## Summary

Keep this open. Current main contains useful mitigations for gateway startup readiness, runtime-deps churn, supervised EADDRINUSE exits, and stuck-session recovery, but the reported root causes are not fully closed: the session-store lock still spans durable atomic write work, plugin restart cleanup remains limited to registered host hooks, and the Manifest/clawrouter listener source is not present in the checked-in plugin tree.

## What This Changes

The issue requests fixes for gateway WebSocket starvation after plugin restarts caused by a Manifest port-binding retry loop, accumulating process signal handlers, and long session persistence lock holds.

## Best Possible Solution

Split the work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a plugin lifecycle cleanup surface for process-level resources, and redesign session persistence so durable writes cannot block latency-sensitive gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the full triad. The report has concrete logs, probes, and independent confirmations, but there is no deterministic current-main reproduction; the session-lock/WS-starvation portion is better evidenced than the Manifest-specific port loop.

## Solution Assessment

No, current main is not a complete solution. It has useful readiness, diagnostics, and runtime-deps mitigations, but the durable-write lock scope, arbitrary process listener cleanup, and missing Manifest listener idempotence remain unresolved.
