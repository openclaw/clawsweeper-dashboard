## Summary

Keep this open. The issue discussion now includes fresh 2026.4.29 reproduction data showing the gateway starvation symptom is still active, and current main still shows mitigations rather than proof that the reported root causes are removed: session persistence still completes durable atomic writes inside the session-store lock, plugin replacement cleanup is limited to registered host hooks, and the reported Manifest/clawrouter listener is not present in the checked-in plugin tree to verify idempotent binding.

## What This Changes

Fix a gateway restart leak triad where Manifest port rebinding, accumulated signal handlers, and long session JSONL persistence lock holds can starve local WebSocket handshakes.

## Best Possible Solution

Treat this as a maintainer-owned gateway stability cluster: make the Manifest/clawrouter listener startup idempotent, add or require a plugin lifecycle cleanup surface for process-level resources such as signal handlers and sockets, and redesign session persistence so slow durable writes cannot block latency-sensitive gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the complete triad on current main. The issue has concrete logs and fresh 2026.4.29 reproduction data for the gateway starvation symptom, but I did not find a deterministic local reproduction path or source proof for the Manifest-specific port loop in the checked-in tree.

## Solution Assessment

No, current main is not yet the best or complete solution. The inspected changes are mitigations around readiness, stale session recovery, and startup pressure; the maintainable fix needs separate root-cause work on listener idempotence, plugin cleanup semantics, and session persistence latency.
