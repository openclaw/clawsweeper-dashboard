## Summary

Keep this open. Current main still matches the session-store fsync/write-lock contributor, plugin host cleanup remains registered-hook scoped rather than a generic process-resource cleanup mechanism, the reported Manifest/clawrouter listener implementation is not present in the checked-in source, and the newly linked #76881 only addresses one arm of the broader regression.

## What This Changes

The issue reports a gateway restart/runtime regression where Manifest port retries, signal listener accumulation, and session-store write-lock stalls can starve WebSocket handshakes.

## Best Possible Solution

Review and land or revise #76881 for the session-store fsync lock cascade, then split the Manifest listener idempotency and plugin process-resource cleanup/CPU-spin causes into focused owner-trackable fixes with regression coverage.

## Reproduction Assessment

Partial yes. The session-store fsync/write-lock contributor is source-reproducible on current main and corroborated by several issue comments, but the full Manifest/signal/session triad is still nondeterministic from this checkout.

## Solution Assessment

No, closing as fixed is not the best solution. The current open PR is a good narrow direction for the session-store fsync arm, but the umbrella should remain open until that lands and the remaining Manifest/signal/CPU-spin causes are fixed or split to canonical follow-ups.
