## Summary

Keep this open. Current main has several mitigations around Gateway readiness, stuck-session recovery, and runtime-deps churn, but the latest issue discussion reports the starvation on v2026.4.29 and the checked-in source still leaves the session-store lock held through durable atomic writes; the Manifest listener source is not present to verify a port-bind fix, and plugin cleanup remains registered-hook scoped.

## What This Changes

The issue reports a Gateway restart/resource-leak cluster involving Manifest port rebinding, signal-handler accumulation, and session persistence lock starvation that blocks WebSocket handshakes.

## Best Possible Solution

Split the root causes into maintainable fixes: make the Manifest/clawrouter listener startup idempotent, add or enforce cleanup/dedup for plugin-owned process resources, and redesign session-store persistence so slow durable writes cannot block Gateway handshakes or session lanes.

## Reproduction Assessment

Partial yes. The session-store starvation path has strong current reproduction evidence from v2026.4.29 comments and source-backed lock/fsync behavior, but the complete three-part Manifest/signal/session triad does not yet have a deterministic current-main reproducer.

## Solution Assessment

No. Current main is not the complete solution; the existing changes reduce symptoms and improve diagnostics, but they do not remove the session persistence critical section or prove the Manifest and signal-handler leak sources are fixed.
