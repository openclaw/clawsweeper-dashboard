## Summary

Keep this open. Current main has useful mitigations and diagnostics, but the reported gateway resource-starvation class is not proven fixed, and newer issue comments show the session/write-lock starvation symptom still reproducing after v2026.4.29.

## What This Changes

The report asks to fix gateway restart resource leaks that can starve WebSocket handshakes: Manifest port rebinding, accumulating process signal handlers, and session-store write-lock/fsync stalls.

## Best Possible Solution

Split this reliability umbrella into maintainer-owned fixes: idempotent Manifest/clawrouter listener startup, generic plugin process-resource cleanup/dedup, and bounded-latency session persistence with an agreed durability contract.

## Reproduction Assessment

Partial yes. The latest comments provide a high-confidence reproduction path for the session-write-lock and event-loop starvation symptom on v2026.4.29, but the full three-part triad is still not deterministically reproducible from current main alone.

## Solution Assessment

No. The current implementation contains mitigations and observability, but it does not remove the underlying lock/fsync latency path, prove process-listener cleanup for arbitrary plugin resources, or verify a Manifest bind fix.
