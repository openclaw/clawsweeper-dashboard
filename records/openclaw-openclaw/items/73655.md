## Summary

Keep this open. Current main has several mitigations around startup readiness, runtime dependency churn, stuck-session recovery, and observability, but the reported root causes are not proven fixed and some are still visibly present in the code path. The latest discussion also reports the same starvation shape on 2026.4.29, so this is an active maintainer investigation rather than stale cleanup.

## What This Changes

The issue requests fixes for Gateway restart starvation caused by Manifest port rebinding, accumulating process signal handlers, and session persistence lock/I/O delays.

## Best Possible Solution

Keep this as the canonical maintainer-owned umbrella until the remaining root causes are split or fixed: make Manifest/clawrouter startup idempotent, add a generic lifecycle path for plugin-owned process resources such as signal handlers and sockets, and redesign session persistence so slow durable writes cannot starve Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the complete triad. The issue has strong logs and multiple deployment confirmations through 2026.4.29, but I did not find a deterministic current-main reproduction harness or a checked-in Manifest listener implementation to exercise locally.

## Solution Assessment

No, the current mitigations are not the complete solution. The narrow maintainable path is to remove the starvation sources in the owning subsystems rather than only extending timeouts, adding readiness probes, or recovering after stale lanes appear.
