## Summary

Keep this open. Current main has added several mitigations around session-store writes, large-session listing, event-loop diagnostics, and session-lock timeout policy, but the source and discussion still do not prove that the reported Manifest port loop, generic signal-handler accumulation, and WebSocket starvation root causes are fixed.

## What This Changes

The issue reports a post-update Gateway restart regression where Manifest port collisions, process signal listener accumulation, and session persistence or lock contention combine to starve WebSocket handshakes while HTTP health remains responsive.

## Best Possible Solution

Split the umbrella into maintainer-owned fixes: make the Manifest/clawrouter listener startup idempotent, define a plugin lifecycle cleanup surface for process-level resources, and keep session persistence/listing work bounded enough that Gateway WebSocket handshakes stay responsive under load.

## Reproduction Assessment

Partial yes. Current source gives a source-level path for session persistence and transcript-lock contention, but there is no deterministic current-main reproduction for the full Manifest/signal/session triad.

## Solution Assessment

No. Closing on the current mitigations would not be the best way to solve this; the maintainable path is to split and fix the remaining resource lifecycle and persistence boundaries with explicit maintainer ownership.
