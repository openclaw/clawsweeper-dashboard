## Summary

Keep this open. Current main still contains the source-backed session-store starvation shape, and the latest discussion reports v2026.4.29 reproductions after the release entry that referenced this issue; the Manifest listener source also was not found in-tree, so the full restart/resource leak triad is not implemented on main.

## What This Changes

The issue reports gateway WebSocket handshake starvation after plugin restarts from Manifest port rebinding, process signal-handler accumulation, and long session-store persistence under lock.

## Best Possible Solution

Split the work into maintainable fixes: make Manifest/clawrouter listener startup idempotent, enforce cleanup for plugin-owned process resources, and redesign session persistence so durable writes cannot block gateway handshakes or session lanes.

## Reproduction Assessment

Partial yes. The session-store starvation path has high-confidence current-release reproduction evidence from v2026.4.29 comments and source-backed lock/fsync behavior, but the complete Manifest plus signal-listener plus session-store triad is not yet deterministically reproduced on current main.

## Solution Assessment

No. Current main has mitigations and diagnostics, but the source still leaves session-store persistence inside the lock and does not prove Manifest listener or process signal-handler cleanup is fixed.
