## Summary

Keep this open. Current main has several useful mitigations around gateway readiness, stale lock cleanup, runtime-deps churn, and large-session responsiveness, but the reported root causes are not fully closed: session-store persistence still holds the store lock through atomic fsync work, plugin replacement cleanup remains hook-scoped, and the reported Manifest/clawrouter listener is not present in the checked-in plugin tree to verify an idempotent listener fix.

## What This Changes

The issue asks OpenClaw to fix a gateway restart stability triad involving Manifest port rebinding, process signal-handler accumulation, and slow session JSON/session-store writes starving WebSocket handshakes.

## Best Possible Solution

Split the work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a plugin lifecycle cleanup surface for process-level resources such as signal handlers and sockets, and redesign session persistence so slow durable writes cannot block latency-sensitive gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the full triad. The issue has concrete logs, probes, and cross-host confirmations, but I did not find a deterministic current-main reproduction path; the session-lock and WS-starvation portion is better evidenced than the Manifest-specific port loop.

## Solution Assessment

No, current main is not yet the best or complete solution. The mitigations improve readiness and recovery, but they do not remove the durable-write lock scope, arbitrary process-listener cleanup gap, or missing Manifest listener idempotence proof.
