## Summary

Keep this open. Current main has useful mitigations for handshake timing, event-loop readiness, stuck-session recovery, and runtime-deps churn, but the reported root causes are not fully closed: session-store writes still occur while holding the store lock, durable atomic writes still await fsync work, plugin replacement cleanup still depends on registered plugin cleanup hooks, and the Manifest/clawrouter listener implementation is not present in the checked-in plugin tree to verify an idempotent port-bind fix. The latest discussion also reports the WS/CPU-starvation symptom still reproducing on 2026.4.29, so this is not stale or implemented.

## What This Changes

The issue asks OpenClaw to fix a gateway restart leak/starvation triad involving Manifest port rebind retries, accumulating process signal handlers, and slow session JSON persistence starving WebSocket handshakes.

## Best Possible Solution

Split the remaining work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, ensure plugin-owned process resources such as signal handlers and sockets are registered and deduplicated through a cleanup lifecycle, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the complete triad. The issue has strong logs, probes, and multiple 4.26/4.29 confirmations for WS/session starvation, but I did not find a deterministic current-main reproduction path or a checked-in Manifest listener implementation to exercise locally.

## Solution Assessment

No, current main is not yet the complete solution. The existing changes are useful mitigations, but the narrowest maintainable fix is a set of targeted lifecycle and persistence changes that remove the underlying starvation sources rather than only extending timeouts or adding recovery after stalls.
