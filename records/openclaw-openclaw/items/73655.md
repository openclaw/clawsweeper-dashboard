## Summary

Keep this open. Current main contains meaningful mitigations for readiness diagnostics, startup/runtime-deps fanout, session listing, session-store writer serialization, and longer write-lock waits, but source inspection still does not prove the reported resource triad is fixed end to end.

## What This Changes

The issue reports a gateway restart/resource leak cluster where Manifest port retries, process signal listener growth, and slow session persistence can starve WebSocket handshakes while HTTP health remains responsive.

## Best Possible Solution

Split this into maintainer-owned root-cause work: make Manifest/clawrouter listener startup idempotent, define a generic plugin lifecycle cleanup surface for process-level resources, and redesign session persistence so slow store writes cannot starve gateway handshakes or session lanes.

## Reproduction Assessment

No for the full issue on current main. There is a source-backed partial path for session-store persistence contention and multiple production reports on 2026.4.26/2026.4.29, but the complete Manifest/signal/session triad was not reproduced against current main.

## Solution Assessment

No. Closing on the current mitigations would not be the best solution because they improve symptoms and observability without proving the Manifest listener, arbitrary plugin resource cleanup, and session persistence root causes are all removed.
