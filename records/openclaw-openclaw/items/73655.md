## Summary

Keep this open. Current main still has the session-store fsync path inside the per-store writer, and the newly linked PR #76881 is open and targets only that strand, while the Manifest/restart and signal-handler parts of the reported triad remain unresolved or at least unproven on main.

## What This Changes

The issue reports Gateway WebSocket starvation after restart under plugin, cron, and session load, with alleged Manifest port retries, signal-listener accumulation, and session-store fsync lock holds.

## Best Possible Solution

Review and land or revise #76881 for the session-store durability trade-off, then split the remaining Manifest listener idempotency and plugin/process resource cleanup into targeted follow-up bugs with owner-specific tests.

## Reproduction Assessment

Partial yes. The session-store strand is source-reproducible because current main still holds the writer through `writeTextAtomic` fsyncs, but the full Manifest/signal/session triad remains non-deterministic from source inspection alone.

## Solution Assessment

No, current main is not the best complete solution yet. The open PR is a plausible narrow fix for the fsync strand, but the broader issue still needs maintainer decisions for Manifest startup idempotency and generic process-resource cleanup.
