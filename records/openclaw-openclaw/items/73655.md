## Summary

Keep this open. Current main has mitigation work for gateway readiness, stuck-session recovery, runtime-deps churn, and diagnostics, but the reported root causes are not proven fixed, and the latest issue discussion includes v2026.4.29 reproductions of the session-store/WS starvation symptom.

## What This Changes

The issue reports a three-part gateway restart regression involving Manifest port rebinding, process signal-listener accumulation, and slow session-store persistence that can starve WebSocket handshakes.

## Best Possible Solution

Split this into maintainer-owned fixes: make the Manifest/clawrouter listener startup idempotent, add a generic plugin lifecycle cleanup or dedup surface for process-level resources, and redesign session-store persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Partial yes. The comments provide high-confidence reproduction evidence for the session-store/WS starvation symptom on v2026.4.29, but the complete three-part restart triad is still emergent rather than deterministically reproducible on current main.

## Solution Assessment

No. Current main contains useful mitigations and diagnostics, but it does not yet implement or prove the narrow root-cause fixes for all three reported resource leaks.
