## Summary

Keep this open. Current main still serializes session-store saves through the writer queue and durable fsync path, while the Manifest listener and signal-handler roots are either outside the checked-in bundled source or only covered by hook-scoped cleanup; #76881 is useful partial follow-up, not proof that the full triad is fixed.

## What This Changes

The issue requests fixes for a gateway restart/starvation regression where plugin port retries, signal listener buildup, and session-store write-lock/fsync work can block WebSocket handshakes while HTTP health remains green.

## Best Possible Solution

Keep the umbrella open, review or land the focused session-store fsync opt-out if it preserves atomic rename semantics, and split the remaining Manifest listener idempotency plus process-resource cleanup roots into separately testable fixes.

## Reproduction Assessment

Partial yes from source inspection: current main still queues session-store updates through saveSessionStoreUnlocked and writeTextAtomic fsync. The full Manifest/signal/session triad remains nondeterministic and was not live-reproduced in this read-only review.

## Solution Assessment

Unclear as a complete solution. The #76881 direction is a plausible narrow repair for the session-store fsync contributor, but closing the whole report would require separate proof for Manifest port binding and signal-handler cleanup.
