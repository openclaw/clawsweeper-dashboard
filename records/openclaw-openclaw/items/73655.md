## Summary

Keep this open. Current main has useful mitigations for event-loop readiness, stuck-session recovery, runtime-deps churn, startup diagnostics, and supervised EADDRINUSE handling, but the reported root causes are not fully closed: the Manifest/clawrouter listener is not present in this checkout, session-store persistence still awaits durable atomic writes while holding the store lock, and plugin cleanup remains limited to registered host-hook resources rather than arbitrary process listeners or sockets. The discussion also has multi-host confirmations and a downstream Paperclip retry/workaround link, so this remains an active gateway lifecycle/performance item rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to fix a Gateway restart/runtime stability triad: Manifest/clawrouter port rebinding, process signal-listener accumulation, and long session persistence lock holds that can starve WebSocket handshakes.

## Best Possible Solution

Split the work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes while preserving the intended durability contract.

## Reproduction Assessment

Unclear for the full triad: the report provides concrete probes, logs, and multi-host confirmations, but not a deterministic one-command reproduction. The session-lock/WS-starvation path is source-supported; the Manifest port-loop path cannot be verified from this checkout because the listener source is not present.

## Solution Assessment

No, the current main mitigation path is not the best complete solution. It improves diagnosis and resilience, but the maintainable fix needs owner-level changes to listener idempotence, plugin lifecycle cleanup contracts, and session persistence lock scope/durability.
