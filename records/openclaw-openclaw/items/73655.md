## Summary

Keep this open. Current main has useful Gateway readiness and diagnostics mitigations, but the reported root causes are not fully resolved: session-store writes still run under the store lock through durable atomic fsync work, plugin replacement cleanup is limited to registered host hooks, and the reported Manifest/clawrouter 2099/2100 listener implementation is not present in the checked-in plugin tree to verify or fix.

## What This Changes

The issue asks OpenClaw to fix a Gateway restart leak cluster: a Manifest/clawrouter EADDRINUSE retry loop, process signal-handler accumulation, and session JSONL write-lock starvation that blocks WebSocket handshakes.

## Best Possible Solution

Keep the issue open and split the fix into maintainer-owned root causes: make Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear. The issue has concrete logs, probes, and multiple independent confirmations for the WS/session-lock starvation shape, but there is no deterministic current-main reproduction for the full triad and the Manifest listener source is not visible in this checkout.

## Solution Assessment

No. Current main has readiness, retry, startup, and cleanup mitigations, but it is not the best complete solution because the lock scope, arbitrary process-listener cleanup, and Manifest listener idempotence remain unresolved.
