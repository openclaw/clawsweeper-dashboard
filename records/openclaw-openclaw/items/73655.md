## Summary

Keep this open. Current main has several mitigations around gateway readiness, stuck-session recovery, runtime-deps churn, and diagnostics, but it does not prove the reported root causes are fixed: session-store persistence still awaits durable write work inside the writer critical section, plugin replacement cleanup remains registered-hook scoped, and the reported Manifest/clawrouter 2099/2100 listener source is not present in this checkout to verify idempotent binding.

## What This Changes

The issue reports a gateway restart resource regression where Manifest port rebinding, accumulating process signal handlers, and slow session persistence can starve Gateway WebSocket handshakes while HTTP health checks still pass.

## Best Possible Solution

Split the work into maintainer-owned fixes: make the Manifest/clawrouter listener startup idempotent, add or enforce cleanup/dedup for plugin-owned process resources, and redesign session persistence so slow durable writes cannot starve Gateway handshakes or session lanes.

## Reproduction Assessment

Partial yes. The latest v2026.4.29 comments provide a high-confidence reproduction path for the session-store/CPU starvation symptom, but the complete Manifest plus signal plus session triad remains nondeterministic and is not reproducible from source alone.

## Solution Assessment

No. Current main contains useful mitigations and diagnostics, but the narrow maintainable fix is still to remove the root causes across plugin listener startup, plugin lifecycle cleanup, and session persistence latency.
