## Summary

Keep open: current main still has the short-only plugin approval contract, and the open closing PR is the active implementation path rather than a merged fix. The request is a useful additive plugin/API feature, but it is not implemented on main and still needs normal PR review plus maintainer API judgment around the deferred custom-prompt behavior.

## What This Changes

Request adds optional long-form plugin approval context and optional custom prompt delivery so capable chat channels can show more than the 256-character fallback description.

## Best Possible Solution

Land a focused additive longDescription contract through protocol schema, gateway payload storage, SDK/renderers, docs, and tests, then track deliverPrompt separately if maintainers want that lifecycle change.

## Reproduction Assessment

Yes for the current limitation: source inspection shows plugin.approval.request accepts only description capped at 256 characters and rejects extra fields. I did not run a live chat flow because this is an additive feature/API request rather than a regression in an existing contract.

## Solution Assessment

Unclear as a complete solution: the additive longDescription direction is plausible and has an open implementation PR, but deliverPrompt:false and channel-specific behavior still need maintainer API/product review. The safer path is to land longDescription narrowly and split custom delivery into a separate decision.
