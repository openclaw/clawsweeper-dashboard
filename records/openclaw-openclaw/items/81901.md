## Summary

Keep open: current main still only supports the 256-character plugin approval description, and a linked open PR now attempts the longDescription path but has not merged and still needs review/proof handling.

## What This Changes

Request an additive long-form plugin approval payload for rich chat channels, plus an optional prompt-delivery opt-out, while preserving the short description fallback.

## Best Possible Solution

Land a focused additive longDescription contract through gateway schema/handler storage, SDK types/renderers, channel presentation, docs, and tests, then track deliverPrompt separately if maintainers want that lifecycle change.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows plugin.approval.request rejects descriptions over 256 characters, rejects extra fields, and the node-policy helper slices descriptions to 256; I did not run a live chat flow because this is a feature/API request rather than a broken existing contract.

## Solution Assessment

Unclear as currently proposed. The additive longDescription direction is plausible and has an open implementation PR, but deliverPrompt:false and per-channel ownership still need maintainer API/product review before the full request is the narrowest maintainable solution.
