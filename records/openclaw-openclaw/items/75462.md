## Summary

Keep this PR open with changes requested. The warning log is useful, but the proposed recovery path is not sufficient because completion failures can occur after the run is already marked ended, and the orphan-recovery path skips normal ended runs.

## What This Changes

The PR replaces the empty `waitForSubagentCompletion` catch block in `src/agents/subagent-registry-run-manager.ts` with a warning log and a scheduled orphan-recovery pass.

## Best Possible Solution

Revise the patch so the catch keeps operator-visible logging but routes ended-run completion failures through the lifecycle cleanup/announce retry path, with a focused regression test and changelog entry. Broader durable completion delivery work should remain with the related subagent completion planning issues.

## Reproduction Assessment

Yes. A focused unit reproduction can make `agent.wait` return `ok` and then make `completeSubagentRun` reject after terminal state has been persisted; the PR's orphan recovery will skip that already-ended non-restart-aborted run.

## Solution Assessment

No. Logging the swallowed error is the right direction, but scheduling orphan recovery is not the narrowest maintainable recovery for failures after terminal persistence; retrying the ended-run cleanup/announce lifecycle is safer.
