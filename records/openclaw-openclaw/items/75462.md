## Summary

Keep this PR open. The underlying bug is real on current main, but the proposed recovery path is incomplete: after terminal state is persisted, `completeSubagentRun` can still throw, and orphan recovery skips ended non-restart-aborted runs, so the PR can still leave the run unannounced. The existing ClawSweeper comment correctly identifies the narrow blocker plus the missing changelog entry.

## What This Changes

The PR replaces `waitForSubagentCompletion`'s empty catch with warning logging and orphan-recovery scheduling, and also adds command-queue priority/starvation scheduling with tests.

## Best Possible Solution

Revise the PR so the catch logs the failure and routes already-ended runs back through an ended-run cleanup/announce retry path, or resets lifecycle state so existing recovery can actually process the run. Keep the command-queue scheduler separate unless it is required for this fix, and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can make `waitForAgentRun` return `ok`, then make `completeSubagentRun` throw after terminal state is persisted, for example from browser cleanup; the proposed orphan recovery then skips the ended run.

## Solution Assessment

No. Logging is the right first step, but scheduling orphan recovery is not the narrow recoverable path for completion failures after `endedAt` is set; an ended-run cleanup/announce retry is the safer implementation.
