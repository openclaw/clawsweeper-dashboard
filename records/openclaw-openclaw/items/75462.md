## Summary

Keep this PR open: it addresses a real swallowed subagent-completion error path, but the proposed recovery call does not recover the important failure mode after terminal state has already been persisted. The existing ClawSweeper review is still supported by current main: logging is useful, but the PR needs a lifecycle cleanup/announce retry path plus the required changelog entry before merge.

## What This Changes

The PR replaces the silent `waitForSubagentCompletion` catch with warning logs and scheduled orphan recovery, and also adds command-queue priority/starvation scheduling with tests.

## Best Possible Solution

Revise the PR so the catch logs the failure and then re-enters a path that can retry ended-run cleanup/announce work, or explicitly resets lifecycle state before using orphan recovery. Add focused regression coverage for `completeSubagentRun` throwing after terminal persistence, add the active-version changelog entry, and split unrelated command-queue priority scheduling unless it is required for this bug fix.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to make `waitForAgentRun` return `ok`, then make `completeSubagentRun` throw after `endedAt` and `outcome` are persisted, such as through `cleanupBrowserSessionsForLifecycleEnd`; the PR then schedules orphan recovery, which skips that ended run.

## Solution Assessment

No. Logging the caught error is the right direction, but orphan recovery is not the narrow recoverable mechanism for already-ended completion failures; a lifecycle cleanup/announce retry path is the safer fix.
