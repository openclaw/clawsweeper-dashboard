## Summary

Keep this PR open because it targets a real swallowed-error path, but the proposed recovery call does not actually recover the post-terminal completion failures it is meant to handle. Orphan recovery skips already-ended runs unless they match the restart-aborted timeout case, so cleanup or announce failures after `endedAt` is persisted can still remain stuck with only a warning.

## What This Changes

The PR adds warning logging plus orphan-recovery scheduling to `waitForSubagentCompletion` errors, and also adds command-queue priority/starvation scheduling APIs with tests.

## Best Possible Solution

Revise the PR so completion failures after terminal wait are routed through an ended-run cleanup/announce retry path, or reset lifecycle state so existing recovery can actually process the run. Keep the warning log, add focused regression coverage for the post-`endedAt` failure case, and add the required active changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can make `waitForAgentRun` return `ok` and then make `completeSubagentRun` throw after terminal state is persisted, such as during browser cleanup; the PR then schedules orphan recovery, which skips the ended non-restart-aborted run.

## Solution Assessment

No. Logging the caught error is useful, but orphan recovery is not the narrow recovery path for already-ended completion cleanup or announce failures; a lifecycle cleanup/announce retry is the safer fix.
