## Summary

This PR is useful but not merge-ready: it adds diagnostics for the swallowed completion error, but the proposed recovery path does not actually recover the main failure mode after a run has already been marked ended. It also bundles an unrelated command-queue priority scheduler into a subagent completion fix and needs the required user-facing changelog entry.

## What This Changes

The PR replaces `waitForSubagentCompletion`'s empty catch with warning logs plus orphan-recovery scheduling, and adds command-queue priority/starvation scheduling with tests.

## Best Possible Solution

Keep the PR open and revise it so completion failures after a terminal wait are routed through an ended-run cleanup/announce retry path, or otherwise reset the lifecycle state to something the existing retry machinery can actually process. Split the command-queue priority scheduler unless it is required for this fix, and add focused regression coverage plus a changelog entry for the user-facing subagent reliability change.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to make `waitForAgentRun` return `ok`, then make `completeSubagentRun` throw after terminal state is persisted, such as from `cleanupBrowserSessionsForLifecycleEnd`; the new catch schedules orphan recovery, but orphan recovery skips the ended non-restart-aborted run.

## Solution Assessment

No. Logging the caught error is the right first step, but scheduling orphan recovery is not the narrow recoverable path for completion/cleanup failures after `endedAt` is already set; the safer fix is to re-enter or schedule the lifecycle cleanup/announce retry for that same ended run.
