## Summary

Keep this PR open, but do not merge it as-is. The logging change is useful, but the recovery call it adds is not effective for completion failures that happen after terminal state has already been persisted, because orphan recovery skips ordinary ended runs.

## What This Changes

The PR replaces the empty `waitForSubagentCompletion` catch block with a warning log and a scheduled orphan-recovery pass in `src/agents/subagent-registry-run-manager.ts`.

## Best Possible Solution

Revise this PR so the catch keeps the warning log but routes post-terminal completion failures through a retryable lifecycle cleanup/announce path, with focused regression coverage and the required changelog entry. Leave broader durable-delivery planning in the linked subagent completion issues.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to make `waitForAgentRun` return `ok`, then have `completeSubagentRun` reject after terminal state has been written; the proposed orphan recovery then skips the already-ended non-restart-aborted run.

## Solution Assessment

No. Logging the swallowed error is the right direction, but `scheduleOrphanRecovery` is not the best recovery path for ended-run cleanup/announce failures; retrying or re-entering the lifecycle cleanup path is safer and narrower.
