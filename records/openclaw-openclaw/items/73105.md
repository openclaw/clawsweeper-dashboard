## Summary

Keep open. The proposed fallback addresses a real current-main gap, but the latest provided diff still emits the synthetic lifecycle directly from `unsubscribe()`, which can overtake pending serialized session events and finalize chat before block-reply work flushes; it also lacks the required user-facing changelog entry.

## What This Changes

This PR modifies embedded pi session subscription state and lifecycle handling to synthesize a terminal lifecycle event during unsubscribe when `agent_end` was not observed, with unit coverage for fallback behavior.

## Best Possible Solution

Keep the backend subscription-level fallback, but route the fallback terminal lifecycle through the same serialized session-event ordering or track an in-flight `agent_end` so teardown only synthesizes a terminal when the terminal event was genuinely missed. Add a focused regression test for pending async flush plus unsubscribe and add the active-version changelog entry.

## Reproduction Assessment

Yes. The original issue has a high-confidence code-level path: timeout aborts the active session fire-and-forget, unsubscribe removes the listener, and current gateway chat finalization depends on lifecycle `end` or `error`. The PR regression can be reproduced with a stub session that queues async `message_end`/`onBlockReplyFlush` work or a queued `agent_end`, then calls `unsubscribe()` before the promise settles.

## Solution Assessment

No, not in the current diff. The subscription seam is the right place to guarantee a terminal lifecycle event, but emitting directly from teardown bypasses existing event queue semantics; queueing the fallback or explicitly tracking terminal-event in-flight state is safer.
