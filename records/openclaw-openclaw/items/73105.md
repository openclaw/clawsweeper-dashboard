## Summary

Keep open. Current main still does not implement the fallback terminal lifecycle, so the PR is not obsolete, but the proposed teardown fallback still has a concrete ordering regression and is missing the required changelog entry for a user-facing Control UI fix.

## What This Changes

The PR shares terminal lifecycle idempotency in embedded subscription state, emits a synthetic terminal lifecycle event during unsubscribe when `agent_end` was missed, and adds unsubscribe-fallback regression coverage.

## Best Possible Solution

Keep the backend subscription-level fallback, but make terminal fallback emission preserve the same serialized ordering as normal session events, or explicitly track `agent_end` as seen/in-flight before synthesizing a terminal event. The final PR should also include focused regression coverage for pending async flush plus unsubscribe and a changelog entry for the user-facing stuck-Stop fix.

## Reproduction Assessment

Yes. The original bug has a high-confidence code path through timeout, fire-and-forget `activeSession.abort()`, finally-block `unsubscribe()`, and missing lifecycle `end`/`error`; the PR regression can be reproduced with a stub session that leaves an async `message_end` or `onBlockReplyFlush` pending, then calls `unsubscribe()` before it settles.

## Solution Assessment

No, not in the current PR shape. The subscription seam is the right place to guarantee exactly one terminal lifecycle event, but emitting directly from teardown bypasses the established event queue and can finalize chat before pending message flush work completes.
