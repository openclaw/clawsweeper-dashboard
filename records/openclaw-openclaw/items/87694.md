## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor base-only billing overrides in the clamp:** `src/agents/auth-profiles/usage.ts:513`
  - body: `resolveBillingDisableCeilingMs` returns the raw `billingMaxMs`, but failure writes use `calculateDisabledLaneBackoffMs`, which forces the effective max to be at least the base. With this PR, a user preserving a longer pause via `auth.cooldowns.billingBackoffHours: 5` but no explicit `billingMaxHours` can write a ~5h disable and then have normal profile ordering clamp it back to 15m, ignoring the configured base. Use the same effective ceiling as the write path or add compatibility coverage for this upgrade case.
  - confidence: 0.9
- **[P1] Preserve implicit billing cooldown compatibility:** `src/agents/auth-profiles/usage.ts:453-454`
  - body: Changing these defaults moves existing users with no explicit `auth.cooldowns` config from a 5h/24h billing pause to about 5m/15m. That can make billing-blocked providers retry much sooner after upgrade, so this needs explicit maintainer-approved policy or a compatibility-preserving recovery path before merge.
  - confidence: 0.86
- **[P1] Gate the runtime state clamp behind an upgrade decision:** `src/agents/auth-profiles/order.ts:247-251`
  - body: Calling the clamp from `resolveAuthProfileOrder` changes persisted billing-disable semantics during normal auth-profile reads before the existing expiry sweep. That fixes the stale-lockout case, but it is also an auth/session-state upgrade behavior change and should be narrowed, migrated, or explicitly accepted by auth-provider owners before merge.
  - confidence: 0.82
