## Summary

Keep this PR open: the direction addresses a real auth-provider lockout, and the contributor removed the one-off proof artifacts, but the current clamp can still leave stale persisted billing lockouts stuck across fresh requests because it recomputes a new now-plus-cap window without persisting or anchoring the original window. The shortened defaults also remain a compatibility-sensitive auth-provider policy change that needs owner acceptance before merge.

## What This Changes

The PR lowers billing auth cooldown defaults to about 5 minutes/15 minutes, adds an on-read clamp for stale billing `disabledUntil` values during auth profile ordering, and updates docs plus auth-profile regression tests.

## Best Possible Solution

Make the stale billing clamp durable or anchored so it elapses across fresh requests, keep explicit operator overrides intact, then get auth-provider owner acceptance for the shortened defaults and upgrade behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the PR-introduced bug: load a stale persisted billing `disabledUntil`, call `resolveAuthProfileOrder`, wait past the new cap, then create a fresh store/request; the clamp recomputes from the new `now` instead of preserving an elapsed window. I did not run a mutating test in this read-only review.

## Solution Assessment

No, not yet: lowering the defaults is a plausible narrow fix for the reported lockout, but the current on-read clamp is not durable across fresh requests and therefore does not reliably solve the persisted-state recovery path. The safer implementation should persist the clamped value or anchor it to the original failure/window, with a fresh-store regression test.
