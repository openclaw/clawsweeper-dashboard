## Summary

Keep open: the PR targets a real split-UTF-8 corruption path, but it is stale against current main, lacks after-fix real behavior proof, and should move the buffering into the shared decoder instead of reintroducing child-adapter-only decoding.

## What This Changes

The PR replaces child-process stdout/stderr per-chunk Buffer decoding with per-stream Node StringDecoder instances and end flushing.

## Best Possible Solution

Move stateful UTF-8 buffering into createWindowsOutputDecoder, keep child.ts wired through that helper, preserve Windows fallback behavior, add split-CJK coverage, and require redacted real behavior proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main still has a per-chunk Buffer.toString("utf8") default decoder path, and a read-only Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run.

## Solution Assessment

No. StringDecoder or equivalent stateful UTF-8 decoding is the right primitive, but the maintainable fix belongs in createWindowsOutputDecoder so child stdout/stderr preserve shared flushing and Windows legacy-codepage fallback.
