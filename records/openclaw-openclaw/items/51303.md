## Summary

Keep open: the underlying UTF-8 boundary bug is valid, but this branch is not ready because current main now routes child output through the shared decoder, where the remaining unsafe default UTF-8 path should be fixed, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR changes child process stdout and stderr decoding to use per-stream StringDecoder instances instead of per-chunk Buffer.toString().

## Best Possible Solution

Make the default UTF-8 path in createWindowsOutputDecoder() boundary-safe, keep the child adapter using that shared decoder, add split-CJK coverage for the decoder and adapter wiring, and verify the reported route or an equivalent child stream.

## Reproduction Assessment

Yes for the core byte-boundary failure from source and a focused Node probe: current main’s default createWindowsOutputDecoder() path decodes each Buffer chunk with buffer.toString("utf8"), and split 测试 bytes produce replacement characters. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. StringDecoder is the right primitive, but this PR targets stale local child adapter handlers; the maintainable fix belongs in createWindowsOutputDecoder() so Windows legacy fallback and shared flush behavior stay intact.
