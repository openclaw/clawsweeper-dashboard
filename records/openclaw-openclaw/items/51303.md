## Summary

Keep open: the PR targets a real UTF-8 split-byte corruption bug, but current main now routes child stdout/stderr through a shared Windows-aware decoder, and this stale child-local patch would need revision plus real behavior proof before it is mergeable.

## What This Changes

The PR adds per-stream `node:string_decoder` `StringDecoder` instances to child-process stdout and stderr handlers so split UTF-8 chunks are buffered and flushed on stream end.

## Best Possible Solution

Put stateful UTF-8 buffering inside `createWindowsOutputDecoder`, keep child stdout/stderr wired through that helper, preserve Windows legacy-codepage fallback, add split-CJK regression coverage, and require redacted real output proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current source has a per-chunk `Buffer.toString("utf8")` default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the maintainable current-main fix belongs in `createWindowsOutputDecoder` rather than a child-local replacement.
