## Summary

Keep open: the PR targets a real byte-boundary corruption path, but its direct child-adapter StringDecoder patch is stale against current main’s shared output decoder and still lacks real after-fix behavior proof.

## What This Changes

The PR replaces per-chunk stdout/stderr `chunk.toString()` decoding in `src/process/supervisor/adapters/child.ts` with per-stream Node `StringDecoder("utf8")` instances and end flushes.

## Best Possible Solution

Move boundary-safe default UTF-8 streaming into `createWindowsOutputDecoder()`, keep the child adapter on that abstraction, add split-CJK regression coverage, and attach redacted real behavior proof for the affected route.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main’s default decoder still decodes each Buffer with `buffer.toString("utf8")`, and a focused Node probe shows split Chinese bytes produce U+FFFD that `StringDecoder` avoids. I did not live-reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive for pure UTF-8, but the current best fix should live in `createWindowsOutputDecoder()` so child output stays on the shared decoder and preserves Windows legacy-codepage fallback behavior.
