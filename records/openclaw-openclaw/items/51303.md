## Summary

Keep this PR open, but it needs changes before merge. The UTF-8 boundary bug is real and the proposed fix was directionally right for the older child adapter, but current main now routes child stdout/stderr through the shared Windows-aware output decoder, so a direct `StringDecoder` patch in `child.ts` would bypass shipped Windows codepage behavior and leave the remaining default decoder bug in the wrong layer.

## What This Changes

The PR imports Node `StringDecoder` in the child process adapter and uses per-stream decoder instances to decode stdout and stderr chunks across UTF-8 byte boundaries.

## Best Possible Solution

Implement boundary-safe UTF-8 streaming in `src/infra/windows-encoding.ts` so the default/non-legacy path buffers split UTF-8 code points while preserving the Windows legacy-codepage fallback and valid-UTF-8-first behavior introduced for #50519.

## Reproduction Assessment

Yes. Splitting a multibyte UTF-8 character across chunks reproduces U+FFFD corruption with per-chunk `Buffer.toString("utf8")`, and current main still uses that behavior in the shared decoder default path.

## Solution Assessment

No. The direct `StringDecoder` change in `child.ts` was appropriate for the old code shape, but current main centralizes child output decoding in `createWindowsOutputDecoder()`, so the narrow maintainable fix belongs in that shared decoder.
