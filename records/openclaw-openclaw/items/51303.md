## Summary

Keep this PR open with changes requested. The UTF-8 split-chunk bug is real, but current main now routes child stdout and stderr through the shared Windows-aware decoder, so the submitted direct `StringDecoder` change is stale and would put the fix in the wrong layer.

## What This Changes

The PR changes `src/process/supervisor/adapters/child.ts` to decode child stdout and stderr chunks with per-stream Node `StringDecoder` instances and flush them on stream end.

## Best Possible Solution

Implement boundary-safe UTF-8 streaming inside `createWindowsOutputDecoder()` so the default and UTF-8 paths buffer split multibyte code points, while preserving the Windows legacy-codepage fallback and keeping the child adapter on the shared decoder contract.

## Reproduction Assessment

Yes. Splitting a multibyte UTF-8 character across chunks reproduces U+FFFD corruption with per-chunk `Buffer.toString("utf8")`, and current main still uses that behavior in the shared decoder default path.

## Solution Assessment

No. The submitted child-adapter patch was appropriate for the older code shape, but the narrow maintainable fix now belongs in `src/infra/windows-encoding.ts` so all child stdout/stderr decoding paths share the same boundary-safe behavior.
