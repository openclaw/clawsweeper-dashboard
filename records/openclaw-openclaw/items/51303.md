## Summary

Keep open: the bug signal is real, but this PR is not merge-ready because current main now routes child stdout/stderr through createWindowsOutputDecoder, so the child-local StringDecoder patch would conflict with or bypass Windows codepage handling; it also lacks after-fix real behavior proof and is currently conflicting.

## What This Changes

The PR changes child-process stdout and stderr decoding to use per-stream node:string_decoder StringDecoder instances with end flushes for UTF-8 chunks.

## Best Possible Solution

Put stateful UTF-8 buffering inside createWindowsOutputDecoder, keep child stdout/stderr wired through that helper, preserve Windows legacy-codepage fallback, add split-CJK regression coverage, and require redacted real output proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current source has a per-chunk UTF-8 fallback path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the maintainable current-main fix belongs in createWindowsOutputDecoder rather than a child-local replacement.
