## Summary

Keep open. The Greptile comment correctly identified the original UTF-8 boundary bug, but current main now routes child stdout/stderr through the shared Windows-aware decoder, so this PR’s direct child-adapter `StringDecoder` patch is no longer the right implementation boundary.

## What This Changes

The PR changes child-process stdout and stderr decoding to use per-stream UTF-8 `StringDecoder` instances instead of per-chunk `toString()` decoding.

## Best Possible Solution

Keep the useful source work open, then land a narrow repair in `createWindowsOutputDecoder()` with regression coverage for default UTF-8 streaming paths while preserving the Windows legacy-codepage fallback.

## Reproduction Assessment

Yes, at source level. Splitting a multibyte UTF-8 character across chunks reproduces U+FFFD corruption with the current default `buffer.toString("utf8")` branch used by `createWindowsOutputDecoder()`.

## Solution Assessment

No. The PR was reasonable for the older child-adapter code shape, but current main’s maintainable fix belongs in the shared decoder so stdout and stderr keep the Windows-aware decoding contract.
