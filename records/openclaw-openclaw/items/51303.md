## Summary

Keep open: the PR targets a real UTF-8 chunk-boundary bug, but the branch is stale against current main’s shared process-output decoder and would bypass the Windows legacy-codepage fallback if rebased in the same shape. It also still lacks after-fix real behavior proof for the reported streaming path.

## What This Changes

The PR replaces child-process stdout/stderr per-chunk `toString()` decoding with per-stream `StringDecoder("utf8")` instances and end flushes in `src/process/supervisor/adapters/child.ts`.

## Best Possible Solution

Move boundary-safe UTF-8 streaming into `createWindowsOutputDecoder()`, keep the child adapter on that shared abstraction, add split-CJK coverage for both the default decoder and adapter wiring, and validate the real streaming route or an equivalent child stream.

## Reproduction Assessment

Yes for the core byte-boundary failure from source and a focused Node probe: current main’s default child-output decoder still calls `buffer.toString("utf8")` per chunk. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive for UTF-8 chunk boundaries, but the maintainable fix belongs in the shared `createWindowsOutputDecoder()` path so Windows legacy-codepage fallback and shared flush behavior are preserved.
