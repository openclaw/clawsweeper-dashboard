## Summary

## What This Changes

The PR replaces child stdout/stderr per-chunk `chunk.toString()` decoding with per-stream Node `StringDecoder` instances and end flushing.

## Best Possible Solution

Keep the canonical issue open and land a narrow replacement that makes `createWindowsOutputDecoder` boundary-safe on its default UTF-8 path while preserving Windows legacy-codepage fallback and adding split-CJK child-output proof.

## Reproduction Assessment

Yes for the core byte-boundary failure: source inspection shows the shared default path still uses per-chunk `Buffer.toString("utf8")`, and a focused Node probe reproduced replacement characters for split Chinese bytes. The full WebChat/cursor2api route was not live-run.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the PR's child-local implementation is no longer the best fix because current main centralizes child stdout/stderr decoding in `createWindowsOutputDecoder` for Windows compatibility.
