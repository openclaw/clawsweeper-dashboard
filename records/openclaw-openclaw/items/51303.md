## Summary

Keep open: this PR targets a real UTF-8 split-byte bug, but current main now routes child stdout/stderr through the shared `createWindowsOutputDecoder`, so this branch needs revision into that helper plus real behavior proof before merge.

## What This Changes

The PR replaces child adapter stdout/stderr per-chunk `Buffer.toString()` decoding with per-stream `node:string_decoder` `StringDecoder` instances and flushes them on stream end.

## Best Possible Solution

Move stateful UTF-8 buffering into `createWindowsOutputDecoder`, keep `child.ts` wired through that helper, add split-CJK coverage for the default decoder and child adapter wiring, then require redacted real behavior proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current source has a per-chunk `Buffer.toString("utf8")` default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the proposed child-local fix is stale; the maintainable fix now belongs in `createWindowsOutputDecoder` so child stdout/stderr preserve shared flushing and Windows legacy-codepage fallback.
