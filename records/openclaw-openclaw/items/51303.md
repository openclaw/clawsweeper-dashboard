## Summary

Keep open: the UTF-8 boundary bug is still plausible on current main, but this PR is stale against the current shared decoder path, conflicts with current main, and lacks after-fix real behavior proof.

## What This Changes

The branch adds Node StringDecoder decoding and end flushing directly to the child adapter stdout/stderr handlers.

## Best Possible Solution

Fix the default UTF-8 branch of createWindowsOutputDecoder with stateful decoding, keep child.ts on the shared abstraction, add split-CJK regression coverage, and attach redacted real child-output or WebChat proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main still has a per-chunk Buffer.toString("utf8") path in the shared decoder, and a read-only Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

No. StringDecoder or equivalent stateful UTF-8 decoding is the right primitive, but the maintainable fix should live in createWindowsOutputDecoder so child stdout/stderr preserve shared flushing and Windows legacy-codepage fallback.
