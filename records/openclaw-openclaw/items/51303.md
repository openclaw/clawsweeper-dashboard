## Summary

Keep open: the linked UTF-8 split-chunk bug is still present in current main’s default decoder path, but this PR is stale against the shared decoder refactor, conflicts with main, and lacks required after-fix real behavior proof.

## What This Changes

The PR replaces child process stdout/stderr per-chunk `toString()` decoding with per-stream Node `StringDecoder` instances and end flushing in `src/process/supervisor/adapters/child.ts`.

## Best Possible Solution

Make the default UTF-8 branch in `createWindowsOutputDecoder()` boundary-safe, keep the child adapter on that shared decoder, add split-CJK coverage for the decoder and adapter wiring, and verify the reported route or an equivalent real child stream.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main’s default decoder path uses per-chunk `buffer.toString("utf8")`, and a focused Node probe shows split Chinese bytes become replacement characters. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive, but this PR edits stale child adapter handlers; the maintainable fix belongs in `createWindowsOutputDecoder()` so shared flush behavior and Windows legacy-codepage fallback stay intact.
