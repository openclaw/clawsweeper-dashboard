## Summary

Keep open: the UTF-8 split-byte bug is real, but this branch is stale against current main’s shared child-output decoder path and still lacks after-fix real behavior proof.

## What This Changes

The PR replaces child adapter stdout/stderr per-chunk `toString()` decoding with per-stream Node `StringDecoder` instances and end flushing.

## Best Possible Solution

Revise or replace this PR so the default UTF-8 branch of `createWindowsOutputDecoder()` is boundary-safe, the child adapter keeps using the shared decoder, split-CJK regression coverage is added, and redacted real behavior proof is attached.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main’s default decoder path uses per-chunk `buffer.toString("utf8")`, and the focused Node probe shows split Chinese bytes become replacement characters. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive, but this PR applies it in stale child adapter handlers instead of the shared decoder, so it would bypass the current abstraction and miss shared coverage.
