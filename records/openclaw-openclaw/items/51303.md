## Summary

This PR targets a real UTF-8 boundary bug, but it should stay open because the branch is stale against the current shared Windows output decoder path, is merge-conflicting, and still lacks after-fix real behavior proof for the reported child-output/WebChat behavior.

## What This Changes

The PR changes `src/process/supervisor/adapters/child.ts` to decode child stdout and stderr Buffer chunks with per-stream `StringDecoder` instances and flush them on stream end.

## Best Possible Solution

Put stateful UTF-8 buffering inside `createWindowsOutputDecoder`, keep child stdout/stderr wired through that shared helper, add split-CJK tests for the default decoder path, and require redacted real behavior proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: source inspection shows the current default decoder path still uses per-chunk UTF-8 Buffer decoding, and a focused Node probe reproduces replacement characters for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the proposed child-local patch is no longer the best shape because current main centralizes child output decoding in `createWindowsOutputDecoder`.
