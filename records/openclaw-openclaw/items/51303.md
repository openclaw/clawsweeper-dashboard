## Summary

Keep open: this targets a real UTF-8 split-byte bug and the linked issue remains open, but the branch is now conflicting and predates current main's shared output decoder, so it needs revision and real behavior proof before merge.

## What This Changes

The PR changes child process stdout/stderr handlers to decode Buffer chunks with `node:string_decoder.StringDecoder` and flush remaining decoder state on stream end.

## Best Possible Solution

Revise or replace this PR so `createWindowsOutputDecoder` uses stateful UTF-8 decoding on its default path, preserves Windows codepage fallback and flush semantics, adds split-CJK regression coverage, and includes real behavior proof.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main's default decoder branch uses per-chunk `buffer.toString("utf8")`, and a focused Node probe shows split Chinese bytes produce U+FFFD while `StringDecoder` preserves them. The full WebChat/cursor2api route was not live-run.

## Solution Assessment

No. `StringDecoder` is the right primitive, but this PR targets an older child adapter; the maintainable current-main fix belongs in the shared output decoder while preserving the Windows legacy-codepage fallback.
