## Summary

Keep open: the PR targets a real split UTF-8 corruption path, but it is not merge-ready because current main now routes child output through `createWindowsOutputDecoder`; the fix needs to move into that shared helper, preserve Windows fallback behavior, and include real behavior proof.

## What This Changes

This PR replaces child-process stdout/stderr per-chunk Buffer decoding in `child.ts` with `node:string_decoder` `StringDecoder` and stream-end flushing.

## Best Possible Solution

Move stateful UTF-8 buffering into `createWindowsOutputDecoder`, keep `child.ts` using that helper, add split-CJK regression coverage for the default decoder and child adapter wiring, and require redacted real behavior proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main has a per-chunk `Buffer.toString("utf8")` default decoder path, and a read-only Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run.

## Solution Assessment

No. `StringDecoder` or equivalent stateful UTF-8 decoding is the right primitive, but the maintainable fix belongs in `createWindowsOutputDecoder` so child stdout/stderr preserve shared flushing and Windows legacy-codepage fallback.
