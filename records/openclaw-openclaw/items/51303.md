## Summary

Keep open: this PR targets a real split UTF-8 corruption bug, but current main now owns child output decoding through the shared Windows-aware decoder; the direct child-adapter patch is stale, conflicting, and missing after-fix real behavior proof.

## What This Changes

The PR changes child-process stdout/stderr Buffer decoding from per-chunk `toString()` calls to per-stream `StringDecoder` instances with end flushing.

## Best Possible Solution

Move stateful UTF-8 buffering into `createWindowsOutputDecoder`, keep `child.ts` wired through that helper, add split-CJK coverage for the default decoder and child adapter wiring, then require redacted real behavior proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main has a per-chunk `Buffer.toString("utf8")` default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

No. Stateful UTF-8 decoding is the right primitive, but the maintainable fix now belongs in `createWindowsOutputDecoder` so child stdout/stderr preserve shared flushing and Windows legacy-codepage fallback.
