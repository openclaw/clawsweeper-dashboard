## Summary

Keep open: this is a real UTF-8 chunk-boundary bug, but the PR is now stale against current main's shared Windows-aware output decoder and still needs after-fix real behavior proof before merge.

## What This Changes

This PR changes `src/process/supervisor/adapters/child.ts` to decode stdout/stderr Buffer chunks with Node `StringDecoder` and flush decoder tails on stream end.

## Best Possible Solution

Move boundary-safe default UTF-8 streaming into `createWindowsOutputDecoder()`, keep the child adapter on that shared abstraction, add split-CJK regression coverage for the default decoder and adapter wiring, and attach redacted real behavior proof for the affected route.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main still has a per-chunk default UTF-8 decode path, and a focused Node probe shows split Chinese bytes produce U+FFFD with `Buffer.toString("utf8")`. I did not live-reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive for pure UTF-8, but the current maintainable fix should live in `createWindowsOutputDecoder()` so it preserves the Windows legacy-codepage fallback already on main.
