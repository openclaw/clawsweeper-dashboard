## Summary

Keep open: this is a plausible fix for a real UTF-8 chunk-boundary bug, but the branch predates current main’s shared output decoder, would bypass Windows decoder behavior when rebased, and still lacks required real behavior proof.

## What This Changes

The PR replaces child-process stdout/stderr per-chunk `toString()` decoding with per-stream `StringDecoder("utf8")` instances and flushes them on stream end.

## Best Possible Solution

Make the default UTF-8 path in `createWindowsOutputDecoder()` boundary-safe, keep the child adapter using that shared decoder, add split-CJK coverage for the decoder and adapter wiring, and verify the real reported route or an equivalent child stream.

## Reproduction Assessment

Yes for the core byte-boundary failure from source and a focused Node probe: current main’s default `createWindowsOutputDecoder()` path still decodes each Buffer chunk with `buffer.toString("utf8")`, and split `测试` bytes produce replacement characters. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive, but this PR targets older local adapter handlers; the maintainable fix belongs in `createWindowsOutputDecoder()` so Windows legacy fallback and shared flush behavior stay intact.
