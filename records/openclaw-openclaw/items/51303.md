## Summary

Keep open: the UTF-8 boundary bug remains plausible on current main, but this PR targets a stale child-adapter location and needs revision plus real behavior proof before merge.

## What This Changes

The branch adds per-stream node:string_decoder StringDecoder decoding and end flushing to stdout/stderr handlers in src/process/supervisor/adapters/child.ts.

## Best Possible Solution

Fix the default UTF-8 branch of createWindowsOutputDecoder with stateful decoding, keep child.ts on the shared abstraction, add split-CJK regression coverage, and attach redacted real child-output or WebChat proof before landing.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main still has a per-chunk Buffer.toString("utf8") path, and a focused Node probe reproduces U+FFFD corruption for split Chinese bytes. I did not live-run the full WebChat/cursor2api setup.

## Solution Assessment

No. StringDecoder or an equivalent stateful UTF-8 decoder is the right primitive, but this PR applies it in stale child-adapter handlers instead of the shared decoder that current main uses.
