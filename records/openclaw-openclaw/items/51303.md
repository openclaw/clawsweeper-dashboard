## Summary

Keep open: the reported UTF-8 boundary bug is still source-reproducible on current main, and this branch needs rework against the shared output decoder plus real behavior proof before it is merge-ready.

## What This Changes

The PR replaces per-chunk child stdout/stderr `chunk.toString()` decoding with `StringDecoder` buffering and end flushing.

## Best Possible Solution

Update the shared output decoder so the default UTF-8 path is streaming and boundary-safe, preserve the Windows legacy-codepage fallback, add split-CJK regression coverage, and verify the linked WebChat scenario before merge.

## Reproduction Assessment

Yes for the implicated byte-boundary path: source inspection plus a Node split-byte check show per-chunk UTF-8 decoding corrupts split CJK bytes, and current main still uses that path in the default decoder. I did not live-reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No, not as-is. `StringDecoder` is the right primitive for the old direct child adapter, but current main routes child output through `createWindowsOutputDecoder`, so the maintainable fix should update that shared decoder and keep Windows codepage behavior.
