## Summary

Keep open: the PR addresses a real split-UTF-8 corruption path, but the branch now patches the older child-adapter decode site instead of the current shared output decoder and it still lacks after-fix real behavior proof.

## What This Changes

This PR decodes child-process stdout/stderr Buffer chunks with per-stream Node `StringDecoder("utf8")` instances and flushes decoder tails on stream end.

## Best Possible Solution

Move boundary-safe UTF-8 streaming into `createWindowsOutputDecoder()`, preserve the Windows legacy-codepage fallback, add split-CJK coverage for the default decoder and child-adapter wiring, and validate the linked WebChat route or an equivalent real child stream.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main’s default output decoder still calls `buffer.toString("utf8")` per chunk, and a Node probe showed split Chinese bytes become U+FFFD. I did not reproduce the full WebChat/cursor2api setup.

## Solution Assessment

No. `StringDecoder` is the right primitive for pure UTF-8, but the current maintainable fix belongs in `createWindowsOutputDecoder()` so the child adapter keeps the shared Windows legacy-codepage behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84
