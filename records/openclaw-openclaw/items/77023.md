## Summary

Keep open: the runtime/docs direction is coherent, but the PR removes previously documented queue-mode literals from validation without adding a doctor/raw-config migration, so existing configs using `messages.queue.mode: "steer"` or per-channel steering modes can fail before the new fallback resolver helps.

## What This Changes

The PR makes mid-run prompts steer active runs by default, narrows `/queue` and `messages.queue` to fallback modes, batches Codex `turn/steer` delivery, and updates related docs, generated config schema, changelog, and tests.

## Best Possible Solution

Keep the default-steering simplification, but add a focused legacy doctor migration/rule for retired queue-mode config and session override values before removing them from the public schema.

## Reproduction Assessment

Yes for the review finding from source inspection: the PR removes old queue literals from config validation while current load code fails closed before runtime queue normalization. I did not execute the PR branch in this read-only review.

## Solution Assessment

No: the implementation needs a compatibility repair so existing documented `messages.queue` values are migrated or clearly repairable by `openclaw doctor --fix` before the schema rejects them.
