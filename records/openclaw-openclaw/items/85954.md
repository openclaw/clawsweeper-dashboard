## Summary

Keep open: current main and v2026.6.9 still render iMessage approval prompts as SDK plain text plus the plain reaction hint, while the issue has protected maintainer/product-decision labels that make this a maintainer-held UX choice rather than conservative cleanup.

## What This Changes

Request iMessage-only Markdown decoration for approval prompt headers, labels, reaction choices, and possibly the manual `/approve` fallback line so Apple clients render native attributed-body styling.

## Best Possible Solution

If maintainers want this UX, keep SDK approval text channel-agnostic and add a small iMessage-only decoration pass before send, with focused renderer tests and live iMessage proof.

## Reproduction Assessment

Yes: source inspection on current main and v2026.6.9 shows the approval renderer still emits plain prompt text, while the sender already supports Markdown-derived formatting ranges when markers are present. I did not run tests because this review is read-only and test commands may write artifacts.

## Solution Assessment

Yes, if maintainers accept the UX direction: the narrowest maintainable fix is an iMessage-owned formatting pass in `approval-native.ts` after SDK payload generation and before send. Changing SDK approval text would be worse because it would leak iMessage presentation markers to other channels.
