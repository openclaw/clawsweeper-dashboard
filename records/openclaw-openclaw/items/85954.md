## Summary

Keep open: current main and latest release v2026.6.9 still have generic iMessage Markdown-to-attributed-body support, but the approval prompt builders still emit plain SDK text plus a plain reaction hint. The protected maintainer/product-decision labels also make this a maintainer-held UX decision rather than conservative cleanup.

## What This Changes

The issue requests iMessage-only Markdown markers around approval prompt headers and labels so existing attributed-body formatting renders them as native emphasis.

## Best Possible Solution

If maintainers want this UX, keep SDK approval text channel-agnostic and add a small iMessage-only decoration pass before send, with focused renderer tests and live iMessage proof.

## Reproduction Assessment

Yes: source inspection on current main and v2026.6.9 shows the approval renderer still emits plain prompt text, while the sender already supports Markdown-derived formatting ranges when markers are present. I did not run tests because this review is explicitly read-only and test commands may write artifacts.

## Solution Assessment

Yes, if maintainers accept the UX direction: the narrowest maintainable fix is an iMessage-owned formatting pass in approval-native.ts after SDK payload generation and before send. Changing SDK approval text would be worse because it would leak iMessage presentation markers to other channels.
