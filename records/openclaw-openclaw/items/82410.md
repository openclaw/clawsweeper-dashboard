## Summary

Keep open for review, not cleanup: the compatibility problem is plausible and the linked discussion points to existing OpenAI-compatible `reasoning_content` handling, but this branch cannot merge because it fails TypeScript and does not read the current replay shape; it also lacks after-fix real behavior proof.

## What This Changes

The PR changes `convertAnthropicMessages()` to copy a non-empty `reasoning_content` property from assistant messages into serialized Anthropic Messages payloads.

## Best Possible Solution

Implement a provider-gated Anthropic Messages compatibility path only after confirming the real proxy wire contract, read replayed reasoning from the actual OpenClaw thinking-block representation, preserve direct Anthropic request shape, and add focused regression coverage plus real proxy proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the reported provider runtime failure because no real proxy proof is attached and I did not live-test that route.

## Solution Assessment

Is this the best way to solve the issue? No. The patch reads a field current assistant messages do not have and applies a non-standard payload field too broadly; a maintainable fix should be provider-gated and based on the existing thinking-block replay contract.
