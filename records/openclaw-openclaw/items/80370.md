## Summary

Keep open: the telemetry feature is plausible, but the patch misses the real direct skill command-dispatch path and the external proof is only targeted tests, not real behavior proof.

## What This Changes

The PR adds a `skill.used` diagnostics event exported through OTEL and Prometheus, plus docs/tests and a separate skills watcher polling env override.

## Best Possible Solution

Wire `skill.used` at the trusted skill command-dispatch boundary or pass trusted skill context into that tool path, keep labels bounded, and demonstrate the exported metric/span from a real run.

## Reproduction Assessment

Yes for the review finding: source inspection shows direct command-dispatch invokes a tool with `skillName` but does not pass the `skillsSnapshot` required by the new detector. I did not run tests because this was a read-only review.

## Solution Assessment

No. The exporter side is a reasonable direction, but the command-dispatch implementation is wired to the wrong context and should be fixed at the trusted dispatch boundary or by passing trusted skill context through that path.
