## Summary

Keep open: the PR is a plausible diagnostics feature, but the latest head still has blocking correctness issues around trusted command attribution and double hook wrapping, and the external contributor proof is test-only.

## What This Changes

The PR adds a `skill.used` diagnostic event and exports skill usage through Prometheus and OpenTelemetry, with docs and targeted tests.

## Best Possible Solution

Thread trusted skill-read and command-dispatch metadata through the existing hook context or command catalog, keep exporter labels bounded, and land only after maintainer product approval plus redacted real telemetry proof.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and PR head shows the double hook wrapper, params-derived command attribution, and prompt-visible snapshot versus command-catalog mismatch. I did not run a live Gateway telemetry scrape.

## Solution Assessment

No. The exporter shape is plausible, but the emission path should use trusted dispatch/read context and command-visible skill metadata rather than model/tool params and a second hook wrapper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools on current main. Wrapping the selected inline command tool again can rerun hook policy, loop tracking, and tool execution diagnostics; pass the skill telemetry context into the existing construction-time hook context instead.
  - confidence: 0.93
- **[P2] Require trusted dispatch context for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:196-227`
  - body: This treats any successful wrapped tool call with `commandName` and `skillName` params as skill command usage. Those are ordinary tool params at this generic hook boundary, so normal tool calls can spoof `skill.used`; carry a trusted dispatch marker or skill metadata from inline command dispatch instead.
  - confidence: 0.9
- **[P2] Use command-visible skill metadata for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:223-227`
  - body: `resolvedSkills` contains prompt-visible skills, but command specs are built from eligible user-invocable entries. A hidden user-invocable skill command can dispatch its tool and still miss `skill.used`; use the command catalog or full runtime skill entry.
  - confidence: 0.9
