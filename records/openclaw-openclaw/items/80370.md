## Summary

Keep this PR open, but it is not merge-ready: the feature is useful in principle, yet the current patch still derives trusted command telemetry from ordinary tool params, double-wraps already wrapped inline tools, misses some command-visible skills, and only supplies test-based proof.

## What This Changes

Adds a `skill.used` diagnostic event for skill reads and command-dispatched skill tools, exports it through Prometheus and OTEL, and updates docs, changelog, and tests.

## Best Possible Solution

If maintainers want this in core, thread trusted skill-read and command-dispatch metadata through the existing hook context and command catalog, keep exporter labels bounded, and require redacted live telemetry proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and the PR head shows the existing construction-time wrapper, the PR’s second inline wrapper, the params-derived command telemetry path, and the prompt-snapshot versus command-catalog mismatch. This is a feature PR rather than a current-main bug report.

## Solution Assessment

No. The exporter shape is plausible, but the emission path should use trusted dispatch/read context and command-visible skill metadata rather than a second wrapper and model/tool-controlled params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Thread skill context through the existing tool wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools, and `wrapToolWithBeforeToolCallHook()` does not no-op for already wrapped tools. Wrapping the inline command tool again can rerun hook policy, loop tracking, and tool diagnostics; pass the skill context into the construction-time hook context instead.
  - confidence: 0.93
- **[P2] Require trusted command-dispatch context for telemetry:** `src/agents/pi-tools.before-tool-call.ts:196-227`
  - body: This treats successful wrapped tool calls with `commandName` and `skillName` params as command-dispatched skill usage. Those fields are ordinary tool params at this generic hook boundary, so normal tool calls can spoof trusted `skill.used`; carry a trusted dispatch marker or metadata from inline command dispatch instead.
  - confidence: 0.91
- **[P2] Use the command catalog for dispatched skill metadata:** `src/agents/pi-tools.before-tool-call.ts:213-227`
  - body: `resolvedSkills` contains prompt-visible skills, while slash command specs are built from eligible user-invocable entries. A prompt-hidden user-invocable skill command can dispatch its tool and still miss `skill.used`; use the command catalog or full runtime skill entry for command telemetry.
  - confidence: 0.9
