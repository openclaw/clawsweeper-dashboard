## Summary

Keep open: the feature is plausible, but the PR still emits skill telemetry from duplicated and tool-param-derived hook context, and the external real behavior proof is only targeted tests.

## What This Changes

The PR adds a `skill.used` diagnostics event for skill file reads and command-dispatched skill tools, exports it through OTEL and Prometheus, and updates docs, changelog, and tests.

## Best Possible Solution

Route `skill.used` emission through trusted runtime read and command-dispatch context on the existing tool hook, use command-visible skill metadata, keep labels bounded, and land after real telemetry proof.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows command telemetry derived from tool params and inline command tools wrapped a second time, while current main shows `createOpenClawTools` already wraps tools by default.

## Solution Assessment

No. The exporter shape is reasonable, but the emission path should use trusted dispatch/read context and the command-visible skill catalog rather than a second wrapper and model/tool-controlled params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools, so wrapping the inline command tool again runs hook policy and tool execution diagnostics twice. Thread the skill telemetry context through the existing creation-time hook context instead.
  - confidence: 0.93
- **[P2] Gate command telemetry on trusted dispatch context:** `src/agents/pi-tools.before-tool-call.ts:218-227`
  - body: This treats any successful wrapped tool call with `commandName` and `skillName` params as command-dispatched skill usage. Ordinary tool calls can supply those fields, so the event should come from trusted inline command dispatch context, not tool params.
  - confidence: 0.9
- **[P2] Use command-visible skills for inline telemetry:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: The inline path looks up the skill in `targetSessionEntry.skillsSnapshot`, whose `resolvedSkills` only contains prompt-visible skills. A hidden but user-invocable skill command can dispatch correctly and still miss `skill.used`; use the command catalog or full runtime skill entry.
  - confidence: 0.9
