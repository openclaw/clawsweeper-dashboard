## Summary

Keep open: the latest head still has blocking correctness issues in the telemetry emission path, and the supplied proof is test-only rather than real exported telemetry from a run.

## What This Changes

The PR adds a `skill.used` diagnostic event for skill file reads and command-dispatched skill tools, exports it through diagnostics-otel and diagnostics-prometheus, and updates diagnostics docs, changelog, and tests.

## Best Possible Solution

Route `skill.used` emission through trusted runtime read and command-dispatch context on the existing hook wrapper, use command-visible skill metadata, keep labels bounded, and require redacted live telemetry proof before merge.

## Reproduction Assessment

Yes for the blocking review findings: source inspection of the latest head shows the nested hook wrapper, params-derived command activation, and prompt-snapshot-only skill lookup. This is a feature PR, so there is no current-main bug reproduction path to run.

## Solution Assessment

No. The exporter shape is reasonable, but the emission source should use trusted dispatch/read context and the command-visible skill catalog instead of adding a second wrapper and deriving command usage from tool params.

  - confidence: 0.9
