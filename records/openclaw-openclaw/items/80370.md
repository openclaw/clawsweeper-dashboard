## Summary

Keep open: this is a useful diagnostics feature, but the latest head is not merge-ready because the telemetry emission path still has source-visible correctness blockers and the PR body only supplies test output as proof.

## What This Changes

The branch adds `skill.used` diagnostics for known `SKILL.md` reads and command-dispatched skill tools, exports them through Prometheus and OTEL, and updates docs, changelog, and targeted tests.

## Best Possible Solution

Thread trusted skill-read and skill-command metadata through the existing hook context and command/full skill catalog, keep labels bounded, and land only after maintainer diagnostics-contract approval plus redacted real telemetry proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the duplicate `skillsSnapshot` property, second inline wrapper, params-derived command matching, and prompt-snapshot lookup. The requested telemetry itself is a feature, not a bug reproduction.

## Solution Assessment

No. The exporter shape is plausible, but the emission path should use trusted dispatch/read context and the command/full skill catalog instead of a second wrapper and model-controlled tool params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Pass the telemetry snapshot without overwriting it:** `src/agents/pi-embedded-runner/run/attempt.ts:1410`
  - body: `skillsSnapshot: skillsSnapshotForToolTelemetry` is overwritten by the later `skillsSnapshot: skillsSnapshotForRun` in the same object, so the fallback snapshot built for telemetry never reaches tool construction. That leaves read-based `skill.used` missing for runs whose original snapshot lacks `resolvedSkills`.
  - confidence: 0.98
- **[P2] Thread inline telemetry through the existing hook:** `src/auto-reply/reply/get-reply-inline-actions.ts:314-323`
  - body: `resolveSkillDispatchTools()` already returns tools from `createOpenClawTools()`, which wraps them with the before-tool-call hook. Wrapping the selected inline command tool again reruns policy and diagnostics for the same command; pass the skill telemetry context into the construction-time hook instead.
  - confidence: 0.93
- **[P2] Require trusted dispatch context for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:200-207`
  - body: This treats any successful wrapped tool call with `commandName` and `skillName` params as real skill usage. Those fields are ordinary tool arguments, so non-command tools can spoof `skill.used`; carry a dispatcher-owned marker or catalog entry instead.
  - confidence: 0.9
- **[P2] Resolve command telemetry from the command catalog:** `src/agents/pi-tools.before-tool-call.ts:227-231`
  - body: Command specs include user-invocable skills, but `resolvedSkills` is prompt-visible only. Hidden user-invocable command skills can dispatch successfully and then be skipped by `findResolvedSkillByName`; use the dispatch catalog or full skill entries.
  - confidence: 0.9
