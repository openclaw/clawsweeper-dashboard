## Summary

Keep open: the feature is not implemented on current main, but this PR still has blocking correctness findings, only test-based proof, and needs maintainer product-scope judgment because related skill-telemetry requests were previously routed toward plugin/ClawHub work.

## What This Changes

The PR adds a `skill.used` diagnostic event for SKILL.md reads and skill command dispatches, exports it through Prometheus and OTEL, and updates related docs, changelog, and tests.

## Best Possible Solution

If maintainers want this in core diagnostics, emit bounded `skill.used` events from trusted read/dispatch context and the command-visible skill catalog; otherwise keep skill usage analytics in a plugin or ClawHub-owned integration.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and PR head shows the double hook wrapper, params-derived command matching, and prompt-snapshot versus command-catalog mismatch. The feature itself is new product surface, so there is no current-main bug reproduction path.

## Solution Assessment

No. The exporter shape is plausible, but the emission path should use trusted skill-read and command-dispatch metadata, or the work should remain plugin/ClawHub scope if maintainers do not want this as a core diagnostics contract.

  - body: `resolvedSkills` only contains prompt-visible skills, while slash command specs are built from eligible user-invocable entries. Hidden user-invocable skill commands can dispatch a tool and still miss telemetry; use the command catalog or full runtime skill entry.
  - confidence: 0.9
