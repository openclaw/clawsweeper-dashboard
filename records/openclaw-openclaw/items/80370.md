## Summary

Keep this PR open: the telemetry feature is plausible, but the current patch still has blocking source-level correctness issues and the external contributor proof is test-only.

## What This Changes

The PR adds a `skill.used` diagnostic event, Prometheus and OTEL exports for skill usage, docs, changelog text, and targeted tests.

## Best Possible Solution

Thread trusted skill-read and command-dispatch metadata through the existing hook context, use command-catalog or full runtime skill metadata for command telemetry, keep labels bounded, and land only after redacted real telemetry proof.

## Reproduction Assessment

Yes for the review findings: source inspection of current main and PR head shows the existing construction-time hook wrapper, the PR's second wrapper, and params-derived command telemetry. I did not run the PR branch in this read-only review.

## Solution Assessment

No. The requested telemetry is useful, but the implementation should avoid double wrapping and should derive command skill usage from trusted dispatch/catalog context rather than generic tool params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools on current main. Wrapping the selected inline command tool again can rerun hook policy, loop tracking, and tool execution diagnostics; thread the skill telemetry context into the creation-time hook context instead.
  - confidence: 0.93
- **[P2] Require trusted dispatch context for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:196-204`
  - body: This treats a successful wrapped tool call with `commandName` and `skillName` params as skill command usage. Those fields are ordinary tool params at this generic hook boundary, so normal tool calls can spoof `skill.used`; carry a trusted dispatch marker or skill metadata from inline command dispatch instead.
  - confidence: 0.9
- **[P2] Use command-visible skill metadata for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:213-227`
  - body: `resolvedSkills` contains prompt-visible skills, but command specs are built from eligible user-invocable entries. A hidden user-invocable skill command can dispatch its tool and still miss `skill.used`; use the command catalog or full runtime skill entry.
  - confidence: 0.9
