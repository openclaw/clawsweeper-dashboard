## Summary

Keep this PR open: the diagnostics/exporter direction is plausible, but the patch misses two production paths it claims to cover, so `skill.used` telemetry would be incomplete. External real behavior proof is also still tests-only.

## What This Changes

The PR adds `skill.used` diagnostics emitted from tool execution and exports them through Prometheus and OpenTelemetry docs, services, and tests.

## Best Possible Solution

Thread the exact per-run skill catalogue into the before-tool-call telemetry for embedded reads and inline command-dispatched tools, keep exporter labels bounded, then add redacted real output showing the new Prometheus or OTEL signal.

## Reproduction Assessment

Yes. Source inspection shows the introduced gaps: inline skill tool dispatch calls `createOpenClawTools` without a skill snapshot while the new matcher requires `ctx.skillsSnapshot`, and copied-workspace sandbox runs build their skill prompt from `effectiveWorkspace` while the new hook receives the original snapshot.

## Solution Assessment

No. The exporter implementation is reasonable, but the emission source needs to be wired to the same production skill catalogue used by the read and command-dispatch paths rather than only the direct test hook context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the sandbox skill catalogue for read telemetry:** `src/agents/pi-embedded-runner/run/attempt.ts:898`
  - body: In copied-workspace sandboxes (`ro`/`none`), this passes the original `params.skillsSnapshot` into the hook even though the run intentionally rebuilds the skill prompt from `effectiveWorkspace`. A model that reads the `SKILL.md` path it was shown in the sandbox will not match the original snapshot path, so the new `skill.used` event is missing for sandboxed runs. Thread a snapshot derived from the same run skill entries used for the prompt.
  - confidence: 0.88
- **[P2] Wire command telemetry into inline skill dispatch:** `src/agents/pi-tools.before-tool-call.ts:261-272`
  - body: The command matcher only emits when `ctx.skillsSnapshot` contains the skill. Production command-dispatched skills are executed through inline actions via `createOpenClawTools(...)`, whose default hook context has no skill snapshot, even though it passes `commandName` and `skillName` in tool args. The new test wraps the hook directly with a fake snapshot, so real `/skill` tool dispatch still exports no `skill.used` metric.
  - confidence: 0.9
