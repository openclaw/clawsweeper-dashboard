## Summary

Keep open: this recent external feature PR is not implemented on main, and the latest head still has source-proven telemetry correctness gaps plus only test-based proof. The earlier ClawSweeper review's sandbox and inline-wiring concerns were addressed by the force-push, but command activation is still derived from tool args and prompt-only snapshots miss command-only skills.

## What This Changes

The PR adds a `skill.used` diagnostic event for known skill reads and command-dispatched skill tools, exports it through diagnostics-otel and diagnostics-prometheus, and updates docs, changelog, and tests.

## Best Possible Solution

Keep the bounded exporters, but derive skill-used events from non-model-controlled skill/catalog context shared by prompt reads and command dispatch, then land with live telemetry proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows command telemetry can be triggered from ordinary tool params, and current main's skill snapshot contract shows command-only skills can be omitted from the snapshot used by inline telemetry.

## Solution Assessment

No. The exporter shape is reasonable, but the emission source should use explicit runtime command-dispatch context or a full command-visible skill catalog instead of trusting tool args and prompt-only snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trust command dispatch context instead of tool args:** `src/agents/pi-tools.before-tool-call.ts:217-226`
  - body: The command detector treats any successful tool call carrying `commandName` and `skillName` as a command-dispatched skill. Those fields are part of the tool params, so a normal model-driven tool call can spoof `skill.used` with `activation="command"`; gate command telemetry on non-model-controlled inline dispatch context instead.
  - confidence: 0.88
- **[P2] Use command-visible skills for inline telemetry:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: Inline command dispatch wraps the tool with `targetSessionEntry.skillsSnapshot`, but that snapshot's `resolvedSkills` is built from prompt-visible entries. Skills with `disable-model-invocation: true` are still valid slash commands, so command-dispatched skill tools for those skills will not emit `skill.used`; pass the command catalog/full runtime skill entry instead.
  - confidence: 0.9
