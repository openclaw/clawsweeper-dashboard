## Summary

Keep open. The feature is useful, but the current PR head still has blocking telemetry emission issues around trusted command context and double wrapping, and the external PR only supplies test-based proof rather than real runtime telemetry proof.

## What This Changes

The PR adds a `skill.used` diagnostic event for skill reads and command-dispatched skill tools, exports it through OTEL and Prometheus, and updates docs, changelog, and tests.

## Best Possible Solution

Route `skill.used` emission through trusted read and command-dispatch runtime context on the existing hook wrapper, use command-visible skill metadata, keep exporter labels bounded, and land only after redacted live telemetry proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection of PR head shows command telemetry derived from tool params and a second inline wrapper, while current main already wraps tools during `createOpenClawTools`.

## Solution Assessment

Is this the best way to solve the issue? No. The exporter shape is plausible, but the emission path should use trusted dispatch/read context and the command-visible skill catalog rather than a second wrapper and tool-controlled params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools, so wrapping the selected inline command tool again reruns hook policy and emits duplicate tool execution diagnostics. Pass the skill telemetry context through the existing construction-time hook context instead.
  - confidence: 0.93
- **[P2] Gate command telemetry on trusted dispatch context:** `src/agents/pi-tools.before-tool-call.ts:218-227`
  - body: This treats any successful wrapped tool call with `commandName` and `skillName` params as command-dispatched skill usage. Ordinary tool calls can supply those fields, so emit `skill.used` from trusted inline command dispatch context instead of tool params.
  - confidence: 0.91
- **[P2] Use command-visible skills for inline telemetry:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: The inline path looks up the skill in `targetSessionEntry.skillsSnapshot`, whose `resolvedSkills` contains prompt-visible skills. A hidden but user-invocable skill command can dispatch correctly and still miss `skill.used`; use the command catalog or full runtime skill entry.
  - confidence: 0.9
