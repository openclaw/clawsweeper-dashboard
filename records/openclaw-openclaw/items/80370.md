## Summary

Keep open. The feature is useful and not present on current main, but the latest PR head still has blocking correctness issues in the trusted emission path, and the external proof is only targeted tests rather than real telemetry output.

## What This Changes

The PR adds a `skill.used` diagnostic event for known `SKILL.md` reads and command-dispatched skill tools, then exports it through OTEL, Prometheus, docs, changelog, and tests.

## Best Possible Solution

Thread trusted skill-read and command-dispatch metadata into the existing tool hook, use command-visible or full skill metadata for command skills, keep exporter labels bounded, and land after redacted real telemetry proof.

## Reproduction Assessment

Yes for the review findings. Source inspection of the latest PR head shows the double wrapper and param-derived command telemetry, and current-main source shows the existing hook wrapper plus the prompt-only `resolvedSkills` snapshot.

## Solution Assessment

No. The exporter shape is reasonable, but the emission path should use trusted dispatch/read context and the command-visible skill catalog rather than a second wrapper and model/tool-controlled params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:324-333`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools on current main, so wrapping the selected inline command tool again reruns hook policy and duplicates tool execution diagnostics. Thread the skill telemetry context into the creation-time hook context instead of wrapping a wrapped tool.
  - confidence: 0.93
- **[P2] Require trusted command-dispatch context for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:196-227`
  - body: This treats a successful wrapped tool call with `commandName` and `skillName` params as skill command usage. Those fields are ordinary tool params at a general hook boundary, so a normal tool call can spoof `skill.used`; carry a trusted dispatch marker or skill metadata from inline command dispatch instead.
  - confidence: 0.9
- **[P2] Use command-visible skill metadata for command telemetry:** `src/agents/pi-tools.before-tool-call.ts:213-227`
  - body: `resolvedSkills` contains only prompt-visible skills, but command specs are built from eligible user-invocable entries. A hidden user-invocable skill command can dispatch its tool and still miss `skill.used`; use the command catalog or full runtime skill entry.
  - confidence: 0.9
