## Summary

Keep this PR open: the feature is useful, but the current patch still has blocking telemetry correctness issues and the external-contributor real behavior proof is only targeted tests, not live telemetry output.

## What This Changes

The PR adds a trusted `skill.used` diagnostic event for skill file reads and command-dispatched skill tools, exports it through OTEL and Prometheus, and updates diagnostics docs, changelog, and tests.

## Best Possible Solution

Route skill-used emission through explicit runtime read and command-dispatch context, keep labels bounded, use the command-visible skill catalog, and land only after redacted live telemetry proof is supplied.

## Reproduction Assessment

Yes. Source inspection of the PR head shows command telemetry is inferred from tool params, inline command tools are wrapped a second time, and prompt-only snapshots exclude command-only hidden skills.

## Solution Assessment

No. The exporter shape is reasonable, but the emission source should use explicit non-model-controlled runtime context and the command-visible skill catalog rather than nested wrappers and tool argument fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools, so wrapping the selected inline tool again makes command-dispatched skill tools run hook policy and emit tool execution diagnostics twice. Pass the skill telemetry context into the existing wrapper instead of nesting another wrapper.
  - confidence: 0.93
- **[P2] Trust command dispatch context instead of tool args:** `src/agents/pi-tools.before-tool-call.ts:217-226`
  - body: This treats any successful tool call carrying `commandName` and `skillName` params as command-dispatched skill usage. Those fields are model/tool-controlled on ordinary calls, so command telemetry should be gated on explicit inline dispatch context.
  - confidence: 0.9
- **[P2] Use command-visible skills for inline telemetry:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: The inline wrapper receives `targetSessionEntry.skillsSnapshot`, but that snapshot is built from prompt-visible skills. Skills with `disable-model-invocation: true` can still run as slash commands, so command telemetry needs the command catalog or full runtime skill entry.
  - confidence: 0.9
