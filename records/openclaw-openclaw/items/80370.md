## Summary

Keep open: the feature is useful, but the latest PR head still has blocking correctness issues around hook wrapping, trusted command attribution, and command-visible skill lookup, and the external PR only supplies targeted test proof rather than real telemetry proof.

## What This Changes

The branch adds a `skill.used` diagnostic event for known `SKILL.md` reads and command-dispatched skill tools, exports it via OTEL and Prometheus, and updates docs, changelog, and tests.

## Best Possible Solution

Route `skill.used` emission through explicit runtime read and command-dispatch context on the existing tool hook, include command-visible skill metadata, keep labels bounded, and land after real telemetry proof.

## Reproduction Assessment

Yes. Source inspection of the latest PR head and current main shows the nested hook wrapper, param-derived command telemetry, and prompt-visible-only snapshot lookup without needing a live run.

## Solution Assessment

No. The exporter shape is reasonable, but the emission source should use trusted runtime dispatch/read context and the command-visible skill catalog rather than a second wrapper, tool-controlled params, and prompt-only snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass skill context through the existing hook wrapper:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: `createOpenClawTools()` already returns before-tool-call wrapped tools, so wrapping the selected inline command tool again runs hook policy and tool execution diagnostics twice for command-dispatched skills. Pass the skill telemetry context through the existing creation-time hook context instead.
  - confidence: 0.93
- **[P2] Gate command telemetry on trusted dispatch context:** `src/agents/pi-tools.before-tool-call.ts:218-227`
  - body: This treats any successful wrapped tool call with `commandName` and `skillName` params as command-dispatched skill usage. Ordinary model/tool-controlled calls can set those fields, so the event should be driven by trusted inline command dispatch context.
  - confidence: 0.9
- **[P2] Use command-visible skills for inline telemetry:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-332`
  - body: The inline wrapper looks up the skill in `targetSessionEntry.skillsSnapshot.resolvedSkills`, but snapshots only retain prompt-visible skills. Skills hidden from model invocation can still expose slash commands, so command telemetry needs the command catalog or a full runtime skill entry.
  - confidence: 0.9
