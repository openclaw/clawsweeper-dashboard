## Summary

Keep open. Current main still has the reported mapper bug: session-backed memory hits can retain `source: "sessions"`, but the final built-in `memory_search` result shaping still hard-codes `corpus: "memory"`; because open PR #72886 uses closing syntax for this issue and PR #71898 overlaps, the issue should stay open until one focused fix lands or the PRs are resolved.

## What This Changes

The issue requests that built-in `memory_search` preserve session transcript provenance by returning `corpus: "sessions"` when the underlying hit has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable solution because backend results already carry source provenance and the tool path already accepts `corpus=sessions`; the stricter type-tagging idea from the discussion is useful future design context, not required for this bug.
