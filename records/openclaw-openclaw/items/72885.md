## Summary

Keep open. Current main still flattens authorized session-source built-in memory hits to `corpus: "memory"` in the final `memory_search` response, while an open closing PR and an overlapping open PR already track the narrow fix.

## What This Changes

The issue requests that `memory_search` surface session-transcript hits with `corpus: "sessions"` when the underlying result has `source: "sessions"`.

## Best Possible Solution

The end state should be a narrow memory-core fix that derives surfaced built-in result `corpus` from `MemorySearchResult.source`, widens the surfaced type, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives session visibility filtering; current main then maps it to `corpus: "memory"` in `extensions/memory-core/src/tools.ts`.

## Solution Assessment

Yes. The narrow proposed direction, deriving the surfaced corpus from the existing result `source`, is the smallest maintainable fix; the stricter type-tagging approach raised in discussion is useful context but broader than this bug requires.
