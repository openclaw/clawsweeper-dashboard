## Summary

Keep open. Current main still has the reported source-level bug: session-backed memory hits can survive filtering with `source: "sessions"`, but the final `memory_search` response mapper still hard-codes built-in hits to `corpus: "memory"`; the linked open PRs should be reviewed or consolidated before this issue closes.

## What This Changes

The issue asks `memory_search` to surface session transcript hits with `corpus: "sessions"` instead of `corpus: "memory"` when the underlying result source is sessions.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The narrow fix is to derive the surfaced corpus from the existing `result.source` and add a regression test; the broader strict type-tagging idea from the discussion is useful design context but not required for this bug.
