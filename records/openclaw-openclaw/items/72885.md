## Summary

Keep open. Current main still flattens built-in `memory_search` results to `corpus: "memory"` after session visibility filtering, while the backend result and tool contract already preserve `source: "sessions"`; open PR #72886 closes this issue and PR #71898 overlaps, so the issue should stay open until one fix lands or the PRs are resolved.

## What This Changes

The issue asks `memory_search` to surface session transcript hits with `corpus: "sessions"` instead of flattening them to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type, and adds regression coverage without changing ranking, filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A source-level reproduction is any allowed `MemorySearchResult` with `source: "sessions"` that reaches `surfacedMemoryResults`; current main then returns it with `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the output corpus from the existing `result.source` is the narrowest maintainable fix; the stricter source-tagging idea from the discussion is useful future design context, not required for this bug.
