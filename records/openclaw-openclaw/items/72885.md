## Summary

Keep open. Current main still preserves `source: "sessions"` internally but flattens surfaced built-in `memory_search` hits to `corpus: "memory"`, and the issue is paired with open fix PRs that should be reviewed or consolidated before this tracker closes.

## What This Changes

Correct `memory_search` so built-in session transcript hits surface with `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the backend hit `source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the KinthAI index-time tagging comment is useful context but larger than this bug requires.
