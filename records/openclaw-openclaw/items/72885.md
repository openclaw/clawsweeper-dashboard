## Summary

Keep open. Current main still flattens surfaced built-in `memory_search` hits to `corpus: "memory"` after preserving `source: "sessions"` internally, and the issue is already paired with open PR #72886 plus overlapping PR #71898, so maintainers should review and land one focused fix rather than close this before merge.

## What This Changes

The issue requests that `memory_search` surface session-transcript hits with `corpus: "sessions"` instead of labeling every built-in memory result as `corpus: "memory"`.

## Best Possible Solution

Land one narrow memory-core change that derives surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`; current main’s mapper will return `corpus: "memory"` for that hit.

## Solution Assessment

Yes. The proposed source-based mapping is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the stricter index-time tagging model discussed in the comments is useful context but broader than needed here.
