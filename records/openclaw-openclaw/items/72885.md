## Summary

Keep open. Current main still has the reported source-level bug: session-backed memory hits can survive visibility filtering with `source: "sessions"`, but the final `memory_search` response mapper still hard-codes every built-in hit to `corpus: "memory"`; open PRs should be reviewed or consolidated before this issue closes.

## What This Changes

The issue asks for built-in `memory_search` session transcript hits to surface as `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable path because the backend already records source provenance and the tool schema already accepts `corpus: "sessions"`; the broader strict type-tagging idea from the discussion is useful future design context, not required for this bug.
