## Summary

Keep open. Current main still preserves session-source hits through the memory search pipeline but flattens every surfaced built-in hit to `corpus: "memory"`, and an open closing PR still tracks the fix.

## What This Changes

The issue asks `memory_search` to surface transcript-backed hits with `corpus: "sessions"` when the underlying result has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable path because the backend already records source provenance and the tool schema already accepts `corpus: "sessions"`; the stricter type-tagging idea from the discussion is useful design context but not required for this bug.
