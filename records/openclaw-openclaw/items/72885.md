## Summary

Keep open. Current main still preserves session provenance internally but hard-codes surfaced built-in `memory_search` results to `corpus: "memory"`, and the latest release has the same mapper. The issue is also paired with open closing PR #72886 and overlapping PR #71898, so maintainers should review or consolidate those PRs before closing it.

## What This Changes

The issue requests that transcript-backed `memory_search` hits surface `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core change that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`; current main’s mapper returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes. Deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the stricter index-time tagging model discussed in comments is useful context but broader than needed here.
