## Summary

Keep open. Current main still preserves session provenance internally but flattens every surfaced built-in memory_search result to corpus="memory", and the issue is already paired with open fix PRs that should be reviewed or consolidated before this tracker closes.

## What This Changes

memory_search should surface built-in session-transcript hits with corpus="sessions" instead of labeling them as corpus="memory".

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result corpus from result.source, widens the surfaced result type to include "sessions", and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock memory.manager.search so it returns a MemorySearchResult with source="sessions", execute memory_search, and inspect details.results; current main’s mapper still returns corpus="memory" for that surfaced hit.

## Solution Assessment

Yes. The proposed source-derived corpus mapping is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts corpus="sessions"; the stricter index-time tagging model discussed in comments is useful context but broader than needed for this bug.
