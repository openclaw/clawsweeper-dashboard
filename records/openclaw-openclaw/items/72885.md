## Summary

Keep open. Current main still has the reported source-level bug: session-backed hits can survive the memory visibility path with `source: "sessions"`, but the final `memory_search` result mapper still hard-codes built-in hits to `corpus: "memory"`; the open linked fix PRs should be reviewed or consolidated before this issue closes.

## What This Changes

The issue asks for built-in `memory_search` session transcript hits to surface `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The narrowest maintainable fix is to derive the surfaced corpus from the existing `result.source` and add a focused tool-layer regression test; the stricter type-tagging idea from the discussion is useful context but not required for this bug.
