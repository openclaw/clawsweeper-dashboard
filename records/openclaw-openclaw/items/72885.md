## Summary

Keep open. Current main still preserves session provenance internally but hard-codes surfaced built-in `memory_search` hits to `corpus: "memory"`, and the latest release has the same mapper. The issue is also paired with open closing PR #72886 and overlapping PR #71898, so it should remain open until maintainers land or reject a fix.

## What This Changes

Request to preserve session transcript provenance by returning `corpus: "sessions"` for `memory_search` hits whose backend `source` is `"sessions"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`; current main’s mapper still returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes. Deriving the surfaced corpus from existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the stricter index-time tagging model discussed in comments is useful context but broader than needed for this bug.
