## Summary

Keep open. Current main still preserves session provenance internally but hard-codes surfaced built-in memory_search hits to corpus: "memory", and the latest release still has the same mapper. The issue also has an open closing PR (#72886) plus an overlapping focused PR (#71898), so it should remain open until maintainers choose and land a fix.

## What This Changes

The issue requests that memory_search surface session transcript hits with corpus: "sessions" instead of flattening all built-in memory hits to corpus: "memory".

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from result.source, widens the surfaced result type to include "sessions", and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock memory.manager.search so it returns a MemorySearchResult with source: "sessions", execute memory_search, and inspect details.results; current main’s mapper will still return corpus: "memory" for that surfaced hit.

## Solution Assessment

Yes. Deriving the surfaced corpus from the existing result.source is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts corpus: "sessions"; the KinthAI comment’s stricter index-time tagging model is useful context but broader than needed here.
