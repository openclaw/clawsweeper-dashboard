## Summary

Keep open. Current main still has the reported source-level bug: session-source memory search hits survive filtering with `source: "sessions"`, but the final `memory_search` response mapper hard-codes all built-in hits to `corpus: "memory"`; open PR #72886 closes this issue and open PR #71898 overlaps, so the issue should remain open until one focused fix lands.

## What This Changes

Fix `memory_search` so built-in session transcript hits surface with `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The narrow fix is to derive the surfaced corpus from the existing `result.source` and add a regression test; the stricter provenance-tagging idea from the discussion is useful design context but not required for this bug.
