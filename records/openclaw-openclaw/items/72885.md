## Summary

Keep open. Current main still preserves session provenance internally but flattens every surfaced built-in `memory_search` hit to `corpus: "memory"`, and this valid issue is already paired with open fix PRs that should be reviewed or consolidated before the tracker closes.

## What This Changes

The issue requests that built-in `memory_search` results surface session-transcript hits as `corpus: "sessions"` when the underlying `MemorySearchResult.source` is `"sessions"`, with regression coverage.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix; the stricter index-time tagging model discussed in comments is useful context but broader than needed for this bug.
