## Summary

Keep open. Current main still preserves session provenance internally but flattens every surfaced built-in `memory_search` hit to `corpus: "memory"`, and this issue is already paired with open candidate fix PRs that should be reviewed or consolidated before the tracker closes.

## What This Changes

Requests that built-in `memory_search` results label session-transcript hits with `corpus: "sessions"` instead of `corpus: "memory"` when the underlying hit has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The proposed source-derived corpus mapping is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the stricter index-time tagging model discussed in comments is useful context but broader than needed for this bug.
