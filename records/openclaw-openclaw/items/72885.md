## Summary

Keep open. Current main still flattens built-in session transcript hits to `corpus: "memory"` at the final `memory_search` result-shaping step, and the issue is paired with open fix PR #72886 plus overlapping PR #71898.

## What This Changes

The issue asks for `memory_search` to surface built-in session transcript hits with `corpus: "sessions"` instead of flattening them to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the backend hit `source` is the narrowest maintainable fix; the stricter index-time tagging model discussed in the KinthAI comment is useful broader context but not required for this bug.
