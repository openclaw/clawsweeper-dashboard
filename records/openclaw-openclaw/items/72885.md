## Summary

Keep open. Current main still flattens session-source `memory_search` hits to `corpus: "memory"` after visibility filtering, and the issue is already paired with open closing PR #72886 plus overlapping PR #71898, so it should remain open until a focused fix merges.

## What This Changes

Report asks `memory_search` to surface session-transcript hits with `corpus: "sessions"` instead of `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage without changing ranking, visibility filtering, citations, supplements, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. The narrow maintainable fix is to derive the surfaced corpus from the existing `result.source` and add a tool-level regression test; the stricter type-tagging idea in the discussion is useful design context but not required for this bug.
