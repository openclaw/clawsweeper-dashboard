## Summary

Keep open. Current main still preserves session provenance internally but flattens surfaced built-in `memory_search` hits to `corpus: "memory"`, and the issue is paired with open fix PRs that should be reviewed or consolidated before this tracker closes.

## What This Changes

The issue requests that `memory_search` surface session transcript hits with `corpus: "sessions"` instead of flattening them to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the backend hit `source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the stricter index-time tagging model discussed in comments is useful broader context but not required for this bug.
