## Summary

Keep open. Current main and the latest release still flatten surfaced built-in `memory_search` hits to `corpus: "memory"` even when the backend hit carries `source: "sessions"`, and open fix PR #72886 plus overlapping PR #71898 should be reviewed or consolidated before this tracker closes.

## What This Changes

The issue requests that session transcript hits returned by `memory_search` surface as `corpus: "sessions"` instead of `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the backend hit `source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the broader strict type-tag approach discussed in comments is useful context but not required for this bug.
