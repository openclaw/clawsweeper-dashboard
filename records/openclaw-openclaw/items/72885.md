## Summary

Keep open. Current main still flattens surfaced built-in `memory_search` hits to `corpus: "memory"` even after preserving `source: "sessions"`, and open PR #72886 plus overlapping PR #71898 should be reviewed or consolidated before this tracker closes.

## What This Changes

The issue asks `memory_search` to surface session-transcript hits as `corpus: "sessions"` when the backend hit has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core fix that derives surfaced built-in result `corpus` from `result.source`, widens the surfaced result type to `"memory" | "sessions"`, and adds regression coverage without changing ranking, visibility filtering, supplements, citations, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any authorized `MemorySearchResult` with `source: "sessions"` that survives `filterMemorySearchHitsBySessionVisibility`; current main then maps it through `surfacedMemoryResults` and returns `corpus: "memory"`.

## Solution Assessment

Yes. Deriving the surfaced corpus from the backend hit `source` is the narrowest maintainable fix because the backend already records provenance and the tool schema already accepts `corpus: "sessions"`; the KinthAI comment’s stricter index-time tagging model is useful context but not required for this bug.
