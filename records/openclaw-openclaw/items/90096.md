## Label Justifications

- `P2`: The issue affects document understanding for PDF-bearing channel and Control UI turns, but it is bounded to media prompt construction rather than a core runtime outage.
- `impact:message-loss`: Inbound PDF attachment content can be reduced to a media marker for text-only locked-down agents, so the meaningful document text is not delivered to the model.
- `impact:security`: The requested fix explicitly depends on preserving managed-media-only resolution and untrusted document boundaries instead of granting broad file or exec access.

## Summary

Keep open: current main still routes managed inbound PDFs as media markers through the auto-reply and Control UI embedded-runner paths, and the existing PDF extractor is not wired into those prompt paths.

## What This Changes

Request automatic bounded, untrusted text extraction for managed inbound PDF attachments in auto-reply and embedded-runner prompts.

## Best Possible Solution

Add a shared managed-inbound-PDF context builder that resolves only media-store inbound refs, extracts bounded text with the existing document-extract path, wraps it as untrusted content, and feeds both auto-reply and embedded-runner prompts without broad read or exec tools.

## Reproduction Assessment

Yes by source inspection, not by a live Signal/Control UI run: current main routes PDF MediaPaths through marker-only prompt construction, so a focused test can reproduce the missing extracted text in both paths.

## Solution Assessment

Unclear until maintainer policy is confirmed; the requested shared bounded extraction path looks like the right owner boundary, but automatic untrusted PDF injection needs an explicit safety/failure contract.
