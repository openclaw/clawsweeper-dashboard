## Label Justifications

- `P2`: The issue affects PDF understanding for locked-down agents but is bounded to media prompt construction and gateway media handling rather than a core outage.
- `impact:security`: The requested fix must preserve managed-inbound-media resolution and untrusted-document boundaries without granting broad file or exec access.

## Summary

Keep open: current main still does not satisfy the large managed-PDF case, and the canonical implementation is the still-open linked PR at https://github.com/openclaw/openclaw/pull/92220.

## What This Changes

The issue asks OpenClaw to inject bounded, untrusted text extracted from managed inbound PDF media into auto-reply and embedded-runner prompts without broad file or exec access.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92220 if maintainers accept the extraction envelope and managed-media fallback contract, then close this issue after that merge.

## Reproduction Assessment

Yes, at source level: current main still has 5 MB / 4-page extraction defaults and rejects a 6 MB sandboxed PDF offload before dispatch. I did not run a live Signal or Control UI repro in this read-only review.

## Solution Assessment

Yes, the linked implementation direction appears best because it reuses the existing media-understanding extraction and injection path instead of adding a parallel prompt-context module. The remaining decision is whether maintainers accept the larger default extraction envelope and managed-media fallback behavior.
