## Summary

Current main and released builds already address the reported reboot loop: stale managed Chromium locks are recovered, failed launch/readiness paths clean up the launched process, and repeated managed launch failures are per-profile circuit-broken instead of spawning Chrome indefinitely.

## What This Changes

The issue requests stale Chromium Singleton* lock recovery, bounded retry/backoff for CDP startup failures, failed-spawn cleanup, and browser doctor/reset recovery guidance.

## Best Possible Solution

Keep the shipped browser-plugin recovery and cooldown behavior; users still on openclaw 2026.4.10 should upgrade, and any stricter PID-name or process-tree hardening should be tracked only with a fresh current-main reproduction.

## Reproduction Assessment

Yes for the historical failure class: regression tests model stale SingletonLock and repeated CDP readiness failure paths. No unresolved current-main failure was reproduced because the reviewed main and v2026.4.26 release code now recover or cooldown instead of retrying indefinitely.

## Solution Assessment

Yes. The shipped extension-owned fix is the narrow maintainable solution: recover dead/foreign managed-profile locks, clean up failed launches, and bound repeated per-profile launches without adding core policy or a broad startup migration.
