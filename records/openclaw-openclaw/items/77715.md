## Summary

Codex review can close this in OpenClaw because the reported failure is in the external Honcho plugin’s session-id mapping, while current OpenClaw only supplies canonical cron/subagent session keys through documented plugin hooks and does not vendor @honcho-ai/openclaw-honcho.

## What This Changes

The issue reports Honcho memory failures when external plugin code turns long OpenClaw cron/subagent session keys into Honcho session IDs over the service's 100-character limit.

## Best Possible Solution

Keep OpenClaw's canonical session keys unchanged and land/release a deterministic truncate-or-hash mapping in @honcho-ai/openclaw-honcho, with users updating that plugin.

## Reproduction Assessment

Yes from source, but not by running a live Honcho setup: current OpenClaw builds isolated cron run keys as agent:...:cron:...:run:..., and upstream openclaw-honcho appends the provider without bounding the result against Honcho's 100-character schema.

## Solution Assessment

No for this repository. The narrow maintainable fix is in @honcho-ai/openclaw-honcho's buildSessionKey mapping; changing OpenClaw's canonical session-key shape would risk core routing and lifecycle behavior.
