## Summary

Keep open. The linked WhatsApp GIF metadata problem is still present on current main and latest release, but this PR is not merge-ready because it puts provider-sourced `externalAdReply` fields into the agent-facing message body instead of the existing fenced untrusted metadata path, and the supplied proof is unit/build output only.

## What This Changes

This PR changes WhatsApp inbound media placeholders to emit `<media:gif>` for `videoMessage.gifPlayback` and append `externalAdReply` title/source/body metadata as placeholder attributes for image and video messages.

## Best Possible Solution

Classify GIF media in the WhatsApp plugin, keep the body placeholder compact, and attach `externalAdReply` metadata through `payload.untrustedStructuredContext` with extractor, dispatch, prompt-rendering, and real WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main maps every WhatsApp `videoMessage` to `<media:video>` and never reads `gifPlayback` or `externalAdReply`, while Baileys v7.0.0-rc13 exposes those fields. No live WhatsApp GIF-picker payload was exercised in this read-only review.

## Solution Assessment

No for the metadata portion. GIF classification belongs in the WhatsApp inbound path, but external ad metadata should flow through the existing `untrustedStructuredContext` prompt path rather than inline pseudo-XML attributes.
