## Summary

Keep open. The GIF classification part is useful, but the PR still serializes provider-sourced `externalAdReply` title/body/source data into the agent-facing media placeholder body instead of the existing fenced untrusted structured-context path, and the supplied proof is unit/build output only.

## What This Changes

The PR changes WhatsApp inbound media placeholder extraction to emit `<media:gif>` for `videoMessage.gifPlayback` and append `externalAdReply` title/source/body metadata as image/video placeholder attributes.

## Best Possible Solution

Classify GIF media in the WhatsApp plugin, keep the body placeholder compact, and attach `externalAdReply` metadata through `payload.untrustedStructuredContext` with extractor, dispatch, prompt-rendering, and real WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main maps every WhatsApp `videoMessage` to `<media:video>` and never reads `gifPlayback` or `externalAdReply`. No live WhatsApp GIF-picker payload was exercised in this read-only review.

## Solution Assessment

No for the metadata portion. GIF classification belongs in the WhatsApp inbound path, but external ad metadata should flow through the existing `untrustedStructuredContext` prompt path rather than inline pseudo-XML attributes.
