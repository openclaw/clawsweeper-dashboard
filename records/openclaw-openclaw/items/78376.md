## Summary

Keep this open: source inspection supports the reported runtime-isolation failure, but the maintainable fix boundary is unclear between OpenClaw loader compatibility and the external Weixin plugin using the existing channel runtime seam.

## What This Changes

The issue reports that @tencent-weixin/openclaw-weixin v2.4.1 times out on OpenClaw 2026.5.4 because runtime setup and monitor code observe different module instances, with a separate Content-Length rejection.

## Best Possible Solution

Settle the ownership boundary, then either harden the core loader contract for ESM external plugin dynamic imports or coordinate an external Weixin plugin release that uses ctx.channelRuntime and omits manual Content-Length headers.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection of @tencent-weixin/openclaw-weixin@2.4.1 supports the reported timeout path because the entrypoint sets a module-global runtime while the monitor waits on that global instead of using the gateway-provided channelRuntime.

## Solution Assessment

Unclear: the narrowest fix may be in the external plugin, but the report also identifies a plausible OpenClaw loader compatibility edge for ESM packages loaded through jiti and later native dynamic imports.
