## Summary

Keep open: current source plus the external plugin tarball supports the runtime-isolation report, and current main does not clearly resolve the module-identity path; the right fix boundary needs maintainer judgment between OpenClaw loader compatibility and an external Weixin plugin release.

## What This Changes

The issue reports that @tencent-weixin/openclaw-weixin v2.4.1 times out on OpenClaw 2026.5.4 because runtime state is split across module instances, with a separate manual Content-Length failure path under undici.

## Best Possible Solution

Settle the owner boundary, then either harden or diagnose OpenClaw’s dual entry/module-loading behavior, or coordinate an external Weixin plugin release that uses ctx.channelRuntime and omits manual Content-Length headers.

## Reproduction Assessment

Yes, source-reproducible: the v2.4.1 package shows register setting a module-global runtime while the monitor waits on a runtime.js global and startAccount does not pass ctx.channelRuntime. No live QR scan or running monitor reproduction was established.

## Solution Assessment

Unclear: the narrowest runtime fix may be external plugin code using ctx.channelRuntime, but the dual source/runtime entry behavior also looks like a host loader compatibility or diagnostic gap that maintainers should decide deliberately.
