## Summary

Keep open. Current main still has the TaskFlow SQLite migration gap reported here: upgraded `flow_runs` tables can retain a NOT NULL `owner_session_key` column while new managed-flow writes only provide `owner_key`, and two open PRs already target this exact failure.

## What This Changes

Webhook `create_flow` should create a managed TaskFlow owned by the route-bound `sessionKey` on upgraded installs without failing on a legacy `flow_runs.owner_session_key` constraint.

## Best Possible Solution

Ship one TaskFlow-owned SQLite migration that preserves existing rows, maps legacy owner data into `owner_key`, removes the stale `owner_session_key` constraint, recreates indexes, and adds regression coverage for upgraded databases and webhook-driven managed flow creation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a legacy `flow_runs` table with `owner_session_key TEXT NOT NULL`: current `ensureSchema()` leaves that constraint in place, then managed-flow persistence inserts only `owner_key`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the TaskFlow SQLite store migration layer; changing the webhook body contract would contradict the documented route-bound session model and would not repair upgraded databases.
