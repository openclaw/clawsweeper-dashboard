## Summary

Keep open. Current main still has the TaskFlow SQLite migration gap: upgraded `flow_runs` tables can retain a `NOT NULL owner_session_key` column while new managed-flow writes only provide `owner_key`, and two open PRs already target this exact failure.

## What This Changes

The issue asks for webhook `create_flow` to create a managed TaskFlow owned by the route-bound `sessionKey` instead of failing on a legacy SQLite `owner_session_key` constraint.

## Best Possible Solution

Land one TaskFlow SQLite migration that preserves existing rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates indexes, and keeps regression coverage for upgraded legacy and hybrid databases.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a legacy `flow_runs` table with `owner_session_key TEXT NOT NULL`: current `ensureSchema()` leaves that constraint in place, then managed-flow persistence inserts only `owner_key`.

## Solution Assessment

Yes. The narrow maintainable fix is in the TaskFlow SQLite migration layer; changing the webhook request contract would contradict the bound-session docs and would not repair upgraded databases.
