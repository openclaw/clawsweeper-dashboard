## Summary

Keep open. Current `main` still leaves legacy `flow_runs` tables with a blocking `owner_session_key` NOT NULL column while new managed-flow inserts write only `owner_key`, so the reported webhook `create_flow` failure remains plausible and specifically targeted by open fix PRs.

## What This Changes

Webhook `create_flow` should create a managed TaskFlow owned by the route-bound `sessionKey` on upgraded installs without failing on a legacy `flow_runs.owner_session_key` constraint.

## Best Possible Solution

Normalize legacy TaskFlow SQLite databases in the store layer by rebuilding or otherwise replacing old `flow_runs` schemas, preserving existing rows, mapping legacy owner data into `owner_key`, removing the blocking legacy column shape, and adding regression coverage for webhook-driven managed flow creation on upgraded databases.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a legacy `flow_runs` table with `owner_session_key TEXT NOT NULL`: current `ensureSchema()` leaves that constraint in place, then managed-flow persistence inserts only `owner_key`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the TaskFlow SQLite migration/store layer; changing the webhook request contract to require owner fields would contradict the documented route-bound session model and would not repair upgraded databases.
