## Summary

Keep open. Current main and the latest release still have the TaskFlow SQLite migration gap: upgraded `flow_runs` tables can retain `owner_session_key TEXT NOT NULL`, while new managed-flow writes only provide `owner_key`; two open PRs already target this exact failure.

## What This Changes

The issue requests a fix for webhooks `create_flow` failing on upgraded SQLite stores that still enforce the legacy `flow_runs.owner_session_key` column.

## Best Possible Solution

Land one TaskFlow SQLite migration that preserves existing rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates indexes, and keeps regression coverage for legacy and hybrid upgraded databases.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a legacy or hybrid `flow_runs` table with `owner_session_key TEXT NOT NULL`: current `ensureSchema()` leaves that constraint in place, then managed-flow persistence inserts only `owner_key`.

## Solution Assessment

Yes. The narrow maintainable fix is in the TaskFlow SQLite migration layer; changing the webhook request contract would contradict the bound-session docs and would not repair upgraded databases.
