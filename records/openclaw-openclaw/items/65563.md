## Label Justifications

- `P1`: The issue describes an upgraded-install runtime failure that breaks webhook-triggered managed TaskFlow creation for real users.
- `impact:session-state`: The failure is in persisted TaskFlow ownership/session state migration and blocks creation of owner-bound flow records.

## Summary

Keep open. Current main and the latest release still have the TaskFlow SQLite migration gap: upgraded `flow_runs` tables can retain `owner_session_key TEXT NOT NULL`, while managed-flow writes only provide `owner_key`; the next step is to consolidate and land one of the existing migration PRs rather than close this issue.

## What This Changes

The issue reports that the webhooks plugin `create_flow` action fails on upgraded SQLite stores because `flow_runs.owner_session_key` remains `NOT NULL` while current writes populate `owner_key`.

## Best Possible Solution

Land one canonical TaskFlow SQLite migration that preserves existing rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates indexes, and keeps regression coverage for legacy and hybrid upgraded databases.

## Reproduction Assessment

Yes. A legacy or hybrid `flow_runs` table with `owner_session_key TEXT NOT NULL` is source-reproducible on current main because `ensureSchema()` leaves that constraint in place while managed-flow persistence inserts only `owner_key`.

## Solution Assessment

Yes. The narrow maintainable fix is in the TaskFlow SQLite migration layer; changing the webhook request contract would contradict the bound-session docs and would not repair upgraded databases.
