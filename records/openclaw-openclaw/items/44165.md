## Summary

Keep open. Current main still treats snapshot-provided channel metadata/order as the complete Channels page list, so a custom-only gateway snapshot can still hide built-in channel cards. The PR is useful source work, but it should be refreshed before merge because it appends a hard-coded legacy eight-channel list instead of the current catalog-backed bundled channel order.

## What This Changes

The PR exports a built-in channel order, updates `resolveChannelOrder()` to append that order after snapshot-provided channel IDs with de-duplication, and adds focused resolver unit tests.

## Best Possible Solution

Refresh or replace the PR with a frontend-only resolver change that merges snapshot-provided custom/loaded IDs with the current catalog-backed bundled channel order, preserves snapshot IDs first, de-dupes, keeps unknown/custom cards on the generic renderer path, and adds regression coverage for custom-only `channelMeta`, mixed `channelOrder`, and no-snapshot fallback.

## Reproduction Assessment

Yes. Build a `ChannelsStatusSnapshot` whose `channelMeta` contains only a custom ID such as `icenter`; current main's resolver returns only that ID, and `renderChannels()` renders exactly the resolver output.

## Solution Assessment

No as submitted. The frontend-only merge boundary is the right direction, but the PR's hard-coded legacy list is not the narrowest maintainable solution now that current main has catalog-backed bundled channel ID metadata.
