## Summary

Keep open: current main still has the reported Control UI channel-card gap, and this PR is the active fix path, but the branch is not merge-ready because it hard-codes a legacy built-in list and still lacks contributor real behavior proof.

## What This Changes

The PR exports a built-in channel order list, appends it to snapshot-provided channel IDs in `resolveChannelOrder()`, and adds focused resolver unit tests.

## Best Possible Solution

Land a corrected UI-only fix that preserves snapshot/custom IDs first, appends de-duplicated built-in IDs from the catalog-backed channel order, includes focused coverage, and includes real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a custom-only `channelMeta` or `channelOrder` snapshot makes `resolveChannelOrder()` return only that custom ID, and `renderChannels()` renders only those returned IDs.

## Solution Assessment

No as submitted. Appending missing built-ins at the Channels UI boundary is the right direction, but the appended IDs should come from the current catalog-backed channel order instead of a frozen legacy UI list.
