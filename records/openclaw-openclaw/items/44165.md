## Summary

Keep open: current main still has a source-reproducible Channels UI gap, and this PR is a targeted fix path, but it only has test/CI evidence and still needs after-fix real behavior proof before merge.

## What This Changes

The PR exports and changes the Channels page order resolver to append de-duplicated built-in channel IDs after snapshot-provided channel IDs, and adds focused unit coverage for custom-only, mixed, and null snapshots.

## Best Possible Solution

Land this UI-only behavior after the contributor adds redacted real setup proof showing a custom-only channel snapshot still renders built-in channel cards, ideally keeping the appended built-in order aligned with the channel catalog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a custom-only `channelMeta` or `channelOrder` snapshot makes current main return only that custom ID, and `renderChannels()` renders only the returned IDs.

## Solution Assessment

Yes, on the proposed UI-only boundary. Merging snapshot-provided IDs with de-duplicated built-in IDs preserves custom ordering and avoids changing backend plugin loading, though a catalog-derived built-in order would be slightly more maintainable.
