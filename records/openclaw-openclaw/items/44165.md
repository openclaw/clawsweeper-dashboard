## Summary

Keep open. The PR targets a real Control UI bug and the frontend merge boundary is appropriate, but the submitted fix hard-codes the old built-in channel list instead of using current main's catalog-backed bundled channel order, so it would still omit newer bundled channels in custom-only snapshots.

## What This Changes

The branch exports a built-in channel order constant, changes `resolveChannelOrder()` to append that order after snapshot-provided channel IDs with de-duplication, and adds resolver unit tests.

## Best Possible Solution

Refresh or replace the PR with a frontend-only resolver change that merges snapshot-provided custom/loaded IDs with the current catalog-backed bundled channel order, preserves snapshot IDs first, de-dupes, keeps unknown/custom cards on the generic renderer path, and adds regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. A `ChannelsStatusSnapshot` with `channelMeta` containing only a custom ID such as `icenter` exercises the bug: current main returns only that ID from `resolveChannelOrder()`, and `renderChannels()` renders exactly the resolver output.

## Solution Assessment

No as submitted. Appending missing built-ins in the frontend is the right narrow boundary, but the hard-coded legacy list is not maintainable now that current main has catalog-backed bundled channel metadata.
