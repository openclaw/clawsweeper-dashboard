## Summary

Keep this PR open. Current main still couples the Agents Skills panel and top-level Skills tab through the same `skillsFilter` state, while the PR's state split is a narrow, coherent fix with no blocking review finding; the remaining work is maintainer review, rebase/mergeability, and focused regression coverage.

## What This Changes

The PR adds an `agentSkillsFilter` app/view-state field and rewires only the Agents Skills panel read/write bindings to use it instead of the top-level Skills tab's `skillsFilter`.

## Best Possible Solution

Land a rebased equivalent of the state split if independent Agents Skills and top-level Skills filters are intended, and add a focused regression test that exercises the app render boundary rather than only the leaf views.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: enter filter text through Agents -> Skills, then switch to the top-level Skills tab; current main reads and writes `state.skillsFilter` from both surfaces, so the filter carries over and can drive the Skills tab to `No skills found.`

## Solution Assessment

Yes. Splitting the state at the app/view-state boundary and rewiring only the Agents Skills bindings is the narrowest maintainable fix for independent filters; the safer completion is to add a small regression test before merge.
