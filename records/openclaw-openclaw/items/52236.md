## Summary

Keep open for maintainer follow-up, but do not merge as-is: current main has not implemented this route-persistence fix, while the patch writes the new route to the base session and still does not fix heartbeat thread routing.

## What This Changes

## Best Possible Solution

Revive or replace this with a narrow fix that persists non-direct Mattermost delivery context on the actual thread-scoped session entry, preserves heartbeat stale-thread safeguards unless an explicit turn-source thread is present, and proves cron or heartbeat thread wake delivery in Mattermost.

## Solution Assessment

No. This is a plausible prerequisite, but it is not the best fix because it writes non-direct thread context to the base route entry and does not address the heartbeat mode that intentionally drops session-stored thread IDs.

Full review comments:

- **[P1] Persist route updates on the thread session entry:** `extensions/mattermost/src/mattermost/monitor.ts:1380`
  - body: This widened block still resolves the last-route write from `route.sessionKey`, so a thread-scoped inbound message stores its route on the base Mattermost session instead of the actual `ctxPayload.SessionKey`. Cron delivery looks up the requested thread session first and only falls back when that entry has no route, so after thread B is the latest inbound message, a cron for thread A can inherit B's stored thread route and post to the wrong thread.
  - confidence: 0.86
- **[P2] Do not claim this fixes heartbeat thread routing:** `extensions/mattermost/src/mattermost/monitor.ts:1381-1383`
  - body: The new comment says this prevents cron/heartbeat wakeups from losing thread context, but heartbeat delivery still calls session target resolution in heartbeat mode, which intentionally drops session-derived thread IDs unless an explicit turn source supplied the thread. Please either fix the heartbeat path with a stale-thread-safe source or narrow this comment and PR claim to the cron/session route this patch actually covers.
  - confidence: 0.82
