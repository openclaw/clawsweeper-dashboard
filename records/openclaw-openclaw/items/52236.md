## What This Changes

The PR removes the direct-only guard around Mattermost last-route persistence and stores `effectiveReplyToId` as `deliveryContext.threadId` for inbound Mattermost sessions.

## Best Possible Solution

Replace or consolidate this with a current-main fix that persists non-direct delivery context on the active session or shared core fallback path, then keep https://github.com/openclaw/openclaw/issues/45082 open until that selected fix lands with proof.

## Reproduction Assessment

Yes, at source level: current main carries non-direct Mattermost `OriginatingTo` and `MessageThreadId` but still only provides `updateLastRoute` for direct messages. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting route context is the right direction, but this branch writes non-direct thread context to the shared main session on its old base; the safer solution is active-session persistence or the core fallback tracked by https://github.com/openclaw/openclaw/pull/77442.
