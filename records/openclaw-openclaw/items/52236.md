## Summary

Keep open for maintainer handling, but do not merge as-is: current main and v2026.6.6 still have the reported Mattermost route-persistence gap, while this abandoned draft is incomplete and can write thread context at the wrong route boundary.

## What This Changes

The PR removes the direct-message-only guard around Mattermost `updateLastRoute` and passes `effectiveReplyToId` as `threadId` when persisting inbound delivery context.

## Best Possible Solution

Replace or refresh this with a consolidated Mattermost-owned fix that persists routable delivery context on the exact active thread session, preserves main-session route policy, covers cron/heartbeat/message-tool paths, and includes redacted live Mattermost proof.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.6 still gate Mattermost `updateLastRoute` to direct messages, while scheduled delivery depends on stored route context and heartbeat mode deliberately drops inherited session thread IDs.

## Solution Assessment

No. The PR is a plausible prerequisite, but it is not the best fix because it broadens route persistence without targeting the exact active thread session or resolving heartbeat's thread-source rule.

- **[P1] Persist the route on the active thread session:** `extensions/mattermost/src/mattermost/monitor.ts:1380`
