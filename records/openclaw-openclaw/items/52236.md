## Summary

Keep open: current main still has the direct-only Mattermost route persistence gap, but this draft is not mergeable as-is because it writes non-direct thread context to the shared main session, lacks real Mattermost proof, and the author marked it abandoned.

## What This Changes

The PR removes Mattermost's direct-message-only guard around inbound last-route persistence and adds `effectiveReplyToId` as `deliveryContext.threadId`.

## Best Possible Solution

Replace or substantially revise this draft with a current-main fix that persists non-direct Mattermost route context on the active thread session, preserves main-session heartbeat safety, and includes focused regression tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes, by source inspection. Current main records Mattermost last-route context only for direct posts while non-direct thread turns carry the active `SessionKey`, `OriginatingTo`, and `MessageThreadId`.

## Solution Assessment

No. Persisting non-direct route context is the right direction, but this patch writes public thread context to the shared main session and does not resolve the heartbeat/current-thread cron policy called out in review discussion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist non-direct routes on the active session:** `extensions/mattermost/src/mattermost/monitor.ts:1380`
  - body: Removing the direct-only guard still writes the route through `route.mainSessionKey` while the active turn records a thread-scoped `SessionKey`. Cron delivery checks the requested thread session before falling back to main, so the intended thread entry can remain unroutable or inherit another thread's main-route write.
  - confidence: 0.88
- **[P1] Avoid storing public thread context on the main session:** `extensions/mattermost/src/mattermost/monitor.ts:1395`
  - body: The patch writes every non-DM Mattermost route, including public thread context, into the shared main session. Heartbeat delivery intentionally drops inherited session thread IDs for safety, so later main-session heartbeats or manual deliveries can jump into the last public thread.
  - confidence: 0.82
