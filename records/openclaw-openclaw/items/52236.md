## Label Justifications

- `P2`: This is a normal-priority Mattermost thread-routing fix with real message-delivery impact but limited channel-specific blast radius.
- `merge-risk: 🚨 compatibility`: The patch changes durable last-route behavior for existing Mattermost non-direct sessions and can alter where scheduled or heartbeat deliveries go after upgrade.
- `merge-risk: 🚨 session-state`: The diff writes Mattermost thread context into persisted session delivery state for session types that currently do not update that route.
- `merge-risk: 🚨 message-delivery`: The touched state decides where later cron, heartbeat, or queued replies are delivered, so a wrong write can misroute visible messages.

## Summary

Keep open for maintainer handling rather than auto-closing: this PR is an abandoned draft and currently conflicting, but it is paired with the same-author canonical open report at https://github.com/openclaw/openclaw/issues/45082. The patch is not merge-ready because it broadens Mattermost last-route writes without targeting the active thread session entry, so it can mis-associate session state and route later replies to the wrong place.

## What This Changes

The PR removes the direct-message guard around Mattermost last-route persistence and adds `threadId` to the stored delivery context.

## Best Possible Solution

Use a refreshed Mattermost-owned fix that writes route context to the correct active session/thread entry, preserves stale-thread protections for heartbeat unless maintainers choose otherwise, and keeps https://github.com/openclaw/openclaw/issues/45082 as the canonical report until a proven fix lands.

## Reproduction Assessment

Yes, at source level. Current main still gates Mattermost `updateLastRoute` to direct messages, and cron delivery resolves the thread session before falling back to the main session entry; I did not run a live Mattermost repro in this read-only review.

## Solution Assessment

No. Removing the guard is a plausible prerequisite, but this PR is not the best fix because it can write non-direct thread state to the shared main route and does not address the heartbeat policy called out in review discussion.

Full review comments:

- **[P1] Persist non-direct routes on the active session entry:** `extensions/mattermost/src/mattermost/monitor.ts:1380`
  - body: Removing the direct-message guard makes non-direct Mattermost turns update the existing last-route path without proving it writes the active thread session entry. Cron delivery checks the targeted thread session first and only then falls back to main, so storing the latest thread on the shared main route can make thread A inherit thread B, or make main-session deliveries jump into a public thread.
  - confidence: 0.88
- **[P2] Stop claiming heartbeat routing is fixed here:** `extensions/mattermost/src/mattermost/monitor.ts:1381-1383`
  - body: The new comment and PR description include heartbeat wakeups, but heartbeat resolution deliberately drops session-derived thread IDs unless the turn source explicitly carries one. Saving `threadId` in the session entry does not complete that path, so this needs either a heartbeat-specific fix or narrower wording before merge.
  - confidence: 0.86
