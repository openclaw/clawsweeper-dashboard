## Summary

Keep open for human handling: this draft was marked incomplete and abandoned by its author, lacks after-fix Mattermost proof, and still has blocking session-route/message-delivery risks that overlap the canonical open Mattermost thread work.

## What This Changes

The PR removes the Mattermost direct-only last-route guard and adds `effectiveReplyToId` as `deliveryContext.threadId` when recording inbound route context.

## Best Possible Solution

Consolidate the useful analysis into the canonical Mattermost thread-routing work, preferably via active-session or shared core delivery-context persistence, and keep https://github.com/openclaw/openclaw/issues/45082 open until a proven fix lands.

## Reproduction Assessment

Yes, at source level: current main carries Mattermost non-direct `OriginatingTo` and `MessageThreadId`, while explicit last-route updates are still direct-only. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting non-direct route context is the right direction, but this patch writes thread state through the shared main route and does not settle heartbeat stale-thread policy; the safer path is active-session or shared core delivery-context persistence.
