## Summary

Keep open. The underlying Mattermost thread-routing bug is still real on current main, but this abandoned draft only persists route metadata and does not complete the heartbeat/current-thread delivery behavior it claims to fix.

## What This Changes

The PR removes Mattermost's direct-message guard around inbound last-route recording and adds `threadId: effectiveReplyToId` to the persisted Mattermost delivery context.

## Best Possible Solution

Keep the canonical Mattermost thread-context issue open and land a complete Mattermost-local fix that covers route persistence, explicit thread-scoped heartbeat routing, and the cron/tool current-thread gap, or close this draft after maintainers choose a replacement path.

## Reproduction Assessment

Yes for source-level reproduction: current main creates Mattermost thread-scoped session keys for channel posts, but only persists last-route delivery context for direct messages. A live Mattermost cron/heartbeat run was not performed.

## Solution Assessment

No. Persisting `deliveryContext.threadId` for all Mattermost session types is a plausible prerequisite, but this PR is not a complete fix because heartbeat routing still drops session-derived thread IDs unless the plugin opts into route-thread preservation and the cron/tool gap remains unresolved.
