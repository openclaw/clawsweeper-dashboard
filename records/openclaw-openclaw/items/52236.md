## Summary

Keep open for maintainer handling: the branch targets a real Mattermost non-direct delivery-context gap, but it is draft/abandoned, lacks real behavior proof, still has blocking session-route defects, and overlaps open canonical work such as https://github.com/openclaw/openclaw/pull/77442 and https://github.com/openclaw/openclaw/issues/45082.

## What This Changes

The branch removes the Mattermost direct-message guard around last-route persistence and stores `effectiveReplyToId` as delivery-context `threadId`.

## Best Possible Solution

Consolidate this analysis into canonical Mattermost thread-routing work, preferably by persisting route context on the active thread/session entry or a shared core fallback path, with focused tests and redacted Mattermost proof before closing the linked issue.

## Reproduction Assessment

Yes, at source level: current main carries Mattermost non-direct `OriginatingTo` and `MessageThreadId`, while explicit last-route updates are still direct-only. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting non-direct route context is the right direction, but this patch writes through the existing base/main route and does not settle heartbeat stale-thread policy; the safer solution is active-session or shared core delivery-context persistence.

Overall confidence: 0.88

Full review comments:

- **[P1] Persist non-direct routes on the active session:** `extensions/mattermost/src/mattermost/monitor.ts:1380-1395`
  - body: The patch removes the direct-only guard but still writes through the existing last-route target derived from `route.sessionKey`, not the `threadContext.sessionKey` used for the active turn. Thread-scoped Mattermost sessions can therefore remain without their own delivery route and later cron delivery can fall back to unrelated main/base-session state.
  - confidence: 0.88
- **[P1] Avoid writing public thread context to the main session:** `extensions/mattermost/src/mattermost/monitor.ts:1395`
  - body: Adding `threadId: effectiveReplyToId` to the shared route update can make later main-session or heartbeat deliveries reuse the last public Mattermost thread. If the goal is only to fix thread-session wakeups, this needs a per-session write or shared fallback path instead of overwriting shared route state.
  - confidence: 0.84

  - body: The new comment says cron and heartbeat wakeups lose thread context, but current heartbeat delivery deliberately drops inherited session thread ids unless they come from an explicit turn source. Saving this field alone does not fix that path.
