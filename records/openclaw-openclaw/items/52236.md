## Summary

Keep open. Current main does not implement the PR’s intended Mattermost route persistence, but this draft should not merge as-is: the author and review discussion both identify that saving `threadId` is only a prerequisite, while heartbeat routing and cron current-thread scheduling remain unresolved under #45082/#52120.

## What This Changes

The PR removes Mattermost’s direct-message guard around inbound last-route persistence and adds the effective Mattermost reply root as `threadId` in the persisted delivery context.

## Best Possible Solution

Keep #45082 as the canonical Mattermost thread-routing bug and either finish a Mattermost-local fix covering inbound route persistence, heartbeat behavior for explicitly thread-scoped sessions, and cron/tool current-thread scheduling with focused tests, or close this abandoned draft after a replacement path is chosen.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: with `replyToMode: "all"`, Mattermost computes a thread-scoped session key and `MessageThreadId`, but current main only writes `updateLastRoute` for direct messages. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

No. Persisting `deliveryContext.threadId` for all Mattermost session types is a useful prerequisite, but it is not the complete fix because heartbeat resolution still drops session-derived thread IDs and the cron tool’s automatic current-context inference only covers agent-turn jobs.
