## Summary

Keep open for maintainer handling: the Mattermost routing bug is source-reproducible on current main, but this draft branch is abandoned, lacks real Mattermost proof, and writes thread route state to the shared main session instead of the active thread session.

## What This Changes

The PR removes the Mattermost direct-only last-route guard and adds `effectiveReplyToId` as `deliveryContext.threadId` when persisting inbound route context.

## Best Possible Solution

Replace or consolidate this with a current-main fix that persists non-direct delivery context on the active session or a shared core fallback path, then keep https://github.com/openclaw/openclaw/issues/45082 open until the selected fix lands with proof.

## Reproduction Assessment

Yes, at source level: current main carries non-direct Mattermost `OriginatingTo` and `MessageThreadId` but only provides `updateLastRoute` for direct messages. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting route context is the right direction, but this branch writes non-direct thread context to the shared main session; the safer solution is active-session persistence or the core fallback tracked by https://github.com/openclaw/openclaw/pull/77442.
