## Summary

Keep open. Current main still has the Mattermost route-persistence gap this PR targets, but this draft PR is abandoned and incomplete: the proposed one-file change does not cover heartbeat thread routing, and the author already confirmed the remaining heartbeat and cron-tool gaps in the PR discussion. The remaining work should stay coordinated under the canonical Mattermost thread-context issue and companion PR rather than being cleaned up as implemented.

## What This Changes

The PR removes Mattermost's direct-message guard around inbound last-route recording and adds `threadId: effectiveReplyToId` to the persisted Mattermost delivery context.

## Best Possible Solution

Keep the canonical Mattermost thread-context issue open and either land a complete Mattermost-local fix covering route persistence, heartbeat preservation for explicitly thread-scoped sessions, and the cron/tool current-thread gap, or close this draft after maintainers choose a replacement or decide to stop pursuing Mattermost thread-routing work.

## Reproduction Assessment

Yes for source-level reproduction: with Mattermost `replyToMode: "all"`, current main creates a thread-scoped session key for channel posts but only updates the last-route delivery context for direct messages. A live Mattermost cron/heartbeat run was not performed in this read-only review.

## Solution Assessment

No. Persisting `deliveryContext.threadId` for all Mattermost session types is a plausible prerequisite, but this PR is not the best complete fix because heartbeat routing still drops session-derived thread IDs and the cron tool still lacks a current-thread system-event target.
