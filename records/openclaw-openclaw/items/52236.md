- `merge-risk: 🚨 message-delivery`: The patch can route cron, heartbeat, or main-session replies to the wrong Mattermost location or thread.

## Summary

Keep open for maintainer handling: the PR targets a real Mattermost thread-routing gap, but the branch is draft/abandoned, lacks real Mattermost proof, and the patch can write public thread route state to the shared main session instead of the active thread-scoped session.

## What This Changes

The PR removes the Mattermost direct-only last-route guard and persists `effectiveReplyToId` as `deliveryContext.threadId` for inbound route context.

## Best Possible Solution

Replace or consolidate this with a current-main fix that persists non-direct Mattermost delivery context on the active thread session or a shared core fallback path, with tests and real Mattermost proof, while keeping https://github.com/openclaw/openclaw/issues/45082 open until a fix lands.

## Reproduction Assessment

Yes, at source level: current main carries non-direct Mattermost `OriginatingTo` and `MessageThreadId` but only supplies `updateLastRoute` for direct messages. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting non-direct route context is the right direction, but this branch writes thread context to the shared main session and does not settle heartbeat stale-thread policy; the safer fix is active-session persistence or the core fallback tracked by https://github.com/openclaw/openclaw/pull/77442.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist non-direct routes on the active session:** `extensions/mattermost/src/mattermost/monitor.ts:1393`
  - body: The patch removes the direct-only guard but still writes through `route.mainSessionKey`. For a thread-scoped Mattermost turn, that can leave the requested thread session entry without its own delivery context, so cron delivery can still miss the route or inherit an unrelated main-session route.
  - confidence: 0.88
- **[P1] Avoid writing public thread context to the main session:** `extensions/mattermost/src/mattermost/monitor.ts:1398`
  - body: Adding `threadId: effectiveReplyToId` to the main-session route means later main-session heartbeat or last-route sends can reuse the last public Mattermost thread. That changes main-session routing safety and can send replies into the wrong thread.
  - confidence: 0.82
- **[P2] Do not claim heartbeat routing is fixed here:** `extensions/mattermost/src/mattermost/monitor.ts:1385-1386`
  - body: The new comment says heartbeat wakeups lose thread context, but current heartbeat delivery deliberately drops inherited session thread IDs unless they come from an explicit turn source. Saving this field alone does not fix the heartbeat path, so the implementation or comment needs to narrow the claim.
  - confidence: 0.84
