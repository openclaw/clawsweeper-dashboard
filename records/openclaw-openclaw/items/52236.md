## Summary

Keep open for maintainer handling: the PR points at a real Mattermost thread-routing gap, but the patch writes thread route state to the shared main session, lacks after-fix real Mattermost proof, and the author later marked the branch incomplete and abandoned.

## What This Changes

The PR removes the Mattermost direct-only last-route guard and adds `threadId: effectiveReplyToId` when persisting inbound delivery context.

## Best Possible Solution

Replace or consolidate this branch with a current-main fix that persists non-direct Mattermost delivery context on the active thread session or shared core fallback path, with tests and redacted real Mattermost proof, while keeping the linked issue open until a fix lands.

## Reproduction Assessment

Yes, at source level: current main carries non-direct Mattermost thread and originating route fields but only supplies explicit last-route updates for direct messages. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No. Persisting non-direct route context is the right direction, but this patch writes thread context to the shared main session and does not settle heartbeat stale-thread policy; the safer solution is active-session persistence or the core fallback tracked by https://github.com/openclaw/openclaw/pull/77442.
