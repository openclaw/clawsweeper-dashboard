## Summary

Keep this open. Current main still matches the reported macOS node failure path: Gateway serves A2UI behind scoped canvas auth, but `MacNodeRuntime` resolves A2UI through the shared Gateway connection and does not use or refresh the active node-scoped canvas capability before returning `A2UI_HOST_UNAVAILABLE`.

## What This Changes

Gateway’s macOS node A2UI host should be reachable so `canvas.a2ui.push`, `canvas.a2ui.pushJSONL`, and `canvas.a2ui.reset` succeed from a paired macOS node.

## Best Possible Solution

Update the macOS node runtime/coordinator boundary so A2UI uses the active `GatewayNodeSession` scoped canvas URL, refreshes that capability and retries once when readiness fails, and adds focused macOS coverage for the stale/missing capability path while keeping Gateway canvas auth unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: invoke a macOS node `canvas.a2ui.push` or reset when the shared canvas URL is missing or the node-scoped capability is stale; current `MacNodeRuntime.ensureA2UIHost()` resolves once and returns `A2UI_HOST_UNAVAILABLE` without refreshing the node session capability.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `GatewayNodeSession.currentCanvasHostUrl()` and `refreshNodeCanvasCapability()` contract from the macOS node path, matching the iOS/Android retry pattern instead of weakening Gateway auth or changing A2UI routing.
