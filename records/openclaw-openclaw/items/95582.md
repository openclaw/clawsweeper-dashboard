## Summary

Keep open: current main and latest release still create `backup create` staging and archive-temp artifacts and only clean them in-process, with no source evidence of a startup sweep for hard-kill leftovers.

## What This Changes

The issue asks for `backup create` to sweep stale backup temp staging directories and temp archives from prior hard-killed runs before starting a new backup.

## Best Possible Solution

Add a bounded `backup create` startup sweep for aged own-artifacts, covering `openclaw-backup-` staging dirs in the chosen temp root and output-sibling UUID temp archives with a concurrency grace guard.

## Reproduction Assessment

Yes. Source inspection matches the reporter's path: `createBackupArchive` creates a staging dir and temp archive, then removes both only in `finally`, so a hard kill can leave them behind; I did not run a destructive kill repro in this read-only review.

## Solution Assessment

Yes. A narrow startup sweep owned by the backup implementation is the maintainable fix, with the memory temp cleanup in https://github.com/openclaw/openclaw/pull/92891 as useful precedent but not a direct fix for this surface.
