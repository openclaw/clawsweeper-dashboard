## Summary

Keep open: current `main` and the latest release still create the reported backup staging directory and output-adjacent temp archive, and cleanup still only runs during normal in-process unwinding. The issue is now paired with an open closing PR, so it should remain open until that fix lands or maintainers choose another path.

## What This Changes

The issue requests that `backup create` sweep stale `openclaw-backup-*` staging directories and `<output>.<uuid>.tmp` archive files left by hard-killed prior runs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/95600 or a narrow equivalent that sweeps both artifact classes with an active-owner guard, then close this issue and reconcile the overlapping backup temp reports.

## Reproduction Assessment

Yes, at source level: current `createBackupArchive` creates the reported temp archive and staging directory and removes both only in an in-process `finally`. I did not run a destructive hard-kill reproduction in this read-only review.

## Solution Assessment

Yes, the requested next-run startup sweep is the right repair shape, provided it is tightly scoped and protects active backup runs. The current best path is to review and land the linked open PR rather than closing this issue before merge.
