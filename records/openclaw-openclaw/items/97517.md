## Label Justifications

- `P1`: The issue affects an active live model-switch workflow and can make a working cross-harness Codex switch appear rejected or running on the wrong runtime.
- `impact:session-state`: The reported behavior is a mismatch between persisted session model fields, liveModelSwitchPending, and the actual active runtime state.
- `impact:auth-provider`: The affected flow switches between Anthropic/claude-cli and Codex/OpenAI app-server provider identity, so wrong status or reset messages can mislead provider/auth diagnosis.

## Summary

Keep open: current main still has the generic model-override reset wording that can report an allow-list rejection, and the inspected status/runtime paths do not prove that a pending cross-harness Codex app-server switch is reflected as the active runtime. The related session-model issues overlap, but they do not fully cover this false reset message plus native `/status` runtime-label drift.

## What This Changes

Report that `/status` and model-override reset messaging misrepresent an active cross-harness switch from `claude-cli` to the Codex app-server.

## Best Possible Solution

Separate true allow-list rejection from Codex runtime startup or cold-start failure, and make native `/status` report one effective live runtime/model identity or an explicit pending/runtime mismatch coordinated with #92415/#93056.

## Reproduction Assessment

No, not as a live reproduction in this review. Source inspection confirms the reset-message, pending live-switch, status-label, and Codex binding paths, giving a medium-confidence source-reproducible path but not the exact cold-start failure trigger.

## Solution Assessment

Yes, the best fix belongs in the session/runtime identity and error-reporting boundary rather than adding config or plugin surface. It should distinguish disallowed models from runtime startup failures and make `/status` consume the same effective live runtime authority.
