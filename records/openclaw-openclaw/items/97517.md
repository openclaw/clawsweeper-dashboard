## Summary

Current main does not yet cover the issue's central failure modes: allowed Codex-routed model switches can still be reported with stale/disallowed allow-list wording, and native `/status` still derives identity from persisted selection/runtime status paths rather than the Codex app-server binding that proves the active thread/model during this failure class. Adjacent fixes for session status usage and default-model switching are useful but do not make this issue obsolete.

## What This Changes

The issue requests guardrails so `/model` and `/status` distinguish allowed Codex app-server model routing from stale or disallowed model overrides during live session switches.

## Best Possible Solution

Keep this issue open until native `/status` and model-override reset handling share a canonical active-session identity that distinguishes allow-list denial, stale overrides, and Codex app-server startup or binding failures.
