## Summary

Keep open: current main does pass `file:` config env values through literally, but this PR creates a new undocumented config-secret shorthand and currently bypasses the existing file SecretRef safety model, so it needs maintainer product/security approval plus hardening before merge.

## What This Changes

The PR adds `file:<relpath>` handling to `applyConfigEnvVars` and a new test file for file-backed config env values under the OpenClaw state directory.

## Best Possible Solution

If maintainers approve the shorthand, implement it through the existing file SecretRef provider or equivalent guardrails, then document the contract and add a changelog entry.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping existing env values and `${VAR}` placeholders, so a `file:` value remains literal. I did not run a live xAI request because the read-only config path establishes the behavior.

## Solution Assessment

No. The PR demonstrates the requested behavior, but the maintainable path is to reuse or mirror the existing file SecretRef safety model and only ship the new shorthand after maintainer approval.
