## Summary

Keep open: the linked behavior is source-reproducible on current main, but this PR adds a new config value scheme that still needs maintainer approval and has a concrete security blocker in the proposed file-read path.

## What This Changes

The PR adds `file:<relpath>` handling for config env vars in `openclaw.json` and tests file content resolution, trimming, missing files, traversal rejection, and non-overwrite behavior.

## Best Possible Solution

If maintainers approve the shorthand, implement it through a shared SecretRef-grade bounded file reader or equivalent guarded helper, then document and changelog the supported semantics.

## Reproduction Assessment

Yes, source-level: current main applies config env strings literally after skipping existing values and unresolved `${VAR}` placeholders, so a `file:` value stays literal. A live xAI request is not needed to verify this config-path behavior.

## Solution Assessment

No: the PR demonstrates the desired behavior, but direct synchronous reads from config-selected paths are not the safest maintainable solution. The safer path is maintainer-approved shorthand backed by the same safety model as file SecretRefs, plus docs and changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:42`
  - body: This new `file:` path goes from state-dir string containment to `fs.readFileSync`. A symlink inside the state dir can escape that containment, a FIFO can block Gateway startup, and a large file can be read into memory, while the existing file SecretRef provider rejects unsafe paths and enforces timeout/maxBytes. Please route this through that safety path or add equivalent checks before reading.
  - confidence: 0.88

Please update the environment/config docs with the supported `file:` syntax, relative path base, trimming, and failure behavior if this API is accepted.
