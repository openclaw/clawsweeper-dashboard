## Summary

Keep open: current main still leaves `file:` config env values literal, but this PR adds new config-secret syntax and the proposed implementation bypasses the established file SecretRef safety checks, so it should not merge as-is.

## What This Changes

The PR adds `file:<relpath>` handling to `applyConfigEnvVars` plus tests for resolving config env values from files under the OpenClaw state directory.

## Best Possible Solution

If maintainers approve the shorthand, implement it through the existing SecretRef file-provider safety model or equivalent regular-file, symlink, permission, timeout, and size checks, then document and changelog the contract.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a live xAI request because the read-only config path is enough to establish current behavior.

## Solution Assessment

No: the PR demonstrates the requested behavior but reads state-dir files synchronously without the established file SecretRef guardrails. The safer path is to reuse or mirror that provider model after maintainer approval of the new shorthand.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:41`
  - body: This new `file:` path goes straight from the state-dir prefix check to `fs.readFileSync`. Please route this through that safety path or add equivalent checks before reading.
