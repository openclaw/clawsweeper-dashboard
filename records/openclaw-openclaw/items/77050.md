## Summary

Keep open: the PR targets a valid source-reproducible config env bug, but the current patch introduces a custom-state-dir failure path during models.json fingerprinting and misses the required release note.

## What This Changes

The PR adds `file:<relpath>` handling to `applyConfigEnvVars` plus tests for file content, CRLF trimming, missing files, traversal rejection, and existing-env precedence.

## Best Possible Solution

Keep the PR open and make file-ref resolution state-dir-aware across runtime and fingerprint callers, with regression coverage for custom `OPENCLAW_STATE_DIR` and a changelog entry.

## Reproduction Assessment

Yes, at source level: current `applyConfigEnvVars` assigns `file:` values literally, and the config load path applies those env vars before provider/model auth resolution. The PR's remaining blocker is also source-reproducible through `buildModelsJsonFingerprint` calling `createConfigRuntimeEnv(params.config, {})`.

## Solution Assessment

No, not yet: adding resolution at the config-env boundary is reasonable, but the patch must avoid resolving state-dir files from synthetic empty env callers or pass the correct state-dir env through those paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep file refs out of synthetic env fingerprints:** `src/config/config-env-vars.ts:134`
  - body: The new resolver runs for every `applyConfigEnvVars` caller, but `buildModelsJsonFingerprint` calls `createConfigRuntimeEnv(params.config, {})` with an empty env. Pass the correct state-dir env into that path or keep fingerprinting from reading secret files.
