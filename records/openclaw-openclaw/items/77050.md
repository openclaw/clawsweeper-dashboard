## Summary

Keep open: current main still treats `file:` env values literally, but this PR adds a new undocumented config shorthand and the latest head still performs direct synchronous file reads without the established file SecretRef guardrails or a changelog entry.

## What This Changes

The PR adds `file:<relpath>` handling to `applyConfigEnvVars` plus tests for reading env values from files under the OpenClaw state directory.

## Best Possible Solution

Decide the config contract first; if accepted, implement `file:` env values through SecretRef-equivalent safe file reads and update docs plus changelog.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after skipping `${VAR}`, so `file:secrets/xai-api-key.txt` remains literal. The linked xAI auth failure matches that path.

## Solution Assessment

No: the PR demonstrates the requested behavior, but the direct read path is not the safest maintainable implementation. The safer route is either to keep using documented SecretRefs or to reuse their file-provider safety model if maintainers approve the shorthand.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:43`
  - body: This new `file:` branch goes from a state-dir containment check straight to `fs.readFileSync`. A FIFO or other special file under the state dir can block Gateway startup, and a large file can be read into memory, while the existing file SecretRef provider rejects unsafe paths and enforces timeout and maxBytes. Please route this through that safety path or add equivalent checks before reading.
