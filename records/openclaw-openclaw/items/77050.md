## Summary

Keep this PR open: the branch demonstrates the requested `file:` env behavior, but it adds a new undocumented config value scheme and currently bypasses the existing file SecretRef safety model, so it needs maintainer product/security approval before merge.

## What This Changes

The PR adds `file:<relpath>` handling to config env values in `applyConfigEnvVars` and adds tests for reading files under the OpenClaw state directory.

## Best Possible Solution

If maintainers approve the shorthand, implement it through the existing SecretRef file-provider safety model or equivalent guardrails, then document and changelog the contract.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping existing values and `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. No live xAI request is needed to verify that code path.

## Solution Assessment

No: the direct state-dir read is not the best way to solve it because the repository already has a file SecretRef provider with stronger validation, timeout, and size controls. The safer solution is to reuse or mirror that model after maintainers approve the new shorthand.

## Review Findings
