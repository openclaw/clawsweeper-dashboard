## Summary

Keep open: current main leaves `file:` config env values literal, but this PR is not merge-ready because it adds a new undocumented config value scheme and reads config-selected files outside the existing SecretRef file-provider safety checks.

## What This Changes

The PR adds `file:<relpath>` resolution for `openclaw.json` config `env` values and tests state-dir-relative file-backed env values.

## Best Possible Solution

If maintainers approve this shorthand, implement it through the existing SecretRef file-provider validation model or an equivalent shared helper, then document and changelog the new contract.

## Reproduction Assessment

Yes, source-level: current main applies config `env` strings literally after only skipping existing env values and unresolved `${VAR}` references, so `file:secrets/xai-api-key.txt` remains literal. A live xAI request is not needed to verify that config-path behavior.

## Solution Assessment

No. The PR demonstrates the requested behavior, but the safer maintainable solution is to reuse or mirror the SecretRef file-provider guardrails after maintainers approve the new shorthand.

## Review Findings
