## Summary

Keep open: current main still copies `file:` config env values literally and this PR is the open implementation candidate for the linked report, but the proposed direct file read needs maintainer approval for a new config-secret contract, SecretRef-grade safety checks, and real behavior proof before merge.

## What This Changes

The PR adds `file:<relpath>` handling to `src/config/config-env-vars.ts` and tests that state-dir-relative files populate config env vars without overriding existing env values.

## Best Possible Solution

Keep structured SecretRef objects as the canonical file-backed secret path unless maintainers approve `file:` env shorthand; if approved, implement it through the existing file-provider safety model or an equivalent guarded helper with docs and focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main copies config env strings into the runtime environment and only skips existing env values and `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. A live xAI request is not needed to verify that config-path behavior.

## Solution Assessment

No: the PR demonstrates the requested shorthand, but the direct synchronous file read is not the safest maintainable implementation for secret-bearing config. The safer path is to reuse or mirror the existing SecretRef file-provider guardrails after maintainers decide the shorthand should exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:41`
  - body: This new `file:` path goes from state-dir containment directly to `fs.readFileSync`. A symlink inside the state dir can escape that containment, a FIFO can block gateway startup, and a large file can be read into memory, while the existing file SecretRef provider already rejects unsafe paths and enforces timeout and maxBytes limits.
  - confidence: 0.88
