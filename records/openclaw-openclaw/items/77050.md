## Summary

Keep this PR open: it targets a real source-level config behavior gap, but the patch adds a new config-driven file read shorthand that bypasses the existing SecretRef file-provider safety model and still needs maintainer product/security approval before merge.

## What This Changes

The PR teaches `applyConfigEnvVars` to resolve `file:<relpath>` values relative to the OpenClaw state directory and adds unit coverage for file-backed env vars.

## Best Possible Solution

If maintainers approve the shorthand, implement it through the existing SecretRef file-provider guardrails or an equivalent shared helper, then document and changelog the supported contract.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping existing env and `${VAR}` placeholders, so `file:...` remains literal. A live xAI request is not needed to establish this config-path behavior.

## Solution Assessment

No. The PR demonstrates the requested behavior, but the current implementation is not the safest path because it bypasses the established file SecretRef safety model and adds an undocumented config shorthand.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:41`
  - body: This new `file:` path goes from state-dir containment to `fs.readFileSync`. A FIFO or other special file under the state dir can block Gateway startup, and a large file can be read into memory, while the existing file SecretRef provider rejects unsafe paths and enforces timeout/maxBytes. Please route this through that safety path or add equivalent checks before reading.
  - confidence: 0.86
