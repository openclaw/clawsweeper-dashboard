## Summary

Keep open: current main leaves `file:` config env values literal and this PR targets the linked report, but the branch still needs maintainer product/security approval and hardening before merge.

## What This Changes

The PR adds `file:<relpath>` handling to `applyConfigEnvVars` plus tests for file-backed config env values under the OpenClaw state directory.

## Best Possible Solution

If maintainers approve this shorthand, implement it through the existing SecretRef file-provider safety model or equivalent guarded helper, then document and changelog the contract.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping existing env and `${VAR}` placeholders, so `file:...` remains literal. A live xAI request is unnecessary to establish this config-path behavior.

## Solution Assessment

No: the PR demonstrates the requested behavior but is not the safest implementation because it bypasses the established file SecretRef guardrails. The safer path is to reuse or mirror that provider model after maintainer approval of the new shorthand.
