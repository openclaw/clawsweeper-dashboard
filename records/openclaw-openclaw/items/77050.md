## Summary

Keep open: the PR addresses a source-reproducible config path, but it adds a new public `file:` env shorthand and currently bypasses the existing SecretRef file-safety model, so it needs maintainer product/security approval before merge.

## What This Changes

The PR adds `file:<relpath>` resolution in config env vars plus tests for state-dir-relative file-backed env values.

## Best Possible Solution

If maintainers approve this shorthand, implement it through the existing SecretRef file-provider safety model or equivalent guarded helper, then document and changelog the contract.

## Reproduction Assessment

Yes, source-level: on current main `applyConfigEnvVars` assigns config env strings literally after only skipping existing env values and `${VAR}` placeholders, so `file:...` stays literal. A live xAI request is not needed to verify this config-path behavior.

## Solution Assessment

No: the PR demonstrates the requested behavior, but reading config-selected paths directly is not the safest implementation. The safer maintainable path is to reuse or mirror the SecretRef file-provider guardrails after maintainer approval of the new shorthand.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate file refs before reading them:** `src/config/config-env-vars.ts:41`
  - body: This new `file:` path goes from state-dir containment to `fs.readFileSync`. Please route this through that safety path or add equivalent checks before reading.
