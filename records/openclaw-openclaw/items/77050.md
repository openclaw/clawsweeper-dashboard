## Summary

Keep open: the PR targets a real source-level gap for `file:` strings, but it adds a new undocumented config env shorthand and bypasses existing file SecretRef guardrails, so it needs product/security approval and hardening before merge.

## What This Changes

The PR adds `file:` value resolution in `applyConfigEnvVars` plus tests for reading config env values from files under the OpenClaw state directory.

## Best Possible Solution

Adopt a single documented secret-loading contract: either keep env values inline/`${VAR}` plus structured SecretRefs, or approve `file:` as a shorthand implemented through the same hardened file-provider semantics.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after skipping existing env values and `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a gateway/provider request because this read-only review established the path from source.

## Solution Assessment

No: direct synchronous file reads are not the best safe implementation for a secret-bearing config surface. If maintainers want the shorthand, it should reuse or mirror the file SecretRef provider's realpath, regular-file, permission, timeout, and size checks and update docs/changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
