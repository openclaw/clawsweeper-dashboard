## Summary

Keep open: the branch demonstrates the requested shorthand, but it adds a new config-driven secret file read contract and bypasses the existing file SecretRef safety model, so it needs maintainer product/security approval and hardening before merge.

## What This Changes

The PR adds `file:<relpath>` handling to config env application plus tests for reading state-directory files and rejecting missing files, traversal, CRLF, and existing-env overrides.

## Best Possible Solution

If maintainers approve this config shorthand, implement it through the existing file SecretRef resolver or an equivalent guarded helper, then document and changelog the new contract; otherwise steer users to structured file SecretRefs.

## Reproduction Assessment

Yes, source-level: current main assigns config env strings directly after only skipping existing values and `${VAR}` placeholders, so `file:secrets/xai-api-key.txt` remains literal. I did not run a live provider request in this read-only review.

## Solution Assessment

No: the patch is not the best way yet because it creates a direct config file-read path instead of reusing the established SecretRef file-provider guardrails. The safer path is an approved config contract plus the shared guarded file-secret resolution model.
