## Summary

Keep open: this is a fresh external bug report with concrete production liveness traces, and current main still processes Codex app-server notifications on the gateway event-loop thread without proof that the reported wedge is fixed.

## What This Changes

Investigate Codex runtime notification handling that appears to starve the Node event loop during `account/rateLimits/updated` and `mcpServer/startupStatus/updated` notifications.

## Best Possible Solution

Instrument and reproduce the Codex notification path, then keep only minimal ordered bookkeeping on the gateway event-loop turn while deferring or bounding expensive notification processing with regression coverage for bursty rate-limit and MCP startup notifications.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The reporter provided concrete live cron and MCP steps with liveness logs, and source inspection confirms the current notification path remains present on main.

## Solution Assessment

Unclear: deferring or budgeting notification work is a plausible direction, but profiling is needed before choosing between macrotask yielding, worker-thread parsing/projection, narrower synchronous bookkeeping, or upstream Codex handling.
