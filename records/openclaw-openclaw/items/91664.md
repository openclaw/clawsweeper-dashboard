## Summary

Keep open: current main still has a plausible source-level path where dashboard sessions created without an explicit key get a fresh UUID-backed key, while labels are stored on the individual session row. ## What This Changes

The issue reports that Control UI dashboard session labels are saved under volatile dashboard session keys and disappear after a Gateway restart or browser reconnect.

## Best Possible Solution

Define the intended dashboard session identity contract, then make reconnect/restart preserve labels through that canonical identity with focused UI and Gateway regression coverage.

## Reproduction Assessment

No: source inspection shows the volatile sessions.create fallback and per-row label storage, but I did not establish a live current-main reconnect reproduction. The normal refresh path can also preserve a selected session via URL/settings state, so the exact failing path remains the gap.

## Solution Assessment

Unclear: carrying labels forward is plausible, but the best fix should first decide whether dashboard labels belong to a stable dashboard key or should migrate by underlying sessionId. A narrow fix should avoid adding a broad runtime fallback stack around session metadata.
