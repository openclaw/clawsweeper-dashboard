## Summary

Keep this PR open for maintainer review: current main and the latest release still have TTL-only `agentRunCache` behavior, and the PR is a focused, well-proven candidate fix, but it deliberately changes how many terminal run snapshots remain available under fan-out.

## What This Changes

The branch adds a fixed 5,000-entry FIFO cap to gateway `agentRunCache` and a regression test that overflows the cache through lifecycle events.

## Best Possible Solution

Land the focused cap once gateway owners accept or adjust the terminal-snapshot retention budget, and keep the linked issue open until this PR or an equivalent fix merges.

## Reproduction Assessment

Yes. Source inspection shows unique terminal lifecycle events with fresh run IDs flow into `recordAgentRunSnapshot`, which currently stores snapshots with only TTL pruning; I did not run a stress loop because this review is read-only.

## Solution Assessment

Yes for the fix shape. A local cap on the existing in-process cache is the narrowest maintainable memory bound; only the exact cap value and eviction budget need maintainer acceptance.
