## Summary

Keep open: this PR still addresses a real unbounded in-process gateway cache on current main, and I found no blocking patch defect. The contributor supplied sufficient copied live runtime proof; the remaining merge concern is the intentional terminal-snapshot retention tradeoff from evicting oldest entries after the hard cap.

## What This Changes

The PR adds a 5,000-entry FIFO cap to `agentRunCache` and a regression test that drives unique lifecycle end events through `emitAgentEvent`.

## Best Possible Solution

Land the fixed hard-cap mitigation after gateway owners accept the 5,000-entry terminal-snapshot retention budget, and keep the linked issue open until the PR merges.

## Reproduction Assessment

Yes. Source inspection of current main shows `agentRunCache` is pruned by a 10-minute TTL but has no size bound, so a stream of unique terminal lifecycle events can grow it until TTL pruning removes old entries.

## Solution Assessment

Yes. A local hard cap at the lifecycle snapshot cache is the narrowest maintainable fix, matches existing bounded Map patterns, avoids a new config surface, and leaves gateway dedupe as the primary terminal-state path for normal agent calls.
