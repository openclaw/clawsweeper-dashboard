## Summary

Keep open: current main and the latest release still lack the cache cap, and this PR is a focused candidate with sufficient proof. The remaining maintainer question is whether the fixed 5,000-entry FIFO retention budget is acceptable for the `agent.wait` lifecycle snapshot fallback.

## What This Changes

Adds a 5,000-entry FIFO cap to gateway `agentRunCache`, plus testing-only cache helpers and a regression test that drives terminal lifecycle events past the cap.

## Best Possible Solution

Land the focused cap once gateway owners accept or adjust the terminal-snapshot retention budget, and keep the linked issue open until this PR or an equivalent fix merges.

## Reproduction Assessment

Yes. Unique lifecycle terminal events with fresh run IDs flow through `recordAgentRunSnapshot`, which on current main stores each snapshot with only TTL pruning; I did not run a stress loop because this review is read-only.

## Solution Assessment

Yes. A local hard cap on the existing in-process cache is the narrowest maintainable fix; a config knob or persistent store would add product and upgrade surface that this bug does not need, though the exact cap value needs maintainer acceptance.
