## What This Changes

Adds a 5,000-entry FIFO cap and regression coverage for the gateway agent-run terminal snapshot cache.

## Best Possible Solution

Keep the hard size bound, but make every accepted snapshot update refresh the run’s eviction position—or evict by snapshot timestamp—and add a repeated-run overflow regression through `waitForAgentJob`.

## Reproduction Assessment

Yes at source level: current main admits every unique terminal snapshot during the ten-minute TTL, and the submitted runtime output demonstrates the proposed cap engaging at 5,000 entries. The update-order defect follows directly from JavaScript Map replacement semantics and the changed eviction loop.

## Solution Assessment

No, not yet; adding a hard bound is the correct direction, but insertion-order eviction must reflect the latest accepted snapshot rather than a run’s first insertion.

- **[P2] Refresh a run's position when its snapshot is updated:** `src/gateway/server-methods/agent-job.ts:79-84`
  - body: `Map.set()` on an existing key does not move that key to the end of insertion order. Delete and reinsert accepted updates, or evict by `ts`, and add a repeated-run overflow test that reads the result through `waitForAgentJob`.
