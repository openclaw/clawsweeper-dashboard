## Summary

Keep open: current main does not implement the proposed fallback, and this PR still has a high-confidence phantom-agent routing bug, reuses an explicit-binding match label for an implicit path, lacks real Discord proof, and changes a permission-sensitive routing convention that needs maintainer judgment.

## What This Changes

The PR adds Discord-only fallback routing from an unmatched non-default account ID to a same-named agent ID and adds resolver tests for the happy path and default-account guard.

## Best Possible Solution

If maintainers accept the same-name routing convention, update the branch with an existing-agent guard, a distinct implicit match reason, focused tests/docs, and redacted live Discord proof; otherwise keep explicit bindings as the supported path.

## Reproduction Assessment

Yes, at source level. Current main passes Discord accountId into resolveAgentRoute and falls through to the default route when no explicit binding matches; the PR's phantom-agent edge follows from pickFirstExistingAgentId echoing sanitized IDs when the agent list is empty.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs an existing-agent guard, a separate implicit match reason, docs/tests, and real Discord proof before merge.
