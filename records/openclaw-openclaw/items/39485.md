## Summary

Keep open. Current main still has the Discord account-name routing gap described by the linked issue, but this PR should not merge as-is because it can synthesize phantom agent routes and hides implicit matches behind the explicit binding label.

## What This Changes

The PR adds a Discord-only implicit accountId-to-agentId fallback after explicit route bindings and adds resolver tests for positive matching and the default-account guard.

## Best Possible Solution

Have a maintainer confirm the implicit Discord account-name convention, then land a guarded resolver change that only matches configured agents, exposes a distinct implicit match reason, covers positive/default/empty-agent-list cases, and updates changelog/docs as needed.

## Reproduction Assessment

Yes. The linked issue gives concrete multi-account Discord steps, and current main shows the resolver falls through from explicit binding tiers to the default route when no binding matches.

## Solution Assessment

No. The central resolver is the right implementation boundary if maintainers accept the convention, but this patch needs a configured-agent guard, a distinct match label, and release-facing documentation before it is the narrow maintainable fix.

  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and routes to a phantom agent instead of falling through to the default.
