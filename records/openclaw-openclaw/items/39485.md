## Summary

Keep open: the PR targets a real Discord multi-account routing problem, but it is not merge-ready because it lacks real Discord proof, introduces a phantom-agent route when no agents are configured, and changes a permission-sensitive routing contract that needs maintainer approval.

## What This Changes

The PR adds a Discord-only implicit fallback that routes unmatched non-default account IDs to same-named agents and adds two resolver unit tests.

## Best Possible Solution

Land a revised version only after maintainers approve the implicit fallback contract, implicit matches are limited to configured agents, diagnostics distinguish implicit from explicit bindings, docs/tests are aligned, and redacted Discord runtime proof is supplied.

## Reproduction Assessment

Yes, at source level. The PR helper calls pickFirstExistingAgentId, which echoes a sanitized account ID when no agents are configured, so the new normalized equality check accepts any non-default Discord account instead of preserving the default route.

## Solution Assessment

No. Same-name Discord fallback may be a reasonable direction, but this implementation needs a configured-agent guard, distinct diagnostics, docs alignment, real Discord proof, and maintainer product/security approval before merge.
