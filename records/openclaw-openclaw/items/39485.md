## Summary

Keep open: this PR targets a real Discord routing gap, but the patch can route unmatched accounts to non-existent agents, reuses explicit-binding diagnostics for an implicit convention, lacks real Discord proof, and still needs maintainer product/security approval.

## What This Changes

Adds a Discord-only fallback in the shared route resolver that maps unmatched non-default `accountId` values to a same-named agent, plus two resolver tests.

## Best Possible Solution

Land a revised version only after it requires an existing configured agent, emits a distinct implicit match reason, updates docs/tests for the routing contract, and includes redacted real Discord proof after maintainer product/security approval.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured; no live Discord repro was run in this read-only review.

## Solution Assessment

No. Same-name Discord fallback may be a reasonable direction, but this implementation needs a configured-agent guard, distinct diagnostics, docs alignment, real Discord proof, and maintainer product/security approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:179-184`
  - body: When `cfg.agents` is absent or empty, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this helper accepts any non-default Discord account as a match and routes to a non-existent agent instead of preserving the default fallback.

  - confidence: 0.84
