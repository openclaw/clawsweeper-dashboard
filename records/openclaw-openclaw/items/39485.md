## Summary

Keep open. Current main still lacks the implicit Discord account-name routing this PR tries to add, but the patch changes a routing permission boundary and still has a source-proven phantom-agent regression before it can merge.

## What This Changes

The PR adds a Discord-only fallback in `resolveAgentRoute` to route unmatched non-default Discord `accountId`s to same-named agents and adds two resolver tests.

## Best Possible Solution

Have a maintainer confirm the implicit Discord account-name convention, then land a guarded resolver change that only matches configured agents, uses a distinct match reason, covers positive/default/empty-agent cases, and updates changelog/docs as needed.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: Discord passes `accountId` to `resolveAgentRoute`, the resolver checks explicit bindings and then defaults, and the docs currently require explicit account bindings.

## Solution Assessment

No. The central resolver is the right implementation boundary if maintainers accept the convention, but this patch needs a configured-agent guard, a distinct `matchedBy` value, and release-facing documentation before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require an existing agent for implicit matches:** `src/routing/resolve-route.ts:181-185`
  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and routes to a phantom agent instead of falling through to the default. Check the configured agent list before returning the candidate and add no-agent coverage.
  - confidence: 0.91
- **[P3] Use a distinct implicit match reason:** `src/routing/resolve-route.ts:826-828`
  - body: This path did not come from an explicit account binding, but it logs and returns `matchedBy: "binding.account"`. Add a dedicated value such as `implicit.account` so diagnostics can distinguish configured bindings from the naming-convention fallback.
  - confidence: 0.84

  - confidence: 0.78
