## Summary

Keep open: this is a plausible Discord routing fix, but it changes the documented explicit-binding contract, lacks real Discord after-fix proof, and the patch introduces a permission-sensitive phantom-agent route when no agents are configured.

## What This Changes

The PR adds a Discord-only fallback that maps an unmatched non-default accountId to a same-named agent and adds resolver tests for the positive and default-account cases.

## Best Possible Solution

Have maintainers decide the Discord routing contract; if accepted, land a guarded implicit account mapping with distinct diagnostics, docs/tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, source-level. Current main passes Discord accountId into resolveAgentRoute and falls through to the default route when explicit binding tiers miss; the PR-introduced phantom-agent regression follows from pickFirstExistingAgentId returning sanitized input when no agents exist.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch must require an existing configured agent, distinguish implicit matches from explicit bindings, document the contract, and include real Discord proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:184`
  - body: When cfg.agents is absent or empty, pickFirstExistingAgentId echoes sanitizeAgentId(accountId), so this helper accepts any non-default Discord account and routes to an agent that does not exist instead of falling back to the default route.
  - confidence: 0.91

  - body: This path did not match an explicit account binding, but it reports binding.account in route state and debug logs. A separate matchedBy value would let operators distinguish configured bindings from the naming-convention fallback.
  - confidence: 0.84
