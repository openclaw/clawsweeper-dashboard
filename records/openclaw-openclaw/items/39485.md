## Summary

Keep open: this PR targets a real linked Discord routing gap, but the proposed patch is not merge-ready because it can route non-default Discord accounts to phantom agents, reports implicit matches as explicit account bindings, and lacks contributor real behavior proof.

## What This Changes

The PR adds a Discord-only resolver fallback that maps an unmatched non-default `accountId` to a same-named agent and adds resolver tests for the fallback and default-account guard.

## Best Possible Solution

Land this PR or an equivalent maintainer-approved routing change only after it requires an existing configured agent, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof; explicit bindings remain the supported workaround until then.

## Reproduction Assessment

Yes, at source level. Current `main` falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs a configured-agent existence guard, a distinct implicit match reason, docs alignment, and real Discord proof before merge.
