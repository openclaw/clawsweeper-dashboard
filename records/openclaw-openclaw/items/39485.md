## Summary

Keep open: the PR targets a real open Discord routing report, but it is not safe to merge as-is because it introduces a phantom-agent routing path, reuses an explicit-binding diagnostic label for an implicit match, lacks real Discord behavior proof, and still needs maintainer approval for the implicit same-name routing contract.

## What This Changes

The PR adds a Discord-only implicit fallback in `resolveAgentRoute` that maps unmatched non-default `accountId` values to same-named agents and adds two resolver tests.

## Best Possible Solution

Have maintainers decide the Discord routing contract; if accepted, land a guarded fallback or explicit account-binding contract with distinct diagnostics, tests, docs/changelog alignment, and redacted real Discord proof.

## Reproduction Assessment

Yes, at source level. Current main passes Discord `accountId` into `resolveAgentRoute` and falls through to the default agent when explicit binding tiers miss; the PR-introduced empty-agent phantom route is also visible through `pickFirstExistingAgentId`.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch should first require an existing configured agent, use a distinct implicit match reason, update release-facing docs, and include real Discord proof.
