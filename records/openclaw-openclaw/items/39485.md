## Summary

Keep open: current main has not implemented this behavior, and the patch is not merge-ready because it changes Discord routing policy while still allowing phantom-agent routes and ambiguous diagnostics.

## What This Changes

The PR adds a Discord-only implicit accountId-to-agentId fallback in the shared route resolver and two resolver tests for same-name routing and the default-account guard.

## Best Possible Solution

If maintainers accept the convention, implement it as a guarded resolver tier that only matches configured non-default Discord agents, uses a distinct implicit match reason, covers positive/default/empty-agent cases, and updates changelog/docs.

## Reproduction Assessment

Yes. Source inspection shows Discord passes accountId into the shared resolver, and current main falls through to the default route when no explicit binding matches a same-named Discord account and agent.

## Solution Assessment

No. The shared resolver is the right boundary if maintainers accept the behavior, but this patch needs an existing-agent guard, a distinct matchedBy value, and release-facing docs/changelog before it is the narrow maintainable fix.

  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and routes to a non-existent agent instead of falling through to the default.
