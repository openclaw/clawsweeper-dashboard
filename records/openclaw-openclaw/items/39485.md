## Summary

Keep open: current main still does not implement implicit Discord accountId-to-agent routing, and this PR targets the open linked routing issue, but the patch is not merge-ready because it introduces a phantom-agent route, ambiguous diagnostics, and lacks real Discord runtime proof.

## What This Changes

The PR adds a Discord-only fallback in the shared route resolver that maps a non-default accountId to a same-named agent and adds two resolver tests.

## Best Possible Solution

Have maintainers decide the Discord same-name routing contract; if accepted, land a guarded implicit account mapping with distinct diagnostics, docs/tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, source-level. Current main passes Discord accountId into resolveAgentRoute and falls through to the default route when explicit binding tiers miss; the PR-introduced phantom-agent path follows from pickFirstExistingAgentId returning sanitized input when no agents exist.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch should require an existing configured agent, distinguish implicit matches from explicit bindings, document the contract, and include real Discord proof.
