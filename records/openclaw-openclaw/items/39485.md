## Summary

Keep open: current main still lacks the Discord accountId-to-agentId fallback needed for #39428, and this PR is useful but cannot merge as written because it can route to synthesized agent IDs and reports implicit matches as explicit bindings.

## What This Changes

The PR adds a Discord-only implicit accountId-to-agentId route fallback after explicit binding tiers and adds two route resolver tests for matching and default-account behavior.

## Best Possible Solution

Revise the fallback in the central route resolver so it remains Discord-only, runs after explicit binding tiers, skips accountId "default", requires a configured agent-list match, returns a distinct reason such as "implicit.account", and covers the positive, default-account, and empty-agent-list cases in routing tests.

## Reproduction Assessment

Yes. #39428 gives concrete multi-account Discord steps, and current main shows why the request falls to the default route; the PR-specific regression is reproducible with a Discord route, a non-default accountId, and no configured agents.

## Solution Assessment

No, not as currently proposed. The central resolver is the right place for this behavior, but the patch needs an explicit configured-agent guard and a distinct match label before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require an existing agent for implicit account matches:** `src/routing/resolve-route.ts:178-182`
  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so the normalized equality check succeeds and this fallback routes to a phantom agent instead of the default. Add an explicit configured-agent presence check before returning the candidate.
  - confidence: 0.91
- **[P3] Use a distinct label for implicit account matches:** `src/routing/resolve-route.ts:826-828`
  - body: This route did not come from an explicit account binding, but it is reported as `binding.account`. Operators and downstream diagnostics need to distinguish configured bindings from this naming-convention fallback, so add a dedicated matchedBy value such as `implicit.account`.
  - confidence: 0.84
