## Summary

Keep open: current main and the latest release still use explicit binding/default routing, while this PR changes permission-sensitive Discord route selection and still has correctness, security, and real-behavior-proof blockers.

## What This Changes

The PR adds a Discord-only fallback that maps an unmatched non-default `accountId` to a same-named agent and adds resolver tests for the positive case and the `default` account guard.

## Best Possible Solution

Accept implicit Discord account-to-agent routing only with maintainer approval, an existing-agent guard, distinct implicit-match diagnostics, docs/tests, and redacted live Discord proof; otherwise keep explicit `bindings[].match.accountId` as the supported contract.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary only if maintainers accept the same-name convention, but this patch needs an existing-agent check, a distinct match reason, docs alignment, and real Discord proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:184`
  - body: When `cfg.agents` is absent or empty, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this helper accepts any non-default Discord account and routes to a non-existent agent instead of preserving the default route.

  - confidence: 0.84
