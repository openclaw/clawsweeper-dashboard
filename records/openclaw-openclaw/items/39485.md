## Summary

Keep open: the PR targets a real Discord multi-account routing gap, but it is not merge-ready because it adds an implicit permission-sensitive routing mode with a source-level phantom-agent defect, conflicts with main, and lacks real Discord proof.

## What This Changes

The PR adds a Discord-only fallback in the shared route resolver that maps unmatched non-default account IDs to same-named agents and adds resolver unit tests.

## Best Possible Solution

If maintainers accept implicit same-name Discord routing, land it only after it matches configured agents, uses distinct diagnostics, covers default and empty-agent cases, resolves conflicts, documents the contract, and proves the route in a real Discord multi-account setup.

## Reproduction Assessment

Yes, at source level. Current main receives Discord accountId in resolveAgentRoute and falls through to the default agent after explicit binding tiers; the PR-introduced empty-agent defect follows from pickFirstExistingAgentId returning a sanitized ID when no agents are configured.

## Solution Assessment

No, not as submitted. Same-name Discord fallback may be useful, but it should land only after maintainer approval with configured-agent guarding, distinct diagnostics, conflict resolution, and real Discord proof.
