## Summary

Keep open: the PR targets a real linked Discord routing pain point, but current main has not implemented this behavior and the branch still has a P1 phantom-agent regression, missing real Discord proof, a conflicting head, and unresolved routing-contract/security questions.

## What This Changes

The PR adds a Discord-only fallback in `resolveAgentRoute` that maps unmatched non-default `accountId` values to same-named agent IDs and adds resolver tests for the positive and default-account cases.

## Best Possible Solution

If maintainers accept same-name Discord routing, land it with a configured-agent existence guard, distinct implicit-match diagnostics, docs/tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account id when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary only if maintainers accept the same-name Discord convention, but this patch should require an existing configured agent, distinguish implicit matches from explicit bindings, document the contract, and include real Discord proof.
