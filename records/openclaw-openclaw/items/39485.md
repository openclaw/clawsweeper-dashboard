## Summary

Keep open: current main still falls back to the default agent when explicit Discord account bindings miss, and this PR is the open implementation for https://github.com/openclaw/openclaw/issues/39428. The patch is not merge-ready because it can create phantom agent routes, reuses the explicit binding diagnostic label, lacks real Discord proof, and needs maintainer approval for the new implicit routing convention.

## What This Changes

The PR adds a Discord-only fallback in `resolveAgentRoute` that maps unmatched non-default `accountId` values to same-named agents and adds two resolver tests.

## Best Possible Solution

Have maintainers decide whether Discord same-name account routing belongs in the contract; if accepted, land a guarded implementation with a distinct match reason, docs/tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main routes Discord `accountId` through `resolveAgentRoute` and falls through to the default when explicit binding tiers miss; the PR-introduced phantom route is visible because `pickFirstExistingAgentId` echoes a sanitized ID when no agents are configured.

## Solution Assessment

No. The resolver is a plausible boundary only if maintainers accept the convention, but this patch should require an existing configured agent, use a distinct implicit match reason, update docs/tests, and include real Discord proof.
