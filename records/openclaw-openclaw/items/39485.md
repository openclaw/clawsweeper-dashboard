## Summary

Keep open: current main still lacks the Discord accountId-to-agentId fallback, and this PR is a focused attempt for the linked bug, but the proposed patch has a blocking no-agent-list routing regression and an ambiguous match label that should be revised before merge.

## What This Changes

The PR adds a Discord-only implicit fallback after explicit routing bindings so a non-default Discord accountId matching an agent ID routes to that agent, plus two routing regression tests.

## Best Possible Solution

Keep the implementation direction but revise it before merge: add a narrow Discord-only fallback after explicit binding tiers, skip accountId "default", require the candidate to be present in agents.list, return a distinct match reason such as "implicit.account", and add regression coverage for positive matching, default-account behavior, and the empty-agent-list guard.

## Reproduction Assessment

Yes. #39428 provides concrete multi-account Discord reproduction steps, and the current source path shows why it falls to the default agent; the PR-specific regression is also reproducible with a Discord route, non-default accountId, and an absent or empty agents.list.

## Solution Assessment

No, not as currently proposed. The fallback belongs in the central route resolver, but it should verify an existing configured agent and use a distinct match label before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require an existing agent for implicit account matches:** `src/routing/resolve-route.ts:178-182`
  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so the normalized equality check succeeds and this fallback routes to a phantom agent instead of the default. Add an explicit configured-agent presence check before returning the candidate.
  - confidence: 0.91
- **[P3] Use a distinct label for implicit account matches:** `src/routing/resolve-route.ts:826-828`
  - body: This route did not come from an explicit account binding, but it is reported as `binding.account`. Operators and downstream diagnostics need to distinguish configured bindings from this naming-convention fallback, so add a dedicated matchedBy value such as `implicit.account`.
  - confidence: 0.84
