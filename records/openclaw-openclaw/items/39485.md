## Summary

Keep open: current main and v2026.5.12 still use explicit routing bindings and default fallback, while this PR remains a plausible but security-sensitive Discord routing change with a real patch defect and missing real Discord proof.

## What This Changes

Adds a Discord-only fallback in `resolveAgentRoute` that maps an unmatched non-default `accountId` to a same-named agent, plus two resolver tests.

## Best Possible Solution

Land this PR or a replacement only after it requires an existing configured agent, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs a configured-agent existence guard, a distinct implicit match reason, docs alignment, and real Discord proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:179-184`
  - body: When `cfg.agents` is absent or empty, `pickFirstExistingAgentId` returns `sanitizeAgentId(accountId)`, so this helper accepts any non-default Discord account and routes to a non-existent agent instead of preserving the default fallback.
  - confidence: 0.91
- **[P3] Report implicit matches distinctly:** `src/routing/resolve-route.ts:831`
  - body: This path did not match an explicit binding, but it returns and logs `binding.account`; add a distinct match reason so diagnostics can separate configured bindings from the naming-convention fallback.
  - confidence: 0.84
