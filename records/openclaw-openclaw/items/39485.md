## Summary

Keep open: the PR targets a permission-sensitive Discord routing gap, but the proposed fallback can route to phantom agents, reuses explicit-binding diagnostics, and has no real Discord behavior proof. It also changes the documented explicit-binding contract, so maintainer product/security approval is needed before merge.

## What This Changes

The PR adds a Discord-only resolver fallback that routes an unmatched non-default `accountId` to a same-named agent and adds two resolver tests.

## Best Possible Solution

Accept a maintainer-approved Discord account-routing design that only maps configured agents, reports implicit matches distinctly, updates docs/tests, and is proven with redacted live Discord evidence.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs a configured-agent existence guard, a distinct implicit match reason, docs alignment, and real Discord proof before merge.
