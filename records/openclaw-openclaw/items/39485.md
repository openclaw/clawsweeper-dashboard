## Summary

Keep this PR open: it targets a real Discord routing gap tracked by https://github.com/openclaw/openclaw/issues/39428, but the current patch introduces a phantom-agent routing path, lacks real Discord proof, and changes a permission-sensitive default routing convention that needs maintainer approval.

## What This Changes

The PR adds a Discord-only fallback that routes unmatched non-default `accountId` values to same-named agent IDs and adds resolver tests for the positive and default-account cases.

## Best Possible Solution

Accept implicit Discord account-to-agent routing only with an existing-agent guard, distinct diagnostics, docs/tests, and redacted live Discord proof; otherwise keep explicit account bindings as the supported contract.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account id when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs a configured-agent existence guard, a distinct implicit match reason, docs, and real Discord proof before merge.
