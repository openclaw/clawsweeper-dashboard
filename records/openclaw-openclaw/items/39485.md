## Summary

Keep open: current main still lacks the implicit Discord accountId-to-agentId fallback, but this external PR cannot merge as-is because it adds a permission-sensitive routing convention with a phantom-agent edge and no real behavior proof.

## What This Changes

The PR adds Discord-only implicit routing from unmatched non-default account IDs to same-named agents and adds resolver tests.

## Best Possible Solution

Have maintainers decide the Discord routing contract; if accepted, land a guarded fallback after explicit binding tiers with distinct diagnostics, docs/changelog alignment, tests, and redacted real Discord proof.

## Reproduction Assessment

Yes, at source level. Current main sends Discord `accountId` into `resolveAgentRoute` and falls through to the default agent after explicit binding tiers; the PR-introduced no-agent phantom route follows from `pickFirstExistingAgentId` echoing sanitized IDs when no agents are configured.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch should require an existing configured agent, use a distinct implicit match reason, update release-facing docs, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:181-185`
  - body: When `cfg.agents` is absent or empty, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and routes to a non-existent agent instead of falling back to the default route.
  - confidence: 0.91
- **[P3] Use a distinct implicit match reason:** `src/routing/resolve-route.ts:831`
  - body: This path did not come from an explicit account binding, but it returns `matchedBy: "binding.account"`. Add a dedicated reason such as `implicit.account` so logs and downstream diagnostics can distinguish configured bindings from the naming-convention fallback.
  - confidence: 0.84
