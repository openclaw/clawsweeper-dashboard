## Summary

Keep open: current main still uses explicit binding tiers followed by the default agent, so the central Discord account-to-agent fallback is not implemented, and this PR still has blocking correctness, diagnostics, real-proof, and routing-contract issues noted by prior review comments.

## What This Changes

The PR adds a Discord-only fallback that maps an unmatched non-default `accountId` to a same-named agent ID and adds two resolver tests for positive matching and the default-account guard.

## Best Possible Solution

Accept implicit Discord account-to-agent routing only if maintainers approve the convention, with an existing-agent guard, distinct match diagnostics, focused docs/tests, and redacted real Discord proof; otherwise keep explicit `bindings[].match.accountId` as the supported contract.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch needs a configured-agent existence guard, a distinct implicit match reason, docs alignment, and real Discord proof before merge.
