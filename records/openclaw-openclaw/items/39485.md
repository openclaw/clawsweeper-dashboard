## Summary

Keep open: current main still uses explicit binding tiers followed by the default-agent fallback, and this PR is the active linked implementation for the open Discord routing report. It should not merge yet because the patch can create phantom-agent routes, reuses an explicit-binding diagnostic label for an implicit fallback, lacks contributor real Discord proof, and needs maintainer product/security approval for the new same-name routing convention.

## What This Changes

Adds a Discord-only resolver fallback that maps an unmatched non-default `accountId` to a same-named agent and adds two resolver tests.

## Best Possible Solution

Land this PR or a replacement only after it requires an existing configured agent, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof with maintainer product/security approval.

## Reproduction Assessment

Yes, at source level. Current main falls through to the default route after explicit binding tiers, and the PR-introduced phantom-agent path follows from `pickFirstExistingAgentId` echoing a sanitized account ID when no agents are configured; I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. The shared resolver is a plausible boundary if maintainers accept the convention, but this patch still needs a configured-agent existence guard, a distinct implicit match reason, docs alignment, and real Discord proof before merge.
