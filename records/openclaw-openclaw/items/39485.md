## Summary

Keep open: current main still falls back to the default route after explicit binding tiers, but this PR changes a Discord permission-boundary routing convention and is not mergeable as-is because the implicit path can synthesize non-existent agents and needs clearer diagnostics plus release notes.

## What This Changes

The PR adds a Discord-only fallback in `resolveAgentRoute` that maps unmatched non-default `accountId` values to same-named agents and adds two resolver tests.

## Best Possible Solution

Have a maintainer confirm whether OpenClaw should support this implicit Discord account-name convention; if accepted, land a guarded resolver change with a distinct match reason, empty-agent coverage, and changelog/docs alignment.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord passes `accountId` to `resolveAgentRoute`, the resolver checks explicit binding tiers, and then returns the default route without an implicit same-name fallback.

## Solution Assessment

No. The central resolver is the right boundary if maintainers accept the convention, but this patch needs an existing-agent guard, a distinct implicit match reason, and release-facing documentation before it is the narrow maintainable fix.

## Review Findings

  - body: When no agents are configured, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and routes to a phantom agent instead of falling through to the default.
