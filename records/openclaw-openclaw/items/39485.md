## Summary

Keep open: current main still lacks implicit Discord accountId-to-agentId routing, but this PR has blocking routing/security concerns and only unit-test evidence. The remaining path needs maintainer approval for the implicit routing contract plus contributor-provided real Discord proof.

## What This Changes

The PR adds a Discord-only implicit fallback that maps unmatched non-default `accountId` values to same-named agents and adds resolver tests for the positive and default-account cases.

## Best Possible Solution

If maintainers accept the convention, land a guarded implicit mapping that only matches configured agents, uses distinct diagnostics, documents/tests the contract, and includes redacted live Discord proof.

## Reproduction Assessment

Yes, source-level. Current main passes Discord `accountId` into `resolveAgentRoute` and falls through to the default route when explicit binding tiers miss; the PR-introduced no-agent phantom route follows from `pickFirstExistingAgentId` returning sanitized input when no agents exist.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch must require an existing configured agent, distinguish implicit matches from explicit bindings, and include real Discord proof.
