## Summary

Keep open: current main still lacks the requested implicit Discord same-name routing, but this PR should not merge as-is because it can create phantom agent routes, reuses the explicit binding diagnostic reason, lacks live Discord proof, and needs maintainer approval against the current explicit-bindings contract.

## What This Changes

This PR adds a Discord-only implicit accountId-to-agentId fallback after explicit binding tiers and adds resolver tests for same-name matching plus the default-account guard.

## Best Possible Solution

Have maintainers decide the Discord routing contract; if accepted, land a guarded fallback with distinct diagnostics, no phantom agents, docs/changelog alignment, focused tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level: current main passes Discord `accountId` into `resolveAgentRoute` and falls through to the default agent after explicit binding tiers. The PR-introduced phantom-agent edge is also source-reproducible because `pickFirstExistingAgentId` echoes requested IDs when no agents are configured.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers accept the convention, but this patch should require an existing configured agent, use a distinct implicit match reason, update docs/changelog as needed, and include live Discord proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a configured agent for implicit matches:** `src/routing/resolve-route.ts:181-185`
  - body: When `cfg.agents` is absent or empty, `pickFirstExistingAgentId` returns a sanitized copy of `accountId`, so this guard accepts any non-default Discord account and returns a route for a non-existent agent instead of falling back to the default.
