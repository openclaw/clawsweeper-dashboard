## Summary

Keep open: current main still lacks the requested Discord account-name fallback, but this PR is not safe to merge as-is because it changes permission-sensitive routing, can synthesize phantom agent routes, and has no real Discord behavior proof.

## What This Changes

The PR adds a Discord-only fallback in `resolveAgentRoute` that maps unmatched non-default `accountId` values to same-named agents and adds resolver tests.

## Best Possible Solution

Land either the existing explicit-binding contract or a maintainer-approved guarded Discord fallback with no phantom agents, distinct diagnostics, docs/changelog alignment, focused tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main routes a Discord `accountId` through the shared resolver and falls through to the default route when no explicit binding matches; the PR-introduced no-agent-list regression is also visible from `pickFirstExistingAgentId`.

## Solution Assessment

No. The resolver boundary is plausible if maintainers accept the convention, but this patch must require an existing configured agent, use a distinct match reason, align docs/changelog, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect
