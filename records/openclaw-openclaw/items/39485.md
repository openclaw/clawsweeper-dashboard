## Summary

Keep open: the PR is focused on a real Discord routing concern, but it is not merge-ready because it changes permission-sensitive default routing, is currently conflicting, lacks real Discord proof, and introduces a P1 phantom-agent routing defect.

## What This Changes

The branch adds a Discord-only fallback that maps an unmatched non-default `accountId` to a same-named configured agent and adds resolver tests for the positive and default-account cases.

## Best Possible Solution

Land a revised, maintainer-approved Discord routing policy that only maps to configured agents, reports implicit matches distinctly, documents the contract, and is proven in a redacted real Discord multi-account setup.

## Reproduction Assessment

Yes, at source level: current main passes Discord account IDs into `resolveAgentRoute` and falls through to the default after explicit binding tiers, while the PR-introduced empty-agent-list defect follows from `pickFirstExistingAgentId` echoing sanitized IDs when no agents are configured. I did not run live Discord proof in this read-only review.

## Solution Assessment

No. Same-name Discord fallback may be useful, but this implementation needs a configured-agent guard, distinct match diagnostics, docs/tests, conflict resolution, real Discord proof, and maintainer product/security approval before merge.
