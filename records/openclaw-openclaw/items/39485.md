## Summary

Keep open: this is a plausible routing fix path, but the patch should not merge as-is because it can create phantom agent routes, reuses explicit-binding diagnostics for implicit routing, lacks real Discord behavior proof, and needs maintainer approval for the implicit same-name account routing policy.

## What This Changes

The branch adds a Discord-only implicit fallback from unmatched non-default `accountId` values to same-named agents and adds resolver tests for the positive case and default-account guard.

## Best Possible Solution

If maintainers accept the convention, land a guarded Discord routing implementation with an agent-existence check, distinct implicit match reason, docs/changelog alignment, focused tests, and redacted real Discord proof.

## Reproduction Assessment

Yes, source-level: current main falls through to the default agent after explicit binding tiers, and the PR-introduced no-agent phantom route follows from `pickFirstExistingAgentId` echoing sanitized IDs when the agent lookup is empty. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. The resolver is a plausible boundary if maintainers approve the routing convention, but this patch needs an existing-agent guard, a distinct implicit match reason, docs alignment, and real runtime proof before merge.
