## Summary

Keep this PR open, but do not merge the current patch as-is. The reported symptom is real on current main, and the contributor supplied useful proof, but the implementation puts exit-code state inside shared doctor execution and changes the plain `openclaw doctor` contract where the documented automation path is already `openclaw doctor --lint`.

## What This Changes

The PR makes final doctor config validation set a non-zero process exit code for invalid final config and makes the top-level doctor command pass that code through.

## Best Possible Solution

Keep the invalid-config detection, but route exit status through an explicit doctor result or top-level wrapper decision so embedded/update callers retain control and maintainers can intentionally choose whether plain doctor or only lint fails closed.

## Reproduction Assessment

Yes, from source inspection: current main prints final `Invalid config:` errors in `runFinalConfigValidationHealth`, then the plain doctor wrapper exits 0 after `doctorCommand` returns. I did not run tests or mutate the checkout because this review is read-only.

## Solution Assessment

No. The patch fixes the visible symptom, but the current implementation uses process-global exit state inside shared doctor health and changes the public plain-doctor exit contract without resolving the existing `doctor --lint` automation boundary.
