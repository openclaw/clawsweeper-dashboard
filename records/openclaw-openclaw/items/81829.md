## What This Changes

This PR threads a `finalConfigInvalid` doctor health result from final config validation to the top-level doctor command and updates tests/callers so plain `openclaw doctor` exits 1 when the persisted config remains invalid.

## Best Possible Solution

Land a rebased version only after maintainers confirm plain doctor should fail on final invalid config and full CLI proof shows the exit code; otherwise keep lint as the automation surface.

## Reproduction Assessment

Yes, source-reproducible: current main prints final `Invalid config:` diagnostics in `runFinalConfigValidationHealth`, then the plain doctor wrapper still exits 0 after `doctorCommand` returns. I did not run the CLI because this was a read-only review.

## Solution Assessment

Yes for the code shape once product direction is accepted: carrying a typed result from final validation to the top-level CLI keeps update callers on their own exit policy. The safer alternative is to preserve plain-doctor compatibility and keep automation on `doctor --lint` if maintainers do not want this contract change.
