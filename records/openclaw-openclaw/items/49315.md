## Summary

Keep this PR open, but do not merge the current patch. The requested feature is real because current main still lacks `--profile-id` on `models auth login`, but the PR applies the requested id after provider auth results are already built, which can leave provider-owned config such as `auth.order` pointing at the original profile id.

## What This Changes

The branch adds `--profile-id` to `models auth login`, remaps single-profile auth results to the requested id, adds a regression test, and updates CLI, gateway authentication, FAQ, and OpenAI provider docs for named auth profiles.

## Best Possible Solution

Implement the feature by passing a typed requested login profile target into provider auth before each provider constructs `ProviderAuthResult`; providers that can safely honor it should build profiles and config patches from that target, while unsupported or multi-profile methods should reject deliberately. Add focused tests for OpenAI Codex and at least one configPatch provider, update docs, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main reproduces the feature gap by inspection because `models auth login` has no `--profile-id` option, and the PR defect is reproducible at source level with providers such as Microsoft Foundry that build `configPatch.auth.order` from the original profile id before the PR's remap runs.

## Solution Assessment

No. Rewriting profile ids after `method.run` is not the best fix because providers may already have built config patches, defaults, or side effects around their original id; the safer solution is to thread the requested target into provider auth result construction.
