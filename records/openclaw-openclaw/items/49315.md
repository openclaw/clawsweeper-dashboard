## Summary

Keep open: current main still lacks named login-profile support for `models auth login`, but this PR is not merge-ready because it applies the requested id after provider auth result construction, which can leave provider-owned routing config pointing at the old profile id. The PR also lacks the required changelog entry for a user-facing CLI feature.

## What This Changes

The PR adds a `--profile-id` flag to `models auth login`, remaps single-profile auth results to that id, adds focused profile-remapping coverage, and updates CLI, gateway, FAQ, and OpenAI provider docs for named login profiles.

## Best Possible Solution

Add a typed requested login-profile target to the provider auth context before providers construct `ProviderAuthResult`; implement the OpenAI Codex OAuth/device-code behavior against that target, define unsupported-provider behavior deliberately, validate collisions and provider ownership, and update docs, changelog, and focused tests around the final contract.

## Reproduction Assessment

Yes. Current main reproduces the feature gap by source inspection because `models auth login` has no `--profile-id`; the PR defect is reproducible at source level with Microsoft Foundry, which embeds its original profile id in `configPatch.auth.order` before the PR remaps returned profiles.

## Solution Assessment

No. Applying the requested id after `method.run` is not the best fix because providers may already have built config patches, defaults, or side effects around the original id; the safer path is to thread a typed profile target into provider auth before result construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Thread profile targets before providers build config:** `src/commands/models/auth.ts:294-297`
  - body: This remaps only the profiles passed to persistence after `method.run` has already built the full `ProviderAuthResult`. Providers can embed the original profile id in `configPatch` such as `auth.order`, so the requested id can be persisted while routing metadata still points at the old id. Pass the requested profile target into the provider auth context before result construction, or restrict the flag to methods whose result is profile-id independent.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/cli/models-cli.ts:310-313`
  - body: This adds user-facing CLI behavior and public docs but does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing feature changes, so the release notes would omit the new flag.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but the auth-profile routing change can desynchronize the credential users think they selected from provider-owned routing metadata.

Concerns:

- **[medium] Profile remap can desynchronize auth selection:** `src/commands/models/auth.ts:294`
  - body: Because the requested profile id is applied after provider result construction, provider-supplied config such as auth order can continue to reference the original profile id. That can make operators believe a named credential controls selection while stale provider-owned ids remain in routing metadata.
  - confidence: 0.82
