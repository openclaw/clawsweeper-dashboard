## Summary

Keep this PR open. Current main still lacks the requested `models auth login --profile-id` surface, so the PR is addressing a real gap, but the proposed post-login profile rewrite is not safe enough to merge because provider-owned config patches can still reference the original profile id.

## What This Changes

The PR adds `--profile-id` to `models auth login`, remaps single-profile auth results to that id, adds a focused regression test, and updates CLI, gateway authentication, FAQ, and OpenAI provider docs for named login profiles.

## Best Possible Solution

Revise or replace the PR so the requested login profile target is threaded through a typed provider auth context before providers construct profiles and config patches. OpenAI Codex browser OAuth and device-code should honor that target, unsupported providers should reject or ignore it deliberately, collisions and provider ownership should be validated, `--agent` behavior should remain intact, and docs/changelog/tests should cover the final contract.

## Reproduction Assessment

Yes. Current main reproduces the feature gap by inspection because the login CLI has no `--profile-id` option; the PR defect is also reproducible at source level with a provider like Microsoft Foundry that embeds its original `profileId` in `configPatch.auth.order` before the PR remaps returned profiles.

## Solution Assessment

No. Applying the requested id after `method.run` is not the best fix because providers may have already built config patches, defaults, or side effects around the original id; the safer path is to pass a typed requested target into provider auth methods before result construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread profile targets before providers build config:** `src/commands/models/auth.ts:294-297`
  - body: This remaps only the profiles passed to persistence after `method.run` has already built the full `ProviderAuthResult`. Providers can embed the original profile id in `configPatch` such as `auth.order`, so the requested id can be persisted while routing metadata still points at the old id. Pass the requested profile target into the provider auth context before result construction, or restrict the flag to methods whose result is profile-id independent.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but the auth-profile routing change can desynchronize the credential users think they selected from provider-owned routing metadata.

Concerns:

- **[medium] Profile remap can desynchronize auth selection:** `src/commands/models/auth.ts:294`
  - body: Because the requested profile id is applied after provider result construction, provider-supplied config such as auth order can continue to reference the original profile id. That can make operators believe a named credential controls selection while stale provider-owned ids remain in routing metadata.
  - confidence: 0.82
