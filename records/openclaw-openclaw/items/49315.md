## Summary

Keep open: current main still lacks the requested login flag, but this PR is not merge-ready because it remaps profile ids only after provider auth has already run and it has no real behavior proof.

## What This Changes

The PR adds `--profile-id` to `models auth login`, remaps single-profile auth results before persistence, adds focused tests, and updates CLI/gateway/OpenAI auth docs.

## Best Possible Solution

Land a rebased provider-auth contract change that adds `--profile-id`, forwards a typed target before provider credential/config writes, rejects ambiguous or unsupported methods, and updates docs/tests with real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI flag and provider-auth capability. Source inspection confirms current main lacks `--profile-id` on `models auth login`, while `paste-token` has that flag.

## Solution Assessment

No as currently proposed: the direction is useful, but remapping returned profiles after provider auth is not the safest contract. The maintainable path is to pass a typed target into `ProviderAuthContext` before running the provider, or reject methods that cannot safely honor it.
