## Summary

Keep open: the requested capability is still absent from current main and the canonical feature issue remains open, but this PR should not merge as-is because the target profile is remapped only after provider auth runs and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a `--profile-id` option to `models auth login`, remaps single-profile auth results before persistence, adds helper coverage, and updates CLI/gateway/OpenAI auth docs.

## Best Possible Solution

Land a rebased provider-auth contract change that adds `--profile-id`, forwards a typed target before provider credential/config writes, rejects ambiguous or unsupported methods, and updates docs/tests with real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI flag and provider-auth capability. Source inspection confirms current main still lacks `--profile-id` on `models auth login`, while `paste-token` has that flag.

## Solution Assessment

No as currently proposed: the direction is useful, but remapping returned profiles after provider auth is not the safest contract. The maintainable path is to pass a typed target into provider auth before credentials/config are created, or reject methods that cannot safely honor it.
