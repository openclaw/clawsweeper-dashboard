## Summary

Keep open for maintainer review: current main still lacks the requested feature and the related open issue tracks this PR as the implementation candidate, but the branch is conflicting, lacks real behavior proof, and has a correctness issue in post-auth remapping.

## What This Changes

The PR adds a `models auth login --profile-id` option, remaps single-profile provider auth results to that id, adds focused coverage, and updates CLI/gateway/OpenAI auth docs.

## Best Possible Solution

Define the public target-profile login contract, then land a rebased implementation that threads the requested id through provider-safe auth context before side effects, rejects unsupported ambiguous methods, updates docs/tests, and includes redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR for a new CLI flag. Source inspection on current main confirms `models auth login` still lacks `--profile-id` while `paste-token` has it.

## Solution Assessment

No as currently proposed: the direction is useful, but the PR head remaps after provider auth methods run and can leave side-effecting providers with the original profile already written. The safer shape is to pass the requested profile into providers before side effects or reject methods that cannot safely honor it.

## Review Findings

Overall correctness: patch is incorrect
