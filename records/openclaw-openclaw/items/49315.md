## Summary

Keep open: current main and the latest release still lack the requested login-time profile flag, but this PR is not merge-ready because it remaps provider auth results after the provider method has already produced credentials/config and it lacks real behavior proof.

## What This Changes

The branch adds `--profile-id` to `openclaw models auth login`, remaps single-profile provider auth results to that ID, adds a focused helper test, and updates auth-profile docs.

## Best Possible Solution

Land a rebased provider-auth contract change that passes a typed target into supported provider auth methods before credentials/config are produced, rejects ambiguous or unsupported methods clearly, and includes docs, tests, and real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI flag and provider-auth capability. Source inspection confirms current main and v2026.5.18 still lack `--profile-id` on `models auth login`, while `paste-token` has separate profile-id handling.

## Solution Assessment

No as currently proposed: the direction is useful, but post-return profile remapping is not the safest provider-auth contract. The maintainable path is to pass a typed target into provider auth before credentials/config are created, or reject methods that cannot safely honor it.
