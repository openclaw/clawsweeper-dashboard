## Summary

Keep open: the feature is still not implemented on current main, but the PR is not merge-ready because it remaps the profile only after provider auth runs, lacks after-fix real behavior proof, and is now conflicting with current auth CLI changes.

## What This Changes

This PR adds `--profile-id` to `models auth login`, post-processes single-profile auth results under that requested ID, adds focused tests, and documents named OAuth login profiles.

## Best Possible Solution

Land or replace this with a rebased provider-auth contract change that passes a typed requested profile target into supported provider auth methods before credentials/config are produced, rejects unsupported or ambiguous methods clearly, updates docs/tests, and includes real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI/provider-auth capability. Source inspection confirms current main and v2026.5.19 expose `--profile-id` on `paste-token` only and have no login-time target profile contract.

## Solution Assessment

No as currently proposed: the direction is useful, but post-return profile remapping is not the safest provider-auth contract. The maintainable path is to pass a typed target into provider auth before credentials/config are created, or reject methods that cannot safely honor it.
