## Summary

Keep open: current main still does not provide a login-time profile target, but this PR is not merge-ready because it rewrites returned profiles after provider auth has already run and lacks real behavior proof.

## What This Changes

The PR adds `--profile-id` plumbing for `models auth login`, remaps single-profile auth results, adds focused coverage, and documents named OAuth login profiles.

## Best Possible Solution

Implement login-time profile targeting as a provider-auth contract: pass a typed target into provider auth before credentials/config are created, reject unsupported or ambiguous methods, and update docs/tests around that behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI/provider-auth capability. Source inspection confirms current main still lacks `--profile-id` on `models auth login`, while related auth-profile surfaces exist elsewhere.

## Solution Assessment

No as currently proposed: the direction is useful, but post-return profile remapping is not the narrowest maintainable provider-auth contract. The safer path is to forward the target profile before provider auth runs, or reject methods that cannot honor it.
