## What This Changes

This PR adds a `--profile-id` option to `models auth login`, remaps single-profile auth results to that ID, adds a helper test, and documents named OAuth profile login flows.

## Best Possible Solution

Land or replace this with a rebased provider-auth contract change that passes a typed target into supported auth methods before credentials/config are produced, rejects ambiguous or unsupported methods clearly, and includes docs, tests, and real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI/provider-auth capability. Source inspection confirms current main and v2026.5.19 expose `--profile-id` on `paste-token` only and have no login-time target profile contract.

## Solution Assessment

No as currently proposed: the direction is useful, but post-return profile remapping is not the safest provider-auth contract. The maintainable path is to pass a typed target into provider auth before credentials/config are created, or reject methods that cannot safely honor it.
