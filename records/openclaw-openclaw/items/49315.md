## Summary

Keep open: current main still lacks the requested `models auth login --profile-id` capability, and the canonical feature request remains open, but this PR cannot merge as-is because the branch is conflicting/outdated, lacks contributor real behavior proof, and its branch-head remap runs after provider auth side effects.

## What This Changes

The PR adds a `--profile-id` option to `models auth login`, remaps single-profile auth results before persistence, adds focused coverage, and documents named OAuth profile login flows.

## Best Possible Solution

Land a rebased implementation that treats `--profile-id` as a provider-safe auth target before side effects when possible, rejects unsupported ambiguous methods, preserves current auth-command behavior, updates docs/tests, and includes redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR for a new CLI flag. Source inspection confirms current main still lacks `--profile-id` on `models auth login`, while `paste-token` has its own `--profile-id` option.

## Solution Assessment

No as currently proposed: the direction is useful, but the PR head remaps after provider auth methods run and can leave side-effecting providers with the original profile already written. The safer shape is to pass the requested profile into providers before side effects or reject methods that cannot safely honor it.
