## Summary

Keep this PR open. Current main still lacks the requested named-profile login surface and the related tracker remains open, but the PR’s generic post-result profile rewrite is not safe enough to merge because provider-owned auth results can contain profile-specific config patches that would stay keyed to the original profile id.

## What This Changes

The PR adds `--profile-id` to `models auth login`, rewrites single-profile auth results to the requested id, adds a focused helper test, and updates CLI, gateway auth, FAQ, and OpenAI provider docs.

## Best Possible Solution

Keep the PR open for revision. The maintainable path is to add the login flag, thread the requested profile target through a typed provider auth context before providers construct profiles/config patches, have OpenAI Codex OAuth and device-code honor that target, validate provider ownership and collisions, preserve `--agent`, and cover CLI/parser, command persistence, provider config-patch, and docs behavior in tests.

## Reproduction Assessment

Yes. A source-level reproduction is enough: apply the PR’s `resolveLoginProfiles` approach to a current provider result like Microsoft Foundry’s, where `profiles[0].profileId` is changed after `configPatch.auth.order` has already been built from the original id.

## Solution Assessment

No. Rewriting only the returned profiles is not the best fix because provider-owned config patches and possible provider side effects remain tied to the original id; the safer solution is to pass the requested target into the provider auth method before result construction and persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread profile targets before providers build config:** `src/commands/models/auth.ts:294-297`
  - body: This remaps only the profiles passed to persistence after `method.run` has already built the full `ProviderAuthResult`. Current providers can embed the original profile id in `configPatch` (for example Microsoft Foundry builds `auth.order` from its provider-owned `profileId`), so `--profile-id microsoft-foundry:work` would persist the renamed profile while leaving config order entries for the original profile. Pass the requested profile into the provider auth context and let the provider build profiles and config patches consistently, or restrict the flag to methods whose result is profile-id independent.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but the auth-profile routing change can desynchronize credential selection metadata from the credential that was actually persisted.

Concerns:

- **[medium] Profile remap can desynchronize auth selection:** `src/commands/models/auth.ts:294`
  - body: Because the requested profile id is applied after provider result construction, provider-supplied config such as auth order can continue to reference the original profile id. That can make operators believe a named credential is controlling selection while stale provider-owned ids remain in routing metadata.
  - confidence: 0.82
