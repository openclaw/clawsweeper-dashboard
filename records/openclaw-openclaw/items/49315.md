## Summary

Keep this PR open: current main still lacks login-time `--profile-id`, the canonical issue remains open, and this branch is the active implementation candidate. It is not merge-ready because it is conflicting, lacks real behavior proof, and remaps profile IDs only after provider auth has already run.

## What This Changes

The PR adds `--profile-id` to `models auth login`, remaps a single returned auth profile to that requested ID, adds profile-remapping tests, and documents named OpenAI Codex login profiles.

## Best Possible Solution

Land or replace this with a rebased provider-auth contract change that passes a typed requested profile target into supported auth methods before credentials/config are produced, rejects unsupported or ambiguous methods clearly, updates docs/tests, and includes real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new CLI/provider-auth capability. Source inspection confirms current main lacks `--profile-id` on `models auth login`, while `paste-token` already has that naming surface.

## Solution Assessment

No as currently proposed: the direction is useful, but post-return profile remapping is not the safest provider-auth contract. The maintainable path is to pass a typed target into provider auth before credentials/config are created, or reject methods that cannot safely honor it.
