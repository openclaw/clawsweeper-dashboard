It should not merge yet because the PR body shows live pre-fix gateway logs but no after-fix real behavior proof from the PR branch.

## What This Changes

The PR guards the Control UI overview secondary `loadUsage(app)` call so it is skipped if the overview refresh is stale after the primary loaders finish.

## Best Possible Solution

Land the narrow stale-refresh guard after after-fix proof, ideally with a focused regression test that stale overview refreshes skip `loadUsage` while active overview refreshes still populate the cost card.

## Reproduction Assessment

Yes, by source inspection: start an overview refresh, have the primary loaders resolve after `host.tab` changes away from `overview`, and current main still enters the secondary array that calls `loadUsage`. I did not run a live browser/gateway repro in this read-only review.

## Solution Assessment

Yes, the pre-call guard is the narrow maintainable fix for the stale-refresh path. The PR should preserve active overview usage loading because the overview cost card uses `usageResult`.
