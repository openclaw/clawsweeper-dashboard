## Summary

Keep open. Current main still does not forward legacy iMessage/imsg tapback reactions to the agent: the monitor only handles ordinary message notifications, subscription only requests attachments, payload/config/schema surfaces have no reaction fields or opt-in, and inbound dispatch still builds a text-message context. The issue is also paired with open implementation work in #39322 and overlapping filtering work in #60274/#60677, so maintainer review should consolidate those rather than close this issue.

## What This Changes

The issue requests forwarding inbound iMessage tapback reactions to the agent as reaction-aware events or messages.

## Best Possible Solution

Keep this as the canonical legacy imsg tapback-to-agent request. Review, rebase, or supersede #39322 with a deliberate opt-in implementation that verifies the imsg reaction RPC contract, adds config/schema/docs, parses reaction metadata, routes reactions as a typed system event or accepted summary format, and reconciles #60274/#60677 so tapbacks do not accidentally trigger ordinary replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: legacy imsg only subscribes for attachments, only handles `message` notifications, and has no reaction payload/config path, so inbound tapbacks cannot be intentionally forwarded as reaction events.

## Solution Assessment

Unclear for the open implementation direction until maintainers choose the product behavior. Forwarding can be the right fix if it is opt-in and reaction-aware, but the safer end state must also preserve filtering for synthetic tapback text so reactions do not become unwanted normal prompts.
