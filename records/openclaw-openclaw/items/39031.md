## Summary

Keep open. Current main still does not forward legacy iMessage/imsg tapback reactions to the agent: the monitor only handles ordinary message notifications, subscribes only with attachment options, the parser and payload type have no reaction shape, and the inbound context path still builds text-message payloads. The issue is also paired with open implementation/filtering work in #39322, #60677, and #60274, so closing it before maintainer consolidation would be premature.

## What This Changes

The issue requests forwarding iMessage tapback reactions to the agent with target message, reaction, sender, and timestamp metadata.

## Best Possible Solution

Implement a deliberate, opt-in legacy iMessage reaction path in the iMessage plugin: verify the imsg reaction notification contract, parse reaction metadata, apply the same access-control/session/mention policy as inbound messages, surface allowed reactions as agent-visible system events or an approved typed event format, and document/configure the behavior with focused iMessage tests while reconciling the filtering bug reports.
