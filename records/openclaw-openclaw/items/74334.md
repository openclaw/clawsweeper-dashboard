## Summary

Keep open. Current main still has the reported memory-core mismatch: daily ingestion stores heading-prefixed/list-normalized snippets, while promotion relocation compares only whitespace-collapsed live windows and drops every selected candidate that cannot be relocated.

## What This Changes

The issue reports that `memory promote --apply` silently applies zero candidates when stored daily-ingestion snippets cannot be matched by `relocateCandidateRange`.

## Best Possible Solution

Share or faithfully mirror daily-ingestion snippet normalization during relocation, preserve the live-source stale-snippet guard, add a heading-plus-list apply regression, and log one clear warning when every selected candidate fails rehydration.

## Reproduction Assessment

Yes, at source level. Current main still stores heading/list-normalized daily snippets and compares them against whitespace-only live windows; a probe of the reported shape returns no relocation match.

## Solution Assessment

Yes. Aligning the relocation comparison with daily-ingestion normalization is the narrow maintainable fix; falling back blindly to stored candidates would restore counts but weaken the documented live daily-note re-read guard.

Fix the memory-core promotion apply mismatch reported in https://github.com/openclaw/openclaw/issues/74334. Current main stores daily-ingestion snippets with heading prefixes, stripped list markers, semicolon-joined list chunks, and truncation in `extensions/memory-core/src/dreaming-phases.ts`, but `relocateCandidateRange` in `extensions/memory-core/src/short-term-promotion.ts` compares only whitespace-normalized live windows and drops all unmatched candidates. Add regression coverage for a daily-ingested heading-plus-bullets candidate that dry-runs/ranks and then applies successfully, plus keep the existing stale-source test meaningful. Add a single useful warning when every selected candidate fails rehydration if the local logging surface supports it without broad API churn.
