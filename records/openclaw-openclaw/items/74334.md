## Summary

Keep open. Current main still has the mismatched daily-ingestion versus relocation normalization path, still drops candidates whose live range cannot be relocated, and the reporter’s v2026.5.2/v2026.5.3 follow-ups show the latest shipped artifacts they checked still carry the same `if (!relocated) continue` behavior.

## What This Changes

The issue asks memory-core promotion apply to relocate and append qualified daily-ingestion snippets instead of silently dropping them when stored snippet normalization differs from live markdown.

## Best Possible Solution

Share or faithfully mirror daily-ingestion snippet normalization in promotion relocation, preserve the documented live-source stale guard, add a heading-plus-list apply regression, and surface a diagnostic when all selected candidates fail rehydration.

## Reproduction Assessment

Yes, at source level. The current relocation logic returns null for a representative heading-prefixed stored snippet against the matching markdown heading/list window, and the apply loop then drops that candidate.

## Solution Assessment

Yes. Aligning relocation comparison with daily-ingestion normalization is the narrow maintainable fix; the stored-candidate fallback is useful as mitigation but weakens the documented live-note stale guard if used unconditionally.
