## Summary

Keep open. Current main still has the mismatched normalization and silent candidate-drop path, and the reporter’s v2026.5.3 follow-up says the same release artifact still carries the failing `if (!relocated) continue` behavior.

## What This Changes

The issue reports that memory-core promotion apply drops qualified dreaming candidates because stored recall snippets and live relocation windows use incompatible normalization.

## Best Possible Solution

Share or faithfully mirror daily-ingestion snippet normalization in promotion relocation, keep the live-source stale guard, add a heading-plus-list apply regression, and warn when every selected candidate fails rehydration.

## Reproduction Assessment

Yes, at source level. Current main stores heading/list-normalized daily snippets and compares them against whitespace-only live windows; a read-only probe of that exact shape produces no relocation match.

## Solution Assessment

Yes. Aligning relocation comparison with daily-ingestion normalization is the narrow maintainable fix; the reporter’s stored-candidate fallback is useful mitigation but weakens the documented live re-read guard.
