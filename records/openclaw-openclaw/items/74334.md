## Summary

Keep open. Current main still stores daily-ingestion snippets with heading/list normalization but relocates them with whitespace-only comparison, so qualified apply candidates can still be silently dropped before writing `MEMORY.md`.

## What This Changes

Fix memory-core promotion apply so stored daily snippets can be rehydrated against live daily notes when heading/list normalization differs.

## Best Possible Solution

Share or faithfully mirror daily-ingestion snippet normalization in promotion relocation while preserving the live-source stale/deleted guard, then add a regression for heading-plus-list apply promotion and a visible warning when all selected candidates fail rehydration.

## Reproduction Assessment

Yes, at source level. Current main builds heading-prefixed, semicolon-joined daily snippets but relocates with whitespace-only live windows; the read-only probe of that exact shape produced no relocation match.

## Solution Assessment

Yes. Aligning the relocation comparison with daily-ingestion normalization is the narrow maintainable fix; falling back to the stored candidate may restore writes but weakens the documented live re-read guard.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74334 in memory-core. Current main daily ingestion stores snippets through `normalizeDailyHeading`, `normalizeDailySnippet`, and `buildDailyChunkSnippet` in `extensions/memory-core/src/dreaming-phases.ts`, while promotion relocation in `extensions/memory-core/src/short-term-promotion.ts` compares `candidate.snippet` to live windows with whitespace-only `normalizeSnippet` and drops candidates when `relocateCandidateRange` returns null. Implement a narrow fix that lets `applyShortTermPromotions` rehydrate daily-ingestion heading/list chunks against the live source text without trusting stale stored snippets; preserve the documented skip for deleted/stale source. Add one visible warning/audit surface when every selected candidate fails rehydration if there is an existing logger path; otherwise keep the fix scoped. Check open PR #74051 before editing and fold/avoid overlap as maintainer direction allows. Add a single-line `CHANGELOG.md` fix entry because this is user-facing.
