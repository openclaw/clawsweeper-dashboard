## Summary

Keep open. Current main still stores heading/list-normalized daily snippets but relocates them with whitespace-only comparison, so selected promotion candidates can still be dropped before apply.

## What This Changes

The issue reports that memory-core promotion apply cannot rehydrate daily-ingestion snippets because stored snippets and live-source relocation windows use incompatible normalization.

## Best Possible Solution

Share or faithfully mirror daily-ingestion snippet normalization in promotion relocation, keep the live-source stale guard, add a heading-plus-list apply regression, and warn when every selected candidate fails rehydration.

## Reproduction Assessment

Yes, at source level. Current main can be driven with a heading-prefixed daily-ingestion snippet and matching markdown list window, and the current relocation logic returns null.

## Solution Assessment

Yes for the structural direction: align relocation comparison with daily-ingestion normalization while preserving live re-read behavior. The reporter’s stored-candidate fallback is a useful mitigation but is not the best durable fix because it weakens stale-source protection.

## Repair Work Prompt

Fix the memory-core promotion apply rehydration mismatch reported in openclaw/openclaw#74334. Current main stores daily-ingestion chunks with heading prefixes, stripped list markers, `; ` list joins, and truncation, but `relocateCandidateRange` compares those stored snippets against whitespace-only live windows and drops every selected candidate when relocation returns null. Implement the fix in the memory-core owner surface, likely by sharing or mirroring daily-ingestion snippet normalization for relocation comparison, while preserving the documented live daily-note re-read stale guard. Do not solve this by blindly accepting stale stored candidates as the primary path. Add/update regression coverage showing a heading-prefixed daily-ingestion list chunk can be applied and rehydrated from the live note, plus coverage or behavior for warning/diagnostics when every selected candidate fails rehydration. Check open PR #74051 before starting so the repair does not duplicate an existing fix; if #74051 already fully solves this issue, report that instead of creating a second implementation. Add a user-facing CHANGELOG.md fix entry unless the final diff is test-only.
