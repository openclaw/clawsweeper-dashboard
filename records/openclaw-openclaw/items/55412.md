## Summary

Keep open. Current main still treats gateway drain as a terminal reply failure: it records `gateway_draining` and returns restart text to the user instead of retrying the run. The closed retry PR #55470 was not merged, and the broader restart/session-loss reports do not replace this narrow retry gap.

## What This Changes

The issue requests treating `GatewayDrainingError` during gateway restarts as a transient condition that retries rather than surfacing a user-visible agent failure.

## Best Possible Solution

Add a bounded gateway-drain retry at the agent-runner boundary, preferably keyed on the actual `GatewayDrainingError` type or resolved lifecycle error rather than only message text, with tests proving the first drain waits and retries without sending restart text while repeated drain failure exits predictably.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the current unit test constructs a `FallbackSummaryError` containing `GatewayDrainingError` and asserts user-visible restart text plus `gateway_draining`; command-queue tests also show `markGatewayDraining()` causes new enqueues to reject with that error.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if implemented as a bounded retry in the runner loop. A safer version should reuse the existing transient retry shape, guard against duplicate reply-operation finalization, and keep command-queue drain semantics unchanged.
