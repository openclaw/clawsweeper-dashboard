## Summary

Keep open. Current main still handles GatewayDrainingError as a terminal restart lifecycle reply: it marks the reply operation gateway_draining and sends restart text instead of retrying the agent run. The earlier retry PR is closed and unmerged, while the broader restart/session-loss reports do not replace this narrow runner-boundary retry gap.

## What This Changes

The issue requests treating GatewayDrainingError during gateway restart drain windows as transient so the agent run retries instead of surfacing user-visible failure text.

## Best Possible Solution

Add a bounded gateway-drain retry at the agent-runner boundary, keyed on the actual GatewayDrainingError or resolved restart lifecycle error, and keep command-queue drain semantics unchanged. Tests should prove the first drain error waits and retries without user-visible restart text, while repeated drain failure exits predictably.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current tests explicitly assert that fallback exhaustion wrapping GatewayDrainingError returns restart text and records gateway_draining, and command-queue tests assert markGatewayDraining causes new enqueues to reject with GatewayDrainingError.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if implemented as a bounded retry in the existing run loop rather than broad message-string matching; durable post-restart replay should remain a separate broader recovery track.
