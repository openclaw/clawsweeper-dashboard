## Summary

Keep open: current main still converts GatewayDrainingError into terminal restart lifecycle text, and the linked retry PR is closed unmerged, so the narrow runner-boundary retry gap remains.

## What This Changes

The issue asks in-flight agent runs that hit GatewayDrainingError during gateway restart drain windows to wait and retry instead of returning a user-visible failure.

## Best Possible Solution

Add a bounded GatewayDrainingError retry at the agent-runner boundary, preserve command-queue drain rejection semantics, and update focused regression coverage so the first drain error retries while repeated drain failure exits predictably.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current code rejects enqueues during gateway drain, and current runner tests assert that a fallback-wrapped GatewayDrainingError returns restart text instead of retrying.

## Solution Assessment

Yes, with a narrower implementation than message-string matching. The maintainable fix is a one-shot or otherwise bounded retry around the existing run loop for the concrete restart lifecycle error, leaving durable post-restart replay to broader recovery issues.
