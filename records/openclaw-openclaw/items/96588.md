## Summary

This issue should stay open: current main and the latest release still use lexicographic ordering in the model resolver, and several open PRs already target the fix but none has merged.

## What This Changes

The issue requests numeric-aware ordering for model alias resolution so double-digit version suffixes do not select an older model.

## Best Possible Solution

Land one narrow candidate PR that applies numeric-aware ordering at both resolver sort sites with focused regression coverage, then close this issue and retire duplicate PRs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to resolve `parseModelPattern("opus", [claude-opus-4-9, claude-opus-4-10])`; current main still orders those candidates with plain lexicographic `localeCompare`.

## Solution Assessment

Yes. Numeric-aware comparison at the existing `tryMatchModel` sort point is the narrowest maintainable fix; exact provider/model matching and broader provider policy can remain unchanged.
