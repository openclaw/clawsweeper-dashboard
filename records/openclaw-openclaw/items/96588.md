## Label Justifications

- `P2`: This is a normal-priority model-selection bug with a narrow fix and limited blast radius until provider catalogs reach double-digit version suffixes.
- `impact:auth-provider`: The affected path resolves provider/model choices from aliases and partial model patterns, which can silently select an older concrete model id.

## Summary

Keep this open: current main and the latest release still sort matched model ids with plain lexicographic `localeCompare`, and the linked fix PRs are open candidates rather than merged fixes.

## What This Changes

The issue requests numeric-aware model alias resolution so aliases like `opus` select the newest double-digit version suffix instead of the lexicographically highest id.

## Best Possible Solution

Land one focused PR that adds numeric-aware ordering to both resolver comparator branches and regression coverage for double-digit alias selection.

## Reproduction Assessment

Yes, at source level. Resolving `parseModelPattern("opus", [claude-opus-4-9, claude-opus-4-10])` reaches the current `tryMatchModel` alias sort, where plain `localeCompare` ranks `4-9` above `4-10`.

## Solution Assessment

Yes. Numeric-aware collation at the existing resolver ordering point, plus a focused regression test, is the narrowest maintainable fix; the next step is reviewing and landing one of the linked PRs.
