## Summary

Keep open. Current main still supports remote embedding endpoints and local/QMD/MMR ranking only; no current schema, resolved config, runtime HTTP contract, builtin search path, QMD bridge, or plugin reranker API implements a Cohere-style remote reranker.

## What This Changes

The issue requests a configurable remote reranker endpoint for memory search, including Cohere-compatible HTTP calls and builtin/QMD fallback behavior.

## Best Possible Solution

Design an additive remote reranker contract with typed config/schema/help/docs, SecretInput-aware credentials, guarded HTTP, builtin fallback semantics, and an explicit QMD or plugin-provider strategy.

## Reproduction Assessment

Yes, as a source-level capability gap. Current main has strict memorySearch config and runtime paths for embeddings/local reranking only, with no /v1/rerank call path or reranker provider contract.

## Solution Assessment

Unclear. A Cohere-compatible endpoint is plausible, but the maintainable fix depends on product and security decisions about private memory egress, QMD ownership, and core config versus a plugin-facing reranker contract.

## Risks / Open Questions

- Remote reranking would send private memory snippets to operator-configured HTTP endpoints, so SSRF, private-network allowance, proxy behavior, credentials, diagnostics, and redaction policy need design before implementation.
- QMD support may require an explicit OpenClaw/QMD ownership boundary rather than only adding OpenClaw config wiring.
- A plugin-only implementation is not straightforward today because the public plugin contract exposes memory embedding providers, not memory reranker providers.
