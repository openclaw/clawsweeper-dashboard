## Summary

Keep open. Current main still supports remote memory embeddings and local/QMD ranking paths, but it does not implement a Cohere-style remote reranker config, schema, HTTP client, builtin integration, QMD integration, or plugin-facing reranker provider contract; this remains a valid feature request that needs maintainer design and security review.

## What This Changes

Add support for operator-configured remote reranker API endpoints in OpenClaw memory search, including builtin and QMD behavior, timeout/batching policy, and vector-only fallback.

## Best Possible Solution

Add an additive remote reranker design for memory search with typed config/schema/help/docs, SecretRef-aware credentials, guarded HTTP for a Cohere-compatible contract, builtin post-retrieval reranking with vector/hybrid fallback on failure, and a clear QMD strategy; if maintainers prefer plugin ownership, first expose a documented reranker provider API.

## Reproduction Assessment

Yes, as a source-level capability gap. The proposed reranker config shape is absent from the strict schema, and current memory search code has no `/v1/rerank` call path to exercise.

## Solution Assessment

Unclear. The requested Cohere-compatible endpoint shape is plausible, but the best implementation depends on maintainer decisions about remote-snippet security policy, QMD ownership, and whether rerankers belong in core config or a new plugin-facing provider contract.
