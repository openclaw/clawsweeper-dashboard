## Summary

Keep open. Current main still supports remote memory embeddings and local/QMD ranking paths, but it does not expose a Cohere-style remote reranker config, schema, HTTP client, builtin integration, QMD integration, or plugin-facing reranker provider contract.

## What This Changes

Request adding remote reranker endpoint support for memory search so builtin and QMD paths can use Cohere-compatible rerank services instead of only local reranking.

## Best Possible Solution

Add an additive remote reranker design for memory search: typed config/schema/help/docs, SecretRef-aware credentials, guarded HTTP for a Cohere-compatible contract, builtin post-retrieval reranking with vector/hybrid fallback on failure, and a clear QMD strategy. If maintainers prefer plugin ownership, first expose a documented memory reranker provider API so Qwen/Cohere/Jina adapters can live outside core without replacing the whole memory backend.

## Reproduction Assessment

Yes. The missing feature has a high-confidence source-level reproduction path: the proposed `memorySearch.query.hybrid.rerank.remote` config is absent from the strict schema, and neither builtin nor QMD search consumes a `/v1/rerank` endpoint.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation depends on maintainer decisions about the remote-snippet security policy, QMD integration, and whether this belongs in core memory config or a new plugin-facing reranker contract.
