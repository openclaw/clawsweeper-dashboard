## Summary

Keep open. Current `main` still supports remote memory embeddings and local/MMR/temporal ranking paths, but it has no typed config, schema, HTTP client, builtin integration, QMD integration, or plugin-facing contract for a Cohere-style remote reranker endpoint.

## What This Changes

Request support for configuring remote reranker API endpoints for memory search, including builtin and QMD backends with fallback, batching, and timeout behavior.

## Best Possible Solution

Add an additive remote reranker design with typed config/schema/help/docs, SecretInput-aware credentials, guarded HTTP for a Cohere-compatible contract, builtin post-retrieval reranking with vector/hybrid fallback, and an explicit QMD or plugin-provider strategy.

## Reproduction Assessment

Yes, as a source-level capability gap. The proposed config shape is absent from the strict schema, and current memory search has no `/v1/rerank` call path to exercise.

## Solution Assessment

Unclear. The requested Cohere-style endpoint shape is plausible, but the best implementation depends on maintainer decisions about remote-snippet security policy, QMD ownership, and whether rerankers belong in core config or a new plugin-facing provider contract.

## Risks / Open Questions

- Remote reranking would send private memory snippets to operator-configured HTTP endpoints, so SSRF, private-network allowance, proxy behavior, credentials, and redaction policy need explicit design.
- QMD support may require upstream QMD behavior or a clear OpenClaw/QMD ownership boundary rather than only OpenClaw config wiring.
- A plugin-only implementation is not currently straightforward because the documented plugin contract covers memory embedding providers, not memory reranker providers.
