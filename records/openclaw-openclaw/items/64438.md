## Summary

Keep open. Current main still has remote embedding support and local/QMD ranking paths, but no Cohere-style remote reranker config, schema, runtime HTTP client, builtin integration, QMD integration, or plugin-facing reranker provider contract; the related Memory v2 PR is local sidecar scoring rather than the requested remote endpoint.

## What This Changes

Request support for configuring Cohere-style remote reranker endpoints for memory search, including builtin and QMD backends, batching, timeout, and fallback behavior.

## Best Possible Solution

Add an additive, SecretRef-aware remote reranker design with typed config/schema/help/docs, guarded Cohere-compatible HTTP, builtin post-retrieval fallback behavior, and a clear QMD/plugin provider boundary.

## Reproduction Assessment

Yes, as a source-level capability gap: the strict schema rejects the proposed `memorySearch.query.hybrid.rerank.remote` shape and current memory code has no `/v1/rerank` call path to exercise. I did not run a live reranker because there is no current config or runtime integration for it.

## Solution Assessment

Unclear. A Cohere-compatible endpoint is plausible, but the best fix needs maintainer decisions about core config versus a plugin-facing reranker seam, private-memory outbound policy, and QMD ownership.
