## Summary

Keep open. Current main still has remote embedding support and local/QMD ranking paths, but no typed config, runtime HTTP client, builtin integration, QMD integration, or plugin contract for a Cohere-style remote reranker endpoint.

## What This Changes

The issue requests remote Cohere-compatible reranker endpoint support for memory search, including builtin and QMD backends.

## Best Possible Solution

Design an additive remote reranker surface with typed config/schema/help/docs, SecretInput-aware credentials, guarded HTTP for a Cohere-compatible contract, builtin post-retrieval reranking with fallback, and an explicit QMD or plugin-provider strategy.

## Reproduction Assessment

Yes, as a source-level capability gap. The strict memorySearch schema and resolved runtime config have no reranker block, and the current search/remote HTTP paths contain no `/v1/rerank` call path.

## Solution Assessment

Unclear. The requested Cohere-style endpoint is plausible, but the maintainable fix depends on product and security decisions about remote memory snippets, QMD ownership, and whether this belongs in core config or a plugin-facing reranker contract.

## Risks / Open Questions

- Remote reranking would send private memory snippets to operator-configured HTTP endpoints, so SSRF, private-network allowance, proxy behavior, credential handling, and redaction policy need explicit design.
- QMD support may require upstream QMD behavior or a clear OpenClaw/QMD ownership boundary rather than only OpenClaw config wiring.
- A plugin-only implementation is not currently straightforward because the documented plugin contract exposes memory embedding providers, not memory reranker providers.
