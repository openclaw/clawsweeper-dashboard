## Summary

Keep open. Current main still has remote embedding support and local/MMR/QMD reranking only; there is no schema, resolved config, HTTP contract, QMD bridge, or plugin reranker API for a Cohere-style remote reranker endpoint.

## What This Changes

Add opt-in memory-search support for Cohere-compatible remote reranker endpoints, including config, fallback, batching or timeout behavior, and builtin/QMD integration.

## Best Possible Solution

Design an additive, default-off remote reranker contract with typed config/schema/help/docs, SecretInput-aware credentials, guarded HTTP, fallback semantics, and an explicit builtin/QMD/plugin ownership strategy.

## Reproduction Assessment

Yes, as a source-level capability gap. Current main has strict memorySearch config and runtime provider contracts for embeddings/local reranking only, with no /v1/rerank call path or reranker provider contract.

## Solution Assessment

Unclear. A Cohere-compatible endpoint is plausible, but the maintainable fix needs product and security decisions about private memory egress, SecretInput/SSRF policy, fallback behavior, and whether QMD support belongs in core or behind a plugin/QMD seam.

## Risks / Open Questions

- Remote reranking would send query text and candidate memory snippets to operator-configured HTTP endpoints, so SSRF, private-network allowance, proxy behavior, credential handling, diagnostics, and redaction policy need design first.
- QMD support may require an explicit OpenClaw/QMD ownership boundary rather than only adding OpenClaw config wiring.
- A plugin-only implementation is not straightforward today because the public contract exposes memory embedding providers, not memory reranker providers.
