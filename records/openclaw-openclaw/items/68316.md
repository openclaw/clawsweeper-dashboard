## Summary

Close. Current main already ships a more precise fix: after successful block streaming, text-only final payloads stay suppressed, unsent media final payloads are preserved, already-streamed text is removed from media finals, and block-sent media URLs are deduped. The proof commit bb97f193963bf6e611f8c206b7c5c0da947e07c2 is included in v2026.4.29, and this PR branch's simpler filter would be worse because it can resend streamed text as a media caption.

## What This Changes

The PR changes `buildReplyPayloads` and its tests so successful block streaming continues to suppress text-only final payloads but retains media/error final payloads that were not marked as already sent.

## Best Possible Solution

Keep the shipped v2026.4.29/current-main implementation: preserve unsent generated media after block streaming, strip already-streamed text from mixed media finals, and dedupe media URLs that the block pipeline already delivered.

## Reproduction Assessment

Yes. The focused reproduction is `buildReplyPayloads` with block streaming enabled, `didStream()` true, not aborted, and a final text+media payload; current tests assert text-only finals drop, unsent media is preserved with text cleared, and already-sent media drops.

## Solution Assessment

No for the PR branch as-is. Current main is the better solution because it preserves the missing media without re-sending already-streamed text and uses `getSentMediaUrls()` to suppress duplicate attachments.
