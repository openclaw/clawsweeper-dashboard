## Summary

Keep open. The feature is not implemented on current main, the older similar PRs I found are closed and unmerged, and this PR still has a blocking correctness issue in agent-id matching plus insufficient inspectable real behavior proof.

## What This Changes

The PR threads an optional `agentId` through model visibility helpers, gateway `models.list`, Control UI and slash-command model loaders, adds protocol/docs/tests/generated Swift updates, and includes a Discord test narrowing after rebase.

## Best Possible Solution

Keep the feature direction, normalize agent-id matching through the existing helper in both the gateway and model-selection helper paths, add a mixed-case regression test, and include inspectable after-fix proof for the real UI or gateway behavior.

## Reproduction Assessment

Yes for the review finding: the PR branch source exact-compares agent ids while current session parsing and agent config resolution normalize them. A config entry like `id: "Writer"` addressed by normalized `agentId: "writer"` will fall back to defaults by source inspection.

## Solution Assessment

No. Threading `agentId` through the catalog path is the right shape, but the implementation needs to use the existing normalized agent-id contract and the PR needs inspectable real behavior proof before merge.

## Review Findings

- **[P2] Normalize agent IDs before matching allowlists:** `src/agents/model-selection-shared.ts:60-61`
