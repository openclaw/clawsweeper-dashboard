## Summary

Keep open: this is a plausible feature PR, but the patch has blocking correctness gaps around legacy `main` slash-command scoping and normalized agent-id matching, and the external-PR real behavior proof is not inspectable.

## What This Changes

This PR adds per-agent model visibility allowlists using `agents.list[].models` for Control UI model loading, slash commands, and `models.list { agentId }`, with docs and tests.

## Best Possible Solution

Land the feature after reusing the existing normalized agent/session resolution paths for all model-list callers, adding regressions for legacy `main` and normalized agent ids, and attaching redacted real behavior proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the current slash-command resolver treats bare `main` as `DEFAULT_AGENT_ID`, and current agent config lookup normalizes ids, while the PR's new paths do not preserve both behaviors. The requested per-agent allowlist itself is a new feature, so there is no current-main bug reproduction.

## Solution Assessment

No. The direction is maintainable, but this patch should first reuse existing agent-id normalization and legacy session-key handling rather than adding raw string matching and `parseAgentSessionKey`-only scoping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve legacy main scoping for slash commands:** `ui/src/ui/chat/slash-command-executor.ts:775-777`
  - body: For `/model` and `/think` invoked with the supported legacy `main` session key, `parseAgentSessionKey("main")` returns null even though `resolveCurrentSession` maps that same key to `DEFAULT_AGENT_ID`. The new request then omits `agentId`, so a default agent with `agents.list[].models` still sees the global allowlist. Derive the agent id with the same legacy-main fallback used by session resolution and add a regression for bare `main`.
  - confidence: 0.9
- **[P2] Normalize agent ids before matching allowlists:** `src/gateway/server-methods/models.ts:46-47`
  - body: OpenClaw matches configured agents through `normalizeAgentId`, but the new `models.list` gate compares trimmed strings directly. A configured id such as `Writer` remains valid for other per-agent settings, yet `models.list { agentId: "writer" }` is treated as unknown and falls back to defaults. Use the shared normalization path for the known-agent check and the per-agent models lookup.
  - confidence: 0.86
