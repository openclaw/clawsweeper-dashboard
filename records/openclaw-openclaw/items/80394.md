## Summary

Keep open: this is a plausible feature PR with supplied UI proof, but the current patch still leaks a fallback/default model into an agent-scoped allowlist, contradicting the stated per-agent visibility behavior.

## What This Changes

The PR adds per-agent model visibility scoping for `agents.list[].models`, threads optional `agentId` through `models.list`, the Control UI and slash command model loaders, updates protocol schema/Swift bindings, and adds tests/docs.

## Best Possible Solution

Keep the per-agent visible catalog strict to the configured `agents.list[].models` entries unless the agent explicitly opts into additional fallback models, and document the new `models.list.agentId` contract in protocol docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source review confirms current main is globally scoped, and the PR proof/source show the proposed per-agent feature path.

## Solution Assessment

No. The implementation direction is reasonable, but the current fallback handling makes a scoped picker show a model outside the agent's configured allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep scoped allowlists from leaking fallback models:** `src/agents/model-selection.ts:387-390`
  - body: Once `agentId` selects a non-empty `agents.list[].models` record, this path still feeds the normal fallback chain into `buildAllowedModelSetWithFallbacks`; the shared builder then appends those fallback/default models to the visible set. For an agent that only configures two local models but inherits the global default, the picker still shows the global `gpt-5.5` option, which the PR's own El Becario screenshot demonstrates and which contradicts the advertised scoped allowlist.
  - confidence: 0.88
- **[P3] Update the models.list protocol docs:** `src/gateway/protocol/schema/agents-models-skills.ts:203-208`
  - body: Adding `agentId` to `ModelsListParamsSchema` changes the public Gateway contract, but `docs/gateway/protocol.md` still says `models.list` only accepts `view` and describes configured results as driven by `agents.defaults.models`. Please document the optional `agentId` parameter and its fallback semantics alongside the schema change.
  - confidence: 0.78
