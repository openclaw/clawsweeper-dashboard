## Summary

Keep open: current main and the latest release still use embedded fallback rather than reattach-first recovery for generic gateway disconnects, and the requested behavior is a compatibility-sensitive product decision. The linked open PR covers a narrower fresh-session fallback repair, not the policy choice here.

## What This Changes

Request reattach-first recovery for accepted `openclaw agent` gateway CLI runs after disconnect before falling back to local embedded re-execution.

## Best Possible Solution

Have maintainers choose recook-first, reattach-first, or a mode-specific hybrid, then implement the chosen CLI fallback policy with focused tests and live accepted-disconnect proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does confirm current main records accepted run context and still routes generic gateway failures to embedded fallback without a reattach-first wait path.

## Solution Assessment

Unclear until maintainers choose the policy. Reattach-first is a plausible bounded design using existing gateway primitives, but a `--deliver`-only hybrid or a future result-by-runId RPC may be safer than changing all interactive disconnect behavior at once.
