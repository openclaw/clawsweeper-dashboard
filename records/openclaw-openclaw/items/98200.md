## Summary

Keep open: current main still uses embedded fallback for non-timeout gateway transport failures after an accepted run, while the requested reattach-first behavior is a compatibility-sensitive product choice rather than an already-implemented fix.

## What This Changes

Request reattach-first recovery for accepted `openclaw agent` gateway runs after CLI WebSocket loss, falling back to local embedded execution only if wait/result recovery fails.

## Best Possible Solution

Have maintainers choose the disconnect policy first, then implement the chosen layered recovery in `src/commands/agent-via-gateway.ts` with focused fallback tests and redacted live gateway disconnect proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does show the current-main path: accepted gateway state is recorded, generic transport fallback runs embedded with the original local options, and no reattach-first wait path exists.

## Solution Assessment

Unclear until maintainers choose the policy. Reattach-first is a plausible bounded design using existing RPCs, but a `--deliver`-only hybrid or a future result-by-runId RPC may be safer than changing all interactive disconnect behavior at once.
