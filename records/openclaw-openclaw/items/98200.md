## Summary

Keep open: current main and v2026.6.11 still use embedded fallback rather than reattach-first recovery for accepted gateway-run disconnects, and the requested behavior is a compatibility-sensitive CLI/gateway product decision rather than an already-implemented fix.

## What This Changes

This issue requests a reattach-first recovery policy for accepted `openclaw agent` gateway runs when the CLI WebSocket disconnects before the final frame.

## Best Possible Solution

Maintainers should choose recook-first, reattach-first, or a mode-specific hybrid, then implement that CLI fallback policy with focused tests and controlled accepted-disconnect proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection confirms current main records accepted run context and still routes gateway failure to embedded fallback without a reattach-first wait path, but I did not run a live gateway/proxy disconnect scenario.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the policy; reattach-first is plausible with existing gateway primitives, but a `--deliver`-only hybrid or future result-by-runId RPC may be safer than changing all interactive disconnect behavior at once.
