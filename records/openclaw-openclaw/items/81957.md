## Summary

Keep this PR open: the security hardening direction and supplied terminal proof look useful, but the latest head still has two source-reproducible test harness bugs that make the added regression coverage fail before merge.

## What This Changes

The PR tightens high-risk GitHub Actions release, publish, live/e2e, and build-artifact workflows, isolates plugin npm/ClawHub publish auth, pins external Actions, and adds plugin npm token-boundary coverage.

## Best Possible Solution

Keep the hardening approach, repair the focused token-boundary test harness, and leave live release/publish validation to controlled maintainer workflows.

## Reproduction Assessment

Yes. Source inspection of the latest PR head shows the fake npm record is written differently than the success assertions expect, and the missing-auth case reads a record file that is never created; I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No. The workflow/script hardening direction is maintainable, but the PR is not merge-ready until the test harness uses deterministic record writing and handles the no-publish case without reading a nonexistent file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the fake publish record with printf:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:35`
  - body: The fake npm script is generated from a TypeScript template literal, so `\n` becomes a real newline inside the shell-quoted `echo` argument and `echo` appends another newline. The success cases then write two newlines while the tests expect one; use `printf '%s\n' ...` or escape the backslash in the generated script.
  - confidence: 0.92
- **[P2] Avoid reading a missing no-publish record:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:114`
  - body: In the missing mirror-auth case, fake `npm view` exits without writing the record and the script exits before any publish call can create it. `readFileSync(record, "utf8")` will throw `ENOENT`, so this test fails before proving publish was not called; create an empty record first or assert that the file is absent.
  - confidence: 0.93
