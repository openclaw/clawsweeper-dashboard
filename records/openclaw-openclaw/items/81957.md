## Summary

Keep open: the supply-chain hardening direction and refreshed current-head proof look sound, but the new token-boundary Vitest file has two source-level harness defects that would make the focused regression suite fail.

## What This Changes

The PR tightens high-risk GitHub Actions permissions and trusted refs, pins external actions, isolates plugin npm/ClawHub publish auth around OIDC, hardens cache/artifact boundaries, and adds plugin npm token-boundary coverage.

## Best Possible Solution

Keep the hardening approach, repair the focused test harness so it reliably proves the token boundary, and leave live publishing to controlled release workflow validation.

## Reproduction Assessment

Yes. By source inspection, the added Vitest harness writes a different record than it expects in publish cases and reads a missing record in the no-publish case; I did not run the test because this review must keep the checkout read-only.

## Solution Assessment

No. The PR is not merge-ready until the test harness uses a deterministic record write and handles the no-publish case without reading a nonexistent file; the workflow/script hardening direction itself looks maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the fake publish record with printf:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:35`
  - body: The fake npm script is generated from a TypeScript template literal, so `\n` becomes a real newline inside the shell-quoted `echo` argument. Bash then appends another newline, making the record contain two newlines while the tests expect one. Use `printf '%s\n' ...` or escape the backslash in the generated script so the success cases can pass.
  - confidence: 0.92
- **[P2] Avoid reading a missing no-publish record:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:114`
  - body: In the missing mirror-auth case, fake `npm view` exits without writing the record and the script exits before any publish call can create it. `readFileSync(record, "utf8")` will throw `ENOENT`, so this test fails before proving that publish was not called. Create an empty record up front or assert that the file is absent.
  - confidence: 0.93
