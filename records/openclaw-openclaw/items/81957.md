## Summary

Keep open: the updated PR body now supplies terminal proof for the workflow predicate checks, but the patch still leaves a long-lived npm token in the plugin package publish subprocess, so the supply-chain boundary is not ready to merge.

## What This Changes

The PR modifies eight GitHub Actions workflows to tighten release/publish permissions and trusted refs, disable PR executable cache reuse in Testbox, remove the ClawHub token fallback, require OIDC plugin npm publishing, and pin high-risk external actions to SHAs.

## Best Possible Solution

Keep the workflow hardening direction, but run plugin package publish with a sanitized child environment and expose the long-lived npm token only to the dist-tag mirroring path.

## Reproduction Assessment

Yes for the blocking finding: PR head sets NPM_TOKEN in the trusted publish step, and current main passes process.env into the npm publish subprocess. The broader release/publish hardening has not been reproduced by a live publish run.

## Solution Assessment

No. OIDC trusted publishing is the right direction, but this implementation must remove the long-lived token from package publish/lifecycle scope and inject it only for npm dist-tag mirroring.
