## Best Possible Solution

Land the hardening only after the token-boundary fixture passes on the live head, proof is refreshed for that head, and release/security owners accept the stricter publish gates while tracking the remaining dist-tag token redesign separately.

## Solution Assessment

The hardening direction is maintainable because it narrows permissions, pins high-risk Actions, and separates OIDC package publish from npm-token dist-tag mirroring.
