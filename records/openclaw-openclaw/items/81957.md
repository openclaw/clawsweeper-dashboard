## Summary

Keep open: the hardening direction is useful, but the latest PR head still leaves a long-lived npm token in the trusted-publishing package publish step while the current publish wrapper passes inherited environment variables through to npm.

## What This Changes

The PR modifies eight GitHub Actions workflows to tighten release/publish permissions and trusted refs, disable PR executable cache reuse in Testbox, remove ClawHub token fallback, require OIDC plugin package publishing, and pin high-risk external actions to SHAs.

## Best Possible Solution

Keep the workflow hardening, but sanitize the package publish environment so trusted publishing receives no long-lived npm token and inject the token only into the dist-tag mirroring command, then route the corrected branch through release/security review.

## Reproduction Assessment

Yes for the blocking finding: source inspection shows the PR head exports NPM_TOKEN in the trusted-publishing step and current main passes process.env into the npm publish subprocess. The broader release/publish hardening was not reproduced with a live publish run.

## Solution Assessment

No. OIDC trusted publishing is the right direction, but this implementation must remove the long-lived token from package publish/lifecycle scope and expose it only to the dist-tag mirroring path.
