## Summary

Keep open: the workflow hardening is useful and the PR now has terminal proof, but PR head still exports `NPM_TOKEN` in the same trusted-publishing step while the current publish wrapper passes `process.env` into `npm publish`, so the long-lived token remains visible to package publish and lifecycle code.

## What This Changes

The PR edits eight GitHub Actions workflows to pin actions, tighten permissions and ref gates, disable PR build-cache restore/save, and move plugin publishing toward OIDC trusted publishing.

## Best Possible Solution

Keep the workflow hardening direction, but sanitize the package publish subprocess environment and expose the npm token only to the later dist-tag mirroring command.

## Reproduction Assessment

Yes for the blocking review finding: PR head exports `NPM_TOKEN` in the trusted-publishing publish step, and current main passes `process.env` into the `npm publish` subprocess. I did not run live publish because it would mutate external release state.

## Solution Assessment

No. OIDC trusted publishing is the right direction, but this implementation must remove the long-lived npm token from package publish scope and inject it only for dist-tag mirroring.
