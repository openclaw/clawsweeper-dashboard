## Summary

Keep open: the workflow hardening scope is useful and the PR now includes sufficient terminal proof for the non-side-effecting checks, but current head still exposes the long-lived npm token to the trusted-publishing package publish subprocess, which is a blocking security regression.

## What This Changes

The PR edits eight GitHub Actions workflows to tighten release/publish permissions and trusted refs, disable PR executable cache reuse, remove the ClawHub token fallback, switch plugin npm package publish toward OIDC trusted publishing, and pin high-risk external Actions to SHAs.

## Best Possible Solution

Keep the workflow hardening, but ensure package publish runs with OIDC trusted publishing and no NPM_TOKEN in its environment, then inject the token only for stable dist-tag mirroring.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows PR head exports NPM_TOKEN in the trusted-publishing step, and current main passes process.env into the npm publish subprocess. I did not run live publish because it would mutate external npm release state.

## Solution Assessment

No. OIDC trusted publishing is the right direction, but this implementation must remove the long-lived npm token from package publish scope and pass it only to the dist-tag mirror path that needs it.
