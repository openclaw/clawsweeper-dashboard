## Summary

Keep open: the hardening direction is useful and the PR includes terminal proof, but the latest head still leaves the long-lived npm token in the trusted-publishing package publish environment.

## What This Changes

The PR edits eight GitHub Actions workflow files to tighten release/publish permissions and trusted refs, disable PR dist cache reuse, remove the ClawHub token fallback, require plugin npm trusted publishing, and pin external actions to SHAs.

## Best Possible Solution

Keep the workflow hardening scope, but sanitize the plugin package publish subprocess environment and expose the npm token only to the later dist-tag mirroring command.

## Reproduction Assessment

Yes for the blocking review finding: PR head exports NPM_TOKEN in the trusted publish step, and current main passes process.env into the npm publish subprocess. I did not run a live publish because it would mutate external release state.

## Solution Assessment

No. OIDC trusted publishing is the right direction, but this implementation must remove the long-lived npm token from package publish scope and inject it only for dist-tag mirroring.
