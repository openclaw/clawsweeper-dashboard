## Summary

Keep open: the hardening direction is useful, but the patch is not ready because plugin npm publish still exports the long-lived npm token into the publish subprocess and the external PR lacks required real behavior proof.

## What This Changes

This PR modifies eight GitHub Actions workflows to tighten permissions/ref gates, pin actions by SHA, change release/publish auth, and adjust build cache/artifact handling.

## Best Possible Solution

Keep the scoped workflow hardening, but make plugin npm publish run with long-lived npm tokens absent from the publish child environment and expose token auth only to dist-tag mirroring, then add real after-fix proof.

## Reproduction Assessment

Yes for the review finding: PR head exports NPM_TOKEN in the plugin npm Publish step, and current main passes process.env into the npm publish subprocess. The broader workflow hardening has not been reproduced in a real run from the PR evidence.

## Solution Assessment

No. The overall hardening direction is maintainable, but the npm token must be removed from the package publish subprocess and scoped only to dist-tag mirroring before this is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep NPM_TOKEN out of the publish subprocess:** `.github/workflows/plugin-npm-release.yml:247`
  - body: The trusted-publisher mode only clears the script's generated publish auth, but this step still exports `NPM_TOKEN` for the whole `plugin-npm-publish.sh` process. That script invokes `npm publish` through `plugin-npm-package-manifest.mjs`, which passes `process.env` to the child process, so the long-lived token remains visible during package publish/lifecycle instead of being limited to dist-tag mirroring. Please remove it from the publish child environment and inject it only for the later `npm dist-tag add` path.
  - confidence: 0.86
