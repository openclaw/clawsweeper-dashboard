## Summary

Keep open. The hardening direction is useful, but the current head introduces a release workflow regression by blocking the existing Tideclaw alpha ClawHub publish path, and the contributor proof still names an older head after later force-pushes.

## What This Changes

The PR pins high-risk workflow Actions, narrows release/publish workflow permissions and ref gates, splits plugin npm trusted publishing from token-backed dist-tag mirroring, and adds token-boundary regression tests.

## Best Possible Solution

Keep the hardening, align the ClawHub trusted-ref gate with the existing Tideclaw alpha release allowance, refresh proof for the exact current head, and land only after release/security owner review.

## Reproduction Assessment

Yes for the PR-introduced regression from source inspection: dispatching plugin-clawhub-release.yml from a matching Tideclaw alpha branch now fails the new main/release-only gate before the existing alpha validation can run. The broader supply-chain hardening is not a user bug reproduction path.

## Solution Assessment

No as submitted. The maintainable fix is to preserve the same Tideclaw alpha branch allowance already present in release publish, plugin npm publish, and the existing ClawHub trusted-branch validation.
