## What This Changes

The PR hardens high-risk GitHub Actions release, publish, live/e2e, and build-artifact workflows, splits plugin npm publish and dist-tag mirroring auth, pins external Actions, and adds token-boundary coverage.

## Best Possible Solution

Keep the hardening approach, repair the focused token-boundary test harness, then let maintainer-controlled CI/release validation cover the live publish paths.

## Reproduction Assessment

Yes for the review findings. Source inspection of the latest diff shows the fake npm record is written differently from the success assertions and the missing-auth case reads a record file that is never created; I did not run tests because this review is read-only.

## Solution Assessment

No. The workflow/script hardening direction is maintainable, but the patch is not merge-ready until the test harness uses deterministic record writing and handles the no-publish case without reading a nonexistent file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the fake publish record with printf:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:35`
  - body: The fake npm script is generated from a TypeScript template literal, so `\n` becomes a real newline inside the shell-quoted `echo` argument and `echo` appends another one. The success tests expect exactly one trailing newline, so this assertion will fail; use `printf '%s\n' ...` or escape the backslash in the generated script.
  - confidence: 0.92
- **[P2] Avoid reading a missing no-publish record:** `test/scripts/plugin-npm-publish-token-boundary.test.ts:114`
  - body: In the missing mirror-auth case, fake `npm view` exits without writing the record and the script exits before any publish call can create it. `readFileSync(record, "utf8")` will throw `ENOENT` instead of proving publish was skipped; create an empty record first or assert that the file is absent.
  - confidence: 0.93
