## Summary

Keep open. Current main still builds the PDF tool during the per-turn core tool-prep path and resolves PDF model/auth availability before the tool is used; the overlapping #73039 PR is still open rather than merged, so the reported performance problem is not implemented on main yet.

## What This Changes

The issue requests avoiding repeated PDF tool initialization/model-auth resolution during request-time tool preparation because it can trigger long `event_loop_delay` warnings on WeCom/Linux arm64.

## Best Possible Solution

Make PDF tool registration cheap on every turn by caching or deferring PDF model/auth resolution, then land #73039 if its broader cache is safe or split a narrower PDF-only fix.

## Reproduction Assessment

Unclear: there is no high-confidence live reproduction of the WeCom/Linux arm64 10-12s stall. Source inspection does show the current request hot path rebuilds the PDF tool and performs immediate provider/auth resolution when PDF availability is true.

## Solution Assessment

Unclear: a video-style async PDF mode or a new config knob is broader than the first maintainable fix. The safer direction is lazy or cached tool-prep work, with maintainer review of whether #73039 should be the canonical implementation.
