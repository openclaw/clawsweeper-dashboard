## Summary

Keep open. Current main still builds the PDF tool during the core tool-prep hot path, and the reported release has identical code, so the per-request stall is plausible; the exact 10-12s root cause still needs live validation because current source only imports `pdfjs-dist` during actual PDF extraction.

## What This Changes

The issue requests eliminating per-request `pdf` tool initialization stalls that trigger liveness event-loop-delay warnings, either by making PDF setup lazy/asynchronous or cached.

## Best Possible Solution

Make PDF tool registration cheap on every agent turn by caching scoped availability/model/auth resolution or deferring expensive discovery, while keeping `pdfjs-dist` and canvas loading confined to actual PDF execution.

## Reproduction Assessment

Unclear: there is not yet a high-confidence live reproduction of the reported 10-12s WeCom/Linux arm64 stall. Source inspection does show the PDF tool is rebuilt on the request hot path and performs immediate model/auth resolution, so the performance bug is source-reproducible enough to keep open.

## Solution Assessment

Unclear: adding a video-style async PDF completion mode or new config knob is not the narrowest first fix. The better first solution is to make PDF tool prep lazy/cached without changing PDF execution semantics.
