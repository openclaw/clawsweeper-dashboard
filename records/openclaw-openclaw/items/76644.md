## Summary

Keep open. Current main still constructs the PDF tool during the per-turn tool-prep path and resolves PDF model/auth candidates at factory time; the related performance PR #73039 is open but not merged, so no close reason is satisfied.

## What This Changes

Reduce repeated request-time PDF tool initialization/model-auth resolution that is reported to cause `event_loop_delay` liveness warnings.

## Best Possible Solution

Make PDF tool registration cheap on every agent turn by caching or deferring scoped availability/model/auth resolution, preferably by validating #73039 or requesting a narrower PDF-only patch without adding a user-facing config knob unless needed.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported WeCom/Linux arm64 10-12s stall. Source inspection does show current main still builds the PDF tool and resolves auth/model candidates on the request tool-prep path, so the report is source-reproducible enough to keep open.

## Solution Assessment

Unclear. A video-style async PDF mode or new config option is not the narrowest first fix; the maintainable path is to make registration lazy/cached, with maintainer choice between #73039 and a smaller PDF-only repair.
