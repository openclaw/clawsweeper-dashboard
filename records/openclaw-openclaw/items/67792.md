## Summary

Keep open. Current main still builds the Telegram model button fallback label from the provider-local model id, so `openrouter/openai/gpt-5.4-mini` still displays as `openai/gpt-5.4-mini` when there is no display-name override. The provided timeline also shows open draft PR #72752 explicitly closes this issue, so the issue should stay open until that PR is reviewed and landed or closed.

## What This Changes

The issue asks for the Telegram model selection menu to show the full OpenRouter provider/model ref for nested OpenRouter model ids.

## Best Possible Solution

Review and land a narrow Telegram plugin fix that changes only the fallback display label for nested provider-local ids to the canonical provider/model ref, while preserving display-name overrides, callback data, selection resolution, pagination, truncation, and current-model marking, with focused regression tests for OpenRouter nested ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildModelsKeyboard` with provider `openrouter`, model `openai/gpt-5.4-mini`, and no `modelNames` entry; current main falls back to the bare model string.

## Solution Assessment

Yes. Fixing the Telegram `buildModelsKeyboard` fallback label and adding focused unit coverage is the narrowest maintainable solution; no core model parser or callback-data change is needed.
