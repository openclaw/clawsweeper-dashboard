## Summary

Keep open. Current main still builds the Telegram model button fallback from the provider-local model id, and open draft PR #72752 explicitly closes this issue, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests Telegram's model picker to display the full OpenRouter provider/model ref instead of only the provider-local model id.

## Best Possible Solution

Update the bundled Telegram plugin so fallback labels for nested provider-local ids render as canonical provider/model refs, while preserving display-name overrides, callback data, pagination, truncation, and current-model marking with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildModelsKeyboard` with provider `openrouter`, model `openai/gpt-5.4-mini`, and no `modelNames` entry; current main falls back to the bare model string.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram `buildModelsKeyboard` fallback label, not in core model parsing or callback-data selection.
