## Label Justifications

- `P1`: The report describes a user-facing agent/session workflow where completed subagents can leave the parent session misleadingly busy or temporarily unresponsive.
- `impact:session-state`: The core failure mode is stale or mismatched subagent/session lifecycle state persisting after completion.
- `impact:message-loss`: The reporter observed new user messages being queued or ignored while the parent session appeared stuck after subagent completion.

## Summary

Keep open. Current main has partial UI mitigation for the chat picker, but the core lifecycle issue remains source-reproducible: default run-mode subagents still use `cleanup: "keep"`, keep-mode runs are not archived, the sweeper skips those rows, and normal session-list/Sessions UI paths can still surface terminal non-archived subagent sessions while yielded-parent responsiveness is tracked by related open reports.

## What This Changes

The issue requests cleanup or hiding of completed one-shot subagent sessions and immediate parent-session responsiveness after subagent completion.

## Best Possible Solution

Define a single lifecycle contract for one-shot subagent history and yielded parent sessions, then apply it consistently across registry cleanup, `sessions.list` defaults, task reconciliation, attachment retention, Control UI visibility, and regression tests.

## Reproduction Assessment

Yes for the retained completed-session symptom at source level: default run-mode cleanup is `keep`, keep-mode runs have no archive timer, the sweeper skips them, and sessions/UI tests still keep terminal non-archived rows visible. No live current-main reproduction was established for the transient Dashboard unresponsiveness.

## Solution Assessment

No current main behavior fully solves the issue. The chat picker clutter is partially fixed, but the maintainable solution needs a maintainer-approved lifecycle and retention policy plus a unified yielded-parent wake contract.
