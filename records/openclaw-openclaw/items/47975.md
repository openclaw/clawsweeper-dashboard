## Label Justifications

- `P1`: The report covers a user-facing session workflow where completed subagents can leave the parent session unresponsive or misleadingly busy.
- `impact:session-state`: The core failure mode is stale or mismatched subagent/session lifecycle state persisting after completion.
- `impact:message-loss`: The reporter observed new user messages being queued or ignored while the parent session appeared stuck after subagent completion.

## Summary

Keep open. Current main still retains default one-shot subagent sessions/run records, keeps terminal non-archived rows visible in session-list and Control UI paths, and the top-level `sessions_yield` persisted paused-state fix remains in https://github.com/openclaw/openclaw/pull/75662 rather than on main.

## What This Changes

The issue asks for completed one-shot subagent sessions to be cleaned up or hidden after completion and for the parent session to resume processing messages immediately after `sessions_yield`.

## Best Possible Solution

Define the terminal lifecycle contract for one-shot subagent history and yielded parent sessions, then align registry cleanup, `sessions.list`, Control UI filtering, task reconciliation, attachment retention, docs, and regression tests to that contract.

## Reproduction Assessment

Yes for the retained completed-session symptom at source level: default run-mode cleanup is `keep`, keep-mode runs have no archive timer, and list/UI paths still include terminal non-archived rows. No live current-main reproduction was established for the transient Dashboard delay.

## Solution Assessment

No current patch fully solves it. The best path is a maintainer-approved lifecycle policy plus the main-session yield-pause fix or equivalent, rather than a narrow cleanup-only change.
