## Label Justifications

- `P1`: The issue describes repeated real-user stalls and stale subagent state affecting active WebChat, Telegram, macOS, and WSL2 workflows.
- `impact:session-state`: Subagent lifecycle, session rows, task records, attachment state, and yielded-parent state can diverge from terminal run state.
- `impact:message-loss`: Fresh parent-session messages and completion/status messages can be delayed or disappear enough to look queued, ignored, or lost to users.

## Summary

Keep open: current main still only partially mitigates the report; one-shot subagent cleanup defaults to retained history, keep-mode runs avoid archive timers, task maintenance can preserve stale running subagent tasks when the child session row still exists, and the parent responsiveness half overlaps live open scheduler/yield reports.

## What This Changes

The issue asks for completed or failed run-mode subagent sessions to stop lingering in session/task state and for the parent session to accept messages promptly after subagent completion.

## Best Possible Solution

Define one core lifecycle contract for one-shot subagent retention, task reconciliation, attachment cleanup, session-list/UI visibility, and parent completion delivery, then implement it consistently with history retained only by explicit policy.

## Reproduction Assessment

Yes for the retention/task-state half by source inspection; the full parent responsiveness timing still needs a live WebChat or Telegram reproduction.

## Solution Assessment

Unclear: immediately deleting all completed subagent sessions would reduce stale rows but may violate the current cleanup=keep history contract and does not by itself fix task reconciliation or parent delivery.
