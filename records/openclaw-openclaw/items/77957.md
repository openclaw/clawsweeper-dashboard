## Summary

Keep open: current main and v2026.6.10 still finalize completed voice calls by persisting the terminal call record and clearing active state, without a supported post-call Telegram/session notification path. The closest prior relay PR was closed unmerged as the wrong vehicle, so this remains a valid product gap that needs maintainer direction before implementation.

## What This Changes

The issue requests visible post-call delivery for completed inbound voice-call transcripts, such as a Telegram/user notification, originating-session summary, or explicit delivery failure instead of silent persistence.

## Best Possible Solution

Define a plugin-owned, opt-in post-call delivery contract for target session/channel, summary content, failure reporting, and privacy defaults, then implement it in the voice-call runtime with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: completed calls reach `call.ended`, then `finalizeCall`, which persists and removes active state without any post-call delivery callout. I did not place a live Twilio/Telegram call in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. An opt-in plugin-owned post-call delivery path is safer than unconditional Telegram delivery or a core-specific shortcut.
