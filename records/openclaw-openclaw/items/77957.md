## Summary

Keep open: current main still has no post-call notification or delivery surface for completed voice-call transcripts, and the related report is closed as stale rather than serving as an active canonical tracker.

## What This Changes

Request post-call user or Telegram notification, or a visible delivery failure, when completed inbound voice-call transcripts are currently only saved to `calls.jsonl`.

## Best Possible Solution

Define a plugin-owned post-call completion contract that summarizes completed transcript-bearing calls, delivers to an approved session or channel target with logged failures, documents the config/API, and adds regression coverage around `call.ended` and realtime close.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection and the detailed reporter evidence: completed realtime calls enter `call.ended`, are persisted, and have no current post-call delivery path; no live Twilio run was executed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the delivery policy; the fix should stay in the voice-call plugin or a generic plugin event seam, not an ad hoc Telegram-only scanner.
