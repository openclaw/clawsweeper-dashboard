## Summary

Keep open: current main and the latest shipped voice-call docs still persist and finalize completed calls without a post-call Telegram/session notification path. A prior closed PR explored post-call relay, but maintainer follow-up explicitly split that product scope out rather than landing it.

## What This Changes

Request a post-call delivery path so completed inbound voice-call transcripts reach Telegram, the originating session, or the user instead of only being persisted to `calls.jsonl`.

## Best Possible Solution

Define an opt-in, plugin-owned post-call completion contract that can summarize transcript-bearing calls, deliver to an approved session or configured channel, report delivery failures visibly, and document the config/API with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows inbound/realtime calls persist transcript-bearing records and finalize through `call.ended` without a notification path, and the reporter provided live Twilio plus Telegram evidence; no live call was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the delivery policy. A plugin-owned post-call hook or delivery contract is the maintainable direction; a Telegram-only special case or external `calls.jsonl` scanner would be brittle.
