## Label Justifications

- `P2`: Completed voice calls can silently fail to notify the user after transcript persistence, but the affected scope is limited to voice-call post-call delivery semantics.
- `impact:message-loss`: The issue is about a completed call transcript not being delivered visibly to Telegram or another user/session channel.

## Summary

Keep open: current main and the latest release still finalize completed voice calls by persisting the terminal record and cleaning active state without a post-call Telegram/session notification path, and the prior post-call relay PR was closed as the wrong merge vehicle rather than merged.

## What This Changes

Requests post-call delivery or visible delivery failure for completed voice-call transcripts, especially to Telegram or the originating session.

## Best Possible Solution

Define plugin-owned, opt-in post-call delivery semantics for target session/channel, transcript summary, failure reporting, and privacy defaults, then implement that path in the voice-call runtime with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: `call.ended` reaches `finalizeCall`, which persists and removes active call state without any post-call delivery callout. I did not place a live Twilio call in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract; an opt-in plugin-owned post-call delivery path is safer than unconditional Telegram delivery or a core-specific shortcut.
