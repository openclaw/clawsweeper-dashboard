## Summary

Keep open: current main still has no post-call notification or delivery surface for completed voice-call transcripts, and the linked earlier report was closed as stale rather than serving as an active canonical tracker.

## What This Changes

Completed inbound voice calls should deliver or visibly surface a post-call transcript summary or delivery failure instead of only persisting to `calls.jsonl`.

## Best Possible Solution

Define a plugin-owned post-call completion contract that summarizes completed transcript-bearing calls, delivers to an approved session or channel target with logged failures, documents the config/API, and adds regression coverage around realtime and `call.ended` completion.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection plus detailed reporter evidence: completed realtime calls enter `call.ended`, are persisted, and have no current post-call delivery path; no live Twilio run was executed.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose delivery policy; the durable fix should stay in the voice-call plugin or an additive generic plugin event seam, not an ad hoc Telegram-only scanner.
