## Summary

Keep open. Current main has active-schema validation, legacy doctor migrations, unknown-key stripping, config version stamping, and Control UI diff rendering, but it still does not implement the requested `openclaw doctor --config-diff` CLI workflow or a version-to-version migration report based on `meta.lastTouchedVersion`; the related issues are adjacent rather than superseding this CLI feature request.

## What This Changes

Add an `openclaw doctor --config-diff` workflow that compares a config's last-touched version with the current schema, reports unknown, removed, renamed, new, and invalid enum entries, and optionally applies safe migrations with a backup.

## Best Possible Solution

Keep the issue open until maintainers either implement or explicitly reject a CLI upgrade-diff workflow. A complete solution should add doctor command plumbing, docs, and tests; use a trustworthy historical baseline keyed from `meta.lastTouchedVersion`; report unknown keys, invalid enum values, removed/renamed/new fields, and migration hints; and keep automatic fixes limited to explicit safe migrations with backups.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing surface is high-confidence from command registration, option types, doctor flow, tests/docs, and schema inspection on current main.

## Solution Assessment

Unclear until maintainer design is settled. The requested CLI direction is plausible and matches real upgrade pain, but historical schema storage, suggestion ownership, and `--fix` safety need product/code-owner decisions before a broad automated implementation.
