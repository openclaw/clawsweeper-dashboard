## Summary

Keep open: current main has doctor lint/repair scaffolding, unknown-key cleanup, live config schema lookup, and `meta.lastTouchedVersion`, but it does not implement the requested version-to-version `openclaw doctor --config-diff` schema comparison or migration-suggestion mode. The closest open thread, https://github.com/openclaw/openclaw/issues/79166 with https://github.com/openclaw/openclaw/pull/79734, covers previewing `doctor --fix` mutations rather than owning schema snapshot diffs across releases.

## What This Changes

The issue requests an `openclaw doctor --config-diff` workflow that compares a user's config against versioned schema changes and suggests or applies safe migrations.

## Best Possible Solution

Keep this open for a maintainer product decision on a core doctor/schema design: either broaden the doctor dry-run work to cover release-to-release config schema diffs, or define a separate `doctor --config-diff` path backed by the current schema, explicit doctor migrations, and documented version metadata.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does confirm current main lacks the requested `doctor --config-diff` flag and versioned schema-diff implementation.

## Solution Assessment

Unclear without maintainer product direction. The narrowest maintainable path is likely to extend doctor’s existing migration and repair-preview contract, but historical schema snapshots and auto-fix semantics need an explicit decision first.
