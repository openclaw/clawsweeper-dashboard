## Summary

Current main and shipped releases now solve the central stale-config problem through the canonical doctor workflow: invalid configuration is loaded best-effort, known legacy shapes receive actionable migration guidance, unknown keys are listed during preview, and `doctor --fix` applies the repaired configuration. Historical schema archives and a catalog of every newly available key would be separate product scope rather than necessary upgrade recovery.

## What This Changes

Add a doctor config-diff mode that identifies stale or invalid settings across upgrades and proposes or applies safe migrations.

## Best Possible Solution

Keep the shipped explicit doctor migration registry and preview/fix pipeline as the single compatibility source of truth, adding focused migration rules whenever future shipped configuration changes invalidate an older shape.

## Reproduction Assessment

Yes at source level. Current doctor code deterministically identifies known legacy shapes and unknown keys, previews their migration or removal, and can apply the repaired configuration; recreating the obsolete 2026.3.2 installation is unnecessary for this close decision.

## Solution Assessment

No, the proposed per-version schema snapshot system is not the best implementation. The shipped explicit doctor migration pipeline is narrower, authoritative, and consistent with the repository’s current-schema-only compatibility policy.
