## Summary

Keep open. The patch has a concrete correctness blocker: it sanitizes post-conversion user-message `name` fields, but the inspected current serializer path and pinned dependency source do not show Telegram `senderName` becoming that field, matching the earlier related discussion that asked for a failing outbound payload.

## What This Changes

The PR adds a shared sender-name sanitizer, applies it to user-role Chat Completions messages after pi-ai conversion, and adds helper plus OpenAI transport unit tests.

## Best Possible Solution

Capture or construct the rejected outbound provider payload, then sanitize only the actual provider-bound field while preserving raw sender names for policy and display, with focused regression coverage and a changelog entry.

## Reproduction Assessment

No. The Telegram steps are concrete, but current source and dependency inspection do not establish a high-confidence path from Telegram `senderName` to Chat Completions `messages[i].name`, and the PR tests do not reproduce that path.

## Solution Assessment

No. Sanitizing post-conversion user-message `name` fields is only the best fix if those fields are actually serialized; the safer path is to prove the rejected outbound field first.
