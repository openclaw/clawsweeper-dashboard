## Summary

Keep open: current main has TTS audio delivery, but it does not carry word-level timing metadata or implement desktop/web playback highlighting, and changing TTS into post-text follow-up delivery is a multi-surface product/API decision rather than conservative cleanup.

## What This Changes

The issue requests desktop/web karaoke-style TTS word highlighting plus sending generated TTS audio as a follow-up message after visible text.

## Best Possible Solution

Design an optional timestamp-bearing speech-provider result contract, propagate it through transcript/webchat state, add fallback desktop/web highlighting, and make TTS delivery ordering an explicit channel/product decision.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection still confirms current main lacks timestamp propagation and word-highlight rendering, while TTS audio remains media attached to a reply payload.

## Solution Assessment

Unclear as a direct implementation plan: carrying optional timing metadata is a plausible direction, but globally changing TTS into text-then-audio follow-ups needs maintainer product/API design because existing channel delivery relies on caption/media sequencing.
