## Summary

Keep open: current main still lacks an end-to-end timestamp/highlight path and still treats TTS audio as media on the reply payload, while the related open PR is not merged and has already been reviewed as needing proof and fixes.

## What This Changes

Requests desktop/web TTS word-level highlighting from provider timing metadata and changing TTS audio delivery to a text-then-audio follow-up message.

## Best Possible Solution

Design an additive timestamp-bearing speech result contract, propagate it through webchat rendering with fallback behavior, and make TTS delivery ordering an explicit channel/product decision.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence current main lacks the requested timestamp/highlight pipeline and still sends TTS as media attached to payloads.

## Solution Assessment

Unclear as a direct implementation plan: timestamp metadata is plausible, but audio follow-up delivery should be designed as an explicit channel/UI policy rather than changing shared media sequencing globally.
