## Summary

Keep open: current main still has ordinary TTS audio delivery but no timestamp-bearing TTS contract, no desktop/web word-highlight playback, and no text-then-audio follow-up delivery mode. The related open PR only partially attempts the timestamp side, is not merged, and does not resolve the broader delivery-order request.

## What This Changes

The issue requests word-level TTS playback highlighting in the desktop/web UI and separate post-text TTS audio delivery instead of bundling audio with the text payload.

## Best Possible Solution

Design this as explicit product/API work: add an optional timestamp-bearing TTS contract, propagate it through transcript/UI state with fallback playback, and decide delivery ordering per channel instead of globally changing media sequencing.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main lacks timestamp propagation and desktop/web highlight rendering, while TTS audio is still attached to the reply payload.

## Solution Assessment

Unclear as a direct implementation plan: timestamp metadata is plausible, but globally changing TTS to text-then-audio follow-ups needs maintainer product/API design because existing outbound code intentionally sends text with the first media item.
