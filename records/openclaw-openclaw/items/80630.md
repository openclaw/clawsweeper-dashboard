## Summary

Keep open: current main still lacks word-level TTS timing propagation and desktop/web karaoke highlighting, and the follow-up audio delivery request crosses SDK, provider, UI, and channel delivery semantics. The related PR at https://github.com/openclaw/openclaw/pull/80646 is open, partial, and not a merged implementation path for this issue.

## What This Changes

Add desktop/web karaoke-style word highlighting for TTS playback and send generated TTS audio as a separate follow-up message after the text.

## Best Possible Solution

Design an optional timestamp-bearing TTS result contract, propagate it through provider and speech-core paths into transcript/webchat state with fallback highlighting, and make text/audio ordering an explicit channel/product policy.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken documented behavior. Source inspection confirms current main lacks timestamp propagation/highlighting and still treats synthesized audio as reply media.

## Solution Assessment

Unclear as a direct implementation plan: optional timing metadata is plausible, but the combined request should be designed as an SDK/provider/UI contract plus an explicit channel delivery policy rather than a global media sequencing change.
