## Summary

Keep open: current main still has TTS audio delivery but does not propagate word-level timing metadata or render synchronized desktop/web highlighting, and the requested text-then-audio delivery order is a cross-surface product/API decision. The related open PR only attempts part of the timestamp work and is not merged or sufficient to close this issue.

## What This Changes

Request word-synchronized TTS playback/highlighting in the desktop/web UI and TTS audio delivery as a separate follow-up message instead of attaching it to the same reply.

## Best Possible Solution

Design an optional timestamp-bearing speech-provider result contract, propagate it through reply/webchat state, add fallback desktop/web highlighting, and make text/audio delivery ordering an explicit channel/product policy.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main lacks timestamp propagation and desktop/web highlighting, while current TTS audio remains attached to the reply payload.

## Solution Assessment

Unclear as a direct implementation plan: carrying optional timing metadata is plausible, but globally changing delivery into text-then-audio follow-ups needs maintainer product/API design because current channel sequencing intentionally treats text as the first media caption.
