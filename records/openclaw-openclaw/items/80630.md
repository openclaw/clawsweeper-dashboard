## Summary

Keep open: current main still supports TTS audio delivery, but it does not carry timestamp metadata into the TTS contracts, does not render desktop/web word highlighting, and still sends media/caption-style payloads rather than a dedicated text-then-audio follow-up sequence. The related open PR only attempts part of this and is not merged or merge-ready.

## What This Changes

Request word-synchronized TTS highlighting in the desktop/web UI and delivery of TTS audio as a separate follow-up message after text.

## Best Possible Solution

Split the design into an optional timestamp-bearing speech result contract with desktop/web highlighting, then make TTS follow-up delivery an explicit per-channel product/API decision.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request rather than broken documented behavior. Source inspection of current main confirms no timestamp propagation or UI highlighting and confirms current media/caption-style TTS delivery.

## Solution Assessment

Unclear as a direct implementation plan: optional timing metadata is plausible, but globally changing TTS audio into follow-up messages is not a narrow cleanup and needs maintainer product/API design.
