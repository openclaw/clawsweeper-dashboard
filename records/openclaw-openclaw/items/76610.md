## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the existing lock, so the reported unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

Persist Matrix crypto IDB snapshots atomically instead of writing JSON directly to the final snapshot path.

## Best Possible Solution

Use `writeJsonFileAtomically()` inside the existing Matrix IDB file lock, keep the storage path and timing unchanged, and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level. Current main routes Matrix init, periodic flush, and shutdown persistence through `persistIdbToDisk()`, which still writes directly to the final snapshot file; I did not run a live crash-timing repro.

## Solution Assessment

Yes. Reusing the existing plugin SDK atomic JSON writer under the already-present file lock is the narrowest maintainable fix because it preserves callers while adding temp-file replacement, fsync, permissions, and cleanup semantics.
