## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON file, while Matrix startup, periodic flush, and shutdown persistence all funnel through this path; the existing SDK atomic JSON helper is available and already used by nearby Matrix persistence code.

## What This Changes

Make Matrix crypto IDB snapshot persistence atomic so unclean shutdowns cannot leave `crypto-idb-snapshot.json` truncated or partially rewritten.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside the current Matrix IDB file lock, remove the direct final-path write, and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level. Current main captures the Matrix crypto snapshot and writes it directly to the active JSON path, so an interrupted final-path write can leave the only snapshot truncated; a deterministic regression can mock the write or replacement failure.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` inside the existing lock is the narrowest maintainable fix because the helper already owns temp-file replacement, permissions, fsync, and cleanup semantics.
