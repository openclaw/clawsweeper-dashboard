## Summary

Keep open. Current main still writes the Matrix crypto IndexedDB snapshot directly to the final JSON path, so the reported unclean-shutdown corruption risk remains valid and narrowly fixable with the existing atomic JSON writer.

## What This Changes

The issue requests atomic persistence for Matrix crypto IndexedDB snapshots so `crypto-idb-snapshot.json` cannot be left partially written after an unclean shutdown.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer under the current Matrix IDB file lock, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix init, periodic flush, and shutdown persistence through `persistIdbToDisk()`, which still writes directly to the final snapshot path. I did not establish a live kill/OOM timing repro.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` inside the existing file lock is the narrowest maintainable fix because the helper already implements temp-file replacement, fsync, restrictive permissions, and cleanup.
