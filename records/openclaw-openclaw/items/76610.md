## Summary

Keep open. Current main and the latest release still write the Matrix crypto IDB snapshot directly to the final JSON path, so the source-proven corruption risk remains and is narrow enough to fix with the existing atomic JSON helper.

## What This Changes

Matrix crypto-store persistence should replace the direct `crypto-idb-snapshot.json` write with an atomic snapshot write.

## Best Possible Solution

Use `writeJsonFileAtomically()` inside the existing snapshot file lock and add focused regression tests proving successful output plus preservation of the previous valid snapshot when replacement fails.

## Reproduction Assessment

Yes, at source level: all Matrix crypto snapshot flush paths route through `persistIdbToDisk()`, which still writes directly to the final snapshot file. I did not live-time a kill/OOM during the write, so the failure should be covered with mocked write or replacement failures.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` is the narrowest maintainable fix because it is already a plugin SDK subpath and already implements temp-file replacement, fsync, permissions, trailing newline, and cleanup.
