## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the existing file lock, so the unclean-shutdown corruption risk remains valid and narrow enough for a focused repair.

## What This Changes

Matrix crypto-store persistence should replace direct final-path snapshot writes with atomic JSON replacement.

## Best Possible Solution

Use writeJsonFileAtomically() inside the existing Matrix IDB file lock, remove the direct write/chmod pair, add preservation and temp-cleanup regression tests, and add a user-facing changelog fix entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix init, periodic flush, and shutdown persistence through persistIdbToDisk(), which still writes directly to the final snapshot path. I did not live-time a kill or OOM during the write; a focused test can reproduce the failure class by injecting a write or replacement failure.

## Solution Assessment

Yes. Reusing writeJsonFileAtomically() inside the existing file lock is the narrowest maintainable solution because the helper already provides temp-file replacement, fsync, restrictive permissions, directory mode, and cleanup.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76610 by making Matrix crypto IDB snapshot persistence atomic. In `extensions/matrix/src/matrix/sdk/idb-persistence.ts`, replace the direct `fs.writeFileSync(snapshotPath, JSON.stringify(snapshot))` plus `fs.chmodSync` final-path write with the existing `writeJsonFileAtomically()` helper from `openclaw/plugin-sdk/json-store`, keeping the existing `withFileLock` ordering so the snapshot is captured and written while the lock is held. Add focused regression coverage in `extensions/matrix/src/matrix/sdk/idb-persistence.test.ts` for successful atomic persistence, 0o600 mode, trailing newline/parseability, and failure during atomic replacement preserving the previous valid snapshot with no leftover temp files where practical. Do not change the snapshot schema, persistence interval, lock options, restore behavior, token-hash layout, or Matrix crypto bootstrap policy. Add a single-line user-facing fix entry to `CHANGELOG.md` unless the maintainer says this is internal-only.
