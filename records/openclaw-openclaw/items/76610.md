## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path, so the unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

Make Matrix crypto IDB snapshot persistence atomic so unclean shutdowns cannot leave crypto-idb-snapshot.json truncated or partially written.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside the existing snapshot file lock, preserving current restore and prefix behavior while adding regression coverage for successful output and failed replacement preservation.

## Reproduction Assessment

Yes, at source level: current main writes the snapshot directly to the final path, and Matrix init, periodic persistence, and shutdown all route through persistIdbToDisk(). I did not live-time an unclean shutdown during the write.

## Solution Assessment

Yes. Reusing writeJsonFileAtomically() is the narrowest maintainable fix because it is already a plugin SDK subpath, already used by nearby Matrix state persistence, and already provides temp-file replacement, fsync, permissions, and cleanup.
