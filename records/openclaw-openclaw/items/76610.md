## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the advisory lock, so the unclean-shutdown corruption risk remains a valid narrow bug; related #59851 fixed concurrent access locking but did not make the replacement atomic.

## What This Changes

The issue requests making Matrix crypto-store snapshot persistence atomic instead of writing `crypto-idb-snapshot.json` directly in place.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside the current Matrix IDB file lock, preserving the snapshot format and persistence timing while adding focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix crypto init, periodic flush, and shutdown persistence through `persistIdbToDisk()`, which still writes directly to `snapshotPath`. I did not live-time a kill during the write; a deterministic test can mock replacement failure and assert the previous snapshot stays parseable.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` under the existing `withFileLock()` is the narrowest maintainable fix because it preserves the current contract while using the repo’s established temp-file, fsync, rename, chmod, and cleanup path.
