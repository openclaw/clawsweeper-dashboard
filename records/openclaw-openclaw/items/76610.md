## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the existing lock, so the reported unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

The issue requests making Matrix crypto IDB snapshot persistence atomic instead of writing `crypto-idb-snapshot.json` directly.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer under the existing Matrix IDB file lock, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix init, periodic flush, and shutdown persistence through `persistIdbToDisk()`, which still calls `fs.writeFileSync()` on the final snapshot path. I did not live-time a kill or OOM during the write; a fix can reproduce the failure by mocking an atomic write or replacement failure.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` inside the existing file lock is the narrowest maintainable fix because the helper already implements temp-file replacement, fsync, permissions, and cleanup, and nearby Matrix state files already use it.
