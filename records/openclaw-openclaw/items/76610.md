## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the existing lock, so the unclean-shutdown corruption risk remains valid and narrowly fixable with the existing atomic JSON helper.

## What This Changes

The issue requests making Matrix crypto IDB snapshot persistence atomic instead of writing `crypto-idb-snapshot.json` directly.

## Best Possible Solution

Use `writeJsonFileAtomically()` inside the existing Matrix IDB file lock, preserve the snapshot format, and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix crypto init, periodic persistence, and shutdown persistence through `persistIdbToDisk()`, which still writes directly to the final snapshot path. I did not live-time a kill or OOM during the write; a fix should reproduce the failure with a controlled mocked write or replacement error.

## Solution Assessment

Yes. Reusing the existing exported atomic JSON helper inside the existing lock is the narrowest maintainable fix because it keeps the Matrix persistence contract unchanged while using the repository’s established temp-file replacement, fsync, permissions, and cleanup behavior.
