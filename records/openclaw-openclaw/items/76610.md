## Summary

Keep open. Current main and the latest release still write the Matrix crypto IDB snapshot directly to the final JSON path, so the reported corruption window remains and no related open PR fixes this atomic-write gap.

## What This Changes

The issue asks for Matrix crypto IDB snapshot persistence to use atomic replacement instead of writing JSON directly to the final snapshot path.

## Best Possible Solution

Use writeJsonFileAtomically inside the existing Matrix IDB file-lock boundary and add regression coverage that a failed replacement leaves the previous snapshot valid and parseable.

## Reproduction Assessment

Yes, at source level: current main still writes directly to the final snapshot path, and a focused regression can simulate write or replacement failure without depending on crash timing. I did not run a live unclean-shutdown reproduction.

## Solution Assessment

Yes. Reusing writeJsonFileAtomically under the existing withFileLock boundary is the narrowest maintainable repair, and the sibling shutdown/event-driven persistence work should stay separate.
