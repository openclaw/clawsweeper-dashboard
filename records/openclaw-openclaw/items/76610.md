## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path, so the unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

The issue requests atomic Matrix crypto IDB snapshot persistence so interrupted writes cannot corrupt `crypto-idb-snapshot.json`.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside the existing Matrix IDB file lock, preserve the snapshot schema and persistence timing, add focused regression coverage, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes, at source level: current main routes shutdown, post-init, and periodic Matrix crypto persistence through `persistIdbToDisk()`, which still writes directly to the final snapshot path. I did not live-time a kill or OOM during the write.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` under the existing `withFileLock()` block is the narrowest maintainable fix because the helper already provides temp-file replacement, restrictive permissions, directory creation mode, fsync, and cleanup.
