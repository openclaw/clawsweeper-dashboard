## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path, so the unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

The issue requests changing Matrix crypto IDB snapshot persistence from direct final-path JSON writes to an atomic replacement path with regression coverage.

## Best Possible Solution

Use writeJsonFileAtomically() for the Matrix crypto IDB snapshot inside the existing file lock, remove the redundant final-path chmod, add focused failure-preservation coverage, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes, at source level. Current main routes Matrix crypto IDB persistence through persistIdbToDisk(), which still calls fs.writeFileSync() on the final snapshot path; I did not live-time an unclean shutdown during that write.

## Solution Assessment

Yes. Reusing the existing plugin SDK atomic JSON writer inside the existing lock is the narrowest maintainable fix because it preserves the snapshot format while adding temp-file replacement, fsync, permissions, and cleanup behavior already used elsewhere.

## Repair Work Prompt

Fix the Matrix crypto IDB snapshot durability bug from https://github.com/openclaw/openclaw/issues/76610. In extensions/matrix/src/matrix/sdk/idb-persistence.ts, replace the direct final-path fs.writeFileSync(snapshotPath, JSON.stringify(snapshot)) plus chmod path in persistIdbToDisk() with writeJsonFileAtomically() from openclaw/plugin-sdk/json-store, keeping the existing withFileLock boundary and snapshot format semantics. Add focused regression coverage in extensions/matrix/src/matrix/sdk/idb-persistence.test.ts proving successful persistence still restores, file mode remains 0o600 where the platform supports mode checks, and a failed atomic replacement path preserves a previous valid snapshot without leaving test-visible temp artifacts. Keep the fix scoped to Matrix IDB persistence; do not change persistence timing, token hashing, Matrix SDK behavior, config, or unrelated stores. Add a single-line user-facing CHANGELOG.md fix entry because this affects Matrix E2EE durability.
