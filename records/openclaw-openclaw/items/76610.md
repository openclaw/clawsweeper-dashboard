## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path, while an existing plugin SDK atomic JSON helper is available and already used by nearby Matrix persistence code.

## What This Changes

The issue requests making Matrix crypto-store snapshot persistence atomic to avoid truncated or partially written `crypto-idb-snapshot.json` after unclean shutdowns.

## Best Possible Solution

Use `writeJsonFileAtomically()` in `persistIdbToDisk()` and add focused regressions proving successful output remains restorable and failed replacement preserves a previous valid snapshot.

## Reproduction Assessment

Yes, at source level: current main routes Matrix crypto init, periodic persistence, and shutdown through `persistIdbToDisk()`, which writes directly to the final snapshot path. I did not live-reproduce a process kill during the write because this read-only pass is better served by a mocked failure regression.

## Solution Assessment

Yes. Reusing the existing SDK atomic JSON helper is the narrowest maintainable fix because it already provides the needed temp-file replacement, permissions, parent directory mode, fsync, and cleanup behavior through a public plugin SDK subpath.

In `extensions/matrix/src/matrix/sdk/idb-persistence.ts`, replace the direct final-path JSON write in `persistIdbToDisk()` with the existing `writeJsonFileAtomically()` helper from `openclaw/plugin-sdk/json-store`; remove redundant final-file chmod only if the helper fully covers the mode. Keep the fix plugin-local and do not add config flags or change Matrix crypto behavior beyond safer persistence. Add focused tests in `extensions/matrix/src/matrix/sdk/idb-persistence.test.ts` proving the final snapshot is restorable, has mode `0o600`, uses the helper's newline behavior if asserted locally, and that a simulated failed atomic replacement preserves a previous valid snapshot and does not leave temp files.
