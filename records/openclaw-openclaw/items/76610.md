## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path under the existing file lock, so the reported unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

The issue requests atomic Matrix crypto IndexedDB snapshot persistence to avoid a truncated or corrupt `crypto-idb-snapshot.json` after an unclean shutdown.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside the current Matrix IDB file lock, add focused regression tests, and include a user-facing changelog entry.

## Reproduction Assessment

Yes, at source level: current main routes Matrix init, periodic flush, and shutdown persistence through `persistIdbToDisk()`, which still writes directly to `snapshotPath`. I did not live-time a kill or OOM during the write, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` inside the existing file lock is the narrowest maintainable fix because the helper already implements temp-file replacement, fsync, permissions, and cleanup.
