## Summary

Keep open. Current main still writes the Matrix crypto IDB snapshot directly to the final JSON path, so the reported unclean-shutdown corruption risk remains valid and narrowly fixable.

## What This Changes

The issue requests making Matrix crypto-store snapshot persistence atomic in `persistIdbToDisk()` to avoid truncated or partially written `crypto-idb-snapshot.json` files.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer in `persistIdbToDisk()` and add focused regressions proving successful output and failed replacement preservation of the previous valid snapshot.

## Reproduction Assessment

Yes, at source level: Matrix init, periodic flush, and shutdown persist all route through `persistIdbToDisk()`, which still writes directly to the final snapshot path. I did not live-time a kill/OOM during the write; the failure should be covered with mocked atomic replacement/write failures.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` is the narrowest maintainable fix because it is already a plugin SDK subpath, already used by nearby Matrix state files, and already implements temp-file replacement, fsync, permissions, and cleanup.

Fix the Matrix crypto IDB snapshot corruption risk reported in https://github.com/openclaw/openclaw/issues/76610. In `extensions/matrix/src/matrix/sdk/idb-persistence.ts`, replace the direct final-path `fs.writeFileSync(snapshotPath, JSON.stringify(snapshot))` + `fs.chmodSync(snapshotPath, 0o600)` path inside `persistIdbToDisk()` with the existing `writeJsonFileAtomically()` helper from `openclaw/plugin-sdk/json-store`. Keep the file-lock behavior and caller API unchanged. Add focused regressions in `extensions/matrix/src/matrix/sdk/idb-persistence.test.ts` proving the snapshot still restores, the final file has mode `0o600`, the helper's trailing newline/pretty JSON behavior is acceptable, and a simulated atomic write/rename failure leaves a previous valid snapshot parseable and does not leave temp files behind. Do not add a config option or change Matrix startup/recovery policy. Add a user-facing `CHANGELOG.md` fix entry for Matrix E2EE persistence crediting @eldron81-r2d2 if attribution is appropriate; do not credit bots.
