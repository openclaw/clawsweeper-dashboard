Keep open. Current main and the latest release still write the Matrix crypto IDB snapshot directly to the final JSON path, so the reported corruption window remains and no closing PR currently resolves the atomic-write gap.

## What This Changes

The issue requests atomic Matrix crypto IDB snapshot persistence so unclean shutdown cannot leave a truncated or partially written snapshot.

## Best Possible Solution

Use writeJsonFileAtomically inside the existing Matrix IDB file-lock boundary and add regression coverage proving a failed replacement leaves the previous snapshot valid and parseable.

## Solution Assessment

Yes. Reusing writeJsonFileAtomically under the existing withFileLock boundary is the narrowest maintainable repair, while sibling event-driven persistence or shutdown changes should stay separate.
