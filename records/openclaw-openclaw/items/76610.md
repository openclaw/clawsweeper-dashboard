## Summary

Keep open. Current main still writes the Matrix IDB crypto snapshot directly to the final JSON path, while the existing plugin SDK atomic JSON helper is available and already used by nearby Matrix state files.

## What This Changes

The issue requests atomic writes for Matrix crypto-store IDB snapshots to avoid corruption after an unclean shutdown.

## Best Possible Solution

Use the existing plugin SDK atomic JSON writer inside `persistIdbToDisk()` and add focused Matrix persistence regressions for successful atomic output and failed replacement preserving the previous snapshot.

## Reproduction Assessment

Yes, at source level: with Matrix E2EE enabled, init, periodic flush, and shutdown all route through `persistIdbToDisk()`, which currently performs a direct final-path `fs.writeFileSync()`. I did not live-reproduce a kill/OOM during the write because the timing-dependent failure is better covered with a mocked write/rename failure regression.

## Solution Assessment

Yes. Reusing `writeJsonFileAtomically()` is the narrowest maintainable fix because it is already a public plugin SDK subpath, uses the repo's durable temp-file replacement helper, and is already used by nearby Matrix state persistence code.
