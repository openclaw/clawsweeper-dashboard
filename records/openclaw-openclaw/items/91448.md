Full review comments:

- **[P2] Reject targets the runtime parser rejects:** `ui/src/ui/controllers/cron.ts:139-149`
  - body: `isValidTelegramTarget` strips `group:` and numeric topic suffixes before checking the chat segment, so inputs like `group:@mygroup`, `t.me/mygroup/`, or an unsafe `:topic:` id can pass this form validation even though the Telegram runtime keeps or rejects those forms and later throws the same delivery error. Mirror the runtime parser's strict topic and legacy-prefix rules so save-time validation actually blocks unsendable targets.
  - confidence: 0.88
- **[P2] Sync the generated i18n metadata:** `ui/src/i18n/locales/en.ts:1653-1654`
  - body: The PR adds a new cron error key to English and generated locale bundles, but no `ui/src/i18n/.i18n/*.meta.json` files changed. The scoped UI guide treats the non-English bundles as generated output, so the i18n sync metadata needs to come with these locale changes.
  - confidence: 0.9
- **[P2] Use Telegram-specific To helper copy:** `ui/src/ui/views/cron.ts:1109-1111`
  - body: The new Telegram validator rejects phone-like and short user-id values, but the announce field still renders the generic `toPlaceholder`/`toHelp` copy that tells users phone or user id values are acceptable. Switch the helper/placeholder for Telegram so the form does not invite values the new validation blocks.
  - confidence: 0.84
