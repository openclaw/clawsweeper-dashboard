## Summary

Keep this open: the PR is aimed at a real cron dashboard bug, but the current patch is not merge-ready because it narrows Telegram targets below the runtime contract, does not render the new announce-field error, and still lacks real dashboard proof.

## What This Changes

The PR adds numeric-only Telegram cron announce `To` validation, a new cron error locale key across locale bundles, controller regression tests, and two cron view label i18n swaps.

## Best Possible Solution

Keep save-time validation, but mirror the Telegram runtime target contract, render Telegram-specific announce `To` guidance and errors accessibly, sync i18n metadata, and require redacted dashboard proof before merge.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` validate only webhook `deliveryTo`, then forward Telegram announce `delivery.to` to runtime where an unresolvable target can fail with the reported error. I did not run a live dashboard or Telegram cron in this read-only review.

## Solution Assessment

No. Save-time validation is the right layer, but this implementation is too narrow, does not surface the error on the announce field, and needs generated i18n metadata plus real dashboard proof.
