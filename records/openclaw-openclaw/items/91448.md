## Summary

Keep open: the linked cron dashboard bug is still present on current main and v2026.6.6, but this branch is not merge-ready because the numeric-only validator rejects Telegram target forms the runtime currently supports, the new announce error is not rendered inline, and the supplied proof does not exercise the real dashboard flow.

## What This Changes

The PR adds Control UI cron validation for non-numeric Telegram announce `To` values, adds regression tests, updates locale bundles, and switches two cron command labels to existing i18n strings.

## Best Possible Solution

Keep save-time validation, but make it mirror the Telegram runtime target parser, render accessible announce `To` errors, and require redacted dashboard proof before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 validate webhook `deliveryTo` only, then forward announce `delivery.to` into `cron.add` or `cron.update`, matching the linked runtime failure. I did not run a live Telegram cron or full dashboard flow in this read-only review.

## Solution Assessment

No. Save-time validation is the right layer, but this implementation is too narrow for the existing Telegram target contract and does not surface the new announce `To` error in the field.
