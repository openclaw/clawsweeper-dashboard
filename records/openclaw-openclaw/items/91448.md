## Summary

Keep this PR open, but not merge-ready: current main and the latest release still lack Telegram cron announce `To` validation, while this branch adds a numeric-only gate that would block Telegram targets the runtime already supports, leaves the new announce error off the visible field, has i18n metadata drift risk, and only supplies injected-validator proof rather than dashboard proof.

## What This Changes

The PR adds Control UI validation for Telegram cron announce `To` values, adds a new validation string across locale maps, extends cron controller tests, and replaces two hardcoded cron command labels with existing i18n strings.

## Best Possible Solution

Keep save-time validation, but mirror the Telegram runtime target parser, render the announce `To` error accessibly, sync Control UI i18n generated metadata, and collect redacted dashboard proof before merge.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` validate only webhook `deliveryTo`, then forward announce `delivery.to` to runtime where unresolved Telegram targets can fail. I did not run a live dashboard or Telegram cron in this read-only review.

## Solution Assessment

No. Save-time Control UI validation is the right layer, but this implementation is narrower than the existing Telegram target contract and leaves UI error rendering, i18n metadata, and real dashboard proof incomplete.
