## Summary

This PR should stay open, but it is not merge-ready: the useful save-time validation is in the right area, yet the current regex would reject Telegram target forms that current runtime code and tests support, and the supplied proof does not exercise the real dashboard flow.

## What This Changes

The PR adds Control UI cron form validation and tests for Telegram announce `To` values, updates locale bundles with a new cron error string, and changes two cron command labels to use i18n strings.

## Best Possible Solution

Keep save-time validation, but make it mirror the Telegram runtime target contract, render announce `To` errors inline, and require redacted dashboard proof before merge.

## Reproduction Assessment

Yes, at source level: current main validates webhook `deliveryTo` only, then forwards announce `delivery.to` into `cron.add` or `cron.update`, matching the linked runtime failure. I did not run a live Telegram cron or full dashboard flow in this read-only review.

## Solution Assessment

No. Save-time validation is the right layer, but this implementation is too narrow for the existing Telegram target contract and does not surface the new announce `To` error in the field.
