## Summary

Keep this PR open: the underlying cron Telegram target UX bug still exists on current main and v2026.6.6, but this patch is not merge-ready because its numeric-only validation rejects runtime-supported Telegram topic targets and the new announce-mode error is not rendered inline.

## What This Changes

The PR adds Control UI cron form validation for Telegram announce To values, adds the related cron error string across locale bundles, and moves cron Command/CWD labels through i18n.

## Best Possible Solution

Align cron UI validation and copy with the existing Telegram target grammar, render announce To errors inline, add focused controller/view coverage, and require redacted dashboard proof before merge.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 only validate webhook delivery, while the PR's regex rejects Telegram target formats accepted by runtime tests and docs; I did not run the browser UI.

## Solution Assessment

No. Save-time validation is the right layer, but this implementation is too narrow because it does not mirror the runtime Telegram target grammar and does not render the new announce field error.
