## Summary

Keep open. The PR targets a real, source-reproducible gateway/outbound hook gap, but the branch is draft, lacks inspectable after-fix real behavior proof, and is based on obsolete delivery and hook-type shapes that no longer match current main.

## What This Changes

The branch adds optional agent/session identity to plugin message hook context, forwards it through outbound delivery, and makes gateway send return a guarded-denial error when `message_sending` cancels.

## Best Possible Solution

Rebase or replace the PR on current main so gateway send consumes durable `suppressed` outcomes, threads available outbound identity through the live hook contract without dropping existing fields, and coordinates with the canonical send-path issue.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main records hook-cancelled sends as durable suppression, but gateway send still turns non-`sent` results into `No delivery result`, and outbound `message_sending` still lacks session identity. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as written. The gateway-only direction may be acceptable, but the branch must be rebased onto `sendDurableMessageBatch`, `src/plugins/hook-message.types.ts`, and the current mapper/cancel-reason contracts before it is a maintainable fix.
