## Summary

Keep open. Current main and the latest release still run noninteractive repair-mode doctor during update finalization, and that repair path can replace an invalid non-plugin-local config wholesale from last-known-good; current code appears to mitigate the named Discord `allow` alias path, but the broader stale rollback and explicit restore-observability gap remain unresolved.

## What This Changes

Report that `openclaw update` can invoke post-update doctor repair that restores stale last-known-good config and discards intentional `openclaw.json` edits.

## Best Possible Solution

Define a policy-backed update/doctor recovery contract that repairs known legacy fields in place, avoids whole-file last-good restore when it would discard unrelated user-authored sections unless explicitly accepted, and emits durable audit/status/reason evidence whenever full restore remains possible.

## Reproduction Assessment

Yes by source inspection: current main still runs update-time repair-mode doctor and can restore invalid non-plugin-local config from last-good, but a live end-to-end update repro was not run and the named Discord `allow` trigger may now be migrated.

## Solution Assessment

Unclear: adding audit/status evidence is a safe part of the fix, but changing when whole-file last-good restore is allowed needs maintainer policy because it trades off against startup recovery and channel availability.
