## Summary

Keep open. Current main and the latest release still run repair-mode doctor during update finalization and still allow whole-file last-known-good restore for invalid non-plugin-local config; current code partially mitigates plugin-local schema drift and records observe-style audit metadata, but the central silent rollback policy remains unresolved.

## What This Changes

The issue asks OpenClaw to prevent or clearly surface `openclaw update` rolling back `openclaw.json` wholesale from a stale last-known-good snapshot when doctor repairs an invalid config.

## Best Possible Solution

Define a policy-backed update/doctor recovery contract that repairs known legacy fields in place, avoids silent whole-file rollback of unrelated user-authored sections, and emits durable audit/status/reason evidence whenever full restore remains allowed.

## Reproduction Assessment

Yes by source inspection: current main still runs repair-mode doctor during update finalization, and repair preflight can still restore invalid non-plugin-local config from last-known-good. A live end-to-end `openclaw update` repro was not run because this was a read-only review.

## Solution Assessment

Unclear: the issue's additive observability fixes are safe-looking, but changing when whole-file last-known-good restore is allowed is a recovery-policy choice because it trades off against startup/channel availability.
