## Summary

Keep open. Current main still has an automatic post-update doctor repair path and whole-file last-known-good recovery for invalid non-plugin-local config snapshots; current code mitigates some plugin-schema cases and records a restore-like `config.observe` audit record, but it does not clearly provide the requested explicit restore signal or policy guard, and the exact stale Discord `allow` trigger needs a focused current-main repro.

## What This Changes

The issue requests that `openclaw update` stop silently replacing user-edited config with stale last-known-good config, or at least make full-file restores prominent and auditable.

## Best Possible Solution

Define a narrow update/doctor recovery contract: migrate known plugin-owned legacy keys in place, avoid whole-file last-known-good restore when it would discard unrelated user-authored sections, and emit a distinct restore/status/audit signal whenever full restore is intentionally retained.

## Reproduction Assessment

No high-confidence live repro was run in this read-only review. Source confirms the post-update doctor repair and whole-file last-good restore path, but current update-time plugin-validation skip plus Discord `allow` migration means the exact reported trigger still needs a focused fixture or live repro.

## Solution Assessment

Unclear. A distinct audit event or reason artifact would improve forensics, but preventing rollback likely needs a maintainer decision on when doctor may do whole-file last-known-good restore versus targeted migration or an explicit operator-facing recovery path.
