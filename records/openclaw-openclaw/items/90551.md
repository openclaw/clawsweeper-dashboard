## Summary

Keep open. Current main and v2026.6.9 still run repair-mode doctor during update finalization, and doctor preflight can still replace an invalid non-plugin-local config wholesale from last-known-good before targeted legacy migrations run; the remaining fix is a maintainer policy decision around update-time recovery versus stale rollback.

## What This Changes

Prevent `openclaw update` post-update doctor recovery from silently replacing the current `openclaw.json` with a stale last-known-good snapshot and discarding intentional config changes.

## Best Possible Solution

Define a bounded update-time config recovery contract that preserves unrelated current config, limits whole-file last-good restore to true corruption or explicit recovery, and emits durable audit/status/reason evidence for every restore.

## Reproduction Assessment

Yes, at source level: current update finalization still runs repair-mode doctor, and repair preflight can still restore an invalid non-plugin-local config from last-known-good. I did not run a live end-to-end update repro in this read-only review.

## Solution Assessment

Unclear. Adding durable restore observability is only a partial fix; changing when whole-file last-good restore is allowed needs maintainer policy because it trades off against startup/channel availability recovery.
