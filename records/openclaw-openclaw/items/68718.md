## Summary

This PR should close because the useful durable-executor idea is real, but the submitted shape adds a second core execution substrate instead of using the now-landed detached-runtime plugin seam. Current main has the core contract needed for an external executor, while the remaining queue/worker implementation fits OpenClaw’s plugin/ClawHub boundary better than core.

## What This Changes

The branch adds a SQLite-backed `src/minions` job queue with CLI, Gateway protocol schemas, TaskRegistry shadow sync, doctor migration, spawn tracking, docs, tests, and a plugin SDK export, plus a separate install-security scanner symlink change.

## Best Possible Solution

Keep Tasks/TaskFlow canonical in core and move the durable queue/recovery worker into a ClawHub/plugin implementation on the current detached-runtime contract, adding only narrow core API seams if a plugin proves one is missing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the patch-level defects by source inspection of the PR head state-machine paths; the broader durable-executor request is a feature/product direction rather than a single current-main bug reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. The maintainable direction is a plugin-backed durable executor behind the detached-runtime seam, not a new core `src/minions` substrate with shadow-synced task state and a separate operator surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep spawn tracking from expiring live subagents:** `src/agents/tools/sessions-spawn-tool.ts:360-366`
  - body: The row inserted after a successful `sessions_spawn` has a 30s `lock_until`, but the original live subagent run never renews or completes that minion row. Once the lock expires, stall recovery can requeue and relaunch a normal long-running or already-finished subagent even without a gateway crash.
  - confidence: 0.88
- **[P1] Resolve parents after ignored child failures:** `src/minions/queue.ts:842-848`
  - body: `applyChildFailPolicy` only resolves the parent for `remove_dep`. For `ignore` and `continue`, the last failed or dead child leaves the parent in `waiting-children` because no later completion path calls `resolveParentIfDone`.
  - confidence: 0.9
- **[P1] Apply parent policy when stalled children dead-letter:** `src/minions/queue.ts:563-576`
  - body: When `handleStalled()` marks an expired active child as `dead`, it returns the dead job without running `applyChildFailPolicy`. A fan-out parent can therefore stay blocked in `waiting-children`, and `fail_parent` is skipped after the child exhausts stall recovery.
  - confidence: 0.86
- **[P1] Resolve parents after cancelling a child subtree:** `src/minions/queue.ts:384-390`
  - body: `cancelJob()` can cancel a non-root child and its descendants, but it never resolves that child's parent afterward. If the cancelled child was the last unfinished dependency, the parent remains stuck in `waiting-children`.
  - confidence: 0.83
- **[P2] Narrow workspace symlink trust in the scanner:** `src/plugins/install-security-scan.runtime.ts:213-221`
  - body: The install-security scan now allows `node_modules` symlinks to resolve outside the install root when they point at broad sibling workspace-member paths. That weakens the blocked-dependency scan and is unrelated to the minions feature, so it should be split or narrowed with focused tests.
  - confidence: 0.78
