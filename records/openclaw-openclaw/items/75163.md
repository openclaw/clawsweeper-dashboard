## Summary

Keep open. Current main still resolves aliases only against the full raw model string before parsing provider/model refs, so a value like `lmstudio-moe/MoE` can still become provider `lmstudio-moe` with raw model `MoE`; the issue is already paired with open closing PR #75198, so cleanup should wait for that PR or an equivalent shared resolver fix to merge.

## What This Changes

The issue reports that TUI or Control UI mid-session model switching can persist provider-qualified display aliases such as `lmstudio-moe/MoE` instead of canonical model IDs, causing LM Studio invalid-model errors.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach so `provider/Alias` resolves only when the provider prefix matches the configured alias target, with regression coverage for both direct resolver behavior and `sessions.patch` persistence. Avoid a UI-only fix because TUI, Control UI, slash-command, SDK, and other callers share the same session patch boundary.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, then call `sessions.patch` with `model: "lmstudio-moe/MoE"`; current source shows the whole-string alias lookup misses and the value is parsed as raw model `MoE`.

## Solution Assessment

Yes. The best fix is at the shared model resolver/session-patch boundary, because multiple UI and command surfaces persist session model overrides through the same gateway path.
