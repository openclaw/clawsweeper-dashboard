## Summary

Keep open. Current main still resolves only whole raw model strings as aliases before parsing provider/model refs, so a value like `lmstudio-moe/MoE` still reaches the session resolver as provider `lmstudio-moe` plus raw model `MoE`; the issue is also paired with open closing PR #75198, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that TUI and Control UI mid-session model switching resolve provider-qualified display aliases to canonical model IDs before persisting session model overrides.

## Best Possible Solution

Land or adapt the shared resolver approach from PR #75198 so `provider/Alias` resolves only when the provider prefix matches the configured alias target, with regression coverage for both direct resolver behavior and `sessions.patch` persistence.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, submit `sessions.patch` with `model: "lmstudio-moe/MoE"`, and current main follows the code path that parses `MoE` as the raw model id instead of resolving the alias.

## Solution Assessment

Yes. The best fix is at the shared model resolver/session-patch boundary; a TUI-only canonicalization would leave Control UI and other `sessions.patch` callers with divergent behavior.
