## Summary

Keep open. Current main at 7df025f45781fe57d12c84dd8ecdd67dc149ddaf still resolves aliases against the whole raw model string before provider/model parsing, so `lmstudio-moe/MoE` misses the configured `MoE` alias and can continue as raw model `MoE`; the linked open PR #75198 is the right implementation candidate and this issue should stay open until that or an equivalent fix lands.

## What This Changes

Model switching through TUI/Control UI should resolve provider-qualified display aliases to canonical model IDs before `sessions.patch` persists the session override.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach so provider-qualified aliases resolve only when the provider prefix matches the configured alias target, with focused coverage in `src/agents/model-selection.test.ts` and `src/gateway/sessions-patch.test.ts`.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, send `sessions.patch` with `model: "lmstudio-moe/MoE"`, and current main's resolver will treat `MoE` as the raw provider/model suffix instead of the canonical configured model ID.

## Solution Assessment

Yes. A shared resolver/session-patch fix is the narrowest maintainable solution; a TUI-only workaround would leave Control UI, slash-command, SDK, and other `sessions.patch` callers with divergent alias behavior.
