## Summary

Keep open. Current main still does not canonicalize provider-qualified display aliases in the shared model resolver before `sessions.patch` persists a session model selection, and the linked closing PR #75198 is still open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that TUI/Control UI mid-session model switching can pass a display alias such as `lmstudio-moe/MoE` or `MoE` through `sessions.patch` instead of resolving it to the canonical provider/model ID for LM Studio.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach so `provider/Alias` resolves to the canonical configured provider/model only when the provider prefix matches the alias target, with regression coverage for both resolver behavior and `sessions.patch` persistence. Avoid a TUI-only workaround because Control UI, slash-command, SDK, and other session-patch callers share the same boundary.

## Reproduction Assessment

Yes. A source-level reproduction is to configure a model alias such as `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"` and send a session patch with `model: "lmstudio-moe/MoE"`; current main checks only the whole string alias and then parses the suffix as raw model `MoE` instead of canonicalizing it.

## Solution Assessment

Yes. The narrow maintainable fix belongs in shared model resolution/session patching, because that is where the persisted provider/model override is validated and stored for both terminal and browser model switches.
