## Summary

Keep open. Current main still has the provider-qualified alias resolution gap on the sessions.patch path, and open PR #75198 is already the active closing implementation path for the same defect.

## What This Changes

The issue reports that Control UI/TUI mid-session model switching can persist a provider-qualified display alias such as `lmstudio-moe/MoE` instead of the canonical model ID, causing LM Studio to reject the next request.

## Best Possible Solution

Review and land or adapt #75198 so shared model resolution canonicalizes provider-qualified aliases before sessions.patch persists user model overrides, with resolver and sessions.patch regression coverage.

## Reproduction Assessment

Yes, source-level. With alias `MoE` configured for `lmstudio-moe/qwen3.6-35b-a3b`, a sessions.patch model value of `lmstudio-moe/MoE` misses the bare alias key and falls through as provider `lmstudio-moe` with model `MoE`.

## Solution Assessment

Yes. Fixing the shared resolver used by sessions.patch is the narrow maintainable path because it covers the Control UI picker and any other session-patch caller without LM Studio-specific logic.
