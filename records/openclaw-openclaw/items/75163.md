## Summary

Keep this issue open. Current main still shows the shared resolver matching aliases only against the whole raw model string before parsing provider/model, while `sessions.patch` persists the resolved provider/model override; the linked closing PR is still open, conflicting, and not merged, so the user-visible bug is not implemented on main.

## What This Changes

The issue asks for Control UI/TUI mid-session model switching to canonicalize display aliases such as `provider/Alias` before persisting the session model override.

## Best Possible Solution

Land a focused server-side fix at the shared model resolver and `sessions.patch` boundary, with regression coverage for provider-qualified display aliases and duplicate alias/provider cases.

## Reproduction Assessment

Yes, at source level rather than a live run here. Current main routes `sessions.patch` model updates through `resolveAllowedModelRef`, whose alias match does not cover `provider/Alias` before provider/model parsing, and the issue comments include real logs for that same boundary.

## Solution Assessment

Yes, a focused shared resolver plus session patch canonicalization fix is the right layer. Provider fallback or LM Studio-specific changes would not address the shared Control UI `sessions.patch` path.
