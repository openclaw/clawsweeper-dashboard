## Summary

Keep open. Current main still misses provider-qualified alias resolution, and #75198 is the open unmerged closing PR for the same resolver/session-patch defect.

## What This Changes

The issue asks for Control UI/TUI mid-session model switches to resolve configured display aliases like `MoE` to canonical provider model IDs before persisting the session override.

## Best Possible Solution

Land or adapt #75198 so provider-qualified aliases resolve in shared model selection before `sessions.patch` persists overrides, with resolver and session-patch regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure `MoE` as the alias for `lmstudio-moe/qwen3.6-35b-a3b`, then send `sessions.patch` with `model: "lmstudio-moe/MoE"`; current main misses the alias lookup and parses `MoE` as the provider-facing model.

## Solution Assessment

Is this the best way to solve the issue? Yes. Fixing the shared resolver/session-patch boundary is the narrowest maintainable solution because the Control UI model picker persists through that RPC and other session model switch paths reuse the same selection logic.
