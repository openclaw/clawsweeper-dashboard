## Summary

Keep open. Current main still resolves aliases only when the whole input matches the alias key, so `lmstudio-moe/MoE` can still parse as provider `lmstudio-moe` plus provider-facing model `MoE`; open PR #75198 is the paired closing fix candidate and should be handled before this issue is closed.

## What This Changes

The issue asks for TUI/Control UI mid-session model switches to resolve configured aliases to canonical provider model ids before persisting session overrides.

## Best Possible Solution

Land or adapt PR #75198 so provider-qualified aliases resolve canonically at the shared model resolver/session patch boundary with regression coverage.

## Reproduction Assessment

Yes. Configure alias `MoE` for `lmstudio-moe/qwen3.6-35b-a3b`, then patch a session with `model: "lmstudio-moe/MoE"`; current source shows the alias lookup misses and slash parsing preserves `MoE` as the provider-facing model id.

## Solution Assessment

Yes. Fixing the shared resolver/session-patch boundary is the narrowest maintainable solution because UI, TUI, `session_status`, slash-command, and SDK patch callers all converge there.
