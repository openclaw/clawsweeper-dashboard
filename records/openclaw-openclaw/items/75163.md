## Summary

Keep open. Current main still lacks provider-qualified alias canonicalization at the shared resolver/session-patch boundary, and open PR #75198 is already the active closing implementation candidate for this bug.

## What This Changes

Resolve provider-qualified model aliases selected mid-session by TUI/Control UI to canonical provider/model IDs before persisting session overrides.

## Best Possible Solution

Land or adapt PR #75198's shared resolver fix: when a raw session model is `provider/Alias`, resolve the suffix through the alias index only when the provider prefix matches the alias target, then persist canonical provider/model IDs through `sessions.patch` with focused resolver and gateway regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure alias `MoE` for `lmstudio-moe/qwen3.6-35b-a3b` and send `sessions.patch` with `model: "lmstudio-moe/MoE"`; current main parses that as provider `lmstudio-moe` plus raw model `MoE` instead of resolving the alias.

## Solution Assessment

Yes. The best fix direction is the shared resolver/session-patch boundary, because a UI-only normalization would leave `/model`, TUI, SDK, and other `sessions.patch` callers with divergent alias behavior.
