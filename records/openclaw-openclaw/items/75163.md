## Summary

Keep open: current main still lacks provider-qualified alias canonicalization in the shared model resolver, and the open closing PR that targets this issue has not merged. The May 16 repro comment usefully broadens the same Control UI to sessions.patch gap from local aliases to provider/display values, so this is still a live issue rather than cleanup-close material.

## What This Changes

The issue asks OpenClaw to resolve Control UI/TUI model-switch aliases to canonical provider/model IDs before persisting session model overrides.

## Best Possible Solution

Refresh and review the linked server-side resolver fix, or land an equivalent focused change that canonicalizes provider-qualified aliases before sessions.patch stores the model override.

## Reproduction Assessment

Yes, at source level. Current main passes the Control UI/TUI model value through sessions.patch into resolveAllowedModelRef, and the shared resolver only exact-matches the whole raw alias before parsing provider/model, so provider-qualified aliases like lmstudio-moe/MoE are not canonicalized.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the shared server-side model resolver before sessions.patch persistence, not only in UI state, so all gateway callers share the same canonicalization rule.
