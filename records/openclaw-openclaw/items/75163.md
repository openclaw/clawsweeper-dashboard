## Summary

Keep open. Current main still performs only whole-string alias resolution before parsing provider/model refs, so a value like `lmstudio-moe/MoE` can still fall through as provider `lmstudio-moe` plus raw model `MoE`. Open PR #75198 is the linked closing implementation, so this issue should stay open until that PR or an equivalent shared resolver fix lands.

## What This Changes

The issue requests that session model switches resolve provider-qualified display aliases to canonical provider/model IDs before persisting the session override.

## Best Possible Solution

Land or adapt PR #75198's shared resolver change so `provider/Alias` resolves only when the provider prefix matches the configured alias target, with regression coverage at both the resolver and `sessions.patch` persistence boundary.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, then patch a session with `model: "lmstudio-moe/MoE"`; current main checks alias key `lmstudio-moe/moe`, misses configured alias `moe`, and falls through to provider/model parsing.

## Solution Assessment

Yes. The shared resolver plus `sessions.patch` boundary is the narrow maintainable fix because TUI and Control UI both forward selected model values there; a TUI-only workaround would leave other session-patch callers inconsistent.
