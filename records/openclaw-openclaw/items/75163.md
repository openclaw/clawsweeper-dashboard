## Summary

Keep open. Current main still treats a provider-qualified display alias such as `lmstudio-moe/MoE` as provider `lmstudio-moe` plus raw model `MoE`, and the issue is already paired with open closing PR #75198, so the issue should remain open until that PR or an equivalent shared resolver fix lands.

## What This Changes

The issue asks OpenClaw to canonicalize TUI/Control UI mid-session model switches from configured display aliases to real provider/model IDs before persisting the session override.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach so `provider/Alias` resolves only when the provider prefix matches the configured alias target, with regression coverage in the resolver and `sessions.patch` persistence boundary.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, send `sessions.patch` with `model: "lmstudio-moe/MoE"`, and current main will parse `MoE` as the raw model instead of resolving the alias.

## Solution Assessment

Yes. The best fix is at the shared model resolver/session-patch boundary; a TUI-only workaround would leave Control UI, `/model`, SDK, and other `sessions.patch` callers with divergent alias behavior.
