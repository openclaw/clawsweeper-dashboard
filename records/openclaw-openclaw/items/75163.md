## Summary

Keep open. Current main still resolves aliases only against the whole raw model string before parsing provider/model refs, so a session switch value like `lmstudio-moe/MoE` is still parsed as provider `lmstudio-moe` with raw model `MoE`. The issue is also paired with open closing PR #75198, so it should remain open until that PR or an equivalent shared resolver fix lands.

## What This Changes

The issue asks for TUI and Control UI mid-session model switching to resolve display aliases to canonical provider/model IDs before session persistence and provider dispatch.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach so `provider/Alias` resolves only when the provider prefix matches the configured alias target, with regression coverage for both resolver behavior and `sessions.patch` persistence. Avoid a UI-only workaround because other callers share the same session patch boundary.

## Reproduction Assessment

Yes. Configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, then send `sessions.patch` with `model: "lmstudio-moe/MoE"`; current main source shows that value is parsed as raw model `MoE` rather than resolving to the canonical configured model ID.

## Solution Assessment

Yes. The best fix is at the shared model resolver/session-patch boundary so TUI, Control UI, slash-command, SDK, and other `sessions.patch` callers get consistent alias canonicalization.
