## Summary

Keep open. Current main still resolves aliases only when the whole raw model string matches an alias, so a provider-qualified value like `lmstudio-moe/MoE` falls through to provider/model parsing instead of resolving the suffix alias; open PR #75198 already references this issue with closing syntax and is the right repair path.

## What This Changes

The issue asks for TUI/Control UI mid-session model switches to resolve configured display aliases to canonical provider/model IDs before updating session state.

## Best Possible Solution

Land or adapt PR #75198’s shared resolver fix with regression coverage for provider-qualified aliases through `resolveAllowedModelRef` and `sessions.patch`.

## Reproduction Assessment

Yes. Configure a canonical allowed model with alias `MoE`, then send `sessions.patch` with `model: "lmstudio-moe/MoE"`; current source shows the raw string misses alias lookup and parses `MoE` as the provider model id.

## Solution Assessment

Yes. Resolving provider-qualified aliases in the shared model resolver/session-patch path is the narrowest maintainable fix; a TUI-only or LM Studio-specific workaround would leave other session patch callers inconsistent.
