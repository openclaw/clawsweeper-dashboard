## Summary

Keep open. Current main and v2026.5.7 still have the unguarded persisted-snapshot env override path, and the linked unmerged fix PR at https://github.com/openclaw/openclaw/pull/79173 uses closing syntax but is still open/conflicting; this issue should stay open until that lands or is replaced.

## What This Changes

The issue reports that `skills.entries.gh-issues.apiKey` SecretRefs can still be evaluated during WhatsApp/embedded sessions from stale skill snapshots, even after the skill entry is disabled.

## Best Possible Solution

Land or replace the focused guard so disabled skill entries are skipped before env override application from both live entries and persisted snapshots, with regression coverage for raw disabled `apiKey` SecretRefs.

## Reproduction Assessment

Yes, source-level. Current main can reuse a persisted snapshot containing `gh-issues` while current config has `enabled:false` and a raw `apiKey` SecretRef; `applySkillEnvOverridesFromSnapshot` reaches strict `normalizeResolvedSecretInputString` without an `enabled:false` guard.

## Solution Assessment

Yes. The linked PR's approach is the narrow maintainable fix: reuse the existing disabled-skill contract at the env-injection boundary instead of adding channel-specific behavior or a startup migration.
