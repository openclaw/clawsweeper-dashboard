## Summary

Keep open. Current main still has a source-visible path where a persisted skill snapshot can evaluate `skills.entries.<skill>.apiKey` even after the entry is disabled, and disabled skill SecretRefs are intentionally left raw by the runtime resolver; that matches the reporter's workaround and shows the prior skill SecretRef fix did not cover this case.

## What This Changes

The issue reports that `skills.entries.gh-issues.apiKey` remains an unresolved SecretRef during WhatsApp lane sessions, crashing incoming messages even when the skill entry is disabled.

## Best Possible Solution

Keep the existing SecretRef runtime-snapshot design, but make skill env injection and snapshot reuse honor the current skill enabled state so stale snapshots cannot evaluate raw disabled skill SecretRefs.

## Reproduction Assessment

Yes, source-level: a persisted snapshot containing `gh-issues`, current config with `skills.entries.gh-issues.enabled:false` plus a raw SecretRef, and no host `GH_TOKEN` reaches `applySkillEnvOverridesFromSnapshot` and throws during strict apiKey normalization. I did not live-run WhatsApp in this read-only review.

## Solution Assessment

No, the current main implementation is incomplete for this report because it skips disabled skills while building new snapshots but not while applying env overrides from an existing snapshot. The narrow fix should make the snapshot env path skip disabled entries and add regression coverage for the persisted-snapshot SecretRef case.
