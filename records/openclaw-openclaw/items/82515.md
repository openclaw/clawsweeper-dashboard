## Summary

Current main still matches the reported failure path: the Matrix plugin relies on matrix-js-sdk sync behavior that requests MSC4222 state-after, and the upstream SDK only initializes outbound crypto when an m.room.encryption event is present in the final-state events. This should stay open as a narrow Matrix plugin bug with a plugin-local compatibility mitigation.

## What This Changes

Prevent Matrix E2EE outbound setup from being skipped when MSC4222 state_after omits m.room.encryption for a newly joined encrypted room.

## Best Possible Solution

Fix this in the Matrix plugin by avoiding or safely gating MSC4222 state-after sync requests until the SDK/homeserver path reliably preserves m.room.encryption for crypto setup, with regression coverage for the /sync request behavior.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: current main lets matrix-js-sdk request MSC4222 state_after, and upstream only calls onCryptoEvent for encryption events in that final-state set. I did not run a live homeserver reproduction in this read-only pass.

## Solution Assessment

Yes, a plugin-local compatibility mitigation that stops requesting incomplete MSC4222 state_after data is the narrowest maintainable fix. Adding a core policy knob or retrying only after send failure would be broader and less direct.
