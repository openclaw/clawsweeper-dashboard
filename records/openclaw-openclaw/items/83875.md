## Summary

Keep this PR open for normal maintainer review: the current head appears to address the earlier unguarded cwd path and I found no blocking diff bug, but the change remains merge-sensitive because it alters managed updater cwd semantics and lacks live managed-service upgrade proof.

## What This Changes

This PR hardens managed update handoff and update-runner cwd handling for deleted cwd/package-root cases, with regression tests and a changelog entry.

## Best Possible Solution

Land the durable cwd handling once maintainers are comfortable with the cwd contract change and have the managed-service upgrade proof they need for the supported service platforms.

Yes for source-level reproduction: current main calls process.cwd() in update.run root resolution and update-runner candidate gathering before the durable handoff path. I did not run a live macOS LaunchAgent repro in this read-only review.

## Solution Assessment

Yes, with an upgrade-proof caveat: using a prepared existing cwd is a narrow maintainable fix for uv_cwd during managed handoff. The main remaining question is not code shape but whether maintainers want service-level proof before accepting the cwd semantics change.
