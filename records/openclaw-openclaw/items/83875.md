- `P1`: The linked production report shows a managed update path that can stop a macOS gateway service and then fail the restart handoff.

## Summary

Keep open for maintainer review: the PR targets a real managed-update availability bug, current main still has source-reproducible `process.cwd()` crash paths, and the patch plus terminal proof are focused. The remaining question is whether maintainers require live managed-service smoke before accepting the cwd behavior change.

## What This Changes

The PR changes managed update handoff, early startup, npm install env, and update-runner cwd handling so deleted cwd/package-root states fall back to existing directories instead of crashing.

## Best Possible Solution

Land the targeted cwd hardening after maintainers either accept the Linux smoke and focused regression tests or add a managed-service update smoke for the service platforms they want covered.

## Reproduction Assessment

Yes for source-level reproduction: current main still calls `process.cwd()` in the update and startup paths that the production report shows crashing with `ENOENT: uv_cwd`. I did not run the live macOS LaunchAgent scenario in this read-only review.

## Solution Assessment

Yes: hardening the handoff helper, early `isMainModule`, update-runner, and npm env cwd paths is a narrow maintainable fix for the reported failure. The only safer alternative would be preserving package-root cwd as the first choice when it still exists, which is a maintainer compatibility decision rather than a definite correctness bug.
