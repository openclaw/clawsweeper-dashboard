## Summary

Keep this PR open: it targets a real Telegram restart-window media-loss path and is linked to the canonical open issue, but the patch is not merge-ready because the durable retry classifier is broader than the media fetch contract supports and the PR has only test/mock proof, not after-fix Telegram restart proof.

## What This Changes

The PR adds a Telegram media-fetch retry classifier, records transient single-message media download failures as `failed-retryable` for spooled replay, suppresses repeated retry notices on replay, and adds focused Telegram tests.

## Best Possible Solution

Keep the Telegram-owned durable retry direction, but narrow retry classification to true transient or shutdown/abort fetch failures and add redacted live Telegram restart proof before merge.

## Reproduction Assessment

Yes, source-level: current main's single-message media catch path sends a best-effort warning and returns, while bot-core only keeps a spooled replay uncompleted when a failed-retryable result is recorded. I did not run a live Telegram restart-window repro in this read-only review.

## Solution Assessment

No as submitted. The owner boundary is plausible, but the classifier retries all `fetch_failed` errors rather than only transient/shutdown failures, and the PR needs real Telegram restart proof before merge.
