## Summary

Keep open: current main has ACP identity/recovery work that closed the linked normal-turn report, but it does not implement this PR's config-option retry path. The branch still needs a careful port onto current main and real behavior proof before merge.

## What This Changes

The PR adds ACP persistent-session recovery retries for bound sessions, expands stale-binding detection for `ACP_TURN_FAILED`, and adds ACP manager/dispatch tests.

## Best Possible Solution

Port a minimal retry around current main's `setSessionConfigOption` implementation, preserving status-derived config keys and `wireKey` mapping, keep the narrowed stale-binding predicate, remove the dead fallback throw, and attach redacted live proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection shows ACP runtime-option commands call `setSessionConfigOption`, current main has no stale persistent-session retry around that method, and the PR adds mocked coverage for the path.

## Solution Assessment

No, not as currently proposed. The retry seam is plausible, but it must wrap current main's config-key resolution and remove the unreachable fallback before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve current config-key resolution inside the retry:** `src/acp/control-plane/manager.core.ts:580-608`
  - body: The retry loop restores an older `setSessionConfigOption` path: it resolves capabilities without `includeStatusConfigOptionKeys`, validates the user key directly, and sends that raw key to `runtime.setConfigOption`. Current main maps aliases such as `thinking` to advertised keys like `effort` and learns keys from status, so applying this diff would reject or mis-send those config updates.
  - confidence: 0.88
- **[P3] Remove the unreachable retry fallback throw:** `src/acp/control-plane/manager.core.ts:651-654`
  - body: The two-attempt loop returns on success, continues only after the first retry, or rethrows from the catch block; on the second attempt `shouldRetry` is false, so execution cannot fall through to this synthetic `ACP_TURN_FAILED`. Keeping it can obscure the real second-attempt error.
  - confidence: 0.82
