## Summary

Keep open: the PR still has a plausible missing retry improvement for ACP config-option commands, but it is not merge-ready because the branch conflicts with current main, drops current config-key resolution behavior, and lacks required real behavior proof.

## What This Changes

The branch changes ACP persistent-session recovery for `--bind here` sessions by adjusting resume-state clearing, adding a `setSessionConfigOption` retry path, expanding stale-binding detection, and adding manager/dispatch tests.

## Best Possible Solution

Port only the missing config-option retry onto current main's `setSessionConfigOption`, preserving status-derived config keys and `wireKey` mapping, keep the narrowed stale-binding predicate, remove the dead fallback throw, and attach redacted live proof.

## Reproduction Assessment

No live high-confidence reproduction was run for the original bug on current main; the linked issue is already closed with maintainer live Docker proof for the normal-turn path. Source inspection gives a high-confidence reproduction for the PR regression: applying the branch would drop current main's status-derived config-key and `wireKey` path around `setSessionConfigOption`.

## Solution Assessment

No, not as currently proposed. A retry around `setSessionConfigOption` is plausible, but it needs to wrap current main's config-key contract instead of restoring the older raw-key path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve config-key mapping inside the retry:** `src/acp/control-plane/manager.core.ts:580-608`
  - body: The retry loop resolves capabilities without `includeStatusConfigOptionKeys`, validates the raw `key`, and sends that raw `key` to the runtime. Current main maps aliases such as `thinking` to advertised keys like `effort` and learns keys from status, so this would reject or mis-send valid ACP config updates after rebase.
  - confidence: 0.88
- **[P3] Remove the unreachable retry fallback throw:** `src/acp/control-plane/manager.core.ts:651-654`
  - body: The two-attempt loop returns on success, continues only after the first retry, or rethrows from the catch block; on the second attempt `shouldRetry` is false, so execution cannot fall through to this synthetic `ACP_TURN_FAILED`. Keeping it can obscure the real second-attempt error.
  - confidence: 0.82
