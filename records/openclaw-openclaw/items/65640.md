## Summary

Keep this PR open. Current main has adjacent ACP persistent-session recovery for normal turns and the original linked bug was closed after live follow-up validation, but main still lacks the PR's bounded fresh-handle retry around `/acp model`/`setSessionConfigOption()` and still only treats `ACP_SESSION_INIT_FAILED` as a stale-binding cleanup signal. The current PR head appears to have addressed the earlier broad `ACP_TURN_FAILED` cleanup and metadata-write failure concerns; only a non-blocking dead fallback throw remains.

## What This Changes

The PR adds ACP persistent-session recovery for config-option commands, adjusts stale resume-state clearing, broadens stale-binding cleanup to narrowly matched `ACP_TURN_FAILED` resume failures, and adds manager/dispatch tests.

## Best Possible Solution

Review and land this PR or port its narrow remaining behavior onto main: reuse the existing fresh-handle retry seam for `setSessionConfigOption()`, keep the `ACP_TURN_FAILED` stale-binding predicate as specific as the recoverable persistent-session guard, preserve metadata-write failure signaling, and remove the unreachable fallback throw during cleanup.

## Reproduction Assessment

Yes for the remaining source-level gap: current main's `/acp model` path calls `setSessionConfigOption()` once without fresh-handle retry, and the PR adds a focused unit reproduction for a persistent-session `Resource not found` failure. A live current-main reproduction was not run in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: it reuses `prepareFreshHandleRetry()` and matches the existing persistent-session resume-error predicate. The only safer cleanup is to remove the unreachable fallback throw after the retry loop before merge.
