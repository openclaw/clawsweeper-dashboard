## Summary

Keep this PR open, but it needs changes before merge: the recovery direction is useful, yet the branch is stale against current main's runtime config-option handling and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds ACP persistent-session recovery retry for config-option commands, relaxes stale resume-state clearing, expands stale-binding detection for ACP_TURN_FAILED, and adds ACP manager/dispatch tests.

## Best Possible Solution

Port the retry onto current main's `setSessionConfigOption()` while preserving status-derived config-key mapping, remove the dead fallback throw, and add redacted live proof for the bound-session recovery path.

## Reproduction Assessment

No live high-confidence current-main reproduction was established in this read-only review. The linked issue has concrete `/acp spawn --bind here` plus `/acp model` steps, and source inspection shows the PR targets a real config-option retry gap, but current main also has newer ACPX/session-identity recovery work.

## Solution Assessment

No, not as currently proposed against main. The retry seam is a reasonable narrow fix, but the branch must preserve current config-key resolution and remove the unreachable fallback before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve current config key mapping in the retry loop:** `src/acp/control-plane/manager.core.ts:591-610`
  - body: The retry rewrite is based on an older `setSessionConfigOption()` and drops current main's status-derived config-key lookup plus `wireKey` mapping. After rebasing, this would regress backends that expose keys only via status or require aliases such as `thinking` -> `effort`, so the retry should wrap the current main logic instead of restoring raw `key` handling.
  - confidence: 0.86
- **[P3] Remove the unreachable retry fallback throw:** `src/acp/control-plane/manager.core.ts:651-654`
  - body: The two-attempt loop returns on success, continues from the first retry, or rethrows from the catch block; the throw after the loop should not be reachable and would mislead future debugging. Restructure the loop or remove the synthetic `ACP_TURN_FAILED`.
  - confidence: 0.82
