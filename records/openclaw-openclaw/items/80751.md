## Summary

Keep open. The PR is a focused SSRF-policy change and now supplies real terminal proof, but it is draft/security-sensitive and the diff currently ignores explicit `request.allowPrivateNetwork: false` by turning every custom/local configured base URL into exact-origin trust.

## What This Changes

The PR adds exact-origin `allowedOrigins` SSRF trust for custom/local model provider `baseUrl` requests, resolves that trust per fetch URL, exports a helper through the SDK runtime, and adds regression tests.

## Best Possible Solution

Approve the SSRF trust boundary first, preserve explicit-deny semantics with regression coverage, then land the exact-origin provider trust with aligned docs and focused live/self-hosted-provider proof.

## Reproduction Assessment

Yes. Source inspection shows current main does not grant non-loopback custom provider base URLs private-host trust without `request.allowPrivateNetwork`, and the review finding is source-reproducible because the PR sees only a collapsed boolean and cannot distinguish default false from explicit false.

## Solution Assessment

No, not yet. Exact-origin trust is a plausible narrower direction than broad private-network opt-in, but the implementation must preserve explicit `false` opt-out and receive maintainer/security approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit private-network denial:** `src/agents/provider-transport-fetch.ts:458-462`
  - body: This enables exact-origin trust for every `custom` or `local` provider base URL even when the operator set `request.allowPrivateNetwork: false`. `resolveProviderRequestPolicyConfig` collapses default false and explicit false into the same boolean, so the transport cannot tell an unset default from the documented opt-out; loopback/LAN exact-origin requests would still be allowed via `allowedOrigins`. Please carry an explicit-deny signal or skip origin trust when the request explicitly sets false.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The patch intentionally relaxes SSRF handling for configured custom/local model endpoints and currently bypasses the documented explicit-deny posture.

Concerns:

- **[medium] Explicit private-network denial is bypassed:** `src/agents/provider-transport-fetch.ts:458`
  - body: Because the new exact-origin trust is keyed only on `endpointClass`, an operator's `request.allowPrivateNetwork: false` no longer prevents private loopback/LAN access for a matching configured base URL. That weakens an existing security control and should be fixed before approval.
  - confidence: 0.91
- **[medium] Approve the new exact-origin trust boundary:** `src/agents/provider-transport-fetch.ts:458`
  - body: Allowing custom/local provider origins to bypass private-IP checks may be the right product behavior, but it is a security policy change that should be explicitly approved for redirects, proxies, API-family scope, and SDK exposure.
  - confidence: 0.86
