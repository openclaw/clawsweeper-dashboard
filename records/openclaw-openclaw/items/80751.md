## Summary

Keep open. The PR is still draft, changes the model-provider SSRF trust boundary, and the diff still bypasses an explicit private-network denial for custom/local provider origins.

## What This Changes

The PR adds exact-origin `allowedOrigins` SSRF trust for custom/local model provider `baseUrl` requests, wires it through guarded fetch and SDK policy merging, and adds regression tests.

## Best Possible Solution

Approve the SSRF trust boundary first, preserve explicit-deny semantics with regression coverage, then land exact-origin provider trust with docs aligned.

## Reproduction Assessment

Yes. Source inspection shows current main does not grant non-loopback custom provider base URLs private-host trust without `request.allowPrivateNetwork`, and the review finding is source-reproducible because the PR cannot distinguish default false from explicit false.

## Solution Assessment

No, not yet. Exact-origin trust is a plausible narrower direction than broad private-network opt-in, but this implementation must preserve explicit `false` opt-out and receive maintainer/security approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit private-network denial:** `src/agents/provider-transport-fetch.ts:458-462`
  - body: This enables exact-origin trust for every `custom` or `local` provider base URL even when the operator set `request.allowPrivateNetwork: false`. The transport only sees the collapsed `allowPrivateNetwork` boolean, so it cannot distinguish an unset default from an explicit denial; matching loopback/LAN origins would still be allowed through `allowedOrigins`. Carry an explicit-deny signal or skip origin trust when false was configured.
  - confidence: 0.91
