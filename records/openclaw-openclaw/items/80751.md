## Summary

Keep open. This is an active draft PR for a real provider-transport problem, but it changes the SSRF trust boundary and still needs maintainer/security-policy review plus after-fix proof from a real local/self-hosted provider setup.

## What This Changes

The PR adds exact-origin SSRF trust for custom/local model provider base URLs, plumbs an `allowedOrigins` policy through fetch guard and SDK helpers, and adds targeted regression tests.

## Best Possible Solution

Decide the SSRF trust boundary first; if exact-origin trust is approved, land the narrow model-transport/fetch-guard implementation with regression tests, docs or doctor guidance for remaining opt-in cases, and redacted live proof against a local/self-hosted provider.

## Solution Assessment

Unclear. The exact-origin approach is more careful than host-only trust and has useful tests, but it is still a security-policy change and the PR itself asks maintainers whether the boundary and SDK exposure are acceptable.
