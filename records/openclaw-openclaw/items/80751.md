## Summary

Keep open for maintainer/security review. Current main lacks exact-origin private-provider trust, and the final PR head appears to address the earlier explicit-deny defect with real terminal proof, but this is still a draft SSRF trust-boundary change with open maintainer questions about policy scope and SDK exposure.

## What This Changes

The branch adds exact-origin SSRF trust for custom/local model-provider `baseUrl` HTTP requests, preserves explicit opt-out, blocks metadata/link-local trust, and updates provider docs, SDK policy helpers, and regression tests.

## Best Possible Solution

Approve or reject the exact-origin SSRF policy and SDK surface explicitly, then either land this scoped HTTP guarded-fetch implementation with its regression proof or keep the linked issue open for a narrower follow-up such as WebSocket handling.

## Reproduction Assessment

Yes. Source inspection of current main shows custom/local provider fetches without `request.allowPrivateNetwork` only get fake-IP hostname policy, while the SSRF guard still blocks private literal/resolved addresses unless the policy grants private-network or hostname trust.

## Solution Assessment

Unclear until maintainer/security approval. The final branch is a narrower implementation than broad `request.allowPrivateNetwork: true` and preserves explicit `false`, but the trust boundary and public SDK helper are policy decisions, not purely mechanical fixes.
