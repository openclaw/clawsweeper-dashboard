## Summary

Keep open. The PR is active, draft, and security-sensitive: it now has credible after-fix proof for the guarded model-fetch path, but the exact-origin private-provider trust boundary still needs maintainer/security-policy approval before merge.

## What This Changes

The PR adds exact-origin SSRF trust for custom/local provider base URLs, resolves that trust per request URL in the fetch guard, exposes the policy helper through the plugin SDK runtime, and adds regression tests.

## Best Possible Solution

Approve or reject the exact-origin SSRF trust boundary first; if approved, land the narrow transport/fetch-guard change with aligned docs and the supplied proof plus changed-surface checks.

## Reproduction Assessment

Yes, by source inspection: current main does not add exact-origin or allowed-hostname trust for non-loopback private custom provider base URLs unless `request.allowPrivateNetwork` is enabled, so the guarded fetch path rejects those private targets. I did not run a live current-main LM Studio or vLLM reproduction in this read-only review.

## Solution Assessment

Unclear. The exact-origin design is narrower than a host-only or broad private-network opt-in and has useful regression coverage, but the PR itself correctly asks for maintainer/security approval on the boundary and SDK exposure before this can be called the best fix.
