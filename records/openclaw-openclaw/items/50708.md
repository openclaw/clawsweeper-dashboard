## What This Changes

The PR adds a running-state guard in `formatRuntimeStatus` so daemon last-exit status and reason details are omitted from human-readable runtime status when the service reports running.

## Best Possible Solution

Refresh the current-main formatter so running services suppress stale last-exit details while non-running services keep signal labels and diagnostics, then add focused formatter coverage plus redacted real systemd proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `readSystemdServiceRuntime` can return `status: "running"` with parsed `ExecMainStatus` and `ExecMainCode`, and `formatRuntimeStatus` currently renders those fields into gateway status output.

## Solution Assessment

No, not as submitted. Formatter-level suppression is the right boundary, but the branch needs to guard the current `formatLastExitStatus(...)` helper rather than reverting to raw exit formatting, and it needs focused coverage plus real behavior proof.
