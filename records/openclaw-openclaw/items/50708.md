## Summary

Keep this PR open. Current main still renders stale systemd exit details for a running service, but the PR branch is dirty against main and its formatter change predates the current signal-aware last-exit formatter, so it needs a rebase and focused tests before merge.

## What This Changes

The PR changes daemon runtime status formatting to hide last-exit status and reason fields when the runtime status is `running`.

## Best Possible Solution

Land a rebased formatter-level fix that hides last-exit status and reason only in human status output for running services, leaves raw systemd runtime fields available for diagnostics, preserves signal-label formatting for non-running states, and adds focused `runtime-format` regression tests.

## Reproduction Assessment

Yes. A direct formatter input with `status: "running"`, `state: "active"`, `lastExitStatus: 1`, and `lastExitReason: "exit-code"` reproduces the stale display on current main without needing a live systemd host.

## Solution Assessment

Yes in direction, but not as the current branch. Formatter-level suppression is the narrow maintainable fix; the rebased patch should guard the existing `formatLastExitStatus(runtime.lastExitStatus)` path rather than restoring raw last-exit formatting.
