## Summary

Keep open: the underlying status-output bug is still present on current main, and this PR has the right formatter-level direction, but the branch is dirty against current main and would lose current signal-aware exit labels if rebased without care.

## What This Changes

The PR changes `src/daemon/runtime-format.ts` so formatted runtime status hides `lastExitStatus` and `lastExitReason` when `runtime.status` is `running`.

## Best Possible Solution

Land a rebased formatter-level fix that hides last-exit status and reason only in human runtime status output for running services, preserves raw systemd runtime fields and signal labels for non-running states, adds focused `runtime-format` regression tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. A runtime object with `status: "running"`, `state: "active"`, `lastExitStatus: 1`, and `lastExitReason: "exit-code"` still renders stale exit details on current main because `formatRuntimeStatus` appends those fields unconditionally.

## Solution Assessment

Yes in direction, but no as the current branch. Formatter-level suppression is the narrowest maintainable fix because restart and onboarding diagnostics can still consume raw runtime fields; the branch needs a rebase that wraps the guard around `formatLastExitStatus(...)`.
