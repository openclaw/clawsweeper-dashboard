## What This Changes

The PR updates the daemon runtime status formatter to hide last-exit status and reason when the service status is running.

## Best Possible Solution

Land a formatter-level fix that hides only stale human-readable last-exit details for running services while preserving raw systemd runtime data, non-running diagnostics, signal labels, tests, and changelog entry.

## Reproduction Assessment

Yes. From source inspection, `formatRuntimeStatus({ status: "running", state: "active", subState: "running", lastExitStatus: 1, lastExitReason: "exit-code" })` still renders the stale exit details on current main because the formatter appends those fields unconditionally.

## Solution Assessment

Yes in direction, but not as the current branch. Formatter-level suppression is the narrowest maintainable solution because restart and onboarding diagnostics can still consume raw runtime fields, but the guard must be rebased around `formatLastExitStatus(...)`.
