## Summary

Keep open: the stale systemd status bug is still present on current main and this PR remains the canonical linked fix path, but the branch needs real Linux/systemd behavior proof and a small current-main regression fix before merge.

## What This Changes

The PR suppresses `lastExitStatus` and `lastExitReason` in daemon runtime status formatting whenever the service runtime reports `status: "running"`.

## Best Possible Solution

Land a refreshed version that keeps raw runtime fields available, suppresses stale exit details only in the human status line while running, preserves stopped/failed diagnostics, and covers both running and non-running cases.

## Reproduction Assessment

Yes, at source level: current main reads systemd active state plus exit fields and the shared formatter prints exit details whenever present, so a running runtime with stale exit fields will display them. I did not establish a live Linux/systemd reproduction in this read-only review.

## Solution Assessment

Partly yes: moving suppression into the formatter is the right boundary because diagnostics can still access raw runtime fields. The branch still needs to be refreshed so the non-running path keeps current main's signal-aware exit formatting, plus focused coverage and real behavior proof.
