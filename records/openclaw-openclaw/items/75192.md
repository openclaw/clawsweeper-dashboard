## Summary

Keep open: current main still lacks a WebChat affordance to create a new session for a selected non-current configured agent, while the backend and frontend session helper already accept `agentId`.

## What This Changes

The issue requests a WebChat control for creating a new chat session with a user-selected configured agent instead of only using the current/default agent.

## Best Possible Solution

Add a focused WebChat New Chat agent picker that lists configured agents, calls the existing `sessions.create({ agentId })` flow through `createSessionAndRefresh`, switches to the returned session, and leaves the broader sidebar redesign to #49828.

## Reproduction Assessment

Yes. Configure multiple agents, open WebChat on one agent session, and use the current New session control; source and tests show it creates a dashboard session for the active session's agent with no selector for another configured agent.

## Solution Assessment

Yes. Reusing the existing `agentsList` metadata and `sessions.create` `agentId` contract is the narrowest maintainable fix; no new backend endpoint appears necessary.

## Repair Work Prompt

Implement a focused Control UI/WebChat New Chat agent selector for https://github.com/openclaw/openclaw/issues/75192. Current main has a plus/New session control that calls `createChatSession()` and passes the active session's agent to `createSessionAndRefresh`; add a user-visible way to choose another configured agent from existing `state.agentsList`/identity metadata and create a dashboard session for that selected `agentId`. Reuse `createSessionAndRefresh` and the existing `sessions.create({ agentId })` backend contract; do not add a new backend endpoint. Preserve the current busy guards, stale-response protection, draft/attachment preservation, and switch-to-created-session behavior. Avoid broad sidebar/navigation redesign; #49828 tracks that separately. If a selected agent differs from the active session's agent, do not create an accidental cross-agent parent linkage unless the existing contract clearly supports it. Add focused regression tests for selecting a non-current agent and for preserving the existing current-agent default path. Add a user-facing `CHANGELOG.md` entry if the PR changes WebChat behavior.
