## Summary

Keep open. Current main still exposes a WebChat New Session button that resets the active chat via `/new`, while the existing agent/session selectors only switch to existing sessions. The Gateway already supports `sessions.create` with `agentId`, so the remaining gap is the Control UI affordance; adjacent PR #73725 is open and unmerged, so it does not implement this request on main.

## What This Changes

Request to let WebChat users choose a configured agent when creating a new chat session, switch to the new session, and show agent identity in the chat/session UI.

## Best Possible Solution

Layer a selected-agent New Chat picker onto the generic WebChat create/switch flow, reusing `agents.list` and `sessions.create({ agentId })`, then switch to the returned session key and refresh history/session/tool state without adding a new Gateway endpoint unless a concrete missing contract appears.

## Reproduction Assessment

Yes. Configure multiple agents, open the Control UI chat, and use the visible New session button: current main routes that path to `/new` in the active session and has no visible UI path that passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, with sequencing. The narrow maintainable fix is a UI-only addition over existing `agents.list` and `sessions.create` contracts, preferably folded into or staged after #73725 rather than creating a parallel WebChat create-session implementation.
