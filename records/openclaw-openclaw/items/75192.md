## Summary

Keep open. Current main has the WebChat New Chat plumbing and the backend accepts `agentId`, but the UI still creates a new session for the currently active session's agent and does not expose a picker for another configured agent.

## What This Changes

The issue requests a WebChat New Chat flow that lets users choose any configured agent and start a new session for that agent.

## Best Possible Solution

Add a focused Control UI/WebChat selected-agent New Chat flow using `agents.list` metadata and the existing `sessions.create({ agentId })` contract, then switch to the returned dashboard session key. Keep the broader sidebar/chat-list redesign tracked separately in #49828.

## Reproduction Assessment

Yes. Configure multiple agents, open WebChat on one active session, and click New session; current source shows the call derives `agentId` from the active session key and offers no UI path to pass another configured agent.

## Solution Assessment

Yes. The narrowest maintainable fix is UI-only over the existing `agents.list` and `sessions.create` contract; current source does not indicate a need for a new backend endpoint.
