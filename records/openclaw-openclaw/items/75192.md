## Summary

Keep open. Current main still does not expose a WebChat path to create a new session for a selected configured agent: the visible New Session control still sends `/new` in the active session, while the backend already has `sessions.create` with `agentId`. The adjacent open #73725 should be sequenced or extended before starting another automated repair branch.

## What This Changes

The issue requests a WebChat New Chat flow that lets users choose a configured agent, create a new session for that agent, show agent identity in session UI, and switch to the new session.

## Best Possible Solution

Reuse the existing `agents.list` and `sessions.create` contracts to add a focused selected-agent New Chat flow in WebChat, switch to the returned session key with the existing session-switch helper, refresh chat/session/avatar/tool state, and avoid a new backend endpoint unless implementation exposes a concrete missing field.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple agents, open WebChat, and use the visible New Session button; current main sends `/new` in the active session and has no UI path that passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, with sequencing. The narrowest maintainable solution is a UI layer over the existing agent list and session-create APIs, preferably folded into or applied after the adjacent #73725 create/switch work rather than implemented as a separate backend contract.
