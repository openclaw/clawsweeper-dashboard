## Summary

Keep open: current main still creates New Chat sessions for the active session's agent only, while the requested WebChat/Control UI path to choose a different configured agent is not implemented. The backend and UI controller already accept `agentId`, so this is a bounded feature candidate rather than a duplicate of the broader sidebar UX request in #49828.

## What This Changes

Request to let WebChat/Control UI users start a new chat session with a selected configured agent instead of only the active/default agent.

## Best Possible Solution

Add a focused New Chat agent picker in the Control UI/WebChat surface that reuses configured agent metadata and the existing `sessions.create({ agentId })` contract without adding a new backend endpoint or config option.

## Reproduction Assessment

Yes. Source inspection on current main shows New Chat and `/new` route through a zero-argument create path that derives `agentId` from the active session, and existing tests assert that behavior.

## Solution Assessment

Yes. A narrow UI selector over configured agents plus `createSessionAndRefresh(..., { agentId })` is the maintainable fix; no Gateway protocol change is indicated.
