## Summary

Keep open: current main still has no WebChat UI path to create a new session for a different configured agent, even though the backend/session controller already accepts an agentId.

## What This Changes

Add a WebChat/Control UI New Chat flow that lets users select a configured agent and creates/switches to a session for that agent.

## Best Possible Solution

Ship a focused Control UI/WebChat agent picker on New Chat that uses configured agent metadata and the existing sessions.create({ agentId }) contract.

## Reproduction Assessment

Yes. Source inspection on current main shows the toolbar New session button calls a zero-argument callback, and createChatSession forwards the active session's agentId rather than a user-selected one.

## Solution Assessment

Yes. The narrow maintainable solution is to add a selector to the existing New session flow and pass the selected agentId through createSessionAndRefresh/sessions.create; no new Gateway endpoint is indicated.
