## Summary

Keep open. Current main still has the requested WebChat gap: the Control UI can list/switch existing agent sessions and the Gateway already supports `sessions.create` with `agentId`, but the visible New Session control still sends `/new` in the active session and offers no UI path to create a fresh session for a selected configured agent. The open, maintainer-labeled #73725 is adjacent create/switch work, but it is not merged and does not satisfy selected-agent creation on main.

## What This Changes

The issue requests a WebChat/Control UI flow that lets users create a new chat session for a selected configured agent.

## Best Possible Solution

Land or update the generic WebChat create/switch path first, then add a focused selected-agent picker that reuses `agents.list`, calls `sessions.create` with the chosen `agentId`, switches to the returned key, refreshes session/history/avatar/tool state, and avoids Gateway contract changes unless a concrete missing field appears.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple agents, open Control UI chat, click the plus/New Session control, and current main can only dispatch `/new` in the active session; no visible UI path passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, with sequencing. The narrow maintainable solution is a UI addition over the existing `agents.list` and `sessions.create` contracts, preferably layered on or after #73725 rather than adding a separate backend endpoint.
