## Summary

Keep open: current main still lacks a visible WebChat New session agent selector; the existing New session button creates another session for the active session's agent even though the backend and UI helper already accept an explicit `agentId`.

## What This Changes

Add a WebChat flow that lets users choose a configured agent when creating a new chat session, then switches to the created session.

## Best Possible Solution

Add a focused Control UI/WebChat New session picker that lists configured agents from existing metadata, calls the existing `sessions.create({ agentId })` flow, switches to the returned session, and leaves sidebar navigation redesign to #49828.

## Reproduction Assessment

Yes. Configure multiple agents, open WebChat on one agent session, and use the New session control; source and tests show it creates a dashboard session for the active session's agent with no selector for another configured agent.

## Solution Assessment

Yes. A narrow UI-only selector over the existing agents list and `sessions.create` contract is the maintainable fix; no new backend endpoint appears necessary.
