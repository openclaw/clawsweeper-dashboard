## Summary

Keep open. Current main still has the WebChat/Control UI gap: configured agents can be listed and the Gateway can create agent-scoped sessions, but the visible New Session control does not let the user pick an agent and does not call `sessions.create` from the UI.

## What This Changes

Add a WebChat/Control UI flow that lets users choose a configured agent when creating a new chat session, then switches to that new agent-scoped session and displays agent identity.

## Best Possible Solution

Land or update the generic WebChat create/switch path first, then add a focused selected-agent picker that reuses `agents.list`, calls `sessions.create` with the chosen `agentId`, switches to the returned key, refreshes session/history/avatar/tool state, and avoids Gateway contract changes unless a concrete missing field appears.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: with multiple configured agents, the visible New Session control can only trigger `/new` in the active session; no current UI path passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, with sequencing. The narrow maintainable solution is a Control UI addition over existing `agents.list` and `sessions.create`, coordinated with #73725 so the session-create mechanics are not implemented twice.
