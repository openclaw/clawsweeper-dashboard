## Summary

Keep open. Current main still has the reported WebChat gap: configured agents are available and the Gateway can create agent-scoped sessions, but the visible New Session control still resets the active chat via `/new` and no Control UI path creates a fresh session for a selected agent. The adjacent open PR #73725 is relevant sequencing work, and #49828 overlaps broader multi-agent UX, but neither is merged or a duplicate replacement for this selected-agent creation request.

## What This Changes

Add a WebChat/Control UI flow that lets users choose a configured agent when creating a new chat session, then switches to that new agent-scoped session and displays agent identity.

## Best Possible Solution

Keep this issue open and implement a focused Control UI selected-agent New Chat flow after or alongside the generic create/switch work: reuse agents.list for picker metadata, call sessions.create with the chosen agentId, switch to the returned key, refresh sessions/history/avatar/tool state, and avoid adding a new Gateway endpoint unless a concrete missing field appears.

## Reproduction Assessment

Yes. Configure multiple agents, open the Control UI chat, and click the plus/New Session control; source inspection shows that path still sends `/new` in the active session and no UI code passes a selected `agentId` to sessions.create.

## Solution Assessment

Yes, with sequencing. The narrow maintainable solution is a UI addition over existing agents.list and sessions.create contracts, coordinated with #73725 rather than adding a duplicate backend API or a separate session abstraction.
