## What This Changes

The PR guards malformed capability provider rows in direct runtime lookup and the shared capability provider map builder, with focused regression tests.

## Best Possible Solution

Land after extending row isolation to the custom generation provider maps or unifying them behind a guarded shared capability-provider map, with regression coverage for unreadable aliases on a generation family.

## Reproduction Assessment

Yes from source inspection: current main reads provider ids and aliases without row-local guards, and the PR's tests encode those throwing-getter paths. I did not run tests because this review is read-only.

## Solution Assessment

No: guarding direct lookup and the shared map helper is a good boundary direction, but the best fix also covers sibling generation provider maps that bypass the shared helper.
