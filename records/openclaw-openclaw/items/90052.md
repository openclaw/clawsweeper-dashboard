## Summary

Keep open: current main still has the unguarded provider-row behavior, and the PR branch is a useful but incomplete fix because image, video, and music provider registries can still re-read a throwing `aliases` getter after the capability runtime returns the raw provider object.

## What This Changes

The PR guards capability provider `id`/`aliases` reads in plugin capability resolution and the shared provider map builder, with focused regression tests for those paths.

## Best Possible Solution

Complete provider-row isolation at the shared and generation registry boundary so all capability provider families snapshot id/aliases once, skip malformed rows, and keep healthy providers discoverable with focused regression coverage.

## Reproduction Assessment

Yes: source inspection gives a high-confidence path by returning an image, video, or music provider whose `aliases` getter throws from `resolvePluginCapabilityProviders`, then calling the corresponding list or lookup registry. I did not execute it because this review is read-only.

## Solution Assessment

No: this is a plausible partial fix, but not the best complete fix yet. The better fix is to apply the same guarded id/alias boundary to image, video, and music provider maps or return a safe provider snapshot before those registries consume aliases.
