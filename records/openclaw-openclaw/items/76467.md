## Summary

Keep open. Current main still rejects `queueReplyRunMessage()` unless the reply operation is already `running`, while the same registry treats `preflight_compacting` and `memory_flushing` as compacting phases; the issue is also paired with open PR #76485 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Fix a Gateway/WebChat session hang where messages sent around preflight compaction can be queued or dropped and leave the session unresponsive until restart.

## Best Possible Solution

Review and land PR #76485 or an equivalent narrow regression-backed fix that lets compacting-phase WebChat follow-ups drain or receive a clear busy/error response without adding config or restart behavior.

## Reproduction Assessment

Yes at source level: create a reply operation in `preflight_compacting` or `memory_flushing` with a queue-capable backend and `queueReplyRunMessage()` still returns false on current main. I did not establish a live Gateway hang reproduction in this read-only pass.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: align `queueReplyRunMessage()` with the compacting phases already recognized by the registry and cover it with focused regression tests. A broader WebChat-only workaround, restart path, or new config knob would be less appropriate.
