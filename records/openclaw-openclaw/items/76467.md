## Summary

Keep open. Current main has compaction safety timeouts and several recent session-list/event-loop fixes, but I did not find proof that a WebChat follow-up turn queued around `preflight_compacting` reliably drains after rotated preflight compaction; the reported stuck `processing`/cleanup-timeout logs still map to current active reply and embedded-run cleanup paths.

## What This Changes

Fix a Gateway/WebChat liveness bug where a main session can remain processing and stop answering after `maxActiveTranscriptBytes` preflight compaction rotates the active transcript.

## Best Possible Solution

Add focused regression coverage for WebChat turns queued during or immediately after preflight transcript-byte compaction, then fix the narrow terminalization or follow-up draining path so the session replies or returns a clear error without requiring a gateway restart.

## Reproduction Assessment

No high-confidence runtime reproduction was established in this read-only pass. Source inspection does confirm the reported path: `maxActiveTranscriptBytes` triggers preflight compaction, the reply operation enters `preflight_compacting`, and later WebChat messages can be serialized behind active work rather than accepted into the active embedded run.

The narrow maintainable solution is a regression test around preflight compaction plus queued WebChat follow-up cleanup, followed by a targeted fix to terminalize or drain the active session state rather than adding new config or broad restart behavior.
