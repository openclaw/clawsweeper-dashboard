## Summary

Keep open: current main forwards the existing SSRF policy to image reference media loads, but image-generation provider API calls still do not receive `allowRfc2544BenchmarkRange`, so the reported Google/MiniMax fake-IP path remains plausible from source inspection.

## What This Changes

Honor the existing SSRF RFC2544 fake-IP opt-in for image-generation provider HTTP calls such as Google and MiniMax, not only generic fetch or reference-image loads.

## Best Possible Solution

Thread the existing SSRF policy into the provider HTTP/image-generation request path with host-scoped regression coverage, while keeping default SSRF behavior strict.

## Reproduction Assessment

Yes, from source inspection: current main can pass `allowRfc2544BenchmarkRange` to reference media loads, but the provider API POST path receives only `allowPrivateNetwork`. A live Clash/TUN DNS run was not performed in this read-only review.

## Solution Assessment

No current implementation fully solves it. The narrow maintainable fix is to propagate the existing SSRF policy through `GenerateImageParams`/`ImageGenerationRequest` and the shared provider HTTP helpers, or reuse the existing scoped provider fake-IP policy helper for provider-host requests.
