## Summary

Keep open: current main still has no path for RFC2544 fake-IP SSRF policy to reach image provider API guarded fetches. Existing coverage only proves remote reference-image loading gets `tools.web.fetch.ssrfPolicy`, while Google/MiniMax provider POSTs still pass only `allowPrivateNetwork` and `dispatcherPolicy`.

## What This Changes

The issue reports that image-generation provider HTTP calls ignore the existing RFC2544 fake-IP SSRF opt-in when outbound DNS resolves provider hosts into `198.18.0.0/15`.

## Best Possible Solution

Add shared provider HTTP SSRF-policy plumbing for media/image generation so RFC2544 fake-IP ranges are allowed only for the resolved provider API hostname while preserving existing private-network blocking semantics.

## Reproduction Assessment

Yes, from source inspection: current main can reproduce the missing-policy path by using image generation with a Google or MiniMax provider whose API hostname resolves to `198.18.x.x`. The runtime and provider helpers do not carry an RFC2544 SSRF policy into the guarded provider POST.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the shared provider HTTP/media-generation policy path, with provider-host-scoped fake-IP allowance and regression coverage. Per-provider patches would be more brittle and easier to drift.
