## Summary

## What This Changes

The PR moves `retireSessionMcpRuntime` in gateway session cleanup so old-session MCP runtimes are retired immediately after aborting the embedded run instead of only after the run drains successfully.

## Best Possible Solution

Merge the focused lifecycle change after concrete redacted process/log proof, or maintainer-run proof, shows the old MCP PID is reaped on the timed-out reset path.

## Reproduction Assessment

Yes from source: current main only retires the MCP runtime when `waitForEmbeddedAgentRunEnd` returns true, so the timed-out reset path can skip MCP cleanup. I did not run the WSL2/LadybugDB live scenario.

## Solution Assessment

Yes, the proposed location is the narrow owner-boundary fix: MCP runtimes are session-scoped and existing sibling cleanup already retires them during session rollover/run end. Reuse or per-server restart strategy would be a larger product change, not needed for this bug fix.
