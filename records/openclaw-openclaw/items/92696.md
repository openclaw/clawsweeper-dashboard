## What This Changes

The PR moves gateway session MCP runtime retirement to run immediately after `abortEmbeddedAgentRun`, before the 15-second run-end wait, so stale MCP processes are disposed even when cleanup later returns `UNAVAILABLE`.

## Best Possible Solution

Land this cleanup after the contributor supplies concrete redacted process proof, ideally with a focused timeout-branch regression assertion for MCP retirement.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: on current main, a reset/delete cleanup where `waitForEmbeddedAgentRunEnd` returns `false` skips `retireSessionMcpRuntime`; I did not execute a live MCP repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes for the code shape: moving the existing bounded MCP retirement into the session cleanup owner after abort is narrower than adding per-server restart config or pushing stale-PID cleanup into individual MCP servers, but the proof/test gap should be closed before merge.
