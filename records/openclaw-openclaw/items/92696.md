## Summary

Keep open. The stale MCP process problem is real, but this PR is not merge-ready because it retires the MCP runtime before the embedded run has drained, so a failed reset/delete can return UNAVAILABLE while leaving the old run active with closed MCP transports.

## What This Changes

This PR moves `retireSessionMcpRuntime` in `ensureSessionRuntimeCleanup` so gateway reset/delete retires the session MCP runtime immediately after aborting an embedded run, before waiting up to 15 seconds for it to end.

## Best Possible Solution

Use a reset/delete-owned terminalization or deferred-retirement path that only retires MCP runtime after the old run is no longer active, or after an explicit maintainer-approved force-clear contract, with focused timeout-branch tests and real active-MCP reset proof.

## Reproduction Assessment

Yes at source level: current main skips MCP runtime retirement when `waitForEmbeddedAgentRunEnd` returns false after a reset/delete abort. I did not reproduce the live file-locking MCP scenario with real child processes.

## Solution Assessment

No, not as submitted. The PR targets the right cleanup gap, but moving MCP retirement before the drain wait reintroduces a previously flagged lifecycle ordering problem; a safer fix needs run terminalization, force-clear, or deferred retirement semantics plus tests.
