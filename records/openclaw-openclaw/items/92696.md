## Summary

Keep open; the PR targets a real cleanup gap, but the submitted ordering is not merge-ready because it retires the MCP runtime before proving the old embedded run has drained, while current source shows the timeout path can leave that run active.

## What This Changes

The PR moves gateway session MCP runtime retirement ahead of the embedded-agent run drain wait in `src/gateway/session-reset-service.ts`.

## Best Possible Solution

Use a lifecycle-safe cleanup contract that reaps stale MCP processes only after the run drains, is explicitly force-cleared, or has deferred retirement tied to lease release, with focused reset and delete timeout coverage.

## Reproduction Assessment

Yes at source level: current main skips MCP runtime retirement when the embedded run wait returns false after reset/delete abort. I did not reproduce the live file-locking MCP scenario with real child processes.

## Solution Assessment

No, not as submitted. The PR targets a real cleanup gap, but moving runtime retirement before the run-drain result conflicts with the still-active timeout contract and the MCP lease path; a safer fix needs force-clear or deferred-retirement semantics.
