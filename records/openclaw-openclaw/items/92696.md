- `merge-risk: 🚨 availability`: The patch can dispose MCP transports during a reset attempt that still returns `UNAVAILABLE`, which could interrupt tool availability for an aborting but not-yet-ended run.

## Summary

Keep this PR open: it is a focused, non-redundant gateway reset cleanup for a real current-main timeout path, but it still needs stronger contributor proof and maintainer acceptance of the abort-time MCP teardown side effect before merge.

## What This Changes

The PR moves `retireSessionMcpRuntime` earlier in gateway session cleanup so MCP runtimes are disposed after aborting the old embedded run instead of only after the run ends within the grace window.

## Best Possible Solution

Land this gateway reset cleanup only after concrete redacted process/log proof and maintainer agreement that abort-time MCP teardown is intended, while keeping the manager-level rebind fix as a separate related path.

## Reproduction Assessment

Yes. Source inspection shows current main skips `retireSessionMcpRuntime` when `waitForEmbeddedAgentRunEnd` returns false after a reset abort, which matches the stale-process timeout branch this PR changes; I did not run a live file-locking MCP scenario.

## Solution Assessment

Partly. This is a narrow fix for the gateway reset timeout path, but it is not a replacement for the related sessionKey rebind fix and should not merge until maintainers accept the failed-reset side effect and the contributor supplies stronger proof.
