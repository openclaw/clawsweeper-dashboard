## Summary

Keep open. The PR targets a real dashboard/history attribution regression, but the current patch is still partial: it recovers labels only in the raw JSONL fallback while current v3 sessions use the active-branch reader first, and the user-visible fix still lacks a changelog entry.

## What This Changes

The PR adds gateway session-history helpers and regression tests to derive `senderLabel` from hidden `openclaw.runtime-context` custom-message rows while keeping those rows out of `readSessionMessages()` output.

## Best Possible Solution

Apply the runtime-context detection and previous-user sender-label attachment consistently in the SessionManager branch path and the raw JSONL path, keep hidden runtime-context rows out of all visible history projections, add focused v3 active-branch regression coverage, and add the required single-line changelog credit.

## Reproduction Assessment

Yes. A v3 transcript with `id`/`parentId` entries and a hidden `openclaw.runtime-context` custom-message row after a user message will take the `getBranch()` path and return before the PR's raw-loop recovery can attach `senderLabel`.

## Solution Assessment

No. Raw JSONL recovery is useful but not sufficient; the narrow maintainable fix is to share the filtering and label-attachment behavior with the active-branch reader used by current sessions.
