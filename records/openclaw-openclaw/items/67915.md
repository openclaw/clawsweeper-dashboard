## Summary

Keep open. Current main still contains the reported empty-roots client-side hard deny before the server metadata check, and the issue is paired with open, unmerged PR #67916 as the implementation candidate, so closing before that lands or is replaced would be premature.

## What This Changes

The issue requests fixing Control UI local assistant media attachments so an empty bootstrap `localMediaPreviewRoots` state does not permanently render valid server-allowed media as outside allowed folders.

## Best Possible Solution

Keep this issue open until PR #67916 or an equivalent focused replacement lands. The desired end state is to treat empty preview roots as a bootstrap-not-loaded state, fall through to the authenticated server metadata endpoint, preserve populated-roots client blocking for true outside-root paths, and add a regression test for the empty-roots local attachment case.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: render a local assistant `MEDIA:` attachment while `localMediaPreviewRoots` is `[]`; current main returns `Outside allowed folders` before the server metadata fetch can run.

## Solution Assessment

Yes. The narrow maintainable solution is to skip only the client hard-deny while preview roots are empty and keep the server `assistant-media?meta=1` route as the authoritative access check; populated roots should continue to block outside paths client-side.
