- `P2`: This is a normal-priority Control UI media rendering bug that makes available assistant attachments appear inaccessible without crashing the runtime.
- `impact:message-loss`: Assistant media can be suppressed or shown unavailable in WebChat even though the server reports it available.

## Summary

Keep open: current main and the latest release still hard-deny local Control UI assistant attachments when `localMediaPreviewRoots` is empty, and an open linked PR already owns the focused fix path.

## What This Changes

The issue reports Control UI local assistant media attachments being blocked as outside allowed folders during the bootstrap window even though the server media endpoint allows them.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/78034`, or an equivalent focused replacement, so empty bootstrap roots fall through to the authenticated server metadata check while populated-root blocking remains intact.

## Reproduction Assessment

Yes, at source level: current main initializes `localMediaPreviewRoots` as `[]`, then `resolveAssistantAttachmentAvailability` returns `Outside allowed folders` before the `assistant-media?meta=1` server check can run. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes for the linked fix shape: treating empty roots as bootstrap-not-loaded and falling through to the existing authenticated server metadata check is narrower than widening allowed roots or adding config. The issue should stay open until that PR or an equivalent replacement lands.
