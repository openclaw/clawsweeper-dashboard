## Summary

Keep open. Current main still has the reported empty-roots hard-deny path, and the issue is explicitly paired with open PR #67916 as the active implementation candidate, so closing before that PR lands or is replaced would be premature.

## What This Changes

The issue requests that Control UI local assistant media attachments use the server `assistant-media?meta=1` availability check when bootstrap `localMediaPreviewRoots` have not loaded, instead of rendering them as outside allowed folders.

## Best Possible Solution

Keep this issue open until PR #67916, or an equivalent focused replacement, lands. The fix should treat empty `localMediaPreviewRoots` as a not-yet-loaded bootstrap state, fall through to the authenticated server metadata endpoint, preserve populated-roots client-side blocking for genuinely outside paths, and add a regression test for the empty-roots case.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: render a local assistant `MEDIA:` attachment while `localMediaPreviewRoots` is `[]`; the client predicate returns false and `resolveAssistantAttachmentAvailability` returns `Outside allowed folders` before the metadata fetch can run.

## Solution Assessment

Yes. The narrow maintainable direction is to skip only the client allowed-root hard deny while roots are empty and continue relying on the server metadata endpoint for the authoritative local-root check; the populated-roots denial should remain unchanged.
