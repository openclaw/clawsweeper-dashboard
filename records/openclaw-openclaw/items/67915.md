## Summary

Keep open. Current main still contains the empty-bootstrap hard-deny path, and live GitHub shows https://github.com/openclaw/openclaw/pull/78034 is an open closing PR for the same bug, so this issue should stay open until that PR or an equivalent replacement lands.

## What This Changes

The issue reports Control UI local assistant media attachments being shown as unavailable when bootstrap preview roots have not loaded, even though the server media endpoint authorizes the file.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78034 or an equivalent focused patch that skips the client hard-deny only while roots are empty, preserves populated-root blocking, covers local inline media/image rendering, and includes regression plus accepted runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the app starts with `localMediaPreviewRoots` as `[]`, the predicate returns false for empty roots, and `resolveAssistantAttachmentAvailability` returns `Outside allowed folders` before the server metadata check can run.

## Solution Assessment

Yes for the proposed direction. Falling through to the existing authenticated server metadata endpoint only during the empty-bootstrap window is narrower than weakening the populated-roots client gate globally.
