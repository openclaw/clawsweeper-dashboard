## Summary

Keep open: current main still has the Discord/message_tool_only path where an accepted channel mention can produce no committed message-tool delivery and any unmarked final payload is suppressed; the matching fix is the still-open https://github.com/openclaw/openclaw/pull/84485, not merged main behavior.

## What This Changes

A Discord channel mention can complete an agent turn as non_deliverable_terminal_turn without assistant text, message.send delivery, or any visible Discord reply.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84485, or replace it with a narrow Discord source-channel guard that emits a safe diagnostic without leaking private final text.

## Reproduction Assessment

Yes at source level: current main still combines Discord message_tool_only source suppression, non_deliverable_terminal_turn classification, and final-payload suppression unless a bypass marker is present. I did not run live Discord in this read-only review.

## Solution Assessment

Mostly yes: the linked guard PR targets the runReplyAgent/dispatch boundary where source-delivery evidence and suppression context are available. The part needing maintainer judgment is the exact visible fallback policy for previously silent Discord turns.
