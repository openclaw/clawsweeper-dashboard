## Summary

Keep open: current main and v2026.6.1 do clamp a requested `full` policy to `allowlist` when the host approvals file is stricter, but the source comments, tests, and docs define that as the host security floor. Because this issue is security-labelled and has an open linked fix PR, it needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue requests changing exec approval security resolution so a session `security: "full"` override is not clamped by an agent or host `allowlist` approval policy, and notes Discord native approvals lack per-channel security/ask config.

## Best Possible Solution

Maintainers should decide the security policy first: either keep and clarify the current host-floor semantics, or design an explicit opt-in override surface with docs and regression coverage for gateway, node, and channel approval paths.

## Reproduction Assessment

Yes, source-reproducible: current `minSecurity` and `resolveExecHostApprovalContext` resolve requested `full` plus host `allowlist` to `allowlist`. I did not run a live command because this review is read-only, but the source, tests, and docs all expose the behavior.

## Solution Assessment

No, the proposed one-line inversion is not clearly the best fix because it conflicts with the documented host approval floor. The safer path is a maintainer decision on whether an explicit opt-in override should exist, followed by focused tests and docs.
