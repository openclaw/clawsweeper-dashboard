## Summary

Keep open for maintainer judgment: the reported clamp is real, but current source, tests, docs, and the latest release define host approvals as the stricter floor, so flipping `minSecurity` would be a security policy change rather than a straightforward bug fix.

## What This Changes

Requests changing exec security precedence so a session or config `security="full"` can override an agent or host `allowlist` policy, and notes missing per-channel exec `security`/`ask` config.

## Best Possible Solution

Keep host approvals as the enforceable floor; if maintainers want per-session or per-channel full/off workflows, design an explicit high-power policy surface with docs, migration expectations, and focused tests instead of inverting `minSecurity`.

## Reproduction Assessment

Yes for the observed clamp: current source and an existing policy test show requested `security="full"` with host `security="allowlist"` resolving to `allowlist`. I did not run tests because this review is read-only.

## Solution Assessment

No for the proposed fix: swapping `minSecurity` would contradict current docs and tests by allowing session/config policy to broaden host approvals. A safer path would be an explicit maintainer-approved policy surface if this override is desired.
