## Summary

Keep open: the PR targets a real agents/process-exec bug, but the current patch misses the local background-exec completion format and can now promise an automatic wake for cases where current runtime intentionally does not enqueue one.

## What This Changes

The branch changes yielded exec result wording and adds heartbeat prompt detection/tests for retained background exec sessions.

## Best Possible Solution

Keep the PR direction, but make the background-completion detector cover the actual local producer formats and keep yielded-result guidance conditional when automatic notification is not guaranteed, then add real background exec heartbeat proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: local background exec completion events are emitted as `Exec completed/failed (...)`, then current heartbeat prompt selection falls through to the existing HEARTBEAT_OK or relay prompt.

## Solution Assessment

No, this is a plausible fix shape but not the best current implementation because it keys off an `id=` token that does not cover the main local producer and it removes fallback guidance for no-notification cases.

Full review comments:

- **[P1] Handle the local completion event format:** `src/infra/heartbeat-events-filter.ts:211-213`
  - body: The new background detector only looks for an `id=` token and then checks that value in `finishedSessions`, but the local background exec notifier emits `Exec completed/failed (<session id prefix>, ...) :: output` from `maybeNotifyOnExit`. That means the ordinary `exec background=true` or auto-yield completion path still falls through to the old HEARTBEAT_OK/relay prompt this PR is meant to replace; cover the structured local format or carry a real session id through the event, and add that producer to the new test.
  - confidence: 0.93
- **[P2] Do not promise a wake when none may be sent:** `src/agents/bash-tools.exec.ts:1915-1917`
  - body: This text is used for every yielded/backgrounded exec, but current runtime only enqueues completion events when `notifyOnExit` is enabled, a session key exists, and the completion is not a quiet successful exit unless `notifyOnExitEmptySuccess` is set. With the new wording, an agent can be told to continue waiting for an automatic notification that will never arrive; keep process poll/log confirmation guidance or make the promise conditional on the actual notify settings.
  - confidence: 0.88
