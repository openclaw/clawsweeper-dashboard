## Summary

Keep open: the PR targets a real current-main/docs mismatch, but the proposed auth fallback skips the shared-secret limiter precheck and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a trusted-proxy local-direct password fallback in `src/gateway/auth.ts` when `allowLoopback` is true, plus two gateway auth regression tests.

## Best Possible Solution

Keep the documented local-direct fallback only if gateway/security owners accept that policy, and implement it with the shared-secret limiter precheck, the no-forwarded-header boundary, regression coverage, and aligned docs.

## Reproduction Assessment

Yes, source-level. Current main returns the trusted-proxy rejection before password auth can run, while docs and the linked issue describe a loopback password fallback path for this configuration.

## Solution Assessment

No. The PR changes the right branch, but it is not mergeable until the fallback checks the existing shared-secret limiter before accepting a matching password and the auth-boundary policy gets owner review.
