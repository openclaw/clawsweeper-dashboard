## Summary

Keep open: the PR targets a real current-main trusted-proxy/password mismatch, but it is not merge-ready because the new fallback bypasses the shared-secret lockout check and the external PR only provides unit-test proof, not real gateway behavior proof.

## What This Changes

The PR adds a trusted-proxy local-direct password fallback in `src/gateway/auth.ts` and two gateway auth regression tests for matching and wrong passwords.

## Best Possible Solution

Align the trusted-proxy contract by restoring the documented local password fallback with the normal rate-limit pre-check and live proof, or updating docs and credential behavior if auth owners choose fail-closed mode.

## Reproduction Assessment

Yes, by source inspection: current main returns the trusted-proxy failure before password auth while docs and credential tests still describe or supply the local password fallback. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR targets the right code path, but it omits the rate-limit lockout check that the previous fallback implementation had and needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Check lockout before fallback password auth:** `src/gateway/auth.ts:494-500`
  - body: This new trusted-proxy fallback calls `authorizePasswordAuth` directly, but the shared `limiter.check` gate is below the trusted-proxy return and is never reached. Wrong passwords can record failures, yet a later matching password will still be accepted while that key is locked out. Mirror the normal password-mode pre-check before calling `authorizePasswordAuth` and cover the lockout case.
  - confidence: 0.93

## Security Review

Status: needs_attention
