## Summary

Keep open: the PR targets a real documented gateway auth mismatch, but the patch has a blocking auth correctness issue and still lacks after-fix real behavior proof. The new fallback bypasses the shared-secret rate-limit precheck before accepting a matching password.

## What This Changes

The branch adds a trusted-proxy local-direct password fallback in `src/gateway/auth.ts` and two regression tests in `src/gateway/auth.test.ts`.

## Best Possible Solution

Keep the documented fallback, but enforce the shared-secret limiter before accepting it, preserve the no-forwarded-header boundary, add a lockout regression test, and require auth/secops review plus redacted real runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns the trusted-proxy failure before password auth can run, and the linked issue provides a concrete loopback/password configuration with `authProvided=password` and a missing-header rejection.

## Solution Assessment

No. The PR changes the right auth branch, but it is not the best mergeable shape until it runs the existing shared-secret limiter check before accepting the fallback password.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Check the limiter before accepting fallback passwords:** `src/gateway/auth.ts:494`
  - body: The new trusted-proxy fallback calls `authorizePasswordAuth` before the shared-secret limiter check below this branch can run, so a local-direct client that is already locked out can still authenticate by presenting the right password. Move the existing `limiter.check`/`rate_limited` handling in front of this fallback and add a regression for a denied limiter.
  - confidence: 0.9
