## Summary

Keep open: the PR targets a real docs/runtime regression in trusted-proxy mode, but the current patch still changes a security-owned auth boundary and has a narrow correctness issue around when password fallback may override trusted-proxy failures.

## What This Changes

The PR adds a local-direct password fallback in trusted-proxy auth when `allowLoopback` is true and updates gateway auth tests for success, limiter lockout, and wrong-password behavior.

## Best Possible Solution

Gateway/security owners should decide whether to accept a short-term password fallback; if accepted, it should be limited to true non-proxy local-direct misses with limiter coverage, while the longer-term service-identity RFC remains the cleaner contract.

## Reproduction Assessment

Yes, source-level. Current main returns the trusted-proxy rejection before password auth can run, while current docs describe a local-direct password fallback for internal trusted-proxy callers.

## Solution Assessment

No. The PR restores the documented path and fixes the earlier limiter gap, but it should not let password fallback override arbitrary trusted-proxy failures such as denied proxy identity, and the auth-boundary policy still needs owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Constrain fallback to non-proxy failures:** `src/gateway/auth.ts:488-493`
  - body: The fallback runs after any trusted-proxy rejection as long as the socket looks local-direct. A same-host proxy request that carries identity headers but fails `allowUsers` can still succeed by presenting the gateway password when it lacks standard `X-Forwarded-*` headers, so the trusted-proxy allowlist no longer owns that request. Gate the fallback to true non-proxy/missing-identity cases or disqualify requests carrying configured proxy identity/verification headers.
  - confidence: 0.86

## Security Review

Status: needs_attention
