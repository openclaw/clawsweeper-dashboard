## Summary

Keep open with a blocking correctness finding: the helper re-resolves the stale current model identity, so cross-provider /model switches can still use the old context window; real behavior proof is also insufficient.

## What This Changes

The PR adds a `refreshCurrentModelFromRegistry()` call before `checkCompaction` reads `contextWindow` in `AgentSession`.

## Best Possible Solution

Synchronize the running session from the canonical selected provider/model identity at the shared model-switch boundary, or make post-turn consumers resolve from that selected identity, then prove it with cross-provider regression coverage and real Telegram or CLI output.

## Reproduction Assessment

Yes, source-level. The Telegram picker persists a different provider/model while a running `AgentSession` can keep the old `agent.state.model`, and `checkCompaction` then reads that stale snapshot for `contextWindow`.

## Solution Assessment

No. This patch is a plausible local mitigation, but it looks up the registry from the stale current identity; the maintainable fix should start from the newly selected provider/model or update the live session at the shared switch boundary.
