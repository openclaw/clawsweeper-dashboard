## Summary

The PR should stay open but not merge as written: it targets a real stale-session bug, yet the added refresh searches the registry with the stale model’s provider and id, so it cannot adopt the cross-provider model selected in the reported Telegram flow. The paired issue remains open, and the PR also lacks a focused regression test and observed after-fix runtime proof.

## What This Changes

Adds a call to refresh the session model from the registry before `checkCompaction` reads its `contextWindow`.

## Best Possible Solution

Propagate the newly selected provider/model into the running session at the shared persisted-selection-to-live-session boundary, preserve existing pending-switch semantics, and cover both compaction threshold and context-usage behavior with a cross-provider regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Telegram persists a different provider/model while the running session can retain its old model object, and compaction reads that live snapshot.

## Solution Assessment

No. The proposed helper resolves only the stale model identity, so the narrow maintainable fix is to synchronize the running session from the newly selected identity at the shared model-switch boundary.
