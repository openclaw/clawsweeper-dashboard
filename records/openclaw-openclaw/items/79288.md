## Summary

This PR targets a valid stale-running maintenance bug tracked in https://github.com/openclaw/openclaw/issues/79226, but it should stay open because the new stale-age liveness check bypasses documented legacy CLI child-session backing and the external PR has no after-fix real behavior proof.

## What This Changes

This PR adds a 24-hour stale-running reconciliation rule to task maintenance, updates background task docs, and adds regression tests for stale child-session task rows.

## Best Possible Solution

Keep the stale-running reconciliation in task maintenance, but make fresh-evidence checks reuse runtime-specific backing semantics for legacy CLI tasks and require real `tasks maintenance --apply` proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main can leave stale running rows active when persisted child/session backing is still present, and the PR regression is also source-reproducible with a legacy CLI task older than 24 hours, no run/source id, and a freshly updated non-chat child session.

## Solution Assessment

No, not as-is. Task maintenance is the right surface, but the implementation should preserve legacy CLI child-session freshness before the 24-hour rule can be considered the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve legacy CLI child-session freshness:** `src/tasks/task-registry.maintenance.ts:342-343`
  - body: The new stale-age path treats every CLI task without an active run context as lacking fresh active-work evidence. That bypasses the documented/current fallback for legacy CLI tasks without run/source identity, so an old legacy CLI task with a recently updated non-chat child session would be marked `lost` after 24 hours even though current maintenance keeps that child-session-backed work live. Please reuse the child-session freshness check when `hasCliRunIdentity(task)` is false.
  - confidence: 0.88
