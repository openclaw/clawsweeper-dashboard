## Summary

Keep open: current main and v2026.6.6 include Codex-specific redacted stderr-tail diagnostics, but the broader requested ACPX subprocess stderr persistence across post-init failures and harnesses is still missing and needs product/security ownership.

## What This Changes

The issue requests durable per-subprocess ACPX stderr logs for post-mortem debugging after successful initialization, with possible retention and log-location configuration.

## Best Possible Solution

Adopt an upstream ACPX-owned diagnostic artifact that is bounded, redacted, retention-aware, documented, and then consume the released ACPX change from OpenClaw.

## Reproduction Assessment

Yes, source-level: current OpenClaw only has Codex-wrapper stderr-tail enrichment, and acpx@0.10.0 still keeps only bounded startup stderr plus verbose mirroring for the general child process path. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear as proposed: durable stderr diagnostics are useful, but the direct unbounded per-process log directory shape is not the best maintainable solution without upstream ACPX ownership, redaction, retention, storage, and default-policy decisions.
