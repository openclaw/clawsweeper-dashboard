## Label Justifications

- `P2`: This is a normal-priority ACPX diagnostics feature with real operator value but no proven emergency outage in current main.
- `impact:security`: The requested durable stderr logs can contain credentials, prompts, local paths, provider diagnostics, and MCP environment details.
- `impact:crash-loop`: The motivating failures include post-init child crashes, OOMs, segfaults, and silent ACPX stalls that are hard to diagnose today.

## Summary

Keep open. Current main has only the shipped Codex-wrapper stderr diagnostic path, while the broader ACPX stderr/run-artifact contract for Claude, Gemini, custom agents, and post-init post-mortem debugging is still unresolved and security-sensitive.

## What This Changes

The issue requests durable ACPX subprocess stderr logging, optionally with configurable log location and retention, for post-mortem debugging after successful startup.

## Best Possible Solution

Define durable ACPX stderr/run artifacts at the supported `acpx/runtime` boundary, then expose reviewed OpenClaw config/docs for redaction, file permissions, retention, location, and real ACPX smoke proof.

## Reproduction Assessment

Yes. Source inspection on current main shows there is no general ACPX stderr artifact config or runtime option, while the existing implementation only captures redacted Codex wrapper stderr for generic Codex ACP failures.

## Solution Assessment

No. The raw always-on tee proposed in the issue is useful as a production workaround, but the maintainable upstream shape needs a reviewed artifact contract with redaction, permissions, retention, and ownership at the current `acpx/runtime` boundary.
