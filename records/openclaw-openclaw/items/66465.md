## Summary

Keep open: current main and the latest release only partially address the diagnostic gap with Codex-specific redacted stderr-tail enrichment, while the broader ACPX child-process stderr persistence requested for post-init failures across harnesses is still missing and needs maintainer product/security direction.

## What This Changes

Request to persist ACPX child-process stderr after initialization to a per-subprocess diagnostic log, with possible retention and log-root configuration, for post-mortem debugging.

## Best Possible Solution

Adopt a plugin/upstream-owned ACPX diagnostic artifact that is bounded, redacted, retention-aware, and documented, then consume the released acpx change from OpenClaw rather than adding an ad hoc unbounded OpenClaw-only log path.

## Reproduction Assessment

Yes, as a source-level capability gap rather than a live bug repro. acpx 0.10.0 still only keeps bounded startup stderr and non-verbose post-init stderr has no durable log, while OpenClaw's current additions are Codex-wrapper-specific.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the best fix should be upstream/plugin-owned with redaction, retention, storage-path, and default on/off decisions rather than directly adding an unbounded per-process log directory.
