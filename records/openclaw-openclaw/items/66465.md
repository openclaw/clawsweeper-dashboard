## Summary

Keep open: current main and the latest release only provide a narrower Codex-wrapper stderr tail for generic Codex ACP internal errors, while the requested durable, general ACPX subprocess stderr persistence still needs maintainer product and security design.

## What This Changes

Persist ACPX child-process stderr beyond startup into durable per-subprocess logs for post-mortem debugging.

## Best Possible Solution

Design a plugin-owned ACPX diagnostic logging contract with redaction, retention, operator controls, and focused tests, then consume it through the OpenClaw ACPX plugin instead of adding an ad hoc always-on core log path.

## Reproduction Assessment

Yes for source-level behavior: upstream acpx 0.10.0 only retains startup stderr for startup failures and mirrors later chunks only in verbose mode. I did not run a live failing child process because the current source and dependency code are enough to confirm the requested durable capture is missing.

## Solution Assessment

Unclear: persistent stderr capture is useful, but the proposed always-on raw log path is not the best shape until maintainers decide redaction, retention, opt-in/defaults, and whether the implementation belongs in openclaw/acpx first.
