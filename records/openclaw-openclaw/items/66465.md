## Summary

Keep open: current main only partially addresses the diagnostic gap with Codex-specific redacted stderr-tail enrichment, while the broader ACPX child-process stderr persistence requested for post-init failures across harnesses is still missing and needs maintainer product/security direction.

## What This Changes

The issue requests persistent ACPX child-process stderr logs for post-initialize failures and post-mortem debugging.

## Best Possible Solution

Adopt a plugin/upstream-owned ACPX diagnostic artifact that is bounded, redacted, retention-aware, and documented, then consume the released acpx change from OpenClaw rather than adding an ad hoc unbounded log path.

## Reproduction Assessment

Yes, as a source-level capability gap rather than a live bug repro. Current OpenClaw only enriches Codex-specific generic failures from wrapper logs, while acpx 0.10.0 still drops non-verbose post-init child stderr.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the best fix should be upstream/plugin-owned with redaction, retention, storage-path, and default on/off decisions rather than directly adding an unbounded per-process log directory.
