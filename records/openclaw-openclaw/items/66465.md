## Summary

Keep open: current main and the latest release still do not provide generic durable ACPX subprocess stderr persistence; current source only has Codex-wrapper-specific redacted stderr-tail diagnostics, while the requested ACPX-wide artifact needs maintainer product and security decisions.

## What This Changes

The issue requests durable per-subprocess ACPX stderr logs for post-initialize failures, with optional retention and log directory controls.

## Best Possible Solution

Adopt an ACPX-owned durable diagnostic artifact that is bounded, redacted, retention-aware, and documented, then consume the released ACPX change from OpenClaw.

## Reproduction Assessment

Yes. Source inspection shows upstream ACPX v0.11.2 still keeps only startup stderr plus verbose mirroring, while OpenClaw only adds a Codex-wrapper-specific stderr tail.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the maintainable solution should be ACPX-owned with redaction, retention, storage path, and default on/off decisions.
