## Summary

Keep open: current main and v2026.6.6 only have Codex-specific redacted stderr-tail diagnostics, while the requested durable ACPX subprocess stderr persistence for post-init failures across harnesses is still missing and needs maintainer product/security direction.

## What This Changes

The issue requests durable on-disk logging of ACPX child-process stderr after startup so operators can debug post-init failures.

## Best Possible Solution

Adopt an ACPX-owned diagnostic artifact that is bounded, redacted, retention-aware, and documented, then consume the released acpx change from OpenClaw rather than adding an unbounded OpenClaw-only log path.

## Reproduction Assessment

Yes, as a source-level capability gap rather than a live bug repro. The current acpx runtime still keeps only bounded startup stderr and no durable non-verbose post-init stderr log, while OpenClaw's current additions are Codex-wrapper-specific.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the best fix should be ACPX/plugin-owned with redaction, retention, storage-path, and default on/off decisions rather than directly adding an unbounded per-process log directory.
