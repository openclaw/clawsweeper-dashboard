## Summary

Keep open. Current main still only retains ACPX startup stderr in memory and mirrors later stderr to the gateway stream when verbose logging is enabled; it does not provide the requested durable post-initialize stderr artifact, and the product/security tradeoff around storing child-process diagnostics remains unresolved.

## What This Changes

The issue requests durable ACPX child-process stderr logs for post-mortem debugging after a subprocess successfully initializes.

## Best Possible Solution

Define an ACPX-owned diagnostic artifact contract with explicit opt-in/default behavior, retention, secure location, and documentation, then implement stderr persistence in that contract with focused runtime tests.

## Reproduction Assessment

Yes, from source inspection: current main keeps only a bounded startup stderr buffer and drops or only mirrors post-init stderr depending on verbose mode. I did not run a live ACPX child because the missing behavior is directly visible in the runtime code path.

## Solution Assessment

Unclear: durable stderr artifacts are a plausible fix, but the proposed always-on file tee plus optional config needs maintainer decisions for sensitive log retention and runtime artifact ownership before it is the best OpenClaw implementation.
