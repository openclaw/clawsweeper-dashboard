## Summary

Keep open. Current main still does not persist ACPX child-process stderr after initialization: OpenClaw delegates child stdio to the pinned `acpx@0.6.1` runtime, exposes no stderr artifact/config contract, and the overlapping artifact PR remains open and unmerged.

## What This Changes

Request persistent per-subprocess ACPX stderr logging, with optional log directory and retention controls, so post-initialization child failures can be debugged after the fact.

## Best Possible Solution

Add bounded, permissioned stderr artifact support in the upstream `acpx` runtime, then wire typed options through the bundled ACPX plugin config, manifest/UI hints, docs, and focused tests. Review or split #53997 as the likely implementation path if maintainers agree on the artifact contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: run an ACP child that writes to stderr after `initialize` with verbose mode off; current `acpx@0.6.1` only updates the bounded startup buffer and optional verbose mirroring, while OpenClaw exposes no persistent stderr option.

## Solution Assessment

No, not as the raw default tee sketched in the issue body. The maintainable solution should keep child lifecycle behavior in the `acpx` runtime boundary and add bounded storage, retention, redaction/privacy handling, and OpenClaw config/docs/tests.
