## Summary

Keep open. Current main still does not persist ACPX child stderr after initialization: the wrapper delegates child stdio to `acpx@0.7.0`, exposes no stderr artifact/config contract, and the overlapping artifact PR remains open with security review findings.

## What This Changes

The issue requests persisting ACPX child-process stderr to disk after initialization for post-mortem debugging, with optional log directory and retention controls.

## Best Possible Solution

Define bounded stderr/run-artifact support in the published `acpx` runtime, then thread sanitized metadata, retention/privacy policy, OpenClaw plugin config/docs, and focused ACPX tests through the wrapper.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: run an ACP child that writes to stderr after `initialize` with verbose mode off, and current `acpx@0.7.0` buffers only startup stderr plus optional verbose mirroring, with no file artifact path.

## Solution Assessment

No. The always-on raw tee sketched in the issue body is not the safest maintainable shape; a bounded upstream `acpx` artifact contract with redaction, permissions, and retention is the better direction.
