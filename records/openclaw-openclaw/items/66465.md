## Summary

Keep open: current main and v2026.6.8 only provide Codex-specific redacted stderr-tail diagnostics, while the requested durable ACPX subprocess stderr persistence across post-init failures and harnesses is still missing and needs product/security direction.

## What This Changes

The issue requests persistent per-subprocess ACPX stderr logs for post-mortem debugging, with possible retention and log-directory configuration.

## Best Possible Solution

Adopt an ACPX-owned diagnostic artifact that is bounded, redacted, retention-aware, and documented, then consume the released acpx change from OpenClaw rather than adding an unbounded OpenClaw-only log path.

## Reproduction Assessment

Yes, as a source-level capability gap rather than a live failure repro. Current acpx still keeps only bounded startup stderr plus verbose mirroring, and OpenClaw's current persistence is Codex-wrapper-specific.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the maintainable path should be ACPX/plugin-owned with redaction, retention, storage-path, and default on/off decisions before implementation.
