## Summary

Keep open. Current main only partially addresses this through merged Codex wrapper stderr diagnostics; it still lacks a general ACPX subprocess stderr/run-artifact contract for Claude, Gemini, custom agents, and post-init post-mortem logs, while the overlapping terminal-artifacts PR remains open with security/product proof blockers.

## What This Changes

Request persistent per-subprocess ACPX stderr logs, with optional retention/root config, for post-mortem debugging after successful initialization.

## Best Possible Solution

Define durable ACPX run/stderr artifacts at the published `acpx` runtime boundary, then expose reviewed OpenClaw config/docs for redaction, permissions, retention, and real smoke proof.

## Reproduction Assessment

Yes. Source inspection against current main and the inspected `acpx@0.7.0` package is enough to show the broad stderr artifact feature is still missing, despite the narrower Codex wrapper diagnostics now on main.

## Solution Assessment

No. The user-visible need is valid, but an always-on raw tee in the OpenClaw wrapper is not the safest maintainable shape; the solution should be an upstream artifact contract or a reviewed successor to the terminal-artifacts PR with redaction, retention, permissions, and proof.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because durable ACPX stderr can contain credentials, prompts, local paths, provider diagnostics, and MCP environment details.

Concerns:

- **[medium] Design redaction and retention before persisting stderr:**
  - body: Persisting raw child-process stderr by default can copy secrets, prompts, provider output, proxy/APM diagnostics, and local paths into long-lived files, so the feature needs explicit redaction, file permissions, location, and retention behavior.
