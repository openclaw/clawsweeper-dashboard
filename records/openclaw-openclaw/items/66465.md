## Summary

Keep open. Current main still does not persist ACPX child stderr after initialization: OpenClaw delegates child-process handling to acpx@0.7.0, exposes no stderr artifact/config surface, and the related artifact PR still needs security/product review.

## What This Changes

Request durable per-subprocess ACPX stderr logs, retention, and optional configuration so post-initialize child failures can be debugged after exit.

## Best Possible Solution

Define bounded, sanitized stderr/run-artifact support in openclaw/acpx, then expose a narrow OpenClaw plugin config/docs surface with retention, permissions, and privacy policy.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: acpx v0.7.0 buffers only startup stderr and mirrors chunks to process.stderr when verbose is enabled, while current OpenClaw passes no stderr artifact option.

## Solution Assessment

No. The always-on raw tee sketched in the issue body is not the safest maintainable shape; the better path is a bounded upstream acpx artifact contract with redaction, permissions, and retention policy.
