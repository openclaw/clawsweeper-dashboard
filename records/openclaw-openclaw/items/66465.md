## Summary

Keep open. Current main still does not persist ACPX child-process stderr after initialization: the OpenClaw plugin delegates process lifecycle to the pinned `acpx@0.6.1` runtime, exposes no stderr artifact/config contract, and the overlapping artifact PR #53997 remains open and unmerged.

## What This Changes

Request to persist ACPX child-process stderr after initialization to durable per-subprocess disk logs, with possible retention and log-directory configuration for post-mortem debugging.

## Best Possible Solution

Define bounded, permission-aware stderr/run-artifact support in the published `acpx` runtime, then consume it through typed OpenClaw plugin config, manifest/UI hints, docs, changelog, and focused ACPX tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: run an ACP child that writes to stderr after `initialize` with verbose mode off, and current `acpx@0.6.1` only keeps startup stderr plus optional verbose mirroring. I did not execute a live ACP child because this was a read-only review and dependencies are not installed locally.

## Solution Assessment

No, the raw default tee sketched in the issue body is not the best maintainable shape. The safer path is a bounded artifact contract in `acpx` with explicit OpenClaw config/docs/tests and privacy/retention handling.
