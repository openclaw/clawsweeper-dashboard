## Summary

Keep open. Current main still does not persist ACPX child-process stderr after initialization: the bundled plugin delegates lifecycle ownership to the pinned `acpx@0.6.1` runtime, the OpenClaw config/manifest expose no stderr log contract, and the overlapping artifact PR is still open and unmerged. The report is concrete and useful, but the fix needs maintainer/security judgment because durable raw stderr can expose sensitive local/runtime data and likely belongs first in the upstream ACPX runtime boundary.

## What This Changes

Request to persist ACPX child-process stderr to durable per-subprocess logs, with optional retention and configuration, so post-initialize failures can be debugged after the fact.

## Best Possible Solution

Add bounded, permissioned, redacted stderr artifact support in upstream `openclaw/acpx` with typed options for log directory, retention, and discoverability, then wire those options through `extensions/acpx` config, manifest/UI hints, docs, and focused tests for startup and post-initialize stderr capture.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: spawn an ACP child that writes to stderr after initialize with verbose mode off, and current `acpx@0.6.1` only updates the bounded startup buffer plus verbose mirroring, while OpenClaw exposes no persistent stderr log configuration.

## Solution Assessment

No, not as the raw tee sketched in the issue body. The maintainable path should keep child lifecycle behavior in upstream `acpx` and add retention, permissions, redaction/privacy handling, and OpenClaw config/docs/tests rather than writing unbounded raw stderr from the wrapper by default.

## Risks / Open Questions

- Persisting raw ACP stderr can capture credentials, environment details, local paths, prompt/tool content, provider diagnostics, or third-party preload output unless the design includes redaction, restrictive permissions, retention, and discoverability.
- A wrapper-only OpenClaw implementation could drift from `acpx`, which owns the child-process lifecycle; the scoped ACPX guidance points shared runtime behavior upstream.
- Open PR #53997 overlaps this need but is broad and unmerged, so maintainers need to decide whether to review/split it or pursue a narrower upstream ACPX change.
