## Summary

Keep open: current main and v2026.6.6 only provide Codex-specific redacted wrapper stderr tails, while general durable ACPX subprocess stderr capture for post-init failures remains missing and needs product/security design.

## What This Changes

Request durable ACPX child-process stderr logging so operators can debug post-initialize ACP agent failures after the in-memory startup buffer is no longer useful.

## Best Possible Solution

Adopt a bounded, redacted, retention-aware ACPX-owned diagnostic artifact, preferably in openclaw/acpx first, then consume that released behavior from OpenClaw.

## Reproduction Assessment

Yes, as a source-level capability gap. Current acpx 0.10.0 still keeps only bounded startup stderr and non-verbose post-init stderr is not durably persisted; OpenClaw's current code only adds Codex-wrapper stderr tails for generic Codex ACP failures.

## Solution Assessment

Unclear as proposed. Durable stderr diagnostics are useful, but the direct raw per-process log directory is not the best shape without ACPX ownership, redaction, retention, storage-location, and default on/off decisions.
