- `impact:message-loss`: Assistant document attachments can be delivered in chat but become difficult or impossible to download with the intended filename.

## Summary

Keep open: current main and the latest release still stream WebChat assistant-media documents without a `Content-Disposition` filename/download hint, and open candidate fixes exist but have not landed.

## What This Changes

WebChat assistant-media document downloads should use attachment disposition with the original filename while keeping inline media previews working.

## Best Possible Solution

Review and land one focused fix, likely by repairing https://github.com/openclaw/openclaw/pull/77912 or an equivalent replacement, that adds safe `Content-Disposition` filenames for non-inline assistant-media files plus focused WebChat tests.

## Reproduction Assessment

Yes, at source level. Current main streams the assistant-media file without `Content-Disposition`, and the document anchor lacks a `download` attribute; I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is narrow: set a sanitized attachment disposition for non-inline document/binary responses, keep images/audio/video inline, and add regression tests around the gateway response and WebChat document link.
