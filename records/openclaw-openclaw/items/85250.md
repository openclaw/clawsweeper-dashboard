## Summary

Keep open: the docs promise `sessions_yield` waits for subagent completion events, and current source/tests confirm yielded parents are idle while webchat completion delivery depends on the dormant direct-handoff path that the report says is not waking promptly.

## What This Changes

The issue asks OpenClaw to wake a `sessions_yield` parent webchat session immediately when a spawned `mode: "run"` subagent completes.

## Best Possible Solution

Keep `sessions_yield` as an idle end-turn state, but ensure dormant requester completion delivery starts a parent agent turn and streams the webchat result without waiting for another user message, backed by a focused regression test.

## Reproduction Assessment

Yes, partially from source: current tests show `sessions_yield` leaves the parent idle and queue steering cannot wake it, while the issue gives a concrete webchat sequence and observed delay. I did not run a live webchat reproduction, so the exact runtime failure remains unproven in this pass.

## Solution Assessment

No, the suggested fix of treating yielded sessions as active is probably not the safest first move because current tests assert the yielded parent is idle. The narrower maintainable fix is to prove and repair the dormant direct-handoff/webchat wake path.
