## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks the proposed Mission Control runtime dashboard surface, and the PR head has two concrete correctness blockers already visible from source and review discussion.

## What This Changes

The branch adds Gateway dashboard summary/delta, incident/tool/process/autonomy runtime data, Control UI Mission Control panels and polling, plus related process-session metadata, node, script, Docker/test, navigation, and style changes.

## Best Possible Solution

A maintainable end state would be a narrower current-base runtime-ops change that adds typed dashboard.summary/dashboard.delta, tool activity, process visibility, incident state, and autonomy posture through existing Gateway method/scope patterns, with docs/protocol updates and focused Gateway/UI regression coverage.

## Reproduction Assessment

Yes for the blocking review findings. The startup crash is reproducible by firing a cron or agent-event dashboard callback before requestContext is assigned, and the tool-status bug is reproducible by feeding a phase: "result" event with isError: true and a plain string/object result into the PR registry.

## Solution Assessment

No. The product direction is plausible, but this PR is too broad and currently has confirmed correctness blockers; a narrower rework that fixes startup scheduling and tool failure classification is safer before reviewing the larger API/UI shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard dashboard scheduling until context exists:** `src/gateway/server.impl.ts:421-422`
  - body: The new callbacks close over `requestContext` before it is assigned. If cron or an agent event fires during the startup awaits, this passes `undefined` into `scheduleDashboardDelta`, whose `WeakMap.has(context)` throws and can crash the callback path. Move context construction before registering these callbacks or guard the closure until the context is available.
  - confidence: 0.94
- **[P2] Classify result failures from isError:** `src/gateway/tool-activity-registry.ts:204-205`
  - body: `handleToolExecutionEnd` emits result events with `data.isError`, but the registry ignores that flag and only inspects `data.result`. Failed tools whose result is a plain string/object will be marked `completed`, so Mission Control underreports recent failures and shows the wrong tool status.
  - confidence: 0.91
