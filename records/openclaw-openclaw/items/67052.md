## Summary

Keep open: current main still lacks a prompt content-completion or finalizing transition for the TUI; the watchdog and related merged PRs only mitigate stale/lost terminal events.

## What This Changes

Request that the TUI stop showing active streaming promptly after visible assistant content finishes, even if gateway lifecycle finalization is still doing post-run work.

## Best Possible Solution

Add a backward-compatible content-completion or finalizing path between gateway and TUI, keeping watchdog recovery and covering delayed lifecycle finals, hidden/tool-only output, concurrent runs, and reconnects.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a chat delta sets TUI status to streaming, lifecycle final can wait on post-response flush work, and current protocol has no content-complete/finalizing state before final or the watchdog.

## Solution Assessment

Unclear. The issue is real, but the maintainable fix requires maintainer UX/protocol choice between an additive gateway signal and a carefully bounded TUI heuristic.
