## Summary

Keep open: current main still couples the TUI `streaming` status to chat deltas until a later terminal final/error/abort path or the watchdog timeout, and related merged PRs are mitigations rather than the requested prompt content-completion transition.

## What This Changes

The issue asks for the TUI to stop showing `streaming` promptly when visible assistant content has finished, instead of waiting for delayed final lifecycle delivery.

## Best Possible Solution

Add a backward-compatible content-completion or finalizing path that lets the TUI stop presenting visible text generation as active streaming while preserving final lifecycle cleanup and watchdog recovery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a chat `delta` sets the TUI to `streaming`, visible text can then stop while lifecycle final waits on post-run flush work, and there is no typed content-complete state before `final` or the watchdog.

## Solution Assessment

Unclear. The issue is real, but maintainers should choose between an additive gateway content-complete/finalizing event and a bounded TUI heuristic before implementation.
