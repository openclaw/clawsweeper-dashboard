## Summary

Keep open: current main and the latest release still couple the TUI streaming indicator to chat delta/final/lifecycle timing, with only watchdog recovery; related merged TUI work explicitly left the prompt content-completion/finalizing UX unresolved.

## What This Changes

The issue asks for the TUI to stop showing `streaming` promptly after visible assistant content finishes instead of waiting for delayed final lifecycle delivery.

## Best Possible Solution

Add a backward-compatible content-completion or finalizing path between gateway and TUI, keeping watchdog recovery and covering delayed lifecycle finals, hidden/tool-only output, concurrent runs, and reconnects.

## Reproduction Assessment

Yes, at source level. A chat delta sets the TUI status to `streaming`, lifecycle final can wait on post-response flush work, and current protocol has no content-complete/finalizing state before final or the watchdog.

## Solution Assessment

Unclear. The issue is real, but the maintainable fix requires a maintainer UX/protocol choice between an additive gateway signal and a carefully bounded TUI heuristic.
