## Summary

Keep open. Current main has watchdog, orphan-final, and reconnect mitigations for stale TUI streaming state, but it still lacks the requested prompt content-complete/finalizing transition; visible assistant output can stop while the footer remains `streaming` until lifecycle/chat finalization or the watchdog timeout.

## What This Changes

The issue requests that the TUI stop presenting a response as actively streaming once visible assistant content has finished, instead of waiting on delayed final lifecycle/chat completion.

## Best Possible Solution

Add a backward-compatible content-complete or finalizing path across the gateway/TUI boundary so visible assistant text generation stops being shown as active streaming while post-run finalization continues; keep the watchdog as recovery.

## Reproduction Assessment

Yes. A source-level reproduction is an active TUI run that receives assistant `delta` events, then visible text stops while lifecycle/chat final is delayed; current main remains `streaming` until terminal handling or the watchdog timeout. I did not run a live TUI session in this read-only review.

## Solution Assessment

Yes. A narrow, additive content-complete/finalizing state is the maintainable direction; a TUI-only silence heuristic would be easier but risks hiding legitimate active tool work or late terminal failures.
