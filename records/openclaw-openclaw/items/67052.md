## Summary

Keep open. Current main has stale-streaming mitigations, but it still lacks the requested prompt content-complete or finalizing transition: TUI deltas set the status to streaming, normal cleanup is still terminal-event driven, and the gateway emits chat final from lifecycle terminal handling that can wait for async flush work. Merged PR #73749 also explicitly left this issue open as remaining content-completion timing work.

## What This Changes

The issue asks for the TUI to stop showing active streaming promptly after assistant-visible response content finishes, instead of waiting tens of seconds for the final lifecycle/chat event.

## Best Possible Solution

Add a compatible content-completion or finalizing path that lets the TUI stop presenting assistant-visible answer generation as active `streaming` once visible content is complete, while still tracking the run until final/error/abort cleanup finishes. Keep the watchdog as recovery, update typed protocol/docs if a new event or field is introduced, and cover delayed lifecycle finals, concurrent runs, hidden/tool-only output, local `/btw`, and reconnect fallback behavior.

## Reproduction Assessment

Yes. The source-level reproduction is an active TUI run that receives assistant `delta` events, stops producing visible text, then waits while backend lifecycle terminal handling flushes/persists before `chat` final arrives; the status remains `streaming` until that terminal path or the watchdog fires.

## Solution Assessment

Yes. A narrow TUI/gateway state-machine change is the maintainable direction; a broad immediate idle shortcut or pure silence timeout would be riskier because it could hide active runs, tool-only work, or delayed final/error state.
