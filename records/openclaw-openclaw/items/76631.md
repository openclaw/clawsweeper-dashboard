## Summary

Keep open: current main still has the risky Prometheus allowlist snippet without an adjacent strict-mode warning, and source confirms omitted plugin ids are disabled under a non-empty allowlist. The PR is useful, but one warning sentence overstates current slot and channel exceptions.

## What This Changes

The PR adds a Warning callout to the Prometheus quick start explaining that `plugins.allow` is strict and should be omitted or include all desired plugin ids on existing gateways.

## Best Possible Solution

Merge a precise Prometheus warning beside the snippet that describes strict allowlist behavior, avoids overstating slot/channel exceptions, and leaves runtime activation unchanged.

## Reproduction Assessment

Yes for the docs gap and source behavior: current main has the allowlist snippet without a warning, and `resolvePluginActivationDecisionShared` disables omitted ids under a non-empty allowlist. I did not run a live gateway or docs preview.

## Solution Assessment

Mostly yes: an additive docs warning is the narrowest maintainable fix for the quick-start pitfall. The safer version should soften the blanket disablement example or name the memory/context slot and conditional bundled-channel exceptions.
