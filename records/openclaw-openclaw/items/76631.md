## Summary

Keep open. Current main still shows the Prometheus quick start with `plugins.allow: ["diagnostics-prometheus"]` and no local warning, while source and existing docs confirm that a non-empty `plugins.allow` is restrictive; the PR is a focused docs-only clarification with no blocking review finding.

## What This Changes

The PR adds a `<Warning>` callout to `docs/gateway/prometheus.md` explaining that `plugins.allow` is strict and must include other desired plugins on existing gateways.

## Best Possible Solution

Merge a focused docs warning once wording is acceptable, while leaving broader `plugins.allow` runtime semantics to the linked follow-up discussion.

## Reproduction Assessment

Yes for the documented behavior: source inspection shows omitted plugin ids are disabled when `plugins.allow` is non-empty, and current docs show the risky Prometheus snippet without the new warning. No live gateway reproduction was run because this is a read-only docs PR review.

## Solution Assessment

Yes. An additive warning beside the snippet is the narrowest maintainable docs fix for the immediate user-facing pitfall; larger changes to bundled/runtime allowlist policy belong in #75575.
