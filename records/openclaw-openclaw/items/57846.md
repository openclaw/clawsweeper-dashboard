## Summary

Keep open. Current main still has one Mattermost mention-gating decision for channel and thread traffic: the monitor computes and passes a thread root, but the gate only accepts explicit mentions, authorized control commands, or onchar triggers, and the strict Mattermost config/docs do not expose a thread-specific mention policy.

## What This Changes

Request a Mattermost config option that keeps @mention requirements for top-level channel messages while allowing unmentioned follow-up replies in bot-participated CRT threads.

## Best Possible Solution

Implement this as a Mattermost plugin-owned thread mention policy: extend the account/group config schema and docs, compute an implicit mention only for safe bot-participated or active-session thread replies, preserve top-level requireMention behavior, and add focused monitor-gating/config regression tests plus a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes. The high-confidence check is the pure mention-gate path: with kind=channel, threadRootId set, requireMention true, wasMentioned false, and canDetectMention true, current main still drops as missing-mention because threadRootId is not part of the effective mention decision.

## Solution Assessment

Yes, a Mattermost plugin-local fix is the narrowest maintainable direction; no core API gap is apparent because the monitor already has threadRootId and session context, while Slack provides an adjacent implicit-thread-mention pattern to adapt carefully.
