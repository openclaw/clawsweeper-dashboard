## Summary

Keep open: current `main` fixes the older `system.which` object-map parser, but the reported `openclaw skills check/list` path still builds skill status without remote eligibility, so the `apple-notes` `darwin`/`memo` requirements can still show missing on a Linux CLI process.

## What This Changes

The issue asks for remote macOS node eligibility to make the bundled `apple-notes` skill eligible in `openclaw skills check/list` when the connected node resolves `memo`.

## Best Possible Solution

Teach the CLI skill status path to include live remote macOS eligibility using the existing remote eligibility contract, while preserving the gateway runtime path and offline-node safeguards.

## Reproduction Assessment

Yes from source, but not from a live remote Mac run in this review. The CLI `skills check/list` path does not pass `eligibility.remote`, while the shared evaluator only clears `memo` and `darwin` when that remote context is supplied.

## Solution Assessment

Yes, the best fix is to wire existing remote eligibility into the CLI status path rather than changing `apple-notes` metadata or reopening the already-fixed parser issue. The repair should use live connected-node state so stale paired metadata does not make offline nodes eligible.
