## Summary

Keep open. Current main still waits for LINE event processing before returning HTTP 200 in the Node handler, Express-style middleware, and shared-path gateway route, and current tests lock in the blocking/error-before-ACK behavior. The related fix PR was closed unmerged and did not cover the shared monitor route.

## What This Changes

The issue requests that LINE webhooks acknowledge with HTTP 200 immediately after valid signature/body handling, then continue event processing asynchronously so slow LLM replies do not cause LINE request timeouts and expired reply tokens.

## Best Possible Solution

Implement the fix inside the LINE plugin: after raw body read, signature validation, target resolution, and JSON parsing succeed, send the 200 response immediately and run event processing fire-and-forget with runtime error reporting. Cover the reusable Node handler, Express middleware, and monitor shared-path route together, with regression tests proving ACK-before-processing.

## Reproduction Assessment

Yes. A signed LINE POST whose handleWebhook/onEvents promise stays pending reproduces the source-level issue; current tests already assert headers are not sent until processing completes and that processing failure returns 500 instead of an already-sent 200.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but the literal two-location patch is incomplete for current main. The monitor shared-path route must be fixed at the same time so installed gateway behavior is covered.
