## Summary

Keep open. Current `main` and the latest release still wait for LINE event processing before returning HTTP 200 in the Node handler, Express-style middleware, and shared monitor route; the prior fix PR is closed unmerged and only covered two of the three ingress paths.

## What This Changes

Request LINE webhook handlers to acknowledge valid webhooks immediately and process events asynchronously so slow replies do not produce LINE request timeouts or invalid reply-token failures.

## Best Possible Solution

Fix this in the LINE plugin by ACKing only after trusted parsing/target resolution, then running event processing fire-and-forget with logged failures across all three webhook entry points.

## Reproduction Assessment

Yes. A signed LINE POST whose `handleWebhook` or `onEvents` promise stays pending reproduces the source-level failure: current handlers do not send HTTP 200 until that promise resolves, and existing tests assert that behavior.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix, but it must include the shared monitor route in addition to the two reusable handlers from the earlier PR.
