## Summary

Keep open. Current main and the latest release still wait for LINE event processing before returning HTTP 200 in the Node handler, Express-style middleware, and shared monitor route; the open closing PR https://github.com/openclaw/openclaw/pull/78186 owns the focused fix but is not merged.

## What This Changes

The issue asks OpenClaw's bundled LINE webhook handlers to acknowledge valid webhooks immediately and continue slow LLM/event processing asynchronously.

## Best Possible Solution

Review, rebase or replace, and land https://github.com/openclaw/openclaw/pull/78186 so all trusted LINE webhook ingress paths ACK after validation and process events asynchronously with regression coverage.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending handleWebhook or onEvents promise is enough: current main awaits that promise before HTTP 200, and the existing LINE test asserts the response remains unsent while processing is pending.

## Solution Assessment

Yes. ACKing after signature/body/target validation and moving only trusted event processing into a caught asynchronous path is the narrow maintainable fix, matching LINE's async-processing guidance and OpenClaw's Telegram precedent.
