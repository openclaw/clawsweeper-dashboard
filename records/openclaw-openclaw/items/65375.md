## Summary

Keep open. Current `origin/main` still waits for LINE event processing before sending HTTP 200 in the Node handler, Express-style middleware, and shared monitor route, while https://github.com/openclaw/openclaw/pull/78186 is an open unmerged closing PR that now owns the fix path.

## What This Changes

The issue asks OpenClaw to acknowledge LINE webhooks immediately after validation and continue slow agent processing asynchronously so LINE does not time out or expire reply tokens.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78186, or close it with a replacement, so all LINE ingress paths ACK after trusted validation and process events asynchronously with regression coverage.

## Reproduction Assessment

Yes. A signed LINE POST with events and a pending `handleWebhook` or `onEvents` promise is enough: current main awaits that promise before sending HTTP 200, and the existing Node handler test asserts `headersSent` is still false while processing is pending.

## Solution Assessment

Yes. ACKing after signature/body/target validation and then running LINE event processing fire-and-forget with runtime error logging is the narrow maintainable fix, already supported by the receive ACK policy contract and Telegram precedent.
