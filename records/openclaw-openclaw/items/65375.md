## Summary

Keep open. Current `main` and the latest tagged release still await LINE webhook event processing before sending HTTP 200 in all three relevant paths, and existing tests preserve the blocking/error-before-ACK behavior. The linked fix PR was closed unmerged, so this remains a narrow valid repair candidate.

## What This Changes

The issue asks for LINE webhooks to acknowledge HTTP 200 immediately after validation and continue slow event processing asynchronously so reply-token expiry does not surface as a false 429.

## Best Possible Solution

Implement the fix in the LINE plugin only: after pre-auth body guards, signature validation, target resolution, and JSON parsing succeed, send the 200 response immediately and run event processing fire-and-forget with runtime error reporting. Preserve the existing reply-token-to-push fallback, loading animation, in-flight limiter, and replay/dedupe behavior, then update regression tests to prove ACK-before-processing for the Node handler, Express middleware, and monitor route.

## Reproduction Assessment

Yes. A signed LINE webhook POST whose `handleWebhook` or `onEvents` promise remains pending reproduces the source-level failure: current code does not send headers until that promise resolves, and current tests explicitly assert that blocking behavior.

## Solution Assessment

Yes, the async ACK-after-validation direction is the narrowest maintainable fix. The issue's two-location patch is incomplete for current `main`, so the best implementation should cover `webhook-node.ts`, `webhook.ts`, and `monitor.ts` together.
