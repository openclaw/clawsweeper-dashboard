## Summary

Keep open: current `main` still validates send payloads against the canonical `message` field only, while the existing normalization path handles routing aliases but not model-generated text-body aliases like `SendMessage`. The report is specific, matches the source path, and is narrow enough for a focused fix.

## What This Changes

Issue requests that model-generated `SendMessage` text arguments for the core `message` tool be normalized to the canonical `message` field before validation.

## Best Possible Solution

Add bounded pre-validation normalization for common send-body aliases such as `SendMessage`, `content`, or `text` into canonical `message` only when `message` is absent, then cover it with focused regression tests.

## Reproduction Assessment

Yes by source inspection, though not by live Discord run: a send payload with only `SendMessage` bypasses the current alias normalizer and reaches the required canonical `message` read as missing.

## Solution Assessment

Yes: central alias normalization before send validation is the narrow maintainable fix, provided canonical `message` remains authoritative and aliases are only used as fallback text bodies.
