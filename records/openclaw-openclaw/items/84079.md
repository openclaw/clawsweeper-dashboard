## Summary

Keep open: current main and the latest shipped release still accept only the canonical `message` send-body field, while the linked fix PR is open and not merged. The reporter's logs match the source path: `SendMessage` is never normalized before validation, so the send is rejected before Discord delivery.

## What This Changes

Normalize model-generated send-body aliases such as `SendMessage` to canonical `message` before `message` tool send validation.

## Best Possible Solution

Land a narrow core message-action fix that fills `message` from `SendMessage`, `content`, or `text` only when canonical `message` is absent, with focused regression coverage and no broad second source of truth for plugin parameters.

## Reproduction Assessment

Yes by source inspection, though not by a live Discord run: a send payload with only `SendMessage` bypasses the current normalizer and then fails the required canonical `message` read.

## Solution Assessment

Yes: central pre-validation alias normalization is the narrowest maintainable fix, provided canonical `message` remains authoritative and aliases are fallback-only. The issue should remain open while the linked PR is reviewed or replaced.
