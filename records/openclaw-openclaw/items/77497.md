## Summary

Keep open: current `main` still omits event-time text from the cron alert bodies named in the report, and open implementation PRs are already linked with closing syntax for this issue.

## What This Changes

Request to include the cron event timestamp in single-failure, repeated-failure, and auto-disable alert bodies so delayed channel delivery is not mistaken for a fresh failure.

## Best Possible Solution

Land one canonical implementation that formats cron alert event time from existing run metadata, follows the existing cron schedule timezone fallback, and leaves alert routing and job schema unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an additive feature request. Source inspection gives a high-confidence current-main check: the relevant cron alert builders still omit event-time text while event metadata such as `runAtMs` is already present nearby.

## Solution Assessment

Yes, keeping this open for the linked implementation PR is the right path. The narrow maintainable fix is timestamp formatting and plumbing through the existing cron alert boundary, not a new routing mode or config surface.
