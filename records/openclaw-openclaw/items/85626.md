- `impact:message-loss`: The failure prevents the resumed user turn from reaching the selected CLI runtime and producing the expected reply.

- `impact:auth-provider`: The failure crosses auth profile/epoch rotation and provider/runtime routing for `claude-cli`.

## Summary

Keep open: current main and v2026.5.20 still show the reported CLI-runtime versus harness-selection asymmetry, and an open linked fix PR uses closing syntax for this issue, so the issue should stay open until one resolver fix lands.

## What This Changes

The issue reports `MissingAgentHarnessError` on resumed CLI-backed sessions because `selectAgentHarnessDecision` treats `claude-cli` as a missing plugin harness instead of a valid CLI runtime.

## Best Possible Solution

Land one narrow resolver fix that routes supported CLI runtime aliases or configured CLI backends through the PI/CLI boundary while preserving strict `MissingAgentHarnessError` behavior for unknown plugin harness ids, then close this issue and related duplicates.

## Reproduction Assessment

Yes for the core failure path by source inspection: a forced `claude-cli` runtime entering `selectAgentHarnessDecision` reaches `MissingAgentHarnessError` because only plugin harnesses are checked. I did not execute the full resumed-session auth-rotation flow in this read-only review.

## Solution Assessment

Yes, the requested short-circuit is the right narrow boundary fix if it reuses existing CLI-runtime classification and keeps unknown plugin harness ids fail-closed. The implementation path should come from one reviewed open fix PR rather than a new duplicate branch.
