- `impact:auth-provider`: The failure involves auth epoch/profile rotation and CLI runtime/provider routing for `claude-cli`.

## Summary

Keep open: current main still has the reported resolver asymmetry, with `selectAgentHarnessDecision` throwing for forced CLI runtime ids while the model-fallback availability path explicitly short-circuits CLI runtimes.

## What This Changes

The issue reports that resumed sessions with stale `cliSessionBindings.*.authEpoch` can throw `MissingAgentHarnessError` for valid CLI runtimes such as `claude-cli`.

## Best Possible Solution

Keep CLI backend ids out of embedded harness selection by sharing the existing CLI-runtime classification with `selectAgentHarnessDecision`, then add focused coverage for resumed or override-driven Claude CLI session resolution.

## Reproduction Assessment

Yes for the core failure path by source inspection: a forced runtime of `claude-cli` entering `selectAgentHarnessDecision` reaches `MissingAgentHarnessError` because only plugin harnesses are checked. I did not execute the full resumed-session auth-rotation flow in this read-only review.

## Solution Assessment

Yes, the narrow fix is to centralize or reuse CLI-runtime detection so valid CLI backend ids bypass embedded harness lookup consistently, without changing plugin harness fail-closed behavior for real missing harnesses.
