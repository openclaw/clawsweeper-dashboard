## Label Justifications

- `P1`: The PR targets a recurring Telegram inbound stall where users can stop receiving agent responses until restart or manual queue repair.
- `merge-risk: 🚨 message-delivery`: The new path can fail a live-owned Telegram update and process later same-lane messages, changing whether an inbound message is delivered, dropped, or duplicated.
- `merge-risk: 🚨 session-state`: The diff mutates durable ingress claim state based on local handler absence and age, which can diverge from the real state of a live owner process.

## Summary

Keep open: Mantis now supplies real Telegram proof, but the patch still fails live-owned Telegram ingress claims using only this gateway's local handler absence plus claim age, which can dead-letter work that another live worker is still processing.

## What This Changes

The PR adds Telegram polling-session recovery that fails timed-out live-owned spooled claims before draining later same-lane updates, plus a regression test for that path.

## Best Possible Solution

Gate live-owner claim failure on a maintainer-approved stale-owner signal, such as diagnostic non-progress evidence or a durable owner lease, while keeping the token-guarded queue mutation and the Telegram proof coverage.

## Reproduction Assessment

Yes. The Mantis Telegram Desktop proof shows current main leaving the stuck-lane message unanswered while the candidate replies, and source inspection shows main keeps fresh other-live-owner claims guarded.

## Solution Assessment

No. The PR addresses a real failure, but failing another live owner's claim from the drain loop is not the safest proven boundary without diagnostic-backed non-progress proof, lease semantics, or explicit maintainer policy.
