## Summary

Keep open for maintainer review: this PR targets a real P1 Telegram ingress stall, but the diff fails live-owned claims from the local drain loop based only on age and this gateway's handler state, so it needs policy approval and real behavior proof before merge.

## What This Changes

The PR adds a Telegram polling-session path that fails timed-out spooled ingress claims owned by another live process and adds a regression test for later same-lane update drainage.

## Best Possible Solution

Recover only claims that have a maintainer-approved stale/non-progress proof, such as diagnostic stuck-session evidence or a durable owner lease, and validate that later Telegram messages drain without duplicate or dropped delivery.

## Reproduction Assessment

Yes, source inspection shows current main keeps other-live-process Telegram claims guarded while later same-lane updates remain pending; the linked issue adds production logs and database rows for the same symptom. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear. The PR is aimed at the right failure, but failing another live owner's claim from the drain loop is not yet the safest proven owner boundary; diagnostic-backed non-progress proof or a lease-style stale-owner rule would be more maintainable.
