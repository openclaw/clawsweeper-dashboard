## Summary

Keep open: the report is plausible, and current main still has an unbounded post-core fresh-process wait that can leave `openclaw update` stuck after the package-manager step if the child stays alive before writing its result.

## What This Changes

Report asks for Windows `openclaw update` to continue post-update checks and return to the PowerShell prompt after the package-manager update completes.

## Best Possible Solution

Add a bounded post-core child wait with clear diagnostics and termination, plus a regression for the child-never-exits/no-result path, while leaving the external-updater redesign to the broader tracker.

## Reproduction Assessment

No high-confidence live reproduction yet: the report gives a concrete Windows PowerShell symptom, and source inspection shows a matching unbounded post-core child wait, but I did not execute the Windows update path.

## Solution Assessment

No, closing as implemented would be premature: current fixes cover result-file and explicit-exit paths, but the parent still waits indefinitely if the post-core child remains alive before writing its result.
