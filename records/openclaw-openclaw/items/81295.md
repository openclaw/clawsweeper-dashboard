## Summary

This should stay open: the report matches a real gateway startup/plugin-planning boundary, and current main still filters runtime loading through a startup ID plan instead of plainly loading every explicitly enabled non-bundled plugin. The metadata snapshot path is broader than “stock only” on main, but I did not find enough proof that the reported npm/global/workspace plugin drop is fixed end to end.

## What This Changes

The issue reports that gateway deferred loading skips enabled non-bundled plugins discovered from npm/global/workspace roots and `plugins.load.paths`.

## Best Possible Solution

Make gateway startup planning include explicitly trusted non-bundled plugins when their config says they should run, while preserving manifest-first discovery, allowlist/denylist gates, and lazy loading for untrusted ambient discoveries.

## Reproduction Assessment

No live reproduction was run, but source inspection gives a high-confidence path to reproduce with a fixture: create a non-bundled plugin discovered from an install record or `plugins.load.paths`, set `plugins.entries.<id>.enabled=true`, and assert gateway startup does not pass it to `loadOpenClawPlugins` unless it matches a startup predicate.

## Solution Assessment

Unclear as a final patch, but the maintainable direction is narrow: fix the startup planner/load handoff for explicitly trusted non-bundled plugins and add regression coverage, rather than bypassing the metadata snapshot or broad-loading every discovered plugin.
