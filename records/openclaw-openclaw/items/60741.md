## Summary

Keep open. Current main covers much of the requested session management surface, but the visible WebChat New session control still resets the active session instead of creating and switching to a preserved dashboard session; open PR #73725 is already the focused implementation path for that remaining gap.

## What This Changes

The issue requests ChatGPT-style WebChat/Control UI multi-session management with a session list, switching, preserved new sessions, naming, and deletion.

## Best Possible Solution

Land #73725 or an equivalent narrow change that makes the WebChat New session affordance call sessions.create, switch Chat to the returned key, refresh the session options, keep the old session visible in sessions.list, and leave /new or sessions.reset as explicit reset behavior with focused UI/controller coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: click the New session plus button in WebChat and it calls onNewSession, which sends /new and rewrites the active session store entry instead of creating a new switchable session.

## Solution Assessment

Yes for the proposed direction, but not for current main. Reusing the existing sessions.create backend contract is the narrowest maintainable fix; changing reset semantics or adding another metadata store would be broader than needed.
