## What This Changes

The PR updates session-write-lock orphan classification so untracked current-process locks with matching process start time are reclaimed during acquisition and stale-lock cleanup, with regression tests and a changelog entry.

## Best Possible Solution

Land this PR or an equivalent current-main patch after CI triage, preserving the current-PID plus matching-starttime plus untracked-`HELD_LOCKS` guard and shared cleanup/acquisition inspection path, then close #75805 and reconcile #49603/#49660 against the landed fix.

## Reproduction Assessment

Yes. Create a fresh `.jsonl.lock` owned by `process.pid` with `starttime` equal to the current process start time while `HELD_LOCKS` has no normalized-session entry, then call `cleanStaleLockFiles()` or `acquireSessionWriteLock()`; current main preserves or waits on that lock because the valid-starttime orphan-self path returns false and cleanup bypasses the helper.

## Solution Assessment

Yes, functionally. Matching current PID and process start time while requiring the lock to be absent from `HELD_LOCKS` is narrower than deleting all current-PID locks, and sharing the inspection path between acquisition and cleanup avoids divergent recovery behavior.
