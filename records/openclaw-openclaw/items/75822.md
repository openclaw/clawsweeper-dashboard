## Summary

Keep open. Current main still preserves or waits on a fresh current-process lock when the lock payload has a valid matching starttime, and this PR is a narrow functional fix with regression coverage and no blocking code-review finding. The remaining work is maintainer merge/reconciliation because #49660 overlaps the older cleanup path and a few exact-head checks were still running at review time.

## What This Changes

The PR updates session write-lock orphan-self detection so untracked current-process locks with matching starttime can be reclaimed during acquisition and stale-lock cleanup, with tests and a changelog entry.

## Best Possible Solution

Land this PR or an equivalent current-main patch after exact-head checks finish and maintainers reconcile #49660, preserving the current-PID, matching-starttime, and absent-HELD_LOCKS guards while keeping acquisition and cleanup classification shared.

## Reproduction Assessment

Yes. Create a fresh .jsonl.lock owned by process.pid with starttime equal to the current process start time while HELD_LOCKS has no normalized-session entry, then call cleanStaleLockFiles() or acquireSessionWriteLock(); current main keeps or waits on that lock.

## Solution Assessment

Yes, functionally. Matching starttime plus an absent HELD_LOCKS entry is narrower than deleting every current-PID lock, and sharing the inspection path avoids divergent acquisition and startup/doctor cleanup behavior.
