## Summary

Keep open. Current main still preserves current-process lock files that have a valid matching starttime when they are absent from `HELD_LOCKS`, while this PR adds a narrow shared cleanup/acquisition path with regression coverage and a changelog entry; the remaining blockers are merge-gate/canonicalization work, not a close reason.

## What This Changes

The PR updates session write-lock orphan-self detection so untracked current-process locks with matching starttime can be reclaimed during acquisition and stale-lock cleanup, with tests and a changelog entry.

## Best Possible Solution

Land this PR or an equivalent current-main patch after CI and related-PR reconciliation, preserving the current-PID, matching-starttime, and absent-`HELD_LOCKS` guard while sharing the inspection path between acquisition and startup cleanup.

## Reproduction Assessment

Yes. A source-level reproduction is to create a fresh `.jsonl.lock` owned by `process.pid` with `starttime` equal to the current process start time while `HELD_LOCKS` has no normalized-session entry, then call `cleanStaleLockFiles()` or `acquireSessionWriteLock()`; current main preserves or waits on that lock.

## Solution Assessment

Yes, functionally. The PR’s matching-starttime plus absent-`HELD_LOCKS` guard is narrower than deleting all current-PID locks, and sharing the inspection path avoids divergent acquisition and cleanup behavior; the merge path still needs validation and related-PR canonicalization.
