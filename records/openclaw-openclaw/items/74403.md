## Summary

Current main still applies `extra_body` after provider wrapping without removing stale `reasoning_content`, so the linked DeepSeek disabled-thinking failure remains plausible; this PR is useful but still needs the required changelog entry and maintainer confirmation of the generic cleanup scope.

## What This Changes

This PR adds post-`extra_body` cleanup to remove `reasoning_content` from OpenAI-completions messages when the final payload has `thinking.type: "disabled"`, plus a DeepSeek V4 regression test.

## Best Possible Solution

Land this or an equivalent final-payload cleanup with regression coverage and a one-line changelog entry, while preserving enabled-thinking replay behavior and leaving the broader `extra_body` collision discussion to the separate framework-level issue.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: DeepSeek V4 can add or preserve `reasoning_content`, then the current wrapper order lets `extra_body` make the final payload `thinking.type: "disabled"` without a final cleanup.

## Solution Assessment

Yes, with a scope check. Cleaning the final payload after `extra_body` matches the actual request state; if maintainers want stricter owner boundaries, the safer alternative is the same cleanup gated to DeepSeek-compatible payloads.
