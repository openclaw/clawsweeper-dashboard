## Summary

Keep open: the APK use case is still absent from current main, but this PR is not merge-ready because it expands the host-local attachment security boundary, the PEM fallback can send printable private-key files, and the branch conflicts with current media-core MIME ownership.

## What This Changes

The PR adds APK and PEM-style certificate MIME mappings, host-read media allowlist entries, docs wording, and focused media/outbound tests for host-local attachment sends.

## Best Possible Solution

Rebase onto current main, move accepted MIME changes into media-core, keep APK only if maintainers accept that boundary, and either split certificate support out or add certificate-only validation with private-key rejection.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding: PR-head source adds PEM/X.509 aliases to the text fallback, and that fallback accepts printable text without certificate-only checks. The requested APK capability is feature behavior supported by contributor terminal and Telegram-path proof rather than a current-main bug reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. APK support may be a narrow addition, but PEM-style certificates cannot safely reuse the generic text gate; the better path is certificate-only validation or private-key rejection plus a rebase to current media-core ownership.
