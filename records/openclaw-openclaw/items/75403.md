## Reproduction Assessment

Yes from source inspection: current main allows cleanup to run before Feishu records removable typing state, so the one-shot stop can no-op and the later successful start can leave a persistent reaction behind. I did not run a live Feishu API reproduction in this read-only review.

## Solution Assessment

Yes, with compatibility acceptance: serializing cleanup in the shared helper is narrower than making Feishu block reply start again and it preserves the fire-and-forget reply path. The safer alternative, if maintainers reject duplicate stop, is a Feishu-owned pending-aware cleanup path.
