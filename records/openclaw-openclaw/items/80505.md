## Summary

Keep open: the latest head addresses the earlier cancellation and silence-allocation findings and includes sufficient live Discord proof, but source review found one remaining cancellation-state bug in the new prebuffer path.

## What This Changes

The PR changes Discord/OpenAI realtime voice handling to send `noise_reduction: null` for OpenAI backend sessions, append Discord speaker-turn trailing silence, prebuffer raw PCM playback, and add focused tests/changelog coverage.

## Best Possible Solution

Preserve the stability changes, but make cancelled prebuffer cleanup explicitly clear the exact-speech response state when no Discord audio resource was started, with a focused regression test.

## Reproduction Assessment

Yes for the remaining review finding by source inspection: the PR buffers audio before playback, marks exact speech as started on the first buffered chunk, and `@discordjs/voice` does not transition an already-idle player on `stop(true)`. The original live Discord failure is supported by the PR body logs rather than an independent local live repro.

## Solution Assessment

No, not as-is. The null noise-reduction, trailing-silence, and prebuffer direction is reasonable, but the cancellation path needs to finish exact-speech bookkeeping when it discards audio before playback starts.
