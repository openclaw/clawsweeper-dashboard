## Summary

Keep this PR open: it is a focused Discord/OpenAI realtime voice bug fix with credible live proof, but the proposed prebuffer path introduces a cancellation leak and the new trailing-silence allocation needs bounds before merge.

## What This Changes

The PR changes the OpenAI realtime backend session payload, Discord realtime speaker-turn closing, Discord raw PCM playback prebuffering, tests, and changelog for Discord VC stability.

## Best Possible Solution

Keep the stability fix direction, but discard buffered output on cancelled responses and clamp trailing-silence duration before allocation, with regression coverage for both paths.

## Reproduction Assessment

Yes for the blocking regression introduced by this PR: source inspection shows cancelled responses still call the same finisher that now starts playback. The original live Discord failure was not reproduced locally, but the PR body supplies credible after-fix runtime logs.

## Solution Assessment

No, not as-is. The null noise-reduction and prebuffer approach is plausible, but the playback finisher must distinguish normal completion from cancellation and the new silence buffer needs finite bounds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Discard prebuffered audio when a response is cancelled:** `extensions/discord/src/voice/realtime.ts:706`
  - body: `finishOutputAudioStream` now force-starts playback before ending the stream, but the caller uses it for both `response.done` and `response.cancelled`. If a barge-in cancels a response before the preroll threshold, the buffered assistant audio is played anyway instead of being discarded, weakening interruption behavior.
  - confidence: 0.9
- **[P2] Clamp trailing silence before allocating it:** `extensions/discord/src/voice/realtime.ts:822`
  - body: `silenceDurationMs` comes from raw realtime provider config at the Discord layer, then directly controls `Buffer.alloc(silenceBytes)`. A very large finite value can allocate a huge buffer and stall or crash the voice process; clamp to a reasonable maximum before allocation.
  - confidence: 0.82

## Security Review
