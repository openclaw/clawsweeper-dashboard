## Summary

Keep open: this is a plausible, well-scoped Discord/OpenAI realtime voice bug fix with supplied live proof, but the new prebuffer path can play cancelled assistant audio and the trailing-silence allocation needs a bound before merge.

## What This Changes

The PR disables backend GA input noise reduction for OpenAI realtime voice, appends trailing silence on Discord speaker-turn close, prebuffers Discord raw PCM playback, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the owner-local Discord/OpenAI realtime fix direction, but distinguish normal completion from cancellation so preroll is discarded on cancelled responses and clamp finite trailing-silence duration before allocation.

## Reproduction Assessment

Yes for review purposes: the PR body provides live Discord VC logs showing the addressed multi-turn behavior after the patch, and source inspection gives a clear reproduction path for the cancellation/prebuffer regression. I did not independently run live Discord in this read-only review.

## Solution Assessment

No, not yet. The provider-local payload and Discord playback direction is appropriate, but the proposed implementation should discard cancelled preroll audio and bound the trailing-silence allocation before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Discard preroll when realtime responses are cancelled:** `extensions/discord/src/voice/realtime.ts:709`
  - body: `finishOutputAudioStream()` now force-starts buffered audio before ending the stream, and `onEvent` calls it for both normal `response.done` and cancelled responses. If the model is interrupted before the 25-frame preroll threshold, audio that was never played gets started at cancellation time instead of being discarded, weakening barge-in and leaking cancelled speech. Treat `response.cancelled` and `response.done` with cancelled status as a clear/reset path rather than a forced playback completion.
  - confidence: 0.88
- **[P2] Clamp trailing-silence duration before allocating:** `extensions/discord/src/voice/realtime.ts:824`
  - body: `silenceDurationMs` is read from arbitrary provider config and converted directly into `Buffer.alloc(silenceBytes)`. A bad value can allocate a very large buffer or throw during speaker-turn close, taking down the realtime voice loop. Validate it as finite/non-negative and cap the extra trailing silence to a small maximum before allocating.
  - confidence: 0.8
