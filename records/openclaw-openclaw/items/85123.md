## Label Justifications

- `P1`: The patch touches plugin runtime loading and can break existing packaged installs at startup or plugin activation.
- `merge-risk: 🚨 compatibility`: The loader would start writing a dist node_modules alias in packaged installs that currently treat the package tree as read-only.
- `merge-risk: 🚨 availability`: Permission failures during alias creation can stop plugin loading before the targeted external plugin runtime runs.
- `merge-risk: 🚨 security-boundary`: The patch adds a process-wide NODE_PATH entry that changes later bare-module resolution outside the specific plugin load call.

## Summary

Keep open: the PR is aimed at a real external-plugin SDK resolution gap, but the current patch would mutate the packaged OpenClaw dist tree during plugin loading and prepend a process-wide NODE_PATH entry. That conflicts with the documented read-only package-tree contract and can break root-owned packaged installs; the PR also lacks after-fix real behavior proof.

## What This Changes

The branch adds Discord SDK facade exports for component and poll sends, exposes a generated SDK alias package through NODE_PATH for external plugin runtimes, and adds createRequire-oriented regression coverage.

## Best Possible Solution

Supply the SDK alias through a build, package, install, or plugin-local contract that preserves read-only packaged installs, and keep any global module-resolution change narrowly justified and tested.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows current native createRequire paths do not preserve the loader alias map after plugin load, but I did not run a live packaged external plugin because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? No: the SDK alias goal is valid, but runtime writes to the package tree and process-wide NODE_PATH mutation are not the narrowest maintainable fix for packaged external plugins.
