## Summary

Keep open: current main does not show a runtime path that can synthesize cache-write tokens when a provider only reports cache hits, but it does support persisting `cacheWrite` when upstream emits a write-side counter. The remaining work is a narrow documentation and regression-test clarification, not an auto-close.

## What This Changes

The issue asks whether session `cacheWrite` telemetry should be populated when `cacheRead` is nonzero or documented as intentionally unavailable for some providers.

## Best Possible Solution

Document session cache counters as provider-reported telemetry, preserve upstream `cacheWrite` when present, and avoid deriving write tokens from cache hits unless a provider contract explicitly exposes that value.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the reported shape: OpenAI-style usage with cached tokens produces `cacheRead` and leaves `cacheWrite` at zero, while the session store only persists a nonzero write count if upstream usage already includes one. I did not run a live provider fleet probe.

## Solution Assessment

Yes, the maintainable fix is to clarify the provider-reported contract and add a focused regression if needed; synthesizing cache writes from cache reads would make session telemetry less accurate for providers that do not expose write-side accounting.
