## Summary

Keep this PR open. Current main still uses exact mutating-action fingerprints, so the target-mismatched retry behavior proposed here is not implemented, but the current PR head still needs maintainer review because it can clear an unresolved same-tool failure without evidence that the later success is the actual retry.

## What This Changes

This PR changes agent tool-error recovery so a successful mutating call with the same tool/action type can clear a previous `lastToolError` despite different target fingerprints, escapes fingerprint separators, and adds handler, subscriber, and tool-mutation tests.

## Best Possible Solution

Define the recovery boundary explicitly before merge. A maintainable fix should either preserve exact-fingerprint clearing, add a narrower retry signal for target-mismatched recovery, or change the UI to show/collapse both the failed attempt and later success instead of inferring recovery from tool/action type alone.

## Reproduction Assessment

Yes. Current main has a checked-in path: fail `write` on `/tmp/a.txt`, succeed `write` on `/tmp/b.txt`, and `getLastToolError()?.toolName` remains `write`; the PR false-clear path can be reproduced by making the second write unrelated rather than a retry.

## Solution Assessment

No/unclear. The PR addresses a real UX complaint and has useful coverage, but clearing by same tool/action type is not enough to prove the failed operation recovered; a safer solution needs an explicit retry relationship or a UI treatment that preserves both attempts.
