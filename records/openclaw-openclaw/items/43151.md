## Summary

Keep open: current main still has the reported warning behavior, but this PR should not merge as-is because its target-agnostic same-action fallback can hide unresolved failures and it lacks the required changelog entry.

## What This Changes

The PR changes agent tool-error recovery to clear `lastToolError` after successful same tool/action-type mutating calls with different target fingerprints, escapes fingerprint separators, and adds handler, subscriber, and mutation tests.

## Best Possible Solution

Define the recovery boundary explicitly: keep exact-fingerprint clearing, add a real retry relationship for target-mismatched recovery, or preserve both the failed attempt and later success in the UI, then update tests and changelog around that contract.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `lastToolError` set for failed `write /tmp/a` then successful `write /tmp/b`, and the checked-in subscriber test asserts that behavior.

## Solution Assessment

No/unclear. The PR addresses a real UX complaint, but clearing by same tool/action type is not the narrowest maintainable proof of recovery; an explicit retry signal or UI representation of both attempts is safer.
