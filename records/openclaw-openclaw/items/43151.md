## Summary

Keep this PR open. Current main still keeps the exact mutating-action fingerprint contract, the checked-in subscribed-session regression test still expects a same-tool success on a different target to leave the prior write failure unresolved, and this PR is the active implementation attempt for the linked tool-warning UX problem rather than work already implemented on main.

## What This Changes

This PR changes agent tool-error handling so a successful mutating call with the same tool/action type can clear a previous `lastToolError` despite different target fingerprints, and adds tool-mutation plus subscriber tests for retry and false-clear cases.

## Best Possible Solution

Keep this PR open for maintainer review. The best end state is either a landed implementation that clearly defines and tests when a target-mismatched tool success counts as recovery, or an explicit maintainer close explaining that exact target fingerprints are the intended warning boundary.

## Reproduction Assessment

Yes. Current main has a checked-in regression path: fail `write` on `/tmp/a.txt`, succeed `write` on `/tmp/b.txt`, and `getLastToolError()?.toolName` remains `write`; the payload builder can then render the warning from that retained state.

## Solution Assessment

No/unclear. The PR is focused and has useful coverage, but the current same tool/action-type fallback still lacks a signal that the later success is actually a retry of the failed operation rather than an unrelated later mutation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Keep retry clears tied to an actual retry:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:833-841`
  - body: For single-action tools like `write`, the new same-action-type path treats `tool=write|path=/protected` and `tool=write|path=/tmp/log` as the same action type. That means any later successful write can clear an unresolved failed write and suppress the warning even when the failed target was never retried. Require an additional retry signal or narrower match before clearing target-mismatched failures.
