## Summary

Keep this PR open. Current main has not implemented the requested recovery behavior, but the PR still clears unresolved failures on later same tool/action-type successes without proving the failed operation was actually retried, so the warning semantics need maintainer review before merge.

## What This Changes

This PR changes agent tool-error recovery to clear `lastToolError` after successful same tool/action-type mutating calls with different target fingerprints, escapes fingerprint separators, and adds handler, subscriber, and mutation tests.

## Best Possible Solution

Define the recovery boundary explicitly: keep exact-fingerprint clearing, add a real retry relationship for target-mismatched recovery, or show both the failed attempt and later success, then update tests and changelog around that contract.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `lastToolError` set for fail `write /tmp/a` then success `write /tmp/b`, and the PR's target-agnostic action-type fallback can be followed from the diff for unrelated same-type successes.

## Solution Assessment

No/unclear. The PR addresses a real UX complaint, but clearing by same tool/action type is not the narrowest maintainable proof of recovery; an explicit retry signal or a UI that preserves both attempts is safer.
