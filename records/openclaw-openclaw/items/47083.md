## Summary

Keep open. Current main still has the stale-token display problem in the active TUI path and in the Control UI session table formatter, so the contribution remains relevant; however, the submitted TUI guard drops fresh `null` resets and can leave stale footer state visible, so the PR is not merge-ready as submitted.

## What This Changes

The PR adds `totalTokensFresh` handling to TUI and Control UI session token display paths so stale token totals are hidden.

## Best Possible Solution

Land a narrow repair that carries `totalTokensFresh` through the active current-main TUI `TuiSessionList`/session-info path, clears or suppresses cached footer totals when freshness is false, allows fresh `totalTokens: null` to clear state, makes Control UI token formatting return n/a for stale rows, and adds focused regressions plus a user-facing changelog entry crediting the contributor.

## Reproduction Assessment

Yes. On current main, seed TUI state with a numeric `sessionInfo.totalTokens`, then apply a session row or patch marked `totalTokensFresh: false`; the TUI path has no freshness branch and the Control UI formatter still renders any non-null total. The PR-specific regression is reproducible from the diff: a fresh `totalTokens: null` update is ignored by the new non-null guard.

## Solution Assessment

No, not as submitted. The direction is correct, but the TUI logic must handle freshness before value/null checks and should clear stale cached state; the current PR also needs to be ported to the current `TuiSessionList` type location and covered by tests.
