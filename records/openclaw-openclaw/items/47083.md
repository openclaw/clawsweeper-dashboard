## Summary

Keep open: current main still has the stale-token display path, and this PR targets the right TUI/UI boundary, but the submitted TUI guard can preserve an old total when freshness is false or when a fresh null should clear it; the external PR also lacks real behavior proof.

## What This Changes

The PR adds `totalTokensFresh` plumbing to TUI and Control UI session rows and hides displayed `totalTokens` when freshness is false.

## Best Possible Solution

Rebase the targeted fix, make TUI `applySessionInfo` clear retained totals when freshness is false while allowing fresh null clears, keep the Control UI formatting guard, and add focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-level: current main documents `totalTokensFresh: false` as stale, the gateway emits that flag, and the TUI/UI formatting paths still ignore it in the places above. I did not run a live TUI or Control UI session.

## Solution Assessment

No, not as submitted. The fix boundary is right, but the TUI condition must handle freshness before retaining state so stale rows clear and fresh null resets are not dropped.
