## Summary

Keep this PR open: the TUI startup-summary direction is still useful and not implemented on current main, but the current patch can miss the active session and the supplied proof still exercises helper output rather than the actual TUI/Gateway startup path.

## What This Changes

The PR adds a first-connect TUI startup summary fetched from Gateway `sessions.list`, scopes it to the current agent, skips reconnects, and updates related TUI plus adjacent test code.

## Best Possible Solution

Target the current session directly in the Gateway lookup, add coverage for that path, and supply redacted proof from an actual TUI/Gateway startup showing the summary renders once and reconnect does not duplicate it.

## Reproduction Assessment

Yes from source inspection: an older current session outside the top ten same-agent sessions will be missed because the new lookup is unsearched and then filtered client-side by `currentSessionKey`. I did not run the interactive TUI/Gateway path in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the best fix is to fetch or search for the active session directly instead of scanning a recent-session window, then prove the real startup display path.
