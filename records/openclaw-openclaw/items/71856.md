## Summary

Keep open: this is a focused TUI startup-summary change, but the current branch can display a summary from an arbitrary agent session when the active TUI session is global or unknown because the new lookup excludes those keys and then falls back to the first returned agent row.

## What This Changes

The PR adds TUI startup-summary helpers/tests and a first-connect Gateway `sessions.list` lookup that renders a current-agent session preview while skipping local mode and reconnects.

## Best Possible Solution

Land a scoped TUI startup summary only after the selection logic is safe for global/unknown sessions, covered by focused tests, and still kept first-connect-only and best-effort.

## Reproduction Assessment

Yes, source-reproducible: use a TUI configuration where the active key is `global` or `unknown`; the PR's first-connect path sends an agent-scoped list request that excludes those keys and then falls back to `sessions[0]`. I did not run a live TUI because this was a read-only review.

## Solution Assessment

No, not yet. The current-agent scoping and reconnect gating are the right direction, but the fallback should not show a different agent session when the active session is not present in the scoped result.

- **[P2] Skip fallback summaries for global sessions:** `src/tui/tui.ts:1500`
  - body: When the active TUI session is `global` or `unknown`, `createStartupConversationSummaryListParams(currentAgentId)` excludes those keys and only returns agent-scoped rows.
