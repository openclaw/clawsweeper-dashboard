## Summary

## Best Possible Solution

Land a narrow TUI-only implementation that renders only an exact current-session summary or an explicitly tested prior-session summary, keeps web-search validation on the shared catalog, and includes current-head redacted TUI/Gateway proof.

## Reproduction Assessment

Yes for the patch defects by source inspection: the PR filters `listSessions` to current-agent/non-global/non-unknown rows and then falls back to `sessions[0]`, while current main supports global and unknown TUI session keys. I did not run a live TUI/Gateway repro in this read-only review.

## Solution Assessment

No. The maintainable path is to keep web-search validation on the existing shared catalog and make startup-summary selection exact or explicitly prior-session, with tests and current-head real behavior proof.
