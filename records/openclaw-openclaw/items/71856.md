## Summary

Keep this PR open: current main does not implement the requested TUI startup summary, and the branch still has a correctness issue where it skips the restored/current session and can summarize a different same-agent session. The supplied terminal proof covers helper gating and scoped params, so this needs a focused PR repair rather than closure.

## What This Changes

The PR adds a Gateway-backed TUI startup summary lookup scoped to the current agent and gated away from local mode and reconnects, with focused TUI helper tests and unrelated branch churn.

## Best Possible Solution

Keep the first-connect, current-agent-scoped lookup, but select the restored/current session row first with canonical session-key matching and focused tests, then reduce the branch to the intended TUI surface before maintainer UX review.

## Reproduction Assessment

Yes for the PR finding by source inspection: latest head restores and loads `currentSessionKey`, then the new summary code intentionally excludes that row while sessions.list is newest-first. It is not a current-main bug reproduction because current main does not have this startup-summary path.

## Solution Assessment

No: the scoped params and reconnect guard are good, but skipping the active session is not the narrowest solution for existing-session recovery. The safer path is to match `currentSessionKey` first and only use a prior-session fallback if maintainers explicitly want that UX.
