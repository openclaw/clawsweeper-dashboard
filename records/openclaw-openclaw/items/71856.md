## Summary

Keep open: this PR is useful source work, and the latest patch addresses the earlier undefined-helper review feedback, but the current head still runs the startup-summary insertion on every non-local reconnect and fetches/render session preview fields without scoping the request to the current agent. Current main and docs already model TUI sessions as current-agent scoped, so the unscoped preview read is both a functional and security-sensitive blocker before merge.

## What This Changes

The PR modifies `src/tui/tui.ts` to format a startup summary and fetch recent Gateway sessions on TUI connection, using the latest non-current session title or last-message preview.

## Best Possible Solution

Revise this PR so the startup summary is inserted only during the initial non-local connection, the `sessions.list` request is scoped to the current TUI agent, best-effort failure handling stays nonblocking, tests cover reconnect and cross-agent ordering, and the feature is recorded in `CHANGELOG.md` before merge.

## Reproduction Assessment

Yes for the PR regressions: simulate Gateway reconnect after the first connection to observe duplicate insertion, and mock `sessions.list` with a newer session from another agent to confirm the unscoped lookup can select and render that preview. I did not run tests because this was a read-only review, but the source path is direct.

## Solution Assessment

No in its current form. Fetching Gateway session preview data is a reasonable implementation direction, but the maintainable fix must use the existing `agentId` contract, avoid reconnect duplication, and add focused tests plus changelog coverage.
