## Summary

This PR still needs changes before merge: the latest head addresses the missing helper and reconnect guard, but the startup summary lookup excludes the restored/current session, so opening an existing session can summarize another recent same-agent session instead.

## What This Changes

The branch adds a Gateway-backed TUI startup summary lookup scoped to the current agent, skips local mode and reconnects, adds focused TUI tests, updates the changelog, and also carries unrelated formatting/import churn in plugin and test files.

## Best Possible Solution

Keep the current-agent-scoped, first-connect-only lookup, but prefer the restored/current session row with canonical session-key matching and only fall back to another prior same-agent session if maintainers intentionally want that UX.

## Reproduction Assessment

Yes for the PR finding by source inspection: the patched flow restores and loads currentSessionKey, then the new summary code intentionally excludes that row while sessions.list is newest-first. This is not a current-main bug reproduction because the startup-summary behavior is new in this PR.

## Solution Assessment

No: the scoped params and reconnect guard are good, but excluding the active restored session is not the narrowest solution for the linked existing-session recovery request. The safer path is to match currentSessionKey first and add a focused regression test for that case.
