## Summary

Keep open: the feature is not implemented on current main and the PR remains relevant to the linked startup-summary request, but the current patch is not safe to merge because it can append duplicate summaries on reconnect and can render another agent's recent session preview.

## What This Changes

The PR adds a TUI startup-summary formatter and a non-local Gateway `sessions.list` call after history load to display a previous session title or last-message preview as system lines.

## Best Possible Solution

Implement this as an initial-connect-only, current-agent-scoped startup summary with best-effort failure handling, focused tests for reconnect and multi-agent session stores, and an Unreleased changelog entry before merge.

## Reproduction Assessment

Yes. Trigger `client.onConnected` again after a disconnect to observe duplicate startup-summary lines, and mock `sessions.list` with a newer session from another agent to confirm the unscoped lookup can select and render it.

## Solution Assessment

No in the current PR form. Using Gateway session preview data is a reasonable direction, but the maintainable version must scope by the current agent, avoid reconnect duplication, and include tests and changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard startup summaries on reconnect:** `src/tui/tui.ts:1105`
  - body: This block sits in `client.onConnected`, which main uses for both initial connects and reconnects via the `reconnected` flag. Because the new condition is only `!isLocalMode`, every Gateway reconnect can fetch sessions and append another copy of the startup summary; gate this to the first connection or track a one-time insertion flag.
  - confidence: 0.93
- **[P2] Scope the session lookup to the current agent:** `src/tui/tui.ts:1106-1111`
  - body: The new `listSessions` request asks for derived titles and last-message previews but does not pass `agentId`. `sessions.list` only filters by agent when that option is present, so a multi-agent TUI can render another agent's newer title or message preview on startup; pass the current agent id and keep global/unknown excluded.
  - confidence: 0.9

  - body: This is a user-facing `feat` change, but the PR only modifies `src/tui/tui.ts`. Repo policy requires an Unreleased `CHANGELOG.md` entry with contributor credit for user-visible features before merge.
  - confidence: 0.82
