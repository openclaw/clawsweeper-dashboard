## Summary

Keep open: the PR is useful and has now supplied real Gateway output, but the SDK projection has a blocking replay bug because it trusts `deltaText` even when it has no prior cumulative snapshot for the run stream.

## What This Changes

The PR adds optional `deltaText` to Gateway chat delta events, carries its state through cleanup paths, updates SDK chat projection, and adds docs plus Gateway/protocol/SDK tests.

## Best Possible Solution

Keep the additive protocol field, but make the SDK preserve a full first replayed chat snapshot before landing the Gateway/protocol/SDK tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: emit enough chat projection deltas to trim earlier per-run replay entries, then start `run.events()` so the first retained projection has no previous text while its `deltaText` is only a suffix.

## Solution Assessment

No, not as currently patched. The Gateway-side additive field is maintainable, but the SDK should only prefer `deltaText` when it has a known previous snapshot or otherwise emit the cumulative `message` as a full replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve full first replayed chat snapshot:** `packages/sdk/src/client.ts:278`
  - body: When `run.events()` starts after earlier chat projections were trimmed from the per-run replay cache, `previousChatProjectionText` is undefined even though Gateway `deltaText` is only the suffix since the last broadcast. Returning that suffix here drops the already-streamed prefix; use `deltaText` only with a known previous snapshot, or mark the first seen projection as a full replacement.
