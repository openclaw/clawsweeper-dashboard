## Summary

Keep open: the PR still prunes the `SessionManager.fileEntries` array that Pi uses as the persistence backing store, so later rewrite paths can serialize the shortened live-memory view and drop older JSONL transcript history.

## What This Changes

The PR adds a bounded Pi SessionManager wrapper, routes CLI and embedded session-manager opens through it, adds memory-retention and port-diagnostic tests, and updates the changelog.

## Best Possible Solution

Keep the bounded live-memory approach, but decouple pruning from the persistence backing data or make rewrite paths preserve the full JSONL transcript, with a focused rewrite-after-prune regression test.

## Reproduction Assessment

Yes for the blocking review finding. Source inspection shows the PR prunes `fileEntries`, Pi `_rewriteFile()` serializes `fileEntries`, and wrapped CLI/embedded paths can still call `_rewriteFile()` after pruning.

## Solution Assessment

No. The wrapper direction is plausible, but it is not the best mergeable solution until live-memory pruning is separated from persisted-transcript rewrite semantics.

- **[P2] Preserve disk history when rewrites follow pruning:** `src/agents/pi-session-manager-memory-bound.ts:197`
