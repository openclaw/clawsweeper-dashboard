## Summary

Keep open: the memory-bounding direction is useful, but the patch is not merge-ready because pruned live `fileEntries` can later be written back to disk by existing `_rewriteFile()` paths, which can truncate the persisted transcript the PR promises to preserve. The current proof is also stale for the current head and Pi version, and unrelated delivery/UI changes still need review or splitting.

## What This Changes

The PR adds a bounded wrapper for Pi `SessionManager`, routes gateway and CLI SessionManager opens through it, and also changes source-delivery matching, macOS channel config rendering, port diagnostics, browser-origin tests, pnpm setup, and process-test behavior.

## Best Possible Solution

Keep the memory fix, but make durable transcript storage independent from the pruned live view, add regression coverage for prune plus rewrite/reopen, and split or revert unrelated delivery/UI/CI changes unless they receive their own proof.

## Reproduction Assessment

Yes, at source level: the wrapper prunes `fileEntries`, and Pi 0.75.4 `_rewriteFile()` serializes that same array while current OpenClaw runner paths still call `_rewriteFile()`. I did not run a live gateway soak in this read-only review.

## Solution Assessment

No. Bounding live memory is the right direction, but the current implementation must preserve full durable transcript state across rewrite/reopen paths and should not bundle unrelated delivery or macOS UI changes into the memory fix.
