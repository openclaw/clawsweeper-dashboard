The memory-bound wrapper can still truncate persisted transcripts through existing rewrite paths, the lockfile is out of sync with current manifests and fails frozen install, and the supplied real behavior proof is stale for the current head/dependency graph.

## What This Changes

The PR adds a bounded Pi SessionManager wrapper for CLI and embedded runner paths, plus related regression tests, port diagnostics, gateway origin hardening, macOS formatting, changelog text, and lockfile changes.

## Best Possible Solution

Land a focused memory-bound implementation only if it preserves full persisted transcripts across all rewrite/flush paths, aligns the lockfile with current manifests, and includes current-head proof against the actual installed dependency graph.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows pruning fileEntries before existing _rewriteFile calls can truncate persisted history, and CI reproduces the frozen-lockfile failure. The original long-running memory leak still needs current-head live or soak proof.

## Solution Assessment

No. The wrapper direction is plausible, but this implementation needs a full persisted-history buffer or rewrite-safe design, a regenerated lockfile, and fresh proof before it is the maintainable fix.
