## Summary

Keep open: the memory-retention fix is plausible and the supplied terminal proof is sufficient, but the current branch still regresses deprecated runtime config helper attribution and the new pnpm runtime dependency patch needs explicit maintainer approval.

## What This Changes

The PR patches `@earendil-works/pi-coding-agent@0.74.0` via pnpm to prune live SessionManager entries after compaction, adds regression coverage, updates port diagnostics/tests, and carries related test snapshot/import updates.

## Best Possible Solution

Keep the SessionManager memory bound only if maintainers approve this pnpm patch or an upstream release, while preserving existing plugin runtime config attribution for all helper methods.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or open a persisted Pi SessionManager on 0.74.0, append many entries through compaction, and observe `fileEntries`/`byId` retaining loaded and appended entries without pruning.

## Solution Assessment

No, not as submitted. The SessionManager pruning boundary is narrow, but the branch should preserve both current and deprecated config helper scoping and still receive explicit dependency-patch approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve deprecated config helper scoping:** `src/plugins/registry.ts:2402-2405`
  - body: This proxy now scopes only `current`/`mutateConfigFile`/`replaceConfigFile`, leaving `loadConfig` and `writeConfigFile` from the spread `config` object. Those deprecated helpers are still part of `PluginRuntime`, and their warnings rely on plugin scope for source attribution and per-plugin de-duplication, so external plugins would regress to anonymous warnings. Keep the old wrappers while adding wrappers for the new helpers and restore coverage for both paths.
  - confidence: 0.88
