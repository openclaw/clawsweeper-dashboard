## Summary

Keep open: the memory-retention fix is aimed at a real gateway/session problem, but the current patch can still turn bounded live memory into persisted transcript truncation, and the supplied proof is stale for the current head/runtime contract.

## What This Changes

The PR adds an OpenClaw wrapper around Pi `SessionManager`, routes gateway/CLI SessionManager opens through it, adds focused memory-retention and port diagnostic tests, and also changes source-delivery matching, browser-origin behavior, macOS channel config rendering, and the changelog.

## Best Possible Solution

Keep the memory fix direction, but separate bounded live context from full persisted rewrite state, add prune/rewrite/reopen regression coverage, and split or explicitly approve the unrelated delivery/UI/origin changes before merge.

## Reproduction Assessment

Yes, at source level: current main opens Pi `SessionManager` directly in active gateway/CLI paths, and upstream keeps session entries resident in `fileEntries`/`byId`. I did not run a multi-day gateway soak in this read-only review.

## Solution Assessment

No. The wrapper bounds live memory by mutating the same `fileEntries` array used for full-file rewrites; the safer path is to keep a full persistence source or make every rewrite reload/merge the full transcript before writing.
