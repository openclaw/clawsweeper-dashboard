## Summary

Keep open: this is a plausible narrow fix for the linked usage-cost cache ENOTSUP crash, but it still needs affected-environment real behavior proof, and the new comment overstates the `O_EXCL` network-filesystem contract.

## What This Changes

The PR replaces the session-cost usage cache lock's temp-file hard-link publish step with direct exclusive `writeFile(..., { flag: "wx" })` in `src/infra/session-cost-usage.ts`.

## Best Possible Solution

Land one narrow cluster fix that removes the hard-link-only failure mode, preserves busy/stale/release semantics, includes focused lock coverage, and has redacted proof from an affected filesystem class.

## Reproduction Assessment

Yes, as a source-backed path: current main calls `fs.promises.link` for the usage-cost lock and rethrows non-EEXIST errors, matching the linked ENOTSUP report. I did not run a live SMB/NFS/virtiofs-style mount in this read-only review.

## Solution Assessment

Unclear whether this PR is the best cluster path: replacing the hard-link step with exclusive create is the right direction, but this branch lacks affected-environment proof and tests, while https://github.com/openclaw/openclaw/pull/81388 appears to cover the same issue with a fallback plus regression coverage.
