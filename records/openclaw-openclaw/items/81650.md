## Summary

Keep open: this is a focused bug-fix PR for a source-proven ENOTSUP lock failure, and the code direction is reasonable, but the external PR still needs real after-fix proof on the affected filesystem class before merge; I also found one non-blocking comment accuracy issue.

## What This Changes

The PR replaces the usage-cost cache refresh lock's temp-file hard-link publish step with direct `writeFile(..., { flag: "wx" })` exclusive creation in `src/infra/session-cost-usage.ts`.

## Best Possible Solution

Land a narrow portable lock fix in the session-cost cache path with regression coverage, accurate filesystem-contract wording, and real proof from an affected filesystem or equivalent container/shared-mount setup.

## Reproduction Assessment

No live unsupported filesystem was mounted in this review, so I do not have a high-confidence runtime reproduction. The issue is source-reproducible: current main publishes the lock with `fs.promises.link` and rethrows non-EEXIST errors, matching the linked ENOTSUP report.

## Solution Assessment

Yes, replacing the hard-link publish dependency with exclusive create is the narrowest maintainable direction. The best version should add focused regression coverage, soften the network-filesystem guarantee, and include real after-fix proof from the affected environment class.

## Review Findings
