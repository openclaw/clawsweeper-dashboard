## Summary

Keep open: current main has public channel-turn primitives and projected claim hooks, but it does not expose the requested full-fidelity, loop-marked, trust-gated redispatch capability for a before_dispatch claim-then-rerun flow.

## What This Changes

Request a supported Plugin SDK/runtime API for trusted plugins to claim an inbound event, acknowledge it, and redispatch the original full-fidelity turn through the reply path with loop markers and reply-gate bypass semantics.

## Best Possible Solution

Add an additive, versioned SDK/runtime contract that reuses channel-turn or finalized-context primitives, exposes observable loop markers, and gates replay/bypass authority to explicitly trusted plugins with docs and contract tests.

## Reproduction Assessment

Yes for the API gap by source inspection: before_dispatch receives only a projected event/context, and runtime.channel.turn has no redispatch or marker-aware re-entry method. I did not run a live plugin reproducer because this is a missing SDK capability rather than broken documented behavior.

## Solution Assessment

Unclear: a redispatch method is plausible, but maintainers need to choose the safest contract shape for trust gating, loop markers, reply-gate bypass, and result reporting. A prepared-turn or plugin-callable agent invocation API may be safer than directly replaying the full dispatch pipeline.
