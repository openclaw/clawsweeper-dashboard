## Label Justifications

- `impact:security`: The requested redispatch authority would need explicit trusted-plugin gating and loop controls before exposure.

## Summary

Keep open: current main and v2026.6.11 still lack the requested trusted hook-level redispatch/takeover Plugin SDK contract; existing inbound helpers are channel receive-path helpers and do not provide loop markers, reply-gate control, dedupe disposition, or reply result reporting for `before_dispatch` claim-then-rerun.

## What This Changes

Request a trusted plugin-callable full-fidelity inbound redispatch capability for `before_dispatch` claim-then-rerun, including loop-safety, reply-gate bypass control, and result reporting.

## Best Possible Solution

Design a narrow trusted Plugin SDK contract for claim-then-rerun that defines dedupe disposition, loop-marker propagation, reply-gate behavior, result reporting, docs, and security tests before implementation.

## Reproduction Assessment

Yes, by source inspection: current main projects a limited `before_dispatch` payload and treats `{ handled: true }` as terminal after committing inbound dedupe. I did not run a live minimal plugin reproducer.

## Solution Assessment

Unclear as a final API shape: a trusted redispatch seam is plausible, but maintainers still need to choose the supported SDK surface and security model before implementation.
