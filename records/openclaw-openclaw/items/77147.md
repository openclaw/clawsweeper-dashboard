## Summary

Keep open: current main has a public channel turn kernel and projected claim hooks, but it does not expose the requested full-fidelity, loop-marked, trust-gated redispatch surface for a `before_dispatch` claim-then-rerun flow.

## What This Changes

Request a supported plugin-callable inbound redispatch API with full original event fidelity, loop-safe markers, reply-gate bypass semantics, trust gating, and a reply result surface.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or finalized inbound context rather than exposing internal dispatch functions, with tests and docs for markers, replay safety, and delivery policy.

## Reproduction Assessment

Yes for the API gap by source inspection: `before_dispatch` receives a projected event and the public `runtime.channel.turn` surface has no redispatch method. I did not run a live plugin reproducer because this is a new SDK capability request rather than a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around channel-turn reuse, trust gating, loop markers, and reply-gate bypass semantics. A narrower plugin-callable agent invocation or prepared-turn API may be safer than replaying the full dispatch pipeline directly.
