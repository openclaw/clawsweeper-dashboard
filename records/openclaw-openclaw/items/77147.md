## Summary

Keep open: current main and the latest release still lack a supported full-fidelity redispatch API for `before_dispatch` claim-then-rerun, and the remaining work is a new security-sensitive Plugin SDK/runtime contract rather than cleanup-close material.

## What This Changes

Request a supported, trust-gated Plugin SDK/runtime API for a `before_dispatch` hook to claim an inbound turn, acknowledge it, and re-enter the normal reply path with full inbound context and loop-safety.

## Best Possible Solution

Design an additive, trust-gated SDK turn invocation contract, or a safer prepared-turn/agent-invocation alternative, that preserves inbound context, exposes loop-safe metadata to hooks, and defines reply-gate bypass semantics.

## Reproduction Assessment

Yes for the API gap by source inspection: current main exposes turn helpers but no redispatch method, and `before_dispatch` receives projected fields before handled results short-circuit model dispatch. I did not run a live plugin because this is a new SDK capability request, not broken existing contract behavior.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, replay markers, and reply-gate bypass. A narrower prepared-turn or plugin-callable agent invocation API may be safer than direct full-pipeline replay.
