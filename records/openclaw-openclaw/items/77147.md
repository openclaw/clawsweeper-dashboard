## Summary

Keep open: current main exposes channel turn helpers and projected `before_dispatch` claiming, but it still lacks the requested full-fidelity, marker-aware, trust-gated redispatch capability. This is a new SDK/runtime feature with delivery-policy and trust-boundary implications, not a stale or already-implemented bug fix.

## What This Changes

Request to add a trusted plugin SDK/runtime API that can claim an inbound `before_dispatch` event, acknowledge it, then re-enter the normal reply path with full inbound context and loop markers.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract around the existing channel turn kernel or finalized inbound context, with tests and docs for loop markers, replay safety, and delivery-policy bypass semantics.

## Reproduction Assessment

Yes for the API gap by source inspection: current `before_dispatch` receives a projected event and `runtime.channel.turn` has no redispatch method. I did not run a live plugin reproducer because this is a new SDK capability request rather than a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the narrowest maintainable fix needs maintainer design around turn-kernel reuse, trust gating, markers, and reply-gate bypass. A prepared-turn or agent-invocation API may be safer than direct full-pipeline replay.

## Risks / Open Questions

- A redispatch API could create replay loops or let trusted plugins bypass reply-gate policy unless marker visibility and trust boundaries are explicit.
