## Label Justifications

- `impact:message-loss`: The current claim path can silence or dedupe a claimed inbound instead of allowing the plugin to re-enter the normal reply path.

## Summary

Keep open: current main and the latest release still lack a supported plugin-callable redispatch/replay contract for `before_dispatch` claim-then-rerun, and the remaining work is a security-sensitive SDK/runtime design rather than cleanup-close material.

## What This Changes

The issue requests a trusted plugin SDK API for full-fidelity inbound redispatch after a `before_dispatch` claim, including loop markers, reply-gate bypass semantics, and a result surface.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime turn invocation contract that preserves inbound context, exposes loop-safe metadata to hooks, and reuses channel turn/finalized-context machinery without exposing brittle internal dispatch functions.

## Reproduction Assessment

Yes for the API gap by source inspection: current main and `v2026.5.27` expose only terminal `before_dispatch` handling plus existing inbound/reply helpers, not a supported full-fidelity redispatch contract. I did not run a live plugin because the remaining ask is a new SDK capability, not a broken documented behavior.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but maintainers need to choose the trust boundary, replay marker semantics, and whether the safer surface is prepared-turn dispatch or a plugin-callable agent invocation endpoint.
