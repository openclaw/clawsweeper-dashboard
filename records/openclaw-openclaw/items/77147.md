## Summary

Keep open: current main still lacks the requested full-fidelity, loop-marked, trust-gated plugin redispatch surface, and the remaining work is an SDK/security design decision rather than conservative cleanup.

## What This Changes

Request a supported Plugin SDK capability that lets trusted plugins claim an inbound turn, acknowledge it, and re-enter the normal reply path with full original context and loop-safety metadata.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or finalized inbound context, with explicit loop markers, replay controls, docs, and regression coverage.

## Reproduction Assessment

Yes for the API gap by source inspection: current main exposes projected before_dispatch hooks and documented channel turn helpers, but no full-fidelity redispatch method. I did not run a live plugin because this is a new SDK capability request, not a broken existing contract.

## Solution Assessment

Unclear: redispatch is a plausible direction, but the best solution needs maintainer design around trust gating, loop markers, reply-gate bypass, and whether a prepared-turn or agent-invocation API is safer than full pipeline replay.

## Risks / Open Questions

- Redispatch plus reply-gate bypass expands trusted plugin authority over replay and delivery policy, so the trust boundary needs explicit maintainer design.
- A hook-level API built from the current projected before_dispatch fields would lose quote, media, reply, or channel context unless it reuses the channel turn/finalized-context contracts.
- The right public shape may be a prepared-turn or plugin-callable agent invocation surface rather than direct full pipeline replay.
