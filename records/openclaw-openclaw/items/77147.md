## Label Justifications

- `P2`: This is a normal-priority plugin SDK capability gap with real message-delivery impact but it is not a current core outage or urgent regression.
- `impact:security`: The requested API would grant trusted plugins authority to replay inbound turns and bypass reply-gate policy, which is a security boundary decision.
- `impact:message-loss`: Current `{ handled: true }` behavior can silence a claimed inbound and commit dedupe, preventing a plugin from safely re-running the same logical message.

## Summary

Keep open: current main and the latest release still lack a supported plugin-callable claim-then-rerun redispatch contract. Source confirms `before_dispatch` is a projected handled/silence hook that commits inbound dedupe, while `runtime.channel.inbound.*` is a channel-owned ingress surface rather than the requested trusted redispatch path.

## What This Changes

The issue requests a supported plugin SDK capability for trusted plugins to claim an inbound `before_dispatch` event, acknowledge quickly, and redispatch the original full-fidelity turn with loop-safe and reply-gate-aware semantics.

## Best Possible Solution

Design a narrow trusted plugin SDK redispatch or equivalent agent-invocation API that carries full inbound context and explicit loop/reply-gate semantics, with docs and regression/security tests for the `before_dispatch` claim-then-rerun path.

## Reproduction Assessment

Yes by source inspection: on current main, a `before_dispatch` handler returning `{ handled: true }` short-circuits dispatch and commits inbound dedupe, while the hook event lacks full inbound context or a loop marker. I did not run a live direct-address plugin, so the live channel behavior remains unproven here.

## Solution Assessment

Unclear as proposed: the need is valid, but the exact public API shape should be a maintainer design decision rather than exposing internal dispatch helpers directly. The safer direction is a small trusted SDK contract with explicit replay, loop, result, and reply-gate semantics.
