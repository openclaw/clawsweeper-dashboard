## Summary

Keep this open: current main still lacks a supported Plugin SDK redispatch contract with full inbound fidelity, loop markers, and reply-gate bypass semantics, and the linked takeover/dedupe report was closed as a duplicate of this canonical design thread.

## What This Changes

The issue requests a trusted Plugin SDK API for full-fidelity inbound redispatch after a `before_dispatch` plugin claims an event.

## Best Possible Solution

Keep this as the canonical SDK design thread and decide on one narrow trusted re-entry contract, or explicitly document the supported non-redispatch agent invocation alternative if maintainers reject redispatch.

## Reproduction Assessment

Yes, at source level: current main has no redispatch/takeover/marker fields on `before_dispatch`, and handled results commit inbound dedupe. I did not run a live plugin because the remaining work is a new SDK capability and security design, not a broken documented command path.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but maintainers still need to choose between a trusted redispatch API and a narrower plugin-callable agent/reply invocation path. A safe implementation must be one canonical SDK contract with trust gating and loop protection, not plugin imports of internal dispatch helpers.
