## Summary

Keep open: current main exposes projected `before_dispatch` claim hooks and channel turn helpers, but not the requested full-fidelity, loop-marked, trust-gated redispatch API. This is a new security-sensitive SDK capability and needs maintainer design rather than cleanup closure.

## What This Changes

Request to add a plugin-callable full-fidelity inbound redispatch capability for `before_dispatch` claim-then-rerun flows.

## Best Possible Solution

Design an additive trusted-plugin SDK/runtime contract that reuses the channel turn kernel or finalized inbound context, with explicit loop markers, reply-gate policy, tests, and docs.

## Reproduction Assessment

Yes for the API gap by source inspection: the public hook is projected and `runtime.channel.turn` has no redispatch method. A live plugin reproducer is not necessary to confirm this is a missing SDK capability rather than a broken existing contract.

## Solution Assessment

Unclear: redispatch is a plausible shape, but the best fix needs maintainer design around trust gating, loop markers, reply-gate bypass, and whether a prepared-turn or plugin-callable agent invocation API is safer.

## Risks / Open Questions

- Redispatch plus reply-gate bypass expands trusted plugin authority over replay and delivery policy, so loop markers, trust gating, and audit semantics need explicit design.
