## Summary

Keep open: current main and v2026.6.6 still lack a supported plugin-callable redispatch/replay contract for before_dispatch claim-then-rerun, and the remaining work is a security-sensitive Plugin SDK design decision rather than stale cleanup.

## What This Changes

Requests a trust-gated Plugin SDK capability that lets a before_dispatch plugin claim an inbound, acknowledge it, and rerun the original full-fidelity inbound through the normal reply path with loop-safety, reply-gate bypass, and result reporting.

## Best Possible Solution

Keep this as the canonical Plugin SDK design thread and define a narrow trusted redispatch or equivalent agent-reply invocation contract that reuses the existing channel inbound/reply kernel without exposing broad internals.

## Reproduction Assessment

Yes, by source inspection: current main has no redispatch/takeover result in before_dispatch, no runtime.channel redispatch method, and handled before_dispatch commits inbound dedupe. I did not run a live plugin because the central gap is a missing SDK capability rather than a failing documented command.

## Solution Assessment

Unclear: a redispatch API may be right, but maintainers should choose the narrow trust-gated contract and decide whether it should rerun the dispatch pipeline or expose a safer plugin-callable reply invocation path.
