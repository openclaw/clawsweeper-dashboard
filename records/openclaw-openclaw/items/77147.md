## Summary

Keep open: current main and the latest release still lack a supported full-fidelity redispatch API for `before_dispatch` claim-then-rerun, and the remaining work is a new security-sensitive Plugin SDK/runtime contract rather than cleanup-close material.

## What This Changes

The issue requests a trusted Plugin SDK capability that lets a plugin claim an inbound `before_dispatch` turn, acknowledge it, then redispatch the original full-fidelity event through the normal reply path with loop markers and reply-gate bypass semantics.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime turn invocation contract that preserves inbound context, exposes loop-safe metadata to hooks, and reuses the channel turn/finalized-context machinery without exposing brittle internal dispatch functions.

## Reproduction Assessment

Yes for the API gap by source inspection: current main exposes turn helpers but no redispatch method, and `before_dispatch` receives projected fields before handled results short-circuit model dispatch. I did not run a live plugin because this is a new SDK capability request, not a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, replay markers, and reply-gate bypass. A narrower prepared-turn or plugin-callable agent invocation API may be safer than direct full-pipeline replay.
