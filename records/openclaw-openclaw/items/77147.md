## Summary

Keep open: current main still lacks the requested trusted hook-level redispatch/takeover Plugin SDK contract; existing channel inbound helpers are for channel-owned receive paths and do not solve `before_dispatch` claim-then-rerun with loop markers, reply-gate control, and result reporting.

## What This Changes

The issue requests a trusted Plugin SDK capability for a `before_dispatch` hook to claim an inbound event, acknowledge it, and redispatch the full-fidelity original turn through the normal reply path with loop-safety and reply-gate control.

## Best Possible Solution

Design a narrow trusted Plugin SDK contract for claim-then-rerun that explicitly defines dedupe disposition, loop-marker propagation, reply-gate behavior, result reporting, docs, and security tests before implementation.

## Reproduction Assessment

Yes, by source inspection: current main projects a limited `before_dispatch` payload and treats `{ handled: true }` as terminal after committing inbound dedupe. I did not run a live minimal plugin reproducer.

## Solution Assessment

Unclear as a final API shape: a trusted redispatch seam is plausible, but maintainers still need to choose between hook return semantics, `runtime.channel.inbound` extension, or a separate agent/reply invocation contract.
