## Summary

Keep this open: current main still has terminal `before_dispatch` handled semantics, and the existing `runtime.channel.inbound.*` bridge is not a supported trusted claim-then-rerun API with markers, reply-gate bypass, and result reporting.

## What This Changes

The issue requests a trusted Plugin SDK capability for plugins to claim an inbound turn, acknowledge it, and redispatch the original full-fidelity event through the normal reply path.

## Best Possible Solution

Design a narrow trusted Plugin SDK/runtime contract for claim-then-rerun, including loop markers, dedupe semantics, reply-gate behavior, and result reporting, or document the maintainer-approved non-redispatch alternative.

## Reproduction Assessment

Yes by source inspection: `before_dispatch` returning handled commits inbound dedupe and exits before normal model dispatch. I did not run a live plugin reproducer, and the remaining ask is a new SDK capability rather than a documented behavior regression.

## Solution Assessment

Unclear as a final API shape; a trusted redispatch seam is plausible, but maintainers need to choose whether this belongs in `runtime.channel.inbound`, hook return semantics, or a separate agent/reply invocation contract.
