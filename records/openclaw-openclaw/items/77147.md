## Summary

Keep open: current main and the latest release still lack the requested supported full-fidelity redispatch API, and the remaining work is a new security-sensitive Plugin SDK capability rather than a cleanup-close candidate.

## What This Changes

Expose a trust-gated Plugin SDK capability that lets a `before_dispatch` hook claim an inbound turn, acknowledge it, and re-enter the normal reply path with full context and a loop marker.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime turn invocation contract that preserves full inbound context, exposes loop-safe metadata to hooks, and reuses the channel turn/finalized-context machinery instead of exposing internal dispatch functions.

## Reproduction Assessment

Yes for the API gap by source inspection: the public runtime turn API has no redispatch method, and `before_dispatch` receives only projected metadata before short-circuiting on `handled`. I did not run a live plugin because this is a new SDK capability request, not a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, replay markers, and reply-gate bypass. A narrower prepared-turn or plugin-callable agent invocation API may be safer than direct full-pipeline replay.
