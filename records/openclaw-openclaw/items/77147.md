## Summary

Keep open: current main still has public channel turn helpers and projected claim hooks, but it does not expose the requested full-fidelity, loop-marked, trust-gated redispatch API for a before-dispatch claim-then-rerun flow.

## What This Changes

The issue requests a supported plugin-callable full-fidelity inbound redispatch capability for before_dispatch claim-then-rerun flows.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or finalized inbound context, with documented and tested marker, replay-safety, and delivery-policy semantics.

## Reproduction Assessment

Yes for the API gap by source inspection: current main has projected before_dispatch handling and no public redispatch method on `runtime.channel.turn`. I did not run a live plugin because this is a new SDK capability request, not a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, loop markers, reply-gate bypass, and whether a prepared-turn or plugin-callable agent invocation API is safer than replaying the full dispatch pipeline.
