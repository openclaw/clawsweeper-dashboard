## Summary

Keep open: current main still lacks the requested supported plugin-callable redispatch/replay contract for `before_dispatch` claim-then-rerun, and the remaining work is a new SDK/runtime/security design rather than cleanup-close material.

## What This Changes

The issue requests a trust-gated Plugin SDK API that lets a `before_dispatch` hook claim an inbound turn, acknowledge it, then re-enter the normal agent reply path with original inbound context plus loop and reply-gate bypass metadata.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime turn invocation contract that preserves inbound context, exposes loop-safe metadata to hooks, and reuses channel turn/finalized-context machinery without exposing brittle internal dispatch functions.

## Reproduction Assessment

Yes for the API gap by source inspection: current main exposes turn and legacy reply helpers, but no supported `before_dispatch` redispatch contract with full inbound context, loop marker, reply-gate bypass, trust declaration, and synchronous result semantics. I did not run a live plugin because this is a new SDK capability request rather than broken existing behavior.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, replay markers, and reply-gate bypass. A narrower prepared-turn or plugin-callable agent invocation API may be safer than direct full-pipeline replay.
