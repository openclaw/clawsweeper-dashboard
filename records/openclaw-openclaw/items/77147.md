## Summary

Keep open: current main and v2026.6.10 still lack the requested trusted hook-level redispatch/takeover contract; existing channel inbound helpers cover channel-owned receive paths, not full-fidelity `before_dispatch` claim-then-rerun with loop markers, reply-gate bypass, and result reporting.

## What This Changes

The issue requests a supported Plugin SDK capability for trusted plugins to claim an inbound `before_dispatch` event, acknowledge it, and rerun the original full-fidelity turn through the normal reply path.

## Best Possible Solution

Design a narrow trusted Plugin SDK contract for claim-then-rerun that names dedupe disposition, loop-marker propagation, reply-gate behavior, result reporting, docs, and security tests before implementation.

## Reproduction Assessment

Yes by source inspection: current main projects a limited `before_dispatch` payload and treats `{ handled: true }` as terminal after committing inbound dedupe. I did not run a live minimal plugin reproducer.

## Solution Assessment

Unclear as a final API shape: a trusted redispatch seam is plausible, but maintainers still need to choose between hook return semantics, `runtime.channel.inbound` extension, or a separate agent/reply invocation contract.
