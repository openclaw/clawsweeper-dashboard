## Summary

Keep open: current main has channel turn and reply dispatch helpers, but it still lacks the requested supported, marker-aware redispatch surface for a `before_dispatch` claim-then-rerun flow.

## What This Changes

Request a supported Plugin SDK redispatch capability so a `before_dispatch` hook can claim, acknowledge, and re-enter the inbound reply pipeline with full original context.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or a prepared-turn/finalized-context path, with docs and tests for replay markers, delivery policy, and failure results.

## Reproduction Assessment

Yes for the API gap by source inspection: `before_dispatch` receives a projected event and the public turn runtime has no redispatch method. A live plugin reproducer is not necessary to prove the missing SDK capability.

## Solution Assessment

Unclear: redispatch is a plausible shape, but the best fix needs maintainer design around trust gating, loop markers, and reply-gate bypass semantics. A narrower prepared-turn or plugin-callable agent invocation contract may be safer than replaying the full dispatch pipeline directly.
