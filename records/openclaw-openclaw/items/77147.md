## Summary

Keep open: current main still lacks a supported plugin-callable full-fidelity redispatch path for `before_dispatch` claim-then-rerun. The remaining work is a new SDK/runtime capability with security and product-design implications, not conservative cleanup-close material.

## What This Changes

The issue requests a trusted Plugin SDK/runtime API that lets a plugin claim a `before_dispatch` turn, acknowledge it, then rerun the original inbound turn with full context and loop-safe bypass metadata.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime turn invocation contract that preserves inbound context, exposes loop-safe metadata to hooks, and reuses the channel turn/finalized-context machinery without exposing brittle internal dispatch functions.

## Reproduction Assessment

Yes for the API gap by source inspection: current main exposes turn helpers but no redispatch method, and `before_dispatch` receives projected fields before handled results short-circuit model dispatch. I did not run a live plugin because this is a new SDK capability request rather than broken existing behavior.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the best fix needs maintainer design around trust gating, replay markers, and reply-gate bypass. A narrower prepared-turn or plugin-callable agent invocation API may be safer than direct full-pipeline replay.
