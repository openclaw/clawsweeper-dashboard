## Summary

Keep open: current main exposes a public channel turn kernel and a projected `before_dispatch` claim hook, but it still lacks the requested full-fidelity, marker-aware, trust-gated redispatch API for claim-then-rerun flows.

## What This Changes

Request a plugin SDK/runtime capability for trusted plugins to claim a `before_dispatch` event, acknowledge it, then re-enter the inbound agent reply path with the original turn context and loop markers.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or finalized inbound context and documents replay markers, reply policy, and result semantics.

## Reproduction Assessment

Yes for the API gap by source inspection: `PluginRuntimeChannel.turn` has no redispatch method and `before_dispatch` receives only projected fields. I did not run a live plugin reproducer because this is a new SDK capability request rather than a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch shape is plausible, but the safest design needs maintainer decisions around trust, replay markers, and reply-gate semantics. A prepared-turn or plugin-callable agent invocation API may be safer than exposing full pipeline replay directly.

## Risks / Open Questions

- Redispatch plus reply-gate bypass would expand trusted plugin authority over replay and visible-delivery policy, so loop markers and trust gating need explicit design.
- A hook-level API that reconstructs turns from projected fields could drop quote, media, reply, or channel-specific context unless it reuses the existing finalized-context or turn-kernel contracts.
- The best product shape is still open: direct redispatch, prepared-turn invocation, or a plugin-callable agent reply API may each carry different safety and ownership tradeoffs.
