## Label Justifications

- `P2`: This is a meaningful Plugin SDK/channel dispatch capability gap with limited blast radius but real design and security implications.
- `impact:security`: A plugin-callable redispatch authority would need explicit trust gating and loop controls before exposure.
- `impact:message-loss`: The current handled path can silence a claimed inbound turn instead of letting the plugin deliver the promised substantive reply.

## Summary

Keep this open: current main still has terminal `before_dispatch` handled semantics, and the existing channel inbound runtime helpers do not provide the trusted claim-then-rerun API, loop marker, reply-gate bypass, or result surface requested here.

## What This Changes

Expose a trusted Plugin SDK/runtime capability that lets a `before_dispatch` plugin claim an inbound turn, acknowledge it, and rerun the full-fidelity reply path later.

## Best Possible Solution

Design a narrow trusted Plugin SDK contract for claim-then-rerun, including loop markers, dedupe disposition, reply-gate behavior, result reporting, and docs/tests for both current-main and released upgrade behavior.

## Reproduction Assessment

Yes, by source inspection: current main projects a limited `before_dispatch` payload and treats `{ handled: true }` as terminal after committing inbound dedupe. I did not run a live plugin reproducer, and the remaining request is a new supported SDK capability rather than a documented behavior regression.

## Solution Assessment

Unclear as a final API shape: a trusted redispatch seam is plausible, but maintainers still need to choose whether this belongs in hook return semantics, `runtime.channel.inbound`, or a separate agent/reply invocation contract.
