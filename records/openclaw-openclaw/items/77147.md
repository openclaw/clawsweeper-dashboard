- `impact:security`: The requested redispatch and reply-gate bypass authority changes plugin trust boundaries and needs explicit gating.

## Summary

Keep open. Current main and v2026.6.6 still lack a supported plugin-callable redispatch/takeover contract, and the existing `before_dispatch` handled path remains terminal because it commits inbound dedupe; the related takeover report was closed into this issue as the canonical design thread.

## What This Changes

Request to expose a trusted, full-fidelity plugin SDK redispatch capability for `before_dispatch` claim-then-rerun flows, including loop markers, reply-gate bypass semantics, and a result surface.

## Best Possible Solution

Design a narrow trusted-plugin redispatch or equivalent agent-invocation contract that preserves inbound fidelity, makes loop/dedupe behavior explicit, defines reply-gate semantics, and lands with SDK docs plus focused tests.

## Reproduction Assessment

Yes for the current limitation by source inspection: `before_dispatch` exposes only a projected hook contract, and `{ handled: true }` commits inbound dedupe and returns. I did not run a live plugin because the remaining ask is a new SDK capability and trust model rather than a broken documented behavior.

## Solution Assessment

Unclear. A redispatch surface is plausible, but maintainers should first choose between pipeline redispatch, an explicit agent-invocation API, or a takeover/defer contract, and align it with the trusted-plugin design.
