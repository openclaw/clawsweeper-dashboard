- `impact:security`: A redispatch authority would need explicit trusted-plugin gating and loop controls before exposure.
- `impact:message-loss`: The current handled path can silence a claimed inbound turn and commit dedupe before a plugin can rerun the substantive reply path.

## Summary

Keep this open: current main and the latest release still expose terminal `before_dispatch` handling and channel inbound helpers, but not the requested trusted full-fidelity redispatch/takeover contract with loop markers, reply-gate bypass semantics, and result reporting.

## What This Changes

Request a supported Plugin SDK/runtime capability for trusted plugins to claim an inbound `before_dispatch` event, acknowledge it, and redispatch the original full-fidelity turn through the normal reply path.

## Best Possible Solution

Design a narrow trusted Plugin SDK contract for claim-then-rerun, with explicit loop markers, dedupe disposition, reply-gate behavior, result reporting, docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection: current main projects a limited `before_dispatch` payload and treats `{ handled: true }` as terminal after committing inbound dedupe. I did not run a live plugin reproducer.

## Solution Assessment

Unclear as a final API shape: a trusted redispatch seam is plausible, but maintainers still need to choose whether this belongs in hook return semantics, `runtime.channel.inbound`, or a separate agent/reply invocation contract.
