## Summary

Keep open: current main still exposes projected claim hooks and channel-turn adapter helpers, but not the requested full-fidelity, loop-marked, trust-gated redispatch capability. This is a new security-sensitive SDK feature, not a cleanup close or narrow repair.

## What This Changes

Request a supported Plugin SDK method for trusted plugins to claim a `before_dispatch` turn, acknowledge it, then rerun the original full-fidelity inbound event through the normal reply path with loop markers and reply-gate bypass semantics.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses the channel turn kernel or finalized inbound context, with documented marker, replay, bypass, and audit semantics plus focused regression tests.

## Reproduction Assessment

Yes for the API gap by source inspection: the public runtime channel turn surface has no redispatch method, and `before_dispatch` receives only projected fields. I did not run a live plugin reproducer because this is a new SDK capability request rather than a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch API is plausible, but maintainer design should decide whether full pipeline replay, prepared-turn reuse, or a narrower agent invocation surface is the safest maintainable contract.
