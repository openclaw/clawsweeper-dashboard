## Summary

Keep open: current main still lacks the requested supported full-fidelity redispatch surface for `before_dispatch` claim-then-rerun flows, and the remaining design crosses SDK, delivery-policy, and trust-boundary decisions.

## What This Changes

The issue requests a supported plugin SDK redispatch capability for `before_dispatch` claim-then-rerun flows with full inbound fidelity, loop markers, reply-gate bypass, trust gating, and result reporting.

## Best Possible Solution

Design an additive, trust-gated SDK/runtime contract that reuses channel-turn or finalized inbound-context machinery, with tests and docs for fidelity, markers, replay safety, and delivery-policy bypass.

## Reproduction Assessment

Yes for the API gap by source inspection: the public turn API has no redispatch method, the `before_dispatch` hook receives only projected fields, and the handled path returns before model dispatch. I did not run a live plugin because this is a new SDK capability request rather than a broken existing contract.

## Solution Assessment

Unclear: the requested redispatch surface is plausible, but the narrowest maintainable solution needs maintainer design around trust gating, loop markers, and reply-gate semantics. A prepared-turn or plugin-callable agent invocation API may be safer than replaying the entire dispatch pipeline directly.

## Risks / Open Questions

- The requested reply-gate bypass and replay authority would expand plugin power over delivery policy and needs explicit trust-boundary design.
- A hook-level redispatch API built from the projected `before_dispatch` event could lose quote, media, reply, or channel context unless it reuses finalized inbound context or channel-turn contracts.
- Loop-marker semantics need to be observable enough for plugins to avoid recursion without exposing spoofable replay controls to untrusted plugins.
