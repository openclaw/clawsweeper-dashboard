## Summary

Keep open: current main has related commentary/tool-text leak hardening, but I did not find high-confidence evidence that the exact bare `exec{...}` Telegram leak is fixed or obsolete. The closest shipped work diagnoses unsupported tool-call-shaped assistant text and documents provider/model incompatibility, but it intentionally does not execute raw text as a tool call.

## What This Changes

The issue reports Telegram/shell-channel output exposing raw `exec{"command":"openclaw status"...}` text instead of running the status tool and returning the version.

## Best Possible Solution

Maintainers should decide whether bare tool-name-prefixed JSON such as `exec{...}` should be added to the unsupported-tool-text diagnostic and hidden from user-visible replies, while preserving the structured-tool-call-only execution boundary.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection indicates the exact `exec{...}` shape is not covered by the current diagnostic tests or documented structured-tool-call boundary, but the reporter's model/provider details are missing.

## Solution Assessment

Unclear. The safest current direction is not to promote arbitrary `exec{...}` text into execution; a maintainer should decide whether to add a narrow sanitizer/diagnostic extension or treat this as provider configuration support.
