## Summary

Keep open. Current main has adjacent hardening for heartbeat-token stripping, sanitized direct exec-followup fallback, Telegram topic approval routing, and provider-context inheritance, but the reported composition still appears live: session-backed approved exec completions re-enter the ordinary delivered agent path, and overflow recovery still retries after compaction without a durable silent/internal delivery intent.

## What This Changes

The issue requests that Telegram topic async exec approval followups preserve internal/no-op delivery intent across context-overflow compaction so `HEARTBEAT_OK` and orchestration text are not posted to users.

## Best Possible Solution

Add a bounded runtime/protocol seam that marks internal/no-op delivery intent for exec/system followup runs and preserves it through gateway agent requests, overflow-compaction retries, and Telegram topic delivery. The regression should prove internal text is suppressed while real approved command output still reaches the originating topic.

## Reproduction Assessment

Unclear. The issue includes detailed 2026.4.26 logs and current-main code still exposes the relevant path, but there are no deterministic steps and no exact regression for Telegram topic approval followup plus overflow compaction.

## Solution Assessment

No current patch is proposed, and current main is not enough to call this solved. The best fix is to carry structured delivery intent across the followup/compaction boundaries, not to rely on broader `HEARTBEAT_OK` text stripping.
