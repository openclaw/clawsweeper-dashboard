## Summary

Keep open. Current main has partial HEARTBEAT_OK stripping and newer heartbeat_respond hardening, but approved exec completions still enter a normal Gateway agent followup and overflow recovery still retries the original prompt/current-message metadata outside the narrower mid-turn precheck path, so the reported Telegram async-followup/context-overflow leak class remains credible.

## What This Changes

The issue asks OpenClaw to keep Telegram async exec completion and heartbeat followups internal across context-overflow compaction instead of delivering orchestration text as user-visible chat.

## Best Possible Solution

Add a typed internal/no-op delivery intent for async exec completion runs and make overflow-compaction retries continue the same turn idempotently instead of re-submitting the original channel prompt/currentMessageId.

## Reproduction Assessment

No, there is not a high-confidence deterministic runtime reproduction in the checkout. The redacted logs show the incident and current source still composes the reported path, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the narrow maintainable direction is a structured delivery-intent plus retry-state fix. Token filtering alone is not enough because the bug also involves async followup classification and context-overflow replays.

Do not make all exec completions internal-only, do not add a user config option, and do not change unrelated channel behavior. Add regression coverage for exec approval followup delivery intent, Gateway propagation, PI overflow retry state, and Telegram topic/thread routing.
