## Summary

Keep open. Current main has adjacent fixes for metadata-only async exec completion noise and Telegram approval-followup routing, but the reported composition remains unproven as fixed: approved exec followups still re-enter the normal `agent` RPC path, the RPC schema has no silent/internal delivery intent, and overflow recovery still retries after compaction.

## What This Changes

The issue asks OpenClaw to prevent Telegram topic async exec followup and heartbeat/internal orchestration text from reaching user-visible chat after context overflow and compaction retry.

## Best Possible Solution

Add a bounded structured delivery-intent path for internal/no-op followup runs across exec approval followup, gateway `agent`, embedded runner, and compaction retry, with Telegram topic coverage proving internal text is suppressed while real command results still deliver.

## Reproduction Assessment

Unclear. The issue has credible redacted incident logs and current source still exposes the relevant route, but there are no deterministic steps or exact regression test for Telegram topic approval followup plus overflow compaction.

## Solution Assessment

No current patch is proposed, and current main is not enough to call solved. The safest implementation direction is structured delivery intent, not broader `HEARTBEAT_OK` string filtering or making all async exec completions internal-only.
