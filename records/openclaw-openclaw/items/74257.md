## Summary

Keep open. Current main has partial HEARTBEAT_OK stripping and some compaction-retry hardening, but async exec completions still enter the ordinary Gateway agent delivery path and overflow recovery still retries raw prompts outside the narrower mid-turn precheck path, so the reported Telegram leak class remains credible and not fully superseded.

## What This Changes

Prevent Telegram async exec approval followups from leaking HEARTBEAT_OK/internal followup text during context-overflow compaction recovery.

## Best Possible Solution

Add a typed internal/no-op delivery intent for async exec completion runs and make overflow retries resume the in-flight turn without re-submitting channel inbound/current-message metadata, while preserving real exec result delivery to Telegram topics.

## Reproduction Assessment

No. The production Telegram overflow plus approval-expiry sequence is not deterministic from the checkout, but the issue body provides concrete redacted logs and current source still composes the vulnerable path, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is structured delivery intent plus retry-state preservation; token filtering alone is not enough because the failure also involves async followup classification and compaction replay.
