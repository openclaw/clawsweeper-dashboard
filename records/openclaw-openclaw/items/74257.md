## Summary

Keep open. Current main has adjacent protections for plain HEARTBEAT_OK and metadata-only exec events, but the reported Telegram topic plus approved async exec followup plus overflow-compaction retry path is still live and lacks an exact regression test or delivery-intent seam.

## What This Changes

The issue asks OpenClaw to keep internal heartbeat and async exec followup text out of user-visible Telegram topic replies when context-overflow compaction resumes an approved exec/tool flow.

## Best Possible Solution

Add a bounded runtime/protocol delivery-intent path for internal or no-op followup runs, then cover the Telegram topic approval-followup plus overflow-compaction retry case so internal orchestration text is suppressed while real approved-command results still reach the originating topic.

## Reproduction Assessment

Unclear. The report includes credible redacted production logs and current main still exposes the relevant surfaces, but there are no deterministic reproduction steps and no exact regression test for the Telegram topic approval followup plus overflow-compaction composition.

## Solution Assessment

No current patch is proposed, and current main is not enough to call this solved. The maintainable solution is to carry delivery intent as structured runtime state across exec followup and compaction boundaries, not to add broader HEARTBEAT_OK/string filtering.
