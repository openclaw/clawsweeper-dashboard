## Summary

Keep open. Current main has partial HEARTBEAT_OK suppression and newer heartbeat-tool hardening, but approved exec completions are still modeled as ordinary Gateway agent followups, and overflow recovery still retries the original prompt/current-message metadata except for the narrower mid-turn precheck path, so the reported Telegram async-followup/context-overflow leak class is still credible and not cleanly superseded.

## What This Changes

The issue asks OpenClaw to keep Telegram async exec/heartbeat followup text internal after context-overflow compaction while preserving in-flight turn state and approval routing.

## Best Possible Solution

Add a typed internal/no-op delivery intent for async exec completion runs and preserve retry state across Gateway dispatch, session resume, and overflow-compaction retry so internal orchestration text is consumed while real exec results still reach the originating Telegram topic.

## Reproduction Assessment

No, there is not a high-confidence deterministic runtime reproduction in the checkout. The redacted logs show the incident and current source still composes the reported path, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the narrow maintainable direction is a structured delivery-intent plus retry-state fix. Token filtering alone is not enough because the bug also involves async followup classification and context-overflow replays.
