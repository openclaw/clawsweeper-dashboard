## Summary

Keep open. Current main has useful adjacent protections for plain HEARTBEAT_OK suppression, Telegram approval callback errors, and metadata-only exec-event suppression, but the combined Telegram topic + approved async exec followup + context-overflow compaction retry path is still live and not covered by an exact regression. The related open reports #66126 and #73149 remain useful context, but neither fully replaces this Telegram approval-expiry instance.

## What This Changes

Prevent Telegram topic async exec approval followups and heartbeat/internal completion text from being delivered as ordinary visible messages after context-overflow compaction retries.

## Best Possible Solution

Add a bounded runtime/protocol seam that preserves internal or no-op delivery intent across approved exec followup runs and overflow-compaction retries, with Telegram-topic regression coverage proving internal text is suppressed while real approved-command output still reaches the originating topic.

## Reproduction Assessment

Unclear. The report has credible OpenClaw 2026.4.26 production logs and current code still exposes the relevant surfaces, but there are no deterministic reproduction steps and I found no current test for the exact Telegram topic approval followup plus overflow-compaction composition.

## Solution Assessment

No current patch is proposed, and current main is not enough to call this solved. The narrow maintainable repair is to carry delivery intent as runtime state across the gateway followup and compaction boundaries, not to rely only on final-message HEARTBEAT_OK filtering.
