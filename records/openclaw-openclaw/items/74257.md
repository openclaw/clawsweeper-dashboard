## Summary

Keep open. Current main still models approved async exec completions as normal deliverable Gateway agent followups, and the overflow retry path still rebuilds from the original prompt except for the narrower mid-turn precheck case, so the reported Telegram/context-overflow leak class is not implemented away or cleanly superseded.

## What This Changes

The issue asks OpenClaw to keep Telegram async exec completion and heartbeat-only followups internal during context-overflow recovery instead of delivering orchestration text to the user-visible chat.

## Best Possible Solution

Add an additive structured delivery-intent contract for approved async exec completion runs and preserve it through Gateway agent dispatch, session resume, and overflow-compaction retry so internal/no-op orchestration text is consumed while real exec results still route to the originating Telegram topic.

## Reproduction Assessment

Unclear. The redacted production logs show the incident and current source still composes the same path, but there is no deterministic end-to-end Telegram topic approval-overflow reproduction in the checkout.

## Solution Assessment

Yes for the requested direction. A typed runtime delivery-intent and compaction retry state fix is the narrow maintainable path; broader HEARTBEAT_OK filtering or dropping all exec completions would be brittle.
