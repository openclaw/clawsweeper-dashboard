## Summary

Keep open. Current main fixes the duplicate persisted Telegram message replay half of the report and strips some exact `HEARTBEAT_OK` followup payloads, but relayable exec heartbeat events can still force a stripped `HEARTBEAT_OK` back into delivery, and the broader system-event/compaction retry contract remains open rather than fully implemented.

## What This Changes

The issue asks OpenClaw to prevent Telegram topic async exec/heartbeat followups from leaking internal text during context-overflow recovery and to preserve in-flight turn state.

## Best Possible Solution

Land or replace the narrow no-op system-event suppression from https://github.com/openclaw/openclaw/pull/73785, then define a shared compaction-resume/delivery-intent contract for exec, heartbeat, cron, and channel followups before closing this Telegram-specific report.

## Reproduction Assessment

No, there is not a high-confidence deterministic end-to-end Telegram reproduction in the checkout. The redacted incident logs plus current source give a credible source-level path through exec/system-event no-op delivery and compaction retry state.

## Solution Assessment

Yes for the narrow direction, but not complete on current main. Token filtering alone is only a partial mitigation; the maintainable fix needs structured no-op/delivery intent and stable retry state for system-event followups.
