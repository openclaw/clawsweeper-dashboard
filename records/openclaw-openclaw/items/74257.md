## Summary

Keep open. Current main still routes approved async exec completions with an existing Telegram session through a normal Gateway `agent` followup, and the overflow-compaction retry path still rebuilds the attempt from the original prompt unless it is the narrower mid-turn precheck case.

## What This Changes

The issue reports that Telegram topic exec approval followups can leak internal heartbeat or async-completion text after context overflow and approval expiry.

## Best Possible Solution

Add an additive structured delivery-intent contract for approved async exec completion runs and preserve it across Gateway agent dispatch, session resume, and overflow-compaction retry so internal/no-op orchestration text is consumed while real exec results still route to the originating Telegram topic.

## Reproduction Assessment

Unclear. The issue has credible redacted production logs and current source/test seams still match the reported flow, but no deterministic local or live reproduction exists in the report or checkout.

## Solution Assessment

No. Current main’s string filters and direct fallback mitigations are useful, but the narrow maintainable fix is typed runtime delivery intent rather than broader `HEARTBEAT_OK` filtering or silently dropping all exec completions.
