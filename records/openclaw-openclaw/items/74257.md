## Summary

Keep open. Current main has adjacent protections for stray `HEARTBEAT_OK`, Telegram approval callback failures, topic-aware approval routing, and overflow compaction, but the exact Telegram topic + approved async exec followup + context-overflow retry path still resumes through a normal deliverable `agent` turn without a durable silent/internal delivery intent. The prior ClawSweeper comment remains directionally correct: related #66126 and #73149 are useful context, but they do not replace this Telegram approval-expiry instance.

## What This Changes

The issue reports that Telegram topic async exec followups can leak internal heartbeat/followup text into user-visible chat after context-overflow compaction and expired approval/resume handling.

## Best Possible Solution

Add a bounded runtime/protocol delivery-intent seam that marks internal/no-op followup obligations and carries that state through approved exec completion, gateway `agent` continuation, queued followup delivery, and overflow-compaction retry. The regression should prove internal text is suppressed for the Telegram topic incident while real approved exec output still reaches the originating topic.

## Reproduction Assessment

Unclear. The issue has credible production logs and current main still exposes the relevant code paths, but there are no deterministic reproduction steps and no exact regression for the combined Telegram topic approval followup plus overflow-compaction retry path.

## Solution Assessment

No current patch is proposed, and current main is not enough to call this solved. The best repair is to preserve delivery intent as structured runtime state across the followup and compaction boundaries, not to add broader text-based suppression.
