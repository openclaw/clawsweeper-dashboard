## Summary

Keep open. Current main has broader heartbeat-token cleanup and structured heartbeat work, but the reported Telegram async exec followup path still uses an ordinary Gateway agent continuation without a typed internal/silent delivery intent across the overflow-compaction retry boundary.

## What This Changes

The issue asks OpenClaw to keep internal async exec completion and heartbeat-only followups out of user-visible Telegram topic messages during context-overflow recovery.

## Best Possible Solution

Add a typed runtime delivery-intent contract for async exec completion followups, preserve it through Gateway agent RPC and overflow-compaction retry, and suppress only internal/no-op orchestration output while still delivering real command results to the originating Telegram topic.

Add a narrow, additive delivery-intent path for async exec completion followups and preserve it through `src/agents/bash-tools.exec-approval-followup.ts`, `src/gateway/protocol/schema/agent.ts`, `src/gateway/server-methods/agent.ts`, and the embedded runner/compaction retry path. Add focused tests for Telegram topic/session followups plus overflow-compaction retry, and update `CHANGELOG.md` because this is user-facing. If the Gateway protocol field is public, update the relevant Gateway protocol docs. Do not solve this by broad global string filtering alone.
