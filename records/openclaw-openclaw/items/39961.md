## Summary

Keep open. Current main still lacks a text-repetition streaming guard, and the PR addresses a real gap also tracked by #44965 and #63411. The patch is not mergeable yet because the abort lifecycle can still emit buffered repetitive text, the abort flag can persist across turns, strict config schemas reject the new config key, and suffix-cycle detection does not honor the configured range.

## What This Changes

The PR adds a core text-repetition detector, config types and resolver state, streaming-handler abort wiring, and unit tests for repeated text output detection.

## Best Possible Solution

Keep the PR open and request changes toward a robust core streaming circuit breaker: schema-backed configuration, documented/default behavior, abort-time buffer clearing or flush suppression, clean per-message state reset, suffix-cycle bounds that match config, and focused tests for chunked, non-chunked, message_end, and agent_end paths.

## Reproduction Assessment

Yes. The current gap is reproducible by source inspection of current main's text_delta/text_end path, and the PR defects are reproducible by tracing the abort-before-message_end lifecycle and strict config-schema paths on PR head.

## Solution Assessment

No, not yet. A core streaming repetition guard is a plausible solution, but this patch needs lifecycle, buffer, schema, and configured-threshold fixes before it is the narrowest maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Suppress fallback emissions after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When detection fires, this branch sets the abort flag and calls `session.abort()`, but it leaves the block buffers intact and only some `message_end` branches check the flag. If abort/error termination skips `message_end`, or if later queued assistant events arrive with full text, fallback flush/final emission can still surface the repetitive output the guard was meant to stop. Clear the buffers and make fallback emissions honor the abort state when setting it.
  - confidence: 0.89
- **[P2] Reset repetition-abort state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:184-188`
  - body: The PR caches the resolved guard config at `message_start`, but it does not clear `abortedByTextRepetitionGuard` there. If a guard-triggered abort terminates before `handleMessageEnd`, the flag can stick into a later assistant message and cause valid block output to be dropped by the guarded final-send paths.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:631`
  - body: This exposes `tools.textRepetitionGuard` in the exported TypeScript config types, but the strict runtime `AgentToolsSchema` and `ToolsSchema` still accept only `loopDetection`. Normal config validation will reject the new option before `resolveTextRepetitionGuardConfig` can read it, so add the matching zod schema/help/generated docs entries.
  - confidence: 0.94
- **[P2] Size suffix scanning from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix-cycle detector hardcodes `tail` to 600 characters even though `maxCyclePatternLength` is configurable. If an operator raises the configured max pattern length and relaxes other detectors, suffix loops in the advertised range can never be checked; size the tail from `maxCyclePatternLength * minCycleRepeats` or cap the config accordingly.
  - confidence: 0.82
