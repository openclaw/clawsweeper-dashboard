Full review comments:

- **[P1] Suppress every flush path after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:299-300`
  - body: When this branch sets `abortedByTextRepetitionGuard` and returns after `session.abort()`, abort/error termination may skip `message_end`; current `handleAgentEnd` still calls `flushBlockReplyBuffer`, which emits chunker or `state.blockBuffer` content. Clear or suppress buffered output as soon as the guard fires and carry the abort state through agent-end flushes, otherwise the repeated text can still be delivered.
  - confidence: 0.88
- **[P1] Reset the guard latch for the next assistant message:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:184-188`
  - body: `abortedByTextRepetitionGuard` is set to true when the guard fires, but the branch does not reliably clear that latch at message start, reset, or final cleanup. After one repetition abort, later valid messages can have their final/block replies suppressed; reset the latch and cursor at a reliable new-message or run boundary.
  - confidence: 0.86
- **[P1] Use a compatible rollout for stream aborts:** `src/agents/text-repetition-guard.ts:26`
  - body: `enabled: true` makes automatic stream aborts active for every existing agent after upgrade, unlike the established tool-loop detection default and without a halt/confirm UX. Make this opt-in, or land it only after an explicit maintainer decision plus upgrade proof for legitimate repetitive structured output.
  - confidence: 0.84
- **[P2] Register the config key in the strict schemas:** `src/config/types.tools.ts:331`
  - body: The PR exposes `tools.textRepetitionGuard` in TypeScript config types, but normal OpenClaw config validation uses strict Zod schemas that do not accept that key. Add a matching schema to both global and per-agent tools config, with parse tests, so users can actually disable or tune the guard.
  - confidence: 0.9
