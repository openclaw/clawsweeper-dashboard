## Summary

Keep open: current main still lacks a semantic streaming repetition guard, but this PR is not merge-ready because the abort path can still leak buffered output, guard state is not lifecycle-safe, config validation is incomplete, and real behavior proof is missing.

## What This Changes

The PR adds a configurable assistant text-repetition detector, wires it into embedded Pi streaming, adds config types/resolution, and adds focused detector tests.

## Best Possible Solution

Land a repaired stream-level circuit breaker that suppresses buffered output on abort, resets/latches guard state across lifecycle paths, registers schema/docs/tests, defines default/context policy, and includes redacted real provider or channel proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: text is buffered before abort, lifecycle flush/reset paths do not consistently honor the guard state, and strict schemas omit the new config key. The original provider-loop behavior still lacks after-fix real environment proof.

## Solution Assessment

No, not as currently patched. A stream-level guard is plausible, but the implementation needs lifecycle-safe suppression, schema coverage, maintainer-approved defaults/policy, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: The detector runs after the triggering chunk has already been appended to `deltaBuffer` and the block reply buffer. If `session.abort()` terminates before `message_end`, `flushBlockReplyBuffer` can still drain that buffered repetitive output because the fallback flush path does not check the abort flag.
  - confidence: 0.91
- **[P2] Latch off text handling after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: Returning from only the triggering update does not stop later queued `text_end` or `message_end` events. Those paths can still compute final stream data or emit block replies, so the abort flag needs an early suppression/clear path across update, end, and final flush handling.
  - confidence: 0.84
- **[P2] Reset repetition guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: `handleMessageStart` caches the config but neither it nor `resetAssistantMessageState` clears `textRepetitionLastCheckedLen` or `abortedByTextRepetitionGuard`. Abort/error paths can skip `message_end`, leaving the next assistant message with stale throttle or abort state.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in strict schemas:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises `tools.textRepetitionGuard` in public config types, but the strict runtime schemas still omit that key. Normal config validation will reject the option instead of applying the overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps the effective pattern length regardless of `maxCyclePatternLength`. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
