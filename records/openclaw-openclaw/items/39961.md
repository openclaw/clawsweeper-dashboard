## Summary

Keep open. Current main still lacks a streaming text-repetition guard, and this PR is unique relevant work, but the proposed patch is not merge-ready because the abort path can still flush buffered repetitive output and the new config key is type-only without runtime schema support.

## What This Changes

The PR adds a core text-repetition detector, config types and resolver state, streaming-handler abort wiring, and unit tests for repeated text output detection.

## Best Possible Solution

Keep this PR open and either harden it into a robust core streaming circuit breaker or make an explicit maintainer decision not to add one. A mergeable version should cover strict config schema/help/docs, abort-time buffer clearing, queued update handling, message_end and handleAgentEnd fallback flushes, state reset between turns, and focused tests for chunked and non-chunked replies.

## Reproduction Assessment

Yes. The current gap is reproducible by source inspection of the text_delta/text_end path on main, and the PR defects are reproducible through the abort-before-message_end lifecycle and strict config-schema paths described in the findings.

## Solution Assessment

No, not yet. A core streaming guard is a plausible direction, but this patch needs lifecycle and config integration fixes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Suppress fallback flush after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:297-298`
  - body: When detection fires, this branch sets `abortedByTextRepetitionGuard` and calls `session.abort()`, but buffered block content is left intact and only `handleMessageEnd` checks the flag. If abort/error termination skips `message_end`, `handleAgentEnd` still calls `flushBlockReplyBuffer()` and can emit the repetitive output the guard was meant to discard. Clear the buffers or make the fallback flush honor the abort flag when setting it.
  - confidence: 0.89
- **[P2] Reset the abort flag between assistant messages:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:575`
  - body: The cleanup added here resets the throttle cursor but never clears `abortedByTextRepetitionGuard`. After one guarded abort, later assistant messages keep the flag set, so the `message_end` block-reply branches guarded by `!ctx.state.abortedByTextRepetitionGuard` can drop valid future replies. Reset the flag at message start and in final cleanup.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in the Zod config schemas:** `src/config/types.tools.ts:627`
  - body: This line exposes `tools.textRepetitionGuard` in exported TypeScript types, but the strict runtime schemas still only accept `loopDetection`, so normal config validation rejects the new option before `resolveTextRepetitionGuardConfig` can read it. Add matching global and per-agent Zod schema entries, plus generated schema/help/labels/docs as needed.
  - confidence: 0.94
