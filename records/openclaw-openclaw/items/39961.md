## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks a stream-level text repetition guard, and this PR is useful source work, but the proposed patch still has blocking lifecycle and config-schema defects plus a maintainer policy decision around default-on core abort behavior.

## What This Changes

The PR adds a core text-repetition detector, per-agent/global config types and resolver wiring, streaming-handler abort integration, and unit tests for repetitive assistant output.

## Best Possible Solution

Land a maintained core streaming circuit breaker after rebase with schema-backed config, docs/changelog, abort-time buffer suppression, per-message state reset, configured detector bounds, focused lifecycle tests, and explicit maintainer-approved default behavior.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the current-main gap and the PR defects: main has no semantic text guard, while PR head sets the abort flag after buffering and leaves strict config schemas and terminal flush paths incomplete.

## Solution Assessment

No, not yet. A core streaming repetition guard is a plausible direction, but this patch needs lifecycle, schema/docs, configured-threshold, stale-branch, and default-policy fixes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Suppress terminal flush after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When detection fires, the chunk has already been appended to deltaBuffer and the block reply buffer. If abort terminates the run before message_end, handleAgentEnd still calls flushBlockReplyBuffer, which drains buffered repetitive content without checking the abort flag; clear or suppress those buffers when setting the abort state.
  - confidence: 0.9
- **[P2] Reset the repetition abort flag at message boundaries:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:543-546`
  - body: The PR sets abortedByTextRepetitionGuard and later uses it to suppress message_end block output, but the flag is never reset when a new assistant message starts. After one repetition abort, a later valid message can inherit the sticky flag and have its block reply dropped.
  - confidence: 0.91
- **[P2] Register textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:334-335`
  - body: This exposes tools.textRepetitionGuard in the exported TypeScript config types, but the strict AgentToolsSchema and ToolsSchema still do not accept that key. Normal config validation will reject the new option before the resolver can read it, so add the matching zod schema/help/generated docs entries.
  - confidence: 0.95
- **[P2] Size suffix scanning from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix-cycle detector hardcodes the inspected tail to 600 characters even though maxCyclePatternLength is configurable. If an operator raises the configured max pattern length and relaxes the other detectors, suffix loops in the advertised range are never checked.
  - confidence: 0.84
