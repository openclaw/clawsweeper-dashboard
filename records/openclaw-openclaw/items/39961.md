## Summary

Keep open. Current main still lacks a streamed text-repetition guard, but this PR cannot merge as-is because it lacks real behavior proof and has source-visible lifecycle/config defects around abort suppression, buffer flushing, state reset, and schema registration.

## What This Changes

The PR adds a configurable streaming text-repetition detector, wires it into assistant message updates, adds state/config types, and covers the detector with unit tests.

## Best Possible Solution

Land a repaired or replacement assistant-stream guard that suppresses buffered repeated output across text_delta, text_end, message_end, and agent_end paths, registers schema/docs for conservative config, and leaves broader halt/resume/context sanitization policy to the canonical issues.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch appends text before aborting, leaves later/fallback flush paths able to emit buffered text, and omits strict schema support. The original provider-loop behavior still lacks after-fix proof from a real OpenClaw run.

## Solution Assessment

No, not as currently patched. The assistant stream hook is a plausible direction, but it needs lifecycle-safe buffer clearing/suppression, state reset, schema/docs coverage, configured suffix sizing, a rebase, and maintainer-approved defaults before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-302`
  - body: The detector runs after this chunk has already been appended to blockBuffer or blockChunker. If abort terminates the run before message_end, handleAgentEnd can still flush those buffers through flushBlockReplyBuffer, which does not check abortedByTextRepetitionGuard, so repeated text can still be delivered.
  - confidence: 0.91
- **[P2] Suppress text handling after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: After abortedByTextRepetitionGuard is set, later queued text_delta/text_end/message_end events still enter the normal text and final emission paths. Add an early latch check that clears or suppresses text processing once the guard has fired.
  - confidence: 0.84
- **[P2] Reset guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: handleMessageStart resolves the new config but resetAssistantMessageState does not clear textRepetitionLastCheckedLen or abortedByTextRepetitionGuard. Abort/error paths can skip message_end cleanup, so the next assistant message may skip checks or discard valid output.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in strict schemas:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises tools.textRepetitionGuard in public TypeScript config, but AgentToolsSchema and ToolsSchema are strict and still omit that key. Normal config validation will reject the new option instead of applying overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps effective pattern length regardless of maxCyclePatternLength. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
