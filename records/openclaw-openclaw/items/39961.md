## Summary

Keep this PR open: current main still has no stream-level text repetition guard, and the PR is useful, but the proposed patch still has merge-blocking integration gaps around abort cleanup and config validation.

## What This Changes

The PR adds a configurable agent text-repetition detector, wires it into assistant streaming updates, and adds unit coverage for repeated text patterns.

## Best Possible Solution

Land a schema-backed streaming repetition guard after fixing abort cleanup, state reset, config docs/baselines, and regression tests; keep broader context sanitization/manual confirmation follow-up in the related open issues.

## Reproduction Assessment

Yes, from source inspection of the PR defects. An abort before message_end leaves block buffers available to handleAgentEnd flushing, and the new config key is absent from strict runtime schemas.

## Solution Assessment

No, not as currently patched. The detector idea fits the affected streaming surface, but the implementation needs lifecycle cleanup, state reset, and full config-contract support before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear buffered replies when the guard aborts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:297-298`
  - body: When repetition is detected this sets the abort flag and returns, but if `session.abort()` ends the run before a `message_end` event, `handleAgentEnd` still calls `flushBlockReplyBuffer()`, which drains `blockChunker` or `blockBuffer`. That can emit the repetitive text the guard was meant to suppress, so clear/reset the buffers here or make the lifecycle flush honor the abort flag.
  - confidence: 0.88
- **[P2] Reset guard state at each assistant message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:184-188`
  - body: The new guard state is initialized once and only `textRepetitionLastCheckedLen` is reset in the `message_end` cleanup path. Abort/error paths can skip `message_end`, leaving a stale checked length that delays later detection and a sticky abort flag that suppresses valid future block replies. Reset both fields at the assistant-message boundary, ideally inside `resetAssistantMessageState`.
  - confidence: 0.86
- **[P2] Accept textRepetitionGuard in the runtime schemas:** `src/config/types.tools.ts:331`
  - body: The PR exposes `tools.textRepetitionGuard` and agent-level overrides in TypeScript types, but the strict zod schemas still only accept `loopDetection`. Users who configure the advertised key will hit validation errors instead of applying thresholds, so the runtime schema, generated config schema/docs metadata, and schema help need to be updated together.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/agents/text-repetition-guard.ts:1`
  - body: This is a user-facing `feat` that changes agent streaming behavior and introduces new config, but the PR does not add a `CHANGELOG.md` entry. Add a single-line entry in the active Changes section before merge.
  - confidence: 0.82
