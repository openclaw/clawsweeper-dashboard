## Summary

Keep open. Current main still lacks a semantic assistant text repetition guard, and this PR is the open implementation candidate, but the patch has blocking source-level defects and no after-fix real behavior proof.

## What This Changes

The PR adds a TypeScript text-repetition guard, config types/resolver, assistant stream abort wiring, state fields, and unit tests for repeated assistant text loops.

## Best Possible Solution

Land a repaired stream breaker that is schema-backed, documented, conservative by maintainer policy, clears buffered output before any lifecycle flush, resets guard state at message boundaries, and tracks confirmation/context sanitization separately if not included.

## Reproduction Assessment

Yes for the PR defects via source inspection: the abort path runs after buffering, lifecycle flushing ignores the new abort flag, and strict config schemas omit the advertised key. The original live model-loop behavior still lacks after-fix real-run proof in this PR.

## Solution Assessment

No, not as currently patched. The assistant stream boundary is plausible, but this needs abort cleanup, state reset, schema/docs coverage, and maintainer-approved default policy before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear buffered replies when the guard aborts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-302`
  - body: When this branch aborts, the repetitive chunk has already been appended to blockBuffer or blockChunker. If the run ends before message_end, handleAgentEnd still calls flushBlockReplyBuffer, which drains those buffers without checking abortedByTextRepetitionGuard; clear/reset the buffers here or make every flush path honor the abort flag before returning.
  - confidence: 0.88
- **[P2] Reset repetition guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: handleMessageStart resets the normal assistant buffers but leaves textRepetitionLastCheckedLen and abortedByTextRepetitionGuard from the previous assistant message. Abort/error paths can skip message_end, so a stale cursor can delay checks and a sticky abort flag can drop a valid later reply.
  - confidence: 0.87
- **[P2] Accept textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:335`
  - body: The PR exposes tools.textRepetitionGuard and per-agent overrides in TypeScript only, but AgentToolsSchema and ToolsSchema still accept only loopDetection. Users configuring this advertised key through normal strict config validation will get an unknown-key error instead of applying the guard settings.
  - confidence: 0.9
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 chars, so raising maxCyclePatternLength above 600 / minCycleRepeats cannot work even though the config claims it can. Size the tail from the configured maximum cycle length and repeats so longer configured loops are actually checked.
  - confidence: 0.78
