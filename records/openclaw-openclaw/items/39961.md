## Summary

Keep open. Current main still lacks a text-repetition stream guard, but this PR is not merge-ready: the head is conflicting, source inspection confirms lifecycle bugs in the abort/buffer path, strict config schema support is missing, and the contributor has not posted real after-fix OpenClaw behavior proof.

## What This Changes

This PR adds a configurable assistant-stream text repetition detector, aborts the session when repetition is detected, wires config/state, and adds unit tests.

## Best Possible Solution

Land a repaired or replacement assistant-stream guard that clears or suppresses buffered repeated output across update, end, abort, and final-flush paths, registers strict schema/docs for conservative config, and leaves broader halt/resume/context-sanitization policy to the canonical issues.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch buffers text before aborting, fallback flush paths ignore the abort flag, guard state can survive abort/error paths, and strict schemas omit the new config key. The original provider-loop behavior still lacks real after-fix proof.

## Solution Assessment

No, not as currently patched. A stream-level guard is plausible, but this implementation needs lifecycle-safe buffer suppression, state reset, strict schema coverage, suffix sizing fixes, a rebase, real behavior proof, and maintainer-approved defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-302`
  - body: The detector runs after the triggering chunk has already been appended to blockBuffer or blockChunker. If abort terminates the run before message_end, handleAgentEnd can still flush those buffers through flushBlockReplyBuffer, which does not check abortedByTextRepetitionGuard, so repeated text can still be delivered.
  - confidence: 0.91
- **[P2] Suppress text handling after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: After abortedByTextRepetitionGuard is set, later queued text_delta, text_end, and message_end events still enter normal text and final emission paths. Add an early latch check that clears or suppresses text processing once the guard has fired.
  - confidence: 0.84
- **[P2] Reset guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: handleMessageStart resolves the new config, but resetAssistantMessageState does not clear textRepetitionLastCheckedLen or abortedByTextRepetitionGuard. Abort/error paths can skip message_end cleanup, so the next assistant message may skip checks or discard valid output.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in strict schemas:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises tools.textRepetitionGuard in public TypeScript config, but AgentToolsSchema and ToolsSchema are strict and still omit that key. Normal config validation will reject the new option instead of applying overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps effective pattern length regardless of maxCyclePatternLength. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
