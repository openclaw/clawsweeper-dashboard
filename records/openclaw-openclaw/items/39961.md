## Summary

Keep open. Current main still lacks a semantic streamed-text repetition guard, and this PR remains a useful implementation candidate, but the branch still has source-proven correctness blockers, no after-fix real OpenClaw proof, and a default-on abort policy that needs maintainer acceptance.

## What This Changes

The PR adds a text-repetition detector, global/agent config types and resolver, assistant stream abort wiring, state fields, and unit tests.

## Best Possible Solution

Land a repaired or replacement schema-backed stream guard that clears buffered repeated output before any flush, resets state at message boundaries, validates and documents conservative config/defaults, and handles confirmation or context sanitization through explicit follow-up design.

## Reproduction Assessment

Yes for the PR defects by source inspection: the abort branch runs after buffering, fallback flushing ignores the new abort flag, start-boundary state is not reset, and strict runtime schemas omit the advertised key. The original provider-loop behavior still lacks after-fix real-run proof from this contributor setup.

## Solution Assessment

No, not as currently patched. The assistant stream hook is plausible, but the branch needs abort cleanup across lifecycle flushes, state reset at message start, schema/docs coverage, configured suffix sizing, maintainer-approved default policy, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies when the guard aborts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-302`
  - body: The detector runs after this chunk has already been appended to blockBuffer or blockChunker. If abort ends the run before message_end, handleAgentEnd can still flush those buffers through flushBlockReplyBuffer, which does not check abortedByTextRepetitionGuard, so the repeated text can still be delivered.
  - confidence: 0.91
- **[P2] Ignore later chunks after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: After abortedByTextRepetitionGuard is set, later queued text_delta/text_end events still enter this handler and can append or emit more text before the session abort is fully honored. Add an early latch check that clears/suppresses text processing once the guard has fired.
  - confidence: 0.82
- **[P2] Reset guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: handleMessageStart resolves the new config but leaves textRepetitionLastCheckedLen and abortedByTextRepetitionGuard from the previous turn. Abort/error paths can skip message_end, so a later assistant message may skip checks or discard valid block output.
  - confidence: 0.9
- **[P2] Accept the new config key in runtime schemas:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises tools.textRepetitionGuard in public TypeScript config, but AgentToolsSchema and ToolsSchema still only accept loopDetection. Strict config validation will reject the new key instead of applying guard overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps the effective pattern length regardless of maxCyclePatternLength. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
