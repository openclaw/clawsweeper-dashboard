## What This Changes

The PR adds a core streaming text-repetition detector, per-agent/global config types and resolver state, message-handler abort wiring, and unit tests for repeated assistant text output.

## Best Possible Solution

Land a maintained core stream-level circuit breaker only after schema-backed config, docs/changelog, abort-time buffer suppression, per-message state reset, configured suffix bounds, focused lifecycle tests, and maintainer-approved default behavior are in place.

## Solution Assessment

No, not yet. A core streaming repetition guard is a plausible direction, but this patch needs lifecycle, schema/docs/changelog, configured-threshold, rebase, and default-policy work before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Suppress buffered flush when repetition abort fires:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When this branch detects repetition, the triggering chunk has already been appended to deltaBuffer and the block buffer. If abort ends the run before message_end, handleAgentEnd still calls flushBlockReplyBuffer, which drains the buffered repetitive output without checking the abort flag; clear or suppress the buffers when setting the abort state.
  - confidence: 0.91
- **[P2] Reset repetition guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:186-188`
  - body: abortedByTextRepetitionGuard and textRepetitionLastCheckedLen are only initialized/finalized, so an abort path that skips message_end can leave the next assistant message with a sticky abort flag or stale cursor. Reset both at the same start-of-message boundary as the other assistant state.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:333-335`
  - body: This adds tools.textRepetitionGuard to the exported TypeScript config surface, but the strict AgentToolsSchema and ToolsSchema still only accept loopDetection. Normal config validation will reject the new option before the resolver can read it, so add matching zod schema, help, and generated-doc coverage.
  - confidence: 0.94
- **[P2] Size suffix scanning from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector hardcodes a 600-character tail even though maxCyclePatternLength is configurable. If an operator raises the configured maximum and relaxes other detectors, suffix loops in the advertised range are never examined; derive the tail from maxCyclePatternLength * minCycleRepeats or cap the config accordingly.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/text-repetition-guard.ts:27-28`
  - body: This is a user-facing feature that changes streaming behavior and exposes a new config key, but the branch does not update CHANGELOG.md. Repo policy requires a changelog entry for user-facing feat, fix, and perf changes before merge.
  - confidence: 0.87
