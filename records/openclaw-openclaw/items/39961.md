## Summary

Keep open. Current main still lacks a text-repetition guard and the PR targets a real gap, but the patch is not merge-ready because lifecycle flush paths can leak repeated output, strict config schemas reject the advertised key, suffix scanning ignores configured bounds, and the external PR has no real behavior proof.

## What This Changes

The PR adds a configurable text-repetition detector, wires it into assistant streaming aborts, adds state fields/resolver/types, and introduces focused unit tests.

## Best Possible Solution

Land a repaired or replacement stream guard that suppresses buffered repeated output across all lifecycle flushes, resets state at message boundaries, validates and documents conservative config, and leaves confirmation/context sanitization to the canonical follow-up design.

## Reproduction Assessment

Yes for the PR defects by source inspection: the abort branch runs after buffering, fallback flushing ignores the new abort flag, start-boundary state is not reset, and strict schemas omit the new key. The original provider-loop behavior still lacks after-fix proof from a real OpenClaw run.

## Solution Assessment

No, not as currently patched. The assistant stream hook is plausible, but the branch needs lifecycle-safe buffer clearing, start-boundary reset, schema/docs coverage, configured suffix sizing, and maintainer-approved default/UX policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-302`
  - body: The detector runs after this chunk has already been appended to blockBuffer or blockChunker. If abort ends the run before message_end, handleAgentEnd can still flush those buffers through flushBlockReplyBuffer, which does not check abortedByTextRepetitionGuard, so the repeated text can still be delivered.
  - confidence: 0.91
- **[P2] Suppress later text events after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: After abortedByTextRepetitionGuard is set, later queued text_delta/text_end events still enter this handler and can append or emit more text before the session abort is fully honored. Add an early latch check that clears or suppresses text processing once the guard has fired.
  - confidence: 0.82
- **[P2] Reset repetition guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: handleMessageStart resolves the new config but leaves textRepetitionLastCheckedLen and abortedByTextRepetitionGuard from the previous turn. Abort/error paths can skip message_end, so a later assistant message may skip checks or discard valid block output.
  - confidence: 0.9
- **[P2] Add runtime schema support for textRepetitionGuard:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises tools.textRepetitionGuard in public TypeScript config, but AgentToolsSchema and ToolsSchema are strict and still only accept loopDetection. Normal config validation will reject the new key instead of applying guard overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps the effective pattern length regardless of maxCyclePatternLength. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
