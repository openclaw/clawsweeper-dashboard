## What This Changes

The PR adds a default-enabled configurable text repetition detector to embedded assistant streaming, wires it into message updates, adds config types/resolution, and adds detector unit tests.

## Best Possible Solution

Land a maintainer-approved stream circuit breaker that suppresses every buffered flush path after abort, resets guard state on every reliable boundary, has schema/docs/tests, and either preserves compatibility with an opt-in default or implements the agreed halt-and-confirm UX.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Suppress buffered flushes when the guard aborts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When repetition is detected, this branch sets the abort flag and returns, but it leaves the current block buffer/chunker intact. If `session.abort()` terminates the run before a `message_end` event, `handleAgentEnd` still flushes the buffered block reply, so the repetitive output the guard is meant to stop can still be delivered; clear the buffer or make `flushBlockReplyBuffer` honor the guard immediately when aborting.
  - confidence: 0.9
- **[P1] Reset guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-188`
  - body: The guard cursor and abort flag are only cleaned up on `message_end`, but abort/error paths can skip that event. Because `handleMessageStart` is already the reliable new-message boundary here, leaving these fields stale can skip checks or drop the next valid block reply after a repetition abort.
  - confidence: 0.87
- **[P1] Keep the stream abort guard opt-in until policy is settled:** `src/agents/text-repetition-guard.ts:28`
  - body: This default enables automatic stream aborts for every existing agent with no config change. That is a compatibility-sensitive behavior change from the existing tool-loop detector's opt-in default and can cut off legitimate repetitive structured output, so it needs maintainer-approved policy or an opt-in rollout before merge.
  - confidence: 0.86
- **[P2] Add strict schema support for the new config key:** `src/config/types.tools.ts:335`
  - body: The PR exposes `tools.textRepetitionGuard` in TypeScript types, but the strict Zod `AgentToolsSchema` and `ToolsSchema` on main do not accept that key. Users who try to configure the advertised thresholds through normal config validation will get an unknown-key failure instead of an applied override.
  - confidence: 0.9
- **[P2] Size the suffix scan from the configured range:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix-cycle detector hardcodes the inspected tail to 600 characters, so configs with a larger `maxCyclePatternLength` can never actually check their requested cycle length range. Derive the tail size from `maxCyclePatternLength * minCycleRepeats` or clamp the accepted config so the runtime behavior matches the advertised setting.
  - confidence: 0.78
