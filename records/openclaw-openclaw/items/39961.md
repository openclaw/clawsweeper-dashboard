## Summary

Keep open: current main still lacks a streaming text-repetition guard, and this PR is useful but not mergeable until abort cleanup, state reset, runtime schema support, configured suffix bounds, and changelog/docs coverage are fixed.

## What This Changes

The PR adds a configurable agent text-repetition guard, wires it into assistant streaming updates, adds config types/resolution state, and adds unit tests for repeated text patterns.

## Best Possible Solution

Land a schema-backed streaming repetition guard that clears buffered output on abort, resets state at assistant-message boundaries, honors advertised config, updates docs/config baselines/changelog, and leaves broader confirmation/context-sanitization behavior to the related issues.

## Reproduction Assessment

Yes, for the PR defects from source inspection: an abort can skip message_end while handleAgentEnd still flushes block buffers, and tools.textRepetitionGuard is exposed in types but rejected by strict runtime schemas.

## Solution Assessment

No. The detector belongs in the streaming surface, but this patch needs lifecycle-level buffer suppression, start-of-message state reset, full config schema/docs support, and configured suffix bounds before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Clear buffered replies when the guard aborts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When repetition is detected, this sets the abort flag and returns, but abort can end the run before message_end. handleAgentEnd still calls flushBlockReplyBuffer(), which drains blockChunker/blockBuffer without checking this flag, so the repetitive text can still be delivered; clear/reset the buffers here or make lifecycle flushing honor the abort.
  - confidence: 0.88
- **[P2] Reset guard state at each assistant message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: The new cursor and abort flag are initialized once and only the cursor is reset in message_end cleanup. Abort/error paths can skip message_end, leaving a stale checked length and sticky abort flag for the next assistant message; reset both fields at the assistant-message boundary, ideally in resetAssistantMessageState.
  - confidence: 0.86
- **[P2] Accept textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:334`
  - body: The PR exposes tools.textRepetitionGuard and agent-level overrides in TypeScript types, but the strict zod schemas still only accept loopDetection. Users who configure the advertised key will get validation errors instead of applying thresholds, so schema, generated config metadata, and help/docs need to move together.
  - confidence: 0.9
- **[P2] Size suffix scanning from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices only the last 600 characters, so configs that raise maxCyclePatternLength or lower minCycleRepeats cannot actually scan the advertised range. Size the tail from maxCyclePatternLength * minCycleRepeats, bounded by the configured window, so customized long-cycle detection works.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/text-repetition-guard.ts:1`
  - body: This is a user-facing feature that changes agent streaming behavior and introduces new config, but the PR does not update CHANGELOG.md. Add a single-line entry in the active Changes section before merge.
  - confidence: 0.82
