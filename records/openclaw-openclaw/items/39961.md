## Summary

Keep open: current main still lacks a semantic streaming text-loop guard, and this PR remains a plausible implementation candidate, but the current patch is not mergeable because abort lifecycle and config-schema defects remain while the default-on core abort policy needs maintainer review.

## What This Changes

The PR adds a core text-repetition detector, config types and resolver state, streaming-handler abort wiring, and unit tests for repeated text output detection.

## Best Possible Solution

Land a maintained core streaming circuit breaker only after it has schema-backed configuration and docs, abort-time buffer suppression, clean per-message state reset, configured suffix bounds, focused streaming lifecycle tests, and maintainer-approved default behavior.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the current-main gap and the PR defects: current main has no text guard, while PR head sets the abort flag after buffering and leaves strict config schemas and terminal flush paths incomplete.

## Solution Assessment

No, not yet. A core streaming repetition guard is a plausible solution, but this patch needs lifecycle, buffer, schema, configured-threshold, and default-policy fixes before it is the narrowest maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Suppress fallback emissions after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: When detection fires, this branch sets the abort flag and aborts the session, but it leaves the already-appended block buffers intact. If termination skips `message_end`, or later terminal cleanup runs, the existing `flushBlockReplyBuffer` path can still emit the repetitive content the guard was meant to stop; clear or suppress those buffers when setting the abort state.
  - confidence: 0.89
- **[P2] Reset abort state when each message starts:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:186-188`
  - body: `abortedByTextRepetitionGuard` is initialized once but not cleared at the next `message_start`. If an abort ends the stream before `handleMessageEnd`, the sticky flag can make a later valid assistant message drop its block output, so reset this state at the same boundary as the other assistant-message state.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in runtime schemas:** `src/config/types.tools.ts:335`
  - body: This exposes `tools.textRepetitionGuard` in the exported TypeScript config types, but the strict runtime `AgentToolsSchema` and `ToolsSchema` still accept only `loopDetection` in this area. Normal config validation will reject the new option before the resolver can read it, so add the matching zod schema/help/generated docs entries.
  - confidence: 0.94
- **[P2] Size suffix scanning from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix-cycle detector hardcodes `tail` to 600 characters even though `maxCyclePatternLength` is configurable. If an operator raises the configured max pattern length and relaxes other detectors, suffix loops in the advertised range are never checked; size the tail from `maxCyclePatternLength * minCycleRepeats` or cap the config accordingly.
  - confidence: 0.82
