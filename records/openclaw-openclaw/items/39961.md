## Summary

Keep open. Current main still lacks a text-output repetition guard and the related core issues remain open, but this branch is conflicting and has blocking lifecycle/schema defects plus missing real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds a configurable assistant-stream text repetition detector, aborts the session when repetition is detected, wires config/state, and adds unit tests.

## Best Possible Solution

Repair or replace the PR with a conservative schema-backed assistant-stream guard that clears buffers on abort, suppresses later flush paths, handles replay/context policy, and is proven in a real OpenClaw run.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch buffers text before aborting, fallback flush paths ignore the abort flag, guard state can survive abort/error paths, and strict schemas omit the new config key. The original provider-loop behavior still lacks real after-fix proof.

## Solution Assessment

No, not as currently patched. A stream-level guard is plausible, but this implementation needs lifecycle-safe buffer suppression, state reset, strict schema coverage, suffix sizing fixes, a rebase, real behavior proof, and maintainer-approved defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:300-301`
  - body: The detector runs after the triggering chunk has already been appended to `deltaBuffer` and the block reply buffer. If `session.abort()` ends the run before `message_end`, `handleAgentEnd` still calls `flushBlockReplyBuffer`, which drains the buffered repetitive output because that path does not check `abortedByTextRepetitionGuard`.
  - confidence: 0.91
- **[P2] Latch off text handling after repetition abort:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:292-302`
  - body: Returning only from the triggering update does not stop later queued `text_end` or `message_end` events. Those paths can still compute final stream data or emit block replies, so the abort flag needs an early suppression/clear path across update, end, and final flush handling.
  - confidence: 0.84
- **[P2] Reset repetition guard state at message start:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:187-191`
  - body: `handleMessageStart` caches the new config but does not clear `textRepetitionLastCheckedLen` or `abortedByTextRepetitionGuard`. Abort/error paths can skip `message_end` cleanup, leaving the next assistant message with stale throttle or abort state.
  - confidence: 0.9
- **[P2] Register textRepetitionGuard in strict schemas:** `src/config/types.tools.ts:334-335`
  - body: The PR advertises `tools.textRepetitionGuard` in public config types, but the strict `AgentToolsSchema` and `ToolsSchema` still omit that key. Normal config validation will reject the option instead of applying overrides.
  - confidence: 0.92
- **[P2] Size suffix scans from configured cycle limits:** `src/agents/text-repetition-guard.ts:229-230`
  - body: The suffix detector always slices the last 600 characters, which caps the effective pattern length regardless of `maxCyclePatternLength`. Configs that intentionally allow longer repeated blocks can miss the loops they asked the guard to detect.
  - confidence: 0.78
