Full review comments:

- **[P1] Keep the new abort guard out of existing defaults:** `src/agents/text-repetition-guard.ts:28`
  - body: `enabled: true` turns this new stream abort heuristic on for every existing agent as soon as the PR lands. Current tool-loop detection defaults off, and this branch has no maintainer-approved default policy or upgrade proof showing that legitimate repetitive structured output will not be cut off, so this should be opt-in or explicitly approved with coverage before merge.
  - confidence: 0.9
- **[P1] Clear buffered replies before aborting repetition:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:301`
  - body: By this point the triggering chunk has already been appended to `deltaBuffer` and the block reply buffer/chunker. If `session.abort()` ends the run before `message_end`, current `handleAgentEnd` still flushes that buffer, so the repeated text can still reach the channel; clear or suppress the buffer/flush path when the guard fires.
  - confidence: 0.9
- **[P1] Reset repetition state at the message boundary:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:186-188`
  - body: The new cursor and abort flag are not reset by `resetAssistantMessageState`, and the finalizer only resets the cursor. When an abort skips `message_end`, the next assistant message can inherit a stale abort flag/check length, dropping valid block replies or delaying future repetition checks.
  - confidence: 0.86
- **[P2] Wire the new config key into the strict schemas:** `src/config/types.tools.ts:335`
  - body: Adding `tools.textRepetitionGuard` to TypeScript config types is not enough because `AgentToolsSchema` and `ToolsSchema` are strict and still omit the key. Users who configure the advertised guard settings through normal config validation will get an unknown-key failure instead of applying the override.
  - confidence: 0.92
