## Summary

Keep this PR open. Current main still has the Slack interactive-reply wake gap, but this PR is not ready to merge as-is because the new wake path manually recreates normal Slack routing and misses runtime conversation-binding and account/chat-type reply-mode semantics that the regular inbound path already preserves.

## What This Changes

The PR adds a Slack reply-button wake helper that dispatches an agent turn after authorized interactive reply actions, plus regression tests for dispatch, threading, dedupe IDs, and delivery behavior.

## Best Possible Solution

Land one canonical Slack interaction wake implementation that reuses or factors the normal Slack inbound routing semantics, including runtime conversation bindings, thread session keys, parent inheritance, per-chat-type reply modes, authorization, dedupe, and same-thread delivery, then close or supersede the competing PRs against that chosen design.

## Reproduction Assessment

Yes. The issue path is concrete from the linked bug report and current main: with Slack `interactiveReplies` enabled, a reply-button click reaches `handleSlackBlockAction`, enqueues a system event, and updates the UI, but no dispatch occurs unless another reply run is already active.

## Solution Assessment

No, not as currently proposed. The direct-dispatch approach may be viable, but this implementation should first preserve the same routing and reply-mode contracts as `prepareSlackMessage` or share that code rather than duplicating a partial version in the interaction handler.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve runtime conversation bindings when waking:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:692-696`
  - body: This wake path derives `SessionKey` by applying `resolveThreadSessionKeys` directly to `route.sessionKey`. The regular Slack inbound path first runs `resolveRuntimeConversationBindingRoute` for the routed conversation/thread and only falls back to thread-key derivation when there is no binding, so a button click in a bound Slack conversation can dispatch against a different session than normal messages in the same thread.
  - confidence: 0.84
- **[P2] Resolve reply mode for the Slack chat type:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:732-760`
  - body: The new delivery path uses `ctx.replyToMode`, which is only the account default, when computing `replyThreadTs` and calling `deliverReplies`. Normal Slack inbound routing resolves `resolveSlackReplyToMode(account, chatType)`, so configs using `replyToModeByChatType` or `dm.replyToMode` will thread button-triggered replies differently from regular messages in the same conversation.
  - confidence: 0.8
