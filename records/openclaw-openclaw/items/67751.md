## Summary

Keep open: the PR targets a real Slack interaction wake bug, but the current patch still diverges from the normal Slack inbound routing and delivery contracts enough to block merge.

## What This Changes

The PR adds a Slack reply-button wake helper that dispatches an agent turn after authorized interactive reply actions, plus regression tests for wake dispatch, threading, dedupe IDs, parent inheritance, and delivery behavior.

## Best Possible Solution

Land one canonical Slack interaction wake implementation that reuses or factors the normal Slack inbound routing semantics, including runtime bindings, chat-type reply modes, thread keys, parent inheritance, authorization, dedupe, and same-thread delivery, then supersede competing Slack wake PRs against that design.

## Reproduction Assessment

Yes. Current main has a concrete path: with Slack interactiveReplies enabled, a reply-button click reaches handleSlackBlockAction, enqueues a system event, and updates the UI, but no dispatch occurs unless another reply run is active.

## Solution Assessment

No. A direct wake path may be viable, but this implementation should first share or faithfully reproduce prepareSlackMessage routing and delivery semantics instead of duplicating a partial version in the interaction handler.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve runtime conversation bindings when waking:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:692-696`
  - body: This wake path derives SessionKey by applying resolveThreadSessionKeys directly to route.sessionKey. The regular Slack inbound path first checks runtime conversation bindings for the routed thread/root conversation and only derives thread keys when no binding owns the route, so bound Slack conversations can dispatch button clicks to a different session than normal messages in the same thread.
  - confidence: 0.84
- **[P2] Resolve reply mode for the Slack chat type:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:732-760`
  - body: The new delivery path uses ctx.replyToMode, which is only the account default, when computing replyThreadTs and calling deliverReplies. Normal Slack inbound routing resolves resolveSlackReplyToMode(account, chatType), so replyToModeByChatType or dm.replyToMode overrides will thread button-triggered replies differently from regular messages in the same conversation.
  - confidence: 0.8
