## Summary

Keep open: the hardening goal is plausible, but this PR is not mergeable as-is because it misses the active Docker setup path on current main, changes documented Docker exposure behavior, adds a non-portable secure Compose surface, and lacks real after-fix behavior proof.

## What This Changes

The PR masks gateway token output in a Docker setup script, binds Docker Compose ports to loopback, adds `docker-compose.secure.yml`, broadens env-file ignores, and pins selected CI Actions to SHA refs.

## Best Possible Solution

Prepare a rebased, maintainer-reviewed hardening patch against `scripts/docker/setup.sh`, `docker-compose.yml`, and the current CI workflow; either drop `docker-compose.secure.yml` or make it portable, documented, and opt-in.

## Reproduction Assessment

Yes. Source inspection of current main shows the active setup script still prints the gateway token and Compose still publishes unqualified host ports; the PR diff also exposes the hardcoded secure Compose defaults.

## Solution Assessment

No. This PR is not the best solution as-is because it edits a stale setup path, changes documented Docker exposure policy, and introduces a hardcoded secure Compose surface; a narrower rebased patch is safer.

- **[P1] Move token masking to the active Docker setup script:** `docker-setup.sh:259`
  - body: Current main's Docker onboarding now lives in `scripts/docker/setup.sh`, which still prints `OPENCLAW_GATEWAY_TOKEN`. This hunk masks the retired root setup path, so rebasing it as-is leaves the active setup leak in place.

  - body: The new secure Compose file defaults config and workspace mounts to `/home/cmdshadow/...`. On machines without that account, users can create or mount unintended host paths for secret and state data, so the tracked secure config is not portable.
  - confidence: 0.92
- **[P2] Make the tailnet bind address configurable:** `docker-compose.secure.yml:24`
  - body: This port mapping binds `10.8.0.1` unconditionally. Compose fails on hosts without that address, and on hosts that do have it the gateway is exposed beyond loopback despite this file being introduced as a secure config.
  - confidence: 0.9
- **[P2] Keep the health-check hint self-contained:** `docker-setup.sh:288`
  - body: The script now prints `$OPENCLAW_GATEWAY_TOKEN` literally, but setup is normally executed rather than sourced. After it exits, the caller's shell usually lacks that variable, so the copied health command expands to an empty token and fails auth.
  - confidence: 0.88
