## Summary

This PR should stay open but is not merge-ready: the security goal is still relevant, but the branch is stale against current main and has blocking Docker/CI correctness and proof gaps.

## What This Changes

The branch masks token output in the old root Docker setup script, changes Compose port publishing to loopback, adds a secure Compose file, broadens env-file ignores, and pins several CI action refs.

## Best Possible Solution

Rework the branch against current main: mask token output in scripts/docker/setup.sh without breaking copyable diagnostics, settle Docker publish-host policy with docs, drop host-specific secure Compose defaults, and pin workflow actions only to valid commit SHAs.

## Reproduction Assessment

Yes. Source inspection on current main shows the active Docker setup script still prints the gateway token, and the PR diff itself shows the hardcoded secure Compose paths/IP and tag-object Gradle pin; no Docker runtime was run in this read-only review.

## Solution Assessment

No. The current branch is not the best way to solve the issue because it targets a deleted setup file, adds non-portable secure Compose defaults, changes Docker exposure policy without docs alignment, and leaves one action pin as a tag object.

  - confidence: 0.9

  - body: The Gradle action ref uses the annotated `v4` tag object SHA. Full action pinning should use the peeled commit SHA for the tag target, otherwise the audit trail is not the intended immutable commit ref.
  - confidence: 0.88

  - body: The script now prints `$OPENCLAW_GATEWAY_TOKEN` literally, but the setup script is normally executed rather than sourced. After it exits, copy/pasting the health command usually expands an empty token and fails auth.
