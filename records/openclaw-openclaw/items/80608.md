## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Register curator commands through registerCli:** `extensions/skill-curator/src/cli.ts:98-106`
  - body: `api.registerCommand` requires a flat command with a `handler`; it does not support `subcommands`, `run`, `options`, `display`, or `ctx.options`. As written, the curator shell commands will not satisfy the current plugin command contract.
  - confidence: 0.96
- **[P1] Use the runtime lifecycle contract:** `extensions/skill-curator/src/index.ts:19-26`
  - body: `registerRuntimeLifecycle` requires an `id` and registers cleanup metadata. `activate` and `deactivate` are not lifecycle callbacks in the current SDK, so this registration is outside the host contract and will be rejected or ignored.
  - confidence: 0.94
- **[P1] Constrain skill names before moving directories:** `extensions/skill-curator/src/run.ts:86-100`
  - body: `moveSkillToArchive` joins the sidecar skill key directly into paths before `fs.rename`. A crafted `.usage.json` key such as `../../outside` can move directories outside `skills/`; normalize the name and verify resolved paths stay under the skills root.
  - confidence: 0.9
- **[P2] Stop importing curator private internals from workshop:** `extensions/skill-workshop/src/tool.ts:24-30`
  - body: `skill-workshop` lazy-imports another plugin's private `src/telemetry.js`, which breaks the extension boundary. Expose a public API seam or keep the integration inside curator instead of depending on another plugin's private files.
  - confidence: 0.87
- **[P2] Add hook guardrails before agent_end mutations:** `extensions/skill-curator/src/index.ts:30-35`
  - body: A new bundled `agent_end` handler needs the hook allowlist/update path and should resolve live plugin config inside the handler. Reading startup `api.pluginConfig` means runtime config changes may not stop automatic curator mutations.
  - confidence: 0.83
