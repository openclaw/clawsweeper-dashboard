## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Register curator commands through the CLI contract:** `extensions/skill-curator/src/cli.ts:98-106`
  - body: `api.registerCommand` only accepts a flat command with a `handler`; it does not support `subcommands`, `run`, `options`, `display`, or `ctx.options`. This will fail the current plugin command contract and the advertised shell subcommands will not register.
  - confidence: 0.96
- **[P1] Use a real runtime lifecycle registration:** `extensions/skill-curator/src/index.ts:19-26`
  - body: `registerRuntimeLifecycle` requires an `id` and only registers cleanup metadata. `activate` and `deactivate` are not lifecycle callbacks in the current SDK, so this object is outside the host contract.
  - confidence: 0.94
- **[P1] Constrain sidecar skill names before moving directories:** `extensions/skill-curator/src/run.ts:86-100`
  - body: `moveSkillToArchive` joins the sidecar skill key directly into source and destination paths before `fs.rename`. A crafted `.usage.json` key such as `../../outside` can move directories outside `skills/`; validate the normalized name and resolve within the skills root first.
  - confidence: 0.9
- **[P2] Keep skill-workshop off curator private internals:** `extensions/skill-workshop/src/tool.ts:24-30`
  - body: This lazy import reaches from `skill-workshop` into another plugin's private `src/telemetry.js`, breaking the bundled plugin boundary. Expose a public API seam or keep the integration inside curator instead.
  - confidence: 0.87
- **[P2] Register long-lived hooks with live-config guardrails:** `extensions/skill-curator/src/index.ts:30-35`
  - body: Adding a new bundled `agent_end` handler needs the hook allowlist/update path and should resolve live plugin config inside the handler. Reading startup `api.pluginConfig` means runtime config changes may not stop automatic curator mutations.
  - confidence: 0.83
