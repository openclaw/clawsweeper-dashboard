## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Register curator commands through the CLI contract:** `extensions/skill-curator/src/cli.ts:98-106`
  - body: `api.registerCommand` requires a flat command definition with `handler`; it does not accept `subcommands`, `run`, `options`, `display`, or `ctx.options`. As written, the advertised `openclaw curator ...` commands are outside the current plugin command contract and will not register correctly.
  - confidence: 0.96
- **[P1] Use a real runtime lifecycle registration:** `extensions/skill-curator/src/index.ts:19-26`
  - body: `registerRuntimeLifecycle` requires an `id` and only tracks cleanup metadata. Passing `activate` and `deactivate` without an id is rejected by the registry and does not create plugin activation callbacks.
  - confidence: 0.94
- **[P1] Constrain sidecar skill names before moving directories:** `extensions/skill-curator/src/run.ts:86-100`
  - body: `moveSkillToArchive` and restore join `skillName` directly into source and destination paths before `fs.rename`. A crafted `.usage.json` or CLI skill name with traversal segments can move directories outside `skills/`; normalize and resolve within the skills root before mutating.
  - confidence: 0.9
- **[P2] Keep skill-workshop off curator private internals:** `extensions/skill-workshop/src/tool.ts:24-30`
  - body: This lazy import reaches from `skill-workshop` into another plugin's private `src/telemetry.js`, breaking the extension boundary that third-party plugins also see. Expose a public seam or keep the integration owned by curator instead.
  - confidence: 0.87
