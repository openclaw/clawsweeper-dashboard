Full review comments:

- **[P1] Avoid overwriting current exec approvals from legacy state:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This import writes the legacy raw approvals into the single SQLite row and then deletes the source. `writeExecApprovalsRawToSqlite` updates an existing row on conflict, so a stale legacy file can roll back current approvals after the user has already saved SQLite state; skip or merge when current SQLite state exists, and keep the legacy file when it is not imported.
  - confidence: 0.9
- **[P1] Merge legacy device-auth tokens instead of replacing rows:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: `writeDeviceAuthStoreSnapshot` treats the legacy snapshot as authoritative and deletes rows for other device IDs or roles before upserting. During doctor repair, a stale `identity/device-auth.json` can roll back current device tokens and then remove the source file; merge per token or skip when SQLite already has newer rows.
  - confidence: 0.89
- **[P1] Validate web-push snapshots before deleting subscriptions:** `src/commands/doctor/legacy/push-web.ts:47-50`
  - body: The import accepts any parsed object, including `{}` or arrays, and then calls a snapshot writer that deletes all `web_push_subscriptions` when no rows are present. A malformed legacy file can therefore wipe current web-push registrations and delete the retry source; validate the snapshot shape before writing.
  - confidence: 0.88
- **[P1] Reject malformed node-host config before upserting:** `src/commands/doctor/legacy/node-host-config.ts:42-43`
  - body: `coercePartialConfig(JSON.parse(raw)) ?? {}` lets malformed-but-parseable objects become an empty config, and `saveNodeHostConfig` normalizes that into a fresh `nodeId` before upserting the single row. That can overwrite an already-current node identity during doctor repair; require a valid legacy node-host payload before saving and removing the source.
  - confidence: 0.86
