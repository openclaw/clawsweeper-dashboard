## Summary

Keep open. This PR has protected `security` and `maintainer` labels, and the current head still has a source-reproducible migration bug in the new legacy plugin-binding approval import path.

## What This Changes

The branch reapplies the broad database-first runtime-state refactor, moving runtime state, docs, tests, Android/macOS state paths, and many store/helper surfaces toward SQLite after the prior PR was reverted.

## Best Possible Solution

Keep the SQLite refactor as a review PR, thread the explicit doctor env through every legacy migration helper, confirm the Node/runtime policy with owners, and add representative after-fix proof before merge.

## Reproduction Assessment

Yes, source-reproducible. On the PR head, call `maybeRepairLegacyRuntimeStateFiles({ env: { OPENCLAW_STATE_DIR: <custom> } })` with a legacy plugin-binding approval file in that custom state dir while `process.env` points elsewhere; the new plugin-binding probe/import resolves against `process.env` instead of the caller env.

## Solution Assessment

No. The broad SQLite direction may be intended, but this helper should accept an env parameter and pass it through both the existence check and import/write path so doctor migrations remain profile-scoped.
