## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid overwriting newer exec approvals from legacy files:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This imports the legacy JSON into the single SQLite approvals row and then removes the file unconditionally. If a user already has newer SQLite approvals plus a stale legacy file, doctor rolls policy back and removes the rollback source; the import needs to skip or merge when SQLite is already authoritative.
  - confidence: 0.92
- **[P1] Merge legacy device auth instead of replacing rows:** `src/commands/doctor/legacy/device-auth-store.ts:39`
  - body: `writeDeviceAuthStoreSnapshot` ultimately deletes tokens for other device IDs and roles before upserting the legacy snapshot. A stale `identity/device-auth.json` can therefore roll back or remove newer SQLite device tokens during doctor migration.
  - confidence: 0.91
- **[P1] Reject malformed web-push snapshots before import:** `src/commands/doctor/legacy/push-web.ts:46-50`
  - body: This accepts any parsed object as a web-push snapshot, including `{}` or an array, and `persistState` deletes all `web_push_subscriptions` rows when the incoming snapshot has no rows. A malformed legacy file can wipe existing registrations and then be removed.
  - confidence: 0.9
- **[P1] Validate node-host payload before saving SQLite config:** `src/commands/doctor/legacy/node-host-config.ts:42-43`
  - body: Malformed-but-parseable payloads like `{}` are coerced into an empty config and saved, which can normalize a fresh node identity over an existing SQLite node-host row before deleting the legacy file. Reject invalid legacy config instead of treating it as a successful import.
  - confidence: 0.88
- **[P1] Merge pairing snapshots instead of replacing live rows:** `src/commands/doctor/legacy/pairing-files.ts:51-58`
  - body: The import writes pairing state whenever `readJsonIfExists` returns anything other than `undefined`, while the writer deletes rows not present in the incoming snapshot and deletes all rows for empty snapshots. Missing, stale, or reduced legacy files can erase newer pending/paired device or node rows.
  - confidence: 0.9
