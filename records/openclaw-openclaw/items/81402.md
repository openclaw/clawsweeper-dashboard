## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid overwriting current exec approvals from legacy state:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: `doctor --fix` now writes any legacy `exec-approvals.json` into the single SQLite config row and removes the source file. If a user already has newer SQLite approvals plus a stale legacy file, this rolls security policy back and removes the rollback artifact; skip or merge based on freshness instead of unconditional replacement.
  - confidence: 0.88
- **[P1] Merge legacy device-auth tokens instead of replacing rows:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: The legacy device-auth import writes the parsed snapshot through `writeDeviceAuthState`, whose implementation deletes other device rows and roles missing from the imported file. A stale `identity/device-auth.json` can therefore roll back or erase newer SQLite device tokens during doctor repair; import should merge only missing/older rows or refuse stale replacement.
  - confidence: 0.86
- **[P1] Validate web-push snapshots before deleting subscriptions:** `src/commands/doctor/legacy/push-web.ts:47-50`
  - body: This treats any parsed object, including `{}` or an array, as a valid web-push snapshot. The persistence path converts that to zero rows and deletes every existing `web_push_subscriptions` row before the legacy file is removed, so malformed legacy state can wipe current push registrations.
  - confidence: 0.87
- **[P2] Keep unreadable pairing files for retry:** `src/commands/doctor/legacy/channel-pairing.ts:263-272`
  - body: When `readLegacyPairingStore` returns null after a read or parse failure, the loop still deletes the source `*-pairing.json` file and counts it as processed. That can silently discard pending pairing requests without importing them, so only remove the file after a successful parse/import or preserve it for operator repair.
  - confidence: 0.84
