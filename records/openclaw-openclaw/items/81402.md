## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid overwriting current exec approvals from legacy state:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This import writes the legacy approvals JSON into SQLite and then removes the file whenever it exists. If a user already has newer SQLite approval state and a stale legacy file remains, `doctor --fix` rolls the policy back and deletes the rollback source.
  - confidence: 0.91
- **[P1] Merge legacy device-auth tokens instead of replacing rows:** `src/commands/doctor/legacy/device-auth-store.ts:39`
  - body: The importer treats a valid legacy snapshot as authoritative. `writeDeviceAuthStoreSnapshot` deletes token rows for other device ids and missing roles, so a stale `identity/device-auth.json` can roll back current SQLite device tokens during doctor repair.
  - confidence: 0.88
- **[P1] Validate web-push snapshots before deleting subscriptions:** `src/commands/doctor/legacy/push-web.ts:47-50`
  - body: Any parsed object reaches `writeWebPushRegistrationStateSnapshot`, but the persistence path deletes all `web_push_subscriptions` when the snapshot has no rows. A stale or malformed legacy file can therefore wipe newer push registrations and then remove the source file.
  - confidence: 0.9
- **[P2] Keep unreadable pairing files for retry:** `src/commands/doctor/legacy/channel-pairing.ts:263-271`
  - body: When `readLegacyPairingStore` returns null for malformed or unreadable JSON, this still removes the legacy pairing file. That drops pending pairing requests without importing them and prevents an operator from fixing the file and rerunning doctor.
  - confidence: 0.87
