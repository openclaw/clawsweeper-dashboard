Overall confidence: 0.88

Full review comments:

- **[P1] Keep legacy auth state until parsing succeeds:** `src/commands/doctor/legacy/auth-profile-state.ts:30-33`
  - body: `loadJsonFile()` can return `undefined` for malformed JSON, but this path coerces that into an empty auth-profile state, saves it, and then unlinks the legacy source. A truncated `auth-state.json` would be erased with no retry source, so guard deletion on a successful semantic parse/import.
  - confidence: 0.91
- **[P1] Pass the doctor env into transcript migration:** `src/commands/doctor/legacy/session-transcript-health.ts:424`
  - body: `noteSessionTranscriptHealth()` resolves legacy session dirs from `process.env`, and the doctor contribution calls it without `ctx.env`. Scoped doctor repairs can therefore read/delete transcripts from the caller's default state instead of the requested target; pass the env through the helper and all SQLite writes.
  - confidence: 0.88
- **[P1] Thread scoped env through legacy cron migration:** `src/commands/doctor/legacy/cron.ts:347-349`
  - body: `maybeRepairLegacyCronStore()` resolves both the legacy cron store path and SQLite store key from defaults instead of the doctor env. A scoped `OPENCLAW_STATE_DIR` repair can import or remove cron state in the wrong store, so this path needs an env parameter carried through detection, imports, and deletes.
  - confidence: 0.86
- **[P1] Keep exec approvals when import cannot parse:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: `writeExecApprovalsRawToSqlite()` normalizes JSON parse failures to an empty approvals config, but the legacy import then removes the source file. A malformed `exec-approvals.json` would silently reset approvals and lose the only recoverable source, so make parse/import failure preserve the file.
  - confidence: 0.9
- **[P1] Merge legacy pairing rows instead of replacing SQLite rows:** `src/commands/doctor/legacy/pairing-files.ts:52-57`
  - body: The migration writes legacy pending/paired snapshots through `writePairingStateRecord()`, whose implementation deletes rows not present in the incoming snapshot. If SQLite already has newer pairing records while stale legacy files still exist, doctor can delete the newer rows; migration should merge legacy rows or only replace when the target table is empty.
  - confidence: 0.84
