## Summary

Keep this PR open. It is a broad, protected SQLite runtime-state refactor with current security-sensitive review findings on auth/profile migration and OAuth refresh locking, so it is not a cleanup-close candidate.

## What This Changes

The PR reapplies the database-first runtime-state refactor, moving active runtime state from file-backed JSON/JSONL/sidecar stores into global and per-agent SQLite databases across core, apps, plugins, docs, scripts, and tests.

## Best Possible Solution

Keep the SQLite refactor as a maintainer-reviewed storage PR, fix the auth-store locking and doctor env propagation, confirm Node/dependency policy, then require representative migration/runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible. The doctor env bug follows from calling the new plugin-binding probe/import without the caller env, and the OAuth lost-update race follows from a provider/profile lock around whole-store read-modify-write saves.

## Solution Assessment

No. The broad SQLite direction may be intended, but the current branch should preserve profile-scoped doctor env handling and serialize OAuth writes by auth store, not only by provider/profile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Serialize OAuth refresh writes per auth store:** `src/agents/auth-profiles/oauth-manager.ts:328-335`
  - body: The refresh critical section now locks only `provider+profileId`, but it still reads and saves the whole auth-profile store. Concurrent refreshes for two profile IDs in the same store can each read the same old row and then overwrite the other profile's newly refreshed token on save.
  - confidence: 0.9
- **[P2] Thread doctor env through approval imports:** `src/commands/doctor/legacy/plugin-conversation-binding.ts:13-16`
  - body: `maybeRepairLegacyRuntimeStateFiles` accepts an explicit env and passes it into the other legacy migrations, but this helper resolves the approval file from global `process.env`. Doctor runs for a non-default `OPENCLAW_STATE_DIR` or `OPENCLAW_HOME` can skip or migrate approval state from the wrong profile during the SQLite cutover.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR touches authorization and credential state, and the current diff has concrete approval-migration and OAuth-token lost-update risks.

Concerns:

- **[medium] Serialize OAuth token refresh writes by store:** `src/agents/auth-profiles/oauth-manager.ts:328`
  - body: OAuth refresh tokens are credential state. Locking only by provider/profile while saving an entire auth-profile store can drop another concurrent profile's freshly refreshed token, causing stale or invalid credential state under concurrent refresh traffic.
  - confidence: 0.88
- **[medium] Keep plugin approval migration profile-scoped:** `src/commands/doctor/legacy/plugin-conversation-binding.ts:13`
  - body: Plugin-binding approvals are authorization state. Resolving the legacy approval file from global `process.env` can skip or import approvals from the wrong state profile when doctor is run with an explicit env.
  - confidence: 0.86
