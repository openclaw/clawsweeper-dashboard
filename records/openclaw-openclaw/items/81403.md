## Summary

Keep open: the bug-fix direction is plausible, but this PR is not mergeable yet because the wrapper can introduce an empty PATH segment, the new command-wrapper assertion expects the wrong escaped string, the RC-file regression test does not model startup-file ordering, and after-fix real behavior proof is still missing.

## What This Changes

The PR threads `tools.exec.pathPrepend` into exec spawning, adds a POSIX shell PATH prologue, adds helper and exec tests, and documents gateway PATH precedence against shell startup files.

## Best Possible Solution

Keep the late-shell PATH prepend approach only if it conditionally appends the old PATH, preserves safe env handling, adds a true startup-file regression test, and includes redacted real macOS/gateway proof.

## Reproduction Assessment

No high-confidence live reproduction is attached. Source inspection shows current main applies `pathPrepend` before spawning the POSIX shell, so the reported ordering problem is plausible, but the PR’s current test does not reproduce startup-file behavior.

## Solution Assessment

No. Moving the prepend into the spawned shell is a plausible narrow fix, but the current diff needs safer PATH concatenation, corrected tests, and real behavior proof before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Avoid appending an empty PATH segment:** `src/agents/bash-tools.exec-runtime.ts:602`
  - body: The wrapper always expands to `OPENCLAW_PREPEND_PATH:$PATH`. If PATH is empty after removing the configured entries, the trailing colon adds the current directory to command lookup; use a conditional form such as `${PATH:+:$PATH}` so the hardened path does not widen execution search.
  - confidence: 0.91
- **[P2] Expect the shell-expanded wrapper string:** `src/agents/bash-tools.exec-runtime.test.ts:720`
  - body: This assertion expects a literal backslash before `${OPENCLAW_PREPEND_PATH}`, but the source template literal uses the backslash only to avoid JavaScript interpolation and emits `${OPENCLAW_PREPEND_PATH}` at runtime. As written, the new test fails or guards the wrong command string.
  - confidence: 0.95
- **[P2] Simulate the RC override before the wrapper runs:** `src/agents/bash-tools.test.ts:862`
  - body: This command changes PATH after the wrapper prologue, then expects `/evil/bin` to stay first. That models user-command behavior rather than `~/.zshenv` or `/etc/zshenv` startup behavior, so it does not prove the precedence guarantee this PR documents.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff touches PATH hardening and currently can add a current-directory PATH element when the pre-wrapper PATH is empty.

Concerns:

- **[medium] Avoid current-directory PATH entries:** `src/agents/bash-tools.exec-runtime.ts:602`
  - body: Shells treat empty PATH entries, including a trailing colon, as the current directory. The new wrapper always appends `:$PATH`, so a minimal or stripped PATH can accidentally broaden executable lookup while claiming to harden precedence.
  - confidence: 0.88
