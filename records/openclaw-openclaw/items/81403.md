## Summary

Keep open. The PR targets a plausible exec PATH precedence bug and the final-shell wrapper is the right general boundary, but the added regression test is not valid against the current host-env sanitization and mocked exec harness, so the patch should not merge as-is.

## What This Changes

The PR passes `tools.exec.pathPrepend` into host exec runtime, wraps POSIX commands to prepend PATH after shell startup, adds path helper coverage, and updates exec docs.

## Best Possible Solution

Keep the final-shell PATH prepend approach, but land it with a regression test that uses a real shell startup path or accurately models sanitized env plus wrapper execution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding: the current mock reads `echo $PATH` from spawn env only while `BASH_ENV` is blocked from inherited host env, so the new test path is source-reproducible as invalid without running the branch.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the runtime boundary is maintainable, but the regression test should exercise actual shell startup ordering or a mock that applies the wrapper before this PR is merged.
