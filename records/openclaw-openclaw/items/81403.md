## Summary

Keep open: the intended host-gateway PATH hardening is plausible, but this patch still has a command-execution PATH edge case, incorrect or misleading regression coverage, and no after-fix real behavior proof.

## What This Changes

This PR threads `tools.exec.pathPrepend` into host-gateway exec runtime, wraps POSIX commands with an in-shell PATH export, adds path-prepend helper/tests, and documents the precedence guarantee.

## Best Possible Solution

Land a narrow host-gateway PATH wrapper only after it avoids empty PATH segments, has a startup-file regression test, and includes redacted real gateway proof from the reported environment.

## Reproduction Assessment

No. The PR body lists a macOS gateway scenario but leaves the evidence blank, and current source already uses startup-suppressed shell args for common POSIX shells, so the current-main failure path still needs proof.

## Solution Assessment

No. Applying the final prepend inside the POSIX shell may be the right direction, but this implementation needs the empty-PATH guard, corrected tests, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid adding an empty PATH segment:** `src/agents/bash-tools.exec-runtime.ts:602`
  - body: When the base PATH is empty after removing configured prepends, this expands to `<prepend>:`. POSIX treats a trailing empty segment as the current directory, which weakens this hardening change by letting cwd command lookup back in; build the export as `${OPENCLAW_PREPEND_PATH}${PATH:+:$PATH}` or equivalent.
  - confidence: 0.86
- **[P2] Fix the wrapper assertion to match the runtime string:** `src/agents/bash-tools.exec-runtime.test.ts:720`
  - body: The wrapper source uses `\${...}` only to escape JS template interpolation, so the spawned command contains `${OPENCLAW_PREPEND_PATH}` without a backslash. This assertion looks for `\${...}` and will fail when the new test is run.
  - confidence: 0.9
- **[P3] Simulate startup PATH changes before the command runs:** `src/agents/bash-tools.test.ts:862`
  - body: This test puts `/evil/bin` in the user command, so it runs after the wrapper and even expects `/evil/bin` first. It does not exercise `.zshenv` or another startup hook that runs before `-c`, so it does not prove the precedence guarantee being documented.
  - confidence: 0.88

## Security Review
