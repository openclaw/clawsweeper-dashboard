## Summary

Keep open. Current main contains several meaningful mitigations, but the remaining issue is not fully implemented: bundled runtime dependency and mirror preparation still use synchronous filesystem traversal and locks, and Gateway startup still treats plugin runtime-deps loading as a readiness-blocking phase. Maintainer comments and the merged follow-up PRs also frame the landed work as partial rather than a complete async/cooperative fix.

## What This Changes

The issue requests that bundled plugin runtime mirror and runtime-dependency preparation stop synchronously blocking Gateway and pi-agent/plugin runtime paths for tens of seconds.

## Best Possible Solution

Keep this issue open until packaged/global-install profiling shows bounded event-loop lag for startup plugin loading and first pi-agent/tool construction. The durable fix should keep the package-level convergence improvements, then make the remaining mirror/runtime-deps preparation cooperative or move it off latency-critical Gateway paths without weakening bundled dependency validation.

## Reproduction Assessment

Yes for the reported release path: the issue includes a versioned npm-global repro, CPU attribution, standalone mirror benchmarks, and a smaller acpx-only startup-readiness repro in comments. For current main, the code still exposes the synchronous path, but the exact remaining wall-clock severity needs fresh packaged profiling after the merged mitigations.

## Solution Assessment

Unclear. The current main changes are good partial mitigations, but they are not the complete requested solution because the hot mirror path and lock remain synchronous and readiness can still wait on plugin-runtime-deps work.
