## Summary

Keep open. Current main contains useful partial mitigations for repeated dist-mirror scans, hardlink churn, and runtime-deps reinstall loops, but the central event-loop/readiness problem is not fully resolved: bundled runtime preparation still uses synchronous filesystem paths, Gateway startup still pre-stages runtime deps before plugin bootstrap completes, and the issue discussion explicitly leaves the broader async/readiness fix open with PR #75183 as related follow-up.

## What This Changes

Request to make bundled plugin runtime-deps and runtime-mirror preparation stop synchronously blocking Gateway and pi-agent paths, especially repeated packaged dist mirror sweeps and Gateway startup readiness staging.

## Best Possible Solution

Keep the issue open until #75183 or an equivalent maintainer-approved fix lands and packaged-install profiling shows bounded Gateway HTTP readiness plus bounded first-agent event-loop lag for bundled runtime-deps and mirror preparation.

## Reproduction Assessment

Yes. The issue body provides a versioned npm-global reproduction, CPU attribution, and standalone mirror benchmarks, and NikolaFC added an acpx-only isolated-HOME startup-readiness repro; current source still contains the synchronous paths those reproductions target.

## Solution Assessment

Unclear. Current main is a useful partial solution, but the remaining fix needs maintainer choice between narrower readiness deferral/caching and a broader async/cooperative runtime-mirror preparation path.
