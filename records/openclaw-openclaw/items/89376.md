## Summary

Keep open for maintainer judgment: current main and the latest release intentionally prefer a supported system Node over version-manager Node for managed daemon services, but the installer can still detect/link the PATH Node, so the mise/systemd mismatch is real enough to clarify or deliberately change.

## What This Changes

The issue asks Linux systemd gateway service generation to use the Node runtime found via mise instead of a globally installed apt/NodeSource Node.

## Best Possible Solution

Keep the stable system-Node daemon default unless maintainers choose an explicit opt-in, and add clear installer/gateway diagnostics or docs so mise users understand which Node the managed service will run.

## Reproduction Assessment

Yes for the source-level behavior: current code treats mise as version-managed and prefers a supported system Node for daemon services. A live Ubuntu/mise run is still needed to prove a runtime failure rather than an expected-but-surprising service file.

## Solution Assessment

No clear best fix yet: unconditionally making systemd use mise conflicts with shipped tests and audit policy. The maintainable path needs a maintainer decision between clearer diagnostics/docs, an explicit opt-in wrapper/runtime path, or changing the daemon default.
