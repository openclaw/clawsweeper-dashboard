## Summary

Keep open for maintainer judgment: current main and the latest release deliberately prefer a supported system Node for managed Gateway services when a version-manager Node such as mise is active, while the installer can still detect and link the PATH Node, so the reported mismatch is real but not a safe automatic bug fix.

## What This Changes

The reporter wants the Linux systemd Gateway service generated by the installer to use the mise-installed Node that the installer detected instead of an apt/global Node.

## Best Possible Solution

Maintain the safe system-Node default unless maintainers intentionally add or document an explicit version-manager daemon opt-in, and make the installer/service messaging clear about that split.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a credible path: with a mise process execPath and a supported /usr/bin/node, the service planner selects the system Node; I did not run a live Ubuntu 24.04 mise plus apt install.

## Solution Assessment

Is this the best way to solve the issue? Unclear: automatically switching managed services to mise conflicts with current daemon best-practice policy, so maintainers should first choose between documentation, existing wrapper guidance, or an explicit opt-in service-runtime override.
