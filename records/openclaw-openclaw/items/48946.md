## Label Justifications

- `P2`: This is a normal-priority Feishu reliability improvement with limited channel scope, not an urgent core runtime regression.
- `merge-risk: 🚨 message-delivery`: The PR can miss recovery entirely because it is not started, and its reconstructed events can route messages with empty or wrong chat metadata.
- `merge-risk: 🚨 session-state`: Recovered messages are dispatched with a fresh chatHistories map instead of the monitor's shared state, losing group context and ordering assumptions.
- `merge-risk: 🚨 security-boundary`: The raw @ fallback can process unrelated group-history content through the agent path even when the bot was not addressed.

## Summary

Keep open: the missed-message problem is plausible and still not implemented on current main, but this PR is not mergeable because the new recovery module is unintegrated, does not match current Feishu/SDK contracts, can replay unrelated group history into the agent path, and lacks real Feishu proof.

## What This Changes

The PR adds a Feishu message recovery module intended to poll recent group chat history and replay missed bot-directed messages after gateway downtime.

## Best Possible Solution

Design Feishu catchup as an opt-in monitor-lifecycle feature that uses the current client factory, SDK list APIs, structured mentions, persistent dedupe, shared queue/history state, bounded cursors, tests, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection and the pinned SDK contract show no lifecycle caller, invalid imports, wrong client/API use, wrong message field paths, and shared-state bypass. No live Feishu tenant proof establishes the original offline missed-message symptom or the proposed recovery behavior.

## Solution Assessment

No. A standalone interval scanner that is not wired into the monitor and bypasses existing queue, dedupe, structured mention, shared history, and privacy controls is not the maintainable solution.
