## Label Justifications

- `P2`: This is a Feishu channel reliability improvement with limited blast radius, not an urgent core runtime outage.
- `merge-risk: 🚨 message-delivery`: The module is not started by the monitor and reconstructs events with wrong chat/message metadata, so recovered messages can be dropped or misrouted.

- `merge-risk: 🚨 security-boundary`: The broad `@` fallback can route unrelated Feishu group-history messages into the agent/model path.

## Summary

Keep open: the Feishu catch-up goal is still a valid channel reliability feature, but this PR is not merge-ready because the added module is not wired into the monitor lifecycle, does not match current Feishu/Lark SDK and dedupe contracts, bypasses existing message state, and lacks real Feishu recovery proof.

## What This Changes

The PR adds `extensions/feishu/src/feishu-message-recovery.ts`, a standalone Feishu recovery service that polls chats every 30 seconds and tries to replay missed bot-mentioned messages.

## Best Possible Solution

Design Feishu catch-up as an opt-in monitor-lifecycle feature using `createFeishuClient`, current SDK list APIs, structured mentions, persistent cursors/dedupe, shared queue/history state, focused tests, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection plus the pinned SDK type contract show the lifecycle, import, API, field-path, and shared-state problems. No live Feishu tenant proof establishes the original offline missed-message symptom or the proposed recovery behavior.

## Solution Assessment

No. A standalone interval scanner that is not wired into the monitor and bypasses existing queue, dedupe, structured mention, shared history, and privacy controls is not the maintainable solution.
