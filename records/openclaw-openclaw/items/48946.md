## Summary

Keep open: the feature direction is plausible, but the current PR is not mergeable because the new recovery module is not wired into the Feishu monitor lifecycle and conflicts with current Feishu client, dedupe, message-shape, mention-policy, queue, and history contracts.

## What This Changes

This PR adds a new Feishu plugin message-recovery module that polls recent chats and attempts to replay missed bot-mention messages after offline periods.

## Best Possible Solution

Rework this as a Feishu-owned recovery path integrated with the monitor lifecycle and shared inbound handler contracts, using the current client/list APIs while preserving group policy, structured mentions, dedupe, queueing, and conversation history.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR imports non-exported dedupe symbols, never starts from the monitor lifecycle, reads a non-existent `account.client`, and uses SDK/message shapes that conflict with current main and the Lark SDK types.

## Solution Assessment

No. The feature is useful, but this patch is not the best implementation because it bypasses the current monitor lifecycle, client factory, message-list API, structured mention policy, dedupe claims, sequential queue, and shared chat history.

Full review comments:

- **[P1] Use the exported Feishu dedupe helpers:** `extensions/feishu/src/feishu-message-recovery.ts:15`
  - body: `dedup.ts` does not export `hasRecordedMessage`, and `hasRecordedMessagePersistent` is private, so this new file is incompatible with the Feishu plugin's current TypeScript surface. Use the current claim/has-processed/finalize helpers instead.
  - confidence: 0.97
- **[P1] Start recovery from the monitor lifecycle:** `extensions/feishu/src/feishu-message-recovery.ts:45-62`
  - body: The new service exports start/stop/manual functions but nothing calls them, so no interval is ever started or stopped and missed Feishu messages still are not recovered after reconnect.
  - confidence: 0.96
- **[P1] Create a Feishu client before polling:** `extensions/feishu/src/feishu-message-recovery.ts:89-90`
  - body: `resolveFeishuAccount` returns resolved credentials/config, not an object with `account.client`; casting `account.client as Lark.IM` leaves recovery without a usable SDK client. Follow the existing `createFeishuClient(account)` path.
  - confidence: 0.96
- **[P1] Use the current chat and message list APIs:** `extensions/feishu/src/feishu-message-recovery.ts:95-113`
  - body: The Lark SDK exposes chat discovery as `client.im.chat.list(...)` and history as `client.im.message.list(...)`; `listChatMembers`/`listChatMessage` with `sort`/`order`/`limit` do not match the dependency contract and will not fetch usable messages.
  - confidence: 0.94
- **[P1] Populate recovered events from message-list items:** `extensions/feishu/src/feishu-message-recovery.ts:141-153`
  - body: Message-list items expose identifiers such as `chat_id`, `msg_type`, `body.content`, and `mentions` on the item shape, not under `msg.body.chat_id/chat_type/message_type`, so recovered events will be malformed or routed with empty/wrong chat metadata.
  - confidence: 0.92
- **[P2] Dispatch recovered messages through shared handler state:** `extensions/feishu/src/feishu-message-recovery.ts:156-163`
  - body: Calling `handleFeishuMessage` directly with a fresh `new Map()` skips the monitor's shared conversation history and serialized inbound queue, so recovered messages can lose context and race active inbound handling.
  - confidence: 0.89
- **[P2] Filter recovered messages with structured mentions:** `extensions/feishu/src/feishu-message-recovery.ts:132-135`
  - body: The raw `content.includes("@")` fallback treats emails, pasted handles, and broadcast mentions as bot-directed recovery candidates, conflicting with Feishu's documented mention policy. Use the structured `mentions` data and existing bot-mention rules.
  - confidence: 0.86
