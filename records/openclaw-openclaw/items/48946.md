## Summary

Keep open: the Feishu catchup goal is useful, but this branch is not merge-ready because it adds an unwired recovery file, uses APIs and helper imports that do not match current main, bypasses monitor state, and lacks real Feishu reconnect proof. The remaining work needs maintainer/product judgment around opt-in history polling, privacy, lifecycle, and session semantics before a repair lane is safe.

## What This Changes

The PR adds a Feishu message recovery module that polls recent chats and tries to replay bot-mentioned messages missed while the gateway was offline.

## Best Possible Solution

Design Feishu catchup as an opt-in monitor-lifecycle feature using `createFeishuClient`, current SDK chat/message list APIs, structured mentions, persistent cursors/dedupe, shared queue/history state, focused tests, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection plus the pinned SDK contract show the lifecycle, import, API, field-path, and shared-state problems. No live Feishu tenant proof establishes the original offline missed-message symptom or the proposed recovery behavior.

## Solution Assessment

No. A standalone interval scanner that is not wired into the monitor and bypasses existing queue, dedupe, structured mention, shared history, and privacy controls is not the maintainable solution.
