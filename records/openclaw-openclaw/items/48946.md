## Summary

Keep this PR open because the underlying Feishu missed-message recovery gap is still not implemented on current main, but the proposed patch is not mergeable as written. Static review against current main confirms build, lifecycle, SDK, event-shape, mention-filtering, and shared-state integration blockers that would prevent reliable recovery.

## What This Changes

Adds a new Feishu message recovery module that periodically scans recent group chat messages, skips recorded message IDs, and replays messages that appear to mention the bot, with start/stop/stats/manual recovery entry points.

## Best Possible Solution

Keep the PR open for rework or supersede it with a Feishu-owned implementation that hooks startup/reconnect recovery into the existing monitor lifecycle, uses `createFeishuClient` with current `im.chat.list` and `im.message.list` contracts, reconstructs events from current message-list item fields, dispatches through the shared claim/debounce/sequential-queue/history path, respects Feishu group and structured mention policy, documents any required scopes, and adds focused regression tests plus live API proof where feasible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR diff imports a non-exported dedupe symbol, has no caller for `startMessageRecovery`, reads `account.client` even though `ResolvedFeishuAccount` has no client field, and bypasses the current handler path; no live Feishu account is needed to reproduce those blockers.

## Solution Assessment

No. The direction targets the right Feishu gap, but this patch is not the narrowest maintainable fix because it adds a side polling module outside the monitor lifecycle and does not reuse current client, message-list, group-policy, dedupe, queue, or history contracts.
