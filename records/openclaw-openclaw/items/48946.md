## Summary

Keep open. Current main does not implement Feishu missed-message recovery, so this cannot be closed as implemented, and the PR branch is not mergeable as-is because the new module is unwired, uses non-existent current-main helpers and account fields, calls the wrong Lark SDK APIs, reconstructs malformed events, and bypasses the shared monitor state.

## What This Changes

The PR adds a Feishu message-recovery module that periodically polls recent group-chat messages, skips previously recorded IDs, attempts to replay bot mentions, and exposes start, stop, stats, and manual recovery functions.

## Best Possible Solution

Rework or supersede the PR with a Feishu-owned implementation that hooks recovery into `monitorSingleAccount`, uses `createFeishuClient` with `im.chat.list` and `im.message.list`, reconstructs events from the current message-list item shape including structured mentions, dispatches through the shared claim, debounce, queue, and history path, documents any required Feishu scopes, and adds focused regression plus live API proof where feasible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR diff imports a symbol current main does not export, has no lifecycle caller, reads a non-existent `account.client` field, and uses SDK/message shapes that conflict with current main and the Lark SDK types.

## Solution Assessment

No. The direction addresses a real Feishu gap, but this patch is not the best way to solve it because it adds a side polling module outside the monitor lifecycle and does not reuse the current client, message-list, group-policy, dedupe, queue, or shared history contracts.
