## Summary

Keep open: the Feishu catchup problem is plausible, but this patch is not mergeable because the new scanner is unintegrated, uses nonexistent/private helpers, calls SDK APIs that do not match the pinned Lark contract, and lacks real Feishu reconnect proof.

## What This Changes

The PR adds a standalone Feishu message recovery module that periodically polls chat history and replays bot-mentioned messages after downtime.

## Best Possible Solution

Design Feishu catchup inside the monitor lifecycle behind explicit opt-in config, using the current client factory, SDK list APIs, structured mentions, existing dedupe/queue/shared histories, bounded cursors, tests, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows no monitor caller, invalid dedupe imports, wrong client/API assumptions, and event reconstruction that does not match current contracts. No live Feishu tenant proof was established for the original missed-message symptom.

## Solution Assessment

No. A standalone interval scanner that bypasses monitor lifecycle, shared state, queueing, structured mention policy, and opt-in privacy controls is not the best maintainable solution.
