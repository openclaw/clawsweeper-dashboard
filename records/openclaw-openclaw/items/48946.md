## Summary

Keep this PR open but do not merge it as-is: the diff is a standalone Feishu recovery scanner that is not wired into the monitor lifecycle, does not match current Feishu account/client/dedupe contracts, and lacks after-fix real behavior proof. The missed-message recovery direction may be useful, but it needs Feishu owner/product review before a safe implementation.

## What This Changes

The PR adds a new Feishu message recovery module that polls recent chats every 30 seconds and replays unprocessed bot-mention messages.

## Best Possible Solution

Implement Feishu catchup inside the monitor lifecycle behind an explicit safe config/policy surface, using SDK list APIs, structured mentions, shared dedupe, shared queueing, shared chat history, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: static source review against current main shows the module is never started, imports unavailable dedupe helpers, reads account.client, and calls SDK methods/fields that do not match current contracts. The original live reconnect behavior was not reproduced in a real Feishu setup here.

## Solution Assessment

No. The feature direction is useful, but a standalone interval scanner that bypasses the Feishu monitor lifecycle, client factory, dedupe/queue contracts, shared chat history, and structured mention policy is not the narrow maintainable solution.
