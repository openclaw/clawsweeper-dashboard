## Summary

Keep open. The PR targets a real Feishu missed-message gap, but current main does not implement the proposed recovery behavior and the branch is not mergeable as-is: the added module is unwired, imports a non-exported dedupe helper, assumes a client field current resolved accounts do not have, uses the wrong chat/message SDK path, reconstructs event metadata from the wrong fields, and bypasses shared monitor state. Prior review comments raised the same blockers.

## What This Changes

The PR adds a new Feishu message recovery module that periodically polls recent group messages, skips recorded message IDs, replays apparent bot mentions, and exposes start/stop/stats/manual recovery entry points.

## Best Possible Solution

Rework or supersede this with a Feishu-owned implementation that hooks recovery into the existing monitor lifecycle, uses the current createFeishuClient plus im.chat.list/im.message.list contracts, reconstructs events from the current message-list item shape, dispatches through the shared claim/debounce/sequential-queue/history path, honors group and structured mention policy, documents any required scopes, and adds focused regression plus live API proof where feasible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR diff imports a symbol current main does not export, has no lifecycle caller, reads a non-existent account.client field, and uses SDK/message shapes that conflict with current main code. No live Feishu account is needed to reproduce those merge blockers.

## Solution Assessment

No. The direction addresses a valid Feishu gap, but this patch is not the narrowest maintainable solution because it adds a side polling module outside the monitor lifecycle and does not reuse current client, message-list, group-policy, dedupe, queue, or shared history contracts.
