## Summary

Keep open: the Feishu catchup direction is useful, but this patch is not mergeable against current main because it is a standalone scanner that is never started, imports unavailable dedupe helpers, uses a nonexistent account client, calls APIs that do not match the pinned Lark SDK, and lacks live Feishu proof.

## What This Changes

The PR adds a new Feishu message recovery module that polls recent chat history and replays missed bot-mention messages after gateway downtime.

## Best Possible Solution

Implement Feishu catchup inside the monitor lifecycle behind an explicit opt-in config and policy surface, using `createFeishuClient`, SDK `im.chat.list`/`im.message.list`, structured mentions, existing dedupe/debounce/queueing, shared chat history, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection against current main shows the module is never started, imports unavailable helpers, reads a nonexistent client, and calls SDK methods/fields that do not match the pinned contract. The original live Feishu reconnect loss was not reproduced in a real Feishu setup here.

## Solution Assessment

No. The feature direction is useful, but a standalone interval scanner that bypasses the monitor lifecycle, client factory, dedupe/debounce/queue contracts, shared chat history, structured mention policy, and an opt-in safety surface is not the narrow maintainable solution.
