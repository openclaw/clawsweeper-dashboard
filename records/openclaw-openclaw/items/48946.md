## Summary

Keep this PR open: the Feishu catchup goal is useful, but the current patch is not wired into the monitor lifecycle, does not match the current Feishu/Lark SDK or local dedupe contracts, and still lacks real Feishu behavior proof.

## What This Changes

The PR adds a new Feishu message recovery module that polls chat history and tries to replay bot-mentioned messages missed while the gateway was offline.

## Best Possible Solution

Rework catchup inside Feishu's monitor lifecycle behind an explicit opt-in config, using `createFeishuClient`, SDK `im.chat.list`/`im.message.list`, structured mentions, existing dedupe/debounce/queueing, shared chat history, cursor/retry bounds, and redacted live Feishu proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the module is never started, imports non-exported dedupe helpers, assumes a nonexistent account client, and calls SDK methods that are not in the pinned Lark SDK. The original live Feishu missed-message behavior was not reproduced in a real Feishu setup here.

## Solution Assessment

No. A standalone interval scanner that bypasses the monitor lifecycle, existing queue/debounce/dedupe contracts, structured mention policy, shared history, and an opt-in safety surface is not the maintainable fix.
