## Summary

Keep this PR open for maintainer handling: the Feishu missed-message gap is valid, but this patch is not mergeable because it is not wired into the monitor lifecycle and conflicts with current Feishu client, dedupe, message-shape, queue, history, and mention-policy contracts.

## What This Changes

The PR adds a Feishu message recovery module that polls recent chats and attempts to replay missed bot-mention messages after offline periods.

## Best Possible Solution

Rework this as a Feishu-owned recovery path integrated into the existing monitor lifecycle and shared inbound handler contracts, with documented scopes and focused regression plus live API proof where feasible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR imports non-exported dedupe symbols, has no lifecycle caller, reads a non-existent `account.client`, and uses SDK/message shapes that conflict with current main.

## Solution Assessment

No. The feature direction is useful, but this patch is not the narrowest maintainable solution because it bypasses the current monitor lifecycle, client factory, message-list API, structured mention policy, dedupe claims, sequential queue, and shared chat history.

  - body: `resolveFeishuAccount` returns resolved account data, not an object with `account.client`, so casting `account.client as Lark.IM` leaves the recovery tick without a usable SDK client. Follow the current `createFeishuClient(account)` path.
  - confidence: 0.96
- **[P1] Use the current chat and message list APIs:** `extensions/feishu/src/feishu-message-recovery.ts:94-110`
  - body: The code calls `listChatMembers` for chat discovery and `listChatMessage` with `sort`/`order`/`limit`, but current Feishu code uses `client.im.chat.list` for chats and `client.im.message.list` with `sort_type`/`page_size` for message listing. Recovery will fail before it can fetch usable messages.
  - confidence: 0.93
- **[P1] Populate recovered event metadata from message items:** `extensions/feishu/src/feishu-message-recovery.ts:147-153`
  - body: Recovered events read `chat_id`, `chat_type`, and `message_type` from `msg.body`, but current message-list handling treats `chat_id`/`chat_type` as item-level fields and the message type as `msg_type`. This falls back to empty or wrong routing metadata for recovered group messages.
  - confidence: 0.91
- **[P2] Dispatch recovered messages through shared handler state:** `extensions/feishu/src/feishu-message-recovery.ts:162-169`
  - body: Calling `handleFeishuMessage` directly with a fresh `new Map()` skips the monitor's shared conversation history and per-chat sequential queue, so recovered messages can lose context and race active inbound handling.
  - confidence: 0.89
- **[P2] Use structured mentions for recovery filtering:** `extensions/feishu/src/feishu-message-recovery.ts:132-137`
  - body: The fallback `content.includes("@")` treats emails, pasted handles, and broadcast mentions as recovery candidates, which conflicts with Feishu's documented bot-mention policy and can replay unrelated group messages when mention requirements are disabled or misclassified.
  - confidence: 0.86
