## Summary

Keep open, but this patch is not mergeable as-is: current main still lacks Feishu catch-up, while this PR's new module is unwired, mismatches current Feishu account/client/dedup contracts, calls the wrong Lark SDK APIs, and has no real Feishu behavior proof.

## What This Changes

The PR adds a Feishu message recovery module that periodically polls recent chat messages, checks dedupe state, and replays messages it considers missed through `handleFeishuMessage`.

## Best Possible Solution

Implement Feishu catch-up inside the existing monitor lifecycle with the current Lark client, shared dedupe and chat history paths, bounded backfill policy, regression coverage, and real Feishu proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the added file imports non-exported dedup helpers, is not started by `monitor.account.ts`, and calls APIs that do not match the pinned Lark SDK types. I did not establish a live Feishu gateway reproduction for the original missed-message feature request.

## Solution Assessment

No. The requested direction is useful, but a standalone 30-second polling module is not the narrowest maintainable fix until it is integrated with the monitor lifecycle, current SDK calls, shared history/dedupe state, and an explicit backfill policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Wire recovery into the monitor lifecycle:** `extensions/feishu/src/feishu-message-recovery.ts:42`
  - body: The new module is never imported or called by the Feishu monitor, so `startMessageRecovery` never runs and missed messages are still not recovered after reconnect. The monitor currently only registers event handlers and starts webhook/WebSocket monitoring.
  - confidence: 0.97
- **[P1] Use exported dedup helpers:** `extensions/feishu/src/feishu-message-recovery.ts:14`
  - body: This import references `hasRecordedMessage` and `hasRecordedMessagePersistent`, but current `dedup.ts` does not export either symbol. The added file will fail to build before any recovery logic can run.
  - confidence: 0.98
- **[P1] Create a Lark client from the resolved account:** `extensions/feishu/src/feishu-message-recovery.ts:89-90`
  - body: `resolveFeishuAccount` returns a `ResolvedFeishuAccount` with credentials and config, not an object with a `client` property. This path should use the existing `createFeishuClient(account)` contract before calling Feishu APIs.
  - confidence: 0.98
- **[P1] List chats through `client.im.chat.list`:** `extensions/feishu/src/feishu-message-recovery.ts:93-97`
  - body: `listChatMembers` is not the chat-discovery API on the pinned Lark client, and it cannot enumerate the bot's chats. Recovery will fail before it has chat IDs to scan.
  - confidence: 0.96
- **[P1] Fetch history through `client.im.message.list`:** `extensions/feishu/src/feishu-message-recovery.ts:103-113`
  - body: The pinned Lark SDK exposes history retrieval as `client.im.message.list` with `sort_type` and `page_size` parameters; `listChatMessage` with `sort`, `order`, and `limit` does not match that contract.
  - confidence: 0.95
- **[P1] Populate recovered events from root message fields:** `extensions/feishu/src/feishu-message-recovery.ts:147-153`
  - body: List-message items expose routing fields such as `chat_id`, `chat_type`, and `msg_type` on the item root, while `body` carries content. Reading those fields from `msg.body` creates empty or wrong chat metadata and routes recovered group messages incorrectly.
  - confidence: 0.95
- **[P2] Drop the broad at-sign fallback:** `extensions/feishu/src/feishu-message-recovery.ts:132-137`
  - body: `content.includes("@")` treats email addresses, pasted handles, and `@all` as bot mentions, so recovery can reprocess unrelated historical group messages and send unintended replies. Use structured mention data or the existing bot-specific mention checks.
  - confidence: 0.9
- **[P2] Reuse the monitor's shared chat history:** `extensions/feishu/src/feishu-message-recovery.ts:157-164`
  - body: Passing a fresh `new Map()` for every recovered message discards the group conversation context that `monitorSingleAccount` keeps in its shared `chatHistories` map, so recovered replies lose prior turns.
  - confidence: 0.88
