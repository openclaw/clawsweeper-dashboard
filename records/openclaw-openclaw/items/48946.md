## Summary

Keep open, but this PR is not merge-ready: source review confirms the added Feishu recovery module is unintegrated, does not match current Feishu runtime contracts, and lacks live Feishu catchup proof.

## What This Changes

This PR adds `extensions/feishu/src/feishu-message-recovery.ts`, a periodic Feishu chat-history polling module intended to replay missed bot-mention messages after gateway downtime.

## Best Possible Solution

Rework catchup inside Feishu's monitor lifecycle behind an explicit opt-in config, using the client factory, SDK chat/message list contracts, structured mentions, existing dedupe/debounce/queueing, shared chat history, cursor/retry limits, and live Feishu proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows no caller starts the module, its imports and account/client assumptions do not match current main, and its SDK calls/field paths do not match the pinned Lark SDK. The original live Feishu missed-message behavior was not reproduced in a real Feishu setup here.

## Solution Assessment

No. The feature direction is useful, but a standalone interval scanner that bypasses monitor lifecycle state, current dedupe/debounce/queue contracts, structured mention policy, and an opt-in safety surface is not the maintainable fix.

  - confidence: 0.96

  - body: `resolveFeishuAccount` returns a credentials/config snapshot, not an object with `account.client`, so `client` is undefined at runtime. Resolve the runtime account and pass it through `createFeishuClient(account)`.
  - confidence: 0.96
- **[P1] Call the SDK chat and message list APIs:** `extensions/feishu/src/feishu-message-recovery.ts:95-113`
  - body: The pinned Lark SDK exposes chat discovery/history through `client.im.chat.list` and `client.im.message.list` with `sort_type` and `page_size`. `listChatMembers`, `listChatMessage`, `sort`, `order`, and `limit` do not match that contract.
  - confidence: 0.94
- **[P1] Populate recovered events from message fields:** `extensions/feishu/src/feishu-message-recovery.ts:147-153`
  - body: Message-list items expose `chat_id`, `msg_type`, `body.content`, and `mentions` on the item shape, not under `body.chat_id`, `body.chat_type`, or `body.message_type`, so recovered messages route with empty or wrong metadata.
  - confidence: 0.92
- **[P2] Filter recovered history with structured mentions:** `extensions/feishu/src/feishu-message-recovery.ts:132-135`
  - body: The raw `content.includes("@")` fallback treats emails, pasted handles, and broadcast mentions as bot-directed. Use Feishu `mentions` and existing mention helpers so the scanner does not replay unrelated history.
  - confidence: 0.88
- **[P2] Dispatch through shared monitor state:** `extensions/feishu/src/feishu-message-recovery.ts:156-163`
  - body: Directly calling `handleFeishuMessage` with a fresh `Map` skips the monitor's shared chat history and inbound debounce/processing path, so recovered messages can lose context and race live inbound handling.
  - confidence: 0.89
