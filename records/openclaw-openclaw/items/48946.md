## Summary

Keep open: the Feishu catchup behavior is not implemented on current main, but this PR is not mergeable because the added module is unintegrated, does not match current Feishu/Lark contracts, lacks live Feishu proof, and can replay unrelated group-history content.

## What This Changes

Adds a Feishu message recovery module that polls recent group chat messages every 30 seconds and tries to replay missed bot mentions after gateway downtime.

## Best Possible Solution

Rework Feishu catchup inside the monitor lifecycle behind an explicit opt-in config, using the client factory, SDK list contracts, structured mentions, existing dedupe/debounce/queueing, shared history, cursor/retry limits, and live Feishu proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the module is never started, imports non-exported dedupe helpers, assumes a nonexistent `account.client`, and calls SDK APIs/fields that do not match the pinned Lark SDK. The original live Feishu missed-message behavior was not reproduced here.

## Solution Assessment

No. The feature direction is useful, but a standalone interval scanner that bypasses the monitor lifecycle, shared dispatch state, structured mention policy, dedupe/debounce queues, and an opt-in safety surface is not the maintainable fix.
