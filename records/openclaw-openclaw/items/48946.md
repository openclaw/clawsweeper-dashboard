## Summary

Keep this PR open, but it is not mergeable as-is. The feature goal is useful for Feishu users, yet the current branch adds an unintegrated standalone scanner with type/runtime breakages, an over-broad history replay filter, and no live Feishu catch-up proof.

## What This Changes

The PR adds `extensions/feishu/src/feishu-message-recovery.ts`, a standalone Feishu polling service intended to fetch recent group history every 30 seconds and replay bot mentions missed while the gateway was offline.

## Best Possible Solution

Implement Feishu catch-up inside the existing monitor lifecycle behind explicit opt-in config and policy, using `createFeishuClient`, SDK `im.chat.list`/`im.message.list`, structured mentions, current dedupe/debounce/queueing, shared chat history, and live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection against current main shows the module is never started, imports unavailable helpers, reads a nonexistent client, and calls SDK methods/fields outside the pinned contract. The original live Feishu reconnect loss was not reproduced in a real Feishu setup here.

## Solution Assessment

No. The feature direction is useful, but this standalone interval scanner bypasses the monitor lifecycle, client factory, dedupe/debounce/queue contracts, shared chat history, structured mention policy, and an opt-in safety surface.
