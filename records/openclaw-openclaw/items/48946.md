## Summary

Keep open for human rework: this branch does not satisfy the Feishu monitor, client, dedupe, or message-shape contracts, and it lacks live Feishu recovery proof, so it is not safe to merge or auto-repair as-is.

## What This Changes

This PR adds a standalone Feishu message-recovery module that polls recent chats/messages every 30 seconds and replays bot-mention messages.

## Best Possible Solution

Implement Feishu missed-message catchup inside the Feishu monitor lifecycle using shared inbound handler contracts, SDK list APIs, structured mentions, dedupe, queueing, and shared chat history, then prove it with redacted live Feishu recovery output.

## Reproduction Assessment

Yes. Static source review gives a high-confidence reproduction path for the PR defects: applying this patch would import non-exported helpers, never start the recovery service, read a non-existent `account.client`, and call SDK/message shapes that conflict with current main and the Lark SDK types.

## Solution Assessment

No. The feature direction is useful, but this patch bypasses the current lifecycle, client factory, structured mention policy, dedupe claims, sequential queueing, and shared conversation history, so it is not the narrow maintainable solution.

  - confidence: 0.96

  - body: `resolveFeishuAccount` returns credentials/config, not an object with `account.client`; casting `account.client as Lark.IM` leaves recovery without a usable SDK client. Follow the existing `createFeishuClient(account)` path.
  - confidence: 0.96
- **[P1] Use the current chat and message list APIs:** `extensions/feishu/src/feishu-message-recovery.ts:95-113`
  - body: The Lark SDK exposes chat discovery and history as `client.im.chat.list(...)` and `client.im.message.list(...)`; `listChatMembers`/`listChatMessage` with `sort`/`order`/`limit` do not match the dependency contract and will not fetch usable messages.
  - confidence: 0.94
- **[P2] Filter recovered messages with structured mentions:** `extensions/feishu/src/feishu-message-recovery.ts:132-135`
  - body: The raw `content.includes("@")` fallback treats emails, pasted handles, and broadcast mentions as bot-directed recovery candidates. Use the structured `mentions` data and existing bot-mention rules to avoid replaying unrelated group messages.
  - confidence: 0.86
- **[P1] Populate recovered events from message-list items:** `extensions/feishu/src/feishu-message-recovery.ts:141-153`
  - body: Message-list items expose identifiers such as `chat_id`, `msg_type`, `body.content`, and `mentions` on the item shape, not under `msg.body.chat_id/chat_type/message_type`, so recovered events will be malformed or routed with empty/wrong chat metadata.
  - confidence: 0.92
- **[P2] Dispatch recovered messages through shared handler state:** `extensions/feishu/src/feishu-message-recovery.ts:156-163`
  - body: Calling `handleFeishuMessage` directly with a fresh `new Map()` skips the monitor's shared conversation history and serialized inbound queue, so recovered messages can lose context and race active inbound handling.
  - confidence: 0.89
