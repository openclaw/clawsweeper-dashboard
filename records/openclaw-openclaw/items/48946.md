## Summary

Keep open for maintainer/Feishu-owner review: the recovery idea is useful, but this PR is not mergeable because the added module is not wired into the monitor lifecycle, does not match current Feishu SDK/account/dedupe contracts, and lacks live Feishu recovery proof.

## What This Changes

The PR adds a new Feishu message recovery module that polls recent chat history every 30 seconds and replays unprocessed bot-mention messages after gateway downtime.

## Best Possible Solution

Implement Feishu catchup inside the monitor lifecycle behind an explicit safe config/policy surface, using SDK list APIs, structured mentions, shared dedupe/debounce/queueing, shared chat history, and redacted live proof.

## Reproduction Assessment

Yes for the PR defects: source inspection against current main shows the module is never started, imports unavailable dedupe helpers, reads a nonexistent account.client, and calls SDK methods/fields that do not match current contracts. The original live Feishu reconnect loss was not reproduced in a real Feishu setup here.

## Solution Assessment

No. The feature direction is useful, but a standalone interval scanner that bypasses the Feishu monitor lifecycle, client factory, dedupe/debounce/queue contracts, shared chat history, and structured mention policy is not the narrow maintainable solution.
