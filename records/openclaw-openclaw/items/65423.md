## Summary

This PR should stay open, but it is not merge-ready. The central reliability idea is still not implemented on current main, while the branch needs to preserve current-main auth-order semantics, test production behavior, and add real after-fix proof.

## What This Changes

The PR adds a private Fisher-Yates shuffle so spawned embedded-runner subagent runs randomize automatic auth-profile candidates, plus a new shuffle-focused test file.

## Best Possible Solution

Move spawned-run candidate ordering into a production-owned helper that disperses automatic profiles while preserving locked, provider-preferred, plugin-harness, explicit-order, and cooldown semantics, then cover that path with focused regression tests and real concurrent-run proof.

## Reproduction Assessment

Yes for the source-level behavior: current main builds an ordered `profileCandidates` list and the auth controller starts each run at index 0. I did not live-reproduce the provider-side 429 cascade in a real concurrent subagent setup.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the patch should be rebased onto current main, preserve provider-preferred and plugin-harness ordering semantics, and test production candidate selection instead of a copied helper.
