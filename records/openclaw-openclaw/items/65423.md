## Summary

Keep this PR open, but it still needs changes before merge: the direction is useful, yet the branch is stale against current auth-profile selection semantics, its tests do not exercise production code, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a Fisher-Yates shuffle in the embedded PI runner so spawned subagent runs randomize automatic auth-profile candidates, plus a new shuffle-focused test file.

## Best Possible Solution

Add a production-owned candidate-ordering helper for embedded runs that disperses spawned automatic profiles while preserving locked, user-preferred, provider-preferred, cooldown, plugin-harness, and explicit-order semantics, then test that helper or the embedded selection path directly.

## Reproduction Assessment

Yes for the source-level behavior: current main resolves ordered profiles and passes `providerOrderedProfiles` unchanged into `profileCandidates` when no profile is locked. I did not live-reproduce the external 429 cascade.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the patch should be rebased onto current main, preserve provider-preferred ordering, and test production candidate selection instead of a local helper copy.
