## Summary

Keep open. Current main still builds embedded-run auth candidates in a deterministic order for spawned runs, so the goal is not obsolete, but this branch needs revision before merge because current main now has provider-preferred auth selection and the added tests exercise a copied helper rather than production behavior. The earlier Greptile and ClawSweeper comments are directionally correct: useful narrow work, not ready as-is.

## What This Changes

This PR adds a Fisher-Yates helper in the embedded runner, uses `params.spawnedBy` to shuffle automatic auth-profile candidates for subagent runs, and adds a standalone shuffle test file.

## Best Possible Solution

Implement a production-owned candidate-ordering helper for embedded runs that disperses spawned automatic runs across healthy profiles while keeping user-locked profiles, provider-preferred profiles, cooldown handling, and documented auth-order semantics intact, then cover that helper or the embedded selection path directly.

## Reproduction Assessment

Yes for the source-level behavior: with multiple eligible profiles and `spawnedBy` set, current main still starts `profileCandidates` from the same ordered list. The provider-specific cascading 429 symptom is plausible but not live-reproduced here.

## Solution Assessment

No, not as currently proposed. The direction is narrow and useful, but the patch should be rebased onto current main, preserve provider-preferred selection, test production candidate ordering, and add the required changelog entry.
