## What This Changes

The PR changes `extensions/telegram/src/message-cache.ts` so normalized Telegram cached message nodes require `messageId: string` instead of inheriting the optional public reply-chain field.

## Best Possible Solution

Refresh this focused PR on current main, rerun Dependency Guard plus the Telegram type/lint/message-cache checks, and land the cache-only type repair if exact-head validation stays green.

## Reproduction Assessment

Yes. Source inspection shows the shared reply-chain entry type allows optional `messageId`, while Telegram cache normalization rejects id-less messages and all cache operations key normalized nodes by a string id; no local typecheck was run because this review is read-only.

## Solution Assessment

Yes, after branch refresh. Tightening only the Telegram cache node type is narrower than making cache callers handle an id-less normalized state that the cache already rejects, and it leaves the public `ReplyChain` contract unchanged.
