## Summary

Keep open: this is a focused, correct PR for an unmerged Telegram cache type regression; source, dependency types, supplied terminal proof, and live CI status support the change, and I found no blocking review finding.

## What This Changes

The PR changes the Telegram message-cache node type so normalized cached nodes keep a required `messageId` while retaining the richer cached `sourceMessage` shape.

## Best Possible Solution

Land the focused type repair after the normal maintainer merge step so the Telegram reply-chain cache keeps its current behavior without widening cache callers to handle missing ids.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run the extension tsgo commands from the PR body against current main, where the cache node alias inherits optional `messageId` from `MsgContext["ReplyChain"]` even though normalized cache nodes always have a concrete id.

## Solution Assessment

Yes. Tightening the Telegram cache node type at the normalized-cache boundary is the narrowest maintainable fix and avoids spreading impossible `undefined` handling through cache upsert and lookup callers.
