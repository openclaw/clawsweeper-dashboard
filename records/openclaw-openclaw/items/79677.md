## Summary

Keep open: current main still rejects or drops the requested `qwen` and `qwen-chat-template` `thinkingFormat` values, and the existing Qwen/vLLM wrappers do not make arbitrary JSON provider compat values valid. Upstream pi docs describe both formats, so this is a plausible narrow enhancement rather than an already-implemented cleanup. ([github.com](https://github.com/badlogic/pi-mono/blob/main/packages/coding-agent/docs/models.md?utm_source=openai))

## What This Changes

Allow `qwen` and `qwen-chat-template` as `compat.thinkingFormat` values in JSON model config for local OpenAI-compatible Qwen backends.

## Best Possible Solution

Add generic OpenAI-compatible Qwen thinking compat support across config validation, catalog normalization, and the OpenAI-completions request builder, while preserving the existing Qwen and vLLM plugin-specific paths.

## Reproduction Assessment

Yes from source inspection: a JSON model config with `compat.thinkingFormat: "qwen"` or `"qwen-chat-template"` would fail schema validation or be dropped by catalog normalization on current main. I did not run a live backend, but the allowlists are explicit.

## Solution Assessment

No, a schema-only fix is likely incomplete because the OpenAI-completions transport also needs to map these compat values to top-level `enable_thinking` or `chat_template_kwargs.enable_thinking`. The narrow maintainable path is to update schema/types/normalization, transport behavior, tests, and docs together.
