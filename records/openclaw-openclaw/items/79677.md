## Summary

Keep open: current main rejects the requested `thinkingFormat` enum values, but this is not a safe one-line cleanup because a recent maintainer refactor deliberately moved Qwen thinking payloads into provider-owned Qwen/vLLM plugin params instead of generic core compat.

## What This Changes

Allow JSON model compat configs to use `thinkingFormat: "qwen"` and `"qwen-chat-template"` for local Qwen OpenAI-compatible backends.

## Best Possible Solution

Decide the ownership boundary, then either re-enable the two Qwen formats across schema/types/catalog/runtime tests or document and provide a plugin-owned local-Qwen parameter path for llama.cpp backends.

## Reproduction Assessment

Yes for the config gate: current main directly rejects the requested values in `ModelCompatSchema` and strips them in catalog normalization. I did not run a live llama.cpp request, but the source path for the schema rejection is high-confidence.

## Solution Assessment

Unclear. The requested one-line schema widening conflicts with the current provider-owned Qwen/vLLM design and would need matching catalog and runtime coverage to be maintainable.
