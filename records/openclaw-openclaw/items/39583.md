## Summary

Keep open: current main still lacks stringified content-block parsing, but this external PR is not merge-ready because its guard can drop unrelated JSON strings and it lacks real vLLM proof.

## What This Changes

The PR adds parsing for JSON-stringified chat content arrays in `extractTextFromChatContent` plus focused helper tests for string, array, stringified, mixed, and edge cases.

## Best Possible Solution

Ship a rebased helper fix that recognizes only valid stringified chat content arrays, preserves plain and unrelated JSON strings plus current coercion hardening, adds regression coverage, and includes redacted vLLM runtime proof.

## Reproduction Assessment

Yes at source level: current main returns a raw JSON string for a stringified `[{type:"text", text:"hello"}]` payload because the string branch never parses JSON, while the array branch extracts `hello`. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified chat blocks is the right narrow direction, but the guard must distinguish real chat content blocks from unrelated JSON arrays and preserve current main's coercion hardening.
