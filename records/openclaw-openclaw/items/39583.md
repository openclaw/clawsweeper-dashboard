## Summary

Keep open. Current main and the latest release still return JSON-stringified chat content-block arrays through the plain string path, so the PR is not obsolete; however the current PR guard is too broad and can drop non-chat JSON strings that happen to contain typed objects.

## What This Changes

The PR changes `extractTextFromChatContent` to parse JSON-stringified chat content-block arrays and adds unit coverage for plain strings, arrays, stringified arrays, mixed blocks, and fallback cases.

## Best Possible Solution

Keep the shared helper as the normalization boundary, but parse strings only when the parsed value has a recognizable chat content-block shape, mirror the real-array extraction semantics for text and mixed media/text blocks, preserve arbitrary JSON strings, and add regression tests for text-only blocks, mixed blocks, invalid JSON, empty arrays, and typed non-chat JSON arrays.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `extractTextFromChatContent(JSON.stringify([{ type: 'text', text: 'hello' }]))` on current main: the string branch returns the normalized raw JSON string, while passing the same value as an array extracts `hello`.

## Solution Assessment

No, not as-is. Parsing stringified content-block arrays in `extractTextFromChatContent` is the right narrow direction, but the PR should use a stronger chat-block shape guard and be rebased onto the current helper so it does not regress arbitrary JSON string handling or newer sanitizer coercion behavior.
