## Summary

Keep this PR open: it targets a real vLLM/OpenAI-compatible content extraction bug, but the current patch can drop unrelated JSON strings and still lacks after-fix real behavior proof.

## What This Changes

The PR parses JSON-stringified chat content-block arrays in `src/shared/chat-content.ts` and adds focused tests in `src/shared/chat-content.test.ts`.

## Best Possible Solution

Rebase over current main, parse only recognized chat content-block arrays while preserving plain-string behavior and coercion hardening, add the missing regression test, and attach redacted vLLM/OpenAI-compatible runtime proof.

## Reproduction Assessment

Yes at source level: current main returns a JSON-stringified `[{"type":"text","text":"hello"}]` as plain string while real arrays extract `hello`. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified chat blocks is the right direction, but the detection must require a recognized chat-content shape before leaving plain-string handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new guard accepts any parsed JSON array whose objects contain a `type` key. A string like `'[{"type":"event","payload":"x"}]'` now recurses into array handling and returns null instead of preserving the original text, and callers that skip null text can silently omit that content. Require a recognized chat content-block shape before switching away from plain-string handling.
  - confidence: 0.9
