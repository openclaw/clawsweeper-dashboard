## What This Changes

The PR parses JSON-stringified chat content block arrays in `extractTextFromChatContent` and adds focused text-extraction tests.

## Best Possible Solution

Rebase onto current main, add a narrow stringified content-block parser that trims only the parse candidate and recognizes real chat content block shapes, preserve current coercion hardening, cover the regressions, and attach redacted runtime proof.

Yes at source level: current main returns a stringified `[{"type":"text","text":"hello"}]` payload through the string branch unchanged, while true array content extracts text. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified blocks in the shared helper is a plausible narrow fix, but this diff needs trimming, stricter chat-block recognition, a rebase, and real runtime proof before it is the best solution.

  - body: The linked vLLM report shows nested content with newlines after the serialized array.   - confidence: 0.9
