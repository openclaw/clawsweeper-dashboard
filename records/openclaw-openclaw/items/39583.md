## What This Changes

The PR adds JSON-stringified content-block parsing to `extractTextFromChatContent` and new shared helper unit coverage.

## Best Possible Solution

Keep the transport-level structured-content normalization, and only add helper/string parsing if it is restricted to supported chat block shapes and backed by redacted vLLM or OpenAI-compatible runtime proof.

## Reproduction Assessment

Yes, source-reproducible for the helper path: current main returns a JSON-stringified content-block array string as plain text, while real array content already extracts text. I did not run tests because this review is read-only.

## Solution Assessment

No as currently written. Parsing stringified content blocks can be the right narrow fix, but the PR needs a stricter supported-shape guard and real runtime proof, while current main already handles structured provider arrays at the transport layer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat blocks, but the array path only extracts `type === "text"`. Inputs like `[{"type":"event","payload":"x"}]` now become `null` instead of preserving the original string, so callers that omit null text can silently drop content; require a supported chat content shape before recursing.
