## Summary

This PR should stay open because current main still lacks the vLLM JSON-stringified content-block parsing, but the patch needs a tighter detection guard and real vLLM behavior proof before merge.

## What This Changes

The PR adds JSON-stringified content-block parsing to `extractTextFromChatContent` and adds unit coverage for plain strings, arrays, stringified arrays, mixed blocks, and edge cases.

## Best Possible Solution

Keep the focused content-block parser, integrate it with the current helper, narrow detection to recognizable chat content block shapes, and require real vLLM/OpenAI-compatible response proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main returns a JSON-stringified content-block array as a normalized plain string, while the array path already extracts text blocks. I did not run tests because this review is read-only and avoids commands that may create artifacts.

## Solution Assessment

No as currently written. Parsing stringified content blocks is the right narrow direction, but the `type`-only guard should be tightened so ordinary JSON string content is preserved unless it has a recognizable chat content-block shape.
