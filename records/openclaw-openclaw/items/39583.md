## Summary

Keep open: current main still treats JSON-stringified content-block arrays as plain strings, so the linked vLLM bug is not implemented. The PR is useful but cannot merge as-is because its JSON-array heuristic can drop valid plain string content and the external contributor has not supplied after-fix vLLM runtime proof.

## What This Changes

The PR adds parsing for JSON-stringified chat content-block arrays in `extractTextFromChatContent` and adds helper tests for plain strings, arrays, stringified arrays, and mixed content blocks.

## Best Possible Solution

Update the shared extractor to parse only recognized JSON-stringified chat content arrays, preserve plain strings and unrelated JSON arrays, keep current coercion hardening, add regression coverage, and attach redacted vLLM runtime proof.

## Reproduction Assessment

Yes at source level: current main returns a raw JSON string for `JSON.stringify([{type:"text",text:"hello"}])` because the string branch never parses JSON, while the array branch extracts `hello`. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified chat blocks is the right narrow direction, but the PR must use a stronger chat-content shape check and preserve current main's coercion hardening before merge.
