## Summary

Keep open: the PR addresses a real vLLM/openai-completions text extraction bug that still exists on current main, but the latest patch is not merge-ready because its `type`-only JSON-array detection can drop arbitrary JSON string content and the branch is currently dirty against main.

## What This Changes

The branch teaches `extractTextFromChatContent` to parse JSON-stringified content-block arrays and adds colocated unit tests for string, array, stringified-array, mixed-block, sanitizer, and join behavior.

## Best Possible Solution

Land a rebased, narrow helper change that recognizes stringified chat content block arrays without consuming arbitrary JSON strings, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes, source-level. On current main, `extractTextFromChatContent` returns all string inputs after sanitize/normalize, so a JSON-stringified text-block array is preserved as JSON instead of using the existing array extraction path.

## Solution Assessment

No. The direction is right, but the current guard is too broad and the branch is dirty against current main; the maintainable fix is a stricter chat-block shape integrated with the current coerce/sanitize helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON array detection before recursing:** `src/shared/chat-content.ts:21-22`
  - body: The new guard treats any JSON array of objects with a `type` key as chat content. A literal string such as `[{"type":"event","payload":"x"}]` now recurses into the array extractor and returns `null` because there are no text blocks, whereas string inputs should be preserved as text unless they are convincingly chat content blocks. Please require a stronger content-block shape before switching away from plain-string handling.
  - confidence: 0.9
