## Summary

Keep open: this PR targets a real source-reproducible vLLM prompt normalization bug, but the PR head still has a guard regression that can drop unrelated JSON string content, and current main does not already implement the fix.

## What This Changes

The PR parses JSON-stringified chat content-block arrays inside `extractTextFromChatContent` and adds shared helper tests for string, array, stringified-array, and fallback inputs.

## Best Possible Solution

Keep the shared helper as the normalization boundary, parse only recognizable chat content-block JSON strings, preserve unrelated JSON strings, mirror mixed media/text array extraction, add regression tests, and include the required changelog entry before merge.

## Reproduction Assessment

Yes, from source inspection. On current main, `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows the string branch and preserves normalized JSON text, while the equivalent array path extracts `hello`.

## Solution Assessment

No, not as the PR currently stands. Parsing stringified content-block arrays in `extractTextFromChatContent` is the right narrow boundary, but the guard needs a stronger chat-block shape so arbitrary JSON strings are preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require chat-block shape before parsing JSON strings:** `src/shared/chat-content.ts:21-23`
  - body: The new string parsing heuristic treats any JSON array of objects with a `type` key as chat content blocks. A string like `[{"type":"event","payload":"x"}]` now recurses into array handling and returns `null` because there are no text blocks, whereas the prior helper preserved that string; require recognizable chat content-block shape before leaving the plain-string path.
  - confidence: 0.89
