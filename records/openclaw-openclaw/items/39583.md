## Summary

Keep open: current main still has the reported source-level behavior, and this PR is a plausible targeted fix, but the current patch can drop arbitrary JSON-string content and lacks real vLLM behavior proof before merge.

## What This Changes

The PR changes `src/shared/chat-content.ts` to parse JSON-stringified arrays with object `type` fields through the existing content-block extractor and adds colocated unit tests for plain strings, arrays, stringified arrays, mixed content, and edge cases.

## Best Possible Solution

Land a rebased helper change that only recognizes convincing stringified chat content-block arrays, preserves arbitrary JSON strings, adds focused regression coverage, and verifies the real vLLM path with redacted evidence.

## Reproduction Assessment

Yes, source-level. On current main, `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows the plain string branch instead of the existing array extraction path, so it preserves the JSON text.

## Solution Assessment

No. The direction is right, but the current `type`-only guard is too broad and the branch is stale; the safer fix is a stricter chat-block parser plus regression tests and real vLLM proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Require a real chat-block shape before parsing:** `src/shared/chat-content.ts:21-22`
