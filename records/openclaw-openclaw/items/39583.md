## Summary

Keep open: the PR addresses a real remaining vLLM/chat-content bug, but current main has moved on and the proposed detection guard still has a concrete data-loss edge case for JSON strings that are not chat content blocks.

## What This Changes

The PR parses JSON-stringified chat content arrays in `extractTextFromChatContent` and adds unit coverage for plain strings, arrays, stringified arrays, and mixed content blocks.

## Best Possible Solution

Port the useful stringified-content-block handling onto current main's `coerceChatContentText`-based helper with a stricter chat-block predicate, preserve arbitrary JSON strings, add regression tests, and include the required changelog entry.

## Reproduction Assessment

Yes, from source inspection. On current main, a JSON-stringified content-block array enters the string branch and is returned as normalized plain text, while equivalent real arrays are extracted through the block loop.

## Solution Assessment

No, not as-is. Parsing stringified content blocks is the right direction, but the latest PR guard is too broad and should distinguish chat content blocks from arbitrary JSON arrays with a `type` field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-chat JSON arrays when probing content blocks:** `src/shared/chat-content.ts:21-23`
  - body: The new string path recurses on any parsed array whose items merely have a `type` key. A JSON string like `[{"type":"event","payload":"x"}]` previously returned the original text, but this patch would call the array path and return null because there are no `type === "text"` blocks, so callers can silently drop content. Require a stronger chat-content shape before switching away from plain string handling.
  - confidence: 0.88
