## Summary

Keep open: current main still lacks stringified content-block extraction, but this PR needs a stricter JSON-block guard and real vLLM behavior proof before merge.

## What This Changes

The branch makes `extractTextFromChatContent` parse JSON-stringified arrays of content blocks and adds unit coverage for plain strings, arrays, stringified arrays, mixed blocks, and edge cases.

## Best Possible Solution

Port a conservative parser into current `src/shared/chat-content.ts` that only unwraps recognized chat content-block arrays, preserves arbitrary JSON strings, covers false-positive arrays and mixed media in tests, and verifies a real vLLM openai-completions response.

## Reproduction Assessment

Yes, at source level: on current main, a JSON string like `[{"type":"text","text":"hello"}]` stays on the plain-string branch and would be returned as raw JSON instead of extracted text. I did not see live vLLM proof in the PR.

## Solution Assessment

No, not as written. The shared helper is the right place, but the PR's `type`-only detection should be narrowed so non-chat JSON strings are preserved instead of becoming null.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat content blocks. A plain string such as `[{"type":"event","payload":"x"}]` would now recurse into the array path and return `null`, so callers that drop null text silently lose valid string content; require a real text block or a stronger known content-block shape before switching away from plain-string handling.
