## Summary

Keep open: current main still lacks the vLLM/stringified-content fix, but this PR should not merge as-is because the parser is top-level-only, can drop unrelated JSON strings, lacks real vLLM proof, and is currently conflicting.

## What This Changes

The PR adds JSON-array parsing to `extractTextFromChatContent` for stringified content blocks and adds focused unit tests.

## Best Possible Solution

Refresh this into a conservative shared parser that detects known chat content-block arrays, unwraps nested stringified text values used by vLLM/OpenAI-compatible paths, preserves unrelated JSON strings, and includes focused tests plus redacted live vLLM proof.

## Reproduction Assessment

Yes for source-level reproduction, but not as a live vLLM run. Current main returns top-level strings unchanged and leaves text-block `text` values unchanged, so stringified content-block arrays can remain raw text and be serialized again.

## Solution Assessment

No, not as currently written. The direction is right, but the fix should share parsing across string and text-block paths and use a narrower chat-block shape before switching away from plain-string handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Unwrap stringified blocks inside text blocks:** `src/shared/chat-content.ts:24`
  - body: The reported nested shape can arrive as an array like `[{type:"text", text:"[{...}]")}]`, but this change only parses when the whole `content` argument is a string. The existing array path will still push that `text` value as raw JSON, so the linked vLLM double-serialization case can remain visible; move the parsing into a shared helper and apply it to text block values too.
  - confidence: 0.86
- **[P2] Narrow JSON-block detection before dropping strings:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat content. A normal string such as `[{"type":"event","payload":"x"}]` will now recurse into the array path and return `null` instead of preserving the string, so require a stronger known content-block shape before parsing.
  - confidence: 0.83
