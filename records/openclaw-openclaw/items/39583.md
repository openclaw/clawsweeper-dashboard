## Summary

Keep open: current main still does not parse JSON-stringified content-block arrays, so the PR is not obsolete, but the current branch can drop unrelated typed JSON strings and still lacks real after-fix vLLM proof.

## What This Changes

The PR parses JSON-stringified chat content-block arrays in `extractTextFromChatContent` and adds unit coverage for plain, array, stringified, mixed, sanitizer, and join cases.

## Best Possible Solution

Land a conservative parser that only switches to array extraction for supported chat content-block shapes, preserves unrelated JSON strings, adds regression coverage, and includes redacted vLLM/OpenAI-compatible runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main sends a JSON-stringified content-block array through the plain string path, while real array content already extracts text. I did not run tests because this review is read-only.

## Solution Assessment

No as currently written. Parsing stringified content blocks is the right narrow fix, but the branch needs a stricter supported-shape guard and real vLLM runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat blocks, but the array path only extracts `type === "text"`.
