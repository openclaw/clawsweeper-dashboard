## Summary

Keep open: current main still does not parse vLLM-style JSON-stringified chat content arrays, and this PR is a plausible fix path, but the PR head has a blocking guard regression and lacks required real behavior proof.

## What This Changes

This PR adds JSON parsing for stringified chat content-block arrays in `extractTextFromChatContent` and adds focused tests for that helper.

## Best Possible Solution

Rework the PR to parse only recognized chat content-block arrays, preserve current main's coercion hardening, cover unrelated `type`-key JSON arrays, and attach redacted vLLM runtime proof.

## Reproduction Assessment

Yes at the helper/source level: current main returns JSON-stringified arrays as plain text while real arrays extract text blocks, matching the linked vLLM nested-content symptom. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified chat blocks is the right direction, but the guard should require a recognized chat-content block shape before leaving plain-string handling.
