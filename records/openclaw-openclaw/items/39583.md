## Summary

Keep open: current main still does not handle literal JSON-stringified chat content block arrays, but this PR should not merge as-is because the latest parser heuristic can drop unrelated typed JSON strings and no real vLLM after-fix proof has been posted.

## What This Changes

The PR changes `src/shared/chat-content.ts` to parse JSON-stringified content block arrays before text extraction and adds `src/shared/chat-content.test.ts` coverage for strings, arrays, stringified arrays, mixed-content arrays, and edge cases.

## Best Possible Solution

Add a narrowly guarded parser or transport normalization for JSON-stringified OpenAI-compatible chat block strings, preserve unrelated JSON text, and back it with focused helper coverage plus redacted vLLM runtime proof.

## Reproduction Assessment

Yes for the helper path from source: current main returns string input as normalized plain text, while real arrays extract `type === "text"` blocks. I did not run tests because this review is read-only.

## Solution Assessment

No as currently written. Parsing stringified blocks is a plausible narrow fix, but the PR needs a stronger supported-chat-block guard and real vLLM proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new string parser treats any JSON array of objects with a `type` key as chat blocks. A string such as `[{"type":"event","payload":"x"}]` now recurses into the array path and returns `null`, so callers that omit null text can silently drop content that current main preserves; require a supported chat content shape before switching away from plain-string handling.
