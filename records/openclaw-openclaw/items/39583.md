## Summary

Keep open: the linked vLLM serialization bug is still source-reproducible on current main and this PR targets the right helper, but the current patch has an overbroad JSON parser guard and still lacks real vLLM behavior proof.

## What This Changes

The PR adds JSON parsing for stringified content-block arrays in `extractTextFromChatContent` and adds colocated tests for plain strings, arrays, stringified arrays, mixed content, and edge cases.

## Best Possible Solution

Land a rebased helper change that recognizes only convincing stringified chat content-block arrays, preserves non-chat JSON strings, adds regression coverage, and includes redacted real vLLM proof.

## Reproduction Assessment

Yes, source-level. On current main, `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows the plain string path and preserves the JSON text instead of using the existing array extraction path; I did not establish a live vLLM repro.

## Solution Assessment

No. Parsing stringified content-block arrays in the shared helper is the right direction, but this PR's `type`-only guard is too broad and should be narrowed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require a real chat-content block before parsing:** `src/shared/chat-content.ts:21-22`
  - body: The new guard treats any JSON array whose objects have a `type` key as chat blocks. Inputs like `[{"type":"event","payload":"x"}]` now recurse into the array path and return `null`, while current string handling preserves the original content, so callers that skip null can silently drop valid text. Require recognized chat-content block shapes, and at least one extractable text block, before leaving string handling.
  - confidence: 0.9
