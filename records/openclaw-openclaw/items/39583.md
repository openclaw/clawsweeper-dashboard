## Summary

The PR still targets a real current-main bug, but it should not merge as-is because the JSON-array detection can drop arbitrary string content and the external contributor has not provided real vLLM behavior proof.

## What This Changes

The PR modifies `src/shared/chat-content.ts` to parse JSON-stringified chat content-block arrays before text extraction and adds colocated unit tests for plain, array, stringified-array, mixed-content, and invalid-input cases.

## Best Possible Solution

Land a rebased helper change that recognizes convincing stringified chat content-block arrays, preserves arbitrary JSON strings, keeps focused regression tests, and includes redacted proof from the real vLLM path.

## Reproduction Assessment

Yes, source-level for the helper behavior. On current main, `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows the plain string branch instead of the existing content-block array extraction path, so it preserves the JSON text.

## Solution Assessment

No. The PR is pointed at the right helper, but the current `type`-only heuristic is too broad; the safer solution is a stricter chat-block parser that switches away from plain-string handling only when extraction is convincing.

## Review Findings
