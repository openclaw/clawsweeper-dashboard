## Summary

Keep open: the vLLM nested-content bug is still not handled on current main, but this branch has a blocking parser regression and still lacks after-fix real vLLM proof.

## What This Changes

The PR adds JSON-stringified chat content-block parsing to `extractTextFromChatContent` and adds colocated unit tests for plain strings, arrays, stringified arrays, mixed content, and fallback cases.

## Best Possible Solution

Land a rebased helper change that recognizes only convincing stringified chat content-block arrays, preserves arbitrary JSON strings, adds focused regression coverage, and includes redacted real vLLM proof.

## Reproduction Assessment

Yes, source-level. On current main, `JSON.stringify([{ type: "text", text: "hello" }])` follows the string branch in `extractTextFromChatContent` and never reaches the existing array text-block extractor.

## Solution Assessment

No. The direction is right, but the PR's `type`-only guard is too broad; the safer fix is a stricter chat-block parser that requires recognized content-block shape and extractable text before leaving the string path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before parsing:** `src/shared/chat-content.ts:21-22`
  - body: The type-only guard treats any stringified JSON array of objects with a `type` field as chat blocks. A payload like `[{"type":"event","payload":"x"}]` now recurses into the array extractor and returns `null`, while current string handling preserves the content; require a recognized chat block with extractable text before leaving the string path.
  - confidence: 0.9
