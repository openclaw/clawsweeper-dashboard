## Summary

Keep open: the PR targets a real vLLM/OpenAI-compatible response formatting problem and current main still leaves the shared helper's string path unparsed, but the branch is not merge-ready because the new heuristic can drop unrelated JSON text, misses whitespace-wrapped stringified arrays from the linked report shape, conflicts with current main, and lacks required real vLLM proof.

## What This Changes

The PR adds JSON parsing for string inputs that look like chat content-block arrays in `extractTextFromChatContent` and adds helper tests for plain, array, stringified, mixed, sanitizer, and join behavior.

## Best Possible Solution

Rebase onto current main, add a narrowly recognized stringified content-block parser that trims only for detection, preserves unrelated JSON/plain strings and current coercion hardening, covers the regressions, and attach redacted vLLM/OpenAI-compatible runtime proof.

## Reproduction Assessment

Yes at source level: current main's string branch sanitizes and normalizes without parsing, so a stringified `[{"type":"text","text":"hello"}]` payload remains raw while array content extracts text. I did not run a live vLLM setup in this read-only review.

## Solution Assessment

No as currently written; parsing stringified blocks in the shared helper is a plausible narrow direction, but the PR must trim the JSON candidate and require a recognized chat-content shape so unrelated JSON strings are preserved.

- **[P2] Trim before detecting stringified content arrays:** `src/shared/chat-content.ts:14`
  - body: The linked vLLM report shows the nested content string ending with escaped newlines after the JSON array. Because this checks `startsWith("[")` and `endsWith("]")` on the raw string before parsing, that reported shape still falls through as plain text and can remain double-serialized; trim the candidate for detection/parsing while keeping the existing plain-string fallback.
  - confidence: 0.86
- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`

A string like `[{"type":"event","payload":"x"}]` now recurses into array handling and returns `null` instead of preserving the original text, so require a recognized chat content-block shape before switching away from plain-string handling.
  - confidence: 0.9
