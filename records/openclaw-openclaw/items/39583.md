## Summary

Keep open: current main still preserves JSON-stringified block arrays as strings, but the PR head can drop unrelated JSON string content with `type` keys and still lacks redacted after-fix vLLM runtime proof.

## What This Changes

The PR teaches `extractTextFromChatContent` to parse JSON-stringified content-block arrays and adds Vitest coverage for the helper.

## Best Possible Solution

Rebase onto current main, parse only recognized chat content-block arrays while preserving unrelated JSON strings, keep regression coverage, and attach redacted vLLM `openai-completions` proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection shows current main returns a JSON-stringified content-block array as plain string while actual arrays extract text; no live vLLM run was established.

## Solution Assessment

Is this the best way to solve the issue? No as currently written; parsing stringified supported chat blocks is the right direction, but the guard must validate recognized chat-content shapes before leaving plain-string handling.
