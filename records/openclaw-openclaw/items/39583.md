## Summary

Keep open: the PR targets a real helper-level bug, but current main still returns JSON-stringified content blocks as raw text and the proposed guard can drop unrelated JSON strings; external vLLM proof is also missing.

## What This Changes

The PR updates `extractTextFromChatContent` to parse JSON-stringified content-block arrays and adds unit tests for plain, array, stringified, and mixed content.

## Best Possible Solution

Rebase onto current main, parse only validated chat content-block JSON strings, keep focused regression tests, and add redacted vLLM runtime proof before merge.

## Reproduction Assessment

Yes at the helper level: current main returns string content unchanged, so a JSON-stringified `[{"type":"text","text":"hello"}]` array remains raw JSON by source inspection. The end-to-end vLLM flow has not been reproduced in this review or proven in the PR.

## Solution Assessment

No. The shared helper is the right surface, but the proposed type-only detection is too broad and should be narrowed before it changes plain-string handling.
