## Summary

This PR is still useful because current main does not parse JSON-stringified content-block arrays, but it is not merge-ready: the parsing heuristic can drop unrelated JSON strings with a `type` key and the PR only shows unit-test proof, not a real vLLM run.

## What This Changes

The PR updates `extractTextFromChatContent` to parse JSON-stringified chat content block arrays and adds unit tests for plain strings, arrays, stringified arrays, mixed blocks, and option handling.

## Best Possible Solution

Rebase the shared extraction fix onto current main, use a stricter content-block parser that preserves unrelated JSON strings, keep focused regression tests, and add redacted real vLLM output proof before merge.

## Reproduction Assessment

Yes at the helper level: current main returns string content unchanged, so a JSON-stringified `[{"type":"text","text":"hello"}]` array remains raw JSON by source inspection. The live vLLM user flow has not been reproduced or proven in this PR.

## Solution Assessment

No. The shared helper is the right surface, but the proposed type-only detection is too broad and should be narrowed before it changes plain-string handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Narrow JSON-block detection before dropping strings:** `src/shared/chat-content.ts:21-23`
  - body: The new guard accepts any parsed array of objects with a `type` key, so a plain string like `[{"type":"event","payload":"x"}]` recurses into the array path and returns `null` instead of preserving the original text. Require a stronger chat-content shape, such as recognized text blocks, before treating string input as content blocks.
