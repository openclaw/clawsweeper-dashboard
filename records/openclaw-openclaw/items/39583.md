## Summary

Keep open. Current main still lacks handling for JSON-stringified OpenAI/vLLM content blocks, but the PR is not merge-ready because the type-only JSON-array guard can drop legitimate string content, the branch conflicts with current shared text normalization, and no real vLLM proof has been posted.

## What This Changes

The PR parses string inputs that look like JSON arrays of chat content blocks and adds unit coverage for plain, array, stringified, mixed, and edge-case content.

## Best Possible Solution

Rebase onto current `src/shared/chat-content.ts`, parse only well-shaped stringified chat-content arrays with extractable `type: "text"` blocks while preserving non-chat JSON strings, and add focused regression coverage plus redacted real vLLM output proof.

## Reproduction Assessment

Yes, source-reproducible. A focused check such as `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows current main's string branch and returns the raw JSON string, while the array branch would extract `hello`.

## Solution Assessment

No, not as submitted. Parsing stringified content blocks is the right direction, but the guard should be narrowed to recognized/extractable chat content blocks and rebased onto the current shared extractor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate parsed arrays on real chat text blocks:** `src/shared/chat-content.ts:21-23`
  - body: The new guard accepts any parsed object array with a `type` key, then recurses into the array extractor that only keeps `type === "text"`. A valid plain string like `[{"type":"event","payload":"x"}]` now becomes `null` instead of remaining text, so callers that skip null content can lose data. Require an actual extractable `type: "text"` block or known chat content-part shape before switching away from plain-string handling.
  - confidence: 0.88
