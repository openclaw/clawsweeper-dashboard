## Summary

Keep open. Current main and the latest release still lack JSON-stringified content-block parsing, so the central fix is not obsolete, but this PR should not merge yet because the final guard can drop non-chat typed JSON strings, it does not address the nested stringified text pattern seen in related reports, and it has no real vLLM proof.

## What This Changes

The PR adds JSON-stringified content-block parsing to `extractTextFromChatContent` and adds unit coverage for plain strings, arrays, stringified arrays, and mixed text/image blocks.

## Best Possible Solution

Update the shared helper on current main to parse only recognized content-block arrays, recurse through nested serialized text payloads without false positives, preserve `coerceChatContentText` hardening, and prove it with focused tests plus redacted vLLM runtime output.

## Reproduction Assessment

Yes at source level: current main returns a stringified `[{"type":"text","text":"hello"}]` as raw text, and the PR head source shows the typed-array false-positive path. I did not run a live vLLM reproduction in this read-only review.

## Solution Assessment

No. The fix belongs in the shared helper, but the current patch should narrow detection to real chat text blocks, handle nested serialized block text from the reported symptom, and keep current coercion behavior before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve typed JSON strings that are not chat blocks:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat content. A plain string such as `[{"type":"event","payload":"x"}]` now recurses into the array path and returns `null` because there are no text blocks, where the helper previously preserved the non-empty string; require a recognized chat text shape or at least one extractable text block before switching away from plain-string handling.
  - confidence: 0.9
- **[P2] Unwrap nested serialized block text:** `src/shared/chat-content.ts:25`
  - body: The reported nested output includes a text block whose `text` is itself a serialized content-block array. After this new top-level parse, the array path takes that inner JSON string as ordinary text and returns it, so nested/double-serialized content can still leak; recurse through text block payloads or add a focused regression test for that shape.
  - confidence: 0.82
