## Summary

Keep open: this is a plausible fix for a real stringified content-block bug, but the current PR head is still too broad and no after-fix vLLM runtime proof is present, so it is not ready to merge or close.

## What This Changes

The branch adds JSON parsing for stringified content-block arrays in `extractTextFromChatContent` plus unit tests for string, array, stringified, mixed, and edge cases.

## Best Possible Solution

Land a current-main version of a conservative stringified content-block parser that recognizes supported chat text shapes, preserves unrelated JSON strings, adds focused regression tests, and includes redacted vLLM/OpenAI-compatible runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main returns a JSON-stringified `[{"type":"text","text":"hi"}]` payload through the plain string path, while real array content already extracts `hi`. I did not run tests because this review is read-only and avoids commands that may create artifacts.

## Solution Assessment

No as currently written. Parsing stringified content blocks is the right narrow direction, but the guard needs a supported content-block shape instead of `type` alone, and the PR still needs real vLLM runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat blocks, but the array path only extracts `type === "text"`. Inputs like `[{"type":"event","payload":"x"}]` or `[{"type":"output_text","text":"x"}]` now become `null` instead of being preserved or extracted, so callers that omit null text can silently drop content. Require a stronger supported content-block shape before parsing.
  - confidence: 0.9
