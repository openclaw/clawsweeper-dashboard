## Summary

Keep open: current main and the latest release still lack the JSON-stringified content-block handling, but this PR is not merge-ready because the `type`-only parsing guard can drop valid plain string content and no real vLLM behavior proof has been posted.

## What This Changes

The PR updates `extractTextFromChatContent` to parse JSON-stringified arrays of chat content blocks and adds unit tests for strings, arrays, stringified arrays, mixed blocks, and edge cases.

## Best Possible Solution

Tighten JSON-array recognition to proven chat content-block shapes, add regression coverage for both vLLM stringified blocks and arbitrary JSON strings that must remain text, then require redacted real vLLM output proof before merge.

## Reproduction Assessment

Yes for the focused helper path: current main returns JSON-stringified arrays as plain strings, and the PR regression is source-reproducible from the `type`-only guard. The full vLLM runtime path has not been live-verified in the PR.

## Solution Assessment

No. Parsing stringified content blocks in the shared helper is a plausible narrow fix, but the current guard should be stricter before switching a plain string into array extraction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before dropping string content:** `src/shared/chat-content.ts:22-23`
  - body: The new guard treats any JSON array of objects with a `type` key as chat content, so a legitimate string such as `[{"type":"event","payload":"x"}]` now recurses into the array path, finds no text blocks, and returns `null` instead of the original text. Callers that skip null or empty extracted text can silently lose user or assistant content; require a stronger chat-content-block shape before parsing.
  - confidence: 0.89
