## Summary

Keep open: current main still lacks the helper behavior for literal JSON-stringified content-block arrays, but the PR head is not merge-ready because its type-only parser heuristic can drop unrelated typed JSON strings and no after-fix vLLM runtime proof is posted.

## What This Changes

The PR parses JSON-stringified chat content block arrays in `extractTextFromChatContent` and adds unit coverage for plain strings, arrays, stringified arrays, and edge cases.

## Best Possible Solution

Add a narrowly guarded parser or transport normalization for JSON-stringified OpenAI-compatible chat block strings, preserving unrelated JSON text and backing it with focused helper coverage plus redacted vLLM runtime proof.

## Reproduction Assessment

Yes for the helper path from source: current main treats all strings as plain text, while real arrays extract `type === "text"` blocks. I did not run tests because this review is read-only.

## Solution Assessment

No as currently written. Parsing stringified blocks can be a narrow fix, but the PR needs a supported-shape guard and real vLLM proof; current main's transport normalization covers structured arrays/objects, not literal stringified arrays.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new string parser treats any JSON array of objects with a `type` key as chat blocks. A string such as `[{"type":"event","payload":"x"}]` now recurses into the array path and returns `null`, so callers that omit null text can silently drop content that current main preserves; require a supported chat content shape before switching away from plain-string handling.
