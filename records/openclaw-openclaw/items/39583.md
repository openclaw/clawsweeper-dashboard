## Summary

Keep this PR open. Current main still does not parse JSON-stringified chat content-block arrays in `extractTextFromChatContent`, but the PR head has a guard regression that can drop unrelated JSON text and it still lacks real vLLM behavior proof before merge.

## What This Changes

The PR changes `extractTextFromChatContent` to parse JSON-stringified content-block arrays and adds Vitest coverage for strings, arrays, mixed blocks, and edge cases.

## Best Possible Solution

Rebase and narrow the parser to recognized chat content-block shapes, preserve unrelated JSON text, cover mixed and invalid shapes, and add redacted vLLM runtime proof.

## Reproduction Assessment

Yes for the helper path from source: current main treats string input as plain text while array input extracts `type === "text"` blocks. I did not run a live vLLM reproduction in this read-only review.

## Solution Assessment

No as currently written. Parsing stringified supported chat blocks is a plausible narrow fix, but the current guard accepts any `type`-key object array and should be restricted before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new parser treats any JSON array of objects with a `type` key as chat blocks. A plain string such as `[{"type":"event","payload":"x"}]` now recurses into the array path and returns `null`, so callers that omit null text can silently drop content that current main preserves; require a supported chat content shape before switching away from plain-string handling.
