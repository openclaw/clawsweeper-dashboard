## Summary

## What This Changes

The PR adds JSON parsing for string inputs that look like content-block arrays in `extractTextFromChatContent` and adds tests for plain, array, stringified-array, mixed-block, and option-handling cases.

## Best Possible Solution

Land a rebased helper change that trims only the parse candidate, recognizes real chat text-block shapes, preserves unrelated JSON strings and current coercion hardening, and includes focused regressions plus redacted vLLM runtime proof.

## Reproduction Assessment

Yes at source level: current main returns any stringified content-block array as plain string content while already-array content blocks are text-extracted. I did not establish a live vLLM reproduction in this read-only review.

## Solution Assessment

No as currently written; parsing stringified blocks is a plausible narrow fix, but this branch needs a rebase, stricter chat-block detection, whitespace-tolerant parsing, and real provider proof before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Trim before detecting stringified content arrays:** `src/shared/chat-content.ts:15`
  - body: The parser only runs when the raw string starts with `[` and ends with `]`, so a provider response with surrounding whitespace or a trailing newline still takes the old path and is returned as raw JSON. Trim a parse candidate before the boundary check while preserving the original string for the fallback path.
  - confidence: 0.88
- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: Checking only for a `type` key treats unrelated JSON arrays like `[{"type":"event","payload":"x"}]` as chat blocks, then the array path finds no text blocks and returns `null` instead of preserving the original string. Require a stronger chat content shape, such as at least one valid text block, before switching away from plain-string handling.
  - confidence: 0.9
