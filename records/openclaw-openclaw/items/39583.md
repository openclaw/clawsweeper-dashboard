## Summary

Keep open: the vLLM serialization bug is still plausible on current main, but this PR is not merge-ready because contributor real behavior proof is missing and the current JSON-array guard can drop unrelated JSON text.

## What This Changes

The PR parses JSON-stringified chat content-block arrays in `extractTextFromChatContent` and adds Vitest coverage for plain strings, arrays, stringified arrays, mixed blocks, and edge cases.

## Best Possible Solution

Rebase the PR, narrow parsing to recognized chat content-block shapes while preserving unrelated JSON text, add focused regression coverage, and attach redacted vLLM runtime proof.

## Reproduction Assessment

Yes for the helper path from source: current main treats string input as plain text while array input extracts `type === "text"` blocks. I did not establish a live vLLM reproduction in this read-only review.

## Solution Assessment

No as currently written. Parsing stringified supported chat blocks is a plausible narrow fix, but the current guard accepts any `type`-key object array and should be restricted before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Narrow JSON-block detection before recursing:** `src/shared/chat-content.ts:21-23`
  - body: The new parser treats any JSON array of objects with a `type` key as chat blocks.
