## Summary

Keep open. Current main still preserves JSON-stringified content-block arrays as raw strings, so the vLLM/openai-completions bug is still real, and this PR is useful source work; however, the updated type-only parse guard can now drop arbitrary JSON string content that happens to contain objects with a `type` key, so it needs one focused fix before merge.

## What This Changes

The PR adds JSON parsing for stringified chat content-block arrays in `extractTextFromChatContent` and adds unit coverage for plain strings, arrays, stringified arrays, mixed blocks, and fallback cases.

## Best Possible Solution

Keep the shared helper as the normalization boundary, but parse strings only when the parsed value has a recognizable chat content-block shape, preserve arbitrary JSON strings such as typed event arrays, mirror the real-array text extraction semantics for mixed media/text blocks, and add focused regression tests plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. On current main, `extractTextFromChatContent(JSON.stringify([{ type: "text", text: "hello" }]))` follows the string branch and returns the normalized raw JSON string, while passing the same value as an array extracts `hello`.

## Solution Assessment

No, not as-is. Parsing stringified content-block arrays in `extractTextFromChatContent` is the right narrow direction, but the type-only guard should be tightened so non-chat JSON string content is preserved, and the branch should be rebased onto the current helper implementation.
