## Summary

Keep open: current main still lacks the requested cron running-state read model, but this PR should not merge while the queued `cron.run` acknowledgment reports `currentStatus: "running"` before the command-lane task has actually marked the job running, and real behavior proof is still missing.

## What This Changes

The PR adds derived cron read-state fields, enriches manual `cron.run` acknowledgments, updates gateway protocol typing/schema, and adds cron/gateway regression tests.

## Best Possible Solution

Add a read-only projection for actually running cron jobs while keeping the manual run response semantically queued or accepted until stored state shows `runningAtMs`, then prove the gateway behavior in a real cron flow.

## Reproduction Assessment

Yes, by source inspection rather than live execution. Current main returns generic queued acks from `enqueueRun`, has no derived `currentStatus` read model, and existing regression tests show accepted manual runs can wait behind the cron command lane.

## Solution Assessment

No. The read-model projection is a reasonable direction, but reusing `currentStatus: "running"` in the enqueue acknowledgment conflates queued acceptance with persisted running state; a queued/accepted field or a later stored-state signal is safer.
