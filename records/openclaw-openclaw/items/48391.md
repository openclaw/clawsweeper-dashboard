## Summary

Keep open: current main still does not implement the PR's gateway cron read-model fields, and the merged CLI status work only covers `openclaw cron * --json`. The PR is not merge-ready because the `cron.run` acknowledgment uses persisted read-model status wording before the job is actually marked running, and the contributor still has not supplied real behavior proof.

## What This Changes

The PR adds derived running-state fields to cron read responses, enriches `cron.run` acknowledgments, updates cron protocol schema/types, and adds cron/gateway regression coverage.

## Best Possible Solution

Keep the additive read-only running projection if maintainers want that gateway surface, but use distinct queued or accepted acknowledgment metadata until a run is actually marked running.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows `enqueueRun` returns after queueing while `runningAtMs` is set later in `prepareManualRun`, so immediate `cron.list` can disagree with the acknowledgment. The broader PR is an additive protocol feature rather than a broken documented existing contract with a current-main end-user repro.

## Solution Assessment

No, not as written. The read projection is plausible, but `cron.run` should not expose persisted read-model `currentStatus` wording until the run is actually reflected by `runningAtMs`.
