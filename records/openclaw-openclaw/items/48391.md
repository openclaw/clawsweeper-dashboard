## Summary

Keep open: current main still lacks the gateway cron read-model fields, and the PR still has an actionable protocol correctness concern plus missing real behavior proof.

## What This Changes

The PR adds derived cron read-state fields to `cron.list`/`cron.listPage`, enriches `cron.run` acknowledgments, updates gateway cron schemas, and adds regression tests.

## Best Possible Solution

Keep the additive read-only running projection if maintainers want this gateway surface, but keep `cron.run` acknowledgment metadata distinct from persisted `currentStatus` until the run is actually marked running.

## Reproduction Assessment

Yes for the review finding: source inspection shows `enqueueRun` returns after queueing while `runningAtMs` is set later in `prepareManualRun`, so an immediate `cron.list` can disagree with the acknowledgment. The broader PR is an additive protocol feature rather than a broken existing documented contract with an end-user repro here.

## Solution Assessment

No, not as written. The read projection is plausible, but `cron.run` should use queued or accepted acknowledgment metadata instead of the persisted read-model `currentStatus` name before execution starts.
