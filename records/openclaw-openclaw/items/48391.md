## Summary

Keep open: current main still lacks the proposed cron running-state read fields, but the PR has a blocking acknowledgment semantics issue, missing real behavior proof, and a dirty/stale branch against current main.

## What This Changes

The PR adds derived cron running-state fields to cron list/read schemas, richer `cron.run` in-progress acknowledgment metadata, and cron/gateway regression tests.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but separate queued/accepted acknowledgment metadata from persisted running state and refresh the branch onto current cron protocol fields.

## Reproduction Assessment

Yes. Source inspection shows `enqueueRun` returns immediately after queueing while `runningAtMs` is persisted later in `prepareManualRun`, so an immediate `cron.list` can disagree with the acknowledgment.

## Solution Assessment

No, not as written. The read-model projection is a reasonable direction, but the `cron.run` acknowledgment should not reuse `currentStatus: "running"` before persisted running state exists.
