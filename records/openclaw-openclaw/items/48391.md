## Summary

Keep open: current main still lacks the requested cron running-state read model, but this PR should not merge as-is because the new `currentStatus: "running"` acknowledgment is emitted while the run is only queued and can disagree with `cron.list`. The branch also needs rebasing onto current cron schema/list-page changes.

## What This Changes

The PR adds computed cron read-state fields for running jobs, enriches `cron.run` enqueue acknowledgments with in-progress metadata, and updates gateway schema/tests.

## Best Possible Solution

Land the read-model projection on current main, keep writable patch state free of derived fields, and make manual-run acknowledgments distinguish queued/accepted work from actually running work.

## Reproduction Assessment

Yes. Source inspection gives the path: keep the Cron lane busy or queue multiple manual runs; `enqueueRun` returns before `prepareManualRun` sets `runningAtMs`, so the ack can say running while `cron.list` has no running read state.

## Solution Assessment

No. The read-model projection is a good direction, but returning live `currentStatus: "running"` from `enqueueRun` before persisted running state exists is not the narrowest maintainable protocol contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report queued cron runs as queued, not running:** `src/cron/service/ops.ts:604-608`
