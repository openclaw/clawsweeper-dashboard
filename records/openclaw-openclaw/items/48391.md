## Summary

Keep open: the PR addresses a real missing cron read-model surface on current main, but it should not merge while the immediate `cron.run` acknowledgment reports `currentStatus: "running"` before the run has actually persisted `runningAtMs`, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds derived cron running-state fields to list responses, enriches manual `cron.run` acknowledgments, updates gateway cron schemas/types, and adds cron/gateway regression coverage.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but make queue acceptance metadata distinct from persisted running status and preserve current-main cron schema/list behavior.

## Reproduction Assessment

Yes, by source inspection: PR head returns `currentStatus: "running"` from `enqueueRun` before the queued task calls `run` and persists `runningAtMs`. Current-main regression coverage also shows manual runs can sit queued before the first run starts.

## Solution Assessment

No. The read-model projection is a reasonable direction, but the immediate `cron.run` response should describe accepted, queued, or pending work rather than reuse the stored-state `currentStatus` name before that state exists.

## Review Findings
