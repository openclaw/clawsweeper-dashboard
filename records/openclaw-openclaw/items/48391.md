## Summary

Current main still lacks the proposed cron read projection and richer manual-run acknowledgment, so this PR is not obsolete; however, the patch still has a protocol-semantic blocker around reporting queued work as running and lacks external-PR real behavior proof.

## What This Changes

The PR adds derived cron running-state read fields, richer manual `cron.run` in-progress acknowledgment metadata, cron protocol schema updates, and cron/gateway regression tests.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but use queued or accepted acknowledgment metadata for `cron.run` instead of the persisted read-model `currentStatus` name.

## Reproduction Assessment

Yes for the review finding: source inspection shows `enqueueRun` returns immediately after queueing while `runningAtMs` is persisted later in `prepareManualRun`, so an immediate `cron.list` can disagree with the acknowledgment. The broader PR is a protocol feature rather than a broken existing documented contract with a runnable end-user repro here.

## Solution Assessment

No, not as written. The read projection is a reasonable direction, but the `cron.run` acknowledgment should not reuse `currentStatus: "running"` before persisted running state exists.
