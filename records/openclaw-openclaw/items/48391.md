## Summary

Keep open; the PR is useful but not merge-ready because the new `cron.run` acknowledgment can report queued work as `running`, and after-fix real behavior proof is still missing.

## What This Changes

The PR adds derived cron read-state fields, enriches manual `cron.run` acknowledgments, updates gateway schema/types, and adds cron/gateway regression tests.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but make the immediate `cron.run` response describe queued or accepted work until `runningAtMs` is actually persisted, then add real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than live execution: current main lacks the new derived cron read-state fields, and existing queue tests show `enqueueRun` can acknowledge work before the command lane actually starts it.

## Solution Assessment

No. The read-model projection is a reasonable direction, but reusing `currentStatus: "running"` in the immediate enqueue acknowledgment conflates queue acceptance with persisted running state; queued/accepted/pending metadata would be safer.
