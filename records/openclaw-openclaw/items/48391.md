## Summary

Keep this PR open: current main still lacks the proposed gateway cron read-model fields, but the branch should not merge as-is because the new `cron.run` acknowledgment reports `currentStatus: "running"` before the queued run has actually set the persisted running marker, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds derived cron running-state fields to `cron.list`/`cron.listPage`, enriches `cron.run` acknowledgments, updates cron protocol schema, and adds cron/gateway regression tests.

## Best Possible Solution

Keep an additive read-only running projection if maintainers want that gateway surface, but keep `cron.run` acknowledgment metadata distinct from persisted `currentStatus` until the run is actually marked running.

## Reproduction Assessment

Yes for the review finding: source inspection shows `enqueueRun` returns after queueing while `runningAtMs` is set later in `prepareManualRun`, so an immediate `cron.list` can disagree with the acknowledgment. The broader PR is an additive protocol feature rather than a broken existing documented contract with an end-user repro here.

## Solution Assessment

No, not as written. The read projection is plausible, but `cron.run` should use queued or accepted acknowledgment metadata instead of the persisted read-model `currentStatus` name before execution starts.
