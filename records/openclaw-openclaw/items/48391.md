## Summary

Keep this PR open. Current main still lacks the cron read-model fields and richer manual-run acknowledgment, but the branch is not ready to merge as-is because it needs to preserve current cron state schema fields and settle the queued-versus-running acknowledgment semantics.

## What This Changes

This PR adds cron read-model running-state fields to list/listPage responses, expands manual cron.run enqueue acknowledgments, and updates cron/gateway protocol tests and schemas.

## Best Possible Solution

Land a reviewed, rebased variant that keeps persisted cron state unchanged, adds separate read-only cron job/read-state schema and TypeScript types, projects both list paths through that read model, preserves all current state fields, and uses explicit queued/pending acknowledgment wording or documents the accepted-in-flight semantics clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: start a manual cron run that has not completed, call cron.list on current main, and the returned stored job has runningAtMs but no currentStatus/lastCompleted* projection; enqueueRun also returns only the old queued ack. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is. The read-model direction is maintainable, but the branch should first preserve current skipped-run state schema and avoid reporting a queued manual run as persisted running state unless that semantic is explicitly renamed or documented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve skipped-run state in the schema:** `src/gateway/protocol/schema/cron.ts:228-242`
  - body: Current main persists and exposes `state.consecutiveSkipped` for skipped-run alerts. This new shared `CronJobStateProperties` block omits that existing field, so rebasing or merging this branch would remove it from the protocol schema while runtime cron state still writes it. Carry `consecutiveSkipped` into the shared properties before landing.
  - confidence: 0.88
- **[P2] Report queued manual runs separately from running jobs:** `src/cron/service/ops.ts:605-607`
  - body: `enqueueRun` returns `currentStatus: "running"` immediately after putting work on the cron lane, but the job is only marked running later when the queued task calls `prepareManualRun` and sets `runningAtMs`. If the lane is busy, a client can receive this ack and then poll `cron.list` to see no running state, or the queued task can later skip. Use queued/pending acknowledgment metadata, or reserve the run before claiming the read-model status.
  - confidence: 0.86
