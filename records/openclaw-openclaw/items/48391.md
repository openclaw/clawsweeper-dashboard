## Summary

Keep open. Current main still lacks the derived cron read model and richer manual-run acknowledgment, but this PR should not merge as-is because it drops an existing protocol state field and leaves the queued-versus-running acknowledgment semantics unresolved.

## What This Changes

This PR adds derived cron running-state fields to list/listPage responses, expands `cron.run` enqueue acknowledgments with in-progress metadata, and updates cron/gateway protocol tests and schemas.

## Best Possible Solution

Land a rebased variant that keeps persisted cron state unchanged, adds additive read-only list/listPage fields, preserves all existing cron state schema fields, and uses explicit queued/pending acknowledgment wording unless the run is actually reserved as running.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main returns stored cron jobs directly from `list()`/`listPage()` and `enqueueRun()` returns only the old queued acknowledgment. The queued-versus-running race is also source-reproducible because `runningAtMs` is set later inside the cron lane.

## Solution Assessment

No, not as-is. The read-model direction is maintainable, but the patch should preserve the full current state schema and avoid reusing `currentStatus: "running"` for work that is only queued unless maintainers explicitly choose that protocol contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve skipped-run state in the schema:** `src/gateway/protocol/schema/cron.ts:228-242`
  - body: Current main exposes `state.consecutiveSkipped` and failure-alert code/tests use it for skipped-run alerts. The new shared state properties block omits that existing field, so merging would drop it from the protocol state schemas while runtime cron code still writes and reads it.
  - confidence: 0.88
- **[P2] Report queued manual runs separately from running jobs:** `src/cron/service/ops.ts:605-607`
  - body: `enqueueRun` returns `currentStatus: "running"` immediately after queueing work, but `runningAtMs` is only written later when the queued task calls `prepareManualRun`. When the lane is busy, callers can receive a running ack and then poll `cron.list` to see no running state, or the task can later skip.
  - confidence: 0.86
