## Summary

Keep open: current main still lacks the proposed cron read-model fields, so the PR is not obsolete, but it should not merge as written because the manual-run acknowledgment reports a queued run as `currentStatus: "running"` before persisted running state exists and the PR still lacks real behavior proof.

## What This Changes

The PR adds derived cron running-state read fields, richer `cron.run` acknowledgment metadata, gateway protocol schema updates, and focused cron/gateway regression tests.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but use acknowledgment metadata that clearly describes queued or accepted work separately from persisted job running state and refresh it against current cron protocol fields.

## Reproduction Assessment

Yes. Source inspection shows `enqueueRun` returns immediately after queueing while `runningAtMs` is only persisted later inside `prepareManualRun`, so the PR acknowledgment can disagree with an immediate `cron.list` read.

## Solution Assessment

No, not as written. The read-model projection is a reasonable direction, but the `cron.run` acknowledgment should not reuse the persisted read-model `currentStatus` name before that state exists.
