## Summary

This PR should stay open because current main still lacks the proposed cron read-model fields, but the patch is not ready as-is: the new `cron.run` acknowledgment reports `currentStatus: "running"` before the queued cron task has actually marked the job running.

## What This Changes

The PR adds derived cron read-state fields, expands `cron.run` acknowledgment metadata, updates gateway protocol schemas, and adds cron/gateway regression coverage.

## Best Possible Solution

Keep the derived read-model direction, but make the manual-run acknowledgment distinguish queued/accepted work from persisted running state, then rebase the schema and tests onto current main.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the requested read-model fields, and the PR's new ack can be returned before `prepareManualRun` persists `runningAtMs`.

## Solution Assessment

No, not as-is. The read-model projection is reasonable, but `currentStatus: "running"` in the immediate enqueue ack overstates a queued state; a separate queued/accepted status or earlier persisted state marker would be clearer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report queued cron runs as pending:** `src/cron/service/ops.ts:604-607`
  - body: `enqueueRun` returns this payload immediately after queueing the command, but `runningAtMs` is only written later when the cron lane executes `prepareManualRun`. If the lane is busy, clients can receive `currentStatus: "running"` while `cron.list` still reports no running state, or the queued task can later skip as already-running. Use a queued/accepted status field here, or persist the running state before returning this ack.
  - confidence: 0.9
