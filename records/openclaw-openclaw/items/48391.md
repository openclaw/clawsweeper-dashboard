## Summary

Keep open: the requested cron read-model improvement is not on current main, but this PR is not merge-ready because the queued-run acknowledgment can report a false running state, the schema split needs to preserve current-main cron state fields, and external real behavior proof is still missing.

## What This Changes

The PR adds derived cron read-state fields to cron list responses, enriches cron.run enqueue acknowledgments, updates gateway protocol schema/types, and adds cron/gateway regression tests.

## Best Possible Solution

Land a rebased additive protocol change that keeps persisted/update state separate from read-only fields, preserves current cron state fields, and reports manual-run acknowledgments as queued or accepted until runningAtMs is actually reserved.

## Reproduction Assessment

Yes, at source level. Current main returns stored cron job state without the new read-model fields, and the PR's queued-run race is visible because enqueueRun returns before prepareManualRun persists runningAtMs.

## Solution Assessment

No. The read-model direction is reasonable, but the current patch is not the best merge shape because it reports queued work as running and its schema split does not yet preserve current-main cron state fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report queued cron runs as queued, not running:** `src/cron/service/ops.ts:605-607`
  - body: `enqueueRun` returns this payload immediately after putting work on the cron command lane, but `runningAtMs` is not written until the queued task later enters `prepareManualRun`. If the lane is backed up, or the task later skips as already-running, clients can display `currentStatus: "running"` for a job that is only queued. Use an accepted/queued acknowledgment or reserve the run before claiming it is running.
  - confidence: 0.9
- **[P2] Preserve existing cron state fields in the read schema:** `src/gateway/protocol/schema/cron.ts:228-241`
  - body: The new shared `CronJobStateProperties` object is also used by `CronJobReadStateSchema`, but the branch's copy is missing current-main state fields such as `lastDiagnostics`, `lastDiagnosticSummary`, and `consecutiveSkipped`. After rebasing, keep the full current state schema in the read schema so generated protocol types and clients do not lose those existing response fields.
  - confidence: 0.84
