## Summary

Keep open. Current main still lacks the requested cron read-model fields and richer manual-run acknowledgment, but this PR should not merge as-is because it drops an existing skipped-run state field from the refactored schema and reports queued manual runs as `running` before the persisted running marker exists.

## What This Changes

This PR adds derived cron running-state fields to list/listPage responses, expands `cron.run` enqueue acknowledgments with in-progress metadata, and updates cron/gateway protocol tests and schemas.

## Best Possible Solution

Land a rebased variant that keeps persisted cron state unchanged, adds separate read-only cron job/read-state schema and TypeScript types, projects both list paths through that read model, preserves all existing state fields including skipped-run counters, and uses explicit queued/pending acknowledgment wording or an actual reservation before claiming running state.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: start a manual cron run, read `cron.list`, and the response can contain `runningAtMs` without the derived read-model fields; `enqueueRun()` also still returns only the old queued acknowledgment.

## Solution Assessment

No, not as-is. The read-model direction is maintainable, but the branch should preserve the full current state schema and avoid reporting a queued manual run as the same `currentStatus` used for persisted running jobs unless maintainers explicitly accept and document that contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve skipped-run state in the schema:** `src/gateway/protocol/schema/cron.ts:228-242`
  - body: Current main persists and exposes `state.consecutiveSkipped` for skipped-run alerts. This new shared state properties block omits that existing field, so merging this branch would remove it from the protocol state schemas while runtime cron code still writes and tests the value. Carry `consecutiveSkipped` into the shared properties before landing.
  - confidence: 0.88
- **[P2] Report queued manual runs separately from running jobs:** `src/cron/service/ops.ts:605-607`
  - body: `enqueueRun` returns `currentStatus: "running"` immediately after queueing work, but the job is only marked running later when the queued task calls `prepareManualRun` and sets `runningAtMs`. If the lane is busy, clients can receive a running acknowledgment and then poll `cron.list` to see no running state, or the queued task can later skip. Use queued/pending metadata, or reserve the run before claiming read-model running state.
  - confidence: 0.86
