## Summary

Keep open: current main still lacks the proposed cron running-state read model, but the PR should not merge as-is because the immediate `cron.run` acknowledgment reports `currentStatus: "running"` before the queued cron lane has persisted the running marker, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds derived cron read-state fields, enriches manual `cron.run` acknowledgments, updates gateway cron schema/types, and adds cron/gateway regression coverage.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but make the immediate `cron.run` response describe queued or accepted work until `runningAtMs` is actually persisted, then update the branch on current main with real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than live execution: current main lacks the new derived cron read-state fields, and the queue path shows `enqueueRun` can acknowledge work before the command lane starts and persists `runningAtMs`.

## Solution Assessment

No. The read-model projection is a reasonable direction, but reusing `currentStatus: "running"` in the immediate enqueue acknowledgment conflates queue acceptance with persisted running state; queued, accepted, or pending metadata would be safer.

## Review Findings

Overall correctness: patch is incorrect
