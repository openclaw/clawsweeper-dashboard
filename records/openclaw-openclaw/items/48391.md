## Summary

Keep open: current main still lacks the proposed cron read-model fields, and the PR still has a concrete protocol correctness blocker around reporting queued work as `running` before the persisted running marker exists.

## What This Changes

The PR adds derived cron read-model fields for running jobs, enriches `cron.run` acknowledgments, and updates gateway cron schemas plus focused cron/gateway tests.

## Best Possible Solution

Keep the read-only running projection if maintainers want this protocol surface, but use acknowledgment metadata that clearly describes queued or accepted work separately from persisted job state.

## Reproduction Assessment

Yes. Source inspection shows `enqueueRun` returns immediately after queueing while `runningAtMs` is only persisted later inside `prepareManualRun`, so the PR ack can disagree with an immediate `cron.list` read.

## Solution Assessment

No, not as written. The read-model projection is a reasonable direction, but the `cron.run` acknowledgment should not reuse the persisted read-model `currentStatus` name before that state exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report queued cron runs as pending:** `src/cron/service/ops.ts:605-607`

Use accepted or queued acknowledgment metadata instead of the persisted read-model status name.
