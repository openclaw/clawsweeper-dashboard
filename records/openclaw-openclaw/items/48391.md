## Summary

Keep open: the cron read-model/protocol change is not on current main and may still be useful, but the PR should not merge as-is because its `cron.run` acknowledgment reports queued work as `currentStatus: "running"` before the persisted running marker exists, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds derived cron read-state fields to list/listPage responses and gateway schemas, enriches `cron.run` enqueue acknowledgments, and updates cron/gateway regression tests.

## Best Possible Solution

Keep a read-only running projection if maintainers want this protocol surface, but make `cron.run` acknowledgment metadata describe accepted or queued work separately from persisted job state and preserve the current writable schema boundary.

## Reproduction Assessment

Yes. Source inspection shows current main only persists `runningAtMs` inside `prepareManualRun`, while the PR head returns `currentStatus: "running"` from `enqueueRun` immediately after queueing.

## Solution Assessment

No, not as written. The read-model projection is a reasonable direction, but the enqueue acknowledgment should use distinct queued/accepted metadata instead of the stored read-model `currentStatus` name before that state exists.
