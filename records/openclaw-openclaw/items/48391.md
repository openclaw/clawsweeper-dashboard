## Summary

Keep open: the PR contains useful cron/protocol work that is not on current main, but it still needs real behavior proof and has a blocking acknowledgment-semantics issue before merge.

## What This Changes

The branch adds derived cron read-state fields to list/page results, enriches `cron.run` acknowledgments, updates gateway protocol schema, and adds cron regression tests.

## Best Possible Solution

Keep the additive read-only running projection if maintainers want this gateway surface, but make `cron.run` acknowledgment metadata clearly queued/accepted until execution actually sets `runningAtMs`.

## Reproduction Assessment

Yes for the review finding: source inspection shows `enqueueRun` returns immediately after queueing while `prepareManualRun` sets `runningAtMs` later, so an immediate `cron.list` can disagree with the acknowledgment. The broader PR is an additive protocol feature rather than a broken existing documented current-main contract.

## Solution Assessment

No, not as written. The read projection is plausible, but the `cron.run` response should use queued or accepted acknowledgment metadata instead of the persisted read-model `currentStatus` name before execution starts.
