## Summary

Keep open: current main only partially overlaps through CLI JSON status, the gateway/protocol read-model fields are not landed, and the proposed branch still has a blocking queued-vs-running acknowledgment bug plus no real behavior proof.

## What This Changes

The PR adds computed cron running-state fields to cron list/read output, enriches manual cron run acknowledgments, and updates cron/gateway protocol tests.

## Best Possible Solution

Rebase the useful read-model work, keep derived fields out of writable patch state, and make `cron.run` acknowledgments distinguish queued/accepted work from actually running state before updating generated clients/docs/tests.

## Reproduction Assessment

Yes. Source inspection gives the path: keep the Cron command lane busy or queue manual runs; `enqueueRun` returns before the queued command calls `prepareManualRun` and persists `runningAtMs`, so this PR can report `currentStatus: "running"` while `cron.list` has no running read state.

## Solution Assessment

No. The read-model projection is a good direction, but using live `currentStatus: "running"` in the enqueue acknowledgment before persisted running state exists is not the narrowest maintainable protocol contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report queued cron runs as queued:** `src/cron/service/ops.ts:604-608`

On a busy Cron lane, this new `currentStatus: "running"` acknowledgment can disagree with `cron.list` and may later be skipped as `already-running`; use queued/accepted metadata or omit live status until the stored marker exists.
