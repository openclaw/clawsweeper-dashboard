## Summary

Keep this PR open. Current main still passes MCP `EmbeddedResource` tool-result blocks through the bundle-MCP boundary unchanged, while the PR applies the normalization at the right boundary, adds focused regression coverage, and has no blocking review findings.

## What This Changes

The PR converts MCP `EmbeddedResource` tool-result blocks to Pi text content in bundle-MCP materialization, adds materializer regression tests, and records the fix in the changelog.

## Best Possible Solution

Land the boundary-level normalization after maintainer approval. Keep provider transports generic, and handle any historical transcript replay compatibility as a separate narrow follow-up if it becomes user-visible.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a bundle-MCP `CallToolResult` with `content: [{ type: "resource", resource: { text: "..." } }]` is cast through unchanged on current main, then provider converters that only read top-level `text`/`image` content emit a placeholder or empty text.

## Solution Assessment

Yes. Normalizing at the bundle-MCP to Pi `AgentToolResult` boundary is the narrowest maintainable fix because it converts MCP-specific content before all provider transports and replay paths that expect Pi text/image content.
