## Summary

Keep this PR open. Current main still passes MCP `resource` tool-result blocks through the bundle-MCP boundary as non-Pi content, while this PR applies the narrow boundary normalization with focused regression coverage and green head checks; I found no blocking review findings.

## What This Changes

The PR converts MCP `EmbeddedResource` tool-result blocks into Pi text content at the bundle-MCP materialization boundary, adds regression tests, and records a changelog fix for #75674.

## Best Possible Solution

Land this boundary-level normalization after maintainer approval, keeping provider transports generic and leaving broader MCP block coverage for separate follow-ups if needed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A bundle-MCP `CallToolResult` containing `content: [{ type: "resource", resource: { text: "..." } }]` is cast through unchanged on current main, after which the OpenAI and Anthropic converters do not treat it as text.

## Solution Assessment

Is this the best way to solve the issue? Yes. Normalizing at `toAgentToolResult` is the narrowest maintainable fix because it converts MCP-specific content before all Pi provider transports and replay paths that expect only text/image content.
