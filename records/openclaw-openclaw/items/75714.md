## Summary

Keep this PR open. Current main still passes MCP EmbeddedResource tool-result blocks through the bundle-MCP boundary as Pi tool content even though downstream provider transports only handle text/image blocks; the PR applies the narrow boundary conversion with focused tests, a changelog entry, and green public check runs.

## What This Changes

This PR normalizes MCP `type: "resource"` tool-result content into Pi text content in bundle-MCP materialization, adds materializer regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Land the boundary-level normalization after maintainer review. Keep provider transports generic, and handle other MCP block variants such as audio or resource links as separate follow-ups if they become observable user problems.

## Reproduction Assessment

Yes. A source-level reproduction is clear: return a bundle-MCP `CallToolResult` with `content: [{ type: "resource", resource: { text: "..." } }]`; current main keeps that block unchanged, and provider converters only read top-level text/image content.

## Solution Assessment

Yes. Normalizing MCP resource blocks at the bundle-MCP to Pi `AgentToolResult` boundary is the narrowest maintainable fix because it converts MCP-specific content before all provider transports and replay paths consume it.
