## Summary

Keep open. Current main still has the raw MCP content pass-through that causes `resource` tool results to be invisible to provider transports, and this PR applies the narrow boundary-level normalization recommended in the linked report. I found no concrete correctness or security finding in the proposed diff, so this should proceed through normal CI and maintainer review rather than cleanup closure.

## What This Changes

The PR adds bundle-MCP content normalization for MCP `resource` tool-result blocks, regression tests for the materializer, and a user-facing changelog entry for #75674.

## Best Possible Solution

Land the boundary-level normalization after CI proves the focused materializer test and changed lane. If maintainers want compatibility for already-saved raw resource transcripts, handle that separately in provider replay sanitization rather than broadening this PR.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, an MCP `CallToolResult` with `content: [{ type: "resource", resource: { text: "..." } }]` passes through `toAgentToolResult` unchanged, then provider conversion sees no top-level text and emits a placeholder or empty output.

## Solution Assessment

Yes. Normalizing MCP `resource` content at the bundle-MCP to Pi `AgentToolResult` boundary is the narrowest maintainable fix because downstream provider transports already correctly consume Pi text/image blocks and should remain provider-generic.
