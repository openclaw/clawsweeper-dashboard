## Summary

Keep open: the report matches a current-main Discord plugin bug in the read-only/setup inspection paths. Runtime token resolution is snapshot-aware, but the inspectors still classify raw per-account SecretRefs as configured_unavailable before consulting the active resolved snapshot.

## What This Changes

The issue reports a 2026.5.3 regression where external Discord plugin per-account SecretRef tokens show configured_unavailable and keep accounts unusable until replaced with plaintext tokens.

## Best Possible Solution

Update the Discord plugin’s read-only and setup account inspectors to use the same runtime snapshot-aware token resolution as resolveDiscordToken, then add regression coverage for named per-account SecretRefs and a user-facing changelog entry.

## Reproduction Assessment

Yes: source inspection gives a high-confidence current-main reproduction path by setting a source config with channels.discord.accounts.work.token as a SecretRef plus an active runtime snapshot with the resolved token, then calling inspectDiscordAccount or inspectDiscordSetupAccount for work. I did not execute tests because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is inside the Discord plugin inspectors, because core already provides runtime snapshots and the Discord runtime resolver already uses them. Changing core SecretRef policy would be broader than needed.
