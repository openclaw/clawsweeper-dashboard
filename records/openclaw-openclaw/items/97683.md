## Summary

Keep open: the fix is aimed at the right IRC decoder boundary, but the current two-regex implementation still misses valid surrogate-pair escapes when they are preceded by another `\uXXXX` escape, which can change delivered IRC text.

## What This Changes

The PR changes the IRC literal-escape decoder to preserve lone surrogate-range `\uXXXX` sequences, decode valid surrogate-pair escapes, and adds protocol regression tests.

## Best Possible Solution

Keep the decoder-stage fix, but replace the two-pass non-overlapping regex with left-to-right `\u` escape handling that decodes valid pairs wherever they appear and preserves only unpaired surrogate escapes literally.

## Reproduction Assessment

Yes for the PR blocker: a focused Node check of the proposed algorithm shows `\u0041\uD83D\uDE00` is not decoded to `A😀`. Current main source and `v2026.6.10` also show the unguarded decoder bug the PR is trying to fix.

## Solution Assessment

No. The fix belongs in `decodeLiteralEscapes`, but the current two-pass regex is not the best shape because invalid non-overlapping pair matches can suppress later valid surrogate-pair decoding.
