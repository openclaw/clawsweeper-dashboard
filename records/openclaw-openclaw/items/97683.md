## Summary

Keep open: current main and `v2026.6.10` still decode surrogate-range IRC `\uXXXX` literals with `String.fromCharCode`, while this PR provides the focused guard, regression coverage, and sufficient live-output proof with no blocking findings.

## What This Changes

The PR updates the IRC literal escape decoder and protocol tests so surrogate-range `\uXXXX` values stay literal instead of becoming lone UTF-16 surrogate code units.

## Best Possible Solution

Keep this PR open and land the narrow decoder guard with its regression tests after normal maintainer review and required gates finish.

## Reproduction Assessment

Yes. Source inspection shows current main decodes `\uD800` and `\uDFFF` through `String.fromCharCode`, and a Node probe confirms those values become lone surrogate code units before IRC sanitization.

## Solution Assessment

Yes. Guarding the IRC literal escape decoder is the narrowest maintainable fix for this source of lone surrogates; broader outbound Unicode policy can remain separate from this focused decoder bug.
