## Label Justifications

- `P1`: The issue describes unexpected paid background provider usage from an existing heartbeat workflow after setup changes model routing.
- `impact:session-state`: Heartbeat can reuse the existing session/default model path, sending accumulated session context through the newly promoted provider.
- `impact:auth-provider`: The root path is provider-auth setup changing the default model and provider selection without preserving the prior primary.

## Summary

Keep open. Current main and the latest shipped release still omit `preserveExistingDefaultModel` from the setup provider-auth path, heartbeat can inherit that changed default, and the linked closing PR is still open rather than merged.

## What This Changes

Prevent setup provider auth from silently promoting a paid Google Gemini default model into existing heartbeat background traffic.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81304 or an equivalent narrow fix that forwards `preserveExistingDefaultModel: true` from setup provider auth, with regression coverage for existing primary models.

## Reproduction Assessment

Yes. By source inspection rather than a live billing run, setup still applies provider auth defaults without preserving an existing primary, and heartbeat without `heartbeat.model` falls through to the resolved session/default model.

## Solution Assessment

Yes. Reusing the existing `preserveExistingDefaultModel` contract in setup is the narrow maintainable fix; changing heartbeat policy or adding Google-specific behavior would be broader than necessary.
