## Summary

Keep open. Current main still lets setup/provider auth replace an existing primary model during Google API-key setup, and heartbeat runs without an explicit heartbeat model still inherit the resolved default model; the broader migration issue and the closed commitments issue do not supersede this narrow setup/heartbeat bug.

## What This Changes

The issue asks OpenClaw to prevent Google API-key setup from silently switching an existing default model so recurring heartbeat background runs start using paid Gemini.

## Best Possible Solution

Apply the existing default-preservation contract to setup provider auth when an existing primary model is present, while preserving first-run default selection and explicit model-picker choices.

## Reproduction Assessment

Yes, by source inspection. Start with an existing agents.defaults.model.primary and heartbeat enabled without heartbeat.model, run setup with the Google Gemini API-key auth choice, and the setup path can replace the primary while later heartbeat resolution falls through to that new default.

## Solution Assessment

Yes. The narrow maintainable repair is to pass the existing preserveExistingDefaultModel seam through setup/provider auth and add focused regression coverage, instead of special-casing Google or changing heartbeat runtime model selection.
