## Summary

Keep open: current main and the latest release still let setup apply provider auth with `setDefaultModel: true` without the existing-primary preservation flag, so configuring Google auth can still promote the paid Gemini default that heartbeat may inherit.

## What This Changes

Configuring Google Gemini auth through setup should not silently replace an existing primary model and route background heartbeat runs through paid Gemini usage.

## Best Possible Solution

Mirror configure-auth by passing `preserveExistingDefaultModel` through setup provider auth when an existing primary exists, add focused setup regression coverage, and leave broader paid-default consent policy to https://github.com/openclaw/openclaw/issues/74395.

## Reproduction Assessment

Yes, source-level: setup forwards `setDefaultModel` without `preserveExistingDefaultModel`, the provider-auth helper only preserves when that flag is true, and heartbeat can inherit the resulting default. I did not run a live paid Google billing reproduction.

## Solution Assessment

Yes for the narrow bug: reusing the existing preservation seam in setup is the smallest maintainable fix and avoids Google-specific or heartbeat-specific policy changes. Broader price-tier confirmation belongs in the related product-policy issue.
