## Summary

Keep open. Current main still has the paid-background-use path for setup: the setup wizard applies provider auth with default-model switching enabled and without existing-default preservation, Google API-key auth still supplies a Gemini default that can overwrite the primary model, and heartbeat only avoids normal default-model resolution when `heartbeat.model` is explicitly configured.

## What This Changes

Configuring a paid Google Gemini API key should not silently switch an existing default model and make heartbeat consume paid Gemini background turns without explicit user intent.

## Best Possible Solution

Add an explicit paid-background boundary in setup: preserve an existing primary model unless the user explicitly confirms switching it, ensure Google's API-key `applyConfig` path cannot bypass that preservation, and keep heartbeat paid usage behind an existing explicit default or `heartbeat.model` choice with regression coverage.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: start from a config with an existing primary model and non-empty `HEARTBEAT.md`, run setup with the Google Gemini API-key auth choice, then observe the setup path can switch the primary to Gemini and heartbeat without `heartbeat.model` resolves the normal default model in the main session.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the existing `preserveExistingDefaultModel` seam in the setup wizard and add Google/provider-auth regression tests, rather than special-casing heartbeat or Google in core runtime selection.
