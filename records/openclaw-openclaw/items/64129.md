## Summary

Keep open: current main and the latest release still leave the setup/onboard provider-auth path able to promote Google's default model over an existing primary, and heartbeat can inherit that changed default for background runs.

## What This Changes

The issue reports that configuring a paid Google Gemini key through setup changed the default model and caused existing heartbeat traffic to run as paid background Gemini usage.

## Best Possible Solution

Apply the existing default-preservation contract to setup provider auth, add focused setup regression coverage, and keep broader paid-default consent policy on the related product-policy issue.

## Reproduction Assessment

Yes, source-level: setup forwards `setDefaultModel` without `preserveExistingDefaultModel`, provider-auth only preserves when that flag is true, and heartbeat can inherit the resulting default. I did not run a live paid Google billing reproduction.

## Solution Assessment

Yes for the narrow bug: reusing the existing setup/provider-auth preservation seam is the smallest maintainable repair and avoids Google-specific or heartbeat-specific policy changes.
