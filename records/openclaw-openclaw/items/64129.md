## Summary

Keep open. Current main still contains the reported setup-to-heartbeat path: CLI setup can switch an existing primary model to Google Gemini during provider auth, and heartbeat without an explicit heartbeat model continues to resolve through the normal default/session model path. The prior ClawSweeper comment is still consistent with the code, and the later #75334 cross-reference is a related but distinct commitments extractor issue rather than a superseding fix.

## What This Changes

The issue asks OpenClaw to prevent paid Google Gemini auth setup from silently turning existing heartbeat background runs into paid Gemini usage before explicit user confirmation.

## Best Possible Solution

Keep the issue open and land a narrow regression fix that preserves an existing primary model during setup provider auth unless the user explicitly chooses to switch it, while keeping heartbeat paid-model use behind an explicit default or heartbeat.model choice and covering Google API-key behavior with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: start with an existing primary model and enabled heartbeat without heartbeat.model, run setup with the Google Gemini API-key auth choice, and the setup path can switch the primary to Gemini while heartbeat resolves the normal default model for its next main-session turn.

## Solution Assessment

Yes. The best repair direction is to use the existing preserveExistingDefaultModel seam in setup/provider auth and add focused Google/default-model/heartbeat regression coverage, rather than special-casing Google or heartbeat deep in core runtime selection.
