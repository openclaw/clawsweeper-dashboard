## Label Justifications

- `P1`: Unexpected paid background provider usage can affect real users immediately and is tied to an existing setup/auth workflow.
- `impact:auth-provider`: The issue is about provider auth changing model routing/default selection after a Gemini key is configured.
- `impact:session-state`: The reported heartbeat runs reuse existing session/default context when no heartbeat-specific model is configured.

## Summary

Keep open. Current main still lets setup provider auth apply a provider default without `preserveExistingDefaultModel`, and heartbeat can inherit that new default; the linked closing fix PR is still open and conflicting.

## What This Changes

The issue asks OpenClaw to stop provider setup from silently switching existing heartbeat/background traffic onto a newly configured paid Gemini model.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81304 or an equivalent narrow setup fix that passes the existing preservation flag and adds regression coverage for setup re-runs with an existing primary model.

## Reproduction Assessment

Yes, by source inspection. Setup applies provider auth with default-setting enabled but without default preservation, and heartbeat without `heartbeat.model` falls through to the session/default model.

## Solution Assessment

Yes. The narrow maintainable repair is to reuse the existing `preserveExistingDefaultModel` contract in setup provider auth rather than adding Google-specific or heartbeat-specific policy.
