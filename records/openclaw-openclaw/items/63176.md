## Summary

Keep this PR open because the requested plugin ACP runtime surface is still absent from current main and the downstream task-dispatch use case is concrete. The branch is useful and the author addressed earlier bot feedback around lazy loading, delivery target resolution, and missing runId handling, but it is not ready to merge as-is because the new privileged ACP surface is globally gated instead of plugin-scoped and the new runtime file violates current config-loading guardrails.

## What This Changes

The branch adds a gated `api.runtime.acp.spawn()`/`api.runtime.acp.prompt()` plugin runtime surface, a `plugins.allowAcpSpawn` config key, tests/mocks for the new runtime helpers, and Discord thread-binding snowflake normalization.

## Best Possible Solution

Land a rebased version only after the ACP capability is scoped to an explicitly trusted plugin identity, the runtime uses the active runtime config snapshot instead of ambient `loadConfig()`, and the public plugin runtime docs/changelog describe the operator opt-in and security model.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. The current-main gap and the PR blockers are statically reproducible with `rg` plus the config-boundary guard rules, and the PR body provides concrete manual validation steps for the intended downstream workflow.

## Solution Assessment

No, not as currently proposed. Exposing ACP through `api.runtime` is a plausible narrow direction, but the safer implementation needs per-plugin authorization and current-runtime config access before this privileged external-harness path is mergeable.

## Security Review

Status: needs_attention

Summary: The diff intentionally exposes a privileged ACP orchestration path, but the proposed authorization is a root flag that is not scoped to the calling plugin.
