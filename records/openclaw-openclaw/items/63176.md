## Summary

Keep open: current main still lacks the requested ACP plugin runtime helpers, and the PR addresses a real downstream plugin need, but the current patch is not merge-ready because it exposes host-side ACP orchestration through a global plugin flag and violates the runtime config boundary.

## What This Changes

The PR adds `api.runtime.acp.spawn()` and `api.runtime.acp.prompt()`, `plugins.allowAcpSpawn` config/schema wiring, runtime tests/mocks, and Discord channel snowflake normalization for ACP thread binding.

## Best Possible Solution

Land a rebased version only after ACP access is scoped to an explicitly trusted plugin identity, the gate uses the active runtime config path, and public docs/changelog explain the operator opt-in and host-side security model.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. The current-main gap and merge blockers are statically reproducible with `rg`, the PR patch, and the current config-boundary guard.

## Solution Assessment

No, not as currently proposed. Exposing ACP through `api.runtime` is a plausible narrow direction, but host-side ACP needs per-plugin authorization and active runtime config access before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope ACP runtime access to the calling plugin:** `src/plugins/runtime/runtime-acp.runtime.ts:23-25`
  - body: `assertAcpRuntimeEnabled()` only checks the root `plugins.allowAcpSpawn` flag, so enabling ACP for one trusted plugin grants every loaded plugin access to `api.runtime.acp.spawn()` and `prompt()`. Because ACP harnesses run host-side, this needs plugin runtime identity plus a per-plugin capability gate before merge.
  - confidence: 0.86
- **[P2] Use the active runtime config snapshot:** `src/plugins/runtime/runtime-acp.runtime.ts:23`
  - body: This new runtime module lives under `src/plugins/runtime`, where the config-boundary guard rejects ambient `loadConfig()` calls. Pass the active runtime config through the call path, or use the established runtime config snapshot seam, so the architecture check stays green after rebase.
  - confidence: 0.93
- **[P3] Document the ACP plugin runtime capability:** `src/plugins/runtime/types-core.ts:121-133`
  - body: The PR adds a public plugin runtime API and operator config key, but there is no plugin runtime doc or changelog entry for the contract, opt-in setting, and privilege warning. Add those before shipping this plugin-facing feature.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff intentionally exposes privileged host-side ACP orchestration to plugins, but authorization is a global flag rather than a per-plugin capability.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/plugins/runtime/runtime-acp.runtime.ts:23`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can call `api.runtime.acp.spawn()` or `prompt()`. Since ACP can launch and continue external host-side coding harness sessions, the gate should be tied to the calling plugin identity or another explicit trusted-plugin capability.
  - confidence: 0.86
