## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope ACP runtime access to the calling plugin:** `src/plugins/runtime/runtime-acp.runtime.ts:24-26`
  - body: `assertAcpRuntimeEnabled()` checks only the top-level `plugins.allowAcpSpawn` flag, so enabling ACP for one trusted plugin would give every loaded plugin host-side ACP spawn and prompt authority. Gate this through caller plugin identity and per-plugin capability before merging this API.
  - confidence: 0.9
- **[P2] Use the active runtime config snapshot:** `src/plugins/runtime/runtime-acp.runtime.ts:24`
  - body: This new runtime module calls ambient `loadConfig()` from under `src/plugins/runtime`, where runtime helpers are expected to use the active config snapshot or a passed config seam. This can trip the config-boundary guard and read a different config than the active gateway path.
  - confidence: 0.88
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` sends a gateway `agent` request without `acpTurnSource: "manual_spawn"`, unlike native ACP spawn. Follow-up prompts initiated by a trusted plugin can be treated as automatic dispatch and blocked when ACP dispatch policy is disabled.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff exposes privileged host-side ACP orchestration to plugins with a global opt-in instead of caller-scoped authorization.

Concerns:

- **[high] Global ACP gate grants all plugins spawn access:** `src/plugins/runtime/runtime-acp.runtime.ts:24`
  - body: With `plugins.allowAcpSpawn: true`, any loaded plugin can call the proposed ACP spawn and prompt helpers. Since those helpers can start or continue host-side coding harness sessions, the permission should be tied to the calling plugin identity or an equivalent per-plugin trust contract.
  - confidence: 0.9
