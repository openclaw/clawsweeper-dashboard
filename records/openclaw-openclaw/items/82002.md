## Label Justifications

- `P2`: This is a normal-priority Telegram channel bug with clear source impact and live reports, but it is bounded to reply-context handling rather than a core outage.
- `impact:session-state`: The bug can remove reply-chain context from the agent-visible turn, causing the session to drift from the user's Telegram reply target.
- `impact:message-loss`: The reported failure silently drops quoted bot-message context before it reaches the agent, making short Telegram replies behave like standalone messages.

## Summary

Keep open. Current main and the latest release still lack the bot/self supplemental-context exception, while the linked closing PR is still open and conflicting; the merged reply-chain cache work only covers part of the surrounding Telegram reply-context problem.

## What This Changes

The issue asks for Telegram replies to bot-sent messages to preserve agent-visible reply/quote context even when group allowlist visibility would otherwise filter the bot account.

## Best Possible Solution

Rebase and review https://github.com/openclaw/openclaw/pull/82079, or land an equivalent narrow replacement, then leave broader durable reply-context work to https://github.com/openclaw/openclaw/issues/88032 and stale commitment delivery to https://github.com/openclaw/openclaw/issues/86356.

## Reproduction Assessment

Yes for source-level reproduction: configure Telegram `contextVisibility: "allowlist"` with a group allowlist that admits the user but not the bot, then reply to a bot-sent message. Current main can filter that reply target because the bot sender is checked only against `effectiveGroupAllow`; I did not run a fresh live Telegram session.

## Solution Assessment

Yes, the narrow best fix is to treat Telegram bot/self reply targets as allowed supplemental quote context before the allowlist gate, while keeping non-bot forwarded-origin redaction intact. The broader first-class durable reply-context contract is already tracked separately.
