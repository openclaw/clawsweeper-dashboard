## Summary

Keep open. Current main partially satisfies the request through the shared `channels.modelByChannel` surface for channel/group-style conversations, including the Discord channel interest raised in discussion, but ordinary DM matching is still not wired and `openclaw config schema` still exports `channels.properties` as empty.

## What This Changes

Add persistent model overrides in config for channel, group, and direct-message conversations.

## Best Possible Solution

Finish the existing generic `channels.modelByChannel` contract: add stable direct-message candidate resolution from finalized context and persisted session origin data, expose the shared key in the exported config schema, document accepted DM key forms, and preserve the current precedence where explicit session/runtime model overrides beat config defaults.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: a direct conversation with `channels.modelByChannel.<provider>.<dm-key>` has no DM candidate inputs in the resolver call, while `openclaw config schema` is backed by a generated schema whose `channels.properties` is empty.

## Solution Assessment

Yes. Completing `channels.modelByChannel` is the narrowest maintainable direction because the shared config surface already exists and avoids provider-specific `channels.whatsapp.*.model` or `channels.slack.*.model` shapes.
