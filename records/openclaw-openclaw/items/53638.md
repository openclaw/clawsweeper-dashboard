## Summary

Keep open. Current main exposes the shared `channels.modelByChannel` config and schema/help surface, and channel/group matching exists, but ordinary DM matching still lacks direct-peer resolver inputs, direct-peer candidate construction, and documented DM key forms.

## What This Changes

The issue requests persistent per-channel, per-group, and per-DM model selection in config, below explicit session overrides and above global defaults.

## Best Possible Solution

Complete the shared `channels.modelByChannel` contract for direct conversations by deriving normalized DM candidates from message context, persisted origin/session data, native provider IDs, and direct session keys while preserving current override precedence.

## Reproduction Assessment

Yes, at source level: configure a direct-message entry under `channels.modelByChannel`, then route a DM whose identifiers live in `OriginatingTo`, `NativeDirectUserId`, `From`, `To`, or session origin fields rather than `groupId` or a parent channel key. Current main has those identifiers available but never passes them into the resolver.

## Solution Assessment

Yes. Extending the existing shared `channels.modelByChannel` resolver and docs is the narrow maintainable direction, and it avoids adding provider-specific nested model fields with competing precedence rules.
