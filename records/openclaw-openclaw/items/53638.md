## Summary

Keep open. Current main and the latest release have the shared channels.modelByChannel surface for channel/group targets, but ordinary per-DM matching still lacks resolver inputs/candidate construction, and the public schema/docs do not expose stable DM key forms.

## What This Changes

Request persistent config-level model overrides for channel, group, and DM conversations, preferably through the existing shared channels.modelByChannel contract rather than provider-specific nested model fields.

## Best Possible Solution

Complete the generic channels.modelByChannel contract for direct conversations by deriving normalized DM candidates from finalized context, stored session origin, native ids, and direct session keys, then document and expose those key forms without adding provider-specific nested model fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a specific DM entry under channels.modelByChannel, then process a direct message whose context only identifies the peer through SessionKey, OriginatingTo, To, NativeChannelId, or stored origin fields; the current resolver never receives those values.

## Solution Assessment

Yes. Finishing the shared channels.modelByChannel resolver/docs/schema path is the maintainable solution because it avoids separate WhatsApp, Slack, Discord, or Telegram nested model fields with competing precedence rules.
