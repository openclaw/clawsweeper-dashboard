## Summary

Keep open. Current main supports the channel/group part through the generic `channels.modelByChannel` surface, but specific per-DM matching and the public config-schema/docs surface for DM key forms are still incomplete.

## What This Changes

The issue requests persistent per-channel, per-group, and per-DM model selection in `openclaw.json` instead of ephemeral runtime `/model` overrides.

## Best Possible Solution

Complete the generic `channels.modelByChannel` contract for direct conversations, expose it in the public config schema and docs, and avoid adding competing provider-specific nested `model` fields.

## Reproduction Assessment

Yes, at source level: configure a specific DM entry under `channels.modelByChannel`, then route a direct message whose available identifiers are session/direct-peer fields rather than `groupId` or parent channel fields. Current resolver inputs cannot build that specific DM candidate.

## Solution Assessment

Yes. Completing the shared `channels.modelByChannel` resolver/schema/docs path is the maintainable solution and avoids provider-specific nested `model` fields with competing precedence rules.
