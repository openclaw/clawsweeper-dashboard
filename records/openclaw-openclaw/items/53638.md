## Summary

Keep open. Current main has the shared `channels.modelByChannel` config, schema, docs, and channel/group matching, but ordinary DM-specific matching and documented DM key forms are still missing; the adjacent open ingress PR only closes a different runtime path for `channels.modelByChannel`.

## What This Changes

The issue requests persistent model selection per channel, group, and DM in `openclaw.json` so conversations can use different models without manual runtime overrides.

## Best Possible Solution

Finish the existing `channels.modelByChannel` contract for direct conversations by deriving normalized DM candidates from finalized context, persisted origin/session data, native provider IDs, and direct session keys while preserving current override precedence.

## Reproduction Assessment

Yes, at source level: configure a specific DM entry under `channels.modelByChannel`, then route an ordinary DM whose identifiers live in context/session origin rather than `groupId` or a parent group/channel key. Current main has those identifiers available but never passes them into `resolveChannelModelOverride`.

## Solution Assessment

Yes. Extending the existing shared `channels.modelByChannel` resolver and docs is the narrow maintainable direction, and it avoids adding provider-specific nested `model` fields with competing precedence rules.
