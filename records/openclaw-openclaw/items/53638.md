## Summary

Keep open. Current main and v2026.5.22 have the shared `channels.modelByChannel` config, schema, docs, and channel/group matching, but specific per-DM matching remains unfinished because the resolver still receives group/thread-style inputs and does not accept ordinary direct session keys.

## What This Changes

The issue requests persistent model selection in config at channel, group, and DM scope so conversations can use different models without ephemeral `/model` overrides.

## Best Possible Solution

Finish the existing `channels.modelByChannel` contract for direct conversations by deriving normalized DM candidates from context, session metadata, direct session keys, and plugin-owned conversation bindings, then document the accepted DM key forms without adding provider-specific nested `model` fields.

## Reproduction Assessment

Yes, at source level. Configure a specific DM entry under `channels.modelByChannel` and route an ordinary direct conversation; current source has no direct-peer candidate inputs and direct session keys are not accepted by `parseRawSessionConversationRef`.

## Solution Assessment

Yes. Extending the existing shared `channels.modelByChannel` resolver and docs is the narrow maintainable direction, and it avoids competing provider-specific `model` fields.
