## Summary

Keep open. Current main has the shared `channels.modelByChannel` config, schema, docs, and group/channel resolver path, but the per-DM part remains unfinished because ordinary direct-peer identifiers are not passed into the resolver and direct session keys are not parsed as model override conversation refs.

## What This Changes

The issue requests persistent config-level model overrides per channel, group, or DM, falling back to channel defaults and then the global agent default.

## Best Possible Solution

Finish `channels.modelByChannel` for direct conversations in the shared resolver, document the accepted DM key forms, and avoid adding competing provider-specific nested `model` fields.

## Reproduction Assessment

Yes. At source level, configure a specific DM entry under `channels.modelByChannel` and route an ordinary direct conversation; current main has no ordinary direct-peer candidate input, and direct session keys are rejected by the group/channel-only conversation-ref parser.

## Solution Assessment

Yes. Extending the existing generic `channels.modelByChannel` resolver is the narrow maintainable path because current main already chose that shared config surface and it avoids provider-specific duplicate `model` fields.
