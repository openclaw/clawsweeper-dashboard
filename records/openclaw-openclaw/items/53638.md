## Summary

Keep open. Current main has the shared `channels.modelByChannel` surface and a direct-chat wildcard path, but specific per-DM matching and public config-schema/docs coverage for DM key forms are still incomplete.

## What This Changes

Add persistent config-based model selection per channel, group, and DM, using the shared channel model override surface rather than ephemeral `/model` overrides.

## Best Possible Solution

Finish the generic `channels.modelByChannel` contract for direct conversations by deriving normalized DM candidates from finalized context, stored session origin, native ids, and direct session keys, then expose and document those key forms without adding provider-specific nested `model` fields.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure a specific DM entry under `channels.modelByChannel`, then route a direct message whose `SessionKey`/`OriginatingTo` identify that DM but whose resolver inputs have no `groupId` or parent-session key. The current resolver cannot build that per-DM candidate and only provider wildcard direct overrides are covered.

## Solution Assessment

Yes, completing the shared `channels.modelByChannel` contract is the maintainable path. It preserves the established generic config surface and avoids adding separate WhatsApp, Slack, or Discord nested `model` fields with competing precedence rules.
