## Summary

Keep open. Current main covers channel/group-style pins through the shared `channels.modelByChannel` path, but specific DM matching still lacks direct-peer resolver inputs and `openclaw config schema` still strips channel child properties from the public schema output.

## What This Changes

Request persistent per-channel, per-group, and per-DM model selection in config instead of only global defaults or ephemeral session overrides.

## Best Possible Solution

Complete the shared `channels.modelByChannel` contract for direct conversations and expose that existing shared key in schema/docs, without adding provider-specific nested `model` fields.

## Reproduction Assessment

Yes, at source level. A specific direct-message entry under `channels.modelByChannel` cannot match when the resolver only receives direct chat context without `groupId` or parent-session candidates, because ordinary direct-peer identifiers are not accepted as resolver inputs.

## Solution Assessment

Yes. The maintainable path is to finish the existing shared `channels.modelByChannel` resolver, schema, docs, and tests rather than adding provider-specific nested model fields with separate precedence rules.
