## Summary

Keep open. Current main and v2026.6.8 already have the shared `channels.modelByChannel` config/schema surface for channel and group routing, but specific DM overrides still cannot match ordinary direct conversations because the resolver and its callers do not pass direct peer identifiers.

## What This Changes

Request persistent config-level model routing for channel, group, and DM conversations instead of ephemeral `/model` overrides for those contexts.

## Best Possible Solution

Complete the existing generic `channels.modelByChannel` direct-conversation path, document supported DM key forms, and avoid adding provider-specific nested `model` fields unless maintainers deliberately choose a new config surface.

## Reproduction Assessment

Yes. Source inspection shows a specific DM override cannot match ordinary direct conversations because resolver inputs omit direct peer ids, parent-session parsing excludes `direct`, and tests only cover wildcard direct fallback.

## Solution Assessment

No, not in the originally requested shape. The maintainable path is to finish the existing shared `channels.modelByChannel` direct-message resolver instead of adding provider-specific nested `model` fields.
