## Summary

Keep open. Current main still does not implement the requested compaction modes, and this PR remains useful source work, but the patch is not merge-ready because the new guard does not consistently honor the documented mode semantics and it conflicts with current-main compaction machinery.

## What This Changes

The PR extends agent compaction config/schema/help with `warn`, `error`, and `none`, adds a Pi pre-prompt token guard, changes Pi auto-compaction and reserve-floor behavior, adds a changelog entry, and adds a bundled `ceospace` skill placeholder.

## Best Possible Solution

A mergeable version should rebase onto current main, remove unrelated skill scope, settle the user-visible semantics for each mode, and implement those semantics through the existing precheck, overflow-recovery, config schema/help/docs, hooks/finalization, and focused tests rather than adding a parallel guard path.

## Reproduction Assessment

Yes. For `warn`, source inspection of PR head shows the warning path leaves `skipPromptForCompactionGuard` false and therefore still reaches prompt submission. For `none`, the PR only disables Pi auto-compaction while current overflow recovery still calls `contextEngine.compact()`.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the best implementation because it adds a second token guard instead of extending the current precheck and overflow lifecycle, and it changes reserve-floor behavior that current main already fixed more narrowly.

- **[P1] Stop warn mode before submitting the prompt:** `src/agents/pi-embedded-runner/run/attempt.ts:2924-2933`
  - body: The `warn` branch emits the warning and appends assistant text, but it never sets `skipPromptForCompactionGuard`. Execution then enters the later prompt block, so a near-limit warn-mode turn still calls the model even though the mode is described as stopping before overflow.
  - confidence: 0.93
- **[P1] Honor none mode in overflow recovery:** `src/agents/pi-settings.ts:109-110`
  - body: Disabling Pi's internal auto-compaction for `mode === "none"` does not disable OpenClaw's separate overflow-recovery loop, which still compacts and retries after model-limit errors. Thread the resolved mode into that recovery path so `none` can actually fail at the model limit as documented.
  - confidence: 0.89
- **[P2] Reuse the existing precheck budget path:** `src/agents/pi-embedded-runner/run/attempt.ts:2918-2921`
  - body: The new guard computes its own raw token total and threshold instead of using the existing precheck path that applies the safety margin, effective reserve capping, unwindowed pressure, and tool-result routing. That can make `warn` and `error` fire at different boundaries from the real model-submission budget.
  - confidence: 0.84

  - body: Lowering `DEFAULT_PI_COMPACTION_RESERVE_TOKENS_FLOOR` from 20,000 to 1,000 weakens the default reserve for large-context models. Current main already caps the 20,000 floor for small windows, so this global reduction would regress the intended headroom outside the small-window case.
