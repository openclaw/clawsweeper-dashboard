## Summary

Keep open: the issue has fresh 2026.5.7 live confirmation, no protected-label or maintainer-authored cleanup rule applies, and current source/tests still define Telegram per-peer DM session persistence as expected behavior rather than obsolete behavior.

## What This Changes

Telegram direct-message turns with `session.dmScope: "per-channel-peer"` should persist and list a durable `agent:<id>:telegram:direct:<peer>` session entry instead of creating unrecoverable per-turn history.

## Best Possible Solution

Add a full Telegram private-message regression that proves two consecutive DMs reuse and list the same per-peer session, then fix the first writer or lookup path that creates per-turn JSONL files without durably upserting the session key.

## Reproduction Assessment

No, not on current `main`. The issue now has strong v2026.5.6 and v2026.5.7 live evidence, but I did not establish a current-main live Telegram reproduction and the current focused tests exercise only narrower persistence paths.

## Solution Assessment

Unclear until the full pipeline failure is pinned down. The best repair direction is narrow: preserve existing dmScope semantics and fix the specific Telegram/session-store path that fails to persist the route-derived key.
