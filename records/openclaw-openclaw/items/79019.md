## Summary

Keep open: the report is recent, has two independent live confirmations on v2026.5.6 and v2026.5.7, and current main still has no post-release implementation change that clearly fixes Telegram direct-session persistence.

## What This Changes

Telegram direct messages with `session.dmScope: "per-channel-peer"` should persist and index `agent:<id>:telegram:direct:<peerId>` sessions so history is recoverable across turns.

## Best Possible Solution

Add a focused regression that drives a Telegram direct DM through the same dispatch path used in production, then make the durable session store and agent transcript lookup consistently use the route-derived per-peer key.

## Repair Work Prompt

Fix the Telegram direct-session persistence regression from https://github.com/openclaw/openclaw/issues/79019. With `session.dmScope: "per-channel-peer"`, consecutive Telegram DMs from the same peer must persist and reuse `agent:<id>:telegram:direct:<peerId>` in `sessions.json`, appear in `openclaw sessions`, and load prior JSONL history instead of starting each turn with `messages=0`. Start by adding a regression that exercises the production Telegram inbound dispatch/agent-session lookup path, not only `buildTelegramMessageContextForTest` plus `turn.recordInboundSession`; the existing recreate helper already covers that narrower path and did not catch the release bug. Investigate `extensions/telegram/src/bot-message-context.session.ts`, `extensions/telegram/src/bot-message-dispatch.ts`, `src/channels/session.ts`, `src/auto-reply/reply/dispatch-from-config.ts`, `src/agents/command/session.ts`, and `src/config/sessions/*` for stale store cache, exact-vs-normalized key lookup, async metadata recording, or a path that re-derives/collapses the direct key. Keep the fix generic at the channel/session seam where possible, do not add Telegram IDs or compatibility migrations to core runtime, and add a single-line changelog fix entry if the branch ships a user-visible repair.
