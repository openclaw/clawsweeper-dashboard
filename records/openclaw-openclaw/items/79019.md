## Summary

Keep open: the report now has two independent live confirmations through v2026.5.7, and current docs/tests define per-channel-peer Telegram DM session rows as durable session-store entries, but I did not establish a failing current-main live reproduction during this read-only pass.

## What This Changes

Telegram DMs with `session.dmScope: "per-channel-peer"` should persist and list `agent:<id>:telegram:direct:<peerId>` session rows instead of starting fresh JSONL sessions each turn.

## Best Possible Solution

Add a current-main regression that drives a real Telegram private inbound turn through queueing, session init, JSONL creation, and `sessions.json` listing, then fix the first writer or store-maintenance path that drops the direct session row.

## Reproduction Assessment

No, not yet for current main. The issue has strong v2026.5.6 and v2026.5.7 live reproduction evidence, but the checked current-main source path should persist `route.sessionKey`, so a focused current-main test or live Telegram check is still needed.

## Solution Assessment

Unclear. Reusing the route-derived session key for persistence is the narrow maintainable direction if a writer re-derives DMs, but the standard path already does that on current main, so the fix should first identify the divergent writer/store path instead of changing `resolveSessionKey` broadly.
