## Summary

Keep open: the configured `dmScope: "per-channel-peer"` behavior is documented as an existing isolation contract, and the issue now has two live reports, including a 2026.5.7 confirmation, showing Telegram DMs route and reply but do not become recoverable direct-session entries. Current main has no obvious post-release fix or open PR specifically closing this report.

## What This Changes

The issue reports that Telegram direct messages under `session.dmScope: "per-channel-peer"` reply successfully but do not persist or list a recoverable `agent:<id>:telegram:direct:<peer>` session entry.

## Best Possible Solution

Add a full Telegram private-message regression for `dmScope: "per-channel-peer"`, then fix the first runtime writer or queue path that creates a JSONL without a stable `sessions.json` entry for the route-derived direct session key.

## Reproduction Assessment

No high-confidence current-main reproduction was established in this review. The issue has strong live reproduction evidence on 2026.5.6 and 2026.5.7, but the standard current-main source path still appears to persist the route-derived key, so the failing full runtime path needs to be captured.

## Solution Assessment

Unclear until the divergent writer path is identified. Reusing the route-derived `SessionKey` for all persistence is the right narrow direction, but the safest repair starts with a full Telegram reply-turn regression rather than changing `resolveSessionKey` semantics globally.
