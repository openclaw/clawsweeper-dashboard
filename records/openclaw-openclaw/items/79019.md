The issue asks OpenClaw to persist and list Telegram direct-message sessions under the route-derived `agent:<id>:telegram:direct:<peerId>` key when `session.dmScope` is `per-channel-peer`.

## Best Possible Solution

Add a full Telegram private-message regression that proves two consecutive DMs reuse and list the same per-peer session, then fix the first writer, lookup, or timing path that creates per-turn JSONL files without durably upserting the session key.
