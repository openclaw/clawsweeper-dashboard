## Summary

Keep open: two independent live reports show Telegram DMs route and reply but do not produce a durable `telegram:direct:*` session row, and that violates the documented `per-channel-peer` DM isolation/session listing contract. Current source inspection shows the standard path should persist the route-derived key, so cleanup should not close this until a focused current-main regression identifies the divergent writer, concurrency, or maintenance path.

## What This Changes

Telegram direct-message sessions with `session.dmScope: "per-channel-peer"` should persist and list under `agent:<id>:telegram:direct:<peerId>` instead of starting fresh each turn.

## Best Possible Solution

Add a current-main regression that drives a real Telegram private inbound turn through the session-store writer, then fix the first path that fails to upsert or retain the route-derived direct key while preserving shared-DM behavior for `dmScope: "main"`.

## Reproduction Assessment

Yes for the latest shipped release via two redacted live Telegram reports and gateway logs; not yet as an executed current-main repro in this checkout. The focused reproduction path is to send repeated allow-listed Telegram DMs with `dmScope: "per-channel-peer"`, then inspect `sessions.json` and `openclaw sessions --json` for the expected `agent:main:telegram:direct:<peer>` row.

## Solution Assessment

Unclear until the failing writer is isolated. Reusing the route-derived `SessionKey` is the right direction, but current main's standard source path already does that, so the safest fix is a focused regression plus a narrow repair to the divergent Telegram/native-command/session-store path.
