## Summary

Keep open: the latest issue discussion includes fresh v2026.5.7 live evidence of Telegram DM context loss, and current main still documents and routes `per-channel-peer` direct sessions as durable per-sender sessions rather than obsolete behavior.

## What This Changes

The issue reports that Telegram DMs with `session.dmScope: "per-channel-peer"` route and reply but do not persist/list `agent:<id>:telegram:direct:<peer>` session entries.

## Best Possible Solution

Add a full Telegram private-message regression that proves consecutive DMs reuse and list the same per-peer session, then fix the first writer or lookup path that creates fresh JSONLs without durably upserting the direct session key.

## Reproduction Assessment

No, not against current main with a live Telegram run. The issue has strong v2026.5.6 and v2026.5.7 live evidence, and current source still preserves the expected contract, but the full current-main failure path has not been reproduced here.

## Solution Assessment

Unclear until the full pipeline failure is pinned down. The maintainable direction is narrow: preserve existing `dmScope` semantics and fix whichever Telegram/session-store path fails to persist the route-derived direct key.
