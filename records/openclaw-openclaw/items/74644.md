## Summary

Keep open. Current main still routes media audio provider execution through provider auth and `requireApiKey` before calling the plugin's `transcribeAudio`, and the open closing PR that addresses this remains unmerged.

## What This Changes

The issue asks OpenClaw to let plugin-owned local/no-auth media audio providers run without failing provider API-key resolution.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75005 or an equivalent explicit media-provider synthetic/no-auth opt-in with regression coverage and docs.

## Reproduction Assessment

Yes, by source inspection. Configure a plugin-only media audio provider with no credentials and trigger transcription; current main resolves provider auth with `requireApiKey` before the provider handler can run.

## Solution Assessment

Yes in direction, but it is not merged yet. A provider-owned synthetic/no-auth opt-in is narrower than a global auth bypass because normal profile, env, literal, and configured-provider auth can still take precedence.
