## Summary

Keep open. Current main still routes media audio provider execution through model-provider auth and `requireApiKey` before calling the plugin's `transcribeAudio`, and the specific closing implementation remains open and unmerged at https://github.com/openclaw/openclaw/pull/75005.

## What This Changes

The issue requests that plugin media audio providers support explicit local/no-auth STT execution without being blocked by the core API-key resolver.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75005 or an equivalent provider-owned media synthetic-auth opt-in with regression coverage and docs.

## Reproduction Assessment

Yes. By source inspection, configure a plugin-only media audio provider with no credentials and trigger transcription; current main reaches `resolveProviderExecutionAuth`, calls `requireApiKey`, and can fail before `transcribeAudio` runs.

## Solution Assessment

Yes in direction, but it is not on main yet. The linked PR's explicit provider-owned synthetic/no-auth fallback is narrower than a broad missing-auth bypass because normal literal, env, profile, and configured-provider auth still take precedence.
