## Summary

Keep open. Current main still requires resolved provider auth before calling media audio/video provider implementations, and open PR #75005 is already the canonical closing implementation candidate for this issue.

## What This Changes

Allow media-understanding audio/STT providers that explicitly represent local or no-auth services to run without a real API key.

## Best Possible Solution

Review and land PR #75005, or an equivalent explicit provider-owned synthetic/no-auth media auth hook, with regression coverage for local audio/video providers and docs that distinguish opted-in no-auth providers from missing credentials.

## Reproduction Assessment

Yes. Current main reproduces from source: a media audio provider without resolved credentials reaches resolveProviderExecutionContext, then requireApiKey, before transcribeAudio can run.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit opt-in synthetic/no-auth hook is narrower than bypassing media auth globally and preserves credential failures for unmarked providers.
