## Summary

Keep open. Current main still resolves media provider auth before invoking audio/video media-understanding providers, and the linked open PR #75005 is the existing implementation path with closing syntax for this issue.

## What This Changes

The issue asks for mediaUnderstandingProviders to support explicitly local or no-auth audio/STT providers without requiring auth-store API-key credentials first.

## Best Possible Solution

Review and land PR #75005, or an equivalent explicit provider-owned synthetic/no-auth media execution hook, with focused audio/video regression coverage and docs that distinguish opted-in local providers from missing credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a plugin-only audio media provider without provider credentials and trigger transcription; current main reaches resolveProviderExecutionContext and requireApiKey before transcribeAudio.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit opt-in synthetic/no-auth hook is the narrow maintainable solution because it avoids bypassing auth globally for ordinary remote providers.
