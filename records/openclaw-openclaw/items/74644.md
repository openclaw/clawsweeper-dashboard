## Summary

Keep open. Current main still resolves provider execution auth and requires an API key before invoking audio media-understanding providers, and open PR #75005 is already the canonical implementation candidate for this issue.

## What This Changes

Allow explicitly local or no-auth media understanding audio/STT providers to run without requiring an auth-store or API-key credential.

## Best Possible Solution

Review and land PR #75005 or an equivalent explicit provider-owned synthetic/no-auth media auth hook, with tests for local audio/video providers and docs that distinguish opted-in no-auth providers from missing credentials.

## Reproduction Assessment

Yes. Current main reproduces from source: configure a media audio provider without credentials and the runner reaches requireApiKey before transcribeAudio, so the provider implementation never gets control.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit opt-in synthetic/no-auth hook is narrower than bypassing media auth globally and preserves the existing credential failure for unmarked providers.
