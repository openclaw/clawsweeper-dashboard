## Summary

Keep open. Current main still resolves media audio/video provider auth through the model-provider API-key path before invoking the plugin media provider, and the provided GitHub context shows open PR #75005 already references this issue with closing syntax, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

Allow plugin-owned local or no-auth media-understanding audio providers to run without hard-requiring auth-store API-key credentials.

## Best Possible Solution

Land PR #75005 or an equivalent explicit provider-owned synthetic/no-auth media execution hook, with focused audio/video regression coverage and docs that distinguish opted-in local providers from ordinary missing credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure a plugin-only audio media provider without provider credentials and trigger transcription; current main reaches resolveProviderExecutionAuth before transcribeAudio.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit opt-in synthetic/no-auth hook is the narrow maintainable fix because it avoids bypassing auth globally for unmarked remote providers.
