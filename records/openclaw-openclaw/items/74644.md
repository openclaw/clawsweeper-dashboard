## Summary

Keep open. Current main still routes media audio/video execution through provider API-key resolution before calling the media provider, and the linked fix PR is open and unmerged, so the issue should remain open until that fix or an equivalent one lands.

## What This Changes

The issue asks for media-understanding audio/STT providers to support explicit local or no-auth execution without requiring auth-store API keys.

## Best Possible Solution

Land PR #75005 or an equivalent explicit provider-owned synthetic/no-auth media execution hook, with focused audio/video regression coverage and docs that distinguish opted-in local providers from ordinary missing credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure a plugin-only audio media provider without provider credentials and trigger transcription; current main reaches resolveProviderExecutionAuth before transcribeAudio.

## Solution Assessment

Yes in direction, but it is not on main yet. An explicit opt-in synthetic/no-auth hook is the narrow maintainable fix because it avoids a global credential bypass for unmarked remote providers.
