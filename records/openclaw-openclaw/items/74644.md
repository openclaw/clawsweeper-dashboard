## Summary

Keep open. Current main still resolves model-provider auth and calls `requireApiKey` before invoking a media provider's `transcribeAudio`; the existing synthetic local fallback is tied to model-provider config/provider hooks, not a media-only provider registration. Open PR #75005 explicitly targets this issue and is still unmerged, so the issue should remain open pending that implementation path.

## What This Changes

Allow local or no-auth media understanding audio/STT provider plugins to run without requiring auth-store API keys.

## Best Possible Solution

Land PR #75005 or an equivalent explicit media-provider synthetic-auth opt-in that covers audio and video without globally bypassing auth for ordinary providers.

## Reproduction Assessment

Yes. Configure a plugin-only media audio provider such as `local-audio` with no auth profile, env key, or `models.providers.<id>.apiKey`, then trigger transcription; current main resolves model-provider auth before `transcribeAudio` and can throw the missing-key error.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow maintainable fix is an explicit provider-owned media no-auth/synthetic-auth opt-in rather than treating every missing key as acceptable.
