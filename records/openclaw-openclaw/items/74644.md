## Summary

Keep open. Current main still routes audio/video media-understanding provider execution through provider auth resolution and `requireApiKey` before invoking plugin provider code, and the issue is already paired with open, unmerged PR #75005 as the canonical implementation path.

## What This Changes

Allow plugin-owned local or no-auth media-understanding audio/STT providers to opt into execution without failing API-key resolution before `transcribeAudio` runs.

## Best Possible Solution

Review and land or revise PR #75005 as the canonical fix: add an explicit provider-owned no-auth/synthetic media auth contract for local audio/video providers while preserving fail-closed behavior for unmarked providers.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: configure a plugin media audio provider without credentials and `runProviderEntry` reaches `resolveProviderExecutionContext`/`requireApiKey` before calling `transcribeAudio`.

## Solution Assessment

Yes, in direction. An explicit opt-in media auth hook is narrower and safer than bypassing provider auth globally because ordinary remote providers without credentials continue to fail closed.
