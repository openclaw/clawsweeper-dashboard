## Summary

Keep open. Current main still requires resolved provider auth before invoking audio media-understanding providers, and the open, unmerged PR #75005 is already the canonical implementation path for this issue.

## What This Changes

The issue requests a first-class no-auth or synthetic-auth path so local/self-hosted media-understanding audio providers can transcribe without auth-store API keys.

## Best Possible Solution

Review and land PR #75005, or an equivalent explicit provider-owned synthetic/no-auth media auth hook, with audio/video tests and docs that distinguish opted-in no-auth providers from missing credentials.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure a plugin-only audio media provider without credentials, trigger audio transcription, and the runner reaches requireApiKey before transcribeAudio.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit opt-in synthetic/no-auth hook is narrower and safer than bypassing media auth globally because unmarked providers still fail closed.
