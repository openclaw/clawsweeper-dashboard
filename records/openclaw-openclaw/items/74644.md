## Summary

Keep open. Current main still resolves media audio provider execution through provider auth and `requireApiKey` before invoking plugin `transcribeAudio`, while the linked closing PR #75005 is still open and unmerged in the provided GitHub context.

## What This Changes

Allow plugin-owned local or no-auth media audio providers to run STT without auth-store API keys when they explicitly opt into that behavior.

## Best Possible Solution

Land PR #75005 or an equivalent explicit media-provider synthetic/no-auth opt-in with regression coverage and docs, while preserving the current fail-closed default for unmarked providers.

## Reproduction Assessment

Yes, by source inspection. Configure a plugin-only audio media provider without credentials and trigger transcription; current main resolves provider auth with `requireApiKey` before calling `transcribeAudio`, so the provider never gets a chance to handle a local no-auth request.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit plugin/provider opt-in for synthetic non-secret media auth is the narrow maintainable fix; a global auth bypass would be riskier.
