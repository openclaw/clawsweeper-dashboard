## Summary

Keep open. Current main still resolves media provider auth before calling audio/video provider implementations, and an open unmerged PR explicitly tracks the canonical fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests a no-auth/local execution path for `mediaUnderstandingProviders` so plugin-owned audio/STT providers can run without auth-store API keys.

## Best Possible Solution

Review and land or revise PR #75005 as the canonical fix: add an explicit synthetic/no-auth media provider hook for local/self-hosted audio and video providers, document it for plugin authors, and keep ordinary providers without credentials failing closed.

## Reproduction Assessment

Yes. Current main reproduces from source: configure a plugin media audio provider without credentials and the runner reaches `resolveProviderExecutionContext` and `requireApiKey` before calling `transcribeAudio`.

## Solution Assessment

Yes in direction, but not yet merged. An explicit opt-in synthetic auth hook is the narrowest maintainable solution because it supports local/no-auth media providers without weakening auth checks for normal remote providers.
