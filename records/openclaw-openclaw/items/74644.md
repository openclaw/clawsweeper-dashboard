## Summary

Keep open. Current main still routes media audio provider execution through provider auth and `requireApiKey` before invoking `transcribeAudio`, and PR #75005 is the open closing implementation path rather than a merged fix.

## What This Changes

Allow mediaUnderstandingProviders to support explicitly local or no-auth audio transcription without requiring auth-store API keys.

## Best Possible Solution

Land PR #75005 or an equivalent explicit media-provider no-auth/synthetic-auth opt-in with regression coverage and docs, while preserving fail-closed behavior for unmarked providers.

## Reproduction Assessment

Yes, by source inspection. Configure an explicit plugin-only audio media provider without credentials and current main resolves provider auth with `requireApiKey` before calling `transcribeAudio`.

## Solution Assessment

Yes in direction, but it is not merged yet. An explicit provider opt-in is the narrow maintainable fix; a global media auth bypass would be riskier.
