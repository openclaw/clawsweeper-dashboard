## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Stop shipping the Linux app as the CLI client:** `apps/linux-gtk/crates/gateway-client/src/protocol.rs:221-228`
  - body: The gateway client IDs are a closed contract, but this new desktop app hardcodes `cli` mode and requests admin, approvals, and pairing scopes. That bypasses a Linux-specific client policy and gives the app CLI-equivalent authority before maintainers have accepted that security model.
  - confidence: 0.94
- **[P1] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }`; this sends `{ path, value }`, so the save action is rejected instead of persisting the edited config.
  - confidence: 0.97
- **[P1] Use skillKey when toggling skills:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: `skills.update` requires `skillKey`, not `skillId`. The current toggle request fails validation, leaving the switch state out of sync with the actual installed skill state.
  - confidence: 0.96
- **[P2] Use a valid skills.install payload:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` accepts the legacy `{ name, installId }`, ClawHub, or upload payload shapes; sending only `skillId` cannot install the selected skill.
  - confidence: 0.93
- **[P1] Wrap cron enablement under patch:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-331`
  - body: `cron.update` requires mutable fields under `patch`. This sends `enabled` at the top level while optimistically changing the UI chip, so validation failure leaves the app showing a state that was not saved.
  - confidence: 0.96
- **[P2] Probe channels with supported status params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `channel`, `probe`, and `timeoutMs`; `channelId` is not part of the schema, so the per-channel probe path will not target the selected channel correctly.
  - confidence: 0.9
- **[P2] Call the registered gateway restart method:** `apps/linux-gtk/crates/app/src/views/control_room.rs:66`
  - body: The registered restart RPC is `gateway.restart.request`, not `gateway.restart`, so the Control Room restart button reports an unknown method instead of requesting a safe restart.
  - confidence: 0.92
- **[P2] Read pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, but this view reads `requests`, so real pending pair requests are shown as empty.
  - confidence: 0.91
- **[P2] Use the local ws URL in onboarding:** `apps/linux-gtk/crates/app/src/views/onboarding.rs:99-102`
  - body: The app's resolved default gateway URL is plain `ws://127.0.0.1:18789`, but the onboarding field still defaults to `wss://127.0.0.1:18789`, so fresh users who accept the default try TLS against the local non-TLS gateway.
  - confidence: 0.9
- **[P1] Persist or remove the settings save flow:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-234`
  - body: `Save & Reconnect` only mutates process environment variables, and the inline TODO says the existing client is not rebuilt. The setting is lost on restart and may not affect the active connection, so the UI should either persist and reconnect or avoid claiming it saved.
  - confidence: 0.89
