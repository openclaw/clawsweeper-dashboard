## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Advertise the current gateway protocol version:** `apps/linux-gtk/crates/gateway-client/src/protocol.rs:218-222`
  - body: Current main only accepts non-probe clients whose advertised protocol range includes 4. This app hard-codes protocol 3 for both min and max, so the gateway closes the handshake with `protocol mismatch` before any UI flow can connect.
  - confidence: 0.96
- **[P1] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }`, but this button sends `{ path, value }`. Users can edit the JSON, but the Save request is rejected by the gateway schema and never persists.
  - confidence: 0.94
- **[P2] Use current skill payload fields:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-126`
  - body: `skills.update` requires `skillKey`, and `skills.install` requires a supported install shape such as `{ name, installId }` or `{ source: "clawhub", slug }`. Sending `skillId` makes skill toggles and installs fail validation.
  - confidence: 0.92
- **[P2] Wrap cron enablement in a patch object:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` requires the mutable fields under `patch`, but the toggle sends `enabled` at the top level. The UI updates the chip optimistically while the gateway rejects the request, leaving visible cron state out of sync.
  - confidence: 0.91
- **[P2] Pass supported channel probe params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, `timeoutMs`, and optional `channel`; this button sends `channelId`. The Probe action therefore returns an invalid-request error instead of testing channel health.
  - confidence: 0.9
- **[P2] Read pending node pairs from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-99`
  - body: `node.pair.list` returns `{ pending, paired }`, but this code reads `requests`. Real pending approvals are shown as empty, so operators cannot approve or reject pair requests from the Instances screen.
  - confidence: 0.9
- **[P2] Call the supported restart request method:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The gateway exposes `gateway.restart.request` and `gateway.restart.preflight`, not `gateway.restart`. The Control Room Restart button will always fail with an unknown method.
  - confidence: 0.88
- **[P1] Persist and rebuild connection settings:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-236`
  - body: Save & Reconnect only mutates process environment variables, then tells users to restart. Those values are lost on a real restart and the existing `GatewayClient` is never rebuilt, so changed URLs or tokens do not reliably apply.
  - confidence: 0.87
- **[P2] Use the local ws default consistently:** `apps/linux-gtk/crates/app/src/views/settings.rs:55-58`
  - body: The gateway client default is `ws://127.0.0.1:18789`, but the onboarding and settings UI still prefill `wss://127.0.0.1:18789`. Fresh local users who accept the visible default attempt TLS against the plain local listener and fail to connect.
  - confidence: 0.86
