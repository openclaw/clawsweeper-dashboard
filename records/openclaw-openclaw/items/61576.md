## Summary

Keep open. The Linux companion work is useful source material, but the branch is not mergeable against current main because multiple management views still send payloads that current gateway validators reject, the auth-token config path does not match OpenClaw config, and the PR still needs product/security review against the parallel Linux companion track.

## What This Changes

This PR adds a Rust/GTK4/libadwaita Linux companion app under `apps/linux-gtk`, including a Rust gateway client, GTK management views, desktop metadata, README, Cargo workspace, and lockfile.

## Best Possible Solution

Choose a single Linux companion direction, then bring that implementation onto typed current gateway protocol helpers with focused credential/TLS review, packaging decisions, and Linux desktop validation before merge.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to compare the PR's hard-coded RPC payloads with current-main TypeBox schemas; those validators reject the shown config, cron, channels, skills, and pair-list shapes, and normal generated config uses `gateway.auth.token` rather than `gateway.token`.

## Solution Assessment

No. A Rust/GTK app may be a viable track, but this branch is not the best mergeable solution until maintainers choose it over the parallel C/GTK path and the implementation stops hand-rolling stale gateway request shapes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Read the configured gateway auth token:** `apps/linux-gtk/crates/gateway-client/src/config.rs:21-25`
  - body: The config parser only deserializes `gateway.token`, but OpenClaw stores shared-secret auth under `gateway.auth.token`/`gateway.auth.password`. A normal generated `openclaw.json` will therefore connect without the configured token unless users set an env var manually.
  - confidence: 0.93
- **[P2] Send empty params when loading config:** `apps/linux-gtk/crates/app/src/views/config.rs:137-138`
  - body: `config.get` is validated against an empty params object on current main, so passing `{ "path": "" }` makes the Config view fail before it can load the file. Send `{}` for `config.get` and reserve path lookup for `config.schema.lookup`.
  - confidence: 0.95
- **[P2] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` accepts `{ raw, baseHash? }`, but this sends `{ path, value }`. The Save button therefore submits an invalid request and cannot persist edits even after the user enters valid JSON.
  - confidence: 0.96
- **[P2] Build cron.add requests with the current schema:** `apps/linux-gtk/crates/app/src/views/cron.rs:147-151`
  - body: `cron.add` requires structured `schedule`, `sessionTarget`, `wakeMode`, and `payload` fields. Sending a plain schedule string plus top-level `agentId` and `message` is rejected as invalid.
  - confidence: 0.94
- **[P2] Wrap cron enablement in a patch:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` requires changed fields under `patch`, but this toggle sends `enabled` at the top level. The request fails validation while the UI optimistically changes the status chip.
  - confidence: 0.95
- **[P2] Use supported channels.status probe params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` only accepts `probe` and `timeoutMs`; passing `channelId` is rejected. The Probe button will fail instead of running the channel health probe.
  - confidence: 0.94
- **[P2] Use skillKey for skill updates:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: Installed skill toggles submit `skillId`, but `skills.update` requires `skillKey`. Enabling or disabling a skill from this view is rejected and does not persist.
  - confidence: 0.94
- **[P2] Use valid install params for missing skills:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` requires either `{ name, installId }` or `{ source: "clawhub", slug }`, but the Install button sends only `skillId`. Missing-skill installs therefore fail validation.
  - confidence: 0.93
- **[P2] Read node pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, not `requests`. Reading `payload.requests` hides real pending node approvals, so operators cannot approve or reject devices from the Instances screen.
  - confidence: 0.94
- **[P2] Remove or implement the unsupported restart RPC:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The Restart button calls `gateway.restart`, but current main does not register that gateway RPC method. This primary recovery action will return unknown-method errors until it uses an existing service control path or a deliberately added server method.
  - confidence: 0.9
- **[P2] Persist settings instead of process-only env vars:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-236`
  - body: Save & Reconnect only mutates process environment variables and then tells the user to restart, but those values disappear when the app exits. URL/token changes are not actually saved for the requested restart path.
  - confidence: 0.89
- **[P2] Add a handshake timeout before reconnecting:** `apps/linux-gtk/crates/gateway-client/src/client.rs:258-260`
  - body: The connection attempt waits indefinitely for `connect.challenge` and ignores unrelated text frames, so a WebSocket endpoint that connects but never speaks the OpenClaw handshake can stall the reconnect loop forever. Add a bounded handshake timeout and return an error to the backoff loop.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff adds a new credentialed desktop gateway client and the PR itself leaves TLS, device identity, and token persistence review open.

Concerns:

- **[medium] Guard the TLS verification bypass before merge:** `apps/linux-gtk/crates/gateway-client/src/client.rs:234`
  - body: When `OPENCLAW_TLS_ACCEPT_INVALID` is set, the client installs a verifier that accepts any TLS certificate for `wss://` URLs. Before shipping a credentialed desktop client, this needs explicit dev-only UX, tests, and documentation so users do not send gateway tokens through an unintended MITMable path.
  - confidence: 0.86
- **[medium] Finish token persistence and storage review:** `apps/linux-gtk/crates/app/src/views/settings.rs:219`
  - body: The Settings view writes the gateway token into process environment variables and the README documents env-token launch flow, while the PR marks token persistence as a known security gap. The merge path should use the project’s credential/config semantics deliberately rather than leaving token handling as an ad hoc app-local pattern.
  - confidence: 0.82
