Full review comments:

- **[P1] Define a desktop client identity:** `apps/linux-gtk/crates/gateway-client/src/protocol.rs:221-228`
  - body: This new Linux desktop app handshakes as the CLI with CLI mode and broad operator scopes. Current main has a closed client-id registry for native apps, so merging this would make gateway policy, diagnostics, and auth decisions treat a desktop app as the CLI instead of a supported Linux client.
  - confidence: 0.91
- **[P1] Send raw text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: The Config view sends `{ path, value }`, but current `config.set` only accepts `{ raw, baseHash? }`. The Save button will be rejected as an invalid request, so users can edit JSON but cannot persist it through this screen.
  - confidence: 0.96
- **[P1] Use supported skill RPC parameters:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: Installed skill toggles send `skillId`, while current `skills.update` requires `skillKey`. These requests fail validation, so the switch can appear to change a skill while backend config is unchanged.
  - confidence: 0.93
- **[P2] Install skills with a valid install target:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: The install button sends only `skillId`, but `skills.install` requires either `{ name, installId }`, a ClawHub `{ source: "clawhub", slug }`, or an upload target. The button will fail for available skills instead of installing them.
  - confidence: 0.89
- **[P1] Wrap cron enablement in patch:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-331`
  - body: `cron.update` accepts a job id plus a `patch` object, but this switch sends `enabled` at the top level. The request fails validation after the UI chip updates optimistically, leaving visible cron state out of sync with the scheduler.
  - confidence: 0.94
- **[P2] Use supported channel probe parameters:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` allows `probe`, `timeoutMs`, and `channel`, not `channelId`. The Probe action will return an invalid-request error instead of checking channel health.
  - confidence: 0.86
- **[P2] Read pending node pairs from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns pending pair requests under `pending`, but this view reads `requests`. Real approvals will be hidden as an empty list, so operators cannot approve or reject pending node pair requests from this screen.
  - confidence: 0.88
- **[P2] Call the supported restart RPC:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The restart button calls `gateway.restart`, but current main exposes `gateway.restart.preflight` and `gateway.restart.request`. This primary recovery control will fail with an unknown method instead of requesting a restart.
  - confidence: 0.9
- **[P1] Persist or rebuild connection settings:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-236`
  - body: Save & Reconnect only mutates process environment variables, while the running client was created from an earlier config and those values disappear on app restart. The UI says settings were saved, but remote URLs and tokens are neither durable nor applied to the active client.
  - confidence: 0.88
