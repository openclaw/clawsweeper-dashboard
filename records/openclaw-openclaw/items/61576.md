## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Define a desktop client identity:** `apps/linux-gtk/crates/gateway-client/src/protocol.rs:221-228`
  - body: The gateway client IDs are a closed contract, but this new desktop app hardcodes `cli` mode and requests admin, approvals, and pairing scopes. That makes policy, diagnostics, and authorization treat the Linux app as the CLI and gives it CLI-equivalent authority before maintainers have accepted that security model.
  - confidence: 0.94
- **[P1] Read the canonical gateway auth token:** `apps/linux-gtk/crates/gateway-client/src/config.rs:21-26`
  - body: Current gateway config stores the shared token under `gateway.auth.token`, but this parser only reads `gateway.token`. A normal configured gateway will therefore start the app without the token unless the user manually supplies `OPENCLAW_GATEWAY_TOKEN`.
  - confidence: 0.9
- **[P1] Send raw text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }` with no additional properties; this sends `{ path, value }`, so the gateway rejects the save request and the Config view cannot persist changes.
  - confidence: 0.95
- **[P2] Use skillKey for skills.update:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: The current `skills.update` schema expects `skillKey`, not `skillId`. Toggling an installed skill will send an invalid payload and fail against the gateway.
  - confidence: 0.93
- **[P2] Install skills with a supported payload:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` accepts either `{ name, installId }`, a ClawHub `{ source, slug }` payload, or an upload payload. Sending only `skillId` is not one of the supported shapes, so the Install button cannot work.
  - confidence: 0.92
- **[P2] Wrap cron edits in a patch object:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` requires `{ id/jobId, patch: { ... } }`; this sends `enabled` at the top level. The UI also flips the chip optimistically, so users can see a disabled/enabled state that the gateway rejected.
  - confidence: 0.92
- **[P2] Probe channels with supported status params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, `timeoutMs`, and optional `channel`; `channelId` is rejected as an unexpected property, so the per-channel Probe action cannot call the current gateway API.
  - confidence: 0.9
- **[P2] Call the registered restart RPC:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: Current gateway descriptors register `gateway.restart.preflight` and `gateway.restart.request`, not `gateway.restart`. The Control Room restart button will call an unknown method.
  - confidence: 0.92
- **[P2] Read pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, but this view reads `requests`. Pending pair requests will be hidden and the app will show the empty state even when approvals are waiting.
  - confidence: 0.91
- **[P3] Keep the documented gateway default in sync:** `apps/linux-gtk/README.md:61-65`
  - body: The README documents a default WebSocket URL of `wss://127.0.0.1:9443`, while the app code defaults to `ws://127.0.0.1:18789`. Users following the docs can end up debugging the wrong endpoint.
  - confidence: 0.85
