## Summary

Keep open. This is an active implementation candidate for a still-missing Linux companion app, but the branch has current gateway-contract mismatches, missing real behavior proof, and security/product decisions that block merge rather than cleanup-close it.

## What This Changes

The PR adds a Rust/GTK4/libadwaita Linux companion app under `apps/linux-gtk`, including a Rust gateway client, management views, desktop metadata, CSS, README, and Cargo workspace files.

## Best Possible Solution

Choose one Linux companion direction in the umbrella tracker, then align the accepted implementation with typed current gateway protocol helpers, durable credential handling, focused TLS/token review, platform docs/labels, and real Linux desktop proof before merge.

## Reproduction Assessment

Yes. The high-confidence source reproduction path is to compare the PR's hard-coded gateway RPC payloads with current `src/gateway/protocol/schema/*` validators; multiple actions would be rejected by current main.

## Solution Assessment

No. A Rust/GTK companion may be viable, but this branch is not the best mergeable solution until maintainers choose the Linux app track, the RPC shapes match current main, and credential/TLS behavior has proof and review.

## Security Review

Status: needs_attention

Summary: The PR adds a credentialed native gateway client and leaves concrete TLS verification and token persistence concerns open.

Concerns:

- **[medium] Gate the TLS verification bypass:** `apps/linux-gtk/crates/gateway-client/src/client.rs:234`
  - body: When `OPENCLAW_TLS_ACCEPT_INVALID` is set, the Rust client installs a verifier that accepts any certificate for `wss://` URLs. Before shipping a credentialed desktop client, this needs explicit dev-only UX, documentation, and tests so tokens are not sent through an unintended MITMable path.
  - confidence: 0.86

  - body: The Settings view writes gateway tokens only into process environment variables while the PR marks token persistence as a known security gap. The merge path should align with OpenClaw credential/config semantics instead of leaving ad hoc app-local token handling.
