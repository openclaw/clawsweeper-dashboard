## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Advertise the current gateway protocol version:** `apps/linux-gtk/crates/gateway-client/src/protocol.rs:218`
  - body: This client advertises protocol 3, but current main accepts protocol 4 as the required client range. The gateway rejects the connect request with a protocol mismatch before the Linux app can authenticate or load any data.
  - confidence: 0.97
- **[P1] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }`, but this request sends `{ path, value }`. Users can edit JSON in the Config view, but Save will be rejected as an invalid request.
  - confidence: 0.96
- **[P1] Persist connection settings instead of process env only:** `apps/linux-gtk/crates/app/src/views/settings.rs:221-236`
  - body: The Save flow only mutates process environment variables and then tells users settings were saved. The running client is not rebuilt and the values are gone after a real restart, so endpoint/token changes are neither durable nor reliably applied.
  - confidence: 0.94
- **[P2] Use skillKey for skills.update:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: The current skills update schema accepts `skillKey`, not `skillId`. Toggling an installed skill will therefore fail validation instead of enabling or disabling the backend skill setting.
  - confidence: 0.93
- **[P2] Pass a valid install payload to skills.install:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` requires either legacy `{ name, installId }`, ClawHub `{ source: "clawhub", slug }`, or upload params. Sending only `skillId` means the Install button cannot install missing skills.
  - confidence: 0.92
- **[P2] Wrap cron enablement in a patch object:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-331`
  - body: `cron.update` requires mutable fields inside `patch`, but this toggle sends `enabled` at the top level. The UI updates the chip optimistically while the gateway rejects the request, leaving visible cron state out of sync.
  - confidence: 0.94
- **[P2] Send supported channel status params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, `timeoutMs`, and optional `channel`, not `channelId`. The Probe button will return an invalid-request error instead of probing the selected channel.
  - confidence: 0.9
- **[P2] Read pending node pairs from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns pending requests under `pending`, but this code reads `requests`. Real pair approvals are hidden as an empty list, so operators cannot approve or reject pending nodes from this screen.
  - confidence: 0.9
- **[P2] Call a supported restart request method:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The Control Room invokes `gateway.restart`, which is not in the advertised core gateway method catalog. The primary Restart button therefore fails with an unknown-method error rather than requesting a restart.
  - confidence: 0.86
