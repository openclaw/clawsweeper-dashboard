## Summary

Keep open. Current main does not contain a Linux GTK companion app, and this PR is still an active alternative implementation for #75 rather than obsolete work. The branch is not merge-ready: the latest head still has multiple gateway RPC contract mismatches against current main, plus the PR itself calls out unresolved security review, packaging, and desktop-environment validation.

## What This Changes

This PR adds a new Rust/GTK4/libadwaita Linux companion app under `apps/linux-gtk/`, including a Rust gateway-client crate, GTK management views, desktop metadata, styling, README, and Cargo lockfile.

## Best Possible Solution

Keep the PR open for maintainer review, decide whether the Rust/GTK track, the C/GTK track in #59859, or a merged design should carry #75, then require the chosen implementation to use typed/current gateway RPC contracts, complete credential/TLS security review, and add focused Linux app validation before merge.

## Reproduction Assessment

Yes. The blocking issues have high-confidence reproduction paths by sending the PR's hard-coded RPC payloads to a current-main gateway: the TypeBox validators reject the mismatched `config.*`, `skills.*`, `cron.*`, and `channels.status` params, and `node.pair.list` returns `pending` rather than `requests`.

## Solution Assessment

No. The PR is a plausible implementation track, but it is not the best mergeable solution yet because several management views are wired to stale or incorrect gateway contracts and the project still needs a maintainer decision against the parallel #59859 implementation.

Overall confidence: 0.89

Full review comments:

- **[P2] Send empty params when loading config:** `apps/linux-gtk/crates/app/src/views/config.rs:137-138`
  - body: `config.get` is validated against an empty params object on current main, so passing `{ "path": "" }` makes the Config view fail before it can load the file. Send `{}` for `config.get` and reserve path lookup for `config.schema.lookup`.
  - confidence: 0.93
- **[P2] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` accepts `{ raw, baseHash? }`, but this sends `{ path, value }`. The Save button therefore submits an invalid request and cannot persist edits even after the user enters valid JSON.
  - confidence: 0.95
- **[P2] Build cron.add requests with the current schema:** `apps/linux-gtk/crates/app/src/views/cron.rs:147-151`
  - body: `cron.add` no longer accepts a plain schedule string plus top-level `agentId` and `message`; it requires structured `schedule`, `sessionTarget`, `wakeMode`, and `payload` fields. New jobs created from this form will be rejected as invalid.
  - confidence: 0.91
- **[P2] Wrap cron enablement in a patch:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` requires the changed fields under `patch`, but this toggle sends `enabled` at the top level. The request fails validation while the UI optimistically changes the chip, leaving users with stale cron state.
  - confidence: 0.95
- **[P2] Use supported channels.status probe params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` only accepts `probe` and `timeoutMs`; passing `channelId` is rejected. The Probe button will always fail instead of running the channel health probe.
  - confidence: 0.94
- **[P2] Use skillKey for skill updates:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: Installed skill toggles submit `skillId`, but `skills.update` requires `skillKey`. As written, enabling or disabling a skill from this view is rejected by the gateway and does not persist.
  - confidence: 0.94
- **[P2] Use valid install params for missing skills:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` requires either `{ name, installId }` or `{ source: "clawhub", slug }`, but the Install button sends only `skillId`. Missing-skill installs therefore fail validation.
  - confidence: 0.92
- **[P2] Read node pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, not `requests`. Reading `payload.requests` hides real pending node approvals, so operators cannot approve or reject devices from the Instances screen.
  - confidence: 0.93
- **[P2] Remove or implement the unsupported restart RPC:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The Restart button calls `gateway.restart`, but current main does not register that gateway RPC method. This primary recovery action will return unknown-method errors until it uses an existing service control path or a new server method is added deliberately.
  - confidence: 0.86
- **[P2] Persist settings instead of process-only env vars:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-236`
  - body: Save & Reconnect only mutates process environment variables and then tells the user to restart, but those values disappear when the app exits. URL/token changes are not actually saved for the requested restart path.
  - confidence: 0.88
- **[P2] Add a handshake timeout before reconnecting:** `apps/linux-gtk/crates/gateway-client/src/client.rs:258-260`
  - body: The connection attempt waits indefinitely for `connect.challenge` and ignores unrelated text frames, so a WebSocket endpoint that connects but never speaks the OpenClaw handshake can stall the reconnect loop forever. Add a bounded handshake timeout and return an error to the backoff loop.
  - confidence: 0.82
- **[P3] Use the local ws URL in Settings defaults:** `apps/linux-gtk/crates/app/src/views/settings.rs:53-58`
  - body: The gateway client default is `ws://127.0.0.1:18789`, but Settings still pre-fills `wss://127.0.0.1:18789`. Fresh local users who save that default will try TLS against the plain local listener and fail to connect.
  - confidence: 0.8
