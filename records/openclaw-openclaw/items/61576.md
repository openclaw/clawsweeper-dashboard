## Summary

Keep open. Current main still has no Linux companion app, the competing Linux GTK/C implementation at https://github.com/openclaw/openclaw/pull/59859 is open rather than merged, and this Rust/GTK branch still has blocking gateway-contract and security-review issues that need maintainer product direction before repair or merge.

## What This Changes

This PR adds a new Rust/GTK4/libadwaita Linux companion app under `apps/linux-gtk` with a Rust gateway client, GTK management views, desktop metadata, README, Cargo workspace, and lockfile.

## Best Possible Solution

Choose one Linux companion direction, then bring that implementation onto current typed gateway protocol helpers with focused credential/TLS review, packaging decisions, and Linux desktop validation before merge.

## Reproduction Assessment

Yes. The high-confidence path is source reproduction by comparing the PR branch's hard-coded RPC payloads and config parsing with current-main TypeBox schemas and registered gateway methods; I did not need a live GTK run to prove those request-shape failures.

## Solution Assessment

No. A Rust/GTK app may still be viable, but this branch is not the best mergeable solution until maintainers choose the Linux companion track and the implementation stops hand-rolling stale gateway request shapes.

Full review comments:

- **[P2] Read the configured gateway auth token:** `apps/linux-gtk/crates/gateway-client/src/config.rs:21-25`
  - body: The parser only reads `gateway.token`, but current config labels and docs use `gateway.auth.token`/`gateway.auth.password`. A normal generated config will connect without the configured shared secret unless users also set an env var.
  - confidence: 0.93

  - body: `config.get` validates against an empty params object on current main, so passing `{ "path": "" }` makes the Config view fail before it can load the file.
  - confidence: 0.95

  - body: `config.set` accepts `{ raw, baseHash? }`, but this sends `{ path, value }`. The Save button therefore submits an invalid request and cannot persist edits.
  - confidence: 0.96

  - body: `cron.add` requires structured `schedule`, `sessionTarget`, `wakeMode`, and `payload` fields. Sending a plain schedule string plus top-level `agentId` and `message` is rejected by the current validator.
  - confidence: 0.94

  - body: `cron.update` requires changed fields under `patch`, but this toggle sends `enabled` at the top level, so enable/disable actions fail validation while the UI optimistically changes state.
  - confidence: 0.95

  - confidence: 0.94

  - confidence: 0.94

  - confidence: 0.93

  - confidence: 0.9

  - confidence: 0.89

  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff adds a credentialed desktop gateway client with TLS-bypass and token-storage decisions still unresolved.

Concerns:

- **[medium] Guard the TLS verification bypass before merge:** `apps/linux-gtk/crates/gateway-client/src/client.rs:234`
  - body: When `OPENCLAW_TLS_ACCEPT_INVALID` is set, the client installs a verifier that accepts any TLS certificate for `wss://` URLs. A shipped credentialed desktop client needs explicit dev-only UX, tests, and documentation so gateway tokens are not sent through an unintended MITMable path.
  - confidence: 0.86
- **[medium] Finish token persistence and storage review:** `apps/linux-gtk/crates/app/src/views/settings.rs:219`
  - body: The Settings view writes the gateway token into process environment variables and the PR marks token persistence as a known gap. The merge path should use OpenClaw's credential/config semantics deliberately rather than leaving token handling as an app-local env pattern.
  - confidence: 0.82
