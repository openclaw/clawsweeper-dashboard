## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send raw config payloads to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-226`
  - body: The gateway's `config.get` takes `{}` and `config.set` accepts `{ raw, baseHash? }`, but this view sends `{ path: "" }` when loading and `{ path, value }` when saving. With strict validation, the config editor cannot load or save against current main.
  - confidence: 0.95
- **[P1] Use current skill install and update identifiers:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: `skills.update` requires `skillKey`, and `skills.install` requires either `{ name, installId }`, a ClawHub slug, or an upload payload. Sending only `skillId` means the app's enable/disable and install actions are rejected by the gateway.
  - confidence: 0.94
- **[P1] Wrap cron updates in the supported patch payload:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-331`
  - body: The current `cron.update` contract requires `{ id|jobId, patch: ... }`, but this code sends top-level `enabled`; `cron.add` similarly sends a string schedule and agent/message fields instead of the required job shape. Cron management will fail or drift from the UI's optimistic state.
  - confidence: 0.94
- **[P2] Use the supported channel status parameter:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `channel`, not `channelId`, and rejects unknown properties. Passing `channelId` here prevents the per-channel refresh/probe path from working against the current gateway.
  - confidence: 0.91
- **[P2] Read pending pair requests from the gateway response:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns pending requests under `pending`, but this view reads `payload.requests`. That leaves real pending pairing requests invisible in the Instances view even when the gateway returns them.
  - confidence: 0.88
- **[P2] Call the gateway restart request method:** `apps/linux-gtk/crates/app/src/views/control_room.rs:66`
  - body: Current main exposes `gateway.restart.request` and `gateway.restart.preflight`; there is no `gateway.restart` server method. The restart button therefore sends an unknown RPC method instead of requesting a restart.
  - confidence: 0.92
