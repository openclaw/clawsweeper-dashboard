Full review comments:

- **[P2] Read the configured gateway auth secret:** `apps/linux-gtk/crates/gateway-client/src/config.rs:21-25`
  - body: Current OpenClaw configs store gateway shared secrets under `gateway.auth.token` or `gateway.auth.password` and may use auth modes or SecretRefs, but this reader only looks for `gateway.token`. A normal configured gateway will therefore connect without the required shared secret unless the user also sets `OPENCLAW_GATEWAY_TOKEN`.
  - confidence: 0.92
- **[P2] Call `config.get` with the supported empty params:** `apps/linux-gtk/crates/app/src/views/config.rs:137-138`
  - body: `config.get` is schema-validated with `additionalProperties: false`, so sending `{ "path": "" }` is rejected before the Config view can load the document. This makes the editor fail against current gateway builds.
  - confidence: 0.94
- **[P2] Save config through `raw` and `baseHash`:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` requires the full raw config string plus the base hash from `config.get`; `{ path, value }` fails validation and bypasses the conflict guard. The Save button cannot persist edits against current main.
  - confidence: 0.95
- **[P2] Use `skillKey` and valid install params:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: `skills.update` expects `skillKey`, and `skills.install` expects an install id, ClawHub slug, or upload id. Sending `skillId` makes both enable/disable and install actions fail validation instead of updating the backend.
  - confidence: 0.92
- **[P2] Send cron updates in a `patch` object:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` requires `{ id|jobId, patch: { enabled } }`, but the toggle sends `enabled` at top level while immediately changing the chip. The UI can claim a job is disabled even though the gateway rejected the mutation.
  - confidence: 0.94
- **[P2] Use the current `cron.add` payload shape:** `apps/linux-gtk/crates/app/src/views/cron.rs:147-151`
  - body: `cron.add` requires a named job with structured `schedule`, `sessionTarget`, `wakeMode`, and `payload`; a top-level schedule string, agentId, and message are rejected by the schema. New-job creation is nonfunctional as written.
  - confidence: 0.91
- **[P2] Probe channels with supported status params:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, `timeoutMs`, and optional `channel`, not `channelId`. The Probe button therefore sends an invalid request and reports an error instead of checking channel health.
  - confidence: 0.9
- **[P2] Read pending node pairs from `pending`:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, but the Instances view reads `requests`. Real pending pair requests will be hidden and operators cannot approve or reject them from this screen.
  - confidence: 0.9
- **[P2] Call the registered gateway restart RPC:** `apps/linux-gtk/crates/app/src/views/control_room.rs:65-67`
  - body: The gateway registers `gateway.restart.request`, not `gateway.restart`. The Control Room restart button will always hit an unknown method and cannot restart the gateway.
  - confidence: 0.89
