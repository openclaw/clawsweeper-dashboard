Full review comments:

- **[P1] Read gateway auth from the current config contract:** `apps/linux-gtk/crates/gateway-client/src/config.rs:21-25`
  - body: This only deserializes `gateway.token`, but current OpenClaw stores gateway credentials under `gateway.auth` and supports password, trusted-proxy, and SecretRef-backed secrets. Users with normal current configs can launch this app without the credential it needs and fail to authenticate, so resolve the same effective auth contract the gateway/CLI use instead of adding a competing token path.
  - confidence: 0.9
- **[P1] Send config writes through raw and baseHash:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }`, but this sends `{ path, value }`. The Save button will be rejected by current gateways, making the Config view's write path nonfunctional and risking drift from the hash-aware config editor contract.
  - confidence: 0.93
- **[P1] Call skills.update with skillKey:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-127`
  - body: Current `skills.update` requires `skillKey`, not `skillId`. Installed skill toggles will fail validation and the UI can leave users believing enablement changed when backend config did not.
  - confidence: 0.9
- **[P1] Use a supported skills.install payload:** `apps/linux-gtk/crates/app/src/views/skills.rs:160-163`
  - body: `skills.install` requires either `{ name, installId }`, `{ source: "clawhub", slug }`, or upload params, but this sends only `skillId`. The Install button therefore cannot install missing skills against the current gateway contract.
  - confidence: 0.9
- **[P1] Wrap cron enablement in a patch object:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-330`
  - body: `cron.update` accepts the job id plus `patch`, but this sends `enabled` at the top level. Enable/disable requests will fail validation while the status chip has already been changed optimistically.
  - confidence: 0.92
- **[P2] Use supported channel probe parameters:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, optional `timeoutMs`, and optional `channel`; this sends `channelId` and never sets `probe: true`. The Probe button will return invalid-request errors instead of running the current gateway health check.
  - confidence: 0.9
- **[P2] Read pending node pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-99`
  - body: `node.pair.list` returns the pairing list as `{ pending, paired }`, but this reads `requests`. Real pending approvals will be displayed as empty, hiding pairing actions operators need to accept or reject.
  - confidence: 0.87
