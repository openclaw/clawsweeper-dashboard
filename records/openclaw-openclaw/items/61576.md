  - body: Current main requires protocol 4 and closes mismatched WebSocket handshakes, but this app sends protocol 3 for both `minProtocol` and `maxProtocol`. As written, the companion cannot connect to a current gateway.
  - confidence: 0.98
- **[P1] Send raw config text to config.set:** `apps/linux-gtk/crates/app/src/views/config.rs:222-225`
  - body: `config.set` validates `{ raw, baseHash? }` with no extra properties, but this view sends `{ path, value }`. The Save button will be rejected by current main and cannot persist edited config.
  - confidence: 0.97
- **[P1] Use skillKey and valid install parameters:** `apps/linux-gtk/crates/app/src/views/skills.rs:123-163`
  - body: Current `skills.update` expects `skillKey`, and `skills.install` expects a legacy `{name, installId}`, ClawHub, or upload payload. Sending `skillId` makes skill toggles and installs fail validation.
  - confidence: 0.96
- **[P1] Wrap cron enablement in patch:** `apps/linux-gtk/crates/app/src/views/cron.rs:327-331`
  - body: `cron.update` accepts an id/jobId plus a `patch` object, but this toggle sends `enabled` at the top level. The UI updates the chip optimistically even though current main rejects the request.
  - confidence: 0.94
- **[P2] Probe channels with supported parameters:** `apps/linux-gtk/crates/app/src/views/channels.rs:126-128`
  - body: `channels.status` accepts `probe`, `timeoutMs`, and optional `channel`; `channelId` is an extra property. The Probe button therefore fails validation instead of checking channel health.
  - confidence: 0.92
- **[P2] Read node pair requests from pending:** `apps/linux-gtk/crates/app/src/views/instances.rs:96-100`
  - body: `node.pair.list` returns `{ pending, paired }`, but the Instances view reads `requests`. Real pending pairing approvals will render as empty, hiding operator actions.
  - confidence: 0.9
- **[P1] Persist connection settings before claiming they are saved:** `apps/linux-gtk/crates/app/src/views/settings.rs:219-236`

  - confidence: 0.86
