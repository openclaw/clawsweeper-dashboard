## Label Justifications

- `P1`: The issue describes a broken Slack Agent/Assistant DM workflow where real user messages fragment into separate sessions and disrupt continuous conversation context.
- `impact:session-state`: The central failure mode is session fan-out into multiple sibling conversations, which fragments agent context and persona state.
- `impact:message-loss`: Inbound Slack DMs are effectively routed away from the user's expected visible DM session, even though transport delivery succeeds.

## Summary

Keep open. Current main and v2026.6.8 still route Slack assistant DM thread timestamps into thread-suffixed sessions, and the open linked fix PR is not merged, so this issue is still necessary.

## What This Changes

The issue requests that Slack Agent/Assistant DM messages route into the existing DM session instead of spawning a new thread-suffixed sibling session for each assistant thread.

## Best Possible Solution

Land a maintainer-approved Slack plugin routing fix that preserves assistant metadata while preventing unwanted DM session fan-out, or explicitly choose a different Slack assistant-thread session policy.

## Reproduction Assessment

Yes, at source level: current main passes Slack assistant DM `assistantThreadTs` into the thread id path, and `resolveThreadSessionKeys` suffixes non-empty thread ids. I did not run a live Slack workspace reproduction in this read-only review.
