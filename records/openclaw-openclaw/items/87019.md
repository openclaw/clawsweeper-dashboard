## Label Justifications

- `P2`: The report describes a real Slack DM continuity problem with a clear source path, but it is limited to the optional Slack Agent/Assistant capability and needs maintainer contract review.
- `impact:session-state`: The central failure mode is fan-out into multiple sibling sessions, which fragments conversation context and agent persona state.
- `impact:message-loss`: Inbound Slack DMs are effectively misrouted away from the user's expected existing DM session, even though they are not dropped at transport level.

## Summary

Keep open. Current main and the latest release still intentionally route Slack assistant DM messages with an assistant thread timestamp into thread-scoped OpenClaw sessions, while the report gives plausible user evidence that this breaks continuous DM context and needs a maintainer routing-contract decision.

## What This Changes

Requested Slack DM routing change so Agent/Assistant-wrapped DM threads collapse into the existing user DM session instead of creating a new OpenClaw thread session per assistant thread.

## Best Possible Solution

Maintainers should decide the Slack assistant-thread session contract, then make one plugin-owned change that preserves assistant context metadata while routing DM continuity according to that decision and updating tests/docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: current source and tests reproduce the OpenClaw key-derivation side when distinct `assistantThreadTs` values arrive on DM events, but a real Slack workspace is still needed to verify the reporter's per-message assistant-thread emission.

## Solution Assessment

Is this the best way to solve the issue? Unclear: collapsing assistant DM thread IDs is a narrow fix for continuity, but it changes a documented/tested assistant-thread isolation contract, so maintainers should first choose the default or config shape.
