## Label Justifications

- `P1`: The issue describes a broken Slack Agent/Assistant DM workflow where real user messages fragment into separate sessions and disrupt continuous conversation context.
- `impact:message-loss`: Inbound Slack DMs are effectively misrouted away from the user's expected visible DM session, even though transport delivery succeeds.
- `impact:session-state`: The central failure mode is session fan-out into multiple sibling conversations, which fragments agent context and persona state.

## Summary

Keep open. Current main and the latest release still route Slack assistant DM thread timestamps into thread-suffixed OpenClaw sessions, and the open linked closing PR owns the active fix path.

## What This Changes

Slack Agent/Assistant DMs should route consecutive messages from the same user into one visible DM session instead of a new sibling session per assistant thread.

## Best Possible Solution

Review and land the linked Slack plugin fix, or replace it with a maintainer-approved routing contract that preserves assistant metadata while preventing unwanted DM session fan-out.

## Reproduction Assessment

Yes at source level: current main routes Slack assistant DM `assistantThreadTs` into `canonicalThreadId`, and `resolveThreadSessionKeys` suffixes any non-empty thread id. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes for triage: keeping this issue open behind the linked closing PR is the right path because the problem remains on main and in v2026.6.6. The final default/config shape still needs maintainer acceptance before the fix lands.
