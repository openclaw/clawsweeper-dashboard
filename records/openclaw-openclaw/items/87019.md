## Label Justifications

- `P1`: The issue reports a broken Slack Agent/Assistant DM workflow where real user messages fragment into separate sessions and disrupt continuous conversation context.
- `impact:session-state`: The central failure is session fan-out into sibling Slack DM sessions, fragmenting agent context and persona state.
- `impact:message-loss`: Inbound Slack DMs can route away from the user's expected visible DM conversation, making replies and context effectively misdelivered.

## Summary

Keep open. Current main and the latest release still route Slack Agent/Assistant DM assistant thread timestamps into thread-suffixed sessions, and the linked closing PR remains open rather than merged.

## What This Changes

The issue asks Slack DMs wrapped by Slack Agent/Assistant assistant threads to route into the existing user DM session instead of creating a sibling session per assistant thread timestamp.

## Best Possible Solution

Review and land or replace the linked Slack routing fix with a maintainer-approved assistant-DM collapse contract, aligned tests, and docs, then close this issue from the merged PR.

## Reproduction Assessment

Yes, at source level: current main passes Slack assistant DM assistantThreadTs into the thread-id path, and resolveThreadSessionKeys suffixes non-empty thread ids. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear as final product direction: collapsing assistant DM sessions addresses the reported root cause, but current docs and tests intentionally preserve assistant thread sessions. Maintainers need to choose default behavior versus an opt-in setting before this can be called the best fix.
