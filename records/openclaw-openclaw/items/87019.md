## Label Justifications

- `P1`: The issue describes a broken Slack Agent/Assistant DM workflow where real user messages fragment into separate sessions and disrupt continuous conversation context.
- `impact:message-loss`: Inbound Slack DMs are effectively routed away from the user’s expected visible DM session, even though transport delivery succeeds.
- `impact:session-state`: The central failure mode is session fan-out into multiple sibling conversations, which fragments agent context and persona state.

## Summary

Keep open. Current main and the latest release still route Slack Agent/Assistant direct-message thread timestamps into thread-suffixed sessions, and the linked closing PR remains open, so this issue is still necessary.

## What This Changes

The issue asks Slack direct messages wrapped in Agent/Assistant assistant_app_thread events to keep using the user’s existing DM session instead of creating a new session per assistant thread timestamp.

## Best Possible Solution

Review and either land or replace the linked Slack routing fix, with a maintainer-approved policy for whether assistant DM thread collapse is default behavior or an opt-in configuration.

## Reproduction Assessment

Yes, at source level: current main passes Slack assistant DM assistantThreadTs into the thread id path, and resolveThreadSessionKeys suffixes non-empty thread ids. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear as final product direction: the linked opt-in config PR is the active candidate, but maintainers still need to accept the Slack default/config contract before it can be called the best fix.
