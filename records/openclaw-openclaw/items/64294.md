## Summary

Keep open: current main still lacks the requested LLM runtime helper, and the latest PR is closer than the earlier review but still lets request-supplied caller metadata override host-scoped plugin identity for audit/cost attribution.

## What This Changes

The PR adds `api.runtime.llm.complete` to the plugin runtime/SDK, injects an LLM capability into context-engine hooks, updates docs/changelog/API baseline, and includes related tests plus generated viewer-runtime churn.

## Best Possible Solution

Ship a session/request-scoped LLM completion helper with host-derived caller attribution, explicit maintainer-approved authority/cost semantics, updated docs/changelog/API baseline, and regression tests for plugin identity spoofing plus context-engine session binding.

## Reproduction Assessment

Yes for the PR review finding: static inspection of the PR head shows `params.caller` wins over the scoped plugin/authority caller even though registry code scopes calls by plugin id. This is a source-level PR review reproduction, not an end-user runtime reproduction.

## Solution Assessment

No. The direction is plausible and several earlier blockers were addressed, but the patch should derive audit/cost caller identity from the host scope before this credential-using public SDK surface is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Derive audit caller from runtime scope:** `src/plugins/runtime/runtime-llm.runtime.ts:69`
  - body: The registry wraps `llm.complete` in `withPluginRuntimePluginIdScope`, but `resolveScopedCaller` returns `params.caller` before the scoped plugin or authority caller. A plugin can therefore log and return its completion as another plugin or `host`, which breaks the audit/cost attribution this API adds; prefer the host-scoped caller and keep request metadata separate.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR still has a concrete audit-attribution issue in a new credential-using model completion capability.

Concerns:

- **[medium] Caller attribution can be spoofed:** `src/plugins/runtime/runtime-llm.runtime.ts:69`
  - body: Because request-supplied `caller` metadata overrides the runtime-scoped plugin identity, an untrusted or buggy plugin can misattribute LLM usage and cost/audit logs to another caller while using host model credentials.
  - confidence: 0.88
