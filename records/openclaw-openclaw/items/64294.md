## Summary

Keep open. Current main still does not implement the proposed plugin/context-engine LLM completion surface, but the PR is not merge-ready: the branch adds a public model-inference capability with unresolved request/session binding, config-boundary, SDK baseline, docs/changelog, and security/API review concerns, and GitHub reports the branch as dirty against main.

## What This Changes

The PR adds `api.runtime.llm.complete()` to `PluginRuntimeCore`, exports `PluginLlmComplete*` SDK types, implements a lazy runtime LLM facade, and injects `llm` into context-engine `compact`, `afterTurn`, and `maintain` runtime contexts with related test mock updates.

## Best Possible Solution

Ship an additive LLM runtime helper as a session-bound, request-scoped SDK surface with explicit plugin authority and cost controls, updated docs/changelog/API baseline, and tests for default and non-default agent model/auth resolution; keep #45865 as the broader feature tracker until a corrected PR lands.

## Reproduction Assessment

Yes for the PR review findings: static inspection of the PR head plus current-main docs and guards reproduces the config-boundary and SDK-baseline problems, and the multi-agent routing issue follows directly from the default-agent fallback. This is a PR review, not an end-user runtime reproduction.

## Solution Assessment

No. The requested capability is plausible, but this patch is not the safest implementation because it reloads config inside a runtime module and does not bind injected completions to the active session agent; a safer path is a request/session-bound facade using the active runtime config snapshot and maintainer-approved controls.
