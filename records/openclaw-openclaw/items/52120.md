## Best Possible Solution

Replace or rework this on current main so Mattermost provides thread context and auto-thread resolution before shared outbound route/session preparation, then prove the user flow in a real Mattermost thread.

Yes, at source level. Current main lacks Mattermost `buildToolContext`/`resolveAutoThreadId`, and the discussion includes concrete `replyToMode: "all"` live reproduction where message-tool sends escape an active Mattermost thread.

## Solution Assessment

No. The PR addresses part of the bug, but the current-main-safe fix should resolve the thread root before outbound route/session preparation rather than only inside Mattermost `handleAction`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Move inherited threads before dispatch:** `extensions/mattermost/src/channel.ts:201-203`
  - body: This fallback sets the implicit Mattermost thread only inside `handleAction`, after current main has already prepared the outbound mirror/session route. A send can therefore post in a thread while `resolveMattermostOutboundSessionRoute` and transcript mirroring stay on the channel-root session; expose this through `resolveAutoThreadId` so the route and delivery use the same root id.
  - confidence: 0.86
- **[P2] Base off-mode promotion on the real thread root:** `extensions/mattermost/src/channel.ts:495-496`
  - body: `threadTs` comes only from `MessageThreadId`/`ReplyToId`, but Mattermost currently derives those from `effectiveReplyToId`, which is undefined for threaded inbound posts when `replyToMode` is `off`. That makes this off-to-all promotion unreachable for the real off-mode thread case; carry the actual `post.root_id`/message thread id separately before promoting.
  - confidence: 0.84
- **[P2] Gate first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:207-212`
  - body: This flips `hasRepliedRef` for any explicit `replyToId`, even if the tool send targets a different Mattermost channel. An explicit cross-channel threaded send can consume the active thread's first-reply slot and make the next same-thread send fall back to the root, so the state transition needs the same active-channel guard as implicit inheritance.
  - confidence: 0.78
