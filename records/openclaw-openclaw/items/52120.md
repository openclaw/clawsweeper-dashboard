## Summary

Keep open: the Mattermost bug is valid and current main still lacks a complete fix, but this abandoned branch is not mergeable as-is because it resolves inherited thread context too late, mishandles active first/off thread contexts, and lacks exact-branch real behavior proof.

## What This Changes

The PR adds Mattermost tool-context construction, fallback reply target inheritance in the send action, and unit tests for message-tool sends without explicit reply targets.

## Best Possible Solution

Rework or replace this PR on current main with a Mattermost-owned buildToolContext plus resolveAutoThreadId path that carries the actual active thread root before routing and mirroring, backed by focused tests and redacted real Mattermost proof.

## Reproduction Assessment

Yes, at source level. Current main still lacks Mattermost buildToolContext/resolveAutoThreadId, and the linked issue plus a later real-world comment give concrete replyToMode all Mattermost steps where message(action="send") posts at the channel root.

## Solution Assessment

No, not as-is. The fix should resolve the inherited Mattermost thread root before shared routing and mirroring, and it should use the actual active root for first/off modes instead of adding a late fallback only inside plugin handleAction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:204-206`
  - body: This fallback chooses the inherited replyToId inside the Mattermost action, after shared outbound routing and mirror preparation have already run. A tool send can post in a thread while route, mirror, and session metadata stay rooted at the channel; add a Mattermost resolveAutoThreadId or equivalent pre-dispatch resolver.
  - confidence: 0.86
- **[P2] Promote active threaded first-mode sends:** `extensions/mattermost/src/channel.ts:492-493`
  - body: buildToolContext preserves replyToMode="first" even when the current Mattermost session is already inside a thread. After hasRepliedRef is true, later tool sends stop inheriting the active root and escape to the channel root; active threaded sessions should keep using their current root.
  - confidence: 0.81
- **[P2] Read off-mode thread context from the real root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: The off-to-all promotion checks MessageThreadId or ReplyToId, but Mattermost currently derives both from effectiveReplyToId, which is undefined for replyToMode="off" even when the inbound post has a root_id. Carry the actual Mattermost thread root into tool context.
  - confidence: 0.78
- **[P2] Scope first-reply consumption to the active target:** `extensions/mattermost/src/channel.ts:211-218`
  - body: This flips hasRepliedRef for any explicit replyToId under replyToMode="first". An explicit threaded send to another channel can consume the current session's first-reply slot, causing the next send in the active thread to fall back to the channel root.
  - confidence: 0.72
