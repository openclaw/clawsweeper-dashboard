## Summary

## What This Changes

Adds Mattermost message-tool thread-context inheritance by adding a threading `buildToolContext` hook, falling back to `currentThreadTs` in `handleAction`, and adding Mattermost channel unit tests.

## Best Possible Solution

Replace or rework this on current main so Mattermost supplies `buildToolContext` and `resolveAutoThreadId` before shared outbound routing, with focused tests and redacted live Mattermost proof.

## Reproduction Assessment

Yes, at source level. Current main lacks Mattermost tool-context and auto-thread hooks for proactive sends, and the discussion includes a concrete real-world `replyToMode: "all"` Mattermost reproduction where message-tool sends escape an active thread.

## Solution Assessment

No. The direction is useful, but resolving thread inheritance only inside `handleAction` is too late for shared outbound route/session preparation; the maintainable fix should use Mattermost threading hooks before dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Move inherited threads before dispatch:** `extensions/mattermost/src/channel.ts:201-203`
  - body: This fallback sets the implicit Mattermost thread only inside `handleAction`, after current main has already prepared the outbound mirror/session route. A send can therefore post in a thread while `resolveMattermostOutboundSessionRoute` and transcript mirroring stay on the channel-root session; expose this through `resolveAutoThreadId` so the route and delivery use the same root id.
  - confidence: 0.86
- **[P2] Base off-mode promotion on the real thread root:** `extensions/mattermost/src/channel.ts:495-496`
  - body: `threadTs` comes only from `MessageThreadId`/`ReplyToId`, but Mattermost currently derives those from `effectiveReplyToId`, which is undefined for threaded inbound posts when `replyToMode` is `off`. That makes this off-to-all promotion unreachable for the real off-mode thread case; carry the actual `post.root_id` or message thread id separately before promoting.
  - confidence: 0.84
- **[P2] Gate first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:207-212`
  - body: This flips `hasRepliedRef` for any explicit `replyToId`, even if the tool send targets a different Mattermost channel. An explicit cross-channel threaded send can consume the active thread's first-reply slot and make the next same-thread send fall back to the root, so the state transition needs the same active-channel guard as implicit inheritance.
  - confidence: 0.78
