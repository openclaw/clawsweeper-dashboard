## Summary

Keep open. Current main still lacks Mattermost tool-thread hooks, but this external branch is not merge-ready: it is conflicting and abandoned, lacks after-fix Mattermost proof, and resolves inherited threads too late for shared outbound routing.

## What This Changes

The branch adds Mattermost message-tool thread-context inheritance via `buildToolContext`, a `handleAction` fallback, and unit tests in the Mattermost channel plugin.

## Best Possible Solution

Replace or rework this on current main with Mattermost-owned `buildToolContext` plus `resolveAutoThreadId`/actual-root handling before plugin dispatch, with focused tests and redacted live Mattermost proof.

## Reproduction Assessment

Yes, at source level. Current main only reads explicit Mattermost reply fields and lacks Mattermost `buildToolContext`/`resolveAutoThreadId`, while the linked issue and later comment provide concrete `replyToMode: "all"` reproduction steps.

## Solution Assessment

No, not as-is. The direction is useful, but the fix should resolve the active Mattermost thread root in the shared pre-dispatch routing path instead of relying on a late `handleAction` fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve inherited threads before plugin dispatch:** `extensions/mattermost/src/channel.ts:201-203`
  - body: The fallback chooses the inherited Mattermost reply inside `handleAction`, after shared reply injection and outbound route preparation have already run. Because the new tool context also exposes `currentMessageId`, current main can write `params.replyTo` first and bypass this root fallback; move the inheritance into `resolveAutoThreadId` or an equivalent pre-dispatch path.
  - confidence: 0.87
- **[P2] Promote active first-mode threads consistently:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the inbound Mattermost message is already in a thread, preserving `replyToMode="first"` lets later message-tool sends escape to the channel root after `hasRepliedRef` flips. Current docs say Mattermost `first` and `all` are equivalent once a thread root exists, so active threaded sessions should keep using that root.
  - confidence: 0.8
- **[P2] Carry the actual off-mode root into tool context:** `extensions/mattermost/src/channel.ts:484-496`
  - body: The off-to-all promotion depends on `threadTs` from `MessageThreadId` or `ReplyToId`, but current Mattermost inbound code derives those from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when the post has `root_id`. Build the tool context from the actual Mattermost thread root so off-mode threaded sessions can stay in the current thread.
  - confidence: 0.78
