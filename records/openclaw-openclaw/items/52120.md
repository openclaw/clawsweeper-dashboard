## Summary

Keep open. Current main still has the Mattermost message-tool thread-context gap, but this PR is abandoned, merge-conflicting, lacks after-fix real Mattermost proof, and its patch resolves the inherited root too late in the send pipeline.

## What This Changes

The PR adds Mattermost `buildToolContext`, late `handleAction` thread inheritance, and unit tests for message-tool sends without explicit `replyTo`.

## Best Possible Solution

Replace or rework this on current main with Mattermost-owned `buildToolContext` plus pre-dispatch `resolveAutoThreadId`/actual-root handling, coordinated with the open Mattermost thread-routing issue and companion PRs.

## Reproduction Assessment

Yes, at source level. Current main lacks Mattermost `buildToolContext` and `resolveAutoThreadId`, the linked issue gives concrete `replyToMode: "all"` steps, and a later comment reports the same real Mattermost `root_id` escape.

## Solution Assessment

No, not as-is. The fix should carry the actual Mattermost thread root into the shared pre-dispatch reply/thread routing path instead of relying on a late plugin `handleAction` fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve inherited threads before plugin dispatch:** `extensions/mattermost/src/channel.ts:201-203`
  - body: The fallback chooses the inherited Mattermost reply inside `handleAction`, after shared reply injection and outbound route preparation have already run. Because the new tool context also exposes `currentMessageId`, current main can write `params.replyTo` first and bypass the Mattermost root fallback; move this into `resolveAutoThreadId` or an equivalent pre-dispatch path.
  - confidence: 0.87
- **[P2] Promote active first-mode threaded sends:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the inbound Mattermost message is already in a thread, preserving `replyToMode="first"` lets later message-tool sends escape to the channel root after `hasRepliedRef` flips. Current docs say Mattermost `first` and `all` are equivalent once a thread root exists, so active threaded sessions should keep using that root.
  - confidence: 0.8
- **[P2] Carry the real off-mode root into tool context:** `extensions/mattermost/src/channel.ts:484-496`
  - body: The off-to-all promotion depends on `threadTs` from `MessageThreadId` or `ReplyToId`, but current Mattermost inbound code derives those from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when the post has `root_id`. Build the tool context from the actual Mattermost thread root so off-mode threaded sessions can stay in the current thread.
  - confidence: 0.78
