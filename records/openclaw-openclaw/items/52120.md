## What This Changes

The PR adds Mattermost tool-context plumbing and adapter-side fallback logic so message-tool sends from an active Mattermost thread can inherit the current thread root when no explicit reply target is supplied.

## Best Possible Solution

Land a refreshed Mattermost/threading fix on current main that resolves the thread root before both send and mirror routing, preserves current attachment and presentation handling, and closes the linked issue only after proof.

## Reproduction Assessment

Yes. From source inspection, current main and v2026.6.6 compute Mattermost direct-send reply targets only from replyToId/replyTo and do not build Mattermost tool context, matching the reported root-channel send; I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

No. This branch is a plausible old mitigation, but it resolves implicit thread context inside the adapter after outbound mirror routing and now conflicts with current main; the best fix is a refreshed shared/Mattermost-owned threading path before route preparation.
