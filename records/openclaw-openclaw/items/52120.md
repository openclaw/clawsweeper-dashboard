## Summary

Keep open. Current main still lacks the Mattermost tool-context hook needed for the linked thread-inheritance bug, but this PR is stale, abandoned by its author, and conflicts with current main's shared outbound threading flow, so it should not merge as-is.

## What This Changes

The PR adds Mattermost message-tool thread inheritance by building Mattermost tool context, falling back send replies to the active thread root, and adding unit coverage for the new send-path guards.

## Best Possible Solution

Adapt the Mattermost-owned fix to current main: add a Mattermost buildToolContext plus pre-dispatch auto-thread integration so message-tool sends use the active thread root before routing/mirroring, preserve explicit reply precedence and same-channel guards, promote active thread sessions consistently with Mattermost semantics, and cover the behavior in focused Mattermost and shared outbound tests.

## Reproduction Assessment

Yes. The PR body and linked #45082 give a concrete Mattermost reproduction with replyToMode="all" in a thread-scoped session, and current main source inspection shows the missing buildToolContext/currentThreadTs path that makes message-tool sends lose the active thread.

## Solution Assessment

No. The requested direction is right, but this PR is not the best current-main solution because it applies the fallback inside Mattermost handleAction after current main has already resolved generic reply injection and because it preserves first mode inside an existing Mattermost thread.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Resolve Mattermost threading before generic reply injection:** `extensions/mattermost/src/channel.ts:504`
  - body: On current main, send actions call the shared reply injector before Mattermost handleAction runs. Because this PR returns a same-channel currentChannelId and currentMessageId, the shared path can set params.replyTo to the latest post id first, masking the new currentThreadTs fallback and routing/mirroring under the wrong Mattermost root. Move this through resolveAutoThreadId/transport or another pre-dispatch path that uses the thread root.
  - confidence: 0.78
- **[P2] Promote active first-mode threads to all:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the active Mattermost session is already inside a thread, preserving replyToMode="first" makes later tool sends stop inheriting currentThreadTs after hasRepliedRef is true and fall back to the channel root. Mattermost's documented behavior keeps first/all equivalent once a thread root exists, so active thread context should use all-mode semantics.
  - confidence: 0.82
