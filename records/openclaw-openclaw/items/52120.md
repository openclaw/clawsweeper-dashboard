## Summary

Keep open: the Mattermost bug is still source-reproducible and this PR remains useful context, but the branch should not merge as-is because its late plugin fallback conflicts with current shared routing/reply injection and the author has abandoned it.

## What This Changes

The PR adds Mattermost threading tool context, late send-action replyTo inheritance from currentThreadTs, and unit coverage for thread-context message-tool sends.

## Best Possible Solution

Rework or replace this on current main with Mattermost-owned buildToolContext plus pre-dispatch resolveAutoThreadId/current-root handling, coordinated with the open Mattermost thread-routing issue and companion PRs.

## Reproduction Assessment

Yes, at source level. Current main lacks Mattermost buildToolContext/resolveAutoThreadId, the linked issue gives concrete replyToMode all steps, and a later comment reports the same real Mattermost root_id escape.

## Solution Assessment

No, not as-is. The fix should carry the actual Mattermost thread root into shared pre-dispatch reply/routing before plugin send, rather than adding only a late handleAction fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve inherited threads before plugin dispatch:** `extensions/mattermost/src/channel.ts:201-203`
  - body: The fallback chooses the inherited Mattermost reply inside handleAction, after shared reply injection and outbound mirror/session routing have already run. On current main, the new buildToolContext can also cause the shared runner to write params.replyTo from currentMessageId first, so this line treats that as explicit and bypasses currentThreadTs; move the Mattermost root resolution into resolveAutoThreadId or an equivalent pre-dispatch path.
  - confidence: 0.87
- **[P2] Promote active first-mode threaded sends:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the inbound Mattermost message is already in a thread, preserving replyToMode="first" means later message-tool sends stop inheriting the active root after hasRepliedRef flips and can escape to the channel root. Active threaded sessions should keep using the current thread root instead of gating on first-mode state.
  - confidence: 0.8
- **[P2] Carry the real off-mode root into tool context:** `extensions/mattermost/src/channel.ts:484-496`
  - body: The off-to-all promotion depends on threadTs from MessageThreadId or ReplyToId, but current Mattermost inbound code derives those from effectiveReplyToId, which is undefined for replyToMode="off" even when the post has root_id. Build the tool context from the actual Mattermost thread root so off-mode threaded sessions can stay in the current thread.
  - confidence: 0.78
- **[P2] Scope first-reply consumption to the active target:** `extensions/mattermost/src/channel.ts:207-212`
  - body: This flips hasRepliedRef for any explicit replyToId under replyToMode="first". An explicit threaded send to another channel can consume the active session's first-reply slot and make the next send in the current thread fall back to the channel root; only consume it for the active target or the implicit same-channel path.
  - confidence: 0.72
