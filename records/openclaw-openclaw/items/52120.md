## Summary

Keep this PR open as useful Mattermost bug-fix context, but it is not merge-ready: current main still lacks Mattermost tool-thread inheritance, while this abandoned/conflicting branch resolves the implicit thread inside the channel handler after shared outbound routing has already chosen the mirror/session route.

## What This Changes

The PR adds Mattermost tool-thread context, falls back to session thread context in Mattermost message-tool sends without an explicit reply target, and adds unit tests for replyToMode guard paths.

## Best Possible Solution

Replace or rebase this branch with a current-main fix that feeds Mattermost implicit thread IDs into the shared pre-dispatch route path, coordinates with the related Mattermost thread PRs, and includes redacted real Mattermost proof.

## Reproduction Assessment

Yes, at source level. Current main has no Mattermost buildToolContext/resolveAutoThreadId and a later user comment reports repeated replyToMode all message-tool sends escaping active Mattermost threads.

## Solution Assessment

No. The handler-local fallback is not the best current-main shape because shared outbound routing prepares reply/thread IDs before plugin dispatch; Mattermost should plug into that route-time contract instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Move implicit thread resolution before outbound routing:** `extensions/mattermost/src/channel.ts:207`
  - body: This fallback only changes the Mattermost handler's local replyToId. On current main, the shared runner prepares the outbound mirror/session route before invoking the plugin handler, so the message can be posted in the thread while the mirrored transcript and follow-up session route stay attached to the channel root.
  - confidence: 0.86
- **[P2] Derive off-mode promotion from the actual thread root:** `extensions/mattermost/src/channel.ts:484-489`
  - body: The new off-to-all promotion depends on context.MessageThreadId or context.ReplyToId, but Mattermost currently sets those from effectiveReplyToId, which is undefined for threaded input when replyToMode is off. Real off-mode threaded sessions therefore will not populate currentThreadTs through this path, even though the tests stub the already-promoted context.
  - confidence: 0.82
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:214-220`
  - body: The patch flips hasRepliedRef for any non-empty replyToId under replyToMode="first", including explicit sends to a different Mattermost channel. That can consume the current session's first-reply slot before a later send back to the active thread, causing that later send to miss the inherited thread.
  - confidence: 0.74
