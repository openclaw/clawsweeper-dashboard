## Summary

Keep this PR open for maintainer takeover or replacement: the underlying Mattermost bug is still source-reproducible on current main and has a later real-world reproduction comment, but this branch resolves the fallback too late in the send path and still misses active-root cases.

## What This Changes

The PR adds Mattermost message-tool thread-context inheritance and regression tests for sends without an explicit reply target.

## Best Possible Solution

Rework or replace this PR on current main with a Mattermost-owned buildToolContext plus resolveAutoThreadId path that carries the actual active thread root before shared routing and mirror preparation.

## Reproduction Assessment

Yes, at source level. Current main still lacks Mattermost buildToolContext/resolveAutoThreadId, and the linked issue plus a later real-world comment give concrete replyToMode all Mattermost steps where message(action="send") posts at the channel root.

## Solution Assessment

No, not as-is. The fix should surface the inherited Mattermost thread root before shared routing and mirroring, and should use the actual active thread root instead of resolving the fallback only inside plugin handleAction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve Mattermost thread inheritance before dispatch:** `extensions/mattermost/src/channel.ts:204-206`
  - body: The fallback chooses the inherited replyToId only inside the Mattermost action, after shared outbound routing and mirror preparation have already run. Add a Mattermost resolveAutoThreadId or equivalent pre-dispatch hook so delivery, session routing, and mirroring all use the same thread.
  - confidence: 0.86
- **[P2] Keep active first-mode sessions on the active thread:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the current Mattermost session is already inside a thread, preserving replyToMode="first" makes later tool sends stop inheriting the active root after hasRepliedRef is true and post at the channel root. Active threaded sessions should keep using their root instead of being treated like a fresh top-level first reply.
  - confidence: 0.81
- **[P2] Build off-mode context from the actual thread root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: The off-to-all promotion depends on MessageThreadId or ReplyToId, but current Mattermost derives those from effectiveReplyToId, which is undefined for replyToMode="off" even when the inbound post has a root. Use the actual Mattermost thread root/session context so off-mode threaded sessions can be represented correctly.
  - confidence: 0.78
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:211-218`
  - body: The hasRepliedRef flip runs for any non-empty explicit replyToId under replyToMode="first", even if the send targets another channel. That can consume the active session's first-reply slot and make later sends in the current thread fall back to the channel root.
  - confidence: 0.72
