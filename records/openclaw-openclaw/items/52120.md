## Summary

Keep open: the Mattermost bug is real and the PR contains useful source work, but the current branch is abandoned, merge-conflicting, lacks after-fix real behavior proof, and still has correctness issues around where the active thread root is resolved.

## What This Changes

The PR adds Mattermost message-tool thread context inheritance for sends without explicit reply targets and adds unit coverage for the new guard paths.

## Best Possible Solution

Replace or rework this branch with a current-main Mattermost fix that builds tool context from the actual active thread root and resolves auto-threading before mirror/session routing and plugin dispatch.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main has no Mattermost buildToolContext/resolveAutoThreadId and the Mattermost send handler only forwards explicit reply fields, while the discussion includes a real Mattermost report of message-tool sends escaping active threads.

## Solution Assessment

No, not as a direct merge. The direction is useful, but the branch resolves implicit roots inside the plugin too late for shared routing and builds first/off-mode context from fields that do not always carry the actual Mattermost root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve the root before shared reply injection:** `extensions/mattermost/src/channel.ts:504`
  - body: Returning `currentMessageId` here lets the shared runner set `params.replyTo` before Mattermost `handleAction` runs. In a threaded child-post session that explicit value wins over `currentThreadTs`, so routing and delivery can use the child post instead of the active root.
  - confidence: 0.82
- **[P2] Keep active first-mode sessions on the root:** `extensions/mattermost/src/channel.ts:493-496`
  - body: When the active Mattermost session is already in a thread, preserving `replyToMode="first"` means later tool sends stop inheriting after `hasRepliedRef` is true and post at channel root. Current monitor delivery keeps first-mode threaded replies on the existing root, so the tool context should do the same.
  - confidence: 0.81
- **[P2] Build off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:484-489`
  - body: This only promotes `off` to `all` when `MessageThreadId` or `ReplyToId` is populated, but current Mattermost derives both from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when `post.root_id` exists. Use the actual Mattermost root instead.
  - confidence: 0.78
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:207-213`
  - body: The new flip runs for any explicit `replyToId` in `first` mode, including sends to another target. That can consume the current session's first-reply slot and make a later same-thread send fall back to the channel root.
  - confidence: 0.72
