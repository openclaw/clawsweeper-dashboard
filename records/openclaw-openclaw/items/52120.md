## Summary

Keep this PR open for maintainer handling rather than cleanup-close: the Mattermost bug is still source-reproducible on current main, but this abandoned external branch is conflicting, lacks after-fix real behavior proof, and still has correctness issues around when and where thread context is resolved.

## What This Changes

The PR adds Mattermost thread-context inheritance to the message tool send action and unit tests for implicit and explicit `replyTo` handling.

## Best Possible Solution

Replace or rework this branch with a current-main Mattermost fix that builds tool context from the actual active thread root and resolves auto-threading before shared route/mirror preparation, with focused regression coverage and real Mattermost proof.

## Reproduction Assessment

Yes, at source level. Current main still lacks Mattermost `buildToolContext`/`resolveAutoThreadId`, and the linked issue plus later PR comment give concrete `replyToMode: "all"` Mattermost steps where `message(action="send")` posts at the channel root.

## Solution Assessment

No, not as-is. The fix should surface the inherited Mattermost root before shared routing/mirroring and use the actual active thread root for first/off-mode cases, rather than setting `replyToId` only inside `handleAction`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:204-206`
  - body: This fallback chooses the inherited `replyToId` only inside the Mattermost action, after shared outbound routing and mirror preparation have already run. Add a Mattermost `resolveAutoThreadId` or equivalent pre-dispatch hook so delivery, session routing, and mirroring all use the same thread.
  - confidence: 0.84
- **[P2] Keep active first-mode sessions on the thread:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the current Mattermost session is already inside a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting the active root after `hasRepliedRef` is true and post at the channel root. Active threaded sessions should keep using their root instead of being treated like a fresh top-level first reply.
  - confidence: 0.81
- **[P2] Build off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: The off-to-all promotion depends on `MessageThreadId` or `ReplyToId`, but current Mattermost derives those from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when the inbound post has a root. Use the actual Mattermost thread root/session context so off-mode threaded sessions can be represented correctly.
  - confidence: 0.77
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:211-218`
  - body: The new `hasRepliedRef` flip runs for any non-empty explicit `replyToId` under `replyToMode="first"`, even if the explicit send targets another channel. That can consume the active session's first-reply slot and make later sends in the current thread fall back to the channel root.
  - confidence: 0.72
