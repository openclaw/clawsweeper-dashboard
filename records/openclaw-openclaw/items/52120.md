## Summary

Keep open: the underlying Mattermost bug remains source-reproducible and has fresh user confirmation, but this PR is not mergeable as-is because its fallback runs too late and mishandles first/off-mode thread roots.

## What This Changes

Adds Mattermost `buildToolContext`, implicit thread inheritance in `handleAction`, and regression tests for message-tool sends without explicit `replyTo`.

## Best Possible Solution

Replace or rework this branch with a current-main Mattermost-owned fix that builds context from the actual active root and resolves auto-threading before shared routing, mirroring, and plugin dispatch.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main has no Mattermost `buildToolContext`/`resolveAutoThreadId`, and its send action only forwards explicit reply fields; a 2026-05-04 comment also reports repeated live Mattermost failures with `replyToMode: "all"`.

## Solution Assessment

No as a direct merge. The PR’s direction is useful, but resolving the implicit root inside Mattermost `handleAction` is too late for shared outbound routing/mirroring and the context builder needs the actual active root for first/off-mode cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:204`
  - body: This fallback runs only inside the Mattermost plugin action, after the shared message runner has already applied reply/thread state and prepared the mirror/session route. Expose the inherited root through `threading.resolveAutoThreadId` or an equivalent pre-dispatch hook so routing and delivery use the same thread.
  - confidence: 0.82
- **[P2] Keep active first-mode sessions on the thread:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the current Mattermost session is already inside a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting the active root after `hasRepliedRef` is true and post at the channel root. Active threaded sessions should keep using their root instead of being treated like a fresh first reply.
  - confidence: 0.81
- **[P2] Build off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: The off-to-all promotion depends on `MessageThreadId`/`ReplyToId`, but Mattermost currently derives those from `effectiveReplyToId`, which is intentionally undefined for `replyToMode="off"` even when the inbound post has a root. Use the actual Mattermost thread root/session context so off-mode threaded sessions can be represented correctly.
  - confidence: 0.76
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:211-218`
  - body: The new `hasRepliedRef` flip runs for any non-empty explicit `replyToId` under `replyToMode="first"`, even if the explicit send targets another channel. That can consume the active session's first-reply slot and make later sends in the current thread fall back to the channel root.
  - confidence: 0.72
