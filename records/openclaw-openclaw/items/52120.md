## Summary

Keep open: the Mattermost bug is valid and current main still lacks the needed tool-context/auto-thread handling, but this abandoned PR should not merge as-is because it resolves the fallback after shared routing and still mishandles active `first` and `off` threaded sessions.

## What This Changes

The PR adds a Mattermost threading tool-context builder, send-action fallback to active thread context, and unit coverage for `replyToMode` guard paths.

## Best Possible Solution

Land a current-main Mattermost fix that uses the existing plugin threading seams to supply the actual active thread root before shared reply injection, mirror routing, and plugin dispatch, with focused regression coverage across `all`, `first`, and `off` modes.

## Reproduction Assessment

Yes. The linked issue and later PR comment give concrete Mattermost `replyToMode: "all"` steps, and current-main source shows message-tool sends only honor explicit reply fields while Mattermost lacks the tool-context/auto-thread hooks needed to supply the thread root.

## Solution Assessment

No. The direction is right, but this PR resolves Mattermost thread inheritance inside plugin `handleAction` after shared reply injection and routing have already run, and its builder still does not derive active `first` and `off` thread roots correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:204`
  - body: This fallback runs inside Mattermost `handleAction`, but the shared runner has already applied reply injection and prepared mirror routing before plugin dispatch. Once `buildToolContext` is present, generic reply injection can write `params.replyTo` from `currentMessageId`, masking this fallback and routing session state to the latest message instead of the Mattermost thread root; wire this through `resolveAutoThreadId` or an equivalent pre-dispatch hook.
  - confidence: 0.78
- **[P2] Promote active first-mode sessions to the root:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the active Mattermost session is already in a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting `currentThreadTs` after `hasRepliedRef` is true and post at the channel root. Current Mattermost threaded reply behavior keeps follow-ups on the active root, so the tool context should preserve the root for active threaded sessions.
  - confidence: 0.82
- **[P2] Derive off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: The builder promotes `off` only when `MessageThreadId` or `ReplyToId` is present, but current Mattermost derives those fields from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when the inbound post has a thread root. Use the actual Mattermost thread root/session context instead of the already-filtered reply id.
  - confidence: 0.7
