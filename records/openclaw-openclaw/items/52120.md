## Summary

Keep open: current main still has the Mattermost message-tool thread-context gap, but this abandoned and conflicting PR is not the right patch to merge as-is because it resolves the root too late and mishandles first/off-mode thread context.

## What This Changes

The PR adds Mattermost `buildToolContext`, `handleAction` implicit thread inheritance, and unit tests for message-tool sends without explicit `replyTo`.

## Best Possible Solution

Replace or rework this branch with a current-main Mattermost-owned fix that builds tool context from the actual active thread root and resolves auto-threading before shared reply injection, mirror routing, and plugin dispatch.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main lacks Mattermost `buildToolContext`/`resolveAutoThreadId`, and the send action only forwards explicit reply fields, so a thread-scoped `message(action="send")` without `replyTo` has no path to the active root.

## Solution Assessment

No as a direct merge. The fix direction is useful, but resolving the implicit root inside Mattermost `handleAction` is too late and the context builder should use the actual active thread root for first/off-mode cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:204`
  - body: This fallback runs only inside the Mattermost plugin action, after the shared message runner has already applied `replyTo` and prepared the mirror/session route. Thread inheritance needs to be exposed through `threading.resolveAutoThreadId` or an equivalent pre-dispatch hook so routing and delivery use the same root.
  - confidence: 0.82
- **[P2] Keep active first-mode threads on the root:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the current Mattermost session is already inside a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting the active root after `hasRepliedRef` is true and post at the channel root. The context builder should keep active threaded sessions on their root instead of treating them like a fresh first reply.
  - confidence: 0.81
- **[P2] Build off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:481-493`
  - body: This promotion depends on `MessageThreadId`/`ReplyToId`, but current Mattermost derives those from `effectiveReplyToId`, which is intentionally undefined for `replyToMode="off"` even when the inbound post has a root. Use the actual Mattermost thread root/session context so off-mode threaded sessions can be represented correctly.
  - confidence: 0.76
- **[P2] Scope first-reply consumption to the active channel:** `extensions/mattermost/src/channel.ts:211-218`
  - body: The new `hasRepliedRef` flip runs for any non-empty explicit `replyToId` under `replyToMode="first"`, even if the explicit send targets another channel. That can consume the active session's first-reply slot and make later sends in the current thread fall back to the channel root.
  - confidence: 0.72
