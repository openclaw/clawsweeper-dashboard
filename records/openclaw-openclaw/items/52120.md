## Summary

Keep this PR open: the Mattermost message-tool bug is still present on current main and the PR is useful source work, but the proposed patch is not mergeable as-is because it injects Mattermost thread fallback after shared reply/routing logic has already run.

## What This Changes

The PR adds Mattermost `buildToolContext` support, late `handleAction` thread fallback logic, and unit tests so message-tool sends without explicit `replyTo` inherit active Mattermost thread context.

## Best Possible Solution

Adapt the useful Mattermost-owned fix onto current main by adding a Mattermost tool-context builder plus `resolveAutoThreadId`/pre-dispatch routing support that supplies the active thread root before shared reply injection, mirror routing, and plugin dispatch.

## Reproduction Assessment

Yes. Current main source shows Mattermost tool sends only honor explicit `replyTo` fields and the Mattermost plugin does not provide the hook needed to expose the active thread root; the PR discussion also includes concrete `replyToMode: "all"` live repro steps.

## Solution Assessment

No. The direction is right, but this patch resolves Mattermost thread inheritance inside plugin `handleAction`, after shared reply injection and outbound session routing have already made decisions, and its context builder mishandles active `first` and `off` thread cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve Mattermost auto-threading before dispatch:** `extensions/mattermost/src/channel.ts:203`
  - body: Choosing the fallback here is too late: the shared runner has already called `resolveAndApplyOutboundReplyToId` and prepared the outbound mirror route before Mattermost `handleAction` runs. With `buildToolContext` setting `currentMessageId`, shared reply injection can write the latest post id into `params.replyTo`, masking this thread-root fallback and routing the send under the wrong session; wire Mattermost through `resolveAutoThreadId` or an equivalent pre-dispatch hook instead.
  - confidence: 0.84
- **[P2] Promote active first-mode threads to the root:** `extensions/mattermost/src/channel.ts:493-496`
  - body: When the active Mattermost session is already inside a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting `currentThreadTs` after `hasRepliedRef` is true and post at the channel root. Current Mattermost thread handling keeps `first` follow-ups on the existing root, so the tool context should promote active threaded sessions rather than preserve `first`.
  - confidence: 0.82
- **[P2] Build off-mode context from the actual root:** `extensions/mattermost/src/channel.ts:484-496`
  - body: This promotes `off` to `all` only when `MessageThreadId` or `ReplyToId` is present, but current Mattermost derives those fields from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even when the inbound post has a thread root. Use the actual Mattermost thread root/session context instead of the already-filtered reply id.
  - confidence: 0.76
