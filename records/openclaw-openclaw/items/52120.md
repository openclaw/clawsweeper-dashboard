## Summary

Keep open. The Mattermost thread-context bug still appears valid on current main, but the abandoned PR branch is not safe to merge as-is because it wires the fix after newer shared outbound threading has already injected generic reply state and it preserves `first` mode in active Mattermost threads contrary to the current Mattermost thread behavior.

## What This Changes

The PR adds Mattermost `threading.buildToolContext` support plus Mattermost `handleAction` send-path fallback logic and unit tests so message-tool sends without explicit `replyTo` inherit the active Mattermost thread.

## Best Possible Solution

Adapt the useful Mattermost-owned fix to current main: add a Mattermost tool-context builder that supplies normalized conversation id, active thread root, account-scoped reply mode, and `hasRepliedRef`; integrate with `resolveAutoThreadId` or the shared outbound threading helper so the thread root is applied before routing, mirroring, and plugin dispatch; keep explicit reply precedence and same-channel guards; and add focused tests for `all`, active `first`, `off`, explicit replies, and cross-channel sends.

## Reproduction Assessment

Yes. The PR body and linked #45082 give concrete Mattermost `replyToMode: "all"` thread-scoped message-tool steps, and current main source inspection shows the missing Mattermost `buildToolContext`/`resolveAutoThreadId` path needed for the message tool to inherit the active thread root.

## Solution Assessment

No. The requested direction is correct, but this PR is not the best current-main solution because it resolves Mattermost thread inheritance inside plugin `handleAction` after the shared runner has already applied generic reply state, and it intentionally stops active `first`-mode threads from continuing on the existing Mattermost root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Route Mattermost auto-threading before generic reply injection:** `extensions/mattermost/src/channel.ts:204`
  - body: On current main, adding `currentChannelId` and `replyToMode` lets the shared message-action runner set `params.replyTo` from `currentMessageId` before Mattermost `handleAction` runs. That explicit reply then masks this fallback, so a send from inside an existing Mattermost thread can use the latest message id as `root_id` instead of the thread root. Wire Mattermost through `resolveAutoThreadId` or an equivalent shared pre-dispatch hook instead of resolving only here.
  - confidence: 0.78
- **[P2] Promote active first-mode threads to all:** `extensions/mattermost/src/channel.ts:492-493`
  - body: When the active Mattermost session is already inside a thread, preserving `replyToMode="first"` makes later tool sends stop inheriting `currentThreadTs` after `hasRepliedRef` is true and post at the channel root. Current Mattermost thread handling keeps `first` follow-ups on the active root, so the effective tool mode should continue to thread once a root exists.
  - confidence: 0.82
- **[P2] Derive off-mode promotion from actual thread presence:** `extensions/mattermost/src/channel.ts:481-493`
  - body: This promotes `off` to `all` only when `MessageThreadId` or `ReplyToId` is present, but Mattermost currently derives those fields from `effectiveReplyToId`, which is undefined for `replyToMode="off"` even if the inbound post has a thread root. If the intent is to keep existing off-mode threads threaded, build the tool context from the actual thread root/session context instead of the already-filtered reply id.
  - confidence: 0.7
