## Summary

Keep open: this is an active protected `maintainer`-labeled PR, and the current head still needs maintainer attention because the new strict npm peer planner can abort third-party peer-conflict installs before the documented legacy install path runs.

## What This Changes

The PR replaces managed npm peer-pin discovery with a temp npm package-lock-only planner, threads timeout and rollback handling through install peer sync, updates tests, and adds a changelog entry.

## Best Possible Solution

Preserve the lockfile-planner refactor only if it keeps the documented legacy managed-install behavior, likely by adding a non-host peer fallback that can hydrate/read a legacy-peer lock plan without rejecting installs that current main can handle safely.

## Reproduction Assessment

No live current-main reproduction was established in this read-only review. The source-level path is clear: a non-`openclaw` peer resolution failure from the temp npm planner reaches the early `npm peer plan failed` path before the legacy managed install flow can run.

## Solution Assessment

No: the planner direction is reasonable, but the current diff is not the best fix while it treats non-host peer planning failures as fatal before preserving the existing legacy-peer install behavior. A safer patch would keep rollback while falling back for third-party peer conflicts.
