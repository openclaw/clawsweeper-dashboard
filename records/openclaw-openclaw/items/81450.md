## Summary

Keep open: this is an active protected maintainer-labeled PR, and the latest head still has a blocking managed-peer correctness issue in the new lockfile version lookup.

## What This Changes

The PR replaces managed npm peer discovery by installed-tree scanning with npm package-lock planning, fallback handling, install rollback/timeout plumbing, tests, and a changelog entry.

## Best Possible Solution

Keep the npm lockfile planner direction, but restrict peer version pinning to the root lockfile entry or fall back to the peer range when the root entry is absent, with a regression test for nested incompatible peer entries.

## Reproduction Assessment

Yes. A synthetic lockfile where a peer is absent at `node_modules/<peer>` but present only under a nested package path exercises the current PR code path and shows the wrong version can be pinned.

## Solution Assessment

No, not as currently written. The planner approach is maintainable, but the fallback version lookup should not promote nested transitive lockfile versions into root-managed peer pins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit peer version lookup to root lockfile entries:** `src/infra/npm-managed-root.ts:335-349`
  - body: When the root lock entry for a peer is absent, this fallback scans every `packages` entry and returns the first matching package name. A nested `node_modules/a/node_modules/<peer>` entry can therefore pin an incompatible transitive version into the managed root instead of using the requested peer range, leaving the installed plugin with an unsatisfied peer.
  - confidence: 0.9
