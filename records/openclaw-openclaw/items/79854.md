## Summary

Keep open: current main still has the reported CLI-backed transcript asymmetry, and the remaining fix is a public Gateway contract decision rather than conservative cleanup. The adjacent open PR only filters transcript-only OpenClaw artifacts; it does not make `sessions.get` hydrate Claude CLI history or clarify `sessionFile` semantics.

## What This Changes

The issue asks for `sessions.get`, `chat.history`, and `sessionFile` metadata to have clearer, less divergent transcript behavior for CLI-backed sessions.

## Best Possible Solution

Define one Gateway transcript contract for CLI-backed sessions, then either make `sessions.get` share `chat.history` hydration behind a compatible API shape or explicitly expose/document wrapper transcript versus CLI transcript metadata.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings['claude-cli'].sessionId` reaches Claude CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` gateway scenario during this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` may solve the empty transcript symptom, but endpoint semantics, wrapper-only access, and transcript metadata naming are public protocol choices. A safer maintainable path is an additive contract clarification before changing defaults.
