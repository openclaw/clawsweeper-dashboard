## Summary

Keep open: current main still has two transcript surfaces for CLI-backed sessions, with `sessions.get` reading only the OpenClaw session transcript while `chat.history` hydrates bound Claude CLI project history. The related open PR is scoped to artifact filtering, not this CLI hydration and `sessionFile` contract decision.

## What This Changes

The issue asks OpenClaw to clarify or change the Gateway session transcript contract so `sessions.get`, `chat.history`, and `sessionFile` do not mislead consumers for CLI-backed sessions.

## Best Possible Solution

Define the public Gateway transcript contract, then implement a compatibility-aware path such as additive scope metadata or an opt-in wrapper-only mode, with docs and tests covering CLI-backed sessions.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions.get` reads only `entry.sessionFile`, while `chat.history` imports bound Claude CLI project messages before projection; I did not run the live macOS Claude CLI scenario in this read-only review.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` fixes the empty-transcript symptom, but endpoint defaults, wrapper-only access, and `sessionFile` naming are public Gateway contract choices that should be made additively.
