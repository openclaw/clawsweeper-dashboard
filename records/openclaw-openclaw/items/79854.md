## Summary

Keep open: current main still has the reported Gateway transcript split, with `sessions.get` reading the OpenClaw wrapper transcript and `chat.history` using the CLI hydration path. The remaining fix is a public Gateway API contract decision, not conservative cleanup closure.

## What This Changes

The issue asks for `sessions.get`, `chat.history`, and `sessions.json` transcript metadata to stop misleading Gateway clients for CLI-backed sessions.

## Best Possible Solution

Define the Gateway transcript contract additively, then either share CLI hydration in `sessions.get` with an explicit wrapper-only escape hatch or document/expose wrapper and CLI transcript metadata as separate fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a CLI-backed session with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run a live macOS `claude-cli` gateway session.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the empty-conversation symptom, but endpoint semantics, a wrapper-only escape hatch, and `sessionFile` naming are public API decisions.
