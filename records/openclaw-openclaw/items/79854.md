## Summary

Keep open: current main still has the reported source-level divergence between `sessions.get` and `chat.history`, and changing endpoint semantics or session metadata names is a Gateway protocol decision rather than a conservative cleanup close.

## What This Changes

The issue requests a clearer and/or unified transcript contract because `sessions.get` returns OpenClaw wrapper transcript rows while `chat.history` hydrates CLI-backed chat history.

## Best Possible Solution

Define the Gateway transcript contract explicitly, then either make `sessions.get` share CLI hydration behind a compatible API shape or document and expose wrapper transcript metadata with additive fields.

## Reproduction Assessment

Yes. A session entry with `cliSessionBindings['claude-cli'].sessionId` is enough to prove the split from source: `chat.history` imports CLI history while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may match user expectations, but the endpoint semantics, wrapper-only escape hatch, and session metadata naming are public protocol decisions; an additive contract clarification is safer than an automated behavior change.
