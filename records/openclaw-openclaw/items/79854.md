## Summary

Keep open: current main still has separate transcript surfaces for CLI-backed sessions, with `sessions.get` reading only the OpenClaw wrapper transcript while `chat.history` imports Claude CLI history. The related open PR addresses transcript artifact filtering, not this API contract or `sessionFile` naming problem.

## What This Changes

The issue asks OpenClaw to make CLI-backed session transcript reads less misleading by hydrating `sessions.get`, renaming or clarifying `sessionFile`, or documenting the wrapper-versus-chat transcript distinction.

## Best Possible Solution

Define the Gateway transcript contract, then either share `chat.history` CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch, or add additive metadata and docs that distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes, by source inspection. A session entry with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` still reads only `entry.sessionFile`; I did not run the live macOS Claude CLI scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the observed empty transcript, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
