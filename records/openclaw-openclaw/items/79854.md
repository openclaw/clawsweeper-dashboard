## Summary

Keep open: current main still has the reported CLI-backed transcript split, and changing `sessions.get` defaults or `sessionFile` metadata is a public Gateway API contract decision rather than a safe cleanup close.

## What This Changes

The issue asks OpenClaw to clarify or align `sessions.get`, `chat.history`, and `sessionFile` semantics for CLI-backed sessions whose conversation lives in an external CLI transcript.

## Best Possible Solution

Define one public Gateway transcript contract, then either hydrate CLI-backed `sessions.get` with an explicit wrapper-only escape hatch or document a separate wrapper-transcript surface and expose CLI transcript metadata additively.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches external CLI import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may solve the observed empty transcript, but endpoint semantics, wrapper-only access, and persisted metadata naming are public protocol decisions that should be settled first.
