## Summary

Keep open: current main still has different transcript semantics for CLI-backed sessions, with `sessions.get` reading only the OpenClaw wrapper transcript while `chat.history` hydrates Claude CLI history. The linked open PR is adjacent artifact filtering work, but it does not resolve the CLI hydration or misleading `sessionFile` contract.

## What This Changes

The issue asks for `sessions.get` and `sessionFile` semantics to either return the full CLI-backed chat history or clearly expose/document wrapper-only transcripts.

## Best Possible Solution

Define the Gateway transcript contract, then either share `chat.history` CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch, or add clear additive metadata/docs that distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes. Source inspection shows a session with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run the live macOS `claude-cli` scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may solve the observed empty transcript, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
