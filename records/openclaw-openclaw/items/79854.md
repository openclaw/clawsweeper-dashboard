## Summary

Keep open: current main still has divergent transcript read behavior for CLI-backed sessions, and the right public protocol shape needs maintainer judgment before cleanup.

## What This Changes

Request to make `sessions.get`, `chat.history`, and `sessionFile` metadata consistent and self-explanatory for CLI-backed sessions.

## Best Possible Solution

Define one public transcript contract for Gateway clients, then either make `sessions.get` share `chat.history` CLI hydration with an explicit wrapper-only escape hatch or document a separate wrapper-transcript surface and expose CLI transcript metadata additively.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` gateway scenario during this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` may solve the observed empty transcript, but the endpoint semantics, `wrapperOnly` escape hatch, and `sessionFile` naming are public protocol decisions. A safer maintainable direction is an additive contract clarification before changing defaults.
