## Summary

Keep open: current main still has the reported CLI-backed transcript split, and changing `sessions.get` or `sessionFile` semantics is a Gateway API contract decision rather than a narrow cleanup close.

## What This Changes

The issue asks for Gateway transcript reads and session metadata to distinguish CLI-backed chat history from OpenClaw wrapper transcript storage.

## Best Possible Solution

Define an additive Gateway transcript contract: either share `chat.history` CLI hydration behind an explicit wrapper-only escape hatch, or document and expose wrapper transcript and CLI transcript metadata as distinct surfaces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a session entry with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the observed empty transcript, but endpoint defaults, a wrapper-only escape hatch, and `sessionFile` naming need Gateway API product judgment.
