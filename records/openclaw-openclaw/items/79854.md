## Summary

Keep open: current main still returns only the OpenClaw wrapper transcript from `sessions.get`, while `chat.history` runs the CLI-backed import path for Claude CLI sessions. The remaining fix is a Gateway API contract decision, not a safe cleanup close.

## What This Changes

The issue requests clearer and more consistent Gateway transcript reads for CLI-backed sessions, especially around `sessions.get`, `chat.history`, and `sessionFile` semantics.

## Best Possible Solution

Define the public Gateway transcript contract, then either share `chat.history` CLI hydration from `sessions.get` with an explicit wrapper-only escape hatch, or keep the wrapper surface separate and document/add additive metadata for CLI transcript locations.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session with `cliSessionBindings.claude-cli.sessionId` reaches Claude CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run the macOS Claude CLI live scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the reported empty transcript, but default semantics, any `wrapperOnly` option, and `sessionFile` naming are public protocol choices that should be decided before implementation.
