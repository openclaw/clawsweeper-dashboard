## Summary

Keep open: current main still has the reported CLI-backed transcript split, where `sessions.get` reads only the OpenClaw wrapper transcript while `chat.history` hydrates Claude CLI history. The related open PR is useful adjacent work for transcript-only artifact filtering, but it does not resolve the CLI hydration or `sessionFile` contract question.

## What This Changes

The issue asks OpenClaw to clarify or unify Gateway transcript reads and session metadata so CLI-backed sessions do not mislead clients into reading wrapper-only history as the full chat transcript.

## Best Possible Solution

Define the Gateway transcript contract first, then either share CLI hydration between `sessions.get` and `chat.history` with an explicit wrapper-only escape hatch or document `sessions.get` as wrapper transcript and expose additive CLI transcript metadata.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not perform a live macOS `claude-cli` Gateway run.

## Solution Assessment

Unclear: the suggested fixes are plausible, but choosing between default hydration, a wrapper-only option, renamed fields, or documentation is a public Gateway API contract decision. The safest maintainable path is an additive contract clarification before changing default response semantics.
