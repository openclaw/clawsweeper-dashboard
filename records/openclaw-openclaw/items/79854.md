## Summary

Keep open: current `main` still has separate transcript semantics for CLI-backed sessions, with `sessions.get` reading only the OpenClaw wrapper transcript while `chat.history` hydrates Claude CLI history. The related open PR is useful context for transcript artifact filtering, but it does not resolve the CLI hydration or `sessionFile` contract reported here.

## What This Changes

The issue asks OpenClaw to make Gateway transcript APIs and `sessionFile` metadata less misleading for CLI-backed sessions.

## Best Possible Solution

Define the Gateway transcript contract, then either share CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch, or add additive metadata and docs that clearly distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes, by source inspection. A session entry with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only the OpenClaw session transcript path.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may fix the observed empty transcript, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
