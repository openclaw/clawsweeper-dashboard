## Summary

Keep open: current main still reads `sessions.get` from the OpenClaw wrapper transcript while `chat.history` imports Claude CLI history, and the linked open PR is adjacent artifact filtering rather than the CLI hydration or `sessionFile` contract change requested here.

## What This Changes

Request to make Gateway transcript reads less misleading by hydrating CLI-backed `sessions.get` results like `chat.history` or documenting/renaming wrapper transcript fields.

## Best Possible Solution

Define the Gateway transcript contract, then either share `chat.history` CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch, or add additive metadata/docs that distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes, by source inspection. A session with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run the live macOS Claude CLI scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may fix the observed empty transcript, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
