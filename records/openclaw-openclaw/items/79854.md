## Summary

Keep open: current main still has divergent Gateway transcript behavior for CLI-backed sessions, and choosing whether `sessions.get` should hydrate chat history or stay wrapper-scoped is a public API/product decision.

## What This Changes

The issue asks Gateway session reads to stop misleading clients by hydrating `sessions.get` for CLI-backed sessions, exposing clearer transcript metadata, or documenting the wrapper-vs-chat transcript split.

## Best Possible Solution

Define an explicit Gateway transcript contract, then implement the chosen additive API/docs path so clients can reliably request either display chat history or wrapper transcript metadata.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` gateway scenario in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` fixes the observed empty transcript, but the endpoint default, wrapper-only escape hatch, and `sessionFile` metadata shape need maintainer API judgment.
