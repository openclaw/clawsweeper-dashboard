## Summary

Keep open: current main still has two transcript surfaces for CLI-backed sessions, with `chat.history` importing Claude CLI project history and `sessions.get` reading only the OpenClaw session transcript. The related open PR covers transcript-artifact filtering and visible limits, not the `sessions.get` hydration contract or misleading `sessionFile` semantics.

## What This Changes

The issue asks OpenClaw to make `sessions.get` return or clearly distinguish the full CLI-hydrated chat transcript from the wrapper transcript, and to clarify `sessionFile` semantics.

## Best Possible Solution

Define the Gateway transcript contract, then either share `chat.history` CLI hydration through `sessions.get` with additive wrapper-transcript access, or add explicit metadata/docs that distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes, by source inspection. A session with `cliSessionBindings['claude-cli'].sessionId` is hydrated by `chat.history`, while `sessions.get` still reads only `entry.sessionFile`; I did not run the live macOS Claude CLI scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the empty transcript symptom, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
