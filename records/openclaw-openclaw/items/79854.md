## Summary

Keep open: current main still routes `sessions.get` through the OpenClaw `sessionFile` transcript while `chat.history` hydrates Claude CLI JSONL history, and no merged fix currently defines the public contract for wrapper versus chat transcript reads.

## What This Changes

Request to make `sessions.get` and `sessionFile` behavior less misleading for CLI-backed sessions by returning hydrated chat history or exposing/documenting wrapper versus CLI transcript sources.

## Best Possible Solution

Define the public Gateway transcript contract, then either share `chat.history` CLI hydration from `sessions.get` behind an explicit wrapper-only escape hatch or keep wrapper transcripts separate with additive metadata and docs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings['claude-cli'].sessionId` reaches Claude JSONL import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS Claude CLI gateway scenario in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` would address the observed empty transcript, but endpoint semantics, wrapper-only access, and transcript metadata naming are public Gateway API decisions.
