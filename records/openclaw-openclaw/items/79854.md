## Summary

Keep open: current main still has the reported source-level split between `sessions.get` and `chat.history` for CLI-backed sessions, and the remaining fix needs a Gateway protocol decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop misleading Gateway clients about CLI-backed transcripts by hydrating `sessions.get`, exposing clearer transcript metadata, nulling missing wrapper files, or documenting the wrapper-vs-chat distinction.

## Best Possible Solution

Define the Gateway transcript contract first, then either share the existing CLI hydration path with `sessions.get` behind a clear compatibility contract or document a separate wrapper-transcript surface with additive metadata.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings["claude-cli"].sessionId` reaches Claude project-file import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. Proxying `sessions.get` to hydrated chat history would address the empty-conversation symptom, but the endpoint semantics, wrapper-only escape hatch, and `sessionFile` naming are public Gateway protocol decisions.
