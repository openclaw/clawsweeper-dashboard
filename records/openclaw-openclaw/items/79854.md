## Summary

Keep open: current main still has the reported CLI-backed transcript divergence, and changing `sessions.get` defaults or renaming transcript metadata is a public Gateway API decision rather than a safe cleanup close.

## What This Changes

The issue asks OpenClaw to make `sessions.get`, `chat.history`, and `sessionFile` semantics clear and useful for CLI-backed sessions where the real conversation lives in provider-owned JSONL files.

## Best Possible Solution

Define a Gateway transcript contract for CLI-backed sessions, then either share `chat.history` hydration from `sessions.get` with an explicit wrapper-transcript escape hatch or clearly expose/document separate wrapper and provider transcript metadata.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` gateway scenario in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` would address the visible mismatch, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions. An additive contract clarification is the safer first step before changing defaults.
