## Summary

Keep open: current main still has the reported split between the wrapper transcript returned by `sessions.get` and the CLI-hydrated conversation returned by `chat.history`; the related open PR is adjacent artifact filtering and does not settle the public transcript contract or `sessionFile` semantics.

## What This Changes

The issue requests clearer or unified Gateway transcript behavior for CLI-backed sessions, especially around `sessions.get`, `chat.history`, and `sessionFile` metadata.

## Best Possible Solution

Define the Gateway transcript contract, then either share `chat.history` CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch, or add additive metadata and docs that distinguish wrapper and CLI transcript files.

## Reproduction Assessment

Yes, by source inspection. A session entry with a Claude CLI binding is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run the live macOS Claude CLI scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may solve the empty-transcript symptom, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol decisions that should be made additively.
