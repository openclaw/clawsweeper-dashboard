## Summary

Keep open: current main still has the reported `sessions.get` versus `chat.history` split for CLI-backed sessions, and the remaining fix is a Gateway API contract decision rather than conservative cleanup. The discussion correctly points to related open PRs, but those are adjacent or incomplete for this exact `sessions.get`/`sessionFile` behavior.

## What This Changes

The issue asks OpenClaw to stop presenting `sessions.get` and `sessionFile` as chat transcript sources for CLI-backed sessions, either by hydrating `sessions.get`, exposing clearer transcript metadata, or documenting the wrapper-only distinction.

## Best Possible Solution

Define the Gateway transcript contract first, then either make `sessions.get` share the CLI hydration path with an explicit wrapper-transcript escape hatch or document and expose the wrapper/CLI transcript distinction additively.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a session entry with a Claude CLI binding reaches CLI import code through `chat.history`, while `sessions.get` still reads only `entry.sessionFile`. I did not run the original macOS `claude-cli` scenario live in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` would address the empty transcript symptom, but the endpoint semantics, wrapper-only compatibility, and `sessionFile` naming need a maintainer API decision before automation changes defaults.
