## Summary

Keep open: current main still has source-level divergence where CLI-backed `chat.history` can hydrate Claude CLI JSONL turns, while `sessions.get` reads only the OpenClaw session transcript file; the public Gateway contract needs maintainer direction before changing defaults or metadata names.

## What This Changes

The issue asks OpenClaw to align or clearly document `sessions.get`, `chat.history`, and `sessionFile` semantics for CLI-backed sessions.

## Best Possible Solution

Define one public transcript contract for Gateway clients, then either share `chat.history` CLI hydration through `sessions.get` with an explicit wrapper-transcript escape hatch or document a separate wrapper-transcript surface and expose CLI transcript metadata additively.

## Reproduction Assessment

Yes, at source level: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches the Claude CLI import path through `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` Gateway scenario in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` would solve the empty transcript symptom, but the default endpoint semantics, wrapper-only escape hatch, and `sessionFile` metadata shape are public API decisions. A safer maintainable direction is an additive contract clarification before changing defaults.
