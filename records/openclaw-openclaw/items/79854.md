## Summary

Keep open: current main still gives CLI-backed sessions two different transcript surfaces, with `sessions.get` reading only the OpenClaw session transcript while `chat.history` imports bound Claude CLI history. The related open PR is artifact-filtering work, not a merged or safe canonical fix for the CLI hydration or `sessionFile` contract issue.

## What This Changes

The issue asks for `sessions.get` and `sessionFile` semantics to stop misleading gateway consumers when CLI-backed runtimes hydrate chat history elsewhere.

## Best Possible Solution

Define the public Gateway transcript contract, then share CLI hydration with `sessions.get` only through an additive, compatibility-aware shape or document/metadata-label the wrapper-only transcript surface clearly.

## Reproduction Assessment

Yes, by source inspection: `sessions.get` reads only `entry.sessionFile`, while `chat.history` imports bound Claude CLI messages before projection. I did not run the live macOS Claude CLI scenario because this review is read-only.

## Solution Assessment

Unclear; proxying `sessions.get` to `chat.history` may fix the empty transcript symptom, but endpoint semantics, wrapper-only access, and `sessionFile` naming are public protocol choices that should be made additively.
