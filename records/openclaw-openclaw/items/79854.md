## Summary

Keep open: current main still separates wrapper-transcript reads from CLI-hydrated chat history, and the related open PR only addresses transcript-artifact filtering, so this is not implemented or safely duplicate-closed.

## What This Changes

The issue asks OpenClaw to clarify or change Gateway transcript APIs so CLI-backed sessions do not make `sessions.get` look like the full chat transcript while `chat.history` returns the hydrated conversation.

## Best Possible Solution

Define the Gateway transcript contract first, then implement it additively: either make `sessions.get` share CLI hydration with an explicit wrapper-only escape hatch, or document and expose separate wrapper and CLI transcript metadata clearly.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings['claude-cli'].sessionId` reaches Claude CLI JSONL import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. The suggested fixes are plausible, but choosing hydrated defaults, wrapper-only opt-ins, field names, and docs wording is a public Gateway protocol decision rather than a narrow mechanical bug fix.
