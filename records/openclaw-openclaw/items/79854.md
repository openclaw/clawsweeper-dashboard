## Summary

Keep open: current main still has the reported CLI-backed transcript split, and the linked open PRs cover adjacent history filtering/import work rather than the public `sessions.get` contract and `sessionFile` semantics called out here.

## What This Changes

The issue asks for `sessions.get` and `sessionFile` semantics to stop misleading gateway clients when CLI-backed sessions store the real conversation outside the OpenClaw wrapper transcript.

## Best Possible Solution

Define the Gateway transcript contract, then either make `sessions.get` share CLI hydration with an explicit wrapper-only escape hatch or document it as wrapper-only and expose CLI transcript metadata additively.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import code through `chat.history`, while `sessions.get` only reads `entry.sessionFile`; I did not run the live macOS `claude-cli` scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may fix the observed empty portal transcript, but the endpoint semantics, wrapper-only compatibility, and transcript metadata names are public Gateway API decisions.
