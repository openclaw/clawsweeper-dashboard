## Summary

Keep open: current main still has the reported Gateway transcript split, and resolving it requires a maintainer API-contract decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to make CLI-backed session transcript reads less misleading by aligning or documenting `sessions.get`, `chat.history`, and `sessionFile` semantics.

## Best Possible Solution

Define the public transcript contract first, then either make `sessions.get` share the `chat.history` CLI hydration path with an explicit wrapper-transcript escape hatch or document/expose separate wrapper and CLI transcript surfaces additively.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`; the reporter also supplied concrete 2026.5.7 command output.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would solve the empty transcript symptom, but the endpoint semantics, wrapper-only access, and `sessionFile` naming need maintainer API judgment before a safe implementation.
