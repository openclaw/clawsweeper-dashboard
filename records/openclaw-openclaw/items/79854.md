## Summary

Keep open: current main still has the CLI-backed transcript split described here, and the linked open PR only addresses transcript-only artifact filtering rather than the Claude CLI hydration and `sessionFile` semantics reported in this issue.

## What This Changes

Clarify or align Gateway transcript reads so CLI-backed sessions do not make `sessions.get` look like the full conversation while `chat.history` hydrates external CLI turns.

## Best Possible Solution

Define the public Gateway transcript contract, then either share CLI-hydrated history with an explicit wrapper-transcript escape hatch or document and expose wrapper-vs-CLI transcript metadata additively.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a session entry with `cliSessionBindings['claude-cli'].sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may solve the empty-conversation symptom, but default endpoint semantics, a wrapper-only escape hatch, and `sessionFile` naming are public protocol decisions that maintainers should settle first.
