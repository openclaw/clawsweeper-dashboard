## Summary

Keep open: current main still has the reported Gateway transcript asymmetry, and the durable fix needs an API contract decision before changing `sessions.get` defaults or `sessionFile` metadata.

## What This Changes

The issue asks OpenClaw to make CLI-backed session transcript reads and `sessionFile` metadata unambiguous for Gateway clients.

## Best Possible Solution

Define an explicit Gateway transcript contract that separates wrapper transcript metadata from hydrated chat history, then update behavior and docs additively so clients no longer infer semantics from `sessionFile`.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches Claude CLI import through `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run a live macOS `claude-cli` gateway scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` may solve the empty transcript symptom, but the endpoint semantics, wrapper-only escape hatch, and `sessionFile` naming are public Gateway protocol decisions.
