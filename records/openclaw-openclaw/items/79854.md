## Summary

Keep open: current main still has divergent read paths for CLI-backed sessions, and the adjacent open transcript-artifact PR does not address CLI history hydration or `sessionFile` semantics. Changing `sessions.get` defaults or adding wrapper/CLI transcript fields needs a Gateway API contract decision first.

## What This Changes

The issue asks for Gateway transcript APIs to stop misleading consumers when CLI-backed sessions make `sessions.get` return only wrapper transcript rows while `chat.history` returns hydrated conversation history.

## Best Possible Solution

Define the public transcript contract, then either share `chat.history` CLI hydration from `sessions.get` behind an explicit wrapper-only escape hatch or document a separate wrapper-transcript surface with additive CLI transcript metadata.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a session entry with `cliSessionBindings.claude-cli.sessionId` reaches CLI import code through `chat.history`, while `sessions.get` reads only `entry.sessionFile`; I did not run a live macOS `claude-cli` gateway scenario.

## Solution Assessment

Unclear. Proxying `sessions.get` to `chat.history` would address the empty-transcript symptom, but endpoint semantics, a `wrapperOnly` escape hatch, and `sessionFile` naming are public protocol decisions that should be settled before implementation.
