## Summary

Keep open: current main still shows the reported API mismatch, and there is already an open related PR, but it targets transcript artifact filtering rather than the CLI hydration and `sessionFile` contract problem reported here.

## What This Changes

The issue requests that Gateway session transcript APIs and metadata stop misleading clients by either hydrating `sessions.get` like `chat.history`, renaming or nulling wrapper transcript metadata, or documenting the wrapper-only behavior.

## Best Possible Solution

Define the public Gateway transcript contract additively, then either share CLI hydration between `sessions.get` and `chat.history` with an explicit wrapper-transcript escape hatch, or clearly expose/document wrapper and CLI transcript metadata as separate fields.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a session entry with `cliSessionBindings.claude-cli.sessionId` is hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile`. I did not run a live macOS `claude-cli` gateway scenario in this read-only review.

## Solution Assessment

Unclear: proxying `sessions.get` to `chat.history` would address the empty transcript symptom, but endpoint semantics, backwards compatibility for wrapper-only callers, and `sessionFile` naming need maintainer protocol judgment before implementation.
