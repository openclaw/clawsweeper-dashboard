## Summary

Keep open: current main still has divergent transcript read behavior for CLI-backed sessions, and the public Gateway contract for `sessions.get` versus `chat.history` needs maintainer judgment before changing defaults or metadata names.

## What This Changes

The issue requests clearer `sessions.get` and `sessionFile` semantics so CLI-backed sessions expose hydrated chat history or explicitly document/expose wrapper-only transcript metadata.

## Best Possible Solution

Define the Gateway transcript contract, then either share CLI hydration in `sessions.get` with an explicit wrapper/debug path or expose clearly named wrapper and CLI transcript metadata with matching docs and tests.

## Reproduction Assessment

Yes. A current-main source reproduction is clear: a session entry with `cliSessionBindings.claude-cli.sessionId` can be hydrated by `chat.history`, while `sessions.get` reads only `entry.sessionFile` and returns the wrapper transcript rows.

## Solution Assessment

Unclear. Proxying `sessions.get` to the `chat.history` hydration path may solve the observed empty transcript, but default payload semantics, a wrapper-only escape hatch, and `sessionFile` naming are public Gateway API choices.
