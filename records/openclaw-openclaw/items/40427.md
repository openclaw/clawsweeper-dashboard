## Summary

Keep open. The PR targets a real config warning log-spam problem, but current main has only solved the main gateway/public-load hot paths while the proposed patch still has merge-blocking logging-state and output-format regressions.

## What This Changes

The PR adds a per-config-path warning fingerprint cache in `createConfigIO().loadConfig()` plus a focused unit test for repeated warning loads and re-logging after config changes.

## Best Possible Solution

Adopt one canonical config diagnostics policy that dedupes unchanged warning/error payloads, clears state across all config transitions, preserves multiline output, and covers public load plus gateway reload behavior with regression tests.

## Reproduction Assessment

Yes. A warning-producing config followed by repeated `createConfigIO({ logger }).loadConfig()` calls reproduces the direct duplicate warning path; the PR-introduced stale-state bug follows from warning config A, then a missing or invalid config, then warning config A again.

## Solution Assessment

No. The proposed direction is useful, but the patch is not mergeable as written because it leaves stale fingerprints on non-warning exits, does not preserve current multiline output, and needs to be reconciled with the broader maintainer config diagnostics branch.
