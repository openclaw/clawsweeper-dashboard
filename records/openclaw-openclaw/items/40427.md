## Summary

Keep open for maintainer handling. Current main still has a direct non-deduped `Config warnings:` path, but this PR is not mergeable as written because it can suppress re-emerged warnings, would regress multiline warning output on rebase, and overlaps the broader maintainer config diagnostics work in #38295. Review target: https://github.com/openclaw/openclaw/pull/40427

## What This Changes

The PR adds a per-config-path fingerprint cache around `Config warnings:` logging in `createConfigIO().loadConfig()` and adds a focused unit test for repeated warning loads.

## Best Possible Solution

Adopt one canonical config diagnostics policy that dedupes unchanged warning and error payloads, clears state across config transitions, preserves multiline output, and covers direct load plus gateway reload behavior with regression tests.

## Reproduction Assessment

Yes. A warning-producing config followed by repeated `createConfigIO({ logger }).loadConfig()` calls reproduces the duplicate warning path; warning config A, then a missing or invalid config, then warning config A again reproduces the PR-introduced stale fingerprint case.

## Solution Assessment

No. The direction is useful, but this patch is not the best merge candidate until it clears fingerprints on every non-warning exit, preserves current multiline output, adds the required changelog entry, and is reconciled with #38295.
