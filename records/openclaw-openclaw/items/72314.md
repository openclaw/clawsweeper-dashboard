## Summary

Keep this PR open for normal maintainer review and merge handling. Current main still lacks an idless inbound dedupe backstop, while the latest PR head appears to solve that narrow gap and addresses the earlier review concerns about provider-ID key churn and a dead guard.

## What This Changes

The PR adds a SHA-256 content/timestamp fallback for inbound auto-reply dedupe when `MessageSid` is absent, preserves provider-message-id keys, extends inbound dedupe tests, and adds a changelog entry.

## Best Possible Solution

Land this PR after maintainer review and green branch checks; keep the separate Control UI duplicate-message report in #72892 tracked independently because that report describes different sender identities and message IDs, not idless retries.

## Reproduction Assessment

Yes. Source inspection on current main shows an idless `MsgContext` returns no inbound dedupe key, and dispatch proceeds for invalid claims; the PR’s focused tests exercise the fixed retry path and its collision boundaries.

## Solution Assessment

Yes. The latest patch is a narrow maintainable fix: it changes only inbound dedupe key construction, preserves provider-ID suffixes and the existing three-part route-key shape, and requires both text and timestamp for the fallback.
