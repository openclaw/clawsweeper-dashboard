## Summary

Keep open: the PR addresses a real idless inbound dedupe gap that still exists on current main, but the current patch can still miss text retries when an earlier body field is blank, and the live PR is currently conflicting.

## What This Changes

The PR adds a route-scoped SHA-256 content/timestamp fallback dedupe key for inbound messages without provider message IDs, plus focused tests and a changelog entry.

## Best Possible Solution

Repair the fallback body resolver to normalize candidate text fields independently, include BodyForCommands, preserve the attachment-only boundary, then rebase and rerun the focused auto-reply proof.

## Reproduction Assessment

Yes at source level: current main requires MessageSid for inbound dedupe keys, so an idless retry returns invalid and can re-enter dispatch. I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not as written. The route-scoped content/timestamp fingerprint is the right narrow shape, but the current implementation can ignore real text when an earlier body field is blank.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Normalize each candidate body before fingerprinting:** `src/auto-reply/reply/inbound-dedupe.ts:42`
  - body: The fallback picks `ctx.CommandBody ?? ctx.RawBody ?? ctx.Body` before trimming. A blank `CommandBody` is still defined, so an idless retry with real text in `RawBody`, `Body`, or the finalized `BodyForCommands` returns null and bypasses the new dedupe path. Normalize each candidate independently and include `BodyForCommands` so empty fields do not mask the text body.
  - confidence: 0.88
