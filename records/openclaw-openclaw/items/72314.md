## Summary

Keep this PR open: current main still has the idless inbound dedupe gap, and the PR is a focused fix, but the new content fallback should be sender-scoped before merge to avoid collapsing distinct group/channel messages.

## What This Changes

The PR adds a route-scoped timestamp/text content fingerprint fallback to inbound auto-reply dedupe for messages without provider IDs, plus regression tests and a changelog entry.

## Best Possible Solution

Keep the PR open and sender-scope the idless content fallback while preserving raw provider-ID keys and the timestamp/text/attachment boundaries.

## Reproduction Assessment

Yes, by source inspection. Current main returns no dedupe key without MessageSid, and the PR finding is reproducible from the key construction by comparing two idless same-route/same-body/same-timestamp contexts with different senders.

## Solution Assessment

No, not as written. The narrow maintainable fix is the PR's fallback approach plus sender identity in the idless content key so provider-ID behavior remains unchanged and distinct group messages do not collapse.
