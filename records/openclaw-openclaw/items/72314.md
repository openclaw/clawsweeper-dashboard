## Summary

Keep open. Current main still does not dedupe inbound auto-reply retries when the provider message id is missing, while this PR adds a narrow content/timestamp fallback with focused tests and sufficient module-level real behavior proof. I did not find a functional or security blocker in the diff.

## What This Changes

The PR adds a route-scoped SHA-256 content/timestamp fallback dedupe key for inbound auto-reply messages without `MessageSid`, plus regression tests and a changelog entry.

## Best Possible Solution

Land a narrow inbound-dedupe fix that preserves provider-id keys, adds an idless fallback only for timestamped text retries, and keeps attachment-only or timestampless messages out of content-only collapse.

## Reproduction Assessment

Yes, source-level. Current main returns no dedupe key when `MessageSid` is missing, and dispatch only suppresses duplicate or inflight claims, so an idless retry proceeds without dedupe.

## Solution Assessment

Yes. The PR keeps the established provider-id path unchanged and scopes the new fallback by the existing session/route key plus timestamped text, which is the narrow maintainable fix for idless retries.
