## Summary

Keep open: this is a focused and useful bug-fix PR, but the current patch needs one bounded repair before merge because the new idless fallback can hash media placeholder bodies and suppress distinct attachment-only turns that current main would accept.

## What This Changes

The PR adds a route-scoped SHA-256 content/timestamp fallback key for idless auto-reply inbound retries, plus regression tests and a changelog entry, while preserving raw provider message-id keys.

## Best Possible Solution

Land the fallback only after the content fingerprint distinguishes real text retries from media/attachment-only placeholders, while keeping raw provider-message-id keys unchanged.

## Reproduction Assessment

Yes, source-level. Current main returns no inbound dedupe key when `MessageSid` is missing, and dispatch only skips `duplicate` or `inflight` claims, so an idless retry can proceed as a new turn.

## Solution Assessment

No, not in the current form. The provider-id compatibility fix is good, but the content fallback should exclude media-placeholder-only contexts or include a stable media identity before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Exclude media placeholders from the fallback:** `src/auto-reply/reply/inbound-dedupe.ts:40`
  - body: The fallback hashes `CommandBody`/`RawBody`/`Body` without checking whether the context is actually media-only. Existing media paths can carry placeholder bodies like `<media:audio>` while also setting `MediaPath`, so an idless attachment retry or a distinct attachment with the same route/timestamp/placeholder would share a `content:` key and the second turn would be skipped. Gate this fallback on real text content or include a stable media identity before deduping media turns.
  - confidence: 0.86
