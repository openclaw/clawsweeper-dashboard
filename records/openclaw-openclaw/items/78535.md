## Summary

Current main and the latest release already treat overloaded/capacity provider failures as fallback-eligible and route them through the configured model fallback chain before surfacing an error.

## What This Changes

The issue asks for temporary provider overload/capacity errors to trigger configured fallback models instead of failing the Telegram main-session request immediately.

## Best Possible Solution

Keep the shipped fallback behavior; if overload still surfaces on v2026.5.5 or newer, collect the model source and fallback-decision logs in a fresh report.

## Reproduction Assessment

No failing current-main reproduction was established; source and regression coverage show the reported v2026.5.4 behavior is now handled by the fallback path. The remaining reproducer would be an older-release comparison or a live provider overload, which was not needed for the implemented-on-main decision.

## Solution Assessment

Yes. Classifying overloaded provider errors as failover errors and letting the existing model fallback runner walk configured candidates is the narrow maintainable fix, while explicit user model selections remain strict by documented policy.
