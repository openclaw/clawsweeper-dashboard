## Label Justifications

## Summary

Keep open: current main still reads memory-core short-term recall through the capped private file store while writes remain uncapped, and workspace failures are still logged inside dreaming without making the cron run fail.

## What This Changes

Fix memory-core dreaming so oversized short-term recall stores do not silently stop promotion after upgrade.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read and write size behavior, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, from source inspection: an oversized `short-term-recall.json` read still goes through `privateFileStore` without a `maxBytes` override, and fs-safe defaults reads to 16 MiB. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before adding any optional user-facing config.
