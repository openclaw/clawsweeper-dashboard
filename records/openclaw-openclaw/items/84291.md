## Label Justifications

- `P1`: A recent regression can repeatedly block nightly dreaming promotion for affected upgraded workspaces while normal cron and doctor surfaces still look healthy.
- `impact:session-state`: The affected data is memory-core short-term recall state used to promote workspace memory into durable dreaming artifacts.

## Summary

Keep open: current main still has the reported memory-core read/write size mismatch, and the shipped v2026.5.19 release still contains the same affected recall-store path. The source-level bug is clear, but the durable fix needs a memory-core owner decision on retention, migration, size limits, and how cron/doctor should surface partial workspace failures.

## What This Changes

The issue asks OpenClaw to stop silently failing dreaming promotion when an upgraded workspace has a `memory/.dreams/short-term-recall.json` larger than 16 MiB.

## Best Possible Solution

Define a memory-core-owned size and retention policy that preserves readable upgraded recall stores, prevents future unreadable growth, and records failed dreaming workspace outcomes for cron and doctor status.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an oversized `memory/.dreams/short-term-recall.json` is read through `privateFileStore` without a max override, fs-safe applies a 16 MiB default read cap, and the write path has no matching cap.

## Solution Assessment

No patch is proposed here. A one-line cap bump would be incomplete; the maintainable fix should align read/write policy, upgraded-store handling, and visible failure reporting before adding any optional user config.
