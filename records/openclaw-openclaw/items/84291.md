## Label Justifications

- `P1`: This is a reported regression that can repeatedly block nightly memory dreaming for affected upgraded workspaces while the normal cron status remains healthy.
- `impact:session-state`: The failure prevents short-term recall state from being promoted into durable memory and leaves memory/dreaming state stale.

## Summary

Keep open: current main still routes the short-term dreaming store through an uncapped memory-core write path and a default-capped fs-safe read path, so a valid recall store over 16 MiB can still block dreaming promotion while the cron handler reports the sweep as handled.

## What This Changes

The issue reports a v2026.5.18 regression where memory-core dreaming cannot read an oversized `short-term-recall.json`, logs the per-workspace failure, and leaves cron or doctor status looking healthy.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read/write size behavior, and surfaces failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a valid `memory/.dreams/short-term-recall.json` over 16 MiB, then run dreaming so `readStore` hits `privateFileStore` without a `maxBytes` override and fs-safe throws before ranking or promotion.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before adding any optional user-facing config.
