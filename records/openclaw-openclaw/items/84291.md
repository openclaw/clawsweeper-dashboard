## Summary

Keep open: rechecking current main confirms the same memory-core read/write size-policy mismatch and the same silent per-workspace dreaming failure path described by the report and prior ClawSweeper review.

## What This Changes

The issue asks OpenClaw to stop memory-core dreaming from silently failing when an upgraded workspace has a short-term recall store larger than 16 MiB.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read and write size limits, and reports failed dreaming workspaces through cron/doctor diagnostics with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows an oversized `short-term-recall.json` read follows the uncapped memory-core `privateFileStore` path into fs-safe's 16 MiB default read limit, and the dreaming sweep catches that failure per workspace.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core read/write size policy and upgrade handling before adding optional user-facing config.
