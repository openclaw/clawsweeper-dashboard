## Summary

Keep open: current main still has the memory-core read/write size-policy mismatch that can make an oversized short-term recall store unreadable during dreaming, and the workspace failure is logged inside the dreaming sweep rather than surfaced as a failed cron or doctor lint finding.

## What This Changes

The issue reports a v2026.5.18 regression where oversized `memory/.dreams/short-term-recall.json` blocks memory-core dreaming promotion while `cron list` and `doctor --lint` still look healthy.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read and write size behavior, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, from source inspection: an oversized `short-term-recall.json` read goes through `privateFileStore(...).readJsonIfExists(...)` without a cap override, and `@openclaw/fs-safe` 0.2.4 defaults root reads to 16 MiB. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The best fix should align memory-core read/write size policy and upgrade handling before adding any optional user-facing config.
