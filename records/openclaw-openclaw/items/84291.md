## Label Justifications

- `P1`: This is a shipped regression that can disable nightly dreaming promotion for affected real workspaces while reporting success in normal status surfaces.
- `impact:session-state`: The affected store is memory/dreaming state, and the failure prevents session-memory consolidation for oversized workspaces.

## Summary

Keep open: current main still has the reported memory-core read/write size mismatch, and dreaming logs per-workspace failures while returning handled success, so cron status can remain green for an affected oversized recall store.

## What This Changes

Fix the v2026.5.18 regression where dreaming promotion silently fails for workspaces whose `memory/.dreams/short-term-recall.json` exceeds 16 MiB.

## Best Possible Solution

Define a memory-core upgrade and retention policy that keeps existing recall stores readable, prevents recurrence by aligning read/write limits or pruning, and surfaces per-workspace dreaming failures through cron or doctor diagnostics.

## Reproduction Assessment

Yes, from source inspection: an oversized `short-term-recall.json` read goes through `privateFileStore` without a `maxBytes` override, and fs-safe 0.2.4 defaults reads to 16 MiB. I did not run the live packaged cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before adding optional user-facing config.
