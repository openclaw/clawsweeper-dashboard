## Label Justifications

- `P1`: The issue is a recent regression that can silently disable nightly memory consolidation for affected production workspaces after upgrade.
- `impact:session-state`: The affected state is persisted memory recall and dreaming promotion state failing to advance while user-visible diagnostics stay healthy.

## Summary

Keep open: current `main` and the latest stable release still have the source-backed read/write size mismatch for memory-core short-term recall, and workspace-level dreaming failures are logged without turning the cron result or doctor lint path into an actionable failure.

## What This Changes

The issue asks OpenClaw to fix a v2026.5.18 memory-core dreaming regression where oversized `short-term-recall.json` files fail reads, block promotion, and are not surfaced by cron or doctor diagnostics.

## Best Possible Solution

Define a memory-core-owned upgrade/read-write policy that keeps existing recall stores readable, prevents future unreadable growth, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, at source level: an oversized `short-term-recall.json` is read through `privateFileStore` without an override, and `@openclaw/fs-safe` 0.2.4 defaults root reads to 16 MiB. I did not run the live managed cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The best fix should first align memory-core’s read/write size and upgrade policy, then add cron or doctor visibility; exposing a user config knob alone would not address existing unreadable stores.
