## Label Justifications

- `P1`: The issue reports a released regression that can stop nightly memory dreaming for affected workspaces while cron and doctor surfaces still look healthy.
- `impact:session-state`: The affected file is memory-core short-term recall state used to promote session and memory context into durable dreaming artifacts.

## Summary

Keep open: current main still has the reported read/write size mismatch for the memory-core short-term recall store, and workspace-level dreaming failures are logged while the cron handler returns success.

## What This Changes

The issue reports that memory-core dreaming should not silently stop promotion when an upgraded workspace has a short-term recall JSON file larger than the fs-safe 16 MiB read default.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, bounds future growth, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a `memory/.dreams/short-term-recall.json` larger than 16 MiB, then run the current dreaming promotion path that reads it through `privateFileStore(...).readJsonIfExists(...)` without a max override. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The best fix should align memory-core read/write size policy, handle existing oversized stores safely, and decide whether cron or doctor should become the visible failure surface before adding optional user config.
