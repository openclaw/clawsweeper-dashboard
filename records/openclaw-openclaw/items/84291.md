## Summary

Keep open: current main and the v2026.5.18 release still show the reported memory-core read/write size mismatch, and the failed workspace path is logged inside dreaming rather than turned into a failed cron or doctor lint result.

## What This Changes

The issue reports a v2026.5.18 regression where memory-core dreaming cannot read oversized `memory/.dreams/short-term-recall.json` files and does not surface the per-workspace failure in cron status or doctor lint.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read/write size behavior, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, from source inspection: an oversized `short-term-recall.json` read goes through `privateFileStore` without a max override, and fs-safe 0.2.4 defaults reads to 16 MiB. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before adding any optional user-facing config.
