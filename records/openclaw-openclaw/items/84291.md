## Label Justifications

- `P1`: The issue describes a release regression that blocks an existing nightly memory/dreaming workflow for affected users while cron still appears healthy.
- `impact:session-state`: The failure prevents short-term recall state from being promoted into durable dreaming and memory artifacts.

## Summary

Keep open: current main still has the source-level failure path described in the report, and the issue is not a duplicate of the related dreaming reports because this one is specifically the private file-store read cap versus an oversized recall JSON.

## What This Changes

The issue reports that dreaming promotion silently fails after upgrade when `memory/.dreams/short-term-recall.json` exceeds the 16 MiB private file-store read limit.

## Best Possible Solution

Adopt an owner-approved memory-core policy that keeps the recall store readable across upgrades, prevents future write/read cap drift, preserves or backs up user memory before trimming, and surfaces per-workspace dreaming failures in the user-facing diagnostics path.

## Reproduction Assessment

Yes, at source level: a private `short-term-recall.json` larger than 16 MiB will hit fs-safe's default read cap through `privateFileStore(...).readJsonIfExists(...)`, while the current write path does not prevent that file from being produced. I did not run a live cron reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice: the narrow bug is real, but the best fix needs maintainer direction on whether to trim with backup, increase or override the cap for this store, and how strongly to report partial workspace failure. A workaround that only asks users to delete the file is not the best durable solution.
