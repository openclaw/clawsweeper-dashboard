## Label Justifications

- `P1`: This is a reported regression that blocks nightly dreaming promotion for affected upgraded workspaces while the cron can still look healthy.
- `impact:session-state`: The failure is in the memory-core short-term recall store used to preserve and promote session/memory state.

## Summary

Keep open: current main and v2026.5.19 still have the reported memory-core read/write size mismatch, and the remaining fix needs an owner decision about recall-store retention, upgrade migration, and diagnostic surfacing.

## What This Changes

The issue reports that memory-core dreaming fails silently after upgrade when `memory/.dreams/short-term-recall.json` is larger than the fs-safe read limit.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, prevents future unreadable growth, and reports failed dreaming workspaces through cron or doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows an oversized `short-term-recall.json` read goes through fs-safe's default 16 MiB read cap while memory-core writes the same store without a matching cap; I did not run the live cron path.

## Solution Assessment

No patch is proposed here. The best solution should align memory-core read/write size policy and upgrade handling before adding an optional user-facing config knob.
