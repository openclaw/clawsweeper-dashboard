- `impact:session-state`: The failure blocks promotion from short-term dreaming recall state into durable memory and dream reports.

## Summary

Keep open: current main still has the reported memory-core read/write limit mismatch, and the per-workspace failure is caught as a logged dreaming failure rather than surfaced as a failed cron run or doctor lint finding.

## What This Changes

The issue reports a v2026.5.18 regression where memory-core dreaming promotion silently fails when `memory/.dreams/short-term-recall.json` exceeds 16 MiB.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read and write limits, and surfaces failed dreaming workspaces in status/doctor/cron diagnostics with regression coverage.

## Reproduction Assessment

Yes from source inspection: an oversized `short-term-recall.json` read goes through `privateFileStore` without a max override, and fs-safe 0.2.4 defaults reads to 16 MiB. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before adding any optional user-facing config.
