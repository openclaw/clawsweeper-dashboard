## Label Justifications

- `P1`: This is a latest-release regression that can silently stop nightly memory-core dreaming promotion for affected real workspaces.
- `impact:session-state`: The failure blocks promotion from short-term recall state into durable memory and dream reports.

## Summary

Keep open: current main still has the reported memory-core read/write size-policy mismatch, and dreaming still converts per-workspace promotion failures into a logged completion rather than cron or doctor-visible failure state.

## What This Changes

The issue requests fixing a v2026.5.18 regression where oversized memory-core short-term recall stores make dreaming promotion fail silently and stay invisible to cron/doctor status.

## Best Possible Solution

Adopt a memory-core-owned policy that keeps existing large recall stores readable, prevents future unreadable growth, preserves or backs up user recall data during upgrade, and reports per-workspace dreaming failures through cron and doctor status.

## Reproduction Assessment

Yes. Source inspection shows an oversized `short-term-recall.json` read goes through `privateFileStore` without a max override, and fs-safe 0.2.4 defaults reads to 16 MiB; I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core read/write size policy, upgrade handling, and diagnostics before adding any optional user-facing config.
