## Summary

Keep open: current main and v2026.5.18 still have the reported memory-core read/write size-policy mismatch, and the per-workspace dreaming failure is logged without becoming a failed cron or doctor lint result.

## What This Changes

The issue reports a v2026.5.18 regression where memory-core dreaming cannot read oversized `short-term-recall.json` files and hides the per-workspace failure from cron/doctor status.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read and write size limits, and surfaces failed dreaming workspaces in cron/doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes, from source inspection: an oversized `short-term-recall.json` read goes through `privateFileStore` without a max override, and fs-safe 0.2.4 defaults reads to 16 MiB. I did not run the live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should align memory-core’s read/write size policy and upgrade handling before considering any optional user-facing config.
