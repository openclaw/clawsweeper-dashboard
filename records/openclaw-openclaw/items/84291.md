## Summary

Keep open: current main and v2026.5.18 still have the reported oversized short-term recall read path, and the related session-corpus cleanup work does not cover already oversized stores, read/write cap alignment, or cron/doctor failure surfacing.

## What This Changes

The issue reports that memory-core dreaming silently fails after upgrade when `memory/.dreams/short-term-recall.json` is larger than 16 MiB.

## Best Possible Solution

Choose a memory-core-owned upgrade policy that keeps existing recall stores readable, aligns read/write size behavior or retention, and reports failed dreaming workspaces through cron and doctor diagnostics with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows an oversized `short-term-recall.json` read still goes through `privateFileStore` without a max override, and fs-safe 0.2.4 still applies the 16 MiB default read cap.

## Solution Assessment

No patch is proposed here. The manual archive workaround unblocks one workspace, but the maintainable fix should define memory-core retention/upgrade behavior and status reporting rather than only raising a byte limit.
