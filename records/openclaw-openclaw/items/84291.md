## Summary

Keep open: current main and v2026.5.20 still have the reported read/write size mismatch and swallowed per-workspace dreaming failure, and no linked PR currently closes this exact oversized recall-store bug.

## What This Changes

The issue reports a v2026.5.18 regression where oversized `memory/.dreams/short-term-recall.json` files make memory-core dreaming fail without visible cron or doctor status.

## Best Possible Solution

Memory-core should define an owner-approved recall-store size and upgrade policy that keeps existing oversized stores readable or safely migrates them, aligns future writes with read limits, and surfaces per-workspace dreaming failures through cron or doctor diagnostics.

## Reproduction Assessment

Yes. Source inspection shows an oversized `short-term-recall.json` goes through `privateFileStore(...).readJsonIfExists(...)` with fs-safe’s default 16 MiB read cap, while the matching write path has no cap; I did not run a live cron path in this read-only review.

## Solution Assessment

No patch is proposed here. The narrowest maintainable fix should first settle memory-core’s retention, migration, and diagnostic policy, then add focused regression coverage for oversized stores and cron/doctor visibility.
