## Summary

Keep open: this recent external PR attempts to address the linked Discord voice report, but the diff introduces two blocking defects and still lacks real Discord behavior proof. Current main still documents separate Discord `/voice` TTS, `/talkvoice` Talk voice, and `/vc` voice-channel control surfaces, so this is not obsolete cleanup.

## What This Changes

The PR adds Discord `/vc list` and `/vc chat [mode]` subcommands in `extensions/discord/src/voice/command.ts`.

## Best Possible Solution

Restore the documented Discord split first: `/voice` should reach TTS, `/talkvoice` should manage Talk voice, and any new `/vc` additions should preserve authorization and either persist changes correctly or stay read-only.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR's `/vc chat mode` branch only replies while current `DiscordVoiceManager.join()` reads configured `discordConfig.voice`, and `/vc list` reads manager status without the authorization gate used by `status` and `leave`. I did not run a live Discord setup.

## Solution Assessment

No. The PR adds new `/vc` surfaces instead of the narrow linked `/voice` routing fix, and the added list/chat behavior needs authorization, persistence, and UX decisions before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist the requested voice mode before confirming it:** `extensions/discord/src/voice/command.ts:349-351`
  - body: The new `/vc chat mode` path validates the option and tells the user the mode was set, but it never writes config, updates the manager, or changes the active session. `DiscordVoiceManager.join()` still reads the existing configured mode, so the confirmation is false after rejoin.
