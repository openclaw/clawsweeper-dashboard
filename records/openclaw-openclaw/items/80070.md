## Summary

Keep open: the PR is not mergeable as written because it does not fix the linked Discord `/voice` routing bug and introduces a typecheck break, an access-control gap, a false mode-setting confirmation, and no real Discord behavior proof.

## What This Changes

The branch modifies `extensions/discord/src/voice/command.ts` to add `/vc list` and `/vc chat [mode]` subcommands.

## Best Possible Solution

Fix the Discord native command split so `/voice` reaches TTS and `/talkvoice` manages Talk voice; any later `/vc` additions should preserve authorization and only confirm changes that are actually persisted or applied.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defects: the new chat command calls `getString` with an unsupported arity, list returns session data without the existing access gate, and the mode setter only replies while join still reads `discordConfig.voice`.

## Solution Assessment

No. This is not the narrow fix for the linked Discord `/voice` routing bug; it expands `/vc` and needs build, authorization, persistence, and UX decisions before it can be considered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Call getString with the supported signature:** `extensions/discord/src/voice/command.ts:339`
  - body: `CommandInteraction.options` is an `OptionsHandler`, and `getString` only accepts the option name. Passing the extra `false` argument here breaks TypeScript checking before the new command can ship.
  - confidence: 0.95
- **[P2] Authorize list before exposing voice sessions:** `extensions/discord/src/voice/command.ts:285-287`
  - body: `/vc list` reads `manager.status()` and replies with active channel mentions without the `ensureVoiceCommandAccess` gate used by `status` and `leave`. In allowlisted guilds this can disclose active voice channel IDs to unauthorized users.
  - confidence: 0.9
- **[P2] Persist the requested voice mode before confirming it:** `extensions/discord/src/voice/command.ts:349-351`
  - body: The new `/vc chat mode` path validates the mode and says it was set, but it never updates config, manager state, or the active session. `DiscordVoiceManager.join()` will still use the existing configured mode, so the confirmation is false.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR exposes active Discord voice session channel IDs without the existing voice-command authorization gate.

Concerns:

- **[medium] Voice session listing bypasses access checks:** `extensions/discord/src/voice/command.ts:285`
  - body: The added list command returns guild voice session channel mentions from `manager.status()` without calling `ensureVoiceCommandAccess`, unlike the existing status and leave paths. That can leak active voice channel IDs to users not authorized for voice commands in that guild or channel.
  - confidence: 0.9
