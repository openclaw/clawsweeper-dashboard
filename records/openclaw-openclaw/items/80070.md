## Summary

Keep open: the patch should not merge as-is because source review confirms an access-control regression and a false mode-setting confirmation, and the external PR still lacks real Discord behavior proof.

## What This Changes

The PR adds Discord `/vc list` and `/vc chat [mode]` subcommands in `extensions/discord/src/voice/command.ts`.

## Best Possible Solution

Restore the documented Discord `/voice` and `/talkvoice` split, and only extend `/vc` if new subcommands preserve authorization and actually persist or apply mode changes.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR's list branch reads and replies with session status before any access gate, and the chat mode branch only replies while `join()` still reads the configured voice mode.

## Solution Assessment

No. This is not the narrow fix for the linked Discord `/voice` routing bug, and the added `/vc` behavior needs authorization, persistence, and UX decisions before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Authorize list before exposing voice sessions:** `extensions/discord/src/voice/command.ts:285-287`
  - body: `/vc list` returns active channel mentions from `manager.status()` without the `ensureVoiceCommandAccess` gate used by `status` and `leave`. In allowlisted guilds, an unauthorized user could learn active voice session channel IDs.
  - confidence: 0.9
- **[P2] Persist the requested voice mode before confirming it:** `extensions/discord/src/voice/command.ts:349-351`
  - body: The new `/vc chat mode` path validates the option and tells the user the mode was set, but it never writes config, updates the manager, or changes the active session. `DiscordVoiceManager.join()` will still use the existing configured mode, so the confirmation is false after rejoin.
  - confidence: 0.9
