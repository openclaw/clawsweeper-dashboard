## Summary

Keep open: this recent external PR lacks after-fix Discord proof and the diff has blocking correctness plus access-control problems before it can be considered for merge.

## What This Changes

The PR modifies `extensions/discord/src/voice/command.ts` to add `/vc list` and `/vc chat [mode]` subcommands for listing active sessions and showing or claiming to set Discord voice mode.

## Best Possible Solution

Restore the documented Discord split first: `/voice` should reach TTS, `/talkvoice` should manage Talk voice, and any new `/vc` additions must preserve authorization and either persist changes correctly or stay read-only.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR's mode branch only replies while current join logic reads the configured voice mode, and the new list branch returns manager status without the existing access gate.

## Solution Assessment

No. This is not the narrow fix for the linked Discord `/voice` routing bug, and the added `/vc` behavior needs authorization, persistence, and UX decisions before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist the requested voice mode before confirming it:** `extensions/discord/src/voice/command.ts:349-351`
  - body: The new `/vc chat mode` path validates the option and tells the user the mode was set, but it never writes config, updates the manager, or changes the active session. `DiscordVoiceManager.join()` will still use the existing configured mode, so the confirmation is false after rejoin.
  - confidence: 0.9
- **[P2] Authorize list before exposing voice sessions:** `extensions/discord/src/voice/command.ts:285-287`
  - body: `/vc list` returns active channel mentions from `manager.status()` without the `ensureVoiceCommandAccess` gate used by `status` and `leave`. In allowlisted guilds, an unauthorized user could learn active voice session channel IDs.
  - confidence: 0.88

## Security Review

Status: needs_attention
