## Summary

Keep open: the current PR head still fails type checking, exposes Discord voice session details before authorization, confirms `/vc chat` mode changes without applying or persisting them, and lacks real Discord after-fix proof.

## What This Changes

The PR adds Discord `/vc list` and `/vc chat [mode]` subcommands in `extensions/discord/src/voice/command.ts`.

## Best Possible Solution

Restore the documented Discord `/voice` and `/talkvoice` routing split first; only extend `/vc` if the subcommands are authorized, persisted, documented, and proven in a real Discord setup.

## Reproduction Assessment

Yes. CI reproduces the typecheck failure, and source inspection shows `/vc list` reads sessions before authorization while `/vc chat` only replies instead of updating config or the active manager.

## Solution Assessment

No. This is not the narrowest fix for the linked Discord `/voice` routing bug, and the added `/vc` behavior needs authorization, persistence, documentation, and UX approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add override modifiers to new command members:** `extensions/discord/src/voice/command.ts:277-307`
  - body: The branch fails `check-prod-types` with TS4114 because the new `ListCommand` and `ChatCommand` members override `Command` properties without the required `override` modifier. Until these are fixed, the package will not typecheck.
  - confidence: 0.95
- **[P2] Authorize list before exposing voice sessions:** `extensions/discord/src/voice/command.ts:286-288`
  - body: `/vc list` returns active channel mentions from `manager.status()` without the `ensureVoiceCommandAccess` gate used by `status` and `leave`. In allowlisted guilds, an unauthorized user could learn active voice session channel IDs.
  - confidence: 0.9
- **[P2] Persist the requested voice mode before confirming it:** `extensions/discord/src/voice/command.ts:350-351`
  - body: This branch validates `mode` and says it was set, but it never writes config, updates the manager, or changes the active session. `DiscordVoiceManager.join()` still reads `params.discordConfig.voice`, so the confirmation is false after rejoin.
  - confidence: 0.9
