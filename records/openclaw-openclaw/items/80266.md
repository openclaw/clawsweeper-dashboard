## Summary

Keep this PR open for normal review: it adds a new install/update setup-script execution surface, but source inspection still shows env-policy and archive staging defects, missing real install/update proof, and unresolved product/security choices around the canonical setup contract.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and runs skill-supplied setup scripts after skill install/update paths, with tests, a changelog entry, and a restricted `runCommandWithTimeout` environment option.

## Best Possible Solution

Settle one canonical setup contract, reuse the established host/skill env policy, make `SKILL_DIR` and rollback semantics stable, document the trust model, and require redacted real install/update proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the setup env policy bypass and archive staging-path semantics on the PR head. For the linked setup-hook request itself, reproduction is not applicable because it is a new feature surface.

## Solution Assessment

No. The feature may be useful, but this patch is not the best mergeable shape until the env policy, path/rollback semantics, docs/security contract, unrelated generated drift, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Filter setup env through the existing skill env policy:** `src/agents/skills-setup.ts:106-115`
  - body: `runSkillSetupHook` forwards any declared `requires.env` key from `process.env`. Existing skill env handling blocks execution-affecting names such as `NODE_OPTIONS`, `BASH_ENV`, and `OPENSSL_CONF`, so setup hooks should reuse or match that sanitizer before running third-party scripts.
  - confidence: 0.94
- **[P2] Keep SKILL_DIR stable for archive hooks:** `src/agents/skills-archive-install.ts:184-187`
  - body: Archive installs run the setup hook from `installPackageDir.afterInstall`, which receives the temporary `.openclaw-install-stage-*` directory before publish. Scripts that persist or report `SKILL_DIR` get a path that is removed after success instead of the installed skill directory promised by the feature contract.
  - confidence: 0.9
- **[P2] Remove the unrelated Canvas bundle hash drift:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes only the Canvas A2UI `.bundle.hash` without changing Canvas inputs or output. The bundle script derives that file from Canvas package/app sources, so this unrelated generated drift can be rewritten by build/package validation.
  - confidence: 0.84
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:38`
  - body: The PR adds the public `metadata.openclaw.setup` field but does not document timing, env forwarding, timeout ownership, failure behavior, rollback semantics, or the install-time code execution trust model.
  - confidence: 0.8
