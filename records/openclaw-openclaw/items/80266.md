## Summary

Keep this PR open, but it is not merge-ready: the direction matches a real feature request, yet the patch still has env-policy, configured-secret, staging-path, docs, unrelated-file, and real-proof blockers for a new install-time code execution surface.

## What This Changes

This PR adds `metadata.openclaw.setup.script` parsing and runs skill-supplied setup scripts after metadata installs and ClawHub archive installs or updates.

## Best Possible Solution

Land one canonical setup-hook design only after env forwarding reuses existing policy and config resolution, `SKILL_DIR` has stable semantics, the public docs define the contract, unrelated drift is removed, and redacted real install/update proof is supplied.

## Reproduction Assessment

Yes for the PR defects by source inspection: the changed lines show direct env forwarding, staged archive hook paths, unused test code, and unrelated generated-file drift. For the linked request itself, bug reproduction is not applicable because it adds a new capability.

## Solution Assessment

No. The direction may be viable, but this patch is not the narrowest maintainable solution until it composes with the existing env/security contract, stable archive publish semantics, docs, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Filter setup env through the existing skill policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` copies each declared `requires.env` value directly from `process.env`. That bypasses the current skill env policy that blocks execution-affecting names such as `NODE_OPTIONS` or `OPENSSL_CONF`, so the hook env should be built through the same sanitizer/host-exec policy before spawning third-party setup code.
  - confidence: 0.93
- **[P2] Resolve setup env from skill config too:** `src/agents/skills-setup.ts:112-115`
  - body: The current skill contract lets `requires.env` be satisfied from `skills.entries.*.env` or `apiKey`, but this runner only reads ambient `process.env`. A skill installed from the UI or gateway with stored config credentials will run its setup script without the credentials the feature is meant to provide.
  - confidence: 0.86
- **[P2] Use stable SKILL_DIR semantics for archive hooks:** `src/agents/skills-archive-install.ts:185-187`
  - body: `installPackageDir` passes a temporary `.openclaw-install-stage-*` directory to `afterInstall` and publishes it afterward. Setup scripts that persist or print `SKILL_DIR` will see a path that disappears, not the final workspace skill directory promised by the new contract.
  - confidence: 0.9
- **[P2] Remove the unused timeout local:** `src/agents/skills-setup.test.ts:34`
  - body: `setupSkillMd` computes `timeout` but never reads it. The core test typecheck enforces unused locals, so this added test file can fail the changed test typecheck until the dead local is removed.
  - confidence: 0.88
- **[P2] Drop the unrelated Canvas bundle hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes only the Canvas A2UI bundle hash without changing the Canvas bundle inputs. That unrelated generated-file drift can make bundle validation expect a Canvas build state this setup-hook PR does not produce.
  - confidence: 0.82
- **[P3] Document the setup hook contract:** `src/agents/skills/types.ts:38`
  - body: The PR exposes a new author-facing `metadata.openclaw.setup` field, but the skill docs do not define timing, env forwarding, failure behavior, rollback semantics, or the security model. Please add the public contract before shipping the new field.
  - confidence: 0.78
