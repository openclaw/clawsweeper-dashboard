## Summary

Keep open: the PR is a plausible implementation of the requested setup hook and it addresses several earlier review points, but the current patch still has blocking lifecycle and security-contract issues, and the supplied proof is still test-only rather than real behavior proof.

## What This Changes

The PR adds a SKILL.md `metadata.openclaw.setup.script` lifecycle hook, parser/types, archive and metadata install invocation, command env support, tests, and a changelog entry.

## Best Possible Solution

Define a deliberate setup-hook lifecycle contract that runs against the durable installed skill directory with rollback, applies existing env sanitization, covers every install kind, documents trust semantics, and includes real CLI/archive proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: `afterInstall` receives `stageDir` before final publish, the metadata `download` branch returns before the new setup block, and the setup env copy bypasses the existing skill env sanitizer. As a feature request, current-main bug reproduction is not applicable.

## Solution Assessment

No. The direction may be viable, but this patch is not yet the narrowest maintainable solution because it still has lifecycle ordering, installer coverage, env-sanitizer, proof, and trust-policy gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Run archive hooks against the durable skill directory:** `src/agents/skills-archive-install.ts:184-186`
  - body: `installPackageDir` invokes `afterInstall` with the staging directory before it moves the package into the final target. That means archive and ClawHub setup scripts see `SKILL_DIR` as a temporary `.openclaw-install-stage-*` path, so any config or vault entry that records the bundle directory will be wrong after install/update succeeds.
  - confidence: 0.91
- **[P2] Run setup after download install specs:** `src/agents/skills-install.ts:516-517`
  - body: The metadata installer still returns immediately for `kind: "download"`, before the new setup-hook block. Skills using the existing download install kind can declare `setup.script` but it will never run, even though `download` is one of the supported install kinds this feature is meant to cover.
  - confidence: 0.88
- **[P1] Filter forwarded setup env with the skill sanitizer:** `src/agents/skills-setup.ts:97-106`
  - body: The setup runner copies every declared `requires.env` value directly from `process.env`. Existing skill env injection runs those values through sanitizer logic that blocks dangerous host env names, so this new install-time code path bypasses the security contract the issue and PR body describe.
  - confidence: 0.84
