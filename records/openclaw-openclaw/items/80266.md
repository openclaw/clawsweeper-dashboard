## Summary

Keep open: this is a plausible implementation of the linked skill setup-hook feature, but the current PR still has blocking correctness/security issues and only test-output proof, not real install/update behavior proof.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and setup-hook execution after skill install/update flows, with tests, changelog text, and a `baseEnv` option for command execution.

## Best Possible Solution

Define one documented setup-hook lifecycle that reuses the existing env sanitizer, gives archive installs durable `SKILL_DIR` semantics, and lands through a single canonical PR with redacted real install/update proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the setup runner forwards declared env directly, archive setup receives a staging path, and the new test helper has an unused local. For the linked feature request itself, reproduction is not applicable because it adds a new capability.

## Solution Assessment

No. The setup-hook direction may be viable, but this patch is not the best mergeable solution until sanitizer parity, durable path semantics, docs/policy, unrelated generated-file drift, and real install/update proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Filter setup env through the host env policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` copies declared `requires.env` values straight from `process.env` into the hook environment. A skill can declare names like `NODE_OPTIONS` or `OPENSSL_CONF` and receive values that current skill env handling blocks elsewhere, so the setup runner should reuse the existing sanitizer/host-env policy before spawning third-party install scripts.
  - confidence: 0.93
- **[P2] Pass archive hooks a durable skill directory:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir` invokes `afterInstall` with the temporary `.openclaw-install-stage-*` directory before publishing it to the real target. Passing that path as `SKILL_DIR` means archive/ClawHub setup scripts can persist a directory that is removed or moved after install, instead of the final skill directory promised by the feature contract.
  - confidence: 0.91
- **[P2] Remove the unused setup timeout variable:** `src/agents/skills-setup.test.ts:34`
  - body: `timeout` is computed but never read in the new test helper. Core test typechecking uses unused-local checks, so this keeps the changed test surface from typechecking cleanly.
  - confidence: 0.88
- **[P2] Remove the unrelated Canvas bundle hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes only the Canvas A2UI bundle hash without changing Canvas bundle inputs. That unrelated generated-file drift can make bundle validation expect a bundle state this setup-hook PR does not produce.
  - confidence: 0.82
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:35`
  - body: This adds a new author-facing `metadata.openclaw.setup` field, but the skill docs still do not describe timing, env forwarding, failure behavior, rollback semantics, or the security model. Please add the docs alongside the public metadata type before shipping the surface.
  - confidence: 0.78
