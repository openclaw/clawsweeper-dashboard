## Summary

Keep open: this is a plausible core feature PR, but the current patch still has correctness and security blockers after the previous review comments, and the supplied proof is test output rather than a real install/update run.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and install/update setup-hook execution for skills, extends process env handling, adds tests and a changelog entry, and changes a Canvas A2UI bundle hash.

## Best Possible Solution

Define a documented setup-hook lifecycle with durable `SKILL_DIR`, rollback semantics, host-env sanitizer reuse, clear trust/consent policy, ClawHub/archive plus metadata installer coverage, and real install/update proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the archive hook receives `stageDir` before publish, and declared setup env is copied from `process.env` without the existing host env block policy. For the linked feature request itself, reproduction is not applicable because this is a new capability.

## Solution Assessment

No. The direction may be viable, but this patch is not yet the best maintainable implementation because the lifecycle path, env policy, docs, and real behavior proof still need work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Filter setup env through the host env policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` copies every declared `requires.env` value from `process.env` into the setup hook environment. Existing skill env handling blocks dangerous names such as `NODE_OPTIONS` even when declared, but this path would forward them to install-time scripts.
  - confidence: 0.9
- **[P2] Run archive hooks with the durable skill path:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir` invokes `afterInstall` with the staging directory before publish, so ClawHub/upload setup scripts see a temporary `.openclaw-install-stage-*` `SKILL_DIR`. Hooks that persist config or vault paths will record the wrong location after install or update succeeds.
  - confidence: 0.9
- **[P2] Remove the unrelated Canvas bundle hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes the Canvas A2UI `.bundle.hash` without changing Canvas A2UI sources or bundle inputs. That generated-asset drift is unrelated to skill setup hooks and can make asset validation expect a bundle state this PR does not provide.
  - confidence: 0.82
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:35`
  - body: This adds a new author-facing `SKILL.md` lifecycle field and env/trust contract, but the docs still describe only the existing `requires` and `install` metadata. Skill authors and operators need the timing, env, failure, rollback, and security model documented before this ships.
  - confidence: 0.78
