## Summary

Keep open: this PR is a plausible implementation for a still-open feature request, but it should not merge as-is because it adds install-time skill-supplied code execution while bypassing existing env filtering, relying on lexical path confinement, and using staged archive paths for `SKILL_DIR`.

## What This Changes

Adds `metadata.openclaw.setup.script` parsing and setup-hook execution after skill install/archive install flows, with tests, a changelog entry, a `runCommandWithTimeout` `baseEnv` option, and an unrelated Canvas A2UI bundle hash change.

## Best Possible Solution

Land one supported setup path only after it reuses the existing env and host-exec policy, realpath-confines script execution, preserves final `SKILL_DIR` and rollback semantics, documents the trust model, and includes redacted real install/update proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch copies declared env names directly, checks setup paths lexically, and receives archive staging paths through `afterInstall`. The linked request itself is a new feature, so there is no current-main bug reproduction beyond confirming the surface is absent.

## Solution Assessment

No. The feature direction may be viable, but this patch is not the best mergeable solution until env-policy parity, realpath confinement, final-directory semantics, docs/security contract, and real behavior proof are addressed; maintainers should also compare it with https://github.com/openclaw/openclaw/pull/82349.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Filter setup env through the existing skill env policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` copies every declared `requires.env` name from `process.env` into the hook. Existing skill env handling blocks execution-affecting keys such as `NODE_OPTIONS`, `OPENSSL_CONF`, and `LD_*`, so setup hooks can currently bypass that policy by declaring those names.
  - confidence: 0.94
- **[P1] Resolve setup script realpaths before execution:** `src/agents/skills-setup.ts:71-73`
  - body: The path guard is only lexical, but `fs.accessSync` and child process execution follow symlinks. A skill can declare a bundle-relative symlink that resolves outside the skill tree, bypassing the intended setup-script confinement and any archive/source scan of the bundled file contents.
  - confidence: 0.88
- **[P2] Keep SKILL_DIR stable for archive setup hooks:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir.afterInstall` receives the temporary staging directory before it is moved to the final skill location, and this value becomes `SKILL_DIR`. Setup scripts that persist or report `SKILL_DIR` will store a `.openclaw-install-stage-*` path that is gone after a successful install/update.
  - confidence: 0.9
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:35-36`
  - body: The PR adds public `metadata.openclaw.setup` fields without updating the skills docs for timing, env forwarding, failure behavior, rollback semantics, timeout ownership, or the trust model. Skill authors need that contract before relying on install-time execution.
  - confidence: 0.82
- **[P3] Remove the unrelated Canvas bundle hash drift:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This branch changes the generated Canvas A2UI `.bundle.hash` even though the PR has no Canvas source or bundle output changes, and current main no longer tracks that file. Keeping it would reintroduce unrelated generated state into a skills setup PR.
  - confidence: 0.84
