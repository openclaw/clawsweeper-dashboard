## Summary

Keep this PR open, but it is not ready to merge. The patch is a plausible implementation direction for the linked feature request, but it still has concrete lifecycle/security blockers, lacks real install/update proof, and overlaps with another open implementation for the same feature.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and runs skill-supplied setup scripts after skill install/update flows, with tests, a changelog entry, `runCommandWithTimeout` env support, and an unrelated Canvas bundle hash change.

## Best Possible Solution

Define and document one setup-hook lifecycle with durable `SKILL_DIR`, host env sanitizer reuse, rollback semantics, timeout/logging/consent rules, ClawHub/update coverage, and real install/update proof before landing an implementation.

## Reproduction Assessment

Yes for the PR defects by source inspection: setup env is copied directly from `process.env`, and archive hooks receive `installPackageDir`'s staging directory before publish. The linked request itself is a feature, so bug reproduction is not applicable.

## Solution Assessment

No. The direction may be viable, but this patch is not yet the best maintainable solution because the env policy, archive path lifecycle, docs, proof, and product/security contract still need work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Filter setup env through the skill env policy:** `src/agents/skills-setup.ts:112-114`
  - body: `runSkillSetupHook` copies every declared `requires.env` value from `process.env` into the setup hook environment. Existing skill env handling always blocks dangerous host execution variables even when declared, so this new install-time subprocess path should reuse that policy instead of forwarding names such as `NODE_OPTIONS` or `OPENSSL_CONF`.
  - confidence: 0.91
- **[P2] Pass archive hooks the final skill directory:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir` calls `afterInstall(stageDir)` before publishing to the canonical target, so ClawHub/upload setup scripts get `SKILL_DIR` set to a temporary `.openclaw-install-stage-*` directory. Hooks that persist config or vault paths will record a directory that disappears after install or update succeeds.
  - confidence: 0.9
- **[P2] Remove the unrelated Canvas bundle hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes the Canvas A2UI `.bundle.hash` without changing the Canvas A2UI sources or bundle inputs. That unrelated generated-asset drift can make bundle validation expect a state this skills lifecycle PR does not provide.
  - confidence: 0.82
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:38`
  - body: Adding `setup?` creates a new author-facing `SKILL.md` metadata field, but the public skill docs still describe only the existing metadata fields and installer specs. Skill authors and operators need the timing, env, failure, rollback, and security semantics documented before this ships.
  - confidence: 0.78
