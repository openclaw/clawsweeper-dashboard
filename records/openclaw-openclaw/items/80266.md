Keep this PR open, but not merge-ready: current main still lacks the requested skill setup-hook capability, while this branch introduces a new install-time code execution surface with unresolved security, product, proof, and current-main drift blockers.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and runs skill-supplied setup scripts after skill install/update paths, with tests, a changelog entry, and a restricted child-process environment option.

## Best Possible Solution

Define one maintainer-approved setup-hook contract with env-policy parity, stable install/update semantics, rollback behavior, public docs, and real install/update proof before merging any implementation.

## Reproduction Assessment

Yes for the PR defects by source inspection: the env forwarding path bypasses existing blocked-name policy, and archive hooks receive a staged directory. For the linked feature request itself, reproduction is not applicable because it adds a new capability that current main source lacks.

## Solution Assessment

No. The direction may be viable, but this patch is not the best mergeable solution until env policy parity, stable `SKILL_DIR` semantics, docs/security contract, branch drift, and real install/update proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Filter setup env through the existing skill env policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` forwards every declared `requires.env` name directly from `process.env`. Existing skill env handling blocks execution-affecting keys such as `NODE_OPTIONS` and `OPENSSL_CONF`, so setup hooks can currently bypass that policy by declaring those names.
  - confidence: 0.93
- **[P2] Keep SKILL_DIR stable for archive setup hooks:** `src/agents/skills-archive-install.ts:184-187`
  - body: Archive installs call the hook from `installPackageDir.afterInstall`, which receives the temporary `.openclaw-install-stage-*` directory before publish. Scripts that persist or report `SKILL_DIR` get a path that is moved away instead of the final installed skill directory promised by the feature contract.
  - confidence: 0.9
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:35`
  - body: The PR adds public `metadata.openclaw.setup` support without documenting timing, env forwarding, failure behavior, rollback semantics, or the trust model in the skills docs, leaving skill authors and reviewers without a stable contract for this new execution surface.
  - confidence: 0.82
