## Summary

Keep this PR open, but it is not mergeable as-is: the patch adds the requested setup-hook shape, but the runner leaks the full host environment to skill scripts, runs outside the archive install rollback window, and does not cover the existing metadata-based `skills.install` path. It also still needs real behavior proof beyond unit tests before merge.

## What This Changes

Adds `metadata.openclaw.setup.script` parsing, a setup-hook runner, archive install/update invocation, tests, and a changelog entry for skill setup hooks.

## Best Possible Solution

Define the setup hook as a constrained lifecycle contract with a controlled base environment, scanner/trust policy, rollback-safe install composition, docs, and coverage for both archive installs and metadata installer runs.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the runner inherits `process.env`, runs after archive publish/backup cleanup, and is only wired into archive install. As a feature request, a current-main bug reproduction is otherwise not applicable.

## Solution Assessment

No. The direction may be viable, but this implementation is not the narrowest maintainable solution until it enforces the declared env contract, preserves update rollback, covers the existing install flow, and has maintainer security review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Pass only declared setup env to hook scripts:** `src/agents/skills-setup.ts:109-110`
  - body: The setup runner builds `scriptEnv` from `process.env`, and `runCommandWithTimeout` also merges command env over the ambient process env. That means any installed skill with `setup.script` can read unrelated gateway secrets, not only `SKILL_DIR`, `OPENCLAW_HOOK_KIND`, and allowlisted `requires.env` values promised by the PR.
  - confidence: 0.97
- **[P2] Keep update rollback intact until setup succeeds:** `src/agents/skills-archive-install.ts:194-196`
  - body: On update, `installPackageDir` has already published the new tree and removed its backup before this code runs. If the setup hook then fails, this cleanup removes `targetDir`, leaving the user with neither the previous skill nor the updated one; keep the hook inside the install rollback window or preserve the backup until setup succeeds.
  - confidence: 0.92
- **[P2] Invoke setup from the existing skills.install flow:** `src/agents/skills-archive-install.ts:187-191`
  - body: This only invokes the hook after archive installs. The existing `installSkill()` path that runs `metadata.openclaw.install` still returns immediately after the installer command, so a skill installed through the current `skills.install` dependency flow will never run its declared setup script.
  - confidence: 0.88
