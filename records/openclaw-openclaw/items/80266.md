## Summary

Keep open: the feature is not on current main, and this PR still has blocking security/lifecycle issues plus only test-based proof. It needs maintainer product/security review before merge.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and setup-hook execution for skill archive and metadata installs, extends exec env handling, adds tests and a changelog entry, and also changes an unrelated Canvas A2UI bundle hash.

## Best Possible Solution

Define and document the setup-hook lifecycle deliberately: durable `SKILL_DIR`, rollback semantics, host-env sanitizer reuse, explicit trust/consent policy, ClawHub/archive plus metadata installer coverage, and real install/update proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the archive hook receives `stageDir` before publish, and setup env forwarding bypasses the existing host-env block policy. For the linked feature request itself, a bug reproduction is not applicable.

## Solution Assessment

No. The direction may be viable, but this patch is not yet the narrowest maintainable solution because lifecycle ordering, env policy, trust/consent, docs, and real behavior proof still need work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Filter setup env through the host env policy:** `src/agents/skills-setup.ts:112-115`
  - body: `runSkillSetupHook` copies every declared `requires.env` value from `process.env` into the setup hook environment. That bypasses the existing skill env sanitizer and host-env block list, so a skill can declare names such as `GITHUB_TOKEN`, `NODE_OPTIONS`, or package-manager override vars and receive them during install/update.
  - confidence: 0.88
- **[P2] Run archive hooks with the durable skill path:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir` calls `afterInstall(stageDir)` before publishing the staged copy to the final workspace path, so archive and ClawHub setup scripts see a temporary `.openclaw-install-stage-*` path as `SKILL_DIR`. Hooks that persist vault/config state will record the wrong location after install or update succeeds.
  - confidence: 0.9
- **[P2] Remove the unrelated Canvas bundle hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This PR changes the Canvas A2UI `.bundle.hash` without changing the A2UI bundle inputs or generated bundle. That unrelated generated-asset drift can make asset validation expect a bundle state that the skill setup-hook change does not provide.
  - confidence: 0.82
- **[P3] Document the setup hook contract before exposing it:** `src/agents/skills/types.ts:35`
  - body: This adds a new author-facing `SKILL.md` lifecycle field and env/trust contract, but the PR only updates code and changelog. Skill authors and operators need docs for when the hook runs, what env is forwarded, failure/rollback behavior, and the security model before this manifest surface ships.
  - confidence: 0.78
