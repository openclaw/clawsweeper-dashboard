## Summary

Keep this PR open for maintainer review, but do not merge as-is: it adds a real setup-hook capability, yet the patch still changes an install/update command-execution boundary without settled product/security direction, real behavior proof, or complete confinement semantics.

## What This Changes

The PR adds `metadata.openclaw.setup.script` parsing and automatically runs skill setup scripts after download, archive, and command-based skill installs, with tests and a changelog entry.

## Best Possible Solution

Maintainers should choose one setup-hook contract first; if core auto-run is accepted, the implementation should reuse shared env sanitization, realpath confinement, final install-path semantics, public docs, and real install/update proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new feature surface. The PR defects are source-reproducible from the head diff: env forwarding bypasses current sanitizer policy, script confinement is lexical, and archive hooks run in the staging directory.

## Solution Assessment

No. The PR is a plausible implementation direction, but it is not merge-ready until maintainers choose the trust boundary and the patch fixes env policy reuse, realpath confinement, archive final-path semantics, docs, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter setup env through the existing sanitizer:** `src/agents/skills-setup.ts:107-115`
  - body: `requires.env` values are copied straight from `process.env` into the setup script. That bypasses the current skill/host env policy that blocks dangerous runtime and shell override keys, so this new command-execution path should reuse or expose that sanitizer before forwarding env.
  - confidence: 0.9
- **[P1] Resolve the setup script realpath before executing:** `src/agents/skills-setup.ts:71-73`
  - body: The containment check only compares the lexical `path.resolve()` result, then `accessSync` and spawn follow symlinks. A skill can declare an inside-looking symlink that points outside the skill directory, so resolve both the skill directory and script realpaths and assert the executed target remains inside.
  - confidence: 0.87
- **[P2] Avoid exposing a staging SKILL_DIR for archive installs:** `src/agents/skills-archive-install.ts:184-187`
  - body: `installPackageDir` calls `afterInstall` with the staging directory before publishing it to the canonical target. Archive setup scripts that persist `SKILL_DIR` will record a dead temporary path after install, so the hook needs final-path semantics or a narrower staged-write contract.
  - confidence: 0.84
