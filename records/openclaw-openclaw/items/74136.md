## Summary

Keep open: current main still has exact-only `ANNOUNCE_SKIP` parsing, and this PR is the active implementation candidate for the linked bug. The code/test/changelog/sub-agent-docs changes look correct, but one FAQ entry still describes `ANNOUNCE_SKIP` as exact-only after the PR broadens the public contract.

## What This Changes

The PR broadens `isAnnounceSkip()` to accept `ANNOUNCE_SKIP` as the trimmed standalone final line, adds focused helper and announce-flow regression tests, and updates the changelog plus sub-agent docs.

## Best Possible Solution

Land this PR or an equivalent narrow patch after syncing the FAQ troubleshooting copy, while preserving strict `REPLY_SKIP` behavior and rejecting embedded or non-final `ANNOUNCE_SKIP` text.

## Reproduction Assessment

Yes, by source inspection. On current main, `isAnnounceSkip("DM summary block\nANNOUNCE_SKIP")` cannot match because the helper compares the whole trimmed string to the token, and the announce delivery paths call that helper.

## Solution Assessment

Yes. The PR's final-line-only check is the narrowest maintainable fix for the reported leak because it preserves exact-token behavior, handles trailing whitespace through trimming, and does not broaden `REPLY_SKIP`.
