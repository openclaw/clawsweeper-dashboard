## Summary

Keep open: current main and the latest release still send the hardcoded branded pairing reply and expose the CLI approval command, while the linked implementation PR is still open and conflicting. The issue requests configurable channel pairing reply text so unapproved senders do not receive hardcoded OpenClaw branding or raw CLI approval syntax.

## Best Possible Solution

Define one shared pairing-response contract with secure defaults, schema/docs/tests, and consistent integration across pairing-capable channels, then land https://github.com/openclaw/openclaw/pull/42617 or a tighter replacement after maintainer security/product approval.
