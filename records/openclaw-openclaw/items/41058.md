The issue requests configurable pairing-message text so operators can remove OpenClaw branding and the raw CLI approval hint from unknown-sender pairing responses.

## Best Possible Solution

Land one maintainer-approved unapproved-DM response contract across shared pairing helpers, config schemas, docs, and affected channel call sites, then close this issue after the chosen PR merges.

Yes, as a source-level current-behavior check rather than a live failing-test repro. `buildPairingReply` still emits the branded text and CLI approval command, and `issuePairingChallenge` sends that text for new pairing requests.

Unclear. Granular configurable text is a plausible fix, but maintainers need to choose whether the better long-term contract is field-level templates or the broader silent/code-only/branded response-mode design.
