## Summary

Keep open: current main and v2026.6.6 still send the branded pairing reply with CLI approval text and no `pairingMessage` config surface, and the remaining work is already paired with open implementation/design PRs that need maintainer review.

## What This Changes

The issue requests a channel-level configurable pairing message so operators can remove OpenClaw branding, sender-label wording, and the raw `openclaw pairing approve` hint from unknown-sender pairing replies.

## Best Possible Solution

Define one maintainer-approved pairing reply customization or response-mode contract at the shared pairing helper/SDK boundary, wire it through channel schemas and call sites, then land or replace the linked implementation PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an unknown DM sender under `dmPolicy: "pairing"` reaches `issuePairingChallenge`, which sends `buildPairingReply` containing the OpenClaw header and CLI approval command; I did not run a live channel in this read-only review.

## Solution Assessment

Unclear. `channels.*.pairingMessage` is a plausible solution, but the safer final shape may be a broader shared response-mode contract aligned with the open security thread and the repo's high bar for new config surfaces.
