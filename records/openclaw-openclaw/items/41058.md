## Summary

Keep open: current `main` and `v2026.5.20` still hardcode the unknown-sender pairing reply and expose no public `pairingMessage` config surface. The linked implementation at https://github.com/openclaw/openclaw/pull/42617 uses closing syntax for this issue, but it is still open and conflicting, so this should stay open until that PR lands, is replaced, or is closed.

## What This Changes

Request a configurable `pairingMessage` text surface for channel DM pairing replies so operators can hide OpenClaw branding and suppress the CLI approval hint sent to unknown senders.

## Best Possible Solution

Define a backward-compatible, SDK-friendly pairing-message contract, preserve the current default reply, document the option, and land or replace the linked implementation with real channel proof while coordinating with the broader unpaired-response security discussion.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug repro; this is a feature/config request. Source inspection gives high confidence that current main and `v2026.5.20` lack `pairingMessage` and still hardcode the pairing reply text.

Is this the best way to solve the issue? Unclear until maintainers settle the public config and security shape. The linked PR is a plausible path, but it needs refresh, review, and proof rather than closing this issue before merge.
