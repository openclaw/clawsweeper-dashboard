## Summary

Keep open: current main and the latest release still send a hardcoded branded pairing reply with CLI approval syntax and no supported `pairingMessage` or `unpairedResponse` config surface. The issue is already paired with an open closing PR and a broader open security PR, both still needing maintainer/design review rather than a new cleanup close.

## What This Changes

Requests configurable text or suppression for pairing replies sent to unapproved DM senders.

## Best Possible Solution

Land or replace the linked implementation with a maintainer-approved shared pairing reply contract that covers the relevant pairing-capable channels without growing config surface more than necessary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an unknown sender under `dmPolicy: "pairing"` reaches `issuePairingChallenge`, which sends `buildPairingReply` containing the OpenClaw header and CLI approval command; I did not run a live channel in this read-only review.

## Solution Assessment

Unclear. `channels.*.pairingMessage` is a plausible solution, but the broader `silent | code-only | branded` response-mode design may be the cleaner shared security contract because the request spans multiple pairing-capable channels.
