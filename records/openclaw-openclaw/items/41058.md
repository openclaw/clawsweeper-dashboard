## Summary

Keep open. Current main still sends the hardcoded OpenClaw-branded pairing reply and CLI approval hint, the strict channel config surfaces do not accept `pairingMessage`, and open PR #42617 explicitly targets this issue with closing syntax, so the issue should remain open until that implementation candidate is reviewed and merged or closed.

## What This Changes

Request an optional per-channel `pairingMessage` config that can customize the unknown-sender DM pairing reply text and optionally suppress the raw `openclaw pairing approve ...` CLI hint.

## Best Possible Solution

Review PR #42617 as the active implementation candidate, or land an equivalent typed config surface that preserves current defaults, validates strict schemas, documents the setting, supports CLI-hint suppression, and applies consistently to pairing-capable channels or leaves explicit follow-up tracking for staged rollout.

## Reproduction Assessment

Yes. Static reproduction on current main is high-confidence: an unapproved sender with `dmPolicy: "pairing"` flows through `issuePairingChallenge` to `buildPairingReply`, which still emits the OpenClaw header and CLI hint, while channel schemas/types reject the requested config key.

## Solution Assessment

Yes, the requested optional typed config with backward-compatible defaults is the narrow maintainable direction. The current main implementation does not solve it yet, and the open PR needs maintainer review for channel coverage and schema/docs completeness.
