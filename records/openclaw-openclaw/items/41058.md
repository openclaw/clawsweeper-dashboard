## Summary

Keep open: current main still sends the hardcoded OpenClaw-branded pairing reply and CLI approval hint, and open PR #42617 uses closing syntax for this issue. The issue should remain open until that implementation candidate is merged, revised, or closed.

## What This Changes

Request optional `channels.*.pairingMessage` config so operators can customize pairing reply labels and suppress the CLI hint sent to unknown DM senders.

## Best Possible Solution

Review PR #42617 as the canonical implementation candidate, then land or revise a typed, documented, backward-compatible `pairingMessage` config with an explicit maintainer decision on channel scope.

## Reproduction Assessment

Yes. Source inspection shows an unapproved sender under `dmPolicy: "pairing"` reaches fixed-label pairing challenge code, and the inspected strict channel schemas do not expose `pairingMessage`.

## Solution Assessment

Unclear until the paired PR is reviewed. A typed optional config with current defaults is the narrow maintainable direction, but all-channel versus iMessage/Telegram-first scope needs maintainer approval.
