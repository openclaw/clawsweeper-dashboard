## Summary

Keep open: current main and v2026.6.10 still require hook capability metadata or explicit hook policy config before Gateway startup loads hook plugins, current docs expose `registerHook`/`before_dispatch` without fully documenting that startup rule, and the linked docs/runtime PRs are open with review blockers.

## What This Changes

The issue asks OpenClaw to document, fail fast, or route non-capability plugin dispatch hooks so plugin authors do not silently register hooks that never run in the Gateway chat path.

## Best Possible Solution

Have maintainers choose the public hook activation contract, then land corrected docs or a runtime/fail-fast change that preserves startup opt-out, allow/deny, and security boundaries.

## Reproduction Assessment

Yes, source-level: current main only starts hook plugins with `activation.onCapabilities: ["hook"]` or explicit hook policy config, so an enabled non-capability plugin that only registers `before_dispatch` has no Gateway startup path. I did not run the live Slack/Gateway deployment reproduction in this read-only review.

## Solution Assessment

Unclear as a product choice: docs-only mitigation is useful but may not be the best final contract, while the runtime PR addresses an overlapping regression but still has startup/security proof blockers. The narrowest maintainable end state needs maintainer confirmation of docs versus fail-fast versus runtime loading.
