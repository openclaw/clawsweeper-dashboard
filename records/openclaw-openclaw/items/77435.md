## Summary

Keep open. Current main partially documents plugin runtime debugging, but it still does not clearly state the Gateway chat startup-selection rule for `before_dispatch` and hook-only/non-capability plugins, while source and tests show that rule controls whether `register(api)` runs.

## What This Changes

Request to fail fast or document how Gateway chat hook routing works for non-capability plugins that register dispatch-time hooks.

## Best Possible Solution

Document the Gateway chat hook activation contract in the plugin hook and troubleshooting docs, including the manifest/config knobs that make hook-only or non-capability plugins load for messaging traffic.

## Reproduction Assessment

Yes, from source: the chat path only sees hooks registered in the active Gateway hook runner, and Gateway startup loads only the plugin ids selected by the startup plan. A non-capability plugin without hook activation or explicit hook policy can be inspectable but absent from the chat runtime.

## Solution Assessment

Yes for a docs-first fix: documenting the current startup routing rule is the narrowest maintainable way to close the plugin-author gap. Fail-fast registration or loading more plugin shapes in chat should be a separate maintainer decision.
