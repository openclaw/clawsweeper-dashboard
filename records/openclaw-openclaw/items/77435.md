## Summary

Keep open. Current main wires `before_dispatch` and explicit hook startup paths, but the public docs still do not clearly connect plugin shape, hook activation metadata, hook policy config, and Gateway chat runtime selection.

## What This Changes

Request to either fail fast or document why non-capability plugins that register `before_dispatch` do not participate in Gateway chat hook routing unless selected for the runtime.

## Best Possible Solution

Document the Gateway chat hook activation contract in the plugin hook docs and troubleshooting flow, including the manifest/config knobs that make hook-only or non-capability plugins load for messaging traffic.

## Reproduction Assessment

Yes, from source: a plugin can be present in CLI inventory while absent from the Gateway startup `pluginIds`, and `before_dispatch` only runs from hooks registered in that active runtime. The exact external plugin was not available for live confirmation.

## Solution Assessment

Yes for a docs-first repair: documenting the existing startup routing rule is the narrowest maintainable fix for the reported silent no-op. Runtime fail-fast behavior would be a separate follow-up if maintainers want stricter enforcement.

## Repair Work Prompt

Treat https://github.com/openclaw/openclaw/issues/77435 as a docs-only repair. Update the plugin hook/troubleshooting docs to explain Gateway chat hook activation for hook-only and non-capability external plugins: `plugins list` and default `plugins inspect` prove inventory/loadability, not that the serving Gateway runtime has invoked `register(api)`; messaging hooks such as `before_dispatch` fire only after the plugin is included in Gateway startup/runtime loading via the existing activation contract, such as `activation.onStartup: true`, `activation.onCapabilities: ["hook"]` plus explicit enablement/allowlist as needed, or explicit hook policy config where applicable. Include the recommended diagnostics (`openclaw plugins inspect <id> --runtime --json`, `openclaw gateway status --deep --require-rpc`, restart the actual Gateway process) and mention that non-bundled conversation-content hooks still require `plugins.entries.<id>.hooks.allowConversationAccess=true`. Do not change runtime startup semantics, add a new config option, or implement fail-fast behavior in this PR. No changelog is expected for a docs-only clarification unless the patch changes behavior or public CLI/API output.
