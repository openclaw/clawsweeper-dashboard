## Summary

Keep open. The issue is paired with an open closing docs PR, and current main still shows the source behavior behind the report: chat-path hooks only run after Gateway startup loads the plugin into the runtime, while the hook docs do not yet clearly document that startup-routing contract.

## What This Changes

Request to fail fast or document how non-capability plugins must be selected by Gateway startup before chat-path hooks such as `before_dispatch` can run.

## Best Possible Solution

Land a docs clarification that ties chat-path hooks to `activation.onCapabilities: ["hook"]` or explicit hook policy, without changing runtime loading semantics unless maintainers choose a stricter follow-up.

## Reproduction Assessment

Yes, from source. Gateway chat dispatch only sees hooks registered in the active runtime registry, and Gateway startup loads an explicit plugin id set rather than every installed or inspectable plugin.

## Solution Assessment

Yes. A docs-first fix is the narrowest maintainable solution for the reported plugin-author gap; fail-fast or broader runtime loading would be a separate behavior change.
