## Summary

Keep open. Current main still routes chat-path hooks only through the Gateway startup plugin plan, the hook reference still does not spell out that routing requirement for non-capability or hook-only plugins, and the linked docs PR that would close the gap is still open.

## What This Changes

Request asks OpenClaw to fail fast or document which plugin shapes and activation metadata route messaging-dispatch hooks such as `before_dispatch` into Gateway chat runtime.

## Best Possible Solution

Land a corrected docs clarification that ties chat-path hook availability to Gateway startup selection via `activation.onCapabilities: ["hook"]` or explicit hook policy, leaving fail-fast or broader runtime loading for a separate SDK contract decision.

## Reproduction Assessment

Yes. Source inspection shows Gateway startup loads a planned plugin id set, `before_dispatch` runs only from the active hook runner, and tests cover explicit hook startup without ambient loading.

## Solution Assessment

Yes for a docs-first fix. Clarifying the existing startup-selection contract is narrower than changing runtime loading or adding fail-fast behavior, though maintainers may choose a later SDK behavior change separately.
