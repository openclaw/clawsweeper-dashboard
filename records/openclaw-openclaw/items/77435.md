## Summary

Keep open. Current main has the hook runner and explicit startup-selection contract in source, plus some CLI diagnostic notes, but the public hook docs still do not clearly state the activation/config needed for a hook-only or non-capability plugin to receive `before_dispatch` in Gateway chat traffic.

## What This Changes

The issue requests fail-fast behavior or documentation clarifying how plugin hook registration is routed for non-capability plugins in the Gateway chat runtime.

## Best Possible Solution

Add a docs-first clarification in the plugin hook and CLI/plugin troubleshooting docs that explains the Gateway startup activation contract for messaging-dispatch hooks such as `before_dispatch`, including the relevant manifest/config knobs and diagnostics.

## Reproduction Assessment

Yes, from source: `before_dispatch` can only run after a plugin has registered with the global hook runner, and Gateway startup loads only plugin IDs selected by the startup plan. A plugin can therefore be inspectable outside the running Gateway but absent from chat-path hook execution.

## Solution Assessment

Yes for a docs-first repair: documenting the startup routing rule is the narrowest maintainable fix for the reported silent no-op. Runtime fail-fast behavior or broader hook activation changes would be a separate API/product decision.
