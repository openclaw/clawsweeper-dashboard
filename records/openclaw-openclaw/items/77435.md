## Summary

Keep open. Current main has the `before_dispatch` hook call and explicit Gateway startup activation paths for hook-capability or hook-policy plugins, but the public hook docs still do not tie messaging-dispatch hooks to that startup routing contract for hook-only or non-capability plugins.

## What This Changes

Request to fail fast or document when hook-only/non-capability plugins are loaded into the Gateway chat runtime for messaging-dispatch hooks.

## Best Possible Solution

Document the Gateway chat hook activation contract in the plugin hook docs and cross-link the manifest activation reference, including how hook-only/non-capability plugins opt into messaging traffic and what plugin inspect/load status does not guarantee.

## Reproduction Assessment

Yes, from source: the chat path only sees hooks registered in the global hook runner, and Gateway startup loads only planned plugin IDs. A non-capability plugin that is installable/inspectable but lacks hook startup activation or explicit hook policy can therefore miss `before_dispatch` registration in the chat runtime.

## Solution Assessment

Yes for a docs-first repair: the current code and tests already define the routing boundary, so documenting it is the narrowest maintainable fix. Runtime fail-fast or always-loading hook plugins would be a separate behavior decision.

## Repair Work Prompt

Fix the docs gap for https://github.com/openclaw/openclaw/issues/77435 without changing runtime behavior. Update docs/plugins/hooks.md to explain that messaging-dispatch hooks such as `before_dispatch` only fire after the plugin has been loaded into the Gateway chat runtime, so `openclaw plugins inspect` showing loaded/installable is not enough by itself. Document the supported opt-in paths for hook-only or non-capability plugins: manifest `activation.onCapabilities: ["hook"]` with plugin enablement/allowlist satisfied, explicit `plugins.entries.<id>.hooks` policy that creates hook startup intent where appropriate, or `activation.onStartup: true` when the plugin truly needs Gateway startup import. Cross-link docs/plugins/manifest.md and update its activation live-consumers wording if needed so `hook` is tied back to Gateway startup planning. Mention allow/deny/global disable behavior and avoid promising that `register()` runs for plugins outside the startup plan. Do not add bundled plugin ids, do not change loader semantics, and do not implement fail-fast behavior in this PR. Add a changelog entry only if maintainers treat this docs fix as user-facing issue resolution.
