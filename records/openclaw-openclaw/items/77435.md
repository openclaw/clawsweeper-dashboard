## Summary

Keep open. Current main has the `before_dispatch` hook path, but the docs still do not clearly connect message-hook availability to Gateway startup activation for hook-only or non-capability plugins, and they leave room to confuse typed `api.on(...)` hooks with internal `api.registerHook(...)` registrations.

## What This Changes

The issue asks OpenClaw to either fail fast or document why non-capability plugins can appear loaded while messaging-dispatch hooks never run in live chat traffic.

## Best Possible Solution

Add focused public docs for Gateway chat hook activation, including the typed-hook API, the startup/runtime loading contract, the manifest/config knobs that make hook-only plugins participate, and the runtime inspection workflow.

## Reproduction Assessment

Yes, from source: the chat path only consults hooks already registered in the global hook runner, and Gateway startup loads a narrowed plugin set. A plugin that is statically inspectable but absent from that startup set can show as installed/loaded outside chat while never registering its message hook in live dispatch.

## Solution Assessment

Yes for a docs-first fix. Documenting the current startup routing rule is the narrowest maintainable solution; changing runtime loading semantics or making registration fail fast would be a larger SDK behavior decision.
