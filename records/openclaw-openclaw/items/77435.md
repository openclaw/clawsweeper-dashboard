## Summary

Keep open. Current main still routes chat-path hooks only through plugins loaded into the Gateway runtime, the hook docs do not yet explain the startup-routing requirement, and the linked closing PR remains open and unmerged.

## What This Changes

The issue requests documentation or fail-fast/runtime behavior for non-capability plugins that register dispatch-time hooks but are not selected for Gateway chat runtime loading.

## Best Possible Solution

Land a corrected docs clarification that ties chat-path hook availability to Gateway startup selection via activation.onCapabilities ["hook"] or explicit hook policy, and leave fail-fast or broader runtime loading for a separate SDK contract decision.

## Reproduction Assessment

Yes, from source. Gateway startup loads a planned plugin id set, dispatch only runs before_dispatch from the active hook runner, and tests cover explicit hook startup without ambient loading.

## Solution Assessment

Yes for a docs-first fix. Clarifying the existing startup-selection contract is narrower than changing runtime loading or adding fail-fast behavior, though maintainers may choose a later SDK behavior change separately.
