## Summary

Keep open. Current main wires `before_dispatch`, but the Gateway chat runtime still loads only selected startup plugins, and the public docs do not clearly say how a non-capability plugin opts into that runtime before its hooks can run.

## What This Changes

Document or fail fast the Plugin SDK hook routing rules for non-capability plugins whose messaging hooks appear loaded but do not run in chat traffic.

## Best Possible Solution

Document the Gateway chat hook activation contract in the plugin hook docs and troubleshooting guide, including the manifest/config knobs that make a hook-only or non-capability plugin load for messaging traffic.

## Reproduction Assessment

Yes, from source: the chat path only sees hooks registered in the global hook runner, and Gateway startup loads an explicit plugin subset. A non-capability plugin without hook activation or hook policy can be loaded by CLI inspection but not by the chat runtime.

## Solution Assessment

Yes for a docs-first fix: documenting the startup routing rule is the narrowest maintainable way to close the reported plugin-author gap. Changing runtime semantics or failing registration should be considered separately if maintainers want stricter behavior.

## Repair Work Prompt

For https://github.com/openclaw/openclaw/issues/77435, update the plugin docs to make Gateway chat hook routing explicit. Document that typed messaging hooks such as `before_dispatch` only run in chat traffic after the plugin is selected for the Gateway startup runtime, and point plugin authors to manifest `activation.onCapabilities: ["hook"]` or explicit `plugins.entries.<id>.hooks` policy as the current activation signals. Clarify how this differs from `openclaw plugins inspect --runtime` and from CLI/plugin install loading. Keep the fix docs-only unless maintainer direction asks for fail-fast runtime behavior; do not change hook semantics, plugin activation, or config schema. No changelog is needed for a docs-only change.
