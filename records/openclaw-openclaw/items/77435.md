## Summary

Keep open. Current main has a source-defined activation path for `before_dispatch`, but the public hook docs still do not clearly connect messaging-dispatch hooks to Gateway startup selection, so the author-facing gap remains.

## What This Changes

Clarify or enforce how non-capability plugins get messaging-dispatch hooks loaded and routed in the Gateway chat runtime.

## Best Possible Solution

Document the Gateway chat hook activation contract in the plugin hook and CLI troubleshooting docs, including the manifest/config knobs that make hook-only or non-capability plugins load for messaging traffic.

## Reproduction Assessment

Yes, from source inspection: chat dispatch only sees hooks registered in the global hook runner, and Gateway startup only loads the selected startup plugin set. A non-capability plugin without hook activation metadata or hook policy can be inspectable by CLI but absent from the chat runtime.

## Solution Assessment

Yes for a docs-first repair: documenting the startup routing rule is the narrowest maintainable solution for the reported silent no-op. Runtime fail-fast behavior can be considered later if maintainers want stricter plugin loading semantics.

## Repair Work Prompt

Treat https://github.com/openclaw/openclaw/issues/77435 as a docs-first repair unless a maintainer explicitly asks for runtime behavior changes. Update the plugin hook docs and related CLI/manifest references to explain the Gateway chat hook activation contract: `before_dispatch` is a typed plugin hook registered with `api.on`, not an internal `api.registerHook` event; the chat runtime uses the global hook runner populated by Gateway startup; external hook-only or non-capability plugins must be loaded into that startup plan via explicit hook activation metadata such as `activation.onCapabilities: ["hook"]` or explicit hook policy config, while remaining subject to enabled/allow/deny rules; `plugins list`/plain `inspect` are not live runtime proof. Clarify that `allowConversationAccess` specifically gates `llm_input`, `llm_output`, `before_agent_finalize`, and `agent_end`, not every dispatch hook. Do not change runtime loading semantics, add config options, or edit plugin baselines. Add a concise changelog entry if required by current OpenClaw policy for closing a user-facing docs issue.
