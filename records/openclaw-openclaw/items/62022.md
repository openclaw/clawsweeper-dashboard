## Summary

Keep open. Current main still uses the documented default of one same-provider rate-limit profile rotation before model fallback, while the reporter wants Codex profiles exhausted first; that is an intentional failover-policy tradeoff rather than an already-fixed bug.

## What This Changes

The issue requests Codex/openai-codex rate-limit handling to continue through remaining auth profiles for the current model before escalating to fallback models.

## Best Possible Solution

Choose a clear Codex rate-limit policy, then either document the capped default and tuning path for Codex users or implement a bounded provider/profile-count-aware exception with matching docs and E2E coverage.

## Reproduction Assessment

Yes. Static inspection of current E2E coverage gives a high-confidence path: with three primary-provider profiles, the default policy expects two primary attempts before fallback, while a high `rateLimitedProfileRotations` override allows all three.

## Solution Assessment

Unclear. Exhausting every profile would match this reporter's expectation, but the current cap is documented and preserves faster fallback for model-scoped quotas, so a targeted policy is safer than a blanket default change.
