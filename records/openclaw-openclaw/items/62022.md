## Summary

Keep open. Current main no longer matches the narrow "first exhausted profile immediately jumps to fallback" case because the embedded runner rotates once to the next auto-selected profile, but it still intentionally does not exhaust every remaining profile by default; the documented default is one same-provider rate-limit rotation before model fallback. That leaves the broader requested behavior as a maintainer product decision rather than a conservative cleanup close.

## What This Changes

The issue requests that Codex rate-limit failures continue through the remaining auth profiles for the same model before escalating to the configured fallback model chain.

## Best Possible Solution

Maintainers should decide the desired default semantics for subscription-style Codex rate limits. If exhausting all profiles is desired, implement it as a bounded provider-specific or profile-count-aware policy and update the failover docs and E2E expectations; otherwise leave the current cap and document the `rateLimitedProfileRotations` tuning path in the issue response.

## Reproduction Assessment

Yes. Current source and E2E tests give a high-confidence reproduction path for the policy: one rate-limited auto profile rotates to the next profile, while a three-profile provider falls back after two primary-provider attempts by default rather than exhausting all profiles.

## Solution Assessment

Unclear. The current implementation is maintainable and documented, but it is not the exact all-profiles-before-fallback behavior requested; changing it safely requires maintainer product judgment because it would alter guarded model-fallback behavior.
