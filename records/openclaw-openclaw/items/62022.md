## Summary

Keep open. Current main rotates to another auth profile once and supports a high `rateLimitedProfileRotations` workaround, but it still intentionally falls back after the default rate-limit rotation cap instead of exhausting all remaining profiles, so the requested Codex behavior remains a maintainer failover-policy decision.

## What This Changes

The issue requests that Codex rate-limit handling continue through all remaining auth profiles for the current model before escalating to configured fallback models.

## Best Possible Solution

Maintainers should choose the desired Codex rate-limit policy, then either keep the capped default and document the tuning path in the issue response or implement a bounded provider/profile-count-aware policy with matching docs and E2E expectations.

## Reproduction Assessment

Yes. Current source and E2E inspection provide a high-confidence reproduction path: with three primary-provider profiles, the default policy expects two primary attempts before fallback, while a high `rateLimitedProfileRotations` override allows all three attempts.

## Solution Assessment

Unclear. Exhausting every profile would match the reporter's expectation, but the current cap is documented and intentionally protects faster fallback for model-scoped quotas, so a narrower provider-specific or profile-count-aware policy would be safer than a blanket default change.
