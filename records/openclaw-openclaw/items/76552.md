## Summary

Keep open. The report is recent, detailed, and still plausible on current main: Codex native runtime still installs synchronous command hooks for `PostToolUse`, and each native hook event still enters an `openclaw hooks relay` CLI process before reaching the in-process bridge/gateway path; current session/history mitigations are useful but do not prove the combined low-core CPU saturation path is fixed.

## What This Changes

The issue asks OpenClaw to reduce high CPU/load spikes from the native Codex runtime hook relay path and concurrent gateway session/history/status work on small Linux hosts.

## Best Possible Solution

Profile this path on a low-core Linux host, then add measured default backpressure or a cheaper relay execution path for native Codex hook events while preserving the documented plugin hook semantics.

## Reproduction Assessment

Unclear for a full failure reproduction: the issue provides concrete low-core telemetry and A/B control cases, and current source confirms the Codex app-server -> command hook relay -> gateway/session paths. I did not establish a deterministic current-main stress run that reproduces the high load average.

## Solution Assessment

Unclear. Backpressure or a cheaper in-process/pooled relay path is the likely direction, but maintainers need profile data before choosing event-specific throttling, batching, disabling, or broader gateway workload admission.
