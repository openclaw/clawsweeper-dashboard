## Summary

Keep open. The report is recent and well-supported, and current `main` still routes native Codex `PostToolUse` through command-based `openclaw hooks relay` processes while gateway session/history work remains only partially mitigated rather than proven fixed.

## What This Changes

The issue asks OpenClaw to reduce high CPU/load on low-core hosts when native Codex runtime hook relay work overlaps with gateway session/history/status activity.

## Best Possible Solution

Profile the low-core native Codex turn path, then add measured default admission control or a cheaper pooled/in-process relay for frequent `PostToolUse` events while preserving documented plugin hook behavior and existing session/history bounds.

## Reproduction Assessment

Unclear for the full high-load failure: the reporter provides strong low-core A/B evidence, and current source confirms the named native Codex hook relay and gateway session/history paths. I did not run a live 2-vCPU Codex stress scenario against current `main`.

## Solution Assessment

Unclear. The likely direction is default relay admission/backpressure or cheaper relay execution, but maintainers need profile data before choosing between event-specific throttling, batching, disabling, or broader gateway workload scheduling.
