## Summary

Keep open. The report is recent, detailed, and still matches current main: native Codex still installs per-event command hooks that invoke `openclaw hooks relay`, and the current relay path has bounds/timeouts but no demonstrated concurrency backpressure for `post_tool_use` bursts. The linked #76607 pdf-tool report is a separate closed hotspot, not a canonical fix for this native Codex relay path.

## What This Changes

The issue asks OpenClaw to reduce CPU saturation from native Codex runtime hook-relay fanout competing with gateway session/history/status work on low-core Linux hosts.

## Best Possible Solution

Profile the low-core native Codex runtime path, then add measured default backpressure or a lower-cost post-tool relay path while preserving hook semantics and the existing session/history bounds.

## Reproduction Assessment

No high-confidence full reproduction was established in this review. The issue provides live A/B evidence on OpenClaw 2026.5.3 and current source confirms the native Codex command-hook path, but I did not run a low-core stress scenario against current main.

## Solution Assessment

Unclear. A measured relay backpressure or cheaper bridge/worker path is likely the right direction, but maintainers need profiling before choosing throttling, batching, disabling, or broader gateway workload admission.
