## Summary

Keep open. The report is recent, detailed, and still matches current `main`: Codex native runtime still installs synchronous command-based hook relays, the relay dispatch path has no visible per-event admission queue for `post_tool_use`, and gateway session/history mitigations are partial rather than proof that the low-core CPU saturation path is fixed.

## What This Changes

Investigate and reduce high CPU/load average on low-core Linux native Codex runs where command-based hook relay and gateway session/history/status work overlap.

## Best Possible Solution

Profile the low-core native Codex path, then add measured default backpressure or cheaper relay execution while preserving existing plugin hook semantics and bounded session/history behavior.

## Reproduction Assessment

Unclear for a full failure reproduction: the issue provides concrete host/process/load evidence, and current source confirms the named relay and Gateway paths. I did not establish a deterministic current-main low-core stress run that reproduces the high load average.

## Solution Assessment

Unclear. Relay backpressure or a cheaper in-process/pooled relay path is plausible, but maintainers need CPU profiles and event timing before choosing throttling, batching, default disabling, or Gateway workload admission.
