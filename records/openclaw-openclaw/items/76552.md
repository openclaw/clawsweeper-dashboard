## Summary

Keep open. The report is recent, detailed, and still plausible on current main: Codex native hooks still install synchronous command relays for tool events, each relay can enter OpenClaw hook processing without visible concurrency admission, and session/history RPCs only have partial bounded-work mitigations.

## What This Changes

The issue reports CPU saturation on low-core Linux hosts during native Codex runtime tasks, with suspected amplification from hook relay subprocesses plus gateway session, history, status, and log work.

## Best Possible Solution

Profile the native Codex low-core path, then add measured default backpressure or a cheaper relay execution path for high-frequency native hook events while preserving existing plugin hook semantics and session/history bounds.

## Reproduction Assessment

Unclear for the full load failure. The issue gives a concrete low-core native Codex scenario and source confirms the named relay plus session/history paths still exist on main, but I did not run a 2-vCPU Codex stress test to reproduce the sustained load averages.

## Solution Assessment

Unclear. Backpressure or a cheaper relay path is likely, but profile data is needed before choosing event-specific throttling, batching, disabling, or broader gateway workload admission.
