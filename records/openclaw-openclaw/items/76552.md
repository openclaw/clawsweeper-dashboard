## Summary

Keep open. The report is recent, detailed, and still plausible on current main: Codex native tool hooks still shell through per-event `openclaw hooks relay` commands, relay invocation still dispatches without visible per-relay concurrency admission, and the session/history work has bounded mitigations but no proof that the combined low-core CPU saturation path is fixed.

## What This Changes

The issue reports CPU saturation on low-core Linux hosts during native Codex runtime tasks, with suspected amplification from hook relay subprocesses plus gateway session, history, status, and log work.

## Best Possible Solution

Profile the native Codex low-core path, then add measured default backpressure or a cheaper relay execution path for high-frequency native hook events while preserving plugin hook semantics and session/history response bounds.

## Reproduction Assessment

Unclear for the full load failure. The issue provides concrete low-core native Codex evidence and current source confirms the named relay plus session/history paths, but I did not run a 2-vCPU Codex stress test that reproduces sustained high load on current main.

## Solution Assessment

Unclear. Backpressure or a cheaper relay path is likely the right direction, but maintainers need profile data before choosing event-specific throttling, batching, disabling, or broader gateway workload admission.
