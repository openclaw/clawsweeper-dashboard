## Summary

Keep open. The report is recent, detailed, and still plausible on current main: the native Codex route still injects per-event command hooks for `PostToolUse`, while session/history work only has partial responsiveness mitigations and no proof that this low-core CPU saturation path is fixed.

## What This Changes

Reduce high CPU and load average during native Codex runtime tasks where hook relay fanout overlaps with Gateway session, history, status, and log work.

## Best Possible Solution

Profile the native Codex low-core path, then add measured relay admission or a cheaper relay execution path that preserves documented hook semantics and existing session/history bounds.

## Reproduction Assessment

Unclear for a full failure reproduction. The issue provides concrete 2026.5.3 A/B evidence and current source confirms the named native hook relay path, but I did not run a low-core current-main stress/profile scenario.

## Solution Assessment

Unclear. The likely fix belongs around native Codex relay admission, process reuse, or cheaper dispatch, but maintainers need profile data before choosing a default policy that might affect synchronous hook behavior.
