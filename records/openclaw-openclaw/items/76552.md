## Summary

Keep open. The report is recent, detailed, and still plausible on current main: the Codex native hook relay still injects per-event `openclaw hooks relay` command hooks, while session/history RPCs only have partial lightweight/bounded mitigations and no proof that this low-core CPU saturation path is fixed.

## What This Changes

The issue reports high CPU/load average on a 2-vCPU Linux host during native Codex runtime tasks, with hook relay work and gateway session/history/status/log activity amplifying event-loop delay.

## Best Possible Solution

Profile this path on a low-core Linux host, then add measured default backpressure or cheaper relay execution for Codex native hook events while keeping existing session/history bounds and plugin hook semantics intact.

## Reproduction Assessment

Unclear for a full failure reproduction: the issue provides concrete load, vmstat, process-tree, and liveness evidence, and current source confirms the named hook relay and session/history paths exist. I did not establish a deterministic current-main low-core Codex stress run that reproduces the high load average.

## Solution Assessment

Unclear. The likely direction is narrower runtime backpressure or a cheaper in-process/pooled relay path, but maintainers need profile data before choosing between event-specific throttling, batching, disabling, or broader gateway workload admission.
