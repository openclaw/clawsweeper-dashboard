## Summary

Keep open. The report is recent, detailed, and still matches current main: the native Codex runtime still installs synchronous command hooks for `PostToolUse`, and I did not find per-relay concurrency admission that would prove the low-core CPU fanout path is fixed.

## What This Changes

The issue reports CPU saturation on low-core Linux hosts when OpenClaw runs the native Codex runtime path with hook relay and concurrent gateway session/history work.

## Best Possible Solution

Profile low-core native Codex turns, then add measured default relay admission/backpressure or a cheaper relay execution path while preserving Codex hook contracts and existing session/history bounds.

## Reproduction Assessment

Unclear for a full failure reproduction. The issue gives concrete low-core A/B samples on 2026.5.3 and current main still has the same `post_tool_use` command relay path, but I did not live-run a current-main 2-vCPU stress scenario.

## Solution Assessment

Unclear. The likely direction is narrower relay backpressure, batching, or a pooled/in-process relay path, but maintainers need profile data before choosing a policy that affects native hook timing and semantics.
