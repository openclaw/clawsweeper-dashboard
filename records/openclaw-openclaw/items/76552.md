## Summary

Keep open. The issue is recent and the current source still matches the reported native Codex path: OpenClaw installs synchronous Codex command hooks for tool events, each event enters through the `openclaw hooks relay` CLI, and I did not find relay-wide concurrency/backpressure around `post_tool_use` bursts.

## What This Changes

Investigate high CPU/load average on low-core hosts when native Codex runtime hook relay work runs alongside gateway session/history/status activity.

## Best Possible Solution

Profile native Codex hook bursts on a 2-vCPU Linux host, then add measured default backpressure or cheaper relay handling for high-frequency `post_tool_use` events while preserving plugin hook semantics.

## Reproduction Assessment

Unclear for a full current-main failure reproduction. Source inspection confirms the named native Codex relay path and the issue provides strong A/B process evidence on 2026.5.3, but I did not run the low-core stress scenario myself.

## Solution Assessment

Unclear. Backpressure or a cheaper relay path is the plausible direction, but the safest fix depends on profile data and a maintainer decision about Codex hook delivery semantics.
