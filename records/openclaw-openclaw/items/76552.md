## Summary

Keep open. The report is recent, detailed, and still matches current `main`: Codex native turns still install synchronous command hooks for `post_tool_use`, the hidden CLI still spawns `openclaw hooks relay` per event, and session/history mitigations are partial rather than proof that this low-core CPU saturation path is fixed.

## What This Changes

The issue asks OpenClaw to reduce CPU saturation from native Codex hook relay fanout and concurrent gateway session/history/status work on low-core Linux hosts.

## Best Possible Solution

Profile the native Codex path on a low-core Linux host, then add measured default backpressure or cheaper relay execution for post-tool events while preserving plugin hook semantics and keeping sessions/history work bounded.

## Reproduction Assessment

Unclear for the full CPU/load failure, because I did not run a current-main low-core Codex stress scenario. Source inspection does confirm the reported trigger path: native Codex `post_tool_use` events still run synchronous hidden `openclaw hooks relay` commands, and the reporter supplied strong A/B host evidence.

## Solution Assessment

Unclear. Backpressure or a cheaper in-process/pooled relay is a plausible narrow fix, but maintainers need profile data before choosing a default policy that could affect hook timing, approvals, and plugin observations.
