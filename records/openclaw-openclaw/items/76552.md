## Summary

Keep open. The report is recent and the current main source still has the named native Codex hook path: synchronous command hooks invoke `openclaw hooks relay` per event, and the direct bridge reduces gateway RPC overhead but does not remove hook process fanout or add post-tool admission/backpressure.

## What This Changes

Report requests reducing CPU fanout and gateway liveness impact when native Codex runtime hook relays run alongside session/history/status work.

## Best Possible Solution

Profile the native Codex route on a low-core Linux host, then add measured default backpressure or a cheaper relay execution path that preserves existing plugin hook semantics.

## Reproduction Assessment

No, not for an end-to-end current-main high-CPU run in this review. Source inspection does confirm the reporter’s trigger path still exists: synchronous native Codex command hooks spawn `openclaw hooks relay` and dispatch post-tool hook work without a visible concurrency gate.

## Solution Assessment

Unclear. The current direct relay bridge is a useful mitigation, but it does not remove command-process fanout or define the backpressure policy needed to safely solve the low-core saturation path.
