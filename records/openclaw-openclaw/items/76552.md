## Summary

Keep open. The issue is recent, detailed, and still plausible on current main: Codex native runtime still injects synchronous command hooks that invoke `openclaw hooks relay`, while session/history paths have bounded/yielding mitigations but no proof that the combined low-core CPU saturation path is fixed.

## What This Changes

The issue asks OpenClaw to reduce high CPU/load average during native Codex runtime tasks by adding measured backpressure or cheaper execution around hook relay and concurrent session/history work.

## Best Possible Solution

Profile the low-core native Codex path, then add measured default backpressure or cheaper relay execution while preserving documented hook semantics and the existing session/history bounds.

## Reproduction Assessment

Unclear for a full failure reproduction. The issue provides concrete load, vmstat, process-tree, and liveness data, and current source confirms the named relay/session paths exist, but I did not establish a deterministic current-main low-core stress run.

## Solution Assessment

Unclear. Backpressure or a cheaper pooled/in-process relay path is likely directionally right, but the best fix depends on profiling and a maintainer policy choice about default hook concurrency versus latency.
