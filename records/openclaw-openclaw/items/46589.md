## Summary

Keep open: current main still lacks the proposed cron success/failure chaining and the narrow Control UI i18n fixes, but this PR needs rebase plus maintainer product/API review before merge because the chaining design overlaps Task Flow and has remaining source-review blockers.

## What This Changes

The PR adds cron UI i18n/test-registration fixes and introduces success/failure cron job chaining through cron types, gateway schema/generated clients, runtime dispatch, and Control UI form fields.

## Best Possible Solution

Land the narrow Control UI i18n/test fixes with generated locale artifacts and a changelog entry, then handle orchestration through Task Flow or a centralized cron post-run dispatcher that covers scheduled, queued manual, and direct service entry points with protocol, docs, and tests updated together.

## Reproduction Assessment

Yes, from source inspection rather than execution. At PR head, a configured chain invoked through CronService.run/ops.run returns after finishPreparedManualRun while chain dispatch is only added around enqueueRun and timer execution paths.

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because post-run behavior is split across entry points and the documented durable orchestration surface is Task Flow.
