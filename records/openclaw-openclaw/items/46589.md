## Summary

Keep open; current main still lacks the cron chaining contract, and the PR remains useful but not merge-ready because its chaining behavior is inconsistent across cron run entry points and its Control UI locale updates are incomplete for the current i18n pipeline.

## What This Changes

The PR adds Control UI cron i18n/test-registration fixes plus success/failure cron job chaining across cron runtime, gateway protocol/generated clients, and the Control UI form.

## Best Possible Solution

Split or land the narrow Control UI i18n fixes with regenerated locale artifacts, then handle orchestration through Task Flow or a centralized cron post-run dispatcher that covers scheduled, queued manual, and direct service runs with protocol, docs, tests, and changelog updated together.

## Reproduction Assessment

Yes. At PR head, configure onSuccessJobId or onFailureJobId and invoke CronService.run()/ops.run() directly; that path returns after finishPreparedManualRun while chain dispatch is only added around enqueueRun and timer execution paths.

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because post-run behavior is split across entry points and the documented multi-step orchestration surface is Task Flow.
