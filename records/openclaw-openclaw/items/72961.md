## Summary

Keep this PR open: the direction is useful, but the current patch does not actually make `HTTPS_PROXY`/`HTTP_PROXY` drive a `wss://` QQ gateway connection because `ProxyAgent` delegates WSS proxy lookup to `wss_proxy`/`ALL_PROXY`, and the PR also lacks the required user-facing changelog entry.

## What This Changes

The PR adds a QQBot WebSocket helper that uses the shared ambient proxy agent, switches gateway connection creation to that helper, and adds proxy/no-proxy regression tests.

## Best Possible Solution

Make QQBot's WebSocket proxy path actually map secure WebSocket URLs to the intended HTTP(S) proxy env vars, cover that dependency behavior in tests, and add one Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes for the code-level failure: dependency source shows `ProxyAgent` resolves `wss:` via `wss_proxy`/`ALL_PROXY`, while the PR's claimed and tested env vars are `HTTPS_PROXY`/`HTTP_PROXY`. Full network reproduction still needs the private QQ egress environment from the PR body.

## Solution Assessment

No. The PR uses the right QQBot-local boundary, but the current agent construction is not the effective dependency-level fix; a safer repair is an opt-in WebSocket proxy mapping through the SDK helper or QQBot wrapper without changing Feishu behavior by accident.
