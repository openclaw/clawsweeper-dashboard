## Summary

Keep this PR open: the patch appears to be a narrow, correct follow-up for a current-main session-origin regression, but it still needs contributor real behavior proof before merge.

## What This Changes

The PR changes session origin merging so internal non-delivery turns preserve a channel-bound session’s native channel identity while real channel switches still reset channel-keyed fields.

## Best Possible Solution

Land this scoped session-origin fix after the contributor adds redacted real runtime proof, or after a maintainer explicitly accepts test-only proof for this pure metadata regression.

## Reproduction Assessment

Yes, at source level. Current main's mergeOrigin clears channel-keyed origin fields on provider/surface/account changes, and chat.send can feed an internal webchat-origin turn through the same metadata path; I did not run tests because this review is read-only.

## Solution Assessment

Yes, this appears to be the best code location: mergeOrigin is the shared metadata boundary, and gating the reset there protects recordSessionMetaFromInbound and updateLastRoute callers together. Passing more skip flags through individual callers would be narrower but would leave sibling internal-turn paths easier to miss.
