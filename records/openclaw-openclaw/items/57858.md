## Summary

Keep this issue open: current main and the latest release still document a bare Claude Code launch and the managed Claude CLI clearEnv list still omits the nesting markers called out by the report. The linked fix PR is the right active path, but it is still open and conflicting, so the issue should remain open until that PR or a narrow replacement lands.

## What This Changes

The issue asks OpenClaw to stop Claude Code worker dispatches from inheriting parent Claude Code nesting environment variables when launched by the bundled coding-agent skill or managed Claude CLI backend.

## Best Possible Solution

Rebase and review https://github.com/openclaw/openclaw/pull/87702, or replace it with the same focused fix across the skill recipe, Claude backend clearEnv list, and parity tests, then close this issue after merge.

## Reproduction Assessment

Yes for a source-level reproduction: current main's skill command lacks the env unsets and the managed backend deletes only clearEnv keys, which omit `CLAUDECODE` and `CLAUDE_CODE_SESSION_ID`. I did not run a live nested Claude Code binary in this read-only review, but the issue discussion adds a recent 2026.6.1 operator reproduction.

## Solution Assessment

Yes, the narrowest maintainable fix is to clear the Claude Code nesting markers at the coding-agent recipe and Claude CLI backend clearEnv boundary. The isolated cron `agentTurn` path is a useful workaround and possible docs follow-up, but it does not replace the root env-strip fix for the documented launch form.
