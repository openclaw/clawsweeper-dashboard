## Summary

Keep open. Current main still exposes the reported failure mode: the bundled coding-agent skill tells agents to run Claude Code directly through the raw bash path, while the host exec sanitizer does not drop the reported Claude nesting variables and the separate Claude CLI backend clearEnv hardening does not apply to that raw skill command.

## What This Changes

The issue requests that bundled coding-agent Claude Code launch guidance unset `CLAUDECODE`, `CLAUDE_CODE_SESSION`, and `CLAUDE_CODE_ENTRYPOINT` before spawning the `claude` CLI.

## Best Possible Solution

Patch the bundled coding-agent skill so every Claude Code launch template scrubs the three nesting variables before invoking `claude`, and add a focused static bundled-skill guidance test to prevent the examples from regressing.

## Reproduction Assessment

Yes for the OpenClaw side: enable the bundled coding-agent skill, run the documented Claude Code `bash background:true` command from an environment containing the reported Claude nesting variables, and current source shows those variables flow through the raw bash env. The exact Claude CLI silent exit is based on reporter evidence rather than a live CLI run here.

## Solution Assessment

Yes, the requested skill guidance fix is the narrowest maintainable first fix because the broken surface is a bundled SKILL.md example, not the managed Claude CLI backend. A broader host-exec scrub could be considered separately but should not block the bundled-skill repair.
