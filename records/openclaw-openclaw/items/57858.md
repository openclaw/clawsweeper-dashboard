## Summary

Keep open: current main and the latest release still tell the bundled coding-agent skill to launch Claude Code directly without unsetting the reported nesting variables, and the existing Claude CLI backend clear-env path does not cover the raw bash command that the skill instructs agents to run.

## What This Changes

Patch the bundled coding-agent skill so Claude Code background dispatch unsets Claude nesting guard environment variables before invoking `claude --print`.

## Best Possible Solution

Update the bundled skill’s Claude Code command templates and rules to unset the three reported nesting variables, then add a focused static guidance test so future edits cannot regress it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the bundled skill’s documented Claude command from an OpenClaw process that has `CLAUDECODE`, `CLAUDE_CODE_SESSION`, or `CLAUDE_CODE_ENTRYPOINT` set; the raw bash path currently inherits `process.env` and the skill does not unset them.

## Solution Assessment

Yes. The narrowest maintainable fix is to correct the bundled skill guidance and test it; a generic host-exec sanitizer for these Claude variables is a plausible defense-in-depth follow-up but broader than the reported broken skill contract.
