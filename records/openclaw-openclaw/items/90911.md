## Label Justifications

- `P2`: This is a well-supported observability improvement with persistent operational impact but no core availability or data-loss failure.
- `impact:other`: Operators cannot reliably attribute local token or cost consumption to individual hook-driven and subagent runs.

## Summary

The requested observability capability remains absent from current main and the latest release, with no duplicate or implementation PR owning it. The issue should remain open for a maintainer decision because the suggested session-store shortcut measures prompt context rather than complete per-run token consumption.

## What This Changes

Persist per-run token and optional cost totals for task and subagent runs, analogous to cron run logs.

## Best Possible Solution

Carry normalized cumulative execution usage into the canonical task and subagent terminalization paths, then persist clearly defined token and optional cost fields through the shared SQLite schema, generated types, stores, and upgrade migration.

## Reproduction Assessment

Yes. Current main and `v2026.6.6` both expose cron token totals while omitting equivalent task and subagent run fields, so the parity gap is directly reproducible from the source schema.

## Solution Assessment

No, not exactly as proposed in the follow-up comment. The additive storage direction is reasonable, but actual cumulative execution usage must be plumbed into terminalization because session `totalTokens` intentionally excludes output tokens.
