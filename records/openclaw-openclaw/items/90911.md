## Summary

Keep open: current main still has the reported parity gap, with `cron_run_logs.total_tokens` persisted but no token/cost columns on `task_runs` or `subagent_runs`; the remaining decision is the exact persisted metric semantics and source.

## What This Changes

The issue requests nullable per-run token and optional cost accounting on `task_runs` and `subagent_runs`, matching cron run-log observability for hook-driven and sub-agent runs.

## Best Possible Solution

Add explicit run-usage fields for task and subagent records only after maintainers choose the token/cost schema and whether the source is provider usage telemetry or context snapshot data.

## Reproduction Assessment

Yes. Source inspection shows the current schema and generated types persist `cron_run_logs.total_tokens` but expose no equivalent token/cost fields on `task_runs` or `subagent_runs`; I did not run a live hook/subagent scenario in this read-only review.

## Solution Assessment

Unclear. Nullable columns are a plausible direction, but the proposed session-store source needs maintainer approval because its `totalTokens` semantics differ from cron run-log aggregate token telemetry.
