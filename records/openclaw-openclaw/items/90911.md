## Summary

Keep open: current main and the latest release still have the parity gap, but the right token/cost semantics need maintainer direction before implementation.

## What This Changes

Persist token and optional cost totals on `task_runs` and `subagent_runs` for hook-driven and subagent run accounting parity with `cron_run_logs`.

## Best Possible Solution

Define the canonical per-run accounting contract, then add nullable `total_tokens` and any approved cost column to `task_runs` and `subagent_runs` with migrations, mappings, finalize-path population, and regression coverage.

## Reproduction Assessment

Yes, at source level: current schema/types/stores show `cron_run_logs.total_tokens` exists while `task_runs` and `subagent_runs` have no token or cost fields. I did not run a live hook or subagent model turn because the DB gap is directly visible in current main and the latest release.

## Solution Assessment

Unclear as written: adding nullable columns is a plausible storage shape, but maintainers should first choose aggregate provider usage versus prompt/context snapshot semantics and whether `cost_total` belongs in the same change.
