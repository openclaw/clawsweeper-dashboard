## Summary

Keep open: current main and the latest release still persist indexed token totals only for cron run logs, while task and subagent run tables lack equivalent token/cost fields; the remaining blocker is a maintainer decision on the durable accounting contract.

## What This Changes

Request to persist per-run token usage, and possibly cost, on `task_runs` and `subagent_runs` for parity with `cron_run_logs.total_tokens`.

## Best Possible Solution

Maintainers should define the canonical per-run accounting contract, then add nullable SQLite columns, migration guards, generated types, store mappings, finalization-time population, and focused regression coverage for task and subagent runs.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.6` reproduces the parity gap: cron run logs expose `total_tokens`, while task and subagent run schemas, generated types, and store mappings lack token or cost fields.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible additive shape, but the best fix depends on maintainer choice for token-vs-cost persistence and whether run totals should come from aggregate telemetry or session snapshots.
