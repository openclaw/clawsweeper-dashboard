## Summary

Current main still lacks run-level token fields for task and subagent records, and no canonical issue or merged PR owns the work. The request is valid and well evidenced, but it needs maintainer judgment on the persisted accounting contract because execution-time run usage is safer than reading a later session snapshot, and storing cost would create a separate historical-pricing contract.

## What This Changes

Persist per-run token usage, and optionally historical cost, on task_runs and subagent_runs for direct local database attribution.

## Best Possible Solution

Define a normalized run-usage payload at the execution boundary, pass it through terminal task and subagent lifecycle updates, persist total tokens consistently in both registries, and add cost only if maintainers explicitly choose immutable historical cost snapshots.

## Reproduction Assessment

Yes. Current-main schema, stores, and cron accounting directly reproduce the missing capability without needing a live model call.

## Solution Assessment

The requested capability is appropriate, but the suggested session-store lookup is not the best implementation; explicit execution-time usage plumbing is the narrower and more reliable accounting boundary.

## Risks / Open Questions

- Reading totalTokens from the session store at finalization could capture stale or overwritten state and does not preserve the execution-time freshness and usage context already available in the run pipeline.
- A persisted cost_total value would be an immutable historical pricing snapshot; without an explicit contract it could disagree with later transcript-derived cost when model pricing metadata changes.
- Task and subagent records have distinct lifecycle owners, so a one-sided or duplicated implementation could produce inconsistent accounting across run types.
