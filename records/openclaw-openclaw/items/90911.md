## Summary

Keep open: current main and the latest release still persist indexed token totals only for cron run logs, while task and subagent run tables lack durable token/cost accounting; the remaining blocker is a maintainer decision on the permanent run-accounting contract.

## What This Changes

Request to persist per-run token usage, and optionally cost, on task_runs and subagent_runs so hook-driven and subagent executions have queryable local-state accounting like cron_run_logs.total_tokens.

## Best Possible Solution

Choose the canonical per-run accounting semantics, then add additive SQLite columns, migration guards, generated types, store mappings, finalization-time population, and regression coverage for both fresh and upgraded databases.

## Reproduction Assessment

Yes for the source-level gap: current main defines cron_run_logs.total_tokens but no equivalent task_runs or subagent_runs token/cost columns, generated types, or store mappings. I did not run a live hook/subagent model flow because the missing persistent fields are directly verifiable from schema and store code.

## Solution Assessment

Unclear until maintainers choose the contract: nullable run-table columns are a plausible additive shape, but token-only versus token-plus-cost and session-derived versus explicitly-plumbed usage need ownership before implementation.
