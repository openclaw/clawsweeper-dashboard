## Summary

Keep open: current main and v2026.6.6 still have the reported state-DB parity gap, and the remaining work is a schema/accounting product decision rather than a strict bug fix.

## What This Changes

Persist run-level token and optional cost totals on task_runs and subagent_runs so hook-driven and sub-agent runs are queryable like cron_run_logs.

## Best Possible Solution

Add an additive, tested SQLite migration and store/finalize plumbing for the accepted run-level accounting fields after maintainers choose the canonical usage source and cost scope.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.6 expose cron_run_logs.total_tokens while task_runs and subagent_runs lack equivalent token/cost columns and store mappings.

## Solution Assessment

Unclear until maintainers decide the accounting contract. The additive column approach is plausible, but the best fix depends on choosing total_tokens-only versus cost_total and whether finalizers read session state or receive usage explicitly.
