## Summary

Keep this open: current main and the latest release still expose cron run token totals but do not persist equivalent token or cost fields for task or subagent run rows. The requested parity is valid, but the exact durable metric and whether cost becomes part of the schema need maintainer direction before implementation.

## What This Changes

The issue requests persisted token and optional cost accounting on task and subagent run records, matching cron run token accounting.

## Best Possible Solution

Define one canonical run-usage metric, add nullable typed SQLite columns and migrations for `task_runs` and `subagent_runs`, populate them from run completion usage rather than timestamp correlation, and cover cron/task/subagent parity with focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main has `cron_run_logs.total_tokens` but no equivalent token/cost columns or store mappings for `task_runs` and `subagent_runs`. I did not run a live SQLite/model scenario because this review is read-only.

## Solution Assessment

No, the implementation direction is not fully settled. Nullable columns are plausible, but sourcing from `SessionEntry.totalTokens` would not match cron aggregate usage unless maintainers intentionally choose prompt/context tokens instead of full run tokens.
