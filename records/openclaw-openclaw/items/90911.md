## Summary

Keep open: current main and the latest release still have the reported parity gap, with token totals indexed for cron runs but no token or cost columns on task or subagent run tables; the remaining work is a durable schema/accounting decision, not conservative cleanup.

## What This Changes

Request to persist per-run token, and optionally cost, totals on `task_runs` and `subagent_runs` for hook-driven and subagent runs.

## Best Possible Solution

Choose the canonical per-run accounting contract, then implement additive SQLite columns, migration guards, generated types, store mappings, finalize-time population, and focused regression coverage for both task and subagent runs.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: current schema, generated types, and stores expose `cron_run_logs.total_tokens` but no equivalent token or cost fields on `task_runs` or `subagent_runs`.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible maintainable direction, but maintainers still need to decide the permanent accounting fields and which runtime path owns the finalized value.
