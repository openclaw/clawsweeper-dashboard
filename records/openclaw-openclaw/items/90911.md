## Summary

Current main confirms the reported parity gap: cron run logs persist `total_tokens`, while task and subagent run tables, generated DB types, and store binders do not expose token or cost totals. This should stay open as a core run-accounting feature because it affects durable shared SQLite state and operator observability, not an external ClawHub-only integration.

## What This Changes

The issue requests nullable token and optional cost accounting columns on `task_runs` and `subagent_runs`, populated like `cron_run_logs.total_tokens`.

## Best Possible Solution

Add a narrow core run-accounting surface for `total_tokens` on both run tables, with generated schema/types, additive column handling, and focused hook/subagent coverage; handle cost only after maintainers settle the durable semantics.

## Reproduction Assessment

Yes, at source level: current main shows `cron_run_logs.total_tokens` but no token or cost columns on `task_runs` or `subagent_runs`, and the corresponding stores do not bind usage fields. I did not run a live DB reproduction because this review is read-only.

## Solution Assessment

Unclear until maintainers settle the schema contract: additive `total_tokens` parity looks like the narrowest maintainable path, while `cost_total` needs explicit semantics before becoming durable state. Timestamp correlation against transcript JSONL is not the best long-term solution for concurrent runs.
