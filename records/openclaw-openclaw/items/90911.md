## Summary

Keep open: current main and v2026.6.6 still only persist indexed token totals for cron run logs, while task and subagent run-table accounting remains missing and needs a maintainer schema/source-of-truth decision.

## What This Changes

Request to persist per-run token and optionally cost accounting on `task_runs` and `subagent_runs` for parity with `cron_run_logs.total_tokens`.

## Best Possible Solution

Maintainers should choose and document the canonical per-run accounting contract, then implement additive SQLite columns, migration guards, generated types, store mappings, finalization-time population, and focused regression coverage for task and subagent runs.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: current main and v2026.6.6 expose `cron_run_logs.total_tokens`, but `task_runs` and `subagent_runs` lack equivalent token or cost fields in schema, generated types, and persistence mappings.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible additive direction, but the best fix depends on a maintainer decision about cost persistence and whether finalization should read session totals or receive usage explicitly.
