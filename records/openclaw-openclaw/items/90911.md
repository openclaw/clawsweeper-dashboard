## Summary

Keep open: current main and the latest inspected release still have the reported state-DB parity gap, while the remaining choice is a product/schema decision rather than a strict bug fix.

## What This Changes

Add per-run token and optional cost columns/population for `task_runs` and `subagent_runs` so hook and subagent runs have state-DB usage parity with `cron_run_logs`.

## Best Possible Solution

Maintainers should choose the canonical per-run accounting contract, then implement additive SQLite columns, migration guards, generated types, store mappings, finalize-time population, and focused regression coverage for task and subagent runs.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: current schema/types and stores expose `cron_run_logs.total_tokens` but no equivalent token or cost fields on `task_runs` or `subagent_runs`.

## Solution Assessment

Unclear. Nullable `total_tokens` on the existing run tables is a plausible core-state solution, but maintainers still need to decide whether cost belongs there and which runtime source should own the finalized value.
