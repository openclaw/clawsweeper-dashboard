## Summary

Keep open: current main and the latest release still persist indexed token totals only for cron run logs, while task and subagent run records lack equivalent token/cost fields and need a maintainer decision on durable accounting semantics.

## What This Changes

The issue requests durable token and optional cost accounting fields on task and subagent run records to match cron run token totals.

## Best Possible Solution

Define the canonical run-accounting contract first, then implement additive SQLite columns, migration guards, generated types, store mappings, finalization-time population, and focused regression coverage for both task and subagent runs.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.9 shows cron_run_logs.total_tokens exists while task_runs, subagent_runs, generated types, migration guards, and store mappings lack token or cost fields.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible additive path, but maintainers still need to choose cost persistence and whether values come from session snapshots or explicitly plumbed aggregate run usage.
