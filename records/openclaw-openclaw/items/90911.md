## Summary

Keep this open: current main and the latest stable release still expose `cron_run_logs.total_tokens` but do not expose equivalent token/cost columns on `task_runs` or `subagent_runs`, and the remaining decision is the product/data-contract shape for persisted run accounting.

## What This Changes

Request to persist per-run token usage, and optionally cost, on `task_runs` and `subagent_runs` for parity with `cron_run_logs.total_tokens`.

## Best Possible Solution

Add a canonical nullable run-accounting shape to the shared state schema, generated Kysely types, migrations, task/subagent record mappings, and finalize paths after maintainers choose the exact token/cost contract.

## Reproduction Assessment

Yes, by source inspection: current main has `cron_run_logs.total_tokens` but no token/cost fields in `task_runs`, `subagent_runs`, or their typed record mappings. I did not run a live hook/subagent scenario because the schema gap itself reproduces the reported missing query surface.

## Solution Assessment

Unclear until maintainers choose the accounting contract. The proposed additive nullable columns are plausible, but the best fix should first settle `total_tokens` versus cost persistence and whether finalize-time session lookup is the canonical source.
