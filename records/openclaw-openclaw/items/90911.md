## Summary

Keep open: current main and the latest release still persist indexed token totals only for cron run logs, while `task_runs` and `subagent_runs` have no comparable token or cost fields. The source gap is real, but the permanent run-accounting contract needs maintainer choice before changing persistent schema.

## What This Changes

The issue requests nullable token/cost accounting fields on task and subagent run records to match cron run token totals.

## Best Possible Solution

Keep this issue open until maintainers choose the durable run-accounting semantics, then add one additive SQLite migration and focused task/subagent store plus finalize-path coverage.

## Reproduction Assessment

Yes at source level: current main defines `cron_run_logs.total_tokens` but does not define matching token/cost fields on `task_runs` or `subagent_runs`. I did not run a live DB migration because this review is read-only.

## Solution Assessment

Unclear: nullable columns plus additive migration are plausible mechanics, but maintainers need to decide the permanent token/cost semantics and whether finalize snapshots session totals or receives explicit executor usage.
