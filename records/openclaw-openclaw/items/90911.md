## Summary

Keep open: current main and latest release still persist indexed token totals only for cron run logs, while task and subagent run tables lack equivalent token/cost fields and the durable accounting source still needs maintainer direction.

## What This Changes

The issue requests nullable per-run token usage, and possibly cost, columns on `task_runs` and `subagent_runs` to match `cron_run_logs.total_tokens`.

## Best Possible Solution

Define the canonical run-accounting contract first, then implement additive SQLite columns, migration guards, generated types, store mappings, finalization-time population, and regression coverage for both task and subagent runs.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows `cron_run_logs.total_tokens` exists while `task_runs` and `subagent_runs` lack token or cost fields in schema, generated types, and store mappings.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible additive path, but maintainers still need to decide cost persistence and whether the value comes from session snapshots or explicitly plumbed aggregate run usage.
