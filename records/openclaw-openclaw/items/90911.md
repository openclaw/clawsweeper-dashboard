## Summary

Keep open. Current main and the latest release still have the parity gap: cron logs persist `total_tokens`, while `task_runs` and `subagent_runs` omit token/cost columns; the report is source-reproducible but needs a maintainer accounting-shape decision before implementation.

## What This Changes

Request to persist per-run token, and possibly cost, accounting on `task_runs` and `subagent_runs` so hook-driven and subagent runs have state-DB parity with `cron_run_logs.total_tokens`.

## Best Possible Solution

Adopt one core run-accounting contract for cron, task, and subagent ledgers, likely additive nullable token columns with migration and focused store/finalize tests after maintainers decide cost semantics.

## Reproduction Assessment

Yes, from source: current schema and generated DB types give `cron_run_logs.total_tokens` while `task_runs` and `subagent_runs` have no token or cost columns. I did not execute a live `/hooks/agent` run in this read-only review.

## Solution Assessment

Unclear until maintainers decide the accounting shape. Nullable `total_tokens` looks like the narrow parity fix, but `cost_total` and whether to snapshot session totals at finalize or plumb explicit run usage are product/accounting choices.
