## Summary

Keep open: current main and the latest release still have the reported SQLite run-table parity gap, and the remaining blocker is a maintainer schema/accounting decision rather than a narrow strict bug fix.

## What This Changes

Request to persist per-run token usage, and possibly cost, on `task_runs` and `subagent_runs` in parity with `cron_run_logs.total_tokens`.

## Best Possible Solution

Maintainers should choose the canonical per-run accounting contract, then implement additive SQLite columns, migration guards, generated types, store mappings, finalize-time population, and regression coverage for task and subagent runs.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: cron run logs expose and migrate `total_tokens`, while task and subagent run tables, generated types, and stores have no equivalent token or cost fields.

## Solution Assessment

Unclear. Nullable run-table columns are a plausible core-state solution, but maintainers still need to decide the permanent token/cost contract and whether the source should be aggregate run usage or the session prompt/context snapshot.
