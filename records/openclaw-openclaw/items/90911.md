## Summary

The request is still necessary on current main: cron run logs persist aggregate usage, while task and subagent run records do not. The contributor’s concrete schema direction is useful, but `SessionEntry.totalTokens` is the wrong source because it is explicitly a prompt/context snapshot that excludes output tokens; maintainers should first define the persisted metric and whether estimated cost belongs in the initial contract.

## What This Changes

Persist per-run token usage, and optionally estimated cost, on task_runs and subagent_runs for local accounting parity with cron runs.

## Best Possible Solution

Define a normalized run-usage contract, carry aggregate usage from executor results into both registries, and persist it atomically at terminal completion through an additive SQLite migration; add estimated cost only if maintainers explicitly want that pricing-derived field in canonical state.

## Reproduction Assessment

Yes. Current-main schema and accounting source provide a high-confidence source reproduction: cron persists aggregate tokens, while task and subagent records do not.

## Solution Assessment

No, not exactly as proposed in the follow-up comment. Nullable columns and migration are appropriate, but the value must come from explicit run-scoped aggregate usage rather than `SessionEntry.totalTokens`; maintainers also need to decide whether estimated cost is part of the first durable contract.
