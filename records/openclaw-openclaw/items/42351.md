Current main and v2026.6.5 still omit this Feishu event handler, so opening a bot private chat deterministically reaches the pinned SDK’s missing-handler warning path. The SDK only warns and returns from that path, however, so the issue remains useful as a narrow warning-noise bug rather than a proven Gateway restart defect; the linked PR was closed unmerged and is not a viable replacement.

## What This Changes

Register Feishu’s informational private-chat-entry event so opening a bot chat no longer emits the SDK’s missing-handler warning.

## Best Possible Solution

Add a plugin-local no-op registration for `im.chat.access_event.bot_p2p_chat_entered_v1` and focused coverage that verifies the event is present in the dispatcher map; investigate any reproducible restart separately using process-exit or supervisor logs.

## Reproduction Assessment

Yes for the warning: current main omits the event, the exact pinned SDK warns for missing handlers, and a user reproduced it on v2026.6.5. No high-confidence reproduction currently ties that warning to a Gateway restart.

## Solution Assessment

Yes, with narrowed scope: a plugin-local no-op handler and handler-map regression test are the smallest maintainable fix; a new config option or Gateway error-path change is unnecessary.
