## Summary

Current main and the latest release still lack the Feishu private-chat-entered handler, so the recurring missing-handler warning remains valid. Keep this issue open because clean open PR https://github.com/openclaw/openclaw/pull/92348 now explicitly owns the narrow fix; the reported Gateway restart remains correlated rather than proven, since the pinned SDK only warns and returns when a handler is absent.

## What This Changes

Register a Feishu handler for `im.chat.access_event.bot_p2p_chat_entered_v1` so opening a bot private chat does not produce an unhandled-event warning or associated disruption.

## Best Possible Solution

Keep this issue as the canonical report until https://github.com/openclaw/openclaw/pull/92348 is reviewed and resolved, land a focused plugin-local no-op registration with event-map coverage, and investigate any reproducible restart separately through Gateway process-exit or supervisor logs.

## Reproduction Assessment

Yes for the missing-handler warning: current main omits the exact event, the pinned SDK warns for unregistered types, and another user reproduced it on v2026.6.5. No high-confidence reproduction currently ties that warning to a Gateway restart.

## Solution Assessment

Yes for the verified warning bug: a no-op entry in the owning Feishu dispatcher is the narrowest maintainable fix and matches the existing ignored read-receipt pattern. Adding configuration or changing generic Gateway handling would introduce unnecessary surface, while restart claims need separate proof.
