## Summary

Keep open: current main and the latest release still do not register `im.chat.access_event.bot_p2p_chat_entered_v1`, while the pinned Lark SDK returns/logs a no-handle result for unregistered event types. The related fix PRs all closed unmerged, so this issue remains the canonical place for the remaining Feishu handler work.

## What This Changes

The issue asks OpenClaw to handle or ignore Feishu `bot_p2p_chat_entered_v1` events so opening a bot private chat does not create unhandled-event warnings or gateway instability.

## Best Possible Solution

Land a narrow Feishu plugin handler that consumes this access event as no-op informational input, add focused registration coverage, and avoid adding a config flag or core fallback for a plugin-owned event.

## Reproduction Assessment

Yes for the missing-handler warning path: current main creates a Lark EventDispatcher, registers no handler for the reported event, and the pinned SDK warns/returns a no-handle result for unregistered event types. The gateway restart part is not live-reproduced here.

## Solution Assessment

Yes for the narrow handler gap: adding a no-op handler in the Feishu plugin’s existing handler map is the best owner-boundary fix for the warning/log-noise behavior. If maintainers want to claim restart prevention, the PR should also include live Feishu WebSocket evidence or a restart-path trace.
