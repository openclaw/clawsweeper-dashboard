## Best Possible Solution

Land a focused Feishu plugin fix that registers a no-op handler next to the existing read-receipt ignore handler, with targeted handler-map or webhook coverage; investigate Gateway restarts separately if they continue after the warning is suppressed.

## Reproduction Assessment

No high-confidence reproduction currently proves that the warning itself causes Gateway restarts.

## Solution Assessment

Yes for the verified warning bug: a no-op entry in the owning Feishu dispatcher is the narrowest maintainable fix and matches the existing ignored read-receipt pattern. A config option or Gateway-wide fallback would add unnecessary surface for this event-specific issue.
