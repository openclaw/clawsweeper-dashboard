## Summary

Keep open. Current main still routes Feishu events through a fixed dispatcher that does not register `im.chat.access_event.bot_p2p_chat_entered_v1`, and the only related implementation PR is closed unmerged. The missing-handler warning is well supported; the exact Gateway restart mechanism still needs live or replay proof before changing generic Gateway supervision.

## What This Changes

The issue requests that Feishu private-chat-entered events be ignored or handled so opening a bot DM does not produce missing-handler warnings or destabilize the Gateway.

## Best Possible Solution

Add a Feishu-owned no-op or low-noise handler for `im.chat.access_event.bot_p2p_chat_entered_v1` in the monitor dispatcher, with focused regression coverage that the handler is registered on the WebSocket path and does not produce the SDK missing-handler response. Treat any broader restart behavior as a separate follow-up only after reproducing the restart mechanism.

## Reproduction Assessment

Partial yes: the missing-handler condition is reproducible from current main by inspecting the registered Feishu dispatcher keys and the SDK's missing-handler response path. The end-to-end Gateway restart still lacks a high-confidence local reproduction without live Feishu/Lark WebSocket credentials or a replay harness.

## Solution Assessment

Yes: the narrowest maintainable fix is to register a Feishu-specific informational handler and cover it in Feishu monitor tests. A generic Gateway restart change would be premature until the restart cause is independently reproduced.
