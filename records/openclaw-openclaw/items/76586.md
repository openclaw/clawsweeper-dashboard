## Summary

Keep open. Current main has partial fixes for the 5.2 plugin externalization fallout, but source inspection still shows a live upgrade recovery gap: update-time doctor defers configured plugin installs, post-core plugin sync skips invalid configs, and `plugins install` remains blocked for the Brave invalid-config case.

## What This Changes

The issue reports a broken 2026.4.29 to 2026.5.2 upgrade path when externalized plugins referenced in config are missing from OpenClaw’s managed plugin install store.

## Best Possible Solution

Make the 5.2 externalized-plugin upgrade repair run to completion for configured official plugins before the config-valid plugin sync gate, and keep install-time invalid-config recovery narrowly scoped to the requested official plugin id.

## Reproduction Assessment

Yes, from source inspection. A config that selects `tools.web.search.provider: "brave"` without an active Brave plugin becomes invalid, package-update doctor defers the missing install under `OPENCLAW_UPDATE_IN_PROGRESS`, and post-core plugin sync then skips because the config is invalid.

## Solution Assessment

No, current main is not yet the best solution. The safer repair is a narrow upgrade recovery path that installs or explicitly carries forward configured official plugin candidates instead of marking the update repair complete while the config remains invalid.

## Repair Work Prompt

Fix the 2026.5.2 externalized plugin upgrade deadlock tracked by https://github.com/openclaw/openclaw/issues/76586. Current main partially fixes official bare-id installs, but a pre-5.2 config that actively uses externalized plugins can still become invalid after update: `tools.web.search.provider: "brave"` fails validation when Brave is missing; package-update doctor runs with `OPENCLAW_UPDATE_IN_PROGRESS=1` and defers configured plugin ids; post-core plugin update then skips because config is invalid; and `openclaw plugins install brave` can still be blocked before the alias branch runs. Implement a narrow recovery path for configured official external plugin installs during update/doctor without broadly bypassing unrelated invalid configs. Preserve managed plugin roots under `~/.openclaw/npm`; do not teach OpenClaw to rely on global `npm install -g` plugin locations. Add regression coverage for Brave web-search provider config, channel/plugin entries such as Discord and voice-call with no install records, update-in-progress deferral not marking the release repair complete, and bare official plugin id install recovery when the only invalid issue is the requested installable provider/channel missing. Add a user-facing CHANGELOG.md fix entry. Stop and ask if the fix would require a new CLI flag, new config option, or broad product policy change.
