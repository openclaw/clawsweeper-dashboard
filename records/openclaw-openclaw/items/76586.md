## Summary

Keep open. Current main has useful catalog and repair pieces, but the reported recovery path still has a real source-level gap: the 2026.5.2 configured-plugin repair is skipped when the source config is already invalid from a missing web-search provider such as Brave, and `plugins install` invalid-config bypass is still limited to explicitly marked recovery cases.

## What This Changes

Ensure upgrades from 2026.4.29 to 2026.5.2 recover configs that reference newly externalized official plugins without trapping users in invalid-config or downgrade loops.

## Best Possible Solution

Make doctor/update repair configured external plugin installs before strict provider/plugin availability validation can block them, then add only the minimal official-plugin install bypass or docs needed for the recovery path to be coherent.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: a config with `tools.web.search.provider: "brave"` and no loaded Brave plugin becomes invalid, and the doctor release repair returns early on invalid source config. I did not live-run the npm upgrade path in this read-only review.

## Solution Assessment

No, current main is not yet the best solution because it has the repair helper but still gates it behind source-config validity. The narrow maintainable fix is to let this specific configured external-plugin repair run from the best-effort parsed config before final strict validation, with regression coverage for Brave plus a channel plugin such as Discord.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76586 as a focused upgrade-recovery bug. Current main can collect/install missing configured external plugins, but the 2026.5.2 release repair is skipped when config validation is already failing for a missing external provider such as `tools.web.search.provider: "brave"`. Make `openclaw doctor --fix` and the post-update doctor path run the configured external plugin install repair from the best-effort parsed/source config before strict plugin/provider availability blocks it. Add regression coverage for Brave web-search provider-only config and at least one configured external channel such as Discord/voice-call. If a `plugins install` bypass is needed, keep it limited to official external recovery metadata; do not teach OpenClaw to use arbitrary global npm locations as plugin roots. Update docs/changelog if user-facing behavior changes.
