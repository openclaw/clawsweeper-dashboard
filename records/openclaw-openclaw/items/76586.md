## Summary

Keep open. Current main still has a live upgrade-recovery gap: update-time doctor can defer configured plugin installs, post-core plugin sync skips invalid configs, and manual `plugins install` still fails closed for the Brave invalid-config case unless the plugin explicitly opts into the narrow recovery path.

## What This Changes

The issue reports a broken 2026.4.29 to 2026.5.2 upgrade path when config references externalized plugins such as Brave, Discord, voice-call, or WhatsApp.

## Best Possible Solution

Repair the 5.2 externalized-plugin upgrade path so configured official external plugins are installed or recovered before invalid config prevents the post-update path, then track unrelated WhatsApp runtime and downgrade-message refinements separately if needed.

## Reproduction Assessment

Yes, from source inspection. A config selecting `tools.web.search.provider: "brave"` without an active Brave plugin becomes invalid, update doctor defers configured plugin installs under `OPENCLAW_UPDATE_IN_PROGRESS`, and post-core plugin sync then skips because the config is invalid.

## Solution Assessment

No, current main is not yet the best solution. The maintainable fix is a narrow recovery path for configured official externalized plugins during update/doctor, without broadening invalid-config install bypasses for unrelated broken configs.
