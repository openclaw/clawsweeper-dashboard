## Summary

Keep open: the PR targets a real Feishu topic-session bug and now includes redacted live log proof, but the current diff still breaks documented normal-group root-key behavior and expands Feishu message lookup before group admission checks.

## What This Changes

The PR changes Feishu topic-session routing and hydration so native topic groups and normal groups using Feishu topic message format can route by canonical `omt_*` topic IDs, with regression tests and a changelog entry.

## Best Possible Solution

Keep the Feishu-local fix, but suppress `rootId` only when the event or hydration result proves a Feishu topic-message-format topic and move the added message lookup behind group admission gates with regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main can route one normal group topic turn by `omt_*` when no root is present and a follow-up by `om_*` when a root is present, while native-topic hydration only covers `chat_type="topic_group"`. I did not execute tests in this read-only review.

## Solution Assessment

No as-is: the PR is in the right Feishu plugin area and has useful live proof, but the resolver-wide `omt_*` priority conflicts with current normal-group contracts and the expanded API lookup should be gated before merge.

## Review Findings

  - body: This change makes any normal `group` event carrying an `omt_*` `thread_id` route by the thread before `root_id`. Move this lookup behind those admission returns or narrow it so ignored group traffic cannot trigger authenticated message reads.
  - confidence: 0.84
