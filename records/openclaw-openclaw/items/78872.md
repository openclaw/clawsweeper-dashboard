## Summary

Keep open: the reported Feishu topic-session bug is plausible from current main, but this PR currently breaks the existing normal-group root-id contract, broadens an early Feishu message lookup before group acceptance checks, and lacks inspectable real behavior proof plus the required changelog entry.

## What This Changes

The PR changes Feishu topic-session routing to prefer valid `omt_*` thread IDs for normal groups, hydrate missing topic thread IDs through the replied-to/root message, and add regression tests for normal groups using Feishu topic message format.

## Best Possible Solution

Revise the Feishu plugin fix to distinguish normal OpenClaw-created threads from normal groups using Feishu topic-message format, keep `om_*` roots for the former, use `omt_*` only when the event/API proves a Feishu topic, defer lookups until after group acceptance checks, and add changelog plus inspectable live proof.

## Reproduction Assessment

Yes from source inspection: current main routes normal `group` events with `root_id=om_*` and `thread_id=omt_*` by the root ID under `group_topic`, while native `topic_group` hydration is the only current pre-routing lookup. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No: the PR targets the right Feishu-local area, but treating every normal-group `omt_*` thread ID as canonical conflicts with existing docs and tests for OpenClaw-created normal group threads. A narrower discriminator or post-access hydration path is safer.
