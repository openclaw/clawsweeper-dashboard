## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, the documented supported route is top-level `bindings[]`, and #63380 is still the paired open implementation candidate for this regression.

## What This Changes

The issue asks OpenClaw to stop rejecting legacy Telegram multi-account configs that put `agentId` under `channels.telegram.accounts.<id>` after upgrading from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review #63380 as the active implementation candidate. The durable outcome should either restore backward-compatible `channels.telegram.accounts.<id>.agentId` with routing semantics and regression coverage, or deliberately keep `bindings[]` as the only supported account-routing path and improve upgrade diagnostics so users do not hit a crash-on-restart path.

## Reproduction Assessment

Yes. The issue provides a concrete multi-account config, and current main’s strict Telegram account schema omits `agentId`, which is enough to reproduce the validation failure at config load.

## Solution Assessment

Unclear. Restoring account-level `agentId` is a plausible compatibility fix, but the best solution must include routing semantics or a doctor/docs migration path rather than only allowing the field in the schema.
