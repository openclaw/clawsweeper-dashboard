## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict named-account config, and #63380 is already the open unmerged implementation candidate for the same regression.

## What This Changes

The issue asks to preserve or migrate Telegram multi-account configs that put `agentId` under `channels.telegram.accounts.<id>` after upgrading from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review #63380 as the active implementation candidate, with the final outcome either restoring/migrating account-level `agentId` with routing semantics and regression coverage or explicitly documenting and diagnosing the `bindings[]` migration.

## Reproduction Assessment

Yes. The reported config places `agentId` under multiple Telegram accounts, and current main parses those accounts through a strict schema that omits that field, so the validation failure is source-reproducible with high confidence.

## Solution Assessment

Unclear until #63380 receives maintainer review. Restoring compatibility is plausible, but the durable fix must either route the legacy account field correctly or guide users to an equivalent `bindings[]` migration.
