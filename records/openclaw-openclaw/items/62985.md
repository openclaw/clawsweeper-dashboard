## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, and #63380 remains the open unmerged implementation candidate paired to this regression.

## What This Changes

The issue asks OpenClaw to keep Telegram multi-account configs with account-level `agentId` valid after upgrading from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review #63380 as the active candidate; the durable outcome should either accept or migrate legacy account-level `agentId` with routing semantics and regression coverage, or make the `bindings[]` migration explicit in doctor and upgrade docs.

## Reproduction Assessment

Yes. The provided config sets `agentId` under `channels.telegram.accounts.*`, and current main parses those accounts with a strict schema that omits that field, so the validation failure is source-reproducible.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring compatibility is plausible, but accepting `agentId` in the schema alone is not enough unless routing uses it or doctor migrates it to `bindings[]`.
