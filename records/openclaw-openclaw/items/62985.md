## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, while https://github.com/openclaw/openclaw/pull/63380 remains open and unmerged as the paired implementation candidate for this regression.

## What This Changes

The issue reports an upgrade regression where Telegram multi-account configs with `channels.telegram.accounts.<id>.agentId` abort validation after moving from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/63380 with a focused fix that aligns schema, exported types, runtime routing semantics, generated config metadata, docs, and regression coverage for legacy account-level `agentId`.

## Reproduction Assessment

Yes, at source level. A config with `channels.telegram.accounts.<id>.agentId` still enters a strict account schema without that field, and current Telegram routing only consumes topic-level `agentId` plus configured/runtime bindings.

## Solution Assessment

Unclear as currently queued. The linked PR is the right active candidate, but the maintainable fix must include real routing semantics and keep generated config metadata/docs aligned, not only make validation accept the field.
