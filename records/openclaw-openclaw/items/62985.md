## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, and #63380 is open and unmerged with closing syntax for this regression.

## What This Changes

Restore or clearly migrate Telegram multi-account configs that set `channels.telegram.accounts.<id>.agentId` after upgrading from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review #63380 as the active implementation candidate, with the final outcome either restoring account-level `agentId` compatibility with routing semantics and regression coverage or making the `bindings[]` migration explicit in doctor and upgrade docs.

## Reproduction Assessment

Yes. The provided config places `agentId` under multiple Telegram accounts, and current main parses those accounts with a strict schema that omits `agentId`, so the validation failure is source-reproducible with high confidence.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring compatibility is plausible, but the durable fix must either route the legacy account field correctly or guide users to an equivalent `bindings[]` migration.
