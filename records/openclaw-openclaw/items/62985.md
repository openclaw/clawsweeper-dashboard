## Summary

Keep open. Current main still rejects Telegram account-level `agentId` through the strict account schema and type, and #63380 is an open, unmerged implementation candidate paired with this issue.

## What This Changes

The issue asks OpenClaw to preserve or migrate Telegram multi-account configs that used `channels.telegram.accounts.<id>.agentId` before the 2026.4.8 upgrade.

## Best Possible Solution

Review #63380 as the active implementation candidate, with the final fix either preserving legacy account-level routing semantics with regression coverage or providing an explicit doctor/docs migration to equivalent `bindings[]` entries.

## Reproduction Assessment

Yes. The provided config places `agentId` under multiple Telegram accounts, and current main parses those accounts with a strict schema that omits `agentId`, so the validation failure is source-reproducible with high confidence.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring schema compatibility is plausible, but the maintainable solution must either route the legacy account field correctly or guide users to an equivalent `bindings[]` migration.
