## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, while #63380 remains open and unmerged with closing syntax for this regression.

## What This Changes

Restore or migrate Telegram multi-account configs that use `channels.telegram.accounts.<id>.agentId` so upgrades from 2026.4.5 do not fail config validation.

## Best Possible Solution

Review #63380 as the active implementation candidate, with the final behavior either restoring backward-compatible account-level routing with regression coverage or making the `bindings[]` migration explicit in doctor and upgrade docs.

## Reproduction Assessment

Yes. The issue provides a concrete multi-account config, and current source shows each account is parsed by a strict schema that omits `agentId`, which is enough for a high-confidence source reproduction of the validation failure.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring compatibility is plausible, but the durable fix must include routing semantics or a migration path rather than only allowing the field in the schema.
