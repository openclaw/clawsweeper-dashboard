## Summary

Keep open. Current main still rejects Telegram account-level `agentId` because the strict account schema and exported account type do not include that field, and #63380 remains open and unmerged with closing syntax for this issue.

## What This Changes

Restore or migrate legacy Telegram multi-account configs that put `agentId` under `channels.telegram.accounts.<id>` so gateway restart does not fail after upgrade.

## Best Possible Solution

Review #63380 as the active implementation candidate, with the final state either restoring legacy account-level `agentId` routing with regression coverage or making the `bindings[]` migration explicit in upgrade docs and doctor output.

## Reproduction Assessment

Yes. The issue provides a concrete multi-account config, and current source shows each account is parsed by a strict schema that omits `agentId`, which is enough to reproduce the validation failure at config load.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring compatibility is plausible, but the best fix must include routing semantics or a migration path rather than only allowing the field in the schema.
