## Label Justifications

- `P1`: The report is an upgrade regression that can make a real Telegram multi-account gateway abort on config validation.
- `impact:crash-loop`: The reported failure aborts gateway startup with `Config invalid` for existing Telegram config.
- `impact:message-loss`: Rejected Telegram account config prevents the affected Telegram accounts from starting and delivering messages.
- `impact:session-state`: The disputed field controls which agent/session a Telegram account should route to in multi-agent setups.

## Summary

Keep open. Current main still rejects the reported Telegram account-level `agentId` shape, and the paired fix candidate remains open, conflicting, and unmerged.

## What This Changes

The issue asks OpenClaw to handle legacy Telegram multi-account configs that put `agentId` under `channels.telegram.accounts.<id>` after an upgrade made them invalid.

## Best Possible Solution

Review, rebase, and either land or reject https://github.com/openclaw/openclaw/pull/63380, or replace it with an equivalent focused fix that restores or migrates legacy account-level `agentId` with routing semantics, docs, metadata, and regression coverage.

## Reproduction Assessment

Yes, at source level. The reported config places `agentId` inside strict Telegram account objects, and current main still lacks account-level schema/type/runtime support for that field.

## Solution Assessment

Unclear. Restoring compatibility may be right, but the durable fix needs real routing or a doctor/docs migration to `bindings[]`; schema-only acceptance would leave old configs validating without necessarily routing correctly.
