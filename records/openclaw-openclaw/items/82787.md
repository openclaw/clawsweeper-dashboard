## Summary

Keep open: the reported timing gap is still plausible on current main because subagent tracking starts the parent wait budget when the wrapper registers the run, while `agent.wait` can time out before the child emits a lifecycle start and later task code intentionally preserves an earlier `timed_out` state over a late success.

## What This Changes

The issue asks OpenClaw to preserve or reconcile successful subagent child results when the child starts only after the parent subagent wait has already timed out.

## Best Possible Solution

Keep this open and make the subagent lifecycle distinguish queued/not-started time from active child runtime, or retain a watcher that reconciles a late child terminal result and notifies the requester without weakening real terminal failure semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: mock or simulate `agent.wait` returning `timeout` before any lifecycle start, then emit a later successful child completion with the same run id; current task-registry behavior preserves `timed_out`. The issue’s local run-id timing evidence matches that path.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if implemented narrowly: separate queue/start and active execution budgets or keep a late-completion reconciler. Raising the timeout alone would mask the symptom without preserving successful child results reliably.
