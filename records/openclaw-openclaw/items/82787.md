## Summary

Keep open: current main still starts the parent subagent wait budget when the wrapper registers the run, maps an `agent.wait` timeout to a subagent timeout outcome, and preserves an earlier `timed_out` task state over a later success signal. The report is concrete, recent, and source-reproducible, with no merged PR or duplicate owning this exact delayed-child-start case.

## What This Changes

Report requests subagent lifecycle reconciliation when a parent wait times out before a delayed child run starts but that child later succeeds.

## Best Possible Solution

Fix the subagent lifecycle so queue/not-started time does not consume the full active child runtime budget, or keep a late watcher that reconciles same-run child terminal results and notifies the requester.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: simulate `agent.wait` timing out before any child lifecycle start, then emit a later successful terminal result for the same run id and observe that the parent/subagent task remains `timed_out`.

## Solution Assessment

Yes. The narrow maintainable fix is to separate queued/start and active execution budgets, or retain a reconciler for late child terminal results; raising the timeout alone would only mask the lost-success behavior.
