## Summary

Keep open: current main still matches the reported failure path where the parent subagent wrapper can consume its wait budget before the child actually starts, then preserve the earlier timeout instead of surfacing the later successful child result.

## What This Changes

The issue requests that parent subagent waits handle delayed child startup and reconcile late child success instead of leaving the requester in a stale timed-out state.

## Best Possible Solution

Implement a narrow subagent lifecycle fix that separates queued/not-started time from active child runtime or keeps a late-child watcher that reconciles terminal child results and notifies the requester.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: simulate `agent.wait` returning `timeout` before a child lifecycle start, then emit a later successful child completion with the same run id and observe the parent task remain `timed_out`. The issue's concrete run-id timing evidence matches that path.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if kept narrow: separate start/queue and active execution budgets or retain a late-completion reconciler. Raising the timeout alone would hide the symptom without reliably preserving successful child results.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/82787 as a narrow subagent lifecycle bug. Current main starts the parent subagent wait budget at registration, so `agent.wait` can time out before the child emits lifecycle start; when the child later succeeds with the same run id, task-registry monotonic terminal handling preserves the parent `timed_out` state and the requester is not notified. Adjust the lifecycle so queued/not-started time is not counted as active child runtime, or keep a watcher/reconciler after the parent wait timeout that observes the child terminal result and updates/delivers the subagent result. Preserve existing monotonic handling for operator cancellation, real failures, real runtime aborts/timeouts, and lost tasks. Do not add a new config option or broad polling loop. Add regression coverage for delayed child start followed by success after the initial parent wait timeout; update changelog if the fix is user-visible during landing.
