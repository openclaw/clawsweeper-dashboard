## Summary

Keep open. Current main partially addresses the reset/restart side by storing Telegram picker selections as user session overrides and preserving them across explicit `/new` or `/reset`, but the Telegram confirmation still does not say the agent default remains unchanged, reset acknowledgements are generic, and broader model-state follow-ups remain open.

## What This Changes

Request clearer Telegram model-switch persistence behavior so `/models` selections do not look like durable agent-default changes and reset/restart fallback is visible.

## Best Possible Solution

Keep core precedence unchanged for now, add clear Telegram-facing scope copy and reset/status messaging for session override versus agent default, and leave durable default reconciliation to the broader model-state design thread.

## Reproduction Assessment

Yes for the remaining UX mismatch: current source and tests show Telegram selection writes session override fields while the success text does not mention that the agent default remains unchanged. The exact production fallback trigger still needs a live or targeted regression reproduction before changing model precedence.

## Solution Assessment

Unclear for a core behavior change. The safest first repair is narrower UI/diagnostic wording because current main already preserves user overrides on explicit reset, while durable-default buttons and single-source reconciliation belong with the broader open model-state work.
