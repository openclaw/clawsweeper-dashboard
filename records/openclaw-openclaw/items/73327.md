## Summary

Keep open. Current main has improved the old `stuck session` signal by classifying active work as `session.long_running` or `session.stalled`, but cron jobs still cannot declare a per-job diagnostic attention threshold; the heartbeat uses one global `diagnostics.stuckSessionWarnMs` for every processing session.

## What This Changes

Request an optional per-cron-job diagnostic threshold so legitimately slow cron jobs can avoid noisy stuck/long-running warnings without raising the global session threshold.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, tools, UI/status/docs, and focused tests without changing execution timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main has no per-job override path, so the global threshold is always used.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected warning window; reusing `timeoutSeconds` would conflate alerting with execution cancellation.
