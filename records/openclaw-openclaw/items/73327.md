## Summary

Keep open. Current main has improved the diagnostic taxonomy from a single stuck-session warning into long-running/stalled/stuck classifications, but it still computes one global `diagnostics.stuckSessionWarnMs` per heartbeat and cron job definitions do not expose a per-job liveness warning threshold. The existing cron `timeoutSeconds` field is an execution timeout, so it does not cover the requested false-positive tuning for legitimately slow jobs.

## What This Changes

Request to add a backward-compatible per-cron-job stuck/session-liveness warning threshold with validation, detector integration, status visibility, tests, and unchanged default behavior.

## Best Possible Solution

Add a bounded top-level per-cron-job session-liveness warning threshold that defaults to the global `diagnostics.stuckSessionWarnMs`, is captured into diagnostic session state when a cron run starts, and is exposed consistently through Gateway protocol, cron normalization, agent tool, CLI/UI/status, docs, and tests without reusing `timeoutSeconds` or changing abort/recovery policy.

## Reproduction Assessment

Yes. A source-level reproduction is to put a cron-backed session into `processing`, let it exceed the resolved global `diagnostics.stuckSessionWarnMs`, and observe that the heartbeat classifies/logs it using the same threshold as every other session. There is no cron job field or session-state value that can vary that threshold per job.

## Solution Assessment

Yes. A per-job liveness warning threshold is the narrow maintainable fix because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or overloading `timeoutSeconds` would either hide unrelated stuck sessions or alter execution-timeout semantics.
