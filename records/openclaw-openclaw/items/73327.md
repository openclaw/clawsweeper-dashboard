## Summary

Keep open. Current main still applies one global diagnostic stuck-session warning threshold to every processing session, while cron job definitions, Gateway protocol schemas, agent tools, CLI/UI surfaces, and docs expose only `timeoutSeconds`, which is an execution-abort timeout rather than the requested per-job warning threshold. The related stuck-session recovery and watchdog issues are adjacent but do not replace this focused false-positive control for slow cron jobs.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120 second stuck-session warning.

## Best Possible Solution

Add a bounded additive per-cron-job stuck warning threshold that defaults to the existing global `diagnostics.stuckSessionWarnMs` behavior when absent. Capture the effective threshold into diagnostic session state when a cron-backed run starts, compare each processing session against its own threshold in the heartbeat, and expose the field consistently through cron normalization, Gateway protocol schemas, agent tools, CLI/UI/status surfaces, docs, and focused tests without changing `timeoutSeconds` abort semantics or broader stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, put a cron-backed diagnostic session into `processing`, and advance past the global `diagnostics.stuckSessionWarnMs`; current main will warn using the same threshold for that cron job as for every other processing session. The reporter's successful 130-210 second cron runs match that path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or alter execution-abort semantics.
