## Summary

Keep open. Current main still applies one global stuck-session warning threshold to all processing diagnostic sessions, and cron job schema/tool/UI/docs expose `timeoutSeconds` for execution aborts but no per-job stuck-warning threshold. The related stuck-session recovery issues are adjacent, not replacements for this narrower false-positive control.

## What This Changes

Request to add an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs can exceed the default 120s warning window without noisy false positives.

## Best Possible Solution

Add a bounded, additive cron job stuck-warning threshold that defaults to the existing global 120000 ms behavior when absent. Capture the effective value into diagnostic session state when a cron-backed run starts, compare the heartbeat against that per-session value, and surface it through cron schema/normalization/tool/CLI/UI/status/docs without changing `timeoutSeconds` abort behavior or the separate stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, mark a cron-backed diagnostic session as `processing`, and advance past 120 seconds; current main compares it to the single global threshold. The reporter’s successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or alter abort semantics.
