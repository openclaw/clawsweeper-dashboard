## Summary

Keep open. Current main has reduced the original noisy `stuck session` WARN path by classifying active work as `session.long_running` or `session.stalled`, but cron jobs still cannot declare their own diagnostic attention threshold; cron types, protocol schemas, UI/tool fields, and the diagnostic heartbeat still use only global `diagnostics.stuckSessionWarnMs` plus execution `timeoutSeconds`.

## What This Changes

The issue requests a per-cron-job configurable stuck-session diagnostic threshold so legitimately slow cron jobs do not emit false-positive attention warnings.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, agent tools, UI/status/docs, and focused diagnostics tests without changing execution-timeout or abort semantics.

## Reproduction Assessment

Yes, at source level. A cron agent-turn job can set `timeoutSeconds`, but there is no per-job diagnostic threshold field, and the heartbeat applies one global `diagnostics.stuckSessionWarnMs` threshold to all processing sessions.

## Solution Assessment

Yes in direction, but it needs maintainer API/config judgment. A per-job diagnostic threshold is narrower than raising the global threshold and avoids conflating execution budget (`timeoutSeconds`) with diagnostic warning classification.
