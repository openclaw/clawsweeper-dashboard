## Summary

Keep open. Current main has improved stuck-session diagnostics into long-running/stalled/stuck classifications, but cron jobs still cannot declare a per-job diagnostic attention threshold; the runtime applies one global `diagnostics.stuckSessionWarnMs` value to all processing sessions.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120s diagnostic threshold.

## Best Possible Solution

Add an additive cron-job diagnostic warning threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced consistently through cron storage, protocol schemas, CLI/tools, UI/status, docs, and focused tests without changing execution-timeout semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: age any processing cron-backed session past the configured global threshold and the heartbeat applies `diagnostics.stuckSessionWarnMs` because no cron job field or per-session threshold exists.

## Solution Assessment

Yes in direction. A per-job diagnostic threshold is the narrowest maintainable fix for slow cron jobs because it preserves aggressive defaults for interactive sessions and avoids conflating observability warnings with `timeoutSeconds` execution aborts.
