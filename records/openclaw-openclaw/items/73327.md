## Summary

Keep open. Current main has reduced the older healthy active-run warning flood, but cron jobs still cannot declare a per-job diagnostic attention threshold; only global diagnostics thresholds and cron execution `timeoutSeconds` are exposed.

## What This Changes

Request adding an optional per-cron-job stuck-session diagnostic threshold so legitimately long cron runs can avoid false-positive stuck-session alerts.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to the global `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, agent tools, UI/status/docs, and focused diagnostics tests without changing execution timeout or stale-marker semantics.

## Reproduction Assessment

Yes, at source level. Current main accepts cron `timeoutSeconds`, but there is no per-job diagnostic threshold field and the heartbeat applies the global `diagnostics.stuckSessionWarnMs` to every processing session.

## Solution Assessment

Yes in direction, but not as an automatic repair. A per-job diagnostic threshold is narrower than raising the global threshold, but it needs maintainer API/config approval and alignment with `timeoutSeconds` and abort behavior.
