## Summary

Keep open. Current main still has only a global `diagnostics.stuckSessionWarnMs` threshold, while cron jobs cannot declare a per-job diagnostic attention threshold for legitimately long runs.

## What This Changes

Request an optional per-cron-job stuck-session warning threshold so slow scheduled jobs do not trigger false-positive session attention logs at the global default.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is stored with the job, captured for cron-created sessions, and surfaced in protocol, tools, UI/status/docs, and focused tests without changing abort or execution-timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold because no per-job override exists.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; raising the global threshold or reusing `timeoutSeconds` would conflate separate contracts.
