## Summary

Keep open. Current main still applies one global diagnostic liveness threshold to all processing sessions, and cron job creation, validation, tools, UI, and docs expose `timeoutSeconds` only as execution timeout, not per-job stuck/long-running warning classification.

## What This Changes

Request a bounded per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger false-positive diagnostics.

## Best Possible Solution

Add an additive per-cron-job diagnostic warning threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured into diagnostic session state for cron-created sessions, and is exposed through cron storage, protocol schemas, tools, UI/status/docs, and focused tests without changing execution timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold to that cron session as every other processing session.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; raising the global threshold or reusing `timeoutSeconds` would either mask unrelated sessions or conflate warning classification with cancellation.
