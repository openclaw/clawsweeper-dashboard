## Summary

Keep open. Current main still applies one global `diagnostics.stuckSessionWarnMs` threshold to every processing diagnostic session, while cron job definitions and protocol/tool/UI-facing cron payloads only expose `timeoutSeconds`, which is a job abort timeout rather than a stuck-warning threshold. The reporter's production evidence matches the current source path, and the related stuck-session recovery/watchdog issues are adjacent rather than a replacement for this per-cron-job false-positive control.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120 second stuck-session warning.

## Best Possible Solution

Add a bounded additive cron job stuck-warning threshold that defaults to the existing global 120000 ms behavior when absent. Capture the effective threshold into diagnostic session state when a cron-backed run starts, compare each session against its own threshold in the heartbeat, and expose it consistently through cron schema, normalization, tools, CLI/UI/status surfaces, docs, and tests without changing `timeoutSeconds` abort semantics or broader stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, mark a cron-backed session as `processing`, and advance past the global threshold; current main compares every processing session against the same `diagnostics.stuckSessionWarnMs` value. The reporter's successful 130-210 second cron runs are consistent with that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or change execution-abort semantics.
