## Summary

Keep open. Current main improved stuck-session diagnostics into long-running/stalled/stuck classifications, but cron jobs still cannot set a per-job diagnostic attention threshold; the runtime applies one global `diagnostics.stuckSessionWarnMs` value to every processing session.

## What This Changes

The issue requests a per-cron-job configurable stuck-session diagnostic threshold so legitimately slow cron jobs do not produce noisy false positive warnings.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, tools, UI/status/docs, and focused tests without changing execution-timeout or abort semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with default or low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold because no per-job override exists.

## Solution Assessment

Yes in direction. A per-job diagnostic threshold is the narrowest maintainable solution because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; reusing `timeoutSeconds` would conflate execution timeout with observability noise control.
