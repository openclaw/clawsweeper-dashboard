## Summary

Keep open. Current main still applies one global stuck-session warning threshold to every processing diagnostic session, while cron job schema, normalization, tool, CLI, UI, and docs expose `timeoutSeconds` for execution aborts but no separate per-job stuck-warning threshold. The related watchdog/recovery reports are adjacent and do not supersede this narrower false-positive control.

## What This Changes

Request to let cron jobs declare a per-job stuck-session warning threshold so legitimately long cron runs do not emit the default 120s stuck-session warnings.

## Best Possible Solution

Add a bounded additive cron job stuck-warning threshold, defaulting to the current global 120000 ms behavior when absent. Capture the effective threshold into generic diagnostic session state when a cron-backed run starts, have the heartbeat compare each session against its own effective threshold, and expose the value consistently through cron schema/normalization/tool/CLI/UI/status/docs without changing `timeoutSeconds` or stuck-session recovery semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default config, mark a cron-backed session as `processing`, and advance past 120 seconds; current main compares that session to the single global threshold. The reporter's successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable solution because it preserves aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or change abort semantics.
