## Summary

Keep open. Current main still applies one global `diagnostics.stuckSessionWarnMs` threshold to every processing diagnostic session, while cron job types, Gateway schemas, agent tool schemas, CLI/UI surfaces, and docs expose `timeoutSeconds` for execution aborts but no per-job stuck-warning threshold. The related stuck-session recovery/watchdog reports are adjacent, but they do not supersede this narrower false-positive control for legitimately slow cron jobs.

## What This Changes

Allow cron jobs to declare an optional per-job stuck-session warning threshold that defaults to the current 120s behavior when unset.

## Best Possible Solution

Add a bounded, additive cron stuck-warning threshold that defaults to the existing global 120000 ms behavior. Capture the effective threshold into diagnostic session state when a cron-backed run enters processing, have the heartbeat compare each session against its own threshold, and expose the setting consistently through cron normalization, protocol schemas, agent tool schema, CLI/UI/status/docs, tests, and changelog without changing execution timeout or recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, mark a cron-backed session as `processing`, and advance time past the global threshold; current main compares every processing session against the same `diagnostics.stuckSessionWarnMs`. The reporter's 130-210 second successful cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves aggressive warnings for interactive sessions while letting known slow cron jobs declare their expected window. Raising the global threshold or reusing `timeoutSeconds` would either hide unrelated stuck sessions or change abort semantics.
