## Summary

Keep open. Current main still applies a single global stuck-session warning threshold to all processing sessions, and the cron job type/schema/tool/CLI/UI/docs surfaces expose `timeoutSeconds` for execution aborts but no per-job warning threshold for legitimately slow cron runs. The related stuck-session recovery/watchdog reports are adjacent, but they do not supersede this narrower cron false-positive control.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so long-running cron jobs can avoid false-positive `stuck session` warnings without raising the global threshold.

## Best Possible Solution

Add a bounded additive cron job stuck-warning threshold, defaulting to the existing global 120000 ms behavior when absent. Capture the effective threshold into diagnostic session state when a cron-backed run starts, have the heartbeat compare each session against its own effective threshold, and expose the value consistently through cron schema/normalization/tool/CLI/UI/status/docs without changing `timeoutSeconds` or stuck-session recovery semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, mark a cron-backed session as `processing`, and advance past 120 seconds; current main compares it to the single global `diagnostics.stuckSessionWarnMs` value. The reporter’s successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it keeps aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would mask unrelated stuck sessions or change abort semantics.
