## Summary

Keep open; current main still lacks a per-cron-job diagnostic liveness threshold, so slow successful cron sessions are still classified against one global `diagnostics.stuckSessionWarnMs`.

## What This Changes

Request adds an optional per-cron-job stuck-session warning threshold for long-running cron jobs while leaving the global default behavior unchanged.

## Best Possible Solution

Add an additive per-cron-job diagnostic warning threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured into diagnostic session state for cron-created sessions, and is exposed consistently through cron storage, protocol schemas, tools, UI/status/docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure diagnostics with the default or a low `stuckSessionWarnMs`, put a cron-backed session into `processing`, age it past the threshold, and observe that current main applies the same global threshold to that cron session as every other processing session.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected warning window; raising the global threshold or reusing `timeoutSeconds` would mask unrelated sessions or conflate warning classification with cancellation.
