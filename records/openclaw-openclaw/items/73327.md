## Summary

Keep open. Current main improved liveness classification and stale-session recovery, but cron jobs still cannot set a per-job diagnostic stuck/long-running threshold; the heartbeat uses one global `diagnostics.stuckSessionWarnMs`, while cron schemas, tools, UI, and docs only expose `timeoutSeconds` for execution timeout.

## What This Changes

Request a bounded per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger false-positive diagnostic alerts.

## Best Possible Solution

Add an additive per-cron-job diagnostic warning threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured into diagnostic session state when cron starts work, and is exposed consistently through cron normalization, gateway schemas, agent tools, UI/status/docs, and focused tests without changing `timeoutSeconds` abort semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold to that cron session as to every other processing session.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; raising the global threshold or reusing `timeoutSeconds` would mask unrelated sessions or conflate warning classification with execution cancellation.
