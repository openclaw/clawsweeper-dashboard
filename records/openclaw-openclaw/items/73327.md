## Summary

Keep open. Current main still uses one global `diagnostics.stuckSessionWarnMs` threshold for all processing sessions, while cron job payload/schema/tool/CLI/UI/docs expose `timeoutSeconds` for execution aborts but no per-job stuck-warning threshold. The related stuck-session recovery/watchdog reports are adjacent and do not supersede this narrower false-positive control for legitimately slow cron runs.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so slow-but-healthy cron jobs can avoid the default 120s diagnostic warning without changing abort behavior.

## Best Possible Solution

Add a bounded additive cron job warning-threshold field, defaulting to the existing global 120000 ms behavior when absent. Capture the effective threshold into diagnostic session state when a cron-backed run starts, compare each processing session against its own effective threshold, and expose the value through cron schema/normalization/tool/CLI/UI/status/docs without changing `timeoutSeconds` or recovery semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable diagnostics, mark a cron-backed diagnostic session as `processing`, and advance past 120 seconds; current main compares it to the single global `diagnostics.stuckSessionWarnMs` value. The reporter’s successful 130-210s cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window. A global bump or reuse of `timeoutSeconds` would either hide real stuck sessions or change abort behavior.
