## Summary

Keep open. Current main still applies one global `diagnostics.stuckSessionWarnMs` to all processing sessions, while cron job schemas, tools, CLI, UI, and docs expose `timeoutSeconds` only as execution timeout, so slow cron jobs cannot tune diagnostic liveness warnings per job.

## What This Changes

Add an optional per-cron-job stuck-session or long-running diagnostic warning threshold while preserving existing defaults when unset.

## Best Possible Solution

Add a bounded additive cron-job diagnostic warning threshold that defaults to the global diagnostics value, is captured into diagnostic session state for cron-started work, and is surfaced consistently through protocol, tools, CLI/UI/status/docs without changing `timeoutSeconds` abort semantics or recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, place a cron-backed session in `processing`, and age past the threshold; current main uses the same threshold for that job as every other processing session.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their diagnostic window; raising the global value or reusing `timeoutSeconds` would conflate separate behaviors.
