## Summary

Keep open. Current main has improved the old warning classification into `session.long_running`, `session.stalled`, and `session.stuck`, but cron jobs still cannot declare a per-job diagnostic attention threshold; every processing session is checked against one global `diagnostics.stuckSessionWarnMs` value.

## What This Changes

Request an optional per-cron-job diagnostic stuck-session/long-running threshold so legitimately slow scheduled jobs do not trigger false-positive attention logs at the global default.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, tools, UI/status/docs, and focused tests without changing execution-timeout or abort semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold because no per-job override exists.

## Solution Assessment

Yes in direction. A per-job diagnostic threshold is the narrowest maintainable solution because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; reusing `timeoutSeconds` would conflate execution timeout with observability noise control.
