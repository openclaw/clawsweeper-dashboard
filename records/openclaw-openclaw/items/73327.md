## Summary

Keep open. Current main has mitigated the older healthy active-run warning flood, but cron jobs still cannot declare a per-job diagnostic attention threshold; the cron payload, protocol schema, agent tool, UI shape, docs, and diagnostic heartbeat still expose only global diagnostics thresholds plus execution `timeoutSeconds`.

## What This Changes

The issue requests a per-cron-job configurable stuck-session diagnostic threshold so legitimately slow cron jobs do not emit false positive stuck-session warnings.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, agent tools, UI/status/docs, and focused diagnostics tests without changing execution-timeout or stale-marker semantics.

## Reproduction Assessment

Yes, at source level. Current main accepts cron `timeoutSeconds`, but there is no per-job diagnostic threshold field and the diagnostic heartbeat applies one global `diagnostics.stuckSessionWarnMs` threshold to all processing sessions.

## Solution Assessment

Yes in direction, but not as an automatic repair. A per-job diagnostic threshold is narrower than raising the global threshold and avoids conflating execution budget with diagnostic warning classification, but it needs maintainer API/config approval.
