## Summary

Keep open. Current main still uses one global `diagnostics.stuckSessionWarnMs` for all processing-session attention, while cron job schemas, tools, CLI, UI, and docs expose execution `timeoutSeconds` only; related PR #43816 is about stale `runningAtMs` lock cleanup and does not supersede this diagnostic-threshold request.

## What This Changes

Request to let cron jobs define an optional per-job stuck-session diagnostic warning threshold so slow but successful jobs do not emit noisy 120s warnings.

## Best Possible Solution

Add a bounded additive per-cron-job diagnostic warning threshold that defaults to the global diagnostics value, is applied to cron-backed session attention, and is exposed consistently through cron normalization, protocol schemas, agent tools, CLI/UI/status surfaces, docs, and focused tests without changing abort semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, age it past that threshold, and observe that the same global threshold is used for that job as for every other processing session.

## Solution Assessment

Yes. A per-job warning threshold is the narrow maintainable direction because it keeps aggressive defaults for interactive sessions while letting known slow cron jobs declare their expected diagnostic window; raising the global threshold or reusing `timeoutSeconds` would conflate separate behaviors.
