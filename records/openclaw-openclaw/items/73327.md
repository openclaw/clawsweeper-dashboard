## Summary

Keep open. Current main has improved the old false-positive `stuck session` wording by classifying active work as `session.long_running` or `session.stalled`, but cron jobs still cannot declare a per-job diagnostic attention threshold; the heartbeat applies one global `diagnostics.stuckSessionWarnMs` value to every processing session.

## What This Changes

Request an optional per-cron-job stuck-session or diagnostic attention threshold so legitimately slow cron jobs do not trip the default 120-second liveness warning window.

## Best Possible Solution

Add an additive cron-job diagnostic threshold that defaults to `diagnostics.stuckSessionWarnMs`, is captured for cron-created sessions, and is surfaced through cron storage, protocol schemas, tools, UI/status/docs, and focused tests without changing execution timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past the threshold; current main applies the same global threshold because no per-job override exists.

## Solution Assessment

Yes. A per-job diagnostic threshold is the narrowest maintainable direction because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window; raising the global threshold or reusing `timeoutSeconds` would mask unrelated sessions or conflate contracts.
