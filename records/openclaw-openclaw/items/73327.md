## Summary

Keep open. Current main has improved liveness classification, but cron jobs still cannot set a per-job diagnostic stuck/long-running threshold; the heartbeat uses one global `diagnostics.stuckSessionWarnMs` for every processing session, and cron schemas/tools/UI/docs expose only `timeoutSeconds`, which is execution timeout, not warning classification.

## What This Changes

Request an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120s diagnostic threshold.

## Best Possible Solution

Add a bounded additive per-cron-job diagnostic warning threshold that defaults to the global diagnostics value, is captured into diagnostic session state when cron starts work, and is exposed consistently through cron normalization, protocol schemas, agent tools, CLI/UI/status surfaces, docs, and focused tests without changing `timeoutSeconds` abort semantics or stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age past the threshold; current main uses the same global threshold for that job as for every other processing session. The emitted event may now be `long_running`, `stalled`, or `stuck` depending on observed activity, but the threshold cannot vary per cron job.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected diagnostic window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated sessions or conflate warning classification with execution cancellation.

## Repair Work Prompt

Implement issue #73327 by adding an optional bounded per-cron-job diagnostic stuck/long-running warning threshold. Prefer a name aligned with the existing diagnostics surface, such as `stuckSessionWarnMs`, unless maintainers have already chosen another name; validate roughly `[1000, 3600000]` ms. Default absent values to the existing global `diagnostics.stuckSessionWarnMs`. Capture the effective threshold into diagnostic session state when a cron-backed run starts so the heartbeat compares each processing cron session against its own threshold. Do not change `payload.timeoutSeconds` execution-abort semantics, stale cron lock cleanup, or broader stuck-session recovery policy. Thread the field through cron types, normalization, store/protocol schemas, agent cron tool flat/nested schemas, CLI add/edit/show/list/status as appropriate, Control UI cron form/type/i18n English strings, docs, and CHANGELOG. Note that PR #43816 is about stale cron lock cleanup for #30096 and should not be treated as the implementation for this issue. Use focused tests for default behavior, custom values below/above the global threshold, validation bounds, protocol/tool schema acceptance, CLI/UI serialization, and docs/status transparency.
