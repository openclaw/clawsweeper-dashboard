## Summary

Keep open. Current main still uses one global diagnostics liveness threshold for every processing session, while cron job schemas, tools, UI types, and docs expose only `timeoutSeconds`, which controls execution abort rather than diagnostic classification.

## What This Changes

The issue requests an optional per-cron-job stuck-session diagnostic threshold so legitimately slow cron jobs do not emit false-positive liveness warnings at the global default.

## Best Possible Solution

Add a bounded additive per-cron-job diagnostic warning threshold that defaults to the global diagnostics value, is captured into diagnostic session state when cron starts work, and is exposed consistently through cron normalization, protocol schemas, agent tools, CLI/UI/status surfaces, docs, and focused tests without changing `timeoutSeconds` abort semantics or stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat with the default or a low `diagnostics.stuckSessionWarnMs`, put a cron-backed session in `processing`, and age it past that threshold; current main applies the same global threshold to that cron job as to every other processing session.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable solution because it preserves aggressive defaults for interactive sessions while letting known slow cron jobs declare their diagnostic window; raising the global threshold or reusing `timeoutSeconds` would conflate separate contracts.

## Repair Work Prompt

Fix the current Cron per-job stuck-session diagnostic threshold issue. Add an optional bounded per-cron-job diagnostic warning threshold, defaulting to `diagnostics.stuckSessionWarnMs`, and keep it separate from `timeoutSeconds` execution abort semantics. Prefer an explicit name that cannot be confused with abort timeout, validate roughly `[1000, 3600000]` ms unless maintainers choose different bounds, and preserve existing jobs when the field is absent. Capture the resolved threshold into diagnostic session state when a cron-backed run starts, and have the diagnostic heartbeat use that per-session value when classifying `session.long_running`, `session.stalled`, or `session.stuck`. Update cron normalization, Gateway protocol schemas/validators, cron agent tool schemas and flat params, CLI/status/UI surfaces, English UI strings if needed, and docs. Add focused tests for default behavior, custom threshold behavior, range validation, and the distinction from `timeoutSeconds`. Do not change stuck-session recovery/abort policy or broaden this into the watchdog/recovery work tracked separately.
