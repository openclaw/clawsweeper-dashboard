## Summary

Keep open. Current main still uses one global stuck-session warning threshold for every processing diagnostic session, while the cron job type, protocol schema, CLI/UI-facing surfaces, and docs expose `timeoutSeconds` for cron execution aborts but no per-job warning threshold. The reporter's production context and the prior ClawSweeper comment remain consistent with the current source; related stuck-session recovery/watchdog reports are adjacent but do not supersede this narrower cron false-positive control.

## What This Changes

The issue requests a per-cron-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120-second stuck-session warning.

## Best Possible Solution

Add a bounded additive cron job stuck-warning threshold that defaults to the existing global 120000 ms behavior when absent. Capture the effective threshold into diagnostic session state when a cron-backed run starts, have the heartbeat compare each session against its own effective warning threshold, and expose it consistently through cron schema, normalization, tools, CLI/UI/status, docs, and tests without changing `timeoutSeconds` abort semantics or stuck-session recovery policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the diagnostic heartbeat, mark a cron-backed session as `processing`, and advance past the global threshold; current main compares every processing session against the same `diagnostics.stuckSessionWarnMs` value. The reporter's successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable solution because it preserves aggressive defaults for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or change execution-abort semantics.
