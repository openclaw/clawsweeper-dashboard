## Label Justifications

- `P2`: This is a normal-priority user-visible model registry bug with limited blast radius to installs carrying an invalid generated plugin catalog.
- `impact:auth-provider`: The failure affects provider/model registry loading and causes otherwise configured model choices to resolve as unknown.

## Summary

Keeping this open: current main and the latest release still return the invalid plugin catalog error from the outer custom-model load, which leaves the registry without valid models, and two open PRs already reference this issue as their closing fix path.

## What This Changes

The issue asks ModelRegistry to isolate invalid generated plugin catalog shards and surface the underlying validation error instead of turning all model resolutions into `Unknown model`.

## Best Possible Solution

Land one focused registry fix that skips only invalid generated plugin catalog shards while retaining or logging the validation error, then handle generator/doctor cleanup for stale catalog rows as a follow-up if needed.

## Reproduction Assessment

Yes, from source inspection: an invalid generated plugin catalog missing `api` triggers `validateConfig()`, the nested `loadCustomModels()` returns an error, and the outer plugin-catalog loop returns that error instead of keeping already parsed models. I did not run the live gateway flow because this review is read-only.

## Solution Assessment

Yes, failure isolation in `ModelRegistry` plus diagnostic surfacing is the narrowest maintainable first fix. Generator hygiene for stale preserved provider rows is useful follow-up work, but valid models should survive one bad generated plugin catalog either way.
