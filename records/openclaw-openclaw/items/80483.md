## Summary

Current main and beta already fix the reported realtime voice startup failure by resolving the configured realtime voice provider directly through the capability-provider resolver before falling back to the active provider list; the affected v2026.5.7 stable package lacked that lookup.

## What This Changes

The issue reports that voice-call realtime startup fails when `realtime.provider` is `openai` but the OpenAI plugin has not already been loaded.

## Best Possible Solution

Keep the direct configured-provider lookup on main/beta and handle the broader manifest-merge PR separately if maintainers still want that additional resolver behavior.

## Reproduction Assessment

Yes for the affected release: v2026.5.7 source lacked the direct configured-provider lookup and the issue includes concrete startup logs showing the active-registry miss. Current main no longer has that failing path for configured realtime voice providers.

## Solution Assessment

Yes. The current direct configured-provider lookup is the narrowest maintainable fix for the reported `realtime.provider: "openai"` startup failure; the open broader resolver PR can be reviewed separately from this issue.
