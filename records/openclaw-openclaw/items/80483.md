## Summary

Current main and the latest stable release already resolve the reported realtime voice provider load-order failure with a targeted configured-provider lookup that can load the requested provider even when another runtime registry is already active.

## What This Changes

The issue reports a realtime voice provider load-order regression where `voice-call` cannot find the configured OpenAI realtime provider if the OpenAI plugin was not already loaded at gateway startup.

## Best Possible Solution

Keep the shipped targeted provider lookup and close this report; handle the related contributor PR as superseded in separate PR triage.

## Reproduction Assessment

Yes. The affected 2026.5.7 path is source/log reproducible from the issue and maintainer package check, while current main no longer follows that failure path because configured realtime voice resolution performs a targeted provider lookup first.

## Solution Assessment

Yes. The shipped direct configured-provider lookup is narrower and more maintainable than forcing OpenAI to load on startup or broadly merging all manifest providers for this path.
