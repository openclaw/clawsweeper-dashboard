## Summary

The reported 2026.5.7 voice-call realtime startup failure is already fixed on current main and shipped in v2026.5.12; configured realtime voice providers now resolve through a direct provider lookup that can load the requested provider instead of depending on the already-active provider list.

## What This Changes

The issue reports that voice-call realtime startup fails when `realtime.provider: "openai"` is enabled but the OpenAI plugin has not already been loaded.

## Best Possible Solution

Keep the shipped direct configured-provider resolution; handle the linked open PR separately only if maintainers still want broader realtime list-merge behavior beyond this reported startup failure.

## Reproduction Assessment

Yes for the affected 2026.5.7 path: the report includes concrete startup logs and a maintainer confirmed the old package missed the configured OpenAI realtime provider. Current main should not reproduce the failure because configured provider lookup now cold-loads by provider id.

## Solution Assessment

Yes. The shipped direct configured-provider lookup is narrower than eagerly loading OpenAI on startup and addresses the reported `realtime.provider: "openai"` failure without changing plugin activation policy.
