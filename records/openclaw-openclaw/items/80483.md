## Summary

This should stay open: the report is recent, externally authored, and matches a real voice-call/provider activation boundary where current main still resolves realtime voice during service startup against the active plugin registry.

## What This Changes

The issue asks OpenClaw to load or resolve the configured realtime voice provider for voice-call startup when that provider plugin is enabled but not otherwise activated.

## Best Possible Solution

Make realtime voice resolution load the configured provider plugin from manifest contracts before voice-call starts, without forcing all provider plugins to start eagerly.

## Reproduction Assessment

No live current-main reproduction was established in this read-only pass. Source inspection plus the reporter's 2026.5.7 startup logs give a credible reproduction path around voice-call realtime startup with OpenAI enabled but not activated.

## Solution Assessment

No, the stop-gap of setting OpenAI `activation.onStartup` to true is not the best long-term fix. A targeted manifest/contract-based load of the configured realtime voice provider is narrower and preserves plugin laziness.
