## What This Changes

The issue reports that voice-call realtime startup fails when the configured OpenAI realtime voice provider has not already been loaded.

## Best Possible Solution

Keep the direct configured-provider lookup on main and let any broader manifest-merge behavior in https://github.com/openclaw/openclaw/pull/80486 be reviewed separately from this already-fixed startup failure.

## Reproduction Assessment

Yes for the affected release path from source and logs: 2026.5.7 used only the provider list/alias map, so an unloaded OpenAI realtime voice plugin could be missed after another provider was active. I did not reproduce the failure live against current main because the current-main source now takes the direct configured-provider path.

## Solution Assessment

Yes. The main fix is narrower than eager-loading provider plugins: it resolves the configured realtime voice provider through the manifest-backed capability provider lookup before using the list fallback.
