## Summary

Current main already contains the targeted runtime-provider lookup needed for this report: configured realtime voice providers are resolved through a provider-specific capability load that requires the owning plugin id, so an enabled-but-not-started OpenAI plugin is no longer skipped once another runtime registry exists. The fix is current-main-only; no release tag contains the proof commit yet.

## What This Changes

The issue reports that voice-call realtime startup fails when `realtime.provider: "openai"` is configured but the OpenAI plugin has not otherwise been eagerly loaded.

## Best Possible Solution

Keep the targeted current-main provider-loading path and ship it in the next release, rather than forcing OpenAI or other realtime providers to start eagerly.

## Reproduction Assessment

No high-confidence failing reproduction remains against current main. The reported `2026.5.7` failure is credible, but the current source path and regression test show configured realtime voice providers missing from the active registry are now loaded by provider id.

## Solution Assessment

Yes. The current fix is the narrow maintainable direction because it preserves plugin laziness and manifest ownership while loading only the configured missing capability provider.
