## Summary

Current main and the latest stable release resolve the configured realtime voice provider directly through the plugin capability resolver, so the reported lazy-load startup failure is fixed and the linked workaround PR is superseded.

## What This Changes

The issue requests fixing `voice-call` realtime startup when `realtime.provider: "openai"` is configured but the OpenAI plugin has not been eagerly loaded.

## Best Possible Solution

Keep the shipped targeted capability-provider resolution so OpenAI can remain lazy-loaded, and handle the linked workaround PR as superseded in its own review.

## Reproduction Assessment

Yes for the affected 2026.5.7 package: configure `voice-call` realtime with OpenAI used only as the realtime provider while another provider is active first. Current main is not expected to reproduce because the configured provider path now cold-loads through the manifest contract.

## Solution Assessment

Yes. The shipped solution is narrower than forcing OpenAI to `onStartup: true` because it preserves lazy plugin activation while resolving the specifically configured realtime voice provider.
