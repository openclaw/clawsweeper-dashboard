## Summary

Keep open. Current main still has the Telegram multi-account resolver mismatch: prompt, skill, and topic context read only the merged account config while mention gating still falls back to channel-level groups, and the linked implementation PR is closed without merging.

## What This Changes

The issue asks OpenClaw to make Telegram multi-account group/topic `systemPrompt`, `skills`, and topic settings resolve consistently with `requireMention` when those settings are configured at `channels.telegram.groups`.

## Best Possible Solution

Implement a Telegram-owned group context fallback for prompt, skill, and topic lookup: use account-scoped groups first, then channel-level groups when the account has no groups, while keeping multi-account routing and merged account config isolation unchanged. Add a focused regression test and a changelog entry; use docs or doctor warnings only if maintainers intentionally decide not to support this config shape.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: configure two Telegram accounts plus channel-level `channels.telegram.groups`, then the merged account config omits `groups`, `resolveTelegramGroupConfig` returns no prompt/topic config, and `resolveTelegramGroupRequireMention` still sees the same channel-level block.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the Telegram prompt/skill/topic resolver follow the existing account-level to channel-level fallback used by mention policy, without changing `mergeTelegramAccountConfig` or account routing semantics.
