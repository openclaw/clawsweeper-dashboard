## Summary

Keep open. Current main still has the reported Telegram resolver mismatch: multi-account account config drops channel-level groups, prompt/skills/topic context only reads that merged snapshot, while mention policy can still fall back to channel-level groups; the linked fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to make Telegram multi-account channel-level group/topic systemPrompt and skills config apply consistently with requireMention, or warn when that config will be ignored.

## Best Possible Solution

Keep the multi-account routing guard from https://github.com/openclaw/openclaw/issues/30673, but add a Telegram-owned prompt/skills/topic lookup that falls back from account-scoped groups to channel-level groups only for prompt context fields, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure two Telegram accounts with only channel-level `channels.telegram.groups`, then the merged account config has no `groups`, the prompt resolver returns no group/topic config, and mention policy still sees channel-level `requireMention`.

## Solution Assessment

Yes. The narrow maintainable fix is to align prompt/skills/topic context lookup with the account-level then channel-level fallback used by mention policy, without changing `resolveTelegramAccount` inheritance or group routing/access semantics.

## Repair Work Prompt

Fix the Telegram multi-account group prompt resolver mismatch for this issue. On current main, `mergeTelegramAccountConfig` intentionally drops channel-level `groups` for multi-account configs, but `resolveTelegramGroupConfig` only reads the merged groups snapshot while `requireMention` still falls back from account groups to channel-level groups. Implement a prompt/skills/topic-only fallback from `channels.telegram.accounts.<accountId>.groups` to `channels.telegram.groups` for inbound message and native-command context, while preserving the multi-account routing/access guard: do not make `resolveTelegramAccount` inherit channel-level groups in multi-account mode, and do not inherit `allowFrom`, `enabled`, `groupPolicy`, or other access-control/routing fields wholesale. Add regression coverage for two Telegram accounts with channel-level group/topic `systemPrompt` and `skills`, plus account-level groups winning when present. Include a `CHANGELOG.md` fix entry for the user-visible bug. Use https://github.com/openclaw/openclaw/pull/64650 as source context, but re-check against current `bot-core.ts` and current helper names before implementing.
