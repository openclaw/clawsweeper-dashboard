## Label Justifications

- `P2`: This is a normal-priority subagent timeout reliability bug with bounded gateway and agent-delivery surfaces but real parent-session workflow impact.
- `impact:session-state`: The child run can reach terminal timeout state while the parent session lacks the corresponding terminal completion state.
- `impact:message-loss`: The reported timeout completion notification can be dropped instead of delivered to the parent session.

## Summary

Keep open: current main has partial explicit-timeout mitigation, but the wait fallback still drops pending timeout snapshots and the reporter's zero-output timeout path still maps to a failed visible-reply delivery branch. Open PRs are useful wait-layer candidates, but none has merged or fully owns the no-output parent-notification behavior.

## What This Changes

The issue asks OpenClaw to notify the parent session when a sub-agent is force-killed by `runTimeoutSeconds`, including timeouts that produce no visible output.

## Best Possible Solution

A single canonical fix should make forced subagent timeouts produce exactly one parent-visible terminal notification, with pending-timeout snapshot preservation and zero-output delivery either repaired in the same change or split into a tracked follow-up.

## Reproduction Assessment

Yes, source-level: current main still has the asymmetric `waitForAgentJob` fallback and the failed `(no output)` completion delivery branch. I did not run a live parent/subagent timeout in this read-only review.

## Solution Assessment

No, current main is only partial. The wait-layer fix is a good bounded slice, but the best solution also needs to cover or explicitly split the zero-output timeout delivery path so the parent receives a terminal notification.
