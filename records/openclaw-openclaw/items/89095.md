## Label Justifications

- `P2`: This is a normal-priority reliability bug in the sub-agent timeout path with limited blast radius but real parent-session workflow impact.
- `impact:message-loss`: The issue is about a terminal sub-agent completion/timeout update being dropped instead of delivered to the parent session.
- `impact:session-state`: The parent and registry can disagree about whether a timed-out child has been surfaced, leaving the parent workflow waiting or polling.

## Summary

Keep open: the reported timeout-notification path is still a real current-main risk because `waitForAgentJob` can drop a pending timeout snapshot when the outer wait timer fires before timeout grace publishes it. Adjacent registry code has some hard-timeout fallback coverage, but the exact pending-timeout fallback remains asymmetric and should get a focused fix plus regression test.

## What This Changes

The issue asks for sub-agent run timeouts to notify the parent session instead of silently dropping the terminal timeout event.

## Best Possible Solution

Make the `waitForAgentJob` outer timer symmetric with the pending timeout path, preserve pending-error precedence, and add a regression that proves parent completion still announces after a forced sub-agent run timeout.

## Reproduction Assessment

Yes, by source: start `waitForAgentJob`, emit an aborted lifecycle timeout, then let the caller timeout fire before `AGENT_RUN_TIMEOUT_RETRY_GRACE_MS`; the current fallback resolves `null` because it checks only pending errors. I did not run that fake-timer repro in this read-only review.

## Solution Assessment

Yes, the reporter's symmetric fallback is the narrowest likely repair, provided it keeps pending-error precedence and gets a regression test for the outer-wait-before-timeout-grace race. The no-output delivery behavior should be left unchanged unless a separate test proves it contributes to this symptom.
