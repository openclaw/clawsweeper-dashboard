## Label Justifications

- `P1`: The latest release report shows a broken sub-agent completion workflow that can leave a yielded parent waiting indefinitely in real use.
- `impact:message-loss`: The child terminal completion notice can be retried and dropped instead of reaching the parent/requester session.
- `impact:session-state`: The child run can be terminal while the parent session remains waiting without the corresponding completion state.

## Summary

Keep open: current main and v2026.6.9 still retain both the pending-timeout wait fallback gap and the no-output failed completion delivery branch, and the latest 2026.6.9 report broadens this to clean stops with no deliverable payload. The open PRs are useful wait-layer candidates, but they do not close the delivery-layer parent notification bug.

## What This Changes

The issue asks OpenClaw to notify the parent/requester session when a sub-agent reaches a terminal state but produces no deliverable completion payload.

## Best Possible Solution

Land one canonical wait-layer fix, then add a delivery-layer terminal notification for required completions with no deliverable payload, coordinated with https://github.com/openclaw/openclaw/issues/90178 and proven by a live parent-session run.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 still have the asymmetric wait fallback and the failed '(no output)' completion delivery branch. I did not run the timing-sensitive live parent/subagent scenario in this read-only review.

## Solution Assessment

No, current main is not the best solution yet. A wait-layer fix is necessary, but the best fix also needs a maintainer-approved terminal notice path for required completions with no deliverable payload.
