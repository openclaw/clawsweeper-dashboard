## Summary

Keep open. The PR targets a real privacy/UX bug, but the patch is incomplete because verbose-off runs can still deliver the full buffered assistant text in the final chat event, and the external PR has only unit-test proof rather than real channel behavior proof.

## What This Changes

The branch gates gateway assistant chat delta/flush delivery to session subscribers based on verbose level, adds gateway regression tests, and carries stale channel/plugin SDK merge fixups.

## Best Possible Solution

Update the PR to gate all session/channel chat payload paths, including final and error events, against fresh verbose state while preserving WS broadcasts; then add lifecycle-end regression coverage and redacted real Telegram/channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on the PR head: emit assistant text with verbose off, then emit lifecycle end; deltas are gated, but emitChatFinal still sends the full buffered final payload to the session/channel subscriber.

## Solution Assessment

No. Gating only delta and buffered flush paths is useful but not sufficient; final/error chat events need the same policy, and the verbose decision should not be frozen in a way that ignores runtime session changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate final chat sends when verbose is off:** `src/gateway/server-chat.ts:496`
  - body: The new guard only wraps delta/flush sends. `emitChatFinal` still turns the buffered assistant text into a final `message` and always calls `nodeSendToSession`, so a verbose-off subscriber can miss all deltas but still receive the complete internal text at lifecycle end. Apply the same session/channel gate to final/error chat sends and cover that lifecycle path.
  - confidence: 0.92
- **[P2] Avoid freezing verbose mode for the entire run:** `src/gateway/server-chat.ts:511-513`
  - body: `resolveVerboseLevel` returns the first cached value for the run, including default/null fallbacks, so later `/verbose` or session config changes during a long stream are ignored. That can keep leaking after verbose is turned off, or keep suppressing after it is turned on. Either re-read session state for assistant streaming decisions or invalidate the cache when session settings change.
  - confidence: 0.78
