## Summary

Keep open: this MEMBER PR has the protected `maintainer` label, and the updated diff now covers both restart paths, but it still drops the new terminal flag when rebuilding WhatsApp status snapshots, so `channels.status` can still report a terminal stop as `not-running`.

## What This Changes

The PR adds a `terminalDisconnect` channel status flag, maps it to a `terminal-disconnect` health reason, skips both health-monitor and manager auto-restarts for that flag, updates WhatsApp monitor state, tests, and changelog.

## Best Possible Solution

Keep the two restart guards, and make the new terminal-disconnect state visible anywhere channel account snapshots are rebuilt so operators and status consumers see the same terminal state the health monitor uses.

## Reproduction Assessment

Yes. Source inspection on current main shows the logged-out/conflict path stops WhatsApp, while generic health and manager recovery treat stopped non-manual channels as restartable; I did not run a live WhatsApp unlink in this read-only review.

## Solution Assessment

No. The restart suppression design is the right narrow direction and the second commit fixes the earlier manager-path blocker, but the PR should also propagate `terminalDisconnect` through rebuilt status snapshots before merge.
