## Summary

Keep open: this is the active candidate for the WhatsApp terminal-disconnect restart loop, but the current head still lets the ChannelManager timed-out recovery branch restart before the new terminal guard runs, and the plugin-facing restart-suppression contract needs maintainer acceptance.

## What This Changes

The PR propagates a `terminalDisconnect` channel snapshot flag from WhatsApp terminal stops and skips gateway health/task auto-restarts when that flag is set, with focused tests.

## Best Possible Solution

Move the terminal guard ahead of every ChannelManager recovery restart branch, clear recovery bookkeeping when skipping, and land only after maintainers accept the generic terminal-stop snapshot contract.

## Reproduction Assessment

Yes, source-reproducible: current main records WhatsApp logged-out/conflict health states but generic health policy collapses stopped accounts to `not-running`, and both health monitor and ChannelManager paths restart non-manual stopped accounts. I did not run a live WhatsApp unlink or replacement-session test.

## Solution Assessment

No, not yet: the contract shape is a plausible narrow bug fix, but the guard must move before timed-out recovery restarts and maintainer acceptance is needed for the plugin-facing restart-suppression field.
