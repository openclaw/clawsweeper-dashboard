Keep open: this is a focused fix attempt for a real Telegram reasoning-delivery bug, but the patch is not sufficient because Telegram still normalizes and suppresses `isReasoning` payloads before they can reach the reasoning lane; it also lacks live Telegram proof.

## What This Changes

The PR changes `src/auto-reply/reply/dispatch-from-config.ts` so Telegram `isReasoning` block and final payloads bypass the generic shared-dispatch suppression guards.

## Best Possible Solution

Land a Telegram-owned durable reasoning conversion that strips `isReasoning` before outbound normalization, bypasses shared TTS for reasoning, preserves generic-channel suppression, adds focused final/block regression coverage, and includes redacted Telegram-visible proof.

## Reproduction Assessment

Yes, source-level: current main suppresses durable reasoning block and final payloads before Telegram delivery, and this PR still hits Telegram's outbound suppression path after the changed guards. I did not run a live Telegram reasoning-model session in this read-only review.

## Solution Assessment

No as written: the shared guard change is only a plausible first step, not the best fix. The safer solution is to convert or strip Telegram reasoning before outbound normalization and keep reasoning out of TTS.
