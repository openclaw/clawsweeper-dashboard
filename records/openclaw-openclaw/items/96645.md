## Summary

Keep open: this is a useful candidate for the Mattermost private-channel namespace bug, but it is not merge-ready because the route still treats ambiguous Mattermost directory `group` metadata as private-channel evidence and the new screenshot does not prove the session store avoided the phantom `channel:<id>` namespace.

## What This Changes

The PR changes Mattermost outbound session routing, bare-id target resolution, and isolated cron delivery route selection so private channels can reuse `group:<id>` session keys while still sending to `channel:<id>`.

## Best Possible Solution

Keep the Mattermost-local cron/current-session direction, but narrow `group` selection to authoritative `P`/`G` evidence or a matching inbound session key, add public/private regression coverage, and require inspectable redacted Mattermost session proof.

## Reproduction Assessment

Yes for the patch defect: source inspection shows Mattermost directory groups include public `O` channels, while the PR treats `resolvedTarget.kind === "group"` as enough to select the `group` namespace. The original private-channel bug is also source-proven, but I did not run a live Mattermost workspace.

## Solution Assessment

No as submitted. The bound-session cron direction and bare-id channel lookup are reasonable, but the route must not treat generic directory `group` metadata as authoritative private-channel evidence.
