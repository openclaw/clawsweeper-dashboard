## Summary

Keep open: the PR targets a real Mattermost session-routing bug, but the current patch still trusts `resolvedTarget.kind === "group"` even though Mattermost directory groups include public `O` channels, so it can move public-channel outbound sessions into the `group` namespace while inbound remains `channel`. The updated PR body adds a redacted before/after table, but it still lacks inspectable live output, logs, screenshot, recording, or an artifact that proves the real Mattermost path after the fix.

## What This Changes

The PR changes Mattermost outbound route keying, opaque target resolution, and cron delivery session-key selection so private-channel deliveries can reuse `group:<id>` session namespaces while keeping `channel:<id>` wire targets.

## Best Possible Solution

Keep the Mattermost-local cron/current-session direction, but narrow `group` selection to authoritative `P`/`G` evidence or a matching inbound session key, add a regression test that directory-resolved public `O` channels remain `channel`, and require inspectable redacted Mattermost proof before merge.

## Reproduction Assessment

Yes. Source inspection shows Mattermost public and private channels are both emitted as directory `group` entries while inbound type `O` remains `channel`, so the PR’s `resolvedKind === "group"` route can reproduce a public-channel namespace split without a live server.

## Solution Assessment

No as submitted. The bound-session cron direction and bare-id channel lookup are reasonable, but the route must not treat generic directory `group` metadata as authoritative private-channel evidence.
