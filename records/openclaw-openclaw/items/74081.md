## Summary

Keep open: the feature is plausible, but the current diff changes generic config substitution before schema validation and does not preserve authored env references through config writes.

## What This Changes

The PR adds JSON string-array coercion for whole-value `${VAR}` config strings plus unit tests around accepted and rejected env values.

## Best Possible Solution

Adopt a schema-aware or explicit list-env contract that only applies to list-typed paths, preserves authored env references during writes, and documents the behavior.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR branch runs array coercion on every string value before validation and current write-back restoration only restores same-type leaves. I did not run tests because this review is read-only.

## Solution Assessment

No. Whole-value `${VAR}` may be a good user-facing direction, but this implementation needs schema/path gating or an explicit list syntax plus write-back preservation to be maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate array coercion to list-typed config paths:** `src/config/env-substitution.ts:200-203`
  - body: `SINGLE_VAR_REFERENCE.test(value)` runs for every config string before validation, so a string-only field such as `agents.defaults.systemPromptOverride: "${PROMPT}"` becomes an array when `PROMPT` contains JSON array text and then fails schema validation. Gate this on list-typed schema/path metadata or use an explicit list syntax.
  - confidence: 0.86
- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:203`
  - body: Returning `asArray` changes the value type from the authored `${VAR}` string before config writes. `restoreEnvVarRefs` only restores same-type string/string or array/array leaves, so a later config write can persist the resolved array and lose the env reference.
  - confidence: 0.91
