## Summary

## What This Changes

This PR changes config env substitution so an exact `${VAR}` placeholder can return `string[]` when the environment value parses as a JSON string array, and adds unit tests for that behavior.

## Best Possible Solution

Keep generic env substitution string-preserving and add list-valued env parsing only through a schema-aware or field-scoped contract for explicit string[] config paths.

## Reproduction Assessment

Yes for the PR-introduced compatibility risk by source inspection: the patch returns string[] from the generic resolver for exact placeholders, while current main always returns a string. I did not run tests because this review is read-only.

## Solution Assessment

No. The requested ergonomics are plausible, but the current approach is too broad; schema-aware or field-scoped parsing is the safer maintainable path.
