## Summary

Keep open, but this PR is not merge-ready: the useful container/K8s ergonomics idea is real, but the patch changes the generic config env-substitution contract before schema validation and can turn existing string or SecretInput config values into arrays.

## What This Changes

The PR changes config env substitution so a whole-value `${VAR}` can resolve to `string[]` when the env value is a JSON string array, with unit coverage for accepted and rejected forms.

## Best Possible Solution

Keep env substitution string-preserving by default; add any JSON-array env support only through explicit schema-aware or field-scoped list config with docs and upgrade proof.

## Reproduction Assessment

Yes, for the PR-introduced risk by source inspection: on the PR branch, a string field like `models.providers.openai.apiKey: "${OPENAI_API_KEY}"` with `OPENAI_API_KEY='["x"]'` would become `string[]` before schema validation. I did not run it because this review is read-only.

## Solution Assessment

No. The generic resolver is the wrong layer for this feature; a schema-aware or field-scoped list-env contract is safer than changing all config string substitutions.
