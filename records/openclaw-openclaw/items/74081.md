## Summary

Keep this PR open, but it is not merge-ready. Current main still has string-only config env substitution, so the contribution targets a real operator gap, but the branch changes the generic string substitution path and can unexpectedly turn existing string fields into arrays while also bypassing current env-reference write-back preservation.

## What This Changes

The PR adds JSON string-array parsing to whole-value config `${VAR}` substitutions and adds focused tests for positive, whitespace, non-array, malformed, mixed-type, empty, and escaped cases.

## Best Possible Solution

If maintainers want this capability, implement it as a schema/path-aware list-field feature or an explicit list-reference syntax, preserve authored env references through read/write round trips, document the contract, add the required changelog entry, and cover validation plus write-back behavior in targeted tests.

## Reproduction Assessment

Yes for the review findings. The high-confidence static path is config read substitution before validation, followed by write-back through `restoreEnvVarRefs`; under the PR, a string field such as `agents.list[].systemPromptOverride: "${PROMPT}"` with `PROMPT='["x"]'` would become an array before string validation, and a list field sourced from `${ALLOW_FROM}` could later write the resolved array back to disk.

## Solution Assessment

No. The proposed whole-value `${VAR}` trigger is narrower than parsing all strings, but it is still global and value-dependent; a path/schema-aware list coercion or explicit syntax is the safer maintainable direction.
