## Summary

Keep this PR open. The use case is real, but the current patch changes generic config env substitution globally, which can break string-typed config fields and weaken env-reference preservation on write-back.

## What This Changes

The PR adds JSON string-array parsing for whole-value `${VAR}` config substitutions and extends `src/config/env-substitution.test.ts` with boundary tests.

## Best Possible Solution

If maintainers want this capability, make env-backed list values explicit or schema/path-aware, preserve authored env refs across read/write, document the new contract, and add changelog plus targeted validation/write-back tests.

## Reproduction Assessment

Yes for the review findings. The high-confidence static path is config read substitution before validation, followed by write-back through `restoreEnvVarRefs`; a string field such as `agents.list[].systemPromptOverride: "${PROMPT}"` with `PROMPT='["x"]'` would become an array before string validation, and a list field sourced from `${ALLOW_FROM}` could later write the resolved array back to disk.

## Solution Assessment

No. The proposed whole-value `${VAR}` trigger is narrower than parsing every string, but it is still global and value-dependent; a path/schema-aware list coercion or explicit list-reference syntax is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:204`
  - body: Returning `asArray` here means the incoming config no longer has the same type as the parsed source string `${ALLOW_FROM}` that `writeConfigFile` gives to `restoreEnvVarRefs`. That helper only restores string/string or array/array leaves, so a read/write round trip can persist the resolved array and lose the authored env reference. Add whole-ref array preservation coverage or keep the resolved type compatible with the parsed source.
  - confidence: 0.91
- **[P2] Gate coercion on list-typed config fields:** `src/config/env-substitution.ts:201-204`
  - body: `SINGLE_VAR_REFERENCE.test(value)` runs for every config string, so any existing string field authored as `${VAR}` becomes an array when the env value is JSON array text. String-only fields such as `agents.list[].systemPromptOverride` are validated as strings after substitution, so this can make a previously valid env-backed config fail validation. Gate this on known list-typed config paths/schema metadata, or use an explicit list syntax.
  - confidence: 0.86
- **[P3] Add the user-facing changelog entry:** `src/config/env-substitution.ts:194`
  - body: This is a user-facing config feature, and repo policy requires user-visible `feat` changes to add a `CHANGELOG.md` entry. The branch only changes config substitution code and tests, so release notes would miss the new config contract.
  - confidence: 0.88
