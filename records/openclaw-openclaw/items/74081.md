## Summary

Keep open; the feature is useful for container deployments, but the current patch changes generic string substitution before schema validation and can also bypass the existing env-reference preservation path during config write-back.

## What This Changes

The branch adds JSON string-array parsing for whole-value `${VAR}` config substitutions and adds boundary tests for that behavior.

## Best Possible Solution

Use an explicit or schema/path-aware list env substitution contract that only applies to list-valued fields, preserves authored env refs on write, and updates docs plus changelog.

## Reproduction Assessment

Yes. A static reproduction is a string field such as `agents.list[].systemPromptOverride: "${PROMPT}"` with `PROMPT='["x"]'`, which this patch turns into an array before string validation; a list field sourced from `${ALLOW_FROM}` can also write the resolved array back because restore is same-type only.

## Solution Assessment

No. Whole-value `${VAR}` is narrower than parsing every string, but it is still global and value-dependent; the safer direction is an explicit syntax or a schema/path-aware coercion with write-back preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate coercion on list-typed fields:** `src/config/env-substitution.ts:200-203`
  - body: `SINGLE_VAR_REFERENCE.test(value)` runs for every config string, so string-only fields such as `agents.list[].systemPromptOverride` become arrays whenever the env value happens to be JSON array text. That makes previously valid env-backed string configs fail validation; gate this on list-typed paths/schema metadata or use an explicit list syntax.
  - confidence: 0.86
- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:203`
  - body: Returning `asArray` here changes an authored `${ALLOW_FROM}` string into an array before write-back. `restoreEnvVarRefs` only restores string/string or array/array leaves, so a read/write round trip can persist the resolved env array and lose the original env reference.
  - confidence: 0.91
- **[P3] Add the config changelog entry:** `src/config/env-substitution.ts:194`
  - body: This is a user-facing config feature, but the branch only changes substitution code and tests. Repo policy expects `feat` config behavior to be reflected in `CHANGELOG.md` so the new contract is visible in release notes.
  - confidence: 0.88
