## Summary

Keep open, but the current patch should not merge: it changes a global, untyped config substitution contract and can turn existing env-backed string fields into arrays before schema validation. This is still a meaningful config ergonomics idea, so it does not meet the close bar for stale, duplicate, implemented, or low-signal cleanup.

## What This Changes

The PR changes config environment substitution so exact `${VAR}` placeholders can return `string[]` when the env value parses as a JSON string array, and adds unit coverage for that behavior.

## Best Possible Solution

Keep generic env substitution string-preserving, then add a typed or field-scoped list-env contract only where `string[]` config paths explicitly allow it.

## Reproduction Assessment

Yes for the PR-introduced compatibility risk by source inspection: after the patch, any exact string placeholder whose env value parses as a JSON string array returns `string[]`, while current main always returns a string from `substituteString`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature need is plausible, but changing the generic untyped resolver is too broad; the safer solution is schema-aware or field-scoped parsing for array-valued config only.
