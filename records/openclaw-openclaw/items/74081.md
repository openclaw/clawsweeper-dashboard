## Summary

Keep this PR open but do not merge it as-is. The implementation is useful source work, but the current diff applies value-based array coercion globally before config validation and before env-reference preservation, so it can break existing string fields and persist values that were authored as env refs.

## What This Changes

The PR changes config env substitution so a whole-value `${VAR}` whose env value is a JSON string array returns `string[]`, with unit coverage for accepted and rejected cases.

## Best Possible Solution

If maintainers want this capability, make list env injection explicit or schema/path-aware, preserve authored `${VAR}` references through read/write, and document the new contract with changelog coverage.

## Reproduction Assessment

Yes. The high-confidence static path is config read substitution before validation, followed by write-back through `restoreEnvVarRefs`; a string field such as `agents.list[].systemPromptOverride: "${PROMPT}"` with `PROMPT='["x"]'` would become an array before string validation.

## Solution Assessment

No. The proposed whole-value `${VAR}` trigger is narrower than parsing every string, but it is still global and value-dependent; schema/path-aware list coercion or an explicit list-reference syntax is the safer maintainable direction.
