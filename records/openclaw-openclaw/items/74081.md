## Summary

Keep this PR open: the feature is plausible, but the current diff changes generic config substitution before validation and has correctness plus env-reference persistence blockers. It also lacks after-fix real behavior proof from the external contributor.

## What This Changes

The branch adds JSON string-array coercion for whole-value `${VAR}` config substitutions and unit tests for the coercion boundaries.

## Best Possible Solution

Define an explicit or schema/path-aware env-backed list contract, preserve authored env references across read/write, then document and changelog the accepted behavior during landing.

## Reproduction Assessment

Yes for the review findings: source inspection shows a string-only config field such as `agents.defaults.systemPromptOverride: "${PROMPT}"` would see an array before validation when `PROMPT` contains JSON array text. A read/write path with `${ALLOW_FROM}` similarly changes parsed string input into an array that same-type env-reference restoration cannot restore.

## Solution Assessment

No. Whole-value `${VAR}` is narrower than parsing every string, but the generic branch is still not the safest maintainable contract without schema/path gating or explicit list syntax plus write-back preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate array coercion on list-typed config fields:** `src/config/env-substitution.ts:198-201`
  - body: `SINGLE_VAR_REFERENCE.test(value)` runs for every config string, so a string-only field such as `agents.defaults.systemPromptOverride: "${PROMPT}"` becomes an array when `PROMPT` contains JSON array text and then fails validation. Gate this on list-typed paths/schema metadata or use an explicit list syntax.
  - confidence: 0.86
- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:201`
  - body: Returning `asArray` changes the value type from the parsed `${VAR}` string before write-back. `restoreEnvVarRefs` only restores same-type string/string or array/array leaves, so a config write can persist the resolved array and lose the authored env reference.
  - confidence: 0.91
