## Summary

Keep open. The cron parser guard is useful and current main still has the nullish `.trim()` crash path, but the branch also adds an unrelated package-update early return that conflicts with the current same-version refresh contract and tests.

## What This Changes

This PR widens `parseAbsoluteTimeMs` to accept nullish input, adds parser tests, and adds an update-command early exit when the current package version equals the resolved target version.

## Best Possible Solution

Keep the nullish cron parser guard with focused parser and cron-list regression coverage plus a changelog entry, and remove or separately redesign the update no-op optimization so it preserves the existing refresh contract.

## Reproduction Assessment

Yes. A malformed persisted cron row with `schedule: { kind: "at" }` can reach `printCronList` -> `formatSchedule` -> `formatIsoMinute` -> `parseAbsoluteTimeMs(undefined)` on current main; the PR-introduced update regression is covered by same-version package update tests that expect install and follow-up work.

## Solution Assessment

No. The parser guard is the narrow cron fix, but bundling the update early return is not the best solution because it contradicts documented and tested package-update behavior.
