## Summary

Keep open. The cron nullish-input guard is useful and current main still has the unsafe `input.trim()` path, but the PR also bundles an unrelated package-update early return that conflicts with current docs and tests for same-version refresh behavior.

## What This Changes

The PR widens `parseAbsoluteTimeMs` to accept nullish inputs and adds parser tests, while also adding an `updateCommand` early return that skips package-manager updates when current and target versions match.

## Best Possible Solution

Keep the cron fix direction, but split the branch before merge: retain the nullish parser guard, add focused parser and CLI malformed-`at` coverage plus the required changelog entry, and remove the package-update early return unless a separate update-contract change updates docs and tests together.

## Reproduction Assessment

Yes. A persisted cron row with `schedule: { kind: "at" }` can flow through `printCronList` to `formatSchedule`, then into `parseAbsoluteTimeMs(undefined)` on current main. The PR-introduced update regression is also reproducible against the existing same-version package-update tests and docs contract.

## Solution Assessment

No. The cron parser guard is the right narrow fix, but this PR is not the best complete solution because it bundles an unrelated update-policy change that contradicts the maintained package-refresh contract.
