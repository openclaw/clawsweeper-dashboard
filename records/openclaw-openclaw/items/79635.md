## Summary

Keep this PR open, but do not merge it as-is: it targets a real linked session-integrity report, yet the new repair heuristic can delete valid tree-shaped transcript leaves and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a load-time session-file repair that drops zero-descendant siblings under a shared parentId, adds session-file-repair regression tests, and updates the changelog.

## Best Possible Solution

Use a compaction-retry-specific reconciliation signal, such as supersession metadata or deterministic retry IDs, and preserve generic tree leaves unless an entry is provably the retry loser.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: a valid transcript with one leaf sibling and one continued sibling would hit the new zero-descendant drop branch. The original linked issue has concrete JSONL evidence, but not a live current-main compaction retry reproduction.

## Solution Assessment

No. Load-time repair may be a useful part of the answer, but this implementation lacks a compaction-specific signal and deletes generic leaf branches that current docs and readers allow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not drop generic leaf branches as orphan forks:** `src/agents/session-file-repair.ts:325-326`
  - body: `readParentLinkedEntry` includes every parent-linked entry, so this branch drops any zero-descendant sibling whenever one sibling under the same parent has descendants. Since session JSONL is a tree, a valid side branch can naturally be a leaf next to a continued branch; this repair would delete that transcript entry on load. Please restrict the drop to entries that are provably compaction-retry losers and add a regression for a legitimate leaf branch.
  - confidence: 0.88
