## Summary

Keep open: current main and v2026.6.6 still expose only coarse TUI run/connection states plus internal model-call diagnostics, not the requested user-facing packet, byte, or last-progress monitor in the status line.

## What This Changes

The issue requests a user-facing live model-connection or packet/activity monitor in the OpenClaw status line so users can tell active model work from a stalled or disconnected run.

## Best Possible Solution

If maintainers want this, expose a compact last-progress or model-I/O liveness indicator in the TUI using the existing diagnostics or a deliberate backend event contract, rather than raw provider-specific packet counts.

## Reproduction Assessment

Not applicable: this is a feature request, not a reproduced broken behavior. Source inspection confirms current main lacks a user-facing packet, byte, or last-model-progress monitor in the TUI status line.

## Solution Assessment

Unclear: literal packet counts are probably not the best product surface. The maintainable path is a maintainer UX decision about which liveness signal to surface and whether it belongs in core TUI status, diagnostics, or both.
