## Summary

Keep open: current main and the latest release provide coarse TUI activity, elapsed-time, disconnect, and watchdog signals plus internal model-call diagnostics, but they still do not expose the requested user-facing packet, byte, or last-progress monitor in the TUI status line.

## What This Changes

The issue requests a user-visible liveness or packet/activity indicator in the OpenClaw status line so users can tell whether a model call is still active or stuck.

## Best Possible Solution

If maintainers want this, define a compact TUI liveness UX and reuse canonical model-call progress or byte diagnostics rather than making the TUI count provider-specific packets itself.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main lacks a user-facing packet, byte, or last-progress monitor in the TUI status line.

## Solution Assessment

Unclear: the requested packet counter is not necessarily the best solution because existing diagnostics already track safer model-call progress signals. The maintainable path is a maintainer UX decision about what liveness signal to surface and how much diagnostic detail belongs in the status line.
