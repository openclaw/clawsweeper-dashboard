## Summary

Keep open: current main and v2026.6.1 already show coarse TUI activity states, elapsed waiting/streaming status, disconnect handling, and internal model-call progress diagnostics, but they do not provide the requested user-facing model connection or packet/last-byte monitor in the status line.

## What This Changes

The issue asks for a status-line indication of whether OpenClaw is actively exchanging data with a model connection during long or stalled runs.

## Best Possible Solution

Expose a stable, low-noise TUI liveness signal based on existing model-call progress, byte counters, or last-progress age only after maintainers choose the UX boundary with the related runtime silence-watchdog work.

## Reproduction Assessment

Not applicable; this is a feature request for finer-grained connection/activity visibility rather than a broken existing behavior. Source inspection confirms current main has coarse activity states and diagnostics, but no requested packet/last-byte status monitor.

## Solution Assessment

Unclear; packet counts may be the wrong abstraction, while surfacing existing model-call last-progress or response-stream diagnostics may be a cleaner solution. Maintainers should decide whether this belongs in TUI status, gateway diagnostics, or the broader silence-watchdog work.
