## Summary

Keep open: current main and v2026.6.6 still provide coarse TUI activity/disconnect states, a streaming watchdog notice, and internal model-call byte/progress diagnostics, but they do not expose the requested packet, byte, or last-model-progress monitor in the TUI status line.

## What This Changes

Request a TUI status-line liveness monitor that shows whether OpenClaw is still exchanging model-call progress during long-running or timeout-prone runs.

## Best Possible Solution

If maintainers want this, expose a compact TUI last-progress/liveness signal derived from existing model-call progress diagnostics instead of raw provider packet counts.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection verifies current main lacks a user-facing packet, byte, or last-progress monitor in the TUI status line.

## Solution Assessment

Unclear: the packet-count proposal is plausible but probably not the best final UX. The maintainable path is a maintainer decision on a compact liveness signal built from existing diagnostic progress data.
