## Summary

Keep open: current main has useful status/doctor drift diagnostics and supervised `openclaw update` syncs official plugins, but direct package-manager core upgrades can still leave external plugins stale and the remaining warn/refuse/auto-sync policy is not settled. The narrower `plugins update --all` recovery gap is already tracked in https://github.com/openclaw/openclaw/issues/94083 with an open candidate PR, but that does not cover the full direct-upgrade silent-failure class.

## What This Changes

The issue requests upgrade-time or startup-time handling for stale official plugin packages so channel/provider failures are not silent after core upgrades.

## Best Possible Solution

Define the official-plugin drift policy for direct package-manager upgrades, then implement a non-silent startup/status/doctor/update path that either realigns trusted official plugins or gives a clear actionable refusal/warning without running package managers during normal gateway startup.

## Reproduction Assessment

Yes for source-level reproduction: direct package-manager upgrades do not run plugin update/sync, while current drift diagnostics are advisory and install-record based. I did not run the full Ubuntu/global-npm gateway scenario live in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The current status/doctor warning and supervised updater sync are useful partial fixes, but they do not fully solve direct-upgrade silent channel/provider failure.
