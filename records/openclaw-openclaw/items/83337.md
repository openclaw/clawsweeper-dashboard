## Summary

Keep open: this looks like a real upgrade-diagnostics gap, not a cleanup-close candidate. Current main has a guarded plugin-sync/convergence path for `openclaw update`, but the reported manual global package upgrade path can bypass that, and the inspected compatibility checks do not clearly catch an older active official plugin package before the channel degrades.

## What This Changes

The issue asks OpenClaw to keep official plugin packages aligned with core upgrades or emit an actionable incompatibility warning instead of silently disabling a configured channel.

## Best Possible Solution

Add a narrow compatibility diagnostic for active official external plugins whose installed package is stale or incompatible with the host, and make the recommended repair path explicit without running package managers during gateway startup.

## Reproduction Assessment

No high-confidence live reproduction was established. The source path is credible because manual global package updates bypass `openclaw update` plugin sync/convergence, while current registry checks only enforce a plugin's minimum host version.

## Solution Assessment

Unclear. Auto-upgrading every plugin during startup would conflict with the documented verify-only startup model, so the safer maintainer direction is an explicit update/doctor convergence path plus clear warnings or refusal for active incompatible official plugin drift.
