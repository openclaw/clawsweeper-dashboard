## Summary

Keep open: this still looks like a real upgrade-diagnostics gap. Current main has guarded plugin sync and post-core convergence for `openclaw update`, but the documented manual package-manager upgrade path can bypass that, and the inspected runtime compatibility checks do not clearly warn when an older official plugin package remains installed before a channel appears disabled.

## What This Changes

The issue reports that manually upgrading OpenClaw core can leave installed official plugins on older versions, causing channels such as Discord to become silently disabled instead of warning or repairing compatibility drift.

## Best Possible Solution

Add an explicit compatibility diagnostic for active official external plugins whose installed package is stale or incompatible with the host, and route repair through `openclaw update` or `openclaw doctor --fix` rather than mutating packages during gateway startup.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the reported path because manual global package updates bypass `openclaw update` plugin convergence, while current compatibility checks do not clearly warn on an older official plugin package built for an older core.

## Solution Assessment

Unclear until maintainers choose the policy for manual global upgrades and exact official plugin pins. Auto-upgrading plugins during startup would conflict with the verify-only startup model, so a warning/refusal plus explicit repair path is the safer direction.
