## Summary

Keep open: current main has partial drift diagnostics in status/doctor, but the reported raw/global upgrade and installed-package mismatch path is still not fully aligned, refused, or surfaced at channel startup/list time. A related QQBot report was closed into this issue, so this remains the canonical thread for the broader policy decision.

## What This Changes

The issue asks OpenClaw to prevent or clearly diagnose official plugin/core version drift after core upgrades because stale channel plugins can appear disabled or stop working silently.

## Best Possible Solution

Define the official-plugin drift policy for raw/global upgrades and pins, then implement a non-silent repair or refusal path using the existing install-record, doctor, status, and plugin-update surfaces without running package managers during normal startup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live current-main run in this review, but the source path is credible: direct package-manager upgrades do not sync plugins, while current diagnostics are advisory and depend on install-record state rather than a guaranteed startup/channel failure check.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide policy for exact official pins and raw/global upgrades; the current doctor/status drift warning is a useful partial fix but not a complete solution for the silent channel-failure class.
