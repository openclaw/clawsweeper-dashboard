## Label Justifications

- `P1`: The report describes a post-upgrade channel workflow where Discord can appear disabled or stop delivering messages with no actionable warning.
- `impact:message-loss`: The affected failure mode is a silently disabled or nonfunctional messaging channel after upgrade.

## Summary

Keep open: current main has guarded plugin convergence for `openclaw update`, but manual global package upgrades and package postinstall can leave old external plugin packages in place, and the inspected compatibility/status/channel-list paths do not yet turn that into an actionable drift warning before channel failure.

## What This Changes

Report or repair plugin/core version drift after core upgrades so external channels do not silently become disabled or nonfunctional.

## Best Possible Solution

Add a narrow compatibility diagnostic for active official external plugins whose installed package is stale or incompatible with the host, and surface clear repair guidance through the chosen status/doctor/update/channel path without running package managers during gateway startup.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the path: manual global upgrades bypass `openclaw update` plugin convergence, postinstall never installs plugin dependencies, and current compatibility checks only catch plugins requiring a newer host.

## Solution Assessment

Unclear. Auto-upgrading plugins during gateway startup would conflict with the documented verify-only startup model, so the safer direction is an explicit update/doctor/status diagnostic with maintainer-approved policy for official plugin pins.
