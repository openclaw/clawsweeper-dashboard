## Label Justifications

- `P1`: The issue describes a post-upgrade channel workflow silently becoming disabled, which can stop real user message delivery until manually repaired.
- `impact:message-loss`: The reported failure mode is a Discord channel appearing disabled or non-functional after upgrade, causing inbound/outbound channel messages to stop.

## Summary

Keep open: current main has a guarded `openclaw update` plugin-sync path, but the reported direct package-manager upgrade with an older official plugin can still bypass that path and lacks a clear stale-plugin compatibility diagnostic.

## What This Changes

The issue requests that OpenClaw detect or repair stale official plugin packages after a core upgrade instead of letting channels fail silently.

## Best Possible Solution

Add a narrow operator-facing compatibility diagnostic for active official external plugins whose installed package is stale or incompatible with the host, and document the supported repair path without running package managers during gateway startup.

## Reproduction Assessment

No high-confidence live reproduction was established. The source path is credible because manual global package upgrades bypass `openclaw update` plugin sync/convergence, while current runtime compatibility checks only enforce a plugin's minimum host version.

## Solution Assessment

Unclear. Auto-upgrading plugins during startup conflicts with the verify-only startup model, so the safer direction is a clear diagnostic plus explicit `openclaw update`, `openclaw plugins update`, or `openclaw doctor --fix` repair guidance after maintainers decide exact-pin policy.
