## Label Justifications

- `P1`: The PR targets a user-facing wake/system-event delivery gap, but as written it can misroute or clear prompt-visible context across sessions.
- `merge-risk: 🚨 message-delivery`: Merging as-is could let an unrelated non-main turn consume a main-session wake event before the intended session sees it.
- `merge-risk: 🚨 session-state`: The diff destructively moves queued system-event context across session keys, so the wrong fallback key or scope can mis-associate session state.
- `merge-risk: 🚨 security-boundary`: The fallback resolves the configured default agent's main key instead of the active agent's key, creating a cross-agent prompt-context movement risk.

## Summary

Keep open. The underlying delivery gap is still source-reproducible, but this branch is not merge-ready because the proposed fallback can drain the wrong agent/session queue, bypass current selected-drain behavior, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to also merge the canonical main-session queue into the active session queue and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, consume only selected generic fallback events, and either restrict the behavior to approved DM isolation scopes or document and test the broader non-main policy explicitly.

## Reproduction Assessment

Yes, at source level. Current main queues untargeted wake/system-event entries under a main session key while per-channel-peer DMs resolve to a different active key, and formatted draining consumes only the active key.

## Solution Assessment

No, not as currently proposed. Receiver-side fallback is plausible, but it should use the active agent's main key, preserve selected-event consumption, and get maintainer approval for any non-DM scope expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selected fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting that context outside the intended conversation unless maintainers explicitly approve and test the broader policy.
  - confidence: 0.78
