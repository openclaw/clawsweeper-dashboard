## Summary

Keep open: the linked bug is still source-reproducible on current main, but this branch is not merge-ready because it drains the configured default main queue wholesale, needs a maintainer scope decision, and lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge canonical main-session events into a non-main DM session drain and adds a per-channel-peer regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id and selected generic-event consumption, with regression coverage for per-channel-peer delivery, multi-agent isolation, and any explicitly approved non-main scopes.

## Reproduction Assessment

Yes, at source level. /hooks/wake and system-event enqueue to a main session key, per-channel-peer DMs use agent:<id>:telegram:direct:<peer>, and current drainFormattedSystemEvents only consumes params.sessionKey; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Receiver-side fallback is plausible, but this branch should derive the fallback from the active agent, preserve selective event consumption, and either narrow the scope or get maintainer-approved coverage for broader non-main delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback draining to the active agent:** `src/auto-reply/reply/session-system-events.ts:84`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent encoded in the active session. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, moving or clearing system-event context across agents; derive the fallback from `params.sessionKey` and use the agent-scoped main resolver.
  - confidence: 0.91
- **[P2] Constrain the main-key fallback policy:** `src/auto-reply/reply/session-system-events.ts:85-89`
  - body: The `mainSessionKey !== params.sessionKey` guard matches group, channel, thread, and custom sessions as well as isolated DMs. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting the context outside the intended conversation; restrict this to approved DM scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:89`
  - body: Current main leaves exec-completion events queued for the heartbeat path by peeking and consuming only selected generic events. The fallback uses `drainSystemEventEntries(mainSessionKey)`, so a normal reply turn can consume heartbeat-owned entries from the main queue; use the same selected-consumption path for the fallback.
  - confidence: 0.84
