## Summary

Keep open: current main still source-reproduces the main-session queue gap for isolated DM sessions, but this PR is not merge-ready because it destructively drains the configured default agent's main queue for every non-main session and lacks inspectable real behavior proof.

## What This Changes

The PR merges the canonical main-session system-event queue into formatted event draining for non-main sessions and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port a receiver-side fallback that derives the active agent's main key from the active session, preserves selected generic-event consumption, and either constrains scope to approved DM isolation or adds explicit tests/docs for broader non-main delivery.

## Reproduction Assessment

Yes at source level: current main queues system-event and wake text to a main-session key, per-channel-peer DMs use a different active key, and formatted draining only consumes the active key. I did not run a live Telegram or Discord reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed. Receiver-side fallback is plausible, but it should use the active agent's main key, preserve selected event consumption, and get maintainer approval for any non-DM scope expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
