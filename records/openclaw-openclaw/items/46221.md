## Summary

Keep this PR open: it targets a source-reproducible system-event routing bug that current main still appears to have, but the branch is not merge-ready because the fallback can drain the wrong agent's main queue, applies to broader session shapes without approved coverage, and lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge canonical main-session events into non-main session drains and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, preserve selected generic-event consumption, and add regression coverage for per-channel-peer delivery, multi-agent isolation, and any approved broader scope.

## Reproduction Assessment

Yes, at source level: main-system-event enqueue paths target canonical main keys while per-channel-peer DMs use different keys, and current main drains only the active key. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selective queue consumption, and either restrict or explicitly document and test broader non-main delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent encoded in the active session. A session such as `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`; derive the fallback from `params.sessionKey` with the agent-scoped resolver.
  - confidence: 0.91
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. The fallback drains the whole main queue, so a normal reply turn can steal heartbeat-owned entries; use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event; restrict this to approved direct-message scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
