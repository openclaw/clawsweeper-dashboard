## Summary

Keep open. Current main still drains formatted system events only from the active session key, so the reported per-channel-peer delivery gap is real, but the PR patch is not merge-ready because its fallback uses the configured default main session and broadens delivery to every non-main session without enough policy and regression coverage.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main active sessions and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict fallback draining to approved direct-message scopes or explicitly document and test the broader non-main-session policy across multi-agent, group, channel, and thread cases.

## Reproduction Assessment

Yes. Enqueue a system event under `resolveMainSessionKey(cfg)` and call `drainFormattedSystemEvents` for `agent:main:telegram:direct:<peer>` with `session.dmScope: "per-channel-peer"`; current main drains only the active key, so the main-key event remains queued. I did not run tests because this review is read-only, but the source path is direct and deterministic.

## Solution Assessment

No. Receiver-side merging is the right direction, but this patch should derive the fallback key from the active session's agent id and should not broaden delivery to every non-main session without maintainer-approved policy and coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the agent encoded in the active session key. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from `params.sessionKey` before merging.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom session keys. That lets an unrelated non-DM turn consume and clear main-session wake context, so restrict this fallback to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff can move untrusted system-event context across agent or conversation boundaries unless fallback key derivation and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using `resolveMainSessionKey(params.cfg)` for fallback draining lets a non-default active agent drain the configured default agent's main queue while missing its own agent-scoped queue; `/hooks/wake` enqueues untrusted text, so this can inject context into the wrong prompt.
  - confidence: 0.86
