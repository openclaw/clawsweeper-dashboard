## Summary

Keep open: this PR addresses a real source-level wake/system-event delivery gap, but the branch is conflicting and the proposed fallback can drain the wrong agent's main queue and broaden prompt-visible delivery beyond the proven DM scope.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main session drains and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port a receiver-side fallback onto current main that derives the fallback main key from the active session's agent id, preserves selected-event consumption, and either restricts scope to approved DM isolation or adds maintainer-approved coverage for broader non-main delivery.

## Reproduction Assessment

Yes at source level: current main queues `/hooks/wake` and `system-event` text to the main-session key while per-channel-peer DMs resolve to a different active key, and formatted draining consumes only selected events from the active key. I did not run a live Telegram/Discord reproduction in this read-only pass.

## Solution Assessment

No, not as currently proposed. Receiver-side fallback is plausible, but the patch should use the active agent's main key, preserve selected event consumption, and get maintainer approval for any non-DM scope expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selected fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
