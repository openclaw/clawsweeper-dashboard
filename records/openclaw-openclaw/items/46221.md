## Summary

Keep open: the underlying bug is still source-reproducible on current main and the PR is useful, but the patch can drain the wrong agent's main queue, over-deliver main-key events to unrelated non-main sessions, drops current selective-consumption behavior, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into an active non-main session and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, selected generic-event consumption, and focused tests for per-channel-peer delivery, multi-agent isolation, and any explicitly approved broader scopes.

## Reproduction Assessment

Yes. Source inspection on current main shows system-event and wake paths enqueue to the main key while per-channel-peer DMs build agent:<id>:telegram:direct:<peer> and drainFormattedSystemEvents only consumes params.sessionKey; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selective generic-event consumption, and either narrow or get approval and test coverage for all non-main session delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the fallback key from the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` chooses the configured default or first agent, so a turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while leaving `agent:ops:main` stuck. Parse the active session's agent id and build that agent's main key before merging queues.
  - confidence: 0.91
- **[P2] Limit fallback draining to approved session scopes:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The guard matches every non-main key, including group, channel, thread, and custom sessions. That can inject and clear a main-session wake in an unrelated conversation; restrict to intended direct-message isolation scopes or add maintainer-approved behavior and tests for the wider policy.
  - confidence: 0.78
- **[P2] Preserve selective main-queue consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion events queued for the heartbeat path by peeking and consuming only selected generic events. The fallback uses `drainSystemEventEntries(mainSessionKey)`, so a normal reply turn can consume heartbeat-owned entries; use the same selected-consumption path for the fallback.
  - confidence: 0.83
