## Summary

Keep open: this PR targets a real source-reproducible system-event delivery bug, but the current patch is not merge-ready because it can drain the default agent’s main queue from another agent’s session, applies the fallback to every non-main session shape, and bypasses current main’s selective event consumption behavior.

## What This Changes

The PR changes formatted system-event draining to also merge the canonical main-session queue into a non-main active session and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, consume only approved generic fallback events, and add regression coverage for per-channel-peer delivery, multi-agent isolation, and any explicitly approved non-main scopes.

## Reproduction Assessment

Yes, from source inspection. Wake/system-event paths enqueue to the canonical main key, current main drains only params.sessionKey, and per-channel-peer DM routing produces a different key.

## Solution Assessment

No. Receiver-side fallback draining is a plausible direction, but this implementation should derive the fallback from the active session's agent id, preserve selective consumption, and avoid broad non-main delivery without maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default or first agent's main key, not the agent encoded in the active session key. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, crossing agent boundaries and leaving the intended queue stuck.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That can deliver and clear main-session wake context in an unrelated conversation, so restrict the fallback to approved DM scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.8
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:88`
  - body: Current main leaves exec-completion events queued for the heartbeat path, but the fallback drains the whole main queue. Use peek plus selected consumption for fallback events too so normal reply prompting does not consume dedicated heartbeat events.
  - confidence: 0.83
