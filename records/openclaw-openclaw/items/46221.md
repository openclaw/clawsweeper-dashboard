## Summary

Keep open: the PR targets a real source-reproducible system-event routing gap, but the current patch is not merge-ready because it can drain the wrong agent's main queue, over-broadly applies to all non-main sessions, and lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to also merge canonical main-session events into non-main session drains and adds a per-channel-peer system-event regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, preserve selective generic-event consumption, and add regression coverage plus real Telegram-visible proof for the approved scope.

## Reproduction Assessment

Yes, at source level: system-event and wake handlers enqueue to the main key, per-channel-peer DMs use a distinct agent:<id>:telegram:direct:<peer> key, and current main drains only params.sessionKey. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selected queue consumption, and restrict or explicitly approve the broader non-main-session behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drains to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` chooses the configured default or first agent, not the agent encoded in the active session key. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback main key from `params.sessionKey`.
  - confidence: 0.91
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. The fallback drains the entire main queue, so a normal reply turn can steal heartbeat-owned entries; use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain fallback delivery to approved session scopes:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That can inject a main-session wake into an unrelated conversation and clear it before the intended session sees it, so restrict the fallback or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
