## Summary

Keep open. The underlying event-routing bug is source-reproducible and current main still drains only the active session key, but this PR is not merge-ready because the fallback can drain the wrong agent's main queue, broadens delivery to all non-main sessions without approved scope, and lacks inspectable real Telegram proof.

## What This Changes

This PR changes formatted system-event draining to merge canonical main-session events into a non-main session drain and adds a per-channel-peer regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id and selected generic-event consumption, with coverage for per-channel-peer delivery, multi-agent isolation, and any explicitly approved non-main scopes.

## Reproduction Assessment

Yes, at source level. System-event and /hooks/wake enqueue to the canonical main session, per-channel-peer DMs use agent:<id>:telegram:direct:<peer>, and current drainFormattedSystemEvents consumes only params.sessionKey; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Receiver-side fallback is plausible, but this patch should derive the fallback from the active agent, preserve selective event consumption, and either narrow the scope or add maintainer-approved coverage for broader non-main delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default or first agent's main key, not the agent encoded in the active session. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`; derive the fallback from `params.sessionKey` with the agent-scoped resolver.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-88`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event; restrict this fallback to approved direct-message scopes or add explicit maintainer-approved coverage for broader non-main delivery.
  - confidence: 0.78
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:88`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by selecting only generic events, but the fallback drains the entire main queue. Consume the same selected generic set from the fallback key so normal reply turns do not steal heartbeat-owned events.
  - confidence: 0.84
