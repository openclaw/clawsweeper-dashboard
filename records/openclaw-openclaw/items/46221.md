## Summary

Keep open: current main still has the source-level system-event delivery gap for isolated DM sessions, but this PR is not merge-ready because its fallback drains the configured default main key broadly, predates current main's selective generic-event consumption contract, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a fallback drain of the canonical main-session system-event queue into formatted active-session event draining and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key, selected generic-event consumption, explicitly approved scope coverage, and redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes, at source level: main enqueue paths still write wake/system events to an agent main key, while per-channel-peer DMs use a different active session key and current formatted draining only consumes the active key. I did not run a live Telegram or Discord reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this branch should be ported to current main with active-agent key derivation, selected fallback consumption, and either constrained direct-message scope or explicit maintainer-approved broader coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal those heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
