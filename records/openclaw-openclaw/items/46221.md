## Summary

This PR should stay open: current main still has the reported main-key delivery gap, but the proposed patch is not merge-ready because it can drain the wrong agent's queue and broadens fallback delivery beyond an approved scope.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main active sessions and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, consume only selected generic fallback events, and either restrict or explicitly document and test the approved non-main delivery scope.

## Reproduction Assessment

Yes. Source inspection shows main enqueue paths still write to the canonical main session key, while per-channel-peer direct-message sessions drain only their own non-main key on current main.

## Solution Assessment

No. Receiver-side fallback is the right direction, but this patch should derive the fallback from the active session's agent id, preserve selective consumption, and avoid broad non-main delivery without maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent's main key, not the agent encoded in `params.sessionKey`. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session's agent id.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That can deliver and clear main-session wake context in an unrelated conversation, so restrict this to approved direct-message scopes or add explicit maintainer-approved coverage for broader behavior.
  - confidence: 0.8
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:89`
  - body: Current main leaves exec-completion events queued for the heartbeat path, but the fallback drains the whole main queue. Use peek plus selected consumption for fallback events too, so normal reply prompting does not consume dedicated heartbeat events.
  - confidence: 0.83
