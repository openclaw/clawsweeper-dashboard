## Label Justifications

- `P1`: The reported bug can silently strand wake/cron/system events for isolated DM sessions, while the PR's current fix can affect message and session boundaries.
- `merge-risk: 🚨 message-delivery`: Merging as-is could consume a main-session wake from an unrelated non-main turn, causing the intended session to miss that event.
- `merge-risk: 🚨 session-state`: The proposed fallback can inject system context into the wrong conversation and clear queued state from another session.
- `merge-risk: 🚨 security-boundary`: The fallback resolves the configured default agent's main key instead of the active agent's key, creating a cross-agent context movement risk.

## Summary

Keep this PR open: the reported main-key system-event delivery gap is still source-valid on current main, but the proposed branch is stale/conflicting, lacks inspectable real behavior proof, and introduces scope/agent-boundary risks that should be fixed before merge.

## What This Changes

This PR changes formatted system-event draining to merge the canonical main-session queue into a non-main active session and adds a system-events regression test for per-channel-peer DMs.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key, selected generic-event consumption, explicitly approved scope coverage, and redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes, at source level: main enqueue paths still write wake/system events to an agent main key, while per-channel-peer DMs use a different active session key and current formatted draining only consumes the active key. I did not run a live Telegram or Discord reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this branch should be ported with active-agent key derivation, selected fallback consumption, and maintainer-approved scope before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal those heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
