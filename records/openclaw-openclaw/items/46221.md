## Summary

Keep open. The receiver-side bug is still source-reproducible on current main, but this PR is not merge-ready because its fallback drains the default main queue too broadly, misses active-agent isolation, and lacks inspectable real behavior proof.

## What This Changes

The PR modifies formatted system-event draining to also merge the canonical main-session queue into the active session drain and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key, selected generic-event consumption, approved scope coverage, and redacted Telegram or terminal proof.

## Reproduction Assessment

Yes, at source level: current main enqueues `/hooks/wake` under a main-session key while peer-scoped DMs use `agent:<id>:telegram:direct:<peer>`, and formatted draining consumes selected events only from `params.sessionKey`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback from the active session's agent, preserve selected queue consumption, and either constrain or explicitly test the broader non-main-session scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope fallback draining to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` picks the configured default or first agent, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal those heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Limit fallback draining to approved session scopes:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
