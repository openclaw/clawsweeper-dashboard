## Summary

Keep open: current main still has the source-level delivery gap, and this PR is useful, but the patch is not merge-ready because it can drain the wrong agent's main queue, applies broader non-main delivery without maintainer approval, and lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge canonical main-session events into non-main session drains and adds a regression test for per-channel-peer DM delivery.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, preserve selected generic-event consumption, and add tests for per-channel-peer delivery, multi-agent isolation, and any explicitly approved broader scopes.

## Reproduction Assessment

Yes at source level: current main enqueues wake/system events to canonical main keys while per-channel-peer DMs use a separate session key and the formatter only consumes the active key. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, consume only selected generic events from the fallback queue, and either restrict or explicitly document and test broader non-main delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback draining to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent encoded in the active session. A session like `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`; derive the fallback from `params.sessionKey` with the agent-scoped resolver.
  - confidence: 0.91
- **[P2] Constrain or prove non-main fallback delivery:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event; restrict this to approved direct-message scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:88`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. The fallback drains the entire main queue, so a normal reply turn can steal heartbeat-owned events; use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
