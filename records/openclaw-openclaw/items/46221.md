## Summary

Keep open: this PR addresses a source-reproducible bug, but the current patch is not merge-ready because it can drain the default agent's main queue from another agent, applies to every non-main session without maintainer scope approval, bypasses current selective event consumption, and still lacks real Telegram behavior proof.

## What This Changes

The PR changes formatted system-event draining to also merge canonical main-session events into non-main session drains and adds a regression test for per-channel-peer DM delivery.

## Best Possible Solution

Port the receiver-side fallback onto current main with active-agent main-key resolution, selected generic-event consumption, and tests for per-channel-peer delivery, multi-agent isolation, and any explicitly approved non-main scopes.

## Reproduction Assessment

Yes, at source level. Current main enqueues system-event and wake-hook entries to the canonical main key, per-channel-peer DMs use `agent:<id>:telegram:direct:<peer>`, and formatted drains consume only the active key; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selected generic-event consumption, and either restrict or explicitly document and test broader non-main delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` chooses the configured default or first agent, not the agent encoded in the active session. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`; derive the fallback from `params.sessionKey` with the agent-scoped resolver.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-88`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event; restrict this fallback to approved direct-message scopes or add maintainer-approved coverage for broader non-main delivery.
  - confidence: 0.78
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:88`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events, but the fallback drains the entire main queue. Consume the same selected generic set from the fallback key so normal reply turns do not steal heartbeat-owned events.
  - confidence: 0.84
