## Summary

Keep this PR open: the underlying delivery bug is still source-reproducible on current main, but the proposed fallback crosses sensitive session boundaries, is stale against the current selective-consumption path, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to also merge canonical main-session events into an active non-main session and adds a regression test for per-channel-peer DM delivery.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, preserve selective generic-event consumption, and add coverage for per-channel-peer delivery, multi-agent isolation, and any explicitly approved non-main scopes.

## Reproduction Assessment

Yes, from source inspection. Current main enqueues wake/system events to the canonical main key, drains formatted events only from params.sessionKey, and per-channel-peer DMs build a different active session key.

## Solution Assessment

No. Receiver-side fallback draining is plausible, but this implementation should derive the fallback from the active session's agent id, preserve selective consumption, and narrow or explicitly document/test the broader non-main scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent's main key, not the agent encoded in the active session key. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, crossing agent boundaries and leaving the intended queue stuck.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That can deliver and clear main-session wake context in an unrelated conversation, so restrict the fallback to approved DM scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.8
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion events queued for the heartbeat path, but the fallback drains the whole main queue. Use peek plus selected consumption for fallback events too so normal reply prompting does not consume dedicated heartbeat events.
  - confidence: 0.83
