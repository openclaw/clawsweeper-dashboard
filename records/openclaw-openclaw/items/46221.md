## Summary

Keep open. Current main still has the reported active-session delivery gap, but this PR is not merge-ready because its fallback drains the configured default agent's main queue, applies to every non-main session shape, and predates current main's selective system-event draining behavior.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main active sessions and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id, preserve generic-only event selection, and add regression coverage for per-channel-peer delivery, multi-agent isolation, and whichever non-main scopes maintainers approve.

## Reproduction Assessment

Yes. Enqueue a generic system event under `resolveMainSessionKey(cfg)` and call `drainFormattedSystemEvents` for `agent:main:telegram:direct:<peer>` with `session.dmScope: "per-channel-peer"`; current main consumes only the active key, so the main-key event remains queued.

## Solution Assessment

No. Receiver-side fallback draining is the right direction, but this implementation should derive the fallback key from the active session, should not silently broaden delivery to every non-main session, and should preserve current main's selective drain behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the fallback key from the active session:** `src/auto-reply/reply/session-system-events.ts:84`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent in `params.sessionKey`. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback with `resolveAgentIdFromSessionKey(params.sessionKey)` plus `resolveAgentMainSessionKey`.
  - confidence: 0.91
- **[P2] Constrain fallback draining to approved scopes:** `src/auto-reply/reply/session-system-events.ts:85-89`
  - body: The `mainSessionKey !== params.sessionKey` guard applies to group, channel, thread, and custom session keys, not just isolated DMs. That lets an unrelated non-DM turn consume and clear main-session wake context, so restrict this to approved scopes or add maintainer-approved coverage for the broader policy.
  - confidence: 0.8
- **[P2] Preserve selective consumption for fallback events:** `src/auto-reply/reply/session-system-events.ts:89`
  - body: Current main leaves exec-completion system events queued for the dedicated heartbeat path, but this fallback drains the entire main queue. Port the merge through `peekSystemEventEntries` and `consumeSelectedSystemEventEntries` so fallback draining does not consume exec completions into a normal reply prompt.
  - confidence: 0.83

## Security Review

Status: needs_attention

  - body: Using `resolveMainSessionKey(params.cfg)` for fallback delivery drains the configured default agent's main queue even when the active session belongs to another agent, which can inject or remove system-event context across agent boundaries.
