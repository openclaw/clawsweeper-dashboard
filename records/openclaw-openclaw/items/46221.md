## Summary

Keep open. The reported per-channel-peer delivery gap is still source-reproducible on current main, but the proposed branch is not merge-ready because it drains the configured default main key rather than the active agent's main key, broadens delivery to every non-main session, bypasses current selective event consumption, and still lacks real Telegram-visible proof.

## What This Changes

The branch changes formatted system-event draining to also merge the canonical main-session queue into non-main session drains and adds a per-channel-peer regression test.

## Best Possible Solution

Port a receiver-side fallback onto current main using the active session's agent id, selected-event consumption, and maintainer-approved scope coverage, then require redacted Telegram proof for /hooks/wake or openclaw system event reaching the active DM.

## Reproduction Assessment

Yes, at source level: gateway system-event and wake hooks enqueue to the main session key, per-channel-peer direct messages use an agent:<id>:<channel>:direct:<peer> key, and current main drains formatted system events only from params.sessionKey. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selected queue consumption, and restrict or explicitly approve the broader non-main-session behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the active session's agent. For a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from `params.sessionKey` instead.
  - confidence: 0.91
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. The fallback drains the entire main queue, so a normal reply turn can steal heartbeat-owned entries; use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting that context outside the intended conversation; restrict this fallback or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
