## Summary

Keep open: current main still has the reported main-key system-event delivery gap, but this branch is not merge-ready because the fallback can drain the wrong agent, applies to every non-main session, consumes the whole fallback queue, and lacks inspectable real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into a non-main active session and adds a per-channel-peer DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent id, selectively consume only generic fallback events, and cover per-channel-peer delivery plus multi-agent isolation and any approved non-main scopes.

## Reproduction Assessment

Yes, from source inspection. The enqueue paths write to a main-session key while per-channel-peer DMs use a different key, and current main drains only the active key.

## Solution Assessment

No. Receiver-side fallback draining is the right direction, but this branch should derive the fallback key from the active session's agent id, preserve selective consumption, and avoid broad non-main delivery without maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent encoded in the active session key. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session's agent id.
  - confidence: 0.91
- **[P2] Constrain main-key fallback scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That can deliver and clear a main-session wake event in an unrelated conversation, so restrict this to approved direct-message scopes or add explicit maintainer-approved coverage for broader behavior.
  - confidence: 0.8
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion events queued for the heartbeat path, but the fallback drains the whole main queue. Use peek plus selected consumption for fallback events too, so normal reply prompting does not consume heartbeat-owned exec completion events.
  - confidence: 0.83
