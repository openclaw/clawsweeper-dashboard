## Summary

Keep open: current main still has a source-level system-event delivery gap for isolated DM session keys, but this PR is not merge-ready because the fallback drains the default agent's main queue, consumes the fallback queue wholesale, needs scope approval, lacks real behavior proof, and is currently dirty against main.

## What This Changes

The PR modifies formatted system-event draining to also drain the canonical main-session queue and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key, selected generic-event consumption, explicit approved scope coverage, and redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes, at source level: current main still enqueues system events under a main-session key while per-channel-peer DMs use `agent:<id>:<channel>:direct:<peer>`, and formatted draining consumes only `params.sessionKey`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback from the active session's agent, preserve selected queue consumption, and either constrain or explicitly test the broader non-main-session scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default or first agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session's agent id before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface; restrict this to approved direct-message scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
