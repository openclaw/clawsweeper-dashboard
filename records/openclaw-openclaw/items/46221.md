## Summary

Keep open: the linked system-event delivery bug remains source-reproducible on current main, but this PR is not merge-ready because the fallback drains the configured default agent's main queue, applies to every non-main session without a settled scope decision, and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes formatted system-event draining to merge canonical main-session queued events into non-main session drains and adds a per-channel-peer regression test.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key and selected generic-event consumption, then add multi-agent/scope coverage plus redacted Telegram proof.

## Reproduction Assessment

Yes, at source level: main enqueue paths still queue under a main key, per-channel-peer DMs use agent:<id>:<channel>:direct:<peer>, and current main formats only params.sessionKey. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The receiver-side fallback is plausible, but this patch should derive the fallback key from the active session's agent, preserve selected queue consumption, and get explicit coverage or approval for the broader non-main-session scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent that owns `params.sessionKey`. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session's agent id.
  - confidence: 0.91
- **[P2] Preserve selective system-event consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface; restrict it to approved DM scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
