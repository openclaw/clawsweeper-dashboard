## Summary

Keep open. The underlying bug is still source-reproducible on current main and the PR is useful, but the patch is not merge-ready: it drains the configured default main key instead of the active agent's main key, broadens delivery to all non-main sessions without approved coverage, and lacks inspectable real behavior proof.

## What This Changes

The PR merges canonical main-session system events into non-main session drains and adds regression coverage for per-channel-peer DM delivery.

## Best Possible Solution

Port a receiver-side fallback that derives the main key from the active session's agent id, preserves selective generic-event consumption, and covers per-channel-peer delivery, multi-agent isolation, and any approved broader non-main scopes.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main enqueues /hooks/wake and system-event notices to the main key while per-channel-peer DMs use a different active key and formatted replies drain only that active key. I did not run a live Telegram/Discord repro in this read-only review.

## Solution Assessment

No. Receiver-side fallback draining is plausible, but this implementation should derive the fallback key from the active session's agent id, preserve selective consumption, and either narrow or explicitly document/test the broader non-main-session behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default or first agent, not the agent encoded in the active session key. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, moving wake context across agents; derive the agent from `params.sessionKey` and use the agent-scoped main key.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting that context outside the intended conversation; restrict this to approved DM scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion events queued for the heartbeat path by consuming only selected generic entries. The fallback drains the whole main queue, so a normal reply turn could consume dedicated heartbeat events before heartbeat delivery; peek/select/consume fallback entries with the same filter.
  - confidence: 0.83
