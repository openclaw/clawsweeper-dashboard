## Summary

This PR should stay open because it targets a real system-event delivery gap, but the current branch is not merge-ready: it can drain the wrong agent's main queue, it bypasses current main's selective system-event consumption, and the broader non-main-session scope still needs maintainer approval plus real behavior proof.

## What This Changes

The PR adds a fallback main-session queue drain to formatted system-event handling and a regression test for per-channel-peer DM delivery.

## Best Possible Solution

Port the receiver-side fallback onto current main using the active session's agent-scoped main key, selected generic-event consumption, maintainer-approved scope coverage, and redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes at source level: current main enqueue paths still write wake/system events to main-session keys, per-channel-peer DMs use a different active session key, and formatted draining only consumes the active key. I did not run a live Telegram or Discord reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed. Receiver-side fallback is a plausible fix direction, but the PR should use the active agent's main key, preserve selected event consumption, and get maintainer approval for any non-DM scope expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` resolves the configured default agent's main key, not the agent that owns `params.sessionKey`. In a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session key before merging queues.
  - confidence: 0.91
- **[P2] Preserve selective fallback consumption:** `src/auto-reply/reply/session-system-events.ts:87`
  - body: Current main leaves exec-completion entries queued for the dedicated heartbeat path by consuming only selected generic events. Draining the entire fallback main queue can steal those heartbeat-owned entries during a normal reply turn, so use the same selected-consumption contract for the fallback key.
  - confidence: 0.84
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, thread, and custom sessions. That can clear a main-session wake event from an unrelated conversation and inject that context outside the intended surface, so restrict this to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
