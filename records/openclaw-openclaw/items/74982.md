## Summary

Keep open. Current main still drops the context-level Telegram `audioAsVoice` flag in both legacy outbound sends and registered plugin `sendPayload`; this PR addresses a remaining gap that is not implemented on main. I found no concrete patch-level blocker or security concern, but it needs maintainer rebase/mergeability review against current main before merge.

## What This Changes

The PR forwards Telegram outbound `audioAsVoice` into existing `asVoice` send options in the legacy outbound adapter and registered plugin `sendPayload`, with outbound adapter regression tests.

## Best Possible Solution

Land a rebased version of this Telegram plugin fix that forwards context-level `audioAsVoice` to `asVoice` in both the legacy outbound adapter and registered plugin `sendPayload`, with targeted coverage for the affected outbound paths.

## Reproduction Assessment

Yes. Code inspection on current main gives a high-confidence reproduction path: call Telegram `sendMedia` or registered `sendPayload` with context `audioAsVoice: true`; the shared outbound layer can supply that flag, but the Telegram adapters do not forward it and the sender only uses voice delivery when `asVoice` is true.

## Solution Assessment

Yes. The PR uses the narrow maintainable fix: map an existing generic outbound context flag to an existing Telegram send option inside the Telegram plugin, without adding Telegram-specific logic to shared core delivery.
