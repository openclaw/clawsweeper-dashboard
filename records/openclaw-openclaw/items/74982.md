## Summary

Keep open. The patch is useful and fixes part of the Telegram voice-note propagation bug, but it still leaves the registered plugin `sendMedia` path dropping `audioAsVoice` and lacks the required user-facing changelog entry.

## What This Changes

The PR threads Telegram outbound `audioAsVoice` into `asVoice` for the legacy `sendMedia`/`sendPayload` adapter and the registered plugin `sendPayload`, with outbound-adapter regression tests.

## Best Possible Solution

Land a narrow Telegram-plugin fix that preserves the existing context-level `audioAsVoice` flag through every legacy and registered outbound media/payload path, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. By code inspection, generic delivery passes `audioAsVoice: true` into media overrides, Telegram registered/legacy adapters must convert that to `asVoice`, and the low-level Telegram sender only uses `sendVoice` when `asVoice` is true.

## Solution Assessment

No, not yet. Mapping the existing context flag to Telegram `asVoice` inside the Telegram plugin is the right boundary, but this PR is incomplete until registered `sendMedia` and the changelog are covered.
