## Summary

Keep open. Current main has a payload-level Telegram voice-note fix, but the PR still carries useful context-level `audioAsVoice` forwarding work and should not be closed as already implemented; review found a remaining registered send-media gap plus the required changelog omission.

## What This Changes

The PR forwards Telegram outbound `audioAsVoice` into `asVoice` for legacy `sendMedia`/`sendPayload` and registered `sendPayload`, with outbound-adapter regression tests.

## Best Possible Solution

Land a narrow Telegram-plugin fix that forwards context-level `audioAsVoice` through every Telegram outbound adapter path, including registered `sendMedia`, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. By code inspection, passing a Telegram outbound context with `audioAsVoice: true` into the current registered or legacy media paths does not produce `asVoice: true`, while the low-level sender only uses `sendVoice` when `asVoice` is true.

## Solution Assessment

Partly. Mapping the existing context flag to Telegram's existing `asVoice` option inside the Telegram plugin is the right boundary, but this PR is incomplete until the registered `sendMedia` path and changelog are covered.
