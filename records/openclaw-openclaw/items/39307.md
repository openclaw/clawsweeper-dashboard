## Summary

Keep open: current main has partial compaction visibility through opt-in notices, a bundled hook, Control UI indicators, and internal post-compaction context, but it still does not satisfy the requested default cross-channel visibility or guaranteed assistant self-disclosure/reload-offer behavior.

## What This Changes

The issue asks OpenClaw to visibly signal mid-thread compaction/context reset and have the assistant disclose and recover after compaction.

## Best Possible Solution

Define a durable compaction/reset disclosure policy, then implement consistent reset signaling plus assistant re-grounding/reload-offer behavior across the reply runner, post-compaction recovery, Control UI, supported chat surfaces, docs, and focused tests.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: emit a compaction event with default config and `onBlockReply` remains silent, while post-compaction context is injected internally rather than guaranteed as user-facing disclosure. I did not run a live chat compaction because the current tests and source already lock in the observable gap.

## Solution Assessment

No, the current partial implementation is not yet the full solution. The maintainable path needs a product decision on default visibility, hook/config interaction, customer-facing suppression, and assistant recovery wording before a focused code change is safe.
