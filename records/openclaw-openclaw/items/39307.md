## Summary

Keep open: current main has useful partial compaction visibility, but the default chat path is still intentionally silent and post-compaction recovery is injected as internal context rather than guaranteed user-facing disclosure plus a reload offer.

## What This Changes

Request a visible user signal for mid-thread compaction/reset and explicit assistant recovery behavior after working context changes.

## Best Possible Solution

Define a durable compaction/reset disclosure policy, then implement consistent reset signaling and assistant re-grounding/reload-offer behavior across the reply runner, post-compaction recovery, Control UI, supported chat surfaces, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence current-main path: with default config, a compaction event leaves `onBlockReply` silent, while recovery context is injected internally rather than guaranteed as visible assistant disclosure.

## Solution Assessment

No. The current partial implementation is useful, but the full solution needs a maintainer decision on default visibility, suppression semantics, and recovery wording before a narrow code change is safe.
