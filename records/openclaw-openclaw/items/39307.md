## Summary

Keep open: current `main` has partial compaction visibility through opt-in chat notices, Control UI indicators, compacted-history dividers, and internal post-compaction context refresh, but it still lacks the requested default cross-channel reset signal plus assistant-side disclosure/reload offer. The open related recovery-notice request also shows this needs maintainer product policy before implementation.

## What This Changes

Request default user-visible signaling and assistant-side disclosure/reload-offer behavior when a session compacts or resets mid-thread.

## Best Possible Solution

Define a durable compaction/reset event and disclosure policy, then implement consistent reset signaling and assistant re-grounding/reload-offer behavior across the reply runner, post-compaction recovery, Control UI, supported chat surfaces, docs, and focused tests.

## Reproduction Assessment

Yes for the observable gap: source inspection on current `main` shows default chat compaction notices remain silent and tests lock that behavior. The only assistant recovery text found is internal post-compaction context, not a user-facing disclosure or reload offer.

## Solution Assessment

No current implementation is not the complete solution. The existing opt-in notices, Control UI status, history divider, and internal grounding are useful partial pieces, but default visibility and assistant disclosure wording need maintainer policy direction before a cross-channel implementation.
