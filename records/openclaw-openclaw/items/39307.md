## Summary

Keep open: current main has useful partial compaction visibility, but default chat disclosure is still opt-in and post-compaction recovery is still internal context rather than a guaranteed user-facing reset notice plus reload offer.

## What This Changes

The issue requests visible mid-thread compaction/reset signaling and explicit assistant recovery behavior after compaction.

## Best Possible Solution

Define a durable compaction/reset disclosure policy, then implement consistent reset signaling and assistant re-grounding/reload-offer behavior across the reply runner, post-compaction recovery, Control UI, supported chat surfaces, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence current-main path: with default config, compaction can leave `onBlockReply` silent while recovery context is injected internally rather than guaranteed as visible assistant disclosure.

## Solution Assessment

No. The existing opt-in notices, bundled hook, Control UI indicator, and Discord reaction are useful partial pieces, but the requested full behavior needs maintainer policy on default visibility, suppression, and recovery wording before a narrow implementation is safe.
