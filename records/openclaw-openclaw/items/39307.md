## Summary

Keep open: current main has useful partial compaction visibility, but default chat notices remain opt-in and post-compaction recovery is still an internal system-event prompt rather than a guaranteed user-facing reset disclosure plus reload offer.

## What This Changes

The issue requests visible mid-thread compaction/reset signaling and explicit assistant recovery disclosure after context is refreshed.

## Best Possible Solution

Define a durable compaction/reset disclosure policy, then implement consistent reset signaling and assistant re-grounding/reload-offer behavior across the reply runner, post-compaction recovery, Control UI, supported chat surfaces, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence current-main path: with default config, compaction can leave `onBlockReply` silent while recovery context is injected internally rather than guaranteed as visible assistant disclosure.

## Solution Assessment

No current implementation is the full solution. The existing opt-in notices, bundled hook, Control UI indicator, and Discord reaction are useful partial pieces, but the default visibility, suppression, and recovery wording policy needs maintainer direction.
