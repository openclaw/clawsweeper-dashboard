## Summary

Keep open. Current main still extracts raw WhatsApp `contextInfo`, but it does not normalize `forwardingScore` / `isForwarded` into the WhatsApp inbound message, finalized context, or plugin hook metadata; the related open PR at https://github.com/openclaw/openclaw/pull/46291 only targets agent prompt context and remains blocked.

## What This Changes

Expose WhatsApp forwarded-message metadata from Baileys to plugin `inbound_claim` and `message_received` hook events.

## Best Possible Solution

Add a typed WhatsApp-owned forwarding extractor and batch-safe canonical metadata carrier, then expose sanitized `isForwarded` / `forwardingScore` through `inbound_claim` and `message_received` hook metadata with focused tests.

## Reproduction Assessment

Yes, source-level. Baileys exposes the requested fields and current main can extract raw `contextInfo`, but there is no carrier from WhatsApp normalization into `inbound_claim` or `message_received` metadata.

## Solution Assessment

Yes as a direction, but the implementation shape is not settled. The maintainable path is an additive WhatsApp metadata pass-through coordinated with batching semantics and the related prompt-context PR.
