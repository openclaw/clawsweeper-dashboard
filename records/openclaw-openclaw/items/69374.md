## Summary

Keep open: current main and the latest shipped tag still extract raw WhatsApp Baileys contextInfo without carrying forwardingScore/isForwarded into the plugin inbound_claim or message_received hook surfaces.

## What This Changes

Expose WhatsApp Baileys forwardingScore/isForwarded metadata to plugin inbound_claim and message_received hook events.

## Best Possible Solution

Add a typed WhatsApp-owned forwarding extractor, define batch-safe semantics, and pass sanitized isForwarded/forwardingScore through the canonical inbound hook context into inbound_claim and message_received metadata.

## Reproduction Assessment

Yes, source-level. Baileys exposes forwardingScore/isForwarded and current main extracts raw contextInfo, but the normalized WhatsApp inbound type and hook mapper have no carrier into inbound_claim or message_received metadata.

## Solution Assessment

Yes as a direction, but the exact API shape needs maintainer approval. The maintainable path is an additive WhatsApp metadata pass-through coordinated with the existing prompt-context PR rather than a body-prefix workaround.
