## Summary

Keep open. Current main still extracts raw WhatsApp `contextInfo` but does not normalize `forwardingScore` / `isForwarded` into WhatsApp inbound messages or plugin hook metadata, and the related open PR only targets prompt context while batch/API semantics remain unresolved.

## What This Changes

Expose WhatsApp Baileys forwarding metadata to plugin `inbound_claim` and `message_received` events so plugins can distinguish forwarded content from user-authored messages.

## Best Possible Solution

Add a typed WhatsApp-owned forwarding extractor and carry sanitized `isForwarded` / `forwardingScore` through canonical hook metadata with explicit debounced-batch semantics and focused coverage.

## Reproduction Assessment

Yes, source-level. A Baileys message with `contextInfo.forwardingScore` / `isForwarded` can reach `extractContextInfo`, but current main has no normalized carrier into `WebInboundMessage` or plugin hook event metadata.

## Solution Assessment

Yes as a direction, but the implementation shape is not settled. The maintainable path is an additive WhatsApp metadata pass-through coordinated with hook API shape and batch semantics.
