## Summary

Keep open. Current main and v2026.5.12 still extract WhatsApp `contextInfo` but do not carry `forwardingScore` / `isForwarded` through the WhatsApp normalized inbound message, finalized context, or plugin hook metadata; the related open PR at https://github.com/openclaw/openclaw/pull/46291 only targets agent prompt context, is conflicting, and does not touch the plugin hook mapper surface.

## What This Changes

Expose WhatsApp Baileys `contextInfo.forwardingScore` / `isForwarded` to plugin `inbound_claim` and `message_received` hook events.

## Best Possible Solution

Add an additive WhatsApp-owned forwarding extractor and pass sanitized `isForwarded` / `forwardingScore` through `WebInboundMessage`, finalized inbound context, and the canonical plugin hook metadata while preserving WhatsApp's explicit hook opt-in privacy model.

## Reproduction Assessment

Yes, source-level. Baileys exposes the fields and current main can extract raw `contextInfo`, but the WhatsApp normalized message, finalized context, and plugin hook mappers have no carrier for them.

## Solution Assessment

Unclear pending maintainer API decision. The requested additive pass-through is the narrow direction, but maintainers should decide whether these become typed event fields or WhatsApp-specific `metadata` entries and how to coordinate with https://github.com/openclaw/openclaw/pull/46291.
