## Summary

Keep open. Current main and the latest release still drop WhatsApp `forwardingScore` / `isForwarded` before plugin hook events; the related open PR at https://github.com/openclaw/openclaw/pull/46291 only targets agent prompt context, not the requested `inbound_claim` / `message_received` plugin-facing hook surface.

## What This Changes

Expose WhatsApp Baileys forwarding metadata to plugin inbound hook events so plugins can distinguish forwarded content from user-authored text.

## Best Possible Solution

Add a typed, additive WhatsApp-owned forwarding extractor and pass sanitized `isForwarded` / `forwardingScore` through the canonical inbound hook context into `inbound_claim` and `message_received` metadata, with focused mapper and WhatsApp tests.

## Reproduction Assessment

Yes, source-level. Baileys 7.0.0-rc11 exposes `IContextInfo.forwardingScore` and `IContextInfo.isForwarded`, and current main can extract raw `contextInfo` but has no carrier from WhatsApp normalization into plugin hook metadata.

## Solution Assessment

Yes as a direction, but it is not implemented yet. The narrow maintainable path is an additive WhatsApp metadata pass-through coordinated with https://github.com/openclaw/openclaw/pull/46291, with maintainers deciding the exact hook event shape.
