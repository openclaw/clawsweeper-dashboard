## Summary

Keep open. Current main still drops WhatsApp forwarded-message fields before plugin hook events, and the related open PR should be reviewed or adapted instead of spawning duplicate repair work.

## What This Changes

Expose WhatsApp `contextInfo.forwardingScore` and `contextInfo.isForwarded` to `inbound_claim` and `message_received` plugin hook events.

## Best Possible Solution

Land a narrow additive WhatsApp-owned pass-through, preferably by adapting https://github.com/openclaw/openclaw/pull/46291 or a follow-up, that extracts forwarding info once, carries it through normalized inbound context, and exposes it through plugin hook metadata with focused tests.

## Reproduction Assessment

Yes, source-level. A Baileys inbound message with `extendedTextMessage.contextInfo.forwardingScore` or `isForwarded` reaches `extractContextInfo`, but current main has no normalized carrier or plugin hook metadata field for those values.

## Solution Assessment

Yes as a direction, but the issue should stay open until implemented. The narrow maintainable path is an additive WhatsApp metadata pass-through, with maintainer review deciding the public hook shape and how it relates to the open forwarding PR.
