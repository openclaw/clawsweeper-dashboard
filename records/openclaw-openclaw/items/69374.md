## Summary

Keep open. Current main and v2026.5.18 still drop WhatsApp forwarding metadata before plugin hook events, and the related open PR only targets agent prompt context while leaving the plugin-facing hook contract unresolved.

## What This Changes

Expose WhatsApp `forwardingScore` and `isForwarded` from Baileys inbound messages to plugin `inbound_claim` and `message_received` hook metadata.

## Best Possible Solution

Add a WhatsApp-owned forwarding extractor and batch-safe typed hook metadata for `inbound_claim` and `message_received`, coordinated with the related prompt-context PR and covered by focused tests plus redacted live proof.

## Reproduction Assessment

Yes, source-level. Baileys exposes the forwarding fields and current main can extract raw `contextInfo`, but no current code path reads or carries those fields into WhatsApp normalized messages or plugin hook events.

## Solution Assessment

Yes as a direction, but the final shape needs maintainer judgment. The narrow maintainable path is additive WhatsApp metadata propagation with explicit batch semantics and plugin-hook contract tests.
