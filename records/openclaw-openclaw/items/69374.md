## Summary

Keep open. Current main and the latest release still do not carry WhatsApp `forwardingScore` / `isForwarded` into plugin hook events, and the related open PR only targets agent prompt context while remaining blocked.

## What This Changes

The issue requests exposing WhatsApp forwarded-message metadata to plugin `inbound_claim` and `message_received` hook payloads.

## Best Possible Solution

Add a typed WhatsApp-owned forwarding extractor, define batch-safe semantics, and pass sanitized `isForwarded` / `forwardingScore` through canonical inbound hook metadata for both plugin events.

## Reproduction Assessment

Yes, source-level. Baileys exposes the fields and current main can extract raw `contextInfo`, but there is no carrier from WhatsApp normalization into `inbound_claim` or `message_received` metadata.

## Solution Assessment

Yes as a direction, but the implementation shape is not settled. The maintainable path is an additive WhatsApp metadata pass-through coordinated with the related prompt-context PR and batching behavior.
