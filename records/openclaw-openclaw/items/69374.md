## Summary

Keep open. Current main still drops WhatsApp `forwardingScore` / `isForwarded` before the plugin hook events, and the related open PR only covers part of the forwarding path rather than resolving the requested plugin-facing hook surface.

## What This Changes

Expose WhatsApp Baileys forwarding metadata from inbound messages to plugin `inbound_claim` and `message_received` hooks.

## Best Possible Solution

Land an additive WhatsApp-owned metadata pass-through into the canonical inbound hook context and plugin event metadata, with focused hook and WhatsApp inbound tests.

## Reproduction Assessment

Yes, source-level. A Baileys inbound message can carry `extendedTextMessage.contextInfo.forwardingScore` or `isForwarded`, but current main has no WhatsApp carrier or plugin hook metadata field for those values.

## Solution Assessment

Yes as a direction, but it is not implemented yet. The narrow maintainable path is an additive WhatsApp metadata pass-through, with maintainer review deciding the plugin hook shape and whether to adapt https://github.com/openclaw/openclaw/pull/46291 or land a follow-up.
