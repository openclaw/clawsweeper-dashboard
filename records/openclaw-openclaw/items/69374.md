## Summary

Keep open. Current main and the latest release still do not carry WhatsApp `forwardingScore` / `isForwarded` into plugin hook events, and the related open PR only targets agent prompt context while it remains blocked on proof and batch semantics.

## What This Changes

Expose WhatsApp Baileys `contextInfo.forwardingScore` and `isForwarded` through normalized inbound messages and plugin `inbound_claim` / `message_received` metadata.

## Best Possible Solution

Add a typed WhatsApp-owned forwarding extractor and carry sanitized `isForwarded` / `forwardingScore` through canonical hook metadata with explicit debounced-batch semantics.

## Reproduction Assessment

Yes, source-level. Baileys exposes the fields and current main can extract raw `contextInfo`, but there is no carrier from WhatsApp normalization into `inbound_claim` or `message_received` metadata.

## Solution Assessment

Yes as a direction, but the implementation shape is not settled. The maintainable path is an additive WhatsApp metadata pass-through coordinated with the related prompt-context PR and batching behavior.
