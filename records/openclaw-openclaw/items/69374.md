## Summary

Keep open. Current main still drops WhatsApp `forwardingScore` / `isForwarded` before plugin hook events, and the related open PR is the right implementation path to review or adapt rather than closing this issue.

## What This Changes

Expose WhatsApp forwarded-message metadata from Baileys inbound context to plugin `inbound_claim` and `message_received` hook events.

## Best Possible Solution

Land a refreshed additive WhatsApp-owned pass-through, preferably by adapting https://github.com/openclaw/openclaw/pull/46291 or a replacement, that extracts forwarding info once, carries it through normalized inbound context, and exposes it through plugin hook metadata with focused tests and real WhatsApp proof.

## Reproduction Assessment

Yes, source-level. A Baileys inbound message with `contextInfo.forwardingScore` or `contextInfo.isForwarded` reaches `extractContextInfo`, but current main has no normalized carrier or plugin hook metadata field for those values.

## Solution Assessment

Yes as a direction, but not implemented yet. The narrow maintainable path is an additive WhatsApp metadata pass-through, with maintainer review deciding the public hook shape and whether to fold it into the open forwarding PR.
