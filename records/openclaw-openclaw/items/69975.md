## Summary

Keep open: the help-text gap is real, but this PR currently documents the wrong default for cron schedules by saying `--tz` defaults to UTC for both `--cron` and offset-less `--at`, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch changes `cron add` and `cron edit` option descriptions so `--tz` mentions offset-less `--at` values and `cron edit --at` references `--tz`.

## Best Possible Solution

Update the two help strings so `--tz` mentions cron expressions and offset-less one-shot datetimes while preserving separate defaults: cron without `--tz` uses the Gateway host timezone, and offset-less `--at` without `--tz` is UTC.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: PR head adds the combined UTC-default help text, while current main runtime, tests, and docs keep cron's omitted-`tz` default as the Gateway host timezone.

## Solution Assessment

No. The requested help-text direction is narrow and valid, but the current PR wording is not the best solution because it collapses cron and one-shot defaults into a single UTC claim.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Remove the UTC default from cron add help:** `src/cli/cron-cli/register.cron-add.ts:89`
  - body: The new `--tz` help says the default is UTC for both `--cron` and offset-less `--at`, but cron schedules without `tz` fall back to the Gateway host local timezone via `resolveCronTimezone`. This can lead users to omit `--tz` when they need UTC cron execution and then get local-time scheduling instead.
  - confidence: 0.94
- **[P2] Remove the UTC default from cron edit help:** `src/cli/cron-cli/register.cron-edit.ts:59`
  - body: The edit command gets the same incorrect UTC default claim. Editing a job with `--cron` and no `--tz` keeps cron evaluation in the Gateway host local timezone, so this help would document the wrong behavior for recurring schedules.
  - confidence: 0.94
