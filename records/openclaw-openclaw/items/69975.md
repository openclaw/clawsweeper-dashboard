## Summary

Keep open. Current main still has the cron help discoverability gap, so this PR remains useful, but the current PR head is not merge-ready because it documents a UTC default for cron schedules that conflicts with runtime behavior, tests, and docs; it also needs the required user-facing changelog entry.

## What This Changes

The PR changes the cron add/edit CLI help strings so `--tz` mentions `--cron` and offset-less `--at` values, and updates `cron edit --at` help to mention `--tz`.

## Best Possible Solution

Land a narrow help-text refresh that says `--tz` applies to cron expressions and offset-less one-shot datetimes, while explicitly preserving the distinct defaults: cron without `--tz` uses the Gateway host local timezone, and offset-less `--at` without `--tz` is UTC. Add a focused help test expectation and the required changelog bullet.

## Reproduction Assessment

Yes. The problem is reproducible by comparing the PR diff's new UTC default wording with current main's `resolveCronTimezone`, cron CLI help test, and docs that distinguish cron host-local behavior from one-shot UTC behavior.

## Solution Assessment

No. The requested direction is the right narrow fix, but the current PR wording is not the best solution because it collapses two different defaults into `default: UTC`; a wording refresh plus changelog entry is safer.
