## Summary

Keep this PR open. The remaining help-text problem is real, but the current branch is not merge-ready because it documents a UTC default for cron schedules that current runtime, tests, and docs define as Gateway host local timezone; the branch is also dirty against current main.

## What This Changes

The PR updates cron add/edit `--tz` help to mention offset-less `--at` values and changes `cron edit --at` help to surface its `--tz` interaction.

## Best Possible Solution

Refresh the PR so `cron edit --at` mentions offset-less datetimes with `--tz`, and so `--tz` help covers both cron expressions and offset-less one-shot datetimes without assigning the same UTC default to both schedule types. The help tests should continue to lock the cron host-local default.

## Reproduction Assessment

Yes. The mismatch is reproducible by reading current main: `cron edit --at` lacks the discoverability text, while `resolveCronTimezone` and the help test show cron defaults to Gateway host local timezone, contradicting the PR head's UTC wording.

## Solution Assessment

No. The requested direction is narrow and maintainable, but the PR's current wording is not the best solution because it collapses distinct defaults for `--cron` and offset-less `--at`; a wording refresh is safer than merging as-is.
