## Summary

Keep this PR open, but the current head still has a blocking resolver bug: it resolves the platform package's `package.json`, while the real `sqlite-vec` variant packages export only the loadable binary subpath. The fallback belongs in the memory-host SDK and the linked regression is credible, but this branch needs the resolver correction before merge.

## What This Changes

The PR adds a memory-host SDK fallback that tries to load the platform-specific `sqlite-vec` package when the meta package is missing, with unit tests and a changelog entry.

## Best Possible Solution

Keep the memory-host SDK fallback, but resolve the exported `sqlite-vec-<platform>-<arch>/vec0.*` subpath directly and add coverage that exercises the real package exports shape while preserving explicit `extensionPath` precedence.

## Reproduction Assessment

Yes. The linked report, current main source, and the PR's terminal output establish the missing-meta-package path; I did not run a local live install in this read-only review, so this is source-backed rather than locally reproduced.

## Solution Assessment

No. The fallback is the right boundary, but the current implementation resolves an unexported `package.json` subpath; the maintainable fix is to resolve the exported `vec0.*` package subpath directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Resolve the exported sqlite-vec loadable path:** `packages/memory-host-sdk/src/host/sqlite-vec-platform-variant.ts:24-25`
  - body: The platform packages expose only the loadable file in their `exports` map, not `package.json`. `createRequire(...).resolve(`${entry.pkg}/package.json`)` throws `ERR_PACKAGE_PATH_NOT_EXPORTED` for an installed variant, so this helper returns `undefined` and the fallback still fails in the only-variant install state. Resolve `${entry.pkg}/${entry.file}` instead.
  - confidence: 0.96
