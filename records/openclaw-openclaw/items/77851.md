## Summary

Keep open: this PR targets a real memory regression, but the proposed fallback resolves `sqlite-vec-<platform>-<arch>/package.json`, and the upstream platform packages do not export `package.json`, so the resolver returns `undefined` in the intended only-variant install state. The PR also still needs after-fix real behavior proof before merge.

## What This Changes

The PR adds a memory-host SDK fallback that tries to load the platform-specific `sqlite-vec` binary directly when the meta `sqlite-vec` package import fails, plus tests and a changelog entry.

## Best Possible Solution

Resolve the exported platform binary subpath, add coverage that reflects the real `sqlite-vec` package exports shape, and include real install or equivalent runtime output showing vector loading succeeds.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Current main source and the linked Ubuntu/npm-global report show the missing-meta-package degradation path, but the only-variant install state was not recreated here.

## Solution Assessment

No. The fallback belongs in the memory-host SDK, but resolving `package.json` is incompatible with the platform package exports; the safer fix is to resolve the exported `vec0.*` subpath directly and prove it in a real install.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Resolve the exported loadable path:** `packages/memory-host-sdk/src/host/sqlite-vec-platform-variant.ts:24`
  - body: The platform packages expose only the loadable file in their `exports` map, not `package.json`. `createRequire(...).resolve(`${entry.pkg}/package.json`)` therefore throws `ERR_PACKAGE_PATH_NOT_EXPORTED` for an installed variant, this helper returns `undefined`, and the fallback still degrades in the exact only-variant scenario. Resolve the exported `${entry.pkg}/${entry.file}` path instead, matching upstream `sqlite-vec`.
  - confidence: 0.95
