## Summary

Keep open. Current main still has Lobster execution exposed through the optional agent tool surface and Task Flow exposed through inspect/cancel/status surfaces, but no explicit chat slash command that manually starts or resumes Lobster or Task Flow workflows. The existing plugin command API makes this feasible, but the requested command is side-effectful enough that maintainers should settle syntax, authorization, session binding, JSON parsing, and approval/resume behavior before implementation.

## What This Changes

Request to add a chat slash command that manually executes Lobster or Task Flow workflows, optionally with JSON arguments.

## Best Possible Solution

Define a narrow Lobster-owned command contract and implement it by reusing the existing embedded Lobster runner plus managed Task Flow runtime, with strict JSON parsing, authorized active-session binding, preserved approval/resume behavior, focused tests, and docs for channel usage and failure cases.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence verification path. Source and docs inspection on current main show no /lobster, /taskflow, or Task Flow run command, while /tasks rejects arguments and only reports task status.

## Solution Assessment

Unclear until maintainers choose the command contract. A plugin-owned command in the Lobster plugin appears narrower than adding a core workflow command, but the security and UX decisions are not specified enough for an automated repair PR.
