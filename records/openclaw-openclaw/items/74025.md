## Summary

Keep open. Current main still exposes Lobster execution as an optional agent tool and Task Flow as inspect/cancel/status surfaces, but there is no explicit `/lobster`, `/taskflow`, or `/tasks ... run` channel command. The plugin command API exists, so this looks feasible, but the side-effectful workflow command contract needs maintainer design before an automated repair PR is safe.

## What This Changes

The issue requests a channel slash command that can manually execute Task Flow or Lobster workflows, with optional JSON arguments.

## Best Possible Solution

Approve a narrow Lobster-owned plugin command contract, then implement it by reusing the existing Lobster runner and managed Task Flow runtime with deliberate JSON parsing, authorized active-session binding, preserved approval/resume semantics, focused tests, and docs for channel usage and failure cases.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence verification path. Source and docs inspection on current main show no `/lobster`, `/taskflow`, or flow-run command, while `/tasks` rejects arguments and only reports task status.

## Solution Assessment

Unclear until maintainers settle the command contract. A plugin-owned command wrapping the existing Lobster/Task Flow execution paths appears narrower than adding a core workflow command, but the security and UX details are not specified enough for an automated fix PR.
