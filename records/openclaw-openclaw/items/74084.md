## Summary

Keep open. The PR targets a real current-main routing cache gap, but it still needs two narrow mechanical follow-ups before merge: make the bounded fingerprint cache semantics match the LRU/FIFO wording and add the required changelog entry.

## What This Changes

The branch adds bindings-reference and SHA-256 content-fingerprint fallback caches to routing evaluated-bindings lookup, plus tests for rebuilt configs with identical or changed bindings.

## Best Possible Solution

Keep the routing optimization if maintainers accept the cross-cfg content-cache tradeoff, make the bounded fingerprint eviction behavior honest or true LRU, add the changelog entry, then land after focused routing tests and the changed gate pass.

## Reproduction Assessment

Yes. Current main can be checked by warming one cfg and then resolving fresh cfg objects with identical bindings while spying on `listBindings`; the PR adds that focused rebuilt-cfg scenario and a changed-content counterexample.

## Solution Assessment

Unclear. The implementation is narrow and plausible, but it should first align the FIFO/LRU semantics and release-note policy, and maintainers should explicitly accept the hashing cost tradeoff.
