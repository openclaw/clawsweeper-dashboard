## Summary

Keep open. Current main still keys evaluated-bindings reuse to the cfg object, while this PR is a plausible narrow routing-cache optimization that still needs its cache semantics and changelog policy cleaned up before merge.

## What This Changes

The PR adds bindings-reference and SHA-256 fingerprint fallback caches for evaluated routing bindings, plus tests for rebuilt configs with identical and changed binding content.

## Best Possible Solution

Keep the optimization if maintainers accept the hashing tradeoff, make the bounded fingerprint cache behavior honest or true LRU, add the required Unreleased changelog entry, then land after focused routing tests and the changed gate.

## Reproduction Assessment

Yes. Current main can be checked by warming one cfg and then resolving fresh cfg objects with identical bindings while spying on `listBindings`; the PR adds that scenario plus a changed-content counterexample.

## Solution Assessment

Unclear. The implementation is narrow and plausible, but it should align eviction and cost semantics first, and maintainers should explicitly accept the SHA-256 stringify tradeoff for cloned binding arrays.
