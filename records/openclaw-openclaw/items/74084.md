## Summary

Keep open. The PR targets a real routing-cache performance gap on current main, and the patch is narrow, but it still needs the small FIFO/LRU alignment and required changelog entry before normal maintainer review and merge.

## What This Changes

The PR adds bindings-reference and SHA-256 content-fingerprint fallback caches to routing evaluated-bindings lookup, plus tests for rebuilt configs with identical or changed bindings.

## Best Possible Solution

Keep the routing optimization if maintainers accept the cross-cfg content-cache tradeoff, make the bounded fingerprint eviction behavior honest or true LRU, add the changelog entry, then land after focused routing tests and changed-gate proof.

## Reproduction Assessment

Yes. Warm one cfg, then resolve fresh cfg objects with identical bindings while spying on listBindings; current main only caches by cfg identity, and the PR adds that rebuilt-cfg scenario plus a changed-content counterexample.

## Solution Assessment

Unclear. The approach is narrow and plausible, but it should first align the FIFO/LRU semantics and release-note policy, and maintainers should accept the hashing cost tradeoff.
