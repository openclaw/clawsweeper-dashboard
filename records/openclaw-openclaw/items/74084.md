## Summary

Keep open. Current main still rebuilds the evaluated binding index for fresh cfg objects because the cache is keyed by the cfg object and binding-array identity only, while this PR adds a real but unmerged fingerprint fallback; the remaining issues are narrow cleanup items rather than grounds to close the PR.

## What This Changes

The PR adds bindings-array and SHA-256 content-fingerprint fallback caches to the routing evaluated-bindings path, plus tests for rebuilt configs with identical and changed binding content.

## Best Possible Solution

Keep the routing optimization if maintainers accept the cross-cfg content-cache tradeoff, make the bounded fingerprint eviction behavior match the stated policy, add the required changelog entry, then land after the focused routing test and changed gate pass.

## Reproduction Assessment

Yes. Current main can be checked by warming one cfg and then resolving rebuilt cfg objects; the PR adds a spy-based test that demonstrates the intended no-rescan behavior for identical content and a rebuild for changed content.

## Solution Assessment

Unclear. The approach is narrow and plausible, but the patch should first align FIFO/LRU semantics and release-note policy, and maintainers should explicitly accept hashing cost versus the avoided binding-index rebuilds.
