## Summary

Keep open. Current main still rebuilds the evaluated-bindings index for fresh config objects with fresh bindings arrays, so the PR is not obsolete; the branch is a plausible narrow performance optimization but still needs two mechanical fixes before merge: align the fingerprint-map eviction semantics with its LRU/FIFO wording and add the required changelog entry.

## What This Changes

The PR adds bindings-reference and SHA-256 bindings-content fallback caches to the routing evaluated-bindings path, plus tests for rebuilt configs with identical and changed binding content.

## Best Possible Solution

Keep the routing optimization if maintainers accept the hashing tradeoff, make the bounded fingerprint eviction behavior match the stated policy, add a single-line Unreleased changelog entry, and land only after the focused routing test and changed gate are green.

## Reproduction Assessment

Yes. Current main can be reproduced by warming one config and then resolving fresh config objects with identical binding content; because the cache is keyed by config object and bindings reference, fresh configs rebuild through `listBindings`. The PR’s added spy test captures the intended no-rescan behavior and the changed-content rebuild path.

## Solution Assessment

Unclear. The implementation is narrow and plausible, but it is not ready as-is because the LRU/FIFO semantics and release-note policy need repair, and maintainers should explicitly accept the hash cost versus avoided binding-index rebuilds.
