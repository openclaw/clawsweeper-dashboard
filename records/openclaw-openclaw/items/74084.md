## Summary

Keep open. The patch is a plausible narrow routing-cache optimization, but it still needs small cleanup before merge: the bounded fingerprint cache is described as LRU while behaving as FIFO, and the runtime perf change lacks the required changelog entry.

## What This Changes

The PR adds bindings-array and SHA-256 content-fingerprint fallback caches to routing evaluated-bindings lookup, plus tests for rebuilt cfg objects with identical or changed bindings.

## Best Possible Solution

Keep the routing optimization if maintainers accept the content-hash cache tradeoff, align the eviction behavior or wording, add the changelog entry, and land after focused routing tests plus changed-gate proof.

## Reproduction Assessment

Yes. Current main can be checked by warming one cfg and then resolving fresh cfg objects with identical bindings while spying on `listBindings`; the PR adds that rebuilt-cfg scenario and a changed-content counterexample.

## Solution Assessment

Unclear. The implementation is narrow and plausibly maintainable, but it should first make the FIFO/LRU semantics honest or true LRU and satisfy release-note policy before maintainers evaluate the hashing tradeoff.
