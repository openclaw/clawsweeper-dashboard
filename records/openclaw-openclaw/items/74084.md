## Summary

Keep open. Current main still only caches evaluated bindings by config-object identity, so this PR is not obsolete or implemented on main; the branch contains plausible routing hot-path work but needs a small eviction-semantics cleanup, required release-note coverage, and normal validation before maintainer merge review.

## What This Changes

The PR adds secondary evaluated-bindings cache lookup by bindings array identity and SHA-256 content fingerprint in `src/routing/resolve-route.ts`, plus tests for rebuilt config objects with identical and changed bindings content.

## Best Possible Solution

Keep the optimization in `src/routing` if routing maintainers accept the immutability and hashing tradeoff, make the fingerprint eviction behavior match the stated policy, add the required changelog entry, and land only after the focused routing test and changed gate are green.

## Reproduction Assessment

Yes. Source inspection shows fresh cfg objects miss the current WeakMap-by-cfg cache and rebuild through `listBindings`; the PR's added spy test is a high-confidence reproduction path for the intended performance behavior, though this read-only review did not execute tests.

## Solution Assessment

Unclear. The proposed cache layers are narrow and plausible, but content fingerprinting adds JSON/hash cost and cross-cfg reuse assumes equivalent binding data stays immutable, so eviction cleanup and validation are needed before calling this the best fix.
