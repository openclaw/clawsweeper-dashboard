## Summary

Keep this PR open: current main does not have this assistant freshness gate, but the proposed branch still has target-matching bugs that can let sibling plugin/config evidence satisfy the gate, and the external PR lacks inspectable real behavior proof.

## What This Changes

The PR adds a default-off assistant-conclusion freshness gate, replay metadata tagging/omission, prompt-build integration, and focused agent tests for plugin/config state questions.

## Best Possible Solution

Rebase onto current main, keep the gate default-off, make freshness matching target-aware for both config and plugin-specific evidence while allowing genuinely broad snapshots, add focused regression tests, and provide real behavior proof before merge.

## Reproduction Assessment

Yes. A focused unit test can ask about plugins.entries.alpha.enabled after recent narrow evidence for plugins.entries.beta.enabled; the current PR logic treats the plugin-state diagnostic type as enough and can skip the required fresh check.

## Solution Assessment

No. The default-off gate direction is plausible, but this implementation is not yet the best fix because target extraction and matching are too broad for nested plugin/config evidence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve nested config target hints:** `src/agents/assistant-conclusion-freshness-gate.ts:225-233`
  - body: `extractConfigTargetHint` returns `plugins.entries` or `plugins.installs` before considering the full dotted path, so a prompt for `plugins.entries.alpha.enabled` is reduced to the root. Since same-or-nested matching is accepted, recent evidence for `plugins.entries.beta.enabled` can be treated as fresh for alpha and skip the intended re-check.
  - confidence: 0.92
- **[P2] Match plugin-state freshness to the queried target:** `src/agents/assistant-conclusion-freshness-gate.ts:261-262`
  - body: `matchesDiagnosticTarget` returns true for every `plugin_install_state`, but the detector now classifies narrow gateway reads such as `plugins.entries.beta.enabled` as `openclaw.plugins_list` with a target. A later question about a different plugin can reuse beta-only evidence and avoid the fresh tool call the gate is meant to require.
  - confidence: 0.87
- **[P3] Ignore openclaw.json when choosing target hints:** `src/agents/assistant-conclusion-freshness-gate.ts:232-233`
  - body: The dotted-path fallback returns the first dotted token, so prompts like "in openclaw.json, is gateway.mode set?" can derive `openclaw.json` instead of `gateway.mode`. That makes fresh key-specific evidence look unrelated and keeps forcing unnecessary tool calls.
  - confidence: 0.83
