## Summary

Keep this PR open: it is a plausible default-off feature, but the current head still has target-matching bugs that can let unrelated plugin/config evidence suppress the intended fresh check, and the external contributor has not provided inspectable real behavior proof.

## What This Changes

The PR adds an opt-in assistant-conclusion freshness gate, replay metadata tagging/omission, prompt-build wiring, and agent tests for plugin/config state questions.

## Best Possible Solution

Keep the gate default-off, rebase onto current main, make freshness matching target-aware for both config and plugin-state evidence, add focused regressions, and require inspectable real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a direct path: place recent `plugins.entries.beta.enabled` gateway evidence in history, then ask about `alpha`; the current matching can treat the unrelated evidence as fresh.

## Solution Assessment

No. The default-off gate is a reasonable direction, but this implementation is not yet the best solution because target extraction and plugin-state matching are too broad and the prompt-build integration needs current-main rebase review.
