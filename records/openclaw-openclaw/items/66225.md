## Summary

Keep this PR open. Current main and the latest shipped tag still have the narrow final-tag regexes in the streaming and user-facing sanitizer paths, so the reported self-closing `<final/>` leak remains valid; however, the PR needs maintainer review because its broader strict-mode scanner can partially parse malformed tags and emit text that should fail closed.

## What This Changes

The PR broadens final-tag stripping in the embedded streaming subscriber and user-facing sanitizer to remove self-closing `<final/>` variants, adds explicit self-closing handling in strict final-tag parsing, and adds sanitizer/streaming regression tests.

## Best Possible Solution

Land this PR or a replacement after rebasing and tightening final-tag parsing into a shared helper that strips `<final/>`, `<final />`, ordinary final tags, and intended attributed forms without treating malformed tag fragments as valid final blocks. Keep focused coverage for sanitizer stripping, non-enforced streaming, enforced streaming, slash-containing attributes, and malformed attribute text, then resolve the linked regression and overlapping PR.

## Reproduction Assessment

Yes. Current-main inspection plus a read-only Node regex check reproduces `<final/>` and `<final />` surviving the affected regexes, and the PR's strict-mode parser risk is reproducible with `<final reason="a>b">SECRET`.

## Solution Assessment

No, not as-is. The PR is directionally right for the user-visible leak, but the best fix should tighten or centralize final-tag parsing so strict mode still fails closed for malformed tags and then be rebased against current main.
