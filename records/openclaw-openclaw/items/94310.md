## Summary

Keep open: the feature is useful, but the Control UI suppression is incomplete because gateway `update.available` events can still restore `updateAvailable` after the initial snapshot is filtered.

## What This Changes

The PR adds Kubernetes container detection, changes CLI status update hints to recommend pulling a newer image in containers, and filters `updateAvailable` out of gateway snapshots when a container is detected.

## Best Possible Solution

Move the container-aware update decision to the shared update-availability source, or filter both snapshot and event delivery from one helper, then add focused proof that status, snapshot, event, and UI banner behavior agree.

## Reproduction Assessment

Yes. Source inspection shows `runGatewayUpdateCheck` can emit `update.available`, `handleGatewayEvent` stores that payload, and the renderer shows the update button from that state.

## Solution Assessment

No. Suppressing only `buildGatewaySnapshot` is too narrow because the live gateway event path remains a supported way for the Control UI to receive update availability.
