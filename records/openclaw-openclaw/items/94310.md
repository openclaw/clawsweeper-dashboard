## Summary

Keep open: the PR is a useful start for container-aware status guidance, but it is not merge-ready because the Control UI update banner can still be repopulated by the async `update.available` event path after the initial snapshot is suppressed.

## What This Changes

Adds Kubernetes env detection, changes CLI update hints to container redeploy guidance, and suppresses `updateAvailable` in gateway snapshots when a container is detected.

## Best Possible Solution

Move the container-aware update policy to the shared update-availability source so status, startup logs, cache, broadcasts, snapshots, and UI all agree while non-container behavior stays unchanged.

## Reproduction Assessment

Yes for the review finding: source inspection shows `scheduleGatewayUpdateCheck` can still broadcast `update.available`, and the Control UI renders `Update Now` from that event. I did not run a live Docker/Kubernetes UI repro in this read-only review.

## Solution Assessment

No. The status formatter change is plausible, but the gateway fix is too narrow because it suppresses only the initial snapshot instead of the shared update availability cache/broadcast/log policy.
