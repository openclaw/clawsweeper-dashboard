## Summary

Keep open: the PR is a useful container-update guidance patch, but it only gates the initial Gateway snapshot while the post-attach update check can still broadcast `update.available` into the Control UI and render the `Update Now` action. The linked container update policy issue also remains open, so maintainer confirmation and stronger real behavior proof are still needed before merge.

## What This Changes

The PR adds Kubernetes/container-aware update guidance to the status formatter, suppresses `updateAvailable` in gateway snapshots for detected containers, and extends container detection with `KUBERNETES_SERVICE_HOST`.

## Best Possible Solution

Apply one container-aware update policy at the update-availability producer so CLI status, startup logs/cache, gateway snapshots/events, and Control UI either suppress local update actions or show redeploy-image guidance consistently, with focused tests and real container/UI proof.

## Reproduction Assessment

Yes, source-level: current main and the PR diff show that `runGatewayUpdateCheck` can emit `update.available`, the UI stores that event, and the renderer shows `Update Now`. I did not run a live Docker or Kubernetes Control UI session.

## Solution Assessment

No. The implementation is a plausible slice, but it is too narrow because it gates status and the initial snapshot while the event producer can still advertise updates to the UI; the safer fix is shared producer-level policy.
