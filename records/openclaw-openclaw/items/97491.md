## What This Changes

The PR changes model-list row normalization so local-baseUrl models are reported as available even when registry availability omits that exact model key, and adds a focused regression test.

## Best Possible Solution

Land the narrow row-level fix if maintainers accept the documented provider-level availability semantics; otherwise design a separate live-readiness probe or field instead of overloading `available`.

## Reproduction Assessment

Yes at source level: current main computes `local` but still derives `available` from registry availability or provider-auth fallback, so a local row omitted from `availableKeys` can render unavailable. I did not run a live Ollama daemon in this read-only review, but the linked issue and PR body provide matching CLI context.

## Solution Assessment

Yes, this is the narrowest maintainable fix for the documented provider-level Auth semantics because it changes the row normalization boundary shared by the relevant row sources. A live readiness probe would be a separate product/API change, not a smaller fix for this display bug.
