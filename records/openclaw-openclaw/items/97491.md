## Summary

Keep open for maintainer review: the PR is a narrow, well-proven candidate fix for the linked local Ollama display bug, but it changes public `models list` availability semantics for local-baseUrl rows, so maintainer acceptance is still needed before merge.

## What This Changes

The PR makes `toModelRow` treat local-baseUrl model rows as available when registry availability omits the exact model key, and adds a focused regression test.

## Best Possible Solution

Land this narrow row-helper fix if maintainers accept local-marker availability semantics; if exact live readiness is desired, add a separate probe or field instead of overloading `available`.

## Reproduction Assessment

Yes, at source level: current `main` computes `local` but can still return `available: false` when `availableKeys` is defined and omits the local model key. I did not run a live Ollama repro in this read-only checkout, but the PR body includes after-fix real-daemon CLI output.

## Solution Assessment

Yes, if maintainers accept provider-level local-marker availability: changing the shared row normalization helper is the narrowest fix for configured, catalog, and discovered rows. A live readiness probe would be a separate product/API change.
