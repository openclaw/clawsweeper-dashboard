## What This Changes

The issue reports that Microsoft Teams streaming in OpenClaw 2026.5.28 delivers each normal-length assistant reply twice: once as the streamed preview and again as a final MessageActivity.

## Best Possible Solution

Fix the Teams plugin's native streaming finalization so default personal-chat streaming produces one visible final reply, and add regression coverage plus live Teams proof for normal-length streamed replies.

## Reproduction Assessment

Yes, source inspection gives a high-signal path: default Teams personal streaming suppresses the normal block payload, then closes the SDK stream through a final MessageActivity path. I did not run a real Teams client, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear until live Teams behavior is verified; the fix belongs in the Teams plugin streaming/finalization boundary, but maintainers should first confirm whether to adapt the SDK final close path or temporarily change default native streaming behavior.
