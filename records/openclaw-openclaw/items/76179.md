## Summary

Keep open: current main still has the reported behavior chain. Owner-only filtering is intentional, but Codex app-server fingerprints the already-filtered per-turn dynamic tool specs, so alternating owner authorization can invalidate the native thread; related #72406 addressed bootstrap refresh visibility, not this thread-compatibility path.

## What This Changes

The issue reports Codex app-server native thread churn when per-turn owner-only tools (`cron`, `gateway`, `nodes`) are removed and restored by authorization changes in long-lived sessions.

## Best Possible Solution

Split Codex dynamic tool schema compatibility from per-turn authorization, keeping native thread continuity stable while the OpenClaw bridge still rejects unauthorized owner-only tool calls fail-closed.

## Reproduction Assessment

Yes. The issue includes a concrete live trajectory, and current source gives a high-confidence source-level reproduction: alternate bound `startOrResumeThread` calls whose dynamic specs differ only by owner-only tools and current main clears the binding instead of resuming.

## Solution Assessment

Unclear for the exact implementation. Given the current protocol types, a stable thread-level schema with bridge-level per-turn authorization looks safer than ignoring only owner-only list differences or assuming resume can update tools.
