## Summary

Keep open: current main still ties Codex native-thread compatibility to the already-filtered per-turn dynamic tool catalog, so owner-only authorization flips can clear the app-server binding and start a new native thread. The issue is source-reproducible and the repair crosses owner-only authorization semantics, so it needs maintainer/security design review before implementation.

## What This Changes

The issue asks OpenClaw to preserve Codex app-server native thread continuity when per-turn owner-only tool availability changes while still enforcing owner-only authorization for the current turn.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn owner authorization so the native thread stays stable across owner-only availability flips while unauthorized tool calls are rejected fail-closed by the OpenClaw bridge.

## Reproduction Assessment

Yes. The reporter provided a concrete live trace, and current source gives a high-confidence source path: alternate the same session between dynamic tool specs with and without cron/gateway/nodes and the fingerprint mismatch clears the binding before thread/start.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus per-turn bridge authorization looks safer than ignoring only owner-only removals because both 74-to-71 and 71-to-74 transitions must preserve the thread without weakening owner-only enforcement.
