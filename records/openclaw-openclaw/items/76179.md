## Summary

Keep open: current main still clears the Codex app-server binding when the already-filtered per-turn dynamic tool catalog changes, and owner-only authorization flips can remove cron, gateway, and nodes from that catalog. The report is source-reproducible, but the repair needs maintainer/security design because thread continuity must not weaken owner-only execution gating.

## What This Changes

The issue requests preserving Codex app-server native thread continuity across owner-only tool availability flips while still denying unauthorized owner-only tool execution.

## Best Possible Solution

Separate Codex thread schema identity from per-turn owner authorization, or use a protocol-supported dynamic tool update path, while keeping unauthorized owner-only tool calls fail-closed in the OpenClaw bridge.

## Reproduction Assessment

Yes. The reporter provided a concrete live trace, and current source gives a high-confidence path: alternate the same session between tool specs with and without cron/gateway/nodes and the fingerprint mismatch clears the binding before thread/start.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus per-turn bridge authorization looks safer than ignoring only owner-only removals because both 74-to-71 and 71-to-74 transitions must preserve the thread without weakening owner-only enforcement.
