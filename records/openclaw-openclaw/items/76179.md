## Summary

Keep open: current main still ties Codex app-server thread compatibility to the already-filtered per-turn dynamic tool catalog, so owner-only authorization flips can clear the binding and start a new native thread. The source path is high-confidence, but the repair is security-sensitive because it must preserve fail-closed owner-only execution. https://github.com/openclaw/openclaw/issues/76179

## What This Changes

Keep a Codex app-server native thread stable across temporary owner-only tool availability flips while continuing to deny unauthorized owner-only tool execution.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn owner authorization so owner-only availability flips preserve the native thread while bridge execution still rejects unauthorized owner-only calls.

## Reproduction Assessment

Yes, from source inspection. Alternate the same Codex app-server session between owner and downgraded non-owner turns: cron/gateway/nodes are removed from toolBridge.specs, the dynamic tool fingerprint changes, and startOrResumeThread clears the binding before starting a new thread.

## Solution Assessment

Unclear for the exact implementation. A stable schema with per-turn bridge authorization or an explicit protocol-level dynamic tool update path are safer directions than ignoring only one owner-only removal transition, because both 74-to-71 and 71-to-74 flips must preserve continuity without weakening authorization.
