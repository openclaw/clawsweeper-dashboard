## Summary

Keep open: current main still has the reported behavior chain. Per-turn owner downgrades change the filtered OpenClaw tool list, Codex app-server fingerprints that already-filtered dynamic catalog, and a fingerprint mismatch clears the binding so the next request starts a new native thread; the related bootstrap snapshot PR does not supersede this root cause.

## What This Changes

The issue reports Codex app-server native thread churn when temporary owner-only tool availability flips `cron`, `gateway`, and `nodes` between otherwise continuous turns.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn tool authorization, preserving native thread continuity across owner-only availability flips while denying unauthorized owner-only calls at execution time.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main gives a source-level reproduction by alternating `startOrResumeThread` calls whose dynamic tool specs differ only because owner-only tools were filtered.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus per-turn bridge authorization looks safer than ignoring one catalog direction, because both downgrade and upgrade transitions must preserve the native thread without weakening owner-only enforcement.
