## Summary

Keep open: current main still fingerprints the already-filtered per-turn Codex dynamic tool list, so owner-only authorization flips can invalidate the Codex app-server thread even though the OpenClaw session continues.

## What This Changes

The issue asks for Codex app-server native thread continuity to survive temporary owner-only tool availability changes while preserving per-turn authorization.

## Best Possible Solution

Split Codex dynamic tool schema identity from per-turn authorization so thread compatibility remains stable while unauthorized owner-only calls are denied by the OpenClaw bridge for that turn.

## Reproduction Assessment

Yes. The issue has a concrete live trace, and current source gives a high-confidence source-level repro by alternating `startOrResumeThread` calls with dynamic tool specs that differ only by owner-only tools.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus fail-closed per-turn bridge authorization is safer than ignoring only owner-only removals, because both 71-to-74 and 74-to-71 transitions need to preserve the native thread.
