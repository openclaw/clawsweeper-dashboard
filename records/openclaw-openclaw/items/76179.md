## Summary

Keep open: current main still has the reported behavior chain. Owner-only filtering is intentional, but the Codex app-server binding fingerprints the already-filtered per-turn dynamic tool list, so a temporary owner-only availability flip can invalidate the native thread instead of only changing turn authorization.

## What This Changes

Prevent Codex app-server native thread restarts when only per-turn owner-only tool availability changes, while preserving owner-only execution enforcement.

## Best Possible Solution

Split Codex dynamic tool schema identity from per-turn authorization so thread compatibility stays stable across owner-only availability flips while the OpenClaw bridge still denies unauthorized calls fail-closed.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current source plus existing tests give a high-confidence source-level reproduction: alternate `startOrResumeThread` calls with dynamic tool specs that differ only by owner-only tools and the binding will be invalidated.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus per-turn bridge authorization is safer than ignoring only owner-only removals, because both 71-to-74 and 74-to-71 transitions need to preserve the native thread without weakening owner-only enforcement.
