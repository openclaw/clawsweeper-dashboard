## Summary

Keep open: current main still invalidates the Codex app-server thread binding when the already-filtered per-turn dynamic tool catalog changes, so owner-only authorization flips can start a new native thread. The issue is source-reproducible and security-sensitive because the fix must preserve fail-closed owner-only execution.

## What This Changes

The issue requests that Codex app-server keep native thread continuity across temporary owner-only tool availability flips while still blocking unauthorized owner-only tool execution.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn owner authorization so the native thread remains stable across owner-only availability flips while the OpenClaw bridge rejects unauthorized owner-only calls fail-closed.

## Reproduction Assessment

Yes. The reporter supplied a concrete live trace, and current main source gives a high-confidence source path: flip senderIsOwner for the same Codex app-server session, remove or restore cron/gateway/nodes in toolBridge.specs, and the fingerprint mismatch clears the binding before thread/start.

## Solution Assessment

Unclear for the exact implementation. A stable thread schema plus per-turn bridge authorization, or a real app-server dynamic tool update path, looks safer than ignoring one direction of owner-only tool changes because both 74-to-71 and 71-to-74 transitions must preserve continuity without weakening authorization.
