## Summary

Keep open: current `main` still ties Codex app-server native-thread compatibility to the already filtered per-turn dynamic tool catalog, so owner-only authorization flips can still invalidate the binding and start a new native Codex thread. The source path remains clear, and the fix needs explicit owner/security review because it must preserve fail-closed execution for owner-only tools.

## What This Changes

The issue reports that Codex app-server thread binding is invalidated when owner-only tool availability changes between turns, causing native thread restarts in one OpenClaw session.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn owner authorization so the native thread remains stable across owner-only availability flips while the OpenClaw bridge rejects unauthorized owner-only tool calls fail-closed.

## Reproduction Assessment

Yes, at source level: alternate the same Codex app-server session between owner and downgraded turns so `toolBridge.specs` changes by `cron`, `gateway`, and `nodes`; current `main` then sees an incompatible fingerprint and starts `thread/start`. I did not run the live Feishu environment in this read-only review.

## Solution Assessment

Unclear for the exact implementation. A stable thread-level schema with per-turn bridge authorization looks safer than ignoring only owner-only removals, because both 74-to-71 and 71-to-74 transitions must preserve the thread without weakening owner-only enforcement.
