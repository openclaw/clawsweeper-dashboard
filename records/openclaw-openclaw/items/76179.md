## Summary

Keep open: current main still fingerprints the already-filtered Codex dynamic tool catalog, so a temporary owner-only authorization downgrade can change the tool list and force a new app-server thread. This is valid but security-sensitive because the fix must preserve fail-closed owner-only execution while separating schema compatibility from per-turn authorization.

## What This Changes

The issue requests preserving Codex app-server native thread continuity when only per-turn owner-only tool availability flips, while still enforcing owner-only authorization for the current turn.

## Best Possible Solution

Split Codex dynamic tool schema identity from per-turn authorization so owner-only availability flips preserve the native thread while unauthorized `cron`, `gateway`, and `nodes` calls still fail closed, with tests covering both 74-to-71 and 71-to-74 transitions.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current source gives a high-confidence reproduction: alternate `startOrResumeThread` inputs whose dynamic tools differ only by owner-only tools after a `senderIsOwner` downgrade, and the stored fingerprint mismatch starts a new thread.

## Solution Assessment

Unclear for the exact implementation. A stable schema plus bridge-side per-turn authorization, or a protocol-supported dynamic-tool update on resume/turn start, is safer than ignoring only owner-only removals because both transition directions must preserve continuity without weakening authorization.
