## Summary

Keep this issue open. Current main still has the reported Mattermost streaming contract mismatch: the docs advertise `channels.mattermost.streaming`, but the Mattermost schema/type/generated metadata do not expose that key and the inbound runtime still forces the draft-preview path without resolving `off | partial | block | progress`. Open PR #73252 is the active implementation candidate, so this should not spawn a duplicate repair lane.

## What This Changes

The issue asks Mattermost to accept the documented streaming modes and fix the draft-preview notification UX so users can disable or choose preview behavior.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, inbound runtime mode resolution, preserved legacy `blockStreaming` compatibility, focused tests for `off` and final delivery, and a follow-up decision on whether final notifications need a separate option.

## Reproduction Assessment

Yes for the documented config mismatch: current main advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound runtime wires preview callbacks unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification run was performed.

## Solution Assessment

Yes, the best direction is a narrow Mattermost-local implementation of the canonical `streaming.mode` contract. A separate `finalNotification` flag should be considered only after users can actually select or disable preview streaming.
