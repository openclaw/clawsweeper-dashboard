## Summary

Keep this issue open. Current main still has the reported Mattermost streaming contract mismatch: public docs advertise `channels.mattermost.streaming`, while the Mattermost schema/types/generated metadata omit that key and the inbound reply path still always wires the draft-preview callbacks without resolving `off | partial | block | progress`.

## What This Changes

The issue asks Mattermost to accept the documented canonical `streaming` preview-mode config, stop silently dropping that key, and address the notification/noisy draft-preview UX.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The end state should add root and per-account `streaming` schema/type/metadata support, resolve preview mode in the inbound runtime, preserve legacy `blockStreaming` compatibility, cover the `off` path and final delivery in tests, and then decide whether a separate final-notification option is still needed.

## Reproduction Assessment

Yes for the documented config mismatch: current main's docs advertise `channels.mattermost.streaming`, while the strict Mattermost schema/type/generated metadata omit it and the inbound runtime wires preview callbacks unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification run was performed.

## Solution Assessment

Yes, the narrow maintainable direction is a Mattermost-local implementation of the documented canonical streaming mode contract. A separate `finalNotification` flag should wait until `streaming.mode` can actually disable or select preview behavior.
