## Summary

Keep this issue open. Current `main` still has the reported Mattermost streaming contract mismatch: the public docs advertise `channels.mattermost.streaming` modes, but the Mattermost schema/type/metadata and inbound runtime still do not implement a preview-mode resolver or `off` path. The related open PRs are useful implementation candidates, not enough to close this tracker yet.

## What This Changes

The issue asks Mattermost to implement the documented `channels.mattermost.streaming` preview modes and fix draft-preview notification/visual-spam behavior.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The end state should add root and per-account `streaming` schema/type/metadata support, resolve preview mode in the inbound runtime, preserve legacy `blockStreaming` compatibility, make `off` bypass draft-preview callbacks/finalization, and add focused schema/runtime/docs coverage before separately deciding whether final-ready notification needs a Mattermost option.

## Reproduction Assessment

Yes for the documented config mismatch: current docs advertise `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound monitor wires draft previews unconditionally. The notification behavior has reporter evidence but was not live-reproduced in this read-only pass.

## Solution Assessment

Yes, the best direction is a narrow Mattermost-local implementation of the documented canonical streaming mode contract. A separate `finalNotification` option should wait until `streaming.mode` can first select or disable the preview behavior correctly.
