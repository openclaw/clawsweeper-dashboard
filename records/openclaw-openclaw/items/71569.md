## Summary

Keep this open. Current main still has the Mattermost streaming docs/runtime mismatch: docs advertise `channels.mattermost.streaming`, but the Mattermost schema/type/metadata omit it and the inbound draft-preview path is still unconditional; open PR #73252 remains the active implementation candidate, while the final-notification UX needs separate product judgment after the mode contract lands.

## What This Changes

The issue asks Mattermost to accept documented `channels.mattermost.streaming` modes and to avoid preview-stream notification/spam behavior that can hide final answers.

## Best Possible Solution

Land or replace PR #73252 with a Mattermost-local implementation of canonical `streaming.mode` schema/type/metadata/runtime support, preserving legacy `blockStreaming`, then make a separate product decision on final notifications or silent draft previews.

## Reproduction Assessment

Yes for the documented config mismatch: current main advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the runtime always registers draft-preview callbacks. The notification symptom is supported by reporter comments/screenshots, but not live-reproduced here.

## Solution Assessment

Yes, the narrowest maintainable direction is Mattermost-local support for the existing channel streaming helpers and regression tests for `off`/final delivery. A new `finalNotification` option should wait until the documented mode contract is actually honored.
