## Summary

Keep this issue open. Current main still has the reported Mattermost streaming contract mismatch: the docs advertise `channels.mattermost.streaming`, but the strict Mattermost schema, public config type, generated metadata, and inbound reply path do not implement a Mattermost preview-mode gate. Open PR #73252 is the active implementation candidate for the schema/runtime mismatch, while the final-notification UX question remains unresolved after that lands.

## What This Changes

The issue asks OpenClaw to implement the documented Mattermost `streaming` config modes and address draft-preview notification and visual-spam behavior.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, inbound runtime mode resolution through the existing preview-streaming helper, preserved legacy `blockStreaming` compatibility, regression tests for `off` and final delivery behavior, and a follow-up product decision on whether finalized preview edits need an explicit notification option.

## Reproduction Assessment

Yes for the documented config mismatch: current main advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound runtime wires draft-preview callbacks unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification reproduction was run.

## Solution Assessment

Yes, the best direction is a narrow Mattermost-local implementation of the canonical `streaming.mode` contract using the existing plugin SDK helper. A new `finalNotification` option should be considered only after users can reliably select or disable preview streaming.
