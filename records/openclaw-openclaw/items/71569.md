## Summary

Keep this open. Current main still has the Mattermost streaming docs/runtime mismatch: the public docs advertise `channels.mattermost.streaming`, while the Mattermost schema, type surface, generated metadata, and inbound runtime path still use legacy block-streaming fields and always initialize draft-preview callbacks; open PR #73252 remains the active implementation candidate for the central bug.

## What This Changes

Mattermost should accept the documented preview streaming modes and avoid draft-preview notification or visual-spam UX when finalizing responses.

## Best Possible Solution

Land or replace PR #73252 with Mattermost-local canonical `streaming.mode` schema, type, metadata, runtime, and regression-test coverage, then decide separately whether final notifications or silent draft previews are desirable.

## Reproduction Assessment

Yes for the documented config mismatch: current main advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the runtime always registers draft-preview callbacks. The notification symptom is plausible from the reporter's Mattermost comments/screenshots but still needs live validation for exact client behavior.

## Solution Assessment

Yes, the narrowest maintainable direction is Mattermost-local adoption of the existing channel streaming helpers while preserving legacy `blockStreaming` compatibility. A new `finalNotification` option should wait until the documented mode contract is actually honored.
