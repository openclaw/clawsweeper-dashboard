## Summary

Keep open: current main and v2026.6.1 now accept the Mattermost `streaming` key, but the remaining Mattermost preview/final delivery behavior is still not fully aligned with the documented modes or the reported notification problem.

## What This Changes

The issue asks Mattermost to honor the documented `channels.mattermost.streaming` modes and avoid draft-preview behavior that notifies on interim drafts while leaving the final answer easy to miss.

## Best Possible Solution

Make Mattermost’s existing documented modes real in the plugin: `off` disables previews, `partial` edits in place, `block` preserves boundary content, and `progress` gives a status preview while delivering a normal final answer with live Mattermost proof.

## Reproduction Assessment

Yes for the code-level mismatch: current main accepts Mattermost streaming modes, but the monitor still routes non-`off` modes through one draft-preview/edit path. No for the exact notification UX, which still needs a live Mattermost client/server proof run.

## Solution Assessment

No, current main is only a partial solution. The config key is fixed, but the best fix is to align Mattermost runtime behavior and docs for `block`/`progress` and make the final-notification policy explicit.
