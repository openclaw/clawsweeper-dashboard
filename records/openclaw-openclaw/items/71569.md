## Summary

Keep open: current main and v2026.6.1 now accept the documented Mattermost `streaming` key, but the remaining mode semantics and final-notification UX are not fully solved because Mattermost still routes preview turns through one draft preview path and finalizes eligible text in place.

## What This Changes

The issue requests Mattermost support for the documented streaming configuration modes and a better final-notification/preview-streaming UX.

## Best Possible Solution

Keep this open and narrow the remaining work to Mattermost-owned streaming mode semantics: preserve the shipped config acceptance, make `block` and `progress` match the docs or update the docs, and require live Mattermost proof for the final-notification behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the remaining path: current Mattermost preview turns finalize eligible final text by editing the draft preview post, and `block` is not wired to a distinct split/chunk behavior.

## Solution Assessment

Unclear as a final product fix. The config parser/resolver fix is good and already shipped, but the remaining UX should be solved by a maintainer-approved Mattermost mode policy plus live client proof, not by adding an ad hoc core notification workaround.
