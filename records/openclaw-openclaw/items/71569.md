## Summary

Keep this issue open. Current main still has the documented Mattermost streaming contract mismatch: the docs advertise `channels.mattermost.streaming`, while the strict Mattermost schema, public type, generated metadata, and inbound runtime do not implement a Mattermost preview-mode gate. Open PR #73252 is the active implementation candidate for the schema/runtime part, and the final-notification UX question remains a separate product behavior decision after the mode contract is fixed.

## What This Changes

The issue requests Mattermost support for the documented `streaming` preview modes and a fix for noisy or silent notification behavior when draft previews finalize.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, inbound runtime mode resolution, preserved legacy `blockStreaming` compatibility, regression tests for `off` and final delivery behavior, and a separate product decision on whether finalized preview edits need an explicit notification option.

## Reproduction Assessment

Yes for the documented config mismatch: current main advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound runtime wires preview callbacks unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification reproduction was run in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow Mattermost-local implementation of the canonical `streaming.mode` contract using the existing plugin SDK helpers. A new `finalNotification` option should be considered only after users can select or disable preview streaming reliably.
