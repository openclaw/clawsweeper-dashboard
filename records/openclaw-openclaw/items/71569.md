## Summary

Keep this issue open. Current `main` still has the reported Mattermost streaming contract mismatch: the public docs advertise `channels.mattermost.streaming`, but the strict Mattermost config schema, public type, generated metadata, and inbound runtime do not implement the documented preview modes. Open PR #73252 is already the active implementation candidate for the schema/runtime mismatch, so this should not spawn a duplicate repair lane.

## What This Changes

The issue asks Mattermost to accept the documented `streaming` preview modes and to address draft-preview notification/visual-noise behavior when final answers arrive as edits.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, inbound runtime mode resolution, preserved legacy `blockStreaming` compatibility, regression tests for `off` and final delivery behavior, and a separate product decision on whether finalized preview edits need an explicit notification option.

## Reproduction Assessment

Yes for the documented config mismatch: current `main` advertises `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound runtime wires preview callbacks unconditionally. The notification symptom has reporter evidence, but I did not perform a live Mattermost notification run.

## Solution Assessment

Yes, the best direction is a narrow Mattermost-local implementation of the canonical `streaming.mode` contract using the existing plugin SDK helpers. A new `finalNotification` option should be considered only after users can select or disable preview streaming reliably.
