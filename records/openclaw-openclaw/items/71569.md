## Summary

Keep this issue open. Current main still has the reported Mattermost streaming contract mismatch: docs advertise `channels.mattermost.streaming`, but the Mattermost schema/type/generated metadata omit that key and the inbound runtime still wires draft-preview callbacks unconditionally.

## What This Changes

Mattermost should accept and honor the documented `channels.mattermost.streaming` preview-mode config and address draft-preview notification or visual-noise UX issues.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The end state should add root and per-account `streaming` schema/type/metadata support, resolve preview mode in the inbound runtime, preserve legacy `blockStreaming` compatibility, test the `off` path and normal final delivery, and then decide whether final notification behavior still needs a separate option.

## Reproduction Assessment

Yes for the documented config mismatch: current main's docs advertise `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound runtime wires preview callbacks unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification run was performed.

## Solution Assessment

Yes, the narrow maintainable direction is a Mattermost-local implementation of the documented canonical streaming mode contract. A separate `finalNotification` flag should be considered only after `streaming.mode` can actually disable or select preview behavior.
