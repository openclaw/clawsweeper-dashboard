## What This Changes

The issue reports duplicate visible channel replies when a message-tool send is followed by a short final acknowledgement such as "Sent above", "OK", or "已发 #xxx".

## Best Possible Solution

Review and land a conservative route-scoped meta-ack suppression through https://github.com/openclaw/openclaw/pull/96680, with normal and follow-up path coverage plus live channel proof, then close this issue from the merged fix.

## Reproduction Assessment

Yes, at source level: current main records successful message-tool sends, routes final and follow-up payloads through the existing duplicate filter, and that filter returns false for short candidates under the 10-character floor. I did not run a live channel replay in this read-only review.

## Solution Assessment

Yes, the best fix belongs at the route-scoped reply-payload dedupe seam rather than lowering the generic duplicate floor, because the floor protects legitimate short replies and both normal and follow-up delivery already share this boundary.
