## Summary

Keep this open: current main and the latest release still only suppress text duplicates above the existing length floor, so short post-message-tool acknowledgements can pass through as visible second messages. The same-author open PR at https://github.com/openclaw/openclaw/pull/96680 is the active candidate fix, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to suppress short final acknowledgements after a same-turn `message(action=send)` already delivered the main channel reply.

## Best Possible Solution

Land a narrow shared reply-payload filter that suppresses only short post-message-tool meta acknowledgements after a same-turn send, while preserving substantive final text and media payloads.

## Reproduction Assessment

Yes at source level: with same-turn `sentTexts` present, a short final payload like `OK` or `已发` falls below the duplicate floor and remains deliverable. I did not run a live channel repro because this review was read-only.

## Solution Assessment

Yes, the best fix boundary is the shared reply-payload dedupe path, not channel-specific adapters. The linked PR uses that shape, but it still needs normal PR review and proof before this issue can close.
