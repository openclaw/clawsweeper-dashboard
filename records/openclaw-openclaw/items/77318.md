## Summary

Keep open: current main does not have the requested consecutive-error loop guard, and this PR is a plausible implementation path, but the new critical detector can be shadowed by the existing ping-pong warning path and the changelog includes an unrelated duplicate entry.

## What This Changes

The PR adds a run-scoped `consecutive_errors` tool-loop detector with a `tools.loopDetection.consecutiveErrorThreshold` config surface, tests, docs, generated schema updates, and a changelog entry.

## Best Possible Solution

Move the consecutive-error critical check ahead of warning-only loop detections, add an alternating two-tool error regression, remove the duplicate changelog bullet, then review this smaller replacement and close or supersede #75924.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: build a same-run A/B alternating error history at `consecutiveErrorThreshold`, then call the expected next tool. In the PR head, `detectToolCallLoop` returns the ping-pong warning before it reaches the new critical consecutive-error branch.

## Solution Assessment

No, not yet: the run-scoped detector direction fits the existing history model, but its current ordering lets a common alternating cascade bypass the intended block threshold. The safer fix is to evaluate the critical consecutive-error detector before warning-only ping-pong and generic returns.
