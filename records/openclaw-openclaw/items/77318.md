## Summary

Keep open: the branch is useful but not merge-ready because the new critical blocker can be shadowed by existing warning-only loop detectors, and the new config validation still accepts an impossible default-threshold setup.

## What This Changes

This PR adds a run-scoped consecutive tool-error loop detector, exposes `tools.loopDetection.consecutiveErrorThreshold`, updates diagnostics/config/docs/changelog artifacts, and adds detector, hook-path, and config tests.

## Best Possible Solution

Keep the detector optional and run-scoped, but make critical consecutive-error blocking win over warning-only detectors, validate the new threshold against resolved defaults, remove the duplicate changelog line, then review/land this smaller replacement and supersede #75924 as appropriate.

## Reproduction Assessment

Yes, source-reproducible: with loop detection enabled, ten alternating read/list failures can satisfy the existing ping-pong warning threshold before the inserted `consecutive_errors` block runs, producing a non-blocking warning instead of the new veto. The config validation gap is also source-reproducible because `historySize` truncation is runtime-enforced while the new refine ignores the default threshold when it is omitted.

## Solution Assessment

No, not yet; the requested direction is narrow and maintainable, but the current PR needs the new critical detector ordered so it cannot be shadowed and the config schema should validate against the resolved/default `consecutiveErrorThreshold`. The duplicate unrelated changelog line should also be dropped before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move consecutive-error blocking before warning detectors:** `src/agents/tool-loop-detection.ts:632`
  - body: The new critical check runs after the ping-pong warning branch. A run that alternates between two failing tools reaches `consecutiveErrorThreshold`, but `getPingPongStreak` can also meet `warningThreshold` and return a non-blocking `ping_pong` warning before this block is evaluated, so the new guard does not stop that cross-tool cascade until the higher ping-pong critical threshold. Move the consecutive-error critical check ahead of warning-only returns or otherwise let the critical error cascade win.
  - confidence: 0.88
- **[P2] Validate history size against the default threshold:** `src/config/zod-schema.agent-runtime.ts:537-541`
  - body: This refine only rejects `historySize < consecutiveErrorThreshold` when the threshold is explicitly set. A config with `historySize: 2` and no `consecutiveErrorThreshold` is accepted, but runtime resolves the threshold to 10 and trims history to two entries, so the new detector can never trigger. Compare against the resolved/default threshold or normalize the default before validation.
  - confidence: 0.84
- **[P3] Drop the duplicated gcloud changelog entry:** `CHANGELOG.md:405`
  - body: The branch adds an unrelated CLOUDSDK_PYTHON changelog line that already exists on current main. Keeping both creates duplicate release notes for the same prior fix; remove the extra added line and keep only the loop-detection entry from this PR.
  - confidence: 0.9
