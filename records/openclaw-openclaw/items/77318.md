## Summary

Keep open: the PR adds a useful detector that is not on current main, but the branch still drops existing veto outcome history for all plugin vetoes and only shows test-style verification rather than real behavior proof.

## What This Changes

The PR adds a run-scoped consecutive-error loop detector with a new `tools.loopDetection.consecutiveErrorThreshold` config/docs/schema/test surface and changes blocked before-tool-call veto outcome recording.

## Best Possible Solution

Keep the run-scoped detector shape, preserve ordinary plugin-veto outcome recording while skipping only tool-loop self-recording, and require real runtime proof that the new block fires.

## Reproduction Assessment

Yes, by source inspection for the review finding. Current main records blocked plugin vetoes into loop history, while the PR head returns all veto blocks without recording an outcome, so repeated ordinary vetoes can lose no-progress escalation data.

## Solution Assessment

No, not as currently patched. The detector belongs in the existing run-scoped history model, but the sticky-blocking repair should skip recording only `tool-loop` vetoes or otherwise keep ordinary plugin veto outcomes in history.
