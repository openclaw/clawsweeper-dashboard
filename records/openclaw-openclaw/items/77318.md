## Summary

Keep open: current main still lacks the requested consecutive-error detector, but this PR is not ready because it drops outcome recording for all vetoed tool calls and still has only test/config-doc proof rather than real runtime proof.

## What This Changes

The PR adds a run-scoped consecutive cross-tool error detector with a new `tools.loopDetection.consecutiveErrorThreshold` config, diagnostics, docs, schema, and tests.

## Best Possible Solution

Keep the run-scoped detector, restore outcome recording for non-`tool-loop` vetoes, rebase against current main, and add redacted real runtime proof that the new block fires.

## Reproduction Assessment

Yes, for the review finding by source inspection: current main records blocked veto outcomes, while PR head returns every veto block before calling `recordLoopOutcome`. A repeated ordinary plugin-before-tool-call veto would therefore lose the result-hash history needed by existing no-progress/global loop detection.

## Solution Assessment

No, not as currently patched. The detector fits the existing run-scoped history model, but the sticky-blocking repair should skip recording only `tool-loop` vetoes or otherwise keep recording non-loop plugin veto results.
