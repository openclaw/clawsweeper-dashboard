## Summary

Keep open: this is a useful targeted guard, but the PR is not mergeable as-is because it introduces a P2 regression in veto outcome tracking, lacks after-fix real behavior proof, and is currently merge-conflicting with main.

## What This Changes

The branch adds a `consecutive_errors` loop detector with `tools.loopDetection.consecutiveErrorThreshold` config/schema/docs/diagnostic type updates and regression tests, plus changes blocked veto outcome recording.

## Best Possible Solution

Keep the run-scoped detector shape, rebase it onto current main, preserve non-loop veto outcome recording, and add real behavior proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main records plugin veto blocked results into loop history, while the PR deletes that recording for all vetoes, so a repeated policy-blocked tool can lose existing no-progress escalation.

## Solution Assessment

No, not as currently patched. The detector belongs in the existing run-scoped history model, but the sticky-blocking repair should skip recording only `tool-loop` vetoes or otherwise preserve ordinary plugin veto outcomes.
