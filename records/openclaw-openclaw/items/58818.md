## Summary

Keep open: current main has partial continuity mechanisms, but it still does not provide the requested user-facing guarantee that the last N raw conversation messages survive both compaction and reset. The remaining work needs maintainer product and security review because raw replay changes context, auth, channel, thread, and clean-reset semantics.

## What This Changes

The issue requests a configurable session-level guarantee that at least the last N raw user/assistant messages remain available across compaction and session resets.

## Best Possible Solution

Define a maintainer-approved continuity contract that unifies reset and compaction behavior, bounds raw-tail replay, preserves auth/channel isolation, and includes cross-runtime tests before implementation.

## Reproduction Assessment

Yes, source-level: current main lacks a session-level `minTailMessages` or carry-forward contract, and the existing raw reseed path is backend opt-in and reason-gated rather than universal. I did not run a live end-to-end repro because the missing contract is directly visible in source and docs.

## Solution Assessment

Unclear as a final design: `minTailMessages` is plausible, but the best fix needs maintainer decisions about defaults, auth/channel isolation, manual clean reset semantics, and whether the contract belongs in core, backend config, context-engine APIs, or a combination.
