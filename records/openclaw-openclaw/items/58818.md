## Summary

Keep open: current main has partial CLI raw-tail reseed and embedded compaction recent-turn retention, but it still does not implement a user-facing guarantee that the last N raw messages survive both compaction and reset. The remaining work is product and security design, not a conservative cleanup close.

## What This Changes

The issue requests a configurable session continuity guarantee that preserves the last N raw transcript messages across compaction and session reset boundaries.

## Best Possible Solution

Keep this as a maintainer-owned design issue for one explicit context-tail contract shared across reset, compaction, CLI reseed, embedded runs, and context engines, with security guards for auth/profile/channel/workspace boundaries.

## Reproduction Assessment

Yes. Source and tests show current main has opt-in CLI raw-tail reseed and embedded recent-turn compaction retention, but reset still creates a fresh successor transcript without a session-level last-N raw-message guarantee.

## Solution Assessment

Unclear as written. The requested contract is plausible and aligned with the problem, but the exact default, config shape, reset semantics, and security boundaries need maintainer product/security decisions before implementation.
