## Summary

Keep open: current main and v2026.6.10 only provide narrower continuity mechanisms, not the requested user-facing guarantee that the last N raw messages survive both compaction and reset. The remaining design changes session context, config defaults, auth boundaries, channel/thread scoping, and manual clean-reset semantics, so it needs maintainer product and security review rather than conservative cleanup.

## What This Changes

Request a configurable session-level raw-tail continuity guarantee that preserves the last N raw conversation messages through compaction and daily, idle, or manual resets.

## Best Possible Solution

Define a maintainer-approved continuity contract that unifies reset and compaction behavior, bounds raw-tail replay, preserves auth/channel/thread isolation, and adds cross-runtime regression coverage before implementation.

## Reproduction Assessment

Yes, source-level: current main lacks a session-level minTailMessages or carry-forward contract, and the existing raw reseed path is backend opt-in, capped, and reason-gated rather than universal. I did not run a live gateway repro because the missing contract is directly visible in source, tests, and docs.

## Solution Assessment

Unclear as a final design: minTailMessages is plausible, but the best fix needs maintainer decisions about defaults, auth/channel isolation, manual clean reset semantics, and whether the contract belongs in core, backend config, context-engine APIs, or a combination.
