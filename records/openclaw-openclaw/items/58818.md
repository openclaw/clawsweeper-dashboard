## Summary

Keep open: current main has useful partial continuity mechanisms, but it still does not provide the requested user-facing guarantee that the last N raw conversation messages survive both compaction and session reset. The remaining work is a product/security design item because raw replay needs explicit auth, channel, thread, and clean-reset boundaries.

## What This Changes

The issue requests a configurable session/context guarantee that preserves a bounded tail of raw user/assistant messages across compaction and session resets.

## Best Possible Solution

Define a maintainer-approved session continuity contract that unifies compaction and reset behavior, bounds raw-tail replay, preserves auth/channel isolation, and includes cross-runtime tests before implementation.

## Reproduction Assessment

Yes, source-level: current main lacks a session-level `minTailMessages` or carry-forward contract, and the existing raw reseed path is backend opt-in and reason-gated rather than a universal reset plus compaction guarantee. I did not run a live end-to-end repro because the missing contract is directly visible in source and docs.

## Solution Assessment

Unclear as a final design: `minTailMessages` is plausible, but the best fix needs maintainer decisions about defaults, auth/channel isolation, manual clean reset semantics, and whether the contract belongs in core, backend config, context-engine APIs, or a combination.
