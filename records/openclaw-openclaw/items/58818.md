## Summary

Keep open: current main still lacks the requested user-facing guarantee that the last N raw conversation messages survive both compaction and session reset. Existing compaction-tail and CLI raw-reseed mechanisms are useful partial coverage, but this remains a product/security design item rather than a cleanup close.

## What This Changes

Request a configurable session context contract that preserves at least the last N raw conversation messages across compaction and reset boundaries.

## Best Possible Solution

Define a maintainer-owned session continuity contract that unifies compaction and reset behavior, bounds raw-tail replay, preserves auth/channel isolation, and includes tests across CLI, embedded, gateway reset, and context-engine paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main proof of the missing capability: there is no last-N raw-tail config, resets mint a new session and archive the old transcript, and raw reseed is only a gated CLI fallback. This is a feature-gap reproduction rather than a live failing command repro.

## Solution Assessment

Unclear as a final design: a `minTailMessages` guarantee is plausible, but the best fix needs maintainer decisions about defaults, auth/channel isolation, and whether this belongs in core, plugin APIs, or both. The current partial CLI and compaction mechanisms are not enough to solve the issue.
