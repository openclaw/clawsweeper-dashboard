## Summary

Keep open: current main and the latest release have partial continuity mechanisms, but they still do not provide the requested user-facing guarantee that the last N raw messages survive both compaction and reset. The remaining work crosses session state, auth boundaries, clean-reset semantics, and config defaults, so it needs maintainer product and security review before implementation.

## What This Changes

Request a configurable session min-tail/carry-forward contract that keeps the last N raw user and assistant messages through compaction and session resets.

## Best Possible Solution

Define a maintainer-approved, security-scoped session continuity contract first, then implement one bounded raw-tail path across compaction and reset with explicit clean-reset and auth/channel isolation tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro for the missing guarantee: current main has no `session.minTailMessages` schema, raw CLI reseed is opt-in and reason-gated, and reset mints a fresh session without raw seed messages. I did not run a live gateway scenario because the source and tests already show the behavior boundary.

## Solution Assessment

Unclear as written: the requested `minTailMessages` direction is plausible, but the safest solution needs maintainer decisions on defaults, auth/profile boundaries, manual clean reset behavior, and whether this belongs in a core context contract or narrower runtime-specific reseed path.
