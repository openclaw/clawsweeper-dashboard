Keep open: current main has partial CLI reseed and startup-memory mechanisms, but it does not implement the requested user-facing guarantee that the last N raw messages survive both compaction and reset. ## What This Changes

The issue requests a configurable session context guarantee that preserves at least the last N raw user/assistant messages through compaction and session reset.

## Best Possible Solution

Keep this as a maintainer-owned design issue for a single core context-tail contract with explicit auth/channel/workspace guardrails, rather than expanding the current narrow CLI fallback piecemeal.

## Reproduction Assessment

Yes, source inspection gives a high-confidence way to verify the missing behavior: current main has no `minTailMessages`/carry-forward config, and reset paths mint fresh sessions while clearing CLI bindings for non-subagent resets. I did not run a live gateway scenario.
