## Summary

Keep open: the reporter narrowed this from a runtime defect to command-facing clarity, and current main plus v2026.6.8 still describe safe restart as draining active work without saying the default deferral is bounded and can force after `gateway.reload.deferralTimeoutMs`.

## What This Changes

The issue asks OpenClaw to clarify safe restart semantics so operators know that default `--safe` restart deferral is bounded and `gateway.reload.deferralTimeoutMs=0` is the current indefinite-wait control.

## Best Possible Solution

Update command help and user docs to state that safe restart uses `gateway.reload.deferralTimeoutMs`, defaults to a bounded 300000ms deferral that can force, and uses config value `0` for indefinite wait; leave runtime semantics unchanged unless maintainers explicitly choose a broader behavior change.

## Reproduction Assessment

Yes, at source level. Current command help/docs promise active-work drain while current runtime and config docs show omitted `gateway.reload.deferralTimeoutMs` means a bounded wait that can force after 300000ms.

## Solution Assessment

Yes for the narrowed issue: a command help/docs clarification is the narrowest maintainable fix. Changing the default behavior or adding per-command indefinite wait would be a broader compatibility/product decision.
