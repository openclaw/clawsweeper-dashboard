- `impact:session-state`: The reported misunderstanding can lead operators to trigger restarts that abort active embedded runs after the configured deferral window.

## Summary

Keep open: current main and the latest release still describe `--safe` as restarting after active work drains, while source and tests show the intended default is a bounded deferral that can force restart on timeout; the reporter already narrowed this to CLI/help clarification.

## What This Changes

Clarify `daemon/gateway restart --safe` help and docs so they explain the default bounded deferral, force-after-timeout behavior, and config-only indefinite wait control.

## Best Possible Solution

Update the CLI option/help text and the daemon/gateway restart docs to say safe restart uses `gateway.reload.deferralTimeoutMs`, defaults to a bounded five-minute wait before force, and uses `0` in config for indefinite safe deferral.

## Reproduction Assessment

Yes. Current source and tests show bounded restart deferral can force after timeout, while the safe restart command copy describes draining without naming that default.

## Solution Assessment

Yes, the best fix is a focused docs/help clarification. Runtime behavior is already configurable and tested, so changing restart semantics or adding a new config surface would be broader than the narrowed report needs.
