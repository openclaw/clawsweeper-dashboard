## Summary

The reporter correctly narrowed this from a runtime defect to a valid help-text mismatch. Current main intentionally uses a bounded safe-restart deferral that becomes forced after timeout, while the command help and nearby CLI docs still imply that restart always waits for active work to drain.

## What This Changes

Clarify that safe restart uses a bounded deferral by default and can force restart after timeout, while indefinite waiting requires `gateway.reload.deferralTimeoutMs=0`.

## Best Possible Solution

Preserve the tested bounded restart contract and update daemon and Gateway restart help plus adjacent public CLI docs to disclose the default five-minute limit, the forced restart outcome, and the configuration-only indefinite-wait control.

## Reproduction Assessment

Yes. Current main directly reproduces the mismatch: the command wording promises restart after draining, while the tested runtime uses a bounded timeout that can force restart.

## Solution Assessment

Yes for the narrowed report. Correcting the help and nearby documentation is the narrowest maintainable solution; changing defaults or adding a per-command wait mode would require separate compatibility and product review.
