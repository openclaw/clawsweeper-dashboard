## Summary

Keep open: the behavior is configurable and appears intended, but current main and v2026.6.6 still describe `--safe` as restarting after active work drains while the default deferral remains bounded and can force after timeout.

## What This Changes

Requested clarification of safe Gateway restart semantics so `--safe` accurately reflects bounded deferral, force-after-timeout, and the config-only indefinite wait behavior.

## Best Possible Solution

Align the public safe-restart contract: either update daemon/gateway help and nearby docs to state the default bounded force-after-timeout and `gateway.reload.deferralTimeoutMs=0`, or intentionally change the CLI semantics with focused restart tests.

## Reproduction Assessment

Yes, at source level. Current main resolves omitted `gateway.reload.deferralTimeoutMs` to `300000`, emits a force intent when that deferral expires, rejects `--safe --wait`, and still describes safe restart as after active work drains.

## Solution Assessment

Unclear. A narrow help/docs clarification is the safest maintainable path unless maintainers intentionally want to change the public safe-restart semantics or add a per-command indefinite wait.
