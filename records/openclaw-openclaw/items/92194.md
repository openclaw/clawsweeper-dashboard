## Summary

The reporter correctly narrowed this from a runtime defect to an operator-facing documentation mismatch: bounded deferral followed by force is intentional and configurable, but current `--safe` help still implies an unqualified wait until work drains. No separate canonical issue or open PR owns that remaining clarification.

## What This Changes

Clarify that `daemon restart --safe` defers active work for a bounded configured interval by default and can then force restart.

## Best Possible Solution

Keep the intended runtime behavior unchanged and make the command help plus adjacent CLI docs explicitly describe the bounded default and the existing `gateway.reload.deferralTimeoutMs=0` indefinite-wait control.

## Reproduction Assessment

Yes. Current and v2026.6.5 source directly show bounded force-on-timeout behavior alongside `--safe` help that only promises restart after active work drains.

## Solution Assessment

Yes for the narrowed issue. Correcting help and nearby docs is the narrowest maintainable solution; changing defaults or adding request-level wait semantics would be a separate product decision.
