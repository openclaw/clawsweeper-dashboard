## Summary

The reporter correctly narrowed this from a runtime bug to an operator-contract mismatch. Current main and release v2026.6.6 still describe safe restart as completing after active work drains, although the documented and tested behavior forces restart after the default five-minute deferral; adjacent restart PRs address different lifecycle concerns.

## What This Changes

Clarify safe-restart help and docs so operators know the default deferral is bounded and can force restart, while `gateway.reload.deferralTimeoutMs=0` waits indefinitely.

## Best Possible Solution

Preserve the existing configurable restart behavior and update the option description, deferred-request output, and daemon/gateway documentation to state the five-minute default, force-after-timeout behavior, and zero-timeout indefinite mode.

## Reproduction Assessment

Yes. Current main directly exposes the mismatch between the `--safe` help/status wording and the documented, tested bounded timeout behavior, although no live CLI reproduction was run during this read-only review.

## Solution Assessment

Yes. A narrow help and documentation correction is the best solution because the reporter confirmed the runtime behavior is intentional and an indefinite-wait configuration already exists.
