## Summary

Keep open: current main still lets symmetric jitter reduce a retry-after-derived delay below the server hint, and open PR #68543 is the active closing implementation candidate. The issue should stay open until that PR, or a narrower replacement, lands.

## What This Changes

Ensure `retryAsync` does not schedule below a finite `retryAfterMs` lower bound after jitter is applied.

## Best Possible Solution

Land #68543 or a narrow replacement that enforces the retry-after lower-bound invariant with deterministic regression coverage, preserves existing exponential-backoff behavior for calls without `retryAfterMs`, and deliberately includes or excludes the memory-host SDK copy.

## Reproduction Assessment

Yes. A static current-main reproduction is `retryAfterMs: () => 1000`, `jitter: 0.5`, `maxDelayMs >= 1000`, and `generateSecureFraction() === 0`, which produces 500 ms before the final clamp.

## Solution Assessment

Unclear for the exact fix shape. The invariant is valid, but maintainers should choose between positive-only jitter and a post-jitter retry-after floor while documenting the intended `maxDelayMs` interaction.
