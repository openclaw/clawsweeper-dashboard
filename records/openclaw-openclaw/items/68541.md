## Summary

Keep open: current main still lets symmetric jitter reduce a retry-after-derived delay below the server hint, and #68543 is open as the active closing implementation candidate.

## What This Changes

The issue asks for `retryAsync` to preserve a server-supplied `retryAfterMs` lower bound when jitter is enabled.

## Best Possible Solution

Review and land #68543 or a narrow replacement that enforces the retry-after lower-bound invariant with deterministic regression coverage and an explicit decision on the memory-host SDK copy.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: `retryAfterMs: () => 1000`, `jitter: 0.5`, `maxDelayMs >= 1000`, and `generateSecureFraction() === 0` schedules 500 ms before the final clamp.

## Solution Assessment

Unclear for the exact implementation shape. The narrow maintainable fix belongs in `retryAsync` with regression coverage, while PR review should choose positive-only jitter versus a post-jitter floor and decide whether the memory-host SDK copy changes too.
