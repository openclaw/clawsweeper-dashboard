## Summary

Keep open: the PR addresses a real, source-reproducible setup bug and includes useful terminal proof, but the current diff still needs revision before merge because it can seed a phantom default account and adds a WeCom-specific branch in shared setup core.

## What This Changes

The PR promotes top-level setup credentials before named-account writes, scopes WeCom setup wizard callbacks to the selected account, and adds regression coverage plus changelog/generated cleanup.

## Best Possible Solution

Land a rebased fix that keeps promotion-before-patch behavior, avoids creating empty default accounts, and handles default-account callback compatibility through a generic setup/plugin contract or plugin-owned update.

## Reproduction Assessment

Yes, source-reproducible: current main skips promotion in the setup adapter path, and the published WeCom 2026.5.7 wizard resolves/writes the default account when callbacks omit accountId. I did not run the interactive wizard because this review is read-only.

## Solution Assessment

No, not as written: promotion-before-patch is the right direction, but the PR should avoid seeding empty default accounts and should not hard-code WeCom behavior in shared core without a generic contract or explicit maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Gate promotion on actual top-level credentials:** `src/channels/plugins/setup-helpers.ts:301-304`
  - body: This new non-default setup path calls the promotion helper even when the existing channel section has no movable account credentials, such as only `enabled`. In that case the helper filters everything out and still creates `accounts.default: {}` before the selected account is patched, so adding the first named account can surface an empty phantom default account. Make the promotion no-op when there are no keys to move, or check that before calling it here.
  - confidence: 0.78
- **[P2] Remove the WeCom id check from shared setup core:** `src/channels/plugins/setup-wizard.ts:197`
  - body: This shared wizard helper now enables default-account callback scoping only when `channelKey` is `wecom`. The same callback shape comes from the generic declarative wizard API, so this fixes one external plugin by embedding its id in core and leaves the behavior unavailable to other plugins unless they get their own core branch. Move this behind a generic setup/plugin contract or keep the compatibility in the WeCom plugin package.
  - confidence: 0.72
