## Label Justifications

- `P1`: The issue describes an active Feishu multi-agent workflow collapsing all users into one agent, which can mix sessions and needs timely maintainer review.
- `impact:session-state`: The reported behavior mixes per-user sessions, memory, and prompts by routing all users to one agent session.
- `impact:security`: Cross-user routing in a messaging channel can expose private context or memory to the wrong user or agent.
- `impact:message-loss`: Messages are dispatched to the wrong configured agent instead of the intended per-user route.

## Summary

Keep open. Current main still has source-level account-scope behavior that can make Feishu multi-account bindings without an explicit `accountId` fall back to the default or first agent, the Feishu docs still show copyable bindings without `accountId`, and the related PR is open with account-isolation concerns.

## What This Changes

The issue asks for Feishu multi-agent routing to dispatch each user's messages to the configured per-user agent instead of collapsing all messages onto one agent.

## Best Possible Solution

Resolve the Feishu account-scope policy first, then land a corrected fix that makes multi-account bindings explicit and safe through docs, CLI/doctor guidance or a narrowly approved bind default, regression tests, and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. Feishu passes the account and direct peer into routing, and current routing/tests show that bindings without `accountId` only match the default account while named-account traffic falls back to the default agent.

## Solution Assessment

Unclear. The existing routing surface can express safe explicit account matching, but maintainers need to decide whether omitted Feishu account scope should stay default-only, become a wizard/doctor warning, or gain a narrowly approved wildcard creation path.
