## What This Changes

The issue reports that seven Feishu user/account bindings all route inbound messages to the same first/default agent instead of isolated per-user agents.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/49769 or an equivalent narrow fix with regression coverage proving multi-account bindings use exact account ids or accountId: "*" while preserving default-account semantics.

## Reproduction Assessment

Yes for the likely route-level failure: current source and tests show omitted accountId bindings miss non-default accounts and fall back to the default/first agent. The exact seven-account Feishu setup is not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Yes: the narrow maintainable fix is to make multi-account binding scope explicit in creation/docs/tests, using exact account ids or accountId: "*", rather than globally reinterpreting omitted accountId.
