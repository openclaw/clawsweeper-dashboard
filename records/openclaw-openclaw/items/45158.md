## Summary

Keep open: current main still has the account-scoped routing behavior that can route non-default Feishu account traffic to the default agent when bindings omit or mis-scope `accountId`, and the related account-scope PR is still open with review blockers.

## What This Changes

The issue reports a Feishu multi-agent setup where messages from multiple users/accounts all route to one agent instead of the configured per-user agents.

## Best Possible Solution

Resolve this through https://github.com/openclaw/openclaw/pull/49769 or a narrow replacement that proves Feishu multi-account bindings use exact account ids or explicit `accountId: "*"` while preserving deliberate default-account semantics.

## Reproduction Assessment

Yes for the likely route-level failure: current source and tests show omitted accountId bindings miss non-default accounts and fall back to the default agent. The exact seven-account Feishu setup was not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Unclear that the open PR's wildcard-default behavior is the best final shape, because it broadens bare multi-account binding scope. The safer fix should make account scope explicit in CLI/docs/doctor behavior and only widen defaults if maintainers approve that routing policy.
