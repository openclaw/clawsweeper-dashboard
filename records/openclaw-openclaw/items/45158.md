## Summary

Keep open: current main still has a source-reproducible Feishu/multi-account routing footgun where bare bindings can miss named accounts and fall back to one agent session. The issue is security- and session-sensitive, and the related account-scope PR at https://github.com/openclaw/openclaw/pull/49769 is still open and conflicting rather than merged.

## What This Changes

The issue reports Feishu multi-agent routing sending messages from distinct users/accounts into the same agent session instead of the configured per-user agents.

## Best Possible Solution

Resolve this through a maintainer-approved account-scope/Feishu binding path that preserves explicit account isolation, documents `peer` and `accountId` usage, adds focused resolver/CLI/doctor coverage, and includes redacted Feishu proof.

## Reproduction Assessment

Yes for the likely source-level failure: current routing code and tests show omitted-account bindings miss named accounts and fall back to `main`. The exact seven-account Feishu setup was not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Unclear as a final product fix. Explicit `peer`/`accountId` guidance and current account-aware routing are necessary, but automatically treating bare multi-account bindings as wildcard routing changes isolation semantics and needs maintainer approval.
