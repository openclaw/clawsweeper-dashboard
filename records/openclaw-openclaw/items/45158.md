## Summary

Keep open: current main still has the account-scoped routing behavior that can miss Feishu bindings when `accountId` is omitted or mis-scoped, then fall back to one default agent. The related account-scope PR remains open and has product/security review blockers, so no close reason is satisfied.

## What This Changes

The issue reports Feishu multi-agent routing sending messages from multiple users/accounts into the same agent session instead of the configured per-user/per-account agents.

## Best Possible Solution

Resolve this through https://github.com/openclaw/openclaw/pull/49769 or a narrow replacement that makes Feishu multi-account bindings use explicit account ids or `accountId: "*"`, with docs/doctor guidance and regression coverage while preserving deliberate default-account semantics unless maintainers approve a policy change.

## Reproduction Assessment

Yes for the likely route-level failure: current source and tests show omitted-account bindings miss named accounts and fall back to the default agent. The exact seven-account Feishu setup was not live-reproduced because the full config and Feishu credentials are unavailable.

## Solution Assessment

Unclear that the open PR's wildcard-default behavior is the best final fix because it broadens bare multi-account binding scope. The safer path is explicit account-scope docs/doctor guidance plus a creation-only convenience only if maintainers approve that routing policy.
