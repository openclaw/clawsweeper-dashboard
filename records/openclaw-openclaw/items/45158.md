## Label Justifications

- `P1`: The report describes a user-facing Feishu routing regression where multiple users or accounts can all land in one agent session.
- `impact:session-state`: Wrong routing can mix transcript, memory, and per-agent state across users configured for isolation.
- `impact:security`: Misrouting can collapse intended user, account, or tool-policy boundaries into the wrong agent context.
- `impact:message-loss`: Messages are delivered to the wrong configured agent, which is a concrete routing failure.

## Summary

Keep open: current main still has a source-reproducible Feishu/multi-account routing failure mode where bare bindings can miss named accounts and fall back to one agent session, and the related account-scope PR remains open and conflicting rather than merged.

## What This Changes

The issue reports seven Feishu agents with separate bindings all receiving traffic through the same agent session instead of per-user/per-account routing.

## Best Possible Solution

Resolve this through a maintainer-approved Feishu/account-scope binding path that preserves explicit account isolation, documents `peer` and `accountId` usage, adds focused resolver/CLI/doctor coverage, and includes redacted Feishu proof.

## Reproduction Assessment

Yes for the likely source-level failure: current routing code and tests show omitted-account bindings miss named accounts and fall back to `main`. The exact seven-account Feishu setup was not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Unclear as a final product fix. Explicit `peer`/`accountId` guidance and account-aware routing are necessary, but automatically treating bare multi-account bindings as wildcard routing changes isolation semantics and needs maintainer approval.
