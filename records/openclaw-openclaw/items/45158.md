## Label Justifications

- `P1`: The issue describes a user-facing Feishu routing regression that can send multiple users or accounts into one agent workflow.
- `impact:session-state`: Wrong routing can mix transcript, memory, and per-agent session state across users configured for isolation.
- `impact:security`: Misrouting can collapse intended user, account, or tool-policy boundaries into the wrong agent context.
- `impact:message-loss`: Messages can be delivered to the wrong configured agent rather than the intended per-user agent.

## Summary

Keep open: current main still has a source-reproducible Feishu/multi-account routing failure mode where bare bindings can miss named accounts and fall back to one agent session. Recent docs and doctor warnings make the behavior clearer, but they do not fully close the reported runtime routing problem or the account-scope product decision.

## What This Changes

The issue reports Feishu multi-agent bindings for different users all resolving to one agent session instead of the intended per-user agents.

## Best Possible Solution

Resolve this through a maintainer-approved Feishu/account-scope binding path that preserves explicit account isolation, documents `peer` and `accountId` usage, adds focused resolver/CLI/doctor coverage, and includes redacted Feishu proof.

## Reproduction Assessment

Yes for the likely source-level failure: current routing code and tests show omitted-account bindings miss named accounts and fall back to `main`. The exact seven-account Feishu setup was not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Unclear as a final product fix. Explicit `peer` and `accountId` guidance is necessary, but automatically treating bare multi-account bindings as wildcard routing changes isolation semantics and needs maintainer approval.
