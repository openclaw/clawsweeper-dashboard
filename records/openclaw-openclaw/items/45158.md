## Summary

Keep open: current main still has a plausible account-scope routing failure for this report, and the related fix PR is open rather than merged. Feishu passes both `accountId` and the sender/chat peer into generic routing, but generic routing only considers bindings without `accountId` for the default account, so non-default Feishu accounts can fall back to the default agent unless bindings use the exact account id or `accountId: "*"`.

## What This Changes

The issue reports that a seven-account Feishu multi-agent setup routes every user's messages to the first Feishu agent instead of the agent selected by that user's binding.

## Best Possible Solution

Keep the issue open until #49769 or an equivalent fix lands, with coverage proving multi-account Feishu or generic multi-account bindings use exact account ids or `accountId: "*"` instead of silently falling back to the default agent.

## Reproduction Assessment

Yes for the likely core routing failure: a static route-level reproduction is already encoded by `src/routing/resolve-route.test.ts`, where a binding without `accountId` misses a non-default account and falls back to `main`. The exact seven-account Feishu environment is not fully reproducible from the report because the config snippets are missing.

## Solution Assessment

Yes, the best direction is to preserve the documented default-account semantics in core routing and fix the creation/docs path so multi-account channel bindings are explicit. Broadly changing omitted `accountId` to mean all accounts would be riskier because current tests and docs define it as default-account only.
