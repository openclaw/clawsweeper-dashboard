- `impact:other`: The issue affects setup diagnostics and can hide actionable skill failures behind platform-incompatible skill noise.

## Summary

Keep open: current main and the latest release still route platform-only skill OS mismatches through the same skills-readiness warning and `doctor --fix` disable path, so the reported Windows noise is not already fixed.

## What This Changes

The issue asks doctor to skip or separately summarize skills whose declared OS requirement does not match the current host instead of listing each one as a missing requirement.

## Best Possible Solution

Keep the OS eligibility logic, but have doctor treat platform-only mismatches as intentionally skipped skills with one concise rollup instead of actionable missing-requirement warnings or disable fixes.

## Reproduction Assessment

Yes, source-reproducible: an allowed skill with `metadata.openclaw.os: ["darwin"]` on a non-Darwin host gets `missing.os`, becomes ineligible, and is emitted by doctor as an unavailable-skill warning. I did not run a live Windows doctor session in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow doctor/skills-readiness classification change: preserve OS-gated eligibility for runtime visibility, but filter or roll up platform-only mismatches before producing actionable warnings and disable repairs.
