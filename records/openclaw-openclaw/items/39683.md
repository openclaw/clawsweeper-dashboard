## Summary

Keep this PR open as a useful implementation candidate, but it should not merge as-is: current main still has the reported Feishu/Lark cron validation gap, while the branch still accepts unproven session-key targets and lacks real behavior proof.

## What This Changes

The branch adds cron service validation, regression tests, docs, and changelog text to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target while allowing unrelated edits to legacy invalid jobs.

## Best Possible Solution

Land one consolidated cron validation fix that fails Feishu/Lark announce create/update without an explicit target unless the write path can prove a deliverable session-derived target, while preserving legacy disable edits and documenting accepted forms.

## Reproduction Assessment

Yes. Source inspection shows current main can construct Feishu/Lark announce delivery without delivery.to, accept it through service/gateway validation, and later fail during store-backed outbound target resolution when no recipient exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this implementation treats any non-empty sessionTarget/sessionKey as target proof instead of verifying that the runtime resolver can load a deliverable Feishu/Lark recipient.
