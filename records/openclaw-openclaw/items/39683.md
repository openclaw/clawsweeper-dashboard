## Summary

Keep open: the linked cron bug is still present on current main/latest release, and this PR is an active fix path, but the branch still accepts a session-key string as target proof and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, cron docs, and a changelog entry to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target while preserving unrelated updates to legacy invalid jobs.

## Best Possible Solution

Keep the cron service/gateway boundary as the enforcement point, but require explicit `delivery.to` or proof from the same runtime resolver path before accepting Feishu/Lark announce delivery.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.7` shows `cron add` can send Feishu/Lark announce delivery without `--to`, and service validation accepts that shape before runtime target resolution can fail.

## Solution Assessment

No. Service-level validation is the right direction, but this PR treats a session key string as sufficient even though runtime delivery resolution is store-backed and can still fail without a real recipient.
