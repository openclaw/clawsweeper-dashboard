## Summary

Keep open: current main still accepts the reported Feishu/Lark cron shape, and this PR is a useful fix path, but the branch still has a correctness blocker and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, docs, and changelog text to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Land one consolidated fix that rejects Feishu/Lark announce create/update without an explicit target unless the write path can prove a deliverable session-derived target, while preserving legacy disable edits and closing https://github.com/openclaw/openclaw/issues/39599 after merge.

## Reproduction Assessment

Yes. Source inspection shows current main can construct Feishu/Lark announce delivery without delivery.to, accept it through service/gateway validation, and later fail during store-backed outbound target resolution when no recipient exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this implementation treats any non-empty sessionTarget/sessionKey as target proof instead of verifying that the runtime resolver can load a deliverable Feishu/Lark recipient.
