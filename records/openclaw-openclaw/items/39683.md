## Summary

Keep open: current main still accepts the reported Feishu/Lark cron shape, and this PR is a plausible narrow fix path, but it still needs real behavior proof and two validation gaps fixed before merge.

## What This Changes

The branch adds Feishu/Lark cron announce target validation in the cron service, regression coverage, CLI cron docs, and a changelog entry.

## Best Possible Solution

Land one consolidated cron service fix that validates explicit targets or the actual runtime-resolvable session context, rejects activation of unresolved legacy jobs, and still permits safe disable/remediation edits.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron add`, the protocol schema, and the cron service can accept Feishu/Lark announce delivery without `delivery.to`, while runtime target resolution can later return the Feishu missing-target error.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch still accepts unproven session-derived targets and skips validation when legacy invalid jobs are re-enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Resolve session-derived Feishu targets before accepting them:** `src/cron/service/jobs.ts:225`
  - body: The new guard treats any `resolveCronDeliverySessionKey(job)` value as target proof, but runtime delivery preflight still resolves with `params.job.sessionKey` rather than the key derived from `sessionTarget`. A session-target-only Feishu/Lark job can pass creation here and still reach runtime with no resolved recipient.
  - confidence: 0.91

  - body: `validateFeishuTarget` is false for an `enabled: true` patch because `patchTouchesDeliveryTarget` ignores activation. That lets a disabled legacy Feishu/Lark announce job with no target be reactivated and fail at runtime; keep disabling and remediation edits allowed, but validate activation paths.
  - confidence: 0.88
