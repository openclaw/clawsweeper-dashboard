## Summary

Keep open: this is a useful repair for a real Feishu/Lark cron delivery bug, but the current branch still has merge-blocking validation gaps and no real behavior proof, and current main has not fixed the reported behavior.

## What This Changes

The PR adds cron-service validation, tests, docs, and a changelog entry to reject Feishu/Lark announce cron jobs that lack an explicit or session-derived delivery target.

## Best Possible Solution

Land one consolidated current-main fix that rejects unresolved Feishu/Lark announce delivery on create, re-enable, run, and delivery-target changes using the same target context the runtime resolver uses, while preserving disable/remediation edits and adding focused proof.

## Reproduction Assessment

Yes, at source level. Current main constructs announce delivery from optional `--channel` and `--to`, service validation has no Feishu/Lark missing-target guard, and runtime outbound resolution can later return the Feishu missing-target error.

## Solution Assessment

No. The fail-fast direction is right, but this PR should validate reactivation and session-derived targets against the same context used at runtime before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Validate re-enabled Feishu jobs before delivery:** `src/cron/service/jobs.ts:703-705`
  - body: `patchTouchesDeliveryTarget` ignores `enabled`, so a stored Feishu/Lark announce job with no target can be disabled as legacy data and later re-enabled with `{ enabled: true }` without this guard running. That recreates the runtime delivery failure this PR is meant to prevent; include activation/re-enable in the validation trigger while still allowing disabling and remediation edits.
  - confidence: 0.93
- **[P1] Use the runtime session key for target checks:** `src/cron/service/jobs.ts:225`
  - body: `hasDeterministicTarget` treats any `resolveCronDeliverySessionKey(job)` value as a valid Feishu/Lark target, but cron execution resolves delivery with `params.job.sessionKey` while gateway passes `sessionTarget` only as the isolated run input session key. A job accepted because `sessionTarget` starts with `session:` can still reach runtime with no effective target, so validate against the same session key/target resolver path that execution uses or require explicit `delivery.to`.
  - confidence: 0.88
