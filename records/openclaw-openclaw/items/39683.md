## Summary

Keep open: current main still accepts Feishu/Lark announce jobs without a delivery target, and this PR is directionally right but still lets unresolved session-derived targets bypass validation.

## What This Changes

The PR adds shared cron service validation, regression tests, docs, and a changelog entry so Feishu/Lark announce jobs require an explicit or session-derived delivery target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the shared cron service as the safety boundary, but accept a missing Feishu/Lark delivery.to only when the code can prove a real stored/session delivery route, then rebase and land with the existing tests, docs, and changelog preserved.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts announce delivery with channel=feishu and no to, and the PR-specific bypass is visible because any non-empty session key satisfies the new guard even when runtime target resolution may still fail.

## Solution Assessment

No, not as currently written. The shared helper is the right place to reject invalid jobs, but the session-derived exception must prove a resolvable route instead of treating an arbitrary session key as a target.
