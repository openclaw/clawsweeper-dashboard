## Summary

Keep open: current main still lacks Feishu/Lark create-time target validation, and this PR is the canonical linked fix path, but the current patch still accepts session-key-only cases without proving a runtime-resolved recipient. External real behavior proof is also missing.

## What This Changes

The PR adds cron service validation, regression tests, cron docs, and a changelog entry for rejecting Feishu/Lark announce jobs without an explicit or session-derived target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the cron service as the enforcement boundary, but require explicit `delivery.to` or a runtime-equivalent resolver proof, then land one linked fix with aligned docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron add` can build Feishu/Lark announce delivery without `--to`, and service validation accepts that shape before runtime target resolution can fail.

## Solution Assessment

No. Service-level validation is the right direction, but the current PR treats a session key string as sufficient even though runtime resolution is store-backed and the isolated run path passes only `job.sessionKey`.
