## Summary

Keep open: current main still accepts the invalid Feishu/Lark cron shape, and this PR is useful but not merge-ready because its session-derived-target exception still lets unresolved jobs through.

## What This Changes

The PR adds cron service validation, regression tests, docs, and a changelog entry requiring Feishu/Lark announce jobs to have an explicit or session-derived delivery target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the shared cron service as the safety boundary, but only accept missing Feishu/Lark `delivery.to` when the implementation can prove a real stored/session delivery route; otherwise reject create/update while still allowing unrelated disable edits to legacy invalid jobs.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts Feishu/Lark announce delivery without `delivery.to`, and the PR-specific bypass is visible from the patch because any non-empty delivery session key satisfies the new guard.

## Solution Assessment

No, not as currently written. Shared service validation is the right place to fix this, but the session-derived exception must be backed by a resolvable route instead of a key string.
