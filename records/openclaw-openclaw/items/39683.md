## Summary

Keep open. Current main still accepts the invalid Feishu/Lark announce jobs described by the linked issue, and this PR is the right narrow surface, but the proposed validation still lets arbitrary custom session targets bypass the missing-target check before failing later at runtime.

## What This Changes

The PR adds cron service validation, regression tests, docs, and a changelog entry so Feishu/Lark announce jobs require an explicit or session-derived delivery target while legacy invalid jobs can still be disabled.

## Best Possible Solution

Keep the shared cron service as the safety boundary, but tighten the Feishu/Lark exception so only an explicit target or a truly resolvable stored/session delivery context satisfies create/update validation, while preserving the legacy-invalid-job disable path and the docs/changelog coverage.

## Reproduction Assessment

Yes. Current main can be verified statically: cron add and the protocol allow omitted to, and createJob/applyJobPatch call a validator that does not inspect Feishu/Lark targets. The PR-specific gap is also reproducible statically because any custom sessionTarget makes resolveCronDeliverySessionKey truthy even when runtime delivery has no stored target to resolve.

## Solution Assessment

No, not as currently written. The shared helper is the right place to cover create and delivery-update paths, but the session-derived allowance needs to prove a real delivery route instead of accepting any non-empty session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require a real session-derived target:** `src/cron/service/jobs.ts:222`
  - body: The new Feishu/Lark guard can be bypassed with any custom session target, for example `session:daily-brief`, because `resolveCronDeliverySessionKey(job)` returns a non-empty string even when there is no stored chat route. Runtime delivery does not reconstruct a Feishu target from that string; it loads the session store entry and otherwise falls through to the same missing-target error. That means the PR still allows unresolved Feishu/Lark jobs to be created without `delivery.to`, which is the bug it is meant to reject.
