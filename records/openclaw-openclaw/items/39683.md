## Summary

Keep open: the PR targets a valid cron bug, but the branch still allows missing Feishu/Lark targets through a session-derived exception that is not proven to match runtime delivery resolution.

## What This Changes

The PR adds cron service validation, regression tests, docs, and a changelog entry requiring Feishu/Lark announce jobs to have an explicit or session-derived delivery target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the shared cron service as the validation boundary, but only accept missing Feishu/Lark `delivery.to` when the same runtime resolver can prove a concrete recipient, while preserving the legacy-disable bypass.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts Feishu/Lark announce delivery without `delivery.to`, and the PR-specific gap is visible because the new exception can be satisfied by a session key that runtime delivery may not resolve.

## Solution Assessment

No, not as currently written. Service-level validation is the right approach, but the session-derived exception should be tied to a runtime-resolvable route instead of a non-empty session key string.
