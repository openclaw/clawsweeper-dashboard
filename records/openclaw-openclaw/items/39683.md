## Summary

Keep this PR open because the underlying bug is still present on current main, but the branch is not landable yet: it still accepts unresolved session-key-only Feishu/Lark jobs, skips validation when re-enabling invalid legacy jobs, and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, docs, and changelog text for rejecting Feishu/Lark announce jobs without an explicit or session-derived delivery target.

## Best Possible Solution

Land one consolidated cron service-boundary fix that requires an explicit Feishu/Lark target or a verified routable session delivery context, rejects activation of unresolved legacy jobs, and still allows safe remediation such as disabling invalid jobs.

## Reproduction Assessment

Yes, source-derived: current main can still create or update Feishu/Lark announce delivery without delivery.to, and runtime resolution can later fail when no routable recipient exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch accepts a session identifier as delivery proof and skips validation for enabled-only updates that can make invalid jobs runnable.
