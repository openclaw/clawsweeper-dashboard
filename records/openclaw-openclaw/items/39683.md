## What This Changes

This PR adds cron service validation, regression tests, docs, and a changelog entry to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target.

## Best Possible Solution

Land one consolidated current-main fix that validates Feishu/Lark announce delivery against the same routable target context used at runtime, rejects activation of unresolved jobs, preserves disable/remediation edits, and includes focused regression coverage plus real proof.

## Reproduction Assessment

Yes, at source level. Current main still permits Feishu announce delivery without delivery.to through the CLI/service/schema path, and outbound target resolution can later return Feishu's missing-target error when no effective target exists.

## Solution Assessment

No, not as currently patched. Service-boundary validation is the right direction, but this PR needs runtime-equivalent target validation, enabled=true activation coverage, conflict resolution, and real behavior proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate the effective Feishu session target:** `src/cron/service/jobs.ts:225-226`
  - body: The new guard treats any resolved cron delivery session key as a routable Feishu/Lark target, but runtime isolated delivery passes `job.sessionKey` into `resolveDeliveryTarget`, not `sessionTarget`, and then resolves through stored delivery context and outbound target resolution. A `sessionTarget`-only or otherwise unresolved session key can pass this guard and still fail at runtime with the same missing-target error.
  - confidence: 0.86
- **[P2] Reject re-enabling unresolved Feishu jobs:** `src/cron/service/jobs.ts:703-705`
  - body: `patchTouchesDeliveryTarget` ignores `enabled`, so `applyJobPatch(job, { enabled: true })` on a disabled legacy Feishu/Lark announce job with no target skips the new guard and can reactivate a job that still misses delivery. Include activation in the validation condition while still allowing `enabled: false` and target-remediation edits.
  - confidence: 0.88
