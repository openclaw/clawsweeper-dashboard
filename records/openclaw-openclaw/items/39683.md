## Summary

Keep this PR open: the linked bug is still source-reproducible on current main and this branch is a plausible fix path, but the current head still accepts unresolved session-key targets, allows re-enabling legacy invalid jobs, and lacks after-fix real behavior proof.

## What This Changes

The PR adds Feishu/Lark announce target validation in cron job create/update paths, regression tests, cron docs, and a changelog entry.

## Best Possible Solution

Land one consolidated service-boundary validation that requires an explicit Feishu/Lark target or verified routable session delivery context, rejects activation of unresolved legacy jobs, and preserves safe disable/edit remediation.

## Reproduction Assessment

Yes, source-reproducible: current main can build and accept Feishu/Lark announce delivery without delivery.to, and runtime target resolution can later fail when no effective target exists. I did not run a live Feishu command in this read-only review.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch still treats any session key as target proof and skips validation for enable-only patches that can make unresolved legacy jobs runnable again.
