## Summary

Keep open. Current main still lacks the Feishu/Lark fail-fast validation, and this PR is the canonical-looking narrow branch after a maintainer repair, but the current head still accepts a session key string as target proof and still lacks real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, docs, and a changelog entry for rejecting Feishu/Lark announce jobs without an explicit or session-derived delivery target while allowing unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the cron service as the enforcement boundary, but require explicit `delivery.to` or a runtime-equivalent resolved target, then land one linked fix with aligned docs and regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main can create an isolated Feishu/Lark announce delivery with no `delivery.to`, and the service only validates webhook targets before runtime resolution can fail. I did not execute the CLI because this was a read-only review and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

No. Service-level validation is the right direction, but this PR is not yet the best fix because a session key string is not the same as a runtime-resolvable Feishu/Lark recipient.
