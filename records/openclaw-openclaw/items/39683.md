## Summary

Keep this PR open. Current main still accepts the reported Feishu/Lark cron announce shape without `delivery.to`, but the PR branch still has a blocking correctness gap around session-derived targets and lacks after-fix real behavior proof.

## What This Changes

This PR adds cron service validation, regression tests, cron docs, and changelog text for rejecting Feishu/Lark announce jobs without an explicit or session-derived delivery target while allowing unrelated edits to legacy invalid jobs.

## Best Possible Solution

Land one narrow cron service validation fix that rejects Feishu/Lark announce delivery unless `delivery.to` is explicit or the same runtime resolver can prove a target, with aligned docs, regression tests, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron add` can submit an isolated Feishu announce delivery object without `to`, and current gateway/service validation accepts that shape before runtime target resolution can fail.

## Solution Assessment

No. Service-level validation is the right boundary, but this PR is not yet the best implementation because it treats any session-derived key string as sufficient even though the runtime delivery path still resolves from `job.sessionKey`.
