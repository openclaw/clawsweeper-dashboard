## Summary

Keep open: current main has not fixed the Feishu/Lark cron delivery bug, and this focused branch remains useful after maintainer repair, but it is not merge-ready because its validation still lets targetless jobs through session-key and re-enable paths and lacks real behavior proof.

## What This Changes

The branch adds cron service validation, regression tests, docs, and changelog text to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target while preserving legacy cleanup edits.

## Best Possible Solution

Rebase the focused branch, validate Feishu/Lark create/update/re-enable paths against an explicit target or the same runtime-resolvable delivery context, preserve disable/repair of legacy invalid jobs, and add redacted real cron command proof.

## Reproduction Assessment

Yes, at source level. Current main builds announce delivery from optional `--channel`/`--to`, service and gateway validation lack a Feishu/Lark missing-target guard, and the outbound resolver can later return Feishu's missing-target error.

## Solution Assessment

No. The fail-fast direction is right, but this implementation treats any session key as a resolved target and lets re-enable patches bypass the guard; it should use runtime target resolution or require explicit `to` in the affected paths.
