## Summary

Keep this PR open: the linked cron bug is still present on current main, but this branch still has a blocking correctness issue in the session-derived target shortcut and still lacks after-fix real behavior proof.

## What This Changes

The branch adds cron service validation, regression tests, docs, and changelog text for rejecting Feishu/Lark announce jobs without an explicit or session-derived delivery target while allowing unrelated edits to legacy invalid jobs.

## Best Possible Solution

Land one clean fix path that enforces Feishu/Lark announce target validation at the cron service/gateway boundary using resolver-backed target proof, keeps legacy-disable edits possible, and documents the accepted target forms.

## Reproduction Assessment

Yes. Source inspection on current main shows cron add can submit Feishu/Lark announce delivery without delivery.to, the service accepts it, and runtime delivery resolution can later fail when no effective recipient exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this implementation treats any non-empty sessionKey or session: target as deterministic even though the runtime resolver still needs a store-backed deliverable target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
