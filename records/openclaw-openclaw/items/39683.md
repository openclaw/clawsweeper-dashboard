## Summary

Keep this PR open: current main still has the Feishu/Lark cron validation gap, and the repaired branch is close, but it still treats a session-key string as proof of a real delivery target and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, cron docs, and a changelog entry to reject Feishu/Lark announce jobs without an explicit or session-derived delivery target while preserving unrelated updates to legacy invalid jobs.

## Best Possible Solution

Keep the cron service/gateway boundary as the enforcement point, but require explicit `delivery.to` or proof from the same store-backed runtime resolver path before accepting Feishu/Lark announce delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main can construct and accept Feishu/Lark announce delivery without `delivery.to`, and runtime target resolution can later fail when no recipient exists. I did not run a live Feishu cron command in this read-only review.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this PR treats `resolveCronDeliverySessionKey(job)` as sufficient even though it only proves an identifier, not a resolvable Feishu/Lark target.
