## Summary

Keep open: the PR addresses a valid cron delivery bug on main, but the current guard still allows Feishu/Lark jobs without `delivery.to` whenever a session key string can be derived, which is weaker than the runtime delivery resolver and can still fail later.

## What This Changes

The PR adds cron service validation, tests, docs, and a changelog entry for rejecting Feishu/Lark announce jobs without an explicit or session-derived delivery target while allowing unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the shared cron service as the validation boundary, but only allow missing Feishu/Lark `delivery.to` when the runtime-equivalent resolver can prove a concrete recipient, while preserving the legacy invalid-job disable/update bypass.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts Feishu/Lark announce delivery without `delivery.to`, and the PR-specific remaining gap is visible because a `sessionTarget`-derived key can satisfy creation without proving a runtime-resolved recipient.

## Solution Assessment

No, not as currently written. Service-level validation is the right approach, but the session-derived exception should be tied to runtime resolver/store proof or removed in favor of requiring explicit `delivery.to`.
