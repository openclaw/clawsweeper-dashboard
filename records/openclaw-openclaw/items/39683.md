## Summary

Keep open: current main still accepts the invalid Feishu/Lark announce cron shape, and this branch remains useful, but the latest head still has merge-blocking validation gaps and lacks real behavior proof.

## What This Changes

The PR adds cron service validation, regression tests, CLI cron docs, and a changelog entry so Feishu/Lark announce cron jobs require an explicit or session-derived delivery target.

## Best Possible Solution

Land one consolidated cron service/gateway fix that validates explicit targets or the actual runtime-resolvable session context, rejects activation of unresolved legacy jobs, and still permits disabling or remediation edits.

## Reproduction Assessment

Yes. Source inspection on current main shows cron add, protocol schema, and service validation can accept Feishu/Lark announce delivery without delivery.to, while runtime target resolution can later fail when no effective target exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this PR still accepts sessionTarget-derived targets that the runtime preflight does not use consistently and skips validation when invalid legacy jobs are re-enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Resolve session-derived targets before accepting Feishu jobs:** `src/cron/service/jobs.ts:225`
  - body: The guard treats `resolveCronDeliverySessionKey(job)` as target proof, but current runtime preflight passes `params.job.sessionKey` to `resolveDeliveryTarget`. A job with only `sessionTarget: "session:..."` can pass creation here and still have no recipient at runtime; validate against the same resolved target the runner will use.
  - confidence: 0.91
- **[P2] Validate Feishu targets when re-enabling jobs:** `src/cron/service/jobs.ts:703-705`
  - body: This skip only looks at delivery/session target patches, so `applyJobPatch(job, { enabled: true })` can reactivate a disabled legacy Feishu/Lark announce job with no `delivery.to`. Keep disable/remediation edits allowed, but validate activation when the resulting job is enabled.
  - confidence: 0.89
