## Summary

Keep this PR open, but it is not merge-ready: it addresses a real source-reproducible cron delivery bug, yet the latest branch still has two correctness gaps and only test/check claims instead of real Feishu/Lark behavior proof.

## What This Changes

The branch adds cron service validation, regression tests, docs, and a changelog entry requiring Feishu/Lark announce cron jobs to have an explicit or deterministic session-derived delivery target.

## Best Possible Solution

Land one consolidated fix that validates Feishu/Lark announce targets against the same runtime-resolvable delivery context, rejects activation of unresolved legacy jobs, still permits disabling/remediation edits, and includes focused regression coverage plus redacted real command proof.

## Reproduction Assessment

Yes. Source inspection on current main shows cron add, protocol schema, and service validation can accept Feishu/Lark announce delivery without delivery.to, while runtime target resolution can later fail when no effective target exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch still accepts sessionTarget-derived proof that the runtime preflight does not use consistently and skips validation when invalid legacy jobs are re-enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use the runtime session key for Feishu target validation:** `src/cron/service/jobs.ts:225`
  - body: The guard accepts a target from resolveCronDeliverySessionKey(job), which can be true solely because sessionTarget starts with session:. The runner passes that sessionTarget-derived key as input.sessionKey, but delivery preflight still resolves with params.job.sessionKey, so a sessionTarget-only Feishu/Lark job can pass creation and still be unresolved at runtime. Validate against the same effective target the runner will use or thread that key into runtime resolution.
  - confidence: 0.91
- **[P2] Validate Feishu targets when re-enabling jobs:** `src/cron/service/jobs.ts:703-705`
  - body: patchTouchesDeliveryTarget is false for applyJobPatch(job, { enabled: true }), so a disabled legacy Feishu/Lark announce job with no target can be reactivated into the same runtime failure. Keep enabled:false and remediation edits allowed, but run the target guard when the resulting job is enabled or the patch turns it back on.
  - confidence: 0.9
