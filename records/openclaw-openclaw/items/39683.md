## Summary

Keep open: current main still lacks the Feishu/Lark create-time validation, and this PR is the canonical narrow branch but still accepts session-key strings that runtime fallback delivery may not resolve; real behavior proof is also missing.

## What This Changes

The PR adds cron service validation, regression tests, cron docs, and a changelog entry for rejecting Feishu/Lark announce jobs without an explicit or session-derived target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Land one narrow cron validation fix that rejects Feishu/Lark announce delivery unless delivery.to is present or the exact runtime delivery resolver can prove a target, with docs and regression tests aligned.

## Reproduction Assessment

Yes, by source inspection. Current main lets cron add build an isolated Feishu/Lark announce job without --to, the service accepts it, and runtime target resolution can later fail when no recipient is available.

## Solution Assessment

No. Service-level validation is the right direction, but accepting any session key as deterministic is not sufficient unless fallback delivery validates or uses that same resolved session key.
