## Summary

Keep open: current main still has the reported Feishu/Lark cron validation gap and this branch remains relevant, but the latest PR head is not correct as-is because it accepts unverified session keys as target proof and skips validation when enabling legacy invalid jobs. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR branch adds Feishu/Lark cron announce target validation in `src/cron/service/jobs.ts`, regression coverage in `src/cron/service.jobs.test.ts`, a cron docs note, and a changelog entry.

## Best Possible Solution

Land one consolidated cron service-boundary fix that requires an explicit Feishu/Lark target or a verified routable session delivery context, rejects activation of unresolved legacy jobs, and still allows safe remediation such as disabling invalid jobs.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI, gateway/protocol, and cron service paths can still accept Feishu/Lark announce delivery without `delivery.to`, while runtime outbound resolution can later fail when no effective target exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch still accepts a session identifier as target proof and skips validation for enable-only updates that can make invalid legacy jobs runnable again.

## Review Findings

That only proves a session key exists; runtime delivery still has to load stored context and can fail with no Feishu/Lark recipient, so create/update can still accept jobs that fail later.
  - confidence: 0.9

  - confidence: 0.86
