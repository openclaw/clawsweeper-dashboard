## Summary

Keep this PR open: the linked Feishu/Lark cron bug is still present on current main, and the PR is a focused candidate, but the latest diff still accepts session-key-only targets without proving that runtime delivery can resolve a recipient.

## What This Changes

The PR updates cron service validation, tests, docs, and changelog text to reject Feishu/Lark announce cron jobs without an explicit or session-derived delivery target while preserving unrelated edits to legacy invalid jobs.

## Best Possible Solution

Keep the cron service/gateway boundary as the enforcement point, but require explicit delivery.to or validate the same store-backed runtime resolver path before accepting Feishu/Lark announce delivery.

## Reproduction Assessment

Yes. Source inspection shows current main can construct and accept Feishu/Lark announce delivery without delivery.to, and runtime target resolution can later fail when no recipient exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this PR treats resolveCronDeliverySessionKey(job) as sufficient even though it only proves an identifier, not a resolvable Feishu/Lark target.
