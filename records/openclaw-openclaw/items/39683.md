## Best Possible Solution

Land one consolidated current-main fix that validates Feishu/Lark announce delivery against the same routable target context used at runtime, rejects activation of unresolved jobs, preserves disable/remediation edits, and includes focused regression coverage plus real proof.

## Reproduction Assessment

Yes, at source level. Current main can build and validate `delivery: { mode: "announce", channel: "feishu" }` without `to`, while outbound resolution can later return a missing-target error when no effective route exists.

## Solution Assessment

No, not as currently patched. Service-boundary validation is the right direction, but this PR needs runtime-equivalent target validation, activation coverage for legacy invalid jobs, conflict resolution, and real behavior proof.
