## Solution Assessment

No. The fail-fast direction is right, but this PR is not the best complete fix until it validates the same routable target context used at runtime and rejects unresolved reactivation while still allowing disable/remediation edits.

## Review Findings

Full review comments:

- **[P2] Validate the effective Feishu session target:** `src/cron/service/jobs.ts:225`
  - body: This check treats any non-empty `sessionTarget`/`sessionKey` as a deterministic Feishu/Lark target, but runtime delivery resolves that key through session-store delivery context before it has a usable `to`. A job like `sessionTarget: "session:missing"` can pass create/update here and still fail later with the same missing-target delivery error, so the guard needs to validate the runtime-equivalent resolved route or require explicit `delivery.to`.
  - confidence: 0.86
- **[P2] Reject re-enabling unresolved Feishu jobs:** `src/cron/service/jobs.ts:703-705`
  - body: Because `validateFeishuTarget` only follows delivery/session-target/session-key changes, `applyJobPatch(job, { enabled: true })` can reactivate a legacy disabled Feishu/Lark announce job that still has no resolvable target. Disabling or unrelated cleanup should remain allowed, but activation needs to run the same unresolved-target guard so the job does not get scheduled and fail at runtime again.
  - confidence: 0.84
