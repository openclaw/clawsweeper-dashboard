## Summary

Keep open: current main still lacks Feishu/Lark fail-fast target validation, but this PR still has two correctness gaps in the proposed validation and lacks required real behavior proof.

## What This Changes

The branch adds cron service validation, regression tests, cron docs, and a changelog entry for rejecting unresolved Feishu/Lark announce jobs without an explicit or session-derived target.

## Best Possible Solution

Land one consolidated service-boundary fix that validates explicit targets or actually resolvable session context, rejects activation of unresolved legacy jobs, and preserves safe disable/remediation edits.

## Reproduction Assessment

Yes. Source inspection shows current main can accept Feishu/Lark announce delivery without delivery.to, and runtime target resolution can later fail when no explicit or default target exists.

## Solution Assessment

No. Service-level fail-fast validation is the right direction, but this branch still treats an unverified session key as target proof and skips validation when re-enabling legacy invalid jobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve session-derived Feishu targets before accepting them:** `src/cron/service/jobs.ts:225`
  - body: The guard treats any `resolveCronDeliverySessionKey(job)` value as delivery proof. Runtime delivery resolution is still responsible for finding a real recipient, so a job with no `delivery.to` can pass creation and still fail later when the stored/default target is missing.
  - confidence: 0.9
- **[P2] Validate Feishu targets when re-enabling jobs:** `src/cron/service/jobs.ts:703-705`
  - body: `patchTouchesDeliveryTarget` is false for an `enabled: true` patch, so a legacy Feishu/Lark announce job without a target can be reactivated without the new validation. Keep disabling and unrelated remediation edits allowed, but validate activation paths.
  - confidence: 0.86
