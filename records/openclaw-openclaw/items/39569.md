## Summary

Keep open. The Feishu bug is still present on current main and this PR targets the right area, but the patch needs rework because the orphan-placeholder fallback can bypass mention gating, parse-time `rootId` clearing breaks topic/thread routing, and real behavior proof is missing.

## What This Changes

The PR changes Feishu mention parsing to infer dropped bot mentions from orphaned reply placeholders, clears `rootId` for mentioned replies, and adds parser tests.

## Best Possible Solution

Rework the Feishu plugin fix in the current parser/routing split: add the strongest available bot-specific dropped-mention detection, preserve topic and configured-thread roots, and land focused regression coverage linked to the open issue.

## Reproduction Assessment

Yes by source inspection. A Feishu group text reply with `parent_id`/`root_id`, text containing the dropped bot placeholder, and a non-empty `mentions` array that omits the bot leaves `mentionedBot=false`; the group `requireMention` gate then returns early.

## Solution Assessment

No. The PR targets the right bug, but treating any orphan placeholder as a bot mention is too broad and clearing `rootId` before scope resolution is not the narrowest maintainable routing fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots for mentioned replies:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` during parse for every mentioned reply happens before `groupSessionScope` and `replyInThread` are known. Topic and explicitly threaded groups rely on that root as the session key and reply target, so addressed replies can collapse to the chat-level session or wrong target.
  - confidence: 0.88
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback treats any unmatched `@_...` token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can satisfy `requireMention` and cause OpenClaw to respond without proof the bot was addressed.
  - confidence: 0.86
