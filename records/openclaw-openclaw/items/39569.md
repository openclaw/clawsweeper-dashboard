## Summary

Keep open. Current main still misses the reported Feishu reply-mention path, but this PR is not safe to merge as-is because it lacks inspectable live proof and introduces broad mention-gating and root-routing regressions.

## What This Changes

The branch modifies Feishu mention parsing to treat orphaned reply placeholders as bot mentions, clears rootId for mentioned replies, and adds parser tests.

## Best Possible Solution

Rework the Feishu plugin fix to detect dropped bot mentions with the strongest bot-specific signal available, preserve topic/thread roots until scope-aware routing, and add focused regression tests plus redacted live proof.

## Reproduction Assessment

Yes by source inspection. A Feishu group reply whose text still includes the bot placeholder but whose non-empty mentions array omits botOpenId returns mentionedBot=false, then the mention-required group ingress returns before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR's any-orphan heuristic and parse-time root clearing are broader than the safe behavior; the maintainable fix should be bot-specific and scope-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing rootId in parseFeishuMessageEvent happens before groupSessionScope is known. In group_topic and group_topic_sender modes, rootId is the topic key, so addressed replies with root_id but no thread_id can fall back to the chat-level session and merge unrelated topic context.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback marks any unmatched @_..._N token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can bypass requireMention and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
