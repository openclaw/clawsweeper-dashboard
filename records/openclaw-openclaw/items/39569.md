## Summary

Keep open. Current main still appears to have the Feishu reply-mention drop path, but this PR is not mergeable as-is because it lacks real behavior proof and broadens mention activation/root routing in ways that can break existing Feishu thread behavior.

## What This Changes

The branch modifies the Feishu plugin to infer bot mentions from orphaned reply placeholders, clear `rootId` for mentioned replies, and add parser regression tests.

## Best Possible Solution

Rework the Feishu plugin fix in the current `bot-content.ts` split with the strongest available bot-specific dropped-mention signal, preserve topic/configured-thread roots, and add focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. A Feishu group reply with `root_id`, text containing the bot placeholder, and a non-empty `mentions` array that omits the bot leaves `mentionedBot=false` on current main and is dropped by the `requireMention` gate.

## Solution Assessment

No. The PR targets the right Feishu plugin bug, but any-orphan placeholder detection is too broad and parse-time `rootId` clearing is not the narrowest maintainable routing fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots for mentioned replies:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` during parse for every mentioned reply happens before group session scope and `replyInThread` are known. Topic and explicitly threaded groups rely on that root as the session key and reply target, so addressed replies can collapse to the chat-level session or wrong target.
  - confidence: 0.88
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback treats any unmatched `@_...` token in a reply as a bot mention. A literal placeholder or dropped non-bot mention can satisfy `requireMention` and cause OpenClaw to respond without proof the bot was addressed.
  - confidence: 0.86
