## Summary

Keep open. Current main still misses the reported Feishu dropped-reply-mention path, but this PR should not merge as-is because its orphan-placeholder heuristic can bypass mention gating and its parse-time root clearing can break topic-scoped routing.

## What This Changes

The PR changes Feishu mention parsing to treat orphan reply placeholders as bot mentions, clears `rootId` for mentioned replies, and adds parser tests.

## Best Possible Solution

Rework the Feishu plugin to use bot-specific dropped-mention evidence and choose reply targets after group session scope is resolved, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, by source inspection. A Feishu group reply whose text still contains the omitted bot placeholder but whose non-empty `mentions` array lacks `botOpenId` leaves `mentionedBot=false`, then the mention-required group gate returns before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR's any-orphan heuristic and parse-time `rootId` clearing are broader than the narrow maintainable repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` in `parseFeishuMessageEvent` happens before `groupSessionScope` is known. In `group_topic` and `group_topic_sender` modes, `rootId` is the topic key, so addressed replies with `root_id` but no `thread_id` can fall back to the chat-level session and merge unrelated topic context.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback marks any unmatched `@_..._N` token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can bypass `requireMention` and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
