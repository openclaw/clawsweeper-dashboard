## Summary

Keep open: current main still has the Feishu reply-mention gap, and this PR needs rework plus inspectable real behavior proof before it is mergeable.

## What This Changes

The PR changes Feishu parsing to infer bot mentions from orphaned reply placeholders, clears `rootId` for mentioned replies, and adds parser tests.

## Best Possible Solution

Rework the Feishu plugin to detect dropped bot mentions with bot-specific evidence and choose reply targets after group session scope is known, preserving topic routes with focused tests and redacted live proof.

## Reproduction Assessment

Yes, at source level. A Feishu group reply whose text still contains the omitted bot placeholder but whose non-empty `mentions` array lacks `botOpenId` leaves `mentionedBot=false`, then mention-required group ingress returns before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR's any-orphan heuristic and parse-time `rootId` clearing are broader than the narrow maintainable repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` in `parseFeishuMessageEvent` happens before `groupSessionScope` is known. In `group_topic` and `group_topic_sender` modes, `rootId` is the topic key, so addressed replies with `root_id` but no `thread_id` can fall back to the chat-level session and merge unrelated topic context.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback marks any unmatched `@_..._N` token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can bypass `requireMention` and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
