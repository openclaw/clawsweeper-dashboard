## Summary

Keep open. The Feishu reply-mention bug remains source-reproducible on current main, but this PR should not merge as-is because the orphan-placeholder heuristic can bypass mention gating and the root-clearing change conflicts with topic-session routing.

## What This Changes

The PR changes Feishu message parsing to infer bot mentions from orphaned reply placeholders, clear reply root metadata for mentioned replies, and add parser regression tests.

## Best Possible Solution

Rework the fix in the current Feishu parser/routing split, use the strongest bot-specific evidence available for dropped reply mentions, preserve topic roots, adjust normal-group thread reply behavior deliberately, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Feishu group text reply with parent_id/root_id, a bot placeholder in text, and mentions missing the bot leaves mentionedBot false on current main, so requireMention drops it.

## Solution Assessment

No. The PR addresses the right bug, but any-orphan placeholder detection is too broad and global rootId clearing is not the narrowest maintainable routing fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots for mentioned replies:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing rootId while parsing every mentioned reply runs before groupSessionScope is known. In group_topic and group_topic_sender modes, that root is the topic key and reply target, so addressed replies can collapse into the chat-level session or the wrong target.
  - confidence: 0.88
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:485`
  - body: The fallback treats any unmatched @_... token in reply text as mentionedBot. A literal placeholder or dropped non-bot mention can satisfy requireMention and trigger OpenClaw even though the bot was not identified.
  - confidence: 0.86
