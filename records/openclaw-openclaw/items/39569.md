## Summary

Keep open. Current main still has the Feishu reply-mention bug, but this PR should not merge as-is because it broadens mention activation, clears thread roots before routing policy is known, and lacks attachable real behavior proof.

## What This Changes

The PR changes the Feishu plugin to infer bot mentions from orphaned reply placeholders, clear rootId for mentioned replies, and add parser regression tests.

## Best Possible Solution

Rework the Feishu plugin fix in the current bot-content split with the strongest available bot-specific dropped-mention detection, preserve topic and configured-thread roots, and add focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. A Feishu group reply with parent_id/root_id, text containing the bot placeholder, and mentions that omit the bot still leaves mentionedBot=false on current main and is dropped by the requireMention gate.

## Solution Assessment

No. The PR targets the right bug, but any-orphan placeholder detection is too broad and parse-time rootId clearing is not the narrowest maintainable routing fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots for mentioned replies:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing rootId during parse for every mentioned reply happens before groupSessionScope and replyInThread are known. Topic and explicitly threaded groups rely on that root as the session key and reply target, so addressed replies can collapse to the chat-level session or wrong target.
  - confidence: 0.88
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback treats any unmatched @_... token in a reply as a bot mention. A literal placeholder or dropped non-bot mention can satisfy requireMention and cause OpenClaw to respond without proof the bot was addressed.
  - confidence: 0.86

## Security Review
