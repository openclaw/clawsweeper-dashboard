## Summary

Keep open: this is a valid Feishu bug path, but the PR is not merge-ready because the orphan-placeholder heuristic can bypass mention-required group ingress, the parse-time `rootId` clearing can break topic-scoped routing, and inspectable real behavior proof is still missing.

## What This Changes

The PR changes Feishu mention parsing to infer dropped bot mentions in reply text, clears reply root routing when a bot mention is inferred, and adds parser tests for those scenarios.

## Best Possible Solution

Land a scoped Feishu-plugin fix that uses bot-specific dropped-mention evidence and chooses reply targets after group session scope is known, preserving topic-mode roots.

## Reproduction Assessment

Yes, at source level. A Feishu group reply whose text still contains the omitted bot placeholder but whose non-empty `mentions` array lacks `botOpenId` leaves `mentionedBot=false`, then mention-required group ingress returns before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR's any-orphan heuristic and parse-time `rootId` clearing are broader than the narrow maintainable repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:808-813`
  - body: Clearing `rootId` in `parseFeishuMessageEvent` happens before `groupSessionScope` is known. In `group_topic` and `group_topic_sender` modes, `rootId` is the topic key, so addressed replies with `root_id` can fall back to the chat-level session or lose the topic reply target.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:481-482`
  - body: The fallback marks any unmatched `@_..._N` token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can bypass `requireMention` and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
