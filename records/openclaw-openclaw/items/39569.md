## Summary

Keep open: the linked Feishu bug is still source-reproducible on current main, but this PR is not merge-ready because the branch conflicts, lacks inspectable real Feishu proof, and introduces routing/gating regressions.

## What This Changes

The branch adds Feishu reply-message orphan mention placeholder detection, clears `rootId` for mentioned replies, and adds parser tests for those cases.

## Best Possible Solution

Rework the Feishu plugin fix to use bot-specific dropped-mention evidence and choose reply targets after group session scope is known, then attach redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A Feishu group reply whose non-empty `mentions` array omits `botOpenId` leaves `mentionedBot=false`, then mention-required group ingress can return before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR’s any-orphan placeholder heuristic and parse-time `rootId` clearing are broader than the narrow maintainable repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` in `parseFeishuMessageEvent` runs before `groupSessionScope` is resolved. For `group_topic` and `group_topic_sender`, `rootId` is the topic/session key, so addressed replies with `root_id` can fall back to the chat-level session and merge unrelated topic context.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: This fallback marks any unmatched placeholder token in reply text as a bot mention. A literal token or dropped non-bot mention can bypass `requireMention` and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
