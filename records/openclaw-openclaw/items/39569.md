## Summary

Keep open: the linked Feishu bug is still source-reproducible on current main, but this PR should not merge as-is because its orphan-placeholder heuristic can bypass mention gating, its parse-time root clearing can break topic-scoped routing, and it lacks inspectable live Feishu proof.

## What This Changes

The PR changes Feishu reply mention parsing to treat orphan text placeholders as bot mentions, clears `rootId` for mentioned replies, and adds parser tests.

## Best Possible Solution

Implement a Feishu-owned, bot-specific dropped-mention detector and choose reply targets after session scope is known, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, by source inspection. A Feishu group reply whose text still contains a bot placeholder but whose non-empty `mentions` array omits `botOpenId` returns `mentionedBot=false`, then mention-required group ingress returns before dispatch.

## Solution Assessment

No. The fix belongs in the Feishu plugin, but this PR's any-orphan heuristic and parse-time root clearing are broader than the narrow maintainable repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve topic roots before clearing reply routing:** `extensions/feishu/src/bot.ts:823`
  - body: Clearing `rootId` in `parseFeishuMessageEvent` happens before `groupSessionScope` is known. In `group_topic` and `group_topic_sender` modes, `rootId` is the topic key, so addressed replies with `root_id` but no `thread_id` can fall back to the chat-level session and merge unrelated topic context.
  - confidence: 0.91
- **[P2] Require bot-specific evidence for orphan placeholders:** `extensions/feishu/src/bot.ts:484`
  - body: The fallback marks any unmatched `@_..._N` token in reply text as a bot mention. A literal placeholder or dropped non-bot mention can bypass `requireMention` and trigger root clearing without evidence that the bot was addressed.
  - confidence: 0.88
