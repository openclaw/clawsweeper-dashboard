## Summary

Keep this open: current main still has a source-visible gap where resolved model params can exist on the runtime model, but the embedded Pi extra-param wrapper only forwards a small options subset and does not patch OpenAI-compatible payload keys such as max_completion_tokens.

## What This Changes

The issue requests that embedded Pi OpenAI-compatible runs honor resolved per-model params, especially max_completion_tokens, instead of keeping the upstream 32000 default.

## Best Possible Solution

Add a narrow generic OpenAI-compatible payload-param bridge in the embedded extra-params path so resolved model params and explicit runtime overrides consistently reach the final provider request without bypassing existing provider plugin hooks.

## Reproduction Assessment

Yes, from source inspection: a focused payload-capture test can configure an openai-completions model with model.params.max_completion_tokens and show the embedded wrapper leaves the upstream 32000 default unless maxTokens is separately mapped. I did not run a live QQBot/DashScope reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow runtime-param propagation change in extra-params/payload wrapping, not a QQBot-specific workaround or a new config option. Preserve explicit runtime overrides and existing provider-owned wrappers.
