## Summary

Keep open: the report describes a current, narrow runtime/config propagation bug. Current code documents and accepts per-model `params`, but the embedded stream wrapper only turns camelCase `maxTokens` into `SimpleStreamOptions.maxTokens`, while the OpenAI-completions transport builds `max_completion_tokens` from `options.maxTokens` or `model.maxTokens`, not from `model.params.max_completion_tokens`.

## What This Changes

Ensure resolved OpenAI-compatible model params, especially `params.max_completion_tokens`, survive into embedded Pi agent OpenAI-completions payloads.

## Best Possible Solution

Add one canonical budget normalization path that maps resolved `max_completion_tokens`/`max_tokens` aliases into the stream option or transport payload before embedded and direct OpenAI-completions requests are sent.

## Reproduction Assessment

Yes, source inspection gives a high-confidence code-path reproduction for the missing propagation: configure per-model `params.max_completion_tokens` without camelCase `maxTokens`, then the embedded extra-param wrapper will not set `options.maxTokens`. I did not run the live QQBot/DashScope provider path.

## Solution Assessment

Yes, the best fix is a narrow normalization change in the existing extra-params/OpenAI transport path, plus regression coverage; adding a QQBot-specific workaround would be the wrong boundary.
