## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution, but not the requested agent-callable `secret.get`/`secret.has`/`secret.list` broker surface; the issue discussion correctly identifies that the remaining blocker is a security/product choice between a trusted bundled plugin and a new external plugin API contract.

## What This Changes

Add a first-class secret broker plugin/tool surface that can resolve named secrets from Bitwarden, Keychain, 1Password, or Vault without persisting returned secret values.

## Best Possible Solution

Define a trusted secret capability contract with metadata-only `has`/`list`, guarded `get`, redacted or non-persistent results, and owner-scoped session gating, then implement either a bundled backend or an external plugin API path based on maintainer direction.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is high-confidence by source inspection: current main exposes configured SecretRef resolution but no agent-callable `secret.*` broker tools or external pre-model redaction contract.

## Solution Assessment

Unclear. The proposed API is plausible, but the safest maintainable solution depends on choosing the trust boundary before implementation rather than adding ordinary plugin tools that return credentials.
