## Summary

Keep this open. Current main has a solid SecretRef resolution foundation, including exec-backed 1Password and Vault examples, but it still lacks lazy agent-callable secret tools and the trusted redaction/session-gating API needed to expose live credential material safely from an external plugin. The reporter’s follow-up correctly frames this as a maintainer product/security decision before implementation.

## What This Changes

Request a generic secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault with transcript redaction and session-scope guards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, a redacted or non-persistent `get` result shape honored before transcript/model replay, and a pre-offer/pre-execution owner-trusted session gate. Then implement the first backend either as a maintainer-approved bundled plugin or after adding the explicit external plugin trust API.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence by source inspection: SecretRefs resolve configured targets eagerly, while no lazy agent-callable `secret.*` tools or trusted external redaction/session-scope API exists.

## Solution Assessment

Unclear. The proposed surface is directionally useful, but the narrow maintainable fix depends on a maintainer product/security choice between bundled trust and a new plugin API trust contract before backend code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before redaction and session-scope trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:370`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool output, while the pre-model tool-result middleware seam is bundled-only, so a trusted redacted or non-persistent result contract is needed before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:85`
  - body: Group/channel allowlists decide who can trigger actions, not what context is safe to expose. Secret tools should be hidden or blocked before execution outside owner-trusted sessions.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1567`
  - body: Both trusted pre-tool policy and tool-result middleware registration are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the key safety controls unavailable.
  - confidence: 0.9
