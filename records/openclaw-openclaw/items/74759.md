## Summary

Keep this open. Current main has SecretRef-backed credential resolution, but it does not implement lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the current plugin trust boundaries make this a product/security design decision rather than a cleanup-close candidate. The reporter's follow-up usefully narrows the decision to a trusted bundled plugin versus a plugin API trust extension.

## What This Changes

Add a generic secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault, with transcript redaction and session-scope guards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, a `get` result shape that is redacted or excluded before transcript/model replay, and a pre-offer owner-trusted session gate. After that, implement either a small trusted bundled backend or an external-plugin API extension plus first backend, depending on maintainer direction.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence by source inspection: SecretRefs resolve configured targets eagerly, and no lazy `secret.*` tool or generic owner-session/redacted-result contract exists.

## Solution Assessment

Unclear. The proposed tool surface is useful, but it is not yet clear whether the best implementation is an in-tree trusted plugin or a plugin API trust extension that lets ClawHub backends ship safely.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes agent-callable tools that can return live credential material before the required redaction and session-trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce normal tool output, while pre-model result middleware is bundled-only, so the redacted or non-persistent result contract needs to be designed first.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group/channel allowlists decide who can trigger actions, not whether the whole context is safe for secret exposure. Secret tools need to be hidden or blocked before execution outside owner-trusted sessions.
  - confidence: 0.9
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1573`
  - body: Both trusted pre-tool policy and pre-model tool-result middleware are bundled-only today. Shipping this as an ordinary external plugin would leave important safety controls unavailable.
  - confidence: 0.91
