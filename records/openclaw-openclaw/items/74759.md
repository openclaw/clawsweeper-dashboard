## Summary

Keep this open. Current main has a solid configured SecretRef provider/runtime foundation, but it still does not expose lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the issue discussion now explicitly needs a maintainer choice between a trusted bundled plugin and a plugin API trust extension.

## What This Changes

Request to add a secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault backends with transcript redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including metadata-only `has`/`list`, redacted or non-persistent `get` results, and an owner/direct-session gate before the tool is offered or executed; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and the safe redaction/session-scope API for them are absent.

## Solution Assessment

Unclear. The requested plugin direction is useful, but external plugin implementation is not the best safe solution until the trusted result and session-scope contract exists; the maintainable path is either a bundled trusted implementation or a small plugin API extension first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before the trust, redaction, and session-scope contracts are settled.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools cannot register pre-model tool-result middleware, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.94
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group/channel allowlists are trigger controls, not a universal context redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions.
  - confidence: 0.9
- **[medium] Keep arbitrary lookup separate from target resolution:** `src/gateway/server-methods/secrets.ts:74`
  - body: The existing `secrets.resolve` path is constrained to known configured target ids. General named lookup needs its own policy and audit semantics rather than overloading the configured-target RPC.
  - confidence: 0.84
