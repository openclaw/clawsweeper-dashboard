## Summary

Keep open: current main has SecretRef resolution for configured credentials, but not the requested agent-callable secret broker; the remaining work is a security-sensitive trust-boundary decision, not a safe cleanup close or automated fix.

## What This Changes

Request to add an agent-callable secret broker with `secret.get`/`secret.has`/`secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault, with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract with metadata-only `has`/`list`, guarded `get`, non-persistent or redacted results, and owner-trusted session gating before deciding whether the backend ships bundled or through a new plugin API seam.

## Reproduction Assessment

Not applicable: this is a feature and security-design request, not a bug report. Source inspection on current main confirms the requested agent-callable `secret.*` broker surface is absent.

## Solution Assessment

Unclear. The proposed API shape is plausible, but the safest maintainable path depends on a maintainer decision between a trusted bundled broker and a plugin API extension for redacted results plus trusted session gating.
