## Summary

Keep this open. Current main has a mature SecretRef provider/snapshot system, but it still does not expose lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the required redaction plus session-scope controls are trusted surfaces that external plugins cannot currently own. The reporter’s follow-up correctly narrowed this to a maintainer product/security choice between a bundled trusted plugin and a plugin-API trust extension.

## What This Changes

The issue requests a first-class secret broker plugin exposing lazy `secret.get`, `secret.has`, and `secret.list` tool surfaces over local secret backends such as Bitwarden, macOS Keychain, 1Password, and Vault.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation: metadata-only `has`/`list`, redacted or non-persistent `get`, and a pre-offer/pre-execution owner/direct-session gate. Then implement the first backend through the maintainer-approved bundled path or after extending the plugin API to safely delegate that trust to external plugins.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested generic plugin shape is useful, but an external plugin is not currently the safest complete solution because the required pre-model result rewriting and trusted pre-tool policy seams are bundled-only; the safer path is a maintainer-approved bundled implementation or a plugin-API trust extension first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes an agent-visible tool that can intentionally return live credential material.

Concerns:

- **[high] Define redacted secret result semantics before `get`:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would return credentials. External plugin tools currently cannot own pre-model tool-result middleware, so OpenClaw needs a trusted redacted or non-persistent result contract before any backend returns secret values.
  - confidence: 0.94
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group and channel sessions are supported, and current allowlists are trigger controls rather than a universal context redaction boundary. Secret tools should be hidden or blocked before execution outside trusted owner/direct sessions.
  - confidence: 0.9
- **[medium] Decide bundled trust versus plugin API trust first:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policy registration are bundled-only today. Shipping this solely as an external plugin would leave the key safety controls unavailable unless the plugin API grows an explicit trust contract.
  - confidence: 0.91
