## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trust/redaction/session-scope contract needed before returning live secrets through tool output.

## What This Changes

Add a first-class secret-broker plugin/API surface over Bitwarden, Keychain, 1Password, and Vault with redacted secret retrieval and trusted session guards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, redacted or non-persistent `get` results before replay/persistence, and owner-trusted session gating; then implement the first backend after the maintainer chooses bundled trust or an external plugin API extension.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is reproducible by source inspection: current main has no `secret.*` agent tools or session-scope annotations, while `secrets.resolve` only resolves configured target IDs from the active SecretRef snapshot.

## Solution Assessment

Unclear. The proposed surface is useful, but the maintainable implementation depends on a maintainer product/security decision between an in-tree trusted plugin and a plugin API extension for external secret broker plugins.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes agent-callable tools that return live credential material before the redaction and session-trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:379`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce ordinary tool output, while agent tool-result middleware is bundled-only, so a trusted redacted or non-persistent result shape is needed before implementation.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group allowlists decide who can trigger actions, not whether all visible context is owner-trusted. Secret tools should be hidden or blocked before execution outside approved owner-trusted session kinds.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1606`
  - body: Both pre-tool trusted policy and result-redaction middleware are bundled-only today. Shipping this as an external plugin without a new trust contract would leave key safety controls unavailable.
  - confidence: 0.9
