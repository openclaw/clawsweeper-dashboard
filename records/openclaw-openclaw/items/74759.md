## Summary

Keep this open. Current main already has configured SecretRef resolution over env/file/exec providers, including 1Password/Vault-style exec examples, but it still does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools. The remaining gap is a product/security design decision: secrets returned through tools need a trusted redaction/non-persistence contract and an owner-trusted session gate before a backend implementation is safe.

## What This Changes

Add a secret broker plugin that exposes agent-callable `secret.get`, `secret.has`, and `secret.list` tools backed by Bitwarden, macOS Keychain, 1Password, or Vault.

## Best Possible Solution

Define the secure secret capability contract first: metadata-only `has`/`list`, a `get` result shape that is redacted or excluded before transcript/model replay, and a pre-offer session-scope gate for owner-trusted contexts. After that, either implement a trusted bundled first backend or add the minimal plugin API trust extension that lets ClawHub backends provide the same safety semantics.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: `secret.*` tools are absent, SecretRefs resolve configured targets eagerly, and the trusted redaction/policy seams remain bundled-only.

## Solution Assessment

Unclear. The proposed tool surface is useful, but the best maintainable solution depends on maintainer direction: either a trusted bundled plugin or a small plugin API extension for safe external secret backends.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes agent-callable tools that intentionally return credential material before the required redaction and trusted-session contracts exist.

Concerns:

- **[high] Define redacted secret output semantics before get:** `src/plugins/registry.ts:374`
  - body: A `secret.get` tool would return live credential values. External plugin tools currently produce ordinary tool output, while pre-model tool-result middleware is bundled-only, so a redacted or non-persistent result contract is needed before implementation.
  - confidence: 0.94
- **[high] Gate secret tools by owner-trusted session scope:** `docs/channels/groups.md:90`
  - body: Group allowlists decide who can trigger actions, not whether all prior or quoted context is safe for secret exposure. Secret tools should be hidden or blocked before execution outside trusted owner sessions.
  - confidence: 0.9
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1577`
  - body: Current registry gates keep trusted tool policies and tool-result middleware bundled-only. Shipping this as an external plugin without a new trust contract would leave the needed safety controls unavailable.
  - confidence: 0.91
