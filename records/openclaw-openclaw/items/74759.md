## Summary

Keep open: this is a security-sensitive feature/API design request. Current main has static, eager SecretRefs and a trusted bundled-only tool-result middleware seam, so an external plugin cannot safely implement redacted secret-returning tools without a new trust/redaction/session-scope contract.

## What This Changes

Request to add first-class secret broker tools for `secret.get`/`secret.has`/`secret.list` backed by Bitwarden, Keychain, 1Password, or Vault with transcript redaction and session-scope guards.

## Best Possible Solution

Keep the issue open for a maintainer/secops API decision; if accepted, land an additive redacted-result and session-scope trust contract first, then ship specific secret backends as plugins or ClawHub packages.

## Reproduction Assessment

Not applicable: this is a new feature/API request, not a broken existing behavior report. Source inspection gives a high-confidence current-main check that no secret broker tools exist and current SecretRefs are env/file/exec-only with eager activation semantics.

## Solution Assessment

Unclear for direct implementation. The requested direction is plausible, but the narrow maintainable path is a maintainer-approved redaction/trust/session-scope API before adding any backend or LLM-callable secret tools.
