## Summary

Keep open. Current main has a strong SecretRef provider and command-target `secrets.resolve` foundation, including 1Password/Vault exec examples, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools. The requested surface also depends on trusted pre-model redaction and pre-tool session-scope checks that external plugins cannot currently own, and the reporter's follow-up correctly frames the remaining choice as a maintainer product/security decision.

## What This Changes

Add a secret broker plugin/tool surface so agents can call `secret.get`, `secret.has`, and `secret.list` against Bitwarden, Keychain, 1Password, or Vault without persisting returned secret values and with session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including `has`/`list` metadata semantics, redacted or non-persistent `get` results, and an owner/direct-session gate before the tool is offered or executed; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe external-plugin redaction/session-scope API are absent.

## Solution Assessment

Unclear. A plain external plugin is not the best fix until OpenClaw has a trusted redacted-result and session-scope contract; the safer path is either a bundled trusted implementation or an additive plugin API extension approved by maintainers first.

## Security Review

Status: needs_attention

Summary: The proposed feature would intentionally return live credential material, so it needs a trusted redaction and session-scope design before implementation.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would return credential material. Current external plugin tools produce normal tool content, while pre-model tool-result rewriting is bundled-only, so OpenClaw needs a trusted redacted or non-persistent result contract first.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:85`
  - body: OpenClaw supports group and channel sessions where allowlists are trigger authorization, not a universal context redaction boundary. Secret tools should be hidden or blocked before execution outside trusted owner/direct sessions.
  - confidence: 0.88
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policies are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the primary safety controls unavailable to the plugin.
  - confidence: 0.9
