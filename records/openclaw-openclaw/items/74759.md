## Summary

Keep this open. Current main still has configured SecretRef resolution and related provider work, but not lazy agent-callable `secret.get` / `secret.has` / `secret.list` tools, and the discussion has converged on a real product/security decision: either a trusted bundled implementation or a plugin API extension that can safely handle redacted secret results and trusted session scoping.

## What This Changes

Add a secret broker plugin exposing agent-callable secret lookup over Bitwarden, Keychain, 1Password, and Vault, with redacted results and trusted-session gating.

## Best Possible Solution

Define a trusted secret capability contract first: metadata-only `has`/`list`, redacted or non-persistent `get` results, and an owner/direct-session gate that hides or blocks the tool before execution. Then implement the first backend through the maintainer-approved path, reusing any landed SecretRef provider seam where appropriate.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested plugin direction is attractive and matches the vision for optional capabilities, but it is not sufficient until core either provides a trusted plugin API extension or maintainers approve a bundled trusted implementation.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before a trusted redaction and session-scope contract exists.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce normal tool results, and the trusted pre-model result middleware path is bundled-only, so core needs a redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group and channel allowlists authorize triggering, but they are not a universal context redaction boundary. Secret tools should be hidden or blocked before execution in group/channel and otherwise untrusted sessions.
  - confidence: 0.88
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policies are bundled-only today. Shipping this as an ordinary external plugin would leave the most important safety controls unavailable to the plugin.
  - confidence: 0.9
