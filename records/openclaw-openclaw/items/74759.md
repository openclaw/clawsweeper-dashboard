## Summary

Keep open: current main has SecretRef resolution for configured credentials, but not the requested agent-callable `secret.get`/`secret.has` broker. The remaining work is a security-sensitive product/API decision around redaction, non-persistence, and trusted session scope.

## What This Changes

Request an agent-callable secret broker plugin/API for `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, and Vault with transcript redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract with metadata-only `has`/`list`, guarded `get`, redacted or non-persistent results, and host-enforced session gating before choosing a bundled plugin or plugin API extension.

## Reproduction Assessment

Not applicable: this is a feature and security-design request, not a broken existing behavior report. Current-main source inspection confirms the requested agent-callable `secret.*` broker surface is absent.

## Solution Assessment

Unclear. The proposed API shape is plausible, but the safest maintainable path depends on a maintainer/secops decision between a trusted bundled broker and a plugin API extension for redacted results plus trusted session gating.
