## Summary

Keep this open. Current main has a configured SecretRef provider/resolution foundation, but it does not expose lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the requested safety model still needs maintainer design because the needed pre-model result redaction and trusted pre-tool gates are bundled-only today.

## What This Changes

Request to add a generic secret broker plugin exposing agent-callable secret lookup over Bitwarden, macOS Keychain, 1Password, and Vault with transcript redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract first: `has` and `list` should expose metadata only, `get` should use a redacted or non-persistent result shape before any transcript/model replay, and tool availability/execution should be blocked outside owner-trusted sessions. Then implement the first backend either as a maintainer-approved bundled plugin or after adding the needed plugin API trust extension.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: SecretRefs resolve configured targets eagerly, while no lazy agent-callable `secret.*` tool or trusted external redaction/session-scope API exists.

## Solution Assessment

Unclear. The proposed plugin surface is directionally useful, but the safest maintainable solution requires deciding the trusted bundled-plugin path versus a generic plugin API extension before backend code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before a trusted result-redaction and session-scope contract exists.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:370`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool output, while pre-model tool-result middleware is bundled-only, so a trusted redacted or non-persistent result contract is needed before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group/channel allowlists are trigger controls, not a universal context or redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions.
  - confidence: 0.88
- **[medium] Decide bundled trust versus plugin API trust first:** `src/plugins/registry.ts:1567`
  - body: The current trusted pre-tool policy and tool-result middleware seams are bundled-only. Shipping this as an external plugin without a new trust contract would leave the key safety controls unavailable to the plugin.
  - confidence: 0.9
