## Summary

Keep this issue open. Current main has SecretRef-backed configured credential resolution, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the discussion correctly leaves the remaining direction to a maintainer product/security decision.

## What This Changes

Request a first-class secret broker capability that exposes agent-callable secret lookup tools over Bitwarden, macOS Keychain, 1Password, or Vault with redaction and session-scope safeguards.

## Best Possible Solution

Define the trusted secret capability contract first, including metadata-only `has`/`list`, redacted or non-persistent `get` results, and owner-trusted session gating, then implement the first backend in the maintainer-chosen bundled or plugin-API path.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is verifiable by source inspection: current main has configured SecretRef resolution but no lazy `secret.*` agent tool surface or owner-trusted session-scope annotation.

## Solution Assessment

Unclear. The proposed surface is useful, but the maintainable solution depends on a maintainer decision between an in-tree trusted plugin and a small plugin API extension that external ClawHub backends can use safely.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it would expose live credential material through agent-callable tools before a public redaction and session-trust contract exists for that path.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:379`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce ordinary tool output, while agent tool-result middleware is bundled-only, so a trusted redacted or non-persistent result shape is needed before implementation.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group allowlists decide who can trigger actions, not whether all visible context is owner-trusted. Secret tools should be hidden or blocked before execution outside approved owner-trusted session kinds.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1606`
  - body: Both pre-tool trusted policy and result-redaction middleware are bundled-only today. Shipping this as an external plugin without a new trust contract would leave key safety controls unavailable.
  - confidence: 0.9
