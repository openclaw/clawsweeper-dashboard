## Summary

Keep this open. Current main has configured SecretRef resolution, but not agent-callable secret broker tools, and the requested redaction plus trusted-session gating need a maintainer security/product decision before implementation.

## What This Changes

Requests an agent-callable secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over local secret-manager backends with transcript redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract first: metadata-only `has`/`list`, redacted or non-persistent `get` results, and owner-trusted session gating, then implement either a bundled backend or an external plugin API extension based on maintainer direction.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is high-confidence by source inspection: current main exposes configured SecretRef resolution but no agent-callable `secret.*` broker tools.

## Solution Assessment

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because `secret.get` would return live credential material through agent tools before the redaction and trusted-session contracts exist.

Concerns:

- **[high] Define redacted secret-result semantics before `get`:** `src/plugins/registry.ts:396`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce ordinary model-visible output, and the trusted pre-model result-rewrite seam is bundled-only, so a redacted or non-persistent result contract is needed first.
  - confidence: 0.93
- **[high] Gate secret tools before model exposure:** `src/plugins/tool-types.ts:41`
  - body: The requested scope guard should hide or block secret tools outside approved owner contexts before execution. Plugin tool options currently only declare names and optionality, so a declarative session-scope policy or bundled trusted gate needs design.
  - confidence: 0.88
- **[medium] Choose bundled trust or plugin API trust first:** `VISION.md:54`
  - body: VISION.md favors plugins and plugin API extensions for optional capabilities, but the needed trusted result and policy seams are not available to external plugins today.
  - confidence: 0.9
