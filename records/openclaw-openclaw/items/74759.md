## Summary

Keep open: current main has configured SecretRef resolution and related provider work in flight, but it still does not provide an agent-callable `secret.get`/`secret.has` broker with redacted, non-persistent results and trusted-session gating. The reporter's follow-up explicitly accepts that this is a product/security direction call between a trusted bundled plugin and a plugin API extension.

## What This Changes

Request a first-class secret broker plugin/API so agents can call `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, or Vault without exposing secrets in transcripts or untrusted sessions.

## Best Possible Solution

Define a trusted secret capability contract with metadata-only `has`/`list`, guarded `get`, redacted or non-persistent results, and host-enforced session gating before choosing a bundled plugin or external plugin API path.

## Reproduction Assessment

Not applicable: this is a feature and security-design request, not a broken existing behavior report. Current-main source inspection confirms the requested agent-callable `secret.*` broker surface is absent.

## Solution Assessment

Unclear. The proposed API shape is plausible, but the safest maintainable path depends on a maintainer/secops decision between a trusted bundled plugin and a plugin API extension for redacted results plus trusted session gating.
