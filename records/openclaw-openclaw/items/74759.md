## Summary

Keep this open. Current main has a SecretRef provider and resolution foundation, but it still does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the external-plugin path lacks the trusted pre-model redaction and pre-tool session-scope controls needed for returning credential material safely. The reporter's follow-up correctly narrows this to a maintainer product/security decision rather than a conservative cleanup close.

## What This Changes

Add a generic secret broker plugin exposing agent-callable `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault with transcript redaction and trusted-session guards.

## Best Possible Solution

Define the trusted secret capability contract first: `has` and `list` should expose metadata only, `get` should produce a redacted or non-persistent result before any transcript/model replay, and the tool should be hidden or blocked outside owner-trusted sessions. After that decision, implement the first backend either as a maintainer-approved bundled plugin or through a generic plugin API extension that external ClawHub plugins can safely use.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: SecretRefs resolve configured targets eagerly, while no lazy agent-callable `secret.*` tool or external redacted-result/session-scope API exists.

## Solution Assessment

Unclear. The proposed surface is directionally useful, but the safest maintainable solution requires choosing the trusted bundled-plugin path or a generic plugin API trust extension before backend code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before a trusted redaction and session-scope contract exists.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:370`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool output, while pre-model tool-result middleware is bundled-only, so a trusted redacted or non-persistent result contract is needed before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:90`
  - body: Group/channel allowlists are trigger controls, not a universal context or redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions.
  - confidence: 0.9
- **[medium] Decide bundled trust versus plugin API trust first:** `src/plugins/registry.ts:1567`
  - body: The current trusted pre-tool policy and tool-result middleware seams are bundled-only. Shipping this as an external plugin without a new trust contract would leave key safety controls unavailable to the plugin.
  - confidence: 0.9
