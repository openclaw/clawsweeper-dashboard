## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution and exec-provider examples for 1Password/Vault, but it does not provide agent-callable `secret.get`/`secret.has`/`secret.list` tools, and external plugins still lack the trusted pre-model redaction/session-scope seams needed to safely return live secret values.

## What This Changes

Request a generic secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault with redaction and session-scope guards.

## Best Possible Solution

Define the trusted secret-result contract and owner-trusted session gate first, then implement either a bundled secret broker or the minimal plugin API extension that lets external ClawHub backends enforce the same guarantees.

## Reproduction Assessment

Not applicable as a feature/security-design request. The gap is reproducible by source inspection: exact-name search finds no requested agent tools, the gateway exposes only `secrets.reload`/`secrets.resolve`, and `secrets.resolve` is scoped to known target IDs.

## Solution Assessment

Unclear. The proposed tool family is reasonable, but the maintainable fix depends on a maintainer decision between bundled trust and extending the external plugin API for redacted secret results and session-scope gating.

## Security Review

Status: needs_attention

Summary: Needs attention because the request would intentionally expose live secret material to agent tool flow before a trusted redaction and owner-session contract exists.

Concerns:

- **[high] Define pre-model redaction before returning secrets:** `src/plugins/registry.ts:379`
  - body: A broker `get` action would return credential material. External plugin tools currently return ordinary model-visible output, while agent tool-result middleware is bundled-only, so a trusted redacted or non-persistent result shape is needed first.
  - confidence: 0.93
- **[high] Gate secret tools by owner-trusted session scope:** `docs/channels/groups.md:83`
  - body: Group allowlists decide who may trigger actions, not whether all visible or historical context is trusted. Secret tools should be unavailable or blocked before execution outside approved owner-trusted sessions.
  - confidence: 0.89
- **[medium] Choose bundled trust or API trust before backend work:** `src/plugins/registry.ts:1606`
  - body: Both trusted pre-tool policy and pre-model tool-result rewriting are bundled-only today. Shipping Bitwarden, Keychain, 1Password, or Vault backends externally without a new trust contract would leave key safety controls unavailable.
  - confidence: 0.9
