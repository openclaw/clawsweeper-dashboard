## Summary

Keep this open. Current main has SecretRef-backed credential resolution and trusted bundled-plugin seams, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the safety contract needed before returning live credentials through tool output.

## What This Changes

Request a first-class secret broker plugin exposing lazy `secret.get`/`secret.has`/`secret.list` tools over Bitwarden, Keychain, 1Password, and Vault with transcript redaction and session-scope guards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, a redacted or non-persistent `get` result shape before transcript/model replay, and a pre-offer owner-trusted session gate. Then implement the first backend only after the maintainer chooses bundled trust or a plugin API trust extension.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: SecretRefs resolve configured targets eagerly, and there are no lazy `secret.*` tools or external-plugin trust seams for pre-model redaction plus session-scope gating.

## Solution Assessment

Unclear. The proposed surface is useful, but the narrow maintainable solution depends on a maintainer decision about trusted bundled implementation versus a new plugin API trust contract before backend code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes agent-callable tools that return live credential material before the required redaction and session trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce normal tool output, while pre-model tool-result middleware is bundled-only, so a trusted redacted or non-persistent result contract is needed first.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: Group/channel allowlists decide who can trigger actions, not what context is safe for secret exposure. Secret tools should be hidden or blocked before execution outside owner-trusted sessions.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1573`
  - body: Both trusted pre-tool policy and tool-result middleware registration are bundled-only today. Shipping this as an external plugin without a new trust contract would leave key safety controls unavailable.
  - confidence: 0.9
