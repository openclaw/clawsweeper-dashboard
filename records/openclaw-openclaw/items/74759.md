## Summary

Keep this open. Current main has an eager SecretRef/runtime snapshot system and command-target `secrets.resolve`, but it still lacks the requested lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the external plugin trust seams needed to return live secret material safely.

## What This Changes

The issue requests a first-class secret broker plugin/tool surface for lazy secret lookup over Bitwarden, macOS Keychain, 1Password, and Vault with redaction and session-scope safeguards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, a `get` result shape that is redacted or non-persistent before transcript/model replay, and an owner-trusted session gate before the tool is offered or executed. Then implement the first backend either as a maintainer-approved bundled plugin or after adding the explicit external plugin trust API.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: SecretRefs resolve eagerly through configured targets, and no lazy agent-callable `secret.*` tools or external trusted redaction/session-scope API exists.

## Solution Assessment

Unclear. The proposed surface is useful, but implementing backends first is not the narrowest maintainable fix until a maintainer chooses bundled trust versus a plugin API trust extension.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credentials through agent tools before the required redaction and session-scope trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:370`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce ordinary tool output, while the trusted tool-result middleware seam is bundled-only, so a redacted or non-persistent result contract is needed first.
  - confidence: 0.94
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:87`
  - body: Secret tools need a hard pre-offer or pre-execution gate for owner-trusted sessions. Existing group/channel allowlists decide who can trigger actions, not what historical or quoted context is safe to expose.
  - confidence: 0.9
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1569`
  - body: Both trusted pre-tool policy registration and tool-result middleware are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the key safety controls unavailable.
  - confidence: 0.91
