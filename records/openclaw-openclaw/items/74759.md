## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution, including exec examples for 1Password and Vault, but it still lacks the requested lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trusted redaction plus session-scope policy needed before live secret values can safely flow through agent tools.

## What This Changes

Request a first-class secret broker plugin or plugin API surface that lets agents lazily call `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, or Vault backends.

## Best Possible Solution

Define an explicit trusted secret capability contract first: metadata-only `has`/`list`, redacted or non-persistent `get` results, and a pre-tool owner/direct-session gate. Then implement the first backend either as a maintainer-approved bundled plugin or after a safe plugin API extension makes the same guarantees available to external ClawHub plugins.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but the requested lazy agent-callable `secret.*` tools and their trusted safety contract do not.

## Solution Assessment

Unclear. A generic external plugin is not currently the best complete solution because external plugins cannot register the needed pre-model tool-result middleware or trusted pre-tool policy; the safer path is maintainer approval for a bundled trusted implementation or an additive plugin API trust design first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes exposing live credential material to agent tools before OpenClaw has a dedicated secret-result and session-scope contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material, but current external plugin tools produce normal tool content while pre-model result rewriting is bundled-only. Core needs a trusted redacted or non-persistent result shape before any backend returns values.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:85`
  - body: OpenClaw group/channel allowlists are trigger controls, not a universal context redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and other untrusted sessions.
  - confidence: 0.88
- **[medium] Choose bundled trust or API trust before implementation:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policy are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the key safety controls unavailable.
  - confidence: 0.9
