## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution, but it still lacks first-class agent-callable secret tools and the request needs a maintainer decision on trusted redaction plus owner-session gating before implementation.

## What This Changes

Add a generic secret broker plugin/API surface for `secret.get`, `secret.has`, and `secret.list` across backends such as Bitwarden, macOS Keychain, 1Password, and Vault.

## Best Possible Solution

Define the trusted secret capability contract first, including metadata-only `has`/`list`, redacted or non-persistent `get` results, and owner-trusted session gating; then implement the first backend in the chosen bundled or plugin-API path.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is reproducible by source inspection: current main has no `secret.*` agent tools, while `secrets.resolve` only resolves known configured target IDs from the active SecretRef snapshot.

## Solution Assessment

Unclear. The proposed surface is useful, but the maintainable implementation depends on a maintainer decision between an in-tree trusted plugin and extending the plugin API so external secret broker plugins can enforce redaction and scope safely.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it proposes agent-callable tools that can return live credential material.

Concerns:

- **[high] Define redacted secret-result semantics before get:** `src/plugins/registry.ts:396`
  - body: A `secret.get` tool would intentionally return credential material. External plugin tools currently produce ordinary tool output, while result middleware that can rewrite output before the model sees it is bundled-only.
  - confidence: 0.93
- **[high] Gate secret tools by owner-trusted session scope:** `docs/channels/groups.md:90`
  - body: Group allowlists decide who can trigger actions, not whether all visible context is owner-trusted. Secret tools should be hidden or blocked before execution outside approved owner-trusted session kinds.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `docs/plugins/sdk-agent-harness.md:170`
  - body: Both pre-tool trusted policy and result-redaction middleware are bundled-only today, so shipping this as an external plugin without a new trust contract would leave key safety controls unavailable.
  - confidence: 0.9
