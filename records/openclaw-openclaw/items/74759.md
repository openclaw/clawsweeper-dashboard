## Summary

Keep this open. Current main supports configured SecretRefs and exec-backed providers, but not the requested agent-callable secret broker tools; the remaining decision is the security/product boundary between a trusted bundled plugin and a new external plugin API contract.

## What This Changes

Add first-class `secret.get`, `secret.has`, and `secret.list` agent tools backed by Bitwarden, macOS Keychain, 1Password, or Vault with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract with metadata-only `has`/`list`, guarded `get`, redacted or non-persistent results, and owner-scoped session gating, then implement the selected bundled or plugin-API path.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is high-confidence by source inspection: current main has SecretRef resolution but no agent-callable secret broker tools or external pre-model redaction contract.

## Solution Assessment

Unclear. The proposed API is plausible, but the best maintainable fix depends on choosing the trust boundary before implementation rather than adding a normal plugin tool that returns credentials.

Status: needs_attention

  - body: The requested scope guard needs to hide or block secret tools before execution in group or untrusted sessions; the existing generic owner-only tool policy is not a complete session-kind/participant trust contract.

  - body: OpenClaw's vision prefers optional capability as plugins, but the needed pre-model secret redaction and trust-policy seams are currently trusted/bundled-only, so this needs maintainer direction before coding.
