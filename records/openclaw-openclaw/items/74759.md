## Summary

Keep this open: current main supports configured SecretRef resolution, including external channel secret contracts, but it still lacks the requested agent-callable `secret.get`/`secret.has`/`secret.list` broker and needs a maintainer trust-boundary decision before implementation.

## What This Changes

Request to add a first-class secret broker plugin/API so agents can check and resolve named secrets from Bitwarden, macOS Keychain, 1Password, Vault, or similar backends without persisting secret values.

## Best Possible Solution

Define a trusted secret capability contract first, with metadata-only `has`/`list`, tightly gated `get`, non-persistent or redacted results, and owner-trusted session scope, then choose bundled backend code or a plugin API extension.

## Reproduction Assessment

Not applicable: this is a feature and security-design request, not a bug report. Source and docs inspection give high confidence that current main has SecretRef resolution but no agent-callable secret broker surface.

## Solution Assessment

Unclear. The proposed API shape is plausible, but the safest maintainable solution depends on choosing the trust boundary and redaction/session-scope contracts before implementation.
