## Summary

Keep open: current main has SecretRef provider resolution for configured credentials, including exec examples for 1Password and Vault, but not an agent-callable `secret.*` broker surface; safe implementation depends on a product/security choice because pre-model redaction and trusted tool policies are bundled-only today.

## What This Changes

Add a first-class secret broker plugin/API surface with agent-callable `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, or Vault with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract with guarded `get`, metadata-only `has`/`list`, redacted or non-persistent result semantics, and owner-trusted session gating before choosing bundled backend code or an external plugin API path.

## Reproduction Assessment

Not applicable: this is a feature and security-design request, not a bug report. Source inspection gives high confidence that current main lacks the requested agent-callable `secret.*` broker surface.

## Solution Assessment

Unclear: the requested API shape is plausible, but the best implementation depends on the maintainer trust-boundary decision between a bundled trusted plugin and a new external plugin API contract.
