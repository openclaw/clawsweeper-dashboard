## Summary

Keep this open. Current main has a configured SecretRef resolution foundation, including exec-backed 1Password/Vault examples, but it still does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools. The requested implementation is blocked on a maintainer product/security choice because the required pre-model result redaction and trusted pre-tool gating seams are currently bundled-only, and the reporter’s follow-up explicitly asks for that direction before coding.

## What This Changes

Add a first-class secret broker plugin/API exposing agent-callable `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, or Vault with redaction and session-scope guards.

## Best Possible Solution

Define the trusted secret capability contract first: metadata-only `has`/`list`, a redacted or non-persistent `get` result shape before transcript/model replay, and a pre-offer/pre-execution owner-trusted session gate. Then implement the first backend either as a maintainer-approved bundled plugin or after adding the explicit plugin API trust extension.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is reproducible by source inspection: SecretRefs resolve configured targets eagerly, while no lazy agent-callable `secret.*` tools or trusted external redaction/session-scope API exists.

## Solution Assessment

Unclear. The proposed surface is directionally useful, but the narrow maintainable solution depends on a maintainer decision about bundled trust versus a new plugin API trust contract before backend code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before the required redaction and session-scope trust contracts exist.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:370`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool output, while the pre-model tool-result middleware seam is bundled-only, so a trusted redacted or non-persistent result contract is needed before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:87`
  - body: Group/channel allowlists decide who can trigger actions, not what context is safe to expose. Secret tools should be hidden or blocked before execution outside owner-trusted sessions.
  - confidence: 0.89
- **[medium] Choose bundled trust or plugin API trust first:** `src/plugins/registry.ts:1567`
  - body: Both trusted pre-tool policy and tool-result middleware registration are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the key safety controls unavailable.
  - confidence: 0.9
