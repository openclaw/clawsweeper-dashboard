## Summary

Keep this open. Current main has a strong configured SecretRef foundation, including exec-backed providers, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trusted result/session-scope contracts needed before live credential material can be safely exposed to an agent. The follow-up discussion correctly narrows this to a maintainer product/security decision between a trusted bundled plugin and a plugin-API trust extension.

## What This Changes

Request adding a secret broker plugin with agent-callable `secret.get`/`secret.has`/`secret.list` tools backed by Bitwarden, macOS Keychain, 1Password, or Vault, with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including metadata-only `has`/`list`, redacted or non-persistent `get` results, and a pre-offer/pre-execution owner/direct-session gate; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature/security-design request. The current-main gap is high-confidence from source inspection: configured SecretRefs and target-bound `secrets.resolve` exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API do not.

## Solution Assessment

Unclear. The requested generic plugin direction is not sufficient by itself because external plugins cannot currently register the trusted pre-model result middleware or trusted pre-tool policy needed for live secrets; the safer path is a bundled trusted implementation or a plugin-API trust extension first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before OpenClaw has a trusted result and session-scope contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool content, while pre-model tool-result rewriting is bundled-only, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: OpenClaw supports group and channel sessions where allowlists are trigger authorization, not a universal context redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions.
  - confidence: 0.88
- **[medium] Choose bundled trust or API trust first:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policies are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the most important safety controls unavailable to the plugin.
  - confidence: 0.9
