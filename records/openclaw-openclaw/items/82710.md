## Summary

Keep open: current main still provides generic model/auth probes plus a separate Codex status command, but it does not expose the requested combined Codex viability report. The closed unmerged implementation attempt at https://github.com/openclaw/openclaw/pull/82996 is useful context, and the latest live incident comment adds evidence that operators still need this diagnostic contract.

## What This Changes

Requests extending `openclaw models status --probe` or a Codex-specific status subcommand to report Codex OAuth, profile order, app-server startup, trivial-turn, last-failure, and fallback viability.

## Best Possible Solution

Define and land a Codex-aware probe contract that keeps Codex-specific checks in the plugin, reports profile/order/refresh/app-server/trivial-turn/fallback state through the existing status surface, and redacts sensitive auth details.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main shows the requested Codex viability fields are absent, and the 2026-05-18 live incident comment demonstrates why the missing diagnostic matters.

## Solution Assessment

Yes directionally, but the best implementation needs maintainer agreement on the public status/probe contract and the cost of running Codex app-server checks. The maintainable path is a plugin-owned Codex diagnostics hook surfaced through existing model status output, not ad hoc duplicated status logic.
