## Summary

Keep open: current main has generic live auth probes and a separate Codex app-server status command, but it still lacks the requested combined Codex viability report for refresh, selected profile/order, app-server startup, trivial-turn success, last runtime failure, and fallback usability. The related merged and open work covers auth-order/status-label/refresh classification pieces, not this product-facing diagnostic contract.

## What This Changes

Request to extend `openclaw models status --probe` or add a Codex-specific probe so operators can distinguish OAuth metadata health from real Codex app-server turn viability.

## Best Possible Solution

Define a Codex-aware diagnostic contract that reuses the existing auth-probe and Codex harness status surfaces, reports effective profile/order, refresh/app-server/trivial-turn/fallback state, and redacts sensitive auth details.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows the current main behavior is generic auth probing plus separate `/codex status`, and it does not expose the requested Codex app-server/fallback viability fields.

## Solution Assessment

Yes directionally, but the best implementation needs maintainer agreement on the output contract and on how far a status command should go in starting Codex app-server turns. A maintainable path would extend the existing probe/status schemas and Codex harness helpers instead of adding an ad hoc parallel report.
