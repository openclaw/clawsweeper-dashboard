## Summary

Keep open. Current main still formats bundled `session-memory` artifacts from the host/process timezone rather than `agents.defaults.userTimezone`, while related open PRs touch the same behavior and need maintainer coordination before this issue can be closed.

## What This Changes

The issue requests that generated memory filenames and `# Session` headers use `agents.defaults.userTimezone` or a dedicated memory-flush timezone instead of UTC/host time.

## Best Possible Solution

Resolve `session-memory` timestamp formatting through existing `resolveUserTimezone` semantics, update docs/tests together, and coordinate with the open human-day/session-memory PRs so one consistent behavior lands.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: use timestamp `2026-04-09T00:44:38.000Z`, configure `agents.defaults.userTimezone: "Asia/Jakarta"`, and run with a different host `TZ`; the current formatter follows host/process timezone instead of the configured zone. I did not run a writing test because this review is read-only.

## Solution Assessment

Yes, reusing the existing `resolveUserTimezone` behavior in `session-memory` is the narrow maintainable direction. The remaining decision is whether maintainers want that extracted as a focused fix or landed through an active broader human-day/session-memory PR.
