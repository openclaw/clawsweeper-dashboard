## Summary

Keep open. Current main still formats bundled `session-memory` filenames, fallback slugs, and headers from the host/process timezone rather than `agents.defaults.userTimezone`, while overlapping open PRs already touch the same behavior and need maintainer coordination.

## What This Changes

Request memory file timestamps and names, especially `session-memory` artifacts, to use the configured user timezone instead of UTC or host-local time.

## Best Possible Solution

Resolve `session-memory` timestamp formatting through existing `resolveUserTimezone` semantics, update docs/tests together, and coordinate with the active human-day/session-memory PRs so one consistent behavior lands.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agents.defaults.userTimezone: "Asia/Jakarta"`, run with a different host `TZ`, trigger `session-memory`, and the current formatter will still follow the host/process timezone.

## Solution Assessment

Yes. Reusing existing `resolveUserTimezone` in `session-memory` is the narrow maintainable direction; the remaining decision is whether maintainers want that extracted as a focused fix or landed through the broader active human-day PR.
