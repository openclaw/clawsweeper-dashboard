## Summary

Keep open. Current main and v2026.6.1 only partially cover the request: pre-compaction memory flush date paths use `agents.defaults.userTimezone`, but bundled `session-memory` filenames and `# Session` headers still use the host/process timezone, and both related implementation PRs are closed unmerged.

## What This Changes

The issue asks for memory file names and header timestamps to follow `agents.defaults.userTimezone` or an explicit memory timezone setting instead of UTC or host-local time.

## Best Possible Solution

Decide the intended memory timestamp contract, then either update `session-memory` to use the existing `userTimezone` helpers with docs/tests or explicitly keep host-local behavior documented.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: set `agents.defaults.userTimezone` to a timezone different from the host `TZ`, trigger `session-memory`, and the formatter still uses the host/process timezone path. I did not run a write-producing repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product contract. Reusing the existing `resolveUserTimezone`/date-stamp helpers is the narrow likely fix if `session-memory` should follow user timezone, while adding `memoryFlushTimezone` would need stronger config-surface justification.
