- `impact:session-state`: The issue affects session memory artifact naming and timestamp context used to correlate saved session state with conversation time.

## Summary

Keep open: current main and v2026.6.10 still make bundled `session-memory` filenames and headers follow host/process timezone instead of `agents.defaults.userTimezone`; the auto-flush memory-core path now uses `userTimezone`, so the remaining issue is the split contract for session-memory artifacts.

## What This Changes

The issue asks for memory-file filenames and header timestamps to use the configured user timezone instead of UTC or host-local time.

## Best Possible Solution

Define one memory artifact timezone contract, preferably reusing existing `agents.defaults.userTimezone` without adding a new config key, then update the bundled `session-memory` handler, focused regression tests, and public hooks/timezone docs together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `agents.defaults.userTimezone` differently from host `TZ`, trigger bundled `session-memory`, and the handler still formats the filename/header from host/process timezone rather than config; no filesystem repro was run because this review is read-only.

## Solution Assessment

Unclear until maintainers confirm the product contract. Reusing the existing configured-timezone helper in `session-memory` is likely the narrowest technical path if accepted, but it changes shipped and documented host-local artifact semantics.
