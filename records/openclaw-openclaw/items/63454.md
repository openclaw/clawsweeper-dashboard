## Summary

Keep open. Current main still formats bundled `session-memory` filenames, fallback HHMM slugs, and `# Session` headers from the host/process timezone path, while `agents.defaults.userTimezone` is only resolved in other time-aware paths such as prompt/current-time and memory-core flush planning.

## What This Changes

Make memory files produced by auto-flush/session-memory use the configured user timezone for filename dates and session header timestamps instead of always using UTC or host-local time.

## Best Possible Solution

Update the bundled `session-memory` hook to resolve `agents.defaults.userTimezone` with the existing timezone helper, use that resolved IANA timezone for filename date, fallback HHMM slug, and `# Session` header, keep host timezone fallback behavior, and document the new behavior in the hook docs.

## Reproduction Assessment

Yes. A focused reproduction can create a hook event with fixed timestamp `2026-04-09T00:44:38.000Z`, config `agents.defaults.userTimezone: "Asia/Jakarta"`, and no matching `TZ`; current source shows the output still follows host/process timezone because the formatting helper never reads the config.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse `resolveUserTimezone`/existing current-time semantics in `session-memory` instead of adding a new config key or changing unrelated memory-core flush behavior, which is already user-timezone-aware.
