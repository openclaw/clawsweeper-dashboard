## Summary

Keep open: latest main and v2026.6.8 still make bundled `session-memory` filenames and headers follow host/env local time, not `agents.defaults.userTimezone`; the narrow fix remains product/compatibility-sensitive because docs currently describe host-local memory artifact dates.

## What This Changes

The issue requests that session memory artifact filenames and headers use the configured user timezone instead of UTC or host-local time.

## Best Possible Solution

Define one memory artifact timezone contract, preferably reusing `agents.defaults.userTimezone` without adding another config key, then update the session-memory handler, focused tests, and docs to match that contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `agents.defaults.userTimezone` differently from host `TZ`, trigger bundled `session-memory`, and the handler still formats the filename/header from host/env timezone rather than config.

## Solution Assessment

Unclear until maintainers confirm the product contract. The narrow technical fix is likely to reuse the existing configured-timezone helper in `session-memory`, but that changes shipped and documented host-local artifact semantics.
