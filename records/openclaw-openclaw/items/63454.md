## Label Justifications

- `P2`: Timezone mismatch in persisted memory/session artifacts is a normal-priority user-facing consistency issue with limited blast radius.
- `impact:session-state`: The issue concerns persisted memory/session artifact names and timestamps drifting from the configured user timezone.

## Summary

Keep open. Current main and the latest release only partially cover this: pre-compaction memory flush planning uses `agents.defaults.userTimezone`, but the bundled `session-memory` hook still formats filenames, fallback slugs, and `# Session` headers from host/process local time, and the public docs describe that host-local behavior.

## What This Changes

Request memory artifact filenames and `# Session` headers to honor `agents.defaults.userTimezone` or a dedicated memory timezone instead of UTC/host-local time.

## Best Possible Solution

Choose one canonical memory-artifact timezone policy, then align `session-memory` code, tests, and docs with the existing `memory-core` flush-plan behavior or explicitly document host-local session-memory as intentional.

## Reproduction Assessment

Yes by source inspection: the bundled `session-memory` hook formats the filename date, fallback HHMM slug, and `# Session` header through `formatLocalSessionTimestamp`, which reads host/process local time and not `cfg.agents.defaults.userTimezone`. I did not run a writing test because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy. The narrow implementation would be to reuse `resolveUserTimezone` for `session-memory`, but current docs still say host-local date and the broader prior fix paths were closed unmerged.
