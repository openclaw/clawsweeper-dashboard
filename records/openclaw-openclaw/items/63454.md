## Summary

Keep open: current main fixes the older UTC-vs-host-local session-memory problem and auto-flush planning already uses `userTimezone`, but the bundled `session-memory` hook still formats filenames and headers from host `TZ` instead of `agents.defaults.userTimezone`, so the central request remains unresolved.

## What This Changes

The issue asks for memory artifact filenames and headers to respect `agents.defaults.userTimezone`, or an explicit memory timezone setting, instead of using UTC or host-local time.

## Best Possible Solution

Define one memory artifact timezone contract, preferably reusing `agents.defaults.userTimezone` for `session-memory` without adding a new config key, then update the handler, focused tests, and docs to match that contract.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure `agents.defaults.userTimezone` to a zone different from the host `TZ`, trigger `session-memory`, and the handler still formats the filename/header from host `TZ`. I did not run a live `/new` flow because this review is read-only.

## Solution Assessment

Unclear until maintainers confirm the product contract. The narrow technical fix is likely to reuse `resolveUserTimezone` in `session-memory`, but the current docs scope `userTimezone` to system prompt context while the shipped host-local behavior is already in the latest release.
