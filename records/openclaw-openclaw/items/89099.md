## Label Justifications

- `P2`: This is a focused gateway/subagent timeout fix with real user impact, but the blast radius is limited to run-wait completion semantics.
- `merge-risk: 🚨 session-state`: The PR changes when pending timeout snapshots become terminal, which can prematurely complete or mis-state subagent run state before retry grace expires.

## Summary

Keep open: the linked bug is credible and the PR targets the right gateway wait path, but the patch should not merge as-is because it returns every pending timeout snapshot before retry grace, not only the hard-timeout case, and the external proof is source/test-only.

## What This Changes

The PR updates `agent.wait` fallback handling to return a pending timeout snapshot when the caller timeout fires before timeout retry grace, and adds focused regression coverage.

## Best Possible Solution

Keep the fallback, but gate it to hard-timeout snapshots and preserve null/retry-grace behavior for soft pending timeouts until later lifecycle evidence settles the run.

## Reproduction Assessment

Yes, source-reproducible: on current `main`, `waitForAgentJob()` only consults pending errors when the caller timeout fires, so a pending hard-timeout snapshot can resolve as `null`. I did not run tests because this review was read-only.

## Solution Assessment

No: the proposed location is right, but the implementation is too broad because the pending-timeout map also contains retry-grace soft timeouts. The safer fix is to return early only for hard-timeout snapshots and keep existing grace behavior for soft aborted ends.
