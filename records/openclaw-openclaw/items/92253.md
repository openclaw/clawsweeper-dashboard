Keep open: the owner boundary is plausible, but the latest head still turns an explicit empty Active Memory agent allowlist into broad activation and the supplied runtime proof does not show final-head, agent-level behavior.

## What This Changes

The PR changes Active Memory config normalization so omitted or empty `config.agents` targets all configured agents or `main`, and updates docs, manifest help, and regression tests.

## Best Possible Solution

Default only an absent `config.agents` field if maintainers approve that policy, preserve explicit `[]` as no targeted agents unless a migration decision says otherwise, and align tests/docs with final-head proof.

## Reproduction Assessment

Yes. Source inspection shows current `main` normalizes omitted or empty `config.agents` to `[]`, and both command and prompt-build gates reject agents outside that list; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. Central normalization is the right layer, but collapsing an explicit empty allowlist into broad activation is not required to fix omitted-field behavior and needs a compatibility decision.
