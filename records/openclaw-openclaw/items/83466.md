## Label Justifications

- `P1`: This is a broken agent/channel workflow on the default `openai/*` Codex app-server path where real inbound image content can be silently absent from vision turns.
- `impact:message-loss`: The issue is about inbound image attachment content being dropped from the model payload even though the text prompt still references the media.

## Summary

Keep open: current main still has a clear Codex app-server media handoff gap where inbound channel media becomes prompt text but not `params.images`, so the model can receive `imagesCount: 0` on image-bearing turns.

## What This Changes

The issue reports that OpenAI Responses Codex app-server turns do not populate `params.images` from inbound `ctx.MediaPath` / `ctx.MediaPaths` attachments.

## Best Possible Solution

Add a bounded Codex app-server media handoff that resolves image attachments from structured inbound context into `params.images` before `turn/start`, preserves caller-supplied images and ordering, and updates the image-tool visibility signal based on images actually delivered.

## Reproduction Assessment

Yes, at source level. Current main turns `ctx.MediaPath(s)` into a prompt note, but the Codex app-server `turn/start` payload only includes images already present in `params.images`, and the inspected reply path does not populate that field from inbound media.

## Solution Assessment

Yes. The narrow fix is to reuse the existing media attachment normalization/cache boundary to populate app-server `params.images` from structured context before the turn is submitted, rather than adding a new config knob or relying on prompt-text parsing.
