## Label Justifications

- `P1`: The issue describes a broken image-bearing agent workflow where users can receive model answers without the attached pixels.
- `impact:message-loss`: Inbound image attachments can be omitted from the model message payload on the reported transport.

## Summary

Keep open: current `main` still has a source-backed gap where Codex app-server/openai-responses turns only forward already-populated `params.images`, while inbound `MediaPath`/`MediaPaths` can reach the prompt only as a media note. The report is recent, detailed, and matches the current implementation, so this is an actionable transport bug rather than stale cleanup.

## What This Changes

The issue reports that openai-responses Codex app-server turns include inbound image attachments only as prompt text and do not populate the image payload sent to the model.

## Best Possible Solution

Add a focused Codex app-server hydration step that converts current-turn image `MediaPath`/`MediaPaths` into `params.images` before `turn/start`, reusing existing media-loading safeguards and preserving explicit image ordering.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path because app-server turn input and telemetry consume only `params.images`, while inbound channel media is independently rendered into prompt text via `MediaPath`/`MediaPaths`. I did not run the live Discord scenario in this read-only pass.

## Solution Assessment

Yes: the best fix is to hydrate structured current-turn media into the existing image payload before the app-server turn starts, rather than relying on the image tool fallback or changing provider defaults.
