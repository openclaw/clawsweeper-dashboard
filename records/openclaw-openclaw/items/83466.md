## Summary

Keep open: current `main` still has a source-backed gap where Codex app-server turns only forward already-populated `params.images`, while inbound `MediaPath`/`MediaPaths` can reach the prompt only as a media note. The report is recent, detailed, and no duplicate or merged fix turned up in targeted searches.

## What This Changes

The issue asks OpenClaw to preserve inbound image attachments for `provider=openai, modelApi=openai-responses` Codex app-server turns by populating `params.images` from current-turn `MediaPath`/`MediaPaths`.

## Best Possible Solution

Add a focused Codex app-server hydration step that converts current-turn image `MediaPath`/`MediaPaths` into `params.images` before `turn/start`, reusing existing media-loading safeguards and preserving explicit image ordering.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path because app-server turn input and telemetry consume only `params.images`, while channel media can independently appear in the prompt through `MediaPath`/`MediaPaths`. I did not run the live Discord scenario in this read-only pass.

## Solution Assessment

Yes: hydrating structured current-turn image media into the existing image payload before the app-server turn starts is narrower and safer than relying on the fallback image tool or changing provider defaults.
