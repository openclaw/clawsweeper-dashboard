## Summary

Keep open. Current main still only marks archived Discord thread sessions stale, generic cleanup preserves external channel/thread rows, and the available delete path requires explicit selected keys rather than providing Discord orphan cleanup or agent-facing GC.

## What This Changes

The issue requests cleanup or deletion support for stale Discord thread session entries that remain listed after backing threads are archived, deleted, or inaccessible.

## Best Possible Solution

Define a maintainer-approved, plugin-aware orphan cleanup flow that previews and revalidates unreachable Discord candidates, then deletes through the existing session-delete lifecycle with explicit permission and transcript-retention semantics.

## Reproduction Assessment

Yes, at source level: current main resets archived Discord thread session rows, preserves Discord-like external conversation rows during generic pruning, and only exposes explicit-key deletion. I did not run a live Discord 403/404 deletion scenario.

## Solution Assessment

No, the current implementation is only a partial manual/operator solution. The maintainable path is a plugin-owned reachability preview/apply flow that reuses existing delete hooks, not unconditional startup polling or unrestricted agent deletion.
