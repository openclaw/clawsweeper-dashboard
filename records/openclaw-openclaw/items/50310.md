## Summary

Keep open. Current main still marks archived Discord thread sessions stale instead of pruning their sessions.json rows, generic cleanup preserves durable channel/thread rows, and the related Dashboard prune issue is broader but does not replace this Discord-specific cleanup and permission-policy work.

## What This Changes

The issue requests automatic cleanup or explicit deletion tooling for stale Discord thread session entries that remain in sessions.json after the backing thread is archived or deleted.

## Best Possible Solution

Add a Discord plugin-owned preview/apply stale-session cleanup flow that verifies reachability, revalidates candidates before deletion, preserves transcript policy, and exposes operator or agent controls only after maintainer-approved permission semantics.

## Reproduction Assessment

Yes, at source level; current main's Discord archive handler sets updatedAt=0 rather than deleting rows, generic maintenance preserves Discord channel/thread keys, and agent-visible tool groups still omit sessions_delete or sessions_gc. No live Discord repro was run.

## Solution Assessment

Unclear until maintainers choose cleanup semantics and permissions. The maintainable direction is plugin-owned reachability preview/apply cleanup rather than unconditional core startup polling or a broad delete tool by default.
