## Summary

Keep open. Current main still only resets archived Discord thread sessions by setting `updatedAt` to 0, preserves durable external channel/thread rows during generic cleanup, and exposes selected-key deletion to gateway/UI callers rather than a Discord orphan cleanup or agent-facing delete/GC flow.

## What This Changes

Request automatic or agent-triggered cleanup of stale Discord thread session entries left in `sessions.json` after backing Discord threads are archived, deleted, or inaccessible.

## Best Possible Solution

Design a Discord plugin-owned orphan cleanup flow that previews and revalidates unreachable candidates, then deletes through the existing session-delete lifecycle with explicit permission and transcript-retention semantics.

## Reproduction Assessment

Yes, at source level: current main resets archived Discord thread sessions to `updatedAt: 0`, preserves external conversation rows during generic cleanup, and lacks an agent-facing delete or GC tool. I did not run a live Discord 403/404 cleanup scenario.

## Solution Assessment

No, current main is only a partial manual/operator solution. The maintainable path is a plugin-owned reachability preview/apply flow that reuses existing delete hooks, not unconditional startup polling or unrestricted agent deletion.
