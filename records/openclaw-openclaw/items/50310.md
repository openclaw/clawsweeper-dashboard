## Summary

Keep open. Current main still resets archived Discord thread sessions by zeroing `updatedAt` and has generic admin cleanup/delete APIs, but it does not perform Discord liveness-based pruning or expose an agent-safe `sessions_delete` or `sessions_gc` surface; related #49259 is broader Dashboard pruning work and does not fully cover this Discord thread lifecycle request.

## What This Changes

Request automatic or controlled cleanup for stale Discord thread session entries left in `sessions.json` after Discord threads are archived or deleted.

## Best Possible Solution

Add a scoped Discord-owned orphan reconciliation path that previews and applies deletion only for confirmed unreachable Discord thread/channel sessions, with store mutation routed through existing Gateway session writers and a clear agent/UI permission model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed `agent:<agentId>:discord:channel:<threadId>`, run the current Discord archive close path, and the store entry remains with `updatedAt = 0`; `sessions.list` then reads it from local store without Discord liveness validation.

## Solution Assessment

Unclear. The current implementation is not enough, and the issue’s proposed options need maintainer choice; a scoped prune preview/apply workflow with Discord reachability checks is safer than a broad unrestricted agent delete tool.
