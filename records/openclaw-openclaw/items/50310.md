## Summary

Keep open. Current main still resets archived Discord thread sessions in place and `sessions.list` still reports persisted store rows without Discord reachability checks; existing admin cleanup/delete surfaces do not provide Discord-aware stale-orphan pruning or an agent-safe GC policy, and the related deleted-channel and Dashboard items do not supersede this Discord archive/delete lifecycle request.

## What This Changes

The issue requests a Discord-aware cleanup path for stale or orphaned thread sessions that remain in `sessions.json` after Discord threads are archived or deleted.

## Best Possible Solution

Add a Discord-owned stale-orphan liveness check plus a Gateway-owned preview/apply cleanup path that revalidates candidates before deletion and uses the existing session writer, transcript archival, and explicit admin or narrowly scoped agent permission boundary.

## Reproduction Assessment

Yes, at source level. An archived Discord thread calls `closeDiscordThreadSessions`, which leaves the row present with `updatedAt = 0`, and `sessions.list` still lists persisted rows without a Discord reachability check.

## Solution Assessment

Unclear. Startup GC, a broad agent delete tool, and TTL expiry each solve part of the report, but the safer maintainable direction is plugin-owned liveness plus Gateway-owned preview/apply deletion semantics.
