## Summary

Keep open. Current main has manual operator deletion and generic cleanup, but Discord archived/deleted thread handling still preserves stale session rows and no agent-facing delete/GC tool or Discord reachability cleanup flow exists.

## What This Changes

The issue requests automatic cleanup or deletion capability for stale Discord thread-backed session entries that remain in sessions.json after the backing thread is archived or deleted.

## Best Possible Solution

Design a Discord/plugin-owned stale-session cleanup flow that previews and revalidates unreachable candidates, then deletes through the existing session-delete path under maintainer-approved permission and transcript-retention rules.

## Reproduction Assessment

Yes, at source level: current main exposes an operator-admin sessions.delete RPC but no model-facing sessions_delete or GC tool, and Discord archive handling marks matching rows stale instead of deleting them. I did not run a live Discord repro.

## Solution Assessment

No, the current implementation is not the full solution. The maintainable direction is a plugin-owned reachability preview/apply cleanup that reuses the existing delete path, not unconditional core startup polling or a broad agent delete tool by default.
