## Summary

Keep open. Current main has admin/UI deletion plumbing, but archived Discord threads are still reset in place, listed from local session-store rows, protected from generic cleanup, and not covered by a Discord reachability-based or agent-safe stale-session GC path.

## What This Changes

Request adding Discord-aware cleanup or agent-safe deletion/GC for stale session-store rows left by archived or deleted Discord threads.

## Best Possible Solution

Add a Discord-aware stale-orphan preview/apply cleanup path that delegates reachability to the Discord plugin, revalidates candidates before deletion, and deletes through the existing Gateway session writer under explicit admin or narrowly scoped agent permissions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: archived Discord thread handling leaves matching rows in `sessions.json` with `updatedAt = 0`, and `sessions.list` still reads those persisted rows without Discord reachability checks.

## Solution Assessment

Unclear. The proposed startup GC, broad agent delete tool, and TTL expiry each cover part of the problem, but the safer maintainable direction is an explicit preview/apply cleanup contract using plugin-owned liveness checks and existing Gateway deletion semantics.
