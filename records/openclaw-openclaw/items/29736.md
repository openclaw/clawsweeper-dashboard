## Summary

Keep this issue open: current main and the latest shipped tag still resolve exec approvals file, socket, CLI/gateway storage, and effective-policy default source text through home-backed paths, while the focused fix PR remains open and needs maintainer/security review.

## What This Changes

The issue reports that exec approvals ignore the active OpenClaw state root and are read or written under the home-backed default instead of `OPENCLAW_STATE_DIR` or profile-specific state roots.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74002 or an equivalent maintainer-approved fix that derives approvals JSON, socket, and effective-policy host paths from the active state-dir/profile contract while preserving intentional fallback and migration semantics.

## Reproduction Assessment

Yes. Source inspection is high-confidence: set `OPENCLAW_STATE_DIR` to a non-default root and `resolveStateDir()` honors it, but current exec approvals resolvers still expand `~/.openclaw/exec-approvals.*` and the CLI/gateway callers use those helpers.

## Solution Assessment

Yes for the bug boundary, but not as an automated new fix PR. The narrow maintainable path is the existing state-dir-aware resolver PR or a maintainer-approved equivalent, with explicit security/product acceptance for migration and socket-root behavior.
