## Summary

Keep open, but not merge-ready: the patch treats every bridge record from another PID as stale even though OpenClaw supports multiple same-user gateway profiles that share the UID-scoped relay directory. The contributor also needs stronger after-fix proof from the submitted branch.

## What This Changes

This PR adds a registration-time scan of the native hook relay bridge directory that deletes `.json` records whose recorded PID differs from the current process.

## Best Possible Solution

Keep the self-healing idea, but constrain pruning to dead or expired records, or to the current relay's own registry file, while preserving live records from other supported gateway profiles and adding focused regression coverage.

## Reproduction Assessment

Yes at source level for the introduced PR bug: the bridge directory is UID-scoped, docs support multiple same-user gateway profiles, and the new predicate removes any record from a different PID. I did not run a live two-gateway repro in this read-only review.

## Solution Assessment

No. The cleanup direction is useful, but deleting every non-current-PID record is too broad; the maintainable fix should verify stale/dead records or limit cleanup to the current relay's own record.
