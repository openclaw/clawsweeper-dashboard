## Summary

Keep this PR open because the reported stale bridge symptom is real and the proposed fix is directionally useful, but the current prune condition is too broad for documented same-user multi-gateway/profile setups and the submitted branch still lacks sufficient real behavior proof after the latest code change.

## What This Changes

The PR adds registration-time cleanup in `src/agents/harness/native-hook-relay.ts` that removes native hook relay bridge `.json` records whose stored PID differs from the current process.

## Best Possible Solution

Keep the self-healing stale-record cleanup, but make it liveness-aware so it removes expired/dead/unreachable bridge records while preserving live records from other same-user gateways/profiles, with focused regression coverage and redacted proof from the submitted branch.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the patch regression: create a live bridge record from one same-user gateway/profile, then have another process register and prune records where `pid !== process.pid`. The original stale-record failure also has useful reporter live output, but I did not run a current-main live repro in this read-only review.

## Solution Assessment

No. The current direction is useful, but the PID inequality is not the best boundary; the safer fix is to delete only records proven dead, expired, or unreachable while preserving live foreign records.
