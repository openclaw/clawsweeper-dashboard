## Summary

Keep open: the stale-prune direction addresses a real native-hook relay failure, but the PID-only deletion predicate can remove bridge records owned by another live same-UID gateway/profile, so this needs a narrower liveness or expiry check before merge.

## What This Changes

The PR adds a registration-time scan in `src/agents/harness/native-hook-relay.ts` that deletes native hook bridge `.json` records whose stored PID differs from the current process.

## Best Possible Solution

Prune only expired records or records whose PID/port is proven dead, preserve live foreign-process bridge records, and add focused regression coverage plus branch-based real proof.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the shared UID directory and PID-only prune would delete live foreign-process records. I did not run a live multi-gateway repro.

## Solution Assessment

No. Pruning stale files is the right area, but the current predicate should verify expiry or process liveness instead of treating every non-current PID as stale.
