## Summary

Keep open: the stale bridge-record symptom is real and the PR is aimed at the right runtime surface, but the current predicate can delete valid bridge records owned by another live same-user OpenClaw gateway/profile process.

## What This Changes

This PR adds registration-time pruning in `src/agents/harness/native-hook-relay.ts` to remove bridge JSON files whose stored PID differs from the current process.

## Best Possible Solution

Prune only records that are actually stale, such as expired records or records whose PID/port no longer represents a live bridge, while preserving valid records owned by other same-user OpenClaw processes.

## Reproduction Assessment

Yes for the PR regression: source inspection shows the new predicate deletes any bridge record with a different PID in a UID-scoped directory. The original stale-record symptom also has useful live logs in the PR body, but the preservation case is not proven.

## Solution Assessment

No; the submitted approach is too broad because non-current PID does not mean stale in a shared same-user bridge directory. The safer fix is to prune only expired, dead, or unreachable records and cover both stale deletion and live preservation in tests.
