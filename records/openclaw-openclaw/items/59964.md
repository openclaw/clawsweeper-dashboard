## Summary

Keep open for maintainer review. Current main already has a newer last-known-good recovery path for invalid startup and hot-reload config states, but it does not implement the PR's explicit `openclaw config recover` manual command, and the PR diff has blocking correctness and privacy issues.

## What This Changes

The PR adds a draft proposal doc plus config recovery code for a new `openclaw config recover` subcommand, backup-candidate recovery helpers, config IO exports, and gateway startup/hot-reload recovery notifications.

## Best Possible Solution

Either close this PR manually in favor of the current last-known-good recovery design, or rework it into a narrow current-main follow-up that adds a manual recovery command on top of the existing recovery policy, docs, tests, and notification boundaries without replacing the newer recovery machinery.

## Reproduction Assessment

Yes for the PR findings: the notification bug follows from a prior in-process config write setting `lastWriteEvent`, then a later invalid filesystem-only edit triggering recovery; the dry-run bug follows from `--from` selecting an invalid candidate while the result still exposes `selectedSource`. For the broader feature request, current-main source/docs provide high-confidence behavior checks but I did not run live tests in this read-only review.

## Solution Assessment

No. The PR's direct backup-ring recovery and cached-channel notification approach is not the best fit for current main; the safer path is to extend the existing last-known-good recovery policy and only add a manual CLI if maintainers still want that product surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope recovery notices to the triggering write:** `src/gateway/config-reload.ts:110-116`
  - body: The reload notifier reads the cached `lastWriteEvent` for every recovery alert. If a later filesystem-only edit corrupts the config, there is no matching write event, so the auto-recovery warning can be delivered to the previous session/thread and expose config failure details to an unrelated conversation. Tie the notice to the actual triggering write or clear the context when the reload is not from that write.
  - confidence: 0.91
- **[P2] Return no selected source for invalid requested backups:** `src/config/io.ts:2886-2887`
  - body: When `--from` names an invalid candidate, `selected` is non-null but unusable, and this branch still returns `selectedSource`/`restoredPath`. `runConfigRecover` treats `selectedSource` as the dry-run success signal, so `openclaw config recover --dry-run --from bak.N` can report it would recover and exit successfully even when that backup is invalid.
  - confidence: 0.88
- **[P2] Remove the draft proposal from user docs:** `docs/PR-PROPOSAL.md:3-8`
  - body: `docs/PR-PROPOSAL.md` is a draft design note in Chinese, including a note that it should be rebased before submission. This repo's docs source is English and user-facing, so this should live in the PR discussion or be replaced by English documentation for the actual shipped command.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete privacy/security concern by routing recovery notices through stale cached channel context.

Concerns:

- **[medium] Recovery notice can go to a stale conversation:** `src/gateway/config-reload.ts:110`
  - body: `notifyReloadFailure` uses `lastWriteEvent` even when the invalid config change was a later filesystem-only edit. Recovery text includes config failure details and file paths, so sending it to the previous channel recipient can leak operator metadata and confuse an unrelated session.
  - confidence: 0.9
