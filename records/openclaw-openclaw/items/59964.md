## What This Changes

The PR adds a config recovery CLI, backup-candidate recovery helpers/exports, startup and hot-reload recovery hooks with channel notifications, and a draft proposal document.

## Best Possible Solution

Keep the current doctor/last-known-good recovery boundary unless maintainers choose a fresh, scoped recovery CLI or auto-reload design that fixes notification routing and preserves plugin-local recovery safeguards.

Yes for the PR findings: the stale-notification path follows from `lastWriteEvent` being reused for later invalid filesystem snapshots, and the dry-run issue follows from the invalid-selection return shape. I did not run tests because this review was read-only.

No. The PR mixes a new CLI, broader auto-recovery policy, channel delivery, and a draft doc; a maintainable fix should be a current-main follow-up after maintainer agreement on the recovery surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope recovery notices to the triggering write:** `src/gateway/config-reload.ts:110-116`
  - body: The reload notifier reads cached `lastWriteEvent` for every recovery alert. A later filesystem-only invalid edit can reuse the previous session/thread and expose config failure details to an unrelated conversation, so the notice must be tied to the actual triggering write or omitted when there is no matching write context.
  - confidence: 0.91
- **[P2] Clear selectedSource for invalid requested backups:** `src/config/io.ts:2886-2887`
  - body: When `--from` names an invalid candidate, the failure return still includes `selectedSource` and `restoredPath`. `runConfigRecover` treats `selectedSource` as dry-run success, so it can claim recovery would succeed from an unusable backup.
  - confidence: 0.88
- **[P2] Remove the draft proposal from user docs:** `docs/PR-PROPOSAL.md:3-8`
  - body: `docs/PR-PROPOSAL.md` is a draft proposal with non-English text and a rebase note, not user-facing documentation for a shipped command. Keep design notes in the PR discussion or replace this with final English docs after the behavior is accepted.
  - confidence: 0.86
