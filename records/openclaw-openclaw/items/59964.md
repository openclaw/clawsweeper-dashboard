## Summary

Keep open, but this PR is not mergeable as written: it has a recovery-notice privacy bug, incorrect dry-run success semantics, candidate validation that bypasses the canonical config read contract, no real behavior proof, and it cuts across current main's doctor-owned recovery policy.

## What This Changes

Adds a `openclaw config recover` CLI, broader config backup recovery for broken configs, gateway hot-reload rollback notices, and a proposal document under `docs/`.

## Best Possible Solution

Rework the feature against current main's doctor-owned recovery model, validate backups through the canonical config read pipeline, keep any automatic runtime restore/notice policy explicitly approved and documented, remove the proposal doc, and add focused recovery and CLI tests plus real runtime proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the stale `lastWriteEvent` notice path, the invalid `selectedSource` dry-run path, and the raw candidate-validation shortcut. The PR still lacks after-fix real behavior proof from a running setup.

## Solution Assessment

No. The proposed direction is plausible, but this implementation is not the safest maintainable path because it conflicts with current recovery policy, misroutes notices, reports invalid dry-runs as recoverable, and skips normal config read semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope recovery notices to the triggering write:** `src/gateway/config-reload.ts:112-124`
  - body: The notifier reuses the cached `lastWriteEvent` for every later recovery alert. A filesystem-only corrupt edit after a previous in-process write can therefore deliver the rollback warning, config path, and failure details to an unrelated prior session/thread; tie the notice to the triggering write or clear the context before notifying.
  - confidence: 0.91
- **[P2] Do not treat invalid requested backups as recoverable:** `src/config/io.ts:2889-2890`
  - body: When `--from` selects an invalid candidate, this branch still returns that candidate in `selectedSource`/`restoredPath`. `runConfigRecover --dry-run` treats `selectedSource` as success, so it can print "would recover" and exit 0 even though recovery cannot happen.
  - confidence: 0.94
- **[P2] Validate candidates through the normal config reader:** `src/config/io.ts:1004`
  - body: Backup candidates are validated directly from raw parsed JSON5, bypassing include resolution, env substitution, and existing config read migrations. A valid backed-up config that relies on those supported features can be rejected as unrecoverable; reuse the canonical read/validation pipeline for candidate checks.
  - confidence: 0.82
- **[P3] Remove the proposal from published docs:** `docs/PR-PROPOSAL.md:3-8`
  - body: This adds a non-English design proposal under `docs/`, including a note that it should be rebased before submission. Design notes belong in the PR body or an issue, and repo docs should remain polished English user/operator documentation.
  - confidence: 0.88
