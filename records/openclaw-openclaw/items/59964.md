## Summary

Keep open and block merge. Current main has a newer last-known-good recovery path for startup and hot reload, but this PR is still not mergeable because it overlaps that design, keeps a security-sensitive stale-notification path, and leaves the manual CLI as a product decision.

## What This Changes

The PR adds a config recovery CLI, backup-candidate recovery helpers/exports, startup and hot-reload recovery hooks with channel notifications, and a draft proposal document.

## Best Possible Solution

Keep the current last-known-good recovery architecture, and only add a narrow manual recovery command if maintainers want that CLI on top of the existing policy, docs, tests, and notification boundaries.

## Reproduction Assessment

Yes for the PR findings: the stale-notification path follows from a prior in-process write setting lastWriteEvent, then a later filesystem-only invalid edit reusing that cached context; the dry-run issue follows directly from the invalid-selection return shape. For the broader behavior, current-main source, docs, and tests give high-confidence checks without executing tests in this read-only review.

## Solution Assessment

No. The PR's direct backup-ring recovery and cached-channel notification approach is not the best fit for current main; a safer path is a focused current-main follow-up that extends the existing last-known-good recovery policy and adds a manual CLI only if maintainers choose that surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope recovery notices to the triggering write:** `src/gateway/config-reload.ts:110-116`
  - body: The reload notifier reads the cached lastWriteEvent for every recovery alert. If a later filesystem-only edit corrupts the config, there is no matching write event, so the auto-recovery warning can be delivered to the previous session/thread and expose config failure details to an unrelated conversation. Tie the notice to the actual triggering write or clear the context when the reload is not from that write.
  - confidence: 0.91
- **[P2] Clear selectedSource for invalid requested backups:** `src/config/io.ts:2886-2887`
  - body: When --from names an invalid candidate, selected is non-null but unusable, and this branch still returns selectedSource/restoredPath. runConfigRecover treats selectedSource as the dry-run success signal, so openclaw config recover --dry-run --from bak.N can report it would recover and exit successfully even when that backup is invalid.
  - confidence: 0.88
- **[P2] Remove the draft proposal from user docs:** `docs/PR-PROPOSAL.md:3-8`
  - body: docs/PR-PROPOSAL.md is a draft design note in Chinese, including a note that it should be rebased before submission. This repo's docs source is English and user-facing, so this should live in the PR discussion or be replaced by English documentation for the actual shipped command.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete privacy/security issue by routing recovery notices through stale cached channel context.

Concerns:

- **[medium] Recovery notice can go to a stale conversation:** `src/gateway/config-reload.ts:110`
  - body: notifyReloadFailure uses lastWriteEvent even when the invalid config change was a later filesystem-only edit. Recovery text includes config failure details and file paths, so sending it to the previous channel recipient can leak operator metadata and confuse an unrelated session.
  - confidence: 0.9
