## Summary

Keep open. The PR contains meaningful config recovery work, but it is not safe to merge because the branch has source-reproducible correctness and security issues, lacks real behavior proof, and still needs maintainer direction for the new recovery CLI/policy surface.

## What This Changes

The PR adds a `openclaw config recover` command, backup-candidate recovery helpers/exports, startup and hot-reload recovery hooks with channel notifications, and a draft proposal document.

## Best Possible Solution

Build on the current last-known-good and doctor-owned recovery architecture, fix the PR's notification and CLI result semantics, remove the draft docs, and add the manual recovery command only if maintainers approve that surface.

## Reproduction Assessment

## Solution Assessment

No. The branch's direct backup-ring recovery and cached-channel notification approach need fixes and maintainer policy direction before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope recovery notices to the triggering write:** `src/gateway/config-reload.ts:110-116`
  - body: The reload notifier reads `sessionKey` and delivery context from the cached `lastWriteEvent` for every recovery alert. A later filesystem-only invalid edit has no matching write event, so the warning can be delivered to the previous session/thread and expose config failure details to an unrelated conversation.
  - confidence: 0.91
- **[P2] Clear selectedSource for invalid requested backups:** `src/config/io.ts:2886-2887`
  - body: When `--from` names an invalid candidate, this branch still returns `selectedSource` and `restoredPath`. `runConfigRecover` uses `selectedSource` as the dry-run success signal, so `openclaw config recover --dry-run --from bak.N` can report it would recover from an invalid backup.
  - confidence: 0.88
- **[P2] Remove the draft proposal from user docs:** `docs/PR-PROPOSAL.md:3-8`
  - body: `docs/PR-PROPOSAL.md` is a draft design note with non-English content, including a note that it should be rebased before submission. It should live in the PR discussion or be replaced with final English user documentation for a shipped command.
  - confidence: 0.86
