## Summary

Keep open. Current main still has the internal `POST /start?headless=false` path, but the public CLI/client/browser-tool/docs surface cannot request a headed runtime start; the profile-lock concern raised in discussion appears covered by current `Singleton*` cleanup, and open PR #75881 is the right paired implementation to review before this issue closes.

## What This Changes

Expose a runtime headed start override for managed browser profiles through `openclaw browser start` and the browser tool without editing config or restarting the Gateway.

## Best Possible Solution

Review and land or close PR #75881; the accepted implementation should wire the existing request-local override through CLI, client, browser tool, docs, tests, and changelog so the issue closes after merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the internal route can accept `headless=false`, while CLI, client helper, browser tool schema/action, and docs expose only the headless=true direction; no live browser handoff was needed for this feature-gap check.

## Solution Assessment

Yes. Reusing the existing request-local `headless` override is the narrowest maintainable direction; this issue should stay open while the linked implementation PR is reviewed rather than spawning duplicate repair work.
