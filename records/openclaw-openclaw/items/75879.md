## Summary

Keep open. Current main already has the lower-level browser start route needed for `headless=false`, but the public CLI, browser client, docs, and agent browser tool still expose only the headless half, so the requested runtime headed handoff surface remains missing.

## What This Changes

Request to expose a runtime headed browser launch override in the browser CLI and browser tool so a managed profile can be stopped and restarted visibly without config edits or Gateway restart.

## Best Possible Solution

Add an additive headed start override that reuses the existing `/start?headless=false` path, threads it through the CLI/browser client/browser tool, documents the stop-then-start handoff, and leaves config-level headless behavior unchanged.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: the HTTP route can accept `headless=false`, while `openclaw browser start`, `browserStart()`, and the browser tool start action provide no public way to send that false override.

## Solution Assessment

Yes. The narrowest maintainable solution is to expose the existing request-level override rather than changing config, startup migration, or running Chrome process semantics.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/75879 as a focused browser plugin change. Add a public runtime headed start override for local managed browser profiles without changing config semantics: add `openclaw browser start --headed` and reject `--headless --headed` together; support `--headless=false` only if it fits the existing Commander patterns safely. Thread the selected override through `browserStart()` and the browser tool `action: "start"` using `headless: false` and/or `headed: true`, including proxy/node start paths. Reuse the existing `/start?headless=false` route and keep existing rejection for remote CDP, attach-only, and existing-session profiles; do not attempt to mutate a running Chrome process. Update docs for `docs/cli/browser.md`, `docs/tools/browser-control.md`, and `docs/tools/browser.md` to show `stop -> start --headed` for human handoff, and add an Unreleased changelog entry. Add focused tests for no flag sending no override, `--headless` sending true, `--headed` or `headless:false` sending false, conflicting flags failing clearly, the route passing `headless:false`, and browser tool host/proxy start forwarding the override. Do not touch dependency pins, plugin architecture, or unrelated browser behavior.
