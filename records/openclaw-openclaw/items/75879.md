## Summary

Keep open. Current main already has the lower-level `/start?headless=false` route behavior, but the public CLI, browser client, agent tool schema/tool wiring, and docs still expose only the headless direction; two open PRs already reference this issue with closing syntax, so it should stay open until one fix lands.

## What This Changes

Expose a runtime headed browser start override for human handoff without editing browser config or restarting the Gateway.

## Best Possible Solution

Land one canonical focused browser plugin PR that threads the existing request-local `headless=false` override through CLI, client, browser tool, docs, and tests, while separately tracking stale profile-lock hardening if it is not included.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: the HTTP route can accept `headless=false`, while the public CLI/client/tool surfaces provide no way to send that false override.

## Solution Assessment

Yes. Exposing the existing request-level override is the narrowest maintainable fix; it avoids config mutation, Gateway restart semantics, and attempts to change a running Chrome process.
