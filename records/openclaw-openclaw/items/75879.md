## Summary

Keep open. Current main still has the internal `/start?headless=false` capability but lacks the public CLI, client, browser-tool, and docs surface requested here; open closing PRs #75880 and #75881 should be reviewed before this issue closes.

## What This Changes

The issue requests a public runtime headed browser start override for managed browser profiles without editing config or restarting the Gateway.

## Best Possible Solution

Land one focused implementation that reuses `/start?headless=false` across the CLI, browser client, browser tool, docs, tests, and changelog, then let the selected closing PR resolve this issue.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the gap: the route accepts `headless=false`, while the public CLI/client/tool surfaces cannot send that false override.

## Solution Assessment

Yes. Reusing the existing request-local start override is the narrowest maintainable fix; the profile-lock concern from discussion is adjacent and already has current-main recovery coverage.
