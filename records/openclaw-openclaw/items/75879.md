## Summary

Keep open. Current main still has the feature gap: the lower-level `/start?headless=false` route path exists, but the CLI, browser client helper, docs, and browser tool still expose only the headless direction; two open PRs already reference this issue with closing syntax, so the issue should stay open until one is reviewed and landed or closed.

## What This Changes

Expose a runtime headed browser start override in the CLI and browser tool/client so human handoff can stop and restart the same managed profile headed without editing config.

## Best Possible Solution

Land one clean implementation PR that adds `--headed` or `headless:false` to the CLI, `browserStart()`, browser tool/node proxy path, docs, tests, and changelog while preserving existing config, restart, and stale-lock semantics.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: `/start?headless=false` can be parsed and applied internally, but no public CLI/tool/client surface constructs that false-valued override.

## Solution Assessment

Yes for the feature direction. The narrow maintainable fix is to expose the existing request-level override and document stop-then-start headed handoff, not mutate config or try to change headed mode on an already-running Chrome process.
