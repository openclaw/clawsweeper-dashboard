## Summary

Keep open. Current main still has the internal `POST /start?headless=false` path, but the public CLI, browser client helper, browser tool schema, and docs still expose only the headless direction; open PRs #75880 and #75881 explicitly target this issue, so it should remain open until one implementation lands.

## What This Changes

The issue requests a runtime headed browser start override for human handoff without editing browser config or restarting the Gateway.

## Best Possible Solution

Land one focused implementation that reuses the existing request-level override across CLI, browser client, browser tool, docs, tests, and changelog, then close or supersede the duplicate PR.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the gap: the HTTP route accepts `headless=false`, but the public CLI/client/tool/docs cannot send or document that false override.

## Solution Assessment

Yes. Reusing the existing `/start?headless=false` request override is the narrowest maintainable solution; the stale-lock concern is adjacent and already has current-main recovery unless a new failing case is demonstrated.
