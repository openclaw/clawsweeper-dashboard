## Summary

Keep open. Current main still exposes the internal `POST /start?headless=false` path but not the requested public CLI, client, browser-tool, or docs surface; two open PRs already reference this issue with closing syntax, so it should close only after one canonical fix lands.

## What This Changes

Request a runtime headed browser start override for human handoff without editing browser config or restarting the Gateway.

## Best Possible Solution

Land one canonical narrow PR that threads `headless=false` through the CLI, browser client, browser tool, docs, and changelog while preserving the existing route and config semantics.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the gap: the route accepts `headless=false`, but the public CLI, client helper, browser tool schema/action, and docs expose only the headless-true side.

## Solution Assessment

Yes. The narrow maintainable fix is to expose the existing request-local override and document the stop-then-start handoff; the profile-lock concern appears already covered by current stale Singleton lock recovery unless new tests prove a separate gap.
