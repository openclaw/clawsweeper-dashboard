## Summary

Keep open. Current main still has the internal request-local `headless=false` route behavior but lacks the requested public CLI, client, browser-tool, and docs surface, and open PR #75881 already references this issue with closing syntax.

## What This Changes

Requests a runtime headed browser handoff flow for OpenClaw-managed profiles without editing config or restarting the Gateway.

## Best Possible Solution

Land one focused implementation that reuses the existing request-level override across CLI, browser client, browser tool/proxy path, docs, tests, and changelog while keeping the existing stale-lock recovery path separate.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the feature gap: `/start` can accept `headless=false`, while public CLI/client/tool/docs surfaces only expose `headless=true` or config-level headed choices.

## Solution Assessment

Yes. Reusing the existing request-local `headless` override after an explicit stop is the narrowest maintainable solution; live mode mutation or config rewrites would broaden the behavior unnecessarily.
