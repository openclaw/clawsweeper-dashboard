## Summary

Keep open. Current main still has the same contract/runtime gap described in the discussion: iMessage accepts and documents `dmHistoryLimit`, but the iMessage monitor only records and injects pending history for group messages. The original `attributedBody` detail remains useful implementation guidance, and related PR #73775 was closed unmerged, so there is no implemented or superseding close reason.

## What This Changes

The issue requests decoded iMessage direct-message history for `dmHistoryLimit`, avoiding blank outbound history caused by relying on macOS Messages rows whose `text` column can be empty while `attributedBody` contains the message text.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in `extensions/imessage` using verified `imsg history` or RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while preserving the accepted config field for compatibility.

## Reproduction Assessment

Yes for the reframed current-main gap: source inspection shows iMessage config/docs expose `dmHistoryLimit`, while the iMessage monitor only records and injects group pending history. The original database-level blank outbound text symptom is plausible but still needs a live macOS/imsg check to reproduce directly.

## Solution Assessment

Unclear. The requested decoded `imsg` history/RPC direction is a plausible runtime fix if iMessage DM backfill is intended, but a docs-contract correction is also maintainable and maintainers have not chosen between those paths after PR #73775 was closed unmerged.
