## Summary

Keep open. Current main still has the iMessage `dmHistoryLimit` contract/runtime gap from the discussion: the field is accepted and the docs still list iMessage as supported, but the iMessage monitor only builds pending history for group messages and does not implement a decoded direct-message history adapter; related PR #73775 was closed unmerged.

## What This Changes

Fix iMessage direct-message history so `dmHistoryLimit` provides decoded, nonblank prior conversation context for outbound messages, or stop presenting iMessage as supporting that DM-history behavior.

## Best Possible Solution

Make the iMessage contract explicit: either implement decoded direct-message history in `extensions/imessage` using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the config field backward compatible.

## Reproduction Assessment

Yes for the reframed current-main gap: source inspection shows iMessage accepts and documents `dmHistoryLimit`, while the runtime history context path is group-only. The attributedBody-specific blank outbound text symptom itself still needs live macOS/imsg verification.

## Solution Assessment

Unclear. The requested decoded `imsg` history/RPC direction is plausible if iMessage DM history is intended, but a docs-contract correction is also coherent and a prior docs-only PR was closed unmerged, so maintainers need to pick the intended product behavior.
