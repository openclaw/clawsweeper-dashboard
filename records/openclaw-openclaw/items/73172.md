## Summary

Keep open. Current main still accepts and documents iMessage direct-message history, but the iMessage runtime only builds group history; the later iMessage catchup work is for missed inbound rows during gateway downtime and does not implement `dmHistoryLimit` conversation history.

## What This Changes

The issue reports that iMessage `dmHistoryLimit` loses outbound text because macOS stores sent content in `attributedBody`, and asks OpenClaw to use decoded `imsg` history/RPC output for DM history injection.

## Best Possible Solution

Make the iMessage contract honest: either implement decoded direct-message history through verified `imsg` history/RPC output with outbound-attributedBody coverage, or remove iMessage from the documented supported list while keeping the schema field compatible.

## Reproduction Assessment

Yes for the current-main contract mismatch: source inspection shows `channels.imessage.dmHistoryLimit` is accepted and documented, while the iMessage monitor only uses group `historyLimit` and only populates `InboundHistory` for groups. The exact blank-outbound `attributedBody` path still needs live macOS or dependency-backed verification.

## Solution Assessment

Unclear. A decoded `imsg` history/RPC adapter is the right runtime fix if iMessage DM history is intended, but the narrower docs-honest fix is also valid if maintainers do not want to support that runtime contract yet.
