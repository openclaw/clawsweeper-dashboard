## Summary

Keep open. Current main still exposes and documents iMessage `dmHistoryLimit`, and the generic session-history limiter treats iMessage like a supported DM-history provider, but the iMessage runtime only builds pending history for group messages. The reporter's `attributedBody` evidence remains useful implementation guidance, while the current-main bug is best framed as an unresolved iMessage DM-history contract gap rather than a proven direct `chat.db` text-column read path.

## What This Changes

The issue asks for iMessage `dmHistoryLimit` history injection to preserve decoded outbound DM message text instead of injecting blank/null outbound entries.

## Best Possible Solution

Make the iMessage contract explicit: either add a decoded direct-message history adapter in the iMessage plugin using verified `imsg` history/RPC output with regression coverage for outbound text, or de-advertise iMessage DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and the schema/docs/generic limiter recognize it, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The exact `attributedBody` database behavior remains reporter-provided macOS evidence rather than a live reproduction from this review.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right implementation path if OpenClaw intends to keep legacy iMessage DM history support, but a docs-honest removal is also coherent and the prior docs-only PR was closed unmerged.
