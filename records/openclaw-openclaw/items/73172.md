## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit` and applies it to generic stored-transcript replay, but the iMessage monitor has no decoded platform DM-history/backfill path; related docs-only PR #73775 closed unmerged and did not resolve the contract.

## What This Changes

The issue asks iMessage direct-message history configured with `dmHistoryLimit` to include decoded outbound message text instead of blank entries after restart or compaction.

## Best Possible Solution

Define the iMessage DM-history contract explicitly: either add a decoded imsg platform-history/backfill adapter with macOS coverage, or document that iMessage lacks Messages.app backfill while preserving the existing stored-transcript limiter.

## Reproduction Assessment

Yes for the current-main gap: source inspection shows iMessage config/docs expose `dmHistoryLimit`, while the iMessage monitor and `InboundHistory` construction are group-only. The original `attributedBody` database symptom remains plausible but still needs live macOS/imsg verification.

## Solution Assessment

Unclear. A decoded imsg adapter is the best fix if iMessage platform history is intended, while a narrower docs correction is safer if maintainers only want to support generic stored-transcript limiting.
