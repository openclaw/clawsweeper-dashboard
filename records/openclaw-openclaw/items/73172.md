## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, but the iMessage plugin only builds pending history for group messages and does not implement a decoded direct-message history adapter; the maintainer comment already reframed the original `attributedBody` theory into this source-backed contract gap, and the related docs-only PR #73775 was closed unmerged.

## What This Changes

The issue requests iMessage direct-message history injection to preserve decoded outbound text, including Messages content stored in `attributedBody`, instead of producing blank/null outbound history entries.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in the iMessage plugin using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while preserving schema compatibility for existing configs.

## Reproduction Assessment

Yes for the current-main contract gap: `channels.imessage.dmHistoryLimit` is accepted and documented, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The original `attributedBody` blank-outbound claim remains useful implementation guidance but was not live-reproduced in this read-only Linux review.

## Solution Assessment

Unclear. A decoded `imsg` history/RPC adapter is the right runtime fix if OpenClaw intends to support legacy iMessage DM history, while a docs-honest removal is also coherent; #73775 taking the docs path was closed unmerged, so this needs maintainer direction.
