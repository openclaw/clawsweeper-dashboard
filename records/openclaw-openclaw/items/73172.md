## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, but the iMessage runtime only builds pending history for group traffic; the original `attributedBody` detail remains useful for the eventual implementation but the current actionable bug is the advertised DM-history support mismatch.

## What This Changes

Make iMessage `dmHistoryLimit` provide decoded direct-message history instead of blank or missing outbound conversation context.

## Best Possible Solution

Make the iMessage DM-history contract honest: implement decoded direct-message history through verified `imsg` history/RPC output, or remove iMessage from the supported DM-history docs while preserving schema compatibility.

## Reproduction Assessment

Yes for the current-main gap: configure `channels.imessage.dmHistoryLimit`, then source inspection shows the value is accepted and documented but never used by the iMessage monitor for direct-message `InboundHistory`. The original database-level `attributedBody` detail still needs a macOS/imsg live check.

## Solution Assessment

Unclear. A decoded `imsg` history/RPC adapter is the right fix if legacy iMessage DM history is intended; a docs/support correction is the narrower maintainable path if it is not intended.
