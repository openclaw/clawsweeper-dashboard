## Summary

Keep open. Current main still exposes iMessage `dmHistoryLimit` in config and docs while the iMessage runtime only builds pending history for group traffic; the original `attributedBody` diagnosis remains useful, but the current actionable gap is the advertised DM-history support mismatch.

## What This Changes

The issue requests decoded iMessage direct-message history for `dmHistoryLimit` so outbound agent messages are not injected as blank context.

## Best Possible Solution

Make the iMessage contract honest: either implement decoded direct-message history through verified `imsg` history/RPC output, or remove iMessage from the documented DM-history support list while preserving schema compatibility.

## Reproduction Assessment

Yes for the current-main gap: configure `channels.imessage.dmHistoryLimit`, then source inspection shows the value is accepted and documented but never used by the iMessage monitor for direct-message `InboundHistory`. The original database-level `attributedBody` claim still needs a macOS/imsg live check.

## Solution Assessment

Unclear. A decoded `imsg` adapter is the right solution if legacy iMessage DM history is intended, while a docs/support correction is the narrower maintainable path if it is not intended.
