## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, but the iMessage runtime only builds pending history for group conversations and does not implement a decoded direct-message history adapter. The maintainer discussion already reframed the report as an iMessage contract gap, and the related docs-only PR #73775 was closed unmerged, so no runtime or docs resolution has landed.

## What This Changes

The issue asks OpenClaw to make iMessage direct-message history honor `dmHistoryLimit` without injecting blank outbound history entries from undecoded macOS Messages data.

## Best Possible Solution

Make the iMessage contract explicit: either add a testable iMessage direct-message history adapter that uses decoded `imsg` history/RPC output and covers outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and the schema/docs/generic session limiter recognize it, but the iMessage monitor only builds `InboundHistory` for group decisions. The exact macOS `attributedBody` blank-outbound path was not live-reproduced here and appears to describe an implementation hazard for any eventual adapter rather than a current direct DB reader.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right runtime direction if OpenClaw intends to support legacy iMessage DM history, but the docs-honest removal path is also coherent and the prior docs-only PR was closed unmerged.
