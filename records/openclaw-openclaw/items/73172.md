## Summary

Keep open. Current main still has an iMessage contract/runtime mismatch: `dmHistoryLimit` is accepted and documented for iMessage, but the iMessage monitor only builds pending history context for group traffic; the original `attributedBody` note remains important implementation guidance rather than evidence of an already-fixed path.

## What This Changes

The issue asks iMessage direct-message history configured with `dmHistoryLimit` to include decoded outbound text instead of blank entries after restart or compaction.

## Best Possible Solution

Make the iMessage contract explicit: either implement decoded direct-message history in `extensions/imessage` using verified `imsg` history/RPC output, or remove iMessage from documented DM-history support while preserving backward-compatible config parsing.

## Reproduction Assessment

Yes for the reframed current-main gap: source inspection shows iMessage config/docs expose `dmHistoryLimit`, while the runtime history key and `InboundHistory` construction are group-only. The original database-level `attributedBody` symptom still needs macOS/imsg live verification.

## Solution Assessment

Unclear. A decoded `imsg` adapter is the right path if legacy iMessage DM history is intended, but a docs/contract correction is narrower if maintainers do not want to support that legacy behavior.
