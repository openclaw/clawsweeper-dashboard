## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, while the iMessage monitor only uses group `historyLimit` and only populates `InboundHistory` for groups; the original `attributedBody` evidence remains relevant to any implementation, but the current actionable gap is still unresolved.

## What This Changes

Fix iMessage `dmHistoryLimit` so direct-message history context contains decoded outbound message text instead of blank or missing history entries.

## Best Possible Solution

Make the iMessage DM-history contract honest: either implement a decoded `imsg` history/RPC adapter with outbound attributed-body coverage, or remove iMessage from the supported DM-history docs while preserving config compatibility.

## Reproduction Assessment

Yes for the current-main contract mismatch: source inspection shows `channels.imessage.dmHistoryLimit` is accepted and documented, but iMessage direct-message `InboundHistory` is never built. The exact outbound `attributedBody` decoding claim still needs a macOS/imsg live check.

## Solution Assessment

Unclear. A decoded `imsg` adapter is the best fix if iMessage DM history is intended; otherwise the narrower maintainable solution is a docs/support correction, and current main has neither.
