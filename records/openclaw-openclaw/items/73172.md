## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, but the iMessage monitor only builds pending history for group messages; the original `attributedBody` detail remains useful for a future decoded-history implementation, but the current actionable gap is the advertised DM-history contract mismatch.

## What This Changes

The issue asks for iMessage direct-message history injection to use decoded message text instead of blank outbound history entries when `dmHistoryLimit` is configured.

## Best Possible Solution

Make the iMessage contract honest: either implement decoded direct-message history through verified `imsg` history/RPC output, or remove iMessage from the documented DM-history support list while preserving schema compatibility.

## Reproduction Assessment

Yes for the current-main support mismatch: configure `channels.imessage.dmHistoryLimit`, then source inspection shows the value is accepted and documented but never used by the iMessage monitor for direct-message `InboundHistory`. The original database-level `attributedBody` claim still needs a macOS/imsg live check.

## Solution Assessment

Unclear. A decoded `imsg` adapter is the right solution if legacy iMessage DM history is intended, while a docs/support correction is the narrower maintainable path if it is not intended.
