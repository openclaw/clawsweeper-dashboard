## Summary

Keep open. Current main still exposes and documents iMessage `dmHistoryLimit`, while the iMessage monitor only computes and injects pending group history; direct-message decisions do not get a history key or `InboundHistory`. The maintainer discussion already reframed the original `attributedBody` theory into this current-main contract gap, and the related docs-only PR #73775 was closed unmerged, so neither runtime behavior nor docs have been resolved.

## What This Changes

The issue requests fixing iMessage direct-message history so `dmHistoryLimit` supplies decoded outbound message text instead of blank history entries.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in the iMessage plugin using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and the schema/docs/generic session-history limiter recognize it, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The original `attributedBody` database evidence remains useful implementation guidance but was not live-reproduced here.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right runtime direction if OpenClaw intends to support legacy iMessage DM history, but a docs-honest removal is also coherent and the prior docs-only PR was closed unmerged.
