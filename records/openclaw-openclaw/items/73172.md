## Summary

Keep open. Current main still has the contract/runtime gap described in the issue discussion: iMessage `dmHistoryLimit` is accepted and documented, but the iMessage monitor only builds pending history context for group messages. The original `chat.db`/`attributedBody` diagnosis remains useful implementation guidance, but current source does not show a direct iMessage DM history loader that would make this already fixed, and related PR #73775 is closed unmerged.

## What This Changes

The issue requests fixing iMessage direct-message history so `dmHistoryLimit` supplies decoded outbound text instead of blank content when macOS stores message bodies outside the `text` column.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in `extensions/imessage` using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the reframed current-main gap: source inspection shows iMessage config/docs expose `dmHistoryLimit`, while the iMessage monitor only records and injects group pending history. The original database-level blank outbound text symptom still needs a macOS/imsg live check to reproduce directly.

## Solution Assessment

Unclear. The requested decoded `imsg` history/RPC direction is plausible if legacy iMessage DM history is intended, but a docs-contract correction is also coherent and maintainers have not chosen between those paths.
