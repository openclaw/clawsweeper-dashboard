## Summary

Keep open. Current main still exposes and documents iMessage `dmHistoryLimit`, while the iMessage runtime only builds pending history for groups. The maintainer discussion already reframed the report into this contract/runtime gap, and the related docs-only PR #73775 is closed unmerged, so neither runtime behavior nor docs have been resolved.

## What This Changes

The issue requests that iMessage `dmHistoryLimit` direct-message history use decoded `imsg` history/RPC output instead of `chat.db.message.text`, so outbound Messages content is not injected as blank context.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in `extensions/imessage` using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the reframed current-main gap: source inspection shows the config/schema/docs accept iMessage `dmHistoryLimit`, but the iMessage monitor only computes group `historyLimit` and only emits `InboundHistory` for group decisions. The original `attributedBody` database behavior is reporter-supplied and was not live-reproduced here.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right runtime direction if legacy iMessage DM history is intended, but a docs-honest removal is also coherent and the prior docs-only PR was closed unmerged.
