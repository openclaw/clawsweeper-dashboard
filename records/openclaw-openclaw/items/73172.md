## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: config and docs expose iMessage `dmHistoryLimit`, but the iMessage runtime only builds pending `InboundHistory` for group traffic, and the related docs-only PR was closed unmerged.

## What This Changes

The issue asks for iMessage direct-message history under `dmHistoryLimit` to include decoded outbound agent text instead of blank content from raw Messages database rows.

## Best Possible Solution

Make the iMessage contract honest: either implement decoded direct-message history through verified `imsg` history/RPC output, or remove iMessage from the documented DM-history support list while keeping schema compatibility.

## Reproduction Assessment

Yes for the current-main gap: configure `channels.imessage.dmHistoryLimit`, then source inspection shows the value is accepted and documented but never used by the iMessage monitor for direct-message `InboundHistory`. The original database-level `attributedBody` claim still needs macOS/imsg live verification.

## Solution Assessment

Unclear. A decoded `imsg` adapter is the right end state if iMessage DM history is intended, while a docs/support correction is narrower if maintainers do not want to support that runtime behavior.
