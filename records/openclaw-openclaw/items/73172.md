## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, and the generic session-history limiter treats `imessage` as supported, but the iMessage plugin monitor only computes group history and only populates `InboundHistory` for group decisions. The maintainer discussion already reframed this as an unresolved iMessage contract gap, and the related docs-only PR #73775 was closed unmerged, so no runtime or docs resolution has landed.

## What This Changes

The issue asks for iMessage `dmHistoryLimit` to inject decoded direct-message history so outbound Messages content stored in `attributedBody` does not become blank in new agent sessions.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in the iMessage plugin using verified `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and the schema/docs/generic limiter recognize it, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The reporter's exact `attributedBody` database evidence remains useful implementation guidance but was not live-reproduced here.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right runtime direction if OpenClaw intends to support legacy iMessage DM history, but a docs-honest removal is also a coherent product choice and the prior docs-only PR was closed unmerged.
