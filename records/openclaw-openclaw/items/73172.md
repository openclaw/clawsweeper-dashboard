## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, and the generic history limiter recognizes iMessage, but the iMessage runtime only builds pending group history from watch payloads; direct-message dispatch has no history key and no `InboundHistory`. The existing maintainer comment and closed docs-only PR leave an unresolved product choice rather than a closeable implemented or duplicate state.

## What This Changes

The issue requests that iMessage direct-message history injected through `dmHistoryLimit` include decoded outbound message text instead of blank entries.

## Best Possible Solution

Make the iMessage contract explicit: either implement a decoded direct-message history adapter in the iMessage plugin using `imsg` history/RPC output with outbound-text regression coverage, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and source inspection shows config/docs/generic tests recognize it, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The reporter's exact `attributedBody` database mechanism is plausible evidence for an implementation, but it was not live-reproduced here.

## Solution Assessment

Unclear. A decoded `imsg` history/RPC adapter is the right runtime fix if OpenClaw intends to support legacy iMessage DM history; otherwise the safer maintainable solution is the docs-honest removal path attempted in #73775.
