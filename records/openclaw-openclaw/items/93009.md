## Summary

Keep open. The updated branch fixes the direct already-wrapped case when source metadata is still present, but it still misses the normalized-wrapper shape from the linked report: normalization can leave the enumerable wrapper marker on the replacement object while dropping the non-enumerable source/context metadata, so the new fallback returns a stale wrapper instead of rebuilding it with the current run context.

## What This Changes

The PR changes `wrapToolWithBeforeToolCallHook` to unwrap and rewrap already wrapped tools when source metadata is present, and adds a focused e2e regression test for direct rewrapping.

## Best Possible Solution

Preserve before_tool_call source/context metadata through `normalizeToolParameters`, then rebuild or dedupe exactly one wrapper with the current `createOpenClawCodingTools` context, with a normalized-wrapper regression test and redacted live cron or isolated-agent proof.

## Reproduction Assessment

Yes from source: an already wrapped tool that passes through `normalizeToolParameters` can retain the enumerable wrapper marker while losing source/context metadata, then reach the PR's fallback path. I did not run the live cron or isolated-agent scenario in this read-only review.

## Solution Assessment

No. The direct rewrap path is a useful partial fix, but the PR does not handle the normalized replacement object that the linked issue describes; the safer fix preserves metadata through normalization and rebuilds one current-context wrapper.
