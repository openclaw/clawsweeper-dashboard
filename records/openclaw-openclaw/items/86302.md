## Summary

Keep open: current main still has the reported source-level bug, but this PR's cumulative `lightHits` filter can preserve stale candidates and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a memory-core phase-signal reader and filters REM dreaming input to keys with `lightHits > 0`, falling back to all entries when no light-staged keys are found.

## Best Possible Solution

Filter REM to same-sweep light-staged keys or an explicitly timestamp-bound staged set, preserve the standalone REM fallback, and add a stale-high-recall versus fresh-light-staged regression test.

## Reproduction Assessment

Yes, at source level: current `runRemDreaming()` passes the full lookback set into REM selection, while phase-signal recording separately tracks cumulative hits and timestamps. I did not run a live multi-day workspace sweep in this read-only review.

## Solution Assessment

No. The PR changes the right runtime path, but filtering on cumulative `lightHits > 0` is not a current light-stage boundary and can keep the reported stale-entry competition alive.
