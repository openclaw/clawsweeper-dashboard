## Summary

Keep open. The PR addresses a real memory-core handoff bug, but the branch still reads cumulative phase-signal state: any entry with a historical `lightHits > 0` remains eligible, so old previously staged high-recall entries can still beat the freshly staged set the linked issue is about; the PR also lacks after-fix real behavior proof.

## What This Changes

The branch adds a light-staged key reader and filters `runRemDreaming()` input to keys with `lightHits > 0`, falling back to all entries when no light-staged keys exist.

## Best Possible Solution

Filter REM to same-sweep light-staged keys, or an explicitly timestamp-bound staged set, preserve an intentional fallback for standalone REM paths, and add a stale-high-recall versus fresh-light-staged regression test.

## Reproduction Assessment

Yes, at source level. Current main passes all REM lookback entries into candidate selection, and the PR source shows the proposed fix still admits every historical `lightHits > 0` key; I did not run a live multi-day workspace sweep in this read-only review.

## Solution Assessment

No. The PR changes the right runtime path, but the helper should use same-sweep light staging or `lastLightAt`/timestamp-bounded state rather than cumulative hit count alone.
