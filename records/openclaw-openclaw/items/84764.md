## Summary

Keep open: the streaming MiniMax fix is focused and has sufficient live proof, but it is not merge-ready because it caps existing user-configured music-generation timeouts above 300s and can still fail long MiniMax generations before the configured deadline.

## What This Changes

This PR changes the bundled MiniMax music provider to request streaming hex responses, decode SSE/audio frames into an MP3 asset, keep JSON URL/inline fallbacks, add provider tests, and add a changelog entry.

## Best Possible Solution

Land the streaming MiniMax provider fix after preserving explicit `req.timeoutMs` values for the streaming POST and adding regression coverage for a configured timeout above 300s.

## Reproduction Assessment

Yes. The current-main failure path is source-reproducible from the 120s default POST timeout and the linked issue/proof supplies live MiniMax timeout evidence; the PR-specific regression is source-reproducible from the timeout helper contract.

## Solution Assessment

No, not yet. Streaming hex decoding is the right direction for MiniMax, but the implementation must preserve caller-configured operation timeouts instead of capping explicit values above 300s.
