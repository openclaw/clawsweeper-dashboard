## Summary

Keep open. The bug is source-reproducible and this PR targets the right Matrix fetch boundary, but the patch still decides synthetic success from original request metadata after redirects and lacks inspectable real behavior proof.

## What This Changes

The PR adds a Matrix transport helper, tests, and changelog entry to rewrite duplicate one-time-key `/keys/upload` 400 responses into synthetic successful empty-count responses.

## Best Possible Solution

Constrain the workaround to the effective final POST `/keys/upload` response, preserve every other Matrix error, add redirect regression coverage, and attach redacted live proof from an affected account.

## Reproduction Assessment

Yes, at source level. The related issue includes an exact duplicate OTK `/keys/upload` 400, current main passes that non-OK response through, and matrix-js-sdk treats non-429 4xx Matrix errors as non-retryable.

## Solution Assessment

No, not as currently written. The guarded-fetch boundary is the right narrow mitigation point, but the synthetic success check must use the effective final request after redirects before the patch is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Match collisions against the final request:** `extensions/matrix/src/matrix/sdk/transport.ts:253-255`
  - body: `fetchWithMatrixGuardedRedirects` can follow redirects away from `/keys/upload` and can rewrite 301/302/303 POSTs to GET, but this check still uses the original URL and method after the final response returns. A redirected request can therefore turn a collision-shaped 400 from another final endpoint into a synthetic 200; match the final URL/effective method and add redirect pass-through coverage.
  - confidence: 0.86
