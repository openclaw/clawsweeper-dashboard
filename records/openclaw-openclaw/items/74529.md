## What This Changes

The PR adds a Matrix SDK helper and guarded-fetch hook that rewrites narrowly matched POST `/keys/upload` one-time-key collision 400 responses into a synthetic 200 with empty `one_time_key_counts`, plus focused tests and a Matrix changelog entry.

## Best Possible Solution

Land the narrow Matrix-owned transport mitigation if Matrix maintainers accept the temporary synthetic response semantics, keep matrix-org/matrix-rust-sdk#6520 as the durable upstream OTK ID-tracking fix, and require focused Matrix tests plus the changed-lane gate before merge.

## Reproduction Assessment

Yes for the failure mechanics: current main has no rewrite, matrix-js-sdk does not retry non-429 4xx responses, and #73480 plus matrix-org/matrix-rust-sdk#6520 provide the exact `/keys/upload` collision body. I did not independently reproduce against a live homeserver in this read-only pass.

## Solution Assessment

Yes, with Matrix maintainer sign-off: the proposed workaround is scoped to the Matrix SDK `fetchFn` path and leaves non-collision 400s and other endpoints untouched. The safer long-term solution remains fixing OTK ID generation/tracking upstream in matrix-rust-sdk.
