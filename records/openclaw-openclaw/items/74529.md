## Summary

Keep open: this PR addresses a real Matrix E2EE bootstrap failure still present on current main, but the patch still trusts the original request URL/method after redirects and lacks inspectable real behavior proof.

## What This Changes

This PR adds a Matrix guarded-fetch workaround that rewrites narrowly matched `/keys/upload` duplicate one-time-key 400 responses to a synthetic 200, with focused tests and a Matrix changelog entry.

## Best Possible Solution

Constrain the rewrite to the effective final Matrix request, add redirect regression coverage, and land only after redacted after-fix proof shows an affected account completes E2EE bootstrap.

## Reproduction Assessment

Yes, at source level: the related issue has exact duplicate OTK `/keys/upload` logs, current main passes non-OK guarded-fetch responses through unchanged, and matrix-js-sdk treats the 400 as fatal. I did not exercise a live affected homeserver/account in this read-only review.

## Solution Assessment

No as currently written. The Matrix guarded-fetch boundary is a reasonable narrow mitigation, but the matcher must use the final redirected URL/effective method and include redirect coverage before this is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Match collisions against the final redirected request:** `extensions/matrix/src/matrix/sdk/transport.ts:253-255`
