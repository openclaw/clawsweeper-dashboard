## Summary

Keep open. The Matrix duplicate OTK failure is source-reproducible and this PR targets the right transport boundary, but the current diff can synthesize success after redirects because it checks the original URL/method, and the claimed live Synapse validation is not backed by inspectable redacted proof.

## What This Changes

Adds Matrix guarded-fetch helpers, tests, and a changelog entry to rewrite duplicate one-time-key `/keys/upload` 400 responses into a synthetic 200 with empty `one_time_key_counts`.

## Best Possible Solution

Constrain the workaround to the effective final POST `/keys/upload` response after redirects, preserve every other Matrix error, add redirect regression coverage, and attach redacted live proof from an affected account.

## Reproduction Assessment

Yes, at source level. The related OpenClaw issue has an exact duplicate OTK `/keys/upload` 400, current main passes that non-OK response through the Matrix SDK fetch boundary, and matrix-js-sdk treats non-429 4xx responses as fatal.

## Solution Assessment

No, not as currently written. The guarded-fetch boundary is the right narrow mitigation point, but the success rewrite must match the effective final URL and method after redirects before it is safe to merge.
