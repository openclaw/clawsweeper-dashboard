## Summary

Keep open. The Matrix E2EE fix direction is plausible, but the current patch still has a redirect-scoping bug and lacks inspectable real behavior proof, so it should not merge yet.

## What This Changes

Adds a Matrix guarded-fetch workaround, tests, and changelog entry to synthesize a successful `/keys/upload` response for duplicate one-time-key collision 400s.

## Best Possible Solution

Constrain the workaround to the effective final POST `/keys/upload` response, preserve every other Matrix error, add redirect regression coverage, and attach redacted live proof from an affected Matrix account.

## Reproduction Assessment

Yes, at source level. The related OpenClaw issue includes an exact duplicate OTK `/keys/upload` 400, current main passes that non-OK response through, and matrix-js-sdk treats the 400 as fatal and non-retryable.

## Solution Assessment

No, not as currently written. The guarded-fetch boundary is the right narrow mitigation point, but the synthetic success check must use the effective final request after redirects before the patch is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Match collisions against the final request:** `extensions/matrix/src/matrix/sdk/transport.ts:254-255`
  - body: `fetchWithMatrixGuardedRedirects` can follow redirects away from `/keys/upload` and can rewrite 301/302/303 POSTs to GET, but this check still uses the original URL and method after the final response returns. A redirected request can therefore turn a collision-shaped 400 from another final endpoint into a synthetic 200; match the final URL/effective method and add redirect pass-through coverage.
  - confidence: 0.86
