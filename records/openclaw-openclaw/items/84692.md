## Label Justifications

## Summary

This issue should stay open: current main has some merge-preserving onboarding code, but ordinary onboard writes still opt out of size-drop rejection and there is no regression coverage proving that reruns preserve `agents.list` plus top-level `bindings`.

## What This Changes

Re-running `openclaw onboard` should preserve existing `agents.list` and top-level `bindings`, or fail before overwriting them.

## Best Possible Solution

Ordinary onboarding should be merge-preserving and fail closed on destructive config shrink, with explicit reset/import carve-outs and regression coverage for `agents.list` plus `bindings`.

## Reproduction Assessment

Unclear for the exact end-to-end empty-array failure on current main, because this was a read-only review. Source inspection does show the reported safety failure: ordinary onboard writes pass `allowConfigSizeDrop: true`, so a size-drop payload is not rejected by the generic config clobber guard.

## Solution Assessment

Yes: the maintainable fix is to preserve existing multi-agent routing state during ordinary onboarding and make destructive shrink paths explicit. Strict dangling-binding validation should be added only in a compatible way, or split if it would fail existing configs at startup.
