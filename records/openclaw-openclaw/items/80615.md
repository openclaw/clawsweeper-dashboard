## Summary

Keep open. The forward streaming migration is useful and the supplied real-behavior proof is adequate, but the PR changes reverse assistant scans from whole-transcript semantics to a fixed tail window, which can miss the latest assistant message in long sessions.

## What This Changes

The PR adds shared transcript streaming/tail helpers, migrates session and gateway transcript scans away from whole-file reads, adds helper tests, and updates the changelog.

## Best Possible Solution

Keep the forward streaming scans, but implement reverse transcript scanning as bounded chunks that continue backward until a decision or beginning of file, with regressions for data beyond the first tail window.

## Reproduction Assessment

Yes, source-reproducible. Current main uses whole-file transcript reads for the linked memory issue, and the PR's new false-negative path follows directly from putting an assistant entry before more than the tail window of later transcript rows.

## Solution Assessment

No, not yet. Forward streaming is the right direction, but fixed-size reverse scans are not the narrowest maintainable fix because they change existing latest-assistant and dedupe semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve full reverse transcript scans:** `src/config/sessions/transcript.ts:150`
  - body: These callers used to scan the whole transcript from the end, but the new helper only inspects the default 4 MiB tail. A long session with more than that much user/tool/malformed output after the last assistant entry will make `/tts latest` report no readable reply and can miss delivery-mirror/gap-fill dedupe. Please make the reverse scan walk backward in bounded chunks until it finds the decisive assistant/mismatch or reaches the beginning of the file, rather than treating the first tail window as authoritative.
  - confidence: 0.89
