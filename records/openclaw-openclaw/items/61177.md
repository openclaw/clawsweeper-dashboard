## Summary

Keep open: the PR is a plausible, narrow macOS exec-approval IPC hardening fix and current main still has unguarded socket writes, but this external PR still lacks after-fix real macOS behavior proof before merge.

## What This Changes

The PR adds a macOS `SO_NOSIGPIPE` descriptor guard to the exec approval socket client/server path and a Swift test that verifies the option with `getsockopt`.

## Best Possible Solution

Land a rebased, macOS-validated version that keeps the client and accepted-fd `SO_NOSIGPIPE` guard plus the regression test, while keeping broader exec approval UX changes out of scope.

## Reproduction Assessment

No, not as a high-confidence full app crash reproduction. Source inspection does show the current client and server response paths write to macOS Unix-domain sockets without `SO_NOSIGPIPE`, which matches Apple's documented failure mode for no-reader writes.

## Solution Assessment

Yes, as a patch direction. Guarding the client fd and each accepted server-side client fd is the narrow maintainable fix for this IPC failure mode; the extra listening-fd guard is harmless, though not essential.
