## Summary

Keep open: current main still lacks the socket descriptor hardening, and the PR is a focused, plausible fix with no blocking code-review finding, but it needs after-fix real behavior proof before merge.

## What This Changes

The PR adds macOS `SO_NOSIGPIPE` hardening to exec approval IPC socket descriptors and a Swift regression test that verifies the option with `getsockopt`.

## Best Possible Solution

Land a rebased, macOS-validated version that preserves the client and accepted-fd `SO_NOSIGPIPE` guard plus the regression test, while keeping broader exec approval UX changes out of scope.

## Reproduction Assessment

No, not as a full app crash reproduction. Source inspection shows current main writes to macOS Unix-domain sockets without `SO_NOSIGPIPE`, and Apple documents this socket option as the per-descriptor way to turn no-reader writes into `EPIPE` instead of `SIGPIPE`. ([developer.apple.com](https://developer.apple.com/library/archive/documentation/System/Conceptual/ManPages_iPhoneOS/man2/setsockopt.2.html?utm_source=openai))

## Solution Assessment

Yes, as a patch direction. Guarding the client fd and accepted server fds is the narrow maintainable fix for this IPC failure mode, and the extra listening-fd guard is harmless under Apple's compatibility guidance. ([developer.apple.com](https://developer.apple.com/library/archive/documentation/NetworkingInternetWeb/Conceptual/NetworkingOverview/CommonPitfalls/CommonPitfalls.html?utm_source=openai))
