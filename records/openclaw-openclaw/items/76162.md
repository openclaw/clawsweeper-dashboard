## Summary

Keep open. Current main still has the reported task-maintenance gap, and an open closing PR (#76199) is the right place to review and land the narrow fix rather than closing this issue early.

## What This Changes

The issue reports that failed `exec-approval-followup` CLI task records can stay `running` forever because maintenance treats the persistent main session as backing an ended embedded run.

## Best Possible Solution

Review and land a narrow task-maintenance fix, likely via PR #76199, that treats run-backed CLI followups as unbacked after their run context disappears while preserving intended fallback behavior for CLI tasks without run-liveness metadata.

## Reproduction Assessment

Yes. A focused current-main repro can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; source inspection shows maintenance will keep it `running`.

## Solution Assessment

Yes, with a scope constraint. The best fix belongs in `hasBackingSession`, but it should make run-backed CLI followups depend on live run context without broadly removing child-session liveness for CLI records that genuinely lack run metadata.
