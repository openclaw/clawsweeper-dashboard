## Summary

Keep open. Current main still has the reported CLI task-maintenance gap, and the previously linked fixing PR is closed unmerged, so this remains a narrow repair candidate.

## What This Changes

The issue asks for `exec-approval-followup` CLI task runs to be reconciled after their embedded run ends instead of staying `running` because the parent session still exists.

## Best Possible Solution

Update task maintenance so run-backed CLI follow-ups depend on live run context once run metadata is present, add an `agent:main:main` regression test, and preserve existing fallback behavior for CLI records that genuinely lack run-liveness metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current `hasBackingSession` will keep it backed after the grace period.

## Solution Assessment

Yes, with a scope constraint. The narrowest maintainable fix is in `hasBackingSession`: make run-backed CLI tasks stop falling through to persistent session existence after their run context disappears, with regression coverage for the exec follow-up shape.
