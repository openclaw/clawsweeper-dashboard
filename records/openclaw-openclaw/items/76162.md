## Summary

Keep open. Current `main` still has the CLI task-maintenance gap, and the issue is already paired with open closing PR #76199, so the right maintainer path is to review and land or close that PR rather than close this issue early.

## What This Changes

Reports that exec approval follow-up CLI task runs can remain `running` after their embedded run ends because maintenance treats the persistent `agent:main:main` session row as backing state.

## Best Possible Solution

Land a narrow task-maintenance fix, likely via PR #76199, that makes run-backed CLI follow-ups depend on live run context while preserving intended behavior for CLI records without run-liveness metadata.

## Reproduction Assessment

Yes. A focused current-main repro can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; source inspection shows maintenance will keep it `running`.

## Solution Assessment

Yes, with a scope constraint. The fix belongs in `hasBackingSession`, but it should make run-backed CLI follow-ups depend on live run context without broadly breaking CLI tasks that lack run metadata.
