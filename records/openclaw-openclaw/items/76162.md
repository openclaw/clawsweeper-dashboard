## Summary

Keep open. Current main still has the CLI task-maintenance gap described here, and the earlier closing PR is now closed unmerged, so this should be handled as a narrow repair rather than cleanup-close.

## What This Changes

The issue reports that exec approval follow-up CLI task runs can remain stuck in `running` because task maintenance treats a persistent parent session key as live backing state after the embedded run has ended.

## Best Possible Solution

Update task maintenance so run-backed CLI followups are marked `lost` after the grace period when their run context is gone, even if `childSessionKey` points at a persistent main session, and cover that exact task shape in tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row, then maintenance will keep it `running`. I did not execute a test because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `hasBackingSession` or its immediate CLI-liveness helper, with a regression test for the persistent main-session followup case; max-age or new configuration would be broader than needed.
