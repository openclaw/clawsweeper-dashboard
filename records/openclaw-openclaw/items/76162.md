## Summary

Keep open. Current main still has the reported CLI task-maintenance gap: inactive run-backed `cli` tasks can fall through to persistent session existence, and the related repair PRs are closed unmerged.

## What This Changes

The issue reports failed `exec-approval-followup` CLI task runs remaining `running` indefinitely because task maintenance treats the persistent parent session as live backing.

## Best Possible Solution

Land a narrow task-maintenance fix that makes run-backed CLI follow-ups depend on live run context after the grace period, with regression coverage for `agent:main:main` and aligned docs/changelog.

## Reproduction Assessment

Yes. A source-level repro can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current `hasBackingSession` will keep it active.

## Solution Assessment

Yes. The narrowest maintainable fix is in `hasBackingSession`: run-backed CLI tasks should not be rescued by persistent session existence once their live run context is gone.

## Repair Work Prompt

Fix the stale `exec-approval-followup` CLI task liveness bug tracked by https://github.com/openclaw/openclaw/issues/76162. Use closed PRs #76199 and #76656 as source context, but implement against current main. In `src/tasks/task-registry.maintenance.ts`, make run-backed `runtime: "cli"` tasks stop falling through to session-existence liveness once `hasActiveCliRun(task)` is false after the existing grace period; preserve intended behavior for CLI tasks that lack authoritative run metadata and for detached media/tool tasks that are still refreshed or have active run context. Add regression coverage for a stale `cli` task with `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a persistent session-store row being marked `lost`, plus a live-run control case. Update `docs/automation/tasks.md` only if wording needs to clarify main-session follow-up liveness, and add a single-line user-facing `CHANGELOG.md` fix entry. Do not broaden this into exec-followup routing, `/new` session rebinding, fallback wording/retry behavior, or bundle-mcp runtime disposal; those are separate issues.
