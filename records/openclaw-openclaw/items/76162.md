## Summary

Keep open. Current `main` still has the reported task-maintenance gap for inactive `cli` task runs whose `childSessionKey` points at the persistent main session, and the related repair PR #76199 is closed unmerged rather than a canonical open fix.

## What This Changes

The issue reports that failed `exec-approval-followup` CLI task runs can remain `running` indefinitely because task maintenance treats the persistent parent session as backing liveness.

## Best Possible Solution

Land a narrow task-maintenance fix that makes run-backed CLI follow-ups depend on active run context after the grace period, with regression coverage for `childSessionKey: "agent:main:main"` and aligned task docs/changelog.

## Reproduction Assessment

Yes. A focused source-level repro can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current `hasBackingSession` will keep it active instead of marking it lost.

## Solution Assessment

Yes. The narrowest maintainable fix is in `hasBackingSession`: once a CLI task has run-liveness metadata and `hasActiveCliRun` is false after the grace period, it should not be rescued by persistent parent-session existence.
