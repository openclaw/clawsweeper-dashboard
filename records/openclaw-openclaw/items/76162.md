## Summary

Keep open. Current main still has the reported CLI maintenance gap: once an `exec-approval-followup` run is no longer active, a persistent `agent:main:main` session row can still make `hasBackingSession` return true, so the sweeper will not mark the stale task lost.

## What This Changes

The issue reports that failed exec approval follow-up CLI task runs can stay `running` indefinitely because task maintenance treats the persistent parent session as proof that the specific run is still alive.

## Best Possible Solution

Teach task maintenance that CLI records with a concrete run id are backed by the live run context, not by persistent main-session existence, while preserving child-session fallback only for CLI records that genuinely lack run-liveness metadata.

## Reproduction Assessment

Yes. A focused test can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current main will keep it `running` because `findTaskSessionEntry` returns the persistent main session.

## Solution Assessment

Yes. The narrow maintainable fix is in `hasBackingSession`: after checking `hasActiveCliRun`, CLI tasks with run metadata should become unbacked when that run context is gone, with regression coverage for `exec-approval-followup` and preservation tests for active run contexts and legacy child-session cases.
