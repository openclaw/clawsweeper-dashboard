## Summary

Keep open. Current main still has the reported CLI task-maintenance gap: a stale run-backed CLI task with `childSessionKey: "agent:main:main"` can be treated as backed by the persistent session row, and PR #76199 is the open closing PR to review and land rather than closing this issue early.

## What This Changes

The issue asks for exec approval follow-up CLI task runs to stop remaining `running` forever after their embedded agent run ends or fails behind a persistent main session key.

## Best Possible Solution

Review and land a narrow task-maintenance fix, likely via PR #76199, that makes run-backed CLI follow-ups depend on live run context once the grace period expires while preserving intended behavior for CLI rows without run-liveness metadata.

## Reproduction Assessment

Yes. A focused current-main repro can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; source inspection shows maintenance will keep it active instead of marking it `lost`.

## Solution Assessment

Yes, with a narrow scope constraint. The maintainable fix belongs in `hasBackingSession`/task-maintenance liveness so run-backed CLI follow-ups stop relying on persistent session existence after their run context disappears.
