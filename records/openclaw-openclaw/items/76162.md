## Summary

Keep open. Current main still has the reported CLI task-maintenance liveness gap, and open PR #76199 references this issue with closing syntax, so the issue should stay open until that fix or an equivalent one merges.

## What This Changes

The issue asks task maintenance to stop treating ended `exec-approval-followup` CLI runs as alive merely because their persistent `agent:main:main` session row still exists.

## Best Possible Solution

Land a narrow task-maintenance fix that treats concrete CLI run liveness as authoritative for follow-up tasks, preserves any intentional legacy fallback, and adds regression coverage for `exec-approval-followup` with `childSessionKey: "agent:main:main"`.

## Reproduction Assessment

Yes. A focused test can seed an old running `runtime: "cli"` record with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current main will keep it running because the persistent session entry still counts as backing.

## Solution Assessment

Yes, directionally. The maintainable fix belongs in `hasBackingSession`, with tests for ended follow-up runs, active CLI runs, and any legacy CLI records that should still be treated as backed.
