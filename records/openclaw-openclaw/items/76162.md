## Summary

Keep open. Current main still has the reported CLI maintenance gap, and an open linked fix PR is the right vehicle to resolve it before this issue closes.

## What This Changes

Exec approval follow-up tasks should reconcile to terminal or lost status when their embedded CLI run is no longer active, rather than staying running because the persistent main session exists.

## Best Possible Solution

Review and land or adjust #76199 so CLI run-backed tasks are considered backed by the live run context, not by persistent main-session existence; after that merge, this issue can close as implemented.

## Reproduction Assessment

Yes. Seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current main will keep it `running` because `hasBackingSession` falls through to `findTaskSessionEntry`.

## Solution Assessment

Yes. The narrow maintainable fix is to make run-backed CLI task liveness depend on the active run context after `hasActiveCliRun`, while preserving separate ACP, subagent, cron, and legacy fallback behavior.
