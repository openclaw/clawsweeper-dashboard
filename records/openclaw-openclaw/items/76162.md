## Summary

Keep open. Current main still lets stale `runtime: "cli"` exec-approval follow-up task rows with `childSessionKey: "agent:main:main"` survive the maintenance sweep because the live-run check falls through to persistent session-row existence.

## What This Changes

Report that failed exec-approval follow-up runs can leave CLI task records stuck in `running`, blocking reload and saturating the gateway.

## Best Possible Solution

Make CLI maintenance treat rows with concrete run metadata as backed by live run context, not by persistent main-session existence, while preserving child-session fallback only for legacy CLI rows without run-liveness metadata.

## Reproduction Assessment

Yes. A focused test can seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current main will keep it `running` through the session-existence fallback.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `hasBackingSession` with regression coverage for inactive main-session follow-ups, active run contexts, and any legacy CLI fallback that must remain supported.
