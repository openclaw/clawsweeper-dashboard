## Summary

Keep open. Current main still lets inactive `runtime: "cli"` task runs fall through to persistent session-key backing, and open PR #76847 already references this issue with closing syntax, so the right cleanup path is to review and land or close that PR rather than close the issue early.

## What This Changes

The issue reports that exec approval follow-up CLI task runs can remain `running` after their embedded run ends because task maintenance treats the persistent main session as live backing state.

## Best Possible Solution

Review and land PR #76847, or an equivalent narrow maintenance fix, so run-backed CLI follow-ups are reconciled from live run context while preserving legitimate isolated child-session task behavior.

## Reproduction Assessment

Yes by source inspection, though I did not run a mutating test command. A stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row will keep `hasBackingSession` true and avoid `lost` reconciliation.

## Solution Assessment

Yes, with a scope constraint. The narrowest maintainable fix is in task maintenance for run-backed CLI liveness, and the existing open PR should be reviewed rather than starting a second repair branch.
