## Summary

Keep open: current main still lacks per-context memory-wiki vaults, but this PR is not merge-ready because no-context CLI/gateway paths can still use unresolved placeholder paths for filesystem writes, real behavior proof is missing, and the branch is currently conflicting with base.

## What This Changes

The PR adds context-aware `vault.path` template expansion for memory-wiki tools, refactors all wiki tools to factory registration, skips memory prompt/corpus supplements for placeholder paths, updates docs, and adds tests.

## Best Possible Solution

Land a rebased version that resolves templates only where trusted invocation context exists, fails closed on CLI/gateway/no-context writes, keeps any `memory_*` limitation intentional and documented, and includes redacted real-run proof of per-workspace vault isolation.

## Reproduction Assessment

Yes. By source inspection, configure `vault.path` as `{workspaceDir}/wiki` and invoke a no-context gateway or CLI write path such as `wiki.init`; that path passes unresolved config into `initializeMemoryWikiVault`, which recursively creates `config.vault.path`.

## Solution Assessment

No. Tool-side templating and the stub-tool catalog behavior are plausible, but the best fix must also fail closed on no-context CLI/gateway paths or define a maintainer-approved context contract for those surfaces.
