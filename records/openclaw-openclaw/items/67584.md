## Summary

Keep this PR open: current main still lacks ctx-aware memory-wiki vault path templating, and the branch contains useful work, but it is not merge-ready because real behavior proof is missing and the no-context CLI/gateway surfaces can still operate on unresolved placeholder paths.

## What This Changes

The PR adds ctx-aware `vault.path` template tokens for memory-wiki tools, converts all wiki tools to factory registration, skips memory prompt/corpus supplements for templated paths, documents the behavior, and adds tests.

## Best Possible Solution

Land a rebased version that resolves templates only where trusted context exists, fails closed on CLI/gateway/no-context filesystem writes, keeps any `memory_*` limitation intentional and documented, and includes redacted real-run proof of per-workspace vault isolation.

## Reproduction Assessment

Yes, by source inspection for the review finding: configure `vault.path` as `{workspaceDir}/wiki`, then invoke a no-context CLI or gateway write path such as `wiki.init`; it passes the unresolved config to `initializeMemoryWikiVault`, which calls recursive `fs.mkdir` on that path.

## Solution Assessment

No. The tool-side templating and stub-tool behavior are a plausible direction, but the best fix must also fail closed on no-context CLI/gateway paths or define a maintainer-approved context contract for those surfaces.
