## Summary

Keep open. The PR addresses useful Twitch/OpenProse behavior, but the current head still has a secret-handling blocker in Twitch token writeback, uses a deprecated config SDK barrel that current boundary guards reject for bundled plugin production code, and lacks required external real behavior proof.

## What This Changes

The PR adds Twitch OAuth refresh token writeback for config-backed accounts, new Twitch unit coverage, and else-chain fixes for one OpenProse fast-loop example.

## Best Possible Solution

Revise token persistence to preserve or skip env-authored token fields through the focused config-mutation surface, keep the OpenProse else-chain fix, add focused regression coverage, and include redacted runtime proof.

## Reproduction Assessment

Yes for the PR-introduced blocker by source inspection: a config-authored env token resolves into the write snapshot, then the PR mutates that resolved value so env-ref restoration treats the token path as intentionally changed. I did not run a live Twitch refresh in this read-only review.

## Solution Assessment

No. Token persistence is useful and the OpenProse branch fix is the right shape, but the writeback needs to preserve env-authored secrets and use the focused config mutation API before this is mergeable.

  - confidence: 0.88

  - body: The helper type allows missing `expiresIn` and `obtainmentTimestamp`, but absence clears `expiresIn` to `null` and synthesizes a new timestamp, making direct helper calls rewrite config even when no metadata actually changed. Either require the Twurple `AccessToken` shape or skip absent optional fields consistently.
