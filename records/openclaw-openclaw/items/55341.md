## Summary

Keep open: current main still lacks both the Twitch token writeback and the OpenProse exclusive-branch fix, but this PR needs changes before merge because the writeback path can turn env-authored Twitch token references into literal secrets and it uses a deprecated config-runtime barrel that current boundary checks reject.

## What This Changes

The PR adds Twitch OAuth refresh-token writeback and tests, and changes an OpenProse fast-loop example so terminal output branches are mutually exclusive.

## Best Possible Solution

Revise the PR so token writeback preserves or skips env-authored Twitch token fields, uses the narrow config-mutation API with explicit after-write intent, keeps the OpenProse else-chain fix, adds regression coverage, and includes redacted runtime proof.

## Reproduction Assessment

Yes for the PR-introduced blocker by source inspection: a config-authored env token resolves into `snapshot.config`, then the PR writes a changed token path that current env-ref restoration will not re-template. I did not run a live Twitch refresh in this read-only review.

## Solution Assessment

No. The direction is useful, but the implementation is not merge-ready until it preserves env-authored token references and uses the supported config-mutation subpath; the OpenProse branch fix itself is the right shape.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve authored env token references:** `extensions/twitch/src/token-writeback.ts:106`
  - body: This clones `snapshot.config`, which is already env-substituted, then writes a changed access-token path. If the user authored the token as `${OPENCLAW_TWITCH_ACCESS_TOKEN}` or another env reference, the refreshed OAuth token can be persisted literally because changed paths are not restored to env refs.
  - confidence: 0.93

  - body: This new production plugin file imports `openclaw/plugin-sdk/config-runtime`, but current main marks that broad barrel deprecated and the config-boundary guard flags it in extension code. Use `config-mutation` and `config-contracts` instead so the write has the supported after-write behavior.
  - confidence: 0.88
