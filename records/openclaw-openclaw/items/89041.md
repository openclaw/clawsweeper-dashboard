## Summary

Keep open for maintainer review: the patch is focused and proof-backed, but it intentionally disables ws 8.21 retained-part limits on Discord gateway sockets and needs explicit acceptance of that availability/security tradeoff before merge.

## What This Changes

The PR sets `maxBufferedChunks` and `maxFragments` to `0` on both Discord gateway WebSocket constructor paths and updates adjacent constructor-option tests.

## Best Possible Solution

Land this Discord-only constructor option mitigation if maintainers accept the retained-part tradeoff; otherwise switch to bounded nonzero caps backed by matching mechanism and live proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the dependency mechanism: ws 8.21.0 source and the PR's deterministic fragment repro show the default retained-part failure path and that `0` disables it; the exact production Discord fragmentation pattern was not forced on demand.

## Solution Assessment

Is this the best way to solve the issue? Yes, this appears to be the narrowest maintainable mitigation if maintainers accept Discord as a trusted gateway peer; pinning ws would remove the security fix globally, while arbitrary higher caps would need production traffic data to justify them.
