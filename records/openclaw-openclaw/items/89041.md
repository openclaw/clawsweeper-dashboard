## Summary

Keep open: current main still leaves both Discord gateway WebSocket constructor paths on ws 8.21.0 receiver defaults, and this PR is the focused candidate fix for the linked Discord gateway failure. The patch looks technically correct, but merging it intentionally disables retained-part guards for this trusted remote peer, so maintainer sign-off on that availability/security tradeoff is still the blocker.

## What This Changes

The PR sets `maxBufferedChunks: 0` and `maxFragments: 0` on both Discord gateway WebSocket constructor paths and adds constructor-option tests.

## Best Possible Solution

Land the two-path Discord gateway override after maintainer sign-off on the trusted-peer tradeoff, or require documented bounded Discord-specific caps if retained-part protection should remain enabled.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the dependency mechanism from source inspection: current main uses ws 8.21.0 defaults on the Discord gateway paths, and ws 8.21.0 raises `WS_ERR_TOO_MANY_BUFFERED_PARTS` when retained-part limits are exceeded. I did not run a live current-main OpenClaw Discord failure.

## Solution Assessment

Is this the best way to solve the issue? Yes as a focused mitigation if maintainers accept the trusted-peer tradeoff; it covers both Discord gateway constructors without downgrading `ws` globally. If maintainers want retained-part protection to remain enabled, bounded Discord-specific caps would be the safer alternative.
