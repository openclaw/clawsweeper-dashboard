## Summary

Keep open: this is a focused candidate fix for the linked Discord/ws 8.21 gateway issue, and current main plus the latest release still use ws defaults on both Discord gateway socket paths. The remaining blocker is maintainer acceptance of the scoped tradeoff: the PR restores pre-8.21 behavior for Discord by disabling retained-part guards on that remote gateway socket.

## What This Changes

The PR sets `maxBufferedChunks: 0` and `maxFragments: 0` on both Discord gateway WebSocket constructor paths and adds constructor-option regression tests.

## Best Possible Solution

Land the two-path Discord gateway override if maintainers accept the trusted-peer tradeoff, or require bounded Discord-specific caps if retained-part protection should remain enabled.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the dependency mechanism: current main uses ws 8.21 defaults on both Discord gateway paths, and ws 8.21 raises `WS_ERR_TOO_MANY_BUFFERED_PARTS` when retained-part limits are exceeded. I did not run a live current-main OpenClaw Discord failure.

## Solution Assessment

Is this the best way to solve the issue? Yes as a focused mitigation if maintainers accept the trusted-peer tradeoff; it covers both Discord gateway constructors without downgrading `ws` globally. If maintainers want retained-part protection to remain enabled, bounded Discord-specific caps are the safer alternative.
