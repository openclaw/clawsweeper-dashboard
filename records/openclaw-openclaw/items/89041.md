## Summary

Keep open for normal maintainer merge review: current main and v2026.6.1 still use ws 8.21.0 defaults on Discord gateway sockets, while this PR applies the targeted Discord-only receiver-limit override with focused tests and convincing supplied live/mechanism proof. ## What This Changes

The PR disables ws 8.21.0 `maxBufferedChunks` and `maxFragments` receiver limits for both Discord gateway WebSocket construction paths and updates focused tests to assert the options.

## Best Possible Solution

Land the targeted Discord-only ws option mitigation once normal CI and maintainer review are satisfied, keeping the scope limited to gateway sockets and preserving tests for both base and monitor construction paths.

## Reproduction Assessment

Yes, with source-backed confidence: ws 8.21.0 finite retained-part defaults are on the current Discord gateway path, and the supplied deterministic repro streams one message over the default fragment limit to trigger `WS_ERR_TOO_MANY_BUFFERED_PARTS`. I did not rerun the live Discord proof locally because this read-only review cannot safely use credentials.

## Solution Assessment

Yes; the PR is the narrowest maintainable mitigation I found because it changes only Discord gateway WebSocket construction and leaves other ws users on the upstream security defaults. A global ws downgrade would be broader, and arbitrary higher limits would be harder to justify without production traffic data.
