## Summary

Keep open for maintainer review: the patch is narrow and appears correct, and the supplied real-behavior proof is sufficient, but it intentionally disables the new ws retained-part caps on Discord gateway sockets so a maintainer should explicitly accept that availability/security tradeoff before merge.

## What This Changes

This PR passes `maxBufferedChunks: 0` and `maxFragments: 0` to both Discord gateway WebSocket constructor paths and updates adjacent tests/assertions.

## Best Possible Solution

Land this Discord-only constructor option change if maintainers accept the retained-part cap tradeoff; otherwise switch to a bounded nonzero limit backed by matching mechanism and live proof.

## Reproduction Assessment

Yes, source-level: current main and the latest release use ws 8.21.0 on the Discord gateway constructors, and upstream ws 8.21.0 source/docs show the default retained-part caps and that `0` disables the checks. This review did not rerun the live Discord scenario, but the PR body supplies redacted live patched logs plus a deterministic fragment-limit repro.

## Solution Assessment

Yes, this is the narrowest mitigation I found: it scopes the opt-out to Discord gateway sockets instead of downgrading `ws` or changing all WebSocket users. It still needs maintainer acceptance because disabling a security-motivated limit is an intentional availability/memory tradeoff.
