## Summary

Keep open for maintainer decision: the patch is focused and the dependency proof supports the mitigation, but merging intentionally disables the `ws` 8.21 retained-part caps on Discord gateway sockets, so the remaining question is accepting that availability/resource tradeoff.

## What This Changes

The PR adds `maxBufferedChunks: 0` and `maxFragments: 0` to both Discord gateway WebSocket constructor paths and updates focused tests to assert those options.

## Best Possible Solution

Make an explicit maintainer choice between this Discord-only trusted-peer mitigation and measured finite caps before landing the linked bug fix.

## Reproduction Assessment

Yes. The current code path and upstream `ws` 8.21 source make the retained-part failure source-reproducible, and the PR body supplies deterministic before/after logs plus a live patched Discord READY/heartbeat run.

## Solution Assessment

This is an acceptable narrow mitigation, not a settled maintainer policy choice. It is better scoped than downgrading `ws` globally because it only changes the two Discord gateway constructors, but maintainers still need to choose disabled caps versus finite measured caps.
