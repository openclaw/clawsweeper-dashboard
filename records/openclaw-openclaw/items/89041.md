## Best Possible Solution

Land the targeted Discord-only mitigation once maintainers accept the trusted-peer availability/resource tradeoff; otherwise replace the disabled limits with maintainer-chosen finite caps.

## Solution Assessment

Is this the best way to solve the issue? Yes as a narrow mitigation: it targets only the two Discord gateway constructors instead of downgrading ws globally or changing unrelated WebSocket users, with the remaining maintainer choice being disabled limits versus finite caps.
