## Summary

Keep open. Current main still lacks tool-level post-write verification for sandbox write/edit and non-workspace host write/edit paths, and this PR targets the right adapter layer, but the submitted patch needs rework before merge because edit recovery can swallow verifier failures and the host hunk is stale against current tilde-aware path handling.

## What This Changes

The PR adds post-write existence, file-type, and byte-size checks to sandbox write/edit and non-workspace host writes, plus a new regression test file for host and sandbox write verification.

## Best Possible Solution

Refresh or replace this PR with a focused adapter-level fix: add shared non-recoverable post-write verification for sandbox and non-workspace host write/edit operations, preserve current tilde expansion, keep writeFileWithinRoot as the workspace-only verifier, add edit-path and write-path regressions, and coordinate but not duplicate #44662's lower-level byte-count hardening.

## Reproduction Assessment

Yes for the code-path class: a mocked bridge or host helper whose writeFile resolves while the file is missing, the wrong type, or the wrong size exercises the false-success path because upstream reports success after ops.writeFile resolves. The original Ubuntu/DeepSeek session was not live-reproduced in this read-only review.

## Solution Assessment

No, not as submitted. Adapter-level post-write verification is the right fix direction, but the patch should prevent edit recovery from converting verifier failures back to success, rebase over current tilde-aware host path handling, and add edit-path regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not recover write verification failures:** `src/agents/pi-tools.read.ts:763-776`
  - body: `createSandboxedEditTool` and host edit tools are wrapped by `wrapEditToolWithRecovery`, which catches post-write errors and can return success when readback suggests the edit applied. Because these new verifier failures are plain `Error`s, a stat/type/size failure in an edit path can still be converted into `Successfully replaced...`, defeating the PR's guarantee. Mark verifier failures as non-recoverable or teach the recovery wrapper to rethrow them, then cover the edit path.
  - confidence: 0.86
- **[P2] Rebase the host verifier over tilde expansion:** `src/agents/pi-tools.read.ts:787-788`
  - body: This host hunk is based on the pre-tilde `writeHostFile` implementation using `path.resolve(absolutePath)`, while current main resolves `expandTildeToOsHome(absolutePath)`. As submitted, the patch does not apply cleanly; rebase it and add verification after the existing tilde-aware resolution so non-workspace `~` write/edit paths keep working.
  - confidence: 0.91
