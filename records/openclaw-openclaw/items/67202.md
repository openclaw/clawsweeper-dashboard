## Summary

Keep open. Current main still lacks post-write readback verification on the affected adapter paths, and this PR is a good direction, but the submitted patch is stale against current host tilde handling and can still let edit recovery convert verifier failures back into success reports.

## What This Changes

The PR adds post-write existence, file-type, and byte-size verification to sandbox write/edit and non-workspace host writes, plus regression tests for host and sandbox write operations.

## Best Possible Solution

Refresh this into a focused adapter-layer fix that preserves tilde-aware host paths, uses a shared non-recoverable verifier for sandbox and non-workspace host writes/edits, adds edit-path regressions, and leaves lower-level atomic byte-count hardening to #44662.

## Reproduction Assessment

Yes for the code-path class: a mocked bridge or host write operation that resolves while stat/readback reports missing, wrong type, or wrong size reproduces the false-success path because upstream success follows ops.writeFile resolution. The original live Ubuntu/DeepSeek transcript was not replayed in this read-only review.

## Solution Assessment

No, not as submitted. Adapter-level post-write verification is the right fix direction, but this patch must rebase over current tilde resolution and prevent edit recovery from masking verifier failures before it is the narrow maintainable solution.
