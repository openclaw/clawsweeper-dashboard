## Label Justifications

- `P1`: The issue reports a current-release runtime workflow silently calling the wrong model/provider after compaction.
- `impact:session-state`: The failure is session model override state drifting from the runtime compaction path.
- `impact:auth-provider`: The observed result is API routing to the wrong provider/model after a user-selected model switch.

## Summary

Keep open: current main and v2026.6.9 still route timeout/overflow compaction through the run-start `provider`/`modelId` after a live `/model` switch when the attempt cannot restart cleanly, and the linked fixes are still open rather than merged.

## What This Changes

Preserve the most recent `/model` session selection through compaction instead of letting compaction use a previous provider/model.

## Best Possible Solution

Land one focused run-loop fix that feeds the pending live model selection into timeout and overflow compaction runtime context, with regression coverage for the pending-switch compaction path.

## Reproduction Assessment

Yes at source level: set `liveModelSwitchPending` via `/model`, then hit timeout or overflow compaction after mid-turn work makes `canRestartForLiveSwitch` false; current main still builds compaction context from stale `provider`/`modelId`. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

Yes, the best fix appears to be a narrow `run.ts` compaction-context repair plus regression test, not a new config option or broad model-selection policy change.
