## Label Justifications

- `impact:session-state`: The reported failure is session model override state drifting from the runtime model used by compaction.
- `impact:auth-provider`: The stale compaction path can route work through a previous provider/model selection after a provider switch.

## Summary

Keep open: current main and the latest release still route timeout/overflow compaction through the run-start provider/model after a pending live `/model` switch, and the linked fix PRs are open rather than merged.

## What This Changes

The issue reports that automatic compaction can use a previous provider/model after a mid-session `/model` switch while the session UI shows the newest selected model.

## Best Possible Solution

Land one focused fix that carries the live provider/model/auth selection into timeout and overflow compaction, with regression coverage for a pending `/model` switch before compaction.

## Reproduction Assessment

Yes, at source level: current main sets `requestedSelection` for a pending live model switch, then timeout and overflow compaction still build context from the prior `provider`/`modelId` locals when the attempt cannot restart cleanly. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow runner/compaction-context repair that treats provider, model, and auth profile as one live selection. Broad model-selection policy changes are riskier unless they also prove the actual compaction runtime route changes.
