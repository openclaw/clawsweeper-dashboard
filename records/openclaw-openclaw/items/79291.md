## Summary

Keep open. Current main still uses weighted signal and age thresholds plus live daily-file rehydration, then writes the rehydrated snippet verbatim to `MEMORY.md`; there is no current-config applicability check or troubleshooting-log staleness gate, and the two linked reports remain open rather than superseding this narrower stale-config case.

## What This Changes

Request staleness-aware deep-memory promotion so outdated troubleshooting/config snippets are not written verbatim to `MEMORY.md`.

## Best Possible Solution

Define the promotion-quality policy for ephemeral troubleshooting and config claims, then implement it in `memory-core` deep ranking/apply with focused regression tests and docs, coordinating with the two related open memory-promotion reports.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The source path is clear: a qualifying short-term candidate whose stale text still exists in the daily note can pass ranking and be written verbatim without checking current config.

## Solution Assessment

Unclear. The requested directions are plausible, but the best maintainable fix needs maintainer policy on whether to distill snippets, classify ephemeral troubleshooting logs, validate config references, or combine those approaches.
