## Summary

Keep open. Current main still lacks the synthetic-placeholder handling, and this PR is the active candidate fix, but the latest patch has one source-reproducible blocker: it can classify a run as `empty_result` even when a later Claude result carries visible text.

## What This Changes

The PR detects Claude CLI `<synthetic>` placeholder assistant records, carries a `syntheticPlaceholder` marker into CLI run metadata, classifies it as fallback-worthy `empty_result`, and adds focused parser, runner, and classifier tests.

## Best Possible Solution

Land a narrowed fix that classifies synthetic placeholders only when the final run has no visible output, preserves legitimate `NO_REPLY`, and keeps the maintainer-approved fallback semantics explicit.

## Reproduction Assessment

Yes. The linked bug is source-reproducible on current main: a Claude `<synthetic>` assistant record followed by an empty result is normalized as plain empty output with no non-GPT fallback classification. The PR blocker is also source-reproducible from the diff: a synthetic marker preserved onto a later non-empty result reaches the new classifier branch before visible-output guards.

## Solution Assessment

No, not yet. Parser-to-runner metadata is the right boundary, but the classifier must not treat a marked run as empty when the same run has visible final output; maintainers also need to accept the fallback semantics for true no-output placeholders.
