## Label Justifications

- `P1`: The source-backed Feishu race can leave real channel accounts unable to receive messages until a gateway restart.
- `impact:message-loss`: The reported failure mode is silently dropped Feishu messages when bot identity remains unknown and recovery is skipped.

## Summary

Keep open. Current main and v2026.6.8 still contain the abort-gated Feishu bot identity recovery path, and the open linked PR is the candidate fix but has not merged.

## What This Changes

The issue reports a Feishu WebSocket lifecycle race where unknown bot identity recovery is skipped if the startup abort signal has already fired.

## Best Possible Solution

Land a Feishu-local fix that schedules recovery for the active lifecycle without allowing stale aborted probes to keep running or overwrite identity state after stop/reload.

## Reproduction Assessment

Yes, at source level: a missing bot identity followed by an already-aborted lifecycle signal skips `startBotIdentityRecovery` in `monitorSingleAccount`. I did not run a live Feishu tenant restart.

## Solution Assessment

Yes for the triage outcome: this issue should stay open while the open candidate PR is reviewed. The best implementation should preserve lifecycle cancellation while ensuring unknown identity in an active lifecycle retries.
