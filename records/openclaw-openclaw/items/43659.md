## Summary

Keep open, but not merge-ready: the Copilot multi-profile cache bug still exists on current main and the latest release, while this PR targets obsolete helper paths instead of the current centralized token helper.

## What This Changes

The PR tries to bind GitHub Copilot API token cache entries to a fingerprint of the requesting GitHub PAT and adds regression tests for same-profile hits, cross-profile misses, and legacy cache misses.

## Best Possible Solution

Refresh or replace this PR with a narrow patch in src/plugin-sdk/provider-auth.ts that preserves integrationId, records a credential fingerprint for the shared cache, and adds regression coverage for cache hit, cross-profile miss, and legacy cache miss.

## Reproduction Assessment

Yes at source level. Current main returns a usable shared Copilot token cache entry before consulting the newly supplied githubToken, so two profiles can receive the same cached Copilot API token; I did not run a live two-account Copilot repro in this read-only pass.

## Solution Assessment

No as submitted. Credential-aware cache reuse is the right fix shape, but the implementation must move to src/plugin-sdk/provider-auth.ts and preserve the current integrationId guard before it is mergeable.
