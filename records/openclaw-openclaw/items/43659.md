## Summary

Keep open. Current main still has an unowned GitHub Copilot API-token cache in the shared provider-auth helper, and this PR is now dirty against main because the implementation moved behind thin wrappers after the branch was last updated. The PR remains useful source work for #43658, but it needs to be rebased and ported to the current shared helper before merge.

## What This Changes

The PR adds a SHA-256-derived GitHub PAT fingerprint to cached GitHub Copilot token entries, requires a fingerprint match for cache hits, updates plugin and legacy token helpers, and adds regression tests for different-profile and legacy-cache misses.

## Best Possible Solution

Port the cache-ownership fix to src/plugin-sdk/provider-auth.ts, keep extensions/github-copilot/token.ts and src/agents/github-copilot-token.ts as thin re-exports, and add focused regression coverage for same-profile cache hits, different-profile cache misses, and legacy unowned-cache misses. Keep #43658 linked until the rebased fix lands or maintainers choose a different cache-key design such as profile-id/per-profile cache files.

## Reproduction Assessment

Yes. On current main, inject a valid CachedCopilotToken into resolveCopilotApiToken via loadJsonFileImpl, then call the helper with a different githubToken; the helper returns the cached Copilot token before the incoming GitHub token affects a fetch. The existing cache-hit test already demonstrates the unowned cache acceptance path.

## Solution Assessment

No as-is. The fingerprint approach is a narrow way to solve the user-visible rotation bug, but the current PR patches stale local helper copies instead of the shared provider-auth implementation that current main exports and uses; the safer path is to move the logic into that central helper and preserve the wrapper structure.
