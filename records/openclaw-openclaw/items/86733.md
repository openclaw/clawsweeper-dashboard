## What This Changes

The PR changes cron timeout policy so main-session `systemEvent` jobs with `wakeMode: "now"` use the existing 60 minute agent-turn safety ceiling, with policy and timer regression tests.

## Best Possible Solution

Land a bounded timeout-policy split that only extends the blocking main-session wake-now systemEvent path, while keeping enqueue-only systemEvent jobs on the generic cron timeout.

## Reproduction Assessment

Yes, source-reproducible: current main applies the generic non-agentTurn watchdog to systemEvent jobs while the wake-now main-session path waits on `runHeartbeatOnce`. I did not execute the timing repro because this was a read-only review and the PR already supplies after-fix terminal proof.

## Solution Assessment

Yes, with maintainer acceptance of the availability tradeoff: the patch limits the longer ceiling to the blocking wake-now main-session path and preserves the generic timeout for next-heartbeat systemEvent jobs. A configurable timeout would be broader and was already avoided by the PR's stated scope.
