## Summary

Current `main` still contains the reported Telegram DM failure path, and the existing regression test still asserts the unwanted visible silent-reply fallback. Because the issue is also paired with open implementation PRs, it should stay open until one of those fixes lands or maintainers choose an equivalent patch.

## What This Changes

The issue requests that successful Telegram direct-message turns with no visible final reply remain silent instead of synthesizing a `NO_REPLY` payload that outbound policy rewrites into visible filler text.

## Best Possible Solution

Keep this issue open and review or land one canonical narrow Telegram adapter fix, preserving real dispatch-error and non-silent delivery-failure fallbacks while changing successful no-visible Telegram DMs into handled silent no-ops with regression coverage.

## Reproduction Assessment

Yes. The current test fixture constructs a Telegram direct session with `queuedFinal: false`, direct silent replies disallowed, and direct rewrite enabled, then asserts that a rewritten visible reply is delivered; the current source follows the same path.

## Solution Assessment

Yes. The best fix is the narrow Telegram adapter change described by the issue and linked PRs, not a broad change to global silent-reply defaults; global default policy can remain a separate maintainer product decision.
