## Summary

Keep open: the Control UI startup direction is useful, but the current patch still lets lower-priority hello work run before the bootstrap-gated chat refresh when the first bootstrap request is pending, so it is not merge-ready yet.

## What This Changes

This PR starts the first Control UI Gateway WebSocket connect in parallel with bootstrap, reprioritizes post-hello startup work, and adds connect-phase timing events with tests.

## Best Possible Solution

Keep the early socket connect, but make the first post-hello chat refresh the gate for deferred bootstrap/identity/health/update work when startup bootstrap is pending, with regression coverage for that combined path.

## Reproduction Assessment

Yes for the review finding by source inspection: with host.controlUiBootstrapReady pending, the chat refresh helper awaits it while the deferred lower-priority block is queued immediately. I did not run the PR’s tests in this read-only review.

## Solution Assessment

No: starting the socket before bootstrap is a plausible owner-boundary fix, but the deferred work should be chained after the bootstrap-gated chat refresh or share the startup bootstrap promise so the intended priority holds under real first-connect timing.
