## Summary

Keep this PR open: it addresses suffix disclosure, but the proposed catch-all branch discloses more of 5- and 6-character secrets than current main and the user-facing security fix still needs a changelog entry.

## What This Changes

The PR changes `maskApiKey` to show one leading character for keys up to four characters and four leading characters for longer keys, then updates unit tests to assert suffix non-disclosure.

## Best Possible Solution

Use one credential-display policy that never reveals suffixes and does not increase disclosed material for short secrets, then cover the shared helper and auth output surfaces with tests plus a changelog entry.

## Reproduction Assessment

Yes. The PR diff and current helper source give a high-confidence reproduction: `abcde`, `abcdef`, and ` short ` reveal four leading characters under the proposed branch while current main reveals two total characters.

## Solution Assessment

No. Removing suffix disclosure is the right direction, but this patch is not the best fix because it increases short-secret disclosure; a safer revision should use constant masking or a short-secret branch that never reveals more material than current main.
