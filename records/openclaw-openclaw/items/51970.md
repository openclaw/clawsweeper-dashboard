## Summary

Keep open for revision. Current main still exposes API-key suffix snippets, and this PR moves in the right direction for long keys, but it increases exposed material for 5- and 6-character credentials, lacks after-fix real CLI/chat proof, and needs a maintainer security-policy choice against the zero-material redaction discussed in https://github.com/openclaw/openclaw/issues/34452 and https://github.com/openclaw/openclaw/pull/39980.

## What This Changes

The PR changes `maskApiKey` and its tests so non-empty values up to four characters show one leading character and longer values show a four-character prefix with no suffix.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, cover boundary behavior in tests, and attach redacted real CLI or chat output for the final policy.

## Reproduction Assessment

Yes, by source inspection. Current main returns first-plus-last masking for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but this implementation increases visible material for short credentials; a short-value guard or constant non-secret marker is safer.
