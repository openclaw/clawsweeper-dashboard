## Summary

Keep this PR open, but do not merge it as-is: it removes suffix disclosure for long keys, yet the new catch-all branch exposes more of 5- and 6-character credentials than current main, and the change needs an explicit maintainer security policy decision.

## What This Changes

The PR changes `maskApiKey` to show one leading character for values up to four characters and four leading characters for longer values, then updates unit tests to assert suffix non-disclosure.

## Best Possible Solution

Adopt an explicit shared credential-display policy that never reveals suffixes and does not increase short-secret disclosure, then cover the helper plus models auth overview/status and chat auth-label surfaces with focused tests and a changelog entry.

## Reproduction Assessment

Yes. The proposed behavior is reproducible from the PR diff and current helper source: values like `abcde`, `abcdef`, and ` short ` deterministically expose more leading secret material under the PR branch than current main.

## Solution Assessment

No. Removing suffix disclosure is the right direction, but this implementation is not the best fix because the catch-all four-character prefix branch exposes most of 5- and 6-character secrets; a safer revision should use constant masking or a separate short-secret branch.
