## Summary

Keep open: current main still exposes key material and this PR targets a real security surface, but the current diff would disclose more of 5- and 6-character credentials and lacks after-fix real behavior proof.

## What This Changes

The PR changes `maskApiKey` to show only a leading one- or four-character prefix plus ellipsis and updates the focused helper tests to assert no tail exposure.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing disclosure for any secret length, then cover the helper plus user-visible auth output with focused tests and redacted real behavior proof.

## Reproduction Assessment

Yes. Source comparison is enough for the blocking finding: current main returns `a...e` and `a...f` for 5- and 6-character inputs, while the PR would return `abcd...`; the helper feeds CLI and chat auth display output.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the catch-all `> 4` branch is not the safest fix because it increases disclosure for short secrets; a short-secret branch or constant redaction is safer.
