## Summary

Keep open: the PR targets a real shared credential-display risk, but the current diff should not merge because it increases disclosure for 5- and 6-character secrets and still lacks after-fix real behavior proof.

## What This Changes

This PR changes `src/utils/mask-api-key.ts` to display one or four leading characters plus `...` and updates the focused helper tests.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing disclosure for any secret length, cover the helper plus CLI/chat auth output, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source comparison is enough for the blocking finding: current main returns `a...e` and `a...f` for 5- and 6-character inputs, while the PR would return `abcd...`, and the helper feeds CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the catch-all `> 4` branch is not the safest fix because it increases disclosure for short secrets; a short-secret branch or constant redaction is safer.
