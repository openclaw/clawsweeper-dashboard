## Summary

Keep open: this PR targets a real credential-exposure problem, but it is not mergeable as-is because the proposed mask increases disclosure for short credentials and the branch has no real CLI/chat proof.

## What This Changes

The PR changes `maskApiKey()` and its unit tests so non-empty credentials show one or four leading characters plus `...` and never reveal trailing characters.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, with focused helper coverage plus redacted real CLI/chat output proof.

## Reproduction Assessment

Yes. Source comparison is enough for the blocking finding: current main returns a two-character first/last mask for 5- and 6-character inputs, while this PR would return a four-character prefix in output paths that feed CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the proposed catch-all four-character prefix increases disclosure for short secrets; a zero-material mask or a short-secret guard is safer.
