## Summary

Keep open. The current main branch still has the reported suffix-exposure behavior, and this PR is directionally useful, but the patch should not merge as-is because its catch-all first-four policy exposes more material for 5- and 6-character credentials and one new test case is placed under the wrong behavior description.

## What This Changes

The PR changes the shared `maskApiKey` helper to show only the first character for values up to four characters and the first four characters for longer values, then updates the unit tests to assert suffix nondisclosure.

## Best Possible Solution

Revise the shared credential-display policy so auth surfaces never expose suffixes and never reveal a larger fraction of short secrets than current main, then cover empty, short, medium, long, whitespace-trimmed, and suffix-nondisclosure cases in the colocated unit test plus the affected auth overview/status tests.

## Reproduction Assessment

Yes. The issue is reproducible by inspecting or exercising `maskApiKey`: current main returns suffix-bearing masks for long values, while the PR diff would make `maskApiKey("abcde")` return `abcd...`.

## Solution Assessment

No. The proposed direction is useful for long keys, but the catch-all first-four branch is not the best fix because it increases disclosure for short credentials; a separate short-key branch or stricter constant mask is safer.
