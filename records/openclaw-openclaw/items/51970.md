## Summary

Keep this PR open but not merge-ready: it addresses a real credential-display leak, but the proposed catch-all branch exposes more material for 5- and 6-character secrets and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes `maskApiKey` and its tests so non-empty values show one leading character up to four characters and otherwise show a four-character prefix with no suffix.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, then cover boundary behavior and attach redacted real CLI or chat proof.

## Reproduction Assessment

Yes, by source comparison. Current main returns first-plus-last masking for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but this implementation increases visible material for short credentials; a short-value guard or constant non-secret marker is safer.
