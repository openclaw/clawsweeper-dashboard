## Summary

Keep this PR open, but not merge-ready: it usefully targets credential-display exposure, yet the proposed fallback reveals more material for 5- and 6-character secrets than current main, lacks after-fix real behavior proof, and still needs maintainer security-policy judgment on whether any key material should be shown.

## What This Changes

The PR changes `maskApiKey` and its unit tests to show one leading character for nonempty values up to four characters and a four-character prefix plus ellipsis for longer values, with no suffix exposure.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, cover boundary behavior in tests, and attach redacted real CLI or chat output.

## Reproduction Assessment

Yes, by source comparison. Current main returns first-plus-last masking for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No, not as-is. Removing suffix exposure is the right direction, but this implementation increases visible material for short credentials; a short-value guard or constant non-secret marker is safer.
