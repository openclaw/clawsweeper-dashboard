## Summary

Keep open: this is a plausible security hardening PR, but it should not merge as-is because source comparison shows it increases disclosure for 5- and 6-character credentials and it still lacks real after-fix CLI/chat proof.

## What This Changes

The branch changes `maskApiKey` and its tests so non-empty values reveal one leading character for inputs up to four characters and four leading characters plus an ellipsis otherwise, with no suffix exposure.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, with focused helper coverage plus redacted real CLI/chat output proof.

## Reproduction Assessment

Yes. Source comparison is high-confidence: current main returns a first/last two-character mask for 5- and 6-character inputs, while this PR would return a four-character prefix in CLI/chat-visible auth label paths.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the catch-all four-character branch discloses more of 5- and 6-character values than current main; zero-material redaction or a short-value guard is safer.
