## Summary

Keep open: the PR targets a real credential-display problem, but the current branch increases disclosure for 5- and 6-character secrets and lacks after-fix real behavior proof.

## What This Changes

This PR changes `maskApiKey` to show one leading character for keys up to four characters and four leading characters for longer keys, with updated unit tests.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without revealing more short-secret material than current main, then cover the helper and auth-output surfaces with focused tests.

## Reproduction Assessment

Yes. Source comparison is enough: the PR changes `abcde` from current `a...e` to `abcd...` and `abcdef` from `a...f` to `abcd...`, so the short-secret disclosure regression is high-confidence without running the branch.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the catch-all `> 4` branch is not the best fix because it reveals most of 5- and 6-character credentials; use a safer short-secret branch or constant redaction.
