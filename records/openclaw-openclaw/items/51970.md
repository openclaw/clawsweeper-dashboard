## Summary

Keep this PR open, but it should not merge as-is. Current main still uses suffix-revealing masking in `maskApiKey`, so the underlying security issue remains real; however, the PR's catch-all first-four policy regresses 5- and 6-character secrets by exposing more of the value than current main, and the related alternative fixes are closed unmerged.

## What This Changes

The PR changes `maskApiKey` to show only the first character for values up to four characters and the first four characters for longer values, then updates unit tests to assert suffix nondisclosure.

## Best Possible Solution

Adopt an explicit shared credential-display policy that removes suffix disclosure without increasing short-secret disclosure, preferably a constant mask or a length-aware policy that reveals less for 5- and 6-character values, then cover the helper plus models auth overview/status and chat auth-label surfaces with focused tests.

## Reproduction Assessment

Yes. The behavior can be reproduced from source inspection or by evaluating the current and proposed helper logic with values like `abcde`, `abcdef`, and `1234567890abcdefghijklmnop`; the short-secret regression is deterministic.

## Solution Assessment

No. The PR is directionally useful for long keys because it removes suffix disclosure, but the catch-all first-four branch is not the best fix because it exposes most of 5- and 6-character secrets; a safer policy needs a separate short-secret branch or no-character mask.
