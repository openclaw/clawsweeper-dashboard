## Summary

Keep open: this is a useful security-hardening direction, but the branch should not merge as-is because it increases disclosure for 5- and 6-character credentials and still lacks after-fix real behavior proof.

## What This Changes

The branch changes `maskApiKey` and its tests so non-empty values reveal one leading character for inputs up to four characters and a four-character prefix plus ellipsis otherwise.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, cover the short-value boundaries, and attach redacted real CLI/chat proof.

## Reproduction Assessment

Yes. Source comparison is high-confidence: current main returns a first/last-character mask for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but this implementation increases exposed material for short credentials; a short-value guard or constant non-secret marker is safer.
