## Summary

Keep this PR open, but it is not merge-ready: the security goal is valid, yet the current patch exposes more material for 5- and 6-character credentials and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `maskApiKey` and its tests to remove suffix exposure by showing one leading character for values up to four characters and a four-character prefix for longer values.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, cover boundary behavior in tests, and attach redacted real CLI or chat output.

## Reproduction Assessment

Yes, by source comparison. Current main returns first-plus-last masking for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but this implementation increases visible material for short credentials; a short-value guard or constant non-secret marker is safer.
