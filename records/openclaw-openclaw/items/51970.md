## Summary

Keep open. The PR is a useful security direction, but the current patch would disclose more material for 5- and 6-character credentials and still lacks after-fix real behavior proof for the CLI/chat output path.

## What This Changes

The PR changes `maskApiKey()` to show only one or four leading characters plus `...` and updates its focused unit tests.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing disclosure for any credential length, cover helper plus CLI/chat auth output, and require redacted real output proof before merge.

## Reproduction Assessment

Yes. Source comparison is enough for the blocking finding: current main returns a two-character first/last mask for 5- and 6-character inputs, while this PR would return a four-character prefix, and the helper feeds CLI/chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but the proposed catch-all four-character prefix increases disclosure for short secrets; full redaction or a short-secret guard is safer.
