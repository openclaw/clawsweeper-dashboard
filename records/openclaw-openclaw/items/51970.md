## Summary

Keep open: the security goal is valid, but the branch currently increases short-credential exposure and still lacks redacted real CLI/chat proof.

## What This Changes

The PR changes `maskApiKey` and its tests so non-empty values reveal one leading character for inputs up to four characters and a four-character prefix plus ellipsis otherwise.

## Best Possible Solution

Adopt a masking policy that removes suffix exposure without increasing visible material for any credential length, cover boundary behavior in tests, and attach redacted real CLI/chat output.

## Reproduction Assessment

Yes. Source comparison is high-confidence: current main returns a first/last-character mask for 5- and 6-character inputs, while this PR returns a four-character prefix in helper paths used by CLI and chat auth labels.

## Solution Assessment

No. Removing suffix exposure is the right direction, but this implementation increases exposed material for short credentials; a short-value guard or constant non-secret marker is safer.
