## Summary

Keep open. The PR is useful defense-in-depth, but it does not fully fix the linked image-tool path because the production image tool can still workspace-resolve `media://inbound/<id>` before `loadWebMedia()` sees it, and the new fail-fast branch can reject valid whitespace-prefixed managed media before the shared normalizer resolves it.

## What This Changes

The PR adds `loadWebMediaInternal()` guards so unresolved `media://` and other scheme-bearing media references are not joined under `workspaceDir`, plus regression tests in `src/media/web-media.test.ts`.

## Best Possible Solution

Amend the PR so managed media references are normalized and passed through before any workspace join, with focused image-tool and loader tests plus a user-facing changelog entry; leave broader sandbox bridge behavior to #74186 unless the same seam requires coordination.

## Reproduction Assessment

Yes. The static reproduction is high-confidence: current main resolves `media://inbound/<id>` under `workspaceDir` in the image tool, and Node path resolution collapses the URI into the reported `media:/` workspace path.

## Solution Assessment

No. The loader guard is a useful backstop, but the narrow maintainable fix also needs to prevent the image-tool workspace join for `isMediaStoreUrl` and resolve normalized valid inbound media before treating a URI as stranded.
