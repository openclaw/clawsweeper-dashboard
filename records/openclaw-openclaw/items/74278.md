## Summary

Keep open. The loader guard is useful defense-in-depth, but the PR still leaves the linked image-tool bug path open and also rejects whitespace-prefixed existing inbound media refs before the shared resolver can normalize them.

## What This Changes

The PR adds `loadWebMediaInternal()` guards and regression tests so unresolved `media://` and other scheme-bearing references are not joined under `workspaceDir`.

## Best Possible Solution

Resolve managed `media://` refs through the shared media-reference/loadWebMedia contract before any workspace or sandbox path join, with focused coverage for the loader, image tool, and sandbox bridge seams.

## Reproduction Assessment

Yes, at source level. `createOpenClawTools()` passes `workspaceDir` into `createImageTool()`, and the current image-tool branch can resolve `media://inbound/<id>` under that workspace before the loader sees the managed URI.

## Solution Assessment

No. The loader guard is useful, but the fix should also prevent image-tool and sandbox pre-resolution and should normalize through the shared resolver before declaring a media URI stranded.
