## Summary

Keep open. The PR is aimed at the right media-loading area, and the author already addressed the Greptile whitespace/test-reuse comments, but the current branch is still not merge-ready: it does not fix the actual image-tool workspace path that can collapse `media://inbound/...` before `loadWebMedia()` sees it, it would fail fast on valid whitespace-prefixed managed media that the shared resolver can already normalize, and it is missing the required user-facing changelog entry.

## What This Changes

The PR adds `loadWebMediaInternal()` guards so unresolved `media://` and other scheme-bearing media references are not resolved under `workspaceDir`, plus regression tests for stranded URI error behavior.

## Best Possible Solution

Amend the PR so managed media references are normalized and preserved before any workspace path join: exclude `refInfo.isMediaStoreUrl` from image-tool workspace resolution, let `loadWebMedia()` resolve trimmed valid inbound `media://` references before applying a stranded-URI guard, keep unsupported locations on the existing error path, add focused coverage for both seams, and leave the sandbox bridge behavior to #74186.

## Reproduction Assessment

Yes. The static reproduction is high-confidence: `createOpenClawTools()` supplies `workspaceDir`, `image-tool.ts` can resolve `media://inbound/<id>` with `path.resolve(workspaceDir, ...)`, and Node turns that into a `media:/` path before `loadWebMedia()` can apply this PR's new guard.

## Solution Assessment

No. The PR's loader guard is useful defense-in-depth, but it is not the best complete fix because it misses the production image-tool workspace join and rejects valid whitespace-prefixed managed media before the shared normalizer can resolve it.
