## Summary

Keep open. The PR is a useful defense-in-depth change, but it does not fully fix the linked image-tool failure because current main can still turn managed media refs into workspace paths before the shared loader runs, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `loadWebMediaInternal()` guards and regression tests so unresolved `media://` and other scheme-bearing media references are not joined under `workspaceDir`.

## Best Possible Solution

Preserve or resolve `media://inbound/*` through the shared media-reference contract before any image-tool workspace join, sandbox bridge resolution, or loader local-path fallback, with focused regressions for all three entry points.

## Reproduction Assessment

Yes. Source inspection shows `media://inbound/<id>` can still reach image-tool workspace resolution on current main, and Node path resolution produces the reported `/workspace/media:/inbound/<id>` shape.

## Solution Assessment

No. The loader guard is useful defense-in-depth, but the fix is incomplete unless managed media refs are preserved before the image-tool workspace join and normalized through the shared resolver before any stranded-URI failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep media URIs out of the image-tool workspace join:** `src/media/web-media.ts:379-386`
  - body: This guard only helps callers that pass a raw `media://` string into `loadWebMedia()`. The normal non-sandbox image-tool path still resolves `media://inbound/<id>` against `workspaceDir` before calling the loader, producing `<workspace>/media:/...` and bypassing the new guard. Exclude managed media refs from that join and add image-tool coverage with `workspaceDir` set.
  - confidence: 0.9
- **[P2] Resolve normalized inbound refs before failing fast:** `src/media/web-media.ts:386`
  - body: The new fail-fast branch runs after the strict raw `resolveMediaStoreUriToPath()` precheck, so `  media://inbound/<existing-id>` becomes `not-found` even though the shared media-reference resolver trims the same value and can hydrate the managed file. Normalize or call the shared resolver before treating a surviving `media://` value as stranded.
  - confidence: 0.88
