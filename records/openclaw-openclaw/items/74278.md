## Summary

Keep open. The PR is on the right loader boundary for preventing URI schemes from being joined under `workspaceDir`, but as written it still rejects the valid whitespace-prefixed managed media URI case described as the root cause instead of resolving the existing inbound file. It also still needs the required user-facing changelog entry.

## What This Changes

This PR adds `loadWebMediaInternal()` guards so unresolved `media://` and other scheme-bearing media references are not resolved under `workspaceDir`, plus regression tests for stranded URI error behavior.

## Best Possible Solution

The end state should be a media loader that resolves canonical and LLM-formatted `media://inbound/<id>` references when the managed file exists, rejects only truly unresolved or unsupported scheme-bearing inputs before workspace joining, and leaves sandbox bridge handling to #74186.

## Reproduction Assessment

Yes. A focused unit test can create an inbound managed media file and call `loadWebMedia('  media://inbound/<id>', { workspaceDir })`; with this PR's guard, that valid reference returns `not-found` before hydration instead of resolving the file.

## Solution Assessment

No. The best narrow fix is to normalize or retry managed-media resolution before the fail-fast stranded-URI guard, then keep the scheme guard as defense-in-depth for values that still cannot resolve.
