## Summary

Keep open: the PR is a focused bug-fix attempt, but it still does not cover the normal image-tool workspace path and it can reject a normalized managed inbound URI that the shared resolver should hydrate. It also still needs after-fix real behavior proof beyond unit tests and type/check output.

## What This Changes

The PR adds a `loadWebMediaInternal` guard for unresolved `media://` URIs, skips loader-level workspace joining for scheme-bearing values, and adds three regression tests in `src/media/web-media.test.ts`.

## Best Possible Solution

Resolve managed `media://` refs through the shared media-reference contract before any image-tool, sandbox bridge, or loader workspace join, with the loader guard kept only as defense-in-depth after normalized resolution fails.

## Reproduction Assessment

Yes, source-level. `createOpenClawTools()` passes `workspaceDir` into `createImageTool()`, and the current non-sandbox image-tool branch can workspace-join `media://inbound/<id>` before the loader receives the managed URI.

## Solution Assessment

No. The loader guard is useful defense-in-depth, but this patch leaves the image-tool pre-resolution path alive and should normalize through the shared resolver before declaring a media URI stranded.

## Review Findings
