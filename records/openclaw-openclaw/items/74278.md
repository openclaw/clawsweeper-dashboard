## Summary

Keep this PR open. The patch is a useful backstop, but it is not merge-ready because the normal image-tool path can still corrupt managed media URIs before the loader sees them, and the new fail-fast branch can reject whitespace-prefixed inbound refs that the shared resolver can otherwise normalize.

## What This Changes

The PR adds `loadWebMediaInternal()` guards and regression tests so unresolved `media://` and other scheme-bearing media references are not joined under `workspaceDir`.

## Best Possible Solution

The maintainable end state is one normalized managed-media path through the image tool, direct loader, and sandbox bridge so `media://inbound/*` is never treated as workspace-relative, with focused regressions and release notes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the image tool receives `media://inbound/<id>`, resolves it under `workspaceDir`, and Node collapses it to a `media:/` path before `loadWebMedia()` can apply this PR's new guard.

## Solution Assessment

No. The loader guard is useful defense-in-depth, but the narrow fix must also preserve `refInfo.isMediaStoreUrl` through the image-tool workspace join and normalize inbound refs before treating them as stranded.
