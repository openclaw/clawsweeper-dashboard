## Summary

The report is not reproducible as a general trigram-tokenizer failure: current SQLite matches long CJK strings as contiguous substrings, consistent with the existing planner. The remaining question is a product/search-semantics decision about whether semantically related but lexically non-matching CJK results should receive a text score, so this should stay open for maintainer review rather than queueing the proposed broad LIKE fallback.

## What This Changes

The issue requests changing memory-core CJK trigram keyword search so hybrid results receive a nonzero text score even when the long CJK query has no current FTS5 MATCH row.

## Best Possible Solution

Define the intended CJK lexical relevance policy first, then add a focused SQLite-backed test using the reporter's exact query/document pair and implement only the narrow planner or scoring change that satisfies it without per-character false positives or unbounded no-hit scans.

## Reproduction Assessment

No for the stated general failure: a current-runtime SQLite probe matched the long quoted CJK substring as documented. The exact indexed text, query, and raw keyword rows are needed to reproduce why the reporter's result had `textScore: 0`.

## Solution Assessment

No. The proposed character-level LIKE plan and fallback after every empty MATCH broaden lexical semantics and can cause expensive scans; the best fix depends on a maintainer decision about whether non-contiguous or partial CJK overlap should count as keyword relevance.
