## Summary

Current main still contains the reported CJK trigram planning and zero-row fallback gap, and a direct SQLite FTS5 check reproduces the silent empty MATCH. Two open PRs explicitly own the fix; the issue should remain open until maintainers select and merge the narrow implementation that preserves existing Latin query semantics.

## What This Changes

Fix memory-core trigram FTS5 keyword search so CJK and mixed short-token queries receive a meaningful lexical score instead of always falling back to vector-only ranking.

## Best Possible Solution

Land a focused CJK-aware planner that splits CJK code points while retaining contiguous non-CJK runs, and retry LIKE only when the plan contains explicit substring terms, preserving existing Latin BM25 and AND semantics.

## Reproduction Assessment

Yes. Current main deterministically builds the problematic quoted CJK MATCH, and a direct node:sqlite FTS5 trigram run confirms that it silently returns zero rows while the intended per-character substring path finds the document.

## Solution Assessment

Yes, the narrow approach represented by https://github.com/openclaw/openclaw/pull/92164 is the best direction because it fixes CJK and mixed-token behavior without globally replacing empty multi-term MATCH results with lower-precision OR matches.
