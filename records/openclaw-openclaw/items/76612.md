## Summary

Keep open. Current main still has a source path that can pass explicit or inherited `cacheRetention` to a Kimi `anthropic-messages` run, and the Kimi plugin does not strip Anthropic `cache_control` markers before the request reaches `api.kimi.com`. The reporter's A/B workaround is credible and separate from #76007's orphan-delta stream fix.

## What This Changes

Kimi Code should not receive Anthropic prompt-cache `cache_control` markers unless the provider explicitly supports them.

## Best Possible Solution

Keep this issue open and land a Kimi-owned payload-policy fix that suppresses Anthropic cache markers for Kimi Code, with regression coverage for inherited `cacheRetention`.

## Reproduction Assessment

Yes for a source-level reproduction, but not for a live request in this review. A Kimi `anthropic-messages` model with inherited or explicit `cacheRetention` flows into the Anthropic transport, which applies `cache_control` before Kimi's payload wrapper runs.

## Solution Assessment

Yes, the best narrow fix is to keep the policy in the Kimi provider boundary: strip or disable Anthropic `cache_control` for Kimi Code while preserving direct Anthropic, OpenRouter, Vertex, and Bedrock cache behavior.
