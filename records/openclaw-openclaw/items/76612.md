## Summary

Keep open. Current main still lets inherited or explicit `cacheRetention` reach the Kimi `anthropic-messages` request path, where Anthropic `cache_control` markers are applied before the Kimi wrapper sends to `api.kimi.com`; the reporter's A/B evidence is credible and distinct from the already-fixed orphan-delta issue.

## What This Changes

The issue asks OpenClaw to stop sending Anthropic `cache_control` prompt-cache markers to Kimi Code unless Kimi explicitly supports them.

## Best Possible Solution

Land a Kimi-owned payload policy fix that suppresses or strips Anthropic `cache_control` markers for Kimi Code, with regression coverage and no broad change to Anthropic, Vertex, Bedrock, OpenRouter, or generic custom-provider cache behavior.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction, but not a live provider run in this review. A Kimi `anthropic-messages` model with inherited or explicit `cacheRetention` reaches `buildAnthropicParams`, receives `cache_control`, and the Kimi wrapper does not remove it before sending.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Kimi provider boundary; globally disabling custom Anthropic-compatible cache markers would regress tested behavior for providers that intentionally opt into cache retention.
