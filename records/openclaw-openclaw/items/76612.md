## Summary

Keep open. Current main can still carry inherited or explicit `cacheRetention` into Kimi's `anthropic-messages` request path, and the Kimi wrapper does not strip Anthropic `cache_control` markers before requests reach `api.kimi.com`; the reporter's A/B evidence is credible and distinct from the already-fixed #76007 stream accumulator issue.

## What This Changes

Request that OpenClaw stop sending Anthropic `cache_control` prompt-cache markers to Kimi Code endpoints unless that provider explicitly supports them.

## Best Possible Solution

Land a Kimi-owned payload policy fix that suppresses or strips `cache_control` for Kimi Code, with regression coverage for inherited `cacheRetention` and no broad change to generic Anthropic/OpenRouter/Vertex/Bedrock cache behavior.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live request in this review. A Kimi `anthropic-messages` model with inherited or explicit `cacheRetention` reaches `buildAnthropicParams`, which applies `cache_control` before the Kimi wrapper sends the request.

## Solution Assessment

Yes. The narrow maintainable fix is at the Kimi provider boundary; globally disabling custom `anthropic-messages` cache markers would regress providers whose explicit cache-retention behavior is already tested.
