## Summary

Current main still appears to send Anthropic `cache_control` markers through the Kimi Code Anthropic-compatible path when `cacheRetention` is active, and the issue is already paired with open PR #76741 using closing syntax. Keep the issue open until that PR or an equivalent Kimi-owned fix lands.

## What This Changes

The issue asks OpenClaw to suppress or strip Anthropic `cache_control` prompt-cache markers for Kimi Code requests so successful Kimi responses do not end as empty `content: []` results.

## Best Possible Solution

Land PR #76741 or an equivalent Kimi-owned payload patch that strips `cache_control` only for Kimi Code, with regression coverage and the existing changelog entry.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: a Kimi `anthropic-messages` model with active `cacheRetention` reaches `buildAnthropicParams`, which injects `cache_control`, and the current Kimi wrapper does not remove it. I did not run a live Kimi request in this read-only review.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable solution: strip the Anthropic cache markers at the Kimi provider boundary rather than changing shared Anthropic-compatible cache semantics for every provider.
