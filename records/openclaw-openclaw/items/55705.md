## Label Justifications

- `P2`: This is a normal-priority Control UI regression that misleads users about active sessions without making the core runtime unusable.
- `impact:session-state`: The affected behavior is cached session state diverging from the current `sessions.list` result and rendering a stale count.

## Summary

Keep open: current main and v2026.6.1 still leave `sessionsResult` unchanged when `sessions.list` resolves to `undefined`, while the Overview card renders `sessionsResult.count`, so a stale Subagent/session count can persist. The open fork PR at https://github.com/BingqingLyu/openclaw/pull/1514 remains the linked implementation candidate rather than a merged fix.

## What This Changes

The issue asks for the Control UI Subagent/session count to match the actual API/CLI session list instead of showing stale active sessions.

## Best Possible Solution

Land or adapt the narrow controller fix so `loadSessions` clears stale `sessionsResult` when `sessions.list` returns no payload, with focused regression coverage for the stale-count path.

## Reproduction Assessment

Yes for source-level reproduction: seed `sessionsResult` with a nonzero count, have `sessions.list` resolve to `undefined`, and `loadSessionsOnce` leaves the old result in place. I did not run a live macOS browser repro in this read-only review.

## Solution Assessment

No, current main is not the best solution because it preserves stale cached state on an undefined response. The best fix is the narrow controller assignment to clear the result, plus a regression test that covers an empty or undefined API response after a prior nonempty result.
