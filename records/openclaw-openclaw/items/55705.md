## Summary

Keep open: current main and latest release still keep the stale `sessionsResult` cache when `sessions.list` resolves successfully with an undefined payload, and the linked fork PR is open but conflicting and not landable as-is.

## What This Changes

The issue reports that the Control UI shows a stale active Subagent/session count even though API/CLI session queries are empty.

## Best Possible Solution

Land a narrow controller fix that clears `sessionsResult` only for empty successful `sessions.list` responses, preserves last-known-good data on generic errors, and adds focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to seed `state.sessionsResult` with a nonzero count, mock `client.request("sessions.list")` to resolve `undefined`, and call `loadSessions`; the current `if (res)` guard leaves the stale result cached.

## Solution Assessment

Yes for the narrow success-path clear, but not the linked PR as-is. The maintainable fix should avoid clearing cached sessions on generic transport/RPC errors unless maintainers intentionally change that UX.
