## Summary

Keep open: current main and v2026.6.6 still keep `sessionsResult` unchanged when `sessions.list` resolves to `undefined`, and an open unmerged/conflicting fork PR is the active fix candidate rather than a landed fix.

## What This Changes

The issue reports the Control UI showing a stale nonzero Subagent/session count even when API/CLI session listing is empty.

## Best Possible Solution

Land or adapt the narrow controller fix so a successful undefined `sessions.list` response clears `sessionsResult`, while preserving last-known-good data on generic errors and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: seed `sessionsResult` with a nonzero result, have `client.request("sessions.list")` resolve to `undefined`, and `loadSessionsOnce` leaves the old result because assignment is guarded by `if (res)`. I did not run a live browser/macOS reproduction in this read-only pass.

## Solution Assessment

No, current main is not the best solution because it preserves stale cached state on an undefined successful response. The narrow maintainable fix is to clear only that success-path result and cover it with a controller test.
