## Summary

Keep open: current main and latest release still preserve stale `sessionsResult` when a successful `sessions.list` response resolves to `undefined`, and an open linked fork PR still targets this issue with closing syntax.

## What This Changes

The issue reports that the Control UI shows a stale Subagent/session count even when API or CLI session listing is empty.

## Best Possible Solution

Review, adopt, or replace the open linked PR with a narrow controller fix that clears `sessionsResult` for empty successful `sessions.list` responses, preserves last-known-good state on generic errors, and adds regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: seed `state.sessionsResult` with a nonzero count, mock `client.request("sessions.list")` to resolve `undefined`, and call `loadSessions`; current code skips assignment and leaves the stale count. I did not run a live macOS browser repro in this read-only review.

## Solution Assessment

Yes, a narrow Control UI controller fix is the best layer because the backend normal empty-list contract already returns a count-bearing result and the stale display comes from UI cache handling. The fix should preserve generic-error last-known-good behavior while clearing empty successful responses.
