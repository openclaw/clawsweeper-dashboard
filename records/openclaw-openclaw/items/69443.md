## Summary

Keep open. Current main still leaves the reported manual Gateway `agent` RPC path actionable: a canonical `agent:<id>:...` session key passed only in `to` is not promoted to the requested session, while the handler defaults the run to the agent main session and later treats `to` as a delivery target. The built-in subagent announce path already passes `sessionKey`, so this is a narrow compatibility/validation gap in generic Gateway/session routing rather than a WeChat plugin bug.

## What This Changes

The issue reports that a subagent callback using `gateway call agent` with a canonical WeChat session key in `to` routes to the main session instead of the intended WeChat direct session.

## Best Possible Solution

Keep the fix in the generic Gateway/session layer. When `agent` RPC receives a canonical session key in `to` without `sessionKey` or `sessionId`, it should either route the run to that explicit session while using stored delivery metadata, or fail fast with a clear error telling callers to pass `sessionKey`; WeChat-specific code should remain outside core.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available from current main: call `agent` with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey`; the Gateway does not select that session key and the session resolver/default path lands on the agent main session.

## Solution Assessment

No. The current behavior is not the best way to solve the issue because it silently treats a session-key-shaped selector as a delivery target and defaults the run to main; a narrow core compatibility or validation guard is safer and maintainable.
