## Summary

Keep open. Current main still selects the Gateway `agent` run session from `sessionKey`, `sessionId`, or the default agent session before treating `to` as delivery metadata, so a canonical session key supplied only in `to` is not honored as the target session.

## What This Changes

Fix Gateway agent RPC routing so subagent callbacks with `to` set to a canonical WeChat session key target that session instead of the agent main session.

## Best Possible Solution

Handle canonical `agent:<id>:...` values in Gateway `agent` `to` as explicit session selectors only when `sessionKey` and `sessionId` are absent, or reject that shape with a clear validation error; keep the fix generic and avoid WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-key selection path before `to` is merged into delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the generic Gateway/session resolver with regression coverage for session-key-shaped `to` values, while preserving existing explicit `sessionKey` and `sessionId` precedence.
