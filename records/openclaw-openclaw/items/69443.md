## Summary

Keep open. Current main still chooses the Gateway `agent` run session from `sessionKey` or `sessionId`, while a canonical session key passed only in `to` is handled as delivery metadata and can still resolve back to the agent main session.

## What This Changes

The issue requests that a subagent Gateway RPC callback using a WeChat session key in `to` route to that explicit session instead of the main session.

## Best Possible Solution

Teach the Gateway agent resolver to handle canonical `agent:<id>:...` values in `to` as explicit session selectors only when no `sessionKey` or `sessionId` is present, or reject them with a clear validation error.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` call with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session selection path before treating `to` as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is in the generic Gateway/session resolver, with regression coverage for session-key-shaped `to` values and no WeChat-specific logic in core.
