## Label Justifications

- `P2`: This is a source-provable routing bug with limited but real impact on subagent callbacks to channel sessions.
- `impact:message-loss`: The reported callback can be delivered or routed to the wrong chat/session instead of the intended WeChat conversation.
- `impact:session-state`: The wrong session key receives the agent run context, causing transcript and conversation state drift.

## Summary

Keep open. Current main and the latest release still select the Gateway agent run session from `sessionKey`, `sessionId`, or the agent default before `to` is treated as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still fall back to the main session.

## What This Changes

The issue asks for Gateway `agent` RPC callbacks with a WeChat session-key-shaped `to` value to target that session instead of the agent main session.

## Best Possible Solution

Add a generic Gateway/session resolver guard so canonical `agent:<id>:...` values supplied only through `to` either select that session without becoming a native delivery target, or fail closed with clear guidance to use `sessionKey`.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows default session selection before `to` is merged as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is generic session-key-shaped `to` handling or rejection in the Gateway/session resolver, with regression coverage and no WeChat-specific core logic.
