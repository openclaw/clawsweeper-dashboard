## Label Justifications

- `P2`: This is a normal-priority routing bug with a clear source path and limited blast radius around Gateway agent callbacks.
- `impact:message-loss`: The reported callback can be misrouted to the main session instead of the intended WeChat session.
- `impact:session-state`: The wrong session receives the callback context, which can drift transcript and memory state away from the intended conversation.

## Summary

Keep open. Current main and v2026.5.18 still choose the Gateway `agent` run session from `sessionKey`, `sessionId`, or default-agent routing before `to` is treated as delivery metadata, so a `to`-only canonical WeChat/OpenClaw session key can still land in the main session path.

## What This Changes

Fix Gateway `agent` RPC so a subagent callback whose `to` value is a canonical WeChat/OpenClaw session key routes to that session rather than the default main session.

## Best Possible Solution

Add a generic Gateway/session resolver guard for canonical `agent:<id>:...` values in `to`, preserving explicit `sessionKey` and `sessionId` precedence and avoiding WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-selection path before `to` is merged as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is generic session-key-shaped `to` handling or rejection in the Gateway/session resolver, with regression coverage and no channel-specific special case.
