## Summary

Keep open. Current main still treats a canonical session key passed only in `to` as a delivery/turn-source value rather than a requested session, so the reported Gateway `agent` RPC path can still land on the agent main session.

## What This Changes

The issue asks for subagent/manual Gateway `agent` RPC callbacks that pass a WeChat direct session key in `to` to route to that explicit session instead of `agent:main:main`.

## Best Possible Solution

Route canonical `agent:<id>:...` session keys passed in `to` as explicit session selectors when `sessionKey` and `sessionId` are absent, while leaving ordinary external delivery targets unchanged and keeping WeChat-specific code out of core.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call Gateway `agent` with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId`; current main does not promote that value to the requested session and the downstream direct-chat resolver can collapse it to the agent main session.

## Solution Assessment

Yes. A narrow generic Gateway/session-layer fix is the best direction because the bug is in session selection semantics, while the WeChat docs make the plugin runtime external to this repository.
