## Summary

Keep open. Current main and the latest release still choose the Gateway agent run session from `sessionKey`, `sessionId`, or the agent default before `to` is treated as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still fall back to the main session.

## What This Changes

The issue requests that Gateway agent RPC callbacks using a WeChat canonical session key in `to` route to that session instead of the main session.

## Best Possible Solution

Add a generic Gateway/session resolver guard for canonical `agent:<id>:...` values in `to`, preserving explicit `sessionKey` and `sessionId` precedence and avoiding WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-selection path before `to` is merged as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is generic session-key-shaped `to` handling or rejection in the Gateway/session resolver, with regression coverage and no channel-specific special case.
