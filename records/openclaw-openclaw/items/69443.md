## Summary

Keep open. Current main and the latest release still choose the Gateway `agent` run session from `sessionKey`, `sessionId`, or the default agent session before `to` is treated as delivery metadata, so a canonical session key supplied only in `to` can still fall back to the main session.

## What This Changes

The issue asks for Gateway `agent` RPC callbacks with a canonical WeChat session key in `to` to route to that session instead of the main session.

## Best Possible Solution

Add a generic Gateway/session resolver guard that treats canonical `agent:<id>:...` values in `to` as explicit session selectors only when `sessionKey` and `sessionId` are absent, or rejects that shape with a clear validation error, without adding WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-key selection path before `to` is merged into delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the generic Gateway/session resolver with regression coverage for session-key-shaped `to` values, while preserving explicit `sessionKey` and `sessionId` precedence.
