## Summary

Keep open. Current main still selects the Gateway `agent` run session from `sessionKey`, `sessionId`, or the default agent session before `to` is used only as delivery metadata, so a canonical session key supplied only in `to` can silently fall back to the main session; no open canonical fix PR was found.

## What This Changes

The issue requests Gateway `agent` RPC to route subagent callbacks to a WeChat session key supplied in `to` instead of falling back to the main agent session.

## Best Possible Solution

Resolve session-key-shaped `to` values generically before defaulting to the main session when `sessionKey` and `sessionId` are absent, with regression coverage and no WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with only `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` follows the default session-key selection path before `to` is merged into delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is in the generic Gateway/session resolver and tests, while preserving explicit `sessionKey` and `sessionId` precedence and avoiding WeChat-specific routing code.
