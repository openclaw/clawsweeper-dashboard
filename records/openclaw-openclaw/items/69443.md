- `impact:message-loss`: The callback can be routed away from the intended WeChat conversation and into the main session path.

## Summary

Keep open. Current main and v2026.5.18 still select the Gateway `agent` run session before `to` is interpreted as delivery metadata, so a `to`-only canonical WeChat/OpenClaw session key can still resolve to the main session path.

## What This Changes

The issue reports that a subagent Gateway RPC callback with `to` set to a canonical WeChat/OpenClaw session key is routed to the main session rather than the specified WeChat session.

## Best Possible Solution

Add generic canonical session-key handling for `to` in the Gateway/session resolver, preserving explicit `sessionKey` and `sessionId` precedence and avoiding WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-selection path before `to` is merged as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is generic canonical `to` session-key handling in the existing Gateway/session resolver, with regression coverage and no channel-specific special case.
