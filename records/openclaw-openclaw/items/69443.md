## Label Justifications

- `P1`: The reported bug can misroute subagent callback results away from the intended channel session in a real agent workflow.
- `impact:message-loss`: The issue is about callback messages being routed to the wrong session instead of the requested WeChat session.
- `impact:session-state`: The wrong session key selection can attach subagent output to the main session transcript and state.
- `impact:security`: Misrouting subagent callback content can expose private results to an unintended session context.

## Summary

Keep open. Current main still selects the Gateway agent run session from `sessionKey`, `sessionId`, or the agent default before `to` is merged as delivery metadata, so a canonical WeChat session key supplied only in `to` can still route work through the main session.

## What This Changes

The issue requests that `gateway call agent` honor a canonical WeChat session key supplied in `to` instead of falling back to the main agent session.

## Best Possible Solution

Add a generic Gateway/session resolver guard for canonical `agent:<id>:...` values in `to` when no explicit selector is present, with focused tests and no WeChat-specific core branch.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey` or `sessionId` still follows the default session-selection path before `to` is merged as delivery metadata.

## Solution Assessment

Yes. The narrow maintainable fix is generic canonical-session-key handling or rejection in the Gateway/session resolver, while preserving explicit `sessionKey` and `sessionId` precedence.
