## Label Justifications

- `P1`: The report describes a broken agent/channel callback workflow where a subagent result can route to the wrong session.
- `impact:message-loss`: The issue is about a callback message being misrouted away from the requested WeChat session.
- `impact:session-state`: The wrong session selection can attach agent work and delivery metadata to the main session instead of the intended channel session.

## Summary

Keep open. Current main and v2026.5.18 still select the Gateway agent run session from `sessionKey`, `sessionId`, or an agent-main default before `to` is used as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still miss the intended WeChat-backed session. The adjacent open work for https://github.com/openclaw/openclaw/issues/41483 targets explicit `--agent --channel --to` per-recipient derivation and does not close this raw Gateway `to`-as-session-key path.

## What This Changes

The issue asks Gateway agent RPC to honor a canonical WeChat-backed session key supplied in `to` for subagent callbacks instead of falling back to the main session.

## Best Possible Solution

Add generic handling or a clear rejection path for canonical `agent:<id>:...` values in `to` before delivery metadata is merged, while preserving explicit `sessionKey` and `sessionId` precedence and avoiding WeChat-specific core logic.

## Reproduction Assessment

Yes, at source level. A Gateway `agent` request with a canonical `agent:main:openclaw-weixin:direct:...` value only in `to` still follows session selection that ignores `to` until after the selected session is loaded.

## Solution Assessment

Yes. The narrow maintainable fix is generic session-key-shaped `to` handling or validation in the Gateway/session resolver, with explicit selector precedence preserved and no channel-specific special case.
