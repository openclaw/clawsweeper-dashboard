## Summary

Keep open. Current main still selects the Gateway `agent` run session from `sessionKey`/`sessionId`, while a canonical session key passed only in `to` remains delivery/source metadata and can still collapse to the agent main session through the direct-chat resolver.

## What This Changes

The issue requests that Gateway `agent` RPC callbacks whose `to` value is a canonical WeChat session key route to that session instead of the agent main session.

## Best Possible Solution

Handle canonical `agent:<id>:...` values in `to` as explicit session selectors only when no `sessionKey` or `sessionId` is present, or reject them with a clear validation error, while keeping WeChat-specific logic out of core.

## Reproduction Assessment

Yes, at source level. Call Gateway `agent` with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey`/`sessionId`; current main does not promote that value to the requested session and the direct resolver can select `agent:main:main`.

## Solution Assessment

Yes. A narrow generic Gateway/session resolver fix is the best direction, with regression coverage for canonical session-key-shaped `to` values and safeguards so hooks or normal delivery targets do not gain unintended session override behavior.
