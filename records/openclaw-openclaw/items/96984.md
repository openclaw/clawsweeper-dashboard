## Summary

Keep open for review/merge rather than cleanup-close: current main still has the unbounded Google TTS success parser, and the PR is the right narrow SDK-reader fix. The remaining blockers are merge-readiness items: contributor proof is mock-only, and the shared 16 MiB cap is a compatibility tradeoff maintainers should see.

## What This Changes

The PR switches Google Gemini TTS success JSON parsing from native `response.json()` to `readProviderJsonResponse` and adds provider test/mock coverage for oversized stream cancellation.

## Best Possible Solution

Land the narrow shared-reader fix after real behavior proof is added and maintainers accept the 16 MiB Google TTS success-response cap, while leaving request construction, retry policy, voice/model selection, and audio extraction unchanged.

## Reproduction Assessment

Yes by source inspection: current main calls raw `res.json()` on the Google TTS success path after `postJsonRequest`, so an oversized successful JSON body bypasses the shared bounded reader. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the code change is the best fix shape: it moves the single success parse point to the existing SDK source-of-truth reader instead of adding a Google-local cap. Merge should wait for real behavior proof and explicit acceptance of the large-response compatibility boundary.
