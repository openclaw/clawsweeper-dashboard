## Summary

Keep open: this is a valid scoped fix for an opaque gateway error, but the latest diff still lacks after-fix real behavior proof and introduces an unsanitized raw channel id into a user-visible error path.

## What This Changes

The PR changes `channels.start` to distinguish disabled bundled channels from unrecognized channel ids and adds an Unreleased changelog entry for the channel login error clarification.

## Best Possible Solution

Land a sanitized, scoped error-message fix with regression coverage and real login/gateway proof, while preserving the remaining auto-config behavior as a separate tracked follow-up if maintainers still want it.

## Reproduction Assessment

Yes. Source inspection shows current main returns `invalid channels.start channel` when `channels.start` receives a bundled channel that is absent from the active plugin registry; I did not run a live login because this read-only review must not mutate local auth or config state.

## Solution Assessment

No, not merge-ready as written. The disabled-versus-unknown message split is a maintainable direction, but the unknown-channel error should sanitize raw input and the contributor still needs real behavior proof before merge.
