## Summary

Keep open: the PR is useful but not ready because the changed gateway branch cannot distinguish a missing channel config block from a typo or unsupported channel, and the user-facing fix still needs changelog coverage.

## What This Changes

This PR changes the gateway `channels.start` invalid-channel branch to return an actionable missing-channel-config error that names the requested channel.

## Best Possible Solution

Land a narrow gateway message fix that preserves accurate invalid-channel handling, adds focused regression coverage and a changelog entry, and keeps the remaining pre-save behavior concern separately tracked.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the handler branch: call `channels.start` with a channel value that `normalizeChannelId` cannot resolve and current main returns the opaque invalid-channel error. A live WhatsApp login was not run because it would mutate local auth state.

## Solution Assessment

No as-is. The narrow gateway-message direction is maintainable, but this patch currently treats every unnormalizable channel as a missing config block and lacks the required changelog entry.
