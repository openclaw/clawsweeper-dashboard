## Summary

Keep open: current main still has native Codex reasoning/plan signals, but the Control UI only renders thinking already embedded in assistant message content and drops Codex plan/progress agent streams. The remaining work is a real UX gap, but it needs maintainer guidance on which Codex app-server events are safe to surface without exposing private reasoning.

## What This Changes

Request Control UI visible progress or thinking status for native Codex runtime runs when reasoning is enabled, instead of only showing the running dots.

## Best Possible Solution

Define an approved Control UI Codex activity surface that consumes safe plan/tool/lifecycle signals or clearly states visible reasoning is unavailable for the selected runtime.

## Reproduction Assessment

Yes, at source level: current main emits native Codex reasoning and plan signals, while the Control UI only renders thinking from assistant message content and ignores non-tool plan events. I did not run a live browser session, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear until maintainers choose the safe UX contract. The narrowest maintainable path is likely to render approved Codex plan/tool/lifecycle progress or an explicit unavailable status, not raw reasoning text.
