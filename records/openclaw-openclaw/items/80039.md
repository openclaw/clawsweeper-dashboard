## Summary

Keep open: current main still has a source-level gap between native Codex plan/reasoning events and the Control UI thinking/progress surface. The Codex app-server path emits reasoning callbacks and `stream: "plan"` agent events, but the browser-side handler ignores plan streams and the chat renderer only shows thinking when assistant message content already contains `type: "thinking"` blocks or legacy tags.

## What This Changes

Native Codex runtime should surface safe thinking/progress state in the Control UI instead of only showing the running dots.

## Best Possible Solution

Add a Control UI-native Codex activity surface that consumes approved plan/tool/lifecycle signals and either renders safe summaries or explicitly says visible reasoning is unavailable for the selected runtime.

## Reproduction Assessment

Yes at source level: native Codex emits reasoning and plan signals, while current Control UI rendering only consumes assistant thinking blocks or tags and ignores `stream: "plan"` events. I did not run a live browser/Codex app-server session.

## Solution Assessment

Unclear until maintainers choose the privacy and UX boundary. The narrow maintainable direction is to render only approved Codex plan/tool/lifecycle progress or show an explicit unavailable state, not to dump raw private reasoning text into chat.
