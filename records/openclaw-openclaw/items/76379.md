## Summary

Keep open: the PR targets a real linked CLI bug, but the current patch is not mergeable because manifest-only command ownership would reject documented plugin CLI roots declared through `api.registerCli` descriptors or commands.

## What This Changes

This PR adds a CLI preflight that rejects roots without built-in or manifest command ownership before proxy startup, changes plugin CLI loading to return an empty registry when no manifest owner is found, and updates CLI tests plus changelog.

## Best Possible Solution

Land a narrow CLI fix that fast-fails truly unknown roots before proxy/runtime plugin loading while preserving plugin CLI roots owned by public descriptor, command, metadata, or manifest contracts.

## Reproduction Assessment

Yes, at source level. The linked issue has concrete `openclaw foo` process-group logs, and current `main` still routes unknown roots through proxy startup plus broad plugin CLI loading before command failure.

## Solution Assessment

No, not as written. The fast-fail direction is right, but the patch should consult or preserve documented plugin CLI metadata ownership instead of requiring manifest activation ownership for every valid plugin root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve descriptor-owned plugin CLI roots:** `src/cli/run-main.ts:305-308`
  - body: This preflight only treats built-in roots or `resolveManifestActivationPluginIds` matches as valid, so a third-party plugin that exposes a root through documented `api.registerCli` descriptors or commands but has no manifest `activation.onCommands`/command alias will now throw before metadata discovery can register it. Please keep the `openclaw foo` fast-fail, but preserve descriptor/commands-owned plugin CLI roots or migrate the public contract with coverage.
  - confidence: 0.9
