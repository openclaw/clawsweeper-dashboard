## Summary

Keep open: the PR addresses a real unknown-command startup bug, but the current diff would reject documented plugin CLI roots that are descriptor-owned through `api.registerCli(..., { descriptors })` but not also declared as manifest command aliases or activation commands.

## What This Changes

The PR adds a pre-proxy unknown CLI root check, narrows plugin CLI loading for unowned primaries, adds regression coverage, and records a changelog fix for `openclaw foo`.

## Best Possible Solution

Keep the early failure for truly unknown roots, but base plugin CLI ownership on the documented descriptor/metadata path or update the public manifest contract with migration coverage before removing the broad fallback.

## Reproduction Assessment

Yes. The linked issue gives concrete `openclaw foo` process/output evidence, and current-main source shows unknown roots can start proxy/plugin loading before command rejection; I did not run a live smoke in this read-only review.

## Solution Assessment

No. The PR is directionally right for `openclaw foo`, but the manifest-only ownership check is narrower than the documented plugin CLI descriptor contract and would reject valid descriptor-owned plugin commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve descriptor-owned plugin CLI roots:** `src/cli/run-main.ts:407-410`
  - body: This preflight only treats built-in roots and manifest activation owners as valid, then throws before plugin CLI metadata or runtime registration can run. Please include descriptor-owned CLI roots in the ownership check, or change the public contract with a migration path before failing closed here.
