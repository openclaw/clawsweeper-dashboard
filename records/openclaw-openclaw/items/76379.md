## Summary

Keep open: the PR is pointed at a real `openclaw foo` failure, but the patch would reject existing built-in subcommands such as `openclaw infer ...` because the new preflight uses `cliCommandCatalog` as the built-in ownership source instead of the actual core/sub-CLI registry.

## What This Changes

The PR adds an early CLI preflight for unowned command roots, narrows plugin CLI runtime loading to manifest-owned roots, adds regression tests, and records a changelog fix for #75287.

## Best Possible Solution

Keep the early unknown-root fast-fail, but derive built-in ownership from the actual core/sub-CLI descriptors or make the catalog authoritative and covered for every built-in root before rejecting.

## Reproduction Assessment

Yes. Source inspection shows `openclaw infer ...` is a built-in sub-CLI and a proxied command today, while the PR would reject proxied roots that are absent from `cliCommandCatalog` and unowned by plugin manifests.

## Solution Assessment

No. The proposed fast-fail is the right direction for unknown roots, but using the network-proxy catalog as the ownership list is not the narrowest maintainable fix because it misses registered built-in sub-CLIs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the real CLI ownership set before rejecting roots:** `src/cli/run-main.ts:266`
  - body: This preflight only recognizes roots in `cliCommandCatalog` or plugin manifests. `infer`/`capability` are built-in sub-CLIs registered from `entrySpecs`, and current tests classify `openclaw infer ...` as a proxied command, so this patch would throw before registering that built-in command. Please derive built-in ownership from the actual core/sub-CLI descriptors, or make the catalog complete and covered, before failing unknown roots.
