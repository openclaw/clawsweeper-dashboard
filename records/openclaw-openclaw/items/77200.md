## Summary

Keep this PR open: it targets a real plugin-loader memory supplement bug, but the patch also changes the register-error rollback path so failed plugins can leave prompt/corpus supplements active.

## What This Changes

The PR changes `src/plugins/loader.ts` to merge previous and current memory corpus/prompt supplements during memory-state restore, and adds a changelog entry for #77039.

## Best Possible Solution

Keep the non-activating success-path preservation fix, but leave failed-register rollback as an exact snapshot restore and add regression coverage for both paths.

## Reproduction Assessment

Yes. The review finding is source-reproducible from the existing failing-register test and the changed catch-path merge; I did not run tests because this review is read-only.

## Solution Assessment

No. The intended #77039 fix belongs in the successful non-activating restore path, while the error path should continue discarding memory state registered by a failing plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep failed memory registrations rolled back:** `src/plugins/loader.ts:2382-2395`
  - body: This same merge is applied in the `catch` rollback path. If `register()` calls `registerMemoryPromptSupplement` or `registerMemoryCorpusSupplement` and then throws, `memoryPluginState` still contains the failed plugin's entries, so filtering for plugin ids missing from the snapshot preserves them instead of rolling them back. That breaks the existing loader rollback contract and leaves failed plugins contributing prompt/corpus state.
  - confidence: 0.92
