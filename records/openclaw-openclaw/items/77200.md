## Summary

This PR should stay open because it targets a valid plugin-loader memory supplement bug, but the patch applies the preservation merge to the register-error rollback path too, which would keep failed plugin memory side effects active after the plugin is marked failed.

## What This Changes

The PR changes `src/plugins/loader.ts` to merge previous and current memory corpus/prompt supplements during memory-state restore and adds a changelog entry for #77039.

## Best Possible Solution

Keep the non-activating success-path preservation fix, leave failed-register rollback as an exact snapshot restore, and add focused regression coverage for both paths.

## Reproduction Assessment

Yes. The review finding is source-reproducible from the PR diff and the existing failing-register test that expects memory prompt/corpus state to be cleared after a plugin throws during `register()`.

## Solution Assessment

No. The preservation merge belongs only in the successful non-activating restore path; the catch path should continue discarding memory state created by a failed registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep failed memory registrations rolled back:** `src/plugins/loader.ts:2382-2395`
  - body: This same merge now runs in the `catch` rollback path. If `register()` adds a memory prompt or corpus supplement and then throws, `memoryPluginState` still contains that failed plugin's entries, so the filter appends them back instead of restoring the pre-register snapshot. That breaks the rollback contract and leaves failed plugins contributing memory state.
  - confidence: 0.92
