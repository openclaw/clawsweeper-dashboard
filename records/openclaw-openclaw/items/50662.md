## Summary

Keep open. The restart-auth problem is still real on current main, but this PR is stale and not merge-ready: its restart probe fix captures the caller environment instead of the restart/service environment, and its rate-limit behavior now conflicts with the current documented failover policy and tests.

## What This Changes

The PR adds config-aware auth resolution to restart-health probes and changes embedded-runner rate-limit failover so configured model fallback wins before same-provider auth-profile rotation, with focused regression tests.

## Best Possible Solution

Split or rebase the useful work onto current main: wire restart-health to the existing config-aware probe-auth helper using the restart/service environment and warning path, and treat any rate-limit rotation default change as a separate failover-policy/docs/config update in src/agents/pi-embedded-runner/run/failover-policy.ts.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main's restart-health probe still uses process.env-only credentials, while current failover-policy tests assert rate_limit rotates a profile before model fallback.

## Solution Assessment

No. The restart-auth direction is plausible, but this patch does not thread the correct environment; the rate-limit change also bypasses the current documented one-rotation default instead of updating the current policy seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread restart env through probe auth:** `src/cli/daemon-cli/restart-health.ts:63`
  - body: `confirmGatewayReachable` captures `process.env`, but restart inspection already accepts an `env` for service-specific runtime/config state. Managed installs whose config path or SecretRef env only exists in the service environment can still load or probe with the caller's credentials, preserving the restart timeout this change is meant to fix. Pass the restart env through config/probe-auth resolution.
  - confidence: 0.86
- **[P2] Respect the current rate-limit rotation policy:** `src/agents/pi-embedded-runner/run.ts:166-169`
  - body: The new guard skips auth-profile rotation for every `rate_limit` when a fallback is configured. Current main documents and tests one same-provider rate-limit rotation by default, tunable via `auth.cooldowns.rateLimitedProfileRotations`; landing this without updating the current policy surface would silently override that operator setting. Rebase this into `run/failover-policy.ts` and docs/tests, or drop this half.
  - confidence: 0.81
