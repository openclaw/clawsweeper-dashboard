## Summary

Keep open: the restart-auth bug is still source-reproducible on current main, but this PR is stale against the current failover policy and needs maintainer review before it can merge.

## What This Changes

The PR changes restart-health to resolve config-backed gateway probe auth and adds rate-limit failover guards/tests that skip auth-profile rotation when a model fallback is configured.

## Best Possible Solution

Split or rebase the useful restart-auth fix onto current main using the service env and existing probe-auth warning path, and handle any rate-limit default change separately through run/failover-policy.ts plus docs/tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still probes restart health with process.env-only gateway credentials, while current failover policy/tests/docs intentionally rotate once on rate_limit before model fallback. I did not run tests because this was a read-only review and pnpm is not available in this checkout environment.

## Solution Assessment

No. The restart-auth direction is plausible, but the patch does not thread the restart/service env into probe-auth resolution, and the rate-limit half bypasses the current documented policy seam instead of updating it intentionally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread restart env through probe auth:** `src/cli/daemon-cli/restart-health.ts:63`
  - body: `confirmGatewayReachable` captures `process.env`, but restart inspection already accepts a service `env` for runtime/config state. Managed installs whose config path or SecretRef env only exists in the service environment can still probe with caller credentials, preserving the false restart timeout this change is meant to fix. Pass the restart env into config/probe-auth resolution and keep unresolved SecretRef warnings visible.
  - confidence: 0.86
- **[P2] Respect the rate-limit rotation policy:** `src/agents/pi-embedded-runner/run.ts:166-169`
  - body: The new guard skips auth-profile rotation for every `rate_limit` when a fallback is configured, but current main documents and tests one same-provider retry controlled by `auth.cooldowns.rateLimitedProfileRotations`. Rebase this into `run/failover-policy.ts` with matching docs/tests, or drop the policy change from this PR.
  - confidence: 0.82
