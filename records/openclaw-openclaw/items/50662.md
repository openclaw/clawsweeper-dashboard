## Summary

Keep open; the restart-auth fix remains useful, but this PR is stale against current main and changes the documented rate-limit fallback policy, so it should not merge as-is.

## What This Changes

The PR changes restart health probing to resolve gateway auth from config and changes embedded runner rate-limit failover guards/tests to prefer model fallback over same-provider auth-profile rotation when a fallback model is configured.

## Best Possible Solution

Rebase or split the restart-auth fix so restart probes use the managed service env and existing config-aware probe-auth warning path; handle immediate rate-limit fallback only through the central failover policy, docs, and tests if maintainers want that behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main's restart probe still reads gateway credentials from `process.env`, while current failover policy/tests/docs intentionally rotate once on `rate_limit` before model fallback. I did not execute tests in this read-only review.

## Solution Assessment

No. The restart-auth direction is useful, but the patch resolves from caller `process.env` and changes rate-limit behavior outside the current policy helper/docs; the safer path is a rebase or split through existing seams.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread service env through restart probe auth:** `src/cli/daemon-cli/restart-health.ts:63`
  - body: `confirmGatewayReachable` resolves config/auth from `process.env`, but restart inspection already receives the managed service env and passes it to `service.readRuntime`. Managed installs whose config path or SecretRef inputs exist only in the service env can still probe with caller credentials and report a false timeout; pass the restart env into config/probe-auth resolution and keep unresolved SecretRef warnings visible.
  - confidence: 0.86
- **[P2] Respect the rate-limit rotation policy:** `src/agents/pi-embedded-runner/run.ts:166-169`
  - body: This skips profile rotation for every `rate_limit` when a fallback is configured, but current main centralizes and documents rate-limited profile rotations via `auth.cooldowns.rateLimitedProfileRotations` with default `1` and `0` for immediate fallback. Updating the old inline guard would regress users relying on one same-provider retry; rebase into `run/failover-policy.ts` and matching docs/tests, or drop this behavior change from the PR.
  - confidence: 0.84
