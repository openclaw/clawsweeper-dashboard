## Summary

Keep open: current main and the latest release still enforce the memory-flush target path and append-only mode, but they do not bound, validate, dedupe, roll back, or expose a stable validation contract for the appended payload.

## What This Changes

The issue requests runtime-level guardrails for pre-compaction memory-flush appends so oversized, duplicated, or invalid daily-memory content is rejected or sanitized before mutation.

## Best Possible Solution

Define a narrow memory-flush append contract at the runtime/plugin boundary, starting with conservative built-in bounds, model-facing rejection, safe telemetry, and focused tests before adding broader user-configurable hooks.

## Reproduction Assessment

Yes. Source inspection on current main shows the allowed memory-flush write path appends any string content after only target-path validation; I did not run a live long-session compaction recurrence in this read-only review.

## Solution Assessment

Unclear as proposed. The narrow guardrail is valid, but the issue mixes built-in bounds, optional validator hooks, rollback behavior, silent-mode enforcement, and telemetry, so maintainers should choose the minimal stable contract before code is queued.
