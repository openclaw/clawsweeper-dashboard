## Summary

Keep open. Current main and the latest release still enforce only the append-only target path for memory-flush writes; the requested bounded size/line validation, duplicate/validator contract, rollback or reject policy, and telemetry are not implemented, and no canonical open PR supersedes this issue.

## What This Changes

The issue requests runtime-level bounded, validated, append-only semantics for pre-compaction memory flush writes to daily memory files.

## Best Possible Solution

Define a narrow maintainer-approved memory-flush append contract that rejects oversized payloads before mutation with conservative built-in defaults, preserves current silent-turn suppression, and separately decides whether validator hooks belong in the memory plugin API.

## Reproduction Assessment

No, not for the full live Feishu long-session incident in this read-only review. The source-level gap is clear: current main appends memory-flush content without bounded size/line validation at the wrapper, plan, or config contract.

## Solution Assessment

Unclear as proposed. Built-in bounded append rejection looks like the narrowest first step, but validator hooks, rollback policy, duplicate semantics, telemetry, and any config/API surface need maintainer product direction before an implementation PR.
