## Summary

Keep open: current main and v2026.6.9 still enforce the target path and append-only write mode, but they do not bound, validate, dedupe, roll back, or expose a stable validation hook for memory-flush append payloads. The remaining work is real, but the contract and security boundary need maintainer direction before an implementation PR is safe.

## What This Changes

This issue requests hard runtime guardrails for pre-compaction memory-flush appends, including size/line bounds, validation or rollback, silent-mode enforcement, and telemetry.

## Best Possible Solution

Define a narrow memory-flush append contract at the runtime/plugin boundary: keep the current single-target append-only guard, add conservative built-in payload limits and safe rejection telemetry first, and only add a validator hook or config surface after maintainers choose the security and upgrade model.

## Reproduction Assessment

Yes at source level. Current main accepts any string content for the memory-flush write wrapper and appends it directly to the planned daily memory file after only a target-path check; I did not rerun the reporter's long Feishu session live.

## Solution Assessment

Unclear. The requested direction is valid, but the broad proposed config and validator hook are not automatically the best fix until maintainers choose the strict defaults, hook boundary, and telemetry/rollback behavior.
