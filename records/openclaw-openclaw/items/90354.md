## Summary

Keep open: current main and the latest release still protect the target path and append mode, but they do not bound, validate, dedupe, roll back, or expose a stable validation hook for memory-flush append payloads.

## What This Changes

Request to add runtime guardrails for pre-compaction memory flush writes so daily-memory appends are small, valid, silent, and observable.

## Best Possible Solution

Define a narrow memory-flush append contract at the runtime/plugin boundary, preferably starting with built-in conservative bounds, model-facing rejection, safe telemetry, and tests before adding broader user-configurable hooks.

## Reproduction Assessment

Yes, source-reproducible: current main accepts any string content for the allowed memory-flush append path after only checking the target file path. I did not run a live long-session compaction repro in this read-only review.

## Solution Assessment

Unclear as proposed: the narrow need is valid, but the issue mixes built-in bounds, optional validator hooks, rollback behavior, silent-mode enforcement, and telemetry. The best fix needs maintainer agreement on the minimal stable contract before code is queued.
