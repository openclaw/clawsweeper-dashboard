## Summary

Keep open. Current main has the adjacent append-only target-path guard, but it does not implement bounded append size, line validation, post-write validation, rollback/reject policy, or hard silent-mode enforcement for memory-flush writes.

## What This Changes

The issue requests bounded and validated append semantics for pre-compaction memory flush writes, including size limits, optional validation, silent-mode enforcement, and structured telemetry.

## Best Possible Solution

Keep this open for maintainer design of a narrow core/plugin memory-flush guardrail that rejects oversized append content before mutation, defines any validator hook contract, and avoids unnecessary config surface.

## Reproduction Assessment

No for the full observed Feishu incident; I did not run a live long-session memory flush. Yes for the missing capability from source inspection: current main has append-only path enforcement but no max append length/line validation or post-write validator path.

## Solution Assessment

Unclear as proposed: the core problem is valid, but the best fix likely needs a smaller maintainer-approved contract than the full example config block, especially for validator hooks, command execution, rollback behavior, and plugin ownership.
