## Summary

Keep open. Current main still lacks the XML `</arg_value>` suffix cleanup and the linked Windows/Qwen bug remains open; this PR is a plausible source fix, but it is not landable as-is because it targets the removed `normalizeToolParams()` path and its record-wide stripping can corrupt legitimate write/edit payload text.

## What This Changes

The PR adds a `stripXmlArgValueSuffix` helper, applies it to legacy normalized tool params and exec `command`, and adds unit tests for malformed `</arg_value>` suffixes.

## Best Possible Solution

Land a rebased, field-scoped fix on current main: strip the malformed suffix at the active tool-argument boundary for read path and exec string options, preserve write/edit content and edit text, add regression and negative tests, and include the required changelog credit.

## Reproduction Assessment

Yes for source-level reproduction: #48780 gives the affected Windows/Qwen setup and corrupted `exec`/`read` examples, and current main forwards raw command/path strings without any sanitizer. Live provider reproduction was not run in this read-only review.

## Solution Assessment

No. The PR's defensive direction is right, but the implementation is not the best fix because it strips every string value and targets a removed normalizer; the maintainable solution is a current-main, field-scoped sanitizer with preservation tests for legitimate XML-like payload text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restrict XML stripping to command/path fields:** `src/agents/pi-tools.params.ts:109`
  - body: This helper runs over every string after `content`, `oldText`, and `newText` have been normalized, so a valid file body or edit text ending in `</arg_value>` is truncated before write/edit executes. Scope stripping to affected command/path-like fields and add negative tests that preserve payload text.
  - confidence: 0.91
- **[P2] Sanitize exec string options consistently:** `src/agents/bash-tools.exec.ts:232`
  - body: The exec handler only fixes `command`, but `workdir`, `host`, `security`, `ask`, and `node` are also string params consumed later. If the provider suffix lands in one of those values, the call can still fail or resolve the wrong option, so strip the suffix from the exec string options before validation/resolution.
  - confidence: 0.78
- **[P3] Allow any trailing XML suffix chevrons:** `src/agents/pi-tools.params.ts:96`
  - body: The regex only matches up to three total `>` characters after `arg_value`, so a value ending with `</arg_value>>>>` leaves residual characters. Use a `>*` suffix matcher to strip the whole malformed closing tag.
  - confidence: 0.84
