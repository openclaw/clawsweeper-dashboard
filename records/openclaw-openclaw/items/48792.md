## Summary

Keep open. The linked bug still appears valid and current main has no `</arg_value>` sanitizer, but this PR needs rework before merge because it targets a stale normalization path, strips too broadly, and leaves exec string options uncovered.

## What This Changes

The PR adds malformed `</arg_value>` suffix stripping to tool parameter normalization and exec command handling, with unit tests for the helper.

## Best Possible Solution

Land a rebased, field-scoped fix on current main: strip the malformed suffix only at active command/path-like argument boundaries, cover exec string options, preserve write/edit payload text, add regression and negative tests, and include the required changelog entry.

## Reproduction Assessment

Yes for source-level reproduction: #48780 gives the affected Windows/Qwen setup and corrupted `exec`/`read` examples, and current main forwards raw command/path strings without a sanitizer. Live provider reproduction was not run during this read-only review.

## Solution Assessment

No. The defensive direction is right, but this implementation is not the best fix because it uses a stale normalizer, mutates all string params, and only sanitizes exec `command`; the safer path is a current-main field-scoped sanitizer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Move stripping to the active current-main boundary:** `src/agents/pi-tools.params.ts:140`
  - body: `normalizeToolParams()` is no longer the active read/write/edit boundary on current main; those tools validate and forward params through `wrapToolParamValidation()`. Rebase the fix onto the current wrappers instead of relying on this removed normalizer, otherwise the PR will not address the current code path cleanly.
  - confidence: 0.9
- **[P1] Restrict XML stripping to command and path fields:** `src/agents/pi-tools.params.ts:140`
  - body: This record-wide call runs after `content`, `oldText`, and `newText` are normalized, so valid write/edit payload text that ends with `</arg_value>` can be truncated before execution. Scope suffix stripping to affected command/path-like fields and add negative tests that preserve payload text.
  - confidence: 0.91
- **[P2] Sanitize exec string options consistently:** `src/agents/bash-tools.exec.ts:230-232`
  - body: The exec handler only strips `command`, but `workdir`, `host`, `security`, `ask`, and `node` are also string params consumed later. If the provider suffix lands in one of those values, exec can still fail or resolve the wrong option, so normalize the full exec string option set before validation/resolution.
  - confidence: 0.8
- **[P3] Allow any trailing XML suffix chevrons:** `src/agents/pi-tools.params.ts:96`
  - body: The regex only matches up to three total `>` characters after `arg_value`, so a value ending with `</arg_value>>>>` leaves residual characters. Use an unbounded trailing-chevron suffix matcher for the malformed closing tag.
  - confidence: 0.84
