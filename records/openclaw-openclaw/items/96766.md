## Summary

Keep open: current main and the latest release still swallow rejected close-time pending sync and provider-initialization promises with empty catches, and the issue is already paired with open fix PRs that should be reviewed instead of closing the report early.

## What This Changes

The issue asks for close-time pending memory sync and provider-initialization rejections to be reported instead of silently swallowed by `awaitPendingManagerWork`.

## Best Possible Solution

Land one canonical close-time diagnostic fix, preferably the proof-positive candidate if maintainers agree, preserve non-throwing shutdown, then close this issue as implemented and retire the duplicate PR.

## Reproduction Assessment

Yes, at source level: current main still catches both rejected pending close-time promises with empty `catch {}` blocks, and `MemoryIndexManager.close()` uses that helper. I did not run a live shutdown scenario that forces an actual SQLite, disk, or provider failure.

## Solution Assessment

Yes: the narrow maintainable fix is to thread an error-reporting callback or logger through `awaitPendingManagerWork` while preserving best-effort, non-throwing close behavior. The issue should remain open until one linked fix PR lands.
