## Summary

Keep open: current main and the latest release still swallow rejected pending sync and provider-initialization work during memory manager close, and the two linked fixing PRs are still open rather than merged.

## What This Changes

The issue reports that memory-core manager close waits for pending sync/provider-initialization promises but discards their rejections with empty catches.

## Best Possible Solution

Land one canonical close-time diagnostic fix that routes rejected pending sync/provider-init promises through the existing memory logger while preserving non-throwing shutdown, then retire the duplicate candidate PR.

## Reproduction Assessment

Yes, at source level: current main still catches both rejected pending close-time promises with empty catches, and MemoryIndexManager.close uses that helper. I did not run a live SQLite, disk-full, or provider-outage shutdown scenario.

## Solution Assessment

Yes: a narrow helper/caller diagnostic path is the best fix shape because it preserves best-effort close behavior while making failures visible. Throwing these close-time errors would be a broader shutdown behavior change.
