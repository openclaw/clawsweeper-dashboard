## Summary

Keep open: current main and the latest release still swallow rejected close-time pending sync/provider-init promises, and two open PRs still reference this issue with closing syntax.

## What This Changes

The issue requests reporting or logging errors from pending memory sync and provider initialization during `MemoryIndexManager.close()` instead of discarding them with empty catches.

## Best Possible Solution

Land one canonical close-time diagnostic fix that preserves non-throwing shutdown behavior, then close this issue and retire duplicate fix PRs.

## Reproduction Assessment

Yes, at source level: current main still catches both rejected pending close-time promises with empty catches, and `MemoryIndexManager.close()` uses that helper. I did not run a live SQLite lock, disk-full, or provider-outage shutdown scenario.

## Solution Assessment

Yes: routing these close-time rejections to the existing memory logger or an equivalent callback is the narrow maintainable fix. Throwing from close would be a broader behavior change than needed.
