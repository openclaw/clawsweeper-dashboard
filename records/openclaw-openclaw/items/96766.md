## Summary

Keep open: current main and the latest release still silently swallow rejected pending sync and provider-initialization work during memory manager close, and two open PRs now explicitly target this issue. The issue should stay open until one canonical fix PR lands.

## What This Changes

The issue requests diagnostic reporting for errors currently swallowed by `awaitPendingManagerWork` during memory-core close-time pending sync and provider initialization waits.

## Best Possible Solution

Land one canonical close-time diagnostic fix that preserves non-throwing shutdown, preferably by reviewing the proof-positive path in https://github.com/openclaw/openclaw/pull/96787 and retiring or rebasing the duplicate https://github.com/openclaw/openclaw/pull/96781.

## Reproduction Assessment

Yes, at source level: current main still catches both rejected pending close-time promises with empty catches, and `MemoryIndexManager.close()` uses that helper. I did not run a live shutdown scenario that forces an actual SQLite, disk, or provider failure.

## Solution Assessment

Yes: routing the rejected pending-work errors through an existing memory logger path while preserving non-throwing close behavior is the narrow maintainable fix. The issue itself should remain open until one linked fix PR merges.
