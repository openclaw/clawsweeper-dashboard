## Summary

Keep open: current main still catches rejected close-time pending sync and provider initialization promises with empty `catch {}` blocks, and the related close/sync race fix only addressed teardown ordering, not silent error reporting.

## What This Changes

The issue reports that pending memory sync and provider initialization errors are swallowed during manager close instead of being logged.

## Best Possible Solution

Thread an error-reporting callback or equivalent memory logger through `awaitPendingManagerWork`, log both pending sync and provider initialization failures during close, and preserve the existing close cleanup/teardown ordering.

## Reproduction Assessment

Yes, at source level: current main still catches both rejected pending close-time promises with empty catch blocks in `awaitPendingManagerWork`. I did not run a live shutdown scenario that forces a real SQLite, disk, or provider failure.

## Solution Assessment

Yes, a narrow logging/error-callback repair in the helper and its close callers is the maintainable fix. Throwing these pending-work errors from close would be a larger behavior change and is not needed to make the failure diagnosable.
