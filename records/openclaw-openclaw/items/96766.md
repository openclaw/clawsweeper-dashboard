## Summary

Keep open: current main and the latest release still swallow both rejected close-time pending sync and provider-initialization promises with empty catches, while adjacent memory sync paths log failures.

## What This Changes

The issue asks OpenClaw to stop silently discarding close-time memory pending sync/provider initialization failures.

## Best Possible Solution

Thread an error-reporting callback or memory logger through `awaitPendingManagerWork`, log both rejected pending-work paths during close, and preserve the existing close ordering.

## Reproduction Assessment

Yes at source level: current main still catches both rejected pending close-time promises with empty `catch {}` blocks. I did not run a live shutdown scenario that forces an actual SQLite, disk, or provider failure.

## Solution Assessment

Yes: a narrow helper/caller logging repair is the maintainable fix. Throwing these close-time errors would be a larger behavior change and is not needed to make the failure diagnosable.
