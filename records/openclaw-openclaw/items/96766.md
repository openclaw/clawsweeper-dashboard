## Summary

## What This Changes

The issue asks for `awaitPendingManagerWork` to report rejected pending memory sync and provider initialization work during manager close instead of silently discarding it.

## Best Possible Solution

Land one canonical memory-core fix that preserves non-throwing close while routing rejected pending sync and provider-init errors to existing diagnostics, then close the duplicate fix PRs and this issue.

## Reproduction Assessment

Yes, at source level: current `main` has empty catches for both pending close-work promises and `MemoryIndexManager.close()` calls that helper. I did not run a live disk-full, SQLite-lock, or provider-outage shutdown scenario.

## Solution Assessment

Yes: adding an error-reporting callback and logging through the existing memory logger is the narrow maintainable fix because it preserves best-effort non-throwing close behavior.
