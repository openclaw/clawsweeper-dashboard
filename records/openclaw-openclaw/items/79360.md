## Summary

Keep open: current main still uses a stateless `image_generate` contract for the OpenAI Codex Responses image path, and the requested continuation IDs plus follow-up inputs are not implemented. This is a useful feature request, but it needs an additive API/privacy design rather than cleanup closure.

## What This Changes

Expose OpenAI Responses image-generation continuation metadata and inputs so `image_generate` can support iterative image edits across turns.

## Best Possible Solution

Design an additive opt-in continuation contract that captures safe Responses/image-generation IDs, exposes them in `image_generate.details`, accepts explicit continuation inputs, preserves stateless defaults, and updates tests/docs.

## Reproduction Assessment

Yes for source-level reproduction: current main’s schema/provider path has no continuation inputs and drops response/call IDs from the Codex image-generation result. I did not run a live Codex OAuth image request, and this is a feature request rather than a broken documented behavior.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the best fix needs maintainer API design around shared versus OpenAI-specific metadata and `store`/privacy behavior.
