## Summary

Keep open: current main still has a stateless `image_generate` contract for the OpenAI Codex Responses image path, so the requested continuation IDs and follow-up inputs are not implemented and need maintainer API/product design.

## What This Changes

The issue requests an opt-in Responses continuation path for OpenAI image generation so `image_generate` can return response/tool-call IDs and reuse them for iterative edits.

## Best Possible Solution

Design an additive opt-in continuation contract that captures safe Responses/image-generation IDs from the OpenAI Codex path, exposes them in tool details, accepts explicit continuation inputs, preserves stateless defaults, and updates tests/docs.

## Reproduction Assessment

Yes, at source level: current main has no continuation inputs and does not preserve response or image-generation call IDs in the Codex image path. I did not establish a live failing bug repro, and this is a new capability rather than broken documented behavior.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the best fix needs maintainer API design around generic versus OpenAI-specific metadata and `store`/privacy behavior.
