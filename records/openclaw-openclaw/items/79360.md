## Summary

Keep open: current main still does not expose OpenAI Responses image-generation continuation IDs or accept continuation/edit inputs on `image_generate`, and the remaining work is an additive API/privacy/product decision rather than an already-implemented bug fix.

## What This Changes

The issue requests an opt-in stateful OpenAI Responses image editing path that returns response/tool-call IDs and accepts continuation inputs such as `previousResponseId`, edit action, and storage controls.

## Best Possible Solution

Design an additive, opt-in continuation contract that captures safe OpenAI Responses/image-generation IDs, surfaces them in `image_generate.details`, accepts explicit continuation inputs, keeps stateless defaults, and updates provider/tool tests plus public docs.

## Reproduction Assessment

Yes, at source level: current main has no continuation inputs in the tool schema/provider request path and drops response/call IDs from the parsed Codex image-generation result. This is a feature gap rather than a failing documented behavior repro.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is plausible, but the best solution should settle generic versus OpenAI-specific fields and storage/privacy defaults before implementation.
