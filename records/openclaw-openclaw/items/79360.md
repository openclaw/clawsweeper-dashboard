## Summary

Keep open: current main still implements the OpenAI Codex image path as a stateless `image_generate` flow, so the requested continuation IDs and follow-up inputs are not present and require maintainer API/privacy design rather than cleanup closure.

## What This Changes

Expose OpenAI Responses image generation response/call IDs and accept opt-in continuation/edit inputs in `image_generate` for iterative image edits.

## Best Possible Solution

Design an additive opt-in continuation contract that captures safe Responses/image-generation IDs from the OpenAI Codex path, exposes them in tool details, accepts explicit continuation inputs, preserves stateless defaults, and updates provider/tool tests plus public docs.

## Reproduction Assessment

Yes for source-level reproduction: current main’s schema, provider request, parser, tests, and docs lack continuation inputs and strip response/call IDs. I did not establish a live failing bug repro, and this is a new capability rather than broken documented behavior.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the best implementation needs maintainer API design around shared versus OpenAI-specific metadata and `store`/privacy semantics.
