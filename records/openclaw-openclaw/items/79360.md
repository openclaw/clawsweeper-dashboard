## Summary

Keep open: current main and v2026.5.28 still only support stateless or reference-image OpenAI image generation, so exposing and reusing Responses continuation IDs remains additive API/product work rather than an implemented fix.

## What This Changes

Request to expose OpenAI Responses image-generation response/tool-call IDs and accept explicit continuation/edit inputs for iterative `image_generate` workflows.

## Best Possible Solution

Design an explicit provider-aware continuation contract for `image_generate` that surfaces safe OpenAI Responses metadata and accepts opt-in continuation/edit inputs while preserving the current stateless default.

Yes, source-reproducible: current code parses Responses image-generation events but only propagates image data and usage/toolUsage, while the tool schema has no continuation inputs. I did not run a live OpenAI flow because this was a read-only review.

## Solution Assessment

Unclear: exposing safe metadata and explicit continuation inputs is plausible, but maintainers need to choose the shared versus OpenAI-specific API shape and storage/privacy semantics before implementation.
