## Summary

Keep open: current `main` and the latest checked release still expose only stateless `image_generate` behavior for the OpenAI Codex Responses image route, while OpenAI’s current docs support the requested continuation patterns. This is a useful feature/API-shape request, but it needs maintainer product judgment around opt-in storage, safe metadata exposure, and the shared provider/tool contract before a fix PR is queued.

## What This Changes

The issue requests exposing and reusing OpenAI Responses image-generation continuation IDs through `image_generate` so follow-up image edits can use stateful Responses context.

## Best Possible Solution

Design an opt-in OpenAI Responses continuation path that preserves today’s stateless default, safely exposes response and image-generation call IDs in `image_generate.details`, accepts explicit continuation/action inputs, and documents/tests the storage and upgrade behavior.

## Reproduction Assessment

Yes, at source level: current `main` shows the Codex image route parses image-generation calls but drops response/call IDs, sends `store: false`, and exposes no continuation input in the tool schema. I did not run a live OpenAI image call in this read-only review.

## Solution Assessment

Unclear pending maintainer product direction: the requested opt-in shape is plausible and matches the OpenAI API, but the best permanent solution needs a decision on safe storage defaults and whether the continuation contract is generic or OpenAI-specific.
