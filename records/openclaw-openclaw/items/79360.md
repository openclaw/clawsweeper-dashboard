## Summary

Keep open: current main still has a stateless OpenAI Codex Responses image path for `image_generate`, and this request needs an additive tool/API design rather than cleanup closure. The related Responses built-in tool visibility issue remains broader and does not supersede this narrower image-edit continuation request.

## What This Changes

Expose opt-in Responses image continuation metadata and follow-up inputs in `image_generate` so OpenAI Codex OAuth image generations can be iteratively edited across turns.

## Best Possible Solution

Design an additive opt-in `image_generate` continuation contract that captures safe Responses/image-generation IDs, exposes them in details, accepts explicit follow-up inputs, preserves stateless defaults, and updates tests and docs.

## Reproduction Assessment

Yes, at source level: current main sends `store: false` without continuation inputs, strips response/call IDs from parsed events, and exposes only generate/list plus reference-image edits. I did not establish a live failing bug path, and this is a new feature rather than broken documented behavior.

## Solution Assessment

Unclear because there is no implementation yet. The requested direction is plausible, but the best fix needs maintainer API design around generic versus OpenAI-specific continuation metadata and storage/privacy semantics.
