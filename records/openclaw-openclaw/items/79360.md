## Summary

Keep open: current main and the latest release still expose only stateless OpenAI Codex Responses image generation through image_generate, while the requested continuation IDs and continuation inputs remain absent. This is a useful feature/API-shape request, but it needs maintainer product direction before an automated fix PR because it changes storage, metadata exposure, and shared tool/provider contract boundaries.

## What This Changes

Expose Responses image generation continuation metadata and explicit opt-in continuation inputs through image_generate for iterative OpenAI image edits.

## Best Possible Solution

Design an opt-in OpenAI Responses continuation path that keeps today’s stateless default, safely exposes response and image-generation call IDs in image_generate.details, accepts explicit continuation/action inputs, and documents/tests storage and upgrade behavior.

## Reproduction Assessment

Yes, at source level: current main still parses image-generation calls but drops response/call IDs, sends store: false, and exposes no continuation input in the tool schema. I did not run a live OpenAI image call in this read-only review.

## Solution Assessment

Unclear pending maintainer product direction: the requested opt-in continuation path is plausible and matches the OpenAI API, but the best permanent solution needs a decision on safe storage defaults and generic versus OpenAI-specific contract shape.
