## Summary

Keep open: current main still has a stateless OpenAI Codex Responses image-generation path, so the requested continuation IDs and follow-up inputs are not implemented and need additive API/product design.

## What This Changes

The issue requests opt-in Responses image-generation continuation support for image_generate, including exposed response/tool-call IDs and follow-up inputs for iterative image edits.

## Best Possible Solution

Design an additive opt-in continuation contract that safely captures Responses/image-generation IDs, exposes them in image_generate details, accepts explicit continuation inputs, preserves stateless defaults, and updates tests/docs.

## Reproduction Assessment

Yes, at source level: current main’s OpenAI Codex image path sends stateless Responses requests and the schema/result path lacks continuation inputs or returned IDs. I did not run a live OpenAI image request, and this is a feature request rather than a broken documented behavior.

## Solution Assessment

Unclear until maintainers choose the API contract. The requested direction is plausible, but the best implementation needs a decision on generic versus OpenAI-specific fields and storage/privacy semantics before coding.
