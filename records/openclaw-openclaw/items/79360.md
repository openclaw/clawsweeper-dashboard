## Summary

Keep open: current main and the latest release still do not expose or reuse OpenAI Responses image-generation continuation handles through `image_generate`. The request is valid at source level, but it remains a feature/API-shape decision rather than a narrow bug fix because it affects metadata exposure, storage/retention defaults, and shared provider/tool contracts.

## What This Changes

The issue requests opt-in `image_generate` support for returning OpenAI Responses image-generation IDs and accepting continuation inputs for iterative image edits.

## Best Possible Solution

Design an opt-in continuation path that preserves today’s stateless default, safely exposes response and image-generation call IDs, accepts explicit continuation/action inputs, and documents/tests storage, retention, and upgrade behavior.

## Reproduction Assessment

Yes, at source level: current main still parses image-generation calls but drops response/call IDs, sends `store: false`, and exposes no continuation input in the tool schema. I did not run a live OpenAI image call in this read-only review.

## Solution Assessment

Unclear pending maintainer product direction: an opt-in continuation path is plausible, but the best permanent shape depends on safe metadata exposure, storage defaults, and generic versus OpenAI-specific contract boundaries.
