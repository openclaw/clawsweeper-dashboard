## Summary

Keep open: current main still has a stateless OpenAI Codex Responses image path for `image_generate`; it extracts generated image bytes and usage metadata but does not expose continuation IDs or accept follow-up continuation inputs. The request is a plausible additive feature that needs maintainer API/product design, especially around storage and privacy semantics.

## What This Changes

Request to add an opt-in stateful OpenAI Responses image-generation path that returns continuation IDs and accepts follow-up continuation inputs through `image_generate`.

## Best Possible Solution

Design a backward-compatible, opt-in continuation contract that safely surfaces Responses/image-generation IDs, accepts explicit continuation inputs, keeps stateless defaults, and updates provider/runtime/tool tests plus public docs.

## Reproduction Assessment

Yes, source-level only: current main’s Codex path posts stateless Responses requests and returns only images/usage metadata, so the missing surface is clear from code and tests. This is a feature request rather than a regression in a documented contract.

## Solution Assessment

Unclear because there is no patch yet. The requested direction aligns with the upstream Responses image-generation API, but the maintainable solution needs API design for shared versus OpenAI-specific fields and storage/privacy behavior.
