## Summary

Keep open. Current main still emits the legacy `Path escapes sandbox root (...)` message without the container-visible workspace mapping, while the Docker sandbox bridge already has the host root and container workdir needed to produce the requested recovery hint; the related closed items cover generated host paths, not this diagnostic path.

## What This Changes

Improve sandbox file-tool escape errors so rejected host-side paths include the sandbox host-root to container-workdir mapping and a recoverable path hint.

## Best Possible Solution

Keep sandbox enforcement unchanged, but format the escape diagnostic from the sandbox fs bridge with the rejected path, sandbox host root, container mount path, and a concise hint to use the container path.

## Reproduction Assessment

Yes, via source inspection. A focused current-main repro is `resolveSandboxFsPathWithMounts` with an input like `/tmp/healthcheck-alert/config.json`, a sandbox workspace root, `defaultContainerRoot: "/workspace"`, and normal mounts; it reaches the legacy `Path escapes sandbox root` branch without using the container mapping.

## Solution Assessment

Yes, the narrowest maintainable solution is a diagnostic-only change in the sandbox file path resolver or its caller. It should not auto-translate arbitrary host paths, weaken sandbox boundaries, or add a config knob.
