## Summary

Keep open: current main still has a source-reproducible MiniMax VLM SSRF sink, but this branch needs rework before merge because it drops env-proxy routing, timeout enforcement, and audit context, and it lacks real behavior proof.

## What This Changes

The PR replaces the MiniMax VLM raw POST with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the returned guarded dispatcher after response handling.

## Best Possible Solution

Rework the MiniMax VLM request to use guarded fetch with MiniMax audit context, timeout forwarding, release cleanup, and trusted env-proxy mode only when the existing proxy gate permits it, then add focused coverage and redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main builds the MiniMax VLM URL from configurable host inputs and sends `Authorization: Bearer` plus image data through raw `fetch()`. I did not use live MiniMax credentials or private-host calls in this read-only review.

## Solution Assessment

No. SSRF-guarded fetch is the right primitive, but this diff hard-codes strict mode and omits timeout and audit context; the maintainable fix should follow the established guarded provider transport pattern.
