## Summary

Keep open: the MiniMax VLM SSRF concern is still real on current main, but this branch is not merge-ready because it drops current proxy and timeout behavior, omits audit context, and lacks real behavior proof.

## What This Changes

The PR routes MiniMax VLM image-understanding requests through strict SSRF-guarded fetch and releases the guarded dispatcher after response handling.

## Best Possible Solution

Rework MiniMax VLM to use guarded fetch with auditContext, timeout forwarding, release cleanup, strict direct mode, and TRUSTED_ENV_PROXY only when shouldUseEnvHttpProxyForUrl permits it, plus focused SSRF/proxy/timeout tests and real proof.

## Reproduction Assessment

Yes, by source inspection rather than live execution: current main derives the MiniMax VLM target from configurable host inputs and sends the bearer token and image body with raw fetch. The PR regressions are also source-backed by current proxy, timeout, and guarded-fetch contracts.

## Solution Assessment

No. SSRF-guarded fetch is the right primitive, but strict-only guarded fetch is not the best fix because current main has established env-proxy and timeout contracts that the branch drops.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve env-proxy routing for MiniMax VLM:** `src/agents/minimax-vlm.ts:80-81`
  - body: Hard-coding `withStrictGuardedFetchMode` ignores the existing env-proxy path unless the managed proxy lifecycle is active. Current main explicitly enables env-proxy dispatch before MiniMax VLM fetches, so proxy-only/operator-egress environments can fail or bypass the intended route; select trusted env-proxy mode when `shouldUseEnvHttpProxyForUrl(url)` permits it and strict mode otherwise.
  - confidence: 0.9
- **[P2] Keep the MiniMax VLM request timeout:** `src/agents/minimax-vlm.ts:80-81`
  - body: Current main computes the caller/default timeout and passes `AbortSignal.timeout(timeoutMs)` into the outbound request. This guarded fetch call passes neither `timeoutMs` nor a signal, so the rebased fix would drop MiniMax VLM timeout behavior and can hang on slow or stuck API calls.
  - confidence: 0.84
- **[P3] Set a MiniMax audit context:** `src/agents/minimax-vlm.ts:80`
  - body: The new guarded fetch call omits `auditContext`, so SSRF-blocked warnings fall back to the generic `url-fetch` label instead of identifying MiniMax VLM. Add an explicit context such as `minimax-vlm` so blocked security events are attributable.
  - confidence: 0.77
