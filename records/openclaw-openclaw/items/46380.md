## Summary

Keep open: the MiniMax VLM endpoint on current `main` still uses an unguarded outbound fetch with bearer credentials, but this PR is stale and would regress current env-proxy and timeout behavior if merged as submitted.

## What This Changes

The PR modifies `src/agents/minimax-vlm.ts` to replace the raw MiniMax VLM `fetch()` call with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and release the guarded dispatcher after response handling.

## Best Possible Solution

Rebase the hardening on current `main` so MiniMax VLM uses the shared SSRF guard with audit context, preserves env-proxy-compatible routing and timeout behavior, and adds focused regression coverage for blocked private hosts and normal public MiniMax calls.

## Reproduction Assessment

Yes from source inspection: current `main` accepts caller-controlled MiniMax VLM hosts and sends bearer credentials plus image data through plain `fetch()`. A concrete repro would set `apiHost` or `modelBaseUrl` to a private/metadata address and observe that the current code does not invoke the SSRF guard before fetch.

## Solution Assessment

No. The PR fixes the right class of issue, but `withStrictGuardedFetchMode()` is not the best current-main implementation because it drops established env-proxy compatibility and the submitted diff predates the timeout code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve MiniMax env-proxy routing:** `src/agents/minimax-vlm.ts:80`
  - body: Current `main` explicitly initializes env-proxy support before MiniMax VLM calls and applies a request timeout. Routing this request through strict guarded mode ignores `HTTP_PROXY`/`HTTPS_PROXY` unless OpenClaw's managed proxy lifecycle is active, so proxy-only deployments that work today can no longer reach MiniMax. Rebase the fix to keep the timeout and use the guard mode/policy that preserves trusted env-proxy routing for this operator-configured provider endpoint.
  - confidence: 0.86
