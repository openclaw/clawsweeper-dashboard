## Summary

Keep open: current main still has the configurable raw MiniMax VLM fetch, but this PR is not mergeable as-is because it drops current proxy and timeout behavior and still lacks real behavior proof.

## What This Changes

The PR replaces the MiniMax VLM raw fetch with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the guarded dispatcher after response handling.

## Best Possible Solution

Rework the MiniMax VLM request to use guarded fetch with MiniMax audit context, timeout forwarding, release cleanup, and trusted env-proxy mode only when `shouldUseEnvHttpProxyForUrl(url)` permits it.

## Reproduction Assessment

Yes, by source inspection: current main builds the MiniMax VLM URL from configurable host inputs and sends `Authorization: Bearer` plus image data through raw `fetch()`. I did not use live MiniMax credentials or private-host calls in this read-only review.

## Solution Assessment

No. SSRF-guarded fetch is the right primitive, but this diff hard-codes strict mode and omits timeout and audit context; the maintainable fix should follow the established provider guarded-fetch pattern.
