## Summary

Keep open. The MiniMax VLM SSRF problem is still real on current main, but this PR is stale against later MiniMax transport changes and is not safe to merge as-is because it bypasses current proxy-compatible egress behavior, drops the current request timeout wiring, and lacks MiniMax-specific audit/test coverage.

## What This Changes

The PR changes `src/agents/minimax-vlm.ts` to replace the MiniMax VLM raw POST fetch with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and release the guarded dispatcher after response handling.

## Best Possible Solution

Revise or replace the PR with a current-main patch that sends MiniMax VLM through the shared guarded provider HTTP helper or an equivalent wrapper, preserves proxy-aware dispatch and request timeouts, adds a MiniMax-specific audit context, and covers private/link-local/metadata hosts for `apiHost`, `MINIMAX_API_HOST`, and `modelBaseUrl`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: set `apiHost`, `MINIMAX_API_HOST`, or configured `modelBaseUrl` to a blocked host such as `http://169.254.169.254`; current main builds that endpoint and raw-fetches it with the bearer token and image body.

## Solution Assessment

No. Direct strict guarded fetch addresses the obvious raw SSRF path, but it is not the best current-main fix because it bypasses proxy-aware provider transport behavior, omits the current timeout, and lacks provider-specific audit/test coverage.

  - body: Blocked SSRF attempts from this provider will log under the generic `url-fetch` context because the guarded fetch call omits `auditContext`. Pass a stable MiniMax label so security logs can identify this provider path.
  - confidence: 0.82
