## Summary

Keep this PR open: the MiniMax VLM SSRF problem is still present on current main, but this branch is not merge-ready because it drops current proxy and timeout behavior and has no real after-fix proof.

## What This Changes

The branch imports the guarded fetch helper into `src/agents/minimax-vlm.ts`, sends MiniMax VLM requests through `withStrictGuardedFetchMode`, and releases the guarded dispatcher after response handling.

## Best Possible Solution

Rework MiniMax VLM to use guarded fetch with `auditContext`, timeout forwarding, `release()` cleanup, strict direct mode, and `TRUSTED_ENV_PROXY` only when `shouldUseEnvHttpProxyForUrl(url)` permits it, plus focused SSRF/proxy/timeout tests and real proof.

## Reproduction Assessment

Yes, by source inspection rather than live execution: current main builds the MiniMax VLM URL from configurable host inputs and sends the API key and image body with raw `fetch`. The proposed regressions are also source-backed by current timeout/proxy code and guarded-fetch tests.

## Solution Assessment

No. The SSRF guard primitive is the right direction, but strict-mode-only is not the best fix because current main has established proxy and timeout contracts that the branch drops.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve env-proxy routing for MiniMax VLM:** `src/agents/minimax-vlm.ts:80-81`
  - body: Hard-coding `withStrictGuardedFetchMode` makes this request ignore normal `HTTP_PROXY`/`HTTPS_PROXY` routing unless the managed proxy lifecycle is active. Current main explicitly enables env-proxy dispatch before MiniMax VLM fetches, so proxy-only/operator-egress environments can fail or bypass the intended route; choose trusted env-proxy mode when `shouldUseEnvHttpProxyForUrl(url)` permits it and strict mode otherwise.
  - confidence: 0.9
- **[P2] Keep the MiniMax VLM request timeout:** `src/agents/minimax-vlm.ts:80-81`
  - body: Current main computes the caller/default timeout and passes `AbortSignal.timeout(timeoutMs)` into the outbound request. This guarded fetch call passes neither `timeoutMs` nor a signal, so the rebased fix would drop MiniMax VLM timeout behavior and can hang on slow or stuck API calls.
  - confidence: 0.84
- **[P3] Set a MiniMax audit context:** `src/agents/minimax-vlm.ts:80`
  - body: The new guarded fetch call omits `auditContext`, so SSRF-blocked warnings will use the generic `url-fetch` label instead of identifying MiniMax VLM. Add an explicit context such as `minimax-vlm` so blocked security events are attributable.
  - confidence: 0.77
