## Summary

Keep open: the SSRF concern is real, but this branch is not merge-ready because it bypasses established proxy routing, drops current timeout behavior, and lacks after-fix real behavior proof.

## What This Changes

The PR replaces MiniMax VLM raw fetch with strict guarded fetch and releases the guarded dispatcher after response handling.

## Best Possible Solution

Rework MiniMax VLM to use guarded fetch with `auditContext`, current timeout handling, `release()` cleanup, strict direct mode, and `TRUSTED_ENV_PROXY` only when `shouldUseEnvHttpProxyForUrl(url)` permits it, plus focused SSRF/proxy/timeout tests.

## Reproduction Assessment

Yes by source inspection: current main builds the MiniMax VLM URL from user/configurable host inputs and sends the API key and image body with raw fetch. The proposed regression is also source-backed by guarded-fetch tests showing strict mode ignores env proxy by default.

## Solution Assessment

No. The SSRF guard family is the right direction, but strict-mode-only is not the narrowest maintainable fix because current MiniMax and shared provider paths preserve timeout and env-proxy behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve env-proxy routing for MiniMax VLM:** `src/agents/minimax-vlm.ts:80-81`
  - body: Hard-coding `withStrictGuardedFetchMode` makes this call ignore normal `HTTP_PROXY`/`HTTPS_PROXY` routing unless the managed proxy lifecycle is active. Current main explicitly enables env-proxy dispatch before MiniMax VLM fetches, and proxy-only deployments would fail or bypass operator egress; select trusted env-proxy mode when `shouldUseEnvHttpProxyForUrl(url)` qualifies and keep strict mode otherwise.
  - confidence: 0.9
- **[P2] Keep the MiniMax VLM request timeout:** `src/agents/minimax-vlm.ts:80-81`
  - body: Current main computes `timeoutMs` and passes `AbortSignal.timeout(timeoutMs)` into the outbound request, with coverage for caller-provided timeouts. This guarded fetch call does not pass `timeoutMs` or a signal, so the rebased fix would drop the MiniMax VLM timeout behavior; thread the same timeout into `fetchWithSsrFGuard`.
  - confidence: 0.84
