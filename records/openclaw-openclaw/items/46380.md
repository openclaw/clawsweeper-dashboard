## Summary

Keep open. The PR addresses a real credentialed MiniMax VLM SSRF surface on current main, but the proposed branch is stale and not safe to merge because it bypasses current proxy-aware egress behavior, drops the MiniMax request timeout, and omits provider-specific SSRF audit context.

## What This Changes

The PR replaces the MiniMax VLM raw POST with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the guarded dispatcher after response handling.

## Best Possible Solution

Revise the fix to route MiniMax VLM through the shared guarded provider HTTP transport or an equivalent wrapper that preserves env-proxy selection, request timeout, audit context, and existing error handling, with tests for blocked private/link-local `apiHost`, `MINIMAX_API_HOST`, and `modelBaseUrl`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: set `apiHost`, `MINIMAX_API_HOST`, or configured `modelBaseUrl` to a blocked host such as `http://169.254.169.254`; current main constructs that endpoint and raw-fetches it with the bearer token and image body.

## Solution Assessment

No. Direct strict guarded fetch addresses the obvious raw SSRF path, but it is not the best current-main fix because it bypasses proxy-aware provider transport behavior, omits the current timeout, and lacks provider-specific audit/test coverage.

  - confidence: 0.82
