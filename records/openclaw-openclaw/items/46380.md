## Summary

Keep open: the MiniMax VLM SSRF sink is still present on current main, but this PR is not merge-ready because the branch is conflicting, loses current proxy/timeout behavior if rebased directly, and has no after-fix real behavior proof.

## What This Changes

The PR replaces the MiniMax VLM raw fetch with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the guarded dispatcher after response handling.

## Best Possible Solution

Rebase and rework the fix to use guarded fetch with MiniMax audit context, timeout forwarding, release cleanup, and trusted env-proxy mode only when the existing proxy gate permits it, plus focused SSRF/proxy/timeout coverage and redacted real proof.

## Reproduction Assessment

Yes, by source inspection: current main builds the MiniMax VLM URL from configurable host inputs and sends `Authorization: Bearer` plus image data through raw `fetch()`. I did not run live MiniMax or private-host calls because this read-only review cannot safely create runtime artifacts or use secrets.

## Solution Assessment

No. SSRF-guarded fetch is the right primitive, but this branch hard-codes strict mode and omits timeout and audit context; the maintainable solution should preserve current proxy and timeout contracts while adding SSRF blocking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve MiniMax VLM env-proxy routing:** `src/agents/minimax-vlm.ts:80`
  - body: Hard-coding `withStrictGuardedFetchMode` ignores the current `HTTP_PROXY`/`HTTPS_PROXY` path. Current main explicitly initializes env-proxy dispatching for this request, and the shared provider helper switches to trusted env-proxy mode only when `shouldUseEnvHttpProxyForUrl(url)` permits it; proxy-only deployments can fail or bypass required egress after a direct rebase.
  - confidence: 0.9
- **[P2] Forward the MiniMax VLM timeout:** `src/agents/minimax-vlm.ts:80-81`
  - body: The guarded-fetch call passes neither `timeoutMs` nor `signal`. Current main computes the caller/default timeout and uses `AbortSignal.timeout(timeoutMs)`, so this patch can remove the MiniMax VLM timeout and leave slow or stuck requests hanging.
  - confidence: 0.86
- **[P3] Set a MiniMax audit context:** `src/agents/minimax-vlm.ts:80`
  - body: The guarded fetch omits `auditContext`, so SSRF-blocked warning logs fall back to the generic `url-fetch` label. Add a MiniMax-specific context so security logs identify this endpoint.
  - confidence: 0.78
