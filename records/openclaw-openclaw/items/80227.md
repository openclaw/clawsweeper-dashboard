## Summary

Keep open. Current main still leaves Discord `/gateway/bot` metadata on the guarded direct fetch while only the gateway WebSocket agent receives the configured proxy, and the linked open fix PR should be reviewed or landed before this issue closes.

## What This Changes

Make Discord gateway metadata REST lookup honor the configured Discord proxy, matching the documented gateway/API proxy behavior.

## Best Possible Solution

Review and land the linked guarded explicit-proxy fix if maintainers accept proxying metadata; otherwise update Discord docs and diagnostics to state that metadata remains direct-only.

## Reproduction Assessment

Yes, source-level: configure `discordConfig.proxy`, create the Discord gateway plugin, and current main still keeps `/api/v10/gateway/bot` on the direct guarded fetch while only the WebSocket agent is proxied. I did not run a live Mainland China/proxy Discord session in this read-only review.

## Solution Assessment

Yes for the code direction: the narrow maintainable fix is to route metadata through `fetchWithSsrFGuard` in guarded explicit-proxy mode while preserving the Discord target allowlist. Merge should wait for review of https://github.com/openclaw/openclaw/pull/80252 and real behavior proof.
