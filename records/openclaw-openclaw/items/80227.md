## Summary

Keep open. Current main still leaves the Discord `/gateway/bot` metadata fetch on the guarded direct fetch while the configured proxy is applied to the WebSocket path, and an open linked PR already targets this issue, so this should not be cleanup-closed before that PR is reviewed or resolved.

## What This Changes

The issue requests that Discord gateway metadata REST lookup use the configured Discord proxy instead of only proxying the gateway WebSocket path.

## Best Possible Solution

Review the linked PR and either land a guarded explicit-proxy metadata fetch or explicitly document and warn if maintainers want gateway metadata to remain direct-only.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket receives the proxy agent. I did not run a live Discord/proxy session in this read-only sweep.

## Solution Assessment

Unclear pending maintainer review of the linked PR. The narrow technical fix is to use the existing guarded explicit-proxy fetch path, not a raw proxy fetch, but the prior maintainer hardening PR means the token-egress policy should be confirmed before closing this issue.
