## Summary

Keep open. Current main still routes Discord gateway metadata through the guarded direct fetch while only the WebSocket agent gets the configured proxy, and the existing test suite explicitly preserves that behavior after a merged security-hardening PR.

## What This Changes

The issue asks for Discord gateway metadata REST lookup to honor the configured Discord proxy consistently with WebSocket and startup REST traffic.

## Best Possible Solution

Route the Discord `/gateway/bot` lookup through `fetchWithSsrFGuard` using a guarded explicit-proxy dispatcher when `channels.discord.proxy` is configured, or document and warn if maintainers intentionally keep metadata direct-only.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create the gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket receives the proxy agent. I did not run a live Mainland China/proxy Discord session in this read-only review.

## Solution Assessment

No, the current direct-only metadata path does not match the documented proxy behavior. The reporter's raw proxy-fetch sketch should be tightened into a guarded explicit-proxy metadata fetch so SSRF protections remain intact.
