## Summary

Keep open. Current main still has the reported source-level proxy mismatch, and the linked open fix PR should be reviewed or landed before this issue is closed.

## What This Changes

The issue asks for Discord gateway metadata REST lookup to honor the configured Discord proxy instead of proxying only the WebSocket connection.

## Best Possible Solution

Review and land the guarded explicit-proxy fix in https://github.com/openclaw/openclaw/pull/80252, or explicitly document a direct-only metadata policy if maintainers choose that security model.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the direct guarded fetch while only the WebSocket gets a proxy agent. I did not run a live Discord/proxy session.

## Solution Assessment

Yes, if maintainers want proxy consistency, the narrow fix is to use `fetchWithSsrFGuard` in guarded explicit-proxy mode with the existing Discord allowlist. A raw proxy fetch would be worse because it could bypass SSRF protections.
