## Summary

Keep open. Current main still leaves Discord gateway metadata on the guarded direct fetch while only the WebSocket agent uses the configured proxy, and the issue is already paired with the open fixing PR https://github.com/openclaw/openclaw/pull/80252, so this should close only after that PR lands or maintainers choose a documented direct-only policy.

## What This Changes

The issue requests routing Discord gateway metadata REST lookup through the configured Discord proxy instead of proxying only the WebSocket path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80252 if its proof, CI, and security review clear; if maintainers want direct-only metadata, update the Discord docs and config help to say so explicitly.

## Reproduction Assessment

Yes, at source level: configure discordConfig.proxy and create the gateway plugin; current main keeps /api/v10/gateway/bot on fetchWithSsrFGuard without proxy policy while only the WebSocket agent is proxied. I did not run a live Discord/proxy session in this read-only review.

## Solution Assessment

No for current main: it contradicts the documented proxy behavior and the current test locks in the bypass. The narrow maintainable fix is guarded explicit-proxy metadata fetch, not a raw proxy fetch that bypasses SSRF protections.
