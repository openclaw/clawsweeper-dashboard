## Summary

Keep open. Current main and the latest shipped tag still route Discord gateway metadata through the guarded direct fetch while only the gateway WebSocket gets the configured proxy, and an open PR already closes this issue for maintainer review.

## What This Changes

Discord proxy configuration should also cover the gateway metadata REST lookup instead of only the gateway WebSocket path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80252, or an equivalent guarded explicit-proxy implementation; if maintainers prefer direct-only metadata, update the Discord docs/UI help and runtime warning accordingly.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create/register the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket receives the proxy agent. I did not run a live Discord/proxy session in this read-only review.

## Solution Assessment

No for current main: the direct-only metadata path conflicts with the documented proxy behavior. The reporter's direction is right, but the maintainable fix should use guarded explicit-proxy mode rather than a raw proxy fetch.
