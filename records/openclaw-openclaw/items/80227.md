## Summary

Keep open. Current main still keeps the Discord gateway metadata lookup on the direct guarded fetch while only the WebSocket agent receives the configured proxy, and the linked open PR https://github.com/openclaw/openclaw/pull/80252 already uses closing syntax for this issue, so the issue should wait for that PR to land or be rejected.

## What This Changes

Make Discord gateway metadata REST lookup honor the configured Discord proxy instead of sending that token-bearing request directly while the WebSocket uses the proxy.

## Best Possible Solution

Review and land the linked guarded explicit-proxy fix if maintainers want proxy consistency; otherwise update docs and runtime messaging to make any direct-only metadata policy explicit.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create the Discord gateway plugin, and current main still leaves `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket agent receives the proxy. I did not run a live Discord/proxy session in this read-only sweep.

## Solution Assessment

Yes in direction, but the raw suggested proxy fetch is not the safest implementation. The maintainable solution is a guarded explicit-proxy metadata fetch that keeps the Discord hostname allowlist and proxy validation, or an explicit documented direct-only policy if maintainers choose that instead.
