## Summary

Keep open. Current main and the latest shipped tag still route Discord `/gateway/bot` metadata through the guarded direct fetch while only the gateway WebSocket receives the configured Discord proxy, and an open closing PR now tracks the narrow fix path.

## What This Changes

The issue reports that Discord gateway metadata REST lookup ignores the configured Discord proxy while the WebSocket path uses it.

## Best Possible Solution

Review and land a narrow Discord plugin fix that routes `/gateway/bot` through the validated loopback proxy using guarded explicit-proxy fetch mode, or explicitly document a direct-only policy if maintainers choose that security tradeoff.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create/register the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the direct guarded fetch while only the WebSocket agent is proxied. I did not run a live Mainland China/proxy Discord session in this read-only review.

## Solution Assessment

Yes, the linked PR’s direction is the narrowest maintainable fix I found because it reuses `fetchWithSsrFGuard` trusted explicit-proxy mode instead of a raw proxy fetch. The issue should stay open until that PR lands or maintainers choose an explicit direct-only metadata policy.
