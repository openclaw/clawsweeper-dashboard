## Summary

Keep open. Current main still has the reported Discord gateway metadata proxy bypass, and an open linked fix PR is the right place to resolve the security-sensitive proxy/SSRF policy before this issue closes.

## What This Changes

The issue reports that Discord gateway metadata REST fetches ignore `channels.discord.proxy` while the gateway WebSocket uses it, causing startup timeouts behind blocked direct Discord access.

## Best Possible Solution

Review and either land the guarded explicit-proxy approach in https://github.com/openclaw/openclaw/pull/80252 or explicitly document/warn that gateway metadata remains direct-only.

## Reproduction Assessment

Yes, at source level. Configure `discordConfig.proxy`, create the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` without an explicit proxy while only the WebSocket gets the proxy agent.

## Solution Assessment

Yes for the linked guarded explicit-proxy direction; no for a raw proxy fetch that would bypass SSRF protections. The narrow maintainable solution is to reuse `fetchWithSsrFGuard` with `trusted_explicit_proxy`, the Discord hostname allowlist, and the existing loopback proxy validation.
