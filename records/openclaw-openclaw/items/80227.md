## Summary

Keep open. Current main and the latest release still route the Discord gateway WebSocket through `channels.discord.proxy` while `/gateway/bot` metadata uses the guarded direct fetch, matching the reporter’s root cause; however, that behavior is now intentional test-covered behavior from a security-adjacent change, so the fix needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks for Discord `/gateway/bot` metadata REST fetches to honor the configured Discord proxy instead of applying the proxy only to the gateway WebSocket.

## Best Possible Solution

Route Discord gateway metadata through a validated, guarded explicit-proxy dispatcher when `channels.discord.proxy` is configured, while keeping the Discord hostname allowlist, capture safety, token handling, and tests/docs aligned.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.7` show the proxy branch only changes the WebSocket agent, while `/gateway/bot` metadata remains on `fetchWithSsrFGuard` without a proxy dispatcher.

## Solution Assessment

No for the suggested unguarded `createProxyFetchImpl` shape. The maintainable fix should use the existing guarded fetch dispatcher policy or equivalent SDK seam so proxy support does not weaken SSRF, capture, or token-routing protections.
