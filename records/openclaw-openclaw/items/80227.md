## Summary

Keep open. Current main still routes Discord gateway metadata through the guarded direct fetch while only the WebSocket agent gets the configured proxy, and the current test suite explicitly locks in that behavior; because a merged maintainer security PR intentionally made that change, this needs Discord/security owner review rather than cleanup closure.

## What This Changes

The issue asks for the Discord account proxy to cover the gateway metadata REST lookup, not only Discord REST client calls and the gateway WebSocket.

## Best Possible Solution

Use the existing guarded explicit-proxy dispatcher path for the Discord `/gateway/bot` lookup while preserving the `discord.com` allowlist and capture behavior; if maintainers want direct-only metadata, update docs and emit a clear warning instead.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, register the gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket gets the proxy agent. I did not run a live proxy/Discord session in this read-only sweep.

## Solution Assessment

No, the current direct-only metadata path is not the best solution for users who configured an account proxy, but the proposed raw proxy fetch should not bypass SSRF protections. The maintainable fix is a guarded explicit-proxy metadata fetch or an explicit documented direct-only policy.
