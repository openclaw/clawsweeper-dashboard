## Summary

Keep open. Current main still leaves the Discord gateway metadata lookup on the guarded direct fetch while only the gateway WebSocket agent receives the configured proxy, and an open linked PR already closes this issue, so the issue should not be cleaned up before that PR is reviewed and merged or rejected.

## What This Changes

The issue asks Discord gateway metadata REST lookups to honor the configured Discord proxy consistently with the WebSocket and regular REST paths.

## Best Possible Solution

Review and land the linked guarded explicit-proxy implementation if it preserves SSRF protections and regression coverage; if maintainers reject proxying this request, update the Discord docs and logs to state the direct-only policy clearly.

## Reproduction Assessment

Yes, at source level. Configure `discordConfig.proxy`, register the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the guarded direct fetch while only the WebSocket agent gets the proxy.

## Solution Assessment

Yes, the best path is to resolve the already-linked PR rather than close this issue now. The proposed direction is narrow if it uses `fetchWithSsrFGuard` in guarded explicit-proxy mode instead of a raw proxy fetch.
