## Summary

Keep open. Current main and the latest release still route Discord `/gateway/bot` metadata through the guarded direct fetch while only the WebSocket agent uses the configured Discord proxy; an open linked fix PR targets this issue, so this issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks Discord gateway metadata REST lookup to honor the configured Discord proxy instead of only proxying the WebSocket path.

## Best Possible Solution

Review the linked guarded explicit-proxy fix or an equivalent patch that preserves the Discord hostname allowlist, SSRF guard, loopback proxy validation, and tests; if maintainers choose direct-only metadata, align docs and warnings with that policy.

## Reproduction Assessment

Yes, at source level. Configure `discordConfig.proxy`, register the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the guarded direct fetch while only the WebSocket uses the proxy agent.

## Solution Assessment

No, the current implementation is not the best end state for the documented proxy behavior. The narrow maintainable fix is a guarded explicit-proxy metadata fetch, not a raw unguarded proxy fetch, with maintainer confirmation of the security policy.
