## Summary

Keep open. Current main still matches the report: Discord gateway metadata uses the guarded direct fetch while only the WebSocket path gets the configured proxy, and the existing test suite explicitly asserts that behavior despite docs/config help promising proxy coverage for startup REST lookups.

## What This Changes

The issue asks for Discord gateway metadata REST lookup to honor the configured Discord proxy instead of bypassing it while the WebSocket uses the proxy.

## Best Possible Solution

Route the Discord `/gateway/bot` metadata lookup through a guarded explicit-proxy dispatcher when `channels.discord.proxy` is configured, or explicitly document and warn that metadata is direct-only if that is the intended security policy.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, register the gateway plugin, and current main keeps `/api/v10/gateway/bot` on `fetchWithSsrFGuard` while only the WebSocket gets the proxy agent. I did not run a live proxy/Discord session in this read-only review.

## Solution Assessment

No for the raw suggested `createProxyFetchImpl` approach; the safer fix is to reuse the guarded explicit-proxy dispatcher path so hostname allowlisting and capture redaction remain intact. If maintainers intend metadata to stay direct-only, docs and runtime diagnostics should say that clearly.
