## Summary

Keep open. Current `origin/main` still leaves the Discord gateway metadata request on the guarded direct fetch while only the gateway WebSocket gets the configured proxy, and an open linked fix PR already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for Discord gateway metadata REST lookup to honor the configured Discord proxy instead of using the proxy only for the gateway WebSocket.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80252 or an equivalent guarded explicit-proxy fix, then close this issue after the fix is merged.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy` and current `origin/main` keeps `/api/v10/gateway/bot` on the direct guarded fetch while only the WebSocket agent is proxied. I did not run a live Discord/proxy session in this read-only review.

## Solution Assessment

Yes, keeping this open is the right triage result because an open PR already tracks the fix with closing syntax. The maintainable code direction is a guarded explicit-proxy metadata fetch, not a raw proxy fetch that bypasses SSRF protections.
