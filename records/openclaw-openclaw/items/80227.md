## Summary

Keep open. Current main still routes Discord gateway metadata through the guarded direct fetch while only the WebSocket agent gets the configured proxy, and an open closing PR now tracks the narrow guarded-proxy fix; because a merged maintainer security PR intentionally set the current direct-only behavior, this needs owner review rather than cleanup closure.

## What This Changes

The issue reports that the Discord plugin applies the configured proxy to WebSocket traffic but not to the gateway metadata REST lookup used during startup.

## Best Possible Solution

Land or revise the linked guarded explicit-proxy fix so `/gateway/bot` uses the configured validated Discord proxy while keeping SSRF protections, or document and warn if maintainers intentionally require metadata to stay direct-only.

## Reproduction Assessment

Yes, at source level: set `discordConfig.proxy`, construct the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the guarded direct fetch while only the WebSocket agent gets the proxy. I did not run a live mainland-China/proxy Discord session in this read-only review.

## Solution Assessment

Yes for the linked direction, if it stays within `fetchWithSsrFGuard` using guarded explicit-proxy mode and the Discord allowlist. A raw proxy fetch would not be the best solution because it would weaken the security boundary on a bot-token-bearing request.
