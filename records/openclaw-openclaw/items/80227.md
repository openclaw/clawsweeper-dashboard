## Summary

Keep open. Current main still routes Discord gateway metadata through the guarded direct fetch while only the WebSocket agent gets the configured proxy, and an open linked PR explicitly closes this issue, so the issue should remain open until that PR lands or maintainers choose a direct-only policy.

## What This Changes

The issue reports that Discord gateway metadata REST fetches bypass the configured Discord proxy while WebSocket traffic uses it.

## Best Possible Solution

Review and land the linked guarded explicit-proxy PR if maintainers accept proxying metadata; otherwise update the Discord docs/config help and warn users that metadata remains direct-only.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, create the Discord gateway plugin, and current main keeps `/api/v10/gateway/bot` on the guarded direct fetch while only the WebSocket agent is proxied. I did not run a live Discord/proxy session in this read-only review.

## Solution Assessment

Yes, the linked PR's guarded explicit-proxy direction is the best known implementation path because it preserves the SSRF guard and Discord allowlist; a raw proxy fetch would be a weaker fix.
