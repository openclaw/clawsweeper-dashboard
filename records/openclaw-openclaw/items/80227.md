## Summary

Keep open. Current main still leaves Discord gateway metadata on the guarded direct fetch while only WebSocket and ordinary Discord REST use the configured proxy, and an open fix PR already references this issue with closing syntax, so the issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

The issue asks for Discord gateway metadata REST lookup to honor the configured Discord proxy consistently with WebSocket and startup REST traffic.

## Best Possible Solution

Review the linked fix PR and either land a guarded explicit-proxy metadata fetch that preserves the Discord allowlist and capture redaction, or intentionally document and warn that gateway metadata remains direct-only.

## Reproduction Assessment

Yes, at source level: configure `discordConfig.proxy`, register the Discord gateway plugin, and current main still keeps `/api/v10/gateway/bot` on the direct SSRF-guarded metadata fetch while only the WebSocket agent receives the proxy. I did not run a live Mainland China/proxy Discord session in this read-only review.

## Solution Assessment

No for the current behavior: direct-only metadata contradicts the documented proxy coverage for Discord gateway/API traffic. The best fix is the linked guarded explicit-proxy approach, pending maintainer/security review because the previous direct-only path was an intentional hardening change.
