## Summary

Keep open. Current main has useful Slack Socket Mode mitigations, but it still constructs Slack's SocketModeReceiver in the gateway process and awaits the same inbound turn dispatch path, so the reported event-loop-starvation/message-loss class is not proved fixed.

## What This Changes

The issue asks OpenClaw to prevent Slack Socket Mode pong timeouts and silent message loss when heavy gateway work starves the Node event loop.

## Best Possible Solution

Design and ship a Slack transport or channel-turn scheduling fix that keeps Socket Mode heartbeat and event ingestion responsive under heavy gateway work, with timeout/native reconnect retained as mitigations.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. The production logs are credible and current source plus Slack SDK source show a same-process heartbeat and awaited dispatch path that can be starved by main-thread work.

## Solution Assessment

Unclear: worker-thread transport isolation may be the right direction, but raising timeouts or relying on reconnects is only a mitigation. Maintainers should choose the final Slack transport/scheduling design with real starvation proof.
