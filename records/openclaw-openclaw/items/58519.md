- `impact:message-loss`: The central failure is Slack messages or replies being silently lost while the gateway appears to have delivered them.

## Summary

Keep open. Current main and v2026.5.18 include useful Slack Socket Mode mitigations, but the transport still runs in the gateway process and awaits the same inbound dispatch path, so the reported starvation-driven pong timeout and message-loss class is not proved fixed.

## What This Changes

The issue asks OpenClaw to prevent Slack Socket Mode pong timeouts and silent message loss when heavy gateway work starves the Node.js event loop.

## Best Possible Solution

Design and ship a Slack-owned transport or scheduling fix that keeps Socket Mode heartbeat and event ingestion responsive under heavy gateway work, with timeout and reconnect tuning retained as mitigations rather than the whole fix.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The posted production logs are credible, and current source plus Slack SDK source show a same-process heartbeat and awaited dispatch path that can be starved by main-thread work.

## Solution Assessment

Unclear. Worker-thread transport isolation may be the right direction, but maintainers should choose between transport isolation, channel-turn scheduling, and heartbeat/liveness instrumentation with real Slack starvation proof.
