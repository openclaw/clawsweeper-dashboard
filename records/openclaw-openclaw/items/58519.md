## Label Justifications

## Summary

Keep open. Current main still constructs Slack Socket Mode inside the gateway process and awaits the same inbound turn path, while the latest issue comment reports the same silent Slack event gap on v2026.5.18; the timeout-config and reconnect improvements are useful mitigations but do not close the requested transport-isolation or scheduling fix.

## What This Changes

The issue asks for Slack Socket Mode heartbeat and message handling to remain responsive during heavy gateway event-loop load, likely through worker-thread transport isolation, faster handoff, or cooperative scheduling fixes.

## Best Possible Solution

Design and ship a Slack-owned transport or scheduling fix that keeps Socket Mode heartbeat, event acknowledgement, and inbound handoff responsive under heavy gateway work, while retaining timeout and reconnect tuning as mitigations.

## Reproduction Assessment

No controlled current-main reproduction was run in this read-only review. The v2026.5.18 live report plus current main source and Slack SDK source give a credible source-reproducible path for same-process heartbeat starvation and silent Slack event gaps.

## Solution Assessment

Unclear. Worker-thread transport isolation may be the right direction, but maintainers should choose between transport isolation, fast acknowledgement and handoff, cooperative scheduling/yielding, and liveness instrumentation with live Slack proof.
