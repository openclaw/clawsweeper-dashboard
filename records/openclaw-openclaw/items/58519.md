## Summary

Keep open. Current main has useful Slack Socket Mode mitigations, including a 15s/default configurable pong timeout and native reconnect, but the WebSocket heartbeat and Slack event handling still run in the gateway process and Slack message turns still await the same inbound dispatch path, so the reported event-loop-starvation class is not proved fixed.

## What This Changes

The issue asks OpenClaw to prevent Slack Socket Mode pong timeouts and silent message loss when heavy gateway work starves the Node.js event loop.

## Best Possible Solution

Design and ship a Slack transport or channel-turn scheduling fix that keeps Socket Mode ack/heartbeat handling responsive under heavy agent work, with starvation/reconnect proof; keep timeout tuning and native reconnect as mitigations.

## Reproduction Assessment

Unclear: two production reports give credible logs and current source confirms same-process timer and awaited dispatch paths, but I did not establish a deterministic current-main Slack load harness that reproduces pong loss.

## Solution Assessment

Unclear: worker-thread transport isolation may be the right direction, but timeout tuning and native reconnect are only mitigations; the final fix should be chosen with maintainer design for Slack SDK ownership and channel-turn scheduling.
