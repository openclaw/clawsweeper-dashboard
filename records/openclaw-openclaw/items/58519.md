## Summary

Keep open. Current main has Slack Socket Mode timeout/pass-through mitigation, but it still runs Bolt SocketModeReceiver and Slack's timer-based heartbeat in the gateway process while Slack message events await the same inbound agent-turn dispatch path, so the event-loop starvation class is not proved fixed.

## What This Changes

Fix Slack Socket Mode message loss when heavy agent work starves ping/pong handling and causes stale WebSocket delivery windows.

## Best Possible Solution

Design and ship a Slack transport or channel-turn scheduling fix that keeps Socket Mode ack/heartbeat handling responsive under heavy agent work, with starvation/reconnect proof; keep timeout tuning as a mitigation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear: two production reports provide credible logs, and source inspection confirms the same-process timer and awaited dispatch path, but this review did not establish a deterministic current-main Slack load harness.

## Solution Assessment

Is this the best way to solve the issue? No for the timeout-only path: public Socket Mode timeouts are a useful mitigation, but the complete fix should isolate or decouple heartbeat/ack handling from long-running agent work and prove it under starvation.
