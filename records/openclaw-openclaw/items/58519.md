## Summary

Keep open. Current main ships the Socket Mode timeout/pass-through mitigation and related Slack media hardening, but it still uses Slack Bolt's SocketModeReceiver inside the gateway process without a worker or transport boundary, so the main-thread starvation class described here is not proved fixed.

## What This Changes

The issue requests isolating Slack Socket Mode keepalive from heavy main-thread agent work so pong deadlines are not missed and replies are not lost during dead socket windows.

## Best Possible Solution

Keep the timeout configuration as a mitigation, but design a Slack/plugin-runtime fix that either isolates Socket Mode transport and ack handling from agent turn work or moves long synchronous turn/context work off the gateway event loop with focused starvation and reconnect validation.

## Reproduction Assessment

Unclear. The report has credible operational reproduction evidence from two production gateways, but this read-only review did not find or run a deterministic local harness that reproduces pong loss against current main.

## Solution Assessment

No. Current main's 15s default and configurable Slack SDK timeouts are useful mitigations for #14248, but they are not the narrowest complete solution for multi-minute event-loop starvation; the durable fix needs transport isolation or dispatch/ack decoupling.
