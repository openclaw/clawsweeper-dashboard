- `impact:message-loss`: The central failure is Slack inbound or outbound messages disappearing while the gateway appears healthy or logs delivery.

## Summary

Keep open. Current main and v2026.5.18 still run Slack Socket Mode in the gateway process through Bolt's SocketModeReceiver, with awaited inbound dispatch on the same runtime path; timeout tuning and reconnect hardening mitigate parts of the problem but do not prove the starvation-driven message-loss class fixed.

## What This Changes

The issue reports Slack Socket Mode pong timeouts and silent message loss when gateway or agent work starves the Node.js event loop.

## Best Possible Solution

Design a Slack-owned transport or scheduling boundary that keeps Socket Mode heartbeat and event ingestion responsive under heavy gateway work, while retaining timeout/reconnect tuning as mitigations rather than treating them as the full fix.

## Reproduction Assessment

No high-confidence local live reproduction was run against current main. The source path is clear and the issue has fresh release-level field evidence, but proving the exact starvation trigger still needs a real Slack workload with event-loop delay and Socket Mode delivery traces.

## Solution Assessment

Unclear. Worker-thread transport isolation is plausible, but the maintainable fix should account for Bolt's full Socket Mode event path, message ack/dispatch ordering, reconnect behavior, and channel-turn scheduling rather than moving only a pong helper.
