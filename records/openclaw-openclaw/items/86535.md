## Summary

Keep open: current main and the latest release still compute active Telegram getUpdates stall elapsed from wall-clock liveness, and the linked closing PR is open but unmerged.

## What This Changes

Telegram polling liveness should reset active getUpdates baselines on the first watchdog tick after a long sleep or event-loop pause instead of forcing a false active-stall restart.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/86541 after redacted live or equivalent Telegram polling proof shows the sleep/resume false restart is gone and genuine active stalls still recover.

## Reproduction Assessment

Yes, at source level. Start a getUpdates call, keep it in flight, advance the liveness wall clock beyond the stall threshold, and current main can report an active stall that the polling-session watchdog turns into a restart.

## Solution Assessment

Yes. The linked PR targets the narrow liveness tracker used by both polling paths and adds focused regression coverage; this issue should remain open until that PR lands or maintainers replace it.
