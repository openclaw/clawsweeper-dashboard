## Summary

Keep open: current main still computes active Telegram getUpdates stall elapsed from wall-clock liveness while a poll is in flight, and the linked closing PR remains open awaiting live Telegram proof and maintainer review.

## What This Changes

Reset Telegram polling liveness baselines after long host sleep or event-loop pauses so a normal in-flight getUpdates call is not reported as an active stall.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/86541 after live Telegram polling proof shows the sleep/resume false restart is gone while real stuck polling still restarts, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a getUpdates call, keep it in flight, advance the liveness wall clock beyond the threshold, and current main reports an active getUpdates stall that the session watchdog turns into a restart.

## Solution Assessment

Yes. The linked PR is the narrowest maintainable fix path because it targets the Telegram liveness tracker directly and adds a sleep-gap regression test; this issue should stay open until that PR lands or is replaced by an equivalent merged fix.
