## Summary

Keep open: current main and v2026.6.9 still use wall-clock active elapsed for in-flight Telegram getUpdates stall detection, and the linked closing PR remains open rather than merged.

## What This Changes

The issue reports that Telegram isolated polling treats a long host sleep or event-loop pause as an active getUpdates stall and restarts polling even when the request duration was normal.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/86541 with a narrow liveness-layer fix that suppresses the first post-gap false restart while preserving genuine active-stall recovery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a getUpdates call, keep it in flight, advance the liveness clock beyond the stall threshold after a long pause, and current main reports an active stall that polling-session turns into a restart.

## Solution Assessment

Yes, the linked PR appears to target the right owner boundary by changing the Telegram liveness tracker and adding focused regression coverage. This issue should stay open until that PR lands or maintainers replace it with an equivalent fix.
