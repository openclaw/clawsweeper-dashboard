## Summary

Keep open: current main and the latest release still compute active Telegram getUpdates stall time from wall-clock liveness, and the linked closing PR remains open as the narrow fix path.

## What This Changes

The issue reports that Telegram polling liveness treats a long sleep/resume or event-loop pause as an active getUpdates stall and forces an unnecessary polling restart.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/86541 after redacted live or equivalent polling proof shows the false post-sleep restart is gone and genuine active stalls still recover.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a getUpdates call, keep it in flight, advance the liveness wall clock beyond the stall threshold, and current main reports an active stall that polling-session turns into a restart.

## Solution Assessment

Yes. The linked PR appears to target the right boundary by changing the Telegram liveness tracker and adding focused regression coverage; this issue should remain open until that PR lands or maintainers replace it.
