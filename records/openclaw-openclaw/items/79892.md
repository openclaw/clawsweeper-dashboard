## Summary

Keep open for maintainer review: the original log-retention patch is plausible, but current main now suppresses the launchd stderr file, so the PR needs a rebase and a decision on whether to keep stdout retention and how to handle legacy stderr logs.

## What This Changes

The PR adds macOS launchd gateway log retention with env-configurable max bytes/archives, hooks it into install/rewrite/restart paths, adds daemon tests, and updates the changelog.

## Best Possible Solution

Rebase the PR on current main, keep only retention that matches actual launchd-owned log files, document any legacy `gateway.err.log` cleanup separately, and leave the CPU-loop investigation in the linked issue cluster.

## Reproduction Assessment

Yes for the remaining stdout retention gap by source inspection: current main still writes launchd stdout to `gateway.log` with no retention hook. No for the original active stderr sink on current main, because macOS launchd stderr is now `/dev/null`, and the CPU loop was not reproduced here.

## Solution Assessment

No as-is against current main: the old-base solution was reasonable when launchd wrote stderr to `gateway.err.log`, but it now needs a rebase and scope adjustment. The narrow maintainable path is to rotate only actual launchd-owned files or explicitly mark legacy stale stderr cleanup as separate behavior.
