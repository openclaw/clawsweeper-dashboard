## Summary

Keep open: this is a useful narrow fix with sufficient macOS terminal proof, but the patch misses the launchd self-restart handoff path, so one restart flow can still bypass retention.

## What This Changes

The PR adds a daemon log-retention helper, wires it into macOS launchd install/rewrite/restart paths, adds unit coverage, and records the fix in the changelog.

## Best Possible Solution

Keep the launchd-owned retention fix, but make restart retention cover both direct kickstart and detached self-handoff restarts before merge.

## Reproduction Assessment

Yes for the log-retention half: current main writes launchd stdout/stderr to fixed files without a cap, and the PR supplies live macOS terminal proof for the proposed rotation behavior. The CPU loop in the linked issue is not reproduced here and remains separate.

## Solution Assessment

No as currently patched: the helper and default cap are a maintainable direction, but the restart hook is placed after the self-handoff early return. Move retention before scheduling the handoff or add equivalent retention to the detached helper path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Run retention before launchd self-handoff:** `src/daemon/launchd.ts:858-861`
  - body: The new retention call runs only after the branch that returns when `restartLaunchAgent` is invoked from inside the managed LaunchAgent. That branch schedules a detached shell helper which only waits and runs `launchctl enable`/`kickstart`/`bootstrap`, so in-process or safe restarts can still restart without capping an oversized `gateway.err.log`. Move retention before this early return or teach the handoff helper to rotate the logs too.
  - confidence: 0.88
