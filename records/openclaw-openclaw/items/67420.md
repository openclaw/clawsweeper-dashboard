## Summary

Keep open: current main still lacks per-agent dreaming control, and this PR is the active implementation path, but the patch still leaves an opt-out fallback bug and lacks required real behavior proof.

## What This Changes

The PR adds `agents.list[].dreaming.enabled` to config docs/types/schema and filters disabled agents from memory dreaming workspace resolution, with a mixed enabled/disabled unit test.

## Best Possible Solution

Land an updated version or successor that adds a documented schema-backed per-agent dreaming config, preserves no-work semantics for explicitly disabled agents, and leaves per-agent scheduling to an explicit maintainer decision.

## Reproduction Assessment

Yes, by source inspection. Current main has only global dreaming config, and the PR's new skip guard can still fall through to default-agent or primary-workspace fallback when every configured agent opts out.

## Solution Assessment

No. The proposed enable/disable surface is useful, but the implementation needs explicit all-disabled semantics, current-main schema alignment, and maintainer scope judgment for per-agent schedules before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve explicit agent opt-outs through fallback:** `src/memory-host-sdk/dreaming.ts:600-602`

When the configured list is all disabled, or when it disables the primary `main` agent, the fallback below still adds the default agent and primary workspace, so the documented opt-out can still run dreaming. Track explicit configured/disabled agents and suppress fallback work for disabled IDs.
