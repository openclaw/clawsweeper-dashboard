## Summary

Keep open. Current main still lacks per-agent dreaming control, and this external PR remains relevant but not merge-ready because it lacks real behavior proof and leaves explicit opt-outs able to fall back into dreaming execution.

## What This Changes

This PR adds an `agents.list[].dreaming.enabled` config field across docs/types/schema and filters disabled agents from `resolveMemoryDreamingWorkspaces()`, with a mixed enabled/disabled unit test.

## Best Possible Solution

Land an updated PR or successor that adds a schema-backed per-agent `enabled` opt-out, suppresses fallback work for explicitly disabled agents, covers mixed/all-disabled/schema cases, and splits schedules if maintainers want that as follow-up scope.

## Reproduction Assessment

Yes, by source inspection: current main lacks `agents.list[].dreaming` in the type/schema, and the PR's skip path can fall through to default or primary workspace fallback when configured agents opt out. No live runtime proof was run during this read-only review.

## Solution Assessment

No. The config surface is plausible, but the current PR needs explicit fallback semantics, current-main schema alignment, all-disabled coverage, and maintainer scope judgment for per-agent schedules before it is the narrow maintainable solution.
