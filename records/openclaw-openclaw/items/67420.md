## Summary

Keep open: current main still lacks per-agent dreaming control, and this PR is relevant, but the latest patch still lets an explicit all-agent opt-out fall back to the default workspace. The PR also needs the required changelog entry and maintainer scope judgment because the linked issue also asks for per-agent schedules.

## What This Changes

This PR adds `agents.list[].dreaming.enabled` across config types/schema/generated schema/docs and filters disabled agents out of memory dreaming workspace resolution with a mixed-agent unit test.

## Best Possible Solution

Keep the PR open and update it so the config contract stays aligned, explicit all-disabled opt-out produces no dreaming work, default fallback remains only for no configured agents, tests cover mixed/all-disabled/schema cases, and the changelog plus linked-issue scope are resolved.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config where the only valid agent, or every valid agent, sets `dreaming.enabled: false`; the PR guard empties `agentIds`, then the existing default-agent and workspace fallbacks can still run dreaming.

## Solution Assessment

No. The feature direction is maintainable, but this implementation is not the best fix until it preserves the difference between no configured agents and all configured agents explicitly opting out; per-agent schedules can be a follow-up only if maintainers narrow the issue linkage.
