## Summary

Keep open. This external feature PR is still relevant because current main has only global dreaming configuration, but the branch has a concrete all-disabled fallback bug and lacks required real behavior proof from a real gateway or dreaming run.

## What This Changes

Adds `agents.list[].dreaming.enabled` to config types, strict schema, generated schema, docs, and memory dreaming workspace resolution, with a mixed enabled/disabled unit test.

## Best Possible Solution

Keep this PR open or land a successor that adds a documented, schema-backed per-agent `enabled` contract, returns no work for explicit all-disabled configs, and splits schedule support into a separate design if needed.

## Reproduction Assessment

Yes. Source inspection of the PR shows that when every valid agent sets `dreaming.enabled: false`, the new guard empties `agentIds` and existing fallback paths can still enqueue dreaming work; I did not run tests because this review is read-only.

## Solution Assessment

No. The enable/disable config surface is useful, but this is not the best implementation until explicit all-disabled configs return no work and the schedule portion of the related request is handled deliberately.
