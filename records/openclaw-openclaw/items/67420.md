## Summary

Keep open. The PR targets a real missing per-agent dreaming control, but it is not merge-ready because the branch is dirty against current main, lacks real behavior proof, and the proposed skip can still reintroduce disabled agents through fallback paths.

## What This Changes

Adds `agents.list[].dreaming.enabled` to agent config types/schema/docs and filters disabled agents out of memory dreaming workspace resolution.

## Best Possible Solution

Refresh the PR on current main, implement a schema-backed opt-out that suppresses default and primary fallback for disabled agents, add help/label metadata and focused tests, and split per-agent schedules if maintainers do not want them in this PR.

## Reproduction Assessment

Yes by source inspection. A config where the only listed agent has `dreaming.enabled: false` would be skipped while collecting ids, then the existing default and primary workspace fallbacks can still schedule dreaming; I did not run a live OOM scenario.

## Solution Assessment

No, not yet. Per-agent opt-out is a plausible narrow mitigation, but the implementation needs current-main schema alignment and must keep explicit opt-outs out of fallback workspaces before it is the best fix.
