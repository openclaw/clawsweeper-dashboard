## Summary

Keep open: current main still lacks per-agent dreaming control, and this external PR remains useful but has a concrete opt-out fallback bug plus missing real-behavior proof before merge.

## What This Changes

The PR adds `agents.list[].dreaming.enabled` to docs/config types/schema/generated schema, skips disabled agents in the memory dreaming workspace resolver, and adds a mixed enabled/disabled unit test.

## Best Possible Solution

Land an updated PR or successor with a schema-backed `agents.list[].dreaming.enabled` contract, explicit no-work semantics for disabled agents, mixed/all-disabled/schema tests, and a split follow-up for per-agent schedules if that scope is not accepted now.

## Reproduction Assessment

Yes, by source inspection. Current main lacks `agents.list[].dreaming`, and the PR's guard can still fall through to default-agent or primary-workspace fallback when configured agents opt out.

## Solution Assessment

No. The enable/disable surface is a reasonable direction, but the current diff needs explicit opt-out fallback handling, current-main schema alignment, and maintainer scope judgment for schedules before it is the narrow maintainable solution.
