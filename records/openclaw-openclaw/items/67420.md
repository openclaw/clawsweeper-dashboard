## Summary

Keep open: the feature gap is real, but the PR still has a functional all-disabled fallback bug, lacks after-fix real behavior proof, and is paired with an open broader schedule-scope issue.

## What This Changes

This PR adds an `agents.list[].dreaming.enabled` config surface in types/schema/docs/generated schema and filters disabled agents out of memory dreaming workspace resolution with a mixed-agent test.

## Best Possible Solution

Rebase the PR onto current main, add the `agents.list[].dreaming.enabled` contract across the current type/schema/help/docs surfaces, preserve default fallback only when no agents are configured, return no work for explicit all-disabled configs, and split per-agent schedules if maintainers want a separate cron design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config where the only valid agent, or every valid agent, sets `dreaming.enabled: false`; the PR guard empties `agentIds`, then the existing default-agent and fallback workspace paths can still run dreaming.

## Solution Assessment

No. The enable/disable direction is a useful narrow feature, but this diff is not the best implementation until it preserves the semantic difference between no configured agents and agents explicitly opting out.
