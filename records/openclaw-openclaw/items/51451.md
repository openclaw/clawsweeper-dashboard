## Summary

Keep open. Current main still lacks the requested HTTP workspace header, related multi-tenancy and binding-workspace threads do not supersede this narrower Gateway request, and the remaining decision is security-sensitive product/API design.

## What This Changes

Requests an `x-openclaw-workspace` header on OpenAI-compatible chat completions to override the agent workspace for one request.

## Best Possible Solution

Design a disabled-by-default trusted Gateway workspace-routing contract with canonical path/root validation, spoofing guidance, docs, and tests, or explicitly reject the HTTP header as outside the supported trust model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main does not read `x-openclaw-workspace` and only carries workspace overrides for internal spawned-run metadata.

## Solution Assessment

Unclear. The header shape is plausible for trusted-router deployments, but the minimal sketch is not the best complete solution until opt-in config, trusted-boundary rules, canonical path validation, tenant roots, and documentation are decided.
