## Summary

Keep open. Current main still does not implement `x-openclaw-workspace` or an equivalent per-request workspace override for the OpenAI-compatible chat completions endpoint; the only current workspace override path is explicitly internal spawned-session inheritance, and the public comments reinforce that this needs a deliberate multi-tenant HTTP trust-boundary design rather than a small unguarded header pass-through.

## What This Changes

Request to add a `x-openclaw-workspace` header on `/v1/chat/completions` so each HTTP request can temporarily override the agent workspace for multi-tenant deployments.

## Best Possible Solution

Design a core Gateway workspace-override contract with explicit enablement, trusted-ingress requirements, canonical path and tenant-root validation, regression tests, and public OpenAI-compatible API documentation while preserving the internal spawned-run inheritance boundary.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-main verification path: source and docs searches find no header support, and the chat completions handler builds ingress command input without any workspace field.

## Solution Assessment

Unclear. The proposed header shape is plausible for gateway-managed multi-tenancy, but the minimal sketch is not the best complete fix until maintainers choose the opt-in, trusted-header, canonical path, and tenant-root policy.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose workspace selection through HTTP ingress.

Concerns:

- **[medium] Define the trusted workspace-header boundary:**
  - body: A raw workspace path header could let an HTTP caller select another tenant's workspace or write outside an intended tenant root unless the feature is explicitly enabled, accepted only from a trusted ingress, canonicalized, and constrained by tested filesystem-root policy.
