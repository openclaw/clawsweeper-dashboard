## Summary

Keep open. Current main still lacks the requested HTTP workspace override, and the comments reinforce both the multi-tenant use case and the spoofing/security boundary that needs maintainer design before implementation.

## What This Changes

The issue requests an opt-in per-request workspace override on `/v1/chat/completions` via the `x-openclaw-workspace` HTTP header.

## Best Possible Solution

Design a disabled-by-default Gateway contract for trusted per-request workspace routing with tenant-root validation, canonical path checks, docs, tests, and preservation of the existing internal spawned-run inheritance behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source verification is high confidence: current main never reads `x-openclaw-workspace`, and workspace overrides only flow from internal spawned-session metadata.

## Solution Assessment

Unclear. The proposed header is plausible for a trusted router deployment, but the minimal sketch is not the best complete solution until opt-in, spoofing, canonical path, and tenant-root policy are decided.
