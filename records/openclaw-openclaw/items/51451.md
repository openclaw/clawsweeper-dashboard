## Summary

Keep open. Current main still lacks `x-openclaw-workspace` or an equivalent per-request workspace override on the OpenAI-compatible chat completions path; the only workspace override surface in main is internal spawned-run metadata. The KinthAI and KeepMyClaw discussion supports a real multi-tenant use case, but the trusted-ingress and filesystem-boundary design is still unresolved.

## What This Changes

Requests an HTTP header on `/v1/chat/completions` that overrides the selected agent workspace for a single request to support tenant-isolated contexts.

## Best Possible Solution

Keep this open for a deliberate core Gateway design: add an opt-in workspace override contract for OpenAI-compatible HTTP, validate trusted ingress and tenant-root constraints before dispatch, keep spawned-run inheritance internal unless intentionally generalized, and document the deployment model on the public API page.

## Reproduction Assessment

Not applicable as a bug reproduction, but there is a high-confidence current-main verification path: exact source/docs search finds no header, and the `/v1/chat/completions` handler builds an ingress command without `workspaceDir`. A mocked endpoint test could reproduce the missing behavior by sending the header and observing that the agent command receives no workspace override.

## Solution Assessment

Unclear. The requested header is a plausible API shape for gateway-managed multi-tenancy, but the implementation sketch is not sufficient by itself because the trust boundary, opt-in config, path validation, tenant-root allowlisting, and session/workspace consistency need maintainer design first.
