## Summary

Keep open. Current main still lacks `x-openclaw-workspace` or an equivalent per-request workspace override on the OpenAI-compatible chat completions path; the only workspace override path I found is the internal spawned-run inheritance flow, and the requested HTTP ingress behavior needs a deliberate security design.

## What This Changes

Add a per-request workspace override for `/v1/chat/completions` using an `x-openclaw-workspace` HTTP header so multi-tenant gateways can route each request to an isolated workspace.

## Best Possible Solution

Keep this open for a core Gateway design that adds an explicit opt-in workspace override contract, validates trusted ingress and tenant-root constraints before dispatch, preserves internal spawned-run semantics, and documents the deployment model on the public API page.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-main verification path: exact source/docs search finds no header, and the chat completions handler builds ingress command input without any workspace field.

## Solution Assessment

Unclear. The proposed header is a reasonable API shape for gateway-managed multi-tenancy, but the issue's minimal implementation sketch is not enough because the trust boundary, opt-in config, canonical path validation, and tenant-root policy need maintainer design first.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose workspace selection through HTTP ingress.
