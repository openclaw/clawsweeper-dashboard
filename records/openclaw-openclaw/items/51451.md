## Summary

Keep open: current main and the latest release still do not implement the requested `x-openclaw-workspace` header, and the open related threads do not supersede this narrower Gateway request. ## What This Changes

Request support for a per-request `x-openclaw-workspace` header on `POST /v1/chat/completions` so a trusted router can select tenant-specific workspace state.

## Best Possible Solution

Decide whether HTTP workspace routing fits the Gateway trust model; if accepted, add a disabled-by-default trusted-proxy contract with canonical tenant roots, spoofing guidance, docs, and denial tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection show current main and `v2026.5.22` do not accept `x-openclaw-workspace`, while only internal spawned runs carry workspace-specific overrides.

## Solution Assessment

Unclear. The header shape is plausible for trusted-router deployments, but the minimal sketch is not the best complete solution until opt-in config, trusted-boundary rules, canonical path validation, tenant roots, and documentation are decided.
