## Summary

Keep open: current main does not implement `x-openclaw-workspace`, and the request is a new security-sensitive gateway capability rather than a broken existing contract. Existing code and docs route OpenAI-compatible requests by configured agent/session, while workspace override is intentionally internal-only for spawned runs.

## What This Changes

The issue requests a per-request `x-openclaw-workspace` HTTP header for `/v1/chat/completions` to override the selected agent workspace for multi-tenant deployments.

## Best Possible Solution

If accepted, add a maintainer-approved trusted-ingress workspace routing contract for OpenAI-compatible HTTP, likely config-gated and constrained by allowlist or tenant mapping, with docs and regression coverage; otherwise document configured multi-agent routing as the supported isolation model.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection shows the current defined behavior is configured agent/session routing rather than a failing workspace-header contract.

## Solution Assessment

Unclear. The raw header shape is not yet proven as the best solution because current HTTP auth is operator-scoped and workspace overrides are internal-only; maintainers should decide the trusted-proxy/config/allowlist contract first.
