## Summary

Keep open. Current main still lacks the requested HTTP workspace override, and adding it would be a security-sensitive Gateway feature that needs maintainer design rather than conservative cleanup.

## What This Changes

The issue requests a per-request workspace override on `/v1/chat/completions` via an `x-openclaw-workspace` HTTP header for multi-tenant deployments.

## Best Possible Solution

Design a first-class, disabled-by-default Gateway contract for trusted per-request workspace routing with canonical path/root validation, deployment docs, tests, and preservation of the existing internal spawned-run inheritance semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Current-main source verification is high confidence: the OpenAI-compatible HTTP handler never reads `x-openclaw-workspace`, and dispatch only forwards workspace overrides from internal spawned-session metadata.

## Solution Assessment

Unclear. The proposed header shape is plausible for a trusted router deployment, but the minimal sketch is not the best complete solution until the opt-in, spoofing, canonical path, tenant-root, and documentation contracts are decided.
