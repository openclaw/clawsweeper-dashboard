## Summary

Keep open. Current main still lacks the requested HTTP workspace header, the related multi-tenancy and binding-workspace threads do not supersede this narrower request, and the remaining work is a security-sensitive Gateway/product design decision.

## What This Changes

Request support for an `x-openclaw-workspace` header on `/v1/chat/completions` that overrides the selected agent workspace for one request.

## Best Possible Solution

Design a disabled-by-default Gateway contract for trusted per-request workspace routing, with canonical path/root validation, trusted-proxy guidance, docs, tests, and preservation of existing internal spawned-run inheritance semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main does not read `x-openclaw-workspace` and only forwards workspace overrides from internal spawned-session metadata.

## Solution Assessment

Unclear. The header shape is plausible for trusted-router deployments, but the minimal sketch is not a complete solution until opt-in config, spoofing boundaries, canonical path validation, tenant roots, and documentation are decided.
