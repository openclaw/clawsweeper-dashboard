## What This Changes

Request support for a per-request `x-openclaw-workspace` header on `/v1/chat/completions` to override the selected agent workspace for that run.

## Best Possible Solution

Decide whether HTTP workspace routing fits the Gateway trust model; if it does, design a disabled-by-default trusted-proxy contract with canonical tenant roots, spoofing guidance, docs, and focused denial tests.

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection show current main and `v2026.5.18` do not accept `x-openclaw-workspace`, while only internal spawned runs carry workspace overrides.

## Solution Assessment

Unclear. The header shape is plausible for trusted-router deployments, but the minimal sketch is not the complete solution until opt-in config, trusted-boundary rules, canonical path validation, tenant roots, and documentation are decided.
