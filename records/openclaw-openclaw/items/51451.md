## Summary

Keep open. Current main still lacks the requested HTTP workspace override, and the remaining work is a security-sensitive Gateway/product design decision rather than conservative cleanup; the broader multi-tenancy thread at https://github.com/openclaw/openclaw/issues/60127 is related but does not supersede this narrower header request.

## What This Changes

The issue requests a per-request Gateway workspace override using an `x-openclaw-workspace` HTTP header for `/v1/chat/completions`.

## Best Possible Solution

Design a first-class, disabled-by-default Gateway contract for trusted per-request workspace routing with canonical path/root validation, deployment docs, tests, and preservation of existing internal spawned-run inheritance semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Current-main source verification is high confidence: the OpenAI-compatible HTTP handler never reads `x-openclaw-workspace`, and dispatch only forwards workspace overrides from internal spawned-session metadata.

## Solution Assessment

Unclear. The header shape is plausible for trusted-router deployments, but the minimal sketch is not the best complete solution until opt-in, spoofing, canonical path, tenant-root, and documentation contracts are decided.
