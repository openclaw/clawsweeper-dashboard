## Summary

Keep open. The report is recent against v2026.5.2 and current main still has a source-level path where Telegram sends visible non-final/tool/block output after a preview exists, then finalizes the answer by editing that older preview unless it has been visible for about one minute.

## What This Changes

Telegram direct replies should preserve chronological bubble order when assistant output spans tool/progress/intermediate messages and a final answer.

## Best Possible Solution

Add a focused Telegram regression for a preview created before visible intermediate output, then make final answer delivery appear after that output while cleaning up the stale preview safely.

## Reproduction Assessment

Yes at source level: current main can edit a short-lived Telegram preview into the final answer after visible non-final output has already been delivered. I did not establish a live Telegram reproduction against current main.

## Solution Assessment

No patch is under review. The narrowest maintainable fix appears to be in Telegram lane delivery/preview finalization, not the shared serialized dispatcher.

## Repair Work Prompt

Fix issue 76529. In Telegram preview streaming, a short-lived answer preview can be created before visible intermediate/tool/block output, then the final answer is applied by editing that older preview, making the final appear above later bubbles in Telegram. Add a focused regression in the Telegram dispatch/lane-delivery tests that models preview text, a visible non-final/tool/block delivery, then final text. Adjust Telegram lane delivery so the final answer appears after visible intermediate output, likely by sending a fresh final and cleaning up/redacting/deleting the stale preview when editing would preserve the wrong visual order. Keep the fix scoped to Telegram preview finalization; do not change shared reply-dispatcher serialization, block-streaming defaults, media/error fallback semantics, or unrelated channel behavior. Add a user-facing CHANGELOG.md fix entry.
