## Summary

Keep open. Current main still has the Telegram preview-streaming path where a short-lived preview can be finalized by editing an earlier bubble after visible non-final/tool/block output has already been sent, matching the reported ordering failure.

## What This Changes

Telegram direct replies should preserve chronological bubble order when assistant output spans streamed preview text, intermediate/tool messages, and a final answer.

## Best Possible Solution

Add a focused Telegram regression for visible intermediate output after an existing preview, then make the final answer appear after that output while cleaning up or replacing the stale preview safely.

## Reproduction Assessment

Yes at source level: current main can send visible non-final output after a short-lived Telegram preview, then finalize the answer by editing that older preview. I did not establish a live Telegram reproduction against current main.

## Solution Assessment

No patch is under review. The narrowest maintainable fix belongs in Telegram preview lane delivery/finalization, not the shared serialized dispatcher.

## Repair Work Prompt

Fix issue 76529. In Telegram preview streaming, a short-lived answer preview can be created before visible intermediate/tool/block output, then the final answer is applied by editing that older preview, making the final appear above later bubbles in Telegram. Add a focused regression in Telegram dispatch/lane-delivery tests that models preview text, a visible non-final/tool/block delivery, then final text. Adjust Telegram lane delivery so the final answer appears after visible intermediate output, likely by sending a fresh final and cleaning up, redacting, or deleting the stale preview when editing would preserve the wrong visual order. Keep the fix scoped to Telegram preview finalization; do not change shared reply-dispatcher serialization, block-streaming defaults, media/error fallback semantics, or unrelated channel behavior. Add a single-line user-facing CHANGELOG.md fix entry.
