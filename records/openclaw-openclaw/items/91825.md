## Summary

Keep open: current main has several adjacent protections for `message_tool_only`, `NO_REPLY`, final-answer selection, and source-reply mirroring, but I did not find a merged fix or current-main proof that suppresses Telegram topic pre-tool/intermediate assistant block replies for this exact multi-tool shape. ## What This Changes

The issue reports that intermediate assistant text between multiple tool calls is delivered as separate Telegram forum-topic messages even when the intended reply is sent through the `message` tool and final text is suppressed with `NO_REPLY`.

## Best Possible Solution

Add a core source-reply regression that simulates a Telegram room-event `message_tool_only` multi-tool turn and suppresses assistant text that appears before or between tool calls unless it is the explicit source reply or a valid final automatic reply.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. The source path is traceable: Telegram room events use `message_tool_only`, while the block-reply delivery handler can still send streamed block replies before terminal payload suppression.

## Solution Assessment

Yes, the best fix appears to be in the core embedded-runner/source-reply delivery boundary rather than a Telegram-only text filter. The repair should prove that pre-tool/intermediate assistant text is not delivered for `message_tool_only` turns while preserving explicit `message(action="send")` source replies.
