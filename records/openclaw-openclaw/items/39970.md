## What This Changes

The PR changes former core iMessage outbound delivery and sender tests to route `audioAsVoice` media through `sendPayload`, suppress audio placeholders, and restore configured text chunking.

## Best Possible Solution

Port the behavior into `extensions/imessage` using the current plugin outbound/message APIs and SDK reply-payload helper semantics, carrying `audioAsVoice` through while preserving configured chunking.

## Reproduction Assessment

Yes, source-level: current main can carry `audioAsVoice` in shared delivery, but the live iMessage adapter has no `sendPayload`, the runtime drops the flag, and the sender still resolves empty audio media to a generic placeholder. I did not run live macOS Messages delivery.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is a current-main port under `extensions/imessage`, not changing removed core paths or adding a custom path that bypasses existing payload helper semantics.
