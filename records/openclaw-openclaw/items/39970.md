## Summary

Keep open: the bug is source-reproducible, but this PR is not mergeable as submitted because current main moved live iMessage delivery into `extensions/imessage`, while the branch edits removed core paths and lacks after-fix real behavior proof.

## What This Changes

The PR modifies the old core iMessage outbound adapter and sender plus shared outbound delivery tests to route `audioAsVoice` media through `sendPayload`, suppress audio placeholders, and restore chunking.

## Best Possible Solution

Port the behavior into `extensions/imessage` with an iMessage-owned payload or send path that carries `audioAsVoice`, avoids generic audio placeholder text for voice-intent audio, sends captions separately when needed, and preserves configured text chunking.

## Reproduction Assessment

Yes, source-level. Current shared delivery can carry `audioAsVoice`, but the live iMessage adapter has no `sendPayload` and its runtime/sender do not accept an `audioAsVoice` option, while docs still define `channels.imessage.textChunkLimit`.

## Solution Assessment

No, not as submitted. The intended fix is narrow, but it needs to be ported to the live `extensions/imessage` plugin path and should reuse chunk-aware payload helpers or equivalent behavior.
