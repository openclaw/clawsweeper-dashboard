## Summary

Keep open: the reported iMessage bug is still valid, but this PR needs changes before merge because it targets removed core iMessage files while current main owns live iMessage delivery under `extensions/imessage`. It also needs real behavior proof beyond test-only verification.

## What This Changes

The PR rewrites the former core iMessage outbound adapter and sender to route `audioAsVoice` media through payload handling, suppress audio placeholders, restore chunking, and add delivery/sender regression tests.

## Best Possible Solution

Port the behavior into `extensions/imessage` with an iMessage-owned payload or send path that carries `audioAsVoice`, suppresses voice-audio placeholder text, sends captions separately when needed, and keeps configured text chunking.

## Reproduction Assessment

Yes, source-level. `sendMessage` and outbound service paths can create `audioAsVoice` payloads, but the live iMessage adapter has no `sendPayload` handler and its runtime does not pass `audioAsVoice` to `sendMessageIMessage`.

## Solution Assessment

No, not as submitted. The intended fix is narrow, but the branch edits stale core paths and its custom payload text branch bypasses chunking; the maintainable fix belongs in the current `extensions/imessage` implementation while preserving the SDK helper semantics.
