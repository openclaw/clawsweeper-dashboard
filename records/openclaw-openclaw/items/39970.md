## Summary

This PR is still aimed at a real iMessage outbound gap, but it is not merge-ready against current main. Main has moved the live iMessage outbound surface into the bundled plugin under `extensions/imessage`, while this branch changes older core iMessage files that no longer exist; the current plugin still drops `audioAsVoice` before the low-level sender.

## What This Changes

The PR branch changes the former core iMessage outbound adapter, low-level sender, and delivery tests to route `audioAsVoice` media through payload semantics, suppress generic audio placeholders, restore text chunking, and add regression coverage.

## Best Possible Solution

Port the useful behavior into the current `extensions/imessage` plugin: add an iMessage-owned payload path or equivalent runtime wiring that carries `audioAsVoice`, sends the intended voice audio without visible placeholder text, sends visible captions separately when needed, and keeps configured text chunking for text-only and channelData-backed payloads.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an iMessage payload with `mediaUrl` and `audioAsVoice: true`: current shared delivery can carry the flag, but the live iMessage plugin exposes no `sendPayload` handler and its runtime sender has no `audioAsVoice` parameter, so the flag is dropped before `sendMessageIMessage`.

## Solution Assessment

No, not as currently submitted. The intended fix direction is useful, but this branch edits removed old core paths and should be replaced or rebased onto the current extension-owned iMessage implementation, with the existing chunking review gap fixed at the same time.
