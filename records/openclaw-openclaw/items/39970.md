## Summary

Keep this PR open: the reported behavior still appears valid on current main, but the branch is stale against the extension-owned iMessage implementation and should be ported or replaced before merge.

## What This Changes

The PR branch changes the former core iMessage outbound adapter, sender, and delivery tests to route media-bearing `audioAsVoice` payloads through payload semantics, suppress generic audio placeholders, restore text chunking, and add regression coverage.

## Best Possible Solution

Port the behavior into `extensions/imessage` with a plugin-owned payload path or equivalent runtime wiring that preserves `audioAsVoice`, suppresses generic audio placeholder text for voice-intent audio, sends visible captions separately when needed, and keeps configured text chunking.

## Reproduction Assessment

Yes, as a high-confidence source-level path: send an iMessage payload with `mediaUrl` and `audioAsVoice: true`; shared delivery carries the flag, but the live iMessage plugin has no `sendPayload` handler and its runtime sender cannot pass `audioAsVoice` to `sendMessageIMessage`.

## Solution Assessment

No, not as currently submitted. The intended behavior is the right narrow fix, but the branch edits removed core paths and should be replaced or rebased into the current extension-owned iMessage implementation while preserving chunking.
