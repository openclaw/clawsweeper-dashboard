## Summary

Keep open. The PR addresses a real iMessage outbound bug, and the discussion already captured both the original regression and the stale-path problem, but current main now owns iMessage under `extensions/imessage`, so this branch needs a replacement/rebase before it can fix the live code path.

## What This Changes

The PR edits the former core iMessage outbound adapter, low-level sender, and delivery tests to route `audioAsVoice` media through payload semantics, suppress audio placeholders, restore text chunking, and add regression coverage.

## Best Possible Solution

Port the behavior into `extensions/imessage` with an iMessage-owned payload path or equivalent runtime wiring that preserves `audioAsVoice`, suppresses generic audio placeholder text, sends visible captions separately when needed, and keeps configured text chunking.

## Reproduction Assessment

Yes, source-level. `sendMessage` can create an `audioAsVoice` media payload, delivery only gets payload semantics when a handler has `sendPayload`, and the live iMessage plugin has no such handler and does not pass `audioAsVoice` into `sendMessageIMessage`.

## Solution Assessment

No, not as currently submitted. The intended fix is narrow and maintainable, but the branch edits removed core paths and should be ported to the extension-owned iMessage implementation while preserving chunking for payload text paths.
