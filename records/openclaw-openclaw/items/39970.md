## Summary

Keep this PR open: it targets a real iMessage outbound bug, but the branch is stale against current main because it changes former core iMessage files while live iMessage delivery now lives under `extensions/imessage`, and the external real-behavior proof gate is still unmet.

## What This Changes

The branch modifies former core iMessage outbound and sender code plus delivery tests to route `audioAsVoice` media through payload semantics, suppress audio placeholders, and restore configured text chunking.

## Best Possible Solution

Port the behavior into `extensions/imessage` using the current message/outbound APIs, carrying `audioAsVoice` through to `sendMessageIMessage` while keeping configured chunking and no-op payload handling intact.

## Reproduction Assessment

Yes, source-level: current main can create and carry `audioAsVoice` payloads in shared outbound delivery, but the live iMessage adapter/runtime/sender path drops that flag and still resolves empty audio media to the generic placeholder. I did not run live macOS Messages delivery.

## Solution Assessment

No, not as submitted. The fix needs to be ported to the current `extensions/imessage` implementation, and the branch's custom payload path should not bypass the configured chunking path for text-only payloads.
