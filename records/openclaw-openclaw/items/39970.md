## Summary

Keep open: the bug appears real, but this branch is stale and incorrect against current main because iMessage now lives under `extensions/imessage`, the live adapter still lacks an iMessage-owned payload path for `audioAsVoice`, and the external PR has no after-fix real behavior proof.

## What This Changes

The branch edits the former core iMessage outbound adapter/sender and shared delivery tests to route `audioAsVoice` media through `sendPayload` while restoring text chunking.

## Best Possible Solution

Port the behavior into `extensions/imessage` using the current plugin outbound/message APIs, carry `audioAsVoice` through the live runtime to the sender, suppress voice-audio placeholder text, send captions separately when needed, and preserve configured chunking.

## Reproduction Assessment

Yes, source-level. `audioAsVoice` can reach shared delivery, but current iMessage has no live `sendPayload` handler and `sendIMessageOutbound` does not pass voice intent into `sendMessageIMessage`; I did not run live Messages delivery.

## Solution Assessment

No, not as submitted. The branch edits stale core files and duplicates payload sending in a way that can bypass chunking; the maintainable fix is a narrow port to the live iMessage plugin while reusing the SDK payload helper semantics.
