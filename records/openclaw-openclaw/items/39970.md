## Summary

Keep open: the reported iMessage behavior still appears valid on current main, but the submitted branch is not mergeable as-is because it edits removed core iMessage paths and lacks required after-fix real behavior proof.

## What This Changes

The PR changes former core iMessage outbound sender/delivery code and tests to route `audioAsVoice` media through `sendPayload`, suppress audio placeholders, and restore text chunking.

## Best Possible Solution

Port the behavior into `extensions/imessage` using the current plugin outbound/message APIs and SDK payload helper semantics, carrying `audioAsVoice` through to the sender while preserving configured chunking.

## Reproduction Assessment

Yes, source-level: current delivery can carry `audioAsVoice`, but live iMessage has no `sendPayload` adapter, drops the voice intent in `sendIMessageOutbound`, and still resolves empty audio media to a generic placeholder. I did not run live macOS Messages delivery.

## Solution Assessment

No, not as submitted. The branch targets stale core paths and adds a custom payload sender that can bypass chunking; the maintainable fix is a narrow port to `extensions/imessage` using the current SDK helper behavior.
