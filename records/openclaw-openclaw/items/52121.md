## Summary

Keep open. Current main does not implement these startup conflict diagnostics, but this PR is not landable as-is because the detector can throw during gateway startup, evaluates raw/default config before runtime and per-agent resolution, and includes unsafe exec hardening advice.

## What This Changes

The PR adds a config conflict detector with tests and calls it during gateway startup to log exec, sandbox, elevated, and no-auth exposure diagnostics.

## Best Possible Solution

Implement diagnostics against effective runtime and per-agent config, keep the warning pass non-fatal, and align every message with the existing security audit and exec docs before merge.

## Reproduction Assessment

Yes. Static review of the PR diff against current main gives high-confidence reproduction paths: the proposed startup hook runs before runtime resolution, the detector calls a throwing sandbox resolver, and the checked settings are not the effective per-agent settings used at runtime.

## Solution Assessment

No. The feature direction is useful, but this implementation should be reworked to use effective runtime/per-agent state, never block startup from a warning path, and reuse docs-aligned security-audit semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep conflict checks from throwing at startup:** `src/config/detect-conflicts.ts:24`
  - body: detectConfigConflicts runs during gateway startup but calls resolveSandboxConfigForAgent, which normalizes SSH sandbox settings and throws for a relative sandbox.ssh.workspaceRoot. A warning-only diagnostic should not turn existing configs into a hard gateway boot failure.
  - confidence: 0.91
- **[P2] Check gateway exposure after runtime bind resolution:** `src/gateway/server.impl.ts:502`
  - body: logConfigConflicts(cfgAtStart) runs before resolveGatewayRuntimeConfig applies CLI/runtime bind, host, auth, and tailscale overrides. That can emit a critical unauthenticated exposure warning from stale file config even when the effective runtime bind/auth state is safe.
  - confidence: 0.88
- **[P2] Resolve conflicts against per-agent overrides:** `src/config/detect-conflicts.ts:24-31`
  - body: The detector reads only the default sandbox mode and global tools settings, but runtime exec/elevated/sandbox behavior resolves agents.list[] overrides per agent. Defaults can look safe while a specific agent has the conflicting state this feature is meant to surface.
  - confidence: 0.87
- **[P2] Remove the exec.security=full hardening advice:** `src/config/detect-conflicts.ts:79`
  - body: The info message recommends exec.security=full for maximum restriction, but current docs and audit semantics treat full as the broadest host exec mode. Operators following this guidance could weaken their exec posture.
  - confidence: 0.93
