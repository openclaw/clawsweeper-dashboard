## Summary

Keep open: current main does not implement this startup conflict detector, and the PR is useful but still has blocking correctness/security issues around effective config resolution and exec-policy guidance. Earlier Greptile feedback on elevated defaults and tailnet exposure was addressed in the branch, but the remaining issues affect common startup and security-warning paths.

## What This Changes

The PR adds a `src/config/detect-conflicts` startup diagnostic with tests and calls it from gateway startup to log sandbox, exec, elevated, and unauthenticated-bind conflicts.

## Best Possible Solution

Rework the diagnostics to use effective runtime gateway state and per-agent exec/sandbox/elevated resolution, keep collection non-fatal, and align all exec-policy wording with existing audit/docs semantics.

## Reproduction Assessment

Yes. Static source review of the PR head against current main gives high-confidence paths: the hook runs before runtime resolution, calls a resolver that can throw, checks raw defaults instead of per-agent state, and emits incorrect exec hardening guidance.

## Solution Assessment

No. Startup warnings are a reasonable direction, but this implementation is not the best merge shape until it uses effective runtime/per-agent state and docs-aligned security semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep conflict checks from throwing at startup:** `src/config/detect-conflicts.ts:24`
  - body: `detectConfigConflicts` runs during gateway startup but calls `resolveSandboxConfigForAgent(config)` directly. That resolver normalizes SSH sandbox settings and throws for a relative `sandbox.ssh.workspaceRoot`, so a warning-only diagnostic can turn existing configs into a hard gateway boot failure.
  - confidence: 0.91
- **[P2] Check exposure after runtime bind resolution:** `src/gateway/server.impl.ts:502`
  - body: `logConfigConflicts(cfgAtStart)` runs before `resolveGatewayRuntimeConfig` applies CLI/runtime bind, host, auth, and tailscale overrides. This can emit a critical unauthenticated-exposure warning from stale file config even when the effective runtime bind/auth state is safe.
  - confidence: 0.88
- **[P2] Resolve conflicts with agent overrides:** `src/config/detect-conflicts.ts:24-31`
  - body: The detector reads only the default sandbox mode and global `tools.*` settings, but runtime exec, elevated, and sandbox behavior resolves `agents.list[]` overrides per agent. Defaults can look safe while a specific agent has the conflicting state this feature is meant to surface.
  - confidence: 0.87
- **[P2] Remove the exec.security=full hardening advice:** `src/config/detect-conflicts.ts:79`
  - body: The info message recommends `exec.security=full` for maximum restriction, but current docs and audit code treat `full` as broad exec trust and prefer allowlist with prompts. Operators following this guidance could weaken their exec posture.
  - confidence: 0.93
