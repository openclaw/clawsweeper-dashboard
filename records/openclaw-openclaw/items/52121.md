## Summary

Keep open: current main does not implement this startup conflict detector, and the PR still has blocking correctness and security-guidance problems around effective runtime state, per-agent overrides, and exec policy semantics.

## What This Changes

The PR adds a startup config conflict detector with tests and calls it during gateway startup to log exec, sandbox, elevated, and unauthenticated-exposure diagnostics.

## Best Possible Solution

Rework the diagnostics to use effective runtime and per-agent config resolution, keep the pass non-fatal, and align all remediation text with the existing security audit and exec-policy docs.

## Reproduction Assessment

Yes. Static review against current main gives high-confidence source reproduction paths for the PR defects: the hook runs before runtime resolution, calls a resolver that can throw, and checks raw defaults instead of effective per-agent state.

## Solution Assessment

No. The requested startup warning feature is reasonable, but this implementation is not the best solution until it uses effective state and docs-aligned security semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make startup conflict detection non-fatal:** `src/config/detect-conflicts.ts:24`
  - body: `detectConfigConflicts` runs during gateway startup and calls `resolveSandboxConfigForAgent(config)`, which normalizes SSH sandbox settings and throws for a relative `sandbox.ssh.workspaceRoot`. A warning-only diagnostic can therefore turn existing configs into a hard boot failure even when the SSH backend is not in use.
  - confidence: 0.91
- **[P2] Check exposure after runtime bind/auth resolution:** `src/gateway/server.impl.ts:502`
  - body: The hook logs conflicts from `cfgAtStart` before runtime bind, host, auth, and tailscale overrides are applied. That can emit a critical unauthenticated-exposure warning for stale file config, or miss the effective listener/auth state that `resolveGatewayRuntimeConfig` later enforces.
  - confidence: 0.88
- **[P2] Inspect per-agent exec and sandbox overrides:** `src/config/detect-conflicts.ts:24-31`
  - body: The detector reads only default sandbox mode plus global `tools.exec` and `tools.elevated`, but runtime behavior resolves `agents.list[]` sandbox and tool overrides per agent. Defaults can look safe while a configured agent has the conflicting state this feature is meant to surface.
  - confidence: 0.87
- **[P2] Remove the exec.security=full hardening advice:** `src/config/detect-conflicts.ts:79`
  - body: The info message recommends `exec.security=full` for maximum restriction, but current audit/docs treat `full` as broad trust and recommend allowlist prompts for tighter posture. Operators following this startup guidance could weaken their exec policy.
  - confidence: 0.93
