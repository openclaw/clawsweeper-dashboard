## Summary

Keep open: the feature direction is useful, but the current PR still has blocking correctness and security issues. Static review against current main shows the added startup pass can throw, runs before effective runtime bind/auth resolution, misses per-agent config, and gives unsafe `exec.security=full` hardening advice.

## What This Changes

The PR adds a config conflict detector with tests and calls it during gateway startup to log exec, sandbox, elevated, and unauthenticated-exposure diagnostics.

## Best Possible Solution

Rework the diagnostics against effective gateway runtime state and per-agent exec/sandbox/elevated resolution, keep the pass non-fatal, and align every message with existing security audit and exec policy semantics.

## Reproduction Assessment

Yes. Static review of the PR diff against current main gives high-confidence source reproduction paths: the hook runs before runtime resolution, calls a resolver that can throw, and checks raw global defaults instead of effective per-agent runtime settings.

## Solution Assessment

No. The requested startup warning feature is reasonable, but this implementation is not the narrowest maintainable solution because it should use effective runtime/per-agent state and docs-aligned security semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep conflict checks from throwing at startup:** `src/config/detect-conflicts.ts:24`
  - body: `detectConfigConflicts` runs during gateway startup but calls `resolveSandboxConfigForAgent(config)` without isolating resolver failures. That resolver normalizes SSH sandbox settings and throws for a relative `sandbox.ssh.workspaceRoot`, so a warning-only diagnostic can turn existing configs into a hard gateway boot failure.
  - confidence: 0.91
- **[P2] Check exposure after runtime bind resolution:** `src/gateway/server.impl.ts:502`
  - body: `logConfigConflicts(cfgAtStart)` runs before `resolveGatewayRuntimeConfig` applies CLI/runtime bind, host, auth, and tailscale overrides. This can emit a critical unauthenticated-exposure warning from stale file config even when the effective runtime bind/auth state is safe.
  - confidence: 0.88
- **[P2] Resolve conflicts with agent overrides:** `src/config/detect-conflicts.ts:24-31`
  - body: The detector reads only the default sandbox mode and global `tools.*` settings, but runtime exec, elevated, and sandbox behavior resolves `agents.list[]` overrides per agent. Defaults can look safe while a specific agent has the conflicting state this feature is meant to surface.
  - confidence: 0.87
- **[P2] Remove the exec.security=full hardening advice:** `src/config/detect-conflicts.ts:79`
  - body: The info message recommends `exec.security=full` for maximum restriction, but current audit/docs treat `full` as broad break-glass exec trust and prefer allowlist with prompts. Operators following this guidance could weaken their exec posture.
  - confidence: 0.93
