## Summary

Keep open: current main still lacks the requested startup conflict detector, and the PR branch is a useful implementation sketch but has blocking correctness and security-guidance issues that should be fixed before merge.

## What This Changes

This PR adds a config-conflict detector and gateway startup hook with tests for exec, sandbox, elevated, exposed gateway, and near-high-safety configuration combinations.

## Best Possible Solution

Refresh the feature so diagnostics are non-throwing, run against effective runtime and per-agent state, reuse existing security-audit semantics where possible, and use docs-aligned exec security guidance, with targeted regression tests for each conflict class.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths: the PR calls a resolver that can throw, logs before runtime bind/auth overrides are resolved, reads only default/global settings, and recommends exec.security=full contrary to current docs.

## Solution Assessment

No. The feature direction is useful, but this implementation checks raw/default config too early and includes incorrect hardening advice; the safer solution is to base warnings on effective runtime/per-agent config and existing audit semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep conflict checks from throwing at startup:** `src/config/detect-conflicts.ts:24`
  - body: detectConfigConflicts runs during gateway startup but calls resolveSandboxConfigForAgent, which normalizes SSH workspaceRoot and can throw for relative roots. A warning-only diagnostic should not block gateway boot for existing configs.
  - confidence: 0.91
- **[P2] Check exposure after runtime bind resolution:** `src/gateway/server.impl.ts:502`
  - body: logConfigConflicts uses cfgAtStart before runtime bind, host, auth, and tailscale overrides are applied. That can emit a critical unauthenticated exposure warning even when the effective runtime bind is loopback.
  - confidence: 0.88
- **[P2] Resolve conflicts per agent:** `src/config/detect-conflicts.ts:24-31`
  - body: The detector only evaluates the default sandbox mode and global tools.* values, but runtime exec/elevated/sandbox behavior resolves agents.list[] overrides per agent. Defaults can look safe while a specific agent has the conflicting state this feature is meant to catch.
  - confidence: 0.87
- **[P2] Remove the exec.security=full hardening advice:** `src/config/detect-conflicts.ts:79`
  - body: The info message recommends exec.security=full for maximum restriction, but current docs and audit semantics define full as the broadest host exec mode. Following this advice can weaken an operator's exec posture.
  - confidence: 0.93
