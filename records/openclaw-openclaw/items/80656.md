## Label Justifications

- `P1`: The PR addresses a live Swift gateway-auth failure that can leave users stuck before pairing on deployed older gateways.
- `merge-risk: 🚨 compatibility`: Changing Swift's default device-auth signature from v3 to v2 affects compatibility behavior across upgraded and older gateway deployments.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily metadata that v3 cryptographically binds into the device signature.

## Summary

Keep this PR open for maintainer/security review: the patch is focused and has credible live proof, but merging it is a compatibility and security-boundary decision because Swift connect signatures would temporarily default back to v2.

## What This Changes

The PR adds a Swift v2 device-auth payload builder, switches the OpenClawKit and macOS Wizard connect signers to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Merge only after maintainers accept the temporary Swift v2 signing policy and own the path back to v3 once deployed gateways uniformly verify v3.

## Reproduction Assessment

Yes: current main and v2026.6.6 Swift signers build v3 payloads, while a v2-only deployed gateway would reject that signature before pairing. The PR body adds redacted live before/after logs from one older Fly gateway; I did not run that older image locally.

## Solution Assessment

Mostly yes as a temporary compatibility mitigation: the patch covers both Swift signers and keeps the v3 builder available. It is not the final security direction, so maintainer/security acceptance of the default v2 policy and restoration plan is still required.
