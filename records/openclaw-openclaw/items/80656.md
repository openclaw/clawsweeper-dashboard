## Summary

Current main still signs Swift OpenClawKit gateway connects with the v3 payload while the gateway accepts both v3 and v2; the PR is a focused compatibility fix with sufficient live proof, but the default-signing downgrade is an auth compatibility/security tradeoff that needs maintainer approval before merge.

## What This Changes

The PR adds Swift OpenClawKit v2 device-auth payload construction, switches Swift gateway connection signing from v3 to v2, adds device auth reset helpers, and covers the v2 payload vector with a Swift test.

## Best Possible Solution

Land a maintainer-approved compatibility path that keeps older deployed Swift OpenClawKit clients pairing with v2-only gateways while documenting or tracking the fleet-upgrade point where v3 metadata-bound signing becomes the default again.

## Reproduction Assessment

Yes for the reported deployed-gateway symptom via the PR body's redacted live logs, and source inspection explains why v3-only Swift signing fails against a v2-only verifier. I did not independently run a live older gateway, so the current-main reproduction is source-backed rather than locally reproduced.

## Solution Assessment

Unclear until maintainers decide the compatibility policy: switching Swift OpenClawKit to v2 is the narrowest fix for older deployed gateways, but it intentionally gives up v3 metadata signing as the default and leaves sibling native signing paths unchanged.
