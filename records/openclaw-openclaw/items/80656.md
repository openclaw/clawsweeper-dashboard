## Summary

Keep this PR open: current main and the latest release still have Swift OpenClawKit sign gateway connect with the v3 payload, while the PR provides a narrow v2-signing compatibility window for deployed gateways that still verify only v2. The patch has no discrete code-review finding, but merging is still a deliberate compatibility/security-boundary tradeoff that needs maintainer landing or pause judgment.

## What This Changes

The PR adds Swift v2 device-auth payload construction, switches Swift connect signing in OpenClawKit and the macOS wizard to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the narrow v2 Swift connect compatibility window only if maintainers accept the temporary metadata-binding reduction, and keep the restoration path to buildV3 or explicit signature-version negotiation tied to the gateway fleet upgrade.

## Reproduction Assessment

Yes, for the reported deployed path: current Swift code signs v3 while the PR body supplies before/after live logs from an older v2-only gateway showing device-signature-invalid before the patch and Connected after it. I did not run a local older-gateway fixture in this read-only review.

## Solution Assessment

Yes as a temporary compatibility mitigation, not as the permanent best end state. The safer long-term solution is to restore buildV3 or add explicit signature-version negotiation once deployed gateways verify v3-before-v2.
