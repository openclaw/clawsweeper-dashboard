## Summary

Keep open for maintainer landing or pause judgment: current main and the latest release still sign Swift connects with v3, while this PR supplies a scoped v2 compatibility fix with sufficient live proof and no concrete correctness findings. The remaining issue is the intentional temporary compatibility/security tradeoff, not contributor action.

## What This Changes

This PR adds Swift v2 device-auth payload helpers, switches the OpenClawKit and macOS wizard connect signers to the v2 compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 compatibility window with the recorded restoration trigger, then restore v3 signing or add explicit signature-version negotiation once deployed gateways support v3-before-v2.

## Reproduction Assessment

Yes, with source and supplied live proof. Current main signs Swift connects with v3, and the PR body provides redacted before/after logs from an older deployed gateway that rejected before pairing and connected after v2 signing; I did not rerun that external gateway locally.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The PR scopes the downgrade to Swift connect signing and leaves server v3-before-v2 verification in place; the long-term solution is restored v3 signing or explicit version negotiation after gateway upgrades.
