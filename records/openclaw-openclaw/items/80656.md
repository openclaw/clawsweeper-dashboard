## Label Justifications

- `P1`: The PR addresses a live Swift gateway-auth failure that can leave users stuck before pairing on deployed older gateways.
- `merge-risk: 🚨 compatibility`: Changing Swift's default device-auth signature from v3 to v2 affects compatibility behavior across upgraded and older gateway deployments.

## Summary

Keep open for maintainer review: the current head appears to address the prior concrete gaps and has sufficient live proof, but the deliberate Swift v2 signing default is still a compatibility and auth-boundary choice that needs owner acceptance before merge.

## What This Changes

The PR adds a Swift v2 device-auth payload builder, switches both Swift connect signers to the compatibility v2 payload, and adds Swift payload-vector tests.

## Best Possible Solution

Merge only after maintainers accept the temporary Swift v2 signing policy and own the path back to v3 once deployed gateways uniformly verify v3.

## Reproduction Assessment

Yes: current main and v2026.5.28 Swift signers build v3 payloads, while the PR body supplies redacted live before/after logs from an older v2-only Fly gateway. I did not run that older gateway image locally.

## Solution Assessment

Mostly yes as a temporary compatibility mitigation: the current patch covers both Swift signers and keeps the v3 builder available. It is not the final security direction, so maintainers still need to accept the default v2 policy and restoration plan.
