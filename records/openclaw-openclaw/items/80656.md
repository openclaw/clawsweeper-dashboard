## Label Justifications

- `P1`: The PR addresses a real Swift Gateway pairing failure where older deployed gateways can reject the client before any pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes across deployed gateway versions and must remain compatible with both older v2-only and newer v3-capable gateways.
- `merge-risk: 🚨 security-boundary`: The compatibility payload omits platform and deviceFamily from the signed Swift connect payload while this temporary window is active.

## Summary

Keep open: current main and the latest release still sign Swift connect device-auth with v3, while this PR is a focused, proof-backed temporary compatibility mitigation for older v2-only gateways. I found no concrete patch defect, but landing still needs maintainer acceptance of the deliberate compatibility and security-boundary tradeoff.

## What This Changes

The PR adds Swift v2 device-auth payload helpers, switches the OpenClawKit and macOS wizard connect signers to a v2 compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 compatibility window with the recorded restoration trigger, then restore v3 signing or add explicit signature-version negotiation once deployed gateways support v3-before-v2.

## Reproduction Assessment

Yes, with source and supplied live proof. Current main Swift signs v3-only, current server code verifies v3 then v2, and the PR body provides redacted before/after logs from an older deployed gateway that rejected before pairing and connected after v2 signing; I did not rerun that external gateway locally.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The PR scopes the downgrade to Swift connect signing and keeps server v3-before-v2 verification in place; the longer-term solution is restored v3 signing or explicit version negotiation after gateway upgrades.
