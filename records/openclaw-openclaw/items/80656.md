## Summary

Keep open: the PR is still useful because current main signs Swift connect requests with v3 while the reported deployed-gateway failure needs a temporary v2 Swift compatibility payload, and a maintainer has accepted that tradeoff with fresh proof.

## What This Changes

The branch adds a Swift v2 device-auth payload builder, switches macOS wizard and OpenClawKit connect signing to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 Swift connect default with the documented restoration trigger, then move Swift signing back to v3 or explicit version negotiation once deployed gateways are upgraded.

Yes. Current main source shows Swift signs v3, and the PR supplies redacted live logs showing an older v2-only Fly gateway rejecting that path with device signature invalid and connecting after v2 signing.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The narrow Swift helper leaves v3 available and a maintainer has accepted the v2 window until gateways can verify v3 before v2.
