## Label Justifications

- `P1`: The PR addresses a live Swift gateway-auth failure that can leave downstream apps stuck before pairing on older deployed gateways.
- `merge-risk: 🚨 compatibility`: Changing Swift's default device-auth signature from v3 to v2 affects upgraded and older gateway compatibility behavior.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily fields that v3 binds into the device signature.

## Summary

Keep open for maintainer/security review: the PR is a focused, proof-backed compatibility fix, but it intentionally makes Swift connect signatures use the legacy v2 payload by default, so the remaining decision is whether that temporary metadata-binding downgrade is acceptable.

## What This Changes

The PR adds a Swift v2 device-auth payload builder, switches both Swift gateway connect signers to the v2 compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Merge only after maintainers explicitly accept the temporary Swift v2 signing default and record the version-gate or rollout trigger for returning Swift connect signing to v3.

## Reproduction Assessment

Yes, by source plus live proof: current main and v2026.6.6 Swift signers build v3 payloads, while the PR body provides redacted live logs from an older v2-only Fly gateway showing the before-fix rejection and after-fix pairing/connection. I did not run that older gateway image locally.

## Solution Assessment

Mostly yes as a temporary compatibility mitigation: the patch covers both Swift signers and current gateway code verifies v2 after v3. It is not the final security direction because docs and current code prefer v3 metadata binding, so maintainer acceptance and a restoration plan are still needed.
