## Label Justifications

- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before a pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes for all Swift clients and may affect future v3-only or metadata-sensitive gateway behavior.

## Summary

Keep this PR open for maintainer landing or pause. The patch is a focused compatibility fix with sufficient live proof and no blocking code finding, but merging it intentionally carries a temporary Swift device-auth compatibility and security-boundary tradeoff.

## What This Changes

The PR adds a Swift v2 device-auth payload builder, routes Swift connect signing through it for older gateway compatibility, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 Swift connect default with the recorded restoration trigger, then move Swift signing back to v3 or explicit version negotiation once deployed gateways are upgraded.

## Reproduction Assessment

Yes. Current main source shows Swift signs v3, and the PR supplies redacted live logs showing an older v2-only Fly gateway rejecting that path with device signature invalid and connecting after v2 signing.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The narrow Swift helper leaves v3 available, and a maintainer has accepted the v2 window until gateways can verify v3 before v2.
