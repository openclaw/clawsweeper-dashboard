## Label Justifications

- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before a pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes for OpenClawKit and the macOS wizard, affecting compatibility expectations across deployed gateway versions.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily from signed material, reducing current Swift signature binding while the temporary window is active.

## Summary

Keep open: this is a focused bug-fix PR with sufficient real-environment proof and no discrete code finding, but it intentionally changes Swift connect signing from v3 to v2 compatibility and still needs maintainer landing or pause judgment on the accepted temporary compatibility/security tradeoff.

## What This Changes

Adds a Swift OpenClawKit v2 device-auth payload builder, switches the shared Swift and macOS wizard connect signers to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary Swift v2 connect default if maintainers accept the recorded tradeoff, then return Swift signing to v3 or explicit signature-version negotiation once deployed gateways verify v3-before-v2.

## Reproduction Assessment

Yes, with source and supplied live proof. Current main Swift signs v3, and the PR body shows redacted before/after logs from an older deployed gateway that rejected before pairing and connected after v2 signing; I did not rerun that external gateway locally.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The narrow Swift helper leaves v3 available and current servers verify v3 before v2; the safer long-term path is restored v3 signing or explicit version negotiation after gateway upgrades.
