## Label Justifications

- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before a pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes for OpenClawKit and the macOS wizard, affecting compatibility expectations across deployed gateway versions.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily from signed material, reducing current Swift signature binding while the temporary window is active.

## Summary

Keep open: current main and v2026.6.6 still sign Swift connect device auth with the v3 payload, while this PR addresses a real older-gateway compatibility failure with supplied live proof and a maintainer-accepted temporary tradeoff.

## What This Changes

The PR adds a Swift v2 device-auth payload helper, switches the Swift OpenClawKit and macOS wizard connect signers to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary Swift v2 connect default with the recorded restoration trigger, then return Swift signing to v3 or explicit signature-version negotiation once deployed gateways are upgraded.

## Reproduction Assessment

Yes, with source and supplied live proof. Current main Swift signs v3, and the PR body provides redacted before/after logs from an older deployed gateway that rejected before pairing and connected after v2 signing; I did not rerun that external gateway locally.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The narrow Swift helper leaves v3 available and current servers verify v3 before v2; the safer long-term path is restored v3 signing or explicit version negotiation after gateway upgrades.
