## Label Justifications

- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before a pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes for all Swift clients and may affect future v3-only or metadata-sensitive gateway behavior.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily from the signed material, reducing the current Swift signature binding until maintainers move back to v3.

## Summary

Keep open. Current main still signs Swift gateway connect requests with the v3 payload, while this PR provides a focused v2 compatibility mitigation with sufficient live proof; the remaining blocker is maintainer/security acceptance of temporarily defaulting Swift connect signatures back to v2.

## What This Changes

The PR adds Swift v2 device-auth payload helpers, switches the macOS wizard and shared OpenClawKit connect signers to the v2 compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Maintainers should either land this scoped Swift v2 compatibility default with an explicit return-to-v3/removal plan, or revise it to preserve v3 by default while still covering older deployed gateways.

## Reproduction Assessment

Yes. Current main source shows Swift signs v3, and the PR supplies redacted live logs showing an older v2-only Fly gateway rejecting that path with device signature invalid and connecting after v2 signing.

## Solution Assessment

Unclear as a permanent fix, but it is an acceptable mitigation. Current gateways accept both v3 and v2 while docs prefer v3, so maintainers need to own the temporary default and removal plan.
