- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before any pending device exists to approve.

## Summary

Keep open for maintainer landing or pause judgment: current main and the latest release still sign Swift connects with the v3 payload, while this focused PR provides the temporary v2 compatibility path that a maintainer explicitly accepted as a deployment-window tradeoff.

## What This Changes

The PR adds a Swift v2 device-auth payload builder, switches OpenClawKit and the macOS wizard connect signing to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 compatibility window with the recorded restoration trigger after current-main mergeability and required checks refresh; later restore v3 signing or negotiate signature version once deployed gateways support v3-before-v2.

## Reproduction Assessment

Yes, with source and supplied live proof. Current main signs Swift connects with v3, and the PR body provides redacted before/after logs from an older deployed gateway that rejected before pairing and connected after v2 signing; I did not rerun that external gateway locally.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The PR scopes the downgrade to Swift connect signing and leaves server v3-before-v2 verification in place; the long-term solution is restored v3 signing or explicit version negotiation after gateway upgrades.
