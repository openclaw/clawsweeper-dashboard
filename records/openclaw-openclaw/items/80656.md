## Label Justifications

- `P1`: The PR addresses a broken real gateway pairing workflow where Swift clients can be rejected before a pending device exists to approve.
- `merge-risk: 🚨 compatibility`: The default Swift connect signature payload changes for all Swift OpenClawKit and macOS wizard clients, affecting older and future gateway compatibility expectations.
- `merge-risk: 🚨 security-boundary`: The v2 payload omits platform and deviceFamily from signed material, reducing the current Swift signature binding until maintainers restore v3 or negotiate versions.

## Summary

Keep open: current main and the latest release still have Swift clients sign the v3 device-auth payload, while this PR provides a narrow, maintainer-accepted temporary v2 compatibility path for deployed gateways that still verify v2 only.

## What This Changes

The PR adds a Swift OpenClawKit v2 device-auth payload builder, switches OpenClawKit and macOS wizard connect signing to that compatibility payload, and adds Swift payload-vector tests.

## Best Possible Solution

Land the scoped temporary v2 Swift connect default with the documented restoration trigger, then move Swift signing back to v3 or explicit signature-version negotiation once deployed gateways are upgraded.

## Reproduction Assessment

Yes. Current main source shows Swift signs v3, and the PR body supplies redacted before/after live logs showing an older v2-only Fly gateway rejecting the v3 path and connecting after v2 signing.

## Solution Assessment

Yes as a temporary mitigation, not as the permanent end state. The narrow Swift helper leaves v3 available, server verification still accepts v3 before v2, and a maintainer accepted the v2 window until gateways can verify v3 before v2.
