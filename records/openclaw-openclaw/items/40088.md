## Summary

Keep open: current main and the latest release still leave unconfigured builtin memory search without a periodic fallback, and the interval path still does not mark memory dirty before syncing. The open linked fix PR now owns the candidate repair, but it needs conflict refresh and maintainer acceptance of the default-on sync behavior before this issue can close.

## What This Changes

Reports that `memory_search` can return stale or missing results after the memory file watcher stops delivering events and no periodic fallback reindexes changed memory files.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/90019 after conflict refresh and maintainer acceptance of the default-on fallback, or choose an explicit watcher-health/opt-in fallback design.

## Reproduction Assessment

Yes at source level: current main resolves unconfigured `intervalMinutes` to `0`, and an interval sync would still not set `dirty` before the dirty-gated memory sync path. I did not establish a live multi-hour chokidar outage reproduction in this read-only pass.

## Solution Assessment

Unclear as a product decision: the open PR targets the right fallback code path, but enabling periodic sync by default changes upgrade behavior. A maintainer should choose between default-on periodic fallback and a narrower watcher-health or opt-in fallback design.
