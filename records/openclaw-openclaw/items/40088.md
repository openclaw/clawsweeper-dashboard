## Summary

Keep open. Current main and v2026.5.19 still leave the reported dead-watcher fallback unresolved: default interval sync is disabled, and a configured interval still calls sync without marking memory files dirty, so the memory-file sync gate can skip after watcher events stop. The earlier watcher directory fix and error listener are useful partial mitigations, but https://github.com/openclaw/openclaw/pull/42042 and https://github.com/openclaw/openclaw/pull/65755 closed unmerged, and https://github.com/openclaw/openclaw/pull/72748 covers adjacent first-search freshness rather than the dead-watcher dirty=false fallback.

## What This Changes

This issue asks OpenClaw to keep memory_search indexes fresh when chokidar stops delivering events by adding watcher resilience and a periodic sync fallback.

## Best Possible Solution

Add a bounded memory-core resilience fix that keeps the existing watcher improvements, enables a conservative periodic fallback by default, and makes interval ticks run the lightweight memory-file scan path without forcing full reindex.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: if chokidar stops delivering events, this.dirty remains false, the default interval is disabled, and even an explicit interval calls sync without making the memory-file gate true. I did not run a live multi-hour gateway repro in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing sync interval as a fallback, respect explicit opt-out, and make interval sync perform the lightweight file listing/hash comparison path rather than a force reindex or a new heartbeat-file mechanism.
