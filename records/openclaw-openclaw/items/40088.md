## Label Justifications

- `P1`: A dead watcher can leave long-lived memory_search indexes stale for real users and break a core agent memory workflow until manual reindex or restart.
- `impact:session-state`: The issue is about memory/index state drifting from persisted workspace memory files during long-running sessions.

## Summary

Keep open. Current main only has the watcher directory-shape fix and an error logger; it still disables periodic memory sync by default, and a configured interval sync can still no-op after watcher death because it calls sync without setting the memory dirty flag.

## What This Changes

The issue asks for memory_search to recover when the memory-core chokidar watcher stops firing so the long-lived vector index does not stay stale.

## Best Possible Solution

Add a periodic safety-net sync for memory files, preserve explicit opt-out, make interval ticks trigger the lightweight memory file scan when watcher events are absent, and keep the existing watcher error logging with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Default intervalMinutes is 0, and even a configured interval calls sync without dirty or force while runSync only scans memory files when dirty, force, or full-reindex is true.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing interval sync surface as a lightweight fallback, preserve explicit 0 opt-out, and cover the dead-watcher/dirty-gate path in tests.
