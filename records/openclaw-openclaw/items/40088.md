## Summary

Keep open: current main and the latest release still leave builtin memory search without a default periodic fallback, and the interval path does not force a memory-file rescan when watcher events are missed. The open fixing PR at https://github.com/openclaw/openclaw/pull/90019 is the right owner for the remaining work and still needs maintainer acceptance of the default-on interval behavior.

## What This Changes

The issue reports that memory_search can return stale or empty results after the file watcher stops delivering memory-file change events.

## Best Possible Solution

Land or reject the linked fixing PR after maintainers decide whether default-on periodic builtin memory sync is acceptable for upgrades; otherwise keep the old default and document or doctor-recommend the opt-in fallback.

## Reproduction Assessment

Yes at source level: current main defaults intervalMinutes to 0, the interval callback does not mark memory dirty, and runSync skips memory-file sync when dirty is false. I did not run a long-lived gateway to reproduce the exact chokidar outage.

## Solution Assessment

Yes, conditionally: the linked PR is the narrow current fix path because it uses the existing intervalMinutes surface and repairs the interval dirty gate. The remaining question is whether maintainers accept changing the default for existing unconfigured users.
