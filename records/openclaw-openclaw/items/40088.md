## Summary

Keep open. Current main fixes the earlier Chokidar v5 glob/FD watcher target problem, but it does not implement the resilience requested here: the default periodic fallback remains disabled, a configured interval tick can still skip memory files when the watcher has stopped marking the manager dirty, and the built-in watcher has no error handler.

## What This Changes

The issue requests memory_search index resilience when chokidar stops delivering file events, including watcher health/error visibility and a periodic fallback so changed memory files are reindexed.

## Best Possible Solution

Add a focused memory-core fix: log watcher errors, make configured interval sync perform a lightweight hash-based memory file scan even when no watcher event marked the manager dirty, preserve explicit intervalMinutes: 0 opt-out, and make an intentional default-cadence decision with tests and changelog/docs updates.

## Reproduction Assessment

Partially yes. The several-hour chokidar failure was not locally reproduced, but the current fallback defect is directly reproducible from code: interval sync calls sync without setting dirty, and runSync skips memory files when dirty is false.

## Solution Assessment

Yes for the technical direction, with one maintainer decision. The narrowest maintainable fix is in memory-core sync handling and memory-search config defaults; a synthetic heartbeat file probe can remain a follow-up unless maintainers want stronger watcher liveness detection now.
