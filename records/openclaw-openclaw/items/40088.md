## Label Justifications

- `P1`: A stale memory index breaks a core long-term memory workflow for users while presenting existing on-disk memory as missing from search.
- `impact:session-state`: The affected state is the memory/search index used to carry context across agent sessions.

## Summary

Keep open. Current main has the watcher error handler, but it still defaults `memorySearch.sync.intervalMinutes` to `0` and configured interval sync does not mark memory files dirty, so a dead watcher can still leave the index stale.

## What This Changes

The issue reports stale `memory_search` results after chokidar stops delivering file events and asks for watcher health/error logging plus a periodic fallback.

## Best Possible Solution

Land a narrow memory-core fix that makes interval sync a lightweight fallback for watcher silence, keeps explicit opt-out behavior, documents the default, and covers the dirty-gate path with focused regression tests.

## Reproduction Assessment

Yes, at source level. After a normal sync clears `dirty`, a stopped watcher means no event sets it again; the default interval is disabled, and even an explicitly configured interval currently calls sync without setting `dirty`, so memory-file indexing is skipped.

## Solution Assessment

Yes, the narrow maintainable direction is the prior PR shape: use the existing `intervalMinutes` setting as an opt-out-respecting fallback, set the memory dirty flag before interval sync, and add focused tests rather than adding a new watcher heartbeat subsystem first.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40088 on current main. The current memory watcher error handler already exists, but default `agents.defaults.memorySearch.sync.intervalMinutes` is still `0`, and `ensureIntervalSync` calls `sync({ reason: "interval" })` without marking memory dirty, so interval sync cannot recover after chokidar stops firing. Implement a narrow memory-core fix using the existing config surface: make the default interval fallback non-zero while preserving explicit `intervalMinutes: 0`, set the memory dirty flag before interval-triggered memory sync so `syncMemoryFiles` lists files and hash-skips unchanged entries, and avoid forcing full reindex on every tick. Consult the closed unmerged PRs https://github.com/openclaw/openclaw/pull/42042 and https://github.com/openclaw/openclaw/pull/65755 for prior review context, but adapt to current `extensions/memory-core` code. Add focused regression tests for default/respected override/explicit zero and for interval sync after dirty was cleared and watcher events stop. Update docs/help if the default or field visibility changes. Do not broaden watcher lifecycle policy or add a new config option unless the existing interval fallback cannot solve the reported bug.
