## Summary

Keep open. Current main has part of the watcher resilience work, but the default sync interval is still disabled and interval ticks still do not mark memory dirty, so a watcher that stops after a successful sync can still leave the index stale. The exact fix PRs were closed unmerged, and the open recursive-memory PR overlaps stale search timing but does not cover the watcher-death fallback.

## What This Changes

The issue asks for memory_search to recover when the chokidar watcher stops delivering file events so modified memory files are re-indexed instead of leaving stale SQLite rows.

## Best Possible Solution

Land a focused memory-core resilience fix that keeps the existing explicit `intervalMinutes: 0` opt-out, enables a modest default fallback interval, and makes interval sync perform a lightweight dirty/hash scan when watcher events stop.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: after a successful sync clears `dirty`, a silently stopped watcher will not set `dirty` again, default `intervalMinutes` is `0`, and search-triggered sync is skipped unless dirty/session-dirty is already true. I did not run a long-lived live gateway reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing interval fallback active by default and make that interval trigger a lightweight memory-file scan; a broader heartbeat or recursive-memory refactor can remain separate.
