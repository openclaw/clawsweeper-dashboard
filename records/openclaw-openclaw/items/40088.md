## Summary

Keep open. Current main fixed some watcher failure modes, but the reported default fallback gap remains: default `intervalMinutes` is still disabled, and a configured interval still does not mark memory dirty before syncing, so a silent watcher stop can still leave memory files stale.

## What This Changes

The issue asks for memory_search resilience when the chokidar watcher stops delivering events, including logging and a periodic fallback so modified memory files are re-indexed.

## Best Possible Solution

Keep the existing watcher fixes and land an approved memory-core fallback: interval sync should actually scan memory files after watcher loss, and maintainers should choose the default cadence or equivalent health-check policy.

## Reproduction Assessment

No full live long-uptime reproduction was established here. Source inspection gives a high-confidence fallback-path reproduction once watcher events stop: default interval sync is disabled, and configured interval sync skips memory files when `dirty` is false.

## Solution Assessment

Unclear until maintainers decide the default background-sync policy. The closed PR direction is technically focused for the bug, but defaulting periodic sync to a nonzero cadence has performance and provider-cost implications.
