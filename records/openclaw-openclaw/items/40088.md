## Summary

Keep open: current main only partially addresses the watcher failure. The memory watcher now logs errors and watches the memory directory directly, but the default periodic fallback is still disabled and the interval path still does not force a lightweight memory-file scan when `dirty` is false, so the stale-index failure can remain after watcher events stop.

## What This Changes

The issue reports that `memory_search` can return stale or empty results after the chokidar watcher stops firing because the memory index is never refreshed and the default interval fallback is disabled.

## Best Possible Solution

Add a bounded memory-core fallback that keeps the existing watcher hardening, makes interval ticks perform a lightweight enumerate/hash sync for memory files, preserves explicit opt-out, and covers both default config and configured interval behavior with regression tests.

## Reproduction Assessment

Yes, at source level: current main still defaults `intervalMinutes` to `0`, and the interval path would not scan memory files unless `dirty` is already true. I did not run the long-lived chokidar failure itself.

## Solution Assessment

Yes, the narrowest maintainable fix is the interval fallback already sketched by the closed PRs: set a safe nonzero default, preserve explicit `0`, and mark or otherwise route interval sync so it performs hash-based memory-file refresh without a forced full reindex.
