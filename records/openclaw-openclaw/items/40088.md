## Summary

Keep open: current main still has the core stale-index failure mode. The watcher now logs errors and watches the memory directory directly, but the default periodic fallback is disabled and the interval path still calls sync without making memory files eligible for enumeration.

## What This Changes

The issue reports that `memory_search` can return stale results after the chokidar memory-file watcher stops firing because there is no effective fallback sync.

## Best Possible Solution

Land a narrow memory-core fix that makes periodic fallback effective for default users while respecting explicit opt-out and adds regression coverage for interval-driven memory-file sync after watcher loss.

## Reproduction Assessment

Yes, source-reproducible: default config resolves `intervalMinutes` to `0`, and even when configured the interval tick does not set `dirty`, while `runSync` skips memory files unless `dirty` is true. I did not establish a live multi-hour chokidar failure in this read-only pass.

## Solution Assessment

Yes: the maintainable fix is to use the existing `sync.intervalMinutes` fallback, make interval sync enumerate changed memory files without a force reindex, and preserve explicit `intervalMinutes: 0` opt-out. The unmerged replacement PR is useful prior art but should be reapplied against current main with focused tests.
