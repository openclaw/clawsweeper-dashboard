## Label Justifications

- `P1`: Silent watcher failure can leave memory_search serving stale state throughout long-running agent workflows without warning.
- `impact:session-state`: The searchable memory index can drift from persisted memory files, causing current memory context to disappear from retrieval.

## Summary

This remains a valid source-reproducible memory-index bug. Current main has watcher-error logging but no default periodic fallback, and the open closing PR now owns the focused implementation.

## What This Changes

Prevent the builtin memory-search index from remaining stale when filesystem watcher events stop by adding a reliable periodic rescan fallback and diagnostics.

## Best Possible Solution

Use the existing lightweight hash-comparison sync path as the safety net, preserve an explicit zero opt-out, and land it only with maintainer acceptance of the default cadence and its upgrade cost.

## Reproduction Assessment

Yes at source level: current main defaults periodic sync off, so silent watcher event loss can leave no timer-based rescan, and the report includes concrete stale SQLite metadata and inotify observations. A real multi-hour current-main chokidar outage has not been reproduced.

## Solution Assessment

Yes for the technical fallback: marking memory dirty before a lightweight interval sync reuses the canonical hash-rescan path instead of forcing a full reindex. The unresolved question is whether that interval should become default-on for all existing unconfigured users.
