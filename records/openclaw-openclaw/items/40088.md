## Label Justifications

- `P1`: A stale memory_search index can break long-running agent memory retrieval for active users even while files exist on disk.
- `impact:session-state`: The issue is about memory index state drifting from persisted workspace memory files during an active gateway session.

## Summary

Keep open: current main has partial watcher hardening, but the built-in memory fallback is still disabled by default and interval sync still does not mark memory dirty before syncing, so the reported stale-index failure remains source-reproducible when watcher events stop.

## What This Changes

The issue asks for memory_search to keep its index fresh when the chokidar watcher stops delivering file events, instead of silently returning stale or empty results.

## Best Possible Solution

Add a bounded memory-core fallback that respects explicit opt-out, gives default users periodic lightweight freshness protection, marks memory dirty before interval scans, and covers watcher-death behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level. With the built-in memory manager, if watcher events stop and dirty remains false, default intervalMinutes is 0; even an opted-in interval currently calls sync without setting dirty while runSync only indexes memory files when dirty, forced, or full-reindexing.

## Solution Assessment

Yes. The narrow maintainable fix is to complete the existing sync fallback contract: mark dirty before interval memory scans, choose a conservative non-zero default while preserving explicit 0, and test that interval sync lists files when watcher events stop.

## Repair Work Prompt

Fix memory_search stale-index fallback for watcher event loss. Current main still defaults agents.defaults.memorySearch.sync.intervalMinutes to 0 and ensureIntervalSync calls sync({ reason: "interval" }) without marking memory dirty, while runSync only indexes memory files when dirty, forced, or full-reindexing. Implement the narrow existing-contract repair: choose a conservative non-zero default interval for memorySearch sync while preserving explicit intervalMinutes: 0 as opt-out; make interval sync perform a lightweight memory-file scan by marking dirty before the interval sync or otherwise passing an explicit lightweight scan signal; do not use force:true for the periodic path because that triggers full reindexing; keep current watcher direct-path/error/no-awaitWriteFinish behavior intact; add regression coverage for default interval resolution, explicit 0, and interval sync after watcher events stop; add a single user-facing changelog fix entry. Do not solve the separate first-search async freshness issue tracked by https://github.com/openclaw/openclaw/issues/52115 unless the same code path is directly required.
