## Summary

Keep open: current main has partially improved memory watcher resilience, but the reported fallback gap remains. Default configs still leave `intervalMinutes` disabled, and even a configured interval still calls `sync()` without marking memory files dirty, so a watcher stall can leave the index stale.

## What This Changes

The issue reports that `memory_search` can return stale results after the chokidar watcher stops delivering file events because no effective periodic fallback reindexes changed memory files.

## Best Possible Solution

Land a narrow memory-core fallback fix that preserves the newer watcher improvements, makes the interval safety net effective, and adds regression tests for default config and dead-watcher interval sync.

## Reproduction Assessment

Yes, at source level: with `intervalMinutes` unset the interval is disabled, and when configured the interval callback still does not set `dirty` before `runSync` checks it. I did not live-reproduce a multi-hour chokidar stall.

## Solution Assessment

Yes: the best fix is the narrow fallback path already explored in the closed PRs, updated onto current main without reverting the direct-directory watcher or changed-file settling work.
