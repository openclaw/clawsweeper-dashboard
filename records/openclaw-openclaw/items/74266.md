## Summary

Keep open. The docs recipe is useful and the earlier Greptile polish was addressed, but the latest head still gives a multi-container setup a `localhost` provider URL that will resolve inside the tenant container instead of reaching the bridge-bound proxy.

## What This Changes

Adds a Gateway docs recipe and sidebar entry for using an external auth-injection proxy with existing provider `baseUrl` and SecretRef/runtime behavior.

## Best Possible Solution

Publish the recipe after it clearly separates same-host or shared-network-namespace `localhost` usage from separate Docker/Kubernetes container deployments, with reachable bridge or host-gateway examples.

## Reproduction Assessment

Yes. By inspection, a tenant container configured with `http://localhost:18080/anthropic` reaches itself, while the recipe’s proxy listens on `172.17.0.1` and identifies tenants by Docker bridge source IP.

## Solution Assessment

No, not as written. The docs-only recipe is the right shape, but the container example needs a reachable proxy address or explicit scoping of `localhost` to same-host/shared-namespace deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a reachable proxy address for containers:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but the proxy skeleton listens on the Docker bridge and keys tenants by bridge source IP. In a separate tenant container, `localhost` is the tenant container itself, so model calls will not route through the auth proxy. Use a bridge/host-gateway address for multi-container deployments, or scope localhost to same-host/shared-network setups.
  - confidence: 0.9
