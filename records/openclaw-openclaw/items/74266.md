## Summary

Keep open. The recipe is useful and uses existing OpenClaw primitives, but it is not merge-ready because the container-oriented example still points OpenClaw at `localhost` while the sample proxy listens on the Docker bridge address; current main already documents and enforces that loopback inside a container is the container itself.

## What This Changes

This PR adds a Gateway docs recipe for an external auth-injection proxy and registers it under the Authentication and secrets docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-host/shared-network-namespace setups from multi-container deployments and uses a container-reachable bridge or host-gateway address where appropriate.

## Reproduction Assessment

Yes. By inspection, a separate tenant container configured with `http://localhost:18080/anthropic` reaches itself, while the sample proxy listens on `172.17.0.1` and identifies tenants by bridge source IP.

## Solution Assessment

No, not as written. A lightweight docs recipe is the right approach, but the address examples need to match the deployment topology so operators do not follow a non-working isolation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy address:** `docs/gateway/external-auth-proxy.md:151`
  - body: The multi-container recipe points `baseUrl` at `http://localhost:18080/anthropic`, but the sample proxy listens on the Docker bridge and keys tenant identity from bridge source IP. Inside a tenant container, `localhost` is the tenant container itself, so model calls will not reach this proxy; use a bridge or host-gateway address for multi-container examples, or explicitly scope `localhost` to same-host/shared-network setups.
