## Summary

Keep open: this is a useful docs-only workaround built on existing provider `baseUrl` and SecretRef primitives, but the current PR head still gives a containerized tenant a `localhost` proxy URL that conflicts with the bridge-bound proxy example, and the claimed field validation is not backed by inspectable real behavior proof.

## What This Changes

The PR adds a new Gateway documentation recipe for an external auth-injection proxy and registers it in the docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-network-namespace localhost setups from multi-container deployments that need a bridge, host-gateway, or service-DNS address, with inspectable redacted deployment proof.

## Reproduction Assessment

Yes. Source inspection is enough: the PR's container-oriented example uses `localhost`, while current docs and runtime guards say loopback inside a container points at that container, not the host-side proxy.

## Solution Assessment

No, not as written. A docs-only workaround is the right shape, but the example should use a container-reachable proxy address or explicitly scope `localhost` to same-namespace deployments before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but inside a tenant container `localhost` is the tenant container itself. Because the sample proxy listens on the Docker bridge and identifies tenants by bridge source IP, readers following this setup will not route model calls through the auth proxy; use a bridge, host-gateway, or service-DNS address for multi-container deployments, or clearly limit localhost to same-namespace setups.
  - confidence: 0.9
