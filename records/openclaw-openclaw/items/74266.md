## Summary

Keep open. The PR is useful docs work and the earlier Greptile feedback was addressed, but the current head is not merge-ready because the container-oriented recipe points the provider `baseUrl` at `localhost` while the sample proxy listens on the Docker bridge; current OpenClaw docs and runtime checks explicitly treat loopback inside a container as the container itself.

## What This Changes

The PR adds an external auth-injection proxy recipe page for provider API-key isolation and registers it in the Gateway authentication/secrets docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-host or shared-network-namespace localhost setups from separate-container deployments that need bridge, host-gateway, or service-DNS addresses.

## Reproduction Assessment

Yes. Source inspection of PR head shows the tenant container example uses `http://localhost:18080/anthropic`, while the sample proxy listens on `172.17.0.1`; current main documents and enforces that loopback inside a container points at the container itself.

## Solution Assessment

No, not as written. The lightweight docs recipe is the right shape, but the container examples should use a reachable bridge, host-gateway, or service address, or explicitly scope localhost to same-host/shared-network setups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but in a tenant container `localhost` is the tenant container itself. Since the sample proxy listens on the Docker bridge and identifies tenants by bridge source IP, readers following this setup will not route model calls through the auth proxy; use a bridge/host-gateway/service address for multi-container deployments or clearly limit localhost to same-host/shared-network setups.
  - confidence: 0.9
