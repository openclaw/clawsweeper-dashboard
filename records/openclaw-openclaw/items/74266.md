## Summary

Keep open: the docs recipe is useful and fits existing OpenClaw primitives, but the current container example can route readers to the tenant container instead of the proxy, and the PR still needs inspectable real behavior proof before merge.

## What This Changes

The PR adds a Gateway docs page for an external auth-injection proxy recipe and registers it in the docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-host/shared-network localhost setups from separate-container deployments that need bridge, host-gateway, or service-DNS addresses.

## Reproduction Assessment

Yes. Source inspection shows the PR's container example uses `localhost`, while current docs and runtime guards establish that loopback inside a container points at the container itself, not the host-side proxy.

## Solution Assessment

No, not as written. The docs-only workaround is the right shape, but the example should use a container-reachable proxy address or explicitly scope `localhost` to same-host/shared-network deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but in a tenant container `localhost` is the tenant container itself. Since the sample proxy listens on the Docker bridge and keys tenant identity from bridge source IP, readers following this setup will not route model calls through the auth proxy; use a bridge, host-gateway, or service-DNS address for multi-container deployments, or clearly limit localhost to same-namespace setups.
  - confidence: 0.9
