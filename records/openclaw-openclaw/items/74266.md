## Summary

Keep open: this docs-only recipe is useful, and the earlier Greptile polish items were addressed, but the current example still points an agent container at loopback while the proxy is described as bridge-bound. That contradicts current OpenClaw container proxy guidance and can make the documented hardening fail.

## What This Changes

This PR adds a Gateway documentation recipe for an external auth-injection proxy and registers it under the Authentication and secrets docs navigation.

## Best Possible Solution

Update the recipe so multi-container deployments use a container-reachable bridge, host-gateway, or service-DNS proxy URL, and reserve localhost only for same-network-namespace or same-host non-container setups.

## Reproduction Assessment

Yes for the review finding. Source inspection is enough: the PR config uses localhost from an agent container while current docs and runtime checks state loopback inside a container points at that container.

## Solution Assessment

No, not as written. A docs-only workaround using existing baseUrl support is the right shape, but the example should use a container-reachable proxy address or clearly scope localhost to same-namespace deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe sets the tenant container's provider `baseUrl` to `http://localhost:18080/anthropic`, but inside that container `localhost` is the tenant itself. Since the proxy skeleton listens on the Docker bridge (`172.17.0.1`) and identifies tenants by bridge source IP, readers following this multi-container setup will not route model calls through the auth proxy; use a bridge, host-gateway, or service-DNS address, or explicitly limit localhost to same-network-namespace deployments.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff is docs-only, but it introduces security-sensitive isolation guidance with a concrete container-networking error.

Concerns:

- **[medium] Container example can miss the auth proxy:** `docs/gateway/external-auth-proxy.md:151`

That can leave operators with a non-working or falsely verified hardening setup unless the docs use a reachable bridge, host-gateway, or service address, or clearly scope localhost to same-namespace deployments.
