## Summary

Keep open. The docs-only recipe is useful, and earlier Greptile polish was addressed, but the current PR head is not merge-ready because its multi-container examples still point provider traffic at `localhost` even though the proxy is described as listening on the Docker bridge.

## What This Changes

The PR adds a Gateway documentation recipe for an external auth-injection proxy and registers it in the Authentication and secrets docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-host/shared-network localhost setups from multi-container deployments and uses bridge or host-gateway addresses that the OpenClaw runtime can actually reach.

## Reproduction Assessment

Yes. By inspection, a tenant container configured with `http://localhost:18080/anthropic` reaches itself, while the same recipe's proxy skeleton listens on `172.17.0.1` and identifies tenants by bridge source IP.

## Solution Assessment

No, not as written. The lightweight docs recipe is the right shape, but the container example needs a reachable bridge or host-gateway URL, or the localhost examples must be scoped to same-host/shared-network deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a container-reachable proxy address:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's multi-container flow keys tenants by Docker bridge source IP and listens on `172.17.0.1`, but the provider config points `baseUrl` at `http://localhost:18080/anthropic`. Inside a tenant container, `localhost` is the tenant container itself, so readers following this setup will not route model calls through the auth proxy. Use a bridge/host-gateway address for multi-container deployments, or scope localhost to same-host/shared-network setups.
  - confidence: 0.9
