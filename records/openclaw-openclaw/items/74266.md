## Summary

Keep open. The docs-only recipe is useful and uses existing SecretRef and provider `baseUrl` primitives, and the earlier Greptile polish was addressed, but the current PR still gives multi-container users a localhost proxy URL that would target the tenant container instead of the external proxy.

## What This Changes

The PR adds a Gateway documentation recipe for an external auth-injection proxy and registers it under Authentication and secrets.

## Best Possible Solution

Publish the recipe after it clearly separates same-host or shared-network-namespace localhost setups from separate-container deployments that need bridge, host-gateway, or service-DNS addresses.

## Reproduction Assessment

Yes. The failure path is source-reproducible: a tenant container configured with `http://localhost:18080/anthropic` reaches itself, while the same page's proxy skeleton listens on `172.17.0.1` and identifies tenants by bridge source IP.

## Solution Assessment

No, not as written. The lightweight docs recipe is the right shape, but the container examples should use a reachable bridge or host-gateway address, or explicitly scope localhost to same-host/shared-network setups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but in a tenant container `localhost` is the tenant container itself. Since the sample proxy listens on the Docker bridge and keys tenant identity from bridge source IP, readers following this setup will not route model calls through the auth proxy; use a bridge/host-gateway/service address for multi-container deployments or clearly limit localhost to same-host/shared-network setups.
  - confidence: 0.9
