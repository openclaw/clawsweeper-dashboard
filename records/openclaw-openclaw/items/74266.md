## Summary

Keep open: the docs-only recipe is useful, but it still documents a container-oriented `localhost` proxy URL that misses the bridge-bound proxy, and the PR lacks inspectable real behavior proof for the security-sensitive setup.

## What This Changes

This PR adds a Gateway docs recipe for using `models.providers.*.baseUrl` with an external auth-injection proxy and registers it in the docs navigation.

## Best Possible Solution

Publish the recipe after it distinguishes same-namespace localhost from multi-container bridge, host-gateway, or service-DNS addresses and includes redacted inspectable deployment proof.

## Reproduction Assessment

Yes. Source inspection is enough: the PR's container-oriented example uses `localhost`, while current docs and runtime guards say loopback inside a container points at that container, not the host-side proxy.

## Solution Assessment

No, not as written. A docs-only workaround is the right shape, but the example should use a container-reachable proxy address or explicitly scope `localhost` to same-network-namespace deployments before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a container-reachable proxy URL:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe points a tenant container at `http://localhost:18080/anthropic`, but inside that container `localhost` is the tenant itself. Since the sample proxy listens on the Docker bridge and identifies tenants by bridge source IP, readers following this setup will not route model calls through the auth proxy; use a bridge, host-gateway, or service-DNS address for multi-container deployments, or clearly limit localhost to same-namespace setups.
  - confidence: 0.9
