## Summary

Keep open. Current main does not contain an equivalent external auth-proxy recipe, and the PR is a useful docs-only workaround for the SecretRef runtime-readability gap, but the latest head still has a blocking container-networking error: the multi-container example uses localhost even though the sample proxy listens on the Docker bridge and identifies tenants by bridge source IP.

## What This Changes

The PR adds `docs/gateway/external-auth-proxy.md` and registers it in `docs/docs.json` as a Gateway authentication and secrets recipe for routing provider requests through an external credential-injection proxy.

## Best Possible Solution

Publish the recipe after it clearly separates same-host/systemd, shared-network-namespace, and multi-container deployments, with address examples that are reachable from the OpenClaw runtime. Keep native credential isolation tracked in #11829 and #10659 rather than implying this recipe replaces it.

## Reproduction Assessment

Yes. The failure path is reproducible by inspection: a tenant container configured with `http://localhost:18080/anthropic` reaches itself, while the same PR's proxy skeleton listens on the Docker bridge address and keys tenant identity from bridge source IP.

## Solution Assessment

No, not as written. The lightweight docs recipe is the right shape, but the container example should use a reachable bridge or host-gateway address, or explicitly say the localhost example only applies when OpenClaw and the proxy share a network namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a reachable proxy address in container examples:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but the proxy skeleton listens on the Docker bridge and identifies tenants by bridge source IP. In a tenant container, `localhost` is the tenant container itself, so readers following this setup will not route model calls through the auth proxy. Use a bridge/host-gateway address for multi-container deployments, or scope localhost to same-host/shared-network setups.
  - confidence: 0.9
