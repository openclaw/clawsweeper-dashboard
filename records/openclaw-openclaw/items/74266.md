## Summary

Keep open. The docs recipe is useful and built on existing OpenClaw primitives, but the latest PR head still gives a container-oriented `localhost` proxy example that misses the bridge-listening proxy in multi-container deployments.

## What This Changes

The PR adds a Gateway docs recipe for an external auth-injection proxy and registers it in the docs navigation.

## Best Possible Solution

Publish the recipe after it clearly separates same-host/shared-namespace setups from multi-container deployments and uses container-reachable bridge, service DNS, or host-gateway addresses where appropriate.

## Reproduction Assessment

Yes. By inspection, a tenant container configured with `http://localhost:18080/anthropic` reaches its own loopback namespace, while the sample proxy listens on the Docker bridge address.

## Solution Assessment

No, not as written. The lightweight docs recipe is the right shape, but the container examples need reachable addressing or explicit scoping before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a reachable proxy address in container examples:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented config points `baseUrl` at `http://localhost:18080/anthropic`, but the proxy skeleton listens on the Docker bridge and identifies tenants by bridge source IP. In a separate tenant container, `localhost` is the tenant container itself, so readers following this setup will not route model calls through the auth proxy. Use a bridge, service DNS, or host-gateway address for multi-container deployments, or scope localhost to same-host/shared-network setups.
  - confidence: 0.9
