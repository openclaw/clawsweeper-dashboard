## Summary

Keep open: current main still contains the doctor/session repair behavior that can rewrite or pin Codex OAuth routes into an `openai/*` PI path, and the Discord symptom adds useful evidence beyond the broader model-routing reports. The right next step is to review the open maintainer fix at https://github.com/openclaw/openclaw/pull/78567 and then verify whether Discord/status still needs a clearer reply-failure surface.

## What This Changes

Prevent upgrade/doctor model-runtime migration and stale session/task state from leaving Discord connected while normal agent replies silently fail.

## Best Possible Solution

Land or revise the maintainer doctor/session-route fix that preserves `openai-codex/*` PI OAuth routes, then verify existing Discord sessions recover and expose model-turn failures clearly enough in status or channel diagnostics.

## Reproduction Assessment

Yes for the doctor/session-route failure from source inspection and related live reports, but not by local execution. Current main can rewrite `openai-codex/*` session routes to `openai/*` plus PI, and https://github.com/openclaw/openclaw/issues/78407 includes a Discord session-store reproduction of the same stale pin failure.

## Solution Assessment

Yes, the existing maintainer PR at https://github.com/openclaw/openclaw/pull/78567 appears to be the narrowest first fix for the migration regression. A separate Discord-visible/status-visible error follow-up may remain only if route repair does not make stalled reply failures diagnosable.
