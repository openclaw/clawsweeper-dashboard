## Summary

Keep open. Current main and the latest shipped release still reject explicit no-auth Gateway startup on non-loopback binds, while the closed implementation PR is unmerged and the open service-identity RFC shows the remaining work is a security/product decision, not conservative cleanup.

## What This Changes

The issue requests explicit Gateway no-auth support for trusted internal deployments where authentication is delegated to external ingress, API gateway, or service-mesh infrastructure.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth or service-identity contract, then align runtime validation, CLI preflight, security audits, install behavior, and docs around that contract.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs an explicit private-ingress or service-identity security contract before changing the Gateway auth guard.
