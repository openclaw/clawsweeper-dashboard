## Summary

Keep open. Current main and v2026.6.5 still reject explicit no-auth Gateway startup on non-loopback listeners, and allowing it is a security/product decision rather than conservative cleanup.

## What This Changes

Request to allow `gateway.auth.mode: "none"` for trusted internal non-loopback Gateway deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Keep the current guard until maintainers define an explicit private-ingress trust contract with dangerous-mode acknowledgement, startup warnings, docs, security-audit visibility, and focused runtime/CLI tests.

## Reproduction Assessment

Yes. Source and focused tests show current main rejects `gateway.auth.mode: "none"` on a LAN bind, and the same guard exists in v2026.6.5.

## Solution Assessment

No. Dropping all validations is not the safest maintainable fix; the better path is an explicit, auditable private-ingress no-auth contract approved by maintainers.
