## Summary

Keep open. Current main and v2026.6.5 still block explicit no-auth Gateway startup on non-loopback listeners, the directly linked implementation closed unmerged, and the related service-identity RFC treats this as a separate security/product decision.

## What This Changes

Request support for `gateway.auth.mode="none"` in trusted internal or private-ingress deployments without forcing a trusted-proxy workaround.

## Best Possible Solution

Define an explicit private-ingress no-auth contract with guardrails, warnings, security-audit visibility, docs, and focused runtime/CLI tests before permitting non-loopback unauthenticated startup.

## Reproduction Assessment

Yes. Current source and tests show that `gateway.auth.mode="none"` is accepted only for loopback startup and rejected on a LAN bind before runtime can serve the requested private-ingress shape.

## Solution Assessment

No. The proposed no-validation behavior is not the safest solution; the maintainable path is an explicit private-ingress trust contract that keeps accidental public exposure distinguishable from intentional delegation.

## Risks / Open Questions

- Allowing full Gateway no-auth on a non-loopback listener could expose WebSocket RPC, Control UI, and enabled HTTP operator surfaces to every peer that can reach the port if the external ingress boundary is absent or misconfigured.
- The related service-identity RFC helps internal callers but does not by itself decide the unauthenticated external/private-ingress user traffic contract requested here.
- Some HTTP API docs already mention private-ingress `auth.mode="none"`, while startup docs and code still block non-loopback no-auth; that mismatch should be resolved with the product decision.
