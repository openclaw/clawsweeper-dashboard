## Summary

Keep open. Current main still blocks the requested non-loopback `auth.mode: "none"` startup path, and the discussion/closed implementation PR show this is a deliberate Gateway auth-boundary product and security decision rather than stale cleanup.

## What This Changes

The issue requests allowing Gateway authentication to be intentionally disabled for trusted internal non-loopback deployments without requiring the trusted-proxy workaround.

## Best Possible Solution

Keep this open until Gateway/security owners either reaffirm loopback-only no-auth and align the HTTP docs, or approve an explicit private-ingress no-auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs explicit Gateway/security approval and likely a private-ingress acknowledgement or service-identity contract rather than silently relaxing the auth boundary.
