## Summary

Keep open. Current main still blocks the requested trusted-internal `auth.mode: "none"` deployment shape on non-loopback binds, and changing that guard is a security/product decision rather than conservative cleanup.

## What This Changes

Request support for intentionally disabling Gateway authentication in trusted internal deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Define the supported private-ingress no-auth trust model explicitly; if accepted, update runtime validation, CLI preflight, security audit severity/warnings, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows non-loopback `auth.mode: "none"` reaches both runtime and CLI no-auth rejection paths, and the runtime-config test asserts that rejection.

## Solution Assessment

Unclear. Allowing existing `auth.mode: "none"` through the non-loopback guard is mechanically narrow, but the best maintainable fix depends on maintainer/security approval for the trust-boundary change.
