## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback listeners; the linked implementation closed unmerged, and the broader service-identity RFC explicitly treats private-ingress no-auth as a separate security/product decision.

## What This Changes

Allow an explicitly configured no-auth Gateway to bind on non-loopback interfaces when authentication is delegated to trusted external infrastructure.

## Best Possible Solution

Define a narrowly scoped, explicitly dangerous private-ingress trust contract with startup warnings, security-audit visibility, Control UI origin enforcement, documentation, and focused fresh-install/runtime tests before permitting non-loopback unauthenticated startup.

## Reproduction Assessment

Yes. Current main has a deterministic source path and focused test showing that `gateway.auth.mode: "none"` is rejected on a LAN bind; a live deployment is unnecessary to establish the requested behavior is absent.

## Solution Assessment

No. Removing all validation is not the best solution because it erases an important safety distinction; an explicit private-ingress trust contract with strong guardrails is the safer maintainable direction.

## Risks / Open Questions

- Allowing full Gateway no-auth on a non-loopback listener could expose WebSocket RPC, Control UI, and enabled operator HTTP surfaces to every peer that can reach the port if the external ingress boundary is absent or misconfigured.
- The open service-identity RFC addresses authentication for internal callers, but it does not establish the trust contract for unauthenticated external user or service traffic requested here.
- Changing the existing guard without a distinct, auditable deployment contract could make an accidentally public bind indistinguishable from an intentionally protected private ingress.
