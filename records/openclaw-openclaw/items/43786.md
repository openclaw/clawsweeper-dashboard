## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback binds, and the closed implementation PR plus the active service-identity RFC show this remains a security/product decision rather than conservative cleanup.

## What This Changes

The issue asks for `gateway.auth.mode: "none"` to work for trusted internal or non-loopback deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Define an explicit private-ingress no-auth or service-identity contract across runtime validation, CLI preflight, audit checks, install behavior, and public docs before relaxing the Gateway auth guard.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection; no live gateway run was needed for this read-only review.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable path needs an explicit private-ingress or service-identity security contract before changing the guard.
