## Summary

Keep open: current main and v2026.6.10 still reject explicit no-auth Gateway startup on non-loopback/container binds, and accepting the request would intentionally relax a Gateway auth boundary that needs maintainer security and product approval.

## What This Changes

Request support for an explicit Gateway `auth.mode="none"` mode on trusted internal/non-loopback deployments so Docker, Kubernetes, or private ingress can delegate authentication to external infrastructure.

## Best Possible Solution

Maintainers should either keep the current token/password/trusted-proxy requirement or explicitly design a named dangerous private-ingress no-auth opt-in with CLI/runtime guards, security audit behavior, docs, and regression coverage.

## Reproduction Assessment

Yes. Source and tests show that explicit `auth.mode="none"` works at the auth layer but is rejected for non-loopback startup, including `bind: "lan"` and container auto-bind paths.

## Solution Assessment

No for the exact proposed shape. `auth mode = none without any validations` is not the safest maintainable solution; if accepted, it should be a deliberate private-ingress policy with warnings, audit behavior, docs, and coverage.
