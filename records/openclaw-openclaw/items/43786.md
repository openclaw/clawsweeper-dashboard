## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback listeners, the direct implementation pull request is closed unmerged, and the remaining work is a security/product decision rather than a safe automated cleanup.

## What This Changes

The issue requests an explicit `gateway.auth.mode = "none"` mode for trusted internal deployments so non-loopback Gateway use can delegate authentication to external infrastructure without a trusted-proxy workaround.

## Best Possible Solution

Define a maintainer-approved private-ingress no-auth contract with explicit dangerous acknowledgement, startup warnings, security-audit coverage, Control UI origin enforcement, docs, and focused runtime/CLI tests.

## Reproduction Assessment

Yes. Source inspection shows `bind: "lan"` with `gateway.auth.mode: "none"` still hits the non-loopback no-auth guard, and the runtime-config test explicitly expects that rejection.

## Solution Assessment

No. The proposed `auth mode = none without any validations` is not the best fix because it removes a safety boundary; the maintainable path needs an explicit, auditable private-ingress mode or acknowledgement with tests and docs.
