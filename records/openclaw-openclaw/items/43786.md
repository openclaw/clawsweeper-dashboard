## Summary

Keep open. Current main still rejects explicit no-auth Gateway startup on non-loopback binds, and the linked closed PR plus the active service-identity RFC show the remaining work is a Gateway security/product-boundary decision rather than cleanup.

## What This Changes

The issue requests support for `gateway.auth.mode: "none"` on trusted internal or Kubernetes-style non-loopback deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Gateway/security owners should decide the supported private-ingress contract, then update runtime validation, CLI preflight, audit behavior, docs, and focused tests together or close this in favor of the service-identity RFC if that becomes the chosen design.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and a regression test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable solution needs an explicit Gateway/security contract such as approved private-ingress no-auth or a service-identity design.
