## Summary

Keep open. Current main and the latest release still reject non-loopback `gateway.auth.mode: "none"`, the direct implementation PR was closed after security review, and the related RFC shows this needs an explicit Gateway auth/security product decision rather than cleanup closure.

## What This Changes

Request support for running the Gateway in trusted internal or private-ingress deployments with authentication intentionally disabled via `gateway.auth.mode: "none"`.

## Best Possible Solution

Either keep `none` loopback-only and align docs around token/password/trusted-proxy alternatives, or approve a deliberate private-ingress auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes. Source and test inspection on current main shows `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection; the latest release contains the same guard.

## Solution Assessment

Unclear. The requested no-validation `none` mode is straightforward for operators, but the closed PR and active RFC show the safer maintainable path needs explicit Gateway auth/security policy approval.
