## Summary

Keep open. Current main still rejects trusted-internal `auth.mode: "none"` deployments on non-loopback binds in both runtime config resolution and CLI preflight; the related RFC is relevant but broader, so this needs maintainer/security product judgment rather than cleanup closure.

## What This Changes

The issue requests allowing OpenClaw Gateway to run in explicit no-auth mode for trusted internal Docker/Kubernetes or gateway-managed deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Make an explicit maintainer/security decision on private-ingress no-auth support; if accepted, implement it deliberately across runtime validation, CLI preflight, security audit severity/warnings, docs, and regression tests.

## Reproduction Assessment

Yes for the current missing capability: source inspection shows `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection, and the focused runtime-config test asserts that rejection. This is a feature-policy reproduction, not a broken existing-contract bug.

## Solution Assessment

Unclear. Directly removing the guard is mechanically narrow, but the best maintainable solution depends on maintainer/security approval and should be aligned with the broader gateway-auth design discussion before changing the trust model.
