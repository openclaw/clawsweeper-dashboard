## Summary

Keep open: current main and the latest release still reject explicit no-auth Gateway startup on non-loopback binds, and changing that would be a deliberate core auth-boundary decision rather than conservative cleanup.

## What This Changes

The issue requests an explicit unauthenticated Gateway mode for trusted internal deployments, including Docker or Kubernetes-style non-loopback binds without the trusted-proxy workaround.

## Best Possible Solution

Maintainers should explicitly choose either the current shipped token/password/trusted-proxy requirement or a dangerous private-ingress no-auth opt-in with runtime, CLI, audit, docs, and regression coverage.

## Reproduction Assessment

Yes. Source and tests show that explicit `auth.mode="none"` works at the auth layer but is rejected for non-loopback startup, including container auto-bind and `bind: "lan"` paths.

## Solution Assessment

No for the exact proposed shape. `auth mode = none without any validations` is not the safest maintainable solution; if maintainers accept this direction, it should be an explicit private-ingress policy with strong warnings and coverage.
