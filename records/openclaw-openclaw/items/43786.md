## Summary

Keep open. Current main and v2026.5.12 still reject explicit `gateway.auth.mode: "none"` on non-loopback Gateway binds, the direct implementation PR was closed unmerged after a high-severity security review, and the linked RFC keeps the remaining work in Gateway auth/security product-decision territory.

## What This Changes

Request support for running the Gateway with authentication intentionally disabled in trusted internal or private-ingress deployments without requiring the trusted-proxy workaround.

## Best Possible Solution

Keep this open until Gateway/security owners either reaffirm loopback-only no-auth and align docs, or approve a deliberate private-ingress auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with explicit `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs explicit Gateway/security approval and likely a private-ingress acknowledgement or service-identity contract rather than silently relaxing the auth boundary.
