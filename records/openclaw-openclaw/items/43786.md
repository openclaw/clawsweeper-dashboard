## Summary

Keep open. Current main still blocks the requested trusted-internal deployment shape, and the linked implementation attempt plus the broader gateway-auth RFC show this is a security/product policy decision rather than a cleanup-close candidate.

## What This Changes

Allow OpenClaw Gateway to run with `gateway.auth.mode: "none"` on trusted internal or Kubernetes-style non-loopback deployments without a trusted-proxy workaround.

## Best Possible Solution

Make an explicit maintainer/security decision on private-ingress no-auth support; if accepted, update runtime validation, CLI preflight, security audit severity, docs, and focused regression tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection, and the focused runtime-config test asserts that rejection.

## Solution Assessment

Unclear. The direct approach from https://github.com/openclaw/openclaw/pull/43820 would implement the request mechanically, but review evidence shows the maintainable path needs an explicit auth/security policy decision first.
