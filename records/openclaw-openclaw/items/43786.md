## Summary

Keep open: current main and the latest release still reject the requested non-loopback `gateway.auth.mode="none"` startup shape, and the exact related PR was closed unmerged after security review, so this remains a maintainer product/security decision rather than cleanup-close material.

## What This Changes

The issue requests explicit no-auth gateway support for trusted internal deployments so operators do not need a trusted-proxy workaround.

## Best Possible Solution

Maintainers should decide the private-ingress no-auth policy first; if accepted, implement it as an explicit dangerous opt-in across runtime startup, CLI preflight, docs, security audit, and focused tests rather than removing validation broadly.

## Reproduction Assessment

Yes. From source, configure `gateway.auth.mode="none"` with a non-loopback bind such as container auto-bind or `--bind lan`, and current runtime/CLI preflight rejects it as unauthenticated.

## Solution Assessment

No, not as proposed. `auth.mode=none` without validations is not the best maintainable/security-safe shape; the safer path is an explicit maintainer-approved unsafe/private-ingress policy with matching runtime, CLI, docs, audit, and tests.
