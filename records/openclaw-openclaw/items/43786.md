## Summary

Keep open: current main and the latest release still reject explicit no-auth Gateway startup on non-loopback binds, and accepting this request would be a deliberate Gateway security/product decision rather than conservative cleanup.

## What This Changes

The issue requests an explicit unauthenticated Gateway mode for trusted internal deployments so operators do not need a trusted-proxy workaround.

## Best Possible Solution

Maintainers should explicitly choose either the current shipped token/password/trusted-proxy requirement or a dangerous private-ingress no-auth opt-in with runtime, CLI, security audit, docs, and regression coverage.

## Reproduction Assessment

Yes. Source and tests show that explicit `auth.mode="none"` works at the auth layer but is rejected for non-loopback startup, including container auto-bind and `bind: "lan"` paths.

## Solution Assessment

No for the exact proposed shape. `auth mode = none without any validations` is not the safest maintainable solution; if maintainers accept this direction, it should be an explicit private-ingress policy with warnings, audit behavior, and coverage.
