## Summary

Keep open: current main and the latest release still reject `gateway.auth.mode="none"` on non-loopback Gateway binds, and enabling that path would change a core auth boundary rather than fix a broken existing contract.

## What This Changes

Request to let trusted internal deployments run the Gateway with `auth.mode=none` on non-loopback/private-ingress binds without requiring trusted-proxy headers.

## Best Possible Solution

Have maintainers explicitly choose either the current shipped token/password/trusted-proxy requirement or a dangerous private-ingress no-auth opt-in with runtime, CLI, audit, docs, and regression coverage.

## Reproduction Assessment

Yes. Source and tests show that explicit `auth.mode="none"` works only on loopback and is rejected for `bind: "lan"`; I did not run a live Gateway because this was a read-only review.

## Solution Assessment

Unclear. The requested no-validation mode is not the safest shape; the maintainable path would be an explicit private-ingress security policy or a confirmed decision to keep the current shipped boundary.
