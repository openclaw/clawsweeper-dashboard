## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback listeners, the linked implementation pull request was closed unmerged, and the broader service-identity RFC was closed without settling this private-ingress trust boundary.

## What This Changes

The issue requests support for running a network-reachable Gateway with `gateway.auth.mode: "none"` so authentication can be delegated entirely to external internal-network infrastructure.

## Best Possible Solution

Define a narrow private-ingress no-auth contract with an explicit dangerous acknowledgement, startup warning, security-audit coverage, Control UI origin enforcement, docs, and focused runtime/CLI tests before permitting non-loopback unauthenticated startup.

## Reproduction Assessment

Yes. Source inspection shows `bind: "lan"` with `gateway.auth.mode: "none"` still hits the non-loopback no-auth guard, and the focused runtime-config test explicitly expects that rejection.

## Solution Assessment

No. `auth mode = none without any validations` is not the best fix because it removes a safety boundary; the maintainable path is an explicit, auditable private-ingress mode or acknowledgement with tests and docs.
