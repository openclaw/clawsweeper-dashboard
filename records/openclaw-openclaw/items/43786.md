## Summary

Keep open. Current main and the latest release still reject explicit Gateway no-auth on non-loopback binds, the direct implementation pull request was closed unmerged, and the related service-identity RFC treats this private-ingress no-auth request as a separate security/product decision.

## What This Changes

Request to let OpenClaw Gateway run with authentication intentionally disabled for trusted internal deployments instead of requiring a trusted-proxy workaround.

## Best Possible Solution

Define a narrow, explicitly dangerous private-ingress trust contract with startup warnings, security-audit visibility, Control UI origin enforcement, documentation, and focused fresh-install/runtime tests before allowing non-loopback no-auth startup.

## Reproduction Assessment

Yes. Current main has a deterministic source path and focused tests showing that gateway.auth.mode none is rejected on a LAN bind, so a live Kubernetes deployment is not needed to prove the requested behavior is absent.

## Solution Assessment

No. The requested no-validation shape is not the best solution because it removes an important safety boundary; the maintainable path is an explicit private-ingress trust mode with guardrails and security review.

## Risks / Open Questions

- Allowing auth.mode none on a non-loopback listener could expose Gateway WebSocket, Control UI, and enabled HTTP operator surfaces to any peer that can reach the port if the external ingress boundary is absent or misconfigured.
- The open service-identity RFC can reduce internal-caller coupling, but it does not by itself define a safe trust contract for unauthenticated external user or service traffic.
- Changing the guard without a distinct, auditable deployment contract would make accidentally public binds harder to distinguish from intentionally protected private-ingress deployments.
