## Summary

Keep open. Current main and v2026.5.12 still intentionally reject explicit `gateway.auth.mode: "none"` on non-loopback Gateway binds, the direct implementation PR was closed after security review, and the related RFC keeps this as a Gateway auth-boundary product/security decision rather than cleanup.

## What This Changes

Request support for intentionally disabling Gateway authentication on trusted internal or non-loopback deployments without requiring the trusted-proxy setup.

## Best Possible Solution

Keep this open until Gateway/security owners either reaffirm the loopback-only no-auth policy and align docs, or approve a deliberate private-ingress auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection; v2026.5.12 contains the same guard.

## Solution Assessment

Unclear. The requested no-validation mode is operationally simple, but the maintainable fix needs explicit Gateway/security approval and likely a private-ingress acknowledgement or service-identity contract instead of silently relaxing the auth boundary.
