## Summary

Keep open. Current main and the latest shipped release still reject `gateway.auth.mode: "none"` on non-loopback Gateway binds, while the closed direct PR and the active gateway-auth RFC show this needs an explicit auth/security product decision rather than cleanup closure.

## What This Changes

Request support for intentionally unauthenticated Gateway deployments on trusted internal or non-loopback networks without the current trusted-proxy workaround.

## Best Possible Solution

Make an explicit Gateway auth/security decision; if accepted, implement a deliberate private-ingress or local-service-auth contract across runtime, CLI, audit, docs, and tests, otherwise keep `none` loopback-only and document supported shared-secret or trusted-proxy alternatives.

## Reproduction Assessment

Yes. Source and test inspection on current main shows `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection, and `v2026.5.12` contains the same guard.

## Solution Assessment

Unclear. The requested no-validation `none` mode is the simplest operator experience, but the closed implementation attempt and active RFC show the safer maintainable path needs explicit auth/security policy approval and likely a deliberate private-ingress or service-identity design.
