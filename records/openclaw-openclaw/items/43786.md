## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback binds; the attempted implementation PR was closed unmerged, and the remaining path is a Gateway auth-boundary product/security decision.

## What This Changes

The issue requests a supported Gateway mode that intentionally disables authentication for trusted internal deployments without requiring trusted-proxy header workarounds.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth or service-identity contract, then align runtime validation, CLI preflight, security audits, install behavior, and public docs around that contract.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs an explicit private-ingress or service-identity security contract before changing the Gateway auth guard.
