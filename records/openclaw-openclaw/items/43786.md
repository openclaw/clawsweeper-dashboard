## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback listeners, the linked implementation PR is closed unmerged, and the remaining decision is security/product policy rather than conservative cleanup.

## What This Changes

The issue requests support for disabling OpenClaw-side Gateway authentication in trusted internal deployments where auth is delegated to external ingress, gateway, or service-mesh infrastructure.

## Best Possible Solution

Define a maintainer-approved private-ingress no-auth contract with explicit operator acknowledgement, startup warnings, security-audit coverage, Control UI origin enforcement, docs alignment, and focused runtime/CLI tests.

## Reproduction Assessment

Yes. Source inspection shows `bind: "lan"` with `gateway.auth.mode: "none"` still hits the non-loopback no-auth guard, and the runtime-config test explicitly expects that rejection.

## Solution Assessment

No. The requested `auth mode = none without any validations` removes a safety boundary; the maintainable path needs an explicit, auditable private-ingress contract with tests and docs.
