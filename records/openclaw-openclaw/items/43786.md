## Summary

Keep open: current main and the latest release still reject the requested non-loopback `gateway.auth.mode="none"` startup shape, and the direct unmerged fix raised a real unauthenticated-gateway exposure concern that needs maintainer product/security direction.

## What This Changes

The issue requests an explicit no-auth gateway mode for trusted internal deployments so OpenClaw can run behind external auth infrastructure without a trusted-proxy workaround.

## Best Possible Solution

Define an explicit private-ingress no-auth policy, likely behind a dangerous acknowledgement flag/config with runtime, CLI, security-audit, docs, and tests, or document that non-loopback no-auth remains unsupported.

## Reproduction Assessment

Yes from source inspection: current main rejects `gateway.bind="lan"` with explicit `gateway.auth.mode="none"` in both runtime config and CLI startup preflight. I did not run tests because this review is read-only, but the existing regression test encodes the rejection.

## Solution Assessment

No: the proposed `none without any validations` is not the best safe solution because the Gateway exposes privileged operator surfaces. A maintainer-approved explicit private-ingress/dangerous opt-in with aligned docs and audit coverage would be safer.
