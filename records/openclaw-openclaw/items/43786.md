## Summary

Keep open. Current main and v2026.5.12 still reject explicit `gateway.auth.mode: "none"` on non-loopback Gateway binds, the direct implementation PR was closed after security review, and the related RFC keeps this as a Gateway auth-boundary product/security decision.

## What This Changes

Request to allow `gateway.auth.mode: "none"` for trusted internal non-loopback deployments without requiring trusted-proxy setup.

## Best Possible Solution

Keep this open until Gateway/security owners either reaffirm the loopback-only no-auth policy and align docs, or approve a deliberate private-ingress/service-identity auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with `gateway.auth.mode: "none"` in runtime validation and has regression coverage asserting that rejection; v2026.5.12 contains the same guard.

## Solution Assessment

Unclear. The requested no-validation mode is operationally simple, but the maintainable fix needs explicit Gateway/security approval and likely a private-ingress acknowledgement or service-identity contract instead of silently relaxing the auth boundary.
