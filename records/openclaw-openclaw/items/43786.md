## Summary

Keep open. Current main still deliberately rejects explicit no-auth Gateway startup on non-loopback listeners, the direct implementation pull request closed unmerged, and the related service-identity RFC leaves this as a security/product trust-boundary decision rather than an implemented cleanup item.

## What This Changes

The issue requests support for running OpenClaw Gateway in trusted internal deployments with authentication intentionally disabled, avoiding trusted-proxy header workarounds.

## Best Possible Solution

Define a narrow, explicit private-ingress no-auth trust contract with startup warnings, security-audit visibility, Control UI origin enforcement, and docs/tests, or intentionally keep the current fail-closed guard and route deployments through trusted-proxy or a future service-identity design.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: configure a LAN or container auto-bind with `gateway.auth.mode: "none"` and current startup validation rejects it; the focused runtime-config tests assert that rejection path.

## Solution Assessment

Is this the best way to solve the issue? No for the proposed shape as written; dropping validation entirely is less maintainable than an explicit private-ingress trust contract or a maintainer decision to keep using trusted-proxy/service-identity boundaries.
