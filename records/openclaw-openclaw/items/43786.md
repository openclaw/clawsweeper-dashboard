## Summary

Keep open. Current main and latest release still reject explicit no-auth Gateway startup on non-loopback binds, and the closed implementation PR plus the active service-identity RFC show this remains a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests an explicit OpenClaw Gateway mode that permits intentionally unauthenticated operation for trusted internal deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth contract with runtime, CLI, install, audit, and docs behavior, or close this in favor of https://github.com/openclaw/openclaw/issues/69066 if service identity becomes the chosen design.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable path needs an explicit private-ingress or service-identity security contract before changing the guard.
