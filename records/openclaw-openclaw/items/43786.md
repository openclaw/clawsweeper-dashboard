## Summary

Keep open. Current main and the latest shipped release still reject `gateway.auth.mode: "none"` on non-loopback binds, the direct implementation attempt was closed unmerged after security concerns, and the related gateway-auth RFC shows this needs an explicit auth/security product decision rather than cleanup closure.

## What This Changes

The issue requests support for intentionally disabling Gateway authentication on trusted internal non-loopback deployments such as Docker or Kubernetes behind external auth infrastructure.

## Best Possible Solution

Make an explicit Gateway auth/security decision; if accepted, add a deliberate private-ingress no-auth contract with runtime and CLI validation, audits, docs, and focused tests, otherwise keep `none` loopback-only and document supported trusted-proxy or shared-secret alternatives.

## Reproduction Assessment

Yes. Source and test evidence on current main show `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection, and the latest release contains the same behavior.

## Solution Assessment

Unclear. The requested no-validation `none` mode is the simplest operator experience, but the closed implementation attempt shows the safer maintainable path needs explicit auth/security policy approval and likely a deliberate private-ingress opt-in.
