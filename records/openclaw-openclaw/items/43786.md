## Summary

Keep open. Current main still blocks explicit no-auth Gateway starts on non-loopback binds, and the remaining request intentionally changes a security boundary that needs maintainer/security product approval rather than an automated cleanup close.

## What This Changes

The issue requests support for running the Gateway with `gateway.auth.mode: "none"` in trusted internal deployments without the current non-loopback auth validation.

## Best Possible Solution

Keep the strict current guard until gateway/security owners approve either a deliberate private-ingress no-auth contract with explicit warnings/audit/docs or a separate service-identity design that avoids weakening user auth.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection.

## Solution Assessment

No, not as proposed. A no-validation `none` mode is operationally simple, but the maintainable path needs explicit gateway/security approval and likely a private-ingress acknowledgement or service-identity contract.
