## Summary

Keep open. Current main still blocks the requested explicit no-auth Gateway startup on non-loopback binds, while the closed implementation PR and open service-identity RFC show this is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests support for intentionally running the Gateway with `gateway.auth.mode: "none"` in trusted internal deployments without the current non-loopback startup validation.

## Best Possible Solution

Gateway/security owners should decide an explicit private-ingress no-auth or service-identity contract that aligns runtime validation, CLI preflight, audits, install behavior, and docs before relaxing the guard.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs an explicit private-ingress or service-identity security contract before changing the Gateway auth guard.
