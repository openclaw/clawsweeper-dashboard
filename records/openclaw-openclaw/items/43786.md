## Summary

Keep open. Current main and v2026.5.22 still reject explicit no-auth Gateway startup on non-loopback binds, the linked implementation PR is closed and unmerged, and the related service-identity RFC shows this is a security/product decision rather than cleanup.

## What This Changes

Request support for running the Gateway in trusted internal or container networks with `gateway.auth.mode: "none"` instead of requiring trusted-proxy or shared-secret auth.

## Best Possible Solution

Define an explicit private-ingress no-auth or service-identity contract with opt-in guardrails, then align runtime validation, CLI preflight, security audit, docs, and focused tests around that contract.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs an explicit private-ingress or service-identity security contract before changing the Gateway auth guard.
