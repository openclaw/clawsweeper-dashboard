## Summary

Keep open. Current main and the latest release still reject explicit `gateway.auth.mode="none"` on non-loopback Gateway binds, the linked implementation PR is closed unmerged, and the remaining path is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests an explicit no-auth Gateway mode for trusted internal deployments so Kubernetes or gateway-fronted setups do not need a trusted-proxy workaround.

## Best Possible Solution

Define a maintainer-approved private-ingress no-auth contract with explicit operator acknowledgement, startup and security-audit warnings, origin/scope behavior, docs alignment, and focused runtime plus CLI tests; do not implement the literal no-validation form.

## Reproduction Assessment

Yes. Source inspection shows `bind: "lan"` with `gateway.auth.mode: "none"` still hits the non-loopback no-auth guard, and the runtime-config test explicitly expects that rejection.

## Solution Assessment

No. The requested `auth mode = none without any validations` removes a safety boundary; the maintainable path needs an explicit, auditable private-ingress contract or a scoped service/principal auth design.
