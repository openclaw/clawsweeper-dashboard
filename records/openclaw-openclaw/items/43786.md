## Summary

Keep open. Current main and v2026.5.20 still reject the requested explicit no-auth Gateway startup on non-loopback binds, while the closed implementation PR, the open service-identity RFC, and the open Tailscale no-auth security report show this needs a maintainer security/product decision rather than conservative cleanup.

## What This Changes

Request support for intentionally running the Gateway with `gateway.auth.mode: "none"` in trusted internal deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth or service-identity contract, then align runtime validation, CLI preflight, security audit behavior, docs, and focused tests before relaxing the guard.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable path needs an explicit private-ingress or service-identity security contract before changing the Gateway auth guard.
