## Summary

Keep open. Current main and the latest release still reject explicit `gateway.auth.mode="none"` on non-loopback Gateway binds, the linked implementation PR is closed unmerged after security feedback, and the remaining work needs a maintainer-approved security/product contract rather than conservative cleanup.

## What This Changes

The issue requests an explicit `gateway.auth.mode="none"` mode for trusted internal or Kubernetes-style deployments where authentication is delegated to external infrastructure instead of OpenClaw trusted-proxy headers.

## Best Possible Solution

Define a maintainer-approved private-ingress no-auth contract with explicit operator acknowledgement, startup/security-audit warnings, origin and scope behavior, docs alignment, and focused runtime plus CLI tests; do not implement the literal no-validation form.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with `gateway.auth.mode: "none"` in runtime config and CLI preflight, and the runtime-config test explicitly expects that rejection.

## Solution Assessment

No. The literal request for `auth mode = none without any validations` is not the best solution because it removes a Gateway security boundary; the maintainable path needs an explicit private-ingress contract or scoped service/principal auth design.
