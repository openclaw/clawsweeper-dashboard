## Summary

Keep open. Current main and the latest release still reject explicit Gateway no-auth on non-loopback binds, the directly linked implementation pull request closed unmerged, and the related service-identity RFC treats this as a complementary security/product decision rather than a replacement.

## What This Changes

Allow Gateway `gateway.auth.mode: "none"` to run in trusted internal non-loopback deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Define an explicit private-ingress trust contract or decide not to support it, while keeping default non-loopback auth required and adding warnings, audit visibility, docs, and focused tests for any approved opt-in.

## Reproduction Assessment

Yes. Current main has a deterministic source path and focused test showing that `gateway.auth.mode: "none"` is rejected on a LAN bind; I did not run tests because this was a read-only review.

## Solution Assessment

No. `auth.mode=none` without validations is not the safest maintainable solution; a narrowly scoped private-ingress opt-in with explicit security guardrails is the better path if maintainers choose to support it.

## Risks / Open Questions

- Allowing full Gateway no-auth on a non-loopback listener could expose WebSocket RPC, Control UI, and operator HTTP endpoints to any peer that can reach the port if the external ingress boundary is absent or misconfigured.
- The open service-identity RFC helps internal callers but does not settle whether unauthenticated external/private-ingress user traffic should be supported.
- Some HTTP endpoint docs already mention private-ingress `auth.mode="none"`, while runtime/config docs still block non-loopback startup; maintainers should reconcile docs with the product decision.
