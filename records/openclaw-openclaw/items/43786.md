## Summary

Keep open. Current main and the latest release still reject explicit no-auth Gateway startup on non-loopback binds, while the closed implementation PR and active service-identity RFC show the remaining work is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests support for intentionally running the Gateway with `gateway.auth.mode: "none"` in trusted internal/private-ingress deployments without needing a trusted-proxy workaround.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth contract covering runtime validation, CLI preflight, audit behavior, install/startup messaging, and public docs, or close this in favor of the service-identity RFC if that becomes the chosen design.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable path needs an explicit private-ingress or service-identity security contract before changing the guard.

- The open RFC at https://github.com/openclaw/openclaw/issues/69066 and the security issue at https://github.com/openclaw/openclaw/issues/50630 need coordination before changing the Gateway auth boundary.
