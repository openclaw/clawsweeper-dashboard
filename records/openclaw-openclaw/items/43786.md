## Summary

Keep open. Current main and v2026.6.5 still reject the requested explicit no-auth Gateway startup on non-loopback listeners; the linked implementation PR is closed unmerged, and the broader service-identity RFC treats private-ingress no-auth as a separate security/product decision.

## What This Changes

Request support for intentionally unauthenticated Gateway operation in trusted internal/private-ingress deployments without requiring a trusted-proxy workaround.

## Best Possible Solution

Approve or reject a narrow private-ingress no-auth contract with explicit dangerous opt-in, startup warnings, security-audit visibility, Control UI origin enforcement, documentation, and focused runtime/CLI tests.

## Reproduction Assessment

Yes. Source inspection shows the current startup and CLI guards reject non-loopback `auth.mode: "none"`, and the focused runtime-config test expects that rejection.

## Solution Assessment

No. The requested `none` mode without validations is not the best solution because it removes an important safety distinction; a guarded private-ingress trust contract is the safer maintainable direction.
