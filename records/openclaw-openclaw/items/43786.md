## Summary

Keep open. Current main still rejects `auth.mode: "none"` on non-loopback Gateway binds, and the requested relaxation is a security/product policy decision rather than conservative cleanup; #69066 is related design work but does not fully supersede this specific deployment request.

## What This Changes

Request support for running the Gateway behind trusted private ingress with `gateway.auth.mode: "none"` on non-loopback binds.

## Best Possible Solution

Decide the trusted-private-ingress no-auth policy in the Gateway auth design, then if approved change runtime validation, CLI preflight, security audit, docs, and regression tests as one coordinated contract.

## Reproduction Assessment

Yes for the current missing capability: source inspection shows `bind: "lan"` with `auth.mode: "none"` reaches the non-loopback no-auth rejection, and the focused runtime-config test preserves that rejection. I did not execute tests because this review was read-only.

## Solution Assessment

Unclear. Removing the guard is mechanically narrow, but the maintainable solution needs explicit security/product approval and should align the startup contract with #69066 and the private-ingress HTTP docs.
