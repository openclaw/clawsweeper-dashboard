## Summary

Keep open. Current main still rejects explicit no-auth Gateway startup on non-loopback binds, while the linked implementation PR closed unmerged and the active service-identity RFC leaves this as a Gateway security/product decision.

## What This Changes

Request to allow `gateway.auth.mode: "none"` for trusted internal or private-ingress deployments without the current non-loopback auth validation.

## Best Possible Solution

Gateway/security owners should define an explicit private-ingress no-auth contract, including runtime and CLI validation, audit/docs behavior, and focused tests, or close this in favor of the service-identity RFC if that becomes the chosen design.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind: "lan"` with explicit `gateway.auth.mode: "none"`, and the focused runtime-config test asserts that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the safer maintainable path needs an explicit private-ingress or service-identity security contract before changing the guard.
