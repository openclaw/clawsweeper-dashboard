## Summary

Keep open. Current main and the latest release still reject explicit `gateway.auth.mode: "none"` on non-loopback Gateway binds, the direct implementation PR was closed after security review, and the related RFC/security threads show this needs an explicit Gateway auth policy decision rather than cleanup closure.

## What This Changes

The issue requests support for running the Gateway with explicit no-auth mode in trusted internal or private-ingress deployments without requiring trusted-proxy auth.

## Best Possible Solution

Keep the issue open until Gateway/security owners either keep no-auth loopback-only and align docs, or approve a deliberate private-ingress auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection; v2026.5.12 contains the same guard.

## Solution Assessment

Unclear. The requested no-validation no-auth mode is operationally simple, but the safer maintainable path needs explicit Gateway/security approval and may require a private-ingress acknowledgement or service-identity design.
