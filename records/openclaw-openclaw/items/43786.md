## Summary

Keep open. Current main and the latest release still block explicit `gateway.auth.mode: "none"` on non-loopback binds, while the closed related PR that tried to allow it was not merged and drew a concrete security objection; the remaining decision is a gateway product/security policy choice, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to support `gateway.auth.mode: "none"` for trusted internal non-loopback deployments so Docker/Kubernetes/ingress setups do not need trusted-proxy header workarounds.

## Best Possible Solution

Make an explicit gateway security/product decision: either keep `none` loopback-only and document the supported ingress path, or add a deliberately named private-ingress/dangerous opt-in with runtime, CLI, doctor, audit, docs, and tests coordinated with the service-identity RFC.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `gateway.bind: "lan"` with `gateway.auth.mode: "none"`, and current runtime config plus CLI preflight reject the non-loopback no-auth startup.

## Solution Assessment

No. The proposed `none without any validations` path is not the best standalone fix because `none` accepts all gateway requests; a safer solution needs an explicit maintainer-approved trust-boundary contract or service-identity design.
