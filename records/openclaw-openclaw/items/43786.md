## Summary

Keep open. Current main still blocks the requested trusted-internal no-auth deployment at both runtime config resolution and CLI startup preflight, and changing that would relax a gateway authentication safety invariant rather than fix a broken existing contract.

## What This Changes

The issue requests allowing `gateway.auth.mode="none"` for trusted internal deployments without requiring trusted-proxy headers or an extra reverse proxy.

## Best Possible Solution

Make an explicit maintainer/security decision on private-ingress no-auth support; if accepted, implement it deliberately across runtime validation, CLI preflight, audit warnings, and docs while preserving Control UI origin, Tailscale/Funnel, and exposed-HTTP guardrails.

## Reproduction Assessment

Yes for the current missing capability: source inspection shows `resolveGatewayRuntimeConfig({ gateway: { bind: "lan", auth: { mode: "none" } } })` still reaches the non-loopback no-auth rejection, and the focused test asserts that rejection. This is a feature-policy reproduction rather than a broken existing-contract bug.

## Solution Assessment

Unclear. The requested exception is mechanically narrow, but the best maintainable solution depends on maintainer/security approval and should be coordinated with the broader gateway-auth RFC rather than applied as a silent guard removal.

## Risks / Open Questions

- Allowing `auth.mode="none"` on non-loopback binds would intentionally expose Gateway HTTP/WebSocket surfaces without shared-secret or trusted-proxy auth if an operator misconfigures ingress or network reachability.
- The docs, CLI/runtime guardrails, and security-audit messaging need to move together; changing only one path would create a dangerous split-brain deployment story.
- The open gateway-auth RFC (#69066) may choose a service-identity design that changes how this request should be implemented.
