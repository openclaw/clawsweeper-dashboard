- `P2`: This is a real Gateway auth improvement request with limited blast radius, but it needs security/product judgment rather than urgent regression handling.
- `impact:security`: The request would intentionally relax the Gateway authentication boundary for non-loopback control-plane exposure.
- `impact:auth-provider`: The requested behavior changes Gateway auth mode handling and provider-facing credential expectations for trusted internal deployments.

## Summary

Keep open. Current main and v2026.5.18 still reject explicit `gateway.auth.mode: "none"` on non-loopback Gateway binds, the direct implementation PR was closed unmerged after security review, and the linked RFC keeps the remaining work in Gateway auth/security product-decision territory.

## What This Changes

Request support for intentionally disabling Gateway auth in trusted internal deployments without adding a trusted-proxy workaround.

## Best Possible Solution

Keep this open until Gateway/security owners either reaffirm loopback-only no-auth and align docs, or approve a deliberate private-ingress auth contract with runtime, CLI, audit, docs, and tests updated together.

## Reproduction Assessment

Yes, at source level. Current main rejects `bind: "lan"` with explicit `gateway.auth.mode: "none"` in runtime validation and has a regression test asserting that rejection.

## Solution Assessment

No, not as proposed. A no-validation mode is operationally simple, but the maintainable fix needs explicit Gateway/security approval and likely a private-ingress acknowledgement or service-identity contract rather than silently relaxing the auth boundary.
