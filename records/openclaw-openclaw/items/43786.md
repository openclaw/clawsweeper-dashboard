## Summary

Keep open. Current main still blocks the requested setup in both the CLI preflight and runtime config path, and the focused runtime test explicitly asserts that `bind: "lan"` with `auth.mode: "none"` rejects. The linked exact fix PR was closed unmerged, while the broader gateway-auth RFC remains open design work rather than a shipped replacement.

## What This Changes

The issue requests support for explicitly running the OpenClaw gateway with authentication disabled in trusted internal deployments, without requiring trusted-proxy workarounds.

## Best Possible Solution

Make an explicit maintainer decision on whether OpenClaw should support a private-ingress no-auth mode. If accepted, implement it as a deliberate opt-in across the CLI preflight and runtime validator, keep Tailscale/Funnel and Control UI origin guardrails intact, update security audit/dangerous-config warnings and docs, and leave broader internal service-identity work to #69066.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `resolveGatewayRuntimeConfig({ cfg: { gateway: { bind: "lan", auth: { mode: "none" }, controlUi: { allowedOrigins: ["https://control.example.com"] } } }, port: 18789 })`, which current tests assert should throw `refusing to bind gateway`.

## Solution Assessment

Unclear. The requested `auth.mode=none` exception directly addresses the trusted internal deployment use case, but it relaxes a gateway safety invariant and should be decided alongside security warnings and the broader gateway-auth RFC rather than applied silently.

## Risks / Open Questions

- Allowing `auth.mode="none"` on non-loopback binds would intentionally expose Gateway HTTP/WebSocket surfaces without shared-secret or trusted-proxy auth if an operator misconfigures ingress.
- Current docs are inconsistent: architecture text mentions private-ingress none, while runtime validation, tests, web docs, and security audit language still treat `auth.mode="none"` as loopback/trusted-local only.
- The open #69066 RFC may change the desired shape by separating internal service identity from user auth, so this should not be treated as a purely mechanical cleanup without maintainer/security review.
