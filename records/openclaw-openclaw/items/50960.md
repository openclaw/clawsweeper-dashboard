## Summary

Keep open: the PR targets a real current-main bug, but the proposed diff admits sensitive skill env keys too broadly, leaves duplicate sensitive-key behavior unsafe, and still lacks after-fix real behavior proof.

## What This Changes

The PR allows sensitive-pattern keys configured in `skills.entries.*.env` through skill env sanitization, tracks skill-injected env state for eligibility checks, and adds regression tests.

## Best Possible Solution

Revise skill env admission so explicit sensitive keys are allowed only for the eligible or snapshot skill set for the active run, preserve hard host-dangerous blocks, reject or isolate conflicting sensitive duplicates, and prove the real exec subprocess path.

## Reproduction Assessment

Yes. Source inspection shows current main stages `skills.entries.*.env` keys and then blocks sensitive-pattern names through the sanitizer, while the PR regressions are source-reproducible through env application before eligibility filtering and first-wins active env state.

## Solution Assessment

No. Allowing explicitly configured sensitive keys is the right owner-boundary fix, but this diff admits them before eligible-skill scoping and leaves duplicate sensitive-key policy unsafe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to eligible skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop allowlists every configured `skills.entries.*.env` key before the run is narrowed to eligible skills. In the no-snapshot path, disabled or otherwise unprompted skills can inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses can inherit those secrets even though the skill is not part of the run.
  - confidence: 0.86
- **[P2] Reject conflicting duplicate sensitive env keys:** `src/agents/skills/env-overrides.ts:163-164`
  - body: Once generic sensitive names are allowlisted, two configured skills can provide different values for the same key such as `GITHUB_TOKEN`. The active env state keeps the first value and increments a count, so later subprocesses silently inherit the first credential; reject conflicting sensitive duplicates or isolate env values by skill/tool scope.
  - confidence: 0.78
