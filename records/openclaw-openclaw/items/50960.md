## Summary

Keep open: the PR targets a source-backed auth regression, but the current branch still lacks real behavior proof and broadens sensitive skill env admission in a way that can expose or misroute secrets before the skill execution boundary is settled.

## What This Changes

The PR allowlists sensitive-pattern keys from `skills.entries.*.env`, moves active skill env tracking into a shared module, adjusts skill eligibility checks, and adds focused unit tests.

## Best Possible Solution

Land a scoped fix that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets, rejects or isolates duplicate sensitive-key conflicts, and includes real behavior proof.

## Reproduction Assessment

Yes, source-backed. Current main stages `skills.entries.gog.env.GOG_KEYRING_PASSWORD` but only allowlists `primaryEnv` and `requires.env`, while the sanitizer blocks `_PASSWORD` names and host exec inherits only what reaches `process.env`.

## Solution Assessment

No. Allowlisting explicit configured keys is directionally right, but this branch admits secrets too broadly and leaves duplicate sensitive-key conflicts; scoped admission plus generic exec stripping is the safer solution.
