## Summary

Keep open: the PR targets a real, source-backed skills env regression, but the branch still needs contributor real behavior proof and security-sensitive scoping fixes before it is mergeable.

## What This Changes

The branch allows sensitive-looking `skills.entries.*.env` keys through skill env sanitization, moves active skill env tracking into a shared module, adjusts skill eligibility checks, and adds regression tests.

## Best Possible Solution

Revise the fix so explicit sensitive skill env values reach only the intended skill-owned command path, hard host-dangerous blocks remain intact, duplicate sensitive conflicts are rejected or isolated, and redacted real exec inheritance proof is added.

## Reproduction Assessment

Yes, source-backed. Current main stages `skills.entries.gog.env.GOG_KEYRING_PASSWORD` but only allowlists `primaryEnv` and `requires.env`; the bundled `gog` skill declares only a binary requirement, so the sanitizer blocks `_PASSWORD` names.

## Solution Assessment

No. Allowlisting explicit configured keys is directionally right for https://github.com/openclaw/openclaw/issues/31583, but this branch admits secrets too broadly and leaves duplicate sensitive conflicts; scoped admission plus conflict handling is the safer fix.
