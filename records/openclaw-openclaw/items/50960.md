## Summary

Keep open: current main still blocks the reported sensitive skill env names, but this PR broadens secret env admission before eligible-skill scoping, leaves duplicate sensitive-key conflicts unresolved, and lacks contributor real-behavior proof.

## What This Changes

The PR allows sensitive-pattern keys from `skills.entries.*.env`, extracts active skill env state, updates skill eligibility checks, and adds regression tests.

## Best Possible Solution

Revise skill env admission so explicit sensitive keys are allowed only for the eligible or snapshot skill set that can run, preserve hard host-dangerous blocks, reject or isolate conflicting sensitive duplicates, and prove the real exec subprocess path.

## Reproduction Assessment

Yes. Source inspection shows current main blocks `GOG_KEYRING_PASSWORD` through the sensitive-name sanitizer, and the PR regressions are source-reproducible through env application before eligible prompt filtering plus first-wins active env state.

## Solution Assessment

No. Allowing explicitly configured sensitive keys is the right direction, but this patch admits them too broadly and does not define safe behavior for duplicate sensitive keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to eligible skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop admits every configured sensitive-pattern key before the run is narrowed to the eligible skills prompt. In the no-snapshot path, disabled or otherwise unprompted skills can inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses can inherit those secrets even though the skill is not part of the run.
  - confidence: 0.86
- **[P2] Reject conflicting duplicate sensitive env keys:** `src/agents/skills/env-overrides.ts:163-164`
  - body: After generic sensitive names are allowed, two configured skills can provide different values for the same key such as `GITHUB_TOKEN`. The active env state keeps the first value and increments a count, so later subprocesses silently inherit the first credential; reject conflicting sensitive duplicates or isolate env values by skill/tool scope.
  - confidence: 0.78
