## Summary

Keep this PR open, but do not merge it as-is: it targets a real current-main bug, yet the patch broadens sensitive env admission before selected-skill scoping, keeps duplicate sensitive-key first-wins behavior, and lacks after-fix real exec proof.

## What This Changes

The PR changes skill env override handling so sensitive-pattern variables configured under `skills.entries.*.env` can pass sanitization, adds active skill env tracking for eligibility, and adds skills regression tests.

## Best Possible Solution

Revise the fix so explicit sensitive skill env keys are admitted only through the intended active skill execution path, hard host-dangerous blocks remain intact, duplicate sensitive conflicts are rejected or isolated, and the real exec inheritance path is proven.

## Reproduction Assessment

Yes, source-backed. Current main stages `GOG_KEYRING_PASSWORD` from `skills.entries.*.env` and then blocks it through the sensitive-pattern sanitizer because configured env keys are not allowlisted; I did not run a live `gog` command in this read-only review.

## Solution Assessment

No. Allowlisting explicit configured keys is directionally right, but this implementation admits keys before selected-skill scoping and keeps duplicate sensitive-key first-wins behavior; the safer fix needs scoped admission plus conflict handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env admission to selected skills:** `src/agents/skills/env-overrides.ts:164`
  - body: This loop allowlists every staged `skills.entries.*.env` key before the run is narrowed to prompted or eligible skills. In no-snapshot runs, disabled, blocked, or unprompted skills can inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses inherit that environment.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env conflicts:** `src/agents/skills/env-overrides.ts:164`
  - body: If two skill configs stage the same sensitive key with different values, `acquireActiveSkillEnvKey()` keeps the first active value and increments a count. After this allowlist change, later subprocesses can silently inherit the wrong credential instead of rejecting or isolating the conflict.
  - confidence: 0.78
