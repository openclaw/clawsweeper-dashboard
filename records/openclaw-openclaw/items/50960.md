## Summary

Keep open: the PR targets a documented bug, but the current patch still has blocking secret-scoping regressions and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR allowlists sensitive-pattern keys from `skills.entries.*.env`, adds active skill env tracking to skill eligibility, extracts that state, and adds focused skills tests.

## Best Possible Solution

Revise the fix so explicit sensitive skill env keys are admitted only for the active eligible skill path, hard host-dangerous blocks remain intact, duplicate sensitive conflicts are rejected or isolated, and the real exec inheritance path is proven.

## Reproduction Assessment

Yes. Source inspection against current main shows configured `GOG_KEYRING_PASSWORD` is staged and then blocked by the sensitive-pattern sanitizer because configured env keys are not allowlisted; host `exec` inherits `process.env` after admission.

## Solution Assessment

No. Allowlisting explicit configured keys is the right boundary, but this implementation does it for every staged skill env key before selected-skill scoping and keeps first-wins duplicate sensitive-key behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env admission to selected skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop allowlists every staged `skills.entries.*.env` key before the run is narrowed to prompted or eligible skills. In no-snapshot runs, disabled, blocked, or unprompted skills can inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses inherit that environment.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env conflicts:** `src/agents/skills/env-overrides.ts:163-164`
  - body: If two skill configs stage the same sensitive key with different values, `acquireActiveSkillEnvKey()` keeps the first active value and increments a count. After this allowlist change, later subprocesses can silently inherit the wrong credential instead of rejecting or isolating the conflict.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch broadens secret env admission into host subprocesses before selected-skill scoping and without duplicate-key conflict handling.

Concerns:

- **[high] Disabled or unprompted skill secrets can reach exec:** `src/agents/skills/env-overrides.ts:163`
  - body: Adding every configured skill env key to the sensitive allowlist can expose `*_TOKEN` or `*_PASSWORD` values from skills that are disabled, blocked, or not included in the prompt to host exec subprocesses.
  - confidence: 0.86
- **[medium] Conflicting sensitive keys can share the wrong credential:** `src/agents/skills/env-overrides.ts:163`
  - body: The process-wide active env map stores one value per key, so widening sensitive env admission lets two skills with the same sensitive env name silently share the first active credential.
  - confidence: 0.78
