## Summary

Keep open: the PR targets a source-backed regression that still exists on current main and the latest release, but the branch is not merge-ready because it broadens sensitive skill env admission into process-wide exec environments without selected-skill scoping, duplicate-key handling, or real behavior proof.

## What This Changes

The PR allowlists all keys configured under `skills.entries.*.env` through the skill env sanitizer, extracts active skill env state into a shared module, adjusts skill eligibility checks, and adds focused unit tests.

## Best Possible Solution

Land or replace this with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic exec strips unrelated active skill secrets and duplicate sensitive keys are rejected or isolated.

No. The PR's broad sensitive-key allowlist fixes the usability symptom but is not the safest permanent boundary; the maintainable path needs selected-skill scoping, generic exec stripping, and duplicate sensitive-key handling.

- **[P1] Gate sensitive env admission to selected skills:** `src/agents/skills/env-overrides.ts:163-165`
  - body: This loop allowlists every configured sensitive env key before the run has narrowed to the selected or prompt-eligible skill. In the no-snapshot path, skill env overrides are applied to all loaded entries, so disabled, blocked, or unprompted skill secrets can become process-wide and be inherited by generic `exec` children; scope admission to the intended skill-owned execution path or strip unrelated active skill keys before host exec.
  - confidence: 0.91
- **[P2] Reject duplicate sensitive env conflicts:** `src/agents/skills/env-overrides.ts:163-165`

If two configured skills use the same sensitive key with different values, later subprocesses can silently receive the first skill's credential, so duplicate sensitive keys need rejection or per-skill isolation before merge.
  - confidence: 0.86
