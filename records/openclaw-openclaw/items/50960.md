## Summary

Keep open: the reported skill env bug is valid, but this PR is not merge-ready because it broadens sensitive env admission before skill scoping, lacks real behavior proof, and conflicts with the related generic exec secret-leak boundary.

## What This Changes

The branch allowlists sensitive-pattern keys from configured skill env overrides, moves active skill env state into a shared module, adjusts skill eligibility checks, and adds regression tests.

## Best Possible Solution

Land a scoped fix that lets intended skill-owned subprocesses receive explicit configured env values while generic exec strips unrelated active skill secrets and duplicate sensitive env conflicts are rejected or isolated, with redacted real exec proof.

## Reproduction Assessment

Yes, source-backed. Current main stages `skills.entries.gog.env.GOG_KEYRING_PASSWORD` but only allowlists `primaryEnv` and `requires.env`; the bundled `gog` skill does not declare that passphrase key, and the sanitizer blocks `_PASSWORD` names before host exec can inherit them.

## Solution Assessment

No as submitted. Allowlisting explicit configured keys addresses the usability bug directionally, but the branch admits secrets too broadly and leaves duplicate sensitive-key conflicts; scoped admission plus generic exec stripping is the safer solution.
