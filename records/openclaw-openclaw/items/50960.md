## Summary

Keep open: the PR targets a real source-backed skill env regression, but the current patch broadens sensitive env admission into process-wide host subprocess environments before skill selection is settled and still lacks contributor real behavior proof.

## What This Changes

This PR allowlists sensitive-pattern keys from `skills.entries.*.env`, moves active skill env tracking into shared state, adjusts skill eligibility checks, and adds focused unit tests.

## Best Possible Solution

Land a scoped fix that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets, rejects or isolates duplicate sensitive-key conflicts, and includes real behavior proof.

## Reproduction Assessment

Yes, source-backed. Current main stages `skills.entries.*.env` keys but only allowlists metadata-declared sensitive keys, and the PR-introduced exposure path is clear from applying overrides before prompt filtering plus host exec inheriting `process.env`.

## Solution Assessment

No. Allowlisting explicit configured keys is directionally useful, but this branch admits secrets too broadly and leaves duplicate sensitive-key conflicts; scoped admission plus generic exec stripping is the safer solution.

- **[P1] Gate sensitive env admission to selected skills:** `src/agents/skills/env-overrides.ts:163-165`
  - body: The added allowlist loop admits every staged `skills.entries.*.env` key before no-snapshot runs narrow the catalog to eligible or prompted skills.   - confidence: 0.86

  - confidence: 0.78
