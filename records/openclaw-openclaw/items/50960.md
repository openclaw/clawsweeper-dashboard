## Summary

Keep open: current main still has the linked skills env regression, but this PR should not merge as-is because it widens sensitive env admission before eligible-skill scoping and leaves duplicate sensitive-key conflicts with first-wins process-wide behavior.

## What This Changes

This PR allows sensitive-pattern keys from `skills.entries.*.env` through skill env sanitization, extracts active skill env state, adjusts skill eligibility around injected env keys, and adds regression tests.

## Best Possible Solution

Revise env injection to operate on the same eligible or snapshot skill set that can be invoked, allow explicit sensitive keys only for that set, keep hard dangerous-host blocks, and reject or isolate conflicting duplicate sensitive keys with targeted regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main blocks the reported sensitive env names, and the PR regressions are source-reproducible through the no-snapshot run path plus first-wins active env state.

## Solution Assessment

No. Allowing explicit sensitive keys is the right direction, but this implementation admits them before eligible-skill scoping and without a conflict policy for duplicate sensitive env names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to eligible skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop admits every configured sensitive-pattern key before the run is narrowed to the eligible skills prompt. In the no-snapshot path, disabled or otherwise unprompted skills can now inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and exec subprocesses can inherit those secrets even though the skill is not part of the run.
  - confidence: 0.86
- **[P2] Reject conflicting duplicate sensitive env keys:** `src/agents/skills/env-overrides.ts:163-164`
  - body: After generic sensitive names are allowed, two configured skills can provide different values for the same key such as `GITHUB_TOKEN`. The existing active env state keeps the first value and increments a count, so later subprocesses silently inherit the first credential; reject conflicting sensitive duplicates or isolate env values by skill/tool scope.
  - confidence: 0.78
