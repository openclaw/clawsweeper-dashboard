## Summary

Keep open for maintainer review: current main still has the linked skill env bug, but this PR broadens sensitive env admission before selected-skill scoping, keeps unsafe duplicate-key behavior, and lacks after-fix real exec proof.

## What This Changes

The PR allowlists sensitive-pattern keys from `skills.entries.*.env`, adds active skill env tracking, changes skill eligibility checks, and adds focused skills tests.

## Best Possible Solution

Revise skill env admission so explicit sensitive keys are allowed only for the selected or eligible skill execution path, preserve hard host-dangerous blocks, reject or isolate conflicting sensitive duplicates, and prove the real exec subprocess path.

## Reproduction Assessment

Yes. Source inspection shows current main stages `skills.entries.*.env` values and then blocks `_PASSWORD`/`_TOKEN` names through the sanitizer because configured env keys are not allowlisted; I did not run a live exec path in this read-only review.

## Solution Assessment

No. Allowing explicit configured sensitive keys is the right boundary, but this implementation does it before selected-skill scoping and keeps first-wins duplicate sensitive keys; a safer fix should scope admission and define conflict handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to selected skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop allowlists every staged `skills.entries.*.env` key before the run is narrowed to the prompted or eligible skills. In normal no-snapshot runs, unprompted or disabled skills can inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses inherit that environment.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env conflicts:** `src/agents/skills/env-overrides.ts:163-164`
  - body: If two skill configs stage the same sensitive key with different values, `acquireActiveSkillEnvKey()` keeps the first active value and increments a count. After this allowlist change, later subprocesses can silently inherit the wrong credential instead of rejecting or isolating the conflict.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff broadens sensitive env admission into `process.env` for host exec paths without selected-skill scoping or duplicate-key conflict policy.

Concerns:

- **[high] Disabled or unprompted skill secrets can reach exec:** `src/agents/skills/env-overrides.ts:163`
  - body: By adding every configured skill env key to the sensitive allowlist while env overrides can run before selected-skill filtering, the patch can expose `*_TOKEN` or `*_PASSWORD` values from skills that are disabled, blocked, or not included in the prompt to host exec subprocesses.
  - confidence: 0.86
- **[medium] Conflicting sensitive keys can share the wrong credential:** `src/agents/skills/env-overrides.ts:163`
  - body: The process-wide active env map stores one value per key, so widening sensitive env admission lets two skills with the same sensitive env name silently share the first active credential.
  - confidence: 0.78
