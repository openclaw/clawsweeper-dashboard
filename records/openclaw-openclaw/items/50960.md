## Summary

Keep open. The PR targets a real regression from #31583, and current main still blocks configured keys such as `GOG_KEYRING_PASSWORD`; however, this patch broadens sensitive env admission before the run is narrowed to eligible skills and leaves duplicate sensitive-key conflicts on the existing process-wide first-wins state.

## What This Changes

The PR allows sensitive-pattern keys from `skills.entries.*.env` through skill env sanitization, adds active skill env state plumbing for eligibility isolation, and adds skills env regression tests.

## Best Possible Solution

Revise the PR so user-configured sensitive env keys are allowed only for the same eligible skill set used by the run or snapshot, hard dangerous host env names remain blocked, and duplicate sensitive env keys with conflicting values are rejected or isolated before any exec subprocess can inherit them.

## Reproduction Assessment

Yes. The source path is enough to reproduce the original bug by configuring `GOG_KEYRING_PASSWORD` in `skills.entries.<skill>.env` and observing current main block it through `sanitizeEnvVars`; the PR regressions can be reproduced with a disabled/unprompted skill carrying a sensitive env key and with two skills configuring different values for the same sensitive key.

## Solution Assessment

No. Allowing explicitly configured sensitive keys is the right direction, but this implementation admits them at the sanitizer layer before eligible-skill scoping and without a duplicate-key policy; the safer fix is to align env injection with the eligible/snapshot skill set and reject or isolate conflicts.

The existing active env state keeps the first value and increments a count, so later subprocesses silently inherit the first credential; reject conflicting sensitive duplicates or isolate env values by skill/tool scope.
  - confidence: 0.78
