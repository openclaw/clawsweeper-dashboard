## Summary

Keep open. The linked bug is source-reproducible and the PR moves in a useful direction, but the current patch is not merge-ready because it lacks real behavior proof and broadens sensitive skill env admission before selected-skill scoping, with duplicate credential conflict risk.

## What This Changes

The branch allowlists sensitive-pattern `skills.entries.*.env` keys, moves active skill env tracking into a shared module, adjusts skill eligibility checks, and adds regression tests.

## Best Possible Solution

Land a scoped fix that combines explicit sensitive env admission with generic exec secret stripping, duplicate conflict handling, focused regression tests, and redacted real exec proof.

## Reproduction Assessment

Yes, source-backed. Current main stages `skills.entries.gog.env.GOG_KEYRING_PASSWORD` but only allowlists `primaryEnv` and `requires.env`, while the bundled `gog` skill declares no passphrase env metadata and the sanitizer blocks `_PASSWORD` names.

## Solution Assessment

No. Allowlisting explicit configured keys is directionally right, but this branch admits secrets too broadly and leaves duplicate sensitive-key conflicts; scoped admission plus generic exec stripping is the safer solution.
