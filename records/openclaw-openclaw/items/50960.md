## Summary

Keep open: this PR targets a real source-reproducible skill env regression, but the current patch still has security-sensitive env scoping and duplicate-secret handling defects, and contributor real behavior proof is missing.

## What This Changes

The PR allowlists sensitive-pattern keys from `skills.entries.*.env`, adds active-skill env tracking for eligibility checks, and adds focused skills test coverage.

## Best Possible Solution

Revise skill env admission so explicit sensitive keys are allowed only for active eligible or snapshot skills, preserve hard host-dangerous blocks, reject or isolate conflicting sensitive duplicates, and prove the real exec subprocess path.

## Reproduction Assessment

Yes. Source inspection shows current main stages `skills.entries.*.env` keys and then blocks sensitive-pattern names through the sanitizer; the PR regressions are also source-reproducible because env admission happens before eligibility filtering and active env state is first-wins.

## Solution Assessment

No. Allowing explicitly configured sensitive keys is the right ownership-boundary fix, but this implementation widens process-wide secret admission before eligible-skill scoping and leaves duplicate sensitive-key conflict handling unsafe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to selected skills:** `src/agents/skills/env-overrides.ts:163-164`
  - body: This loop allowlists every configured `skills.entries.*.env` key before the run is narrowed to eligible or prompted skills. In no-snapshot runs, disabled or unprompted skills can still inject `*_TOKEN` or `*_PASSWORD` values into `process.env`, and host exec subprocesses inherit that environment.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env conflicts:** `src/agents/skills/env-overrides.ts:163-164`
  - body: After generic sensitive names are allowlisted, two configured skills can provide different values for the same key such as `GITHUB_TOKEN`. The active env state keeps the first value and increments a count, so later subprocesses silently inherit the wrong credential.
  - confidence: 0.78

## Security Review

Status: needs_attention

  - body: By adding every configured skill env key to the sensitive allowlist while env overrides are applied before eligible-skill filtering, the patch can expose `*_TOKEN` or `*_PASSWORD` values from skills that are disabled, blocked, or not included in the prompt to host exec subprocesses.
