## Summary

Keep this PR open, but do not merge it as-is. It addresses a real current-main gap for sensitive-pattern keys in `skills.entries.*.env`, and the circular import noted by Greptile has been fixed, but the patch still broadens host secret env admission before skill eligibility is scoped and leaves duplicate sensitive-key conflicts with first-wins process-wide behavior.

## What This Changes

The PR lets env var names configured under `skills.entries.*.env` bypass sensitive-name sanitization, adds active skill env state for eligibility isolation, and adds skills env regression tests.

## Best Possible Solution

Revise the PR so configured sensitive-pattern env keys are admitted only for the skill set active in the run or snapshot, hard dangerous host env names remain blocked, conflicting duplicate sensitive keys are rejected or isolated, and regression tests cover undeclared configured secrets, disabled or unselected skills, duplicate keys, and exec subprocess inheritance.

## Reproduction Assessment

Yes. On current main, configure a skill env key such as `GOG_KEYRING_PASSWORD` that is not declared in skill metadata; the shared sanitizer blocks the `_PASSWORD` pattern and the skill env allowlist does not re-allow configured env keys. The PR findings can be reproduced with one disabled/unselected configured skill or with two skills configuring different values for the same sensitive key.

## Solution Assessment

No. Allowing user-configured sensitive keys is the right direction, but this implementation widens the sanitizer bypass before eligible-skill scoping and keeps process-wide first-wins duplicate handling. The safer fix is to align env injection with the same active/eligible skill set used for the run and define conflict behavior for sensitive duplicates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to selected skills:** `src/agents/skills/env-overrides.ts:172`
  - body: `applySkillEnvOverrides()` is still called on the loaded skill entries before `resolveSkillsPromptForRun()` filters the run to eligible skills. Adding every configured `pendingOverrides` key to `allowedSensitiveKeys` lets `*_TOKEN` and `*_PASSWORD` values from disabled, allowlist-blocked, or otherwise unselected skills enter `process.env` and be inherited by exec subprocesses. Filter to the active/eligible skill set before admitting these secrets.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env key conflicts:** `src/agents/skills/env-overrides.ts:172`
  - body: After this change, generic sensitive keys such as `GITHUB_TOKEN` or `CUSTOM_SERVICE_TOKEN` can be configured by multiple skills, but `acquireActiveSkillEnvKey()` keeps the first active value and only increments a count. A later skill with a different credential silently inherits the first credential for the run. Reject conflicting duplicate sensitive keys or isolate env values by selected skill/tool scope.
  - confidence: 0.78
