## Summary

Keep this PR open, but it is not merge-ready. Current main does not implement BlueBubbles tapback approval resolution, and the branch still has approval-boundary and real-webhook-path issues that match the existing Aisle/Omar review concerns.

## What This Changes

This PR adds BlueBubbles thumbs-up/thumbs-down tapback handling for exec approvals, adds monitor tests, and exports `callGateway` from `openclaw/plugin-sdk/gateway-runtime`.

## Best Possible Solution

Implement BlueBubbles reaction approvals through tracked native approval state: store the delivered approval prompt GUID with the pending approval id and requester/approver identity, resolve only exact supported reactions on that tracked prompt, handle both BlueBubbles reaction webhook and tapback-message shapes, and use the existing approval-scoped gateway resolver or another narrow SDK seam.

## Reproduction Assessment

Yes. The review issues are reproducible from source inspection: current main routes real BlueBubbles `message-reaction`/`reaction` payloads to `processReaction`, while the PR only adds a resolver in the message path; the spoofing path follows directly from resolving IDs out of untrusted `replyToBody` text.

## Solution Assessment

No. The UX goal is valid, but the proposed implementation is not the narrowest maintainable fix because it parses free-form reply text and broadens the gateway SDK instead of binding to tracked pending approval messages and using the approval-scoped helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bind approvals to tracked message GUIDs:** `extensions/bluebubbles/src/monitor-processing.ts:884-888`
  - body: This resolves approvals by regex-parsing `replyToBody`, which is quoted/replied text rather than proof that the user reacted to an OpenClaw-sent approval prompt. An allowlisted participant can send or quote text containing a live `Full id:` and then tapback to resolve it. Track the delivered BlueBubbles message GUID for each pending approval and look up by `replyToId` instead of parsing free-form text.
  - confidence: 0.9
- **[P2] Handle real reaction webhook events:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: This resolver only runs in the normalized message path. Current BlueBubbles webhook handling sends `message-reaction`/`reaction` payloads with `associatedMessageType` to `processReaction`, which this PR leaves as a plain system-event enqueue path, so installations that emit real reaction webhooks still will not resolve approvals.
  - confidence: 0.86
- **[P1] Use the approval-scoped gateway helper:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: Exporting `callGateway` promotes a broad helper that can call arbitrary gateway methods and accepts caller-supplied scopes while resolving host credentials. This PR only needs approval resolution, and current main already has `openclaw/plugin-sdk/approval-gateway-runtime` backed by an approvals-scoped client. Use that narrow helper or a dedicated approval API instead.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR changes an exec approval path and public SDK gateway surface, and both need security hardening before merge.

Concerns:

- **[high] Approval resolution trusts spoofable reply text:** `extensions/bluebubbles/src/monitor-processing.ts:884`
  - body: The proposed handler derives the approval id from `replyToBody` without proving that the reacted-to message is a tracked OpenClaw approval prompt, so an allowlisted actor could craft or quote approval-shaped text and resolve a pending exec approval.
  - confidence: 0.9

  - body: Adding `callGateway` to `plugin-sdk/gateway-runtime` expands the public plugin surface to a helper that resolves host gateway credentials and can request arbitrary methods/scopes; the approval use case should stay on an approvals-scoped helper.
