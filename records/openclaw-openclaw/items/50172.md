## Summary

Keep this PR open: the BlueBubbles approval-reaction UX is useful, but the current patch is not mergeable because it resolves security-sensitive exec approvals from free-form replied-to text and broadens the plugin SDK gateway surface instead of using the existing approval-scoped/native approval path.

## What This Changes

This PR adds BlueBubbles thumbs-up/thumbs-down tapback handling for exec approval prompts, monitor tests, and a `callGateway` export from the plugin SDK gateway runtime.

## Best Possible Solution

Implement BlueBubbles approval reactions through the channel approval capability/native runtime: store delivered approval message GUIDs with pending approval entries, resolve only exact added thumbs-up/thumbs-down reactions from authorized approvers on tracked prompts, and use the existing approval-scoped gateway resolver.

## Reproduction Assessment

Yes. The review issues are source-reproducible from the PR diff and current main: approval ids are parsed from `replyToBody`, native BlueBubbles reaction webhooks route through `processReaction`, and the patch does not bind reactions to tracked pending approval prompt GUIDs.

## Solution Assessment

No. The requested UX should be kept, but this patch is not the best solution because it bypasses the current approvalCapability/nativeRuntime design and uses free-form text plus a broad gateway SDK export where a GUID-bound, approval-scoped path is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind approvals to tracked prompt GUIDs:** `extensions/bluebubbles/src/monitor-processing.ts:884-888`
  - body: This resolves approvals by regex-parsing `replyToBody`, which is quoted or replied text rather than proof that the user reacted to an OpenClaw-sent approval prompt. Track the delivered BlueBubbles message GUID for each pending approval and look up by that reaction target instead of parsing free-form text.
  - confidence: 0.9
- **[P2] Resolve real BlueBubbles reaction webhook events:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: The resolver only runs in the normalized message path. Current BlueBubbles webhook handling routes `message-reaction` and `reaction` payloads with associated message metadata to `processReaction`, so installations that emit native reaction webhooks still will not resolve approvals.
  - confidence: 0.86
- **[P2] Use the approval-scoped gateway resolver:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: This PR only needs approval resolution, but it promotes `callGateway` through the plugin SDK gateway runtime instead of using the existing `openclaw/plugin-sdk/approval-gateway-runtime` helper backed by an approval-scoped gateway client. Keep this path on the narrow resolver or a dedicated approval API.
  - confidence: 0.82
- **[P2] Handle approval reactions before mention gating:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: In mention-gated BlueBubbles groups, tapback texts do not contain a mention or control command, so the message path can return before this resolver runs. Approval reactions need their own approver/prompt binding path rather than depending on normal message mention gating.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch changes exec approval resolution and public SDK gateway access.

Concerns:

- **[high] Approval resolution trusts spoofable reply text:** `extensions/bluebubbles/src/monitor-processing.ts:884`
  - body: The proposed handler derives the approval id from `replyToBody` without proving that the reacted-to message is a tracked OpenClaw approval prompt, so an allowed actor could craft or quote approval-shaped text and resolve a pending exec approval.
  - confidence: 0.9
- **[medium] Broad gateway RPC helper is exposed to plugins:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: Adding `callGateway` to `plugin-sdk/gateway-runtime` exposes a generic helper that resolves host gateway credentials and can request methods beyond the approval use case; this PR should use the approval-scoped helper instead.
  - confidence: 0.82
