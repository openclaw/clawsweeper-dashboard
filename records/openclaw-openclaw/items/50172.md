## Summary

Keep open: current main does not implement BlueBubbles tapback approvals, but this PR is not ready to merge because it resolves approvals from reply text and broad gateway access rather than a tracked prompt, requester-bound approval path.

## What This Changes

The PR adds BlueBubbles thumbs-up/thumbs-down tapback handling for exec approvals, monitor tests, and a public plugin SDK export of `callGateway`.

## Best Possible Solution

Implement tapback approvals through the BlueBubbles/native approval runtime with delivered prompt GUID tracking, requester/approver checks, native reaction webhook handling, the approval-scoped gateway resolver, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. The review issues are source-reproducible from the PR diff and current main: approval ids are parsed from `replyToBody`, native BlueBubbles reaction payloads route to `processReaction`, and current pending approval state is not bound to delivered BlueBubbles prompt GUIDs.

## Solution Assessment

No. The requested UX is useful, but the proposed implementation should be reworked around tracked approval prompt GUIDs and the existing approval-scoped gateway client instead of parsing free-form reply text and exporting generic gateway RPC access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind approval tapbacks to tracked prompt GUIDs:** `extensions/bluebubbles/src/monitor-processing.ts:884-888`
  - body: This resolves approvals by regex-parsing `replyToBody`, which can be quoted or user-controlled text rather than proof that the user reacted to an OpenClaw-sent approval prompt. Track the delivered BlueBubbles message GUID for each pending approval and look up the reaction target instead of trusting prompt-shaped text.
  - confidence: 0.92
- **[P2] Handle native BlueBubbles reaction webhooks:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: The resolver only runs in the normalized message path, but current webhook handling routes `message-reaction` and `reaction` payloads with associated message metadata to `processReaction`. Installations that emit native reaction webhooks will still enqueue a plain reaction event and never resolve the approval.
  - confidence: 0.88
- **[P2] Authorize tapbacks through approval-scoped identity:** `extensions/bluebubbles/src/monitor-processing.ts:895`
  - body: `commandAuthorized` is the channel/control-command gate, not the approval requester or normal `/approve` authorization path. This can make tapback approvals disagree with text approvals and allow another allowed group participant to resolve a request unless the pending approval identity is checked.
  - confidence: 0.84
- **[P2] Keep gateway calls approval scoped:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: This PR only needs approval resolution, but exporting `callGateway` from `plugin-sdk/gateway-runtime` promotes a generic host gateway RPC helper to plugins. Use `openclaw/plugin-sdk/approval-gateway-runtime` or a dedicated approval API so the public seam stays limited to approvals.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: This is a user-facing BlueBubbles feature, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to include a changelog entry before landing.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch changes exec approval resolution and public plugin SDK gateway access.

Concerns:

- **[high] Approval resolution trusts spoofable reply text:** `extensions/bluebubbles/src/monitor-processing.ts:884`
  - body: The handler derives the approval id from `replyToBody` without proving the reacted-to message is a tracked OpenClaw approval prompt, so an allowed actor could craft or quote approval-shaped text and resolve a pending exec approval.
  - confidence: 0.92
- **[medium] Approval authorization is not requester-bound:** `extensions/bluebubbles/src/monitor-processing.ts:895`
  - body: The patch checks the channel command gate rather than the pending approval requester/approver identity, which weakens the approval boundary in allowed group or shared-account contexts.
  - confidence: 0.84

  - body: Adding `callGateway` to the public gateway runtime subpath exposes a generic helper that resolves host gateway credentials and can invoke methods beyond the approval use case.
