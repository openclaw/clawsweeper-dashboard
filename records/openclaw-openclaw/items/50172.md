## Summary

Keep open. The UX goal is valid, but the PR still changes a security-sensitive exec approval path and public plugin SDK surface in a way that needs hardening before merge: it resolves approvals from free-form replied-to text, misses BlueBubbles native reaction webhook events, and adds a broad gateway RPC export instead of using the approval-scoped helper already on current main.

## What This Changes

The PR adds BlueBubbles thumbs-up/thumbs-down tapback handling for exec approval prompts, monitor tests, and a `callGateway` export from `openclaw/plugin-sdk/gateway-runtime`.

## Best Possible Solution

Implement BlueBubbles reaction approvals through tracked native approval state: store the delivered approval prompt GUID with the pending approval id and approver/requester identity, resolve only exact added thumbs-up/thumbs-down reactions on that tracked message, handle both native reaction webhook and tapback-message shapes, and keep the SDK call on the existing approval-scoped resolver.

## Reproduction Assessment

Yes. The review issues are reproducible from source inspection: a BlueBubbles `message-reaction` payload with `associatedMessageType` routes to `processReaction`, while the PR only hooks `processMessage`; the spoofing path follows from resolving IDs out of `replyToBody` text without proving the reacted-to message is a tracked OpenClaw approval prompt.

## Solution Assessment

No. The requested UX is worth keeping, but this PR is not the narrowest maintainable solution because it trusts free-form reply text and adds a broad gateway helper where current main already has an approvals-scoped gateway resolver.

  - body: This resolver only runs in the normalized message path. Current BlueBubbles webhook handling sends `message-reaction` and `reaction` payloads with `associatedMessageType` to `processReaction`, which this PR leaves as a plain system-event enqueue path, so installations that emit real reaction webhooks still will not resolve approvals.
  - confidence: 0.86
- **[P2] Use the approval-scoped gateway resolver:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: This PR only needs approval resolution, but it promotes `callGateway` through the plugin SDK gateway runtime instead of using current main's `openclaw/plugin-sdk/approval-gateway-runtime` helper backed by an approval-scoped gateway client. Use the narrow resolver or a dedicated approval API for this path.
