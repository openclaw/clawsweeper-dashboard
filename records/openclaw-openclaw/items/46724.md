## Summary

Keep open. Current main still exposes an isolated single-Gateway control plane with only operator/node roles, no peer-Gateway method/config surface, and the related open work covers narrower cross-Gateway session routing rather than full federation, HA, peer identity, and routing semantics.

## What This Changes

The issue requests Gateway-to-Gateway federation for mutual Gateway connections, a peer Gateway role, and routed messages across Gateway instances.

## Best Possible Solution

Keep this as the broad Gateway federation design tracker until maintainers define peer identity, trust, routing, config, compatibility, HA, and how it coordinates with https://github.com/openclaw/openclaw/issues/43605, https://github.com/openclaw/openclaw/pull/43656, and https://github.com/openclaw/openclaw/issues/25883.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature and architecture request. Source and docs inspection gives high confidence that current main lacks the requested Gateway peer role, peer method family, and federation config surface.

## Solution Assessment

Unclear. The proposed role="gateway" and gw.send direction is plausible, but the maintainable path needs reviewed protocol, security, config, HA, and session-ownership design before implementation.
