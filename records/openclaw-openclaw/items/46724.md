## Label Justifications

- `P2`: This is a normal-priority architecture feature for multi-Gateway deployments, not a current regression in the default single-Gateway path.
- `impact:session-state`: Gateway federation would move session ownership, transcript visibility, routing, and state coordination across Gateway boundaries.

- `impact:message-loss`: The requested capability is about routing messages between isolated Gateway instances, where delivery semantics are currently undefined.

## Summary

Keep open. Current main still implements a single-Gateway control plane with operator/node roles and isolated multi-Gateway profiles; the narrower cross-Gateway session PR is open and blocked, so this broader federation/HA request still needs maintainer architecture and security review.

## What This Changes

The issue requests Gateway-to-Gateway federation with peer connections, a gateway role, routed messages, and HA/multi-instance deployment support.

## Best Possible Solution

Keep this as the broad Gateway federation design tracker until maintainers define peer identity, trust, routing, config, compatibility, HA semantics, and how it coordinates with the narrower cross-Gateway sessions work.

## Security Review

Status: needs_attention

Summary: Gateway federation is security-sensitive because it would introduce a new cross-Gateway control-plane trust boundary.
