## Summary

Keep open. Current main still has only operator/node Gateway roles, no gateway-peer RPC family, and no peer/federation config surface; related #43605 and #43656 are open and narrower, so they do not supersede this broader federation/HA request.

## What This Changes

Request support for Gateway-to-Gateway federation, including peer Gateway auth/role support and routed messages between isolated Gateway instances.

## Best Possible Solution

Keep this as a core Gateway protocol design item until maintainers define peer identity, authorization, routing semantics, HA scope, config/discovery, compatibility, and docs; #43656 can inform a narrower session-routing subset.

## Reproduction Assessment

Not applicable as a runtime bug. Source and docs provide high-confidence evidence that current main lacks a `role: "gateway"`, gateway-peer RPCs, and federation config.

## Solution Assessment

Unclear. The proposed `role: "gateway"` plus `gw.send` direction is plausible, but the maintainable solution needs a reviewed core protocol and security design rather than a one-off routing patch.
