## Summary

Keep open. Current main still lacks a Gateway peer role, federation RPC, peer-routing config, and HA semantics; the related cross-Gateway messaging tracker and PR are narrower, while the horizontal-scaling issue is a broader job-orchestration discussion rather than a duplicate of Gateway federation.

## What This Changes

Request Gateway-to-Gateway federation so separate Gateways can authenticate as peers, route messages, and support HA or horizontal multi-instance deployments.

## Best Possible Solution

Keep this as a core Gateway protocol design tracker until maintainers define peer identity, trust, routing, config, compatibility, and HA boundaries.

## Reproduction Assessment

Not applicable as a runtime bug. Source and docs inspection gives high confidence that current main accepts only operator/node roles, rejects role="gateway", has no gw.send or gateway.peers surface, and documents isolated multi-Gateway operation.

## Solution Assessment

Unclear. The requested role="gateway" plus gw.send direction is plausible, but the maintainable fix needs reviewed core protocol and security design coordinated with https://github.com/openclaw/openclaw/issues/43605 and https://github.com/openclaw/openclaw/pull/43656.
