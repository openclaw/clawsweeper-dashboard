## Summary

Keep open. Current main still has a single-Gateway control-plane model with operator/node roles only, no peer-Gateway routing method or federation config surface, and the related open work is narrower session-tool routing that still needs product/security review.

## What This Changes

Request Gateway-to-Gateway federation with mutual peer connections, a gateway role, and message routing for HA, scaling, and cross-instance collaboration.

## Best Possible Solution

Keep this as the broad Gateway federation design tracker until maintainers define peer identity, trust, routing, config, compatibility, HA, and coordination with the narrower cross-Gateway session work.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature and architecture request. Source and docs inspection gives high confidence that current main lacks the requested Gateway peer role, peer method family, and federation config surface.

## Solution Assessment

Unclear. The proposed role="gateway" and gw.send direction is plausible, but the maintainable path needs reviewed protocol, security, config, HA, and session-ownership design before implementation.
