## Label Justifications

- `P2`: This is a normal-priority architecture feature for multi-Gateway deployments, not a current regression in the default single-Gateway path.
- `impact:session-state`: Gateway federation would move session ownership, transcript visibility, routing, and state coordination across Gateway boundaries.
- `impact:security`: The request introduces a new trust boundary where one Gateway may authenticate to and trigger work on another Gateway.
- `impact:message-loss`: The requested capability is about routing messages between isolated Gateway instances where delivery semantics are currently undefined.

## Summary

Keep open. Current main still documents and implements a single-Gateway control plane with operator/node roles, local session tools, no peer Gateway auth role, no federation method family, and no peer registry; the related narrower cross-Gateway session PR remains open and blocked, so this broader federation request still needs maintainer architecture and security review.

## What This Changes

Request support for Gateway-to-Gateway federation, including mutual Gateway connections, a Gateway auth role, and routed cross-Gateway messages.

## Best Possible Solution

Keep this as the broad Gateway federation design tracker until maintainers define peer identity, trust, routing, config, compatibility, HA semantics, and coordination with the narrower cross-Gateway sessions work.

## Solution Assessment

Unclear. The proposed role="gateway" and gw.send direction is plausible, but the maintainable path needs reviewed protocol, security, config, routing, HA, and session-ownership design before implementation.

Status: needs_attention

Summary: Gateway federation is security-sensitive because it would introduce a new cross-Gateway control-plane trust boundary.
