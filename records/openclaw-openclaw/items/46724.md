## Label Justifications

- `P2`: This is a normal-priority architecture feature for multi-Gateway deployments, not a current regression in the default single-Gateway path.
- `impact:session-state`: Gateway federation would move session resolution, transcript visibility, and state ownership across Gateway boundaries.
- `impact:security`: The request introduces a new trust boundary where one Gateway may authenticate to and trigger work on another Gateway.
- `impact:message-loss`: The requested capability is about routing messages between isolated Gateway instances with currently undefined delivery semantics.

## Summary

Keep open. Current main and the latest release still document and implement a single Gateway control plane with `operator`/`node` roles, local session-tool routing, no peer-Gateway role or RPC family, and no peer registry; the related sessions PR is narrower, open, conflicting, and blocked on proof/security/product decisions.

## What This Changes

The issue requests Gateway-to-Gateway federation with mutual WebSocket peer connections, a new `role: "gateway"`, and a peer routing method such as `gw.send`.

## Best Possible Solution

Keep this as the broad federation design tracker until maintainers define the peer trust model, routing semantics, config surface, HA goals, and relationship to https://github.com/openclaw/openclaw/issues/43605, https://github.com/openclaw/openclaw/pull/43656, and https://github.com/openclaw/openclaw/issues/25883.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature and architecture request. Source, docs, release-tag, and related-item inspection give high confidence that current OpenClaw lacks the requested peer role, peer method family, and federation config surface.

## Solution Assessment

Unclear. The proposed `role: "gateway"` and `gw.send` direction is plausible, but the maintainable path needs reviewed protocol, security, config, routing, HA, and session-ownership design before implementation.

## Security Review

Status: needs_attention

Summary: Gateway federation is security-sensitive because it would introduce a new cross-Gateway control-plane trust boundary.

Concerns:

- **[medium] Define peer trust before federation:** `docs/gateway/protocol.md:10`
  - body: A peer Gateway role could send session messages, trigger agent work, or expose control-plane state on another Gateway, so the design needs fail-closed peer identity, mutual auth, URL policy, scoped authorization, transcript ownership, and visibility semantics.
  - confidence: 0.9
- **[medium] Protect remote session orchestration surfaces:** `docs/gateway/tools-invoke-http-api.md:116`
  - body: The documented tool policy treats `sessions_spawn` as remote orchestration and `sessions_send` as message injection, so federation must preserve or replace those safety boundaries explicitly.
  - confidence: 0.88
