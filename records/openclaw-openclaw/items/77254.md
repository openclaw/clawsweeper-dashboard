## Summary

Keep open: current main still has the relevant v2026.5.3 resolution shape, including a generic outbound adapter path that returns undefined on a channel-registry miss and a target-resolution fallback that still reads the mutable active plugin registry. The report includes detailed live Discord failures plus a local fallback hotfix that recovered pending deliveries, so this is a credible narrow core/channel-registry bug rather than stale support noise.

## What This Changes

Ensure Discord cron announce delivery and final replies keep resolving the outbound adapter after plugin registry reload or inspection activity.

## Best Possible Solution

Make outbound adapter and target resolution consistently prefer the pinned channel registry and loaded channel plugin fallback, then add regression coverage for a Discord-like channel surviving active registry replacement without broad plugin reloads.

## Reproduction Assessment

No high-confidence executable reproduction was established against current main during this read-only pass. The issue provides detailed live 2026.5.3 steps/logs, and source inspection shows the same generic error paths and registry-fallback inconsistency still exist.

## Solution Assessment

Yes, the best direction is a narrow channel-registry resolution fix with targeted regression tests. Adding a config knob, restarting providers, or reloading the full plugin registry on delivery would be a broader and riskier workaround.
