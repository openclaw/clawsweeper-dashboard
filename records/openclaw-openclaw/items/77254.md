## Summary

Keep open: the report is specific, matches current source-level outbound failure paths, and current main still lacks a generic fallback when the channel registry lookup drops an external Discord adapter.

## What This Changes

The issue reports Discord outbound delivery losing its adapter after plugin registry reloads, causing cron announcements and final replies to fail despite Discord remaining connected.

## Best Possible Solution

Patch the generic outbound channel resolver and adapter loader so loaded or pinned channel plugins remain authoritative across active registry churn, with regression coverage for external channel delivery after registry replacement and no Discord-specific hard-code.

## Reproduction Assessment

No controlled live current-main reproduction was run. The source path is clear: if the Discord entry is absent from the channel registry, adapter loading returns `undefined`, target resolution reports `Unsupported channel`, and delivery throws `Outbound not configured`, matching the production logs.

## Solution Assessment

Yes, the narrow maintainable fix is generic channel-plugin resolution that uses the pinned/loaded channel surface consistently, plus regression tests for registry replacement. A Discord-only workaround would put the fix on the wrong ownership boundary.
