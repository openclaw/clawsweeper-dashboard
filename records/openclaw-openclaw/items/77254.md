The issue asks OpenClaw to keep Discord outbound adapter resolution stable across plugin registry reloads so cron announcements and final replies continue delivering.

Patch the generic channel outbound resolver/loader so target resolution and adapter loading consult the same authoritative loaded or pinned channel-plugin surface, without hard-coding Discord, and add a regression for external channel delivery after registry churn.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77254. The observable bug is that generic Discord outbound delivery can lose its adapter after plugin registry/channel registry churn, producing `Unsupported channel: discord` for cron target resolution and `Outbound not configured for channel: discord` for final replies while the Discord plugin remains connected. Keep the fix generic: do not hard-code Discord, do not add config options, and do not change provider/auth policy. Make target resolution and `loadChannelOutboundAdapter` consult a consistent authoritative loaded/pinned channel-plugin surface, with a safe fallback when the registry entry misses but the loaded channel plugin still has `outbound`. Add or update focused regression coverage for an external channel/plugin whose adapter remains loaded across registry replacement or channel-registry divergence. Include a `CHANGELOG.md` fix entry because this is user-facing. Related context: #54599 is historical registry churn cleanup; #77241 is a distinct Discord npm secret-contract startup issue and should not be folded into this fix unless source evidence proves the same root cause.
