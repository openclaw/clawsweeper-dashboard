## Summary

Keep open: the live 2026.5.3 report still maps to current generic outbound resolution paths, and I did not find current-main proof or coverage showing Discord outbound adapter resolution survives registry churn. The closed #54599 cleanup is related background, while open #77241 and #71127 describe adjacent but distinct failure modes.

## What This Changes

The issue reports Discord cron announcements and final replies losing the outbound adapter after plugin registry/runtime reloads, despite the Discord plugin remaining connected and direct Discord sends succeeding.

## Best Possible Solution

Patch generic channel outbound resolution so loaded or pinned channel plugins remain authoritative across registry replacement, with regression coverage for external channel adapter and target resolution after registry churn.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection gives a credible reproduction path: if Discord plugin lookup returns undefined after registry churn, target resolution emits `Unsupported channel` and shared delivery throws `Outbound not configured`, matching the report.

## Solution Assessment

Yes, the best direction is a generic resolver/loader fix that consistently uses the authoritative channel registry or loaded channel plugin, plus focused regression tests. A Discord-only hard-code or startup-only workaround would leave the shared outbound contract fragile.
