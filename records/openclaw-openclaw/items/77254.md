## Summary

Keep open: the report is specific, matches current generic outbound failure paths, and neither the closed #54599 cleanup nor the open #77241 Discord contract issue supersedes this distinct adapter-loss evidence.

## What This Changes

The issue reports Discord outbound delivery losing its adapter after plugin registry reloads, causing cron announcements and final replies to fail despite the Discord plugin remaining connected.

## Best Possible Solution

Patch the generic outbound channel resolution path so loaded or pinned channel plugins remain authoritative across active registry churn, without hard-coding Discord, and add regression coverage for external channel delivery after registry replacement.

## Reproduction Assessment

No controlled live current-main reproduction was run. Source inspection gives a medium-confidence path: if Discord is absent from the outbound resolution registry, target resolution emits `Unsupported channel` and delivery throws `Outbound not configured`, matching the report.

## Solution Assessment

Yes, the best direction is a narrow generic resolver/loader fix that consistently uses the pinned or loaded channel registry. A Discord-only workaround would be the wrong ownership boundary, and current main does not provide enough terminal proof to close the issue.
