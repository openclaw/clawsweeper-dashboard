Keep open: the report has concrete production evidence and current main still has inconsistent channel-registry resolution paths that can miss an external channel outbound adapter and emit the reported delivery errors.

## What This Changes

Ensure Discord external channel outbound delivery retains its adapter across plugin registry reloads so cron announce delivery and final replies keep sending.

## Best Possible Solution

Make generic outbound channel adapter and target resolution use one consistent pinned-or-loaded channel registry contract across registry replacement, with regression coverage for external channel adapters and no Discord-specific hard-code.

## Solution Assessment

Yes. The narrowest maintainable fix is a generic resolver/loader change at the channel registry seam, plus regression tests for external channel delivery after registry churn; a Discord-only workaround would be the wrong boundary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77254. Generic outbound delivery can lose an external Discord channel outbound adapter after plugin registry reload or pinned-registry mismatch, producing `Unsupported channel: discord` and `Outbound not configured for channel: discord` while the Discord plugin/provider is otherwise loaded. Keep the fix generic in channel plugin resolution and outbound adapter loading; do not hard-code Discord, do not change package-manager install/secret-contract repair behavior, and do not roll in stuck-session recovery policy from #71127. Likely direction: make `loadChannelOutboundAdapter` and `resolveOutboundChannelPlugin` share the same pinned/loaded channel registry semantics, including fallback from an empty pinned channel registry to a loaded/active channel plugin where appropriate, and remove or constrain mutable active-registry fallback so registry churn cannot hide the startup/loaded channel plugin. Add regression coverage for external channel adapter retention after registry replacement or empty pinned-channel state. Update CHANGELOG.md because this is user-facing delivery behavior.
