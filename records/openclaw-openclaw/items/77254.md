Keep open: the report is specific, not covered by the closed #54599 cleanup or the separate open #77241 startup-secret issue, and current main still has generic outbound paths that produce the reported errors when Discord is missing from channel plugin resolution.

## What This Changes

Fix generic outbound delivery so a loaded external Discord channel plugin keeps its outbound adapter and target resolver after plugin registry reloads.

## Best Possible Solution

Make channel plugin registry or loaded-plugin state authoritative for outbound target and adapter lookup across registry churn, with no Discord-specific hard-code, then add a regression that simulates external channel delivery after active registry replacement.

## Reproduction Assessment

No, there is not a high-confidence live current-main reproduction. Source inspection does show the same failure path: missing Discord plugin resolution yields `Unsupported channel`, and missing outbound adapter resolution yields `Outbound not configured`.

## Solution Assessment

Yes, the narrow maintainable solution is a generic resolver/loader fix that preserves loaded or pinned channel plugin state across registry churn. A Discord-only workaround would be the wrong owner boundary.

## Repair Work Prompt

Fix openclaw/openclaw#77254. The observable bug is that generic Discord outbound delivery can lose the external Discord channel plugin after plugin registry reload/churn, causing cron/final-reply paths to emit `Unsupported channel: discord` or `Outbound not configured for channel: discord` even though Discord is loaded and direct CLI sends work. Keep the fix generic: do not hard-code Discord, do not change plugin ownership boundaries, and do not add a new config option. Review `src/infra/outbound/channel-resolution.ts`, `src/channels/plugins/registry-loader.ts`, `src/channels/plugins/outbound/load.ts`, `src/plugins/runtime.ts`, and gateway plugin pin/reload paths. Add focused regression coverage that simulates a loaded external channel with outbound adapter/resolveTarget, then replaces the active plugin registry and verifies both outbound target resolution and delivery adapter loading still use the loaded/pinned channel plugin. Preserve existing behavior for truly missing channels. Add a user-facing changelog fix entry if behavior changes. Related context: #54599 is closed but similar; #77241 is separate startup secret-contract loading; the later comment on this issue mentions recipient loss after stuck-session recovery and should be considered evidence of the same delivery invariant, not necessarily the same trigger.
