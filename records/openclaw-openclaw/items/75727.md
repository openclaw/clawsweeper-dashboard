## Summary

Keep open: the reported gateway/WebChat media bug is real and the PR now has current-head terminal proof, but the current patch still has blocking correctness issues around captions, transcript replacement scope, and duplicate live transcript updates.

## What This Changes

The PR changes gateway and agent media handling to trust non-external structured local-media markers, replace raw Codex `MEDIA:` transcript turns with normalized WebChat inline image messages, and add regression coverage plus a changelog entry.

## Best Possible Solution

Keep the gateway replacement seam, but preserve directive-stripped captions, constrain replacement to the current raw `MEDIA:` turn, and emit only one transcript update for the rewrite.

## Reproduction Assessment

Yes. Current main has an agent media supplement append path without a raw Codex `MEDIA:` replacement path, and the PR adds focused regression tests for the transcript/media seam; I did not execute tests in this read-only review.

## Solution Assessment

No as written. The intended gateway transcript replacement seam is the right place, but the patch needs to preserve captions, target only the current raw media turn, and avoid the second live update before it is merge-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve captions when replacing media replies:** `src/gateway/server-methods/chat.ts:170-171`
  - body: `stripVisibleTextFromMediaSupplement` clears `payload.text` for every media-bearing payload before the replacement content is built. A reply such as `Here is the image.\nMEDIA:...` can be rewritten as `Image reply` or image-only instead of preserving the assistant caption, so keep the directive-stripped visible text for non-TTS image replacements.
  - confidence: 0.91
- **[P2] Replace only the current raw MEDIA turn:** `src/gateway/server-methods/chat.ts:1497-1499`
  - body: The replacement helper searches the entire transcript branch for any assistant `MEDIA:` directive matching the payload. If an older turn reused the same media URL while the current reply did not persist a raw directive, this rewrites history and suppresses the new media append; restrict replacement to the current/latest assistant turn or pass the expected entry id.
  - confidence: 0.88
- **[P2] Avoid emitting a second update after rewrite:** `src/gateway/server-methods/chat.ts:2645-2650`
  - body: `rewriteTranscriptEntriesInSessionFile` already emits a transcript refresh after persisting a changed rewrite. Emitting an inline `message` update immediately afterward can make live history clients refresh to the rewritten branch and then append the same replacement again.
  - confidence: 0.9
