## Summary

Keep open. The PR addresses a real current-main WebChat/Codex media duplication gap, but the replacement path introduces a replay-safety bug by storing UI-only media blocks on the original provider-owned assistant turn.

## What This Changes

The PR changes gateway transcript handling and Pi tool-result media filtering so Codex `MEDIA:` image replies render once as normalized inline media and non-external structured local-media markers are trusted.

## Best Possible Solution

Keep the single-message WebChat behavior, but make the transcript rewrite replay-safe by avoiding provider-owned persistence of UI-only media blocks or by adding an explicit replay filter/test for the rewritten entry.

## Reproduction Assessment

Yes. A source-level reproduction is a Codex/AppServer session where the latest assistant transcript turn contains `MEDIA:<local path>` and the gateway final payload carries equivalent normalized media; current main appends a separate assistant-media transcript entry.

## Solution Assessment

No. The proposed direction is narrow, but the current implementation should be adjusted so display media does not become provider replay content on the original Codex assistant turn.
