## Summary

Keep this PR open but do not merge it as-is: the latest head still has correctness blockers in the media replacement path and the current-head gateway-methods CI is failing on the PR’s own new tests.

## What This Changes

The PR changes gateway and agent media handling so Codex local `MEDIA:` replies become a single normalized inline media assistant message, with tool-media trust, replay sanitation tests, and a changelog entry.

## Best Possible Solution

Keep the raw-directive replacement direction, but preserve visible reply text, replace only the current raw directive turn, make replacement retries idempotent, emit one live transcript update path, then provide fresh current-head proof and green CI.

## Reproduction Assessment

Yes. Source inspection shows current main still appends an assistant-media supplement without replacing a raw Codex `MEDIA:` turn, and the PR’s latest CI reproduces concrete failures in the new gateway directive tests.

## Solution Assessment

No as written. The replacement approach is maintainable, but the current patch drops visible text, can rewrite an older matching turn, loses idempotency across retries, and double-emits replacement updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve text on non-TTS media replies:** `src/gateway/server-methods/chat.ts:170-171`
  - body: `stripVisibleTextFromMediaSupplement` clears `payload.text` for every media reply before content is built. Image and audio replies with visible assistant text then become synthetic labels such as `Image reply` or `Audio reply`; the latest CI failure shows the audio case losing `Here is the audio.`.
  - confidence: 0.94
- **[P2] Only replace the current MEDIA reply:** `src/gateway/server-methods/chat.ts:1497-1499`
  - body: This searches the whole active branch for any assistant message whose `MEDIA:` path matches the payload. If an older turn reused the same output path while the current turn did not persist a raw directive, the helper rewrites history and suppresses the new media append.
  - confidence: 0.87
- **[P2] Keep replacement retries idempotent:** `src/gateway/server-methods/chat.ts:1521-1522`
  - body: The replacement is written through the transcript rewrite path as an `AgentMessage`, but latest CI shows the resulting branch message has no `idempotencyKey`. A retry of the same chat send will not be caught by `transcriptHasIdempotencyKey` after the raw directive is gone, so it can append a second media message.
  - confidence: 0.86
- **[P2] Do not emit the replacement twice:** `src/gateway/server-methods/chat.ts:2642-2648`
  - body: `rewriteTranscriptEntriesInSessionFile` already emits a refresh update after persisting the rewritten branch. Emitting `replaced.message` immediately afterward sends an inline append for the same replacement, so unbounded history streams can refresh to the rewritten transcript and then append the same assistant message again.
  - confidence: 0.9
