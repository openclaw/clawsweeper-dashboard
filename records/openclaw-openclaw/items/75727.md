## What This Changes

The PR updates gateway and agent media handling so Codex local `MEDIA:` replies and structured trusted local-media tool results normalize into one WebChat media message instead of duplicate assistant transcript turns.

## Best Possible Solution

Keep the gateway replacement seam, but preserve directive-stripped captions, constrain replacement to the current raw `MEDIA:` turn, rely on one transcript update path, and refresh current-head proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main has the agent media supplement append path without a raw Codex `MEDIA:` replacement path, and the PR adds targeted tests for that seam. I did not execute the tests in this read-only review.

## Solution Assessment

No as written. The intended gateway transcript replacement seam is the right place, but this patch can drop captions, rewrite an older matching transcript entry, and double-emit replacement updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve captions when replacing media replies:** `src/gateway/server-methods/chat.ts:170-171`
  - body: `stripVisibleTextFromMediaSupplement` clears `payload.text` for every media-bearing payload before `assistantContent` and `mediaMessage` are built. A reply such as `Here is the image.\nMEDIA:...` can be rewritten as `Image reply` or image-only instead of preserving the assistant caption; keep the directive-stripped visible text for non-TTS media replacements.
  - confidence: 0.91
- **[P2] Replace only the current raw MEDIA turn:** `src/gateway/server-methods/chat.ts:1497-1499`
  - body: The replacement helper searches the entire transcript branch for any assistant `MEDIA:` directive matching the payload. If an older turn reused the same media URL while the current reply did not persist a raw directive, this rewrites history and suppresses the new media append; restrict replacement to the current/latest assistant turn or pass the expected entry id.
  - confidence: 0.88
- **[P2] Avoid emitting a second transcript update after rewrite:** `src/gateway/server-methods/chat.ts:2644-2650`
  - body: `rewriteTranscriptEntriesInSessionFile` already emits a refresh update after persisting a changed rewrite. Emitting an inline `message` update immediately afterward can make live history clients refresh to the rewritten branch and then append the same replacement again.
  - confidence: 0.9
