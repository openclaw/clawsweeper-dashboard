## Summary

Keep open. The PR addresses a real WebChat/Codex MEDIA duplication bug that is still present on current main, but the current branch still has three merge-blocking correctness issues in the transcript replacement path.

## What This Changes

The branch changes gateway and agent media handling so non-external structured local media can be trusted and raw Codex MEDIA transcript turns can be replaced with normalized inline media messages.

## Best Possible Solution

Keep the gateway transcript replacement approach, but preserve directive-stripped visible text, limit replacement to the current raw MEDIA assistant turn, and rely on the existing rewrite refresh for live history updates.

## Reproduction Assessment

Yes, source-level. Current main still has the media-bearing assistant-media append path, and https://github.com/openclaw/openclaw/issues/78885 confirms the same raw MEDIA plus gateway-injected duplicate behavior that this PR targets.

## Solution Assessment

No as written. The gateway seam is the right place to solve this, but the current patch can drop captions, rewrite an older matching transcript entry, and emit a second live update after the rewrite.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve captions when replacing media replies:** `src/gateway/server-methods/chat.ts:170-171`
  - body: `stripVisibleTextFromMediaSupplement` now clears `payload.text` for every media-bearing reply before replacement content is built. A reply like `Here is the image.\nMEDIA:...` can be rewritten as `Image reply` or image-only instead of preserving the directive-stripped caption, so keep non-TTS visible text while removing only the MEDIA directive.
  - confidence: 0.91
- **[P2] Replace only the current raw MEDIA turn:** `src/gateway/server-methods/chat.ts:1497-1499`
  - body: The replacement helper scans the whole transcript branch for any matching assistant MEDIA directive. If an older assistant turn used the same media URL and the current reply did not persist a raw directive, this rewrites history and suppresses the new media append; restrict the match to the current/latest assistant entry or pass the expected entry id.
  - confidence: 0.88
- **[P2] Avoid emitting a second update after rewrite:** `src/gateway/server-methods/chat.ts:2644-2650`
  - body: `rewriteTranscriptEntriesInSessionFile` already emits a transcript refresh after persisting a changed rewrite. Emitting a message-bearing update immediately afterward can make SSE history clients refresh to the rewritten branch and then append the same replacement again, so rely on the refresh or add sequencing that prevents a second append.
  - confidence: 0.9
