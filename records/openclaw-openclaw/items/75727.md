## Summary

Keep open. The PR is aimed at a real Gateway/WebChat Codex media bug, but the latest head still has blocking correctness issues in transcript replacement/live updates, and the supplied real behavior proof is for an older head than the current patch.

## What This Changes

The PR changes Gateway and agent media handling so Codex `MEDIA:` replies are normalized into one inline WebChat message and structured local media markers from non-external tool results can be trusted.

## Best Possible Solution

Keep the raw-directive replacement direction, but preserve visible text, constrain replacement to the current raw assistant turn, emit only one transcript update path, and then provide fresh current-head proof.

## Reproduction Assessment

Yes, at source level. Current main can append an assistant-media supplement without replacing a raw Codex `MEDIA:` transcript turn, and the PR adds focused tests for that seam, but I did not run a live current-main repro in this read-only review.

## Solution Assessment

No as written. The replacement approach is the right narrow path, but this patch still needs to preserve visible text, avoid branch-wide replacement, and avoid a second live transcript update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve visible text for image media replies:** `src/gateway/server-methods/chat.ts:170-171`
  - body: `stripVisibleTextFromMediaSupplement` clears `payload.text` for every media-bearing reply before display/replacement content is built. For a reply like `Here is the image.\nMEDIA:<path>`, the replacement can remove the raw directive but lose the model-authored visible text, leaving only the managed image or synthetic fallback text in WebChat.
  - confidence: 0.91
- **[P2] Limit MEDIA replacement to the current reply:** `src/gateway/server-methods/chat.ts:1497-1499`
  - body: The replacement helper scans the whole active branch for any assistant `MEDIA:` directive matching the current payload. If an older turn reused the same output path and the current turn did not persist a raw directive, this rewrites old history and suppresses the current inline image append.
  - confidence: 0.86
- **[P2] Do not emit the replacement update twice:** `src/gateway/server-methods/chat.ts:2642-2648`
  - body: `rewriteTranscriptEntriesInSessionFile` already emits a transcript refresh after persisting the rewritten branch. Emitting `replaced.message` here sends a second message-bearing update, so live unpaginated history streams can refresh to the rewritten transcript and then append the same assistant message inline.
  - confidence: 0.88
