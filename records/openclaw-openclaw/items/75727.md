## Summary

Keep open: the bug is source-reproducible and the PR is addressing the right surface, but the latest head still introduces blocking WebChat transcript regressions and the real-behavior proof has not been refreshed after the last same-surface commit.

## What This Changes

The PR updates gateway and agent media handling so Codex local MEDIA replies can become one normalized inline WebChat image message while external MCP local-media markers remain untrusted.

## Best Possible Solution

Preserve the useful media-trust and replay-sanitization direction, but replace only the current raw MEDIA assistant turn, keep visible caption text after directive stripping, and emit exactly one live transcript update per persisted assistant message.

## Reproduction Assessment

Yes, source-level. Current main can append a managed media assistant supplement after Pi has already persisted a raw Codex MEDIA assistant turn, and the PR includes a focused fixture for that transcript/media path, though I did not execute tests in this read-only review.

## Solution Assessment

No as written. Replacing the raw directive turn is the right approach, but the current patch still has unbounded replacement, visible-text loss, and duplicate transcript-update emission defects that should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid emitting the appended media message twice:** `src/gateway/server-methods/chat.ts:2677-2683`
  - body: `appendAssistantTranscriptMessage` already calls `appendInjectedAssistantMessageToTranscript`, which emits a transcript update with the appended message. This second emit sends the same message again; the history SSE inline path appends every message update without message-id dedupe, so live WebChat clients can see the assistant-media reply twice. Thread `sessionKey` into the original append emit or otherwise make the append path emit once.
  - confidence: 0.91
- **[P2] Preserve visible text for non-TTS media supplements:** `src/gateway/server-methods/chat.ts:2567`
  - body: Passing all media-bearing payloads through `stripVisibleTextFromMediaSupplement` removes the user's visible caption before both WebChat content builders run. For audio/image replies this stores only media blocks or generic fallback text instead of the original caption, and it contradicts the added audio regression that expects `Here is the audio.` to remain in the assistant message. Strip the MEDIA directive text itself, not the whole visible reply text.
  - confidence: 0.88
- **[P2] Bound media replacement to the current assistant turn:** `src/gateway/server-methods/chat.ts:1507-1509`
  - body: The replacement helper still scans the whole active branch for any assistant MEDIA directive matching the payload URL, then returns without appending the current supplement. If a tool reuses a stable output path and the current run did not persist a raw directive, this can rewrite an older assistant turn and drop the current inline image message. Limit replacement to the latest/current-run assistant entry, or append when the match is not the current reply.
  - confidence: 0.84
