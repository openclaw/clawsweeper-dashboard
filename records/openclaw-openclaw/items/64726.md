## Summary

Keep open. Current main partially satisfies the request with batch embedding, exact-hash cache reuse, entry counting, and max-entry pruning, but fuzzy cache reuse, hit/miss/latency metrics, cache warming, gradual model migration, and compressed vector storage are still absent and need maintainer design.

## What This Changes

Request to improve embedding cache performance with provider batch calls, smarter invalidation/reuse, warming, metrics, model-migration support, and storage pruning/compression.

## Best Possible Solution

Keep the shipped batch/exact-cache behavior, then split the remaining work into staged, compatibility-safe changes: metrics first, opt-in fuzzy reuse with safety tests, explicit warming policy, and schema-backed migration/storage updates.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a feature and performance request. The remaining gaps are high-confidence from source inspection: cache reuse is exact-hash only, status lacks hit/miss/latency fields, and provider/model changes still force full reindex.

## Solution Assessment

Not applicable because no patch is under review. The narrow maintainable path is to decompose the broad cache program into smaller design-backed changes rather than treating it as one automated repair.
