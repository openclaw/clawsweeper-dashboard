## Summary

Keep open. Current main partially addresses the request with provider batch embedding, exact-hash cache reuse, cache entry counting, and max-entry pruning, but the requested fuzzy reuse, hit/miss/latency metrics, frequency-based warming, gradual model migration, and compressed cache storage are still not implemented.

## What This Changes

The issue requests optimizing the memory embedding cache with batching, smarter invalidation, warming, statistics, model migration, and cache storage pruning/compression.

## Best Possible Solution

Keep the shipped exact-cache and batch behavior, then split the remaining cache work into staged changes: cache metrics first, opt-in fuzzy reuse with safety tests, a clear warming policy, and backward-compatible schema updates for model migration and storage format changes.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a feature/performance request. The remaining gaps are high-confidence from source: current cache reuse is exact-hash only, status has no hit/miss/latency fields, and model/provider changes still trigger full reindex.

## Solution Assessment

Not applicable because no patch is under review. The narrow maintainable path is to decompose this broad request into smaller design-backed memory-cache changes rather than treating it as one automated rewrite.
