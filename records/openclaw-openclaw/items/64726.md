## Summary

Keep open. Current main only partially satisfies the request: batch embedding, exact-hash cache reuse, cache row seeding during safe reindex, status entry counts, and max-entry pruning exist, but fuzzy reuse, hit/miss/latency metrics, frequency-based warming, gradual model migration, and compressed vector storage are still missing.

## What This Changes

The issue requests improving memory embedding cache behavior with batch embedding, smarter invalidation, cache warming, cache metrics, model migration support, and cache storage optimization.

## Best Possible Solution

Keep the shipped batch and exact-cache behavior, then split the remaining work into staged memory-cache improvements: observable metrics first, opt-in fuzzy reuse with safeguards, a frequency-based warming policy, and backward-compatible schema changes for model migration and storage format updates.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a feature/performance request. The current gap is high-confidence from source inspection: exact hash lookup is the only reuse path, status lacks hit/miss/latency fields, and provider/model changes still require full reindex.

## Solution Assessment

No proposed patch is under review, and one broad cache rewrite would not be the best maintainable path. The safer solution is several smaller design-backed changes with focused tests for cache metrics, fuzzy matching safety, warming policy, migration compatibility, and storage format handling.
