Full review comments:

- **[P1] Sanitize the bridge env before spawning:** `extensions/claude/src/app-server/client.ts:82-85`
  - body: `cfg.appServer.env` comes from plugin config, but the client merges it straight into `process.env` before `spawn`. That bypasses OpenClaw's host-exec env sanitizer, so config can override dangerous variables such as loader, credential, prompt, or PATH-like settings for a subprocess that holds Anthropic credentials. Route the base env plus overrides through `sanitizeHostExecEnv` or an approved SDK/Core seam before spawning, and cover blocked override behavior.
  - confidence: 0.93
- **[P1] Do not bypass release age for the contributor bridge package:** `pnpm-workspace.yaml:12`
  - body: This adds an unversioned `minimumReleaseAgeExclude` for `@zeroaltitude/openclaw-claude-bridge` while the official bundled plugin depends on that contributor-scoped executable. Every future bridge patch could bypass the workspace release-age guard and then run with Anthropic credentials, so this needs OpenClaw package ownership or explicit secops approval with a narrower dependency policy before merge.
  - confidence: 0.9
- **[P1] Register claude in the middleware runtime normalizer:** `src/plugins/agent-tool-result-middleware-types.ts:5`
  - body: The public type now accepts `"claude"`, and the Claude bridge calls `createAgentToolResultMiddlewareRunner({ runtime: "claude" })`, but `AGENT_TOOL_RESULT_MIDDLEWARE_RUNTIMES` still only includes `openclaw` and `codex`. Plugin authors declaring `runtimes: ["claude"]` will be normalized to an empty list, so the new runtime API is advertised but unreachable.
  - confidence: 0.94
- **[P2] Forward source-reply telemetry to the runner:** `extensions/claude/src/app-server/run-attempt.ts:404-407`
  - body: The dynamic-tool bridge records `messagingToolSourceReplyPayloads`, but result construction only copies the regular sent text/media/target fields.   - confidence: 0.92
