## Summary

Keep open. Current main still has the OpenAI-compatible TTS base URL path, `speed`, `instructions`, and `responseFormat`, but it does not retain or merge arbitrary `extraBody`/`extra_body` fields into OpenAI speech requests, so the Kokoro/custom endpoint use case remains unsolved.

## What This Changes

Add optional OpenAI TTS `extraBody`/`extra_body` config that is merged into `/audio/speech` request payloads for custom OpenAI-compatible endpoints.

## Best Possible Solution

Implement this in the bundled OpenAI speech provider: accept sanitized object-valued `extraBody` and `extra_body` on the current `messages.tts.providers.openai` config path, preserve it through provider resolution, merge it into both normal and telephony `/audio/speech` request bodies with documented precedence, and update focused provider tests plus TTS/OpenAI docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a provider-level unit probe with `messages.tts.providers.openai.extraBody` and a custom `baseUrl`; current source inspection shows the value is not retained by normalization and cannot appear in the serialized request body.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-owned OpenAI TTS passthrough; core already has a generic provider config catchall, and existing OpenAI-compatible speech helper code shows this can be handled without a new core schema or plugin API.
