## Summary

Keep open: this draft maintainer PR is useful, but the new override reader is source-path-only and likely no-ops from packaged `dist`, leaving managed plugin installs without the intended security pins.

## What This Changes

The PR copies root npm overrides into managed external plugin npm roots before install, tracks OpenClaw-owned override keys, and documents/changelogs the behavior.

## Best Possible Solution

Resolve the OpenClaw package root with the existing source/dist-aware helper and add built/package coverage or proof before documenting the override guarantee.

## Reproduction Assessment

Yes for the review finding: source inspection shows packaged runtime imports from `dist`, where the new `../../package.json` URL resolves above the package root and falls into the empty-override catch path. A built package smoke should be added as runtime proof.

## Solution Assessment

No; the direction is right, but the hard-coded source-relative package manifest URL is not maintainable across source and packaged runtimes. The narrower fix is to use the existing package-root resolver and cover the dist path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve overrides from the package root at runtime:** `src/infra/npm-managed-root.ts:36`
  - body: This hard-codes `../../package.json` relative to the module URL. In packaged OpenClaw, the CLI loads `./dist/entry.js`, so bundled code runs from `dist` and this URL points above the package root; `readOpenClawManagedNpmRootOverrides()` then catches the read failure and returns `{}`, meaning installed users do not get the security overrides this PR is meant to apply. Please resolve the package root with the existing source/dist-aware helper and cover the built/package path.
