## Summary

Keep open: current main and the latest release still omit DashScope/Qwen from the provider-usage resolver, and the Qwen plugin owns those aliases without usage auth or snapshot hooks. The linked Token Plan PR is adjacent, but it explicitly leaves Alibaba usage accounting as a separate follow-up.

## What This Changes

Request DashScope/Qwen provider usage API support so UI/status surfaces no longer show zero or missing usage while tokens are consumed.

## Best Possible Solution

Add Qwen-owned usage auth and usage snapshot hooks, then map Qwen/DashScope aliases into the existing provider-usage resolver with docs, focused tests, and redacted live endpoint proof.

## Reproduction Assessment

Yes, at source level: resolveUsageProviderId cannot return Qwen/DashScope IDs and models.authStatus filters through that resolver before usage polling. I did not run a live DashScope credential scenario in this read-only review.

## Solution Assessment

No, an allowlist-only change is not the best complete fix because it would make DashScope eligible for polling without provider-owned auth extraction or quota response normalization. The maintainable path is Qwen plugin usage hooks plus alias wiring and endpoint-contract proof.
