## Summary

Keep open: current main and the latest release still exclude Qwen/DashScope from usage-window provider IDs, while the Qwen plugin owns those provider aliases and the UI/status path only renders quota windows when backend usage snapshots are present. The closed DashScope token-streaming fixes cover per-response token capture, not this provider usage-window support.

## What This Changes

Add Qwen/DashScope provider usage API support so UI/status usage windows do not stay empty while those providers are used.

## Best Possible Solution

Implement Qwen-owned `resolveUsageAuth` and `fetchUsageSnapshot` support, with minimal core alias/type/docs wiring and focused regression coverage after verifying the DashScope/Qwen usage endpoint contract.

## Reproduction Assessment

Yes, source-level: on current main, `resolveUsageProviderId` cannot return Qwen/DashScope IDs, and both auth-status and models-status usage paths call that resolver before loading usage summaries.

## Solution Assessment

No, a static-list-only patch is not the best solution. The maintainable fix is provider-owned Qwen usage auth and snapshot hooks plus narrow alias/type/docs/test updates once the endpoint contract is verified.
