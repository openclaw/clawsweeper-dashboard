## Summary

Keep open: current main and the latest release still omit Qwen/DashScope from provider usage-window routing, while the Qwen plugin owns the affected aliases but does not implement usage auth or snapshot hooks. The related merged DashScope fixes address per-response streaming token capture, not provider quota-window snapshots for the UI.

## What This Changes

The issue asks OpenClaw to add DashScope/Qwen provider usage API support so the UI does not show provider usage as 0 while tokens are being consumed.

## Best Possible Solution

Add Qwen-owned usage-window support through existing plugin usage hooks, mapping the Qwen/Model Studio/DashScope aliases and documenting supported plan semantics after verifying the official DashScope contract.

## Reproduction Assessment

Yes, source-level. `resolveUsageProviderId('dashscope')` and `resolveUsageProviderId('qwen')` are excluded by the closed allowlist/type, and the UI auth-status path only loads usage for IDs that pass that resolver.

## Solution Assessment

No, not as a static allowlist-only patch. The maintainable fix is provider-owned Qwen usage auth and snapshot hooks plus narrow alias/type/docs wiring once DashScope usage endpoint semantics are verified.
