## Summary

Keep open: current main and v2026.6.10 still omit Qwen/DashScope from the provider-usage resolver, while the Qwen plugin owns those aliases but does not implement provider-usage auth or snapshot hooks.

## What This Changes

Request to add DashScope/Qwen provider usage API support so UI/status usage surfaces can report provider usage instead of showing zero or empty usage while tokens are consumed.

## Best Possible Solution

Add Qwen-owned usage auth and usage snapshot hooks, then map the Qwen/DashScope aliases into the existing provider-usage resolver with docs, focused tests, and redacted live endpoint proof.

## Reproduction Assessment

Yes, at source level: resolveUsageProviderId cannot return Qwen/DashScope IDs and models.authStatus filters through that resolver before usage polling. I did not run a live DashScope credential scenario in this read-only review.

## Solution Assessment

No, the issue's allowlist-only proposal is not the best complete fix because it would make the provider eligible for polling without provider-owned auth extraction or quota response normalization. The maintainable path is Qwen plugin usage hooks plus alias wiring and endpoint-contract proof.
