## Summary

Keep this open. Current main still does not support Qwen/DashScope provider usage/quota snapshots: the supported usage-provider set excludes Qwen and DashScope, the Qwen plugin owns those provider ids but does not implement the usage auth/snapshot hooks, and the UI/status paths only load provider usage after that resolver returns a supported id.

## What This Changes

Request DashScope/Qwen provider-usage support so UI/status usage does not remain empty or zero while those providers are consuming tokens.

## Best Possible Solution

Add Qwen-owned usage-window support through the existing provider plugin hooks, with minimal core type/alias additions so `qwen`, `qwencloud`, `modelstudio`, and `dashscope` resolve to a bundled Qwen usage provider when a documented endpoint contract exists.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure a Qwen/DashScope provider and the current resolver returns `undefined`, so the status/UI usage paths never request a Qwen/DashScope quota window.

## Solution Assessment

Yes, with one prerequisite. The narrow maintainable direction is provider-owned Qwen/DashScope `resolveUsageAuth` and `fetchUsageSnapshot` support plus alias/type wiring, not only adding `dashscope` to the static list.
