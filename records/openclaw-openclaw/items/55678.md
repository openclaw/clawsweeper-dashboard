## Summary

Keep open: current main and v2026.6.6 still omit Qwen/DashScope from provider-usage IDs, and the bundled Qwen plugin has no usage auth/snapshot hook, so UI/status usage cannot request Qwen provider usage windows. The related merged DashScope fixes cover per-response streaming token capture, not provider quota-window snapshots.

## What This Changes

Request DashScope/Qwen provider usage API support so UI/status usage displays reflect Qwen account or token consumption instead of staying empty or zero.

## Best Possible Solution

Add Qwen-owned `resolveUsageAuth` and `fetchUsageSnapshot` support, plus narrow usage-provider alias/type/docs wiring, after confirming the DashScope/Qwen quota endpoint contract.

## Reproduction Assessment

Yes, source-level: current `resolveUsageProviderId` cannot return Qwen/DashScope aliases, and the auth-status/status paths use that resolver before loading usage summaries. I did not run a live Qwen credential scenario in this read-only review.

## Solution Assessment

No, not as an allowlist-only patch. The maintainable solution is provider-owned Qwen usage auth and snapshot fetching plus narrow alias/type/docs wiring after upstream endpoint semantics are verified.
