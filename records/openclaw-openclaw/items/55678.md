## Summary

Keep open: current main and the latest release still omit DashScope/Qwen from the provider-usage resolver, while the Qwen plugin owns those aliases but does not implement the provider-owned usage auth/snapshot hooks. The remaining work needs provider-owner/product review because the DashScope quota endpoint, credential source, and quota-window semantics are not settled by source alone.

## What This Changes

The issue requests DashScope/Qwen usage support so the UI usage/cost display no longer stays at zero while tokens are being consumed.

## Best Possible Solution

Add Qwen-owned usage auth and usage snapshot support through the existing plugin hooks, then wire the aliases into the usage resolver with docs, focused tests, and redacted live endpoint proof after the provider quota semantics are confirmed.

## Reproduction Assessment

Yes, at source level: current main omits DashScope/Qwen aliases from the usage provider union and resolver, so models.authStatus cannot select them for usage polling. I did not run a live DashScope credential scenario in this read-only review.

## Solution Assessment

No, the issue's allowlist-only proposal is not the best complete fix because it would make the provider eligible for polling without proving provider-owned auth extraction or quota response normalization. The maintainable path is Qwen plugin usage hooks plus alias wiring, docs, tests, and endpoint-contract proof.
