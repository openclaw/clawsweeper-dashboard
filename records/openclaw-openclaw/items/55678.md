## Summary

Keep open: current main and v2026.5.22 still exclude Qwen/DashScope from provider-usage IDs, and the bundled Qwen plugin has no usage auth/snapshot hooks, so status/UI usage surfaces cannot request Qwen usage windows. The related merged DashScope work appears to cover per-response streaming token capture, not provider quota-window snapshots.

## What This Changes

Request support for DashScope/Qwen provider usage so UI and status surfaces show real usage instead of 0 for Qwen/DashScope users.

## Best Possible Solution

Add Qwen-owned `resolveUsageAuth` and `fetchUsageSnapshot` support, with minimal usage-id alias wiring, docs, and focused tests after confirming the DashScope/Qwen quota-window contract.

## Reproduction Assessment

Yes, source-level: current `resolveUsageProviderId` cannot return Qwen/DashScope aliases, and the auth-status/status paths use that resolver before loading usage summaries. I did not run a live Qwen credential scenario in this read-only review.

## Solution Assessment

No, not as an allowlist-only patch. The maintainable fix is provider-owned Qwen usage auth and snapshot fetching plus narrow alias/type/docs wiring after the upstream endpoint semantics are verified.
