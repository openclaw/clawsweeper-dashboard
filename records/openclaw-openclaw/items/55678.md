## Summary

Keep open: current main and the latest release still omit Qwen/DashScope from the provider-usage ID resolver, while the Qwen plugin owns those provider aliases but does not expose provider-owned usage auth/snapshot hooks. The issue remains valid at source level, but the next step is provider-owner/product review because DashScope quota endpoint semantics and UI window behavior are not settled by source alone.

## What This Changes

Request to show real DashScope/Qwen token usage in the UI instead of reporting zero because DashScope is not represented in the usage-provider path.

## Best Possible Solution

Add Qwen-owned usage auth and usage snapshot support through the existing plugin hooks, then wire the aliases into the usage resolver with docs, tests, and redacted live endpoint proof after the provider quota semantics are confirmed.

## Reproduction Assessment

Yes, at source level: the current usage-provider union and resolver omit Qwen/DashScope aliases, so auth-status usage polling cannot select those providers. I did not run a live DashScope credential scenario in this read-only review.

## Solution Assessment

No, the allowlist-only proposal is not the best complete fix because it would enable polling without proving provider-owned auth and quota response normalization. The maintainable path is Qwen-owned resolveUsageAuth/fetchUsageSnapshot support plus alias wiring, docs, tests, and endpoint-contract proof.
