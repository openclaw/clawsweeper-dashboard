## Summary

Keep this open: current main and the latest v2026.5.18 release still exclude Qwen/DashScope from usage-window provider ids, and the Qwen plugin has no usage snapshot hooks; the related merged DashScope streaming-token fix does not cover this provider quota/UI path.

## What This Changes

Request DashScope/Qwen usage-window API support so UI and CLI usage surfaces do not show missing or zero usage while tokens are being consumed.

## Best Possible Solution

Add Qwen-owned usage-window support through provider hooks, with `UsageProviderId` alias wiring, docs, and regression coverage after verifying the official DashScope endpoint contract.

## Reproduction Assessment

Yes, source-level. On current main, `resolveUsageProviderId('dashscope')` or `resolveUsageProviderId('qwen')` returns undefined because the allowlist and union omit them, and the status/UI paths call that resolver before loading usage summaries.

## Solution Assessment

Not as a static-list-only patch. The maintainable solution is Qwen plugin-owned `resolveUsageAuth` and `fetchUsageSnapshot` support plus minimal core alias/type/docs wiring, once the DashScope usage endpoint contract is confirmed.
