## Summary

Keep open. Current main and the latest release still lack Zalo inbound photo media extraction, while this PR has sufficient real Zalo screenshot proof and no concrete patch defect; the remaining blocker is maintainer acceptance of the new sender-triggered remote fetch surface.

## What This Changes

The PR adds Zalo inbound photo extraction, bounded remote download, MIME normalization, and focused tests so detected photos are passed into `buildContext({ media })`.

## Best Possible Solution

Land after maintainer acceptance of the bounded Zalo photo-download behavior and fresh focused validation on the current head, or intentionally pause if automatic inbound media download is not desired.

## Reproduction Assessment

Yes, at source level: current main has no media field on `ZaloInboundMessage` and flattens attachment content to text, while `zca-js` v2.1.2 exposes attachment `href`/`thumb`/`type` fields. I did not run a live Zalo account in this read-only review.

## Solution Assessment

Yes, this is the best owner-boundary shape I found: the Zalo plugin extracts its own transport metadata and feeds the existing inbound media contract. Leaving images to an agent image tool would preserve an unreliable workaround instead of using the channel media pipeline already used by sibling channels.
