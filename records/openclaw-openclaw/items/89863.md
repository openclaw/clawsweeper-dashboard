## Summary

Keep open: current main and the latest release still log package-state checker load failures by reading `entry.pluginId` directly, while this draft MEMBER/maintainer PR proposes a small defensive guard and regression. It is not cleanup-closeable; the remaining decision is maintainer review plus focused regression or CI proof before merge.

## What This Changes

The PR replaces direct package-state warning access to `entry.pluginId` with a safe log-label helper and adds a regression for a throwing `pluginId` getter during checker load failure.

## Best Possible Solution

Land the narrow warning-label guard only after maintainer review accepts the defensive hardening and focused package-state regression or equivalent CI proof is available.

## Reproduction Assessment

Partly. Source inspection reproduces the escape if a package-state catalog entry violates the `pluginId` string contract, but production catalog construction currently normalizes `pluginId` into a plain string before returning entries.

## Solution Assessment

Yes as a narrow defensive mitigation, with a proof caveat. Guarding only the warning label is the least invasive fix if maintainers want contract-violating catalog entries to stay nonblocking; broader loader or catalog changes are not justified by the current evidence.
