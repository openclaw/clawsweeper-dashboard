## Summary

Keep open. Current main still sends immediate OpenResponses metadata, but user-visible `response.output_text.delta` is only emitted after assistant text exists or after the agent command returns; there is no current WebChat early-flush/progress configuration. This is best treated as a product/API improvement rather than an auto-closeable bug because maintainers need to choose the supported progress contract.

## What This Changes

Request a supported WebChat/OpenResponses early visible progress signal so streaming clients do not appear idle before the first assistant `response.output_text.delta`.

## Best Possible Solution

Define an explicit OpenResponses/WebChat progress contract, preferably by rendering existing `response.in_progress` or adding an additive non-transcript progress event before considering an opt-in visible text flush, then document and test the chosen behavior.

## Reproduction Assessment

Yes, for the current behavior from source: visible `response.output_text.delta` waits for assistant text or post-run fallback. I did not live-measure the reported latency distribution on Linux/systemd.

## Solution Assessment

Unclear. A hardcoded assistant sentence is not the best path; maintainers should choose between rendering existing metadata, adding an additive non-transcript progress event, or adding an opt-in early flush setting.
