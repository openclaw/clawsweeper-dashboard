Keep open: current main and the latest release still do not provide an early WebChat/OpenResponses progress signal, and the linked implementation PR is open, conflicting, and blocked on the public Gateway/WebChat contract shape.

## What This Changes

The issue requests a supported early visible progress signal for WebChat `/v1/responses` streams before real assistant text is available.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/88048 with a maintainer-approved non-transcript progress signal under a supported Gateway/OpenResponses surface, while preserving `response.output_text.delta` for real assistant text.

## Reproduction Assessment

Yes, at source level: `/v1/responses` defaults to WebChat, writes structural SSE events immediately, and only writes `response.output_text.delta` after assistant content or the post-run fallback. I did not run a fresh live timing repro in this read-only review.

## Solution Assessment

Unclear. A non-transcript progress event appears more maintainable than an early fake assistant delta, but the supported event/config contract needs maintainer approval before this should be considered solved.
