## What This Changes

The issue requests a supported early visible or progress signal for WebChat requests using `/v1/responses` streaming before real assistant text is produced.

## Best Possible Solution

Define a maintainer-approved non-transcript progress event or explicit opt-in WebChat early-flush contract, keep hardcoded assistant text out of default transcripts, and cover stream order plus UI rendering.

## Reproduction Assessment

Yes. Current source shows structural SSE events are sent immediately, while `response.output_text.delta` is emitted only from assistant stream content or the post-run fallback; the reporter also supplied concrete 2026.5.20 timing data.

## Solution Assessment

Unclear. A hardcoded visible assistant delta is not the best default; maintainers should choose a progress event or explicit opt-in configured text so OpenResponses transcript semantics stay clean.
