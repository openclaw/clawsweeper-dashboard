## Summary

Keep open: this PR addresses a credible external-channel runtime gap and the supplied real-environment proof is sufficient, but the patch still lets promise-returning setters reject outside the sync try/catch and hit the gateway unhandled-rejection exit path.

## What This Changes

The PR adds an optional `ChannelGatewayAdapter.setChannelRuntime` hook, calls it before `startAccount`, and adds server-channel unit coverage for ordering and sync throw handling.

## Best Possible Solution

Keep the new external runtime seam, but make setter invocation tolerant of both synchronous throws and rejected thenables, with regression coverage for the async failure path.

## Reproduction Assessment

Yes, source-level: current main only passes the resolved channel runtime through `ctx.channelRuntime`, while the PR body gives a concrete WSL2 `openclaw-weixin` timeout-to-running proof path. I did not run that live channel setup independently.

## Solution Assessment

No, not as-is. The hook is a narrow maintainable seam, but the implementation must guard promise-like setter failures so the documented non-blocking warning behavior holds for async plugin code too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Catch async setChannelRuntime rejections:** `src/gateway/server-channels.ts:517`
  - body: `setChannelRuntime` is typed as returning `void`, but TypeScript still allows an external plugin to implement it with an `async` function. If that returned promise rejects, this synchronous `try/catch` will not run and the gateway's unhandled-rejection handler can exit the process instead of logging the warning and continuing startup. Capture the return value and attach a rejection handler or reject promise-like setters with a swallowed catch.
  - confidence: 0.88
