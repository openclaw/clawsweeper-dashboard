## Summary

Keep this PR open: it addresses a plausible external channel runtime regression and includes real Weixin startup proof, but the current hook invocation still lets rejected async setters escape as fatal unhandled rejections, so it should not merge as-is.

## What This Changes

The PR adds an optional `ChannelGatewayAdapter.setChannelRuntime` hook, invokes it before `startAccount` with the resolved channel runtime, and adds server-channel unit coverage.

## Best Possible Solution

Land the external runtime seam only after guarding async and sync setter failures while preserving the existing `ctx.channelRuntime` path and adding regression coverage for rejected async setters.

## Reproduction Assessment

Yes, source-level: current `main` only exposes the resolved channel runtime through `ChannelGatewayContext` during `startAccount`, and the PR body gives a concrete WSL2 `openclaw-weixin` timeout-to-running proof path. I did not independently run that live channel setup.

## Solution Assessment

No, not as-is. The hook is a reasonable narrow seam, but the implementation needs to catch promise-returning setter failures so the advertised non-blocking warning behavior holds for async plugin code too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle async setChannelRuntime rejections:** `src/gateway/server-channels.ts:517`
  - body: `ChannelGatewayAdapter.setChannelRuntime` is typed as returning `void`, but TypeScript still allows plugins to implement it with an `async` function. If that returned promise rejects, this sync `try/catch` will not see it and the gateway's fatal unhandled-rejection handler can exit the process instead of logging the warning and continuing startup. Capture the return value and attach a rejection handler, or explicitly reject promise-like setters with a swallowed catch.
  - confidence: 0.88
