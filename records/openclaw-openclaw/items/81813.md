## Summary

Keep open: the PR addresses a real external-channel runtime handoff gap and supplies useful real-environment proof, but the latest head still leaves a public contract race by allowing an async setter while starting the account without waiting for that setter to finish.

## What This Changes

This PR adds an optional `ChannelGatewayAdapter.setChannelRuntime` hook, invokes it with the resolved task-scoped channel runtime before `startAccount`, and adds server-channel tests for ordering and setter failure handling.

## Best Possible Solution

Keep the new external runtime handoff, but make the setter contract match the lifecycle: either sync-only with defensive thenable rejection handling, or intentionally awaited before account startup with clear non-blocking failure behavior.

## Reproduction Assessment

Yes. Source inspection shows current main only provides the resolved channel runtime through `ctx.channelRuntime`, and the PR body gives a concrete WSL2 `openclaw-weixin` timeout-to-running proof path.

## Solution Assessment

No, not as-is. The hook is a narrow maintainable seam, but the type and implementation need to agree on whether asynchronous setters are supported before `startAccount` can rely on the runtime being set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Keep the runtime setter ordered with startAccount:** `src/channels/plugins/types.adapters.ts:344`
  - body: `setChannelRuntime` is now typed as returning `Promise<void>`, but the caller only attaches a catch and immediately queues `startAccount`. An external plugin that awaits before storing the runtime can still start with the old empty slot, despite the contract saying this hook runs before account start. Either keep the setter sync-only and treat thenables as defensive rejection handling, or await the promise before starting the account.
  - confidence: 0.82
