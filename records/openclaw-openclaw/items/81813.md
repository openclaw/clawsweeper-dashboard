## Summary

Close: current main already fixes the captured `api.runtime` failure with the lazy `PluginRuntime` proxy, and the PR's new gateway hook is unnecessary and riskier than the existing runtime plus `ctx.channelRuntime` contract.

## What This Changes

The PR adds an optional `gateway.setChannelRuntime(channelRuntime)` adapter hook, calls it before `startAccount`, and adds unit coverage for call order and thrown setters.

## Best Possible Solution

Keep the current lazy runtime proxy and per-account `ctx.channelRuntime`; add a new gateway hook only if a separate account-scoped plugin API gap is demonstrated.

## Reproduction Assessment

No, not against current main: the loader now passes a lazy runtime proxy into plugin registration, and a current test asserts the captured runtime exposes `channel`. The PR body's WSL2 logs are useful proof for the older beta behavior.

## Solution Assessment

No, the proposed hook is not the best way to solve this now. Current main fixes the root registration-runtime problem generically, while a plugin-level setter before each account start can add account-scoping ambiguity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep the runtime account-scoped:** `src/gateway/server-channels.ts:512-518`
  - body: `setChannelRuntime` is invoked on the shared `plugin.gateway` object without an account id before each account start. A multi-account channel that stores this value as a singleton can have one account overwrite another account's task-scoped runtime, so later runtime-context registrations may be tracked and disposed by the wrong account lifecycle; keep using `ctx.channelRuntime` or make any new hook explicitly account-scoped.
  - confidence: 0.82
