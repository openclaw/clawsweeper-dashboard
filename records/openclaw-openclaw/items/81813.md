## Summary

Keep this PR open. It targets a real external channel-runtime handoff gap and the supplied WSL2 logs are credible real behavior proof, but the latest branch still has a compile-blocking type mismatch in the defensive thenable handling and the async rejection test does not actually prove the manager catch path.

## What This Changes

The branch adds a sync `ChannelGatewayAdapter.setChannelRuntime` hook, calls it before channel `startAccount`, and adds gateway tests for setter ordering and failure handling.

## Best Possible Solution

Keep the new adapter hook, but make the defensive thenable handling typecheck cleanly and add a regression test that fails if the gateway omits its rejection catch.

## Reproduction Assessment

Yes. Source inspection shows current main only hands the resolved channel runtime to `startAccount` through `ctx.channelRuntime`, and the PR body supplies a concrete WSL2 external-plugin timeout-to-running proof path.

## Solution Assessment

No, not as-is. The hook is a narrow maintainable seam, but the latest branch needs a type-safe defensive catch pattern and meaningful async rejection coverage before merge.
