## Summary

Keep open: the reported external-channel runtime problem is plausible and the PR includes real proof, but the patch should not merge as-is because the new public hook has account/lifecycle ambiguity and the added tests are failing CI.

## What This Changes

The PR adds an optional `ChannelGatewayAdapter.setChannelRuntime` hook, calls it before `startAccount`, and adds server-channel tests for ordering and throw handling.

## Best Possible Solution

Expose a lifecycle-safe channel-runtime seam for external plugins, with account-aware semantics and regression coverage using a valid `ChannelRuntimeSurface`.

## Reproduction Assessment

No, not with a local live Weixin setup in this read-only review. Source inspection confirms current main only passes the runtime through `ctx.channelRuntime`, and the PR body supplies real WSL2 logs for the reported external-plugin startup path.

## Solution Assessment

No. The direction may be useful, but the current implementation publishes a task-scoped runtime through a shared adapter hook and the new tests use invalid runtime mocks that fail CI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make the runtime hook account-safe:** `src/gateway/server-channels.ts:512-518`
  - body: This calls the new setter on the shared `plugin.gateway` object with the task-scoped runtime for the account being started. Channel plugins can have multiple accounts and starts run with concurrency, so a later account can overwrite the runtime captured by a long-running loop for an earlier account; the scoped wrapper also owns `runtimeContexts` lease cleanup. Please make the hook account/lifecycle-safe, or keep this runtime on the per-account `startAccount` context.
  - confidence: 0.78
- **[P2] Use a real ChannelRuntimeSurface in the tests:** `src/gateway/server-channels.test.ts:1021`
  - body: The new test creates `runtimeContexts` as a `Symbol`, but `ChannelRuntimeSurface.runtimeContexts` must provide `register`, `get`, and `watch`; `check:test-types` already fails on this line, and the runtime shard reports the new setter assertions see zero calls. Use `createRuntimeChannel()` or a structural `runtimeContexts` mock in both new tests.
  - confidence: 0.94
