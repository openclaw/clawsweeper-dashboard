## Summary

Keep open. Current main still lacks the requested session-run cancel seam, but this PR head is not merge-ready: it exposes a core-owned cancel emitter through the public Plugin SDK, loses aborts for late handlers, wires a global requester in minimal gateways, and is dirty against current main. The later discussion helpfully links active remediation work in #75371 and a fork patch for the boundary/sticky-replay fixes.

## What This Changes

The PR adds a plugin-neutral session-run cancellation target and runtime subpath, wires gateway chat aborts to fan cancellation between core runs and plugin/delegated tasks, stacks a sessions_send delegation hook, updates SDK entrypoint metadata, and adds focused tests.

## Best Possible Solution

Land this as a narrow, plugin-neutral cancel seam after the branch is rebased/split cleanly: expose only plugin-safe observe/request helpers, persist and replay terminal cancellation for late subscribers, skip global requester wiring in minimal gateways, and align changelog/API artifacts with the final public SDK surface.

## Reproduction Assessment

Yes. The blockers are reproducible by static inspection of the PR head: the public runtime subpath exports `emitSessionRunCancel`, terminal cancellation is not stored for late `onSessionRunCancel` subscribers, and startup wires the requester outside the existing minimal-gateway guard pattern.

## Solution Assessment

No. The proposed seam is a good direction, but this PR head is not the safest implementation until the SDK boundary, sticky replay semantics, minimal-gateway guard, changelog, and dirty stacked dependency are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the core cancel emitter out of the plugin SDK:** `src/plugin-sdk/session-run-cancel-runtime.ts:12`
  - body: The public runtime subpath exports `emitSessionRunCancel`, which lets plugin code fire the core-to-plugin fan-out directly. Because the emitter clears its handler bucket, an accidental plugin call can consume handlers before the real core abort and leave delegated cleanup unnotified.
  - confidence: 0.92
- **[P1] Replay cancels for late session-run handlers:** `src/sessions/session-run-cancel.ts:82-94`
  - body: `emitSessionRunCancel` forgets cancellation when no handler is registered yet and deletes the bucket after dispatch. If a delegated runtime registers after the owning run is aborted, it never sees the abort and can keep running; store terminal cancel state long enough to replay it once.
  - confidence: 0.87
- **[P2] Skip requester wiring for minimal test gateways:** `src/gateway/server-startup-early.ts:124-134`
  - body: `startGatewayEarlyRuntime` wires the module-level requester even when `minimalTestGateway` is true. A minimal gateway can replace the process-global requester with test-local chat state, causing later plugin cancellation to route to the wrong gateway state.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `package.json:700-703`
  - body: The patch publishes a new Plugin SDK subpath and hook behavior for plugin authors, but it does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing feature/API changes with appropriate contributor credit.
  - confidence: 0.78
