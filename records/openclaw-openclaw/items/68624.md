## Summary

Keep open. Current main does not implement the cancel seam, so the PR is not obsolete, but the current PR head is not merge-ready because sticky terminal cancel state is retained without production cleanup, minimal gateway startup still wires the process-global requester, and the new public SDK subpath lacks a changelog entry. The discussion has useful follow-up context, including #75371 for part of the cleanup and the private A2A consumer smoke result, but those changes are not folded cleanly into this PR head.

## What This Changes

The PR adds a plugin-neutral `session_run` cancel seam, gateway abort/request wiring, a public Plugin SDK runtime subpath, stacked `sessions_send` delegation hook support, generated SDK metadata, and focused tests.

## Best Possible Solution

Land the neutral cancel seam after folding the lifecycle cleanup and gateway-startup boundary fixes into the latest branch: keep `emitSessionRunCancel` core-internal, bound or explicitly clean sticky terminal replay state from the run/session lifecycle, guard requester wiring for minimal gateways, add the public Plugin SDK changelog/API proof, and rebase the branch so #68622 owns the duplicated `sessions_send` hook changes.

## Reproduction Assessment

Yes. Static inspection of PR head `e32d6e55b5fdfcc2f4aaf69a94f81309376eb267` reproduces the blockers: `TERMINAL_CANCELS` is populated without a production cleanup call, `startGatewayEarlyRuntime` wires the global requester even under `minimalTestGateway`, and the added SDK export has no matching changelog entry.

## Solution Assessment

No. The plugin-neutral seam and current boundary direction are appropriate, but this is not yet the safest implementation until sticky replay state has a bounded lifecycle, minimal gateways stop claiming the global requester, and the public SDK addition is documented/changeloged.
