## Summary

Keep open: the reported API-key wording still exists on current main, but this PR does not yet pass the real provider auth mode from the embedded runner into the new billing-copy path and lacks real behavior proof.

## What This Changes

The PR adds OAuth-aware billing error formatting and helper plumbing, while also carrying unrelated config metadata and gateway bind-timeout changes.

## Best Possible Solution

Carry ResolvedProviderAuth.mode from the embedded runner auth state into both failover and payload error formatting, keep API-key wording for API-key auth, use OAuth-safe copy for OAuth auth, and cover the runner-level path with focused regression tests.

## Reproduction Assessment

Yes at source level: current main's billing formatter returns API-key top-up copy for billing-shaped errors, and the existing auth contract can distinguish oauth from api-key. I did not run a live Anthropic OAuth or Telegram scenario.

## Solution Assessment

No: the proposed helper API is plausible, but the real embedded-runner call sites still do not supply authMode, so the user-visible path remains unchanged. The safer fix is to derive authMode from apiKeyInfo/runtimeAuthState in run.ts and split or justify the unrelated changes.
