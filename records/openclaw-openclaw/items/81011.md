## Summary

Keep open: the updated branch now touches the runner call sites, but it still passes only `runtimeAuthState?.authMode`, which misses OAuth/token auth paths where no prepared runtime credential is created. The PR also still needs real behavior proof before merge and carries unrelated config/gateway changes that should be split or explicitly accepted.

## What This Changes

The PR adds OAuth-aware billing error copy and threads an optional auth mode through assistant error formatting, failover, payload construction, and the embedded run loop.

## Best Possible Solution

Carry the resolved auth mode from `runtimeAuthState?.authMode ?? apiKeyInfo?.mode` into both failover and payload error formatting, keep API-key wording for API-key auth, use OAuth-safe copy for OAuth/token account-limit cases, and split or explicitly accept the unrelated changes.

## Reproduction Assessment

Yes at source level: current main routes billing-shaped assistant errors through a formatter that always emits API-key top-up copy, while the runner already has resolved auth-mode data on `apiKeyInfo`. I did not run a live Anthropic OAuth or Telegram flow.

## Solution Assessment

No: adding OAuth-aware copy is the right direction, but the current branch uses only `runtimeAuthState?.authMode` and can miss OAuth/token paths that do not create prepared runtime auth. The safer narrow fix is to derive auth mode from `runtimeAuthState?.authMode ?? apiKeyInfo?.mode` at both runner call sites.

## Review Findings

- **[P2] Pass the resolved auth mode when runtime auth is absent:** `src/agents/pi-embedded-runner/run.ts:2382`
