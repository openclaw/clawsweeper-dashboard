## Summary

Keep open. The branch added the previously missing runner plumbing, but it passes `runtimeAuthState?.authMode`; Anthropic Claude CLI/OAuth can resolve as `apiKeyInfo.mode === "oauth"` while `runtimeAuthState` is null, so the reported path can still fall back to API-key billing copy. The PR is also draft, lacks real behavior proof, and still carries unrelated config and gateway changes.

## What This Changes

The PR adds an optional auth-mode argument to billing error formatting and threads it through embedded-runner failover/payload formatting, while also including unrelated config metadata and gateway bind-timeout commits.

## Best Possible Solution

Carry the existing resolved provider auth mode, preferably from `apiKeyInfo.mode` or a prepared run-level auth fact, into both failover and payload error formatting; keep API-key billing copy for API-key auth, use OAuth-safe copy for OAuth, add focused regression coverage, and split unrelated commits out of this PR.

## Reproduction Assessment

Yes, at source level. Current main formats billing-shaped assistant errors with API-key copy, and the PR still leaves `authMode` undefined for Anthropic Claude CLI/OAuth when `runtimeAuthState` is null; I did not run a live Max OAuth or Telegram reproduction in this read-only review.

## Solution Assessment

No. The helper API shape is reasonable, but the branch uses nullable runtime-auth exchange state instead of the resolved auth mode source of truth, so the reported OAuth path can remain unchanged; the safer fix is to derive auth mode from `apiKeyInfo.mode` or an equivalent prepared fact and remove unrelated commits.
