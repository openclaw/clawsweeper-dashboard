## Summary

Keep open: the fix direction is useful and the missing-provider proof is convincing, but the provider-available branch resolves Codex without first loading the standalone migration-provider registry, so an installed provider can still be treated as unavailable.

## What This Changes

The PR changes the Codex native-asset doctor warning to branch between direct `migrate codex --dry-run` guidance and a `doctor --fix` first hint based on Codex migration provider resolution, with tests for both paths.

## Best Possible Solution

Use the same provider-load sequence as `openclaw migrate`, then show the doctor-first wording only when Codex truly cannot be resolved.

## Reproduction Assessment

Yes. Source inspection shows the PR resolves the Codex migration provider directly from doctor warning collection, while the working migrate path first loads the standalone migration-provider registry.

## Solution Assessment

No. The UX branch is the right direction, but the implementation should use the same load-before-resolve contract as `openclaw migrate` so installed providers are not reported unavailable.
