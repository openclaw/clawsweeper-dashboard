## Summary

Keep open: the PR is a focused fix with real proof for the provider-missing case, but the new gate appears to false-negative the provider-available path because it resolves the Codex migration provider without the standalone registry load used by the migrate command.

## What This Changes

The PR changes the Codex native-asset doctor warning to choose between direct Codex migration guidance and a doctor --fix-first hint based on Codex migration provider availability.

## Best Possible Solution

Align the doctor warning gate with the migrate command by loading the standalone migration-provider registry before resolving Codex, then cover missing and available provider paths without mocking away that load contract.

## Reproduction Assessment

Yes, by source inspection: the PR calls `resolvePluginMigrationProvider` directly from doctor warning collection, while the working migrate path first loads the standalone migration-provider registry. I did not run a local command because this review is read-only and avoids artifact-generating test runs.

## Solution Assessment

No. The UX branch is the right direction, but the implementation should use the same load-before-resolve contract as `openclaw migrate` so installed providers are not reported unavailable.

## Review Findings
