## Label Justifications

- `impact:message-loss`: The reported stuck SETUP state prevents inbound channel messages from reaching the agent after QR login.

## Summary

Keep open. Current main does not prove the stuck-SETUP problem is fixed: `channels.start` still exists, but `channels login` still calls it after saving local auth, and the gateway can emit the reported `invalid channels.start channel` when the running process has no active registry entry for that external channel.

## What This Changes

The issue asks OpenClaw to stop leaving QR-login channels in SETUP after `openclaw channels login` saves auth and the running gateway rejects the post-save channel start call.

## Best Possible Solution

Define the supported post-login lifecycle for external channel plugins, then either reload/start through an existing gateway contract or detect the stale gateway registry case and tell users to restart without implying the channel start command is missing.

## Reproduction Assessment

No, not with high-confidence live proof. Source inspection shows the reported error path when the running gateway cannot normalize the channel id from its active registry, but the external WeChat plugin flow was not run in this read-only review.

## Solution Assessment

Unclear. Improving the CLI warning is a narrow fix, but making post-save login auto-start newly installed external plugins may change the gateway plugin reload/restart contract and needs maintainer direction.
