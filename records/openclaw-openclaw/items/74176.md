## Label Justifications

- `P2`: This is a normal-priority Telegram feature PR with a concrete security/product blocker but no evidence of an urgent shipped regression.
- `merge-risk: 🚨 compatibility`: Existing or current-main presentation `url` buttons in Telegram DMs would open as Mini Apps instead of ordinary links.
- `merge-risk: 🚨 security-boundary`: Mini App `web_app` launches expose Telegram WebApp data semantics to the destination origin, so automatic conversion broadens the trust boundary.

## Summary

Keep open for maintainer review. Current main and the latest release already preserve ordinary Telegram URL buttons and provide an explicit `webApp`/`web_app` presentation contract, while this PR's unique automatic conversion of HTTPS direct-chat links into Telegram Mini App buttons is a security-sensitive product change, not a conservative cleanup-close candidate.

## What This Changes

The PR carries shared presentation URL buttons into Telegram inline keyboards, auto-renders HTTPS direct-chat URLs as Telegram `web_app` Mini App buttons, and adds Telegram docs/tests for that behavior.

## Best Possible Solution

Refresh onto current main and keep ordinary `url` buttons as links; render Telegram Mini Apps only from the explicit `webApp`/`web_app` presentation contract unless maintainers intentionally approve a narrower opt-in or allowlisted conversion.

## Reproduction Assessment

Yes for the blocking review finding: PR-head source maps HTTPS `button.url` values to Telegram `web_app` whenever direct-chat mode is selected. Mantis screenshots also show the visible Telegram button after the patch, though they do not prove the Mini App security boundary is safe.

## Solution Assessment

No. Current main already has the narrower maintainable solution: ordinary `url` buttons stay links, and explicit `webApp`/`web_app` buttons become Telegram Mini App buttons.
