## Summary

Keep open: current main and v2026.6.10 still use UTC-shaped reset archive suffixes, and changing persisted transcript archive filenames is a compatibility-sensitive product decision rather than an already-implemented bug fix.

## What This Changes

The issue requests that reset/session archive filenames use a configured local timezone, or include a local-date prefix, instead of only UTC timestamp suffixes.

## Best Possible Solution

Have maintainers choose the archive filename contract, likely preserving the machine-readable UTC suffix while adding a compatible local-date search/display key or prefix if local-date discovery should be supported.

## Reproduction Assessment

Yes at source level: the reset archive writer appends a shared UTC ISO file stamp and the archive parser requires a trailing `Z`. I did not run the reporter's Windows/Asia-Shanghai reset scenario, so this is not a live reproduced bug path.

## Solution Assessment

Unclear: using a local-time suffix directly may break archive parsing and cleanup assumptions. A compatible design, such as a local-date prefix or search/display metadata while preserving the UTC suffix, needs maintainer product and upgrade review.
