## Summary

Keep open: current main and v2026.6.10 still use UTC-only reset/deleted archive filename stamps, and the matching implementation is an open, unmerged PR that still needs maintainer review/proof before this issue can close.

## What This Changes

The issue requests session archive/reset filenames that reflect a configured local timezone so local-date searches can find archived sessions.

## Best Possible Solution

Review and either land or reject the open candidate PR after maintainers choose the filename contract; if accepted, preserve the canonical UTC suffix and add local-date discoverability with old-format reader coverage.

## Reproduction Assessment

Yes, at source level: current main formats reset/delete archive names with a UTC ISO file stamp and does not pass `agents.defaults.userTimezone` into the archive writer. I did not live-run the Windows/Asia-Shanghai reset scenario.

## Solution Assessment

Unclear for final product direction: the open PR's local-date prefix is the safer compatibility shape than replacing the UTC suffix, but it still needs maintainer approval and real archive-run proof before landing.
