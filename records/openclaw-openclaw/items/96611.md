## Summary

Keep open: current `main` and the latest release still generate reset archive filenames from UTC ISO file stamps, while the request is a new compatibility-sensitive session archive filename behavior decision rather than an already implemented fix.

## What This Changes

Request that reset/session archive transcript filenames reflect the configured local timezone, or include a local-date prefix, instead of only UTC ISO suffixes.

## Best Possible Solution

Have maintainers choose the archive filename contract, likely preserving the canonical UTC suffix for machine parsing while adding a compatible local-date search/display key or prefix if local-date discovery should be supported.

## Reproduction Assessment

Yes at source level: the reset archive writer appends a shared UTC ISO file stamp and the archive parser requires a trailing `Z`. I did not run the reporter's Windows/Asia-Shanghai reset scenario, so this is not a live reproduced bug path.

## Solution Assessment

Unclear: using a local-time suffix directly may break archive parsing and cleanup assumptions. A compatible design, such as a local-date prefix or search/display metadata while preserving the UTC suffix, needs maintainer product and upgrade review.
