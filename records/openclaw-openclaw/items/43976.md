## Summary

Keep open: current main still supports static channel-to-agent route bindings, TUI-local `/agent` switching, and thread/session-focused `/agents` plus `/focus`, but it does not provide a generic channel command that temporarily switches the active configured agent for a direct conversation. The request is valid and security-sensitive, so it needs maintainer product/security direction before implementation.

## What This Changes

Request a channel command such as `/agents` and `/agent <name>` to switch the active agent for one conversation without editing `openclaw.json` or restarting Gateway.

## Best Possible Solution

Design an authorization-aware per-conversation active-agent state path that reuses routing/session-binding primitives and preserves per-agent session/workspace isolation without hot-editing global `bindings[]`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current channel commands do not provide `/agent <id>` active-agent switching for a direct conversation.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The safer direction is a generic, authorization-aware conversation/session routing design, not chat-driven mutation of global route bindings.
