## Summary

Keep open: current main and v2026.6.6 still support configured bindings, config hot reload, and thread/subagent `/focus` plus `/agents`, but not a chat command that temporarily switches the active configured agent for a direct conversation. The request is valid feature work, but it needs maintainer product/security direction because it changes who can route a channel sender into another agent boundary.

## What This Changes

Request a per-conversation `/agents` and `/agent <name>` command flow to switch the active channel-bound agent without restarting the Gateway or editing global bindings.

## Best Possible Solution

Design an authorization-aware per-conversation active-agent selection path that reuses existing routing/session-binding primitives, preserves per-agent workspace and memory isolation, and avoids hot-editing global bindings or adding ad hoc JSON state.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection give high confidence that current main lacks a `/agent <id>` command for direct active-agent switching.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The safer direction is a generic, authorization-aware conversation/session routing design rather than chat-driven mutation of global bindings or a new standalone override file.
