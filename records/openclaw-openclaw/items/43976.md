## Summary

Keep open: current main still lacks a channel-level `/agent <id>` command that temporarily switches a direct conversation to another configured agent. Existing surfaces cover thread-bound subagent/session listing, TUI-local agent selection, and model delegation tools, so the remaining work needs product and security design before implementation.

## What This Changes

Request a channel command to list configured agents and temporarily switch the current single-chat conversation to another agent without restarting Gateway.

## Best Possible Solution

Design an additive, opt-in conversation-level routing contract that reuses configured agent ids, preserves per-agent session and memory isolation, documents authorization and sandbox rules, and avoids mutating global bindings as the normal command path.

## Reproduction Assessment

Yes, at source level, but this is a feature gap rather than a bug repro. Current main has thread-bound `/agents`, TUI-only `/agent <id>`, configured binding/default routing, and subagent delegation tools, with no channel `/agent <id>` path that temporarily switches a direct chat to another configured agent.

## Solution Assessment

Unclear as proposed. A direct `/agent <id>` chat command may be the right UX, but it should be designed around authorization-aware conversation routing and existing subagent/session boundaries rather than hot-editing global bindings or writing ad hoc state files.
