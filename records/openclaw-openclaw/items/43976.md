## Summary

Keep open: current main and v2026.6.10 still do not provide a channel command that temporarily switches a direct conversation to another configured agent. Existing `/agents` and `/focus` are thread-bound subagent/session controls, TUI `/agent <id>` is a local UI command, and `agents_list`/`sessions_spawn` are model delegation tools, so the remaining work needs maintainer product and security design.

## What This Changes

Request a session-scoped `/agents` and `/agent <id>` command flow to list and temporarily switch the current channel conversation's configured agent without restarting Gateway.

## Best Possible Solution

Design an additive, opt-in conversation-level routing contract that reuses configured agent ids, preserves per-agent session and memory isolation, documents authorization and sandbox rules, and avoids mutating global bindings as the normal command path.

## Reproduction Assessment

Yes at source level, but this is a feature gap rather than a bug repro. Current main has thread-bound `/agents`, TUI-only `/agent <id>`, and configured binding/default routing, with no channel `/agent <id>` path that switches a direct chat to another configured agent.

## Solution Assessment

Unclear as proposed. A direct `/agent <id>` chat command may be the right UX, but it should be designed around authorization-aware conversation routing and existing subagent/session boundaries rather than hot-editing global bindings or writing ad hoc state files.
