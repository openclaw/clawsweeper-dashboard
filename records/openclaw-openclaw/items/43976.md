## Summary

Keep open. Current main still has adjacent `/agents`, `/focus`, route bindings, and plugin-specific dynamic-agent behavior, but not the requested channel-chat `/agent <name>` workflow for temporarily switching a direct conversation to another configured Agent without editing bindings or restarting Gateway.

## What This Changes

The issue requests session-level channel chat commands to list and temporarily switch the current channel conversation's active Agent.

## Best Possible Solution

Define a session-level configured-Agent override or conversation-binding path that reuses existing routing primitives, is gated like other control commands, documents its relationship to `/focus` and `sessions_spawn`, and adds direct-chat precedence coverage.

## Solution Assessment

Unclear. A dedicated `/agent <name>` workflow may be the right UX, but the maintainable solution needs a product decision on routing precedence, authorization, persistence, and how it differs from existing focus/session binding behavior.
