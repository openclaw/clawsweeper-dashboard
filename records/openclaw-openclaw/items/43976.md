## Summary

Keep open: current main has static binding-based Gateway routing, thread-bound subagent commands, and TUI-local agent selection, but it still does not provide a channel-level `/agent <id>` command that temporarily switches a direct conversation's active agent without restarting Gateway. This is a useful multi-agent/session-state feature, but it needs maintainer direction on routing authority, authorization, and state ownership before implementation.

## What This Changes

The issue requests channel/session commands to list agents and temporarily switch the active agent for a direct conversation without restarting Gateway.

## Best Possible Solution

Design a generic, policy-guarded conversation active-agent selection path that reuses session/binding primitives, lists only allowed agents, preserves per-agent workspace and memory isolation, and avoids hot-editing global bindings from chat.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and docs inspection confirm current channel commands do not provide `/agent <id>` active-agent switching for a direct conversation.

## Solution Assessment

Unclear until maintainers choose the product boundary. The safer path is a generic conversation/session binding design with authorization and persistent state rules, not chat-driven hot edits to global Gateway bindings.
