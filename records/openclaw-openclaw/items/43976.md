## Summary

Keep open: current main and v2026.6.6 still provide static configured route bindings plus thread/session-focused `/focus` and `/agents`, but not a direct-channel `/agent <id>` command that temporarily switches the active configured agent. The request remains a security- and session-state-sensitive feature that needs maintainer product/security direction before implementation.

## What This Changes

Request a direct-chat command flow to list configured agents and temporarily switch the current channel/conversation to another agent without restarting Gateway.

## Best Possible Solution

Design an authorization-aware per-conversation agent routing switch that uses canonical session or binding state, preserves per-agent workspace and memory isolation, and avoids hot-editing global `bindings[]` or adding ad hoc runtime JSON state.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current channel commands do not provide direct-conversation `/agent <id>` active-agent switching.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The safer direction is a generic, authorization-aware conversation routing design, not chat-driven mutation of global route bindings or a new unowned override store.
