## Summary

Keep open: current main has static gateway bindings, UI-specific agent selection, and thread-bound subagent commands, but it still does not provide a channel-level `/agent <id>` command that temporarily switches a direct conversation's active agent without restarting Gateway. The request is a valid multi-agent/session-state feature, but it needs maintainer direction on routing authority, authorization, and where the active-agent state belongs.

## What This Changes

Request a channel command flow that temporarily switches the current conversation to another configured agent without editing bindings or restarting Gateway.

## Best Possible Solution

Design a generic, policy-guarded conversation active-agent selection path that reuses session/binding primitives without mutating global bindings or requiring Gateway restart.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and docs inspection confirms current channel commands do not provide `/agent <id>` active-agent switching for a direct conversation.

## Solution Assessment

Unclear until maintainers choose the product boundary. The safer path is a generic conversation/session binding design with authorization and persistent state rules, not hot-editing global bindings from chat.
