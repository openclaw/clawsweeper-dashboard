## Summary

Keep open: current main and v2026.6.10 still lack the requested `/agent <id>` configured-agent switch. Existing `/agents` and `/focus` cover thread-bound subagent/session binding, not configured-agent listing or switching for a direct chat, so the remaining work needs explicit product and security decisions.

## What This Changes

The issue requests session-level commands to list configured agents and switch the current chat to another agent without restarting Gateway.

## Best Possible Solution

Design an authorization-aware conversation-level agent routing override on canonical session or binding state, with explicit target allowlists, workspace and memory isolation, security review, docs, and focused routing/command tests.

## Reproduction Assessment

Yes from source inspection: current `/agents` lists active or bound subagent sessions and current routing uses configured bindings/defaults. There is no `/agent <id>` command path that switches a chat to another configured agent.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The safer direction is an authorization-aware conversation-level routing override over canonical session or binding state, not hot-editing global bindings or adding ad hoc state files.
