## Summary

Current main still routes channel conversations through configured bindings and has no `/agent` or `/agents` command. The request is distinct from the related Control UI shortcut issue and should remain open for a maintainer product decision because agent identity is part of canonical session, transcript, memory, and delivery state.

## What This Changes

Add session-scoped commands to list configured agents and temporarily switch a direct conversation to another agent without restarting the gateway or changing global bindings.

## Best Possible Solution

Define one generic per-conversation agent-selection contract, then implement list, set, and reset commands through the canonical route/session-key and outbound delivery paths with explicit authorization, target-agent transcript ownership, and memory-isolation tests.

## Reproduction Assessment

Not applicable. Current main consistently routes channel messages through configured bindings and has no documented session-switch command, so this is missing capability rather than a reproducible bug.

## Solution Assessment

No. The proposed `sessions_spawn` forwarding approach would create a parallel routing path and blur memory ownership; a first-class override integrated into the existing route, session-key, and delivery contract is the narrowest maintainable solution.
