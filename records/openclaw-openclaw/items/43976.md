## Summary

Keep open: current main and v2026.6.6 still expose configured routing plus thread/session-focused `/focus` and `/agents`, but not a direct-channel `/agent <id>` command that temporarily switches the active configured agent. The request remains useful, but it crosses session ownership and agent authorization boundaries, so it needs maintainer product/security direction before implementation.

## What This Changes

The issue requests session-level commands for dynamically switching the active agent for a channel conversation without restarting the Gateway.

## Best Possible Solution

Design an authorization-aware per-conversation active-agent selection path that reuses existing routing/session-binding primitives, preserves per-agent workspace and memory isolation, and avoids hot-editing global `bindings[]` or adding ad hoc JSON state.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source inspection gives high confidence that current channel commands do not provide `/agent <id>` active-agent switching for direct conversations.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The safer direction is a generic, authorization-aware conversation/session routing design rather than chat-driven mutation of global route bindings or a new standalone override store.
