## Summary

Keep open. Current main still does not provide the requested external-chat workflow to list configured agents and temporarily switch one conversation to another configured agent without restarting the Gateway; the nearby `/agents` and `/focus` commands cover subagent/session bindings, not configured-agent route overrides by agent id.

## What This Changes

Request session-level chat commands to list configured agents and switch the current conversation to a selected configured agent without restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel configured-agent route override that lists available agents, stores a temporary per-conversation agent selection before static binding fallback, supports reset/expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a missing feature, and the absence is high-confidence from the command registry, `/agents` handler, route resolver, and docs inspection.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide command namespace and security policy. A session-level route override before static binding fallback looks narrower and safer than hot-editing global `bindings` from chat.
