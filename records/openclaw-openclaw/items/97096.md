## Summary

This PR is a focused fix for a real runtime-state bug, but it is not merge-ready: the new preservation predicate still conflates stripped refreshes with canonical no-web fast-path snapshots, can clear web metadata for unrelated plugin entries, and has no real gateway/config-write proof.

## What This Changes

Adds `webToolsFromFastPath` provenance to prepared secrets snapshots and changes secrets runtime activation to preserve active web-tool metadata across stripped fast-path refreshes, with regression test updates.

## Best Possible Solution

Carry explicit provenance that distinguishes stripped or partial refresh input from authoritative full configs, then prove both preservation and deletion paths in source tests and a real gateway/config-write run.

## Reproduction Assessment

Yes for source-level reproduction: current main can prepare empty web-tool metadata for a no-web/no-SecretRef source config, and activation publishes `next.webTools` into active state. I did not run tests in this read-only checkout.

## Solution Assessment

No. Activation-layer preservation is a plausible boundary, but this implementation infers stripped input from container presence instead of carrying explicit provenance, which leaves stale-retention and unrelated-plugin-entry clearing bugs.
