## Summary

Keep open. The PR targets a valid linked runtime bug, but the proposed activation heuristic is not safe yet: it can preserve stale web-tool metadata after a canonical config removes web surfaces, while still allowing unrelated plugin-entry stripped refreshes to clear the metadata; the external proof is also only targeted tests and formatting.

## What This Changes

The PR adds a `webToolsFromFastPath` provenance flag to prepared secrets snapshots and changes secrets runtime activation to preserve active web-tool metadata across stripped fast-path refreshes, with regression tests and fixture updates.

## Best Possible Solution

Carry explicit provenance that distinguishes stripped or partial refresh input from a canonical full no-web config, then preserve only ambiguous stripped fast-path refreshes while proving deletion and unrelated plugin-entry writes.

## Reproduction Assessment

Yes for source-level reproduction: the new predicate preserves any fast-path no-container snapshot after a prior full web config, and current fast-path code produces that shape for canonical no-web configs. I did not run tests in this read-only checkout.

## Solution Assessment

No. Activation-layer preservation is a plausible owner boundary, but the best fix needs explicit provenance separating stripped partial refreshes from authoritative full configs rather than inferring that from container presence.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep canonical no-web refreshes authoritative:** `src/secrets/runtime.ts:160-163`
  - body: This branch treats every fast-path snapshot without `tools.web` or `plugins.entries` as ambiguous stripped input. A real `prepareSecretsRuntimeSnapshot({ config: {} })` after a prior full web config also has that shape, so deleting web config would preserve the old active `web_search`/`web_fetch` metadata instead of clearing it. Please carry explicit stripped-refresh provenance, or otherwise keep canonical no-web snapshots authoritative.
  - confidence: 0.9
- **[P2] Do not make unrelated plugin entries clear web tools:** `src/secrets/runtime.ts:132-138`
  - body: `hasRuntimeWebToolMetadataContainer` returns true for any `plugins.entries`, but the fast path still allows plugin entries that do not contain `config.webSearch` or `config.webFetch`. A stripped refresh carrying only an unrelated plugin entry would bypass preservation and clear populated web metadata, leaving the reported config-write drop unfixed for that writer shape.
  - confidence: 0.84
