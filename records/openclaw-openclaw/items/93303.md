## Best Possible Solution

## Reproduction Assessment

Yes from source, though I did not run the test suite here: current main awaits `readMessages` during inbound finalization, and Baileys `readMessages` awaits privacy-query plus receipt socket work. A never-settling socket operation can therefore keep the read-receipt step pending.

## Solution Assessment

Yes with docs/help alignment: wrapping the existing read-receipt promise at the call site reuses the WhatsApp socket-operation timeout seam and preserves the existing non-fatal read-receipt catch. A new timeout knob would be broader than needed.
