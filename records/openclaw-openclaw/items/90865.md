## Label Justifications

- `P2`: The issue describes a normal-priority Discord inbound delivery bug with a full gateway restart workaround and limited blast radius.
- `impact:message-loss`: Inbound messages from newly configured Discord channels can be silently dropped until the Discord provider is restarted.

## Summary

Keep open: current main has the generic Discord channel hot-restart path, but the Discord inbound path still depends on a provider-startup guild/channel snapshot, so the reported silent drop remains plausible unless a focused repro proves the reload actually rebuilds that snapshot for newly added guild channels.

## What This Changes

The issue requests that Discord guild channel entries added to `channels.discord.guilds.*.channels.*` after gateway startup become live for inbound messages without a full gateway restart.

## Best Possible Solution

Keep the existing gateway reload ownership model and add focused proof or a repair showing that a `channels.discord.guilds.*.channels.*` edit reaches changed-path detection, restarts the Discord channel, and rebuilds the inbound guild/channel allowlist without a full gateway restart.

## Reproduction Assessment

No high-confidence full current-main reproduction was established. Source inspection proves the stale-snapshot failure mode if Discord channel restart is missed, but current main also has a generic `channels.discord.*` restart plan, so the missing link still needs focused harness or live proof.

## Solution Assessment

Unclear as reported. The narrow maintainable solution is to prove and repair the existing config reload to Discord channel restart path before adding a separate Discord admin reload command or reconnect-time config reread.
