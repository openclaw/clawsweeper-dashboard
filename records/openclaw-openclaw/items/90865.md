## Summary

Keep open: current main still has the generic Discord config hot-reload-to-channel-restart path, but the report describes plausible Discord inbound message loss and no source, release, or related-item evidence proves the reported post-startup subscription failure is fixed.

## What This Changes

The issue reports that adding a new enabled Discord guild channel to `openclaw.json` while Gateway is running does not make inbound messages from that channel live until a full gateway restart.

## Best Possible Solution

Keep the gateway reload ownership model, capture a focused Discord repro that shows whether the watcher, reload planner, channel stop/start, or monitor snapshot refresh is failing, then repair that path rather than adding reconnect-time config rereads.

## Reproduction Assessment

No. Source confirms Discord snapshots guild/channel allowlists at monitor startup and reconnect alone will not re-read config, but current main and v2026.6.6 also show `channels.discord.*` edits should restart the Discord channel, so the actual failing path still needs a focused repro.

## Solution Assessment

Unclear. Repairing the existing config watcher/reload/channel restart path is the maintainable direction if it fails; reconnect-time disk rereads or a new Discord reload command should wait for proof that the existing reload model cannot cover this case.
