## Summary

Keep open: the report describes a plausible Discord inbound-delivery gap, and current main still depends on the existing config watcher restarting the Discord channel to refresh the startup-only guild/channel allowlist; I did not find a later source/history/release fix or a canonical duplicate that safely closes the issue.

## What This Changes

The issue requests that Discord pick up newly added `channels.discord.guilds.*.channels.*` entries without requiring a full gateway restart.

## Best Possible Solution

Prove and repair the existing `channels.discord.*` hot-reload-to-channel-restart path so the Discord provider refreshes its allowlist after a config edit, instead of adding a parallel reconnect-time disk reread or Discord-specific reload command first.

## Reproduction Assessment

No live Discord repro was run; source inspection gives a focused repro target because the Discord provider keeps startup guild/channel config unless the generic config watcher plans and applies a Discord channel restart.

## Solution Assessment

The existing reload design is the best first repair target; the alternative requests in the issue would add new Discord-specific behavior before proving why the current generic restart path did not refresh the provider.
