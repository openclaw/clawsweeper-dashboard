## Summary

Keep open: the report describes a fresh 2026.6.1 Discord inbound-delivery failure, and current main/latest release still only show the generic config-change-to-Discord-channel-restart path rather than a focused fix or proof for newly added guild channel entries becoming live in the reported runtime path.

## What This Changes

The issue asks OpenClaw to make newly added `channels.discord.guilds.*.channels.*` entries receive inbound Discord events without requiring a full gateway restart.

## Best Possible Solution

Keep the existing gateway reload ownership model, then capture focused proof for a new `channels.discord.guilds.*.channels.*` entry showing the watcher event, changed path, reload plan, Discord stop/start, and post-restart inbound delivery; repair the failing segment or add diagnostics where the chain breaks.

## Reproduction Assessment

No high-confidence live/current-main reproduction is established. Source inspection shows that Discord snapshots guild channel config at provider startup and will drop a newly added channel if the configured channel restart does not happen, but the intended reload path should request that restart.

## Solution Assessment

Unclear. The best fix is probably to prove and repair the existing config watcher -> reload plan -> Discord channel restart path before adding reconnect-time config reads or a separate Discord reload command.
