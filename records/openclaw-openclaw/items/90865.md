## Summary

Keep open: current main has the generic Discord `channels.discord.*` hot-reload-to-channel-restart path, but there is still no focused current-main or live Discord proof that adding a new guild channel after startup actually becomes inbound-live without a full gateway restart.

## What This Changes

The issue reports that Discord channel entries added to `openclaw.json` after gateway startup can send outbound messages but are not subscribed for inbound events until a full gateway restart.

## Best Possible Solution

Prove the running-gateway Discord new-channel flow, then repair the existing hot-reload/restart path or add a targeted diagnostic where that path is skipped.

## Reproduction Assessment

No: the issue gives concrete steps, but this review did not run a live Discord repro or establish a focused current-main failing test for adding a new channel after startup. Source shows both a startup-scoped allowlist and an intended Discord hot-restart path, so the failing link needs direct proof.

## Solution Assessment

Unclear: repairing the existing config watcher/channel restart path is likely the right direction, but adding a separate `discord reload` command or reconnect-time disk reread would be premature without proving why the current path did not fire or did not rebuild the allowlist.
