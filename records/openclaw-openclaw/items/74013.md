## Summary

Keep open. This PR addresses the real Firecrawl self-hosting gap tracked by #63877, but the patch broadens endpoint trust in security-sensitive ways and overlaps with two other open Firecrawl self-hosting PRs that propose different endpoint policies.

## What This Changes

This PR adds Firecrawl `allowSelfHosted` config fields in the plugin manifest and config resolver, passes that flag into Firecrawl endpoint resolution, and adds tests for self-hosted base URLs.

## Best Possible Solution

Land one canonical Firecrawl self-hosting change for #63877 that keeps defaults strict, allows custom public Firecrawl hosts only over HTTPS, allows cleartext HTTP only after private/loopback validation, routes private endpoints through the trusted web-tools guard, and updates manifest, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Current main has a source-and-test reproduction for rejecting HTTP and custom-host Firecrawl base URLs, and the PR-specific regressions are reproducible from the diff by enabling `allowSelfHosted` on only one operation or by configuring a public `http://` Firecrawl base URL.

## Solution Assessment

No. The requested direction is valid, but this PR is not the narrowest maintainable solution because it treats self-hosting as a broad endpoint bypass instead of separating public HTTPS hosts from validated private HTTP targets and the matching guarded fetch mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restrict cleartext self-hosted endpoints to private hosts:** `extensions/firecrawl/src/firecrawl-client.ts:70-73`
  - body: When `allowSelfHosted` is true, `resolveEndpoint` accepts any `http://` Firecrawl base URL, and the request path sends `Authorization: Bearer ...` to that endpoint. Keep public custom hosts HTTPS-only and only allow HTTP after private/loopback validation.
  - confidence: 0.88
- **[P2] Scope self-hosted opt-in to the selected config branch:** `extensions/firecrawl/src/config.ts:202-206`
  - body: `resolveFirecrawlAllowSelfHosted` ORs the search and fetch configs, so opting one operation into self-hosted mode relaxes endpoint validation for the other operation too. Resolve the flag from the same branch as the selected base URL, or make it one explicit shared policy.
  - confidence: 0.86
- **[P2] Use the trusted guard for allowed private endpoints:** `extensions/firecrawl/src/firecrawl-client.ts:315-317`
  - body: The new resolver can produce `http://127.0.0.1...`, but `postFirecrawlJson` still uses `withStrictWebToolsEndpoint`, whose default guard blocks localhost/private targets. A configured local endpoint can pass validation and then fail at runtime.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust and can send bearer credentials to public cleartext HTTP endpoints, so it needs security review before merge.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:70`
  - body: With `allowSelfHosted` enabled, arbitrary `http://` hosts pass endpoint validation while Firecrawl requests include an Authorization bearer header. Public custom hosts should remain HTTPS-only, with HTTP limited to validated private/loopback targets.
  - confidence: 0.88
- **[medium] Self-hosted opt-in leaks between search and fetch:** `extensions/firecrawl/src/config.ts:202`
  - body: The new resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86
