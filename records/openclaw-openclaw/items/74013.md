## Summary

Keep this PR open for maintainer review, but do not merge it as-is: the requested Firecrawl self-hosting capability is valid and matches the open user report, yet this patch broadens endpoint trust in a way that can leak bearer credentials and also overlaps with other open Firecrawl self-hosting PRs that use a narrower private-network policy.

## What This Changes

This PR adds Firecrawl `allowSelfHosted` config fields to the plugin manifest/config resolver, passes that flag into Firecrawl endpoint resolution for search and scrape, and extends endpoint tests for self-hosted URLs.

## Best Possible Solution

Land one canonical Firecrawl self-hosting change for #63877 that keeps defaults strict, permits custom public Firecrawl endpoints only over HTTPS, permits cleartext HTTP only after private/loopback validation, routes allowed private endpoints through the trusted web-tools guard, scopes policy to search versus fetch deliberately, and updates manifest, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Current main has source and test coverage showing HTTP and custom-host Firecrawl base URLs are rejected, and the PR-specific regressions are reproducible from the diff by enabling `allowSelfHosted` for an arbitrary public `http://` endpoint or for only one operation branch.

## Solution Assessment

No. The direction is useful, but this patch is not the best solution because it treats self-hosting as a broad endpoint bypass instead of separating public HTTPS custom hosts from validated private HTTP targets and the matching trusted fetch guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restrict cleartext self-hosted endpoints to private hosts:** `extensions/firecrawl/src/firecrawl-client.ts:70-73`
  - body: When `allowSelfHosted` is true, `resolveEndpoint` accepts any `http://` Firecrawl base URL while `postFirecrawlJson` sends an `Authorization: Bearer ...` header. Keep custom public hosts HTTPS-only and allow HTTP only after validating that the target is private or loopback.
  - confidence: 0.88
- **[P2] Scope self-hosting to the selected config branch:** `extensions/firecrawl/src/config.ts:202-206`
  - body: `resolveFirecrawlAllowSelfHosted` ORs the search and fetch configs, so opting one operation into self-hosted mode can relax endpoint validation for the other. Resolve the flag from the same branch as the selected endpoint, or make the shared policy explicit.
  - confidence: 0.86
- **[P2] Use the trusted guard for private Firecrawl endpoints:** `extensions/firecrawl/src/firecrawl-client.ts:315-317`
  - body: The new resolver can return `http://127.0.0.1...`, but the request still flows through the strict web-tools guard, which blocks localhost and private targets. After private-target validation, use the trusted endpoint guard for those self-hosted requests.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust and can send bearer credentials to cleartext public endpoints, so it needs security-focused maintainer review before merge.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:70`
  - body: With `allowSelfHosted` enabled, arbitrary `http://` hosts pass endpoint validation while Firecrawl requests include an Authorization bearer header. Public custom hosts should remain HTTPS-only, with HTTP limited to validated private or loopback targets.
  - confidence: 0.88
- **[medium] Self-hosted opt-in leaks between operations:** `extensions/firecrawl/src/config.ts:202`
  - body: The new resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86
