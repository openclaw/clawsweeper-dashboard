## Summary

Keep open: the PR addresses a real Firecrawl self-hosting gap tracked by #63877, but the current patch is not mergeable because it permits arbitrary cleartext Firecrawl endpoints with bearer credentials, leaks the opt-in across search and fetch config, and still routes private self-hosted calls through the strict web-tools guard. The overlapping #59666 and #63941 proposals should be resolved into one maintainer-approved endpoint policy.

## What This Changes

The PR adds Firecrawl `allowSelfHosted` config fields to the plugin manifest and config resolver, then passes that flag into endpoint resolution so non-official Firecrawl base URLs can be accepted.

## Best Possible Solution

Land one canonical Firecrawl self-hosting change for #63877 that preserves strict defaults, allows custom public Firecrawl hosts only over HTTPS, allows cleartext HTTP only after private/loopback validation, routes private self-hosted endpoints through the trusted web-tools guard, and updates manifest, docs, changelog, and runtime tests together.

## Reproduction Assessment

Yes. Current-main source and tests reproduce the existing HTTP/custom-host rejection path, and the PR-specific regressions are source-reproducible by enabling `allowSelfHosted` for one operation and following the resulting arbitrary HTTP or private endpoint through the unchanged strict request guard.

## Solution Assessment

No. The opt-in flag is aimed at the right user problem, but the maintainable fix should be a narrower Firecrawl endpoint policy modeled on the existing SearXNG private-HTTP validation and trusted guard path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict cleartext self-hosted endpoints to private targets:** `extensions/firecrawl/src/firecrawl-client.ts:73`
  - body: When `allowSelfHosted` is true, this accepts any `http://` Firecrawl base URL, and the request path sends an `Authorization` bearer header to that URL. Public custom hosts should remain HTTPS-only; cleartext HTTP should be limited to validated loopback/private targets.
  - confidence: 0.88
- **[P2] Scope allowSelfHosted to the selected Firecrawl config:** `extensions/firecrawl/src/config.ts:203`
  - body: `resolveFirecrawlAllowSelfHosted` ORs `webSearch` and `webFetch`, so opting one operation into self-hosted mode relaxes endpoint validation for the other operation too. Resolve the flag from the same config branch as the base URL, or make it one explicit shared policy.
  - confidence: 0.86
- **[P2] Route private self-hosted calls through the trusted guard:** `extensions/firecrawl/src/firecrawl-client.ts:315`
  - body: The new resolver can return a localhost/private HTTP endpoint, but `postFirecrawlJson` still uses `withStrictWebToolsEndpoint`, whose default SSRF policy blocks private targets. The advertised local self-hosted endpoint can pass resolver validation and still fail at runtime.
  - confidence: 0.83
- **[P3] Document and changelog the new endpoint config:** `extensions/firecrawl/openclaw.plugin.json:51-52`
  - body: This adds an operator-facing Firecrawl config knob, but the current docs still say Firecrawl base URLs must use the official HTTPS host and the PR has no changelog entry. Update the docs and `CHANGELOG.md` with the final policy.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust in a way that can send bearer credentials to arbitrary cleartext HTTP hosts and can let one operation's self-hosting opt-in relax another operation.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:73`
  - body: With `allowSelfHosted` true, the resolver accepts arbitrary `http://` hosts while `postFirecrawlJson` sends `Authorization: Bearer ...`; cleartext should be constrained to validated private/loopback targets.
  - confidence: 0.88
- **[medium] Self-hosted opt-in leaks across search and fetch:** `extensions/firecrawl/src/config.ts:203`
  - body: The resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86
