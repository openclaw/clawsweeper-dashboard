## Summary

Keep open: the Firecrawl self-hosting gap is still present on current main, but this PR is not safe to merge as written because it turns self-hosting into a broad endpoint-trust bypass, leaks the opt-in across search and fetch config, and does not align the network guard used for private endpoints.

## What This Changes

The PR adds Firecrawl `allowSelfHosted` config fields to the plugin manifest/config resolver, passes that flag into Firecrawl search and scrape endpoint resolution, and updates endpoint tests to allow self-hosted URLs when opted in.

## Best Possible Solution

Land one canonical Firecrawl self-hosting change that keeps defaults strict, allows custom public Firecrawl hosts only over HTTPS, allows cleartext HTTP only after private/loopback validation, scopes the opt-in to the selected config branch or makes it an explicit shared policy, routes private endpoints through the trusted web-tools guard, and updates manifest, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Current main has a source-and-test reproduction for rejecting HTTP and custom-host Firecrawl base URLs, and the PR regressions are visible from the diff by configuring `allowSelfHosted` with a public `http://` base URL or enabling the flag on only one of search/fetch.

## Solution Assessment

No. The requested direction is valid, but this PR is not the narrowest maintainable fix because it broadens endpoint trust instead of separating HTTPS custom hosts from validated private HTTP targets and the matching guarded fetch mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restrict HTTP self-hosting to private hosts:** `extensions/firecrawl/src/firecrawl-client.ts:70-73`
  - body: When `allowSelfHosted` is true, this condition accepts any `http://` Firecrawl base URL. Firecrawl requests include an Authorization bearer header, so public custom hosts should remain HTTPS-only and HTTP should be allowed only after private/loopback validation.
  - confidence: 0.88
- **[P2] Scope self-hosted opt-in to the selected config branch:** `extensions/firecrawl/src/config.ts:202-206`
  - body: `resolveFirecrawlAllowSelfHosted` ORs the search and fetch configs, so opting one operation into self-hosted mode relaxes endpoint validation for the other operation too. Resolve the flag from the same branch as the selected base URL, or make it one explicit shared policy.
  - confidence: 0.86
- **[P2] Use the trusted guard for allowed private endpoints:** `extensions/firecrawl/src/firecrawl-client.ts:315-317`
  - body: The new resolver can produce `http://127.0.0.1...`, but `postFirecrawlJson` still uses `withStrictWebToolsEndpoint`, whose strict guard blocks localhost/private targets. Route validated private endpoints through the trusted web-tools endpoint helper or an equivalent guarded path.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust and can expose bearer credentials unless the self-hosting policy is narrowed before merge.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:70`
  - body: With `allowSelfHosted` enabled, arbitrary `http://` hosts pass endpoint validation while Firecrawl requests include an Authorization bearer header. Public custom hosts should remain HTTPS-only, with HTTP limited to validated private/loopback targets.
  - confidence: 0.88
- **[medium] Self-hosted opt-in leaks between search and fetch:** `extensions/firecrawl/src/config.ts:202`
  - body: The new resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86

## Risks / Open Questions

- With `allowSelfHosted` enabled, the PR can send Firecrawl bearer credentials to arbitrary public cleartext HTTP endpoints.
- The new `allowSelfHosted` resolver lets a fetch-only opt-in relax search endpoint validation, and vice versa.
- Local/private self-hosted endpoints can pass the new resolver but still fail at runtime because Firecrawl requests continue through the strict web-tools endpoint guard.
- The PR overlaps with #63877, #59666, and #63941, so merging one branch without choosing a canonical endpoint policy risks conflicting fixes.
