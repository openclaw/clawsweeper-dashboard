## Summary

Keep this PR open for maintainer/security review. The self-hosted Firecrawl direction addresses a real open bug, but the patch broadens credentialed endpoint trust too much and overlaps other open attempts to solve the same Firecrawl self-hosting policy.

## What This Changes

The PR adds Firecrawl `allowSelfHosted` config fields, threads that flag into Firecrawl endpoint resolution for search and scrape, and extends endpoint tests for opted-in self-hosted URLs.

## Best Possible Solution

Land one canonical Firecrawl self-hosting fix for #63877 that keeps defaults strict, permits custom public Firecrawl endpoints only over HTTPS, permits cleartext HTTP only after private or loopback validation, routes validated private endpoints through the trusted web-tools guard, scopes search versus fetch policy deliberately, and updates tests, docs, manifest schema, and changelog together.

## Reproduction Assessment

Yes. The regressions are reproducible from the PR diff by enabling `allowSelfHosted` for an arbitrary `http://` endpoint with a Firecrawl API key, and by using `http://127.0.0.1` where endpoint resolution succeeds but the strict web-tools guard still blocks the private target.

## Solution Assessment

No. The requested capability is valid, but this patch is not the best fix because it turns self-hosting into a broad endpoint bypass instead of separating public HTTPS custom hosts from validated private HTTP targets and the matching trusted fetch guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict cleartext self-hosted endpoints to private hosts:** `extensions/firecrawl/src/firecrawl-client.ts:70-73`
  - body: When `allowSelfHosted` is true, `resolveEndpoint` accepts any `http://` Firecrawl base URL, and `postFirecrawlJson` sends `Authorization: Bearer ...` to that URL. Keep public custom hosts HTTPS-only and allow cleartext HTTP only after private or loopback validation.
  - confidence: 0.89
- **[P2] Scope self-hosting to the selected config branch:** `extensions/firecrawl/src/config.ts:199-204`
  - body: `resolveFirecrawlAllowSelfHosted` ORs the search and fetch configs, so opting one operation into self-hosted mode can relax endpoint validation for the other. Resolve the flag from the same branch as the selected endpoint, or make the shared policy explicit and documented.
  - confidence: 0.86
- **[P2] Use the trusted guard for private Firecrawl endpoints:** `extensions/firecrawl/src/firecrawl-client.ts:315`
  - body: The new resolver can return private targets such as `http://127.0.0.1:8787`, but the request still flows through `withStrictWebToolsEndpoint`, which keeps private-network SSRF blocking. After private-target validation, route those self-hosted requests through the trusted endpoint guard.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust and can send bearer credentials to cleartext public endpoints, so it needs security-focused maintainer review before merge.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:70`
  - body: With `allowSelfHosted` enabled, arbitrary `http://` hosts pass endpoint validation while Firecrawl requests include an Authorization bearer header. Public custom hosts should remain HTTPS-only, with HTTP limited to validated private or loopback targets.
  - confidence: 0.89
- **[medium] Self-hosted opt-in leaks between operations:** `extensions/firecrawl/src/config.ts:199`
  - body: The new resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86
