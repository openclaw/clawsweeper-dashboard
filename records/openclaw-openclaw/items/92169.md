## Summary

Keep open: current main still has a source-level path where a `message_tool_only` source reply can terminate the embedded runner after a visible progress or promise send, and the incomplete-turn retry guards skip replay once messaging delivery evidence exists. Related open repair and docs PRs are useful context, but no merged/current-main fix owns this exact Slack/Anthropic promised-work stall yet.

## What This Changes

The issue reports a regression where embedded Anthropic agents on Slack send a visible interstitial promise through the message tool and then end the turn without running the promised follow-up work.

## Best Possible Solution

Define a focused embedded-runner contract for progress-versus-final visible sends, then preserve final source-reply termination while letting progress/promise message-tool sends continue the same turn with Slack/Anthropic proof and focused retry/duplicate-delivery tests.

## Reproduction Assessment

Yes, at source level with medium confidence: `message_tool_only` installs a terminal after-tool hook for qualifying message sends, and the incomplete-turn retry path skips recovery once messaging delivery evidence exists. I did not run a live Slack/Anthropic reproduction in this read-only review.

## Solution Assessment

No, current main is not the best end state because it treats all qualifying implicit source message sends as terminal delivery, including text that can be only a progress promise. The safer repair should preserve final-reply termination while adding a tested continuation or routing path for progress-only and promise-only sends.
