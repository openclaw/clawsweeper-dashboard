## Label Justifications

- `P1`: The report describes a high-frequency regression that stalls real Slack agent workflows until the user manually re-prompts.
- `impact:message-loss`: The user receives an interstitial message but the promised final result/work output is never delivered in that turn.
- `impact:session-state`: The embedded session is marked done after a progress-like message tool call even though the agent has not completed the requested work.

## Summary

Keep open. Current main and the latest checked release still intentionally terminate a `message_tool_only` turn after a delivered implicit `message(action=send)`, which matches the reported interstitial-stall path; the fix needs owner judgment because that terminal hook was merged to prevent duplicate message-tool replies.

## What This Changes

The issue asks the embedded runtime to continue promised tool work after an interstitial Slack-visible `message(action=send)` instead of ending the turn there.

## Best Possible Solution

Define a safe final-vs-progress message-tool contract for `message_tool_only` source turns, then update the embedded runner and focused tests so interstitial promise sends continue while final delivered replies still stop duplicate loops.

## Reproduction Assessment

Yes, from source inspection rather than a live Slack run. A single delivered implicit `message(action=send)` in `message_tool_only` mode returns `terminate: true`, and agent-core then stops instead of invoking the model again.

## Solution Assessment

Unclear until maintainers choose the runtime contract. The current terminal hook is not enough for interstitial promise sends, but simply removing it would conflict with the merged duplicate-reply fix in https://github.com/openclaw/openclaw/pull/84437.
