## Summary

Keep this PR open: the patch addresses a real Langfuse OTel gap, is focused, and the follow-up run-id keying fixes the earlier cross-stamping concern, but it still needs maintainer acceptance of making already-captured prompt/response content visible on Langfuse trace and root-run observation surfaces.

## What This Changes

The PR mirrors captured model-call input/output onto the owning `openclaw.run` span for Langfuse trace-level and root-observation display, with a regression test for sibling runs sharing one OTel trace.

## Best Possible Solution

Merge the run-id-keyed projection only after maintainers explicitly accept the trace/root content visibility expansion under the existing `captureContent` opt-in and CI is clean enough for the branch.

## Reproduction Assessment

Yes. Source inspection shows current `main` exports captured model I/O only on `openclaw.model.call` spans, and the contributor supplied live Langfuse proof that root trace input/output were null before the patch and populated afterward.

## Solution Assessment

Yes, subject to maintainer policy acceptance. Keying root-run stamping by `runId` uses the existing trusted span owner model and the added sibling-run test covers the prior trace-id cross-stamping failure; a separate opt-in would be safer only if maintainers want trace/root visibility to differ from existing `captureContent`.
