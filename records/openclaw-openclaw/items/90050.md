## Summary

Keep open: the useful run-id approach is still blocked because the PR deletes the root run span lookup when synchronous `run.completed` handles, before queued `model.call.completed` events can stamp root I/O, and the broader captured-content visibility still needs maintainer acceptance.

## What This Changes

The PR mirrors captured diagnostics-otel model-call input/output onto root `openclaw.run` spans for Langfuse trace-level and root-observation display, with sibling-run isolation coverage.

## Best Possible Solution

Keep the run-id projection, but preserve root span tracking until queued model-call completions have stamped or drained, then merge only after maintainers explicitly accept the root trace/root observation captured-content visibility.

## Reproduction Assessment

Yes, source-backed. Current main only attaches captured content to model-call spans, and the PR/source path shows a normal sequence can enqueue `model.call.completed` then synchronously handle `run.completed` before the async drain.

## Solution Assessment

No, not yet. Run-id ownership is the right boundary, but deleting root-run tracking at `run.completed` before queued model-call drain is not the best fix; the safer solution stamps before terminal cleanup or defers cleanup until the queue drains.
