## Summary

Keep this PR open, but the current patch is not merge-ready because it keys root-run I/O stamping only by OTel trace id and can write one sibling run's captured prompt/response onto another run span.

## What This Changes

The PR adds diagnostics-otel logic that mirrors model-call input/output onto the `openclaw.run` span for Langfuse trace and observation I/O display.

## Best Possible Solution

Track per-run observation I/O by the diagnostic run span/owner, while separately keeping any intended single trace-level Langfuse summary behavior explicit and covered by sibling-run tests.

## Reproduction Assessment

Yes for the reported single-run Langfuse symptom at source/proof level: current main only puts model I/O on model-call spans, and the PR body includes a live Langfuse API before/after check. I did not independently run Langfuse, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No. Mirroring I/O is the right owner boundary, but using only the OTel trace id is too broad for OpenClaw's existing sibling-run trace topology; the fix should identify the specific run span for observation-level I/O.

Full review comments:

- **[P1] Key run I/O stamping by the run span, not the trace:** `extensions/diagnostics-otel/src/service.ts:2947`
  - body: This lookup is keyed only by OTel trace id, but OpenClaw already supports multiple sibling `openclaw.run` spans in the same trace.
