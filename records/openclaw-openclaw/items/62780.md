## Summary

Keep open. Current main now handles a large part of the observable failure with built-in empty/reasoning/planning result classification that can advance model fallback, but it still does not expose the requested content-quality hook that can inspect completed content, modify or reject it, and turn plugin rejection into a configured fallback attempt.

## What This Changes

Request a post-LLM, pre-delivery hook that can inspect, modify, or reject reply content and route rejected content into model fallback.

## Best Possible Solution

Define a documented, bounded final-reply decision seam before terminal acceptance that exposes content, provider/model, finish reason, usage, and run/session context, and maps replay-safe rejection into the existing fallback chain.

## Reproduction Assessment

Unclear. The report gives a concrete model/channel scenario and current tests reproduce generic empty-result fallback behavior, but there is no deterministic Gemini prompt or captured log; code inspection does verify the requested hook contract is absent.

## Solution Assessment

Unclear. A literal delivery-stage `message:before_send` hook is probably not the safest final shape because late cancellation may occur after streaming or side effects; a runner/finalization decision point before terminal acceptance is the narrower maintainable direction.
