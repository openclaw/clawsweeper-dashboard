## Summary

Keep open. Current main now has built-in replay-safe empty-result recovery and fallback classification for terminal failures, including Gemini-style empty turns, but it still does not provide the requested plugin-facing content-quality gate with provider/finish metadata and reject-to-fallback semantics.

## What This Changes

Requests a post-completion, pre-delivery hook that can inspect reply content and model metadata, modify or reject it, and make rejection trigger configured model fallback.

## Best Possible Solution

Define a documented final-result quality gate before terminal acceptance that exposes reply content plus provider/model metadata and maps replay-safe rejection into the existing model fallback classifier.

## Reproduction Assessment

No high-confidence live reproduction of the Gemini/WhatsApp empty-content behavior is available from the issue data. Source inspection does show the remaining feature gap: no plugin hook can reject completed content into configured fallback on current main.

## Solution Assessment

Unclear. A literal delivery-stage message:before_send hook is likely too late for safe fallback; a final-result classification seam before terminal acceptance is the narrower maintainable direction.
