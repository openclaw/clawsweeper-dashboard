## Summary

Keep open. Current main now covers part of the report with built-in empty/no-visible-result classification that can drive model fallback, but it still does not expose the requested plugin hook that can inspect completed content, return pass/modify/reject, and turn a plugin rejection into a configured fallback attempt.

## What This Changes

Request a pre-delivery or pre-terminal message hook that can pass, modify, or reject completed model output and route rejected content into the model fallback chain.

## Best Possible Solution

Define a documented, bounded plugin/runner decision point before terminal reply acceptance that exposes content, model/provider, finish/stop reason, usage, run/session context, and a decision result; map replay-safe rejection into the existing fallback chain while leaving `message_sending` as delivery rewrite/cancel policy.

## Reproduction Assessment

Unclear. The report gives a concrete provider/model/channel, and current tests cover empty terminal results, but there is no deterministic Gemini prompt or log here; code inspection is enough to verify the requested hook contract is absent.

## Solution Assessment

Unclear. A delivery-stage `message:before_send` hook is probably not the safest final shape because cancellation after streaming or side effects cannot always be retried safely; a runner/finalization decision seam before terminal acceptance is the narrower maintainable direction.
