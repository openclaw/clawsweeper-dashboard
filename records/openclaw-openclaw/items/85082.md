## Summary

Keep open. Current main still has a source-level path where a subagent announce/recovery turn runs inside the requester Telegram topic session and can write fallback provider/model usage into the same session fields that `/status` renders as user-facing model state; the related issues are adjacent observability or delivery work, not a canonical replacement for this attribution bug.

## What This Changes

The issue asks OpenClaw to keep internal subagent announce fallback usage from making Telegram topic `/status` or activity look like the user-facing session switched models.

## Best Possible Solution

Make session usage/status provenance-aware so subagent announce or other internal inter-session runs cannot overwrite user-facing topic model/fallback fields, while exposing internal last-run usage separately if maintainers want that visibility.

## Reproduction Assessment

Yes at source level: the announce path runs the internal handoff on the requester session, and the normal reply/session usage path can persist the fallback provider/model fields that `/status` reads. I did not establish a live Telegram failure path in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to carry provenance through usage/status and separate internal announce model usage from user-facing session model state, rather than closing under broader fallback observability work.
