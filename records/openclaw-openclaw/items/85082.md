## Summary

Keep open. Current main still has a credible source-level path where an inter-session subagent announce runs on the requester Telegram topic session and can update the same model/fallback fields that `/status` presents as user-facing session state; the related fallback observability issues are adjacent but do not own this specific internal-announce attribution bug.

## What This Changes

The issue reports ambiguous Telegram topic status/model attribution after an internal subagent announce or recovery turn uses a fallback model.

## Best Possible Solution

Make session usage/status provenance-aware so inter-session `subagent_announce` or recovery model usage is either labeled as internal last-run data or excluded from the user-facing topic model/fallback fields.

## Reproduction Assessment

Yes at source level: the current announce path runs the internal handoff on the requester session, then normal reply/session usage code can persist fallback provider/model fields that `/status` reads. I did not establish a live Telegram failure path in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to carry provenance into usage/status persistence and separate internal announce model usage from user-facing session model state, rather than closing this as a broad fallback observability duplicate.
