## Summary

Keep open. Current main still has a credible source-level path where a subagent announce runs in the requester Telegram topic session and non-heartbeat fallback usage can update the same provider/model and fallback fields that `/status` renders as the user-facing session state.

## What This Changes

The issue requests that Telegram topic `/status` and activity distinguish user-facing session model state from internal subagent announce/recovery fallback model usage.

## Best Possible Solution

Make session usage/status provenance-aware so subagent_announce and similar internal inter-session runs are either excluded from user-facing model/fallback fields or shown as clearly labeled internal last-run usage, with focused regression coverage.

## Reproduction Assessment

Yes at source level: the current announce path runs an internal handoff on the requester session, and the normal usage/fallback persistence path can write the provider/model fields that `/status` reads. I did not establish a fresh live Telegram failure path in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to carry provenance into session usage/status and separate internal announce model usage from user-facing topic model state. The broader fallback observability issues are adjacent but do not fully own this concrete Telegram attribution bug.
