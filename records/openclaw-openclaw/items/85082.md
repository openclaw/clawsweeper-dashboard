## Summary

Keep open. Current main still has a source-level path where an internal subagent announce or recovery run can execute on the requester Telegram topic session and write fallback model state into the same session fields that `/status` renders, while the related open fallback-observability items do not specifically own this Telegram/internal-announce attribution bug.

## What This Changes

The issue asks OpenClaw to keep Telegram topic `/status` and activity model attribution from presenting an internal subagent announce fallback model as the user-facing topic session model.

## Best Possible Solution

Make session usage and status provenance-aware so `subagent_announce` and other internal inter-session runs do not overwrite the user-facing topic model fields, while still exposing internal fallback usage as separate diagnostic state when useful.

## Reproduction Assessment

Yes at source level: current main runs subagent announce/recovery on the requester session, can persist auto fallback selection and non-heartbeat provider/model usage to that session, and `/status` reads those fields. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to carry provenance into session usage/status and keep internal announce fallback usage separate from user-facing topic model state, rather than closing this under broader fallback observability work.
