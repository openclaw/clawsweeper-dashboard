## Summary

Keep open. Current main still has a credible source-level path where a subagent announce/recovery turn runs on the requester Telegram topic session, then normal fallback and usage persistence can update the same model fields that `/status` renders as user-facing session state; the related fallback/status issues are adjacent but do not own this Telegram internal-announce attribution bug.

## What This Changes

The issue requests that Telegram `/status` and activity distinguish the normal topic session model from fallback model usage by internal subagent announce/recovery turns.

## Best Possible Solution

Make session usage/status provenance-aware so `subagent_announce` internal inter-session runs are excluded from or separately labeled from the user-facing Telegram topic model/fallback fields, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current main routes subagent announce handoffs through the requester session and the ordinary usage/fallback persistence path can update the same fields `/status` reads. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to carry provenance through usage/status persistence and separate internal announce model usage from user-facing session model state; closing under the broader fallback observability issues would leave this concrete bug unowned.
