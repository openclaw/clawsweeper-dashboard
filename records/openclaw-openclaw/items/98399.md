## Summary

Keep open: current main and v2026.6.11 still use the Azure deployment bridge path that the report describes, and there is no current implementation evidence for following or safely handling the Azure WebSocket redirect. The broader Azure Foundry Realtime Talk issue at https://github.com/openclaw/openclaw/issues/87325 overlaps, but this issue has a narrower source-reproducible failure against the existing Azure bridge path.

## What This Changes

The issue reports Azure OpenAI Realtime Talk failing because the gateway WebSocket client does not follow or otherwise handle an Azure 302 redirect during the WebSocket upgrade handshake.

## Best Possible Solution

Implement provider-owned Azure Realtime endpoint/auth handling in the OpenAI plugin, preferably constructing the documented Azure endpoint shape directly or using a tightly bounded same-resource redirect path, with regression tests, docs if user-facing config changes, and live Azure proof when available.

## Reproduction Assessment

Yes from source inspection, but not from a live Azure run: the reported config enters the `azureEndpoint && azureDeployment` branch, current main builds the reported preview-style URL, and ws defaults redirect following off. The exact Azure 302 response still needs live Azure proof before landing a fix.

## Solution Assessment

No, the issue's broad option to follow redirects is only a plausible fix, not automatically the best one. The safer solution is provider-owned Azure endpoint/auth handling or a narrowly constrained Azure redirect retry that preserves the server-side credential boundary.
