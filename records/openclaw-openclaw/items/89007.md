## Label Justifications

- `P1`: The reported orphan ACPX processes can accumulate hundreds of MB each and make real gateway hosts hit OOM under restart or close-failure paths.
- `impact:crash-loop`: The issue is about process-level availability and memory exhaustion from orphaned ACP adapter processes.
- `impact:session-state`: The cleanup depends on ACP session records, process leases, and stale session ownership after gateway restarts.

## Summary

Current main already implements the requested ACPX hardening: ACP runtime close now always falls through to OpenClaw-owned process-tree cleanup, and ACPX startup reaps stale leased or orphaned OpenClaw-owned wrapper trees. The fix is current-main-only and is not contained in the latest release tag v2026.6.6.

## What This Changes

The issue requests PID-level fallback cleanup when ACP runtime close fails and a gateway-startup reaper for orphaned ACP processes left after hard restarts.

## Best Possible Solution

Keep the current ACPX plugin-owned lease and reaper implementation on main and let it ship in the next release rather than adding a broad core process-name killer.

## Reproduction Assessment

Yes, at source level for the affected behavior: pinned acpx@0.10.0 records PIDs but its terminateAgentProcess has no separate OpenClaw-owned PID/tree fallback after detaching. I did not run a live gateway crash repro in this read-only review.

## Solution Assessment

Yes. The current main solution is the maintainable owner-boundary fix because ACPX owns the wrapper leases and validates command, wrapper root, lease id, and gateway instance before killing processes.
