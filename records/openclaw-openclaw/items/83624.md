## Label Justifications

- `impact:message-loss`: The reported tool responses and visible tool cards are routed or rendered in the wrong session context.
- `impact:session-state`: The core symptom is drift between the direct session key used by the WebSocket caller and the session context shown in the UI.

## Summary

Keep open. Current source supports a real session-state bug: `sessions.send` forwards the canonical session key into `chat.send`, but live tool updates for sessions started outside the UI are mirrored as `session.tool` frames that the web UI does not handle, so tools can fail to render in the active direct session even though the run is processing.

## What This Changes

The issue reports that WebSocket `sessions.send` with a direct session key creates or routes work into a group session instead of the existing direct web UI session.

## Best Possible Solution

Route `session.tool` into the same session-key-filtered UI tool-card path as `agent` tool events, while preserving the existing `sessions.send` canonical-key and channel-delivery semantics.

## Reproduction Assessment

No live reproduction was run, but there is a source-level reproduction path: start a run through `sessions.send` from a non-UI WebSocket client, subscribe the UI via session events, and observe that `session.tool` frames are emitted but not handled by the UI tool stream.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing session-scoped tool event and UI filtering path, not add a new session key shape or channel-specific policy.
