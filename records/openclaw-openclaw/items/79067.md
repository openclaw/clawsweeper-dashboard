## Summary

Keep open: this is a plausible focused fix for the linked cron self-heal failure, but the current patch also claims `apply_patch` recovery without using production `apply_patch` arguments, so that part should be corrected before merge.

## What This Changes

The PR adds same-target cross-tool file-mutation recovery in `isSameToolMutationAction`, regression tests, and a changelog entry for the cron false-failure path.

## Best Possible Solution

Keep the mutation-state fix, align the `apply_patch` behavior with production fingerprints or narrow the scope to `edit`/`write`, then merge after exact-head checks are green.

## Reproduction Assessment

Yes from source inspection, without running local tests: current main stores tool-scoped fingerprints, keeps `lastToolError` for `edit` to `write` on the same path, and cron treats the resulting trailing warning payload as fatal. The PR body also supplies live output for that helper-chain path.

## Solution Assessment

Not yet: clearing recovered file mutations before warning rendering is the right layer, but the PR should not claim `apply_patch` recovery until real `apply_patch` targets are fingerprinted or that scope is removed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make apply_patch recovery use real patch targets:** `src/agents/tool-mutation.ts:28`
  - body: This adds `apply_patch` to the file-mutating recovery set, but production `apply_patch` calls only pass patch text under `input`; `buildToolActionFingerprint` never extracts `path` or `oldpath` from that shape. Please either parse single-file patch targets into the fingerprint or remove `apply_patch` from this fallback and the public claim.
