## Summary

Keep this PR open, but do not merge it as-is: the fix targets the right Codex app-server context-loss path, but the new timestamp trigger can repeatedly project Codex’s own mirrored history back into an already-resumed native thread.

## What This Changes

The PR adds a Codex app-server resume check that projects mirrored OpenClaw history when user-visible transcript messages are newer than the saved thread binding, plus regression coverage for that path.

## Best Possible Solution

Preserve the intended cross-runtime history bridge, but distinguish externally newer OpenClaw history from Codex-owned mirrored turns before projection and add a repeated-resume regression test.

## Reproduction Assessment

Yes, from source inspection: binding updatedAt is written on resume before mirroring, and Codex’s own mirrored user/assistant messages are later read as history after that timestamp on the next run.

## Solution Assessment

No, not yet: the direction is right, but the timestamp predicate needs a durable ownership boundary or watermark so it only bridges history the native Codex thread does not already own.

## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid re-projecting Codex-owned mirrored turns:** `extensions/codex/src/app-server/run-attempt.ts:4289-4296`
  - body: This predicate treats any user/assistant message newer than the saved binding as missing from the native Codex thread. But the binding is updated during `thread/resume`, before `mirrorTranscriptBestEffort` writes the just-completed Codex turn back to the same session file. On the next resume, those Codex-owned mirror records are newer than the binding and will cause the full mirrored history block to be sent again, duplicating prior native-thread context every turn. Please filter to history not already owned by the bound Codex thread, or persist a projection/mirror watermark after mirroring, and add a two-resume regression test.
