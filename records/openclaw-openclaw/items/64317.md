## Summary

Keep open. Current main still routes ordinary managed viewport screenshots with a tab WebSocket through raw CDP `Page.captureScreenshot`, and there is still no Hyprland/grim compositor fallback in the browser plugin or browser docs.

## What This Changes

The issue asks OpenClaw to avoid headed Chromium viewport screenshot timeouts on Omarchy/Hyprland/Wayland after successful page load.

## Best Possible Solution

Land or replace a focused browser-plugin fix that uses a Hyprland compositor-capture fallback only for headed Linux viewport screenshots where CDP is known to stall, while preserving current CDP/Playwright behavior for headless, non-Hyprland, full-page, element, labeled, and existing-session paths.

## Reproduction Assessment

Yes for source-level reproduction: current main still reaches `Page.captureScreenshot` for the ordinary managed viewport screenshot path named in the report. The issue also provides concrete affected pages, timings, environment, and same-machine controls, though live compositor validation was not run here.

## Solution Assessment
