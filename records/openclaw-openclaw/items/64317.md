## Summary

Keep open. Current `main` and the latest shipped tag still route ordinary managed viewport screenshots through CDP `Page.captureScreenshot`, and no Hyprland/`grim` fallback is present in the browser plugin source or docs.

## What This Changes

The issue asks OpenClaw to avoid headed Chromium viewport screenshot timeouts on Omarchy/Hyprland/Wayland after a page has loaded successfully.

## Best Possible Solution

Implement or land a narrow browser-plugin fallback for headed Linux Hyprland/Wayland viewport screenshots that uses compositor capture only for the affected viewport path, handles missing `grim`/`hyprctl` cleanup safely, and preserves existing CDP/Playwright behavior elsewhere.

## Reproduction Assessment

Yes for source-level reproduction: ordinary managed viewport screenshots with a per-tab WebSocket still reach `Page.captureScreenshot`, exactly the path named in the report. No for live reproduction in this review, because no Hyprland/Wayland headed browser session was run.

## Solution Assessment

Yes, keeping this open is the best current outcome because neither current `main` nor `v2026.5.2` contains the proposed native fallback. The maintainable fix should stay scoped to the browser plugin’s headed viewport screenshot path rather than changing full-page, element, labeled, headless, or non-Hyprland capture behavior.
