## Summary

Keep open: the report is credible and current main still lacks a preflight for npm `min-release-age`/`before`, while plugin update failures can remain a post-update warning instead of making the aggregate update result fail before restart. Current main has some staged npm swap rollback safeguards, so the repair should focus on the npm quarantine preflight and result/status handling rather than replacing the whole updater.

## What This Changes

The issue requests safer `openclaw update` behavior when npm release-age quarantine blocks recent OpenClaw core or plugin packages.

## Best Possible Solution

Add an update preflight that detects blocking npm `before`/`min-release-age` settings for the target core and plugin specs, aborts before mutation with recovery guidance, and makes plugin post-update failures visibly fail or degrade the aggregate update result before restart.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: configure npm `min-release-age=7` or `before` so a target OpenClaw/plugin version is newer than the allowed date, then run the npm-backed update path. I did not run the destructive global-update flow in this read-only checkout.

## Solution Assessment

Yes, the narrow maintainable fix is to preflight npm release-age filters and correct aggregate status before package swap/restart. The existing staged install and rollback code should be preserved and extended only where proof shows a remaining atomicity gap.
