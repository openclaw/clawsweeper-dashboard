## Summary

Keep open: current main still supports single async image-generation tasks, status lookup, per-call image count, completion wake, and Task Flow as a generic durable orchestration substrate, but it does not provide the requested native multi-item image batch queue or persisted continuation API. This remains a product/API design request rather than a proven regression in an existing contract.

## What This Changes

The issue requests native continuation or queue support so multi-image `image_generate` workflows can run sequential jobs, deliver each result, and preserve batch progress across turn endings or interruptions.

## Best Possible Solution

Design a maintainer-approved generic media/task continuation model, likely reusing Task Flow and the existing task ledger, before adding any image-specific batch wrapper.

## Reproduction Assessment

No, not as a high-confidence bug reproduction. Source inspection confirms the native batch/continuation API is absent, but the issue lacks a current-main failing run, logs, exact provider behavior, or session transcript for the reported three-image workflow.

## Solution Assessment

Unclear. The requested image batch wrapper is plausible, but the maintainable path is probably a generic Task Flow or media-task continuation design rather than an image-only state machine.
