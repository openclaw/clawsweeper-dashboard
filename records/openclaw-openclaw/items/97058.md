## Summary

Keep open: current main and v2026.6.10 still support single async image-generation tasks plus per-call image count, but not the requested native multi-job image queue or persisted continuation API. This is a feature/product-design request rather than a proven regression in an existing contract.

## What This Changes

The issue requests native continuation or queue support for multi-step async image generation batches so multiple image jobs can continue after each item completes.

## Best Possible Solution

Design a maintainer-approved generic media/task continuation model, likely using Task Flow or the task ledger, then wire image_generate through that boundary instead of adding an image-only state machine.

## Reproduction Assessment

No high-confidence bug reproduction is available; the issue has no logs or runnable current-main transcript. Source inspection does confirm that native multi-item image batch continuation is absent today.

## Solution Assessment

Unclear. A batch wrapper is plausible, but the best maintainable solution likely needs a generic Task Flow or media-task continuation design rather than an image-only queue.
