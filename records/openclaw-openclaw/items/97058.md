## Summary

Keep open: current main supports single async image_generate tasks, status, completion wake, fallback delivery, and a per-call image count, but it does not provide the requested native sequential multi-image job queue or persisted continuation state. This is a feature/product-design request rather than a proven regression in an existing image-generation contract.

## What This Changes

The issue asks OpenClaw to add native continuation or queue support for multi-image async image_generate workflows so batches keep progressing across completions and chat interruptions.

## Best Possible Solution

Use a maintainer-approved generic media/task continuation design, preferably building on Task Flow, with image_generate wiring added only after ownership and persistence semantics are settled.

## Reproduction Assessment

No, not as a high-confidence bug reproduction. Source inspection confirms the native batch/continuation API is absent, but the issue lacks a current-main failing run, logs, or exact provider/session transcript for the stop-after-one workflow.

## Solution Assessment

Unclear. A native image batch wrapper is plausible, but the safer maintainable path is likely a generic Task Flow or media-task continuation design rather than an image-only state machine.
