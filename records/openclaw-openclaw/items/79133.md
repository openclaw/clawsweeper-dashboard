## Summary

Keep open: the protected `maintainer` label blocks cleanup closure, and source review supports the reported mismatch between `/model status` provider-row auth and the effective Codex runtime auth used elsewhere.

## What This Changes

Telegram `/model status` should show effective `openai-codex` OAuth for `openai/gpt-5.5` when `agentRuntime.id` is `codex` instead of reporting direct OpenAI auth as missing.

## Best Possible Solution

Reuse the existing effective runtime-auth provider mapping for `/model status`, keep ordinary provider/env/custom-key labels unchanged, and add focused regression coverage for Codex runtime plus `openai-codex` OAuth.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure `agentRuntime.id = codex`, select `openai/gpt-5.5`, provide only an `openai-codex` OAuth profile, and `/model status` will still resolve the `[openai]` row through `resolveAuthLabel('openai', ...)`. I did not run a live Telegram command in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to route `/model status` auth labels through the same effective runtime-auth provider logic already used by `/status` and the models status command. A separate config knob or provider-specific special case would be less consistent.
