## Summary

Keep open for maintainer review, but the patch should not merge as-is: it turns an invalid boolean manifest value into wildcard tool ownership, while current main and public docs define `contracts.tools` as explicit `string[]` tool names. The linked report at https://github.com/openclaw/openclaw/issues/80621 is source-reproducible, but the safer fix is clearer diagnostics/docs unless maintainers intentionally approve the wildcard API tracked at https://github.com/openclaw/openclaw/issues/77797.

## What This Changes

The PR maps `contracts.tools: true` to `["*"]`, treats `"*"` as allowing all plugin tool names, and adds helper tests for wildcard tool-contract behavior.

## Best Possible Solution

Keep explicit `contracts.tools` arrays for now; fix the linked report with clearer invalid-manifest diagnostics and docs, and handle wildcard or dynamic tool ownership through a maintainer-approved design in the canonical feature thread.

## Reproduction Assessment

Yes, at source level: boolean `contracts.tools` is dropped by `normalizeTrimmedStringList`, leaving no declared tool names, and `registerTool` then emits the reported missing-contract diagnostic. I did not run the live gateway flow because this review was read-only.

## Solution Assessment

No. The PR is not the narrowest maintainable fix because it adds wildcard manifest semantics instead of improving diagnostics/docs for the invalid boolean shape; wildcard support needs explicit maintainer API design first.
