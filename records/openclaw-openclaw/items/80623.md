## Summary

Keep open, but the patch should not merge as-is: it turns an invalid manifest value into a new wildcard tool-ownership mode that conflicts with the current explicit `contracts.tools` contract, and the external PR has no real behavior proof.

## What This Changes

Adds `contracts.tools: true` normalization to a `"*"` wildcard, treats `"*"` as allowing any registered tool name, and adds tool-contract unit tests.

## Best Possible Solution

Keep `contracts.tools` as an explicit list unless maintainers approve a new manifest mode; fix the linked problem with clearer invalid-manifest diagnostics and plugin-author docs for the required manifest location.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main drops non-array `contracts.tools: true`, leaving no declared tool names, and `api.registerTool` then emits the reported contract diagnostic. I did not run the live gateway flow because this review is read-only.

## Solution Assessment

No. The wildcard/boolean approach is not the narrowest maintainable fix because current docs, types, and tests define explicit tool-name ownership; a safer repair is an invalid-manifest diagnostic plus docs/examples unless maintainers intentionally approve wildcard semantics.
