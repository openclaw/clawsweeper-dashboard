## Summary

Keep open, but this PR should not merge as-is: current code and docs define `contracts.tools` as an explicit string array, while the patch turns an invalid boolean value into wildcard tool ownership and needs maintainer API approval plus real behavior proof.

## What This Changes

The PR maps `contracts.tools: true` to `["*"]`, treats `"*"` as allowing any plugin tool name, and adds wildcard helper tests.

## Best Possible Solution

Keep `contracts.tools` explicit unless maintainers intentionally approve and document a wildcard manifest mode; otherwise address the linked report with clearer invalid-manifest diagnostics and plugin-author docs.

## Reproduction Assessment

Yes, at source level: current main drops `contracts.tools: true` through array-only normalization and `registerTool` then emits the reported missing-contract diagnostic. I did not run a live gateway flow in this read-only review.

## Solution Assessment

No. Boolean-to-`["*"]` is a new wildcard manifest mode, while current docs and shipped code use explicit tool-name ownership; the safer narrow path is diagnostics/docs unless maintainers approve the API change.
