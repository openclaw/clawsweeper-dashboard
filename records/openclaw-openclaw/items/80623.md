## Summary

Keep open. The PR changes the current explicit `contracts.tools: string[]` manifest contract into boolean-to-wildcard tool ownership, and it still lacks after-fix real behavior proof from a real plugin/runtime flow.

## What This Changes

This PR maps `contracts.tools: true` to `["*"]`, treats `"*"` as allowing any registered plugin tool name, and adds tool-contract helper tests.

## Best Possible Solution

Keep `contracts.tools` explicit unless maintainers approve and document a wildcard manifest mode; otherwise resolve the linked report with clearer invalid-manifest diagnostics and plugin-author docs.

## Reproduction Assessment

Yes. Source inspection on current main shows `contracts.tools: true` normalizes to no declared tool names, then `registerTool` emits the reported missing-contract diagnostic.

## Solution Assessment

No. Treating boolean `true` as wildcard ownership is a new public manifest mode; the narrower maintainable path is clearer diagnostics/docs unless maintainers explicitly choose the API change.
