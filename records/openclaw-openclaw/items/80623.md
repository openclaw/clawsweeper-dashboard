## Summary

Keep open. The PR targets a source-reproducible plugin author pain point, but it changes the public manifest contract by treating an invalid boolean as wildcard tool ownership, while current main docs, types, and shipped behavior require explicit `string[]` tool names; it also still lacks after-fix real behavior proof.

## What This Changes

The PR maps `contracts.tools: true` to `["*"]`, treats `"*"` as allowing any plugin tool name, and adds helper tests for the wildcard behavior.

## Best Possible Solution

Keep `contracts.tools` as explicit tool-name ownership unless maintainers intentionally approve and document a wildcard manifest mode; for the linked report, prefer clearer invalid-manifest diagnostics and plugin-author docs.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` normalizes to no declared names and then `registerTool` emits the exact missing-contract diagnostic; I did not run the live gateway flow in this read-only review.

## Solution Assessment

No. The PR's boolean-to-wildcard mapping changes a documented public contract, so the safer path is diagnostics/docs unless maintainers approve a new wildcard API with matching schema, docs, and end-to-end proof.
