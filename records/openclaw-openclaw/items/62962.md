## Summary

Keep open. Current main still lacks the requested tool security classification contract and schema-driven consumers; adjacent trusted-policy, plugin metadata, derived-path, catalog, and provider `parallel_tool_calls` work remains complementary rather than a merged implementation.

## What This Changes

Add standard tool security metadata fields to tool definitions and use them for approvals, parallel scheduling, interrupt behavior, and classifier input.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract and thread it through core tool definitions, plugin metadata, hook/trusted-policy events, Gateway catalog/effective inventory, docs, API baselines, and conformance checks before adding conservative consumers.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: current main lacks the requested fields and schema-driven consumers across tool types, plugin metadata, hooks, catalog, inventory, conformance, and docs.

## Solution Assessment

No, not as a direct one-off six-field addition. The requested direction is plausible, but the maintainable solution needs maintainer design for taxonomy, redaction, trust tiers, backward compatibility, and how metadata feeds approval and concurrency decisions.
