## Summary

Keep open. Current main still lacks the requested schema-level tool security classification contract and consumers; related open design items at https://github.com/openclaw/openclaw/issues/48510 and https://github.com/openclaw/openclaw/issues/48503 are complementary rather than superseding this narrower read-only/destructive/concurrency/interrupt/classifier-summary request.

## What This Changes

The issue requests standard tool security classification properties so approvals, warnings, interrupt handling, classifier input, and parallel scheduling can use declared tool metadata.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract separate from pre-action decision receipts, then thread safe metadata through tool definitions, plugin metadata, trusted policies, before_tool_call, Gateway inventory, conformance, docs, and conservative consumers.

## Risks / Open Questions

- The taxonomy, trust tiers, and unknown/default semantics would become long-lived public SDK and security-policy contracts.
- A toAutoClassifierInput-style hook could expose secrets, private URLs, file paths, command arguments, or media references without redaction and length bounds.
- Read-only or concurrency-safe flags must not become automatic allow paths around owner-only filtering, allow/deny policy, sandboxing, trusted policies, plugin approvals, or host exec approvals.
- Per-tool concurrency metadata needs a clear relationship with existing provider-level parallel_tool_calls behavior to avoid unsafe scheduling semantics.
- The design should be coordinated with the open capability and provenance issues to avoid overlapping SDK fields.
