## Summary

Keep open. Current main still lacks the requested per-tool security-classification contract and schema-driven consumers; related issues and drafts are adjacent, but they do not supersede the full six-field proposal for destructive/read-only/concurrency/interrupt/open-world/classifier semantics.

## What This Changes

The issue requests schema-level tool security metadata fields for read-only, destructive, concurrency-safe, open-world, interrupt, and classifier-input behavior, plus consumers for approvals and scheduling.

## Best Possible Solution

Design one additive, versioned tool-security metadata contract that covers built-in and plugin tools, projects through hook events, trusted policies, catalog/effective inventory, docs, and conformance, and keeps all metadata subordinate to existing policy gates.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection of origin/main, which shows the requested fields absent from tool types, plugin metadata, hook events, dispatch, conformance, and Gateway projections.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the exact six-field taxonomy and consumers should be reconciled with https://github.com/openclaw/openclaw/issues/48503 and https://github.com/openclaw/openclaw/issues/48510 before implementation.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested API is security-sensitive because these fields would become inputs for allow, block, prompt, audit, scheduling, and classifier decisions.

  - confidence: 0.86

  - body: Read-only or concurrency-safe flags must not become automatic allow paths around owner-only filtering, allow/deny policy, trusted policies, sandbox constraints, plugin approvals, or host exec approvals.
