## Summary

Keep open. Current main still lacks the requested per-tool security classification contract and consumers; the trusted policy and plugin metadata work is adjacent but only covers policy hooks plus display/risk catalog metadata, and the related RFC/draft follow-up explicitly say they do not implement this broader proposal.

## What This Changes

Add six schema-level tool security classification properties and wire them into confirmation, scheduling, abort, classifier, and conformance behavior.

## Best Possible Solution

Define a backwards-compatible public tool security metadata contract, preserve it through core tools, plugin registration, adapters, inventories, docs, and conformance snapshots, then wire conservative consumers that add safety signals without weakening existing policy or approval gates.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The current-main verification path is high-confidence: exact field search is empty and the central tool, adapter, policy, conformance, plugin metadata, catalog, and docs surfaces do not expose the requested contract.

## Solution Assessment

Unclear as currently proposed; the direction is plausible, but the exact contract is security-sensitive and not yet the narrowest maintainable implementation. Maintainers should settle defaults, redaction, compatibility, approval ordering, and concurrency semantics before a fix PR is safe.
