## Summary

Keep open. Current main still lacks the requested six-field tool security classification contract and schema-driven consumers, and the closest related issues remain complementary open design work rather than a complete canonical replacement.

## What This Changes

The issue requests standardized tool security metadata such as read-only, destructive, concurrency, interrupt, open-world, and classifier-summary properties on tool definitions.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract across core tools, plugin metadata, hook/trusted-policy events, Gateway catalog/effective inventory, docs, API baselines, and conformance checks, with conservative consumers that never bypass existing gates.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: the requested fields and consumers are absent from the central tool, hook, metadata, catalog, inventory, conformance, and scheduling surfaces.

## Solution Assessment

No, not as a direct one-off field addition. The proposed fields are plausible, but the maintainable solution needs maintainer design for taxonomy, defaults, trust tiers, redaction, concurrency semantics, and backwards compatibility.
