## Summary

Keep open. Current main still lacks the requested tool security classification contract and schema-driven consumers, and the recent external preflight-adapter comment is useful design input rather than a merged OpenClaw implementation or a reason to close.

## What This Changes

The issue requests standard tool security classification fields so approvals, concurrency, interrupts, and classifier inputs can be driven by tool metadata.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract, thread it through core tool definitions, plugin metadata, hook/trusted-policy events, Gateway catalog/effective inventory, docs, API baselines, and conformance checks, then add conservative consumers.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: current main lacks the requested fields and consumers across tool types, plugin metadata, hooks, catalog/effective inventory, conformance, and docs.

## Solution Assessment

No, not as a direct one-off six-field addition. The maintainable solution needs maintainer SDK/security design for taxonomy, redaction, trust tiers, compatibility, and how classification metadata feeds approval, interrupt, classifier, and concurrency behavior.
