## Summary

Keep open. Current main has adjacent trusted-policy, plugin metadata, derived-path, catalog, and provider `parallel_tool_calls` work, but it still lacks the requested tool security classification contract and schema-driven consumers; https://github.com/openclaw/openclaw/issues/48510 and https://github.com/openclaw/openclaw/issues/48503 remain complementary open design items rather than superseding implementations.

## What This Changes

Request adding standard security classification properties to tool definitions so approval, scheduling, interrupt, network-policy, and classifier decisions can be schema-driven.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract and thread it through core tool definitions, plugin metadata, hook/trusted-policy events, Gateway catalog/effective inventory, docs, API baselines, and conformance checks before adding conservative consumers.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: current main lacks the requested fields and schema-driven consumers across tool types, plugin metadata, hooks, catalog, inventory, and docs.

## Solution Assessment

No, not as a direct one-off six-field addition. The requested direction is plausible, but the maintainable solution needs maintainer design for taxonomy, redaction, trust tiers, backward compatibility, and how metadata feeds approval and concurrency decisions.
