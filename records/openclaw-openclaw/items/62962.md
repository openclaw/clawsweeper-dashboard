## Summary

Keep open. Current main still lacks the requested tool security classification contract and schema-driven consumers, while the related capability/provenance/security-policy threads remain complementary open design items rather than merged implementations.

## What This Changes

The issue requests standard tool classification metadata such as read-only, destructive, concurrency, interrupt, open-world, and classifier-summary properties to drive approval, scheduling, abort, and safety-policy behavior.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract and separate classification from pre-action decision receipts, then thread the safe metadata through tool definitions, plugin metadata, trusted policies, before_tool_call, Gateway catalog/effective inventory, docs, API baselines, and conformance checks before enabling conservative consumers.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: current main lacks the requested fields and schema-driven consumers across tool types, plugin metadata, hook events, dispatch, Gateway projections, conformance, and docs.

## Solution Assessment

No, not as a direct one-off six-field addition. The requested direction is plausible, but the maintainable solution needs maintainer design for taxonomy, redaction, trust tiers, backward compatibility, decision receipts, and how metadata feeds approval and concurrency behavior.
