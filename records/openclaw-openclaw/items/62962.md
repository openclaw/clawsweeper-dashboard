## Summary

Keep open. Current main still lacks the requested six-field per-tool security classification contract and schema-driven consumers; adjacent trusted-policy hooks, owner-only filtering, plugin display/risk metadata, and `parallel_tool_calls` transport settings do not implement this API, and related RFC #71734 plus draft PR #74483 explicitly leave this broader proposal unresolved.

## What This Changes

Add schema-declared tool security classification metadata for read-only, destructive, concurrency, interrupt, external-system, and classifier-input behavior.

## Best Possible Solution

Define a backwards-compatible public tool-security metadata contract, carry it through core tools, plugin registration, runtime adapters, catalog/inventory surfaces, docs, and conformance checks, then add conservative consumers that remain subordinate to existing policy gates.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: exact requested fields are absent and the central tool, adapter, policy, conformance, plugin metadata, docs, and scheduling paths do not expose the requested contract.

## Solution Assessment

No. The proposed direction is plausible, but the specific public contract and consumers need maintainer design decisions around defaults, redaction, approval ordering, backwards compatibility, and concurrency semantics before implementation is the best fix.

## Security Review

Status: needs_attention

Summary: No patch diff exists, but the requested public tool-security metadata contract is security-sensitive and needs maintainer design before implementation.

  - confidence: 0.86

  - body: Read-only or concurrency-safe flags must not become automatic allow paths around owner-only filtering, allow/deny policy, sandbox constraints, plugin approvals, or host exec approvals.
  - confidence: 0.84

  - confidence: 0.82
