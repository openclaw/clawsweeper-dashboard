## Summary

Keep open. Current main still lacks the requested six-field tool security classification contract and schema-driven consumers; adjacent trusted-policy, plugin metadata, hook enrichment, catalog, inventory, and provider `parallel_tool_calls` work does not implement this proposal, and related open design items remain complementary rather than merged superseding fixes.

## What This Changes

The issue requests standard tool security classification properties on tool definitions so prompts, scheduling, interrupts, classifier input, and conformance checks can be driven from schema metadata.

## Best Possible Solution

Design an additive, versioned tool-security metadata contract, carry it through core tools, plugin registration, hook/trusted-policy events, Gateway catalog/effective inventory, docs, API baseline, and conformance checks, then add conservative consumers that never bypass existing policy gates.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The high-confidence verification path is source inspection: the requested fields and schema-driven consumers are absent from the central tool, hook, metadata, catalog, inventory, conformance, and scheduling surfaces.

## Solution Assessment

No, not as a direct one-off field addition. The proposed fields are plausible, but the maintainable solution needs maintainer design for taxonomy, defaults, redaction, trust tiers, backwards compatibility, and how metadata feeds approval and concurrency decisions.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested API is security-sensitive because it would become policy input for allow, prompt, block, classifier, interrupt, and scheduling behavior.

Concerns:

- **[medium] Define classifier redaction before prompt use:** `src/agents/tools/common.ts:31`
  - body: A toAutoClassifierInput-style hook would compress tool arguments for another classifier call; without redaction and length bounds it could expose secrets, private URLs, media references, command arguments, or file paths.
  - confidence: 0.86
- **[medium] Keep metadata subordinate to existing gates:** `src/agents/pi-tools.before-tool-call.ts:533`
  - body: Read-only or safe-to-parallelize flags must not become automatic allow paths around owner-only filtering, allow/deny policy, sandbox constraints, plugin approvals, trusted policy hooks, or host exec approvals.
  - confidence: 0.84
- **[medium] Resolve concurrency semantics before exposing flags:** `src/agents/pi-embedded-runner/extra-params.ts:331`
  - body: The runtime already injects provider-level parallel_tool_calls for supported GPT transports, so a new isConcurrencySafe flag needs one authoritative mapping before it can influence scheduling safely.
  - confidence: 0.82
