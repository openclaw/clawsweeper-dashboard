## Summary

Keep this PR open; the branch is not merge-ready because the advertised symlink-backed redirect path conflicts with the current fs-safe write contract, and the new optional `postHookActions` field is only tolerated after handlers have already run.

## What This Changes

The PR adds `sessionSaveRedirectPath` support to the bundled session-memory hook, introduces `postHookActions`, hardens slug generation, and adds redirect, rollback, and hook-drain tests.

## Best Possible Solution

Keep the PR open, decide the symlink redirect policy, then either add a reviewed safe-write seam for in-workspace symlink targets or remove that support from the PR, while also normalizing `postHookActions` before handlers and adding the changelog entry.

## Reproduction Assessment

Yes. Source inspection at the PR head shows symlink redirect writes go through `writeFileWithinRoot`, whose current implementation rejects existing symlink targets, and shows missing `postHookActions` normalization before handler dispatch.

## Solution Assessment

No. The PR shape is plausible, but the implementation must first align redirect symlink semantics with fs-safe and make the optional hook field safe before handlers run.

  - body: This redirected write still goes through `writeFileWithinRoot`, but that helper rejects existing symlink targets while resolving the write target.   - confidence: 0.9

  - body: The branch makes `postHookActions` optional and documents manually constructed events as compatible, but handlers run before the only `?? []` guard at drain time. Any handler following the new examples with `event.postHookActions!.push(...)` will throw on a legacy hand-built event, get logged as a hook error, and skip deferred behavior. Initialize `event.postHookActions ??= []` before the handler loop and cover a legacy event whose handler pushes an action.
