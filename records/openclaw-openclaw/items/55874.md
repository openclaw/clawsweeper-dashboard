## Summary

Keep open. Current main still lacks the proposed `safetyPosture` config surface, so this PR is not obsolete, but the branch should not merge as-is because the new security-facing knobs are only partially materialized and do not enforce the advertised tool or memory restrictions.

## What This Changes

The PR adds top-level `safetyPosture` config types, Zod schema, generated config metadata, runtime preset expansion for sandbox/session defaults, focused tests, and a few test-helper import cleanups.

## Best Possible Solution

A revised PR should make `safetyPosture` a real additive runtime contract: schema/types/generated metadata plus centralized materialization that sets existing sandbox, session, tool, and memory policy fields, or removes unsupported promises, with docs and tests asserting effective behavior rather than only parsed config shape.

## Reproduction Assessment

Yes for the introduced defects. Source inspection of PR head shows `secureDmMode` is skipped without a preset and `agentProfile` only writes `_safetyPostureResolvedProfile`; current main's tool profile schema/resolver confirms that marker is not consumed.

## Solution Assessment

No. The proposed direction is reasonable, but the implementation is not the narrowest maintainable solution until it rebases onto `materializeRuntimeConfig` and maps each advertised preset to concrete existing policy fields or a clearly documented new contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply the advertised tool profile to real policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: `agentProfile` currently resolves only into `_safetyPostureResolvedProfile`, but current tool policy never reads that marker and only recognizes `minimal`, `coding`, `messaging`, and `full`. A `strict` or `public` posture can therefore validate while leaving high-risk tools governed by the old policy, so the preset must write `tools`/agent policy or add a real consumed profile contract.
  - confidence: 0.93
- **[P2] Honor secureDmMode when no preset is set:** `src/config/apply-safety-posture.ts:31-34`
  - body: The schema accepts `safetyPosture.secureDmMode` as an independent boolean, but `applySafetyPosture` returns unchanged whenever `preset` is omitted. That makes `{ safetyPosture: { secureDmMode: true } }` silently leave `session.dmScope` untouched, despite the field's documented purpose.
  - confidence: 0.9
- **[P2] Enforce or remove the strict memory preset:** `src/config/apply-safety-posture.ts:57-61`
  - body: The strict preset advertises disabled long-term memory, and the test name says memory is disabled, but the implementation explicitly skips memory mapping. Operators could select `strict` and still keep the existing memory configuration, so either map this to the current memory controls or drop the unsupported preset promise.
  - confidence: 0.86
- **[P2] Move posture expansion into runtime materialization:** `src/config/io.ts:1376`
  - body: This patch inserts `applySafetyPosture` into an older `io.ts` load chain, while current main now centralizes load/snapshot/default handling in `materializeRuntimeConfig`. Rebase the change into that materialization flow so all config consumers see the same effective posture.
  - confidence: 0.78
