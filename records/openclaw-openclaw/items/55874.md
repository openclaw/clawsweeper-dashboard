## Summary

Keep open: current main does not implement `safetyPosture`, and the PR still exposes security-posture presets whose runtime behavior does not enforce the advertised tool and memory restrictions.

## What This Changes

The PR adds a top-level `safetyPosture` config schema/types, generated schema metadata, runtime preset expansion, tests, and shared extension test-helper import cleanups.

## Best Possible Solution

Define the safety-posture contract as an additive config surface only after each preset expands through shared runtime materialization into enforceable sandbox, session, tool, and memory settings, with unsupported claims removed or narrowed.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the PR diff and current main shows `secureDmMode` is ignored without a preset, memory mapping is skipped, and `agentProfile` writes a marker no current tool-policy path consumes. I did not run tests because this task is a read-only review.

## Solution Assessment

No. The feature direction is plausible, but the implementation should move into `materializeRuntimeConfig` and either map presets to existing enforceable tool/memory policy fields or introduce a reviewed, documented profile contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply posture profiles to real tool policy:** `src/config/apply-safety-posture.ts:63-67`
  - body: `agentProfile` resolves only into `_safetyPostureResolvedProfile`, but current tool-policy resolution reads configured `tools.profile` and provider profiles. A `strict` or `public` posture can therefore validate without denying the advertised high-risk tools, so the preset needs to write an enforced tool policy or add a consumed profile contract.
  - confidence: 0.93
- **[P2] Honor secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:29-31`
  - body: The schema accepts `safetyPosture.secureDmMode` as an independent boolean, but `applySafetyPosture` returns unchanged whenever `preset` is omitted. `{ safetyPosture: { secureDmMode: true } }` silently leaves `session.dmScope` untouched despite the field's purpose.
  - confidence: 0.9
- **[P2] Enforce or remove the strict memory promise:** `src/config/apply-safety-posture.ts:53-59`
  - body: The strict preset advertises disabled long-term memory and the preset constants encode that value, but the runtime expansion explicitly skips memory mapping. Operators could select `strict` and still keep existing memory indexing or recall behavior, so the preset must map to real controls or stop promising that restriction.
  - confidence: 0.86
- **[P2] Move expansion into runtime materialization:** `src/config/io.ts:1373`
  - body: This inserts `applySafetyPosture` into one load chain, while current main centralizes load and snapshot materialization in `materializeRuntimeConfig`. Put the expansion in that shared materialization path so validation snapshots and other config consumers see the same effective posture.
  - confidence: 0.78
- **[P2] Avoid returning an undeclared profile marker:** `src/config/apply-safety-posture.test.ts:26`
  - body: The new tests access `cfg._safetyPostureResolvedProfile`, but `OpenClawConfig` does not declare that property. The changed test lane should typecheck this file, so either remove the private marker in favor of real config fields or declare a typed internal runtime field if it is intentionally consumed.
  - confidence: 0.82
