## Summary

Keep this PR open: current main still has no `safetyPosture` config surface, but the proposed patch is not ready because the advertised security presets do not fully map to enforced tool, memory, and config materialization behavior.

## What This Changes

The PR adds top-level safety-posture config types/schema, generated config metadata, runtime preset expansion for sandbox/session/tool defaults, focused tests, and a few extension test-helper import cleanups.

## Best Possible Solution

Land an additive safety-posture contract only after each advertised preset expands through current runtime materialization into enforceable sandbox, session, tool, and memory settings, with unsupported claims removed or narrowed.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the PR head shows `secureDmMode` is ignored without a preset, memory mapping is skipped, and `agentProfile` writes an unconsumed marker. Current-main inspection confirms the live tool policy has no `limited` or `public` profile contract.

## Solution Assessment

No. The feature direction is plausible, but the implementation should rebase into `materializeRuntimeConfig` and either enforce the tool/memory promises through existing policy fields or narrow the public preset semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply posture profiles to real tool policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: `agentProfile` resolves only into `_safetyPostureResolvedProfile`, but current tool policy never reads that marker and only resolves configured `tools.profile` / provider profiles. A `strict` or `public` posture can therefore validate without denying the advertised high-risk tools, so the preset needs to write an enforced tool policy or add a consumed profile contract.
  - confidence: 0.93
- **[P2] Honor secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:31-34`
  - body: The schema accepts `safetyPosture.secureDmMode` as an independent boolean, but `applySafetyPosture` returns unchanged whenever `preset` is omitted. `{ safetyPosture: { secureDmMode: true } }` silently leaves `session.dmScope` untouched despite the field's purpose.
  - confidence: 0.9
- **[P2] Enforce or remove the strict memory promise:** `src/config/apply-safety-posture.ts:57-61`
  - body: The strict preset advertises disabled long-term memory and the preset constants encode that value, but the runtime expansion explicitly skips memory mapping. Operators could select `strict` and still keep existing memory indexing or recall behavior, so the preset must map to real controls or stop promising that restriction.
  - confidence: 0.86
- **[P2] Move expansion into runtime materialization:** `src/config/io.ts:1376`
  - body: This inserts `applySafetyPosture` into an older inline load chain, while current main centralizes load and snapshot materialization in `materializeRuntimeConfig`. Put the expansion in that shared materialization path so all config consumers see the same effective posture.
  - confidence: 0.78
