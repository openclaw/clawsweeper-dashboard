## Summary

Keep open for maintainer review: current main does not already implement safetyPosture, but this PR is not safe to merge because it publishes security-posture promises before tool, memory, and secure-DM enforcement are complete, and it lacks real behavior proof.

## What This Changes

Adds a top-level safetyPosture config with development/balanced/strict presets, runtime preset expansion, tests, generated schema metadata, and shared extension test-helper import updates.

## Best Possible Solution

Rebase onto current materializeRuntimeConfig and schema-base behavior, then land only an additive preset contract whose sandbox, session, tool-policy, and memory claims are either enforced through existing runtime paths or removed from public schema/help text.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the latest diff shows secureDmMode can be ignored, strict memory mapping is skipped, and agentProfile writes only an unconsumed marker. I did not run tests or live config because this was a read-only review.

## Solution Assessment

No. The hardening direction matches the closed product discussion, but the current patch exposes security posture promises before active runtime enforcement semantics are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply posture profiles to real tool policy:** `src/config/apply-safety-posture.ts:63-66`
  - body: agentProfile resolves only into _safetyPostureResolvedProfile, but current tool policy reads tools.profile and the core catalog only recognizes minimal/coding/messaging/full. A strict or public posture can therefore validate without denying the advertised high-risk tools.
  - confidence: 0.93
- **[P2] Honor secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:29-31`
  - body: The schema accepts safetyPosture.secureDmMode as an independent boolean, but applySafetyPosture returns unchanged whenever preset is omitted and otherwise ignores the flag when choosing dmScope. A config that sets only secureDmMode: true can silently leave DMs on the main scope.
  - confidence: 0.9
- **[P2] Enforce or remove the strict memory promise:** `src/config/apply-safety-posture.ts:54-59`
  - body: The strict preset advertises disabled long-term memory, but runtime expansion explicitly skips memory mapping. Operators can select strict and still keep existing qmd/session memory indexing or recall behavior active.
  - confidence: 0.86
