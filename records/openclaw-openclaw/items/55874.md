## Summary

Keep open. Current main still does not implement the safetyPosture config surface, and the PR is not merge-ready because the advertised public/limited tool profiles and strict memory posture validate without being enforced, with the runtime hook also targeting an older config materialization path.

## What This Changes

The PR adds top-level safetyPosture config types, Zod/generated schema metadata, runtime preset expansion for sandbox/session/profile defaults, focused tests, and small test-helper import cleanups.

## Best Possible Solution

Make safetyPosture an additive runtime contract only after each advertised preset maps to enforced sandbox, session, tool, and memory policy fields, or narrow the preset promises to the behavior OpenClaw can enforce today.

## Reproduction Assessment

Yes for the PR-introduced defects. Source inspection of PR head shows secureDmMode is skipped when preset is absent and agentProfile only writes an unconsumed marker, while current main's tool policy resolver has no limited/public profile contract.

## Solution Assessment

No. The direction is plausible, but the implementation should be rebased into materializeRuntimeConfig and either enforce the tool/memory promises through existing policy fields or remove unsupported security claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply posture profiles to real tool policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: `agentProfile` currently resolves only into `_safetyPostureResolvedProfile`, but current tool policy never reads that marker and only recognizes `minimal`, `coding`, `messaging`, and `full`. A `strict` or `public` posture can therefore validate while leaving high-risk tools governed by the old policy, so the preset must write the real tools policy or add a consumed profile contract.
  - confidence: 0.93
- **[P2] Honor secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:31-34`
  - body: The schema accepts `safetyPosture.secureDmMode` as an independent boolean, but `applySafetyPosture` returns unchanged whenever `preset` is omitted. `{ safetyPosture: { secureDmMode: true } }` silently leaves `session.dmScope` untouched despite the field's documented purpose.
  - confidence: 0.9
- **[P2] Enforce or remove the strict memory promise:** `src/config/apply-safety-posture.ts:57-61`
  - body: The strict preset advertises disabled long-term memory, and the preset constants encode that, but the implementation explicitly skips memory mapping. Operators could select `strict` and still keep existing memory indexing and recall behavior, so either map this to current memory controls or drop the unsupported preset promise.
  - confidence: 0.86
- **[P2] Rebase posture expansion into config materialization:** `src/config/io.ts:1376`
  - body: This patch inserts `applySafetyPosture` into the older inline config load chain, while current main now centralizes load, snapshot, and validation materialization in `materializeRuntimeConfig`. Rebase the expansion into that materialization flow so all config consumers see the same effective posture.
  - confidence: 0.78
