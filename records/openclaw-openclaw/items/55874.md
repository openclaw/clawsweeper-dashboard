## Summary

Keep open. The safety-posture feature is not present on current main, and the linked discussion explicitly allowed focused hardening follow-ups, but this PR is not mergeable as-is because the advertised security presets are not fully enforced and contributor real behavior proof is still missing.

## What This Changes

The PR adds a top-level `safetyPosture` config schema/type, preset expansion helper, generated schema metadata, tests, and extension test-helper import adjustments.

## Best Possible Solution

Rebase onto current config materialization and land only an additive preset contract whose sandbox, session, tool-policy, and memory claims are either enforced through existing runtime paths or removed from schema/help text.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the latest PR files shows `secureDmMode` can be ignored, strict memory is not wired, and `agentProfile` writes only an unconsumed marker. I did not run tests or live config because this was a read-only review.

## Solution Assessment

No. The hardening direction is plausible, but this patch exposes security-posture promises before the runtime enforcement and product contract are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Honor secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:31-33`
  - body: `secureDmMode` is accepted by the schema as an independent boolean, but this early return means `safetyPosture: { secureDmMode: true }` leaves the config unchanged instead of setting `session.dmScope` to `per-channel-peer`. That makes the documented secure-DM control silently ineffective unless a preset is also present.
  - confidence: 0.93
- **[P2] Wire posture profiles into real tool policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: This stores `limited` or `public` in `_safetyPostureResolvedProfile`, but current tool policy reads `tools.profile` and only recognizes `minimal`, `coding`, `messaging`, and `full`. A strict/public posture can validate while leaving the advertised high-risk tool denylist unenforced.
  - confidence: 0.94
- **[P2] Use the memory preset before exposing it:** `src/config/apply-safety-posture.ts:38`
  - body: `memoryPreset` is assigned but never read, which will fail the repo's `noUnusedLocals` TypeScript checks. It also means the strict preset's `longTermMemoryEnabled: false` promise is not applied to runtime memory behavior.
  - confidence: 0.91
