## Summary

Keep open: current main does not implement `safetyPosture`, and this PR is still not safe to merge because several advertised security posture settings are accepted or documented without being enforced by the current runtime policy paths.

## What This Changes

The PR adds a top-level `safetyPosture` config schema/types, runtime preset expansion for sandbox/session/profile defaults, generated schema metadata, tests, and extension test-helper import cleanups.

## Best Possible Solution

Land an additive safety-posture contract only after each advertised preset maps through `materializeRuntimeConfig` into enforceable sandbox, `dmScope`, tool-policy, and memory controls, with docs and changelog aligned.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection shows `secureDmMode` is ignored without a preset, strict memory mapping is skipped, and `agentProfile` writes a marker that current tool policy never reads.

## Solution Assessment

No. The feature direction is plausible, but the current implementation needs maintainer-approved security semantics and should expand presets through the shared runtime materialization and real policy fields rather than an unconsumed marker.

  - body: The strict preset advertises disabled long-term memory, but the runtime expansion explicitly skips memory mapping. Operators selecting `strict` can still keep existing memory indexing or recall behavior, so the preset needs to map to real controls or stop promising that restriction.
  - confidence: 0.87

  - body: Current main centralizes load and snapshot defaults in `materializeRuntimeConfig`, while this wires `applySafetyPosture` into an older inline load chain. Put the expansion in the shared materialization path so every config consumer sees the same effective posture after rebase.
