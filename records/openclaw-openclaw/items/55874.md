## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Apply secureDmMode even without a preset:** `src/config/apply-safety-posture.ts:31-33`
  - body: `secureDmMode` is accepted as an independent boolean, but this early return means `safetyPosture: { secureDmMode: true }` leaves the config unchanged instead of setting `session.dmScope` to `per-channel-peer`. That makes the isolation control silently ineffective unless a preset is also present.
  - confidence: 0.93
- **[P2] Map posture profiles into real tool policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: This stores `limited` or `public` in `_safetyPostureResolvedProfile`, but current tool policy reads `tools.profile` and only recognizes `minimal`, `coding`, `messaging`, and `full`. A strict/public posture can validate while leaving the advertised high-risk tool denylist unenforced.
  - confidence: 0.94
- **[P2] Apply or remove the strict memory preset:** `src/config/apply-safety-posture.ts:38`
  - body: `memoryPreset` is assigned but never read, which fails the repo's strict unused-local checks and leaves `MEMORY_PRESETS.strict.longTermMemoryEnabled: false` disconnected from runtime memory behavior. Either map it to existing memory controls or remove the exposed preset promise.
  - confidence: 0.91
- **[P2] Type the resolved profile before asserting it:** `src/config/apply-safety-posture.test.ts:24`
  - body: `applySafetyPosture` returns `OpenClawConfig`, but `OpenClawConfig` does not define `_safetyPostureResolvedProfile`, so the new tests' direct property access will fail strict test typechecking. Prefer mapping the profile into the real `tools.profile` contract, or add a typed consumed runtime field before asserting it.
  - confidence: 0.87
