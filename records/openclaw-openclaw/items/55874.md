## Summary

Keep open: current main does not implement the `safetyPosture` surface, and the linked maintainer discussion treats this as reviewable product hardening rather than a shipped fix. The PR still has source-level enforcement gaps for security posture promises and lacks contributor real behavior proof.

## What This Changes

The PR adds `safetyPosture` config types/schema, preset expansion during config load, generated schema metadata, focused tests, and extension test-helper import updates.

## Best Possible Solution

Rebase onto current config materialization and land an additive preset contract only after sandbox, session, tool-policy, memory, and network claims are enforced through existing runtime paths or removed from the public schema/help text.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the PR head shows `secureDmMode` can be ignored, tool profiles are written to an unconsumed marker, and strict memory/network claims are not wired to runtime config. I did not run tests because this was a read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this patch exposes a security-posture config contract before its sandbox, tool, memory, and session semantics are fully enforced.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Apply secureDmMode without requiring a preset:** `src/config/apply-safety-posture.ts:31-33`
  - body: `secureDmMode` is accepted as an independent boolean, but this early return means `safetyPosture: { secureDmMode: true }` leaves the config unchanged instead of setting `session.dmScope` to `per-channel-peer`. That makes the isolation control silently ineffective unless a preset is also present.
  - confidence: 0.93
- **[P2] Map posture profiles into real tool policy:** `src/config/apply-safety-posture.ts:63-68`
  - body: This stores `limited` or `public` in `_safetyPostureResolvedProfile`, but current tool policy reads `tools.profile` and only recognizes `minimal`, `coding`, `messaging`, and `full`. A strict/public posture can validate while leaving the advertised high-risk tool denylist unenforced.
  - confidence: 0.94
- **[P2] Apply the sandbox network preset:** `src/config/apply-safety-posture.ts:45-49`
  - body: `SANDBOX_PRESETS` includes `docker.network: "none"` for balanced/strict postures, but the expansion only copies `mode` and `workspaceAccess`. Users selecting those presets would not get the promised sandbox network isolation unless they also configure it manually.
  - confidence: 0.9
- **[P2] Apply or remove the strict memory preset:** `src/config/apply-safety-posture.ts:38`
  - body: `memoryPreset` is assigned but never read, which fails strict unused-local checks and leaves `MEMORY_PRESETS.strict.longTermMemoryEnabled: false` disconnected from runtime memory behavior. Either map it to existing memory controls or remove the exposed preset promise.
  - confidence: 0.91
- **[P2] Type the resolved profile before asserting it:** `src/config/apply-safety-posture.test.ts:28`
  - body: `applySafetyPosture` returns `OpenClawConfig`, but `OpenClawConfig` does not define `_safetyPostureResolvedProfile`, so the new tests' direct property access will fail strict test typechecking. Prefer mapping the profile into the real `tools.profile` contract, or add a typed consumed runtime field before asserting it.
  - confidence: 0.87
