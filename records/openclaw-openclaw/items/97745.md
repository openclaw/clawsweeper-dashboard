## Summary

Keep open: the OAuth response-bound hardening is useful, but the OpenRouter success path uses the small diagnostic text reader instead of a fail-closed provider JSON/body-limit reader, and the external PR still needs real after-fix proof before merge.

## What This Changes

The PR bounds OpenRouter OAuth success reads and Google OAuth token error/JSON reads with provider HTTP helpers, and adds an OpenRouter unit test that verifies the success path avoids response.text().

## Best Possible Solution

Use provider-local fail-closed JSON/body-limit helpers for the OAuth success paths, add normal and oversized response proof, and coordinate the Google token parser cap with the broader Google OAuth PR.

## Reproduction Assessment

Yes from source inspection: current main still has unbounded OpenRouter OAuth success response.text() and Google token response.text()/json() reads on the reported paths. I did not run a loopback oversized-response repro in this read-only review.

## Solution Assessment

No: provider-local bounded helpers are the right layer, but the OpenRouter success path should use a fail-closed JSON/body-limit reader rather than the truncating diagnostic text reader. The Google token cap is plausible defense-in-depth but overlaps the broader Google OAuth parser PR.
