## Summary

Keep this PR open, but do not merge it as-is: the underlying cron preflight bug is plausible from current source, while the patch threads the raw persisted provider config value into an Authorization header and bypasses OpenClaw's credential-resolution contract.

## What This Changes

The PR adds an optional `apiKey` parameter to cron model-provider preflight and sends `Authorization: Bearer <apiKey>` on the local provider probe request.

## Best Possible Solution

Resolve a real probe credential through the existing provider-auth/SecretRef/non-secret-marker contract, skip the header when only a marker is available, add focused preflight tests for plaintext and marker/SecretRef/authless cases, then attach redacted after-fix LiteLLM or equivalent logs.

## Reproduction Assessment

Yes, from source inspection: current main probes local OpenAI-compatible providers with `GET /models` and no headers, and the existing mocked test already models a 401 response as reachable. I did not run a live LiteLLM cron job in this read-only review.

## Solution Assessment

No, this patch is not the best way to solve it because it uses the persisted `apiKey` field directly instead of a resolved credential that respects SecretRefs, env markers, auth profiles, and non-secret local markers. The safer fix is to reuse the existing provider-auth filtering contract and test the compatibility cases.
