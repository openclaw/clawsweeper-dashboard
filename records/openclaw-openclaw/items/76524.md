## Summary

Keep open. Current main still caps `sessions.usage` totals by the requested page size, while this PR is a narrow source-backed fix with regression coverage and a changelog entry; no blocking functional or security findings were found.

## What This Changes

The PR changes `sessions.usage` so aggregate totals are computed from all matching merged sessions while `limit` only controls rows returned in the `sessions` array, with a regression test and changelog entry.

## Best Possible Solution

Land this PR or an equivalent gateway fix that aggregates across all matching sessions while applying `limit` only to returned session rows, preserving the regression test and changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main slices `mergedEntries` into `limitedEntries` before the aggregate loop, so any range with more matching sessions than `limit` undercounts totals and breakdowns; the PR’s added test models this with two sessions and `limit: 1`.

## Solution Assessment

Yes. The PR uses the narrow maintainable fix: keep the page membership as a set while iterating all merged entries for aggregate data, avoiding protocol or UI contract changes.
