## Summary

Keep open. Current main still slices `mergedEntries` before computing `sessions.usage` totals, while this PR applies the narrow gateway fix with regression coverage and a changelog entry; I found no blocking review findings.

## What This Changes

The PR changes gateway `sessions.usage` to aggregate over all merged session entries while limiting only the returned `sessions` rows, adds a regression test, and adds an Unreleased changelog entry.

## Best Possible Solution

Land this PR or an equivalent gateway fix that computes all matching aggregate data independently from the paginated `sessions` response and keeps the regression test plus changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main slices `mergedEntries` before the aggregation loop, so any date range with more matching sessions than `limit` undercounts totals; the PR test models this with two sessions and `limit: 1`.

## Solution Assessment

Yes. The PR uses the narrow maintainable fix: iterate all merged entries for aggregate data and gate only `sessions.push()` on limited-page membership, without changing the protocol or adding configuration.
