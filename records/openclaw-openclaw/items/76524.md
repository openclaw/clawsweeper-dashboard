## Summary

Keep open for maintainer review and landing. Current main still undercounts `sessions.usage` totals when `limit` is smaller than the matching session set, and the PR’s narrow aggregation/page-splitting approach matches the protocol contract with focused regression coverage and no review findings.

## What This Changes

The PR changes `sessions.usage` to compute totals and aggregate breakdowns from all merged entries while paging only the returned `sessions` list, adds regression coverage, and adds an Unreleased changelog fix entry.

## Best Possible Solution

Land this PR or an equivalent gateway fix that aggregates over all matching sessions while keeping `limit` scoped to the returned `sessions` page and preserving the regression test plus changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main applies `limit` before aggregation, so any date range with more matching sessions than the requested limit undercounts totals and aggregate breakdowns; the PR’s added test models this with two sessions and `limit: 1`.

## Solution Assessment

Yes. The PR uses the narrow maintainable fix: iterate all merged entries for aggregate data and gate only `sessions.push()` on the limited page membership.
