## Summary

Keep open: current main does not implement generic direct `codex-cli` artifact finalization, but this PR still needs real Codex CLI behavior proof, a rebase/product decision after the Codex app-server migration, and fixes for parser scope, log hygiene, and cleanup ordering before it is merge-ready.

## What This Changes

The PR adds `codex-cli` `--output-last-message` temp artifact wiring to the CLI runner, prefers artifact text over stdout, broadens JSONL fallback parsing, and adds parser/runner tests.

## Best Possible Solution

Land only if maintainers still want direct `codex-cli` artifact finalization: keep the parser change Codex-scoped or defer to the terminal-field parser PR, mask and guard the artifact path, rebase onto current main, and require live Codex CLI proof.

## Reproduction Assessment

Yes, source-level: current main's generic CLI runner finalizes successful runs from streamed JSONL or parsed stdout and does not allocate or read a `--output-last-message` artifact. A live current-main failure path was not run, and the direct `codex-cli` path is now a product decision because current docs say Codex agent runs use the app-server harness.

## Solution Assessment

No, not as proposed. The artifact approach is plausible for an explicitly supported direct `codex-cli` backend, but this patch also changes generic JSONL parsing, needs temp-file cleanup/log hygiene fixes, lacks live proof, and must be reconciled with the Codex app-server migration and the open parser PR.
