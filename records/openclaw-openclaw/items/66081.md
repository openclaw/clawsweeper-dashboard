## What This Changes

## Best Possible Solution

Refresh this as a maintainer-approved final-artifact contract at the CLI backend/plugin boundary, then rebase onto current main, keep non-Codex JSONL behavior unchanged, and add real Codex CLI artifact proof before merge.

Full review comments:

- **[P1] Put artifact support behind a backend contract:** `src/agents/cli-runner/execute.ts:102-103`
  - body: Current main no longer registers a bundled `codex-cli` backend, but this helper hard-codes that backend id in the core runner and then appends a Codex-only flag to any plugin/user backend with that name. That can break custom or older `codex-cli` commands and bypasses the plugin-owned CLI backend boundary; add an explicit backend/plugin final-artifact contract or keep this behavior in the owning backend.
  - confidence: 0.84
- **[P1] Scope the new JSONL fallback to Codex:** `src/agents/cli-output.ts:486-490`
  - body: This fallback runs for every JSONL backend, so a non-Codex backend that emits status, tool, or diagnostic records with `message`, `content`, `result`, or `response` can have that text appended to the final user reply. Guard this extraction to the intended Codex path or move it into a declared JSONL dialect.
  - confidence: 0.9
- **[P2] Create the artifact inside the guarded cleanup path:** `src/agents/cli-runner/execute.ts:272-275`
  - body: `prepareCodexLastMessageArtifact()` is awaited after system prompt/image/skills resources may already exist but before the outer `try/finally` begins. If `mkdtemp` fails, the existing cleanup block never runs, so those prior temp resources can leak; allocate this artifact under the same guarded cleanup lifetime.
  - confidence: 0.82
