## Summary

Keep open: this PR is still the canonical implementation surface for the closed artifact-finalization request, and current main does not add `--output-last-message` to the generic CLI runner. The patch is not merge-ready because it broadens JSONL parsing beyond Codex, needs real behavior proof from a live Codex CLI run, and now needs maintainer judgment after the Codex app-server migration.

## What This Changes

The PR adds Codex CLI `--output-last-message` temp artifact wiring, prefers that artifact over parsed stdout for final replies, and adds parser/runner tests for artifact precedence and stdout fallback.

## Best Possible Solution

Keep this as the canonical PR only if maintainers still want direct `codex-cli` artifact finalization; otherwise close it in favor of the Codex app-server path and the separate parser PR.

## Reproduction Assessment

Yes, source-level: current main's generic CLI runner still finalizes successful CLI runs from streamed JSONL or parsed stdout and has no artifact readback. Live reproduction is less direct now because current docs say bundled `codex-cli` is no longer registered by default.

## Solution Assessment

No, not as proposed. The artifact strategy is plausible for an explicit direct Codex CLI backend, but this patch changes generic JSONL parsing, needs cleanup/log hygiene fixes, lacks live proof, and needs rebasing against the app-server migration.

Full review comments:

- **[P2] Scope JSONL fallback to Codex:** `src/agents/cli-output.ts:486-493`
  - body: The new fallback extracts `message`, `content`, `result`, and `response` from every JSONL backend. Custom JSONL backends are documented public surface, so a non-Codex status or tool-result record can now be folded into the final assistant reply; guard this by `providerId === "codex-cli"` or an explicit dialect and cover a non-Codex regression case.
  - confidence: 0.9
- **[P3] Mask the generated artifact path in argv logs:** `src/agents/cli-runner/execute.ts:276-278`
  - body: Adding `--output-last-message <temp-file>` means verbose `cli argv` logs can include the generated temp directory and filename. Command-execution logs already mask injected prompt and image paths, so add a masking case for this generated artifact path too.
  - confidence: 0.82
- **[P3] Move artifact allocation under the cleanup guard:** `src/agents/cli-runner/execute.ts:270-273`
  - body: `prepareCodexLastMessageArtifact` is awaited before the outer cleanup `finally` starts. If that temp allocation fails after earlier prompt image, system prompt, or plugin resources were prepared, those resources can skip cleanup; allocate this inside the guarded block or start the cleanup guard before the new I/O point.
  - confidence: 0.72
