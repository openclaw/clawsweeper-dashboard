## Summary

Keep open: current main still lacks the Codex final-message artifact path, so the PR addresses a real CLI runner gap, but the proposed patch is not merge-ready because it broadens JSONL reply extraction beyond the stated `codex-cli` scope and leaves a low-severity command-log hygiene issue.

## What This Changes

The PR adds Codex CLI `--output-last-message` temp artifact creation, readback, cleanup, artifact-over-stdout final reply selection, and focused parser/runner tests for the `codex-cli` path.

## Best Possible Solution

Land the artifact-preferred Codex CLI finalization path after narrowing the JSONL content fallback to `codex-cli` or an explicit dialect, masking the generated artifact path in verbose argv logs, placing artifact allocation under the cleanup scope, and keeping focused tests for artifact precedence, empty-artifact fallback, and non-Codex JSONL behavior.

## Reproduction Assessment

Yes. Current main inspection reproduces the missing behavior because the runner finalizes from stdout and the bundled `codex-cli` backend does not pass `--output-last-message`; the PR tests simulate the artifact-vs-stdout precedence case without needing a live Codex run.

## Solution Assessment

No, not as proposed. The artifact strategy is the right direction, but the current diff changes generic JSONL backend parsing and needs small command-execution hygiene fixes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope JSONL content fallback to Codex:** `src/agents/cli-output.ts:503-511`
  - body: The new fallback extracts `message`, `content`, `result`, and `response` from every JSONL backend. OpenClaw documents custom JSONL CLI backends, so a backend emitting status or tool-result records with those fields can now have operational text folded into the final assistant reply, outside the PR's stated Codex-only scope. Guard this with `providerId === "codex-cli"` or an explicit dialect and add a non-Codex regression test.
  - confidence: 0.9
- **[P3] Create the artifact inside the cleanup scope:** `src/agents/cli-runner/execute.ts:272-279`
  - body: The PR awaits the Codex artifact preparation before the outer cleanup `finally` is active. If that new temp-file allocation fails after earlier prompt/image/plugin resources were prepared, those resources can be left behind. Move artifact preparation under the same guarded cleanup scope or start the cleanup `try` before this new I/O point.
  - confidence: 0.72
- **[P3] Mask the output-last-message path in logs:** `src/agents/cli-runner/execute.ts:272-279`
  - body: Adding `--output-last-message <temp-file>` means verbose `cli argv` logs can include the generated temp directory and artifact filename. That path is not a credential, but it is command-execution log hygiene and should be masked like existing system prompt and image path arguments.
  - confidence: 0.82
