## What This Changes

The branch adds typed `plugins.*` Gateway RPC schemas, validators, handlers, admin scopes, write rate limits, shared plugin-management services, regenerated Swift protocol models, docs, and tests.

## Best Possible Solution

Keep the service-layer approach only if maintainers accept the admin-scoped remote plugin-management boundary, then fix the install error classification, add a single-line changelog entry, rebase onto current main, and run the targeted plugin/Gateway tests plus the changed gate in Testbox before merge.

## Reproduction Assessment

Yes for the blocking review finding: call `plugins.install` with `source: "path"` and a missing Gateway-host path. The existing path resolver returns an uncoded `path not found` failure, and this PR maps that to `UNAVAILABLE` instead of an invalid request.

## Solution Assessment

No as-is. The shared service layer is a maintainable direction for exposing existing CLI plugin operations to Gateway clients, but the branch should preserve client-error semantics for bad local paths, add the required changelog, and resolve the dirty base before it is mergeable.
