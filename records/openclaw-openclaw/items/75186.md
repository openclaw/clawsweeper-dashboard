## Summary

Keep open. Current main still lacks the proposed Gateway plugin-management RPC surface, so this PR is not obsolete, but it is not mergeable as-is because a client-error mapping bug and the required changelog entry remain, and a maintainer has explicitly asked for deeper API/security review before landing.

## What This Changes

The PR adds typed Gateway `plugins.*` management RPC schemas, validators, handlers, admin scopes, write rate limits, shared plugin-management services, regenerated Swift protocol models, docs, and tests.

## Best Possible Solution

Keep the service-layer direction only if maintainers approve the remote plugin-management boundary, then fix the error semantics and changelog, rebase onto current main, and validate the Gateway/protocol/plugin lanes before merge.

## Reproduction Assessment

Yes for the blocking code review finding: call `plugins.install` with `source: "path"` and a missing Gateway-host path on the PR branch. The existing path installer can return an uncoded deterministic input failure, and the PR maps that path to `UNAVAILABLE` instead of an invalid request.

## Solution Assessment

No as-is. A shared service layer is a plausible maintainable direction, but this branch needs the install error classification fix, a user-facing changelog entry, and explicit maintainer acceptance of the protocol/security boundary before it is the right solution.
