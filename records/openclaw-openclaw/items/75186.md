## Summary

Keep open. Current main does not implement the proposed Gateway plugin-management RPCs, and maintainer discussion says the direction is interesting but not ready to land; the PR still has one correctness blocker plus the required changelog gap.

## What This Changes

The PR adds typed `plugins.*` Gateway RPCs, shared plugin-management services, admin scopes/rate limits, Swift protocol models, docs, and tests for remote plugin list/inspect/doctor/registry/install/update/uninstall/enable/disable operations.

## Best Possible Solution

Keep the PR open for maintainer review, fix the install error classification and changelog entry, rebase onto current main, then run targeted Gateway/plugin tests plus the changed gate before merge.

## Reproduction Assessment

Yes for the blocking review finding: the PR head has `installFailureKind(undefined)` return `unavailable`, while the existing install result type allows failures without a code and local path/target errors can be uncoded deterministic input failures. This was verified by source inspection rather than executing tests because the review was read-only.

## Solution Assessment

No as-is. The shared service-layer direction is plausible, but the branch should preserve client-error semantics for bad install input and add the required changelog entry before maintainers decide whether the remote admin control-plane boundary is acceptable.
