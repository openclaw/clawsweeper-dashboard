## Summary

Keep open. Current main still lacks the requested Gateway plugin-management RPC surface, but this PR is not merge-ready because a deterministic bad local path is still classified as `UNAVAILABLE`, the required changelog entry is missing, and maintainer discussion calls for careful API/security/rebase review before landing.

## What This Changes

The PR adds typed Gateway `plugins.*` management RPCs, shared plugin-management services, admin scopes/rate limits, regenerated Swift protocol models, docs, and handler/service tests.

## Best Possible Solution

Finish maintainer review of the Gateway API and security boundary, rebase onto current main, then fix the error semantics and changelog before running targeted plugin/Gateway tests plus the changed gate.

## Reproduction Assessment

Yes for the blocking review finding: call `plugins.install` with `source: "path"` and a nonexistent Gateway-host path. The existing installer returns uncoded `directory not found` or `file not found`, and this PR maps that undefined code to `UNAVAILABLE`.

## Solution Assessment

No as-is. The shared service-layer direction is plausible, but the branch should preserve invalid-request semantics for bad local paths, add the required changelog entry, and settle the maintainer-reviewed API/security boundary before merge.

Full review comments:

- **[P2] Classify uncoded install failures as invalid requests:** `src/plugins/management-core.ts:75-78`
  - body: When `installPluginFromPath` fails without a `PluginInstallErrorCode`, deterministic input problems such as missing local directories or files reach this fallback and are returned as `UNAVAILABLE`. Gateway clients will treat a bad path as a transient service outage instead of correcting the request, so the undefined-code fallback should stay in the invalid-request class unless the lower layer identifies an operational failure.
