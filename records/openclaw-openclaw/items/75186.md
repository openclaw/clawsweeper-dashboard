## Summary

Keep open: current main still lacks Gateway plugin management RPCs, and the maintainer discussion says the capability is useful but not ready because the protocol, admin/security boundary, host-path semantics, registry state, and app compatibility still need review. The branch is draft/dirty, lacks real Gateway behavior proof, and still has a blocking error-classification issue.

## What This Changes

Adds admin-scoped Gateway `plugins.*` management RPC schemas and handlers, shared plugin management services, Swift protocol models, docs, scopes/rate limits, and focused tests.

## Best Possible Solution

Keep the service-layer direction only if maintainers accept the Gateway boundary, then rebase onto the current method descriptor registry, fix install error classification, and add real Gateway behavior proof.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: PR head maps uncoded install failures to `UNAVAILABLE`, while current `installPluginFromDir` and `installPluginFromFile` return uncoded missing-path failures. Reproduction of the feature itself is not applicable because this is a new RPC surface.

## Solution Assessment

No as-is. The reusable service-layer approach is plausible, but this branch should preserve client-error semantics for bad Gateway-host paths and wait for maintainer product/security/protocol acceptance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Classify missing path installs as invalid requests:** `src/plugins/management-core.ts:75-78`
  - body: When `plugins.install` receives `source: "path"` for a missing Gateway-host path, the underlying path install returns uncoded `directory not found` or `file not found` failures. This helper maps an undefined code to `unavailable`, so clients see a transient service outage for deterministic bad input and may retry instead of correcting the path; classify these uncoded path-input failures as `invalid-request` without changing real operational failures.
  - confidence: 0.9
