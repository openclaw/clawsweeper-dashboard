## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes the reported MCP symptom by removing a stored inherited allowlist that current main uses as a child-session tool-authorization boundary.

## What This Changes

The PR removes propagation of the parent inherited tool allowlist into spawned child sessions while keeping inherited denylist propagation.

## Best Possible Solution

Preserve inheritedToolAllow propagation as the child-session authorization boundary, then fix MCP availability by including authorized MCP tool names in the effective inherited allowlist or by moving inherited allow filtering to the correct materialized-tool stage.

## Reproduction Assessment

Source-reproducible: current main stores inheritedToolAllow and later resolves it during child runtime tool filtering, so deleting the storage removes that enforcement path. I did not run a live MCP subagent scenario in this read-only review.

## Solution Assessment

No; this is not the best fix because it removes the inherited allowlist boundary instead of fixing the MCP materialization/filtering order. A safer fix should keep the stored inherited policy and make authorized MCP tools appear in the effective child tool surface.
