## Summary

Keep open. Current main still supports selecting individual session keys, including channel-derived sessions, but the built-in Control UI and Gateway chat-history contract remain centered on one active session and do not provide a separated/merged channel view toggle.

## What This Changes

Request a Control UI mode that can show channel conversations independently or merge WeChat, Feishu, QQ, and WebChat conversations into one unified timeline.

## Best Possible Solution

Add a core Control UI and Gateway protocol design that preserves current per-session selection as the separated mode, adds an explicit separated/merged chat mode, and exposes a permission-aware aggregation method with timestamp, channel, account, source, and session provenance on each timeline entry.

## Reproduction Assessment

Yes for the feature gap: current-main static verification shows the UI loads chat history for one state.sessionKey and the Gateway chat.history schema accepts one sessionKey. This is a feature request rather than a runtime crash, so no failing test run is required to confirm the missing capability.

## Solution Assessment

Unclear for an implementation because no patch exists yet. The requested direction is reasonable, but the maintainable solution should be a protocol-backed aggregate timeline rather than browser-side ad hoc transcript fetching.
