## Summary

Keep open: this PR addresses the linked exec environment feature and has sufficient gateway-mode proof, but the latest head still exports the internal node sentinel as `OPENCLAW_SESSION_KEY` on direct node-host runs without session metadata, conflicting with the documented empty-string contract.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child-process environments across gateway, sandbox/PTY, node-host, macOS node mode, docs, tests, and changelog.

## Best Possible Solution

Keep the env-injection approach, normalize the direct node-host session env to empty when the internal sentinel is used, add omitted-session node-host coverage, then maintainer-review the public env contract.

## Reproduction Assessment

Yes for source reproduction: current main only injects `OPENCLAW_SHELL=exec`, and PR head's direct node-host path defaults omitted `sessionKey` to `node` before exporting it. I did not run live exec because this review is read-only.

## Solution Assessment

No, not yet: injecting trusted values at child-env composition points is the right direction, but the direct node-host path needs the same sentinel normalization as the companion path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize the node sentinel before exporting session env:** `src/node-host/invoke-system-run.ts:669`
  - body: When `system.run` is invoked without a session key, `parseSystemRunPhase` uses the internal `node` sentinel. This new env merge exports that sentinel verbatim on the direct node-host path, while the docs promise an empty string when unavailable and the companion path already maps `node` to `""`. A host=node exec without session metadata will therefore see `OPENCLAW_SESSION_KEY=node` instead of the documented empty value.
  - confidence: 0.92
