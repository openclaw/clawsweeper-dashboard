## Summary

Keep open: the PR improves gateway and fallback node-host env injection, but it still misses successful macOS app/companion node execution and lacks the required changelog entry for the new user-facing exec contract.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child-process env handling, updates exec/environment docs, and adds gateway/node-host regression tests.

## Best Possible Solution

Ship one always-set trusted exec metadata env contract across gateway, sandbox, PTY, headless node, and macOS app/companion hosts, with parity tests and a changelog entry.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows `preferMacAppExecHost` returns after a successful `runViaMacAppExecHost` response before the new fallback env merge, while Swift passes sanitized env unchanged. I did not run a live macOS companion session.

## Solution Assessment

No. The gateway and fallback node merge is necessary but not sufficient; the same trusted post-sanitization env merge needs to happen in the Swift macOS exec-host paths without letting user env override it.
