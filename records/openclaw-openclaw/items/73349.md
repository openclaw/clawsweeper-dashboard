## Summary

Keep this PR open, but do not merge it as-is. The requested feature is real and current main does not implement it; the latest PR commits appear to address the earlier docs/test wording feedback, but node-host execution still routes the new values through the untrusted `requestedEnv` override payload that shell-wrapper sanitization drops before spawning the final child process, and the user-facing feature is still missing a changelog entry.

## What This Changes

The PR adds docs, tests, and exec-tool changes intended to inject `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into gateway, sandbox, PTY, and node-host child-process environments.

## Best Possible Solution

Preserve the gateway/sandbox/PTY injection direction, but carry session and agent identifiers through node execution as trusted runtime metadata and merge them into the final child environment after shell-wrapper env filtering in both the TypeScript node host and macOS runtime. Keep user env override blocking semantics, document the always-set empty-string behavior, add final-child node-host coverage, and add the required changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `host=node` exec is always shell-wrapped, the PR places the new values in `requestedEnv`, and the receiving node-host sanitizer drops non-allowlisted `OPENCLAW_*` overrides before spawning the child.

## Solution Assessment

No. Adding the values to `requestedEnv` is not the best way to solve node-host execution because that payload is intentionally treated as untrusted env override data; a trusted metadata path merged after sanitizer filtering is the safer maintainable fix.
