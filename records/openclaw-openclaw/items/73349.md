## Summary

Keep open: the PR now includes docs, tests, changelog, and real gateway-mode proof, but PR head still misses the macOS app/companion host path because trusted env injection happens only after a successful companion response would already have returned.

## What This Changes

The PR adds always-set `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to exec child environments for gateway/sandbox/PTY/node-host paths, with docs, changelog, and regression tests.

## Best Possible Solution

Complete the env contract at the node-host boundary so the same trusted always-set metadata reaches fallback `runCommand` and the macOS app/companion execution hosts, then land with TS and Swift parity coverage.

## Reproduction Assessment

Yes for the PR blocker: source inspection of PR head shows the successful `preferMacAppExecHost` path returns before the new fallback env merge, while current Swift forwards sanitized env unchanged. I did not run a live macOS companion session.

## Solution Assessment

No. The gateway and fallback node merges are directionally right, but the trusted metadata env must be composed before both the macOS companion request and fallback execution paths.
