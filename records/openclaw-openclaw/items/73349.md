Full review comments:

- **[P1] Document these env vars as context markers:** `docs/help/environment.md:123-124`
  - body: The new docs say scripts can reliably identify the agent and session from these env vars, while the linked use cases include approval, push-policy, and observability decisions. Because child-process env can be read, logged, forwarded, and changed by the process tree, document these as routing/context hints and state that authorization needs another trust boundary.
  - confidence: 0.88
- **[P1] Do not silently replace explicit env values:** `src/agents/bash-tools.exec-runtime.ts:676-677`
  - body: Placing the trusted values after `...opts.env` means an existing call that explicitly sets `OPENCLAW_AGENT_ID` or `OPENCLAW_SESSION_KEY` now gets a different value, or an empty string, without warning. Either reject those keys as reserved with a clear error or keep the old explicit override behavior until maintainers approve the breaking reservation.
  - confidence: 0.9
- **[P2] Make the clearing tests observe the spawned env:** `src/agents/bash-tools.exec.path.test.ts:351`
  - body: The two clearing tests assert command output, but empty stdout is rendered as `(no output)`, so the focused CI shard fails and the tests still do not prove the keys are present with empty-string values. Capture the supervisor `input.env` for these calls and assert the env object directly.
  - confidence: 0.96
