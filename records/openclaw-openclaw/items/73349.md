## Summary

Keep open: current main still lacks the requested exec child-process env vars, and the latest PR head still has one concrete correctness blocker in the macOS companion exec-host path.

## What This Changes

This PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child-process environments across local, node-host, and macOS exec paths, with docs, changelog, and regression tests.

## Best Possible Solution

Keep the env-var contract, normalize unavailable session metadata to an empty string before any child-process env export, and preserve internal routing fallbacks only for event delivery.

## Reproduction Assessment

Yes. Source inspection of the PR head shows an omitted node-host session key becomes the internal `"node"` fallback, and the macOS companion request/export path still forwards that value into child env.

## Solution Assessment

No. The feature shape is appropriate, but the current head needs one more normalization point for macOS companion forwarding so every exported child env follows the documented empty-string fallback.

On PR #73349, fix exec env injection so `OPENCLAW_SESSION_KEY` is empty when no session metadata is provided in every node-host path. The direct TypeScript `runCommand` path already maps the internal `"node"` sentinel to `""`, but `preferMacAppExecHost` still builds an `ExecHostRequest` with `sessionKey: phase.sessionKey`, and `ExecApprovalsSocket` exports that value. Normalize the request session key before forwarding to the macOS companion, or defensively normalize in the companion too, while preserving event-routing fallback semantics. Add regression coverage for mac app exec-host forwarding with omitted `params.sessionKey` so it cannot pass/export `"node"`. Do not change public env var names, add config, or broaden env override allowlists; the existing changelog entry is sufficient unless docs wording changes.
