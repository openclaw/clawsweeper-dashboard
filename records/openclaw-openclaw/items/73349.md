## Summary

Keep open. The PR addresses the requested exec env contract and its later commits appear to resolve the Greptile docs/test comments, but the host=node implementation still routes trusted values through the untrusted requestedEnv channel where shell-wrapper env filtering drops these OPENCLAW_* keys before the final child process. The user-facing exec feature also still needs a changelog entry.

## What This Changes

The PR injects OPENCLAW_AGENT_ID and OPENCLAW_SESSION_KEY into exec child-process environments across gateway/sandbox/PTY and node-host paths, with docs and regression tests.

## Best Possible Solution

Route the session and agent identifiers as trusted exec runtime metadata, merge them after user env sanitization in gateway/sandbox/PTY and both TypeScript and macOS node-host runtimes, preserve empty-string and user-override prevention semantics, then document and changelog the user-facing env contract.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: host=node exec builds a shell-wrapper argv and forwards env through requestedEnv, while both node-host sanitizers drop non-allowlisted shell-wrapper env overrides before spawning the final child.

## Solution Assessment

No. Adding these values to requestedEnv is not the best fix because that field is treated as untrusted user override data; a separate trusted metadata path is safer and preserves the intended override-blocking semantics.
