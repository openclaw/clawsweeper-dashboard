## Summary

Keep this PR open: the env-var contract is useful and the latest head covers several exec paths, but source inspection still shows missing or inconsistent behavior for macOS node execution and missing-value clearing.

## What This Changes

The PR adds docs, changelog, tests, and exec/node-host env merges to expose `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to exec child processes.

## Best Possible Solution

Keep the env-var contract, but make all exec hosts compose the same trusted metadata values with consistent empty-string clearing before spawning child processes.

## Reproduction Assessment

Yes for the PR blockers: source inspection of latest head shows direct macOS `system.run` still passes `evaluation.env` unchanged, the Swift companion skips empty-value clearing, and the TS node fallback exports the internal `node` session sentinel. I did not run a live macOS node-host session.

## Solution Assessment

No. The direction is maintainable, but the implementation should share one trusted metadata composition model across gateway/sandbox/PTY, TS node fallback, Swift companion, and direct macOS node runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Inject metadata in macOS node runtime too:** `docs/tools/exec.md:83`
  - body: This new exec contract says spawned commands receive the session and agent env vars, but the macOS app node-host path still handles `system.run` in `MacNodeRuntime` and passes `evaluation.env` straight to `executeSystemRun` without adding either key. A user running `exec host=node` against the macOS app will still not see the promised values, so add the same trusted env merge there with Swift coverage.
  - confidence: 0.9
- **[P2] Always clear missing Swift metadata values:** `apps/macos/Sources/OpenClaw/ExecApprovalsSocket.swift:456-462`
  - body: The Swift companion only writes `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` when the request fields are non-empty. That does not match the documented and tested TS behavior where the vars are always present and empty when unavailable; it can leave the keys absent or inherited from the app environment. Assign both keys unconditionally from `request.agentId ?? ""` and `request.sessionKey ?? ""`.
  - confidence: 0.86
- **[P2] Do not export the node sentinel as a session key:** `src/node-host/invoke-system-run.ts:669`
  - body: `phase.sessionKey` has already defaulted to the internal `"node"` fallback when the caller omitted a session key, and this new env merge exports that value as `OPENCLAW_SESSION_KEY`. That violates the PR's empty-string clearing contract and can make callback scripts treat `node` as a real routable session. Preserve the raw requested session key for the env var, or normalize missing values to `""` before spawning.
  - confidence: 0.88
