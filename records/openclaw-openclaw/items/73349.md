## Summary

Keep this PR open. The Greptile docs/test comments appear addressed in the later commits, but the node-host path still carries the new trusted identifiers through the user env-override channel, where shell-wrapper filtering drops `OPENCLAW_*` before the final child process sees them; the branch also still lacks the required changelog entry.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child environments, updates exec/environment docs, and adds gateway/node-host regression tests.

## Best Possible Solution

Carry session and agent identifiers as trusted exec runtime metadata, merge them after user env sanitization in gateway, sandbox, node, and macOS execution, document the always-set empty-string semantics, and add changelog plus final-child coverage.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR puts the identifiers in `requestedEnv`, node exec shell-wraps commands, and node-host shell-wrapper sanitization drops non-allowlisted `OPENCLAW_*` overrides before spawn.

## Solution Assessment

No. The gateway/sandbox runtime injection shape is reasonable, but the node-host implementation should not use `requestedEnv`; trusted identifiers need a separate contract or post-sanitization merge point.
