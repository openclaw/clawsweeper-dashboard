## Summary

Keep open. Current main still lacks same-product cross-platform export/import or restore semantics, and the linked implementation work is unmerged while maintainer discussion has flagged native import/restore as a product and security decision.

## What This Changes

Request `openclaw migrate export` and `openclaw migrate import` commands for selective, cross-platform OpenClaw state/workspace migration.

## Best Possible Solution

Keep this as the narrower OpenClaw-to-OpenClaw migration tracker, then decide whether the first implementation belongs under hardened backup restore/convert or a new migration sub-surface with preview-first validation and explicit secret/state policy.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior. Source and docs give high-confidence proof that current main has backup create/verify and provider-based imports, but not same-product export/import.

## Solution Assessment

Unclear. The requested direction is useful, but current maintainer discussion suggests the best solution may be backup convert/restore or a redesigned import flow rather than adding archive import directly under the existing provider migration namespace.
