## Summary

Keep open. Current main has provider-based `openclaw migrate` imports, `backup create`/`verify`, and manual machine-migration docs, but it still lacks the requested first-party same-product cross-platform export/import or restore workflow; the overlapping restore PR/issues are open and security/product-sensitive rather than a reason to close this tracker.

## What This Changes

Request `openclaw migrate export` and `openclaw migrate import` commands for portable cross-platform OpenClaw state migration.

## Best Possible Solution

Choose one first-party portable state migration/recovery model, then implement it as a hardened preview-first flow with archive validation, path remapping, secret policy, conflict handling, component selection, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, tests, and related GitHub checks give high-confidence proof that current main lacks same-product `migrate export/import` or native cross-platform restore.

## Solution Assessment

Unclear as a direct implementation path. The requested capability is useful, but maintainers should first choose the command namespace and safety model rather than landing an archive-writing import flow piecemeal.
