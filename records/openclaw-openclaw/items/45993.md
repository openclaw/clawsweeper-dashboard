## Summary

Keep open: current main and v2026.5.20 still have provider-based `openclaw migrate` plus `backup create`/`backup verify`, while the requested OpenClaw-to-OpenClaw cross-platform export/import flow remains absent. The work is valid but destructive and security-sensitive, and it overlaps an open closing PR plus the broader native backup/restore umbrella, so it needs maintainer product/security direction before implementation.

## What This Changes

Add `openclaw migrate export` and `openclaw migrate import` for selective cross-platform migration of workspace, config, memory, skills, and related local state.

## Best Possible Solution

Settle the first-party recovery model and namespace first, then implement a guarded export/import or restore flow with preview, archive validation, containment, rollback, conflict handling, secret policy, cross-platform path mapping, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source, docs, latest-release inspection, and related GitHub state give high-confidence proof that provider import and backup create/verify exist while native OpenClaw-to-OpenClaw export/import remains absent.

## Solution Assessment

Unclear. The user problem is valid, but `migrate import` is not yet proven as the best core shape because backup restore, backup convert/platform targeting, manual restore guidance, or a more guarded restore command may be safer.
