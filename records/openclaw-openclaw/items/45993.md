## Summary

Keep open: current main and v2026.5.20 still have provider-based `openclaw migrate` plus `backup create`/`backup verify`, but not same-product cross-platform `migrate export/import`. The issue also has an open closing PR, so the right cleanup path is maintainer product/security review rather than closing the issue first.

## What This Changes

Add first-party cross-platform `openclaw migrate export` and `openclaw migrate import` commands for portable OpenClaw workspace, config, skills, memory, and related state.

## Best Possible Solution

Settle the first-party recovery model and CLI namespace first, then implement a guarded export/import or restore flow with preview, manifest validation, containment, rollback, conflict handling, secret policy, cross-platform path mapping, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source, docs, latest-release inspection, and related GitHub state give high-confidence proof that provider migration and backup create/verify exist while native OpenClaw-to-OpenClaw export/import remains absent.

## Solution Assessment

Unclear. The user problem is valid, but `migrate import` is not yet proven as the best core shape because backup restore, backup convert/platform targeting, guarded manual restore, or a narrower restore command may be safer.
