## Summary

Keep open. Current main has provider-based `openclaw migrate` imports and `openclaw backup create/verify`, but it still does not provide the requested same-product cross-platform archive export/import flow; live GitHub state also shows an open linked implementation PR, so cleanup should not close this before maintainer product/security review.

## What This Changes

The issue asks for `openclaw migrate export` and `openclaw migrate import` commands for cross-platform OpenClaw state migration.

## Best Possible Solution

Settle the native migration/restore safety model first, then either land a reviewed replacement for the linked PR or evolve `backup` with platform-aware export/convert/manual-restore guidance and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-main source and docs give a high-confidence feature-gap check: `migrate` is provider import, `backup` is create/verify, and machine-to-machine migration is still documented as manual copy/extract/doctor steps.

## Solution Assessment

Unclear. The user problem is valid, but prior review shows the exact solution needs maintainer product/security choice between a guarded archive import command and a safer backup convert/manual restore model.
