## Summary

Keep open. Current main still has provider-based imports under `openclaw migrate` and `openclaw backup create/verify`, but it does not provide the requested OpenClaw-to-OpenClaw cross-platform archive export/import flow; the issue is also linked to an open implementation PR that needs maintainer product/security review.

## What This Changes

The issue requests first-party `openclaw migrate export` and `openclaw migrate import` commands for cross-platform OpenClaw workspace, config, skills, memory, and path migration.

## Best Possible Solution

Settle the migration/restore safety model first, then either land a reviewed replacement for the linked PR or evolve `openclaw backup` with platform-aware export/convert/manual-restore guidance and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The feature-gap check is high-confidence from current source/docs: `migrate` is provider-based, `backup` is create/verify only, and machine-to-machine migration remains documented as manual copy/extract/doctor steps.

## Solution Assessment

Unclear. The user problem is valid, but the exact `migrate import` design is not yet proven as the best path because prior maintainer discussion raised a safer `backup` platform-targeting or conversion model.

## Security Review

Status: needs_attention

Summary: The requested native import/restore flow is security-sensitive because it would apply user-provided archives to credential, session, plugin, and workspace paths.
