## Summary

Keep open. Current main has provider-based `openclaw migrate` imports plus `backup create`/`verify` and manual machine-migration docs, but it still lacks the requested same-product cross-platform export/import or restore flow; the closest implementation work is unmerged and prior discussion treats archive restore/import as a product and security decision.

## What This Changes

The issue requests native `openclaw migrate export` and `openclaw migrate import` commands for selective, cross-platform OpenClaw state migration.

## Best Possible Solution

Choose a single first-party recovery model, then implement it as a hardened preview-first flow with explicit archive validation, secret policy, path remapping, conflict handling, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs provide high-confidence evidence that current main has provider imports plus backup create/verify, but not same-product cross-platform export/import.

## Solution Assessment

Unclear. The requested capability is useful, but the narrowest maintainable shape may be `backup restore`/`backup convert` or a redesigned preview-first import flow rather than adding archive import directly to the existing provider-migration namespace.
