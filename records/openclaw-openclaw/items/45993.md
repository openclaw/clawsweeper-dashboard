## Summary

Keep open. The requested first-party cross-platform export/import workflow is not implemented on current main; existing `backup` and `migrate` surfaces cover adjacent archive creation/verification and provider imports only, and the linked review history shows maintainers still need to choose a secure recovery model.

## What This Changes

Request adds `openclaw migrate export` and `openclaw migrate import` for selective cross-platform OpenClaw state/workspace migration with path remapping, environment checks, memory merge, and dependency warnings.

## Best Possible Solution

Settle the first-party recovery scope, then implement either safer backup export/convert/manual-restore improvements or a preview-first import with strict archive validation, canonical target checks, conflict/rollback handling, secret policy, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Current-main source and docs give a high-confidence feature-gap check: `migrate` is provider import/list/plan/apply, `backup` is create/verify, and machine migration docs still describe manual tar/copy plus doctor.

## Solution Assessment

Unclear until maintainers choose the recovery model. A safer path may be backup platform targeting or conversion with manual restore, while automated import would need to integrate with the existing provider-based migrate namespace and carry strong security hardening.
