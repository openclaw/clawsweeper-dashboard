## Summary

Keep open: current main and v2026.5.22 still expose provider-based `openclaw migrate` plus `backup create`/`backup verify`, but they do not implement native OpenClaw-to-OpenClaw cross-platform `migrate export`/`migrate import`. The request remains valid, with useful external Work Capsule context noted in the discussion, but the native import path is destructive and security-sensitive and overlaps open restore/migration PR work, so it needs maintainer product/security direction before implementation.

## What This Changes

Request to add `openclaw migrate export` and `openclaw migrate import` for selective cross-platform migration of OpenClaw workspace, skills, memory, config, secrets/env requirements, and dependency checks.

## Best Possible Solution

Settle the first-party portability model and command namespace, then implement a guarded export/import or restore flow with preview, archive validation, path remapping, secret policy, conflict handling, rollback, docs, and focused cross-platform tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source, docs, release-tag inspection, and live GitHub state confirm current main has provider import plus backup create/verify but not native OpenClaw-to-OpenClaw export/import.

## Solution Assessment

Unclear. The user problem is valid, but maintainers still need to choose whether this belongs under `migrate export/import`, `backup restore`, a profile portability command, or a split core/plugin model before implementation.
