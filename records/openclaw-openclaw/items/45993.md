## Summary

Keep open. Current main still lacks same-product cross-platform export/import or backup restore; `openclaw migrate` is a provider-import surface, `openclaw backup` only creates/verifies archives, and the related implementation proposals still need maintainer product and archive-security direction.

## What This Changes

Add `openclaw migrate export` and `openclaw migrate import` for selective cross-platform migration of OpenClaw state, workspace files, config, secrets, skills, and memory between machines.

## Best Possible Solution

Keep this as the same-product migration tracker until maintainers choose the recovery model and either review/replace the open external implementation or define a safer backup-based path with archive validation, secret handling, path remapping, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The feature-gap check is high-confidence: current CLI registration, implementation, docs, tests, and repository search show no same-product `migrate export`, archive `migrate import`, or `backup restore` surface on main.

## Solution Assessment

Unclear for the exact requested command shape. The user problem is valid, but the maintainable fix needs owner selection between extending `migrate`, extending `backup`, or documenting a safer manual conversion/restore workflow before implementation.
