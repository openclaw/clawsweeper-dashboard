## Summary

Keep open. Current main still lacks same-product cross-platform `migrate export`/archive `migrate import` or `backup restore`, and the remaining work needs maintainer product/security scoping rather than conservative cleanup closure.

## What This Changes

Add `openclaw migrate export` and `openclaw migrate import` for cross-platform migration of OpenClaw state, including selective export, path adaptation, env/secret handling, merge strategy, and dependency checks.

## Best Possible Solution

Keep this as the same-product migration tracker until maintainers choose the recovery model and either review/replace the open implementation or define a safer backup-based path with archive validation, secret handling, path remapping, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The feature-gap check is high-confidence: current main registers provider-based migrate/list/plan/apply plus backup create/verify, while docs still describe manual same-product machine moves.

## Solution Assessment

Unclear for the exact requested command shape. The user problem is valid, but maintainers need to choose between extending `migrate`, extending `backup`, or defining a safer documented restore workflow before implementation.
