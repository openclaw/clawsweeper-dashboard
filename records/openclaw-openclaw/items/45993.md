## Summary

Keep open: current main and v2026.5.19 still expose provider-based `openclaw migrate` plus backup create/verify only, while the requested OpenClaw-to-OpenClaw cross-platform export/import flow remains unresolved and has an open linked implementation that needs product/security review.

## What This Changes

Request adds `openclaw migrate export` and `openclaw migrate import` for selective, cross-platform OpenClaw state migration with path adaptation, secret/env handling, and merge choices.

## Best Possible Solution

Choose the recovery safety model first, then either evolve `openclaw backup` with platform-aware export/convert plus manual restore guidance or add a guarded import flow with preview, archive validation, rollback, conflict, and secret-handling rules.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source, docs, release-tag inspection, and GitHub state give high-confidence proof that create/verify and provider import exist while OpenClaw-to-OpenClaw archive export/import is absent.

## Solution Assessment

Unclear. The user problem is valid, but `migrate import` is not yet proven as the best core shape because backup conversion, manual restore guidance, profile portability, or a more guarded restore command may be safer.
