## Summary

Keep open. Current main still lacks the requested OpenClaw-to-OpenClaw cross-platform export/import flow, related duplicate discussions point back to this tracker, and the linked open implementation remains a product/security review item rather than a cleanup close.

## What This Changes

Request to add `openclaw migrate export` and `openclaw migrate import` for selective, cross-platform OpenClaw state migration with path adaptation, secret/env handling, merge choices, and dependency checks.

## Best Possible Solution

Choose the recovery safety model first, then either evolve `openclaw backup` with platform-aware export/convert and clearer manual restore docs or add a guarded import path with preview, archive validation, rollback, conflict, and secret-handling rules.

Not applicable as a bug reproduction. The feature-gap check is high-confidence from current source and docs: `migrate` is provider-based, `backup` is create/verify only, and machine-to-machine migration remains manual.

## Solution Assessment

Unclear. The user problem is valid, but `migrate import` is not yet proven as the best core shape because backup conversion, manual restore guidance, or a more guarded restore command may be safer.
