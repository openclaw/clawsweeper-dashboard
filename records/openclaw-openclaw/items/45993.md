## Summary

Keep open. Current main still has provider-based `openclaw migrate` imports plus `backup create`/`backup verify` and manual machine-migration docs, but it does not provide the requested same-product cross-platform export/import flow; the open fork PR with closing syntax is an implementation candidate, not a reason to close before merge.

## What This Changes

The issue requests `openclaw migrate export` and `openclaw migrate import` commands for selective cross-platform OpenClaw state migration with path adaptation, env/key checks, memory merge behavior, and dependency warnings.

## Best Possible Solution

Settle the first-party recovery model first, then implement a hardened preview-first flow with explicit archive validation, secret policy, path remapping, conflict handling, rollback behavior, and docs, coordinating with the broader backup/restore umbrella.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, current CLI registration, and related GitHub state give high-confidence proof that current main lacks same-product cross-platform export/import.

## Solution Assessment

Unclear. The requested capability is useful, but maintainers still need to choose whether the maintainable shape is `migrate export/import`, `backup restore`, `backup convert`, or a manual restore flow backed by stronger verification.
