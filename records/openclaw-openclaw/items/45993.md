## Summary

Keep open: current main and v2026.5.20 still expose provider-based migration plus backup create/verify, but not native OpenClaw-to-OpenClaw export/import with cross-platform path adaptation. The request remains valid, but it is broad, destructive, security-sensitive, and currently overlaps open restore/profile/migration work plus a fork PR with closing syntax, so it needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Add first-party `openclaw migrate export` and `openclaw migrate import` commands for selective cross-platform OpenClaw state, workspace, skill, memory, config, and secret-aware migration.

## Best Possible Solution

Settle the first-party recovery model and command namespace first, then implement a guarded export/import or restore flow with preview, archive validation, containment, rollback, conflict handling, secret policy, cross-platform path mapping, docs, and focused tests.

Not applicable as a bug reproduction; this is a feature gap. Source, docs, latest-release inspection, and related GitHub state give high-confidence proof that provider migration and backup create/verify exist while native OpenClaw-to-OpenClaw export/import remains absent.
