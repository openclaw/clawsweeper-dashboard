## Summary

Keep open. Current main has partial responsiveness work, including backend channel-status fan-out and #75986's timing/secondary-refresh decoupling, but source inspection and maintainer routing show the broader Control UI degradation remains unresolved.

## What This Changes

Keep the Control UI responsive over time when channel status probes, provider reconnects, polling, and session/usage refreshes become slow.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that bounds slow channel status/probes, scopes polling to active views, coalesces session reloads, and guards or aborts stale long-running RPC results with regression coverage.

Source inspection does confirm current-main contributors: the 8s channels.status wait, non-abortable browser RPCs, global node polling, and non-chat sessions.changed reloads.

## Solution Assessment

No. Current main is an improvement, but it is not yet the best complete solution because the remaining polling and churn paths are still present and already have overlapping open PRs that need reconciliation.
