Keep open. Current main has backend channel-status fan-out and the merged timing/secondary-refresh work from #75986, but important Control UI contributors from the report remain present, and the latest maintainer routing comment says #75986/#64004 should not close this while #45930 and #69454 are still unresolved.

## What This Changes

The issue asks for the Control UI to remain responsive over time when slow channel status probes, provider reconnect churn, polling, and session reloads occur.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that scopes or slows polling, coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow channel status probes from blocking broader navigation, and adds regression coverage for slow status plus session-event churn.

## Reproduction Assessment

Unclear for a high-confidence end-to-end reproduction: no live long-running dashboard with a slow provider was run. Source inspection does show the current-main contributors reported here, including the 8s channels.status wait, non-abortable browser RPCs, global node polling, and immediate session reloads for non-chat-turn session events.

## Solution Assessment

No, current main is not yet the best solution. Backend status fan-out and #75986's timing/secondary-refresh changes help, but the remaining UI-side churn should be reconciled through the already-linked Control UI PRs instead of closing this issue now.
