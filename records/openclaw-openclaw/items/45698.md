## Summary

Keep this open. Current main has backend channel-status fan-out, but the Control UI-side contributors in the report remain present: the Channels tab still issues an 8s status request, the browser RPC layer has no abort or stale-response guard, node polling still starts globally on connect, and every `sessions.changed` event still triggers an immediate session reload. Open PRs #45930 and #69454 cover overlapping parts of the same responsiveness problem, so this should be reconciled by a maintainer rather than closed or split into another automated repair branch.

## What This Changes

The issue asks for the Control UI to stay responsive over long-lived dashboard sessions even when channel status probes, provider reconnects, or session updates are slow or noisy.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that scopes polling to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow channel status probes from blocking broader navigation, and adds regression coverage for slow status responses plus session-event churn.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction: it requires a long-lived dashboard and a slow or reconnecting provider. The source-level contributors are high-confidence on current main: 8s channel status waits, non-abortable browser RPCs, global node polling, and immediate session-change reloads.

## Solution Assessment

No, current main is not yet the best solution. Backend channel-status concurrency helps, but the UI still lacks stale-request handling, active-view polling, and session-reload controls needed to resolve the progressive stuck behavior.
