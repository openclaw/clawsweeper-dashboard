## Summary

Keep open. Current main still leaves the reported Control UI degradation paths unresolved: backend channel-status fan-out exists, but the UI still has an 8s Channels status wait, non-abortable browser RPCs, global node polling, and immediate session reloads; related PRs #45930, #69454, and #75986 cover overlapping slices rather than a merged terminal fix.

## What This Changes

The issue asks for the Control UI to remain responsive during long-lived dashboard sessions even when channel status probes, provider reconnects, or session updates are slow or noisy.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that scopes polling to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow channel status probes from blocking broader navigation, and adds regression coverage for slow status responses plus session-event churn.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction: it requires a long-lived dashboard and a slow or reconnecting provider. The current source-level contributors are high-confidence: 8s channel status waits, non-abortable browser RPCs, global node polling, and immediate session-change reloads.

## Solution Assessment

No, current main is not yet the best solution. Backend channel-status concurrency helps, but the UI still lacks stale-request handling, active-view polling, and session-reload controls needed to resolve the progressive stuck behavior.
