## Summary

Keep open. Current main has partial responsiveness work, including channel-status fan-out and non-blocking secondary refreshes, but source inspection still shows the reported slow-status, polling, and session-reload contributors, and the latest maintainer routing says the remaining work belongs in the related open Control UI PRs rather than closing this issue now.

## What This Changes

The issue asks for the Control UI dashboard to remain responsive over time when channel status probes, reconnect churn, tab switching, polling, and session refreshes become slow.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that bounds slow status probes, keeps stale or slow tab work from updating the wrong view, scopes or reduces background polling, coalesces session reloads, and adds focused regression coverage before closing this issue with release proof.

## Reproduction Assessment

No, not as a high-confidence end-to-end reproduction. Source inspection shows current-main contributors matching the report, including the 8s channel-status request budget, non-abortable browser RPCs, global node polling, and immediate session reloads for unpatchable sessions.changed events, but no live slow-provider dashboard run was executed.

## Solution Assessment

No, current main is not yet the full solution. The partial backend fan-out and refresh-decoupling work are useful, but the remaining UI-side churn should be reconciled through the open Control UI PRs before this issue is closed.
