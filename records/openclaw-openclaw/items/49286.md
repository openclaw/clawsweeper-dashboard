## What This Changes

This PR expands Discord `MessageType` system-event mapping, adds mapper and inbound preflight regression tests for native Discord system messages, and adds a changelog entry.

## Best Possible Solution

Land one current-main-compatible Discord fix that adds the missing native `MessageType` cases in the Discord mapper, keeps `Default`, `Reply`, `ThreadStarterMessage`, `ChatInputCommand`, and `ContextMenuCommand` as passthroughs, updates mapper plus preflight queue/drop tests against `system-event-runtime`, and includes a properly credited changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: on current main, a `MessageType.ChannelNameChange` message reaches `resolveDiscordSystemEvent`, falls through to `null`, and therefore does not trigger the preflight `enqueueSystemEvent`/drop branch.

## Solution Assessment

Is this the best way to solve the issue? Not as-is: the mapper expansion is the right narrow behavior change, but the branch needs rebase updates for current imports/runtime mocks and the changelog credit before it is the best landable fix.
