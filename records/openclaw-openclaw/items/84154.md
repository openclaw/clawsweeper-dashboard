## Summary

Keep open: current main still records and logs channel admission as transient turn state, but I did not find a durable per-message admission/reason record that would explain an accepted Telegram message recorded without an immediate run. Related fixes cover narrower stale-thread, stale-reply, or queue-claim paths, and the open hook PR is narrower than this diagnostic gap.

## What This Changes

The issue asks OpenClaw to either dispatch normal Telegram inbound messages immediately or durably record the channel admission/reason when an accepted message is recorded without a run.

## Best Possible Solution

Add a shared, redacted durable channel-ingress/turn admission diagnostic surface for accepted channel messages, covering dispatch/observeOnly/skip/drop, decisive gates, record-vs-pending state, and active/stale-run suppression without relying on transient logs.

## Reproduction Assessment

No high-confidence live Telegram reproduction was run. Source inspection does reproduce the diagnostic gap: admission/reason data exists during ingress but is not durably recorded with the session or queue record for later analysis.

## Solution Assessment

No, the issue is not solved by the current narrow fixes or by a Telegram-only hook. The best fix is a shared durable admission diagnostic surface with maintainer-scoped retention and redaction rules.
