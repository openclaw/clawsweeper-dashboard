## Summary

Keep open: current main still records inbound channel session metadata before dispatch and carries admission/reason as transient runtime/log state, but I found no durable per-message admission, reasonCode, or decisive-gate record that would explain an accepted Telegram message recorded without an immediate run.

## What This Changes

The issue asks OpenClaw to persist or expose Telegram/channel ingress and turn admission diagnostics for messages recorded into sessions without dispatch.

## Best Possible Solution

Add a shared, redacted durable channel-turn admission diagnostic surface, then use it to identify and fix any Telegram-specific suppression gate.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection shows the durable admission-diagnostic gap and record-before-dispatch path, but the exact Telegram no-run-until-next-message trigger was not reproduced on current main.

## Solution Assessment

Yes for the diagnostic direction, but not as a Telegram-only guess. The narrow maintainable path is shared redacted admission persistence first, followed by a targeted Telegram fix if the stored reason identifies a Telegram-specific gate.
