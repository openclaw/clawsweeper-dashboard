## Summary

Keep open: current main still has shared channel-turn admission concepts and coarse transient diagnostics, but it does not provide a durable, redacted record explaining why an accepted Telegram message was recorded into session context without dispatching a run. The related fixes cover narrower stale-thread or stale-lock causes, not the generic durable admission/reason trail requested here.

## What This Changes

The issue asks OpenClaw to dispatch accepted Telegram/channel messages immediately or persist durable ingress admission diagnostics when dispatch is suppressed.

## Best Possible Solution

Add a generic, redacted durable channel-turn diagnostic record that captures admission, reason/decisive gate, session-record versus pending-history state, and active/stale run suppression state, then use that trail to fix any remaining Telegram-specific dispatch suppression.

## Reproduction Assessment

No for the exact Telegram no-dispatch-until-next-message behavior: the reporter provided a plausible but nondeterministic scenario and no live current-main repro was established. Source inspection does reproduce the durable diagnostic gap because current main has transient admission/log events and coarse message diagnostics, but no durable per-message admission trail.

## Solution Assessment

Unclear for a direct Telegram-only repair. The best maintainable path is a shared redacted channel-turn diagnostic surface first, followed by a targeted Telegram/session fix once the next occurrence has an admission and reason trail.
