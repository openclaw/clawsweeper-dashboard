## Summary

Keep open: current main and the latest release still leave a source-reproducible pure-text subagent completion loss path when active requester wake fails and the requester-agent handoff then hits a session write-lock error; the related direct-delivery PR remains open and unmerged.

## What This Changes

The issue asks for subagent completion delivery to survive inactive requester sessions and locked requester transcripts by using a bounded direct-message fallback when a usable direct target exists.

## Best Possible Solution

Land or replace the bounded direct-completion fallback with long-output protection and real external-channel proof, while keeping the broader session write-lock contention work tracked in the canonical lock issue.

## Reproduction Assessment

Yes, source-reproducible: current main still rethrows pure-text SessionWriteLock handoff failures after active requester wake failure, and the adjacent test still expects no direct sendMessage call. I did not run a live Feishu gateway reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution yet. The maintainable fix should bound direct fallback text, preserve generated-media recovery, add focused regression coverage, and keep the broader lock-lifetime decision separate or explicitly in scope.
