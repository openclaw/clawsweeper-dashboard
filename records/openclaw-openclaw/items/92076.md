- `impact:message-loss`: A completed subagent result can be suppressed or lost before any visible completion message reaches the user.

## Summary

Keep open: current main and v2026.6.8 still have the pure-text direct-message loss path after requester inactivity or SessionWriteLock handoff failure; the narrower candidate PR was closed and the remaining open PRs are overlapping or blocked rather than a merged fix.

## What This Changes

The issue requests a capped direct-message fallback for top-level subagent text completions when the requester run is inactive or cannot be woken and requester-agent transcript delivery is blocked by a session lock.

## Best Possible Solution

Land a focused agents-path fix that sends capped, mirrored direct text for direct-message subagent completions when the requester is already inactive, active wake fails with the relevant reasons, or requester-agent handoff hits SessionWriteLock, with regression tests and external-channel proof; keep orphan lock cleanup and cron route leases separate.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.8 shows pure-text subagent completions still fall through to requester-agent handoff and can rethrow SessionWriteLock failures instead of using direct-message text fallback.

## Solution Assessment

No close yet. Current main is a partial mitigation for generated media and no-visible-output cases, but the best fix must cover already-inactive requester sessions and lock-failed pure-text handoff with bounded direct-message delivery.
