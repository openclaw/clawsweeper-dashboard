## Label Justifications

- `P1`: A completed background task can be lost in a real direct-message workflow after the requester run times out or is evicted.
- `impact:message-loss`: The user may never receive an already completed subagent result.
- `impact:session-state`: The failure depends on requester-run inactivity and contention on the requester session transcript.

## Summary

Current main only uses text-direct delivery after an incomplete announce-agent result; a requester transcript lock triggers direct fallback only for generated media, so the reported text-completion failure remains actionable.

## What This Changes

Make successful top-level subagent completion text deliverable to a direct-message target when active requester wake fails and requester-agent transcript writing is locked.

## Best Possible Solution

Extend the existing transport-neutral text-direct helper to the active-wake-failed session-lock branch, preserving idempotency and partial-delivery detection while applying a bounded user-visible completion payload and best-effort transcript mirroring.

## Reproduction Assessment

Yes at source level: an inactive requester can force requester-agent handoff, and a session-write-lock error only activates generated-media direct fallback, leaving successful text completion without the equivalent recovery path. The supplied v2026.6.1 logs match this branch sequence, though no live current-main run was performed.

## Solution Assessment

Yes in direction, with refinement: reuse the existing direct text delivery seam rather than create a new channel-specific path, and add explicit idempotency, partial-send, payload-bounding, and transcript-lock regression coverage.
