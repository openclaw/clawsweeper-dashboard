## Summary

The issue remains necessary because current main can still lose a successful text completion after requester wake fails and transcript-backed handoff hits a session lock. The proposed raw direct-result fallback is not yet the best fix because it bypasses the intentional requester-agent mediation/privacy boundary; the separate orphan-self-lock defect is already tracked by https://github.com/openclaw/openclaw/issues/49603.

## What This Changes

Prevent successful subagent completions from being lost when the requester is inactive and transcript-backed completion handoff is blocked by a session lock.

## Best Possible Solution

Design a privacy-safe, transcript-independent completion envelope for the inactive-requester path, preserving requester mediation where possible and handling same-PID orphan-lock recovery separately in the canonical lock issue.

## Reproduction Assessment

Yes at source level: current main follows the reported `no_active_run` outcome into transcript-backed requester-agent handoff, and the reporter supplied matching Feishu logs and focused local runtime checks. An independent live current-main execution was not performed.

## Solution Assessment

No, not as currently proposed: immediately forwarding capped raw child output bypasses an intentional mediation/privacy boundary. The repair should provide transcript-independent delivery without exposing unreviewed child content, while the orphan-lock change remains with the canonical lock issue.
