## Label Justifications

- `P1`: A completed subagent result can fail to reach a real user in a direct-message workflow even though a usable external delivery route exists.
- `impact:message-loss`: The central user-visible symptom is a completed child result being suppressed rather than delivered.
- `impact:session-state`: Requester-run eviction and transcript lock state determine whether the completion delivery path succeeds.

## Summary

Current main still sends a failed active requester wake into the requester-agent handoff before trying the existing direct-text fallback, so a locked requester transcript can still suppress a completed subagent result. The merged stale-announce repair improved retry and message-tool mediation but intentionally did not add raw direct-send fallback, while the broader lock-contention report covers a distinct systemic failure.

## What This Changes

Ensure a top-level subagent completion can reach a direct-message requester when the requester run is inactive and transcript-backed requester-agent delivery is locked or unavailable.

## Best Possible Solution

Reuse the existing direct-message text-delivery seam immediately after requester inactivity or a no_active_run wake result, before transcript-dependent requester-agent handoff, with focused bounded-output and routing regression coverage; keep whole-turn and orphan-lock lifecycle changes in their canonical lock work.

## Reproduction Assessment

Yes at source level: current main explicitly routes a failed active wake into requester-agent handoff before direct-text recovery, matching the reported no_active_run followed by transcript-lock failure. The reporter also supplied a release-runtime log sequence and local hotfix validation, but this review did not execute a live current-main reproduction.

## Solution Assessment

Yes, with a narrower boundary than the full issue discussion: moving the existing direct-message text fallback ahead of transcript-dependent handoff is the smallest maintainable fix for the original symptom. Redesigning session-lock duration or general stale-lock recovery is separate work.
