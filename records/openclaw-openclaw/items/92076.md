## Label Justifications

- `P1`: Subagent completion can be silently lost in real external-message workflows when requester wake and transcript handoff are unavailable.
- `impact:message-loss`: The reported failure drops the user-visible subagent completion instead of delivering a direct or background message.
- `impact:session-state`: The failure depends on requester session activity and transcript lock state, and the proposed recovery changes how completion output is mirrored to session state.

## Summary

Keep open: current main and v2026.6.6 still route active requester wake failure into requester-agent handoff, and pure-text SessionWriteLock failures only get direct fallback in narrower incomplete-output paths. The related fix PR https://github.com/openclaw/openclaw/pull/92643 is open but currently conflicting, so it is not a safe close reason yet.

## What This Changes

The issue reports lost pure-text subagent completion delivery when active requester wake fails and the inactive or locked requester transcript handoff hits SessionWriteLockTimeoutError, plus a related orphan self-lock recovery symptom.

## Best Possible Solution

Land a bounded fix that sends concise direct text only for direct-message subagent completions when active wake or locked handoff fails, preserves generated-media recovery, adds long-output and lock-error regression coverage, and tracks orphan self-lock cleanup through the session-lock issues if separate.

## Reproduction Assessment

Yes from source inspection, though I did not live-reproduce Feishu. Current main routes active wake failure into requester-agent handoff, and pure-text SessionWriteLock errors skip direct text fallback because the catch path only recovers generated media.

## Solution Assessment

Unclear until the open PR is repaired or replaced. The owner surface is plausible, but the best mergeable fix needs bounded direct text, regression coverage, and proof while keeping the broader orphan-lock lifecycle separate if needed.
