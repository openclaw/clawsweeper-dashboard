- `P1`: The issue reports real user-visible loss of subagent completion messages in an agent/channel workflow, and current main still shows the pure-text failure path.
- `impact:message-loss`: The reported symptom is a completed subagent result not reaching the user after requester wake and transcript handoff fail.

## Summary

Keep open: current main and the latest release still leave the pure-text subagent completion path dependent on requester-agent handoff after active wake failure, and the open repair PR is not merged.

## What This Changes

The issue requests direct, bounded subagent completion delivery when the requester run is inactive or the requester transcript path is locked.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/92643 with a bounded direct-text fallback, long-output coverage, and real external-channel proof, while keeping broader session-lock lifetime work tracked separately.

## Reproduction Assessment

Yes, source-reproducible from current main though not live-reproduced here. The active-wake failure path still falls into requester-agent handoff, and the SessionWriteLock catch only tries direct recovery for generated media, not pure text.

## Solution Assessment

Yes, a bounded direct fallback in `src/agents/subagent-announce-delivery.ts` is the right first fix for this failure mode. It should cap or artifact-link long child output and leave broader lock-lifetime cleanup to the related session-lock work if needed.
