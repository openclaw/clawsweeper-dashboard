## Summary

Keep open: current main still has a plausible Zalouser direct-message identity bug at the plugin boundary, and the reporter supplied concrete logs showing the self-ID misroute. The fix should be narrow and plugin-owned.

## What This Changes

The issue reports that first-contact Zalo Personal direct messages can create sessions and delivery targets for the bot's own Zalo ID instead of the sender's peer ID.

## Best Possible Solution

Normalize the remote direct peer inside the Zalouser plugin before routing, rejecting the account's own Zalo ID and covering first-contact DMs with regression tests.

## Reproduction Assessment

Yes at source level, not by live replay: if a direct inbound Zalo message normalizes to the bot account id, current main builds the direct session and reply target from that id. The issue provides concrete first-contact logs for that event shape.

## Solution Assessment

Yes: the narrow maintainable fix is in the Zalouser normalization/routing boundary. Core routing already behaves deterministically once the plugin supplies the correct peer, so a core workaround or new config knob would be broader than needed.
