## Summary

This PR is a valid, focused update-flow fix, but it should stay open because the patch can still miss the managed service package root during the real non-dry-run package install path for the direct managed npm layout shown in the supplied proof.

## What This Changes

The PR redirects package-mode updates to the managed gateway service package root when it differs from the invoking CLI root and adds a CLI regression test for that mismatch.

## Best Possible Solution

Carry a resolved package update target that is proven to match the managed service root, including the direct managed npm root shown in the proof, or refuse redirection for roots the updater cannot safely mutate.

## Reproduction Assessment

Yes by source inspection and the PR's real dry-run proof: current main updates from the invoking root, while a real managed LaunchAgent can point at a different package root. I did not execute the destructive update path in this read-only review.

## Solution Assessment

No. Selecting the service package root is the right direction, but the patch must also make the package install target resolve to that same root for the managed npm layout before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor the selected service root during install:** `src/cli/update-cli/update-command.ts:798`
  - body: Returning `layout.packageRoot` here is not enough to make the actual package install target that root. The supplied real proof uses `$HOME/.openclaw/npm/node_modules/openclaw`, but the existing npm target resolver only recognizes normal `lib/node_modules` global-prefix layouts and otherwise falls back through `npm root -g`; a non-dry-run update can still mutate the shell/global npm install instead of the managed service root. Please either teach the package target resolver to honor this managed root or avoid redirecting to unsupported roots.
  - confidence: 0.88
