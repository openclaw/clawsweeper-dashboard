## Summary

This PR should close as superseded: it identifies a real managed-service update bug and includes useful proof, but the open maintainer PR at https://github.com/openclaw/openclaw/pull/83396 now owns the same root-selection fix and adds the package-root target handling missing from this branch.

## What This Changes

The PR redirects package-mode `openclaw update` planning to the managed gateway service package root when it differs from the invoking CLI root, with a regression test for that mismatch.

## Best Possible Solution

Keep the managed-service package-root fix in the canonical maintainer PR, or revive this branch only if it also makes the actual package-manager install target honor the selected managed root and covers the source-checkout mismatch path.

## Reproduction Assessment

Yes. The PR body gives a real LaunchAgent/source-checkout mismatch proof, and source inspection shows the update path where the selected root can diverge from the invoking root.

## Solution Assessment

No. Redirecting `root` is necessary, but this branch does not fully tie package-manager target resolution to the managed service package root; the superseding maintainer PR is the more complete fix path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep package-manager targeting tied to the redirected root:** `src/cli/update-cli/update-command.ts:2916-2919`
  - body: When invoked from a source checkout and redirected to the managed package root, `installKind` still describes the invoking root. `resolveGlobalManager` only detects the manager for `params.root` when `installKind === "package"`, so the real update can still choose the shell global manager/root even though dry-run prints the managed root. Please pass a package-root/honor-root signal and cover the source-checkout mismatch path, not only the package-shell mismatch.
  - confidence: 0.86
