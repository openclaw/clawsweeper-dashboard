Keep this PR open: the patch is a focused fix for a real source-visible update-target bug, and I did not find a blocking code defect, but the external-PR real behavior proof gate is not satisfied because the body only provides mocked test evidence.

## What This Changes

The PR redirects package updates from the invoking CLI package root to the managed gateway service package root when those package installs differ, and adds a CLI regression test for that mismatch.

## Best Possible Solution

Land this narrow managed-service root selection after real managed-service proof or an explicit maintainer proof override, preserving the existing service-layout and update-global helper boundaries.

## Reproduction Assessment

Yes by source inspection: current main chooses the invoking CLI package root and carries it into package update execution, while the existing service command layout can point at a different managed-service package root. I did not run the destructive live update path in this read-only review.

## Solution Assessment

Yes, the proposed approach is the narrowest maintainable fix because it reuses the existing service-layout summary and only redirects package updates when the service package root differs and is not a source checkout. The remaining blocker is proof from a real managed-service setup, not a different implementation direction.
