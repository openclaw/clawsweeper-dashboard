## Label Justifications

- `P1`: The issue affects real Discord, Telegram, and Feishu message-send workflows and can block ordinary outbound sends or add long recovery latency.
- `impact:message-loss`: The reported failures reject outbound channel sends before the intended message or attachment is delivered.

## Summary

Keep open. Current main still exposes poll creation fields in the broad message tool schema and still rejects ordinary `action: "send"` calls when model-filled poll-looking params are present; an open linked repair PR exists but has not landed.

## What This Changes

The issue asks for ordinary `message.send` to stop failing when GPT-style models auto-populate poll fields or Discord component/modal skeletons from an over-broad message tool schema.

## Best Possible Solution

Land or replace one canonical repair that treats model-filled no-op poll/component defaults as absent for ordinary sends, preserves strict validation for explicit polls and real modals, and makes Discord native component schema exposure an intentional opt-in contract.

## Reproduction Assessment

Yes, by source inspection. A `runMessageAction()` call with `action: "send"` and non-empty poll-looking params still throws before channel dispatch; I did not run a live GPT-family channel flow.

## Solution Assessment

Unclear as a single proposed path. The normalization direction is right, but maintainers need to consolidate the poll intent rule and Discord component schema exposure before landing one canonical fix.
