## Label Justifications

- `P1`: The issue affects real Discord, Telegram, and Feishu message-send workflows and can block ordinary outbound sends or add long recovery latency.
- `impact:message-loss`: The reported failures reject outbound channel sends before the intended message or attachment is delivered.

## Summary

Keep open. Current main and the latest release still have a source-reproducible ordinary-send failure for model-filled poll params, and the Discord component/modal side is split between current no-schema behavior, a still-strict runtime parser, and open PRs that need maintainer contract consolidation.

## What This Changes

The issue requests a narrower or intent-aware `message.send` contract so ordinary sends ignore model-filled poll, component, and modal defaults unless the user explicitly asks for those advanced behaviors.

## Best Possible Solution

Land or replace one canonical repair that treats model-filled no-op poll/component defaults as absent for ordinary sends, preserves strict validation for explicit polls and real modals, and makes any Discord native component schema exposure an intentional opt-in contract.

## Reproduction Assessment

Yes, by source inspection. A `runMessageAction()` call with `action: "send"` and non-empty poll-looking params still throws before channel dispatch, and object-shaped Discord `components.modal` with empty fields still throws if it reaches the parser; I did not run a live GPT-family channel flow.

## Solution Assessment

Unclear as a single proposed path. The normalization direction is right, but maintainers need to consolidate the poll intent rule, Discord component schema exposure, and the separate media/config follow-ups before landing one fix.
