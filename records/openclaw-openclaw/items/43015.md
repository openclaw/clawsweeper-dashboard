## Summary

Keep open. Current main still has the source-reproducible poll false-positive and Discord empty-modal failure paths, and the linked implementation candidates remain open rather than merged; the newer Telegram `actions.poll` config-schema subpoint appears addressed on main, but the central `message.send` reliability bug is not fixed.

## What This Changes

The issue asks for ordinary `message.send` calls to ignore or avoid auto-populated poll, Discord component, and modal fields unless those advanced behaviors are explicitly requested.

## Best Possible Solution

Land a narrow normalization fix that keeps explicit `poll` validation, makes `send` robust to accidental poll/default component fields, treats empty Discord modal skeletons as absent, and tracks broader tool-shape or media-policy improvements separately.

## Reproduction Assessment

Yes. A high-confidence source reproduction remains: a `message.send` payload with nonzero/default-filled poll params reaches the send-side poll guard, and Discord object components with `modal.fields: []` reach a parser that throws.

## Solution Assessment

Partly. The semantic normalization direction is the right narrow fix, but current main has not implemented it and broad per-channel tool splitting or media diagnostics should not block the bug fix.
