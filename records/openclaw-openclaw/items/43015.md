## Summary

Keep open. Current main and the latest release still expose shared poll fields in the unified message tool schema and still reject ordinary `action: "send"` calls when populated poll-looking params are present; PR #46455 remains the direct open implementation candidate for this issue.

## What This Changes

The issue asks for `message.send` schema/runtime behavior that prevents ordinary sends with files from being rejected or altered by auto-populated poll fields or empty Discord components/modal skeletons.

## Best Possible Solution

Review, land, or explicitly supersede PR #46455 with a focused current-main patch that makes advanced poll/component fields action-aware for ordinary sends, preserves strict validation for explicit advanced actions, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection and existing tests provide a high-confidence reproduction path: `message.send` with populated poll-looking params still reaches the send-side guard and rejects before dispatch.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should settle the narrowest schema/runtime split through the open linked PR so normal sends tolerate model-populated noise without weakening explicit poll or modal validation.
