## Label Justifications

- `P1`: The issue blocks real outbound channel send workflows when ordinary messages or attachments are rejected before delivery.
- `impact:message-loss`: The failure prevents intended outbound channel messages or attachments from being delivered.

## Summary

Keep open. Current main still rejects ordinary `action: "send"` calls before channel dispatch when model-filled poll-looking params are present, and the Discord component parser still treats an empty `components.modal` object as enabled; several related repair PRs are open and need one maintainer-approved contract.

## What This Changes

Request is to narrow `message.send` schema/runtime so plain sends ignore model-filled poll fields and empty Discord component/modal skeletons unless poll or component behavior is explicitly requested.

## Best Possible Solution

Consolidate the linked repair work into one maintainer-approved contract that keeps explicit poll and real Discord component/modal validation strict while treating no-op model-filled advanced fields as absent for ordinary sends.

## Reproduction Assessment

Yes, by source inspection. A `runMessageAction()` call with `action: "send"` plus non-empty, non-zero, or true poll creation params still reaches the pre-dispatch poll guard, and tests explicitly preserve that rejection behavior.

## Solution Assessment

Unclear as a final implementation path. The bug boundary is clear, but maintainers still need to choose the canonical contract among stripping poll params, narrowing intent detection, and ignoring empty Discord component skeletons without breaking explicit advanced sends.
