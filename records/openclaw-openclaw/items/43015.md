## Label Justifications

- `P1`: The issue can block real channel sends or attachments before delivery across Telegram, Discord, and Feishu-style message workflows.
- `impact:message-loss`: The reported failures reject intended outbound messages before the user-visible message or attachment is delivered.

## Summary

Keep open. Current main still exposes poll fields on the mixed-action message tool schema and still rejects ordinary `action: "send"` calls when model-filled poll-looking params are present; the linked repair PR is open and has not landed.

## What This Changes

The issue asks OpenClaw to keep auto-filled poll and Discord component/modal fields from breaking ordinary `message.send` sends and attachments.

## Best Possible Solution

Adopt one maintainer-approved message tool contract that keeps ordinary sends narrow or strips no-op model-filled poll/component fields before send dispatch, while preserving explicit poll/modal validation and focused regression tests.

## Reproduction Assessment

Yes, by source inspection. On current main, a `runMessageAction()` call with `action: "send"` and non-empty, non-zero, or true poll creation params still hits the guard before dispatch; reporter comments also confirm Telegram failures.

## Solution Assessment

Unclear as a final implementation path. Treating model-filled no-op advanced fields as absent is the right direction, but maintainers still need to choose the canonical schema split/sanitization and component-modal contract rather than stacking another partial workaround.
