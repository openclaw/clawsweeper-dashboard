## Summary

Keep open. Current main still has the source-reproducible poll and Discord modal default-fill failures, while the likely fix path is split across still-open PRs that need rework/proof before this issue can close.

## What This Changes

The issue asks OpenClaw to keep ordinary `message.send` calls from failing when GPT-family models auto-fill poll or Discord component/modal fields from the generic message schema.

## Best Possible Solution

Land a corrected, source-covered normalization fix, preferably by repairing or replacing the open normalization PR, while leaving media-root policy and any remaining Telegram action-config mitigation to separate focused follow-up if still needed.

## Reproduction Assessment

Yes. A high-confidence source reproduction remains: `message.send` with non-zero/default poll-looking params still reaches the poll-only guard, and a Discord components object with `modal.fields: []` still reaches the parser throw path.

## Solution Assessment

Yes for the semantic normalization direction, but not yet on current main. The best path is to repair or replace the open implementation PRs with a narrow fix that preserves intentional poll validation and treats empty advanced-field defaults as absent.
