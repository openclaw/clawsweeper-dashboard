## Summary

Keep open: current main and v2026.6.6 still source-reproduce the remaining send-blocking cases for schema-padded channel-extra poll fields and empty Discord modal skeletons, while the linked repair paths are open/conflicting or closed unmerged.

## What This Changes

Make generic `message.send` robust to GPT-populated poll and Discord component/modal fields so plain sends do not fail or route as advanced actions unless explicitly requested.

## Best Possible Solution

Land one maintainer-approved repair that treats content-bearing poll fields as the send-blocking intent signal, normalizes empty Discord component/modal skeletons as absent, and preserves explicit poll/component actions with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message.send` still rejects when channel-extra poll fields make `hasPollCreationParams` true, and `components.modal.fields: []` still throws in the Discord parser.

## Solution Assessment

Unclear as a final product choice. The partial merged shared-poll fix is useful but does not cover channel-extra poll fields or empty modal skeletons, so maintainers should consolidate schema scoping and runtime normalization rather than layering another narrow default exception.
