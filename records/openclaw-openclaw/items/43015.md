## Summary

Keep open. Current main still has the source-reproducible `message.send` false-positive path for poll-looking fields, and Discord still treats an empty `components.modal.fields` skeleton as a hard modal opt-in; the Telegram config gate improvement only mitigates one related operator workaround.

## What This Changes

The issue asks for ordinary `message.send` calls to avoid exposing or hard-failing on auto-populated poll and Discord component/modal fields.

## Best Possible Solution

Land a narrow schema/normalization fix that preserves explicit poll/modal validation while treating model-filled advanced-field noise as absent from plain sends.

## Reproduction Assessment

Yes. Source inspection on current main shows `action="send"` plus poll-looking params still reaches the poll-only guard, and a Discord `components.modal.fields: []` skeleton still reaches the parser throw path.

## Solution Assessment

Yes for the narrow normalization and schema-scoping direction, but it is not implemented on current main. The safer path is to review, repair, or replace the linked fork PR rather than leaving the broad send schema coupled to strict advanced-field validation.
