## Summary

Keep open. Current main still has a source-reproducible poll false positive for ordinary `message.send` calls, and the Discord empty-modal runtime path still throws if an auto-filled skeleton reaches it; the Telegram `actions.poll` config mitigation is present now, but it does not resolve the main schema/runtime coupling bug.

## What This Changes

The issue asks for `message.send` schema/runtime behavior to stop treating auto-populated poll and Discord component/modal fields as intentional advanced actions.

## Best Possible Solution

Land a narrow normalization/schema fix that keeps explicit poll/modal validation intact but strips or ignores auto-filled advanced fields for `action="send"`; repair or replace the open fork PR if it remains the best source patch.

## Reproduction Assessment

Yes. There is a high-confidence source reproduction on current main: `action="send"` with poll-looking params still reaches the poll-only guard, and a Discord `components.modal.fields: []` skeleton still reaches the parser throw path.

## Solution Assessment

Yes for the narrow normalization and schema-scoping direction, but it is not implemented on current main. The safer fix is to preserve intentional poll/modal errors while treating empty or model-default advanced fields as absent from plain sends.
