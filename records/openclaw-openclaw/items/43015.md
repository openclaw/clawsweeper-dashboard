- `impact:message-loss`: The reported failures reject intended outbound messages or attachments before they are delivered.

## Summary

Keep open. Current main and the latest release still source-reproduce the poll false-positive path for broad message schemas, and Discord still rejects an empty `components.modal` skeleton if that object reaches the send path; the linked repair pull request is still open rather than landed.

## What This Changes

The issue reports that GPT-family models populate advanced poll/component/modal fields exposed on the generic message tool, causing ordinary sends and attachments to fail.

## Best Possible Solution

Land or replace the linked repair with a maintainer-approved contract that treats model-filled no-op poll/component defaults as absent for plain sends while preserving explicit poll and real modal validation with regression coverage.

## Reproduction Assessment

Yes, at source level. A `runMessageAction()` call with `action: "send"` and non-empty poll-looking params still reaches the guard before dispatch, and a present object-shaped Discord `components.modal` skeleton still reaches strict empty-fields validation.

## Solution Assessment

Unclear as a final implementation path. Treating model-filled no-op advanced fields as absent is the right direction, but maintainers still need to choose the canonical schema split, sanitization, and modal-validation contract.
