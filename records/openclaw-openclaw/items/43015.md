## Summary

Keep open. Current main still source-reproduces the poll false-positive path for mixed-action message schemas, and Discord still treats an empty `components.modal` skeleton as hard modal intent if that object reaches the send path; the linked repair PR with closing syntax is still open.

## What This Changes

Request to narrow or sanitize `message.send` so model-filled poll and Discord component/modal defaults do not block ordinary sends.

## Best Possible Solution

Land or replace the linked repair with a maintainer-approved contract that treats model-filled no-op poll/component defaults as absent for plain sends while preserving explicit `poll` and real modal validation with regression coverage.

## Reproduction Assessment

Yes, at source level. A `runMessageAction()` call with `action: "send"` and non-empty poll-looking params still reaches the guard before dispatch, and a non-empty object-shaped Discord `components.modal` skeleton still reaches the strict empty-fields parser.

## Solution Assessment

Unclear as a final implementation path. Treating model-filled no-op advanced fields as absent is the right direction, but maintainers still need to choose the canonical schema split, sanitization, and modal-validation contract.
