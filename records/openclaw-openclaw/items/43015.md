## Summary

Keep open: current main and the latest release include partial poll-guard fixes, but the reported send-blocking class is still source-reproducible for schema-padded channel-extra poll fields and empty Discord modal skeletons. The linked closing PR is still open and conflicting, so this needs maintainer consolidation rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to narrow `message.send` schema/runtime behavior so plain sends ignore auto-populated poll fields and empty Discord component/modal skeletons unless poll or component behavior is explicitly intended.

## Best Possible Solution

Land one maintainer-approved repair that treats content-bearing poll fields as the only send-blocking poll intent, normalizes empty Discord component/modal skeletons as absent, and preserves explicit poll/component actions with focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path without running a live channel: `message.send` rejects whenever `hasPollCreationParams` is true, and current code/tests still classify `pollDurationSeconds: 60` or `pollPublic: true` as poll intent. The Discord modal half is also source-reproducible because present `components.modal` with empty `fields` still throws before normal send handling.

## Solution Assessment

Unclear as a final product choice: the partial merged fix for shared poll modifiers is useful but not enough for the latest reported channel-extra poll fields or empty modal skeletons. The maintainable fix should consolidate schema scoping and runtime normalization rather than layering more one-off defaults.
