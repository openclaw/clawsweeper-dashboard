## Summary

Keeping this open: current main still exposes shared poll creation fields on the unified message tool and still rejects `action="send"` when those fields look populated; Discord also still treats a non-empty `components.modal` object with empty fields as an enabled modal. An open linked fix PR explicitly references this issue, so it should not be closed before that implementation is reviewed, landed, or replaced.

## What This Changes

The issue asks OpenClaw to keep ordinary `message.send` calls from being broken by auto-populated poll and Discord components/modal fields.

## Best Possible Solution

Land or replace the linked fix with a narrow upstream change that strips or scopes poll fields for `send`, treats empty Discord component/modal skeletons as absent, and preserves explicit poll/modal behavior with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `send` payload with populated poll creation fields reaches the send-side poll guard, and a Discord `send` payload with `components.modal.fields: []` reaches the modal parser error.

## Solution Assessment

Yes. The narrow maintainable fix is action-aware sanitization or schema scoping for `send`, plus ignoring empty advanced-field skeletons while keeping strict validation for real poll/modal requests.
