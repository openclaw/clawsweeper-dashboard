## Label Justifications

- `P1`: The issue affects real channel send workflows and can block ordinary outbound messages or attachments before delivery.
- `impact:message-loss`: The reported failures reject intended outbound channel sends before the user-visible message or attachment is delivered.

## Summary

Keep open. Current main still exposes poll creation fields through the mixed-action message tool schema and still rejects `action: "send"` before dispatch when model-filled poll-looking params are present; the linked repair at https://github.com/BingqingLyu/openclaw/pull/708 is still open and has not landed.

## What This Changes

The issue asks OpenClaw to stop letting model-filled poll and native component/modal fields break ordinary `message.send` text or attachment sends.

## Best Possible Solution

Land or replace the linked canonical repair at https://github.com/BingqingLyu/openclaw/pull/708 with a maintainer-approved contract that ignores no-op model-filled advanced fields for ordinary sends while keeping explicit poll and real modal/component validation strict.

## Reproduction Assessment

Yes, by source inspection. A `runMessageAction()` call with `action: "send"` and non-empty, non-zero, or true poll creation params still hits the guard before dispatch; reporter comments also confirm Telegram failures.

## Solution Assessment

Unclear as a final implementation path. The direction of treating model-filled no-op advanced fields as absent is right, but maintainers still need to choose one canonical poll/schema/component contract rather than landing another partial workaround.
