## Summary

Keep open. Current `main` still has the source-visible poll false-positive path for mixed `message` schemas, and an open linked repair PR already targets this issue and the Feishu companion, so this should stay paired with that review path.

## What This Changes

The issue asks OpenClaw to prevent ordinary `message.send` calls from being rejected when GPT-family models auto-populate advanced poll, component, or modal fields exposed by the shared message schema.

## Best Possible Solution

Land or replace the linked repair with a narrow fix that strips or ignores model-filled advanced defaults for `send`, scopes poll/component schema by action and channel, and keeps explicit poll and real modal validation strict.

## Reproduction Assessment

Yes. Source inspection on current `origin/main` shows `action="send"` plus poll-looking params still reaches the reported guard, and focused tests still expect that rejection; the Discord empty-modal skeleton path is also source-visible.

## Solution Assessment

Yes. The schema/normalization direction in the linked repair PR is the narrowest maintainable path, but this issue should remain open until that PR is reviewed, landed, or replaced.
