## Summary

Keep open. Current main still exposes shared poll creation fields through the unified message tool schema and still rejects ordinary `action: "send"` calls when non-empty or otherwise poll-looking params are present. The Discord generic-tool schema appears narrowed enough that `components` is not included by default, but the empty modal skeleton path remains strict when a populated `components` object reaches Discord handling, and open PR #46455 directly links this issue as an implementation candidate.

## What This Changes

The issue requests action-scoped message tool schemas and runtime handling so ordinary `message.send` calls are not polluted or rejected by auto-populated poll fields or empty Discord components/modal skeletons.

## Best Possible Solution

Review, land, or explicitly supersede PR #46455 with a focused patch that makes poll creation fields action-aware, ignores or strips model-populated advanced-field noise from ordinary sends, preserves strict validation for explicit polls and real Discord components/modals, and adds regression coverage for Telegram/Feishu/Discord send-with-file plus Discord empty-modal skeleton behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains: `message.send` with non-empty poll-looking params still trips `hasPollCreationParams()` and the send-path guard before ordinary send dispatch, and existing tests assert that rejection.

## Solution Assessment

Unclear. The requested direction is maintainable, but no final patch has landed; maintainers should choose the narrowest combination of schema action-scoping and runtime normalization that fixes model-populated noise without weakening explicit poll or real component validation.
