## Summary

Keep open. Current main still has source-visible `message.send` failures for model-filled poll params and empty Discord modal skeletons, and several open repair PRs need consolidation rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to keep ordinary `message.send` payloads from being rejected or rerouted when GPT-family models auto-populate poll and Discord component/modal fields.

## Best Possible Solution

Land or replace one canonical repair that treats model-filled no-op poll/component defaults as absent for ordinary sends, preserves strict validation for explicit poll and real modal requests, and keeps media-root policy separate.

## Reproduction Assessment

Yes, by source inspection. Current main still exposes poll fields in mixed message schemas, still runs the send-side poll guard, and still throws on empty Discord modal skeletons; I did not run a live GPT-family channel flow.

## Solution Assessment

Unclear as a single PR path. The right direction is clear, but maintainers still need to consolidate competing strip-versus-semantic-gate proposals and avoid mixing in the separate media-root policy question.
