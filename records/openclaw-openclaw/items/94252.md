## Summary

Keep this PR open because it is the active candidate fix for stale Dreaming sessions after Gateway restart, but it still has a concrete merge blocker: the startup scrubber is imported as a runtime static import while the same narrative module is intentionally lazy-loaded elsewhere.

The PR exports the Dreaming narrative scrubber, schedules it from memory-core `gateway_start`, and adds a startup regression test for stale `dreaming-narrative-*` session rows and transcripts.

## Best Possible Solution

Keep the startup scrub behavior, load the scrubber lazily or through a small internal runtime boundary, preserve the current accessor-backed cleanup path, and land only after maintainers accept the targeted startup session-state cleanup.

## Reproduction Assessment

Yes. Current main only invokes the Dreaming scrubber from `generateAndAppendDreamNarrative` cleanup while `gateway_start` only reconciles cron state, and the PR body includes live Gateway startup output showing stale row cleanup with fresh and normal sessions preserved.

## Solution Assessment

No, not as written. Reusing the existing memory-core scrubber from startup is the right owner-boundary fix, but the implementation should preserve the lazy module boundary and current accessor-backed cleanup before merge.
