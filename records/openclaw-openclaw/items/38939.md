## Summary

Keep open. The Feishu duplicate-reply problem still appears valid on current main, but this PR is not merge-ready: current main still has only account-local mention gating, while the branch hard-codes the coordinator account as `main`, misses current topic-group semantics, and adds substring-based implicit mention matching. The related #39853 is useful adjacent Feishu mention-metadata work, but it does not replace dispatch-time sibling arbitration.

## What This Changes

The PR adds Feishu mentioned-open-id extraction, dispatch-time group mention arbitration across main and specialist bot accounts, Feishu mention regression tests, an implicit text mention fallback, and a short design note.

## Best Possible Solution

Implement a narrow Feishu-owned arbitration layer in the current receive path that uses structured mentioned bot open_ids, ignores broadcast `@all`, derives the coordinator from Feishu account configuration, supports both `group` and `topic_group`, preserves existing debounce/dedupe/reaction/broadcast behavior, and covers the three policy cases with regression tests and user-facing changelog credit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure multiple enabled Feishu bot accounts in one group, allow the coordinator/default account to dispatch without being mentioned, then send a structured group mention to a sibling bot; current main starts both accounts, dedupes per account, and only checks whether each current bot was mentioned.

## Solution Assessment

No, not as submitted. Dispatch-time arbitration is the right direction, but the branch should derive the coordinator from config, cover current topic-group handling, and avoid substring-only implicit mention matching before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:35`
  - body: This hard-codes the coordinator account as `main`, but Feishu account IDs are configurable and default selection may be `default` or any operator-defined account. In deployments without a literal `main`, multiple specialist mentions can make every mentioned specialist skip while no coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic groups:** `extensions/feishu/src/monitor.ts:94-96`
  - body: Feishu currently treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.84
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention. Match exact mention tokens or drop this fallback path.
  - confidence: 0.88
