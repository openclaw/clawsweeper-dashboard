## Summary

Keep open: the Feishu duplicate-reply problem is still plausible on current main, but this PR is not merge-ready because the policy is hard-coded around a literal `main` account, misses `topic_group`, uses unsafe substring mention matching, and lacks real Feishu after-fix proof.

## What This Changes

The PR adds Feishu inbound mention extraction and main/specialist dispatch arbitration, with regression tests and a short design note for group mention routing.

## Best Possible Solution

Choose one Feishu multi-bot routing contract, then implement it with configured coordinator selection, `group` and `topic_group` coverage, safe mention matching, focused tests, and redacted live Feishu proof.

## Reproduction Assessment

Yes at source level: current main starts every enabled Feishu account, processes messages under each account id, and only detects whether the current bot was mentioned. I did not establish a live Feishu tenant reproduction.

## Solution Assessment

No. Dispatch-time arbitration may be viable, but this branch hard-codes the coordinator, skips `topic_group` chats, and uses substring text fallback matching that can create false positive mentions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account IDs and default-account selection are configurable. Deployments that use `default`, `ops`, or `channels.feishu.defaultAccount` will not get the documented `main + specialist` arbitration semantics and can still duplicate or misroute group replies.
  - confidence: 0.9
- **[P2] Apply arbitration to topic_group chats:** `extensions/feishu/src/monitor.ts:97-99`
  - body: Current Feishu handling treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention.
  - confidence: 0.88
