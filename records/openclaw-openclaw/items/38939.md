## Summary

Keep this PR open: the duplicate Feishu multi-bot reply problem is still present on current main, but the submitted branch is not merge-ready because it has blocking correctness issues, missing real Feishu behavior proof, a conflicting/stale code shape, and overlap with a related metadata proposal.

## What This Changes

This PR adds Feishu dispatch-time mention arbitration, a mentioned-open-id extractor, an implicit text-mention fallback, regression tests, and a design note for multi-bot group mentions.

## Best Possible Solution

Implement Feishu-owned arbitration on current main using the configured default/coordinator account, covering group and topic_group while preserving debounce, broadcast, and @all behavior, then reconcile it with the open metadata proposal.

## Reproduction Assessment

Yes, at source level. Current main dispatches Feishu message events per account, uses account-scoped processing claims, and only knows whether the current bot was mentioned, so a multi-account group can still dispatch from the addressed specialist and from another account with no mention requirement; I did not run a live Feishu repro.

## Solution Assessment

No, not as submitted. Dispatch-time arbitration is the right area, but this patch hard-codes `main`, skips `topic_group`, uses substring mention fallback matching, risks broadcast @all regressions, and needs reconciliation with the related metadata proposal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account IDs and default-account selection are configurable. In deployments without a literal `main` account, multi-specialist mentions can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic groups:** `extensions/feishu/src/monitor.ts:97-99`
  - body: Current Feishu handling treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real Feishu mention. Match exact mention tokens or remove this fallback path.
  - confidence: 0.88
- **[P2] Filter broadcast mentions in the shared extractor:** `extensions/feishu/src/bot.ts:197-205`
  - body: Current main deliberately excludes Feishu broadcast `@all` and `@_all` from bot-mention handling. The new shared extractor should filter those broadcast mentions before feeding `checkBotMentioned` and arbitration, otherwise broadcast-only messages can be treated as addressed bot mentions.
  - confidence: 0.82
