## Summary

Keep open: the Feishu duplicate-reply problem is source-reproducible and not implemented on current main, but this PR is not merge-ready because the submitted arbitration policy has correctness bugs, lacks real Feishu multi-bot proof, and overlaps an open metadata-based approach that needs maintainer direction.

## What This Changes

The PR adds Feishu group mention extraction and dispatch-time arbitration across main/specialist bot accounts, plus Feishu tests and a design note.

## Best Possible Solution

Choose one Feishu multi-bot routing contract, then implement it with configured default-account selection, group and topic_group coverage, preserved broadcast filtering, focused tests, and redacted live proof.

## Reproduction Assessment

Yes, at source level: current main starts one monitor per enabled Feishu account and each receive path claims processing under the accountId namespace before dispatch. I did not run a live Feishu tenant.

## Solution Assessment

No. Dispatch-time arbitration may be viable, but this branch hard-codes main, skips topic_group, uses substring fallback matching, and does not preserve existing broadcast mention filtering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as main, but Feishu account IDs and default-account selection are configurable. Deployments without a literal main account can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic_group chats:** `extensions/feishu/src/monitor.ts:97-99`
  - body: Current Feishu handling treats both group and topic_group as group chats, but this policy returns early unless the raw chat type is exactly group. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as @main can match plain text like @maintenance or @mainbot and mark the bot as mentioned without a real structured Feishu mention.
  - confidence: 0.88
- **[P2] Filter broadcast mentions in the shared extractor:** `extensions/feishu/src/bot.ts:197-211`
  - body: Current main excludes Feishu broadcast @all/@_all before treating a mention as bot-directed. The new shared extractor should filter those broadcast mentions before feeding checkBotMentioned and arbitration, otherwise broadcast-only messages can be treated as addressed bot mentions.
  - confidence: 0.82
