## Summary

Keep open: the duplicate-reply problem is source-reproducible on current main, but this PR still has blocking correctness findings and lacks after-fix Feishu multi-bot behavior proof.

## What This Changes

Adds Feishu mentioned-open-id extraction, dispatch-time group mention arbitration across bot accounts, regression tests, and a design note.

## Best Possible Solution

Implement Feishu-owned arbitration on current main using the configured coordinator/default account, covering group and topic_group, preserving @all filtering and debounce/dedupe behavior, and requiring live Feishu multi-bot proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main starts all enabled Feishu accounts, registers account-local receive handlers, and dedupes by account namespace, so sibling bot accounts can still process the same group event when mention policy lets them dispatch.

## Solution Assessment

No. Dispatch-time arbitration is a plausible direction, but this patch hard-codes the coordinator, excludes topic_group, uses substring fallback matching, and conflicts with the current broadcast mention contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account IDs and default-account selection are configurable. In deployments without a literal `main`, multi-specialist mentions can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic groups:** `extensions/feishu/src/monitor.ts:97-98`
  - body: Current Feishu handling treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.87
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The fallback uses substring matching, so a pattern like `@main` can match plain text such as `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention.
  - confidence: 0.88
- **[P2] Filter broadcast mentions before extraction:** `extensions/feishu/src/bot.ts:197-211`
  - body: Current main requires broadcast `@all`/`@_all` not to count as a bot mention, even when the bot open id is `all`. This shared extractor collects those ids and feeds them into `checkBotMentioned`, so a rebase would regress the existing broadcast-mention contract.
  - confidence: 0.83
