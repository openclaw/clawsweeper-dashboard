## Summary

This PR should stay open because current main still lacks sibling-account Feishu mention arbitration, but the proposed branch is not mergeable as-is: it has blocking correctness issues, overlaps with another open Feishu mention-contract PR, and still lacks real multi-bot Feishu proof.

## What This Changes

The branch adds Feishu mentioned-open-id extraction, dispatch-time group mention arbitration between a main account and specialist bot accounts, regression tests, and a design note.

## Best Possible Solution

Implement Feishu-owned dispatch arbitration on current main using the configured coordinator/default account, covering group and topic_group while preserving broadcast, dedupe, debounce, and @all behavior, then reconcile that with the related metadata proposal.

## Reproduction Assessment

Yes, at source level: run multiple enabled Feishu bot accounts in one group, send a structured mention to a sibling bot while another account can dispatch without being mentioned, and current main has account-local mention gating plus account-scoped receive dedupe. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

No. Dispatch-time arbitration is the right area, but this patch is not the best implementation because it hard-codes the coordinator as main, skips topic_group, uses substring fallback matching, and would regress current broadcast-mention filtering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:35`
  - body: This hard-codes the coordinator as main, but Feishu account IDs and default-account selection are configurable. In deployments without a literal main account, a multi-specialist mention can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic groups:** `extensions/feishu/src/monitor.ts:94-96`
  - body: Current Feishu handling treats both group and topic_group as group chats, but this policy returns early unless the raw chat type is exactly group. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as @main can match plain text like @maintenance or @mainbot and mark the bot as mentioned without a real structured Feishu mention. Match exact mention tokens or remove this fallback path.
  - confidence: 0.88
- **[P2] Filter broadcast mentions in the shared extractor:** `extensions/feishu/src/bot.ts:197-205`
  - body: Current main deliberately excludes Feishu broadcast @all and @_all from bot-mention handling. The new shared extractor should filter those broadcast mentions before feeding checkBotMentioned and arbitration, otherwise broadcast-only messages can be treated as addressed bot mentions.
  - confidence: 0.82
