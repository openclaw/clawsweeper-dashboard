## Summary

Keep this PR open: the Feishu duplicate-reply problem is still source-reproducible on current main, but the proposed branch is not merge-ready because it hard-codes the coordinator account, misses topic groups, adds substring-based implicit mention detection, and lacks real Feishu behavior proof.

## What This Changes

The PR adds Feishu mentioned-open-id extraction, dispatch-time group mention arbitration among main and specialist accounts, regression tests, and a design note.

## Best Possible Solution

Implement Feishu-owned dispatch arbitration on current main that derives the coordinator from configured Feishu account selection, covers group and topic_group, ignores broadcast @all, and preserves existing dedupe, debounce, reaction, and broadcast behavior with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run multiple enabled Feishu bot accounts in one group, send a structured mention to a sibling bot while another account can dispatch without being mentioned, and current main has only account-local mention gating plus per-account duplicate claims.

## Solution Assessment

No. Dispatch-time arbitration is the right direction, but this patch is not the best implementation because it hard-codes main, skips topic_group, uses substring text matching, and bypasses the current @all filtering contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account ids and default-account selection are configurable. In deployments without a literal `main`, multi-specialist mentions can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic groups:** `extensions/feishu/src/monitor.ts:97-98`
  - body: Current Feishu handling treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention.
  - confidence: 0.88
- **[P2] Filter broadcast mentions in the shared extractor:** `extensions/feishu/src/bot.ts:197-211`
  - body: Current main deliberately excludes Feishu broadcast `@all` and `@_all` from bot-mention handling. The new shared extractor should filter those broadcast mentions before feeding `checkBotMentioned` and arbitration, otherwise broadcast messages can be treated as addressed bot mentions.
  - confidence: 0.82
