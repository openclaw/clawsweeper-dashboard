## Summary

Keep this PR open: current main still lacks Feishu sibling-bot mention arbitration, but the submitted patch is not merge-ready because it hardcodes the coordinator account, adds an unsafe implicit text fallback, and does not include after-fix proof from a real Feishu multi-bot group.

## What This Changes

The PR adds Feishu mention extraction, group mention arbitration between a primary account and specialist accounts, regression tests, and a design note.

## Best Possible Solution

Keep the Feishu duplicate-reply fix open, but derive the coordinator from the existing Feishu default-account contract or a reviewed arbitration policy, make implicit text matching token-safe, and require redacted real Feishu multi-bot proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection shows current main has per-account Feishu dispatch and self-only mention detection, so the duplicate-reply path is source-reproducible but not proven in a real Feishu group here.

## Solution Assessment

No, not as submitted. Channel-side arbitration is plausible, but the patch should use configured account selection instead of a literal "main" account and should avoid substring-based implicit mention matches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive coordinator from Feishu account selection:** `extensions/feishu/src/monitor.ts:35`
  - body: The arbitration policy hardcodes `main` as the primary account, but OpenClaw's default account id is `default` and Feishu supports `channels.feishu.defaultAccount`. In deployments using `default`, `main-bot`, or any other primary id, messages mentioning the primary plus another bot can be treated as multi-specialist traffic and skipped by every mentioned account, so derive the coordinator from the existing Feishu account selection contract instead of this literal.
  - confidence: 0.92
- **[P2] Match implicit mention patterns as tokens:** `extensions/feishu/src/bot.ts:256-260`
  - body: `hasImplicitTextMention` uses `normalized.includes(pattern)`, so the fallback `@${accountId}` pattern makes `@main` match text such as `@mainbot` or `@maintenance`. That can mark a group message as mentioning the bot even when Feishu did not deliver a structured mention, causing unintended dispatches; require a token boundary around the matched pattern.
  - confidence: 0.86
