## Summary

Keep open: current main still lacks sibling-account Feishu mention arbitration, but this PR should not merge as submitted because it has blocking correctness issues and no convincing after-fix real Feishu multi-bot proof.

## What This Changes

The PR adds Feishu inbound mention arbitration across a primary account and specialist accounts, shared mention extraction, regression tests, and a design note.

## Best Possible Solution

Choose one Feishu mention strategy, then port it to current main using the configured default account, `group` and `topic_group` coverage, broadcast filtering, focused regression tests, and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: current main can run multiple enabled Feishu accounts, each receive path uses account-local processing claims, and there is no sibling-account arbitration before dispatch. I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

No. Dispatch-time arbitration may be viable, but this patch hard-codes `main`, skips `topic_group`, uses substring fallback matching, and regresses broadcast mention filtering; the metadata direction in https://github.com/openclaw/openclaw/pull/39853 also needs maintainer choice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from Feishu config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account IDs and default-account selection are configurable. In deployments without a literal `main` account, multi-specialist mentions can make every mentioned specialist skip while no configured coordinator handles the message.
  - confidence: 0.91
- **[P2] Apply arbitration to topic_group chats:** `extensions/feishu/src/monitor.ts:94-96`
  - body: Current Feishu handling treats both `group` and `topic_group` as group chats, but this policy returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:256-260`
  - body: The implicit fallback uses substring matching, so a fallback pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention. Match exact mention tokens or remove this fallback path.
  - confidence: 0.88
- **[P2] Filter broadcast mentions in the shared extractor:** `extensions/feishu/src/bot.ts:197-205`
  - body: Current main excludes Feishu broadcast `@all` and `@_all` from bot-mention handling. The new shared extractor should filter those broadcast mentions before feeding `checkBotMentioned` and arbitration, otherwise broadcast-only messages can be treated as addressed bot mentions.
  - confidence: 0.82
