## Summary

Keep open: current main still has the Feishu multi-account duplicate-reply path, but this PR is not mergeable as-is because it hard-codes the coordinator account, misses topic_group chats, introduces substring-based implicit mention matches, and lacks real Feishu multi-bot proof.

## What This Changes

The PR adds Feishu inbound mention extraction, main/specialist dispatch arbitration, regression tests, and a design note for multi-bot group mentions.

## Best Possible Solution

Implement one Feishu multi-bot mention contract using the configured coordinator/default account, cover group and topic_group chats, use boundary-safe mention matching, add focused regression tests, and require redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main starts every enabled Feishu account and each account-scoped handler only checks whether its own botOpenId was mentioned. I did not establish a live Feishu tenant reproduction.

## Solution Assessment

No. Dispatch-time arbitration may be viable, but this patch hard-codes the coordinator, excludes topic_group chats, and adds substring text-mention matching; the safer path needs a configured coordinator contract or structured mention metadata direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive the coordinator account from config:** `extensions/feishu/src/monitor.ts:38`
  - body: This hard-codes the coordinator as `main`, but Feishu account IDs and default-account selection are configurable. Deployments using `default`, `ops`, or another `channels.feishu.defaultAccount` can misroute multi-bot mentions, and multi-specialist mentions can be skipped by all addressed accounts instead of being handled once.
  - confidence: 0.9
- **[P2] Apply arbitration to topic_group chats:** `extensions/feishu/src/monitor.ts:97-99`
  - body: Current Feishu handling treats both `group` and `topic_group` as group-chat surfaces, but this helper returns early unless the raw chat type is exactly `group`. Multi-bot topic groups would keep the duplicate-reply behavior this PR is meant to prevent.
  - confidence: 0.86
- **[P2] Require token boundaries for implicit mentions:** `extensions/feishu/src/bot.ts:259-260`
  - body: The fallback uses substring matching, so a pattern such as `@main` can match plain text like `@maintenance` or `@mainbot` and mark the bot as mentioned without a real structured Feishu mention.
  - confidence: 0.88
