## Summary

Keep open: the default sqlite/composite path looks materially improved and the supplied proof is strong, but the updated delete fallback now assumes `deleteTask` also deletes delivery state even though the store shape and existing test helper support separate `deleteTask` / `deleteDeliveryState` methods.

## What This Changes

The PR reorders task update/delete persistence ahead of in-memory task-registry mutations and adds regression coverage for sqlite-direct divergence plus delete fallback behavior.

## Best Possible Solution

Keep the persist-before-memory fix, but make the non-composite delete fallback remove both task and delivery state through one projected snapshot or an explicitly updated store contract with regression coverage.

## Reproduction Assessment

Yes. The PR body supplies a before/after terminal probe for the main sqlite divergence bug, and source inspection gives a clear remaining reproduction path with a non-composite store that implements `deleteTask` and `deleteDeliveryState` separately.

## Solution Assessment

Not yet. Persisting before memory is the right shape for the default sqlite path, but the fallback needs to preserve the existing separate delivery-state deletion semantics before merge.
