## Summary

Keep open: the core persist-before-memory direction is right, but one non-composite store path still performs task and delivery-state upserts as separate writes, so a delivery-state write failure can leave durable task state advanced while in-memory state stays old.

## What This Changes

The PR reorders task and TaskFlow persistence ahead of in-memory mutation, makes creation/update/delete failure paths explicit, adds try-style runtime helpers, and updates related callers/tests.

## Best Possible Solution

Keep the persist-before-memory approach, but make non-composite task-plus-delivery upserts a single projected snapshot when no composite helper exists, then rerun the focused task store and task-flow suites.

## Reproduction Assessment

Yes from source inspection and the PR’s red/green terminal evidence: current main commits in-memory task state before the SQLite-backed persist helper, so a thrown write can leave memory and durable reads out of sync. I did not run tests because this review is read-only.

## Solution Assessment

No, not quite yet. Persist-before-memory is the right fix shape for the default SQLite path, but the non-composite `upsertTask` plus `upsertDeliveryState` branch must be made atomic before this is the best maintainable solution.
