## Summary

Keep open: the default SQLite path and supplied proof cover the central persist-before-memory bug, but the non-composite store path that has both `upsertTask` and `upsertDeliveryState` still splits a task+delivery upsert into two writes, so a delivery write failure can leave persistent state ahead of memory.

## What This Changes

This PR reorders task and TaskFlow persistence ahead of in-memory mutation, makes task/flow creation failure paths explicit with null or try-style APIs, updates affected agent/gateway/cron/plugin callers, and adds regression tests.

## Best Possible Solution

Make every task+delivery persist path atomic before merge, then have maintainers intentionally accept the new explicit persistence-failure contract for runtime and plugin callers.

## Reproduction Assessment

Yes for the central bug from source and the PR's supplied live-output proof; the remaining non-composite gap is source-reproducible from the split `upsertTask` then `upsertDeliveryState` branch without running the checkout.

## Solution Assessment

No, not yet. Persist-before-memory is the right owner-boundary fix, but the non-composite task+delivery upsert branch needs the same single-operation treatment before this is the best implementation.
