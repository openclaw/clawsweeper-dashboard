## Summary

Keep this PR open: the persist-before-memory direction matches the task-store consistency bug, but the latest head still leaves one blocking custom-store path where a successful update with delivery state can persist the old task row because the snapshot is taken before `tasks.set`.

## What This Changes

The PR reorders task-registry update/delete persistence before in-memory mutation and adds regression tests for sqlite write-failure divergence and delete fallback behavior.

## Best Possible Solution

Land the persist-before-memory approach after every upsert/delete fallback persists a projected pending snapshot and the regression tests cover default sqlite, snapshot-only, and non-composite custom store shapes.

## Reproduction Assessment

Yes. Current main shows the in-memory update/delete happens before transactional sqlite persistence, and the PR body supplies a deterministic injected-store repro for a rolled-back write leaving sqlite unchanged.

## Solution Assessment

No. Persist-before-memory is the right fix shape for the default sqlite path, but this branch is not complete until the non-composite upsert snapshot path projects the pending task and has regression coverage.
