## Summary

This PR is a real fix for the default sqlite/composite task-store divergence, and the author has already addressed earlier delete-path feedback, but the latest patch still leaves a blocking non-composite upsert regression in the fallback store contract.

## What This Changes

The branch moves task-registry upsert/delete persistence before in-memory mutation and adds regression tests for sqlite write failures and delete fallback behavior.

## Best Possible Solution

Keep the persist-before-memory fix, but make non-composite upserts with delivery state use one projected atomic snapshot and add focused coverage for that fallback before merge.

## Reproduction Assessment

Yes. Source inspection shows a concrete current PR path: a task with delivery state, a store with upsertTask but no upsertDeliveryState, and updateTask now calling persistTaskUpsert before tasks.set causes saveSnapshot to receive the old task map.

## Solution Assessment

No. Persist-before-memory is the right default sqlite fix, but the non-composite upsert fallback must also project the pending task, or use one atomic snapshot, before this is the safest maintainable solution.

Full review comments:

- **[P1] Project non-composite upsert snapshots before mutating memory:** `src/tasks/task-registry.ts:1060`
  - body: Moving `persistTaskUpsert(next)` before `tasks.set` means the fallback branch for stores with `upsertTask` but no `upsertDeliveryState` still calls `saveSnapshot({ tasks, ... })` with the old in-memory task map. Updating a task that already has delivery state will persist a stale task, so custom stores can still diverge or lose the update. Project `next` into the snapshot, or use one atomic snapshot for task plus delivery state whenever the composite upsert helper is unavailable.
  - confidence: 0.91
