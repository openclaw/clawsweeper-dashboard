## Summary

Keep this PR open. Current main still has both SHA-1 calls the branch targets, and the proposed SHA-256 swap appears correct for the existing opaque fingerprint/cache-key contracts with no blocking functional or security finding.

## What This Changes

The PR changes `createCatalogFingerprint` and a media-understanding test helper from SHA-1 to SHA-256 while leaving the surrounding call sites and cache-key structure unchanged.

## Best Possible Solution

Merge or otherwise apply this narrow cleanup if maintainers want these two internal cache/fingerprint helpers aligned on SHA-256; any broader repository-wide SHA-1 audit should be tracked separately because other extension, script, and protocol-specific SHA-1 uses remain.

## Reproduction Assessment

Not applicable as a cleanup PR rather than a bug report. Source inspection gives a high-confidence verification path: current main still contains both SHA-1 calls, and a local Node crypto probe confirms SHA-256 preserves the existing string digest contract.

## Solution Assessment

Yes for the stated two-call cleanup: SHA-256 is a drop-in replacement for these opaque, non-persisted comparison/cache keys. It should not be treated as a complete repository-wide SHA-1 purge because `rg` still finds other SHA-1 uses, some of which appear protocol- or extension-specific.
