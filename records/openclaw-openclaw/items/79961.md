## What This Changes

Adds `delete payload.reasoning` in the shared DeepSeek V4 OpenAI-compatible thinking wrapper so the final payload does not keep both nested `reasoning` and top-level `reasoning_effort`.

## Best Possible Solution

Yes. Source inspection shows a non-off OpenRouter DeepSeek V4 request can pass through generic OpenRouter reasoning injection that creates nested `reasoning.effort`, then through the DeepSeek V4 wrapper that adds top-level `reasoning_effort`.

## Solution Assessment

Yes for the code direction: clearing `payload.reasoning` in the enabled DeepSeek V4 branch is the smallest fix consistent with current OpenRouter DeepSeek V4 docs. The merge path still needs real behavior proof and green status checks.
