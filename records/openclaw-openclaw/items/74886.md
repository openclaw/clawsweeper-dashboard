## Summary

Keep open. The report is credible and not fully fixed on the reviewed current main: WhatsApp still does not provide an explicit `onReasoningStream` sink at its reply boundary, while the linked open PR #75258 targets that exact gap. Current guards suppress already-marked reasoning payloads, but the MiniMax fallback and broader stability symptoms remain unproven as fixed.

## What This Changes

The issue reports that upgrading from OpenClaw 2026.4.23 to 2026.4.27 made a WhatsApp direct session unstable, exposed reasoning content, and appeared to persist or fall back to MiniMax instead of the configured Codex primary.

## Best Possible Solution

Review and land or revise PR #75258 with a regression test proving WhatsApp supplies a reasoning-stream sink and does not deliver reasoning chunks. After that, keep any remaining MiniMax fallback or gateway-stability behavior as narrower artifact-backed follow-ups instead of treating this issue as fully implemented.

## Reproduction Assessment

No. The reporter supplied credible rollback data, sanitized transcript counts, and a plausible code-level root cause, but this review did not establish a high-confidence live reproduction of the full leak plus fallback plus instability sequence on current main.

## Solution Assessment

Unclear. Adding a WhatsApp `onReasoningStream` no-op appears to be the narrowest first hardening step for the reasoning leak, but it does not by itself prove the MiniMax fallback state or gateway instability symptoms are fixed.
