## Summary

Keep open. The report is credible and still not proven fixed on current main: WhatsApp suppresses payloads already marked as reasoning, but the WhatsApp inbound dispatch still does not provide an `onReasoningStream` sink, and the paired open PR #75258 targets that exact remaining stream-boundary gap. The MiniMax fallback stickiness and broader gateway instability symptoms also remain separate, unproven-as-fixed parts of the report.

## What This Changes

The issue reports that upgrading from OpenClaw 2026.4.23 to 2026.4.27 made a WhatsApp direct session unstable, leaked internal reasoning text to WhatsApp, and appeared to stick on MiniMax instead of the configured Codex primary model.

## Best Possible Solution

Treat the WhatsApp reasoning disclosure as the first fix: review and land or revise PR #75258 with a regression test proving WhatsApp supplies a reasoning-stream sink and does not deliver reasoning chunks. After that, split any remaining MiniMax fallback or gateway-stability behavior into narrower artifact-backed follow-up issues.

## Reproduction Assessment

No. The reporter supplied credible rollback data and sanitized transcript/artifact counts, and current source shows a plausible missing callback boundary, but there is not a high-confidence live reproduction against current main for the full leak, fallback, and stability sequence.

## Solution Assessment

Unclear. The paired PR's `onReasoningStream` no-op appears to be the narrowest first hardening step for the WhatsApp leak, but it does not by itself prove the MiniMax fallback state or gateway instability symptoms are fixed.
