## Summary

Keep open. Current main still treats any live-PID runtime-deps lock as fresh unless start-time evidence proves PID reuse, so a legacy owner record with a live recycled PID and no `starttime` or `createdAtMs` can still block until the lock timeout; the paired PR #74950 remains the right implementation candidate.

## What This Changes

The issue asks OpenClaw to expire legacy bundled runtime-deps lock files that lack start-time and creation-time metadata by falling back to stale lock-directory or owner-file mtimes.

## Best Possible Solution

Land #74950 or an equivalent narrow change in `shouldRemoveRuntimeDepsLock()` that uses stale lock-directory or owner-file mtimes only for legacy live-PID locks missing stronger ownership timestamps, with focused regression coverage for the Docker recycled-PID case.

## Reproduction Assessment

Yes. A source-level reproduction on current main is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1000, ownerFileMtimeMs: 2000 }, 1_000_000, () => true)`, which returns `false` before mtime fallback can run.

## Solution Assessment

Yes. A narrow lock-predicate fallback plus colocated tests is the most maintainable fix; broader runtime-deps repair work does not address this specific stale legacy lock owner path.
