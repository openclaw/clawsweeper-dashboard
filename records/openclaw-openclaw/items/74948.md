## Summary

Keep open. Current main still treats a live-PID bundled runtime-deps lock as fresh before consulting `createdAtMs` or filesystem mtime fallback, so the legacy Docker lock path remains valid; the paired PR #74950 is the existing implementation candidate rather than a reason to close this issue early.

## What This Changes

Expire stale bundled runtime-deps lock directories created by legacy OpenClaw versions when the owner PID is live but the lock lacks start-time and created-at evidence.

## Best Possible Solution

Review and land PR #74950 or an equivalent narrow change in `shouldRemoveRuntimeDepsLock()` with regression coverage for live-PID legacy locks missing start-time and created-at fields, while preserving active-lock safety when matching start-time evidence exists.

## Reproduction Assessment

Yes. A source-level reproduction on current main is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1000, ownerFileMtimeMs: 2000 }, 1_000_000, () => true)`, which follows the live-PID branch and returns `false` before stale mtime fallback is considered.

## Solution Assessment

Yes. The narrow predicate fallback plus focused tests is the maintainable fix boundary; the broader runtime-deps repair PR #75183 explicitly does not claim this root cause.
