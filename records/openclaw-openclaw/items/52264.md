## Summary

Keep open: current main still has the cwd-sensitive relative `OPENCLAW_STATE_DIR` path, and this PR is a useful follow-up to the linked plugin-registry bug, but the branch still revives a retired legacy env var and carries generated baseline artifacts that should not be merged.

## What This Changes

The PR adds startup normalization for explicit state-dir env overrides in the gateway CLI and server startup paths, with focused path/startup-order tests and generated SDK baseline changes.

## Best Possible Solution

Pin only the supported `OPENCLAW_STATE_DIR` before cwd-sensitive gateway startup work in both CLI and direct server paths, without reviving legacy env names or committing local generated baselines.

## Reproduction Assessment

Yes, as source-reproducible. Current main resolves relative `OPENCLAW_STATE_DIR` at `resolveStateDir` call time and gateway startup does not pin it before cwd-sensitive work; the linked fixture documents the channel becoming unknown after the cwd-changing path.

## Solution Assessment

No, not in the current branch shape. Early startup pinning is the right direction, but the patch should be narrowed to the supported env contract and stripped of generated baseline churn before merge.
