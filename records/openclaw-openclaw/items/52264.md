## Summary

Keep this PR open, but it still needs changes before merge. Current main does not include the startup state-dir pinning, so the useful hardening is not obsolete; the branch still conflicts with the documented legacy-env contract and carries local-only generated SDK baseline files.

## What This Changes

The PR adds startup normalization for explicit state-dir environment overrides in gateway CLI/direct startup paths and adds focused path/order tests.

## Best Possible Solution

Pin only supported `OPENCLAW_STATE_DIR` at the earliest gateway startup boundaries, preserve current restart/config startup behavior, and leave local generated SDK baselines out of the branch.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the relative-state-dir drift: current main resolves `OPENCLAW_STATE_DIR` relative to the cwd at each call. The linked fixture for https://github.com/openclaw/openclaw/issues/52250 demonstrates the channel failure, but I did not rerun it in this read-only review.

## Solution Assessment

No, not in the current branch shape. Early state-dir pinning is the right fix direction, but the branch should not normalize legacy `CLAWDBOT_STATE_DIR`, should drop generated baseline artifacts, and should be reconciled with current gateway startup code.
