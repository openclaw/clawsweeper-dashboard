## Summary

Keep open: the startup pinning is useful and still missing on current main, but the branch should not merge as-is because it reintroduces side effects for retired legacy env names, carries local-only generated baseline files, and is now outdated/conflicting.

## What This Changes

The branch adds a state-dir normalization helper, calls it from gateway CLI and direct server startup, and adds focused path/startup-order tests while also changing generated Plugin SDK baseline files.

## Best Possible Solution

Land a rebased narrow patch that pins supported `OPENCLAW_STATE_DIR` at gateway startup, leaves retired legacy env aliases ignored, and omits local-only generated baseline artifacts.

## Reproduction Assessment

Yes, at source level. Current main still resolves a relative `OPENCLAW_STATE_DIR` from the current cwd at each call, and both gateway startup paths lack an early normalization hook; I did not run the external repro in this read-only review.

## Solution Assessment

No, not in the current branch shape. Startup pinning for `OPENCLAW_STATE_DIR` is the narrow maintainable fix, but this branch should drop `CLAWDBOT_STATE_DIR`, remove generated baselines, and rebase onto current gateway startup code.
