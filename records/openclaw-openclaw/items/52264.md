## Summary

Keep open: the supported state-dir startup pinning is still useful and is not on current main, but this branch needs cleanup before merge. The active blockers are the legacy `CLAWDBOT_STATE_DIR` normalization, which conflicts with the current legacy-env contract, and committed local generated Plugin SDK baseline files.

## What This Changes

The PR normalizes explicit state-directory environment overrides during gateway CLI and direct server startup, adds focused path/order tests, and updates generated Plugin SDK baseline files.

## Best Possible Solution

Land a cleaned fix that pins supported `OPENCLAW_STATE_DIR` before gateway config/plugin discovery on both startup paths, preserves ignored legacy env behavior, removes generated baselines, rebases against current main, and reruns focused proof.

## Reproduction Assessment

Yes. The linked standalone fixture and a focused cwd-change path around `resolveStateDir()` give a concrete way to show relative `OPENCLAW_STATE_DIR` can drift; I did not rerun them in this read-only review.

## Solution Assessment

No. Early pinning for the supported state-dir env var is the right narrow direction, but this branch currently violates the legacy-env contract and includes generated local artifacts that should not be part of the gateway fix.
