## Summary

Keep this PR open: the startup pinning is still useful because current main does not normalize relative `OPENCLAW_STATE_DIR` before gateway startup, but the branch still needs cleanup because it reintroduces retired `CLAWDBOT_*` behavior and carries gitignored generated baseline churn.

## What This Changes

The PR adds state-dir env normalization in gateway CLI and direct server startup, focused path/startup-order tests, and generated Plugin SDK baseline updates.

## Best Possible Solution

Gateway startup should pin only the supported `OPENCLAW_STATE_DIR` before cwd-sensitive loads in both CLI and direct startup paths, with regression coverage and no legacy-env or generated-artifact churn.

## Reproduction Assessment

Yes, at source level. Current main resolves relative `OPENCLAW_STATE_DIR` from the current cwd at each call and both gateway startup paths lack early pinning; the linked issue also provides a deterministic standalone fixture, but I did not run it in this read-only review.

## Solution Assessment

No, not in the current branch shape. Startup pinning for supported `OPENCLAW_STATE_DIR` is the right direction, but the branch should drop `CLAWDBOT_STATE_DIR`, remove generated baseline files, and rebase over current gateway startup code.
