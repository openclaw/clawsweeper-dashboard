## Summary

Keep open. Current main still does not pin relative `OPENCLAW_STATE_DIR` during gateway startup, so the hardening remains useful, but the PR head is conflicting and still includes two merge-blocking cleanup issues.

## What This Changes

The PR adds a state-dir environment normalizer to config paths, calls it from gateway CLI and direct server startup, adds focused tests, and updates local Plugin SDK generated baselines.

## Best Possible Solution

Land a cleaned, rebased change that pins supported `OPENCLAW_STATE_DIR` to an absolute path before config/plugin work in both gateway entry paths, with focused tests and no legacy-env or generated-baseline churn.

## Reproduction Assessment

Yes, source-reproducible: current main resolves relative `OPENCLAW_STATE_DIR` via `path.resolve()` at each call, and neither gateway startup path pins it before later cwd-sensitive work. I did not rerun the external fixture in this read-only pass.

## Solution Assessment

No, not as-is. Early state-dir pinning is the right narrow fix direction, but the branch should only affect supported `OPENCLAW_STATE_DIR`, drop generated baseline files, and be rebased against current main.
