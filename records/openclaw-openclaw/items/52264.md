## Summary

Keep open: current main still has the cwd-sensitive relative `OPENCLAW_STATE_DIR` path, and this PR contains the right narrow fix direction, but the branch still needs cleanup before merge.

## What This Changes

The PR adds startup normalization for state-dir environment overrides, calls it from the gateway CLI and direct server startup paths, adds focused tests, and changes generated Plugin SDK baseline files.

## Best Possible Solution

Land a cleaned, rebased or replacement patch that pins supported `OPENCLAW_STATE_DIR` to an absolute path before gateway config/plugin work in both startup paths, with focused tests and no legacy-env or generated-baseline churn.

## Reproduction Assessment

Yes, source-reproducible: current main resolves relative `OPENCLAW_STATE_DIR` at each call, and neither gateway startup path pins it before later cwd-sensitive config/plugin work. The related report at https://github.com/openclaw/openclaw/issues/52250 also includes a standalone fixture and confirmation of the relative-path failure mode.

## Solution Assessment

No, not as-is. Early state-dir pinning is the right narrow fix, but the branch should only affect supported `OPENCLAW_STATE_DIR`, drop local generated baseline files, and be rebased or replaced against current main.
