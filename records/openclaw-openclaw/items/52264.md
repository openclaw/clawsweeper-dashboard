## Summary

Keep open. The startup state-dir pinning is a useful bug fix and the linked fixture output supports the behavior, but this PR still changes the retired `CLAWDBOT_STATE_DIR` contract, commits local generated SDK baselines, and is not mergeable against current `main`.

## What This Changes

The PR adds `normalizeStateDirEnv`, calls it from gateway CLI and direct server startup, adds focused path/startup tests, and also touches generated Plugin SDK baseline files.

## Best Possible Solution

Land a cleaned branch that pins supported `OPENCLAW_STATE_DIR` before gateway startup paths can observe cwd-sensitive state, while preserving retired legacy-env and generated-artifact contracts.

## Reproduction Assessment

Yes, by source inspection and linked fixture context. Current main resolves `OPENCLAW_STATE_DIR` at `resolveStateDir` call time and both gateway startup paths lack early normalization; I did not rerun the standalone fixture in this read-only pass.

## Solution Assessment

No, not as-is. Early `OPENCLAW_STATE_DIR` pinning is the right narrow fix, but the branch should leave retired `CLAWDBOT_*` ignored, remove generated artifacts, and rebase onto current main before validation.
