## Summary

Keep open: current main still leaves explicit relative `OPENCLAW_STATE_DIR` resolution cwd-sensitive in the gateway startup paths, so the PR addresses a real hardening gap. The branch is not mergeable as-is because it revives legacy `CLAWDBOT_STATE_DIR` handling and commits local generated Plugin SDK baseline files.

## What This Changes

The PR adds a state-dir environment normalizer, calls it from gateway CLI and direct server startup, and adds focused path/startup tests while also changing generated Plugin SDK baseline files.

## Best Possible Solution

Land a cleaned, rebased startup pin for supported `OPENCLAW_STATE_DIR` only, with focused tests and no generated-baseline churn.

## Reproduction Assessment

Yes, source-reproducible: current main resolves relative `OPENCLAW_STATE_DIR` through cwd-sensitive `path.resolve()` at each call, and both gateway startup paths lack an early pin. I did not rerun the external fixture in this read-only review.

## Solution Assessment

No, not as-is. Early startup pinning is the right narrow fix direction, but the PR should not touch legacy `CLAWDBOT_*` env behavior and should drop local generated baseline files before merge.
