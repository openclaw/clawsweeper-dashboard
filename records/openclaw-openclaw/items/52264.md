## Summary

Keep open: current main still has call-time relative state-dir resolution and the central hardening is useful, but this branch is not merge-ready because it re-enables legacy env behavior and carries ignored generated baselines.

## What This Changes

The PR adds startup normalization for `OPENCLAW_STATE_DIR`/`CLAWDBOT_STATE_DIR` in gateway CLI and direct server startup paths, plus focused path and call-order tests.

## Best Possible Solution

Pin only supported `OPENCLAW_STATE_DIR` at the earliest gateway startup boundaries, preserve current restart/config startup logic, and leave local generated SDK baselines out of the branch.

## Reproduction Assessment

Yes. Source inspection shows current main resolves `OPENCLAW_STATE_DIR` through `resolveStateDir()` at call time, and the linked standalone fixture/closed issue demonstrates the cwd-sensitive plugin-channel failure; I did not rerun it in this read-only review.

## Solution Assessment

No, not in the current branch shape. Early state-dir pinning is the right fix direction, but it should not normalize legacy `CLAWDBOT_STATE_DIR` and should not commit ignored generated baselines.
