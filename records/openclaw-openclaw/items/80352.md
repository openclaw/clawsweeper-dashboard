## Summary

Keep open: current main and the latest release still expose memory prompt supplements as a global prompt path with no agentId/sessionKey context, and the remaining fix is a plugin SDK/config scoping product decision rather than conservative cleanup.

## What This Changes

The issue requests per-agent or per-sessionKey scoping for memory-wiki compiled digest prompt injection and generic memory prompt supplements.

## Best Possible Solution

Define one maintainer-approved scoped prompt supplement contract, keep existing builders compatible, and add memory-wiki docs plus regression coverage for an included interactive session and an excluded cron/session-key path.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 build memory prompt supplements without agentId or sessionKey, and memory-wiki's digest builder can only see global config plus available tools. I did not run a live multi-session gateway in this read-only pass.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers should choose whether the narrowest durable fix is builder context, memory-wiki-owned scope config, broader per-agent plugin config, or a coordinated combination with vault-isolation work.
