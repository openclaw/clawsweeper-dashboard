## Summary

Keep open: current main and the latest release still make memory prompt supplements global at prompt-build time. The memory-wiki tools and corpus path already carry session context, but the prompt-supplement builder and memory-wiki digest prompt do not, so this remains a real plugin API/config design gap rather than stale noise.

## What This Changes

The issue requests agent or session-key scoping for memory-wiki compiled digest prompt injection and generic memory prompt supplements.

## Best Possible Solution

Define one scoped memory prompt contract, thread agentId/sessionKey or an explicit config predicate through prompt assembly, and update the Plugin SDK, memory-wiki config/docs, and focused tests without changing the default digest-off behavior.

## Reproduction Assessment

Yes for the current-behavior gap: source inspection shows the prompt supplement builder receives no sessionKey or agentId, and memory-wiki only gates digest injection on includeCompiledDigestPrompt. I did not run a live agent prompt because the missing parameters are clear from the current source and release tag.

## Solution Assessment

Unclear; the request is valid, but the best solution needs maintainer choice between a generic Plugin SDK builder-context extension and a memory-wiki-owned config predicate. A narrow implementation should avoid creating parallel scoping rules that drift from existing memory/session visibility contracts.
