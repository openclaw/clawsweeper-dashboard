## Summary

Keep open: current main still has the reported global memory prompt-supplement behavior, and the request is a plausible new SDK/config capability rather than an already-fixed bug or ClawHub-only plugin idea.

## What This Changes

The issue requests per-agent or per-session-key scoping for memory-wiki compiled digest prompt supplements so they are not injected into every prompt build.

## Best Possible Solution

Add a backward-compatible request-context path for memory prompt supplements and, if maintainers want operator-facing control, expose a narrow memory-wiki scope config with matching schema, docs, and tests.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the prompt supplement builder, legacy system prompt assembly, and context-engine helper do not carry sessionKey or agentId. I did not run a live prompt build because this was a read-only review, but the source path is direct.

## Solution Assessment

Unclear until maintainers choose the public contract. A generic builder-param extension plus optional memory-wiki scope config looks maintainable, but it changes plugin SDK/config behavior and should not be treated as a narrow bug fix.
