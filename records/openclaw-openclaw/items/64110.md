## Summary

Keep open: the PR targets a valid cron deletion race, but the current patch still has a blocking timer error-handling bug and lacks inspectable real behavior proof for the Feishu scenario.

## What This Changes

The PR adds force-reloaded cron job-existence checks before timed/manual execution plus seam tests for jobs removed between reservation and execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks persisted job existence after reservation but before delivery side effects, handles reload failures as per-job outcomes, and includes focused tests plus redacted real Feishu or cron proof.

## Reproduction Assessment

Yes, source-reproducible. Current main can capture a due cron job or prepared manual run, delete the persisted job, and still execute the captured snapshot before any fresh existence check stops side effects.

## Solution Assessment

No. The pre-execution guard is the right narrow direction, but the timer-path jobExists call must be inside handled per-job outcome logic so reload failures do not abort Promise.all and strand running markers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:726`
