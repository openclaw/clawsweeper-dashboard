## Summary

Keep open. The PR targets a real cron deletion race and improves the direction, but the latest patch still introduces a blocking timer-path failure mode: the new existence check can reject before the per-job error handling, which can abort the whole timer tick and skip result application for other completed jobs.

## What This Changes

The PR adds a cron `jobExists` helper, pre-execution deletion guards for timer and manual cron runs, and regression tests covering removed jobs before execution.

## Best Possible Solution

Keep the PR open and revise it so both timer-collected due jobs and queued manual runs re-check deletion after reservation but before side effects, while treating existence-check reload failures as ordinary per-job outcomes and preserving regression coverage for deleted jobs plus reload-error behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to collect or enqueue a cron job, delete it before the worker starts, and observe that current main can still execute the captured `CronJob` snapshot; the PR-specific bug is reproducible by making the new forced reload fail while other due-job outcomes are pending.

## Solution Assessment

No. The proposed direction is appropriate, but the timer guard needs to run inside the per-job failure boundary or otherwise convert reload failures into a `TimedCronRunOutcome` so one bad store read cannot abort the whole timer pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:724`
  - body: The new `jobExists` guard runs before `runDueJob` enters its `try/catch`, but `jobExists` force-reloads the cron store and can throw on a transient read/parse/normalization failure. Since workers are awaited with `Promise.all`, one rejection aborts the tick before completed results are applied, leaving collected jobs with persisted running markers and no finish update. Move this guard into the per-job error path or convert failures into a normal outcome.
  - confidence: 0.86
