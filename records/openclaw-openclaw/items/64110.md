## Summary

Keep open: the cron bug is still plausible on current main and the PR is a useful direction, but the current patch still has a timer-path correctness bug and lacks inspectable real behavior proof before merge.

## What This Changes

The PR adds force-reloaded cron job-existence checks before timed and manual execution plus seam tests for jobs removed between reservation and execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks persisted job existence after reservation but before delivery side effects, handles reload failures as per-job outcomes, and includes focused tests plus redacted real Feishu or cron proof.

## Reproduction Assessment

Yes, source-reproducible. Current main can capture a due cron job or prepared manual run, delete the persisted job, and still execute the captured snapshot before any fresh existence check stops side effects.

## Solution Assessment

No. The pre-execution guard is the right narrow direction, but the timer-path jobExists call must be handled inside per-job outcome logic so reload failures do not abort Promise.all and strand running markers.
