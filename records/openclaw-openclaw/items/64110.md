## Summary

Keep open: current main still has a source-reproducible deleted/queued cron execution gap, and this PR moves in a useful direction, but latest head still has a blocking timer-worker error path and lacks required real behavior proof.

## What This Changes

The PR adds cron job-existence checks before timed and manual execution plus seam tests for jobs removed before execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks persisted job existence after reservation but before delivery side effects, handles reload failures as per-job outcomes, and includes focused tests plus redacted real Feishu or cron proof.

## Reproduction Assessment

Yes, source-reproducible. Current main collects or prepares a cron job, then executes the captured job snapshot before any fresh persisted existence check can notice that the job was deleted.

## Solution Assessment

No. The pre-execution guard is the right narrow direction, but the timer-path check needs to be inside per-job handled outcome logic so reload failures do not abort Promise.all and skip applying completed results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:726`
  - body: The new guard awaits `jobExists` before `runDueJob` enters its `try/catch`. Since `jobExists` force-reloads the store, a read or parse failure rejects the worker; `Promise.all` then skips the completed-results application block, leaving jobs with persisted running markers. Return a per-job error or skipped outcome from inside the handled path instead.
