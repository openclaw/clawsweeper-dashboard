## Summary

Keep open: current main still has the queued cron deletion race, and this PR is aimed at the right boundary, but it is not merge-ready because the timer existence check can fail outside the per-job outcome path and no inspectable after-fix real behavior proof is attached.

## What This Changes

The PR adds cron job-existence checks before timed and queued manual execution, plus regression tests for jobs removed between reservation and execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks job existence after reservation but before timed or manual delivery side effects, contains reload failures as per-job outcomes, and keeps focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level path is to queue or collect a cron job, delete the persisted job before the worker starts, and observe current main executing the captured job snapshot without a pre-execution store recheck.

## Solution Assessment

No. The PR's pre-execution guard is the narrow maintainable direction, but the timed path must catch `jobExists` reload failures inside the per-job outcome path before this is safe to merge.
