## Summary

Keep this PR open: it targets a real cron deletion race that is still present on current main, and the later branch revisions moved the guard before execution as requested. It still has a blocking timer-path regression because the new existence check can reject outside the per-job error handling, and the PR body does not include inspectable after-fix real behavior proof.

## What This Changes

The PR adds a cron `jobExists` helper, pre-execution removal checks for timed and manual cron runs, and regression tests for skipped removed jobs.

## Best Possible Solution

Land a revised cron-service fix that checks deletion after reservation but before delivery side effects for timer and manual queued runs, treats store reload failures as normal per-job skipped/error outcomes, and keeps focused regression coverage on the cron service paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to collect or queue a cron job, delete its stored job before the worker starts, and observe current main executing the captured job snapshot without a pre-execution store recheck.

## Solution Assessment

No. The PR direction is the narrow maintainable fix, but the timer existence check must be handled inside the per-job outcome path so one store reload failure cannot abort the whole timer tick.
