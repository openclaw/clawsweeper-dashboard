## Summary

Keep open: the cron deletion race is source-reproducible and the PR is useful, but the current branch is not mergeable because the timer guard can throw before per-job error handling and the PR lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds a cron `jobExists` recheck before timed/manual cron execution and regression tests for jobs removed after scheduling but before execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks persisted job existence after reservation but before delivery side effects, handles reload failures as per-job outcomes, and keeps focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source path is to collect or queue a cron job, delete the persisted job before the worker starts, and observe current main executing the captured job snapshot without a pre-execution store recheck.

## Solution Assessment

No. The pre-execution guard is the narrow maintainable direction, but the timer path must catch existence-check reload failures inside per-job outcome handling before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:726`
  - body: The new `jobExists` guard runs before `runDueJob` enters its existing `try/catch`. Translate this into a per-job error or skipped outcome instead.
