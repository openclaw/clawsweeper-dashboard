## Summary

Keep open: the PR targets a source-reproducible cron deletion bug that current main still appears to have, but the current head has a blocking timer-path error-handling bug and lacks required after-fix real behavior proof.

## What This Changes

The PR adds cron job existence checks before timed and manual cron execution and adds tests for jobs removed between scheduling or manual-run preparation and execution.

## Best Possible Solution

Land a revised cron-service fix that rechecks persisted job existence after reservation but before delivery side effects, handles reload failures as per-job outcomes, and includes focused tests plus redacted real Feishu or cron proof.

## Reproduction Assessment

Yes. A high-confidence source path is to collect or prepare a cron job, delete the persisted job before execution starts, and observe current main executing the captured job snapshot before any fresh store-existence recheck.

## Solution Assessment

No. The pre-execution guard is the right narrow direction, but the timer-path check needs to be inside per-job handled outcome logic so reload failures do not abort Promise.all and skip applying completed results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:726`
