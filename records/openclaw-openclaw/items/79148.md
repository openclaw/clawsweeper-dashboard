## Summary

Keep open: current main and the latest release still hard-cap inbound prompt history metadata at 20 entries despite the documented historyLimit contract, so the fix remains useful. This release-branch PR still needs real behavior proof or a maintainer proof override, and maintainers should confirm whether to land it as a backport or port the same cap propagation through the current main inbound-context builder.

## What This Changes

The PR makes inbound prompt history serialization accept a caller-provided history cap, passes the resolved Discord/global group history limit into that serializer, and adds a focused unit regression test.

## Best Possible Solution

Wire the resolved inbound history cap through the current channel inbound context path into buildInboundUserContextPrefix, preserve the existing config contract, and land only after focused tests plus real Discord/gateway proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 hard-code prompt history serialization to 20 entries while docs expose configurable historyLimit values. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Mostly yes: threading the resolved cap into prompt serialization is the right bug-fix direction. The safer landing shape on current main is to carry the cap through the generic channel inbound context builder rather than only the older Discord context object on the release branch.
