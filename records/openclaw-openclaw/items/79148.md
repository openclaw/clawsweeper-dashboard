## Summary

Keep open: current main has the hard-coded 20-entry prompt-history cap this PR is trying to fix, but the proposed fallback applies the global group-chat cap to contexts that have not yet carried their own resolved limit, which can suppress or misreport histories from other channel paths such as Slack DM history.

## What This Changes

The PR adds an `InboundHistoryLimit` context field, threads Discord's resolved history limit into reply preparation, and lets inbound prompt metadata serialize beyond the old fixed 20-entry tail.

## Best Possible Solution

Carry the resolved history cap beside `InboundHistory` from each producer, use only that explicit cap in prompt metadata, and keep the 20-entry serializer default when no source cap is known.

## Reproduction Assessment

Yes, source-reproducible: current main always slices prompt metadata history to 20 while docs and channel code allow configured history limits above 20. I did not execute a live Discord scenario because this review was read-only.

## Solution Assessment

No, not as written. The core idea is right, but the fallback should be an explicit prepared fact from the history producer rather than applying the group-chat config to every inbound history context.
