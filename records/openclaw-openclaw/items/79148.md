## Label Justifications

- `P1`: Configured Discord history limits above 20 are silently ignored in prompt context, breaking an established channel history workflow for affected users.
- `merge-risk: 🚨 compatibility`: Merging changes the effective prompt history window for existing users who configured historyLimit above 20, which can alter prompt size and model behavior.

## Summary

Keep open: current main and the latest release still serialize inbound history with the fixed 20-entry cap, so the configured-cap bug remains. The merged history-shaping PR is related, but it centralized the history shape and media metadata without removing this serializer cap; this PR still needs proof and a maintainer decision on backport versus current-main adaptation.

## What This Changes

The PR makes inbound prompt history serialization use a caller-provided history cap, propagates Discord's resolved historyLimit through template context, and adds a regression test for 25 entries.

## Best Possible Solution

Land an adapted fix on the intended branch that carries the resolved history limit into prompt serialization while preserving current delivery-mode options, media history metadata, and focused regression coverage.

## Reproduction Assessment

Yes by source inspection: Discord resolves and collects history with the configured limit, but current prompt serialization slices InboundHistory back to the fixed 20-entry cap. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Yes, mostly: fixing prompt serialization is the right boundary for the reported bug. A current-main version must extend the existing options object rather than dropping sourceReplyDeliveryMode and must preserve the media-history behavior from the merged refactor.
