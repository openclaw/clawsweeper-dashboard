## Summary

The PR addresses a real current-main bug, but its reply-layer fallback introduces a cross-channel settings regression: channels that collect history using their own override can have that history suppressed by a different global limit. The branch needs a focused data-flow repair before merge.

## What This Changes

The PR makes inbound prompt-history serialization use a supplied limit and passes Discord’s resolved history limit through the inbound message context.

## Best Possible Solution

Carry each producer’s already-resolved collection limit alongside `InboundHistory`, make serialization consume that prepared fact, and retain the 20-entry safety default only when no resolved limit is available.

## Reproduction Assessment

Yes. Set `messages.groupChat.historyLimit` to zero and Signal’s channel history limit to five: Signal collects five entries, while the patched shared caller passes zero to serialization and emits none.

## Solution Assessment

No. The serializer should consume the producer’s resolved limit rather than recomputing an incomplete configuration fallback with different precedence.
