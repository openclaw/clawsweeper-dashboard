## Summary

Keep open: current main still has the fixed 20-entry inbound-history serializer cap, and this bot PR targets the right small surface, but the patch is not correct as written because its group-history fallback can mis-cap Slack DM and other producer-specific histories.

## What This Changes

The PR adds an optional inbound history limit to template context, threads Discord's resolved history limit through reply preparation, and makes inbound prompt metadata use a supplied cap instead of the fixed 20-entry cap.

## Best Possible Solution

Carry the producer-resolved history cap beside each `InboundHistory`, use only that explicit cap in the serializer, and keep the safe 20-entry default when no source cap is known.

## Reproduction Assessment

Yes. Source inspection shows current main always serializes at most 20 inbound history entries while docs and channel code allow configured history windows above 20; the PR's new regression is also source-proven through Slack DM history paths.

## Solution Assessment

No, not as written. Passing a cap into the serializer is the right direction, but the cap must come from the producer-resolved history window instead of falling back to `messages.groupChat.historyLimit` for every inbound history context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Carry each history producer's real cap:** `src/auto-reply/reply/get-reply-run.ts:612`
  - body: This fallback uses `messages.groupChat.historyLimit` whenever `InboundHistoryLimit` is absent. Slack DMs and other producers can fetch history with DM, channel, or account limits while still passing only `InboundHistory`; if the group cap is `0` or lower, this patch drops or truncates structured history that was already collected. Pass a producer-resolved limit with the history, or keep the serializer default when no explicit cap is present.
