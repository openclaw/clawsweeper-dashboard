## Summary

Keep open: current main and v2026.6.5 still serialize inbound history with a fixed 20-entry cap, but this PR's fallback can override channel-specific history limits when a producer does not populate the new field.

## What This Changes

The branch adds an inbound history limit field and serializer option so prompt metadata can use a configured cap instead of the fixed 20-entry limit.

## Best Possible Solution

Carry each producer's resolved history limit alongside `InboundHistory`, consume that prepared value in `buildInboundUserContextPrefix`, retain the 20-entry safety default only when no resolved producer limit exists, and preserve the current `sourceReplyDeliveryMode` option.

## Reproduction Assessment

Yes. Configure `messages.groupChat.historyLimit: 0` with a producer-specific positive history limit, such as Signal `historyLimit: 5`: the producer collects five entries, while this PR's shared fallback serializes zero when the new field is absent.

## Solution Assessment

No. The serializer should consume the producer's resolved limit rather than recomputing an incomplete configuration fallback in the reply layer.
