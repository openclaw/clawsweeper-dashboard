## Summary

Keep open: current main still serializes inbound history with a hard 20-entry cap even though Discord can collect a larger configured window, so the PR addresses a real source-reproducible bug. The remaining blockers are missing real Discord behavior proof and a maintainer scope decision for the release-branch patch versus the broader current-main history-limit surface.

## What This Changes

The branch adds an optional inbound history limit to prompt metadata serialization and passes the resolved Discord history limit into that path.

## Best Possible Solution

Land or replace this with a current-main-compatible fix that keeps the collected history window and serialized prompt window aligned, preserves existing prompt options, and has real Discord proof for the configured cap path.

## Reproduction Assessment

Yes from source inspection: Discord can collect more than 20 history entries using its resolved historyLimit, while current-main prompt serialization still slices InboundHistory to 20. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Mostly yes for the Discord bug on the release branch, but not fully proven as the best current-main shape until maintainers decide whether to propagate the cap through the shared inbound context used by sibling channels. A safer current-main implementation would merge the new history-limit option with the existing sourceReplyDeliveryMode option instead of replacing that options object.
