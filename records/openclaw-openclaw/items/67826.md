## Best Possible Solution

Preserve numeric-only setup, remove the stale username-resolution pieces, and decide separately whether target coercion belongs at the resolver boundary with owner-reviewed tests and proof.

Yes for the blocking review finding: current source, docs, tests, maintainer notes, and the PR diff show that @user setup resolution conflicts with numeric-only allowFrom. No high-confidence real path was supplied for non-string resolver inputs reaching Telegram through supported callers.

## Solution Assessment

No. The PR mixes useful defensive normalization with unsupported setup username resolution; a safer path is a narrow rebase that preserves numeric-only allowFrom and either validates at string call sites or explicitly revises the resolver contract.
