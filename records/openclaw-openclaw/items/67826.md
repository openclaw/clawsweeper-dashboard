## Summary

Keep open and do not merge as-is. Current main deliberately keeps Telegram setup `allowFrom` numeric-only, while this PR still reintroduces `@username` setup resolution; the remaining malformed target-input hardening is not proven reachable through the current string-only resolver contract.

## What This Changes

The PR widens Telegram setup allowFrom and runtime target resolver inputs to tolerate null, undefined, and non-string values, normalizes tokenFile inspection, and adds a setup-surface test for malformed allowFrom entries.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram setup, filter or reject malformed setup values without username lookup, and only add runtime target coercion if maintainers intentionally widen or defensively guard the resolver contract with focused tests.

## Reproduction Assessment

Yes for the review finding: current main's Telegram setup test asserts an @user allowFrom entry remains unresolved and does not fetch, while this PR adds a test expecting @user to resolve through getChat. Unclear for the original non-string runtime crash because the supported setup and resolver contracts still pass string[] inputs.

## Solution Assessment

No. The PR mixes useful defensive normalization with stale access-control setup behavior; a narrower fix should keep setup numeric-only and make any target hardening explicit at the supported resolver boundary.
