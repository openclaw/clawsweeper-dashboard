## Summary

Keep open. This PR still conflicts with the current numeric-only Telegram allowFrom setup direction, lacks contributor real behavior proof, and the remaining resolver hardening would require a maintainer decision about whether to widen a string-only contract.

## What This Changes

The PR widens Telegram setup allowFrom and runtime target resolver inputs to tolerate nullish/non-string values, normalizes tokenFile handling, and adds a malformed allowFrom setup test.

## Best Possible Solution

Preserve numeric-only Telegram setup, drop the stale username-resolution pieces, and separately decide whether Telegram target resolution should defensively coerce non-string values behind the string-only resolver contract.

## Reproduction Assessment

No. Current main has string-only setup and resolver contracts, and I did not establish a supported path where null or undefined reaches Telegram target resolution or setup allowFrom resolution.

## Solution Assessment

No. The PR is not the best fix as written because it conflicts with numeric-only Telegram allowFrom setup; a safer path is to keep setup numeric-only and review runtime resolver coercion as a separate contract decision.
