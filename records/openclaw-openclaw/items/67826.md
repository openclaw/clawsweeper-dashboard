## Summary

Keep open and do not merge as-is. Current main has moved Telegram setup to numeric-only `allowFrom` sender IDs, while this PR still carries stale setup username-resolution behavior and broadens resolver inputs beyond the current string-only plugin contract.

## What This Changes

The PR widens Telegram setup and target resolver inputs for malformed values, normalizes Telegram token-file inspection, and adds a setup-surface test for malformed `allowFrom` entries.

## Best Possible Solution

Keep Telegram setup numeric-only, leave legacy username repair in `openclaw doctor --fix`, and add any defensive normalization only at supported string boundaries unless maintainers explicitly change the resolver contract.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path: `telegramSetupWizard.allowFrom.resolveEntries` maps `@user` through `parseTelegramAllowFromId` and the existing test asserts it stays unresolved without a fetch, while the PR adds the opposite expectation.

## Solution Assessment

No. The PR is partly useful defensive hardening, but the setup behavior should preserve the merged numeric-only access-control contract; resolver input widening should be handled as an SDK/caller-boundary decision if maintainers want it.
