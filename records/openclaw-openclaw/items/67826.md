## Summary

Keep open for maintainer review: current main deliberately keeps Telegram setup allowFrom numeric-only while this PR head adds @user resolution coverage, and the remaining target-resolver hardening would broaden a string-only resolver contract without real Telegram proof.

## What This Changes

The PR branch changes Telegram setup and runtime target resolution to coerce or ignore nullish/non-string values, normalizes one tokenFile check, and adds setup-surface coverage.

## Best Possible Solution

Preserve numeric-only Telegram allowFrom setup, remove the stale username-resolution expectation, and only add target/normalization hardening after maintainer approval of the resolver contract boundary.

## Reproduction Assessment

Yes for the blocking review finding: current-main source, docs, and tests establish numeric-only setup and string-only resolver inputs, while the PR head expects @user lookup. No high-confidence real-world reproduction for the asserted undefined-trim crash is provided.

## Solution Assessment

No. The PR mixes defensive normalization with a stale Telegram setup username-resolution path; the safer direction is to keep numeric-only setup and make any malformed-input handling an explicitly reviewed resolver-contract change.
