## Summary

Keep this PR open, but do not merge it as-is. Current main intentionally requires numeric Telegram allowFrom IDs during setup, while the PR head still advertises and performs @username resolution on that setup access-control path; the remaining target hardening also raises a shared resolver contract question rather than being clearly implemented on main.

## What This Changes

The PR widens Telegram setup allowFrom and runtime target resolver inputs to tolerate null, undefined, and non-string values, normalizes tokenFile inspection, and adds setup-surface coverage for malformed allowFrom entries.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram allowFrom setup, keep legacy @username repair in doctor/fix paths, and either validate malformed resolver inputs at the shared caller boundary or add narrowly tested Telegram-only defensive coercion without changing the public resolver contract accidentally.

## Reproduction Assessment

Yes for the stale setup behavior: the PR head can be inspected or exercised through telegramSetupWizard.allowFrom.resolveEntries with entries like ["@user"], while current main's test expects @user to remain unresolved without fetching. For non-string target inputs, reproduction is unclear because the supported CLI and adapter contract pass strings.

## Solution Assessment

No, this PR is not the best way to solve the issue as-is. It mixes useful defensive hardening with obsolete setup username resolution and a resolver contract change; the safer path is a rebase that preserves current numeric-only setup and makes the target-input boundary explicit.
