## Summary

Keep this PR open for maintainer review: it has plausible defensive hardening, but the branch conflicts with the current numeric-only Telegram allowFrom setup contract and lacks real Telegram behavior proof.

## What This Changes

The branch widens Telegram setup allowFrom and runtime target resolver inputs to tolerate nullish or non-string values, normalizes one tokenFile check, and adds a malformed allowFrom unit test.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram allowFrom setup, drop the stale username-resolution pieces, and either validate at existing string boundaries or explicitly broaden the resolver contract with owner-reviewed tests and proof.

## Reproduction Assessment

Yes for the blocking review finding: current-main source, docs, and tests show setup allowFrom is numeric-only, while the PR expects @user to resolve through getChat. A supported real path for non-string resolver inputs is not established.
