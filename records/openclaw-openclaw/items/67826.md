## Summary

Keep this PR open, but it is not merge-ready: the branch conflicts with current Telegram setup policy by expecting `@user` allowFrom resolution again, while current main and the merged numeric-ID work require setup allowlists to use numeric sender IDs. The remaining runtime target hardening may still be useful, but the supported resolver contract is string-only and the contributor has not provided real behavior proof.

## What This Changes

The PR widens Telegram setup allowFrom and target resolver inputs to tolerate nullish/non-string values, normalizes `tokenFile` through `normalizeOptionalString`, and adds a malformed-entry setup test.

## Best Possible Solution

Preserve numeric-only Telegram setup and separately review a narrowly scoped defensive runtime target-normalization change if maintainers decide to widen the string-only resolver contract.

## Reproduction Assessment

No. Current main no longer has the setup username-resolution helper, and the setup/resolver contracts feed `string[]`; I did not establish a supported current-main path where null or undefined reaches Telegram target resolution.

## Solution Assessment

No. The setup part is not the best fix because it conflicts with current numeric-ID authorization policy; a safer patch would keep setup numeric-only and isolate any defensive runtime normalization behind the supported resolver contract.
