## Summary

Keep open. The Mantis Discord lane is useful, but this protected PR needs maintainer/security review because the workflow exposes Discord secrets to setup/install/build on selected refs and the runner can pass with a channel that is not in the configured guild.

## What This Changes

The PR adds a Mantis QA design doc, a `pnpm openclaw qa mantis discord-smoke` QA Lab command with tests and artifacts, a manual secret-backed GitHub workflow, docs glossary updates, and a changelog entry.

## Best Possible Solution

Keep the Mantis runner direction, but merge only after secrets are step-scoped and Discord channel/guild validation is explicit, with regression coverage for mismatched IDs.

## Reproduction Assessment

Yes for the PR findings by source inspection: the workflow env scope and setup action show the secret exposure path, and the runtime discards the guild channel list before accepting the channel. Not applicable as a user bug reproduction because this is a new feature PR.

## Solution Assessment

No. The Mantis lane is a plausible maintainable direction, but the current patch should narrow secret exposure to the smoke step and prove the selected channel belongs to the selected guild before posting or reporting pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope Discord secrets to the smoke step:** `.github/workflows/mantis-discord-smoke.yml:126-129`
  - body: These secrets are defined at job scope, so the selected ref's local setup action, dependency lifecycle scripts, and `pnpm build` all receive the bot token before the smoke command runs. Because the validator allows open PR heads, move the Discord secrets to the `Run Mantis Discord smoke` step, or create a one-shot token file there, so candidate code only sees them when the runner executes.
  - confidence: 0.91
- **[P2] Validate the channel belongs to the selected guild:** `extensions/qa-lab/src/mantis/discord-smoke.runtime.ts:308-314`
  - body: The smoke fetches `/guilds/${guildId}/channels` but discards the result, then accepts any `/channels/${channelId}` the bot can access. A mismatched guild/channel config can pass and post to a channel in another guild; assert the channel id appears in the guild channel list or check `channel.guild_id` before reporting success.
  - confidence: 0.87
