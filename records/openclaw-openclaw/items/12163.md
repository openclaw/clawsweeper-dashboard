## Summary

Keep this issue open. Current main and the latest release still miss the reported OpenAI directive path: OpenAI TTS speed is honored after it reaches the provider, but the OpenAI directive parser claims voice/model only and not speed; the open linked PR is the implementation path until it lands or is replaced.

## What This Changes

The issue asks OpenAI TTS to honor `[[tts:speed=...]]` directives by parsing a speed override and sending it in OpenAI speech requests.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/74089 so OpenAI-owned directive parsing accepts valid speed overrides with focused normal and telephony request-body coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenAI does not claim `speed` in `parseDirectiveToken`, while the directive router only stores provider overrides after `handled: true`.

## Solution Assessment

Yes for the linked PR direction, not for the original core-level sketch. The maintainable fix belongs in the OpenAI provider parser with OpenAI's `0.25..4.0` range and focused directive/request tests.
