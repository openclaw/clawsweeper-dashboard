## Summary

Keep open: current main still drops OpenAI `[[tts:speed=...]]` directive overrides at the OpenAI provider parser, and open PR #74089 is already the active closing implementation path.

## What This Changes

The issue requests OpenAI TTS support for `[[tts:speed=...]]` directive overrides so speech rate reaches the OpenAI request body.

## Best Possible Solution

Review and land or replace PR #74089 with a narrow OpenAI provider parser fix, OpenAI range validation, and focused regression tests; close this issue after the implementation merges.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with OpenAI selected, `[[tts:speed=1.5]]` reaches provider directive parsing, but OpenAI returns unhandled for `speed`, so no provider override is written.

## Solution Assessment

Yes. Provider-owned OpenAI directive handling is the narrow maintainable fix because core routing already delegates provider-specific keys and OpenAI synthesis already serializes speed.
