## Summary

Keep this issue open. Current main still has the reported OpenAI directive gap: OpenAI speed is honored once it reaches the provider, but `[[tts:speed=...]]` is not claimed by the OpenAI directive parser, and the open linked PR should remain the implementation path until it lands or is replaced.

## What This Changes

The issue requests passing `[[tts:speed=...]]` through to OpenAI TTS so users can control OpenAI speech rate per reply.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/74089 so OpenAI-owned directive parsing accepts valid speed overrides and keeps normal plus telephony request serialization covered by focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenAI does not claim `speed` in `parseDirectiveToken`, while the directive router only stores provider overrides after `handled: true`.

## Solution Assessment

Yes for the linked PR direction, not for the original core-level sketch. The maintainable fix belongs in the OpenAI provider parser with OpenAI's `0.25..4.0` range and focused directive/request tests.
