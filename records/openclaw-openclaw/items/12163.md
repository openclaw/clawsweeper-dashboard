## Summary

Keep open: current main still drops OpenAI `[[tts:speed=...]]` directive overrides in the OpenAI provider parser, while the lower OpenAI speed plumbing already exists and https://github.com/openclaw/openclaw/pull/74089 is the active closing implementation path.

## What This Changes

The issue requests OpenAI TTS support for `[[tts:speed=...]]` directive overrides so speech rate reaches the OpenAI audio speech request.

## Best Possible Solution

Provider-owned OpenAI directive parsing should accept the speed aliases under the existing voice-settings policy, validate OpenAI's documented range, keep core routing unchanged, and close this issue only after the linked implementation lands. ([platform.openai.com](https://platform.openai.com/docs/api-reference/audio/createTranscription?lang=javascript&utm_source=openai))

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with OpenAI selected, `[[tts:speed=1.5]]` reaches provider directive parsing, but the OpenAI parser returns unhandled for `speed`, so no provider override is written.

## Solution Assessment

Yes. The narrow maintainable fix is in `extensions/openai/speech-provider.ts` because core routing already delegates provider-specific keys and OpenAI synthesis already serializes speed.
