## Summary

Keep open. Current main still drops the reported OpenAI `[[tts:speed=...]]` directive path because the OpenAI speech provider parser only handles voice/model tokens, even though lower-level OpenAI speed config and request serialization already exist. The issue is also already paired with open PR #74089, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for OpenAI TTS `[[tts:speed=...]]` directives to pass a speech-rate override through to the OpenAI speech request, matching the behavior users have with ElevenLabs.

## Best Possible Solution

Land a provider-owned OpenAI fix that handles `speed` aliases inside `extensions/openai/speech-provider.ts`, gates them on `policy.allowVoiceSettings`, validates OpenAI's `0.25..4.0` range, and adds focused regression tests for preferred-provider and explicit-provider directive routing. Keep the lower-level synthesis plumbing as-is unless tests show a separate config-validation gap.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `parseTtsDirectives` with the OpenAI provider selected and `[[tts:speed=1.5]]`: current routing requires `handled: true`, but OpenAI's parser returns `handled: false` for `speed`, so no OpenAI provider override is created.

## Solution Assessment

Yes. The best direction is the narrow provider-owned parser change proposed by the linked PR, because OpenAI has its own valid speed range and the core directive router already supports provider-specific parsing.
