## Summary

Keep this issue open. Current main still serializes OpenAI speed when it reaches the provider, but the reported `[[tts:speed=...]]` directive path is still blocked because the OpenAI speech provider parser does not claim `speed`; the open linked fix PR is the right implementation path to review and land.

## What This Changes

The issue requests OpenAI TTS support for `[[tts:speed=...]]` directives so a validated speed override reaches the OpenAI `/audio/speech` request.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/74089 so OpenAI-owned directive parsing accepts valid speed overrides, rejects invalid values non-fatally, and keeps request serialization covered by focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenAI does not claim `speed` in `parseDirectiveToken`, while the directive router only stores provider overrides after `handled: true`.

## Solution Assessment

Yes for the linked PR direction, not for the original core-level sketch. The maintainable fix belongs in the OpenAI provider parser with OpenAI's `0.25..4.0` range and focused directive/request tests.
