## Summary

Keep open: current main still drops OpenAI `[[tts:speed=...]]` directive overrides at the provider parser, and PR #74089 is already the active closing implementation path.

## What This Changes

Support OpenAI TTS `[[tts:speed=...]]` directives so per-reply speed reaches the OpenAI speech request.

## Best Possible Solution

Review and land or replace PR #74089 with a narrow OpenAI speech-provider parser fix, OpenAI range validation, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: parse `[[tts:speed=1.5]]` with OpenAI selected, where the router requires `handled: true` but the OpenAI parser returns unhandled for `speed`.

## Solution Assessment

Yes. Provider-owned OpenAI directive handling is the narrow maintainable fix because core routing already delegates provider-specific keys and lower-level OpenAI speed request plumbing exists.
