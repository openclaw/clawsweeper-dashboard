## Summary

Keep open: the PR targets a real `sessions_send` routing bug, but the patch reads only the target entry's top-level `channel` while current session delivery state is normalized across `route`, `deliveryContext`, and `lastChannel`; it also provides only targeted test output for a user-visible cross-channel delivery change.

## What This Changes

The PR hoists target session entry loading in `sessions_send` and passes `targetSessionEntry.channel` into the nested gateway `agent` call instead of the hardcoded internal webchat channel.

## Best Possible Solution

Derive `sendParams.channel` from normalized session delivery context with the internal fallback only when no channel is known, add focused regression coverage, then require real cross-channel proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main still hardcodes internal webchat for the nested `sessions_send` agent call, and current delivery helpers/tests show external route metadata can live outside `entry.channel`. No live multi-channel replay was run in this read-only review.

## Solution Assessment

No. The tool-layer direction is plausible, but reading only `targetSessionEntry.channel` is too narrow because the codebase already centralizes canonical delivery state in normalized session delivery context.
