## Summary

Keep open: this is a focused candidate for the linked hardcoded-channel bug, but it is not merge-ready because the patch reads only the top-level session `channel` while current session delivery state can be canonicalized from `route`, `deliveryContext`, and legacy `last*` fields, and the PR still has only test-output proof.

## What This Changes

The PR moves target session entry loading earlier in `createSessionsSendTool` and uses `targetSessionEntry.channel`, falling back to webchat, for the nested gateway `agent` call.

## Best Possible Solution

Derive the initial `sessions_send` agent channel from normalized session delivery context, keep the internal fallback only when no channel is known, and add focused regression plus redacted real cross-channel proof.

## Reproduction Assessment

We have source-level reproduction evidence: current main still passes `channel: INTERNAL_MESSAGE_CHANNEL` for `sessions_send` gateway agent calls, and adjacent tests still assert webchat-tagged inter-session calls. I did not establish a full live multi-channel replay in this read-only review.

## Solution Assessment

No. The tool-layer fix is plausible, but using only `targetSessionEntry.channel` is too narrow because current routing code already centralizes canonical channel state in normalized session delivery context.
