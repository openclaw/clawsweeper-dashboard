## Summary

Keep open for maintainer review: the PR targets a real `sessions_send` misroute, but the patch derives the delivery channel from only the top-level session `channel` field while current routing helpers already normalize `route`, `deliveryContext`, `lastChannel`, and origin fields. It also still has only targeted test output, not real cross-channel behavior proof.

## What This Changes

The PR loads the resolved target session entry earlier in `sessions_send` and uses `entry.channel` as the nested gateway `agent` call channel instead of the hardcoded internal webchat channel.

## Best Possible Solution

Derive `sendParams.channel` from normalized session delivery state, fall back to internal webchat only when no channel is known, add focused regression coverage for stale-webchat/external-context sessions, and provide real cross-channel proof before merge.

## Reproduction Assessment

Yes at source level: current main still passes `channel: INTERNAL_MESSAGE_CHANNEL` for `sessions_send`, and adjacent helpers/tests show external delivery metadata can live outside `entry.channel`. I did not run a live multi-channel replay in this read-only review.

## Solution Assessment

No. The tool-layer fix is plausible, but reading only `targetSessionEntry.channel` is too narrow because OpenClaw already centralizes canonical delivery state in `deliveryContextFromSession` and outbound target helpers.
