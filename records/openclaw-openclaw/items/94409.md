## Summary

Keep open: this PR targets a real `sessions_send` misroute, but the patch is too narrow because current session routing state is normalized across `deliveryContext`, `route`, `lastChannel`, and `channel`, and the supplied terminal proof does not exercise the production cross-channel path.

## What This Changes

The PR hoists target session entry loading in `sessions_send` and uses its top-level `channel` field for the nested gateway `agent` call instead of `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Derive `sendParams.channel` from normalized session delivery context with the internal fallback only when no channel is known, add focused regression coverage, then require real cross-channel proof or a maintainer proof override.

## Reproduction Assessment

Yes at source level: current main still hardcodes internal webchat for the nested `sessions_send` agent call, and current delivery helpers/tests show external route metadata can live outside `entry.channel`. I did not run a live multi-channel replay in this read-only review.

## Solution Assessment

No. The tool-layer direction is plausible, but reading only `targetSessionEntry.channel` is too narrow because the codebase already centralizes canonical delivery state in normalized session delivery context.
