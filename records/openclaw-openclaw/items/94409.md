## Summary

Keep open: this PR targets a real current-main `sessions_send` hardcoded-channel bug, but the patch is too narrow because it reads only the target session's top-level `channel` instead of the normalized delivery context already used elsewhere for external routes.

## What This Changes

The PR moves target session loading earlier in `src/agents/tools/sessions-send-tool.ts` and uses `targetSessionEntry?.channel` for the nested gateway `agent` call channel instead of always using internal webchat.

## Best Possible Solution

Derive the nested `sessions_send` channel from normalized session delivery context, keep the internal fallback only when no channel is known, and add focused regression plus redacted real cross-channel proof.

## Reproduction Assessment

Yes at source level: current main still passes `channel: INTERNAL_MESSAGE_CHANNEL` for `sessions_send`, and existing delivery-context helpers/tests show external route metadata can live outside top-level `entry.channel`. I did not run a live multi-channel replay in this read-only review.

## Solution Assessment

No. The tool-layer idea is plausible, but the submitted implementation is not the best fix because it bypasses the normalized session delivery context that already handles stale webchat and external route state.
