## Summary

Keep open: this is a focused bug-fix PR for a real `sessions_send` routing problem, but the patch still derives the nested send channel from only `SessionEntry.channel` while current session delivery state can be stored in `deliveryContext`, `route`, or `lastChannel`; the external contributor proof is also not production `sessions_send` proof.

## What This Changes

The PR moves target session entry loading earlier in `sessions_send` and passes `targetSessionEntry?.channel?.trim() || INTERNAL_MESSAGE_CHANNEL` as the nested gateway `agent` channel.

## Best Possible Solution

Derive `sendParams.channel` from the normalized session delivery context with an internal fallback only when no channel is known, add focused regression coverage for external-route session shapes, and require real cross-channel proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes at source level: current main still hardcodes internal webchat for the nested `sessions_send` agent call, and current delivery helpers/tests show external route metadata can live outside `entry.channel`. I did not run a live multi-channel replay in this read-only review.

## Solution Assessment

No. The tool-layer direction is plausible, but reading only `targetSessionEntry.channel` is too narrow because the codebase already centralizes routable session delivery state in normalized delivery context.
