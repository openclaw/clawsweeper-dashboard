## Summary

Keep open: current main still wires plugin-declared skills into OpenClaw’s workspace skill snapshot, but the embedded PI agent resource loader is created without those plugin skill paths, so agent command/discovery surfaces can remain out of sync with the manifest contract.

## What This Changes

The issue asks OpenClaw to make plugin manifest `skills` entries discoverable to the agent runtime’s skill-loading paths instead of only resolving them inside plugin roots.

## Best Possible Solution

Thread enabled plugin skill directories into the embedded PI resource loader’s skill discovery, and mirror the same behavior in compaction if needed, while preserving current OpenClaw precedence and containment checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: enable a plugin with `skills: ["./skills"]`, observe OpenClaw’s workspace loader includes it, then observe the embedded PI `DefaultResourceLoader` is created without the plugin paths that power `resourceLoader.getSkills()` discovery.

## Solution Assessment

Is this the best way to solve the issue? Yes: use the existing dependency seam for additional skill paths or an equivalent contained loader integration, rather than unconditional global symlink publication.
