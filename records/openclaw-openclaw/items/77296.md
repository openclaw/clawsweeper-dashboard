## Summary

Keep open: current main has the prior manifest-to-snapshot wiring, but source inspection still shows plugin-declared skill directories are consumed in place rather than exposed through the file-discovery roots named in the report.

## What This Changes

The issue asks OpenClaw to make manifest-declared plugin skills discoverable by the agent's file-based skill loader, not only by OpenClaw's workspace skill snapshot.

## Best Possible Solution

Add a generic contained search-path or publication integration for enabled plugin skill directories so prompt snapshots and agent file discovery see the same enabled skill set, with no plugin-specific ids.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live gateway failure in this review, but source inspection confirms the reported publication gap: enabled plugin skill dirs are loaded directly and no current path publishes them into the file-discovery roots named by the report.

## Solution Assessment

Is this the best way to solve the issue? The direction is right, but the safest implementation is likely to register plugin skill roots with the agent discovery layer or create a guarded managed mirror rather than adding unconditional global symlinks.

## Repair Work Prompt

Fix the plugin skill discovery gap reported in https://github.com/openclaw/openclaw/issues/77296. Current main resolves enabled manifest `skills` entries and merges them into workspace skill snapshots, but file-based agent skill discovery still cannot find those plugin skill dirs. Implement a generic fix that makes enabled plugin-declared skills discoverable by the agent runtime without hardcoding plugin ids. Prefer passing/registering contained plugin skill roots into the relevant discovery layer; if mirroring is necessary, keep realpath containment, collision handling, cleanup/update behavior, and third-party trust boundaries explicit. Add regression coverage for an enabled external-style plugin with `openclaw.plugin.json` `skills: ["./skills"]`, and preserve existing precedence: workspace, project agent, personal agent, managed, bundled, then extra/plugin-low-precedence roots. Do not change plugin manifest semantics beyond making the documented `skills` field usable. Add a single-line `CHANGELOG.md` fix entry if the implementation changes user-facing behavior.
