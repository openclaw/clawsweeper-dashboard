## Summary

Keep open: current main still shows the plugin manifest skill directories being consumed in place rather than published to the agent file-discovery roots, and open PR #77328 explicitly targets this issue as the fix candidate.

## What This Changes

The issue requests that plugin manifest `skills` entries become discoverable through the agent skill discovery paths or an equivalent supported mechanism.

## Best Possible Solution

Land a generic plugin-skill discovery integration that keeps enabled manifest skills visible to both workspace snapshots and agent file-discovery consumers without hard-coding plugin ids.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main resolves plugin skill directories and loads them directly, while the managed and `.agents/skills` discovery roots remain separate; the live gateway ENOENT was not rerun locally.

## Solution Assessment

Is this the best way to solve the issue? Yes, the requested direction is the right boundary, but the safest implementation is a generic registered discovery path or guarded managed mirror rather than plugin-specific symlinks.
