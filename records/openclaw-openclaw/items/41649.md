## Summary

Keep open: current main does not implement the model-change transcript archive/handoff behavior, and this PR is still a useful implementation candidate, but the proposed patch has blocking correctness and security issues before it can merge.

## What This Changes

The PR adds a default-model-change session handoff flow that archives old transcripts, writes handoff JSON from recent user messages, rotates session IDs, wires reload/startup handling, and injects handoff context on the first new-session turn.

## Best Possible Solution

Implement the model-change path as a maintained gateway/session feature: detect default-model changes across hot reload and restart paths, preserve explicit user overrides, clear stale runtime model/context metadata, rotate or archive affected transcripts, store sanitized handoff data with private atomic file handling and retention semantics that do not drop fresh sessions, and deliver the handoff only when an agent run will actually proceed.

## Reproduction Assessment

Yes. Source inspection on current main shows model config reload leaves sessions untouched, and the PR defects have concrete reproduction paths: switch models with more than five active sessions, or send first-turn traffic that short-circuits after the handoff is consumed.

## Solution Assessment

No, not as written. The direction is plausible, but the patch must preserve all fresh handoffs, defer one-shot consumption until a run is committed, harden handoff storage, persist the documented archive path, add the required changelog entry, and rebase onto current gateway/session code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve all fresh model handoffs:** `src/config/sessions/model-handoff.ts:291-292`
  - body: cleanupHandoffFiles keeps only the newest five handoff files in the whole sessions directory, and archiveAndHandoffOnModelChange calls it immediately after creating handoffs. A model switch with more than five active sessions will delete most newly-created handoffs before those users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only after the run will proceed:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: This reads and deletes the one-shot handoff before later early-return guards run. If the first post-switch event is an unauthorized control command, empty normalized body, or other pre-run rejection, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P3] Persist the archived transcript path:** `src/config/sessions/store.ts:838-840`
  - body: The handoff file is written before archiveSessionTranscripts runs, and this later assignment only mutates the returned object. The persisted JSON never gets archivePath, despite the documented file format, so write the handoff after archiving or rewrite it atomically with the archive path.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/config/sessions/model-handoff.ts:139-147`
  - body: This is a user-facing feat(sessions) change, but the PR does not update CHANGELOG.md. Add a single-line entry under the active release section so the model-change handoff behavior is included in release notes.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds sensitive handoff sidecar storage for recent user messages and should use the same private, atomic write treatment as existing session artifacts.

Concerns:

- **[medium] Write handoff files with private atomic storage:** `src/config/sessions/model-handoff.ts:165`
  - body: createModelHandoff persists recent user messages with fs.writeFileSync and no explicit private mode or atomic replacement. Existing session artifacts use 0o600-style private writes, so this new sidecar should not rely on process umask or non-atomic direct writes.
