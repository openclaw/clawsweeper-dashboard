## Summary

Keep this PR open: current main still does not implement model-change session archival/handoff, but the branch is not mergeable as-is because source review shows blocking correctness issues, security-sensitive handoff storage concerns, and missing real behavior proof.

## What This Changes

The PR adds default-model-change handling that archives old session transcripts, writes recent-user-message handoff files, rotates session IDs, and injects handoff context into the new model's first reply.

## Best Possible Solution

Implement a maintained session reset/handoff service that covers hot reload and restart paths, uses collision-resistant private atomic storage, retains per-session handoffs until use or expiry, consumes only when a run will proceed, and lands with approved prompt/privacy semantics plus real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main only restarts heartbeat and stales model-catalog state for model-related config changes, while the PR's failure modes are visible in the changed files without needing a live gateway run.

## Solution Assessment

No. The direction is plausible, but this branch is not the best mergeable solution until restart handling, sidecar storage, retention, one-shot consumption timing, archive metadata, and prompt/privacy approval are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts on restart resets:** `src/gateway/server-startup.ts:78`
  - body: The startup path only calls clearSessionModelFields, so restart-based reloads clear metadata but leave the old transcript in place and create no handoff file, defeating the PR's model-isolation goal.
  - confidence: 0.88
- **[P1] Use collision-resistant handoff filenames:** `src/config/sessions/model-handoff.ts:131-132`
  - body: Sanitizing and truncating the session key can map distinct sessions to the same handoff file, which can overwrite or leak one session's recent user messages into another session's first turn.
  - confidence: 0.87
- **[P1] Retain fresh handoffs for every affected session:** `src/config/sessions/model-handoff.ts:291-292`
  - body: cleanupHandoffFiles keeps only the newest five files in the entire sessions directory, so a model switch affecting more than five sessions can delete fresh handoffs before users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs after pre-run guards pass:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The one-shot handoff is read and deleted before later control-command and empty-body early returns, so the first post-switch event can discard the handoff without ever sending it to the model.
  - confidence: 0.86
- **[P2] Clear contextTokens in field-only resets:** `src/config/sessions/store.ts:752-755`
  - body: clearSessionModelFields deletes model and modelProvider but leaves contextTokens, so fallback or startup resets can keep the previous model's context window for status and budgeting until another run rewrites the entry.
  - confidence: 0.81
- **[P3] Write archivePath before persisting the handoff:** `src/config/sessions/store.ts:817-824`
  - body: createModelHandoff writes the JSON before archiveSessionTranscripts returns, and the later archivePath assignment only mutates memory, so the documented archivePath is never persisted on disk.
  - confidence: 0.78
