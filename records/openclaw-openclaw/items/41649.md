## Summary

Keep open: current main still lacks automatic model-change transcript archiving or handoff, but this PR has blocking correctness and privacy issues plus no contributor-provided real behavior proof.

## What This Changes

The branch adds config-reload/startup logic that archives session transcripts, writes recent-user-message handoff sidecars, rotates session IDs, and injects handoff context on the new model's first turn.

## Best Possible Solution

Implement model-change handling through a maintained session reset/handoff service that covers hot reload and restart paths, uses collision-resistant private atomic storage or an existing private store, delays one-shot consumption until a model run proceeds, and has approved prompt/privacy semantics.

## Reproduction Assessment

Yes. Source inspection shows current main has no automatic model-change handoff/reset path, and the PR defects are reproducible from the changed code paths without needing a live run.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable solution until restart handling, sidecar privacy, filename collisions, retention, one-shot consumption timing, and archive metadata are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts on restart-based reloads:** `src/gateway/server-startup.ts:78`
  - body: `gateway.reload.mode=restart` and mixed changes that require restart skip the hot-reload handler, but this startup path only clears model fields. That leaves the old transcript in place with no handoff file, so the new default model can still load the previous model's conversation history after restart.
  - confidence: 0.88
- **[P1] Use collision-resistant handoff filenames:** `src/config/sessions/model-handoff.ts:131-132`
  - body: The filename is made by replacing unsupported characters with `_` and truncating, so distinct session keys like keys that differ only by punctuation or after the first 100 characters can map to the same handoff file. Because the file contains recent user messages, a collision can overwrite or leak one session's context into another session's first turn.
  - confidence: 0.87
- **[P1] Preserve handoffs for every affected session:** `src/config/sessions/model-handoff.ts:291-292`
  - body: The cleanup keeps only the newest five handoff files in the entire sessions directory and runs immediately after creating handoffs. A default-model switch with more than five affected sessions will delete fresh handoffs before most users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only after the run will proceed:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The handoff is read and deleted before later early-return guards. If the first post-switch event is an unauthorized control command, empty normalized body, or another pre-run rejection, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P2] Clear stale context tokens in field-only resets:** `src/config/sessions/store.ts:752-755`
  - body: `clearSessionModelFields` is used by the startup and fallback paths, but it only deletes `model` and `modelProvider`. After a restart-based reset or handoff failure, the session can still carry the previous model's context window for status and budgeting until another run rewrites it.
  - confidence: 0.81
- **[P3] Persist the archive path before writing the handoff:** `src/config/sessions/store.ts:837-840`
  - body: The handoff JSON is written before `archiveSessionTranscripts` runs, and the later `archivePath` assignment only mutates the returned object. The on-disk handoff never contains the documented archived transcript path.
  - confidence: 0.78

## Security Review

  - confidence: 0.87
