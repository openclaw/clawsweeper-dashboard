## Summary

Keep open: current main still lacks model-change transcript archive/handoff behavior, and this PR still has blocking restart, retention, storage, and proof gaps.

## What This Changes

The branch adds a model-change session handoff flow that archives old transcripts, stores recent user messages, rotates session IDs, and injects the handoff on the next first turn.

## Best Possible Solution

Implement a maintained model-change reset/handoff service on current session lifecycle code with collision-resistant private atomic storage, per-session retention until safe consumption, approved prompt/privacy semantics, focused regressions, and a redacted live smoke.

No. The direction may be useful, but this branch is not the best mergeable solution until it is rebased to current lifecycle code, fixes the storage/retention/restart defects, and gets real behavior proof plus prompt/privacy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts on restart reloads:** `src/gateway/server-startup.ts:78`
  - body: `gateway.reload.mode=restart` and mixed restart-triggering config changes skip the hot-reload handoff path, while startup only clears model fields. That leaves the old transcript in place, so the replacement model can still inherit prior conversation history after restart.
  - confidence: 0.88
- **[P1] Use collision-resistant handoff paths:** `src/config/sessions/model-handoff.ts:131-132`
  - body: Sanitizing and truncating the session key maps distinct keys such as `a:b` and `a_b`, or long keys sharing the first 100 safe characters, to the same handoff file. Because the file contains recent user messages, a collision can overwrite or expose another session's context.
  - confidence: 0.88
- **[P1] Keep per-session handoffs until consumed:** `src/config/sessions/model-handoff.ts:291-292`
  - body: Cleanup keeps only the newest five handoff files in the whole sessions directory and runs immediately after creating handoffs. A default-model switch with more than five affected sessions can delete fresh handoffs before most users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only after run guards pass:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The one-shot handoff is read and deleted while building system prompts before later early-return guards. If the first post-switch event is unauthorized control-command traffic or another gated event, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P2] Clear context tokens in fallback resets:** `src/config/sessions/store.ts:752-754`
  - body: `clearSessionModelFields` is used by startup and fallback paths but only deletes `model` and `modelProvider`. Those paths can leave the prior model's context window in session status and budgeting until another run rewrites the entry.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The PR introduces concrete privacy-sensitive storage concerns for recent user-message handoff files.

Concerns:

- **[high] Make handoff paths collision-resistant:** `src/config/sessions/model-handoff.ts:131`
  - body: Sanitizing and truncating session keys into filenames can map distinct sessions to one handoff file, which can overwrite or expose recent user messages across sessions.
  - confidence: 0.88
- **[medium] Write handoffs with private atomic storage:** `src/config/sessions/model-handoff.ts:165`
  - body: The handoff JSON contains recent user messages but is written with direct `fs.writeFileSync` and no explicit private mode or atomic replacement, unlike existing private session-store writes.
  - confidence: 0.82
