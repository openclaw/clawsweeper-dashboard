## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the model-change archive and sanitized handoff flow, the narrower linked #41907 is closed unmerged, and the PR branch still has blocking correctness and privacy-hardening issues before it should merge.

## What This Changes

The PR adds a model-change session handoff flow that archives old transcripts, writes per-session handoff JSON from recent user messages, rotates session IDs, wires model config reload/startup paths, and injects the handoff on the first session turn.

## Best Possible Solution

Implement the model-change path as a maintained gateway/session feature: detect default-model changes across hot reload and restart paths, preserve explicit user overrides, clear stale runtime model and context metadata, archive or rotate affected transcripts, store sanitized handoff data with private/atomic file handling and retention semantics, and deliver the handoff only when an agent run will actually proceed.

## Reproduction Assessment

Yes. Current-main source checks show no archive/handoff path, and the PR defects have concrete reproduction paths: switch models with more than five active sessions, or trigger a first-turn early return after a handoff is created.

## Solution Assessment

No, not as written. The proposed direction is plausible, but the patch must preserve all fresh handoffs, defer one-shot consumption until the model run is committed, harden private handoff storage, and rebase onto the current gateway startup/reload structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve all fresh model handoffs:** `src/config/sessions/model-handoff.ts:291-292`
  - body: `cleanupHandoffFiles` keeps only the newest five handoff files globally, and `archiveAndHandoffOnModelChange` runs it immediately after creating one handoff per affected session. A model switch with more than five active sessions will delete most newly-created handoffs before those users send their first post-switch message, so the feature silently fails for the majority of sessions.
  - confidence: 0.92
- **[P2] Consume handoffs only after the run will proceed:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: This reads and deletes the one-shot handoff before later early-return guards run. If the first post-switch event is an unauthorized control command, an empty normalized body, or another pre-run rejection, the handoff is lost without ever reaching the model, and the next real user message no longer gets the model-change context.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a new file containing recent user-message excerpts and should harden that storage before merge.

Concerns:

- **[medium] Write handoff files with private permissions:** `src/config/sessions/model-handoff.ts:165`
  - body: `createModelHandoff` persists recent user messages using `fs.writeFileSync` without an explicit private mode or atomic write. Existing session artifacts such as transcripts and sessions.json use 0o600-style private writes, so this new sensitive sidecar should use the same confidentiality and durability treatment.
