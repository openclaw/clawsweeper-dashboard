## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts when restart reloads run:** `src/gateway/server-startup.ts:78`
  - body: The startup/restart path only calls `clearSessionModelFields`, so restart-based model config reloads still leave old transcripts in place. That defeats the archive/handoff behavior for `gateway.reload.mode=restart` or mixed changes that require restart.
  - confidence: 0.88
- **[P1] Use collision-resistant handoff paths:** `src/config/sessions/model-handoff.ts:131-132`
  - body: Sanitizing and truncating `sessionKey` can map distinct sessions such as `a:b` and `a_b` to the same file. Because the file contains recent user messages, this can overwrite or expose another session's context.
  - confidence: 0.9
- **[P1] Keep handoffs per session until consumed:** `src/config/sessions/model-handoff.ts:291-292`
  - body: The cleanup loop keeps only the newest five handoff files across the whole sessions directory. A model switch affecting more than five sessions can delete fresh handoffs before those users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only when a run will proceed:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The one-shot handoff is read and deleted before later early-return guards. Unauthorized control-command traffic or an empty-body path can consume the handoff without ever sending it to the model.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces privacy-sensitive handoff files without collision-resistant private atomic storage.

Concerns:

- **[high] Make handoff paths collision-resistant:** `src/config/sessions/model-handoff.ts:131`
  - body: Sanitizing and truncating session keys into filenames can map distinct sessions to one handoff file, which can overwrite or expose recent user messages across sessions.
  - confidence: 0.9

  - body: The handoff JSON contains recent user messages but is written with direct `fs.writeFileSync` and no explicit private mode or atomic replacement, unlike the safer storage pattern expected for private session state.
