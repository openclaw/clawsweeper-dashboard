## Summary

## What This Changes

The PR adds a model-change session handoff flow that archives old transcripts, writes recent user-message handoff files, rotates session IDs, and injects the handoff on the next first turn.

## Best Possible Solution

Rework this on current session lifecycle code with collision-resistant private atomic storage, per-session retention until safe consumption, restart/hot-reload parity, focused regressions, and redacted real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts when restart reloads run:** `src/gateway/server-startup.ts:78`
  - body: The startup path only calls `clearSessionModelFields`, so `gateway.reload.mode=restart` or mixed restart-triggering config changes still leave old transcripts in place. That lets the replacement model load the prior conversation history, defeating the archive/handoff behavior this PR is adding.
  - confidence: 0.88
- **[P1] Use collision-resistant handoff paths:** `src/config/sessions/model-handoff.ts:131-132`
  - body: Sanitizing and truncating the session key maps distinct keys such as `a:b` and `a_b`, or long keys with the same first 100 safe characters, to the same file. Because the file contains recent user messages, a collision can overwrite or expose another session's context.
  - confidence: 0.9
- **[P1] Keep handoffs per session until consumed:** `src/config/sessions/model-handoff.ts:291-292`
  - body: Cleanup keeps only the newest five handoff files in the whole sessions directory and runs immediately after creating handoffs. A default-model switch with more than five affected sessions can delete fresh handoffs before most users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs after run guards pass:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The one-shot handoff is read and deleted before later early-return guards. If the first post-switch event is unauthorized control-command traffic or an empty-body normalization path, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P2] Clear context tokens in fallback resets:** `src/config/sessions/store.ts:752-754`
  - body: `clearSessionModelFields` is used by startup and fallback paths but deletes only `model` and `modelProvider`. Those paths can leave the prior model's `contextTokens` in session status and budgeting until another run rewrites the entry.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The PR introduces privacy-sensitive recent-message handoff files without collision-resistant private atomic storage.

Concerns:

- **[high] Make handoff paths collision-resistant:** `src/config/sessions/model-handoff.ts:131`
  - body: Sanitizing and truncating session keys into filenames can map distinct sessions to one handoff file, which can overwrite or expose recent user messages across sessions.
  - confidence: 0.9
- **[medium] Write handoffs with private atomic storage:** `src/config/sessions/model-handoff.ts:165`
  - body: The handoff JSON contains recent user messages but is written with direct `fs.writeFileSync` and no explicit private mode or atomic replacement, unlike the safer storage pattern expected for private session state.
  - confidence: 0.82
