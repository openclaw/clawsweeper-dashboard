## Summary

Keep open: current main still does not implement automatic model-change transcript archiving or handoff, but this PR is not mergeable as-is because source review shows blocking restart, collision, retention, and one-shot-consumption defects, plus missing real behavior proof.

## What This Changes

The branch adds model-change session handling that archives transcripts, writes recent-user-message handoff sidecars, rotates session IDs, and injects handoff context into the new model's first reply.

## Best Possible Solution

Implement a maintained session reset/handoff service that covers hot reload and restart paths, uses collision-resistant private atomic storage or an existing private store, delays one-shot consumption until a run will proceed, and has approved prompt/privacy semantics.

## Reproduction Assessment

Yes, at source level. Current main shows model-related reload only refreshes heartbeat/model-catalog state, and the PR defects are visible from the changed code paths without needing a live gateway run.

## Solution Assessment

No. The direction is plausible, but this branch is not the best mergeable solution until restart handling, private collision-resistant storage, retention, consumption timing, and archive metadata are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts when reload falls back to restart:** `src/gateway/server-startup.ts:78`
  - body: `gateway.reload.mode=restart` and mixed changes that require restart skip the hot-reload handler, but this startup path only clears model fields. That leaves the old transcript in place with no handoff file, so the new default model can still load the previous model's conversation history after restart.
  - confidence: 0.88
- **[P1] Make handoff filenames collision-resistant:** `src/config/sessions/model-handoff.ts:131-132`
  - body: The filename is made by replacing unsupported characters with `_` and truncating, so distinct session keys can map to the same handoff file. Because the file contains recent user messages, a collision can overwrite or leak one session's context into another session's first turn.
  - confidence: 0.87
- **[P1] Keep all fresh handoffs for affected sessions:** `src/config/sessions/model-handoff.ts:291-292`
  - body: The cleanup keeps only the newest five handoff files in the entire sessions directory and runs immediately after creating handoffs. A default-model switch with more than five affected sessions will delete fresh handoffs before most users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only when a run will proceed:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The handoff is read and deleted before later early-return guards. If the first post-switch event is an unauthorized control command, empty normalized body, or another pre-run rejection, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P2] Clear contextTokens in field-only model resets:** `src/config/sessions/store.ts:752-755`
  - body: `clearSessionModelFields` is used by the startup and fallback paths, but it only deletes `model` and `modelProvider`. After a restart-based reset or handoff failure, the session can still carry the previous model's context window for status and budgeting until another run rewrites it.
  - confidence: 0.81
- **[P3] Persist the archive path before writing the handoff:** `src/config/sessions/store.ts:837-840`
  - body: The handoff JSON is written before `archiveSessionTranscripts` runs, and the later `archivePath` assignment only mutates the returned object. The on-disk handoff never contains the documented archived transcript path.
  - confidence: 0.78
