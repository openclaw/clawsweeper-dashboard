## Summary

Keep this PR open: current main still lacks the proposed model-change transcript archive and handoff flow, but the branch is not mergeable as-is because it stores recent user messages in collision-prone, non-atomic handoff files and can lose or skip handoffs in normal reload paths.

## What This Changes

The PR adds model-change session handling that archives old transcripts, writes recent-user-message handoff files, rotates session IDs, and injects handoff context into the first post-switch reply.

## Best Possible Solution

Implement a maintained model-change reset/handoff service that covers hot reload and restart paths, uses private collision-resistant atomic storage, retains per-session handoffs until consumed after pre-run gates, and has approved prompt/privacy semantics plus regression and live proof.

## Reproduction Assessment

Yes, at source level. Current main lacks automatic model-change transcript handoff, and the branch defects are traceable in the changed storage, cleanup, startup, and first-turn consumption paths.

## Solution Assessment

No. The direction may be useful, but this branch is not the best mergeable solution until restart coverage, collision-resistant private storage, retention, consumption timing, and prompt/privacy decisions are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Archive transcripts on restart-based reloads:** `src/gateway/server-startup.ts:75`
  - body: `gateway.reload.mode=restart` and mixed restart-triggering config changes skip the hot-reload handoff path, but startup only clears model fields. That leaves the old transcript in place, so the replacement model can still inherit prior conversation history after restart.
  - confidence: 0.88
- **[P1] Make handoff filenames collision-resistant:** `src/config/sessions/model-handoff.ts:131-132`
  - body: Sanitizing and truncating the session key means distinct keys such as `a:b` and `a_b`, or long keys sharing the first 100 safe characters, resolve to the same handoff file. Because the file contains recent user messages, a collision can overwrite or expose another session's context.
  - confidence: 0.88
- **[P1] Retain fresh handoffs for every affected session:** `src/config/sessions/model-handoff.ts:291-292`
  - body: The cleanup keeps only the newest five handoff files in the entire sessions directory and runs immediately after creating handoffs. A default-model switch with more than five affected sessions will delete fresh handoffs before most users send their first post-switch message.
  - confidence: 0.92
- **[P2] Consume handoffs only after pre-run gates:** `src/auto-reply/reply/get-reply-run.ts:275`
  - body: The one-shot handoff is read and deleted while building system prompts before later early-return guards. If the first post-switch event is an unauthorized command or empty normalized body, the handoff is lost without ever reaching the model.
  - confidence: 0.86
- **[P2] Clear contextTokens in field-only reset paths:** `src/config/sessions/store.ts:749-751`
  - body: `clearSessionModelFields` is used by startup and fallback paths but only deletes `model` and `modelProvider`. After those paths run, the session can still carry the prior model's context window into status and budgeting until another run rewrites it.
  - confidence: 0.81
- **[P3] Persist archive paths before writing handoffs:** `src/config/sessions/store.ts:837-840`
  - body: The handoff JSON is written before `archiveSessionTranscripts` runs, and the later `archivePath` assignment only mutates the returned object. The documented archived transcript path is never persisted to the on-disk handoff file.
  - confidence: 0.78
