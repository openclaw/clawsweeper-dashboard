## Summary

Keep open: current main still has the reported control-flow gap, and the related PRs are useful but not merged proof that the issue is fixed.

## What This Changes

The issue asks for `gateway.restart` continuations to be durably queued, or explicitly reported as not queued, when the restart request coalesces with an in-flight restart.

## Best Possible Solution

Land a narrow restart-boundary fix that either merges an accepted continuation into the pending or in-flight restart sentinel, or returns an explicit non-queued result, with regression coverage for the gateway tool entering the unconsumed in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before the tool's `beforeEmit` hook can persist the sentinel.

## Solution Assessment

No. Current main acknowledges the coalesced restart while dropping the accepted continuation on the in-flight branch; the narrow maintainable fix belongs at the gateway-tool/restart-scheduler boundary, coordinated with #73812.
