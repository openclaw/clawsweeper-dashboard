## Summary

Keep open: current main still has the in-flight coalescing gap. The restart tool accepts a `continuationMessage`, but if `scheduleGatewaySigusr1Restart()` sees an already-emitted, unconsumed restart signal it returns `ok`/`coalesced` before the tool's `beforeEmit` hook can write the sentinel that startup later uses to enqueue the continuation; the related fix PRs were closed unmerged or covered adjacent Telegram delivery behavior rather than this exact path.

## What This Changes

The issue requests that `gateway.restart` either preserve an accepted `continuationMessage` across coalesced in-flight restarts or explicitly report that the continuation was not queued.

## Best Possible Solution

Add a narrow restart-boundary contract fix so an already-emitted coalesced restart either queues the same-session continuation safely and durably, or returns an explicit non-queued result that the agent can act on.

## Reproduction Assessment

Yes, source-reproducible. On current main, leave a restart emitted but unconsumed, then call `gateway.restart` with `continuationMessage`; the scheduler returns from the `hasUnconsumedRestartSignal()` branch before the gateway tool's sentinel-writing hook can run.

## Solution Assessment

Yes. The narrow maintainable solution is in the gateway restart/tool contract: safely preserve the accepted one-shot continuation for the same pending restart, or explicitly report that it was not queued, without folding in broad task checkpointing.
