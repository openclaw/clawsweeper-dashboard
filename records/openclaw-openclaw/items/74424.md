## Summary

Keep open. Current main still has a source-level path where a `gateway` restart with `continuationMessage` returns `ok`/`coalesced` during an already-emitted restart before the tool's sentinel-writing hook can run, and the linked implementation candidate has not landed in this checkout.

## What This Changes

The issue requests that `gateway` restart calls carrying `continuationMessage` either durably queue that continuation across coalesced restarts or explicitly report that no continuation was queued.

## Best Possible Solution

Keep this issue open until PR #73812 or an equivalent narrow fix lands. The end state should make continuation queuing explicit at the scheduler/tool boundary for already-in-flight coalesced restarts, or return an explicit not-queued result, with regression coverage for the unconsumed SIGUSR1 branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but not marked handled, then call the gateway restart tool with `continuationMessage`; current main returns from the coalesced branch before the tool's `beforeEmit` hook can persist the continuation.

## Solution Assessment

No. Current main is not the best solution because it acknowledges a continuation-bearing coalesced restart without queuing or reporting the continuation; the safer fix is a narrow scheduler/tool contract change plus regression coverage.
