## Summary

Keep open: current `main` still has the reported source-level path where a gateway restart request with `continuationMessage` can return `coalesced: true` from the in-flight restart branch before the continuation-writing hook is run. The linked open PR #73812 is already the implementation candidate, so this should not be closed or queued as a duplicate repair job yet.

## What This Changes

Ensure `gateway` restart calls with `continuationMessage` either durably persist and deliver the continuation when they coalesce with an existing restart, or explicitly report that no continuation was queued.

## Best Possible Solution

Keep this issue open until PR #73812 or an equivalent narrow fix lands. The durable end state should persist or merge the accepted continuation for coalesced restart requests, or return an explicit non-queued result, with regression coverage at the gateway-tool/restart-scheduler boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but not yet consumed, then call the gateway restart tool with `continuationMessage`; current `main` returns from the unconsumed coalescing branch before the tool's sentinel-writing hook can run.

## Solution Assessment

No. Current `main` is not the best solution because it can acknowledge a coalesced restart while dropping the accepted continuation; the narrow maintainable repair belongs at the gateway-tool/restart-scheduler boundary rather than in broad task checkpointing.
