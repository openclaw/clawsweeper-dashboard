## Summary

Keep open: current main still has the reported restart-boundary gap, and the related open PR is not merged proof that the continuation contract is fixed.

## What This Changes

The issue asks for gateway restart continuations to be persisted or explicitly reported as not queued when a restart request coalesces with an already in-flight restart.

## Best Possible Solution

Land one canonical restart-boundary fix that either merges an accepted continuation into the pending or in-flight restart sentinel, or returns an explicit non-queued result, with regression coverage for the gateway tool entering the in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with `continuationMessage`; current main returns from the in-flight coalescing branch before the tool's `beforeEmit` hook can persist the sentinel.

## Solution Assessment

No. Current main is not the best solution because it acknowledges a coalesced restart while dropping the accepted continuation; the narrow maintainable repair belongs at the gateway-tool/restart-scheduler boundary, coordinated with the open related PR.
