## Summary

Keep open: current main still has the reported in-flight restart coalescing gap. The gateway tool accepts a continuation and only persists it from the scheduler emit hook, while the already-emitted restart branch returns `ok`/`coalesced: true` before that hook can run; #73812 and #74443 are useful related implementation context, but no merged main fix is present.

## What This Changes

Ensure a gateway restart request with `continuationMessage` either durably queues and delivers that continuation after a coalesced restart or reports that it was not queued.

## Best Possible Solution

Land one narrow restart-boundary fix that either merges an accepted continuation into the pending/in-flight restart sentinel before startup drains it, or returns an explicit non-queued result so the caller can recover, with regression coverage for the gateway tool entering the unconsumed in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway restart tool with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before the tool's sentinel-writing hook can persist the continuation.

## Solution Assessment

Yes, the issue's requested contract is the right maintainable direction: either make accepted continuations durable across coalescing or make non-delivery explicit. The narrowest code boundary is between `createGatewayTool()` and `scheduleGatewaySigusr1Restart()`, not a broader task-checkpoint system.
