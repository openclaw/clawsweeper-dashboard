## Summary

Keep open: current main still has the reported control-flow gap, and the related open PR is implementation context rather than merged proof that this is fixed.

## What This Changes

A gateway restart tool call with `continuationMessage` should not silently drop that continuation when it coalesces with an existing restart.

## Best Possible Solution

Land a narrow restart-boundary fix that either persists or merges the accepted continuation into the pending/in-flight restart sentinel, or returns an explicit non-queued result.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before the tool’s `beforeEmit` hook can persist the sentinel.

## Solution Assessment

Yes. The issue’s proposed boundary is the narrow maintainable direction: preserve the accepted one-shot continuation or explicitly report non-delivery, while leaving the broader checkpoint/resume RFC separate.
