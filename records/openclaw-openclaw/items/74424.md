## Summary

Keep open: current main still has the reported control-flow gap, and the related fix work is not merged proof that the continuation is preserved on the in-flight coalescing path.

## What This Changes

The issue reports that a gateway restart tool call with `continuationMessage` can return success with `coalesced: true` while losing the continuation when a prior restart is already in flight.

## Best Possible Solution

Land a narrow restart-boundary fix that either persists or merges an accepted continuation into the pending/in-flight restart sentinel, or returns an explicit non-queued result, with regression coverage for the gateway tool entering the in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway restart tool with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before the gateway tool's `beforeEmit` hook can write the sentinel.

## Solution Assessment

Yes. The narrow maintainable direction is to preserve the accepted one-shot continuation or explicitly report non-delivery, rather than folding this into the broader checkpoint/auto-resume RFC.
