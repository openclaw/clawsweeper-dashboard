## Summary

Current main still has the reported source-level gap: the gateway tool accepts a continuation and only persists it from a restart emit hook, while the already-in-flight coalescing path returns success before that hook can run. The issue should stay open; the adjacent open PR does not currently appear to cover the exact unconsumed coalescing branch, and the exact narrow PR is closed unmerged.

## What This Changes

The issue reports that gateway restart requests with `continuationMessage` can return a successful coalesced result while dropping the continuation when another restart is already in-flight.

## Best Possible Solution

Keep the issue open until a narrow restart-boundary fix lands that either durably merges the accepted continuation into the pending/in-flight restart sentinel or returns an explicit non-queued result, with regression coverage for the gateway-tool call entering the already-in-flight coalesced branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway restart tool with `continuationMessage`; current main returns from the unconsumed coalescing branch before the tool's sentinel-writing hook can run.

## Solution Assessment

No. Current main is not the best solution because it acknowledges the coalesced restart while dropping the accepted continuation; the narrow maintainable fix belongs at the gateway-tool/restart-scheduler boundary rather than in broader task checkpointing.
