## Summary

Current main still has the reported source-level gap: the gateway restart tool accepts `continuationMessage`, but the unconsumed in-flight coalescing branch returns success before the hook that writes the continuation sentinel can run. This should stay open because the exact fix is not merged on main, and the related PRs are still either open-adjacent or closed unmerged.

## What This Changes

The issue requests that `gateway` restart calls with `continuationMessage` either durably queue and deliver the continuation after a coalesced restart or explicitly report that the continuation was not queued.

## Best Possible Solution

Land a narrow restart-boundary fix that either merges the accepted continuation into the pending or already in-flight restart sentinel, or returns an explicit non-queued result, with regression coverage for the gateway tool entering the unconsumed coalesced branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway restart tool with `continuationMessage`; current main returns from the `hasUnconsumedRestartSignal()` coalescing branch before the tool's sentinel-writing hook can run.

## Solution Assessment

No. Current main acknowledges the coalesced restart while dropping the accepted continuation, so the maintainable fix belongs at the gateway-tool/restart-scheduler sentinel boundary rather than in broader restart checkpointing.
