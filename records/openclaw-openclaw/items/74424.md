## Summary

Keep open: current main still returns `ok`/`coalesced` from the already-emitted restart branch before the gateway tool's sentinel-writing hook can run, so an accepted `continuationMessage` can still be lost. PR #73812 is adjacent continuation work, while PR #74443 was the exact coalesced-continuation repair but is closed and unmerged.

## What This Changes

The issue reports that `gateway.restart` can accept `continuationMessage` during an in-flight coalesced restart without durably queuing or explicitly rejecting that continuation.

## Best Possible Solution

Add a narrow restart-boundary fix that either durably preserves same-session accepted continuations for the already-emitted coalesced path or reports that the continuation was not queued, while leaving broad checkpoint/resume work to its separate trackers.

## Reproduction Assessment

Yes, as a high-confidence source path: leave a SIGUSR1 restart emitted but unconsumed, then call `gateway.restart` with `continuationMessage`. Current main returns from `hasUnconsumedRestartSignal()` before `beforeEmit` can persist that continuation.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the accepted one-shot continuation or explicitly report it was not queued. PR #73812's current diff does not cover this exact persistence gap, and the broader checkpoint RFC does not need to block it.
