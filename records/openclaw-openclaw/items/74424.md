## Summary

Keep open: current `main` at the review SHA still has the reported source-level path where an explicit gateway restart can return `coalesced: true` while skipping the hook that persists the continuation. The linked open PR #73812 remains the implementation candidate, so this issue should not be closed or queued as a duplicate repair job yet.

## What This Changes

The issue asks OpenClaw to preserve or explicitly reject a `gateway.restart` `continuationMessage` when the restart request coalesces with an already scheduled or in-flight gateway restart.

## Best Possible Solution

Keep this issue open until PR #73812 or an equivalent narrow fix lands. The durable end state should either merge/persist the accepted continuation for the in-flight coalesced restart branch or make the tool response explicitly say that no continuation was queued, with regression coverage for that branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but not yet marked handled, then call the gateway restart tool with `continuationMessage`; current `main` returns from the unconsumed coalescing branch before the tool's sentinel-writing hook can run.

## Solution Assessment

No. Current `main` is not the best solution because it can acknowledge a coalesced restart while dropping the accepted continuation; the narrow maintainable fix belongs at the gateway-tool/restart-scheduler boundary rather than in broad task checkpointing.
