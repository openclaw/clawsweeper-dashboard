## Summary

Keep open: current main still has the reported in-flight coalescing gap, because `gateway.restart` accepts `continuationMessage` but the already-emitted restart branch returns `ok`/`coalesced` before the tool's sentinel-writing hook can run. Open PR #73812 is active implementation context, not merged proof that this is fixed.

## What This Changes

A `gateway` restart request with `continuationMessage` should either durably queue the continuation when coalesced with an in-flight restart or explicitly report that it was not queued.

## Best Possible Solution

Land a narrow restart-boundary fix, likely through #73812 or a replacement, that preserves the accepted one-shot continuation or returns an explicit not-queued result.

## Reproduction Assessment

Yes. A high-confidence source path is to leave a SIGUSR1 restart emitted but unconsumed, then call `gateway.restart` with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before `beforeEmit` can persist the continuation sentinel.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve the accepted one-shot continuation or report that it was not queued; the broader checkpoint/resume RFC does not need to block this bug fix.
