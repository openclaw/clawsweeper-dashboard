## Summary

Keep open: current main still has the already-emitted SIGUSR1 coalescing path where a gateway restart with `continuationMessage` can return success before the gateway tool's sentinel-writing hook runs. The linked PR #73812 is useful implementation context, but it is still open and does not prove current main is fixed.

## What This Changes

The issue asks that a gateway restart `continuationMessage` be durably queued or explicitly reported as not queued when the restart request coalesces with an already in-flight restart.

## Best Possible Solution

Land PR #73812 or an equivalent narrow restart-boundary fix that persists or merges the accepted one-shot continuation into the pending/in-flight restart sentinel, or returns an explicit non-queued result.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with `continuationMessage`; current main returns from the unconsumed-signal branch before the tool's `beforeEmit` hook can persist the sentinel.

## Solution Assessment

Yes. The narrow maintainable fix is at the restart scheduling/sentinel boundary; the broader checkpoint/resume RFC is related but not required to honor this existing one-shot continuation contract.
