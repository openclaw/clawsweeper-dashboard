## Summary

Keep open. Current `main` still has the reported restart-boundary gap: the gateway tool accepts a continuation but persists it only from an emit hook, while the already-in-flight coalescing branch returns success before that hook can run. The related open PR is adjacent, and the exact narrow PR is closed unmerged, so this should remain tracked until a maintainer folds in or lands the specific coalesced-continuation fix.

## What This Changes

The issue requests that `gateway` restart calls with `continuationMessage` either durably queue the continuation across coalesced restarts or explicitly report that the continuation was not queued.

## Best Possible Solution

Land a narrow restart-boundary fix that either merges the accepted `continuationMessage` into the pending/in-flight restart sentinel before returning coalesced success, or changes the tool result so agents can see that no continuation was queued. Add regression coverage for the gateway-tool call entering the already-in-flight coalesced scheduler branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `hasUnconsumedRestartSignal()` true, then invoke the gateway restart tool with `continuationMessage`; current `main` returns from the in-flight coalescing branch before the tool's `beforeEmit` sentinel write can run.

## Solution Assessment

No. Current `main` is not the best solution because it acknowledges the coalesced restart while dropping the accepted continuation; the maintainable fix belongs at the gateway-tool/restart-scheduler sentinel boundary, not in the broader checkpoint RFC work.
