## Summary

Keep open: current main still has the reported restart-continuation gap, and the linked #73812/#74443 discussion is useful fix context rather than merged proof. The in-flight coalescing branch returns success before the gateway tool's sentinel-writing hook can run.

## What This Changes

Ensure a gateway restart request with `continuationMessage` either durably queues that continuation across coalesced restarts or explicitly reports that it was not queued.

## Best Possible Solution

Land one narrow restart-boundary fix that preserves the accepted continuation through in-flight coalescing or returns an explicit non-queued result, with regression coverage for the gateway tool entering the unconsumed SIGUSR1 branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before `beforeEmit` can persist the sentinel.

## Solution Assessment

No. Current main is not the best solution because it acknowledges the coalesced restart while dropping the accepted continuation; the maintainable fix belongs at the gateway-tool/restart-scheduler boundary, coordinated with #73812 or the exact #74443 fix idea.
