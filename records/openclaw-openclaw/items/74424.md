## Summary

Keep open: current main still has a source-level path where the gateway restart tool accepts a continuation but the restart scheduler returns `coalesced: true` before the sentinel-writing hook can run. The discussion already identifies open PR #73812 as the implementation candidate, so this issue should wait for that PR or an equivalent narrow fix to land.

## What This Changes

The issue requests that gateway restart `continuationMessage` be durably queued across coalesced restarts, or that the tool response explicitly report when no continuation was queued.

## Best Possible Solution

Land PR #73812 or an equivalent narrow fix that preserves the accepted continuation for coalesced restarts, or changes the gateway tool response to state that no continuation was queued, with regression coverage for the in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart signal unconsumed, then call the gateway restart tool with `continuationMessage`; current main returns from the in-flight coalescing branch before the tool's sentinel-writing hook can run.

## Solution Assessment

Yes, the narrow boundary proposed by the issue is the best direction: fix the gateway-tool/restart-scheduler continuation contract rather than folding this into the broader task-checkpoint RFC in #60864.
