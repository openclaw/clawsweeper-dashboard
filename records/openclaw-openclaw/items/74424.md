## Summary

Keep open: current main still has the reported in-flight coalescing gap, because `gateway.restart` accepts `continuationMessage` but the already-emitted restart branch returns `ok`/`coalesced` before the tool's sentinel-writing hook can run. Open PR #73812 is active implementation context, not merged proof that this is fixed.

## What This Changes

The issue asks `gateway.restart` to preserve or explicitly reject a `continuationMessage` when the restart request coalesces with a pending or in-flight gateway restart.

## Best Possible Solution

Land a narrow restart-boundary fix that either durably merges the accepted continuation into the pending/in-flight restart sentinel or returns an explicit non-queued result, while keeping broader checkpoint/resume work separate.

## Reproduction Assessment

Yes. A high-confidence source path is to leave a SIGUSR1 restart emitted but unconsumed, then call `gateway.restart` with `continuationMessage`; current main returns from `hasUnconsumedRestartSignal()` before `beforeEmit` can persist the continuation sentinel.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve the accepted one-shot continuation or report that it was not queued; the broader task-checkpoint RFC does not need to block this bug fix.
