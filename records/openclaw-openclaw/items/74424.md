## Summary

Keep open: current main still has the reported control-flow gap. The gateway tool accepts a continuation and returns the scheduler result, but the already-in-flight coalescing branch returns success before the sentinel-writing hook can run; the related open and closed PRs are useful context but do not make this implemented in the checked-out main.

## What This Changes

The issue requests that gateway restart continuations be durably queued or explicitly reported as not queued when a restart request coalesces with an in-flight restart.

## Best Possible Solution

Land a narrow restart-boundary fix that either merges an accepted continuation into the pending/in-flight restart sentinel or returns an explicit non-queued result, with regression coverage for the gateway tool entering the unconsumed in-flight coalescing branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a SIGUSR1 restart emitted but unconsumed, then call the gateway tool restart action with continuationMessage; current main returns from hasUnconsumedRestartSignal before the tool's beforeEmit hook can persist the sentinel.

## Solution Assessment

No. Current main is not the best solution because it acknowledges the coalesced restart while dropping the accepted continuation on the in-flight branch; the narrow maintainable fix belongs at the gateway-tool/restart-scheduler boundary.
