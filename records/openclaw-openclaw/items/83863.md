- `P1`: A live ACP/Codex child workflow can show durable success even when no final deliverable reached the parent.
- `impact:message-loss`: The reported failure hides a missing completion message or final artifact behind successful task state.
- `impact:session-state`: Durable task and flow state can diverge from the child session's actual completion contract.

## Summary

Keep open. Current main has partial startTurn-result handling for Codex ACP, but the source still leaves legacy ACP done events, subagent ok finalization, and failed required completion delivery able to present progress-only text as successful durable task state.

## What This Changes

The issue asks OpenClaw to treat ACP/Codex child runs that produce only progress text and no required final deliverable as blocked or failed instead of successful.

## Best Possible Solution

Carry explicit final-deliverable and required-completion-delivery state through ACP and subagent finalization, mapping missing final output or exhausted required delivery to the existing blocked terminal outcome while preserving real final reports as succeeded.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a legacy ACP turn can emit progress text then done, and a subagent ok outcome can complete with frozen progress text before required delivery later fails. I did not execute a live replay because this review was read-only.

## Solution Assessment

Yes, the requested direction is the best maintainable fix if it uses structured final-deliverable and delivery state rather than broad text guessing. The current startTurn failure handling is useful but does not cover the remaining legacy ACP and subagent completion paths.
