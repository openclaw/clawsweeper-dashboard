## Label Justifications

- `P1`: A live ACP/Codex child-task workflow can show success even when the parent receives no final deliverable.
- `impact:message-loss`: The reported failure is a missing completion message/final deliverable being hidden behind successful task state.
- `impact:session-state`: Durable task and flow state can diverge from the child session's actual completion contract.

## Summary

Keep open. Current main has partial ACP terminal-result handling, but source still allows a completed ACP turn or `ok` subagent outcome with only progress text to leave task/flow state as succeeded, and completion retry exhaustion only flips `deliveryStatus` to failed. The related invalid-encrypted-content report is a distinct failure from the same incident cluster, not a duplicate: https://github.com/openclaw/openclaw/issues/83836.

## What This Changes

The issue asks OpenClaw to stop marking ACP/Codex child tasks as cleanly succeeded when an expected final deliverable is missing or completion delivery fails.

## Best Possible Solution

Carry an explicit final-deliverable/completion-delivery contract through ACP and subagent task finalization, mapping missing final output or exhausted completion delivery to the existing blocked terminal outcome while preserving real final reports as succeeded.

## Reproduction Assessment

Yes from source and incident evidence, though not by executing a local replay in this read-only pass. Simulating a parented ACP turn that emits only progress text then reports completed, or a subagent `ok` outcome whose final delivery exhausts retries, follows current code to `status=succeeded` unless a blocked terminal outcome is explicitly set.

## Solution Assessment

Yes, the requested direction is the right fix shape if it uses structured completion/delivery state rather than broad text-only guessing. The maintainable path is to reuse `terminalOutcome=blocked` and add focused regressions for missing final output, retry-limit delivery failure, and valid final output.
