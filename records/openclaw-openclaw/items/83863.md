## Label Justifications

- `P1`: A live ACP/Codex child-task workflow can show success even when the parent receives no final deliverable.
- `impact:message-loss`: The reported failure is a missing completion message or final deliverable being hidden behind successful task state.
- `impact:session-state`: Durable task and flow state can diverge from the child session's actual completion contract.

## Summary

Keep open. Current main has partial startTurn-result handling for Codex ACP, but the durable task contract still permits legacy ACP done events and subagent ok outcomes to surface progress text as successful task state, and completion delivery failure remains represented as deliveryStatus rather than a blocked terminal outcome; the related invalid-encrypted-content report is adjacent but distinct: https://github.com/openclaw/openclaw/issues/83836.

## What This Changes

The issue asks OpenClaw to avoid marking ACP/Codex child tasks as successful when they only produced progress text or failed completion delivery.

## Best Possible Solution

Carry an explicit final-deliverable and completion-delivery contract through ACP and subagent task finalization, mapping missing final output or exhausted required completion delivery to the existing blocked terminal outcome while preserving real final reports as succeeded.

## Reproduction Assessment

Yes from source and incident evidence, though not by executing a local replay in this read-only pass. Simulating a legacy ACP turn that emits only progress text then done, or a subagent ok outcome whose required completion delivery exhausts retries, follows current code to succeeded unless blocked terminal metadata is explicitly set.

## Solution Assessment

Yes, the requested direction is the maintainable fix if it uses structured completion and delivery state rather than broad last-text guessing. The repair should reuse terminalOutcome=blocked and add regressions for missing final output, failed required delivery, and valid final output.
