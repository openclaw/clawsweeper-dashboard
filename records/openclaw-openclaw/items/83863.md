## Label Justifications

- `P1`: A live ACP/Codex child-task workflow can report success even when no final deliverable or completion message reaches the requester.
- `impact:message-loss`: The failure hides a missing completion message or final artifact behind successful task state.
- `impact:session-state`: Durable task, subagent, cron, and parent-session state can diverge from the child run's actual completion contract.

## Summary

Keep open. Current main partially addresses this class by honoring structured ACP `startTurn` failures and by crediting already-delivered subagent mirrors, but legacy ACP `done` events and subagent `ok` finalization can still make durable task state look successful while the required final artifact or completion delivery is missing.

## What This Changes

Require ACP/Codex child tasks that expect a final artifact to distinguish progress-only/interim output and failed completion delivery from successful task completion.

## Best Possible Solution

Carry explicit final-deliverable and required-completion-delivery state through ACP and subagent finalization, mapping missing final output or exhausted required delivery to blocked/failed task state while preserving real final reports as succeeded.

## Reproduction Assessment

Yes, at source level. Simulating legacy ACP progress text followed by `done`, or a subagent `ok` run whose required completion delivery exhausts retries, still follows current code to succeeded task state unless a narrow blocked terminal outcome is explicitly set.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if implemented as structured completion/delivery state rather than broad last-text guessing. Reuse existing blocked/failed terminal surfaces and add regressions for missing final output, failed required delivery, and valid final output.

Fix the false-success completion contract reported at https://github.com/openclaw/openclaw/issues/83863. Current main partially handles structured ACP startTurn failures and requester delivery mirrors, but legacy ACP `done` events and subagent `ok` finalization can still record progress-only output or failed required completion delivery as succeeded. Keep the fix bounded: carry structured final-deliverable/required-delivery state through ACP and subagent task finalization, map missing final output or exhausted required completion delivery to the existing blocked/failed task/flow surface, preserve legitimate final text-only completions as succeeded, and avoid adding a user-facing config option.
