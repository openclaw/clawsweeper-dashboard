## Label Justifications

- `P1`: A live ACP/Codex child workflow can present success even when the parent and user receive no final deliverable.
- `impact:message-loss`: The reported failure hides a missing completion message or final artifact behind successful task state.
- `impact:session-state`: Durable task, subagent, and parent-flow state can diverge from the child session's actual completion contract.

## Summary

Keep open. Current main partially handles structured `startTurn` failures, but the remaining source paths still allow a completed ACP turn or subagent ok outcome to carry only progress text into successful task state while required completion delivery can fail separately.

## What This Changes

The issue asks OpenClaw to treat ACP/Codex child sessions with only progress output or failed required completion delivery as blocked or failed instead of successful.

## Best Possible Solution

Carry explicit final-deliverable and required-completion-delivery state through ACP and subagent finalization, mapping missing final output or exhausted delivery to an existing blocked terminal outcome while preserving genuine final reports as succeeded.

## Reproduction Assessment

Yes, from source inspection and the sanitized incident evidence. A completed legacy ACP stream with only progress text, or a required subagent completion announcement that exhausts retries after ok finalization, follows current code toward successful task state unless explicit blocked or failed metadata is supplied.

## Solution Assessment

Yes, the requested direction is the right fix if implemented as structured final-output and delivery-state handling rather than broad last-text guessing. The repair should reuse blocked terminal outcomes and add regression coverage for missing final output, failed required delivery, and valid final reports.
