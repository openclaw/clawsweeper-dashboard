## Label Justifications

- `P1`: A real ACP/Codex child-task workflow can show success even when the parent or channel receives no final deliverable.
- `impact:message-loss`: The core symptom is a missing completion message or final artifact being hidden behind successful task state.
- `impact:session-state`: Durable task, subagent, flow, and parent cron state can diverge from the child session's actual completion contract.

## Summary

Keep open. Current main has partial ACP startTurn-result handling, but the legacy ACP path and subagent finalization path can still turn progress-only text into succeeded durable task state, and failed required completion delivery remains separate from the terminal task outcome.

## What This Changes

The issue asks OpenClaw to stop marking ACP/Codex child tasks successful when only interim progress text or failed required completion delivery is captured.

## Best Possible Solution

Carry an explicit final-deliverable and required-completion-delivery contract through ACP and subagent task finalization, mapping missing required output or exhausted required delivery to the existing blocked terminal outcome while preserving real final reports as succeeded.

## Reproduction Assessment

Yes from source inspection and the incident-shaped issue evidence, though not from an executed replay in this read-only review. A focused repro can simulate legacy ACP output text followed by done, and a subagent ok outcome with expectsCompletionMessage=true whose final delivery exhausts retries.

## Solution Assessment

Yes, the requested direction is the maintainable fix if it uses structured completion and delivery state instead of guessing from the last assistant text. Reusing terminalOutcome=blocked and existing task-flow behavior is narrower than adding a new public status or config option.
