## Label Justifications

- `P1`: The issue affects active agent/channel workflows by presenting missing child deliverables as successful task completion.
- `impact:message-loss`: The reported failure drops or fails to deliver the child completion while the ledger still looks successful.
- `impact:session-state`: Durable task and subagent state can drift from the real completion contract and mislead parent sessions and UI.

## Summary

Keep open: current main has related ACP/acpx timeout protections, but the new 2026.5.18 incident and source inspection show a remaining ok-result plus delivery-failed path where task status can stay `succeeded` with only progress-style captured text.

## What This Changes

The issue reports that ACP/Codex child sessions with only progress-style output and failed completion announcement are recorded as succeeded in durable task state.

## Best Possible Solution

Capture a redacted current Codex/acpx startTurn/subagent event stream for one failing run, then make the adapter/runtime expose final-deliverable versus interim output and propagate failed completion delivery into an honest blocked or failed task state.

## Reproduction Assessment

No high-confidence live reproduction was run in this checkout. The issue supplies concrete 2026.5.18 live run evidence, and current source shows an ok/completed ACP or subagent outcome can still finalize as succeeded while delivery failure remains only in `deliveryStatus`.

## Solution Assessment

Unclear: the installed hotfix’s progress-only heuristic reduces danger but may be too broad for legitimate text-only completions. The safer fix is an adapter/runtime completion contract plus task-state propagation for failed final delivery.
