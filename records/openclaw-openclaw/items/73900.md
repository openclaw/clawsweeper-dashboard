## Summary

Keep open. Current main still has the reported cron/systemEvent routing path: cron forces `heartbeat.target="last"`, the gateway wrapper shallow-merges that over configured heartbeat defaults, and outbound heartbeat delivery still treats inherited `heartbeat.to` as an explicit destination before falling back to the bound session route. The issue is also paired with open PR #73968, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

A `sessionTarget: "main"` cron `systemEvent` bound to a Telegram forum topic should route heartbeat reminder replies to the bound session/topic instead of inheriting global `agents.defaults.heartbeat.to` and delivering to a DM.

## Best Possible Solution

Land #73968 or an equivalent narrow fix that clears inherited explicit destination fields for cron-forced `target: "last"` main-session `systemEvent` wakes, with regression coverage proving a topic-bound session stays on its group/topic route when global heartbeat defaults include a DM `to`.

## Reproduction Assessment

Yes. The issue provides concrete config, cron payload, session key, and logs, and the current-main source path statically reproduces the merge and explicit-destination precedence without needing a live Telegram run.

## Solution Assessment

Yes. The narrowest maintainable fix is to clear or ignore inherited explicit destination fields only for cron-forced `target: "last"` main-session systemEvent wakes; changing general heartbeat `target: "last"` semantics would be broader and riskier.
