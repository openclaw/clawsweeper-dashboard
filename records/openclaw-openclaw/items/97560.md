## Summary

Keep open: this PR is a credible candidate for the linked unsafe-local data-loss bug, but it still needs real behavior proof and it silently changes configured-file cleanup by skipping all unsafe-local pruning whenever any configured source is unreadable.

## What This Changes

The PR changes memory-wiki unsafe-local sync to detect unreadable configured paths or nested directories, skip imported-source pruning during those outages, and add regression coverage for transient outages plus readable-directory deletion.

## Best Possible Solution

Track availability per configured unsafe-local source scope, preserve entries for unreadable scopes, keep pruning for successfully scanned scopes and intentional deletions, and require redacted real behavior proof before merge.

## Solution Assessment

No. The direction is right, but the current implementation is broader than the best fix because the global unreadable-source guard should be scoped so existing configured-file and available-source pruning behavior is preserved or explicitly approved.
