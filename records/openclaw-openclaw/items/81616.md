## Summary

Keep open: current main still contains a reset-to-MCP-runtime-dispose path and a stdio close path that force-kills the process tree on timeout, matching the reported Windows sidecar termination. The fix should be reviewed by ACP/MCP lifecycle owners because it must preserve prior leak cleanup while avoiding teardown of a reusable OpenCode/vlocal sidecar.

## What This Changes

The issue requests that OpenCode/vlocal sidecar processes survive OpenClaw session completion or reset so later sessions work without restarting the desktop app.

## Best Possible Solution

Adjust the ACP/MCP reset lifecycle so a reusable externally managed OpenCode/vlocal sidecar is not killed on ordinary session rollover, while session-owned MCP workers are still retired and covered by Windows regression tests.

## Reproduction Assessment

No live reproduction yet: I did not run OpenCode Desktop on Windows. The reporter supplied concrete steps and logs, and current main contains the same reset -> MCP runtime dispose -> stdio process-tree kill path, so the failure is source-reproducible with medium confidence.

## Solution Assessment

Unclear: simply removing process-tree cleanup is not the best fix because current tests and related lifecycle work protect against MCP worker leaks. The safer direction is a scoped lifecycle change that preserves cleanup for session-owned workers while keeping the reusable sidecar alive.
