## Summary

Keep open. Current main still has the reported source path: a session reset closes ACP/bundle-MCP runtime handles, stdio MCP transport escalates to `killProcessTree` after a 2s close wait, and Windows `killProcessTree` uses `taskkill /T /PID`, which matches the reporter's vlocal sidecar termination evidence.

## What This Changes

The issue asks OpenClaw to stop killing the vlocal/OpenCode sidecar on session completion or reset so later sessions can reconnect without restarting the app.

## Best Possible Solution

Add an ownership-aware lifecycle path so reusable OpenCode/vlocal sidecars survive session reset while OpenClaw-owned stdio MCP child trees are still cleaned up reliably.

## Reproduction Assessment

No, not as a live current-main reproduction. The issue supplies detailed Windows logs and current source matches the kill path, but I did not run OpenCode Desktop plus OpenClaw on Windows to verify the end-to-end failure.

## Solution Assessment

Unclear. The desired behavior is valid, but changing generic `killProcessTree` or reset-time runtime disposal directly risks reopening MCP process leaks; the safer solution is an explicit ownership/persistence distinction for reusable sidecars.
