## Summary

Keep open: current main still relies on Commander's private `rawArgs` field, so the cleanup is relevant, but this PR is not merge-ready because head CI shows both type-check failures and lazy CLI smoke-test failures introduced by the new argv storage path.

## What This Changes

The branch stores CLI parse argv in program context, resolves it through command ancestors during lazy reparse, passes rewritten argv into `buildProgram`, and adds focused program-context/reparse tests.

## Best Possible Solution

Keep the OpenClaw-owned argv direction, but make stored argv reflect the actual parse invocation, keep argv types compatible with the registry, then rerun the CLI/type gates and add redacted real CLI terminal proof.

## Reproduction Assessment

Yes for the PR defects: GitHub check annotations on the PR head show the readonly argv type error and lazy CLI smoke failures in `program.smoke.test.ts`. I did not run tests locally because this was a read-only review.

## Solution Assessment

No, not yet: owning argv outside Commander is a maintainable direction, but this implementation stores construction-time argv and introduces a readonly-to-mutable type mismatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Store argv from the actual parse call:** `src/cli/program/build-program.ts:22`
  - body: This stores raw argv when `buildProgram` is constructed, but existing callers and tests build the program and later call `program.parseAsync([...], { from: 'user' })`. On this head, lazy reparsing then reads stale construction argv instead of the argv being parsed, and `checks-node-agentic-cli` fails `program.smoke.test.ts` with Commander help exits from `action-reparse`. Capture/update the stored argv for each parse invocation, or keep reparse tied to the argv Commander actually parsed.
  - confidence: 0.93
- **[P2] Keep argv mutability consistent with the registry:** `src/cli/program/build-program.ts:9`
  - body: `buildProgram` now accepts `readonly string[]`, but it still passes that value to `registerProgramCommands`, whose argv parameter is mutable `string[]`. The PR head's `check-prod-types` and `check-test-types` annotations both report this readonly-to-mutable mismatch, so either keep the parameter mutable or copy/narrow before handing it to the registry.
  - confidence: 0.96
