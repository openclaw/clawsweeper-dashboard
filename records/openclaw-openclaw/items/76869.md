## Summary

Keep open: current main still parses Claude CLI stream-json text deltas and emits them as assistant agent events, but the CLI reply-runner path does not bridge those deltas into the partial-preview callback consumed by Telegram. The report is source-reproducible and the discussion also includes an independent live reproduction confirmation.

## What This Changes

Wire Claude CLI assistant deltas into Telegram live preview streaming instead of only delivering final text.

## Best Possible Solution

Add a generic shared reply-runner bridge from current-run assistant CLI events to the existing `onPartialReply` and typing path, with Telegram-focused regression coverage and no new configuration.

## Reproduction Assessment

Yes, source-reproducible: configure a Claude CLI JSONL backend with Telegram preview streaming and the CLI path emits assistant deltas only on the agent-event bus while Telegram listens to `onPartialReply`. I did not run the external Telegram and Claude CLI setup during this read-only review.

## Solution Assessment

Yes, the requested direction is the right boundary if implemented as a shared reply-runner bridge rather than a Telegram-specific CLI shortcut. The fix should reuse the existing partial-preview callback path and preserve final event semantics.

Current main parses Claude `stream-json` deltas and emits `stream: "assistant"` agent events from the CLI runner, but the auto-reply CLI execution path does not bridge those deltas into the existing `onPartialReply`/typing preview callback that Telegram consumes.
