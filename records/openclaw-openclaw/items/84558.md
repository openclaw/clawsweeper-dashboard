## Best Possible Solution

Keep the formatter-owned option, but include or sequence the narrow sanitizer allowlist for the literal `expandable` attribute, then prove the exact head with focused formatter tests and redacted Telegram output.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the generated expandable tag is passed into a sanitizer that still rejects blockquote attributes, so the failure is reproducible from the current code path without running the test suite.

## Solution Assessment

No as written. The Telegram formatter boundary is reasonable, but the branch must preserve the literal `expandable` attribute through the sanitizer or apply the expansion after a safe sanitizer step.
