## Summary

Keep this PR open, but it is not merge-ready: the formatter change creates `<blockquote expandable>` and then still sends it through the Telegram HTML sanitizer path that rejects attributes on `blockquote`, so the advertised output would be escaped instead of sent as native Telegram HTML.

## What This Changes

The PR adds an opt-in Telegram formatter/chunker option to render trailing markdown blockquotes as Telegram-native expandable blockquotes, plus formatter tests.

## Best Possible Solution

Land generated expandable-final-blockquote support together with the exact Telegram sanitizer allowance for `blockquote expandable`, focused markdown/chunking/raw-HTML tests, and fresh real Telegram proof from the corrected head.

## Reproduction Assessment

Yes at source level. The reviewed diff creates `<blockquote expandable>` before calling the existing sanitizer, while current main only allows attribute-free `blockquote` tags, so the new expected output is rejected without needing a live Telegram run.

## Solution Assessment

No as written. The Telegram formatter boundary is a reasonable place for generated expandable blockquotes, but the implementation must also align the sanitizer contract or the generated tag will be escaped before delivery.
