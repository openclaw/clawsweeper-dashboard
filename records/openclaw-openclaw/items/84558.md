## Summary

Keep this PR open, but it is not merge-ready: the final head emits Telegram’s expandable blockquote tag and then still runs it through a sanitizer path that rejects attributes on `blockquote`, so the claimed formatter output is not preserved. The PR also widens the shared MarkdownIR style union for a Telegram-only rendering choice, which needs either a Telegram-local implementation or explicit SDK contract work.

## What This Changes

The PR adds an opt-in Telegram formatter option that promotes a trailing Markdown blockquote to Telegram’s native `<blockquote expandable>` HTML and adds formatter/chunking tests.

## Best Possible Solution

Make expandable blockquotes a Telegram-owned formatter/sanitizer feature unless maintainers deliberately promote a shared MarkdownIR style with SDK docs and contract coverage.

## Reproduction Assessment

Yes for the review finding: the PR head renders `<blockquote expandable>` and immediately passes it into a sanitizer that still rejects attributes on `blockquote`. The new formatter tests assert unescaped output, so the failure path is clear from source inspection without running write-producing test commands.

## Solution Assessment

No as-is; the generated formatter direction is useful, but the final head must preserve the tag through Telegram sanitization and should not widen shared MarkdownIR unless that public contract is intentional.
