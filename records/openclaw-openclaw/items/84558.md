## Summary

Keep open: the latest diff has a blocking sanitizer mismatch. It generates `<blockquote expandable>` before calling the existing Telegram HTML sanitizer, but that sanitizer still treats `blockquote` as a no-attribute simple tag, so the advertised formatter output would be escaped instead of sent as Telegram-native expandable HTML.

## What This Changes

The branch adds an `expandFinalBlockquote` Telegram markdown option, applies it to HTML and chunked rendering, and adds formatter tests for expandable final blockquotes.

## Best Possible Solution

Keep the generated formatter behavior inside the Telegram formatter/chunker boundary, but first add or coordinate the narrow sanitizer allowlist for `<blockquote expandable>` and refresh latest-head Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. The PR renders `<blockquote expandable>` and then passes it through a sanitizer that still rejects blockquote attributes, while Telegram's Bot API docs confirm the target tag itself is valid.

## Solution Assessment

No, not as implemented. The formatter boundary is a good place for generated expandable blockquotes, but the patch must also preserve that exact attribute through OpenClaw's Telegram HTML allowlist or coordinate with the related sanitizer PR.
