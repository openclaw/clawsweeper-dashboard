## Summary

Keep open: the opt-in generated formatter path is useful and the visible proof is adequate, but the current patch applies “final blockquote” expansion per rendered chunk, so an intermediate long blockquote can be wrongly upgraded.

## What This Changes

The PR adds an opt-in Telegram formatter option that renders the final generated markdown blockquote as `<blockquote expandable>` and adds formatter tests.

## Best Possible Solution

Keep the generated formatter option, but make chunked rendering decide expandability from the original message’s final blockquote range and add a regression test for long non-final blockquotes.

## Reproduction Assessment

Yes, by source inspection: chunking slices the MarkdownIR before rendering, and the PR checks `span.end === ir.text.length` on each sliced chunk. I did not run tests because this review is read-only, but the code path is concrete.

## Solution Assessment

No, not merge-ready as written. The generated formatter boundary is the right place for this option, but chunked rendering needs to preserve the original-message meaning of “final” before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep expansion tied to the original final blockquote:** `extensions/telegram/src/format.ts:782`
  - body: `renderTelegramChunksWithinHtmlLimit` renders every sliced chunk with the full `expandFinalBlockquote` option, and `hasFinalTelegramBlockquote` then evaluates against that chunk’s IR. A long non-final blockquote followed by normal text can therefore become `<blockquote expandable>` in an intermediate chunk even though the unchunked path only promotes the trailing blockquote. Compute the expandable range from the original IR, or pass per-chunk context so only the chunk containing the original final blockquote is upgraded.
  - confidence: 0.86
