## Summary

Keep open: the PR is a focused Telegram formatter improvement with useful proof, but the current patch has a blocking chunked-output bug because it decides whether a blockquote is final after slicing the Markdown IR into chunks.

## What This Changes

The PR adds an opt-in `expandFinalBlockquote` Telegram markdown formatter option and tests for normal and chunked expandable final blockquote output.

## Best Possible Solution

Keep this formatter-owned option, but compute expandability from the original document's true trailing blockquote and add coverage for long non-final blockquotes; leave raw HTML sanitizer passthrough to https://github.com/openclaw/openclaw/pull/84082.

## Reproduction Assessment

No live reproduction was run; the bug is source-reproducible from the PR diff. The chunk renderer receives sliced Markdown IR and then checks final-blockquote status against the slice length, not the original message length.

## Solution Assessment

No as written. The generated Telegram formatter boundary is the right place, but final-blockquote expansion must be decided from original document boundaries before or during chunking rather than independently for each chunk slice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep expansion tied to the original final blockquote:** `extensions/telegram/src/format.ts:779`
  - body: In the chunked path, the same options are passed to every sliced MarkdownIR. `hasFinalTelegramBlockquote` then treats any chunk whose blockquote span reaches that chunk's end as final, so a long quoted section followed by normal text can be emitted as `<blockquote expandable>` even though it is not the final blockquote. Decide expandability from the original IR/document boundaries and add a regression with a long non-final blockquote followed by non-quote text.
  - confidence: 0.88
