## Summary

Keep open: this is a focused fix for a real isolated-cron media workflow bug, but the patch still needs a small correctness repair before merge. The new detach predicate makes run-scoped cron media execute inline, but it also skips the only path that caches the media request key for recent duplicate suppression, and the PR still has mock-only real behavior proof.

## What This Changes

The PR changes image, video, and music generation so run-scoped cron sessions execute media generation inline while non-run cron and interactive sessions still detach to background completion.

## Best Possible Solution

Keep the inline cron media behavior, but record recent media task starts for inline and detached task handles, add inline retry regression coverage, and include redacted real cron/provider proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the introduced review finding: run-scoped cron sessions now return false from shouldDetachMediaGenerationTask, and the tools only record the duplicate-guard requestKey in the detach branch. I did not run a live provider cron job in this read-only review.

## Solution Assessment

No, not quite: keeping cron media inline is the right repair direction, but the implementation should preserve duplicate guarding independently of whether completion is detached. The narrow fix is to keep task-start/requestKey recording for any created task handle and only gate background scheduling on the detach policy.
