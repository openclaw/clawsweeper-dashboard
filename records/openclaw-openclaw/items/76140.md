## Summary

Keep open. This PR targets a valid memory-core isolation bug, but the latest patch still has blocking compile/contract problems, incomplete ranking isolation, filesystem safety regressions, and no after-fix real shared-workspace proof.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests for shared workspaces.

## Best Possible Solution

Keep this PR as the likely fix path, but preserve the exported workspace helper contract, implement rank-level agent filtering for every caller, keep new hidden stores on fs-safe/private-file-store primitives, and add real shared-workspace proof before merge.

## Reproduction Assessment

Yes by source inspection. Current main resolves all configured agents for a workspace and ranks/promotes memory entries without a current-agent discriminator, which matches the shared-workspace contamination report.

## Solution Assessment

No. The PR direction is appropriate, but the latest patch is not mergeable because the exported helper and ranking contracts are incomplete, REM preview is still not wired to the new scope, and new filesystem writes bypass existing safety helpers.

Full review comments:

- **[P1] Keep the workspace helper contract compiling:** `src/memory-host-sdk/dreaming.ts:608`
  - body: This removes the `options` parameter from `resolveMemoryDreamingWorkspaces`, but PR-head code still calls it with `{ primaryWorkspaceDir, primaryAgentId }` in `dreaming-phases.ts`, and existing callers such as `src/gateway/server-methods/doctor.ts` do the same. That breaks typechecking and also drops the runtime primary-workspace fallback the helper currently provides.
  - confidence: 0.94
- **[P1] Add the ranker isolation contract before passing it:** `extensions/memory-core/src/dreaming.ts:616-618`
  - body: The new managed dreaming call passes `currentAgentId` and `isShared`, but `RankShortTermPromotionOptions` still does not define those fields and `rankShortTermPromotionCandidates` never filters candidates by agent-scoped corpus path. As written this fails typechecking and, once typed, would still leave the promotion ranker unscoped unless the filter is implemented inside the ranker.
  - confidence: 0.93
- **[P2] Pass REM preview isolation from the CLI:** `extensions/memory-core/src/rem-harness.ts:190-192`
  - body: `previewRemHarness` only forwards isolation metadata when `params.isShared && params.currentAgentId`, but the CLI caller still invokes it without either value. After the ranker accepts these fields, `memory rem-harness` will continue ranking shared-workspace candidates across agents unless the caller resolves the workspace metadata and passes the current agent scope.
  - confidence: 0.86
- **[P2] Keep corpus appends regular-file safe:** `extensions/memory-core/src/dreaming-phases.ts:792-799`
  - body: The replacement realpath check only verifies the parent directory; `fs.appendFile` will still follow a symlink at the final corpus file path. The current code uses `appendRegularFile(... rejectSymlinkParents: true)`, so the per-agent path change should keep that helper rather than reopening a symlink write path.
  - confidence: 0.88
- **[P2] Write provenance through the private file store:** `extensions/memory-core/src/short-term-promotion.ts:1585-1602`
  - body: The new provenance sidecar is an internal JSON store under `memory/.dreams`, but it is created with bare `fs.mkdir`, `fs.readFile`, and `fs.writeFile`. That follows symlinked `.dreams` paths and bypasses the same private-file-store pattern already used for the short-term stores in this file; use `privateFileStore(workspaceDir)` or another fs-safe helper for the sidecar.
  - confidence: 0.82
