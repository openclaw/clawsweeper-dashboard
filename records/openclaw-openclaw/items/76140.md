## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent corpus/provenance handling, and isolation tests for the cross-agent dreaming bug.

## Best Possible Solution

Land one coherent agent-boundary contract across managed dreaming, manual promote/promote-explain, REM preview, SDK-compatible workspace metadata, safe corpus writes, and focused shared-workspace regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows two configured agents sharing one workspace are grouped into one dreaming workspace, then collectSessionIngestionBatches iterates all matched agentIds and appends them to one workspace corpus.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep corpus writes on the safe file helper:** `extensions/memory-core/src/dreaming-phases.ts:803`
  - body: `appendSessionCorpusLines` currently appends through `appendRegularFile({ rejectSymlinkParents: true })`, but this patch switches the corpus write to plain `fs.appendFile`. Since the corpus path is under a user-configured workspace, a symlinked `memory/.dreams/session-corpus` parent can redirect the write outside the workspace; keep the safe helper for both legacy and per-agent corpus paths.
  - confidence: 0.89
- **[P1] Do not fall back to the first agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619-620`
  - body: When `params.currentAgentId` is absent, this fallback passes the first configured agent into shared-workspace ranking. That bypasses the fail-closed contract because deep promotion can still rank and promote one agent's entries without a real hook identity; pass only the actual current agent and let shared ranking return no candidates when it is missing.
  - confidence: 0.92
- **[P1] Make promotion ranking receive agent scope at every caller:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new ranking filter is opt-in through `isShared` and `currentAgentId`, but `memory promote`, `promote-explain`, and REM preview still call this API without those options. Those user-facing paths can still rank shared-workspace entries broadly, and `memory promote --apply` can write another agent's candidate into `MEMORY.md`; derive the same workspace/agent boundary at those callers or require it in the rank API.
  - confidence: 0.9
- **[P2] Keep the exported workspace type compatible:** `src/memory-host-sdk/dreaming.ts:147-149`
  - body: `MemoryDreamingWorkspace` is exported through `openclaw/plugin-sdk/memory-core-host-status`, so making `shared` required breaks existing plugin code that constructs the prior `{ workspaceDir, agentIds }` shape. Make the field optional or keep a narrower internal type while still returning the metadata from the resolver.
  - confidence: 0.84
