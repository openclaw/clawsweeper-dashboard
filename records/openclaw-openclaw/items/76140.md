## Summary

Keep open. The PR targets a real shared-workspace dreaming isolation bug, but the current draft is not mergeable because durable promotion remains workspace-scoped and the added Vitest coverage uses APIs that will not run in this test setup.

## What This Changes

The PR adds a shared-workspace flag, threads `currentAgentId` through memory-core dreaming, writes per-agent session corpus files plus a provenance sidecar, and adds shared-workspace isolation tests.

## Best Possible Solution

Implement agent-scoped dreaming end to end: validate the active agent against workspace membership, scope session corpus queries and deep ranking/promotion by agent without dropping safe non-session signals, add a two-agent shared-workspace regression, and document the isolation contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is one gateway with multiple active agents sharing a workspace: current main groups those agents into one dreaming workspace and ingests each agent's session files into the same workspace-level corpus.

## Solution Assessment

No. The direction is appropriate, but this draft only partially scopes ingestion and provenance; durable ranking/promotion is still unscoped, and the new tests are not executable under the repo's Vitest setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope deep promotion before applying candidates:** `extensions/memory-core/src/dreaming.ts:623`
  - body: Passing `currentAgentId` into `applyShortTermPromotions` only records provenance after candidates have already been selected. `rankShortTermPromotionCandidates` above still reads the whole workspace short-term store, so another agent's entry can still be promoted into `MEMORY.md` in a shared workspace. Filter by agent-scoped sources before ranking/applying.
  - confidence: 0.92
- **[P1] Reject current agents outside the matched workspace:** `extensions/memory-core/src/dreaming-phases.ts:729-730`
  - body: For shared workspaces this returns `[currentAgentId]` without checking that the current agent belongs to `match.agentIds`. Because dreaming fans out across resolved workspaces, a run for agent A can process a different shared workspace for agents B/C and write A's sessions into that corpus. Fail closed unless the current agent is a member of the matched workspace.
  - confidence: 0.86
- **[P2] Preserve non-session recall inputs while filtering corpus entries:** `extensions/memory-core/src/dreaming-phases.ts:677-680`
  - body: In shared mode this filter keeps only `memory/.dreams/session-corpus/<agent>/...` paths. Light and REM still rely on daily memory notes and other short-term recall traces, so shared workspaces would drop legitimate `memory/YYYY-MM-DD.md` and non-session entries. Limit the agent filter to session-corpus entries or explicitly preserve safe non-corpus paths.
  - confidence: 0.88
- **[P2] Use Vitest imports instead of Jest/CommonJS helpers:** `extensions/memory-core/src/dreaming-phases.test.ts:2803-2804`
  - body: This test file runs under Vitest ESM and currently uses `vi`; the added block calls `jest.requireActual` and `require`, which are not defined here. Import `__testing` directly or use Vitest-compatible imports so the isolation tests can run.
  - confidence: 0.95
- **[P2] Replace `fs.pathExists` with an access helper:** `extensions/memory-core/src/dreaming-phases.test.ts:2860-2862`
  - body: `fs` in this file is imported from `node:fs/promises`, which does not provide `pathExists`. These assertions will throw before checking the corpus paths; use `fs.access` or a local helper instead.
  - confidence: 0.96
