## Summary

Keep open. The shared-workspace memory contamination bug is valid and this PR moves in the right direction, but the latest head is still draft, lacks after-fix real behavior proof, and source inspection shows remaining promotion paths that can still cross agent boundaries plus an SDK compatibility break.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with regression coverage and real behavior proof.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main groups agents by shared workspace, loops over every matched agent's sessions, writes one workspace-level corpus, and ranks without agent scope.

## Solution Assessment

No. The PR direction is right, but the latest patch still synthesizes an agent identity for shared ranking, leaves manual/REM ranking callers unscoped, and makes an exported SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When params.currentAgentId is absent, this fallback passes the first configured workspace agent into rankShortTermPromotionCandidates. In a shared workspace that defeats fail-closed isolation because deep promotion can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent identity and let shared ranking return no candidates.
  - confidence: 0.93
- **[P1] Scope all promotion ranking callers to the active agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new filter only runs when callers pass isShared/currentAgentId, but memory promote, promote-explain, and REM preview still call this ranker without those options. In a shared workspace those paths can still rank another agent's corpus entries, and memory promote --apply can write them into MEMORY.md; derive the same boundary at those callers or make the rank API require it.
  - confidence: 0.9
- **[P2] Keep MemoryDreamingWorkspace source-compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: MemoryDreamingWorkspace is exported through openclaw/plugin-sdk/memory-core-host-status, so making shared required breaks plugin code that constructs the existing { workspaceDir, agentIds } shape. Make the field optional or use a narrower internal type while still returning shared from resolveMemoryDreamingWorkspaces.
  - confidence: 0.84
