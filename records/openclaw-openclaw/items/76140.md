## Summary

Keep this PR open: it targets a real memory-core shared-workspace isolation bug, but the draft/conflicting PR is not mergeable because source inspection still shows type-contract breaks, incomplete ranker isolation, REM preview gaps, filesystem-safety regressions, and missing real behavior proof.

## What This Changes

The PR changes memory-core dreaming to mark shared workspaces, thread currentAgentId, write per-agent corpus/provenance data, scope promotion callers, and add isolation/audit tests.

## Best Possible Solution

Keep this PR as a possible fix path, but preserve the workspace helper contract or migrate all callers, implement ranker-level agent filtering for every promotion/preview path, keep hidden memory stores on fs-safe/private-store primitives, and require redacted real shared-workspace proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main resolves every configured agent for a shared workspace and ranks/promotes memory entries without a current-agent discriminator, matching the linked shared-workspace contamination report.

## Solution Assessment

No. The PR direction is appropriate, but the current patch is not the best mergeable solution because it breaks the exported helper contract, passes ranker scope fields without implementing the ranker contract, leaves REM preview unscoped, and reopens filesystem-safety concerns.

  - body: This removes the options parameter from resolveMemoryDreamingWorkspaces, but PR-head code still calls it with primaryWorkspaceDir/primaryAgentId in dreaming-phases, gateway doctor, and an SDK test.   - confidence: 0.95

  - body: The new managed dreaming call passes currentAgentId and isShared into rankShortTermPromotionCandidates, but RankShortTermPromotionOptions still does not define those fields and the ranker does not filter candidates by agent-scoped corpus path. This is both a type error and an incomplete isolation fix.
  - confidence: 0.94
- **[P2] Forward REM preview scope from the CLI:** `extensions/memory-core/src/rem-harness.ts:190-192`
  - body: previewRemHarness only forwards isolation metadata when params.isShared and params.currentAgentId are present, but the CLI caller still invokes it without either value. After the ranker accepts these fields, memory rem-harness will continue ranking shared-workspace candidates across agents.
  - confidence: 0.88
- **[P2] Keep corpus appends on the regular-file helper:** `extensions/memory-core/src/dreaming-phases.ts:799`
  - body: The replacement write checks only the parent directory realpath before calling fs.appendFile, so a symlink at the final corpus file path can still redirect the append. Current main uses appendRegularFile with rejectSymlinkParents here; the per-agent path change should keep that helper.
  - confidence: 0.9
- **[P2] Write provenance through the private store:** `extensions/memory-core/src/short-term-promotion.ts:1602`
  - body: The new provenance sidecar is an internal JSON store under memory/.dreams, but it is created and updated with bare fs mkdir/readFile/writeFile calls. Existing memory-core stores in this file use privateFileStore, so this sidecar should use the same safe store path instead of following symlinked internal paths.
  - confidence: 0.85
- **[P2] Do not reject configured default agents:** `extensions/memory-core/src/dreaming.ts:930-940`
  - body: This guard checks only agents.list, but current configs can use a default primary agent/workspace alongside listed subagents. In that shape ctx.agentId can be the default agent while agents.list contains only subagents, so the PR would skip dreaming for a valid primary agent before it reaches the isolation logic.
  - confidence: 0.82
