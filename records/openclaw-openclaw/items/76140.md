## Summary

Keep open. The linked memory isolation bug is real, but the latest PR head still leaves shared-workspace ranking paths open, breaks a public SDK type shape, lacks real behavior proof, and is currently draft/conflicting.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent corpus/provenance handling, and isolation tests for cross-agent dreaming contamination.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, then validate it with regression tests and a real shared-workspace proof artifact.

## Reproduction Assessment

Yes. Source inspection on current main shows shared workspace sessions and promotion ranking are reachable without an active-agent discriminator; the PR head still exposes the same class of path in managed ranking plus manual promotion/REM preview.

## Solution Assessment

No. The PR direction is appropriate, but the current patch is not the best final fix because it synthesizes identity, leaves user-facing ranking callers unscoped, and makes an exported SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When `params.currentAgentId` is absent, this fallback ranks the shared workspace as the first configured agent instead of failing closed. A shared dreaming run without a real hook identity can still rank and promote one agent's entries, so pass only the real current agent identity and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The rank filter only runs when callers pass `isShared` and `currentAgentId`, but `memory promote`, `promote-explain`, and REM preview still omit those options. In a shared workspace, `memory promote --apply` can still promote another agent's candidate into `MEMORY.md`; derive the same agent boundary at these callers or make ranking require it.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: `MemoryDreamingWorkspace` is exported through the plugin SDK, so making `shared` required breaks plugin code that constructs the existing `{ workspaceDir, agentIds }` shape. Make the field optional or use a narrower internal type while still returning `shared` from `resolveMemoryDreamingWorkspaces`.
  - confidence: 0.84
