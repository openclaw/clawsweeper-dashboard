## Summary

Keep open. The linked shared-workspace dreaming bug is real on current main, but this draft PR is not correct yet: promotion still ranks from the workspace-wide recall store, the new fail-closed path can still read existing shared entries, shared detection is applied across unrelated workspaces, and the added Vitest block uses Jest/CommonJS helpers.

## What This Changes

The PR adds a shared-workspace flag, threads currentAgentId through memory-core dreaming, writes shared workspace session corpus under per-agent paths, filters recall entries by agent path, records promotion provenance, and adds isolation tests.

## Best Possible Solution

Ship an agent-scoped dreaming fix that scopes corpus writes, recall reads, ranking, promotion, and dream narrative generation to the active agent for the matched workspace, with real two-agent shared-workspace regression tests and completed security review.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two configured agents sharing one workspace with separate session transcripts; current main groups them into one dreaming workspace and ingests both agents through the same corpus and recall path.

## Solution Assessment

No. The direction is appropriate, but this PR does not yet scope durable promotion, does not fail closed for existing recall entries without identity, and has blocking test harness issues; the safer fix is to filter by the matched workspace and current agent before any ranking or narrative step.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter candidates before promotion:** `extensions/memory-core/src/dreaming.ts:623`
  - body: `currentAgentId` is only passed into `applyShortTermPromotions`, but `candidates` has already been produced by `rankShortTermPromotionCandidates({ workspaceDir, ... })`, which reads the workspace-wide short-term recall store. In a shared workspace, another agent's `memory/.dreams/session-corpus/<agent>/...` entries can still be ranked and appended to the current workspace `MEMORY.md`, now with the current agent recorded as provenance. Scope ranking/selection to the current agent before applying promotions.
  - confidence: 0.91
- **[P2] Fail closed when shared identity is missing:** `extensions/memory-core/src/dreaming-phases.ts:672-674`
  - body: For `isShared: true` with no `currentAgentId`, this helper returns all recall entries. Session ingestion may skip new files, but old shared or contaminated entries already in the short-term store can still feed light/REM dreaming. Shared workspaces without an agent identity should return no entries or skip the phase.
  - confidence: 0.88
- **[P2] Match shared status to the active workspace:** `extensions/memory-core/src/dreaming-phases.ts:1651`
  - body: `workspaceIsShared` is true if any resolved workspace is shared, not if `params.workspaceDir` is shared. With one shared workspace plus one isolated workspace, the isolated workspace will be filtered as shared and lose normal memory entries unless they live under `session-corpus/${currentAgentId}/`. Match the resolved entry by workspace path before enabling agent-path filtering, and apply the same fix to the REM path.
  - confidence: 0.89
- **[P2] Use Vitest APIs in the new tests:** `extensions/memory-core/src/dreaming-phases.test.ts:2803-2804`
  - body: This file is an ESM Vitest test and imports `vi`, not a Jest global, but the new block calls `jest.requireActual` and later uses CommonJS `require`. It will fail before exercising the isolation assertions; use the existing `__testing` import or `vi.importActual`, and replace `fs.pathExists` with a Node fs/promises-compatible check.
  - confidence: 0.97
