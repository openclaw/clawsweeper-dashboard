## Summary

Keep open. The PR addresses a real memory-core shared-workspace isolation bug, but the latest head still leaves merge-blocking agent-boundary gaps, breaks an exported SDK shape, and lacks after-fix real behavior proof from a real shared-workspace dreaming run.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with focused regression coverage and real shared-workspace proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: two configured agents sharing one workspace are resolved together, their session files are ingested into one corpus path, and promotion ranking has no active-agent discriminator.

## Solution Assessment

No. The PR direction is right, but this patch still synthesizes an agent identity for shared ranking, leaves manual/REM ranking callers unscoped, and makes an exported SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When currentAgentId is absent, this fallback passes the first configured agent into shared-workspace ranking. That defeats fail-closed isolation because deep promotion can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the active agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass isShared/currentAgentId, but memory promote, promote-explain, and rem-harness still omit those options. Those user-facing paths can still rank another agent's shared-workspace entries and memory promote --apply can write them into MEMORY.md.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: MemoryDreamingWorkspace is exported through openclaw/plugin-sdk/memory-core-host-status, so making shared required breaks plugin code that constructs the existing { workspaceDir, agentIds } shape. Make the field optional or keep the required flag in a narrower internal type.
  - confidence: 0.84
