## Summary

Keep open. The linked contamination bug is valid and the PR moves in the right direction, but current head still leaves shared-workspace promotion paths that can bypass the intended fail-closed boundary and also breaks a public SDK type.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace/per-agent corpus/provenance handling, and adds isolation tests plus an audit note for the memory contamination bug.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: shared workspace agents are resolved together, ingested into one corpus, and ranked without an active-agent discriminator; this read-only pass did not run a live failing test.

## Solution Assessment

No. The PR direction is right, but the patch should remove fallback identity synthesis, make shared-workspace ranking scope mandatory for all user-facing promotion paths, and preserve the public SDK type contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop falling back to the first agent during shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When `params.currentAgentId` is absent, this fallback passes the first configured agent into `rankShortTermPromotionCandidates`. In a shared workspace that bypasses the fail-closed contract: ingestion skips, but deep promotion can still rank and promote that first agent's existing corpus entries without a real hook identity. Pass only the real current agent and let shared ranking return no candidates.
  - confidence: 0.91
- **[P1] Make shared-workspace promotion scoping mandatory:** `extensions/memory-core/src/short-term-promotion.ts:1248`
  - body: The new rank filter only runs when callers opt in with `isShared/currentAgentId`, but `memory promote`, `promote-explain`, and the REM harness are not updated. Those user-facing paths can still rank cross-agent shared corpus entries and `memory promote --apply` can promote the wrong agent's memory, so derive the same boundary at those callers or require it in the rank API.
  - confidence: 0.88
- **[P2] Keep MemoryDreamingWorkspace source compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: `MemoryDreamingWorkspace` is exported through `openclaw/plugin-sdk/memory-core-host-status`; making `shared` required breaks plugin/test code that constructs the previous `{ workspaceDir, agentIds }` shape. Make the new field optional or use an internal narrowed type while still returning `shared` from the resolver.
  - confidence: 0.84
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:498`
  - body: This is a user-facing `fix(memory)` behavior change, but the PR does not update `CHANGELOG.md`. Add one single-line active `### Fixes` entry before merge, without inventing a bot or maintainer thanks handle.
  - confidence: 0.82
