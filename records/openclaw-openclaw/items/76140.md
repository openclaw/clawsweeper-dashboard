## Summary

Keep open; the linked bug is valid and the PR is headed in the right direction, but the latest patch still leaves shared-workspace promotion paths unisolated and breaks an exported SDK type.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests for Bug #65374.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, as a source reproduction. Current main groups all agents sharing a workspace, writes ingested sessions into one session corpus path, and ranks promotion candidates without an active-agent discriminator.

## Solution Assessment

No. The PR direction is maintainable, but the current patch still synthesizes an identity for shared ranking, leaves user-facing manual ranking paths unscoped, and makes an exported SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619-620`
  - body: When params.currentAgentId is absent, this fallback passes the first configured agent into rankShortTermPromotionCandidates. In a shared workspace that defeats fail-closed isolation because deep promotion can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent identity and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass isShared and currentAgentId, but memory promote, promote-explain, and rem-harness still call this path without those options. Those user-facing paths can still rank another agent's shared-workspace entries and memory promote --apply can write them into MEMORY.md, so derive the same workspace/agent boundary at those callers or make the rank API require it.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: MemoryDreamingWorkspace is exported through openclaw/plugin-sdk/memory-core-host-status, so making shared required breaks plugin code that constructs the existing { workspaceDir, agentIds } shape. Make the new field optional or use a narrower internal type while still returning shared from resolveMemoryDreamingWorkspaces.
  - confidence: 0.84
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:619`
  - body: This is a user-facing fix(memory) behavior change, but the PR does not update CHANGELOG.md. Add a single-line entry under the active fixes section before merge.
  - confidence: 0.82
