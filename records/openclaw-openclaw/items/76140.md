## Summary

Keep open. The linked cross-agent dreaming bug remains valid on current main, and this draft PR is aimed at the right memory-core boundary, but the patch still has blocking isolation, recall-contract, and test-execution defects before it is mergeable.

## What This Changes

The PR adds shared-workspace metadata, threads agent/session identity through memory-core dreaming, writes agent-scoped session corpus and provenance data, and adds isolation tests.

## Best Possible Solution

Implement a validated agent-scoped dreaming contract end to end: match the active agent to the workspace being swept, fail closed without that match, deliberately handle legacy pooled corpus entries, update recall path matching and rehydration for agent-scoped corpus files, and prove it with executable two-agent shared-workspace tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure two agents with the same workspace, let dreaming ingest sessions, and the resolver returns both agent IDs into one unscoped session corpus and short-term recall store.

## Solution Assessment

No. The PR direction is plausible, but this implementation is not yet the best fix because agent-scoped corpus paths are not accepted by the recall store, missing or mismatched agent identity still leaves fail-open paths, and the new tests will not run under the repo's Vitest setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Accept agent-scoped corpus paths before recording recalls:** `extensions/memory-core/src/dreaming-phases.ts:753-760`
  - body: When `isShared` is true this writes and returns `memory/.dreams/session-corpus/<agent>/<day>.txt`, but `recordShortTermRecalls` still filters through `isShortTermMemoryPath`, whose session-corpus regex only accepts `<day>.txt` directly under `session-corpus`. The isolated corpus lines are appended but never enter the short-term recall store, so later dreaming phases cannot use them.
  - confidence: 0.95
- **[P1] Fail closed shared recall reads without an agent id:** `extensions/memory-core/src/dreaming-phases.ts:672-674`
  - body: In shared mode with no `currentAgentId`, this helper returns every stored recall entry. That preserves the contamination path for cron, hook, or test callers that can reach the read path without an active agent identity, even though ingestion now fail-closes.
  - confidence: 0.92
- **[P2] Validate the shared-workspace agent id before ingesting:** `extensions/memory-core/src/dreaming-phases.ts:729-730`
  - body: For shared workspaces this trusts any non-empty `currentAgentId` and returns it without checking `match.agentIds`. A stale or mismatched identity can make the sweep for one configured workspace ingest another agent's sessions into that workspace instead of failing closed.
  - confidence: 0.86
- **[P2] Scope shared-workspace filtering to the active workspace:** `extensions/memory-core/src/dreaming-phases.ts:1647-1651`
  - body: This computes `workspaceIsShared` with `.some(...)` across all configured workspaces rather than the workspace being swept. If any workspace is shared, isolated workspaces filter their normal recall and daily entries as shared and drop non-agent-scoped paths.
  - confidence: 0.84
- **[P2] Use Vitest-compatible module access in tests:** `extensions/memory-core/src/dreaming-phases.test.ts:2803-2804`
  - body: This test file imports from Vitest and runs as ESM, but the added block calls `jest.requireActual` and later CommonJS `require`. The test process will throw before exercising the isolation assertions.
  - confidence: 0.94
- **[P2] Use fs.promises-compatible existence checks:** `extensions/memory-core/src/dreaming-phases.test.ts:2860-2862`
  - body: `fs` is imported from `node:fs/promises`, which has no `pathExists` method. These assertions will fail at runtime; use `fs.access`, `stat`, or an existing helper.
  - confidence: 0.94
- **[P3] Add the memory fix to the changelog:** `extensions/memory-core/src/dreaming.ts:924-928`
  - body: This is a user-facing `fix(memory)` behavior and isolation change, but the PR file list has no `CHANGELOG.md` entry. OpenClaw policy requires a changelog note for user-facing fixes before merge.
  - confidence: 0.82
