## Summary

Keep open. The underlying cross-agent dreaming bug is valid and this PR is pointed at the right area, but the latest head still leaves shared-workspace promotion paths that can rank or promote with the wrong agent boundary.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests for the memory dreaming contamination bug.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: shared-workspace agents are resolved together, written into one session corpus, and ranked without an active-agent discriminator.

## Solution Assessment

No. The PR direction is right, but the latest patch still synthesizes an agent when identity is missing, leaves manual ranking callers unscoped, and changes an exported SDK type incompatibly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When `params.currentAgentId` is absent, this fallback passes the first configured agent into `rankShortTermPromotionCandidates`. In a shared workspace that bypasses the fail-closed contract because the sweep can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent identity and let shared ranking return no candidates.
  - confidence: 0.91
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass `isShared` and `currentAgentId`, but `memory promote`, `promote-explain`, and REM preview still call this path without those options. Those user-facing paths can still rank another agent's shared-workspace entries and promote them with `--apply`, so derive the same workspace/agent boundary at those callers or make the boundary mandatory in the rank API.
  - confidence: 0.88
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: `MemoryDreamingWorkspace` is exported through `openclaw/plugin-sdk/memory-core-host-status`, so making `shared` required breaks plugin code that constructs the existing `{ workspaceDir, agentIds }` shape. Make the new field optional or use a narrower internal type while still returning `shared` from `resolveMemoryDreamingWorkspaces`.
  - confidence: 0.84
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:523`
  - body: This is a user-facing `fix(memory)` behavior change, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active fixes section before merge.
  - confidence: 0.82
