## Summary

Keep open: this PR targets a real shared-workspace memory dreaming isolation bug, but the latest branch is still draft/conflicting, lacks required real shared-workspace proof, and has blocking correctness/security issues in the updated patch.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent corpus/provenance handling, and isolation tests for the cross-agent dreaming contamination report.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, light/REM ingestion, manual promote/promote-explain, REM preview/doctor, primary-workspace fallback, and fs-safe corpus writes, with regression tests plus real shared-workspace proof.

## Reproduction Assessment

Yes by source inspection, though not by a live run. Current main groups configured agents by shared workspace and ranks/promotes recall entries without an active-agent discriminator, and the linked issue discussion narrows the bug to single-gateway shared-workspace setups.

## Solution Assessment

No. The direction is appropriate, but the latest patch is not the best final fix because it drops primary-workspace fallback, passes isolation options that the ranking contract does not accept or use, leaves REM preview callers unthreaded, and reopens fs-safe append risk.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restore the primary workspace option:** `src/memory-host-sdk/dreaming.ts:608`
  - body: Dropping the resolver options makes existing calls to `resolveMemoryDreamingWorkspaces(cfg, { primaryWorkspaceDir })` silently ignore the active runtime workspace. The existing test that expects `/workspace/main` to be included would fail, and doctor/dreaming paths can omit the current workspace when it is not listed under `agents.list`.
  - confidence: 0.93
- **[P1] Make ranking isolation real:** `extensions/memory-core/src/dreaming.ts:616-618`
  - body: The new calls pass `currentAgentId` and `isShared`, but `RankShortTermPromotionOptions` still does not declare those fields and the ranking loop never reads them. These object literals should fail strict TypeScript, and if the type were widened without filtering, shared promotion ranking would remain cross-agent.
  - confidence: 0.95
- **[P2] Thread identity into REM preview callers:** `extensions/memory-core/src/rem-harness.ts:190-192`
  - body: `previewRemHarness` only scopes ranking when both new params are supplied, but the CLI and gateway doctor callers still invoke it without `currentAgentId` or `isShared`. Shared-workspace REM preview can therefore still show candidates from other agents.
  - confidence: 0.88
- **[P1] Keep corpus appends on fs-safe:** `extensions/memory-core/src/dreaming-phases.ts:792-799`
  - body: The parent `realpath` check does not reject a symlinked target file and has a check/use gap before `fs.appendFile`, so corpus writes can still be redirected after this change. Keep the existing fs-safe append behavior when adding the per-agent subdirectory.
  - confidence: 0.9
