## Summary

Keep open. The related bug still reproduces on current main, and this PR is a useful implementation candidate, but the latest branch still has merge-blocking isolation gaps plus a public SDK compatibility break and missing changelog entry.

## What This Changes

This PR threads agent identity into memory-core dreaming, adds shared-workspace metadata, writes per-agent session corpus/provenance, and adds isolation/audit coverage for cross-agent dreaming contamination.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK-compatible workspace metadata, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows two configured agents sharing one workspace are resolved together, ingested into one workspace-level session corpus, and ranked from a workspace-level recall store without an active-agent discriminator.

## Solution Assessment

No. The PR direction is right, but the current patch still synthesizes an agent when identity is missing, leaves manual ranking paths unscoped, and makes an exported SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When `params.currentAgentId` is absent, this fallback passes the first configured agent into `rankShortTermPromotionCandidates`. In a shared workspace that bypasses fail-closed isolation because the sweep can still rank and promote that agent's entries without a real hook identity; pass only the actual `currentAgentId` and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass `isShared` and `currentAgentId`, but `memory promote`, `promote-explain`, and `rem-harness` still call this path without those options. Those user-facing paths can still show or apply another agent's shared-workspace entries, so derive or require the same workspace/agent boundary there too.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:147-149`
  - body: `MemoryDreamingWorkspace` is exported through `openclaw/plugin-sdk/memory-core-host-status`, so making `shared` required breaks plugin code that constructs the existing `{ workspaceDir, agentIds }` shape. Make the field optional or use a narrower internal return type while still annotating resolver results.
  - confidence: 0.85
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:501-502`
  - body: This is a user-facing `fix(memory)` behavior change, but `CHANGELOG.md` has no entry for this PR or the linked contamination bug. Add a single-line entry under the active fixes section before merge.
  - confidence: 0.82
