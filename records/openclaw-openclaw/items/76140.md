## Summary

Keep open. The PR targets a real current-main memory contamination bug and moves in the right direction, but the latest draft still has merge-blocking agent-boundary gaps and a plugin SDK compatibility issue.

## What This Changes

The branch changes memory-core dreaming and short-term promotion to thread currentAgentId, mark shared workspaces, store per-agent session corpus/provenance data, and add isolation tests plus an audit note.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows shared-workspace agents are resolved together and manual/deep promotion ranking has no active-agent discriminator; the linked issue comments describe the same reproduction shape.

## Solution Assessment

No. The PR direction is appropriate, but the current patch still synthesizes an agent identity for shared ranking, leaves manual promotion callers unscoped, and makes an exported SDK type stricter than the existing contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619`
  - body: When `params.currentAgentId` is absent, this fallback passes the first configured agent into `rankShortTermPromotionCandidates`. In a shared workspace that defeats fail-closed isolation because deep promotion can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent identity and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass `isShared` and `currentAgentId`, but `memory promote` and `promote-explain` still call this path without those options. Those user-facing paths can still rank another agent's shared-workspace entries and `memory promote --apply` can write them into `MEMORY.md`, so derive the same workspace/agent boundary at those callers or make the rank API require it.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: `MemoryDreamingWorkspace` is exported through `openclaw/plugin-sdk/memory-core-host-status`, so making `shared` required breaks plugin code that constructs the existing `{ workspaceDir, agentIds }` shape. Make the new field optional or use a narrower internal type while still returning `shared` from `resolveMemoryDreamingWorkspaces`.
  - confidence: 0.84
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:619`
  - body: This is a user-facing `fix(memory)` behavior change, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active fixes section before merge.
  - confidence: 0.8
