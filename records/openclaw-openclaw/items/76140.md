## Summary

Keep open: this is a plausible fix for a real shared-workspace memory isolation bug, but the latest PR head still has blocking isolation, SDK compatibility, and fs-safe write regressions, and it lacks real shared-workspace behavior proof.

## What This Changes

The branch threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus and provenance handling, and isolation tests.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, CLI promote/promote-explain, REM preview, and SDK compatibility, while preserving fs-safe writes and validating with regression tests plus real shared-workspace proof.

## Reproduction Assessment

Yes by source inspection, though not by running a live repro. Current main resolves all configured agents for a workspace and ranks/promotes recall entries without an active-agent discriminator; related issue discussion narrows this to shared-workspace single-gateway setups.

## Solution Assessment

No. The PR direction is appropriate, but the latest patch is not the best final fix because it synthesizes identity, leaves user-facing ranking callers unscoped, breaks the exported SDK shape, and drops fs-safe corpus append protection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail closed when ranking lacks an agent id:** `extensions/memory-core/src/dreaming.ts:619`
  - body: On shared workspaces, this fallback turns a missing hook identity into the first configured agent. A cron/heartbeat path without a real `ctx.agentId` can still rank and promote that agent's entries instead of returning no candidates.
  - confidence: 0.92
- **[P1] Scope every promotion ranking caller:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new isolation filter only applies when callers pass `isShared` and `currentAgentId`, but `memory promote` and REM preview still call `rankShortTermPromotionCandidates` without those options. In shared workspaces those paths can continue to rank or promote cross-agent store entries.
  - confidence: 0.89
- **[P1] Keep corpus appends symlink-safe:** `extensions/memory-core/src/dreaming-phases.ts:803`
  - body: This replaces the existing `appendRegularFile(... rejectSymlinkParents: true)` call with `fs.appendFile`. If `memory/.dreams/session-corpus` or an agent subdirectory is symlinked, dreaming can append corpus text outside the workspace; keep the fs-safe append when adding per-agent paths.
  - confidence: 0.87
- **[P2] Keep MemoryDreamingWorkspace compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: `MemoryDreamingWorkspace` is exported through `plugin-sdk/memory-core-host-status`, so making `shared` required breaks plugin code that constructs the old `{ workspaceDir, agentIds }` shape. Make the field optional or use an internal enriched type.
  - confidence: 0.84
