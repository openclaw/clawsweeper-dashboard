## Summary

Keep open. The PR targets a real shared-workspace memory isolation bug, but the current head is still draft, lacks after-fix real behavior proof, and leaves concrete cross-agent promotion paths plus an SDK compatibility break.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, per-agent session corpus/provenance handling, and isolation tests for the linked memory contamination bug.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, manual promote/promote-explain, REM preview, and SDK compatibility, then provide redacted real behavior proof from a shared-workspace setup.

## Reproduction Assessment

Yes. Source inspection on current main shows two configured agents sharing one workspace are resolved together, ingested into one workspace-level corpus, and ranked without an active-agent discriminator.

## Solution Assessment

No. The PR direction is right, but this head still synthesizes an identity for shared ranking, leaves manual ranking callers unscoped, and breaks the exported SDK type shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619-620`
  - body: When params.currentAgentId is absent, this fallback passes the first configured agent into ranking. In a shared workspace, that defeats the fail-closed contract because deep promotion can still rank and promote that agent's entries without a real hook identity; pass only the actual current agent identity and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Scope manual promotion ranking to the selected agent:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass isShared and currentAgentId, but memory promote, promote-explain, and REM preview still call this path without those options. Those user-facing paths can still rank another agent's shared-workspace entries and memory promote --apply can write them into MEMORY.md, so derive the same workspace/agent boundary at those callers or make the rank API require it.
  - confidence: 0.89
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:149`
  - body: MemoryDreamingWorkspace is exported through openclaw/plugin-sdk/memory-core-host-status, so making shared required breaks plugin code that constructs the existing { workspaceDir, agentIds } shape. Make the new field optional or use a narrower internal type while still returning shared from resolveMemoryDreamingWorkspaces.
  - confidence: 0.84
