## Summary

Keep open. The PR targets a real memory-core isolation bug and improves several paths, but latest head still has merge-blocking gaps: shared ranking can synthesize an agent identity, manual promotion/REM preview ranking remains unscoped, and the exported SDK workspace type is made incompatibly stricter.

## What This Changes

The PR threads agent identity through memory-core dreaming, adds shared-workspace metadata, agent-scoped corpus/provenance handling, isolation tests, and a source audit note.

## Best Possible Solution

Land one consistent agent-boundary contract across managed dreaming, manual promotion/promote-explain, REM preview, and SDK compatibility, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows two configured agents sharing one workspace are resolved together, ingested into one shared corpus, and ranked without an active-agent discriminator.

## Solution Assessment

No. The PR direction is right, but the latest patch still falls back to a configured agent when no real identity is present, leaves manual ranking paths unscoped, and makes a public SDK type incompatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not synthesize an agent for shared ranking:** `extensions/memory-core/src/dreaming.ts:619-620`
  - body: When params.currentAgentId is absent, this fallback passes the first configured agent into rankShortTermPromotionCandidates. In a shared workspace that defeats fail-closed isolation because the sweep can still rank and promote that agent's entries without a real hook identity; pass only the actual currentAgentId and let shared ranking return no candidates.
  - confidence: 0.92
- **[P1] Make manual promotion use the same agent boundary:** `extensions/memory-core/src/short-term-promotion.ts:1247-1253`
  - body: The new rank filter only runs when callers pass isShared/currentAgentId, but memory promote, promote-explain, and rem-harness still omit those options. Those user-facing paths can still rank another agent's shared-workspace entries and promote them with --apply, so derive the same workspace/agent boundary at those callers or make it mandatory in the rank API.
  - confidence: 0.88
- **[P2] Keep MemoryDreamingWorkspace backward compatible:** `src/memory-host-sdk/dreaming.ts:147-149`
  - body: MemoryDreamingWorkspace is exported through openclaw/plugin-sdk/memory-core-host-status, so making shared required breaks plugin code that constructs the existing { workspaceDir, agentIds } shape. Make the new field optional or use a narrower internal type while still returning shared from resolveMemoryDreamingWorkspaces.
  - confidence: 0.84
- **[P3] Add the memory isolation fix to the changelog:** `extensions/memory-core/src/dreaming.ts:523`
  - body: This is a user-facing fix(memory) behavior change, but the PR does not update CHANGELOG.md. Add a single-line entry under the active fixes section before merge.
  - confidence: 0.82
