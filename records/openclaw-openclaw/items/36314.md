## Summary

Keep open. Current main has adjacent Gateway resilience pieces, including channel health restarts, shallow liveness plus readiness probes, stability diagnostics, hard-coded global error logging/exit handling, and narrow startup config recovery, but it still does not implement the requested coherent Gateway process watchdog, operator-configurable recovery policies, richer /health schema, or generalized degraded startup mode. Related #53684 and #53716 are still open partial work rather than a merged resolution.

## What This Changes

Requests built-in Gateway robustness mechanisms: a process watchdog, global error handling with recovery strategies, richer health checks, and degraded startup behavior.

## Best Possible Solution

Keep this as a Gateway resilience design item and split the accepted scope into focused follow-ups: process crash-loop supervision with bounded stop/notification behavior, explicit recovery-policy hooks for runtime/provider errors, additive health/readiness schema changes, and a defined degraded startup contract that composes with existing diagnostics and config recovery.

## Reproduction Assessment

Not applicable as an executable bug reproduction; this is a broad feature/design request. Source, docs, tests, and related-item context give high-confidence evidence that current main only partially covers the requested resilience surface.

## Solution Assessment

No, a single broad automated implementation is not the best way to solve this issue. The safer path is maintainer scoping, then focused PRs that coordinate with #53684 and #53716 rather than creating an overlapping Gateway recovery surface.
