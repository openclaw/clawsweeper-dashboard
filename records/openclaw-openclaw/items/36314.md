## Summary

Keep open: current main now provides global fatal/transient error handling, HTTP liveness/readiness probes, detailed diagnostics, and narrow last-known-good recovery, but it still lacks the requested unified process watchdog, bounded restart-loop policy, configurable recovery strategies, and generalized degraded startup mode. The active rescue-watchdog work covers part of that gap but remains unmerged and does not own the entire request.

## What This Changes

Add built-in Gateway watchdog supervision, configurable global error recovery, richer health reporting, and degraded startup behavior.

## Best Possible Solution

Converge on one bounded recovery architecture that uses isolated supervision for process failure, reuses existing health/readiness and doctor repair contracts, and adds only explicitly approved, upgrade-safe configuration and notification behavior.

## Reproduction Assessment

Not applicable: this is a multi-part feature proposal rather than a narrowly defined existing-behavior bug. Source inspection confirms that several adjacent mechanisms exist, but the proposed unified watchdog contract does not.

## Solution Assessment

No: the proposed top-level `watchdog` and `errorHandling` configuration is not yet proven to be the narrowest maintainable design. The remaining work should be reconciled with existing daemon supervision, health/readiness, doctor repair, fatal-error handling, and the active rescue-watchdog PR.
