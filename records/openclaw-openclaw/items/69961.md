## What This Changes

Adds a session-memory sanitizer, wires it into transcript reads and memory writes, and adds handler/sanitizer tests for template-token, tool-call XML, role-marker, and NO_REPLY cleanup.

## Best Possible Solution

Update this PR or a focused replacement to reuse shared sanitizer behavior, preserve legitimate content, keep regression coverage on the session-memory path, and attach redacted after-fix hook proof from a real run.

## Reproduction Assessment

Yes. Source inspection shows current main reads session JSONL message text unchanged and writes the resulting `sessionContent` verbatim under `## Conversation Summary`.

## Solution Assessment

No. Sanitizing at transcript and write boundaries is the right shape, but this implementation should reuse or align with existing sanitizer coverage and avoid eliding valid short replies or prose about tool tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the shared model-token sanitizer:** `src/hooks/bundled/session-memory/sanitize.ts:20-21`
  - body: The new token regex still misses formats OpenClaw already strips elsewhere, including `[INST]`, `<s>`, `<start_of_turn>`, reserved-token variants, and harmony markers. Those leaks would still be persisted into session memory and re-injected on the next `/new`, so this path should reuse or align with shared sanitizer coverage.
  - confidence: 0.92
- **[P2] Keep concise replies after stripping tokens:** `src/hooks/bundled/session-memory/sanitize.ts:92-95`
  - body: The `strippedRatio > 0.5 && cleaned.length < 80` branch still elides valid short answers when one leaked token dominates the character count, for example `2+2=4<|im_end|>`. Short factual replies are useful memory context, so the sanitizer should keep the cleaned reply unless the remainder is structurally garbage.
  - confidence: 0.9
- **[P2] Avoid treating prose mentions as tool-call XML:** `src/hooks/bundled/session-memory/sanitize.ts:32`
  - body: The open-ended `<tool_call>` alternative removes everything from any `<tool_call>` mention through end-of-string, so normal prose such as `Use <tool_call> to invoke tools.` can collapse or be elided. The existing visible-text sanitizer checks payload shape and code regions; this path needs equivalent false-positive protection.
  - confidence: 0.9
