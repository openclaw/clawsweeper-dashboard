## Summary

Keep open: the reported session-memory poisoning path is valid and this PR targets the right hook boundaries, but the current patch still has sanitizer correctness gaps and lacks real post-fix hook behavior proof before merge.

## What This Changes

The PR adds a session-memory sanitizer, wires it into transcript reads and memory writes, and adds unit/wiring tests for leaked chat-template tokens, raw tool_call XML, NO_REPLY, and malformed turns.

## Best Possible Solution

Land one focused session-memory fix that reuses or aligns with the shared scaffolding/token sanitizers, preserves legitimate short and technical prose, adds regression coverage, includes the changelog entry, and attaches real memory-file proof from an after-fix hook run.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence reproduction path: a synthetic session JSONL assistant message containing `<tool_call>...</tool_call><|im_end|>` is read unchanged by `getRecentSessionContent` and written verbatim under `## Conversation Summary`.

## Solution Assessment

No. Sanitizing at transcript-read and memory-write boundaries is the right shape, but this implementation should align with the existing shared sanitizer behavior and avoid eliding useful short or technical content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cover the existing special-token formats:** `src/hooks/bundled/session-memory/sanitize.ts:20-21`
  - body: The new token regex still covers only a subset of the model-control tokens OpenClaw already strips elsewhere, so Mistral `[INST]`, Phi `<s>`, Gemma `<start_of_turn>`, GPT-OSS/harmony markers, and reserved-token variants can still be persisted into session memory and re-injected on the next `/new`. Reuse or align with the existing shared token sanitizer and add cases for those formats.
  - confidence: 0.92
- **[P2] Preserve short replies after token cleanup:** `src/hooks/bundled/session-memory/sanitize.ts:92-95`
  - body: This heuristic still elides concise but valid replies when a single leaked token dominates the character count; for example `2+2=4<|im_end|>` cleans to useful content but is replaced with `[malformed turn elided]`. Short factual replies are common memory context, so the sanitizer should strip the token and keep the cleaned reply unless the remainder is structurally garbage.
  - confidence: 0.9
- **[P2] Avoid deleting literal tool_call prose:** `src/hooks/bundled/session-memory/sanitize.ts:32`
  - body: The regex treats any `<tool_call>` without a closing tag as leaked XML through end-of-string, so normal technical prose such as `Use <tool_call> to invoke tools.` or inline-code examples can collapse or be elided. The shared visible-text sanitizer already checks payload shape and code regions; this path needs the same false-positive protection.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/hooks/bundled/session-memory/handler.ts:274`
  - body: This is a user-facing hooks/session-memory fix that changes what gets persisted and re-injected after `/new` or `/reset`. Repo policy requires a single-line active-version `CHANGELOG.md` entry for user-facing fixes, but this branch does not touch the changelog.
  - confidence: 0.98
