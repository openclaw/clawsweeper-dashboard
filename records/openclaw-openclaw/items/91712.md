## Summary

Keep open: the reported diagnostic-state leak is source-plausible, but this PR's unconditional TTL fallback is too broad and can remove recoverable session state before the existing recovery machinery runs. The PR also still lacks after-fix real behavior proof beyond a code snippet.

## What This Changes

The PR changes diagnostic session-state pruning so any entry older than the existing TTL is deleted even when it is not idle.

## Best Possible Solution

Keep the leak fix, but make stale cleanup preserve recoverable processing or queued diagnostic entries until recovery has honored the configured abort window and then add focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level: current main can leave non-mutated diagnostic states after failed recovery, and the PR's added branch would delete any stale state regardless of recovery eligibility. I did not run a live gateway reproduction.

## Solution Assessment

No; the broad TTL fallback is only a plausible leak mitigation and is not the best fix because it bypasses configured recovery timing and queued-state convergence. A safer fix should delete only proven unrecoverable ghost entries or make cleanup aware of recovery windows and queue state.

Full review comments:

- **[P1] Preserve processing state until the abort window:** `src/logging/diagnostic-session-state.ts:61-64`
  - body: This branch deletes every `processing` entry once it is older than the fixed 30 minute TTL, but the heartbeat only requests active-abort recovery once `ageMs >= diagnostics.stuckSessionAbortMs`, and current tests allow operators to raise that value to 48 hours. With that existing configuration, the diagnostic state disappears before recovery can abort/drain the stuck run, so the stale session stops being recoverable.
  - confidence: 0.91
- **[P2] Keep queued idle state until recovery clears it:** `src/logging/diagnostic-session-state.ts:61-64`
  - body: The same `ageMs > SESSION_STATE_TTL_MS` fallback also matches idle entries with `queueDepth > 0`, bypassing the current idle-queued recovery path after failed or skipped recovery attempts. That drops the queue depth and generation record the heartbeat uses to classify stale queued work and release stale bookkeeping, so queued session recovery can stop converging while lane work still exists.
  - confidence: 0.84
