## Best Possible Solution

Keep the leak fix goal, but make cleanup recovery-aware so only proven orphaned diagnostic state is removed after recovery windows and queue-state convergence are respected, then add focused regression coverage and real after-fix proof.

## Reproduction Assessment

Yes, at source level: failed recovery outcomes can leave non-mutated diagnostic state behind, while current pruning only deletes idle empty entries. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No; the broad TTL fallback is only a plausible leak mitigation and is not the best fix because it bypasses configured recovery timing and queued-state convergence. A safer fix should delete only proven orphaned ghost entries or make cleanup aware of recovery windows and queue state.
