## What This Changes

The PR adds a fallback in diagnostic session-state pruning that deletes any entry older than `SESSION_STATE_TTL_MS` regardless of state or queue depth.

## Best Possible Solution

Land one canonical recovery-aware fix that cleans failed-recovery ghost entries only after active and queued recovery can no longer use the state, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes, at source level: failed recovery outcomes do not mutate diagnostic state, while current pruning only deletes idle empty entries. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The age-only fallback is a plausible leak mitigation, but it is not the best fix because heartbeat pruning runs before recovery and can remove state that configured active-abort or queued-idle recovery still needs.
