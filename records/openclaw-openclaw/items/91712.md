## Best Possible Solution

Land one canonical recovery-aware cleanup that prunes failed-recovery ghost entries only after active and queued recovery can no longer use the state, with focused tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows failed recovery outcomes return before mutating diagnostic state, while current main and `v2026.6.6` only TTL-prune idle empty entries; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The age-only fallback is a plausible leak mitigation, but it is not the best fix because heartbeat pruning runs before recovery and ignores configured abort windows and queued-idle recovery state.
