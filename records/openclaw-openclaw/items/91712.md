## Summary

Keep open: current main and v2026.6.6 still have the source-reproducible leak, but this PR's broad TTL fallback can delete recovery-relevant diagnostic state before configured recovery windows run and still lacks real behavior proof.

## What This Changes

The PR adds an age-only fallback to `pruneDiagnosticSessionStates` so stale diagnostic session entries are deleted after the existing TTL even when they are not idle or still have queue depth.

## Best Possible Solution

Keep the leak fix goal, but make cleanup recovery-aware so only proven orphaned diagnostic state is removed, then add focused regression coverage and real after-fix proof.

## Reproduction Assessment

Yes, at source level: failed recovery outcomes can leave non-mutated diagnostic state behind, while current pruning only deletes idle empty entries. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No; the broad TTL fallback is only a plausible leak mitigation and is not the best fix because it bypasses configured recovery timing and queued-state convergence. A safer fix should delete only proven orphaned ghost entries or make cleanup aware of recovery windows and queue state.
