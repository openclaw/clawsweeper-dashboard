## Summary

This should stay open: current main still appears to strip or fail to synthesize `reasoning_content` for Kimi/Moonshot OpenAI-compatible assistant replay, while the reporter’s 2026.5.12 log and Kimi’s own thinking/tool-call docs support the reported failure mode. ([platform.moonshot.ai](https://platform.moonshot.ai/docs/guide/use-kimi-k2-thinking-model.en-US))

## What This Changes

The issue requests Kimi OpenAI-completions replay history to include `reasoning_content` on assistant tool-call messages when thinking is enabled.

## Best Possible Solution

Add Kimi/Moonshot-specific OpenAI-completions replay compatibility so assistant history with tool calls retains or fills `reasoning_content` when thinking is enabled, while keeping stock OpenAI sanitized.

## Reproduction Assessment

Yes, at source level: current main only preserves `reasoning_content` for OpenRouter and DeepSeek/Xiaomi-style routes, while the issue gives a concrete 2026.5.12 Kimi 400 log on the second turn after tool calls. I did not run a live Kimi request in this read-only review.

## Solution Assessment

Yes; the narrow maintainable fix is to extend the existing provider compatibility/replay policy path for Kimi/Moonshot, not to add `reasoning_content` to every OpenAI-compatible assistant message.
