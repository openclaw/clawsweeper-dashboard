## Summary

Keep open for maintainer handling: the central target-account fan-out from the linked report is already fixed on main, but this PR still proposes a broader Telegram-origin ambiguity change and currently has blocking routing regressions plus missing real behavior proof.

## What This Changes

The PR broadens Telegram native exec approval ambiguity checks to Telegram-origin requests, filters native handling by explicit Telegram targets, removes cross-channel session-account fallback, updates approval tests, and adds a Node memory default for local heavy checks.

## Best Possible Solution

Keep the current main target-account implementation; if stricter unbound same-channel routing is desired, land it as a narrow Telegram-only change that preserves `both` mode and cross-channel session bindings with real Telegram proof.

## Reproduction Assessment

Yes, at source level: a multi-account Telegram config with `turnSourceChannel: "telegram"` and no bound Telegram account still falls through the current main `!boundAccountId` branch. I did not run a live Telegram repro because this review is read-only.

## Solution Assessment

No. The proposed direction may be useful, but this patch is not the best solution because it gates `both` mode native handling and removes shared cross-channel session-account binding outside the Telegram-specific bug boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not gate both mode on Telegram targets:** `extensions/telegram/src/exec-approvals.ts:89`
  - body: `matchesTelegramExplicitTargetAccount` treats `approvals.exec.mode === "both"` like pure target forwarding, so a config that keeps session/native Telegram approvals active while forwarding targets elsewhere can return `false` before normal session/origin matching runs. Restrict this gate to pure `targets` mode or to a narrower target-forwarding-only path so `both` keeps session delivery active.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Replacing the stored session-binding fallback with `return null` removes shared account routing for every native approval channel when the request came from another channel but the session store identifies the channel account. That turns a concrete binding into ambiguity and can stop the correct native handler from delivering the approval; keep the fallback or make the Telegram ambiguity guard local to Telegram.
  - confidence: 0.83
