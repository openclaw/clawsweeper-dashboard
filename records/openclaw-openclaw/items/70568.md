## Summary

Keep open: current main still has a source-level same-channel unbound Telegram ambiguity path, but this PR should not merge as-is because it suppresses valid `both` mode handling, removes shared cross-channel account binding, and lacks real Telegram behavior proof.

## What This Changes

The PR broadens Telegram native exec approval ambiguity checks to Telegram-origin requests, filters native handling by explicit Telegram targets, removes cross-channel session-account fallback, updates approval tests, and adds a local tsgo `NODE_OPTIONS` default.

## Best Possible Solution

Keep the current main target-account implementation; if stricter unbound same-channel Telegram routing is desired, land a narrower Telegram-only patch that preserves `both` mode and shared cross-channel bindings with real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main only applies the multi-account ambiguity guard to non-Telegram turn sources, so a `turnSourceChannel: "telegram"` request without a bound Telegram account still falls through to permissive matching. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The requested direction may be valid, but this patch is not the best solution because it gates `both` mode native handling, removes shared cross-channel binding outside the Telegram-specific bug boundary, and mixes in unrelated local-check memory policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not gate both mode on Telegram targets:** `extensions/telegram/src/exec-approvals.ts:89`
  - body: `matchesTelegramExplicitTargetAccount` treats `approvals.exec.mode === "both"` like pure target forwarding, so valid session/native Telegram approvals can return `false` whenever configured targets are non-Telegram or scoped to another Telegram account. Restrict this gate to pure `targets` mode or to a narrower target-forwarding path so `both` keeps session delivery active.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Replacing the stored session-binding fallback with `return null` removes shared account routing for every native approval channel when the request came from another channel but the session store identifies the expected channel account. Keep the fallback or make the Telegram ambiguity guard local to Telegram.
  - confidence: 0.83
