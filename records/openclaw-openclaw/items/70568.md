## Summary

Keep open: the Telegram unbound same-channel ambiguity is still source-reproducible on current main, but this PR is not mergeable as-is because it suppresses valid `both`-mode native/session approvals, removes shared cross-channel session binding, and lacks real Telegram proof.

## What This Changes

The PR changes Telegram exec approval account matching and tests for unbound multi-account approvals, adds target-account scoping coverage, changes shared approval-account binding, and adjusts local tsgo `NODE_OPTIONS` defaults.

## Best Possible Solution

Land a narrower Telegram-local ambiguity guard that preserves `both`-mode session delivery and shared cross-channel bindings, then validate it with focused tests and redacted real Telegram proof.

## Reproduction Assessment

Yes, at source level. On current main, an unbound `turnSourceChannel: "telegram"` request gets no bound account and skips the ambiguity guard because it only applies to non-Telegram sources, so multiple eligible Telegram accounts can pass.

## Solution Assessment

No. The Telegram ambiguity-guard direction is plausible, but this diff also changes `both`-mode target gating and shared account binding beyond the narrow bug; a Telegram-local guard that preserves the existing binding contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve session handling in both mode:** `extensions/telegram/src/exec-approvals.ts:89-92`
  - body: `matchesTelegramExplicitTargetAccount` runs for `mode: "both"` before normal session/origin matching, so configs that forward targets only to another channel or another Telegram account can suppress valid Telegram-native session approvals. Restrict this gate to pure `targets` mode or the target-forwarding path.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Replacing the stored session-binding fallback with `return null` removes shared account routing for every native approval channel when the request came from another channel but the session store identifies the expected channel account. Keep the fallback or make the new ambiguity guard Telegram-local.
  - confidence: 0.83
