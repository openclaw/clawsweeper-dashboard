## Summary

Keep open. Current main now fixes the linked explicit `targets[].accountId` fan-out from #69916, but it does not implement this PR's broader same-channel Telegram unbound-account ambiguity guard, and the PR still has concrete approval-routing regressions that should block merge as written.

## What This Changes

The PR changes Telegram native exec approval matching to reject unbound Telegram-origin approvals in multi-account setups, adds Telegram/native approval regression tests, removes cross-channel session-binding fallback, and adds a local heavy-check `NODE_OPTIONS` memory default.

## Best Possible Solution

Keep the shipped `6dec2e1852` target-account implementation for #69916, then decide separately whether unbound Telegram-origin approvals without any explicit account binding should be rejected in multi-account configs. If maintainers want that stricter behavior, it should be a narrow Telegram-only patch that preserves `both` mode session/native delivery and shared cross-channel session-binding compatibility.

## Reproduction Assessment

Yes. The source-level reproduction is a multi-account Telegram config where an exec approval request has `turnSourceChannel: "telegram"` but no `turnSourceAccountId` or session-bound account; current main falls through to `!boundAccountId`, while the PR adds a regression test that expects both accounts to reject that shape.

## Solution Assessment

No. The direction is plausible for the remaining unbound same-channel ambiguity, but this PR is not the best fix as written because it suppresses valid `mode: "both"` native handling and removes shared cross-channel session-binding fallback outside the Telegram-specific bug boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not gate `both` mode on Telegram targets:** `extensions/telegram/src/exec-approvals.ts:89`
  - body: `matchesTelegramExplicitTargetAccount` applies the target-account gate when `approvals.exec.mode` is `"both"`, so a config that keeps session/native Telegram approvals active while forwarding targets elsewhere can return `false` before normal session/origin matching runs. Restrict this gate to pure `targets` mode or to a narrower target-forwarding-only path so `both` keeps session delivery active.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Replacing the stored session-binding fallback with `return null` removes shared account routing for every native approval channel when the request came from another channel but the session store identifies the channel account. That turns a concrete binding into ambiguity and can stop the correct native handler from delivering the approval; keep the fallback or make the Telegram ambiguity guard local to Telegram.
  - confidence: 0.83
