## Summary

Keep this PR open, but it is not mergeable as-is: the branch still has blocking approval-routing regressions and lacks the real Telegram behavior proof required for this kind of Telegram authorization change.

## What This Changes

The PR broadens Telegram native exec approval ambiguity checks, adds target-account scoping tests, removes the shared cross-channel session-binding fallback, and adds a local tsgo Node memory default.

## Best Possible Solution

Keep the shipped target-account implementation; if stricter unbound same-channel Telegram approval handling is still desired, land a narrow Telegram-only change that preserves `both` mode native handling and shared cross-channel session bindings, with real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main only ambiguity-guards non-Telegram turn sources, so a Telegram-origin request without a bound account still falls through to normal permissive matching. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The proposed branch is not the best fix as written because it can suppress `both` mode native Telegram approvals and removes a shared cross-channel session-account binding that current main and tests preserve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Do not suppress both-mode native approvals:** `extensions/telegram/src/exec-approvals.ts:89`
  - body: `matchesTelegramExplicitTargetAccount` treats `mode: "both"` like pure target forwarding and returns `false` when there is no matching Telegram target. In `both` mode, native/session handling should still run, so configs that forward targets elsewhere can lose valid Telegram approval delivery; restrict this gate to the target-forwarding-only path or allow normal matching to continue.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Replacing the stored session-binding fallback with `return null` removes shared account routing for every native approval channel when the request came from another channel but the session store identifies the expected channel account. That turns a concrete binding into ambiguity and can stop the correct native handler from delivering the approval; keep the fallback or make the Telegram ambiguity guard local to Telegram.
  - confidence: 0.84
