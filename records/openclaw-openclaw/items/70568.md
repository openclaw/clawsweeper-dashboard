## Summary

Current main already implements the observable Telegram multi-bot approval fan-out fix this PR was opened for: native approval handling now honors scoped Telegram `approvals.exec/plugin.targets[].accountId`, the linked report was closed as fixed, and the fix shipped in `v2026.4.29`. The PR branch should not land as-is because it still contains a `mode: "both"` routing regression and an unrelated local heavy-check runtime change.

## What This Changes

The PR changes Telegram native approval routing to reject unbound same-channel multi-account approvals unless exactly one eligible account remains, gates eligible accounts by Telegram forwarding targets, updates approval regression tests, removes cross-channel session account fallback, and adds a local heavy-check `NODE_OPTIONS` default.

## Best Possible Solution

Close this PR and keep the shipped `6dec2e1852` implementation. Any broader no-target ambiguity policy or local heavy-check memory default should be handled as separate, focused work with its own tests and ownership review.

## Reproduction Assessment

Yes for the linked fan-out bug: current main has a focused multi-account Telegram regression that asserts only the configured target account handles the native approval. The PR's broader no-target same-channel policy is not independently reproduced by a remaining open report in the provided context.

## Solution Assessment

No, this PR is no longer the best way to solve the issue. Current main uses a narrower shipped fix that scopes native approvals by explicit exec/plugin target account while preserving unscoped targets; this branch still suppresses valid `mode: "both"` session delivery and bundles unrelated script runtime changes.
