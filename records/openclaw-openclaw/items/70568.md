## Summary

Keep open. Current main already covers the linked explicit target-account fan-out bug, but this PR still proposes a separate stricter rule for unbound Telegram-origin approvals and the patch has blocking routing regressions. The remaining behavior needs maintainer judgment rather than cleanup closure.

## What This Changes

This PR changes Telegram native exec approval matching to reject unbound Telegram-origin approvals in multi-account configs, removes shared cross-channel session-binding fallback, adds Telegram/infra regression tests, and adds a local heavy-check Node old-space default.

## Best Possible Solution

Keep the current target-account implementation on main for the shipped fan-out fix. If maintainers want stricter handling for unbound Telegram-origin approvals, land a narrow Telegram-only change that preserves `mode: "both"` session/native delivery and shared cross-channel session-binding compatibility, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is a multi-account Telegram config with a Telegram-origin approval request that has `turnSourceChannel: "telegram"` but no `turnSourceAccountId` or session-bound Telegram account; current main falls through to the permissive match branch, while this PR adds tests expecting both accounts to reject it.

## Solution Assessment

No. The direction may be valid for the remaining ambiguous same-channel shape, but this patch is not the best fix because it suppresses valid `mode: "both"` native handling and removes a shared cross-channel session-binding fallback outside the Telegram-specific bug boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve native handling when `both` has no Telegram target:** `extensions/telegram/src/exec-approvals.ts:89`
  - body: `matchesTelegramExplicitTargetAccount` runs for `approvals.exec.mode === "both"` and returns `false` when there are no Telegram targets, so configs that forward approvals to another channel while keeping session/native Telegram delivery active will stop handling valid Telegram approvals. Restrict this gate to pure `targets` mode or to the explicit target-forwarding path.
  - confidence: 0.88
- **[P2] Preserve cross-channel session account bindings:** `src/infra/approval-request-account-binding.ts:102`
  - body: Returning `null` here drops the stored session-binding fallback for every native approval channel when the request came from a different channel but the session store identifies the expected channel account. That turns a concrete binding into ambiguity and can prevent the correct native handler from delivering the approval; keep the fallback or make the Telegram ambiguity guard local to Telegram.
  - confidence: 0.83
