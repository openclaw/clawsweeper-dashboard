## Summary

Keep open: current main still source-reproduces the unbound Telegram-origin ambiguity, but this branch is not merge-ready because it conflicts with main, duplicates target-account scoping now present on main, breaks existing approval-routing contracts, and lacks real Telegram proof.

## What This Changes

The PR changes Telegram exec approval account matching and tests for unbound multi-account approvals, adds target-account scoping coverage, removes cross-channel session binding fallback, and appends a local tsgo `NODE_OPTIONS` default.

## Best Possible Solution

Rebase onto current main, keep the shipped target-account scoping and shared binding behavior, add only the Telegram-local unbound-source ambiguity guard, then validate with focused tests and redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main only applies the ambiguity guard to non-Telegram turn sources, so a Telegram-sourced approval with no bound account can pass for every eligible Telegram account; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The unbound Telegram-source guard is the right direction, but this PR also duplicates current target-account scoping, gates `both` mode too broadly, and removes shared cross-channel binding; a narrower Telegram-local change is safer.
