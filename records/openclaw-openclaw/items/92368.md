## Summary

Keep this PR open for revision: the diagnostic gap is valid, but the implementation does not match the one-time lifecycle contract described by the linked issue, and its supplied proof does not exercise a real integration path.

## What This Changes

The PR emits console warnings for unsupported tool-availability diagnostics, assigns `always` signals to malformed expressions, and updates a planner test expectation.

## Best Possible Solution

Keep availability evaluation side-effect free, validate malformed descriptors once during registration or plugin activation, and leave composite-expression diagnostics without a fabricated atomic signal.

## Reproduction Assessment

Yes for the pure API: an empty availability group deterministically returns an `unsupported-signal` diagnostic and causes the planner to hide the descriptor without logging. A live gateway-startup reproduction is not established because no inspected production caller currently invokes the planner.

## Solution Assessment

No. One-time validation at descriptor registration or plugin activation is narrower and matches the linked issue discussion better than adding a console side effect to a reusable evaluator.

- **[P2] Warn once during descriptor activation:** `src/tools/availability.ts:190-196`
  - body: `evaluateToolAvailability` is a reusable evaluation boundary, so this side effect runs again whenever a descriptor is evaluated. The linked issue asks for startup-time validation in plugin activation; emit the warning from that lifecycle owner so one malformed descriptor cannot repeatedly spam logs.
  - confidence: 0.9

  - confidence: 0.95
