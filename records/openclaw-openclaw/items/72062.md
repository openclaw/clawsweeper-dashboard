## Summary

Keep open. Current main still lacks a typed, documented, and enforced negative exec script-path policy; existing per-agent denial remains tool-name based while exec wrapper/script path resolution is used only for allowlist and approval decisions. This is a valid core exec-policy gap, but the API shape and precedence are security-sensitive and should get maintainer design before implementation.

## What This Changes

The issue requests a per-agent exec wrapper/script path deny policy so allowed shell interpreters cannot invoke sensitive wrapper scripts that bypass tool-name denial.

## Best Possible Solution

Add a documented per-agent negative exec script-path policy in the core exec security surface, resolve script candidates with the existing shell-wrapper/interpreter helpers, make deny win before allowlist satisfaction, durable approval reuse, allow-always persistence, and approval prompts, and cover both gateway and node host paths with focused tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure tools.deny for an MCP namespace, allow sh/bash through exec allowlist, then invoke sh or bash with a sensitive wrapper path; current main filters only the exec tool name while the exec policy has no negative script-path check.

## Solution Assessment

Unclear. A negative resolved script-path policy is the right class of fix, but the exact API name, placement, glob semantics, interpreter coverage, and gateway/node precedence should be chosen by maintainers before a repair PR is attempted.
