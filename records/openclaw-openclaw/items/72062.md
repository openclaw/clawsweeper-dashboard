## Summary

Keep open. Current main still has no typed, documented, or enforced negative exec path policy, while `tools.deny` remains tool-name based and exec approvals only use script-path resolution for positive allowlist/approval decisions. The newer open PR #75101 is a plausible implementation path for the same class of gap, but it is not merged and targets the broader #74379 path-deny request, so this issue should stay open for maintainer/security review and linkage rather than be closed early.

## What This Changes

Add a per-agent exec wrapper or script-path deny policy so an agent that may run `sh` or `bash` can still be blocked from invoking configured wrapper scripts by resolved path before approval or allowlist handling.

## Best Possible Solution

Review open PR #75101 against this issue’s wrapper-delegation use case, adapt it if needed to cover per-agent resolved script-path denial before every exec dispatch path, then land a documented, tested deny policy and link this issue as covered by that implementation.

## Reproduction Assessment

Yes. Configure `agents.list[].tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh /path/to/sensitive-wrapper.sh`; current main filters only `tool.name` while wrapper script paths are used only for allowlist/approval matching.

## Solution Assessment

Unclear. A negative resolved path gate before allowlist and approval is the right class of fix, but maintainers need to choose the canonical API shape, precedence, glob semantics, shell/interpreter coverage, and whether #75101’s broader `tools.exec.denyPathPatterns` design should be the implementation for this issue too.

## Security Review

Status: needs_attention

Summary: This issue describes a concrete exec policy gap where per-agent tool-name denies do not constrain otherwise-allowed interpreter invocations of sensitive wrapper paths.

Concerns:

- **[medium] Add exec path denial before host dispatch:** `src/agents/bash-tools.exec.ts:1609`
  - body: Current `createExecTool` rejects unsafe control-shell commands and then proceeds to node, gateway, or sandbox execution without a configured deny-path gate, so a policy added only inside one host approval phase could still miss another dispatch path or elevated-full bypass.
  - confidence: 0.86
- **[medium] Block gateway wrapper paths before approval reuse:** `src/agents/bash-tools.exec-host-gateway.ts:274`
  - body: Gateway exec evaluates allowlist satisfaction and durable approvals without a negative script-path policy, so an allowlisted interpreter/script path can proceed toward execution or prompting instead of failing fast by agent policy.
  - confidence: 0.88
- **[medium] Mirror policy in node system.run:** `src/node-host/invoke-system-run.ts:391`
  - body: The node-host policy phase resolves per-agent exec config and allowlist state without a deny-path phase, so a gateway-only fix would leave node-targeted exec with different security semantics.
  - confidence: 0.84
