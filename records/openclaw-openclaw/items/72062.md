## Summary

Keep open. Current main still lacks a typed, documented, and enforced per-agent negative policy for exec wrapper or script paths; the existing tool deny path is tool-name based, while exec approval code can resolve script paths only for allowlist/approval decisions. Because this is a core exec-security API change, the field shape, interpreter scope, path-resolution semantics, and gateway/node precedence need maintainer agreement before a repair PR is safe.

## What This Changes

Requests a new `agents.list[].tools.deny_wrappers` or equivalent per-agent glob policy that denies resolved shell/interpreter wrapper script paths before exec allowlist matching or approval prompts.

## Best Possible Solution

Add a documented, per-agent negative exec script-path policy in the core exec security surface, reuse the existing shell-wrapper/script candidate resolution, make deny win before allowlist satisfaction, durable approval reuse, allow-always persistence, and approval prompts, then cover both gateway and node exec paths with focused regression tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure per-agent `tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh <sensitive-wrapper>`; current main filters only the exec tool name while exec policy has no script-path deny check.

## Solution Assessment

Unclear. A negative script-path policy is the right class of fix, but the API name, nesting, interpreter coverage, path resolution semantics, and gateway/node precedence should be settled by maintainers because this changes core exec security behavior.
