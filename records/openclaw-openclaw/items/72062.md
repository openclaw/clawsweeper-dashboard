## Summary

Close as superseded. Current main still lacks a per-agent exec wrapper/path deny policy, but the same remaining work is now tracked by the canonical path-deny issue #74379 and open implementation PR #75101, which proposes `tools.exec.denyPathPatterns` plus `agents.list[].tools.exec.denyPathPatterns` as the broader policy surface that covers this wrapper-script use case.

## What This Changes

Request a per-agent policy, proposed as `agents.list[].tools.deny_wrappers` or equivalent, that denies resolved exec wrapper/script paths before approval while preserving existing exec allowlist behavior.

## Best Possible Solution

Continue the remaining work in #74379/#75101 with the broader exec path-deny API, and ensure the final implementation explicitly covers per-agent wrapper script denial, deny-before-approval precedence, gateway and node-host exec paths, docs, and focused tests for `sh`/`bash` wrapper invocations.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: allow `exec`/`sh` through exec policy, deny an MCP namespace through `tools.deny`, then invoke a sensitive wrapper path through `sh` or `bash`; current main only filters the `exec` tool name and has no negative path policy.

## Solution Assessment

Yes for triage routing. A broader `tools.exec.denyPathPatterns`/per-agent path-deny policy is a more maintainable superset of a wrapper-only field, provided the canonical PR preserves the wrapper-script case and applies before allowlist and approval decisions.
