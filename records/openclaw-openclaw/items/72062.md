## Summary

Keep open. Current main still lacks a typed, documented, and enforced negative exec script-path policy; tool denial remains name-based while exec allowlist code can positively match shell script paths. A newer open PR, #75101, appears to be the right implementation candidate, but it is not merged and does not directly close this issue, so this issue should remain open for maintainer review against that PR.

## What This Changes

Request a per-agent exec policy that denies wrapper/script path invocations before allowlist or approval so tool-name denies cannot be bypassed through `exec`.

## Best Possible Solution

Land a documented exec path-deny policy that runs before allowlist satisfaction, durable approval reuse, `allow-always` persistence, and approval prompts, with consistent gateway and node-host behavior. PR #75101 is the likely path, provided maintainers confirm its `denyPathPatterns` semantics cover this issue's resolved wrapper-script use case and per-agent inheritance expectations.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `agents.list[main].tools.deny` for an MCP namespace, allow `sh`/`bash` or the script through exec policy, then invoke a sensitive wrapper path through `exec`; current main filters only the `exec` tool name and has no negative script-path policy.

## Solution Assessment

Unclear. A negative exec path policy is the right class of fix, and #75101 is a plausible broader implementation, but maintainers should confirm API naming, global-versus-agent precedence, glob semantics, and gateway/node ordering before treating this issue as solved.
