## Summary

Keep open: the latest head fixes the prior nullable child-stdin blocker and has sufficient copied live-output proof, but it still needs a narrow repair because input-wait `process poll` results now include advancing idle-duration text that can defeat existing no-progress loop detection.

## What This Changes

The branch adds `process attach`, input-wait metadata and hints, child/PTY stdin writability tracking, docs, changelog coverage, and regression tests for interactive background session recovery.

## Best Possible Solution

Land the attach recovery feature after making input-wait poll/attach status loop-detector-safe, either by normalizing away idle-duration fields in no-progress hashes or by keeping no-output poll text stable.

## Reproduction Assessment

Yes for the review finding by source: current loop detection hashes `process poll` result text, and PR head changes that text as idle duration advances. The underlying attach behavior is supported by the contributor's copied live output rather than rerun in this read-only review.

## Solution Assessment

No, not as-is: the attach direction, docs, tests, and stdin guard repair are aligned, but the dynamic input-wait poll text needs a loop-detector-safe repair before merge. The narrow safer fix is to ignore `idleMs`/`lastOutputAt` and duration text in process no-progress hashing or keep poll's no-output suffix stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve no-progress detection for input-wait polls:** `src/agents/bash-tools.process.ts:549`
  - body: The loop detector treats `process poll` as a known polling tool and hashes the returned text to decide whether identical polls made no progress. Please normalize/ignore the idle duration in loop detection or keep the no-progress poll text stable.
