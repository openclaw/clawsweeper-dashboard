## Summary

Keep open: the feature addresses a real process recovery gap and includes useful proof/tests, but PR head still introduces a loop-detection regression because input-wait polling returns time-varying text while current no-progress hashing includes `process poll` text.

## What This Changes

The PR adds `process attach`, input-wait metadata and hints, stdin writability tracking for child/PTTY adapters, docs, changelog coverage, and regression tests for interactive background sessions.

## Best Possible Solution

Land the attach recovery feature after making input-wait poll/attach status loop-detector-safe while preserving the current scoped, in-memory, no-persistence process model.

## Reproduction Assessment

Yes for the review finding by source: PR head's running `process poll` text includes a formatted idle duration, and current main hashes `process poll` text for no-progress detection. The contributor's copied live output is sufficient for the attach behavior itself.

## Solution Assessment

No, not as-is: the attach direction, docs, stdin state checks, and tests are aligned, but the dynamic input-wait text needs a loop-detector-safe repair before merge. The narrow fix is to normalize idle-duration fields/text in process no-progress hashing or keep no-output poll text stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve no-progress detection for input-wait polls:** `src/agents/bash-tools.process.ts:217`
  - body: The loop detector treats `process poll` as a known polling tool and hashes the returned text, but this helper puts the advancing idle duration into the running poll result. A session waiting on stdin with no new output can therefore produce a different result hash on every poll and avoid the warning/critical no-progress circuit breaker. Please normalize the idle duration out of loop hashing or keep the no-output poll text stable.
  - confidence: 0.89
