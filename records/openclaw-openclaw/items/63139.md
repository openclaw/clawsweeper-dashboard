## Summary

Keep open. Current main still invokes `before_model_resolve` inside each embedded fallback attempt, so an unconditional routing hook can still rewrite every fallback candidate back to the same provider/model; the existing `runId` and model fields are only a partial mitigation.

## What This Changes

The issue requests a corrected or explicit `before_model_resolve` fallback contract so routing plugins do not reapply the same override on every fallback attempt.

## Best Possible Solution

OpenClaw should choose and document the public hook contract, then add regression coverage for a router override plus cross-provider fallback; if the hook stays per-candidate, expose retry metadata in the SDK context.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure multiple fallbacks and a `before_model_resolve` hook that always returns the same provider/model, then the current fallback loop re-enters the embedded runner and re-applies the hook for each candidate.

## Solution Assessment

Unclear. Hoisting the hook best matches the reporter’s once-per-run expectation, while adding `attempt`/`isFallbackRetry` plus docs is the narrower compatible contract for plugins that need per-candidate visibility.
