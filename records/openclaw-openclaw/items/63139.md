## Summary

Keep open. Current main still dispatches `before_model_resolve` from inside each fallback candidate's embedded run, so an unconditional routing plugin can still rewrite every attempted candidate back to the same provider/model. The `runId`, `modelProviderId`, and `modelId` mitigation is present, including the workaround path discussed in the issue, but retry metadata is still missing and the public docs/types do not clearly settle the once-per-run versus once-per-candidate contract.

## What This Changes

The issue asks for `before_model_resolve` to stop reapplying routing overrides on every model fallback candidate, or to expose retry metadata so plugins can detect fallback retries.

## Best Possible Solution

OpenClaw should choose and document the hook contract explicitly. The strongest runtime fix is to run `before_model_resolve` once at the logical run boundary and pass that selection into fallback as the primary candidate; the smaller compatible path is to preserve per-candidate dispatch, add `attempt` and `isFallbackRetry` to `PluginHookAgentContext`, wire them through embedded run call sites, document the behavior, and add regression coverage for routing-plugin failover.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple fallback candidates plus a hook that always returns the same provider/model, then follow the current `runWithModelFallback` -> `runEmbeddedPiAgent` path where the hook override is applied before each candidate's model resolution.

## Solution Assessment

Unclear. The one-shot hook design best matches the reporter's expected logical-run contract, while the retry-metadata/documentation path is narrower and safer for plugins that intentionally observe each fallback candidate.
