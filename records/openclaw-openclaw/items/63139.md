## Label Justifications

- `P2`: This is a valid runtime/plugin routing bug with clear source evidence and limited blast radius, but the fix needs a public SDK contract decision.
- `impact:auth-provider`: The issue is about provider/model routing and fallback behavior when plugin hooks override model selection.

## Summary

Keep open: current main still invokes `before_model_resolve` from inside each embedded fallback attempt, so unconditional routing plugins can still rewrite every fallback candidate back to the same provider/model; `runId` and current-candidate model fields are useful partial mitigations but do not settle the public hook/fallback contract.

## What This Changes

The issue asks OpenClaw to make `before_model_resolve` safe with model fallback, either by firing once per logical run or by exposing retry metadata so routing plugins can detect fallback iterations.

## Best Possible Solution

OpenClaw should choose and document the hook/fallback contract, then add regression coverage for an unconditional router override plus cross-provider fallback; if the hook remains per-candidate, expose retry metadata in the SDK context.

## Reproduction Assessment

Yes, source-level. Configure multiple fallback candidates and an unconditional `before_model_resolve` router; current main re-enters embedded execution per fallback candidate and reapplies the hook before resolving each candidate.

## Solution Assessment

Unclear. Hoisting the hook best matches the once-per-run expectation, while adding `attempt`/`isFallbackRetry` plus docs is the narrower compatible contract for plugins that need per-candidate visibility.
