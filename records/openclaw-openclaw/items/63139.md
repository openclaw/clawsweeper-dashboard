## Summary

Keep open. Current main still invokes `before_model_resolve` inside each embedded fallback attempt, so an unconditional routing hook can still rewrite every fallback candidate back to the same provider/model; existing `runId` and current-candidate model fields only partially mitigate that behavior.

## What This Changes

The issue asks OpenClaw to make `before_model_resolve` safe with model fallback, either by firing it once per logical run or by exposing retry metadata for fallback-aware routing plugins.

## Best Possible Solution

OpenClaw should choose and document the public hook/fallback contract, then add regression coverage for a router override plus cross-provider fallback; if the hook remains per-candidate, expose retry metadata in the SDK context.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure multiple fallback candidates and a `before_model_resolve` hook that always returns the same provider/model, then the fallback loop re-enters `runEmbeddedPiAgent` and reapplies the override for each candidate.

## Solution Assessment

Unclear. Hoisting the hook best matches the once-per-run expectation, while adding `attempt`/`isFallbackRetry` plus docs is the narrower compatible contract for plugins that need per-candidate visibility.
