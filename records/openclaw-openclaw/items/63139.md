## Summary

Keep open. Current main still invokes `before_model_resolve` from inside each embedded fallback candidate attempt, so an unconditional routing plugin can still rewrite every fallback candidate back to its chosen provider/model. The partial `ctx.runId`/model context mitigation is present, including the downstream workaround path discussed in the issue, but the public contract still lacks `attempt`/`isFallbackRetry` and does not document per-candidate hook semantics.

## What This Changes

The issue asks OpenClaw to fix the `before_model_resolve` and model-fallback contract so routing plugins do not defeat runtime failover by reapplying provider/model overrides on every fallback candidate.

## Best Possible Solution

OpenClaw should choose and ship a clear hook contract. The strongest runtime fix is to run `before_model_resolve` once at the logical run boundary and pass that result into fallback as the primary candidate; a smaller compatible path is to keep per-candidate semantics, expose `attempt` and `isFallbackRetry`, document the behavior, and add regression coverage for routing-plugin failover.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple fallback candidates plus a hook that always returns the same provider/model, then follow the current `runWithModelFallback` -> `runAgentAttempt` -> `runEmbeddedPiAgent` path where the hook override is reapplied before every candidate's model resolution.

## Solution Assessment

Unclear. The one-shot hook design best matches the reporter's expected logical-run contract, but the smaller metadata/documentation fix is more backward-compatible for plugins that intentionally observe each fallback candidate.
