## Summary

This PR should stay open: the zh-CN localization direction is useful and the supplied real-behavior proof is sufficient, but the latest head still has a source-reproducible generated i18n metadata defect that should block merge.

## Best Possible Solution

Keep the i18n rendering approach, but regenerate or repair the generated locale metadata so every non-target English fallback introduced by `chat.slashCommands` is tracked truthfully before merge.

## Solution Assessment

No. Routing chat UI text through i18n is the right direction, but merging generated locale outputs with inaccurate fallback metadata would undermine future translation refreshes.
