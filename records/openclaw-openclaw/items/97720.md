## Summary

Keep open: current main still drops bodyless Telegram membership service updates before dispatch and has no member-event config surface, but the request is a new Telegram event/config contract that needs maintainer product/API direction before implementation.

## What This Changes

Request an opt-in, default-off Telegram per-group member join/leave event so agents or skills can react to group membership changes.

## Best Possible Solution

Define an opt-in Telegram membership-event contract that reuses existing group/topic authorization and room-event or system-event semantics, or explicitly extend the plugin API if maintainers want this outside the bundled Telegram plugin.

## Reproduction Assessment

Yes, at source level: a Telegram message update with new_chat_members or left_chat_member but no text/media has no body path, so resolveTelegramInboundBody returns null and dispatch stops. I did not run a live Telegram group join/leave scenario.

## Solution Assessment

Unclear. The per-group flag is plausible, but maintainers should first choose the durable event contract and whether it belongs in the bundled Telegram plugin, a system-event/room-event path, or a plugin SDK extension.
