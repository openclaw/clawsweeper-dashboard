## Summary

Keep open: current main still has no Telegram member join/leave event surface, and bodyless membership service updates still fall out before an agent turn. The request is useful and source-reproducible, but it adds a new opt-in Telegram event/config contract and needs maintainer product/API direction before implementation.

## What This Changes

Request an opt-in, default-off Telegram group setting that forwards member join/leave service updates to the configured agent as minimal inbound events.

## Best Possible Solution

Choose the Telegram membership-event contract first, preferably within the Telegram plugin or a reusable channel event seam, then implement default-off join/leave delivery with access-control, routing, docs, and regression coverage.

## Reproduction Assessment

Yes, from source inspection: a Telegram update with `new_chat_members` or `left_chat_member` and no normal text/media has no synthetic body path, so the current message-context builder returns `null`. I did not run a live Telegram group join/leave scenario in this read-only review.

## Solution Assessment

Unclear: a per-group default-off flag is plausible, but maintainers should first choose whether membership changes are synthetic messages, quiet room events, internal hooks, or a broader channel lifecycle API.
