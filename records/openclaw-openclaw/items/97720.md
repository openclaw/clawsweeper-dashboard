## Summary

Keep open: current main still drops Telegram join/leave service-message updates before they become agent-visible work, and the requested default-off event/config behavior needs maintainer API and product direction rather than conservative cleanup.

## What This Changes

The issue requests an opt-in Telegram group setting that forwards member join and leave service messages to the agent.

## Best Possible Solution

Keep this as a maintainer-owned Telegram plugin design item; if accepted, implement default-off member events as authorized system/room events with docs, config schema, tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level: a Telegram member service message with `new_chat_members` or `left_chat_member` but no ordinary text/media reaches the message path, produces no inbound body, and stops before dispatch. I did not run a live Telegram group join/leave scenario.

## Solution Assessment

Unclear. A per-group default-off flag is plausible, but existing reaction and Slack member handling point toward an authorized system-event shape that maintainers should choose before a PR.
