## Summary

Keep open: current main does not expose Telegram member join/leave updates to agents, and the source path still drops bodyless Telegram service updates. The request is useful, but it adds new default-off Telegram event/config behavior and needs maintainer product/API direction before a safe implementation lane.

## What This Changes

Requested an opt-in Telegram group setting to surface member join and leave service messages to agents.

## Best Possible Solution

Keep this as a maintainer-owned Telegram plugin/API design item and, if accepted, implement a default-off membership-event contract that reuses existing group access controls and room-event/system-event semantics where possible.

## Reproduction Assessment

Yes, at source level: a Telegram update with `new_chat_members` or `left_chat_member` but no normal text/media has no synthetic body path, so `resolveTelegramInboundBody` returns `null` and dispatch stops. I did not run a live Telegram group join/leave scenario in this read-only review.

## Solution Assessment

Unclear. A per-group flag is plausible and has a reaction-notification precedent, but maintainers should first choose whether membership changes are synthetic messages, quiet room events, internal hooks, or a broader channel lifecycle API.
