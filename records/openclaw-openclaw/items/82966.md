## Summary

Keep open: the fix is directionally useful, but the current patch broadens a fresh-only helper and can make preflight compaction skip transcript re-estimation for stale sessions. It also lacks patched-run real behavior proof.

## What This Changes

The branch makes preserved stale session `totalTokens` visible through `resolveFreshSessionTotalTokens`, adds several freshness guards, and updates session/status/gateway tests plus changelog text.

## Best Possible Solution

Keep a preserved-total display path separate from freshness-sensitive compaction logic, and add a regression that stale low cached totals do not suppress transcript-based preflight compaction.

## Reproduction Assessment

Yes at source level: current main filters finite stale totals through `resolveFreshSessionTotalTokens` for `/context` and status/session projection paths. I did not run a live current-main or patched-binary session in this read-only review.

## Solution Assessment

No: the display direction is right, but changing `resolveFreshSessionTotalTokens` globally is too broad while preflight compaction still treats it as fresh-only. A separate preserved-total resolver or an explicit preflight guard is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep preflight compaction on fresh estimates:** `src/config/sessions/types.ts:553-556`
  - body: Removing the stale guard makes `resolveFreshSessionTotalTokens` return a number for stale entries. `runPreflightCompactionIfNeeded` then treats that value as `freshPersistedTokens` and skips transcript estimation, so an old lower total can be passed as the fresh compaction override and a needed preflight compaction can be missed; keep this helper fresh-only there or add a direct freshness guard before using it.
  - confidence: 0.88
