## Summary

Keep open: the reported session-token display bug is valid, but this PR should not merge yet because the helper change also lets preflight compaction treat stale totals as fresh, and the supplied proof does not show the patched behavior running in a real setup.

## What This Changes

The branch changes session token accounting so stale preserved `totalTokens` are displayed by `/context`, status, and gateway session rows while adding freshness guards in memory flush/fork paths and updating tests and the changelog.

## Best Possible Solution

Preserve stale totals for display/status surfaces while keeping preflight compaction, memory-flush, and fork logic on fresh-only token data, with regression coverage for both display and compaction behavior.

## Reproduction Assessment

Yes at source level: current `main` hides stale preserved totals because `/context` and status call the fresh-only resolver. I did not establish a live current-main or patched-binary reproduction in this read-only review.

## Solution Assessment

No: the direction is right, but changing `resolveFreshSessionTotalTokens` globally is too broad unless every freshness-sensitive consumer is updated. A safer solution is a display-safe preserved-total resolver or explicit fresh guards on all runtime gating paths, including preflight compaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep preflight compaction from trusting stale totals:** `src/config/sessions/types.ts:553-556`
  - body: Removing the stale guard makes `resolveFreshSessionTotalTokens` return a number for stale entries. `runPreflightCompactionIfNeeded` stores that as `freshPersistedTokens`, skips transcript usage estimation, and then passes the old total as the `tokenCount` override to preflight compaction, so an outdated lower count can skip a needed refresh/compaction. Keep a fresh-only path here or update this consumer with regression coverage.
  - confidence: 0.87
