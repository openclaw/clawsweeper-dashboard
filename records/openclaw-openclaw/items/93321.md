## Summary

Keep open: current `main` and `v2026.6.9` still use compaction repair paths that preserve aborted orphaned tool-call turns unless the caller opts into dropping them, and https://github.com/openclaw/openclaw/pull/93584 is the active candidate fix that should be reviewed before this issue closes.

## What This Changes

The issue reports that timeout-interrupted compaction can retain orphaned assistant `toolCall` blocks without matching `toolResult` blocks, permanently poisoning later strict-provider requests for the same session.

## Best Possible Solution

Review and land a narrow compaction-boundary repair that drops aborted/error retained tool-use turns before summarization or provider replay; https://github.com/openclaw/openclaw/pull/93584 is the current candidate path.

## Reproduction Assessment

Yes from source inspection: current compaction paths use default repair behavior, and default transcript repair preserves aborted assistant tool-call turns unless callers opt into dropping them. I did not perform a live provider-timeout reproduction in this read-only pass.

## Solution Assessment

Yes: the maintainable fix belongs at the compaction planning/safeguard boundary before LLM summarization or provider replay, not in provider-specific error handling. The linked PR targets that boundary with focused tests and should remain the candidate fix path.
