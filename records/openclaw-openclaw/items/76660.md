## Summary

Keep open. Current main still has the reported Codex app-server parity gap, and #76663 is an open draft PR with closing syntax for this issue, so the issue should remain open until that implementation lands or is replaced.

## What This Changes

The issue requests that Codex app-server source replies include the `message` tool when source delivery is `message_tool_only`, even under restrictive tool profiles such as `coding`.

## Best Possible Solution

Land or replace the narrow Codex app-server fix that forwards `forceMessageTool` for message-tool-only source replies and adds the coding-profile regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: select Codex message-tool-only source delivery with `tools.profile: "coding"`, and dispatch suppresses automatic delivery while the Codex app-server dynamic tool list omits the forced `message` tool.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass `forceMessageTool` from the Codex app-server run when `sourceReplyDeliveryMode === "message_tool_only"`, matching the existing shared factory and embedded-runner behavior.
