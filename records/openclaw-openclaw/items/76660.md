## Summary

Keep open. Current main still has a source-level mismatch between source-reply delivery suppression and the Codex app-server dynamic tool list, and #76663 is already an open fix PR that explicitly closes this issue.

## What This Changes

Ensure Codex app-server source replies using message-tool-only delivery keep the `message` dynamic tool available under restrictive tool profiles such as `coding`.

## Best Possible Solution

The end state should be the narrow Codex app-server parity fix that forwards `forceMessageTool` for message-tool-only source replies and keeps the coding-profile regression covered.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: select Codex message-tool-only source delivery with `tools.profile: "coding"`, and dispatch suppresses automatic delivery while the Codex app-server tool builder does not force `message` into the dynamic tool list.

## Solution Assessment

Yes. The narrowest maintainable solution is to pass `forceMessageTool` from the Codex app-server run when `sourceReplyDeliveryMode === "message_tool_only"`, matching the existing shared tool-factory contract and the linked regression test direction.
