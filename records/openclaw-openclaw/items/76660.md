## Summary

Keep open. Current main still has the Codex app-server/source-reply tool-list mismatch, and open draft PR #76663 is already the same-author implementation candidate with closing syntax for this issue.

## What This Changes

Ensure Codex app-server source replies include the `message` tool when source delivery selects message-tool-only mode under restrictive tool profiles.

## Best Possible Solution

Land a narrow Codex app-server parity fix that forwards `forceMessageTool` for `message_tool_only` source replies and keeps the coding-profile regression covered.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence path: select Codex message-tool-only source delivery with `tools.profile: "coding"`, and dispatch suppresses automatic delivery while the Codex app-server tool builder does not force `message` into the dynamic tool list.

## Solution Assessment

Yes. Forwarding `forceMessageTool` when `sourceReplyDeliveryMode === "message_tool_only"` is the narrowest maintainable fix and matches the shared tool-factory contract plus the existing embedded-runner path.
