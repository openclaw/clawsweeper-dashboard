## Summary

Keep open: the force-pushed fix addresses the earlier stopped-before-drain ordering bug, but the shared clear path now flushes unsent pending drafts before clearing across finalizable draft channels, and the live Telegram proof remains insufficient.

## What This Changes

The PR adds a shared `stopAndDrain()` helper, switches finalizable draft clearing to drain before marking stopped with a 50 ms delay, and exposes a Telegram `waitForDrain` hook.

## Best Possible Solution

Keep the ordering fix in the Telegram final-delivery lifecycle, or add an explicit clear mode that awaits in-flight edits without forcing unsent drafts, with regression coverage and redacted live Telegram ordering proof.

## Reproduction Assessment

No for a high-confidence live reproduction of the original intermittent Telegram race. Source inspection does show the PR-introduced behavior: `stopAndDrain()` flushes pending draft text before `clear()` reads and deletes the message.

## Solution Assessment

No; draining the shared clear/discard lifecycle is broader than needed and can send drafts that should remain discarded. A narrower fix should await only the relevant in-flight Telegram progress edit or expose an explicit drain option used by Telegram final delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not flush unsent drafts while clearing:** `src/channels/draft-stream-controls.ts:59`
  - body: `stopForClear()` is shared by finalizable draft streams, including Discord and Mattermost. Calling `stopAndDrain()` here flushes any throttled pending draft before clear/discard, so a draft that had not been sent yet can now be posted to the channel and then deleted, causing notifications/flicker and leaving a stale message if cleanup fails. Keep clear/discard waiting for in-flight work without forcing unsent pending text, or localize the Telegram ordering wait to the final-send path.
  - confidence: 0.92
