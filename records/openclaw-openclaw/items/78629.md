## Summary

Keep open: the reported Telegram ordering race is plausible, but this patch currently needs real Telegram proof and has a blocking source-level drain bug because the new drain path is called after the stream is already marked stopped.

## What This Changes

The PR adds a shared draft-stream drain method, uses it while clearing finalizable drafts with a short delay, and exposes a Telegram draft-stream drain hook.

## Best Possible Solution

Preserve or repair the in-flight wait guarantee, add focused coverage for stopped drain semantics and Telegram progress final ordering, then require real Telegram proof before merge.

## Reproduction Assessment

No; the PR body only provides an unchecked manual test plan, and I did not establish the live Telegram race against current main. Source inspection does show the proposed stopped-drain path cannot wait on in-flight edits because `flush()` is gated by `isStopped()`.

## Solution Assessment

No; calling a flush-based drain after `markStopped()` skips the drain and removes the current explicit in-flight wait. A safer fix should either drain before marking stopped or make the drain helper explicitly wait in-flight after stopping, with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep drain waiting after the stream is marked stopped:** `src/channels/draft-stream-controls.ts:60`
  - body: `stopForClear()` now calls `markStopped()` before `stopAndDrain()`, but `stopAndDrain()` is implemented as `flush()` followed by `stop()`. Since `flush()` exits while `isStopped()` is true, this path no longer awaits the in-flight Telegram send/edit that the old `waitForInFlight()` handled, so the final `sendMessage` can still race the last preview edit.
  - confidence: 0.94
