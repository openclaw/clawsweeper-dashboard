## Summary

Keep open: this PR targets a valid Telegram progress-draft ordering problem, but the branch is not mergeable because the new drain path stops the stream before its flush loop can wait for in-flight edits, and the attached screenshots do not prove the fixed live ordering.

## What This Changes

This PR adds a shared `stopAndDrain()` draft-stream helper, switches Telegram draft clearing to use it with a 50 ms delay, and exposes a Telegram draft `waitForDrain` hook.

## Best Possible Solution

Repair the clear/final lifecycle so Telegram awaits the last progress edit or deletion before durable final send, with focused stopped-drain regression coverage and redacted live Telegram proof.

## Reproduction Assessment

No for the original live Telegram race; the screenshots and source path identify the surface, but there is no deterministic current-main live reproduction. The PR's drain-ordering defect is source-reproducible from `markStopped()` -> `stopAndDrain()` -> `flush()` exiting while stopped.

## Solution Assessment

No; the proposed helper is called after the stopped flag is set and is implemented as `flush()` then `stop()`, so it skips the wait this fix needs. A safer repair should drain before marking stopped or explicitly await in-flight Telegram work before final send.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep draining in-flight edits after stopping:** `src/channels/draft-stream-controls.ts:60`
