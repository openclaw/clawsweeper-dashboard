## Summary

Keep open. Current main and the latest release still resolve a matched binding's missing target agent through the default-agent fallback, and Telegram inbound routing still enters that shared resolver; the related default-selection issue is adjacent, not a replacement for this stale-binding privacy path.

## What This Changes

The issue requests that Telegram route bindings pointing at unavailable agents stop silently falling back to the main/default agent.

## Best Possible Solution

Fail closed for matched bindings whose target agent is absent, keep default-agent fallback for true no-match routing, and add validation or doctor diagnostics plus docs for stale bindings[].agentId.

## Reproduction Assessment

Yes, at source level: configure agents.list with main/default, add a matching Telegram route binding whose agentId is absent, and resolveAgentRoute will return the default agent through pickFirstExistingAgentId. I did not run a live Telegram gateway in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract, but fail-closed handling for matched stale bindings is the narrowest maintainable direction. A configurable catch-all agent would be a larger feature/config decision.
