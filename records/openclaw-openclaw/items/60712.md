## Summary

Keep open. Current main still routes a matched binding whose target agent is absent through the default-agent resolver, and Telegram inbound traffic still enters that resolver; the related #44375 tracks default selection when no binding handles traffic, not this stale matched-binding privacy behavior.

## What This Changes

The issue requests that Telegram or channel route bindings pointing to missing agents stop silently falling back to the main/default agent.

## Best Possible Solution

Fail closed for matched route bindings whose target agent is absent, preserve the existing default-agent fallback only for true no-match routing, and add config validation or doctor diagnostics plus docs for stale `bindings[].agentId`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `agents.list` with `main` as the default, add a matching Telegram/core route binding whose `agentId` is absent, and observe that `resolveAgentRoute` returns the default agent through `pickFirstExistingAgentId`.

## Solution Assessment

Unclear until maintainers choose the exact product contract, but the requested direction is right: the main agent should not silently absorb traffic from a stale explicit binding. The narrowest maintainable default is to fail closed for matched stale bindings while keeping no-match default routing intact.
