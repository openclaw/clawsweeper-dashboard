## Summary

Keep open. Current main still resolves a matched binding whose target agent is absent through the default-agent fallback, and Telegram inbound routing still enters that shared resolver; the related default-selection issue is adjacent, not a replacement for this privacy-sensitive stale-binding behavior.

## What This Changes

Stop Telegram and shared route bindings that point to missing agents from silently falling back to the main/default agent.

## Best Possible Solution

Fail closed for matched route/configured bindings whose target agent is absent, keep default-agent fallback only for true no-match routing, and add validation or doctor diagnostics plus docs for stale `bindings[].agentId`.

## Reproduction Assessment

Yes, at source level: configure `agents.list` with `main` as default, add a matching Telegram route binding whose `agentId` is absent, and `resolveAgentRoute` will return the default agent through `pickFirstExistingAgentId`. I did not run a live Telegram gateway in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract, but the requested fail-closed direction is the narrowest maintainable fix; adding a configurable catch-all would be a larger feature/config decision.
