## Label Justifications

- `P1`: A stale binding can misroute private Telegram traffic into the wrong default agent workflow for real multi-agent users.
- `impact:security`: The issue is about sensitive messages crossing an intended agent/workspace boundary.
- `impact:message-loss`: Messages addressed to one configured agent can be delivered to a different agent instead.
- `impact:session-state`: The wrong agent id changes the session key and workspace/session store used for the inbound conversation.

## Summary

Keep open. Current main and v2026.5.18 still resolve matched route bindings and configured ACP bindings with missing agent IDs through the default-agent fallback, so the reported wrong-agent/privacy path remains real and needs a maintainer routing-contract decision.

## What This Changes

The issue asks OpenClaw to stop silently routing Telegram messages from matched bindings with missing agent IDs to the default/main agent.

## Best Possible Solution

Fail closed or surface an explicit diagnostic for matched bindings whose target agent is absent, while preserving default-agent fallback only for true no-match routing unless maintainers approve a catch-all contract.

## Reproduction Assessment

Yes, at source level: configure agents.list with an existing default agent and add a matching route or ACP binding whose agentId is absent; the resolver paths call pickFirstExistingAgentId, which returns the default agent. I did not run a live Telegram gateway in this read-only review.

## Solution Assessment

Unclear until maintainers choose the routing contract. The narrowest maintainable fix appears to be fail-closed or explicit diagnostic handling for matched stale bindings, not a broad catch-all feature by default.
