## Summary

Keep open. Current main still resolves a matched binding whose target agent is absent through the default-agent fallback, and Telegram inbound routing still enters that resolver; related #44375 is adjacent default-selection work, not a duplicate of this stale matched-binding privacy behavior.

## What This Changes

The issue asks OpenClaw to stop silently routing messages from stale explicit agent bindings to the default main agent.

## Best Possible Solution

Fail closed for matched route bindings whose target agent is absent, keep default-agent fallback only for true no-match routing, and add validation or doctor diagnostics plus docs for stale `bindings[].agentId`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `agents.list` with `main` as default, add a matching Telegram route binding whose `agentId` is absent, and observe `resolveAgentRoute` return the default agent through `pickFirstExistingAgentId`.

## Solution Assessment

Unclear until maintainers choose the product contract, but the requested direction is correct: the main agent should not silently absorb traffic from a stale explicit binding. The narrow maintainable default is to fail closed for matched stale bindings while preserving no-match default routing.
