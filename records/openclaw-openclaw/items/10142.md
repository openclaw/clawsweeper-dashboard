## Summary

Keep open. Current main and v2026.5.12 still do not expose the requested internal `session:end` hook; the existing lifecycle surface is the typed plugin `session_end` hook, and this issue is paired with open implementation PRs that need maintainer review or rejection before cleanup.

## What This Changes

Request an internal `session:end` hook event that fires when sessions complete so external workflow orchestrators can be signaled without agent-authored cleanup commands.

## Best Possible Solution

Choose one supported lifecycle contract, then either land an internal `session:end` event with docs/tests across completion paths or explicitly document plugin `session_end`/reset hooks as the replacement and close the open implementation candidates accordingly.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection still confirms the current gap: internal hook docs/types expose `session:patch` but not `session:end`, while completion-related support is on the plugin `session_end` surface.

## Solution Assessment

Unclear. A session completion hook is a reasonable direction, but maintainers need to choose whether the maintainable API is an internal `HOOK.md` event or the existing typed plugin lifecycle hooks before accepting one of the open implementations.
