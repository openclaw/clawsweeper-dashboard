## Summary

Keep open. Current `main` still lacks the requested internal `session:end` event: internal hooks expose only `session:patch`, while the typed plugin `session_end` hook covers reset/replacement and shutdown/restart, subagents use `subagent_ended`, and cron completion emits neither lifecycle hook. The open linked PR https://github.com/openclaw/openclaw/pull/61675 is conflicting and addresses lazy reset/memory hooks rather than this broader completion contract, so it is not yet a safe canonical resolution. Likely related people: @shtse8 (high confidence, introduced core wiring for plugin lifecycle hooks), @vincentkoc (high confidence, split and recently maintained plugin hook contracts), and @graciegould (medium confidence, introduced the typed internal `session:patch` contract).

## What This Changes

Requests a typed internal `session:end` hook so workflow orchestrators can react when an agent session completes.

## Best Possible Solution

Define one canonical typed lifecycle contract—preferably extending the plugin hook surface—with explicit reasons, terminal outcome data, and deduplication, then bridge it to internal hook packs only if operator-installed hooks require the same guaranteed boundary; cover reset, shutdown/restart, cron, and subagent paths deliberately rather than equating every agent turn with session termination.

## Reproduction Assessment

Yes at source level. Current main has no typed or documented internal `session:end`, and the cron and subagent terminal paths can be followed directly to their existing, different completion mechanisms.

## Solution Assessment

No, the proposed broad internal event is not yet the best-defined fix. Maintainers should first choose the canonical plugin-versus-internal contract and specify which lifecycle boundaries count as session termination, then implement consistent coverage and deduplication.
