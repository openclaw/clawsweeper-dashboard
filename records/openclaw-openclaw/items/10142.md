## Summary

Keep open: current main and v2026.6.10 still do not expose a documented internal `session:end` event, and the existing `session_end` coverage is the separate typed plugin lifecycle surface. The open linked implementation candidate at https://github.com/openclaw/openclaw/pull/61675 addresses related daily/idle reset hooks but explicitly does not add new hook event names, so the remaining decision is maintainer-owned hook contract direction.

## What This Changes

Requests a new internal `session:end` hook event emitted when sessions complete, with context for workflow orchestration.

## Best Possible Solution

Maintainers should either document typed plugin `session_end` as the supported replacement for this use case or land a narrow internal `session:end` contract with tests across the relevant completion paths.

## Reproduction Assessment

Yes, at source level: current main can be checked by searching internal hook types and docs, which show `session:patch` and plugin `session_end` but no internal `session:end` event. I did not run a live hook install because this is a feature-contract gap rather than a runtime crash reproduction.

## Solution Assessment

Unclear: adding an internal event is a plausible solution, but the existing typed plugin `session_end` surface and the open daily/idle reset PR mean maintainers should first decide the supported hook contract.
