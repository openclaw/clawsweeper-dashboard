## Summary

Keep open: current main and v2026.6.10 still do not expose the requested internal `session:end` hook event, while the implemented `session_end` behavior is the separate typed plugin lifecycle hook and the open linked PR remains a product-contract candidate rather than a merged fix.

## What This Changes

Request an internal `session:end` hook event that fires when an agent session completes so external workflow orchestrators can react without agent-authored cleanup commands.

## Best Possible Solution

Maintainers should either document typed plugin `session_end` as the supported replacement for this workflow use case, or land a narrow internal `session:end` contract with tests and docs for the relevant completion paths.

## Reproduction Assessment

Yes, at source level: current main exposes `session:patch` internally and `session_end` as a typed plugin hook, but no documented or typed internal `session:end` event exists. I did not run a live hook install because the remaining gap is a feature-contract surface rather than a runtime crash.

## Solution Assessment

Unclear: adding an internal event is a plausible solution, but the existing typed plugin `session_end` surface may be the intended owner. Maintainers should decide the supported hook contract before an implementation PR lands.
