## Summary

Keep open. Current main and v2026.5.12 still lack the requested internal `session:end` hook event; the existing `session_end` surface is a typed plugin hook, and open linked implementation PRs still need maintainer review or rejection.

## What This Changes

Request to add an internal `session:end` hook event that fires when a session completes so external workflows can react without agent-issued commands.

## Best Possible Solution

Choose one supported lifecycle contract, then either land an internal `session:end` event with docs/tests across completion paths or explicitly document typed plugin `session_end`/reset hooks as the supported replacement and close the open implementation candidates accordingly.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection confirms the current gap: internal hook docs/types expose `session:patch` but not `session:end`, while completion-related support is on the typed plugin `session_end` surface.

## Solution Assessment

Unclear. A session completion hook is a reasonable direction, but maintainers need to choose whether the maintainable API is an internal `HOOK.md` event or the existing typed plugin lifecycle hook before accepting one of the open implementations.
