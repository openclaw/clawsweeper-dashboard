## Summary

Keep open. Current main and the latest release still do not expose the requested internal `session:end` hook; the existing `session_end` signal is a typed plugin hook, and the linked implementation PRs are still open and conflicting, so maintainers need to choose the API direction before cleanup can close this issue.

## What This Changes

Requests a new internal `session:end` hook event that fires when sessions complete so external workflow orchestrators can proceed without agent-authored cleanup commands.

## Best Possible Solution

Establish one supported session-completion contract: either add an internal `session:end` event with typed guard, docs, and tests across completion paths, or explicitly document typed plugin `session_end` as the supported replacement.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks the requested internal event while exposing adjacent typed plugin lifecycle hooks.

## Solution Assessment

Unclear. A session completion signal is reasonable, but maintainers need to choose whether the maintainable API is an internal `HOOK.md` event or the existing typed plugin lifecycle hook before accepting one path.
