## Summary

Keep this issue open for a maintainer lifecycle-API decision. Current main still lacks the requested internal `session:end` event, and the existing typed plugin hooks do not provide a general agent-task completion signal across cron and subagent runs; the linked open PR addresses daily and idle resets instead of this contract.

## What This Changes

Add a reliable session-completion hook so external workflow orchestrators can react without requiring the agent to run a final CLI command.

## Best Possible Solution

Define a canonical typed plugin event for agent-task completion, distinct from persisted-session finalization, with explicit cron and subagent coverage, outcome metadata, and exactly-once guarantees; avoid adding a parallel internal event unless maintainers deliberately want HOOK.md scripts to be a second lifecycle API.

## Reproduction Assessment

Yes at the source-contract level: current main has no internal `session:end`, and the identifiable cron and subagent completion paths do not emit the typed plugin `session_end` event either.

## Solution Assessment

No, not as an immediately added parallel internal-hook event. The narrower maintainable solution is to separate agent-task completion from persisted-session finalization and expose the former through the canonical typed plugin lifecycle after maintainers approve its semantics.
