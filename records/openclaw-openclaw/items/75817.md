## Summary

Keep this issue open. Current main still contains object-unsafe `channelLogs[channelId]` dereferences in the gateway channel manager, and the global unhandled-rejection handler still exits on generic secondary TypeErrors, so the reported crash path remains actionable.

## What This Changes

The issue requests hardening for a gateway crash where a channel-task rejection can become a secondary `undefined.error` failure during event-loop pressure.

## Best Possible Solution

Add a narrow gateway channel-manager hardening fix: resolve or create a fallback channel logger/runtime before diagnostics, make all channel diagnostic calls object-safe, and cover a rejecting channel task when its logger map entry is absent while keeping the generic global unhandled-rejection policy fatal for unrelated errors.

## Reproduction Assessment

Yes for the proximate crash path. A focused regression test can register a channel plugin, omit that channel from `channelLogs`, make `startAccount` reject, and observe the current code throw from `log.error?.(...)` instead of preserving the original channel failure.

## Solution Assessment

No for the broad suggested top-level suppression. The repository already has a global fatal unhandled-rejection handler, so the safer maintainable fix is local object-safe logging and fallback setup in `src/gateway/server-channels.ts`.

## Repair Work Prompt

Fix the gateway channel-manager crash tracked at https://github.com/openclaw/openclaw/issues/75817. Current main can dereference a missing `channelLogs[channelId]` entry during channel start/stop/health-monitor diagnostics, turning the original channel failure into `Cannot read properties of undefined (reading 'error')` or similar. Keep the fix local to the gateway channel manager: create or resolve a safe fallback channel logger/runtime before diagnostic use, make diagnostic calls object-safe, and do not broaden the global `process.on('unhandledRejection')` fatal policy except for tests proving it remains fatal for unrelated generic errors. Add regression coverage in `src/gateway/server-channels.test.ts` for a channel plugin whose logger entry is absent and whose `startAccount` rejects; the test should assert no process-level unhandled rejection and the original channel error is recorded/logged instead of a secondary missing-logger TypeError. Include a user-facing `CHANGELOG.md` fix entry crediting `@xonaman`. Do not touch unrelated channel/provider behavior or dependency versions.
