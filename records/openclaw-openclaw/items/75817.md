## Summary

Keep this open. Current main still contains the channel-manager missing-logger crash surface that can turn a handled channel task rejection into the reported secondary `undefined.error` TypeError, and the related unhandled-rejection items cover other owners or transport paths rather than this gateway channel lifecycle path.

## What This Changes

The issue reports a Linux gateway crash where event-loop pressure exposes an unhandled channel task rejection that becomes a secondary `undefined.error` TypeError and causes process restart loops.

## Best Possible Solution

Harden the gateway channel manager locally: ensure every channel lifecycle path resolves a real or fallback logger/runtime before diagnostics, make diagnostic logging object-safe, and add a regression test for a rejecting channel task when the logger map lacks that channel id. Preserve the existing fatal global unhandled-rejection policy for unrelated errors.

## Reproduction Assessment

Yes for the proximate crash path. Current code inspection shows an absent `channelLogs[channelId]` plus a rejecting channel task reaches `log.error?.(...)`; a focused regression can omit the logger entry and force `startAccount` to reject, although the broader DNS-pressure scenario was not live-replayed.

## Solution Assessment

No, the broad suggested top-level suppression is not the best fix because the repo already has a global fatal unhandled-rejection handler. The narrow maintainable fix is local channel-manager fallback/object-safe logging with targeted coverage.

## Repair Work Prompt

Fix the gateway channel-manager crash reported at https://github.com/openclaw/openclaw/issues/75817. Current main can dereference a missing `channelLogs[channelId]` in `src/gateway/server-channels.ts` when a channel task rejects or stop diagnostics run after the logger/runtime map diverges from the live channel plugin list. Implement a local hardening fix in the channel manager: resolve/create a safe fallback logger and runtime for channel ids before start/stop diagnostics and plugin lifecycle contexts, make diagnostic calls object-safe, and add a regression test where a channel plugin exists but `channelLogs` lacks that channel id and `startAccount` rejects. Do not add broad top-level `process.on('unhandledRejection')` suppression or weaken the existing fatal policy in `src/infra/unhandled-rejections.ts`. Keep the change scoped to gateway channel lifecycle behavior, update `CHANGELOG.md` because this is user-facing availability behavior, and do not change unrelated channel/provider behavior.
