## Summary

Keep this open. Current main still has a plausible channel-manager crash path where deferred channel plugin startup can run with no per-channel logger entry, and the async channel-task rejection handler then dereferences `log.error` on an undefined logger before the original failure can be logged or recovered.

## What This Changes

The issue requests gateway hardening for an unhandled rejection that becomes a secondary `undefined.error` crash during channel startup under event-loop pressure.

## Best Possible Solution

Harden the channel manager so every loaded channel has a stable logger/runtime surface when it starts or stops, including after deferred plugin loading, and add regression coverage for a rejecting channel task with an absent logger entry while keeping the generic fatal unhandled-rejection policy intact.

## Reproduction Assessment

Yes for the proximate crash path: create a channel manager whose active registry contains a channel but whose `channelLogs` map lacks that id, then make `startAccount` reject. Current main will enter the tracked task catch and throw the secondary `log.error` TypeError.

## Solution Assessment

Yes, a narrow channel-manager/logger fallback fix is the right direction. A broad top-level unhandled-rejection suppression would hide unrelated fatal failures and would not address the stale logger/runtime map that creates this secondary crash.
