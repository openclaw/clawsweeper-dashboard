## Summary

Keep this open. Current main still has the missing channel logger dereference that can turn a channel task rejection into the reported secondary `undefined.error` TypeError, and the related unhandled-rejection reports cover different crash paths rather than superseding this gateway channel-manager issue.

## What This Changes

Harden the Gateway channel manager so a channel rejection under event-loop pressure is logged or recovered instead of crashing through a missing logger object.

## Best Possible Solution

Add a narrow channel-manager hardening fix: resolve or create a fallback channel logger/runtime before start and stop diagnostics, make diagnostic calls object-safe, and add a regression test where a channel task rejects while its logger entry is absent. Keep the generic global unhandled-rejection policy fatal for unrelated errors.

## Reproduction Assessment

Yes for the proximate crash path. A focused test can install a channel plugin, omit that channel from `channelLogs`, make `startAccount` reject, and observe current code throw from `log.error?.(...)` instead of preserving the original channel failure; the broader DNS-pressure timing was not live-run in this read-only review.

## Solution Assessment

No, the broad suggested top-level handler is not the best fix because the repository already has a global fatal unhandled-rejection handler. The maintainable fix is local object-safe logging and fallback channel runtime/logger setup in `src/gateway/server-channels.ts`, with regression coverage.
