## Summary

Keep open. The PR addresses a real media-loss path and now includes live Telegram/browser proof, but the patch is not merge-ready because it unconditionally replaces legitimate assistant-selected media in both block replies and final payload assembly.

## What This Changes

The branch makes embedded agent replies prefer queued tool media over assistant-authored `MEDIA:` directives, preserves attempt-level tool media for final payloads and trajectory metadata, and adds browser screenshot skill guidance.

## Best Possible Solution

Land an agent-level media fix that keeps tool artifacts as the trusted fallback for absent or stale assistant paths while preserving explicit valid assistant media selections, with regression coverage for block replies and final payload merging.

## Reproduction Assessment

Yes, source-reproducible: current main clears queued tool media when an assistant reply already has media, and the PR head has a source-level conflict with the existing assistant-selection integration test. I did not execute tests under the read-only review constraint.

## Solution Assessment

No. The lower-level agent media seam is the right place to fix stale assistant paths, but unconditional tool-media precedence is too broad; the safer repair should distinguish stale/invented assistant paths from legitimate assistant-selected media.
