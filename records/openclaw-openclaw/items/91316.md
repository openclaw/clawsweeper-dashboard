- `impact:data-loss`: The reported hard-reset path clears or archives visible transcript history when the user requested a transcript-preserving soft reset.
- `impact:message-loss`: The Control UI drops the `soft` token and trailing user message before sending the command to the Gateway.
- `impact:session-state`: The bug changes reset-mode selection and transcript/session preservation semantics.

## Summary

Keep open: current main still drops `/reset soft ...` in the Control UI local reset dispatch path, while the Gateway-side reset parser and docs already define soft reset semantics when the full command reaches the backend.

## What This Changes

The issue reports that Control UI should preserve `/reset soft ...` arguments instead of dispatching a hardcoded `/reset`.

## Best Possible Solution

Preserve the full `/reset` command text when Control UI dispatches local reset commands, treat `/reset soft...` as non-destructive for confirmation/refresh decisions, and add focused regression tests for bare reset versus soft reset with a tail.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: type `/reset soft ...` in Control UI and the local reset branch sends only `/reset`. I did not run the browser UI because this review is read-only.

## Solution Assessment

Yes, the best fix is narrow: correct Control UI command forwarding and destructive-reset detection, not the Gateway parser, because Gateway soft-reset parsing and tests already handle the full command body.
