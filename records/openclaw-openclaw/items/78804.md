## What This Changes

The issue reports stale hashed module references after an npm global package update, breaking `sessions_send` and `web_fetch` with `ERR_MODULE_NOT_FOUND`.

## Best Possible Solution

Extend the package postbuild/build verification so long-lived npm global gateways survive package replacement for the affected lazy chunks, with regression coverage using the known stale filenames.

## Reproduction Assessment

Yes at source/package level, but not as a live Windows run. The high-confidence path is to start from `openclaw@2026.5.4`, where the reported chunks exist, replace the package with `openclaw@2026.5.6`, where they do not, then trigger the lazy `sessions_send`/`web_fetch` path.

## Solution Assessment

No, current main is not yet the best solution because its stable alias logic only covers `.runtime`/`.contract` chunks and selected legacy names.
