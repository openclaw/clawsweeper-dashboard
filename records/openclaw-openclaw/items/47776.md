## Summary

Keep this PR open. Current main still does not implement the intended Unix service-mode cleanup for npm-global `node .../openclaw/... gateway` listeners, and the submitted branch is not safe to merge because the gateway change is ineffective against current main’s Unix PID discovery while unrelated Slack/json changes and a runtime log artifact remain in the PR diff.

## What This Changes

The PR adds a service-mode gateway cleanup pass, changes JSON atomic write retry behavior, adds Slack URL-verification handling in the old core Slack monitor path, and commits a runtime log file.

## Best Possible Solution

Land a focused replacement patch that collects raw Unix listener PIDs for the gateway port, verifies each candidate by argv via `/proc/<pid>/cmdline` on Linux and `ps` on macOS before signaling, wires that into service-mode startup, and adds a regression for `lsof` reporting `cnode` while argv identifies `node .../openclaw/dist/entry.js gateway --port 18789`. Leave the Slack, JSON retry, and log-file changes out of this fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from current main: `lsof -Fpc` output with a `cnode` listener is filtered out by `parsePidsFromLsofOutput`, and service mode only calls that filtered cleanup path before binding.

## Solution Assessment

No. The submitted PR is not the best way to solve the issue because its second pass calls a helper that still starts from command-name-filtered Unix candidates on current main, and the branch bundles unrelated/stale changes; a narrow gateway-only replacement is safer.
