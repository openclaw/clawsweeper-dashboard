Keep open, but do not merge as-is: current main still lacks a gateway startup cleanup path that can see npm-global `node ... openclaw ... gateway` listeners, and the branch bundles unrelated stale Slack/JSON/log changes with a webhook body-limit regression.

## What This Changes

## Best Possible Solution

Land a focused replacement patch for gateway startup cleanup: enumerate raw Unix listener PIDs for the gateway port, verify each candidate by argv, preserve self/ancestor exclusions, wire it into service-mode startup, and add regression coverage for `lsof` reporting `node` while argv identifies an OpenClaw gateway.

Yes. A high-confidence source-level reproduction exists on current main: lsof output with `p<pid>/cnode` is dropped by the command-name filter, and service mode only calls that filtered cleanup path before binding.

## Solution Assessment

No. The submitted branch is not the best fix because its second pass still depends on the filtered Unix candidate source and it bundles unrelated stale Slack/JSON/log changes; a narrow gateway-only replacement is safer.
