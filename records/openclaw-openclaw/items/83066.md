## Summary

Keep open: the reported 60s watchdog failure is plausible from current source, but the patch is not merge-ready because the new public cron config keys are not wired into the strict config schema, and the PR still needs stronger real behavior proof.

## What This Changes

The PR raises isolated cron setup/pre-execution watchdog defaults from 60s to 180s and adds `cron.setupWatchdogMs`/`cron.preExecutionWatchdogMs` runtime fields.

## Best Possible Solution

Ship the timeout-policy change only after either completing the public config contract or dropping the config keys, with focused regression coverage and real evidence from an isolated Codex cron run.

## Reproduction Assessment

Yes, source-reproducible: current main arms 60s setup/pre-execution watchdogs for isolated agent-turn cron jobs, and existing fake-timer tests cover the 60.1s pre-execution timeout path. A live slow-hardware reproduction was not established in this read-only review.

## Solution Assessment

No, not as submitted: raising the defaults is a plausible fix direction, but advertised configurability must be wired through schema/help/docs/tests or removed from the patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Wire the new cron keys into the validated config schema:** `src/config/types.cron.ts:66-72`

A real config containing `cron.setupWatchdogMs` or `cron.preExecutionWatchdogMs` will be rejected before the runtime resolver can see it, so the advertised configurability does not work.
