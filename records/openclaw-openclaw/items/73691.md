## Summary

Keep open: current main still appends memory-core promotion sections to `MEMORY.md` without a durable size budget, and draft PR #74088 is an open, unmerged closing implementation candidate.

## What This Changes

Bound memory-core dreaming promotions so `MEMORY.md` cannot grow beyond bootstrap-safe limits and stall Gateway sessions.

## Best Possible Solution

Review and land PR #74088, or an equivalent memory-core promotion fix that enforces a durable pre-write budget while preserving user-authored `MEMORY.md` content.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The full Gateway freeze path is only partially reproducible without the reported lock-timeout logs.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. A compacting or archival promotion policy is the narrow maintainable direction.
