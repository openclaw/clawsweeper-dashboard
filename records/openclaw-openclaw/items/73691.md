## Summary

Keep open: current `main` still has the unbounded memory-core promotion append path, and PR #74088 is an open draft closing implementation candidate that should be reviewed, landed, or replaced before this issue closes.

## What This Changes

The issue requests a durable bound on `MEMORY.md` growth from memory-core dreaming promotions so long-running installs do not exceed bootstrap context limits or wedge the Gateway.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core change that enforces a pre-write promotion budget, preserves user-authored memory, and includes regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a file-size guard. The exact Gateway freeze path remains unclear without the reported lock-timeout logs.

## Solution Assessment

Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. Budgeted compaction or an archive policy is the narrow maintainable direction.
