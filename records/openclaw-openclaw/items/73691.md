## Summary

Keep open: current main still has the reported unbounded append path, and PR #74088 is an open, draft, unmerged closing implementation candidate that should be reviewed, landed, or replaced before this issue closes.

## What This Changes

The issue requests bounding memory-core dreaming promotions so `MEMORY.md` cannot grow past bootstrap limits and wedge Gateway sessions.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core change that enforces a durable pre-write budget for promotion output, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The exact Gateway freeze path remains unclear without the reported lock-timeout logs.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. A bounded compaction or archive policy is the narrow maintainable path.
