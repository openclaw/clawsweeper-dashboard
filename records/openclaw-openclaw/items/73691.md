## Summary

Keep open: current main still has an append-only memory-core promotion writer with no durable `MEMORY.md` size budget, and the provided GitHub context shows PR #74088 is an open, unmerged closing implementation candidate for the same bug. Repository policy says the issue should remain open until that PR, or an equivalent fix, lands.

## What This Changes

The issue requests bounding, compacting, archiving, or restructuring memory-core dreaming promotions so `MEMORY.md` cannot grow beyond bootstrap budgets and contribute to Gateway stalls.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core change that enforces a durable pre-write budget for promotion output, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The exact Gateway freeze path remains unclear because the report has no lock-timeout log or focused Gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. A bounded compaction or archive policy is the narrow maintainable path; startup-only migration would leave future writes unbounded.
