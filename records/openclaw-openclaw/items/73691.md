## Summary

Keep open because current main still has the unbounded memory-core promotion write path, and the linked closing implementation candidate has not merged.

## What This Changes

The issue requests that dreaming promotions keep `MEMORY.md` within bootstrap-safe bounds instead of growing indefinitely and contributing to Gateway freezes.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core fix that enforces a durable pre-write budget for promotion output, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The exact Gateway freeze path remains unclear because the report has no lock-timeout log or focused Gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. A bounded compaction or archive policy is the narrow maintainable direction; startup-only truncation would leave future writes unbounded.
