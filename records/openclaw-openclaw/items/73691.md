## Summary

Keep open: current main still has the append-only memory-core promotion path without a durable `MEMORY.md` size budget, and the provided GitHub context shows PR #74088 is an open, unmerged implementation candidate that uses closing syntax for this issue.

## What This Changes

Requests bounding or compacting memory-core dreaming promotions so `MEMORY.md` cannot grow past bootstrap limits and contribute to Gateway freezes.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core change that enforces a durable size budget before promotion writes, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections to existing content without a size guard. The exact Gateway freeze mechanism is unclear because the issue has no lock-timeout logs or focused Gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the durable guard belongs in `memory-core`'s promotion writer because bootstrap truncation only caps injected prompt context. The precise compaction/archive policy should be settled in review of PR #74088 or an equivalent replacement.
