## Summary

Keep open: current main still has an unbounded memory-core promotion write to `MEMORY.md`, and open draft PR #74088 is the active closing implementation candidate rather than a merged fix.

## What This Changes

Bound dreaming promotions so `memory-core` does not let `MEMORY.md` grow past bootstrap-size limits.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core promotion change that enforces a durable pre-write budget while preserving user-authored `MEMORY.md` content.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the unbounded-growth root path: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The full Gateway freeze remains only partially reproducible without the reported lock-timeout logs.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected context and does not bound disk growth. A compacting or archival promotion policy is the narrow maintainable direction.
