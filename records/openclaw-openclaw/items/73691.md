## Summary

Keep open: current main still has the source-backed unbounded `MEMORY.md` append path, and open draft PR #74088 is an unmerged closing implementation candidate that should be reviewed, landed, or replaced before this issue closes.

## What This Changes

The issue requests bounding or compacting memory-core dreaming promotions so `MEMORY.md` cannot grow past bootstrap-safe limits and wedge Gateway sessions.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core promotion change that enforces a durable pre-write budget while preserving user-authored `MEMORY.md` content.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The full Gateway freeze remains only partially reproducible without the reported lock-timeout logs.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation caps prompt injection only and does not bound disk growth. A compacting or archival promotion policy is the narrow maintainable direction.
