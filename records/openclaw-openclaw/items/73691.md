## Summary

Keep open: current main still appends memory-core promotion sections to `MEMORY.md` without a durable size budget, while PR #74088 is an open, unmerged closing implementation candidate for the same bug.

## What This Changes

The issue requests that memory-core dreaming/deep promotion keep `MEMORY.md` bounded through compaction, archive, or structured storage so bootstrap/session startup is not overwhelmed by unbounded durable-memory growth.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core change that enforces a pre-write budget for promotion output, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded `MEMORY.md` growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard. The exact Gateway freeze path remains unclear because the report has no lock-timeout log or focused Gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in memory-core's promotion writer because bootstrap truncation only caps injected prompt context and does not bound disk growth. The current issue should stay open until the linked PR or an equivalent bounded compaction/archive implementation lands.
