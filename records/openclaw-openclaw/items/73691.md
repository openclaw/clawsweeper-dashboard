## Summary

Keep open: current `main` still has the append-only memory-core promotion writer with no durable `MEMORY.md` size budget, and the provided GitHub context shows PR #74088 is open, unmerged, and uses closing syntax for this issue. Bootstrap truncation only limits injected context; it does not bound the on-disk memory file that dreaming keeps appending to.

## What This Changes

The issue asks memory-core dreaming promotions to keep `MEMORY.md` within bootstrap-size bounds through compaction, archival, or structured storage so long-running Gateway sessions do not freeze.

## Best Possible Solution

Review and land PR #74088 or an equivalent memory-core change that enforces a durable size budget before promotion writes, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs. Bootstrap truncation alone is not sufficient because it leaves the on-disk file unbounded.

## Reproduction Assessment

Yes for the core unbounded-growth bug: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections to existing `MEMORY.md` without a size guard. The exact Gateway freeze mechanism is unclear because the issue does not include lock-timeout logs or a focused Gateway reproduction.

## Solution Assessment

Yes on boundary: the fix belongs in `memory-core`'s promotion writer because that is where durable growth is introduced. The open PR's compaction approach is a plausible narrow solution, but maintainers should review that patch or an equivalent replacement before closing this issue.
