## Summary

Keep open: current `main` still has the unbounded memory-core promotion append path, and open draft PR #74088 is the active closing implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Bound `MEMORY.md` growth from memory-core dreaming promotions so bootstrap context stays within budget and the Gateway does not freeze after long-running dreaming use.

## Best Possible Solution

Review and land PR #74088, or an equivalent memory-core promotion change that enforces a durable pre-write budget while preserving user-authored `MEMORY.md` content.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for unbounded growth: repeated successful `applyShortTermPromotions` calls with distinct candidates append new promotion sections without a size guard; the full Gateway freeze remains only partially proven without the reported lock-timeout logs.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the durable cap belongs in memory-core's promotion writer because bootstrap truncation only limits injected context and does not bound disk growth.
