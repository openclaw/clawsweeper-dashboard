## Summary

Keep open: this is a focused, still-useful bugfix PR because current main still runs full tuple anti-joins for every same-database legacy memory migration, and no merged replacement covers this row-count fast path. I found no blocking correctness issue, but the change is upgrade-sensitive because it affects the verifier that decides when legacy memory tables can be dropped.

## What This Changes

This PR changes the memory-host SDK legacy memory-index migration to use row-count verification for empty canonical tables, keep full anti-join verification for prefilled canonical tables, and add a regression test for duplicate legacy meta rows.

## Best Possible Solution

Land this focused verifier change after exact-head checks, while keeping the broader legacy sidecar import work in the separate Memory Core migration PR.

## Reproduction Assessment

Yes, source-reproducible: current main copies legacy rows and then verifies each table with full tuple anti-joins on the memory-store open path. I did not run the large-store benchmark locally, but the PR body includes file-backed SQLite timing evidence for the claimed bottleneck.

## Solution Assessment

Yes. The narrow fix is to skip the expensive anti-join only when the canonical table was empty before the copy, while retaining the full conflict check for prefilled canonical tables; broader sidecar migration belongs in the separate Memory Core PR.
