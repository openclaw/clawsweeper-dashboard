## Summary

## What This Changes

The PR wraps llm-core tool argument schema validation failures for unreadable plugin-provided schemas and adds hostile-schema regression tests.

## Best Possible Solution

Keep the validation boundary in llm-core, but make unsupported-schema errors an internal branded type and regression-test that extension getter text cannot reach tool error results.

## Reproduction Assessment

Yes. Current main reads plugin tool schemas directly during validation and forwards thrown validation messages into tool error results; I did not run tests because this review is read-only.

## Solution Assessment

No. Wrapping schema operations in validateToolArguments is the right boundary, but prefix-matching errors is not the best fix because a private/branded unsupported-schema error avoids the raw-message bypass without adding product surface.
