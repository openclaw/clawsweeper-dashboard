## Summary

Keep open: this protected maintainer draft addresses a real LLM-core validation bug, but the current patch is not merge-ready because its unsupported-schema guard still trusts an extension-controlled error-message prefix and overlaps stronger open LLM-core guard branches.

## What This Changes

This PR wraps `validateToolArguments()` schema reads and TypeBox validation operations, then adds regression tests for top-level and nested unreadable tool parameter schemas.

## Best Possible Solution

Land one canonical `validateToolArguments()` guard that uses a private/branded unsupported-schema error and covers dependency-read accessor bypasses, then close or supersede the overlapping LLM-core branches.

## Reproduction Assessment

Yes, from source without executing tests in this read-only review. Current main and `v2026.6.6` pass `tool.parameters` directly into TypeBox conversion/compilation, and TypeBox reads schema keyword properties that can surface proxy or accessor failures.

## Solution Assessment

No. Wrapping schema operations at `validateToolArguments()` is the right boundary, but message-prefix matching is not the best fix because a branded internal error avoids the raw-message bypass without adding product surface.
