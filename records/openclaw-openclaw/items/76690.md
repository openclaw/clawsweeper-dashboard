Keep open: current main still advertises, configures, and tests the Codex OAuth image route while source inspection shows it sends the exact hosted `image_generation` payload that the Windows and macOS reports say the Codex backend rejects.

## What This Changes

The issue reports that the advertised OpenAI Codex OAuth route for `openai/gpt-image-2` reaches the Codex Responses backend but fails with a hosted-tool 400 instead of generating an image or clearly reporting unavailable support.

## Best Possible Solution

Keep the fix provider-local: use a live-supported Codex Responses image payload if one exists; otherwise classify this exact hosted-tool rejection as Codex OAuth image generation unavailable and align docs, tests, and changelog with API-key Images API guidance.
