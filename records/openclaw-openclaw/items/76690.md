## Summary

Keep open: current main still advertises and selects the Codex OAuth image route, then sends the hosted `image_generation` tool payload that two issue reports say the Codex Responses backend rejects.

## What This Changes

The issue asks OpenClaw to make `openai/gpt-image-2` image generation work through Codex OAuth or return a clear unavailable diagnostic instead of the backend hosted-tool 400.

## Best Possible Solution

Keep the fix provider-local: live-verify whether Codex supports any image-generation payload, and if it does not, classify this exact hosted-tool rejection as Codex OAuth image generation unavailable with docs, tests, and changelog aligned.

## Reproduction Assessment

Yes, source-backed but not live-proven here: run the reported `openclaw infer image generate --model openai/gpt-image-2 ...` command under a Codex OAuth-only profile, which current main still routes to the reported hosted-tool payload. Two independent reports provide the raw backend 400, but no local credential was available for a live replay.

## Solution Assessment

No, current main is not the best way to solve it because it continues to promise and send a route that fails with a raw backend 400 in two reports. The narrow maintainable fix is in the OpenAI plugin: use a verified supported Codex payload if one exists, otherwise return a clear unavailable error and steer users to the direct OpenAI Images API route.
