## Summary

Keep open: current main still advertises and selects the OpenAI Codex OAuth image route, then sends the hosted `image_generation` Responses payload that two reports say the Codex backend rejects with HTTP 400.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation with OpenAI Codex OAuth fails even though the OpenAI provider docs advertise that route.

## Best Possible Solution

Keep the fix provider-local: use a live-supported Codex image request shape if one exists, otherwise classify this exact hosted-tool rejection as Codex OAuth image generation unavailable and steer users to the direct OpenAI Images API route.

## Reproduction Assessment

Yes, source-backed: with a Codex OAuth-only profile and `openai/gpt-image-2`, current main selects `openai-codex` and sends the hosted `image_generation` `tool_choice` payload. I did not mark it live reproduced because no usable OpenAI/Codex credential was present here, but the Windows and macOS reports show the same backend 400.

## Solution Assessment

Yes, a provider-local fix is the narrow maintainable path. The repair should verify the live Codex contract, then either switch to a supported payload or turn this exact rejection into a clear unavailable error with docs/tests/changelog aligned.
