## Summary

Keep open: current main still advertises and selects Codex OAuth image generation, then sends the hosted `image_generation` tool-choice payload that two reports say the Codex Responses backend rejects.

## What This Changes

Fix `openai/gpt-image-2` image generation under OpenAI Codex OAuth, or clearly report that this route is unavailable instead of surfacing the backend tool-choice error.

## Best Possible Solution

Keep the fix provider-local: use a verified supported Codex image payload if one exists; otherwise classify this exact hosted-tool rejection as Codex OAuth image generation unavailable and align docs, tests, and changelog.

## Reproduction Assessment

Yes, source-backed: under an OAuth-only `openai-codex` profile, current main routes `openai/gpt-image-2` to the same Codex Responses hosted-tool payload reported by Windows and macOS users. I did not mark it live reproduced because this read-only review found no local OpenAI/Codex credential.

## Solution Assessment

Unclear until the live provider contract is checked. The narrow maintainable solution is in the OpenAI plugin: switch to a supported Codex payload or turn this exact 400 into a clear unavailable error with direct OpenAI Images API guidance.
