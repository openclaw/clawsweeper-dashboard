## Summary

Keep open: current `main` and the latest release still advertise and select the Codex OAuth image route, then send the hosted `image_generation` Responses payload that two reports say the Codex backend rejects.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation through OpenAI Codex OAuth reaches the Codex Responses backend but fails with `Tool choice 'image_generation' not found in 'tools' parameter`.

## Best Possible Solution

Keep the repair provider-local: either switch to a live-supported Codex image request shape, or classify this exact hosted-tool rejection as Codex OAuth image generation unavailable and point users to the direct OpenAI Images API route.

## Reproduction Assessment

Yes, as a source-backed path: current `main` still routes an OAuth-only `openclaw infer image generate --model openai/gpt-image-2 ...` request to the reported Codex hosted-tool payload. I did not mark it live reproduced because no usable Codex credential was available here.
