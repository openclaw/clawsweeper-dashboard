## Summary

Keep open: current main and the latest release still document and route `openai/gpt-image-2` through Codex OAuth using the same hosted `image_generation` Responses request family, while the issue has two matching user reports and no current live proof that the backend now accepts that route.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation through OpenAI Codex OAuth reaches the Codex Responses backend but fails because the backend rejects the hosted `image_generation` tool.

## Best Possible Solution

Run a live current-release Codex OAuth image-generation smoke, then either keep the documented route with proof or add a targeted unavailable diagnostic plus docs/tests for accounts where the hosted tool is rejected.

## Reproduction Assessment

No high-confidence live current-main reproduction is available here. Source inspection shows current main and the latest release still send the reported request family, and two users supplied matching live failures on OpenClaw `2026.5.2`.

## Solution Assessment

Unclear. Catching this 400 and pointing users to direct `OPENAI_API_KEY` may be a good mitigation, but maintainers first need to decide whether Codex OAuth image generation remains a supported route.
