Keep open: current main still advertises and selects the Codex OAuth image route, then sends the same hosted `image_generation` tool payload that the issue and macOS confirmation say the Codex Responses backend rejects.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation fails when OpenClaw routes it through OpenAI Codex OAuth instead of the public OpenAI API-key Images API path.

## Best Possible Solution

Keep the repair provider-local: use a live-supported Codex image request shape if one exists; otherwise classify this exact hosted-tool rejection as Codex OAuth image generation unavailable and align the OpenAI docs, image-generation docs, regression tests, and changelog.
