Keep open: current main still documents Codex OAuth image generation for `openai/gpt-image-2` and still sends the Codex Responses hosted `image_generation` request shape that the Windows report and macOS confirmation say the backend rejects.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation through OpenAI Codex OAuth fails with `Tool choice 'image_generation' not found in 'tools' parameter` instead of succeeding or reporting the route as unavailable.

## Best Possible Solution

Keep the documented Codex OAuth route only if a live-supported request shape can be verified; otherwise add OpenAI-provider-local handling for this exact hosted-tool rejection, surface clear API-key Images API guidance, and align docs, tests, and changelog.
