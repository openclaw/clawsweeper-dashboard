## Summary

Keep open: current main still advertises and selects the Codex OAuth image route, then sends the hosted `image_generation` tool payload that the Windows and macOS reports say the Codex Responses backend rejects.

## What This Changes

The issue reports that `openclaw infer image generate --model openai/gpt-image-2` fails under Codex OAuth because the Codex Responses backend rejects the hosted `image_generation` tool payload.

## Best Possible Solution

Keep the fix provider-local: use a live-supported Codex image request shape if one exists; otherwise classify this hosted-tool rejection as Codex OAuth image generation unavailable and steer users to the direct OpenAI Images API route.

## Reproduction Assessment

Yes, as a source-backed path: run the reported `openclaw infer image generate --model openai/gpt-image-2 ...` command under a Codex OAuth-only profile, which current main still routes to the reported hosted-tool payload. I did not mark it live reproduced because no usable OpenAI/Codex credential was available in this review environment.

## Solution Assessment

No, current main is not the best way to solve it because it continues to promise and send a route that fails with a raw backend 400 in two reports. The narrow maintainable solution is in the OpenAI plugin, with docs/tests/changelog aligned to the verified backend behavior.
