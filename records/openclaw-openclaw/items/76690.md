## Summary

Keep open: current main still advertises and selects Codex OAuth image generation, then sends the same hosted `image_generation` tool payload that the Windows report and macOS confirmation say the Codex Responses backend rejects.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation with OpenAI Codex OAuth fails on v2026.5.2 even though the OpenAI provider docs promise that route.

## Best Possible Solution

Keep the Codex OAuth image route only if a live-supported request shape can be verified; otherwise handle the exact hosted-tool rejection in the OpenAI provider, guide users to the API-key Images API route, and align docs, tests, and changelog.

## Reproduction Assessment

No live current-main reproduction was established here because no usable OpenAI/Codex auth was present and `pnpm` was unavailable. Source inspection plus two precise user reports gives a high-confidence path through the same route and payload on current main.

## Solution Assessment

Unclear until the live provider contract is checked. The narrow maintainable fix is provider-local: use a verified Codex request shape if one exists, or classify this exact 400 as Codex OAuth image generation unavailable without silently falling back to API-key routing.
