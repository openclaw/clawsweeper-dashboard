## Summary

Keep open: current main and the latest release still advertise and send the Codex OAuth image-generation route, while two users report the backend rejecting that hosted tool route and no merged/current proof shows the failure is resolved.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation via OpenAI Codex OAuth reaches the Codex Responses backend but fails when the backend rejects the hosted `image_generation` tool choice.

## Best Possible Solution

Run a live current-release Codex OAuth image-generation smoke, then either keep the documented route with proof or add a targeted unavailable diagnostic plus docs/tests for accounts where the hosted tool is rejected.

## Reproduction Assessment

No high-confidence live current-main reproduction is available here. Source inspection shows current main and `v2026.6.8` still send the reported request family, and two users supplied matching live failures on OpenClaw `2026.5.2`.

## Solution Assessment

Unclear. Catching this 400 and pointing users to direct `OPENAI_API_KEY` may be a good mitigation, but maintainers first need to decide whether Codex OAuth image generation remains a supported route.
