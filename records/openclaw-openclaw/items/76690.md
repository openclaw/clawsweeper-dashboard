## Summary

Keep open: current main still documents Codex OAuth image generation as supported and routes Codex-only OpenAI image requests to the same hosted `image_generation` Responses payload that two users report the ChatGPT Codex backend rejects. The related open issue at https://github.com/openclaw/openclaw/issues/81151 is adjacent, but it also covers a different 401/scope failure and `gpt-5.5-pro` fallback behavior, so it is not a clean duplicate.

## What This Changes

The issue reports that `openclaw infer image generate --model openai/gpt-image-2` fails under Codex OAuth-only auth because the Codex Responses backend rejects the hosted `image_generation` tool.

## Best Possible Solution

Keep the fix provider-local: verify the current Codex Responses image contract, then either send a supported payload or classify this exact rejection as an unavailable Codex image route with docs pointing users to direct OpenAI Images API auth.
