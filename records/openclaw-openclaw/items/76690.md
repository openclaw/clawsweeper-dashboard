## Summary

Keep open: current main still documents and selects the Codex OAuth image route, then sends the hosted `image_generation` tool payload that the Windows and macOS reports say the Codex Responses backend rejects. The newer related issue is useful adjacent context, but it reports a different 401/scope failure plus `gpt-5.5-pro` fallback behavior, so it should not absorb this exact 400 report yet.

## What This Changes

`openai/gpt-image-2` image generation with only OpenAI Codex OAuth should either succeed or return a clear unavailable-route diagnostic instead of surfacing the raw hosted-tool 400.

## Best Possible Solution

Keep the fix provider-local: verify whether Codex Responses currently supports image generation with Codex OAuth, and if not, classify this exact rejection as an unavailable route with clear docs pointing users to the direct OpenAI Images API path.
