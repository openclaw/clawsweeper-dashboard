## Summary

Keep open: current main still lacks the `message` eager-warmup skip and still drops Discord/Telegram channel-declared execution modes at the channel wrapper layer, so the PR targets a real gap, but the new `resolveExecutionMode` wrappers need a narrow return-type fix before merge.

## What This Changes

The PR skips eager model-context warmup for `openclaw message`, forwards Discord and Telegram message-action execution modes through their channel wrappers, and threads caller-owned manifest registry metadata into provider-policy defaulting.

## Best Possible Solution

Land the latency fix after tightening the two execution-mode wrappers so they always return the declared mode and after the targeted tests plus changed gate pass.

## Reproduction Assessment

Yes. Current-main inspection reproduces the functional gap by showing `message` is not skipped for eager warmup and the channel wrappers do not expose `resolveExecutionMode`; the PR's type issue is reproducible through strict TypeScript checking of the new wrapper return expressions.

## Solution Assessment

Mostly yes. Skipping warmup at the primary `message` command and forwarding plugin-owned execution modes is the narrow maintainable direction, but the wrapper implementation should return an explicit fallback mode before merge.
