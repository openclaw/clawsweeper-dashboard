## Summary

Keep open: the fix direction is useful, but the PR is not merge-ready because it conflates RPC auth presentation with host remoteness and can regress provider auth flows that already handle RPC-visible URLs differently. The external proof is also not inspectable yet.

## What This Changes

The branch adds a `presentsAuthChallenge` prompter capability, includes OAuth authorization URLs in provider wizard prompts, marks RPC wizard sessions as auth-presenting, and adds focused tests.

## Best Possible Solution

Keep host locality and auth-presentation separate: only the specific OAuth flows that need in-band URL delivery should opt into manual/auth-presenting behavior, then add inspectable Windows companion or RPC wizard proof.

## Reproduction Assessment

Yes. Source inspection shows current main can expose only the redirect input as a wizard text step while placing the authorization URL in logs or gateway browser behavior; the PR-introduced xAI/local RPC regression is also traceable from source.

## Solution Assessment

No, not as submitted. Carrying the URL in protocol-visible prompt text is the right narrow direction, but using `isRemote` as the presentation capability is too broad; preserve host locality and use `presentsAuthChallenge` only where needed.
