## Summary

Keep open: the PR targets the right Discord bug boundary, but it still leaves successful component-style and voice sends with `threadName` silently ignored, and the supplied proof is mocked test output rather than real behavior proof from an OpenClaw/Discord run.

## What This Changes

The PR forwards `threadName` through the Discord message send adapter, adds a warning for the regular `sendMessage` runtime path, and adds focused adapter/runtime tests.

## Best Possible Solution

Keep the warning-based fix in the Discord plugin, but attach the same warning to every successful `sendMessage` branch that accepts `threadName`, with focused tests for the covered send variants and redacted real behavior proof.

## Reproduction Assessment

Yes. Current main exposes `threadName` in the message-tool schema, while the Discord `send` adapter/runtime do not consume, rename, or warn on it for the reported send path; I verified this by source inspection rather than a live Discord call.

## Solution Assessment

No. Returning an explicit warning is a narrow maintainable fix, but this PR currently adds it only to the regular send return and misses component-spec and voice send returns that can also carry `threadName`.
