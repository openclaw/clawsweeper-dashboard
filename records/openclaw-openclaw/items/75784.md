## Summary

Keep open. Current main still has a concrete WebChat transcript hygiene gap: the history projection hides empty rows and heartbeat prompt/OK pairs, but it does not inspect user-message provenance, and orphan-recovery resume prompts are still sent through `agent` as ordinary user text without `inputProvenance`. The related closed heartbeat/system-event reports cover older pollution paths, but this report adds a still-actionable subagent recovery and provenance-display case.

## What This Changes

The issue requests that runtime-generated heartbeat, gateway restart, session repair, and subagent recovery prompts stop appearing in WebChat as user-authored messages.

## Best Possible Solution

Add a narrow display contract for runtime-generated user-role rows: orphan-recovery resume, subagent announce/progress, heartbeat/system-event, and repair-only handoff prompts should carry structured provenance and then be hidden or rendered as internal/system events in WebChat while preserving normal external user messages.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction can construct `chat.history` raw messages containing a user-role inter-session or orphan-recovery prompt and verify `projectRecentChatDisplayMessages` currently leaves it visible unless it matches the older heartbeat/empty-row filters.

## Solution Assessment

No, current main is not a complete solution. The maintainable fix is to tag the missing orphan-recovery resume path with provenance and extend the WebChat projection/tests to use that provenance rather than matching arbitrary prompt text.
