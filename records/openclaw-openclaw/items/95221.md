## Summary

Keep this PR open: the direct EADDRNOTAVAIL guard is useful, but the reported probe-forced fallback path can still discard the original socket error and emit the misleading pinned-IP fallback warning; the updated proof also shows gateway startup, not the changed failure path.

## What This Changes

This PR adds an EADDRNOTAVAIL fail-fast diagnostic to Telegram fetch fallback classification and a regression test that expects no retry or misleading DNS fallback log for that direct fetch error.

## Best Possible Solution

Carry the original probe fetch error into fallback promotion, or gate forced fallback through the same local-socket-allocation decision, then add focused probe coverage and redacted real transport proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level: the canonical issue's `codes=none, reason=probe timeout/network error` log matches probeTelegram calling forceFallback without the original fetch error. I did not run a live socket-exhaustion Telegram repro in this read-only review.

## Solution Assessment

No: the classifier guard is a useful part of the fix, but it is not the best complete solution because the probe forceFallback boundary still discards the original local socket allocation error. The safer fix carries the original error into that decision while preserving retry for non-EADDRNOTAVAIL network failures.
