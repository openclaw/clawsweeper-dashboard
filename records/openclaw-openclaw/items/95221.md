## Summary

Keep open: the EADDRNOTAVAIL classifier is in the right owner boundary, but the new forceFallback guard also disables existing Telegram probe timeout IPv4 fallback because ordinary timeout errors are not accepted by the fetch fallback classifier.

## What This Changes

The PR adds EADDRNOTAVAIL detection to Telegram fetch fallback, changes TelegramTransport.forceFallback to accept an optional error, threads probe errors into forceFallback, and updates focused tests.

## Best Possible Solution

Keep the narrow EADDRNOTAVAIL fail-fast classifier, but make forced fallback preserve existing timeout and manual recovery unless the original error is specifically a local socket allocation failure; then add regression proof for both paths.

## Reproduction Assessment

Yes, source-level. On PR head, probeTelegram forwards caught errors into forceFallback, and the new guard rejects ordinary timeout errors because shouldUseTelegramTransportFallback only accepts internal unhealthy errors, configured fallback codes, or a code-less fetch failed envelope.

## Solution Assessment

No. The EADDRNOTAVAIL classifier belongs in Telegram transport, but the forced-fallback gate is too broad and should preserve existing timeout fallback while suppressing only local socket allocation failures.
