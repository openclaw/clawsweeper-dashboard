## Summary

Keep open: the direct EADDRNOTAVAIL classifier is useful, but the PR head still promotes forced fallback unconditionally, so probe-driven EADDRNOTAVAIL can still advance through Telegram fallback attempts. The supplied proof also does not cover the latest forced-fallback path.

## What This Changes

This PR adds Telegram EADDRNOTAVAIL fail-fast handling in fetch fallback classification, threads probe errors into forceFallback, and updates focused fetch/probe tests.

## Best Possible Solution

Gate both automatic fetch fallback and forced probe/client fallback on the same EADDRNOTAVAIL classifier, preserve valid remote-network fallback cases, and land one canonical Telegram transport fix with live or equivalent probe proof.

## Reproduction Assessment

## Solution Assessment

No. The direct fetch classifier is the right layer, but the current forceFallback change is only a partial fix because it preserves unconditional promotion; the same gate needs to exist at the forced promotion boundary.
