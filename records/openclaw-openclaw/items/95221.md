## What This Changes

## Best Possible Solution

Carry the original probe fetch error, or an equivalent local-socket-allocation decision, through the forced fallback boundary so `EADDRNOTAVAIL` skips every automatic IP-rotation path while valid timeout and IPv4 recovery fallback still work.

## Reproduction Assessment

Yes from source: current main has a probe retry catch path that calls `forceFallback` without the original error, and `forceFallback` logs/promotes with a synthetic error. That source path matches the reported `codes=none` misleading fallback log without requiring live socket exhaustion.

## Solution Assessment

No. The direct-fetch classifier guard is useful, but the best fix must also gate probe-forced fallback on the original `EADDRNOTAVAIL` error because that is the path named by the reported log.
