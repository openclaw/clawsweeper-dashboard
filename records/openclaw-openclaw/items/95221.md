## Summary

Keep this PR open: it adds a useful direct EADDRNOTAVAIL classifier guard, but the reported startup-probe path can still force the pinned-IP fallback after losing the original error, so the central bug is not fully fixed.

## What This Changes

The PR changes Telegram fetch fallback classification so EADDRNOTAVAIL logs a local socket allocation diagnostic and skips remote IP fallback, with a focused regression test.

## Best Possible Solution

Carry the original probe fetch error, or an equivalent local-socket-allocation decision, through the probe and forced-fallback boundary so all EADDRNOTAVAIL-driven automatic IP rotation is skipped while timeout and remote reachability fallbacks remain intact.

## Reproduction Assessment

Yes, at source level: the reported `codes=none, reason=probe timeout/network error` log matches `probeTelegram` forcing fallback without passing the caught error to `forceFallback`. I did not run a live socket-exhaustion Telegram repro in this read-only review.

## Solution Assessment

No: the direct fetch classifier is a useful part of the fix, but it is not the best complete solution because startup probing still discards the original local socket allocation error before fallback promotion.
