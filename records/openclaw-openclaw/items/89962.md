## Summary

Keep open for maintainer review: this is a focused Discord message-loss fix with sufficient real Discord proof and no concrete patch defect found, but it intentionally changes the delivery policy after a voice-send rejection.

## What This Changes

The PR changes Discord outbound `audioAsVoice` payload delivery so rejected voice sends can fall back to visible text or TTS supplement text, with regression tests for fallback text, reply targeting, and duplicate suppression.

## Best Possible Solution

Land this shape if maintainers want Discord final replies to prefer visible text over total loss after voice-send rejection; otherwise narrow the catch to failure classes known to happen before Discord accepts the voice message.

## Reproduction Assessment

Yes, at source level: current main awaits Discord `sendVoice` in the `audioAsVoice` payload branch before any visible text send, so a rejected voice send aborts the payload. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, this is the right owner boundary and a focused fix for the final payload path, not a duplicate of the related unavailable-adapter/media-attachment PR. The only unsettled part is maintainer acceptance of the broad voice-failure-to-text fallback policy.
