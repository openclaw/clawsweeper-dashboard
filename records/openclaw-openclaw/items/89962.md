## Summary

## What This Changes

The PR wraps Discord `audioAsVoice` payload delivery so rejected voice sends can fall back to visible text or TTS supplement text, with regression tests for fallback text, reply targeting, and duplicate suppression.

## Best Possible Solution

Land this shape if maintainers want Discord final replies to preserve visible answers after voice-send rejection; otherwise narrow fallback to failure classes known to happen before Discord accepts the voice message.

## Reproduction Assessment

Yes, at source level: current main awaits Discord voice delivery before any visible text path, so a rejected `sendVoice` aborts the payload send. The contributor also supplied after-fix live Discord output for the patched path.

## Solution Assessment

Yes for the reported failure: the Discord payload layer owns the voice-first send, and the PR preserves reply targets plus TTS supplement behavior. The remaining question is policy, namely whether the fallback should catch all voice-send rejections or only known pre-send failures.
