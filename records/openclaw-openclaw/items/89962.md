## Summary

Keep open for maintainer review: current main and v2026.6.9 still await Discord voice delivery before any text fallback, while this PR adds focused fallback code, regression coverage, and live-output proof. The remaining decision is whether maintainers accept the Discord message-delivery policy change around catch-all voice-send failures.

## What This Changes

The PR changes Discord outbound payload delivery so failed `audioAsVoice` sends can fall back to visible text or TTS supplement text, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.9 await `sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, and the PR body includes after-fix live output showing forced voice conversion failure followed by a fetched text fallback message.

## Solution Assessment

Yes, if maintainers accept the fallback policy. The change is in the Discord outbound payload owner and reuses the existing TTS supplement contract; the safer alternative is to limit fallback to known pre-send failures if ambiguous duplicate delivery is unacceptable.
