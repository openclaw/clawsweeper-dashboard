## Summary

Keep open: current main and the latest release still await Discord voice delivery before any text fallback in the `audioAsVoice` payload branch, while this PR adds a focused fallback path with regression tests and sufficient live Discord output. I found no blocking patch defect, but maintainers still need to accept the message-delivery policy change and the small duplicate-delivery ambiguity before merge.

## What This Changes

The PR changes Discord `audioAsVoice` final payload delivery to catch rejected voice sends and send visible text or TTS supplement text as a fallback, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.9` await `sendContext.sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, so a rejected voice send exits the payload path.

## Solution Assessment

Yes, if maintainers accept the fallback policy. The change is in the Discord outbound payload owner and reuses the existing TTS supplement contract; the safer alternative is to limit fallback to known pre-send failures if ambiguous duplicate delivery is unacceptable.
