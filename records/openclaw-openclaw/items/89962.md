## Summary

Keep open for maintainer review: the PR fixes a real Discord message-loss path that is still present on current main and the latest release, but it intentionally changes failed voice-note final replies into visible text fallback and needs maintainer acceptance of that delivery policy.

## What This Changes

This PR wraps Discord `audioAsVoice` final payload delivery so rejected voice-note sends fall back to visible text or TTS supplement text, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 await `sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, so a rejected voice send exits the payload path; I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback policy. The change is in the Discord outbound payload owner and reuses the existing TTS supplement contract; the safer alternative is to limit fallback to known pre-send failures if ambiguous duplicate delivery is unacceptable.
