## Summary

Keep open for maintainer review: current main and the latest release still lack a text fallback after a rejected Discord `audioAsVoice` voice send, and this PR is focused with real Discord proof, but merging it intentionally changes Discord message-delivery behavior.

## What This Changes

The PR changes the Discord outbound payload path so failed `audioAsVoice` voice-note delivery can fall back to visible text or hidden TTS supplement text, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 await `sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, so a rejected voice send exits the payload path. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback policy; the change is in the Discord outbound payload owner and reuses the existing TTS supplement contract. The safer alternative is to limit fallback to known pre-send failures if duplicate delivery after ambiguous voice-send failures is unacceptable.
