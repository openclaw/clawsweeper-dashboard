## Summary

Keep open for maintainer review: current main and the latest release still await Discord `sendVoice` before any text fallback in this final-payload path, and the PR is a focused fix with sufficient live-output proof. The remaining decision is whether maintainers accept visible text fallback, and its duplicate-delivery ambiguity, after a rejected voice-note send.

## What This Changes

The PR changes Discord outbound payload handling so rejected `audioAsVoice` voice delivery can fall back to visible text or hidden TTS supplement text, with regression tests.

## Best Possible Solution

Keep the Discord-owned fallback and tests if maintainers accept text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, at source level: current main awaits `sendContext.sendVoice` in the Discord `audioAsVoice` payload branch before any fallback text can be sent, so a rejected voice send exits the payload path. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the delivery policy; the implementation is in the Discord outbound payload owner and reuses the existing reply, retry, and TTS supplement contracts. The safer alternative is to limit fallback to known pre-send failures if duplicate delivery after ambiguous voice-send failures is unacceptable.
