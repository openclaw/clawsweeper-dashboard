## Summary

Keep open: current main and the latest release still await Discord voice delivery before any text fallback, so the branch addresses a real remaining message-delivery gap. The patch looks focused and proof-backed, but merge should stay with maintainer review because it intentionally changes failed voice-note final replies into visible text fallback.

## What This Changes

The PR adds Discord `audioAsVoice` fallback handling so a rejected first voice-note send can post visible text or hidden TTS supplement text, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, source-reproducible: current main awaits `sendVoice` before later text delivery in the Discord `audioAsVoice` payload branch, so a rejected first voice send exits before fallback text. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes if maintainers accept the delivery policy; the change is in the Discord outbound payload owner and reuses the existing TTS supplement contract. The safer alternative is to narrow fallback to known pre-send failures if ambiguous Discord request failures must remain hard errors.
