## Summary

Keep open: this is a focused Discord message-loss bug fix with convincing proof and regression coverage, and current main plus v2026.6.8 still have the no-fallback voice-send path. The remaining blocker is maintainer acceptance of the intentional fallback policy and duplicate-delivery ambiguity.

## What This Changes

The branch wraps Discord `audioAsVoice` final-payload voice delivery so rejected voice sends fall back to visible text or hidden TTS supplement text, with regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text as recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send voice failures.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 await `sendVoice` before sending visible text in the Discord `audioAsVoice` payload branch, so a rejected voice send exits before text delivery. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback policy; the change is in the Discord outbound payload owner and reuses the existing TTS supplement contract. The safer alternative is to limit fallback to known pre-send failures if duplicate delivery after ambiguous voice-send failures is unacceptable.
