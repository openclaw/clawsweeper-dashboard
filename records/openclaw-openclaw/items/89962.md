## Summary

Keep open for maintainer review: current main can lose a Discord final reply when voice delivery rejects, and this PR appears to address that bug with a scoped fallback, focused tests, and real Discord proof. The remaining maintainer question is whether the text fallback after retry-exhausted voice failure is the intended delivery policy.

## What This Changes

The PR changes the Discord `audioAsVoice` payload path to send visible fallback text, including TTS supplement text, when voice-note delivery rejects, and adds regression coverage for reply targeting and duplicate suppression.

## Best Possible Solution

Land a scoped Discord payload fallback after maintainer acceptance of the retry-exhausted voice-to-text policy and normal CI on the exact head.

## Reproduction Assessment

Yes. Source inspection of current main shows the payload path awaits voice delivery before text delivery, so a rejected `sendVoice` prevents the visible reply; the PR body also supplies after-fix live Discord output.

## Solution Assessment

Yes. The fix is in the Discord payload owner path, reuses the existing send context and retry behavior, preserves reply fanout, and leaves direct media sending out of scope where the PR does not claim to change it.
