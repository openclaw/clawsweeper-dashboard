## Summary

Keep open for maintainer review: current main and v2026.6.8 still let a rejected Discord `audioAsVoice` payload voice send abort before text fallback, and this PR is a focused Discord-plugin fix with sufficient real Discord proof; the remaining decision is accepting text fallback and duplicate-delivery policy after voice-send rejection.

## What This Changes

The PR changes Discord `sendPayload` handling so failed first `audioAsVoice` voice delivery falls back to visible text or hidden TTS supplement text, with regression tests for reply targeting and duplicate-suppression behavior.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow fallback to known pre-send failures.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 await `sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, so a rejected voice send exits the payload path. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback policy; the change is in the Discord outbound payload owner and reuses the existing TTS supplement contract. The safer alternative is to limit fallback to known pre-send failures if duplicate delivery after ambiguous voice-send failures is unacceptable.
