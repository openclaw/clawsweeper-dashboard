## Summary

Keep open for maintainer review: the patch is a focused Discord bug fix with sufficient real behavior proof and no concrete code-review findings, but it intentionally changes message-delivery behavior after a rejected voice send.

## What This Changes

The PR changes Discord `audioAsVoice` payload delivery so a rejected voice send can fall back to visible text or TTS supplement text, with focused regression tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send failures.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.9 await `sendVoice` before any text fallback in the Discord `audioAsVoice` payload branch, so a rejected voice send exits the payload path.

## Solution Assessment

Yes, if maintainers accept the fallback policy. The change is in the Discord outbound payload owner and reuses the existing TTS supplement contract; the safer alternative is to limit fallback to known pre-send failures if ambiguous duplicate delivery is unacceptable.
