## Summary

Keep open: current main and v2026.6.8 still await Discord `audioAsVoice` final-payload voice delivery before any visible text can send, while this PR adds a focused fallback and regression tests. The remaining question is maintainer acceptance of the Discord message-delivery policy, not a concrete patch defect.

## What This Changes

The PR changes the Discord outbound payload path so rejected `audioAsVoice` voice-note delivery can fall back to visible text or hidden TTS supplement text, with focused tests.

## Best Possible Solution

Land the Discord-owned fallback if maintainers accept visible text recovery for rejected voice-note final replies; otherwise narrow the catch to known pre-send conversion or runtime failures.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 await `sendVoice` before sending visible text in the Discord `audioAsVoice` payload branch, so a rejected voice send exits before text delivery. I did not run a live current-main Discord repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback policy; the change is in the Discord outbound payload owner and reuses the existing TTS supplement contract. The safer alternative is to limit fallback to known pre-send failures if duplicate delivery after ambiguous voice-send failures is unacceptable.
