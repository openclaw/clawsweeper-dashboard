## Summary

Keep open: current main and latest stable still suppress `context_management` on the `openai-chatgpt-responses` path, and the exact companion fix PR is closed unmerged, so the remaining maintainer question is whether to emit the payload or diagnose unsupported config.

## What This Changes

The issue reports that `responsesServerCompaction` and `responsesCompactThreshold` are read for OpenAI/Codex Responses sessions but not emitted as `context_management` on outbound `openai-chatgpt-responses` payloads.

## Best Possible Solution

Define the ChatGPT/Codex Responses server-compaction contract once, then either wire the existing config into the outbound payload with focused regression coverage or surface a clear runtime/config diagnostic when the route cannot honor it.

## Reproduction Assessment

Yes for the source-level payload gap: current main still reads and documents the compaction config, but the ChatGPT Responses payload policy cannot emit `context_management` on this route. No live long-session reproduction was run in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the ChatGPT/Codex Responses contract. The narrow payload-policy repair is plausible, but sibling Codex source uses separate compaction mechanisms and no `context_management` field, so a diagnostic may be the safer end state if that field is unsupported.
