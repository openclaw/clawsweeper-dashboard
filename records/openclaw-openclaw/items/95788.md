## Summary

Keep open: current main and the latest release still build the ChatGPT Responses payload with server-compaction emission disabled, while related merged work addressed auth/preflight routing rather than the outbound `context_management` gap.

## What This Changes

The issue asks OpenClaw to either emit configured OpenAI Responses server compaction settings on the `openai-chatgpt-responses` request path or clearly diagnose that those settings are unsupported there.

## Best Possible Solution

Define the ChatGPT Responses server-compaction contract once, then either wire supported config into the outbound payload with focused tests or surface a clear runtime/config diagnostic when the route cannot honor it.

## Reproduction Assessment

Yes for the emitted-payload bug from source: current main reads the compaction config for preflight but the Responses builder cannot emit `context_management`. No live long-session reproduction was run in this review.

## Solution Assessment

Unclear until maintainers confirm the route contract. The missing payload is clear, but upstream Codex currently has no `context_management` request field and uses a store-disabled OpenAI route, so a diagnostic may be safer than forcing emission.
