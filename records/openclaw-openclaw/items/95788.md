## Summary

Keep open: current main still has the reported split-brain behavior where the compaction threshold config can be read for OpenAI models, but the ChatGPT/Codex Responses payload path has no proven route that emits `context_management` or tells the user it is unsupported.

## What This Changes

The issue asks OpenClaw to honor or explicitly diagnose `responsesServerCompaction` and `responsesCompactThreshold` on the `openai-chatgpt-responses` / Codex OAuth request path.

## Best Possible Solution

Define the Codex Responses compaction contract, then either emit a supported `context_management` payload with focused source and live proof, or add a clear diagnostic/docs/preflight guard so users do not believe a suppressed threshold is active.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main reads `responsesCompactThreshold` in the compaction gate but the ChatGPT/Codex Responses payload policy cannot emit `context_management`. I did not run a live current-main Codex/OAuth long-session canary in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the Codex backend contract. The maintainable fix is not merely passing the params through; it must either prove the payload is valid for the ChatGPT/Codex route or explicitly diagnose that the config is direct OpenAI Responses only.
