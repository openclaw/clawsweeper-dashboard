## Summary

Keep open: current main and v2026.6.10 still read the OpenAI Responses compaction config but do not emit `context_management` for the `openai-chatgpt-responses` path, and the exact companion fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to either emit configured OpenAI Responses `context_management` for Codex-backed `openai-chatgpt-responses` sessions or clearly diagnose that the config is unsupported on that route.

## Best Possible Solution

Define the ChatGPT/Codex Responses compaction contract once, then either wire the configured threshold into the outbound payload with focused regression coverage or add a clear runtime/config diagnostic when the route cannot honor it.

## Reproduction Assessment

Yes for the source-level behavior: current main reads `responsesServerCompaction` and `responsesCompactThreshold`, but the `openai-chatgpt-responses` payload policy still cannot enable `context_management` under the current store gate. I did not run a live long-session reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the ChatGPT/Codex Responses contract. The narrow emission fix proposed in the closed companion PR is plausible, but sibling Codex source currently has separate compaction paths and no `context_management` request field, so a diagnostic may be the safer product answer.
