## Summary

Keep open: current main still reads the server-compaction config but the ChatGPT Responses payload policy still cannot emit `context_management` on that path, and the exact companion fix PR is closed unmerged.

## What This Changes

The issue reports that `responsesServerCompaction` and `responsesCompactThreshold` are accepted for OpenAI/Codex Responses sessions but are silently suppressed before the outbound ChatGPT Responses payload is sent.

## Best Possible Solution

Define the ChatGPT/Codex Responses server-compaction contract once, then either wire the configured threshold into the outbound payload with focused tests or emit a clear diagnostic when the route cannot honor it.

## Reproduction Assessment

Yes for the source-level issue: current main reads the compaction config but the ChatGPT Responses payload policy still blocks `context_management` emission. No live long-session reproduction was run in this read-only pass.

## Solution Assessment

Unclear until maintainers confirm the ChatGPT/Codex Responses contract. The narrow repair is likely in the payload policy/wrapper tests, but a diagnostic may be safer if that backend intentionally rejects the field.
