## Summary

Keep open: current main still has the reported Linux gateway lifecycle routing path, where effective root plus a non-root SUDO_USER is treated as sudo-to-root and sent to the sudo user's systemd user manager before direct root --user handling. The related open system-unit PR targets /etc/systemd/system units and does not resolve this root-owned systemd user-service stale-env case.

## What This Changes

The issue reports that gateway stop/start/status misroute root-owned Linux systemd user-service actions when a root shell still has stale non-root SUDO_* variables.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that targets root's direct --user manager when the effective service context is root's user unit, with regression coverage that preserves the intentional sudo paths.

## Reproduction Assessment

Yes at source level: current main deterministically routes effective root plus non-root SUDO_USER to --machine <sudo-user>@ --user before direct root --user handling, and the reporter provided concrete live Linux logs plus an env-unset workaround. I did not execute a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with tests is the best direction. It should not become a new config option or broad system-scoped unit implementation, though it should avoid regressing the existing sudo behavior.

## Repair Work Prompt

Fix the Linux gateway systemd user-scope resolver for stale sudo variables in a root-owned user-service context. Current main treats effective uid 0 plus a non-root SUDO_USER as sudo-to-root and sends gateway stop/restart/status/is-enabled to `systemctl --machine <sudo-user>@ --user`, even when USER=root, LOGNAME=root, HOME=/root, XDG_RUNTIME_DIR=/run/user/0, and the managed unit path is `/root/.config/systemd/user/openclaw-gateway.service`. Keep the fix narrow in `src/daemon/systemd.ts`: target root's direct `systemctl --user` manager for the root user-service context, preserve intentional `sudo openclaw ...` and `sudo -u ...` behavior covered by https://github.com/openclaw/openclaw/pull/72655, and do not add a config option or implement system-scoped unit support from https://github.com/openclaw/openclaw/pull/57276. Add focused regressions in `src/daemon/systemd.test.ts` for stale non-root SUDO_USER with root env expecting direct --user for availability and lifecycle paths; extend status/lifecycle tests only if needed. Include a user-facing changelog entry unless maintainers classify the fix as internal-only. Validate with the focused node test wrapper and `git diff --check`; if feasible, also capture live Linux proof using a root-owned systemd user service with stale SUDO_* variables, redacting private usernames/paths where needed.
