## Summary

Keep open. Current main still has a source-level path that treats any effective-root process with a non-root `SUDO_USER` as sudo-to-root and routes gateway stop/restart/status systemctl calls to that user's systemd manager, which matches the reported stale-root-shell failure and is not covered by the existing sudo-u regression tests.

## What This Changes

Report that Linux gateway lifecycle commands from an effective-root shell with stale `SUDO_*` variables target the sudo user's systemd user manager instead of root's user manager.

## Best Possible Solution

Add a narrow systemd scope-resolution guard that targets the root user manager when the resolved environment and root-owned user unit indicate root's own service, while preserving intentional sudo-to-root handling for non-root user services.

## Reproduction Assessment

Yes, at source level. With effective uid 0 and non-root `SUDO_USER`, current main chooses the sudo user's machine scope before direct `--user`, while the unit path still resolves from root's HOME; the reporter also supplied concrete live commands and a workaround.

## Solution Assessment

Yes, the best direction is a focused daemon systemd fix rather than a broad root policy change. The implementation should key off the resolved service owner/unit context and add regression tests for stale `SUDO_*` in root shells.
