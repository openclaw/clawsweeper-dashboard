Keep open: current main still deterministically routes effective-root plus non-root `SUDO_USER` to the sudo user's systemd user manager, so the reported root-owned user-service lifecycle bug remains valid.

## What This Changes

## Best Possible Solution

Land a narrow daemon systemd resolver fix that targets root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves intentional sudo paths.

## Repair Work Prompt

Fix the Linux gateway systemd user-scope resolver bug reported in https://github.com/openclaw/openclaw/issues/81410. Current main treats effective root plus non-root SUDO_USER as sudo-to-root and always targets `systemctl --machine <sudo-user>@ --user`, which is wrong for an already-root shell managing `/root/.config/systemd/user/openclaw-gateway.service` with `HOME=/root`, `USER=root`, `LOGNAME=root`, and `XDG_RUNTIME_DIR=/run/user/0`. Implement a narrow fix in `src/daemon/systemd.ts` so root's direct `systemctl --user` manager is used when the resolved/current service context is root's user unit, while preserving the intentional sudo caller behavior from https://github.com/openclaw/openclaw/pull/72655 and the permission-denied fallback behavior from https://github.com/openclaw/openclaw/pull/62337. Add focused regression tests in `src/daemon/systemd.test.ts` for stop/restart/status or the shared executor path with stale non-root `SUDO_*` variables plus root env, and keep existing sudo-to-root and `SUDO_USER=root` tests passing. Do not add a config option, do not implement `/etc/systemd/system` system-unit support in this fix, and do not change unrelated update-helper behavior. Add a single `CHANGELOG.md` fix entry if repo policy expects one for this user-visible lifecycle fix. Collect live Linux root systemd user-service proof with private host/user details redacted if feasible before landing.
