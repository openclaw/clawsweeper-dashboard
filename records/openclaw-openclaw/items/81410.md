## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo user's machine-scoped systemd user manager before trying root's direct `--user` manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The adjacent system-unit PR is related Linux systemd work, but it targets `/etc/systemd/system` units rather than this stale root-shell user-service case.

## What This Changes

Gateway lifecycle commands should target root's systemd user service when the effective user, home, and user unit path are root-owned despite stale non-root `SUDO_*` variables.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves the existing sudo paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with tests is the best direction. It should not add a config option or broaden into system-scoped unit support, but it must preserve real sudo-to-root and sudo-u behavior.
