## Label Justifications

- `impact:crash-loop`: The issue affects gateway process lifecycle availability: stop/start/status can report success or stopped state while the root user service keeps running.

## Summary

Keep open: current main and the latest shipped release still route effective root plus a non-root `SUDO_USER` to the sudo user's systemd user manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The related system-scoped-unit PR is adjacent Linux systemd work, but it does not specifically fix the stale root-shell user-service case.

## What This Changes

The issue asks OpenClaw gateway lifecycle commands to target root's systemd user service when running as effective root with root's home/runtime context, even if stale `SUDO_*` variables are present.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regressions that preserve the existing sudo caller and sudo-u behavior.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with regression coverage is the best direction. It should not add a config option or broaden into system-scoped unit support, and it must preserve intentional sudo-to-root and sudo-u behavior.
