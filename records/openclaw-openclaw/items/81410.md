## Summary

Keep open: the reported stale `SUDO_USER` root-shell path still exists on current `main` and in the latest release, and no linked or discovered PR closes this issue. The source path matches the reporter's logs: unit discovery can use `HOME=/root` while `systemctl` operations are redirected to `--machine <SUDO_USER>@ --user`.

## What This Changes

The issue asks gateway start, stop, and status to target the root systemd user service when effective root is managing a root-owned gateway unit despite stale `SUDO_*` variables.

## Best Possible Solution

Adjust the Linux systemd scope resolver to prefer the service owner implied by the managed unit path, `HOME`, and effective root context for root-owned units, while preserving intentional sudo-to-root non-root management and adding focused stale-`SUDO_*` regression coverage.

## Reproduction Assessment

Yes, at source level. The reporter provided concrete release logs and current main still routes effective root plus non-root `SUDO_USER` to `systemctl --machine <sudo-user>@ --user` while resolving the unit path from `HOME=/root`; I did not run a live root systemd repro in this read-only sweep.

## Solution Assessment

Yes, a narrow fix in the systemd scope resolver is the best direction. The safer alternative to broad `SUDO_USER` removal is to suppress stale sudo scope only when the effective/root-owned unit context proves the root user service is the lifecycle target.
