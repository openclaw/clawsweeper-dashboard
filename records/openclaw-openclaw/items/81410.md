## Summary

Keep open: current main still routes effective root plus non-root `SUDO_USER` straight to the sudo caller's machine-scoped user manager, so the reported root-owned systemd user-service stop/start/status mismatch remains actionable.

## What This Changes

The issue requests that Linux gateway lifecycle commands ignore stale sudo environment scope when managing a root-owned systemd user service from an effective-root shell.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that targets root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves intentional sudo and sudo-u paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with tests is the best path. It should not add a config option or broaden into system-scoped unit support, but it must preserve real sudo-to-root and sudo-u behavior.
