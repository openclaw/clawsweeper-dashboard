## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo user's `--machine ... --user` scope before root's direct `--user` manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The related system-unit PR is adjacent Linux systemd work, but it does not resolve this stale root-shell user-service case.

## What This Changes

Ensure Linux gateway lifecycle and status commands in an effective-root shell with root home and stale `SUDO_*` variables target root's systemd user service rather than the previous sudo user's scope.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage preserving intentional sudo paths.

Yes, at source level: current main deterministically routes effective root plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling, and the reporter provided live Linux logs plus an env-unset workaround. ## Solution Assessment

Yes, a focused resolver fix with tests is the best direction. It should not add a config option or become the system-scoped unit implementation, but it must preserve existing sudo behavior.
