## Summary

Keep open: current main and the latest release still prefer the non-root `SUDO_USER` machine scope whenever the effective UID is root, so a stale sudo environment can still misroute root-owned user-service lifecycle commands. The related system-scope work fixes `/etc/systemd/system` units, but it does not cover this root-owned user unit case.

## What This Changes

The issue reports that gateway lifecycle commands from a root shell with stale sudo environment variables target the sudo user's systemd user scope instead of root's user service.

## Best Possible Solution

Fix the Linux systemd adapter so an installed root-owned user unit is controlled through root's user manager when the active environment identifies root as the service owner, while preserving the existing sudo caller scope contract for normal sudo-to-root flows.

## Reproduction Assessment

Yes, at source level: a root process with non-root `SUDO_USER` still prefers `systemctl --machine <sudo-user>@ --user` before direct root `--user`, even when the root user unit path is the installed service. I did not run a live root/systemd VM repro in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow owner-aware systemd scope decision in `src/daemon/systemd.ts`, plus focused regression tests. A new flag or doc-only workaround would leave a supported gateway lifecycle command misrouting the service owner.
