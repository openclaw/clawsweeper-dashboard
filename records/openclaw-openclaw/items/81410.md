## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo caller's systemd user scope before trying root's direct `--user` manager, so the reported root-owned user-service lifecycle mismatch remains actionable.

## What This Changes

The issue asks OpenClaw gateway stop/start/status to target root's systemd user service when a root shell has stale non-root sudo environment variables.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage preserving intentional sudo paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective root plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling, and the reporter provided live Linux logs plus an env-unset workaround. I did not execute a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes: a focused resolver fix with tests is the narrowest maintainable direction. It should not add a config option or become the system-scoped unit implementation, but it must preserve existing sudo behavior.
