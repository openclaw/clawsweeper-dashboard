## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo caller's machine-scoped systemd user manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The related open system-unit PR addresses `/etc/systemd/system` units, not this stale root-shell user-service case.

## What This Changes

The issue asks gateway stop/start/status to target root's systemd user service when run from an effective-root shell with stale non-root `SUDO_*` variables.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves existing sudo paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with tests is the best direction. It should not add a config option or broaden into system-scoped unit support, but it must preserve real sudo-to-root and sudo-u behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81410 with a bounded daemon/systemd change. Current main routes effective uid 0 plus a non-root `SUDO_USER` to `systemctl --machine <sudo-user>@ --user` before direct root `--user`, even when the command context is root's user unit (`HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, unit under `/root/.config/systemd/user/openclaw-gateway.service`). Update `src/daemon/systemd.ts` so root-owned user-service lifecycle/status paths choose root's direct `--user` manager in that stale-env case, while preserving intentional sudo-to-root and sudo-u target behavior from https://github.com/openclaw/openclaw/pull/72655. Add focused regression coverage in `src/daemon/systemd.test.ts` for stop/restart/status or is-enabled with stale non-root `SUDO_*` under root, and keep existing sudo machine-scope tests passing. Do not broaden this into system-scoped `/etc/systemd/system` support and do not add a config option. Add a single changelog fix entry if this becomes a user-facing fix PR.
