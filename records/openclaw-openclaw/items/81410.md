## Summary

Keep this issue open: current `main` and the latest release still route effective-root plus non-root `SUDO_USER` through the sudo caller's user manager before direct root `--user`, while the report is specifically about a root-owned user unit under `/root`.

## What This Changes

The issue asks gateway lifecycle/status commands in a root shell with stale `SUDO_*` variables to target the root-owned systemd user service instead of the sudo caller's user scope.

## Best Possible Solution

Keep this open and land a narrow daemon/systemd fix that chooses the systemd user manager from the installed user unit/current root HOME in the root-owned unit case while preserving existing sudo and system-scope behavior.

## Reproduction Assessment

Yes, source-reproducible: with effective uid 0 and non-root `SUDO_USER`, current code sets `preferMachineScope` and returns `systemctl --machine <sudo-user>@ --user ...` before direct root `--user`. I did not live-run the root systemd service scenario in this read-only review.

## Solution Assessment

Yes, the best fix appears to be in `src/daemon/systemd.ts` where the installed user-unit owner and systemctl user-scope selection meet. The open dueling-unit PR is adjacent but not the best solution for this report because it explicitly preserves user-first lifecycle behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81410. Current `src/daemon/systemd.ts` treats effective uid 0 plus non-root `SUDO_USER` as sudo-to-root and immediately uses `systemctl --machine <sudo-user>@ --user ...`, even when `HOME=/root` and the installed user unit is `/root/.config/systemd/user/openclaw-gateway.service`. Update the systemctl user-scope selection so root-owned user units target root's user manager in that case, while preserving the intentional sudo-to-root and `sudo -u` behavior from https://github.com/openclaw/openclaw/pull/72655 and the system-scope behavior from https://github.com/openclaw/openclaw/pull/87618. Add focused tests for effective uid 0, `HOME=/root`, `USER`/`LOGNAME=root`, non-root stale `SUDO_USER`, user unit path present, and expected direct `systemctl --user` calls; keep existing sudo-caller tests passing. Do not add new config/env flags or edit `CHANGELOG.md`; update docs only if user-visible guidance changes. If feasible, include a redacted real Linux root systemd user-service smoke after the unit tests.
