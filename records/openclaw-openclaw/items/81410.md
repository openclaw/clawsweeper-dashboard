## Summary

Keep open: current main still has the reported source-level bug. The Linux systemd resolver sends effective-root commands with a non-root `SUDO_USER` to the sudo caller's user manager, while unit discovery still uses the current `HOME`, so a root-owned user unit under `/root` can be discovered but lifecycle/status calls target the wrong systemd user scope.

## What This Changes

The issue asks gateway lifecycle and status commands run from an already-root shell with stale `SUDO_*` variables to target root's systemd user service instead of the stale sudo caller's user manager.

## Best Possible Solution

Teach the Linux systemd user-scope resolver to recognize the already-root/root-owned user-unit context and use root's direct user manager, with focused regressions that keep the intended sudo caller behavior intact.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective-root plus non-root `SUDO_USER` to the sudo user's machine scope, while the reporter supplied concrete live Linux output and the `env -u SUDO_*` workaround. I did not run a live systemd host during this read-only review.

## Solution Assessment

Yes, a narrow resolver fix with regression coverage is the best direction. It should not become a system-service redesign or a new configuration option, and it must explicitly preserve the existing sudo-to-root behavior while fixing the stale-root-shell case.
