## Summary

Keep open. Current main still treats effective-root plus a non-root `SUDO_USER` as a sudo-to-root command and targets `systemctl --machine <sudo-user>@ --user` before trying root's direct `--user` manager, so the reported root-owned systemd user service can be missed.

## What This Changes

The issue asks gateway start/stop/status to ignore stale sudo environment when managing a root-owned Linux systemd user service from an effective-root shell.

## Best Possible Solution

Add a narrow systemd scope resolver guard that selects root's direct user manager only when the effective environment and unit path prove a root-owned service, while preserving genuine sudo-to-root behavior.

## Reproduction Assessment

Yes, source-level. Mocking euid 0 with non-root `SUDO_USER`, root `USER`/`LOGNAME`, root home/runtime, and a root unit path should show current main choosing `--machine <sudo-user>@ --user` before direct `--user`; I did not run a live root systemd host in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to refine systemd user-scope resolution for a proven root-owned service and add regression coverage, not to add a new config flag or broadly remove the existing sudo-to-root behavior.
