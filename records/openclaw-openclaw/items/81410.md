## Summary

Keep open: current main still has a source-level path that matches the report. The Linux systemd resolver treats any effective-root process with a non-root `SUDO_USER` as a sudo-to-root invocation and sends lifecycle/status calls to `systemctl --machine <sudo-user>@ --user`, while the unit discovery path can still read the root-owned user unit from `/root`, producing the wrong service target and contradictory status output.

## What This Changes

The issue asks gateway lifecycle and status commands to ignore stale sudo environment scope when an effective-root shell is managing a root-owned systemd user service.

## Best Possible Solution

Update the Linux systemd user-scope resolver to target root's direct user manager when the effective/root service context points at root's user unit, while preserving real sudo-to-root and sudo-u behavior with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective-root plus non-root `SUDO_USER` to the sudo user's machine scope, and the reporter provided a concrete Linux root-user service reproduction plus an environment-variable workaround. I did not run a live systemd host in this read-only review.

## Solution Assessment

Yes, a narrow resolver fix with regression tests is the best direction; it should not become a broad system-service overhaul or a new configuration knob. The implementation should explicitly protect the existing sudo-to-root behavior while handling the stale-root-shell case.
