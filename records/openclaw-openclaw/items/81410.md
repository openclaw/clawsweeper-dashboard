## Summary

Keep open. Current main and the latest shipped release still prefer a non-root `SUDO_USER` machine scope whenever the process is effectively root, so a root-owned systemd user service can be mis-targeted; the related sudo fixes do not cover this stale-root-shell case.

## What This Changes

Gateway lifecycle commands should ignore stale non-root `SUDO_*` variables when the effective/root-owned service context indicates root's systemd user manager.

## Best Possible Solution

Fix shared systemd user-scope resolution so lifecycle/status commands target the service owner implied by the resolved unit path and root environment, with regressions for stale root-shell `SUDO_*` plus existing sudo-to-root and `sudo -u` cases.

## Reproduction Assessment

Yes, source-level: with effective uid 0 and non-root `SUDO_USER`, the current scope resolver chooses the non-root machine scope before direct root `--user`, and the lifecycle/status paths all use that helper. I did not run a live root systemd service in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow change in `src/daemon/systemd.ts` shared scope resolution with focused tests. Avoid a broad service-scope redesign or a new config knob unless maintainers explicitly choose a larger policy change.
