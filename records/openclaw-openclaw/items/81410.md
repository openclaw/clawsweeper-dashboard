## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo caller's machine-scoped user manager before root's direct `--user` manager, so the reported root-owned user-service stop/start/status mismatch remains actionable.

## What This Changes

The issue reports that gateway lifecycle commands run from a root shell with stale `SUDO_*` variables manage the sudo caller's systemd user service instead of root's user service.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that targets root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves intentional sudo paths.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81410 narrowly. Current `resolveSystemctlUserScope` routes effective root plus non-root `SUDO_USER` to `systemctl --machine <sudo-user>@ --user`, which is wrong when the effective service context is root's user unit (`HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, root unit under `/root/.config/systemd/user`). Update the daemon systemd resolver so root-owned user-service lifecycle/status paths target root's direct `--user` manager despite stale `SUDO_*` variables. Preserve intentional sudo-to-root/sudo-u behavior covered by PR 72655; do not broaden this into `/etc/systemd/system` support or change the system-scoped-unit policy from PRs 57276/68909. Add focused regression coverage in `src/daemon/systemd.test.ts` for stale non-root `SUDO_USER` with root env/unit context, and update lifecycle/status tests only if needed. Add a changelog entry only if the repair is user-visible per repo policy.
