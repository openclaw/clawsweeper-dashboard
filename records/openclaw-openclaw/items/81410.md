## Summary

Keep open: current main and the latest release still route effective-root systemd user-scope calls through non-root `SUDO_USER` machine scope, so the reported stale-sudo root-shell lifecycle bug remains source-reproducible.

## What This Changes

The issue reports that gateway lifecycle commands run from an effective root shell can use stale sudo environment variables and target the wrong systemd user scope.

## Best Possible Solution

Add an explicit, tested root-owned user-service targeting rule that prevents false success when stale `SUDO_*` variables point at the wrong user manager while preserving intentional sudo-to-caller lifecycle flows.

## Reproduction Assessment

Yes. Source inspection shows the current user-scope systemctl path still chooses the non-root `SUDO_USER` machine scope whenever the process is effectively root, matching the reported stale-sudo root-shell failure mode.

## Solution Assessment

Unclear. A narrow code fix is plausible, but maintainers need to choose the supported distinction between a stale root shell with inherited `SUDO_*` variables and intentional `sudo openclaw ...` targeting of the invoking user's service.
