## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo caller's machine-scoped systemd user manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The related system-scoped-unit PR targets `/etc/systemd/system` units and does not cover this stale root-shell user-service case.

## What This Changes

The issue reports that gateway lifecycle commands in an effective-root shell with stale non-root `SUDO_*` variables target the sudo user's systemd user manager instead of root's user service.

## Best Possible Solution

Land a narrow daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves existing sudo paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes, a focused resolver fix with tests is the best direction. It should not add a config option or broaden into system-scoped unit support, but it must preserve real sudo-to-root and sudo-u behavior.

Update the daemon systemd resolver so root-owned user-service lifecycle/status commands target root's direct `systemctl --user` manager when the effective environment and resolved unit path indicate root, while preserving intentional sudo-to-root and sudo-u behavior from https://github.com/openclaw/openclaw/pull/72655.
