Revise the implementation to use one maintainer-approved sentinel consistently across cron selection, cron fallback policy, subagent selection, public docs, and regression coverage, then provide real after-fix runtime proof.

- **[P2] Exclude the sentinel from user override metadata:** `src/agents/subagent-spawn-plan.ts:75`
  - body: For `modelOverride: "@default"`, model selection falls through to the configured/default model, but this predicate still treats the non-empty sentinel as user-supplied.   - confidence: 0.93
- **[P2] Keep `@default` on the default cron fallback path:** `src/cron/isolated-agent/model-selection.ts:119`
  - body: This normalizes `payload.model: "@default"` away for model selection, but `resolveCronFallbacksOverride` still inspects the raw payload and classifies any non-empty model as a payload override. That makes `@default` use a different fallback policy than an omitted model, so the sentinel is not fully equivalent to following the live default.
  - confidence: 0.88
- **[P2] Align the object-form test with the function type:** `src/agents/openclaw-tools.subagents.sessions-spawn.model.test.ts:239`
  - body: This test passes an object into `resolveSubagentModelAndThinkingPlan`, but that function still declares `modelOverride?: string`; CI's `check-test-types` job reports TS2322 at this line. Either widen the function type if object-shaped overrides are valid here, or move the assertion to the lower-level selector that already accepts `unknown`.
  - confidence: 0.95
- **[P3] Document the public sentinel before exposing it:** `src/agents/model-default-sentinel.ts:9`
  - body: This adds a supported operator value, but the cron and subagent docs still describe model fields as ordinary explicit overrides and say unresolved cron models fail instead of falling through. Add user-facing docs for spelling, precedence, and migration before shipping the new contract.
  - confidence: 0.84
