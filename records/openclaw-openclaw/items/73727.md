## Summary

Keep open: current main still lacks this sentinel capability, but this PR is not ready because the patch has source-visible correctness defects and the external-contributor real behavior proof is still missing.

## What This Changes

The PR adds an `@default` sentinel for cron and subagent stored model overrides so they can resolve through the live default model at execution time.

## Best Possible Solution

Settle one public sentinel contract, centralize normalization across cron selection, fallback policy, subagent metadata, docs, and tests, then merge only after redacted real runtime proof is provided.

## Reproduction Assessment

Yes. Source inspection shows current main treats sentinel-like cron model values as explicit model refs, and PR-head inspection plus CI logs reproduce the metadata, fallback-policy, and type mismatches.

## Solution Assessment

No. The shared helper is a plausible direction, but the current patch is not the best maintainable solution until fallback policy, subagent metadata, typed tests, docs, and sentinel spelling are aligned.

- **[P2] Exclude the sentinel from user override metadata:** `src/agents/subagent-spawn-plan.ts:75`
  - body: For `modelOverride: "@default"`, model selection falls through to the configured/default model, but this predicate still treats the non-empty sentinel as user-supplied. The child session patch then records `modelOverrideSource: "user"`, so later fallback persistence treats a default-following spawn as an explicit user pin.
  - confidence: 0.93
- **[P2] Keep the sentinel on the default cron fallback path:** `src/cron/isolated-agent/model-selection.ts:119`
  - body: This normalizes `payload.model: "@default"` away for model selection, but `resolveCronFallbacksOverride` still inspects the raw payload and classifies any non-empty model as a payload override. With no configured fallbacks that returns `[]` instead of the omitted-model/default path, giving the sentinel different fallback behavior than no model.
  - confidence: 0.89
- **[P2] Align the object-form test with the function type:** `src/agents/openclaw-tools.subagents.sessions-spawn.model.test.ts:239`
  - body: This added test passes `{ primary: "@default" }` into `resolveSubagentModelAndThinkingPlan`, but that function still declares `modelOverride?: string`. Either widen the function type if object-shaped overrides are valid here, or move the assertion to the selector that accepts `unknown`.
  - confidence: 0.95
- **[P3] Document the public sentinel before exposing it:** `src/agents/model-default-sentinel.ts:9`
  - body: This introduces a supported operator value, but the cron docs still say unresolved `--model` values fail instead of falling through, and the subagent docs still describe `model` as an ordinary override. Add docs for spelling, precedence, and migration before shipping the new contract.
  - confidence: 0.84
