Full review comments:

- **[P2] Exclude the sentinel from user override metadata:** `src/agents/subagent-spawn-plan.ts:75`
  - body: For `modelOverride: "@default"`, model selection falls through to the configured/default model, but this predicate still treats the non-empty sentinel as user-supplied. The persisted session patch then records `modelOverrideSource: "user"`, so later runs can treat a default-following spawn as an explicit pin.
  - confidence: 0.93
- **[P2] Keep the sentinel on the default cron fallback path:** `src/cron/isolated-agent/model-selection.ts:119`
  - body: This normalizes `payload.model: "@default"` away for model selection, but `resolveCronFallbacksOverride` still inspects the raw payload and classifies any non-empty model as a payload override. That gives `@default` a different fallback policy than an omitted model.
  - confidence: 0.88
- **[P2] Align the object-form test with the function type:** `src/agents/openclaw-tools.subagents.sessions-spawn.model.test.ts:239`
  - body: This test passes an object into `resolveSubagentModelAndThinkingPlan`, but that function still declares `modelOverride?: string`; CI reports TS2322 at this line. Either widen the function type if object-shaped overrides are valid here, or move this assertion to the selector that already accepts `unknown`.
  - confidence: 0.95
- **[P3] Document the public sentinel before exposing it:** `src/agents/model-default-sentinel.ts:9`
  - body: This introduces a supported operator value, but the cron and subagent docs still describe model fields as ordinary explicit overrides and cron docs still say unresolved values fail instead of falling through. Add docs for spelling, precedence, and migration before shipping the contract.
  - confidence: 0.84
