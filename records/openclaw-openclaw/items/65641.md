## Summary

Keep open: this PR targets a real gateway diagnostics bug, but the branch still broadens local loopback auth semantics and lacks required real behavior proof before merge.

## What This Changes

The PR preserves explicit `deviceIdentity: null`, appends early gateway connect errors to RPC close diagnostics, changes loopback device-identity omission, and updates gateway/TUI tests.

## Best Possible Solution

Revise the patch to preserve explicit constructor null and thread early connect-error details into current main's `GatewayTransportError` path while retaining the narrow backend shared-auth identity-omission contract.

## Reproduction Assessment

Yes, by source inspection: the PR nulls device identity for default no-token/no-password loopback calls, and `GatewayClient` only loads stored device-token auth when a device identity exists. I did not run live gateway commands in this read-only review.

## Solution Assessment

No. The PR addresses real gaps, but broad loopback/no-auth identity omission is not the narrowest maintainable fix; preserve explicit null and connect-error diagnostics without weakening paired local auth.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve device identity for paired loopback sessions:** `src/gateway/call.ts:195`
  - body: This branch treats every loopback call without `token` or `password` as device-less. `GatewayClient` only loads cached per-device auth when a device identity exists, so paired local sessions that rely on stored device-token auth can stop presenting credentials before the RPC runs.
