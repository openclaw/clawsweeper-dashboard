## Summary

Keep open: the linked bug-fix work is still useful because current main still drops explicit `deviceIdentity: null` and does not thread `onConnectError` into `callGateway` close errors, but the PR as written has a blocking gateway auth regression and needs revision before merge.

## What This Changes

The PR preserves explicit `deviceIdentity: null`, threads early connect-error details into gateway RPC close errors, adds gateway/TUI regression tests, and broadens loopback device-identity omission.

## Best Possible Solution

Revise or replace the PR with a narrow patch that preserves explicit constructor `deviceIdentity: null`, includes early connect-error details in `callGateway` transport failures, and retains the current backend/shared-token loopback identity-omission contract with regression coverage.

## Reproduction Assessment

Yes. Source inspection against current main shows the null-coercion path and missing `onConnectError` plumbing, and the PR diff shows the broad loopback/no-token identity omission that would break paired stored-token sessions.

## Solution Assessment

No. The PR targets real gaps, but broad loopback/no-auth identity omission is not the narrowest maintainable fix; the safer solution is to honor explicit null only when requested and keep the documented backend shared-auth exception.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve device identity for paired loopback sessions:** `src/gateway/call.ts:192-193`
  - body: This branch treats every loopback call without `token` or `password` as device-less. `GatewayClient` only loads cached per-device auth when a device identity exists, so paired local sessions that rely on stored device-token auth can stop presenting credentials before the RPC runs. Please keep omission limited to the current backend shared-token/password path, or only honor an explicit caller-provided `deviceIdentity: null`.
  - confidence: 0.91
