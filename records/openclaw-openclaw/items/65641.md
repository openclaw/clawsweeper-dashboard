## Summary

Keep this PR open, but it is not merge-ready as-is. Current main still has the reported null-preservation and early connect-error diagnostic gaps, but the branch widens loopback device-identity omission in a way that can suppress stored device-token auth for paired local sessions, and the contributor has not provided real after-fix behavior proof.

## What This Changes

The PR preserves explicit `deviceIdentity: null`, threads early gateway connect errors into close-error diagnostics, changes loopback device-identity selection, and adds gateway/TUI regression tests.

## Best Possible Solution

Revise the patch to preserve explicit constructor `deviceIdentity: null` and include early connect-error details in `callGateway` failures while retaining the current backend/shared-auth loopback identity-omission contract.

## Reproduction Assessment

Yes. Source inspection shows current main still coerces explicit null and does not pass `onConnectError` through `callGateway`, while the PR source shows default loopback/no-token calls would force `deviceIdentity: null` and bypass stored device-token auth.

## Solution Assessment

No. The PR targets real gaps, but broad loopback/no-auth identity omission is not the narrowest maintainable fix; explicit null preservation plus current shared-auth carveout is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve device identity for paired loopback sessions:** `src/gateway/call.ts:195`
  - body: This branch treats every loopback call without `token` or `password` as device-less. `GatewayClient` only loads cached per-device auth when a device identity exists, so paired local sessions that rely on stored device-token auth can stop presenting credentials before the RPC runs. Keep omission limited to the current backend shared-token/password path, or only honor an explicit caller-provided `deviceIdentity: null`.
  - confidence: 0.91
