## Summary

Keep open: the intent is plausible, but the patch can fail open when the relay CLI cannot know the originating session's before-tool policy after the bridge and gateway are unavailable; it also needs inspectable real behavior proof before merge.

## What This Changes

The PR changes the native hook relay unavailable fallback so Codex PreToolUse returns no output when no before-tool-call policy is detected, and updates CLI tests for that behavior.

## Best Possible Solution

Carry an explicit policy-required/no-policy fallback mode from relay registration into the hook command or unavailable response, keep PreToolUse fail-closed whenever policy state is unknown, and add tests plus real proof for both no-policy and policy-present failures.

## Reproduction Assessment

Yes, from source inspection: the PR's added branch returns empty stdout based on process-local `hasBeforeToolCallPolicy()` after the CLI can no longer query the originating relay. I did not run a live Codex native relay in this read-only review.

## Solution Assessment

No. The safer solution is to make no-policy fallback explicit in the relay command/registration or keep unavailable PreToolUse fail-closed whenever policy state is unknown.
