## Summary

Keep open: the fix direction is plausible, but the patch still treats `resolveProviderAuth(...oauthMarker...)` as if `apiKey` were the live OAuth token. Current main returns the OAuth marker in `apiKey` and the real access token only as `discoveryApiKey`, so the reported Minimax portal auth failure is not fixed by this branch.

## What This Changes

The PR changes the Minimax portal provider catalog to call `ctx.resolveProviderAuth` for OAuth profiles and adds three catalog regression tests.

## Best Possible Solution

Fix Minimax portal OAuth in the provider-owned catalog/runtime-auth path so stored OAuth profiles produce request-ready credentials, with a regression test using the real resolver/auth store and redacted real behavior proof of a model request or equivalent runtime auth path.

Yes, from source inspection: current main can put `minimax-oauth` into the portal provider config while runtime auth rejects non-secret markers as real credentials. I did not run a live Minimax request because that requires a valid Minimax OAuth account/subscription.

## Solution Assessment

No. Calling the existing resolver is the right direction, but this implementation uses the wrong resolver field and the added tests mock away the real contract that needs coverage.
