## Summary

Keep this PR open: the Minimax bug is plausible and the resolver seam is relevant, but this patch treats the catalog resolver as returning a live OAuth token even though current core returns the OAuth marker in `apiKey` and the live token in `discoveryApiKey`.

## What This Changes

The PR routes the Minimax portal catalog through `ctx.resolveProviderAuth(...oauthMarker...)` and adds three Minimax catalog regression tests.

## Best Possible Solution

Align the Minimax fix with the actual provider-auth contract and add coverage that exercises the real resolver/runtime path from a portal OAuth profile through request auth, not only a mocked catalog resolver.

## Reproduction Assessment

Unclear for a live Minimax request because this review did not run against a real Minimax portal OAuth account. Source inspection does reproduce the PR-specific problem: the production resolver returns the OAuth marker as `apiKey`, not the live access token mocked by the new test.

## Solution Assessment

No; using the existing resolver seam may be the right direction, but this patch relies on an incorrect `resolveProviderAuth` return shape and does not prove the reported request path is fixed.

## Review Findings
