## Summary

Keep open for maintainer review: current main still lacks the requested runtime axis, but this PR is not merge-ready because it mixes mock provider debug plans into runtime tool-call evidence. The related correction in https://github.com/openclaw/openclaw/issues/80236 shows that this can turn a mock/native mismatch into a hard runtime-drift claim, and the broader corrected harness is being pursued in https://github.com/openclaw/openclaw/pull/80323.

## What This Changes

This PR adds a private-QA Pi-vs-Codex runtime-pair harness, report mode, suite summary schema, runtime override, tests, multipass forwarding, and a release-check job.

## Best Possible Solution

Use the corrected harness direction from https://github.com/openclaw/openclaw/pull/80323 or revise this PR so provider-plan diagnostics are separate from transcript/app-server runtime tool evidence before any merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature PR; source inspection gives high-confidence evidence that current main lacks the requested runtime-pair surface, and the PR body provides terminal proof that the proposed lane runs.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is: the implementation records mock provider plans as runtime tool evidence. The safer maintainable fix is to keep provider-plan diagnostics separate from actual runtime transcript/app-server tool calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep provider plans out of runtime tool evidence:** `extensions/qa-lab/src/runtime-parity.ts:864-868`
  - body: `captureRuntimeParityCell` replaces transcript-derived tool calls with `/debug/requests` whenever the mock provider is present. In Codex app-server cells the mock can plan tools that are not declared or executed by the runtime, so the report hard-fails mock/native mismatches as runtime drift; the related approval-followthrough correction already reclassified this exact signal as a mock artifact. Store provider plans separately, or only use transcript/app-server events for `RuntimeParityCell.toolCalls`.
  - confidence: 0.91
