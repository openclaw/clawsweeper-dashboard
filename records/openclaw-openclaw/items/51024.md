## Summary

Keep open: current main still uses the fixed two-hour stale-marker cleanup, so this PR addresses a valid source-reproducible cron recovery bug. The patch looks mechanically sound, but it still needs the required changelog entry and fresh validation before merge.

## What This Changes

The PR derives cron stale `runningAtMs` cleanup from each job's resolved execution timeout plus grace/floor logic, logs the resolved stale threshold, and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land the timeout-aware stale-marker cleanup after adding the changelog entry, refreshing the branch, and running targeted cron regression tests plus the changed gate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an `agentTurn` cron job with `timeoutSeconds: 900` and persisted `state.runningAtMs` older than about 16 minutes but younger than two hours; current main still treats it as running because cleanup only checks the fixed two-hour threshold.

## Solution Assessment

Yes, with the changelog gap fixed. Reusing `resolveCronJobTimeoutMs` is the narrowest maintainable fix because stale-marker recovery follows the same timeout contract already used for cron execution.
