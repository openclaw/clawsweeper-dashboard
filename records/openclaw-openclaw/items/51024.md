## Label Justifications

- `P2`: This is a focused cron scheduling recovery bug with real user impact when stale running markers block jobs, but the blast radius is limited to cron maintenance behavior.
- `merge-risk: 🚨 compatibility`: The diff changes stale-marker cleanup timing for existing default-timeout cron jobs from a fixed two-hour window to effective timeout plus grace.
- `merge-risk: 🚨 message-delivery`: If a marker is cleared too aggressively, cron jobs can run again earlier than before and duplicate scheduled delivery side effects.

## Summary

Keep open: current main still uses the fixed two-hour cron stale-marker threshold, so the PR targets a real remaining recovery bug, but the branch is currently conflicting with main, lacks inspectable real behavior proof, and must preserve newer stale-marker skip handling during rebase.

## What This Changes

The PR changes cron maintenance to clear stale `runningAtMs` markers using `resolveCronJobTimeoutMs(job)` plus grace/minimum fallback logic and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land one rebased timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves the current post-clear skip guard and unlimited-timeout fallback, and includes redacted real cron/gateway recovery proof.

## Reproduction Assessment

Yes from source inspection. Current main still clears `runningAtMs` only after the fixed two-hour threshold, so an `agentTurn` job with `timeoutSeconds: 900` and a roughly 16-minute-old stale marker remains blocked even though its own timeout window has elapsed.

## Solution Assessment

The timeout-aware direction is the right maintainable fix, but this branch is not the best merge shape yet. It needs a current-main rebase that preserves post-clear skip handling plus inspectable real behavior proof and maintainer acceptance of the shorter default-timeout recovery windows.
