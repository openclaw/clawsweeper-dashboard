## Summary

Keep open: current main still uses the fixed two-hour stale-marker cleanup, so this PR is not obsolete, but the branch conflicts with current cron skip-guard behavior and still lacks inspectable real behavior proof before merge.

## What This Changes

The PR changes cron stale `runningAtMs` cleanup to use each job's resolved timeout plus grace/minimum handling, and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land one rebased timeout-aware stale-marker cleanup that derives recovery from the execution timeout policy, preserves the post-clear skip guard and unlimited-timeout fallback, and includes redacted gateway proof.

## Reproduction Assessment

Yes, from source inspection. Current main uses a fixed two-hour `STUCK_RUN_MS`, so an `agentTurn` job with `timeoutSeconds: 900` and a stale marker just past its own timeout window remains blocked until the two-hour threshold.

## Solution Assessment

No, not mergeable as-is. Timeout-aware cleanup is a maintainable direction, but this branch needs a current-main rebase that preserves post-clear skip handling plus maintainer acceptance of the shorter default-timeout recovery windows.
