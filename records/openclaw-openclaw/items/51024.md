## Summary

Keep open. Current main still has the fixed two-hour stale-marker cleanup, while the PR addresses a real cron recovery gap and the patch looks narrowly correct; the branch is currently dirty/stale and needs maintainer review, rebase, and fresh validation before merge.

## What This Changes

The PR changes cron stuck-run cleanup to derive the stale `runningAtMs` threshold from each job's effective execution timeout, adds a grace/floor fallback, logs the resolved threshold, and adds a regression test for short-timeout isolated agent-turn jobs.

## Best Possible Solution

Rebase this PR onto current main, keep the timeout-aware stale-marker helper in the cron service, preserve the conservative fallback for effectively unlimited jobs, and land it after targeted cron regression tests plus the normal changed gate pass.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated cron `agentTurn` job with `timeoutSeconds: 900` and persisted `state.runningAtMs` older than 16 minutes but younger than two hours; current main still treats it as running because cleanup only uses the fixed two-hour threshold.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it reuses the existing cron execution timeout contract instead of adding another timeout configuration path; the remaining work is rebase and validation, not a different design.
