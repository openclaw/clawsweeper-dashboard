## Summary

Keep open. Current main still uses the fixed two-hour stale `runningAtMs` threshold, while this PR remains a plausible, narrow fix that reuses the existing cron execution timeout policy; the only concrete PR-review issue I found is the missing required changelog entry for a user-visible cron recovery fix.

## What This Changes

The PR changes cron stale `runningAtMs` cleanup to derive the threshold from each job's effective timeout plus grace/floor logic, and adds regression coverage for a short-timeout isolated `agentTurn` job.

## Best Possible Solution

Land the timeout-aware stale-marker helper after adding the required changelog entry, refreshing the branch against current main, and running targeted cron regression tests plus the normal changed gate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` cron job with `timeoutSeconds: 900` and persisted `state.runningAtMs` older than about 16 minutes but younger than two hours; current main still treats it as running because cleanup only uses the fixed two-hour threshold.

## Solution Assessment

Yes, with the changelog gap fixed. Reusing `resolveCronJobTimeoutMs` is the narrowest maintainable path because stale-marker recovery follows the same timeout contract that already governs cron execution, rather than adding another independent timeout setting.
