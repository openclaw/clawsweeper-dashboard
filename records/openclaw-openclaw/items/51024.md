## Summary

Keep open: the cron stale-marker bug still exists on current main and this PR is useful, but the live branch is conflicting, lacks inspectable real behavior proof, and its head predates the current post-clear skip guard that prevents silently advancing never-executed overdue cron slots.

## What This Changes

The PR replaces the fixed two-hour cron stale `runningAtMs` cleanup threshold with a threshold derived from `resolveCronJobTimeoutMs(job)` plus grace/minimum handling and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land a rebased timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves the post-clear skip guard and unlimited-timeout fallback, and includes redacted real gateway proof for short explicit-timeout recovery.

## Reproduction Assessment

Yes, from source inspection. Current main still clears `runningAtMs` only after a fixed two-hour threshold, so an `agentTurn` job with `timeoutSeconds: 900` can stay blocked after its own timeout window has elapsed.

## Solution Assessment

No, not as currently mergeable. Timeout-aware cleanup is the right maintainable direction, but this branch needs a current-main rebase that preserves post-clear skip handling plus maintainer acceptance of the shorter default-timeout recovery windows.
