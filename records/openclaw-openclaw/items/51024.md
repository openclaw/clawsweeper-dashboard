## Summary

Keep open. Current main still uses the fixed two-hour cron stale-running threshold, and this PR’s code direction looks sound with no blocking correctness or security findings, but the external PR still lacks inspectable after-fix real behavior proof and overlaps with another open stale-lock fix path.

## What This Changes

The branch replaces cron’s fixed stale `runningAtMs` threshold with a per-job timeout-derived threshold, logs that threshold, and adds a focused regression test for short-timeout agent-turn jobs.

## Best Possible Solution

Land one canonical timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves the post-clear due-slot skip behavior, keeps the conservative fallback for unlimited jobs, and includes redacted real cron/gateway recovery proof.

## Reproduction Assessment

Yes. A high-confidence source repro is an `agentTurn` cron job with `timeoutSeconds: 900` and persisted `state.runningAtMs` older than about 16 minutes but younger than two hours; current main still treats it as running because cleanup uses the fixed two-hour threshold.

## Solution Assessment

Yes for this PR’s code direction. Reusing the existing cron timeout policy is the narrow maintainable fix, and this branch preserves the current post-clear skip behavior, but it still needs real proof and current-main validation before merge.
