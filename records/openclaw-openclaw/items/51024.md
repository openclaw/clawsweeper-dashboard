## Summary

Keep open. Current main still uses a fixed two-hour stale `runningAtMs` threshold, while this PR is the canonical remaining timeout-aware cleanup path after https://github.com/openclaw/openclaw/pull/43816 was closed in favor of it; it still needs contributor real behavior proof plus rebase and merge validation before landing.

## What This Changes

The PR updates cron maintenance to compute stale running-marker thresholds from `resolveCronJobTimeoutMs(job)` with grace/minimum/fallback handling and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land one rebased timeout-aware stale-marker cleanup that reuses the existing cron timeout policy, preserves post-clear skip behavior and the unlimited-timeout fallback, and includes redacted real cron/gateway recovery proof.

## Reproduction Assessment

Yes, by source inspection. On current main, an `agentTurn` cron job with `timeoutSeconds: 900` and a persisted `runningAtMs` older than about 16 minutes but younger than two hours remains blocked because cleanup still waits for the fixed two-hour threshold.

## Solution Assessment

Yes for the code direction. Deriving stale-marker recovery from `resolveCronJobTimeoutMs(job)` is the narrow maintainable fix, but this branch is not merge-ready until it is rebased, backed by inspectable real behavior proof, and revalidated.
