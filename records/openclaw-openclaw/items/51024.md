## Label Justifications

- `P2`: This is a focused cron scheduling recovery bug with real user impact when stale running markers block jobs, but the blast radius is limited to cron maintenance behavior.
- `merge-risk: 🚨 compatibility`: The diff changes stale-marker cleanup timing for existing default-timeout cron jobs from a fixed two-hour window to effective timeout plus grace.
- `merge-risk: 🚨 message-delivery`: If a marker is cleared too aggressively, cron jobs can run again earlier than before and duplicate scheduled delivery side effects.

## Summary

Keep open: current main still uses a fixed two-hour stale-marker threshold, so this PR addresses a real cron recovery gap, but it is not merge-ready without a rebase that preserves current-main skip handling and inspectable real behavior proof.

## What This Changes

The PR changes cron stale `runningAtMs` cleanup to use each job's resolved timeout plus grace/minimum fallback logic and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land one rebased timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves the post-clear skip guard and unlimited-timeout fallback, and includes redacted real gateway proof for short explicit-timeout recovery.

## Reproduction Assessment

Yes, from source inspection. Current main still clears `runningAtMs` only after a fixed two-hour threshold, so an `agentTurn` job with `timeoutSeconds: 900` can stay blocked after its own timeout window has elapsed.

## Solution Assessment

No, not as currently mergeable. Timeout-aware cleanup is the right maintainable direction, but this branch needs a current-main rebase that preserves post-clear skip handling plus maintainer acceptance of the shorter default-timeout recovery windows.
