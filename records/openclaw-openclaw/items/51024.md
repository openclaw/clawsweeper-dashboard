## Summary

Keep open: current main still uses the fixed two-hour stale `runningAtMs` cleanup window, and this PR is the current canonical branch for the timeout-aware cron fix after the overlapping older PR was closed. It still needs contributor real behavior proof and a rebase before merge.

## What This Changes

The branch derives cron stale running-marker cleanup from each job's resolved timeout plus grace/minimum thresholds, logs the threshold, and adds a short-timeout `agentTurn` regression test.

## Best Possible Solution

Land one rebased canonical timeout-aware stale-marker cleanup that reuses `resolveCronJobTimeoutMs`, preserves the current post-clear skip guard, keeps the conservative unlimited-timeout fallback, and includes redacted real cron/gateway recovery proof.

## Reproduction Assessment

Yes. A high-confidence source repro is an `agentTurn` cron job with `timeoutSeconds: 900` and persisted `state.runningAtMs` older than about 16 minutes but younger than two hours; current main still treats it as running because cleanup uses the fixed two-hour threshold.

## Solution Assessment

Yes for the code direction. Reusing the existing cron timeout policy is the narrow maintainable fix, but the branch still needs rebase and real behavior proof before it is merge-ready.
