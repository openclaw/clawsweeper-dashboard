## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep bridge capabilities out of stored canvas metadata:** `src/agents/mcp-ui-resource.ts:51-58`
  - body: `fetchMcpAppView` stores the app `sessionKey` in `mcpApp` metadata and `buildMcpAppCanvasJson` later serializes that metadata with `toolInput` and `toolResult`. Assistant canvas JSON can be logged, exported, replayed, or retained outside the live bridge lifetime, so this leaks both a bridge capability and potentially sensitive tool payloads; mint a transient server-side capability or resolve the live session without persisting these values.
  - confidence: 0.92
- **[P1] Revalidate runtimes before serving app bridge calls:** `src/gateway/server-methods/mcp-app-proxy.ts:26-28`
  - body: `resolveRuntime` returns `manager.getExisting(sessionId)` before loading the latest config or checking the runtime fingerprint. If `mcp.apps.enabled` is disabled or an MCP server is removed/rotated while an app view is open, iframe bridge calls can keep using the stale runtime contrary to current config.
  - confidence: 0.9
- **[P1] Reuse canonical session IDs for restored app runtimes:** `src/gateway/server-methods/mcp-app-proxy.ts:35`
  - body: When no mapping exists, restored app views create a runtime under `mcp-app:${sessionKey}`. Later turns and cleanup use the canonical session id, so this forks MCP runtime state and can leave server processes alive after the real session is disposed.
  - confidence: 0.88
- **[P1] Constrain app-initiated resource reads:** `src/gateway/server-methods/mcp-app-proxy.ts:99`
  - body: `mcp.readResource` forwards any iframe-supplied URI to the MCP server once apps are enabled. The bridge should restrict reads to advertised or approved app resources with clear type and size policy; otherwise MCP-provided HTML can become a broad resource proxy through the gateway.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive MCP iframe bridge with app-initiated tool/resource access and persisted session/tool metadata that needs hardening before merge.

Concerns:

- **[high] Persisted bridge capability and tool payloads:** `src/agents/mcp-ui-resource.ts:51`
  - body: The patch stores `sessionKey` in app metadata and serializes tool input/result into canvas JSON, which can outlive the live bridge and leak sensitive payloads or capability material.
  - confidence: 0.92
- **[high] Broad app resource proxy:** `src/gateway/server-methods/mcp-app-proxy.ts:99`
  - body: The bridge accepts arbitrary app-supplied resource URIs and forwards them to the MCP runtime without limiting reads to the UI resource set or enforcing a host-side policy.
  - confidence: 0.84
- **[medium] Unbounded HTML resource persistence:** `src/agents/mcp-ui-resource.ts:38`
  - body: The UI resource fetch decodes MCP-provided HTML/blob content into a stored canvas document without an obvious size limit at this boundary, creating availability and retention risk for untrusted MCP resources.
  - confidence: 0.78

  - body: Returning an existing runtime before config revalidation can keep app bridge policy enabled after the operator disables MCP Apps or changes server configuration.
