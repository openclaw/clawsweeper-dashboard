## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cap MCP App resources before decoding and writing:** `src/agents/mcp-ui-resource.ts:49-53`
  - body: When MCP Apps are enabled, this accepts any `text` string or base64 `blob` from `resources/read`, decodes it, and passes the whole HTML string into `createCanvasDocument`, whose HTML path writes the full value to disk. The MCP SDK schema only constrains these fields as strings, so a configured server can return an oversized UI resource and exhaust memory or disk before the host rejects it. Add an explicit byte cap for text and blob resources, and cover both paths in tests.
  - confidence: 0.92
- **[P2] Render disabled MCP App previews through the canvas frame:** `ui/src/ui/chat/tool-cards.ts:478-480`
  - body: This early return hides any persisted canvas preview tagged with `mcpApp` whenever `mcp.apps.enabled` is false. The PR body says disabled apps should fall back to the existing canvas/embed rendering, and current main renders assistant canvas previews through `renderPreviewFrame`; after a user disables the flag or reloads history, the preview disappears instead. Let the disabled path use `renderPreviewFrame` and reserve `<mcp-app-view>` for the enabled bridge.
  - confidence: 0.89
