## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop persisting MCP session and tool payload data:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: `buildMcpAppCanvasJson` serializes the app's live `sessionKey` plus the full `toolInput`/`toolResult` into canvas/chat JSON. Canvas previews are persisted and can flow through history, logs, exports, or context paths, so the app bridge should derive session capability from live UI context and persist only minimal non-sensitive view metadata.
  - confidence: 0.9
- **[P1] Bound MCP App resources before decoding or writing:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: Server-provided text or base64 HTML is decoded without a maximum and then passed to canvas document creation, whose HTML entrypoint is written to disk. An enabled MCP server can therefore force large memory allocation or persistent state growth before the host has a chance to reject the resource.
  - confidence: 0.88
- **[P1] Apply resource CSP and iframe permissions:** `ui/src/ui/components/mcp-app-view.ts:249-250`
  - body: The MCP Apps dependency exposes resource-level CSP and permissions metadata, including a helper for iframe `allow`, but the host renders the iframe with fixed policy and no resource-derived permissions. That can both deny legitimate app permissions and fail to enforce the app-declared connection/resource/frame boundaries that the host contract expects.
  - confidence: 0.86
- **[P2] Keep restored app runtimes under canonical sessions:** `src/gateway/server-methods/mcp-app-proxy.ts:31-42`
  - body: When the gateway cannot resolve the stored session key, it creates a fallback `mcp-app:${sessionKey}` runtime for restored views. That can start MCP processes outside the normal run/session lifecycle, leaving authorization and cleanup tied to a synthetic owner instead of the canonical session state.
  - confidence: 0.78
