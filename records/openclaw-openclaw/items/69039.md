## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep bridge capability metadata out of stored canvas data:** `src/agents/mcp-ui-resource.ts:62-67`
  - body: `fetchMcpAppView` stores `runtime.sessionKey` in `mcpApp`, and `buildMcpAppCanvasJson` serializes the same metadata with tool input/result into assistant canvas JSON. Canvas/chat metadata persists into transcripts, manifests, logs, export, and replay paths, so this leaks a live bridge capability and potentially sensitive tool payloads outside the iframe lifetime; use a transient server-side capability lookup and avoid serializing tool payloads unless explicitly redacted and approved.
  - confidence: 0.93
- **[P1] Revalidate current config before proxying app bridge calls:** `src/gateway/server-methods/mcp-app-proxy.ts:28-31`
  - body: `resolveRuntime` returns a cached runtime from `peekSession` before loading current config or rechecking the MCP fingerprint. If an operator disables `mcp.apps.enabled` or changes/removes an MCP server while an app view is open, existing iframe bridge calls can keep using the stale app-enabled runtime contrary to current config.
  - confidence: 0.91
- **[P1] Do not fork restored views into synthetic MCP sessions:** `src/gateway/server-methods/mcp-app-proxy.ts:38-40`
  - body: Restored views with no session-key mapping create a runtime under `mcp-app:${sessionKey}`. Normal runs and cleanup use canonical session ids, so this forks runtime state and can leave MCP server processes alive after the real session is disposed; reuse/adopt the canonical session identity or require a valid existing mapping.
  - confidence: 0.88
- **[P1] Constrain app-initiated resource reads:** `src/gateway/server-methods/mcp-app-proxy.ts:107-117`
  - body: `mcp.readResource` forwards arbitrary iframe-supplied URIs to the MCP server once apps are enabled. The ext-apps contract exposes server resources as a host capability, but OpenClaw still needs host-side policy limiting reads to advertised/approved app resources plus MIME and size constraints; otherwise MCP HTML becomes a broad resource proxy through the gateway.
  - confidence: 0.87
- **[P2] Put a hard cap on MCP App HTML resources:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: This decodes and persists text/blob resource contents without a byte limit. A buggy or hostile MCP server can force large memory and disk writes through canvas creation, so the decode/write path needs an explicit size cap before storing app HTML.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces an MCP-provided iframe bridge that can call tools/resources, and the current patch still has concrete persisted-capability, stale-policy, broad-resource, and unbounded-content concerns.

Concerns:

- **[high] Persisted bridge capability metadata:** `src/agents/mcp-ui-resource.ts:62`
  - body: `sessionKey` and tool input/result metadata are serialized into canvas/chat state, which can outlive the live iframe and leak bridge capability context or sensitive tool payloads.
  - confidence: 0.93
- **[high] Stale app-enabled runtime policy:** `src/gateway/server-methods/mcp-app-proxy.ts:28`
  - body: The proxy can return a cached runtime before reloading current config, so bridge calls may continue after `mcp.apps.enabled` is disabled or server config changes.
  - confidence: 0.91
- **[high] Synthetic session runtime escape:** `src/gateway/server-methods/mcp-app-proxy.ts:39`
  - body: Creating `mcp-app:${sessionKey}` runtimes for restored views can split cleanup and ownership from canonical run sessions, leaving MCP processes or capabilities alive longer than intended.
  - confidence: 0.88
- **[high] Broad app resource proxy:** `src/gateway/server-methods/mcp-app-proxy.ts:107`
  - body: Iframe-supplied resource URIs are forwarded to the MCP runtime without an OpenClaw-side resource allowlist or content policy at the bridge boundary.
  - confidence: 0.87
- **[medium] Unbounded app HTML persistence:** `src/agents/mcp-ui-resource.ts:49`
  - body: MCP app HTML resource content is decoded and persisted without an explicit byte cap, creating memory and disk pressure risk from oversized server responses.
  - confidence: 0.82
