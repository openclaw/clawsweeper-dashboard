Full review comments:

- **[P1] Keep bridge capabilities out of persisted canvas metadata:** `src/agents/mcp-ui-resource.ts:51-56`
  - body: This embeds `runtime.sessionKey` into the MCP app metadata that is serialized with the canvas payload, and the same payload includes the tool input/result.   - confidence: 0.89
- **[P1] Revalidate runtimes before serving app bridge calls:** `src/gateway/server-methods/mcp-app-proxy.ts:25-31`
  - body: Returning the existing runtime here skips the current-config/fingerprint path, so disabling `mcp.apps.enabled` or changing/removing an MCP server can leave an already-rendered app view calling tools/resources through the stale runtime.
  - confidence: 0.9
- **[P2] Reuse canonical session IDs for restored runtimes:** `src/gateway/server-methods/mcp-app-proxy.ts:38-40`
  - body: When the session mapping is missing, this creates a new runtime under `mcp-app:${sessionKey}` instead of the canonical run session ID. Later normal cleanup is keyed by the real session, so restored app views can leave orphaned MCP runtimes or server processes behind.
  - confidence: 0.84

## Risks / Open Questions

- Persisting `runtime.sessionKey` and full tool input/result into canvas metadata can turn a live app bridge capability and sensitive MCP payloads into durable transcript-like state.
- App bridge calls can keep using stale runtimes after `mcp.apps.enabled` or `mcp.servers` changes, so disabling the feature, rotating credentials, or removing a server may not take effect for already-rendered app views.
- Restored app views can create runtimes under synthetic session IDs that normal session cleanup does not own, risking leaked MCP runtimes or processes.
- The branch adds a new `mcp.apps.enabled` config/default surface and new browser bridge dependency, so maintainers need an explicit trust-model and upgrade decision before merge.
