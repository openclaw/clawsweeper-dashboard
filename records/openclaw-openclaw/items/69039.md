## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep bridge capabilities out of persisted canvas metadata:** `src/agents/mcp-ui-resource.ts:55`
  - body: `fetchMcpAppView` writes `runtime.sessionKey` into `mcpApp` metadata, and that metadata is then serialized into assistant canvas JSON. Since that content can survive in transcript, export, or model-context surfaces, the PR turns the live app bridge capability for `mcp.*` calls into persisted content; bind bridge auth from active UI/server state or a short-lived server-side handle instead.
  - confidence: 0.9
- **[P1] Revalidate runtimes before serving app bridge calls:** `src/gateway/server-methods/mcp-app-proxy.ts:29-31`
  - body: `resolveRuntime` returns the mapped runtime from `getExisting` before loading current config or checking the runtime fingerprint. If an operator disables `mcp.apps.enabled`, removes a server, or rotates MCP config while an app view is open, the iframe can keep calling the old runtime and removed server; route app calls through a revalidation/adoption path before proxying.
  - confidence: 0.88
- **[P2] Bind restored app views to the canonical session runtime:** `src/gateway/server-methods/mcp-app-proxy.ts:38-40`
  - body: The fallback path creates a runtime under `mcp-app:${sessionKey}` when the session-key map misses. Normal chat turns and cleanup use the real session id, so restored app views can leave a second MCP runtime and server processes outside the session lifecycle; rebind to the canonical session id or explicitly adopt/dispose this runtime when the real session resumes.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-boundary concerns around persisted bridge capabilities, sensitive tool payload retention, and stale runtime authorization.

Concerns:

- **[high] Persisted app bridge capability:** `src/agents/mcp-ui-resource.ts:55`
  - body: `runtime.sessionKey` is copied into canvas metadata for the MCP app view, making a live bridge capability persist in assistant-rendered content rather than staying bound to the active UI session.
  - confidence: 0.9
- **[medium] Sensitive tool payloads stored in canvas JSON:** `src/agents/mcp-ui-resource.ts:87`
  - body: `toolInput` and `toolResult` are copied into the canvas JSON payload; MCP tool arguments and results can contain user, workspace, or service data that should not be broadly retained without a redaction policy.
  - confidence: 0.84
- **[high] Stale runtime can preserve old MCP access:** `src/gateway/server-methods/mcp-app-proxy.ts:29`
  - body: The app proxy returns an existing runtime without current config revalidation, so disabled apps or removed MCP servers may remain callable through an already-open bridge.
  - confidence: 0.88
- **[medium] Unscoped resource reads need policy review:** `src/gateway/server-methods/mcp-app-proxy.ts:102`
  - body: Once apps are enabled, `mcp.readResource` accepts any URI for the selected server rather than constraining reads to the resource that produced the app view or a reviewed resource policy.
