## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop persisting MCP session and tool payload data:** `src/chat/canvas-render.ts:23-29`
  - body: This metadata becomes durable canvas/chat state, so storing `sessionKey`, `toolInput`, and full `toolResult` can leak session identifiers and sensitive tool payloads through history, exports, previews, or logs. Keep only non-sensitive app identity in the document and authorize bridge calls from in-memory gateway/UI state.
  - confidence: 0.92
- **[P1] Bound MCP App resources before decoding or writing:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: A configured MCP server can return arbitrary text or base64 blob content, and this path decodes it into HTML and persists it as a canvas document without a raw or decoded byte cap. Enabling MCP Apps should reject oversized resources before decode and before persistence to avoid memory or disk exhaustion.
  - confidence: 0.9
- **[P1] Enforce MCP Apps resource CSP metadata:** `ui/src/ui/components/mcp-app-view.ts:249-250`
  - body: The ext-apps contract exposes `_meta.ui.csp` and `_meta.ui.permissions` with secure defaults, but the host uses a fixed CSP/sandbox while granting bridge capabilities. Parse the resource metadata and map it to CSP, iframe allow, and sandbox policy so apps only receive the server-declared network/resource/frame/permission surface.
  - confidence: 0.86

## Security Review

Status: needs_attention

## Risks / Open Questions

- The PR persists sessionKey plus raw tool input/result data into canvas/chat metadata, which can leak session identifiers and sensitive tool payloads through history, exports, previews, or logs.
- A configured MCP server can return arbitrarily large text or base64 HTML resources that the new path decodes and persists without an MCP-app-specific cap.
