## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop persisting MCP session and tool payload data:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: `buildMcpAppCanvasJson` serializes the MCP app metadata plus full tool input and result into canvas/chat JSON, and the view metadata already includes the runtime session key. That makes live app-session capability and potentially sensitive tool payloads durable in chat/canvas state instead of keeping them in the live bridge context.
  - confidence: 0.9
- **[P1] Bound MCP App resources before decoding or writing:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: The server-provided HTML path accepts either `text` or base64 `blob` content and decodes it without an explicit byte limit before creating a canvas document. A trusted-but-buggy or compromised enabled MCP server can force large memory allocation and persistent canvas growth.
  - confidence: 0.88
- **[P1] Apply resource CSP and iframe permissions:** `ui/src/ui/components/mcp-app-view.ts:249-250`
  - body: The iframe is rendered with fixed sandbox/CSP attributes and no `allow` permissions derived from the MCP Apps resource metadata. The dependency contract exposes resource `_meta.ui.csp`, `_meta.ui.permissions`, and `buildAllowAttribute`, so this can both deny legitimate apps and fail to enforce declared resource/network/frame boundaries.
  - confidence: 0.86
- **[P2] Keep restored app runtimes under canonical sessions:** `src/gateway/server-methods/mcp-app-proxy.ts:31-42`
  - body: When a restored view has only a session key mapping miss, the proxy creates a new `mcp-app:${sessionKey}` runtime. That runtime is separate from the normal run/session lifecycle, so cleanup and authorization state can drift from the canonical session that subsequent turns use.
  - confidence: 0.78
