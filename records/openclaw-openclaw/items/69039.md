## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep bridge capability data out of stored canvas JSON:** `src/agents/mcp-ui-resource.ts:62-67`
  - body: This persists `sessionKey` in `mcpApp`, and `buildMcpAppCanvasJson` then serializes that metadata together with full tool input/result into the tool-result JSON. That JSON becomes chat/canvas state, so a bridge capability and sensitive tool payloads can leak through stored transcripts, logs, exports, or replayed message content. Persist only non-sensitive app identifiers and use live UI/session context, or an explicitly redacted store, for bridge data.
  - confidence: 0.92
- **[P1] Revalidate MCP runtimes before bridge proxy calls:** `src/gateway/server-methods/mcp-app-proxy.ts:29-31`
  - body: `resolveRuntime` returns an existing runtime for a supplied session key without reloading current config, while the runtime fingerprint still excludes `mcp.apps.enabled`. If an operator disables MCP Apps or changes/removes a server while an app view is open, these gateway methods can keep using the stale runtime and continue app bridge calls contrary to current config. Route bridge calls through the same current-config validation path used by `getOrCreate`, and include the apps gate in the relevant fingerprint.
  - confidence: 0.86
- **[P1] Use canonical session IDs for restored app runtimes:** `src/gateway/server-methods/mcp-app-proxy.ts:38-40`
  - body: On a session-key mapping miss, the proxy creates a runtime under `mcp-app:${sessionKey}`. Later agent turns and cleanup use the real session id, so restored app views can leave separate runtimes and MCP server processes outside the canonical session lifecycle. Resolve the real current session id or reject the bridge call when no canonical live/restorable session can be proven.
  - confidence: 0.84
- **[P2] Bound MCP App resource size before persisting it:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: The PR decodes MCP App HTML from `text` or base64 `blob` and writes it into a canvas document without an MCP-App-specific size limit. A configured server can return a very large UI resource and force memory and disk growth during normal tool execution. Enforce a documented maximum before decoding/persisting and cover the rejection path in the MCP App resource tests.
  - confidence: 0.78
