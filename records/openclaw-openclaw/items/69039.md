## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep bridge capabilities out of stored canvas metadata:** `src/agents/mcp-ui-resource.ts:55`
  - body: `sessionKey` is serialized into the canvas manifest, and the same MCP app metadata is later copied into assistant-visible canvas JSON with tool input/result payloads. Those surfaces can persist in chat history, exports, or logs, so the bridge capability and sensitive tool data should come from in-memory/server-side context instead of stored message metadata.
  - confidence: 0.93
- **[P1] Revalidate runtimes before serving app bridge calls:** `src/gateway/server-methods/mcp-app-proxy.ts:29-31`
  - body: When a mapped runtime exists, `resolveRuntime` returns it before loading current config, so changing `mcp.apps.enabled` or removing/rotating an MCP server can leave app bridge calls using stale policy and stale server processes. Route bridge calls through the canonical config-fingerprint path or explicitly revalidate/dispose before serving them.
  - confidence: 0.9
- **[P1] Reuse canonical session IDs for restored app runtimes:** `src/gateway/server-methods/mcp-app-proxy.ts:39`
  - body: If the session-key mapping is missing, the proxy creates a runtime under `mcp-app:${sessionKey}`. Later agent turns and cleanup use the real session id, so restored app views can fork MCP session state and leave orphaned server processes outside normal session disposal.
  - confidence: 0.88
- **[P1] Constrain app-initiated resource reads:** `src/gateway/server-methods/mcp-app-proxy.ts:106-109`
  - body: `mcp.readResource` forwards an arbitrary iframe-supplied URI to the MCP server. Since the host advertises resource proxy support, app code should be limited to approved/advertised UI resources and bounded response sizes rather than becoming a generic resource read proxy for any configured server.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive MCP iframe bridge with concrete capability persistence, stale authorization/config, and resource-proxy concerns.

Concerns:

- **[high] Bridge capability persisted into chat/canvas metadata:** `src/agents/mcp-ui-resource.ts:55`
  - body: `sessionKey` plus tool input/result data are serialized into MCP app canvas metadata, creating an avoidable stored bearer-capability and sensitive-payload exposure surface.
  - confidence: 0.93
- **[high] Stale or synthetic runtime access can bypass current config intent:** `src/gateway/server-methods/mcp-app-proxy.ts:29`
  - body: The proxy can return an existing runtime without current config revalidation and can create fallback runtimes under synthetic ids, which can keep disabled or removed MCP server access alive outside canonical session lifecycle.
  - confidence: 0.88
- **[medium] App resource proxy is too broad:** `src/gateway/server-methods/mcp-app-proxy.ts:106`
  - body: The bridge forwards arbitrary app-supplied resource URIs to configured MCP servers without an allowlist, advertised-resource check, or response-size cap.
  - confidence: 0.84
- **[medium] MCP UI resource decoding and persistence are unbounded:** `src/agents/mcp-ui-resource.ts:38`
  - body: The PR decodes text/blob UI resources and writes them as canvas documents without a clear byte limit before decode or persistence, so a trusted-but-buggy MCP server can cause memory or disk pressure.
  - confidence: 0.8
