## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop persisting MCP session and tool payload data:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: `buildMcpAppCanvasJson` serializes `view.mcpApp` plus full tool input/result into canvas JSON; because `view.mcpApp` can include the live session key, this stores sensitive session-linked app data in chat/canvas state instead of keeping it in the live UI context or a redacted metadata shape.
  - confidence: 0.9
- **[P1] Bound MCP App resources before decoding or writing:** `src/agents/mcp-ui-resource.ts:49-53`
  - body: Server-controlled `text` or base64 `blob` HTML is accepted and decoded without an explicit size limit before being written as a canvas document. An enabled MCP server can exhaust memory or persistent canvas storage; cap both raw and decoded payload size before materialization.
  - confidence: 0.88
- **[P1] Apply resource CSP and iframe permissions:** `ui/src/ui/components/mcp-app-view.ts:249-250`
  - body: The app iframe uses a fixed sandbox/CSP and never consumes resource `_meta.ui.csp` or `_meta.ui.permissions` from the MCP Apps contract. That makes required connect/resource/frame domains and iframe `allow` policy unenforced or unavailable for apps that declare them.
  - confidence: 0.86
- **[P2] Keep restored app runtimes under canonical sessions:** `src/gateway/server-methods/mcp-app-proxy.ts:31-38`
  - body: When the session-key mapping is missing, the proxy creates a runtime under a synthetic `mcp-app:${sessionKey}` id. After a reload/restart, continuing the canonical session can create a second runtime and leave cleanup/authorization ownership split across two session ids.
  - confidence: 0.78
