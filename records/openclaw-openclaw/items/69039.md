## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop persisting MCP app session and tool data:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: `buildMcpAppCanvasJson` spreads `view.mcpApp`, which includes `sessionKey`, and then stores raw `toolInput`/`toolResult` in the canvas/chat JSON. That makes a bearer-like session capability and user/tool payloads durable in chat history, logs, exports, or rendered metadata. Keep session authority in in-memory UI context and persist only non-sensitive view identity.
  - confidence: 0.91
- **[P1] Enforce MCP resource CSP when hosting app HTML:** `ui/src/ui/components/mcp-app-view.ts:249-250`
  - body: The iframe only applies a static `navigate-to`/`form-action` CSP attribute, while the MCP Apps contract puts `connect-src`, resource, frame, and base-uri policy on the UI resource metadata for the host to enforce. Because the fetched HTML is persisted and served through the canvas host without a CSP header, compliant apps can break and undeclared network/resource access is not constrained by the server-declared policy.
  - confidence: 0.88
- **[P1] Bound MCP app resources before decode and persistence:** `src/agents/mcp-ui-resource.ts:49-53`
  - body: `fetchMcpAppView` accepts arbitrary text or base64 `blob` content, decodes it, and writes the HTML into a canvas document without raw or decoded byte limits. A trusted but buggy MCP server can consume large memory or disk when `mcp.apps.enabled` is on; add enforced limits before decode and before persistence.
  - confidence: 0.89
