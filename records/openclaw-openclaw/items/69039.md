## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep MCP App payloads out of stored canvas JSON:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: `buildMcpAppCanvasJson` serializes the view metadata together with raw `toolInput` and `toolResult`; the spread also carries the view `sessionKey` created earlier. Those values can contain user data or bridge routing capability and then live in chat/canvas history, logs, exports, and future transcript projections, so persist only minimal non-sensitive metadata and keep session/runtime data outside the transcript payload.
  - confidence: 0.9
- **[P1] Cap MCP App resources before decoding and writing:** `src/agents/mcp-ui-resource.ts:49-53`
  - body: The MCP UI resource path accepts arbitrary text or base64 blob content and decodes it before writing it as a canvas document. With MCP Apps enabled, a configured server can return very large HTML and exhaust memory or disk; reject oversized text/blob payloads before decode/write and add focused tests for the limit.
  - confidence: 0.88
- **[P2] Render disabled MCP Apps through the canvas fallback:** `ui/src/ui/chat/tool-cards.ts:478-480`
  - body: This early return makes saved MCP App canvas previews disappear whenever `mcp.apps.enabled` is false. The PR promises the disabled path falls back to normal canvas/embed rendering, so this should render the ordinary preview frame instead of returning `nothing`.
  - confidence: 0.86
- **[P2] Bind restored MCP Apps to canonical session ownership:** `src/gateway/server-methods/mcp-app-proxy.ts:31-42`
  - body: When the session-key mapping is missing, the proxy creates a runtime under `mcp-app:${sessionKey}`. A later agent turn for the same session uses the real run `sessionId`, so restored views and active runs can own separate MCP runtimes/processes until idle sweep; rehydrate or require canonical session binding before creating the runtime.
  - confidence: 0.82
