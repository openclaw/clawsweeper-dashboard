## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cap MCP App resources before decoding and writing:** `src/agents/mcp-ui-resource.ts:49-54`
  - body: `fetchMcpAppView` accepts arbitrary `text` or base64 `blob`, decodes it, and writes the result as a canvas HTML document. A configured MCP server can return a very large app resource and exhaust memory or disk before any OpenClaw host-side limit runs; add byte caps before decode/write and cover oversized text/blob responses.
  - confidence: 0.92
- **[P1] Avoid persisting session keys and full tool payloads:** `src/agents/mcp-ui-resource.ts:96-99`
  - body: The canvas JSON spreads `view.mcpApp` and then stores raw `toolInput` plus the full `toolResult`, so chat/canvas history carries session-scoped app metadata and potentially sensitive tool data. Persist only the minimal redacted display metadata and keep any live session capability in a scoped runtime-side association.
  - confidence: 0.86
- **[P2] Render disabled app previews through the canvas fallback:** `ui/src/ui/chat/tool-cards.ts:478-480`
  - body: When `mcp.apps.enabled` is false, this returns `nothing` for a persisted MCP App preview, so the already-created canvas iframe disappears instead of using the normal non-bridge canvas preview path promised in the PR body. Let disabled app previews fall through to `renderPreviewFrame` and update the test that currently locks in the blank preview.
  - confidence: 0.9
