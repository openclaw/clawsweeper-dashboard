## Summary

Keep open: the branch still reconciles the dropdown from request-time `sessions.patch.resolved`, which does not prove or fix the linked effective-runtime fallback mismatch. The supplied screenshots show the UI shell disconnected from the Gateway, not a real dropdown fallback/default flow.

## What This Changes

The PR changes `switchChatModel` in `ui/src/ui/chat/session-controls.ts` to capture `sessions.patch`, read its `resolved` model for explicit dropdown selections, and rewrite the per-session `chatModelOverrides` cache.

## Best Possible Solution

Repair the PR so the dropdown reconciles from a true effective runtime/session state source, or explicitly narrow it to request-time model normalization without closing the fallback issue.

## Reproduction Assessment

Yes at source level: current main prefers cached dropdown overrides before server session rows, and the PR reads request-time patch metadata rather than a completed runtime fallback source. I did not run a live WebChat/provider fallback reproduction in this read-only review.

## Solution Assessment

No. The live patch is too early in the lifecycle because `sessions.patch.resolved` is computed immediately after applying the selected model patch; the safer fix needs refreshed/effective runtime state or a maintainer-approved selected-versus-effective contract.
