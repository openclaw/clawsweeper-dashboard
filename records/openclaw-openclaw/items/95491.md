## Summary

Keep open: the branch is not merge-ready because the reconciliation treats `state.sessionsResult` as a fresh effective-runtime source, but current session loading can return before a refresh completes and session rows still prefer the selected model over runtime identity. The supplied screenshots show a disconnected dashboard and helper-level checks, not the claimed WebChat fallback/default flow.

## What This Changes

The PR updates WebChat `switchChatModel` to refresh sessions after `sessions.patch` and rewrite `chatModelOverrides` from the refreshed session row for explicit model selections.

## Best Possible Solution

Use a fresh authoritative model source for this UI cache, backed by regression coverage for pending session loads and selected-versus-effective model state, then add redacted real WebChat/Gateway proof for fallback and Default behavior.

## Reproduction Assessment

Yes, source-level: current code shows the PR reads a session row after a refresh helper that can return before fresh data is available, and current row projection is selected-first rather than effective-runtime-first. I did not run a live provider fallback browser flow in this read-only review.

## Solution Assessment

No. Reading `state.sessionsResult` immediately after `refreshSessionOptions` is neither guaranteed fresh nor guaranteed to be the actual backend fallback model; a safer fix needs a fresh authoritative patch result or an explicit selected/effective model contract.
