## What This Changes

The PR changes WebChat’s model switch handler to refresh sessions after `sessions.patch` and rewrite the per-session dropdown cache from the refreshed session row or `patched.resolved`.

## Best Possible Solution

Use a true effective-runtime source, or reconcile/clear the dropdown cache when runtime fallback is persisted, while preserving Default and pending-switch semantics with focused WebChat/Gateway proof.

## Reproduction Assessment

Yes, at source level: current main is cache-first, the PR reconciles around immediate patch/session-row state, and the real fallback persistence path is later in the agent runtime. I did not run a live provider fallback in this read-only review.

## Solution Assessment

No. This is a plausible partial request-time normalization, but the best fix needs a real effective-runtime source or fallback-persistence reconciliation rather than only the immediate post-patch session row.
