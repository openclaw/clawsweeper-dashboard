## Summary

Keep open: current main and the latest release still pin the ACPX stack whose ACP SDK message dispatcher can throw the reported TypeError on a numeric inbound message, and no canonical OpenClaw fix PR appears to own the remaining work.

## What This Changes

The issue reports ACP `sessions_spawn(runtime="acp", agentId="claude")` failing when ACPX receives a numeric or non-object message, leaving the spawned session output empty.

## Best Possible Solution

Fix the non-object message guard at the ACP SDK/acpx boundary, consume the approved fixed dependency in `@openclaw/acpx`, and cover request, response, primitive, null, string, and array payloads plus a live Claude ACP smoke.

## Reproduction Assessment

Yes. The current pins resolve to an ACP SDK dispatcher that uses `"method" in message`, and a numeric `1` payload produces the exact TypeError text; I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

Yes, the reporter's object guard is the right fix shape, but the best layer is the ACP SDK/acpx boundary rather than catching the crash after OpenClaw receives an unhandled runtime failure.
