Keep open: current main and v2026.6.8 still pin the ACPX/Claude ACP dependency path whose dispatcher can throw the reported `in`-operator TypeError on primitive messages, and the only direct fix PR is closed unmerged and conflicting.

## What This Changes

The issue asks OpenClaw ACPX/Claude ACP handling to tolerate primitive ACP adapter messages so `sessions_spawn(runtime="acp")` does not fail with an `in`-operator TypeError.

## Best Possible Solution

Apply a maintainer-approved fix at the ACP SDK/acpx or local dependency-patch boundary, with regression coverage for request-like, response-like, primitive, null, string, and array messages plus a live ACP smoke when feasible.

## Reproduction Assessment

Yes, source-reproducible: the pinned dependency applies the `in` operator to `message` before narrowing it to an object, and Node produces the reported TypeError for primitive `1`. I did not run a live Claude ACP session in this read-only pass.

## Solution Assessment

Yes, the best direction is a guard at the ACP SDK/acpx dependency boundary rather than catching an already-unhandled runtime failure in OpenClaw. The closed unmerged PR shows the likely patch shape but is not a viable canonical fix as-is.
