## What This Changes

The issue requests concurrent or fast-lane same-session agent turns so quick follow-up messages are not blocked behind a long-running Discord or chat turn.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/57239 or consolidate the remaining design with https://github.com/openclaw/openclaw/issues/53023, preserving default per-session serialization unless an explicit, tested fast-lane policy is approved.

## Reproduction Assessment

Yes, at source level. Current main still routes ordinary same-session follow-up work through enqueue/wait behavior and per-session lanes rather than starting a second normal tool-capable turn.

## Solution Assessment

Unclear. A fast-lane or concurrent-turn path could solve the latency problem, but the best version needs maintainer decisions around ordering, session state, tool safety, and whether it belongs as explicit /btw behavior, a configurable lane policy, or a narrow quick-reply mode.
