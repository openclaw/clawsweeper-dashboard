## Summary

Keep open: current main and the latest release still serialize one active run per session. Default steering, followup, collect, and interrupt reduce some latency, but they do not provide the requested parallel or fast-lane same-session turns, and the related PRs either merged only narrower steering behavior or remain open/conflicting for narrower admission and steering fixes.

## What This Changes

The issue requests concurrent same-session agent turns so quick follow-up messages can run without waiting behind a long active turn.

## Best Possible Solution

Keep this as the canonical design item for same-session fast-lane or parallel turns, resolve the narrower steering/admission PRs first, then choose a safety model that preserves transcript and tool-ordering guarantees before adding any config or mode surface.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: start an active same-session run, then send another normal prompt; current main routes that prompt through steering, interrupt, or queued followup/collect behavior instead of starting a second same-session run. I did not run a live Discord timing scenario in this read-only review.

## Solution Assessment

Unclear. Parallel slots or a fast-lane mode are plausible, but the best solution needs maintainer decisions about transcript ordering, tool safety, rate limits, abort ownership, config surface, and how it composes with the open steering/admission fixes.
