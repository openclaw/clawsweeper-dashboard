## Summary

Keep open: current main and the latest release still preserve one active run per session. Steering, followup, collect, and interrupt mitigate some waits, but they do not implement the requested same-session parallel or fast-lane turns, and the open implementation candidates still need maintainer design/review.

## What This Changes

The issue requests concurrent or fast-lane same-session message handling so short prompts are not blocked behind long-running agent turns.

## Best Possible Solution

Keep this as the canonical design item for same-session fast-lane or parallel turns, resolve the narrower steering/admission candidates, then choose a bounded safety model before adding any config or mode surface.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: start an active same-session run, then send another normal prompt. Current main routes it through steering, interrupt, or queued followup/collect behavior instead of starting a second same-session run.

## Solution Assessment

Unclear. Parallel slots or a fast-lane mode are plausible, but the best solution needs maintainer decisions about transcript ordering, tool safety, rate limits, abort ownership, config surface, and how it composes with steer/followup/collect/interrupt.
