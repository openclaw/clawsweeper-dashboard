## Summary

Keep open. Current main still serializes normal tool-capable turns per session and only offers steering/followup/interrupt behavior plus the narrower tool-less `/btw` side-question path. An open unmerged PR already references this issue with closing syntax, so this should remain open until that PR is reviewed, landed, replaced, or closed.

## What This Changes

The issue requests per-session concurrent or async agent turns so quick Discord or chat follow-ups can respond while a long turn is still running.

## Best Possible Solution

Keep the issue open while maintainers review the linked fast-lane PR as the narrow implementation candidate. The durable end state should either land a bounded active-session quick-reply path with tests or explicitly design safe per-session concurrency across session isolation, transcript consistency, tool conflict handling, and rate-limit controls.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a long normal run in one session followed by another normal message in the same session; current main routes the second message through steering/followup/interrupt behavior rather than starting an independent tool-capable turn.

## Solution Assessment

Unclear. The open fast-lane PR is a plausible narrow mitigation, but the requested full concurrent turns need deeper safety design; reviewing that PR is better than closing this issue or queueing duplicate repair work.
