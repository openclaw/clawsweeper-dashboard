## Summary

Keep open. Current main has default steering and `/btw` side-question support, but normal per-session agent runs are still serialized and there is still an open closing PR for the fast-lane subset, so this should stay open for maintainer scheduler/product review rather than be closed as implemented.

## What This Changes

Request concurrent agent turns within a chat session so quick follow-ups can run while a long turn is still active.

## Best Possible Solution

Keep the current safe serialization/steering defaults until maintainers choose the session scheduler model, then land either the linked fast-lane PR or a broader design that includes transcript, tool-safety, rate-limit, and upgrade tests.

## Reproduction Assessment

Yes, source-level rather than live-run: current main still routes normal active-session messages through steering or follow-up queue paths and serializes embedded agent runs through a session lane. I did not run a live Discord timing reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path: full concurrent normal turns is a product/scheduler design, not a narrow bug fix. The safer path is maintainer approval of the desired mode plus focused safety tests, with the linked fast-lane PR treated as one candidate subset.
