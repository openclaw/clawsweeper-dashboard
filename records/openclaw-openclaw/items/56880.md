## Summary

Keep open. Current main and the latest release still intentionally serialize normal runs per session; steering, `/queue interrupt`, and `/btw` cover useful subsets but do not provide general concurrent per-session turns, and the linked fast-lane PR is still open and conflicting rather than a merged fix.

## What This Changes

The issue requests concurrent or fast-lane message handling within a single chat session so short follow-ups are not blocked behind a long-running agent turn.

## Best Possible Solution

Keep this as the canonical session-scheduler/product issue until maintainers choose a safe model, then land or replace the linked fast-lane work with transcript, tool-safety, rate-limit, and upgrade coverage.

## Reproduction Assessment

Yes, source-level: current main routes active-session normal messages through steering or follow-up queue paths and serializes embedded agent runs through a session lane. I did not run a live Discord timing reproduction in this read-only review.

## Solution Assessment

Unclear: full concurrent per-session turns are a product and scheduler design, not a narrow bug fix. The safer path is a maintainer-selected mode or fast-lane subset with explicit safety tests and upgrade behavior.
