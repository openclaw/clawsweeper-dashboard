## Summary

Keep open: current main and the latest release still guarantee one active run per session, with steering/followup/collect/interrupt as partial latency tools rather than true parallel or fast-lane same-session turns. The related PRs either merged narrower steering behavior, remain open for narrower admission/steering fixes, or are conflicting/unproven fast-lane attempts, so this still needs maintainer product and safety direction.

## What This Changes

The issue requests concurrent or fast-lane handling for same-session messages so short follow-ups are not blocked behind long-running agent turns.

## Best Possible Solution

Keep this as the canonical design item for same-session parallelism, and decide whether to extend the existing queue/steer/interrupt model with a safe fast-lane or parallel-turn mode after the narrower active-run steering and admission fixes are resolved.

## Reproduction Assessment

Yes, source inspection gives a high-confidence way to reproduce the reported current behavior: start an active run, send another same-session message, and current main routes through steering/followup/collect/interrupt rather than starting a concurrent quick turn. I did not run a live Discord timing scenario in this read-only review.

## Solution Assessment

Unclear. Parallel slots or fast-lane replies are plausible, but the best solution needs maintainer decisions about session state, tool safety, rate limits, config surface, and how it composes with the open steering/admission fixes.
