## Summary

Keep open. Current main still sends Grok web_search through the shared 30 second timeout on the xAI Responses path, and the related timeout PR remains open rather than merged or shipped.

## What This Changes

The issue requests a provider-appropriate default timeout for bundled Grok web_search so xAI Responses searches are not aborted at the shared 30 second boundary.

## Best Possible Solution

Land or refresh a narrow xAI-owned fix that preserves the shared 30 second default for other providers, gives Grok a longer default only when no user timeout is configured, and adds focused regression coverage plus docs/schema updates if a provider-specific timeout knob is exposed.

## Reproduction Assessment

Yes. The reporter gave direct `requestXaiWebSearch(...)` steps with timing at the 30 second boundary, and current main confirms that an unset Grok timeout still flows as 30 seconds into the xAI Responses request path.

## Solution Assessment

Yes. The narrow maintainable direction is provider-owned: adjust the Grok fallback or expose an xAI/Grok-specific setting without raising the shared web_search timeout for every provider; #43609 appears aligned but remains unmerged.
