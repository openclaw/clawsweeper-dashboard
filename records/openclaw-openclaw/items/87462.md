- `impact:auth-provider`: The failure is in auth profile cooldown state and provider/model fallback routing.

## Summary

Keep open: current main and the reported release still have a source-visible path where timeout cooldowns are profile-wide while model fallback checks availability per candidate model, so a transient timeout can make same-provider fallbacks look unavailable.

## What This Changes

The issue reports that a transient Google timeout marks every Google auth profile/fallback candidate as unavailable, exhausting the same-provider model fallback chain without matching Google API errors.

## Best Possible Solution

Keep the issue open and add a narrow regression fix so transient timeout handling cannot make same-provider model fallbacks inherit a shared profile-wide cooldown unless the failure is genuinely profile-wide.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced: current code stores timeout cooldowns without cooldownModel and later checks all same-provider fallback candidates against that profile-wide cooldown. The live Google claim still needs runtime proof or a focused regression test.

## Solution Assessment

Yes, the best fix is a bounded auth/fallback repair rather than a new config option: preserve profile-wide disables for auth/billing, but prevent transient timeouts from poisoning unrelated same-provider fallback models.
