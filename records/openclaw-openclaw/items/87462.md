## Summary

Keep open: current main and the reported v2026.5.26 release still have a source-visible path where timeout cooldowns are profile-wide while fallback/auth selection checks availability per candidate model, so a transient timeout can make same-provider Google fallbacks appear unavailable before a useful provider call happens.

## What This Changes

The issue reports that a transient timeout on one Google model marks the shared Google auth profile unavailable and exhausts the configured same-provider fallback chain.

## Best Possible Solution

Patch core auth-profile/fallback handling so provider-started transient timeouts do not make unrelated same-provider model fallbacks unavailable, and add a focused Google-style regression test.

## Reproduction Assessment

No live high-confidence repro is available yet. Source inspection shows timeout cooldowns are profile-wide while fallback selection is model-aware, which plausibly reproduces the reported same-provider chain exhaustion.

## Solution Assessment

Yes, the best path is a narrow core auth-profile/model-fallback repair with regression coverage. A new config option or plugin-level workaround would add surface area without addressing the shared fallback contract.
