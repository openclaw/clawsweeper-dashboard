## Summary

Keep this issue open. Current main still implements TUI `/new` as a local `setSession(tui-<uuid>)` switch, while the hook-visible `command:new` path is tied to `sessions.reset`; an open, unmerged PR already references this issue with closing syntax and should be reviewed or replaced rather than closing the tracker early.

## What This Changes

Restore hook-visible new-session behavior for TUI `/new` while preserving the isolated per-client `tui-<uuid>` session key behavior.

## Best Possible Solution

Land PR #49945 or an equivalent focused fix on main: keep the unique `tui-<uuid>` session key, route that key through the existing `sessions.reset` path with reason `new`, and update the TUI command-handler regression test so `/new` covers both session isolation and hook visibility while `/reset` remains an in-place reset.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `/new` currently calls only local `setSession(uniqueKey)`, while documented and tested `command:new` hook delivery is behind `sessions.reset(reason="new")`. I did not run a live TUI hook because this review was read-only, but the current code path directly matches the reported failure.

## Solution Assessment

Yes. Reusing `sessions.reset` for the newly allocated TUI session key is the narrowest maintainable path because it preserves the existing gateway hook contract instead of inventing a second synthetic event path in the TUI.
