## Summary

Keep open: current main and v2026.6.10 still route TUI `/new` through local `setEmptySession`, while gateway `sessions.reset` remains the hook-emitting path. Closed upstream fix attempts and a conflicting fork PR do not provide a safe canonical close path.

## What This Changes

The issue reports that TUI `/new` creates isolated `tui-*` sessions but no longer emits hook-visible `command:new` behavior for hooks such as session-memory and command-logger.

## Best Possible Solution

Restore TUI `/new` through the existing gateway/session lifecycle so hooks observe the session being left, while preserving unique `tui-*` isolation and avoiding a parallel TUI-only hook event or public hook-source override.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: TUI `/new` calls local `setEmptySession(uniqueKey)`, while docs and gateway code show `command:new` hooks are emitted through gateway reset/create hook paths.

## Solution Assessment

Yes on the requested end state, but current main is not the solution. The safest fix should reuse the gateway/session lifecycle with correct previous-session attribution instead of emitting a synthetic TUI-only event or resetting the wrong key.
