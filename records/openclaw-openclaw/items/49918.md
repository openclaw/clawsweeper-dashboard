## Summary

Keep this issue open: current main still has the reported TUI `/new` hook gap, and an open linked fix PR already owns the implementation path, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for TUI `/new` to keep isolated `tui-<uuid>` sessions while still emitting hook-visible `command:new` new-session behavior.

## Best Possible Solution

Land or replace the open linked PR with a focused fix that preserves isolated TUI `tui-*` session switching while emitting the existing hook/lifecycle contract for the session being left, with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows TUI `/new` only switches local session state, while the documented and tested `command:new` hook path is behind `sessions.reset(reason="new")`.

## Solution Assessment

Yes, with a constraint. Reusing the existing reset hook contract is the narrow maintainable direction, but the implementation should attribute hooks to the session being left before switching to the isolated TUI key.
