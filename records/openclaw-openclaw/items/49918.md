## Summary

Keep open: current main and the latest shipped release still route TUI `/new` through a local empty-session switch, so the reported hook-visible `command:new` regression remains source-reproducible. The issue is valid, but an open closing PR already owns the proposed repair path, so this should stay open until that PR is reviewed, refreshed, replaced, or closed.

## What This Changes

The issue asks OpenClaw to restore hook-visible new-session/reset behavior for TUI `/new` while preserving isolated `tui-<uuid>` sessions.

## Best Possible Solution

Refresh, review, or replace the linked fix so TUI `/new` fires the existing hook-visible reset/new-session lifecycle for the session being left while keeping isolated `tui-<uuid>` session keys.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: TUI `/new` reaches `setEmptySession(uniqueKey)` without `client.resetSession`, while gateway `sessions.reset` is the path that emits `command:new` hooks. No live TUI hook run was executed under the read-only review constraint.

## Solution Assessment

No, current main is not the best solution because it preserves TUI isolation but bypasses the hook-visible reset path. The safer fix should reuse the existing session reset or parent-session hook lifecycle instead of adding a parallel TUI-only event emitter.
