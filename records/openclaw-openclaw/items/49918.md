## Summary

Keep this issue open: current main and the latest release still make TUI `/new` create only a local isolated empty `tui-<uuid>` session, while the documented hook contract still says `/new` should fire `command:new` and session-memory hooks. The earlier target-repo fix PRs are closed unmerged, and the remaining fork PR is not a target-repo landing path.

## What This Changes

The issue asks OpenClaw to restore hook-visible `command:new` behavior for TUI `/new` without losing the per-client `tui-<uuid>` session isolation.

## Best Possible Solution

Land a narrow target-repo TUI fix that preserves per-client `tui-<uuid>` isolation while routing `/new` through the existing hook-visible session lifecycle, with regression tests for both isolation and `command:new` emission.

## Reproduction Assessment

Yes from source inspection: current TUI `/new` calls `setEmptySession` only, while `/reset` calls `client.resetSession` and docs/tests define that reset path as hook-visible. I did not run a live TUI hook session because this review is read-only.

## Solution Assessment

Yes, the best fix is a narrow TUI/session-lifecycle repair that keeps the unique `tui-<uuid>` key and restores the existing `command:new` hook contract. Reverting to the old shared-session reset would risk reintroducing the cross-client broadcast bug fixed by the isolation change.
