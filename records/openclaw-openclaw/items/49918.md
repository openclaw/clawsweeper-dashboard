## Summary

Keep open: current main and the latest release still route TUI `/new` through a local empty-session switch, so the hook-visible `command:new` regression remains source-reproducible. ## What This Changes

The issue requests restoring hook-visible `/new` behavior for TUI sessions while preserving isolated per-client `tui-<uuid>` session keys.

## Best Possible Solution

Restore TUI `/new` to emit the existing hook-visible reset/new-session lifecycle for the session being left, while keeping isolated `tui-<uuid>` session keys and reusing the gateway/session lifecycle contract rather than adding a parallel TUI-only event path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current TUI `/new` calls `setEmptySession(uniqueKey)` without `client.resetSession`, while gateway reset is the path that emits `command:new` hooks.

## Solution Assessment

Yes on the requested end state, but current main is not the solution. The maintainable fix should restore the existing gateway/session hook lifecycle with correct previous-session attribution rather than emitting a synthetic TUI-only hook event.
