## Summary

Keep this issue open. Current main still routes TUI `/new` through a local `setSession(tui-<uuid>)` switch instead of the gateway reset path that emits `command:new`, and open PR #49945 already references this issue with closing syntax, so the tracker should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests restoring hook-visible `command:new` behavior for TUI `/new` while preserving isolated per-client `tui-<uuid>` sessions.

## Best Possible Solution

Land PR #49945 or an equivalent focused fix that keeps the unique TUI session key but routes `/new` through the existing reset hook contract with regression coverage for both isolation and `command:new` visibility.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `/new` currently calls local `setSession(uniqueKey)`, while `command:new` hook delivery is implemented and tested behind `sessions.reset(reason="new")`.

## Solution Assessment

Yes. Reusing the existing `sessions.reset` path for the new TUI session key is the narrowest maintainable fix because it preserves the documented hook contract without adding a second synthetic event path.
