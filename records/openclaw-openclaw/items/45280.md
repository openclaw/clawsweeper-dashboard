## Summary

Keep this issue open. Current `main` still has phone-number-scoped voice-call sessions in the classic response path, no `sessionScope` config or manifest field, and realtime consults still fall back to caller/callee phone when no call-level session key exists. The related #59245 and #71262 context is useful but does not supersede this narrower per-call isolation request.

## What This Changes

The issue requests a voice-call plugin `sessionScope` option so operators can choose current per-phone memory or per-call isolation keyed by call id.

## Best Possible Solution

Implement this in the bundled voice-call plugin as a strict `sessionScope` enum with `per-phone` as the default and `per-call` as an opt-in, resolve and persist `call.sessionKey` when each call record is created, and reuse that key from both classic response generation and realtime consult. Cover repeated calls from the same phone in tests for both the default shared-history behavior and the new isolated per-call behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: two calls with the same `from` value both pass through `generateVoiceResponse`, which derives the same `voice:<normalizedPhone>` key regardless of `callId`, and existing tests assert that phone-derived key.

## Solution Assessment

Yes. A plugin-owned `sessionScope` option with `per-phone` as the compatibility default is the narrowest maintainable fix; moving this into core session policy or changing the default would be broader and riskier than the reported receptionist use case requires.

## Repair Work Prompt

Fix the voice-call plugin per-call session isolation request. Add a strict `sessionScope` config option to the bundled `voice-call` plugin with values `per-phone` and `per-call`; keep `per-phone` as the default. Resolve a single voice call session key when each `CallRecord` is created: phone-derived for `per-phone`, `voice:call:${callId}` or another documented call-id-derived key for `per-call`. Persist that key on the call record and reuse it from both `generateVoiceResponse` and realtime consult instead of recomputing phone scope independently. Update `VoiceCallConfigSchema`, config normalization, `openclaw.plugin.json` configSchema/uiHints, docs, README if needed, and `CHANGELOG.md` with user-facing credit to @pondcountry. Add focused tests proving repeated inbound calls from the same phone share history by default and use separate session keys with `sessionScope: "per-call"`; also cover realtime consult using the persisted key. Do not change the default behavior, do not move this policy into core session routing, and do not add bundled-plugin IDs to core.
