## Summary

Keep open. Current main still lacks a user-configurable per-call session scope for the voice-call plugin: classic responses are keyed by caller phone number, realtime consult falls back to phone identity whenever one exists, and neither the runtime schema nor plugin manifest accepts the proposed option. The related realtime consult/tool work and broader outbound task-call request do not supersede this narrower memory-isolation request.

## What This Changes

Add a voice-call plugin session-scope option so operators can choose per-call session keys instead of sharing conversation history by caller phone number.

## Best Possible Solution

Implement the option in the bundled voice-call plugin as a strict enum, defaulting to the current per-phone behavior. Resolve and persist the effective call session key when a call record is created, reuse that key in both classic response generation and realtime consult, and update manifest metadata, docs, and focused tests for both shared-history and isolated-history modes.

## Repair Work Prompt

Implement openclaw/openclaw#45280 in the bundled voice-call plugin. Add a strict `sessionScope` config enum such as `"per-phone" | "per-call"`, defaulting to the current `per-phone` behavior. Expose it in `VoiceCallConfigSchema`, `extensions/voice-call/openclaw.plugin.json` config schema/UI metadata, and `docs/plugins/voice-call.md`. Resolve the effective session key once when a call record is created and persist it as `CallRecord.sessionKey`; for `per-phone`, keep the current `voice:<normalizedPhone>` behavior when a phone number exists, and for `per-call`, use a call-id-scoped key such as `voice:call:<callId>`. Reuse the persisted key in both `generateVoiceResponse` and realtime consult. Preserve existing outbound custom `sessionKey` behavior and do not change the default away from phone-scoped sessions. Add focused tests proving repeated inbound calls from the same phone share history when unset/defaulted and are isolated when `sessionScope: "per-call"`; update realtime consult tests so it uses the persisted call session key. Add a single-line user-facing changelog entry under the active version with `Thanks @pondcountry`. Do not move voice-call-specific policy into core.
