## Summary

Keep open. Current main still matches the report: `repairLaunchAgentBootstrap` defaults to `repaired` after a successful `bootstrap` but then unconditionally runs `launchctl kickstart -k`, and the existing unit test still asserts that fresh-bootstrap sequence.

## What This Changes

Avoid running `launchctl kickstart -k` after `repairLaunchAgentBootstrap` successfully bootstraps a fresh macOS LaunchAgent, while preserving the bounce for already-loaded services.

## Best Possible Solution

Gate `kickstart -k` to the `already-loaded` branch, leave a fresh bootstrap as a successful `repaired` result, and cover both branches with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The high-confidence source reproduction is the current mocked fresh-bootstrap test path in `src/daemon/launchd.test.ts`, which returns `repaired` and records a later `kickstart -k`; the reporter also supplied live macOS logs showing the same sequence.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable change because it preserves the existing bounce when bootstrap reports an already-loaded job while avoiding a needless SIGTERM of a freshly loaded RunAtLoad agent.
