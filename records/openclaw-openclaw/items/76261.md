## Summary

Keep open. Current `main` still has the reported launchd repair behavior: a successful bootstrap takes the `repaired` path but immediately runs `launchctl kickstart -k`, and the existing unit test currently asserts that kill-and-restart sequence for a fresh bootstrap.

## What This Changes

The issue reports that macOS LaunchAgent bootstrap repair kills a freshly bootstrapped Gateway by always running `launchctl kickstart -k` after `bootstrap`.

## Best Possible Solution

Gate `kickstart -k` to the `already-loaded` bootstrap branch, keep fresh bootstrap as a successful `repaired` result, and add regression coverage for both paths.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with mocked `bootstrap` success in `src/daemon/launchd.test.ts`, `repairLaunchAgentBootstrap` currently records a later `kickstart -k` call on the `repaired` path.

## Solution Assessment

Yes. The issue’s proposed direction is the narrowest maintainable fix because it preserves the existing bounce for genuinely already-loaded jobs while avoiding a needless SIGTERM of a fresh RunAtLoad process.

## Repair Work Prompt

Fix the macOS LaunchAgent bootstrap repair race reported in openclaw/openclaw#76261. In `src/daemon/launchd.ts`, change `repairLaunchAgentBootstrap` so `launchctl kickstart -k` runs only when `bootstrap` reports the service was already loaded (`status: "already-loaded"`); when `bootstrap` succeeds and returns `status: "repaired"`, return success without killing the freshly spawned Gateway. Update `src/daemon/launchd.test.ts` so the fresh-bootstrap path asserts no kickstart call, while exit 130 / "already exists in domain" still kickstart and kickstart failures remain typed on that branch. Add a single-line user-facing `CHANGELOG.md` fix entry. Do not change broader restart semantics, runtime-deps locking, launchd install/stop behavior, or plugin staging logic.
