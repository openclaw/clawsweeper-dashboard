## Summary

Keep open. Current `main` still matches the report: a successful fresh `bootstrap` leaves the repair status as `repaired`, then `repairLaunchAgentBootstrap` immediately runs `launchctl kickstart -k`, and the existing unit test still asserts that sequence.

## What This Changes

The issue requests skipping `launchctl kickstart -k` after a fresh LaunchAgent bootstrap while preserving the bounce for already-loaded launchd jobs.

## Best Possible Solution

Gate `kickstart -k` to the `already-loaded` repair branch, return `repaired` directly after a successful fresh bootstrap, and add regression tests for both branches.

## Reproduction Assessment

Yes. The current mocked fresh-bootstrap test path returns `{ ok: true, status: "repaired" }` while recording a later `kickstart -k`, and the reporter supplied live macOS logs showing the same sequence.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it preserves the existing bounce for already-loaded jobs while matching `installLaunchAgent`'s established no-kickstart-after-bootstrap behavior.
