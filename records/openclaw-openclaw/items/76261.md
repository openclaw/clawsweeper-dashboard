## Summary

Keep open. Current `main` still matches the report: a fresh `bootstrap` leaves `repairStatus` as `repaired`, but `repairLaunchAgentBootstrap` immediately runs `launchctl kickstart -k`, and the current unit test still asserts that sequence.

## What This Changes

The issue asks OpenClaw to stop `repairLaunchAgentBootstrap` from `kickstart -k`-terminating a newly bootstrapped macOS LaunchAgent while preserving the already-loaded bounce path.

## Best Possible Solution

Gate `kickstart -k` to the `already-loaded` bootstrap branch, let a successful fresh bootstrap return `repaired` without a bounce, and add regression coverage for both paths.

## Reproduction Assessment

Yes. The high-confidence source reproduction is the current mocked fresh-bootstrap unit path in `src/daemon/launchd.test.ts`, which returns `repaired` and records a later `kickstart -k`; the reporter also supplied live macOS logs showing the same sequence.

## Solution Assessment

Yes. The proposed conditional kickstart is the narrowest maintainable fix because it preserves the existing already-loaded restart behavior while avoiding a needless SIGTERM of a freshly loaded RunAtLoad agent.
