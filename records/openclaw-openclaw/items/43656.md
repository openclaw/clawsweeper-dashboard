## Summary

Keep open; current main does not implement cross-Gateway session routing, but this PR is not ready to merge because it bypasses existing Gateway URL guardrails and has blocking remote spawn/session semantics defects.

## What This Changes

The PR adds a `gateway.peers` config map plus `gateway`, `gatewayUrl`, and `gatewayToken` routing paths for `sessions_send` and `sessions_spawn`.

## Best Possible Solution

Define an additive trusted Gateway federation contract first, then rebase the session routing work to preserve URL/auth guardrails, idempotency, lane behavior, docs/changelog, and regression coverage.

## Reproduction Assessment

Yes. Static inspection of PR head shows remote `sessions_spawn` sends an `agent` RPC without required `idempotencyKey`, and explicit `gatewayUrl` inputs bypass the current guarded `resolveGatewayOptions` path.

## Solution Assessment

No. The feature direction addresses a real multi-Gateway gap, but the current patch is not the best solution until the federation trust model and existing auth/URL guardrails are designed into the implementation.

  - body: The helper returns arbitrary explicit `gatewayUrl` values and the new session paths pass them directly to `callGateway`.   - confidence: 0.9

  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`. `AgentParamsSchema` requires a non-empty `idempotencyKey`, so cross-Gateway spawn requests can be rejected as invalid params before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, but both remote branches pass the peer URL to `callGateway` as an explicit override. `callGateway` requires explicit credentials for URL overrides, so tokenless peers fail before reaching the remote Gateway unless a separate trusted-peer auth path is added.
  - confidence: 0.86
- **[P2] Reject sessionKey and label together on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote `sessions_send` branch only rejects the neither-provided case, then silently prefers `sessionKey` when both `sessionKey` and `label` are supplied. The local path rejects this combination, so cross-Gateway behavior can route to an unintended session instead of surfacing an input error.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:127-133`
  - body: The remote `sessions_send` branch hard-codes the global nested lane, while current main scopes nested work with `resolveNestedAgentLaneForSession(resolvedKey)`. Peer sends can reintroduce cross-session nested-lane contention.
