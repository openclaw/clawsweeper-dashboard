## Summary

Keep open. The PR targets a real feature gap that is still present on current main, but the patch is not merge-ready: it introduces a new cross-Gateway routing/config surface with blocking protocol and URL-policy defects, lacks real two-Gateway proof, and still needs maintainer product/security judgment.

## What This Changes

Adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`, with a peer-resolution helper and tests.

## Best Possible Solution

Define and review a cross-Gateway peer trust contract, preserve the existing URL/auth hardening or replace it with an explicitly approved peer policy, satisfy the Gateway `agent` protocol, and prove send plus spawn across two real Gateways before landing.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote `sessions_spawn` omits required `idempotencyKey`, ad-hoc `gatewayUrl` bypasses the current allowlist, tokenless peers conflict with explicit-auth rules, and remote send target validation diverges from local behavior. I did not establish a real two-Gateway runtime repro because this is a read-only review and the PR provides no real behavior proof.

## Solution Assessment

No. The `gateway.peers` direction is plausible, but this patch does not yet preserve the existing URL/auth contract or satisfy the Gateway `agent` protocol, so the safer path is a reviewed peer policy plus focused implementation and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.9
- **[P2] Require credentials for configured peer URL overrides:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Reject ambiguous remote send targets:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87
