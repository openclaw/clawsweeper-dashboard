## Summary

Keep open. The feature targets a real multi-Gateway gap and closes an open paired tracker, but the PR still has blocking runtime/security issues, lacks real multi-Gateway behavior proof, and needs maintainer product/security direction before this trust boundary is added.

## What This Changes

The PR adds a `gateway.peers` config map plus `gateway`/`gatewayUrl`/`gatewayToken` paths that route `sessions_send` and `sessions_spawn` through remote Gateway RPC calls, with schema/help/type/test additions.

## Best Possible Solution

Define an additive trusted Gateway federation/session-routing contract first, then rebase the implementation to preserve URL/auth guardrails, idempotency, per-session lanes, docs, tests, and real behavior proof.

## Reproduction Assessment

Yes for the review findings. PR-head source shows remote `sessions_spawn` sends an `agent` RPC without required `idempotencyKey`, and explicit `gatewayUrl` inputs bypass the current guarded agent-tool URL path.

## Solution Assessment

No. The feature direction addresses a real multi-Gateway gap, but this patch is not the best way to solve it until the federation trust model and existing auth/URL guardrails are designed into the implementation.

Overall confidence: 0.91

Full review comments:

- **[P1] Keep explicit gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper returns arbitrary explicit `gatewayUrl` values, and the new session paths pass them directly to `callGateway`, bypassing the existing agent-tool allowlist in `resolveGatewayOptions`. Keep raw URL overrides on the guarded path and reserve any bypass for trusted configured peers with a documented policy.
  - confidence: 0.9
- **[P1] Include idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`. `AgentParamsSchema` requires a non-empty `idempotencyKey`, so cross-Gateway spawn requests can be rejected as invalid params before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, but both remote branches pass the peer URL as a `callGateway` URL override. `callGateway` requires explicit credentials for URL overrides, so tokenless peers fail before reaching the remote Gateway unless a separate trusted-peer auth path is added.
  - confidence: 0.86
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only rejects the neither-provided case, then silently prefers `sessionKey` when both `sessionKey` and `label` are supplied. The local path rejects this combination, so cross-Gateway behavior can route to an unintended session instead of surfacing an input error.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:133`
  - body: The remote `sessions_send` branch hard-codes the global nested lane, while current main scopes nested work with `resolveNestedAgentLaneForSession(resolvedKey)`. Peer sends can reintroduce cross-session nested-lane contention.
  - confidence: 0.78
