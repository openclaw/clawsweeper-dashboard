## Summary

Keep open. The feature is still absent from current main and the linked tracker is still open, but this PR is not mergeable as-is because it introduces cross-Gateway trust-boundary behavior with concrete runtime and security defects.

## What This Changes

The PR adds `gateway.peers` plus `gateway`, `gatewayUrl`, and `gatewayToken` routing for `sessions_send` and `sessions_spawn`, with config schema/help changes and peer-resolution tests.

## Best Possible Solution

Define an additive trusted Gateway federation/session-routing contract first, then rebase the implementation to preserve URL/auth guardrails, idempotency, lane behavior, docs, and regression coverage.

## Reproduction Assessment

Yes, for the blocking PR defects by source inspection: the remote spawn request omits required `idempotencyKey`, explicit `gatewayUrl` bypasses the guarded option path, and tokenless peers conflict with `callGateway` override auth requirements. I did not run a live multi-Gateway setup.

## Solution Assessment

No. The feature direction addresses a real distributed-Gateway gap, but this patch is not the best maintainable solution until the federation trust model and existing auth/URL guardrails are designed into the implementation.

  - confidence: 0.86
