## Summary

Keep open. This is a plausible implementation PR for the open cross-Gateway feature request, but the current patch still has blocking runtime and security issues, and the external contributor has not provided real after-fix multi-Gateway behavior proof.

## What This Changes

The PR adds a `gateway.peers` config map and `gateway`/`gatewayUrl`/`gatewayToken` parameters to `sessions_send` and `sessions_spawn` so session tools can route calls to another Gateway.

## Best Possible Solution

Land a narrower cross-Gateway implementation only after maintainers accept a fail-closed peer trust model, keep raw ad-hoc URLs behind existing allowlists, preserve Gateway protocol validation, and add real multi-Gateway proof.

## Reproduction Assessment

Yes for the source-level blockers: the PR's remote spawn request omits the required `idempotencyKey`, and tokenless peer entries reach `callGateway`'s explicit-URL credential guard. I did not run live multi-Gateway verification in this read-only review.

## Solution Assessment

No. The named-peer direction may be viable, but bypassing the existing URL allowlist and adding direct cross-Gateway session routing without an accepted trust model is not the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw gatewayUrl behind the tool allowlist:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The explicit `gatewayUrl` branch returns any ws(s) string, and the sessions fast paths pass it directly to `callGateway`, bypassing `resolveGatewayOptions` and the existing `validateGatewayUrlOverrideForAgentTools` SSRF guard. That gives any agent with these tools a new arbitrary Gateway WebSocket target path; restrict raw URLs to the existing allowlist or support only configured peer names with a reviewed trust contract.
  - confidence: 0.9
- **[P1] Include idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires that field. Cross-Gateway spawn will be rejected as invalid params before the remote agent run starts.
  - confidence: 0.94
- **[P2] Reject tokenless peers before returning a URL override:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: `gateway.peers.<name>.token` is optional and the tests cover tokenless peers, but the helper can return an explicit URL with `gatewayToken: undefined`. `callGateway` rejects URL overrides without explicit credentials, so those peers fail locally before connecting; require credentials or make no-auth peer mode explicit and covered end to end.
  - confidence: 0.88
- **[P2] Preserve label and sessionKey exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:74`
  - body: The remote `sessions_send` branch only rejects the case where neither target is provided, so passing both `sessionKey` and `label` silently ignores the label. The local path rejects this combination to avoid ambiguous targeting, and the remote path should keep the same validation.
  - confidence: 0.86
