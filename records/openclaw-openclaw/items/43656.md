## Summary

Keep open. Current main still has local-only session tools, while this PR is a real feature proposal but remains incorrect, conflicting, security-sensitive, and missing real two-Gateway behavior proof.

## What This Changes

Adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` cross-Gateway paths for `sessions_send` and `sessions_spawn`, with a peer-resolution helper and config/tests updates.

## Best Possible Solution

Define an approved cross-Gateway peer trust contract, keep URL/auth policy fail-closed, satisfy the Gateway `agent` protocol, and land only with real send/spawn proof across two Gateways.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote spawn omits required `idempotencyKey`, ad-hoc `gatewayUrl` bypasses the existing agent-tool allowlist, tokenless peers conflict with explicit-URL auth, and remote send does not preserve local target exclusivity. I did not run a two-Gateway repro because this is a read-only review and the PR has no real behavior proof.

## Solution Assessment

No. `gateway.peers` may be a plausible direction, but this patch does not yet preserve existing URL/auth policy or satisfy the Gateway protocol, so it needs a maintainer-approved trust model plus focused fixes and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.9
- **[P2] Reject tokenless peer overrides before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Preserve session target exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a cross-Gateway routing surface and currently lets ad-hoc `gatewayUrl` bypass the existing agent-tool URL allowlist.

Concerns:

- **[medium] Ad-hoc gatewayUrl bypasses existing allowlist:** `src/agents/tools/gateway-peer.ts:66`
  - body: Existing agent Gateway tools route URL overrides through `resolveGatewayOptions`, which constrains targets to loopback or configured remote URLs; this PR's explicit `gatewayUrl` branch returns arbitrary ws(s) URLs and the session tools pass them directly to `callGateway`. That outbound control surface needs a reviewed policy before merge.
