## Summary

Keep open. Current main still has local-only session tools, and the PR targets a real multi-Gateway gap, but the proposed routing still conflicts with existing Gateway URL/auth guardrails and the `agent` protocol contract, so it needs maintainer product/security review before merge.

## What This Changes

The PR adds a `gateway.peers` config map and `gateway`/`gatewayUrl`/`gatewayToken` routing for `sessions_send` and `sessions_spawn`, plus config labels/help and peer-resolution tests.

## Best Possible Solution

Define an additive Gateway federation or trusted-peer contract first, then implement session-tool routing with preserved URL/auth guardrails, valid `agent` payloads, credential/TLS requirements, docs, and end-to-end remote send/spawn tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: in the PR branch, `sessions_spawn` with `gateway` or `gatewayUrl` reaches a remote `agent` call without the required `idempotencyKey`, and explicit `gatewayUrl` inputs bypass the current allowlist path.

## Solution Assessment

No. The direction addresses a real multi-Gateway gap, but this patch is not the narrowest maintainable fix until raw URL guardrails, tokenless peer behavior, idempotency, lane semantics, and federation trust are resolved.
