The PR adds a `gateway.peers` config surface and `gateway`/`gatewayUrl`/`gatewayToken` parameters so `sessions_send` and `sessions_spawn` can forward work to another Gateway.

## Best Possible Solution

Define the cross-Gateway trust, URL policy, credential, routing, transcript ownership, and visibility contract first, then rework this PR or a replacement branch to preserve those boundaries with live two-Gateway proof.

Yes for the blocking PR defects by source inspection: `AgentParamsSchema` requires `idempotencyKey`, current Gateway URL policy is allowlisted, and the PR bypasses that path while omitting the required spawn field. I did not run a live two-Gateway reproduction in this read-only review.

## Solution Assessment

No. The `gateway.peers` direction may be useful, but this implementation should not merge until the cross-Gateway trust model is approved and the concrete protocol, auth, targeting, and proof blockers are fixed.
