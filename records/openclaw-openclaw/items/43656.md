## Summary

Keep this PR open for normal review: the feature is still relevant to the open cross-Gateway sessions request, but the current patch has source-provable protocol and security blockers plus no real two-Gateway behavior proof.

## What This Changes

Adds a `gateway.peers` config map, a peer-resolution helper, new session-tool gateway parameters, tests, and remote branches for `sessions_send` and `sessions_spawn`.

## Best Possible Solution

Rework the feature around a documented peer auth/addressing contract, preserve local session-tool validation semantics, satisfy the Gateway `agent` protocol, and prove send plus spawn across two real Gateways.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote `sessions_spawn` omits required `idempotencyKey`, tokenless peers conflict with current `callGateway` explicit-auth rules, and remote `sessions_send` diverges from the local target contract. I did not run the PR branch because this review is read-only.

## Solution Assessment

No. The `gateway.peers` direction is plausible, but this patch bypasses the existing agent-tool URL policy and does not satisfy the current Gateway agent protocol; a reviewed peer trust contract with consistent validation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it and the handler validates params before dispatch. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep gatewayUrl behind the agent-tool URL policy:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.88
- **[P2] Require or explicitly support tokens for peer overrides:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Reject ambiguous remote send targets:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote branch only checks for the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that ambiguous combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87

## Security Review

Status: needs_attention
