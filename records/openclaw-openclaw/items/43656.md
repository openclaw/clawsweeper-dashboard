## Summary

Keep open: current main still has only local session tools and no `gateway.peers`, while this PR remains a live but incorrect feature implementation. The diff adds a security-sensitive cross-Gateway target surface, has source-proven protocol/URL/auth issues, and lacks contributor-provided two-Gateway proof.

## What This Changes

Adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`, with a peer-resolution helper and tests.

## Best Possible Solution

Define an approved cross-Gateway peer trust contract, keep URL/auth hardening explicit, satisfy the Gateway `agent` protocol, and land it with live two-Gateway send/spawn proof.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote spawn omits `idempotencyKey`, the ad-hoc `gatewayUrl` path bypasses the current allowlist, tokenless peer config conflicts with `callGateway` auth requirements, and remote send accepts both `sessionKey` and `label`. I did not run a two-Gateway repro because this is a read-only review and the PR provides no real behavior proof.

## Solution Assessment

No. `gateway.peers` is a plausible direction, but this patch does not yet preserve the existing URL/auth contract or satisfy the Gateway `agent` protocol, so it needs a reviewed peer policy plus focused implementation and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.9
- **[P2] Reject tokenless peer overrides before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Preserve session target exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87
