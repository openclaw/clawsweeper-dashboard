## Summary

Keep open. Current main still has local-only session tools and no `gateway.peers` config, while the PR head still has blocking protocol/auth-policy defects and lacks required real two-Gateway behavior proof.

## What This Changes

The PR adds a `gateway.peers` config map and `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`.

## Best Possible Solution

Define an approved cross-Gateway trust, URL, auth, and session-routing contract, then land a rebased implementation with regression coverage and real send/spawn proof across two Gateways.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote spawn omits required `idempotencyKey`, ad-hoc `gatewayUrl` bypasses the existing allowlist, tokenless peers conflict with `callGateway`, and remote send does not preserve target exclusivity. I did not run a live two-Gateway repro because this is a read-only review and the PR has no real behavior proof.

## Solution Assessment

No. `gateway.peers` may be a plausible direction, but this patch does not yet preserve existing URL/auth policy or satisfy the Gateway protocol, so it needs maintainer-approved trust semantics plus focused fixes and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.9
- **[P2] Reject tokenless peer overrides before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type and tests making `token` optional.
  - confidence: 0.91
- **[P2] Preserve session target exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:70-74`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87
