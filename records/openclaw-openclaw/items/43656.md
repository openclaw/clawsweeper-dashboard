## Summary

Keep open: current main still lacks the requested cross-Gateway session feature, but this PR is not mergeable as-is because source inspection shows protocol, auth-policy, and target-validation defects plus missing real behavior proof.

## What This Changes

The PR adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`, with peer resolution tests and config schema/help updates.

## Best Possible Solution

Define the cross-Gateway peer trust, auth, URL, routing, transcript ownership, and visibility contract first, then implement the session tools against the Gateway protocol with focused tests and live two-Gateway proof.

## Reproduction Assessment

Yes for the blocking PR defects from source inspection: remote spawn omits required `idempotencyKey`, explicit `gatewayUrl` bypasses the current allowlist, tokenless peers conflict with `callGateway` auth requirements, and remote send accepts both `sessionKey` and `label`. I did not run a two-Gateway repro, and the PR has no real behavior proof.

## Solution Assessment

No. `gateway.peers` may be a plausible direction, but this patch does not yet preserve the existing URL/auth/session contracts or satisfy the Gateway `agent` protocol, so it needs a reviewed peer policy plus focused implementation and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the current loopback/configured-remote allowlist.
  - confidence: 0.9
- **[P2] Reject tokenless peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` requires explicit credentials for URL overrides. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Preserve session target exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87
