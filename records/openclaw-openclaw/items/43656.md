Full review comments:

- **[P1] Include idempotencyKey in remote spawns:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires that field and the gateway agent handler validates requests against it. Any cross-gateway spawn will be rejected as invalid params before the remote run starts.
  - confidence: 0.97
- **[P1] Keep remote session URLs behind the gateway allowlist:** `src/agents/tools/sessions-send-tool.ts:87-90`
  - body: This branch exposes `gatewayUrl` on `sessions_send` and passes the resolved URL directly to `callGateway`, bypassing the existing agent-tool `resolveGatewayOptions` allowlist for loopback or configured `gateway.remote.url`. That turns model-provided session-tool input into operator-scoped WebSocket traffic to arbitrary hosts instead of an approved gateway target.
  - confidence: 0.9
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-73`
  - body: The remote branch accepts both `sessionKey` and `label` and silently prefers `sessionKey`, while the local path still rejects that combination. Cross-gateway sends can therefore ignore the caller's label and route to an unintended remote session instead of returning the established input error.
  - confidence: 0.86
- **[P2] Require credentials for configured peers:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: `resolveGatewayPeerOptions` returns a URL override even when a named peer has no token, but `callGateway` rejects URL overrides without explicit credentials. The PR examples and tests treat tokenless peers as valid, so those calls will fail before reaching the remote gateway.
  - confidence: 0.84
