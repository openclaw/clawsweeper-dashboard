## Summary

Keep open. The requested cross-Gateway session feature is not on current main and this PR is paired with the still-open tracker at https://github.com/openclaw/openclaw/issues/43605, but the branch has blocking correctness/security issues and no after-fix real two-Gateway proof.

## What This Changes

The PR adds a `gateway.peers` config map and `gateway`/`gatewayUrl`/`gatewayToken` handling in `sessions_send` and `sessions_spawn` to forward session traffic to remote Gateways.

## Best Possible Solution

Rebase and redesign the cross-Gateway session path around an approved trust/URL/auth model, then land it with protocol-compliant requests, consistent validation, focused tests, and redacted real two-Gateway proof.

## Reproduction Assessment

No complete real-runtime reproduction was established for the new feature. The blocking PR failures are source-reproducible: remote `sessions_spawn` omits the required `idempotencyKey`, tokenless peers conflict with `callGateway` auth requirements, and ad-hoc URLs bypass the existing agent-tool URL policy.

## Solution Assessment

No. The named-peer direction is plausible, but this implementation bypasses existing URL policy, leaves remote spawn invalid, and needs a maintainer-approved trust model plus real two-Gateway proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add idempotencyKey to remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path forwards to the gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Any cross-Gateway spawn that reaches the remote gateway will be rejected as invalid params before the run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the existing allowlist:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: This helper returns any tool-supplied `gatewayUrl`, and the new call sites pass that directly to `callGateway`, bypassing `resolveGatewayOptions` and its agent-tool URL allowlist. Configured peers may need a new reviewed path, but ad-hoc URL params should not skip the existing security policy.
  - confidence: 0.88
- **[P2] Reject or require credentials for tokenless peers:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: The helper can return a peer URL with `gatewayToken: undefined`, and the tests advertise tokenless peers as valid. `callGateway` rejects URL overrides without explicit credentials, so those peers fail before any remote request is attempted unless the schema/docs require a token or the auth model is changed deliberately.
  - confidence: 0.9
- **[P2] Preserve sessionKey/label exclusivity for remote sends:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote branch only rejects the case where neither target is provided. If callers provide both `sessionKey` and `label`, it silently prefers `sessionKey`, unlike the local path which returns an input error; that can route a remote message to an unintended session.
  - confidence: 0.86
