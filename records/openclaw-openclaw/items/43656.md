## Summary

Keep open. The PR targets a real open feature gap, but the current branch still has blocking runtime issues, lacks real multi-Gateway behavior proof, and needs maintainer product/security review for the new cross-Gateway trust boundary.

## What This Changes

The PR adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` parameters to `sessions_send` and `sessions_spawn`, forwarding selected session operations to a remote Gateway.

## Best Possible Solution

Keep the linked feature issue open and either revise this PR with a maintainer-approved peer trust model, fixed runtime contract, tests, docs, and real multi-Gateway proof, or redirect the narrower session-tool work into the broader federation design.

## Reproduction Assessment

Not applicable as a bug; this PR implements a new feature. Source review gives high-confidence reproduction of the patch blockers: remote spawn omits the required `agent.idempotencyKey`, and tokenless peer entries flow into `callGateway` URL override auth.

## Solution Assessment

No. `gateway.peers` is a plausible direction, but this branch is not the best mergeable solution until it fixes the protocol/auth/input-validation bugs, resolves conflicts, and provides maintainer-approved trust semantics plus real multi-Gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass an idempotency key to remote spawns:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote branch calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires that field. A cross-gateway `sessions_spawn` request will be rejected as invalid params before the remote run starts, so generate and forward a stable idempotency key like the send path does.
  - confidence: 0.96
- **[P2] Require auth before returning peer URL overrides:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: `gateway.peers` can omit `token`, and this helper returns `{ gatewayUrl, gatewayToken: undefined }` for that case. Both session tools then pass the explicit URL into `callGateway`, whose override guard requires explicit credentials, so the documented tokenless peer configuration fails before contacting the remote Gateway.
  - confidence: 0.92
- **[P2] Reject mixed sessionKey and label on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote path only checks the neither-provided case, then silently prefers `sessionKey` when both `sessionKey` and `label` are supplied. The local path rejects that ambiguous input; preserving the same exclusivity avoids sending a cross-gateway message to an unintended session.
  - confidence: 0.89
- **[P3] Validate peer URLs as real Gateway origins:** `src/config/zod-schema.ts:813-818`
  - body: The schema only checks for a `ws://` or `wss://` prefix, so malformed values such as `wss://` still pass config validation and fail later in the call path. Parse with `URL` and enforce a valid ws/wss origin so users get the config error at load time.
  - confidence: 0.78
