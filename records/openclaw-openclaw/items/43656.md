## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep remote session URLs behind the agent-tool allowlist:** `src/agents/tools/sessions-send-tool.ts:87-90`
  - body: The remote path passes `peerOpts.gatewayUrl` directly to `callGateway`, but `resolveGatewayPeerOptions` also returns raw caller-supplied `gatewayUrl`. That skips the existing `resolveGatewayOptions` allowlist for loopback or configured `gateway.remote.url`, allowing session tools to open credentialed WebSocket traffic to arbitrary hosts instead of an approved gateway target.
  - confidence: 0.94
- **[P1] Include idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch forwards an `agent` request without `idempotencyKey`, but `AgentParamsSchema` requires that field. Cross-gateway spawn will be rejected as invalid params before the remote run starts.
  - confidence: 0.97
- **[P2] Do not advertise tokenless peers until calls support them:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: The helper returns configured peers with `gatewayToken: undefined`, and the tests cover that as expected behavior, but `callGateway` rejects explicit URL overrides without explicit credentials. A `gateway.peers.<name>` entry without a token will fail locally before reaching a peer gateway, so either make the token required or design an explicit auth-none peer mode.
  - confidence: 0.9
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch only checks that at least one target is present; when both `sessionKey` and `label` are provided it silently prefers `sessionKey`. The local path rejects that combination, so cross-gateway sends can behave differently and target an unintended remote session.
  - confidence: 0.86
