## Summary

Keep open. Current main does not implement cross-Gateway session targeting, and the proposed PR is not merge-ready because the patch still has protocol, auth, and trust-boundary blockers in the remote send/spawn paths.

## What This Changes

The PR adds a `gateway.peers` config surface plus `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`, with a new peer-resolution helper and helper tests.

## Best Possible Solution

Define an additive Gateway federation or trusted-peer contract first, then implement session-tool routing with valid Gateway protocol payloads, preserved local safety semantics, credential/TLS requirements, docs, and end-to-end remote send/spawn coverage.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: `sessions_spawn` with `gateway` or `gatewayUrl` reaches the remote `agent` method without `idempotencyKey`, and explicit `gatewayUrl` inputs visibly bypass the current agent-tool allowlist.

## Solution Assessment

No. The direction addresses a real multi-Gateway gap, but this patch is not the narrowest maintainable solution until the raw URL guardrail, tokenless peer semantics, protocol payload, and federation trust model are resolved.
