## Summary

Keep open: current main still has local-only session tools, and this PR is useful source work for the linked feature request, but it is not ready to merge because the patch has blocking correctness and security findings around remote Gateway routing, auth guardrails, idempotency, and unresolved federation policy.

## What This Changes

The PR adds a `gateway.peers` config map plus `gateway`/`gatewayUrl`/`gatewayToken` routing for `sessions_send` and `sessions_spawn`, with schema/help labels and peer-resolution tests.

## Best Possible Solution

Define an additive Gateway federation or trusted-peer contract first, then rebase and implement session routing with preserved URL/auth guardrails, required idempotency, credential/TLS policy, lane semantics, docs/changelog, and remote send/spawn tests.

## Reproduction Assessment

Yes. Static inspection of PR head `85c5b02419c2a2390465f74d2daa45a047d7e746` gives a high-confidence path: remote `sessions_spawn` sends an `agent` RPC without the required `idempotencyKey`, and explicit `gatewayUrl` inputs bypass the current guarded `resolveGatewayOptions` path.

## Solution Assessment

No. The direction addresses a real multi-Gateway gap, but this patch is not the narrowest maintainable fix until raw URL guardrails, peer credentials, idempotency, lane behavior, and federation trust semantics are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep explicit gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns arbitrary explicit `gatewayUrl` values and the new session paths pass them directly to `callGateway`. That bypasses the current agent-tool allowlist in `resolveGatewayOptions`; keep raw URL overrides on the guarded path and reserve any bypass for trusted config peers with a documented policy.
  - confidence: 0.9
- **[P1] Include idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`. `AgentParamsSchema` requires a non-empty `idempotencyKey`, so cross-Gateway spawn requests can be rejected as invalid params before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, but both remote branches pass the peer URL to `callGateway` as an explicit override. `callGateway` requires explicit credentials for URL overrides, so tokenless peers fail before reaching the remote Gateway unless a separate trusted-peer auth path is added.
  - confidence: 0.86
- **[P2] Reject sessionKey and label together on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` branch reads both values but only rejects the neither-provided case, then silently prefers `sessionKey`. The local path rejects this input, and the remote branch can route to an unintended session instead of surfacing an input error.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:127-133`
  - body: The remote `sessions_send` branch hard-codes the global nested lane, while current main scopes nested work with `resolveNestedAgentLaneForSession(resolvedKey)`. Peer sends can reintroduce cross-session nested-lane contention.
  - confidence: 0.78
