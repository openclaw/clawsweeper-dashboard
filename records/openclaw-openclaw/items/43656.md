## Summary

Keep open. The cross-gateway session feature is not on current main and is paired with the still-open feature request, but this PR is not mergeable as-is because the remote paths still violate the gateway protocol/auth contract and lack real two-gateway proof.

## What This Changes

The PR adds `gateway.peers` config plus `gateway`, `gatewayUrl`, and `gatewayToken` parameters so `sessions_send` and `sessions_spawn` can route through remote gateway RPC calls.

## Best Possible Solution

Rework this against current main around a documented peer auth/addressing contract, keep remote tool validation consistent with local sessions behavior, satisfy the gateway `agent` protocol, and prove the flow against two real gateways before closing https://github.com/openclaw/openclaw/issues/43605.

## Reproduction Assessment

Yes for the blocking PR bugs from source inspection: remote `sessions_spawn` omits the required `idempotencyKey`, and tokenless peers flow into `callGateway` as explicit URL overrides without explicit credentials. I did not run the PR branch because this review is read-only and the branch is merge-conflicting.

## Solution Assessment

No. The proposed direction addresses a real feature gap, but the current patch should not merge until it aligns with gateway auth policy, preserves existing validation semantics, and includes real cross-gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Any cross-gateway spawn will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep ad-hoc gatewayUrl behind the tool URL policy:** `src/agents/tools/gateway-peer.ts:65-68`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools then bypass `resolveGatewayOptions`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist. Named config peers may need a new policy, but ad-hoc URLs should not silently bypass the current agent-tool restriction.
  - confidence: 0.82
- **[P2] Reject tokenless peers before calling overridden URLs:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: `gateway.peers.<name>.token` is optional in the new config and tests, but `callGateway` rejects explicit URL overrides without explicit credentials. Peers that resolve with `gatewayToken: undefined` will fail before reaching even an unauthenticated remote gateway; either require/reject missing tokens clearly or use a deliberately approved unauthenticated peer path.
  - confidence: 0.9
- **[P2] Reject sessionKey with label on remote sends:** `src/agents/tools/sessions-send-tool.ts:74-80`
  - body: The remote branch only checks that at least one target is present, so `sessionKey` plus `label` silently uses `sessionKey`. The local branch rejects this ambiguous combination, and remote sends should preserve that contract to avoid routing to the wrong session.
  - confidence: 0.88

## Security Review
