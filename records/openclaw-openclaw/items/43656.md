## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Keep remote session URLs behind the allowlist:** `src/agents/tools/sessions-send-tool.ts:87-90`
  - body: This branch resolves `gateway`/`gatewayUrl` and then passes `peerUrl` directly to `callGateway`, bypassing `validateGatewayUrlOverrideForAgentTools`. Current tests reject non-allowlisted agent-tool Gateway URLs for SSRF hardening, so model-invoked `sessions_send`/`sessions_spawn` calls would gain arbitrary WebSocket reachability.
  - confidence: 0.97
- **[P1] Implement remote spawn instead of messaging a main session:** `src/agents/tools/sessions-spawn-tool.ts:158-179`
  - body: The remote path builds `agent:<id>:main` and calls the ordinary `agent` method, then reports `status: accepted`. That does not create an isolated child session, register lifecycle/cleanup/timeout state, apply spawn runtime/model/context/sandbox/attachment settings, or preserve completion delivery, so `sessions_spawn` would mean a different operation when `gateway` is set.
  - confidence: 0.98
- **[P1] Include an idempotency key in remote agent requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The Gateway `agent` protocol requires `idempotencyKey`, and the server uses it as the run id and dedupe key. The remote spawn request omits it, so the request is invalid before the remote run can start.
  - confidence: 0.99
- **[P2] Make peer credentials required or fail before connecting:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: `gateway.peers.<name>.token` is documented and typed as optional, but `callGateway` requires explicit credentials for any URL override. A tokenless peer such as the PR body's `studio` example will fail later with a generic URL-override auth error; either require the token in the peer contract or reject tokenless peers clearly in the helper.
  - confidence: 0.9
