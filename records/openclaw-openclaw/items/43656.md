## Summary

Keep open: current main still lacks cross-Gateway session routing, and this PR is the linked implementation candidate, but the current head is not mergeable because source review shows protocol, URL-policy, and auth-contract defects plus missing real two-Gateway proof.

## What This Changes

Adds a `gateway.peers` config map plus `gateway`/`gatewayUrl`/`gatewayToken` parameters to `sessions_send` and `sessions_spawn`, with a helper and tests for remote Gateway routing.

## Best Possible Solution

Define a reviewed peer auth/addressing contract, keep existing agent-tool URL hardening, satisfy the Gateway `agent` protocol, and prove send plus spawn across two real Gateways before landing.

## Reproduction Assessment

Yes, for the blocking PR defects from source inspection: remote `sessions_spawn` omits required `idempotencyKey`, `gatewayUrl` bypasses the current allowlist, and tokenless peers conflict with explicit-auth rules. I did not run the PR branch because this review is read-only and no real two-Gateway proof was provided.

## Solution Assessment

No. The `gateway.peers` direction is plausible, but this patch does not preserve the existing URL/auth contract or the Gateway `agent` protocol, so a reviewed peer trust contract with consistent validation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it and the handler validates params before dispatch. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep gatewayUrl behind the agent-tool URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`, so an agent tool call can target arbitrary ws(s) endpoints instead of the existing loopback/configured-remote allowlist.
  - confidence: 0.88
- **[P2] Reject tokenless peer overrides before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers may resolve to an explicit URL with `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Tokenless peers therefore fail before reaching the remote Gateway despite the config type making `token` optional.
  - confidence: 0.91
- **[P2] Reject ambiguous remote send targets:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination, and the remote path should keep the same contract to avoid routing to the wrong session.
  - confidence: 0.87
