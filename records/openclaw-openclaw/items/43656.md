Overall confidence: 0.92

Full review comments:

- **[P1] Keep session-tool gateway URLs behind the allowlist:** `src/agents/tools/sessions-send-tool.ts:98-100`
  - body: The new remote path resolves `gateway`/`gatewayUrl` and then passes `peerOpts.gatewayUrl` directly to `callGateway`, bypassing the current agent-tool `resolveGatewayOptions` allowlist that only permits loopback or the configured `gateway.remote.url`. Since `sessions_send` and `sessions_spawn` are model-callable and security-sensitive, this exposes authenticated cross-session traffic to arbitrary WSS targets unless maintainers explicitly replace that boundary with a reviewed peer allowlist.
  - confidence: 0.94
- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch sends an `agent` request with `message`, `sessionKey`/`label`, and delivery metadata, but it omits `idempotencyKey`. The gateway protocol requires `idempotencyKey` on `agent` params, so every cross-gateway spawn will be rejected as invalid before the remote run starts.
  - confidence: 0.98
- **[P2] Do not model remote spawn as a main-session send:** `src/agents/tools/sessions-spawn-tool.ts:158-187`
  - body: This branch implements `sessions_spawn({ gateway })` by constructing `agent:<agentId>:main` and sending a normal `agent` message, then returning that session key instead of a spawned child session. That loses the existing `sessions_spawn` contract for isolated child runs, child tracking, cleanup, and completion ownership, so callers asking to spawn remote work can silently get a main-session message instead.
  - confidence: 0.87
