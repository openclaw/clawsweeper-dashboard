## Summary

Keep open. The PR is a plausible implementation candidate for the open cross-gateway sessions request, but the current patch still has blocking runtime and security issues, lacks real multi-gateway behavior proof, and touches a trust-boundary product decision.

## What This Changes

The PR adds `gateway.peers` config plus `gateway`/`gatewayUrl`/`gatewayToken` parameters so `sessions_send` and `sessions_spawn` can route through a remote gateway.

## Best Possible Solution

Keep the linked tracker open and either revise this PR into a fail-closed peer contract with canonical URL validation, explicit auth semantics, preserved gateway protocol fields, docs, tests, and real multi-gateway proof, or redirect it into the broader federation design at https://github.com/openclaw/openclaw/issues/46724.

## Reproduction Assessment

Yes for the blocking patch defects from source inspection: remote `sessions_spawn` omits the required `idempotencyKey`, tokenless peer entries hit the existing explicit-auth guard, and explicit `gatewayUrl` bypasses the current allowlist. I did not run a live multi-gateway setup.

## Solution Assessment

No. The `gateway.peers` direction may be viable, but this patch is not the best mergeable shape until it preserves the existing URL/auth guardrails, satisfies the gateway `agent` protocol contract, and proves the real cross-gateway path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep explicit gatewayUrl overrides on the allowlist:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: For the explicit `gatewayUrl` path this helper returns whatever URL the tool call supplied, and both session tools pass it straight into `callGateway`. That bypasses the existing `validateGatewayUrlOverrideForAgentTools` loopback/`gateway.remote.url` allowlist for model-supplied session tool args, so a session tool call can connect to an arbitrary WebSocket when it includes any token string. Keep ad-hoc URL overrides on the existing guardrail, or limit the bypass to administrator-configured peers with equivalent validation.
  - confidence: 0.92
- **[P1] Include idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote spawn branch calls the gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires that field. A `sessions_spawn(..., gateway=...)` request will be rejected as invalid params before the remote run starts, so generate and forward an idempotency key like the send path does.
  - confidence: 0.95
- **[P2] Reject tokenless peers before calling an override URL:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can omit `token`, and the tests advertise that path, but `callGateway` requires explicit credentials for any override URL. A configured peer like the PR's `studio` example therefore fails locally with `gateway url override requires explicit credentials` instead of attempting the peer call; require a token/password-capable peer credential or return a clear config error before routing.
  - confidence: 0.9
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:74-80`
  - body: The remote `sessions_send` branch only rejects the case where neither target is provided, so a call with both `sessionKey` and `label` silently prefers `sessionKey`. The local tool rejects that combination to avoid ambiguous routing; keep the same validation on the remote path so the message cannot go to an unintended session.
  - confidence: 0.86
