## Summary

Keep open: current main does not implement cross-Gateway session routing, and the PR still has blocking source-level defects plus missing real two-Gateway proof and unresolved trust/URL policy decisions.

## What This Changes

The PR adds a `gateway.peers` config map and `gateway`/`gatewayUrl`/`gatewayToken` routing paths for `sessions_send` and `sessions_spawn`.

## Best Possible Solution

Define an approved cross-Gateway trust, URL, auth, and session-routing contract, then rebase and fix or replace this implementation with regression coverage and live two-Gateway proof.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: remote spawn omits required `idempotencyKey`, ad-hoc `gatewayUrl` bypasses the existing tool URL policy, tokenless peers conflict with `callGateway`, and remote send does not preserve target exclusivity. I did not run a live two-Gateway repro in this read-only review.

## Solution Assessment

No. The `gateway.peers` direction may be plausible, but this patch does not preserve URL/auth policy or satisfy the Gateway protocol, so it needs maintainer-approved trust semantics plus focused fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` path calls Gateway method `agent` without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-Gateway spawns will be rejected as invalid params before the remote run starts.
  - confidence: 0.95
- **[P1] Keep gatewayUrl behind the existing URL policy:** `src/agents/tools/gateway-peer.ts:65-67`
  - body: The helper returns any caller-supplied `gatewayUrl`, and the session tools pass it directly to `callGateway`. Existing agent Gateway tools validate overrides through `resolveGatewayOptions`, so this creates a competing arbitrary target path for model-callable tools.
  - confidence: 0.9
- **[P2] Reject or support tokenless peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can resolve to `gatewayToken: undefined`, but `callGateway` rejects explicit URL overrides without explicit credentials. Entries documented and tested without `token` will fail before contacting the remote Gateway.
  - confidence: 0.91
- **[P2] Preserve session target exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote `sessions_send` branch only checks the no-target case, so when `sessionKey` and `label` are both present it silently prefers `sessionKey`. The local path rejects that combination to avoid routing to the wrong session.
  - confidence: 0.87
