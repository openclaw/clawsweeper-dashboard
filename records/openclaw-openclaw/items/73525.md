## Summary

Keep open. Current main still preserves the reported split: plain `openclaw status` stays on the fast read-only channel table path, while `openclaw channels status` renders from the live gateway `channelAccounts` payload. #73662 is an open, unmerged implementation candidate with closing syntax for this issue, so this issue should stay open until that PR lands or is superseded.

## What This Changes

The issue asks for the Channels section in plain `openclaw status` to show configured/running channels consistently with `openclaw channels status`.

## Best Possible Solution

Review and land #73662, or an equivalent narrow fix, so the visible Channels table can fill missing rows from the existing gateway account snapshot while preserving the fast default status path, regression coverage, and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, another user reproduced the same behavior with Discord and Telegram, and current main still has a source/test path where default status can render no rows while the live `channelAccounts` payload would be shown by `openclaw channels status`.

## Solution Assessment

Unclear as a merged solution because no fix is on main yet. The direction in #73662 looks narrow and maintainable because it reuses the existing gateway snapshot to fill missing visible rows instead of making default status fully deep or changing channel discovery wholesale.
