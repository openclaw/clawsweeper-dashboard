## Summary

Keep open. Current main still has the reported split: the fast `openclaw status` path disables live channel status and setup fallback, while the visible Channels table is built only from read-only discovery; #73662 is an open, unmerged implementation candidate with closing syntax, so this issue should remain open until that PR lands or is superseded.

## What This Changes

The issue asks for `openclaw status` to populate its Channels table from configured or running channels instead of showing only headers while `openclaw channels status` shows live accounts.

## Best Possible Solution

Review and land #73662, or an equivalent narrow fix, so the visible Channels table can fill missing rows from the existing gateway account snapshot while preserving the fast default status path and adding regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue gives concrete CLI steps, another user reproduced the split with Discord and Telegram, and current main still contains the source/test path where fast status can render no rows while `openclaw channels status` renders `channelAccounts`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a merged solution because no fix is on main yet. The #73662 direction appears narrow and maintainable because it reuses the existing gateway snapshot to fill missing visible rows instead of making default status fully deep or changing channel discovery wholesale.
