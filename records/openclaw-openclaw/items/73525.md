## Summary

Keep open. Current main still preserves the reported split between the fast `openclaw status` channel table and the live `openclaw channels status` output, and #73662 is an open, unmerged implementation candidate with closing syntax for this report.

## What This Changes

The issue requests that plain `openclaw status` populate its Channels table for configured/running channels instead of rendering only headers when `openclaw channels status` shows live accounts.

## Best Possible Solution

Review and land #73662, or an equivalent narrow fix, so the visible Channels table can augment missing read-only rows from the existing gateway account snapshot while preserving the fast default status path and adding regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, another user confirmed the same behavior with Discord and Telegram, and current main still has a source/test path where default status can render no rows while live `channelAccounts` would be shown by `openclaw channels status`.

## Solution Assessment

Unclear as a merged solution because no fix is on main yet. The direction in #73662 looks like the narrowest maintainable fix: reuse the existing gateway snapshot to fill missing visible rows instead of making the default status path fully deep or changing channel discovery wholesale.
