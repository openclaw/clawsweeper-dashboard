## Summary

Keep open. Current main still preserves the reported split: default text `openclaw status` skips live `channels.status` and builds the visible Channels table only from read-only discovery, while the linked fix PR #73662 is open and unmerged.

## What This Changes

The issue requests that `openclaw status` populate its Channels table with the configured and running channels that are visible through `openclaw channels status`.

## Best Possible Solution

Resolve this by reviewing and landing #73662 or an equivalent narrow fix that augments the visible Channels table from the existing gateway snapshot when read-only discovery returns no rows, while preserving the fast default status path when the gateway is unreachable.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, another user confirmed the same behavior with multiple channels, and current main still has a source/test path where default status can render no rows while `channels.status` contains live accounts.

## Solution Assessment

Unclear as a merged solution because no fix is on main yet. The narrow direction in #73662 appears maintainable: reuse the existing gateway snapshot only to fill missing visible rows instead of changing channel runtime discovery wholesale.
