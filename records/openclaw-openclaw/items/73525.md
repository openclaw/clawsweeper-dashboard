## Summary

Keep open. Current main still preserves the reported split: fast `openclaw status` builds the visible Channels table from read-only channel discovery, while `openclaw channels status` formats the gateway `channelAccounts` payload. PR #73662 is an open implementation candidate with closing syntax, so the issue should stay open until that PR is reviewed, landed, or superseded.

## What This Changes

The issue requests that `openclaw status` populate its Channels table with the configured and running channel accounts already visible through `openclaw channels status`.

## Best Possible Solution

Resolve this by reviewing and landing PR #73662 or an equivalent narrow fix that fills missing visible Channels rows from the existing gateway `channels.status` snapshot while preserving the fast status path when the gateway is unreachable.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, another user confirmed the same split with multiple channels, and current main has a source/test path where fast status can render no rows while `channels.status` contains live channel accounts.

## Solution Assessment

Yes for the requested direction, but the exact implementation is still pending review. The maintainable fix should reuse the existing gateway status snapshot to fill the display gap rather than changing channel runtime discovery wholesale.
