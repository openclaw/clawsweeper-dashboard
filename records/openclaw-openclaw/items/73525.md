## Summary

Keep open. Current main still routes fast `openclaw status` through the read-only Channels table without consuming the gateway account snapshot that `openclaw channels status` renders, and #73662 is an open unmerged implementation candidate with closing syntax for this issue.

## What This Changes

Request that `openclaw status` populate its Channels table consistently with `openclaw channels status` for configured and running channels.

## Best Possible Solution

Land #73662, or an equivalent narrow mainline fix, so the visible Channels table can use the existing gateway channel account snapshot when read-only discovery produces no rows.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue gives concrete CLI steps, another user confirmed the same split with Discord and Telegram, and current main still has the source path where fast status skips the gateway account data that the dedicated channels status command renders.

## Solution Assessment

Is this the best way to solve the issue? Unclear until a fix is merged. The #73662 direction appears narrow because it reuses the existing gateway snapshot for missing visible rows without turning fast status into full deep status.
