## Summary

Keep open. Current main still gates default text status off live channel status and setup fallback, then renders the visible Channels table only from read-only rows; #73662 remains open and unmerged as the implementation candidate.

## What This Changes

The issue asks for default `openclaw status` to populate its Channels table consistently with `openclaw channels status` when channels are configured and running.

## Best Possible Solution

Land #73662 or an equivalent narrow fix that lets the visible Channels table use the existing gateway account snapshot when read-only discovery produces no rows, while preserving the fast default status path.

## Reproduction Assessment

Yes. The issue provides concrete CLI steps, another user confirmed the same split across Discord and Telegram, and current source shows plain `openclaw status` can skip the gateway account data that `openclaw channels status` renders.

## Solution Assessment

Unclear because no solution is merged yet. The #73662 approach looks maintainable because it reuses the existing `channels.status` payload instead of turning default status into a fully deep probe, but the augment-versus-replace behavior still needs review.
