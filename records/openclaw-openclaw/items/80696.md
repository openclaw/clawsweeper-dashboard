## Summary

Keep open: current main and the latest release still ship and audit for `RestartSec=5`, while the supervised lock path still depends on `/healthz` becoming healthy before exit 78 can stop systemd respawns.

## What This Changes

The issue requests a gateway/systemd lifecycle fix so the default unit’s restart delay no longer races the documented exit-78 duplicate-start protection during slow gateway warmup.

## Best Possible Solution

Align the shipped systemd unit, service audit, Linux/gateway docs, and regression tests around a restart delay that gives normal gateway startup enough time to expose `/healthz`, while preserving the existing exit-78 contract.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection gives a clear reproduction model: a systemd duplicate start before the incumbent exposes `/healthz` keeps the exit-78 path from firing while the shipped unit retries after 5 seconds.

## Solution Assessment

Yes for a focused repair that updates the systemd restart timing contract and its audit/docs/tests. A `Type=notify` redesign or lock-ownership rewrite may be useful later, but it is broader than needed to address the reported default-unit race.
