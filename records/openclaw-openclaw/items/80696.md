## Summary

Keep open: current main and the latest release still ship `RestartSec=5`, and doctor still treats 5s as the preferred systemd value even though the documented exit-78 guard only fires after a duplicate starter sees a healthy `/healthz` responder. The related closed EADDRINUSE loop fix does not cover this warmup-timing race, and the broader crash-recovery ladder issue is not a narrow owner for this specific default-unit bug.

## What This Changes

The issue requests changing the default Linux systemd gateway restart behavior so the exit-78 duplicate-start protection is not defeated by restarting before the previous gateway becomes healthy.

## Best Possible Solution

Land a narrow Linux gateway service fix that makes the generated unit, service audit, docs, and tests use a restart delay longer than realistic startup-to-health time, while leaving Type=notify/watchdog or graduated crash recovery for separate design work.

## Reproduction Assessment

No, not as a live WSL2/systemd cascade in this read-only review. The source path is reproducible: current main and v2026.5.7 use `RestartSec=5`, the exit-78 path only fires after a healthy `/healthz` probe, and service-mode stale cleanup can kill an existing port holder before that path succeeds.

## Solution Assessment

Yes for a narrow first fix: raising the generated and audited systemd restart delay, with docs and regression tests, is the smallest maintainable repair for the reported workaround. Type=notify, watchdogs, or a crash-recovery ladder are broader follow-ups rather than the first patch for this issue.
