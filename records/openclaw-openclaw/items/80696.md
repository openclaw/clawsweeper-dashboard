## Summary

Keep open: current main and the latest shipped release still use `RestartSec=5` while the documented exit-78 lock recovery only fires after `/healthz` responds, so the reported systemd timing race remains source-visible and has no merged canonical fix.

## What This Changes

The issue asks OpenClaw to change the shipped systemd gateway restart timing or lock/health semantics so slow gateway warmup does not defeat the exit-78 restart-loop guard.

## Best Possible Solution

Align the generated systemd unit, service audit, docs, and tests around a restart delay long enough for normal `/healthz` startup, while preserving the existing exit-78 contract.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection and the reporter's journal timing show a credible current-main path: the unit restarts after 5s, while exit 78 only fires once an existing gateway answers `/healthz`.

## Solution Assessment

Yes for a focused repair that updates the systemd restart timing contract and its audit/docs/tests. A `Type=notify` redesign is broader and should only be pursued if maintainers prefer that direction.
