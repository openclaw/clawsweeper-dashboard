## Summary

Current main still renders and recommends a 5s systemd restart delay while the documented exit-78 protection only works after an existing gateway can answer `/healthz`; the issue is source-supported and not duplicated by the earlier EADDRINUSE fix, so it should stay open for a targeted gateway/systemd repair.

## What This Changes

The issue requests changing the default systemd gateway restart behavior so `RestartPreventExitStatus=78` can stop restart cascades when gateway startup takes longer than `RestartSec=5`.

## Best Possible Solution

Update the canonical systemd unit, audit guidance, docs examples, and focused lifecycle tests so systemd restarts wait long enough, or otherwise defer stale cleanup until an existing gateway has had a fair health-probe window before it is killed.

## Reproduction Assessment

Yes at source level, but not with a live WSL2 systemd run in this read-only review. Current main renders `RestartSec=5`, documents exit-78 only after a healthy `/healthz`, and still has service-mode stale PID cleanup before the supervised recovery loop can observe a healthy prior process.

## Solution Assessment

Yes, the narrowest maintainable path is to align the canonical systemd timing and doctor/docs guidance with the documented exit-78 health handoff. `Type=notify` may be more robust later, but it is broader than the immediate bug boundary.

## Repair Work Prompt

Fix the systemd gateway restart timing race reported in https://github.com/openclaw/openclaw/issues/80696. Current main renders and recommends `RestartSec=5`, while gateway-lock exit-78 protection only fires after the existing gateway answers `/healthz`; slow startup under WSL2 can let service-mode stale PID cleanup kill the prior instance before it becomes healthy. Keep the fix scoped to canonical systemd behavior: update `src/daemon/systemd-unit.ts` and `src/daemon/service-audit.ts` so the default/recommended restart delay gives the gateway enough time to become healthy, update the systemd examples in docs, and add focused tests for the rendered unit/audit/docs-adjacent behavior. If you instead change stale-PID cleanup, preserve non-systemd and manual duplicate-start behavior and add regression coverage for the supervised health handoff. Do not add a new config option or broad `Type=notify` implementation unless maintainer review explicitly chooses that path. Add a changelog entry because this changes user-visible gateway/systemd behavior.
