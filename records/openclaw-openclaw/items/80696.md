## Summary

Keep open: current main still ships `RestartSec=5` with `Restart=always` and exit-78 protection that only fires after a successful `/healthz` probe, so the reported warmup race remains source-backed and unresolved.

## What This Changes

The issue asks OpenClaw to adjust the systemd gateway restart timing or lock behavior so exit-78 duplicate-start protection works when gateway warmup exceeds the current restart delay.

## Best Possible Solution

Update the systemd gateway lifecycle contract so the generated unit, audit/doctor checks, docs, and regression tests use a restart delay or lock behavior that lets normal startup expose `/healthz` before duplicate-start protection is evaluated.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection does show the path: current units restart after 5s, exit 78 requires a healthy `/healthz`, and gateway startup can spend significant work before the HTTP listener binds.

## Solution Assessment

Yes for a focused systemd lifecycle repair that updates the default restart timing plus audit/docs/tests. A `Type=notify` redesign may be more robust, but it is broader than needed for a first fix unless maintainers choose that direction.
