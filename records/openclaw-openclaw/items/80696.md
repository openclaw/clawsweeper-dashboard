## Summary

Keep open: current main and the latest release still ship the 5s systemd restart delay and doctor/audit still treats 5s as preferred, while the documented exit-78 protection depends on the previous gateway being healthy on `/healthz`.

## What This Changes

The issue requests changing the gateway systemd restart/health timing so duplicate supervised restarts do not kill a still-warming gateway before exit 78 can stop the restart loop.

## Best Possible Solution

Align the shipped systemd defaults, doctor/audit guidance, docs, and focused regression coverage around a restart delay that gives normal gateway startup enough time to expose `/healthz`, while preserving the exit-78 loop-break contract.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection still supports the reported path: current units use `RestartSec=5`, exit 78 requires a healthy `/healthz` responder, and startup can spend time before the HTTP listener is bound.

## Solution Assessment

Yes for a focused repair that updates the systemd restart timing contract and its audit/docs/tests. A `Type=notify` redesign may be more robust, but it is broader than needed for the first maintainable fix.
