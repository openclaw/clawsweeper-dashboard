## Summary

Keep open: current main still ships and audits the Linux systemd gateway unit around `RestartSec=5`, while the documented exit-78 protection depends on a duplicate starter observing an existing healthy `/healthz` responder. The report is recent, detailed, source-aligned, and distinct from the earlier fixed EADDRINUSE exit-78 loop, so it should be handled as a narrow gateway/systemd bug rather than closed.

## What This Changes

Raise or otherwise harden the gateway systemd restart timing so gateway-lock exit-78 protection can break duplicate-start restart cascades on slower hosts.

## Best Possible Solution

Update the canonical Linux systemd service contract and matching doctor/docs/tests so duplicate-start protection has enough time to observe a healthy existing gateway, or add an equally narrow warmup-aware stale-process guard that preserves the documented exit-78 behavior.

## Reproduction Assessment

Yes at source level: current main has the reported `RestartSec=5` service/audit/doc default, exit-78 relies on `/healthz`, and stale restart cleanup can kill verified gateway PIDs without a health/warmup check. I did not establish a live WSL2/systemd timing reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes for a narrow first fix: raising the canonical `RestartSec` and aligning doctor/docs/tests is the smallest maintainable change for the title problem. A `Type=notify` readiness redesign or a deeper stale-kill policy change may be better long-term, but should not be required to fix the default unit race.
