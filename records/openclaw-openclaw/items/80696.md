## Summary

Keep open: current main still ships and audits the same 5s systemd restart delay while the documented exit-78 guard depends on an existing gateway becoming reachable on `/healthz`; no closing PR or canonical issue supersedes this specific timing bug.

## What This Changes

The issue asks OpenClaw to prevent systemd gateway restart storms by aligning the default service restart timing and lock behavior with slow gateway warmup.

## Best Possible Solution

Rebaseline the generated systemd unit, service audit, docs, and regression tests around a restart delay or equivalent lock policy that lets normal startup expose `/healthz` before the exit-78 duplicate guard is needed.

## Reproduction Assessment

No, not as a live end-to-end repro in this review. The source path is clear on current main: a 5s systemd restart delay races a `/healthz`-dependent exit-78 guard while the gateway can hold the lock before binding HTTP.

## Solution Assessment

Yes for a focused repair that updates the systemd timing contract and its audit/docs/tests while preserving the existing exit-78 behavior. A `Type=notify` redesign may be stronger later, but it is broader than the first maintainable fix for this report.
