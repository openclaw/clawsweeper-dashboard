## Summary

Keep open: current main and the latest release still ship and recommend `RestartSec=5`, while the documented exit-78 protection depends on a duplicate starter first observing a healthy `/healthz` responder. The report remains a concrete gateway/systemd lifecycle bug with source-backed timing risk.

## What This Changes

The issue asks OpenClaw to adjust its systemd gateway restart timing or lock behavior so slow gateway warmup does not defeat the documented exit-78 loop breaker.

## Best Possible Solution

Update the shipped systemd restart timing contract, audit guidance, docs, and focused regression coverage so normal gateway warmup has time to expose `/healthz` before a supervised duplicate can defeat exit 78.

## Reproduction Assessment

Yes at source level, but not from a live WSL2/systemd run in this review. Current main still combines `RestartSec=5` with exit-78 behavior that only fires after a healthy `/healthz` probe, so a focused test can simulate slow warmup exceeding the supervisor restart interval.

## Solution Assessment

Yes for a narrow repair that changes the systemd restart timing contract and aligned audit/docs/tests. A full `Type=notify` redesign may be more robust later, but it is broader than needed to fix the reported default-unit bug.
