## Summary

Keep open: current main and the latest shipped release still use `RestartSec=5`, and the service audit still recommends 5s, while the documented systemd exit-78 path still depends on an existing `/healthz` responder becoming healthy before the next supervised duplicate start.

## What This Changes

The issue requests changing the shipped gateway systemd restart timing or lock behavior so `Restart=always` does not race slow `/healthz` warmup and defeat the documented exit-78 duplicate-start protection.

## Best Possible Solution

Align the generated systemd unit, service audit, Linux/gateway docs, and regression tests around a restart delay that gives normal gateway startup enough time to expose `/healthz`, while preserving the existing exit-78 contract.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection does show the reported path on current main: `RestartSec=5`, health-gated exit 78, and startup work before the HTTP listener exposes `/healthz`.

## Solution Assessment

Yes for a focused repair that updates the systemd restart timing contract and its audit/docs/tests. A `Type=notify` redesign is a broader alternative and should not be the first repair unless maintainers choose that direction.
