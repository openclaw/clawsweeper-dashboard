Fix https://github.com/openclaw/openclaw/issues/80696. Current main renders and audits the Linux gateway systemd unit with `RestartSec=5`, but the supervised gateway-lock loop only exits 78 after an existing gateway answers `/healthz`; on slower WSL2 warmup this can let a new start arrive before the previous process is healthy and keep the restart cascade alive. Prefer the narrow fix: raise the generated systemd `RestartSec` default to a warmup-safe value such as 60s, update service audit/doctor messaging and docs/examples to match, and add or update regression tests. Keep the existing `RestartPreventExitStatus=78` contract and unmanaged duplicate-start behavior. Do not add a new config flag or implement `Type=notify` unless maintainers explicitly ask for that broader design. If you touch gateway-lock pre-bind ownership semantics, keep it narrowly scoped and preserve stale-lock recovery. Add a changelog entry because this changes user-visible service behavior.
