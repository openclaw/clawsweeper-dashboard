## Summary

Keep open: the PR targets a real stuck-turn symptom, but the patch shortens active embedded-run abort timing using item-level terminal-looking Codex progress that the linked P1 report says is not durable run/session completion evidence.

## What This Changes

The PR adds a 60-second terminal-progress orphan threshold in `src/logging/diagnostic.ts` and diagnostic tests for terminal versus non-terminal stale embedded-run progress.

## Best Possible Solution

Define a durable orphan predicate for embedded Codex runs before shortening active-abort timing, then prove true orphan recovery and delayed-useful-progress safety with focused tests and redacted runtime logs.

## Reproduction Assessment

Yes at source level: current code and tests show terminal-looking Codex progress can lead to active abort recovery after the abort threshold, and the PR's fake-timer tests exercise the proposed earlier threshold. I did not run a live Codex/channel reproduction in this read-only review.

## Solution Assessment

Unclear: the patch is narrow for the timeout symptom, but it uses a terminal-looking item event plus elapsed time without proving the embedded run is truly orphaned. A safer solution would require durable run/session completion or another explicit no-progress invariant before early active abort.
