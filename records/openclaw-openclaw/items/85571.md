## Summary

Keep open: the PR targets a real diagnostic stall symptom, but it currently changes active embedded-run recovery to abort much sooner from the same terminal-looking item signal that an open P1 report says is not a durable run/session completion signal. It also still needs after-fix real behavior proof beyond mocked/local test output before merge.

## What This Changes

The PR adds a 60-second terminal-progress orphan recovery threshold in `src/logging/diagnostic.ts` and adds diagnostic tests for terminal versus non-terminal stale embedded-run progress.

## Best Possible Solution

Define a durable orphan predicate for embedded Codex runs before shortening active-abort timing, then prove true orphan recovery and delayed-useful-progress safety with focused tests and redacted real runtime logs.

## Reproduction Assessment

Yes at source level: current tests and code show terminal-looking Codex progress can lead to active abort recovery after the abort threshold, and the PR's fake-timer tests exercise the proposed earlier threshold. I did not run a live Codex/channel reproduction because this review is read-only.

## Solution Assessment

Unclear: the patch is narrow for the timeout symptom, but the proposed predicate uses a terminal-looking item event plus elapsed time without proving the embedded run is truly orphaned. A safer solution would require a durable run/session terminal signal or another explicit no-progress invariant before early active abort.
