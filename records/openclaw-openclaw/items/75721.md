## Summary

Keep open: this maintainer-labeled PR is an active, unmerged implementation candidate, but the latest head still has blocking plugin install/runtime issues and a concrete supply-chain ordering regression. The central direction can be repaired, but this patch should not merge as-is.

## What This Changes

The PR removes bundled/runtime dependency staging, moves npm and git plugin installs into OpenClaw-owned user roots, externalizes QQ Bot as downloadable, and updates plugin loading, doctor cleanup, docs, CI, Docker, and tests around install-time plugin dependency ownership.

## Best Possible Solution

Revise the PR while keeping the install-time dependency model: preserve SDK alias compatibility, make npm/git package roots first-class discovery/index/load/uninstall/update targets, scan git plugin source before package-manager work, narrow doctor cleanup to OpenClaw-generated legacy artifacts, and then rerun focused plugin install/runtime tests plus the changed gate.

## Reproduction Assessment

Yes. The blockers reproduce from the latest PR source by inspecting wrapper exports, install roots versus discovery roots, git install ordering, doctor cleanup target collection, and current uninstall safety; no mutating test run was needed for these code-path regressions.

## Solution Assessment

No. The product direction may be maintainable, but this implementation is not the narrowest safe fix until it preserves compatibility, discovery, cleanup, uninstall, and pre-install scanning contracts around the new roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve default-safe SDK alias wrappers:** `src/plugins/plugin-sdk-dist-alias.ts:12-13`
  - body: The generated alias unconditionally re-exports `default` from every SDK subpath. Most `dist/plugin-sdk/*.js` files only have named exports, so ESM rejects the wrapper during instantiation before packaged plugins can use named SDK imports. Keep the previous namespace-safe fallback or only re-export `default` when the source module actually has one.
  - confidence: 0.94
- **[P1] Add discovery for package install roots:** `src/plugins/install.ts:1146-1147`
  - body: Npm plugins now install under `.openclaw/npm/node_modules/<pkg>` and git plugins under `.openclaw/git/...`, but discovery still starts from bundled, `.openclaw/extensions`, workspace, and configured load paths. Install records are applied after candidates are found, so a successful package install can be recorded without appearing in the refreshed registry or Gateway load set.
  - confidence: 0.9
- **[P2] Scan git plugin sources before npm install:** `src/plugins/git-install.ts:292-309`
  - body: This path runs `npm install` inside the cloned git repository before `installPluginFromInstalledPackageDir` performs the plugin package source safety scan. Even with lifecycle scripts disabled, rejected source can now trigger dependency resolution and network/package-manager work first; keep source validation ahead of dependency convergence for git installs.
  - confidence: 0.84
- **[P1] Stop deleting plugin node_modules during doctor cleanup:** `src/commands/doctor/shared/plugin-dependency-cleanup.ts:37-46`
  - body: The cleanup treats any `node_modules` directory under extension roots as removable legacy state, but the PR also makes local/path/archive installs rely on preinstalled dependencies. Running `openclaw doctor --fix` after such an install can delete valid runtime dependencies and break the plugin on the next load.
  - confidence: 0.9
- **[P2] Delete managed npm installs on uninstall:** `src/plugins/install.ts:1146-1147`
  - body: The new npm install target is `.openclaw/npm/node_modules/<pkg>`, while uninstall safety still only trusts paths in the managed `extensions` layout. Uninstall can fall back to removing `.openclaw/extensions/<pluginId>` and leave the actual npm package directory behind, so repeated install/uninstall cycles accumulate stale plugin code.
  - confidence: 0.86
