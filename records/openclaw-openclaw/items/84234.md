## Summary

Keep open: the fix direction is valid and the supplied live proof is useful, but the write-side flush gate has a blocking session-store integration bug. When the flush check fails, the PR omits replacement metadata but does not explicitly clear an existing persisted Claude CLI binding, so a stale session id can survive and hit the same missing-transcript reset on the next turn.

## What This Changes

The PR adds a Claude CLI transcript-flush gate before persisting CLI session bindings and invalidates reusable Claude CLI sessions whose JSONL transcript ends with an unmatched tool_use.

## Best Possible Solution

Keep the two recovery gates, but make a failed Claude CLI flush explicitly clear the provider's persisted binding state and align the orphan transcript probe with the deterministic path if the related probe PR lands first.

## Reproduction Assessment

Yes. Current main source shows CLI session ids are persisted without a write-side flush gate and only checked before later reuse, while the linked issue and PR body provide live logs for the observed missing-transcript context-loss and orphan-tool no-reply symptoms.

## Solution Assessment

No, not yet. The approach is maintainable, but the current write-side implementation must clear existing persisted bindings when the flush gate fails; otherwise the stale session id can survive the exact path this PR is trying to fix.

- **[P1] Clear the persisted binding when the flush gate fails:** `src/agents/cli-runner.ts:573-586`

Please add an explicit clear signal/path and a regression test with a pre-existing Claude CLI binding.
