## Summary

Keep this PR open for maintainer handling. The active/stopped steering direction is useful and still targets a real current-main bug, but the branch is dirty against current main and the added production rate gate can suppress valid rapid steering messages before they reach the active run.

## What This Changes

The PR changes embedded and reply-run steering gates from streaming-only to lifecycle-aware active/stopped checks, removes stale reply-run streaming parameters, adds a shared steer-injection size/rate policy with tests, and edits the changelog.

## Best Possible Solution

Replay the lifecycle-aware active/stopped steering fix onto current `embedded-agent-runner` paths, remove or redesign the throttle so valid rapid steering messages are not dropped, and leave changelog generation to release tooling.

Yes, at source level. Current main and `v2026.6.8` still gate active steering on `isStreaming`, while the docs describe active-run delivery at tool/model boundaries; I did not run a live channel scenario in this read-only pass.

## Solution Assessment

No for the current branch. The lifecycle-aware gate is the right fix direction, but this PR adds a dropping throttle and needs a rebase onto the current embedded-runner paths before it is a safe fix.
