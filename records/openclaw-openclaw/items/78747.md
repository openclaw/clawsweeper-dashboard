## Summary

Keep open: the source-level cache-prefix issue is still present on current main, and this PR is a narrow functional fix with sufficient real traffic proof; the only issue I found is a small changelog attribution cleanup.

## What This Changes

The PR moves OpenAI Responses tool conversion ahead of params construction so `tools` serializes before `input`, adds a key-order regression test, and records the change in the changelog.

## Best Possible Solution

Land the shared-builder key-order fix with its regression test after adding the missing changelog contributor credit; keep broader prompt-cache work in separate focused issues.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main inserts `input` before a later `params.tools` assignment, and the OpenAI SDK serializes the resulting object with `JSON.stringify`. I did not run a live gateway in this read-only review, but the PR body and comment thread include real Codex traffic evidence matching that source path.

## Solution Assessment

Yes. Moving only the existing `tools` value ahead of `input` in the shared Responses params builder is the narrowest maintainable fix because it preserves field values and changes only serialization order, with a direct regression test for the cache-sensitive contract.
