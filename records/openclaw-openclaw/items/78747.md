## Summary

Keep open: the cache-order fix is valid and has convincing real traffic proof, but the current PR head still regresses existing Responses `tool_choice` support from main, so it needs a small repair before merge.

## What This Changes

The PR hoists OpenAI Responses tool conversion before the params literal, emits `tools` before trailing `input`, adds key-order coverage, and adds a changelog entry.

## Best Possible Solution

Land the key-order fix with `tool_choice` included alongside the hoisted `tools` path and keep both the existing `tool_choice` test and the new key-order regression test.

## Reproduction Assessment

Yes. Source inspection shows the current builder inserts `input` before a later `tools` assignment and the OpenAI SDK serializes the object with `JSON.stringify`; the PR body plus an independent production comment provide real cache-drop evidence.

## Solution Assessment

Mostly yes. Hoisting the existing converted tools value before `input` is the narrow maintainable fix, but the branch must carry forward current-main `toolChoice` forwarding before it is mergeable.
