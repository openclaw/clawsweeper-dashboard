## What This Changes

The PR reorders OpenAI Responses request construction so converted `tools` are inserted before the per-turn `input`, adds a key-order regression test, and edits `CHANGELOG.md`.

## Best Possible Solution

Land a rebased version that keeps current Responses projection and `tool_choice` semantics, emits `tools` before per-turn `input`, and removes the normal-PR `CHANGELOG.md` edit.

## Reproduction Assessment

Yes, source inspection reproduces the request-order bug: current main and `v2026.6.8` insert `input` before appending `tools`, and the OpenAI SDK serializes the object with `JSON.stringify`. I did not rerun live traffic, but the PR body includes after-fix production Codex logs and byte-prefix measurements.

## Solution Assessment

Yes, after rebase this is the narrowest maintainable fix: moving the existing converted tools into the params literal before `input` addresses the byte-prefix cause without changing field values or provider routing. A custom serializer or fetch-layer rewrite would be broader and harder to keep aligned with the shared builder.
