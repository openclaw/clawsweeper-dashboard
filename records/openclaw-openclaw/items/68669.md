## Summary

Keep open: the duplicate browser-cleanup dispatch is real and the proposed per-run marker is the right narrow direction, but the current diff still returns before the shared completion tail, which can delay final delivery behind a slow first browser cleanup call.

## What This Changes

The PR adds a `browserCleanupDispatchedAt` run-record flag, resets it on rearm, guards subagent browser cleanup dispatch, and adds a parallel-completion lifecycle regression test.

## Best Possible Solution

Keep the per-run browser cleanup dispatch marker, but make duplicate completion callers skip only `cleanupBrowserSessionsForLifecycleEnd` and still continue the existing runtime-retirement and announce-cleanup tail with slow-cleanup regression coverage.

## Reproduction Assessment

Yes. Current main has two completion paths that call `completeSubagentRun(..., triggerCleanup: true)` for the same run, and the PR body supplies before/after terminal output demonstrating duplicate cleanup dispatch across 100 parallel-completion trials.

## Solution Assessment

No, not as currently written. The per-run marker is the maintainable fix shape, but the duplicate branch should skip only the browser cleanup wrapper and continue finalization instead of returning from `completeSubagentRun`.
