## Summary

Keep open. The PR targets a real current-main duplicate browser-cleanup dispatch, but the current patch returns from `completeSubagentRun` on the duplicate path before the normal runtime-retirement and announce-cleanup tail can run.

## What This Changes

The branch adds a `browserCleanupDispatchedAt` run-record flag, resets it on rearm, guards browser cleanup dispatch, and adds a parallel-completion regression test.

## Best Possible Solution

Deduplicate only the browser cleanup wrapper, then let the existing runtime-retirement and announce-cleanup path proceed, with regression coverage for a slow first cleanup plus a duplicate completion signal.

## Reproduction Assessment

Yes. Source inspection shows current main starts both completion paths for the same registered subagent, and both converge on `completeSubagentRun(..., triggerCleanup: true)` while the browser cleanup wrapper is unguarded.

## Solution Assessment

No, not as currently written. The per-run dispatch marker is the right narrow shape, but the guard must skip only `cleanupBrowserSessionsForLifecycleEnd` rather than returning before finalization can be leased and started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Continue the cleanup tail after the duplicate browser guard:** `src/agents/subagent-registry-lifecycle.ts:876-877`
  - body: The new guard returns from `completeSubagentRun` when a concurrent caller already set `browserCleanupDispatchedAt`. That skips `retireRunModeBundleMcpRuntime` and `startSubagentAnnounceCleanupFlow` for the duplicate signal, so a slow first browser cleanup still blocks final delivery instead of allowing the second caller to continue the existing finalization path. Please skip only `cleanupBrowserSessionsForLifecycleEnd` and let the tail run.
