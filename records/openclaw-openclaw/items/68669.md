## Summary

Keep open. The duplicate cleanup dispatch is real and the supplied proof is adequate, but the current patch returns from `completeSubagentRun` when the second caller sees the new flag, so that caller no longer reaches the existing announce cleanup path while the first caller is still awaiting browser cleanup.

## What This Changes

The PR adds a `SubagentRunRecord.browserCleanupDispatchedAt` flag, uses it to skip duplicate browser-session cleanup dispatch in `completeSubagentRun`, resets it on rearm, and adds a lifecycle regression test.

## Best Possible Solution

Deduplicate only `cleanupBrowserSessionsForLifecycleEnd` while still allowing duplicate completion signals to continue into the shared announce cleanup/finalization lease, with regression coverage for a slow first browser cleanup call.

## Reproduction Assessment

Yes, at source level. Current main starts both completion paths for a registered subagent, both can call `completeSubagentRun(..., triggerCleanup: true)`, and there is no current browser cleanup dispatch guard.

## Solution Assessment

No, not as currently written. A per-run dispatch marker is the right narrow shape, but it should skip only the browser cleanup wrapper rather than returning before the shared announce cleanup path can run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Continue announce cleanup after skipping browser cleanup:** `src/agents/subagent-registry-lifecycle.ts:873-875`
