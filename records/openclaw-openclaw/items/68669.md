## Summary

Keep open: the reported duplicate browser-cleanup dispatch still exists on current main, but this PR still has a blocking correctness issue because the duplicate guard returns before the existing runtime-retirement and announce-cleanup tail can run.

## What This Changes

The PR adds a `browserCleanupDispatchedAt` run-record flag, resets it on rearm, guards browser cleanup dispatch, and adds a parallel-completion regression test.

## Best Possible Solution

Deduplicate only the browser cleanup wrapper, then continue the existing runtime-retirement and announce-cleanup path, with regression coverage for a slow first cleanup plus a duplicate completion signal.

## Reproduction Assessment

Yes. Current main starts both completion paths for the same registered subagent, and both can reach `completeSubagentRun(..., triggerCleanup: true)` while the browser cleanup wrapper has no per-run dispatch marker; the PR body also includes terminal before/after trial output.

## Solution Assessment

No, not as currently written. The per-run marker is the right narrow approach, but the duplicate branch must skip only `cleanupBrowserSessionsForLifecycleEnd` instead of returning before finalization can be leased and started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Continue the cleanup tail after the duplicate browser guard:** `src/agents/subagent-registry-lifecycle.ts:876-877`
  - body: The new guard returns from `completeSubagentRun` when a concurrent caller already set `browserCleanupDispatchedAt`. That skips `retireRunModeBundleMcpRuntime` and `startSubagentAnnounceCleanupFlow`, so a slow first browser cleanup can still delay final delivery instead of allowing the duplicate caller to continue the existing finalization path. Please skip only the browser cleanup wrapper and let the tail run.
