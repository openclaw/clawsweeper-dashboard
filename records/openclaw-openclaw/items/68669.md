## Summary

The PR addresses a real duplicate browser-cleanup dispatch bug, but the current patch still returns before the shared announce/finalization path when a duplicate completion signal arrives after the new flag is set.

## What This Changes

Adds a `browserCleanupDispatchedAt` run-record flag, guards browser cleanup dispatch in subagent completion, resets it on rearm, and adds a parallel-completion regression test.

## Best Possible Solution

Deduplicate only the browser cleanup wrapper while preserving the normal announce cleanup and finalization path, with a regression test for a slow first cleanup and a duplicate completion signal.

## Reproduction Assessment

Yes, at source level. Current main starts both completion paths for a registered subagent and both can call `completeSubagentRun(..., triggerCleanup: true)` against an unguarded browser cleanup wrapper; the PR body also supplies before/after terminal counts.

## Solution Assessment

No, not as currently written. A per-run dispatch marker is the right narrow shape, but the guard should skip only `cleanupBrowserSessionsForLifecycleEnd` instead of returning before announce cleanup can be leased and started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Continue announce cleanup after skipping browser cleanup:** `src/agents/subagent-registry-lifecycle.ts:873-875`
  - body: This return exits `completeSubagentRun` as soon as another caller has set `browserCleanupDispatchedAt`, so the duplicate embedded completion signal cannot reach `retireRunModeBundleMcpRuntime` or `startSubagentAnnounceCleanupFlow`. Since the first caller sets the flag before awaiting browser tab close, a slow first cleanup now keeps completion delivery behind that close instead of letting the second caller skip only browser cleanup and continue the existing finalization path.
  - confidence: 0.86
