## Summary

Keep this PR open. The bug is valid and the proof is sufficient, but the current patch still has a blocking cleanup-ordering bug: the new dispatch guard returns before the existing announce cleanup/finalization path can run.

## What This Changes

The branch adds a per-run `browserCleanupDispatchedAt` field, guards browser cleanup dispatch in `completeSubagentRun`, resets it on rearm, and adds a parallel-completion regression test.

## Best Possible Solution

Deduplicate only the browser cleanup wrapper while still allowing the normal runtime retirement and announce cleanup/finalization path to proceed, with regression coverage for a slow first browser-cleanup call plus a duplicate completion signal.

## Reproduction Assessment

Yes, at source level. Current main starts both completion paths for a registered subagent and both can call `completeSubagentRun(..., triggerCleanup: true)` without a browser cleanup dispatch guard; the PR body also supplies before/after terminal counts.

## Solution Assessment

No, not as currently written. A per-run dispatch marker is the right narrow shape, but the guard should suppress only `cleanupBrowserSessionsForLifecycleEnd` instead of returning before announce cleanup can be leased and started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Continue announce cleanup after skipping browser cleanup:** `src/agents/subagent-registry-lifecycle.ts:873-875`
  - body: This early return skips more than the duplicate browser cleanup call. If another completion caller already set `browserCleanupDispatchedAt`, the second caller exits before `retireRunModeBundleMcpRuntime` and `startSubagentAnnounceCleanupFlow`; with a slow first browser cleanup, completion delivery is now held behind that first cleanup instead of letting the duplicate signal skip browser work and start the existing guarded cleanup/finalization path. Please make the flag skip only the browser cleanup dispatch.
  - confidence: 0.84
