## Summary

Keep open. The reported duplicate browser-cleanup dispatch is real and current main still lacks a guard, but this PR should not merge as-is because the new guard returns from the whole completion tail before announce cleanup can start.

## What This Changes

Adds a `browserCleanupDispatchedAt` run-record field, guards browser cleanup dispatch in `completeSubagentRun`, resets it on rearm, and adds a parallel-completion regression test.

## Best Possible Solution

Deduplicate only the browser cleanup wrapper while preserving the normal announce cleanup/finalization path, with regression coverage for a slow first browser-cleanup call and a duplicate completion signal.

## Reproduction Assessment

Yes, at source level. Current main starts both completion paths for a registered subagent and both can call `completeSubagentRun(..., triggerCleanup: true)` against an unguarded browser cleanup wrapper; the PR body also supplies before/after terminal counts.

## Solution Assessment

No, not as currently written. A per-run dispatch marker is the right narrow shape, but the guard should suppress only the browser cleanup dispatch rather than returning before announce cleanup can be leased and started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Continue announce cleanup after skipping browser cleanup:** `src/agents/subagent-registry-lifecycle.ts:873-875`
  - body: This guard returns from `completeSubagentRun` as soon as another caller has set `browserCleanupDispatchedAt`, so the duplicate embedded completion signal can no longer reach `retireRunModeBundleMcpRuntime` or `startSubagentAnnounceCleanupFlow`. Because the first caller sets the flag before awaiting browser tab close, a slow or stuck browser cleanup now holds completion delivery behind that first cleanup instead of letting the second caller skip the already-drained browser work and start the existing announce cleanup flow. Please make the flag skip only `cleanupBrowserSessionsForLifecycleEnd` while still allowing the shared cleanup/finalization path to proceed once.
  - confidence: 0.84
