## Summary

Keep open: the duplicate browser-cleanup dispatch is real and the PR is narrowly aimed at it, but the current guard returns from `completeSubagentRun` before the remaining runtime-retirement and announce-cleanup tail.

## What This Changes

The PR adds `browserCleanupDispatchedAt` to subagent run records, resets it on rearm, uses it to dedupe browser cleanup dispatch, and adds a parallel-completion regression test.

## Best Possible Solution

Keep the dispatch flag, but make it suppress only `cleanupBrowserSessionsForLifecycleEnd`; duplicate completion callers should still continue through runtime retirement and announce cleanup with a held-first-cleanup regression.

## Reproduction Assessment

Yes, at source level. Current main can route both the in-process listener and gateway wait path into `completeSubagentRun(..., triggerCleanup: true)`, and the PR body supplies before/after terminal counts for the duplicate wrapper call.

## Solution Assessment

No. The dispatch flag is the right boundary, but returning from `completeSubagentRun` is too broad; the safer fix is to bypass only browser cleanup and keep the remaining completion tail live.
