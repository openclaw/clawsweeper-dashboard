- `merge-risk: 🚨 message-delivery`: The new early return can prevent a duplicate completion path from reaching announce cleanup and final delivery while the first browser cleanup is pending.

## Summary

Needs changes before merge: the PR targets a real duplicate browser-cleanup dispatch path, but the new guard returns from the whole completion function, so duplicate completion callers can no longer drive runtime retirement or announce cleanup while the first browser cleanup is still pending.

## What This Changes

The PR adds a `browserCleanupDispatchedAt` flag to subagent run records, resets it on rearm, uses it to dedupe browser cleanup dispatch, and adds a parallel-completion lifecycle regression test.

## Best Possible Solution

Keep the dedicated dispatch flag, but make it guard only the browser cleanup wrapper; duplicate completion callers should still continue through runtime retirement and announce cleanup, with a regression that holds the first cleanup promise.

## Reproduction Assessment

Yes. Current source and the PR body give a high-confidence path: parallel `completeSubagentRun(..., triggerCleanup: true)` calls duplicate the browser cleanup wrapper on main, and the supplied terminal proof shows before/after counts.

## Solution Assessment

No. The dispatch flag is the right boundary, but returning from `completeSubagentRun` is too broad; the safer fix is to bypass only browser cleanup and keep the remaining completion tail live.

## Review Findings

- **[P2] Keep duplicate completions on the finalization path:** `src/agents/subagent-registry-lifecycle.ts:955`
