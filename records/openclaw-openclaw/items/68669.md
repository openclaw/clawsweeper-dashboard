## Summary

Keep this PR open: the underlying duplicate browser-cleanup bug is real and the supplied proof is strong, but the current patch still returns from the whole completion method on the duplicate path, which can skip the completion finalization tail that delivers/cleans up the run.

## What This Changes

The PR adds a `browserCleanupDispatchedAt` run-record flag, resets it on rearm, guards browser session cleanup dispatch, and adds a parallel-completion lifecycle regression test.

## Best Possible Solution

Keep the dispatch flag, but make it suppress only `cleanupBrowserSessionsForLifecycleEnd`; duplicate completion callers should still continue through runtime retirement and `startSubagentAnnounceCleanupFlow`, with a regression where a held first cleanup still lets a duplicate caller reach the tail.

## Reproduction Assessment

Yes. Current source shows both the listener and gateway wait paths can call `completeSubagentRun(..., triggerCleanup: true)`, and the PR body supplies before/after terminal output for the duplicate cleanup dispatch path.

## Solution Assessment

No. The dispatch-flag direction is good, but returning from the whole completion method is too broad; the safer fix is to skip only browser cleanup while preserving the completion tail, or to adopt the broader lifecycle ordering work if maintainers choose that path.
