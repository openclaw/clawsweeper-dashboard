## Summary

Keep open: the PR fixes the duplicate browser-cleanup dispatch shape, but the current guard still returns from the whole `completeSubagentRun` tail and can skip runtime retirement plus announce cleanup for duplicate completion callers.

## What This Changes

The PR adds `browserCleanupDispatchedAt` to subagent run records, resets it on rearm, uses it to dedupe browser cleanup dispatch, and adds a parallel-completion lifecycle regression test.

## Best Possible Solution

Keep the dedicated dispatch flag, but make it guard only the browser cleanup wrapper; duplicate completion callers should still continue through runtime retirement and announce cleanup, with a regression that holds the first cleanup promise.

## Reproduction Assessment

Yes. Current source and the PR body give a high-confidence path: parallel `completeSubagentRun(..., triggerCleanup: true)` calls duplicate the browser cleanup wrapper on main, and the supplied terminal proof shows before/after counts.

## Solution Assessment

No. The dispatch flag is the right boundary, but returning from `completeSubagentRun` is too broad; the safer fix is to bypass only browser cleanup and keep the remaining completion tail live.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep duplicate completions on the finalization path:** `src/agents/subagent-registry-lifecycle.ts:955`
  - body: When a duplicate completion arrives after the first caller sets `browserCleanupDispatchedAt`, this `return` skips the rest of `completeSubagentRun`. That means the duplicate caller cannot reach `retireRunModeBundleMcpRuntime` or `startSubagentAnnounceCleanupFlow`, so a slow or held first browser cleanup can still hold completion delivery/final cleanup behind it. Make this guard skip only the browser cleanup wrapper and continue through the tail.
  - confidence: 0.89
