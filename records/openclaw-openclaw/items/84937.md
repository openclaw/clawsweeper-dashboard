## Summary

Keep this PR open, but it is not merge-ready: the Discord interaction seam is useful, while the bundled `/ask` command is unscoped, the rejection paths can edit the shared prompt after the dispatcher pre-acknowledges interactions, malformed component specs can now throw through native command handling, and the supplied proof only shows build artifacts rather than real Discord behavior.

## What This Changes

Adds a bundled Ask plugin with a Discord native `/ask` command, a new `plugin-sdk/discord-interactions` subpath, Discord native component reply registration, docs, tests, and workspace metadata.

## Best Possible Solution

Land a narrow, documented Discord interaction SDK/native-command component seam if maintainers want that API, and keep `/ask` Discord-scoped or publish it externally through ClawHub unless core bundling is explicitly approved.

## Solution Assessment

No. The reusable Discord interaction seam may be a good direction, but the bundled unscoped Ask plugin plus unguarded component parsing and missing live proof is not the narrowest maintainable merge path.

Full review comments:

- **[P1] Scope `/ask` to Discord only:** `extensions/ask/index.ts:24-26`
  - body: This command has no `channels` restriction and uses `nativeNames.default`, while current plugin commands default to every channel and the default native alias applies to all native providers.   - confidence: 0.96

  - body: The Discord plugin-interactive dispatcher acknowledges matched component interactions before invoking the handler, and after that `reply()` edits the original response. These rejection branches can therefore overwrite the shared `/ask` prompt instead of sending an ephemeral error to the clicker; use `ctx.respond.followUp(...)` for invalid, expired, unauthorized, and duplicate responses or avoid pre-ack before access checks.
  - confidence: 0.91
- **[P2] Treat malformed component specs as non-renderable:** `extensions/discord/src/monitor/native-command-reply.ts:67-70`
  - body: `readDiscordComponentSpec` throws for invalid objects, but this new renderability path calls it without a guard. A plugin returning malformed `channelData.discord.components` can now throw out of native command handling instead of falling back to the existing empty-visible-reply warning, so catch parse errors and return `undefined` here.
