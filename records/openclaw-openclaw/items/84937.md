## What This Changes

This PR adds a bundled Ask plugin with a Discord native /ask command, adds Discord component handling for native plugin command replies, and exposes a new plugin SDK subpath for Discord interactive handler types.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use follow-ups for rejected /ask interactions:** `extensions/ask/interactive-handler.ts:52-55`
  - body: These rejection branches call `ctx.respond.reply(..., ephemeral: true)`, but Discord plugin interactive handlers are acknowledged before the plugin runs. After that deferred message update, `reply()` edits the original component message, so unauthorized, expired, or duplicate clicks can overwrite the shared `/ask` prompt instead of sending a private denial; use `followUp()` for these branches or avoid the pre-ack before access checks.
  - confidence: 0.9
- **[P2] Treat malformed component specs as non-renderable:** `extensions/discord/src/monitor/native-command-reply.ts:70`
  - body: This helper forwards any object-shaped `channelData.discord.components` into `readDiscordComponentSpec`, whose parser throws on invalid or incomplete specs. Existing array-only behavior ignored unknown shapes, so a bad plugin reply can now bubble out of native command handling instead of falling back to `DISCORD_EMPTY_VISIBLE_REPLY_WARNING`; catch parse/build failures and treat them as non-renderable.
  - confidence: 0.86
- **[P3] Add routing metadata for the new Ask plugin:** `extensions/ask/openclaw.plugin.json:2`
  - body: Adding `extensions/ask` creates a new plugin and docs surface, but the labeler has no Ask entry even though the repo policy requires new plugin surfaces to update `.github/labeler.yml` and labels. Add the Ask plugin routing so future changes get the right maintainer labels.
  - confidence: 0.78
