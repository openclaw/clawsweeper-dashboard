## Summary

Keep open: current main still lacks the requested visible coordinator-to-ACP Discord-thread delivery path that bypasses main-agent Discord channel-session metadata creation, and the remaining work is a product/API/security decision rather than stale cleanup.

## What This Changes

The issue requests a supported visible agent-to-agent delivery path into ACP-bound Discord threads without creating `agent:main:discord:channel:<thread_id>` session entries.

## Best Possible Solution

Add a narrow, explicit visible proxy-delivery contract after maintainer API and security review, preserving normal outbound session metadata everywhere except the authorized ACP-bound proxy path.

## Reproduction Assessment

Yes from source for the contract gap: current main lacks `visible` or `proxyOnly` parameters and still records outbound visible-send session metadata. I did not run a live dual-Discord ACP setup, so the Discord 503 cascade itself is not live-reproduced here.

## Solution Assessment

Unclear until maintainers choose the API and security semantics. A narrow explicit visible proxy-send contract is safer than a blanket ACP-bound thread exclusion rule.
