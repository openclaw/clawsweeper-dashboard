## Summary

Keep open. Current main still lacks the requested visible coordinator-to-ACP-thread delivery path that bypasses main Discord channel-session metadata creation; the related A2A work now surfaces follow-up/announce output after an internal send, but it does not provide the initiating visible proxy-only instruction path described here.

## What This Changes

The issue requests a supported visible coordinator-to-ACP-thread message path that does not create or pollute the main Discord channel session.

## Best Possible Solution

Add a narrow, explicit, authorized visible proxy-delivery contract for ACP-bound Discord threads that bypasses main-session metadata only for that call, preserves normal outbound session recording elsewhere, and documents the supported coordination workflow.

## Reproduction Assessment

Yes for the feature gap by source inspection: current schemas reject visible/proxyOnly-style fields, sessions_send injects internally with deliver:false, and normal outbound message sends still record session metadata. I did not run a live dual-Discord-account reproduction in this read-only review.

## Solution Assessment

Unclear. The requested options are plausible, but maintainers should choose the API and authorization semantics; a narrow authorized proxy-delivery path is safer than a blanket ACP-bound-thread exclusion.
