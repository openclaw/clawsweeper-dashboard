## Summary

Keep open. Current main has solved adjacent A2A announce/reply delivery into bound threads, but it still lacks the requested visible proxy-only initiating message path that avoids writing main Discord channel-session metadata.

## What This Changes

The issue requests a visible coordinator-to-ACP-bound Discord thread delivery path that does not create main Discord channel-session metadata.

## Best Possible Solution

Add a narrow, explicit, authorized visible proxy-delivery contract for ACP-bound Discord threads that bypasses main-session metadata only for that call, preserves normal outbound session recording elsewhere, and documents the supported workflow.

## Reproduction Assessment

Yes for the feature gap by source inspection: current schemas reject visible/proxyOnly-style fields, sessions_send injects internally with deliver:false, and normal outbound message sends still record session metadata. I did not run a live dual-Discord-account reproduction in this read-only review.

## Solution Assessment

Unclear. The requested options are plausible, but maintainers should choose the API and authorization semantics; a narrow authorized proxy-delivery path is safer than a blanket ACP-bound-thread exclusion.
