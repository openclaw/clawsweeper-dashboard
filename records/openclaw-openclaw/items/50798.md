## Summary

Keep open. Current main and v2026.5.22 still lack the initiating visible proxy-only ACP Discord-thread delivery contract; the existing `sessions_send` path injects internally and can announce the worker reply, while normal `message` sends still create outbound session metadata.

## What This Changes

The issue requests a supported visible coordinator-to-ACP Discord thread delivery path that does not create a main Discord channel session for the target thread.

## Best Possible Solution

Add a narrow, explicit, authorized visible proxy-delivery contract for ACP-bound Discord threads that bypasses main-session metadata only for that call and preserves normal outbound session recording elsewhere.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.22 have no visible/proxyOnly field, `sessions_send` starts the nested run with `deliver:false`, and normal outbound message sends still write session metadata. I did not run a live dual-Discord-account reproduction.

## Solution Assessment

Unclear. The proposed options are plausible, but maintainers should choose the public API and authorization semantics; a narrow authorized proxy-delivery path is safer than a blanket ACP-bound-thread exclusion.
