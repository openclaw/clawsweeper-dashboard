## Summary

Keep open. Current main and latest release still lack the requested initiating visible proxy-only ACP Discord-thread delivery contract; existing `sessions_send` can surface an announce/reply after an internal send, while normal `message`/Gateway sends still create outbound session metadata.

## What This Changes

Request a supported way for a coordinator agent to post a visible instruction into an ACP-bound Discord thread without creating a main Discord channel-session entry.

## Best Possible Solution

Add a narrow, explicit, authorized visible proxy-delivery contract for ACP-bound Discord threads that bypasses main-session metadata only for that call and preserves normal outbound recording elsewhere.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.22 have no visible/proxyOnly field, `sessions_send` starts the nested run with `deliver:false`, and normal outbound message sends still write session metadata. I did not run a live dual-Discord-account reproduction.

## Solution Assessment

Unclear until maintainers choose the public contract. A narrow authorized proxy-delivery path is safer than a blanket ACP-bound-thread exclusion because it preserves existing message/session metadata behavior by default.
