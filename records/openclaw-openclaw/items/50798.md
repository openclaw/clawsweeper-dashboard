## Summary

Keep open: current main and the latest release have fixed adjacent bound-thread routing and worker reply mirroring, but OpenClaw still lacks the requested supported visible coordinator-to-ACP Discord-thread delivery path that avoids creating main-session metadata. The remaining work is a compatibility-sensitive product/API decision around visible delivery, authorization, loop prevention, and session-metadata bypass.

## What This Changes

The issue requests a supported visible agent-to-agent message path into ACP thread-bound Discord sessions without creating `agent:main:discord:channel:<thread_id>` sessions.

## Best Possible Solution

Choose an explicit supported visible delivery contract for ACP-bound sessions, with authorization and loop-prevention semantics, then implement it at the agent/gateway/channel boundary with Discord proof.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the product gap: current schemas and handlers expose only internal `sessions_send`/`sessions.send` delivery and no visible or proxy-only session bypass.

## Solution Assessment

Unclear until maintainers choose the API shape; a narrow explicit visible proxy-delivery contract appears safer than a blanket ACP-bound thread exclusion, but the permanent owner boundary and authorization model are product decisions.
