## Summary

Keep open. Current main still has the reported inbound-dispatch hot path: `dispatchReplyFromConfig` calls `ensureRuntimePluginsLoaded` for inbound replies, and that helper still falls through to `resolveRuntimePluginRegistry` instead of returning when a populated active runtime plugin registry already exists. The linked implementation PR #74118 is still open and unmerged with closing syntax, so this issue should remain open until that PR or an equivalent replacement lands.

## What This Changes

The issue requests that inbound dispatch avoid rebuilding and re-registering runtime plugins when the active plugin registry is already populated.

## Best Possible Solution

Keep this issue open until PR #74118 or an equivalent replacement lands; the shipped end state should make inbound dispatch reuse the already-populated runtime plugin registry and add focused regression coverage proving that a valid active registry avoids a rebuild.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: inbound dispatch still calls `ensureRuntimePluginsLoaded`, the helper still constructs reduced load options and calls `resolveRuntimePluginRegistry`, and the resolver still falls through to `loadOpenClawPlugins` when those options are incompatible with the active boot cache key.

## Solution Assessment

Unclear. The proposed active-registry fast path is the narrowest call-site fix for this issue, but a loader-level compatibility change could be the better shared solution if maintainers want to address the related strict cache-key misses together.
