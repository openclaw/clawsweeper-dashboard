## Summary

Keep open. Current main still has the reported inbound-dispatch path: dispatch loads `ensureRuntimePluginsLoaded`, that helper always calls the runtime plugin resolver with explicit reduced load options, and the resolver still falls through to `loadOpenClawPlugins` on an incompatible active cache key. The linked PR #74118 remains the implementation candidate from the supplied GitHub context, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that inbound dispatch calls `ensureRuntimePluginsLoaded` with reduced plugin-load options, causing active registry cache-key misses and repeated full plugin reloads per message.

## Best Possible Solution

Land PR #74118 or an equivalent focused fix that makes inbound dispatch reuse an already-populated active runtime plugin registry, with regression coverage proving the active-registry case does not rebuild or re-run plugin registration.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: current main still dispatches through `ensureRuntimePluginsLoaded`, the helper still supplies explicit reduced load options, and the resolver still reloads when those options do not match the active gateway boot cache key.

## Solution Assessment

Unclear. The proposed helper-level active-registry fast path is a narrow fix for this call site, but maintainers should decide whether to keep that local short-circuit or broaden loader compatibility semantics because the same cache-key family has related reports.
