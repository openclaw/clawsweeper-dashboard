## Summary

Keep open. Current main still has the reported inbound-dispatch path and no active-registry fast path in `ensureRuntimePluginsLoaded`; the issue is also paired with open PR #74118, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

Request that `ensureRuntimePluginsLoaded` reuse an already active plugin registry during inbound dispatch instead of rebuilding runtime plugins on every inbound message.

## Best Possible Solution

Keep this issue open until PR #74118 or an equivalent fix lands; the shipped behavior should avoid rebuilding the runtime plugin registry during inbound dispatch when a valid active registry already exists, with focused regression coverage for that no-reload path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still routes inbound replies through `ensureRuntimePluginsLoaded`, passes explicit reduced load options to the resolver, and falls through to a full plugin load when the active registry cache key is not compatible.

## Solution Assessment

Yes for this issue's hot path. A focused active-registry short-circuit or equivalent no-reload guard in `ensureRuntimePluginsLoaded` is the narrowest maintainable fix; broader subset-compatible loader cache behavior can be handled as separate follow-up work if maintainers want it.
