## Summary

Keep open. Current main still has the reported inbound-dispatch path, and the linked implementation PR is open and unmerged; that PR should be reviewed or replaced rather than closing this issue early.

## What This Changes

The issue requests that inbound dispatch avoid rebuilding runtime plugins when `ensureRuntimePluginsLoaded` is called after a valid active plugin registry already exists.

## Best Possible Solution

The shipped end state should make inbound dispatch reuse the already-populated runtime plugin registry when it is valid for this ensure-only call, with focused regression coverage proving that an active registry avoids a rebuild.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: inbound dispatch calls the ensure helper, the helper passes reduced explicit load options, and the resolver falls through to a full load when the active registry cache key is incompatible.

## Solution Assessment

Unclear. The active-registry fast path proposed in PR #74118 is the narrowest fix for this hot path, but maintainers should decide whether this should stay local to `ensureRuntimePluginsLoaded` or be generalized in loader compatibility handling.
