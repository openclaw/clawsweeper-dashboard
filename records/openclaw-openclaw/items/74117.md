## Summary

Keep open. Current main still has the reported inbound-dispatch plugin reload path, and the supplied GitHub context shows an open, unmerged paired PR with closing syntax for this issue, so maintainer cleanup should review, land, or replace that PR rather than close the issue early.

## What This Changes

The issue requests that inbound dispatch avoid rebuilding runtime plugins when the active plugin registry is already loaded.

## Best Possible Solution

Keep this issue open until #74118 or an equivalent replacement lands; the shipped end state should make inbound dispatch reuse an already-active runtime plugin registry and include focused regression coverage proving no rebuild when a valid active registry exists.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: inbound dispatch calls `ensureRuntimePluginsLoaded`, the helper always calls `resolveRuntimePluginRegistry` with reduced explicit options, and the resolver falls through to `loadOpenClawPlugins` when those options do not match the broader gateway boot cache key.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed active-registry fast path is the narrowest call-site fix for this issue, while a loader-level subset-compatibility change may be preferable if maintainers want one generic fix for related cache-key misses.
