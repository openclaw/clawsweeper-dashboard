## Summary

Keep open. Current main still has the reported inbound-dispatch path: replies call `ensureRuntimePluginsLoaded`, that helper still calls `resolveRuntimePluginRegistry` without an active-registry fast path, and the linked implementation PR is open and unmerged.

## What This Changes

Request that `ensureRuntimePluginsLoaded` reuse an already-populated active runtime plugin registry during inbound dispatch instead of rebuilding and re-registering plugins on each message.

## Best Possible Solution

Keep this issue open until PR #74118 or an equivalent replacement lands; the shipped end state should make inbound dispatch reuse the already-populated runtime plugin registry and add focused regression coverage proving that a valid active registry avoids a rebuild.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: inbound dispatch still calls the ensure helper, the helper still passes reduced explicit load options to the resolver, and the resolver still falls through to a full plugin load after a cache-key compatibility miss.

## Solution Assessment

Unclear. The proposed active-registry fast path is the narrowest fix for this call site, but maintainers should decide whether to solve this specific hot path or broaden compatibility handling in the plugin loader for the related cache-miss class.
