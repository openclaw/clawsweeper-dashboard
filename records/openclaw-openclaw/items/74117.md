## Summary

Keep open. Current main still has the reported inbound-dispatch path, and the paired implementation PR is open and unmerged, so this issue should stay open until maintainers review, land, or replace that fix.

## What This Changes

The issue requests that `ensureRuntimePluginsLoaded` stop rebuilding the runtime plugin registry on every inbound dispatch when an active registry is already loaded.

## Best Possible Solution

The shipped end state should make inbound dispatch reuse the already-populated runtime plugin registry when it is valid for this ensure-only call, with focused regression coverage proving that an active registry avoids a rebuild.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: inbound dispatch calls the ensure helper, the helper always calls the resolver with reduced explicit load options, and the resolver falls through to a fresh plugin load when active-registry compatibility misses.

## Solution Assessment

Unclear. The active-registry fast path proposed in PR #74118 is a narrow plausible fix, but maintainers should decide during review whether that should stay local to `ensureRuntimePluginsLoaded` or be generalized in loader compatibility handling.
