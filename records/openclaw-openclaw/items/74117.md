## Summary

Keep open. Current main still has the reported inbound-dispatch path: `dispatchReplyFromConfig` calls `ensureRuntimePluginsLoaded`, that helper still calls `resolveRuntimePluginRegistry` with explicit load options, and the resolver still falls through to `loadOpenClawPlugins` when the active registry cache key is incompatible. The linked closing PR #74118 remains the implementation candidate rather than proof that main is fixed.

## What This Changes

Make `ensureRuntimePluginsLoaded` reuse an already-active runtime plugin registry during inbound dispatch instead of rebuilding and re-registering plugins after gateway startup has loaded them.

## Best Possible Solution

Land PR #74118 or an equivalent narrow fix that lets inbound dispatch reuse the already-populated active runtime plugin registry, with focused regression coverage proving a valid active registry avoids a rebuild. Broader loader cache-key compatibility work can remain separate if maintainers want to address related cache misses together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: inbound dispatch calls the helper, the helper passes explicit reduced load options, and the resolver reloads when those options do not match the gateway boot registry cache key.

## Solution Assessment

Unclear. The proposed helper-level active-registry fast path is the narrowest fix for this hot path, but a loader-level compatibility change could be the better shared solution if maintainers want one cache semantics fix for the related strict-key reports.
