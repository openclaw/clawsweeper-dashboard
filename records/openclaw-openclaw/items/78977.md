## Summary

Keep open: the patch targets a plausible Responses continuation bug, but it changes proxy storage behavior without an explicit opt-in, lacks real LiteLLM/proxy proof, and omits the required changelog entry.

## What This Changes

The branch changes the OpenAI Responses payload policy so disable mode omits the `store` field for proxy-like endpoints and adds a unit test for a LiteLLM-style route.

## Best Possible Solution

Add an explicit provider/model Responses persistence opt-in or maintainer-approved default policy, document the retention tradeoff, cover native/proxy no-regression cases, and include real LiteLLM/proxy proof.

## Reproduction Assessment

No live high-confidence reproduction is available because the failing LiteLLM proxy is private. Source inspection and the linked logs give a clear current-main path: proxy-like Responses routes can send `store:false` while replaying prior Responses item IDs.

## Solution Assessment

No. Omitting `store:false` is small in code, but for OpenAI-backed proxies it can rely on the documented `store:true` default and change retention behavior without the explicit opt-in requested in the related issue discussion. ([platform.openai.com](https://platform.openai.com/docs/api-reference/responses/object?lang=node.js&utm_source=openai))
