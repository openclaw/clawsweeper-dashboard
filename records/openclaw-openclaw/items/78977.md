## Summary

Keep open: the linked bug is source-reproducible, but this patch should not merge as-is because it changes proxy retention defaults without an explicit opt-in and still lacks after-fix real proxy proof.

## What This Changes

The PR changes the Responses payload policy so disable mode omits `store` for proxy-like endpoints and adds a LiteLLM-style unit test.

## Best Possible Solution

Add or approve an explicit provider/model Responses persistence contract, document the retention tradeoff, cover native/proxy cases, and validate a real LiteLLM/proxy multi-turn run.

## Reproduction Assessment

No, not as a live repro: the failing LiteLLM proxy is private. Source inspection gives a clear current-main path where Responses item ids are replayed while `store:false` is applied to proxy-like routes.

## Solution Assessment

No. The diff targets the right branch, but omitting `store:false` globally for proxies changes retention semantics; an explicit persistence opt-in or maintainer-approved documented default is the safer solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate proxy response storage behind an explicit opt-in:** `src/agents/openai-responses-payload-policy.ts:329`
  - body: This makes every proxy-like Responses route omit `store:false`; OpenAI-backed proxies then inherit the Responses API default of `store:true`, which can persist application state without a documented operator opt-in.
