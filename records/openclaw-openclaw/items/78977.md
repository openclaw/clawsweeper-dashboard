## Summary

Keep open: the reported failure path is source-reproducible, but this patch changes proxy persistence defaults without an explicit opt-in and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `resolveOpenAIResponsesPayloadPolicy` so disable mode omits `store` for proxy-like Responses endpoints and adds a LiteLLM-style unit test.

## Best Possible Solution

Add an explicit provider/model Responses persistence opt-in or a maintainer-approved default, document the retention tradeoff, cover native/proxy no-regression cases, and prove a real LiteLLM/proxy multi-turn run.

## Reproduction Assessment

No live high-confidence reproduction is available because the failing LiteLLM proxy is private. Source inspection and the reporter's logs give a clear current-main path: proxy-like Responses routes can send `store:false` while replaying prior Responses item ids.

## Solution Assessment

No. The patch addresses the source path, but omitting `store:false` globally for proxies can change retention behavior; an explicit provider/model persistence opt-in is the safer maintainable direction unless maintainers approve the default change.

## Review Findings
