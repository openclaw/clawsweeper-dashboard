## Summary

Current main and the shipped v2026.5.12 release already preserve OpenAI-completions visible assistant text when the same response also contains native tool calls, and replay serializes assistant text alongside tool_calls instead of dropping it; the report was against 2026.5.7.

## What This Changes

The issue asks OpenClaw to preserve assistant text emitted with an OpenAI-completions tool call so the next turn can see the prior text and tool result instead of repeating the same tool.

## Best Possible Solution

Keep the shipped parser and replay behavior, and have affected 2026.5.7 users update to v2026.5.12 or newer; only reopen with a current-main transcript if Qwen still repeats tools after the preserved text/tool-call path is present.

## Reproduction Assessment

No live high-confidence reproduction against current main was established. The reported code path is clear from source and shipped tests: an OpenAI-completions chunk containing both visible content and tool_calls now produces both text and toolCall blocks.

## Solution Assessment

Yes. Preserving and replaying the assistant text/tool-call pair is the narrow contract fix; adding a generic duplicate-tool-call suppressor would be broader and could hide valid repeated tool use.
