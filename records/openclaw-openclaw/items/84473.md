## Summary

Keep open: current main still parses the ClawHub search `--limit` option with `Number.parseInt` in both plugin and skill search commands, so typo-like values can be accepted before the request path. The issue is narrow, source-reproducible, and I did not find a duplicate or protected-label reason to close it.

## What This Changes

Require strict positive-integer validation for `--limit` on `openclaw plugins search` and `openclaw skills search`.

## Best Possible Solution

Add a shared strict positive-integer option parser for these search commands, reject malformed/decimal/non-positive values before ClawHub is called, and cover both commands with focused regression tests.

## Reproduction Assessment

Yes, from source inspection. Current main parses both search `--limit` values with `Number.parseInt`, so `1abc` becomes `1`, `5.5` becomes `5`, and non-positive parsed values are defaulted or omitted downstream instead of being rejected.

## Solution Assessment

Yes. The maintainable fix is strict CLI-side integer validation using Commander-supported parser errors, plus tests proving invalid values fail before ClawHub mocks are called.
