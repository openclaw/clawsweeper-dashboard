## Summary

Keep open: current main and the latest release still parse both search `--limit` options with `Number.parseInt`, and the open fixing PR is linked with closing syntax but has not merged.

## What This Changes

Require `openclaw plugins search` and `openclaw skills search` to reject non-whole or non-positive `--limit` values before calling ClawHub.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84480 or an equivalent strict positive-integer Commander parser with focused CLI regression coverage, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected command registrations use `Number.parseInt`, and the downstream search paths either forward parsed partial positives or default invalid non-positive values instead of failing fast.

## Solution Assessment

Yes. Validating `--limit` as a whole positive integer at the Commander option parser boundary is the narrow maintainable fix because it stops invalid input before either ClawHub search action runs.
