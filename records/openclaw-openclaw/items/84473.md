## Summary

Keep open: current main and the latest release still use permissive `Number.parseInt` parsing for both affected search commands, and https://github.com/openclaw/openclaw/pull/84480 is an open linked fix with closing syntax. This issue should close after that PR or an equivalent strict parser fix lands.

## What This Changes

The issue asks for `openclaw plugins search` and `openclaw skills search` to reject partially numeric, decimal, zero, or negative `--limit` values before calling ClawHub.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84480 or an equivalent strict positive-integer parser with focused coverage for both search commands, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected Commander options use `Number.parseInt`, and the downstream search paths either forward parsed partial positives or default invalid non-positive values instead of failing fast.

## Solution Assessment

Yes. The narrow maintainable fix is to validate `--limit` as a whole positive integer at the Commander option boundary before either ClawHub search path can run.
