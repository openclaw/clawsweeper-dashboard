## Summary

Keep open: current main and the latest shipped release still parse `--limit` with `Number.parseInt` for both ClawHub plugin and skill searches, so partially numeric values can reach the search path instead of failing fast.

## What This Changes

Require `openclaw plugins search` and `openclaw skills search` to reject non-whole or non-positive `--limit` values before calling ClawHub.

## Best Possible Solution

Add a complete positive-integer parser for these two ClawHub search limits, preserve the omitted-limit default, and cover invalid values in focused CLI tests.

## Reproduction Assessment

Yes from source inspection: both command registrations use `Number.parseInt`, and the downstream search paths either forward truthy partial values or default non-positive values. I did not run the full CLI in this read-only review.

## Solution Assessment

Yes: failing fast in the CLI option parser before any ClawHub call is the narrow maintainable fix, with tests for `1abc`, `5.5`, `0`, and `-1`.

## Repair Work Prompt

Fix the current-main bug where `openclaw plugins search` and `openclaw skills search` accept partial or non-positive `--limit` values via `Number.parseInt`. Add or reuse a complete positive-integer parser for these two search commands only; reject values like `1abc`, `5.5`, `0`, and `-1` with a clear CLI error and exit before `searchClawHubPackages` or `searchSkillsFromClawHub` is called. Preserve omitted `--limit` default behavior and existing plugin result limiting semantics. Add focused regression coverage for both commands, avoid changing unrelated generic parsers that intentionally keep partial parse behavior, and add a user-facing changelog fix entry.
