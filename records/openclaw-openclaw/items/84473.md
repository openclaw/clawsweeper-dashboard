## Summary

Keep open: current main and the latest release still use permissive `Number.parseInt` handling for both affected `--limit` options, and an open linked PR with closing syntax already targets the fix.

## What This Changes

The issue asks for `plugins search --limit` and `skills search --limit` to reject partial, decimal, and non-positive numeric values before calling ClawHub.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84480 or an equivalent strict positive-integer Commander parser with focused coverage, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected command registrations use `Number.parseInt`, and the downstream search paths either forward parsed partial positives or normalize invalid values instead of failing fast.

## Solution Assessment

Yes. The narrow maintainable solution is to validate `--limit` as a whole positive integer at the Commander option boundary before either ClawHub search action can run.
