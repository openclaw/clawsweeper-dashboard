## Summary

Keep open: current main still parses `plugins search --limit` and `skills search --limit` with `Number.parseInt`, so partially numeric values remain accepted; the linked fixing PR is open with closing syntax and should resolve this issue when merged.

## What This Changes

The issue requests strict positive whole-integer validation for `openclaw plugins search --limit` and `openclaw skills search --limit`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84480 or an equivalent strict positive-integer parser with focused plugin and skill CLI coverage, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected command registrations use `Number.parseInt`, and the downstream search paths either forward parsed partial positives or default invalid non-positive values instead of failing fast.

## Solution Assessment

Yes. The narrow maintainable fix is to validate `--limit` as a whole positive integer in the CLI parser before any ClawHub call, with regression coverage for both plugin and skill search parsing.
