## Summary

Keep open: current main and the latest release still accept partially numeric `--limit` values, and the linked fixing PR is open with closing syntax, so this issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks `openclaw plugins search` and `openclaw skills search` to reject invalid `--limit` values as positive whole integers before making ClawHub requests.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84480 or an equivalent shared strict Commander positive-integer parser with focused CLI coverage, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected command registrations use `Number.parseInt`, and the downstream search paths either forward parsed partial positives or default invalid non-positive values instead of failing fast.

## Solution Assessment

Yes. The narrow maintainable solution is to validate `--limit` as a whole positive integer at the Commander option parser boundary before either ClawHub search action runs.
