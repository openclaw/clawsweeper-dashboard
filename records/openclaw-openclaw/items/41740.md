## Summary

This issue is still necessary: current main still routes Discord approval resolution through the shared operator-approval Gateway client without carrying a TLS fingerprint, and the related shared fix PR is open but not merged.

## What This Changes

The issue asks Discord/native exec approval Gateway clients to pass the local TLS fingerprint when connecting to a self-signed local WSS Gateway.

## Best Possible Solution

Refresh and land the shared bootstrap TLS fingerprint propagation from the related PR or an equivalent guarded implementation, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level: local TLS config produces a WSS target, current approval bootstrap cannot return a TLS fingerprint, and the operator approval GatewayClient never receives one. I did not run a live Discord self-signed Gateway repro in this read-only pass.

## Solution Assessment

Yes. A shared bootstrap-layer fingerprint resolver is the best fix because it covers Discord and other native approval/bootstrap callers consistently instead of adding a Discord-only workaround.
