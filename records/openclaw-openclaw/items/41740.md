## Summary

Keep this open: current main still routes Discord approval decisions through the shared approval Gateway resolver, and the shared operator-approvals client still constructs `GatewayClient` without a `tlsFingerprint`. The related fix work is in open, unmerged PRs, so the issue is valid but should not spawn a duplicate repair lane.

## What This Changes

Discord exec approvals should trust a local self-signed WSS Gateway by propagating the resolved local TLS fingerprint into the approval Gateway client.

## Best Possible Solution

Land a narrow shared bootstrap/operator-approvals TLS fingerprint fix that mirrors `callGateway` precedence and covers local WSS, remote WSS, CLI/env overrides, and remote fallback cases with focused regression tests.

## Reproduction Assessment

Yes, source-level: Discord approval decisions reach `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed WSS handling. I did not run a live Discord/Gateway repro in this read-only pass.

## Solution Assessment

Yes, the maintainable fix belongs in the shared operator-approvals/bootstrap TLS path rather than a Discord-only workaround. The open PRs are directionally useful, but the final landing should be narrow and cleanly reviewed.
