## Summary

Keep this open: current main still routes Discord approval decisions through the shared operator approvals Gateway client, and that client still constructs `GatewayClient` without `tlsFingerprint`. The related fix work exists in open PRs, but it has not landed and still needs a narrow, reviewed landing path.

## What This Changes

Discord exec approval resolution should pass the local self-signed Gateway TLS fingerprint to `GatewayClient` when using a local WSS Gateway.

## Best Possible Solution

Land a narrow shared bootstrap/operator-approvals TLS fingerprint fix that reuses the existing `callGateway` precedence rules and covers local WSS, remote WSS, CLI/env overrides, and remote fallback cases.

## Reproduction Assessment

Yes, source-level. Current Discord approval resolution reaches `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed WSS handling.

## Solution Assessment

Yes, the maintainable fix belongs in the shared operator approvals/bootstrap TLS path rather than a Discord-only workaround. The open PRs are directionally useful, but the final patch should be narrow and cleanly rebased.
