## Summary

Keep open: current main and v2026.5.12 still omit `tlsFingerprint` from the shared operator-approvals `GatewayClient` path, while active PRs already target the same fix, so this should remain tracked without a duplicate repair lane.

## What This Changes

Discord exec approval decisions should connect to a local self-signed WSS Gateway by carrying the resolved local TLS fingerprint into `GatewayClient`.

## Best Possible Solution

Land one shared bootstrap TLS fingerprint propagation fix that mirrors `callGateway` precedence and covers operator approvals with regression tests plus local WSS proof.

## Reproduction Assessment

Yes, source-level. Discord approval decisions reach `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed WSS handling.

## Solution Assessment

Yes for the shared bootstrap direction, not for a Discord-only special case. The maintainable fix is to resolve the selected TLS fingerprint once in the shared bootstrap path and pass it through direct `GatewayClient` call sites.
