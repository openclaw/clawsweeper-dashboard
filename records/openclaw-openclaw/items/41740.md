## Summary

Keep open: current main still routes Discord exec approval decisions through the shared operator-approvals Gateway client, and that client still constructs GatewayClient without a TLS fingerprint. The open shared fix at https://github.com/openclaw/openclaw/pull/80204 remains the right path, so this issue should close only after that PR or an equivalent shared bootstrap fix lands.

## What This Changes

The issue requests that Discord exec approvals pass the local Gateway TLS fingerprint when connecting to a self-signed local WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint resolver/pass-through fix, then close this issue with the merge commit and proof.

## Reproduction Assessment

Yes, by source inspection. Discord approvals reach the shared operator approvals Gateway client, that client omits tlsFingerprint, and GatewayClient only enables the pinned self-signed WSS path when tlsFingerprint is provided.

## Solution Assessment

Yes for the shared bootstrap direction, but not for a Discord-only patch. The narrow maintainable solution is to resolve the existing Gateway TLS fingerprint in shared bootstrap and pass it to all bootstrap-created GatewayClient instances.
