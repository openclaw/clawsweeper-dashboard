## Summary

Keep open. Current main still leaves the reported gap in the shared operator-approval Gateway client path: Discord approval buttons delegate to that client, and the client is still created without a TLS fingerprint even though GatewayClient only enables pinned self-signed TLS handling when one is supplied. The closed #41741 and #46232 are useful prior implementation context but were not merged, and #66279 is adjacent rather than a complete replacement for this Discord/native-approval path.

## What This Changes

Discord exec approval resolution should pass the local self-signed Gateway TLS fingerprint to GatewayClient when connecting to the local TLS-enabled wss:// Gateway without a URL override.

## Best Possible Solution

Resolve the local Gateway TLS runtime in the shared operator-approval Gateway bootstrap/client path under the same conditions used by callGateway, pass the resulting tlsFingerprint into GatewayClient, and add focused regression coverage for local self-signed wss:// plus CLI/env override and remote-config cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with local Gateway TLS enabled and no URL override, the Discord button path reaches withOperatorApprovalsGatewayClient, which creates GatewayClient without tlsFingerprint, and GatewayClient therefore does not enter its pinned self-signed TLS path.

## Solution Assessment

Yes. The best fix boundary is the shared operator-approval Gateway client/bootstrap path, not a Discord-only wrapper patch, because Discord and other native approval clients can call the same Gateway approval client.
