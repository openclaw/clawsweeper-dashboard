## Summary

Keep open: current main and the latest release still route Discord approval decisions through the shared operator-approvals Gateway client, and that client still constructs GatewayClient without a TLS fingerprint. The active canonical fix remains https://github.com/openclaw/openclaw/pull/80204, so this issue should close only after that shared bootstrap fix or an equivalent lands.

## What This Changes

The issue asks Discord exec/native approval Gateway clients to pass the local self-signed Gateway TLS fingerprint when connecting to a local TLS-enabled WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint resolver/pass-through fix, then close this issue with the merge commit and proof.

## Reproduction Assessment

Yes by source inspection. Discord approvals reach createOperatorApprovalsGatewayClient, that client omits tlsFingerprint, and GatewayClient only enables the pinned self-signed WSS path when tlsFingerprint is provided.

## Solution Assessment

Yes for the shared bootstrap direction, but not for a Discord-only patch. The narrow maintainable solution is to resolve the existing Gateway TLS fingerprint in shared bootstrap and pass it to all bootstrap-created GatewayClient instances.
