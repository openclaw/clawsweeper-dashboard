## Summary

Keep open: current main and the latest release still leave native approval bootstrap clients without a TLS fingerprint, and the current fix candidate is the still-open, conflicting https://github.com/openclaw/openclaw/pull/80204 rather than merged code.

## What This Changes

The issue reports Discord native exec approvals failing against a local self-signed TLS Gateway because the operator-approval GatewayClient path does not receive the local TLS fingerprint.

## Best Possible Solution

Refresh and land the shared gateway bootstrap TLS fingerprint fix so operator approvals, ACP, and MCP clients all receive the same scoped local/remote WSS pin data as callGateway.

## Reproduction Assessment

Yes, source-level: Discord native approvals route through the shared operator-approval GatewayClient path, which currently does not receive tlsFingerprint, while GatewayClient only enables self-signed WSS pin handling when that option is present. I did not run a live Discord/self-signed Gateway repro in this read-only review.

## Solution Assessment

Yes: the shared bootstrap resolver/pass-through in the open candidate PR is the best layer because operator approvals, ACP, and MCP share this connection bootstrap. A Discord-only fix would be too narrow and would leave sibling bootstrap clients with the same TLS gap.
