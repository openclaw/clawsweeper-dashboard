## Summary

Keep open. Current main no longer uses the old Discord-only direct GatewayClient construction, but Discord exec/native approvals now flow through the shared operator-approval Gateway client, and that shared client still constructs GatewayClient without tlsFingerprint; https://github.com/openclaw/openclaw/pull/62532 is the active open fix path but is conflicting and previously reviewed as needing changes.

## What This Changes

Discord exec approvals should connect to a local self-signed TLS Gateway by passing the resolved local Gateway TLS fingerprint into GatewayClient.

## Best Possible Solution

Land a corrected shared Gateway fix, through https://github.com/openclaw/openclaw/pull/62532 or a replacement, that propagates TLS fingerprints for operator approvals under the same local, remote, and override guards as callGateway with focused regression coverage.

## Reproduction Assessment

Yes, at source level. With local Gateway TLS enabled and no URL override, Discord approval flows reach GatewayClient through createOperatorApprovalsGatewayClient without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is present.

## Solution Assessment

No, not as originally scoped to Discord-only code. The maintainable fix now belongs in the shared operator-approval Gateway client or bootstrap path so Discord and other native approval clients get the same TLS behavior without duplicating policy.
