## Summary

Keep open: current main and the latest release still route Discord approval decisions through the shared operator-approvals Gateway client, and that client still constructs GatewayClient without a TLS fingerprint. A newer open fix, https://github.com/openclaw/openclaw/pull/80204, targets the shared bootstrap path and should be reviewed/landed before this issue closes.

## What This Changes

The issue reports that Discord exec approvals fail against a local self-signed TLS Gateway because the approval GatewayClient path does not receive the local TLS fingerprint.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS resolver/pass-through fix, then close this issue with the merge commit and proof.

## Reproduction Assessment

Yes. Source inspection shows Discord approvals reach createOperatorApprovalsGatewayClient, which does not pass tlsFingerprint, while GatewayClient requires that option for pinned self-signed WSS handling.

## Solution Assessment

Yes for the shared bootstrap direction, but not for the older Discord-only patch shape. The narrow maintainable fix is to resolve the existing Gateway TLS fingerprint once in the shared bootstrap path and pass it to all bootstrap-created GatewayClient instances.
