## Summary

Keep open: current main and the latest release still route Discord approval decisions through the shared operator-approval Gateway client without passing a TLS fingerprint, while https://github.com/openclaw/openclaw/pull/80204 is the current unmerged fix path.

## What This Changes

The issue requests that Discord native exec approvals connect to a local self-signed WSS Gateway with the same local TLS fingerprint handling used by other Gateway call paths.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint propagation fix, then close this issue after it reaches main.

## Reproduction Assessment

Yes, source-level. Discord approval decisions reach withOperatorApprovalsGatewayClient, which constructs GatewayClient without tlsFingerprint, while GatewayClient only uses the self-signed WSS fingerprint path when that option is present.

## Solution Assessment

The original Discord-only fix location is no longer the best layer because current Discord code uses the shared approval gateway resolver. The best fix is to extend the shared bootstrap/TLS fingerprint path so operator approvals and sibling bootstrap clients inherit the same guarded behavior as callGateway.
