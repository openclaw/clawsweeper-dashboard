## What This Changes

The issue requests that Discord/native exec approval clients pass the local Gateway TLS fingerprint into GatewayClient for self-signed local WSS Gateway connections.

## Best Possible Solution

Keep this issue open as the canonical bug report, refresh and review https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap fix, and close this issue only after the TLS fingerprint propagation lands with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main’s bootstrap/operator approval path constructs GatewayClient without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS trust when that option is present. I did not run a live Discord/local Gateway scenario because this review is read-only.

## Solution Assessment

Yes, the best fix is a shared bootstrap TLS fingerprint resolver that feeds all bootstrap clients, not a Discord-only copy of the callGateway logic. The open related PR is the right shape but needs a current-main refresh before it can land.
