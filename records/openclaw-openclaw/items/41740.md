## Summary

Keep open. Current main still routes Discord approval buttons through the shared operator-approval Gateway client, and that client is constructed without the TLS fingerprint needed for pinned local self-signed `wss://` connections.

## What This Changes

Discord exec approvals should connect to a local self-signed TLS Gateway by propagating the resolved local Gateway TLS fingerprint into the approval Gateway client path.

## Best Possible Solution

Resolve the local Gateway TLS runtime in the shared operator-approval Gateway bootstrap/client path under the same guards used by `callGateway`, pass that fingerprint to `GatewayClient`, and add focused regression coverage for local `wss://`, CLI/env override, and remote-config cases.

## Reproduction Assessment

Yes, at source level. With local Gateway TLS enabled and no URL override, the Discord button path reaches `GatewayClient` without `tlsFingerprint`, so Node's default TLS handling can reject the local self-signed certificate.

## Solution Assessment

Yes. The shared operator-approval Gateway client/bootstrap path is the narrowest maintainable fix boundary because Discord buttons and other native approval runtimes use the same client construction path.
