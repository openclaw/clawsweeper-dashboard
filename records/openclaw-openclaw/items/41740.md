- `impact:security`: The affected behavior is TLS certificate fingerprint handling for local self-signed Gateway connections.

## Summary

Keep open: current main and the latest release still route Discord exec approval decisions through the shared operator-approvals Gateway client, and that shared bootstrap path still cannot carry a TLS fingerprint. The open shared fix at https://github.com/openclaw/openclaw/pull/80204 remains the right implementation path.

## What This Changes

Discord exec approvals should reuse local Gateway TLS fingerprint resolution when connecting to a local self-signed WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint resolver/pass-through fix, then close this issue with the merge commit and release/proof.

## Reproduction Assessment

Yes. By source inspection, Discord approvals reach the shared operator approvals Gateway client, that client omits tlsFingerprint, and GatewayClient only enables the pinned self-signed WSS path when tlsFingerprint is provided.

## Solution Assessment

Yes for the shared bootstrap direction, but not for a Discord-only patch against the old direct client path. The narrow maintainable solution is to resolve the existing Gateway TLS fingerprint in shared bootstrap and pass it to all bootstrap-created GatewayClient instances.
