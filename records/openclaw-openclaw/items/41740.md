## What This Changes

The issue requests that Discord exec approval GatewayClient construction inherit the local self-signed Gateway TLS fingerprint.

## Best Possible Solution

Refresh and land the shared bootstrap TLS fingerprint fix in https://github.com/openclaw/openclaw/pull/80204, preserving local/remote/CLI/env target-selection guards and regression coverage for operator approvals plus sibling bootstrap clients.

## Reproduction Assessment

Yes, source-level: current main sends operator approvals, ACP, and MCP bootstrap clients into GatewayClient without tlsFingerprint, while GatewayClient only enables self-signed WSS fingerprint checking when that option is present. I did not run a live Discord/local Gateway repro in this read-only review.

## Solution Assessment

Yes for the active shared-fix direction, but no for the original Discord-only scoped fix. The best layer is the shared bootstrap/TLS resolver because current Discord approval decisions and sibling ACP/MCP clients share the same missing GatewayClient bootstrap contract.
