## Summary

## What This Changes

The issue asks for Discord exec approval handling to trust the existing local Gateway TLS fingerprint when connecting to a self-signed local WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint propagation fix, then close this issue with merge proof against a local self-signed WSS Gateway.

## Reproduction Assessment

Yes. Source inspection shows the Discord approval path reaches a shared bootstrap-created GatewayClient that cannot receive tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is supplied.

## Solution Assessment

Yes for the shared bootstrap direction. A Discord-only patch would leave ACP/MCP/native approval bootstrap clients with the same missing TLS fingerprint path.
