## Summary

Keep open. Current main still routes Discord approval button decisions through the shared operator-approval Gateway client, and that client is constructed without the local TLS fingerprint required for pinned self-signed `wss://` Gateway connections; the two related fix PRs are closed unmerged, and #66279 covers an adjacent TUI surface rather than replacing this report.

## What This Changes

The issue requests that Discord exec approvals pass the local Gateway TLS fingerprint when connecting to a self-signed local `wss://` Gateway.

## Best Possible Solution

Resolve the local Gateway TLS runtime in the shared operator-approval Gateway bootstrap/client path under the same guards used by `callGateway`, pass that fingerprint to `GatewayClient`, and add focused regression coverage for local `wss://`, CLI/env override, and remote-config cases.

## Reproduction Assessment

Yes, at source level. With local Gateway TLS enabled and no URL override, the Discord button path reaches `GatewayClient` without `tlsFingerprint`, while `GatewayClient` only enables pinned self-signed TLS handling when that option is present.

## Solution Assessment

Yes, with a small boundary adjustment. The issue’s requested behavior is correct, but the narrow maintainable fix belongs in the shared operator-approval Gateway bootstrap/client path rather than only in the Discord button file.
