## Summary

Keep open: current main still omits the local TLS fingerprint from the shared gateway bootstrap clients, and the open related PR is a candidate fix but has not merged.

## What This Changes

The issue requests that Discord/native exec approvals pass the local gateway TLS fingerprint when connecting to a self-signed local WSS gateway.

## Best Possible Solution

Keep this issue open as the source report and finish the open shared bootstrap TLS fingerprint PR as the canonical fix.

## Reproduction Assessment

Yes, at source level. Current main sends the operator approval bootstrap path into `GatewayClient` without a `tlsFingerprint`, while the sibling `callGateway` path resolves and forwards one for local self-signed WSS targets.

## Solution Assessment

Yes, the shared bootstrap resolver is the best layer because it fixes native approval, ACP, and MCP clients consistently instead of copying TLS selection into one Discord-only caller. The best next step is to finish the open related PR, not create another overlapping fix.
