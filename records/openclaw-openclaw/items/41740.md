## Summary

Keep this issue open: current main and the latest release still route native/bootstrap Gateway clients into GatewayClient without a TLS fingerprint, while an open broader PR is the active repair path but has not merged.

## What This Changes

The issue reports Discord exec approvals failing against a local self-signed TLS Gateway because the approval GatewayClient path does not receive the local TLS fingerprint.

## Best Possible Solution

Land a refreshed shared bootstrap TLS fingerprint fix through https://github.com/openclaw/openclaw/pull/80204 or an equivalent narrow replacement, then close this issue with release provenance.

## Reproduction Assessment

Yes, source-level: current main sends operator-approval, ACP, and MCP bootstrap clients into GatewayClient without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is present. I did not run a live Discord/self-signed Gateway repro in this read-only review.

## Solution Assessment

No, not as a Discord-only one-off patch: the better fix is shared bootstrap TLS fingerprint propagation so operator approvals, ACP, and MCP follow the same target-selection rules. The open related PR is the right repair direction but needs a current-main refresh before merge.
