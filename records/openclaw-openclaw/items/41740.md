## Summary

Keep open. Current main still constructs the shared operator-approval Gateway client without a TLS fingerprint, so Discord approval-button decisions can still fail against local self-signed WSS Gateways; https://github.com/openclaw/openclaw/pull/62532 is the active fix path but is open and previously reviewed as needing changes.

## What This Changes

Discord exec approval decisions should reuse local Gateway TLS fingerprint pinning when resolving approvals over a self-signed local WSS Gateway.

## Best Possible Solution

Land a corrected shared Gateway fix that propagates TLS fingerprints for operator approvals under the same local, remote, and override guards as callGateway, with focused regression coverage.

## Reproduction Assessment

Yes, at source level. With local Gateway TLS enabled and no URL override, Discord button decisions reach GatewayClient through the shared operator-approval client without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is present.

## Solution Assessment

Yes, with the fix boundary adjusted to the shared client path. The maintainable solution is to extend the shared operator-approval bootstrap/client TLS selection rather than adding Discord-only logic, and the active PR should be corrected or replaced there.
