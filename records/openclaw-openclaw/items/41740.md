## Summary

Keep open: current main and v2026.5.20 still omit TLS fingerprint propagation from shared bootstrap-created Gateway clients used by Discord/native approvals. The open shared fix at https://github.com/openclaw/openclaw/pull/80204 remains the canonical implementation path, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for Discord/native approval Gateway clients to reuse local self-signed TLS fingerprint handling when connecting to a local WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint propagation fix, then close this issue with merge proof for native approval clients against a local self-signed WSS Gateway.

## Reproduction Assessment

Yes. Source inspection shows the Discord/native approval path reaches shared bootstrap clients that cannot receive tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is supplied.

## Solution Assessment

Yes for the shared bootstrap direction, but not for another Discord-only patch against the older direct client path. The narrow maintainable solution is to resolve the existing Gateway TLS fingerprint in shared bootstrap and pass it to all bootstrap-created GatewayClient instances.
