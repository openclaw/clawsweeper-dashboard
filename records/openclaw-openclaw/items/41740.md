## Summary

Keep this open: current main still routes Discord approval decisions through the shared approval Gateway resolver, and the shared operator-approvals client still builds `GatewayClient` without a TLS fingerprint. The remaining fix is already represented by an open PR, so this should not spawn a duplicate repair lane.

## What This Changes

Discord exec approvals should pass the local Gateway TLS fingerprint when connecting to a local self-signed `wss://` Gateway.

## Best Possible Solution

Land a narrow shared operator-approvals/bootstrap TLS fingerprint fix that mirrors `callGateway` precedence and includes focused regression coverage plus real runtime proof.

## Reproduction Assessment

Yes, source-level: Discord approval decisions reach `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed `wss://` handling. I did not run a live Discord/Gateway repro in this read-only pass.

## Solution Assessment

No for the original Discord-only shape; current main has moved the affected construction into the shared operator-approvals path. The maintainable fix is to propagate TLS fingerprint data through shared bootstrap/client code using the same precedence as `callGateway`.
