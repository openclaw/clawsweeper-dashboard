## Summary

Keep open: current main and the latest release still leave shared Gateway bootstrap clients without a TLS fingerprint, while https://github.com/openclaw/openclaw/pull/80204 is the active but unmerged repair candidate.

## What This Changes

Discord/native approval Gateway clients should trust the configured local self-signed WSS Gateway by passing the resolved local TLS fingerprint into GatewayClient.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/80204 as the shared Gateway bootstrap TLS fingerprint fix, with focused coverage for local WSS, remote pins, URL overrides, and non-TLS targets.

## Reproduction Assessment

Yes, at source level: current main sends operator approval, ACP, and MCP bootstrap clients into GatewayClient without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS trust when that option is present. I did not run a live Discord flow in this read-only review.

## Solution Assessment

Yes, the best fix direction is shared bootstrap TLS fingerprint resolution and pass-through rather than restoring a Discord-only copy of the logic. The open related PR is already attempting that canonical fix and should be refreshed or reviewed rather than duplicated.
