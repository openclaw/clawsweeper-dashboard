Keep this issue open: current main and the latest release still route native/operator approval Gateway clients through bootstrap data that lacks a TLS fingerprint, while a broader open PR now appears to own the fix path but has not merged.

## What This Changes

The issue requests that Discord exec approvals trust the local self-signed Gateway TLS certificate by passing the resolved Gateway TLS fingerprint into GatewayClient.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap fix that resolves local/remote/override/non-WSS TLS fingerprint selection once and forwards it to operator approval, ACP, and MCP GatewayClient callers.

## Reproduction Assessment

Yes, source-level: current main sends operator approval bootstrap clients into GatewayClient without tlsFingerprint, while GatewayClient only enables pinned self-signed WSS handling when that option is present. I did not run a live Discord/Gateway repro in this read-only review.

## Solution Assessment

No, not exactly as the original Discord-only patch shape. The better current fix is the shared bootstrap resolver/pass-through path because the same invariant affects native approvals and other bootstrap clients.
