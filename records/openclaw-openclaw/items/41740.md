## Summary

Keep open: current main and the latest release still route Discord approval decisions through the shared operator-approvals Gateway client, and that client still constructs `GatewayClient` without a TLS fingerprint. The remaining fix is represented by an open PR, so this should stay open without creating a duplicate repair lane.

## What This Changes

Discord exec approval decisions should pass the resolved local Gateway TLS fingerprint when connecting to a local self-signed `wss://` Gateway.

## Best Possible Solution

Review and land a narrow shared operator-approvals/bootstrap TLS fingerprint fix that mirrors `callGateway` precedence, with focused regression coverage and real local WSS approval proof.

## Reproduction Assessment

Yes, source-level. Discord approval decisions reach `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed WSS handling.

## Solution Assessment

Yes for the current shared-path direction, but not for the older Discord-only patch shape. The maintainable fix belongs in the shared operator-approvals/bootstrap TLS path and should mirror `callGateway` precedence.
