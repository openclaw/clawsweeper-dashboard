## Summary

Keep open: current main and the latest release still omit TLS fingerprint propagation from the shared bootstrap/operator-approval client path, and the related fix PR is open but not merged.

## What This Changes

The issue reports Discord/native exec approvals failing against a local self-signed TLS Gateway because the approval GatewayClient is created without the local TLS fingerprint.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint fix, then close this issue after merge.

## Reproduction Assessment

Yes, source-level: current main sends operator-approval bootstrap clients into GatewayClient without tlsFingerprint, while GatewayClient only enables self-signed WSS fingerprint checking when that option is present. I did not run a live Discord/local TLS repro in this read-only review.

## Solution Assessment

Yes: a shared bootstrap TLS fingerprint resolver is the best layer because it fixes operator approvals and sibling ACP/MCP bootstrap clients consistently instead of copying TLS selection into only the Discord path. The open implementation still needs refresh/review before it can close this issue.
