## Summary

Keep open: current main and the latest release still route Discord/native approval resolution through shared Gateway bootstrap clients that do not carry a TLS fingerprint, while an open conflicting PR is the active candidate fix.

## What This Changes

The issue reports Discord exec approval failures against a local self-signed WSS Gateway because the approval/bootstrap GatewayClient path does not forward the local TLS fingerprint.

## Best Possible Solution

Land a refreshed shared Gateway bootstrap TLS fingerprint propagation fix, preferably through https://github.com/openclaw/openclaw/pull/80204 or an equivalent branch, then close this issue from merged commit proof.

## Reproduction Assessment

Yes at source level: local TLS selects a wss:// loopback target, normal callGateway can resolve the local TLS fingerprint, but approval/bootstrap clients construct GatewayClient without that option. I did not run a live Discord approval flow in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix is shared bootstrap TLS fingerprint propagation with local, remote, override, fallback, and non-WSS tests. A Discord-only patch is now too narrow because Discord uses the shared operator approval Gateway path.
