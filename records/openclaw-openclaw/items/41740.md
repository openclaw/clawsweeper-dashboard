## Summary

Keep this issue open: current main centralizes native approval connections in the shared operator-approvals client, but that client still constructs `GatewayClient` without the TLS fingerprint required to trust a local self-signed gateway; all linked fix PRs closed without merging.

## What This Changes

Fix native Discord exec approvals so they connect to a local self-signed TLS gateway using the canonical certificate fingerprint.

## Best Possible Solution

Resolve the local TLS fingerprint in the shared gateway bootstrap or operator-approvals client, apply it only to canonical local `wss://` targets, and add coverage proving both local self-signed success and unchanged remote/override behavior.

## Reproduction Assessment

Yes at source level: current main omits the certificate pin that `GatewayClient` needs for an otherwise untrusted local self-signed certificate, matching the reporter's configured reproduction. This review did not run a live gateway instance.

## Solution Assessment

No, the original Discord-only patch is no longer the best boundary because native approval connectivity is now centralized. The shared operator-approvals path should own fingerprint resolution with explicit local-versus-remote provenance checks.
