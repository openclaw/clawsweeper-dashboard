## Summary

Keeping this open: current main still routes Discord approval decisions through the shared operator approvals client, and that client constructs `GatewayClient` without `tlsFingerprint` even though the client only enables pinned self-signed WSS handling when that option is supplied. An open fix PR already targets the shared path, so this should stay open until that fix lands or is replaced.

## What This Changes

Discord exec approvals should connect to a local self-signed TLS Gateway by passing the resolved local TLS fingerprint into the Gateway client path.

## Best Possible Solution

Land or replace the shared operator approvals TLS fingerprint fix in https://github.com/openclaw/openclaw/pull/62532, with guards for local loopback versus remote URL and override cases plus focused Gateway tests.

## Reproduction Assessment

Yes, at source level. The current Discord resolver reaches `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option to accept a pinned local self-signed WSS certificate.

## Solution Assessment

Yes, the maintainable fix belongs in the shared operator approvals Gateway client or its bootstrap helper, not in a Discord-only path. The open PR is the right direction once its fallback handling is corrected and validated.
