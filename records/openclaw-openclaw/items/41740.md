- `P2`: This is a normal-priority local Gateway/native approval bug with a clear source-level repro and limited blast radius.
- `impact:security`: The affected behavior is TLS certificate fingerprint handling for local self-signed Gateway connections.

## Summary

Keep open: current main and the latest release still route Discord approval decisions through shared Gateway bootstrap clients that cannot carry a TLS fingerprint. The open shared fix at https://github.com/openclaw/openclaw/pull/80204 remains the right implementation path, but it has not merged yet.

## What This Changes

Discord exec approvals should pass the configured local Gateway TLS fingerprint to `GatewayClient` when connecting to a local self-signed WSS Gateway.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80204 or an equivalent shared bootstrap TLS fingerprint propagation fix, then close this issue with merge proof for native approval clients against a local self-signed WSS Gateway.

## Reproduction Assessment

Yes. Source inspection shows the Discord/native approval path reaches the shared operator approvals Gateway client, that client cannot receive a `tlsFingerprint` from bootstrap, and `GatewayClient` only enables pinned self-signed WSS handling when `tlsFingerprint` is provided.

## Solution Assessment

Yes for the shared bootstrap direction, but not for another Discord-only patch against the older direct client path. The narrow maintainable solution is to resolve the existing Gateway TLS fingerprint in shared bootstrap and pass it to all bootstrap-created `GatewayClient` instances.
