## Summary

Keep this PR open, but do not merge it as-is: the new `readPageState` contract is internally inconsistent for `selector`, and the LinkedIn-specific state mode needs maintainer API-boundary approval or removal before it becomes a browser plugin contract.

## What This Changes

The branch adds a browser `readPageState` act action with bounded page text reads, a `linkedin_invitations` state classifier/card extractor, and a wait-path switch from `innerText` to `textContent`.

## Best Possible Solution

Keep the browser plugin surface generic and internally consistent, then let the LinkedIn-owned skill/plugin parse invitation semantics from a generic bounded page-state read unless maintainers explicitly approve a service-specific browser contract.

## Reproduction Assessment

Yes for the patch defect: source inspection reproduces the route mismatch because `readPageState` accepts `selector` but the top-level `/act` selector guard excludes that kind. The original LinkedIn cron failure is supported by PR-body live-output claims, not independently rerun here.

## Solution Assessment

No; a generic bounded page-state read is a plausible fix, but adding LinkedIn invitation classification to the generic browser plugin is broader than the owner boundary requires. The safer path is a generic read contract plus LinkedIn parsing in the owning skill/plugin, or explicit maintainer approval for the service-specific API.
