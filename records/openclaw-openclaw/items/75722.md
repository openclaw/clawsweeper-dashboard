## Summary

Keep open with changes requested: the PR is the right privacy-hardening direction, but the current head still drops archived transcript owner metadata and diverges from the existing memory_search visibility contract.

## What This Changes

The PR threads caller session context into memory-wiki wiki_search/wiki_get, filters shared-memory session hits, guards direct shared-memory reads, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Reuse the SDK transcript identity contract in memory-wiki so wiki_search/wiki_get and memory_search authorize live and archived transcript paths the same way.

## Reproduction Assessment

Yes. Current main is source-reproducible for the original bug because wiki_search maps shared-memory session hits directly and wiki_get reads shared-memory candidates without a visibility guard; the PR regression is source-reproducible with a sessions/<agent>/<stem>.jsonl.deleted.* path and an empty live store.

## Solution Assessment

No, not as written. The memory-wiki-side fix is the narrow maintainable place, but it should use extractTranscriptIdentityFromSessionsMemoryHit and pass archivedOwnerAgentId for archived paths before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve archived transcript owner metadata:** `extensions/memory-wiki/src/query.ts:1253-1260`
  - body: The new checker reduces archived paths such as sessions/main/deleted-stem.jsonl.deleted.* to only a stem before resolving session keys.
