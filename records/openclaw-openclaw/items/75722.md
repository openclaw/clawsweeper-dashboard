## Summary

Keep open: the PR targets a real current-main session visibility gap, but the latest diff still drops archived transcript owner metadata, lacks after-fix real behavior proof, and carries protected maintainer/security-sensitive context.

## What This Changes

The branch threads agent session and sandbox context into memory-wiki `wiki_search`/`wiki_get`, filters shared-memory session hits, guards direct shared-memory reads, adds regression tests, and adds a changelog security entry.

## Best Possible Solution

Keep the fix in memory-wiki, align it with memory-core's owner-aware transcript visibility contract, add archived `wiki_search`/`wiki_get` regression coverage, and merge only after maintainer security review plus real behavior proof.

## Reproduction Assessment

Yes, as a source-level reproduction against current main: `searchMemoryWiki` maps shared session hits directly and `getMemoryWikiPage` reads shared-memory candidates without a visibility guard. I did not run a live scenario because this review was constrained to read-only inspection.

## Solution Assessment

No, not as written. The memory-wiki boundary is right, but the PR should use `extractTranscriptIdentityFromSessionsMemoryHit` and pass `archivedOwnerAgentId` for archived paths before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve archived transcript owner metadata:** `extensions/memory-wiki/src/query.ts:1250-1257`
  - body: The checker reduces archived paths such as `sessions/main/deleted-stem.jsonl.deleted.*` to only a stem before resolving session keys.
