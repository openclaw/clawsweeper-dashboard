Keep open for maintainer review. The PR addresses a real current-main privacy gap in memory-wiki and I did not find a discrete blocking defect in the patch, but the protected maintainer label plus GHSA/session-visibility surface require explicit human approval before merge.

## What This Changes

The branch threads sandbox/session context into memory-wiki `wiki_search` and `wiki_get`, filters shared session-memory hits and direct reads by session visibility, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep the enforcement inside the memory-wiki plugin, rely on the existing SDK session-visibility helpers, and land the PR after maintainer approval and green validation if the gateway/no-session behavior is intentional.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: call memory-wiki shared-memory search or direct get with a caller session while the shared memory backend can return another session's transcript path; the current code maps or reads those results without the SDK visibility guard.

## Solution Assessment

Yes. The PR appears to be the narrow maintainable fix because it threads trusted tool context into the plugin and applies the same session visibility/transcript-hit contracts already used by `memory_search`, instead of adding a core special case.
