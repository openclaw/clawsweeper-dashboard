## Summary

Keep open: the bug is real and the PR moves in the right direction, but the patch is too narrow because it adds a reset-only timestamp parser while current main already has a broader reset/deleted session archive timestamp contract.

## What This Changes

The PR parses `.jsonl.reset.<timestamp>Z` paths inside memory-core temporal decay and adds regression tests for reset archive decay and ranking.

## Best Possible Solution

Use one canonical session archive timestamp contract for temporal decay, exposed through the memory-host SDK boundary, so reset and deleted archives decay consistently without extension deep imports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: session archive hits are indexed as logical `sessions/<agent>/<basename>` paths, while current temporal decay resolves non-memory timestamps under `workspaceDir` and skips decay when that stat fails.

## Solution Assessment

No. Parsing before `fs.stat` is the right fix shape, but this PR should reuse the canonical reset/deleted archive timestamp contract instead of adding a duplicate reset-only parser.
