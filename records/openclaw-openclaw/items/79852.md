## Summary

Keep open: the reported temporal-decay bug is valid and the PR has a focused fix direction, but the current patch duplicates the archive timestamp contract with a reset-only parser that leaves other accepted archive paths inconsistent.

## What This Changes

The PR adds filename-based timestamp parsing for session-reset archives in memory-core temporal decay and adds regression tests for reset archive decay and ranking.

## Best Possible Solution

Expose and reuse the canonical session archive timestamp parser through the memory-host SDK facade, then decay reset and deleted archive paths consistently before the fs.stat fallback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: indexed session archive paths are logical sessions/<agent>/... paths, while current temporal decay resolves non-memory timestamps against workspaceDir and skips decay when fs.stat fails. I did not run a live archive search in this read-only review.

## Solution Assessment

No, not yet: parsing archive timestamps before fs.stat is the right fix shape, but the PR should reuse the existing reset/deleted archive timestamp contract instead of adding a local reset-only regex.
