## Label Justifications

- `P2`: This is a normal-priority web provider credential detection fix with limited but real web-tool blast radius.
- `merge-risk: 🚨 compatibility`: Existing configs that previously treated unresolved env-ref strings as truthy credentials can change provider selection after upgrade.
- `merge-risk: 🚨 auth-provider`: The changed helper participates in web provider credential availability and provider-auth fallback decisions for web_search and web_fetch.

## Summary

Keep open for maintainer review: the patch is a narrow, source-backed fix with sufficient live output, but it intentionally changes web provider availability when env-style refs are unresolved, so maintainers need to accept the compatibility and auth-provider semantics before merge.

## What This Changes

The PR changes shared web provider credential detection so primary and fallback env-style SecretRefs only count as credentials after env resolution succeeds, while literals and non-env SecretRefs keep their existing behavior.

## Best Possible Solution

Land the shared-helper fix after maintainers accept the stricter env-ref credential semantics; keep this rule centralized instead of duplicating provider-specific checks.

## Reproduction Assessment

Yes. Source inspection on current main shows an env shorthand can parse as a SecretRef and then still be counted as a nonempty literal credential before env resolution; I did not run a local command in this read-only review.

## Solution Assessment

Yes. The shared helper is the best fix boundary because both web_search and web_fetch call it; caller-level fixes would duplicate provider credential semantics and risk drift.
