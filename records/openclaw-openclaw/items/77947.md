## Summary

Keep open: the PR is active and has sufficient live proof for the main Telegram single-message path, but PR head still swallows failed inline-button edits after marking the stream delivered. The protected `maintainer` label and the documented fresh-final contract change also require maintainer acceptance before merge.

## What This Changes

The PR rewrites Telegram text streaming so final text stops and reuses the active stream message, removes archived preview/fresh-final delivery paths, updates Telegram QA/tests/docs, and adds a changelog fix entry.

## Best Possible Solution

Restore fallback for failed required button edits while preserving the accepted single-stream final behavior, then have the Telegram owner decide whether to retire the documented fresh-final paths.

## Reproduction Assessment

No high-confidence live before-fix reproduction was run in this read-only review. The blocking regression is source-reproducible at PR head because failed `editStreamMessage` calls are caught after delivery is marked, and the new test asserts no fallback send.

## Solution Assessment

No. The single-stream direction is reasonable if accepted, but the current PR head is not the best fix until failed inline-button edits either fall back to normal final delivery or leave the stream unfinalized instead of reporting success.
