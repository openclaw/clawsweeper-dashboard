## Summary

Keep open: the PR still needs real Telegram proof and the current patch can mark a stream final as delivered even when the final edit failed, while also removing documented fresh-final behavior without updating the docs.

## What This Changes

The PR rewrites Telegram text streaming so final replies stop/update the active stream message, removes archived preview and fresh-final fallback handling, updates Telegram and QA coverage, and adds a changelog entry.

## Best Possible Solution

Land a version that preserves failed-edit fallback, aligns the docs with the intended Telegram semantics, and includes real Telegram proof that a streamed final produces one visible SUT message without duplicates.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection is enough to review the PR regression paths, but the PR discussion does not show a real Telegram before/after run for the duplicate-response symptom.

## Solution Assessment

No as submitted. Finalizing in the stream message may be the right narrow direction for duplicate suppression, but the patch needs failed-edit fallback plus docs/product alignment before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restore fallback when the final stream edit fails:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:124-140`
  - body: After `stopDraftLane` runs, this path treats any remaining stream message id as delivered. `TelegramDraftStream.stop()` flushes through code that catches edit/send failures and returns `false` while leaving an existing message id, so a failed final edit can be reported as `stream-finalized` and suppress the normal final send even though Telegram still shows the stale preview. Please only mark delivered after a known landed final update, or fall back to `sendPayload` when the final stream update fails.
  - confidence: 0.91
- **[P2] Align the Telegram streaming docs:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:149-157`
  - body: This final path now treats the stream message as delivered and only sends follow-up chunks, but the Telegram and Streaming docs still promise fresh final messages after visible intermediate output or old previews. Please update those docs to the new single-stream contract, or keep the old fallback where that UX is still intended.
  - confidence: 0.87
