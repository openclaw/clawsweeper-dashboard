## Summary

Keep open: the PR still has blocking Telegram inline-button delivery regressions at head. The live proof covers the successful single-stream paths, but source review shows failed button edits are reported as delivered and long buttoned finals can emit keyboards before the answer is complete.

## What This Changes

The PR rewrites Telegram text streaming to finalize on the active stream message, removes archived preview/fresh-final paths, updates Telegram QA coverage, docs, and the changelog.

## Best Possible Solution

Preserve the accepted single-stream final behavior while making button-required final delivery fall back or attach keyboards exactly once, then have the Telegram owner explicitly accept the fresh-final contract change.

## Reproduction Assessment

Yes for the PR-introduced blockers from source: PR head catches failed button edits after marking delivery and preserves button data on long follow-up chunks. The original duplicate-final happy path has after-fix live proof, but I did not run a live before-fix reproduction in this read-only review.

## Solution Assessment

No. The single-stream direction may be acceptable if an owner approves it, but the current patch reports buttoned finals delivered after failed button edits and can duplicate or misplace inline keyboards on long final chunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fall back when button edits fail:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:170-176`
  - body: After the final text is stopped into the stream, inline buttons are applied with a second edit. If that edit fails, this catch only logs after delivery was already marked, so Telegram can show the final answer without its required buttons; current main falls back to normal final delivery for comparable final-edit failures. Treat failed button edits as fallback instead of returning `preview-finalized`.
  - confidence: 0.9
- **[P2] Keep buttons off intermediate long-final chunks:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:181-185`
  - body: When a final response is split and carries Telegram inline buttons, the stream message gets buttons and each remaining chunk is sent with `followUpPayload(payload, chunk)`, which preserves `channelData.telegram.buttons`. Current main deliberately avoids preview reuse for buttoned long finals so normal sending keeps the keyboard on the last chunk only; fall back for buttoned long finals or attach buttons only once at the end.
  - confidence: 0.84
