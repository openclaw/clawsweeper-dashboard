## Summary

Keep open: the PR has sufficient live proof for the duplicate-final happy path, but current head still introduces inline-button delivery regressions that should block merge.

## What This Changes

The PR rewrites Telegram text streaming to finalize on the active stream message, updates progress-mode final delivery, replaces related tests, adds Telegram QA coverage, and updates docs/changelog.

## Best Possible Solution

Keep the single-stream final direction only if the Telegram owner accepts the contract change, while preserving reliable inline-button delivery and existing chunk keyboard placement.

## Reproduction Assessment

Yes for the PR-introduced blockers from source: failed button attachment returns preview-finalized, and long follow-up chunks reuse the buttoned payload. The duplicate-final improvement has after-fix live QA proof, but I did not run a live before/after reproduction in this read-only review.

## Solution Assessment

No. The single-stream approach may be acceptable after owner approval, but the current implementation should fall back or attach inline keyboards exactly once instead of swallowing button-edit failures and preserving buttons on follow-up chunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fall back when button edits fail:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:170-176`
  - body: After the final text is stopped into the stream, inline buttons are applied with a second edit. If that edit fails, this catch only logs after delivery was already marked, so Telegram can show the final answer without its required buttons; current main falls back to normal final delivery for comparable final-edit failures. Treat failed button edits as fallback instead of returning `preview-finalized`.
  - confidence: 0.9
- **[P2] Keep buttons off intermediate long-final chunks:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:181-185`
  - body: When a final response is split and carries Telegram inline buttons, the stream message gets buttons and each remaining chunk is sent with `followUpPayload(payload, chunk)`, which preserves `channelData.telegram.buttons`. Current main avoids preview reuse for buttoned long finals so normal sending keeps the keyboard on the last chunk only; fall back for buttoned long finals or attach buttons only once at the end.
  - confidence: 0.84
