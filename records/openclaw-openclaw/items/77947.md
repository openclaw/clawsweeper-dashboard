## Summary

This PR has already landed on current main, so the cleanup outcome is implemented-on-main. The landed patch still has two inline-button regressions that should be handled as follow-up work rather than keeping this now-merged PR open.

## What This Changes

The PR changes Telegram text streaming so final replies stop and reuse the active stream message, updates progress-mode final delivery, replaces related Telegram tests, adds live QA coverage, and updates streaming docs/changelog.

## Best Possible Solution

Keep the landed single-stream Telegram final contract on main, and track a narrow follow-up repair for inline-button fallback and one-time keyboard placement if maintainers want to preserve that behavior.

## Reproduction Assessment

Yes for the review findings: source inspection shows failed button edits are logged after delivery is marked, and long follow-up chunks preserve the original button payload. The duplicate-final improvement itself has after-fix live Telegram QA proof in the PR body/comments.

## Solution Assessment

No for the landed patch as-is. Main now implements the desired single-stream final behavior, but the inline-button handling should fall back or attach keyboards exactly once instead of swallowing button-edit failures and repeating buttons on follow-up chunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fall back when button edits fail:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:170-176`
  - body: After the final stream text is marked delivered, inline buttons are applied with a second edit. If that edit fails, the catch only logs and still reports `preview-finalized`, so Telegram can show the final answer without the required buttons; fall back to normal delivery or avoid marking the stream delivered until the buttoned edit succeeds.
  - confidence: 0.9
- **[P2] Attach buttons only once for long streamed finals:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:185`
  - body: When a long final response carries Telegram inline buttons, the stream message gets buttons and each remaining chunk is sent through `followUpPayload(payload, chunk)`, which preserves `channelData.telegram.buttons`. Because each follow-up is delivered as its own reply, the keyboard can be attached repeatedly; strip buttons from follow-ups or choose one attachment point.
  - confidence: 0.84
