## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow; existing surfaces cover immediate default-model writes, fallback list edits, OpenRouter scanning, and auth-profile probes, but not stage, verify, promote, and rollback behavior.

## What This Changes

Request a staged model migration UX that can add a candidate as fallback, verify it, promote it to primary, and roll back to the old model if needed.

## Best Possible Solution

Define and implement a maintainer-designed openclaw models migration flow that stages any provider/model candidate, verifies the intended agent surfaces, preserves the old primary in the fallback chain, persists rollback state, and documents/tests config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, live issue discussion, and related-item search give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent on current main.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than treating the narrower fallback-test issue as the whole solution.
