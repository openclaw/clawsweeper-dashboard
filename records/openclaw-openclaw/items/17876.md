## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the live discussion already contains maintainer guidance to keep this as product work rather than close it as stale, duplicate, or implemented.

## What This Changes

Request a safer `openclaw models` migration flow that stages a new model, verifies it before promotion, preserves fallback/rollback state, and can restore the prior primary model.

## Best Possible Solution

Define a maintainer-approved `openclaw models` migration workflow that stages an arbitrary model candidate, verifies the intended agent surfaces, promotes it only after success, preserves the old primary in the fallback chain, persists rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, related-issue, and live discussion inspection give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent on current main.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than an OpenRouter-only scan extension or fallback-test-only tweak.
