## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the discussion already includes maintainer guidance that this needs product follow-up rather than cleanup closure.

## What This Changes

The issue requests a staged model migration UX that can test a new model before promotion, preserve the old primary as fallback, and roll back if needed.

## Best Possible Solution

Define and implement a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies intended agent surfaces before promotion, preserves the old primary in the fallback chain, persists rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Current-main source and docs inspection gives a high-confidence verification path: the requested test/migrate/promote/rollback workflow is absent.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built on existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than an OpenRouter-only scan extension or rollback-only tweak.
