## Summary

Keep open. Current main still lacks the staged, provider-generic model migration workflow requested here; existing code covers immediate default-model writes, fallback list edits, OpenRouter free-model scanning, and auth probes, but not stage/test/promote/rollback behavior.

## What This Changes

The issue requests a staged `openclaw models` migration workflow that adds a candidate as fallback, verifies it, promotes it to primary, and supports rollback.

## Best Possible Solution

Define and implement a maintainer-designed `openclaw models` migration flow that stages any provider/model candidate, verifies the intended agent surfaces, preserves the old primary in the fallback chain, persists rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, related-issue, and live discussion inspection give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent on current main.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than an OpenRouter-only scan extension or fallback-test-only tweak.
