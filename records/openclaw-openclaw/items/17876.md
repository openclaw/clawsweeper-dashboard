## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow: the available surfaces cover immediate primary updates, manual fallback editing, status probes, OpenRouter-specific scan/probe, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback. The discussion also includes explicit maintainer keep-open guidance, and related #6599 is narrower fallback-chain verification work rather than a full migration workflow.

## What This Changes

Request a staged model migration UX that lets users add a candidate as fallback, test it, promote it to primary, preserve the old primary as fallback, and roll back if needed.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies it against the intended agent surfaces before promotion, preserves the previous primary in the fallback chain, stores rollback state, and documents/tests the config transitions while coordinating with #6599 for fallback-chain verification.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/stage/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or rollback-only tweak to `models set`.
