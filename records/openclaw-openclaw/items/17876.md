## Summary

Keep open. Current main still lacks a staged, provider-generic model migration workflow; the existing model surface covers status/probe, immediate default setting, manual fallback management, OpenRouter scan, and runtime failover, while the discussion already includes maintainer guidance to keep this open.

## What This Changes

Request a staged model migration workflow that can add a candidate as fallback, verify it, promote it to primary, preserve the old primary as fallback, and roll back.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies it against intended agent surfaces before promotion, preserves the previous primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The current-main verification path is high confidence: source and docs inspection show no test/migrate/stage/promote/rollback workflow exists.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or a rollback-only tweak to `models set`.
