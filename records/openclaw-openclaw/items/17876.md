## Summary

Keep open. Current main still has immediate model setting, manual fallback management, auth/profile probes, OpenRouter scan support, and runtime failover, but it does not provide the requested staged test/migrate/promote/rollback workflow; the related open fallback-test issue covers only a narrower piece.

## What This Changes

The issue requests a staged model migration workflow that can test a candidate model before promotion, preserve the old primary as fallback, and roll back if the new primary fails.

## Best Possible Solution

Define and implement a provider-generic `openclaw models` migration flow that stages a candidate, verifies the intended agent surfaces, promotes it only after success, preserves the old primary as fallback, persists rollback state, and documents/tests the transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and live issue inspection on current main give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on the existing set, fallback, status/probe, auth, scan, and runtime failover primitives rather than an OpenRouter-only or fallback-test-only patch.
