## Summary

Keep open. Current main still provides immediate model setting, fallback-list management, OpenRouter-specific scan/probe behavior, and runtime failover, but not the staged, provider-generic migration workflow requested here; the live discussion also contains explicit maintainer keep-open guidance.

## What This Changes

Request a staged model migration UX that can add and test a candidate model, promote it to primary, preserve the old primary as fallback, and roll back.

## Best Possible Solution

Define and implement a provider-generic `openclaw models` migration flow that stages candidates, verifies the intended agent surfaces, promotes only after verification, preserves the old primary in the fallback chain, persists rollback state, and documents/tests the transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, related-thread, and live discussion inspection verify that staged test/migrate/promote/rollback behavior is absent on current main.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than an OpenRouter-only scan extension or fallback-test-only tweak.
