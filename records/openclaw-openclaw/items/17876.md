## Summary

Keep open: current main and v2026.6.5 still provide immediate model selection, manual fallback management, auth probes, OpenRouter scanning, and runtime failover, but not the requested staged verification, promotion, previous-primary preservation, or rollback workflow. The narrower fallback-test request at https://github.com/openclaw/openclaw/issues/6599 remains open and does not cover the full migration lifecycle.

## What This Changes

Add a staged model migration workflow that tests a candidate before promotion, preserves the previous primary as fallback, and supports rollback.

## Best Possible Solution

Design one provider-generic migration workflow that stages and probes a candidate, atomically promotes it while preserving the former primary, records narrowly owned rollback state, and leaves unrelated fallback preferences unchanged.

## Reproduction Assessment

Not applicable; current immediate model switching and runtime failover operate according to their documented contracts, while this issue requests a new staged migration capability.

## Solution Assessment

Unclear: staged migration is a sound safety direction, but maintainers must first define the provider-generic verification depth, command UX, fallback ordering, and rollback persistence contract. Reusing one probe layer with https://github.com/openclaw/openclaw/issues/6599 would be preferable to parallel implementations.
