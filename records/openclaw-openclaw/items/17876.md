## Summary

Keep open: current main and v2026.5.22 still lack the requested staged model migration workflow, and the discussion already has explicit keep-open guidance. Existing code supports immediate primary-model setting, manual fallback list edits, auth probes, OpenRouter scan/probe support, and runtime failover, but not candidate staging, verify-before-promote, old-primary fallback preservation, or one-command rollback.

## What This Changes

Add a staged model migration UX for testing a new model as fallback, promoting it to primary only after verification, preserving the old primary, and rolling back if needed.

## Best Possible Solution

Design a provider-generic `openclaw models` migration flow that stages a candidate, verifies real agent and sub-agent execution before promotion, preserves the old primary as fallback, stores rollback metadata, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, related-item, and latest-release checks show no staged migrate/test/promote/rollback surface on current main or v2026.5.22.

## Solution Assessment

Unclear until maintainers choose the command UX and verification depth. The maintainable path is to build on existing model config, fallback, auth-probe, and scan primitives rather than adding a provider-specific partial workflow.
