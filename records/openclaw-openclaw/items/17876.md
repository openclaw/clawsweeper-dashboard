## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow; the existing surfaces cover immediate default changes, manual fallback list edits, OpenRouter-specific scan/probe support, auth probes, and runtime failover, and a maintainer comment already directed that this stay open for product follow-up.

## What This Changes

The issue requests a staged model migration workflow that can test a candidate model before promotion, preserve the previous primary as fallback, and support rollback.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies intended agent surfaces before promotion, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The current-main verification path is high confidence: source and docs inspection show no staged test/migrate/promote/rollback workflow exists.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or a rollback-only tweak to `models set`.
