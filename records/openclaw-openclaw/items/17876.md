## Summary

Keep open. Current main and v2026.5.12 still lack the requested staged model migration flow; the existing pieces cover immediate default-model writes, manual fallback lists, OpenRouter scan/probe, auth-profile probes, and runtime failover, but not provider-generic staging, verification-before-promotion, old-primary preservation, or rollback state.

## What This Changes

Request a staged `openclaw models` migration workflow that can test a candidate model before promoting it to primary and roll back if needed.

## Best Possible Solution

Design one provider-generic `openclaw models` migration workflow that reuses the existing model config, auth probe, and fallback primitives while coordinating the fallback-chain probe portion with https://github.com/openclaw/openclaw/issues/6599.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, live issue context, and v2026.5.12 inspection give high-confidence proof that the requested staged migration workflow is not present.

## Solution Assessment

Unclear until maintainers define the UX and verification depth. The maintainable direction is to build on the existing model config, auth probe, scan, and fallback primitives instead of adding an OpenRouter-specific or fallback-only partial command.
