## Summary

Keep open. Current main and the latest release still lack the requested staged test/promote/rollback model migration workflow; the existing `models set`, fallback-list, auth-probe, OpenRouter scan, and runtime failover pieces are useful building blocks but do not provide provider-generic candidate staging, verification before promotion, old-primary preservation, or rollback, and the issue discussion already has explicit keep-open product-design guidance.

## What This Changes

The issue requests a staged model migration workflow that can test a candidate model before promoting it to primary and allow rollback.

## Best Possible Solution

Design and ship a documented provider-generic `openclaw models` migration flow that stages arbitrary candidates, verifies real execution before promotion, preserves the previous primary as fallback, persists rollback state, and coordinates the fallback-chain probe surface with https://github.com/openclaw/openclaw/issues/6599.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection on current main and v2026.5.12 gives high-confidence proof that the requested staged test/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the UX and verification depth. The safer direction is to reuse existing model config, auth, probe, and fallback primitives in a provider-generic workflow rather than adding an OpenRouter-specific or fallback-only partial command.
