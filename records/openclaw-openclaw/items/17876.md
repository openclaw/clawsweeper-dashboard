## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow; existing code supports immediate default changes, manual fallback edits, OpenRouter-specific scan/probe, auth probes, and runtime failover, and maintainer discussion already says this needs product follow-up.

## What This Changes

Request a staged `openclaw models` migration workflow that tests a candidate model as fallback before promotion and supports rollback.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies intended agent surfaces before promotion, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The current-main verification path is high confidence: source and docs inspection show no staged test/migrate/promote/rollback workflow exists.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built around existing set, fallback, status/probe, scan, auth, and runtime failover primitives, not an OpenRouter-only scan extension or a rollback-only tweak to `models set`.
