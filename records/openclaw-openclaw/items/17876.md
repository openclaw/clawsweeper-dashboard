## Summary

Keep open: current main and v2026.6.5 still have immediate model setting, manual fallback management, provider/auth probes, OpenRouter scanning, and runtime failover, but not the requested staged test/promote/rollback model migration workflow. The related fallback-test and all-model health-check issues cover useful probe pieces, but neither owns the full migration lifecycle.

## What This Changes

Request a staged model migration workflow that tests a candidate model before promotion, preserves the old primary as fallback, and supports rollback.

## Best Possible Solution

Design one provider-generic model migration workflow that stages and probes a candidate, atomically promotes it while preserving the former primary, records narrowly owned rollback state, and reuses a shared probe layer with the related diagnostics work.

## Reproduction Assessment

Not applicable as a bug reproduction; this issue requests a new staged migration capability. Source inspection gives high confidence that current main lacks the requested test/promote/rollback workflow.

## Solution Assessment

Unclear until maintainers define the product contract. The staged migration direction fits the problem, but the best implementation should share probe infrastructure with https://github.com/openclaw/openclaw/issues/6599 and https://github.com/openclaw/openclaw/issues/63145 instead of adding a parallel partial path.
