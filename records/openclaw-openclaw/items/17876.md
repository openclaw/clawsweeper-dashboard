## Summary

Keep open. Current main still provides immediate model setting, manual fallback management, runtime failover, auth probes, and OpenRouter scan helpers, but not the requested staged test/promote/rollback migration workflow; the live discussion also has explicit keep-open guidance, and the related open work at https://github.com/openclaw/openclaw/issues/6599 and https://github.com/openclaw/openclaw/issues/63145 covers narrower probe pieces rather than the full migration flow.

## What This Changes

The issue requests a staged model migration workflow that can test a candidate model before promotion, preserve the old primary as fallback, and roll back if the new primary fails.

## Best Possible Solution

Design and ship a provider-generic models migration flow that stages arbitrary candidates, verifies real execution paths before promotion, preserves the previous primary as fallback, persists rollback state, and documents how it relates to https://github.com/openclaw/openclaw/issues/6599 and https://github.com/openclaw/openclaw/issues/63145.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection on current main confirm the requested staged test/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the UX and verification contract. The maintainable direction is a provider-generic workflow that reuses existing model config, auth, probe, and fallback primitives instead of adding a provider-specific partial migration path.
