## Summary

Keep open. Current main still lacks a staged, provider-generic model migration workflow; the existing model surfaces cover immediate default changes, manual fallback list management, OpenRouter-specific scan/probe support, and runtime failover rather than test-before-promote with rollback, and the issue discussion already has maintainer keep-open guidance.

## What This Changes

Request a staged `openclaw models` migration workflow that can add a candidate fallback, verify it, promote it to primary, preserve the old primary as fallback, and roll back.

## Best Possible Solution

Add a maintainer-designed, provider-generic `openclaw models` migration workflow that stages any candidate, verifies intended agent surfaces before promotion, preserves the previous primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The current-main verification path is high confidence: source and docs inspection show no staged test/migrate/promote/rollback workflow exists.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or a rollback-only tweak to `models set`.
