## Summary

Keep open. Current main still lacks the requested staged model migration workflow; existing model set, fallback, auth-probe, OpenRouter scan, and runtime failover paths are useful primitives but do not provide candidate staging, verification-before-promotion, old-primary preservation, or rollback.

## What This Changes

The issue requests a staged model migration workflow that tests a candidate model before promotion, preserves the old primary as fallback, and supports quick rollback.

## Best Possible Solution

Define and implement a provider-generic `openclaw models` migration flow that stages a candidate, verifies the intended agent surfaces, promotes it only after success, preserves the old primary as fallback, persists rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and live issue inspection on current main give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on existing set, fallback, status/probe, auth, scan, and runtime failover primitives rather than an OpenRouter-only or fallback-test-only patch.
