## Summary

Keep open: current main and v2026.6.8 still lack the staged model migration workflow, and the nearest existing pieces are immediate primary setting, manual fallback list edits, provider/auth probes, and OpenRouter-specific scan support rather than candidate staging, verification-before-promotion, old-primary preservation, and rollback.

## What This Changes

Request to add a staged model migration workflow that can stage a candidate model, verify it, promote it to primary while preserving fallback safety, and roll back if needed.

## Best Possible Solution

Add a maintainer-designed core `openclaw models` migration workflow that stages an arbitrary candidate, verifies it before promotion, preserves the old primary as fallback, persists rollback state safely, and documents/tests the config transitions alongside related probe features.

## Reproduction Assessment

Not applicable as a failing-bug reproduction: this is a feature request. Source inspection does confirm the requested staged migration commands and rollback flow are absent on current main.

## Solution Assessment

Unclear until maintainers choose the product contract. A staged migration workflow is plausible and aligned, but it should be reconciled with the existing fallback-chain test and all-model health-check proposals before implementation.
