## Summary

Keep open. Current main still lacks the requested staged model migration workflow; existing code supports immediate default-model writes, fallback list edits, OpenRouter free-model scanning, auth probes, and runtime failover, but not provider-generic stage/test/promote/rollback behavior.

## What This Changes

The issue requests a staged model migration UX that can add a candidate model as fallback, test it, promote it to primary, preserve the old primary, and roll back if needed.

## Best Possible Solution

Define and implement a maintainer-designed `openclaw models` migration flow that stages an arbitrary candidate, verifies the intended agent surfaces, preserves the old primary in the fallback chain, persists rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, release, and live issue inspection give a high-confidence verification path that staged test/migrate/promote/rollback behavior is absent.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built on the existing set, fallback, status/probe, scan, auth, and runtime failover primitives rather than an OpenRouter-only extension or fallback-test-only tweak.
