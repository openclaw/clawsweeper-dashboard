## Summary

Keep open: current main still has the timeout runtime-option path implicated by the report, and the issue is already paired with an open fixing PR that uses closing syntax. It should close only after the chosen fix PR lands and proves the Claude ACP spawn path.

## What This Changes

Report that ACPX forwards an unsupported `timeout` session config option to Claude/non-Codex ACP agents, causing `sessions_spawn` to fail with `ACP_TURN_FAILED`.

## Best Possible Solution

Consolidate the linked PRs around a manager-level fix that skips or fail-softs auto-injected timeout controls for backends that do not advertise them, keeps explicit user config writes strict, and closes this issue after merge with live Claude ACP proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main still emits `timeout` from runtime options and either rejects or forwards it for Claude/non-Codex ACP agents, and the issue plus the broader open report provide matching live logs and direct-ACPX controls.

## Solution Assessment

No for the narrow wrapper-only fix as the complete solution; the manager runtime-control path can reject the unsupported timeout before the acpx wrapper sees it. The maintainable fix should handle auto-injected timeout controls at the runtime-options/application boundary while preserving strict explicit config writes.
