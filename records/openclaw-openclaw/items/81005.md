## Summary

Keep open: current main has source-level coverage for the reported Claude ACP timeout leak, but https://github.com/openclaw/openclaw/pull/81006 is still open with closing syntax and needs maintainer disposition before this issue is closed.

## What This Changes

The issue reports that ACPX forwards unsupported `timeout` config options to Claude or other non-Codex ACP agents during `sessions_spawn`, causing `ACP_TURN_FAILED`.

## Best Possible Solution

Keep the current capability-aware contract: enforce turn timeouts inside OpenClaw, emit timeout only for backends that advertise a timeout alias, suppress it for Claude ACP, and resolve the open paired PR before closing this issue as implemented or superseded.

## Reproduction Assessment

No, not against current main: source shows the Claude ACP path now suppresses timeout and the control-plane option builder omits timeout when advertised keys exclude timeout aliases. The 2026.5.7 failure path is high-confidence from the report logs and dependency contract, but I did not live-run ACPX/Claude here.

## Solution Assessment

Yes: capability-aware filtering with a known Claude ACP guard is the narrow maintainable direction. The open universal-drop PR should be reconciled with current main rather than closing this issue while the paired PR remains unresolved.
