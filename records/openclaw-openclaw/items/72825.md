## Summary

Keep open. Current main still has the GitHub Copilot capability gap, but this PR is not merge-ready because its session context fallback returns Anthropic’s 1,048,576-token constant while the PR’s Copilot capability table and supplied upstream evidence use exactly 1,000,000 tokens; it also lacks the required user-facing changelog entry. The paired report and overlapping Copilot catalog PRs remain open, so this is still an implementation candidate rather than obsolete.

## What This Changes

The PR adds GitHub Copilot known-capability overrides for Claude Opus 4.6/4.7 1m variants, enables xhigh for the 4.7 1m internal model, adjusts session context fallback behavior after model switches, and adds targeted unit tests.

## Best Possible Solution

Land a provider-owned Copilot capability fix that keeps synthetic model metadata, default model definitions, session/status context windows, output caps, and thinking levels aligned with Copilot’s exact advertised limits, while leaving dynamic /models catalog discovery to the broader follow-up if maintainers choose that direction.

## Reproduction Assessment

Yes. Current main reproduces the original issue from source through the 128k/8192 Copilot fallbacks and xhigh allowlist omission; the PR’s remaining bug is reproducible by comparing its 1_000_000 capability table to the added fallback that returns 1_048_576.

## Solution Assessment

No, not as currently patched. The narrow known-capability table is a maintainable fix boundary, but the shared session fallback should return or derive Copilot’s exact 1_000_000 cap and the user-facing fix needs a changelog entry.
