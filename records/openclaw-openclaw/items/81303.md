Adds a `session.maintenance.preserveKeys` option to session maintenance types and resolved config so specific session keys can be exempted from pruning.

## Best Possible Solution

Land this only after maintainers accept the public config contract, the key is added to schema/help/docs/tests, and configured keys are merged with active-session preservation across runtime writes and cleanup paths.

Yes for the blocking patch defects from source inspection: the strict config schema lacks `preserveKeys`, and the cleanup callers still preserve only `activeSessionKey`. I did not run mutating CLI proof in this read-only review.

## Solution Assessment

No. Adding type fields and a resolved property alone is incomplete; the maintainable path is to align the config contract and then pass a merged preserve set into every prune/cap path that should honor it.
