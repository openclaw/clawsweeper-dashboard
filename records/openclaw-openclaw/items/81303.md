## Summary

Keep this PR open but do not merge it as-is: it is a plausible implementation path for the open preserve-keys feature, but the patch only changes TypeScript types/resolution while strict config validation and the destructive maintenance call sites still do not honor the configured keys.

## What This Changes

The PR adds a `preserveKeys` field to session maintenance config/resolved types and returns configured values from `resolveMaintenanceConfigFromInput`.

## Best Possible Solution

Implement the accepted preservation contract across schema, types, help/docs, and all save/load/cleanup/disk-budget paths with focused regression coverage, or choose a runtime-provider/default-protection approach instead.

## Reproduction Assessment

Yes, for the PR defects by source inspection: the patch exposes `maintenance.preserveKeys` only in types/resolution, while strict schema and maintenance removal callers still omit it. I did not run mutating CLI proof in this read-only review.

## Solution Assessment

No. A type/resolver-only patch is not the maintainable solution for a new public config option; the feature needs full config-contract plumbing, caller integration, docs/help, and tests after maintainer approval of the semantics.
