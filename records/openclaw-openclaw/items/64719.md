## Summary

Keep this PR open: the underlying channel tool-exposure concern is source-reproducible on current main, but the proposed patch still has blocking access-control scope issues, can break memory flush, does not actually wire `exec` through runtime owner-only authorization, and lacks required real behavior proof.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only helper constant, and expands related unit tests.

## Best Possible Solution

Revise this into a scoped untrusted-external-channel policy, preserve internal providers and memory flush, and wire any `exec` owner-only hardening through the runtime authorization source with real channel/runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns unfiltered tools for channel provider names without explicit policy entries, and auto-reply carries the originating provider into tool construction; I did not run a live channel repro.

## Solution Assessment

No. The hardening direction is valid, but this PR's default deny is too broad, can break memory flush, and leaves the claimed `exec` owner-only behavior outside the runtime authorization source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: The new fallback applies to every non-empty `messageProvider`, not only external auto-reply channels. Internal/control values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event` can lose exec and file tools before owner-only authorization, which regresses trusted internal flows outside the stated threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: Memory-triggered runs are first narrowed to `read` and `write`, and channel memory compaction forwards the originating provider. This default deny removes both remaining tools for normal channel providers unless group policy explicitly re-allows them, so channel memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec through runtime owner-only policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes a helper constant and tests. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
