## Summary

Keep open. The underlying channel auto-reply tool-access gap is still present on current main, but this PR should not merge as-is because the default-deny fallback is overbroad and the exec owner-only hardening is not enforced through the runtime policy.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise the policy so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply contexts while preserving internal/trusted providers, channel memory-flush `read`/`write` behavior, and documented operator allow/deny semantics. If `exec` owner-only hardening remains in scope, wire it through `src/agents/tool-policy.ts` or exec tool metadata with runtime authorization coverage, and review the result alongside #64718.

## Reproduction Assessment

Yes. The issue and PR regressions are reproducible by source-level checks: current main returns the original tool list for providers such as `telegram`, while the PR diff would deny `read`/`write` after memory-flush narrowing and would not make `exec` owner-only in `applyOwnerOnlyToolPolicy`.

## Solution Assessment

No. The direction is valid security hardening, but the current diff is not the narrowest maintainable fix because it applies defaults to trusted/internal providers, breaks memory-flush tool availability, and updates the wrong owner-only enforcement source for `exec`.
