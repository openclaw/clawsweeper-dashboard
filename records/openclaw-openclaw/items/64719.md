## Summary

Keep this PR open: current main still lacks dangerous-tool denial for untrusted channel providers, but the proposed branch is not merge-ready because it applies the deny fallback too broadly, can break memory compaction, and changes an owner-only helper that current main has since removed.

## What This Changes

The PR adds a default deny list for exec and filesystem tools in message-provider tool filtering, threads group allow overrides into that filter, adds exec to an owner-only tool-name helper, and expands related unit tests.

## Best Possible Solution

Revise this into a scoped untrusted-external-channel tool policy that preserves internal providers and memory compaction, and coordinate any exec owner-only hardening with the current runtime policy and https://github.com/openclaw/openclaw/pull/64718.

## Reproduction Assessment

Yes, at source level: current main returns the original tool list for channel providers without explicit policy entries, and auto-reply/memory paths carry the originating provider into tool construction. I did not run a live channel reproduction.

## Solution Assessment

No, not as written. The security direction is valid, but the maintainable fix needs a scoped untrusted-channel boundary, memory/internal-provider coverage, and current-runtime exec authorization if exec owner-only hardening remains in scope.
