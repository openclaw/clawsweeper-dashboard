## Summary

Keep this PR open: current main still lacks dangerous-tool denial for external channel auto-reply, but the proposed patch has blocking access-control scope bugs and needs maintainer/security review before merge.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related tests.

## Best Possible Solution

Revise the policy so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply, preserve internal providers and memory-flush read/write behavior, and wire any `exec` owner-only claim through `src/agents/tool-policy.ts` or exec tool metadata before coordinated security review with https://github.com/openclaw/openclaw/pull/64718.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main returns unfiltered tools for providers such as `telegram`, while the PR diff would remove `read`/`write` from channel memory-flush runs and would not make `exec` owner-only in `applyOwnerOnlyToolPolicy`.

## Solution Assessment

No. The hardening direction is valid, but the current implementation is broader than the threat model and updates the wrong enforcement source for the `exec` owner-only claim.

  - confidence: 0.88
- **[P2] Wire exec owner-only through runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant and tests. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside provider filtering can still receive `exec`.
  - confidence: 0.92
