## Summary

Keep open: current main still has the reported channel tool-policy gap, but the PR head has blocking access-control regressions, missing real behavior proof, an unresolved overlap with https://github.com/openclaw/openclaw/pull/64718, and is currently conflicting with main.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to an owner-only tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise this work into a scoped security policy for untrusted external auto-reply providers, preserve internal providers and memory flush, and wire any `exec` owner-only hardening through the runtime authorization source with real channel/runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the original tool list for channel providers without explicit allow/deny entries, and auto-reply stores the originating provider into tool construction; I did not run a live channel repro.

## Solution Assessment

No. The hardening direction is valid, but this PR over-scopes the default deny, can break memory flush and internal providers, and does not wire `exec` through runtime owner-only policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: The new fallback applies to every non-empty `messageProvider`, not only external auto-reply channels. Current main uses internal/control provider values such as `webchat` and heartbeat/event sources, and this filter runs before owner-only authorization, so the patch can remove exec and file tools from trusted internal flows outside the stated threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: Memory-triggered runs are first narrowed to `read` and `write`, and channel memory compaction forwards the originating provider. The default deny removes both remaining tools for normal channel providers unless group policy explicitly re-allows them, so channel memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec through runtime owner-only policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes a helper constant and its tests. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
