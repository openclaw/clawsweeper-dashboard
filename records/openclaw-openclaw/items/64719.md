## Summary

Keep open. The PR targets a real current-main gap for channel-provider tool filtering, but the proposed patch still has blocking access-control scope issues and an ineffective `exec` owner-only enforcement path.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise the policy to deny dangerous filesystem and exec tools only for untrusted external channel auto-reply contexts, preserve internal providers and memory-flush behavior, and wire any `exec` owner-only hardening through the actual runtime authorization source.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main returns unfiltered tools for providers such as `telegram`, while the PR diff would strip `read`/`write` from channel memory-flush runs and would not make `exec` owner-only in `applyOwnerOnlyToolPolicy`.

## Solution Assessment

No. The hardening direction is valid, but the current implementation is broader than the threat model and updates the wrong enforcement source for the `exec` owner-only claim.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:39-41`
  - body: The new fallback applies the dangerous-tool deny list to every non-empty `messageProvider`. Current main also uses internal/trusted provider values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, so this can remove exec and file tools from owner-controlled flows outside the channel auto-reply threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read/write tools:** `src/agents/pi-tools.message-provider-policy.ts:39-41`
  - body: Channel memory flush runs are narrowed to `read` and `write` before this filter and carry the originating provider. The default deny then removes both tools for normal channel providers, leaving the flush run without the tools it needs to persist memory unless an unrelated group override happens to re-enable them.
  - confidence: 0.88
- **[P2] Enforce exec owner-only in runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes the helper constant and tests. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside provider filtering can still receive `exec`.
  - confidence: 0.92
