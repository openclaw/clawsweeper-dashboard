## Summary

Keep open: current main still lacks dangerous-tool denial for external channel providers, but this PR is not safe to merge as-is because the default deny applies too broadly, can break channel memory flush, and does not actually enforce the new `exec` owner-only claim at runtime.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool constant, and expands related unit tests.

## Best Possible Solution

Keep the PR open and revise the policy so dangerous filesystem/exec tools are denied for untrusted external channel auto-reply contexts while internal/trusted providers, channel memory flush, and runtime owner-only authorization remain coherent.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the original tool list for channel providers with no explicit allow/deny entry, and auto-reply forwards the originating channel into that filter; I did not run a live channel repro in this read-only review.

## Solution Assessment

No. The hardening direction is valid, but this diff needs tighter provider scoping, memory-flush preservation, and real runtime `exec` authorization wiring before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted channel providers:** `src/agents/pi-tools.message-provider-policy.ts:38-40`
  - body: The new fallback deny list applies to every non-empty `messageProvider`, not only external auto-reply channels. Current main uses internal/trusted provider values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, so owner-controlled internal runs can lose exec and file tools outside this PR's threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:38-40`
  - body: Memory-triggered runs are narrowed to `read` and `write` before message-provider filtering, and channel memory compaction forwards the originating channel. For normal channel providers this deny list removes both remaining tools unless group policy happens to re-allow them, so memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec owner-only through runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
