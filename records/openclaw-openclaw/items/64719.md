## Summary

Keep open: current main still lacks dangerous-tool denial for external channel providers, but this PR is not mergeable as-is because the broad default-deny policy can strip tools from internal providers and memory flush, `exec` owner-only is not enforced at runtime, and the external PR lacks real behavior proof.

## What This Changes

This PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, marks `exec` in the owner-only core tool-name constant, and updates related unit tests.

## Best Possible Solution

Revise the policy so untrusted external auto-reply channels lose dangerous tools while internal providers, channel memory flush, explicit operator policy, and runtime `exec` authorization are coherently covered, then verify with focused tests and redacted live/runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the original tool list for channel providers with no explicit allow/deny entry, and prepared auto-reply runs pass the originating channel into that filter; I did not run a live channel repro.

## Solution Assessment

No. The hardening direction is valid, but this diff over-scopes default denies, can break memory flush, and tests a constant that runtime `exec` authorization does not use.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: The new fallback deny list applies to every non-empty `messageProvider`, not only external auto-reply channels. Current main passes internal values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event` through this same filter, so owner-controlled internal runs can lose exec and file tools outside the stated threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: Memory-triggered runs are narrowed to `read` and `write` before message-provider filtering, and channel memory compaction forwards the originating provider. This default deny removes both remaining tools for normal channel providers unless group policy happens to re-allow them, so memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec owner-only through runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
