## Label Justifications

- `P0`: The item concerns unauthenticated channel access to exec and filesystem tools, which is a potential security-boundary bypass.
- `merge-risk: 🚨 compatibility`: The default deny fallback can remove existing tools from trusted internal/control providers and operator workflows outside the stated channel threat model.
- `merge-risk: 🚨 session-state`: The fallback can strip read/write from channel memory compaction, risking lost or stale session memory persistence.
- `merge-risk: 🚨 security-boundary`: The PR changes tool-access control semantics and leaves the claimed exec owner-only hardening outside the runtime authorization source.

## Summary

Keep this PR open: it addresses a real security-sensitive gap, but the latest patch still over-scopes the default deny policy to trusted/internal providers, can break channel memory compaction, leaves the claimed exec owner-only hardening unenforced at runtime, and has only unit-test proof.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider tool filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise this into a scoped untrusted-channel tool policy that preserves internal providers and memory compaction, and either wire exec through runtime owner-only authorization or split it into a coordinated follow-up with https://github.com/openclaw/openclaw/pull/64718.

## Reproduction Assessment

Yes, source-reproducible. Current main passes non-allow/deny channel providers through unchanged, auto-reply carries the originating provider into tool construction, and the PR-head fallback applies broadly to every non-allowlisted provider.

## Solution Assessment

No. The hardening direction is valid, but this patch over-scopes the fallback and leaves exec owner-only as a helper-constant-only change; a scoped policy with runtime authorization coverage is safer.
