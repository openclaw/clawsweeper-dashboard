## Summary

Keep open: the hardening target is valid and still missing on current main, but this PR has blocking access-control scope regressions, an ineffective runtime `exec` owner-only change, missing real behavior proof, and overlap with https://github.com/openclaw/openclaw/pull/64718.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider tool filtering, passes group allow overrides into that filter, adds `exec` to an owner-only helper constant, and expands related unit tests.

## Best Possible Solution

Keep this PR open and revise the policy so untrusted external channel auto-reply loses dangerous tools while internal providers, memory flush, explicit operator policy, and runtime `exec` authorization remain coherent.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the original tool list for channel providers without explicit allow/deny entries, and prepared auto-reply runs pass the originating provider into that filter; I did not run a live channel repro.

## Solution Assessment

No. The hardening direction is valid, but the current PR over-scopes default denies, can break memory flush, and does not wire the claimed `exec` owner-only behavior through runtime policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: The new fallback applies to every non-empty `messageProvider`, not only external auto-reply channels. Current main also uses internal values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event` on this field, so owner-controlled internal runs can lose exec and file tools outside the stated threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: Memory-triggered runs are narrowed to `read` and `write` before message-provider filtering, and channel memory compaction forwards the originating provider. This default deny removes both remaining tools for normal channel providers unless group policy explicitly re-allows them, so memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec through runtime owner-only policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
