## Summary

Keep open. The PR addresses a real security gap on current main, but the current diff has blocking access-control scope issues and still needs redacted runtime proof before merge.

## What This Changes

The PR adds a default dangerous-tool deny list for message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool constant, and expands unit tests.

## Best Possible Solution

Revise the policy to deny dangerous filesystem and exec tools only for untrusted external channel auto-reply contexts, preserve internal providers and memory-flush read/write behavior, and wire any `exec` owner-only hardening through runtime authorization before coordinated security review.

## Reproduction Assessment

Yes. Source inspection shows current main returns the original tools for non-empty providers not in the allow/deny maps, and auto-reply forwards the resolved originating channel into this filter; I did not run a live channel repro in this read-only review.

## Solution Assessment

No. The hardening direction is valid, but this diff applies default denies to all non-empty providers, can break memory flush, and changes a helper constant instead of runtime exec authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Scope default denies to untrusted external providers:** `src/agents/pi-tools.message-provider-policy.ts:41-43`
  - body: The new fallback builds `DEFAULT_DENIED_TOOLS` for every normalized provider outside the node allowlist. Current main also uses internal providers such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, and message-provider filtering runs before owner-only authorization, so owner-controlled internal runs can lose exec and file tools outside the channel auto-reply threat model.

For Telegram/Discord/etc. this deny list then removes both allowed tools unless group policy explicitly re-allows them, so channel memory persistence can stop working.
  - confidence: 0.88
- **[P2] Wire exec owner-only through runtime authorization:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant. Runtime owner-only filtering is driven by `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
