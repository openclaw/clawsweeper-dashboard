## Summary

Keep this PR open. The security hardening target is real and not implemented on current main, but the PR still has blocking policy-scope and runtime-authorization issues, lacks after-fix real behavior proof, and overlaps with related open security PRs.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to an owner-only helper constant, and expands unit tests.

## Best Possible Solution

Revise the policy so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply while internal providers, memory-flush read/write behavior, explicit operator policy semantics, and runtime `exec` authorization stay coherent.

## Reproduction Assessment

Yes, source-reproducible. Current main returns all tools for channel providers without explicit policy entries, and prepared auto-reply passes the originating provider into the message-provider filter; I did not run a live channel repro.

## Solution Assessment

No. The hardening direction is valid, but this diff applies default denies to every non-empty provider, can strip memory-flush read/write, and changes an owner-only helper without runtime enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope default denies to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: The new fallback applies `DEFAULT_DENIED_TOOLS` to every non-empty `messageProvider`, not only external auto-reply channels. Current main uses internal values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, so owner-controlled internal runs can lose exec and file tools outside the stated threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read and write tools:** `src/agents/pi-tools.message-provider-policy.ts:42-46`
  - body: Memory-triggered runs are narrowed to `read` and `write` before message-provider filtering, and channel memory compaction forwards the originating provider. This default deny removes both remaining tools for normal channel providers unless group policy explicitly re-allows them, so memory persistence can silently stop working.
  - confidence: 0.88
- **[P2] Wire exec through runtime owner-only policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only changes this helper constant. Runtime owner-only filtering uses `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside message-provider filtering can still receive `exec`.
  - confidence: 0.92
