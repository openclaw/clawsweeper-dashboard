## Label Justifications

- `P0`: The item concerns unauthenticated channel access to exec and filesystem tools, which is a potential security-boundary bypass.
- `merge-risk: 🚨 compatibility`: The PR's fallback can remove tools from existing internal/control providers and operator workflows outside the stated channel threat model.
- `merge-risk: 🚨 session-state`: The fallback can strip read/write from channel memory compaction, risking lost or stale session memory persistence.
- `merge-risk: 🚨 security-boundary`: The PR changes tool-access control semantics and leaves exec owner-only hardening outside the runtime authorization source.

## Summary

Keep open. Current main still has the channel-provider dangerous-tool gap, but the PR head is not mergeable as-is because its default deny applies beyond untrusted channels, can strip read/write from memory compaction, and does not make exec owner-only at runtime.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider tool filtering, passes group allow overrides into that filter, adds exec to an owner-only helper constant, and expands unit coverage.

## Best Possible Solution

Revise this into a scoped untrusted-channel tool policy that preserves internal providers and memory compaction, and either wire exec through runtime owner-only authorization or split it into a coordinated follow-up with https://github.com/openclaw/openclaw/pull/64718.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the original tool list for non-allow/deny channel providers and auto-reply carries the originating provider into tool construction; I did not run live channel proof.

## Solution Assessment

No. The hardening direction is valid, but this patch over-scopes the fallback and leaves exec owner-only as a helper-constant-only change; a scoped policy with runtime authorization coverage is safer.
