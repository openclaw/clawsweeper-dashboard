## Summary

Keep open: the PR targets a plausible WebChat/Gateway bug, but the patch edits the non-agent final assembly path while the reported media append path requires an agent run, so the audio can still be absent from agent-run final broadcasts. It also lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/gateway/server-methods/chat.ts` to always use delivered final payloads for WebChat final assembly and adds an assistant-media idempotency key to the later transcript append.

## Best Possible Solution

Move the fix to the agent-run final broadcast path so WebChat receives the same audio content already appended to the transcript, with regression coverage for final chat payload media and duplicate-transcript behavior.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the agent-run media path can persist audio to the transcript while the final chat event is still built from buffered text only, so the defect is source-reproducible but still needs WebChat/TTS runtime proof.

## Solution Assessment

No. The PR changes the non-agent final assembly branch and adds an unsafe idempotency key there; the narrower maintainable fix should bridge media into the agent-run final broadcast path and handle duplicate transcript append results explicitly.
