## Summary

Keep open: this is a plausible WebChat/Gateway bug-fix PR, but the patch has blocking source-level issues and still lacks after-fix real behavior proof for the reported WebChat/TTS flow.

## What This Changes

The PR changes WebChat final handling in `src/gateway/server-methods/chat.ts` to always derive final payloads from delivered replies and to pass `${clientRunId}:assistant-media` when appending the broadcast assistant transcript message.

## Best Possible Solution

Revise the gateway fix to target the reachable agent-run media delivery path, preserve a full assistant final message when transcript dedupe occurs, add focused regression coverage, and attach redacted real WebChat/TTS proof.

## Reproduction Assessment

Yes for the patch defect by source inspection; the reported WebChat/TTS symptom itself still lacks live proof. Current source shows the changed payload branch is guarded by `!agentRunStarted` while assistant-media appending requires `agentRunStarted`, and duplicate idempotency returns no message for the final broadcast to reuse.

## Solution Assessment

No. The patch changes a branch that does not clearly reach the reported already-appended media state and reuses an idempotency key whose duplicate path does not return the assistant message needed for a normal final event.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Fix the reachable media-final path:** `src/gateway/server-methods/chat.ts:2582`
  - body: Changing this expression does not hit the path described in the PR: this whole block only runs under `if (!agentRunStarted)`, while `appendedWebchatAgentMedia` can only be set after `onAgentRunStart` flips `agentRunStarted` and the helper accepts a media-bearing payload. Please move the fix to the reachable agent-run final/media delivery path or add a regression that demonstrates this branch is reached.
  - confidence: 0.86
- **[P2] Preserve the assistant message when deduping:** `src/gateway/server-methods/chat.ts:2686`
  - body: The new `idempotencyKey` reuses the same `${clientRunId}:assistant-media` key that the earlier assistant-media append writes. On that duplicate path `appendAssistantTranscriptMessage` returns `{ ok: true }` without `message` or `messageId`, but this block still builds the final WebChat event from `appended.message`, so the result can lose the assistant role, timestamp, and id. Either fetch/reuse the existing assistant message or skip the transcript append while constructing a full assistant final message for broadcast.
  - confidence: 0.84
