## Summary

Keep open: the reported WebChat/Gateway audio-delivery bug is plausible and paired with an open issue, but this PR is not mergeable because it changes the non-agent final branch while the media append path requires an agent run, and it lacks after-fix real behavior proof.

## What This Changes

The PR removes the raw-final-payload clearing branch in `src/gateway/server-methods/chat.ts` and adds an assistant-media idempotency key to the non-agent WebChat final transcript append.

## Best Possible Solution

Move the fix to the agent-run final broadcast path so WebChat receives the same audio content already appended to the transcript, with regression coverage for final chat payload media and duplicate-transcript behavior.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection on current main shows the agent-run media helper can append audio to the transcript while the agent-run final broadcast still builds only buffered text.

## Solution Assessment

No. The PR edits the non-agent final assembly branch, but the reported media-append condition requires `agentRunStarted`; the safer repair is to carry the prepared media into the agent-run final broadcast path and handle transcript idempotency duplicates explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fix the agent-run media broadcast path:** `src/gateway/server-methods/chat.ts:2582-2584`
  - body: This assembly runs only under `if (!agentRunStarted)`, but `appendWebchatAgentMediaTranscriptIfNeeded` returns unless an agent run has started. For the reported WebChat TTS case, this code is not reached, so the final chat event still comes from `server-chat.ts` and carries only buffered text without the audio block.
  - confidence: 0.9
- **[P2] Handle duplicate idempotency hits before broadcasting:** `src/gateway/server-methods/chat.ts:2686`
  - body: Adding the idempotency key here means a retry with an existing transcript key can make `appendAssistantTranscriptMessage` return `ok: true` without `message`. This branch then builds the final broadcast from `appended.message`, which can emit a malformed assistant message missing the normal envelope fields.
  - confidence: 0.82
