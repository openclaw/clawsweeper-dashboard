## Summary

Keep open: current main and v2026.6.9 can send Telegram polls and request Telegram poll-answer updates, but there is still no current-main `poll_answer` handler that routes votes into an agent-visible turn. The open draft fix candidate at https://github.com/openclaw/openclaw/pull/95309 now owns the implementation path, so this issue should stay open until that PR or a replacement lands.

## What This Changes

Route Telegram non-anonymous poll votes into agent-visible turns/events, or explicitly document Telegram poll votes as display-only.

## Best Possible Solution

Land a Telegram plugin fix that stores pollId-to-session context in canonical plugin state, authorizes the voter, routes a concise system event to the correct direct/topic session, and includes live Telegram proof; otherwise document polls as display-only by explicit maintainer decision.

Yes. Source inspection shows current main can send public Telegram polls and request `poll_answer` updates, but no handler routes a later vote into a message path or system event.

## Solution Assessment

Unclear. The open PR's plugin-state context and `poll_answer` handler are a plausible direction, but the best fix still needs maintainer confirmation of poll-vote semantics and live Telegram proof before this issue should close.
