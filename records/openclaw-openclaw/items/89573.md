## Summary

Keep open: current main and the latest release still subscribe to Telegram `poll_answer` updates but have no handler that routes those updates into an agent turn or system event. The prior implementation PR was closed unmerged and pointed back to this issue as the canonical remaining work, so cleanup closure would lose a real Telegram interaction gap.

## What This Changes

Route Telegram non-anonymous `poll_answer` votes for bot-created polls into agent-visible turns/events, or document Telegram poll votes as display-only.

## Best Possible Solution

Make the product decision, then either add Telegram-owned `poll_answer` routing through existing authorization/session patterns and canonical storage or document Telegram polls as outbound-only/display-only.

## Reproduction Assessment

Yes, source-reproducible. Current main requests `poll_answer` updates but only has message, reaction, and callback handlers, so a poll vote has no route into an agent turn without a live Telegram run.

## Solution Assessment

Unclear until maintainers choose semantics. A Telegram-owned handler using existing event authorization and canonical state storage is plausible, while a docs-only display-only decision is also maintainable if poll votes should not wake agents.
