## Summary

Keep open: current main and the latest release still request Telegram `poll_answer` updates, but the Telegram bot registers no `poll_answer` handler or agent-visible route. The closed implementation PR was not merged and was explicitly closed back to this issue, so there is no safe cleanup close.

## What This Changes

The issue requests that Telegram `poll_answer` updates from non-anonymous bot-created polls be routed into agent-visible turns/events, or clearly documented as display-only.

## Best Possible Solution

Add a Telegram-owned `poll_answer` route using existing authorization, session-routing, and canonical state patterns, or explicitly document Telegram polls as display-only if maintainers choose that product contract.

## Reproduction Assessment

Yes. Source inspection shows current main requests `poll_answer` updates and can create public Telegram polls, but no handler routes a later vote into a message path or system event.

## Solution Assessment

Unclear until maintainers choose poll-vote semantics. A Telegram-owned handler with canonical state storage is plausible, while a docs-only display-only decision is also maintainable if poll votes should not wake agents.
