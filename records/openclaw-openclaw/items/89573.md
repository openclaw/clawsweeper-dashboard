## Summary

Keep open: current main and the latest release still have no Telegram poll_answer handler, so non-anonymous poll votes are not routed into agent-visible turns or system events. A prior implementation PR exists, but it was closed unmerged after review found design/proof problems, so the remaining work needs a maintainer decision rather than cleanup closure.

## What This Changes

Route Telegram non-anonymous poll votes into agent-visible events, or document Telegram polls as outbound display-only UI.

## Best Possible Solution

Choose the product semantics first, then either add a Telegram-owned poll_answer route using existing authorization/session patterns and canonical state storage, or document Telegram polls as outbound-only.

## Reproduction Assessment

Yes, from source. Current main subscribes to poll_answer through Telegram allowed updates, but there is no handler that routes poll_answer into synthetic messages or system events; I did not run a live Telegram account reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose semantics. Routing poll_answer through the existing Telegram authorization/session pattern is plausible, but a docs-only display-only decision is also maintainable if poll votes should not wake agents.
