## Summary

Keep open: current main still forwards the oversized compaction reserve token budget, while this PR applies the narrow wrapper clamp with focused regression coverage and a changelog entry. I found no blocking functional or security issue in the patch, but exact-head CI still had unrelated-looking failures and in-progress jobs at inspection time, so this needs maintainer validation and merge rather than cleanup closure.

## What This Changes

The branch clamps compaction summary `reserveTokens` by the selected model output limit before calling Pi `generateSummary`, adds reserve-token clamping tests, updates an existing compaction mock, and records a changelog fix entry.

## Best Possible Solution

Land the OpenClaw-side wrapper clamp with its regression coverage once exact-head validation is clean, and keep any upstream Pi helper change as a separate dependency follow-up.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main can pass `reserveTokens: 300000` into pinned Pi `generateSummary`, which derives a 240000 output-token request, while the affected Claude 4.6 catalog entries advertise a 128000 output limit.

## Solution Assessment

Yes. Clamping in `summarizeChunks` is the narrowest maintainable OpenClaw-side fix because the built-in fallback and staged summarization paths pass through that wrapper before the pinned Pi helper derives provider `maxTokens`.
