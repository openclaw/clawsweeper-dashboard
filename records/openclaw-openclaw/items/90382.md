## Summary

Keep open: current `main` and the latest release still route OpenAI ChatGPT OAuth responses streaming through the SDK without an explicit SSE `Accept` header, while the guard rejects a successful stream response that has a missing content type.

## What This Changes

The issue reports a `v2026.6.1` regression where `openai/gpt-5.5` via ChatGPT OAuth fails in the OpenClaw responses stream path with `invalid_provider_content_type`.

## Best Possible Solution

Add a default `Accept: text/event-stream` for the `openai-chatgpt-responses` SDK stream transport, with regression tests for header merge precedence and the guarded stream path.

## Reproduction Assessment

Yes, at source level. The reporter gives a concrete probe command and current `main` shows the SDK stream route can send JSON Accept while the guarded fetch rejects a 200 response with missing content type.

## Solution Assessment

Yes, the narrowest maintainable fix is to set the SSE Accept header on the ChatGPT responses SDK stream request path, matching the sibling direct Codex HTTP implementation without adding config or fallback behavior.
