## Summary

Keep open. Current main still lets the affected gateway-backed JSON commands reach their generic stderr-only failure path before writing stdout JSON, and two open implementation PRs explicitly close this issue, so cleanup should wait for one to land.

## What This Changes

The issue requests structured stdout JSON for gateway-backed `--json` command transport failures instead of empty stdout plus human-readable stderr.

## Best Possible Solution

Land one focused JSON transport-error implementation with regression coverage, then let the linked PR close this issue and supersede the other open implementation.

## Reproduction Assessment

Yes, source-level. Current main awaits the affected gateway RPCs before reaching their JSON output branches, while typed close/timeout `GatewayTransportError` values are rethrown to the generic stderr/exit path.

## Solution Assessment

Yes. A shared serializer for typed gateway transport errors, applied only to JSON-mode command failure paths, is the narrow maintainable fix; the remaining choice is which open PR envelope to land.
