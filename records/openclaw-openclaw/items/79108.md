## Summary

Keep open: current main still routes the affected commands through a throwing gateway RPC path before any JSON is written, while the docs and `gateway probe` establish a machine-readable JSON expectation for these automation surfaces.

## What This Changes

The issue asks gateway-backed `--json` commands to emit a structured JSON failure on gateway connection errors, or to document any success-only JSON behavior.

## Best Possible Solution

Add a shared gateway CLI JSON failure envelope for transport/connect errors, reuse the structured probe fields where practical, and cover the three reported commands with regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes from source inspection: the affected commands await gateway RPC calls before any JSON write, so transport failure exits through stderr-only error handling. I did not run a live Windows packaged reproduction in this read-only review.

## Solution Assessment

Yes: emitting a structured nonzero JSON failure for `--json` gateway transport errors is the narrowest maintainable fix and matches the existing `gateway probe --json` precedent better than documenting success-only behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79108. On current main, `openclaw health --json`, `openclaw gateway health --json`, and `openclaw devices list --json` can exit on gateway transport/connect failure without writing parseable JSON to stdout. Add a narrow shared error-envelope path for gateway-backed `--json` command failures, preferably using existing `GatewayTransportError` fields and the `gateway probe --json` failure precedent. Preserve existing successful payload shapes, keep human diagnostics on stderr, do not change gateway protocol or config behavior, and do not broaden device local-fallback policy. Add focused tests for the three affected commands and update user-facing docs/changelog if behavior changes.
