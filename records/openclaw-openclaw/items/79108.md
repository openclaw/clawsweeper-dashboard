## Summary

Keep open. Current main still has source-level paths where the affected `--json` commands can throw a `GatewayTransportError` before writing stdout JSON, and two open closing PRs are already linked as implementation candidates.

## What This Changes

The issue requests structured JSON stdout for gateway-backed `--json` command connection failures in `health`, `gateway health`, and `devices list`.

## Best Possible Solution

Land one focused implementation with a shared `GatewayTransportError` JSON serializer for JSON-mode gateway command failures, then let the merged closing PR close this issue.

## Reproduction Assessment

Yes, source-level. The issue gives concrete unavailable-gateway commands, and current main still awaits gateway RPC success before writing JSON in the affected command paths.

## Solution Assessment

Yes. A shared serializer for typed gateway transport errors, applied only to JSON-mode command error paths while preserving non-JSON stderr behavior, is narrower and more maintainable than documenting success-only JSON or parsing stderr.
