## Summary

Keep open. Current main still lets gateway transport failures escape from the listed `--json` commands before any stdout JSON write, while `gateway probe` is the inspected path that serializes a failed probe result.

## What This Changes

Request gateway-backed CLI commands to emit structured stdout JSON, or document limitations, when gateway connection fails under `--json`.

## Best Possible Solution

Add a shared gateway CLI JSON failure formatter for transport close/timeout errors and apply it to the affected `--json` command paths while preserving human stderr behavior for non-JSON mode.

## Reproduction Assessment

Yes, source-level. On current main, the affected commands await gateway RPC calls before writing JSON, and typed close/timeout failures therefore leave through the existing stderr/exit paths.

## Solution Assessment

Yes. The maintainable fix is a shared `GatewayTransportError` to JSON error contract in the gateway-backed CLI wrappers, plus targeted regression tests, rather than per-command stderr parsing or a docs-only exception.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79108. Gateway-backed `--json` commands should emit a structured JSON error object on stdout when the gateway connection closes or times out, while still exiting non-zero. Start with the reported commands: `openclaw health --json`, `openclaw gateway health --json`, and `openclaw devices list --json`. Prefer a shared helper based on `isGatewayTransportError`/`GatewayTransportError` over command-specific string parsing. Preserve existing human stderr behavior for non-JSON mode and do not change `gateway probe` except to keep its current failure behavior as the control case. Add focused regression tests for the JSON failure paths, update `CHANGELOG.md` because this is user-facing CLI behavior, and update docs only if the fix intentionally leaves any advertised `--json` command success-only.
