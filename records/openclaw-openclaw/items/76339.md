## Summary

Keep this open. Current main still has the source-visible Linux gateway install path that can select a supported nvm/fnm/volta/asdf Node for the managed systemd service while doctor/audit already treat that service shape as non-standard, and the open linked fix PR has not merged yet.

## What This Changes

The issue asks OpenClaw to stop `gateway install --force` from regenerating an upgraded Linux systemd gateway service with version-manager Node/PATH and stale inline gateway-token metadata.

## Best Possible Solution

Review and land the open gateway service runtime PR if its tests and changelog remain correct, then verify whether the inline-token and RPC-timeout observations are resolved or need a narrower follow-up.

## Reproduction Assessment

Yes for the core service-shape regression by source inspection: current main can select the invoking version-manager Node for `ExecStart` and PATH while audit/doctor classify that shape as needing repair. I did not live-reproduce the CPU-bound RPC timeout on an Ubuntu host.

## Solution Assessment

No current-main fix is present. The narrowest maintainable fix is to align install with doctor/audit by preferring a supported system Node for managed Linux gateway services while preserving fallback when no supported system Node exists.
