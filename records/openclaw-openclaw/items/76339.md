## Summary

Keep open. Current main still has a source-visible path where a supported nvm/fnm/volta/asdf `process.execPath` can become the managed gateway service `ExecStart` and PATH, and two open PRs still reference this issue with closing intent.

## What This Changes

Reports that shipped v2026.5.2 `gateway install --force` can regenerate a Linux systemd gateway service with nvm `ExecStart`/PATH and inline `OPENCLAW_GATEWAY_TOKEN` after an upgraded install.

## Best Possible Solution

Land or supersede the open gateway service Node-selection PRs with focused Linux version-manager regression coverage and explicit proof that forced reinstall does not persist generated gateway tokens inline.

## Reproduction Assessment

Yes for the core Node/PATH service-shape regression by source inspection: current main can still feed a supported version-manager `process.execPath` into the service install plan. The inline-token and CPU-bound RPC timeout portions do not yet have a high-confidence current-main reproduction path.

## Solution Assessment

No for current main; install and doctor still disagree on the canonical Linux gateway service runtime. The narrow maintainable fix is to align install with the system-Node repair/audit policy, then separately verify whether token persistence or RPC timeout symptoms remain.
