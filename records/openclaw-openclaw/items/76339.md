## Summary

Keep this issue open. Current main still has a source-visible path where `gateway install --force` can regenerate a Linux systemd service with a version-manager Node `ExecStart` and PATH entry that OpenClaw's own service audit warns about; the closed #76313 context does not cover this narrower reinstall regression.

## What This Changes

The report asks for Linux `gateway install --force` after an upgraded install to preserve the repaired service shape: no nvm service runtime/PATH, no inline `OPENCLAW_GATEWAY_TOKEN`, and healthy `--require-rpc` status.

## Best Possible Solution

Align Linux systemd install and doctor repair around one hardened service plan: prefer a supported system Node for managed services, keep version-manager PATH entries out, and ensure gateway auth secrets are loaded from config or owner-only env files rather than inline unit metadata.

## Reproduction Assessment

Yes for the core service-shape regression by source inspection: a supported nvm `process.execPath` is selected for `ExecStart` and injected into service PATH. No high-confidence local reproduction was established for the CPU-bound RPC timeout or every inline-token source.

## Solution Assessment

No, current main is not the best solution because doctor and install can produce different service definitions for the same repaired Linux gateway. The safer direction is to share the doctor/audit canonical service plan with `gateway install --force` and cover the upgrade reinstall path in tests.
