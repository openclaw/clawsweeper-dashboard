## Summary

Keep this issue open. Current main still has the source-visible Linux gateway install path that can select a supported nvm/fnm/volta/asdf Node for `ExecStart` and inject its bin dir into the managed service PATH, and the linked closing PR #76346 is open rather than merged.

## What This Changes

The issue reports that `openclaw gateway install --force` on v2026.5.2 can regenerate a Linux systemd gateway service with nvm Node/PATH and inline `OPENCLAW_GATEWAY_TOKEN`, leading to service-audit warnings and RPC timeouts after an upgrade.

## Best Possible Solution

Review and land or close #76346, then verify whether the inline-token and RPC-timeout evidence is fixed by that change or should be split into a separate focused issue.

## Reproduction Assessment

Yes for the core service-shape regression by source inspection: current main can still choose a version-manager Node for `ExecStart` and add its bin dir to PATH while audit/doctor warn about that shape. The inline-token and RPC-timeout portions are not fully reproduced from source in the same path.

## Solution Assessment

Yes for the nvm/PATH bug: aligning gateway install with doctor/audit by preferring a supported system Node is the narrow maintainable fix. It is unclear whether that also resolves the reported inline-token and RPC-timeout symptoms, so the linked PR should verify or split those pieces.
