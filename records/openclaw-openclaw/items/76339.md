## Summary

Keep this issue open. Current main still has a source-visible divergence: `gateway install` can choose a supported nvm `process.execPath` and inject its bin directory into PATH, while the service audit flags that same shape and doctor repair migrates to system Node when available.

## What This Changes

The issue asks for `gateway install --force` after uninstall/reinstall to generate the same hardened Linux systemd service shape as doctor repair: no version-manager `ExecStart`/PATH and no inline gateway token.

## Best Possible Solution

Unify Linux systemd install and doctor repair around one canonical service plan: use a supported system Node for managed gateway services when available, keep version-manager paths out of `ExecStart` and service PATH, and move managed gateway secrets to file/runtime sources rather than inline unit metadata.

## Reproduction Assessment

Yes for the core service-shape regression by source inspection: current main can select an nvm `process.execPath` for `ExecStart` and add its bin directory to the service PATH. No live VPS/systemd reproduction was established for the CPU-bound RPC timeout or every inline-token source.

## Solution Assessment

No, current main is not the best solution because doctor and `gateway install --force` can produce different Linux systemd service definitions for the same repaired gateway. The safer fix is to share the canonical doctor/audit service plan with forced install and cover the upgrade reinstall path in tests.
