## Summary

Keep open: the PR improves the non-gateway `executeSendAction` path, but the TTS transform is still placed after gateway message-action dispatch, so Telegram gateway sends can still bypass TTS and send raw directives. The supplied proof also does not exercise a real TTS-enabled message send.

## What This Changes

The PR adds lazy-loaded TTS payload transformation to `executeSendAction` and forwards the transformed payload through prepared, local plugin-action, and core fallback sends.

## Best Possible Solution

Apply TTS once before gateway/plugin/core send branch selection, or explicitly plumb the transformed payload through every branch, then add focused regression coverage and redacted live behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has final-reply TTS normalization, but gateway message-action sends return before `executeSendAction`, and the PR still leaves that bypass in place.

## Solution Assessment

No. The patch is a useful partial fix for non-gateway sends, but the maintainable fix needs the normalization boundary moved before branch selection or carried into the gateway action parameters too.
