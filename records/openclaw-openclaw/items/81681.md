## Summary

Keep open: this targets a real source-reproducible TTS gap, but the patch is incomplete because it runs too late for gateway message actions, uses a TTS kind that the default config suppresses, and does not propagate synthesized media to legacy plugin-action params. The supplied terminal proof covers only the TTS gate, not a real `message(action=send)` path after the fix.

## What This Changes

The branch lazily loads the TTS runtime in `executeSendAction`, transforms the default send payload, syncs transformed text into params, and sends the transformed payload through prepared/core fallback branches.

## Best Possible Solution

Move TTS normalization to the shared message-action payload construction or gateway handoff so direct, prepared, plugin-action, and gateway sends all receive the same transformed payload, then add focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main's `executeSendAction` has no TTS gate/runtime call while final delivery does use the TTS payload path. I did not run a live provider repro in this read-only review.

## Solution Assessment

No. The PR is not the best fix as written because the chosen insertion point misses gateway actions, `kind: "tool"` is suppressed by the default TTS mode, and synthesized media is not copied into legacy plugin-action params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply TTS before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263-271`
  - body: This block only runs inside `executeSendAction`, but `handleSendAction` returns from `runGatewayPluginMessageActionOrNull` before calling it when a channel action resolves to `"gateway"`; Telegram send actions do that. A Telegram `message(action=send)` still forwards the original params to the gateway with raw `[[tts:text]]` tags and no synthesized media, so the PR misses one of the reported paths.
  - confidence: 0.91
- **[P2] Use a TTS kind that can synthesize visible sends:** `src/infra/outbound/outbound-send-service.ts:276`
  - body: Passing `kind: "tool"` keeps the default `messages.tts.mode: "final"` path from generating audio because `maybeApplyTtsToPayload` returns before synthesis for non-final kinds. With the default config, this only strips directives and still does not attach audio for `message(action=send)`, contrary to the final-reply parity this PR is trying to restore.
  - confidence: 0.88
- **[P2] Pass synthesized media into plugin action params:** `src/infra/outbound/outbound-send-service.ts:280-283`
  - body: The legacy plugin-action branch reads `params.ctx.params`, but after TTS the patch only copies back `message`. When synthesis succeeds, fields such as `mediaUrl`, `mediaUrls`, and `audioAsVoice` stay only on `defaultPayload`, so channels without `prepareSendPayload` can still send text without the generated audio.
  - confidence: 0.86
