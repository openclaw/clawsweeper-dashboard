## Summary

Keep this PR open but do not merge it as-is: the underlying TTS send bug is valid, but the branch still misses gateway-mode sends, default TTS synthesis, and plugin-action media propagation, and the supplied proof does not exercise the changed real send path.

## What This Changes

The branch adds lazy TTS runtime loading in `executeSendAction`, applies `maybeApplyTtsToPayload` to the default send payload, syncs transformed text into send params, and passes the transformed payload to the core fallback.

## Best Possible Solution

Normalize TTS once before message-action dispatch or gateway handoff so gateway, prepared, plugin-action, and core fallback sends all receive the same final-visible TTS payload, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Current-main source shows `message(action=send)` reaches outbound send routing without TTS normalization while final delivery uses the existing TTS normalizer; I did not run a live provider repro in this read-only review.

## Solution Assessment

No. This PR is not the best fix as written because the insertion point misses gateway actions, `kind: "tool"` suppresses synthesis under the default TTS mode, and synthesized media is not propagated into plugin-action params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Normalize before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263-271`
  - body: This TTS block only runs inside `executeSendAction`, but `handleSendAction` returns from the gateway-action path before calling it when a channel resolves actions to `"gateway"`. Telegram send actions do that, so gateway-mode sends still forward raw TTS tags and no synthesized media.
  - confidence: 0.91
- **[P2] Use final-visible TTS semantics for message sends:** `src/infra/outbound/outbound-send-service.ts:276`
  - body: Passing `kind: "tool"` means the default `messages.tts.mode: "final"` path returns before synthesis. With default config this can strip directives but still will not attach audio for visible `message(action=send)` replies.
  - confidence: 0.88
- **[P2] Copy synthesized media into plugin-action params:** `src/infra/outbound/outbound-send-service.ts:280-283`
  - body: The legacy plugin-action branch reads `params.ctx.params`, but the patch only syncs transformed `message`. When synthesis succeeds, `mediaUrl`, `mediaUrls`, and `audioAsVoice` remain only on `defaultPayload`, so plugin-action sends can still lose the generated audio.
  - confidence: 0.86
