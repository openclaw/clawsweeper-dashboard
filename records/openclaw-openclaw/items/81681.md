## Summary

Keep open: the linked bug is valid, but this PR still has blocking correctness gaps and the supplied proof does not exercise an after-fix message-tool send. The branch does not cover gateway-mode sends, default TTS mode synthesis, or synthesized media propagation into legacy plugin-action params.

## What This Changes

The branch adds a lazy TTS gate/runtime call inside `executeSendAction`, syncs transformed text into send params, and routes the transformed payload through prepared/core fallback sends.

## Best Possible Solution

Move TTS normalization to the shared message-action payload construction or gateway handoff so prepared, plugin-action, core fallback, and gateway sends all receive one transformed payload, then add focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main routes `message(action=send)` through outbound send handling without a TTS hook, while final reply delivery invokes the TTS payload path. I did not run a live provider repro in this read-only review.

## Solution Assessment

No. The PR is not the best fix as written because the insertion point misses gateway actions, uses a non-final TTS kind that suppresses synthesis under the default mode, and does not pass synthesized media into plugin-action params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply TTS before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263-271`
  - body: This block only runs inside `executeSendAction`, but `handleSendAction` returns from `runGatewayPluginMessageActionOrNull` before calling it when a channel action resolves to gateway mode. Telegram send actions do that, so those `message(action=send)` calls still forward raw `[[tts:text]]` tags and no synthesized media.
  - confidence: 0.91
- **[P2] Use a TTS kind that can synthesize visible sends:** `src/infra/outbound/outbound-send-service.ts:276`
  - body: Passing `kind: "tool"` keeps the default `messages.tts.mode: "final"` path from generating audio because `maybeApplyTtsToPayload` returns before synthesis for non-final kinds. With default config this strips directives but still does not attach audio, which does not match final-reply parity.
  - confidence: 0.88
- **[P2] Pass synthesized media into plugin action params:** `src/infra/outbound/outbound-send-service.ts:280-283`
  - body: The legacy plugin-action branch reads `params.ctx.params`, but after TTS this patch only copies back `message`. When synthesis succeeds, `mediaUrl`, `mediaUrls`, and `audioAsVoice` remain only on `defaultPayload`, so channels without `prepareSendPayload` can still send text without the generated audio.
  - confidence: 0.86
