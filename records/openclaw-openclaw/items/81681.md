## Summary

Keep this PR open: it targets the right TTS gap, but the patch is incomplete because the transformed payload does not reach fallback core sends or gateway/plugin-handled sends. It also needs real behavior proof beyond Vitest output before merge.

## What This Changes

The PR lazy-loads the TTS runtime in `executeSendAction` and applies `maybeApplyTtsToPayload` to the default send payload before prepared plugin send handling.

## Best Possible Solution

Apply existing TTS normalization at a shared outbound send boundary, or explicitly plumb the transformed payload through every gateway, plugin, prepared-payload, and core fallback branch, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes. Source inspection shows the current message-tool send path lacks a complete TTS hook, and the PR still leaves channels without `prepareSendPayload` plus gateway/plugin actions able to send raw directives; I did not run a live provider reproduction.

## Solution Assessment

No. The current patch is not the best complete fix because it mutates `defaultPayload` only before prepared-payload handling instead of ensuring the transformed payload reaches every send branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route the TTS payload through every send branch:** `src/infra/outbound/outbound-send-service.ts:273`

That leaves message-tool sends able to post `[[tts:text]]` literally.
