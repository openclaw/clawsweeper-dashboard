## Summary

Keep open: the linked bug is valid and this PR targets the right area, but the patch only transforms the prepared-payload branch and leaves other message-tool send branches able to forward raw TTS directives.

## What This Changes

This PR lazy-loads the TTS runtime in `executeSendAction` and applies `maybeApplyTtsToPayload` to the default message payload before prepared plugin send payload handling.

## Best Possible Solution

Apply the TTS transform once before send branch selection, or plumb the transformed payload through every gateway, plugin, prepared-payload, and core fallback branch, then add focused regression coverage plus redacted live behavior proof.

## Reproduction Assessment

Yes, from source inspection: current `executeSendAction` has no TTS call and the PR diff still does not carry the transformed payload into every send branch. I did not run a live TTS provider/channel reproduction.

## Solution Assessment

No. The PR is not the best complete fix because it mutates `defaultPayload` only where `prepareSendPayload` consumes it; the safer boundary is before gateway/plugin/core branch selection or explicit transformed-payload plumbing through each branch.
