## Summary

Keep this PR open: it targets a valid TTS bug, but the patch still runs TTS only inside `executeSendAction`, so gateway plugin sends and local plugin action sends can still bypass the transformed payload. The supplied proof also does not exercise a real `message(action=send)` TTS send.

## What This Changes

The PR lazy-loads existing TTS helpers in `executeSendAction`, applies `maybeApplyTtsToPayload` to the default outbound payload, and passes that transformed payload to prepared and core fallback sends.

## Best Possible Solution

Apply TTS once before gateway/plugin/core send branch selection, or explicitly plumb the transformed payload through every branch, then add focused regression coverage and redacted live behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main has no TTS normalization in the message-tool send path while final delivery does apply it, and the PR still leaves gateway/plugin action branches able to bypass the new transform.

## Solution Assessment

No. The patch is a good start for prepared and core fallback sends, but it is not the best complete fix because the transform remains below branch selection and does not cover Telegram gateway sends or local plugin action sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route TTS before gateway and plugin send branches:** `src/infra/outbound/outbound-send-service.ts:263`
  - body: The new TTS normalization runs inside `executeSendAction`, but `handleSendAction` can return from `runGatewayPluginMessageActionOrNull` before this function is called, and the local plugin-action fallback still dispatches the original `params.ctx.params`. Telegram gateway sends and iMessage action sends can therefore still post raw `[[tts:text]]` directives. Move the transform before branch selection or plumb the transformed payload into those branches too.
