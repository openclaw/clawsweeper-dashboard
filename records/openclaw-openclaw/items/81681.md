## Summary

Keep open: the linked TTS bug is valid, but this PR still misses gateway-mode sends, uses a TTS kind that does not synthesize audio under the default mode, and does not propagate synthesized media into legacy plugin-action params. The supplied proof is also insufficient for the real message-tool send behavior.

## What This Changes

The PR lazily loads the TTS runtime inside `executeSendAction`, applies `maybeApplyTtsToPayload` to send payloads, syncs transformed text into action params, and routes the transformed payload through prepared/core fallback sends.

## Best Possible Solution

Move TTS normalization to the shared message-action payload or gateway handoff before dispatch so direct, prepared, plugin-action, and gateway sends receive one final-visible transformed payload with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main's message-tool send path builds and dispatches outbound payloads without TTS normalization, while final-reply delivery does call the TTS normalizer; I did not run a live provider in this read-only review.

## Solution Assessment

No. The PR's `executeSendAction`-only patch is not the best fix because it misses gateway dispatch, uses non-final TTS kind under the default final mode, and syncs only text rather than the full synthesized payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply TTS before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263-271`
  - body: This transformation only runs inside `executeSendAction`, but `handleSendAction` returns from `runGatewayPluginMessageActionOrNull` before that point for gateway-mode channels. Telegram uses gateway mode, so those sends still forward raw `[[tts:...]]` params with no synthesized media.
  - confidence: 0.91
- **[P2] Use a kind that can synthesize visible sends:** `src/infra/outbound/outbound-send-service.ts:276`
  - body: Passing `kind: "tool"` keeps the default `messages.tts.mode: "final"` path from generating audio because the TTS runtime returns before synthesis for non-final kinds. With default config this can strip directives but still not attach audio.
  - confidence: 0.88
- **[P2] Pass synthesized media into plugin action params:** `src/infra/outbound/outbound-send-service.ts:280-283`
  - body: The legacy plugin-action branch dispatches `params.ctx.params`, but the patch only copies back `message`. When synthesis succeeds, `mediaUrl`, `mediaUrls`, and `audioAsVoice` stay off the params read by plugin actions, so those channels can still send text without audio.
  - confidence: 0.86
