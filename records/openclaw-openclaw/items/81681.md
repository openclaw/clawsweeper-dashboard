## Summary

Keep this PR open, but it is not merge-ready. The underlying bug is source-reproducible on current main, and the PR moves in the right area, but the patch still misses gateway sends, default TTS mode semantics, synthesized media propagation for plugin actions, and after-fix real behavior proof.

## What This Changes

The PR adds a lazy TTS runtime call inside `executeSendAction`, rewrites the default send payload, syncs transformed text back into params, and passes the transformed payload into core fallback sends.

## Best Possible Solution

Move TTS normalization to the shared message-action payload construction or gateway handoff so direct, prepared, plugin-action, and gateway sends all receive the same transformed payload, with visible message-tool sends treated as final-equivalent for default TTS semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `message(action=send)` reaches outbound send paths without a TTS hook, while final reply delivery uses the existing TTS normalizer; I did not run a live provider repro in this read-only review.

## Solution Assessment

No. The PR's insertion point is too late for gateway actions, uses a non-final kind that suppresses default synthesis, and does not propagate synthesized media into legacy plugin-action params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply TTS before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263-271`
  - body: This TTS block only runs inside `executeSendAction`, but `handleSendAction` returns from `runGatewayPluginMessageActionOrNull` before that for gateway-mode sends. Telegram uses gateway mode for message actions, so a visible Telegram `message(action=send)` still forwards the original params with raw TTS tags and no synthesized media.
  - confidence: 0.91
- **[P2] Use a final-equivalent TTS kind for visible sends:** `src/infra/outbound/outbound-send-service.ts:276`
  - body: Passing `kind: "tool"` keeps the default `messages.tts.mode: "final"` path from generating audio because `maybeApplyTtsToPayload` returns before synthesis for non-final kinds. With default config this strips directives but still does not attach audio for the visible send path this PR is trying to fix.
  - confidence: 0.88
- **[P2] Pass synthesized media into plugin action params:** `src/infra/outbound/outbound-send-service.ts:280-283`
  - body: The legacy plugin-action branch reads `params.ctx.params`, but after TTS this patch copies back only `message`. When synthesis succeeds, `mediaUrl`, `mediaUrls`, and `audioAsVoice` remain only on `defaultPayload`, so channels without `prepareSendPayload` can still send text without the generated audio.
  - confidence: 0.86
