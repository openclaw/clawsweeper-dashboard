## Summary

This PR should stay open but not merge as-is: it targets a real source-proven TTS bug, but the new normalization runs after the gateway dispatch branch and text-only sync still drops generated audio for local plugin-action sends; the supplied proof does not exercise the real message-tool path.

## What This Changes

The PR lazy-loads TTS helpers in `outbound-send-service`, applies `maybeApplyTtsToPayload` inside `executeSendAction`, syncs transformed text back into params, and passes the transformed payload to core fallback sends.

## Best Possible Solution

Apply TTS normalization once before gateway/plugin/core send branch selection and plumb the complete transformed payload, including text and generated audio fields, into every branch with focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: gateway-backed sends return before `executeSendAction`, and local plugin-action sends read media fields from raw params that the PR does not update.

## Solution Assessment

No. The current PR is a useful start, but the narrow maintainable fix is to normalize before branch selection and propagate the full transformed payload rather than adding a partial transform inside only `executeSendAction`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply TTS before gateway action dispatch:** `src/infra/outbound/outbound-send-service.ts:263`
  - body: The new TTS normalization runs inside `executeSendAction`, but `handleSendAction` returns from `runGatewayPluginMessageActionOrNull` before this function is called. Telegram resolves sends to gateway mode, so visible Telegram sends can still post raw `[[tts:text]]` directives with no audio. Move normalization before gateway dispatch and pass the transformed payload into gateway action params too.
  - confidence: 0.92
- **[P2] Pass TTS audio fields into plugin action params:** `src/infra/outbound/outbound-send-service.ts:280-284`
  - body: The follow-up only copies transformed text back to `params.ctx.params.message`. When TTS succeeds, `maybeApplyTtsToPayload` returns audio fields such as `mediaUrl` and `audioAsVoice`; local plugin-action sends like Matrix read those fields from `params`, so they would strip the tag text but still send no generated audio. Sync the full transformed payload or route these sends through the prepared payload path.
  - confidence: 0.86
