## Summary

Keep open. Current main still has the reported cron scheduler asymmetry: one-shot jobs classify transient errors and schedule short retries, while recurring cron/every errors only apply a backoff lower bound and still choose the later natural schedule. The issue is also paired with open PR #46423, so it should stay open until that PR or a focused replacement lands.

## What This Changes

Recurring cron/every jobs should retry transient execution errors with backoff instead of waiting until the next natural schedule.

## Best Possible Solution

Land a focused scheduler change, likely by reviewing/refining PR #46423 or replacing it with a narrower PR, that classifies recurring cron/every execution errors, schedules retryable transient failures on the short backoff path, preserves natural-schedule behavior for permanent errors, and updates regression tests plus docs/type comments to match the final contract.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: a daily cron or every job that returns a transient error such as a network timeout enters the recurring error branch, computes endedAt plus backoff, and then selects the later natural schedule instead of retrying soon.

## Solution Assessment

Yes. The narrow maintainable direction is to apply transient classification in the recurring error path while keeping permanent-error natural scheduling; this issue should not close before the linked implementation PR merges or a focused replacement lands.
