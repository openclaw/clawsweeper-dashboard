## Summary

Keep open. Current main still has the reported scheduler asymmetry: one-shot jobs classify transient errors and schedule short retries, while recurring cron/every errors still choose the later natural schedule/backoff floor and do not call the transient classifier; the issue is also paired with open closing PR #46423, so it should stay open until that PR or a focused replacement lands.

## What This Changes

Recurring cron/every jobs should retry transient execution errors on short backoff instead of waiting for the next natural schedule.

## Best Possible Solution

Land a focused scheduler change, likely by reviewing or replacing PR #46423, that applies transient classification in the recurring error path, keeps permanent errors on the natural-schedule/backoff-floor behavior, and updates regression tests plus public retry-contract text.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: make a recurring daily cron or long-period every job return a transient error such as a timeout, and current main will compute a short backoff but select the later natural next run instead of retrying soon.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing transient classifier in the recurring error branch while preserving current permanent-error scheduling semantics; closing before the linked implementation merges would be premature.
