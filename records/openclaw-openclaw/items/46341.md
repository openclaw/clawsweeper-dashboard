## Summary

Keep open. Current main still schedules recurring cron errors at the later of the natural next run and the backoff floor, so long-period jobs can miss the documented short retry path; the related direct-fix PR was closed unmerged as a dirty branch, and the remaining fix needs a cron retry policy decision.

## What This Changes

This issue requests recurring cron/every jobs to retry transient failures with backoff instead of waiting for the next natural schedule.

## Best Possible Solution

Adopt an explicit recurring error retry contract, then update `applyJobResult`, reload/recompute behavior, docs/types, and regression tests together.

## Reproduction Assessment

Yes, by source inspection. The recurring error path on current main uses the later natural schedule/backoff lower bound, so a daily network-style failure can remain scheduled for the next day instead of the documented short retry ladder.

## Solution Assessment

Unclear. The suggested transient-only branch is plausible, but the best fix depends on maintainer alignment between generic recurring backoff docs and the one-shot-only `cron.retry` config contract.
