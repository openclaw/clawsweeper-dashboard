## Summary

Keep this open. Current main still routes recurring cron errors through the natural-schedule lower-bound logic, but the public config/docs currently scope transient retry policy to one-shot jobs, so this needs a maintainer policy call rather than an automatic cleanup close.

## What This Changes

Recurring cron and every jobs are requested to retry transient failures with backoff instead of waiting for the next natural schedule.

## Best Possible Solution

Define the recurring-job transient retry policy explicitly, then update `applyJobResult`, regression tests, and cron retry docs together if maintainers want recurring jobs to retry before their next natural schedule.

## Reproduction Assessment

Yes, by source inspection. The recurring error path currently uses the later of `normalNext` and `backoffNext`, so a daily job with a network-style error can remain scheduled for the next day instead of the short backoff.

## Solution Assessment

Unclear. The suggested transient branch is a plausible narrow implementation, but the best fix needs maintainer confirmation on whether recurring jobs should share one-shot `cron.retry` semantics or have a separate policy.
