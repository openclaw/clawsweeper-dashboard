## Summary

Keep open: current main still has the reported Crestodian routing gap. The media tools and `/tts` command exist in the normal agent/gateway path, but Crestodian remains a setup/repair helper with a closed non-media command vocabulary, so this needs a maintainer product decision rather than cleanup closure.

## What This Changes

The issue asks for Crestodian/REPL to route configured xAI image generation and TTS requests through the available media tools instead of interpreting them as setup/plugin/overview commands.

## Best Possible Solution

Define a generic Crestodian media/tool routing contract that reuses the existing agent/gateway media surfaces, or explicitly keep Crestodian as setup-only and make the normal agent TUI handoff the supported media path.

## Reproduction Assessment

Yes. Current-main source shows Crestodian only resolves a closed setup/repair/admin command set and its planner is constrained to that same non-media vocabulary; I did not run the live xAI OAuth flow.

## Solution Assessment

Unclear. Exposing media in Crestodian may be useful, but the best fix needs maintainer direction on direct tool invocation versus a normal-agent-TUI handoff, and it should not be xAI-specific.
