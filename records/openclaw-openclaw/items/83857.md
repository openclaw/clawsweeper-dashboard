## Summary

Keep open: current main still lacks a Crestodian media-routing contract, and the existing source/docs frame Crestodian as a typed setup and repair helper rather than a normal agent tool runner. The reporter's xAI-specific symptoms are source-backed, but the fix requires maintainer product/API judgment before a repair lane should choose the command surface.

## What This Changes

The issue asks for Crestodian/REPL to route xAI image-generation and TTS requests to the available media tools/providers when the same xAI setup works through `openclaw infer`.

## Best Possible Solution

Define a generic Crestodian media-routing contract, or explicitly document that users should switch to the normal agent TUI for media tools; avoid xAI-specific routing in Crestodian.

## Reproduction Assessment

Yes from source inspection: Crestodian messages are resolved through the typed operation parser and planner allow-list, and neither path includes image-generation or TTS operations. I did not run the reporter's live xAI OAuth setup.

## Solution Assessment

Unclear as filed: exposing media actions in Crestodian may be useful, but the best solution needs a maintainer decision on whether Crestodian should stay setup/repair-only or reuse a generic gateway/agent tool contract for media execution.
