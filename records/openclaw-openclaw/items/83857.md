## Summary

Keep open: current main still routes Crestodian input through the constrained setup/repair command planner, while the requested image/TTS behavior would expand Crestodian into normal agent tool execution or require an explicit handoff policy.

## What This Changes

The issue requests that Crestodian/REPL route xAI image generation and TTS requests to the same enabled tools/providers that work through `openclaw infer`.

## Best Possible Solution

Decide the Crestodian boundary explicitly: either teach Crestodian to hand `/tool` and `/tts` media requests into the normal agent TUI with a clear message, or approve and design a narrow safe media/TTS command bridge.

## Reproduction Assessment

Yes from source inspection: current main sends Crestodian input through a typed command parser and planner allowlist that do not include `image_generate`, `/tool`, or `/tts`. I did not run a live xAI Crestodian session.

## Solution Assessment

Unclear as a direct implementation: routing arbitrary media tools through Crestodian is not the narrowest safe fix without a maintainer boundary decision. A safer first direction is explicit handoff to the normal agent TUI or a documented refusal unless maintainers approve direct tool bridging.
