## Summary

Keep open. Current main and v2026.5.22 have useful partial mitigations, including root optional sharp metadata and bounded original-image fallback, but they do not solve the Android/Termux sharp gap because sharp 0.34.5 still has no android-arm64 prebuilt and OpenClaw has no Termux-native post-update rebuild or supported Gateway-hosting policy.

## What This Changes

The issue requests Android/Termux-safe handling for sharp native-module failures after OpenClaw updates, via rebuild tooling, a repair command, or graceful image fallback behavior.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, make image handling survive updates through a documented system backend path, Termux-native sharp repair in doctor/update, or a narrow original-image fallback for inputs proven safe without sharp.

## Reproduction Assessment

Yes, source-reproducible. Current code still reaches sharp or system image backends for resize-required and HEIC inputs, and sharp 0.34.5 has no android-arm64 prebuilt; I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. The current root optional dependency and fallback chain are good partial mitigations, but the durable update/rebuild fix depends on whether maintainers want Android/Termux to be a supported Gateway host.
