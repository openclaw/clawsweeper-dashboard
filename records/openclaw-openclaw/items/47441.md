## Summary

Keep open. Current main and v2026.5.22 shipped important mitigations for the generic Sharp packaging problem, but the Android/Termux Gateway case remains unresolved: Sharp still has no android-arm64 optional package, OpenClaw has no Termux-native rebuild/update/doctor repair path, and the system-tool fallback does not discover Termux's package prefix by default.

## What This Changes

Request support for Android/Termux Gateway image handling when Sharp has no android-arm64 prebuilt, via Termux repair, a native-module repair command, or safe raw-image fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, add a documented Termux-aware doctor/update repair or backend discovery path while preserving the current safe fallbacks.

## Reproduction Assessment

Yes, at source level: Sharp 0.34.5 has no android-arm64 optional package and current OpenClaw has no Termux rebuild or repair path. I did not run a live Termux device, and current fallbacks narrow the failure to cases without a usable backend or where resize/HEIC/size policy still requires processing.

## Solution Assessment

Unclear. The current root optional dependency and bounded fallback are good mitigations, but the durable fix depends on whether maintainers want Android/Termux to be a supported Gateway host and where native-module repair belongs.
