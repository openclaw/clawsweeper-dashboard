## Summary

Keep open. Current main and v2026.5.20 have useful mitigations for generic missing sharp and in-limit original-image pass-through, but the Android/Termux failure remains reachable because sharp 0.34.5 has no android-arm64 prebuilt while OpenClaw still needs an image backend for resize-required, HEIC, and over-cap media.

## What This Changes

The issue asks OpenClaw to handle Android/Termux sharp native-module failures after updates, either through Termux-aware rebuild/repair or graceful image fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, make image handling survive updates through a documented system backend, Termux-native sharp rebuild path, or explicit doctor/update repair while preserving bounded safe original-image fallbacks.

## Reproduction Assessment

Yes, source-reproducible. sharp 0.34.5 resolves Android as android-arm64 and has no matching prebuilt package, while OpenClaw still reaches sharp or system image backends for resize, HEIC conversion, and over-cap images; I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. Current main's root optional dependency, system-backend chain, and in-limit original pass-through are useful partial mitigations, but the durable fix depends on whether maintainers want to support Android/Termux as a Gateway host.
