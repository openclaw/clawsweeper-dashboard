## Summary

Keep open. Current main and the latest tagged release still have reachable image paths that require the sharp-backed media-understanding runtime when Android/Termux lacks a usable sharp/libvips install, and the repository does not yet provide either a bounded backend-unavailable fallback or Termux-aware native-module repair. The prior ClawSweeper comments are consistent with the current source: this remains a focused image-ingestion bug rather than a stale or duplicate report.

## What This Changes

Android/Termux users report that the sharp-backed image tool fails after updates because sharp resolves the runtime as android-arm64, has no matching prebuilt/libvips package, and source rebuild expects an unavailable NDK cross-compiler.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion and optimization: preserve the original only when MIME type, byte size, and header-derived dimensions prove the image is already within provider limits; otherwise omit it or return a clear backend-unavailable error. Keep automatic Termux native rebuild tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: force the bundled sharp import/load to fail, then process a non-GIF web/local image with default optimization, a resize-required managed image block, a HEIC input, or a PNG alpha resize path. The exact Android/Termux device setup was not live-tested here.

## Solution Assessment

Partly: the graceful fallback direction is the narrowest maintainable first fix for the user-visible image failure. The suggested post-update rebuild or fix-native-modules command is broader native packaging work and should not be bundled into an automated repair without maintainer approval.
