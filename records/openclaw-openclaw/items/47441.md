## Summary

Keep open. Current main and v2026.5.18 include important mitigations, including root `sharp` metadata, external image-tool fallbacks, and in-limit original-image pass-through, but Sharp 0.34.5 still has no `android-arm64` prebuilt and OpenClaw still loads Sharp first, so Android/Termux hosts can still fail after updates unless a system backend or manual rebuild is present.

## What This Changes

The issue asks OpenClaw to avoid post-update Android/Termux image-tool failures caused by Sharp lacking an `android-arm64` prebuilt and falling into a Termux-incompatible native build.

## Best Possible Solution

Maintainers should decide the Android/Termux Gateway support boundary; if supported, document and test the required image backend or Termux repair path while preserving bounded original-image fallbacks.

## Reproduction Assessment

Yes, source-reproducible: Sharp 0.34.5 has no Android prebuilt target while OpenClaw still reaches Sharp-first image processing and only partially falls back. I did not run the reporter's Android/Termux setup live.

## Solution Assessment

Unclear. Current main's root dependency metadata and bounded fallbacks are useful partial fixes, but the durable answer requires a maintainer choice between supported Termux repair, documented system-backend requirements, or declaring Android Gateway hosting unsupported.
