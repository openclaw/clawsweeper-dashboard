## Summary

Keep open. Current main still ships sharp as the non-macOS image-processing backend, still has reachable resize/optimization paths that throw when that backend cannot load, and update only retries npm installs with `--omit=optional` rather than repairing Android/Termux native-module toolchains.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after update, either through rebuild tooling, a repair command, or graceful image fallback.

## Best Possible Solution

Add a bounded image fallback that preserves already-safe JPEG/PNG/GIF/WebP inputs when sharp is unavailable and emits clear omissions/errors for inputs that require resize or conversion; leave Termux native rebuild automation as separate packaging work.

## Reproduction Assessment

Yes. Force the bundled sharp import/load to fail, then call loadWebMedia on an optimizable non-GIF image or exercise resizeToJpeg, HEIC conversion, or managed-image resize; current main routes those paths to the sharp-backed backend and throws.

## Solution Assessment

Partly. The graceful fallback is the narrowest maintainable first fix for the user-visible image failure, while post-update Termux rebuild tooling should remain a separate maintainer-scoped packaging decision.
