## Summary

Keep open. Current main and v2026.5.22 include useful Sharp packaging and bounded fallback mitigations, but the reported Android/Termux update failure is not fully solved because Sharp 0.34.5 still has no android-arm64 prebuilt package and OpenClaw has no Termux-aware native-module repair policy.

## What This Changes

The issue asks for a durable fix for Android/Termux Sharp native-module failures after OpenClaw updates, via rebuild tooling, a repair command, or graceful image fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, add a documented doctor/update or generic backend fallback path that preserves image size, dimension, and decoder safety checks.

## Reproduction Assessment

No live Android/Termux current-main reproduction was run. Source-level reproduction is credible because Sharp 0.34.5 lacks android-arm64 packages, current resize/HEIC paths still need a usable backend, and the update path can omit optional native dependencies after failure.

## Solution Assessment

Unclear. The shipped root optional dependency and bounded raw-image fallback are useful mitigations, but the durable fix depends on whether maintainers want OpenClaw update or doctor to own Termux native-module repair.
