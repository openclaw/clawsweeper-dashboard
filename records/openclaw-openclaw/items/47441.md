## Summary

Keep open. Current main has a useful in-limit image fallback, but the Android/Termux native sharp failure is not fully fixed: sharp remains the non-macOS backend, sharp 0.34.5 has no Android prebuilt package, update fallback can omit optional deps, and images that need conversion or resizing still depend on the sharp-backed path.

## What This Changes

The issue asks OpenClaw to stop breaking image handling on Android/Termux after updates when sharp/libvips cannot load or build.

## Best Possible Solution

Keep the current in-limit fallback, review the supported-platform packaging fix, and separately decide an Android/Termux native-dependency strategy or explicit backend-unavailable behavior for resize/convert-required images.

## Reproduction Assessment

Yes by source inspection: on a platform where sharp import/load fails, current main still throws through resizeToJpeg, HEIC conversion, and over-cap loadWebMedia paths. I did not run a live Android/Termux device repro, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. The in-limit fallback is a good narrow mitigation, but the remaining Android/Termux update failure needs a maintainer decision on native dependency packaging and rebuild policy; the open root optional-dependency PR only addresses part of the broader class.
