## Summary

Keep open. Current main and v2026.5.22 have partial mitigations for missing Sharp, but they still do not solve the Android/Termux-specific problem: Sharp 0.34.5 has no android-arm64 prebuilt, OpenClaw has no Termux-native post-update rebuild or repair command, and the public Android docs still describe Android as a companion node rather than a Gateway host.

## What This Changes

The issue asks for Android/Termux image handling to survive `openclaw update` despite Sharp lacking an android-arm64 prebuilt, via a Termux-native rebuild, native-module repair command, or graceful fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, add a documented system-backend or doctor/update repair path for Termux-native image processing while preserving the existing safe original-image fallback.

## Reproduction Assessment

Yes, source-reproducible. Current main still depends on Sharp or external image backends for resize/HEIC paths, and Sharp 0.34.5 has no android-arm64 prebuilt; I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. The current root optional dependency, system-backend fallback, and bounded original-image fallback are good mitigations, but a durable Termux rebuild/update command depends on whether maintainers want Android/Termux to be a supported Gateway host.
