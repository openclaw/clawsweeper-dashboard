Keep open. Current main adds important mitigations, including root Sharp metadata and system image-tool fallbacks, but the Android/Termux Sharp prebuilt gap remains and the durable answer still needs a maintainer decision on whether Termux Gateway hosting is supported or should be documented as unsupported.

## What This Changes

Request to make image processing survive Android/Termux updates where Sharp has no android-arm64 prebuilt and source builds assume an unavailable NDK cross-compiler.

## Best Possible Solution

Maintainers should decide the Android/Termux Gateway support boundary; if supported, add documented and tested image backend requirements or Termux native-module repair while preserving safe original-image fallbacks.

## Reproduction Assessment

Yes, source-reproducible: Sharp 0.34.5 has no android-arm64 optional binary package while OpenClaw still imports Sharp first and falls back only to installed system tools or under-cap originals. I did not run the reporter's Android/Termux setup live.

## Solution Assessment

Unclear. Current main's Sharp-first system-tool fallback is a useful partial mitigation, but the report's post-update Termux repair and Android Gateway support boundary need maintainer judgment.
