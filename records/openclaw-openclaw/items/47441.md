## Summary

Keep open. Current main still uses sharp as the non-macOS image-processing backend, reachable image optimization and resize paths still throw when sharp/libvips cannot load, and the update path only retries generic optional dependency omission rather than providing Termux-aware repair.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after updates, including post-install repair, a fix command, or graceful image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in image ingestion that preserves raw, already-safe images and emits a clear omission/error for inputs that need resize or conversion; decide Termux rebuild automation separately.

## Reproduction Assessment

Yes. Force the bundled sharp import/load to fail, then call loadWebMedia on an optimizable non-GIF image or exercise resizeToJpeg, HEIC conversion, or managed-image resize; current main routes those paths to the sharp-backed backend and throws.

## Solution Assessment

Partly. The graceful fallback is the narrowest maintainable first fix for the user-visible image failure, while post-update Termux rebuild tooling should remain a separate packaging decision.
