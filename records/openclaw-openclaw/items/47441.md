## Summary

Keep open. Current main has partial mitigations, but the Android/Termux sharp failure remains reachable for resize-required, HEIC, and over-cap media because sharp 0.34.5 has no android-arm64 package and OpenClaw still needs an image backend; the durable fix depends on the Android/Termux Gateway support boundary.

## What This Changes

Requests Android/Termux-safe image handling after updates when sharp cannot load, via rebuild tooling, a fix-native-modules command, or graceful raw-image fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, make image handling survive updates through a documented system backend, Termux-native sharp rebuild path, or explicit doctor/update repair while keeping bounded original-image fallbacks.

## Reproduction Assessment

Yes, source-reproducible. sharp 0.34.5 has no android-arm64 optional package while OpenClaw still reaches sharp-backed or system image backends for resize, HEIC conversion, and over-cap images; I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. Current main's original-image fallback and system backend chain are useful partial mitigations, but a durable update/rebuild fix depends on whether maintainers want Android/Termux to be a supported Gateway host.
