## Summary

Keep open. Current main and v2026.5.22 include useful Sharp packaging, external-backend, and bounded raw-image fallback mitigations, but they do not fully resolve the reported Android/Termux update failure because Sharp still has no android-arm64 prebuilt and OpenClaw still lacks a Termux-aware native dependency repair policy.

## What This Changes

The issue asks OpenClaw to stop recurring Android/Termux Sharp image-module failures after updates, either through update-time native rebuild repair, a repair command, or graceful image fallback.

## Best Possible Solution

Decide the Android/Termux Gateway support boundary; if supported, add a documented Termux-aware native image dependency repair path in update or doctor while preserving the current bounded media fallbacks and clear backend-unavailable errors.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source and dependency evidence make the failure credible because Sharp 0.34.5 has no android-arm64 package and current OpenClaw still needs a usable backend for resize-required, HEIC, or over-cap image paths.

## Solution Assessment

Unclear. The shipped optional dependency, external backend chain, and bounded raw-image fallback are good mitigations, but the durable fix depends on whether maintainers want update or doctor to own Termux native-module repair.
