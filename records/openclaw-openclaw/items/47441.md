## Summary

Keep open. Current main and the latest release still have reachable non-macOS image paths that load sharp, sharp 0.34.5 still has no Android optional binary packages in the lockfile, and the existing fallback only covers already-under-cap non-HEIC images rather than Android/Termux update or native-module repair.

## What This Changes

The issue requests durable handling for sharp image-processing failures on Android/Termux after OpenClaw updates.

## Best Possible Solution

Maintainers should decide the Android/Termux Gateway support boundary; if supported, add tested native-module repair or replace sharp with a backend that supports the platform while keeping safe raw-image fallbacks for already-valid inputs.

## Reproduction Assessment

Yes, by source and dependency-contract inspection: current non-sips image resize/HEIC paths still load sharp, and sharp 0.34.5 has no android-arm64 optional binary packages. I did not run the reporter's Android/Termux setup live.

## Solution Assessment

Unclear. The bounded safe-original fallback is useful but partial; fully resolving this needs a maintainer choice between supported Termux native rebuilds, a different image backend, or documented unsupported-platform behavior.
