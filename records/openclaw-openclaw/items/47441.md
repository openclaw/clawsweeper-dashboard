## Summary

Keep open. Current main still has reachable image-processing paths that hard-require the sharp-backed media-understanding runtime when Android/Termux lacks a usable sharp/libvips install, while the update path only retries generic optional-dependency installs and does not provide Termux-aware repair.

## What This Changes

The issue requests Android/Termux-safe handling for sharp native module failures after updates, either through rebuild tooling or a graceful image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion/optimization, and keep automatic Termux native rebuild tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: force the bundled sharp import/load to fail, then process a non-GIF optimized image, a HEIC input, or a managed image that requires resize. The exact Android/Termux device setup was not live-tested in this read-only review.

## Solution Assessment

Partly: the graceful fallback direction is the narrowest maintainable first fix for the user-visible failure. The post-update rebuild and fix-native-modules ideas are broader native packaging work and should not be bundled into the first automated repair.
