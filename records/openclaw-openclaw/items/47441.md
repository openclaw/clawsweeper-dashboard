## Summary

Keep open. Current main and the latest release still have a source-reachable missing/broken Sharp path for Android/Termux: Sharp remains the resize/HEIC backend, the root package still does not declare Sharp, the Sharp package set has no Android ARM64 prebuilt in the lockfile, and the remaining decision is broader than a simple cleanup close.

## What This Changes

The issue asks OpenClaw to survive Android/Termux Sharp native-module failures after updates, either by repairing native modules or by safely falling back to raw images.

## Best Possible Solution

Keep this open until maintainers choose the Android/Termux stance, while separately landing or superseding the root Sharp package fix and preserving a bounded fallback for already-safe images.

## Reproduction Assessment

Yes, by source and dependency inspection rather than live Android hardware: Sharp is still required for resize/HEIC paths and the locked Sharp package set has no Android ARM64 prebuilt. The exact Termux node-gyp compiler failure was not locally reproduced.

## Solution Assessment

Unclear. The existing in-limit fallback is a useful mitigation, but a complete fix needs a maintainer decision between root package metadata, Android/Termux native rebuild support, alternate image backend behavior, or documenting Android Gateway as unsupported.
