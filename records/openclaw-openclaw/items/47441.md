## Summary

Keep open. Current main and v2026.5.22 include useful mitigations, including root `sharp` optional metadata and bounded original-image fallback, but they do not resolve the reported Android/Termux sharp native-module gap or define whether Android/Termux is a supported Gateway host.

## What This Changes

The issue asks OpenClaw to survive Android/Termux `sharp` failures after updates, either through Termux-aware native rebuild repair, a native-module utility, or safer image fallback behavior.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, add a documented doctor/update repair or system-backend path for native image processing while preserving the current capped original-image fallback.

## Reproduction Assessment

Yes, source-reproducible: sharp 0.34.5 has no android-arm64 prebuilt, OpenClaw still calls sharp or system image backends for resize/HEIC paths, and tests show the missing-optimizer fallback is intentionally bounded. I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. The current root optional dependency and capped original-image fallback are good mitigations, but the durable Termux rebuild/update path depends on whether maintainers want Android/Termux to be a supported Gateway host.
