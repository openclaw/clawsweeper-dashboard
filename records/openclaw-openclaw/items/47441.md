## Label Justifications

- `P2`: The issue affects a real but non-default Android/Termux Gateway image workflow and needs platform-support judgment, not emergency handling.
- `impact:message-loss`: The reported failure can suppress or omit image payloads in channel-driven media workflows when the image backend is unavailable.

## Summary

Keep open. Current main and v2026.5.19 include useful mitigations, including root `sharp` metadata, a system-backend image chain, and bounded original-image pass-through, but Android/Termux still has a reachable post-update failure path because Sharp 0.34.5 has no `android-arm64` prebuilt and OpenClaw still needs an image backend for resize-required, HEIC, and over-cap inputs.

## What This Changes

The issue asks OpenClaw to survive Android/Termux `sharp` native-module failures after updates, either through Termux-aware rebuild/repair or graceful image fallback.

## Best Possible Solution

Define the Android/Termux Gateway support boundary; if supported, make image handling survive updates through a documented system backend, Termux-native Sharp rebuild path, or explicit doctor/update repair while keeping bounded safe original-image fallbacks.

## Reproduction Assessment

Yes, source-reproducible. Sharp 0.34.5 has no Android prebuilt target while OpenClaw still reaches Sharp or external image backends for resize, HEIC conversion, and over-cap images; I did not run the reporter's Termux device live.

## Solution Assessment

Unclear. Current main's root optional dependency, system-backend fallback, and in-limit pass-through are useful partial mitigations, but the durable fix requires maintainers to choose whether OpenClaw owns Android/Termux Gateway native-module repair or documents that hosting mode as unsupported.
