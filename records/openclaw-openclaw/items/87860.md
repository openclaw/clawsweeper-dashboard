## Summary

Keep open. Current main and the latest release still have the QQ Bot outbound dispatcher swallow lane rejection or watchdog timeout without sending user feedback, and the linked closing PR is still open/conflicting rather than merged.

## What This Changes

The issue asks QQ Bot to send a brief user-facing error reply when lane dispatch rejects or the response watchdog times out before any normal reply.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/87863 with a focused QQ Bot dispatcher fix that sends one sanitized best-effort failure reply, suppresses late duplicate timeout delivery, and adds regression plus real QQ Bot proof.

## Reproduction Assessment

Yes. From source, make QQ Bot runtime.channel.inbound.run reject or let responseTimeoutMs fire before any deliver call; the current catch only clears timeoutId and sends no text.

## Solution Assessment

Yes for the issue direction: the narrow owner-boundary fix belongs in the QQ Bot outbound dispatcher. The current open PR still needs repair/proof before merge, while the broader RFC at https://github.com/openclaw/openclaw/issues/92672 should not close this channel-specific bug by itself.
