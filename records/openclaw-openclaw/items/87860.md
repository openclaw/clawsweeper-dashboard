## Summary

Keep open. Current main and v2026.5.27 still have the QQ Bot outbound dispatcher swallow the lane/timeout race rejection without calling the existing error sender, so the reported no-feedback failure path remains a narrow channel-delivery bug.

## What This Changes

QQ Bot should send a brief user-facing error reply when lane-level model, rate-limit, or timeout failures occur before a normal response.

## Best Possible Solution

Patch the QQ Bot outbound dispatcher to send one sanitized best-effort failure reply on pre-response dispatch rejection or watchdog timeout, preserving existing streaming/tool fallback behavior and adding focused regression tests.

## Reproduction Assessment

Yes for a source-level repro: stub QQ Bot `runtime.channel.inbound.run` to reject, or let `responseTimeoutMs` fire before any deliver call, and the current catch exits without sending text. I did not run a live QQ Bot account in this read-only review.

## Solution Assessment

Yes, with bounds: the reporter's suggested catch-block error reply matches the existing QQ Bot helper, but the implementation should avoid leaking raw provider details and prevent double-sends after streaming or tool fallback output.
