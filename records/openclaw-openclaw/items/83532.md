## Summary

Keep open: current main has reconnect refresh, auth-error suppression, exponential WebSocket backoff, and chat.send dedupe, but the reported Safari idle 401 plus unattended retry/billing loop is not proven fixed and has no canonical fixing PR.

## What This Changes

The issue asks for Control UI responses to arrive after Safari idle periods without manual refresh, and for auth/reconnect failures to stop retrying in a way that can replay expensive chat sends.

## Best Possible Solution

Reproduce the Safari idle 401 path, then add a bounded auth-expired retry/reconnect policy that reconciles chat history on resume without duplicate chat.send dispatch.

## Reproduction Assessment

No. The report has useful Safari idle timing, 401 console evidence, and billing-loop context, but it still lacks a deterministic current-main reproduction, WebSocket close trace, exact version, and gateway/client retry logs.

## Solution Assessment

Unclear. A session-expired/retry cap is plausible, but current main already has auth non-reconnect guards, backoff, reconnect refresh, queued-send replay, and server idempotency, so the best fix depends on proving which path fails.
