## Summary

Keep open: the recovery idea is useful and backed by production hotfix context, but the patch is not ready because it can trust stale trajectory markers and misses pointer-discoverable trajectory files.

## What This Changes

The PR adds pre-open embedded-runner recovery that archives a transcript and trajectory when the latest trajectory `session.ended` reports `non_deliverable_terminal_turn`, then starts the turn with a clean session and focused helper tests.

## Best Possible Solution

Keep the pre-open recovery approach, but resolve trajectories through the canonical pointer-aware helper, prove the marker is current before renaming, archive or refresh the pointer consistently, and add regression coverage for those cases.

## Reproduction Assessment

Yes at source level: current main records non-deliverable terminal markers and opens existing transcripts without this recovery gate. I did not run the live Feishu production replay in this read-only review.

## Solution Assessment

No. The recovery boundary before transcript replay is the right layer, but the implementation needs pointer-based trajectory discovery and stale-marker protection before it is the best fix.

  - body: This rename is driven only by the latest `session.ended` found in the trajectory file.
