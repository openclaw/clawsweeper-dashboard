## Summary

Keep open. Current main still has one shared `mcp.servers` registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume the merged MCP map before any agent-specific tool filtering can prevent startup or injection.

## What This Changes

Request optional per-MCP-server agent scoping so multi-agent gateways only start or expose MCP servers for matching agents.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config; omitted scope should preserve today’s all-agents default.

## Reproduction Assessment

Yes. Source-level reproduction is a multi-agent config with any `mcp.servers` entry: current main builds one shared server map and consumes it without an effective-agent filter before runtime startup or CLI injection.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn and pre-connect filtering; existing `tools.deny` or post-catalog filtering is not enough.
