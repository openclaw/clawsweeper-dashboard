## Summary

Keep open. Current main still lacks an effective per-agent MCP server scope, and the existing tool allow/deny path runs after shared MCP config merge/startup paths, so this is a real multi-agent resource and access-boundary gap rather than a cleanup close.

## What This Changes

Request an optional per-agent scope on configured MCP servers so multi-agent gateways only spawn or inject servers for matching agents.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, cataloged, cached, written to Pi settings, or injected into CLI backend config.

## Reproduction Assessment

Yes. A source-level reproduction is any multi-agent config with `mcp.servers`: current main builds one shared MCP server map and consumes it without an effective-agent filter before runtime startup or CLI injection.

## Solution Assessment

Unclear. The proposed `agents` field is plausible, but maintainers should first approve field shape, glob semantics, and security-boundary rules; the maintainable direction is pre-spawn/pre-connect filtering rather than `tools.deny`.

- **[medium] Define MCP scope before server startup:** `src/agents/pi-bundle-mcp-runtime.ts:228`
  - body: Per-agent MCP scoping must be applied before `resolveMcpTransport`, stdio spawn, HTTP/SSE connection, tool cataloging, fingerprinting, and runtime caching; hiding tools later would still start or connect to disallowed servers.
  - confidence: 0.92
- **[medium] Filter CLI MCP config before backend injection:** `src/agents/cli-runner/bundle-mcp.ts:175`
  - body: CLI runtime adapters currently receive the merged MCP map and inject it into downstream backend config, so a scoped server must be removed before config is written or passed to Claude, Codex, Gemini, or similar backends.
  - confidence: 0.86
