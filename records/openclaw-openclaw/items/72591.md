## Summary

Keep open. Current main still has no typed, documented, or enforced per-agent scope for `mcp.servers.*`, and the related closed duplicate points back to this as the canonical thread while adjacent MCP identity and `tools.deny` reports cover distinct work.

## What This Changes

The issue requests an optional `agents` scope on `mcp.servers.*` so MCP servers are only started or exposed for matching agents.

## Best Possible Solution

Add a backward-compatible, schema-backed MCP server scope resolved against the effective agent before configured or bundled MCP servers are spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config.

## Reproduction Assessment

Yes for the current limitation, not as a strict bug: source inspection shows current main has no enforced `mcp.servers.*.agents` contract and consumes a shared MCP server map without effective-agent filtering. I did not run a live multi-agent memory measurement.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The maintainable direction is pre-spawn and pre-injection filtering; `tools.deny` or post-catalog filtering alone does not satisfy the reported resource and guardrail goals.
