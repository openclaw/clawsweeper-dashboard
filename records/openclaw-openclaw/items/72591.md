## Summary

Keep open. Current main still has a shared `mcp.servers` registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume the merged MCP server map before any agent-specific policy could avoid startup, connection, cataloging, or backend config injection.

## What This Changes

Request an optional `mcp.servers.<name>.agents` scope to limit MCP servers to matching agents while preserving the current all-agent default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before configured or bundled MCP servers are spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config.

## Reproduction Assessment

Yes for the current limitation, not as a strict bug: source inspection shows current main builds one shared MCP server map from `mcp.servers` and consumes it before effective-agent filtering. A live multi-agent memory measurement was not run.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The maintainable direction is pre-spawn and pre-connect filtering; `tools.deny` or other post-catalog filtering is not enough for the reported resource and guardrail goals.
