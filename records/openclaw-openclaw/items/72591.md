## Summary

Keep open. Current main still treats `mcp.servers` as a global registry, and the embedded Pi, project-settings, compaction, and CLI backend paths consume the merged MCP map before any agent-specific tool policy can prevent startup or downstream config injection.

## What This Changes

Request an optional per-agent scope on `mcp.servers.*` so only matching agents spawn or receive each MCP server.

## Best Possible Solution

Add a backward-compatible MCP server scope that is resolved against the effective agent before MCP servers are spawned, connected, cataloged, cached, written to Pi settings, or injected into CLI backend config; omitted scope should preserve today’s all-agents default.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence behavior gap: `mcp.servers` is merged globally and consumed by embedded Pi and CLI paths before agent-specific tool policy can prevent startup or config injection.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The maintainable boundary is pre-spawn and pre-connect filtering; post-catalog `tools.deny` filtering does not solve the process or data-access exposure.

Status: needs_attention

Summary: No patch is under review, but the requested feature is security-sensitive because MCP scope would become a pre-startup and remote-connection boundary.

Concerns:

- **[medium] Define MCP scope before server startup:** `src/agents/pi-bundle-mcp-runtime.ts:228`
  - body: Any implementation must apply effective-agent scope before transport resolution, stdio process spawn, HTTP/SSE connection, tool cataloging, fingerprinting, and runtime caching; hiding tools later would still start or connect to disallowed MCP servers.
  - confidence: 0.92

  - body: CLI runtime adapters currently receive the merged MCP map and inject it into downstream backend config. Per-agent scoping must happen before this config is written or passed to Claude, Codex, Gemini, or similar backends.
