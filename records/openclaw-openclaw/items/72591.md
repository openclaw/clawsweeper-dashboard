## Summary

Keep open. Current main still treats `mcp.servers` as one shared registry and only applies agent tool policy after MCP server config has already been merged and, for embedded Pi, cataloged through the session MCP runtime. The memory impact in the report and the least-privilege concern raised in the discussion are still valid.

## What This Changes

Request adding optional per-MCP-server agent ID or glob scoping so only matching agents spawn or connect to that server, while omitted scope preserves the current all-agent behavior.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, injected into downstream CLI backend config, cataloged, fingerprinted, or cached. Omitted scope should keep the current all-agents default, with aligned config types, schema/help/docs, and focused embedded Pi plus CLI backend tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current gap is high-confidence from source inspection. A multi-agent config with any `mcp.servers` entry still feeds the same merged MCP server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-catalog filtering does not solve process/RAM use or least-privilege startup boundaries.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested feature is security-sensitive because MCP server scoping would become a pre-spawn and remote-connect access boundary.

Concerns:

- **[medium] Define MCP scoping before server startup:** `src/agents/pi-bundle-mcp-runtime.ts:228`
  - body: Any implementation must apply the effective-agent scope before `resolveMcpTransport`, stdio process spawn, HTTP/SSE connection, downstream CLI config injection, tool cataloging, fingerprinting, and runtime caching; otherwise disallowed agents could still start or connect to sensitive MCP servers while only hiding their tools later.
  - confidence: 0.92
- **[medium] Filter CLI backend MCP config before injection:** `src/agents/cli-runner/bundle-mcp.ts:175`
  - body: CLI runtime adapters currently receive the merged MCP map and inject it into downstream backend config. Per-agent scoping must happen before this config is written or passed to Claude, Codex, Gemini, or similar backends.
