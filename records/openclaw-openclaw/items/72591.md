## Summary

Keep open. Current main still treats `mcp.servers` as a shared registry with no typed or documented per-agent MCP scope, and the runtime paths still merge/materialize the whole server map before any agent-specific tool filtering can help.

## What This Changes

Request an optional per-MCP-server agent scope so configured MCP servers spawn only for matching agent IDs or patterns, defaulting to the current all-agents behavior.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, injected into CLI backend config, cataloged, fingerprinted, or cached.

## Reproduction Assessment

Yes for source-level reproduction: a multi-agent config with any `mcp.servers` entry still flows through one merged server map for each session/runtime path. I did not run a live process-count test in this read-only review, but the current call path has no pre-spawn agent filter.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn and pre-connect filtering of the effective MCP server map; `tools.deny` or post-catalog filtering does not address process count or least privilege.
