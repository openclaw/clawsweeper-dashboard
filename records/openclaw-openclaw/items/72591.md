## Summary

Keep open. Current main still has a single shared `mcp.servers` registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume the merged MCP map before any per-agent tool policy can prevent startup, connection, or downstream injection.

## What This Changes

Add optional per-MCP-server agent scoping so only matching agents spawn or receive that server while omitted scope preserves the current all-agents default.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config; omitted scope should preserve today’s all-agents default.

## Reproduction Assessment

Yes. A source-level reproduction is any multi-agent config with a `mcp.servers` entry: current main builds one shared server map and consumes it without an effective-agent filter before runtime startup or CLI injection.

## Solution Assessment

Unclear until maintainers approve public config and security semantics. The narrow maintainable direction is pre-spawn and pre-connect filtering; existing `tools.deny` or post-catalog filtering does not solve the process and access-boundary problem.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because MCP scoping would become a pre-spawn and pre-connect access boundary.

Concerns:

- **[medium] Define scope before MCP startup:** `src/agents/pi-bundle-mcp-runtime.ts:228`
  - body: Any implementation must apply the effective-agent scope before `resolveMcpTransport`, stdio process spawn, HTTP/SSE connection, tool cataloging, fingerprinting, and runtime caching; hiding tools later would still start or connect to disallowed MCP servers.
  - confidence: 0.92
- **[medium] Filter CLI backend MCP config before injection:** `src/agents/cli-runner/bundle-mcp.ts:175`
  - body: CLI runtime adapters currently receive the merged MCP map and inject it into downstream backend config. Per-agent scoping must happen before this config is written or passed to Claude, Codex, Gemini, or similar backends.
  - confidence: 0.86
- **[medium] Include effective scope in runtime cache identity:** `src/agents/pi-bundle-mcp-runtime.ts:474`
  - body: Session MCP runtimes reuse a fingerprint derived from the loaded server map. If different agents see different scoped MCP sets, the effective scoped config must participate in the cache key or use isolated runtimes.
  - confidence: 0.84
