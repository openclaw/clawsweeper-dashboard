## Summary

Keep open. Current main still has a shared MCP server registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume that shared map before per-agent tool policy can avoid startup, remote connection, or config injection.

## What This Changes

Request adding optional per-agent scoping for entries under `mcp.servers` so only matching agents spawn or receive those MCP servers.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before configured or bundled MCP servers are spawned, connected, cataloged, cached, written into downstream CLI backend config, or exposed to tool policy; omitted scope should preserve today’s all-agents default.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure multiple agents with any `mcp.servers` entry and current main builds one shared server map that embedded Pi and CLI backend paths consume before effective-agent filtering can prevent startup or injection.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn and pre-injection filtering; `tools.deny` or other post-catalog filtering is not enough for the reported RAM and least-privilege goals.

  - body: Any implementation must apply effective-agent scope before transport resolution, stdio process spawn, HTTP/SSE connection, tool cataloging, fingerprinting, and runtime caching; hiding tools later would still start or connect to disallowed MCP servers.
  - confidence: 0.92

  - body: CLI runtime adapters currently receive the merged MCP map and inject it into downstream backend config. Per-agent scoping must happen before this config is written or passed to Claude, Codex, Gemini, or similar backends.
