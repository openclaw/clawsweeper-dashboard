## Summary

Keep open. Current main still has a gateway-wide MCP server registry and materializes the full configured server map before per-agent tool policy runs, so the reported memory and least-privilege problem remains valid.

## What This Changes

Add optional per-agent scoping for `mcp.servers` entries so only matching agents start or receive those MCP servers while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the active agent before any MCP server is spawned, connected, injected into CLI backend config, cataloged, or fingerprinted. Omitted scope should preserve current all-agents behavior, with config types/schema/help/docs and focused tests covering embedded Pi, CLI backend overlays, config reload/cache behavior, and interaction with existing tool policy.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A config with multiple agents and any `mcp.servers` entry still feeds the same merged server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config syntax and trust model. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not enough for the memory or guardrail goal.
