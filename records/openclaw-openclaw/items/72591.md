## Summary

Keep open. Current main still has no typed or documented per-agent MCP server scope; the proposed `agents` field would be parsed as inert unknown data, and both embedded Pi and CLI backend paths consume the shared MCP server map before agent-specific tool filtering can avoid startup, connection, or config injection. The issue discussion also correctly identifies a least-privilege concern beyond RAM use.

## What This Changes

The issue requests an optional per-MCP-server agent allowlist or glob scope so configured MCP servers are spawned or injected only for matching agents.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any MCP server is spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config; omitted scope should remain available to all agents.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure multiple agents plus any `mcp.servers` entry, and current main builds one shared server map that embedded Pi and CLI backend paths consume before agent-specific tool filtering.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn, pre-connect, and pre-injection filtering; post-catalog `tools.deny` does not solve the reported RAM or least-privilege problem.
