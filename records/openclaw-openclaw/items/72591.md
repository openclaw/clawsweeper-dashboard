## Summary

Keep open. Current main still has no typed, documented, or enforced per-agent scope for `mcp.servers.*`; the closed duplicate points back to this as the canonical thread, while the open MCP identity and CLI `tools.deny` reports are adjacent but distinct.

## What This Changes

Request adding optional per-agent/glob scoping on `mcp.servers.*` so each agent only starts or exposes the MCP servers it needs.

## Best Possible Solution

Add a backward-compatible, schema-backed MCP server scope resolved against the effective agent before MCP spawn/connect/catalog/cache and CLI backend config injection, with docs, validation, and doctor diagnostics.

## Reproduction Assessment

Yes for the current limitation, not as a strict bug: source inspection shows no enforced `mcp.servers.*.agents` contract and no effective-agent filter before MCP merge/startup. I did not run a live multi-agent memory measurement.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The maintainable direction is pre-spawn and pre-injection filtering; `tools.deny` or prompt-only hiding would not satisfy the reported resource and guardrail goals.
