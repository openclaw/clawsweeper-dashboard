## Summary

Keep open. Current main still treats `mcp.servers` as a gateway-wide server map and starts/connects configured servers before per-agent tool policy can hide their tools, so the reported process/RAM waste and least-privilege concern remain valid.

## What This Changes

The issue requests an optional per-MCP-server agent allowlist, including agent IDs or glob patterns, so multi-agent gateways only spawn or connect MCP servers for matching agents while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any server is spawned, connected, injected into downstream CLI backend config, cataloged, fingerprinted, or exposed to tool policy. Omitted scope should keep the current all-agents behavior, with config types/schema/help/docs and focused tests updated for embedded Pi, CLI bundle MCP, saved MCP config, config reload/cache behavior, and existing tool policy interaction.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path. A multi-agent config with any `mcp.servers` entry still feeds the same merged server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not sufficient for the reported memory use or guardrail boundary.
