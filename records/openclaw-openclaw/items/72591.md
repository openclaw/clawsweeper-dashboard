## Summary

Keep open. Current main still has a gateway-wide MCP server registry and materializes configured servers before per-agent tool policy runs, so the reported process/RAM and least-privilege problem remains. The fix should be maintainer-scoped because it changes a public config contract and MCP pre-spawn security boundary.

## What This Changes

Request an optional per-MCP-server agent scope so configured MCP servers are only spawned for matching agents while omitted scope preserves the current all-agents behavior.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any server is spawned, connected, injected into downstream CLI backend config, cataloged, or fingerprinted. Omitted scope should keep the current all-agents default, and the change should update config types/schema/help/docs plus focused tests for embedded Pi, CLI bundle MCP, config reload/cache behavior, and existing tool policy interaction.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A config with multiple agents and any mcp.servers entry still feeds the same merged server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config shape. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; tools.deny or post-materialization filtering is not sufficient for the reported memory and least-privilege boundary.
