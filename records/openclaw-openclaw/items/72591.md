## Summary

Keep open. Current main still treats configured MCP servers as a gateway/session-wide registry and materializes the full server map before agent-specific tool policy runs, so the reported process/RAM waste and least-privilege gap remain.

## What This Changes

Add optional per-MCP-server agent scoping so configured MCP servers only start or connect for matching agents while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any configured or bundled MCP server is spawned, connected, injected into CLI backend config, cataloged, fingerprinted, or cached. Omitted scope should keep the current all-agents default, with updated config types/schema/help/docs and focused embedded Pi plus CLI backend tests.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A multi-agent config with any mcp.servers entry still feeds the same merged MCP server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; tools.deny or post-materialization filtering does not solve the process/RAM or least-privilege boundary.
