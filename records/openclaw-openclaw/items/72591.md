## Summary

Keep open. Current main still has no typed, documented, or enforced per-agent scope on `mcp.servers.*`, and the closed duplicate https://github.com/openclaw/openclaw/issues/78787 points back to this issue as the canonical thread while related identity and `tools.deny` reports remain distinct.

## What This Changes

Add an optional `agents` filter on `mcp.servers.*` so only matching agents spawn or connect to that MCP server, with the current all-agent behavior preserved when omitted.

## Best Possible Solution

Add a backward-compatible, schema-backed MCP server scope resolved against the effective agent before configured or bundled MCP servers are spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config.

## Reproduction Assessment

Yes for the current limitation, not as a strict bug: source inspection shows current main has no enforced `mcp.servers.*.agents` contract and consumes a shared MCP server map without effective-agent filtering. I did not run a live multi-agent memory measurement.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The maintainable direction is pre-spawn and pre-injection filtering; `tools.deny` or other post-catalog filtering is not enough for the reported resource and guardrail goals.
