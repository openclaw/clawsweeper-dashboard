## Summary

Keep open. Current main still lacks a typed or documented per-agent MCP server scope: configured MCP servers are merged into one global map and materialized before the final per-agent tool policy pass, so the reported process/RAM cost and pre-spawn least-privilege concern remain.

## What This Changes

Add an optional per-server agent filter under `mcp.servers.*` so MCP servers are spawned only for matching agents while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any server is spawned, connected, fingerprinted, cataloged, materialized, or injected into downstream CLI backend config. Omitted scope should keep the current all-agents behavior, and the change should update config types/schema/help/docs plus focused tests for embedded Pi, CLI bundle MCP, config reload/cache behavior, and existing tool policy interaction.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A multi-agent config with any `mcp.servers` entry still feeds the same merged server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not sufficient for the reported memory and least-privilege boundary.
