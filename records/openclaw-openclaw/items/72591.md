## Summary

Keep open. Current main has a shipped Codex-app-server-only `mcp.servers.<name>.codex.agents` projection, but it still lacks the requested first-class cross-runtime `mcp.servers.<name>.agents` scope that would prevent unneeded MCP startup/connect and enforce the boundary for embedded Pi and CLI backend injection.

## What This Changes

Request adds optional per-agent scoping for configured MCP servers so only matching agents receive or start those servers while omitted scope preserves current all-agent behavior.

## Best Possible Solution

Define one backward-compatible, schema-backed `mcp.servers.<name>.agents` contract and resolve it against the effective agent before MCP startup/connect, cataloging, runtime caching, project snapshots, and CLI/Codex backend config injection.

## Reproduction Assessment

Yes for the feature gap: current main has only `mcp.servers.<name>.codex.agents`, while the generic merge, embedded runtime, and CLI backend config paths still consume all configured servers without an effective-agent filter. I did not live-run the reporter's 12-agent memory measurement.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The likely maintainable solution is pre-startup/pre-injection filtering, not hiding tools after MCP servers have already been started or projected.
