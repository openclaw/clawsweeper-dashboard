## Summary

Current main still does not implement per-agent MCP server scoping: configured MCP servers are accepted as a gateway-wide map and each session runtime iterates that map before per-agent tool policy can remove visibility. The request is valid, recently discussed, and security-relevant because a follow-up comment explicitly frames it as a guardrail for sensitive MCP servers, so it should remain open for maintainer-approved config/API design rather than conservative cleanup.

## What This Changes

The issue requests an optional per-MCP-server agent allowlist, including agent IDs or glob patterns, so multi-agent gateways only spawn or connect MCP servers for matching agents while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before any server is spawned, connected, injected into downstream CLI backend config, cataloged, fingerprinted, or exposed to tool policy. Omitted scope should keep the current all-agents behavior, and the change should update config types/schema/help/docs plus focused tests for embedded Pi runtime, CLI bundle MCP config, saved MCP validation, config reload/cache behavior, bundle MCP interactions, and existing tool policy interaction.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A multi-agent config with any `mcp.servers` entry still feeds the same merged server map into each session runtime before per-agent tool policy can apply.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not sufficient for the reported process/RAM usage or guardrail boundary.
