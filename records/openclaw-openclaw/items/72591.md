## Summary

Keep open. Current main still has a shared `mcp.servers` registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume the merged MCP map before agent-specific tool filtering can prevent startup, connection, or injection.

## What This Changes

Request to add optional per-MCP-server agent scoping so configured MCP servers only start or get injected for matching agents while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible, typed, documented MCP server scope that is resolved against the effective agent before configured or bundled MCP servers are spawned, connected, cataloged, cached, written to project settings, or injected into CLI backend config.

## Reproduction Assessment

Yes. Source-level reproduction is a multi-agent config with any `mcp.servers` entry: current main builds one shared MCP map and consumes it without an effective-agent filter before runtime startup or CLI injection.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-start and pre-injection filtering; existing `tools.allow`/`tools.deny` only hides tools after MCP materialization.
