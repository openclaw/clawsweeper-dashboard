## Summary

Keep open. Current main still has a shared MCP server registry with no typed or documented per-agent scope, and both embedded Pi and CLI backend paths consume the merged MCP map before agent-level tool policy can avoid MCP startup, connection, or downstream config injection.

## What This Changes

Request an optional per-agent or glob-based scope on `mcp.servers.*` so only matching agents start or receive those MCP servers.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the effective agent before configured or bundled MCP servers are merged into runtime catalogs, connected/spawned, written to CLI backend config, cached, or advertised; omitted scope should preserve today’s all-agents default.

## Reproduction Assessment

Yes. A source-level reproduction is a multi-agent config with any `mcp.servers` entry: current main builds one shared server map and consumes it without an effective-agent filter before embedded runtime startup or CLI backend injection.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The requested pre-start scoping direction is the right boundary, while existing `tools.deny` or post-catalog filtering is not enough to solve the reported process and connection fan-out.
