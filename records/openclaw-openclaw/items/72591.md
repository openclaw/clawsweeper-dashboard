## Summary

Keep open. Current main still treats `mcp.servers` as a shared registry and materializes the merged server map before agent-specific tool policy runs, so the reported process/RAM waste and the least-privilege concern from the discussion remain valid.

## What This Changes

Request adding optional per-MCP-server agent ID or glob scoping so only matching agents spawn or connect to that server, while omitted scope preserves the current all-agent behavior.

## Best Possible Solution

Add a backward-compatible MCP server scope that is resolved against the effective agent before any configured or bundled MCP server is spawned, connected, injected into downstream CLI backend config, cataloged, fingerprinted, or cached. Omitted scope should keep the current all-agents default, with aligned config types, schema/help/docs, and focused embedded Pi plus CLI backend tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current gap is high-confidence from source inspection. A multi-agent config with any `mcp.servers` entry still feeds the same merged MCP server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config and security semantics. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-catalog filtering does not solve process/RAM use or least-privilege startup boundaries.
