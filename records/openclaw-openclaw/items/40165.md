## Summary

Keep open: current main has several NO_REPLY defenses, but the exact prefix/suffix behavior requested here is not consistently implemented and partly conflicts with existing tests that preserve mixed-content NO_REPLY strings.

## What This Changes

The issue requests gateway-level handling that suppresses leading NO_REPLY payloads and strips trailing NO_REPLY suffixes before user-visible delivery.

## Best Possible Solution

Unify silent-token normalization behind a maintainer-approved shared policy across dispatcher, route-reply, outbound payload planning, live chat projection, docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection: current tests reject `NO_REPLY more` as a prefix sentinel and preserve mixed outbound strings like `thanks NO_REPLY`. I did not run tests or live channels because this review was read-only.

## Solution Assessment

Unclear. The proposed broad gateway helper is directionally useful, but maintainers should first reconcile the existing exact-only and mixed-content preservation contracts so natural text is not stripped incorrectly.
