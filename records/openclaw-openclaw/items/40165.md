## Summary

Keep open: current main and v2026.5.19 include several NO_REPLY hardening fixes, but the broad prefix/suffix suppression policy requested here is still not implemented consistently and remains a product/security delivery-policy decision.

## What This Changes

The issue requests robust NO_REPLY handling that suppresses malformed leading-token replies and strips or suppresses trailing-token replies before they reach external channels.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests with explicit compatibility tradeoffs and live delivery proof.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show exact, glued-leading, and cron trailing cases are handled, while separated leading and mixed NO_REPLY literals remain deliverable in shared normalization/outbound paths. I did not run live Telegram, WhatsApp, or cron transport proof in this read-only review.

## Solution Assessment

No, a gateway-local regex is not the best complete fix because the behavior crosses shared token helpers, reply normalization, outbound planning, cron delivery, gateway display/history, and channel adapters. The safer direction is a single maintainer-approved classifier and rollout plan with regression and transport proof.
