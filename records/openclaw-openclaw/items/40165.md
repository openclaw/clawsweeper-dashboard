## Summary

Keep open: current main has partial NO_REPLY hardening, including exact Telegram suppression and cron trailing-token suppression, but shared helpers and tests still preserve whitespace-separated prefix forms and mixed outbound text. The remaining request is a cross-surface silent-token policy decision, not a cleanup-close candidate.

## What This Changes

Request robust NO_REPLY handling that suppresses prefix misuse, strips or suppresses suffix misuse, and clarifies model prompt guidance across gateway and delivery paths.

## Best Possible Solution

Define one documented silent-token classifier and policy, then apply it consistently through reply normalization, outbound planning, cron delivery, Gateway history/projection, and Telegram delivery.

## Reproduction Assessment

Yes, source-reproducible: current tests assert separated prefix forms and mixed outbound NO_REPLY text are preserved, while cron tests cover only exact/trailing suppression. I did not run a live Telegram or cron probe in this read-only review.

## Solution Assessment

No, a gateway-local regex is not the best complete fix because the behavior crosses shared token helpers, outbound planning, cron delivery, docs, and channel delivery. The maintainable path is a single policy decision plus shared implementation and regression coverage.
