## Summary

Keep open: current main and the latest release have several NO_REPLY hardening fixes, but they still do not implement the broad prefix/suffix policy requested here, and the remaining behavior needs a maintainer policy decision before another fix PR is safe.

## What This Changes

The issue requests a robust NO_REPLY delivery policy that suppresses or strips malformed sentinel text in prefix, suffix, and mixed positions before channel delivery.

## Best Possible Solution

Define one maintainer-approved silent-token policy, then apply it consistently across shared token helpers, reply normalization, outbound planning, cron delivery, gateway display/history, and channel-visible tests.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show exact, glued-leading, and cron trailing cases are handled in places while separated leading and mixed NO_REPLY literals remain deliverable. I did not run live Telegram, WhatsApp, or cron transport proof in this read-only review.

## Solution Assessment

No: a gateway-local regex is not the best complete solution because the behavior crosses shared token helpers, reply normalization, outbound planning, cron delivery, gateway display/history, and channel adapters. The safer path is a single maintainer-selected policy with focused regression and live delivery proof.
