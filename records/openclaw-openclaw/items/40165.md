## Summary

Keep open: current main has partial NO_REPLY hardening, but the requested robust prefix/suffix behavior is still inconsistent across shared token helpers, reply normalization, outbound delivery, cron, Gateway display, docs, and overlapping open PRs.

## What This Changes

Request broader NO_REPLY handling so prefix-position sentinel text is suppressed and suffix-position sentinels are stripped or suppressed across Gateway and delivery paths.

## Best Possible Solution

Define one shared silent-token classifier and documented policy, then wire it through normalization, directive parsing, outbound planning, cron/subagent delivery, Gateway projection/history, Telegram delivery, and regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main's tests and helpers encode exact-only, glued-leading-only, strip-and-deliver, and cron-trailing-only behavior. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear: the requested gateway-level regex is plausible defense in depth, but the maintainable fix is a shared classifier and explicit policy so prefix, suffix, own-line, exact, direct/group, streaming, and cron behavior do not diverge.
