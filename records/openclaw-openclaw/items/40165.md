## Summary

Keep open: current main and the latest release have partial NO_REPLY hardening, but the requested robust prefix/suffix policy is still inconsistent across shared reply normalization, outbound delivery, cron delivery, and Gateway display paths.

## What This Changes

Request to strip or suppress malformed NO_REPLY prefix and suffix output so the control token does not leak into user-visible channel replies.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, Gateway display/history, and channel-visible tests after maintainers choose the compatibility boundary.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show exact-token, glued-leading, and scoped cron trailing handling, while separated leading and shared mixed NO_REPLY literals remain deliverable. I did not run live transport proof in this read-only review.

## Solution Assessment

No, the issue's gateway-local regex is not the best complete fix because the behavior spans shared token helpers, reply normalization, outbound planning, cron delivery, Gateway display, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the chosen policy.
