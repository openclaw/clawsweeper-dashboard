## Summary

Keep this open: current main and the latest release have several partial NO_REPLY fixes, but they still do not define the broader prefix/suffix or suppress-if-present policy requested here.

## What This Changes

Request broad gateway NO_REPLY handling that suppresses malformed prefix-position replies, strips or suppresses suffix tokens, and clarifies model prompt guidance.

## Best Possible Solution

Define one shared silent-token policy, then apply it consistently across auto-reply normalization, gateway display, cron announce/direct delivery, message-tool sends, and streaming with regression coverage for exact, suffix, glued-leading, whitespace-leading, reasoning-prefixed, and any approved suppress-if-present behavior.

## Reproduction Assessment

Yes, at source level. Current helpers and tests show exact, suffix, glued-leading, and reasoning-prefixed cases are partly handled while broad prefix or suppress-if-present behavior is not the current contract.

## Solution Assessment

Unclear. Broader suppression is plausible, but current docs and tests preserve substantive content in some mixed NO_REPLY cases, so maintainers should choose the policy before implementation.
