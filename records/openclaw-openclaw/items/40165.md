## What This Changes

The issue requests broader NO_REPLY handling so prefix malformed responses are suppressed and suffix tokens are stripped or otherwise handled before user-visible delivery.

## Best Possible Solution

Define one documented silent-token classifier and policy, then apply it consistently through reply normalization, outbound planning, cron delivery, gateway projection/history, and channel delivery.

Yes, source-reproducible: current tests and code paths show exact and trailing cases are handled in places, while separated prefix forms and mixed outbound NO_REPLY text are still preserved. I did not run live Telegram or cron validation in this read-only review.

No, a gateway-local regex is not the best complete fix because the behavior crosses shared token helpers, outbound planning, cron delivery, docs, and channel delivery. The maintainable path is a single policy decision plus shared implementation and regression coverage.
