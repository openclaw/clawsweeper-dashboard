## Summary

Current main fixes the requested suffix cleanup and several glued-prefix/cron variants, but it intentionally does not implement the issue’s broader rule that a whitespace-separated leading `NO_REPLY` suppresses the entire response. The remaining behavior is still necessary to decide because it changes message-delivery and security semantics: suppressing the whole response prevents internal narration leaks, while stripping or preserving content avoids discarding potentially substantive output.

## What This Changes

Treat malformed `NO_REPLY` at both the beginning and end of model output as a delivery control, stripping substantive suffix cases and suppressing malformed prefix cases, with clearer prompt wording.

## Best Possible Solution

Define a single typed normalization outcome for silent-token handling and use it across normal replies, streaming, isolated cron announce, and command cron delivery; preserve the shipped suffix cleanup, add explicit policy tests for whitespace-separated leading tokens, and update the system prompt only after maintainers choose whether those cases mean full suppression or cleaned delivery.

## Reproduction Assessment

Yes at source level: `startsWithSilentToken` requires visible content to be glued directly to the token, so a whitespace-separated prefix such as `NO_REPLY\n\nHello` does not enter the leading-token cleanup path. The discussion also supplies concrete Telegram and cron examples, although no live current-main transport run was performed in this read-only review.

## Solution Assessment

No, the proposed gateway-local regex is not the best maintainable fix because current main already owns this behavior in shared auto-reply and cron normalization helpers. The best fix is one canonical normalization result with an explicit maintainer decision for whitespace-separated prefix semantics.
