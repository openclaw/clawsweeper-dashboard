## Summary

Keep open: current main has partial NO_REPLY hardening, but it still does not implement the requested robust prefix/suffix policy consistently across shared helpers, outbound delivery, cron, Gateway display, and Telegram-adjacent paths. Related open PRs cover narrower pieces with different semantics, so this needs maintainer policy selection before a safe repair branch.

## What This Changes

The issue requests robust NO_REPLY handling that suppresses malformed leading silent-token replies and strips trailing NO_REPLY suffixes without leaking the token to users.

## Best Possible Solution

Define and document one shared silent-token classifier and policy, then apply it consistently through reply normalization, outbound planning, cron/subagent delivery, Gateway projection/history, Telegram delivery, and regression tests.

## Reproduction Assessment

Yes, source-reproducible: current tests and helpers encode exact-only, glued-leading-only, strip-and-deliver, outbound mixed-token preservation, and cron trailing-only behavior. I did not run a live Telegram or cron reproduction in this read-only review.

## Solution Assessment

No: the issue's gateway-local regex is not the best complete fix because the behavior spans shared helpers, outbound delivery, cron, Telegram, and Gateway display paths. The maintainable path is a shared classifier plus an explicit documented policy.
