## Summary

Keep open: current main and v2026.5.22 still have mixed NO_REPLY behavior split across paths, with shared outbound deliberately preserving prefix/suffix literals while cron has narrower trailing-token hardening. This remains a security-sensitive delivery-policy decision with overlapping open work, not a cleanup close.

## What This Changes

The issue requests broader NO_REPLY handling so malformed prefix outputs are suppressed and suffix outputs are stripped or otherwise suppressed before channel delivery.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests while preserving media and legitimate NO_REPLY mentions.

## Reproduction Assessment

## Solution Assessment

No, the issue's gateway-local regex sketch is not the best complete fix because the behavior crosses shared token helpers, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the selected compatibility policy.
