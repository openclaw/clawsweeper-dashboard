## Summary

Keep open: current main and v2026.5.22 still have inconsistent NO_REPLY handling. Some auto-reply and cron paths strip or suppress mixed tokens, but shared outbound normalization still preserves mixed prefix/suffix literals, so the central delivery-policy problem remains and needs maintainer security/product review.

## What This Changes

Request robust handling for malformed NO_REPLY outputs that appear before or after substantive text before they reach user-visible gateway/channel delivery.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests while preserving media payloads and legitimate NO_REPLY mentions.

## Reproduction Assessment

Yes, at source level. Current main tests preserve mixed prefix and suffix NO_REPLY literals in shared outbound normalization, while other paths strip or suppress them; no live transport proof was run in this read-only review.

## Solution Assessment

No, the issue's gateway-local regex sketch is not the best complete fix because the behavior crosses shared token helpers, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the selected compatibility policy.
