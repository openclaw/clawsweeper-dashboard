## Summary

Keep open: current main and v2026.5.22 still have mixed NO_REPLY handling split across delivery paths, with shared outbound deliberately preserving prefix/suffix literals while cron only suppresses narrower trailing-token forms. This remains a security-sensitive delivery-policy decision, not a cleanup close.

## What This Changes

The issue requests broader NO_REPLY normalization so prefix, suffix, and mixed silent-control tokens are stripped or suppressed consistently before user-visible delivery.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests while preserving media and legitimate NO_REPLY mentions.

## Reproduction Assessment

Yes, at source level: current main tests preserve mixed `NO_REPLY thanks for the update` and `thanks NO_REPLY` in shared outbound normalization, while cron suppresses only narrower exact/trailing-token forms. I did not run live transport proof in this read-only review.

## Solution Assessment

No, the issue's gateway-local regex sketch is not the best complete fix because the behavior crosses shared token helpers, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the selected compatibility policy.
