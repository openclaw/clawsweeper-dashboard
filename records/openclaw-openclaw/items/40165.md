## Summary

Keep open: current main has several NO_REPLY hardening fixes, including exact-token suppression, glued-leading stripping, and cron trailing-token suppression, but the broader prefix/suffix policy requested here is still inconsistent across shared reply normalization, outbound planning, cron delivery, and gateway display paths.

## What This Changes

The issue requests broader NO_REPLY handling so prefix forms suppress delivery and suffix forms are stripped or suppressed across gateway and delivery paths.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests with explicit compatibility tradeoffs.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show exact, glued-leading, and cron trailing cases are handled, while separated leading and mixed NO_REPLY literals remain deliverable in shared outbound paths. I did not run live transport proof in this read-only review.

## Solution Assessment

No, the gateway-local regex proposed in the issue is not the best complete fix because the behavior crosses token helpers, reply normalization, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the chosen compatibility policy.
