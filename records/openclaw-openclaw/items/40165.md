## Summary

Keep open: current main has partial NO_REPLY hardening, but separated leading tokens and mixed outbound/non-trailing NO_REPLY text are still intentionally preserved, so the remaining behavior needs a maintainer product/security decision rather than cleanup closure.

## What This Changes

Request to make NO_REPLY handling robust across prefix and suffix positions so malformed control tokens are suppressed or stripped before gateway/channel delivery.

## Best Possible Solution

Define and document one shared, case-insensitive silent-token classifier and delivery policy, then apply it consistently through reply normalization, outbound planning, cron delivery, gateway projection/history, and channel delivery tests.

## Reproduction Assessment

Yes, source-reproducible: current tests and code paths show exact, glued-leading, and trailing cases are handled in places while separated leading and mixed outbound NO_REPLY text remain preserved. I did not run a live Telegram or cron probe in this read-only review.

## Solution Assessment

No, a gateway-local regex is not the best complete solution because the behavior crosses shared token helpers, reply normalization, outbound planning, cron delivery, gateway display/history, docs, and channel delivery. The safer path is a single documented policy plus focused regression coverage across those surfaces.
