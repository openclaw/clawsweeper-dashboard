## Summary

Keep open: current main still has conflicting NO_REPLY behavior across reply normalization, outbound delivery, gateway display, and docs, and related comments plus open PRs show this needs a maintainer policy decision rather than cleanup closure.

## What This Changes

The issue requests gateway-level handling that suppresses leading NO_REPLY output, strips trailing NO_REPLY sign-off tokens from substantive content, and clarifies prompt guidance.

## Best Possible Solution

Choose one shared silent-token policy, then apply it consistently through token helpers, reply normalization, directive parsing, dispatcher delivery, outbound payload planning, gateway/WebChat projection, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested cases can be checked directly: token tests reject `NO_REPLY more`, outbound tests preserve `thanks NO_REPLY`, and normalizer tests strip some suffix cases.

## Solution Assessment

Unclear. The proposed gateway defense is useful, but maintainers first need to reconcile exact-only preservation, own-line suppression, and broad trailing-token suppression so natural user-facing text is not dropped incorrectly.
