## Summary

Keep open. Current main has partial NO_REPLY hardening, including exact-token suppression, lead-fragment suppression, glued-leading cleanup, and one reply-normalization suffix-strip path, but it still does not implement the requested shared contract across gateway chat projection, outbound payload planning, and routeReply/channel delivery.

## What This Changes

Request to normalize NO_REPLY more robustly by suppressing exact and prefix-position silent tokens, stripping terminal NO_REPLY from substantive replies, and tightening prompt guidance.

## Best Possible Solution

Define one shared silent-token normalization/projection contract and use it consistently at gateway live/final chat projection, reply normalization, routeReply/channel delivery, and outbound payload planning. The contract should cover exact suppression, separated-prefix suppression, glued-prefix cleanup, terminal-suffix stripping with cleaned-content delivery, media behavior, direct/group silent policy interaction, and safe embedded-token exceptions, then update prompts/docs to match.

## Reproduction Assessment

Yes. Current-main source and regression tests give a high-confidence static reproduction path: mixed prefix and suffix examples are still preserved in outbound payload tests, and the gateway projector has no branch that strips terminal NO_REPLY or suppresses separated-prefix content.

## Solution Assessment

No. The current partial implementation is not the best end state because suffix cleanup, prefix handling, and lead-fragment suppression are split across separate helpers and surfaces; a shared helper with focused contract tests is the safer maintainable path.
