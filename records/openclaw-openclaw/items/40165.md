## Summary

Keep open: current main has partial NO_REPLY hardening, but it still preserves several mixed or separated prefix/suffix forms and the remaining behavior is a cross-path delivery policy decision rather than a finished implementation.

## What This Changes

The issue requests robust NO_REPLY handling so malformed prefix/suffix tokens are suppressed or stripped before user-visible delivery.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests with explicit false-positive tradeoffs.

## Reproduction Assessment

Yes, source-level: current source and tests show exact, glued-leading, and cron trailing cases are handled, while separated leading and mixed NO_REPLY literals remain deliverable in shared paths. I did not run live transport proof in this read-only review.

## Solution Assessment

No, the gateway-local regex proposed here is not the best complete fix because the behavior crosses shared token helpers, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the selected compatibility policy.
