## Summary

Keep open: current main still has exact-only and strip-and-deliver NO_REPLY behavior outside a cron-specific trailing-token path, and recent comments plus active PRs show the remaining work needs a maintainer policy choice rather than cleanup closure.

## What This Changes

Request a shared gateway/delivery policy that handles malformed prefix and suffix NO_REPLY outputs without leaking the token or unintended model text to users.

## Best Possible Solution

Define one shared silent-reply classifier and policy, align docs/tests with that decision, and apply it through normalization, directive parsing, outbound planning, cron, gateway display/history, and channel delivery.

## Reproduction Assessment

Yes, source-level. Current main still has tests and helpers that preserve text around mixed NO_REPLY in shared delivery paths, while cron has a separate trailing-token suppression path, so the inconsistent behavior can be traced without a live channel run.

## Solution Assessment

Unclear. A gateway-level regex would address some examples, but the maintainable solution is a shared classifier and explicit policy boundary so exact, prefix, suffix, own-line, direct/group, cron, and streaming behavior do not diverge.
