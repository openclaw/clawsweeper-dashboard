## Summary

Keep open: current main has partial NO_REPLY hardening, but docs and tests still preserve separated leading and mixed NO_REPLY text in several paths, while related open PRs are still trying to settle the remaining silent-token policy.

## What This Changes

Requests a gateway and delivery policy that suppresses exact/prefix NO_REPLY and strips or suppresses suffix NO_REPLY across channels.

## Best Possible Solution

Define and document one shared case-insensitive silent-token classifier and delivery policy, then apply it consistently across reply normalization, directive/outbound planning, cron/subagent delivery, gateway projection/history, and channel-visible tests.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show exact/glued/trailing cases are handled in places while separated leading and mixed NO_REPLY text remain preserved in others. I did not run a live Telegram, WhatsApp, or cron probe in this read-only review.

## Solution Assessment

No, a gateway-local regex is not the best complete solution because the behavior crosses shared token helpers, reply normalization, directive parsing, outbound planning, cron delivery, gateway display/history, and docs. The safer path is one maintainer-selected policy plus focused regression and real delivery proof.
