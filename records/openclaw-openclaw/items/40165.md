## Summary

Keep open: current main and v2026.5.20 have partial NO_REPLY hardening, but shared delivery paths still intentionally preserve several mixed prefix/suffix forms, so this remains an unresolved delivery-policy decision with overlapping open work.

## What This Changes

The issue asks OpenClaw to treat malformed NO_REPLY prefix/suffix output as a shared suppression or stripping policy rather than only suppressing exact-token replies.

## Best Possible Solution

Define one documented silent-reply classifier and apply it consistently across reply normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and channel-visible tests while preserving media and non-text payloads.

## Reproduction Assessment

Yes, at source level: current main tests and source show exact, glued-leading, and cron trailing cases are handled while separated leading and shared mixed NO_REPLY literals can remain deliverable. I did not run live transport proof in this read-only review.

## Solution Assessment

No, the issue's gateway-local regex sketch is not the best complete fix because the behavior crosses shared token helpers, directive parsing, outbound planning, cron delivery, gateway display/history, and channel adapters. The maintainable path is one shared classifier plus tests and docs for the selected compatibility policy.
