## What This Changes

The issue requests robust NO_REPLY normalization so leading malformed silent replies are suppressed and trailing NO_REPLY markers are stripped from otherwise visible replies.

## Best Possible Solution

Centralize silent-token projection for user-visible text, apply it consistently in gateway live chat and outbound delivery, preserve media-only behavior, and update regression tests plus changelog for prefix suppression and suffix stripping.

## Reproduction Assessment

Yes, by source inspection: `projectLiveAssistantBufferedText` returns raw mixed text after only attached-leading cleanup, and `src/infra/outbound/payloads.test.ts` currently asserts mixed prefix/suffix NO_REPLY payloads are delivered unchanged. I did not execute tests in this read-only sweep.

## Solution Assessment

Partly. A gateway guard is useful for webchat, but the best maintainable fix is shared normalization across gateway projection and outbound payload planning rather than one channel-specific check.
