## Summary

Keep this open: current main has partial `NO_REPLY` hardening for suffix, glued-leading, and reasoning-prefixed cases, but it still preserves some leading-token content and has not resolved the broader suppress-if-present delivery policy requested here.

## What This Changes

The issue requests gateway/reply handling that suppresses or strips malformed `NO_REPLY` tokens in both leading and trailing positions before user-visible delivery.

## Best Possible Solution

Define the canonical silent-token policy once in shared auto-reply normalization, then apply it consistently across gateway, cron, streaming, and routed delivery with regression coverage for exact, suffix, glued-leading, whitespace-leading, and any approved suppress-if-present behavior.

## Reproduction Assessment

Yes, at source level: current tests and normalization code show exact, suffix, glued-leading, and reasoning-prefixed cases are handled, while broad leading-token or suppress-if-present behavior is not the current contract. I did not run a live Telegram or cron reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation direction: current main already solves several narrow leaks, but changing to broader suppression would alter delivery semantics and should be a maintainer product decision before code changes.
