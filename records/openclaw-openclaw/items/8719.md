## Summary

Keep open. Current main has useful hardening pieces, but it still does not implement the requested native data-centric Security Profile v1.1, the discussion already has an explicit keep-open review, and the related filesystem/network work only covers narrower slices of the requested model.

## What This Changes

Request a native secure-by-default Security Profile v1.1 with data sensitivity levels, cross-tool filesystem/exec/network enforcement, audit logging, approval/elevation, session isolation, and plugin trust controls.

## Best Possible Solution

Keep this as the native Security Profile umbrella until security/product owners either define the v1.1 contract or split it into canonical issues for filesystem ACLs, exec policy, network/export policy, audit logs, approval/elevation, session isolation, plugin verification, config integrity, and platform sensory inputs; use ClawMoat from the discussion as external prior art, not a core replacement.

## Reproduction Assessment

Not applicable as a bug reproduction. This is a broad security architecture feature request, and source/docs inspection gives high confidence that current main lacks the native Security Profile contract rather than exposing one discrete failing command path.

## Solution Assessment

No for a single autonomous fix. The maintainable path is security/product decomposition across filesystem, exec, network/export, audit, approval/elevation, session isolation, plugin trust, config integrity, and platform-input surfaces before implementation.
