## Summary

Keep open. Current main still lacks a native data-centric Security Profile v1.1; the implemented hardening is narrower workspace-only filesystem policy, optional operator proxy routing, trusted-plugin guidance, and related guardrails, while the requested profile spans filesystem, exec, network/export, audit, approvals, elevation, session isolation, platform sensory inputs, and plugin/skill verification.

## What This Changes

Request a native secure-by-default OpenClaw Security Profile v1.1 with data sensitivity levels, cross-tool policy enforcement, network controls, audit logging, approvals, TTL elevation, isolation, and plugin/skill verification.

## Best Possible Solution

Keep this as the native Security Profile umbrella until maintainers define the v1.1 threat model and split enforceable controls into canonical filesystem, exec, network/export, audit, approval/elevation, isolation, platform-input, config-integrity, and plugin/skill trust work.

Not applicable. This is a broad feature/security-profile request, and source/docs inspection shows the native profile contract is missing rather than exposing one discrete failing command path.
