## Summary

Keep open: the underlying Control UI bug is still source-reproducible on current main, but this PR should not merge as-is because it routes documented `list` and `status` modes to the bare summary response, adds an undocumented `info` alias, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR reserves `status`, `list`, and `info` in the Control UI `/model` slash-command executor so they use the bare model-info branch, and adds executor tests including a normal model-selection case.

## Best Possible Solution

Land or replace one narrow Control UI fix that routes `/model list` and `/model status` to the documented shared behavior, preserves normal model selection, and only adds `info` after a contract/docs decision.

## Reproduction Assessment

Yes, source-backed with high confidence: `/model status` parses to `args: "status"`, Control UI passes that arg unchanged, and current main sends every non-empty `/model` arg to `sessions.patch`. I did not run a live Control UI session in this read-only review.

## Solution Assessment

No. Avoiding the bad `sessions.patch` call is necessary, but this PR collapses the documented `list` and `status` responses into the bare summary branch and adds an undocumented `info` mode.
