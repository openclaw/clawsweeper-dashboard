## Summary

Keep open: current main still sends non-empty `/model` arguments such as `status` and `list` to `sessions.patch`, but this PR should not merge as-is because it collapses documented `list` and `status` behavior into the bare summary path, reserves undocumented `info`, and omits the required changelog entry.

## What This Changes

The PR adds Control UI `/model` reserved subcommands for `status`, `list`, and `info`, with tests that those tokens show the existing model summary instead of calling `sessions.patch` and that a normal model name still patches.

## Best Possible Solution

Align Control UI `/model` handling with the shared directive/docs contract: preserve normal model selection, route `/model list` to the model browser/list output, route `/model status` to detailed provider/auth/endpoint status, and leave `info` unreserved unless the public contract is deliberately extended.

## Reproduction Assessment

Yes, by source inspection. On current main, `/model status` and `/model list` parse as non-empty args and `executeModel` sends those values to `sessions.patch`; the linked bug report also includes gateway logs showing the rejected `openai-codex/status` and `openai-codex/list` model refs.

## Solution Assessment

No. The PR prevents the bad patch call, but it does so by dropping the documented `list` and `status` semantics and adding an undocumented `info` reservation; the narrower fix is to implement the existing list/status contract in the Control UI local executor.
