## Summary

Keep open: current main still has the Control UI bug, but this PR is not merge-ready because it collapses documented `list` and `status` modes into the bare `/model` summary, adds an undocumented `info` alias, and still lacks after-fix real Control UI proof.

## What This Changes

The branch adds a Control UI `/model` reserved-token check for `status`, `list`, and `info`, plus tests that those tokens avoid `sessions.patch` and that a normal model name still patches.

## Best Possible Solution

Land or replace one narrow Control UI fix that routes `/model list` to the documented list/picker behavior, routes `/model status` to detailed model status output, preserves normal model selection, and leaves `info` unreserved unless the shared contract and docs are deliberately extended.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: `/model status` parses to `args: "status"`, dispatch passes it unchanged, and current `executeModel` sends every non-empty arg to `sessions.patch`. I did not run a live Control UI session in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The PR prevents the bad patch call, but it returns the wrong documented UI for `list` and `status` and adds an `info` alias that is not part of the shared command contract.
