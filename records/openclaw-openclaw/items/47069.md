## Summary

Keep this PR open, but it is not merge-ready: it fixes the bad `sessions.patch` path only partly, lacks after-fix Control UI proof, and conflicts with the documented `/model list` and `/model status` behavior.

## What This Changes

The PR changes the Control UI `/model` executor to treat `status`, `list`, and `info` as model-info requests and adds executor regression tests.

## Best Possible Solution

Land one rebased canonical Control UI fix that routes `/model list` and `/model status` to the documented picker/status flows, preserves `/model <provider/model>`, and either removes or documents `/model info`.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: `/model status` parses to `args: "status"`, dispatch preserves it, and current `executeModel` sends every non-empty `/model` arg to `sessions.patch`.

## Solution Assessment

No. Avoiding the bad `sessions.patch` call is necessary, but this PR collapses documented `list` and `status` modes into the bare summary branch and adds an undocumented `info` keyword.
