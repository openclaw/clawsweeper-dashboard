## Summary

Keep open: the latest force-push restored shared profile fan-out, but the new provider-specific env vars now combine with that fan-out in a way that can persist a Zen key into the Go profile or vice versa. The PR also still needs stronger real behavior proof because the posted `doctor --provider` command is not a supported CLI path.

## What This Changes

This PR adds `OPENCODE_ZEN_API_KEY` and `OPENCODE_GO_API_KEY` precedence for OpenCode Zen/Go providers, updates plugin manifests and docs, and adds env lookup tests.

## Best Possible Solution

Keep provider-specific env lookup, but make persisted profile writes source-aware: shared `OPENCODE_API_KEY` can fan out to both profiles, while provider-specific Zen/Go keys should write only their matching profile or resolve each provider key independently, with onboarding tests and supported real proof.

## Reproduction Assessment

Yes for the merge blocker at source level: `createProviderApiKeyAuthMethod` writes one captured credential to every configured profile id, and the PR keeps shared OpenCode profile ids while changing the credential source to provider-specific env vars. I did not execute tests because this review is read-only.

## Solution Assessment

No, not yet. Split env precedence is a maintainable direction, but persisted setup needs to avoid writing one provider-specific credential into both provider profiles and the proof needs to use supported commands.
