## Summary

Keep open. Current main still uses a fixed 20s client timeout for file-chooser arming, so this PR is not obsolete; the branch fixes the main client path, but it still leaves the CLI explicit `--timeout-ms` path racing its server-side operation deadline and lacks the required changelog entry for a user-facing browser upload fix.

## What This Changes

This PR changes browser file-chooser arming to use the browser action timeout budget, raises the CLI upload wrapper default timeout from 20s to 60s, and adds client assertions for default and explicit file-chooser request timeouts.

## Best Possible Solution

Land the narrow browser upload timeout alignment after padding the CLI explicit-timeout transport deadline or sharing the timeout resolver, adding the required changelog entry, and keeping upload path guardrails, Playwright semantics, and the broader reliability work out of this PR.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main sends `/hooks/file-chooser` with a 20,000ms wrapper timeout, while the hook route can do tab/path/Playwright work before responding; the PR's client test directly asserts the intended default and explicit transport deadlines.

## Solution Assessment

No, not yet as proposed. The browser client fix is the right narrow direction, but the CLI explicit-timeout wrapper should receive the same slack as the client API path and the user-facing fix needs a changelog entry before merge.
