## Summary

Keep open. The PR addresses a real source-reproducible browser upload timeout by aligning the API client with the existing browser action timeout budget, but it still leaves the CLI explicit-timeout path without transport slack and lacks after-fix real behavior proof.

## What This Changes

The PR changes browser file-chooser arming to use the browser action timeout budget, raises the CLI upload default transport timeout to 60s, and adds client timeout assertions.

## Best Possible Solution

Land the timeout alignment only after padding the explicit CLI timeout path or sharing the same resolver, then add redacted after-fix live upload proof while leaving broader upload reliability to https://github.com/openclaw/openclaw/issues/38844.

## Reproduction Assessment

Yes, source-reproducible. Current main sends `/hooks/file-chooser` with a fixed 20,000ms client deadline while the route can perform tab resolution, upload-path validation, Playwright setup, and optional clicking before responding; I did not run a live browser flow in this read-only review.

## Solution Assessment

No, not yet. The core browser client timeout direction is the narrow maintainable fix, but the CLI explicit-timeout wrapper should receive the same slack or a shared resolver, and the external PR still needs after-fix real behavior proof.
