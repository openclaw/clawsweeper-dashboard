## Summary

Keep open. Current main still has the fixed 20s file-chooser client timeout, and this PR is useful but not merge-ready because the CLI explicit-timeout path can still race the browser request boundary and the required user-facing changelog entry is missing.

## What This Changes

The PR changes browser file-chooser arming to use the browser action timeout budget, raises the CLI upload default transport timeout to 60s, and adds client timeout assertions.

## Best Possible Solution

Land the narrow timeout alignment after the CLI upload wrapper uses the same explicit-timeout slack model or a shared resolver, add the required changelog entry, and keep broader stale-ref/error-mapping reliability work tracked separately in #38844.

## Reproduction Assessment

Yes. Current main gives /hooks/file-chooser a fixed 20s client timeout while the route can do tab resolution, path validation, Playwright/CDP attach work, and optional clicking before responding; the proposed client test directly reproduces the timeout-budget regression at the request layer.

## Solution Assessment

No, not quite as proposed. The browser client change is the right narrow direction, but the CLI explicit-timeout path should not reuse the exact server operation timeout as the transport deadline, and the user-facing fix needs its changelog entry before merge.
