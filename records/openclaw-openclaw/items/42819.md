## Summary

Keep open: current main and the latest release can auto-detect Playwright-managed Chromium in Docker, but the requested Docker packaging behavior is still missing because the image still exports the hidden cache path and does not create stable `chrome` or `chromium` commands on PATH.

## What This Changes

The issue requests Docker browser-enabled builds to put Chromium on PATH and store Playwright browser files outside `/home/node/.cache/ms-playwright`.

## Best Possible Solution

Keep the shipped autodetection behavior, then have maintainers explicitly decide the Docker path and PATH-symlink contract before landing a focused Dockerfile/docs/tests change with browser-enabled, browser-disabled, and upgrade proof.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` still set Docker’s Playwright browser path to `/home/node/.cache/ms-playwright`, and the Docker test asserts that contract. I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. The merged autodetection is the best fix for brittle versioned executable paths, but moving the Docker browser store to `/opt` or adding PATH symlinks is a separate compatibility-sensitive Docker packaging decision.
