## Label Justifications

- `P3`: The historical Docker browser-path regression had limited blast radius and is now resolved through automatic discovery.
- `impact:other`: The issue concerns Docker browser startup and executable discoverability, which does not fit the more specific owned impact classes.

## Summary

Current main resolves the reported regression by automatically discovering Playwright-managed Chromium across revisioned cache directories, so Docker users no longer need to hardcode a path that changes after browser upgrades. Moving the files to `/opt` or adding a PATH symlink is no longer necessary to solve the observable browser-launch problem.

## What This Changes

Make Docker’s Playwright-managed Chromium available through a stable path or discovery mechanism instead of requiring a revision-specific cache path.

## Best Possible Solution

Keep automatic Playwright Chromium discovery as the supported contract so Docker browser startup remains independent of Playwright revision names and internal cache placement.

## Reproduction Assessment

Yes for the historical failure mode: a configured path containing Playwright’s revision directory becomes invalid when that browser revision changes. Current main avoids that failure by enumerating the cache dynamically.

## Solution Assessment

Yes. Automatic managed-browser discovery fixes the actual runtime dependency on versioned paths without introducing a new filesystem location, PATH contract, or migration for existing Docker users.
