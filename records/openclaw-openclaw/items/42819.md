## Summary

Keep open. Current main and the latest release still keep Docker Playwright Chromium under `/home/node/.cache/ms-playwright` and do not create `chrome` or `chromium` commands on PATH; the merged autodetection work only removes the need for hardcoded versioned paths. The remaining Docker packaging decision is already paired with open implementation PR https://github.com/openclaw/openclaw/pull/42771, so this issue should stay open until that PR is landed, refreshed, or explicitly closed.

## What This Changes

The issue asks Docker browser-enabled images to store Playwright Chromium outside the hidden home cache and expose a stable `chrome` or `chromium` command on PATH.

## Best Possible Solution

Settle the Docker browser path contract through https://github.com/openclaw/openclaw/pull/42771 or a maintainer replacement, with migration guidance, Dockerfile tests, docs, and real Docker proof for browser-enabled and browser-disabled images.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.19` shows Docker still exports `/home/node/.cache/ms-playwright` and lacks Dockerfile-created `chrome` or `chromium` PATH commands; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. The merged autodetection work is the narrower fix for brittle versioned Chromium paths, while moving the default browser store and adding PATH aliases is a compatibility-affecting Docker packaging decision that should be settled in the linked PR or a maintainer replacement.
