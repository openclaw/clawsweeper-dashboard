## Summary

Keep open. Current main and the latest release still use `/home/node/.cache/ms-playwright` for Docker Playwright Chromium and do not expose a Dockerfile-created `chrome` or `chromium` command on PATH; the merged autodetection work only removed OpenClaw's need to hardcode the revisioned executable path.

## What This Changes

The issue asks for Docker browser assets to live outside the hidden Playwright cache path and for a stable Chrome or Chromium executable on PATH.

## Best Possible Solution

Maintainers should decide the Docker browser path contract; if changing it, land a focused Docker/docs/tests update with stable `chrome`/`chromium` PATH behavior plus browser-enabled, browser-disabled, runtime-install, and upgrade/mount proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the Dockerfile still exports the hidden Playwright cache path and lacks Dockerfile-created `chrome` or `chromium` commands on PATH; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. Current main's autodetection is the narrowest shipped fix for OpenClaw's own brittle versioned executable discovery, but the remaining `/opt` path and PATH-alias request needs a maintainer-approved compatibility and migration plan.
