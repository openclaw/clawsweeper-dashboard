## Summary

Keep open: current main and v2026.6.6 already solve the brittle versioned Playwright executable lookup through browser auto-detection, but the requested Docker image contract is still missing because the image keeps browsers under `/home/node/.cache/ms-playwright` and does not expose stable `chrome` or `chromium` commands on PATH.

## What This Changes

The issue requests that Docker browser-enabled images stop relying on versioned Playwright Chromium paths under `/home/node/.cache/ms-playwright` and expose a stable browser binary on PATH, preferably with browser files under `/opt`.

## Best Possible Solution

Keep the shipped Playwright-cache auto-detection, then land a maintainer-approved Dockerfile/docs/test change only if OpenClaw wants a new `/opt` browser-store and PATH-command contract with upgrade guidance and Docker proof.

## Reproduction Assessment

Yes by source inspection. Current main still exports `/home/node/.cache/ms-playwright` from the Dockerfile and has no Dockerfile-created stable `chrome` or `chromium` command; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear as a product direction. The merged auto-detection path is the narrow fix for brittle versioned executable paths, while moving the browser store to `/opt` and exposing PATH commands is a compatibility-sensitive Docker packaging choice.
