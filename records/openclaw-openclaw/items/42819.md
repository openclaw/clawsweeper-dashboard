## Summary

Keep open. Current `main` and the latest release still keep Docker Playwright browsers under `/home/node/.cache/ms-playwright` and do not create stable `chrome` or `chromium` commands on PATH; merged autodetection only removes the need for a hardcoded versioned path. The issue is paired with the open closing PR https://github.com/openclaw/openclaw/pull/42771, so it should stay open until that PR is refreshed, landed, or explicitly closed.

## What This Changes

The issue asks Docker images to store Playwright browser assets outside the hidden home cache and expose a stable browser binary on PATH.

## Best Possible Solution

Review, refresh, and either land or close https://github.com/openclaw/openclaw/pull/42771 or a maintainer replacement with a chosen Docker browser path contract, migration docs, Dockerfile tests, and real Docker proof for browser-enabled and browser-disabled images.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.20` shows Docker still exports `/home/node/.cache/ms-playwright` and lacks Dockerfile-created `chrome` or `chromium` PATH commands; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. Current autodetection is the narrowest fix for brittle versioned paths, while moving the Docker store and adding PATH aliases is a compatibility-affecting packaging choice that should be settled through the linked PR or a maintainer replacement.
