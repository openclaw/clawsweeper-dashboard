## Summary

Keep open. Current main solves the brittle versioned executable-path problem for OpenClaw by auto-detecting Playwright-managed Chromium, but it still stores Docker browser assets under `/home/node/.cache/ms-playwright` and does not expose a Dockerfile-created `chrome` or `chromium` command on PATH. This issue is also paired with the open closing PR at https://github.com/openclaw/openclaw/pull/42771, so maintainers should decide that path before cleanup.

## What This Changes

Request Docker browser-enabled images to expose a stable Chrome/Chromium command and store Playwright browser assets outside the hidden node cache.

## Best Possible Solution

Maintain the shipped cache-autodetection behavior unless maintainers explicitly choose a Docker packaging migration; if they do, refresh or replace https://github.com/openclaw/openclaw/pull/42771 with docs, contract coverage, and Docker build/run proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Docker build: current main still sets the Docker browser cache to `/home/node/.cache/ms-playwright` and lacks a Dockerfile-created `chrome` or `chromium` symlink. I did not run Docker in this read-only review.

## Solution Assessment

Unclear. Current main's autodetection is a narrow maintainable fix for the brittle versioned path, but moving the Docker browser cache to `/opt` and exposing PATH commands is a packaging/migration decision that should be handled through the linked PR or a maintainer replacement.
