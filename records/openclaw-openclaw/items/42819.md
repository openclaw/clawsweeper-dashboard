## Summary

Keep open. Current main and the latest release still install Docker Playwright Chromium under `/home/node/.cache/ms-playwright` and do not create a `chrome` or `chromium` PATH alias; the merged autodetection work only solved OpenClaw's brittle versioned executable discovery, while the direct target-repo PR was closed unmerged for proof and compatibility reasons rather than as not planned.

## What This Changes

The issue asks Docker images to install Playwright browsers outside the hidden home cache and expose a stable Chrome/Chromium binary on PATH.

## Best Possible Solution

Maintainers should decide the Docker browser path contract; if changing it, land a focused target-repo fix with migration docs, stable `chrome`/`chromium` PATH behavior, Dockerfile coverage, and browser-enabled plus browser-disabled image proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the Dockerfile still exports the hidden Playwright cache path and lacks Dockerfile-created `chrome` or `chromium` commands on PATH; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. Current main is the narrowest shipped solution for brittle versioned executable discovery, but moving the Docker store to `/opt` and adding PATH aliases needs a maintainer-approved compatibility and migration plan.
