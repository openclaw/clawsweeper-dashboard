## Label Justifications

- `P2`: The issue described a real but bounded regression affecting browser-enabled Docker workflows.
- `impact:other`: The concrete impact was brittle Docker browser executable discovery, outside the narrower owned impact categories.

## Summary

Current main and v2026.6.5 automatically discover the revisioned Playwright-managed Chromium installed in the Docker image, so users no longer need to configure a changing absolute executable path; the alternative `/opt` and PATH-symlink design is unnecessary for the reported failure and carried mount-compatibility hazards.

## What This Changes

Request stable Docker Chromium access without revision-specific cache paths, preferably with Playwright browser assets outside the hidden node cache.

## Best Possible Solution

Keep automatic discovery of Playwright-managed Chromium as the stable interface, avoiding revision-specific user configuration and global symlinks; reconsider the storage location only if a distinct permissions or persistence failure is reported.

## Reproduction Assessment

Yes at source level: the reported revision-specific Playwright layout is explicit, and the shipped regression test exercises that exact directory shape. I did not rebuild the Docker image during this read-only review, but the merged related PR includes positive live container proof.

## Solution Assessment

Yes. Runtime discovery is narrower and more maintainable than relocating the browser cache or adding global symlinks, and it avoids coupling executable availability to volume mounts and Playwright's revision directory names.
