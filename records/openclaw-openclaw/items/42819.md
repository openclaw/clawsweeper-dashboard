## Summary

Keep open. Current main partially addresses the brittle versioned path by auto-detecting Playwright-managed Chromium, but it still installs Docker browsers under `/home/node/.cache/ms-playwright` and does not create a stable `chrome` or `chromium` command on PATH. The same-author implementation PR https://github.com/openclaw/openclaw/pull/42771 is still open with closing syntax, so this issue should stay open until that PR or a maintainer replacement is resolved.

## What This Changes

Request Docker browser-enabled images to store Playwright Chromium outside the hidden node cache and expose a stable browser command on PATH.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/42771 or a maintainer replacement after deciding whether the Docker contract should migrate to `/opt` and PATH symlinks, while preserving the current Playwright cache autodetection fix.

## Reproduction Assessment

Yes. Source inspection of current main shows the browser-enabled Dockerfile still installs into `/home/node/.cache/ms-playwright` and only exposes the `openclaw` CLI on PATH; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear. Current main's autodetection is a narrower fix for the brittle versioned executable path, but moving the Docker browser store to `/opt` and adding PATH symlinks is a broader packaging contract that needs maintainer path-selection before refresh or replacement.
