## Summary

Keep open: current main and v2026.6.6 solve the brittle versioned Playwright executable lookup by auto-detecting the Docker cache, but they still keep Docker browsers under `/home/node/.cache/ms-playwright` and do not expose stable `chrome` or `chromium` commands on PATH.

## What This Changes

The issue requests Docker browser-enabled images to expose a stable Chrome/Chromium executable on PATH and store Playwright browser files outside the hidden node cache directory.

## Best Possible Solution

Make a maintainer-approved Docker browser packaging decision: either keep the home-cache path with auto-detection, or intentionally move to `/opt` plus PATH symlinks with migration guidance, docs/tests, and Docker proof for browser-enabled and browser-disabled images.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.6 still set the Docker Playwright browser path to `/home/node/.cache/ms-playwright` and do not create Dockerfile `chrome` or `chromium` PATH symlinks. I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear as an implementation direction: current main is a good fix for the brittle versioned executable lookup, but the remaining `/opt`/PATH contract changes Docker upgrade behavior and needs maintainer approval before a fix shape is accepted.
