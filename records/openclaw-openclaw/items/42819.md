## Summary

Keep open: current main and the latest release solve the brittle versioned Playwright executable lookup through auto-detection, but the requested Docker image contract still is not implemented because the image keeps browsers under `/home/node/.cache/ms-playwright` and does not expose stable `chrome` or `chromium` commands on PATH.

## What This Changes

The issue requests Docker browser-enabled images to store Playwright browser files outside the hidden `.cache` path and expose a stable Chrome/Chromium binary on PATH.

## Best Possible Solution

Keep the shipped Playwright-cache auto-detection, then land a maintainer-approved Dockerfile/docs/test change only if OpenClaw wants a new `/opt` browser-store and PATH-command contract with upgrade guidance and Docker proof.

## Reproduction Assessment

Yes by source inspection. Current main and `v2026.6.6` still export `/home/node/.cache/ms-playwright` from the Dockerfile and no Dockerfile-created stable `chrome` or `chromium` PATH command is present; I did not run a live Docker build in this read-only review.

## Solution Assessment

Unclear as a product direction. The merged auto-detection path is the narrow fix for brittle versioned executable paths, while moving the browser store to `/opt` and exposing PATH commands is a compatibility-sensitive Docker packaging choice.
