## Summary

Keep open. Current main now auto-detects Playwright-managed Chromium in the Docker cache, but the requested Docker packaging contract is still not present: the image still uses `/home/node/.cache/ms-playwright`, does not create a stable `chrome` or `chromium` command on PATH, and the paired open implementation PR remains unresolved.

## What This Changes

Request Docker browser-enabled builds to expose a stable Chrome/Chromium command and install or persist Playwright browser assets outside the hidden node cache path.

## Best Possible Solution

Maintainers should decide the Docker packaging contract, then either refresh and land https://github.com/openclaw/openclaw/pull/42771 or replace it with a focused fix that includes Dockerfile contract coverage, docs, and real Docker build/run proof.

## Reproduction Assessment

Yes by source inspection for the remaining Docker packaging request: current main still sets `PLAYWRIGHT_BROWSERS_PATH` to `/home/node/.cache/ms-playwright` and has no Dockerfile-created `chrome` or `chromium` symlink. I did not run a Docker build in this read-only review.

## Solution Assessment

Unclear. The merged cache autodetection is a narrow fix for the brittle versioned path, but moving the browser payload to `/opt` and adding a PATH command is a packaging and migration decision that should be handled through the paired PR or a maintainer replacement.
