## Summary

Keep this PR open: the private-session-directory security goal is still valid on current main, but the branch is not merge-ready because its managed/custom classifier can use a different on-disk config snapshot than the caller's actual store target, and final-head real behavior proof is still missing.

## What This Changes

The PR adds private-permission session directory helpers and routes setup, onboarding, agent creation, session store, transcript, lock, and Pi runner session directory creation through managed/custom directory handling.

## Best Possible Solution

Resolve managed-versus-custom ownership with the session target, pass that closed fact through lock, store, transcript, setup, and agent creation paths, and harden only explicitly managed session directories with upgrade proof.

## Reproduction Assessment

Yes, at source level: current main creates managed session store, transcript, setup, onboarding, and lock directories with ordinary recursive mkdir, so POSIX modes remain umask-dependent. I did not run a live before/after proof in this read-only review.

## Solution Assessment

No in its current form; the privacy objective is correct, but rediscovering managed ownership from ambient config is the wrong layer. The safer fix is to carry managed/custom provenance from target resolution into the write helpers.
