## Summary

Keep open: current main still creates several managed session directories with plain recursive mkdir, so the hardening is not obsolete, but this PR is not merge-ready because the new managed/custom classifier reloads ambient config inside the path helper and can choose the wrong chmod policy for callers that resolved paths from another config snapshot.

## What This Changes

The PR adds private-permission helpers for managed session directories and routes setup, onboarding, agent creation, session store, transcript, lock, and Pi runner flows through them while trying to preserve plain mkdir behavior for custom paths.

## Best Possible Solution

Thread an explicit managed/custom ownership decision or resolved store template from config resolution into store, transcript, lock, and bootstrap helpers; then refresh against current main and add redacted terminal or log proof for fresh, upgrade, custom-path, and symlink cases.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main still creates managed session store, setup, and lock directories with plain recursive mkdir, and PR head's classifier can be traced to ambient config loading without needing a live filesystem repro.

## Solution Assessment

Is this the best way to solve the issue? No. The hardening direction is right, but permission policy should be carried from the caller's resolved config or ownership mode instead of rediscovered from ambient config inside shared path helpers.
