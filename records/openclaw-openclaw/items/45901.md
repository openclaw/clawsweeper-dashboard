## Summary

Keep open. The security goal is valid and current main still uses ordinary recursive mkdir for session paths, but this branch is currently conflicting and still has a source-backed blocker: the PR-head classifier reloads ambient config inside low-level path checks, so permission policy can diverge from the caller's explicit or in-memory session target.

## What This Changes

The PR hardens managed session directory creation across setup, onboarding, agent creation, session store, transcript, lock, and Pi runner paths while trying to preserve plain mkdir for caller-supplied custom paths.

## Best Possible Solution

Refresh the branch, then make session target resolution return a closed managed/custom ownership result and pass that provenance through setup, agent creation, store, transcript, and lock paths; keep custom directories on plain mkdir unless maintainers explicitly approve stricter policy with upgrade proof.

## Reproduction Assessment

The PR-head regression risk is source-reproducible from the ambient config classifier, but I did not run a live filesystem reproduction in this read-only review.

## Solution Assessment

No. The privacy goal is appropriate for managed session storage, but rediscovering ownership from ambient config inside path helpers is not the best fix; carrying managed/custom provenance from the resolver that selected the target is narrower and safer.
