## Summary

Keep open: current main still has the managed session-directory privacy gap and this PR contains useful hardening, but the branch is conflicting/stale and would regress current session write-lock contracts before it can merge.

## What This Changes

The PR adds private managed session-directory creation helpers, ports setup/onboarding/probe/agent/session-store/transcript/lock callers to use them, and adds permission and symlink-path tests.

## Best Possible Solution

Land a rebased or replacement PR that ports the managed-session directory hardening onto current main while preserving the current lock timeout, non-reentrant lock, session-store writer, and custom-path contracts.

## Reproduction Assessment

Yes. Source inspection on current main shows fresh managed session paths still use bare recursive mkdir; on a POSIX host with a normal umask, the repro is to create a fresh managed session directory through setup or session-store creation and inspect its mode.

## Solution Assessment

No. The hardening direction is right, but this branch is not the best merge vehicle until it is rebased or replaced without regressing current session write-lock and store-writer contracts.
