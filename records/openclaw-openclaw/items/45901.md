## Summary

Keep open: the security objective is valid and current main still lacks private managed session-directory creation, but this PR is not merge-ready because it still classifies managed paths by reloading ambient on-disk config, lacks real behavior proof, and is currently dirty against main.

## What This Changes

The PR adds private managed session-directory creation plus symlink and identity hardening across setup, onboarding, agent creation, session store, transcript, and lock paths.

## Best Possible Solution

Resolve managed-versus-custom ownership when deriving session targets, pass that closed fact through store, transcript, and lock writes, keep caller-supplied custom paths on plain mkdir, and add final-head real behavior proof for fresh, upgrade, custom, and alias layouts.

## Reproduction Assessment

Yes at source level: current main uses ordinary recursive mkdir in session store and lock paths, so POSIX modes depend on umask rather than a private-directory guarantee. I did not run a live reproduction because this review was read-only.

## Solution Assessment

No in its current form; the privacy goal is correct, but rediscovering managed status from ambient config is not the best owner boundary. The safer path is to carry managed/custom provenance from target resolution into every write path.
