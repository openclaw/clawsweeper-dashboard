## Summary

Keep open: current main and v2026.6.8 still have mode-less managed session directory creation, so the central security hardening remains useful. ## What This Changes

The PR adds private managed session-directory helpers, routes setup/onboarding/agent/session store/transcript/lock/probe paths through them, and adds regression tests for modes, symlink handling, and custom path preservation.

## Best Possible Solution

Refresh or replace this with a narrow fix that passes explicit managed/custom ownership or the resolved store template through the session helpers, preserves custom paths, and includes final-head proof for fresh, upgrade, and symlink cases.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source for the underlying bug class: current main and v2026.6.8 still create managed session/setup/store/lock directories with recursive mkdir and no private mode; I did not run a live filesystem repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The hardening direction is right, but shared path helpers should not choose permission policy from ambient config when callers already resolved concrete store paths from specific config snapshots.
