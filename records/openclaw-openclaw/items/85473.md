## Summary

Keep open: current main and v2026.6.9 still route dreaming through memory-core as the engine, with only slot-owned config reading and ClawHub workaround/docs around LanceDB. The remaining request is a public plugin protocol/product decision, not implemented or superseded.

## What This Changes

Request to define a plugin-facing dreaming protocol so non-memory-core memory plugins can own Light/Deep/REM consolidation while OpenClaw schedules and orchestrates phases.

## Best Possible Solution

Settle a narrow, documented memory-dreaming plugin contract, or explicitly decide that role-slot selection plus external ClawHub plugins is the product boundary, while preserving memory-core default behavior and upgrade safety.

## Reproduction Assessment

Not applicable as a feature/API request; source inspection is enough to show current main lacks the requested public lifecycle methods even though it can read dreaming settings from the selected memory slot.

## Solution Assessment

Unclear until maintainers choose the product/API direction; the maintainable path is a documented SDK/role-slot contract with contract tests, not an ad hoc branch that bypasses plugin SDK boundaries.
