## Summary

Keep open: current main has slot-aware dreaming config and a memory-core sidecar, but no plugin-facing dreaming lifecycle contract; the request still needs maintainer API/product design before implementation.

## What This Changes

Request a plugin-facing dreaming protocol so alternate memory plugins can implement Light, Deep, and REM consolidation without depending on memory-core private file layouts.

## Best Possible Solution

Define a versioned MemoryPluginCapability.dreaming contract, keep the existing memory-core sidecar behavior compatible during migration, and update memory-core, docs, UI, and contract tests as the reference implementation.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that the current memory plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A typed SDK capability is the maintainable direction; more memory-core private-file exceptions would preserve the coupling this request is trying to remove.
