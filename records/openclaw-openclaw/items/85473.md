## Summary

Keep open: current main has partial memory-slot awareness for dreaming config, but the dreaming engine, cron sidecar, and storage lifecycle are still memory-core/file-layout based, with no plugin-facing dreaming protocol for alternate memory plugins.

## What This Changes

The issue asks for dreaming to become a memory-plugin protocol so non-memory-core backends can implement the dreaming lifecycle.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract and migrate memory-core to implement it first, with docs/UI/tests updated so external memory plugins can opt in without relying on memory-core private files.

## Reproduction Assessment

Not applicable as a normal bug repro; this is a feature/API request. Source inspection verifies the current plugin contract has no dreaming lifecycle methods and memory-core still runs the file-backed sweep.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
