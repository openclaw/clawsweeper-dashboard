## Summary

Keep open: current main and v2026.6.9 still provide selected-memory-slot dreaming config plus a memory-core sidecar, not a documented lifecycle protocol that arbitrary memory plugins can implement. The ClawHub LanceDB dreaming plugin is useful for affected users today, and the open memory role-slot PR is related, but the requested core/plugin-SDK product decision remains unresolved.

## What This Changes

Request to expose dreaming as a memory-plugin protocol with shared config, scheduler orchestration, and lifecycle methods for non-memory-core memory plugins.

## Best Possible Solution

Keep this as the canonical design item until maintainers either bless a ClawHub-only model for LanceDB dreaming or sponsor a documented plugin-SDK dreaming lifecycle that preserves memory-core defaults and adds compatibility tests/docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection gives high confidence that current main lacks a general dreaming lifecycle protocol for arbitrary memory plugins.

## Solution Assessment

Unclear until maintainers decide the API boundary. A standalone ClawHub plugin works for LanceDB users today, but a core-managed dreaming protocol would be a broader public plugin contract requiring SDK, docs, scheduler, and compatibility design.
