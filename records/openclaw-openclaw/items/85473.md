## Summary

Keep open: current main and v2026.6.9 still provide selected-memory-slot dreaming config plus a memory-core sidecar, not a documented dreaming lifecycle protocol that arbitrary memory plugins can implement. The ClawHub LanceDB dreaming plugin and the open memory role-slot PR are useful related work, but they leave the core/plugin-SDK API decision unresolved.

## What This Changes

Request to expose a plugin-facing dreaming protocol, config surface, and runtime orchestration so memory plugins other than memory-core can implement Light/Deep/REM consolidation.

## Best Possible Solution

Keep this as the canonical design item until maintainers either choose a ClawHub-only model or sponsor a narrow documented plugin-SDK dreaming lifecycle contract that preserves memory-core defaults and adds compatibility tests/docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection gives high confidence that current main lacks a general dreaming lifecycle protocol for arbitrary memory plugins.

## Solution Assessment

Unclear until maintainers decide the API boundary. A standalone ClawHub plugin works for LanceDB users today, while a core-managed dreaming protocol would be a broader public plugin contract requiring SDK, docs, scheduler, and compatibility design.
