## Summary

Keep open: current main has partial memory-slot awareness for dreaming config, but the actual dreaming engine and storage lifecycle are still memory-core/file-layout based and there is no plugin-facing dreaming protocol for alternate memory plugins.

## What This Changes

Request a memory plugin dreaming protocol so non-memory-core memory plugins can declare config and implement light/deep/REM storage and consolidation hooks.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract and migrate memory-core to implement it first, with docs/UI/tests updated so external memory plugins can opt in without relying on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms the current plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
