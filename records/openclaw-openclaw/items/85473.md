## Summary

Keep open: current main only partially decouples dreaming config/loading; the runtime still loads memory-core as the sidecar engine and uses memory-core file-backed state. The requested public protocol is a real plugin API design item, not stale or already implemented.

## What This Changes

Expose a plugin-facing dreaming protocol so alternate memory plugins can own the dreaming lifecycle instead of depending on memory-core private files.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core to implement it first, and keep current sidecar compatibility until alternate memory plugins can opt in cleanly.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms the current plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent SDK and storage boundary. A narrow plugin capability contract is the maintainable direction; adding more memory-core sidecar exceptions would preserve the coupling this issue is trying to remove.
