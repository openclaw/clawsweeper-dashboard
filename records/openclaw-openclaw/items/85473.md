## Summary

Keep open: current main and v2026.6.9 still provide memory-core-owned dreaming plus selected-memory-slot config/sidecar behavior, not a documented dreaming lifecycle protocol that arbitrary memory plugins can implement. The ClawHub LanceDB dreaming plugin is useful for affected users today, and the open role-slot PR is related, but both leave the core/plugin-SDK product decision unresolved.

## What This Changes

Request to expose dreaming as a memory plugin protocol so non-memory-core memory plugins can provide query, promotion, reporting, and storage behavior while OpenClaw orchestrates scheduling and phases.

## Best Possible Solution

Keep this as the canonical design item until maintainers either bless a ClawHub-only model or sponsor a narrow documented plugin-SDK dreaming lifecycle contract that preserves memory-core defaults and adds compatibility tests/docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection gives high confidence that current main lacks a general dreaming lifecycle protocol for arbitrary memory plugins.

## Solution Assessment

Unclear until maintainers decide the API boundary. A standalone ClawHub plugin works for LanceDB users today, but a core-managed dreaming protocol would be a broader public plugin contract requiring SDK, docs, scheduler, and compatibility design.
