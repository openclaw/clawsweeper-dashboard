## Summary

Keep open: current main supports loading memory-core as a dreaming sidecar for an alternate memory slot, but it still does not expose a plugin-facing dreaming lifecycle protocol; the engine, storage, cron, and docs remain memory-core/file-layout centered.

## What This Changes

The issue requests a public memory-plugin dreaming protocol so alternate memory plugins can implement query, promotion, state, config, and lifecycle hooks instead of relying on memory-core internals.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core to implement it first, then update docs, config/UI/status, loader tests, and external-plugin guidance so alternate memory plugins can opt in without reading memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that the current memory plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent public SDK shape. The maintainable direction is a versioned memory-plugin capability, while adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
