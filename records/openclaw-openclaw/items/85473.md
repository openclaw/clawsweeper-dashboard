## Summary

Keep open: current main has partial slot-aware dreaming config and memory-core sidecar loading, but the dreaming lifecycle is still implemented by memory-core against its private file/state layout and there is no public memory-plugin dreaming protocol for external plugins such as memory-lancedb-pro.

## What This Changes

Request a plugin-facing dreaming protocol so any memory plugin can own Light/Deep/REM storage, promotion, and state reporting instead of relying on memory-core internals.

## Best Possible Solution

Define a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core to implement it first, and update docs/UI/tests so external memory plugins can opt in without relying on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms current main has no plugin-facing dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would preserve the current private-file coupling.
