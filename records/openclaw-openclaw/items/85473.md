## Summary

Keep open: current main has slot-aware dreaming config and a memory-core sidecar for non-core memory slots, but it still lacks a plugin-facing dreaming lifecycle contract; this needs maintainer API/product design before implementation.

## What This Changes

The issue requests a plugin-facing dreaming protocol so memory plugins other than memory-core can participate in light/deep/REM consolidation.

## Best Possible Solution

Define a versioned memory dreaming capability in the plugin contract, keep memory-core behavior compatible during migration, and make memory-core the reference implementation with docs and contract tests.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that current main lacks a plugin-facing dreaming lifecycle contract.

## Solution Assessment

Unclear until maintainers choose the permanent API shape. A typed memory plugin capability is likely the maintainable direction, but scheduler ownership, UI/status behavior, and migration compatibility need design approval first.
