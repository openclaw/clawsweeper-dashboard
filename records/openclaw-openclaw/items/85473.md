## Summary

Keep open: current main and the latest release still expose dreaming as a memory-core-owned engine with slot-aware config and a memory-core sidecar, not as a plugin-facing dreaming lifecycle protocol. The newly linked ClawHub package is useful external evidence that users are working around the gap, but it does not replace the requested core SDK/API design.

## What This Changes

The issue requests a memory plugin dreaming protocol so non-memory-core memory plugins can participate in Light, REM, and Deep consolidation.

## Best Possible Solution

Define a versioned MemoryPluginCapability.dreaming contract, keep memory-core sidecar behavior compatible during migration, and update memory-core, docs, UI/status, and contract tests as the reference implementation.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that the current memory plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A typed SDK capability is the maintainable direction; more memory-core private-file exceptions would preserve the coupling this issue is trying to remove.
