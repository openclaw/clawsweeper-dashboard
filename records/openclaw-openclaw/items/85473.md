## Summary

Keep open: current main supports running memory-core as a dreaming sidecar for another selected memory plugin, but there is still no plugin-facing dreaming lifecycle protocol; the storage and phase implementation remain memory-core/file-layout owned.

## What This Changes

Request to expose dreaming as a memory plugin protocol so alternate memory plugins can implement the consolidation lifecycle instead of relying on memory-core internals.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core onto it first, and update docs/UI/tests so external memory plugins can opt in without depending on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms current main lacks plugin-facing dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
