## Summary

Keep open: current main and the latest release still route dreaming through memory-core rather than a plugin-facing dreaming lifecycle contract. The new ClawHub package is a useful external workaround for LanceDB users, but it does not implement the requested OpenClaw memory-plugin protocol or update the built-in `/dreaming` and Dreams UI assumptions.

## What This Changes

Request a public memory-plugin dreaming protocol so non-memory-core memory plugins can participate in Light, REM, and Deep consolidation.

## Best Possible Solution

Settle a versioned memory dreaming capability in the Plugin SDK, keep memory-core sidecar behavior compatible during migration, and use memory-core plus an alternate memory plugin as contract-test implementations.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that current main has no plugin-facing dreaming lifecycle contract.

## Solution Assessment

Unclear until maintainers choose the permanent API shape. A typed, documented SDK capability with compatibility tests is the maintainable direction; adding more memory-core private-file exceptions would keep the coupling this issue reports.
