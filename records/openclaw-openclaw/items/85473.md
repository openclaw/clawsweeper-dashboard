## Summary

Keep open: current main solves the older slot-coexistence problem by loading memory-core as a dreaming sidecar, but it still does not expose a plugin-facing dreaming lifecycle that alternate memory plugins can implement. The linked memory-lancedb-pro issues show this is an active external plugin gap, and the remaining work is a public SDK/product-boundary decision rather than stale cleanup.

## What This Changes

The issue requests a plugin-facing dreaming protocol so memory plugins other than memory-core can implement scheduled Light, REM, and Deep consolidation.

## Best Possible Solution

Define a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core to implement it first, and update docs/UI/tests so external memory plugins can opt in without depending on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms current main has no plugin-facing dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
