## Summary

Keep open: current main and v2026.6.8 still provide memory-core-owned dreaming plus selected-slot config/sidecar support, not a public dreaming lifecycle protocol that any memory plugin can implement. The ClawHub LanceDB dreaming plugin and the open memory role-slot PR are useful related paths, but neither replaces the remaining core/plugin-SDK product decision.

## What This Changes

Request to expose dreaming as a memory-plugin protocol so non-memory-core memory plugins can own Light, REM, and Deep consolidation lifecycle behavior.

## Best Possible Solution

Maintain this as the canonical product/API design item until maintainers either bless the ClawHub-only model or land a narrow documented plugin-SDK dreaming lifecycle contract with compatibility tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection does give high confidence that current main lacks a general dreaming lifecycle protocol for arbitrary memory plugins.

## Solution Assessment

Unclear until maintainers decide the API boundary. A standalone ClawHub plugin works for LanceDB users today, while core-managed scheduling/provider hooks would be a broader public plugin contract.
