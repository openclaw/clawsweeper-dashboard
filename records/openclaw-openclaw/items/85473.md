## Summary

Keep open: current main and the latest release still expose dreaming as memory-core-owned sidecar/runtime behavior, not as a general memory-plugin lifecycle protocol. The open multi-slot memory-role PR is relevant groundwork but explicitly leaves the richer dreaming protocol out of scope, and the ClawHub LanceDB plugin is a useful workaround rather than an OpenClaw core implementation.

## What This Changes

The issue requests a plugin-facing dreaming protocol so non-memory-core memory plugins can implement Light, Deep, and REM consolidation.

## Best Possible Solution

Make a maintainer design decision first: either bless external ClawHub dreaming plugins as the supported model, or define a narrow SDK dreaming lifecycle API with config/docs/tests and upgrade-safety proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection gives a high-confidence current-main check that a general plugin-implemented dreaming lifecycle protocol is still absent.

## Solution Assessment

Unclear until maintainers choose the API boundary. A standalone ClawHub plugin addresses LanceDB users today, while a core-managed scheduler/provider protocol would create a broader public plugin contract that needs design approval.
