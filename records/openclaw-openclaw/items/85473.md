## Summary

Keep open: current main still treats dreaming as memory-core-owned runtime behavior, while the ClawHub LanceDB dreaming plugin shows a useful workaround rather than a core plugin-facing dreaming lifecycle contract. The remaining decision is product/API ownership, not a narrow bug fix.

## What This Changes

Request to expose dreaming as a memory plugin protocol so non-memory-core memory plugins can participate in Light, Deep, and REM consolidation.

## Best Possible Solution

Decide the memory plugin contract first: either bless the ClawHub plugin path as the supported extension model, or land a narrow core/plugin-SDK dreaming lifecycle API with compatibility tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection and the issue discussion do show current main lacks a general dreaming lifecycle protocol for arbitrary memory plugins.

## Solution Assessment

Unclear until maintainers decide the API boundary. A standalone ClawHub plugin works for LanceDB users today, but a core-managed scheduler/provider protocol would be a broader public plugin contract and should not be treated as a narrow repair.
