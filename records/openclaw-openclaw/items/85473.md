## Summary

Keep open: current main has partial config and sidecar coexistence for alternate memory slots, but the dreaming lifecycle still runs through memory-core file-based internals and no plugin-facing dreaming protocol exists.

## What This Changes

Request to expose dreaming as a memory plugin protocol so alternate memory plugins can query, promote, and report dreaming state without relying on memory-core private files.

## Best Possible Solution

Add a narrow, versioned MemoryPluginCapability.dreaming contract and migrate memory-core to implement it first, with docs, UI, loader, and tests updated so external memory plugins can opt in without relying on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms the current plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
