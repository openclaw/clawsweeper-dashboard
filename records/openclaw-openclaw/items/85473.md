## Summary

Keep open: current main has slot-aware dreaming config and sidecar loading, but the dreaming lifecycle still lives in memory-core internals and the public memory plugin capability has no query/promote/status protocol for alternate memory plugins.

## What This Changes

Request to expose dreaming as a memory plugin protocol so non-memory-core memory plugins can implement storage-backed Light/Deep/REM participation while OpenClaw orchestrates scheduling.

## Best Possible Solution

Define a narrow, versioned MemoryPluginCapability.dreaming contract, migrate memory-core to implement it first, and document/test how alternate memory plugins opt in without depending on memory-core private files.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection confirms the current plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A narrow SDK capability is the maintainable direction; adding more memory-core sidecar exceptions would keep the coupling this issue is trying to remove.
