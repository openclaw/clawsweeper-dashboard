## Summary

Keep open: current main and the latest release still expose memory dreaming as memory-core-owned behavior rather than a plugin-facing dreaming lifecycle, so the external memory-plugin problem remains and needs maintainer API/product design.

## What This Changes

Request a plugin-facing dreaming protocol so non-memory-core memory plugins can implement light/deep/REM consolidation instead of relying on memory-core internals.

## Best Possible Solution

Define a versioned MemoryPluginCapability.dreaming contract, keep memory-core behavior compatible as the reference implementation, and update SDK/docs/UI/status/contract tests around that stable boundary.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that the current memory plugin contract has no dreaming lifecycle methods.

## Solution Assessment

Unclear until maintainers choose the permanent plugin API shape. A typed SDK capability with memory-core as the reference implementation is the maintainable direction; more memory-core private-file exceptions would preserve the coupling this issue is trying to remove.
