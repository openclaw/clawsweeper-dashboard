## Summary

## What This Changes

The PR routes `channels status --timeout` through the shared CLI timeout parser and adds command-flow tests for invalid timeout input plus the probe-mode default.

## Best Possible Solution

Rebase the branch on current main, keep the shared-parser fix with the 10s and 30s fallbacks, update the assertion to current parser text, preserve channel filtering, and add redacted real CLI output proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main still uses raw `Number(...)` in `channelsStatusCommand`, so invalid timeout input is not rejected at the command boundary before the gateway request is built.

## Solution Assessment

Mostly yes. Reusing `parseTimeoutMsWithFallback(..., { invalidType: "error" })` is the narrow maintainable fix, but this branch needs the current parser assertion, a rebase, and real behavior proof before merge.
