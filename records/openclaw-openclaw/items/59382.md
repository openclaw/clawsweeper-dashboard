## Summary

Keep open. Current main still parses `channels status --timeout` with raw `Number(...)`, so the requested validation is not implemented on main; the PR is a narrow and appropriate fix, but it is still missing the required user-facing changelog entry before landing.

## What This Changes

The PR changes `channels status` to parse `--timeout` with `parseTimeoutMsWithFallback(..., { invalidType: "error" })`, adds default timeout constants, and adds command-flow tests for probe fallback and invalid timeout rejection.

## Best Possible Solution

Add the required changelog entry, then land this PR or the equivalent focused change so `channels status` rejects invalid timeout input through the shared CLI parser before any gateway call. Treat the Commander probe-default question as a separate follow-up unless maintainers want to fold that behavior into this PR intentionally.

## Reproduction Assessment

Yes. A source-level reproduction on current main is `channelsStatusCommand({ timeout: "0" })`: line 169 parses it to `0` and forwards it toward the gateway instead of throwing the shared parser's `invalid --timeout: 0` error before any gateway work.

## Solution Assessment

Yes, with the changelog fixed. Reusing `parseTimeoutMsWithFallback(..., { invalidType: "error" })` at the command boundary is the narrowest maintainable fix for this CLI validation gap; changing gateway protocol clamping would be broader and would affect remote callers.
