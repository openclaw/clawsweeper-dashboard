## Summary

Keep this PR open. Current main and v2026.4.27 still parse `channels status --timeout` with raw `Number(...)` and forward that value into the gateway request, while the PR is a narrow command/test change that aligns the command with the existing shared timeout parser. I found no blocking correctness or security finding in the provided diff.

## What This Changes

This PR changes `channels status` to parse `--timeout` with `parseTimeoutMsWithFallback(..., { invalidType: "error" })` and adds command-flow coverage for probe default timeout and invalid timeout rejection.

## Best Possible Solution

Land this PR, or apply the equivalent focused change, so `channels status` rejects invalid timeout input through the shared parser before any gateway call while keeping the gateway protocol's existing timeout clamping for remote callers. Keep the focused command-flow tests with the option-validation describe block and confirm the intended probe default at the CLI boundary during maintainer review.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `channelsStatusCommand({ timeout: "0" })` or `channelsStatusCommand({ timeout: "not-a-number" })` on current main: the command uses raw `Number(...)` and reaches the gateway path instead of throwing the shared parser's `invalid --timeout` error first.

## Solution Assessment

Yes. The PR's approach is the narrowest maintainable fix for the command-surface bug because it reuses the existing parser rather than changing gateway protocol validation or duplicating timeout parsing logic.
