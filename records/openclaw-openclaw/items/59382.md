## Summary

Keep open. Current main still accepts invalid `channels status --timeout` input at the command boundary, and this PR is a narrow correct fix, but the user-facing CLI behavior change still needs the required changelog entry before merge.

## What This Changes

The PR changes `channels status` to use the shared CLI timeout parser and adds command-flow tests for probe defaults and invalid timeout rejection.

## Best Possible Solution

Land this focused CLI validation change with an Unreleased changelog entry, while keeping the gateway clamp as a defensive server-side guard.

## Reproduction Assessment

Yes. Source inspection of current main shows `channelsStatusCommand({ timeout: "0" })` computes `timeoutMs = 0` and forwards it toward the gateway instead of throwing the shared parser's `invalid --timeout: 0` error.

## Solution Assessment

Yes, with the changelog fixed. Reusing `parseTimeoutMsWithFallback(..., { invalidType: "error" })` at the command boundary is the narrowest maintainable fix; changing gateway protocol clamping would be broader and affect remote callers.
