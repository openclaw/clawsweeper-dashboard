## Summary

Keep open. Current main still has the reported behavior: the plugin-owned `inbound_claim` handled path returns no final-queued signal, Telegram still uses that false signal plus no delivered reply to synthesize the silent fallback, and the issue is paired with open unmerged PR #70288, so it should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue requests that plugin-owned async `inbound_claim` handling suppress Telegram's silent-reply fallback until the plugin's asynchronous response can arrive.

## Best Possible Solution

Land PR #70288 after expanding it, or land an equivalent maintainer patch, so a plugin-owned async claim produces a durable response-committed signal through the direct dispatch, buffered wrapper, and Telegram fallback path while preserving missing-plugin, no-handler, declined, and error behavior. Add regression coverage at the direct dispatch level, wrapper level, and Telegram fallback level.

## Reproduction Assessment

Yes. A plugin-owned binding whose targeted `inbound_claim` returns `{ handled: true }` currently produces `queuedFinal: false` with zero immediate delivery, which satisfies Telegram's silent-fallback condition before the plugin's async response arrives.

## Solution Assessment

No, current main is not the best solution because it still reproduces the regression. The proposed direction is plausible, but a fix that only flips the direct return value is incomplete unless the buffered dispatch wrapper and Telegram fallback semantics preserve that async-claim signal.
