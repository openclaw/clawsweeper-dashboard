## Summary

Keep open: current main and the latest release still preserve the reported mismatch between plugin-owned async inbound claims and Telegram's silent-reply fallback, and the only linked fix PR is closed unmerged.

## What This Changes

The issue reports Telegram sending a silent-reply fallback before a plugin-owned async response after `inbound_claim` returns `{ handled: true }`.

## Best Possible Solution

Add an explicit committed external-response signal for plugin-owned async claims, preserve it through dispatch finalization, and use it to suppress Telegram's silent fallback while keeping missing, declined, error, cancelled, and failed local-delivery paths unchanged.

## Reproduction Assessment

Yes, at source level. A handled plugin-owned claim returns `queuedFinal: false` with zero local final counts, and Telegram still has a no-visible-delivery group fallback path gated by false `queuedFinal`.

## Solution Assessment

No, current main is not the best solution because it still reproduces the regression shape. The reporter's one-line `queuedFinal` flip is incomplete unless dispatch finalization preserves zero-count external commitments; the linked PR's `finalResponseCommitted` contract is the safer narrow fix.
