## Summary

Keep open: current main and the latest release still preserve the reported mismatch between plugin-owned async inbound claims and Telegram's silent-reply fallback, and the only linked fix PR is closed unmerged.

## What This Changes

The issue reports that Telegram sends a silent-reply fallback before an asynchronously responding plugin finishes after `inbound_claim` returns `{ handled: true }`.

## Best Possible Solution

Add a small dispatch contract that explicitly marks plugin-owned async claims as a committed final response, preserve it through dispatch finalization, and use it to suppress Telegram's no-delivery silent fallback while leaving missing, declined, error, cancelled, and failed local delivery paths unchanged.

## Reproduction Assessment

Yes, at source level. A plugin-owned `inbound_claim` handled result returns `queuedFinal: false` with zero counts, and Telegram still has a no-delivery silent fallback path that runs when `queuedFinal` remains false.

## Solution Assessment

No, current main is not the best solution because it still reproduces the regression. The safest repair is the explicit committed async-response signal from the closed prior PR, rather than only flipping `queuedFinal` in one branch where the wrapper can erase it.
