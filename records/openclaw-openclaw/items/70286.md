## Summary

Keep open: current main and the latest release still preserve the reported plugin-owned async claim versus Telegram silent-reply mismatch, and the only linked fix PR was closed unmerged despite the current source still showing the failing signal path.

## What This Changes

The issue asks for plugin-bound `inbound_claim` handlers that return `{ handled: true }` to suppress Telegram silent-reply fallback while the plugin sends its async response.

## Best Possible Solution

Add an explicit internal committed external-response signal for plugin-owned async claims, preserve it through dispatch finalization, and use it to suppress Telegram’s silent fallback without changing missing-plugin, no-handler, declined, error, cancelled, or failed local-delivery behavior.

## Reproduction Assessment

Yes, at source level. A plugin-owned handled claim returns `queuedFinal: false` with zero local final counts, and Telegram still has a no-visible-delivery group fallback path gated by false `queuedFinal`.

## Solution Assessment

Yes, the narrow maintainable direction is an explicit committed-response signal rather than only flipping `queuedFinal`, because the dispatch finalizer currently ties `queuedFinal` to local final counts.
