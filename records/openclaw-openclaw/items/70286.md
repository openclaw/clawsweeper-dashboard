## Summary

Keep open: current main still has the reported behavior, and the linked open PR is the implementation candidate rather than a reason to close the issue early.

## What This Changes

Prevent the Telegram silent-reply fallback from firing after a plugin-owned `inbound_claim` handler returns `{ handled: true }` and responds asynchronously.

## Best Possible Solution

Land PR #70288 after expanding the contract as needed, or land an equivalent maintainer patch that preserves an async plugin-claim signal through direct dispatch, the buffered wrapper, and Telegram fallback handling with regression coverage.

## Reproduction Assessment

Yes. A plugin-owned binding whose targeted `inbound_claim` returns `{ handled: true }` with no immediate reply produces `queuedFinal: false`; Telegram then satisfies its no-delivery fallback condition before the plugin's async response arrives.

## Solution Assessment

No, current main is not the best solution because it still reproduces the regression. The proposed direct flip is directionally useful, but the wrapper path needs to preserve the async-claim signal instead of reducing it to a final-message count.
