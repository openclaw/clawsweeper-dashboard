## Summary

Keep open: current main still has the reported source-level mismatch. Plugin-owned handled claims report no committed final response, the wrapper would erase a zero-count `queuedFinal`, and Telegram can still enter the silent-reply fallback path; the linked fix PR is closed unmerged.

## What This Changes

The issue requests suppressing Telegram silent-reply fallback when a plugin-bound `inbound_claim` handler returns `{ handled: true }` and responds asynchronously.

## Best Possible Solution

Add an internal committed external-response signal for plugin-owned async claims, preserve it through dispatch finalization, and use it to suppress Telegram’s silent fallback without changing missing-plugin, no-handler, declined, error, cancelled, or failed local-delivery behavior.

## Reproduction Assessment

Yes, at source level. A plugin-owned handled claim still returns `queuedFinal: false`, finalization still requires a local final count, and Telegram still has a group fallback path gated by false `queuedFinal`.

## Solution Assessment

Yes for the broader direction, but the one-line `queuedFinal: true` proposal is incomplete because finalization can erase it when local final counts are zero. The maintainable fix is an explicit committed external-response signal carried through dispatch and checked by Telegram fallback logic.
