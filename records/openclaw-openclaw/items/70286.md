## Summary

Keep open: current main still has the reported plugin-owned async claim versus Telegram silent-reply mismatch, and the only linked fix PR is closed unmerged rather than a canonical landed fix.

## What This Changes

The issue reports that Telegram can send a silent-reply fallback before an async plugin response after a plugin `inbound_claim` handler returns `{ handled: true }`.

## Best Possible Solution

Add an explicit internal committed-response signal for plugin-owned async claims, preserve it through dispatch finalization, and have Telegram suppress silent fallback only for that signal while leaving declined, missing-plugin, no-handler, error, cancellation, and failed local delivery behavior unchanged.

## Reproduction Assessment

Yes, at source level. A plugin-owned handled claim returns `queuedFinal: false` with zero local final counts, and Telegram still has a group silent-fallback path gated by `!queuedFinal` and no visible delivery.

## Solution Assessment

Yes. The narrow maintainable fix is an explicit committed-response signal rather than only changing `queuedFinal`, because current dispatch finalization intentionally reconciles `queuedFinal` against local final delivery counts.
