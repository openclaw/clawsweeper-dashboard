## Label Justifications

- `P1`: This is a source-proven regression in a real Telegram plugin-bound channel workflow that can send spurious fallback messages before the actual plugin response.
- `impact:message-loss`: The issue is about incorrect channel message delivery behavior: a fallback reply is delivered even though the plugin-owned response is already committed asynchronously.

## Summary

Keep open: current main and the latest shipped release still preserve the reported mismatch, where plugin-owned `inbound_claim` handling reports no queued final response while Telegram can still send the silent-reply fallback. The linked fix PR at https://github.com/openclaw/openclaw/pull/70288 was closed unmerged, and exact follow-up searches only found this issue plus that closed PR.

## What This Changes

The issue asks OpenClaw to stop Telegram silent-reply fallback text from racing ahead of an async plugin response after a plugin-bound `inbound_claim` returns `{ handled: true }`.

## Best Possible Solution

Add an internal committed external-response signal for plugin-owned async claims, preserve it through dispatch finalization, and use it to suppress Telegram’s silent fallback without changing missing-plugin, no-handler, declined, error, cancelled, or failed local-delivery behavior.

## Reproduction Assessment

Yes, at source level. A plugin-owned handled claim returns `queuedFinal: false` with zero local final counts, dispatch finalization keeps that false, and Telegram’s no-visible-delivery fallback path is still gated on `!queuedFinal`.

## Solution Assessment

Yes, the best fix is not only flipping one boolean in the plugin branch; it should carry an explicit external-response commitment through dispatch finalization and let Telegram suppress only that fallback path.
