## Summary

Keep open: the PR is a focused Telegram cache performance change with real routing proof, but it introduces a blocking cache-format migration regression for existing current-main cache files. The protected `maintainer` label also means this should stay in maintainer handling rather than cleanup close.

## What This Changes

The branch changes Telegram reply-chain cache persistence from full JSON-array rewrites to newline-delimited append records with compaction, tests, and a changelog entry.

## Best Possible Solution

Keep the append-log design, but make the first read/append path safely load or compact the current JSON-array snapshot shape before switching the file to JSONL, with a regression test for that upgrade path.

## Reproduction Assessment

Yes, from source inspection: current main writes one JSON-array cache snapshot, while the PR reads only JSONL entries and appends without first converting the old file. I did not run a filesystem repro because this review must remain read-only.

## Solution Assessment

No, not yet: append records with bounded compaction are the right narrow performance direction, but the implementation should preserve current-main persisted cache files before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve existing snapshot cache files:** `extensions/telegram/src/message-cache.ts:173`
  - body: Current main persists this cache as a single JSON array with no trailing newline. With this reader, that existing line is ignored, and the first append is concatenated to it, so any gateway that already has a cache file reloads an empty or invalid cache until a later compaction. Please load or compact the old array shape before appending JSONL records.
