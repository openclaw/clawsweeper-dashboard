## Summary

Keep open. Current main supports the requested acknowledgement behavior through the recommended BlueBubbles iMessage path, but the still-documented legacy `channels.imessage`/`imsg` channel requested by the report has no ack/read-receipt capability, config/schema surface, or inbound acknowledgement call. Because legacy imsg is deprecated but still documented, this needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

The issue requests native acknowledgement support for iMessage inbound messages, either through read receipts or tapback-style acknowledgements, so senders can tell the agent received queued messages.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported for this feature, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and focused tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure legacy `channels.imessage` with top-level `messages.ackReactionScope` and trace accepted inbound handling: the legacy config/schema has no channel-owned ack/read field and the monitor dispatch path contains no acknowledgement call.

## Solution Assessment

Unclear. Native acknowledgement in deprecated `imsg` may be the right fix only if maintainers want to keep expanding that path; otherwise the safer solution is to document BlueBubbles as the iMessage acknowledgement path and explicitly leave legacy imsg without this feature.
