## Summary

Keep open. Current main has BlueBubbles read receipts/tapback acknowledgements and a generic inbound message hook, but the legacy `channels.imessage`/`imsg` path described here still lacks channel-owned acknowledgement behavior; whether to add that to a deprecated channel or document BlueBubbles as the supported answer needs maintainer product direction.

## What This Changes

Request native acknowledgement for iMessage inbound messages, via `ackReactionScope` tapbacks/read receipts or an inbound hook, so senders can tell queued messages were received.

## Best Possible Solution

Make the supported iMessage direction explicit: either add legacy `imsg` acknowledgement with channel-owned implementation, schema/docs, and regression tests, or document that read receipts and tapback acknowledgements require BlueBubbles and are out of scope for legacy iMessage.

## Reproduction Assessment

Yes for source-level reproduction: current main shows legacy `channels.imessage` dispatching inbound messages without any ack/read/tapback path, while BlueBubbles has that behavior. I did not do a live macOS Messages run, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as an implementation choice: adding native legacy `imsg` ack support would be a maintainable fix only if maintainers want to keep investing in that deprecated channel. The safer product path may be to document BlueBubbles as the supported acknowledgement path and scope legacy iMessage out.

## Risks / Open Questions

- Legacy `imsg` is documented as legacy and may be removed, so adding acknowledgement support there may conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs maintainer UX/security judgment: tapbacks add conversation noise, read-receipt activation can steal focus and need Accessibility/Automation permissions, and native macOS API support is not established in the current code.
- The existing `message_received` hook can support custom workarounds, but it is observation-only/fire-and-forget and does not provide native legacy iMessage read/tapback acknowledgement semantics.
