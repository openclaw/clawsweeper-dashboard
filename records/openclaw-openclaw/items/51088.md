## Summary

Keep open. Current main supports iMessage read receipts and tapback acknowledgements through the recommended BlueBubbles channel, and it also has a generic inbound `message_received` hook, but the still-documented legacy `channels.imessage`/`imsg` path requested in the report has no native acknowledgement config, capability, schema support, or read/tapback behavior. Because that path is deprecated but still documented, this needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

Request native iMessage acknowledgement so inbound messages visibly mark as read or receive a tapback, or an inbound hook that lets users run custom acknowledgement behavior without polling.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and focused tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure legacy `channels.imessage` with `messages.ackReactionScope` and trace inbound handling: the legacy config/schema has no ack/read field and the monitor dispatch path contains no acknowledgement call.

## Solution Assessment

Unclear. Native acknowledgement in deprecated `imsg` may be the right fix only if maintainers want to keep expanding that path; otherwise the narrower maintainable solution is to document BlueBubbles as the iMessage acknowledgement path and leave legacy `imsg` without this feature.
