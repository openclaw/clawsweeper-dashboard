## Summary

Keep open. Current main supports iMessage read receipts and tapback acknowledgements through the recommended BlueBubbles channel, and the issue discussion includes useful workaround context, but the still-documented legacy `channels.imessage`/`imsg` path requested here has no native acknowledgement setting, capability, schema support, or inbound read/tapback behavior. Because that path is deprecated but still documented, this needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

Add native iMessage inbound message acknowledgement, via read receipts or tapback acknowledgement, so senders can see that OpenClaw received messages before an agent turn completes.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and focused tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure legacy `channels.imessage` with `messages.ackReactionScope` and trace inbound handling: the legacy config/schema has no channel-owned ack/read field and the monitor dispatch path contains no acknowledgement call.

## Solution Assessment

Unclear. Native acknowledgement in deprecated `imsg` may be the right fix only if maintainers want to keep expanding that path; otherwise the narrower maintainable solution is to document BlueBubbles as the iMessage acknowledgement path and explicitly leave legacy `imsg` without this feature.

## Risks / Open Questions

- Legacy `imsg` is documented as legacy and may be removed, so adding native acknowledgement there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs product and privacy judgment: tapbacks add conversation noise, while activation-driven read receipts can steal focus and require Accessibility/Automation permissions.
- The current generic `message_received` hook supports custom automation but is fire-and-forget and does not give users a supported legacy iMessage read/tapback contract.
