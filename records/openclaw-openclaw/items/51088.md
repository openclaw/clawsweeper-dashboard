## Summary

Keep open. Current main has BlueBubbles read-receipt and tapback support plus a generic inbound hook, but the legacy `channels.imessage`/`imsg` path described by this issue still lacks native acknowledgement config, schema, capability, or dispatch behavior; because that path is documented as legacy but still supported, this needs maintainer product direction.

## What This Changes

Request native iMessage inbound acknowledgement through read receipts or tapback acknowledgements, ideally by extending existing acknowledgement scope handling or inbound hook support.

## Best Possible Solution

Make the supported iMessage direction explicit: either add legacy `channels.imessage` acknowledgement with schema/docs/tests and a safe macOS mechanism, or document that read receipts and tapback acknowledgements require `channels.bluebubbles`.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms the requested legacy iMessage acknowledgement surface is absent on current main while BlueBubbles has the related behavior.

## Solution Assessment

Unclear. Extending legacy `imsg` could solve the reported workflow, but the docs steer new iMessage deployments to BlueBubbles, so maintainers should first decide whether legacy acknowledgement support is in scope.

- Adding acknowledgement behavior to deprecated legacy `imsg` may conflict with the documented BlueBubbles migration path.

- The existing `message_received` hook can help custom workflows but is fire-and-forget and does not guarantee native channel acknowledgement semantics.
