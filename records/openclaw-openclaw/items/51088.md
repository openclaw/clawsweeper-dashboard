## Summary

Keep open. Current main solves this for the recommended BlueBubbles iMessage path and exposes a generic `message_received` hook, but the legacy `channels.imessage`/`imsg` path described by the report still lacks native read-receipt or tapback acknowledgement behavior while remaining documented as legacy-supported.

## What This Changes

Request native iMessage acknowledgement so senders can see receipt confirmation via read receipts or tapback acknowledgements when messages are queued or processed.

## Best Possible Solution

Make the supported iMessage path explicit: either add channel-owned legacy `imsg` acknowledgement with schema/docs/tests, or document that read receipts and tapback acknowledgements require `channels.bluebubbles` and are out of scope for legacy `channels.imessage`.

## Reproduction Assessment

Yes for the source-level gap: current main's legacy `imessage` monitor reaches `dispatchInboundMessage` without ack/read handling and its strict config schema has no ack/read keys. I did not live-run macOS Messages, so this is source-reproducible rather than a live reproduction.

## Solution Assessment

Unclear. The AppleScript workaround is not the best maintainable solution, but choosing between legacy `imsg` support and a BlueBubbles-only direction is a maintainer product decision rather than a narrow bug fix.

## Risks / Open Questions

- Legacy `imsg` is deprecated and may be removed, so adding new acknowledgement behavior there could conflict with the intended BlueBubbles migration path.
- The requested mechanisms need UX/security judgment: read-receipt activation can steal focus or require Automation/Accessibility permissions, while tapbacks can add conversation noise.
- The current legacy imessage RPC surface inspected here does not establish a native macOS or `imsg` API for read receipts/tapbacks, so implementation scope may depend on upstream capability.
