## Summary

Keep open: the confirmed-thread idea is useful, but the PR is still incorrect because the shared assertion can reject existing threaded media/upload sends that only have the requested thread fallback, and the contributor proof is still unit/mock-only.

## What This Changes

The PR adds a Slack `confirmedThreadTs` result field, asserts threaded outbound sends against Slack-confirmed thread evidence, and adds focused Slack send/outbound tests.

## Best Possible Solution

Keep the confirmed-thread contract for `chat.postMessage` and block/card paths, preserve threaded media/uploads unless upload confirmation is actually available, and require redacted live Slack proof before merge.

## Reproduction Assessment

Yes, source-level: at PR head, a threaded `sendMedia` or no-block media payload reaches the new shared assertion, while the upload-only send path returns no `confirmedThreadTs` when no follow-up chat message is posted. I did not run live Slack or local tests in this read-only review.

## Solution Assessment

No. Separating requested and Slack-confirmed thread evidence is the right direction, but asserting it in the common wrapper is too broad; the safer fix preserves upload-only threaded media behavior unless a real upload confirmation source exists.
