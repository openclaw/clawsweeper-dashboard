## Summary

Keep open: the PR is not merge-ready because the new guard trusts `result.threadTs`, but current `sendMessageSlack` can fill that field from the requested thread timestamp rather than Slack-confirmed response data; real Slack proof is also absent.

## What This Changes

The PR adds a Slack outbound adapter assertion that fails threaded payload sends when the send result lacks the requested thread timestamp, plus regression tests for missing, mismatched, and matching thread timestamps.

## Best Possible Solution

Expose confirmed Slack `message.thread_ts` evidence separately from requested or fallback thread state, assert that confirmed field at the outbound boundary, and keep the linked origin-tuple issue open until real Slack proof covers the threaded completion path.

## Reproduction Assessment

Yes for the PR defect at source level: current main and `v2026.6.10` show `sendMessageSlack` may synthesize returned `threadTs` from `opts.threadTs`, so a missing Slack confirmation can pass the new guard. The broader linked Slack origin-tuple issue still lacks a reduced live reproduction in this review.

## Solution Assessment

No. The adapter-level check is only safe after the sender result differentiates confirmed Slack `message.thread_ts` from requested fallback state; as written, it is too trusting.
