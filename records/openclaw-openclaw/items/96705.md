## Summary

Keep open: the patch is not merge-ready because it checks a `threadId` field that the real Slack send result does not expose, so successful threaded Slack replies can be rejected; external real behavior proof is also missing.

## What This Changes

The PR adds a Slack outbound adapter check that throws when a threaded send result does not confirm the requested thread timestamp, plus regression tests for confirmed and unconfirmed payload sends.

## Best Possible Solution

Move confirmation into the Slack send result contract using Slack `message.thread_ts` or root `ts` evidence, then keep the linked broader origin-tuple issue open until live Slack proof covers delegated threaded completion.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the real Slack send result returns `threadTs`, while the PR checks `threadId` and then compares the child reply `messageId` to the parent thread timestamp. The broader linked issue still lacks a live reduced Slack reproduction.

## Solution Assessment

No. The best fix is not to assert against a mocked `threadId`; it should make `sendMessageSlack` expose confirmed thread evidence, then assert that contract at the outbound boundary.
