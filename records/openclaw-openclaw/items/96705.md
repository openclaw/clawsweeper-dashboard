## Summary

Keep open: the bugfix direction is useful, but the patch still trusts `result.threadTs` even though `sendMessageSlack` can synthesize that value from the requested thread timestamp, and the PR still lacks real Slack behavior proof.

## What This Changes

Adds a Slack outbound adapter assertion that fails threaded payload sends unless the send result has a matching `threadTs`, plus regression tests for missing, mismatched, and matching thread results.

## Best Possible Solution

Expose confirmed Slack `message.thread_ts` evidence separately from requested or fallback thread state, assert that confirmed field at the outbound boundary, and keep the linked origin-tuple issue open until real Slack proof covers the threaded completion path.

## Reproduction Assessment

## Solution Assessment

No. The adapter-level check is only safe after the sender result differentiates confirmed Slack `message.thread_ts` from requested fallback state; as written, it is too trusting.
