## Summary

Keep this PR open: the worker-thread direction has strong before/after proof for the event-loop starvation bug, but the current head can still publish false prepared auth for providers whose synthetic auth exists only in the active runtime registry, and it also carries an unrelated minified diff-viewer runtime asset change that needs removal or build provenance.

## Best Possible Solution

Land the worker-based prewarm only after the worker preserves the same runtime synthetic-auth facts as the main process and the unrelated generated asset change is removed or source-proven.
