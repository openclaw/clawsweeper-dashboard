## Summary

Keep open: the worker-thread direction addresses the gateway starvation class, but the current head can still publish false unauthenticated answers for runtime-only synthetic-auth providers because the worker does not receive the main process synthetic-auth registry state.

## What This Changes

This PR moves provider-auth prewarming into a worker thread, serializes warm snapshots, passes sanitized runtime auth-store presence, rewires gateway startup/reload/logout rewarm paths, and adds packaging/release coverage for the worker entry.

## Best Possible Solution

Preserve the exact main-process provider-auth facts across the worker boundary, including active synthetic-auth refs, then re-run the focused auth/gateway/release tests and responsiveness proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows a concrete path where a provider plugin exposing resolveSyntheticAuth only through the active runtime registry is visible on current main but not reconstructed in the worker. I did not run a live plugin reproduction in this read-only review.

## Solution Assessment

No. Moving the expensive warm path off the main event loop is a good fix shape, but the current worker boundary must preserve runtime synthetic-auth facts before it is the best maintainable solution.

- **[P1] Carry runtime synthetic-auth refs into the worker:** `src/agents/model-provider-auth.ts:577-580`
