## Summary

Keep open. Current main still full-reindexes changed session transcripts, and this PR is the right direction for the linked performance issue, but the latest patch still has concrete merge blockers in offset accounting, Plugin SDK API baseline generation, and changelog placement against current main.

## What This Changes

The PR adds hash-based incremental session indexing for memory-core, new chunk offset storage metadata, SDK exports for offset-aware chunking, session-sync tests/benchmarks, and a changelog entry.

## Best Possible Solution

Land a corrected incremental session-sync implementation in memory-core that reuses unchanged session embeddings, preserves accurate offsets for long-line and overlap cases, keeps the generated Plugin SDK API baseline aligned, and records the user-facing performance change in the current changelog section.

## Reproduction Assessment

Yes. Current main can be verified statically: changed sessions call indexFile, and writeChunks clears existing rows before reinserting. The PR blockers are also statically reproducible from the latest head: the split-line offset cursor is inconsistent, the SDK baseline is absent from the diff, and the changelog is based on the old Unreleased section.

## Solution Assessment

No. Hash-based incremental session sync is the right maintainable approach for the reported performance problem, but this exact patch should not merge until the offset accounting, generated API baseline, and changelog placement are corrected.

## Review Findings

Overall correctness: patch is incorrect

  - body: For long source lines split across multiple chunks, the first flush advances `charOffset` as if a newline existed between adjacent split segments while later segments do not include that newline.
