## Summary

Keep open. This remains useful source work for the open session-memory performance bug and supersedes the earlier closed, unmerged PR #59577, but the latest patch is not merge-ready because offset metadata for split long lines is still incorrect and generated SDK/changelog surfaces need cleanup.

## What This Changes

The PR adds hash-based incremental indexing for growing session memory files, SQLite chunk offset metadata, SDK exports, memory-core and host-sdk tests/benchmarks, and an unreleased changelog entry.

## Best Possible Solution

Land a corrected incremental session-sync implementation in memory-core that reuses unchanged session embeddings, preserves accurate offsets for long-line and overlap cases, keeps the generated Plugin SDK API baseline aligned, and places the changelog bullet under the existing active Changes section.

## Reproduction Assessment

Yes. Current main can be verified statically: changed session files call indexFile, and writeChunks clears existing rows before reinserting. The PR blockers are also statically reproducible from the latest diff: split-line offset math is inconsistent, the SDK baseline file is unchanged, and the changelog inserts a duplicate section heading.

## Solution Assessment

No. Hash-based incremental session sync is the right maintainable direction for the linked performance issue, but this exact patch should not merge until the offset accounting, SDK baseline, and changelog placement are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep split-line offsets contiguous:** `packages/memory-host-sdk/src/host/internal.ts:505-515`
  - body: For long source lines split across multiple chunks, the first flush advances `charOffset` as if a newline existed between adjacent split segments, and later chunks subtract two from `currentRawChars` even when they contain only a non-first segment. That makes the next chunk start one character late and stores the wrong offset range for long session messages.

  - body: This inserts a second `### Changes` heading inside the active version instead of adding the Memory/sessions bullet under the existing one. The changelog should keep one active Changes section, so move the bullet up and drop the duplicate heading.
