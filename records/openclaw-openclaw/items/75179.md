## Summary

Keep open. Current main still has the full session reindex behavior, and this PR is the active replacement for closed #59577, but the latest head remains merge-blocked by offset accounting and generated SDK/changelog issues. The direction is sound, but the patch should not land unchanged.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, persists chunk offset metadata in SQLite, exports offset-aware chunking through the memory host SDK, and adds memory-core tests/benchmarks plus a changelog entry.

## Best Possible Solution

Land a corrected memory-core incremental session sync that reuses unchanged session embeddings, keeps original-content offsets contiguous for long-line and overlap cases, updates the generated Plugin SDK API baseline, and places the user-facing changelog entry in the active Changes section.

## Reproduction Assessment

Yes. Current main can be verified statically: changed session files call indexFile, and writeChunks clears existing rows before reinserting. The PR blockers are also statically reproducible from latest head: split-line offset math is inconsistent, the SDK baseline is absent from the PR file list, and the changelog is on stale active-version structure.

## Solution Assessment

No. Hash-based incremental session sync is the right maintainable direction, but this exact patch should be repaired before merge; the safer path is to keep the behavior scoped to memory-core while fixing offset math and generated/public contract state.
