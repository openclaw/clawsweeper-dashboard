## Summary

Keep this PR open, but do not merge it as-is. The incremental session-sync direction still addresses the real performance problem tracked by #40919 and the stale predecessor #59577, but the current patch preserves hash-matched rows on forced session reindexes and records incorrect offsets for multi-line chunks; the existing ClawSweeper discussion already identified the same concrete blockers, and current source/diff inspection confirms them.

## What This Changes

The PR adds hash-based incremental indexing for memory-core session files, new start/end offset chunk metadata and schema columns, SDK exports, and focused memory-core/host SDK tests and benchmarks.

## Best Possible Solution

Land the incremental session-sync optimization only after forced syncs still clear and rebuild session rows, offset accounting is correct for multi-line and split-line chunks, and focused tests cover both append-growth efficiency and repair-mode behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction for the blocking regression is a forced memory sync on an unchanged session after stale or corrupted indexed rows exist; current main rebuilds through indexFile, while the PR would route that session through hash-preserving incremental sync.

## Solution Assessment

No. Incremental sync is the right direction for ordinary session growth, but this implementation needs a forced mode or a retained indexFile path for full reindexes, plus corrected offset cursor accounting before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve forced session rebuilds:** `extensions/memory-core/src/memory/manager-sync-ops.ts:886`
  - body: This replacement also runs for full reindexes. The new incremental helper preserves same-hash chunks and embeddings, so `openclaw memory index --force` can leave stale or corrupted session rows in place instead of performing the documented repair rebuild. Keep the current `indexFile` path when `needsFullReindex` is true, or add an explicit forced mode that clears first.
  - confidence: 0.93
- **[P3] Advance offsets past separator newlines:** `packages/memory-host-sdk/src/host/internal.ts:511`
  - body: After flushing a chunk that spans multiple source lines, `currentRawChars - 1` advances the next start offset to the separator newline after the last emitted line. The new persisted offset range then points at the newline instead of the next content character, so consumers of this metadata get off-by-one ranges.
  - confidence: 0.82
