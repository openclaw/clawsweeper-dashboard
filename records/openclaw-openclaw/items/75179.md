## Summary

Keep open. Current main still rebuilds and rewrites every chunk for a changed session file, so the underlying performance bug remains valid, and this PR is the active replacement for the superseded #59577 work. The patch is not merge-ready because it expands a public SDK surface without regenerating the API baseline, records incorrect offsets for split long lines, and lacks the required changelog entry.

## What This Changes

This PR adds hash-based incremental indexing for session memory files, including offset-aware chunking and stored offsets, memory index schema columns, public memory-host SDK exports, and memory-core session benchmark/incremental tests.

## Best Possible Solution

Keep the incremental session-sync design in memory-core, but land it only after the offset-aware chunking contract is correct, the public SDK baseline is regenerated, the changelog is updated, and targeted memory-core plus Plugin SDK checks pass.

## Reproduction Assessment

Yes. The current-main performance issue is reproduced by growing a session file and observing that selected changed sessions still call indexFile, whose write path clears all chunks before reinserting. The PR offset defect is reproducible by passing one long line through chunkMarkdownWithOffset and checking that later split segments get offsets as if extra newlines existed.

## Solution Assessment

No. The proposed direction is the right maintainable shape for session growth, but this exact patch should not merge until the offset metadata and generated public SDK contract are fixed; the changelog should also be added before merge.
