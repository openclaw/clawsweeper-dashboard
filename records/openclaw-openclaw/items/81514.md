## Summary

Keep open. Current main still has the order-sensitive isolated cron status path, and the linked draft PR uses closing syntax for this issue but is not merged, so the issue should remain open until that PR is revised and landed or replaced.

## What This Changes

The issue asks for isolated cron agent-turn run status to be order-independent when a recoverable tool error is handled and fallback output is produced.

## Best Possible Solution

Define the recovered-tool-error contract for isolated cron runs, then land a narrow classifier and regression-test change that clears only explicitly recovered failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: error then success is nonfatal, while the same successful text before a later error remains fatal and finalizeCronRun maps that to the stored run status.

## Solution Assessment

Unclear. The broad any-success approach is too weak for claimed-success failures, so the best fix should require an explicit recovery or acknowledgement signal and preserve fatal boundaries for unresolved tool/provider errors.
