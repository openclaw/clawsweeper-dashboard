## Summary

Keep open. Current main still makes isolated cron status depend on payload ordering: an error followed by successful text is nonfatal, while successful text followed by a later non-message error remains fatal and is persisted as `status: error`.

## What This Changes

The issue asks for deterministic isolated cron run status when an agent recovers from a tool failure and emits a fallback summary.

## Best Possible Solution

Define the recovered-tool-error contract for isolated cron runs, then update `resolveCronPayloadOutcome` and regression coverage so status follows that recovery signal rather than incidental payload order.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass `resolveCronPayloadOutcome` an error payload followed by successful text to get nonfatal, then reverse the order with a later non-message error to get fatal; `finalizeCronRun` persists that boolean as the run status.

## Solution Assessment

Unclear. The smallest suggested fix would change existing tests that keep real trailing errors fatal, so maintainers should choose the policy boundary before implementation.
