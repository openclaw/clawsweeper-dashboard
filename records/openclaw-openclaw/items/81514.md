## Summary

Keep open. Current main and v2026.6.6 still have a source-reproducible recovered-output path where terminal assistant text can lose to error payloads, and the live linked PRs are still open rather than merged.

## What This Changes

The issue reports isolated cron run status flipping between `ok` and `error` for recovered tool failures because status classification depends on payload ordering instead of a terminal recovery signal.

## Best Possible Solution

Review and land the viable resolver-level fix, likely https://github.com/openclaw/openclaw/pull/84156 if it supersedes the draft https://github.com/openclaw/openclaw/pull/81585, or close those PRs after choosing a different cron status contract.

## Reproduction Assessment

Yes, at source level. Current main has a focused helper test that combines all-error payloads with terminal assistant-visible recovery text and still expects the last error to win; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a maintainer policy question, but the fix belongs in `resolveCronPayloadOutcome` because both interim retry handling and final cron persistence consume that classifier. A summary-only workaround would leave persisted status and failure-alert behavior unreliable.
