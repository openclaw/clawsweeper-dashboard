## Summary

Keep open. Current main and v2026.6.8 cover some plain recovered tool-warning cases, but the broader isolated-cron status contract remains unresolved: all-error payloads plus terminal assistant text still resolve as fatal, later comments include the opposite false-OK case, and the active classifier PR is open but conflicting.

## What This Changes

The issue reports isolated cron run status flipping between `ok` and `error` when an agent recovers from tool errors and emits terminal output.

## Best Possible Solution

Keep this issue as the canonical cron status-contract thread until maintainers land a refreshed classifier fix or explicitly split the false-OK sentinel and diagnostics cases into narrower follow-ups.

## Reproduction Assessment

Yes, source-level. Current main still has a focused helper test where all payloads are errors and preferred terminal assistant text exists, yet the last error payload wins; I did not run a live daemon cron reproduction in this read-only review.

## Solution Assessment

Unclear. The resolver is the right layer for the false-error path because both interim handling and final cron persistence consume it, but maintainers still need to choose whether terminal assistant text, explicit sentinels, or typed failure signals are the canonical status contract.
