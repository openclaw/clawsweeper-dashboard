## Summary

Keep open. Current `main` still makes isolated cron status depend on whether a non-error payload appears after the last error payload, so the reported recovered-tool-error status flip remains source-reproducible; the related final-assistant issue and open PR do not yet settle this exact status policy.

## What This Changes

Requests deterministic `openclaw cron runs` status for isolated `agentTurn` jobs when the agent recovers from a tool error and completes via a fallback path.

## Best Possible Solution

Define the recovered-tool-error status contract for isolated cron runs, then update `resolveCronPayloadOutcome` and focused regression tests so recovered failures are classified by an explicit recovery signal rather than incidental payload ordering while true run-level failures stay fatal.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: `resolveCronPayloadOutcome` reports nonfatal for error-then-success payloads and fatal for success-then-error payloads, and `finalizeCronRun` persists that boolean as `ok` or `error`.

## Solution Assessment

Unclear. The smallest suggested change, treating any successful payload as recovery, would overturn tests that keep real trailing errors fatal, so maintainers should choose the recovered-versus-unresolved error policy before implementation.
