## Summary

Keep open. Current main has narrowed some recovered-tool-warning cases, but the broader isolated cron status contract remains unsettled and recent issue discussion reports the same false-error pattern in production; the issue also has an open closing PR, so it should stay open until that implementation path is reviewed, landed, or closed.

## What This Changes

The issue asks for isolated `agentTurn` cron run status to reflect the terminal recovered outcome rather than incidental intermediate tool-error payload ordering.

## Best Possible Solution

Define the isolated cron terminal-outcome contract, then land one focused fix that covers recovered success, terminal failure sentinels or typed failure signals, failure-alert accounting, and regression tests across the helper and finalization paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path through `resolveCronPayloadOutcome`: construct a recovered run where final assistant text exists but the remaining error payload does not satisfy the current narrow recovery gates, and `run.ts` will persist that fatal boolean as cron `status: error`.

## Solution Assessment

Unclear. The direction is valid, but the best fix needs maintainer policy because the issue comments now include both false-error and false-OK cases, so a general terminal-output rule may be too broad without a typed outcome or sentinel contract.
