## Summary

Keep open. Current main and v2026.6.10 still pin the generic all-error-payload plus terminal assistant-text path as fatal, while cron run status and failure counters are still derived from that classifier; open related PRs are candidate fixes but none has landed as the canonical solution.

## What This Changes

The issue reports isolated cron `agentTurn` runs persisting unreliable `ok` or `error` status when the agent recovers from tool errors and emits a terminal outcome.

## Best Possible Solution

Keep this issue open as the canonical cron status-contract thread until maintainers land or supersede the classifier fix and decide how terminal assistant success/failure sentinels should affect persisted status.

## Reproduction Assessment

Yes, source-level. Current main has focused helper coverage showing all-error payloads plus preferred final assistant text still resolve to the last error payload, and cron status is derived from that classifier; I did not run a live daemon cron reproduction in this read-only review.

## Solution Assessment

Unclear as the final product contract. `resolveCronPayloadOutcome` is the right layer for the false-error path, but maintainers still need to choose the canonical status contract for terminal assistant success/failure text versus typed fatal signals.
