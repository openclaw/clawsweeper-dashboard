## Summary

Keep open. Current main and the latest release still use an order-sensitive recovered-tool-error classifier for isolated cron runs, and the adjacent open self-correction PR does not repair this status policy; the remaining fix needs maintainer judgment because existing tests intentionally keep some trailing errors fatal.

## What This Changes

Request that isolated cron `agentTurn` runs report stable `ok`/`error` status when an agent handles a recoverable tool failure and completes through a fallback path.

## Best Possible Solution

Define the recovered-tool-error contract for isolated cron runs, then update `resolveCronPayloadOutcome` and focused regression coverage so status follows an explicit recovery signal rather than incidental payload ordering while preserving true run-level failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: pass `resolveCronPayloadOutcome` an error payload followed by successful text to get nonfatal, then reverse the order with the same success text before a later non-message error to get fatal; `finalizeCronRun` maps that directly to `ok` or `error`.

## Solution Assessment

Unclear. The helper is the right boundary, but the suggested broad rule that any successful payload makes errors nonfatal would overturn existing tests that keep real trailing errors fatal, so maintainers should choose the recovered-error policy first.
