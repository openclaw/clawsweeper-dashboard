## Summary

Keep open. Current main and the latest release still classify isolated cron payload errors by whether successful text appears after the last error, and the linked closing PR is open and draft rather than merged.

## What This Changes

The issue asks for isolated cron run status to classify recovered tool errors deterministically instead of depending on payload ordering within an agent turn.

## Best Possible Solution

Land a narrow fix through the linked PR or a replacement that uses an explicit recovery signal for isolated cron runs while preserving fatal handling for unresolved trailing tool, canvas, provider, and run-level errors.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `resolveCronPayloadOutcome` returns nonfatal for `[error, success text]` but fatal for `[success text, later non-message error]`, and `run.ts` maps that boolean directly to the stored cron run status.

## Solution Assessment

Unclear. The linked PR is the right place to finish the issue, but the current proposed boundary is not yet the best fix because it can clear trailing exec failures after claimed success without a concrete recovery signal.
