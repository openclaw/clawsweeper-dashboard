## Summary

Keep open: current main still blocks this path by rejecting `sessions_spawn` attachments for `runtime: "acp"`, and the ACP turn dispatch path does not pass parsed images into `acpManager.runTurn`. The issue is valid, but it is a new ACP image-routing capability rather than a strict regression against the documented current limitation.

## What This Changes

The issue requests ACP `sessions_spawn` image content to reach Codex native vision the same way direct `acpx codex exec` does.

## Best Possible Solution

Plumb image attachments through the existing ACP turn attachment contract for ACP `sessions_spawn`, keep size/redaction/non-image safeguards, update the docs, and prove the path with focused tests plus a real Codex/acpx image smoke.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `sessions_spawn` with `runtime: "acp"` and any attachment, and current main returns the unsupported-attachments error before ACP dispatch.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse `AcpTurnAttachment` plumbing rather than adding a new image tool or config option, while preserving existing attachment limits and rejection behavior for unsupported payloads.
