## Summary

Keep open: current source supports a plausible ACPX/Codex capability gap rather than an obsolete or external report, because ACP-spawned Codex sessions bypass the normal OpenClaw image-tool setup and run through an isolated Codex wrapper.

## What This Changes

Make `sessions_spawn` with `runtime: "acp"` and `agentId: "codex"` expose the same image-analysis capability available through direct `acpx codex exec`.

## Best Possible Solution

Fix this in the ACPX/Codex bootstrap with a narrow image-capability bridge or corrected Codex ACP configuration, add regression coverage, and preserve isolated `CODEX_HOME` without broad default tool exposure.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a medium-confidence path: ACP spawn uses `spawnAcpDirect` and `acpManager.runTurn`, while normal image tooling is built only on the standard OpenClaw tool path and ACPX Codex uses an isolated `CODEX_HOME`.

## Solution Assessment

No proposed patch is present. The maintainable fix should be ACPX/Codex-owned parity, not making core agents depend on bundled plugin ids or copying the full host Codex config wholesale.
