## Summary

Keep this PR open, but it is not merge-ready: the patch changes the provider/model override trust boundary by deriving LLM authorization from lossless-claw `summaryModel` rather than preserving the documented explicit `llm` policy.

## What This Changes

The branch modifies `src/plugins/config-normalization-shared.ts` so lossless-claw entries with `config.summaryModel` synthesize an `llm` model-override policy during plugin config normalization.

## Best Possible Solution

Preserve `plugins.entries.<id>.llm.allowModelOverride` and `llm.allowedModels` as the only runtime LLM authorization source, then fix the context-engine/runtime config lifecycle for the linked LCM startup denial with focused regression and real gateway proof.

## Reproduction Assessment

No live current-main LCM run was executed here. The patch defect is source-reproducible: the PR writes trusted `llm` policy from `summaryModel`, while current runtime and docs require explicit `llm.allowModelOverride` authorization.

## Solution Assessment

No. This is not the best fix because it turns plugin config into authorization; the safer fix is to preserve explicit `llm` trust and repair the context-engine/runtime config lifecycle.
