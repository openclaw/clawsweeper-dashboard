## What This Changes

The branch guards channel startup-maintenance metadata access and warning logging against throwing plugin getters, with colocated Vitest coverage for throwing `lifecycle` and `id` getters.

## Best Possible Solution

Rebase the draft onto current main, keep the per-plugin metadata guard in the startup-maintenance runner, and land after focused regression or CI proof covers the new test.

## Reproduction Assessment

Yes, from source inspection. Current main and v2026.6.6 both read `plugin.lifecycle` before the guarded hook call, so a throwing getter can abort the loop; I did not run a local repro because this review was read-only.

## Solution Assessment

Yes. Guarding metadata access in `runChannelPluginStartupMaintenance` is the narrowest maintainable fix and matches the existing best-effort hook failure policy; loader-level policy or a config knob would be broader than needed.
