## Summary

Keep this PR open as useful source work, but it is not merge-ready. Current main still lacks the proposed `*Share` config surface, and the PR diff has unresolved runtime bugs around fallback context windows, preflight floor resolution, plus a missing required changelog entry for a user-facing config feature.

## What This Changes

The PR adds four ratio-based compaction budget config sibling fields, updates schema/help/generated metadata, threads model context-window values into Pi compaction and memory-flush paths, and adds focused tests.

## Best Possible Solution

Land this only after rebasing onto current main, keeping the additive share fields if maintainers approve the API, resolving shares only from authoritative context-window metadata, applying the same reserve-floor resolver in preflight paths with and without memory-flush plans, and updating generated config metadata, docs/help, changelog, and focused tests together.

## Reproduction Assessment

Yes. Current-main absence is reproducible by source/schema search, and the PR bugs are reproducible by static control-flow inspection of the changed paths: floor shares are skipped when no flush plan exists, and fallback/default context windows are treated as real share bases.

## Solution Assessment

No, not yet. Ratio sibling fields may be a reasonable solution, but the current patch is not the narrowest maintainable merge shape until it preserves absolute fallbacks for unknown windows and consistently resolves the new floor-share setting across all compaction gates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve floor shares when no flush plan exists:** `src/auto-reply/reply/agent-runner-memory.ts:390-393`
  - body: When no memory-flush plan is configured, `memoryFlushPlan` is null and the preflight threshold still falls back to the raw absolute `reserveTokensFloor` or `20_000`. A config that only sets `reserveTokensFloorShare` is ignored on this path, so large-window sessions compact later than the configured share intends.
  - confidence: 0.91
- **[P2] Preserve absolutes for fallback context windows:** `src/auto-reply/reply/memory-flush.ts:40-49`
  - body: `applyMemoryFlushSharesToPlan` treats every numeric `contextWindowTokens` as authoritative, but the caller can pass the default 200k fallback when model metadata is unknown. That makes share thresholds override the plan's absolute values for custom or unknown models, contrary to the PR's documented fallback behavior.
  - confidence: 0.86
- **[P2] Use authoritative context budgets for shares:** `src/agents/pi-embedded-runner/run/attempt.ts:916-917`
  - body: Passing `params.model.contextWindow` into share resolution can use a synthesized default window from model fallback construction rather than a real model limit. Unknown or custom small-window models can therefore compute share budgets from 200k instead of preserving absolute/default budgets; the direct compaction path needs the same fix.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/zod-schema.agent-defaults.ts:133`
  - body: This adds user-facing `feat(config)` keys, but the PR file list does not include `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-visible config features before merge.
  - confidence: 0.9
