## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Build deferred candidates with the final account id:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: When `deferApplyUntilValidated` is enabled, the PR resolves `draftAccountId` only after earlier setup steps have already built `next` using the original account id. A plugin whose `resolveAccountId` derives the account from completed input can then validate a candidate under one id while the config was applied under another.
  - confidence: 0.88
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: This returns the original `accountId` even after deferred validation re-resolves `draftAccountId`, so callers such as account recording, binding prompts, and post-write hooks can operate on the wrong account for derived ids.
  - confidence: 0.9
- **[P2] Validate deferred candidates without sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `validateSetupCandidate` reruns `validateInput` against `draftInput`, but that object only includes fields touched in the wizard. Credential-preserving updates can therefore fail validators that require a credential in `input` even though the assembled `candidateCfg` remains valid.
  - confidence: 0.84
- **[P1] Keep service-specific add flags in plugin metadata:** `src/cli/channels-cli.ts:171-172`
  - body: Current main already supports plugin-owned `cliAddOptions` and generic setup input capture, so hardcoding `--community-id` and `--community-access-token` into the shared CLI creates a competing source of truth for VK options and exposes those flags independent of the plugin.
  - confidence: 0.86
