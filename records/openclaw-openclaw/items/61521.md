## Summary

Keep this PR open, but do not merge this revision as-is. Current main still lacks the requested complete/async candidate-validation seam, and the PR is useful source work, but the deferred wizard path can validate and return a candidate assembled under the wrong account and can reject valid update flows by rerunning field validation on sparse draft input.

## What This Changes

This PR adds complete/async channel setup validation hooks, wires the setup wizard and `openclaw channels add` to validate candidate configs before writing, adds VK-oriented setup input/CLI fields, and adds a focused wizard validation test.

## Best Possible Solution

Keep the useful seam, but revise it so deferred setup builds a single candidate config from the original config plus the complete collected draft input, validates that candidate with candidate-aware hooks, consistently uses and returns the resolved account id, adds coverage for derived account ids and kept existing credentials, and includes the required changelog entry before maintainer review.

## Reproduction Assessment

Yes. A focused wizard test can enable `deferApplyUntilValidated`, derive `resolveAccountId` from `communityId`, then enter a separate credential prompt; the current PR code applies prompt patches with sparse inputs and validates/returns the wrong candidate path. Another focused update test can keep an existing credential and show `validateInput` rejecting the sparse draft even though `candidateCfg` is valid.

## Solution Assessment

No. The proposed direction is plausible, but this implementation validates the already-mutated per-step config instead of a freshly assembled candidate and reuses field-level validation in a context where it lacks complete input. The safer fix is candidate-first assembly plus complete/async candidate validation only at the deferred boundary.

  - body: When deferred mode is enabled, `next` has already been built by applying each prompt as a one-field input. If `resolveAccountId` depends on assembled fields such as `communityId`, later credential steps can be applied under the original account, then this block validates that stale or mixed `next` as the final candidate and returns the original account id. Build the candidate once from `cfg` plus the full `draftInput`, validate that config, and return/use the resolved account id.
  - confidence: 0.91

Keep field validation on per-step input or make the final validation candidate-aware.
