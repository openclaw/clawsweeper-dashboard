## Summary

Keep open for maintainer review: the PR is a plausible new channel setup contract, but the branch still has source-proven correctness bugs, conflicts with current main's plugin-owned CLI option path, and lacks external real behavior proof.

## What This Changes

The PR adds complete and async channel setup validation hooks, defers setup writes until candidate validation, adds VK-oriented setup inputs/CLI flags, and adds a focused setup-wizard test.

## Best Possible Solution

If maintainers want this SDK contract, rework it on current main so candidate assembly is generic, account ids are propagated consistently, sparse draft validation is avoided, channel-specific setup flags stay in plugin metadata, and the new contract is documented.

## Reproduction Assessment

Yes, source-reproducible. A deferred wizard can derive accountId from later input while earlier prompt mutations and the returned account id still use the original value, and skipped/kept credentials can leave draftInput sparse before final validation.

## Solution Assessment

No. The direction may be right, but this implementation should not merge until candidate assembly, account-id propagation, sparse validation, and plugin-owned CLI option issues are fixed against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build deferred candidates from the complete draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: In deferred mode, `next` has already been mutated prompt by prompt using the original account id. If a later field changes `resolveAccountId`, validators and the eventual write can see a candidate assembled under a different account than the one being validated; build one candidate from the original config plus the complete draft before validating.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: The deferred path computes `draftAccountId` for validation but still returns the original `accountId`. Downstream account recording, bindings, and config-changed hooks can then refer to a different account than the validated candidate.
  - confidence: 0.9
- **[P2] Avoid final field validation on sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` contains only fields changed during this wizard run, while kept credentials or skipped prompts may exist only in `candidateCfg`. Re-running field-level `validateInput` here can reject a valid deferred update before the candidate-aware validators run.
  - confidence: 0.88
- **[P2] Keep VK setup flags in plugin metadata:** `src/cli/channels-cli.ts:171-172`
  - body: Current main already registers channel-specific `channels add` flags from plugin `cliAddOptions` and forwards non-control option values generically. Hardcoding VK flags in the shared CLI exposes them without the VK plugin and bypasses the metadata-owned setup surface.
  - confidence: 0.84
- **[P3] Preserve SecretInput typing for community tokens:** `src/commands/channels/add.ts:25-26`
  - body: `ChannelSetupInput.communityAccessToken` is declared as `SecretInput`, but `ChannelsAddOptions` redeclares it as `string`, narrowing programmatic callers away from `SecretRef` values. Remove the duplicate field or type it as `SecretInput` so command options match the setup contract.
  - confidence: 0.78
