## Summary

Keep this PR open, but it is not merge-ready: the proposed core setup API is still useful to discuss, yet the branch is stale against current main, duplicates plugin-owned CLI option handling, and has source-level setup contract bugs that can mis-record derived channel accounts.

## What This Changes

The PR adds complete and async channel setup validation hooks, deferred setup-wizard and `channels add` candidate validation, VK-oriented setup inputs and CLI flags, and a focused setup-wizard test.

## Best Possible Solution

Rebase on current main and design a plugin-owned pre-write candidate validation hook that assembles once, validates once, propagates the final account id, keeps VK flags in plugin metadata, and includes real CLI or wizard proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head shows stale deferred account-id propagation, sparse final validation input, and duplicated CLI option ownership against current main. The requested change itself is a feature/API proposal, not a current-main bug reproduction case.

## Solution Assessment

No: the current patch is not the best form because it mixes a core setup API change with hardcoded VK CLI fields and does not validate one final assembled candidate consistently. A safer solution would keep channel-specific flags in plugin metadata and make the shared hook operate on the final candidate config and resolved account id only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep VK setup flags plugin-owned:** `src/cli/channels-cli.ts:171-172`
  - body: Current main already registers `channels add` options from channel package metadata via `cliAddOptions`. Adding `--community-id` and `--community-access-token` directly to core makes VK flags visible even when the VK plugin is absent and creates a competing source of truth for external channel setup options.
  - confidence: 0.93
- **[P1] Build deferred candidates with the final account id:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: The wizard mutates `next` during individual prompt steps, then resolves `draftAccountId` from the aggregated `draftInput` only afterward. A plugin that derives account id from completed input can validate `next` under `draftAccountId` even though parts of that candidate were applied under earlier or partial account ids, so the candidate being validated is not necessarily the final runnable config.
  - confidence: 0.9
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: When deferred validation is enabled, this returns the original `accountId` even after resolving `draftAccountId`. The setup flow records the returned account id for bindings and post-write hooks, so derived-account plugins can write config under one account while the rest of setup records another.
  - confidence: 0.95
- **[P2] Avoid validating sparse draft input as complete setup input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` only contains prompt values that changed in this run, but `validateInput` is the existing field-level setup validator and can require credentials in `input`. Re-running it here can reject valid updates that preserve an existing credential even though `candidateCfg` is complete.
  - confidence: 0.88
