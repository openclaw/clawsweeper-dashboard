## Summary

Keep open: current main does not implement the requested candidate-config setup validation seam, but this PR is not mergeable as-is because the deferred wizard path has account/candidate consistency bugs, the branch predates current main’s plugin-owned CLI option surface, and the external real-behavior proof gate is still unsatisfied. This is not a ClawHub close for the core API question, although the VK channel itself has already been routed to community plugin/ClawHub work.

## What This Changes

The PR adds complete and async channel setup validation hooks, defers wizard and non-interactive channel config writes until candidate validation, extends setup input with VK-oriented fields, and adds a focused setup wizard test.

## Best Possible Solution

If maintainers want this SDK contract, rebase it onto current main and implement candidate validation generically: build one complete candidate from the full draft before write, propagate the resolved account id consistently, avoid sparse final field validation, keep plugin-specific flags in plugin metadata, and add regression plus real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. A focused wizard test can derive `accountId` from a later setup field while earlier prompt mutations were applied to the original account, and a kept/skipped credential path can hit `validateInput` with sparse `draftInput`; I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but this implementation should not merge until the candidate assembly, deferred account id, sparse validation, and metadata-owned CLI option issues are fixed; the VK channel implementation itself should remain a community plugin/ClawHub concern.

  - body: In deferred mode, `next` has already been mutated prompt-by-prompt before validation.   - confidence: 0.9

  - confidence: 0.88
