## Summary

Keep open. Current main does not implement the requested complete/async channel setup validation seam, and this PR still contains useful feature work, but the branch is blocked by source-proven correctness issues, current main's plugin-owned CLI option contract, missing inspectable real behavior proof, and a dirty/conflicting head.

## What This Changes

The branch adds complete/async channel setup validation hooks, defers setup wizard and `channels add` config writes until candidate validation, adds VK-oriented setup input/CLI fields, and adds a focused wizard test.

## Best Possible Solution

Keep the deferred validation seam idea, but rework it on current main so candidate config is assembled once from the original config plus complete draft input, candidate-aware validation runs before writes, resolved account ids are used consistently, and VK options live in plugin metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a focused wizard test can enable deferred validation, derive `resolveAccountId` from `communityId`, and show the mixed candidate/stale returned account id; another update test can keep an existing credential and hit the sparse `validateInput` rejection.

## Solution Assessment

No. The feature direction is plausible, but this branch validates the wrong deferred candidate, re-runs field validation on incomplete draft input, returns the wrong account id, and predates current main's plugin-owned setup option flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build the deferred candidate from the full draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: When deferred mode is enabled, `next` has already been mutated one prompt at a time before validation. If `resolveAccountId` depends on a later field such as `communityId`, patches can land under the initial account while the final validation uses a different account, so rebuild the candidate once from the original config and complete draft input.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: This path computes `draftAccountId` for deferred validation but still returns the original `accountId`. Downstream account recording, bindings, and config-changed hooks can then refer to a different account than the validated candidate.
  - confidence: 0.9
- **[P2] Skip final field validation on sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` contains only values changed during this wizard run, while kept credentials or skipped prompts can exist only in `candidateCfg`. Re-running `validateInput` here can reject a valid deferred update before the complete/async candidate validators run.
  - confidence: 0.88
- **[P2] Move VK setup flags to plugin metadata:** `src/cli/channels-cli.ts:168-170`
  - body: Current main registers channel-specific setup flags from plugin `cliAddOptions`. Hardcoding `--community-id` and `--community-access-token` in the shared CLI would expose VK-specific flags even without the VK plugin and conflicts with the current plugin-owned option contract.
  - confidence: 0.84
