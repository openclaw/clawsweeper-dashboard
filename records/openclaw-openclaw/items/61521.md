## Summary

Keep this PR open. Current main still lacks candidate-config complete/async setup validation, but the PR needs correctness fixes and a rebase onto the current plugin-owned CLI option surface before it is mergeable.

## What This Changes

This PR adds complete and async channel setup validation hooks, defers wizard and `openclaw channels add` writes until candidate validation, adds VK-oriented setup input fields, and adds a focused wizard validation test.

## Best Possible Solution

Keep the deferred validation seam, but rework it so candidate config is assembled once from the original config plus complete draft input, validation receives the same candidate that would be written, resolved account ids are used consistently, plugin-owned CLI flags stay in metadata, and the feature has changelog and regression coverage.

## Reproduction Assessment

Yes. A focused wizard test can enable deferred validation, derive `resolveAccountId` from `communityId`, then keep or enter a credential; the PR code validates a per-step-mutated candidate and returns the original account id, and a sparse update can reproduce the final `validateInput` rejection.

## Solution Assessment

No. The direction is maintainable, but this PR's implementation validates the wrong candidate in deferred mode, re-runs field validation on sparse draft input, returns the wrong account id, and hardcodes VK setup flags despite current main's generic plugin-owned option surface.
