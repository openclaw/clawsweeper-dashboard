Overall confidence: 0.9

Full review comments:

- **[P1] Build deferred candidates with the final account id:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: In deferred mode, `next` is built during prompts with the original `accountId`, but validation later recomputes `draftAccountId` from complete input. If `resolveAccountId` maps fields like `communityId` to a different account, the candidate being approved is not the config for the account validators see.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: The deferred path computes `draftAccountId` for validation but returns `{ cfg: next, accountId }`. Downstream account recording, bindings, completion notes, or post-write hooks can then use a different account id from the validated candidate.
  - confidence: 0.9
- **[P2] Validate deferred candidates without sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` only contains values touched during this wizard run. Re-running `validateInput` on it can reject a valid deferred update when a credential was kept or a prompt was skipped and the value exists only in `candidateCfg`.
  - confidence: 0.88
- **[P1] Keep VK flags in plugin metadata:** `src/cli/channels-cli.ts:171-172`
  - body: Current main registers `channels add` flags from plugin `cliAddOptions` and forwards non-control options generically, but this diff hardcodes VK flags in shared CLI. That exposes VK options even when VK is not installed and creates a second source of truth for plugin-owned setup fields.
  - confidence: 0.86
- **[P3] Preserve SecretInput for community tokens:** `src/commands/channels/add.ts:25-26`
  - body: `ChannelSetupInput.communityAccessToken` is `SecretInput`, but `ChannelsAddOptions` redeclares it as `string`, so programmatic callers cannot pass a `SecretRef` through the same setup contract.
  - confidence: 0.78
