## Summary

Keep this PR open, but it should not merge as-is: the new deferred setup contract is useful, yet the patch has source-reproducible account-id/candidate-validation bugs, hardcodes VK-specific CLI flags into shared CLI, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds complete and async channel setup validation hooks, defers setup wizard and `channels add` writes until candidate validation, adds VK-oriented setup fields/CLI flags, and adds a focused setup-wizard test.

## Best Possible Solution

If maintainers want this contract, land a revised main-based patch that assembles and validates one candidate config atomically, propagates resolved account ids, keeps channel-specific setup flags plugin-owned, updates SDK docs/types, and includes real behavior proof.

## Reproduction Assessment

Yes, for the PR review findings: source inspection of the PR head shows a deferred wizard can resolve a later `draftAccountId` while validating and returning state tied to the original account, and `draftInput` can be sparse when final validation runs. This is source-reproducible without running the full VK stack.

## Solution Assessment

No. The direction may be useful, but this patch should be reworked so candidate assembly, account-id propagation, sparse input handling, and plugin-owned CLI options align with current main before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build deferred candidates from the complete draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: In deferred mode, `next` has already been mutated one prompt at a time using the original `accountId`. If later draft input changes `resolveAccountId`, this validates a candidate assembled under one account while telling validators to use another; assemble the candidate once from the original config plus the complete draft before validating.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: The deferred path computes `draftAccountId` for validation but still returns the original `accountId`. Downstream account recording, bindings, and config-changed hooks can then refer to a different account than the validated candidate.
  - confidence: 0.9
- **[P2] Avoid final field validation on sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` contains only values changed during this wizard run, while kept credentials or skipped prompts may exist only in `candidateCfg`. Re-running field-level `validateInput` here can reject a valid deferred update before the candidate-aware validators run.
  - confidence: 0.88
- **[P2] Keep channel-specific add flags plugin-owned:** `src/cli/channels-cli.ts:171-172`
  - body: Current main already registers channel-specific `channels add` flags from plugin `cliAddOptions` and forwards non-control option values generically. Hardcoding VK flags in the shared CLI exposes them without the VK plugin and bypasses the metadata-owned setup surface.
  - confidence: 0.84
- **[P3] Preserve SecretInput typing for community tokens:** `src/commands/channels/add.ts:25-26`
  - body: `ChannelSetupInput.communityAccessToken` is declared as `SecretInput`, but `ChannelsAddOptions` redeclares it as `string`, narrowing programmatic callers away from `SecretRef` values. Remove the duplicate field or type it as `SecretInput` so command options match the setup contract.
  - confidence: 0.78
