## Summary

Keep open: current main still lacks candidate-aware setup validation, and this PR is a plausible core SDK/API seam, but it is not mergeable as written because the deferred wizard path has blocking correctness bugs, the branch is conflicting, and the external real-behavior proof gate is still unmet.

## What This Changes

The PR adds complete and async channel setup validation hooks, defers wizard and non-interactive channel writes until candidate validation, extends setup input/CLI flags with VK-oriented fields, and adds a focused wizard test.

## Best Possible Solution

Define a generic plugin-owned candidate-validation setup contract, rebase it onto current main's metadata-driven add options, fix candidate/account-id propagation, and require redacted real CLI or wizard proof before merge.

## Reproduction Assessment

Yes, source-reproducible. A focused wizard scenario can derive `accountId` from a later setup field while earlier prompt mutations were applied to the original account, and kept/skipped credential paths can make final `validateInput` see sparse `draftInput`.

## Solution Assessment

No. The feature direction is plausible, but this patch mixes the generic contract with VK-specific shared CLI flags and needs the deferred candidate assembly, resolved account id, and sparse validation issues fixed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Build the deferred candidate from the full draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: In deferred mode, `next` has already been mutated prompt by prompt before validation. If a later field changes `resolveAccountId`, validators can see a mixed candidate instead of the config that would be written, so build one candidate from the original config plus the complete draft before validating.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: The deferred block computes `draftAccountId` for validation, but the adapter still returns the original `accountId`. Downstream account recording, bindings, and config-changed hooks can then refer to a different account than the validated candidate.
  - confidence: 0.9
- **[P2] Skip final field validation on sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` contains only fields changed in this wizard run, while kept credentials or skipped prompts may exist only in `candidateCfg`. Re-running `validateInput` here can reject a valid deferred update before the candidate-aware validators run.
  - confidence: 0.88
- **[P2] Move VK setup flags to plugin metadata:** `src/cli/channels-cli.ts:168-170`
  - body: Current main already registers channel-specific add flags from plugin `cliAddOptions` and forwards those values generically. Hardcoding VK flags in the shared CLI exposes them without the VK plugin and bypasses the metadata-owned setup surface.
  - confidence: 0.84
