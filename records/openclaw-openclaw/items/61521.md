## Summary

Keep this PR open, but it should not merge as-is. Current main still lacks the requested complete/async channel setup validation hooks, while the PR branch has concrete correctness issues in deferred candidate assembly/account-id propagation and needs to be rebased onto the current generic channel setup option flow.

## What This Changes

This PR adds complete/async channel setup validation hooks, defers wizard and `openclaw channels add` writes until candidate validation, adds VK-oriented setup input/CLI fields, and adds a focused wizard validation test.

## Best Possible Solution

Keep the deferred validation seam, but rework it on current main so candidate config is assembled once from the original config plus complete draft input, candidate-aware validation runs before writes, resolved account ids are used consistently, plugin-owned CLI setup options stay metadata-driven, and the feature has changelog and focused regression coverage.

## Reproduction Assessment

Yes. A focused wizard test can enable deferred validation, derive `resolveAccountId` from `communityId`, then enter or keep a credential; the PR code validates a per-step-mutated candidate and returns the original account id. A second update test with an existing kept credential can reproduce the sparse final `validateInput` rejection.

## Solution Assessment

No. The proposed feature direction is plausible, but this implementation validates the wrong candidate in deferred mode, re-runs field validation with incomplete draft input, returns the wrong account id, and predates current main's generic setup option surface.

Full review comments:

- **[P1] Build the deferred candidate from the full draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: When deferred mode is enabled, `next` has already been mutated one prompt at a time. If `resolveAccountId` depends on a later field such as `communityId`, earlier/later patches can land under the wrong account while this block validates the mixed candidate. Build the candidate once from the original config plus the complete `draftInput`, then validate that result.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: This block computes `draftAccountId` for deferred validation, but the adapter still returns the original `accountId`. Downstream account recording, bindings, and config-changed hooks can then point at a different account than the candidate that was validated.
  - confidence: 0.9
- **[P2] Skip final field validation on sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` only contains values changed during this wizard run, while kept credentials or skipped prompts can exist only in `candidateCfg`. Re-running `validateInput` here can reject a valid deferred update before the candidate-aware complete/async validators run.
  - confidence: 0.88

  - body: Current main already adds channel setup flags from plugin metadata, so hardcoding VK-specific flags into the shared `channels add` command would expose those options even without the VK plugin and conflicts with the current generic setup option flow. Move these flags to the VK plugin metadata when rebasing.
  - confidence: 0.84

  - confidence: 0.82
