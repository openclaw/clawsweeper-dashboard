## Summary

Keep open: this is a real feature/API proposal that is not implemented on current main, but the current PR head still has source-reproducible correctness blockers and lacks inspectable real behavior proof.

## What This Changes

The PR adds complete and async channel setup validation hooks, defers setup wizard and `channels add` writes until candidate validation, adds VK-oriented setup input/CLI fields, and adds a focused seam test.

## Best Possible Solution

Land a revised main-based setup contract only if maintainers want it: assemble one candidate config from the complete input, validate it before writes, propagate resolved account ids consistently, keep VK flags plugin-owned, update SDK docs/types, and include real behavior proof.

## Reproduction Assessment

Yes, for the PR review findings: source inspection of the PR head shows inconsistent deferred account-id handling and sparse draft validation without needing the full VK stack. The requested feature itself is not a bug reproduction case.

## Solution Assessment

No. The direction may be useful, but this patch should be reworked against current main so candidate assembly, account-id propagation, sparse input handling, and plugin-owned CLI options match the existing channel setup architecture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build deferred candidates from the final draft:** `src/channels/plugins/setup-wizard.ts:945-950`
  - body: In deferred mode, `next` is still mutated during each prompt using the original `accountId`. If `resolveAccountId` derives an account from the completed input, validators receive that later account id while `candidateCfg` was assembled under another account, so the config being approved can differ from the account validators think they are checking.
  - confidence: 0.91
- **[P1] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: The deferred path computes `draftAccountId` for validation but returns `{ cfg: next, accountId }`. Downstream account recording, bindings, completion notes, or post-write hooks can then use a different account id from the validated candidate.
  - confidence: 0.9
- **[P2] Validate legacy fields against complete input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` only contains values touched during this wizard run. Re-running `validateInput` on it can reject a valid deferred update when a credential was kept or prompt skipped and the value exists only in `candidateCfg`; either validate the complete candidate-aware shape or skip field validation here.
  - confidence: 0.88
- **[P2] Keep channel-specific flags in plugin metadata:** `src/cli/channels-cli.ts:171-172`
  - body: Current main registers `channels add` flags from plugin `cliAddOptions` and forwards non-control options generically, but this diff hardcodes VK flags in shared CLI. That exposes VK options even when VK is not installed and bypasses the plugin-owned setup surface.
  - confidence: 0.84
- **[P3] Preserve SecretInput for community tokens:** `src/commands/channels/add.ts:25-26`
  - body: `ChannelSetupInput.communityAccessToken` is `SecretInput`, but `ChannelsAddOptions` redeclares it as `string`, so programmatic callers cannot pass a `SecretRef` through the same setup contract.
  - confidence: 0.78
