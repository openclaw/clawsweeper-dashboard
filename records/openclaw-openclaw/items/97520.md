## Summary

Keep open: this is a focused auth-provider bug fix, but the malformed-key detector misses documented non-interactive onboarding command forms, and the external PR proof is only test/fixture output rather than real behavior proof.

## What This Changes

The branch adds a shared malformed API-key detector for OpenClaw onboarding command text, applies it to auth profile eligibility, provider API-key setup validation, auth health/doctor reporting, and regression tests.

## Best Possible Solution

Broaden the malformed-command detector to reject `openclaw onboard` invocations where `--auth-choice` appears after other flags, or adopt the provider-owned variant-covered path from https://github.com/openclaw/openclaw/pull/97344, then add redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current docs include `openclaw onboard --non-interactive --auth-choice ...`, while the PR detector only matches `--auth-choice` immediately after `onboard`. A live Z.AI network run was not performed.

## Solution Assessment

No: the shared eligibility boundary is a plausible place to reject malformed stored keys, but this implementation is too narrow for documented command forms. The safer fix is variant-aware parsing with tests, or the provider-owned path from the related PR if maintainers prefer that boundary.
