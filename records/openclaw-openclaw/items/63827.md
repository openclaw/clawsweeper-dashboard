## Summary

Keep this PR open for maintainer handling: the linked dmPolicy preservation bug still appears present on current main, but this branch is not mergeable as-is because the dmPolicy change targets the old command file, has a compile-time scoping error already identified in review comments, and bundles unrelated security, dependency, advisory, test, changelog, and gateway changes.

## What This Changes

The PR adds an `initialValue: current` default to the old channel DM policy prompt and also bundles gateway, channel, dependency, test, changelog, and security-advisory changes.

## Best Possible Solution

Land a narrow current-main fix in `src/flows/channel-setup.prompts.ts` that reads the current policy before prompting and passes it as `initialValue`, with focused regression coverage for preserving `allowlist` and other non-default policies; handle the unrelated GHSA, dependency, and gateway changes in separate secops-owned PRs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a channel with `dmPolicy: "allowlist"`, re-run channel setup, choose to configure DM policies, and the current-main prompt has no `initialValue` before `current` is read, so the default selection can overwrite the existing policy.

## Solution Assessment

No. This PR is not the best way to solve the issue because its stated one-line fix is scoped to the old command implementation and will not compile; the safer path is a small patch against the current flow plus targeted tests.
