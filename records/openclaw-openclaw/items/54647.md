## Summary

Keep this PR open: it targets a valid gateway bug that is still present on current main, but the branch has a blocking provider-preservation regression and still lacks after-fix real behavior proof.

## What This Changes

The PR updates gateway session-row model resolution and tests so ACP session rows read `entry.acp.runtimeOptions.model` and avoid default-model fallback when the ACP runtime model is unknown.

## Best Possible Solution

Rework the current gateway session-row resolver so ACP rows use explicit ACP runtime metadata when present, leave `modelProvider` and `model` absent when the ACP runtime model is unknown, and preserve existing generic-session provider, transcript, and display normalization behavior with focused regressions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a gateway `sessions.list` ACP row with only `entry.acp.runtimeOptions.model`, or with no ACP runtime model, still flows through the generic resolver and default-model fallback on current main.

## Solution Assessment

No. The PR targets the right bug, but the current patch re-parses prefixed model strings even when an explicit provider is stored and needs rework against current session-list resolver behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve explicit providers for prefixed models:** `src/gateway/session-utils.ts:1058-1060`
  - body: When `modelProvider` is already stored, this helper should treat it as authoritative.   - confidence: 0.9
