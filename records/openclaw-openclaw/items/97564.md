- `P2`: This is a normal-priority agent-runtime fallback reliability bug with a concrete provider/model routing failure mode and limited blast radius.
- `impact:auth-provider`: The affected behavior is model/provider fallback routing after the primary provider reports the selected model is unavailable.

## Summary

Keep open: current main and v2026.6.10 still encode the retry-limit `model_not_found` terminal exception the report points at; the normal direct model-not-found path already falls back, so the remaining work is a narrow retry-limit edge rather than a broad fallback absence.

## What This Changes

The issue reports that `model_not_found` failover reasons are treated as terminal, so configured model fallbacks are skipped when a primary model disappears.

## Best Possible Solution

Keep the existing direct `model_not_found` fallback behavior and close the retry-limit gap with a small policy/test change rather than adding a new config flag.

## Reproduction Assessment

Yes, source-reproducible: current main returns `return_error_payload` for retry-limit exhaustion with `failoverReason: "model_not_found"` even when `fallbackConfigured` is true. I did not run a live provider-decommission repro.

## Solution Assessment

Yes for the narrow path: update the existing retry-limit failover decision and tests while preserving terminal behavior when no fallback is configured. A new config flag would be heavier than the existing documented fallback contract requires.
