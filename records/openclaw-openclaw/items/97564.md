- `P2`: This is a normal-priority provider/model fallback reliability bug with a concrete source path and limited blast radius.
- `impact:auth-provider`: The issue affects provider/model routing and whether a configured fallback model is attempted after a provider returns `model_not_found`.
- `impact:message-loss`: A configured fallback chain can be skipped, leaving the user with a hard error instead of a fallback-generated reply.

## Summary

Keep open: current main and the latest release still exclude `model_not_found` from the retry-limit path that throws into model fallback, even though the broader model-fallback layer already treats model-not-found as a fallback reason when a fallback candidate exists.

## What This Changes

The issue reports that `model_not_found` is treated as terminal, so configured model fallbacks are not attempted when a primary model disappears.

## Best Possible Solution

Align the retry-limit failover decision with the existing model-fallback contract by treating `model_not_found` as fallback-eligible only when an explicit downstream fallback remains, and update focused tests/docs without adding a new config flag by default.

## Reproduction Assessment

Yes at source level: current main carries `model_not_found` into `lastRetryFailoverReason`, but the retry-limit decision excludes that reason and returns a local error payload instead of throwing into model fallback. I did not run a live provider repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to adjust the existing retry-limit failover policy and regression tests so it matches the already-supported `runWithModelFallback` behavior for `model_not_found`, while keeping exhausted/no-fallback cases terminal.
