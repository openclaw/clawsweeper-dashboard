## Summary

Keep open. Current main still rejects Android for managed Gateway service lifecycle, and the Android docs define Android as a companion node that requires a Gateway on another host. AndroidForClaw is useful external context, but it does not implement OpenClaw's own `openclaw gateway start` support or make this clearly closeable as ClawHub work.

## What This Changes

Request Android/Termux support for running the OpenClaw Gateway directly on an Android device, including local dashboard access and remote node connections without a PC or VPS.

## Best Possible Solution

Keep this as the canonical maintainer platform thread. The desired end state is either a supported Android/Termux Gateway mode with lifecycle handling and docs, or an explicit product decision that Android remains node-only while docs point users to maintained external options such as AndroidForClaw or a future ClawHub package.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on `process.platform === "android"`, `resolveGatewayService()` misses the registry and throws on the `openclaw gateway start` path; I did not run a live Termux device reproduction in this read-only review.

## Solution Assessment

Unclear. The request is a valid canonical tracker, but the best implementation needs maintainer platform/security judgment before a narrow code fix can be chosen.
