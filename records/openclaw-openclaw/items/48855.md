## Summary

Keep open. Current main still routes `openclaw gateway start` through the managed service resolver, which only supports macOS, Linux, and Windows, while the Android docs explicitly define Android as a companion node rather than a Gateway host. AndroidForClaw is useful external context, but it does not make OpenClaw's own Android/Termux Gateway lifecycle support implemented or clearly closeable as plugin-only work.

## What This Changes

Request support for running the OpenClaw Gateway service directly on Android/Termux so mobile-only users can access the dashboard and remote nodes without a separate PC or VPS.

## Best Possible Solution

Keep this as the canonical platform thread until maintainers decide whether to support an Android/Termux Gateway mode with lifecycle, security, and docs, or explicitly keep Android node-only while documenting maintained external options.

## Reproduction Assessment

Yes. The reported `gateway start` failure is reproducible from source: `runDaemonStart()` resolves the managed service first, and `resolveGatewayService()` throws for platforms outside `darwin`, `linux`, and `win32`, which includes Node on Android.

## Solution Assessment

Unclear. Adding Android to the service registry alone would not be a maintainable fix; the right solution needs a deliberate Android/Termux lifecycle model, security posture, and documentation boundary.
