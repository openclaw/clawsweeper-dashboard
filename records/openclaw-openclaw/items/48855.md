## Summary

Keep open. Current main and the latest shipped service code still keep Android as a companion node while managed Gateway service adapters are limited to macOS, Linux/WSL2, and Windows; the request is a new Android/Termux Gateway-hosting product decision, and the linked AndroidForClaw project is useful external context/workaround rather than a core resolution.

## What This Changes

Request Android/Termux Gateway service support so an Android device can host the Gateway/dashboard without a separate PC, VPS, or always-on Linux box.

## Best Possible Solution

Maintainers should decide the official Android/Termux Gateway-hosting policy, then either add a maintained Termux/mobile Gateway mode with docs and validation or explicitly keep Android node-only while documenting supported workarounds and external projects like AndroidForClaw.

## Reproduction Assessment

Not applicable as a feature request. The current unsupported behavior is high-confidence from source and docs: Android is documented as node-only and the service resolver has no Android adapter.

## Solution Assessment

Unclear. Adding Termux service support might solve the request, but the maintainable path needs a platform/product decision first; changing only the service registry would not address Android lifecycle, security, dashboard, and persistence semantics.
