## Summary

Keep open: the CJK false-overflow bug is credible, but this branch is not merge-ready because the one-line divisor change also reduces non-CJK and JSON/object tool-result pressure, leaves the sibling guard estimate unaligned, and still has only unit/synthetic proof.

## What This Changes

This PR changes the embedded-agent pre-prompt tool-result token divisor from 2 to 4 and adds/updates CJK and mixed-language token-estimation tests.

## Best Possible Solution

Use a shared CJK-aware tool-result estimator that preserves current conservative Latin/JSON pressure, align the pre-prompt and guard paths, cover both CJK and non-CJK regressions, and add redacted real CJK workflow proof.

## Reproduction Assessment

Yes by source inspection and related prior evidence: current main applies CJK-aware character weighting and then divides tool-result content by 2, which overestimates CJK text at about two tokens per character. I did not run a live CJK workflow in this read-only review.

## Solution Assessment

No. The bug is real, but the global divisor change is too broad because it also halves Latin and JSON/object tool-result pressure; the safer fix is a per-script/shared estimator across the pre-prompt and guard paths.
