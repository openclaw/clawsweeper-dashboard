## Summary

Keep open; the reported CJK false-overflow problem is real, but this patch is not merge-ready because it globally halves tool-result pressure for non-CJK payloads, leaves sibling guard accounting inconsistent, and has no after-fix real behavior proof.

## What This Changes

The PR changes the embedded agent pre-prompt tool-result token divisor from 2 to 4 and adds/adjusts preemptive-compaction unit coverage for CJK and mixed-language tool results.

## Best Possible Solution

Use a shared CJK-aware tool-result estimator that preserves current conservative Latin/JSON pressure, align the pre-prompt and guard paths, cover both CJK and non-CJK regressions, and add redacted real CJK workflow proof.

## Reproduction Assessment

Yes by source inspection and linked prior PR context: current main divides CJK-weighted tool-result chars by 2, which overestimates CJK tool results, and the earlier closed PR includes real CJK false-precheck evidence. I did not run a live CJK workflow in this read-only review.

## Solution Assessment

No. A global divisor change is too broad because it also halves Latin and JSON/object tool-result pressure; the best fix should be per-script/shared across the pre-prompt estimator and tool-result guard.
