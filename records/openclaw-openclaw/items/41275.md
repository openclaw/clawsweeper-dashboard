## Summary

Keep open. Current main still has the Control UI mismatch: the form rejects explicit `timeoutSeconds: 0` and the payload builder omits zero, while the Gateway schema and cron runtime already support zero as no-timeout. The PR is aimed at the right controller path, but it is not merge-ready because it leaves a stale controller test and user-facing timeout copy behind.

## What This Changes

The PR updates Control UI cron timeout validation and payload construction so `timeoutSeconds: 0` is accepted and preserved while malformed or negative timeout values are rejected.

## Best Possible Solution

Amend this branch so the Control UI accepts and preserves explicit `timeoutSeconds: 0`, still rejects negative and non-finite timeout input, updates controller coverage for validation and payload construction, and refreshes the English timeout copy plus generated locale output if the i18n workflow requires it.

## Reproduction Assessment

Yes. Current main gives a high-confidence reproduction path: `validateCronForm` rejects `timeoutSeconds: "0"`, `buildCronPayload` omits explicit zero, and the existing controller test asserts the old invalid-zero behavior.

## Solution Assessment

No, not as currently proposed. The controller parsing and payload direction is narrow and maintainable, but the mergeable fix should update the stale test expectation and timeout copy alongside the code change.
