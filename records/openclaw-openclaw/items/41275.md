## What This Changes

The PR changes the Control UI cron controller so `timeoutSeconds: 0` is accepted, preserved in the payload, and negative or non-finite timeout input is rejected.

## Best Possible Solution

Keep this focused UI-layer fix, update the controller regression tests for zero, malformed, and negative timeout input, and require redacted proof that a cron job with `timeoutSeconds: 0` can be saved through the Control UI.

## Reproduction Assessment

Yes, source-level. Current main rejects `timeoutSeconds: 0` in `validateCronForm`, omits explicit zero in `buildCronPayload`, and the backend schema/runtime tests show zero is already a valid no-timeout value.

## Solution Assessment

Yes for the owner boundary, but the PR is incomplete. The maintainable fix belongs in the UI validation and payload builder, with focused controller tests and real behavior proof before merge.
