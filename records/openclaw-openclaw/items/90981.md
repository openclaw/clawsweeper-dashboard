## Summary

Keep open: current main and v2026.6.6 still keep `sessions_history` bounded to `sessionKey`, `limit`, and `includeTools` with no cursor/export parameter. HTTP cursor pagination and existing export commands are adjacent surfaces, but they do not fully satisfy the requested agent-tool/readable-export workflow, so this remains a maintainer product/API decision rather than a cleanup close.

## What This Changes

The issue requests pagination or export support for `sessions_history` so long conversations can be retrieved without losing older messages to bounded output truncation.

## Best Possible Solution

Define one canonical, visibility-checked session-history pagination/readable-export contract that reuses the existing redaction and cursor semantics where possible, and keep raw mirror-to-path automation in plugin/ClawHub territory unless maintainers explicitly want it in core.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current limitation is source-reproducible: the tool schema and Gateway call lack cursor/export inputs and still apply the bounded history cap.

## Solution Assessment

Unclear until maintainers choose the product shape. Pagination through the existing cursor model looks safer than raising a global cap, while readable export must preserve the current visibility and redaction contracts.
