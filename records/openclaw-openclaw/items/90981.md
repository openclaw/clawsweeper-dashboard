## Summary

Keep open: current main and v2026.6.8 still expose `sessions_history` as a bounded, sanitized recall tool with no model-facing cursor, offset, page token, or clean readable export contract. The existing REST history cursor and `/export-session` command are useful adjacent surfaces, but they do not satisfy this request without a maintainer product and transcript-security decision.

## What This Changes

The issue requests pagination or export support for `sessions_history` so long conversations can be retrieved beyond the current bounded truncated tool result.

## Best Possible Solution

Define one canonical visibility-checked model-facing pagination and readable-export contract, likely reusing existing REST cursor semantics while preserving redaction and avoiding a new global cap knob unless maintainers explicitly choose it.

## Reproduction Assessment

Yes, at source level: current main exposes no `sessions_history` cursor/export input and still caps sanitized tool output at 80 KiB. I did not run a live long-session repro because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the product shape. Reusing the existing HTTP cursor semantics for model-facing pagination looks safer than raising a global cap, while any readable export must preserve visibility, redaction, and reasoning/tool filtering contracts.
