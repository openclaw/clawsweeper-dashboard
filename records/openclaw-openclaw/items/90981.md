## Summary

Keep open: current main and v2026.6.8 still expose `sessions_history` as a bounded sanitized recall tool with no model-facing cursor, page token, offset, readable export, or safe full-history contract. The existing HTTP history cursor and `/export-session` command are useful adjacent surfaces, but they do not satisfy this issue without a maintainer API and transcript-security decision.

## What This Changes

The issue asks for pagination, offset, or readable export support for `sessions_history` so long conversations can be retrieved beyond the current bounded response.

## Best Possible Solution

Define one canonical, visibility-checked model-facing pagination and readable-export contract that reuses existing cursor/redaction behavior where possible, and avoid a new global config size knob unless maintainers explicitly accept that surface.

## Reproduction Assessment

Yes at source level: current main exposes no `sessions_history` cursor/export input and still caps sanitized tool output at 80 KiB. I did not run a live long-session repro because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the product shape. Reusing the existing HTTP cursor semantics for model-facing pagination looks safer than raising a global cap, while any readable export must preserve visibility, redaction, and reasoning/tool filtering contracts.
