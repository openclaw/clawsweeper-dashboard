## Summary

Keep open: current main and v2026.6.6 still expose `sessions_history` as a bounded, redacted recall tool with no cursor, offset, page token, or clean export mode. Adjacent HTTP cursor pagination and `/export-session`/trajectory export exist, but they do not fully solve the requested agent-tool pagination/export workflow.

## What This Changes

Add pagination/cursor or export support to `sessions_history` so long conversations can be retrieved beyond the current bounded tool response.

## Best Possible Solution

Define one canonical, visibility-checked, redacted pagination/export contract for session history, preferably reusing the existing HTTP cursor snapshot semantics and avoiding a global size-limit config knob unless maintainers deliberately approve it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main only supports bounded `sessionKey`/`limit` history and still drops older rows under the hard cap.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Pagination through `sessions_history` is plausible, but the best implementation should reuse existing Gateway cursor semantics and export contracts rather than adding a parallel history format or a broad config knob.
