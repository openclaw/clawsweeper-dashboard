## Summary

Keep open: current main and v2026.6.8 still keep `sessions_history` bounded to `sessionKey`, `limit`, and `includeTools`, with no cursor/export/readable-log contract exposed through the agent tool. The adjacent HTTP history cursor and current-session export commands do not fully satisfy the requested model-facing pagination or clean export workflow, so this remains a maintainer API and security-boundary decision.

## What This Changes

Request to add pagination/offset or clean export support for `sessions_history` so long conversations can be retrieved beyond the current bounded result.

## Best Possible Solution

Define one canonical, visibility-checked session-history pagination and readable-export contract that reuses existing redaction and cursor semantics where possible, and avoid a new config size knob unless maintainers explicitly accept that product surface.

## Reproduction Assessment

Yes, at source level: current main exposes no `sessions_history` cursor/export input and still returns a bounded, capped recall view. I did not run a live long-session repro because this was a read-only cleanup review.

## Solution Assessment

Unclear until maintainers choose the product shape. Pagination through the existing cursor model looks safer than raising a global cap, while readable export must preserve visibility, redaction, and thinking/tool filtering contracts.
