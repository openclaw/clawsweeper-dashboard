## Summary

Keep open. The reporter provided concrete 2026.5.20 logs, and current main still lets `openai/gpt-5.4-nano` run through the Codex searchable dynamic-tool path that asks Codex app-server to use `tool_search`, so the fallback to `gpt-5.4-mini` remains actionable.

## What This Changes

The issue asks for `openai/gpt-5.4-nano` to work through the Codex harness without unsupported `tool_search` errors and model fallback.

## Best Possible Solution

Gate Codex dynamic-tool search and reasoning-effort handling by the selected model or app-server capability so `gpt-5.4-nano` runs without `tool_search`, while capable models keep searchable dynamic tools.

## Reproduction Assessment

Yes, by source plus reporter evidence: configure `openai/gpt-5.4-nano` through the Codex harness with default searchable dynamic tools, then send a turn and Codex app-server can reject `tool_search`. I did not run the live credentialed request in this read-only review.

## Solution Assessment

Yes. The best fix is a narrow capability gate in the Codex app-server tool/reasoning preparation path, not the user workaround of switching to PI or disabling searchable dynamic tools for every model.
