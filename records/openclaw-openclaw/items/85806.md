## Summary

Keep open. The report has concrete 2026.5.20 logs, and current `main` still has `openai/gpt-5.4-nano` flowing through the Codex app-server dynamic-tool path without a no-tools compatibility flag, so the `tool_search` failure is still actionable.

## What This Changes

The issue reports that the Codex harness with `openai/gpt-5.4-nano` sends `tool_search`, gets an OpenAI 400 error, and falls back to `gpt-5.4-mini`.

## Best Possible Solution

Gate Codex dynamic-tool search and reasoning-effort handling by the selected model's real capability, so `gpt-5.4-nano` can run without `tool_search` while larger Codex-capable models keep searchable dynamic tools.

## Reproduction Assessment

Yes, by source and reporter evidence: configure `openai/gpt-5.4-nano` through the Codex harness with default searchable dynamic tools, then send a chat turn and watch Codex app-server reject `tool_search`. I did not run the live OpenAI request because this review is read-only and credential-dependent.

## Solution Assessment

The user workaround of switching to PI is not the best long-term fix. The maintainable repair is to use existing model capability/compat gates so nano does not receive unsupported Codex dynamic-tool search, with regression coverage that larger models keep the current path.
