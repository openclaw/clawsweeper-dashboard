## Summary

Keep open. The reporter provided concrete 2026.5.20 fallback logs, and current main still sends searchable Codex dynamic-tool specs for `openai/gpt-5.4-nano` even though OpenAI's model docs say nano does not support Tool search.

## What This Changes

The issue asks for Codex harness runs on `openai/gpt-5.4-nano` to complete without `tool_search` errors or fallback to a more expensive model.

## Best Possible Solution

Add a model-capability gate in the Codex app-server dynamic-tool path so nano avoids deferred Tool search while supported Codex/OpenAI models keep the existing searchable tool behavior.

## Reproduction Assessment

Yes, by source and reporter evidence: configure `openai/gpt-5.4-nano` through the Codex harness with default searchable dynamic tools, then send a chat turn and Codex app-server rejects `tool_search`. I did not run the live OpenAI request because credentials and a real Codex runtime are required.

## Solution Assessment

No, the current workaround of switching to PI or relying on fallback is not the best fix. The maintainable solution is to gate Codex dynamic-tool search by the selected model's actual Tool search capability with focused regression coverage.
