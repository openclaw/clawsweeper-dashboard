## Summary

Keep open. Current main still has the Codex searchable dynamic-tool path that can ask the app-server to use `tool_search` for `openai/gpt-5.4-nano`, matching the reporter's 2026.5.20 fallback logs; older nano/Codex reports cover route/catalog issues, not this exact `tool_search` rejection.

## What This Changes

The issue reports that OpenAI Codex harness requests with `openai/gpt-5.4-nano` send unsupported `tool_search` tools and fall back to `openai/gpt-5.4-mini`.

## Best Possible Solution

Gate Codex dynamic-tool search and reasoning/tool preparation by the selected model or app-server capability so `gpt-5.4-nano` runs without unsupported `tool_search` while capable models keep searchable deferred tools.

## Reproduction Assessment

Yes. Source plus reporter evidence gives a high-confidence path: configure `openai/gpt-5.4-nano` through the Codex harness with default searchable dynamic tools, then send a turn and the app-server can reject `tool_search`.

## Solution Assessment

Yes. A narrow Codex model/capability gate is the maintainable fix; switching users to PI or globally disabling searchable dynamic tools would either avoid the harness or regress capable Codex models.
