## Summary

Keep open: current main still launches the managed browser with a persistent user-data-dir and exposes live Playwright cookie APIs, but there is no durable cookie save/restore path for managed-profile cookies. The prior same-symptom PR was closed unmerged because its stale plaintext cookie sidecar was not an acceptable security/product shape, so this needs maintainer and security direction rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to persist managed browser cookies across browser and gateway restarts so login sessions survive managed-profile restarts.

## Best Possible Solution

Keep this issue open and design a current `extensions/browser` managed-profile fix that preserves login state without creating an unsafe default cookie sidecar, then prove browser and gateway restart behavior with live Chrome evidence.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review because Chrome/OpenClaw was not run. The current source path, latest-release report, prior same-symptom issue, and absence of any cookie persistence layer make the bug source-reproducible enough to keep open.

## Solution Assessment

Unclear: cookie export/import through CDP is a plausible repair mechanism, especially with `Storage.*`, but the prior default plaintext sidecar was the wrong product/security shape. The best fix needs current-path browser lifecycle ownership, sensitive-state handling, and live restart proof.
