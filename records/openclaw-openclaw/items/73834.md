## Summary

Keep this PR open, but it should not merge as written. Current main still lacks the requested Google Meet barge-in surface, and the linked issue remains open, but the PR head still has a correctness blocker: the new Google Meet side-channel can call the provider hook after immediate mark acknowledgement has already cleared the OpenAI truncation state, leaving provider output able to continue after local playback is cleared.

## What This Changes

The PR adds an optional realtime voice `handleBargeIn()` bridge hook, Google Meet Chrome `bargeInInputCommand` config/metadata/docs, loopback suppression during assistant playback, local playback clearing on detected speech, and focused config/test coverage.

## Best Possible Solution

Keep the draft available, but require a patch update that makes provider-side cancellation or truncation independent of transport mark acknowledgement state, preserves the optional Google Meet-owned side-channel command model, keeps existing external bridge behavior unchanged, and lands with focused Google Meet/OpenAI realtime regression tests plus SDK/config/docs checks.

## Reproduction Assessment

Yes for the PR blocker: the source path is deterministic from Google Meet's `ack-immediately` mark strategy through OpenAI's mark-gated `handleBargeIn()` truncation logic. No live Meet reproduction was run in this read-only review.

## Solution Assessment

No, not as written. The overall architecture is plausible, but external barge-in must reliably stop provider output even when transport marks have already been acknowledged; a safer fix is to keep or derive provider cancellation state separately from playback mark queues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep provider truncation effective for barge-in:** `extensions/google-meet/src/realtime.ts:274`
  - body: The Google Meet command-pair bridge uses `markStrategy: "ack-immediately"`, so `session-runtime` immediately calls `acknowledgeMark()` and the OpenAI bridge clears `markQueue`, `responseStartTimestamp`, and `lastAssistantItemId`. When this new side-channel later calls `bridge?.handleBargeIn()`, OpenAI usually falls through to `onClearAudio()` without sending `conversation.item.truncate`, so the active provider response can keep streaming new audio after the local buffer is cleared. Make external barge-in cancellation/truncation independent of transport mark acknowledgements, or keep the needed response state until barge-in is handled.
  - confidence: 0.87
