## Summary

Keep open: the report describes a plausible provider-qualified routing regression, but current main's normal `/model` path should persist the selected provider/model as a user override and disable fallback, so cleanup would be premature without a focused runtime reproduction.

## What This Changes

The issue reports that `/model qianfan/deepseek-v4-flash` appears to succeed but later routes to another provider with the same model id and ignores the configured fallback expectation.

## Best Possible Solution

Preserve provider-qualified model identity through `/model`, queued followups, status, and fallback resolution, and handle any resilient user-override fallback policy as an explicit maintainer-approved config/API change.

## Reproduction Assessment

No high-confidence current-main reproduction is established. Source inspection shows the normal `/model` path should mark the selection as a strict user override and pass an empty fallback override, so the missing proof is a focused duplicate-provider runtime repro with session state before and after the request.

## Solution Assessment

Unclear. The narrow bug fix would keep strict provider-qualified user overrides from falling through to another provider, while letting user overrides inherit configured fallbacks is a separate product/API decision already explored by https://github.com/openclaw/openclaw/pull/80181.
