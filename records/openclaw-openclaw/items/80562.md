## Summary

Keep open: the report describes a plausible routing bug, but current main and the shipped release both define `/model` user selections as strict and pass an explicit empty fallback override for that path, so the observed fallback needs a narrower runtime/channel reproduction before cleanup or automation.

## What This Changes

The issue reports that selecting `qianfan/deepseek-v4-flash` can appear to succeed while runtime status shows fallback to `Deepseek/deepseek-v4-flash` instead of the configured fallback chain.

## Best Possible Solution

Preserve provider-qualified model identity and make `/model` behavior explicit: strict selections should not silently use another provider, and any resilient fallback behavior should be deliberate and documented.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection shows the normal `/model` path should mark the selection as a user override and disable fallback, so the report needs logs or a focused failing path showing where that state is lost.

## Solution Assessment

Unclear. A narrow bug fix would enforce the documented strict no-hidden-fallback behavior, while using configured fallbacks for user-selected models is a product-policy change already explored by an open opt-in PR.
