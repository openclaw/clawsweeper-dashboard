## Summary

Keep open: the report describes a plausible routing regression, but current source, docs, and release code all say `/model` user selections should be strict and should pass an empty fallback override, so this needs a focused runtime reproduction before cleanup or automation.

## What This Changes

The issue asks OpenClaw to keep duplicate provider/model IDs independent when selected with `/model` and to avoid silently routing the call to a different provider or unintended fallback.

## Best Possible Solution

Preserve provider-qualified model identity through `/model`, queued followups, and fallback resolution; if maintainers want user-selected models to inherit fallbacks, make that an explicit documented policy rather than a hidden redirect.

## Reproduction Assessment

No high-confidence reproduction is established. Source inspection shows the normal `/model` path should mark the selection as a user override and disable fallback, so the next useful proof is a focused duplicate-provider runtime repro with session state before and after the request.

## Solution Assessment

Unclear. The narrow bug fix would keep strict provider-qualified user overrides from falling through to another provider, while using the configured fallback chain for user overrides is a product/API decision already explored by https://github.com/openclaw/openclaw/pull/80181.
