## Summary

Keep open: the PR targets real Nostr DM delivery bugs still present on current main, but it is not merge-ready because the media fallback can expose raw media URLs, the branch needs current-main adaptation, and the PR lacks after-fix real behavior proof.

## What This Changes

The branch changes the Nostr DM subscription filter, account lifecycle, outbound media fallback, and regression tests to restore NIP-04 DM delivery.

## Best Possible Solution

Land a current-main Nostr fix that passes a plain Filter to subscribeMany, keeps startNostrGatewayAccount pending until abort with cleanup, leaves Nostr text-only or validates only public HTTP(S) media URLs, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Configure Nostr and start the gateway: current main still passes an array filter to nostr-tools subscribeMany and returns startAccount immediately, which the gateway treats as an exit; I did not run a live relay smoke.

## Solution Assessment

No as-is. The filter and lifecycle changes are the right direction, but the patch should be adapted to current gateway.ts and should remove or strictly validate the media fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject local media URLs in the Nostr fallback:** `extensions/nostr/src/channel.ts:164-169`
  - body: The new sendMedia adapter copies mediaUrl directly into DM text. Outbound media contexts can contain local paths, file:// URLs, data URLs, or scoped internal references, so this can leak host details and send unusable private references to Nostr recipients; rely on the core text fallback or allow only validated public HTTP(S) URLs.
