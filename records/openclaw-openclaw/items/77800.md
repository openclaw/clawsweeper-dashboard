## Summary

Keep open: current main still rejects plugin tool registration without an exact manifest contract, and an open linked PR is the active fix path; the latest operational comment also confirms the failure still appears on 2026.5.7 and post-merge main.

## What This Changes

The issue reports a 2026.5.2 regression where `api.registerTool()` calls without `contracts.tools` silently fail for dynamic plugin tools and asks for clearer failure behavior, wildcard support, or migration guidance.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/78084; if rejected, choose an explicit `registerTool` error or warning contract and split the startup-warning request if needed.

## Reproduction Assessment

Yes, at source level: current `registerTool` returns after adding the reported diagnostic, and `loader.test.ts` asserts no tool is registered without `contracts.tools`. I did not run tests because this review is read-only.

## Solution Assessment

Unclear: restoring implicit runtime ownership would conflict with the manifest-first plugin contract, while the linked bounded-wildcard PR is plausible but still needs maintainer acceptance and real behavior proof.
