## Summary

Keep open: current main still rejects undeclared plugin tools, and the report raises a real plugin SDK compatibility/migration question rather than something already fixed or safely closeable.

## What This Changes

Plugins that call `api.registerTool()` without manifest `contracts.tools` stopped exposing those tools after 2026.5.2, with only registry diagnostics visible to operators.

## Best Possible Solution

Define an explicit backward-compatible plugin tool migration path that preserves manifest-first discovery while giving dynamic third-party plugins a supported ownership contract or a clear plugin-visible failure.

## Reproduction Assessment

Yes. Source and existing tests show current main drops `api.registerTool()` registrations when `contracts.tools` is absent or mismatched, which matches the reported missing tools and allowlist failure path.

## Solution Assessment

Unclear. Reverting to unconstrained runtime registration is probably not the best fix because it conflicts with manifest-first discovery; the safer direction is a deliberate SDK/manifest compatibility design plus diagnostics and migration docs.
