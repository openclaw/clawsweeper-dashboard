## Summary

Keep open: current main still rejects undeclared plugin tool registrations, and the remaining question is an SDK/product decision rather than a stale cleanup close. The wildcard/dynamic-catalog part is also being tracked in related open issue #77797 by the same author.

## What This Changes

A plugin author asks OpenClaw to stop dropping `api.registerTool()` registrations when `contracts.tools` is missing and to provide a dynamic-tool migration path.

## Best Possible Solution

Define an explicit dynamic plugin-tool ownership contract, then align `registerTool` failure behavior, docs, changelog guidance, and regression tests with that contract.

## Reproduction Assessment

Yes, at source level: current `registry.ts` returns after adding the reported diagnostic, and `loader.test.ts` has a focused fixture that asserts no tool is registered without `contracts.tools`. I did not run tests because this review was required to stay read-only.

## Solution Assessment

Unclear: restoring pre-2026.5.2 implicit runtime ownership would conflict with the documented manifest-first plugin contract. A maintainer should choose between explicit wildcard/prefix ownership, a different dynamic-tool API, or clearer `registerTool` failure semantics.
