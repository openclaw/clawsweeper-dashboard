## Summary

Keep open: current main and v2026.6.10 still skip `api.registerTool()` when manifest `contracts.tools` is absent or not an exact match, and the linked dynamic-contract fix remains open with API, security-boundary, and proof questions.

## What This Changes

The issue asks to restore or make diagnosable dynamic plugin tool registration after v2026.5.2 began rejecting `api.registerTool()` calls without manifest `contracts.tools`.

## Best Possible Solution

Resolve this through a maintainer-approved plugin manifest contract and diagnostics path that preserves manifest-first discovery, keeps explicit allowlists narrow, and lands either the linked PR or a narrower replacement with live proof.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the registry returns without adding tools when `contracts.tools` is absent, and `src/plugins/loader.test.ts` asserts the dropped-tool behavior.

## Solution Assessment

Unclear. Bounded wildcard contracts are plausible, but the final fix needs maintainer API/security-boundary approval; a warning-only change would improve diagnostics without restoring dynamic tool registration.
