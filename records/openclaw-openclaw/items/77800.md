## Summary

Keep this open. Current main and the latest release still exact-match `contracts.tools` and reject undeclared dynamic `api.registerTool()` names through diagnostics, while the linked closing PR is still open and has unresolved API/proof review work.

## What This Changes

The issue asks OpenClaw to restore or explicitly support dynamic plugin tool registration after `api.registerTool()` began rejecting tools not declared in `contracts.tools`.

## Best Possible Solution

Review, revise, and land or close the linked dynamic tool-contract PR so OpenClaw has one documented, tested contract for dynamic plugin tools while preserving manifest-first discovery.

## Reproduction Assessment

Yes. Source inspection shows current main records a diagnostic and returns when `contracts.tools` is absent, and the adjacent loader test asserts that no tools are registered in that case; I did not run a live plugin install because this review is read-only.

## Solution Assessment

Unclear as a final product/API decision. The bounded wildcard direction in the linked PR is plausible, but it is still unmerged and has review concerns around explicit allowlist filtering and real behavior proof.
