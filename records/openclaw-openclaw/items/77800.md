## Summary

Keep open: current main and v2026.6.10 still reject dynamic `api.registerTool()` registrations unless `contracts.tools` exact-matches the runtime tool names, and the linked closing PR is open, unmerged, and still has API/security/proof blockers.

## What This Changes

Restore or replace the pre-2026.5.2 dynamic plugin tool registration path so `api.registerTool()` without a static `contracts.tools` list no longer fails silently and dynamic tool authors have a supported manifest/API path.

## Best Possible Solution

Land or revise the linked dynamic-tool contract path so manifest-first discovery is preserved, concrete allowlists stay narrow, and plugin authors get documented tests plus real plugin/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main records a diagnostic and returns when `contracts.tools` is absent, and the adjacent loader test asserts no tools are registered. I did not run a live plugin repro because this is a read-only review.

## Solution Assessment

Unclear for the final implementation. Bounded wildcard contracts in the linked PR are plausible, but the permanent plugin API direction and allowlist/security semantics need maintainer approval before this issue can close.
