## Summary

Keep this open. Current main and latest release v2026.6.6 still reject `api.registerTool()` when the manifest omits `contracts.tools`, and the active closing implementation at https://github.com/openclaw/openclaw/pull/78084 is still open, unmerged, and pending API/proof review.

## What This Changes

The issue requests restoring or clearly signaling dynamic plugin tool registration when manifests omit `contracts.tools` after the 2026.5.2 plugin contract change.

## Best Possible Solution

Land a maintainer-approved plugin tool contract fix, likely through the linked PR or a replacement, that keeps manifest-first discovery while supporting bounded dynamic namespaces and concrete allowlist filtering.

## Reproduction Assessment

Yes, at source level. Current main records a diagnostic and returns when `contracts.tools` is absent, and the adjacent loader test asserts that no tools are registered in that case; I did not run a live plugin repro in this read-only review.

## Solution Assessment

Unclear for the final implementation. Bounded wildcard ownership in https://github.com/openclaw/openclaw/pull/78084 is a plausible direction, but it still needs maintainer API approval and proof that explicit allowlists do not expose sibling dynamic tools.
