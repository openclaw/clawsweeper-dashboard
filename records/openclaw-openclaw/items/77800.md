## Summary

Keep open: current main still source-reproduces the missing-`contracts.tools` rejection path, the reporter’s dynamic-tool use case is not implemented, and the live linked implementation PR remains open with review blockers.

## What This Changes

The issue asks OpenClaw to address the 2026.5.x change where `api.registerTool()` registrations are rejected when plugin manifests omit `contracts.tools`, especially for dynamic tool plugins.

## Best Possible Solution

Land a maintainer-approved dynamic tool ownership and diagnostic design, likely through https://github.com/openclaw/openclaw/pull/78084 after its API contract, allowlist filtering, docs, tests, and real behavior proof are acceptable.

## Reproduction Assessment

Yes. Current main source-reproduces the report: `registerTool` returns after recording the missing-`contracts.tools` diagnostic, and a focused loader test asserts no tool is registered in that case.

## Solution Assessment

Unclear. Bounded wildcard ownership is a plausible fix for dynamic tools, but the public manifest contract and the exact failure signal still need maintainer API judgment before this issue can close.
