## Summary

Keep open: current main and the latest release still reject `api.registerTool()` when `contracts.tools` is absent, and the issue is paired with an open closing PR that has not landed. The follow-up comment adds useful user-side confirmation and a broader startup-warning angle, but neither the dynamic-tool contract nor the diagnostic behavior is fixed yet.

## What This Changes

The issue asks OpenClaw to stop silently rejecting dynamic plugin tools registered with `api.registerTool()` without exact `contracts.tools` declarations, or to provide a migration path such as wildcard tool contracts.

## Best Possible Solution

Keep this issue open until a maintainer-approved plugin manifest/API contract and diagnostic path lands, likely through https://github.com/openclaw/openclaw/pull/78084 or a successor with docs, tests, and real behavior proof.

## Reproduction Assessment

Yes. Current main source-reproduces the report because `registerTool` returns after recording the missing-`contracts.tools` diagnostic, and a focused loader test asserts no tool is registered in that case.

## Solution Assessment

Unclear. Bounded wildcard ownership is a plausible direction for dynamic tools, but the public contract, allowlist filtering, and failure signal still need maintainer API judgment before this can be considered solved.
