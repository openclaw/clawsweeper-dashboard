## Summary

The underlying image-replay failure remains valid on current main, but this PR is not merge-ready: it broadens a base64-only shared contract by treating every malformed image string as latin1, targets a removed helper path, conflicts with current main, and supplies only synthetic test proof.

## What This Changes

The PR canonicalizes valid ACP inline image base64 and re-encodes every non-base64 string from latin1 bytes, with focused regression tests.

## Best Possible Solution

Keep ImageContent data canonical base64, trace and encode the actual raw-byte producer or narrowly migrate a proven shipped representation, and preserve explicit omission or rejection for unrelated corrupted images.

## Reproduction Assessment

Yes at source level: current main forwards inline image data unchanged into provider-bound attachments, so non-ASCII data reaches a base64 field. The exact real producer of raw latin1 data remains unproven.

## Solution Assessment

No. The shared contract is base64-only, so converting every invalid string from latin1 is broader and less safe than repairing the violating producer or a narrowly proven migration boundary.
