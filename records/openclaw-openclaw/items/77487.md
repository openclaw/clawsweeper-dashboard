## Summary

Keep open: this PR fixes the reported wrong allowlist ID path, but the one-line fallback swap also loses the existing explicit channel-disable guard for external plugins claiming built-in aliases.

## What This Changes

The PR changes plugin auto-enable channel claim resolution to prefer the owning plugin manifest ID over a built-in channel alias and adds a WeCom regression test.

## Best Possible Solution

Keep the plugin-ID preference, but make channel-configured candidates still honor `channels.<channelId>.enabled=false`, add regression coverage for that case, and add the required changelog entry.

## Reproduction Assessment

Yes. The underlying issue is source-reproducible on current main because built-in aliases win over manifest claims, and the PR finding is source-reproducible by following the changed candidate ID through `isPluginExplicitlyDisabled`.

## Solution Assessment

No. The one-line preference change is the right central direction, but it is not the best final fix until it preserves explicit channel-disable semantics and documents the user-facing fix.
