## Label Justifications

- `P1`: The issue breaks first-run external provider onboarding for multiple providers in the current beta and leaves users on the wrong default auth/model path.
- `impact:auth-provider`: The reported failure is provider auth-choice routing and default model/auth state after installing external provider plugins.

## Summary

Keep this issue open: current main still shows the reported onboarding path can retry provider discovery immediately after install without invalidating plugin metadata caches, and the open linked pull request is the active fix candidate rather than a close reason.

## What This Changes

First-time onboarding should continue into the selected external provider credential flow after installing a missing provider plugin instead of falling back to the default OpenAI model/auth path.

## Best Possible Solution

Land a narrow shared onboarding-install fix that invalidates plugin metadata lifecycle caches after a successful install, then prove the same run reaches the selected provider credential prompt and closes this issue after merge.

## Reproduction Assessment

Yes, with high confidence from source and the reporter's matrix: run fresh-profile interactive onboarding with an external provider auth choice, accept the provider plugin install, and observe that the selected credential prompt is skipped. I did not run the live PTY flow in this read-only review.

## Solution Assessment

Yes, the linked fix direction is the right boundary: invalidate the shared plugin metadata lifecycle after a successful onboarding install rather than adding provider-specific retries. The issue should remain open until that candidate is reviewed, merged, and ideally proven against the real onboarding path.
