## Summary

Keep open for review/merge: the patch appears correct and the updated PR body now includes non-test terminal proof, but merge remains compatibility-sensitive because `cron.add` and `cron.update` now fail closed for schedules that previously returned success.

## What This Changes

The PR adds service-layer validation to reject cron expressions with no future run time, maps that error to gateway `INVALID_REQUEST`, and adds cron service regression tests.

## Best Possible Solution

Land the service-boundary rejection only after maintainers accept the stricter `cron.add`/`cron.update` contract; otherwise switch to an explicit auto-disable and recorded schedule-error path.

## Reproduction Assessment

Yes. A high-confidence source path exists: create a cron job with `0 0 30 2 *`; current main maps Croner’s missing next run to `undefined` and persists the enabled job without a future run.

## Solution Assessment

Yes, pending maintainer compatibility approval. The shared service boundary covers gateway RPC and in-process callers; the safer alternative, if fail-closed behavior is rejected, is an explicit auto-disable plus schedule-error record.
