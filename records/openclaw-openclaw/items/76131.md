## Summary

Keep open. The PR is a plausible startup-performance fix, but the latest patch still puts a provider-config fallback ahead of provider-owned dynamic resolution for provider-only configs, does not actually shortcut exact configured rows before target-provider hooks, and is missing the required changelog entry.

## What This Changes

The PR adds a skipPiDiscovery configured-model shortcut, defers PDF auto-model setup behind media/tool policy, registers PDF in the media catalog/group, adds regression tests, and loosens two script signal declaration types.

## Best Possible Solution

Land a narrower fast path that only short-circuits exact configured model rows before expensive provider hooks, leaves provider-only or dynamically discoverable refs on provider-owned dynamic resolution, keeps the PDF policy changes if validation stays green, and records the user-facing performance change in the changelog.

## Reproduction Assessment

Yes. Static reproduction is enough: with skipPiDiscovery and a configured provider that lacks an exact model row, the PR returns the generic fallback before prepareProviderDynamicModel; with an exact configured row, the new branch is skipped because explicitModel is already resolved.

## Solution Assessment

No. The safer solution is an exact configured-model shortcut before target-provider runtime normalization, plus preserving dynamic provider resolution for provider-only configs and adding the changelog line.
