## Summary

Keep open: the current PR head still places the configured-model shortcut both too broadly and too late in the async model resolver, so it can skip provider dynamic hooks for provider-only configs while still failing to avoid provider runtime normalization for exact configured rows; it also lacks the required changelog entry.

## What This Changes

The PR adds a skipPiDiscovery configured-model fast path, defers PDF auto model resolution behind media tool policy, registers PDF in media tool policy/catalog coverage, and loosens two script signal declaration types.

## Best Possible Solution

Land a narrower fast path that returns only exact configured provider model rows before runtime normalization, keeps provider-only configs on the dynamic provider path, preserves the PDF/tool-policy deferral, and adds focused tests plus a single-line changelog entry.

## Reproduction Assessment

Yes. Static review gives a high-confidence path: call resolveModelAsync with skipPiDiscovery and a provider config that lacks an exact models[] row, and the PR branch can return generic fallback metadata before prepareProviderDynamicModel; an exact configured row still enters runtime normalization before the shortcut.

## Solution Assessment

No. The safer solution is to detect exact configured rows before resolveExplicitModelWithRegistry performs provider hook normalization, while leaving provider-level dynamic-discovery configs to the existing prepare/resolve dynamic flow.
