## Summary

Keep open. The PR is directionally useful, but the current branch still has merge-blocking regressions in model dynamic-hook routing and session-list freshness/usage semantics, plus the required changelog entry is missing.

## What This Changes

The PR adds a skipPiDiscovery fast path for configured provider/model refs, defers PDF auto-model setup behind tool policy, adds PDF/media coverage, and caches lightweight sessions.list results.

## Best Possible Solution

Land a narrower PR that keeps the exact configured-model fast path, preserves provider-owned dynamic resolution for provider-only refs, centralizes session-list cache invalidation, retains default usage fallback behavior, and records the user-facing performance change in the changelog.

## Reproduction Assessment

Yes. Static reproduction is enough: use skipPiDiscovery with a configured provider but no exact configured model row, and a cached sessions.list followed by agent send/create; current main tests also show sessions.list with {} recovering transcript usage.

## Solution Assessment

No. The current patch solves the intended latency problem too broadly; the safer solution is an exact configured-model shortcut plus shared cache invalidation and an explicit lightweight-list mode if default usage fallback is meant to change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep provider-only configs on the dynamic path:** `src/agents/pi-embedded-runner/model.ts:1079-1088`
  - body: This early return runs after exact configured-model resolution misses, but resolveConfiguredFallbackModel can synthesize a model for any configured provider. With skipPiDiscovery, provider-only OpenRouter/Ollama/LM Studio-style configs can return before provider dynamic hooks run, dropping provider-owned discovery, caches, and capability metadata.
  - confidence: 0.92
- **[P2] Invalidate cached session lists from agent events:** `src/gateway/server-methods/sessions.ts:328`
  - body: The new cache is cleared only by the local sessions.ts emitter, but agent sends and creates broadcast sessions.changed through a separate helper in server-methods/agent.ts. After a cached sessions.list, those agent updates can leave callers seeing stale rows until the 60-second TTL expires.
  - confidence: 0.88
- **[P2] Preserve default transcript usage fallback:** `src/gateway/session-utils.ts:1925-1926`
  - body: Setting transcriptUsageMaxBytes to 0 unless title or last-message fields are requested makes default sessions.list rows stop recovering stale totalTokens and estimatedCostUsd from transcripts. Current coverage exercises that fallback with opts: {}, so this should remain default behavior or move behind an explicit lightweight option.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `CHANGELOG.md:7`
  - body: This PR changes user-visible first-turn model, media-tool, and session-list performance behavior, so repository policy requires a single-line CHANGELOG.md entry under Unreleased before landing.
  - confidence: 0.9
