## Summary

Keep open: the PR targets a real first-turn performance problem, but the model fast path is still too broad and is placed too late to avoid the runtime hooks it intends to skip.

## What This Changes

The PR adds a configured-model skipPiDiscovery fast path, defers PDF auto model resolution behind media policy, adds PDF policy/catalog coverage, and relaxes two script declaration types.

## Best Possible Solution

Land a narrower fast path that returns only exact configured provider model rows before runtime normalization, leaves provider-only configs on the dynamic provider path, preserves the PDF/tool-policy deferral, and adds focused tests plus one changelog line.

## Reproduction Assessment

Yes. Static review gives a high-confidence path: call resolveModelAsync with skipPiDiscovery and a configured provider that lacks an exact models[] row, and this PR can return synthetic fallback metadata before prepareProviderDynamicModel; for an exact row, the current placement still enters normalizeResolvedModel hooks first.

## Solution Assessment

No. The safer solution is to detect exact configured model rows before resolveExplicitModelWithRegistry/normalizeResolvedModel and keep provider-level dynamic-discovery configs on the existing prepare/resolve dynamic flow.
