The branch adds `filePath` and `path` fallbacks to Telegram `sendMessage` media parsing and tests those aliases while preserving existing `mediaUrl` precedence.

## Best Possible Solution

Land the focused alias mapping with concise Telegram docs for `filePath`/`path` and the existing local media-root boundary, require redacted real Telegram proof, then rerun the targeted Telegram action tests and relevant gates.

Yes. Source inspection on current main shows `sendMessage` maps only `mediaUrl` and `media`, so `filePath`-only or `path`-only inputs leave `mediaUrl` unset and hit `content required.` before upload.

## Solution Assessment

Yes, pending docs and proof. Reusing the existing `mediaUrl` pipeline is the narrowest maintainable solution because Telegram already owns media loading, document forcing, size limits, and local-root checks there.
