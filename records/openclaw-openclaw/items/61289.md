## Summary

Keep this PR open. Current main still has the direct Telegram action gap, and the PR’s final diff is a narrow, mostly correct alignment with the existing media pipeline, but it still needs small public docs and changelog follow-up before merge.

## What This Changes

The PR adds Telegram `sendMessage` `filePath` and `path` media fallbacks in the action runtime and tests that those aliases reach the existing `mediaUrl`/`asDocument` send options.

## Best Possible Solution

Land the focused Telegram alias mapping, preserve existing `mediaUrl` then `media` precedence, keep the focused alias/asDocument tests, and add a small public Telegram docs update plus changelog entry without introducing a broader provider-neutral attachment redesign.

## Reproduction Assessment

Yes. On current main, a focused unit call to `handleTelegramAction` with `action: "sendMessage"` and only `filePath` or only `path` has no media recognized and falls into the no-content path; the PR maps those keys into the existing media send options.

## Solution Assessment

Yes, with docs and changelog follow-up. Routing the aliases into the existing `mediaUrl` pipeline is the narrowest maintainable fix because Telegram already has media loading, document forcing, and local-root guards.
