Keep open. The focused alias mapping is still useful on current main and the final diff avoids the earlier precedence and PR-doc noise, but this external Telegram PR is not merge-ready because it lacks real Telegram behavior proof and still needs a small public docs update.

## What This Changes

The PR adds `filePath` and `path` aliases to Telegram `sendMessage` media resolution and tests that they forward through the existing `mediaUrl` send path, including `asDocument` handling.

## Best Possible Solution

Land the focused alias mapping with a small Telegram docs update and live Telegram proof, while leaving broader attachment abstractions outside this PR and letting maintainers handle changelog during landing.

## Reproduction Assessment

Yes for the source-level gap: current main reads only `mediaUrl` and `media`, so a `filePath`-only or `path`-only Telegram action cannot reach the media pipeline. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, with docs and real behavior proof. Mapping the aliases into the existing `mediaUrl` pipeline is the narrowest maintainable solution because Telegram already owns media loading, local-root checks, and document forcing there.
