## Summary

Keep open: current main and the latest release still keep the synchronous `banner-config-lite` config import on the CLI help banner path, while the target fix PR was closed unmerged and the remaining open fork PR is the same unproven head with a prior cache-regression review finding.

## What This Changes

The issue reports that `openclaw tui -h` hangs indefinitely at 100% CPU because CLI help banner generation reaches a heavy synchronous config read.

## Best Possible Solution

Land or replace the fix with a narrow CLI banner/help change that removes heavy config imports from help generation, preserves `cli.banner.taglineMode` for normal banners, and includes real terminal proof that help exits.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still evaluates the banner/config helper from the Commander help path and still synchronously calls `createConfigIO().loadConfig()`. I did not run the potentially hanging command in this read-only review.

## Solution Assessment

No, current main is not the best final solution because the `mode: "default"` help hook avoids one config read call but leaves the static helper import in the help module graph. The safest solution is a narrow runtime-boundary or help-banner refactor that avoids the heavy config graph while preserving normal banner settings.
