## Summary

Keep open: this PR targets the right Tool Search source-level bug, but the latest patch still strips legitimate flattened target `id`/`name` fields when a different selector key identifies the wrapper tool, and real localModelLean runtime proof is still missing.

## What This Changes

The branch changes Tool Search `readCallArgs` to pass flattened top-level `tool_call` fields as target input when `args` or `input` are absent, filters `id`/`toolId`/`name`, and adds focused regression coverage.

## Best Possible Solution

Land one narrow Tool Search parser fix that preserves explicit `args`/`input`, treats only the actually consumed selector key as wrapper metadata, covers selector-collision/nullish-wrapper cases, and includes redacted localModelLean runtime proof.

## Reproduction Assessment

Yes at source level: current main forwards only nested `args` or `input`, so a flattened payload like `{ id, command }` dispatches `{}`. I did not run the reported Telegram/vLLM session in this read-only review.

## Solution Assessment

No, not yet: `readCallArgs` is the right layer, but the patch should preserve target fields when a different selector key is used and needs real behavior proof before merge.
