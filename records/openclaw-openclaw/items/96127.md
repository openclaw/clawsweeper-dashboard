## Summary

Keep this PR open: it targets a real current Tool Search bug, but the fallback is too narrow because it only removes `id` before forwarding flattened input, and the external PR still has test-only proof rather than an after-fix real local-model run.

## What This Changes

The branch changes Tool Search `tool_call` parsing to forward top-level target parameters when `args` or `input` are not nested, and adds a regression test.

## Best Possible Solution

Normalize flattened calls by removing all wrapper selector/input keys before dispatch, keep nested `args`/`input` precedence, add alias/null regression coverage, and require redacted real local-model proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main discards flattened top-level target fields when `args` and `input` are absent, matching the linked issue logs. I did not run a live local-model or Telegram scenario in this read-only review.

## Solution Assessment

No. Runtime normalization is the right layer for this bug, but the implementation should strip every wrapper key that `readId` accepts, plus wrapper input keys, before forwarding flattened target parameters.
