## Summary

Keep open: current main still has separate non-streaming block-media and final-reply delivery paths, while per-media final suppression is only wired for block streaming. The related prior duplicate reports are closed but not reliable terminal fixes for this Telegram non-streaming case, and the open draft PR at https://github.com/openclaw/openclaw/pull/78355 targets missing TTS delivery rather than duplicate MEDIA delivery.

## What This Changes

The issue requests that non-streaming Telegram delivery send each generated image or audio attachment from a MEDIA directive exactly once.

## Best Possible Solution

Implement a turn-scoped delivered-media dedupe path that covers non-streaming direct block replies and final payload construction, with regression coverage for multiple MEDIA lines and Telegram delivery semantics.
