## Summary

## What This Changes

The issue reports Telegram direct messages stalling during Codex embedded runs with OpenAI Codex OAuth while an explicit CLI session using the same agent/model succeeds.

## Best Possible Solution

Keep the current Codex harness behavior on main and ship it; affected users should update to a build containing the May 18 Codex fixes or a later release.

Yes for the affected release path: current source shows Codex direct/source chats default to message-tool delivery, and the prior forced code-mode-only behavior broke that dynamic tool path. I did not live-run the reporter's Telegram OAuth account in this read-only review.

## Solution Assessment

Yes. The current fix targets the Codex harness default and tool bridge instead of special-casing Telegram: code mode stays enabled, code-mode-only is off by default, and dynamic tools remain registered when code-mode-only is explicitly enabled.
