## Summary

Keep open: current main auto-inherits or preserves thread context in specific paths, but it does not provide the requested per-channel hard gate for outbound top-level sends; this is a product/config-sensitive feature rather than cleanup-close material.

## What This Changes

Request per-channel Slack/Discord configuration that rejects outbound message sends to configured channels when no valid threadId is present.

## Best Possible Solution

Define an opt-in per-channel outbound send policy at the approved channel/plugin boundary, keep existing defaults unchanged, and add Slack/Discord docs plus regression coverage for allowed threaded sends and rejected top-level sends.

## Reproduction Assessment

Not applicable as a feature request: the current source and tests show threadId is optional and top-level sends remain allowed, but there is no broken existing requireThread contract to reproduce.

## Solution Assessment

Unclear until maintainers choose the API boundary: a shared opt-in send policy may be the right solution, but adding a new Slack/Discord config key needs product and compatibility direction first.
