## Summary

Keep open: current main does not implement this API, but the PR still needs maintainer privacy/API approval, stronger real behavior proof, and a hosted-LID lookup fix before merge.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` method and `PnLidEntryResult` type, wires it into the inbound monitor, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

If maintainers approve the API, make it mirror existing WhatsApp LID resolver semantics, document or gate the privacy-sensitive surface, export the type consistently, and require redacted runtime proof of the lookup path.

## Reproduction Assessment

Yes for the blocking patch defects by source inspection: the PR calls Baileys `getPNForLID` for hosted LIDs even though upstream only accepts `@lid`, and it changes resolver ordering relative to current WhatsApp normalization. I did not run tests because this is a read-only review.

## Solution Assessment

No. The requested API may be useful, but the current patch is not the best merge shape until hosted-LID behavior, authDir-first semantics, privacy/API approval, and live lookup proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle hosted LIDs before calling Baileys:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: `lookupPnLidEntry` classifies `@hosted.lid` as a LID, then passes it to Baileys `getPNForLID`. In Baileys 7.0.0-rc.9 that method returns `null` unless `isLidUser(lid)` is true, and upstream `isLidUser` only matches `@lid`, so hosted LIDs without a local reverse file cannot use the dynamic lookup path this API is meant to provide.
  - confidence: 0.86
- **[P2] Check authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:932-934`
  - body: Current WhatsApp normalization calls `jidToE164(..., { authDir })` before `getPNForLID`, and its regression test asserts an authDir reverse mapping avoids the Baileys call. This new path asks Baileys first, so an existing local reverse mapping can resolve differently from inbound sender normalization; reuse `resolveJidToE164` or mirror its ordering.
  - confidence: 0.82
