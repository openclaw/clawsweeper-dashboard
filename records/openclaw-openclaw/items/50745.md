## Summary

Keep open: current main does not provide this WhatsApp lookup API, and this PR is still the only tracked implementation, but it needs maintainer API/privacy review, stronger live lookup proof, and a source fix for local-first LID mapping behavior before merge.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, result type, monitor implementation, listener mocks, and a changelog entry.

## Best Possible Solution

Approve a narrow WhatsApp lookup contract only if maintainers are comfortable exposing this identity mapping, then implement it by reusing or mirroring the existing local-first resolver semantics with hosted-LID coverage and redacted live lookup proof.

## Reproduction Assessment

Yes for the review finding: PR head source calls Baileys before the authDir mapping fallback, while current main source and tests establish local-first LID resolution. The live WhatsApp lookup behavior itself has not been reproduced by the supplied proof.

## Solution Assessment

No, not as proposed. The maintainable path is to align the new API with the existing resolver ordering, define the privacy/API contract, and require real lookup proof before merge.
