## Summary

Keep open: current main still lacks the requested WhatsApp listener API, but this PR needs maintainer approval for the privacy-sensitive lookup surface, stronger real behavior proof, and fixes to match existing LID resolution semantics before merge.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, exports its result type, wires it into the inbound monitor, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

If maintainers approve the API, implement it through the existing WhatsApp LID resolver semantics, document or gate the privacy-sensitive surface, add focused regression coverage, and require redacted live lookup proof.

## Reproduction Assessment

Yes for the blocking patch defects by source inspection: PR head calls Baileys before authDir mapping and passes hosted LIDs to `getPNForLID`, while upstream skips non-`@lid` inputs. I did not run tests or a live WhatsApp session because this was a read-only review.

## Solution Assessment

No. The requested API may be useful, but the current patch is not the best merge shape until it mirrors existing resolver ordering, makes hosted-LID behavior explicit, gets privacy/API approval, and includes redacted live lookup proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle hosted LIDs before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: `lookupPnLidEntry` classifies `@hosted.lid` as a LID and sends it to Baileys `getPNForLID`, but upstream `getPNsForLIDs` filters with `isLidUser`, which only accepts `@lid`. Hosted LIDs without a local reverse file therefore cannot use the dynamic lookup path this new API is meant to provide; route hosted LIDs through the local resolver or make the contract explicit before calling Baileys.
  - confidence: 0.85
- **[P2] Check authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:932-934`
  - body: Existing WhatsApp normalization resolves `jidToE164(..., { authDir })` before `getPNForLID`, and the current regression test asserts an authDir reverse mapping avoids the Baileys call. This new method asks Baileys first, so the same LID can resolve differently or wait on a store call despite a local mapping; reuse `resolveJidToE164` or mirror its order.
  - confidence: 0.82
