## Summary

Keep this PR open. The requested WhatsApp LID lookup API is not on current main, but the proposed patch is not ready to merge because it bypasses current authDir-first LID resolution, clobbers unrelated changelog entries, and exposes a plugin-visible phone-number lookup that needs maintainer API/privacy review.

## What This Changes

The PR adds a WhatsApp `lookupPnLidEntry` listener API/type, implements LID and phone lookup in the inbound monitor, updates mocks and monitor tests, and edits the unreleased changelog.

## Best Possible Solution

If maintainers approve the API, rebase the branch, add the contract in the current `extensions/whatsapp/src/inbound/types.ts` surface, implement lookup through the shared authDir-first `jidToE164`/`resolveJidToE164` semantics, preserve existing changelog entries, add cold-cache/authDir fallback coverage, and document or gate the sensitive lookup.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with a `lid-mapping-*_reverse.json` authDir mapping and missing or empty `lidLookup`, the PR returns `null` in the LID branch while current `jidToE164` tests show the same LID should resolve.

## Solution Assessment

No. The API may be useful, but this implementation is not the best merge shape until it reuses the existing resolver semantics, updates the current listener type location, preserves the changelog, and gets explicit maintainer approval for exposing phone-number lookup to plugins.
