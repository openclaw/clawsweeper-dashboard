## Summary

Keep open: this PR adds a useful WhatsApp runtime API, but current main does not already implement it and the proposed patch still has blocking merge issues around LID fallback behavior, changelog preservation, stale listener-contract placement, and privacy/API approval.

## What This Changes

The branch adds a WhatsApp `lookupPnLidEntry` active listener API/type, implements LID and phone lookup in the inbound monitor, updates mocks/tests, and edits the unreleased changelog.

## Best Possible Solution

If maintainers approve the API, rebase it onto current main, define the contract in the current inbound listener type surface, implement lookup through shared authDir-first resolver semantics, preserve changelog entries, and add privacy gating or documentation for the sensitive identifier lookup.

## Reproduction Assessment

Yes for the blocking patch defect. A source-level reproduction is clear: with a LID input, an authDir `lid-mapping-*_reverse.json`, and unavailable or cold `lidLookup`, the PR returns `null` while current `resolveJidToE164` semantics can resolve the same LID.

## Solution Assessment

No. The API may be useful, but this patch is not the best merge shape until it reuses existing resolver semantics, preserves the changelog, rebases to the current listener contract, and gets maintainer approval for exposing phone-number lookup to plugins.
