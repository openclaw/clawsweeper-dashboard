## Summary

Keep open. Current main still lacks the proposed WhatsApp lookup API, but the branch has a correctness gap for phone inputs and the supplied proof does not show the new API returning a resolved result in a real setup; the earlier `@hosted.lid` review feedback appears addressed in the latest diff.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` method/result type, implements LID-to-phone lookup via Baileys with local authDir fallback, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

Keep the WhatsApp-owned API additive, reuse the existing bidirectional LID mapping helpers so every supported input returns all known mapping data, expose the result type through the supported public barrels, and require a real lookup invocation proof before merge.

## Reproduction Assessment

Yes for the PR finding by source inspection: the PR’s phone-input branches set `lid = ""`, while current main already reads forward `lid-mapping-{phoneDigits}.json` files. I did not run a live WhatsApp lookup reproduction in this read-only review.

## Solution Assessment

No. The proposed location is reasonable, but the implementation should share the existing PN/LID mapping behavior, settle the exported API compatibility shape, and show a real resolved lookup result before merge.
