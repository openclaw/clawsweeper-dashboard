## Summary

Keep this PR open: current main does not provide this WhatsApp lookup API, and the branch contains a plausible focused feature, but the latest diff still drops known PN-to-LID mappings for phone inputs and the supplied proof does not exercise the new behavior.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, exports its result type, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

Keep the WhatsApp-owned API additive, but make it share the existing PN/LID mapping behavior so both LID and phone inputs return all known mapping data, then add direct coverage and real behavior proof.

Yes for the PR finding by source inspection: the phone-JID and E.164 branches set `lid = ""` even though current main can resolve PN-to-LID from authDir forward mapping files. I did not establish a live WhatsApp reproduction path.

## Solution Assessment

No. The proposed location is reasonable, but the implementation should reuse or extract the existing PN/LID mapping helpers so the advertised phone inputs do not lose known LID data.
