## Summary

Keep open: current main does not implement this new WhatsApp listener API, and the PR still needs maintainer API/privacy review plus stronger real lookup proof before it is mergeable.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API and implementation for LID/phone inputs, updates listener mocks and exports, and adds a changelog entry.

## Best Possible Solution

Expose this lookup only if maintainers approve the privacy surface, implemented through existing resolver semantics with focused tests and redacted real lookup proof.

## Reproduction Assessment

Yes for the review findings: PR head source shows the new method calls Baileys before local authDir mapping, while current main resolver and tests establish local-first behavior. The contributor has not shown a real or redacted runtime lookup result for the new API.

## Solution Assessment

No. The useful direction should reuse the existing resolver ordering, define hosted-LID behavior and privacy expectations, and land only with maintainer approval plus real lookup proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prefer authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The new method calls `lidLookup.getPNForLID` before checking the local authDir reverse mapping. Current WhatsApp resolution first uses `jidToE164(..., { authDir })`, and tests assert local mappings avoid Baileys, so this can resolve differently or consult Baileys when local state already has the answer.
  - confidence: 0.86
- **[P2] Handle hosted LIDs without calling getPNForLID:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: `@hosted.lid` is classified as a LID, but the method still sends it to `getPNForLID`. Baileys v7.0.0-rc11 skips JIDs that are not `@lid`, so hosted LID support should route directly through local mappings or the public contract should be narrowed.
  - confidence: 0.82
