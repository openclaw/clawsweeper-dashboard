## Summary

Keep open: current main does not implement this API, and the PR still needs changes because the lookup order diverges from existing WhatsApp resolver semantics while exposing a privacy-sensitive LID-to-phone lookup surface.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API/type, implements it in the inbound monitor with Baileys plus authDir fallback, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

If maintainers want this API, implement it through existing resolver semantics with local mapping first, explicit hosted-LID behavior, focused tests, privacy docs or gating, and redacted live lookup proof.

## Reproduction Assessment

Yes. Source inspection shows PR head calls Baileys before the local `authDir` mapping, while current resolver code and tests require local mapping to win; Baileys source also shows hosted LIDs are not accepted by `getPNForLID`.

## Solution Assessment

No. The API may be useful, but this patch should mirror `resolveJidToE164` ordering, make hosted-LID behavior explicit, and get privacy/API approval plus real lookup proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prefer authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: Existing WhatsApp normalization resolves `jidToE164(..., { authDir })` before calling `getPNForLID`, and the current inbound test asserts a local reverse mapping avoids Baileys. This method calls Baileys first, so the same LID can resolve differently or incur a store lookup despite a local mapping; reuse `resolveJidToE164` or mirror its order.
  - confidence: 0.86
- **[P2] Route hosted LIDs without relying on getPNForLID:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The new detector accepts `@hosted.lid`, but Baileys `getPNsForLIDs` skips anything that is not `isLidUser`, and `isLidUser` only matches `@lid`. Hosted LID inputs without a local reverse mapping therefore go through a dynamic lookup path that cannot resolve them; check local mapping first and make hosted-LID support explicit.
  - confidence: 0.84
