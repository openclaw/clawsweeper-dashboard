## Summary

Keep open: current main does not implement this WhatsApp LID lookup API, and the PR still needs maintainer approval plus fixes for resolver parity and changelog preservation before merge.

## What This Changes

This PR adds a WhatsApp `lookupPnLidEntry` active listener API/type, implements LID and phone lookup in the inbound monitor, updates mocks/tests, and edits the unreleased changelog.

## Best Possible Solution

If maintainers approve the API, add it in the current `ActiveWebListener` type surface, reuse `resolveJidToE164` authDir-first semantics, preserve changelog entries, and document or gate the privacy-sensitive lookup.

## Reproduction Assessment

Yes for the blocking patch defect. Source inspection shows a LID input with an authDir `lid-mapping-*_reverse.json` and unavailable or cold `lidLookup` would return `null` in the PR while current inbound normalization resolves the same LID.

## Solution Assessment

No. The API may be useful, but this patch is not the best merge shape until it reuses existing resolver semantics, preserves the changelog, rebases to the current listener contract, and gets maintainer approval for exposing phone-number lookup to plugins.
