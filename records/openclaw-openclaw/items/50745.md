## Summary

Keep this PR open: current main still lacks the proposed WhatsApp lookup API, but the branch adds a privacy-sensitive phone-number resolution surface with source-reproducible correctness gaps and insufficient real lookup proof for merge.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, implementation, mock updates, and a changelog entry for resolving LID or phone inputs to a phone-number result.

## Best Possible Solution

If maintainers want this API, implement it by reusing the existing resolver semantics, preserving authDir-first behavior, making the hosted-LID contract explicit, and documenting or gating the privacy-sensitive surface with redacted live proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows the Baileys-first lookup order, while current main code and tests show authDir-first resolver semantics. The contributor has not provided a live or redacted runtime lookup reproduction for the new API itself.

## Solution Assessment

No. The API may be useful, but this patch should mirror existing resolver ordering, avoid or explicitly contract hosted-LID behavior, and receive privacy/API review plus real lookup proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prefer authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The new method calls `lidLookup.getPNForLID` before `jidToE164(..., { authDir })`. Current inbound resolution first uses local reverse mappings and the existing monitor test asserts that path avoids Baileys, so this API can resolve differently or unnecessarily consult Baileys when a local mapping exists.
  - confidence: 0.86
- **[P2] Avoid sending hosted LIDs to the plain Baileys LID lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The method classifies `@hosted.lid` as a LID but passes it to `getPNForLID`; Baileys 7.0.0-rc10 filters values that are not `@lid`. Hosted LID inputs therefore only work through local mapping, so route them directly to the authDir path or narrow/document the contract.
  - confidence: 0.84
