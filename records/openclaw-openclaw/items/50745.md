## Summary

Keep this PR open: current main does not have the proposed WhatsApp lookup API, but the current patch still has resolver-order and hosted-LID issues, needs maintainer approval for the privacy-sensitive API surface, and does not yet show a real lookup result.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, exports its result type, implements it in the inbound monitor, updates listener mocks, and adds a changelog entry.

## Best Possible Solution

If maintainers want this API, implement it through the existing resolver semantics with local mapping first, explicit hosted-LID behavior, focused tests, privacy docs or gating, and redacted live lookup proof.

## Reproduction Assessment

Yes. Source inspection shows PR head calls Baileys before the local `authDir` mapping, while current resolver code and tests require local mapping to win; upstream Baileys source also shows hosted LIDs are not accepted by `getPNForLID`.

## Solution Assessment

No. The API may be useful, but the current patch is not the best merge shape until it mirrors existing resolver ordering, makes hosted-LID behavior explicit, gets privacy/API approval, and includes real lookup proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prefer authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: Existing WhatsApp normalization resolves `jidToE164(..., { authDir })` before calling `getPNForLID`, and the current inbound test asserts a local reverse mapping avoids Baileys. This method calls Baileys first, so the same LID can resolve differently or incur a store lookup despite a local mapping; reuse `resolveJidToE164` or mirror its order.
  - confidence: 0.86
- **[P2] Route hosted LIDs without relying on getPNForLID:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The new detector accepts `@hosted.lid`, but Baileys `getPNsForLIDs` skips anything that is not `isLidUser`, and `isLidUser` only matches `@lid`. Hosted LID inputs without a local reverse mapping therefore go through a dynamic lookup path that cannot resolve them; check local mapping first and make hosted-LID support explicit.
  - confidence: 0.84
