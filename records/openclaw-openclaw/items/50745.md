## Summary

Keep this PR open for human review: current main does not already implement the new WhatsApp lookup API, but PR head still needs API/privacy approval, stronger real behavior proof, and fixes to align with existing LID resolver semantics.

## What This Changes

The PR adds a WhatsApp `ActiveWebListener.lookupPnLidEntry` API, implementation, mock updates, and a changelog entry for resolving LID or phone inputs to phone-number metadata.

## Best Possible Solution

If maintainers want this API, implement it by reusing the existing local-first resolver semantics, making hosted-LID behavior explicit, and documenting or gating the privacy-sensitive surface with redacted live proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows the Baileys-first lookup order and hosted-LID call path, while current main code and tests show authDir-first resolver semantics. The contributor has not provided a live or redacted runtime lookup reproduction for the new API itself.

## Solution Assessment

No. The API may be useful, but this patch should mirror existing resolver ordering, avoid or explicitly contract hosted-LID behavior, and receive privacy/API review plus real lookup proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prefer local authDir mappings before Baileys lookup:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The new method calls `lidLookup.getPNForLID` before `jidToE164(..., { authDir })`. Current inbound resolution first uses local reverse mappings and existing tests cover that local mapping path, so this API can unnecessarily consult Baileys or resolve differently when a local mapping already exists.
  - confidence: 0.86
- **[P2] Route hosted LIDs away from getPNForLID:** `extensions/whatsapp/src/inbound/monitor.ts:934`
  - body: The method classifies `@hosted.lid` as a LID but passes it to `getPNForLID`; Baileys v7.0.0-rc11 `getPNsForLIDs` skips values that are not `isLidUser`, and `isLidUser` only matches `@lid`. Hosted LID inputs therefore only work through local mapping, so route them directly to the authDir path or narrow the documented contract.
  - confidence: 0.84
