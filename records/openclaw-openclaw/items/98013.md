## Summary

Keep open: the updated branch now routes classified `server_error` into the retry path, but it still breaks the promised conservative fallback for harnesses that do not emit explicit current-attempt metadata.

## What This Changes

The PR adds optional `currentAttemptReplayMetadata` to embedded attempt results, derives it during normalization, uses it for silent provider-error retry gating, includes `server_error` in retry eligibility, and updates retry tests.

## Best Possible Solution

Land one canonical fix that emits or preserves explicit current-attempt replay metadata at the right producer boundary, keeps classified `server_error` coverage, preserves absent-field fallback, and includes real gateway or provider proof.

## Reproduction Assessment

Yes at source level: current main blocks retry on cumulative replay metadata, and PR head shows the new fallback can be bypassed because normalization always creates `currentAttemptReplayMetadata`. I did not run tests or a live OpenRouter 5xx reproduction because this review is read-only.

## Solution Assessment

No as submitted: separating current-attempt metadata and routing `server_error` are the right direction, but computing the new field unconditionally in normalization is the wrong compatibility boundary. The safer fix is to have attempt producers explicitly emit current-attempt metadata or preserve absence so legacy fallback remains conservative.
