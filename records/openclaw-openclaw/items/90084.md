- `merge-risk: 🚨 availability`: A version-manager Node path can disappear or stop resolving after upgrades or without shell init, causing the managed Gateway service to fail.

## Summary

Keep open but not merge-ready: this reverses the current daemon policy that prefers a supported system Node for supervised services, while current tests, doctor docs, and service audit still treat version-manager service runtimes as upgrade-fragile.

## What This Changes

The branch changes daemon Node resolution so a supported current Node executable, including mise/nvm/fnm-style paths, wins over a supported system Node when generating service commands.

## Best Possible Solution

Keep the supported-system-Node default unless maintainers approve an explicit opt-in runtime path flow with matching tests, docs, and live Linux systemd proof.

## Reproduction Assessment

Yes for the PR regression from source: current main and tests expect version-manager execPath to fall back to supported system Node, while the patch returns the version-manager path. I did not run tests or live systemd proof because this review is read-only.

## Solution Assessment

No: unconditional active-node preference conflicts with the current daemon runtime policy, service audit, and doctor documentation. The safer solution is a maintainer-approved explicit runtime pin or opt-in path, with upgrade and live systemd proof.
