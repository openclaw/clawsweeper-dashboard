## Label Justifications

- `P2`: This is a bounded daemon install/runtime behavior change tied to a real user report, but it is not an emergency or release-wide outage.
- `merge-risk: 🚨 compatibility`: The diff changes a shipped default for generated service Node paths when supported system Node and active version-manager Node both exist.
- `merge-risk: 🚨 availability`: Version-manager Node paths can disappear or stop resolving after upgrades or without shell initialization, causing a managed Gateway service to fail.
- `merge-risk: 🚨 security-boundary`: The diff moves the long-running service executable from the stable system-Node preference to a mutable user-managed runtime path without explicit opt-in.

## Summary

Keep open and do not merge as-is: the patch flips a shipped daemon runtime default to version-manager Node paths while current main, v2026.6.1 tests/docs/audit logic still prefer supported system Node for managed services, and the supplied proof is unit-test output rather than a real Ubuntu mise plus systemd run.

## What This Changes

The PR changes `resolvePreferredNodePath` and its tests so a supported active mise/nvm/fnm Node wins over a supported system Node for daemon service commands.

## Best Possible Solution

Keep the supported-system-Node default unless maintainers approve an explicit validated runtime pin or a deliberate default-policy change with aligned docs, audit/tests, and live upgrade proof.

## Reproduction Assessment

Yes, from source for the selector behavior: with a supported version-manager `execPath` and supported system Node, current main and v2026.6.1 choose system Node while this PR chooses the version-manager path. I did not establish a live Ubuntu mise plus systemd before/after run.

## Solution Assessment

No: unconditional active-node preference is not the best fix because it conflicts with current service audit, docs, tests, and released behavior. The safer maintainable path is explicit validated runtime pinning or a clear maintainer-approved default-policy change with upgrade proof.
