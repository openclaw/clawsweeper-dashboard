## Summary

Keep open: current main deliberately prefers a supported system Node for daemon installs, and this PR reverses that default without updating the policy, tests, docs, or real Linux mise/systemd proof.

## What This Changes

The PR changes daemon Node resolution so a supported current Node executable is returned even when it comes from a version manager instead of preferring a supported system Node.

## Best Possible Solution

Keep the system-Node preference as the default for supervised services unless maintainers approve an explicit opt-in path with matching tests, docs, and Linux mise/systemd proof.

## Reproduction Assessment

Yes for the PR regression from source: current tests and source expect version-manager execPath to fall back to supported system Node, while this patch returns the version-manager path. I did not run tests because this review is read-only.

## Solution Assessment

No: unconditional active-node preference is not the best fix against current policy because it conflicts with daemon tests, doctor docs, and service-audit behavior. A safer path is default compatibility plus a maintainer-approved explicit opt-in if that behavior is desired.
