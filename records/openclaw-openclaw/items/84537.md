- `impact:session-state`: The report concerns valid persisted cron chat sessions disappearing from the UI selector even though session state still exists in the backend.

## Summary

Keep open: current main and the shipped 2026.5.18 Control UI still apply a 120-minute session-list cutoff on chat refresh paths, while the backend honors that cutoff by filtering out older session rows. That matches the reported disappearance of older cron-backed custom sessions and is a narrow UI/session-discovery bug rather than a stale or duplicate cleanup item.

## What This Changes

The issue asks for cron-backed custom chat sessions to remain discoverable in the Control UI chat/session selector after upgrade instead of being hidden by the activeMinutes:120 filter.

## Best Possible Solution

Keep the issue open and fix the Control UI chat session refresh path so persistent cron/custom sessions remain selectable while preserving the existing bounded list behavior and backend filtering semantics.

## Reproduction Assessment

Yes, from source inspection: create or have a persistent cron/custom session whose updatedAt is older than 120 minutes, then trigger a Control UI chat refresh that calls sessions.list with activeMinutes:120. I did not run a live UI reproduction because this review is read-only, but the current source path is direct.

## Solution Assessment

Yes, the maintainable fix is to change the Control UI chat/session selector refresh behavior rather than backend session storage. The safest shape is a bounded chat selector request that does not exclude older persistent cron/custom sessions while retaining explicit archive and per-run filtering.
