## Summary

Keep open: current main and v2026.5.19 still send `activeMinutes: 120` from Control UI chat refresh paths, and the gateway filters older session rows when that parameter is present. The linked open PR is the right implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for older valid cron/custom chat sessions to remain discoverable in the Control UI instead of being hidden by the chat session refresh `activeMinutes: 120` filter.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84683 or an equivalent Control UI fix that omits `activeMinutes` only for chat session discovery while preserving bounded limits and existing filtering semantics.

## Reproduction Assessment

Yes. From source inspection, a persistent cron/custom session with `updatedAt` older than 120 minutes is excluded when a Control UI chat refresh sends `sessions.list` with `activeMinutes: 120`, because the gateway filters by that cutoff.

## Solution Assessment

Yes. The best fix boundary is the Control UI chat/session discovery request shape, not backend session storage or a new config option; the linked PR follows that direction by keeping the bounded selector request but removing the active-window cutoff.
