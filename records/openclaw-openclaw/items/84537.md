## Label Justifications

- `P2`: The original bug is user-visible Control UI session discovery breakage with a clear, limited UI/session-listing blast radius.
- `impact:session-state`: The report concerns valid persisted sessions disappearing from the Control UI selector even though backend session state still exists.

## Summary

Close as implemented on current main: the chat/session refresh path no longer sends the 120-minute active window, so older valid sessions are not filtered out by the gateway before the picker can render them.

## What This Changes

The issue reports that Control UI chat/session discovery hides older valid cron-backed sessions because chat refreshes send a hard-coded activeMinutes:120 filter.

## Best Possible Solution

Keep current main's recency-free bounded chat session discovery, and route any separate cron-filter default or broader session-kind UX decision through the existing selector-filter discussion.

## Reproduction Assessment

No current-main failing path remains for the activeMinutes cutoff: chat refreshes now use activeMinutes 0, and loadSessions omits the parameter when it is not positive. The old behavior is still visible in the v2026.5.20 tag, which matches the release confirmation comment.

## Solution Assessment

Yes. Removing the activeMinutes filter at the Control UI chat request boundary is the narrow maintainable fix; changing the separate default cron-hide toggle would be a distinct product/UX decision.
