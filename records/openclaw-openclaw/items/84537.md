## Summary

Keep open: current main and the latest release still contain the reported Control UI session-discovery behavior, and the linked open PR covers the 120-minute fetch cutoff but not the reporter's newer note that non-current cron sessions are hidden by default in the selector.

## What This Changes

The issue asks for valid cron-backed chat sessions to remain discoverable in the Control UI after upgrade instead of being hidden by the chat refresh lookback or default cron-session filter.

## Best Possible Solution

Review or update the linked PR so chat discovery omits the recency cutoff and persistent custom cron sessions remain selectable, while preserving the bounded limit and existing protection against noisy cron-run/session clutter.

## Reproduction Assessment

Yes. Source inspection shows chat refreshes send activeMinutes:120 and the gateway filters older updatedAt rows; the latest comment's second path is also source-reproducible because sessionsHideCron defaults true and skips non-current cron session keys.

## Solution Assessment

Partially. Removing activeMinutes from chat refreshes is the right boundary for the first bug, but the new sessionsHideCron finding means the linked PR may need a maintainer UX decision or follow-up before this issue is fully resolved.
