## Summary

Keep open: current main and the latest shipped release still send Control UI chat/session refreshes through a hard-coded 120-minute activeMinutes filter, and the Gateway filters rows by updatedAt before returning sessions. That source path matches the reported disappearance of valid older cron/custom sessions, so this remains a current, narrow session-discovery bug.

## What This Changes

The issue asks for older valid cron-backed chat sessions to remain discoverable in the Control UI chat/session selector instead of being hidden by a hard-coded 120-minute activeMinutes filter.

## Best Possible Solution

Fix the Control UI chat/session selector refresh path so persistent cron/custom sessions remain selectable even when older than two hours, while preserving bounded recent-session behavior and the Gateway activeMinutes contract for callers that explicitly ask for recency filtering.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create or keep a cron/custom session whose updatedAt is older than 120 minutes, then trigger a Control UI chat refresh that calls sessions.list with activeMinutes:120. I did not run a live UI reproduction because this review is read-only.

## Solution Assessment

Yes, the maintainable fix is to adjust the Control UI chat/session selector request shape or merge behavior rather than changing all session storage semantics. The repair should keep older persistent cron/custom sessions discoverable without making every recency-filtered sessions.list caller unbounded.
