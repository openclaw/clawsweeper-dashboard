- `P2`: This is a normal-priority user-visible Control UI bug with clear source paths and limited blast radius around chat session discovery.
- `impact:session-state`: The issue is about valid persisted cron chat sessions disappearing from the UI selector even though their session state remains present.

## Summary

Keep open: current main and v2026.5.19 still send `activeMinutes: 120` from chat refresh paths, and the newer issue comment identifies a second current-main selector path where `sessionsHideCron` defaults to hiding non-current cron sessions. The linked fix PR is open and unmerged, so this issue should remain the canonical tracker until that PR or a successor covers both user-visible paths.

## What This Changes

The issue asks for long-lived valid cron-backed chat sessions to remain discoverable in the Control UI chat selector after upgrade instead of being excluded by a 120-minute fetch cutoff or default cron-session hiding.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84683 or a successor that removes the chat `activeMinutes` cutoff and explicitly handles persistent cron session visibility while preserving the existing clutter controls for per-run and archived sessions.

## Reproduction Assessment

Yes. Source inspection shows a persistent cron session older than 120 minutes is excluded when chat refresh sends `activeMinutes: 120`, and a second current-main path hides non-current cron sessions when `sessionsHideCron` is true.

## Solution Assessment

Unclear for the full issue. The linked PR is the right narrow boundary for the activeMinutes cutoff, but the later default cron-hide report needs a maintainer UX decision or a more targeted rule for persistent cron sessions.
