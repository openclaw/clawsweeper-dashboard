## Summary

Keep open: the doctor SecretRef false-positive is valid, but this PR still lacks real behavior proof and the current diff would resolve exec-backed SecretRefs during doctor without an explicit opt-in. The original task timestamp portion is already covered by the merged task registry work, so the remaining path is doctor-only consolidation.

## What This Changes

This PR changes Gateway doctor auth health to resolve a SecretRef-backed `gateway.auth.token` before warning, adds doctor e2e coverage for resolved and non-token SecretRef cases, and adds a changelog entry.

## Best Possible Solution

Consolidate on a doctor-specific SecretRef check that resolves env/file refs only after token auth is required, avoids exec provider execution without explicit opt-in, reports skipped exec refs accurately, and leaves task timestamp behavior to the already merged task registry implementation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure local token auth with a resolvable `gateway.auth.token` SecretRef and run `openclaw doctor`; current main sees the SecretRef but the synchronous resolver drops SecretRef object values and emits the unavailable warning.

## Solution Assessment

No, not as written. Resolving after `needsToken` is the right shape for env/file refs, but this PR uses a helper path that also executes exec-backed SecretRef providers during doctor without an explicit policy or opt-in.
