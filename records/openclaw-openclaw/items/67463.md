## Summary

Keep open. Current main still has the Gateway doctor SecretRef false-positive path, while the task timestamp part of the original PR is already covered by merged #71871; the latest PR diff remains the focused implementation candidate pending maintainer review and exact-head validation.

## What This Changes

This PR resolves SecretRef-managed Gateway tokens in doctor after token auth is needed, adds doctor e2e coverage, and records the Gateway doctor fix in CHANGELOG.md.

## Best Possible Solution

Land a doctor-only SecretRef fix that resolves the configured token only after token auth is required, while leaving task timestamp behavior anchored to #71871's shipped implementation.

## Reproduction Assessment

Yes. Configure local Gateway token auth with gateway.auth.token as a resolvable SecretRef and run openclaw doctor; current main strips SecretRef token values before the doctor token-needed check and can emit the unavailable warning even though Gateway startup can resolve the token.

## Solution Assessment

Yes. The current PR approach reuses the existing SecretRef-aware Gateway token resolver behind the needsToken decision, which is narrower than making synchronous auth resolution execute secret providers or resolving tokens for non-token auth modes.
