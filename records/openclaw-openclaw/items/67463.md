## Summary

Keep open: the Gateway doctor SecretRef false-positive is a real current-main bug, but this PR still needs contributor real behavior proof and has a concrete exec-SecretRef command-execution issue before it can merge.

## What This Changes

The PR changes Gateway doctor auth health to resolve a SecretRef-backed `gateway.auth.token` before warning, adds doctor E2E coverage for resolved and non-token cases, and adds a changelog entry.

## Best Possible Solution

Consolidate into one doctor fix that resolves safe env/file Gateway token SecretRefs only after token auth is required, avoids implicit exec execution, reports exec refs as unverified or opt-in, preserves regression coverage, and leaves task timestamp behavior anchored to the shipped task implementation.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure local token auth with a resolvable `gateway.auth.token` SecretRef and run `openclaw doctor`; current main detects the SecretRef, `resolveGatewayAuth()` ignores SecretRef object values, and the warning branch fires.

## Solution Assessment

No, not as written. Reusing the existing SecretRef-aware token resolver after `needsToken` is the right shape for env/file refs, but this PR also executes exec-backed providers during doctor without an explicit opt-in.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip exec SecretRefs during doctor:** `src/flows/doctor-health-contributions.ts:179`
  - body: This `resolveGatewayAuthTokenForService()` call delegates into the generic SecretRef resolver, whose exec provider path invokes configured commands.
