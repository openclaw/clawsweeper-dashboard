## Summary

Keep open: current main still hard-omits `skill_workshop` for sandboxed tool sets, and the requested opt-in crosses a sandbox/security boundary plus adds config surface, so it needs maintainer/security direction rather than cleanup closure.

## What This Changes

The issue requests a default-off `skills.workshop.allowSandboxedAgents` option so sandboxed agents can receive the built-in `skill_workshop` tool.

## Best Possible Solution

Keep the secure default, then if maintainers approve sandboxed Skill Workshop access, implement one canonical opt-in surface with schema/docs/tests and an explicit sandbox security rationale.

## Reproduction Assessment

Yes from source inspection: `createOpenClawTools({ sandboxed: true })` omits `skill_workshop`, and the current test suite asserts that behavior. I did not run a live sandboxed agent repro in this read-only review.

## Solution Assessment

Unclear: the requested default-off flag identifies the right construction gate, but maintainers still need to choose whether the best opt-in is a new `skills.workshop` key or reuse of `tools.sandbox.tools.alsoAllow`.
