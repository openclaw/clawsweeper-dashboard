## Summary

Keep open: current main and the latest release still hard-omit `skill_workshop` for sandboxed tool sets, while the requested opt-in would add config surface across a sandbox/security boundary that needs maintainer/security direction rather than cleanup closure.

## What This Changes

The issue requests a default-off `skills.workshop.allowSandboxedAgents` or equivalent opt-in so sandboxed agents can use the built-in `skill_workshop` tool.

## Best Possible Solution

Keep default-deny behavior, then if maintainers approve the boundary, implement one canonical opt-in or warning path with schema, docs, focused tests, and live sandbox proof.

## Reproduction Assessment

Yes. Source inspection shows `createOpenClawTools({ sandboxed: true })` omits `skill_workshop`, the current test suite asserts that omission, and sandbox skill mounts are read-only; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Unclear. The requested default-off flag is plausible, but maintainers still need to decide whether the best owner is a new `skills.workshop` key, the existing sandbox tool allowlist, or a warning-only path.
