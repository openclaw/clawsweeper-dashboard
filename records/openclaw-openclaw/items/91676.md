## Summary

Current main still intentionally omits `skill_workshop` from sandboxed sessions, and no merged or open replacement owns the request. The capability is plausible and vision-aligned, but exposing a host-side skill-authoring tool across the sandbox boundary and choosing between the existing sandbox allowlist and a new config key require explicit product and security decisions.

## What This Changes

Add an operator-controlled, default-off way for sandboxed agents to receive the built-in `skill_workshop` tool.

## Best Possible Solution

Adopt one canonical, default-off sandbox policy that controls both construction and filtering of `skill_workshop`, with security-owner approval and regression coverage proving per-agent workspace and proposal isolation.

## Reproduction Assessment

Yes, at source level. Calling the current tool-construction path with `sandboxed: true` unconditionally omits `skill_workshop`; the reporter also supplied detailed real-environment lifecycle and isolation observations, though this read-only review did not independently rerun them.

## Solution Assessment

Unclear as proposed. A default-off opt-in is directionally appropriate, but maintainers should first decide whether `tools.sandbox.tools.alsoAllow` should become the canonical construction-time authorization rather than adding a parallel `skills.workshop` flag.

## Risks / Open Questions

- A new `skills.workshop.allowSandboxedAgents` key could duplicate or conflict with `tools.sandbox.tools.alsoAllow`, leaving two policy surfaces that must agree before the tool is usable.
- `skill_workshop` is a host-side capability that can apply persistent workspace skill changes, so sandbox exposure needs an explicit contract covering workspace confinement, proposal isolation, approval policy, and hostile tool arguments.
- Adding another config key increases a compatibility-sensitive public surface; the existing policy mechanism should be preferred unless it cannot safely express construction-time availability.
