- `P1`: The in-product update path is broken for recommended Linux systemd-supervised installs, leaving users on the old version unless they know the manual shell workaround.
- `impact:crash-loop`: The report is about process lifecycle availability: systemd kills the helper during the supervised gateway restart, so the managed update process never completes.

## Summary

Keep open: current `main` still uses a plain detached helper for managed update handoff while the documented/generated systemd unit uses `KillMode=control-group`, and the linked fix PR is still open with closing syntax. The issue should stay open until that PR lands or maintainers choose a different fix path.

## What This Changes

The issue reports that the Control UI Update button is a no-op on user-systemd installs because the managed update handoff helper is killed with the gateway service cgroup before it runs.

## Best Possible Solution

Review and land a systemd-specific handoff fix, likely the linked PR's transient user scope approach, while preserving the existing non-systemd detached-helper behavior and adding regression coverage for the never-ran helper case.

## Reproduction Assessment

Yes. The reporter provides concrete Linux user-systemd upgrade steps and logs, and current main still combines a detached helper spawn with `KillMode=control-group`; I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to launch the systemd handoff helper outside `openclaw-gateway.service`'s cgroup, with tests and real systemd proof, then close this issue after the linked PR merges.
