## Label Justifications

- `P1`: The in-product update path is broken for recommended Linux systemd-supervised installs, leaving users stuck on the old version unless they manually upgrade from a shell.

## Summary

Keep open: current main still starts the managed update helper as a plain detached child before scheduling the supervised gateway restart, and systemd's documented/generated `KillMode=control-group` keeps that child in the unit kill boundary. Two open PRs already claim this fix, so this issue should stay open until one lands or the fix path is rejected.

## What This Changes

Make Control UI managed updates on systemd-supervised installs survive the gateway unit restart instead of leaving the handoff stuck at `managed-service-handoff-started`.

## Best Possible Solution

Land one focused systemd handoff fix that launches the update helper in a transient user scope or service outside `openclaw-gateway.service`, preserves the existing non-systemd paths, and adds a clear stale-handoff diagnostic.

## Reproduction Assessment

Yes. The reporter provided a concrete Linux user-systemd upgrade reproduction, and current source still starts the helper as a detached child before the systemd-supervised gateway restart; I did not run the live systemd upgrade in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to keep the managed-service update contract but make the systemd path escape the service cgroup before restart, with coverage and a clearer diagnostic for helpers that never run.
