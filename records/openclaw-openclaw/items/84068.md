## Label Justifications

- `P1`: The recommended Linux systemd-supervised Control UI update path remains broken for affected users, though a manual shell upgrade workaround exists.
- `impact:crash-loop`: The reported failure is a process-lifetime availability bug where systemd kills the updater helper during a supervised gateway restart.

## Summary

Keep open: current `main` still starts the managed update helper as a plain detached child before the supervised gateway restart, and two open PRs already reference this issue as their fix target. This should close only after one fix lands and proves the Linux user-systemd update path.

## What This Changes

Fix systemd-supervised Control UI updates so the handoff helper survives the gateway unit restart and runs the package update instead of leaving the sentinel stuck at `managed-service-handoff-started`.

## Best Possible Solution

Land one systemd-specific handoff fix that launches the helper outside `openclaw-gateway.service` while preserving launchd, Windows task, and non-supervised behavior, then verify a real Linux user-systemd upgrade end to end.

## Reproduction Assessment

Yes. The reporter provides a concrete Linux user-systemd upgrade failure, and current `main` still combines a detached child helper with a `KillMode=control-group` systemd unit and an immediate supervised restart.

## Solution Assessment

Yes. The maintainable fix is to make only the systemd handoff escape the service cgroup, for example via a transient user scope/service, while preserving the existing managed-service contract and adding a clearer never-ran diagnostic.
