## Label Justifications

- `P1`: The in-product update path is broken for recommended Linux user-systemd managed installs on current main and the latest shipped release.

## Summary

Keep open: current main still launches the managed update helper as a detached child inside the gateway context while systemd units use `KillMode=control-group`, and two open closing PRs exist but neither is merged.

## What This Changes

The issue reports that Control UI managed updates under user-systemd create a handoff helper, then the gateway restart kills it before it runs `openclaw update`.

## Best Possible Solution

Land one canonical systemd-specific handoff fix that launches the helper outside `openclaw-gateway.service`, with focused tests and live Linux user-systemd update proof.

## Reproduction Assessment

Yes. The reporter provided a concrete Linux user-systemd reproduction, and current source plus the generated `KillMode=control-group` unit show why a plain detached child can die before writing `handoff.log`; I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

Yes. Escaping the helper into a systemd user scope or transient unit before restarting the gateway is the narrow maintainable fix, and the open closing PRs are the right path to review rather than closing this issue now.
