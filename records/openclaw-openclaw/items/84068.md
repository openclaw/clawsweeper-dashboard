## Label Justifications

- `P1`: The in-product update path is broken for recommended Linux user-systemd installs on current shipped code, even though manual shell upgrades remain available.
- `impact:crash-loop`: The failure is rooted in supervised process lifecycle handling: the updater helper is killed by the service cgroup and the managed update workflow remains unavailable after restart.

## Summary

Keep open: current main and the latest shipped release still launch the managed update helper as a plain detached child before restarting a systemd unit that uses control-group killing, and two open closing PRs are already tracking the systemd-scope fix.

## What This Changes

The issue asks for systemd-supervised Control UI/package update handoffs to launch outside the gateway service cgroup so the helper survives the service restart and reports a clearer failure if it never runs.

## Best Possible Solution

Review and land one focused systemd managed-update handoff fix that moves the helper into a transient user scope before restarting the gateway, keeps non-systemd behavior intact, and proves the full user-systemd update path.

## Reproduction Assessment

Yes. The reporter gave a concrete Linux user-systemd upgrade failure, and current source plus systemd documentation show the helper remains inside a KillMode=control-group unit when only detached-spawned; I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

Yes. A transient user scope for systemd handoffs is the narrow maintainable repair because it preserves the managed-update contract while moving only the helper that must outlive the gateway service cgroup.
