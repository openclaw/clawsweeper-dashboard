## Label Justifications

- `P1`: The in-product update flow is broken for recommended Linux systemd-supervised installs, leaving users unable to update through the Control UI even though manual shell upgrades work.

## Summary

Keep open: current main and the latest shipped release still launch the managed update helper as a plain detached child under the supervised gateway process, while the linked closing PR is open and not merged. The report remains a concrete systemd update bug with a high-confidence source reproduction path.

## What This Changes

Fix the Control UI/package-manager update handoff on user-systemd installs so the helper survives the gateway unit restart and reports a useful failure if it never runs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84151 or an equivalent scoped systemd handoff, then close this issue after the merged fix is on main with systemd upgrade proof.

## Reproduction Assessment

Yes, at source level: current main starts the helper as a detached child under a generated systemd unit that uses `KillMode=control-group`, then schedules the gateway restart. I did not run a live systemd upgrade in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the maintainable fix is to move the systemd helper into a transient user scope or service before the gateway restarts, while keeping the existing managed-service update contract. The open closing PR is the right implementation candidate to review before this issue closes.
