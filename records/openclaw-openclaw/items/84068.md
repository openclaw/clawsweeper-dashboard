## Label Justifications

- `P1`: The in-product update path is broken for recommended Linux systemd-supervised installs, leaving users stuck on the old gateway version unless they use a manual shell workaround.

## Summary

Keep open: current main and the latest shipped v2026.5.20 code still launch the managed update helper as a plain detached child before restarting the systemd unit, so the reported cgroup-kill race remains real until one of the linked fix PRs merges.

## What This Changes

Fix systemd-supervised Control UI/package update handoffs so the helper survives the gateway unit restart and does not leave the sentinel stuck at `managed-service-handoff-started`.

## Best Possible Solution

Land one scoped systemd handoff fix with real user-systemd proof, preserve non-systemd behavior, then close this issue from the merged PR and retire the duplicate branch.

## Reproduction Assessment

Yes: the reporter gives a concrete Linux user-systemd upgrade failure, current main still spawns the helper as a detached child inside the service context, and systemd docs confirm `KillMode=control-group` kills unit-cgroup children. I did not run a live upgrade in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes: moving only the systemd handoff into a transient user scope is the narrow maintainable fix for a helper that must outlive `openclaw-gateway.service`. The remaining maintainer decision is which open PR to land and what live upgrade proof is sufficient.
