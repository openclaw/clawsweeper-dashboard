## Label Justifications

- `P1`: The in-product update workflow is broken for recommended Linux user-systemd installs on current main and the latest listed release.

## Summary

Keep open: current main and the latest release still launch the managed update handoff helper as a plain detached child before the supervised restart, which is consistent with the reported systemd cgroup kill race. Two open PRs already close this issue, so the issue should stay open until maintainers land or reject the linked fix path.

## What This Changes

The issue reports that Linux systemd-supervised Control UI updates create the handoff helper but the service restart kills it before it runs, leaving the update stuck at `managed-service-handoff-started`.

## Best Possible Solution

Review and land one canonical systemd handoff fix from https://github.com/openclaw/openclaw/pull/84151 or https://github.com/openclaw/openclaw/pull/85414, then close this issue as implemented once the merged path has acceptable Linux user-systemd proof.

## Reproduction Assessment

Yes, at source level: current main plain-detached-spawns the helper and then schedules a supervised restart, while generated systemd units use `KillMode=control-group`. I did not perform a live systemd package upgrade in this read-only review.

## Solution Assessment

Yes, the maintainable fix direction is to launch the systemd handoff helper in a transient user scope or equivalent unit outside `openclaw-gateway.service`, while preserving the existing non-systemd paths and adding clearer diagnostics if the helper never runs.
