## Label Justifications

- `P1`: The in-product update workflow is broken for recommended Linux user-systemd installs and can leave users unable to update from the Control UI without a manual shell workaround.

## Summary

Keep open: current `main` and the latest shipped release still launch the managed update helper as a plain detached child while generated Linux systemd units use `KillMode=control-group`, so the reported helper-killed-before-work race remains valid. Two open PRs already reference this issue with closing links, so this issue should stay open until one canonical fix lands.

## What This Changes

Fix the Control UI/package-update handoff on user-systemd installs so the helper escapes the gateway unit cgroup before restart and reports a clearer never-ran failure.

## Best Possible Solution

Land one canonical scoped-systemd handoff fix that uses a transient user scope or equivalent durable systemd boundary, preserves non-systemd handoff behavior, and adds a clear diagnostic for helpers that never create `handoff.log`.

## Reproduction Assessment

Yes. The reporter provided concrete Linux user-systemd steps, and current source still starts the helper inside a unit that uses `KillMode=control-group`; I did not run a live upgrade in this read-only pass.

## Solution Assessment

Yes. Escaping the systemd unit cgroup before restarting the gateway is the narrow maintainable fix, and the issue should close only after one linked implementation PR merges.
