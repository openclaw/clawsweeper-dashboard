## Label Justifications

- `P1`: The in-product update flow is broken for recommended Linux systemd-supervised gateway installs, even though a manual shell upgrade workaround exists.

## Summary

Keep open: current main still uses a plain detached helper for managed-service updates, while generated user-systemd units keep child processes in the service cgroup. The issue is concrete, source-reproducible, and already has an open linked fix PR that should be reviewed or replaced before the issue closes.

## What This Changes

The issue reports that the Control UI update path fails under user systemd because the detached update handoff helper remains in the gateway service cgroup and is killed before running.

## Best Possible Solution

Review and land the linked systemd scoped-handoff fix, or replace it with an equivalent narrow fix that launches the update helper outside the gateway service cgroup and adds a clear never-ran diagnostic.

## Reproduction Assessment

Yes. The reporter provides concrete Linux user-systemd evidence, and current main shows a detached helper followed by a supervised restart under KillMode=control-group; I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to preserve the managed-service update contract while making the systemd path launch the helper in a transient user scope or equivalent unit outside openclaw-gateway.service, with clearer diagnostics if the helper never runs.
