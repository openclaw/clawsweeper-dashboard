## Summary

Keep open: current main still has the reported two-step cross-agent denial path, and the first visibility error still only names `tools.sessions.visibility=all` even though the documented contract also requires `tools.agentToAgent` for cross-agent targeting.

## What This Changes

The issue requests clearer cross-agent session-tool diagnostics and/or doctor/config validation when multi-agent operators configure subagent allowlists but direct session visibility and agent-to-agent policy still block cross-agent send/history/status.

## Best Possible Solution

Keep the explicit cross-agent security defaults, but update the shared session-tool denial path and any narrow doctor/config lint so operators see both required controls before troubleshooting target-agent health.

Yes, from source inspection: default visibility resolves to `tree`, and cross-agent targets hit the visibility-only forbidden message before `tools.agentToAgent` is evaluated. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes for a diagnostic-only repair: improving the shared denial message and focused lint is narrower and safer than auto-enabling direct cross-agent access from `subagents.allowAgents`. Automatic migration or policy broadening should remain a separate maintainer decision.
