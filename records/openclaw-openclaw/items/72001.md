## Summary

Keep open. The PR addresses the startup validator gap only partly: Gmail mapped delivery still rejects the templated preset session key without `hooks.allowRequestSessionKey=true`, and the unconditional prefix write can invalidate existing static-shadow/default-session configs.

## What This Changes

The PR adds a Gmail prefix merge helper, writes `hooks.allowedSessionKeyPrefixes: ["hook:"]` from Gmail setup, adds hook tests, and adds a changelog fix entry.

## Best Possible Solution

Revise the wizard to emit a complete bounded Gmail session policy for the effective mapping it installs, or preserve static-session configurations by using or respecting a static mapping path.

## Reproduction Assessment

Yes. The source-level path is high-confidence: the PR emits the templated Gmail preset plus prefixes, but current dispatch still rejects `mapping-templated` session keys unless `allowRequestSessionKey` is true.

## Solution Assessment

No. Prefix seeding is necessary for startup validation, but by itself it does not make `/hooks/gmail` delivery work and can regress supported static-shadow/default-session setups.
