## Summary

Keep open. The PR addresses a real Gmail wizard startup failure, but it is not merge-ready because it writes only the prefix allowlist: mapped Gmail delivery still fails the templated-session dispatch gate, and the unconditional prefix can invalidate supported configs that shadow the Gmail preset with a static mapping and use a non-`hook:` default session.

## What This Changes

The PR adds a Gmail helper for merging `hooks.allowedSessionKeyPrefixes`, writes `"hook:"` from `openclaw webhooks gmail setup`, adds hook/gateway tests, and adds a changelog fix entry.

## Best Possible Solution

Revise the wizard to emit a complete Gmail hook session policy for the effective mapping it installs: either enable bounded templated Gmail sessions with `allowRequestSessionKey=true` and matching prefixes, or preserve static-shadow/default-session configs by using a static mapping path.

## Reproduction Assessment

Yes. The source-level path is high-confidence: the wizard installs the Gmail preset, the preset has a templated `sessionKey`, startup requires prefixes, and mapped delivery still rejects templated session keys unless `hooks.allowRequestSessionKey` is true; tests were not run because this review is read-only and `pnpm` is unavailable in PATH.

## Solution Assessment

No. Seeding `allowedSessionKeyPrefixes` is necessary for startup validation but is not sufficient for runtime `/hooks/gmail` delivery, and doing it unconditionally can regress existing static-shadow/default-session configurations.
