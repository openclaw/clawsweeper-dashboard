## Summary

Keep open. The reported Gmail wizard bug is real on current main, but this PR is not merge-ready: it only seeds `allowedSessionKeyPrefixes`, while the active hook dispatch path also requires `allowRequestSessionKey=true` for templated Gmail session keys, and the unconditional prefix write can break or broaden existing session-policy configurations.

## What This Changes

The PR adds a Gmail helper for merging `hooks.allowedSessionKeyPrefixes`, has `openclaw webhooks gmail setup` write `"hook:"` beside the Gmail preset, adds Gmail/gateway tests, and appends a changelog fix entry.

## Best Possible Solution

Update the wizard so it emits a complete, internally consistent Gmail hook session policy for the effective mapping it installs: templated Gmail routing should enable request-derived session keys only with an appropriately bounded namespace, while configs that shadow the preset with static mappings or rely on an existing default session key should remain loadable. Add tests for startup validation, mapped `/hooks/gmail` dispatch, and static-shadow/default-session compatibility.

## Reproduction Assessment

Yes. The read-only source path on current main reproduces the original startup failure: the wizard writes the Gmail preset, the preset has a templated session key, and `resolveHooksConfig` requires prefixes. The PR regressions are reproducible by code inspection against `resolveHookSessionKey` and the existing static-shadow/default-session tests; tests were not run because this review was read-only.

## Solution Assessment

No. Seeding `allowedSessionKeyPrefixes` is necessary for one validator error, but it is not sufficient for Gmail delivery and is too broad for some existing configs. A safer fix should update the full session policy conditionally or install a static mapping when request-derived session keys should stay disabled.
