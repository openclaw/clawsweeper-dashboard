## Summary

Keep open. The PR addresses a real current-main Gmail wizard/config gap, but the proposed prefix-only patch is not merge-ready because mapped Gmail delivery still fails the existing templated-session policy and the unconditional prefix can invalidate supported static-shadow/default-session configurations.

## What This Changes

The PR adds a helper to merge `hooks.allowedSessionKeyPrefixes`, has Gmail setup write `"hook:"` alongside the Gmail preset, adds Gmail/gateway tests, and adds a changelog fix entry.

## Best Possible Solution

Revise the wizard so it emits a complete Gmail hook session policy for the effective mapping it installs: templated Gmail routing should set `hooks.allowRequestSessionKey=true` with a bounded Gmail/hook prefix allowlist, while configs that shadow the preset with static mappings or use a non-`hook:` default session remain loadable. Add coverage for startup validation, mapped `/hooks/gmail` dispatch, and the static-shadow/default-session compatibility case.

## Reproduction Assessment

Yes. Current main provides a source-level reproduction path: the wizard writes the Gmail preset, the preset has a templated session key, and `resolveHooksConfig` requires prefixes. The PR regressions are also reproducible by code inspection against the dispatch gate and existing static-shadow/default-session tests; tests were not run because this review is read-only.

## Solution Assessment

No. Prefix seeding fixes only the startup validator failure; it does not enable templated mapped Gmail delivery because `allowRequestSessionKey` stays false, and it can break existing configs by applying a prefix policy when the templated preset is shadowed.
