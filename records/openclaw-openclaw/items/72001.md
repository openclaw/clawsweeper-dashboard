## Summary

Keep open: current main still has the Gmail wizard/preset mismatch, but this PR only fixes startup validation and leaves mapped Gmail delivery blocked by the existing session-key runtime gate; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a Gmail prefix merge helper, writes `hooks.allowedSessionKeyPrefixes: ["hook:"]` during Gmail setup, adds hook/gateway tests, and adds a changelog entry.

## Best Possible Solution

Gmail setup should emit a config whose startup validation and mapped delivery policy agree, with regression coverage for both config resolution and `/hooks/gmail` dispatch.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the wizard installs the templated Gmail preset, startup validation requires prefixes, and mapped dispatch still rejects templated keys unless `hooks.allowRequestSessionKey` is enabled; I did not run live Gmail/Pub/Sub in this read-only review.

## Solution Assessment

No. Adding only `allowedSessionKeyPrefixes` is necessary for startup validation, but it is not the narrow complete fix because runtime mapped delivery still rejects the templated Gmail key and static-session configurations can be broken by the injected prefix.
