## Summary

Keep open: the approval-routing direction is useful, but this PR is not merge-ready because it still has two source-reproducible routing bugs, lacks after-fix real behavior proof, and mixes in a security-sensitive guarded-fetch transport change.

## What This Changes

The PR changes exec/plugin approval routing, Telegram approval delivery and authorization, approval docs/tests, deterministic approval prompt state propagation, and guarded-fetch undici dispatcher handling.

## Best Possible Solution

Rebuild the approval routing change on current main's shared approval request handler, carry approvalKind/agentId/sessionKey/turnSourceTo/account inputs through the route decision, keep Telegram delivery targets separate from approval authority, and split guarded-fetch transport work into its own reviewed change.

## Reproduction Assessment

Yes, from source inspection: configure approvals exec/plugin mode as targets with a Telegram target, trigger an approval from an origin surface that also has approval clients, and current main broadcasts before forwarding ownership is known. The PR-specific blockers are also source-reproducible for agentId-only exec filters and Web UI/TUI-origin targets-only approvals.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it patches stale per-handler routing, misses two suppression cases, and carries unrelated guarded-fetch network-boundary work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agentId into targets-only filtering:** `src/infra/exec-approval-surface.ts:92-97`
  - body: `shouldDisableInitiatingSurfaceForExecTargetsOnly` filters with only `sessionKey`. When an exec approval has `agentId` but no `sessionKey` and `approvals.exec.agentFilter` is configured, targets-only suppression can miss and the gateway can still broadcast an origin approval prompt after forwarding accepted the request.
  - confidence: 0.9
- **[P2] Apply targets-only suppression before UI returns:** `src/infra/exec-approval-surface.ts:119-128`
  - body: The new targets-only branch in the early return only handles the missing-channel case. Web UI and TUI sources still return `enabled`, so forwarded approvals in `approvals.exec.mode = "targets"` can still produce actionable origin-surface prompts instead of target-only routing.
  - confidence: 0.87
