## Summary

Keep open. The PR targets a real remaining approval-routing gap, but the branch is not merge-ready: current main still broadcasts approval requests before forwarding ownership is known, while the PR misses two targets-only suppression cases and mixes in an unrelated guarded-fetch transport change.

## What This Changes

The PR changes exec/plugin approval routing, Telegram approval delivery and authorization, approval docs/tests, deterministic approval prompt state propagation, and guarded-fetch undici dispatcher handling.

## Best Possible Solution

Rebuild the approval routing change on current main's shared approval request handler: decide forwarding ownership before origin broadcasts, carry approvalKind/agentId/sessionKey/turnSourceTo/account inputs through the route decision, keep Telegram delivery targets separate from approval authority, and split guarded-fetch transport work into a separate security-reviewed change.

## Reproduction Assessment

Yes. Configure approvals exec/plugin mode as targets with a Telegram target, trigger an approval from an origin surface that also has approval clients, and current main can broadcast before forwarding ownership is known; the PR-specific blockers reproduce with exec agentFilter plus agentId-only requests and with Web UI/TUI-origin targets-only approvals.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it patches older per-handler routing, misses two suppression cases, and carries unrelated guarded-fetch network-boundary work; a narrower rebased shared-handler patch is safer.
