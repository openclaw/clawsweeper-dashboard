## Summary

Keep this PR open but not merge-ready: the approval-routing direction is useful and current main still has a source-visible targets-mode routing problem, but the branch lacks real Telegram proof, still has two routing blockers, and mixes an SSRF/network-boundary fetch change into the approval PR.

## What This Changes

The PR changes exec/plugin approval forwarding and Telegram approval handling for explicit targets mode, adds admin-terminal docs/tests, updates deterministic approval prompt state, and modifies guarded-fetch dispatcher fetch behavior.

## Best Possible Solution

Rework this onto the current shared approval request handler, carry approvalKind, agentId, sessionKey, turnSourceTo, and account inputs through the route decision, keep delivery targets separate from approval authority, split guarded-fetch transport work, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes, with high confidence from source inspection: configure approvals.exec or approvals.plugin with mode "targets" and a Telegram target, then trigger an approval from an origin surface that has approval clients. Current main broadcasts the origin approval event before forwarding ownership is known, and the PR-specific remaining blockers are visible in the branch source.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it misses agentId and Web UI/TUI suppression cases, needs to target the current shared handler, and mixes a guarded-fetch network-boundary change into the approval-routing work.
