## Summary

Keep this PR open. The central approval-routing problem is still present on current main, but the branch is not merge-ready because it misses two targets-only suppression paths and includes an unrelated guarded-fetch transport change in a security-sensitive boundary.

## What This Changes

The PR changes exec/plugin approval routing, Telegram native approval delivery and authorization, approval docs/tests, deterministic prompt state propagation, and guarded-fetch undici dispatcher handling.

## Best Possible Solution

Implement the routing policy on current main's shared approval request handler: compute an approval-kind-aware route descriptor before broadcasting, include agentId/sessionKey/turnSourceTo/account inputs in the turn-source decision, suppress origin broadcasts only when forwarding owns the route, keep Telegram delivery targets separate from approval authority, and split guarded-fetch changes into a separate security-reviewed PR.

## Reproduction Assessment

Yes. Current main can be checked by configuring targets-mode forwarding to Telegram and triggering exec/plugin approvals from an origin with approval clients; the code broadcasts before forwarding ownership is known. The PR findings are reproducible with approvals.exec.agentFilter plus agentId without sessionKey, and with Web UI/TUI origins in targets-only mode.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it targets older per-handler routing, misses two suppression cases, and mixes in an unrelated guarded-fetch transport change; the safer path is a narrowed rebase onto the shared approval handler plus a separate fetch-guard review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agentId through targets-only filtering:** `src/infra/exec-approval-surface.ts:92-97`
  - body: Targets-only suppression filters with only `{ sessionKey }`. If an exec approval has `agentId` but no `sessionKey` while `approvals.exec.agentFilter` is configured, the filter returns false and the origin approval prompt can still be broadcast after forwarding accepted the request. Include `agentId` in the helper params and filter request.
  - confidence: 0.9
- **[P2] Apply targets-only suppression before Web UI/TUI returns:** `src/infra/exec-approval-surface.ts:115-128`
  - body: The early return marks Web UI and TUI sources as enabled before targets-only suppression runs. In `approvals.exec.mode = "targets"`, forwarded approvals from those origins can still get origin approval broadcasts instead of target-only routing. Move the suppression check ahead of this return for these origins.
  - confidence: 0.87
