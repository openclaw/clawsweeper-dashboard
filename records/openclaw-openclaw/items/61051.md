## Summary

Keep open: the target-mode approval-routing problem is still valid on current main, but this PR is not merge-ready because it lacks real behavior proof, still has source-reproducible routing bugs, and mixes an approval change with security-sensitive guarded-fetch transport work.

## What This Changes

This PR changes exec/plugin approval routing, Telegram approval delivery and authorization, approval docs/tests, deterministic approval prompt state propagation, and guarded-fetch dispatcher handling.

## Best Possible Solution

Rebuild the approval routing change on the shared approval request handler, pass the full route context through the decision, keep Telegram delivery targets separate from approval authority, and split guarded-fetch transport work into a dedicated review.

## Reproduction Assessment

Yes. From source inspection, configure approvals.exec or approvals.plugin mode as targets with a Telegram target, trigger an approval from an origin surface with approval clients, and current main broadcasts before forwarding ownership is known; the PR-specific blockers are also source-reproducible in the agentId-only and Web UI/TUI paths.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it patches a stale per-handler path, misses two suppression cases, and includes unrelated guarded-fetch network-boundary work.
