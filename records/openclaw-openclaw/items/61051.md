## Summary

Keep open: the approval-routing goal is still valid on current main, but this branch is dirty against main, still has source-reproducible routing blockers, and mixes in a guarded-fetch network-boundary change.

## What This Changes

This PR changes exec/plugin approval forwarding, Telegram native approval authorization and delivery, admin-terminal documentation/tests, deterministic approval prompt state tracking, and guarded-fetch dispatcher handling.

## Best Possible Solution

Rebuild the approval-routing change on current main's shared approval request handler, carry approvalKind/agentId/sessionKey/turnSourceTo/account inputs through route decisions, keep Telegram delivery targets separate from approval authority, and split guarded-fetch work into its own reviewed change.

## Reproduction Assessment

Yes. From source inspection, configure approvals.exec or approvals.plugin mode as targets with a Telegram target and an origin surface that also has approval clients; current main broadcasts before forwarding ownership is known, and the PR-specific blockers reproduce for agentId-only filters and Web UI/TUI origins.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it is dirty against current main's shared handler, misses two targets-only suppression cases, and mixes in guarded-fetch transport behavior.
