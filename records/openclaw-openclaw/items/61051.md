## Summary

Keep open: the approval-routing problem is still source-reproducible on current main and the PR direction is useful, but this head is not merge-ready because it misses two targets-only suppression cases, lacks real Telegram behavior proof, and mixes in a guarded-fetch network-boundary change.

## What This Changes

The PR changes exec/plugin approval routing, Telegram approval delivery and authorization, approval docs/tests, deterministic approval prompt propagation, and guarded-fetch undici dispatcher handling.

## Best Possible Solution

Rebuild the approval-routing fix on current main's shared approval handler, carry approvalKind/agentId/sessionKey/turnSourceTo/account inputs through one route decision, keep Telegram delivery separate from approval authority, and split guarded-fetch transport work into its own reviewed change.

## Reproduction Assessment

Yes, source-reproducible: configure approvals exec/plugin mode as targets with a Telegram target, trigger an approval from an origin surface that also has approval clients, and current main broadcasts before forwarding ownership is known. The PR-specific blockers are also source-reproducible from the agentId-only filter path and the Web UI/TUI early return.

## Solution Assessment

No. The direction is useful, but this PR is not the best merge shape because it patches older per-handler routing, misses two suppression cases, and carries unrelated guarded-fetch network-boundary work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agentId into targets-only filtering:** `src/infra/exec-approval-surface.ts:92-97`
  - body: `shouldDisableInitiatingSurfaceForExecTargetsOnly` only evaluates filters with `request: { sessionKey }`, so `approvals.exec.agentFilter` is ignored when an exec approval has `agentId` but no `sessionKey`. In that case the origin stays enabled and the gateway can still broadcast a duplicate origin approval after forwarding accepted the request.
  - confidence: 0.9
- **[P2] Apply targets-only suppression before UI returns:** `src/infra/exec-approval-surface.ts:115-128`
  - body: The early Web UI/TUI return runs before the targets-only check. With `approvals.exec.mode = "targets"`, forwarded approvals from those origins can still get actionable origin broadcasts instead of target-only routing.
  - confidence: 0.87
