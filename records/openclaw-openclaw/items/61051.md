## Summary

Keep open, but this PR should not merge as-is: the target-routing problem still exists on current main, while the branch lacks real Telegram behavior proof, is merge-conflicting, has two source-reproducible routing defects, and mixes an unrelated guarded-fetch transport change into approval routing.

## What This Changes

This PR changes exec/plugin approval forwarding to favor explicit targets, tightens Telegram approval delivery and authorization behavior, adds approval admin-terminal docs/tests, propagates deterministic approval prompt state, and modifies guarded-fetch dispatcher handling.

## Best Possible Solution

Rebuild the approval routing change on current main's shared approval request handler, carry approvalKind, agentId, sessionKey, turnSourceTo, and account inputs through the route decision, keep delivery targets separate from approval authority, and split guarded-fetch transport work into a separate reviewed change.

## Reproduction Assessment

Yes, from source inspection: configure approvals.exec or approvals.plugin with mode "targets" and a Telegram target, then trigger an approval from an origin surface with approval clients. Current main broadcasts before forwarding ownership is known, and the PR-specific agentId-only and Web UI/TUI blockers are visible in the PR source.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it misses two suppression cases and mixes approval routing with a guarded-fetch transport change that should be reviewed separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agentId into targets-only filtering:** `src/infra/exec-approval-surface.ts:92-97`
  - body: `shouldDisableInitiatingSurfaceForExecTargetsOnly` only filters with `sessionKey`, so `approvals.exec.agentFilter` is ignored when an exec approval has `agentId` but no `sessionKey`. In that case targets-only routing can be treated as non-matching and the gateway can still broadcast an origin approval prompt after forwarding accepted the request.
  - confidence: 0.9
- **[P2] Apply targets-only suppression before UI returns:** `src/infra/exec-approval-surface.ts:119-128`
  - body: The early return only runs the targets-only suppression for a missing channel, then returns `enabled` for Web UI and TUI origins. Forwarded approvals in `approvals.exec.mode = "targets"` can therefore still produce actionable origin-surface prompts instead of target-only routing.
  - confidence: 0.87
