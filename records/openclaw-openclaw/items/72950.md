## Summary

Keep open: current main still has the reported core plugin-config contract gap. The source, docs, and tests all point to `plugins.entries.<id>.config` as the resolved plugin config surface; existing env substitution only helps when a placeholder was already baked into the read-only config, and there is no generic writable overlay or plugin-declared env mapping path.

## What This Changes

The issue requests a generic way for plugin config to be overridden by env vars or a writable overlay so policy-locked sandbox deployments can change per-plugin settings without rebuilding images.

## Best Possible Solution

Add a generic plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, optionally merges a maintainer-approved writable overlay and explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only that validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows the only resolved, validated, and passed plugin config is `entry.config`. The external NemoClaw EACCES command is supplied in the issue discussion but was not live-run in this review.

## Solution Assessment

Unclear for the exact implementation. The requested direction is appropriate, but env-only is likely incomplete for the reporter's restart-without-rebuild loop; a schema-validated overlay plus explicit env mapping is the safer maintainable direction.
