## Summary

Keep open: current main and the latest release still route native plugin config through the baked `plugins.entries.<id>.config` value, while the active discussion shows this affects multiple policy-locked plugin deployments and needs a maintainer contract decision.

## What This Changes

The issue requests a generic environment-variable override and/or writable overlay path for `plugins.entries.<id>.config` so policy-locked sandboxes can change plugin config without rebuilding images.

## Best Possible Solution

Define a schema-validated plugin config resolver that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, then document precedence and locked-sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection shows plugin config is resolved, validated, and exposed from `plugins.entries.<id>.config` without a declared env override or writable overlay path. The external sandbox filesystem denial was not live-run here.

## Solution Assessment

Unclear for the exact implementation: env-only support may not cover runtime reconfiguration when sandbox env is fixed, while overlay-only misses standard ecosystem env conventions. A generic resolver with documented precedence is the maintainable direction, but it needs maintainer approval first.
