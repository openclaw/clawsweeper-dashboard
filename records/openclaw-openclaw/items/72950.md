## Summary

Keep open: current main still lacks a generic env-declared or writable-overlay resolution path for plugin config, and the requested change affects the core plugin config contract enough that maintainers should choose precedence, reload, and security semantics first.

## What This Changes

Add an operator-updatable plugin config path for policy-locked sandboxes, via plugin-declared env overrides and/or a writable overlay merged with `plugins.entries.<id>.config`.

## Best Possible Solution

Add a generic, schema-validated plugin config resolver that merges baked config, an approved writable overlay, and explicit env mappings with documented precedence and reload behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current source shows plugin config is read, validated, and exposed from `plugins.entries.<id>.config`. The NemoClaw EACCES path is supplied in the issue discussion, but I did not live-run that external sandbox here.

## Solution Assessment

Unclear for the exact implementation. The requested direction is valid, but a host-owned, schema-validated overlay plus explicit env mapping is safer than letting plugins read arbitrary `process.env` independently.
