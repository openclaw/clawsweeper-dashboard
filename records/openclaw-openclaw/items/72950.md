## Summary

Keep open: current main still has no generic env-var override or writable-overlay path for plugin config. The loader validates and passes only the baked `plugins.entries.<id>.config` object, and the follow-up comments show this affects policy-locked plugin deployments beyond one observability setup.

## What This Changes

The issue requests a runtime env-var or writable-overlay path for plugin configuration so policy-locked sandboxes can change `plugins.entries.<id>.config` without rebuilding images.

## Best Possible Solution

Add an explicit plugin-config resolution layer in core: start with baked `plugins.entries.<id>.config`, merge an approved writable overlay and/or plugin-declared env mappings with documented precedence, validate the final object against the plugin manifest schema, and expose only that validated result through `api.pluginConfig`.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the OpenClaw side with high confidence: only `entry.config` is resolved, validated, and passed to plugins; the NemoClaw EACCES command is supplied in the issue and is not needed to confirm the missing core seam.

## Solution Assessment

Yes on the overall direction: an explicit host-side overlay/env-resolution layer before manifest validation is the maintainable fix. Passing raw `process.env` wholesale into plugin registration would be a weaker alternative because it bypasses the unified config and schema contract.
