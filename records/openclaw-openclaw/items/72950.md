## Summary

Keep open: current main still resolves, validates, and exposes plugin config from baked `plugins.entries.<id>.config`; global `${VAR}` substitution is only a partial workaround and does not provide a writable overlay or plugin-declared env override contract for policy-locked sandboxes.

## What This Changes

Request env overrides and/or a writable overlay for plugin config so policy-locked deployments can reconfigure plugins without rebuilding images.

## Best Possible Solution

Add a generic, schema-validated plugin config resolution layer that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, then document precedence for locked-down deployments.

## Reproduction Assessment

Yes for the OpenClaw side: current-main source inspection shows the only resolved, validated, and exposed plugin config is `plugins.entries.<id>.config`. The NemoClaw Landlock `EACCES` command is supplied in the issue discussion, but this review did not live-run that external sandbox.

## Solution Assessment

Unclear for the exact implementation. A writable host-owned overlay plus explicit env mappings is safer than an env-only path or letting every plugin independently read arbitrary process env.
