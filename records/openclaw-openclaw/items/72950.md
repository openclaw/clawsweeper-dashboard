## Summary

Keep open: current main still has only the baked `plugins.entries.<id>.config` path for plugin config resolution and validation. The reported need for a writable overlay or explicit env override contract remains valid, and the comments add useful signal that this affects policy-locked deployments beyond one observability plugin.

## What This Changes

The issue requests an env-var override and/or writable-overlay source for `plugins.entries.<id>.config` so plugin config can change in policy-locked sandboxes without rebuilding images.

## Best Possible Solution

Add an additive core plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, merges a maintainer-approved writable overlay and/or explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence OpenClaw-side reproduction: the resolver, schema, validation path, and API handoff all use only `plugins.entries.<id>.config`; the sandbox write failure is supplied in the issue discussion.

## Solution Assessment

Unclear for the exact implementation. The host-side overlay/env-resolution direction is reasonable, but maintainers should choose the writable path, merge semantics, precedence, and trust boundary before an implementation PR proceeds.
