## Summary

Keep open: current main still has the same core plugin-config contract gap. Plugin config is resolved and validated from `plugins.entries.<id>.config`; existing env substitution can help only when a placeholder is already baked into the read-only config, and the current code does not provide a writable overlay or plugin-declared env mapping for post-build reconfiguration.

## What This Changes

Request a generic way for plugin config to be overridden from declared environment variables and/or a writable overlay so policy-locked sandboxes can change per-deployment plugin settings without rebuilding the image.

## Best Possible Solution

Add a generic plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, optionally merges a maintainer-approved writable overlay and explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side. Source inspection on current main shows the only resolved, validated, and passed plugin config is `entry.config`; the NemoClaw EACCES command is supplied in the issue discussion but was not live-run here.

## Solution Assessment

Unclear for the exact implementation. The direction is appropriate, but a schema-validated overlay plus explicit env mapping is safer than asking every plugin to read arbitrary process env independently.
