## Summary

Keep open: current main still exposes plugin config from the baked `plugins.entries.<id>.config` object, and the existing env substitution path is only a partial placeholder workaround, not a declared per-plugin override or writable overlay.

## What This Changes

Request a runtime-friendly plugin config path so policy-locked sandboxes can override `plugins.entries.<id>.config` through explicit env mappings or a writable overlay without rebuilding images.

## Best Possible Solution

Add a generic plugin-config resolution layer that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, then validates the final object against the plugin manifest schema and documents precedence for locked-down deployments.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows the resolved, validated, and exposed plugin config still comes from `plugins.entries.<id>.config`. The external NemoClaw EACCES behavior is supplied in the issue discussion, but was not live-run in this checkout.

## Solution Assessment

Unclear for the exact implementation. A host-owned, schema-validated overlay plus explicit env mapping is safer than letting each plugin read arbitrary process env independently, but the trust and precedence choices need maintainer review.
