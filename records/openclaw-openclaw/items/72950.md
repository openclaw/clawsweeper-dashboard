## Summary

Keep open: current main still resolves, validates, and injects only the baked `plugins.entries.<id>.config` object. The generic `$include` and `${VAR}` config features are useful adjacent mechanisms, but they do not provide a plugin-declared env override contract or a writable sibling overlay for policy-locked sandbox deployments, and the comments show this affects more than one plugin environment.

## What This Changes

Request a core-supported plugin config override or writable overlay path so policy-locked sandbox deployments can change plugin settings without rebuilding the sandbox image.

## Best Possible Solution

Add an additive core plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, merges a maintainer-approved writable overlay and/or explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only that validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw-side limitation: source inspection on current main shows only `entry.config` is validated and passed to plugins. The sandbox write failure is issue-supplied and would require a NemoClaw/OpenShell environment to live-verify.

## Solution Assessment

Unclear for the exact proposal until maintainers choose the contract. The safest direction is core-owned overlay/env resolution before plugin activation and schema validation, rather than asking each plugin to read arbitrary `process.env` independently.
