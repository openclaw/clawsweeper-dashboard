## Summary

Keep open: current main still resolves, validates, and passes only the baked plugin entry config. The existing `$include` and `${VAR}` mechanisms are useful adjacent config features, but they do not provide the requested writable sibling overlay or plugin-declared env override path for policy-locked sandbox deployments.

## What This Changes

Requests a core plugin-config override mechanism so policy-locked sandboxes can change `plugins.entries.<id>.config` via declared env mappings and/or a writable overlay without rebuilding the image.

## Best Possible Solution

Add an additive core plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, merges a maintainer-approved writable overlay and/or explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the OpenClaw side: `entry.config` is the only value resolved, validated, and passed to plugins; the NemoClaw EACCES path is supplied by the issue discussion and would require that sandbox to live-run.

## Solution Assessment

Yes for the requested direction, but the exact model needs maintainer choice. A host-side overlay/env-resolution contract validated before plugin activation is safer than asking each plugin to read arbitrary `process.env` independently.
