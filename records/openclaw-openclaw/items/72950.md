## Summary

Keep open: current main still validates and exposes plugin config from the baked plugin entry config only, while the comments show this affects more than one policy-locked sandbox/plugin setup. The remaining work changes the plugin config contract and needs maintainer product/security direction rather than conservative cleanup closure.

## What This Changes

Requested declared env-var overrides and/or a writable overlay source for `plugins.entries.<id>.config` so policy-locked sandboxes can change plugin config without rebuilding images.

## Best Possible Solution

Define a generic, schema-validated plugin config resolver that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, then document precedence and locked-sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection shows plugin config is resolved, validated, and exposed from `plugins.entries.<id>.config` without a declared env override or writable overlay path. The external sandbox filesystem denial was not live-run here.

## Solution Assessment

Unclear for the exact implementation: env-only support does not fully cover the reporter's runtime reconfiguration loop, while overlay-only misses standard ecosystem env conventions. The narrowest maintainable direction is a generic resolver with documented precedence and validation semantics.
