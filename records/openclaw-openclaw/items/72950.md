## Summary

Keep open: current main still resolves, validates, and passes only the baked `plugins.entries.<id>.config` object. The issue discussion also shows this is broader than one observability deployment, with a +1 for policy-locked environments and a related openclaw-weixin report, so this remains a real plugin-config contract gap rather than stale support noise.

## What This Changes

Request an additive env-var override and/or writable overlay path for plugin config so policy-locked sandbox deployments can reconfigure `plugins.entries.<id>.config` without rebuilding images.

## Best Possible Solution

Add a generic plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, optionally merges a maintainer-approved writable overlay and explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the OpenClaw side: the only value resolved, validated, and passed to plugins is `entry.config`; the sandbox EACCES command is supplied in the issue discussion and would require NemoClaw to live-run.

## Solution Assessment

Unclear for the exact implementation, but the requested direction is appropriate. A host-side overlay and explicit env mapping contract validated before plugin activation is safer than asking every plugin to read arbitrary `process.env` independently.
