## Summary

Keep open: current main and the latest release still resolve `api.pluginConfig` from `plugins.entries.<id>.config` after normal config loading, without a plugin-declared environment override contract or writable overlay path. The discussion also broadens the impact beyond one observability/NemoClaw setup, including another policy-locked plugin deployment.

## What This Changes

The issue requests a core plugin-config override path, either through explicit env-var mappings, a writable overlay merged with `plugins.entries.<id>.config`, or documentation of the current build-time constraint.

## Best Possible Solution

Define a generic, schema-validated plugin config resolver with documented precedence for baked config, approved writable overlay, and explicit plugin-declared env mappings, then coordinate the writable path contract with policy-locked sandbox runtimes.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows plugin runtime config is resolved and validated from `plugins.entries.<id>.config` only. The external sandbox filesystem denial was not live-tested here.

## Solution Assessment

Unclear for the exact implementation. The requested direction is valid, but env-only support would not cover the reporter's full restart-without-rebuild loop, and overlay location, merge depth, precedence, validation, and trust boundaries need maintainer design first.
