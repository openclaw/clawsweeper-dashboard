## Summary

Keep open: current main still lacks a generic plugin-config overlay or explicit env-mapping path, the discussion corroborates this as a policy-locked deployment problem beyond one plugin, and no canonical open fix PR was found.

## What This Changes

Allow plugin config to be changed in policy-locked deployments without rebuilding, via explicit env overrides and/or a writable overlay for `plugins.entries.<id>.config`.

## Best Possible Solution

Add a generic plugin-config resolution layer that merges baked config, an approved host-owned writable overlay, and explicit plugin-declared env mappings, then validates the final object against the plugin manifest schema and documents precedence for policy-locked deployments.

## Reproduction Assessment

Yes for the OpenClaw side: current source inspection shows the only resolved, validated, and exposed plugin config path is `plugins.entries.<id>.config`. The external NemoClaw EACCES command is supplied in the issue discussion but was not live-run in this checkout.

## Solution Assessment

Unclear for the exact implementation. The requested direction is valid, but a host-controlled overlay plus explicit env mapping is safer than letting plugins read arbitrary process env or weakening the read-only sandbox policy.
