## Summary

Keep open: current main still has no host-side plugin config overlay or plugin-declared env override contract. The adjacent `$include` and `${VAR}` features are useful, but they still require config content to be present in the resolved config tree, which does not solve a read-only `openclaw.json` deployment; the discussion also shows the same pain outside the original observability/NemoClaw case.

## What This Changes

Request adding env-var override and/or writable overlay sourcing for `plugins.entries.<id>.config` so plugin config can be changed in policy-locked sandbox deployments without rebuilding the image.

## Best Possible Solution

Add an additive core plugin-config resolution layer that starts with baked `plugins.entries.<id>.config`, merges a maintainer-approved writable overlay and/or explicit plugin-declared env mappings, validates the final object against the plugin manifest schema before activation, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows `entry.config` is the only value resolved, validated, and passed to plugins. The NemoClaw EACCES runtime step is supplied in the issue discussion and would need that sandbox to live-run.

## Solution Assessment

Unclear for the exact implementation, but the requested direction is valid. A host-side overlay/env-resolution contract validated before plugin activation is safer than asking every plugin to read arbitrary `process.env`, while env-only support may not satisfy the restart-without-rebuild loop if sandbox env injection is also fixed at image creation.
