## Summary

Keep open: current main still has the reported plugin-config contract gap, and the discussion points to a broader policy-locked deployment problem rather than a one-off plugin workaround.

## What This Changes

Add a runtime-reconfigurable plugin config path, via plugin-declared env overrides and/or a writable overlay merged with `plugins.entries.<id>.config`, so policy-locked deployments do not require image rebuilds for config changes.

## Best Possible Solution

Add a generic plugin-config resolution layer that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, and documents precedence for policy-locked deployments.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows the only resolved, validated, and exposed plugin config is `plugins.entries.<id>.config`. The NemoClaw EACCES command is supplied in the issue discussion, but I did not live-run that external sandbox here.

## Solution Assessment

Unclear for the exact implementation. The direction is right, but a host-owned, schema-validated overlay plus explicit env mapping is safer than letting every plugin read arbitrary process env independently.
