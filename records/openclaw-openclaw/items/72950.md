## Summary

Keep open: current main still has a source-local plugin-config contract gap. Plugin config is resolved and validated from `plugins.entries.<id>.config`; there is generic `${ENV}` substitution for config strings, but no core-declared env mapping or writable overlay path that would let a policy-locked sandbox change plugin config without rebuilding or pre-baking placeholders.

## What This Changes

The issue requests a runtime-reconfigurable path for plugin config, preferably explicit env-var overrides and/or a writable overlay merged with `plugins.entries.<id>.config` for policy-locked sandboxes.

## Best Possible Solution

Add a generic plugin-config resolution layer that starts from baked `plugins.entries.<id>.config`, optionally merges a maintainer-approved writable overlay and explicit plugin-declared env mappings, validates the final object against the plugin manifest schema, exposes only the validated result through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence OpenClaw-side reproduction: the resolver and loader only consume `entry.config`; the live NemoClaw EACCES step is supplied in the issue discussion and is consistent with the missing core overlay path.

## Solution Assessment

Unclear for the exact design, but the requested direction is valid. A host-owned overlay plus explicit env mapping validated before plugin activation is safer than telling every plugin to read arbitrary `process.env` independently.
