## Summary

Keep open: the reported Telegram media failure is valid, but this PR should not merge as-is because it conflicts with current fallback metadata behavior and introduces an unthrottled warning path.

## What This Changes

The PR adds a Telegram user warning when `resolveMedia()` returns null and suppresses inbound media placeholders when no media attachments resolve.

## Best Possible Solution

Adapt the Telegram fix on current main so null media-resolution failures produce an accurate, rate-limited user-visible signal while preserving `<media:...> [file_id:...]` fallback metadata and covering media groups.

## Reproduction Assessment

Yes, from source inspection: `resolveTelegramFileWithRetry()` can return null, `resolveMedia()` propagates null, and current main then builds an empty `allMedia` list without a null-result warning. The remaining behavior is locked by current body-helper tests that preserve fallback metadata.

## Solution Assessment

No. The warning direction is useful, but suppressing placeholders is no longer the narrowest maintainable fix because current main intentionally preserves Telegram media metadata; the warning path also needs throttling and media-group coverage.
