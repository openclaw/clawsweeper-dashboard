## Summary

Keep this PR open, but it is not merge-ready. The Telegram null-media path still lacks a user-visible warning on current main, while this stale PR would conflict with newer fallback metadata behavior and introduces an unthrottled warning path.

## What This Changes

The PR adds a Telegram warning reply when inbound media resolution returns null and suppresses inbound media placeholders when no attachment was resolved.

## Best Possible Solution

Keep the fallback <media:...> [file_id:...] metadata, add an accurate rate-limited warning for null media resolution, and cover single-message plus media-group failures in Telegram plugin tests.

## Reproduction Assessment

Yes. A focused mocked Telegram handler test can make ctx.getFile() return a file-too-big failure or exhaust retries so resolveMedia() returns null; current source then continues with allMedia = [], and current body-helper tests cover the captioned-video fallback metadata behavior.

## Solution Assessment

No. The warning intent is useful, but suppressing placeholders whenever allMedia is empty conflicts with current fallback metadata, and the new warning path needs throttling plus album handling before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve fallback media metadata:** `src/telegram/bot-message-context.body.ts:141`
  - body: This change makes the media placeholder empty whenever allMedia is empty. Current main intentionally keeps <media:video> [file_id:...] for captioned failed downloads so the agent still knows media was present and can use the Telegram file_id; rebasing this as-is would drop that metadata and fail the current regression test.
  - confidence: 0.9
- **[P2] Handle all-failed media groups before dropping markers:** `src/telegram/bot-message-context.body.ts:141`
  - body: The new warning branch runs only after the media_group_id early return. If every item in an album resolves to null, suppressing the placeholder can leave the album with no user-visible failure signal; add album-level warning or fallback handling before removing the marker.
  - confidence: 0.86
- **[P2] Throttle repeated media-failure replies:** `src/telegram/bot-handlers.ts:1031-1036`
  - body: This branch sends a Telegram warning for every null media resolution in an allowed chat. A sender can repeatedly submit media that fails getFile() and make the bot spam the chat or consume Bot API send capacity; add a per-chat, per-sender, or media-group cooldown before sending the automatic warning.
  - confidence: 0.8
- **[P3] Use reason-neutral fallback warning text:** `src/telegram/bot-handlers.ts:1031`
  - body: resolveTelegramFileWithRetry() returns null for both Bot API size-limit failures and exhausted transient retries, but this message points mainly at the 20 MB limit. Use wording that also covers temporary download failures unless the failure reason is carried through explicitly.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but it introduces an unthrottled Telegram auto-reply path for media failures.

Concerns:

- **[medium] Unthrottled media-failure warnings can spam chats:** `src/telegram/bot-handlers.ts:1031`
  - body: Any permitted sender can repeatedly trigger null media resolution and cause the bot to send one warning reply per message, which can spam group chats or consume Telegram send capacity. Add cooldown, dedupe, or scope limits before merge.
  - confidence: 0.8
