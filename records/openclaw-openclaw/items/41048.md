## Summary

Keep open, but this PR is not merge-ready against current main. The underlying Telegram null-media failure is still source-reproducible, while the branch is stale against the current plugin layout and would now drop fallback media metadata introduced by a newer shipped Telegram fix.

## What This Changes

The PR adds a Telegram warning reply when media resolution returns null and suppresses media placeholders when no attachment was resolved.

## Best Possible Solution

Update or replace the Telegram plugin fix so null getFile failures produce accurate, rate-limited user-visible feedback while preserving fallback file_id metadata and handling all-failed media groups.

## Reproduction Assessment

Yes, source-reproducible. A mocked Telegram ctx.getFile() that throws a file-too-big error or exhausts retries makes resolveMedia() return null, and current main then proceeds with allMedia = [] outside the thrown-error warning path.

## Solution Assessment

No. The warning intent is useful, but the PR's placeholder suppression conflicts with current fallback metadata, misses all-failed albums, and introduces an unthrottled auto-reply path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve fallback media metadata:** `src/telegram/bot-message-context.body.ts:141`
  - body: Conditioning the placeholder on allMedia.length drops the current fallback '<media:video> [file_id:...]' metadata for captioned failed downloads. Current main has a regression test for that metadata, so rebasing this as-is would hide useful Telegram file_id context from the agent.
  - confidence: 0.9
- **[P2] Warn when an entire media group fails:** `src/telegram/bot-message-context.body.ts:141`
  - body: The new warning branch only runs after media_group_id messages have already returned into the album buffer. If every album item resolves to null and the placeholder is suppressed, the album can be dropped without either an agent-visible marker or a user-visible warning.
  - confidence: 0.86
- **[P2] Throttle repeated media-failure replies:** `src/telegram/bot-handlers.ts:1031-1036`
  - body: This branch sends a Telegram warning for every null media resolution in an allowed chat. A sender can repeatedly submit media that fails getFile() and make the bot spam the chat or consume Telegram send capacity, so the warning needs per-chat, per-sender, or media-group cooldown/dedupe.
  - confidence: 0.82
- **[P3] Use reason-neutral warning text:** `src/telegram/bot-handlers.ts:1031`
  - body: resolveTelegramFileWithRetry() returns null both for the Bot API file-size limit and for exhausted transient retries. The fallback warning should either carry the failure reason through or use wording that covers temporary download failures too.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but it introduces an unthrottled Telegram auto-reply path for media failures.

Concerns:

- **[medium] Unthrottled media-failure warnings can spam chats:** `src/telegram/bot-handlers.ts:1031`
  - body: Any permitted sender can repeatedly trigger null media resolution and cause the bot to send one warning reply per message, which can spam group chats or consume Telegram Bot API send capacity. Add cooldown, dedupe, or scope limits before merge.
  - confidence: 0.82
