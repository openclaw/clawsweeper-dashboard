## Summary

Keep open; the PR targets a valid Telegram null-media failure but is not merge-ready because current main now preserves fallback file_id metadata that this patch would remove, media groups still need failure handling, and the new warning path is unthrottled.

## What This Changes

The PR branch adds a Telegram warning reply when media resolution returns null and suppresses placeholder-only inbound media text when no attachment was resolved.

## Best Possible Solution

Rework the Telegram plugin fix so null media-resolution failures produce an accurate, rate-limited user-visible signal while preserving current fallback placeholder/file_id metadata for single messages and all-failed media groups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock Telegram getFile to hit the file-too-big or retry-exhausted branch, resolveMedia() returns null, and the current handler continues with allMedia = [] while body-helper tests cover the fallback metadata behavior.

## Solution Assessment

No. The warning direction is useful, but suppressing placeholders conflicts with current shipped fallback metadata and the warning path needs throttling plus media-group handling before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve fallback media metadata:** `src/telegram/bot-message-context.body.ts:141`
  - body: Current main intentionally keeps `<media:video> [file_id:...]` for captioned failed downloads so the agent still knows media was present and can use the Telegram file_id. Rebasing this suppression as-is would drop that metadata and fail the current body-helper regression test.
  - confidence: 0.9
- **[P2] Handle all-failed media groups before dropping markers:** `src/telegram/bot-message-context.body.ts:141`
  - body: Media-group updates return before the new single-message warning branch, so an album where every item resolves to null can lose the placeholder signal without any user-visible failure. Add album-level fallback or warning handling before removing markers.
  - confidence: 0.86
- **[P2] Throttle repeated media-failure replies:** `src/telegram/bot-handlers.ts:1031-1036`
  - body: This branch sends a Telegram warning for every null media resolution in an allowed chat. A permitted sender can repeatedly submit media that fails getFile() and make the bot spam the chat or consume Bot API send capacity; add per-chat, per-sender, or media-group cooldown before sending.
  - confidence: 0.82
- **[P3] Use reason-neutral fallback warning text:** `src/telegram/bot-handlers.ts:1031`
  - body: resolveTelegramFileWithRetry() returns null for both Bot API size-limit failures and exhausted transient retries, but this message points mainly at the 20 MB limit. Use wording that also covers temporary download failures unless the failure reason is carried through explicitly.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but it introduces an unthrottled Telegram auto-reply path for media failures.

Concerns:

- **[medium] Unthrottled media-failure warnings can spam chats:** `src/telegram/bot-handlers.ts:1031`
  - body: Any permitted sender can repeatedly trigger null media resolution and cause the bot to send one warning reply per message, which can spam group chats or consume Telegram send capacity. Add cooldown, dedupe, or scope limits before merge.
  - confidence: 0.82
