## Summary

This PR should stay open, but not merge as-is: current main already preserves Telegram fallback media metadata via #59948, while this PR would drop that signal, and the added user warning needs cooldown, more accurate wording, and media-group coverage.

## What This Changes

The PR adds a Telegram warning when media resolution returns null and suppresses inbound media placeholders when no attachments resolve.

## Best Possible Solution

Adapt the Telegram fix on current main so null media-resolution failures produce an accurate, rate-limited user-visible signal while preserving `<media:...> [file_id:...]` fallback metadata and covering media groups.

## Reproduction Assessment

Yes, from source inspection: `resolveTelegramFileWithRetry()` can return null, `resolveMedia()` propagates null, and the current single-message handler has no null-result warning branch. The metadata fallback behavior is covered by `bot-message-context.body.test.ts`.

## Solution Assessment

No. The warning direction is useful, but suppressing placeholders is no longer the narrowest maintainable fix because current main intentionally preserves Telegram media metadata; the warning path also needs throttling and media-group coverage.
