## Summary

Keep open: the PR addresses a real Telegram media-ingest bug, but the patch is stale against current main and would regress current fallback metadata while adding an unthrottled auto-reply path.

## What This Changes

The PR adds a Telegram warning when media resolution returns null and suppresses generated media placeholders when no attachment resolves.

## Best Possible Solution

Adapt the Telegram fix on current main so null getFile failures produce an accurate, rate-limited user-visible warning while preserving metadata fallback and covering media groups.

## Reproduction Assessment

Yes, from source inspection: `resolveTelegramFileWithRetry()` can return null, `resolveMedia()` propagates null, and the current handler then proceeds with `allMedia = []` without a null-result warning. Current tests also prove fallback metadata is intentionally preserved for captioned failed downloads.

## Solution Assessment

No. The warning direction is useful, but suppressing placeholders is not the best current-main fix because it drops fallback file_id context, misses all-failed media groups, and lacks warning throttling.
