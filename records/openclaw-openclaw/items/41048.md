## Summary

Keep this PR open. It targets a real Telegram null media-resolution gap, but the branch is not safe to merge as-is because current main has since shipped placeholder plus `file_id` preservation for failed captioned media, while this diff suppresses that metadata, misses media-group failures, and adds unthrottled warning replies.

## What This Changes

The PR adds a Telegram warning reply when inbound media resolution returns null and changes body construction to omit media placeholders when no attachment was resolved.

## Best Possible Solution

Update or replace the Telegram plugin change so null media-resolution failures produce an accurate, rate-limited user-visible failure signal while preserving useful caption and `file_id` metadata for the agent, including media-group cases.

## Reproduction Assessment

Yes. A focused mocked Telegram handler test can make `ctx.getFile()` return the file-too-big failure or exhaust retries so `resolveMedia()` returns `null`; current source then reaches `allMedia = []`, and the body helper behavior is covered by the captioned-video test.

## Solution Assessment

No. The warning intent is useful, but suppressing placeholders when `allMedia` is empty conflicts with shipped metadata behavior and leaves albums plus repeated warning spam unresolved; the safer fix is a throttled, album-aware warning path that preserves fallback metadata.

## Security Review

Status: needs_attention
