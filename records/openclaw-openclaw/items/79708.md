## Summary

Keep open: current main still has a narrow memory-core cleanup gap that matches the fresh Windows `EBUSY` unlink report. The code retries transient rename failures during atomic swaps, but temp/backup SQLite cleanup still removes base/WAL/SHM files once and in parallel.

## What This Changes

Requests that `openclaw memory index --force` tolerate transient Windows locks during atomic SQLite reindex cleanup instead of failing and leaving temp files.

## Best Possible Solution

Teach memory-core atomic reindex cleanup to use transient-lock retry/backoff for temp and backup SQLite sidecars, remove sidecars sequentially, and add focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, but not live-reproduced here: `removeMemoryIndexFiles` currently propagates `fs.rm` `EBUSY`/`EPERM`/`EACCES` from temp SQLite sidecars, matching the reporter's Windows logs.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrowest fix is in `extensions/memory-core/src/memory/manager-atomic-reindex.ts`, sharing transient-lock resilience with cleanup rather than changing Ollama/provider behavior or adding configuration.
