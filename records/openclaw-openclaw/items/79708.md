## Summary

Close as implemented: current main now hardens memory-core atomic reindex cleanup for the reported Windows temp SQLite `EBUSY` path, with retry/backoff, sequential sidecar removal, pre-cleanup DB close handling, regression tests, and a changelog entry. The fix is on current main but not yet proven in a shipped release tag.

## What This Changes

Make `openclaw memory index --force` tolerate transient Windows locks on temporary SQLite, WAL, and SHM files during atomic reindex cleanup.

## Best Possible Solution

Keep the current memory-core cleanup implementation on main, close this temp-file cleanup bug, and leave the broader persistent Windows `EPERM` rename/copy fallback tracked separately at https://github.com/openclaw/openclaw/issues/78640.

## Reproduction Assessment

Yes for the original bug: reporter logs and the former source path showed cleanup `fs.rm` failures propagating from temp SQLite sidecars. Current main no longer has the no-retry cleanup path, so I did not live-reproduce a failing Windows run.

## Solution Assessment

Yes. The current main fix is the narrow owner-local solution: close temp DB handles before cleanup, retry transient file removal errors, and remove sidecars sequentially without changing providers or adding configuration.
