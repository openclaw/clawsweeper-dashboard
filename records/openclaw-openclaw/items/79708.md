## Summary

Close as implemented on current main: the memory-core atomic reindex cleanup path now retries transient Windows file-lock errors while removing temporary SQLite files, removes sidecars sequentially, and has focused regression coverage plus an Unreleased changelog entry.

## What This Changes

The issue reports that `openclaw memory index --force` can fail on Windows when temporary SQLite index files are removed while still locked.

## Best Possible Solution

Keep the current memory-core implementation on main and let the next release ship the fix; broader Windows SQLite lock hardening can remain in narrower follow-up reports if it reproduces separately.

## Reproduction Assessment

Yes for the historical bug: the reporter supplied Windows `EBUSY` cleanup logs, and the earlier source path removed temp SQLite base/WAL/SHM files without transient-lock retry. I did not live-reproduce the failure on Windows in this read-only pass, and current main now contains the fix.

## Solution Assessment

Yes. The current implementation is the narrow owner-module fix: close temp SQLite handles, retry transient cleanup file errors, remove sidecars sequentially, and cover the behavior without provider or config changes.
