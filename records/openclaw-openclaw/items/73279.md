## Summary

Keep open. The PR addresses a real duplicate Discord auto-thread reply problem, but the current diff is unsafe because it gives unconditional final-response guidance while current main defaults Discord guild/channel turns to message-tool-only delivery and suppresses automatic final delivery in that mode. It also lacks the required user-facing changelog entry.

## What This Changes

The PR changes Codex app-server developer guidance and the bundled Discord skill to prefer final assistant responses for normal current-conversation replies, and adds a thread-lifecycle unit test for that wording.

## Best Possible Solution

Revise the prompt and skill wording so it follows the resolved source reply delivery mode: final-response guidance for automatic delivery, message-tool guidance for private/tool-only Discord channel turns, focused regression coverage for both modes, and a single-line changelog entry for the user-visible fix.

## Reproduction Assessment

Yes. The original duplicate path is documented by #73278 for Discord `autoThread: true`, and the PR-introduced regression can be reproduced by tracing a default Discord guild turn to `sourceReplyDeliveryMode: "message_tool_only"`, where dispatch suppresses automatic final delivery.

## Solution Assessment

No. The PR points in the right direction for automatic auto-thread delivery, but unconditional final-response wording is not the narrowest maintainable fix because current default Discord guild/channel delivery still requires `message(action=send)` for visible output.
