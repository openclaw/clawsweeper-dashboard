## Summary

Keep open: the branch is a plausible fix for the GBK read bug, but it still needs changes before merge because it duplicates the shared Windows decoder, mishandles standard TextDecoder labels, and the current proof covers only the explicit override rather than the latest default fallback.

## What This Changes

The PR adds an optional read-tool `encoding` argument, a default Windows console-codepage fallback for non-UTF-8 text, GBK regression coverage, and a repro script.

## Best Possible Solution

Use the shared `decodeWindowsOutputBuffer` for the default read path, either remove the explicit `encoding` API or keep it with intact TextDecoder labels and maintainer-owned API intent, then add current-head proof for the default Windows path.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` decode non-image read buffers as UTF-8, while the linked GBK byte examples are invalid UTF-8 and mojibake under that path. I did not run a live Chinese Windows filesystem repro in this read-only review.

## Solution Assessment

No. The default fallback direction is right, but this implementation copies the shared decoder, has a concrete explicit-label bug, and needs maintainer confirmation on whether the new `encoding` tool API should ship.

Full review comments:

- **[P1] Call the shared Windows decoder:** `src/agents/sessions/tools/read.ts:277-286`
  - body: The no-encoding branch reimplements the strict UTF-8 -> Windows console codepage -> UTF-8 fallback even though `src/infra/windows-encoding.ts` already exports and tests `decodeWindowsOutputBuffer` for that invariant.   - confidence: 0.86

  - body: This normalization strips underscores before passing the label to `TextDecoder`, so a standard explicit value like `shift_jis` becomes unsupported `shiftjis` and the read call fails. Keep supported labels intact apart from trim/lowercase, or map aliases explicitly, before exposing a free-form encoding parameter.
  - confidence: 0.88
