## What This Changes

The PR adds an optional `encoding` parameter to the built-in read tool, decodes explicit non-UTF-8 reads with `iconv-lite`, updates wrapper argument cleanup, adds GBK test/repro coverage, and changes root/plugin dependency metadata.

## Best Possible Solution

Prefer a dependency-free default Windows codepage fallback using the existing in-tree decoder; keep an explicit `encoding` API only if maintainers align the public contracts and approve the dependency/security direction.

## Solution Assessment

No as written: the explicit parameter is a plausible mitigation, but this branch should not merge with the blocked dependency graph, failed shrinkwrap check, and stale exported contract. The dependency-free Windows fallback is the safer default unless maintainers deliberately choose the new API.
