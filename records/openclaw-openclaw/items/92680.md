## Summary

Keep open: this is a useful attempt at the GBK read-tool bug, but the branch is not merge-ready because it adds a blocked external dependency graph change, leaves the exported read-tool contract stale, and competes with a narrower in-tree decoder approach. Current main and the latest release still decode read-tool text as UTF-8, so the underlying user problem remains real.

## What This Changes

This PR adds an optional `encoding` parameter to the built-in read tool, decodes explicit non-UTF-8 reads with `iconv-lite`, updates argument cleanup, adds GBK test/repro coverage, and changes dependency/shrinkwrap files.

## Best Possible Solution

Prefer the existing dependency-free Windows codepage-aware decoder for the default read text path; keep an explicit `encoding` API only if maintainers deliberately align the public contracts and approve the dependency direction.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` decode non-image read buffers as UTF-8, while the linked GBK byte examples mojibake under that path. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the explicit `encoding` parameter is a plausible mitigation, but this PR should not merge with the blocked dependency graph, failing shrinkwrap check, and stale exported contract. The narrower dependency-free default fallback is the safer current direction unless maintainers choose the new API deliberately.
