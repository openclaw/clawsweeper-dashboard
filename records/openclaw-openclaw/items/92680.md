## Summary

Keep open: current main and the latest release still have the GBK read-tool problem, but this PR is not merge-ready because it adds a blocked external dependency, includes unrelated shrinkwrap drift, and exposes a new read-tool argument without aligning the exported tool-call contract.

## What This Changes

The PR adds an optional `encoding` argument to the read tool, decodes explicit non-UTF-8 reads with `iconv-lite`, normalizes that argument in the OpenClaw read wrapper, adds GBK regression/repro coverage, and updates dependency manifests/lockfiles.

## Best Possible Solution

Reuse the existing Windows codepage-aware decoder for the default read text path, and keep an explicit encoding API only if maintainers deliberately align the public contract and dependency approval path.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode non-image read buffers as UTF-8, while the linked report provides GBK bytes that mojibake under that path. I did not run a live Chinese Windows filesystem repro in this read-only review.

## Solution Assessment

No as written: the explicit `encoding` parameter proves a possible mitigation, but the blocked dependency graph, unrelated shrinkwrap drift, and stale exported contract make this the wrong merge shape. The safer default repair is the in-tree Windows codepage-aware decoder unless maintainers intentionally choose a new API.
