## Summary

Keep open: the linked GBK read bug is still present on current main and this PR contains a useful focused approach, but the current head is not merge-ready because the new model-facing `encoding` input is not aligned with existing read argument repair and it breaks valid TextDecoder labels such as `shift_jis`.

## What This Changes

The PR adds an optional `encoding` string to the built-in read tool schema/type, decodes text buffers with `TextDecoder` when supplied, and adds GBK regression/repro coverage.

## Best Possible Solution

Merge an explicit read `encoding` API only after valid TextDecoder labels and read argument repair are aligned; otherwise prefer the existing Windows console-codepage fallback path for default reads.

## Reproduction Assessment

Yes, source-level: current main decodes text read buffers as UTF-8, while the linked issue supplies GBK bytes that mojibake under that path. I did not run a live Chinese Windows filesystem session in this read-only review.

## Solution Assessment

No, not as submitted: explicit encoding is a plausible supported path from the issue, but the implementation must align argument repair and label handling before it is the best mergeable fix. The existing Windows codepage fallback remains a viable alternative for default reads.
