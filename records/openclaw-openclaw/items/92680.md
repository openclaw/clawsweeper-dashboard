## Summary

Keep open: the branch targets a real open read-tool bug that is still present on current main and the latest release, but the current patch needs a small encoding-label correctness fix and real proof for the default Windows codepage fallback before merge.

## What This Changes

The PR adds an optional `encoding` argument to the built-in read session tool, changes default text decoding to strict UTF-8 with a Windows console-codepage fallback, updates the read input contract, and adds GBK test/repro coverage.

## Best Possible Solution

Land a narrow read-tool fix that reuses the existing Windows codepage-aware decoder for default reads, keeps any explicit encoding override only with correct `TextDecoder` label handling and aligned contracts, and includes GBK regression plus real Windows/default-flow proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` convert non-image read buffers with UTF-8, while the reported GBK byte pattern requires a legacy decoder. I did not run a live Chinese Windows read session in this read-only review.

## Solution Assessment

No as written: the default Windows codepage fallback is the right layer, but the explicit encoding API needs correct label handling, clearer schema text, and real default-flow proof. If maintainers do not want the new API, the safer alternative is a fallback-only read-tool patch using the existing helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve TextDecoder labels with underscores:** `src/agents/sessions/tools/read.ts:261`
  - body: The new explicit encoding path strips every character except letters, numbers, and hyphens before calling `TextDecoder`. That turns a supported label like `shift_jis` into `shiftjis`, which Node rejects, so explicit reads for that common legacy encoding fail even though the platform decoder supports it. Let `TextDecoder` handle labels directly after trimming/case normalization, or add a narrow alias map instead of deleting underscores.
  - confidence: 0.88
- **[P3] Describe the actual default decoding path:** `src/agents/sessions/tools/read.ts:45-46`
  - body: The schema tells the model that `encoding` defaults to UTF-8, but the no-encoding path now tries strict UTF-8 and then the Windows console codepage. Update this text so models can rely on the automatic fallback instead of unnecessarily supplying an override.
  - confidence: 0.78
