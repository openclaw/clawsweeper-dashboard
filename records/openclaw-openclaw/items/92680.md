## Summary

## What This Changes

## Best Possible Solution

Keep the read API unchanged, use the shared Windows codepage decoder on the read path, and land one canonical PR only after read-tool-level GBK/UTF-8 regression coverage and real or maintainer-approved equivalent proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the reported GBK bytes mojibake under that path while decoding correctly as GBK. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

Partly: reusing decodeWindowsOutputBuffer and keeping the read API unchanged is the right implementation layer, but the current test/proof is not the best merge-ready validation because it does not assert correct Windows GBK output.
