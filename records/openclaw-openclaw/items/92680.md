## What This Changes

This PR adds an optional read-tool `encoding` parameter, decodes explicit non-UTF-8 text reads with `TextDecoder`, updates `ReadToolInput`, and adds GBK test/repro coverage.

## Best Possible Solution

Reuse the existing strict-UTF-8-then-Windows-codepage decoder for the default read text path, and keep an explicit `encoding` parameter only if maintainers deliberately want the new API with label-safe decoding and focused tests.

## Reproduction Assessment

## Solution Assessment

No: the explicit `encoding` parameter is a plausible mitigation, but it is not the narrowest default fix for the linked Windows workflow. The existing Windows codepage-aware decoder can solve the default read path without requiring user or model opt-in.
