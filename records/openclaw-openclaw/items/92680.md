## Summary

Keep open: the dependency issue is resolved, but the latest head still needs maintainer API/fix-shape review because it adds a model-facing read `encoding` parameter while duplicating the existing Windows codepage decoder and lacks real proof for the default Chinese Windows read path.

## What This Changes

The PR adds an optional read-tool `encoding` input, a strict-UTF-8 then Windows-codepage fallback, GBK-focused tests, and a standalone repro script.

## Best Possible Solution

Route default text reads through the existing Windows codepage-aware decoder or a shared file-decoding helper, add an asserting GBK no-encoding regression path, and expose explicit `encoding` only if maintainers accept that API.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still force read-tool text buffers through UTF-8, while the reported GBK bytes decode to mojibake under UTF-8 and readable Chinese through `TextDecoder('gbk')`. I did not establish a live Chinese Windows read-tool reproduction in this read-only review.

## Solution Assessment

No, not yet: reusing the existing Windows decoder for the default path is narrower than duplicating it in `read.ts`, and the explicit `encoding` API needs maintainer direction. The current branch is a plausible fix, but not the best maintainable shape as submitted.
