## Summary

Keep this PR open, but it is not merge-ready: the useful GBK read-tool fix is still needed, yet this branch duplicates the shared Windows decoder policy and adds a new model-facing `encoding` API that needs maintainer direction.

## What This Changes

The PR adds an optional read-tool `encoding` parameter, changes text decoding to strict UTF-8 with Windows codepage fallback, updates the read input type, and adds GBK tests plus a repro script.

## Best Possible Solution

Use one canonical read text decode path that reuses the shared Windows output decoder for default behavior, and keep the explicit encoding override only if maintainers approve the new read-tool API surface.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` read non-image file buffers with `buffer.toString("utf-8")`, which turns the reported GBK byte examples into mojibake. I did not run a live Chinese Windows filesystem session in this read-only review.

## Solution Assessment

No as currently shaped: the default Windows fallback is the right repair direction, but duplicating the shared decoder policy is not the best implementation, and the explicit `encoding` parameter needs maintainer approval as a new tool API.
