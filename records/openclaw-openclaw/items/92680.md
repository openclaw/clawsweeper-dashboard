## Summary

Keep open: the linked GBK read bug is still present on current main, but this PR is not merge-ready because it adds a blocked root dependency and expands the read-tool API without aligning the exported contract.

## What This Changes

The PR adds an optional `encoding` argument to the built-in read tool, decodes text with `iconv-lite`, normalizes the new argument in the OpenClaw wrapper, and adds GBK regression/repro coverage plus root dependency entries.

## Best Possible Solution

Reuse the existing dependency-free Windows codepage-aware decoder for the read text path, and keep an explicit encoding API only if maintainers deliberately align its exported contract and approval path.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode non-image read buffers as UTF-8, while the linked report provides GBK bytes that mojibake under that path. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the explicit `encoding` parameter is a plausible mitigation, but the PR should not merge with the blocked dependency graph or stale exported contract. The safer default repair is to reuse the existing Windows codepage-aware decoder unless maintainers choose the new API deliberately.
