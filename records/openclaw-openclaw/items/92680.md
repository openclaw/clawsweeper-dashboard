## Summary

Keep open: the linked GBK read bug is still present on current main and in v2026.6.6, and this PR contains useful proof, but the branch is not merge-ready because it adds a blocked root dependency graph change and leaves the new read-tool API contract stale.

## What This Changes

The PR adds an optional `encoding` argument to the read session tool, decodes non-UTF-8 buffers with `iconv-lite`, normalizes malformed XML suffixes for that argument, adds GBK regression/repro coverage, and changes the root dependency graph.

## Best Possible Solution

Reuse or extend the existing dependency-free Windows codepage-aware decoder for the read-tool text path, remove the root dependency graph change, and keep an explicit encoding API only if maintainers deliberately align the exported contract and tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 decode non-image read buffers with UTF-8, while the linked issue provides GBK bytes that mojibake under that path. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the explicit `encoding` parameter is a plausible mitigation, but this PR should not merge with the blocked dependency graph or stale exported contract. The safer default repair is to reuse the existing Windows codepage-aware decoding path unless maintainers choose the new API deliberately.
