## What This Changes

The PR updates the built-in read session tool to add an optional `encoding` argument, decode explicit non-UTF-8 reads with `TextDecoder`, and fall back to the Windows console codepage when default UTF-8 decoding fails.

## Best Possible Solution

Land one canonical dependency-free read fix: preserve valid UTF-8, decode GBK on Chinese Windows through the existing codepage pattern, and keep the explicit encoding override only if maintainers accept that model-facing API with valid label handling.

## Reproduction Assessment

## Solution Assessment

Not yet: the default Windows fallback direction is reasonable, but the explicit encoding API needs maintainer acceptance and the current label normalization breaks valid encoding labels.
