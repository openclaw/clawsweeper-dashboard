## Summary

Keep open: this is useful work for the linked GBK read-tool bug, but the patch is not merge-ready because it adds a blocked dependency graph change even though current Node/OpenClaw code already has GBK-capable decoding primitives, and the new read parameter is not reflected in the exported input contract.

## What This Changes

The PR adds an optional `encoding` parameter to the built-in read tool, decodes text buffers with `iconv-lite`, normalizes malformed XML suffixes for that argument, adds GBK regression/repro coverage, and declares `iconv-lite` as a direct dependency.

## Best Possible Solution

Use the existing dependency-free `TextDecoder`/Windows-codepage path for GBK decoding, remove the package graph change, and keep any explicit `encoding` parameter only if its exported contract and tests are aligned.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and GBK bytes mojibake under that path. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the explicit `encoding` parameter is a plausible mitigation, but the best mergeable shape should avoid the blocked dependency graph and align the exported read input contract. Maintainers should also decide whether this should instead reuse the existing default Windows codepage fallback.
