The PR adds an optional `encoding` parameter to the built-in read tool, adds GBK regression/repro coverage, and changes default text reads to try strict UTF-8 before a Windows console-codepage fallback.

## Best Possible Solution

Route the default read text path through the shared Windows decoder, keep valid UTF-8 unchanged, decide whether the explicit encoding override belongs in the tool API, and prove GBK-on-Windows or an equivalent no-encoding fallback path without adding dependencies.

## Solution Assessment

No: the branch is close, but the best fix should reuse the existing exported Windows decoder for the default path instead of duplicating it, and maintainers still need to decide whether the explicit `encoding` override belongs in the tool API.
