The PR adds an optional `encoding` argument to the built-in read tool, decodes text buffers with `iconv-lite`, updates read-argument cleanup, adds GBK test/repro coverage, and changes package dependencies.

## Best Possible Solution

Use the existing strict-UTF-8 then Windows-codepage decoder in the read text path, remove `iconv-lite` and the new read API unless maintainers explicitly approve them, and add focused GBK regression coverage plus real OpenClaw proof.

No: the explicit `encoding` parameter is a plausible mitigation, but the best current fix is the narrower in-tree Windows codepage fallback unless maintainers intentionally approve a broader read-tool API.
