## Summary

Keep open: the linked read-tool bug is still present on current main and this branch contains useful proof, but it cannot merge as-is because it changes the root dependency graph and adds a model-facing read argument without the shared contract and dependency approval settled.

## What This Changes

The PR adds an optional `encoding` argument to the built-in read tool, decodes text buffers with `iconv-lite`, normalizes that argument in the OpenClaw wrapper, adds GBK-focused coverage, and updates dependency files.

## Best Possible Solution

Use the existing dependency-free Windows codepage-aware decoder for the default read text path, and keep an explicit encoding API only if maintainers deliberately align its shared contract and approve the dependency path.

## Reproduction Assessment

Yes. Current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the linked report supplies GBK bytes that mojibake under that path; I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as submitted. The explicit parameter is a plausible mitigation, but the dependency graph and exported contract are unresolved, and the in-tree Windows decoder is the narrower default repair for the reported Windows flow unless maintainers choose a new API deliberately.
