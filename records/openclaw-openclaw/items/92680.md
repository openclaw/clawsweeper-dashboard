## Summary

Keep open: the PR is a useful explicit-encoding approach for the linked GBK read bug, but the current patch still has two concrete blockers before merge: it breaks valid TextDecoder labels and leaves the read tool-call repair path unaware of the new argument.

## What This Changes

The PR adds an optional `encoding` input to the built-in read tool, decodes text buffers with `TextDecoder`, updates the read input contract, and adds GBK regression/repro coverage.

## Best Possible Solution

Merge the explicit encoding API only after valid TextDecoder labels and read argument repair are aligned; otherwise prefer the existing Windows codepage fallback path for default reads.

## Reproduction Assessment

Yes, source-level: current main decodes read-tool text buffers as UTF-8, and the linked GBK bytes mojibake under that path. The PR proof shows explicit `encoding: "gbk"` decodes those bytes, but no live Chinese Windows filesystem run was performed in this review.

## Solution Assessment

No, not as submitted: explicit encoding is a plausible supported path from the issue, but the implementation must align argument repair and label handling before it is mergeable. The existing Windows codepage fallback remains a viable alternative for default reads.
