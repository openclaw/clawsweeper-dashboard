## Summary

Keep open: current main and the latest release still have the UTF-8-only read bug, and this PR is an active useful fix attempt, but the latest branch is not merge-ready because the explicit encoding path corrupts valid TextDecoder labels and the supplied proof does not exercise the newly added default Windows fallback path.

## What This Changes

This PR adds an optional read-tool `encoding` argument, changes text reads to strict UTF-8 with Windows codepage fallback, updates the read input contract and tests, and adds a GBK repro script.

## Best Possible Solution

Land one dependency-free read decoding path that preserves valid TextDecoder labels, reuses or stays aligned with the existing Windows codepage decoder, and includes focused proof for both explicit GBK and default Windows fallback behavior.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, while the reported GBK bytes require GBK/TextDecoder handling to avoid mojibake. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the dependency-free direction is appropriate, but the implementation should preserve TextDecoder labels and prove the latest default Windows fallback path before merge. Maintainers also need to decide whether the explicit `encoding` API should remain or whether the canonical fix should be default fallback only.
