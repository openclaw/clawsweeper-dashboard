## Summary

Keep open: the linked GBK read bug is real and still present on current main and v2026.6.6, but this PR is not merge-ready because it adds a blocked direct dependency, creates a new model-facing read API without updating shared contracts, and overlaps with several unmerged sibling fixes that require a maintainer direction choice.

## What This Changes

The PR adds an optional `encoding` parameter to the read session tool, decodes text buffers with `iconv-lite`, normalizes that argument in the wrapper, adds GBK-focused regression/repro coverage, and updates root dependency files.

## Best Possible Solution

Use the existing dependency-free Windows codepage-aware decoder for the default read text path, and keep an explicit encoding API only if maintainers deliberately align its shared contract and approve the dependency or dependency-free decoding path.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the linked report supplies GBK bytes that mojibake under that path. I did not run a live Chinese Windows filesystem proof in this read-only review.

## Solution Assessment

No as written: the explicit parameter is a plausible mitigation, but the dependency graph and exported contract are unresolved, and the in-tree Windows decoder is the narrower default repair for the reported Windows flow unless maintainers choose a new API deliberately.
