## Summary

Keep open, but this patch is not merge-ready: it proves explicit `encoding: "gbk"` reads, while the reported Chinese Windows default read path still needs the existing Windows codepage-aware fallback wired into the read tool.

## What This Changes

The branch adds an optional `encoding` argument to the built-in session read tool, decodes explicit non-UTF-8 text with `TextDecoder`, updates the exported read input type, and adds GBK test/repro coverage.

## Best Possible Solution

Wire the read text path through the existing strict-UTF-8-then-Windows-codepage decoder for default reads; consider an explicit encoding parameter only as a separate maintainer-approved API addition.

## Reproduction Assessment

Yes at source level: current main decodes read text buffers as UTF-8, and a byte probe for the reported GBK sample produced mojibake under UTF-8 and readable text under GBK. I did not run a live Chinese Windows read session.

## Solution Assessment

No: explicit encoding is a plausible new API, but it is not the best bug fix for ordinary Chinese Windows reads. The narrower maintainable solution is to reuse the existing Windows codepage-aware decoder on the default read path.
