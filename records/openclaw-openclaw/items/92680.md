## Summary

Keep open: current main and v2026.6.6 still use UTF-8-only decoding for read-tool text, so the linked GBK bug is not implemented on main. The latest PR head removed the earlier dependency and model-facing API blockers and now uses the shared Windows decoder, but its added test and repro only prove non-empty output rather than correct Chinese Windows codepage decoding, and the PR body proof is stale for the current diff.

## What This Changes

The branch wires read-tool text decoding through `decodeWindowsOutputBuffer`, adds a GBK-byte regression test, and adds a standalone repro script.

## Best Possible Solution

Keep the read API unchanged, use the shared Windows decoder in the read text path, and require proof or focused coverage that no-encoding GBK reads return Chinese on a Chinese Windows/codepage-936 path while UTF-8 remains unchanged.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the issue's GBK byte sample mojibakes under UTF-8 while decoding correctly with GBK. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

Mostly yes: reusing `decodeWindowsOutputBuffer` is the right owner-boundary fix and avoids a new read API. The current branch is not merge-ready until the tests/proof assert the Windows codepage output instead of only non-empty text.
