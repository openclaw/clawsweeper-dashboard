## Summary

Keep open: the linked read-tool bug is still present on current main and the PR contains useful work, but it is not merge-ready because it adds a blocked dependency graph change and a model-facing read parameter without aligning the shared contract.

## What This Changes

The PR adds an optional read-tool `encoding` parameter, GBK decoding through `iconv-lite`, malformed-argument cleanup for that parameter, GBK regression/proof coverage, and root lock/shrinkwrap dependency changes.

## Best Possible Solution

Resolve the bug through one canonical read-decoding design: preferably reuse the existing dependency-free Windows codepage fallback for the default read path, or deliberately ship the explicit `encoding` API with aligned contracts and approved dependency/security handling.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and a Node byte probe shows the reported GBK bytes mojibake under UTF-8 but decode correctly as GBK. I did not run a live Chinese Windows filesystem session.

## Solution Assessment

No as written: the explicit `encoding` parameter is a plausible mitigation, but this PR should not merge with a blocked dependency graph and stale shared contract. The safer default repair is the existing Windows codepage-aware fallback unless maintainers deliberately choose the new API.
