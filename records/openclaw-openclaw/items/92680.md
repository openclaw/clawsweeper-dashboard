## Summary

Keep open: this PR targets a real read-tool bug that is still present on current main and the latest release, but the branch is not merge-ready because it duplicates the existing Windows codepage decoder and the new explicit encoding override rewrites valid TextDecoder labels into invalid ones.

## What This Changes

The PR adds an optional `encoding` input to the read session tool, decodes explicit non-UTF-8 reads with `TextDecoder`, and adds a default invalid-UTF-8 fallback through the Windows console codepage path with tests and a repro script.

## Best Possible Solution

Converge this branch or a sibling on the existing `decodeWindowsOutputBuffer` default path, add focused default GBK read coverage/proof, and keep the explicit `encoding` parameter only if maintainers approve that API surface.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` decode non-image read buffers as UTF-8, which mojibakes the GBK bytes from the linked report. I did not run a live Chinese Windows read session in this read-only review.

## Solution Assessment

No as submitted: the default fallback is a good direction, but the branch should reuse the existing decoder and fix explicit encoding label handling before merge. The explicit `encoding` parameter itself also needs a maintainer API decision.
