## Label Justifications

- `P1`: The issue breaks real node-host exec in YOLO mode when an approvals socket token is present, blocking a core agent execution workflow.
- `impact:security`: The affected surface is exec approval policy resolution and socket-token handling, which is part of the host command execution security boundary.

## Summary

Keep open: current main and the latest release still gate the full/off fast path on absence of a socket token, and an open linked PR is the active fix path.

## What This Changes

Fix node-host exec approval resolution so YOLO full/off mode does not fall through to socket approval when `exec-approvals.json` contains a socket token.

## Best Possible Solution

Land a narrow fix that lets full/off resolve through the no-persistence fast path regardless of socket-token presence, with regression coverage for an existing token-bearing approvals file while preserving socket behavior for prompting modes.

## Reproduction Assessment

Yes, source-reproducible: create a full/off approvals file with `socket.token`, then `resolveExecApprovals("main", { security: "full", ask: "off" })` follows the socket-token path instead of returning the empty-token fast path. I did not execute that test because this sweep was read-only.

## Solution Assessment

Yes, the narrow fix proposed by the linked PR is the best direction: full/off already means no approval prompt, so socket-token presence should not force socket approval unless the caller explicitly requires a socket.
