## Summary

Keep this issue open: current main and the latest release still carry the token-gated full/off resolver behavior, and the open linked fix PR is the right place to resolve the remaining code and proof questions.

## What This Changes

The issue reports that a token-bearing `exec-approvals.json` can make trusted `host=node` YOLO exec fail with `SYSTEM_RUN_DENIED` even though the effective policy is `security=full, ask=off`.

## Best Possible Solution

Review and land the linked PR, or a replacement, that keeps full/off YOLO execution independent of socket-token presence while preserving token-file hardening and proving the real node path.

## Reproduction Assessment

Yes for the source-level resolver behavior: current main still routes token-bearing full/off approvals files through the socket-capable `ensureExecApprovals` path. The exact Linux headless `SYSTEM_RUN_DENIED` failure still needs live paired-node proof because current source does not require the socket path on Linux.

## Solution Assessment

Unclear as a complete fix until PR review finishes. Removing the token condition is the narrowest likely behavior fix, but the final patch should preserve token-file `0600` hardening and include inspectable node execution proof.
