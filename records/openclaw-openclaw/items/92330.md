## Summary

Keep this issue open: current main still has the token-gated YOLO fast path, and https://github.com/openclaw/openclaw/pull/92334 is an open closing PR for the same bug. Source inspection supports the fast-path drift, but the exact Linux headless socket-denial chain needs review because the socket exec-host path is Darwin/app-gated in current source.

## What This Changes

Fix host=node exec so full/off YOLO policy does not fall into approval/socket denial when `exec-approvals.json` already contains a socket token.

## Best Possible Solution

Review and land the linked fix PR if it preserves full/off YOLO execution without weakening non-YOLO approval enforcement.

## Reproduction Assessment

Yes from source inspection for the token-gated fast path: `resolveExecApprovals` only returns the no-socket full/off result when no socket token is present, while `ensureExecApprovals` generates and saves a token. The exact Kali/Linux socket-denial chain is less certain because current source only uses the socket exec host on Darwin with app exec enabled.

## Solution Assessment

Unclear for the full observed failure: removing the token condition is the narrowest fix for the token-gated source behavior, but maintainers should validate that it matches the reported Linux headless denial path before landing the open PR.
