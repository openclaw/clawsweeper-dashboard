## Summary

Keep open: this PR targets a real Telegram selected-text Quote regression, but it is not merge-ready because the new fallback classifier sends supported rich-only inline content to legacy `sendMessage`, where OpenClaw escapes those tags, and the PR has no real Telegram client proof.

## What This Changes

The PR adds content-aware Telegram rich-message routing so `richMessages: true` text/edit delivery falls back to `sendMessage` when no selected rich block tags are present, with related unit tests updated.

## Best Possible Solution

Keep the content-aware fallback, but only use `sendMessage` for content that can render readably there; keep rich-only inline/block content on `sendRichMessage` or intentionally normalize it before fallback, then add real Telegram proof.

## Reproduction Assessment

Yes at source level. Current main sends `richMessages: true` delivery through `sendRichMessage` while plain/off delivery uses `sendMessage`, and the linked issue has strong Telegram client A/B evidence; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Content-aware routing is the right owner-boundary direction, but this implementation is too broad because it treats supported rich-only inline tags as disposable while the fallback path escapes them.
