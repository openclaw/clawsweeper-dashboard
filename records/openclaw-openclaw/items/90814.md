## Summary

This should stay open: current main still has a source-visible per-row path from `sessions.list` into workspace-scoped plugin metadata resolution when configured model refs need manifest normalization, and the active workspace remains process-global. The existing lightweight-row mitigation avoids provider runtime normalization, but it does not prove the concurrent workspaceDir cache-key churn described here is fixed.

## What This Changes

The issue requests stable plugin metadata lookup during `sessions.list` row building so concurrent agent or cron registry changes cannot force repeated plugin metadata scans.

## Best Possible Solution

Make one `sessions.list` row-building pass use a stable request-scoped workspace or precomputed manifest metadata snapshot across its async yields, while preserving live global registry updates outside that request scope.

## Reproduction Assessment

Yes, at source level: current main still has a per-row model resolution path that can load manifest metadata using the process-global active workspace, and the async row loop yields between batches. I did not run the live multi-agent/crons diagnostic scenario, so this is not marked fully reproduced.

## Solution Assessment

Yes, the request-scoped workspace pin or equivalent precomputed manifest snapshot is a better fix than adding another fallback cache, because it addresses the concurrency boundary directly. The final PR should prove both stable reads inside one list call and live global updates after the scoped call exits.
