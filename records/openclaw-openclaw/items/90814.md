## Summary

Keep open: current main still lacks the request-scoped active plugin workspace pin, and the same reporter has an open paired PR targeting this exact concurrency regression. The newer single-row PR is adjacent but does not replace the workspace-dir race fix.

## What This Changes

The issue reports that `sessions.list` under concurrent agents/crons rereads mutable global plugin-registry `workspaceDir` per row, causing repeated plugin metadata work and severe control-plane latency.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90819, or an equivalent request-scoped workspace snapshot with regression coverage, then close this issue after merge proof.

## Reproduction Assessment

Yes for the source-level race: current main's `sessions.list` row loop yields between batches while manifest normalization reads `workspaceDir` from mutable global state. I did not run a live multi-agent timing repro, and later memo work may change the exact scan count.

## Solution Assessment

Yes: the paired PR's request-scoped workspace pin is the narrow owner-boundary fix for the race. Cache improvements alone do not guarantee that every row in one list request uses the same workspace snapshot.
