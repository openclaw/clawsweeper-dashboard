## Summary

The issue should stay open: current main and the latest v2026.6.6 release still lack the requested request-scoped plugin workspace pin, and the same reporter has an open, mergeable paired PR for this exact concurrency regression. The merged single-row session metadata fix at https://github.com/openclaw/openclaw/pull/92362 is adjacent but explicitly does not cover the workspace-dir concurrency problem.

## What This Changes

The issue requests fixing `sessions.list` so one awaited row-building batch uses a stable plugin-registry `workspaceDir` instead of rereading globally mutable plugin metadata scope per row under concurrent agent or cron activity.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90819, or an equivalent request-scoped workspace pin with regression coverage, then close this issue as implemented after merge proof.

## Reproduction Assessment

Yes for a source-level path: current main still has an awaited `sessions.list` row loop, a global-only plugin workspace getter, and a workspace-keyed metadata snapshot lookup. I did not run a live multi-agent gateway repro in this read-only review, so this is not marked as fully reproduced.

## Solution Assessment

Yes, the narrow request-scoped workspace pin in the paired PR is the best visible fix shape because it removes mutable global workspace reads from the awaited batch without changing public config or row contracts. The merged single-row metadata fix is adjacent but not a substitute for the concurrency pin.
