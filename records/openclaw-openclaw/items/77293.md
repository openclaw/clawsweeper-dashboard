## Summary

Keep open: current main still routes plugin API compatibility through the generic semver range check, so an OpenClaw stable correction runtime can still reject a plugin floor like `>=2026.5.3`. The issue is already paired with open closing PR #77450; the missing runtime-chunk half was not reproduced in the prior review and current tarball import/inventory checks cover that artifact class.

## What This Changes

The issue reports that `openclaw@2026.5.3-1` breaks plugin installs and updates by omitting an `install.runtime-*` chunk and exposing `2026.5.3-1` as a plugin API version that fails `>=2026.5.3` ranges.

## Best Possible Solution

Land or supersede #77450 with a narrow plugin API normalization fix, and keep the tarball import/inventory checker in release validation for `install.runtime-*` chunks.

## Reproduction Assessment

Partially. The plugin API mismatch is source-reproducible through `resolveCompatibilityHostVersion` into `satisfiesPluginApiRange`; the missing runtime-chunk half was not reproduced here and the prior comment reports the current registry tarball listing did not contain the failing chunk pair.

## Solution Assessment

Yes for the linked fix direction. Normalizing only OpenClaw stable correction host versions in the plugin API compatibility path is narrower than changing the runtime version globally and preserves real prerelease rejection.
