## Summary

Keep open: current `main` still routes the plugin API host version through the generic semver range checker, so a stable correction runtime such as `2026.5.3-1` can still be rejected against a plugin API floor of `>=2026.5.3`. The missing runtime-chunk half has source-side tarball guardrails now, but I did not prove the exact published artifact failure is gone.

## What This Changes

The issue asks OpenClaw to restore plugin install/update flows for `openclaw@2026.5.3-1` by shipping referenced install runtime chunks and accepting stable correction versions as compatible plugin API hosts.

## Best Possible Solution

Teach plugin API compatibility to treat OpenClaw stable correction versions as same-base stable hosts while preserving real prerelease rejection, then add focused regression coverage and keep the tarball inventory checks as the artifact safety net.

## Reproduction Assessment

Yes for the plugin API mismatch by source inspection: a runtime version of `2026.5.3-1` flowing into a ClawHub package with `pluginApiRange: ">=2026.5.3"` reaches the generic semver comparator and is ranked below the base stable version. The missing runtime-chunk artifact half is not reproduced from this checkout.

## Solution Assessment

Yes: the narrow maintainable fix is to normalize or specially compare OpenClaw stable correction versions in `satisfiesPluginApiRange`, not to rely on the current `2026.5.4` package version avoiding this one suffix.
