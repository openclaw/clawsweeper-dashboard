## Summary

Keep open: the report is concrete, and current main still has an unfixed correction-suffix compatibility gap even though the tree has moved to 2026.5.4 and has package-integrity guards that may catch part of the runtime chunk failure.

## What This Changes

The issue reports npm package 2026.5.3-1 breaking plugin install/update because the package omits an `install.runtime-*` chunk and exposes a plugin API version that official plugins reject.

## Best Possible Solution

Normalize plugin API compatibility for OpenClaw numeric correction releases and add package/release validation that proves the `install.runtime-*` sidecar is present in npm tarballs and postinstall inventories.

## Reproduction Assessment

Yes for the plugin API half from source: the raw `2026.5.3-1` runtime version flows into semver range checks and compares below `>=2026.5.3`. The missing tarball chunk has concrete reporter logs but was not live-reproduced because this review is read-only.

## Solution Assessment

No current-main implementation fully solves it: the version bump to 2026.5.4 avoids this exact release pairing, but the correction-suffix compatibility rule remains unfixed. The narrow solution is to normalize only plugin API compatibility versions while keeping normal gateway/version ordering intact, then lock the runtime chunk packaging path with tests.
