## Summary

Keep open. Current main has durable delivery queues, platform receipts, and Slack/channel-final transcript mirrors, but it still does not link assistant transcript entries to expected Discord/platform message receipts, expose delivery state in Control UI, or provide an idempotent reconciler; the only directly linked PR is still open and explicitly scoped as a partial mitigation.

## What This Changes

Request a delivery receipt and reconciliation layer for channel-scoped sessions so transcripted assistant outputs can be matched to expected client/channel delivery and repaired when delivery drifts.

## Best Possible Solution

Design one generic delivery ledger keyed by transcript entry and channel target, record provider message ids/failures through the existing durable delivery lifecycle, expose the state in session APIs/Control UI, and add an idempotent operator-triggered reconciler.

## Reproduction Assessment

No high-confidence live reproduction is present for the intermittent Discord drop. Source inspection does confirm current main lacks the requested transcript-entry to provider-message delivery ledger, reconciler, and UI/API delivery-state surface.

## Solution Assessment

Unclear as written: the goal is the right reliability direction, but the durable implementation should extend the existing delivery queue, receipt, and commit-hook lifecycle rather than adding a parallel transcript sidecar.
