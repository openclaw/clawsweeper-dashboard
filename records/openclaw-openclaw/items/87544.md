## Summary

Keep open. Current main has partial durable send receipt and queue primitives, but the central request remains missing: a transcript-entry-to-provider-message delivery ledger, Discord reconciliation, and Control UI visibility for drift between transcript and channel delivery.

## What This Changes

Requested behavior: track and reconcile delivery state between durable assistant transcript entries and the originating Discord/channel client surface for channel-scoped sessions.

## Best Possible Solution

Design one generic delivery ledger keyed by transcript entry id and channel target, have Discord record provider receipts/failures into it, expose that state in session APIs/Control UI, and add an idempotent operator-triggered reconciler.

## Reproduction Assessment

No high-confidence reproduction for the intermittent Discord drop is provided. Source inspection does confirm that the requested durable transcript-entry to provider-message delivery ledger and reconciler are not present on current main.

## Solution Assessment

The requested direction is plausible, but the best implementation should extend the existing durable send, receipt, and queue recovery primitives instead of adding a separate parallel transcript sidecar.
