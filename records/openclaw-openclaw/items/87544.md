## Summary

Keep open. Current main and v2026.6.9 still have queue-local send recovery and channel receipts, but not the requested durable transcript-entry-to-provider-message ledger, Discord reconciliation semantics, or session/Control UI delivery-state surface.

## What This Changes

The issue requests durable delivery receipts and reconciliation between channel-scoped assistant transcript entries and their expected client/channel delivery.

## Best Possible Solution

Design a single SQLite-backed delivery ledger keyed by transcript entry and expected channel target, feed it from the outbound send/recovery lifecycle, and expose delivery state through Gateway/session APIs and Control UI.

## Reproduction Assessment

No live Discord reproduction is available in this review. Source inspection gives a high-confidence way to verify the product gap: current main lacks a transcript-entry-to-provider-message delivery ledger and delivery-state UI/API surface.

## Solution Assessment

Unclear as a final implementation. The requested direction is right, but the durable implementation should extend the existing outbound receipt/recovery lifecycle rather than create a separate sidecar that can drift from queue state.
