## Summary

Keep open. Current main and v2026.6.9 still have queue-local delivery state, channel receipts, and best-effort transcript mirrors, but not the requested transcript-entry-to-provider-message ledger, Discord reconciliation contract, Control UI delivery-state surface, or idempotent redelivery path.

## What This Changes

The issue requests durable delivery receipts and reconciliation for assistant transcript entries expected to appear in their originating channel-scoped client surface.

## Best Possible Solution

Design a generic SQLite-backed delivery ledger keyed to transcript entries and expected channel targets, feed it from outbound receipt/recovery lifecycle events, expose it through session APIs and Control UI, and keep redelivery idempotent through provider-message receipts.

## Reproduction Assessment

No live Discord reproduction was run in this read-only review. Source inspection gives a high-confidence way to verify the remaining product gap: current main lacks a transcript-entry-to-provider-message ledger and delivery-state UI/API surface.

## Solution Assessment

Yes for keeping the issue open. The desired direction is valid, but the implementation should extend the existing outbound receipt/recovery lifecycle with a maintainer-approved persistent ledger rather than add an isolated sidecar that can drift.
