## Summary

Keep open. Current main and v2026.6.8 still have durable queue-local send state, normalized channel receipts, and transcript mirrors, but not the requested transcript-entry-to-provider-message ledger, Discord reconciliation semantics, Control UI/session delivery-state surface, or idempotent redelivery path.

## What This Changes

The issue requests durable delivery receipts and reconciliation so channel-scoped assistant transcript entries can be tied to client-visible Discord delivery state.

## Best Possible Solution

Design a generic SQLite-backed delivery ledger keyed by transcript entry and expected channel target, feed it from the outbound receipt/recovery lifecycle, expose it through session APIs and Control UI, and add an idempotent operator reconciliation path.

## Reproduction Assessment

No live Discord reproduction is available in this review. Source inspection gives a high-confidence way to verify the product gap: current main lacks a transcript-entry-to-provider-message delivery ledger and delivery-state UI/API surface.

## Solution Assessment

Unclear as a final design. The requested direction is right, but the durable implementation should extend the existing outbound receipt/recovery lifecycle rather than create a separate sidecar that can drift from queue state.
