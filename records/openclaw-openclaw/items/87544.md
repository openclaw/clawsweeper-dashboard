## Summary

Keep open. Current main has durable outbound queueing, normalized provider receipts, and recent Slack transcript-mirroring work, but it still does not provide the requested transcript-entry-to-provider-message delivery ledger, Discord reconciliation, or Control UI/API visibility for transcript-to-client drift; the related open PR is only a draft mitigation and explicitly leaves the broader design open.

## What This Changes

Request adding delivery receipt and reconciliation support for channel-scoped sessions so transcripted assistant replies can be correlated with expected Discord/client delivery.

## Best Possible Solution

Design one generic delivery ledger keyed by transcript entry and channel target, have Discord record provider receipts/failures into it, expose that state through session APIs/Control UI, and add an idempotent operator-triggered reconciler.

## Reproduction Assessment

No high-confidence live reproduction of the intermittent Discord drop is available. Source inspection does confirm current main lacks the requested transcript-entry-to-provider delivery ledger, Discord unknown-send reconciliation, and UI delivery-state surface.

## Solution Assessment

The requested direction is plausible, but the best implementation is not settled. It should extend the existing durable queue, MessageReceipt, session transcript, Gateway, and Control UI contracts instead of adding a parallel sidecar without maintainer agreement.
