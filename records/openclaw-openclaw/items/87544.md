## Summary

Keep open. Current main and v2026.6.8 still have durable queue state, platform send receipts, and narrow transcript mirrors, but they do not provide the requested transcript-entry-to-provider-message delivery ledger, Discord reconciliation semantics, Control UI/session delivery-state surface, or idempotent redelivery path.

## What This Changes

The issue requests durable delivery receipts and reconciliation for channel-scoped assistant transcript entries so operators can detect and repair transcript-present but client-missing replies.

## Best Possible Solution

Design a generic SQLite-backed delivery ledger keyed by transcript entry and expected channel target, feed it from the outbound receipt/recovery lifecycle, expose it through session APIs and Control UI, and add an idempotent operator reconciliation path.

## Reproduction Assessment

No live reproduction is present for the intermittent Discord drop. Source inspection does give a high-confidence way to verify the reported product gap: current main lacks a transcript-entry-to-provider-message delivery ledger and delivery-state UI/API surface.

## Solution Assessment

Unclear as proposed. The goal is right, but the durable implementation should extend the existing outbound receipt/recovery lifecycle rather than create a separate sidecar that can drift from queue state.
