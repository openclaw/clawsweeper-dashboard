## Summary

Keep open. Current main and the latest release have durable outbound queues and normalized platform receipts, but they still do not persist a transcript-entry to provider-message delivery ledger, expose channel delivery state in session/Control UI chat data, or provide an idempotent reconciler for the Discord/client drift described here.

## What This Changes

The issue requests durable delivery receipts and reconciliation for channel-scoped assistant outputs so transcript entries can be matched to expected client/channel delivery.

## Best Possible Solution

Design one generic delivery ledger keyed by transcript entry and channel target, record receipt/failure/reconciliation state through the existing durable delivery lifecycle, expose it through session APIs and Control UI, and add an idempotent operator-triggered reconciler.

## Reproduction Assessment

No high-confidence live Discord reproduction is available for the intermittent drop. Source inspection does confirm current main lacks the requested transcript-entry to provider-message delivery ledger, idempotent reconciler, and session/Control UI delivery-state surface.

## Solution Assessment

Unclear as proposed: the reliability goal is sound, but the best implementation should extend the existing durable delivery lifecycle and receipt model rather than adding an ad hoc transcript sidecar.
