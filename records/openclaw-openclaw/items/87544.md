## Summary

Keep open. Current main has durable outbound queue state and narrow transcript mirror markers, but it still lacks the requested transcript-entry-to-provider-message delivery ledger, Discord reconciliation semantics, Control UI delivery-state surface, and idempotent redelivery path; the linked mitigation PR is open, draft, and explicitly partial.

## What This Changes

Request a durable transcript-to-channel delivery receipt and reconciliation layer for channel-scoped sessions so operators can detect and repair transcript/client drift.

## Best Possible Solution

Design one generic delivery ledger keyed by transcript entry and channel target, feed it from the durable delivery commit/reconciliation lifecycle, expose it through session APIs and Control UI, and add an idempotent operator-triggered reconciler.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction of the intermittent Discord drop. Source inspection does confirm current main lacks the requested transcript-entry-to-provider-message ledger, Discord reconciler, and Control UI delivery-state surface.

## Solution Assessment

Unclear as proposed: the goal is right, but the durable implementation should extend the existing delivery queue, receipt, and commit-hook lifecycle. A separate transcript sidecar would be riskier than a single source of delivery truth.
