## Summary

Keep open: current main still has the reported silent-drop path. The followup runner logs and fails the operation, then returns before delivery, while the main reply path already has known-failure payload handling for billing and rate-limit failures that this path should reuse.

## What This Changes

The issue requests a user-facing channel/thread notice when followup reply generation fails before producing output because the model provider rejects the request for billing, quota, or rate-limit reasons.

## Best Possible Solution

Reuse or factor the existing known run-failure payload classifier in the followup runner catch path, then deliver that payload through the existing followup delivery route while preserving `message_tool_only` suppression.

## Reproduction Assessment

Yes, at source level. A rejected `runWithModelFallback` or embedded agent call reaches the followup catch block, which records and logs the failure but returns before `sendFollowupPayloads`; the reporter's logs show that exact path for billing failures.

## Solution Assessment

Yes, the issue should be fixed in code, but the best fix is to reuse the main path's known-failure payload logic rather than adding separate quota string matching in the followup runner.
