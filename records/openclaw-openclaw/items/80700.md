## Summary

Keep open. Current main still has the reported followup-runner gap: provider billing/rate-limit failures can become thrown failover errors, but the followup catch only logs and returns before routing any user-facing payload.

## What This Changes

The issue asks for a visible Slack/Telegram/etc. followup notice when billing, quota, or rate-limit rejection prevents reply generation.

## Best Possible Solution

Add a narrow followup-runner failure-surfacing path that reuses existing billing/rate-limit classification and sanitized copy, routes it through `sendFollowupPayloads`, and covers it with focused followup-runner tests.

## Reproduction Assessment

Yes, source-reproducible. If `runWithModelFallback` throws a billing/rate_limit FailoverError or FallbackSummaryError during a followup run, current main logs and returns before calling the existing outbound delivery path; no live Slack/Anthropic run was executed in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to handle known billing/rate-limit/fallback-summary failures in the followup catch with existing sanitized failure-copy helpers and the existing followup delivery route, while leaving generic failures and `message_tool_only` behavior unchanged.
