## Summary

Keep open: current main still has a queued followup failure path that logs billing/quota/rate-limit fallback rejection and returns without delivering a user-facing payload, while the immediate agent path already has explicit billing/rate-limit surface behavior.

## What This Changes

Request that queued followup failures caused by billing/quota/rate-limit provider rejection post a short notice to the originating channel/thread instead of only logging.

## Best Possible Solution

Add a narrow queued-followup failure notice path that reuses the existing billing/rate-limit classification and delivery machinery, with regression coverage for billing, rate-limit, and `message_tool_only` suppression.

## Reproduction Assessment

Yes, at source level. A queued followup whose `runWithModelFallback` call rejects with a billing/rate-limit `FallbackSummaryError` reaches the catch in `createFollowupRunner`, logs, and returns before `sendFollowupPayloads`; I did not run a live provider-quota exhaustion setup.

## Solution Assessment

Yes, the narrow maintainable fix is to route only classified billing/quota/rate-limit queued-followup failures through the existing followup delivery path, while preserving `message_tool_only` and adding focused tests.
