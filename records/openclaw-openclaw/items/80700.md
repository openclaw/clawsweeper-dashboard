The issue asks OpenClaw to surface a user-facing Slack/Telegram/etc. notice when follow-up reply generation fails on provider billing/quota/rate-limit rejection.

## Best Possible Solution

Land a narrow fix that sends a safe origin-channel error notice for provider billing/quota/rate-limit follow-up failures, reuses the shared failover classifiers, and preserves explicit delivery-suppression boundaries for non-error output.

## Solution Assessment

Yes in direction, but the fix should use the existing provider error classifiers and route a generic error notice intentionally rather than treating it like normal follow-up output. The linked PR is the active implementation candidate but needs review blockers resolved before this issue can close.
