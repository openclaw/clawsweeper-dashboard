## Summary

Keep open. The PR targets a real P1 stale-session bug and includes useful live proof, but the latest diff still needs changes because startup recovery now treats all markerless running rows as stale even though recovery is scheduled after the gateway is ready.

## What This Changes

This PR adds stale `running` row healing to `sessions.abort` no-active-run handling, broadens main-session restart recovery to markerless running rows, and adds regression tests.

## Best Possible Solution

Keep the abort no-active-run healing, but gate markerless restart recovery to rows known to predate the current gateway process or another durable stale signal, with a regression proving fresh current-process `running` rows are left alone.

## Reproduction Assessment

Yes, source-level with high confidence. Current `main` leaves the no-active-run abort path without store reconciliation and marker-gates restart recovery; the PR's new startup race follows from the delayed post-ready recovery scan.

## Solution Assessment

No. The abort healing direction is right, but the restart recovery change needs a stale ownership or startup-cutoff guard before it is the safest fix.
