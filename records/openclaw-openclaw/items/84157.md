## Summary

Keep open: the abort no-active-run healing is useful, but the markerless restart recovery path is still too broad and can treat current-process session state as abandoned.

## What This Changes

This PR clears stale persisted `running` session rows after `sessions.abort` returns no-active-run, broadens main-session restart recovery to markerless `running` rows, and adds focused regression tests.

## Best Possible Solution

Keep the abort no-active-run healing, but restrict markerless restart recovery to rows known to predate the current process or carrying another durable stale signal, with regression coverage for fresh current-process rows.

## Reproduction Assessment

Yes. Current main’s stale-row gap and the PR’s markerless recovery risk are source-reproducible from the abort path, restart recovery selector, and startup ordering.

## Solution Assessment

No. The abort healing direction is narrow and appropriate, but markerless restart recovery needs a pre-process or durable-stale ownership guard before it is safe to merge.

Full review comments:

- **[P1] Gate markerless recovery to pre-start stale rows:** `src/agents/main-session-restart-recovery.ts:422-438`
  - body: This branch treats every markerless `running` row as stale unless an embedded run or active command lane is visible. The recovery scan is scheduled after HTTP bind/post-ready startup, and normal lifecycle start persists `status: "running"` with `abortedLastRun: false`; regular agent activity is tracked in gateway context rather than this recovery module. A fresh current-process row can therefore be resumed or failed as abandoned. Please require a process-start/pre-bind cutoff or durable restart marker, and add a regression for a current-process markerless row.
