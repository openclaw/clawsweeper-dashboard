## Summary

Keep open: the stale-row fix is useful and source-reproducible, but the latest restart-recovery guard still uses a cutoff captured after the gateway has already bound HTTP, so markerless current-process runs can still be mistaken for abandoned startup rows.

## What This Changes

This PR heals stale `running` rows after `sessions.abort` returns no-active-run, broadens restart recovery to markerless `running` rows with a timestamp guard, and adds regression tests.

## Best Possible Solution

Keep the abort no-active-run healing, but gate markerless restart recovery to rows that predate the current process startup or carry another durable stale signal, with a regression proving fresh current-process markerless rows are skipped.

## Reproduction Assessment

Yes, source-level with high confidence: current main has the abort/restart gaps, and the remaining PR race follows from HTTP bind preceding post-ready recovery scheduling.

## Solution Assessment

No. The abort healing direction is right, but the markerless restart recovery guard should use process-start ownership or another durable stale signal rather than a schedule-time cutoff.
