## Summary

Keep open. The PR targets a real stuck-session bug and has useful proof, but the new abort healing path only checks the canonical key and can miss stale rows stored under existing legacy, deleted-agent, or case-variant store keys.

## What This Changes

Adds stale `running` row healing for `sessions.abort` no-active-run outcomes, broadens main-session restart recovery to markerless `running` rows, and updates regression tests.

## Best Possible Solution

Keep the no-active-run and restart-recovery direction, but update abort healing to mutate the resolved freshest stored row across `storeKeys` and retain focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main can return `no-active-run` without clearing persisted `running` state, and the PR body includes before/after live output; I did not execute tests because this review is read-only.

## Solution Assessment

No, not as written. The recovery direction is the right narrow fix, but the abort branch should use the existing resolved store-key/freshest-match contract instead of only `store[canonicalKey]`.
