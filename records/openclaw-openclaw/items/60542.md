## Summary

Keep open. Current main and the latest release still reuse a persisted main-session row based on session id and freshness timestamps without reconciling whether a terminal row is stale relative to its transcript, and the only exact fix PR was closed unmerged.

## What This Changes

The issue asks for persisted main-session state to stay consistent with the linked transcript so stale terminal rows cannot wedge new input after restart or recovery.

## Best Possible Solution

Add targeted main-session initialization or session-file reconciliation that detects terminal rows older than their transcript or lifecycle tail and safely rolls forward or rotates the session with regression coverage.

## Reproduction Assessment

Yes, at source level with high confidence. Seed the main session store with a terminal row whose updatedAt or endedAt predates newer transcript content; current initSessionState can reuse it because freshness checks ignore transcript tail freshness.

## Solution Assessment

Yes. A narrow reconciliation in initSessionState or the session-file helper is the maintainable fix; the closed PR's core idea should be reimplemented cleanly without unrelated files and with focused regression coverage.
