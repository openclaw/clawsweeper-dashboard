## Summary

Keep open. Current main still reuses a finite `updatedAt` main-session row without checking whether a terminal registry entry is stale relative to its transcript, and the adjacent stale-running fix does not cover this terminal-row divergence.

## What This Changes

The issue asks OpenClaw to keep persisted main-session registry rows consistent with their transcripts so stale terminal rows cannot wedge new input after restart.

## Best Possible Solution

Add a narrow main-session initialization or session-file reconciliation that detects terminal rows older than their transcript or lifecycle tail and safely rolls forward or rotates without changing normal terminal sessions or explicit reset behavior.

## Reproduction Assessment

Yes, as a source-level path. Seed `agent:main:main` with a terminal row whose timestamp is older than the referenced transcript, then call `initSessionState`; current main will treat the row as fresh/reusable because it only checks finite `updatedAt` and reset policy.

## Solution Assessment

Yes. A targeted `initSessionState` or `session-file` reconciliation with regression coverage is the narrowest maintainable fix; a broad startup migration or watchdog-only approach would carry more risk and leave persistence inconsistent.

## Repair Work Prompt

Fix the persisted main-session terminal-row divergence reported in https://github.com/openclaw/openclaw/issues/60542. Add a narrow reconciliation in `initSessionState` and/or `src/config/sessions/session-file.ts` so a main-session row already marked terminal is not blindly reused when its referenced transcript or lifecycle tail is newer than the row's persisted terminal timestamp. Preserve normal terminal sessions, explicit `/new` and `/reset` behavior, provider-owned session handling, and the existing #60250 stale-running metadata-patch fix.
