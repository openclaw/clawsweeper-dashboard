## Summary

Keep open. Current main has partial mitigations, but the low-power host performance problem remains credible and unresolved across plugin-tool materialization, session-store discovery compatibility, fixed concurrency defaults, and open startup blockers called out in the follow-up discussion.

## What This Changes

The issue requests low-power host performance work across default plugin loading, plugin dependency staging, session/node workspace scans, and concurrency defaults.

## Best Possible Solution

Keep this open as an umbrella, but split implementation into bounded fixes: credential-aware or lazy plugin-tool materialization, idempotent runtime-deps staging, compatibility-safe session discovery caching, host-aware low-power concurrency defaults, and linked startup blockers such as #76202.

## Reproduction Assessment

Unclear for the full 30+ second chain because it requires the reporter’s low-power host, storage, plugin inventory, network timeouts, and workspace/session state. The static ingredients and live comments are credible: many default-enabled plugins, fixed 4/8 concurrency defaults, disk-backed session discovery, and linked startup blockers are visible on current main.

## Solution Assessment

No, not as one patch. The best maintainable path is to split by subsystem and preserve compatibility contracts, especially disk-only ACP sessions and plugin SDK/runtime boundaries.
