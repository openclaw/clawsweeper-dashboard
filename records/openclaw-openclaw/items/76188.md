## Summary

Keep open. Current main has partial mitigations, including a cheaper `node.list` path and async yielding in `sessions.list`, but the broader low-power saturation cluster remains unresolved across default-enabled provider/tool materialization, disk-backed session discovery, fixed concurrency defaults, Discord startup behavior, and model-pricing startup work.

## What This Changes

The issue asks OpenClaw to reduce compounded low-power host CPU and event-loop saturation caused by eager plugin loading, repeated plugin staging, expensive session/node discovery, and fixed high concurrency defaults.

## Best Possible Solution

Split this into bounded fixes: credential-aware or lazy provider/tool materialization, measured idempotent runtime-dependency staging, bounded cached session discovery, no-op Discord native-command disablement, pricing fetch backpressure, and an explicit low-power concurrency profile.

## Reproduction Assessment

Unclear for the whole chain because it depends on the reporter's hardware, storage, network, plugin inventory, and session data. The individual inputs are credible and source-backed on current main, and the comments add independent low-power host data points.

## Solution Assessment

Unclear as one combined change; the best route is to split by subsystem and validate each hot path separately. Current main is a partial mitigation, not a terminal solution.
