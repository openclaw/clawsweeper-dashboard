## Summary

Keep open. Current main has partial session-list mitigations, but the central low-power performance chain is not resolved: many bundled plugins are still default-enabled, plugin-tool prep still walks manifest metadata and can fall back to runtime loading, disk-only session-store discovery remains intentional behavior, and the default concurrency caps are still fixed at 4 agent runs plus 8 subagent runs.

## What This Changes

The issue requests low-power host performance fixes across default plugin activation, plugin runtime dependency staging, session/node listing I/O, and host-aware concurrency defaults.

## Best Possible Solution

Keep this open and split it into bounded performance work: credential-aware or lazy plugin-tool materialization, idempotent runtime-deps staging, bounded async session-store discovery that preserves intentional disk-only ACP sessions, and a host-aware low-power concurrency profile.

## Reproduction Assessment

Unclear for the full issue chain because it depends on low-power hardware, slow storage, plugin inventory, and large workspace/session state. The static inputs remain source-backed on current main: many default-enabled plugins, disk-backed session-store discovery, and fixed 4/8 concurrency defaults.

## Solution Assessment

Not applicable as a single patch; the requested direction is valid, but one broad automated change would cross plugin activation policy, runtime staging, Gateway session compatibility, and default resource policy. The safer solution is to split and validate each subsystem independently.
