## Summary

Keep open. Current main has partial mitigations, but the reported low-power host problem still has source-backed contributors across plugin tool materialization, disk-backed session discovery, fixed concurrency defaults, startup network work, and the same-day comments add credible independent gateway-idle CPU data plus the open Discord subissue #76202.

## What This Changes

Request to reduce a chain of low-power host CPU and event-loop bottlenecks across default plugin activation, plugin runtime staging/reload, session/node listing, startup network work, and fixed agent/subagent concurrency defaults.

## Best Possible Solution

Split this into bounded performance work: credential-aware or lazy plugin-tool materialization, idempotent runtime-dep staging, bounded async session/agent-store discovery with disk-only compatibility preserved, host-aware low-power concurrency defaults, and startup network opt-outs or caching.

## Reproduction Assessment

Unclear for the full issue chain. The reporter and follow-up commenter provide detailed timings and host data, and current source confirms several static contributors, but reproducing the 30s+ event-loop saturation requires the same low-power hardware, storage, network, plugin, and session-store conditions.

## Solution Assessment

Unclear as one fix because the requested direction spans plugin activation policy, gateway/session compatibility, Discord startup behavior, model catalog/pricing fetches, and runtime defaults. The maintainable path is to split by subsystem and validate each with focused performance tests plus a low-power smoke profile.
