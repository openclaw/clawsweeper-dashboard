## Summary

Keep open. Current main has partial mitigations for session listing and model-catalog timeouts, but the broader low-power host performance chain remains unresolved across plugin activation/tool materialization, disk-backed session discovery, fixed concurrency defaults, and linked startup work.

## What This Changes

The issue asks OpenClaw to reduce low-power host CPU and event-loop saturation by pruning or lazily loading unused plugins, avoiding repeated runtime-dependency staging, reducing session/node scans, and capping concurrency by host capacity.

## Best Possible Solution

Split this into narrow performance fixes that preserve current plugin APIs and disk-only session compatibility while adding host-aware defaults and explicit opt-outs for expensive startup paths.

## Reproduction Assessment

Unclear for the full chain: the 30+ second event-loop stalls depend on hardware, storage, workspace size, plugin inventory, credentials, and network timeouts. The static reproduction inputs are high-confidence because current main still has many default-enabled plugins, disk-backed session discovery, and fixed 4/8 concurrency defaults.

## Solution Assessment

No as a single automated repair; the requested direction spans several subsystems with different compatibility constraints. The maintainable path is separate fixes for plugin-tool laziness, runtime-dep staging idempotence, bounded session/node discovery, low-power concurrency defaults, and linked startup RPC/network blockers.
