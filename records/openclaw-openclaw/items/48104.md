## Summary

Keep open: current main has execution-bias prompting, structured exec approval/tool-failure states, and limited refusal/error metadata, but it still does not define the trusted operator diagnostics contract or reliably distinguish pre-tool model soft refusals from authorization and SSH/tool failures.

## What This Changes

The issue requests clearer handling and observability when model safety behavior blocks explicitly authorized read-only operational diagnostics such as SSH log and service checks.

## Best Possible Solution

Define a maintainer-approved operator diagnostics contract that preserves hard authorization in policy/approvals, then add narrow refusal-versus-authorization-versus-tool-failure observability with live provider proof.

## Reproduction Assessment

No. The issue describes observed soft refusals, but it lacks the provider/model, transcript, configured tool policy, approval/elevated state, and SSH target needed for a high-confidence current-main reproduction.

## Solution Assessment

Unclear. A broad trusted-admin mode might help, but the safer maintainable path is to define the hard-policy boundary first and then add narrow observability or routing changes with live provider evidence.
