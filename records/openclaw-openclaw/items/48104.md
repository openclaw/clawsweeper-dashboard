## Summary

Keep open: current main has execution-bias prompting, planning-only/reasoning-only retry guards, exec approval states, and limited refusal metadata, but it still does not define the requested trusted operator diagnostics contract or classify pre-tool model soft refusals separately from authorization and SSH/tool failures.

## What This Changes

The issue requests better execution and observability when model safety behavior blocks explicitly authorized operator diagnostics such as SSH read-only checks.

## Best Possible Solution

Define a maintainer-approved operator diagnostics contract that preserves hard exec authorization, then add narrow refusal-versus-authorization-versus-tool-failure observability with live provider proof.

## Reproduction Assessment

No. The report describes observed soft refusals, but it lacks the provider/model, transcript, configured tool policy, approval/elevated state, and SSH target needed for a high-confidence current-main reproduction.

## Solution Assessment

Unclear. A broad trusted-admin mode is not the safest solution by itself; the maintainable path is to define the hard-policy boundary first, then add narrow observability or routing changes with live provider evidence.
