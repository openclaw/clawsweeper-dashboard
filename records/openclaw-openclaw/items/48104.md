## Summary

Keep open: current main and v2026.5.12 already have execution-bias prompting, hard exec-approval policy, structured exec approval/tool states, and limited refusal metadata, but they still do not define a trusted operator diagnostics contract or reliably classify pre-tool model soft refusals separately from authorization and SSH/tool failures.

## What This Changes

The issue requests a trusted operator diagnostics path that prevents model-level soft refusals from masking authorized SSH/read-only infrastructure work and makes refusal, authorization, and tool failures distinguishable.

## Best Possible Solution

Define a maintainer-approved operator diagnostics contract that preserves hard authorization in policy/approvals, then add narrow refusal-versus-authorization-versus-tool-failure observability with live provider proof.

## Reproduction Assessment

No. The issue describes observed soft refusals, but lacks the provider/model, transcript, configured tool policy, approval/elevated state, and SSH target needed for a high-confidence current-main reproduction.

## Solution Assessment

Unclear. A broad trusted-admin mode is not sufficient by itself; the safer maintainable path is to define the hard-policy boundary first, then add narrow observability or routing changes with live provider evidence.
