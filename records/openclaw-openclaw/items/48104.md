## Summary

Keep open. Current main has useful adjacent work around execution bias, strict-agentic plan-only retries, exec approvals, diagnostics, trajectory export, and generic refusal metadata, but it still does not implement the central product gap: a first-class distinction between SSH/tool failure, authorization or approval denial, and model-level soft refusal for explicitly authorized read-only operational diagnostics.

## What This Changes

The issue requests better handling and observability when model safety behavior blocks explicitly authorized operational tasks such as SSH diagnostics, log inspection, and service-health checks.

## Best Possible Solution

Add a core agent-runtime observability and policy design pass that records structured run outcomes for tool transport failure, authorization/approval denial, and model or tool-routing soft refusal; then refine prompt/runtime guidance for explicitly authorized read-only infrastructure diagnostics without weakening destructive-action approvals.

## Reproduction Assessment

Yes, at the product-behavior level: a focused regression can simulate a provider response that answers cautiously to an SSH/log-inspection request while exec is available and authorized, then assert the run records a model/tool-routing refusal instead of a tool failure or approval denial. The current issue does not include a live transcript, so the exact provider-specific reproduction remains unspecified.

## Solution Assessment

Unclear. The current adjacent implementation is not the best complete solution because strict-agentic retries and generic refusal metadata only cover some stall/refusal shapes; the safer path is a small explicit failure-source taxonomy plus targeted prompt/runtime guidance for read-only diagnostics.
