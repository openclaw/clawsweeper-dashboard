## Summary

Keep open. Current main has static SecretRef support and model-provider-scoped auth hooks, but it does not expose the cross-domain Credential Provider Plugin capability requested here. The RFC also needs maintainer API and security design review rather than ClawHub closure, because the missing piece is a core Plugin SDK contract that external providers would implement.

## What This Changes

The issue requests a new Credential Provider Plugin capability so plugins can supply, refresh, rotate, health-check, and revoke credentials for model providers, channels, skills, tools, and other plugins.

## Best Possible Solution

Keep the RFC open for maintainer-led API and security design. If accepted, the maintainable path is an additive, versioned credential-provider capability with explicit trust policy, per-session scoping, integration boundaries for SecretRef/AuthProfileStore, and one proof provider; AWS or Volcengine implementations can then live as plugins on top of that seam.

## Reproduction Assessment

Not applicable; this is an RFC rather than a bug. The current-main verification path is source and docs inspection showing no credential-provider registration API and static/eager SecretRef semantics.

## Solution Assessment

Unclear as an implementation plan, but keeping the design discussion open is the right current outcome. The requested direction is plausible under the plugin-capability model, but the exact API, trust model, and rollout need maintainer review before code is queued.
