## Summary

Keep open: current main and the latest release still provide static `SecretRef` plus model-provider auth/profile machinery, but no cross-domain credential-provider lifecycle API, plugin registration contract, consumer request path, per-session scoping, or settled trust model. The open SecretRef-source work at https://github.com/openclaw/openclaw/pull/72548 and related issue https://github.com/openclaw/openclaw/issues/71593 are useful narrower pieces, but they do not close this broader credential lifecycle RFC.

## What This Changes

Requests an RFC-level Credential Provider Plugin capability for cross-domain credential acquisition, refresh, rotation, interactive auth, per-session scoping, and trust boundaries parallel to SecretRef.

## Best Possible Solution

Keep the RFC open for maintainer and security-led API design; if accepted, land an additive SDK/runtime contract with an explicit trust model, SecretRef/AuthProfileStore boundaries, per-session scoping, and proof provider before moving concrete backends to plugins or ClawHub.

Not applicable; this is an RFC for a new credential capability, not a broken existing behavior report. Source, docs, release, and related-PR inspection give a high-confidence current-main status: the requested cross-domain credential-provider lifecycle API is absent.

## Solution Assessment

No for automatic implementation. The direction is plausible, and the linked AWS, Volcengine, auth-proxy, and broker discussions are useful, but the maintainable path is a maintainer-approved SDK and security contract before any provider backend lands.

## Risks / Open Questions

- The RFC is security-sensitive because it asks plugins to supply credentials across models, channels, skills, tools, and other plugins; authorization, redaction, caching, audit, refresh, and failure semantics need explicit design before implementation.
- Several overlapping open tracks cover pieces of the problem, including pluggable SecretRef sources, masked secrets, external auth proxying, and agent-callable secret broker APIs, but none settles the full value-return versus proxy/broker trust model raised in the latest discussion.
- A narrow automated PR could accidentally create a second credential source of truth that conflicts with `SecretRef`, `AuthProfileStore`, and provider-owned OAuth refresh behavior.
