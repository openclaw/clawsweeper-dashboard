## Summary

Keep open. Current main still implements the static, eager SecretRef model only, while this RFC asks for a new cross-domain credential-provider lifecycle API with interactive auth, refresh/rotation, per-session scoping, and explicit trust boundaries; the adjacent pluggable SecretRef-source work is open and narrower, not an implementation of this RFC.

## What This Changes

Add a Credential Provider Plugin capability for cross-domain credential acquisition, refresh, rotation, health checks, and interactive auth flows parallel to SecretRef.

## Best Possible Solution

Keep the RFC open for maintainer and secops-led API design; if accepted, land an additive versioned SDK/runtime capability with explicit trust policy, per-session scoping, SecretRef/AuthProfileStore boundaries, and a proof provider before moving concrete backends to plugins or ClawHub.

## Reproduction Assessment

Not applicable; this is an RFC for a new capability, not a broken existing behavior report. The high-confidence verification path is source and docs inspection showing no credential-provider registration API and static env/file/exec SecretRef semantics on current main.

## Solution Assessment

No for automatic implementation. The direction is plausible, but the maintainable path is a maintainer-approved SDK and security contract first; the open pluggable SecretRef-source PR is adjacent, narrower work rather than a complete solution for this RFC.
