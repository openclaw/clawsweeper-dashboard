## Summary

Keep open: current main still lacks the RFC's credential-provider registration API, consumer request path, lifecycle engine, per-session scoping contract, and plugin trust policy. The AWS offer, the Authsome broker comment, and open SecretRef-provider PRs are useful adjacent paths, but they do not settle the broader security-sensitive API design.

## What This Changes

The issue requests a new Credential Provider Plugin capability for cross-domain credential acquisition, refresh, rotation, health checks, interactive auth flows, and plugin-backed credential sharing.

## Best Possible Solution

Keep the RFC open for maintainer and security-led API design; if accepted, land an additive SDK/runtime contract with an explicit trust model, SecretRef/AuthProfileStore boundaries, per-session scoping, and a proof provider before moving concrete backends to plugins or ClawHub.

Not applicable; this is an RFC for a new capability, not a broken existing behavior report. Source and docs inspection gives a high-confidence current-main status: OpenClaw has static SecretRef and provider auth/profile machinery but no credential-provider lifecycle API.

## Solution Assessment

No for automatic implementation. The direction is plausible, and the AWS/Authsome comments plus prior-art plugin are useful, but maintainers need to choose the security model and SDK contract before any backend or lifecycle engine lands.
