## Summary

Keep open: current main still has static SecretRef and provider auth/profile machinery, but not the RFC’s credential-provider registration API, consumer request path, lifecycle engine, per-session scoping contract, or plugin trust policy. The AWS Secrets Manager offer, Authsome broker model comment, and open SecretRef-provider PRs are useful adjacent work, but they do not settle the broader security-sensitive API design.

## What This Changes

Requests an RFC-level core Plugin SDK/runtime credential-provider capability for plugin-owned credential acquisition, refresh, rotation, scoping, and cross-consumer use alongside SecretRef.

## Best Possible Solution

Keep the RFC open for maintainer and security-led API design; if accepted, land an additive SDK/runtime contract with an explicit trust model, SecretRef/AuthProfileStore boundaries, per-session scoping, and a proof provider before moving concrete backends to plugins or ClawHub.
