## Summary

Keep open: current main still has static SecretRef and provider-auth/profile machinery, but it lacks the RFC's credential-provider registration API, consumer request path, lifecycle engine, per-session scoping contract, and plugin trust policy. The AWS Secrets Manager offer and Authsome broker comment are useful implementation/design context, but they do not close the broader security-sensitive API decision.

## What This Changes

The issue proposes a first-class Credential Provider Plugin capability for dynamic credential acquisition, lifecycle management, cross-plugin consumption, and SecretRef/AuthProfileStore bridging.

## Best Possible Solution

Keep the RFC open for maintainer and security-led API design; if accepted, land an additive SDK/runtime contract with an explicit trust model, SecretRef/AuthProfileStore boundaries, per-session scoping, and a proof provider before moving concrete backends into plugins.

Not applicable; this is an RFC for a new capability, not a broken existing behavior report. Source inspection gives a high-confidence current-main status: OpenClaw has static SecretRef and provider auth/profile machinery but no credential-provider lifecycle API.

## Solution Assessment

No for automatic implementation. The direction is plausible and the AWS/Authsome comments are useful, but maintainers need to choose the security model and SDK contract before any backend or lifecycle engine lands.
