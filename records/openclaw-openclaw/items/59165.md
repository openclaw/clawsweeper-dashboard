- `P2`: This is a concrete but broad security/auth-provider feature RFC rather than an active outage or narrow regression.
- `impact:security`: The proposal would define how plugins supply, refresh, share, isolate, and revoke sensitive credential material.
- `impact:auth-provider`: The proposal directly intersects provider authentication, AuthProfileStore bridging, SecretRef-adjacent resolution, and credential routing for model/channel/tool consumers.

## Summary

Keep open: current main still has static SecretRef and provider auth/profile machinery, but no first-class credential-provider plugin registration, lifecycle engine, consumer request API, per-session scoping contract, or trust policy. The AWS Secrets Manager offer, Authsome broker comment, and narrower open SecretRef-provider work are useful adjacent inputs, not a replacement for the core SDK/security decision.

## What This Changes

The issue proposes a new Credential Provider Plugin capability for plugins to supply, refresh, rotate, health-check, and revoke typed credentials across OpenClaw consumers alongside the existing SecretRef system.

## Best Possible Solution

Keep the RFC open for maintainer and security-led SDK design, then land an additive contract with an explicit trust model, SecretRef/AuthProfileStore boundaries, per-session scoping, lifecycle semantics, and at least one proof provider before moving vendor backends to plugins.

## Reproduction Assessment

Not applicable; this is an RFC for a new capability, not a broken existing behavior report. Source and docs inspection give high confidence that current main has SecretRef and provider-auth machinery but no credential-provider lifecycle API.

## Solution Assessment

Unclear for final design and no for automatic implementation. The direction is plausible, and the AWS/Authsome discussion plus narrower open SecretRef PRs are useful, but maintainers need to choose the security model and SDK contract before any backend or lifecycle engine lands.

## Risks / Open Questions

- A credential-provider API would create a new cross-plugin security boundary for raw secrets, refresh tokens, session-scoped credentials, and brokered outbound access.
- The RFC still needs a maintainer choice between providers returning raw credential values to OpenClaw and broker/proxy providers that keep raw secrets outside the OpenClaw process.
- The proposed API overlaps SecretRef, AuthProfileStore, provider setup, plugin config, and tool/skill execution, so an ad hoc implementation could create competing credential sources or unclear fail-closed behavior.
