## Summary

Keep open. Current main still has static SecretRef support and model-provider-scoped auth hooks, but not the cross-domain Credential Provider Plugin capability proposed here; the remaining work is security-sensitive SDK/API design, not stale cleanup.

## What This Changes

The issue proposes a new Credential Provider Plugin capability for cross-domain credential acquisition, lifecycle management, interactive authentication flows, and optional bridges to SecretRef/AuthProfileStore.

## Best Possible Solution

Keep the RFC open for maintainer-led API and security design; if accepted, land an additive versioned SDK capability with explicit trust policy, per-session scoping, SecretRef/AuthProfileStore integration boundaries, and one proof provider.

## Reproduction Assessment

Not applicable; this is an RFC, not a broken existing behavior report. The current-main verification path is source and docs inspection showing no credential-provider registration API and closed, static SecretRef semantics.

## Solution Assessment

Unclear as an implementation plan, but keeping the design discussion open is the right current outcome. The narrowest maintainable path is a maintainer-approved SDK/security contract first, with partial SecretRef-source work treated as adjacent rather than a full replacement.
