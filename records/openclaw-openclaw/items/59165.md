- `P2`: The issue is a broad but concrete security/auth-provider feature request, not an active outage or narrow regression.
- `impact:security`: The proposal defines cross-plugin credential sharing, lifecycle management, and trust boundaries for sensitive credential material.
- `impact:auth-provider`: The proposal directly affects provider auth, AuthProfileStore bridging, model/channel credential routing, and SecretRef-adjacent resolution.

## Summary

Current main still has static SecretRef and model-provider auth/profile machinery, but no cross-domain credential-provider lifecycle API, registration method, consumer request path, per-session scoping contract, or plugin trust policy. The AWS implementation offer and open pluggable SecretRef-source PR are useful adjacent context, but they do not implement this RFC’s broader security-sensitive API.

## What This Changes

The issue proposes a new Credential Provider Plugin capability for typed, lifecycle-managed credentials consumed across models, channels, skills, tools, and plugins, parallel to SecretRef.

## Best Possible Solution

Keep the RFC open for maintainer and security-led API design; if accepted, land an additive SDK/runtime contract with explicit trust policy, SecretRef/AuthProfileStore boundaries, per-session scoping, and a proof provider before moving concrete backends to plugins or ClawHub.

Not applicable; this is an RFC for a new capability, not a broken existing behavior report. The high-confidence verification path is source and docs inspection showing no credential-provider registration API and static env/file/exec SecretRef semantics on current main.

## Solution Assessment

No for automatic implementation. The direction is plausible, and the AWS offer plus prior-art plugin are useful, but the maintainable path is a maintainer-approved SDK and security contract before any provider backend lands.
