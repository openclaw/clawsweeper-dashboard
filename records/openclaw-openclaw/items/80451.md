## Summary

Keep open: current main still lacks a billing-account or quota-bucket identity in model fallback, so same-provider Gemini siblings can remain eligible after a native Google `RESOURCE_EXHAUSTED` monthly-cap failure. The related circuit-breaker and classifier PRs are relevant but do not supersede this same-account sibling suppression request.

## What This Changes

Request account-scoped failover suppression so Gemini sibling models sharing the same Google billing account are skipped after a billing-cap `RESOURCE_EXHAUSTED` failure.

## Best Possible Solution

Define an additive provider/account quota-scope contract, classify native Google monthly-cap errors into the non-transient billing/quota lane, and suppress only matching sibling candidates while preserving cross-provider fallback.

## Reproduction Assessment

Yes, at source level: current main builds explicit sibling fallback candidates, classifies Google `RESOURCE_EXHAUSTED` as `rate_limit`, and has no billing-account suppression key. A live exhausted Google billing account would still be needed to verify the exact log volume.

## Solution Assessment

Unclear: the direction is reasonable, but the maintainable fix depends on a maintainer decision about whether quota scope is keyed by auth profile, explicit billing-account metadata, provider-owned account/project metadata, or a new quota-bucket contract.
