## Summary

Current main still has the same-account Gemini failover gap, and the adjacent open quota/classification PRs do not yet provide a billing-account scope that can suppress within-run sibling model attempts. This should stay open for maintainer API/product direction rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to skip Gemini fallback candidates that share a billing account after a Google monthly spend-cap RESOURCE_EXHAUSTED failure.

## Best Possible Solution

Define a provider/auth billing-account scope signal, classify terminal Google spend-cap failures precisely, and suppress only same-scope fallback candidates for the current run plus an appropriate short host cooldown.

## Reproduction Assessment

Yes, from source inspection: configure a Gemini primary, an OpenAI fallback, and Gemini sibling fallbacks, then a Google RESOURCE_EXHAUSTED/429 failure is treated as rate_limit and different Gemini models remain eligible through model-scoped cooldown behavior. A live exhausted Google billing account would still be needed to confirm the reporter's exact journal volume.

## Solution Assessment

Unclear as an implementation path: the requested direction is sound, but maintainers need to decide whether account scope comes from provider metadata, auth-profile/account IDs, a new billingAccount field, or coordination with the open quota circuit-breaker work. A provider-wide skip would be a risky overcorrection for multi-account Google setups.
