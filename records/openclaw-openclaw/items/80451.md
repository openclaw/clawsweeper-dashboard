## Summary

Keep open. Current main still lacks an account-scoped failover suppression path: configured Gemini siblings remain candidates, the fallback loop continues after each failed candidate, and Google-style 429/RESOURCE_EXHAUSTED remains in the rate_limit lane rather than a terminal billing-account scope.

## What This Changes

Request account-scoped failover suppression so Gemini sibling models on the same Google billing account are skipped after a terminal RESOURCE_EXHAUSTED monthly cap.

## Best Possible Solution

Define a provider/account-scope quota contract, classify terminal Google billing caps into that scope, and make fallback skip only candidates sharing the resolved account within the current run, with focused regression coverage for a Gemini-to-OpenAI-to-Gemini chain.

## Reproduction Assessment

Yes from source inspection: current main builds the configured Gemini sibling candidates and has no per-run account-scope suppression after the first RESOURCE_EXHAUSTED failure. I did not reproduce the reporter's exact 15-line cascade against a live exhausted Google billing account.

## Solution Assessment

Unclear: the requested direction is reasonable, but the best implementation needs maintainer/API judgment on whether the account scope comes from auth profiles, provider metadata, or a new public config field. A provider-only skip would be too coarse.
