## Summary

The current failover loop still has the same-account gap: configured fallbacks are tried in order, Google RESOURCE_EXHAUSTED is classified as rate_limit, and provider cooldown skipping only applies when auth profiles are already unavailable before an attempt. This is not safe to close or auto-fix because the requested account scope needs a product/API decision and overlaps open quota/failover work.

## What This Changes

The issue requests billing-account-aware failover suppression so Gemini sibling models sharing one Google billing account are skipped after an account-scoped cap failure.

## Best Possible Solution

Define a small provider/account-scope contract for terminal billing/quota failures, then teach model fallback to carry that fact through the current run and skip only candidates that share the same resolved account scope, with focused regression coverage for a Gemini-to-OpenAI-to-Gemini chain.

## Reproduction Assessment

Yes from source inspection, but not from a live exhausted Google billing account: current main builds the configured sibling candidates and has no per-run billing-account suppression after the first RESOURCE_EXHAUSTED failure. A live journal reproduction would still be needed to verify the reporter’s exact 15-line cascade.

## Solution Assessment

Unclear: the direction is reasonable, but choosing between provider-level cooldown, auth-profile/account scoping, or a new billingAccount config field is a maintainer product/API decision. A provider-only skip would be too coarse for users with multiple Google accounts or Vertex projects.
