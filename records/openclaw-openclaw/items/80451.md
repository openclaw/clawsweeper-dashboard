## Summary

Current main still has the reported same-account failover gap: configured sibling Gemini models remain candidates, `RESOURCE_EXHAUSTED`/HTTP 429 is still routed through `rate_limit`, and the cooldown checks are provider/profile/model scoped rather than billing-account scoped. This should stay open because the durable fix needs maintainer API/product direction, not conservative cleanup.

## What This Changes

Request to skip Gemini fallback siblings that share a Google billing account after an account-scoped billing or quota cap failure.

## Best Possible Solution

Define a small provider/account-scope contract for terminal billing or quota failures, then carry that scope through one fallback run so only candidates sharing the resolved account scope are skipped, with focused Gemini-to-OpenAI-to-Gemini coverage.

## Reproduction Assessment

Yes from source inspection: current main builds the configured sibling candidates and has no per-run billing-account suppression after the first `RESOURCE_EXHAUSTED` failure. A live exhausted Google billing account would still be needed to verify the reporter's exact journal volume.

## Solution Assessment

Unclear: the requested direction is reasonable, but choosing provider-level cooldown, auth-profile/account scoping, or a new `billingAccount` config field is a maintainer API/product decision. A provider-only skip would be a risky overcorrection for multi-account Google setups.
