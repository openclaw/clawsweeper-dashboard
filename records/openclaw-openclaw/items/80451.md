## Summary

Current main still has the reported account-scope gap: Gemini sibling models remain configured fallback candidates, Google `RESOURCE_EXHAUSTED`/HTTP 429 is still treated as `rate_limit`, and the current cooldown state is provider/profile/model scoped rather than billing-account scoped. This should stay open because the durable fix needs maintainer/API direction on how OpenClaw identifies shared billing-account scope.

## What This Changes

Request billing-account-aware failover so a hard Gemini spending-cap failure suppresses sibling Gemini attempts on the same account.

## Best Possible Solution

Define a small provider/account-scope contract for terminal billing or quota failures, carry that scope through one fallback run, and skip only candidates sharing the resolved account scope, with focused Gemini-to-OpenAI-to-Gemini coverage.

## Reproduction Assessment

Yes from source inspection: current main builds and walks the configured sibling candidates, and there is no per-run billing-account suppression after the first RESOURCE_EXHAUSTED failure. Live confirmation of the exact journal volume would require an exhausted Google billing account.

## Solution Assessment

Unclear: the requested direction is reasonable, but choosing provider-level cooldown, auth-profile/account scoping, or a new billingAccount-style config surface is a maintainer API/product decision. A provider-only skip would be risky for multi-account Google setups.
