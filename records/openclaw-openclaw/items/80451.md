## Summary

Keep open. Current main still lacks a billing-account scope concept for model fallback: Google sibling models remain configured candidates, native Google `RESOURCE_EXHAUSTED`/HTTP 429 still lands in the transient rate-limit lane, and rate-limit cooldowns are deliberately model-scoped rather than billing-account-scoped.

## What This Changes

The issue asks OpenClaw failover to skip sibling Gemini models on the same Google billing account after a billing-account cap failure.

## Best Possible Solution

Define a provider/account-scope contract for terminal billing or quota failures, then carry that resolved scope through a fallback run so only candidates sharing the same account scope are skipped.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configured Gemini siblings remain candidates, HTTP 429/RESOURCE_EXHAUSTED remains rate_limit, and model-scoped cooldown bypass allows a different Gemini model to be attempted. A live over-cap Google account would only be needed to confirm the reporter's exact journal volume.

## Solution Assessment

Unclear as an implementation plan. The direction is valid, but maintainers need to choose account/profile/provider scoping before a safe fix; a provider-wide skip would be too broad for multi-account and proxy setups.
