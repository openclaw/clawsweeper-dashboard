## Summary

The report is concrete, but the failure is isolated to the downstream QClaw desktop renderer while the reporter says the OpenClaw Dashboard in Chrome, Gateway, and WeChat channel are healthy; current OpenClaw source has no QClaw app code and the official macOS WebChat path is native SwiftUI rather than an embedded V8 renderer.

## What This Changes

The issue reports that QClaw v0.2.24 on macOS ARM64 lets the user type in the desktop chat box but cannot send because the embedded renderer appears stuck in a V8 compilation loop.

## Best Possible Solution

Route the QClaw-only renderer hang to the QClaw desktop maintainers; OpenClaw should keep the native WebChat/Gateway path and only reopen this repo path if current OpenClaw Mac app or browser Dashboard reproduces the same send failure.
