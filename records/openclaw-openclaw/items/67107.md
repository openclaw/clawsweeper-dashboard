## Summary

Keep this open: current main still has the reported installer/docs UX gap. The Node >=22.14 runtime floor appears intentional, but `install.sh` still treats all Darwin hosts as generic macOS and proceeds into Homebrew `node@24` installation without a Big Sur/macOS 11 preflight or public install-doc warning.

## What This Changes

The issue requests that OpenClaw either support Node.js on macOS Big Sur or clearly fail/document unsupported macOS hosts before attempting a Node/Homebrew install.

## Best Possible Solution

Preserve the Node >=22.14 runtime contract, add a narrow macOS preflight or friendly failure path for unsupported old macOS before Homebrew Node installation, and update the public install/Node docs so Big Sur-era hosts know the supported path or limitation up front.

## Reproduction Assessment

Yes for the source-level UX gap: current main shows the Darwin path proceeds to Homebrew Node 24 without checking the macOS product version, and the install docs do not document Big Sur/macOS 11 limits. Live Big Sur logs would only refine the exact upstream failure wording.

## Solution Assessment

Yes: a narrow installer/docs fix is the best way to solve this issue. Changing the runtime floor to Node 18 would conflict with the current package engine and runtime guard, while clearer preflight/docs directly address the user-visible failure.
