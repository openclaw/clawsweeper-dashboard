## Summary

Current main still lacks the Big Sur/macOS-version preflight or public install-doc warning needed for this report, so this should stay open as a focused installer/docs UX bug. The runtime Node floor is intentionally Node >=22.14; the remaining gap is that the macOS installer still enters the generic Homebrew Node 24 path before explaining what older macOS hosts can or cannot do.

## What This Changes

The issue requests that OpenClaw handle macOS Big Sur installation failure by supporting a compatible Node path or failing early with clear unsupported-platform documentation instead of surfacing late Homebrew/Node errors.

## Best Possible Solution

Preserve the Node >=22.14 runtime contract, add an installer preflight or fallback that treats older macOS releases deliberately before trying Homebrew Node 24, and update the public install/Node docs to describe the supported macOS/Node path or unsupported Big Sur behavior.

## Reproduction Assessment

Yes for the source-level UX gap: current main can be inspected to show Darwin is only mapped to generic macOS and the Node install path still tries Homebrew node@24 without a product-version check. Live Big Sur failure logs would improve confidence in the exact Homebrew wording, but they are not needed to verify the missing OpenClaw guard.

## Solution Assessment

Yes, the best path is a narrow installer/docs fix rather than Node 18 support. Node 22.14 remains the runtime floor and Node 24 remains the default, so the installer should either accept a valid Node 22.14+ path or explain why the host cannot use the default path before attempting a failing Homebrew install.
