## Summary

Keep open: current main still routes all Darwin hosts through generic macOS handling and the Homebrew node@24 bootstrap path when Node is missing or too old, while install docs still present macOS support without an old-macOS caveat. Node 18 support should not be added, but the Big Sur failure path remains a valid installer/docs bug.

## What This Changes

The issue asks for the macOS installer to handle Big Sur’s Node/Homebrew incompatibility gracefully or document the unsupported path before installation fails.

## Best Possible Solution

Keep the Node >=22.16 runtime contract, add a macOS version-aware guard before the Homebrew node@24 bootstrap path, and update install docs with clear old-macOS guidance that points users to a supported Node 22 path or Docker instead of Node 18.

## Reproduction Assessment

Yes for source-level reproduction: on Darwin with no supported Node, current main detects only generic macOS, checks Node, and then attempts Homebrew node@24 without a Big Sur/macOS version guard. I did not live-run on Big Sur, so this is source_reproducible rather than reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is installer/docs handling that preserves Node >=22.16 and avoids the blind Homebrew node@24 path on old macOS. Adding Node 18 support would conflict with current package metadata and runtime guard.
