## Summary

Keep open: current main still routes all Darwin hosts through the generic macOS installer path and attempts Homebrew `node@24` when Node is missing or too old, while install docs still say the installer handles Node automatically without an old-macOS caveat. The actionable bug is installer/docs clarity or fallback for Big Sur, not adding Node 18 support.

## What This Changes

The issue asks for the installer to handle macOS Big Sur machines more gracefully when the default Node 24/Homebrew path cannot work.

## Best Possible Solution

Keep the Node >=22.19 contract, add a macOS-version-aware guard before the Homebrew Node 24 path, direct Big Sur users to the supported Node 22/local-prefix path or a clear unsupported message, and update install docs/tests.

## Reproduction Assessment

Yes for source-level reproduction: on Darwin with missing or too-old Node, current main reaches the Homebrew `node@24` installer path without checking the macOS product version. I did not live-run Big Sur, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes: the narrow maintainable solution is installer/docs handling around the existing Node 22.19+ contract, not adding Node 18 runtime support. The existing `install-cli.sh` local-prefix Node 22 path gives a supported direction to document or recommend.
