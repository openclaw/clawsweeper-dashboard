## Summary

Keep open: current main still sends old macOS hosts through the generic macOS installer path and Homebrew `node@24` bootstrap, while install docs still describe macOS installer support without an old-macOS caveat. The actionable bug is installer/docs clarity around the existing Node requirement, not adding Node 18 runtime support.

## What This Changes

The issue asks OpenClaw to handle macOS Big Sur systems that cannot satisfy the current Node requirement through the installer instead of failing late or implying Node 18 should work.

## Best Possible Solution

Keep the Node >=22.16 runtime contract, add a macOS-version-aware guard before the Homebrew node@24 bootstrap path, and update install docs with clear old-macOS guidance and supported alternatives.

## Reproduction Assessment

Yes for source-level reproduction: on Darwin with no supported Node, current main detects only generic macOS and then attempts Homebrew node@24 without checking the macOS version. I did not live-run Big Sur, so this is source_reproducible rather than reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is installer/docs handling that preserves Node >=22.16 and avoids the blind Homebrew node@24 path on old macOS. Adding Node 18 support would conflict with current package metadata and runtime guard.
