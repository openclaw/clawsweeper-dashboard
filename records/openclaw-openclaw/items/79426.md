## Summary

Keep open. Current main still drops the explicit cron timeout signal when the payload value equals `agents.defaults.timeoutSeconds`, and this PR targets that real source-level bug, but the external PR still lacks required after-fix proof from a real cron run.

## What This Changes

The PR plumbs a `runTimeoutOverrideMs` field from isolated cron preparation through embedded runner attempts so the LLM idle watchdog can honor explicit cron `timeoutSeconds` values even when they equal the agent default.

## Best Possible Solution

Land the explicit cron timeout signal after maintainer review and redacted real cron proof, then use that merged fix to resolve the related cron idle-timeout reports that share this root cause.

## Reproduction Assessment

Yes, source-level. Current main collapses explicit cron `timeoutSeconds` into `timeoutMs`, then drops `runTimeoutMs` when it numerically equals the configured default, while the resolver only honors uncapped cron behavior when `runTimeoutMs` is present.

## Solution Assessment

Yes. Carrying the prepared explicit-timeout fact from cron into the embedded attempt is narrower and more maintainable than changing the global idle-timeout policy or adding a new config knob; merge should still wait for real behavior proof.
