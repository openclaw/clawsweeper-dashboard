## Summary

Keep open. The PR still contains useful feature work for Gateway recovery, but current main has not implemented the watchdog command and the branch has blocking package/runtime, validation, proof, and security issues that require contributor and maintainer follow-up rather than cleanup closure.

## What This Changes

The PR adds startup-error file persistence, fatal-error wiring, a `gateway watchdog` CLI subcommand, and a Bash watchdog script that polls Gateway health, restarts the Gateway, logs events, and sends desktop notifications.

## Best Possible Solution

Land a rebased and reviewed implementation only after the watchdog boundary is clear; the final shape should use the current stability/fatal-hook architecture, package runtime assets correctly, keep secrets out of argv, validate CLI input, document the command, and include focused tests plus real runtime proof.

No runtime command was run because this was a read-only review.

## Solution Assessment

No. The feature direction is useful, but this exact branch is not the best merge shape until maintainers choose the watchdog boundary and the branch is rebased with the functional and security blockers fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package file list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file, so the feature is unusable outside a source checkout unless the script is packaged or moved into a shipped runtime asset.
  - confidence: 0.94
- **[P1] Read the token from environment defaults:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI forwards `OPENCLAW_GATEWAY_TOKEN`, but the script initializes `TOKEN` to an empty string and only updates it from `--token`. Env-authenticated gateways will be probed without the configured token and can be incorrectly treated as down.
  - confidence: 0.9
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or exit. On normal successful startup this block never executes, leaving stale `startup-error.txt` content visible after the Gateway has recovered.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:585-590`
  - body: Both numeric parsers can return `NaN`, and that value is forwarded into shell arithmetic. Inputs such as `--interval foo` or `--max-attempts foo` should fail in Commander before starting the watchdog script.
  - confidence: 0.9
- **[P2] Handle child process spawn errors explicitly:** `src/cli/gateway-cli/run.ts:615`
  - body: Wrapping `spawn("bash", ...)` in try/catch does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned child process as an `error` event. Add an error listener so the CLI reports a controlled failure.
  - confidence: 0.86
- **[P2] Keep gateway tokens out of child process argv:** `src/cli/gateway-cli/run.ts:604`
  - body: When `--token` is supplied, the command appends the gateway token to the long-running Bash child process arguments. Since local process listings can expose argv, pass the token only through the child environment and make the script read that env default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention: the watchdog CLI can forward a gateway auth token into long-running child-process argv, which may expose it through local process listings.

Concerns:

- **[medium] Gateway token can be exposed in watchdog argv:** `src/cli/gateway-cli/run.ts:604`
  - body: `opts.token` is appended as `--token <value>` when spawning the Bash watchdog. Because the watchdog is intended to run continuously, that token can remain visible in process listings; use `OPENCLAW_GATEWAY_TOKEN` or another non-argv secret path instead.
  - confidence: 0.82
