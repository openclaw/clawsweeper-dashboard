## Summary

Keep open: current main still lacks the proposed Gateway watchdog command/startup-error artifact, and the branch has concrete merge blockers rather than an obsolete implementation.

## What This Changes

The PR adds startup-error persistence, fatal-error wiring, and a `openclaw gateway watchdog` command that launches a Bash watchdog script for health checks, restarts, logging, and desktop notifications.

## Best Possible Solution

Land a rebased maintainer-approved implementation, or a narrower successor, that fits the current stability/fatal-hook architecture, packages any runtime watchdog asset, validates CLI inputs, keeps secrets out of process argv, documents the operator contract, and includes focused tests plus real behavior proof.

No runtime command was run because this was a read-only review.

## Solution Assessment

No. The feature direction is useful, but this exact branch should not merge until maintainers choose the watchdog boundary and the branch is rebased onto the current stability/fatal-hook architecture with the functional and security blockers fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package `files` list does not include that script. Installed package users would get a `gateway watchdog` command pointing at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P1] Read the token from environment defaults:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI forwards `OPENCLAW_GATEWAY_TOKEN`, but the script initializes `TOKEN` to an empty string and only updates it from `--token`. Env-authenticated gateways will be probed without the configured token and can be incorrectly treated as down.
  - confidence: 0.9
- **[P2] Clear startup errors when startup succeeds:** `src/cli/gateway-cli/run.ts:462-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to keep running after successful startup. On normal startup this block never executes, leaving stale `startup-error.txt` content visible after recovery.
  - confidence: 0.88
- **[P2] Reject invalid numeric options before spawning:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN`, and that value is forwarded into shell arithmetic. Inputs such as `--interval foo` or `--max-attempts foo` should fail in Commander before starting the watchdog script.
  - confidence: 0.9
- **[P2] Handle child process spawn errors explicitly:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in try/catch does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned child process as an `error` event. Add an error listener so the CLI reports a controlled failure.
  - confidence: 0.86
- **[P2] Keep gateway tokens out of child process argv:** `src/cli/gateway-cli/run.ts:603-605`
  - body: When `--token` is supplied, the command appends the gateway token to the long-running Bash child process arguments. Since local process listings can expose argv, pass the token only through the child environment and make the script read that env default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention: the watchdog CLI can forward a gateway auth token into a long-running child-process argv, which may expose it through local process listings.

Concerns:

- **[medium] Gateway token can be exposed in watchdog argv:** `src/cli/gateway-cli/run.ts:604`
  - body: `opts.token` is appended as `--token <value>` when spawning the Bash watchdog. Because the watchdog is intended to run continuously, that token can remain visible in process listings; use `OPENCLAW_GATEWAY_TOKEN`/script env defaults instead.
  - confidence: 0.82
