## Summary

Keep open. Current main still lacks this PR's `gateway watchdog` command and `startup-error.txt` writer, but the PR head is not merge-ready: it is currently unmergeable, overlaps broader open rescue-watchdog work, and still has concrete blocking defects in packaging, auth defaults, startup-error cleanup, option validation, and child-process error handling.

## What This Changes

The PR adds startup-error file persistence for fatal CLI/Gateway startup failures and registers an `openclaw gateway watchdog` subcommand that spawns a Bash health-check/restart/notification script.

## Best Possible Solution

Land a rebased implementation only after the watchdog boundary is clear. The final shape should package any runtime asset, honor Gateway auth defaults, clear diagnostics at the actual successful startup boundary, use current fatal-hook/stability-bundle architecture, validate CLI input before spawning shell code, handle child-process errors, document the public command, and include the required changelog entry and focused tests.

## Reproduction Assessment

Yes for the review findings. The breakages are reproducible by static inspection of the PR head: the command points to an unpackaged script, the script ignores the forwarded env token, startup-error cleanup sits after a long-running loop, parsers can forward `NaN`, and `spawn` errors need an `error` listener.

## Solution Assessment

No. The PR addresses a real failure-recovery request, but this branch is not the best mergeable solution until it is rebased and its shell watchdog scope is reconciled with the open integrated rescue-watchdog design.

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the npm package file list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P1] Read the watchdog token from the environment default:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI advertises `OPENCLAW_GATEWAY_TOKEN` as the default and forwards it to the child environment, but the script initializes `TOKEN` to an empty string unless `--token` is passed. Token-protected gateways using env auth will be treated as unhealthy and may enter restart/manual-intervention flows incorrectly.
  - confidence: 0.9
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or restart handling. On normal successful startup the block is not reached, leaving stale `startup-error.txt` content visible after recovery.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN`, and those values are forwarded into shell arithmetic. Inputs like `--interval foo` or `--max-attempts foo` should fail with a Commander validation error instead of starting a script that aborts with an opaque shell error.
  - confidence: 0.9
- **[P2] Handle child process spawn errors with an error listener:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in `try/catch` does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, missing Bash can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
- **[P3] Add the required changelog entry for the new command:** `src/cli/gateway-cli/run.ts:576`
  - body: This PR adds a user-facing `gateway watchdog` feature but does not touch `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to have a changelog entry, so the release notes would omit the new command.
  - confidence: 0.82
