## Summary

Keep open. Current main still lacks the proposed watchdog command/startup-error file, and this PR remains unmerged with missing real behavior proof, merge-dirty state, overlapping watchdog direction, and blocking packaging/runtime defects.

## What This Changes

The PR adds startup-error persistence, wires fatal CLI/Gateway errors to it, and registers a `gateway watchdog` command that runs a Bash health-check/restart/notification script.

## Best Possible Solution

Choose the canonical Gateway recovery design, then land a rebased implementation that uses current fatal-hook/stability architecture, packages runtime assets, validates input, documents operator behavior, and includes focused tests plus real behavior proof.

No live command was run because this was a read-only review.

## Solution Assessment

No. The recovery feature is useful, but this patch should not merge until maintainers settle the watchdog boundary and the branch fixes the runtime defects, stale rebase state, tests/docs, and real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Package the watchdog script with the CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The new command resolves `scripts/openclaw-watchdog.sh` at runtime, but package metadata does not include that script path. Installed package users would get a `gateway watchdog` command that points at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P1] Honor the watchdog token environment default:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI forwards `OPENCLAW_GATEWAY_TOKEN` to the child process, but the script initializes `TOKEN` to an empty string and only sets it from `--token`. Env-authenticated gateways will be probed without the token and can be incorrectly treated as down.
  - confidence: 0.9
- **[P2] Clear startup errors after the gateway starts:** `src/cli/gateway-cli/run.ts:462-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to keep running after successful startup. On normal startup this block never executes, leaving stale `startup-error.txt` content visible after recovery.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN`, and that value is forwarded into shell arithmetic. Inputs such as `--interval foo` or `--max-attempts foo` should fail in Commander before starting the watchdog script.
  - confidence: 0.9
- **[P2] Handle watchdog spawn errors on the child process:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in `try/catch` does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned child process as an `error` event. Add an error listener so the CLI reports a controlled failure.
  - confidence: 0.86
