## Summary

Keep open for maintainer review. The patch looks coherent and the supplied proof is sufficient, but it intentionally makes an `operator.read` inventory request start or connect configured MCP transports, which needs gateway/security owner acceptance before merge.

## What This Changes

The PR makes `tools.effective` materialize policy-filtered bundled MCP tools into an `mcp` source group and updates the protocol, docs, UI labels, i18n output, and tests.

## Best Possible Solution

Land this narrow `tools.effective` fix only if gateway/security owners accept live MCP listing as a session-inventory read side effect; otherwise require an opt-in or cached catalog design before merge.

## Reproduction Assessment

Yes, from source: current main's `tools.effective` handler uses only the static inventory resolver while the embedded run path materializes MCP tools later. I did not run the harness in this read-only review, but the PR includes before/after Gateway RPC proof with a real stdio MCP server.

## Solution Assessment

Yes if maintainers accept the read-path side effect: the PR reuses the existing session MCP runtime, materializer, and final policy path instead of creating a second policy source. If `operator.read` must remain side-effect-free, the safer solution is an explicit opt-in or cached MCP catalog source.
