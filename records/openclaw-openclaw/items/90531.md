## Summary

Keep open: current main still carries Telegram sender identity into the inbound context, but the shared user-turn transcript path only persists text/provenance/media and does not write `__openclaw` sender metadata into the JSONL user record.

## What This Changes

Persist sender metadata such as `__openclaw.senderId` for Telegram group-chat user messages in session JSONL so transcript readers can distinguish participants.

## Best Possible Solution

Propagate existing inbound sender identity through the shared user-turn transcript recorder into a canonical `__openclaw` metadata block, with Telegram group coverage and no new config surface.

## Reproduction Assessment

Yes, source-level: Telegram sets `SenderId`, but the current user-turn transcript writer does not copy that identity into the persisted user message. A live Telegram JSONL repro was not run during this read-only review.

## Solution Assessment

Yes: the best fix is a narrow shared transcript-metadata propagation from existing `MsgContext` sender fields, not a Telegram-only ad hoc JSONL writer or a new config option.
