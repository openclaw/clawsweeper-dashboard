## Summary

Keep open: current main and v2026.6.8 still omit sender metadata from user-turn JSONL persistence even though Telegram and the shared inbound context already carry the sender identity. The remaining work is already represented by https://github.com/openclaw/openclaw/pull/90552, which is open and still needs maintainer review before this issue can close.

## What This Changes

The issue requests that group chat user messages persisted to agent session JSONL include a `__openclaw` sender metadata envelope such as `senderId`, matching direct-chat attribution behavior.

## Best Possible Solution

Review and land or replace the open shared transcript fix that forwards existing SenderId/SenderName/SenderUsername into `__openclaw` while keeping phone/E.164 metadata out of persisted JSONL.

## Reproduction Assessment

Yes, source-level. Telegram and the shared channel context set SenderId/SenderName/SenderUsername, but current main's userTurnInput and persisted-message builder omit those fields before JSONL persistence; I did not run a live Telegram group repro.

## Solution Assessment

Yes, with maintainer privacy review. The shared user-turn transcript path is the right boundary instead of a Telegram-only writer, and https://github.com/openclaw/openclaw/pull/90552 is the current canonical fix candidate.
