## Summary

Keep this open: current main and the latest release still preserve the helper-level Unix group-kill default when `detached` is omitted, although the named supervisor/service-managed path now passes actual detached state and is better protected than the original report described.

## What This Changes

The issue requests changing `killProcessTree` so omitted `{ detached: false }` cannot group-kill a non-detached macOS child and SIGTERM the gateway process group.

## Best Possible Solution

Have maintainers choose and document the permanent kill-tree contract, then land a focused fix that keeps supervisor/service-managed safety while preserving intentional detached process-group cleanup with regression tests.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still group-kill on Unix when `detached` is omitted. I did not live-reproduce the macOS Telegram SIGTERM loop, and the current supervisor caller now passes detached state for service-managed and fallback children.

## Solution Assessment

Unclear. The suggested kill-side PGID invariant addresses the helper hazard, but a direct `ps -p <pid>` leader check could lose the tested ability to kill a still-live detached process group after the leader PID exits.
