## Summary

Keep open. Current `main` and `v2026.4.29` still make Unix `killProcessTree` use negative-PID group kill unless callers explicitly pass `detached:false`, and current tests preserve that default despite the documented gateway SIGTERM hazard.

## What This Changes

The issue requests that Unix process-tree termination verify a PID is actually a process-group leader before using `process.kill(-pid, ...)`.

## Best Possible Solution

Enforce the process-group-leader precondition inside `kill-tree` itself, preserving direct-PID behavior for non-leaders and keeping known detached group leaders covered.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on Unix, call `killProcessTree(pid)` for a non-group-leader child and current code will attempt `process.kill(-pid, "SIGTERM")`; the live macOS launchd reproduction still needs a real host check after a patch.

## Solution Assessment

Yes, the suggested direction is the narrowest maintainable fix if implemented with a bounded PGID check and regression tests. Auditing every caller would leave the same unsafe default available for future callers.
