## Summary

Keep open: current main and latest release still preserve Unix group-kill when `detached` is omitted, and this issue now has an open closing fix candidate at https://github.com/openclaw/openclaw/pull/94697. The issue should stay open until that PR lands or maintainers choose a different bounded process-tree fix.

## What This Changes

The issue reports that Unix process-tree cleanup can SIGTERM a macOS launchd-managed gateway when callers omit `detached:false` and `killProcessTree` group-kills a PID that is not its own process-group leader.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94697, or replace it with an equivalent focused fix that prevents non-leader group kills while preserving valid detached process-tree cleanup.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still default omitted-detached Unix cleanup to group-kill, and current callers still omit the actual spawn detachment fact. I did not run the live macOS launchd Telegram gateway scenario in this read-only review.

## Solution Assessment

Yes, keeping this issue open behind the open closing PR is the best current path. The durable fix should either verify process-group leadership in the helper or carry actual spawn-time detached facts through every cleanup path; a broad unqualified default flip alone is not the best fix.
