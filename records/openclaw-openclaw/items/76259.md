## Summary

Keep open. Current `main` and the shipped `v2026.4.29` tag still default omitted Unix `killProcessTree` options to group-kill, current tests assert that behavior, and several callers still omit `detached`, so the reported regression remains valid and narrow.

## What This Changes

The issue reports that `killProcessTree` still defaults omitted Unix `detached` options to group-kill, allowing non-group-leader children to SIGTERM a macOS launchd-managed gateway.

## Best Possible Solution

Move the process-group safety invariant into `src/process/kill-tree.ts` so negative-PID kills only run for verified or explicitly known process-group leaders, with omitted-option callers covered by regression tests.

## Reproduction Assessment

Yes. A focused reproduction is `killProcessTree(pid)` or `killProcessTree(pid, { graceMs })` on Unix, which current tests show calls `process.kill(-pid, "SIGTERM")`; the issue also provides macOS launchd and Telegram trace evidence for the production path.

## Solution Assessment

Yes. A kill-side guard is the narrowest maintainable fix because it protects existing and future omitted-option callers while preserving group-kill where the target is actually a detached process group leader.
