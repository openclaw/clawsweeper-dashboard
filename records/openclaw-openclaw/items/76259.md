## Summary

Keep open. Current `main` and `v2026.4.29` still default omitted `killProcessTree` options to Unix group-kill, while the existing test suite preserves that default despite the documented gateway SIGTERM hazard for non-group-leader children.

## What This Changes

The issue requests making POSIX process-tree termination safe when callers omit `detached`, so a non-group-leader child cannot trigger a gateway-wide group kill.

## Best Possible Solution

Move the process-group safety invariant into `src/process/kill-tree.ts`: only use negative-PID group kill when the target is verified to be its own process-group leader, and cover omitted-option callers with regression tests.

## Reproduction Assessment

Yes. A focused current-main reproduction is `killProcessTree(pid)` or `killProcessTree(pid, { graceMs })` on Unix, which reaches `process.kill(-pid, 'SIGTERM')`; the reporter's macOS launchd trace supports the full gateway symptom.

## Solution Assessment

Yes. A kill-side guard is the narrowest maintainable fix because it protects current and future callers that omit detachment state while preserving group kill for verified detached process groups.
