The issue asks OpenClaw to prevent Unix `killProcessTree` callers that omit detachment metadata from group-killing the gateway process group on macOS service-managed installs.

## Best Possible Solution

Land a narrow core process-helper fix that makes group-kill safe by construction, audits current callers, and adds regression coverage without unrelated PR churn.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still default Unix `killProcessTree` and `signalProcessTree` to negative-PID group kill when detachment is omitted, and omitted-option callers remain. I did not run the macOS launchd scenario, so this is not a live reproduced result.

## Solution Assessment

Yes in direction: the best fix belongs in the shared process-tree helper plus caller audit, not as another one-off caller opt-in. The patch should either verify process-group leadership before negative-PID signals or otherwise require explicit proven detachment while preserving detached descendant cleanup.
