## Summary

Keep open. Current main and the latest `v2026.5.2` release still default omitted Unix `killProcessTree` calls to process-group termination, current tests preserve that behavior, and open PR #76398 explicitly references this issue as the fix candidate.

## What This Changes

The issue reports that `killProcessTree` still defaults to Unix process-group termination when callers omit `detached`, which can SIGTERM a launchd-managed gateway.

## Best Possible Solution

Review and land or close #76398; the desired end state is direct-pid termination by default and process-group termination only for explicitly detached process groups.

## Reproduction Assessment

Yes. Source inspection shows current `killProcessTree(pid)` and `killProcessTree(pid, { graceMs })` on Unix still call the negative-PID group-kill path, and the current test suite asserts that default behavior.

## Solution Assessment

Yes. A kill-side safe default is the narrow maintainable fix because it protects current and future omitted-option callers while preserving explicit group-kill for detached children.
