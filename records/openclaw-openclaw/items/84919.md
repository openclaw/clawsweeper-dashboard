## Summary

Keep open: current main and v2026.5.19 still construct Gemini GenerateContent payloads that can combine `cachedContent` with request-level `systemInstruction`, `tools`, and `toolConfig`. The linked fix PR at https://github.com/openclaw/openclaw/pull/84958 is open and includes useful live Gemini proof, so this issue should remain open until that PR or an equivalent complete fix lands.

## What This Changes

The issue reports that Gemini cached-content requests with `cacheRetention` and tools can send incompatible request-level prompt/tool fields alongside `cachedContent`.

## Best Possible Solution

Review and land a complete fix through https://github.com/openclaw/openclaw/pull/84958 or an equivalent PR, covering both explicit `cachedContent` and managed `cacheRetention` paths before closing this issue with commit and release proof.

## Reproduction Assessment

Yes for source-level reproduction: current main visibly emits the conflicting request fields, and current tests assert that shape. I did not run a live Gemini API request during this read-only issue review.

## Solution Assessment

Yes. The maintainable fix is to keep cached-content handles on GenerateContent requests while moving or omitting incompatible prompt/tool request fields; the linked open PR is the active implementation path, but the issue should stay open until it merges.
