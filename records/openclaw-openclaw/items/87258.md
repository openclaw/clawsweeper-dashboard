## Summary

Keep open: current `main` and the latest release still compute the file-log hostname once at module load, so an empty startup value can be cached as `unknown` for the process lifetime. The issue is narrow, source-backed, and fits a focused logger fix with regression coverage.

## What This Changes

The issue requests that structured file-log metadata resolve the gateway hostname lazily so macOS logs do not get stuck with `hostname: "unknown"` after startup.

## Best Possible Solution

Keep the existing `hostname` JSONL field but resolve it at write time or through a small recoverable helper, preserving log format and redaction behavior while adding coverage for an initially empty hostname.

## Reproduction Assessment

Yes for the source-level cache bug: if `os.hostname()` is empty when the module loads, current `main` stores `unknown` in `HOSTNAME` and reuses it for later file-log records. I did not live-reproduce the intermittent macOS Node startup timing in this Linux checkout.

## Solution Assessment

Yes: lazy hostname resolution inside the file-log path is the narrowest maintainable fix. It should keep the existing top-level `hostname` field, avoid new config, and add focused regression coverage for an initially empty hostname.
