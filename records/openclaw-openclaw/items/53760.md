## Summary

Keep this issue open. Current main still does not enforce DM-level `toolsBySender`: direct session keys are deliberately excluded from the group-policy resolver, Signal only exposes `toolsBySender` under group config, and Telegram's typed direct `tools`/`toolsBySender` config is not connected to the enforced tool-filtering path. The prior ClawSweeper comment matches current main after re-checking the same source paths.

## What This Changes

Add enforced DM-level `toolsBySender` support for direct-message channels such as Signal and Telegram so allowlisted senders can receive different tool access policies.

## Best Possible Solution

Implement a generic direct-message sender tool-policy resolver parallel to the group resolver, backed by verified inbound sender identity, shared config/schema/docs for DM-level `tools` and `toolsBySender`, and regression tests for Signal and Telegram proving one allowlisted DM sender can retain tools while another is denied.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to evaluate a Telegram or Signal direct session key with sender metadata: the session parsers do not produce group context and `resolveGroupToolPolicy` returns no policy, so DM-level `toolsBySender` cannot affect the enforced tool pipeline.

## Solution Assessment

Yes. The best direction is a core/channel-policy contract for direct-message sender tool policy; prose workspace instructions or channel-local shortcuts would not provide the same enforceable, shared policy boundary and could drift from bundled-tool filtering.
