## Summary

Keep this issue open. Current main still has only a group/channel tool-policy resolver: direct session keys are excluded from the trusted conversation parser, the resolver returns no policy when there is no group id, and the core plus bundled-tool filtering paths only consume that group-scoped result. Telegram has typed direct `toolsBySender` config, but it is not wired into enforcement; Signal exposes `toolsBySender` only under group config.

## What This Changes

Request enforced `toolsBySender` restrictions for allowlisted direct-message senders on DM-capable channels such as Signal and Telegram.

## Best Possible Solution

Add a generic direct-message sender tool-policy resolver beside the group resolver, backed by verified inbound sender identity and shared config/schema/docs for DM-level `tools` and `toolsBySender`. Cover Signal and Telegram with regressions proving one allowlisted DM sender can keep tools while another is denied.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to evaluate a direct Telegram or Signal session with sender metadata: the session parser does not create group context, and `resolveGroupToolPolicy` returns no policy, so DM-level `toolsBySender` cannot affect tool filtering.

## Solution Assessment

Yes. The issue points at the right product boundary: enforce a shared direct-message sender policy in the channel/tool-policy contract rather than relying on workspace prose or channel-local shortcuts.
