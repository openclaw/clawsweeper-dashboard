## Summary

Keep open. Current main still lacks OpenRouter audio transcription, and this PR is useful directionally, but the proposed patch is draft/stale against current main and has build, behavior, metadata, and HTTP-policy blockers before it can merge.

## What This Changes

The PR adds an OpenRouter `tools.media.audio` transcription provider using `chat/completions` `input_audio`, registers it, adds focused tests, and documents the configuration.

## Best Possible Solution

Land a refreshed implementation that extends the existing OpenRouter media-understanding provider with audio support, preserves image behavior, uses provider HTTP policy helpers, updates manifest metadata/docs/changelog, and keeps auto-detection unchanged unless maintainers explicitly choose otherwise.

## Reproduction Assessment

Yes. Source inspection on current main is enough: configure `tools.media.audio.models` with `provider: "openrouter"`, and the runner has no eligible OpenRouter audio provider because the registered provider has no `transcribeAudio` hook.

## Solution Assessment

No. The feature direction is appropriate, but this patch is not the best current-main solution because it imports helpers from the wrong SDK subpath, bypasses provider HTTP policy handling, and would overwrite the existing image media provider instead of extending it.
