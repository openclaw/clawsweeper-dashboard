## Summary

Keep open. The OpenRouter audio-transcription gap is real on current main, but this draft PR is stale against the current OpenRouter media provider and would fail or regress current behavior unless repaired.

## What This Changes

The PR adds an explicit OpenRouter `tools.media.audio` transcription provider using `chat/completions` `input_audio`, plus plugin registration, focused tests, and audio documentation.

## Best Possible Solution

Land a refreshed implementation that extends the existing OpenRouter media-understanding provider with audio support, preserves image behavior, uses the current provider HTTP policy helpers, updates manifest metadata/docs/changelog, and keeps auto-detection unchanged unless maintainers choose an explicit OpenRouter audio priority.

## Reproduction Assessment

Yes. Current-main source inspection is enough: configure `tools.media.audio.models` with `provider: "openrouter"`, and the runner has no eligible OpenRouter audio provider because the registered provider lacks `transcribeAudio`.

## Solution Assessment

No. The feature direction is appropriate, but the proposed patch is not the best current-main solution because it imports helpers from the wrong SDK subpath, replaces rather than extends the existing provider, and bypasses HTTP policy handling.
