## Summary

Keep open. Current main still has the documented default active-run steering path and `steeringMode: "all"` behavior called out in this issue, while the prefill/replay fixes only partially cover the reported TUI amnesia loop and recovery-regeneration symptoms.

## What This Changes

The issue reports a v2026.5.7 TUI regression where streaming follow-ups are steered by default, gateway-injected assistant entries can trigger Anthropic prefill rejection, and recovery failures lead to repeated stale responses.

## Best Possible Solution

Keep this open and have runtime owners define the intended TUI active-run semantics, then split fixes and regression coverage across queue steering, provider replay, and recovery-loop handling while coordinating with https://github.com/openclaw/openclaw/issues/48003, https://github.com/openclaw/openclaw/pull/73365, https://github.com/openclaw/openclaw/issues/79614, and https://github.com/openclaw/openclaw/issues/62322.

## Reproduction Assessment

No high-confidence end-to-end TUI reproduction was established. Source inspection does reproduce the default active-run steering path, but the repeated-response loop and Anthropic 400 need a live or fixture-backed transcript replay against current main.

## Solution Assessment

Unclear. Blindly changing `steeringMode: "all"` or the default queue mode is not obviously the best fix because current docs and tests intentionally make `steer` the default; the safer path is a maintainer decision plus targeted regression tests.
