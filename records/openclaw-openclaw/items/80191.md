## Summary

Keep open. Current main still intentionally defaults normal inbound messages to active-run steering, and the report combines that queueing behavior with partly-fixed prefill hygiene and separately tracked recovery instability, so this is not safe conservative cleanup.

## What This Changes

The issue reports a v2026.5.7 TUI regression where in-stream follow-up messages are implicitly steered, assistant-role transcript injection can trigger Anthropic prefill rejection, and failed recovery can repeat prior answers.

## Best Possible Solution

Keep the issue open, split the remaining work into a TUI/current-main repro plus an explicit decision on whether the TUI should keep default steering or prefer followup/collect semantics, and route the recovery-prefill piece through the existing recovery issue.

## Reproduction Assessment

Unclear. Source inspection confirms the default steering path and the recovery-prefill ingredient, but this review did not replay the reporter's TUI JSONL transcripts or produce a deterministic current-main amnesia loop.

## Solution Assessment

Unclear. Forcing every streaming TUI message into a new turn is not obviously the best fix because current docs define `steer` as the default; the safer direction is to isolate any broken steering/prefill behavior and then choose TUI queue semantics explicitly.
