## Summary

Keep this open. Current main still has the active-run steering behavior the report calls out, while the prefill/replay side is only partially mitigated and the recovery-loop symptom needs fixture or live TUI proof before this can be closed as fixed or duplicate.

## What This Changes

The issue reports a v2026.5.7 TUI regression where streaming follow-ups are implicitly steered into the active run, assistant-prefill transcript entries can trigger provider 400s, and recovery failures can lead to repeated stale answers.

## Best Possible Solution

Keep this open while runtime owners decide the intended TUI active-run semantics, then split any fix into scoped queue/steering, replay/prefill, and recovery regression coverage coordinated with https://github.com/openclaw/openclaw/issues/48003, https://github.com/openclaw/openclaw/issues/79614, and https://github.com/openclaw/openclaw/pull/73365.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established. Source inspection does reproduce the default active-run steering path on current main, but the TUI amnesia loop and provider 400 need a live or fixture-backed transcript replay.

## Solution Assessment

Unclear. Changing `steeringMode: "all"` or the default queue mode alone is not clearly the best fix because current docs and tests intentionally make `steer` the default; the safer path is an owner decision on TUI queue semantics plus targeted replay/recovery tests.
