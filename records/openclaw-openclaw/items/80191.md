## Summary

Keep this open. Current main still has the default active-run steering path the report calls out, while the prefill side is only partially covered by existing replay/payload mitigations, so there is not enough evidence to close this as implemented, duplicate, or stale.

## What This Changes

The issue reports a v2026.5.7 TUI regression where streaming follow-ups are steered into the active run, assistant-prefill-shaped transcript entries can trigger provider 400s, and recovery failures can lead to repeated responses.

## Best Possible Solution

Keep this open and have the runtime owners decide the intended TUI active-run semantics, then split the fix into scoped queue/steering, provider replay, and recovery regression coverage while coordinating with https://github.com/openclaw/openclaw/issues/79614, https://github.com/openclaw/openclaw/issues/48003, and https://github.com/openclaw/openclaw/pull/73365.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this review. Source inspection does reproduce the current default steering path, but the TUI amnesia loop and provider 400 need a live or fixture-backed transcript replay against current main.

## Solution Assessment

Unclear. Blindly changing `steeringMode: "all"` or the default queue mode is not obviously the best fix because current docs and tests intentionally make `steer` the default; the safer path is an owner decision on TUI queue semantics plus targeted replay/recovery tests.
