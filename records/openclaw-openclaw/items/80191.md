## Summary

Keep this open. Current main still contains the default streaming-follow-up steering path reported here, while the prefill/replay and recovery symptoms are only partially mitigated and need a live or fixture-backed regression pass before this can be closed.

## What This Changes

The issue reports a v2026.5.7 TUI regression where streaming follow-up messages are steered into the active run, assistant-prefill payloads trigger provider rejection, and recovery failures can lead to repeated stale responses.

## Best Possible Solution

Define the intended TUI active-run semantics first, then split fixes into targeted queue/steering, provider replay, and recovery-loop regression coverage coordinated with the open steering and stale-reply follow-ups.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established. Source inspection confirms the current default steering path and adjacent recovery/prefill ingredients, but the reported TUI amnesia loop and provider 400 need a live or fixture-backed replay against current main.

## Solution Assessment

Unclear. Blindly changing `steeringMode: "all"` or the default queue mode is not obviously the best fix because current docs and tests intentionally make `steer` the default; the safer path is an owner decision plus targeted regression tests.
