## Label Justifications

- `P1`: The issue describes a real gateway/ACP workflow where users lose the terminal error event and see a silently stopped response.
- `impact:message-loss`: The affected behavior is a client-facing `chat` error event that is generated as an agent payload but not delivered to connected clients.

## Summary

Keep open: current main still has the source path described in the report, and the latest release tag shows the same behavior. The timeout is converted into an `isError` reply payload, but once `agentRunStarted` is true the gateway post-dispatch branch only updates transcript state and does not emit a `chat` error event.

## What This Changes

The issue requests that LLM idle timeout errors returned after an agent run has started be broadcast to connected gateway clients instead of only being written to the session log.

## Best Possible Solution

Teach the gateway post-dispatch path to turn terminal returned `isError` replies after `agentRunStarted` into a `chat` error event and failure dedupe state, without mirroring normal agent replies into the transcript.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: make `dispatchInboundMessage` call `onAgentRunStart` and then return a final payload with `isError: true`; the current `.then()` branch reaches only `emitUserTranscriptUpdate()` and never calls `broadcastChatError()`. I did not execute the test because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix, but the implementation should also keep dedupe/error state consistent and avoid duplicate transcript writes. The existing `broadcastChatError()` helper and gateway test harness are the right surfaces.
