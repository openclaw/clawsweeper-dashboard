## Label Justifications

- `P1`: The issue affects a real gateway/ACP workflow where connected clients lose the terminal error event and the response appears to silently stop.
- `impact:message-loss`: The generated error payload is recorded internally but the client-facing `chat` error event is not delivered.

## Summary

Keep open: current main and the latest release still have the reported gateway path, and two open closing PRs now target the same bug. The issue should remain open until one fix is reviewed and merged.

## What This Changes

The issue reports that returned LLM idle-timeout `isError` payloads after an agent run starts are logged but not broadcast as terminal `chat` error events.

## Best Possible Solution

Review and land one focused gateway fix that broadcasts returned terminal `isError` payloads as chat errors, records error dedupe state, and adds regression coverage for the agent-started path.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to make `dispatchInboundMessage` call `onAgentRunStart`, deliver a final payload with `isError: true`, and observe the current post-dispatch branch reach only `emitUserTranscriptUpdate()` plus an `ok` dedupe entry.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway post-dispatch branch, reusing the existing `broadcastChatError()` contract while keeping transcript ownership and dedupe state consistent.
