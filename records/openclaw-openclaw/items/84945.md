## Label Justifications

- `P1`: The issue describes a real gateway/ACP workflow where clients lose the terminal error event and users see a silently stopped response.
- `impact:message-loss`: The affected behavior is a generated client-facing `chat` error event that is not delivered to connected clients.

## Summary

Keep open: current main and the latest release still have the gateway branch that drops returned `isError` payloads after `agentRunStarted`, and https://github.com/openclaw/openclaw/pull/84953 plus https://github.com/openclaw/openclaw/pull/85176 are still open implementation candidates.

## What This Changes

The issue asks for gateway chat to broadcast a terminal `state: "error"` event when an agent-started run resolves with an `isError` reply payload such as an LLM idle timeout.

## Best Possible Solution

Review and land a single gateway fix that broadcasts returned `isError` payloads after agent start and records error dedupe state, then close the duplicate or superseded fix PR.

## Reproduction Assessment

Yes. A high-confidence source-level repro is to make `dispatchInboundMessage` call `onAgentRunStart` and then resolve with a final `ReplyPayload` containing `isError: true`; the current branch reaches only transcript update plus OK dedupe instead of `broadcastChatError()`.

## Solution Assessment

Yes. The narrow maintainable solution is in `src/gateway/server-methods/chat.ts`: detect returned error payloads after agent start, emit the existing `chat` error event, and set failure dedupe state without mirroring normal Pi assistant turns.
