## Label Justifications

- `P1`: The report describes an active gateway chat workflow where users can receive no terminal error feedback after a real model idle timeout.
- `impact:message-loss`: The affected behavior suppresses the terminal error chat event that connected clients need to display the failure.
- `impact:session-state`: Without a final or error chat event, clients can remain out of sync with the completed failed run.

## Summary

Keep open: current main still has the reported gateway path where an agent-started `chat.send` run drains returned `isError` payloads into `deliveredReplies` but only emits a transcript update, so connected clients can miss the terminal error event.

## What This Changes

The issue requests that LLM idle timeout error payloads returned after an agent run starts be broadcast as gateway chat errors instead of being silently dropped.

## Best Possible Solution

Add a focused gateway fix that treats agent-started, final/block `isError` replies with no successful assistant content as a terminal `chat` error event, with regression coverage in the existing `chat.send` seam tests.

## Reproduction Assessment

Yes for source-level reproduction: current `run.ts` returns idle-timeout `{ isError: true }` payloads, `dispatch-from-config` routes returned replies through the dispatcher, and `chat.ts` does not broadcast an error once `agentRunStarted` is true. I did not execute a live 120s model timeout in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the gateway `chat.send` post-dispatch branch, not in provider timeout handling. The implementation should classify delivered error-only replies and call the existing `broadcastChatError` path while preserving normal agent-run transcript ownership.
