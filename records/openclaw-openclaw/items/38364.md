## Summary

Keep open: current main still lacks a structured Control UI reply target in `chat.send`, and the Control UI terminal status remains a coarse `done`/`interrupted` signal. Open related work covers useful pieces, but no merged implementation or single canonical item fully supersedes this combined protocol and UX request.

## What This Changes

The issue requests structured reply-to metadata from Control UI chat sends into agent sessions, plus clearer handling for completed runs that produce no user-facing assistant output.

## Best Possible Solution

Define an additive typed reply-target contract for Control UI `chat.send`, propagate it into canonical inbound runtime context, and coordinate output-aware terminal status/fallback semantics with https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes. Source inspection on current main shows `chat.send` rejects/omits reply-target fields, `CurrentInboundPromptContext` has no structured reply object, and Control UI terminal status only distinguishes `done` from `interrupted`.

## Solution Assessment

Unclear until maintainers choose the protocol and UX contract. A quote-only UI workaround is too narrow for structured reply metadata, while a wording-only status tweak is too narrow for visible-output terminal semantics.
