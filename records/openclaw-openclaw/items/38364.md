## Summary

Keep open: current main and latest release still lack a structured outgoing reply target in `chat.send`, and Control UI terminal status is still only a coarse done/interrupted indicator without visible-output semantics. The narrower right-click Reply PR and related delivery-contract issue overlap, but they do not fully supersede this combined session-protocol plus output-state request.

## What This Changes

The issue requests structured reply-to metadata in Control UI agent sessions and less misleading user-facing completion notices when a run ends without meaningful assistant output.

## Best Possible Solution

Define an additive typed reply-target contract for Control UI `chat.send`, propagate it into canonical inbound session context, and extend terminal run state with visible-output classification so the UI can distinguish normal replies, no-output completion, and interruption without relying on quoted-text workarounds.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: current `chat.send` rejects unknown fields and exposes no reply target, while Control UI terminal status only records `done` or `interrupted`; I did not run a live browser scenario in this read-only review.

## Solution Assessment

Yes in direction, but the maintainable solution needs an owner-approved typed end-to-end reply contract and normalized output-aware terminal state. A quoted-text-only workaround or a one-off status wording change is too narrow for the requested behavior.
