## Summary

Keep open: current main and the latest release still lack a structured Control UI reply target in `chat.send`, and the WebChat terminal status remains a coarse `done`/`interrupted` signal. The linked right-click Reply and runtime-context PRs cover useful pieces, and the durable final-fallback issue covers the no-output half, but no single merged item supersedes this combined protocol and UX request.

## What This Changes

Request to expose structured reply-to metadata from Control UI chat sessions to agents and make no-output completion states less misleading.

## Best Possible Solution

Define an additive typed reply-target contract for Control UI `chat.send`, propagate it into canonical inbound runtime context, and coordinate output-aware terminal status/fallback behavior with https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes. Source inspection on current main shows `chat.send` rejects/omits reply-target fields, `CurrentInboundPromptContext` has no structured reply object, and Control UI terminal status only distinguishes `done` from `interrupted`.

## Solution Assessment

Unclear until maintainers choose the protocol and UX contract. A quote-only UI workaround is too narrow for structured reply metadata, while a wording-only status tweak is too narrow for visible-output terminal semantics.
