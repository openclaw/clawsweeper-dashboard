## Summary

Keep open. Current main still has no structured reply-target path from Control UI or Gateway chat sends into agent-visible context, while the no-output display side is partly covered by existing `NO_REPLY`/control-token suppression.

## What This Changes

The issue asks Control UI and Gateway chat/session sends to expose structured reply-to metadata to agent sessions and avoid misleading no-output completion notices.

## Best Possible Solution

Add a backward-compatible reply-target field to `chat.send` and `sessions.send`, pass sanitized parent id/body/sender metadata from Control UI into existing `MsgContext` reply fields, and preserve the current silent-output filtering behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: closed `chat.send` and `sessions.send` schemas reject reply metadata, Control UI does not send a reply target, and `chat.send` builds `MsgContext` without the existing reply fields.

## Solution Assessment

Yes. The narrow maintainable direction is additive protocol/UI plumbing into the existing `MsgContext` reply fields; encoding reply intent in message text would not provide reliable structured parent-message metadata.
