## Summary

Keep open. Current main still lacks structured reply-target plumbing from Control UI or Gateway sends into agent-visible context; the silent-token display side is partly covered, but the central reply metadata request remains unresolved.

## What This Changes

Expose structured reply-to metadata from Control UI/WebChat sends into agent sessions and make no-output completion notices less misleading.

## Best Possible Solution

Add a backward-compatible reply-target contract to chat.send and sessions.send, pass sanitized Control UI reply metadata into existing MsgContext reply fields, and preserve the current silent-output filtering behavior.

## Reproduction Assessment

Yes for the source-level gap: current main has closed chat.send and sessions.send schemas without reply metadata, Control UI does not send a reply target, and chat.send builds MsgContext without ReplyTo fields. I did not run a live browser repro because this is primarily a protocol/UI capability gap.

## Solution Assessment

Yes at the direction level: additive protocol/UI plumbing into existing MsgContext reply fields is the narrow maintainable path. Maintainers should settle the exact payload shape and coordinate with the related Dashboard reply UI issue before implementation.
