## Summary

Keep open. Current main still lacks structured reply-target plumbing from Control UI or Gateway sends into agent-visible context; the silent-token display path is partly covered, but that does not resolve the central reply metadata request.

## What This Changes

Expose Control UI reply-to metadata to agent sessions and avoid misleading no-output completion notices.

## Best Possible Solution

Add a backward-compatible reply-target contract to chat.send and sessions.send, pass sanitized Control UI reply metadata into existing MsgContext ReplyTo fields, and keep the current silent-output filtering behavior.

## Reproduction Assessment

Yes for the source-level capability gap: current main has closed chat.send and sessions.send schemas without reply metadata, Control UI does not send a reply target, and chat.send does not populate MsgContext ReplyTo fields. I did not run a live browser repro because this is primarily a protocol/UI capability gap.

## Solution Assessment

Yes at the direction level: additive protocol/UI plumbing into existing MsgContext reply fields is the narrow maintainable path. Maintainers should settle the exact payload shape and coordinate it with the open Dashboard reply UI tracker before implementation.
