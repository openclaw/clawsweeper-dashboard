## Summary

Keep open. Current main still lacks the structured reply-target path from Control UI/Gateway sends into agent-visible session context, while the confusing no-output/silent-reply display path is already partly covered by current UI and history filtering.

## What This Changes

Request to carry reply-to metadata from Control UI/chat sends into agent sessions and avoid misleading completion notices when no useful assistant text is produced.

## Best Possible Solution

Add an additive Gateway protocol reply-target field for chat.send and sessions.send, map sanitized parent id/body/sender metadata into existing MsgContext ReplyTo* fields, update Control UI/docs/tests, and keep the current silent-output filtering behavior intact.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: chat.send and sessions.send reject unknown reply metadata, Control UI sends no reply target, and chat.send builds MsgContext without ReplyTo* fields.

## Solution Assessment

Yes. The narrow maintainable solution is additive protocol/UI plumbing into the existing MsgContext reply fields; encoding reply intent in message text would not provide reliable structured parent-message metadata.
