## Summary

Keep open: current main and v2026.6.6 still lack a structured Control UI `chat.send` reply target, and terminal run UI still collapses success into generic `done` without visible-output semantics; the related open reply-metadata PR only covers runtime handoff for another issue and does not resolve the Control UI/no-output request.

## What This Changes

Expose structured reply-to metadata from Control UI chat sends to agent sessions and make no-output or interrupted completions less misleading to users.

## Best Possible Solution

Define one additive reply-target contract for Control UI `chat.send`, propagate it through Gateway/session context without treating quoted bodies as trusted metadata, and extend terminal run state so Control UI can distinguish visible reply, no visible reply, and interrupted or aborted outcomes.

## Reproduction Assessment

Yes. Source inspection shows the current `chat.send` schema and Control UI request payload have no reply target, while terminal run UI records only `done` or `interrupted` without visible-output classification.

## Solution Assessment

Unclear at implementation detail level. The requested direction is right, but the best fix needs maintainer-approved protocol and terminal-state semantics rather than a one-off quoted-text workaround or localized wording tweak.
