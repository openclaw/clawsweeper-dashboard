## Summary

Keep open. Current main still lacks a structured reply-to path through `chat.send` or `sessions.send` into the agent-facing `MsgContext`; the existing no-output handling covers exact `NO_REPLY`/empty-final suppression, but that does not satisfy the reply-metadata half of the report.

## What This Changes

Expose structured reply-to metadata from Control UI/chat session sends to agent sessions and avoid misleading completion notices when no user-facing assistant text exists.

## Best Possible Solution

Implement an additive Gateway and Control UI reply metadata flow that captures a selected parent message, validates it on `chat.send`/`sessions.send`, maps it into the existing `MsgContext.ReplyTo*` fields, documents the contract, and keeps the current silent-output filtering behavior intact.
