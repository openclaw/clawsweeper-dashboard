## Summary

Keep open. Current main partially addresses the no-output/no-reply display problem by filtering silent assistant finals, but the requested structured reply-to metadata is still absent from the Control UI chat send path and Gateway chat protocol, and the related open issues cover adjacent UI/threading work rather than fully superseding this metadata request.

## What This Changes

The issue asks for Control UI chat replies to expose structured reply-to metadata to agent sessions and for no-output completion states to avoid misleading user-facing success notices.

## Best Possible Solution

Define an additive Gateway/WebChat reply metadata contract, wire Control UI selected reply targets into agent session context, and coordinate any UI affordance work with the open right-click reply tracker while keeping the existing silent-output filtering behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live browser run: current Control UI chat.send and the closed Gateway schema have no reply-to field, and the renderer only displays existing reply directives. The no-output half is partly source-proven as addressed by current NO_REPLY/HEARTBEAT_OK filtering tests.

## Solution Assessment

Unclear as a concrete implementation because the metadata shape, cross-surface semantics, and relationship to the open UI reply/threading work need maintainer scoping. The maintainable direction is an additive protocol/session contract plus focused UI wiring, not ad hoc text-only inference.
