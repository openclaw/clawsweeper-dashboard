## Summary

Keep open. Current main still lacks a structured Control UI reply-target path through `chat.send` into agent `MsgContext`; silent-token/no-output handling is partly covered, but the remaining reply metadata contract is not implemented or fully superseded by the related open UI/threading work.

## What This Changes

The issue requests structured reply-to metadata from Control UI chat replies into agent sessions and clearer handling for no-output completion notices.

## Best Possible Solution

Define an additive Control UI reply metadata contract from selected reply target through `ChatSendParamsSchema`, transcript/session semantics, and `MsgContext`, coordinated with the open UI quote-reply and channel-threading work while preserving current silent-output suppression.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive reply-metadata feature. Yes at source level: current main's `chat.send` schema, Control UI request payload, and `MsgContext` construction have no reply target field, so a selected Control UI reply target cannot reach the agent session.

## Solution Assessment

Unclear as an immediate patch because the change spans UI, Gateway protocol, transcript/session semantics, and agent context. The maintainable direction is an additive contract that does not regress existing `NO_REPLY`, empty-output, interruption, or channel-threading behavior.
