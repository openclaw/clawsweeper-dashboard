## Summary

Keep open: current main and v2026.6.6 still cannot carry an exact Control UI reply target into agent context, and the composer still reduces terminal success to a generic `Done` state without representing whether visible assistant output existed. The related reply-UX issue deliberately avoids protocol metadata, while the broader threading issue does not own these outcome semantics. Likely related people: @vincentkoc (high confidence, recent Control UI/gateway chat work), @steipete (medium confidence, recent protocol and run-lifecycle work), and BunsDev (medium confidence, status-indicator cleanup history).

## What This Changes

Expose structured Control UI reply targets to agent sessions and distinguish visible-reply success from empty, interrupted, aborted, or cancelled completion.

## Best Possible Solution

Add one typed reply-target field to the existing `chat.send` contract, propagate it into the canonical inbound message context, and extend normalized terminal outcome state with visible-final-output information so the UI can present precise completion feedback.

## Reproduction Assessment

Yes for the source-level gaps: current main has no `chat.send` reply target and its terminal UI cannot distinguish a completed run with visible assistant output from one without it. A live trace would still be valuable for choosing the exact fallback wording and validating tool-only cases.

## Solution Assessment

Yes in direction, but the best maintainable implementation is a typed end-to-end reply contract plus normalized output-aware terminal state, not a quoted-text-only workaround or suppression of one reported localized string.
