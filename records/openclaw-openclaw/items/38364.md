## Summary

Keep open. Current main still lacks a structured Control UI reply-to contract in `chat.send`, while the no-output confusion has been partly addressed by silent-token filtering and compact run statuses; the related right-click reply issue and open reply-threading PR do not supersede this broader metadata request.

## What This Changes

The issue requests structured reply-to metadata from Control UI chat sends into agent sessions plus less confusing handling for runs with no user-facing assistant output.

## Best Possible Solution

Keep this as the structured Control UI reply-metadata design tracker, preserve the existing silent-output suppression, and coordinate an additive protocol/UI/session contract with https://github.com/openclaw/openclaw/issues/16896 and https://github.com/openclaw/openclaw/pull/62262.

## Reproduction Assessment

Not applicable as a strict bug reproduction because the remaining work is primarily a new reply metadata/threading capability. Source inspection gives high confidence that the current Control UI `chat.send` path lacks structured reply-to metadata.

## Solution Assessment

Unclear as an immediate patch because the remaining work crosses UI affordance, Gateway protocol, and agent-session transcript semantics. The maintainable direction is an additive contract that does not regress current `NO_REPLY` and heartbeat suppression behavior.
