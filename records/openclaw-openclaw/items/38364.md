## Summary

Keep open: current main and latest release still lack a structured Control UI reply target in `chat.send`, and Control UI run status remains a coarse done/interrupted signal without visible-output semantics. The open right-click Reply PR and the durable final-fallback issue cover useful pieces, but neither fully supersedes this combined session-protocol and terminal-UX request.

## What This Changes

Request to expose structured reply-to metadata from Control UI/webchat into agent session context and make no-output run completion notices output-aware instead of generically successful.

## Best Possible Solution

Define an additive typed reply-target contract for Control UI `chat.send`, propagate it into canonical inbound session context, and extend terminal run state with visible-output classification while coordinating final-fallback delivery with the existing channel contract issue.

## Reproduction Assessment

Yes. Source inspection on current main shows `chat.send` rejects unknown fields and lacks a reply target, while Control UI terminal state only records `done` or `interrupted`; I did not run a live browser scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol and UX contract. A quote-only UI workaround is too narrow for structured reply metadata, and a one-off wording tweak is too narrow for output-aware terminal semantics.
