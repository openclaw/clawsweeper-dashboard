## Summary

Keep open. Current main still lacks a structured Control UI reply-to path from selected chat message through `chat.send` into agent session context; the no-output/`NO_REPLY` side is partly handled, but the remaining reply metadata scope is not implemented or superseded by the related UI reply and channel-threading items.

## What This Changes

Expose Control UI reply-to metadata to agent sessions and make no-output completion notices less misleading.

## Best Possible Solution

Define an additive Control UI reply metadata contract from selected UI reply target through `chat.send` into `MsgContext` and transcript semantics, while preserving current silent-output suppression and coordinating with the separate UI reply affordance and channel-threading work.

## Reproduction Assessment

No, not as a strict bug reproduction: the remaining work is an additive reply metadata capability. Source inspection gives high confidence that current main still lacks a structured Control UI `chat.send` reply metadata path.

## Solution Assessment

Unclear as an immediate patch because the change spans UI, Gateway protocol, and agent-session context. The maintainable direction is an additive contract that does not regress current `NO_REPLY` and heartbeat suppression behavior.
