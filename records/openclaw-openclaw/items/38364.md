## Summary

Keep open. Current main still lacks the central structured reply-to path into agent sessions: the Gateway `chat.send` and `sessions.send` schemas are closed without reply metadata, `sessions.send` forwards no parent/reply fields, the Control UI send helper cannot supply them, and the `chat.send` `MsgContext` construction does not populate existing `ReplyTo*` fields. The no-output side has adjacent `NO_REPLY`/empty-output suppression and classification coverage, so the remaining actionable gap is the reply metadata plumbing.

## What This Changes

Request to expose structured reply-to or parent-message metadata from Control UI/chat sends to agent sessions and avoid misleading no-output completion notices when no meaningful assistant reply exists.

## Best Possible Solution

Implement a focused additive protocol and UI change: add validated reply metadata to `chat.send` and likely `sessions.send`, let the Control UI composer select and clear a parent/reply target, map the sanitized bounded target id/body/sender into `MsgContext.ReplyTo*`, update protocol/docs/client artifacts/tests, and preserve the existing silent-output filtering behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a `chat.send` or `sessions.send` payload containing a reply target would be rejected by the closed schemas today, and the current Control UI state/request path has no field that could send such metadata.

## Solution Assessment

Yes, the best path is an additive Gateway protocol and Control UI mapping into the existing `MsgContext.ReplyTo*` fields. A workaround that encodes reply intent in message text or display-only `[[reply_to_*]]` directives would not reliably expose a structured parent message id to agent sessions.
