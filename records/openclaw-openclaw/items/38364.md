## Summary

Keep open. Current main still lacks a structured Control UI reply-target path from the composer through `chat.send` into agent `MsgContext`; the no-output/silent-token half is partly handled, but the remaining reply metadata contract is not implemented or superseded by the related open UI and channel-threading work.

## What This Changes

Request to propagate Control UI reply-to metadata into agent sessions and avoid misleading no-output completion notices.

## Best Possible Solution

Define an additive reply metadata contract from selected Control UI reply target through `ChatSendParamsSchema`, transcript/session semantics, and `MsgContext`, coordinated with https://github.com/openclaw/openclaw/issues/16896, https://github.com/openclaw/openclaw/issues/7817, and https://github.com/openclaw/openclaw/pull/62262 while preserving current silent-output suppression.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive metadata/UX feature. Source inspection gives high-confidence evidence that current main has no `chat.send` reply target field, no Control UI reply payload, and no reply fields populated into `MsgContext`.

## Solution Assessment

Unclear as an immediate patch because the change spans UI selection state, Gateway protocol, transcript/session semantics, and agent context. The maintainable direction is an additive contract coordinated with https://github.com/openclaw/openclaw/issues/16896, https://github.com/openclaw/openclaw/issues/7817, and https://github.com/openclaw/openclaw/pull/62262.
