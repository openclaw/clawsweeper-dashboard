## Summary

Keep open. Current main still lacks a structured Control UI reply-to contract from `chat.send` into agent session context; the no-output confusion is partly addressed, but related reply UI/threading work does not supersede the remaining metadata scope.

## What This Changes

The issue asks for structured reply-to metadata from Control UI chat sends to reach agent sessions and for no-output completion states to avoid misleading user-visible success notices.

## Best Possible Solution

Define an additive Control UI reply metadata contract from selected UI reply target through `chat.send` into `MsgContext`/transcript semantics, while preserving the existing silent-output suppression and coordinating with the separate UI reply affordance and channel threading work.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a strict bug reproduction, because the remaining work is a new reply metadata/threading capability; source inspection gives high confidence that current main lacks the structured Control UI `chat.send` reply metadata path.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an immediate patch because the change spans UI, Gateway protocol, and agent-session context. The maintainable direction is an additive contract that does not regress current `NO_REPLY` and heartbeat suppression behavior.
