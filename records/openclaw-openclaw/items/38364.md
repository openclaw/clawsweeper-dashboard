## Summary

Keep open. Current main still lacks a structured Control UI reply-target path through `chat.send` into agent `MsgContext`; no-output and silent-token handling is partly covered, but the remaining reply metadata contract is not implemented or superseded by the related open UI/threading work.

## What This Changes

Request to carry Control UI reply-to metadata into agent sessions and avoid misleading no-output completion notices.

## Best Possible Solution

Define an additive reply metadata contract from selected UI reply target through `ChatSendParamsSchema`, transcript/session semantics, and `MsgContext`, coordinated with the open UI quote-reply and channel-threading work while preserving current silent-output suppression.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive reply-metadata feature. Source inspection gives high-confidence evidence that current main has no `chat.send` reply target field, no Control UI reply payload, and no reply fields populated into `MsgContext`.

## Solution Assessment

Unclear as an immediate patch because the change spans UI selection state, Gateway protocol, transcript/session semantics, and agent context. The maintainable direction is an additive contract that keeps current `NO_REPLY`, empty-output, interruption, and channel-threading behavior intact.
