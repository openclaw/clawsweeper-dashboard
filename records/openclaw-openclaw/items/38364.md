## Summary

Keep open. Current main still lacks a structured Control UI reply-target path through `chat.send` into agent `MsgContext`; silent/no-output handling is partly covered, but the related open UI and channel-threading work does not supersede the remaining protocol/session design.

## What This Changes

Expose Control UI reply-to metadata to agent sessions and make no-output completion handling less misleading.

## Best Possible Solution

Define an additive Control UI reply metadata contract from selected reply target through `ChatSendParamsSchema`, `MsgContext`, and transcript semantics, coordinated with the open UI reply and channel-threading work while preserving current silent-output suppression.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive reply metadata capability. Source inspection on current main gives high confidence that the Control UI `chat.send` path still cannot carry a structured reply target into the agent session.

## Solution Assessment

Unclear as an immediate patch because the change spans UI, Gateway protocol, and session context semantics. The maintainable direction is an additive contract that does not regress existing `NO_REPLY`, empty-output, and interruption handling.
