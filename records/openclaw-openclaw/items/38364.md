## Summary

Keep open. Current main still lacks a structured Control UI reply-target path from outbound chat send through Gateway protocol into agent MsgContext; silent-token/no-output display handling is partly covered, but the remaining metadata contract is additive and needs maintainer API/UX scoping.

## What This Changes

Request to pass reply-to or parent-message metadata from Control UI chat sends into agent sessions and improve user-facing empty completion handling.

## Best Possible Solution

Define an additive selected-reply metadata contract from Control UI selection through ChatSendParamsSchema, transcript/session semantics, and MsgContext, coordinated with the open Control UI quote-reply and channel-threading work while preserving current silent-output suppression.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive metadata/UX feature. Source inspection gives high-confidence evidence that current main has no chat.send reply target field, no Control UI reply payload, and no reply fields populated into MsgContext.

## Solution Assessment

Unclear as an immediate patch because the change spans UI selection state, Gateway protocol, transcript/session semantics, and agent context. The maintainable direction is an additive contract coordinated with https://github.com/openclaw/openclaw/issues/16896, https://github.com/openclaw/openclaw/issues/7817, and https://github.com/openclaw/openclaw/pull/62262.
