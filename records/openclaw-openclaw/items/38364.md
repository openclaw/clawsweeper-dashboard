## Summary

Keep open. Current main still lacks a structured Control UI reply-target path through `chat.send` into agent `MsgContext`; silent-token/no-output handling is partly covered, but the related open UI reply and channel-threading items do not supersede the remaining protocol/session design work.

## What This Changes

The issue requests structured Control UI reply-to metadata in agent sessions and clearer handling when chat runs complete without meaningful assistant text.

## Best Possible Solution

Define an additive Control UI reply metadata contract from selected UI reply target through `chat.send` into `MsgContext` and transcript semantics, while preserving current silent-output suppression and coordinating with the separate UI reply affordance and channel-threading work.

## Reproduction Assessment

No as a strict bug reproduction; this is mainly an additive reply metadata capability. Source inspection gives high confidence that current main still lacks a structured Control UI `chat.send` reply metadata path.

## Solution Assessment

Unclear as an immediate patch because the change spans UI, Gateway protocol, and agent-session context. The maintainable direction is an additive contract that does not regress current `NO_REPLY`, empty-output, and interruption handling.
