## Summary

Keep this PR open as the canonical feature discussion, but do not merge the current branch: current main still lacks binding-level workspace overrides, while the PR head is stale, conflicting, missing real behavior proof, and has blocking reply-time workspace selection defects.

## What This Changes

The PR adds an optional `bindings[].workspace` route field, threads it into `ResolvedAgentRoute`, and makes reply setup prefer it over the agent workspace.

## Best Possible Solution

Thread a validated workspace from the already selected binding or route into reply setup as the single source of truth, document the public config contract, and cover precedence, wildcard, account, and collision cases before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head calls an unimported helper, reads an option outside `GetReplyOptions`, and scans bindings with incomplete scope. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but the current implementation should be replaced with selected-route workspace threading plus validation, docs, tests, conflict resolution, and real behavior proof.
