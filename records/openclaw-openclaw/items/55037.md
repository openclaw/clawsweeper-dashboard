## Summary

Keep open: current main still lacks the requested route-binding workspace feature, and the related feature issue was already closed in favor of this PR. The PR is not merge-ready because source inspection shows build/runtime defects, unsafe partial workspace selection, merge conflicts against current routing code, and missing real behavior proof.

## What This Changes

The PR adds `workspace?: string` to route bindings and schema, carries it on resolved routes, and tries to prefer it over the agent workspace in `getReplyFromConfig`.

## Best Possible Solution

Thread the selected binding workspace through the resolved route or inbound context, validate and document `bindings[].workspace`, preserve existing session and ACP binding behavior, and add focused precedence/collision tests before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head calls an unimported helper and reads a nonexistent optional field. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the already-selected route rather than a second partial binding scan, and it needs validation, docs, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:62`
  - body: `resolveBindingWorkspace` calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is imported or the selected route data is wired through instead.
  - confidence: 0.94
- **[P1] Read an existing session key source:** `src/auto-reply/reply/get-reply.ts:194`
  - body: `opts` is optional and `GetReplyOptions` has no `sessionKey` member, so this both violates the type contract and can throw when callers omit options. Use the already-derived session key or a typed route/context field.
  - confidence: 0.95
- **[P1] Use the selected binding for workspace selection:** `src/auto-reply/reply/get-reply.ts:62-66`
  - body: The helper re-scans bindings by agent id and peer-id suffix only, ignoring channel, account, guild, team, role, parent-peer, and wildcard scope. That can pick a workspace from an unrelated binding or skip valid non-peer bindings; the workspace should come from the route resolution result.
  - confidence: 0.93
- **[P2] Reject blank binding workspaces:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts whitespace-only values for a public cwd override. Trim and reject empty values, or normalize them before applying precedence so invalid config cannot select an unintended directory.
  - confidence: 0.82

## Security Review
