## Summary

Keep open: current main still lacks binding-level workspace overrides, and the closed related issue now points to this PR as the canonical implementation surface. The branch is not mergeable as-is because it re-derives routing scope for workspace selection, reads a non-existent reply option, omits public config metadata/docs/tests, and lacks real behavior proof.

## What This Changes

The PR adds `bindings[].workspace`, threads it onto `ResolvedAgentRoute`, and tries to prefer that workspace over the agent-level workspace in reply setup.

## Best Possible Solution

Implement binding-level workspace as a public config contract through the existing route resolver, then thread the selected binding workspace into reply/runtime setup with focused tests and docs.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows `GetReplyOptions` has no `sessionKey`, while the patch reads `opts.sessionKey`, and the new lookup re-matches route bindings by agent/peer suffix instead of the selected route scope.

## Solution Assessment

No. The feature direction is reasonable, but the patch should use the already resolved route/binding workspace instead of a second partial scan, and it needs public config metadata, docs, tests, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the resolved route for workspace selection:** `src/auto-reply/reply/get-reply.ts:65`
  - body: `resolveBindingWorkspace` only re-checks agent id plus a peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution instead of a second partial scan.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This line fails strict typing and can throw for direct callers before the normal workspace fallback runs.
  - confidence: 0.95
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the strict schema leaves schema labels/help and docs unaware of the new public config key, so generated config reference and UI/help surfaces will not describe it.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain change, but its duplicate workspace lookup can select the wrong cwd across scoped route collisions.

Concerns:

- **[medium] Workspace override can cross route scope:** `src/auto-reply/reply/get-reply.ts:65`
