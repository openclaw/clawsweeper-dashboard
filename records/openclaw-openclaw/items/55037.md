## Summary

Keep this PR open, but it is not merge-ready: current main still lacks binding-level workspace routing, while the PR head has blocking source-level defects, missing real behavior proof, and unresolved workspace/session security semantics.

## What This Changes

The PR adds a `bindings[].workspace` route config field, threads it through `ResolvedAgentRoute`, and tries to make reply setup prefer that binding workspace over the agent default.

## Best Possible Solution

Thread the selected binding's validated workspace through the resolved route or inbound context as the single source of truth, document the public config contract, add focused precedence/collision tests, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head calls an unimported helper, reads an option that is not in `GetReplyOptions`, and scans bindings with incomplete scope. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but the maintainable implementation should use the selected route/binding rather than a second partial scan, and it needs validation, docs, tests, rebase, and real behavior proof.
