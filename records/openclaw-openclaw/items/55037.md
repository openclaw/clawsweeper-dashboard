## Summary

Keep this PR open. Current main still lacks per-binding workspace support, but the proposed patch is not mergeable because reply setup re-derives the binding with a lossy peer-id/session-key scan, reads a non-existent `opts.sessionKey`, and does not update the public config metadata surface.

## What This Changes

The PR adds an optional route-binding `workspace` config field, carries it on `ResolvedAgentRoute`, and tries to prefer it over the agent-level workspace during reply execution.

## Best Possible Solution

Implement the feature through the existing route resolver: add the binding workspace to the route contract and generated config docs, then pass the already resolved workspace into reply/runtime setup without re-scanning bindings.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: apply the PR and TypeScript rejects `opts.sessionKey`, while two scoped Slack bindings with the same peer id or a team/account-only binding can select the wrong workspace or ignore the override.

## Solution Assessment

No. The requested behavior belongs in core, but the current patch duplicates partial routing logic; the safer fix is to consume the workspace from the already selected `ResolvedAgentRoute`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the resolved route for workspace selection:** `src/auto-reply/reply/get-reply.ts:65`
  - body: `resolveBindingWorkspace` matches only agent ID plus a peer-id suffix, so bindings that differ by account/team/guild/roles can collide and non-peer route bindings with a workspace never apply. Carry the workspace from the binding selected by `resolveAgentRoute` instead of doing a second lookup.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This line fails strict typechecking and direct callers that omit options would throw before normal workspace fallback runs.
  - confidence: 0.95
- **[P2] Publish the new binding key in config metadata:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the strict Zod schema leaves labels/help/generated config docs unaware of the new public key, so the config reference/UI surface will not describe it.
  - confidence: 0.84

## Security Review

Status: needs_attention
