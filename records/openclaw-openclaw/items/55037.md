## Summary

Keep this PR open because current main still lacks the requested binding-level workspace feature and the related feature issue was closed in favor of this PR, but the current branch is not mergeable and has source-level correctness and security blockers.

## What This Changes

The PR adds an optional `workspace` field to route bindings, carries it on resolved routes, and tries to prefer it over the agent workspace in reply handling.

## Best Possible Solution

Thread the selected binding workspace through the current route/inbound context path, validate and document `bindings[].workspace`, preserve existing session and ACP binding behavior, and add focused precedence and collision tests before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head calls an unimported helper, reads a nonexistent optional field, and re-scans route bindings with partial criteria. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the already selected route or inbound context instead of a second partial binding scan, and it needs validation, docs, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:62`
  - body: `resolveBindingWorkspace` calls `listRouteBindings(cfg)`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is imported or, preferably, the selected route data is wired through instead.
  - confidence: 0.94
- **[P1] Read an existing session key source:** `src/auto-reply/reply/get-reply.ts:194`
  - body: `opts` is optional and `GetReplyOptions` has no `sessionKey` member, so this violates the type contract and can throw when callers omit options. Use the already-derived session key or a typed route/context field.
  - confidence: 0.95
- **[P1] Use the selected binding for workspace selection:** `src/auto-reply/reply/get-reply.ts:62-66`
  - body: The helper scans all route bindings by agent id and peer-id suffix only, ignoring channel, account, guild, team, role, parent-peer, and wildcard scope. That can pick a workspace from an unrelated binding or skip valid non-peer bindings; the workspace should come from the route resolution result.
  - confidence: 0.93
- **[P2] Reject blank binding workspaces:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts empty or whitespace-only values for a public cwd override. Trim and reject empty values, or normalize them before applying precedence so invalid config cannot select an unintended directory.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch adds cwd selection logic that can choose the wrong local workspace.

Concerns:

- **[medium] Workspace override can cross route scope:** `src/auto-reply/reply/get-reply.ts:62`
  - body: The lookup chooses a workspace by agent id and peer-id suffix without checking the selected binding's full channel/account/guild/team/role scope. Since workspace controls local files and runtime cwd, scoped collisions can run the agent in the wrong directory.
  - confidence: 0.9
