## Summary

Keep open: current main still lacks binding-level workspace routing and the related feature issue was closed in favor of this PR, but the patch has blocking route-scope correctness/security issues and no real behavior proof.

## What This Changes

The PR adds a `bindings[].workspace` route-binding config field, carries it through route resolution, and tries to prefer it over the agent-level workspace during reply setup.

## Best Possible Solution

Implement binding-level workspace as a documented public config contract through the existing route resolver, then thread the selected binding workspace into reply/runtime setup with focused tests and docs.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows the patch reads `opts.sessionKey` despite no such `GetReplyOptions` field, and it re-matches route bindings by agent/peer suffix instead of using the selected route scope.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix should use the already resolved binding/route workspace and publish the config metadata instead of adding a second partial lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:65`
  - body: `resolveBindingWorkspace` only re-checks agent id plus a peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution instead of a second partial scan.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This access fails the type contract and can throw for direct callers before the normal workspace fallback runs.
  - confidence: 0.95
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the strict schema leaves generated labels/help and user docs unaware of the new public config key, so config reference and UI/help surfaces will not describe it.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention: there are no supply-chain changes, but the new duplicate workspace lookup can select the wrong cwd across scoped route collisions.

Concerns:

- **[medium] Workspace override can cross route scope:** `src/auto-reply/reply/get-reply.ts:65`
  - body: The lookup chooses a workspace by agent id and peer-id suffix without checking channel, account, guild, team, or role scope. Because workspace controls bootstrap files, media staging, hooks, local plugins, and tool cwd, a scoped collision can run the agent in the wrong directory.
  - confidence: 0.9
