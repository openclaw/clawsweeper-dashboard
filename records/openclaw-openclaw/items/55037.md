## Summary

Keep this PR open as the canonical implementation surface for a feature current main still lacks, but it is not merge-ready: the branch conflicts with current main, lacks real behavior proof, and re-derives workspace from partial route data instead of the selected binding.

## What This Changes

The PR adds a `workspace` field to route bindings, carries it on resolved routes, and tries to prefer that binding workspace over the agent default in reply setup.

## Best Possible Solution

Thread the selected binding's validated workspace through the route/inbound context as a single source of truth, document the public config contract, and test route-scope collisions plus upgrade behavior before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head calls an unimported helper, reads a nonexistent option, and scans bindings by only agent plus peer suffix. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the already selected binding/route instead of a second partial scan, and it needs validation, docs, tests, rebase, and real behavior proof.
