## Summary

Keep open: current main still lacks the requested feature, and the related feature issue was closed as superseded by this PR. The branch is not mergeable as-is because the workspace choice is re-derived from an incomplete route match, the reply options contract is wrong, config validation/docs are incomplete, and the external contributor has not provided real behavior proof.

## What This Changes

This PR adds a public `bindings[].workspace` route-binding field, carries it on resolved routes, and tries to prefer it over the agent workspace during reply setup.

## Best Possible Solution

Carry the selected binding workspace through the resolved route or message context, validate and document `bindings[].workspace`, preserve existing binding session overrides, and add focused precedence and scoped-collision tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: `GetReplyOptions` has no `sessionKey`, `listRouteBindings` is not imported, and the helper re-matches only agent plus peer suffix. The underlying request is a new config feature, not a current-main bug.

## Solution Assessment

No. The feature direction is plausible, but the maintainable fix should use the binding already selected by route resolution rather than doing a second session-key scan.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:59-65`
  - body: `resolveBindingWorkspace` re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution.
  - confidence: 0.94
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This breaks the type contract and can throw for callers that omit options before the normal workspace fallback runs.
  - confidence: 0.95
- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:59`
  - body: The new helper calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is removed or the selected route data is wired through correctly.
  - confidence: 0.9
- **[P2] Validate binding workspaces before accepting config:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts blank strings. Trim and reject empty values, or normalize blanks before applying precedence so an invalid binding value cannot bypass the agent workspace.
  - confidence: 0.86
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: This adds a public strict config key, but current help/docs metadata has no `bindings[].workspace` entry. Add the config metadata and generated/docs surface with the schema change.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch can choose the wrong cwd because workspace selection is re-derived from a partial route match.
