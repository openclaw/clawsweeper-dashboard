## Summary

Keep open: current main still does not implement per-binding workspaces, and this PR is the canonical implementation surface, but the branch is not mergeable as-is because the reply path has source-proven build/type defects, unsafe workspace selection, and missing real behavior proof.

## What This Changes

The PR adds a route-binding `workspace` config field, carries it on resolved routes, and tries to prefer it over the agent workspace during reply setup.

## Best Possible Solution

Thread the selected binding workspace through the route or inbound context, validate and document `bindings[].workspace`, preserve existing session and ACP binding behavior, and add focused precedence and collision tests before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch calls an unimported helper, reads a non-existent optional field, and re-matches bindings from partial route data. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the selected route instead of a second session-key scan and needs validation, config metadata/docs, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:62`
  - body: `resolveBindingWorkspace` calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is removed or the selected route data is wired through correctly.
  - confidence: 0.93
- **[P1] Pass a typed session key before reading it:** `src/auto-reply/reply/get-reply.ts:194`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This breaks the type contract and can throw for callers that omit options before the normal workspace fallback runs.
  - confidence: 0.95
- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:62-65`
  - body: The helper re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution.
  - confidence: 0.94
- **[P2] Reject blank binding workspaces:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts blank or whitespace-only values for a public cwd override. Trim and reject empty values, or normalize blanks before applying precedence so invalid config cannot select an unintended workspace.
  - confidence: 0.84
- **[P2] Add public config metadata for workspace:** `src/config/zod-schema.agents.ts:42`
  - body: This adds a strict public config key, but the config help/label metadata and agent binding docs do not include `bindings[].workspace`. Add the metadata/docs surface with the schema change so generated help and validation guidance stay complete.
  - confidence: 0.82
