## Summary

Keep open: current main still lacks binding-level workspace overrides and the closed companion issue points here as the canonical implementation, but this PR is not merge-ready because it conflicts with current routing code, lacks real behavior proof, and re-derives workspace selection outside the selected route.

## What This Changes

The PR adds a `bindings[].workspace` config field, threads it through route resolution, and tries to prefer that binding workspace when building replies.

## Best Possible Solution

Thread a validated workspace from the already selected route or binding into reply setup as the single source of truth, then document and test precedence, account/channel scope, wildcard/parent-peer behavior, and blank-value rejection before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head calls an unimported helper, reads an option outside `GetReplyOptions`, and scans route bindings with incomplete scope. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should be replaced with selected-route workspace threading plus validation, docs, tests, conflict resolution, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the selected binding for workspace selection:** `src/auto-reply/reply/get-reply.ts:62-66`
  - body: `resolveBindingWorkspace` scans all route bindings by agent id and peer-id suffix, ignoring channel, account, guild, team, roles, parent-peer, wildcard matches, and binding order. In multi-account or scoped routing setups this can pick a workspace from an unrelated binding; the workspace should come from the already selected route/binding.
  - confidence: 0.94
- **[P1] Read a typed session key source:** `src/auto-reply/reply/get-reply.ts:194`
  - body: `opts` is optional and `GetReplyOptions` has no `sessionKey` member, so `opts.sessionKey` both violates the reply options contract and can throw when callers omit options. Use the finalized context/session key or a typed route/context field instead.
  - confidence: 0.95
- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:62`
  - body: `resolveBindingWorkspace` calls `listRouteBindings(cfg)`, but this file does not import or define that symbol. The branch will fail type/build checks unless the helper is imported or, preferably, the selected route data is wired through instead.
  - confidence: 0.95
- **[P2] Reject blank binding workspaces:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts empty and whitespace-only values for a public cwd override. Trim and reject empty values, or normalize them before applying precedence so invalid config cannot select an unintended directory.
  - confidence: 0.82
