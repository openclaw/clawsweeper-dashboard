## Summary

Keep this PR open. Current main still does not support route-binding workspace overrides, and the PR remains useful, but the current patch is not mergeable: reply execution tries to rediscover the matched binding from a session-key suffix instead of consuming the resolved route, and it reads an option field that is not part of GetReplyOptions. Existing review comments already flagged the same route-scope problem, and the later peer-case fix only addressed one symptom.

## What This Changes

The PR adds an optional bindings[].workspace field to route binding types/schema, carries a workspace value on ResolvedAgentRoute, and tries to prefer that workspace when getReplyFromConfig chooses the runtime workspace.

## Best Possible Solution

Keep the feature in core, but implement it through the existing route resolver: add workspace to the route binding contract and generated config metadata, have resolveAgentRoute attach the matched binding workspace alongside session overrides, and pass that resolved route into reply/runtime setup so every binding tier uses the same scope decision.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: apply the PR and getReplyFromConfig either fails typechecking because GetReplyOptions lacks sessionKey or can dereference opts when omitted, and a two-account Slack setup with the same peer ID can select the wrong workspace because the lookup ignores account/channel/team/guild scope.

## Solution Assessment

No. The requested behavior is reasonable, but this patch is not the best way to solve it because it duplicates routing with a lossy session-key suffix scan; the safer fix is to consume ResolvedAgentRoute.workspace from the already-selected binding.
