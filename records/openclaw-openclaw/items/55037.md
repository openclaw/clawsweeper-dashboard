## Summary

Keep this PR open as the canonical implementation candidate, but it is not mergeable as-is: current main still lacks route-binding workspaces, and the branch has source-level correctness, security, config-contract, and real-proof blockers.

## What This Changes

The PR adds a public `bindings[].workspace` route-binding field, threads it through route resolution, and tries to prefer it over the agent-level workspace during reply setup.

## Best Possible Solution

Carry the selected binding workspace through the resolved route or message context, validate and document `bindings[].workspace`, preserve existing binding session overrides, and add focused tests for precedence and scoped-collision cases.

## Reproduction Assessment

Yes for the PR defects by source inspection: the patch reads `opts.sessionKey` even though `GetReplyOptions` has no such field, and it re-matches bindings by partial peer suffix instead of full selected route scope. The underlying request is a feature/config addition, not a current-main bug report.

## Solution Assessment

No. The feature direction is plausible, but the maintainable implementation should use the already selected route or message context and publish the full config contract instead of adding a second binding scan from a session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:62-65`
  - body: `resolveBindingWorkspace` re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This breaks the type contract and can throw for callers that omit options before the normal workspace fallback runs.
  - confidence: 0.95
- **[P1] Resolve the undefined route-binding helper:** `src/auto-reply/reply/get-reply.ts:59`
  - body: The new helper calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is removed or the selected route data is wired through correctly.
  - confidence: 0.9
- **[P2] Validate and normalize binding workspaces:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts blank strings; the reply path treats a blank binding value as present and bypasses the agent workspace before workspace setup falls back to the default. Trim and reject empty values, or normalize blanks before applying precedence.
  - confidence: 0.86
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding a public strict config key without schema help/generated baseline/docs leaves operator help and config reference surfaces unaware of `bindings[].workspace`. Add the metadata and docs with the schema change.
  - confidence: 0.84
