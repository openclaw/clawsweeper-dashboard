## Summary

Keep open: current main still lacks the requested per-binding workspace feature and the closed related issue intentionally points to this PR as the canonical review surface, but this branch is not merge-ready because it is conflicting, missing real behavior proof, and has blocking source-level defects in workspace selection.

## What This Changes

The PR adds a binding-level `workspace` config field, carries it on resolved routes, and tries to prefer it over the agent default workspace in reply setup.

## Best Possible Solution

Thread the already-selected binding's validated workspace through the inbound route/context into reply dispatch, document the new config contract, and test precedence, route-scope collisions, blank values, and upgrade behavior before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head calls an unimported helper, reads a nonexistent `GetReplyOptions` member, and re-scans route bindings with incomplete scope. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the already selected route/context instead of a second partial scan, and it needs validation, docs, tests, rebase, and real behavior proof.
