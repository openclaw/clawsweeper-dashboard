## Summary

Keep open: current main still lacks per-binding workspace support, and this PR is useful source work but not merge-ready because the reply path duplicates partial routing logic, reads a nonexistent option, misses public config metadata/docs/changelog, and can choose the wrong workspace.

## What This Changes

The PR adds `bindings[].workspace`, carries it on `ResolvedAgentRoute`, and tries to prefer that binding workspace over the agent-level workspace during replies.

## Best Possible Solution

Publish `bindings[].workspace` as a documented config contract and pass the matched binding's workspace through the existing route-resolution and dispatch path, preserving current route-session behavior and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: applying this PR leaves `opts.sessionKey` invalid against `GetReplyOptions`, and the helper can choose the wrong workspace by re-matching only agent id plus peer suffix instead of the route selected by `resolveAgentRoute`.

## Solution Assessment

No. The requested direction is reasonable, but the patch should use the already selected `ResolvedAgentRoute` workspace and preserve current `bindings[].session` handling instead of adding a second partial binding scan.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

  - body: `resolveBindingWorkspace` re-scans bindings by exact `agentId` and peer-id suffix, so scoped bindings can collide and non-peer bindings with `workspace` never apply. The diff also does not import `listRouteBindings`; carry the workspace from the binding selected by `resolveAgentRoute` instead.
  - confidence: 0.94

  - confidence: 0.95
- **[P2] Preserve binding session overrides:** `src/routing/resolve-route.ts:803`
  - body: Current main already passes `matched.binding.session` into `choose`; this PR uses that argument for `workspace`. A rebase must carry both values or `bindings[].session.dmScope` route overrides will stop applying.
  - confidence: 0.86
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the type/schema leaves config labels, help, generated docs baseline, user docs, and changelog unaware of the new public key.
  - confidence: 0.86
