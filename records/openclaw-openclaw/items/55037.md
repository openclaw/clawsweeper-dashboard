## Summary

Keep open: the feature is not implemented on current main and this PR still has merge-blocking defects in workspace selection, reply option typing, and public config publication.

## What This Changes

The PR adds a `bindings[].workspace` route-binding field, carries it on `ResolvedAgentRoute`, and tries to prefer it over the agent-level workspace during reply setup.

## Best Possible Solution

Implement binding workspace through the selected route-resolution contract, then publish and test the new config key across metadata, docs, and reply/runtime setup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defects: `opts.sessionKey` is not a valid optional reply field, and same-agent bindings with different channel/account/team/guild scope but the same peer suffix can select the wrong workspace.

## Solution Assessment

No. The requested feature is reasonable, but this patch duplicates partial routing logic; the narrower maintainable fix is to carry the selected binding workspace from `resolveAgentRoute` into reply/runtime setup.

## Review Findings
