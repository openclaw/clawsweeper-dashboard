## Summary

Keep open: current main does not implement per-binding route workspaces, and the PR still has blocking correctness issues in how reply execution chooses the workspace.

## What This Changes

The PR adds an optional route-binding `workspace` field, carries it on `ResolvedAgentRoute`, and tries to prefer it over the agent-level workspace during replies.

## Best Possible Solution

Implement the feature through the existing route resolver contract so the selected binding's workspace is passed into reply/runtime setup, then publish the new config key through metadata, docs, tests, and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defects: applying the patch makes `opts.sessionKey` invalid, and scoped bindings can select or skip the wrong workspace because the PR re-matches only agent id plus peer suffix.

## Solution Assessment

No. The requested direction is reasonable, but this patch duplicates partial routing logic; the narrower maintainable fix is to use the already selected `ResolvedAgentRoute` workspace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the resolved route for workspace selection:** `src/auto-reply/reply/get-reply.ts:65`
  - body: `resolveBindingWorkspace` matches only agent ID plus a peer-id suffix, so account/team/guild/role-scoped bindings can collide and bindings without `match.peer.id` never apply. The workspace should come from the binding already selected by `resolveAgentRoute`, not a second partial scan.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This line fails strict typechecking and direct callers that omit options can throw before the normal workspace fallback runs.
  - confidence: 0.95
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the strict schema leaves the public config metadata/help/docs unaware of the new key, so generated config reference and UI surfaces will not describe it.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/types.agents.ts:42`
  - body: This is a user-facing `feat` that changes the public config contract. Repo policy requires a `CHANGELOG.md` entry before merge.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the PR can choose the wrong workspace directory for scoped route collisions.

Concerns:

- **[medium] Workspace override can cross route scope:** `src/auto-reply/reply/get-reply.ts:65`
