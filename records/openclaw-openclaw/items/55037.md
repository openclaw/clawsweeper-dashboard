## Summary

Keep open: current main still lacks binding-level route workspaces, and this PR is the canonical open implementation, but the patch is not mergeable as written because it re-derives workspace from an incomplete session-key match, reads a non-existent option, misses an import, and lacks real behavior proof.

## What This Changes

The PR adds an optional `bindings[].workspace` route-binding field, carries it on `ResolvedAgentRoute`, and tries to prefer it over the agent workspace during reply setup.

## Best Possible Solution

Carry the selected binding workspace through the resolved route or inbound message context, validate and document `bindings[].workspace`, preserve existing session and ACP binding behavior, and add focused precedence and collision tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the diff calls an unimported helper, reads `opts.sessionKey` even though `GetReplyOptions` has no such field and `opts` is optional, and re-matches route bindings using only agent plus peer suffix. The underlying request is a new public config feature, not a current-main bug.

## Solution Assessment

No. The feature direction is plausible, but the maintainable fix should use the binding already selected by route resolution rather than a second session-key scan, and it needs validation, config metadata, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

  - body: `resolveBindingWorkspace` re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution.
  - confidence: 0.94

  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This breaks the type contract and can throw for callers that omit options before the normal workspace fallback runs.
  - confidence: 0.95
- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:59`
  - body: The new helper calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is removed or the selected route data is wired through correctly.
  - confidence: 0.9
- **[P2] Validate binding workspaces before accepting config:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts blank strings. Trim and reject empty values, or normalize blanks before applying precedence so an invalid binding value cannot bypass the agent workspace.
  - confidence: 0.86

  - body: This adds a public strict config key, but the config help and label metadata have no `bindings[].workspace` entry. Add the config metadata/docs surface with the schema change.
  - confidence: 0.84
