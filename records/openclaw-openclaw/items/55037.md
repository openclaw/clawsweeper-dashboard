## Summary

Keep open: current main still lacks binding-level workspace overrides, but this PR is not merge-ready because it re-matches bindings outside the selected route, reads a nonexistent reply option, omits public config metadata, and lacks real behavior proof.

## What This Changes

The PR adds `bindings[].workspace`, threads it through route resolution, and tries to prefer it over the agent-level workspace during reply setup.

## Best Possible Solution

Carry the selected binding workspace through the resolved route or message context, validate and document `bindings[].workspace`, preserve existing session override behavior, and add focused tests for precedence and scoped-collision cases.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the patch reads `opts.sessionKey` even though `GetReplyOptions` has no such field, and it re-matches route bindings by partial peer suffix instead of full selected route scope. The underlying request is a feature/config request, not a broken existing behavior report.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix should use the already selected route/binding workspace and publish the config contract instead of adding a second partial lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:65`
  - body: `resolveBindingWorkspace` re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace needs to come from the binding already selected by route resolution.
  - confidence: 0.93
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This access breaks the type contract and can throw for direct callers before the normal workspace fallback runs.
  - confidence: 0.95
- **[P2] Publish the binding workspace config key:** `src/config/zod-schema.agents.ts:42`
  - body: Adding `bindings[].workspace` only to the strict schema leaves generated help/docs metadata unaware of the new public config key, so the config reference and operator help surfaces will not describe it.
  - confidence: 0.84
