## Summary

Keep open: current main still lacks this feature, and this PR remains the canonical review surface, but the branch has blocking implementation, security, mergeability, and real-behavior-proof gaps.

## What This Changes

This PR adds optional route-binding `workspace` config, threads it into route resolution, and tries to prefer it over the agent workspace during reply setup.

## Best Possible Solution

Rebase onto current main, carry the selected binding workspace through the resolved route or inbound context, validate and document the config key, preserve session/ACP binding behavior, and add focused precedence and collision tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the diff calls an unimported helper, reads a non-existent optional field, and re-matches route bindings from partial data. The underlying request is a new config feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should use the selected route instead of a second session-key scan and needs validation, config metadata/docs, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the selected route for workspace selection:** `src/auto-reply/reply/get-reply.ts:59-65`
  - body: `resolveBindingWorkspace` re-checks only agent id plus peer-id suffix, so channel/account/guild/team/role-scoped bindings can collide and non-peer bindings with a workspace never apply. The workspace should come from the binding already selected by route resolution.
  - confidence: 0.94
- **[P1] Do not read sessionKey from reply options:** `src/auto-reply/reply/get-reply.ts:191`
  - body: `GetReplyOptions` does not define `sessionKey`, and `opts` is optional. This breaks the type contract and can throw for callers that omit options before the normal workspace fallback runs.
  - confidence: 0.95
- **[P1] Import or remove the route-binding helper:** `src/auto-reply/reply/get-reply.ts:59`
  - body: The new helper calls `listRouteBindings`, but this file does not import or define that symbol. The branch will fail type-check/build unless the helper is removed or the selected route data is wired through correctly.
  - confidence: 0.9
- **[P2] Validate binding workspace values:** `src/config/zod-schema.agents.ts:42`
  - body: `workspace: z.string().optional()` accepts blank strings. Trim and reject empty values, or normalize blanks before applying precedence so an invalid binding value cannot bypass the agent workspace.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch can choose the wrong local workspace because cwd selection is re-derived from a partial route match.

Concerns:

- **[medium] Workspace override can cross route scope:** `src/auto-reply/reply/get-reply.ts:62`
  - body: The lookup chooses a workspace by agent id and peer-id suffix without checking channel, account, guild, team, or role scope. Since workspace controls local files and runtime cwd, scoped collisions can run the agent in the wrong directory.
  - confidence: 0.9
