## Summary

## What This Changes

The issue asks for restart-interrupted cron runs to recover as failed instead of being swept to lost when the persisted reservation timestamp differs from the task run id start timestamp.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96174 if its implementation holds up; otherwise apply the same narrow fix in cron timer persistence with regression coverage for due jobs and startup catch-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main persists a reservation timestamp, builds the task run id from a later start timestamp, then recovery requires those values to be equal. I did not run a live crash/restart cycle during this read-only review.

## Solution Assessment

Yes, aligning the persisted running marker with the task run id timestamp before creating the task row is the narrow maintainable fix. The linked open PR appears to target that boundary, so this issue should stay open until that PR or an equivalent fix merges.
