The issue requests fixing cron restart recovery so interrupted cron task rows are marked failed instead of lost when reservation and task-start timestamps differ.

## Reproduction Assessment

Yes. Source inspection still shows current main persists a reservation timestamp, builds the cron task run id from a later started-at timestamp, and then requires those values to match during task recovery.

## Solution Assessment

Yes. Aligning the persisted running marker with the task run id timestamp before creating the task row is the narrow owner-boundary fix; the linked open PR targets that boundary.
