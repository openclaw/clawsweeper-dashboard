## Summary

Keep open: current main still persists a cron reservation timestamp, creates the task run with a later started-at timestamp, then requires those values to be equal during durable recovery, so the reported interrupted-run false `lost` path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to recover restart-interrupted cron task rows as failed instead of marking them lost when reservation and run-id timestamps differ.

## Best Possible Solution

Make scheduled and startup catch-up cron runs use one durable run timestamp across `runningAtMs`, task run id, run logs, and recovery, then add regression coverage with a deliberately advancing clock.

## Reproduction Assessment

Yes, at source level: seed a stale cron task whose run id encodes `startedAt` while the cron store has `lastRunAtMs = startedAt - 7ms`, then run task maintenance and observe recovery miss and lost reconciliation. I did not execute a live crash/restart cycle in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to align the durable cron recovery key with the task run id at the cron producer boundary, not to add another task-maintenance fallback that guesses across mismatched timestamps.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/96161. Current main persists a cron reservation timestamp in `job.state.runningAtMs`, then later creates the task run id with a newer `startedAt`; restart recovery copies the persisted reservation into `lastRunAtMs`, while task maintenance requires equality with the run-id timestamp and falls through to `lost`. Keep the fix in the cron/task recovery boundary: make scheduled and startup catch-up runs use one durable timestamp for the running marker, task run id, run-log/recovery key, and final `lastRunAtMs`, or otherwise persist the exact task-run started-at before execution so restart recovery can recover the same id. Cover both `onTimer` due jobs and startup catch-up. Preserve manual-run behavior, do not add config/env knobs, do not add runtime fallbacks for old malformed state, and do not edit `CHANGELOG.md`. Add focused regression coverage with an advancing clock proving an interrupted cron task whose reservation timestamp differs from the run-id timestamp is recovered as failed with `cron: job interrupted by gateway restart` instead of `lost / backing session missing`; keep existing run-log recovery and non-authoritative liveness tests green.
