## Summary

Keep this issue open: current main still has the cron recovery timestamp mismatch, and the open linked PR is the active candidate fix rather than a reason to close before merge.

## What This Changes

The issue reports that restart-interrupted cron task rows are marked `lost` instead of recovered as failed because cron recovery compares a persisted reservation timestamp with the task run id's later started-at timestamp.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96174 or an equivalent timer-persistence fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows current main persists a reservation timestamp, builds the task run id from a later start timestamp, then requires equality during recovery; I did not run a live crash/restart cycle.

## Solution Assessment

Yes. Aligning the durable running marker with the task run id timestamp before task-row creation is the narrow owner-boundary fix, and the linked open PR targets that path.
