## Summary

Keep this issue open: current main still has the cron recovery timestamp mismatch, and the linked open fixing PR is the active path rather than a reason to close before merge.

## What This Changes

The issue asks OpenClaw to recover restart-interrupted cron task rows as failed instead of lost when the persisted reservation timestamp differs from the task run-id start timestamp.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96174 if its implementation holds up; otherwise apply the same narrow cron timer persistence fix with regression coverage for due jobs and startup catch-up.

## Reproduction Assessment

Yes. Current main source still has the reported mismatch: persisted cron `runningAtMs` can be a reservation timestamp while the task run id is created from a later `startedAt`, and recovery requires equality.

## Solution Assessment

Yes. The best fix is to make the durable running marker and task run id use the same claimed start timestamp before task-row creation; the linked open PR targets that boundary.
