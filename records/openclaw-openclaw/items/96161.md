## Summary

Keep this issue open: current main still has the reported cron recovery timestamp mismatch, and the open linked fixing PR is the active path rather than a reason to close before merge.

## What This Changes

The issue reports that interrupted cron runs can be marked `lost` after restart because durable recovery compares the task run id `startedAt` against a persisted reservation timestamp.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96174, or an equivalent narrow fix that makes the persisted cron running marker match the task run id timestamp before closing this issue.

## Reproduction Assessment

Yes. Current main persists a reservation timestamp, builds the task run id from a later start timestamp, then recovery requires those two values to be equal.

## Solution Assessment

Yes. Aligning the durable running marker with the task run id timestamp before creating the task row is the narrow maintainable fix, and the linked open PR targets that boundary.
