## Summary

Keep open: current main has adjacent heartbeat scheduling, task records, cron setup watchdogs, session liveness diagnostics, and subagent run timeouts, but it still lacks the requested runtime-side task watchdog for accepted-but-not-started channel work and delegated progress heartbeats.

## What This Changes

Request a runtime task-watchdog configuration that emits visible status or blocker messages for accepted work that does not start and enforces heartbeat/progress deadlines for long-running or delegated agent work.

## Best Possible Solution

Design a generic task-lifecycle watchdog around accepted channel-agent work, task records, and subagent lifecycle events, then document its config, SLA defaults, and violation-delivery policy alongside existing heartbeat and diagnostics contracts.

## Reproduction Assessment

Not applicable: this is a new orchestration guardrail request, not a broken existing contract. Source and docs inspection verify adjacent heartbeat, task, diagnostics, and subagent timeout mechanisms but no current taskWatchdog/no-start/progress-SLA surface.

## Solution Assessment

Unclear until maintainers decide the product/API contract. A runtime-side watchdog is directionally appropriate, but SLA defaults, event ownership, subagent progress semantics, and system-event versus channel-announcement delivery need design before a focused implementation is safe.
