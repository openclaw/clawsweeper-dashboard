## Summary

Keep open: current main has heartbeat scheduling, task records, subagent run timeouts, chat abort deadlines, and lost-task reconciliation, but not the requested runtime watchdog for accepted work that has not started or delegated work that needs progress heartbeats.

## What This Changes

Request a runtime task watchdog that alerts on accepted channel-agent work that does not start and enforces progress heartbeats for long-running or delegated subagent work.

## Best Possible Solution

Design a generic task-lifecycle watchdog around channel-agent accepted work, task records, and subagent lifecycle events, then document the config and delivery policy alongside the run-level idle/max timeout contract.

## Reproduction Assessment

Not applicable: this asks for a new orchestration guardrail rather than a broken documented behavior. Source and docs inspection show adjacent heartbeat, timeout, and task-maintenance mechanisms but no current task-watchdog contract to reproduce against.

## Solution Assessment

Unclear until maintainers decide the product/API contract. A runtime-side watchdog is directionally appropriate, but SLA defaults, event ownership, subagent progress semantics, and channel/systemEvent delivery need design before a focused implementation is safe.
