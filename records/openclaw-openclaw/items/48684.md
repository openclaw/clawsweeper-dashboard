## Label Justifications

- `P1`: Bare abort messages can be delayed or swallowed in active chat workflows, leaving users with a broken remote stop path.
- `impact:message-loss`: The issue is about debounce merging or swallowing a user command message before the session can process it.
- `impact:session-state`: A swallowed abort leaves the active session in the wrong control state because the user's stop intent is never applied.

## Summary

Keep open. Current main still treats bare abort phrases as debounceable text, and open linked pull requests already claim the fix path, so this issue should stay open until one implementation lands or the paired PRs are closed.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound debounce immediately instead of being delayed or merged with later text.

## Best Possible Solution

Review and land the broader linked fix path that updates shared and plugin-local debounce predicates while preserving command-auth, mention-gating, and documented same-sender coalescing exceptions.

## Reproduction Assessment

Yes. Source inspection on current main shows `shouldDebounceTextInbound({ text: "stop", cfg })` still reaches `hasControlCommand("stop")`, while the separate abort-aware detector recognizes bare `stop`, `abort`, and `wait`.

## Solution Assessment

Yes. Reusing the existing abort-aware control detector at debounce gates is the narrow maintainable fix, provided auth and mention-gating call sites that need slash-command semantics stay unchanged.
