## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce predicate bypasses only slash/text commands through `hasControlCommand()`, while the existing abort-aware helper separately recognizes standalone abort phrases such as `stop`, `abort`, and `wait`. The linked fix PR #48916 is closed unmerged, so there is no merged implementation to close against.

## What This Changes

The issue requests that standalone abort phrases bypass inbound message debouncing immediately across debounced chat channels instead of being delayed or merged into normal text.

## Best Possible Solution

Use abort-aware command detection at every debounce boundary that decides whether text can be buffered, keep normal text/media debounce behavior unchanged, preserve the BlueBubbles same-sender DM coalescing exception, and add focused regression tests for bare abort phrases.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop")`, which does not consult abort triggers, so the predicate treats the abort phrase as debounceable text.

## Solution Assessment

Yes. The narrow maintainable fix is to use `isControlCommandMessage()` or the plugin-runtime equivalent in shared and plugin-local debounce gates, with regression coverage for `stop`, `abort`, and `wait`; changing queue timing or command semantics would be broader than necessary.

## Repair Work Prompt

Fix this issue by making standalone abort phrases bypass inbound message debounce across the shared channel helper and bundled plugin-local debounce gates. On current main, `shouldDebounceTextInbound()` uses `hasControlCommand()`, so `stop`, `abort`, and `wait` are debounced even though `isControlCommandMessage()` already recognizes abort triggers. Update `src/channels/inbound-debounce-policy.ts` to use abort-aware detection, add regression coverage in `src/channels/inbound-debounce-policy.test.ts`, and audit bundled plugins with local debounce predicates (`extensions/feishu`, `extensions/mattermost`, `extensions/msteams`, `extensions/whatsapp`, `extensions/bluebubbles`) so bare abort/control messages do not get buffered there either. Preserve existing behavior for blank text, normal text, media/attachments, Telegram forwarded burst coalescing, and the documented BlueBubbles `coalesceSameSenderDms` exception. Use plugin SDK/runtime command helpers rather than importing core internals into plugin production code. Add a one-line changelog fix entry credited to @sherly-techtree and any source PR author whose closed PR work is materially reused. Do not change debounce timing, queue policy, provider behavior, or unrelated command authorization semantics.
