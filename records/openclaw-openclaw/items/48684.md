## Summary

Keep open. Current main and the latest shipped tag still route shared inbound debounce through slash-command detection, while the abort-aware helper and abort trigger set already cover bare phrases; the linked repair PR is closed unmerged and no open PR currently owns this exact fix.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound text debounce immediately instead of being delayed or merged as normal text.

## Best Possible Solution

Use the existing abort-aware command detection helper at every debounce boundary that is meant to flush control messages, keep normal text/media debounce behavior unchanged, and add focused regression coverage for bare abort phrases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop")`, which only has `/stop` as the registered text command alias, so the predicate returns `true` even though `stop` is in `ABORT_TRIGGERS`.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `isControlCommandMessage()` or the plugin-runtime equivalent at debounce gates, not add new configuration or broaden unrelated authorization and mention-gating decisions.
