## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce predicate and several plugin-local debounce gates use slash-command detection, while the existing abort-aware helper already recognizes standalone abort phrases. The linked fix PR #48916 is closed unmerged, so there is no merged implementation to close against.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound message debouncing and flush immediately like `/stop`.

## Best Possible Solution

Use abort-aware command detection at every debounce boundary that is meant to flush control messages, keep normal text/media debounce behavior unchanged, preserve the documented BlueBubbles coalescing exception, and add focused regression tests for bare abort phrases.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop") === false` and returns `true`, while `isAbortTrigger("stop")` is separately registered and tested as an abort phrase.

## Solution Assessment

Yes. The narrow maintainable solution is to use `isControlCommandMessage()` or the existing plugin-runtime equivalent at debounce gates, with regression coverage proving bare abort phrases flush immediately and ordinary text still debounces.
