## Summary

Keep open. Current main still routes inbound debounce bypass through slash-command detection while the documented abort path already recognizes standalone abort phrases; the linked fix PR #48916 is closed unmerged, so there is no merged implementation to close against.

## What This Changes

Bare `stop`, `abort`, and `wait` messages should bypass inbound debounce immediately instead of being delayed or merged.

## Best Possible Solution

Use abort-aware control-message detection at shared and plugin-local debounce gates, preserve the BlueBubbles coalescing opt-in, and add focused regression coverage for bare abort phrases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand()`, while `isControlCommandMessage()` and `isAbortTrigger()` separately recognize the same phrase.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing abort-aware helper at debounce boundaries, with tests proving bare abort phrases bypass debounce and ordinary text still debounces.
