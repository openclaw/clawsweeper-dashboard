## Label Justifications

- `P1`: Bare stop messages can be delayed or swallowed in active chat workflows, leaving users with a broken remote abort path.
- `impact:message-loss`: The reported failure is a user command message being merged or swallowed by debounce before processing.
- `impact:session-state`: A swallowed abort leaves the active session running despite the user's stop intent.

## Summary

Keep open. Current main and latest release still route bare abort text through the slash-command-only debounce predicate, and an open linked PR explicitly closes this issue, so the cleanup path is to review or land that PR rather than close the issue early.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound text debounce immediately instead of being delayed or merged into normal chat text.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83348, or close it with an explicit replacement plan that fixes both the shared helper and plugin-local debounce predicates while preserving auth, mention-gating, and documented coalescing exceptions.

## Reproduction Assessment

Yes. Source inspection on current main shows bare `stop` reaches `hasControlCommand()` at the debounce gate, while `stop`, `abort`, and `wait` are only recognized by the separate abort-aware detector and trigger set.

## Solution Assessment

Yes. Reusing `isControlCommandMessage()` only at debounce bypass boundaries is the narrow maintainable fix, while slash-only auth and mention-gating call sites should stay unchanged.
