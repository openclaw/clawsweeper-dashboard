## Summary

Keep open. Current main still has the reported debounce gap: shared and several plugin-local debounce gates use slash-command detection, while the existing abort-aware helper and docs already recognize standalone abort phrases.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound message debouncing like `/stop` does.

## Best Possible Solution

Use abort-aware command detection at each debounce boundary that should flush control messages, preserve normal text/media debounce behavior and the documented BlueBubbles coalescing exception, and add focused regression coverage for bare abort phrases.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `shouldDebounceTextInbound({ text: "stop", cfg })` still reaches `hasControlCommand("stop") === false` and returns `true`, while abort tests register `stop`, `abort`, and `wait` as abort phrases.

## Solution Assessment

Yes. Reusing `isControlCommandMessage()` or the plugin-runtime equivalent is the narrow maintainable fix; changing only the shared helper would be incomplete because several bundled plugins keep local debounce predicates.
