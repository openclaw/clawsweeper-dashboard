## Label Justifications

- `P1`: Bare abort messages can be delayed or swallowed in active chat workflows, leaving users with a broken remote stop path.
- `impact:message-loss`: The issue is about debounce merging or swallowing a user command message before the session can process it.
- `impact:session-state`: A swallowed abort leaves the active session in the wrong control state because the user's stop intent is never applied.

## Summary

Keep open. Current main still shows the shared debounce gate treating bare abort phrases as debounceable text, and there are open linked fix PRs that explicitly close this issue, so the cleanup path is to review or land one of those PRs rather than close the issue early.

## What This Changes

Bare `stop`, `abort`, and `wait` abort phrases should bypass inbound debounce instead of being delayed or merged across debounced chat channels.

## Best Possible Solution

Review and land the broader linked fix path that updates shared and plugin-local debounce predicates while preserving auth, mention-gating, and documented coalescing exceptions.

## Reproduction Assessment

Yes. Source inspection on current main shows `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop")`, while bare stop/abort/wait are only recognized by the separate abort-aware path.

## Solution Assessment

Yes. Reusing the existing abort-aware command detector at debounce gates is the narrow maintainable fix, provided auth and mention-gating call sites that need slash-command semantics stay unchanged.
