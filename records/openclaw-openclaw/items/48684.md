## Summary

Keep open. Current main still has the reported debounce gap: shared inbound debounce and several plugin-local debounce gates bypass only slash/text commands through `hasControlCommand()`, while the existing abort-aware helper already recognizes standalone abort phrases.

## What This Changes

The issue asks for bare abort phrases such as `stop`, `abort`, and `wait` to bypass inbound message debounce immediately across debounced chat channels.

## Best Possible Solution

Use the existing abort-aware command detection helper at every debounce boundary that is meant to flush control messages, preserve documented channel-specific coalescing exceptions, and add focused regression coverage for bare abort phrases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop")`, which does not consult `ABORT_TRIGGERS`, so the predicate still returns debounceable text on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `isControlCommandMessage()` or the plugin-runtime equivalent at debounce gates, not add new configuration or change unrelated authorization and mention-gating decisions.
