## Summary

Keep open. Current `main` and the latest release still have the reported debounce gap: shared and plugin-local debounce gates use slash-command detection, while docs and abort helpers already recognize standalone abort phrases. The related fix PR at https://github.com/openclaw/openclaw/pull/48916 is closed unmerged, so the issue remains actionable.

## What This Changes

Bare abort phrases like `stop`, `abort`, and `wait` should bypass inbound debounce across debounced chat channels.

## Best Possible Solution

Use abort-aware command detection at every debounce boundary that should flush control messages, preserve documented coalescing exceptions, and add focused regression coverage for bare abort phrases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop")`, which does not consult the abort trigger set, so the predicate returns debounceable text.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `isControlCommandMessage()` or the plugin-runtime equivalent at debounce gates, while avoiding changes to unrelated authorization and mention-gating semantics.
