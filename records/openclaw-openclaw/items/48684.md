## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce predicate uses slash-command detection, while the abort-aware helper and documented standalone abort phrases remain separate; the related fix PR is closed unmerged.

## What This Changes

Standalone abort phrases such as `stop`, `abort`, and `wait` should bypass inbound message debouncing immediately across debounced chat channels.

## Best Possible Solution

Use abort-aware command detection at shared and plugin-local debounce gates, keep media and normal-text debounce behavior unchanged, preserve the BlueBubbles coalescing exception, and add regression tests proving `stop`, `abort`, and `wait` flush immediately where control commands are supposed to bypass debounce.

## Reproduction Assessment

Yes. A source-level reproduction is direct: on current main, `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop") === false` and returns `true`, so the abort phrase remains debounceable instead of flushing immediately.

## Solution Assessment

Yes. The narrow maintainable direction is to use `isControlCommandMessage()` or the existing plugin-runtime equivalent at debounce boundaries, with focused tests for bare abort phrases and unchanged normal text/media behavior.

## Repair Work Prompt

Fix the inbound debounce abort-trigger gap tracked by https://github.com/openclaw/openclaw/issues/48684.
