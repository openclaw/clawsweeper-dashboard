## Summary

Keep this open. Current main still resolves and consumes Discord component interactions by the clicked component id, while the Discord docs describe non-reusable components as single-use by default; the related closed #54233 mixed this same remaining bug with another Discord edit issue and does not supersede the panel-wide consumption problem.

## What This Changes

Discord non-reusable component panels should expire all sibling buttons and selects after any component in the panel is clicked, instead of consuming only the clicked component.

## Best Possible Solution

Implement Discord plugin-owned panel-wide consumption for non-reusable component messages, likely by assigning a shared group key when entries are built/registered for the same message and adding a registry operation that invalidates sibling entries after the first authorized click. Preserve `reusable: true`, unauthorized-click behavior, modal trigger/submission handling, plugin interactive handlers, and persistent registry behavior.

## Reproduction Assessment

Yes. A focused repro can register two non-reusable button entries with the same `messageId`, click one id, then click the other; current source deletes only the first id, so the second still resolves and dispatches.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Discord plugin component registry/handler path, not in core channel code or the generic plugin API.

## Repair Work Prompt

Fix the Discord component lifecycle bug where `components.reusable !== true` consumes only the clicked button/select instead of expiring sibling components in the same component message/panel. Keep the fix inside the Discord plugin-owned component builder/registry/handler path; do not add core channel behavior or change unrelated component APIs. Likely approach: add a shared consumption/group key to non-link component entries created from the same `buildDiscordComponentMessage` result and persisted through `registerDiscordComponentEntries`, then make the authorized non-reusable click consume/invalidate all entries in that group while preserving `reusable: true`, unauthorized-click behavior, modal trigger/submission behavior, plugin interactive handlers, and persistent registry fallback. Add focused regression coverage for two non-reusable buttons sharing one message where the first click dispatches and the second returns the expired response without dispatching. Also preserve existing tests for single-button consume and reusable buttons. Add a user-facing `CHANGELOG.md` fix entry credited to the reporter, `Thanks @fujiwarakasei`, unless the maintainer decides this is internal-only.
