## Summary

Keep this PR open, but it is not merge-ready: the stale-SUDO_USER fix is aimed at a real Linux gateway lifecycle bug, but the shared resolver change can regress existing sudo-to-root installs before the target unit exists, and the supplied proof is macOS unit-test/build output rather than live systemd behavior.

## What This Changes

The PR makes `resolveSystemctlUserScope` async, checks for a non-root `SUDO_USER` unit file before preferring machine-scoped `systemctl`, and updates daemon systemd tests.

## Best Possible Solution

Keep the stale-`SUDO_USER` fix, but choose root `--user` only when a current/root-owned unit is known to own the lifecycle, preserve install-safe sudo-to-root machine scope, and prove both paths on Linux.

## Reproduction Assessment

Yes. The linked bug is source-reproducible from current main's effective-root plus non-root `SUDO_USER` machine-scope path, and the PR regression is source-reproducible because install probes systemd before a fresh unit exists.

## Solution Assessment

No, not yet. The patch is a plausible stale-env heuristic, but placing it in the shared resolver is too broad; the safer fix needs installed-unit/current-owner context or an install-safe branch.
