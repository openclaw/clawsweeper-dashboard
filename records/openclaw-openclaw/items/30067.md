## Summary

Keep open. Current `main` still lacks the requested first-run setup heads-up before the normal chat handoff, and the request remains narrow core onboarding UX rather than ClawHub-only work.

## What This Changes

Request a notification during initial setup that background initialization can temporarily contribute to provider API rate-limit errors before the first chat session starts.

## Best Possible Solution

Add a concise first-run provider-rate-limit heads-up in the classic setup finalization flow before the TUI/Web/Later choice, cover it with focused onboarding tests and docs, and coordinate wording with the adjacent ETA issue without changing provider runtime, auth, fallback, cooldown, quota, or retry behavior.

## Reproduction Assessment

Yes for the UX gap: current-main source and docs inspection show no setup notice before the first chat handoff. The original live 429 frequency is not independently reproducible from the report because it lacks provider, account, logs, and version details.

## Solution Assessment

Yes, the narrowest maintainable solution is expectation-setting copy in onboarding finalization plus docs/tests. Runtime quota handling should remain separate because the issue asks for a setup heads-up, not a provider retry or cooldown behavior change.
