## Summary

Keep open. Current main and the latest release still lack a proactive setup or first-chat provider quota heads-up; existing docs and runtime handling explain 429/quota errors only after the fact, and the related onboarding ETA work only partially covers this provider-specific expectation gap.

## What This Changes

Request to add a proactive warning during initial setup before chat that provider rate limits may occur while first-run work completes.

## Best Possible Solution

Add maintainer-approved, provider-neutral first-run quota guidance in the onboarding or first-chat handoff path, coordinated with the separate onboarding ETA work where useful.

## Reproduction Assessment

Yes at source level for the missing proactive heads-up: current setup and finalizer paths do not show provider quota/rate-limit guidance before chat. No high-confidence live reproduction exists for the claimed concurrent-background-process cause because the issue lacks provider details, logs, and steps.

## Solution Assessment

No, not exactly as proposed. A provider-neutral first-run quota note is safer than telling users background setup processes caused rate-limit errors without provider-specific proof.
