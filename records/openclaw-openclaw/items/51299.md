## Summary

Keep open. Current main still documents and runs the legacy iMessage path as a gateway-spawned `imsg rpc` child process, while the new Sequoia `.app`/PPPC guidance from the discussion needs maintainer security and product validation before OpenClaw endorses it.

## What This Changes

The issue requests a supported way to grant macOS Full Disk Access and Automation to the legacy iMessage LaunchAgent process on Sequoia/SIP.

## Best Possible Solution

Define the supported legacy imsg stance: either document headless LaunchAgent as unsupported on Sequoia and route users to BlueBubbles, or add maintainer-approved signed app/user-session guidance without recommending TCC database bypasses.

## Reproduction Assessment

Yes at source level: configure `channels.imessage`, run the gateway under the per-user LaunchAgent, and send/probe through `imsg`; current main still attributes the spawned RPC process to the gateway process chain. I did not perform live macOS TCC validation.

## Solution Assessment

Unclear. The `.app` wrapper path in the latest comment is plausible and useful, but the maintainable OpenClaw solution should be an explicit docs/runtime policy reviewed for macOS TCC and security implications.
