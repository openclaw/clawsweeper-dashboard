## Summary

Keep open. Current main and the latest release still run iMessage through a Gateway-spawned `imsg rpc` child while the macOS local path still manages a launchd/Node Gateway; the new `.app` wrapper comment is useful evidence, but OpenClaw has not yet documented or validated it as the supported FDA/Automation path.

## What This Changes

The issue asks for a supported way to grant Full Disk Access and Messages Automation to iMessage when the OpenClaw Gateway runs as a macOS LaunchAgent.

## Best Possible Solution

Define one supported macOS iMessage launch path: document bare LaunchAgent `imsg` as unsupported unless a signed GUI/session or MDM-approved identity owns the permissions, or implement routing through a stable signed OpenClaw app/runtime while preserving explicit TCC consent.

## Reproduction Assessment

Yes at source level, but not with live macOS TCC proof. Configure `channels.imessage`, run the Gateway as the per-user LaunchAgent, and current main still spawns `imsg rpc` from that Gateway process context while OpenClaw and upstream `imsg` docs require FDA and Automation on the launching process context.

## Solution Assessment

Unclear. The signed `.app` wrapper loaded in the GUI domain is plausible and matches the current discussion, but OpenClaw should not endorse it, an MDM profile, or app-broker routing until maintainers validate the security and support model.
