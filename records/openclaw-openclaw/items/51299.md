## Summary

Keep open. Current main still exposes the legacy `imsg` iMessage path as a gateway-spawned child process and still gives process-context TCC prompt guidance for LaunchAgent/SSH users, while BlueBubbles is only the recommended alternative and #43440 covers a narrower Full Disk Access parent-process problem.

## What This Changes

The issue asks for supported guidance or a workaround for granting Full Disk Access and Automation to the legacy iMessage LaunchAgent process on macOS with SIP enabled.

## Best Possible Solution

Keep a supported, explicit macOS iMessage path: update legacy `imsg` docs/setup/status to state whether headless LaunchAgent is unsupported or route it through a signed GUI/session path, recommend BlueBubbles for new deployments, and leave #43440 for the narrower FDA parent-process runtime work if pursued.

## Reproduction Assessment

Yes, at source level: configure legacy `channels.imessage`, run the gateway as the per-user LaunchAgent, and probe/send through `imsg`; current main still spawns `imsg rpc` from the gateway process that would be the TCC-attributed parent. I did not perform live macOS TCC validation.

## Solution Assessment

Unclear. A docs/status diagnostic update is the safest immediate repository change, but endorsing an app wrapper, MDM profile, or brokered Automation/FDA flow needs maintainer and security validation.
