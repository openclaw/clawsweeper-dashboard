## Summary

Keep open. Current main still runs iMessage through a Gateway-spawned `imsg rpc` child, while the macOS app local path still manages a launchd/Node Gateway; related open issues cover adjacent TCC problems, but the supported iMessage FDA/Automation LaunchAgent stance still needs maintainer/security direction.

## What This Changes

The issue asks for a supported way to grant Full Disk Access and Messages Automation to the `imsg` iMessage channel when OpenClaw Gateway runs as a macOS LaunchAgent under Sequoia/SIP.

## Best Possible Solution

Define one supported macOS iMessage path: either document LaunchAgent `imsg` as unsupported without a signed GUI/session wrapper, or provide validated guidance/runtime routing through a stable signed app/MDM-compatible identity while preserving explicit TCC consent.

## Reproduction Assessment

Yes at source level, but not with live macOS TCC proof. Configure `channels.imessage`, run the Gateway as the per-user LaunchAgent, and the current code still spawns `imsg rpc` from that Gateway process context while docs require FDA and Automation there.

## Solution Assessment

Unclear. The `.app` wrapper loaded in the GUI domain is plausible and matches the current discussion, but OpenClaw should not endorse it, an MDM profile, or app-broker routing until maintainers validate the security and support model.
