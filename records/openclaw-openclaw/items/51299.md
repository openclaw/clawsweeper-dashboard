## Summary

Keep open. Current main still runs the legacy iMessage transport by spawning `imsg rpc` from the gateway process, and the docs still tell LaunchAgent/SSH users to trigger TCC prompts from that same process context. The linked #43440 overlaps the Full Disk Access parent-process problem, but this issue also covers Automation `-1743` and headless LaunchAgent guidance, so it should not be closed as a duplicate yet.

## What This Changes

Request a supported workaround or guidance for running the legacy iMessage `imsg` channel from a macOS LaunchAgent on Sequoia/SIP when Full Disk Access and Automation TCC grants fail.

## Best Possible Solution

Keep a supported, explicit path for iMessage on macOS: update legacy `imsg` docs/setup/status to explain the headless LaunchAgent TCC limitation, recommend BlueBubbles or an interactive signed app/user-session path for new deployments, and link the narrower FDA parent-process work in #43440 if a runtime fix is pursued.

## Reproduction Assessment

Yes. The reported path is concrete: configure legacy `channels.imessage`, run the gateway as the per-user LaunchAgent on macOS with SIP enabled, then probe or send via `imsg`; current main still spawns `imsg rpc` from that gateway process, matching the reported TCC attribution path. I did not perform live macOS TCC validation in this checkout.

## Solution Assessment

Unclear. A simple documentation and diagnostic update is the safest immediate repo change, but endorsing an `.app` wrapper, MDM profile, or any preapproval flow needs maintainer/security validation against macOS TCC behavior.
