## Summary

Current source still starts the legacy iMessage transport as `imsg rpc` from the gateway process, and the docs still advise same-context interactive approval even though this report says LaunchAgent approval cannot be prompted in that context. The related FDA issue covers a narrower parent-process grant problem, but this report also includes Automation/LaunchAgent/TCC setup behavior, so it should remain open for maintainer follow-up.

## What This Changes

The issue asks for a supported way to grant Full Disk Access and Automation to the legacy iMessage `imsg` channel when OpenClaw runs as a macOS LaunchAgent under a dedicated user.

## Best Possible Solution

Document and/or implement a maintainer-approved LaunchAgent-safe iMessage path: either route legacy `imsg` through a GUI-approved context, clearly steer LaunchAgent users to BlueBubbles, or add a supported macOS app/node bridge if the legacy channel is expected to work headlessly.

## Reproduction Assessment

Unclear for live reproduction. Source inspection confirms OpenClaw still spawns `imsg rpc` from the gateway process and current docs still depend on process-context TCC grants, but I did not run a macOS Sequoia LaunchAgent with SIP to reproduce the GUI prompt failure.

## Solution Assessment

Unclear. A narrow docs update is possible, but the durable answer needs maintainer choice between legacy imsg support, BlueBubbles guidance, or a macOS app/node execution path for TCC-sensitive work.
