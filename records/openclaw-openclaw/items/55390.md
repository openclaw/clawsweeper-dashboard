## Summary

Keep open. Current main still lacks the local schema artifact maintenance requested here, and the PR is useful but not merge-ready because its startup `$schema` backfill can damage or repeatedly trip over include-owned configs and the user-facing feature is missing a changelog entry.

## What This Changes

The PR adds best-effort local `openclaw_schema.json` generation, `$schema` backfilling on CLI startup/config writes, an `update.maintainConfigJsonSchema` opt-out, docs, and targeted tests.

## Best Possible Solution

Rebase the feature onto current main, keep local schema generation best-effort, preserve custom `$schema` values, avoid mutating root configs whose authored source is owned by `$include`, and add regression coverage plus the required changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a config authored as a root `$include` with no `$schema`, followed by a read-style CLI command that runs the new startup maintenance; the PR path tries to persist `$schema` from the resolved snapshot, while current main’s include-owned write tests show that touching that root must be preserved or rejected.

## Solution Assessment

No. Local schema generation is a reasonable direction, but writing a resolved config snapshot during startup is not the safest implementation; the narrow fix is an authored-root-aware mutation path or skipping root `$schema` backfill for include-owned roots while still writing the schema file when safe.
