## Summary

Keep open. The local schema maintenance feature is still not on current main, but this PR should not merge as-is because it still has an include-owned config write blocker, a docs hierarchy regression, missing real behavior proof, and a dirty/conflicting head.

## What This Changes

The PR adds CLI/config code, docs, and tests to generate and maintain `openclaw_schema.json`, default `$schema`, and expose `update.maintainConfigJsonSchema`.

## Best Possible Solution

Repair or replace the branch with an include-aware, idempotent schema maintenance path that preserves custom `$schema`, avoids unsafe startup config mutation, keeps docs hierarchy intact, and includes redacted after-fix proof.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows root `$include` configs with no `$schema` go through `writeConfigFile(nextConfig, writeOptions)`, while current main rejects writes that touch include-owned root paths. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. Local schema maintenance is a reasonable direction, but the startup path should patch only the authored root when safe or skip root mutation for include-owned configs while still doing best-effort schema-file generation.
