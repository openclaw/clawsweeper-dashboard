## Summary

Keep open. Current main still lacks automatic local schema artifact maintenance, and this PR remains a plausible implementation candidate, but it has a source-reproducible include-owned config write regression, a docs hierarchy regression, merge conflicts, and missing real behavior proof before merge.

## What This Changes

The PR adds default-on CLI/config behavior to maintain `openclaw_schema.json` beside `openclaw.json`, backfill `$schema`, expose `update.maintainConfigJsonSchema`, update docs, and add tests.

## Best Possible Solution

Rebase and land or replace this PR with an include-aware implementation that keeps schema generation best-effort, avoids unsafe root-config mutation, preserves custom `$schema` values and current schema metadata, keeps docs hierarchy intact, and includes real behavior proof.

## Reproduction Assessment

Yes for the blocking PR behavior at source level: use a root config authored with `$include` and no `$schema`, then run a normal CLI command that triggers startup maintenance. I did not execute it because this review is read-only, but the PR write path and current-main include-owned write guard make the failure path clear.

## Solution Assessment

No. Local schema artifact maintenance is a reasonable direction, but this implementation should patch only the authored root safely or skip root mutation for include-owned configs instead of writing the resolved snapshot during startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve include-owned configs during schema backfill:** `src/config/local-json-schema.ts:96-98`

For a root config authored with `$include`, that startup write either flattens the authored root on the old base or repeatedly warns after rebase against current main's include-owned write guard; patch the authored root directly or skip root mutation for include-owned configs.
  - confidence: 0.88

  - confidence: 0.74
