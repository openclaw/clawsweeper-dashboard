## Summary

Keep open: current main still lacks local `openclaw_schema.json` maintenance, and this PR is the active implementation for the open schema-maintenance feature, but the branch has blocking correctness issues and lacks real after-fix behavior proof.

## What This Changes

The PR adds default-on CLI maintenance that writes `openclaw_schema.json`, backfills `$schema`, exposes `update.maintainConfigJsonSchema`, updates docs, and adds config/CLI tests.

## Best Possible Solution

Repair this PR or replace it with an include-aware implementation that keeps schema generation best-effort, backfills only the authored root safely or skips include-owned root mutation, preserves custom `$schema`, keeps docs headings intact, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: use a root `openclaw.json` authored with `$include` and no `$schema`, then run a normal CLI command that triggers the PR startup maintenance. The PR writes from `snapshot.config`, while current-main write preparation rejects root include-owned patches.

## Solution Assessment

No. Local schema artifact maintenance is a reasonable direction, but this branch should patch the authored root safely or skip root mutation for include-owned configs instead of writing the resolved runtime snapshot during startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve include-owned configs during schema backfill:** `src/config/local-json-schema.ts:96-98`
  - body: `maintainLocalConfigJsonSchemaArtifacts` derives `nextConfig` from the resolved runtime snapshot and passes it to `writeConfigFile` when `$schema` is missing. For a root config authored with `$include`, current-main write preparation rejects that root patch, so normal CLI runs warn and never maintain the schema artifacts. Patch only the authored root safely or skip root mutation for include-owned configs while still allowing best-effort schema-file generation.
  - confidence: 0.9
- **[P3] Keep Paths as a peer config section:** `docs/cli/config.md:47`
  - body: Changing `### Paths` to `#### Paths` nests the general `config get/set/patch` path syntax under the `config schema` section. Leave `Paths` at the existing peer heading level or move the schema note under its own subsection.
  - confidence: 0.74
