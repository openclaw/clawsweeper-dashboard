## Summary

This PR should stay open: current main still does not implement automatic local `openclaw_schema.json` maintenance, and this PR remains the active implementation candidate for the paired feature issue. The patch is not merge-ready because startup backfill writes from the resolved config snapshot, which is unsafe for `$include`-owned roots, and the user-facing feature still needs a changelog entry.

## What This Changes

The PR adds best-effort local config JSON schema generation, startup/config-write `$schema` backfilling, an `update.maintainConfigJsonSchema` opt-out, docs, schema metadata, and tests.

## Best Possible Solution

Land a rebased implementation that keeps schema generation best-effort, preserves custom `$schema` values, avoids root mutation for `$include`-owned configs, and records the user-facing behavior in the active changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a root `$include` config with no root `$schema`, followed by a read-style CLI command that runs the PR's startup maintenance; the PR writes from `snapshot.config` while current main explicitly rejects include-owned root writes.

## Solution Assessment

No. Local schema generation is a reasonable feature direction, but writing a resolved config snapshot during startup is not the narrowest safe implementation; the backfill should patch only the authored root when safe or skip root `$schema` mutation for include-owned roots while still updating the schema file.
