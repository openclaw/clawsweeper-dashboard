## Summary

Keep open. Current main has several Active Memory latency mitigations, but it still does not implement warm standby or persistent embedded-agent session reuse; uncached recalls still create a fresh Active Memory sub-agent session and run through the embedded runner setup path. The discussion also has multiple independent timing reports plus related regression context, so this is a real remaining performance/design item rather than stale cleanup.

## What This Changes

The issue requests warm standby or persistent session reuse for Active Memory embedded-agent recalls to avoid repeated cold-start setup latency.

## Best Possible Solution

Keep this as Active Memory embedded-agent warm reuse work. A maintainer-quality end state would add bounded TTL/invalidation for repeated Active Memory recalls, prove that repeated eligible turns skip expensive setup, and preserve strict per-parent-session isolation so recall state cannot leak across users, agents, channels, or model/auth changes.

## Reproduction Assessment

Yes for the missing reuse behavior: static inspection and a focused mock of two uncached Active Memory prompt-build calls can show a fresh sub-agent session id and a new runEmbeddedPiAgent call each time. The exact latency numbers still require a live gateway/provider timing run.

## Solution Assessment

Unclear. Warm standby/session reuse is a plausible solution, but the best implementation boundary is not settled because it may belong in the Active Memory plugin, a generic embedded-runner/session API, or a hybrid with explicit invalidation and isolation contracts.
