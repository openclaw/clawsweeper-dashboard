## Summary

Keep open. Current main still lacks Active Memory warm standby or persistent embedded-session reuse; the newer related reports narrow specific overhead sources, but they do not replace the Active Memory lifecycle and isolation design requested here.

## What This Changes

Request a warm-standby, pre-initialized, or persistent embedded-agent session mechanism so Active Memory does not cold-start `runEmbeddedPiAgent` on every recall.

## Best Possible Solution

Design a bounded per-parent-session Active Memory warm cache or generic embedded-runner reuse seam with explicit TTL, invalidation, and isolation semantics before implementation.

## Reproduction Assessment

Yes for the missing reuse behavior: source inspection shows uncached recalls still create a fresh Active Memory sub-agent session id/key and enter `runEmbeddedPiAgent` each time. The exact current wall-clock latency still needs a live gateway/provider timing run.

## Solution Assessment

Unclear. Warm standby/session reuse is plausible, but maintainers should first choose the boundary between an Active Memory plugin cache, a generic embedded-runner reuse API, or a hybrid with strict isolation.
