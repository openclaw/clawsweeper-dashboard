## Summary

Keep open: current main still lacks Active Memory warm standby or persistent embedded-session reuse, and the closed latency reports only cover adjacent startup/tool-factory bottlenecks rather than this lifecycle design request.

## What This Changes

The issue requests warm standby or persistent session reuse for Active Memory embedded agent runs to reduce repeated cold-start latency.

## Best Possible Solution

Design a bounded per-parent-session Active Memory warm cache or generic embedded-runner reuse seam with explicit TTL, invalidation, isolation semantics, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing reuse behavior: current source still creates a fresh Active Memory embedded session on uncached recalls. I did not reproduce the reported 10-30s latency in a live gateway/provider setup.

## Solution Assessment

Unclear. Warm standby is plausible, but maintainers should first choose whether the boundary belongs in Active Memory, the generic embedded runner, or a hybrid API with strict isolation and invalidation rules.
