## Summary

Keep open: current main still lacks Active Memory warm standby or persistent embedded-session reuse, while the discussion has multiple corroborating latency reports and the related closed reports only cover adjacent startup, tool-factory, and cache bottlenecks.

## What This Changes

Request adding warm standby or persistent session reuse for Active Memory embedded agent runs to avoid per-message cold-start latency.

## Best Possible Solution

Design a bounded per-parent-session Active Memory warm cache or generic embedded-runner reuse seam with explicit TTL, invalidation, isolation semantics, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing reuse behavior: current source still creates fresh Active Memory embedded subagent state on uncached recalls. I did not reproduce the reported 10-30s latency in a live gateway/provider setup.

## Solution Assessment

Unclear. Warm standby is plausible, but maintainers should first choose whether the boundary belongs in Active Memory, the generic embedded runner, or a hybrid API with strict isolation and invalidation rules.
