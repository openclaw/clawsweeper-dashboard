## Summary

Keep open: the WhatsApp plugin is the right boundary and the reporter's live reconnect context supports the user problem, but this PR is not mergeable as submitted because the opt-in age gate lets invalid timestamps through and the public config/schema/docs surfaces are incomplete.

## What This Changes

The PR adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` config, resolves them per account, and passes them into append-upsert handling after reconnect.

## Best Possible Solution

Repair this PR into a safe opt-in WhatsApp feature with strict timestamp handling, generated config/schema/docs updates, focused regression tests, and a changelog entry; keep #50093 as the canonical broader backfill follow-up if needed.

## Reproduction Assessment

Yes for the patch defects: an append upsert with `replyToOfflineMessages: true` and a missing or non-finite `messageTimestamp` follows the PR's `msgTsMs &&` branch and bypasses the max-age check. The existing WhatsApp monitor harness already covers append timestamp behavior on current main.

## Solution Assessment

No, not as submitted. The extension-owned opt-in direction is maintainable, but the implementation must reject invalid timestamps and publish the config through generated schema/docs before users can rely on it.
