## Summary

Keep open. The PR targets a real WhatsApp offline `append` gap and builds on useful #21719/#50093 context, but it is not merge-ready because the opt-in path can process invalid/untimestamped history and the new config keys are missing from generated validation/schema surfaces.

## What This Changes

Adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` config, resolves them per account, and passes them into inbound append-message handling.

## Best Possible Solution

Keep the extension-owned opt-in feature direction, but require finite timestamp enforcement, generated config metadata/schema/docs/changelog updates, and focused tests before merge; leave #50093 as the canonical follow-up for broader history sync if needed.

## Reproduction Assessment

Yes for the patch defects: the existing WhatsApp monitor harness can emit `append` upserts with stale or non-finite timestamps and pass `replyToOfflineMessages: true`, and config validation can be checked against a `channels.whatsapp.replyToOfflineMessages` sample. Live reconnect proof is not required to catch these specific regressions.

## Solution Assessment

No, not as submitted. The WhatsApp plugin is the right boundary for this opt-in behavior, but the implementation must reject missing/invalid timestamps and publish the config through generated validation/docs surfaces before it is safe and usable.

Full review comments:

- **[P2] Reject untimestamped offline appends:** `extensions/whatsapp/src/inbound/monitor.ts:773`
  - body: With `replyToOfflineMessages` enabled, an invalid or missing `messageTimestamp` becomes `0`, and the `msgTsMs && Date.now() - msgTsMs > maxAgeMs` guard is false. That lets untimestamped history-sync messages bypass the flood-prevention age cap and get enqueued, even though the existing default-path tests treat non-finite append timestamps as stale.
  - confidence: 0.91

  - confidence: 0.88
