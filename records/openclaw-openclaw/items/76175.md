## Summary

Keep open. The PR targets a real WhatsApp offline-message gap at the right plugin boundary, but the submitted patch is not mergeable because the opt-in path weakens the timestamp age gate and the new config keys are not published through generated validation/docs surfaces.

## What This Changes

The PR adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` config, resolves them per account, and passes them into append-upsert handling so offline catch-up messages can be processed when enabled.

## Best Possible Solution

Keep the WhatsApp plugin-owned opt-in direction, but require strict finite timestamp enforcement, generated config metadata/schema/docs/changelog updates, and focused append tests before merge; leave broader reconnect backfill tracked in #50093.

## Reproduction Assessment

Yes for the patch defects: the existing WhatsApp monitor harness can emit append upserts with stale or non-finite timestamps, and config validation can be checked against a `channels.whatsapp.replyToOfflineMessages` sample. The reporter's live timeline also supports the underlying current-main gap, but live WhatsApp reconnect proof is not required for these review findings.

## Solution Assessment

No, not as submitted. The WhatsApp plugin is the right boundary for an opt-in, but the implementation must reject missing/invalid timestamps and publish the config through generated validation/docs surfaces before it is safe and usable.
