## Summary

Keep open, but this needs changes before merge. The WhatsApp plugin is the right boundary for a bounded reconnect/offline-message recovery feature, but the submitted branch still has a timestamp age-gate regression, does not publish the new config through generated validation/schema surfaces, lacks after-fix real WhatsApp proof, and overlaps with the product direction being discussed in https://github.com/openclaw/openclaw/issues/50093 and https://github.com/openclaw/openclaw/pull/80642.

## What This Changes

The PR adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` config, resolves the fields per account, and uses them to process bounded `append` upserts after reconnect.

## Best Possible Solution

Land one plugin-owned reconnect recovery path that caps replay, rejects missing or non-finite timestamps, dedupes by message ID, publishes any new config through generated surfaces, documents the behavior, and is proven on a real WhatsApp reconnect.

## Reproduction Assessment

Yes for the patch defects: source inspection shows the invalid-timestamp opt-in branch and generated schema mismatch, and the existing WhatsApp monitor harness can exercise those paths. No after-fix live WhatsApp reconnect proof is present.

## Solution Assessment

No, not as submitted. The WhatsApp plugin boundary is appropriate, but this branch needs timestamp hardening, generated config/schema/docs alignment, focused tests, and a maintainer choice against the narrower related reconnect PR.
