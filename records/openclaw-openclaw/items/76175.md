## Summary

Keep open: this is useful WhatsApp work, but the current PR head is not mergeable because the new config is not wired through strict validation/generated surfaces and the opt-in append path weakens the timestamp flood guard.

## What This Changes

Adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` config fields, resolves them per account, and passes them into append upsert handling so recent offline messages can be processed after reconnect.

## Best Possible Solution

Land this as an extension-owned opt-in WhatsApp feature only after the branch preserves the default append behavior, rejects missing or non-finite offline timestamps, publishes the config through validation/docs/generated surfaces, and adds focused regression coverage.

## Reproduction Assessment

Yes for the PR defects: source inspection shows an append upsert with `replyToOfflineMessages: true` and a missing/non-finite timestamp can bypass the intended age cap, and config validation can reject `channels.whatsapp.replyToOfflineMessages` because the strict schema lacks the key. The live reconnect problem has reporter evidence, but these blockers do not require live WhatsApp proof.

## Solution Assessment

No, not as submitted. The WhatsApp plugin is the right boundary for this opt-in behavior, but the current patch needs strict timestamp handling, Zod/generated schema updates, docs/changelog, fixture updates, and focused tests before it is the maintainable fix.
