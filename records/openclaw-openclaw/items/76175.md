## Summary

Keep open. Current main only handles very recent WhatsApp append upserts, and this PR targets a real remaining gap, but the branch is not merge-ready because the opt-in path can bypass its age cap for invalid timestamps and the new config is not exposed through the runtime/generated config surfaces.

## What This Changes

The PR adds WhatsApp `replyToOfflineMessages` and `offlineMessageMaxAgeSeconds` settings and passes them into append-upsert handling after reconnect.

## Best Possible Solution

Keep the WhatsApp-plugin-owned opt-in direction, but require finite timestamp enforcement, runtime and generated config schema updates, docs/changelog coverage, and focused tests before merge.

## Reproduction Assessment

Yes for the patch defects: the existing WhatsApp monitor harness can emit append upserts with stale or non-finite timestamps, and the strict WhatsApp config schema can be checked with a `channels.whatsapp.replyToOfflineMessages` sample. The original reconnect problem is also supported by the PR author's live test comments, though broader backfill remains separate.

## Solution Assessment

No, not as submitted. The extension boundary is the right place for this opt-in behavior, but the implementation must reject invalid timestamps and publish the config through the runtime/generated schema surfaces before users can safely enable it.

  - confidence: 0.9
