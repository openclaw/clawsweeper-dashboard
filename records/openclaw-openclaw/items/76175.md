## Summary

Keep open. The WhatsApp offline-message opt-in addresses a real gap, but the PR is not merge-ready because the opt-in age guard lets invalid timestamps bypass the flood cap and the new config keys are not wired into the strict runtime/generated schema, docs, or changelog.

## What This Changes

The PR adds WhatsApp config fields for processing append/offline messages after reconnect and threads those options through account resolution into inbound monitoring.

## Best Possible Solution

Keep the WhatsApp plugin-owned opt-in direction, but require finite timestamp rejection, zod/generated config schema updates, docs/changelog, and focused tests before merge while leaving #50093 as the broader backfill tracker.

## Reproduction Assessment

Yes for the patch defects: the existing monitor harness can emit append upserts with missing or non-finite timestamps, and config validation can be checked with a `channels.whatsapp.replyToOfflineMessages` sample. The reporter's live reconnect comments also support the underlying user problem.

## Solution Assessment

No, not as submitted. The WhatsApp plugin is the right boundary for this opt-in behavior, but the implementation must reject invalid timestamps and publish the config through the strict schema/generated/docs surfaces before it is safe and usable.
