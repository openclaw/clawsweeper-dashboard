## Summary

Keep open. This PR targets a real P1 secrets migration bug, but the current patch is still not merge-ready because it narrows `.env` cleanup too far for file/exec SecretRef migrations and still cross-pairs independent env-key and migrated-value sets.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing to env ref IDs recorded during config/auth-profile mutation and adds same-value provider regression tests.

## Best Possible Solution

Add a maintainer-approved target-owned `.env` scrub contract, likely through secret target registry metadata, that pairs each eligible env key only with values owned by that same target or provider across env, file, exec, auth-profile, and plugin targets.

## Reproduction Assessment

Yes, by source inspection and the linked issue proof; this read-only review did not execute the write path. The PR head shows file/exec targets can add values without keys and the scrubber still receives independent key/value sets.

## Solution Assessment

No. The env-ref-ID filter is a plausible narrow mitigation, but the best fix is an explicit target-owned key/value scrub contract that covers sibling target classes without reintroducing over-scrub or under-scrub cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind scrub keys to their owning values:** `src/secrets/apply.ts:705`
  - body: Passing separate `scrubbedValues` and `scrubbedEnvKeys` still forms a cross product: one target can contribute an env key while another contributes the matching value, causing `scrubEnvRaw` to delete a live env line that was not migrated as that key/value pair. Build a per-env-key value map from the target that owns both sides instead.
  - confidence: 0.9
- **[P1] Preserve file and exec cleanup coverage:** `src/secrets/apply.ts:697-698`
  - body: Because `scrubbedEnvKeys` is only populated when `target.ref.source === "env"`, a target migrated to a file or exec SecretRef can add an old plaintext value but no eligible env key, and the new early return skips `.env` cleanup entirely. That regresses the documented cleanup for known `.env` secret keys whose values were migrated.
  - confidence: 0.91
