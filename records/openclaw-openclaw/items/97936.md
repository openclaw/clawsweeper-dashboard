## Label Justifications

- `P1`: The PR targets a high-priority secrets migration bug that can delete or preserve provider credentials incorrectly in the default apply/configure workflow.
- `merge-risk: 🚨 compatibility`: The diff changes shipped `secrets apply` cleanup semantics for non-env SecretRef migrations that previously scrubbed matching known `.env` secret keys.
- `merge-risk: 🚨 security-boundary`: The changed path handles credential scrubbing and can leave plaintext credential residues or remove live credential sources.

## Summary

Keep open: this is a focused bug-fix PR, but it is not merge-ready because the new env-key tracking misses file/exec SecretRef cleanup and still carries independent key/value sets instead of target-owned pairs.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing by tracking env ref IDs for migrated targets and adds same-value credential regression tests.

## Best Possible Solution

Use an explicit env-scrub ownership contract, preferably target-registry-owned, and pass a per-env-key set of owned migrated values into `scrubEnvRaw` with coverage for env, file, exec, provider-auth, auth-profile, and plugin targets.

## Reproduction Assessment

Yes by source inspection and prior same-root proof. A plan that migrates a file/exec SecretRef target with `scrubEnv` enabled now has migrated values but no `scrubbedEnvKeys`, so `.env` cleanup is skipped for that target class.

## Solution Assessment

No. This is a plausible narrow mitigation, but the maintainable fix needs target-owned env-key/value pairing rather than env-ref-id ownership plus one global migrated-value set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bind scrub keys to their owning values:** `src/secrets/apply.ts:705`
  - body: Passing separate `scrubbedValues` and `scrubbedEnvKeys` sets still forms a cross product: one target can contribute `OPENAI_API_KEY` while another contributes the matching value, causing `scrubEnvRaw` to delete a live env line that was not migrated as that key/value pair. Build a per-env-key value map from the target that owns both sides instead.
  - confidence: 0.87
- **[P2] Preserve file and exec cleanup coverage:** `src/secrets/apply.ts:697-698`
  - body: Because `scrubbedEnvKeys` is only populated when `target.ref.source === "env"`, a target migrated to a file or exec SecretRef can add an old plaintext value but no eligible env key, and the new early return skips `.env` cleanup entirely. That regresses the documented/shipped behavior that scrubs known `.env` secret keys whose values were migrated.
  - confidence: 0.9
