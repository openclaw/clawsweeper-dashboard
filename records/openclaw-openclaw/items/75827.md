## Summary

This issue remains valid on current main: the Docker Release workflow still has no main-branch push trigger, the Docker docs still list `main` as a common pre-built image tag, and live GHCR metadata shows `ghcr.io/openclaw/openclaw:main` still pointing at the March 26 image revision. The reporter's narrower point that no published image contained `29ed5266` is now mitigated by `latest`/`2026.5.2`, but the stale `:main` channel contract still needs maintainer release-policy handling.

## What This Changes

The issue asks OpenClaw to make the GHCR `:main` image track git `main` again, or stop guiding users toward `main` when only release images are current.

## Best Possible Solution

Define the Docker image channel contract and align workflow, docs, and maintainer guidance around it: guarded `main`/`main-slim` publishing from main pushes, or explicit deprecation of Docker `:main` as a current-main test path.

## Reproduction Assessment

Yes. Current-main source inspection shows no normal main-branch trigger for Docker Release, GitHub Actions shows the last main-branch Docker Release run on 2026-03-26, and live GHCR metadata still points `:main` at that March 26 revision.

## Solution Assessment

Unclear because no patch is proposed and both plausible fixes carry release-policy implications. Re-enabling main-push publishing is the direct behavioral fix, while deprecating Docker `:main` may be safer if release tags are intended to be the only supported published image artifacts.
