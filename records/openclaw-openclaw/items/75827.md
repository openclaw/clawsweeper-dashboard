## Summary

Keep open. Current main still does not rebuild the GHCR `:main` image from branch pushes, Docker docs still advertise a `main` tag, and live registry inspection confirms `:main` points to a March image rather than current main; resolving this needs an explicit Docker channel/release decision, not cleanup closure.

## What This Changes

Requests that `ghcr.io/openclaw/openclaw:main` either track git `main` automatically or that docs and maintainer guidance stop presenting it as current main.

## Best Possible Solution

Define one Docker image channel contract and align workflow, docs, and maintainer guidance: either guarded `main`/`main-slim` publishing from branch pushes with attestations, or explicit deprecation of Docker `:main` as a current-main test path.

## Reproduction Assessment

Yes. The source-level reproduction is to inspect `.github/workflows/docker-release.yml` and see no branch-main trigger, and the live registry reproduction is `skopeo inspect docker://ghcr.io/openclaw/openclaw:main`, which currently reports a 2026-03-26 image revision.

## Solution Assessment

Unclear until release maintainers choose the desired channel contract. Auto-publishing `:main` is technically direct because main-tag resolution already exists, but a docs/guidance change may be safer if OpenClaw does not want mutable Docker images from every main push.
