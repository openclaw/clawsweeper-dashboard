## Summary

Keep open. The report is valid against current main: the Docker release workflow has tag-push/manual triggers only, while the same workflow still contains `refs/heads/main` tag-resolution branches and the Docker docs still advertise a `main` image tag. The related #73532 example is useful impact evidence, and the older #46076 cleanup covered release-tag publishing rather than the moving `:main` contract.

## What This Changes

The issue asks OpenClaw to either rebuild `ghcr.io/openclaw/openclaw:main` on each push to git `main` or stop directing users to a Docker tag that does not match current main.

## Best Possible Solution

Define one explicit Docker channel contract and make code and docs match it: either restore a guarded main-branch image publishing path for `:main`/`:main-slim`, or deprecate that tag in docs and maintainer guidance so users are sent to a release tag, npm/git `main`, or source build when a fix is only on current main.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: inspect `.github/workflows/docker-release.yml` on current main and observe that only `v*` tag pushes/manual backfills trigger it, while the issue’s Docker pull/manifest data provides an external registry reproduction.

## Solution Assessment

Unclear until maintainers choose the desired release contract. Auto-building `:main` is technically direct because the workflow already has main-tag branches, but a docs/guidance change may be safer if OpenClaw does not want mutable Docker images published from every main push.
