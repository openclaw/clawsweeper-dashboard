## Summary

Keep open. Current main still has a Docker release workflow that only runs on version tags or manual backfills, while Docker docs still list `main` as a common pre-built tag and live GHCR inspection shows `:main` still points to a March image revision.

## What This Changes

The issue requests that `ghcr.io/openclaw/openclaw:main` either track git `main` again or that OpenClaw stop directing users to it as a current-main test image.

## Best Possible Solution

Define one Docker image channel contract and align workflow, docs, and maintainer guidance: either guarded `main`/`main-slim` publishing from main pushes with attestation checks, or explicit deprecation/removal of Docker `:main` as a current-main test path.

## Reproduction Assessment

Yes. Source inspection reproduces the missing auto-rebuild path by showing no branch-main trigger in `.github/workflows/docker-release.yml`, and live registry inspection reproduces the stale artifact by showing `:main` was created on 2026-03-26 with revision `4ed58956375be8b16fb3559699f1b0cb3661dbce`.

## Solution Assessment

Unclear because no patch is proposed and the two plausible fixes are release-policy choices. Restoring `:main` publishing is direct but supply-chain sensitive; deprecating the tag in docs and guidance may be safer if OpenClaw wants Docker images to remain release-tag driven.
