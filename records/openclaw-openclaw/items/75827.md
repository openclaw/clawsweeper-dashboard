## Summary

Keep open: the issue is still valid on current main. The Docker Release workflow cannot run on ordinary `main` pushes, the Docker docs still present `main` as a common pre-built image tag, and live GHCR metadata still shows `ghcr.io/openclaw/openclaw:main` at a March 26 revision; the newer `latest` release image only resolves the reporter's temporary “no image has the cited fix” point, not the stale `main` channel.

## What This Changes

The issue asks OpenClaw to align the GHCR Docker `main` image tag with the git `main` branch or stop directing users to that tag as current-main proof.

## Best Possible Solution

Define the Docker image channel contract and align workflow, docs, and maintainer guidance: either re-enable guarded `main`/`main-slim` publishing from main pushes with existing attestations, or deprecate Docker `:main` as a current-main testing path and direct users to release tags or local builds.

## Reproduction Assessment

Yes. Current-main source inspection shows ordinary main pushes cannot run the Docker Release workflow, and live GHCR inspection shows `:main` still points to the March 26 image revision instead of current main.

## Solution Assessment

Unclear because no patch is proposed and both plausible fixes are policy choices. Re-enabling `main` publishing is the direct behavioral fix; deprecating or documenting the tag may be safer if only release images should be published.
