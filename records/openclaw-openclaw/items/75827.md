## Summary

This issue should stay open. Current main still advertises `ghcr.io/openclaw/openclaw:main` as a common Docker tag and the Docker release workflow contains `:main` publishing branches, but the workflow is only triggered by `v*` tag pushes or manual tag backfills, so normal git `main` commits do not rebuild the image. A live GHCR manifest/config check confirms the `:main` image still points to a March 26 build rather than current main or the latest release.

## What This Changes

The issue requests that the GHCR Docker `:main` tag be rebuilt automatically on git `main` pushes, or that maintainer guidance/docs stop treating `:main` as current main.

## Best Possible Solution

Make the Docker image policy explicit: either wire Docker Release to build and push `:main` on accepted git `main` pushes while preserving release/backfill behavior, or remove/clarify the `:main` guidance so users are sent to a release tag until a fix has shipped.

## Reproduction Assessment

Yes. Current source inspection plus a live GHCR registry config check reproduce the reported mismatch: `:main` is a March 26 image at revision `4ed58956375be8b16fb3559699f1b0cb3661dbce`, not current main `58f2d17e9e05f76c382c47e8a533af3595df0231`.

## Solution Assessment

Unclear. Auto-building `:main` on every main push is the closest behavior to the existing docs and workflow tag logic, but it is a release/publishing policy change; a docs-only correction is safer operationally but does not satisfy users who expect a current-main image.
