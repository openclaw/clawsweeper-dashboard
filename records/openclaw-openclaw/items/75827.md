## Summary

Keep open. Current main still has a tag/manual-only Docker publishing workflow, Docker docs still advertise `main` as a common pre-built tag, and live GHCR inspection confirms `:main` is still pinned to a March image.

## What This Changes

The issue asks OpenClaw to either rebuild `ghcr.io/openclaw/openclaw:main` from every git `main` push or stop directing users to a stale `:main` Docker image for current-main verification.

## Best Possible Solution

Define one Docker image channel contract and make workflow, docs, and maintainer guidance agree: either guarded `main`/`main-slim` publishing from main pushes, or explicit deprecation of Docker `:main` as a current-main artifact.

## Reproduction Assessment

Yes. Source inspection reproduces the missing auto-rebuild path by showing no branch-main trigger in `.github/workflows/docker-release.yml`, and live registry inspection reproduces the stale artifact with a March 26 `:main` image revision.

## Solution Assessment

Unclear because no patch is proposed and both plausible fixes are release-policy choices. Restoring `:main` publishing is direct, while documenting release-tag/source-build guidance may be safer if Docker images are intended to stay release-driven.
