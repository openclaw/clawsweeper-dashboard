## Summary

Keep open. Current main still has a reproducible Docker channel mismatch: the Docker release workflow does not run on normal pushes to git main, docs still list the floating `main` image tag, and live GHCR metadata shows `ghcr.io/openclaw/openclaw:main` still points to the March 26 image revision rather than current main. The fix is release/supply-chain policy, so this should stay with maintainers instead of being auto-cleaned or auto-repaired.

## What This Changes

The issue asks OpenClaw to either rebuild `ghcr.io/openclaw/openclaw:main` from every git main push or stop directing users to that stale floating Docker tag.

## Best Possible Solution

Define the Docker image channel contract and align workflow, docs, and maintainer guidance around it: either guarded `main`/`main-slim` publishing from main pushes, or explicit deprecation of Docker `:main` as a current-main test path.

## Reproduction Assessment

Yes. `skopeo inspect` against both amd64 and arm64 `ghcr.io/openclaw/openclaw:main` shows March 26 image revisions, while current main is `983064f5f8199422f813006fbeff3e1f7013f185` and the workflow source has no normal main-branch trigger.

## Solution Assessment

Unclear because no patch is proposed and both plausible fixes are policy choices. Re-enabling guarded main-push publishing is the direct behavioral fix, while deprecating or documenting the Docker `:main` tag may be safer if releases are intended to be the only published image artifacts.
