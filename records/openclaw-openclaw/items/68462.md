## Summary

Keep open. Current main still has the direct `.bak` restore gap, and this PR is the right implementation lane but is not merge-ready because it is conflicting, lacks real behavior proof, and has blocking detector gaps.

## What This Changes

The PR adds a parsed-config redaction-placeholder detector, uses it to refuse async and sync `.bak` restores, and adds config recovery tests for that refusal path.

## Best Possible Solution

Land a rebased guard before `copyFile` and `copyFileSync` that follows current sensitive path semantics and the actual `maskApiKey` contract, while tracking backup creation, rotation, and CLI validation as separate follow-up scope.

## Reproduction Assessment

Yes. Source inspection on current main shows that a suspicious live config plus a valid-looking `.bak` with redacted sensitive values reaches the async or sync backup copy path without inspecting `backupParsed` for pollution.

## Solution Assessment

No as-is. Refusing polluted backups before direct restore is the right fix shape, but the detector should use schema/path-sensitive placeholder detection and accept actual `maskApiKey` outputs before merge.

## Review Findings

Full review comments:

- **[P2] Scan schema-sensitive paths before restoring backups:** `src/config/backup-pollution.ts:12-20`
  - body: `findRedactedSecretSites` only checks the leaf key against this suffix list, but supported sensitive config values can live under leaves such as `value`, `key`, `passphrase`, arbitrary request header names, or local service env names.   - confidence: 0.91

  - body: `maskApiKey` preserves whatever leading and trailing characters the original secret had, but this regex only allows a fixed punctuation subset around the ellipsis. Masked values containing other punctuation can be missed, so the restore guard can still copy a polluted backup.
  - confidence: 0.84
