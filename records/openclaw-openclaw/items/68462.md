## Summary

Keep open. Current main still has the direct `.bak` restore gap, so the PR targets a real config recovery bug, but the proposed detector misses schema-sensitive TLS fields and some actual `maskApiKey` outputs; it should not merge until those detector gaps are fixed and the branch is rebased.

## What This Changes

The PR adds a redacted-secret backup pollution scanner and wires async/sync suspicious config recovery to refuse restoring polluted `.bak` files.

## Best Possible Solution

Keep the `.bak` restore guard, but implement one shared redacted-placeholder detector aligned with schema-sensitive config paths and actual `maskApiKey` output shapes, then cover gateway token, TLS material, punctuation-bearing masks, and both async/sync recovery paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed a recoverable `.bak` with a redacted sensitive value and a live update-channel-only config; current main reaches `copyFile`, and the PR still misses cases such as `models.providers.*.request.tls.passphrase: "***"`.

## Solution Assessment

No as-is. The guard belongs in this recovery path, but the detector is too narrow; the safer fix is to reuse or align with the repository's sensitive config metadata and the real masking helper contract.
