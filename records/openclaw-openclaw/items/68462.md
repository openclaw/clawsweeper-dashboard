## Summary

Keep open. The PR targets a real direct `.bak` restore pollution bug that still exists on current main, but the proposed detector is too narrow for current sensitive config paths and actual `maskApiKey` output, so it needs focused changes before merge.

## What This Changes

The PR adds a redacted-secret backup pollution scanner and wires async/sync suspicious config recovery to refuse restoring polluted `.bak` files, with unit and integration tests.

## Best Possible Solution

Keep the direct `.bak` restore guard, but make redacted-placeholder detection path/schema-aware, accept actual `maskApiKey` shapes, cover async and sync restore, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed `openclaw.json.bak` with a redacted sensitive value and make the live config an update-channel-only clobber; current main reaches `copyFile` without a backup pollution check. The PR's own test covers a gateway token case, but additional cases such as request TLS passphrase/key and masked punctuation remain reproducible gaps.

## Solution Assessment

No as-is. The recovery-path guard is the right narrow place to fix the outage, but the detector should align with existing sensitive config schema/path semantics instead of relying only on a small leaf-name suffix list and fixed punctuation regex.
