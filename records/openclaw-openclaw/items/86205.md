## Summary

Keep this PR open: the bug-fix direction is useful and current main does not already preserve optimistic TUI user messages through history reloads, but the patch still clears pending user entries too early and can reproduce the same visible message-loss class during a later stale history or error refresh.

## What This Changes

The PR changes TUI send/history handling to store outbound user messages as pending chat-log entries, preserve and reconcile them during history rebuilds, and adds focused regression coverage plus a changelog entry.

## Best Possible Solution

Keep pending user entries until persisted history reconciliation removes the matching message, refresh the branch against current main, and leave release-note context in the PR body instead of editing `CHANGELOG.md`.

## Reproduction Assessment

Yes from source inspection: after `commitPendingUsers()` clears the pending map on first run adoption, a later `loadHistory()` can call `clearAll({ preservePendingUsers: true })` with no pending entry left to restore. I did not run tests because this was a read-only review.

## Solution Assessment

No: pending-user storage is the right direction, but clearing all pending entries before persisted history catches up defeats the preservation guarantee. The narrower fix is to keep them pending until `reconcilePendingUsers()` matches persisted history or an explicit send failure removes the failed entry.
