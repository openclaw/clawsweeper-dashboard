## Summary

Keep open: the pending-row approach is the right TUI layer for stale history reloads, but the branch is not merge-ready as a standalone patch because it adds ChatLog pending user rows without the matching abort cleanup covered by the open companion work.

## What This Changes

The PR changes TUI sends to render outbound prompts as pending ChatLog user rows, preserves/reconciles those rows during history reloads, drops the row on send failure, and adds focused TUI regression coverage.

## Best Possible Solution

Land one coherent pending-message lifecycle: pending prompts survive stale history until persisted, are dropped or restored on abort, are removed on send failure, and are cleared on session changes, preferably by folding or sequencing this PR with https://github.com/openclaw/openclaw/pull/86200.

## Reproduction Assessment

Yes, at source level: current main renders optimistic user text with addUser while loadHistory clears and rebuilds the chat log, and the PR's focused tests exercise the stale-history preservation path. I did not run a live Gateway/TUI reproduction in this read-only review.

## Solution Assessment

No, not as a standalone branch: using ChatLog pending rows is the right layer, but the best fix also owns the abort and pending-row cleanup lifecycle so abandoned rows cannot be restored later.
