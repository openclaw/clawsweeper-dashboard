## What This Changes

This PR changes TUI chat-log, command, event, and session state so optimistic user rows are keyed as pending entries, survive history rebuilds, rekey to accepted run IDs, and clean up on failed or early aborted sends, with focused tests.

## Best Possible Solution

Land the keyed pending-row approach after maintainer review, current-main refresh, and CI/proof gates, keeping optimistic display state owned by ChatLog and reconciled only through canonical history or registered run events.

## Reproduction Assessment

Yes at source level: current main renders optimistic sends as unkeyed user rows and loadHistory clears the chat log, while the PR adds focused tests for stale history rebuilds, run-ID reassignment, send failure, and abort cleanup. I did not run a live Gateway/provider reproduction in this read-only review.

## Solution Assessment

Yes, this looks like the right layer for the reported TUI rendering bug: ChatLog owns the pending visual row, event handlers mark run registration, and session history performs reconciliation. A gateway protocol change could also help someday, but it is broader than needed for stabilizing local optimistic rendering.
