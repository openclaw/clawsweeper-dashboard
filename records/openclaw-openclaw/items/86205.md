## Summary

Keep open: this is a valid TUI bug-fix PR and current main still has the vulnerable send/history shape, but the proposed patch should not merge as-is because it clears pending-user state on the first run event before history necessarily contains the user turn. The protected `maintainer` label also means this PR needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR changes the TUI send/history path to store optimistic outbound user messages as pending chat-log entries, preserve and reconcile them across history rebuilds, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep optimistic TUI user entries as recoverable pending chat-log state until `chat.history` reconciliation confirms the persisted user turn, while still dropping them on send failure or session switch.

## Reproduction Assessment

Yes, source inspection shows the current-main path: outbound messages are added as ordinary user entries and `loadHistory()` clears the log. I did not run a live TUI/Gateway repro in this read-only review.

## Solution Assessment

No, the direction is right but the event-side `commitPendingUsers()` timing is too early. The safer fix is to keep pending entries restorable until history reconciliation proves the user turn has persisted.
