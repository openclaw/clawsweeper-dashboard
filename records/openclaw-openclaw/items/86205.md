The PR changes TUI chat-log and state handlers so submitted prompts become pending user rows that survive history rebuilds, rekey across accepted run IDs, reconcile with persisted history, and clean up on failed or early-aborted sends, with focused TUI coverage.

## Best Possible Solution

Land a coordinated TUI-state fix that preserves pending prompt rows across stale history reloads and abort/send-error paths after TUI owners confirm merge order and focused TUI validation.

## Solution Assessment

Yes, this is the right layer for the issue: the Gateway already keys chat sends by the client idempotency run ID, while the failure is TUI-local preservation/reconciliation during history reloads. A protocol change is not required for this fix, though merge order with adjacent TUI state work still needs maintainer care.
