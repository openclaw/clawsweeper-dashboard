## Summary

Keep this protected maintainer draft open: the stale-history preservation idea is useful, but the current branch still leaves a pending ChatLog user row alive when a queued send is aborted, so it is not merge-ready.

## What This Changes

The PR changes TUI optimistic outbound messages to pending chat-log user rows, preserves/reconciles them across history reloads, and updates focused TUI regression coverage.

## Best Possible Solution

Land the pending-row preservation only after the same lifecycle removes or restores the pending user row consistently for send failure, abort, session switch, and history reconciliation.

## Reproduction Assessment

Yes, by source inspection: after the PR adds a pending user row at send time, abortActive clears only state flags, and the aborted/history path can reload history while preserving pending rows. I did not run an interactive TUI reproduction in this read-only review.

## Solution Assessment

No, not yet: preserving pending rows is the right local layer for stale history rebuilds, but the branch is incomplete until abort cleanup participates in the same pending-row lifecycle.
