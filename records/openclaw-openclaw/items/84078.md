## Summary

Keep this open: the current source still has the Slack capability gap and required durable-send failure path, but the exact `message_tool_only` runtime failure needs maintainer choice and live Slack proof before selecting a fix.

## What This Changes

The issue reports that Slack visible source replies routed through `message_tool_only` can fail when `message(action="send")` uses required durable delivery without Slack `reconcileUnknownSend` support.

## Best Possible Solution

Add Slack unknown-send reconciliation if feasible; otherwise make Slack `message_tool_only` source replies unambiguously use best-effort/tool guidance while preserving explicit required durable failures for callers that truly request them.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection shows required Slack message sends fail without `reconcileUnknownSend`, but current main defaults ordinary message-tool sends to best-effort unless `bestEffort:false` is present, so the exact Slack session/tool args still need confirmation.

## Solution Assessment

Unclear. The maintainable fix depends on whether maintainers prefer Slack reconciliation, a targeted source-reply best-effort downgrade, or stronger tool/prompt guidance for adapters that cannot reconcile unknown sends.
