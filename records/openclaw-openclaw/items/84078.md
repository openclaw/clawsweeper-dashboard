## Summary

Keep open: current main still has the reported Slack capability gap. The added local logs make the source-level path stronger: Slack `message_tool_only` source replies can reach `message(action="send")` with `bestEffort:false`, and required sends still reject Slack because the adapter does not declare `reconcileUnknownSend`.

## What This Changes

Ensure Slack `message_tool_only` source replies do not fail durable-send preflight when Slack lacks `reconcileUnknownSend`.

## Best Possible Solution

Add Slack unknown-send reconciliation if feasible; otherwise handle the Slack source-reply capability mismatch at the source-reply/tool guidance boundary while preserving explicit required durable failures for callers that truly request them.

## Reproduction Assessment

Yes at source level, but not as a live Slack run in this review. The current code path maps Slack group/channel `message_tool` replies to `message_tool_only`, preserves explicit `bestEffort:false`, turns that into required durability, and then rejects Slack because it lacks `reconcileUnknownSend`; the reporter's added logs match that path.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix needs maintainer choice between Slack reconciliation, a scoped best-effort downgrade for source replies, or config/tool guidance that prevents impossible required sends.
