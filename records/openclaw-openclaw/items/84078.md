## Summary

Keep this open: current main still has the Slack capability gap and the required durable-send rejection path, and the reporter added matching redacted Slack logs with `bestEffort: false`. The fix choice still needs maintainer product judgment and live Slack proof because ordinary message-tool sends default to best-effort unless the tool call explicitly opts into required durability.

## What This Changes

The issue reports that Slack `message_tool_only` source replies can fail when `message(action="send")` is invoked with required durable semantics because Slack lacks `reconcileUnknownSend`.

## Best Possible Solution

Add Slack unknown-send reconciliation if feasible; otherwise make Slack `message_tool_only` source replies negotiate an explicit best-effort/tool-guidance path when the active adapter cannot reconcile unknown sends, while preserving required durable failures for callers that deliberately request them.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection supports the reported failure for Slack required durable sends without `reconcileUnknownSend`, and the reporter added matching redacted logs with `bestEffort: false`.

## Solution Assessment

Unclear. The maintainable fix depends on whether maintainers want Slack reconciliation, a targeted best-effort downgrade for source replies, config validation, or stronger tool/prompt guidance for adapters that cannot reconcile unknown sends.
