## Label Justifications

- `P2`: This is a normal-priority Slack delivery reliability bug with user-visible missing replies, limited to Slack source replies configured through the message-tool path and required durability.
- `impact:message-loss`: The reported failure prevents visible Slack assistant replies from being posted when the source reply is routed through the message tool with required durable semantics.

## Summary

Keep open: current main still lets Slack group/channel `message_tool` source replies reach the required durable-send path when the tool call includes `bestEffort:false`, and Slack still does not expose `reconcileUnknownSend`; the reporter added redacted live logs showing that exact argument shape, so this needs maintainer contract choice rather than cleanup close.

## What This Changes

The issue reports Slack group/channel visible source replies configured for `message_tool` failing when `message(action="send")` is run as a required durable send against a Slack adapter without `reconcileUnknownSend`.

## Best Possible Solution

Add Slack unknown-send reconciliation if feasible; otherwise make Slack `message_tool_only` source replies unambiguously avoid impossible required durability while preserving explicit required durable failures for callers that truly request them.

## Reproduction Assessment

Yes, source-reproducible with high confidence: `messages.groupChat.visibleReplies: "message_tool"` can select `message_tool_only`, Slack lacks `reconcileUnknownSend`, and `bestEffort:false` drives the message tool into the required durable preflight that throws the reported error. I did not perform live Slack Socket Mode reproduction in this read-only review.

## Solution Assessment

Unclear. The report identifies the failing contract, but the best fix depends on maintainer choice between Slack reconciliation, a narrowly scoped source-reply best-effort downgrade, and/or stronger prompt/config guidance for adapters that cannot reconcile unknown sends.
