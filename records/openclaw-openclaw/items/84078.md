## Summary

Keep this open: current main still has the Slack capability gap and the required durable-send rejection path, and the reporter's follow-up narrows the live failure to `message(action="send")` calls with `bestEffort:false`. The remaining decision is which contract to own: Slack reconciliation, a targeted best-effort path for tool-only source replies, or clearer tool/config negotiation.

## What This Changes

The issue asks OpenClaw to prevent Slack `message_tool_only` visible source replies from failing when message-tool sends require durable delivery but Slack does not support `reconcileUnknownSend`.

## Best Possible Solution

Add Slack unknown-send reconciliation if maintainers want required durable Slack sends; otherwise scope a source-reply-specific best-effort/tool-guidance or validation path that preserves explicit required durable failures.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection plus the reporter's added `bestEffort:false` logs give a high-confidence source reproduction path for the durable preflight failure on Slack.

## Solution Assessment

Unclear. Slack reconciliation is the safest durable-delivery fix, while a narrower source-reply best-effort/tool-guidance change could preserve visible replies but needs maintainer agreement on retry-safety semantics.
