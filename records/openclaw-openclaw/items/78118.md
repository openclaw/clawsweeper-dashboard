## Summary

Current main and v2026.5.4 already prevent the reported duplicate Discord bot-token runtime: duplicate enabled accounts resolve to one active account, duplicate accounts are surfaced as disabled, and gateway startup does not launch the duplicate client.

## What This Changes

The issue asks OpenClaw to prevent duplicate enabled Discord accounts with the same bot token/application from creating a second native-command runtime that can time out slash commands.

## Best Possible Solution

Keep the shipped duplicate-token account guard and open a narrower follow-up only if maintainers want generic unknown-command ACK hardening beyond this duplicate-account failure.

## Reproduction Assessment

No. Source inspection on current main shows duplicate Discord accounts with the same resolved bot token are filtered before gateway startup, so the reported two-client native-command timeout path is no longer available through normal config startup.

## Solution Assessment

Yes. Filtering duplicate bot-token accounts at the Discord plugin account boundary is the narrowest maintainable fix for this report because it prevents the second runtime client while preserving per-account native-command settings for distinct bot tokens.
