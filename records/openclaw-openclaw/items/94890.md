## Summary

Keep this PR open, but it is not merge-ready: the current branch still changes the shipped abort default and treats ownerless model-call markers as long-running before the related cleanup lands, while the supplied proof is still unit-test output rather than a real stalled CLI/provider recovery trace.

## What This Changes

The PR adds `hasActiveModelCall` to diagnostic activity snapshots, uses it for model-call classification and active-abort recovery, lowers the default stalled-run abort window, and updates focused diagnostics tests.

## Best Possible Solution

Keep the non-embedded model-call recovery idea, but preserve the shipped abort default unless maintainers approve a policy change, require live ownership or ownerless-marker cleanup before changing classification, and keep replay deduplication tracked on the canonical issue.

## Reproduction Assessment

Yes for source-level review: current main has a focused diagnostic test where a stale model call without embedded-run ownership emits a stalled diagnostic and does not recover. No live Xiaomi/Codex/provider stalled-call recovery proof is supplied.

## Solution Assessment

No as-is. Adding a model-call activity snapshot field is a plausible part of the fix, but using it without an owner or cleanup boundary and lowering the shipped default is not the narrowest safe solution.
