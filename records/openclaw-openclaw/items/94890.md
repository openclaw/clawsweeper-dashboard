## Summary

Keep this PR open, but it is not merge-ready: it is a useful partial fix for stuck model-call recovery, yet the patch changes shipped abort defaults, changes ownerless model-call classification, and still lacks real CLI/provider recovery proof.

## What This Changes

The PR adds `hasActiveModelCall` to diagnostic activity snapshots, uses it for model-call classification and recovery, lowers the default stalled abort fallback, and updates diagnostics tests plus a tooling test-plan entry.

## Best Possible Solution

Preserve the shipped abort default unless maintainers approve a shorter policy, keep ownerless stale activity from being hidden, add real redacted CLI/provider recovery proof, and keep restart replay/deduplication tracked separately if this PR remains partial.

## Reproduction Assessment

Yes for source-level review: current main has a focused diagnostic test where a stale model call without embedded-run ownership emits `session.stalled` and does not recover. No live Xiaomi/Codex/provider stalled-call recovery proof is supplied.

## Solution Assessment

No as-is. Adding a model-call activity snapshot field is plausible, but changing the shipped abort default and treating ownerless model markers as long-running needs maintainer approval or a safer ownership boundary.
