## Summary

Keep this PR open, but not merge-ready: the current head removed the earlier abort-threshold default change, yet it still uses a non-owner-specific model-call flag for both classification and active-abort recovery, and the contributor proof remains unit-test-only rather than real CLI/provider recovery output.

## What This Changes

Adds `hasActiveModelCall` to diagnostic session activity snapshots and uses it to classify and recover stalled model calls in sessions without active embedded-run ownership, with focused diagnostics tests.

## Best Possible Solution

Expose an ownership-aware model-call snapshot signal, or land the ownerless-marker cleanup first, so non-embedded active model calls can recover without treating stale ownerless markers as live work; then add redacted real runtime proof.

## Reproduction Assessment

Yes for source-level review: current main has a focused diagnostic test where a stale model call without embedded-run ownership emits `session.stalled` and does not call recovery. I did not run a live Xiaomi/Codex/provider stalled-call repro in this read-only review.

## Solution Assessment

No as-is. Adding a model-call activity snapshot signal is a plausible part of the fix, but the signal needs current-owner semantics or ownerless-marker cleanup before it can safely drive classification and active abort.
