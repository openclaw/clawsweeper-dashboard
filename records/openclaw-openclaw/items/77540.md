## Summary

Keep open: the cache direction is useful and current main does not already implement it, but the PR head can still let an older in-flight node pairing read repopulate the cache after mutation invalidation, and the changelog omits the new node/cron endpoint performance behavior.

## What This Changes

The PR adds short-lived caching and in-flight coalescing for `sessions.list`, manifest model-id normalization, `node.list`/`node.describe`, and cron delivery previews used by `cron.list`.

## Best Possible Solution

Keep the scoped performance caches, add stale in-flight write suppression plus regression coverage for node pairing cache fills after invalidation, and document the node and cron cache behavior in the changelog.

## Reproduction Assessment

Yes. Source-reproducible: start `node.list` or `node.describe` while `loadNodePairingListsCached()` is in flight, mutate pairing state so `clearNodePairingListCache()` runs, then let the older read resolve and store a pre-mutation snapshot for later list/describe calls until the 1-second TTL expires.

## Solution Assessment

No. The short-TTL cache approach is appropriately narrow, but the node pairing cache needs the same generation guard already used by the sessions cache before this is the maintainable fix.
