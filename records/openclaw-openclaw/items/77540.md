## Summary

Keep open: the PR addresses real gateway hot paths, but the node pairing cache can still be repopulated by an older in-flight read after mutation invalidation, and the changelog omits the added node and cron endpoint performance behavior.

## What This Changes

The PR adds short-lived caching and in-flight coalescing for `sessions.list`, `node.list`/`node.describe`, `cron.list` delivery previews, and manifest model-id normalization.

## Best Possible Solution

Keep the performance caches, add a generation guard and regression coverage for node pairing cache fills after invalidation, and document the node/cron endpoint behavior before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows an in-flight `Promise.all` can start, a node pairing mutation can call `clearNodePairingListCache()`, and the older promise can later write `nodePairingListCache` for the same list/describe path.

## Solution Assessment

No. The cache strategy is a narrow perf mitigation, but node pairing needs the same generation/token guard already used by the sessions-list cache plus a changelog update for node and cron behavior.
