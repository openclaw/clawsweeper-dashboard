## Summary

Keep open: the force-push addresses the earlier `sessions.list` cache invalidation blockers, but the newly added node pairing cache can still repopulate stale data after a pairing mutation resolves during an older in-flight list read.

## What This Changes

The PR adds short-lived caches for gateway `sessions.list`, manifest model-id normalization policy lookup, and `node.list`/`node.describe` pairing reads, plus invalidation hooks, tests, and changelog entries.

## Best Possible Solution

Keep the performance caches, but add a generation/token guard to the node pairing cache and regression coverage for an in-flight list resolving after a pairing mutation before merge.

## Reproduction Assessment

Yes by source inspection: start `node.list` or `node.describe` while the pairing-list Promise is in flight, mutate pairing state, then let the older read resolve. The older result can store at line 113 and be reused until the 1-second TTL expires.

## Solution Assessment

No: the sessions cache now uses the right generation guard, but the new node pairing cache needs the same stale in-flight write suppression. That is a narrower fix than removing the cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Prevent stale node pairing reads from refilling the cache:** `src/gateway/server-methods/nodes.ts:113`
  - body: `clearNodePairingListCache()` drops the current pointer, but an older `Promise.all([listDevicePairing(), listNodePairing()])` can still resolve afterward and assign `nodePairingListCache`. After approve/reject/remove/request/rename, a later `node.list` or `node.describe` can therefore reuse a pre-mutation pairing snapshot for up to the TTL; add a generation/token check before storing resolved entries.
