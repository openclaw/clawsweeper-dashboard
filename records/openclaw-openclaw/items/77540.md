## Summary

Keep open. The follow-up commits appear to address the earlier `sessions.list` cache invalidation findings, but the latest branch adds a node-pairing list cache that can still return stale `node.list`/`node.describe` pairing state after device/node pairing writes.

## What This Changes

The PR adds short-lived caches for gateway `sessions.list`, node list/describe pairing data, and manifest model-id normalization policy lookups, with tests and changelog entries.

## Best Possible Solution

Keep the performance caches only after node/device-pairing invalidation uses a shared generation-guarded boundary and stale in-flight node-list reads cannot repopulate cache entries.

## Reproduction Assessment

Yes, by source inspection: start a slow `node.list`, mutate paired device or node state, then request `node.list`/`node.describe` within the cache TTL. The pre-mutation pairing snapshot can be reused or written back after invalidation.

## Solution Assessment

No. Caching the read-heavy lists is reasonable, but this patch needs the same generation-checked invalidation pattern for the node cache and must clear it from all device/node pairing mutation paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Invalidate node cache for device-pairing writes:** `src/gateway/server-methods/nodes.ts:106-113`
  - body: The cached snapshot includes `listDevicePairing()`, but `clearNodePairingListCache()` is private to `nodes.ts` and is only called from node-pair handlers. Device approvals/removals in `server-methods/devices.ts` and the websocket bootstrap path mutate the same paired-device state that `node.list` uses, so repeated `node.list`/`node.describe` calls can show stale paired device rows until the TTL expires.
  - confidence: 0.9
- **[P2] Version node cache writes after invalidation:** `src/gateway/server-methods/nodes.ts:113`
  - body: Clearing the cache during a pairing mutation drops the in-flight promise, but the older promise can still run its `.then()` and assign `nodePairingListCache = entry` after the mutation. Use a generation/token check like the sessions cache so pre-invalidation reads are returned to their caller but not stored for later requests.
  - confidence: 0.87
