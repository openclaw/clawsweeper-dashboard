## Summary

Keep this PR open: the optimization is useful, but the proposed sessions.list cache has two concrete freshness bugs that can serve stale rows after real session updates.

## What This Changes

The PR adds short-lived caching for gateway `sessions.list` results and manifest model-id normalization policy lookups, plus changelog entries.

## Best Possible Solution

Keep the performance cache only after invalidation is shared across all session mutation/message/lifecycle paths and in-flight results cannot repopulate stale entries after a clear.

## Reproduction Assessment

Yes by source inspection: current-main session/message/lifecycle broadcasters outside `sessions.ts` bypass the new cache clear, and an in-flight list can write after a clear. A focused test can reproduce stale `sessions.list` rows by listing, mutating through an external event path, then listing again within the TTL.

## Solution Assessment

No: caching `sessions.list` is a reasonable narrow optimization, but this implementation needs a shared/versioned invalidation boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Invalidate cache from every session-change source:** `src/gateway/server-methods/sessions.ts:325`
  - body: The cache is cleared only from this local helper, but current main also emits `sessions.changed` directly from `server-chat.ts` and `server-session-events.ts` for lifecycle/message updates. Those paths can change transcript previews and row metadata, so a later `sessions.list` with the same params can reuse stale rows for up to 60 seconds.
  - confidence: 0.91
- **[P2] Prevent stale in-flight results from repopulating the cache:** `src/gateway/server-methods/sessions.ts:206`
  - body: If a slow `sessions.list` is in flight when a session mutation clears the cache, this code still writes the pre-mutation result after the await completes. Add an invalidation generation/token check so results started before a clear are returned to their caller but not stored for later requests.
  - confidence: 0.88
