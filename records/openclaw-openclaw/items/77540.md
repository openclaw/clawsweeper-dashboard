## Summary

Keep open: the performance direction is useful, but the PR still lets stale in-flight node pairing reads refill the cache after pairing mutations, and the changelog omits the node and cron endpoint performance changes.

## What This Changes

The PR adds short-lived caches and in-flight coalescing for sessions.list, node.list/node.describe pairing reads, cron.list delivery previews, and manifest model-id normalization, with invalidation hooks, tests, and changelog entries.

## Best Possible Solution

Keep the performance caches, add a generation/token guard and regression coverage for node pairing cache fills after mutation invalidation, and document the node and cron endpoint cache behavior in the changelog.

## Reproduction Assessment

Yes. Source-reproducible: start node.list or node.describe while loadNodePairingListsCached is in flight, mutate pairing state, then let the older read resolve and store a pre-mutation nodePairingListCache entry that later list/describe calls can reuse until the 1-second TTL expires.

## Solution Assessment

No. The PR uses the right generation-guard pattern for sessions, but the node pairing cache needs the same stale in-flight write suppression; that is narrower and safer than removing the cache.
