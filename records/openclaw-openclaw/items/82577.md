## Summary

Keep this PR open: the core behavior is a plausible fit for OpenClaw's plugin model, and the Max.ru channel-plugin context is useful, but the patch currently has blocking test/scoping issues and no after-fix real behavior proof.

## What This Changes

The PR adds a channel metadata listing helper for bundled and installed channel plugins, then switches `openclaw configure` channel removal and setup discovery to use it.

## Best Possible Solution

Implement the core metadata support through the existing scoped catalog helpers, with focused tests for non-bundled channel labels in configure removal and setup discovery.

## Reproduction Assessment

Yes, source-reproducible: current main's removal wizard only labels ids returned by `listChatChannels()`, so a configured non-bundled channel falls back to its raw id. I did not run the interactive command because this review is read-only.

## Solution Assessment

No, not as written: the direction is reasonable, but the implementation should reuse or parameterize the resolved channel catalog so workspace/env scope and origin priority are preserved, and the existing mocks/tests need updating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Update channel metadata mocks for the new export:** `src/commands/configure.channels.ts:1`
  - body: Both configure and setup-discovery tests mock `channels/chat-meta.js` with only `listChatChannels`; after these callers import `listAllChatChannels`, Vitest will fail during module import before exercising the behavior. Export `listAllChatChannels` from those mocks and add coverage for the non-bundled label path when switching the callers.
  - confidence: 0.9
- **[P2] Keep setup metadata scoped to the requested workspace:** `src/commands/channel-setup/discovery.ts:121`
  - body: `resolveChannelSetupEntries` already carries `workspaceDir` and `env` into catalog discovery so scoped workspace/config entries can beat global entries. Calling a parameterless `listAllChatChannels()` here can seed `metaById` from the default environment first, then prevent the correctly scoped catalog entry with the same id from replacing it. Pass the scope through or build the baseline from the already-resolved catalog entries.
  - confidence: 0.86
