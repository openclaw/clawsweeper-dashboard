## Summary

Keep open. Current main still routes every Control UI textarea input through slash-menu updates and a Lit `@state()` draft assignment, so the reported per-keystroke render path remains present. Related PR #62790 is open and unmerged, so this should stay open for maintainer review rather than being closed as implemented.

## What This Changes

The issue reports 500ms-1s character display delays while typing in the Control UI/WebChat composer on macOS Safari or Chrome and asks for immediate keystroke responsiveness.

## Best Possible Solution

Resolve this through #62790 or an equivalent narrow Control UI fix that keeps ordinary draft typing out of full Lit host re-renders, avoids unnecessary host updates for non-slash input, preserves explicit draft synchronization paths, and adds regression coverage showing draft edits do not rebuild the heavy chat thread.

## Reproduction Assessment

Yes for the underlying code path: open the Control UI chat composer and type; current source shows every input event calls the draft callback, which assigns the `@state()` `chatMessage`. I did not run a live timing profile, so the exact delay magnitude is not independently measured here.

## Solution Assessment

Yes, the narrow maintainable fix is to remove ordinary draft typing from the reactive host render path while keeping deliberate draft updates for send, clear, abort, history, and slash-command behavior. Reviewing and landing #62790 or a same-scope replacement is the best path.
