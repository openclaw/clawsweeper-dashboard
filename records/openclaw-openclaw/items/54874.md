## Summary

Keep open. Current main and v2026.5.19 still route ordinary WebChat input through Lit update paths, and the focused implementation PR is still open, conflicting, and not merged.

## What This Changes

The issue reports 500ms-1s delay on each keystroke in the OpenClaw WebChat input on macOS browsers.

## Best Possible Solution

Land or replace the focused WebChat fix so ordinary typing no longer updates expensive host/chat state while preserving slash-menu behavior, history recall, failed-send restore, and detached-send draft sync with tests and browser proof.

## Reproduction Assessment

Yes, at source level. Current main wires each textarea input to `updateSlashMenu` and `onDraftChange`, and `onDraftChange` assigns reactive `chatMessage`; a live browser trace would still be needed to measure the exact 500ms-1s delay.

## Solution Assessment

Yes, keeping this issue open is the best current outcome. The maintainable path is to repair, land, or replace the focused WebChat PR rather than closing the issue before the fix reaches main.
