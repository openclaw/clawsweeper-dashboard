## Summary

Keep open. Current main still has the WebChat per-keystroke reactive render path, and the related implementation PR at https://github.com/openclaw/openclaw/pull/62790 remains open and unmerged with prior review feedback to address.

## What This Changes

Typing in the Control UI webchat textarea is reported to lag 500ms-1s per keystroke because draft typing still drives reactive host state updates.

## Best Possible Solution

Land or replace the existing focused Control UI fix so ordinary draft typing does not update Lit state per keystroke while programmatic draft sync paths remain explicit and tested.

## Reproduction Assessment

Yes, at source level. Current main still wires the textarea input event to `onDraftChange`, which assigns `chatMessage` while it is a Lit `@state()`, matching the reported per-keystroke host render path; I did not measure the exact browser delay.

## Solution Assessment

Yes for the overall direction of making the draft non-reactive, but not as a separate issue repair job. The best path is to review, rebase, land, or replace https://github.com/openclaw/openclaw/pull/62790 with current-main coverage for programmatic draft sync.
