## Summary

Keep open. Current main still gates `memory-lancedb` auto-capture through plugin-local hard-coded regexes, has no custom trigger config, and rejects short CJK phrases such as the reporter's examples before regex matching because of the `text.length < 10` guard.

## What This Changes

The issue requests broader i18n support or configurable trigger patterns for `memory-lancedb` automatic memory capture so CJK memory-worthy phrases are not skipped.

## Best Possible Solution

Implement this inside the bundled `memory-lancedb` plugin: add a safe custom trigger config surface, broaden tested default CJK coverage for the reported phrases, make the short-message eligibility check CJK-aware, update manifest/UI hints/docs/changelog, and leave core memory APIs unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: short CJK examples such as `以后都用这个` are rejected by `shouldCapture` at the `text.length < 10` guard, and longer phrases are limited to the fixed trigger regex list.

## Solution Assessment

Yes. A plugin-owned config/default-trigger update is the narrowest maintainable fix; LLM-based intent detection would be broader product work and a new core API is not needed for this report.
