## Label Justifications

- `P1`: `sessions_send` timeout misclassification can break real cross-session agent workflows even though the current main path has partial mitigation.
- `impact:message-loss`: The issue is about a cross-session reply being suppressed from the inline `sessions_send` result after the target run accepts the message.

## Summary

Keep open. Current main no longer treats every soft wait-window expiry as a hard timeout, but it still does not return a late post-timeout reply inline, and the issue is paired with an open closing PR that needs review/rebase rather than issue closure.

## What This Changes

The issue requests post-timeout compensation so `sessions_send` can distinguish wait-window expiry from hard delivery failure and return a late reply or accepted status when the target run was accepted.

## Best Possible Solution

Review, rebase, and land or replace the linked PR with compensation that returns `ok` only for a real new post-baseline reply while preserving terminal timeout and delayed A2A behavior.

## Reproduction Assessment

Yes at source level: start the target run successfully, make the first `agent.wait` return a non-terminal timeout, then make a new assistant reply appear before a follow-up history read. Current main returns `accepted` and relies on delayed A2A instead of returning that late reply inline.

## Solution Assessment

Yes in direction: post-timeout compensation in the `sessions_send` and run-wait boundary is the narrow maintainable fix. The active PR still needs review against current main so it does not mask terminal timeouts or drop delayed A2A delivery.
