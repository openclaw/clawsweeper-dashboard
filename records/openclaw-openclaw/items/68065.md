## Summary

Keep open. Current main now returns `accepted` for non-terminal wait-window expiry, but it still lacks the requested inline late-reply compensation path, and the same reporter has an open closing PR that should be reviewed, rebased, landed, or superseded.

## What This Changes

The issue asks `sessions_send` to distinguish first `agent.wait` window expiry from delivery failure by returning `accepted` or an inline late reply after post-timeout compensation.

## Best Possible Solution

Review, rebase, and land or replace the linked PR with compensation that returns `ok` only for a real new post-baseline reply, returns `accepted` for provably accepted no-reply timeouts, and preserves terminal timeouts plus delayed A2A delivery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: make the target run start succeed, make the first `agent.wait` return a non-terminal timeout, then add a post-baseline assistant reply before a follow-up history read; current main returns `accepted` with delayed A2A instead of inline `ok`.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: post-timeout compensation at the `sessions_send` and run-wait boundary is the narrow maintainable fix, but the implementation must be reconciled with current terminal-timeout and delayed-A2A behavior.
