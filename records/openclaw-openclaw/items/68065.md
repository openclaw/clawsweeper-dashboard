## Label Justifications

- `P1`: A misleading `sessions_send` timeout can break real cross-session delegation workflows and already has an open fix PR and production evidence.
- `impact:message-loss`: The issue is about callers losing or misclassifying an accepted inter-session reply when the first wait window expires.

## Summary

Keep this issue open: current main and the latest release no longer treat a non-terminal first wait-window expiry as a hard timeout, but the requested inline post-timeout reply compensation is still owned by the open same-author closing PR, which needs maintainer review rather than a separate cleanup close.

## What This Changes

The issue asks for `sessions_send` to distinguish an accepted run whose first `agent.wait` expires from a hard send failure, returning an accepted state or late reply instead of a bare timeout.

## Best Possible Solution

Review, refresh, and either land or deliberately narrow/close the open compensation PR so `sessions_send` has one maintainer-approved timeout contract across inline replies, accepted soft timeouts, and terminal failures.

## Reproduction Assessment

Yes, source-level: current main returns early from the wait helper on timeout and the `sessions_send` timeout branch returns `accepted` rather than reading history to return a late reply inline. I did not run a live timeout-edge scenario in this read-only review.

## Solution Assessment

Unclear until maintainer review of the open PR: current main already fixes the hard-timeout classification for non-terminal waits, while the proposed extra inline compensation may be the better user contract if maintainers accept the compatibility and delivery semantics.
