## Summary

Keep this issue open. Current main still treats the first nonzero-timeout `agent.wait` timeout from `sessions_send` as a hard `status: "timeout"` without post-timeout reply or accepted-delivery compensation, and the same reporter has open implementation PR #68196 with explicit closing syntax for this issue.

## What This Changes

The issue requests that `sessions_send` distinguish wait-window expiry from actual delivery failure by checking for a late assistant reply or proven acceptance before returning `status: "timeout"`.

## Best Possible Solution

Land or supersede the linked implementation path with a narrow fix in the `sessions_send`/run-wait boundary: after a successful `agent` start, handle wait-window expiry by checking for a new assistant reply since the pre-wait baseline, returning `ok` for a late reply, returning `accepted` only with a real acceptance signal, and preserving hard timeout when delivery cannot be established.

## Reproduction Assessment

Yes. A focused gateway mock where `agent` returns a run id and the first `agent.wait` returns `timeout` will hit the current `sessions_send` timeout branch and return `status: "timeout"` without any post-timeout `chat.history` compensation.

## Solution Assessment

Yes, the requested fix boundary is the narrow maintainable one. The open PR direction matches the issue, but maintainers should review the exact acceptance-proof semantics before landing it.
