## Label Justifications

- `P1`: The issue affects cross-session delegation where callers rely on `sessions_send` to distinguish accepted work, replies, and real timeout failures.
- `impact:message-loss`: A target-session reply can miss or be misclassified by the inline `sessions_send` result when it materializes after the first wait window expires.

## Summary

Keep this issue open: current main and v2026.6.10 partially address soft wait-window expiry by returning `accepted`, but they still do not implement the requested inline late-reply compensation, and https://github.com/openclaw/openclaw/pull/68196 remains open with closing syntax.

## What This Changes

The issue asks for `sessions_send` to return `accepted` or a late reply instead of a hard timeout when the first wait window expires after the target run was accepted.

## Best Possible Solution

Refresh or supersede the linked PR with a narrow compensation fix that preserves pending-error and terminal-timeout precedence, then close this issue when that path lands.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level reproduction: current `run-wait` returns before reading history after non-`ok` waits, so a just-after-timeout reply cannot become the inline `ok` result requested here.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final product contract. The fix belongs at the `sessions_send` and `run-wait` boundary, but maintainers need to choose between current delayed-delivery `accepted` semantics and inline late-reply compensation while preserving terminal timeout precedence.
