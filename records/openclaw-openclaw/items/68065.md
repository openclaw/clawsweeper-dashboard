## Label Justifications

- `P1`: The issue affects cross-session delegation where callers rely on `sessions_send` to distinguish accepted work, replies, and real timeout failures.
- `impact:message-loss`: A target-session reply can still miss the inline `sessions_send` result when it materializes after the first wait window expires.

## Summary

Keep this issue open: current main and the latest release no longer treat every non-terminal wait-window expiry as a hard timeout, but they still do not implement the requested post-timeout late-reply compensation, and the same-author linked PR remains open with closing syntax.

## What This Changes

The issue asks for `sessions_send` to distinguish `agent.wait` wait-window expiry from delivery failure and return either a late reply or accepted delivery state after the first timeout.

## Best Possible Solution

Refresh or supersede the linked PR with a narrow compensation fix that preserves pending-error and terminal-timeout precedence, then close this issue when that path lands.

## Reproduction Assessment

Yes, for source-level reproduction: current `run-wait` returns before reading history after non-`ok` waits, so a just-after-timeout reply cannot become the inline `ok` result requested here. I did not run a live timeout-edge scenario in this read-only review.

## Solution Assessment

Unclear as a final product contract. The fix belongs at the `sessions_send` and `run-wait` boundary, but maintainers need to choose between current delayed-delivery `accepted` semantics and inline late-reply compensation while preserving terminal timeout precedence.
