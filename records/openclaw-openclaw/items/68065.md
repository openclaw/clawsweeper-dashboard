## Label Justifications

## Summary

Keep this issue open: current main and the latest release no longer treat every first `agent.wait` timeout as a hard failure because non-terminal wait timeouts return `accepted`, but the requested post-timeout reply compensation is still missing. The linked implementation PR is open and should be reviewed, refreshed, or explicitly superseded before this issue is closed.

## What This Changes

The issue asks for `sessions_send` to compensate after a first `agent.wait` timeout so accepted target runs and late replies are not reported as hard delivery timeouts.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/68196 first: either refresh it as the narrow post-timeout reply compensation fix with regression coverage, or explicitly supersede it if maintainers accept the current `accepted`/delayed-delivery semantics.

## Reproduction Assessment

Yes for source-level reproduction of the remaining gap: a timeout result returns before the helper reads a post-timeout assistant snapshot, so a just-after-timeout reply is not returned inline. The exact original hard-timeout misclassification is only partially current because non-terminal wait timeouts now return `accepted`.

## Solution Assessment

Unclear whether the linked PR's exact inline compensation contract is the final best fix, but the fix belongs at the `sessions_send`/`run-wait` boundary rather than in channel-specific code. Maintainers should reconcile the PR with the shipped soft-timeout behavior before closing this issue.
