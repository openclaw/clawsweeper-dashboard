## Summary

Keep open. Current main no longer treats every non-terminal wait-window expiry as a hard timeout, but it still does not implement the requested inline late-reply compensation, and the same reporter has an open closing PR that needs rebase/review or replacement.

## What This Changes

`sessions_send` should distinguish accepted-but-not-ready waits from hard send failures and return a late inline reply when it appears immediately after the first `agent.wait` timeout.

## Best Possible Solution

Rebase and review the linked PR, or replace it with a narrow compensation patch that preserves terminal timeouts, returns `ok` only for a real new post-baseline reply, and keeps delayed A2A delivery alive when no inline reply is available.

## Reproduction Assessment

Yes at source level: make the target run start succeed, make the first `agent.wait` return a non-terminal timeout, and add a post-baseline assistant reply before a follow-up history read. Current main returns `accepted` and schedules delayed A2A rather than returning that late reply inline.

## Solution Assessment

Unclear for the currently open PR as-is. Post-timeout compensation is the right boundary, but the patch must be rebased and preserve current terminal-timeout and delayed-A2A behavior before this issue is resolved.
