## Summary

Keep open. Current main partially fixes the original hard-timeout symptom by returning `accepted` for non-terminal wait-window expiry, but it still does not implement the requested inline late-reply compensation, and the same reporter has an open closing PR paired with this issue.

## What This Changes

The issue asks for `sessions_send` to return `accepted` for accepted-but-not-ready wait-window timeouts and `ok` when a late reply is discoverable after the first timeout.

## Best Possible Solution

Rebase, review, and land or replace the linked PR with compensation that preserves current terminal-timeout handling and delayed A2A delivery.

## Reproduction Assessment

Yes at source level: make the target run start succeed, make the first `agent.wait` return a non-terminal timeout, and add a post-baseline assistant reply before a follow-up history read. Current main returns `accepted` and schedules delayed A2A instead of returning that late reply inline.

## Solution Assessment

Unclear for the open PR as-is. Post-timeout compensation is the narrow maintainable boundary, but it must be reconciled with current main’s terminal-timeout and delayed-A2A behavior before this issue is resolved.
