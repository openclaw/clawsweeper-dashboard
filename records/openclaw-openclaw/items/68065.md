## Summary

Keep open. Current main still returns a hard `status: "timeout"` from `sessions_send` as soon as the nonzero-timeout `agent.wait` window expires, and the same reporter has open implementation PR #68196 with closing syntax for this issue.

## What This Changes

Change `sessions_send` so a successful agent start followed by a wait-window timeout can return a late reply or accepted-delivery status instead of misclassifying it as a hard send timeout.

## Best Possible Solution

Review, land, or supersede #68196 with a narrow fix at the `sessions_send` and run-wait boundary: after a successful agent RPC start, compensate a wait timeout by checking for a new assistant reply since the baseline, returning `ok` for a late reply, returning `accepted` only with a real acceptance signal, and preserving hard timeout when delivery cannot be established.

## Reproduction Assessment

Yes. A focused mock where `agent` returns a run id and the first `agent.wait` returns `timeout` hits the current `sessions_send` timeout branch and returns `status: "timeout"` before any compensating `chat.history` read.

## Solution Assessment

Yes. The requested boundary is the narrow maintainable solution for this specific misclassification; broader gateway-routing recovery should remain separate unless evidence shows the same root cause.
