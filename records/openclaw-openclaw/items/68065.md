## Label Justifications

- `P1`: The issue affects cross-session delegation where callers rely on `sessions_send` wait results to know whether work was accepted, replied, or actually failed.

## Summary

Keep this issue open: current main and `v2026.6.8` only partially address the original hard-timeout noise by returning `accepted` for non-terminal wait-window expiry, but the requested post-timeout late-reply compensation is still missing. The open linked fix PR at https://github.com/openclaw/openclaw/pull/68196 uses closing syntax and should be reviewed, refreshed, or explicitly superseded before this issue closes.

## What This Changes

The issue asks for `sessions_send` to distinguish wait-window expiry from failed delivery and return either a late reply or an accepted delivery state when the target run was accepted.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/68196 first: refresh it as the narrow post-timeout compensation fix with regression proof, or explicitly supersede it if maintainers accept the current `accepted`/delayed-delivery semantics.

## Reproduction Assessment

Yes for source-level reproduction of the remaining gap: the timeout path returns before a post-timeout assistant snapshot is read, so a just-after-timeout reply cannot become the inline `ok` result requested here. I did not run a live timeout-edge scenario in this read-only review.

## Solution Assessment

Unclear as a final product contract. The fix belongs at the `sessions_send`/`run-wait` boundary, but maintainers should reconcile the linked PR with current terminal-timeout, pending-error, and delayed-delivery behavior before accepting inline compensation.
