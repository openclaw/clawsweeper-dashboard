## Label Justifications

- `P1`: A false hard timeout or missing late reply in `sessions_send` can break active cross-session agent workflows for affected users.
- `impact:message-loss`: The issue is about an accepted inter-session send being reported without the real target reply, which can make callers miss delivered messages.

## Summary

Keep open. Current main and `v2026.6.1` only convert a non-terminal `agent.wait` timeout into `status: "accepted"`; they still do not perform the requested post-timeout reply read, and https://github.com/openclaw/openclaw/pull/68196 remains the open closing PR for that remaining behavior.

## What This Changes

Issue requests that `sessions_send` distinguish accepted-but-not-ready waits from hard delivery failures and return a late reply when one appears after the first wait timeout.

## Best Possible Solution

Land or rework https://github.com/openclaw/openclaw/pull/68196, or an equivalent narrow patch, so post-timeout reply snapshots are checked without treating terminal or pending-error timeouts as successful sends.

## Reproduction Assessment

Yes for the remaining source-level gap: timeout results return before any post-timeout history read, so a late reply cannot be returned inline on current main. I did not execute a live timing race in this read-only review.

## Solution Assessment

Yes, the best current resolution is to keep this issue open behind the active linked PR rather than close it as implemented by the partial `accepted` behavior on main.
