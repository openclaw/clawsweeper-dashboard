## What This Changes

The branch adds a seven-day default TTL to Telegram message-cache plugin-state entries and adds a generic SQLite write-path update that assigns TTLs to existing NULL-expiry rows in the same namespace, with focused tests.

## Best Possible Solution

Keep the Telegram message-cache TTL only if maintainers accept the retention change, and move old NULL-row cleanup to a scoped Telegram migration/backfill or explicit plugin-state contract that preserves `registerIfAbsent` false no-op semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the PR defect: seed a live NULL-expiry row, call `registerIfAbsent` on an already-live key with a TTL-enabled store, and the patch mutates namespace TTLs even though the claim returns false.

## Solution Assessment

No. Adding a Telegram message-cache TTL may be acceptable, but the generic write-path backfill is not the best fix because it changes the keyed-store contract; a scoped Telegram migration/backfill is safer.
