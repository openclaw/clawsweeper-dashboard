## Summary

Keep open for maintainer handling: the released cap fix already covers the original row-limit outage, but this TTL follow-up is not merge-ready because its generic backfill changes plugin-state `registerIfAbsent` semantics and the PR still has only test-based proof.

## What This Changes

The branch adds a seven-day default TTL to Telegram message-cache plugin-state rows and updates the generic SQLite plugin-state store to assign a TTL to existing NULL-expiry rows in a TTL-enabled namespace.

## Best Possible Solution

Keep the Telegram message-cache TTL only if maintainers accept the retention change, but move existing NULL-row cleanup to a scoped Telegram migration/backfill path or another explicit contract that preserves generic keyed-store semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the PR defect: seed a live NULL-expiry row, call registerIfAbsent on a live key with a TTL-enabled store, and the patch mutates namespace TTLs even though the claim returns false.

## Solution Assessment

No. Adding a Telegram message-cache TTL may be acceptable, but the generic write-path backfill is not the best fix because it changes the public keyed-store contract; a scoped Telegram migration/backfill is safer.

- **[P1] Preserve registerIfAbsent no-op TTL semantics:** `src/plugin-state/plugin-state-store.sqlite.ts:525-531`
