## Summary

Keep open for maintainer review: the Telegram TTL direction is plausible, but the new core plugin-state normalization changes per-entry TTL semantics across all namespaces and still lacks real upgrade proof.

## What This Changes

The PR adds a seven-day default TTL for Telegram message-cache plugin-state rows and updates core plugin-state writes so NULL-expiry rows in a TTL-written namespace receive an expiry.

## Best Possible Solution

Keep the Telegram cache lifetime bounded, but scope NULL-expiry normalization to an explicit Telegram/default-TTL migration path or opt-in core API and prove the upgrade behavior on a real plugin-state SQLite path before merge.

## Reproduction Assessment

Yes, at source level: current main opens `telegram.message-cache` without `defaultTtlMs`, and no-TTL plugin-state writes store `expires_at = NULL`. I did not execute a live Telegram or production SQLite reproduction in this read-only review.

## Solution Assessment

No, not as currently shaped: adding a Telegram default TTL uses the existing contract, but the generic core migration rewrites permanent rows for any TTL write. A narrower Telegram/default-TTL migration or explicit opt-in core API would be safer.
