## Summary

Keep open. The useful Telegram message-cache TTL follow-up is not on current main, but this draft branch is currently conflicting, lacks real behavior proof, and still introduces a generic plugin-state contract regression by mutating namespace TTLs from a false `registerIfAbsent` claim.

## What This Changes

The PR adds a seven-day TTL for Telegram message-cache plugin-state rows and updates SQLite plugin-state writes to assign TTLs to existing no-expiry rows in the same namespace.

## Best Possible Solution

Land a scoped Telegram retention fix only after maintainers accept the seven-day context lifetime, preserve keyed-store no-op semantics, handle old NULL rows through a Telegram-specific migration/backfill or explicit SDK contract, and add real upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows the PR defect: with a TTL-enabled store, `registerIfAbsent` can update existing NULL-expiry rows before returning false for a duplicate claim.

## Solution Assessment

No. The Telegram TTL may be acceptable, but the generic SQLite backfill changes the public keyed-store no-op contract; a scoped Telegram migration/backfill or explicit SDK contract change is safer.
