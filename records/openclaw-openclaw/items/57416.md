## Summary

Keep open for maintainer handling. The patch should not merge as-is because it weakens the current and shipped workspace dotenv trust boundary: current main blocks `OPENCLAW_CONFIG_PATH` both explicitly and through the fail-closed `OPENCLAW_` prefix rule, and the docs/changelog now describe that as intentional security behavior.

## What This Changes

The PR removes `OPENCLAW_CONFIG_PATH` from the workspace dotenv blocklist, precomputes the global dotenv path before workspace loading, and updates dotenv tests to allow project-local config while keeping `OPENCLAW_STATE_DIR` blocked.

## Best Possible Solution

Keep the shipped fail-closed workspace dotenv policy unless maintainers explicitly approve a safer project-local config selection design. If the workflow is still desired, it should be a documented trusted opt-in with tests proving it cannot redirect global `.env`, state directories, credentials, plugin roots, or other runtime controls.

## Reproduction Assessment

Yes. Current-main tests provide a high-confidence reproduction path: place `OPENCLAW_CONFIG_PATH=./evil-config.json` in a workspace `.env`, call `loadDotEnv`, `loadWorkspaceDotEnvFile`, or `loadCliDotEnv`, and assert the key remains unset. I did not run tests because this review was read-only.

## Solution Assessment

No. Removing `OPENCLAW_CONFIG_PATH` from the workspace dotenv blocklist is not the best fix because it conflicts with the later shipped fail-closed `OPENCLAW_*` security policy; shell/global env or a new trusted opt-in is safer.
