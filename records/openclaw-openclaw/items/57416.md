## Summary

Keep open. The PR addresses a real linked regression report, but its current patch conflicts with current main's documented and tested fail-closed workspace dotenv policy by allowing a workspace `.env` to set an `OPENCLAW_*` runtime-control key.

## What This Changes

The PR removes `OPENCLAW_CONFIG_PATH` from the workspace dotenv blocklist, precomputes the global dotenv path before workspace loading, and updates dotenv tests to expect project-local config while keeping `OPENCLAW_STATE_DIR` blocked.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a documented trusted opt-in for project-local config selection, with tests across workspace, CLI, gateway, and global dotenv loading.

## Reproduction Assessment

Yes. Current-main tests provide a high-confidence path: place `OPENCLAW_CONFIG_PATH` in a workspace `.env`, call `loadDotEnv` or `loadWorkspaceDotEnvFile`, and assert the key remains unset; I did not execute tests because this review was read-only.

## Solution Assessment

No. Removing `OPENCLAW_CONFIG_PATH` from the workspace blocklist is not the safest fix because current main reserves the full `OPENCLAW_*` namespace for trusted shell/global config; shell/global env or a new explicit trusted opt-in is the safer direction.
