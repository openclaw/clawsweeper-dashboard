## Summary

Keep open for maintainer/security judgment: the PR is a focused attempt to fix the linked project-local dotenv complaint, but it relaxes a documented workspace dotenv trust boundary that current main and the later merged hardening PR intentionally keep fail-closed.

## What This Changes

The PR removes the workspace-dotenv block on `OPENCLAW_CONFIG_PATH`, keeps `OPENCLAW_STATE_DIR` blocked, precomputes the global dotenv path earlier, and updates dotenv tests to expect project-local config loading.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy unless maintainers approve a documented trusted opt-in; if approved, revise the implementation with matching security docs and CLI/gateway regression coverage.

## Reproduction Assessment

Yes, by source inspection: put `OPENCLAW_CONFIG_PATH` in a workspace `.env`, then `loadWorkspaceDotEnvFile` filters it before config resolution on current main. I did not execute tests because this review is read-only.

## Solution Assessment

No. Removing the block for one runtime-control key is not the safest merge path while current code, tests, docs, and the later hardening PR reserve `OPENCLAW_*` for trusted shell/global config.
