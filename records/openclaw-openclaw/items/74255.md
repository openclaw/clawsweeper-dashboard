## Summary

Keep open. Current main still has the reported host-side precheck bug and this PR is the active fix path, but the head is not merge-ready because the opt-in preassembly path can lose the raw history when an engine mutates its input, and the public SDK baseline, docs, and changelog follow-through are still missing.

## What This Changes

The PR adds optional `AssembleResult.promptAuthority`, makes embedded-runner overflow prechecks use assembled messages by default with an explicit preassembly opt-in, and adds runner/precheck regressions.

## Best Possible Solution

Land this PR after preserving true preassembly history for the opt-in path, regenerating the SDK API baseline, documenting the new prompt-authority contract, adding the required changelog entry, and waiting for exact-head checks; then the linked host-side issue can close while LCM-side cache/provenance work stays in its own repository.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: assemble a small context-engine prompt from a much larger history, then the precheck can still use the larger preassembly snapshot and force compaction. The PR adds tests for the intended default and opt-in behaviors, but it still needs the mutating-engine opt-in case covered or made impossible by contract.

## Solution Assessment

Yes directionally, but this head is not the best final implementation yet. Defaulting to assembled prompt authority with an explicit preassembly opt-in is the narrow maintainable fix, but the opt-in implementation must snapshot before assembly or otherwise enforce immutability, and public SDK/docs/changelog artifacts must be aligned.
