## Summary

Keep open. This PR is aimed at the right host-side bug, but the latest reviewed head is not merge-ready because the opt-in preassembly overflow mode can lose the true pre-assembly snapshot, and the public SDK/docs/changelog follow-through is still missing.

## What This Changes

The PR adds `AssembleResult.promptAuthority`, defaults embedded-runner overflow prechecks to assembled messages, preserves pre-assembly gating for opt-in engines, and adds runner/precheck regressions.

## Best Possible Solution

Land the host-side assembled prompt-authority fix after preserving the true pre-assembly snapshot for opt-in engines, updating the Plugin SDK API baseline and public docs, adding the required changelog entry, and rerunning focused exact-head validation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: provide a context engine that returns a small assembled prompt from a huge raw transcript, then the runner still passes the larger pre-assembly snapshot into the overflow precheck. The PR-specific opt-in regression can be reproduced with an engine that mutates the `messages` array in place before returning `promptAuthority: "preassembly_may_overflow"`.

## Solution Assessment

No, not as currently patched. The direction is the narrow maintainable fix, but the final implementation should snapshot pre-assembly messages before calling `assemble()` for opt-in mode and complete the SDK/docs/changelog updates.
