## Summary

Keep open. The latest PR head is a narrow and directionally correct host-side fix for the linked context-engine overflow bug, and it also addressed earlier type-derivation/allocation review comments, but it still lacks required public-contract follow-through: the Plugin SDK API baseline is not updated, the context-engine docs do not document the new field, and the changelog has no entry. The PR is also currently reported dirty against its base, so it needs branch repair before merge.

## What This Changes

The PR adds optional `AssembleResult.promptAuthority`, makes assembled context the default overflow-precheck authority in the embedded runner, preserves pre-assembly overflow gating for explicit opt-in engines, and adds regression coverage for both paths.

## Best Possible Solution

Keep the PR open, preserve the assembled-default/explicit-opt-in runtime design, add the missing SDK baseline/docs/changelog follow-through, repair the dirty branch state, then rerun the focused runner tests plus Plugin SDK API checks before landing so the linked bug closes after merge.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: a context engine can replace `activeSession.messages` with a small assembled prompt, while the later precheck still receives the larger pre-assembly snapshot and chooses the larger estimate.

## Solution Assessment

Yes, directionally. Making assembled messages authoritative by default with an explicit `preassembly_may_overflow` opt-in is the narrow maintainable fix; the proposed patch is not merge-ready until the public SDK contract, docs, changelog, and dirty-branch state are repaired.
