## Summary

Keep open: the MiniMax VLM success-path read is a valid hardening target, but this patch swallows the new overflow error and the PR body only shows static typecheck/code-inspection proof, so it is not merge-ready.

## What This Changes

The PR replaces MiniMax VLM success-path `res.json()` with a 16 MiB `readResponseWithLimit` read followed by manual JSON parsing.

## Best Possible Solution

Keep the MiniMax VLM success-path byte cap, preserve the explicit overflow error, and add runtime proof or focused regression coverage for oversized and normal success responses before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main still uses unbounded `res.json()` on successful MiniMax VLM responses. I did not run a streaming oversized-body repro in this read-only review.

## Solution Assessment

No, not yet: bounding the read is the right direction, but this implementation should not swallow the overflow error and should include runtime proof of the bounded-read behavior.
