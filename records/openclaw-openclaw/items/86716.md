## Summary

Keep open: the PR targets a real Discord delivery-accounting bug, but the new `false` interaction-delivery result is not wired into dispatcher failure accounting, and the supplied proof does not exercise any changed path.

## What This Changes

The PR changes Discord interaction reply delivery to return a boolean result, fails sanitizer-empty final Discord replies, and recognizes both `send` and `sendMessage` tool-send actions.

## Best Possible Solution

Propagate failed interaction sends into dispatcher failed or cancelled accounting, keep the sanitizer and send-extraction fixes, then add a native-dispatch regression plus redacted real Discord proof for one changed path.

## Reproduction Assessment

Yes from source: the dispatcher queues final counts before delivery and only subtracts failed or cancelled counts, while the PR head returns successfully when `deliverDiscordInteractionReply` reports `false`. I did not run a live expired Discord interaction in this read-only review.

## Solution Assessment

No as written. Returning a boolean from `deliverDiscordInteractionReply` is useful, but the best fix must integrate that failure signal with the dispatcher result/fallback contract that decides whether a visible reply landed.
