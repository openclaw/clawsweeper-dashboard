## Summary

Keep this PR open: the direction is useful, but the patch does not yet fix the central Discord reply-accounting path because failed interaction delivery can still leave a nonzero queued final count that suppresses the fallback. The real-behavior proof is also too indirect for the changed failure modes.

## What This Changes

The PR changes Discord reply accounting so interaction replies return explicit delivery status, sanitizer-empty final replies fail, and both Discord `send` action names extract targets.

## Best Possible Solution

Thread Discord interaction delivery success into the native-agent accounting/fallback boundary, keep the focused sanitizer and send-extraction fixes, and add real proof for at least one changed Discord behavior path before merge.

## Reproduction Assessment

Yes from source: current-main can swallow expired Discord interaction errors, and the shared dispatcher increments queued final counts before delivery. The PR tests the callee but not the native-agent accounting guard that decides whether fallback is skipped.

## Solution Assessment

No as written. Returning `false` from `deliverDiscordInteractionReply` is a useful start, but the best fix must integrate that result with the dispatcher delivery-result contract or the fallback/accounting decision that currently reads queued counts.
