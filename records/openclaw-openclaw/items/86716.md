## Summary

Keep this PR open: the Discord delivery-accounting direction is useful, but the patch still lets a false interaction delivery return normally from the native command deliver callback, so dispatcher accounting can keep the reply counted as queued/delivered.

## What This Changes

The PR hardens Discord reply delivery accounting by returning explicit interaction delivery status, failing sanitized-empty final replies, and normalizing Discord message-tool send extraction for `send` and `sendMessage`.

## Best Possible Solution

Keep the explicit Discord delivery result, but wire false delivery into dispatcher-visible failed or cancelled accounting and add a native command dispatcher regression test for the expired-interaction path.

## Reproduction Assessment

Yes from source inspection: the PR head returns normally on `delivered === false`, while current dispatcher accounting only subtracts failed or cancelled deliveries, so the queued final count remains nonzero.

## Solution Assessment

No; the approach is close, but false Discord delivery must propagate into dispatcher accounting instead of only returning from the Discord callback.
