## Summary

Keep open: the Discord delivery-accounting fix is still useful, but this PR is not merge-ready because a false interaction delivery still resolves the dispatcher callback successfully, leaving final delivery counted and fallback suppressed; the supplied live output also does not demonstrate one changed Discord path.

## What This Changes

The PR changes Discord interaction reply delivery to return a boolean result, fails sanitizer-empty final Discord replies, recognizes both `send` and `sendMessage` as send actions, and adds focused tests.

## Best Possible Solution

Wire false Discord interaction delivery into dispatcher failed/cancelled accounting, keep the sanitizer and send-extraction fixes, add a focused native-dispatch regression, then provide redacted real Discord proof for one changed path.

## Reproduction Assessment

Yes from source: the dispatcher queues final counts before delivery and only subtracts failed or cancelled counts. On the PR head, a false Discord interaction delivery returns from the callback without creating either signal.

## Solution Assessment

No as written. Returning a boolean from `deliverDiscordInteractionReply` is a reasonable direction, but the best fix must feed that failure signal into the dispatcher result/fallback contract that decides whether a visible reply landed.
