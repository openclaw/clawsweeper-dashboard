## Summary

Keep open. The PR targets a real over-broad cooldown problem, but the current diff is not mergeable because it bypasses OpenAI Codex WHAM cooldown probing for ambiguous failures and leaves user-facing docs/changelog out of sync.

## What This Changes

The PR changes the PI embedded runner auth-profile failure policy and unit tests so shared profiles only cooldown for selected auth-health reasons while non-auth failure reasons return null.

## Best Possible Solution

Keep the narrower cooldown direction, but make the policy provider-aware or split a separate Codex WHAM path: Copilot format/model-path failures should not poison a shared profile, while OpenAI Codex WHAM-probed ambiguous failures should still reach provider-owned cooldown classification with regression coverage for both cases.

## Reproduction Assessment

Yes. The source path is high-confidence: current main maps 400/422 unclassified provider rejections to format, passes shared non-timeout reasons to markAuthProfileFailure, and the PR would null ambiguous Codex reasons before the existing WHAM-covered markAuthProfileFailure path.

## Solution Assessment

No. The proposed blanket allowlist is too broad; the safer maintainable fix is a provider-aware policy that filters request/model-path failures without disabling Codex WHAM probing for ambiguous profile-health signals.
