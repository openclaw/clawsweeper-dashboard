## Summary

Keep open: current main still has the broad auth-profile cooldown behavior this PR is trying to narrow, so the contribution is useful, but the proposed blanket null mapping would also bypass the OpenAI Codex WHAM cooldown path for ambiguous failures before provider-specific handling can run.

## What This Changes

The PR changes the PI embedded runner auth-profile failure policy to cooldown profiles only for selected auth-health reasons and adds focused unit coverage for filtered and retained failover reasons.

## Best Possible Solution

Revise the PR so client-side payload/model-path failures such as the reported GitHub Copilot assistant-prefill 400 do not poison shared auth health, while openai-codex ambiguous failures still reach WHAM-based cooldown classification. Add regression coverage for both paths and the required changelog entry.

## Reproduction Assessment

Yes. The source path is high-confidence: PR-filtered reasons become null, null reasons return early in maybeMarkAuthProfileFailure, and current tests show unknown openai-codex failures are expected to reach WHAM fallback cooldown handling.

## Solution Assessment

No. The direction is right, but the current implementation is too broad; the safer solution is provider-aware narrowing or a split Codex WHAM path rather than globally dropping empty_response, no_error_details, unclassified, and unknown.
