## Summary

Keep this PR open, but do not merge it as-is. The reported cooldown pollution is a real direction to fix, yet the patch drops ambiguous failure reasons before the existing OpenAI Codex WHAM cooldown probe can classify exhausted Codex profiles, so it would regress current Codex auth-profile behavior.

## What This Changes

The PR changes `resolveAuthProfileFailureReason` and its focused tests so only selected auth-health failover reasons can put a shared auth profile into cooldown.

## Best Possible Solution

Keep the narrower auth-health policy direction, but make the split provider-aware: ignore Copilot/client-side format and model-path failures for shared auth cooldowns while still allowing OpenAI Codex ambiguous failures to reach `markAuthProfileFailure` and WHAM probing. Add regression coverage for both the Copilot non-auth case and the Codex ambiguous-cooldown case.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: the PR maps `unknown` and related ambiguous reasons to null, `maybeMarkAuthProfileFailure` returns early on null, and current tests require `unknown` OpenAI Codex failures to produce a WHAM-backed cooldown.

## Solution Assessment

No. The patch is directionally useful but too broad because it removes provider-specific Codex cooldown classification; the safer solution is a provider-aware policy or separate probe path that preserves WHAM while preventing Copilot format errors from poisoning shared auth health.
