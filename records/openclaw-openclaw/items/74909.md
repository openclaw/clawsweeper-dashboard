## Summary

Keep open. The reported format-cooldown bug class is real and current main already has the narrow format exclusion, but this PR remains incorrect because its broader null mapping bypasses OpenAI Codex WHAM probes and drops overloaded cooldown state that current docs/tests require; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR changes the PI embedded runner auth-profile failure resolver to allowlist only selected auth-health reasons for profile cooldown and adds unit coverage for filtered reasons.

## Best Possible Solution

Keep the shipped format/server_error exclusions, preserve WHAM-probed ambiguous reasons and existing overloaded transient cooldowns, and consider any further reason filtering only as a provider/reason-aware change with regression coverage and real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than a live run: 400/422 payload rejections can classify as format, and current main already skips profile cooldown for format. The PR’s broader filtered reasons also trace directly through maybeMarkAuthProfileFailure to skipped WHAM and overloaded cooldown handling.

## Solution Assessment

No. Narrowing request-shape format cooldown was the right fix and is already on main, but this branch’s blanket allowlist is too broad and should be replaced with a narrower provider/reason-aware policy.
