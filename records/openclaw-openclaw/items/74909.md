## Summary

Keep open. The reported Copilot cooldown cascade is plausible on current `main`, but this PR has a blocking compatibility issue: it also filters ambiguous failure reasons before the OpenAI Codex WHAM cooldown probe can run, which current code and tests treat as intentional behavior.

## What This Changes

The PR changes the PI embedded runner auth-profile failure policy to cooldown profiles only for auth, auth_permanent, billing, rate_limit, and session_expired reasons, with focused unit coverage for filtered and retained reasons.

## Best Possible Solution

Keep the narrower auth-health direction, but make the policy provider-aware or split a separate Codex WHAM probe path. Copilot `format`/model-path failures should not cooldown a shared profile, while Codex `unknown` and other WHAM-probed ambiguous failures should still reach provider-owned cooldown classification with regression tests for both paths.

## Reproduction Assessment

Yes. The source path is high-confidence: a non-null failover reason reaches `markAuthProfileFailure`, while a null reason returns early; current tests also cover the Codex `unknown` WHAM fallback that this PR bypasses.

## Solution Assessment

No. The proposed direction is maintainable, but the blanket null mapping is not the best fix because it removes provider-specific health probing for OpenAI Codex; the safer alternative is provider-aware narrowing.
