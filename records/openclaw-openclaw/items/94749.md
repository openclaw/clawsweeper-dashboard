## Summary

Keep open: the reported Codex turn/start overflow is real, but this PR is not safe to merge because the new final clamp slices from the beginning and can drop the current user request instead of trimming older context; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a final length clamp to `fitCodexProjectedContextForTurnStart` so the concatenated Codex turn prompt is sliced to `maxChars`.

## Best Possible Solution

Bound the overflow path with a tail-preserving clamp that keeps the current request whenever possible, then add focused regression tests and redacted after-fix Codex app-server proof.

## Reproduction Assessment

Yes at source level: with `beforeContext + afterContext` over `maxChars`, the PR returns a left slice of the final prompt, while current main places the current user request in the suffix. I did not run a live Codex app-server repro in this read-only review.

## Solution Assessment

No. A final clamp is plausible, but this implementation is the wrong shape because it can trim the newest user input; the safer fix is a tail-preserving overflow branch with focused tests.
