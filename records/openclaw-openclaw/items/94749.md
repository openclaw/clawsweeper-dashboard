## Summary

Keep open: current main still has the linked Codex turn-start overflow bug, but this PR is not merge-ready because its front-slice clamp can discard the current user request in the overflow branch and the PR lacks required after-fix real behavior proof.

## What This Changes

The PR adds a final length clamp to `fitCodexProjectedContextForTurnStart` so the concatenated projected prompt cannot exceed `maxChars`.

## Best Possible Solution

Use a tail-preserving overflow fallback for the non-context text, add regression coverage for `beforeContext + afterContext > maxChars`, and require real after-fix proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: if the non-context wrapper plus current request exceed `maxChars`, current main removes the context slice and still returns over-limit text. I did not run tests or a live app-server repro because this review was constrained to read-only inspection.

## Solution Assessment

No. A front `slice(0, maxChars)` is not the best fix because it can discard the current request tail; the maintainable fix should preserve recent/current text with the existing tail-preserving truncation behavior and prove that case with regression coverage.
