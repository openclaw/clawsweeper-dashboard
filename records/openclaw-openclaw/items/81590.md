## Summary

Keep open. The latest head appears to address the earlier protocol, port, and basic SecretInput drift and includes plausible live proof for the happy path, but source review still finds false-restart cases in auth selection plus a security-sensitive SecretRef exec-resolution gap in the new watchdog.

## What This Changes

The PR adds a Python gateway health watchdog that performs an authenticated WebSocket connect probe, can kickstart the macOS gateway service, and adds Python/Vitest coverage for protocol, port, and SecretInput contracts.

## Best Possible Solution

Consolidate this with the broader gateway recovery design, and make any watchdog probe reuse a maintained gateway client/auth/SecretRef contract before restart behavior lands.

## Reproduction Assessment

Yes for the review findings. Source inspection shows configs such as password-mode auth, no-auth mode, or config-first tokens can make the watchdog report failure against a healthy gateway, and exec SecretRefs use a direct subprocess path instead of the hardened resolver.

## Solution Assessment

No. The authenticated connect probe is a useful direction, but duplicating gateway auth and SecretRef resolution in a restart script is more fragile than reusing the maintained client/status path or exposing a small generated contract helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor gateway auth mode before probing:** `scripts/gateway_health_watchdog.py:453-470`
  - body: `_load_gateway_auth()` returns env credentials before checking `gateway.auth.mode` or config-first service auth. A healthy gateway using password mode, no-auth mode, or a config token that differs from `OPENCLAW_GATEWAY_TOKEN` will fail two probes and can be kickstarted unnecessarily; choose credentials with the same auth-mode and precedence rules as the gateway runtime.
  - confidence: 0.9
- **[P1] Keep exec SecretRefs on the hardened resolver path:** `scripts/gateway_health_watchdog.py:356-364`
  - body: The new resolver runs `secrets.providers.*.command` directly, but the maintained SecretRef resolver first validates command path ownership/symlink/trustedDirs and enforces no-output/max-output limits. A launchd watchdog should not bypass that security contract when resolving gateway auth; reuse the existing resolver/client path or mirror those checks before spawning provider commands.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a local code-execution path for exec SecretRefs that bypasses the maintained resolver's path and output safety checks.

Concerns:

- **[medium] Exec SecretRefs bypass hardened resolver checks:** `scripts/gateway_health_watchdog.py:356`
  - body: `subprocess.run` executes the configured provider command without the current resolver's command ownership, symlink, trustedDirs, no-output-timeout, and max-output enforcement, which weakens the SecretRef execution contract for a repeatedly-run watchdog.
  - confidence: 0.86
