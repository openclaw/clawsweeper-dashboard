## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route watchdog restarts through the gateway restart coordinator:** `scripts/gateway_health_watchdog.py:808`
  - body: When kickstart is allowed, this path calls `launchctl kickstart -k` directly. That skips the current restart preflight that defers while active work is running and marks restart recovery state, so enabling the watchdog can interrupt live replies or background work instead of using the managed gateway restart flow.
  - confidence: 0.9
- **[P2] Use the existing gateway probe URL/TLS path:** `scripts/gateway_health_watchdog.py:495-499`
  - body: `probe_ws_upgrade()` opens a raw TCP socket and sends a plain `ws://` upgrade regardless of `gateway.tls.enabled`. A valid TLS-enabled local gateway can therefore be reported unhealthy and, with kickstart enabled, restarted unnecessarily; the existing TypeScript probe/client path already carries the URL/TLS behavior.
  - confidence: 0.86
- **[P2] Preserve exec SecretRef resolver policy:** `scripts/gateway_health_watchdog.py:354-365`
  - body: The Python resolver executes configured exec providers directly and does not enforce the shared resolver's secure-path, no-output-timeout, and max-output policies. Gateway auth SecretRefs are a security boundary, so this should delegate to the existing resolver or avoid exec-backed secrets in this script.
  - confidence: 0.86
- **[P2] Honor resolved config and state directories:** `scripts/gateway_health_watchdog.py:59-69`
  - body: The watchdog derives log, state, and config defaults from `OPENCLAW_HOME`, then writes its own JSON state file. Installs using `OPENCLAW_STATE_DIR`, config path candidates, or canonical state resolution can be probed or persisted in the wrong location.
  - confidence: 0.84
