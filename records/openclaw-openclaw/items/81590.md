## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor the resolved state and config directories:** `scripts/gateway_health_watchdog.py:232-267`
  - body: The watchdog hard-codes OpenClaw home/config/log/state discovery around `OPENCLAW_HOME` and `~/.openclaw`, but the runtime supports separately resolved config and state directories. A valid install using those dirs can be probed against the wrong config/log/state and then reported unhealthy or restarted when the real gateway is fine.
  - confidence: 0.88
- **[P2] Use WSS for TLS-enabled local gateways:** `scripts/gateway_health_watchdog.py:514`
  - body: The probe always opens `ws://127.0.0.1:<port>/` and never consults the gateway TLS setting. TLS-enabled local gateways are valid current setups, so this script can mark them unhealthy and, when kickstart is enabled, restart them unnecessarily.
  - confidence: 0.82
- **[P1] Restart through the managed gateway lifecycle:** `scripts/gateway_health_watchdog.py:696-735`
  - body: Even behind the new opt-in env var, `kickstart_gateway()` directly kills matching gateway processes and calls `launchctl kickstart`. That bypasses the existing lifecycle path that defers restarts while active runs or embedded runs are present, so enabling the watchdog can interrupt live replies or cron work instead of using the safe restart coordinator.
  - confidence: 0.9
- **[P1] Preserve exec SecretRef security checks:** `scripts/gateway_health_watchdog.py:332-337`
  - body: The script resolves `exec` SecretRefs with an unbounded `subprocess.check_output(..., shell=True)` call. The shared resolver enforces policy around shell usage, timeout, executable resolution, environment, and max output; reimplementing it here weakens the gateway auth secret boundary for installs using exec-backed secrets.
  - confidence: 0.86
