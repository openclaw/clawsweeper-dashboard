## Summary

Keep this PR open, but it is not ready to merge: the authenticated handshake probe is useful in principle, but the new restart-capable watchdog can misclassify healthy gateways when the port or auth token uses supported non-default configuration.

## What This Changes

The PR adds a local `scripts/gateway_health_watchdog.py` watchdog that probes the gateway WebSocket `connect` handshake and changes one live model test helper return type for lint.

## Best Possible Solution

Keep the authenticated handshake idea, but implement it through existing gateway client/config credential resolution and configured port handling before deciding whether this restart-capable ops script belongs in core.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run the script against a healthy gateway configured on a non-default port, or with `gateway.auth.token` stored as a SecretRef and no plain `OPENCLAW_GATEWAY_TOKEN` in the watchdog environment. In either case the script reports probe failure despite supported current-main configuration.

## Solution Assessment

No. Completing the authenticated `connect` handshake is the right direction, but this implementation should reuse the existing gateway client/config secret contract and honor the configured endpoint before it is safe as a watchdog that can restart services.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor the configured gateway port:** `scripts/gateway_health_watchdog.py:55-56`
  - body: The watchdog always probes `127.0.0.1:18789` and later treats two failed probes as `ws-upgrade-fail`, but OpenClaw supports `gateway.port`, `--port`, and `OPENCLAW_GATEWAY_PORT`. A healthy gateway on a non-default port would be marked unhealthy and can be kickstarted unnecessarily, so the script needs to resolve or require the actual configured endpoint before restart logic is enabled.
  - confidence: 0.91
- **[P2] Resolve SecretRef-managed gateway tokens:** `scripts/gateway_health_watchdog.py:206-215`
  - body: `_load_gateway_token()` only accepts `OPENCLAW_GATEWAY_TOKEN` or a plaintext string at `gateway.auth.token`. Current gateway auth supports SecretRef-managed tokens, including daemon/onboarding flows, so those installations would report a missing token, accumulate WS probe failures, and may restart a healthy gateway. Reuse the existing credential resolution path or fail inert when the auth source cannot be resolved.
  - confidence: 0.88
