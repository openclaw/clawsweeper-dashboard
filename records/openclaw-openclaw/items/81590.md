## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Read the current protocol package constants:** `scripts/gateway_health_watchdog.py:155`
  - body: The script reads `src/gateway/protocol/version.ts`, but current main exposes protocol constants from `packages/gateway-protocol/src/version.ts`. As written, the watchdog can return `unable to resolve gateway protocol version from repo` before it ever performs the authenticated connect probe.
  - confidence: 0.95
- **[P1] Restart through the managed gateway lifecycle:** `scripts/gateway_health_watchdog.py:655`
  - body: Even behind `GATEWAY_WATCHDOG_ALLOW_KICKSTART`, this calls `launchctl kickstart -k` directly. That bypasses the existing `scheduleGatewaySigusr1Restart` deferral, cooldown, and session-aware restart machinery, so enabling the watchdog can interrupt active gateway work outside the managed lifecycle.
  - confidence: 0.9
- **[P2] Preserve the shared exec SecretInput policy:** `scripts/gateway_health_watchdog.py:358-366`
  - body: The Python `exec` SecretInput resolver runs the configured command itself instead of using the shared gateway SecretInput resolver. That creates a second security-sensitive auth secret path that can drift from shared timeout, trusted path, output, cleanup, and unavailable-secret behavior.
  - confidence: 0.86
- **[P2] Use the TLS-aware gateway probe path:** `scripts/gateway_health_watchdog.py:495-499`
  - body: The probe always opens a plaintext socket to `127.0.0.1`, while current gateway connection details can resolve local probes to `wss://` with a TLS fingerprint when `gateway.tls.enabled` is set. A healthy TLS-enabled local gateway can therefore be marked unhealthy.
  - confidence: 0.82
