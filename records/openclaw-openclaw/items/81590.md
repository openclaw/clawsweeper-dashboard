## Summary

Keep open: the proposed watchdog is not safe to merge because it hardcodes an obsolete gateway protocol version and misses supported gateway configuration paths, so it can restart healthy gateways. The supplied terminal proof is useful context, but it does not line up with the current main protocol contract.

## What This Changes

The PR adds a Python gateway health watchdog script that probes and kickstarts a local gateway, plus a one-line live model test return-type annotation change.

## Best Possible Solution

If this watchdog is wanted, implement it through the maintained gateway client or daemon health path, with tests covering current protocol constants, token/password credential resolution, SecretInput-backed credentials, and configured ports.

## Reproduction Assessment

Yes. Source inspection is enough: current main requires gateway protocol 4, while the added script advertises protocol 3, so a current gateway will reject the connect probe with protocol mismatch.

## Solution Assessment

No. Duplicating the gateway handshake in Python with hardcoded protocol, auth, and port values is less maintainable than reusing the existing GatewayClient/health command or shared generated protocol constants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the current gateway protocol version:** `scripts/gateway_health_watchdog.py:275-276`
  - body: This connect probe advertises protocol 3, but current main exports `PROTOCOL_VERSION` and `MIN_CLIENT_PROTOCOL_VERSION` as 4 and the server rejects clients whose `maxProtocol` does not cover the current protocol. As written, the watchdog will mark a healthy current gateway as failed and can kickstart it after the debounce window.
  - confidence: 0.95
- **[P2] Resolve the configured gateway auth mode:** `scripts/gateway_health_watchdog.py:206-215`
  - body: The probe only accepts `OPENCLAW_GATEWAY_TOKEN` or a raw `gateway.auth.token` string. Supported gateways can use password auth, env password auth, or SecretInput-backed credentials, so those healthy configurations will be reported as missing a token and treated as unhealthy.
  - confidence: 0.86
- **[P2] Use the configured gateway port:** `scripts/gateway_health_watchdog.py:55-56`
  - body: The script always probes port 18789 even though OpenClaw resolves the gateway port from `--port`, `OPENCLAW_GATEWAY_PORT`, `gateway.port`, then the default. A healthy gateway on a configured non-default port will be misreported as down and the watchdog will kickstart the wrong service path.
  - confidence: 0.82
