## What This Changes

The issue requests retry behavior for transient MiniMax timeouts so exhausting the configured fallback chain does not immediately fail the whole user request.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73374 or an equivalent idempotency-gated transient retry policy, then close this issue after post-merge validation confirms MiniMax timeout behavior.

## Reproduction Assessment

Yes for source-level behavior: current main walks each configured fallback candidate once and throws a fallback summary after exhaustion. I did not reproduce a live MiniMax service timeout during this read-only review.

## Solution Assessment

No for the broad suggested exponential backoff around whole requests: that can replay non-idempotent work. The safer solution is the active idempotency-gated same-model transient retry approach or an equivalent maintainer-approved policy.
