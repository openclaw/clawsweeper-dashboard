## Summary

Keep open. Current main handles timeout as a model-fallback signal, but the requested retry after MiniMax fallback exhaustion is not implemented: configured fallback runs disable the existing same-model idle-timeout retry, and the fallback loop tries each candidate once before throwing a summary error.

## What This Changes

Add retry/backoff behavior for MiniMax timeout failures so exhausting MiniMax M2.7 and M2.7-highspeed does not immediately fail the whole request.

## Best Possible Solution

Implement a bounded, replay-safe timeout retry path that covers configured fallback chains, preferably by extending the existing idle-timeout retry guard so it only retries when no assistant text, tool calls, messaging sends, approvals, or other side effects occurred. Keep MiniMax catalog/auth unchanged unless provider-specific evidence shows the fix must be scoped to MiniMax.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core behavior with a mocked timeout/fallback path: current tests already simulate timeout-to-fallback, and the source shows same-model idle retry is disabled when fallbacks are configured. A live MiniMax-only reproduction remains unverified.

## Solution Assessment

Is this the best way to solve the issue? Unclear as written: a bounded replay-safe timeout retry is appropriate, but a broad exponential retry for every provider timeout could duplicate side effects or stall requests. The safer path is to reuse the existing idle-timeout safety gates and add focused regression coverage.

## Repair Work Prompt

Fix the MiniMax/API timeout fallback-exhaustion behavior tracked at https://github.com/openclaw/openclaw/issues/62464. Current main classifies timeouts and walks model fallbacks, but `allowSameModelIdleTimeoutRetry` is disabled when `fallbackConfigured` is true and `runWithModelFallback` tries each candidate once before `FallbackSummaryError`. Add a narrow, bounded retry for timeout/idle-timeout failures in configured fallback-chain runs, preferably reusing the existing `canRestartForLiveSwitch` and `MAX_SAME_MODEL_IDLE_TIMEOUT_RETRIES` safety gates so retries only occur before assistant text, tool calls, messaging sends, approvals, or other side effects. Do not add unrelated fallback models, do not change MiniMax auth/onboarding/catalog behavior, and do not add broad retries for non-timeout failures. Add regression coverage for a configured fallback chain where timeout candidates get the intended bounded retry behavior and still surface a clear summary when retries are exhausted. Because this is user-facing runtime behavior, add a one-line `CHANGELOG.md` fix entry crediting `Thanks @jaydenlichong29-create` if the repo's active changelog section is available.
