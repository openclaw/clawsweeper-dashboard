## Summary

Keep open. Current main still makes one pass over the configured model fallback chain and throws after exhaustion; the open related PR covers idle-timeout-to-fallback signaling, but not the broader retry/backoff policy requested here.

## What This Changes

The issue asks OpenClaw to recover from MiniMax timeout failures by retrying boundedly instead of failing the whole request after the configured fallback chain is exhausted.

## Best Possible Solution

Review and land the idle-timeout fallback PR if acceptable, then make a separate maintainer decision on a bounded, replay-safe retry policy in the shared fallback path rather than a MiniMax-only retry loop.

## Reproduction Assessment

Yes for the source-level behavior: current main walks each fallback candidate once and can throw a fallback summary after timeout failures. No live MiniMax outage reproduction was run in this read-only review.

## Solution Assessment

Unclear as suggested: generic exponential backoff across whole model runs risks duplicate side effects. The safer direction is a narrow, replay-safe retry policy coordinated with the open idle-timeout fallback PR and covered by focused fallback tests.
