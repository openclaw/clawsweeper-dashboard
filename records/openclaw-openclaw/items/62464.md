## What This Changes

The issue requests retry/backoff behavior for MiniMax provider timeouts after configured model fallback candidates are exhausted.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73374 or an equivalent idempotency-gated same-model transient retry policy, then validate MiniMax/provider timeout behavior before closing this issue.

## Reproduction Assessment

Yes for source-level behavior: current main walks each configured fallback candidate once and throws a fallback summary after exhaustion. I did not reproduce a live MiniMax API timeout in this read-only review.

## Solution Assessment

No for the broad suggested exponential backoff around whole requests, because it can replay non-idempotent work. The safer path is the active idempotency-gated same-model transient retry design or an equivalent maintainer-approved policy.
