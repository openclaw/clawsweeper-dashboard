## Summary

Keep open. Current main now handles the adjacent idle-timeout-to-fallback gap, but it still does not implement a retry/backoff policy after configured model fallback candidates time out and are exhausted; that remaining behavior needs maintainer policy judgment because retries can replay side effects.

## What This Changes

Request retry/backoff behavior for MiniMax timeouts so timed-out requests do not fail after the configured fallback chain is exhausted.

## Best Possible Solution

Define a shared, replay-safe timeout retry policy in the agent fallback path, including retry count, backoff, user-visible status/error behavior, and how it coordinates with configured model fallbacks.

## Reproduction Assessment

Yes for the source-level no-retry behavior: current main tries each fallback candidate once and throws a fallback summary after exhaustion. No live MiniMax outage reproduction was run in this read-only review.

## Solution Assessment

Unclear as suggested. A generic exponential backoff around whole runs is risky because of replay side effects; the maintainable direction is a bounded retry policy with explicit replay-safety gates and focused fallback tests.
