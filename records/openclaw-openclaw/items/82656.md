## Summary

Keep this PR open: it addresses the common false-like values, but the proposed gate still enables QQBot debug logging for any unrecognized non-falsy string, which leaves the linked explicit-truthy expectation only partially implemented. The PR also still needs after-fix real behavior proof before merge.

## What This Changes

The PR changes QQBot debug logging to disable logs for a hardcoded set of false-like `QQBOT_DEBUG` values and adds table tests for false-like and truthy values.

## Best Possible Solution

Make QQBot debug logging opt in only for explicit truthy values, keep the helper local to preserve the engine boundary, and cover false-like plus unknown-string cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `QQBOT_DEBUG=0` or `false` on current main and the `!!process.env.QQBOT_DEBUG` guard passes before `debugLog`, `debugWarn`, or `debugError` writes to the console.

## Solution Assessment

No. The false-list approach is a useful partial fix, but the linked behavior and existing core helper point to an explicit truthy allowlist as the narrower maintainable solution.
