## Summary

Keep open: the PR addresses the reported QQBot cron preview regression and now includes live proof plus changelog credit, but the new QQBot explicit parser can hide mismatched provider prefixes before the shared runtime guard rejects them.

## What This Changes

The PR adds QQBot explicit cron target parsing, repeated-prefix rejection, cron preview prefix de-duplication, regression tests, and a changelog entry.

## Best Possible Solution

Keep the QQBot-owned parser and preview approach, but make `parseExplicitTarget` decline or preserve non-QQBot provider-prefixed targets so the shared mismatch guard still rejects them, with a regression test for explicit QQBot delivery plus another provider prefix.

## Reproduction Assessment

Yes, source-level: current main can duplicate `qqbot:` in cron preview labels and lacks a QQBot explicit-target parser, matching the linked report. I did not run a live current-main Gateway in this read-only review.

## Solution Assessment

No, not quite: the PR uses the right channel-owned parser seam and proves the reported preview fix, but the new parser should not consume foreign provider-prefixed targets before shared validation can reject them.

## Review Findings

- **[P2] Keep foreign provider prefixes visible to validation:** `extensions/qqbot/src/engine/messaging/target-parser.ts:128-130`

Return `null` or otherwise preserve raw non-QQBot provider-prefixed targets so the shared guard still sees them.
