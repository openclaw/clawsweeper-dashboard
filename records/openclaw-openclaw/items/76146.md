## Summary

Keep open. The PR is pointed at a real CLI SecretRef bug, but the submitted patch still misses a documented web-search credential fallback and introduces a build-sensitive static-plus-dynamic import pattern; it also needs the required user-facing changelog entry.

## What This Changes

The PR resolves SecretRefs for the `capability web search` CLI path before invoking the web-search runtime, adds a web-search command secret target helper, and extends related target/coverage tests.

## Best Possible Solution

Merge a focused CLI SecretRef fix that resolves every credential path the web-search runtime can consume, keeps imports build-safe, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main has a high-confidence reproduction path by configuring a web-search API key as a SecretRef and running `openclaw capability web search --query ...`; the remaining Gemini fallback gap is also traceable from the provider code without needing live credentials.

## Solution Assessment

No, not as submitted. The shared command SecretRef resolver is the right foundation, but the target set must include runtime credential fallbacks such as `models.providers.google.apiKey`, and the config import should be made static before merge.
