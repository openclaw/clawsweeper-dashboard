## Summary

Keep open: current main has compaction-specific branch/restore and general session creation primitives, but it still lacks the requested named native save/load snapshot workflow; the related open restore PR is narrower and does not safely supersede the broader product design.

## What This Changes

Add named native session checkpoints that can save, restore, reuse, and fork transcript, context, model selection, and related session state.

## Best Possible Solution

Define a native SQLite-backed snapshot data model and protocol that uses SessionManager DAG-safe operations, separates non-destructive branch from destructive restore, and explicitly specifies retention, model/config capture, lifecycle hooks, and optional archive import before adding command and UI surfaces.

## Reproduction Assessment

Not applicable: the absence of named native session snapshots is directly verifiable from current commands, Gateway APIs, tests, and docs, but it is a requested capability rather than a broken existing contract.

## Solution Assessment

No: named snapshots are a valid product direction, but the proposed raw JSONL checkpoint storage and single load operation do not satisfy current SQLite ownership, transcript-DAG integrity, lifecycle-hook, and branch-versus-restore requirements.
