## Summary

Keep open. Current main has useful adjacent recovery primitives, but it still does not implement the requested named `/session save` and `/session load` workflow, reusable session templates, or restore/import from reset/deleted transcript archives; the issue discussion also has an explicit maintainer keep-open signal.

## What This Changes

The issue requests explicit session snapshot save/load, restore, and branching semantics for long-running sessions.

## Best Possible Solution

Define one native session snapshot model on top of SessionManager, session store metadata, compaction checkpoint artifacts, reset/deleted transcript archives, and subagent fork support, then expose coherent save/list/load/branch/import semantics through chat commands, Gateway protocol, CLI/Control UI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request. The current-main gap is high-confidence from docs and source: `/session` only accepts `idle`/`max-age`, and the available checkpoint/fork primitives are narrower than named session snapshots.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer choices on restore semantics, retention, archive import, permissions, and cache-preserving fork behavior before a narrow maintainable fix can be selected.
