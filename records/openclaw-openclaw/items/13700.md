## Summary

Keep open. Current main has useful adjacent primitives for compaction checkpoint branch/restore and subagent transcript forking, but it still lacks the requested named `/session save` and `/session load` workflow, reusable session templates, and restore/import from reset or deleted transcript archives; the recent maintainer discussion also explicitly kept this design open.

## What This Changes

Requested named session snapshots that can save, load, restore, or fork an OpenClaw session context checkpoint.

## Best Possible Solution

Define one core-owned session snapshot lifecycle that builds on SessionManager, session store metadata, compaction checkpoint artifacts, reset/deleted transcript archives, and subagent fork support, then expose coherent save/list/load/branch/import semantics through commands, Gateway protocol, UI/CLI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request. The current-main gap is high-confidence from docs and source: `/session` only accepts `idle`/`max-age`, and available checkpoint/fork primitives are narrower than named session snapshots.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer choices on restore behavior, retention, archive import, permissions, and cache-preserving fork semantics before a narrow maintainable fix can be selected.
