## Summary

Current main has strong adjacent primitives for compaction-checkpoint restoration, SessionManager-based resume/import, and transcript forking, but still lacks user-created named snapshots. The open `/resume` PR covers only prior-session recovery, leaving this issue necessary as the canonical product-design tracker.

## What This Changes

Add native named session checkpoints that users can save, load, restore, and fork while preserving relevant transcript, context, lineage, and model state.

## Best Possible Solution

Define separate resume, named-checkpoint restore, and fork operations over canonical SessionManager and Gateway state, with explicit lineage, authorization, retention, model-selection, and destructive-replacement semantics; do not create a parallel workspace JSONL snapshot subsystem.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a missing capability. Exact target-main source and public docs provide a high-confidence verification path showing that no named save/load or general native resume command exists.

## Solution Assessment

No, the proposed single JSONL save/load mechanism is not the best final design. Current OpenClaw and Codex ownership contracts distinguish resume, restore, import, and fork, so the maintainable path is to expose deliberate operations over those canonical primitives.
