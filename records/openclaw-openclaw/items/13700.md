## Summary

Keep open. Current main has adjacent compaction checkpoint recovery and subagent transcript forking, but it still lacks the requested named `/session save` and `/session load` workflow, reusable templates, and restore/import from reset or deleted transcript archives; recent maintainer discussion also kept this design open.

## What This Changes

Add native session snapshot commands and APIs to save, load, restore, and fork conversation checkpoints across session state.

## Best Possible Solution

Define one core-owned session snapshot lifecycle that composes SessionManager, session store metadata, compaction checkpoint artifacts, reset/deleted transcript archives, and subagent fork support, then expose coherent save/list/load/branch/import semantics through commands, Gateway protocol, UI/CLI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request. The current-main gap is high-confidence from docs and source: `/session` only accepts `idle`/`max-age`, while checkpoint/fork primitives are narrower than named session snapshots.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation requires maintainer choices on restore behavior, retention, archive import, permissions, and cache-preserving fork semantics before a narrow maintainable fix can be selected.
