## Summary

Keep open. Current main has adjacent compaction checkpoint branch/restore and native subagent transcript-fork primitives, but it still lacks the requested named `/session save` and `/session load` workflow, reusable session templates, and restore/import from `.reset` or `.deleted` transcript archives; the maintainer discussion also explicitly kept this design open.

## What This Changes

The issue requests explicit session snapshot commands to save, load, restore, and fork conversation context checkpoints for rollback, branching, batch reuse, and recovery after resets or restarts.

## Best Possible Solution

Define one native session snapshot model that builds on SessionManager, session store metadata, compaction checkpoint artifacts, reset/deleted transcript archives, and subagent fork support, then expose coherent save/list/load/branch/import semantics through chat commands, Gateway protocol, CLI/Control UI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request. The current-main gap is high-confidence from docs and source: `/session` only accepts `idle`/`max-age`, and the available checkpoint/fork primitives are narrower than named session snapshots.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer choices on restore semantics, retention, archive import, permissions, and cache-preserving fork behavior before a narrow maintainable fix can be selected.
