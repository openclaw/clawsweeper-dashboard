## Summary

Keep open: current main has adjacent compaction checkpoint restore and native subagent transcript fork primitives, but it still lacks the requested named `/session save` and `/session load` snapshot workflow. The active restore PR covers one archive-restore slice and the related fork/resume issue covers another API slice, so this remains the broader product/API tracker.

## What This Changes

Request native session snapshot support for saving, loading, restoring, forking, and reusing context checkpoints across long-running sessions.

## Best Possible Solution

Define one native session snapshot lifecycle over session storage, compaction checkpoints, reset/deleted archives, and subagent fork primitives, then expose agreed save/list/load/branch/restore behavior through commands, Gateway protocol, SDK/UI/CLI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, protocol, UI, live issue, and related-PR inspection give high confidence that current main lacks the requested named save/load snapshot workflow while adjacent primitives exist.

## Solution Assessment

Unclear pending maintainer API direction. The requested slash commands are plausible, but the maintainable path is a unified session snapshot lifecycle contract rather than treating compaction restore, archive restore, or subagent transcript fork as the whole solution.
