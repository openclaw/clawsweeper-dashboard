## Summary

Keep open. Current main has important partial mitigations for media tasks, diagnostic recovery, and subagent completion delivery, but the core reported failure mode is still not clearly closed: subagent runs still enter a single global `subagent` lane with default concurrency 8, and source/tests explicitly do not treat that cap as a per-parent spawn throttle.

## What This Changes

The issue reports gateway event-loop saturation, 1012 restarts, and lost subagent work when heavy subagents or long-running tool work run concurrently.

## Best Possible Solution

Profile current main under the reported heavy subagent fan-out and then make subagent startup/completion work scheduler-aware, bounded, or offloaded while preserving liveness recovery and delivery guarantees.

## Reproduction Assessment

No, there is not a high-confidence live current-main reproduction from this review. The production logs are concrete and the current source still permits the reported concurrent path, but I did not run the three-heavy-model fan-out on a gateway here.

## Solution Assessment

Unclear because no patch is under review. The safest direction is profiling-led scheduler/offload work; the current partial fixes are useful but do not prove the original restart path is resolved.
