## Summary

Keep open. Current main still allows the reported parallel native subagent burst: the default subagent lane cap remains 8, the Gateway applies that cap directly to the shared `subagent` lane, and `spawnSubagentDirect()` still dispatches child runs to that lane. The related draft PR #75601 is not merged and needs maintainer review/profiling before this can be treated as fixed.

## What This Changes

The issue requests fixing Gateway event-loop saturation and 1012 service restarts when several heavy `sessions_spawn` subagents are started in parallel.

## Best Possible Solution

Profile packaged v2026.4.29 and current main under the reported three-way `sessions_spawn` workload, then keep Gateway RPC/liveness responsive with the narrowest measured mitigation: a bounded subagent startup/model-prep queue, targeted yields/caching, worker or process isolation, or a supervisor/liveness adjustment if the restart path is the real fault. Review #75601 against that evidence before deciding whether to land, revise, or replace it.

Unclear. The issue gives a high-quality reproduction recipe and logs, and current code confirms the parallel burst is still allowed, but I did not live-run the Ollama `deepseek-v4-pro:cloud` workload against a Gateway to reproduce the 5s event-loop delay and 1012 close end to end.
