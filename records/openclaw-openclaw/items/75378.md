## Summary

Keep open. Current main still permits the reported three-subagent burst on the shared in-process subagent lane, and agent startup/model resolution still runs inside the gateway process. Recent provider/runtime deferral changes are adjacent performance work, but they do not add a subagent startup queue, worker offload, or proof that the reported 1012 restart path is gone.

## What This Changes

The issue reports gateway event-loop saturation and 1012 restart when three heavy thread-bound `sessions_spawn` subagents start in parallel, requesting nonblocking model startup, spawn throttling, better diagnostics, and tunable stuck-session thresholds.

## Best Possible Solution

Profile packaged v2026.4.29 and current main under the reported parallel `sessions_spawn` workload, identify the blocking startup phase, then keep Gateway RPC/liveness responsive by adding a bounded subagent startup/model-prep queue, targeted yielding, caching, or worker/process isolation at the measured hotspot.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, and source review confirms current main still permits the reported three-way subagent burst, but I did not have a live Ollama cloud/gateway setup to reproduce the event-loop delay and 1012 restart end to end.

Not applicable because there is no patch. The proposed directions are plausible, but the best fix should first distinguish gateway prep starvation from provider/model backend latency and then throttle, yield, cache, or offload only the responsible phase.
