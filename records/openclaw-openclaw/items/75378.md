## Summary

Keep open. The report is current, detailed, and now independently corroborated, while current main still allows the reported subagent fan-out and still runs the relevant startup and completion bookkeeping inside the Gateway process without a measured fix for the saturation path.

## What This Changes

The issue requests preventing parallel `sessions_spawn` subagents from saturating the Gateway event loop and causing 1012 service-restart disconnects under heavy model/subagent completion load.

## Best Possible Solution

Keep the issue open and land a measured Gateway reliability fix that keeps RPC, heartbeats, and liveness responsive during subagent fan-out: profile the reported workload, then bound or isolate the specific hot phase with a subagent startup/completion queue, targeted event-loop yields, async session-store work, worker/process offload, or a supervisor/liveness adjustment if that is the actual restart trigger.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path for the released build: run the reported three parallel `sessions_spawn` calls with `mode="session"`, `thread=true`, and a heavy Ollama `:cloud` model, then watch Gateway liveness, `agent.wait`, and 1012 close behavior. I did not live-run it against current main, but current code review shows the same concurrency and completion paths remain available.
