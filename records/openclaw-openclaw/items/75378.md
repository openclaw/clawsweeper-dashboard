## Summary

Keep open. The report is current, concrete, and current main still permits the reported three-way native subagent burst: subagents default to an eight-wide shared gateway lane, child runs are dispatched directly onto that lane, and embedded-run plugin/model preparation still happens in the gateway process. There is no patch or merged follow-up that clearly fixes this workload yet.

## What This Changes

The issue requests that parallel native subagent spawning with heavy models avoid saturating the gateway event loop, timing out child runs, or causing 1012 service-restart closes.

## Best Possible Solution

Profile v2026.4.29 and current main under the reported three-way sessions_spawn workload, identify the blocking startup/model-prep phase, then keep gateway RPC/liveness responsive with the narrowest measured mitigation: bounded subagent startup/model-prep concurrency, targeted async yielding/caching, offloading, or a supervisor/readiness change if the restart path is the actual fault.

## Reproduction Assessment

Unclear. The issue gives a high-quality reproduction recipe and logs, and source review confirms current main still permits the reported burst, but I did not live-run the Ollama deepseek-v4-pro:cloud workload to reproduce the 5s event-loop delay and 1012 close end to end.

## Solution Assessment

Not applicable because there is no patch. The proposed queue/offload/config ideas are plausible, but the best fix should be selected after profiling so OpenClaw throttles or isolates the measured hotspot instead of broadly reducing useful parallelism.
