## Summary

Keep open. The report is recent, concrete, and independently corroborated, and current main still has the reported concurrency shape: three heavy subagents are below the default subagent lane cap, startup dispatch still uses the shared in-process subagent lane, and completion/capture/delivery still runs through Gateway-side work.

## What This Changes

The issue reports that spawning three heavy `sessions_spawn` subagents in parallel saturates the Gateway event loop, triggers liveness degradation, and closes clients with a 1012 service restart.

## Best Possible Solution

Keep this open until a profiled runtime fix keeps the Gateway responsive under a three-heavy-subagent burst while preserving subagent completion, cleanup, and delivery semantics.

## Reproduction Assessment

Unclear. The issue gives concrete steps, logs, and an independent macOS confirmation, but I did not live-run the heavy three-subagent workload against current main during this read-only review.

## Solution Assessment

Not applicable because there is no current patch. A one-off yield before `agent` dispatch may help one startup slice, but the safer direction is a profiled throttle/offload or completion-path change that addresses the measured Gateway event-loop saturation end to end.
