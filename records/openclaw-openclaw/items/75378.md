## Summary

Keep open. The report is current, concrete, and not fixed by current main: subagent starts still share the gateway `subagent` lane with a default concurrency of 8, child runs are dispatched directly onto that lane, and the embedded runner still performs runtime plugin/model preparation inside the gateway process. The exact hot phase still needs profiling before choosing a throttle, offload, or liveness/supervisor change.

## What This Changes

The issue reports that spawning three heavyweight thread-bound session subagents in parallel can saturate the gateway event loop, trigger liveness warnings, close WebSocket clients with 1012 service restart, and crash the subagent runs.

## Best Possible Solution

Profile v2026.4.29 and current main under the reported three-way `sessions_spawn` workload, identify the blocking startup phase, then keep Gateway RPC/liveness responsive with the narrowest measured fix: a bounded subagent startup/model-prep queue, targeted yields/caching, worker or process isolation, or a supervisor/liveness adjustment if the restart path is the real fault.

## Reproduction Assessment

Unclear. The issue gives a high-quality reproduction recipe and logs, and code review confirms current main still permits the reported parallel subagent burst, but I did not live-run the Ollama `deepseek-v4-pro:cloud` workload against a gateway to reproduce the 5s event-loop delay and 1012 close end to end.

## Solution Assessment

Not applicable because there is no patch. The proposed queue/offload/config directions are plausible, but the best fix should be chosen after profiling the reported path so OpenClaw throttles or isolates only the measured hotspot.
