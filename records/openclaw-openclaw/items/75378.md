## Summary

Keep open. Current main has a small Gateway `agent` accepted-ack delay, but it still permits the reported three-way subagent burst under the default subagent lane cap and still runs startup, wait, history read, session-store, and completion delivery work through the Gateway process. The independent macOS confirmation points at subagent completion processing as an active bottleneck, so this is not safe to close as implemented or stale.

## What This Changes

The issue asks OpenClaw to keep the Gateway responsive when multiple heavy `sessions_spawn` subagents start and complete in parallel, avoiding event-loop saturation, 1012 service restarts, and lost subagent work.

## Best Possible Solution

Keep the issue open and profile the reported workload on both v2026.4.29 and current main, then keep Gateway RPC/liveness responsive with the narrowest measured change: a bounded subagent startup/model-prep queue, offloaded or chunked completion/session-store work, targeted yielding/caching, or a supervisor/liveness adjustment only if restart policy is the actual failure amplifier.

## Reproduction Assessment

Yes, at the report level. The issue provides a concrete three-subagent `sessions_spawn` recipe, exact model/version/environment, and logs, and a second user confirmed comparable macOS completion stalls; the remaining gap is live profiling rather than missing reproduction data.

## Solution Assessment

Not applicable because there is no proposed patch on this issue. Current main’s accepted-ack delay is useful but not a complete solution for the reported startup burst or the confirmed completion-processing stalls, so the best fix should be selected after profiling.
