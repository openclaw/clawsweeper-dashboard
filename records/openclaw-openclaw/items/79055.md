## Summary

Keep open: current main still has static startup agent context, consult-time fast context, and direct realtime Twilio stream setup, but no async per-call preload hook before the first realtime model turn. The recent Discord/active-memory comment reinforces this as a broader lifecycle/API design request, not conservative cleanup.

## What This Changes

Add an async, timeout-bounded pre-answer hook for realtime voice calls that can load caller/contact context and inject sanitized per-call instructions before the first model turn.

## Best Possible Solution

Design an additive Voice Call or realtime/plugin-SDK lifecycle seam that receives bounded call metadata, runs under a hard timeout, injects only sanitized per-call context into realtime instructions, fails open, logs redacted status, and ships with focused config, docs, and handler/runtime tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source inspection on current main gives a high-confidence path showing startup agentContext and consult-time fastContext exist, but no async per-call preload happens before the first realtime turn.

## Solution Assessment

Yes in direction, but not as an automated bug fix. A first-class preload hook is the maintainable alternative to compiled-runtime patching, while the API placement, timeout budget, privacy contract, and cross-channel scope need maintainer approval first.

## Risks / Open Questions

- A pre-answer hook can add answer/setup latency for Twilio or other realtime transports, so timeout and fail-open behavior need explicit API design.
- Caller/contact/property context can contain private data, so sanitizer, prompt-injection framing, and log redaction rules need maintainer review.
- The right owner boundary is unresolved: the seam could live in the Voice Call plugin, realtime-voice SDK, generic plugin hooks, or a narrower active-memory integration.
- The Discord side-agent comment suggests related non-voice demand, so a voice-only fix may leave a broader lifecycle gap if maintainers want one shared contract.
