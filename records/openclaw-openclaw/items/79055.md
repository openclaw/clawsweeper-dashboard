## Summary

Keep open: current main still has setup-time realtime instructions and consult-time fast context, but no supported async pre-answer hook that enriches an inbound realtime call from caller metadata before the first spoken turn. The linked outbound objective and first-turn latency items are adjacent, not replacements for this inbound caller/contact preload API.

## What This Changes

The issue requests a supported async pre-answer preload hook so realtime voice calls can load inbound caller/contact context before the first model turn.

## Best Possible Solution

Design a narrow Voice Call or realtime plugin lifecycle seam that receives authenticated per-call metadata, enforces a hard timeout and redaction/logging rules, scopes the returned context to the call/session, and falls through to normal connection on failure.

## Reproduction Assessment

Yes at source level: current main carries inbound From/To/direction into a stream token and then starts the realtime bridge with prebuilt instructions, with no supported async caller-context preload hook before the first spoken turn. No live call was run, so this is not a full live reproduction.

## Solution Assessment

Unclear as an implementation plan: the stable hook direction is maintainable, but maintainers need to choose whether it belongs in Voice Call, the realtime voice SDK, or a broader pre-reply lifecycle. A safe solution needs timeout, fail-open behavior, redaction, logging, and call-scoped context semantics before code starts.
