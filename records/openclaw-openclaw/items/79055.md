## Summary

Keep open: current main still has realtime voice context only as startup instructions, static agent context, and consult-time fast context; it does not expose the requested async, fail-open per-call preload hook before the first realtime model turn. The related open Voice Call PR is useful context, but it does not close this issue and already has caller-identity/security concerns that require maintainer review.

## What This Changes

Request an async pre-answer hook for realtime voice calls that loads sanitized caller/contact context before the first model turn.

## Best Possible Solution

Design an additive Voice Call or plugin-SDK lifecycle seam that receives bounded call metadata, runs under a hard timeout, injects sanitized session-scoped context into realtime instructions before the first turn, logs only redacted status, and documents the caller-identity trust model.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing contract. Source inspection against current main shows the requested lifecycle point is absent from config, TwiML handling, realtime session setup, and provider contracts.

## Solution Assessment

Yes in direction, but not as an automated bug fix. The maintainable path is a first-class lifecycle seam, while the exact API, timeout, privacy, and caller-identity semantics need maintainer/security review.

## Risks / Open Questions

- A pre-answer hook can delay Twilio answer/stream setup, so timeout, cancellation, and fail-open behavior need explicit design.
- Caller/contact/property context may contain private data, so sanitizer, log-redaction, retention, and provider exposure rules need review.
- Caller-ID-based matching is low assurance in current Voice Call docs; private context must not be treated as verified identity without a stronger trust model.
- The existing open PR covers a narrower caller-context approach and already has security/storage/docs blockers, so it should not be treated as the final design.
