## Summary

Keep open: the reported race is real and the PR targets the right runtime area with credible after-fix proof, but the current boolean guard creates a restart regression because it is set on stop and never reset or generation-scoped.

## What This Changes

Adds a stopped guard to native approval delivery so in-flight entries are dropped or unbound after handler stop, with a regression test for the bindPending-to-activeEntries race.

## Best Possible Solution

Keep the in-flight cleanup fix, but make the stop guard lifecycle-aware, preferably with a per-stop generation token so old deliveries are dropped without poisoning future starts on the same handler.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: delivery awaits occur before activeEntries insertion while onStopped clears that map. The linked issue and PR also provide deterministic stub-plugin before/after counts, though I did not rerun the E2E in this read-only review.

## Solution Assessment

No, not as written. The runtime-level guard is the right direction, but a permanent stopped boolean is not the narrowest safe fix because the exported handler supports stop/start reuse; a generation-scoped guard avoids the leak without breaking later deliveries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset stopped guard across handler restarts:** `src/infra/approval-handler-runtime.ts:445`
  - body: The new closure flag is set to true in onStopped and is never cleared. The handler returned here is the same start/stop runtime exported through the plugin SDK, and createExecApprovalChannelRuntime.start() can be called again after stop; after such a restart every delivery will hit this guard and return null, so approvals silently stop being presented. Use a per-lifecycle generation or reset the guard for a new start while still rejecting deliveries from the stopped generation.
  - confidence: 0.86
