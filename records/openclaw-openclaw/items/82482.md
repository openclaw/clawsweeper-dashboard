## Summary

Keep open: this is a focused bug fix with sufficient real-behavior proof, but the patch still has a merge-blocking lifecycle bug because the new stopped flag stays latched after any stop.

## What This Changes

The PR adds a stopped-state guard around native channel approval delivery and a unit test for stopping during bindPending.

## Best Possible Solution

Keep the in-flight cleanup behavior, but make the stopped state start/stop-generation scoped or reset it when the runtime starts again, with regression coverage for restart reuse.

## Reproduction Assessment

Yes. Source inspection shows the current-main race window, and the PR/linked issue provide deterministic stub-plugin steps plus before/after sideband logs; I did not execute the setup in this read-only review.

## Solution Assessment

No, not as written. The guard is the right narrow fix shape for the leak, but it must not permanently disable later deliveries on the same handler object after a stop/start cycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset the stopped guard for later starts:** `src/infra/approval-handler-runtime.ts:445`
  - body: This flag is set in onStopped but is never cleared. The approval runtime supports start() after stop(), so after one stop every later delivery on the same handler would return null or immediately unbind instead of registering a pending approval. Make the guard generation-scoped or reset it when the runtime starts again.
