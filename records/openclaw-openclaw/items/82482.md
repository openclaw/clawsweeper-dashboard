## Summary

Keep this PR open: the reported leak is credible and the patch is pointed at the right area, but the one-way stopped flag would make a reused approval handler silently stop delivering approvals after its first stop.

## What This Changes

This PR adds a stopped-state guard around in-flight channel approval delivery and a regression test for stopping between bindPending and activeEntries insertion.

## Best Possible Solution

Keep the in-flight cleanup behavior, but implement it with a stop generation or a lifecycle reset so the same handler can deliver approvals after a stop/start cycle.

## Reproduction Assessment

Yes. Source inspection shows the current main race window around deliverPending, bindPending, activeEntries.set, and onStopped; the PR body also supplies before/after real-environment sideband counts for the same path.

## Solution Assessment

No, not as written. The guard should be tied to a specific stop generation or reset on start, instead of permanently latching every future delivery into the stopped path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reset the stopped guard for later starts:** `src/infra/approval-handler-runtime.ts:445`
  - body: This flag is set in onStopped but is never cleared. The returned approval handler uses the normal start()/stop() runtime surface, where stop() makes the runtime startable again; after one stop, every later delivery from the same handler would return null or immediately unbind instead of registering a pending approval. Please make the guard generation-scoped, or reset it when the runtime starts, so restart reuse still works.
  - confidence: 0.86
