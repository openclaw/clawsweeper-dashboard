## Summary

Keep open: the reported approval-delivery race is real on current main, and this PR is close, but the branch still has a stopped-delivery cleanup gap after a native entry has already been created.

## What This Changes

The PR adds a stopped-state guard to channel approval delivery and a regression test for stop during an in-flight native approval binding.

## Best Possible Solution

Keep the focused in-flight cleanup fix, but make stopped delivery cleanup handle entries that have already been created before bindPending and cover that exact window with a deterministic regression test.

## Reproduction Assessment

Yes. Current-main source shows the await/cleanup race, and the linked issue plus PR body provide deterministic stub-plugin before/after logs; the additional review finding is source-reproducible from Matrix registering reaction state in deliverPending before bindPending.

## Solution Assessment

No, not as written. The stopped guard is the right narrow shape for the bindPending-window leak, and the author correctly identified that production bootstrap recreates handlers, but the early return after deliverPending drops already-created native state without cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clean up entries delivered before bind:** `src/infra/approval-handler-runtime.ts:515-521`
  - body: Returning here drops an entry that `deliverPending` already created without giving the native runtime a cleanup pass. Matrix registers its reaction target during `deliverPending` before `bindPending`; if stop lands in this window, the entry is not bound, unbound, finalized, or tracked, so native reaction state can remain after stop. Route delivered entries through a cleanup path instead of returning `null` as soon as `stopped` is observed.
  - confidence: 0.88
