## Summary

Keep open: current main still supports bundle-declared LSP defaults and configured stdio LSP startup, but not workspace language detection, PATH/server discovery, install/update flows, user overrides, or an LSP status/diagnostics dashboard. The related diagnostics proposal is still open, and prior/open related PRs do not provide a clean canonical implementation of this broader product/API work.

## What This Changes

Request automatic LSP server detection, discovery, optional installation, smart configuration, overrides, updates, and diagnostics for common languages.

## Best Possible Solution

Define a scoped LSP detection and lifecycle contract with explicit install authority, documented override precedence, diagnostics/status events, and clear plugin or bundle ownership before implementing phased language defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature proposal. The current-main gap is source-verifiable from the bundle-backed loader, runtime, docs, and absence of a standalone LSP detection surface.

## Solution Assessment

No, not as a direct broad core auto-installer. The maintainable path needs product/API/security design for detection, install permission, overrides, diagnostics, and plugin-boundary ownership before implementation.
