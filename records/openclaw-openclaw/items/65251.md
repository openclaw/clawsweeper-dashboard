## Summary

Keep open. Current main has bundle-declared LSP defaults and an embedded stdio LSP runtime, but it still does not implement the requested zero-configuration workspace scan, package-manager detection, PATH/server discovery, optional install/fix flow, override semantics, update notices, or LSP health dashboard.

## What This Changes

Request automatic detection, configuration, optional installation, and diagnostics for LSP servers based on project files and existing developer environment clues.

## Best Possible Solution

Keep this as a product/API design item. The maintainable end state should add explicit, safe workspace detection and installed-server discovery, keep installation and repair opt-in, define user override semantics, expose LSP status through CLI/UI diagnostics, and use a documented generic plugin or bundle API where language-specific behavior should not live in core.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-main gap is verifiable by source and docs: LSP support is loaded from enabled bundle configuration and does not inspect workspace files, package manifests, editor configs, or PATH.

## Solution Assessment

No, the current bundle-backed LSP implementation is not the complete solution requested here. A safer solution needs scoped design for detection, install authority, overrides, diagnostics, and plugin-boundary ownership rather than ad hoc core heuristics.
