## Summary

Keep open: current main still only supports bundle-declared LSP defaults and configured stdio LSP startup. Workspace language detection, PATH/server discovery, install flows, overrides, updates, and user-visible diagnostics/status are still broad feature and product-design work, with the related diagnostics proposal still open at https://github.com/openclaw/openclaw/issues/65252.

## What This Changes

Request automatic LSP server detection, configuration, installation suggestions, overrides, updates, and status based on workspace files, package manifests, PATH, editor clues, and language heuristics.

## Best Possible Solution

Define a scoped LSP detection/lifecycle contract with explicit install authority, documented overrides, diagnostics/status events, and clear plugin or bundle ownership before implementing phased language defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature proposal. The current-main gap is source-verifiable from the bundle-backed loader, runtime, docs, and absence of a standalone LSP detection surface.

## Solution Assessment

No, not as a direct core auto-installer. The maintainable path needs product/API/security design for detection, install permission, overrides, diagnostics, and plugin-boundary ownership before implementation.
