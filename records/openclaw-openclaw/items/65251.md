## Summary

Keep open. Current main still supports static bundle-declared LSP defaults and a configured-server runtime, but it does not implement workspace language scanning, project-file/package-manager detection, PATH discovery, opt-in server installation/repair, update notices, or a dashboard flow.

## What This Changes

The issue proposes automatic workspace language/LSP server detection, optional install/fix flows, smart configuration, and LSP diagnostics/status UI.

## Best Possible Solution

Design a scoped LSP control plane that keeps language-specific defaults plugin-owned, adds any missing generic plugin/API seams, and makes installation/repair explicit, reversible, and diagnosable.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-main gap is verifiable from source and docs: LSP servers are loaded from enabled bundle configuration and spawned only from configured server entries.

## Solution Assessment

No, not as a direct all-in-core patch. The maintainable path is a design-backed API/UX split for detection, discovery, install authority, overrides, and diagnostics before implementation.
