## Summary

Keep open. Current main has bundle-declared LSP defaults, but it still does not provide the requested zero-configuration project scan, PATH-based server discovery, optional install/fix flow, user override model, update notices, or LSP health dashboard.

## What This Changes

The issue requests automatic LSP server detection, configuration, optional installation, and diagnostics based on workspace files and common language/project signals.

## Best Possible Solution

Keep this as a product/API design item. The maintainable end state should add safe workspace detection and PATH discovery, keep installs and one-click repairs explicitly opt-in, define user override precedence, expose LSP health through CLI/UI diagnostics, and add a generic plugin or bundle API if language-specific detection should live outside core.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-main gap is verifiable from source and docs: LSP support is loaded from enabled bundle declarations, returns no tools when no servers are configured, and has no documented project scan, PATH discovery, install, or dashboard path.

## Solution Assessment

No, current bundle-backed LSP support is not the full solution for this request. The best path is a scoped API and product design that separates safe detection, explicit install authority, overrides, and diagnostics instead of adding ad hoc core heuristics.
