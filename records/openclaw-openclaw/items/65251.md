## Summary

Keep open: current main still only supports static bundle-declared LSP defaults and configured stdio server startup, while workspace detection, PATH/server discovery, install flows, overrides, updates, and dashboard UX remain new feature/product work.

## What This Changes

Request automatic detection, configuration, optional installation, and status UX for LSP servers based on workspace files and installed tools.

## Best Possible Solution

Keep this as a maintainer design item for a scoped LSP detection/lifecycle API, explicit install authority, documented overrides, diagnostics/status events, and clear plugin or bundle ownership for language defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature proposal. The current-main gap is source-verifiable from the bundle-backed loader, runtime, docs, and absence of a standalone LSP detection surface.

## Solution Assessment

No, a direct core auto-installer should not be assumed as the best fix. The maintainable path needs product/API/security design for detection, install permission, overrides, diagnostics, and plugin-boundary ownership before implementation.
