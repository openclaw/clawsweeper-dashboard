## Summary

Keep open: current main still provides static bundle-declared LSP defaults and startup for configured stdio servers, but the requested workspace detection, PATH discovery, install/update flow, override model, and dashboard/status UX remain broad product and API work.

## What This Changes

Request automatic detection and configuration of LSP servers from project files, installed tools, language heuristics, and editor clues, with optional install/fix flows and diagnostics.

## Best Possible Solution

Keep this as a maintainer design item for a scoped LSP detection/lifecycle API with explicit install authority, documented overrides, diagnostics/status events, and plugin or bundle ownership for language defaults.

## Reproduction Assessment

Not applicable; this is a feature proposal rather than a bug report. The current-main gap is source-verifiable from the static bundle-backed LSP loader/runtime, docs, tests, and absence of a standalone detection/install lifecycle.

## Solution Assessment

No direct core auto-installer should be assumed as the best fix. The maintainable path needs product, API, and security design for detection, install permission, overrides, diagnostics, and plugin-boundary ownership before implementation.
