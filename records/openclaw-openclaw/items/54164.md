## Summary

Keep open. The reported sessions_send gateway freshness bypass appears fixed on current main and present in the latest release, but #57086 is still open, unmerged, and explicitly tied to this issue. Cleanup policy says the issue should stay open until that PR is reviewed, merged, closed as superseded, or split for any remaining node-event work.

## What This Changes

The issue asks for inter-agent sessions reached through sessions_send to apply normal session freshness evaluation so stale stored session IDs rotate at the daily or idle reset boundary.

## Best Possible Solution

Keep the shipped gateway freshness behavior for sessions_send, resolve #57086 by closing it as superseded or landing/splitting any remaining node-event fix, then close this issue as implemented. If maintainers want task-scoped inter-agent reset semantics beyond daily/idle freshness, track that as a separate product/API follow-up.

## Reproduction Assessment

Yes. The issue gives a concrete two-agent sessions_send path, and current regression tests model the stale daily-reset case with both stored and explicit stale session IDs.

## Solution Assessment

Yes for the reported sessions_send path: evaluating freshness in the gateway before choosing the resolved sessionId is the narrow maintainable fix. The remaining node-event sessionId forwarding covered by #57086 is adjacent and should be handled through that PR or a separate follow-up.
