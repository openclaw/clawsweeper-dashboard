## Summary

Keep open. Current main partially mitigates the reported registration-time provider discovery path, but the shared model-config resolver and image/video/music execution helpers still have the eager provider-listing shape, and the paired closing PR remains open.

## What This Changes

The issue asks for image/video/music generation tool model-config resolution to defer capability-provider listing when complete explicit modelConfig already exists.

## Best Possible Solution

Land #74097 or an equivalent main change that makes the shared media model-config resolver accept lazy provider discovery for image/video/music execution, while preserving provider lookup for incomplete config and explicit list/status actions.

## Reproduction Assessment

Yes. Current main still has a static reproduction path: the three media model-config resolvers call listRuntime*GenerationProviders before resolveCapabilityModelConfigForTool can return a complete explicit modelConfig.

## Solution Assessment

Yes for the remaining code path. A provider-list thunk is the narrowest maintainable fix because it avoids only the unnecessary explicit-config lookup and preserves the existing fallback behavior when config is incomplete.
