## Summary

Keep open. Current main has partly mitigated the original registration-time path by making generation-tool availability use lazy provider discovery, but the shared model-config resolver and the image/video/music execution resolvers still materialize provider lists before an explicit complete modelConfig can short-circuit. The paired closing PR #74097 is still open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests deferring image/video/music capability-provider listing when an explicit complete tool modelConfig is already sufficient, avoiding unnecessary plugin-registry reload work.

## Best Possible Solution

Land #74097 or an equivalent main change that makes the shared media model-config resolver accept lazy provider discovery for image, video, and music execution, while preserving provider lookup for incomplete config and explicit list/status actions.

## Reproduction Assessment

Yes. Static reproduction on current main is high confidence: the three media model-config resolvers call listRuntime*GenerationProviders before resolveCapabilityModelConfigForTool can return a complete explicit modelConfig. The original registration-time availability path is partly mitigated, so the remaining reproduction is the execution/model-config resolver path.

## Solution Assessment

Yes. A provider-list thunk is the narrowest maintainable fix for the remaining path because it avoids only the unnecessary explicit-config lookup while preserving existing fallback behavior when config is incomplete.
