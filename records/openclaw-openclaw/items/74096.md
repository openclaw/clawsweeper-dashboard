## Summary

Keep open. Current main partially mitigates the reported registration-time provider discovery by making generation-tool availability checks lazy, but the issue is still paired with open, unmerged PR #74097 using closing syntax, and the shared model-config resolver still accepts an eagerly computed provider array in current main.

## What This Changes

Avoid eagerly listing image/video/music generation capability providers when complete explicit model config lets the media tool resolve without provider discovery.

## Best Possible Solution

Keep this issue open while maintainers review or supersede PR #74097. The desired end state is that media generation registration and execution avoid provider discovery when complete explicit tool model config is already available, while preserving provider lookup for incomplete config and explicit list/status actions.

## Reproduction Assessment

Yes. The registration path can be reproduced with a providers thunk spy and complete explicit model config, and current tests cover that short-circuit; the remaining eager execution helper is directly visible in source.

## Solution Assessment

Yes for the remaining code path. Making the shared resolver accept lazy provider discovery is the narrowest maintainable completion, provided it is reconciled with current main's existing registration-time availability thunk and the provider-runtime cache mitigation stays separate.
