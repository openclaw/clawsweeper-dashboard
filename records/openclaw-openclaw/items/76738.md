## Summary

Keep this PR open: current main does not already contain the new opt-in reaction tracking, and the PR has a concrete P2 blocker where Discord retargets status reactions before the requested reaction is known to have succeeded. The protected `maintainer` label also means this is not a cleanup-close candidate.

## What This Changes

The PR adds `trackToolCalls`/`track_tool_calls` to the message reaction tool, forwards sanitized tool-start args, retargets Discord status reactions to an opted-in reacted message, updates tool emoji resolution, docs, tests, and the changelog.

## Best Possible Solution

Keep the feature opt-in, but defer retargeting until the reaction target is confirmed usable or restore the original inbound status controller when the tracked reaction cannot be applied, with a Discord regression test for that failure path.

## Reproduction Assessment

Yes for the review finding: the PR source shows `onToolStart` calls the binding path before the reaction tool result is known, then swaps and clears the controller. A focused test can simulate a failed tracked reaction target and assert progress/done reactions remain on the original inbound message.

## Solution Assessment

No: the schema, docs, and sanitized-args wiring are reasonable, but switching controllers before the reaction succeeds is not the safest implementation. The narrower maintainable fix is to prove the tracked target first or fall back to the original status controller on first tracked-target failure.
