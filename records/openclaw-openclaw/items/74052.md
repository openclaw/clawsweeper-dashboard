## Summary

Keep open. Current main still has the inherited/default-account-before-agent-binding behavior this PR targets, and the provided diff is focused with no concrete functional or security blocker found.

## What This Changes

The PR updates the message tool and outbound message-action runner so omitted `accountId` resolves as explicit param first, calling-agent binding second, inherited/default account last, with regression coverage for the combined default-plus-binding case.

## Best Possible Solution

Land this PR or an equivalent narrow shared outbound change after CI confirms the targeted regression and typecheck lanes. The desired end state is explicit `accountId` remains authoritative, omitted `accountId` prefers the calling agent's configured channel/account binding, and inherited/default account is only the final fallback.

## Reproduction Assessment

Yes. On current main, call `runMessageAction` with both `defaultAccountId` and a different configured `bindings` account for `agentId`, while omitting `params.accountId`; the current runner selects the default before checking the binding.

## Solution Assessment

Yes. The proposed patch is the narrowest maintainable fix I found: it preserves explicit params, stops the wrapper from manufacturing an explicit `accountId` from inherited context, moves default fallback after binding lookup, and adds focused regression coverage.
