## Summary

Keep open. The PR is aimed at the right outbound-account precedence bug, but the force-pushed head still resolves message-tool SecretRefs against the inherited account before the runner can select the calling agent's bound account, and it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes the message tool and outbound message-action runner so omitted `accountId` prefers the calling agent's channel binding before inherited/default account context, with runner regression coverage.

## Best Possible Solution

Keep the narrow precedence change, but select the same preferred bound account before message-tool SecretRef scoping or defer account-scoped SecretRef resolution until after account selection, then add a `CHANGELOG.md` fix entry.

## Reproduction Assessment

Yes. A high-confidence repro is `createMessageTool` with inherited `agentAccountId: "parent-account"`, a calling-agent binding to `account-b`, account-scoped channel SecretRefs for both accounts, and omitted `params.accountId`; the PR can dispatch as `account-b` after resolving only the inherited account's SecretRefs.

## Solution Assessment

No, not as currently patched. The runner priority change is the right direction, but the tool setup phase must use the same selected account for SecretRef scoping or local channel actions can fail in the exact multi-account case being fixed.
