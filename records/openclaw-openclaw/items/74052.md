## Summary

Keep open: the PR fixes the runner precedence path for the linked bug, but it still lets message-tool SecretRef scoping run with the inherited account before the bound account is selected, and it is missing the required user-facing changelog entry.

## What This Changes

The PR changes message-tool outbound account resolution so omitted `accountId` prefers the calling agent's channel binding over inherited/default account context, with runner regression coverage.

## Best Possible Solution

Keep the precedence change, but select the same preferred account before command-time SecretRef scoping or defer account-scoped SecretRef resolution until after runner account selection, then document the fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main reproduces with omitted `params.accountId`, `defaultAccountId: "parent-account"`, and a calling-agent binding to `account-b`; the PR-specific gap reproduces through `createMessageTool` with account-scoped SecretRefs for inherited and bound accounts.

## Solution Assessment

No. The runner change is the right narrow direction, but the patch is incomplete until message-tool setup resolves scoped secrets for the same account that dispatch will use and the changelog is updated.
