## Summary

Keep open. The PR targets a valid outbound account-selection bug and the runner precedence change is the right direction, but the patch is not ready to merge because the message tool still scopes command-time secret resolution to the inherited account before the new binding-based account is selected. The branch also lacks the required user-facing changelog entry.

## What This Changes

The PR changes the message tool and outbound message-action runner so explicit accountId stays authoritative while omitted accountId prefers the calling agent's configured channel/account binding before inherited/default account, with focused runner regression tests.

## Best Possible Solution

Keep the narrow account-precedence direction, but finish the implementation so the account chosen for omitted message-tool calls is also the account used for command-time channel secret scoping. Then add the required changelog entry and land after the targeted runner/tool tests and exact-head CI are green or clearly unrelated.

## Reproduction Assessment

Yes. Current main can be reproduced statically by calling runMessageAction with params.accountId omitted, defaultAccountId set to a parent account, and cfg.bindings mapping the calling agent to a different account; line 942 selects the default before the binding. The PR-specific gap can be reproduced through createMessageTool with account-scoped SecretRefs for parent and bound accounts, because secret allowedPaths are chosen before the runner selects the bound account.

## Solution Assessment

No, not as currently patched. The runner priority change is the right core fix, but the message-tool config/secret-resolution phase must use the same preferred account or local channel actions such as Slack can dispatch under an account whose account-scoped secrets were not resolved.
