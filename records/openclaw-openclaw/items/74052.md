## Summary

Keep open. The PR addresses a real current-main outbound account-precedence bug, but the patch is not mergeable yet because the message tool can still resolve channel SecretRefs for the inherited account before the runner later dispatches under the bound account; it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes `createMessageTool` and `runMessageAction` so explicit message `accountId` values stay authoritative while omitted accounts prefer the calling agent's channel/account binding before inherited/default account, with focused runner regression cases.

## Best Possible Solution

Keep the account-precedence direction, but make the selected account a shared decision used by both command-time secret scoping and `runMessageAction`, then add targeted SecretRef regression coverage and the required Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction is a `createMessageTool` call with omitted `params.accountId`, inherited `agentAccountId` set to a parent account, a binding from the calling agent to another Slack account, and account-scoped Slack SecretRefs; the PR still scopes secrets to the parent before the runner selects the bound account.

## Solution Assessment

No. The runner priority change is the right narrow core fix, but the current PR splits account selection from secret resolution and omits the required changelog entry; resolving the preferred account before SecretRef scoping or deferring account-scoped resolution is safer.
