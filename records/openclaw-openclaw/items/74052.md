## Summary

Keep open. Current main still has the account-precedence bug described by the linked issue, and this PR's runner-level direction is correct, but the patch is not ready because message-tool SecretRef scoping still uses the inherited/default account before dispatch can choose the calling agent's bound account. That leaves the exact omitted-account multi-account path able to dispatch under an account whose scoped credentials were not resolved, and the user-facing fix also needs a changelog entry.

## What This Changes

The PR changes outbound message account selection so omitted message-tool accountId prefers the calling agent's channel/account binding over inherited default account while preserving explicit overrides, with focused runner regression tests.

## Best Possible Solution

Keep the narrow precedence change, but make message-tool secret scoping and dispatch use the same resolved account choice for omitted accountId calls: explicit accountId first, calling-agent binding second, inherited/default account last. Add a message-tool regression for account-scoped SecretRefs plus the required changelog entry, then land this with the linked issue.

## Reproduction Assessment

Yes. Current main is reproducible by calling runMessageAction with params.accountId omitted, defaultAccountId set to a parent account, and cfg.bindings mapping the calling agent to another account; the runner selects the default before the binding. The PR-specific gap is reproducible through createMessageTool with account-scoped SecretRefs on parent and bound accounts, because allowedPaths are chosen from agentAccountId before the runner selects the bound account.

## Solution Assessment

No, not as currently patched. The runner priority split is the right core fix, but the maintainable solution needs the wrapper's command-time SecretRef scope to use the same selected account or a shared helper before dispatch.
