## Summary

Keep open: the PR targets a real outbound account-precedence bug, but the current patch can dispatch under the bound account after resolving only the inherited account's scoped SecretRefs, and it is missing the required changelog entry.

## What This Changes

The PR updates the message tool and outbound runner so an omitted `accountId` prefers the calling agent's channel binding before inherited/default account context, with regression coverage.

## Best Possible Solution

Keep the narrow precedence change, but select the same preferred bound account before message-tool SecretRef scoping or defer account-scoped SecretRef resolution until after account selection, then add an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. A high-confidence source repro is `createMessageTool` with inherited `agentAccountId: "parent-account"`, a calling-agent binding to `account-b`, account-scoped channel SecretRefs for both accounts, and omitted `params.accountId`; at the PR head the runner can dispatch as `account-b` after the tool resolved only `parent-account` SecretRefs.

## Solution Assessment

No, not as currently patched. The runner precedence change is the right direction, but the tool setup phase must use the same selected account for SecretRef scoping or local channel actions can fail in the exact multi-account case being fixed.
