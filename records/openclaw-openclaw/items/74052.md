## Summary

Keep open. The linked account-selection bug is still present on current main, and this PR is a useful narrow fix direction, but the latest diff still leaves message-tool SecretRef scoping tied to the inherited account before the runner can select the calling agent's bound account; it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes the message tool and outbound message-action runner so omitted `accountId` values prefer the calling agent's channel binding before falling back to inherited/default account context, while preserving explicit overrides and adding runner regression cases.

## Best Possible Solution

Keep the narrow precedence change, but make message-tool setup and runner dispatch agree on the same preferred account before account-scoped SecretRefs are resolved, or defer account-scoped SecretRef resolution until after selection. The final patch should keep explicit `accountId` authoritative, prefer the calling agent's bound channel account when omitted, fall back to inherited/default account last, and document the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main can be reproduced with `runMessageAction` using omitted `params.accountId`, `defaultAccountId: "parent-account"`, and a binding from the calling agent to `account-b`; source line 942 chooses the default before the binding. The PR-specific gap can be reproduced through `createMessageTool` with account-scoped channel SecretRefs for inherited and bound accounts, because secret `allowedPaths` are chosen before the bound account is selected.

## Solution Assessment

No, not as currently patched. The runner precedence change is the right narrow direction, but the tool setup phase must use the same selected account for SecretRef scoping or local channel actions can run with unresolved bound-account configuration.
