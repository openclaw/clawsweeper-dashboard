## Summary

Keep open; the PR targets a real current-main account-precedence bug, but the refreshed diff still leaves SecretRef scoping on the inherited account and omits the required user-facing changelog entry.

## What This Changes

The PR changes outbound message account resolution so omitted `accountId` prefers the calling agent's channel binding over inherited/default context, with regression coverage for runner dispatch.

## Best Possible Solution

Keep the precedence fix, but select the same effective bound account before account-scoped SecretRef resolution or defer that resolution until after selection, then add an Unreleased Fixes changelog entry for #73619.

## Reproduction Assessment

Yes. Current main reproduces with `runMessageAction` using omitted `params.accountId`, `defaultAccountId: "parent-account"`, and a binding from the calling agent to `account-b`; the PR-specific gap reproduces through `createMessageTool` with account-scoped SecretRefs for inherited and bound accounts.

## Solution Assessment

No, not as currently patched. The runner priority change is the right direction, but the tool setup phase must use the same selected account for SecretRef scoping or local channel actions can dispatch with unresolved bound-account config.
