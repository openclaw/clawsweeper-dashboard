## Summary

Keep open: the patch addresses part of the score-scale starvation, but it does not cover the linked `wiki_search corpus=all` path and can return fewer than `maxResults` when one corpus has spare capacity.

## What This Changes

The PR adds a changelog entry, a memory-core regression test, and changes `memory_search corpus=all` to cap memory and supplement results before sorting.

## Best Possible Solution

Apply one deterministic corpus-diversity merge helper or equivalent behavior to both `memory_search` and `wiki_search`, guarantee representation when both corpora have hits, and backfill sparse-side capacity without adding a config flag.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main raw-sorts wiki integer scores against memory cosine scores for all-corpus searches and then slices to `maxResults`. The linked issue also provides concrete tool calls, though this read-only review did not run the reporter's private vault.

## Solution Assessment

No. The direction is appropriate, but this PR is incomplete because it leaves `wiki_search corpus=all` unchanged and the half-cap can drop available results when one corpus has fewer hits.

## Review Findings

Overall correctness: patch is incorrect

  - body: The linked bug reports both `memory_search corpus=all` and `wiki_search corpus=all`, but this PR only changes `memory_search`.   - confidence: 0.92
- **[P2] Backfill unused capacity after per-corpus caps:** `extensions/memory-core/src/tools.ts:332-333`
  - body: With `maxResults=5`, one wiki hit, and five memory hits, the new slices keep only three memory rows plus one wiki row, returning four results despite enough matches. Fill unused capacity from the other corpus before the final slice so the cap prevents starvation without dropping available hits.
  - confidence: 0.89
