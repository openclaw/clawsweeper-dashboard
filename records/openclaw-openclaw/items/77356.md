## Summary

Keep open: the PR targets a valid source-backed bug, but the patch does not cover the linked `wiki_search corpus=all` path and the new cap can drop available results instead of filling spare capacity.

## What This Changes

The PR adds a changelog entry, a memory-core regression test, and a `memory_search corpus=all` merge change that caps memory and wiki supplement rows before sorting.

## Best Possible Solution

Apply the same deterministic corpus-diversity merge behavior to both `memory_search` and `wiki_search`, guarantee representation when both corpora have hits, and fill unused capacity from the other corpus without adding a config flag or plugin API change.

## Reproduction Assessment

Yes, for the underlying bug by source inspection: current main merges memory/wiki results for `corpus=all`, raw-sorts by incomparable `score` scales, and slices to `maxResults`. I did not run the reporter's private vault or tests because this review is read-only.

## Solution Assessment

No. The direction is appropriate, but the current PR is incomplete because it leaves `wiki_search corpus=all` unchanged and the per-corpus slice can under-fill results when one corpus has fewer than half the requested limit.
