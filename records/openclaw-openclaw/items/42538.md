## Summary

Current main now threads live channel runtime state into gateway health snapshots, and the same implementation is present in the latest v2026.5.22 release, so the reported WhatsApp health/status mismatch is no longer necessary to track separately.

## What This Changes

The issue asks for the health endpoint to report the same live WhatsApp running and connected state as channels.status.

## Best Possible Solution

Keep the shipped runtime-aware health snapshot implementation and close this issue as fixed in v2026.5.22.

## Reproduction Assessment

Yes. The original mismatch is source-reproducible from the old path described in the issue and duplicate review: health lacked runtime state while channels.status read context.getRuntimeSnapshot; I did not run a live WhatsApp account during this read-only review.

## Solution Assessment

Yes. Passing the prepared ChannelRuntimeSnapshot into getHealthSnapshot is the narrow maintainable fix because it reuses the gateway’s existing runtime source of truth without adding WhatsApp-specific core policy.
