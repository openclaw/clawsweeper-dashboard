## Summary

Close as implemented: current main now threads live channel runtime into health snapshots and refreshes cached health when runtime lifecycle differs, so the reported WhatsApp running/connected mismatch is covered by source, tests, history, and shipped release provenance.

## What This Changes

The issue requests that gateway health report WhatsApp running/connected from the same live channel runtime state used by channels.status.

## Best Possible Solution

Keep the shipped runtime-backed health snapshot and cache-freshness implementation; open a new report only if a current release still shows health/status parity failure with fresh output.

## Reproduction Assessment

No live WhatsApp current-main repro was run in this read-only review. The old failure path is source-backed, and current source plus regression tests show runtime fields now flow into health and stale cache entries refresh.

## Solution Assessment

Yes. The best fix is in the shared gateway health path: pass ChannelRuntimeSnapshot into getHealthSnapshot and refresh cached health when runtime lifecycle differs, rather than adding WhatsApp-specific status logic.
