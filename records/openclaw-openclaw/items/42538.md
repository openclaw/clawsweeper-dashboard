## Summary

Current main implements the requested health/runtime merge: gateway health refresh now passes the channel manager runtime snapshot into health snapshot generation, the snapshot builder merges per-account runtime state before WhatsApp reads `running` and `connected`, and the health RPC refreshes instead of serving cached data when runtime lifecycle fields diverge.

## What This Changes

Fix the gateway health response so WhatsApp `running` and `connected` match live channel runtime state instead of defaulting to false while `channels.status` reports true.

## Best Possible Solution

Keep the current runtime-backed gateway health path and close this issue; any future mismatch should be filed as a new, versioned report with current health JSON, `channels.status` output, and gateway diagnostics if it reproduces after this main-line fix.

## Reproduction Assessment

Yes. The original issue has concrete reproduction steps, and the code-level reproduction path is high confidence: without a runtime snapshot, WhatsApp summary defaults missing `running` and `connected` to false; current main now supplies that runtime before summary building.

## Solution Assessment

Yes. Threading the existing channel manager runtime snapshot through health refresh and using the shared channel snapshot builder is the narrow maintainable fix, and the added cache-divergence guard addresses stale cached false negatives without changing direct CLI channel ownership.
