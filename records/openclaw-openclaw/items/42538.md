## Summary

Close: current main and v2026.5.12 already wire channel runtime snapshots into health generation and cover the runtime-backed channel health path with regression tests. The recent note about https://github.com/openclaw/openclaw/pull/83081 is correct that that PR is distinct, but the reported health/status mismatch was fixed separately by https://github.com/openclaw/openclaw/pull/72417.

## What This Changes

The issue reports that the gateway `health` RPC returns stale `running` and `connected` values for WhatsApp while `channels.status` shows the live connected state.

## Best Possible Solution

Keep the generic runtime-backed gateway health implementation from https://github.com/openclaw/openclaw/pull/72417; no WhatsApp-specific health shim is needed.

## Reproduction Assessment

Yes for the original bug: the reporter supplied a concrete live WhatsApp sequence, and current regression tests exercise the same runtime-backed health snapshot path. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes. The merged fix updates the shared gateway health path to carry channel runtime state into health snapshots and refresh stale cache entries, which is narrower and more maintainable than a WhatsApp-only special case.
