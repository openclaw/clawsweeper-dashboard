## Summary

Current main and the v2026.5.20 release already thread live channel runtime state into the gateway health snapshot, so the reported WhatsApp health/status mismatch is implemented; the later health-path change is distinct from the config repair PR that a maintainer called out as not fixing this issue.

## What This Changes

The issue asks for the gateway health endpoint to report WhatsApp running and connected values from the same live runtime state used by channels.status.

## Best Possible Solution

Keep the shipped runtime-aware health snapshot implementation and close the issue as fixed in v2026.5.20.

## Reproduction Assessment

Yes for the originally reported behavior: the issue gives exact gateway calls and current tests/source show the mismatch was in the health snapshot path rather than channels.status. I did not run a live WhatsApp session in this read-only review, but the source path and regression tests are high-confidence.

## Solution Assessment

Yes. Threading the channel manager runtime snapshot into health generation and refreshing stale cached health when runtime differs is the narrow owner-boundary fix, without changing channels.status semantics.
