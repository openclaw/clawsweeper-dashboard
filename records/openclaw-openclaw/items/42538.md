## Summary

Current main and the `v2026.6.6` release now implement the requested health/status parity: gateway health refresh passes live channel runtime state into `getHealthSnapshot`, and the health snapshot builder merges `running` and `connected` before channel summaries are built. The earlier maintainer note correctly kept this open before that path changed, but the shipped source and regression coverage now address the reported mismatch.

## What This Changes

The issue asks for the gateway `health` endpoint to report the same live WhatsApp `running` and `connected` state that `channels.status` already reports.

## Best Possible Solution

Keep the shipped runtime-backed health snapshot implementation and close this report as fixed in the released gateway health path.

## Reproduction Assessment

Yes, source-level. The old path was documented in the duplicate review as building health snapshots without runtime state, and current main now has source and regression-test proof that live runtime `running` and `connected` values are merged into health output.

## Solution Assessment

Yes. Passing the gateway runtime snapshot through health refresh into the channel account snapshot builder is the narrow owner-boundary fix and matches the `channels.status` source of truth without adding config or product policy.
