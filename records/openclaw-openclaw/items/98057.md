## Summary

Keep open: this PR is still necessary because current main and the latest release still bind `feishu_chat` to the first enabled Feishu account, while the PR moves chat/member lookup onto the shared contextual account resolver with focused tests and redacted live/staged proof. I found no concrete patch defect, but this is a Feishu credential-routing boundary change that needs maintainer acceptance before merge.

## What This Changes

The PR changes `feishu_chat` registration from a static first-account tool to a context-built tool using `createFeishuToolClient`, and adds chat account-routing regression tests.

## Best Possible Solution

Land this as the narrow chat-account resolver alignment, or deliberately fold it into the broader Feishu routing PR, while keeping the broader non-inbound binding-precedence issue open until that surface is resolved.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.10` use `accounts[0]` for chat registration and client creation, so `ctx.agentAccountId` cannot affect `feishu_chat`. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes for the chat subset: reusing the existing Feishu tool-account resolver is the narrowest maintainable fix and matches sibling Feishu tools. It is not the full solution for the broader non-inbound account-binding issue tracked at https://github.com/openclaw/openclaw/issues/96657.
