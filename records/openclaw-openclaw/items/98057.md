## Summary

Keep open: current main and the latest release still route `feishu_chat` through the first enabled account, while this PR moves chat onto the existing contextual Feishu tool-account resolver with adequate proof. I found no blocking patch defect, but landing still needs maintainer acceptance of the Feishu chat credential-boundary change and should not close the broader non-inbound routing issue.

## What This Changes

The PR changes `feishu_chat` registration and execution so chat, members, and member_info actions use the shared Feishu tool-account resolver with contextual account routing instead of a registration-time first account.

## Best Possible Solution

Land this as the narrow chat-account resolver alignment if maintainers accept the credential-boundary change, or deliberately fold it into the broader Feishu routing PR while keeping the broader issue open.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.10` use `accounts[0]` for chat registration and client creation, so `ctx.agentAccountId` cannot affect `feishu_chat`. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the chat subset: reusing the existing Feishu tool-account resolver is the narrowest maintainable fix and matches sibling Feishu tools. It is not the full solution for the broader non-inbound binding-precedence issue tracked at https://github.com/openclaw/openclaw/issues/96657.
