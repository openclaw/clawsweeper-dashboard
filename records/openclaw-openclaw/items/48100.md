## Summary

Keep open: this is a real new core plugin-hook proposal, not already implemented on current main, but the branch is not ready because the hook bypasses the raw-conversation access gate, cannot reliably honor suppress/reinject before streaming delivery, and lacks real behavior proof.

## What This Changes

The branch adds an `after_agent_complete` plugin hook plus an agent-runner wrapper that can suppress delivery or reinject context after a completed agent turn.

## Best Possible Solution

Design the seam at the actual finalization or delivery boundary, explicitly define streaming behavior, apply existing prompt-injection and conversation-access policies, and land it with focused hook-runner and reply-runner coverage plus redacted real gateway proof.

## Reproduction Assessment

Yes for the blocking review issues: source inspection shows block replies can enqueue or send inside `runAgentTurnWithFallback` before the proposed hook runs, and the raw `response` event is not tied to the existing conversation-access gate. The broader feature itself is not a bug reproduction.

## Solution Assessment

No. Wrapping the whole agent turn after `runAgentTurnWithFallback` is not the narrowest maintainable solution because delivery, transcript, and permission semantics need to be resolved at the hook contract boundary first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate raw response access behind conversation policy:** `src/plugins/types.ts:1545`
  - body: The new event exposes the final assistant `response`, but the hook is only added to the hook-name list and prompt-injection gate. Current registry policy blocks raw conversation hooks via `CONVERSATION_HOOK_NAMES`; without adding this hook there, a non-bundled plugin can register it and read raw assistant content without `allowConversationAccess=true`.
  - confidence: 0.87
- **[P1] Run the hook before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:42-48`
  - body: The wrapper waits for `runAgentTurnWithFallback` to finish before consulting the hook. That run can already enqueue or send block replies through the delivery handler, so `{ suppress: true }` or reinjection can arrive after stale content has become user-visible, contradicting the pre-delivery contract.
  - confidence: 0.86
- **[P2] Expose the reinjected pipeline to the caller:** `src/auto-reply/reply/agent-runner-hooks.ts:97-99`
  - body: On reinjection this reassigns only the local `blockReplyPipeline`. The caller still flushes and stops the original pipeline after the wrapper returns, so buffered chunks from the final reinjected attempt may never be awaited or may be skipped.
  - confidence: 0.82
