## Summary

Keep open, but do not merge as-is. The PR proposes a plausible core plugin lifecycle seam, but the submitted patch exposes raw assistant responses without the existing conversation-access gate and runs suppress/reinject after important session and block-streaming side effects have already happened.

## What This Changes

The PR adds an `after_agent_complete` plugin hook with suppress and reinject results, wires it into `runReplyAgent`, and adds hook types/runner methods.

## Best Possible Solution

Define one maintained pre-delivery/finalization contract that can revise or suppress before delivery/session side effects, gate raw response access with `allowConversationAccess`, document the public API, and add focused hook-runner plus reply-runner tests.

## Reproduction Assessment

Yes for the blocking review findings via source inspection: the patch exposes `response` without a conversation-access gate and replaces block pipelines locally while current `runReplyAgent` flushes the caller pipeline. No live runtime proof was provided by the contributor.

## Solution Assessment

No as submitted. A safer direction is to align with the existing `before_agent_finalize`/final-reply hook work and run the decision before delivery and session side effects, with policy gates and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate response access with conversation permission:** `src/plugins/types.ts:1545`
  - body: The new event exposes the full assistant `response`, but the patch only adds the hook to the prompt-injection list. Current registry policy protects raw conversation hooks through `allowConversationAccess`; without adding this hook to that protected set, a non-bundled plugin can read reply content without the opt-in required for `llm_output`, `before_agent_finalize`, and `agent_end`.
  - confidence: 0.88
- **[P2] Keep the active block pipeline reachable:** `src/auto-reply/reply/agent-runner-hooks.ts:97-98`
  - body: On reinject this creates a new `blockReplyPipeline` that is scoped inside the wrapper, while `runReplyAgent` still flushes, stops, and final-payload-dedupes with the original pipeline after the wrapper returns. Block-streamed chunks from the final attempt can be unawaited or stale pipeline state can drive final delivery decisions.
  - confidence: 0.82
- **[P2] Run suppression before persisting hidden assistant turns:** `src/auto-reply/reply/agent-runner-hooks.ts:45-49`
  - body: `runAgentTurnWithFallback()` completes the embedded turn before the hook fires, so suppressing here only skips outbound delivery; the assistant draft has already been incorporated into the session snapshot used by later turns. That breaks the expected "discard before the user sees it" semantics for stale-response suppression unless the hook runs earlier or rolls back the turn state.
  - confidence: 0.8
