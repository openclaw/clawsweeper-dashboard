## Summary

Keep open; current main does not provide `after_agent_complete`, and this branch still needs API/security/product decisions plus proof before it can be considered merge-ready.

## What This Changes

The PR adds an `after_agent_complete` plugin hook, hook runner, and reply-agent wrapper so plugins can suppress delivery or reinject context after an agent run.

## Best Possible Solution

Define one supported lifecycle boundary: use `before_agent_finalize` where bounded final-answer revision is enough, or rebase a new after-complete/pre-delivery API with conversation-access gating, explicit delivery/transcript semantics, docs, tests, and real proof.

## Reproduction Assessment

Not applicable as a current-main bug; this PR proposes a new plugin lifecycle feature. The blocking risks are source-reproducible by tracing the PR wrapper after `runAgentTurnWithFallback` and comparing the new hook to current conversation-access policy.

## Solution Assessment

No. The direction may be useful for Speedtrap, but this branch is not the best mergeable shape until the hook contract is rebased onto current APIs and the security, delivery, transcript, and proof requirements are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate completion hooks as conversation hooks:** `src/plugins/types.ts:1540-1549`
  - body: The new event exposes raw assistant `response` text, but the diff only classifies `after_agent_complete` as prompt-injection gated. Non-bundled plugins need the same `allowConversationAccess=true` gate as `before_agent_finalize`, `llm_output`, and `agent_end`, or external plugins can read assistant output without that explicit trust opt-in.
  - confidence: 0.91
- **[P1] Defer delivery before suppress or reinject:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: `runAgentTurnWithHooks` waits for `runAgentTurnWithFallback()` to finish before invoking the hook, but the inner run can already enqueue or send block replies. A later `suppress` or `reinject` result cannot honor the advertised pre-delivery contract unless delivery is deferred or those modes are excluded for streamed/block-delivered output.
  - confidence: 0.88
- **[P1] Define transcript handling for suppressed drafts:** `src/auto-reply/reply/agent-runner-hooks.ts:86-87`
  - body: Returning an empty final payload after `runAgentTurnWithFallback()` suppresses delivery, but the completed assistant turn has already been written into the session path. The next turn can be conditioned on a response the user never saw unless the wrapper restores pre-hook state or the API explicitly documents transcript-preserving suppression.
  - confidence: 0.82
- **[P2] Return or flush the active block pipeline:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject, the wrapper replaces only its local `blockReplyPipeline`; `runReplyAgent` still owns and flushes the original pipeline after the wrapper returns. Buffered output from the final reinjected attempt can be missed or left unawaited unless the active pipeline is returned or flushed inside the wrapper.
  - confidence: 0.84
