## Summary

Keep open: current main does not implement `after_agent_complete`, and this PR is not mergeable as-is because the new response-observing/reinjecting lifecycle contract still has security, delivery, session-state, and proof gaps.

## What This Changes

The PR adds an `after_agent_complete` hook, hook runner, and reply-agent wrapper so plugins can inspect assistant output, suppress delivery, or reinject context.

## Best Possible Solution

Define one public hook boundary: either use the shipped `before_agent_finalize` path where bounded final-answer revision is enough, or land a rebased after-complete API that enforces conversation access, defines delivery and transcript semantics, and ships with focused tests, docs, and real proof.

## Reproduction Assessment

Not applicable as a current-main bug; this PR proposes a new plugin lifecycle feature. The blocking defects are source-reproducible by tracing the proposed wrapper after `runAgentTurnWithFallback` and comparing the new hook to current conversation-access policy.

## Solution Assessment

No. The direction may be useful for Speedtrap, but this branch is not the best mergeable shape until the hook contract is rebased onto current APIs and the security, delivery, transcript, and proof requirements are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate completion hooks as conversation hooks:** `src/plugins/types.ts:1537-1548`
  - body: The new event exposes raw assistant `response` text, but the diff only classifies `after_agent_complete` as prompt-injection gated. Non-bundled plugins should need `allowConversationAccess=true` like `before_agent_finalize`, `llm_output`, and `agent_end`, or external plugins can read assistant output without that explicit trust opt-in.
  - confidence: 0.91
- **[P1] Defer delivery before suppress or reinject:** `src/auto-reply/reply/agent-runner-hooks.ts:44-50`
  - body: `runAgentTurnWithHooks` waits for `runAgentTurnWithFallback()` to finish before invoking the hook, but the inner run can already enqueue or send block replies. A later `suppress` or `reinject` result cannot reliably honor the advertised pre-delivery contract unless delivery is deferred or those modes are explicitly excluded for streamed/block-delivered output.
  - confidence: 0.88
- **[P2] Return the active block pipeline to the caller:** `src/auto-reply/reply/agent-runner-hooks.ts:99-102`
  - body: On reinject, the wrapper replaces only its local `blockReplyPipeline`; `runReplyAgent` still owns and flushes the original pipeline after the wrapper returns. Buffered output from the final reinjected attempt can be missed or left unawaited unless the active pipeline is returned or flushed inside the wrapper.
  - confidence: 0.84
