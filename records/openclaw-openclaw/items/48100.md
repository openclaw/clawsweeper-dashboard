## Summary

Keep open: the feature is plausible, but this branch is not mergeable as-is because the new prompt-mutating hook is not actually blocked by the branch’s registry policy path, the wrapper can run after delivery/transcript side effects, and the PR does not include real behavior proof.

## What This Changes

Adds an `after_agent_complete` plugin hook around the auto-reply agent runner so plugins can suppress a completed reply or reinject context for another agent pass.

## Best Possible Solution

Provide one documented delivery or finalization seam in the current hook architecture that enforces prompt/conversation policies and clearly defines streaming and transcript behavior.

## Reproduction Assessment

Yes. The policy bypass is source-reproducible on the PR head: `after_agent_complete` is added to the prompt-injection list, but the branch registry only blocks `before_prompt_build` and constrains `before_agent_start`, so the new hook still registers.

## Solution Assessment

No. The proposed wrapper adds a new core hook at a fragile post-run boundary; current main’s `before_agent_finalize` is a safer final-answer revision anchor unless maintainers explicitly want a separate delivery-stage hook with defined suppress semantics.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.89

Full review comments:

- **[P1] Enforce prompt-injection policy for this hook:** `src/plugins/types.ts:1434`
  - body: Adding `after_agent_complete` to `PROMPT_INJECTION_HOOK_NAMES` is not enough on this branch: `registerTypedHook` only blocks `before_prompt_build` and constrains `before_agent_start`, then lets any other prompt-injection hook register. A plugin with `allowPromptInjection=false` can still return `reinject` and mutate the next prompt.
  - confidence: 0.91
- **[P2] Prevent hook decisions after block replies are sent:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: The wrapper waits for `runAgentTurnWithFallback` to finish before asking the hook, but that run can already enqueue or flush block replies through the existing `onBlockReply` path. For suppress or reinject decisions, users can receive stale first-attempt content before the hook verdict, which breaks the promised pre-delivery contract.
  - confidence: 0.84
- **[P2] Avoid keeping suppressed drafts in session history:** `src/auto-reply/reply/agent-runner-hooks.ts:86-87`
  - body: The suppress branch drops delivery only after the completed run has already updated the embedded session. That leaves an assistant draft the user never saw in future context, so a suppressed stale answer can still influence the next turn.
  - confidence: 0.78
