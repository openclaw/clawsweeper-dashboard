## Summary

Keep this PR open for maintainer review: it is a new core plugin lifecycle feature, overlaps with newer hook surfaces on current main, has blocking security/delivery issues in the proposed patch, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an `after_agent_complete` plugin hook that can suppress a completed response or reinject context for another agent pass before final reply delivery.

## Best Possible Solution

Decide the product boundary between `before_agent_finalize` and a true delivery-stage suppression hook, then implement the chosen hook with conversation-access gating, deferred delivery/session semantics, docs, contract tests, and real gateway proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the new event exposes response text without the existing conversation-access gate and that reinject creates a pipeline the caller cannot flush. The requested feature itself is not a current-main bug reproduction.

## Solution Assessment

No. The proposed wrapper is not the best mergeable shape yet because it fires after paths that can already deliver output and it misses the raw-conversation privacy classification; a safer implementation should either extend `before_agent_finalize` or add a true deferred delivery-stage hook.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate completion hooks as conversation-access hooks:** `src/plugins/types.ts:1548`
  - body: `after_agent_complete` exposes the draft assistant `response`, but the PR only adds it to the prompt-injection hook list. Since registry policy only requires `allowConversationAccess=true` for names in the conversation hook set, non-bundled plugins could read raw response content without the explicit privacy opt-in; add this hook to the conversation-access classification and tests.
  - confidence: 0.88
- **[P1] Return the active block pipeline after reinjection:** `src/auto-reply/reply/agent-runner-hooks.ts:103`
  - body: On reinject this wrapper stops the original pipeline and creates a replacement, but the caller still flushes and stops only the original pipeline after the wrapper returns. Any buffered output from the final reinjected attempt can be left unflushed or unawaited, so return/flush the active pipeline or keep delivery on a caller-owned pipeline.
  - confidence: 0.84
- **[P2] Defer delivery until the hook decides:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: The hook runs after `runAgentTurnWithFallback`, but that path can already enqueue block-streamed replies or directly send media through `onBlockReply`. A later `{ suppress: true }` cannot guarantee that nothing was delivered, so the implementation needs a deferred delivery buffer or an enforced unsupported-mode path before advertising pre-delivery suppress/reinject semantics.
  - confidence: 0.78
