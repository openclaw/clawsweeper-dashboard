## Summary

Keep open. The feature is not implemented on current main, but the branch is not mergeable as-is because the new response-observing hook bypasses the existing conversation-access gate, cannot reliably honor pre-delivery suppress/reinject semantics on streaming paths, and lacks real behavior proof.

## What This Changes

The branch adds an `after_agent_complete` plugin lifecycle hook plus a reply-runner loop that can suppress a completed response or append injected context for another agent pass.

## Best Possible Solution

Rebase onto the current hook contract architecture, use or extend `before_agent_finalize` where final-answer revision is enough, and add any separate suppress/reinject hook only with conversation-access gating plus explicit streaming and transcript semantics.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a new plugin lifecycle feature. The blocking PR defects are source-reproducible from the PR head and current hook-policy and reply-delivery paths.

## Solution Assessment

No. The direction is plausible, but this PR is not the best mergeable form until it is rebased onto current hook architecture, gated for conversation access, and scoped against streaming and transcript behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require conversation access for completion hooks:** `src/plugins/types.ts:1548`
  - body: The new event exposes the assistant `response`, but the hook is only added to prompt-injection policy. Non-bundled plugins that have not opted into `allowConversationAccess` can therefore observe raw conversation output by registering this hook; add the hook to the conversation-access policy when rebasing.
  - confidence: 0.91
- **[P1] Defer streaming until completion hooks decide:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: This wrapper calls `runAgentTurnWithFallback()` before `runAfterAgentComplete()`, but the current run path can enqueue or directly send block replies during that call. A later suppress or reinject verdict cannot honor the claimed pre-delivery contract unless delivery is deferred or the hook explicitly excludes those streaming paths.
  - confidence: 0.88
- **[P2] Return the active reinject pipeline to the caller:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject, the wrapper replaces only its local `blockReplyPipeline`; `runReplyAgent` still flushes and stops the original pipeline after the wrapper returns. Buffered output from the final reinjected attempt can be missed unless the active pipeline is returned or final flushing is owned inside the wrapper.
  - confidence: 0.84
