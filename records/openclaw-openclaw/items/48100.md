## Summary

Keep open: this is a new core plugin lifecycle feature, not something current main already implements, and the branch still has blocking delivery, policy, and proof gaps before it is mergeable.

## What This Changes

The PR adds an `after_agent_complete` plugin hook that can inspect the agent response, suppress delivery, or reinject context for another run.

## Best Possible Solution

Rebase onto the current hook architecture, reuse `before_agent_finalize` where final-answer revision is sufficient, and only add a separate completion hook if its access policy, delivery ordering, streaming, transcript, docs, tests, and proof semantics are explicit.

## Reproduction Assessment

Not applicable as a current-main bug; this PR proposes a new lifecycle feature. The blocking findings are source-reproducible from the PR diff and current main's hook-policy and reply-delivery paths.

## Solution Assessment

No. The direction may support speedtrap, but this branch is not the best mergeable form until it is rebased onto current hook contracts and the policy, streaming, pipeline, and transcript semantics are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate completion hooks with conversation access:** `src/plugins/types.ts:1540-1549`
  - body: The new event exposes assistant `response` text, but the hook is only classified as prompt-injection capable. Non-bundled plugins could observe raw assistant output without the explicit `allowConversationAccess=true` gate required for comparable conversation hooks.
  - confidence: 0.92
- **[P1] Defer block delivery until completion hooks decide:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: The wrapper runs `runAgentTurnWithFallback()` before firing the hook, but current block delivery can enqueue or send user-visible chunks during that call. A later `suppress` or `reinject` result cannot reliably honor the claimed pre-delivery behavior.
  - confidence: 0.88
- **[P2] Return the active reinject pipeline:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject, the wrapper replaces only its local `blockReplyPipeline`, while `runReplyAgent` still flushes the original pipeline after the wrapper returns. Buffered output from the final reinjected attempt can be missed unless the active pipeline is returned or flushing is owned inside the wrapper.
  - confidence: 0.84
