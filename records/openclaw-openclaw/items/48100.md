## Summary

Keep this PR open for maintainer review rather than cleanup-close: the requested hook is not implemented on current main, but the branch adds a new core plugin API with blocking policy/delivery issues and no after-fix real behavior proof.

## What This Changes

The PR adds an `after_agent_complete` plugin hook and wraps reply-agent execution so plugins can inspect a completed response, suppress delivery, or reinject context for another run.

## Best Possible Solution

Rebase onto the current hook architecture, reuse `before_agent_finalize` where final-answer revision is enough, and only add a separate completion hook if its conversation-access, delivery, streaming, transcript, docs, tests, and proof semantics are explicit.

## Reproduction Assessment

Not applicable as a current-main bug; this PR proposes a new lifecycle feature. The blocking review findings are source-reproducible from the PR diff and current main's hook-policy and reply-delivery paths.

## Solution Assessment

No. The direction may support speedtrap, but this branch is not the best mergeable form until it is rebased onto current hook contracts and the policy, streaming, pipeline, and transcript semantics are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require conversation access for completion hooks:** `src/plugins/types.ts:1537-1548`
  - body: The new event exposes assistant `response` text, but the hook is only added to prompt-injection policy. Non-bundled plugins without `allowConversationAccess=true` could observe raw assistant output by registering this hook, so classify it with the conversation-access gated hooks when rebasing.
  - confidence: 0.91
- **[P1] Defer delivery until the completion hook decides:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: `runAgentTurnWithHooks` calls `runAgentTurnWithFallback()` before `runAfterAgentComplete()`, but current reply delivery can enqueue or send block replies during that call. A later `suppress` or `reinject` result cannot honor the claimed pre-delivery contract unless delivery is deferred or the hook contract excludes those modes.
  - confidence: 0.88
- **[P2] Return the active reinject pipeline to the caller:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject, the wrapper replaces only its local `blockReplyPipeline`; `runReplyAgent` still owns the original pipeline after the wrapper returns. Buffered output from the final reinjected attempt can be missed unless the active pipeline is returned or final flushing is owned inside the wrapper.
  - confidence: 0.84
