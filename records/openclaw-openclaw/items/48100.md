## Summary

Keep this PR open, but it is not merge-ready: the hook fills a plausible plugin API gap, yet the current diff has source-reproducible privacy and delivery-order bugs, is stale against the current hook contract surface, and lacks real behavior proof.

## What This Changes

The PR adds an `after_agent_complete` plugin lifecycle hook that can suppress a completed response or reinject context for another agent run before delivery.

## Best Possible Solution

Rebase against the current hook SDK and design a documented, conversation-access-gated completion hook that runs before any deliverable output, with focused tests and redacted real behavior proof.

## Reproduction Assessment

Yes. The blockers are source-reproducible from the PR head and current main: the hook exposes raw responses without the conversation gate, runs after inner fallback delivery paths can flush, and targets a hook type surface that has moved on main.

## Solution Assessment

No. The requested capability is useful, but this diff is not the narrowest maintainable solution until the privacy gate, delivery timing, current hook-types surface, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate completion hooks with conversation access:** `src/plugins/types.ts:1548-1549`
  - body: The new event exposes the full agent `response`, but the patch only adds the hook to prompt-injection gating. Current registry policy blocks raw conversation hooks via `isConversationHookName`; without adding this hook to that gate, non-bundled plugins can read generated replies without `allowConversationAccess`.
  - confidence: 0.89
- **[P1] Run suppression before block replies can flush:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: The wrapper awaits `runAgentTurnWithFallback()` before asking hooks whether to suppress or reinject. That inner run can call `onBlockReplyFlush` and flush queued block replies before the hook returns, so a suppress/reinject result can arrive after user-visible output has already been sent.
  - confidence: 0.86
- **[P2] Register the hook on the current hook-types surface:** `src/plugins/types.ts:1364-1368`
  - body: Current main moved hook names, prompt/conversation gate lists, and the handler map to `src/plugins/hook-types.ts`; this patch still edits the old `types.ts` hook block. After rebase, the hook will not be recognized by `isPluginHookName` or the registry unless those additions move to the current hook-types surface.
  - confidence: 0.82
