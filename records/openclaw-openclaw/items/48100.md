## Summary

Keep this PR open for maintainer review, but it is not merge-ready: the proposed hook is useful, current main does not implement `after_agent_complete`, and the branch still has correctness, security-policy, and proof gaps around a new core plugin lifecycle API.

## What This Changes

The PR adds an `after_agent_complete` plugin lifecycle hook with suppress/reinject behavior and wires it into the reply agent runner plus plugin hook types.

## Best Possible Solution

Design this as a maintainer-owned lifecycle seam that composes with current `before_agent_finalize`/delivery hooks, enforces both prompt-injection and conversation-access policy, and documents/tests streaming plus transcript semantics before exposing it to plugins.

## Reproduction Assessment

Yes for the PR defects by source inspection: the wrapper runs after `runAgentTurnWithFallback`, while current delivery and session code can emit or persist during that run. A live gateway repro was not run because this was a read-only review.

## Solution Assessment

No. Wrapping the completed turn is not the safest way to provide a pre-delivery suppress/reinject contract; the hook should be integrated at an existing finalization or explicit pre-delivery stage with clear streaming, transcript, access-policy, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Defer streamed delivery until the hook decides:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: This calls `runAgentTurnWithFallback` before asking `after_agent_complete`, but that run can already send partial/block replies. If the hook later returns `suppress` or `reinject`, users may have already received the stale first attempt, so the advertised pre-delivery contract is broken for streaming-enabled channels.
  - confidence: 0.9
- **[P1] Preserve the final block pipeline across reinjects:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject this replaces `blockReplyPipeline` only inside the wrapper. The caller still flushes and stops the original pipeline after `runAgentTurnWithHooks` returns, so block replies from the final reinjected attempt may not be awaited or flushed in order.
  - confidence: 0.86
- **[P1] Require conversation access for response-reading hooks:** `src/plugins/types.ts:1548`
  - body: The new event exposes the assistant response text, but the hook is only classified for prompt-injection policy. Non-bundled plugins should not be able to read raw conversation output unless `allowConversationAccess` permits this hook too.
  - confidence: 0.84
