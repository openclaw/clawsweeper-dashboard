## Summary

Keep this PR open because current main still lacks the requested lifecycle hook, but the proposed branch is not merge-ready: it exposes assistant response text without the current conversation-access gate, cannot honor suppress/reinject before streaming delivery, and lacks real behavior proof.

## What This Changes

The PR adds an `after_agent_complete` plugin hook, a hook runner method, and an agent-runner wrapper that can suppress a completed reply or reinject context for another agent pass.

## Best Possible Solution

Land a current-main refit that adds the hook through the current hook contracts, gates it as both prompt-injection and conversation-access, and either defers delivery until hook verdicts or explicitly scopes the hook away from streaming modes.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a new plugin lifecycle feature. The blocking PR issues are source-reproducible from the post-`runAgentTurnWithFallback` wrapper, current conversation-access classification, and caller-owned block pipeline flush path.

## Solution Assessment

No. The hook direction is plausible, but this PR is not the best mergeable form until privacy gating, streaming/pre-delivery behavior, and current hook architecture integration are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require conversation access for completion hooks:** `src/plugins/types.ts:1548`
  - body: The new event exposes the full assistant `response`, but the hook is not classified with current raw conversation hooks. Non-bundled plugins without `hooks.allowConversationAccess=true` could observe conversation content through this handler, unlike `llm_output` or `agent_end`; add the hook to the conversation-access classification when refitting it onto current main.
  - confidence: 0.91
- **[P1] Prevent streaming output before hook verdicts:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: This wrapper waits for `runAgentTurnWithFallback()` to finish before calling the hook, but that path can already enqueue or flush block replies while the run is in progress. A later `suppress` or `reinject` result cannot retract content that reached the channel, so this needs deferred delivery or an explicit guard that prevents this hook from running with streaming delivery modes.
  - confidence: 0.88
- **[P2] Return the active reinject pipeline to the caller:** `src/auto-reply/reply/agent-runner-hooks.ts:101-103`
  - body: On reinject this code replaces only the wrapper's local `blockReplyPipeline`; after the wrapper returns, `runReplyAgent` still flushes and stops the original pipeline. Buffered output from the final reinjected attempt can be missed unless the active pipeline is handed back or final pipeline flushing is owned inside the wrapper.
  - confidence: 0.84
