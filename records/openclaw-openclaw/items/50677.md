## Summary

Keep this open. Current main still truncates the prompt-facing skills catalog by count and character budget, still reports only an included/total count inside the model-facing skills prompt, and still has no priority, pinned, or required-skill config contract. The related omitted-name PR and broader priority issue are both still open, so the requested visibility and prioritization behavior has not landed.

## What This Changes

The issue requests operator-visible and configurable handling for skills prompt truncation so user-created or critical skills are not silently omitted from the agent-visible catalog.

## Best Possible Solution

Keep this open until OpenClaw ships an omitted-skill discovery or operator-visible warning path, documents the skills prompt budget and truncation outcome in public docs, and explicitly decides whether priority, pinning, or context-required skills belong in the skills config contract or remain tracked by narrower follow-ups such as #50199.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to build a workspace skills prompt with many eligible skills and low `skills.limits.maxSkillsPromptChars` or `maxSkillsInPrompt`; existing tests already assert compact and count truncation, including a case where no skill fits.

## Solution Assessment

No. Current main is not the best resolution because it still keeps truncation recovery model-facing and count-only; #42637 is a plausible narrow omitted-name improvement, while priority/pinning and user-visible warning semantics need maintainer API/product review.
