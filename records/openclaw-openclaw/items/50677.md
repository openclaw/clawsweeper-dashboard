## Summary

Keep open. Current main and v2026.5.22 still truncate the prompt-facing skills list to a fitting prefix with only a count-based model-facing note; docs now mention the budget knobs, but omitted-skill recovery, guaranteed priority/pinning, and user-visible warning semantics have not landed. The open omitted-key PR and the open priority issue cover useful pieces, but neither resolves this broader report.

## What This Changes

The issue asks OpenClaw to stop silently dropping eligible user-created skills from `available_skills` when the skills prompt budget is exceeded, and to add visibility or priority controls for that truncation path.

## Best Possible Solution

Review and land or replace the omitted-key recovery work, document truncation behavior in the public skills docs, and make an explicit maintainer decision on priority/pinning and operator-visible warning semantics before adding new config.

## Reproduction Assessment

Yes, at source level. Current main sorts prompt-facing skills alphabetically, applies count and character budgets, and emits only a count-based model-facing note when skills are omitted; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a complete solution. Omitted lookup keys are the narrow recovery path, while priority, pinning, required skills, and user-visible warnings need maintainer product/API direction.
