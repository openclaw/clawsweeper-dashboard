## Summary

Keep open. Current main and latest release v2026.6.10 still truncate prompt-visible skills to an alphabetical fitting prefix with a count-only model-facing note, while the omitted-skill recovery PR and the priority/pinning API decision remain open.

## What This Changes

The issue asks OpenClaw to make skills prompt truncation visible and controllable, especially for user-created skills that fall out of the model-facing skills list.

## Best Possible Solution

Land or replace the omitted-skill recovery path, document the skills prompt budget behavior, and make an explicit maintainer decision on whether priority or pinning belongs in the canonical skills config contract.

## Reproduction Assessment

Yes, as a source-level reproduction: current main still slices the prompt-visible skills list under `maxSkillsPromptChars` and emits only a count-based warning. I did not run a live model session because this review is read-only, but the code path and existing tests clearly exercise the truncation behavior.

## Solution Assessment

Unclear. Listing omitted lookup keys is the best narrow repair and is already covered by an open PR, but priority, pinning, per-topic required skills, and user-visible warnings are product/API choices.
