## Summary

Keep open: current main still has only time-based pruning for gateway auth-rate-limit entries, and the linked implementation PR is open rather than merged, so this issue should stay open until that fix or an equivalent bounded-limiter change lands.

## What This Changes

The issue requests a hard cap for gateway auth-rate-limit entries so unique-IP failed-auth floods cannot grow the in-memory map without bound between prune cycles.

## Best Possible Solution

Land a reviewed bounded-limiter fix that caps entries on insert, preserves locked-out entries, and aligns config, docs, and tests if the cap is operator-visible.

## Reproduction Assessment

Yes. Source inspection on current main shows recordFailure inserts a Map entry for each new non-exempt scope/IP key, while only prune removes stale unlocked entries, so a unique-IP failure loop before prune would grow size with unique key count.

## Solution Assessment

Yes. The narrow maintainable solution is an insertion-time entry cap that evicts only non-locked entries; the linked PR is the right implementation candidate to review instead of closing this issue early.
