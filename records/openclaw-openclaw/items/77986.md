## Summary

Current main still has the uncapped auth-rate-limit entries map described here, and the issue is already paired with open closing PR #77987, so it should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Add a hard cap to the gateway auth rate limiter's per-scope/IP entries map so unique-IP failure floods cannot grow memory between prune cycles while preserving locked-out entries.

## Best Possible Solution

Land or replace PR #77987 with a narrow gateway-auth limiter fix that bounds non-locked entries, preserves active lockouts, updates regression coverage, and keeps the public config/docs aligned if a maxEntries knob is accepted.

## Reproduction Assessment

Yes, from source inspection: repeatedly calling recordFailure with unique non-loopback IPs creates a new Map entry each time, and only prune can later remove stale entries. I did not execute a runtime proof in this read-only review, but the current-main path is clear.

## Solution Assessment

Yes, the linked PR direction is the right narrow shape: enforce a bounded entry count on new inserts while skipping currently locked entries. The issue should remain open so maintainers can review and land that implementation or an equivalent one.
