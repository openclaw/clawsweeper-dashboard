## Summary

Keep open: current main still has no hard cap for the gateway auth limiter entries map, and open PR #77987 is already linked as the implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Add a hard bound to the gateway auth rate limiter's per-scope/IP entries map under unique-IP failed-auth floods while preserving locked-out entries.

## Best Possible Solution

Land a reviewed fix that caps non-locked auth-rate-limit entries on insert, preserves locked-out entries, and updates schema/docs/changelog if the cap is exposed.

## Reproduction Assessment

Yes. Source inspection shows `recordFailure` inserts a new Map entry for each new non-loopback `{scope, ip}`, and only `prune()` removes stale unlocked entries, so looping unique IP failures before a prune tick would grow `size()` with unique IP count.

## Solution Assessment

Yes. The narrow maintainable fix is an auth-rate-limit insertion-time cap that evicts only non-locked entries; the linked PR is the right candidate to review rather than closing this issue early.
