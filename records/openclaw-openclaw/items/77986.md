## Summary

Keep open: current main still has no hard cap for gateway auth rate-limiter entries, and the linked closing PR remains open and dirty rather than merged. The issue is valid DoS-hardening work, but the active fix path should be reviewed and landed before this issue closes.

## What This Changes

The issue asks for a hard cap on gateway auth rate-limiter entries so unique-IP failed-auth floods cannot grow the in-memory map between prune ticks while locked entries remain protected.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77987 or an equivalent bounded-limiter fix that caps insertions, evicts only non-locked entries, and updates the public config contract if the cap is configurable.

## Reproduction Assessment

Yes. Source inspection on current main shows `recordFailure` inserts one entry per new non-loopback scope/IP key and no current code path caps `entries.size` before the periodic prune.

## Solution Assessment

Yes. The narrow maintainable fix is an insertion-time entry cap that preserves active lockouts; the linked PR is the right candidate once its config-contract and merge-state issues are resolved.
