## Label Justifications

- `P1`: A production Telegram report shows sessions_spawn stalls can block an entire chat session queue for users until manual recovery.
- `impact:session-state`: The issue is about subagent runs and parent sessions remaining stuck in processing/active state instead of reaching a terminal outcome.
- `impact:message-loss`: Queued chat messages behind the stuck spawn receive no acknowledgement or reply, creating user-visible message suppression.

## Summary

Keep open: current main has hard wall-clock subagent run timeouts and generic stuck-session recovery, but it still lacks the requested subagent-specific activity nudge-to-kill watchdog, stall config keys, and per-spawn controls; the June 10 Telegram production report keeps this user-visible rather than speculative.

## What This Changes

Add two-stage activity-based subagent stall recovery that nudges inactive children, then kills them and notifies the parent/user, with global and per-spawn controls.

## Best Possible Solution

A maintainer-sponsored design should decide the default-on policy, config shape, and user-visible failure semantics, then implement subagent activity tracking/recovery through the existing registry, control, and diagnostic recovery paths.

## Reproduction Assessment

No live repro was run; source inspection confirms the requested stall knobs and staged subagent recovery are absent, and the June 10 Telegram comment gives a credible production failure path.

## Solution Assessment

Unclear: two-stage recovery is plausible, but maintainers should choose how it relates to runTimeoutSeconds, diagnostics active-abort recovery, and the #87445 manual kill surface before implementation.
