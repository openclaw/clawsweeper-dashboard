## Summary

Keep open. Current main still lacks the requested two-stage subagent stall recovery: it has wall-clock run timeouts, manual kill/steer controls, stale active-run cleanup, and restart/orphan recovery, but no per-subagent tool-inactivity timestamp, nudge state, automatic nudge injection, or kill escalation after inactivity. The related implementation PR #43017 is closed unmerged, while #39127 and #39141 remain broader open activity/watchdog design tracks rather than shipped behavior.

## What This Changes

The issue requests configurable two-stage subagent stall recovery based on time since child tool activity, with a nudge phase followed by automatic kill escalation and per-spawn overrides.

## Best Possible Solution

Define the canonical child-activity signal, then either implement this in the subagent registry/control path or explicitly fold it into the broader session activity/watchdog design. A maintainable implementation should add strict config and `sessions_spawn` override fields, persist auditable activity/nudge/escalation state, deliver nudges through the existing child-session messaging path, terminate through the existing subagent kill/cancellation path, document the behavior, and cover config/runtime/docs with focused tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is source/docs inspection on current main: the strict config schema, `sessions_spawn` schema, run record type, and registry wait/sweep paths expose only wall-clock timeout and cleanup behavior, not the requested inactivity nudge/kill state machine.

## Solution Assessment

Unclear. The two-stage approach is plausible, but the safest route is for maintainers to choose whether it should be a targeted subagent feature now or part of the broader #39127/#39141 session activity/watchdog model to avoid parallel activity systems.
