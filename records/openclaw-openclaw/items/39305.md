## Summary

Keep open: current main has adjacent hard-timeout and diagnostic abort-drain recovery, but the requested two-stage sub-agent nudge/kill policy, per-spawn overrides, and user-visible recovery behavior remain unsettled product work.

## What This Changes

Request adds escalating sub-agent stall recovery that nudges inactive child runs, then kills and notifies the parent if inactivity continues.

## Best Possible Solution

Define one maintainer-approved recovery policy that reuses existing timeout, diagnostic activity, and subagent-control primitives without creating a second source of session truth.

## Reproduction Assessment

Partially. The June 10 production logs show the stuck sessions_spawn symptom, and current source confirms the requested nudge/kill config and policy are absent; I did not run a live current-main Telegram reproduction.

## Solution Assessment

Unclear. A two-stage recovery policy is plausible, but maintainers need to decide how it should compose with current diagnostic abort-drain recovery, runTimeoutSeconds, manual kill/reap, and user-visible channel notifications.
