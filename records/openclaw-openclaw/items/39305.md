## Summary

Keep open: current main has hard `runTimeoutSeconds` handling and lost-context recovery, but it still lacks the requested default-on, tool-inactivity-based sub-agent nudge-to-kill recovery and parent/user notification path.

## What This Changes

The issue requests two-stage stall recovery for stalled sub-agent runs, with a nudge followed by automatic termination and parent notification.

## Best Possible Solution

Design a maintainer-approved recovery policy that reuses the current sub-agent run manager and lifecycle seams, adds focused inactivity detection only where needed, and defines default thresholds plus parent/user notification semantics explicitly.

## Reproduction Assessment

Yes, but only at source and production-report level. Current code and docs show no default inactivity-based sub-agent recovery, and the June 10 comment gives concrete production symptoms, but I did not run a live Telegram reproduction against current main.

## Solution Assessment

Unclear as proposed. The recovery goal is valid, but the best fix should consolidate with existing hard timeout, blocked-liveness, lost-context, and watchdog surfaces instead of adding a parallel sweeper/config stack without maintainer direction.
