## Summary

Keep open: current main has hard subagent runTimeoutSeconds enforcement and generic stale/lost-context cleanup, but it still lacks the requested activity-based subagent nudge-to-kill watchdog, stall config keys, and per-spawn stall controls; the June 10 Telegram production report keeps this user-visible and high impact.

## What This Changes

The issue requests two-stage sub-agent stall recovery that nudges an inactive child agent and then kills it if it does not resume tool activity.

## Best Possible Solution

A maintainer-sponsored design should decide the default policy, config shape, per-spawn exposure, and channel-visible failure semantics, then implement subagent activity tracking and recovery through the existing registry/lifecycle paths without duplicating the session activity watchdog work.

## Reproduction Assessment

Partly. Source inspection shows current main lacks the requested stall nudge/kill mechanism, and the June 10 comment gives production incident data, but I did not run a live Telegram or gateway reproduction against current main.

## Solution Assessment

Unclear. A registry-owned watchdog is plausible, but default-on behavior, model-visible/per-spawn controls, queue-level recovery, and user-visible channel notification need maintainer direction first.
