## Summary

Keep open: current main has generic stuck-session detection/recovery and configurable `runTimeoutSeconds`, but it does not implement the requested default-on two-stage sub-agent inactivity nudge/kill or user-visible parent/channel notice. The June 10 production report gives concrete chat-queue impact, so this needs maintainer product review rather than cleanup close.

## What This Changes

Request default-on two-stage sub-agent stall recovery that nudges after inactivity, kills after continued inactivity, and notifies the parent/channel with related config/default decisions.

## Best Possible Solution

Evolve the existing diagnostic stuck-session recovery/session-activity path into a single owner for sub-agent no-progress recovery, then decide defaults, nudge behavior, parent error propagation, and channel-visible notice policy before adding new config or tool surface.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source and recent production logs identify the blocked `sessions_spawn`/diagnostic path, while current main shows only generic stuck-session recovery and no two-stage sub-agent nudge/kill surface.

## Solution Assessment

Unclear as proposed: the recovery goal is valid, but the best implementation should build on existing diagnostic recovery and session-activity work rather than add a parallel subagent sweeper/config stack without maintainer defaults and notice semantics.
