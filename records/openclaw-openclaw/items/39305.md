## Label Justifications

- `P1`: A recent production Telegram-agent report shows a sessions_spawn stall blocking a group chat serial queue for hours, which is an urgent real-user channel workflow failure.
- `impact:session-state`: The issue centers on subagent runs and parent sessions remaining in running or processing state without reliable recovery.
- `impact:message-loss`: When the parent chat session is stuck behind the dead child run, subsequent user messages receive no acknowledgement or reply.

## Summary

Keep open: current main has partial mitigations for subagent timeouts, diagnostic aborts, and internal kill helpers, but it does not implement the requested two-stage subagent stall nudge-to-kill contract. The June 10 production comment makes this a high-impact chat/session reliability issue, while the default-on recovery behavior and new config surface still need maintainer product direction.

## What This Changes

Request adds default-on two-stage stall recovery for subagents, with nudge and kill thresholds plus config/per-spawn controls when a child stops making tool-call progress.

## Best Possible Solution

Define a maintainer-approved gateway/subagent recovery contract that reuses current diagnostics/session activity and kill primitives, preserves safe upgrade defaults, and emits a bounded parent/user-visible terminal event when a child blocks a serial chat queue.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The missing two-stage recovery surface is source-reproducible on current main, and the June 10 comment provides production symptoms, but a current-main Telegram/live replay is still needed before treating it as a strict reproduced bug.

## Solution Assessment

Unclear whether the exact proposed default-on nudge-to-kill design is the best final shape. A safer path is for maintainers to settle how this composes with session activity, diagnostic recovery, manual kill/reap-stale, and chat-visible timeout notices before implementation.
