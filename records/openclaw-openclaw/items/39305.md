## Label Justifications

- `P1`: Fresh production evidence describes repeated hours-long outages where one stuck spawn makes an interactive group session unusable.
- `impact:session-state`: The child run and parent session remain stuck without a user-workflow terminal transition.
- `impact:message-loss`: Subsequent conversation messages queue indefinitely and receive no acknowledgement or reply.

## Summary

Keep open: current main now enforces explicit child-run deadlines in the registry, but fresh production evidence reports that a blocked `sessions_spawn` tool call can still hold the parent session queue indefinitely. The reliability problem remains actionable, while the proposed default-on nudge/kill settings require a separate product decision.

## What This Changes

Add escalating inactivity recovery for stalled sub-agents, with a nudge followed by forced termination, parent notification, and configurable per-default or per-spawn thresholds.

## Best Possible Solution

Guarantee that a child deadline or confirmed unrecoverable stall terminates or safely detaches the child, returns a canonical terminal error from `sessions_spawn`, notifies the parent, and releases the serialized session queue; consider a conservative nudge stage afterward using canonical run-progress state rather than parallel registry policy.

## Reproduction Assessment

No high-confidence live current-main reproduction was run, but the June 10 production diagnostics provide a concrete source-level path: `sessions_spawn` remains the active blocked tool call while later messages queue behind it. Current code exposes the relevant deadline, completion, and stuck-session diagnostic seams for a focused reproduction.

## Solution Assessment

No. A two-stage nudge/kill watchdog may be useful, but the narrowest maintainable first fix is to prove and enforce the existing child-deadline-to-parent-terminal and queue-release contract; default inactivity thresholds should be decided separately.
