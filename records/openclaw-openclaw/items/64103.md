## Summary

Keep open: current `main` and the latest release still expose terminal-looking session `status` values without an explicit resumability signal, and the linked closing PR is open but unmerged. The issue should stay open until that PR or a maintainer-approved replacement lands.

## What This Changes

The issue asks OpenClaw to clarify session-list status versus resumability so agents resume existing sub-agent sessions instead of spawning duplicates.

## Best Possible Solution

Land a maintainer-approved, backward-compatible session-list contract that exposes resumability explicitly, either by refreshing the linked PR or replacing it narrowly.

## Reproduction Assessment

Yes at the source-contract level: current main exposes terminal-looking last-run statuses while the reactivation path accepts ended subagent runs and starts a replacement run. I did not establish a live LLM-orchestrator duplicate-spawn scenario in this read-only review.

## Solution Assessment

Unclear until maintainers accept the exact API contract, but the additive `resumable` signal in the linked PR is a safer narrow fix than renaming shipped status values. A broader lifecycle model would be more invasive and should not block this bug path unless maintainers reject the additive contract.
