## Summary

Keep this issue open: current main and v2026.6.6 still expose terminal-looking session `status` values without resumability metadata, while the active linked PR is the open implementation path and has not landed.

## What This Changes

The issue asks OpenClaw to make session status semantics clear so orchestrator agents resume existing sessions instead of spawning duplicates after last-turn states like failed, timeout, or done.

## Best Possible Solution

Land or explicitly replace the linked additive `resumable` implementation, then close this issue once main exposes a clear resumability contract for session rows.

## Reproduction Assessment

Yes, at source level: current main exposes terminal-looking `status` values without resumability metadata while the runtime can reactivate ended subagent runs. I did not run a deterministic live orchestrator transcript.

## Solution Assessment

Yes, the linked additive `resumable` path appears to be the narrowest compatible fix if maintainers accept the field semantics. A breaking rename of status values would be riskier for existing consumers.
