## Label Justifications

- `P2`: This is a bounded but real session-state behavior problem that can cause duplicate subagent sessions and token waste without making core unusable.
- `impact:session-state`: The reported failure is agents misreading session state and abandoning resumable session context for duplicate sessions.

## Summary

Keep open: current main and the latest release still expose terminal-looking session `status` values without a model-visible resumability signal, and the open linked fix PR is not merged. The issue should stay open until https://github.com/openclaw/openclaw/pull/89101 or a narrow replacement lands.

## What This Changes

The issue asks OpenClaw to make session `status` semantics or resumability explicit so agents do not treat last-turn outcomes as permanent session lifecycle states.

## Best Possible Solution

Land a maintainer-approved, backward-compatible session list/gateway row contract that makes resumability explicit, refreshing the linked PR or replacing it narrowly if needed.

## Reproduction Assessment

Yes at the source-contract level: current main exposes terminal-looking latest-run statuses, while the reactivation path accepts ended subagent runs and starts a replacement run. I did not run a live LLM orchestrator duplication scenario in this read-only review.

## Solution Assessment

Unclear until maintainers accept the exact API contract, but the additive `resumable` signal in the linked PR is safer than renaming shipped status values. A broader lifecycle redesign would be more invasive and should not block the narrow bug path unless maintainers reject the additive contract.
