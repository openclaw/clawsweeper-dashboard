## Summary

Keep open. Current main still exposes terminal-looking legacy session statuses through persistence, gateway rows, and agent-facing `sessions_list` without explicit resumability or status-scope metadata, while `sessions.send` can still target and reactivate existing sessions.

## What This Changes

The issue requests clarifying session `status` semantics so agents resume existing sessions instead of spawning duplicates after `done`, `failed`, or `timeout` last-turn states.

## Best Possible Solution

Keep the legacy `status` union for compatibility, add explicit resumability/status-scope or targetability metadata to gateway and agent-facing session rows, and update prompts, docs, and focused tests around that contract.

## Reproduction Assessment

Yes, at source level. Current `sessions_list` can still return `done`, `failed`, or `timeout` without resumability context, while `sessions.send` can send to an existing session key and reactivate completed subagent sessions.

## Solution Assessment

No for the proposed breaking rename. The best maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.
