## Summary

Keep open. Current main and v2026.5.12 still expose terminal-looking legacy session statuses through persistence, gateway rows, and agent-facing `sessions_list`, while `sessions.send` can still target an existing `sessionId` and reactivate completed subagent sessions.

## What This Changes

The issue requests clearer agent-facing session status semantics so `done`, `failed`, `timeout`, and `killed` do not make resumable sessions look permanently closed.

## Best Possible Solution

Keep the legacy `status` union for compatibility, add explicit resumability and status-scope/targetability metadata to gateway rows and agent-facing `sessions_list`, and align prompts, docs, and tests around that contract.

## Reproduction Assessment

Yes, at source level. A session row can still list `status: "done"`, `"failed"`, or `"timeout"` without resumability context, while `sessions.send` can send to an existing `sessionId` and reactivate completed subagent sessions.

## Solution Assessment

No for the proposed breaking rename. The narrow maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.

## Repair Work Prompt

Fix the ambiguous session status contract reported at https://github.com/openclaw/openclaw/issues/64103. Keep the existing `running | done | failed | killed | timeout` status values for backward compatibility; do not implement the breaking rename. Add explicit backward-compatible metadata that tells gateway clients and agent-facing `sessions_list` consumers whether a session is resumable/targetable by `sessions_send` and what scope the `status` value represents. Use current send constraints as the source of truth: a row with a usable `sessionId` and allowed send policy should not look permanently dead merely because the last/current run status is `done`, `failed`, or `timeout`; rows without a transcript/session id or with denied/deleted targets should not be marked resumable.
