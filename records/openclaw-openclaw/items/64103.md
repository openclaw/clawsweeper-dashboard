## Summary

Keep open. Current main still exposes terminal-looking session `status` values through persisted session entries, gateway session rows, and the agent-facing `sessions_list` output, while `sessions.send` can still target an existing session and reactivate completed sub-agent sessions.

## What This Changes

The issue requests clearer session status semantics so agents do not treat last-turn states such as `failed`, `timeout`, or `done` as non-resumable session lifecycle states.

## Best Possible Solution

Add backward-compatible resumability and status-scope metadata to gateway rows and `sessions_list`, keep the existing `status` union, and update tool descriptions, prompt guidance, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list a session row whose `status` is `done`, `failed`, or `timeout`: current main forwards that bare value through gateway and `sessions_list`, while `sessions.send` still targets existing entries with a `sessionId` and can reactivate completed sub-agent sessions.

## Solution Assessment

No for the proposed breaking rename. The maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.
