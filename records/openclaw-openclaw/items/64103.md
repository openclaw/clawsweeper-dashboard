## Summary

Keep open. Current main still exposes terminal-looking session `status` values through persisted entries, gateway rows, and `sessions_list` without an explicit resumability or status-scope field; the prompt/docs discourage polling but do not tell orchestrators that these sessions can be resumed with `sessions_send`.

## What This Changes

Clarify session status semantics so agents do not treat `failed`, `timeout`, or `done` as proof that a resumable session should be replaced.

## Best Possible Solution

Add backward-compatible resumability/status-scope metadata to session rows and agent tool output, then update tool descriptions, default prompt guidance, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list a session entry whose `status` is `failed`, `timeout`, or `done`; current main forwards that bare value through gateway and `sessions_list`, while `sessions.send` can still target an existing `sessionId`.

## Solution Assessment

No for the breaking rename proposed as the cleanest option. The safer fix is additive metadata and guidance that preserves the existing status union while making resumability explicit.
