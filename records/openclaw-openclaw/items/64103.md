## Summary

Keep open. Current main still exposes terminal-looking session status values through persisted session entries, gateway rows, session-change payloads, and agent-facing `sessions_list` without an explicit resumability or status-scope signal; the prompt/docs only discourage polling and still do not explain that `failed`, `timeout`, `done`, or `killed` are not necessarily a permanent no-resume lifecycle state.

## What This Changes

The issue requests clearer session status semantics so orchestrator agents do not mistake last-turn states such as `failed`, `timeout`, or `done` for non-resumable session lifecycle states.

## Best Possible Solution

Add a backward-compatible session row/tool contract that preserves existing `status` values while making resumability and status scope explicit, then update tool descriptions, system prompt guidance, and public session-tool docs so orchestrators prefer `sessions_send` for visible existing sessions unless the session is inaccessible, deleted, or policy-denied.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list or construct a visible session row with `status: "failed"`, `"timeout"`, or `"done"`; current main returns that bare status through `sessions_list` with no resumability/scope metadata, matching the agent-facing ambiguity in the report.

## Solution Assessment

Yes. The narrow maintainable solution is a non-breaking additive contract plus prompt/docs guidance; renaming the existing `status` values would be cleaner conceptually but riskier for current clients.
