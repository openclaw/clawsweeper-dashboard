## Summary

Keep open. Current main still exposes terminal-looking session status values through persisted session entries, gateway session rows, and the agent-facing `sessions_list` result without an explicit resumability or status-scope signal; the docs and default prompt also still lack the orchestration guidance requested here.

## What This Changes

The issue asks OpenClaw to make session status semantics clearer so orchestrator agents do not treat `failed`, `timeout`, or `done` as proof that a session cannot be resumed with `sessions_send`.

## Best Possible Solution

Add a backward-compatible session row/tool contract that preserves existing `status` values while making resumability and status scope explicit, then update tool descriptions, system prompt guidance, tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to construct or list a session row with `status: "failed"`, `"timeout"`, or `"done"`; current main forwards that bare value through gateway and `sessions_list` without companion resumability metadata.

## Solution Assessment

Yes. The best fix is additive metadata plus prompt/docs guidance; renaming existing status values would be conceptually cleaner but riskier for current clients.
