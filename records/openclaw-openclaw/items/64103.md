## Label Justifications

- `P2`: The issue describes a real bounded session-orchestration behavior that can waste model calls and lose continuity without making the core runtime unusable.
- `impact:session-state`: Agents can abandon resumable sessions and create duplicate state because the listed session state is ambiguous.

## Summary

Keep open. Current main still exposes terminal-looking session statuses to agents without explicit resumability or status-scope metadata, while the same sessions can still be targeted by `sessions_send`, so the reported session-state confusion remains actionable.

## What This Changes

The issue asks OpenClaw to clarify that session `status` values such as `failed`, `timeout`, and `done` describe the last turn outcome rather than whether the session can be resumed.

## Best Possible Solution

Keep the legacy `status` union for compatibility, add explicit agent-visible resumability/status-scope metadata, and update tool descriptions, docs, prompts, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level. Current `sessions_list` can still return `done`, `failed`, or `timeout` without resumability context, while `sessions.send` can target and reactivate existing completed subagent sessions.

## Solution Assessment

No for the proposed breaking rename. The safer fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.
