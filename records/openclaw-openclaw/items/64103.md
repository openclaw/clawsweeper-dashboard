## Summary

Keep open. Current main still exposes terminal-looking `status` values through the persisted session contract, Gateway rows, and `sessions_list` without a companion resumability or status-scope field, while `sessions.send` can still target an existing session by `sessionId` and reactivate completed sub-agent sessions.

## What This Changes

Clarify that session `status` values describe the latest run/turn state rather than permanent session lifecycle, so orchestrators resume existing sessions instead of spawning duplicates.

## Best Possible Solution

Add backward-compatible resumability/status-scope metadata to Gateway rows and `sessions_list`, keep the existing status union, and update agent guidance, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list a session row whose `status` is `done`, `failed`, or `timeout`: current main forwards that bare value through Gateway and `sessions_list`, while `sessions.send` still targets existing entries with a `sessionId` and can reactivate completed sub-agent sessions.

## Solution Assessment

No for the proposed breaking rename. The maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.
