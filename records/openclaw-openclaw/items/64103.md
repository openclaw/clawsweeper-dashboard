## Label Justifications

- `P2`: This is a bounded but real session-state bug that can cause duplicate subagent sessions and token waste without crashing core runtime.
- `impact:session-state`: The reported failure is agents misreading session state and abandoning resumable session context for duplicate sessions.

## Summary

Keep open: current main and the latest release still expose terminal-looking session `status` values without a model-visible resumability signal, while source shows ended subagent sessions can be reactivated. The open linked fix PR is the right active path, so this issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to make session status semantics clear so agents resume existing sessions with `sessions_send` instead of spawning duplicate subagent sessions.

## Best Possible Solution

Land a maintainer-approved, backward-compatible session list/gateway row contract that makes resumability explicit, refreshing the linked PR or replacing it narrowly if needed.

## Reproduction Assessment

Yes at the source-contract level: current main exposes terminal-looking last-run statuses, while the reactivation path accepts ended subagent runs and starts a replacement run. I did not run a live LLM orchestrator duplication scenario in this read-only review.

## Solution Assessment

Unclear until maintainers accept the exact API contract, but the additive `resumable` signal in the linked PR is a safer fix shape than renaming shipped status values. A broader lifecycle model would be more invasive and should not block this narrow bug path unless maintainers reject the additive contract.
