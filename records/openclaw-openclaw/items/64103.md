## Label Justifications

- `P2`: The issue is a bounded behavior bug that can waste paid model calls and fragment orchestration state without making the core runtime unusable.
- `impact:session-state`: Agents can abandon resumable sessions and create duplicate session state because the listed session status is ambiguous.

## Summary

Keep open. Current main still returns terminal-looking `status` values to agent-facing session list consumers without explicit resumability or status-scope metadata, while `sessions.send` can reactivate completed subagent sessions.

## What This Changes

The issue requests clearer session status semantics so agents do not treat `done`, `failed`, or `timeout` as proof that a session cannot be resumed.

## Best Possible Solution

Preserve the legacy `status` union for compatibility, add explicit resumability/status-scope metadata to session rows, and update tool descriptions, docs, prompts, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level. Current `sessions_list` can still expose `done`, `failed`, or `timeout` without resumability context, while `sessions.send` can target and reactivate an existing completed subagent session.

## Solution Assessment

No for the proposed breaking rename. The safer maintainable path is additive metadata and guidance that keeps existing clients compatible while making resumability explicit to agents.
