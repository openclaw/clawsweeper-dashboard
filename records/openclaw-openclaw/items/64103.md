## Summary

Keep open. Current main still exposes terminal-looking session `status` values through the persisted session contract, gateway rows, and `sessions_list`, while `sessions.send` can still target and reactivate an existing completed sub-agent session; the provided related reports do not supersede this narrower status/resumability contract issue.

## What This Changes

The issue asks OpenClaw to make session status output clearly distinguish last-run or lifecycle state from whether an existing session can be resumed with `sessions_send`.

## Best Possible Solution

Add backward-compatible resumability/status-scope metadata to gateway rows and `sessions_list`, keep the existing `status` values, and update tool descriptions, prompt guidance, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list a session row whose `status` is `done`, `failed`, or `timeout`: current main forwards that bare value through gateway and `sessions_list`, while `sessions.send` still targets existing entries with a `sessionId` and can reactivate completed sub-agent sessions.

## Solution Assessment

No for the proposed breaking rename. The maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.

## Repair Work Prompt

Fix the session status clarity bug reported at https://github.com/openclaw/openclaw/issues/64103. Current main exposes `running | done | failed | killed | timeout` through persisted sessions, gateway rows, and `sessions_list`, but sessions with terminal-looking statuses can still be resumed with `sessions_send`. Implement a backward-compatible additive contract; do not rename or remove the existing `status` values. Add explicit agent/client-facing metadata that makes sendability or status scope clear, deriving it from the same constraints `sessions.send` uses where practical and preserving thread-scoped/session visibility rules. Update `sessions_list` output types, gateway row types/builders, tool descriptions, default prompt guidance, docs, and focused regression tests. Add a `CHANGELOG.md` fix entry. Stop and ask for maintainer direction if the implementation would require a breaking status rename, a new config option, or a broad session lifecycle redesign.
