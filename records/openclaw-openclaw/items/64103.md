## Label Justifications

- `impact:session-state`: The reported failure is agents misreading session state and abandoning resumable session context for duplicate sessions.

## Summary

Keep open: current main and the latest release still expose terminal-looking session `status` values without a model-visible resumability signal, while source shows ended subagent sessions can be reactivated. The open linked fix path is https://github.com/openclaw/openclaw/pull/89101, so this issue should stay open until that PR or a narrow replacement lands.

## What This Changes

The issue asks OpenClaw to make session-list status semantics and resumability explicit so orchestrator agents resume existing subagent sessions instead of spawning duplicates.

## Best Possible Solution

Land a maintainer-approved, backward-compatible session list/gateway row contract that makes resumability explicit, using the linked PR or a narrow replacement if that branch cannot be refreshed safely.

## Reproduction Assessment

Yes at the source-contract level: current main exposes terminal-looking last-run statuses while the reactivation path accepts ended subagent runs and starts a replacement run. I did not run a live orchestrator duplicate-spawn scenario in this read-only review.

## Solution Assessment

Unclear until maintainers accept the exact API contract, but the additive `resumable` signal in the linked PR is safer than renaming shipped status values. A broader lifecycle model would be more invasive and should not block a narrow bug fix unless maintainers reject the additive contract.
