## Summary

Keep open. Current main still exposes terminal-looking session `status` values through persisted session entries, gateway session rows, and agent-facing `sessions_list`, while `sessions.send` can still target and reactivate existing sessions.

## What This Changes

The issue asks OpenClaw to clarify that session `status` values describe the last turn rather than session lifecycle, so agents do not spawn duplicate sessions.

## Best Possible Solution

Add backward-compatible resumability and status-scope metadata to gateway rows and `sessions_list`, keep the existing `status` union, and update tool descriptions, prompt guidance, docs, tests, and changelog together.

## Reproduction Assessment

Yes, at source level. A listed session row with `status: "done"`, `"failed"`, or `"timeout"` still carries that bare value through gateway and `sessions_list`, while `sessions.send` can target an existing `sessionId` and reactivate completed subagent sessions.

## Solution Assessment

No for the proposed breaking rename. The narrower maintainable fix is additive metadata and guidance that preserves the existing status union while making resumability explicit for agents and clients.
