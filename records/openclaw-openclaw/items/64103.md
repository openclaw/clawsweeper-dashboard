## Summary

Keep open: current main still exposes the ambiguous session `status` values without a `resumable` signal or model-facing status semantics, and the open closing PR owns the proposed fix but is still unmerged and conflict-marked.

## What This Changes

The issue asks OpenClaw to make session status output and guidance clear enough that agents resume existing sessions instead of spawning duplicates after `failed`, `timeout`, `done`, or `killed` last-turn states.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/89101, then land a backward-compatible resumability signal and status-semantics guidance before closing this issue.

## Reproduction Assessment

Yes, at source level: current main exposes only last-turn-looking `status` values while `sessions_send` can reactivate ended subagent runs. I did not run a live orchestrator flow that demonstrates duplicate spawning.

## Solution Assessment

Yes for the broad direction: additive `resumable` metadata and status guidance is safer than breaking status renames. The exact linked PR still needs conflict resolution and maintainer review before this issue can close.
