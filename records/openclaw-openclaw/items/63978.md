## Summary

Keep open. The bug is source-reproducible and the PR uses the right session-header mechanism, but the patch only guards one sync reader while other transcript-usage paths can still read stale usage after reset; it also lacks after-fix real behavior proof and the live PR is conflicting.

## What This Changes

The PR adds a first-line session header id check to `readLatestSessionUsageFromTranscript` plus one regression test for stale transcript usage after reset.

## Best Possible Solution

Add a shared conservative session-header validator before every transcript usage extraction path, with regression coverage for sync latest, async latest/recent, latest-recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can append a stale provided `sessionFile` candidate, and several usage readers extract transcript usage after path resolution without checking the first-line session header id.

## Solution Assessment

No. The header-id check is the right mechanism, but applying it only to one sync helper leaves other usage readers exposed; a shared validator at the resolved transcript read boundary is safer.
