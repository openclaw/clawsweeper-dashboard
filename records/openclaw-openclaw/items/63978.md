Keep this PR open: it targets a real stale-transcript usage bug, but the branch is now incomplete against current main because only one usage reader is guarded while session-list and async/bounded readers can still consume a stale transcript candidate. It also still needs contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds a first-line session-header check to `readLatestSessionUsageFromTranscript` and a regression test for stale usage after reset.

## Best Possible Solution

Rebase on current main, centralize conservative first-line session-header validation across all transcript usage readers, preserve headerless legacy transcripts, and add regression coverage for latest, async, and bounded session-list fallback paths.

## Reproduction Assessment

## Solution Assessment

No. The session-header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves bounded and async usage-reader paths unguarded.
