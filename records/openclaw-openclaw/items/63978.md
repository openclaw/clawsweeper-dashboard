## Summary

Keep open. The PR is a focused and useful fix, but it is incomplete against current main: session listing now uses the bounded transcript usage reader, and this patch only guards the full-file reader, so stale transcript usage can still reach a visible usage surface. The PR also needs the required user-facing changelog entry before landing.

## What This Changes

The PR adds a first-line session-header ID check to `readLatestSessionUsageFromTranscript` and a filesystem regression test for stale post-reset transcript usage.

## Best Possible Solution

Update the PR so the session-header validation is shared by both usage readers, preserving legacy/headerless behavior while rejecting parseable mismatched session headers in full-file and bounded reads, then add focused regression coverage and a single-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction is to make an old transcript discoverable via `sessionFile` with a first-line session header for session A, then request usage for session B; current main's full-file and bounded readers both extract usage without checking that header.

## Solution Assessment

No. The first-line header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` is no longer the narrow complete fix because current session listing uses `readRecentSessionUsageFromTranscript`.
