## Summary

Keep open: the reported stale-transcript bug is valid, but the PR is incomplete because it guards only one sync usage helper while other current-main usage readers can still consume a stale candidate; it also lacks required real behavior proof.

## What This Changes

The PR adds a first-line session-header id check to `readLatestSessionUsageFromTranscript` and a colocated regression test for stale post-reset transcript usage.

## Best Possible Solution

Add a shared conservative session-header id validator at the resolved transcript read boundary before every usage extraction path, with regression coverage for sync/async latest, recent/latest-recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can resolve a stale provided `sessionFile` candidate and then extract transcript usage without checking that the first-line session header id matches the requested session.

## Solution Assessment

No. The session-header check is the right mechanism, but applying it only to one sync helper leaves other usage readers exposed; a shared validator across all usage readers is the narrow maintainable fix.
