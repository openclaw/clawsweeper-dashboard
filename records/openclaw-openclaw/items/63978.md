## Summary

Keep open: the stale-transcript bug is valid, but this PR only guards one sync usage reader. Current main still has session-list and async usage paths that can read usage from a mismatched transcript, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a first-line transcript session-header check to `readLatestSessionUsageFromTranscript` and a regression test for a stale post-reset transcript candidate.

## Best Possible Solution

Add a shared conservative transcript-session-header validator before every transcript usage extraction path, with regression coverage for sync/async latest, recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can include a stale `sessionFile` candidate, and the transcript usage readers extract usage after `findExistingTranscriptPath` without checking that the first-line session header matches the requested session id.

## Solution Assessment

No. The header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves bounded and async usage-reader paths unguarded.
