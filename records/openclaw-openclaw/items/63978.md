## Summary

Keep open: the stale transcript usage bug is source-valid, but this branch is not ready because it guards only one transcript usage reader and lacks external after-fix real behavior proof.

## What This Changes

The PR adds a first-line session-header check to `readLatestSessionUsageFromTranscript` plus a regression test for a stale post-reset transcript file.

## Best Possible Solution

Centralize conservative session-header validation across the sync/async full-file and bounded usage readers after transcript path resolution, preserve headerless legacy transcripts, rebase the branch, and add focused coverage for latest/recent paths.

## Reproduction Assessment

Yes, at source level. Current main can resolve stale `sessionFile` candidates and its usage readers extract usage without checking the first-line session id; I did not run a filesystem repro because this review was read-only.

## Solution Assessment

No. The header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves the sessions-list bounded reader and async/recent readers exposed.
