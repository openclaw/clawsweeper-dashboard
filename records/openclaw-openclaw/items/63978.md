## Summary

Keep this PR open, but it still needs changes before merge. The stale-header check is a useful direction, yet the patch only guards the full-file usage reader while current session-list usage recovery goes through the bounded reader, and the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds a first-line session-header ID check to `readLatestSessionUsageFromTranscript` and a filesystem regression test for stale post-reset transcript usage.

## Best Possible Solution

Rebase the PR on current main, share one conservative session-header validation helper across full-file and bounded usage readers while preserving headerless legacy behavior, and add focused coverage for the bounded reader used by session listing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make a transcript with a first-line session header for session A discoverable through `sessionFile`, then request usage for session B; current main's full-file and bounded readers extract usage without checking that header.

## Solution Assessment

No. The first-line header check is the right mechanism, but applying it only to the full-file reader is incomplete because current session-list usage recovery uses the bounded reader.
