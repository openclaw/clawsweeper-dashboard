## Summary

Keep open: this is a plausible bug fix, but the patch only guards one transcript usage reader while current main still has other stale-candidate usage paths, and the PR lacks required after-fix real behavior proof.

## What This Changes

Adds a first-line session-header id check to `readLatestSessionUsageFromTranscript` and a regression test for stale transcript usage after reset.

## Best Possible Solution

Move the header-id validation into a shared helper used by all transcript usage extraction paths, then add regression coverage for latest, recent, async, latest-recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can append a stale provided `sessionFile` candidate, and multiple usage readers consume the resolved transcript without checking the first-line session header id.

## Solution Assessment

No. The session-header id check is the right mechanism, but applying it to only one sync helper is not the narrowest maintainable fix because other usage readers share the same stale-candidate boundary.
