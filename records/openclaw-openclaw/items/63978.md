## Summary

Keep open: the stale transcript bug is real, but the patch is incomplete because it only guards the sync latest usage reader while async, recent, and session-list usage readers still accept stale transcript candidates; the PR also lacks required real behavior proof.

## What This Changes

The PR adds a first-line session-header id check in `readLatestSessionUsageFromTranscript` and a regression test for stale transcript usage after reset.

## Best Possible Solution

Add a shared conservative session-header validator at the resolved transcript usage-read boundary and cover sync latest, async latest/recent, latest-recent, bounded recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can append a stale provided `sessionFile` candidate and the usage readers then extract usage without validating the first-line session header id.

## Solution Assessment

No. The header-id check is the right mechanism, but applying it to only one sync helper is not the narrowest maintainable fix because other usage readers remain exposed.
