## Summary

Keep open: the stale-transcript bug is valid and not fixed on current main, but this PR is incomplete against the current transcript usage surface and still lacks external real behavior proof.

## What This Changes

The PR adds a first-line session-header ID check to one transcript usage reader and a filesystem regression test for stale post-reset transcript usage.

## Best Possible Solution

Rebase on current main, centralize conservative session-header validation across the sync/async full-file and bounded usage readers, preserve headerless legacy behavior, and add regression coverage for the session-list fallback.

## Reproduction Assessment

Yes, at source level. A stale transcript with a first-line header for session A can still be selected for session B and current main's usage readers extract usage without checking the header id.

## Solution Assessment

No. The header check is the right mechanism, but this PR applies it only to one sync full-file reader while current main also uses bounded and async usage readers.
