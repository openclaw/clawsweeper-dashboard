## Summary

Keep open: the bug is source-backed and the PR uses the right session-header mechanism, but it only guards the sync latest usage reader while current main also has async and recent/session-list usage paths that can resolve the same stale transcript candidate.

## What This Changes

The PR adds a first-line session header id check to `readLatestSessionUsageFromTranscript` and a regression test for stale transcript usage after reset.

## Best Possible Solution

Add a shared conservative session-header validator before every transcript usage extraction path, with regression coverage for sync latest, async latest/recent, latest-recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can append a stale provided `sessionFile` candidate, and usage readers extract transcript usage after path resolution without checking the first-line session header id.

## Solution Assessment

No. The header-id check is the right mechanism, but applying it only to one sync helper leaves other usage readers exposed; a shared validator at the resolved transcript read boundary is safer.
