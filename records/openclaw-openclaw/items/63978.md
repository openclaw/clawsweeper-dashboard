## Summary

Keep open: the bug direction is valid, but the patch only guards one usage reader while current main still has unguarded async/recent and session-list fallback paths, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a first-line transcript session-header check to `readLatestSessionUsageFromTranscript` and a regression test for stale post-reset transcript usage.

## Best Possible Solution

Add a shared conservative session-header validator at the transcript usage-read boundary, cover every sync/async latest and recent usage reader, and coordinate with the reset-path work in https://github.com/openclaw/openclaw/pull/77772.

## Reproduction Assessment

Yes, at source level. Current main can append a stale provided `sessionFile` candidate and the usage readers then extract usage without checking the transcript header id against the requested session id.

## Solution Assessment

No. The header-id check is the right mechanism, but applying it only to the sync latest helper leaves the async/latest-recent/recent and session-list fallback readers exposed.
