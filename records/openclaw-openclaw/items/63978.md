## Summary

Keep open: the reported stale-transcript bug is valid and the header check is the right direction, but this patch only guards one usage reader while current main has bounded and async usage readers that can still read mismatched transcript headers; the external PR also lacks real behavior proof before merge.

## What This Changes

The PR adds a first-line transcript session-header check to `readLatestSessionUsageFromTranscript` and a regression test for stale usage after reset.

## Best Possible Solution

Rebase on current main, centralize conservative session-header validation across the sync/async full-file and bounded usage readers, preserve headerless legacy behavior, and cover the session-list fallback.

## Reproduction Assessment

Yes, at source level. Current main can resolve stale generated transcript candidates and its usage readers extract usage without checking the first-line session id; I did not run a filesystem repro because this review had to remain read-only.

## Solution Assessment

No. The header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves current bounded and async usage readers unguarded.
