## Summary

Keep open: the PR fixes the right class of stale-transcript bug for one helper, but current main has other transcript usage readers and the session-list fallback still able to consume a mismatched transcript candidate.

## What This Changes

The PR adds a first-line session-header mismatch check to `readLatestSessionUsageFromTranscript` and a regression test for stale usage after reset.

## Best Possible Solution

Rebase on current main and add a shared conservative transcript-session-header validator used by all transcript usage readers, with regression coverage for latest, async, and bounded session-list fallback paths.

## Reproduction Assessment

Yes, at source level. Current main can resolve stale transcript candidates, and the usage readers extract transcript usage without checking that the first-line session header matches the requested session id.

## Solution Assessment

No. The header check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves bounded and async usage-reader paths unguarded.
