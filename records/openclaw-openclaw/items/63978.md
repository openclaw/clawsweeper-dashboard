## Summary

Keep open: the reported stale-transcript bug is source-reproducible, but this PR only guards one sync usage reader while async, recent, and session-list fallback usage reads can still extract usage from a stale candidate transcript. The external contributor also has not provided after-fix real behavior proof.

## What This Changes

The PR adds a first-line session-header mismatch check to `readLatestSessionUsageFromTranscript` and a colocated regression test for stale transcript usage after reset.

## Best Possible Solution

Add a shared conservative session-header validator before every transcript usage extraction path, with regression coverage for sync, async, recent, and session-list fallback reads.

## Reproduction Assessment

Yes, at source level. Current main can still include stale `sessionFile` candidates, and the usage readers extract transcript usage after path resolution without comparing the first-line session header id to the requested session id.

## Solution Assessment

No. The header-id check is the right mechanism, but applying it only to `readLatestSessionUsageFromTranscript` leaves other transcript usage readers exposed.
