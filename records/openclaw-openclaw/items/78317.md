## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle path-based upload-file inputs:** `extensions/imessage/src/actions.ts:607-615`
  - body: This adapter advertises `upload-file`, but the generic message action contract and docs allow `media`, `path`, or `filePath` while this branch still requires a base64 `buffer` plus `filename`. A normal `message({ action: "upload-file", filePath: ... })` call will throw before reaching imsg; resolve the file through the media access helpers or stop advertising the generic action.
  - confidence: 0.89
- **[P2] Keep formatting off the RPC send path:** `extensions/imessage/src/send.ts:216-237`
  - body: `sendMessageIMessage` now strips Markdown and sends `params.formatting` through regular JSON-RPC `send`, but upstream imsg only documents `text/file/service/region/chat target` for RPC `send`; formatting ranges are applied through `send-rich --format`. Basic sends like `**bold**` therefore become unstyled `bold` instead of preserving the original text or using the supported rich path.
  - confidence: 0.87
- **[P3] Add labeler coverage for the migration page:** `.github/labeler.yml:55-59`
  - body: This PR adds `docs/channels/imessage-from-bluebubbles.md`, but the iMessage label rule still only matches the existing iMessage doc. Add the migration page to the same labeler globs so future docs edits route to the channel owners.
  - confidence: 0.82
- **[P3] Narrow the cache-mode security claim:** `extensions/imessage/src/monitor-reply-cache.ts:132-138`
  - body: The comment says owner-only file modes close a same-UID read/injection vector, but `0600` and `0700` do not stop another process running as the same Unix user from reading or writing those files. Narrow the claim to other local users, or add an actual integrity boundary if same-UID tampering remains in scope.
  - confidence: 0.86
