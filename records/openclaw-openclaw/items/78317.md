## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Handle file paths for upload-file:** `extensions/imessage/src/actions.ts:609-614`
  - body: This adapter advertises `upload-file`, but the shared message tool schema lets agents pass `media`, `path`, or `filePath` and this handler only accepts a base64 `buffer`. A normal `message({ action: "upload-file", filePath: ... })` call will throw before reaching `imsg`; resolve file/media inputs through the provided media access helpers or stop advertising `upload-file`.
  - confidence: 0.87
- **[P2] Keep formatting off the plain send path:** `extensions/imessage/src/send.ts:220-238`
  - body: `sendMessageIMessage` now strips Markdown markers and adds `params.formatting` to regular RPC `send`, but upstream `imsg` only applies formatting through `send-rich --format`; RPC `send` ignores this field. On basic/SIP-on sends, `**bold**` becomes plain `bold`, so either preserve the original Markdown or route formatted sends through the supported bridge path.
  - confidence: 0.88
- **[P3] Add labeler coverage for the migration page:** `.github/labeler.yml:55-59`
  - body: This PR adds `docs/channels/imessage-from-bluebubbles.md`, but the iMessage label rule still only matches the existing iMessage doc. Add the new migration page to the relevant labeler globs so future edits route to the channel owners.
  - confidence: 0.81
- **[P3] Correct the same-UID cache-hardening claim:** `extensions/imessage/src/monitor-reply-cache.ts:132-138`
  - body: The new comment says owner-only file modes close a same-UID read/injection vector, but `0600`/`0700` do not stop another process running as the same Unix user from reading or writing those files. Narrow the claim to other local users, or add a real integrity boundary if same-UID tampering remains in scope.
  - confidence: 0.86
