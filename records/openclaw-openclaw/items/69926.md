## What This Changes

The issue requests optional per-group WhatsApp `allowFrom` at root and account scope so sender authorization can vary by group instead of using one channel-wide `groupAllowFrom`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81053 or a narrow replacement that adds the WhatsApp-owned schema, type, runtime, docs, generated metadata, and regression coverage while preserving channel-level fallback.

## Reproduction Assessment

Yes for the feature gap: current main’s strict WhatsApp group schema omits `allowFrom`, and the inbound/command authorization path passes only the resolved account/channel `groupAllowFrom`. I did not run tests in this read-only review.

## Solution Assessment

Yes. The additive group-level override is the narrowest maintainable direction because it stays within the WhatsApp plugin/config surface and mirrors existing group prior art without changing unrelated admission or denylist behavior.
