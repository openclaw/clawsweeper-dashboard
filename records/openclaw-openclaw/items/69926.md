## Summary

Keep open. The prior ClawSweeper assessment still matches current main: WhatsApp group entries do not accept per-group `allowFrom`, and runtime sender authorization still resolves one account/channel-level group allowlist rather than a group-specific override.

## What This Changes

Request adds optional WhatsApp `groups.<jid>.allowFrom` and account-scoped equivalents to override channel-level `groupAllowFrom` per group.

## Best Possible Solution

Add a WhatsApp-owned optional per-group sender allowlist in types, strict schema, generated metadata, runtime authorization, tests, docs, and changelog while preserving existing channel-level fallback when the group entry omits `allowFrom`.

## Reproduction Assessment

Yes for the feature gap. Static source inspection on current main shows the config type/schema reject `groups.<jid>.allowFrom`, and both inbound and command authorization use only the resolved account/channel-level group allowlist.

## Solution Assessment

Yes. The requested additive group-level override is the narrowest maintainable direction because it mirrors existing channel prior art without changing WhatsApp group admission, denylist behavior, or global account inheritance semantics.
