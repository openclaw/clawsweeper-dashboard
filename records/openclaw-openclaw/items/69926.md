## Summary

Keep open. Current main still lacks WhatsApp per-group sender authorization: the WhatsApp group config type, strict schema, generated metadata, inbound access gate, and command authorization path all use only account/channel-level `groupAllowFrom` with `allowFrom` fallback. The related WhatsApp system-prompt work, multi-account inheritance discussion, and deny/blocklist PR do not supersede this narrower feature gap.

## What This Changes

The issue requests optional WhatsApp per-group `allowFrom` sender authorization at root and account-scoped group config entries, with a defined group value overriding channel-level `groupAllowFrom`.

## Best Possible Solution

Implement an additive WhatsApp plugin change that adds `allowFrom` to root and account-scoped group entries, treats a defined group value as the sender-authority override for that group, preserves existing `groupAllowFrom` and `allowFrom` fallback when undefined, updates generated config metadata and docs, and covers both normal inbound replies and command authorization.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: `channels.whatsapp.groups.<jid>.allowFrom` is absent from the public type and strict schema, and the runtime never looks up the matched group entry when deciding sender authorization.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable solution because it mirrors existing channel prior art while keeping WhatsApp group admission, sender authorization, and the separate multi-account inheritance issue distinct.
