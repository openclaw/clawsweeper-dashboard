## Summary

Keep open: current main still has the unconditional Kubernetes ConfigMap-to-PVC copy path, but this PR's one-time copy guard conflicts with the documented ConfigMap redeploy workflow and lacks real Kubernetes after-fix proof.

## What This Changes

The PR changes the Kubernetes init container to use `set -eu` and copy `openclaw.json` and `AGENTS.md` from the ConfigMap to the PVC only when destination files are missing.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then implement it consistently with docs: either seed once with reset/resync guidance, or preserve ConfigMap redeploy updates with backup/conflict handling.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main copies ConfigMap files into the PVC on every init-container run, so a pod restart overwrites PVC edits; I did not run a real Kubernetes rollout.

## Solution Assessment

No. The one-time copy guard fixes the preservation case but is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:35`

This breaks the documented redeploy/update workflow and can leave stale settings in production.
