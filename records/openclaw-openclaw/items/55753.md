## Summary

Keep open: the current Kubernetes overwrite path is source-reproducible, but this PR changes the documented ConfigMap redeploy contract and can leave security-relevant config stale, so maintainers need to choose the config ownership/sync policy before merge.

## What This Changes

The PR changes the Kubernetes init container to enable `set -eu` and copy `openclaw.json` and `AGENTS.md` from the ConfigMap only when the PVC destination files are missing.

## Best Possible Solution

Make the Kubernetes config ownership contract explicit, then implement it consistently in the manifest and docs with either documented seed-once reset guidance or redeploy-safe backup/conflict handling.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main overwrites the PVC-backed files on each init-container run, while the PR guard would skip later ConfigMap copies once those files exist.

## Solution Assessment

No. The guard is a narrow preservation mechanism, but it is not the best complete fix until maintainers decide whether Kubernetes config is ConfigMap-declarative or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34-36`
  - body: The docs tell operators to edit `scripts/k8s/manifests/configmap.yaml` and rerun `deploy.sh`, which applies manifests and restarts the deployment. After this guard creates the PVC files once, later ConfigMap changes to `openclaw.json` or `AGENTS.md` are ignored, leaving stale gateway/auth or agent instructions in place after a normal redeploy.
  - confidence: 0.9
