## Summary

Keep open: the PR fixes the PVC overwrite symptom, but it also changes Kubernetes config ownership from ConfigMap-driven redeploys to one-time seeding without maintainer direction or real rollout proof.

## What This Changes

The PR adds `set -eu` and guards the Kubernetes init-container copies of `openclaw.json` and `AGENTS.md` so existing PVC files are not overwritten.

## Best Possible Solution

Define the Kubernetes config ownership contract, then implement either seed-once behavior with reset/resync guidance or ConfigMap sync with backup/conflict handling across the manifest, deploy script, and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main copies ConfigMap files into the PVC on every init-container run, so a pod restart overwrites PVC edits; I did not run a real Kubernetes rollout.

## Solution Assessment

No. The one-time guard fixes the preservation case, but it is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34`
  - body: The new guard makes ConfigMap copies one-time-only: after the first pod start, later `deploy.sh` runs apply the ConfigMap and restart the deployment but never propagate updated `openclaw.json` or `AGENTS.md` into the PVC. This breaks the documented redeploy/update workflow and can leave stale settings in production.
  - confidence: 0.9
