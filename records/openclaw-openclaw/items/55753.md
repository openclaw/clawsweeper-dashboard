## Summary

Keep this PR open: it addresses a real Kubernetes PVC overwrite problem, but it intentionally changes the ConfigMap-to-PVC ownership contract and still lacks usable real Kubernetes rollout proof.

## What This Changes

The PR changes the Kubernetes deployment init container to use `set -eu` and seed `openclaw.json` and `AGENTS.md` from the ConfigMap only when the PVC files are missing.

## Best Possible Solution

Define the Kubernetes config ownership contract, then either document seed-once PVC ownership with explicit reset/resync steps or implement safe reconciliation that preserves runtime edits without silently ignoring ConfigMap updates.

## Reproduction Assessment

Yes, source-reproducible: current main unconditionally copies the ConfigMap files into the PVC on each init-container run, and the PR would skip those copies after the files exist. I did not run a live Kubernetes rollout.

## Solution Assessment

No. The guard is a reasonable seed-once implementation if maintainers choose PVC ownership, but the PR is not the best complete fix until docs, reset guidance, and proof cover both fresh install and existing-PVC redeploy behavior.
