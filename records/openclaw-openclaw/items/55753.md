## Summary

Keep open: the PR addresses a plausible overwrite/data-loss concern, but it changes the Kubernetes config ownership contract and currently has a blocking ConfigMap redeploy regression plus missing real rollout proof. The author acknowledged the ConfigMap trade-off, so this needs maintainer direction rather than cleanup close.

## What This Changes

The PR changes the Kubernetes init container to use `set -eu` and copy `openclaw.json` and `AGENTS.md` from the ConfigMap into the PVC only when the destination file is absent.

## Best Possible Solution

Define the Kubernetes config ownership contract, then implement and document either seed-once PVC ownership with reset/resync guidance or ConfigMap reconciliation with backup/conflict handling.

## Reproduction Assessment

Yes, source-reproducible: current `main` unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not establish a real Kubernetes rollout reproduction.

## Solution Assessment

No. The guard preserves PVC edits, but it is not the best complete solution until maintainers decide whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34-39`
  - body: The new existence guards make ConfigMap-to-PVC copies one-time-only. `deploy.sh` still applies the ConfigMap and restarts the deployment to pick up config changes, so later `openclaw.json` or `AGENTS.md` updates will not reach the PVC and production can run stale settings.
  - confidence: 0.9
