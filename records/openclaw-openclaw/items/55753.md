## Summary

Keep open: the PR addresses a real current-main overwrite path, but the patch changes the Kubernetes config ownership contract by making ConfigMap-to-PVC copies one-time-only, which conflicts with the documented redeploy workflow and still needs maintainer judgment plus real rollout proof.

## What This Changes

The PR adds `set -eu` and file-existence guards around the Kubernetes init-container copies of `openclaw.json` and `AGENTS.md` from the ConfigMap into the PVC.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then align manifests and docs around either seed-once PVC ownership with reset/resync guidance or declarative ConfigMap updates with backup/conflict handling.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main copies ConfigMap files into the PVC on every init-container run, so a pod restart can overwrite PVC edits; I did not run a live Kubernetes rollout in this read-only review.

## Solution Assessment

No. The guard fixes the local-edit preservation case, but it is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34-35`
  - body: The Kubernetes guide tells operators to edit `scripts/k8s/manifests/configmap.yaml` and rerun `deploy.sh`, which applies manifests and restarts the deployment. After this guard creates the PVC files once, later ConfigMap changes to `openclaw.json` or `AGENTS.md` are ignored, so redeploys can leave stale gateway/auth settings in place.
  - confidence: 0.9
