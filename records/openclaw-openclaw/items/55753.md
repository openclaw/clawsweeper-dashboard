## Summary

Keep open: source inspection confirms the data-loss motivation, but the patch changes the documented ConfigMap redeploy contract and still lacks real Kubernetes rollout proof.

## What This Changes

The PR changes the Kubernetes deployment init container to seed `openclaw.json` and `AGENTS.md` from the ConfigMap only when the PVC files are missing, adding `set -eu` and comments about PVC ownership.

## Best Possible Solution

Define the Kubernetes config ownership contract, then either document seed-once PVC ownership with explicit reset/resync steps or implement safe reconciliation that preserves runtime edits without silently ignoring ConfigMap updates.

## Reproduction Assessment

Yes, source-reproducible: current main unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not establish a live Kubernetes rollout reproduction.

## Solution Assessment

No, not yet. The guard is only the best fix if maintainers choose PVC ownership; the PR still needs the documented ownership contract, reset/resync guidance, and real rollout proof.
