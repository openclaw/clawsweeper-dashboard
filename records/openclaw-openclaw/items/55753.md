## Summary

Keep open: source inspection confirms the overwrite problem, but this PR changes the Kubernetes config ownership contract by making ConfigMap-to-PVC copies one-time-only. That conflicts with the documented redeploy/update path, has security impact for stale gateway/auth settings, and the contributor has not provided real after-fix Kubernetes rollout proof.

## What This Changes

The PR changes the Kubernetes init container to add `set -eu` and copy `openclaw.json` and `AGENTS.md` from the ConfigMap to the PVC only when the destination files do not already exist.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then implement and document either seed-once behavior with reset/resync guidance or ConfigMap reconciliation with backup/conflict handling.

## Reproduction Assessment

Yes, by source inspection: current main copies ConfigMap files into the PVC on every init-container run, and `deploy.sh` restarts the deployment after applying manifests. I did not run a real Kubernetes rollout.

## Solution Assessment

No. The one-time guard fixes the PVC-preservation case, but it is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings
