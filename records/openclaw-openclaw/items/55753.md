## Summary

Keep open. The overwrite problem is source-reproducible, but this patch changes the Kubernetes config ownership contract and can silently stop documented ConfigMap redeploy updates from reaching existing PVC-backed deployments; it also lacks real Kubernetes after-fix proof.

## What This Changes

The PR adds `set -eu` and file-existence guards around the Kubernetes init-container copies for `openclaw.json` and `AGENTS.md`.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then implement that contract consistently with docs: either seed-once with reset/resync guidance, or preserve ConfigMap redeploy updates with backup/conflict handling.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current `main` copies ConfigMap files into the PVC on every init-container run, so a pod restart overwrites PVC edits. I did not run a real Kubernetes rollout in this review.

## Solution Assessment

No. The guard fixes one preservation case, but it is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34`

After this guard creates the PVC files once, later ConfigMap changes to `openclaw.json` or `AGENTS.md` are ignored, so redeploys can leave stale gateway/auth settings in place.
