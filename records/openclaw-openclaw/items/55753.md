## Summary

Keep open: the PR addresses a real overwrite problem, but the current patch changes the Kubernetes config ownership model without docs, reset/resync guidance, or real rollout proof. The author acknowledged the ConfigMap-to-PVC trade-off, so this needs maintainer product/infra judgment before merge.

## What This Changes

The PR changes the Kubernetes deployment init container to seed `openclaw.json` and `AGENTS.md` from ConfigMap only when the PVC files are missing.

## Best Possible Solution

Define the Kubernetes config ownership contract, then either document seed-once PVC ownership with explicit reset/resync steps or implement safe reconciliation that preserves runtime edits without silently ignoring ConfigMap updates.

## Reproduction Assessment

Yes, source-reproducible: current main unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not establish a live Kubernetes rollout reproduction.

## Solution Assessment

No, not yet. The guard is a reasonable seed-once implementation only if maintainers choose PVC ownership, but the complete fix needs docs, reset guidance, and proof for both fresh install and existing-PVC redeploy behavior.
