## Summary

Keep this PR open: it targets a plausible Kubernetes data-loss problem, but the branch changes the config ownership contract in a way that conflicts with the documented ConfigMap redeploy/update flow and needs maintainer review before merge.

## What This Changes

The PR changes the Kubernetes init container script to add `set -eu` and only copy `openclaw.json` and `AGENTS.md` from the ConfigMap into the PVC when the destination files do not already exist.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then implement and document either seed-once behavior with reset/resync guidance or ConfigMap reconciliation with backup/conflict handling.

## Reproduction Assessment

Yes, by source inspection: current `main` unconditionally copies ConfigMap files into the PVC on every init-container run, and the PR would skip later ConfigMap updates after the files exist. I did not run a real Kubernetes rollout.

## Solution Assessment

No. The one-time guard addresses PVC preservation, but it is not the best complete fix until maintainers decide whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings
