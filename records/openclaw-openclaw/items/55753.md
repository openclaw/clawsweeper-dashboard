## Summary

Keep open: the change is focused, but it is not merge-ready because the one-time copy guard conflicts with the documented ConfigMap redeploy workflow and the PR still lacks real Kubernetes rollout proof.

## What This Changes

The PR changes the Kubernetes init container to add `set -eu` and copy `openclaw.json` and `AGENTS.md` from the ConfigMap into the PVC only when the destination files are absent.

## Best Possible Solution

Define the Kubernetes config ownership contract, then implement and document either seed-once PVC ownership with reset/resync guidance or ConfigMap reconciliation with backup/conflict handling.

## Reproduction Assessment

Yes, source-reproducible: current `main` unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not establish a real Kubernetes rollout reproduction.

## Solution Assessment

No. The guard addresses PVC preservation, but it is not the best complete solution until maintainers decide whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34-39`
  - body: The new existence guards make ConfigMap copies one-time-only. After the first pod start, `deploy.sh` still applies the ConfigMap and restarts the deployment to pick up config changes, but updated `openclaw.json` or `AGENTS.md` will not reach the PVC, leaving stale settings in production.
  - confidence: 0.9
