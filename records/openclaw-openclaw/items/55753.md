## Summary

Keep open: the PR targets a real Kubernetes PVC overwrite/data-loss concern, but it still changes the config ownership contract without maintainer acceptance, docs/reset guidance, or real rollout proof, while current docs say ConfigMap edits plus redeploy should be picked up.

## What This Changes

The PR changes the Kubernetes init container to seed `openclaw.json` and `AGENTS.md` from the ConfigMap only when missing, adds `set -eu`, and comments that the PVC owns those files after initial seed.

## Best Possible Solution

Define the Kubernetes config ownership contract, then either document seed-once PVC ownership with explicit reset/resync steps or implement safe reconciliation that preserves runtime edits without silently ignoring ConfigMap updates.

## Reproduction Assessment

Yes, source-level: current main unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not run a live Kubernetes rollout.

## Solution Assessment

No, not yet. The guard is a reasonable seed-once implementation only if maintainers choose PVC ownership, but current docs and deploy behavior still promise ConfigMap redeploy pickup and need docs/reset guidance or reconciliation.
