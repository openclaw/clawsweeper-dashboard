## Summary

Keep open. Current main still has the restart clobbering behavior, but the PR's destination-exists guard is not merge-ready because it turns documented ConfigMap updates into seed-once PVC state; the author acknowledged that trade-off, so maintainers need to choose the Kubernetes config ownership contract.

## What This Changes

The PR adds `set -eu` and guards the Kubernetes init-container copies of `openclaw.json` and `AGENTS.md` so ConfigMap defaults are copied only when PVC files are absent.

## Best Possible Solution

A complete fix should make the Kubernetes manifest and docs agree on either declarative ConfigMap sync with backup/conflict handling, or documented seed-once PVC ownership with reset/resync guidance.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path, though I did not run a cluster. Current main copies from ConfigMap on every init-container run, and the PR guard would skip later ConfigMap copies once files exist.

## Solution Assessment

No. The file-exists guard fixes restart clobbering narrowly, but it is not the best complete solution because it silently breaks the documented ConfigMap redeploy/update contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34`
  - body: The Kubernetes docs tell operators to edit `scripts/k8s/manifests/configmap.yaml` and redeploy, and `deploy.sh` applies the manifests before restarting the deployment. Once the PVC already has these files, this guard skips every later ConfigMap copy, so updates to `openclaw.json` or `AGENTS.md` silently stop applying to existing deployments.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff needs attention because the one-time guard can leave existing Kubernetes deployments on stale gateway/auth settings after a ConfigMap hardening update.

Concerns:

- **[medium] Config hardening updates can be ignored:** `scripts/k8s/manifests/deployment.yaml:34`
  - body: `openclaw.json` carries gateway bind/auth/control UI settings, and the documented Kubernetes workflow updates it through ConfigMap plus redeploy. The new destination-file guard can leave an existing PVC-backed deployment running older security settings after operators apply a manifest update.
  - confidence: 0.82
