## Summary

Keep open: the current main behavior can overwrite PVC-edited Kubernetes config on init-container restart, but this PR trades that for stale ConfigMap updates on redeploy and needs maintainer direction plus real rollout proof before merge.

## What This Changes

The PR changes the Kubernetes init container to add `set -eu` and only copy `openclaw.json` and `AGENTS.md` from the ConfigMap to the PVC when destination files are missing.

## Best Possible Solution

Define the Kubernetes config ownership contract explicitly, then implement and document either seed-once behavior with reset/resync guidance or ConfigMap reconciliation with backup/conflict handling.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main copies ConfigMap files into the PVC on every init-container run, so a pod restart overwrites PVC edits; I did not run a real Kubernetes rollout.

## Solution Assessment

No. The one-time copy guard fixes the preservation case but is not the best complete solution until maintainers choose whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap updates on redeploy:** `scripts/k8s/manifests/deployment.yaml:34`
  - body: The new guard makes ConfigMap copies one-time-only: after the first pod start, later `deploy.sh` runs apply the ConfigMap and restart the deployment but never propagate updated `openclaw.json` or `AGENTS.md` into the PVC. This breaks the documented redeploy/update workflow and can leave stale settings in production.
  - confidence: 0.9
