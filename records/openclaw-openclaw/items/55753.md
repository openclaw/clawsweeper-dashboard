## Summary

Keep open. Current main still has the Kubernetes init-container overwrite behavior, but this PR's file-exists guard is not merge-ready because it changes the documented ConfigMap redeploy contract and can leave stale gateway configuration in existing PVC-backed deployments.

## What This Changes

The PR adds `set -eu` and changes the Kubernetes init container in `scripts/k8s/manifests/deployment.yaml` to copy ConfigMap-backed `openclaw.json` and `AGENTS.md` only when the destination files are absent.

## Best Possible Solution

Define the Kubernetes config ownership model explicitly and update manifest logic plus docs together: either keep the ConfigMap as the declarative source with backup/conflict-aware preservation, or make these files seed-once PVC state with a documented reset/resync workflow for operators.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main overwrites PVC-backed files on every init-container run, while the PR's `-f` guard would skip all future ConfigMap updates once those files exist.

## Solution Assessment

No. The destination-exists guard is a narrow fix for restart clobbering, but it is not the best complete solution because it breaks the documented ConfigMap redeploy/update behavior; a safer fix needs explicit sync semantics or a documented seed-once contract.

## Review Findings

Overall confidence: 0.91

- **[P1] Preserve ConfigMap rollout updates:** `scripts/k8s/manifests/deployment.yaml:34`

Once the PVC already has these files, this guard skips every later ConfigMap copy, so updates to `openclaw.json` or `AGENTS.md` silently stop applying to existing deployments.
