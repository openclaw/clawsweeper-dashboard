- `merge-risk: 🚨 compatibility`: Merging changes the documented ConfigMap redeploy workflow by making existing PVC files block future manifest updates.

## Summary

Keep open: the patch addresses a real overwrite hazard, but it changes the Kubernetes deployment contract from ConfigMap-driven redeploys to seed-once PVC ownership without maintainer agreement or real rollout proof.

## What This Changes

The PR updates the Kubernetes init container to use `set -eu` and copy ConfigMap-provided `openclaw.json` and `AGENTS.md` into the PVC only when the destination files do not already exist.

## Best Possible Solution

Define the Kubernetes config ownership contract, then implement and document either seed-once PVC ownership with reset/resync guidance or conflict-aware ConfigMap reconciliation that preserves user edits safely.

## Reproduction Assessment

Yes, source-reproducible: current `main` unconditionally copies ConfigMap files into the PVC on every init-container run, and the PR would skip those copies once the files exist. I did not establish a live Kubernetes rollout reproduction.

## Solution Assessment

No. The existence guard prevents one overwrite mode, but it is not the best complete fix until maintainers decide whether Kubernetes config is declarative ConfigMap state or PVC-owned runtime state.
