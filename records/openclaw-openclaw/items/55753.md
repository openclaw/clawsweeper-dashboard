## Summary

Keep open: the data-loss motivation is source-reproducible, but this PR changes the documented Kubernetes ConfigMap redeploy contract and still lacks real rollout proof. The author’s follow-up acknowledges the ConfigMap-vs-PVC trade-off, but the manifest/docs do not yet define the new ownership or reset path.

## What This Changes

The PR changes the Kubernetes deployment init container to seed `openclaw.json` and `AGENTS.md` from the ConfigMap only when the PVC files are missing.

## Best Possible Solution

Define the Kubernetes config ownership contract, then either document seed-once PVC ownership with explicit reset/resync steps or implement safe reconciliation that preserves runtime edits without silently ignoring operator-applied ConfigMap changes.

## Reproduction Assessment

Yes, source-reproducible: current main unconditionally copies ConfigMap files into the PVC on each init-container run, and the PR would skip those copies once files exist. I did not establish a live Kubernetes rollout reproduction.

## Solution Assessment

No, not yet. The guard is reasonable only if maintainers choose PVC ownership, but the complete fix needs docs, reset/resync guidance, and proof for fresh install plus existing-PVC redeploy behavior.
