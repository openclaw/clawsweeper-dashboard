## Summary

Keep open: the patch addresses a real overwrite/data-loss mode, but it also conflicts with the documented Kubernetes redeploy flow by making ConfigMap-sourced `openclaw.json` and `AGENTS.md` one-time seeds. The existing review thread already identified this trade-off and the author acknowledged it, so maintainer judgment is needed before changing the k8s config ownership contract.

## What This Changes

The PR adds `set -eu` and wraps the Kubernetes init-container copies of `openclaw.json` and `AGENTS.md` with destination-exists checks in `scripts/k8s/manifests/deployment.yaml`.

## Best Possible Solution

Define the Kubernetes config ownership model explicitly and update script plus docs together: either keep ConfigMap as the declarative source with a preservation/backup or conflict-aware update mechanism, or make these files seed-once PVC state with an explicit reset/resync workflow for operators.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: after first pod start creates the files on the PVC, the PR's `-f` guards skip future copies, while `deploy.sh` and the docs expect ConfigMap edits to propagate on redeploy.

## Solution Assessment

No. A simple destination-exists guard is not the best fix because it prevents the documented ConfigMap redeploy workflow from updating existing deployments; a safer fix needs explicit sync semantics or a documented seed-once contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve ConfigMap rollout updates:** `scripts/k8s/manifests/deployment.yaml:34`
  - body: The Kubernetes docs tell operators to edit `scripts/k8s/manifests/configmap.yaml` and redeploy, and `deploy.sh` applies kustomize then restarts the deployment. Once a PVC already has these files, this guard skips every later ConfigMap copy, so updates to `openclaw.json` or `AGENTS.md` silently stop applying to existing deployments.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete security-sensitive stale-config risk by preventing later ConfigMap hardening changes from being copied into existing PVC-backed deployments.

Concerns:

  - body: Because `openclaw.json` includes gateway/auth/bind settings and the documented k8s path updates it through the ConfigMap plus redeploy, the new destination-file guard can leave an existing deployment running older security settings after operators apply a manifest update.
  - confidence: 0.82
