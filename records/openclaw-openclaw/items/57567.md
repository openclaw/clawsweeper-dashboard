## Summary

Keep open. The current main checkout still rejects legacy Feishu account configs with `botName`, has no Feishu doctor compatibility normalizer to rewrite that field, and only repairs unparseable config through prefix stripping or a promoted `.last-good` snapshot rather than the numbered `.bak` rollback requested in this issue; the narrower Feishu fix path is still represented by the open pull request at https://github.com/openclaw/openclaw/pull/65384.

## What This Changes

The issue asks for robust upgrade recovery from v2026.3.24 to v2026.3.28, including valid config migration output, Feishu legacy schema migration, automatic backup rollback, clearer `doctor --fix` repair, and documented config changes.

## Best Possible Solution

Land or replace the Feishu-owned doctor migration, then separately scope a conservative doctor recovery policy for restoring a validated recent backup when `openclaw.json` is unparseable, leaving any auth-profile or memory-loss claims to narrower follow-up reports with logs.

## Reproduction Assessment

Yes. A legacy config with `channels.feishu.accounts.main.botName` is source-reproducible because current Feishu account validation is strict and no Feishu doctor normalizer rewrites the field; the corrupted-JSON recovery gap is also source-traceable because doctor repair does not select numbered `.bak` backups.

## Solution Assessment

No, closing this issue now would be premature. The best repair direction is the Feishu plugin doctor migration plus a separately reviewed backup-restore policy, not treating current last-known-good recovery as a complete fix for the reported numbered-backup rollback behavior.
