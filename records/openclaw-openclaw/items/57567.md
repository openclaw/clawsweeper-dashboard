## Summary

Keep open. Current main has partial last-known-good and prefixed-JSON recovery, but the specific legacy Feishu upgrade path still is not fully repaired: per-account `botName` remains outside the strict schema, Feishu does not expose a doctor compatibility normalizer, and `doctor --fix` still only handles a narrow non-JSON-prefix corruption path rather than restoring arbitrary malformed configs from numbered backups. The narrower related #65177 is still open and does not cover the broader corrupted-config rollback request.

## What This Changes

The issue requests safer upgrade migration from v2026.3.24 to v2026.3.28, including valid config output, Feishu legacy schema migration, backup rollback for corrupted JSON, clearer doctor repair, and documentation.

## Best Possible Solution

Keep this open or split it into two linked tasks: Feishu should add a plugin-owned doctor compatibility migration for legacy `channels.feishu.accounts.*.botName`, `accounts.default`, and moved policy fields, while core doctor should get an explicit, validated backup-restore path for unparseable configs or a narrower canonical follow-up that preserves last-known-good safety rules.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence for the Feishu path: a legacy `channels.feishu.accounts.main.botName` value reaches the active strict Feishu schema and is rejected as an additional property. The arbitrary malformed-JSON rollback path is also source-verifiable as not implemented unless last-known-good or valid-suffix recovery applies.

## Solution Assessment

Not applicable as a patch. The maintainable solution is narrower than the whole umbrella text: implement the Feishu-owned compatibility migration with regression coverage, then separately decide the core doctor policy for restoring valid backups from unparseable configs.
