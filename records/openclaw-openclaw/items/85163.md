## Summary

Keep open: the migration boundary is reasonable, but the latest diff can prompt and emit doctor output before `--json` commands reach the existing JSON-output startup policy, which would break machine-readable CLI workflows for affected macOS users.

## What This Changes

The PR adds a macOS interactive CLI startup hook that offers to migrate legacy Codex OAuth sidecar profiles through doctor, adds a headless Keychain warning, updates docs/changelog wording, and adds focused regression tests.

## Best Possible Solution

Keep doctor as the canonical migration path, but make the startup hook honor machine-readable invocations before prompting and land only after maintainers accept the macOS first-run auth migration UX.

## Reproduction Assessment

Yes, at source level. Current main blocks Keychain reads when `allowKeychainPrompt=false`, and the PR hook tests model the interactive migration path; I did not run a live macOS Keychain prompt.

## Solution Assessment

No, not as-is. Reusing doctor from an interactive CLI hook is the right owner boundary, but it must skip machine-readable `--json` invocations before any prompt or doctor note can reach stdout.

Full review comments:

- **[P1] Skip the startup prompt for JSON output commands:** `src/cli/auto-migrate-legacy-oauth-sidecar.ts:52-57`
  - body: The new hook only skips help/version, non-TTY, CI, read-only, and selected primaries before it calls doctor’s prompt. Treat JSON-output invocations as a skip path and cover that case with a regression test.
