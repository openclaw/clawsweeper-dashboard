## Summary

Keeping this open: current main still has a clear source-level path where a late normal agent completion can recreate a session row after `sessions.delete` has reported success.

## What This Changes

The issue asks for `sessions.delete` to either be a reliable termination boundary after normal agent runs or to document that it is only best-effort cleanup.

## Best Possible Solution

Make successful session deletion durable against late final agent-run metadata writes while preserving current cleanup, archive, and active-run failure behavior.

## Reproduction Assessment

Yes from source inspection, though not from a live run. The current path is `sessions.delete` deleting the row, followed by a stale normal agent completion calling `updateSessionStoreAfterAgentRun`, which recreates a missing `sessionKey`.

## Solution Assessment

Yes for a narrow code fix. The best fix is to make final agent-run persistence respect a concurrent successful delete; adding a new hard-kill API is not necessary to fix this reported delete-success boundary.
