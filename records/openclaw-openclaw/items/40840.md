## Summary

Keep open: current `main` still has a source-level path where a normal agent-run finalizer can recreate a session-store row after `sessions.delete` succeeds, and I found no merged or open canonical fix for this narrower bug.

## What This Changes

The issue asks for `sessions.delete` to behave as a trustworthy termination boundary, or at least not allow the same session key to immediately reappear after a successful delete.

## Best Possible Solution

Make explicit session deletion win over stale post-run metadata persistence with focused regression coverage, while leaving broader `sessions_kill` child-process semantics to a separate product/security discussion.

## Reproduction Assessment

Yes, at source level: a normal agent-run finalizer can persist metadata after `sessions.delete` removes the durable row. I did not run the timing race live in this read-only review.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. A new hard-kill API should remain separate because this issue has a concrete existing `sessions.delete` consistency bug.
