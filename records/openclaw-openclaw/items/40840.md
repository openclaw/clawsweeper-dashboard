## Summary

Keep open: current `main` still allows a source-level interleaving where `sessions.delete` removes and archives a session row, then the normal agent-run finalizer can recreate the same key from stale in-memory state. The broad `sessions_kill` request was already superseded elsewhere, but this narrower delete-boundary bug is still canonical and actionable.

## What This Changes

The issue asks for `sessions.delete` to be a reliable session termination boundary, or to fail/document clearly when deletion cannot prevent same-key re-registration.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage; keep any new `sessions_kill` RPC as a separate API/product discussion.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, delete that session after runtime cleanup drains, then let `updateSessionStoreAfterAgentRun` persist from the stale in-memory snapshot. I did not execute the timing race because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify the contract afterward, but it should not replace fixing the resurrection path.
