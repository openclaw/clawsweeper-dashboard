## Summary

Keep open: current main and the latest release still have a source-level path where a normal agent-run finalizer can recreate a session row after `sessions.delete` succeeds, and I found no merged or open canonical fix for this narrower bug.

## What This Changes

The issue asks for `sessions.delete` to be a reliable termination/delete boundary, or to fail/document clearly when a normal agent run can re-register the same session key.

## Best Possible Solution

Make explicit session deletion win over stale post-run metadata persistence with focused regression coverage, while leaving broader `sessions_kill` child-process semantics to a separate product/security discussion.

## Reproduction Assessment

Yes, at source level: a normal agent run can clear active-run tracking before final metadata persistence, letting `sessions.delete` remove the row before `updateSessionStoreAfterAgentRun` writes the same key back. I did not run the timing race live in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. A new hard-kill API can remain separate because this issue has a concrete existing `sessions.delete` consistency bug.
