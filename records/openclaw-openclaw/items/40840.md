## Summary

Keep open: current main still has a source-visible path where `sessions.delete` can delete and archive a session after waiting for embedded-run cleanup, while the normal agent-run finalizer can later write the same `sessionKey` back to the store.

## What This Changes

The issue asks for `sessions.delete` to either be a reliable hard-kill/delete boundary or explicitly document weaker cleanup-only semantics when normal agent runs are involved.

## Best Possible Solution

Make explicit deletion win over in-flight agent finalization, with regression coverage for delete-during-agent-run and no new hard-kill RPC unless maintainers choose to add one later.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, call `sessions.delete` after the embedded run clears but before `updateSessionStoreAfterAgentRun` persists metadata, and the finalizer can write the deleted key again. I did not run that timing test in this read-only review.

## Solution Assessment

Yes, a narrow code fix is the best next step: prevent post-delete finalization from recreating an entry that was present at run start and removed before final persistence. Documentation can clarify semantics, but it should not replace fixing the store resurrection path.
