## Summary

Keep open: current main still has a source-visible race where `sessions.delete` can delete and archive a session after runtime cleanup, while normal agent-run finalization can later persist metadata back under the same session key.

## What This Changes

The issue asks for trustworthy session termination semantics so a successful `sessions.delete` cannot immediately reappear after a normal agent run.

## Best Possible Solution

Make explicit deletion win over in-flight agent finalization, with regression coverage for delete-during-agent-run and no new hard-kill RPC unless maintainers choose that product direction separately.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing non-main session, let the embedded run clear, call `sessions.delete`, then allow `updateSessionStoreAfterAgentRun` to persist metadata. The finalizer currently has enough stale snapshot state to recreate the deleted key.

## Solution Assessment

Yes, the narrow maintainable fix is to have post-run store finalization detect that the pre-existing row was explicitly deleted and skip recreation, while keeping normal first-run creation behavior intact.

## Repair Work Prompt

Fix the `sessions.delete` resurrection race tracked by https://github.com/openclaw/openclaw/issues/40840. In current main, a normal Gateway `agent` run can finish its embedded active-run cleanup, `sessions.delete` can then remove/archive the row, and `updateSessionStoreAfterAgentRun` can recreate the same `sessionKey` from its stale snapshot. Make explicit deletion win over in-flight agent finalization without adding a new hard-kill RPC or changing unrelated session semantics. Preserve legitimate first-run creation when no prior row existed. Add regression coverage for delete-after-embedded-run-end/before-final-store-persist, likely in `src/agents/command/session-store.test.ts` and/or `src/gateway/server.sessions.delete-lifecycle.test.ts`; update docs/changelog only if the user-visible behavior or documented semantics change. Do not touch unrelated session cleanup, provider, or plugin behavior.
