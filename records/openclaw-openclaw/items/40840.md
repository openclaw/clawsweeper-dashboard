## Summary

Keep open: current `main` still has a source-level path where `sessions.delete` deletes and archives a session row, while a normal agent-run finalizer can later persist the same `sessionKey` back from stale in-memory state. The broader `sessions_kill`/cleanup request is superseded elsewhere, but this narrower delete-boundary bug remains actionable.

## What This Changes

The issue requests reliable session deletion semantics so a successful `sessions.delete` cannot be undone by normal agent-run lifecycle persistence.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage; leave any new `sessions_kill` RPC or broader cleanup UX to separate product/API work.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, delete that session after cleanup drains, then let `updateSessionStoreAfterAgentRun` persist from the stale in-memory snapshot. I did not execute the timing race because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation may clarify semantics afterward, but it should not replace fixing the resurrection path.
