## Summary

Keep open: current main still has a source-level race where a normal agent-run finalizer can recreate a session row after `sessions.delete` deleted and archived it, and there is no closing PR or canonical replacement for this narrower hard-delete semantic gap.

## What This Changes

The issue requests stronger session termination semantics so successful `sessions.delete` cannot be undone by stale normal agent-run persistence.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with focused regression coverage; leave any new `sessions_kill` RPC as a separate product decision.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, call `sessions.delete` after active embedded-run cleanup has drained, then let `updateSessionStoreAfterAgentRun` persist from stale in-memory state. I did not run the timing test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify the contract afterward, but it should not replace fixing the resurrection path.
