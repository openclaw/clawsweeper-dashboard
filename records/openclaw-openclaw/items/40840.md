## Summary

Keep open: current main and the latest release still have a source-level path where `sessions.delete` removes the row, but a normal agent-run finalizer can later write stale metadata back under the same `sessionKey`. The broad session-management feature work is tracked separately, but this narrower delete-boundary bug is still actionable.

## What This Changes

The issue asks for `sessions.delete` to be a reliable termination boundary, or to fail/clarify when a normal agent run can resurrect the same session key.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage; keep broader `sessions_kill`, pause/resume, and session-management UI work in the separate feature track.

## Reproduction Assessment

Yes at source level: start a normal Gateway `agent` run for an existing non-main session, let `sessions.delete` remove it after cleanup drains, then allow `updateSessionStoreAfterAgentRun` to persist from the stale in-memory snapshot. I did not execute the timing race in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation may clarify semantics afterward, but it should not replace fixing the resurrection path.
