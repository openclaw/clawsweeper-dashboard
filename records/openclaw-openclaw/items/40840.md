## Summary

Keep open: current main and the latest release still have a source-visible lifecycle gap where `sessions.delete` can remove/archive a row, then a normal agent-run finalizer can write the same session key back from stale run state. The prior ClawSweeper comment correctly identified this as a narrow session-store race, and no linked closing PR or canonical replacement issue currently owns the remaining fix.

## What This Changes

The issue requests stronger session termination semantics so successful `sessions.delete` cannot be undone by a later normal agent-run store write.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage and no new hard-kill RPC unless maintainers separately choose that product surface.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, let active embedded-run state clear, call `sessions.delete`, then allow `updateSessionStoreAfterAgentRun` to persist and it can recreate the missing key. I did not run that timing test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent a stale post-run metadata write from recreating a session that was explicitly deleted. Documentation can clarify semantics, but it should not replace fixing the store resurrection path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40840 by making explicit `sessions.delete` win over stale in-flight normal `agent` finalization. Current main deletes and archives the row in `src/gateway/server-methods/sessions.ts`, but `src/agents/command/session-store.ts` can build from the caller snapshot/defaults and write `store[sessionKey] = merged` after the fresh writer store no longer has that key.
