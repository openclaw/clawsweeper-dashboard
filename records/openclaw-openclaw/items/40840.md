## Summary

Keep open: current main still allows `sessions.delete` to remove a durable session row while a normal agent-run finalizer can later recreate the same `sessionKey` from stale in-memory metadata.

## What This Changes

The issue requests stronger `sessions.delete` semantics so a successful delete cannot immediately be undone by a completing normal agent run.

## Best Possible Solution

Make explicit session deletion win over stale post-run metadata persistence, with focused regression coverage, while leaving any broader `sessions_kill` API design to separate product work.

## Reproduction Assessment

Yes, at source level: a normal agent run can clear active-run tracking before final metadata persistence, allowing `sessions.delete` to remove the row before `updateSessionStoreAfterAgentRun` recreates it. I did not execute the timing race in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify semantics afterward, but doc-only treatment would not fix the resurrection path.
