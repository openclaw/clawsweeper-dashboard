## Label Justifications

- `P2`: The bug prevents integrations from treating successful session deletion as reliable, but its blast radius is limited to a specific lifecycle race.
- `impact:session-state`: A deleted session key can be recreated by stale post-run state, causing persisted session state to drift from the requested lifecycle action.

## Summary

Current `main` still permits a normal run’s stale post-run finalizer to recreate a session-store row after `sessions.delete` succeeds, and no narrower merged or open fix was found; keep this issue open as a focused session-state repair rather than expanding it into the broader hard-kill request.

## What This Changes

Prevent stale normal-run finalizers from recreating a session-store entry after `sessions.delete` reports success.

## Best Possible Solution

Introduce a lifecycle-owned generation or equivalent compare-before-write fence shared by deletion and post-run persistence, with regression coverage proving stale finalizers cannot resurrect the row while later intentional activity can create a fresh generation.

## Reproduction Assessment

Yes at source level with high confidence: the delete path removes the key without a generation marker, and the normal post-run finalizer can assign that missing key again from captured run metadata. A live timing reproduction was not run during this read-only review.

## Solution Assessment

Yes, the narrow stale-write fence is the best repair because it fixes the existing `sessions.delete` contract at the lifecycle owner boundary without adding a parallel hard-kill API or configuration surface.
