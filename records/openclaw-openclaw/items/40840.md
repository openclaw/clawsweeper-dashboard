## Summary

Keep open. Current main still has the stale-finalizer race: a normal agent run can carry a pre-delete session-store snapshot, `sessions.delete` can remove the key, and the post-run session-store update can write the same key back because it does not distinguish first-time creation from recreation after deletion.

## What This Changes

The issue asks for `sessions.delete` to either be a durable session termination boundary after normal agent runs, fail when that cannot be guaranteed, or document the weaker cleanup semantics.

## Best Possible Solution

Add a durable-delete guard in the post-run session-store finalizer: if the caller's pre-run snapshot contained the key but the locked current store no longer does, skip the finalizer write and leave the key absent. Preserve first-time session creation and normal metadata updates, and cover the race with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to let a normal agent run hold a `sessionStore` snapshot containing the key, delete that key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` execute; current main can recreate the missing key.

## Solution Assessment

Yes. The narrowest maintainable fix is a finalizer guard plus focused tests; a broader native `sessions_kill` API can remain a separate product/API discussion.
