## Summary

Keep open: current main still has a source-level delete/finalization race where `sessions.delete` can remove and archive a row, but a normal in-flight agent run can later persist the same `sessionKey` from its stale run snapshot. The newer `sessions.abort` and cleanup surfaces do not fully supersede the requested hard-delete boundary.

## What This Changes

Make successful `sessions.delete` a trustworthy termination/delete boundary for normal agent runs, or clearly document/replace that semantic.

## Best Possible Solution

Strengthen the existing delete/run-finalization contract so a successful `sessions.delete` cannot be undone by stale completion persistence from a run that was already in flight, while preserving normal future recreation of deleted conversation keys.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal agent run with an existing session snapshot, call `sessions.delete` while the run is draining, then let `updateSessionStoreAfterAgentRun` write the stale same-key row after the delete.

## Solution Assessment

Yes. The narrowest maintainable fix is to make post-run session persistence respect an intervening successful delete, rather than adding a broad new primitive or documenting surprising success semantics as expected.

## Repair Work Prompt

Fix the session delete/finalization race reported here: a successful `sessions.delete` must not be undone by `updateSessionStoreAfterAgentRun` from a normal agent run that was already in flight when the delete happened. Keep the change narrow: prevent stale same-run completion from recreating a deleted row, preserve legitimate future recreation after later inbound messages or new explicit sessions, and do not add a permanent global tombstone or a new public config option. Add regression coverage around the agent-run completion writer and/or gateway delete lifecycle. Keep existing guards intact: main session cannot be deleted, plugin-owned delete checks remain, ACP metadata preservation remains, and session archive/lifecycle hooks still fire on actual delete. Add a user-facing `CHANGELOG.md` fix entry if the repo's current changelog policy requires one.
