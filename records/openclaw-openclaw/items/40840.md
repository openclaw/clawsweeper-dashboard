## Summary

Keep open. Current main still has the narrow delete-boundary bug: `sessions.delete` removes the session row, but the ordinary post-agent-run metadata writer can recreate a missing row through its fallback-entry path, so the issue remains actionable.

## What This Changes

The issue asks for `sessions.delete` to be a reliable delete boundary that stale normal agent-run finalizers cannot immediately undo.

## Best Possible Solution

Add a delete-generation or tombstone fence at the session-store mutation boundary so stale post-run finalizers cannot recreate a successfully deleted session row, while leaving broader hard-kill process control to separate session-management issues.

## Reproduction Assessment

Yes at source level: current main lets `sessions.delete` remove the row, while the ordinary post-agent-run store updater can write a fallback entry when the current persisted row is absent. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Yes for the narrow bug: a stale-finalizer fence at the session-store update/delete boundary is the smallest maintainable repair. A new hard-kill command or receipt payload can be designed separately, but it is not required to stop same-key resurrection after a successful delete.

## Repair Work Prompt

Fix the session delete fence bug described in https://github.com/openclaw/openclaw/issues/40840. A successful `sessions.delete` must prevent stale normal agent-run finalizers from recreating the same session key after the row has been removed. Keep the fix in the core session-store/session lifecycle boundary, likely by adding a delete generation/tombstone or a current-row ownership check so `updateSessionStoreAfterAgentRun` cannot persist a fallback entry after deletion. Do not add broad `sessions_kill` process-control features, new config, or compatibility shims. Add focused regression coverage showing a normal agent run finalizer after `sessions.delete` does not resurrect the row while valid non-deleted finalizers still update metadata. Do not edit `CHANGELOG.md`.
