## Summary

Keep open: current main still has a source-visible lifecycle race where `sessions.delete` can remove and archive a row, while a normal `agent` run finalizer can later write the same `sessionKey` back into `sessions.json`.

## What This Changes

The issue asks for `sessions.delete` to be a reliable terminate/delete boundary, or for OpenClaw to expose/document a stronger session kill semantic.

## Best Possible Solution

Make explicit `sessions.delete` win over late agent-run finalization, with focused regression coverage; add a new hard-kill API only if maintainers separately choose that product direction.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run, let `sessions.delete` complete after embedded cleanup, then let `updateSessionStoreAfterAgentRun` persist against the fresh store. The finalizer still has a current-main path that recreates the deleted `sessionKey`.

## Solution Assessment

Yes, the narrow maintainable fix is to guard late session-store finalization so a deletion that won the writer queue is not undone. Documentation can clarify semantics afterward, but it should not replace fixing the resurrection path.
