## Summary

Keep open: current main still has a source-level path where `sessions.delete` can remove and archive a row, while a normal agent-run finalizer can later write the same `sessionKey` back from stale in-memory state. The broad `sessions_kill` cleanup request is superseded, but this narrower delete-boundary bug remains actionable.

## What This Changes

The issue asks for `sessions.delete` to behave as a reliable termination boundary, fail honestly when it cannot, or document that it is only best-effort store cleanup.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage; keep any new `sessions_kill` RPC or broader cleanup UX separate.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, delete that session after cleanup drains, then let `updateSessionStoreAfterAgentRun` persist from the stale in-memory snapshot. I did not execute the timing race because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation may clarify semantics afterward, but it should not replace fixing the resurrection path.
