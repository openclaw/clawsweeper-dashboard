## Summary

Keep open: current main still has a source-level race where `sessions.delete` removes and archives a session row, but the normal agent-run finalizer can recreate the same `sessionKey` from stale in-memory state afterward. There is no closing PR or canonical replacement for this narrower delete-boundary bug.

## What This Changes

Ensure a successful `sessions.delete` is a durable session termination boundary after normal agent runs, or return/document stronger failure semantics.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with focused regression coverage; leave any new `sessions_kill` RPC as a separate product/API discussion.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, call `sessions.delete` after active embedded-run cleanup has drained, then let `updateSessionStoreAfterAgentRun` persist from stale in-memory state. I did not run the timing test because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify the contract afterward, but it should not replace fixing the resurrection path.
