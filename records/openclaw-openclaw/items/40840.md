- `impact:session-state`: The issue is about a deleted session record being recreated from stale runtime state.

## Summary

Keep open: current main and the latest release still allow `sessions.delete` to remove a row while a normal agent-run finalizer can later recreate the same `sessionKey` from stale in-memory metadata.

## What This Changes

The issue asks for `sessions.delete` to be a reliable deletion boundary after normal agent runs, or to fail/document weaker semantics instead of reporting success before same-key re-registration.

## Best Possible Solution

Make explicit session deletion win over stale post-run metadata persistence, with regression coverage, while keeping broader hard-kill and pause/resume feature design in separate items.

## Reproduction Assessment

Yes, at source level: a normal agent run clears active-run tracking before `agent-command` persists final metadata, so `sessions.delete` can remove the durable row in between and the finalizer can recreate it. I did not execute the timing race in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify semantics afterward, but it should not replace fixing the resurrection path.
