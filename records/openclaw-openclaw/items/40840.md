## Summary

Keep open: current main still lets a stale normal agent-run finalizer recreate a session row after `sessions.delete` has deleted and archived it. The broad cleanup request is already closed with this issue named as the remaining canonical hard-delete semantic gap, and no closing PR currently owns this interleaving.

## What This Changes

The issue asks for `sessions.delete` to be a reliable termination boundary, or for OpenClaw to provide/document a stronger session hard-kill primitive.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage, while leaving any new hard-kill RPC as a separate product decision.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, call `sessions.delete` after active embedded-run state clears, then let `updateSessionStoreAfterAgentRun` persist from stale in-memory state. I did not run that timing test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify semantics afterward, but it should not replace fixing the store resurrection path.
