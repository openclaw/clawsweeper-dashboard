## Summary

Keep open: current main still has a source-level interleaving where `sessions.delete` can delete the durable row and a normal agent-run finalizer can later write the same `sessionKey` back from stale runtime metadata; I found no merged fix or closing PR for this narrower bug.

## What This Changes

The issue asks for `sessions.delete` to be a reliable delete/termination boundary, or to fail or document weaker semantics, when normal agent runs can write stale session metadata after deletion.

## Best Possible Solution

Make explicit session deletion win over stale post-run metadata persistence with focused regression coverage, while leaving any broader `sessions_kill` API design to separate product work.

## Reproduction Assessment

Yes, at source level: a normal agent run can clear active-run tracking before final metadata persistence, letting `sessions.delete` remove the row before `updateSessionStoreAfterAgentRun` writes the same key back. I did not run the timing race live in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify semantics afterward, but doc-only treatment would not fix the resurrection path.
