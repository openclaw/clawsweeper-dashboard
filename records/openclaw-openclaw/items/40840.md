## Summary

Keep open: current main still has a source-visible session lifecycle race where `sessions.delete` can remove/archive a row, while a later normal agent-run finalizer can write the same key back from stale in-memory state. The related broad cleanup issue is closed and points back to this issue for the remaining hard-delete semantic gap, and no closing PR currently owns this exact interleaving.

## What This Changes

The issue asks for `sessions.delete` to either keep a deleted session gone or return failure when normal agent lifecycle work can re-register the same session key.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage, while leaving any new hard-kill RPC as a separate maintainer product decision.

## Reproduction Assessment

Yes, at source level: start a normal Gateway `agent` run for an existing session, allow active embedded-run state to clear, call `sessions.delete`, then let `updateSessionStoreAfterAgentRun` persist and it can recreate the missing key. I did not run that timing test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation can clarify semantics afterward, but it should not replace fixing the store resurrection path.
