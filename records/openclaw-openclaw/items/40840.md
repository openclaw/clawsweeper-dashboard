## Label Justifications

- `P2`: The bug blocks reliable integrator session termination semantics, but the blast radius is limited to a session lifecycle race rather than core runtime availability.
- `impact:session-state`: A deleted session key can be recreated by stale post-run persistence, causing persisted session state to drift from the requested lifecycle action.

## Summary

Keep open: current main still has a source-reproducible stale-finalizer gap where normal post-run session persistence can recreate a missing `sessionKey` after `sessions.delete` succeeds, and no merged or open PR now owns the narrow fence. This should stay scoped to the session-store write boundary; the broader child-process hard-kill request remains separate.

## What This Changes

The issue asks for `sessions.delete` to either provide a hard post-delete boundary against same-key re-registration after normal agent runs or document that it is only best-effort cleanup.

## Best Possible Solution

Introduce a shared lifecycle generation, tombstone, or compare-before-write fence between `sessions.delete` and post-run persistence so stale finalizers cannot resurrect the deleted row, while keeping the broader child-process hard-kill design on its separate track.

## Reproduction Assessment

Yes, at source level: `sessions.delete` removes the row without a stale-writer fence, and `updateSessionStoreAfterAgentRun` can recreate a missing `store[sessionKey]` from captured run metadata. I did not run a live timing reproduction during this read-only review.

## Solution Assessment

Yes, the narrow stale-write fence is the best fix for this issue because it repairs existing delete semantics at the session lifecycle owner boundary without adding a new hard-kill API, config option, or child-process termination policy.

## Repair Work Prompt

Fix the stale-finalizer race where `sessions.delete` can remove a session row but a normal agent run's post-run metadata finalizer later recreates the same `sessionKey`. Keep the scope narrow: add a lifecycle-owned generation, tombstone, or compare-before-write fence shared by the delete path and `updateSessionStoreAfterAgentRun`, so finalizers from the deleted generation cannot write past a successful delete while legitimate later activity can create a fresh session. Do not add a new `sessions_kill` API, a config option, or broader child-process termination behavior; that broader work remains tracked at https://github.com/openclaw/openclaw/issues/12554. Add focused regression coverage for finalizer-after-delete behavior and keep existing restart-recovery guarded persistence and `sessions.delete` lifecycle cleanup semantics intact.
