## Summary

Keep open. Current main and the latest release still have a source-level gap where ordinary post-run session-store finalization can write a missing session key after `sessions.delete` removes it; the existing stale-lifecycle guard only covers a narrower user-facing recovery path, and the related session-kill issue tracks broader process termination rather than this delete fence.

## What This Changes

The issue asks for `sessions.delete` to either keep a successfully deleted session from reappearing after a normal agent run or clearly report that delete was not a hard termination boundary.

## Best Possible Solution

Make successful `sessions.delete` a fence for that session key/session id so stale run finalizers re-read and verify the current row before writing, while leaving broader process-kill API design to the related hard-termination issue.

## Reproduction Assessment

Yes at source level: current main can clear embedded-run activity before outer post-run store finalizers finish, and the ordinary finalizer path writes `store[sessionKey]` even if `sessions.delete` removed it. I did not run a live gateway reproduction during this read-only review.

## Solution Assessment

Yes for the narrow repair: the best fix is a stale-finalizer fence around session-store writes after delete, not a new broad `sessions_kill` API. A receipt/status API could be useful later, but it is not required to stop same-key resurrection.
