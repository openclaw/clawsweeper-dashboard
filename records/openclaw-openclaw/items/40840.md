## Summary

Keep open: current main still has a source-level path where `sessions.delete` removes and archives a store row, but a normal agent-run finalizer can later persist stale in-memory session metadata back under the same `sessionKey`. The broader hard-kill feature request is tracked separately, but this narrower delete-boundary bug remains actionable.

## What This Changes

The issue reports that `sessions.delete` can return success while a normal agent-run finalizer later re-registers the same session key.

## Best Possible Solution

Make explicit session deletion win over stale in-flight agent finalization, with regression coverage; keep broader `sessions_kill`, pause/resume, and UI lifecycle controls separate.

## Reproduction Assessment

Yes at source level: start a normal Gateway `agent` run for an existing session, delete that session after cleanup drains, then let `updateSessionStoreAfterAgentRun` persist from the stale in-memory snapshot. I did not execute the timing race because this review is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to prevent stale post-run metadata persistence from recreating a session that was explicitly deleted. Documentation may clarify semantics afterward, but it should not replace fixing the resurrection path.
