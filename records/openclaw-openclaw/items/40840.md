## Summary

Keep open. Current main still allows a stale post-run session-store finalizer to recreate a key that `sessions.delete` just removed, so the reported terminate boundary remains a valid runtime bug rather than a stale feature request.

## What This Changes

Make `sessions.delete` reliably keep a deleted session key absent after normal agent-run finalizers, or otherwise clarify/provide equivalent hard-kill semantics.

## Best Possible Solution

Add a locked finalizer guard that skips recreating a previously existing session key when the current store no longer has it, while preserving first-time creation and normal metadata updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: let a normal agent run hold a snapshot containing the session key, delete that key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` run; current main can write the key back.

## Solution Assessment

Yes. The narrowest maintainable fix is a finalizer guard plus focused regression coverage; a broader `sessions_kill` API can remain a separate product/API discussion.

## Repair Work Prompt

Fix openclaw/openclaw#40840. Current main lets `updateSessionStoreAfterAgentRun` recreate a session key from a stale caller snapshot after `sessions.delete` has removed that key. Add a durable guard in the post-run session-store finalizer: if the caller snapshot contained `sessionKey` but the locked current store no longer contains it, skip the finalizer write and leave the key absent. Preserve first-time session creation when the caller snapshot did not contain the key, normal metadata updates, ACP/lifecycle preservation, and active-run delete failure behavior. Add focused regression coverage in `src/agents/command/session-store.test.ts` and only add gateway lifecycle coverage if needed. Do not add a new public hard-kill API, config option, or broad session lifecycle redesign. Add a single-line user-facing `CHANGELOG.md` fix entry.
