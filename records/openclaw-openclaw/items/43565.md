## Summary

Keep open. Current main still has only the binary `tools.fs.workspaceOnly` policy, so the feature is not implemented, but this PR is not ready to merge because the head introduces concrete filesystem-boundary regressions and generated config/changelog drift.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config field and threads extra filesystem roots through sandbox path checks, read/write/edit/apply_patch, media loading, config schema/help/types, and regression tests.

## Best Possible Solution

Land a rebased implementation that uses one canonical filesystem policy path, preserves pinned root-contained mutation helpers, resolves sandbox bind mounts before allowed-root matching, keeps apply_patch enabled-by-default semantics, updates generated config/docs artifacts and the changelog, and adds focused tests for the final contract.

## Reproduction Assessment

Yes. Source comparison is enough to reproduce the review failures: PR head line 311 deletes after validation with `fs.rm`, line 601 checks allowed roots before bridge mount resolution, and lines 360-361 disable default apply_patch unless explicitly enabled.

## Solution Assessment

No. The feature direction is useful, but this branch is not the safest maintainable solution until the filesystem boundary regressions, stale generated config output, and release-note gap are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on `sandboxResult.resolved`. Current main uses `removePathWithinRoot()`, so this reintroduces a check-then-use mutation path that can be redirected outside the matched root if a parent is swapped between validation and deletion.
  - confidence: 0.9
- **[P2] Resolve mounted sandbox paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: This wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for Docker bind mounts.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:360-361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main keeps `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:449`
  - body: Adding `tools.fs.allowedRoots` to the runtime schema also changes the generated base config schema and docs baseline. The PR does not update those generated artifacts, so config schema/docs checks will be stale until they are regenerated.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:284-288`
  - body: This adds user-facing config, but the PR does not touch `CHANGELOG.md`. Repo policy requires user-facing feat/fix/perf changes to add a single-line changelog entry with contributor credit, so release notes will miss the new setting until that is added.
  - confidence: 0.78
