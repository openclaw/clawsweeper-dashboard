## Summary

Keep open. Current main still lacks the requested filesystem allowlist, so the PR is not obsolete, but the branch should not merge as-is because it regresses a pinned deletion boundary, rejects the Docker bind-mount use case it is meant to support, disables apply_patch by default, and leaves generated config schema artifacts stale.

## What This Changes

This PR adds `tools.fs.allowedRoots` and threads extra filesystem roots through config schema/help/types, sandbox path checks, host read/write/edit tools, apply_patch, media loading, and regression tests.

## Best Possible Solution

Land a corrected, rebased implementation that uses one canonical filesystem policy path, keeps pinned root-contained helpers for workspace-only mutations, resolves sandbox mount paths before allowed-root matching, preserves existing apply_patch defaults, regenerates config schema/docs artifacts, and adds focused tests for the final contract.

## Reproduction Assessment

Yes. Source comparison against current main and PR head gives a high-confidence reproduction path: default OpenAI configs lose apply_patch, sandbox bind-mount paths are rejected before bridge resolution, and workspace-only delete uses `fs.rm` after validation instead of a pinned helper.

## Solution Assessment

No. The feature direction is useful, but this branch is not the narrowest safe solution until it restores pinned deletion, makes allowed-root checks bridge-aware in sandbox mode, preserves default apply_patch availability, and updates generated config artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on `sandboxResult.resolved`. Current main uses `removePathWithinRoot()`, so this reintroduces a check-then-use mutation path that can be redirected outside the matched root if a parent is swapped between validation and deletion.
  - confidence: 0.9
- **[P2] Resolve mounted sandbox paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs this wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for the Docker bind-mount case.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:360-361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main keeps `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:449`
  - body: Adding `tools.fs.allowedRoots` to the runtime schema also changes the generated base config schema and docs baseline. The PR head still lacks `tools.fs.allowedRoots` in `schema.base.generated.ts`, so config schema/docs checks will be stale until the generated artifacts are updated.
  - confidence: 0.82
