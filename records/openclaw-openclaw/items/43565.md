The PR adds a new `tools.fs.allowedRoots` config field and threads extra filesystem roots through sandbox path checks, host file tools, `apply_patch`, media tools, config help/types/schema, and tests.

## Best Possible Solution

Land one maintainer-approved extra-root design that preserves root-scoped filesystem mutations, current sandbox mount and media contracts, config/docs alignment, and redacted real allow/reject proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore root-scoped apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: Workspace-only deletes validate with `assertSandboxPath` and then call raw `fs.rm` on the resolved absolute path. Current main routes these mutations through the root-scoped helper, so this separates validation from mutation and can be redirected by a concurrent parent or symlink swap outside the matched root.
  - confidence: 0.91
- **[P2] Preserve mounted-root guard options:** `src/agents/pi-tools.read.ts:581-584`
  - body: This wrapper drops current main's `additionalContainerMounts`, `pathParamKeys`, and `normalizeGuardedPathParams` options and then compares only the container-workdir-mapped path against `additionalRoots`. Sandbox bind-mount paths that current main maps to their own guarded host roots can be rejected or forwarded incorrectly after this change.
  - confidence: 0.87
- **[P2] Preserve the media local-roots call signature:** `src/agents/tools/media-tool-shared.ts:75-78`
  - body: Current main callers pass media-source context as a third argument to `resolveMediaToolLocalRoots`, but this patch changes the helper to a two-argument signature. Rebase the allowed-roots threading through the current media contract so image, PDF, music, and video callers keep compiling and enforcing the same policy.
  - confidence: 0.78
