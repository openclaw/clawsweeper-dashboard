## Summary

Keep open. The feature is still absent from current main and this PR is useful source work, but the current head would regress apply_patch delete safety, fails the sandbox bind-mount use case, and leaves generated config metadata plus changelog coverage stale.

## What This Changes

The PR adds a `tools.fs.allowedRoots` config option and threads additional filesystem roots through sandbox path checks, filesystem tools, `apply_patch`, media loaders, schema help/labels, and tests.

## Best Possible Solution

Land a rebased implementation that keeps all workspace-only mutations on matched-root pinned helpers, resolves sandbox bind mounts before allowed-root policy checks, regenerates config artifacts, adds the changelog entry, and proves the filesystem/media contract with focused tests.

## Reproduction Assessment

Yes. Source inspection of current main and PR head is enough to reproduce the review failures: the raw delete, pre-bridge allowed-root check, and missing generated schema/changelog are all visible at fixed lines.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best merge path until it preserves the existing pinned mutation safety boundary, handles Docker bind-mounted roots correctly, and updates generated/user-facing release metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: Current main routes workspace-only deletes through `removePathWithinRoot`, but this branch validates the target and then calls raw `fs.rm(sandboxResult.resolved)`. That separates the boundary check from the mutation, so a parent alias swap after validation can redirect deletion outside the matched root.
  - confidence: 0.9
- **[P2] Resolve bind-mounted paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: The wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge` maps custom bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` will still reject `/shared/file`, so the new setting fails for Docker bind mounts.
  - confidence: 0.87
