## Summary

Keep open. Current main does not implement `tools.fs.allowedRoots`, and this PR is useful source work, but the current branch still has blocking correctness and security issues around pinned deletes, sandbox bind mounts, generated config schema drift, and the required changelog entry.

## What This Changes

The PR adds `tools.fs.allowedRoots` and threads additional filesystem roots through sandbox checks, host read/write/edit, `apply_patch`, media loading, config metadata, and tests.

## Best Possible Solution

Land a rebased implementation that keeps the additive allowed-root contract, uses pinned helpers for all workspace-only mutations, evaluates sandbox allowed roots after mount resolution, and updates generated config/docs/changelog with focused tests.

## Reproduction Assessment

Yes. Source inspection of current main and the PR head reproduces the review failures: the branch deletes with `fs.rm` after validation, compares container paths to host allowed roots before mount resolution, and leaves generated config schema stale.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best merge path until the filesystem-boundary regression, sandbox bind-mount behavior, generated schema drift, and changelog gap are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: Current main routes workspace-only deletes through `removePathWithinRoot()`, but this branch validates an allowlisted target and then calls `fs.rm(sandboxResult.resolved)`. That separates the boundary check from the mutation, so a parent swap after validation can redirect deletion outside the matched root.
  - confidence: 0.9
- **[P2] Resolve mounted sandbox paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: The sandbox wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` will still reject `/shared/file`, so the new setting fails for Docker bind mounts.
  - confidence: 0.87
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:449`
  - body: Adding `tools.fs.allowedRoots` to the runtime schema also changes the generated base config schema and docs metadata. The PR does not update `src/config/schema.base.generated.ts`, so config schema/docs checks and user-facing schema consumers remain stale.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:284-288`
  - body: This adds user-facing configuration, but the PR does not update `CHANGELOG.md`. Repo policy requires a single-line changelog entry for user-facing feature changes so release notes include the new setting.
  - confidence: 0.78
