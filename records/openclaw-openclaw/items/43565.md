## Summary

Keep open: current main does not implement `tools.fs.allowedRoots`, and the PR branch still has concrete merge blockers in filesystem safety, sandbox bind-mount handling, generated config metadata, and changelog coverage.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config option and threads additional filesystem roots through sandbox checks, host read/write/edit, `apply_patch`, media tools, schema help/labels, and regression tests.

## Best Possible Solution

Land a rebased implementation that preserves pinned root-contained mutation helpers, resolves sandbox bind mounts before host-root policy checks, updates generated config artifacts and changelog, and proves the contract with focused tests.

## Reproduction Assessment

Yes. Source inspection of current main and PR head is enough to reproduce the review failures: raw `fs.rm` after validation, pre-bridge allowed-root checks for sandbox bind mounts, and generated schema drift are visible in the diff.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best merge path until it preserves the existing pinned deletion helper, handles bind-mounted sandbox paths through the bridge, and updates generated public config metadata and changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: Current main routes workspace-only deletes through `removePathWithinRoot()`, but this branch validates an allowlisted target and then calls `fs.rm(sandboxResult.resolved)`. That separates the boundary check from the mutation, so a parent swap after validation can redirect deletion outside the matched root.
  - confidence: 0.9
- **[P2] Resolve bind-mounted paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: The sandbox wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge` maps custom bind mounts. A bind like `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` will still reject `/shared/file`, so the new setting fails for Docker bind mounts.
  - confidence: 0.87
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:449`
  - body: Adding `tools.fs.allowedRoots` to the runtime schema also changes the generated base config schema and docs metadata. The PR does not update `src/config/schema.base.generated.ts`, so config schema/docs checks and user-facing schema consumers remain stale.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:284-288`
  - body: This adds a user-facing configuration option, but the PR does not update `CHANGELOG.md`. Repo policy requires a single-line changelog entry for user-facing feature changes so release notes include the new setting.
  - confidence: 0.78
