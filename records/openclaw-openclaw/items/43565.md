## Summary

Keep open: current main does not implement `tools.fs.allowedRoots`, and this PR is useful but not merge-ready because the sandbox wrapper still checks container mount paths against host allowed roots before the sandbox bridge can translate them.

## What This Changes

The PR adds a `tools.fs.allowedRoots` config option and wires extra allowed filesystem roots through sandbox path checks, host fs tools, apply_patch, and media-local-root handling under `workspaceOnly`.

## Best Possible Solution

Land a rebased implementation that routes host and sandbox fs paths through one allowed-root policy, including bridge-aware mount translation, media/apply_patch parity, changelog coverage, and regression tests for bind-mounted sandbox roots.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows sandbox read/write/edit validate a still-container path against host allowedRoots before `SandboxFsBridge.resolvePath` can map mounts like `/agent`. Existing sandbox workspace-only tests cover that mounted-path rejection surface, though I did not execute them in this read-only run.

## Solution Assessment

No. The feature direction is reasonable, but the PR should not merge until sandbox mount paths are translated consistently and the required changelog entry is added.
