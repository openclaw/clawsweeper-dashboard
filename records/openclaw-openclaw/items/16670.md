## Summary

Keep open. Current main still has memory configuration docs, doctor/status diagnostics, and a working manual memorySearch contract, but the setup/onboarding/configure flows do not provide a dedicated memory embedding provider/model/API-key or SecretRef setup step; the request also asks for mandatory onboarding behavior, which is a product and credential-policy decision.

## What This Changes

The issue requests a mandatory setup/onboarding step that configures memory embeddings and verifies the embedding provider for long-term memory search.

## Best Possible Solution

Add a maintainer-approved memory setup experience in onboarding and configure that writes the existing memorySearch fields, supports the chosen secret policy, verifies readiness through a controlled status/probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes for the source-level gap. Current main source inspection shows plain setup only writes workspace/gateway defaults, onboarding has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated setup/configure memory path is likely the right implementation area, but mandatory versus default-on versus prominently optional behavior, credential storage, SecretRef UX, and probe semantics need maintainer direction first.
