## Summary

Keep open: current main and v2026.6.6 still keep memory embedding setup outside onboarding and configure, while the requested mandatory behavior remains a product-policy choice across built-in, keyword-only, and plugin-backed memory modes.

## What This Changes

The issue requests a prominent mandatory onboarding/configure step for memory embedding provider, model, API-key or SecretRef setup, plus readiness verification.

## Best Possible Solution

Add a maintainer-approved memory readiness section to onboarding and `openclaw configure` that configures SecretRef-aware embedding provider/model credentials and verifies readiness, with an explicit required-versus-recommended policy.

## Reproduction Assessment

Yes, at source level: current main's setup and configure flows have no dedicated memory section, embedding credential prompt, or readiness probe. The broader claim of complete memory loss is not reproduced because keyword-only and plugin-backed memory modes exist.

## Solution Assessment

Partly. A first-class memory setup/readiness flow is the narrow maintainable improvement, but making embeddings universally mandatory is not yet the best established policy across built-in, local, keyword-only, and plugin-backed memory modes.
