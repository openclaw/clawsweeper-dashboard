## Summary

Keep open. Current main still has memorySearch docs, doctor diagnostics, and `openclaw memory status --deep`, but `openclaw setup`, onboarding, `openclaw configure`, and onboarding docs do not provide a dedicated memory embedding provider/model/API-key or SecretRef setup step; the mandatory/default-on policy remains a product decision.

## What This Changes

The issue asks for OpenClaw onboarding/setup to include a prominent or mandatory memory embedding provider, model, API key, and readiness setup step.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that configures `agents.defaults.memorySearch`, handles SecretRef-aware credentials, verifies readiness, and documents daemon/Codex OAuth caveats.

## Reproduction Assessment

Yes for the source-level setup gap: current setup, onboarding, configure, and onboarding docs omit a dedicated memorySearch step while memory docs and doctor diagnostics show embeddings matter. I did not run live embeddings because this read-only review is about setup surface and product policy.

## Solution Assessment

Unclear as requested: a dedicated setup/configure path is likely maintainable, but mandatory versus optional/default-on behavior plus credential and probe semantics need maintainer direction first.
