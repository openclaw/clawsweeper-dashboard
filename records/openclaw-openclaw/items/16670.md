## Summary

Keep open. Current main and v2026.5.20 still lack a dedicated `memorySearch` setup/onboarding/configure step; existing memory docs, doctor guidance, and `openclaw memory status --deep` diagnose the problem but do not resolve the requested onboarding UX or mandatory/default-on product decision.

## What This Changes

Add a mandatory or prominent setup/onboarding step for memory embedding provider, model, and API-key or SecretRef configuration.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes the existing `agents.defaults.memorySearch` contract, supports API-key or SecretRef input, verifies readiness, and documents daemon and Codex OAuth caveats.

I did not run a live embedding failure path because this read-only review is about setup surface and product policy.

## Solution Assessment

Unclear as requested. A dedicated setup/configure path is likely the maintainable direction, but mandatory versus optional/default-on behavior and credential/probe semantics need maintainer direction first.
