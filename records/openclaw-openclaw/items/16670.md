## Summary

Keep open: current main and v2026.6.10 still lack a dedicated memory embedding setup path in setup, onboarding, or configure, while the mandatory-versus-prominent-optional policy remains a maintainer product decision.

## What This Changes

The issue requests making memory embedding provider, model, API-key or SecretRef setup a mandatory onboarding wizard step.

## Best Possible Solution

Add a maintainer-approved memory readiness flow in onboarding and `openclaw configure` that uses the existing `agents.defaults.memorySearch` shape, handles API-key or SecretRef input, verifies readiness, and documents the chosen mandatory-versus-optional policy.

## Reproduction Assessment

Yes at source level: current setup, onboarding, and configure flows have no dedicated `memorySearch` section, credential prompt, or embedding readiness probe. I did not run a live wizard because this is a read-only review and the source path is sufficient to confirm the missing surface.

## Solution Assessment

Partly: a first-class memory setup/readiness flow is the maintainable direction, but making it universally mandatory is not yet established across local, FTS-only, provider-specific, and headless setup modes.
