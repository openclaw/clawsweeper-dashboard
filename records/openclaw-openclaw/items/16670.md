## Summary

Keep open: current main and v2026.6.8 still keep memory embedding setup outside setup, onboarding, and configure, while the requested mandatory policy and credential/probe UX still need maintainer direction.

## What This Changes

The issue requests making memory embedding provider, model, API-key setup, and readiness verification a mandatory first-run onboarding step.

## Best Possible Solution

Add a maintainer-approved memory readiness path in CLI onboarding and `openclaw configure` that writes `agents.defaults.memorySearch`, verifies embeddings, and documents the chosen mandatory-versus-recommended policy.

## Reproduction Assessment

Yes, at source level: current setup, onboarding, and configure flows have no dedicated memorySearch section, credential prompt, or embedding readiness probe. The broader complete-memory-loss claim was not reproduced because current docs and code support FTS/keyword and plugin-backed memory paths.

## Solution Assessment

Partly: a first-class memory setup/readiness flow is the narrow maintainable direction, but making it universally mandatory is not yet established across built-in, local, keyword-only, and plugin-backed modes.
