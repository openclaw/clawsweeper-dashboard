## Summary

Keep open. Current main still has memory-search docs and diagnostics, but setup/onboarding/configure do not provide a dedicated memory embedding provider/model/API-key or SecretRef step, and making that step mandatory needs maintainer product and credential-policy judgment.

## What This Changes

The issue asks for `openclaw setup` onboarding to include a prominent mandatory memory embedding provider, model, credential, and verification step.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes the existing `agents.defaults.memorySearch` fields, supports the chosen SecretRef/plaintext policy, verifies readiness through a controlled status/probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes for the source-level gap. Current main source inspection shows setup only writes workspace/gateway defaults, onboarding has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated setup/configure path is likely the right implementation area, but making it mandatory is not clearly the best UX until maintainers decide credential storage, SecretRef behavior, daemon environment handling, and live-probe policy.
