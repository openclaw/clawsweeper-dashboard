## Summary

Keep open. Current main and the latest release still have memory docs, doctor diagnostics, and `openclaw memory status --deep`, but they do not add a dedicated `memorySearch` provider/model/API-key or SecretRef step to setup, onboarding, or configure; the mandatory/default-on choice remains a maintainer product decision.

## What This Changes

The issue requests that onboarding/setup make memory embedding configuration a prominent mandatory step, including provider/model credentials and a readiness check.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes the existing `agents.defaults.memorySearch` contract, supports API-key or SecretRef input, verifies readiness, and documents daemon and Codex OAuth caveats.

## Reproduction Assessment

Yes for the source-level gap: setup, onboarding, configure, and onboarding docs omit a dedicated `memorySearch` setup step while memory docs and doctor diagnostics show embeddings matter. I did not run a live embedding failure path because this read-only review is about setup surface and product policy.

## Solution Assessment

Unclear as requested. A dedicated setup/configure path is likely maintainable, but mandatory versus optional/default-on behavior plus credential and probe semantics need maintainer direction first.
