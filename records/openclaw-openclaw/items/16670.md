## Summary

Keep open. Current main still has memorySearch documentation, diagnostics, and `openclaw memory status --deep`, but the setup/onboarding/configure flows do not provide the requested memory embedding setup path, and the mandatory/default-on question remains a maintainer product and credential-policy decision.

## What This Changes

Request adding a prominent or mandatory onboarding step that configures memory embedding provider/model/API-key and verifies the embedding endpoint.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes existing `agents.defaults.memorySearch` fields, supports the chosen SecretRef/plaintext policy, verifies readiness through a controlled status or embedding probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes for the source-level gap. Current main source inspection shows plain setup only writes workspace/gateway defaults, onboarding has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated setup/configure flow is likely the right implementation area, but mandatory versus optional/default-on behavior, credential storage, SecretRef UX, and probe semantics need maintainer direction first.
