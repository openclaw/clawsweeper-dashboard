## Label Justifications

- `P2`: The missing setup path affects first-run semantic-memory usability but does not disable the core runtime or all memory retrieval modes.
- `impact:session-state`: The request concerns durable cross-session recall and users receiving weaker retrieval than expected when embeddings are unavailable.
- `impact:auth-provider`: Embedding provider selection, model routing, API credentials, and SecretRef handling are central to the requested setup flow.

## Summary

Current main and the latest release still document and implement memory embedding configuration outside onboarding: the setup and configure flows omit a memory section, so the central request remains open. Existing keyword-only fallback and multiple memory backends make mandatory embeddings a product-policy decision rather than a straightforward bug fix.

## What This Changes

Add a prominent or mandatory onboarding step for configuring a memory embedding provider, model, credentials, and readiness verification.

## Best Possible Solution

Add a first-class memory readiness section to onboarding and `openclaw configure` that supports provider, model, and SecretRef-aware credentials plus a deep readiness probe, with maintainers explicitly choosing required versus recommended-with-skip behavior for each supported memory mode.

## Reproduction Assessment

Yes, at source level: current main’s setup and configure flows have no dedicated memory section, embedding credential prompt, or readiness probe. The broader claim of complete memory failure is not reproduced because current builtin memory can fall back to keyword retrieval.

## Solution Assessment

Partly. A first-class memory setup and diagnostics path is the narrow maintainable improvement, but universally mandatory embeddings are not yet the best established policy because keyword-only, local, and plugin-backed memory modes are valid.
