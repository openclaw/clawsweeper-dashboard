## Summary

Keep open. Current main and v2026.5.12 still have memory docs, doctor diagnostics, and `openclaw memory status --deep`, but they do not add a dedicated setup/onboarding/configure step for memory embedding provider/model/API-key or SecretRef setup; the remaining mandatory/default-on/diagnostic-only choice is a product and credential-policy decision.

## What This Changes

The issue asks for onboarding/setup to make memory embedding provider and API-key setup a prominent mandatory step so `memory_search` works across sessions.

## Best Possible Solution

Add a maintainer-approved memory setup path in onboarding and configure that writes the existing `memorySearch` fields, supports the chosen credential policy, verifies readiness through a controlled status/probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes for the source-level gap. Current main and v2026.5.12 show setup/configure/onboarding omit a memory section or dedicated `memorySearch` prompt while the memory docs still require usable provider credentials.

## Solution Assessment

Unclear as requested. A dedicated setup/configure path is likely the right implementation area, but mandatory versus optional/default-on behavior and credential/probe policy need maintainer direction first.
