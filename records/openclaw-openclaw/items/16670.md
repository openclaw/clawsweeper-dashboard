## Summary

Keep open: current main and the latest release still keep memory embedding provider/model/API-key setup outside onboarding and `openclaw configure`, while the requested mandatory behavior is a product and credential-UX decision rather than an already-implemented fix.

## What This Changes

The issue requests a mandatory onboarding wizard step that configures and verifies `agents.defaults.memorySearch` embedding provider credentials.

## Best Possible Solution

Add a maintainer-approved memory readiness path in onboarding and `openclaw configure` that configures SecretRef-aware embedding provider/model credentials and verifies readiness, with an explicit required-versus-recommended policy.

## Reproduction Assessment

Yes, at source level: current `openclaw setup`, setup wizard, and `openclaw configure` have no dedicated memory embedding setup section, credential prompt, or readiness probe. I did not live-run onboarding because this read-only review cannot run artifact-producing tests or commands.

## Solution Assessment

Partly. A first-class memory readiness/setup flow is a maintainable direction, but making it universally mandatory is not established across builtin, QMD, FTS-only, and plugin-backed memory modes.
