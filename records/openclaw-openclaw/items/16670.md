## Summary

Keep open. Current main still has manual memorySearch documentation, doctor diagnostics, and memory status probes, but setup/onboarding/configure do not provide a dedicated memory embedding provider/model/API-key or SecretRef setup step. The requested mandatory behavior also needs maintainer product and credential-policy judgment before a safe implementation path is chosen.

## What This Changes

Request to add a mandatory onboarding step that configures memorySearch embedding provider, model, API-key handling, and readiness verification.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and openclaw configure that writes agents.defaults.memorySearch provider/model plus API-key or SecretRef config, verifies readiness through a controlled probe or memory status path, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon environment behavior.

## Reproduction Assessment

Yes. Static inspection against current main reproduces the gap: setup only writes workspace/gateway defaults, the onboarding wizard has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated memory setup path is likely the right area, but making it mandatory may not be the narrowest maintainable UX until maintainers decide the credential, SecretRef, and live-probe policy.
