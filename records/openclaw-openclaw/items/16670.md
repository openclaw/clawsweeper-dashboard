## Summary

Keep open. Current main still documents and diagnoses memory search, but the setup/onboarding/configure flows do not offer a memory embedding provider/model/API-key or SecretRef step, and making that step mandatory needs maintainer product and credential-policy judgment.

## What This Changes

The issue asks for CLI onboarding/setup to add a prominent mandatory memory embedding setup step that configures `agents.defaults.memorySearch`, captures credentials, and verifies the embedding endpoint.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes existing `agents.defaults.memorySearch`, supports the chosen credential storage policy, verifies readiness through a controlled status/probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes. Static inspection of current main shows setup only writes workspace/gateway defaults, the onboarding flow has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated memory setup section is the likely implementation area, but making it mandatory is not clearly the best UX until maintainers decide the credential, SecretRef, daemon environment, and live-probe policy.
