## Summary

Keep open. Current main provides FTS-only fallback, memory diagnostics, and an embedding readiness probe, but setup, onboarding, and configure still lack a dedicated memory embedding choice. The exact “mandatory” policy needs maintainer judgment because not every valid memory path requires remote credentials.

**Likely related people**
- `steipete` — high confidence; substantial setup/configure/memory history and prior issue review.
- Vincent Koc — medium confidence; significant recent memory runtime and documentation work.
- `Takhoffman` — medium confidence; issue assignee, discussion participant, and setup-area contributor.

## What This Changes

Add a prominent or mandatory onboarding step for selecting and validating memory embedding configuration and credentials.

## Best Possible Solution

Add a prominent provider-aware memory readiness step that explains FTS-only, local, and remote embedding choices, preserves an explicit skip, stores credentials through existing SecretRef-aware configuration, and runs the existing embedding probe when semantic search is selected.

## Reproduction Assessment

Yes. Current main’s setup and configure flows have no dedicated memory embedding step, which is directly verifiable from their registries and orchestration code; this is a source-level UX reproduction rather than a runtime crash reproduction.

## Solution Assessment

No to the exact universally mandatory proposal. A prominent provider-aware choice is the narrower maintainable solution because current OpenClaw supports valid FTS-only and local embedding modes as well as credentialed remote providers.

## Risks / Open Questions

- Making embedding setup mandatory could unnecessarily require provider credentials or a network probe for users satisfied with FTS-only or local embedding paths.
- A live onboarding probe can add latency, provider cost, or first-run failure when the network is unavailable.
- Credential UX must distinguish model authentication from embedding credentials, preserve SecretRef handling, and account for daemon environments that do not inherit an interactive shell.
- The issue’s claims of complete memory loss and uniformly silent failure overstate current main behavior because FTS-only retrieval and explicit diagnostics now exist.
