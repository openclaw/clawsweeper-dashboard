## Summary

Keep open. Current main and the latest release still have memory docs, doctor diagnostics, and `openclaw memory status --deep`, but setup/onboarding/configure do not include a dedicated memory embedding provider/model/API-key or SecretRef step; the mandatory/default-on/diagnostic-only choice needs product and credential-policy direction.

## What This Changes

Request adding a prominent mandatory onboarding/setup step for configuring and verifying memory embedding provider credentials.

## Best Possible Solution

Add a maintainer-approved memory setup path in onboarding and configure that writes existing `memorySearch` fields, supports the chosen credential policy, verifies readiness through a controlled status/probe path, and updates setup/configure/memory docs.

## Reproduction Assessment

Yes for the source-level gap. Current main shows setup/configure/onboarding omit a memory section or dedicated `memorySearch` prompt while memory docs still require usable provider credentials for remote embeddings.

## Solution Assessment

Unclear as requested. A dedicated setup/configure path is likely the right implementation area, but mandatory versus optional/default-on behavior and credential/probe policy need maintainer direction first.
