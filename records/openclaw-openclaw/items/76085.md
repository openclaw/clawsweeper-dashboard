## Summary

Keep open. The PR addresses a real current-main allowlist bypass in the bundled provider runtime path, but the latest branch still needs user-facing docs/changelog/config-baseline coverage and the maintainer-requested note that the empty-allow `respect-allow` asymmetry is intentional.

## What This Changes

Adds an opt-in `plugins.bundledMode: "respect-allow"` mode so bundled provider runtime discovery can honor `plugins.allow` while preserving default compat behavior, with schema/help wiring and focused tests.

## Best Possible Solution

Land the opt-in mode after adding public docs, a changelog entry, the generated config baseline hash, and a small contract note or test covering the empty-allow behavior without changing the default `compat` semantics.

## Reproduction Assessment

Yes. On current main, a restrictive `plugins.allow` still lets bundled provider discovery return non-workspace provider plugins, and the activation compat path can add those discovered ids back into the effective allowlist.

## Solution Assessment

Yes as a direction, but not yet as a shippable patch. The opt-in mode is the narrow maintainable fix because it preserves legacy defaults while giving constrained deployments an explicit runtime gate; the remaining work is release/documentation alignment and the requested contract note.
