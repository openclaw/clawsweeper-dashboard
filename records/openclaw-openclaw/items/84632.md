## Label Justifications

- `P1`: The report describes a shipped regression that can write real provider credentials to generated agent state for active users.
- `impact:security`: The affected behavior can persist resolved secrets in plaintext on disk.
- `impact:auth-provider`: The bug is in provider API-key SecretRef resolution, marker persistence, and audit classification.

## Summary

Keep open: current `main` still has a source-reproducible SecretRef persistence gap in the `models status --probe` path. The command resolves `models.providers.*.apiKey` before probing, but when no source snapshot is already pinned it stores only the resolved config, so later `models.json` generation cannot reliably enforce pre-resolution non-secret markers for custom providers.

## What This Changes

The issue requests that `models status --probe` and other `models.json` regeneration paths never persist resolved plaintext `apiKey` values for SecretRef-backed custom model providers.

## Best Possible Solution

Preserve the pre-resolution source config through models status/probe/embedded generation and persist an audit-accepted non-secret marker for every SecretRef-backed provider `apiKey`, with custom-provider regression coverage.

## Reproduction Assessment

Source-reproducible with high confidence: `models status` resolves `models.providers.*.apiKey`, drops the source snapshot on unpinned command paths, and the generator then uses the resolved config as `sourceConfigForSecrets`. I did not execute the live probe because it can write per-agent state.

## Solution Assessment

The marker-only suggestion is directionally useful but incomplete if the command path has already lost the source snapshot. The maintainable fix should keep the source snapshot available and then choose one canonical marker form that `secrets audit` accepts for arbitrary custom providers.
