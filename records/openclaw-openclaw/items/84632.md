## Label Justifications

- `P1`: The report describes a shipped regression that can write real provider credentials to generated agent state for active users.
- `impact:security`: The affected behavior can persist resolved secrets in plaintext on disk.
- `impact:auth-provider`: The bug is in provider API-key SecretRef resolution, marker persistence, and audit classification.

## Summary

Keep open: current checked-out `main` still has the source-reproducible SecretRef persistence gap, and the linked closing PR is open but not merged. The issue should stay open until https://github.com/openclaw/openclaw/pull/84658 or an equivalent fix lands and the shipped/current-main proof shows generated `models.json` no longer receives resolved custom-provider API keys.

## What This Changes

The issue reports that `openclaw models status --probe` can rewrite per-agent `models.json` with resolved plaintext custom-provider API keys instead of non-secret SecretRef markers.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/84658 so unpinned models commands preserve the pre-resolution source snapshot and custom-provider SecretRef API keys regenerate as non-secret markers; keep https://github.com/openclaw/openclaw/issues/84376 scoped to the separate Codex marker audit case.

## Reproduction Assessment

Yes, source-reproducible with high confidence: `models status` resolves `models.providers.*.apiKey`, the unpinned loader stores only the resolved config, and `models.json` generation then lacks the source snapshot needed to emit custom-provider SecretRef markers. I did not run the live probe because this read-only review must not write agent state.

## Solution Assessment

Yes, preserving the pre-resolution source snapshot through `loadModelsConfigWithSource()` is the narrow maintainable fix. A marker-format-only change would not address the source loss when the generator only sees resolved runtime values.
