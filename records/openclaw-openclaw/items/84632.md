## Label Justifications

- `P1`: The issue describes a shipped regression that can write real provider credentials into generated agent state during a normal operator workflow.
- `impact:security`: The affected path can persist resolved secrets in plaintext on disk.
- `impact:auth-provider`: The bug is in provider API-key SecretRef resolution, marker persistence, and generated model auth state.

## Summary

Keep open: current `main` still has the source-reproducible SecretRef persistence gap, and the linked fix PR is open with closing syntax, so this issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports that `models status --probe` can persist resolved custom-provider SecretRef API keys into generated per-agent `models.json`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84658, or an equivalent source-snapshot fix with the same custom-provider SecretRef regression coverage; keep marker-classification follow-ups such as https://github.com/openclaw/openclaw/issues/84376 separate.

## Reproduction Assessment

Yes. Source inspection shows `models.providers.*.apiKey` is resolved for model commands, the unpinned command path drops the source snapshot, and generated `models.json` then uses the resolved config as `sourceConfigForSecrets`.

## Solution Assessment

Yes. Preserving the pre-resolution source snapshot through `loadModelsConfigWithSource()` is the narrowest maintainable fix because downstream `models.json` generation already has source-authoritative marker enforcement.
