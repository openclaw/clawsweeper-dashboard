## Summary

Keep open. Current main still only supports static Slack credentials; there is no Slack schema, runtime, CLI, or docs surface for oauth.v2 token rotation, and the requested flow is security-sensitive enough to need maintainer design before implementation.

## What This Changes

Add native Slack token rotation in the bundled Slack plugin, including refresh credential storage, oauth.v2.exchange bootstrap, oauth.v2.access refresh, runtime reload, and status visibility.

## Best Possible Solution

Implement this in the bundled Slack plugin with a reviewed credential-storage model, serialized refresh, atomic token persistence, runtime client reload/reconnect, non-secret status/doctor output, and focused Slack docs/tests; add a generic SDK seam only if the plugin cannot do this through existing public APIs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but source inspection gives a high-confidence absence check: the strict Slack schema, runtime resolver, monitor startup, docs, and repo searches have no Slack oauth.v2 rotation path.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the exact config/storage shape needs maintainer review; native support belongs in the Slack plugin, while refresh-token persistence may be safer through existing secret or auth-store patterns than raw config patching.

## Risks / Open Questions

- The safest persistence model for Slack client secrets and rotating refresh tokens is undecided; writing every rotated secret directly into openclaw.json may not match existing OAuth auth-store patterns.
- Slack refresh tokens are one-use and repeated refreshes can revoke older active tokens, so any implementation needs serialized per-account refresh and atomic persistence before runtime token swap.
- Reload/reconnect behavior is non-trivial because Socket Mode and HTTP mode both construct clients with fixed tokens today.
