## Summary

Keep open for maintainer product and security design: current main and v2026.6.10 still model Slack credentials as static bot/app/signing/user tokens, while this asks for a new Slack OAuth refresh-token lifecycle with secure persistence and runtime adoption.

## What This Changes

The issue requests native Slack OAuth token rotation: store refresh credentials, refresh bot tokens with `oauth.v2.access`, and adopt refreshed tokens without an external daemon or manual restart.

## Best Possible Solution

Implement this as a Slack-plugin-owned secure OAuth credential lifecycle with transactional refresh, runtime token adoption, redacted status/recovery UX, docs, and live proof; avoid using `openclaw.json` as the mutable refresh-token store unless maintainers explicitly choose that tradeoff.

## Reproduction Assessment

Not applicable as a bug reproduction: this requests a new Slack OAuth credential lifecycle. Source, docs, related issue search, and shipped-tag inspection show the capability is still missing.

## Solution Assessment

No for the exact proposed mutable `openclaw.json` token-rotation shape. Native Slack rotation is a reasonable direction, but the maintainable fix needs a plugin-owned secure storage and runtime adoption design first.
