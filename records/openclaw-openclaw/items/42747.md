## Summary

Keep open for maintainer product and security design: current main and v2026.6.6 still use static Slack bot/app/signing-token configuration, while native Slack token rotation needs a new OAuth refresh-token lifecycle, storage model, and runtime adoption path.

## Best Possible Solution

Design Slack-plugin-owned OAuth credential storage first, then implement transactional refresh, runtime token adoption, redacted status/recovery UX, docs, and live proof for both Socket Mode and HTTP mode.

## Risks / Open Questions

- Slack refresh-token rotation is security-sensitive because refresh tokens and replacement ordering need durable, transactional storage; a crash after Slack accepts a refresh but before OpenClaw stores the new refresh token could strand the installation.
- The proposed openclaw.json fields would add mutable client secrets, refresh tokens, and expiry state to broad user config unless maintainers choose a narrower credential store.
- Socket Mode, HTTP mode, inbound Bolt authorization, outbound Web API clients, cached write clients, probes, setup/status UX, and reload behavior need one refreshed-token source of truth.
- The related static-token reload issue is not a substitute for this OAuth lifecycle design.
