## Summary

Keep open. Current main still supports static Slack credentials only and treats `token_expired` as a non-recoverable Slack auth failure, so native OAuth token rotation remains unimplemented and security-sensitive.

## What This Changes

Request native Slack token rotation that stores refresh credentials, refreshes bot tokens via `oauth.v2.access`, updates the running Slack runtime, and adds init/status CLI support.

## Best Possible Solution

Implement this inside the Slack plugin with reviewed SecretRef-aware persistence, schema/docs/status/doctor coverage, serialized `oauth.v2.access` refresh, safe bootstrap flow, and runtime reload behavior for both Socket Mode and HTTP mode; add a generic SDK seam only if the plugin cannot persist credentials safely today.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; current-main source inspection confirms the Slack config, secret registry, provider startup, status docs, and auth-error tests lack a rotation/refresh path.

## Solution Assessment

Is this the best way to solve the issue? Directionally yes that the bundled Slack plugin should own token rotation, but the proposed raw config-patching and SIGUSR1 shape should be refined into a reviewed secret-persistence and runtime-reload design first.
