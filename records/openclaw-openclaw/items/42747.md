## Summary

Keep open. Current main still lacks native Slack token rotation: the Slack config/runtime surfaces only resolve static bot/app/user/signing-secret credentials, the monitor passes fixed tokens into Bolt, there is no `oauth.v2.exchange` or `oauth.v2.access` refresh path, and `token_expired` is still treated as a non-recoverable auth failure. Slack's own docs confirm the requested refresh flow and 12-hour expiring-token behavior, so this is a real bundled Slack plugin gap rather than a stale request. citeturn1view0turn1view1

## What This Changes

Request native Slack token rotation for the bundled Slack channel, including refresh credential storage, oauth.v2.exchange bootstrap, oauth.v2.access refreshes, runtime token replacement, and setup/status visibility.

## Best Possible Solution

Implement this in the bundled Slack plugin with schema-backed sensitive rotation credentials or credential-store entries, oauth.v2.exchange bootstrap, serialized oauth.v2.access refresh before expiry, atomic persistence of the new bot token/refresh token/expiresAt, Slack client reconnect or rebuild behavior, non-secret status/doctor signals, and docs/tests for both Socket Mode and HTTP mode. Add a generic plugin SDK persistence seam only if the Slack plugin cannot do this safely through existing public APIs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but current-main inspection gives a high-confidence absence check: the strict Slack schema has no rotation object and the runtime only resolves static tokens.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally, but the exact storage shape needs maintainer review. Native support belongs in the existing Slack plugin, while the safest persistence model may use existing secret/credential-store patterns rather than writing every rotated secret directly into `openclaw.json`.
