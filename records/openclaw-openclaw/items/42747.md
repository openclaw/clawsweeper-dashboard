## Summary

Keep open for explicit maintainer product and security design. Current main and the latest release still use static Slack credentials, while Slack’s supported rotation lifecycle requires secure installation persistence, atomic replacement of single-use refresh tokens, and runtime authorization adoption; no duplicate implementation currently owns that work.

## What This Changes

Add native Slack OAuth token refresh, secure credential persistence, runtime rotation, and setup/status flows.

## Best Possible Solution

Design and approve a Slack-plugin-owned installation credential store and authorization interface, then implement one-time exchange, transactional token-pair rotation, atomic runtime adoption, redacted status/recovery UX, and live proof for both Socket and HTTP modes.

## Risks / Open Questions

- A crash after Slack invalidates the old refresh token but before the replacement pair is durably committed can strand the installation.
- Writing managed refresh tokens and expiry state into openclaw.json would add sensitive mutable runtime state to a configuration surface the repository intentionally keeps narrow.
- Socket Mode and HTTP mode must adopt refreshed authorization consistently across inbound Bolt handling and outbound Web API clients; relying on external file-watch reload alone is not a robust contract.
- The one-time oauth.v2.exchange bootstrap changes rollback behavior because the legacy token expires after rotation begins, requiring explicit recovery and operator guidance.
