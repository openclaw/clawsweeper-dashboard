## Summary

Keep open. Current main still only accepts static Slack credentials and treats expired Slack tokens as auth failures, so the requested oauth.v2 refresh flow remains an unimplemented, security-sensitive feature that needs maintainer design.

## What This Changes

Add native Slack token rotation support with config fields, oauth.v2 refresh behavior, an initialization command, and operational status/docs coverage.

## Best Possible Solution

Implement this in the Slack plugin with reviewed secret persistence, serialized oauth.v2.access refresh, atomic token updates, Socket Mode and HTTP runtime reload behavior, non-secret status/doctor output, docs, and focused tests; add a generic SDK seam only if current plugin APIs cannot persist safely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but current-main source inspection gives a high-confidence absence check across Slack schema, runtime token resolution, monitor startup, docs, and auth-error handling.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally for native Slack plugin support, but the proposed raw config-patching shape is not yet proven to be the safest maintainable storage and reload model.

## Security Review

Status: needs_attention

Summary: No PR diff exists, but the requested feature would add long-lived Slack client and refresh credentials, so storage, redaction, refresh serialization, and reload behavior need explicit security review.

Concerns:

- **[medium] Design refresh-token storage before implementation:** `extensions/slack/src/secret-contract.ts:10`
  - body: The Slack secret registry only covers static app, bot, user, and signing-secret fields today. A rotation implementation must add redaction and persistence for clientSecret and refreshToken without logging secrets or unsafe plaintext churn.
  - confidence: 0.9
- **[medium] Serialize Slack token refreshes:** `extensions/slack/src/monitor/provider.ts:204`
  - body: Slack refresh tokens are replacement credentials, so concurrent oauth.v2.access calls can lose the newest refresh token or leave the running Bolt client using stale credentials unless refresh and persistence are serialized.
  - confidence: 0.86
