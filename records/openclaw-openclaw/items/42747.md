## Summary

Keep open. Current main and v2026.5.22 still only model Slack bot/app/user/signing-secret credentials; native Slack token rotation remains a security-sensitive feature requiring maintainer design for refresh-token storage, oauth.v2.access refresh, runtime rebuild, and setup/status UX.

## What This Changes

Request native Slack OAuth token rotation support so OpenClaw can store refresh credentials, refresh expiring Slack bot tokens with oauth.v2.access, and expose setup/status commands.

## Best Possible Solution

Implement this in the official Slack plugin after security/product review, using a credential-persistence contract and focused coverage for bootstrap, refresh success/failure, redaction, and provider rebuild behavior.

## Reproduction Assessment

Not applicable as a feature request. Current-main source, docs, command-surface search, related-issue search, and v2026.5.22 inspection confirm the requested Slack token-rotation config, OAuth refresh runtime, and setup/status commands are absent rather than broken documented behavior.

## Solution Assessment

Unclear pending maintainer security/product design. The capability belongs in the Slack plugin, but the proposed direct config-file patch plus SIGUSR1 pattern should be replaced or formalized through a reviewed credential-persistence and provider-reload contract.

## Risks / Open Questions

- Adding refresh-token support without an explicit credential contract could persist mutable OAuth refresh material in the wrong place or miss redaction/audit/backup expectations.

- The setup/status UX and oauth.v2.access bootstrap flow require a product decision because the issue proposes new CLI commands and new sensitive config fields.
