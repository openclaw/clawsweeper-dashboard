## Summary

Keep open for maintainer product and security design. Current main and v2026.6.6 still configure Slack with static bot/app/signing/user tokens, while Slack token rotation requires expiring access tokens, single-use refresh-token replacement, and durable secure storage; the related https://github.com/openclaw/openclaw/issues/80346 covers static-token reload reliability, not native OAuth rotation. ([docs.slack.dev](https://docs.slack.dev/authentication/using-token-rotation?utm_source=openai))

## What This Changes

Request native Slack OAuth token rotation support, including bootstrap, refresh-token persistence, proactive oauth.v2.access refresh, runtime token adoption, and operational status/recovery behavior.

## Best Possible Solution

Design Slack-plugin-owned OAuth installation storage and authorization first, then implement exchange, transactional refresh-token rotation, runtime token adoption, redacted status/recovery UX, and live proof for Socket and HTTP modes.

## Reproduction Assessment

Not applicable: this requests a new Slack OAuth credential lifecycle rather than a broken existing OpenClaw contract. Current source, docs, release inspection, and Slack’s official API contract are enough to show the capability is missing.

## Solution Assessment

No, not in the exact proposed shape. Native Slack-plugin support is reasonable, but managed rotation state should use a secure transactional installation/credential store and explicit runtime authorization adoption rather than periodic mutation of general config.
