## Summary

Keep open. Current main has the Telegram `audioAsVoice` delivery handoff, but the TTS tool-result path can still filter out trusted local `/tmp/...ogg` media before it is queued when `builtinToolNames` is present without raw `tts`; open PR #74901 already tracks that remaining gap with closing syntax.

## What This Changes

The issue requests that successful TTS tool output be delivered to Telegram as a voice message instead of only returning `(spoken)` text.

## Best Possible Solution

Land or replace the active narrow fix so trusted structured TTS local media is allowed through the filter only when the result itself marks it trusted, with regression coverage for the `builtinToolNames`-present-but-missing-`tts` case and Telegram voice payload delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `filterToolResultMediaUrls('tts', ['/tmp/reply.opus'], { details: { media: { trustedLocalMedia: true } } }, new Set(['other_tool']))`, which current main reduces to no local media before the queueing handler can deliver it.

## Solution Assessment

No, current main is not yet the complete solution. The merged outbound work is the right Telegram-side foundation, and PR #74901 is the narrow active direction for the remaining trust-filter gap, but this issue should stay open until that patch lands or is replaced.
