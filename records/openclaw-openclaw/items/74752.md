## Summary

Keep open. Current main emits trusted, voice-capable TTS media and now maps `audioAsVoice` through Telegram payload delivery, but the tool-result filter can still drop local TTS audio when `builtinToolNames` is present without raw `tts`; open PR #74901 uses closing syntax for this issue and should be reviewed or replaced rather than closing the issue early.

## What This Changes

The issue requests that the TTS tool’s `(spoken)` result automatically deliver generated audio as a Telegram voice message instead of falling back to a text reply.

## Best Possible Solution

Land or replace PR #74901 with a narrow change that honors structured `trustedLocalMedia` for OpenClaw-owned TTS output, keeps untrusted local paths blocked, and adds regression coverage for the `builtinToolNames`-missing-`tts` case through pending media delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `filterToolResultMediaUrls('tts', ['/tmp/reply.opus'], ttsResultWithTrustedLocalMedia, new Set(['message']))`, which returns no local media on current main even though the TTS result marks the media as trusted and voice-capable.

## Solution Assessment

No, current main is not yet the complete fix. The merged outbound work addresses Telegram `audioAsVoice` delivery, but the remaining filter gap should be fixed in the tool-result media trust path, as PR #74901 proposes.
