## Summary

Keep open. Current main still has a source-level gap for the reported TTS-to-Telegram path: local TTS media can be filtered before queueing when exact built-in tool names omit `tts`, and the Telegram outbound adapter still does not map `audioAsVoice` onto Telegram's `asVoice` send option. Open PR #74901 uses closing syntax for this issue, while merged PR #74920 only covers shared propagation, so the issue should remain open until the coordinated fix lands or is replaced.

## What This Changes

A user reports that the TTS tool returns `(spoken)` with generated Microsoft OGG audio on Telegram, but OpenClaw sends text instead of an automatic Telegram voice message.

## Best Possible Solution

Land or replace the active fix work with one coordinated patch that preserves structured trusted TTS local media without weakening alias/MCP local-file safeguards, maps Telegram `audioAsVoice` to low-level `asVoice`, and adds regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through tool-result handling when `builtinToolNames` is present but lacks raw `tts`; current main can filter the media away before queueing, and the Telegram adapter still drops the voice hint before `sendVoice`.

## Solution Assessment

No, not yet. PR #74901 addresses the local-media trust direction, and PR #74920 addressed shared propagation, but the maintainable fix needs both the safe filter change and the Telegram adapter `audioAsVoice` to `asVoice` handoff.
