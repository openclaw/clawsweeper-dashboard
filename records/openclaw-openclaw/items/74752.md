## Summary

Keep open. Current main has the Telegram voice-send plumbing, but the TTS tool-result delivery path can still drop trusted local `/tmp/...ogg` media before queueing when `builtinToolNames` is present without raw `tts`; open PR #74901 already uses closing syntax for this issue and should be reviewed or replaced rather than racing a duplicate fix.

## What This Changes

The issue requests that successful TTS tool output deliver generated audio as a Telegram voice message instead of only sending the `(spoken)` text.

## Best Possible Solution

Land or replace the active narrow fix so trusted structured TTS local media survives the filter only when the result itself marks it trusted, with regression coverage for the `builtinToolNames`-present-but-missing-`tts` case and Telegram voice payload delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `tts` tool result with `/tmp/reply.opus`, `details.media.trustedLocalMedia: true`, and `builtinToolNames` set to something that does not include `tts`; current main filters that local media to an empty list before queueing.

## Solution Assessment

No, current main is not yet the complete solution. The merged outbound work is the right Telegram-side foundation, and PR #74901 is the narrow active direction for the remaining trust-filter gap, but this issue should stay open until that patch lands or is replaced.
