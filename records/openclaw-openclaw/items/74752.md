## Summary

Keep open. Current main still has the remaining TTS tool-result filtering gap: the TTS tool emits trusted local audio metadata, and Telegram can send `audioAsVoice` payloads as voice notes, but the queueing path can drop the local `/tmp/...opus` media when `builtinToolNames` is present without the exact raw `tts` name. Open PR #74901 already targets this gap with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that Microsoft-backed TTS returns `(spoken)` but Telegram receives a text reply instead of the generated voice message.

## Best Possible Solution

Review, land, or replace PR #74901 with a narrow fix that lets structured TTS `trustedLocalMedia` survive the filter without weakening the existing exact-name protection for untrusted local media, plus regression coverage for the `builtinToolNames`-present-but-missing-`tts` case and Telegram voice payload delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `filterToolResultMediaUrls("tts", ["/tmp/reply.opus"], { details: { media: { mediaUrl: "/tmp/reply.opus", trustedLocalMedia: true, audioAsVoice: true } } }, new Set(["web_search"]))`, which current main reduces to no local media before the queueing handler can deliver it.

## Solution Assessment

Yes for the active direction, not for current main. The narrow maintainable fix is to honor trusted structured TTS media at the filter boundary while preserving the collision/MCP safeguards, then prove the handoff with focused tests.
