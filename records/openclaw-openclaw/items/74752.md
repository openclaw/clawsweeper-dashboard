## Summary

Keep open. Current main now forwards Telegram `audioAsVoice` payloads to the voice-send path, but the TTS tool-result local media can still be filtered out before queueing when `builtinToolNames` is present without the raw `tts` name; open PR #74901 uses closing syntax for this issue and should be reviewed, landed, or replaced before cleanup.

## What This Changes

The issue reports that the `tts` tool returns `(spoken)` for Telegram while delivering visible text instead of the generated audio as a Telegram voice message.

## Best Possible Solution

Land or revise the active PR into one coordinated patch that lets TTS-owned `trustedLocalMedia` survive the filter, keeps the current Telegram `audioAsVoice` to `asVoice` behavior, and adds focused regression coverage for a TTS-originated local OGG/Opus Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `tts` tool result with `details.media.mediaUrl` set to a local `/tmp/...ogg`, `trustedLocalMedia: true`, and `audioAsVoice: true`, while `builtinToolNames` is defined but lacks raw `tts`; current main filters the media to an empty list before queueing.

## Solution Assessment

No, current main is not the full solution yet. The Telegram send handoff is fixed, but the remaining maintainable fix is the narrow media-trust change from #74901 plus regression coverage, not closing the issue before that PR lands or is replaced.
