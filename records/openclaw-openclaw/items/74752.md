## Summary

Keep open. Current main has the merged shared/Telegram payload-side `audioAsVoice` propagation, but the TTS tool-result local-media path can still drop the generated `/tmp/...ogg` before queueing when `builtinToolNames` is present without raw `tts`; open PR #74901 uses closing syntax for this issue, so the issue should remain open until that fix is landed, updated, or replaced.

## What This Changes

The issue asks for the TTS tool's generated audio to be delivered automatically to Telegram as a voice message instead of returning `(spoken)` while the user receives text only.

## Best Possible Solution

Land or replace the active fix with a narrow change that honors structured `trustedLocalMedia` for trusted TTS output without weakening MCP, alias, or unregistered-tool local-file safeguards, and add regression coverage for the `builtinToolNames`-missing-`tts` case through Telegram voice payload delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `filterToolResultMediaUrls('tts', ['/tmp/reply.opus'], ttsResultWithTrustedLocalMedia, new Set(['other_tool']))`, which returns no local media on current main even though the TTS result marks the media as trusted and voice-capable.

## Solution Assessment

No, current main is not yet the complete fix. The merged outbound work solves the Telegram payload handoff, and PR #74901 is the right narrow direction for the remaining trust-filter gap, but the issue should stay open until that patch is reviewed and merged or replaced.
