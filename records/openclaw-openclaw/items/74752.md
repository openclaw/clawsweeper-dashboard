## Summary

Keep open. Current main has the Telegram `audioAsVoice` to `asVoice` outbound handoff, but the TTS tool-result path can still drop trusted local `/tmp/...ogg` media before queueing when `builtinToolNames` is present without raw `tts`; open PR #74901 already tracks that remaining gap with closing syntax.

## What This Changes

The issue reports that Telegram TTS returns `(spoken)` and generates Microsoft OGG audio, but automatic TTS delivery sends visible text instead of a Telegram voice message.

## Best Possible Solution

Land or replace PR #74901 with a narrow fix that honors structured `trustedLocalMedia` for TTS local media while preserving the raw-name collision guard for untrusted tool output, plus regression coverage for `builtinToolNames` present but missing `tts` and Telegram voice payload delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `filterToolResultMediaUrls('tts', ['/tmp/reply.opus'], { details: { media: { mediaUrl: '/tmp/reply.opus', trustedLocalMedia: true, audioAsVoice: true } } }, new Set(['web_search']))`, which current main reduces to no local media before the handler can queue voice delivery.

## Solution Assessment

No, current main is not yet the complete solution. The merged outbound work is the right Telegram-side foundation, and PR #74901 is the narrow active direction for the remaining trust-filter gap, but this issue should stay open until that patch lands or is replaced.
