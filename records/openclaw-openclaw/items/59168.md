## Summary

Keep open. Current main still uses provider/model refs as the model identity and has no accepted provider/name keying or equivalent selectable-profile-to-upstream-model mapping; the related routeTo work is still an open, unmerged product/config contract discussion.

## What This Changes

The issue proposes separating selectable OpenClaw model identity from the upstream API model id by using provider/name as the internal model key and leaving id as the provider-facing model id.

## Best Possible Solution

Define a backward-compatible model profile contract that separates selectable OpenClaw refs from provider-facing ids, then update schema, runtime resolution, params, pickers/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that provider/name keying or an equivalent profile-to-upstream model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing name as the internal key is one possible design, but an explicit profile-routing contract such as routeTo/apiModelId may be safer and needs maintainer product choice before implementation.
