## Summary

Keep open. Current main still uses provider/model ids as the configured model identity and provider-facing runtime id, and there is no provider/name keying or explicit profile-to-upstream model mapping; the related open issue and PR show this config contract still needs maintainer direction.

## What This Changes

Request to use provider/name, or an equivalent profile key, as OpenClaw's internal model identity so multiple selectable profiles can route to the same upstream API model id.

## Best Possible Solution

Define a backward-compatible model profile contract that separates selectable OpenClaw refs from provider-facing ids, then update schema, model selection, params, runtime resolution, pickers/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that provider/name keying or an equivalent profile-to-upstream model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing name as the internal key is one possible design, but a documented profile-routing contract needs maintainer choice before a narrow implementation is safe.
