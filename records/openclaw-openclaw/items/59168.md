## Summary

Keep open: current main still uses `provider/model-id` as the model-ref contract and has no merged provider/name or equivalent routed model-profile contract; the closest implementation path is still the open derived-model `routeTo` PR.

## What This Changes

The issue proposes changing internal model identity from `provider/id` to `provider/name`, with `id` reserved for the upstream API model name and model-profile reasoning levels attached to entries.

## Best Possible Solution

Define one backward-compatible routed model-profile contract, then update schema/help/docs, picker/listing behavior, runtime model resolution, session/fallback handling, migration or doctor guidance, and focused regression coverage together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that current main still lacks provider/name keying or an equivalent merged routed model-profile mapping.

## Solution Assessment

Unclear. Reusing `name` as the internal key is plausible, but an explicit `routeTo` or `apiModelId`-style contract may be safer and needs maintainer product judgment before implementation.
