## Summary

Keep this open. Current main still treats configured provider model `id` as the selectable/runtime key and uses `name` as display metadata, so the same-provider duplicate-API-ID profile problem remains; the related derived-model `routeTo` PR is open and unmerged, and a narrower suffix-stripping issue was already closed in favor of this canonical discussion.

## What This Changes

The issue proposes changing OpenClaw's internal model key from `provider/id` to `provider/name` so one upstream API model ID can have multiple selectable provider-local profiles.

## Best Possible Solution

Choose one canonical model-profile contract, likely either a reviewed `routeTo`/derived-ref surface or a documented provider/name migration, and include schema, runtime, UI, session-state, docs, and doctor-migration behavior in the same design.

## Reproduction Assessment

Yes, at source level: current main still resolves configured provider rows and runtime model metadata by provider plus `id`, while `name` remains display metadata. I did not run a live config because this was a read-only review, but the code path is direct and covered by current source inspection.

## Solution Assessment

Unclear: provider/name would solve the exact reported problem, but it is a broad config-contract migration. A dedicated derived-ref route such as the open `routeTo` proposal may be a narrower maintainable solution if maintainers accept the API and migration shape.
