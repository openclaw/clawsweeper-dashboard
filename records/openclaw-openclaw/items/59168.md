## What This Changes

The issue proposes using a provider/name-style internal model-profile key so multiple selectable profiles can share one upstream API model id.

## Best Possible Solution

Keep this as the canonical maintainer design issue until OpenClaw chooses one model-profile contract and implements it consistently across schema, runtime routing, UI/session state, docs, and doctor migration.

## Reproduction Assessment

Yes, at source level: current main still builds selectable and configured model keys from provider plus `id`, while configured row `name` remains display metadata. I did not run a live config because this was a read-only cleanup review, but the code path is direct.

## Solution Assessment

Unclear: using `provider/name` would solve the reported duplicate API-id profile gap, but it repurposes a display field and needs maintainer approval versus an explicit routed-profile field such as a redesigned `routeTo`/`apiModelId` contract.
