## Summary

Keep open. Current main still lacks provider/name keying or an equivalent selectable model-profile to provider-facing model-id contract, and the related derived-model routing PR is still open rather than shipped.

## What This Changes

The issue proposes decoupling OpenClaw's internal model key from the upstream API model id so one provider can expose multiple selectable profiles for the same API model.

## Best Possible Solution

Define one backward-compatible selectable model-profile routing contract, then update schema/help/docs, model listing and picker behavior, runtime dynamic preparation, params/context budgeting, sessions, fallbacks, migration, and regression tests together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that current main still has no provider/name keying or equivalent selectable profile-to-upstream-model mapping.

## Solution Assessment

Unclear. Reusing `name` as the internal key is plausible, but a dedicated `routeTo` or `apiModelId`-style contract may be safer and needs maintainer product judgment before implementation.
