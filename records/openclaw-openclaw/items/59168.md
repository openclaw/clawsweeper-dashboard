## Summary

Keep this open: current main and the latest release still key selectable/configured model entries by `provider/model id`, and the only concrete routed-profile implementation is still an open PR with unresolved contract and upgrade-safety questions.

## What This Changes

The issue proposes decoupling OpenClaw's selectable/internal model key from the provider-facing API model id, using `provider/name` and expanding model-profile reasoning configuration.

## Best Possible Solution

Define one backward-compatible selectable model-profile contract, then update config types, strict schemas, docs, model listing/pickers, runtime routing, session/fallback/context handling, doctor or migration guidance, and regression proof together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that current main and `v2026.5.22` still lack provider/name keying or an equivalent selectable profile-to-upstream-model mapping.

## Solution Assessment

Unclear. Reusing `name` as the internal key is plausible, but `routeTo` or `apiModelId` may be safer because `name` is already display-oriented; maintainers should choose the public contract before implementation.
