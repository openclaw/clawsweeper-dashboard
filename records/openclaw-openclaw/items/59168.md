## Summary

Keep open. Current main still keys configured model aliases/params and runtime resolution by provider/model id, not provider/name or an explicit profile-to-API-model mapping, and the related narrower issue https://github.com/openclaw/openclaw/issues/62629 plus open PR https://github.com/openclaw/openclaw/pull/60506 show the config contract is still unsettled.

## What This Changes

Requests a model identity/config redesign where selectable OpenClaw model keys are decoupled from provider-facing API model ids, with per-profile params and reasoning support.

## Best Possible Solution

Define a backward-compatible model profile contract that separates selectable OpenClaw refs from provider-facing ids, then update schema, model selection, params, picker/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that the requested provider/name keying or equivalent profile-to-API-model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing name as the internal key is one possible design, but the safer maintainer path is to decide an explicit documented profile-routing contract before changing long-lived provider/model id semantics.
