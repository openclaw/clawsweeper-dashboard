## Summary

Keep open. Current main still has no provider/name keying or equivalent profile-to-upstream-model contract; this remains a broad model config design issue rather than a closeable implemented or duplicate cleanup item.

## What This Changes

The issue requests using `provider/name` as the selectable internal model key while keeping `id` as the provider-facing API model id, plus optional reasoning-level profile behavior.

## Best Possible Solution

Define a backward-compatible model profile/routing contract that separates selectable OpenClaw refs from provider-facing model ids, then update schema, runtime resolution, params, pickers/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that provider/name keying or an equivalent selectable-profile-to-upstream-model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing `name` as the internal key is one possible design, but an explicit routing/profile contract such as `routeTo` or `apiModelId` may be safer and needs maintainer product judgment first.
