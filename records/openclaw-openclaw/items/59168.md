## Summary

Keep open. Current main still uses provider/model-id refs as the selectable model identity and does not expose an accepted provider/name, routeTo, apiModelId, or equivalent profile-to-upstream-model mapping; the overlapping routeTo work remains an open PR with unresolved contract/integration questions.

## What This Changes

Request to separate selectable OpenClaw model identity from the upstream API model id, using provider/name keys and allowing multiple per-model configuration profiles.

## Best Possible Solution

Define a backward-compatible model profile/routing contract that separates selectable OpenClaw refs from provider-facing ids, then update schema, runtime resolution, params, pickers/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection on current main gives high confidence that provider/name keying or an equivalent profile-to-upstream-id mapping is still absent.

## Solution Assessment

Unclear. The provider/name design is one possible solution, but an explicit routeTo/apiModelId/profile contract may be safer and should be decided before implementation; https://github.com/openclaw/openclaw/pull/60506 is not merged.
