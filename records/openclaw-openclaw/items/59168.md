## Summary

Keep open. Current main still keys configured model aliases, params, catalog metadata, and runtime resolution by provider/model id, not by provider/name or any explicit profile-to-provider-model mapping; the related narrower issue and open PR show the public config contract is still unsettled.

## What This Changes

Request a model identity/profile contract that separates OpenClaw's selectable internal model key from the provider-facing API model id, with optional reasoning-level profile semantics.

## Best Possible Solution

Define a backward-compatible model profile contract that separates selectable OpenClaw refs from provider-facing ids, then update schema, model selection, params, picker/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that provider/name keying or an equivalent profile-to-API-model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing name as the internal key is one possible design, but the safer maintainer path is to decide an explicit documented profile-routing contract before changing long-lived provider/model id semantics.
