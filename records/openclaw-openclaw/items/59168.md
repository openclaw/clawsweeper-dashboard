## Summary

Keep open. Current main still lacks the requested provider/name keying or an equivalent route/profile mapping: the config schema only accepts the existing per-model entry fields, provider model definitions still treat `id` as the model id and `name` as metadata, and runtime payload paths still send `model.id`; the related `routeTo` PR remains open and needs config-contract review.

## What This Changes

Use a distinct selectable model profile key, proposed as `provider/name`, while keeping `id` as the upstream API model id and allowing model-entry reasoning levels.

## Best Possible Solution

Define a backward-compatible model profile/route contract that separates selectable OpenClaw refs from provider-facing model ids, then update schema, runtime resolution, pickers/status, sessions, fallbacks, docs, and migration together.

## Reproduction Assessment

Not applicable as a feature/config-contract proposal. Source inspection gives high confidence that provider/name keying or an equivalent profile-to-upstream model mapping is still absent on current main.

## Solution Assessment

Unclear. Reusing `name` as the internal key is one possible design, but https://github.com/openclaw/openclaw/pull/60506 shows the maintainable answer still needs a chosen route/profile contract before implementation.
