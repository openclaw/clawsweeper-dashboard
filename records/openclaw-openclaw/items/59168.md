- `P2`: The request affects model/provider routing and configuration ergonomics, but it is a normal-priority design improvement rather than an urgent outage.
- `impact:session-state`: Changing the internal model key would affect active model identity, model overrides, context/profile lookup, and persisted session state.
- `impact:auth-provider`: The request changes provider/model routing and the provider-facing model id used for runtime API calls.

## Summary

Keep open. Current main and the latest release still treat selectable/configured model identity as provider plus model id, while this issue asks for a new durable model-profile/API-id contract that needs maintainer product and migration direction.

## What This Changes

Request to use `provider/name` as the internal model profile key while keeping `id` as the upstream API model id, plus allow model-level reasoning levels.

## Best Possible Solution

Keep this as the canonical design issue until maintainers choose one model-profile/API-id contract and implement it consistently across schema, runtime resolution, UI/session state, docs, and doctor migration.

## Reproduction Assessment

Yes, at source level. Current main still resolves configured/selectable model identity from provider plus model id, while `name` remains metadata and no profile-to-API-id mapping exists.

## Solution Assessment

Unclear. The issue identifies a real gap, but `provider/name` is not proven to be the best permanent contract over an explicit `apiModelId`, `routeTo`, or another profile field.
