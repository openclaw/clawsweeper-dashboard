## Summary

Keep open. Current main and v2026.6.10 still key selectable/configured model identity as provider plus model id, while this issue asks for a new model-profile/API-id contract that needs maintainer product and migration direction.

## What This Changes

The issue requests decoupling OpenClaw's internal selectable model key from the upstream API model id, with provider/name suggested as the internal key.

## Best Possible Solution

Keep this as the canonical design issue until maintainers choose one model-profile/API-id contract and implement schema, runtime resolution, UI/session state, docs, and doctor migration together.

## Reproduction Assessment

Yes, at source level. Current main still resolves configured and selectable model identity from provider plus model id, while name remains metadata and no profile-to-API-id mapping exists.

## Solution Assessment

Unclear. The gap is real, but provider/name is not proven better than an explicit apiModelId, routeTo, or another model-profile contract that preserves display names and upgrade semantics.
