## Summary

Keep open. Current main still treats selectable model refs as `provider/model id`, `name` remains catalog/display metadata, and the related derived-model routing work is still open with unresolved contract questions.

## What This Changes

Request to use `provider/name` as OpenClaw's internal model key while reserving `id` for the provider-facing API model id and allowing model-entry reasoning levels.

## Best Possible Solution

Design and land a documented model-profile routing contract that separates selectable OpenClaw refs from provider API ids and aligns schema, docs, pickers, params lookup, session persistence, runtime resolution, and migration behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an architectural feature proposal. Source inspection does confirm current main still lacks the requested selectable-profile/API-id separation.

## Solution Assessment

Unclear. Decoupling selectable profile keys from provider API ids is a real need, but using `name` as the internal key needs maintainer product/API judgment because `name` is currently display/catalog metadata and an explicit route/profile field may be safer.
