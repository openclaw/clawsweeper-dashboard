## What This Changes

## Best Possible Solution

Keep this as the canonical maintainer design issue until OpenClaw chooses one model-profile contract and implements it consistently across schema, runtime routing, UI/session state, docs, and doctor migration.

## Reproduction Assessment

Yes, at source level: current main still builds catalog and runtime lookups around provider plus `id`, while `name` remains display metadata. I did not run a live provider config because this read-only review can verify the architectural gap from source and docs.

## Solution Assessment

Unclear as the final solution: `provider/name` would solve the duplicate API-id profile gap, but it repurposes display metadata and needs maintainer approval versus an explicit profile or routed API-id contract.
