## Summary

Keep open. Current main still treats the selectable model reference as provider plus model `id`; `name` remains display metadata, and duplicate configured model IDs under one provider collapse in catalog, UI, and runtime lookup paths. The related open context-profile and generation-parameter issues overlap with the motivation but do not supersede this broader model-profile key proposal.

## What This Changes

The issue proposes using `provider/name` as OpenClaw’s internal model key so one upstream API model ID can have multiple local profiles while preserving `id` as the API model identifier.

## Best Possible Solution

Design a backward-compatible model profile identity contract that separates local profile key from upstream API model id, migrates existing `provider/model` references safely, and updates selection, catalog, session persistence, UI, docs, and provider-runtime lookup together with regression coverage for duplicate same-provider API ids.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure two `models.providers.<provider>.models[]` rows with the same `id` and different `name` or params, then inspect configured model view or runtime lookup: current main keys and dedupes by `provider/id`, so the profiles cannot remain distinct.

## Solution Assessment

Unclear. The requested `provider/name` direction is plausible, but maintainers should compare it against an explicit profile `key` or `apiModelId` because the change affects public config and persisted state.
