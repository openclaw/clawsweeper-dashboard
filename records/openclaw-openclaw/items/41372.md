## Summary

Keep this open. Current main has improved several config-validation recovery surfaces, but the field report still has concrete remaining cron cleanup: the automation docs still describe plain `openclaw cron list` as listing all jobs while the CLI/service default to enabled-only, the JSON wrapper shapes for cron scripting are still not described in the cron docs, and `cron add` remains create-only rather than an idempotent setup path.

## What This Changes

The issue requests upstream fixes or follow-up triage for a bundled production field report covering config validation crash loops, Discord receive failures, cron CLI/docs mismatches, isolated web-search failures, onboarding token changes, resource exhaustion, and update-process handling.

## Best Possible Solution

Keep the issue open until the remaining cron operator-facing gaps are resolved or split: update the cron docs to match enabled-only list behavior and document the JSON wrapper contracts, then make an explicit product call on whether cron needs an upsert/idempotent setup command or a documented safe setup pattern. Any still-current live Discord, isolated web-search, onboarding-token, update-process, and small-VPS resource findings should be moved into focused follow-ups with fresh reproduction evidence.

## Reproduction Assessment

Yes for the remaining cron docs mismatch: current main can be checked by comparing the docs text with the CLI `--all` flag and the service's enabled-only default. Unclear for the live Discord, Brave search, onboarding, update, and VPS resource sections because this read-only pass did not exercise those environments.

## Solution Assessment

No, not as a single umbrella. The report is valuable field evidence, but the maintainable path is focused cron documentation/product cleanup plus separate validated follow-ups for the live runtime incidents rather than closing or fixing all 25 findings as one unit.
