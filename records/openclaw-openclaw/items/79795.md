## Summary

Keep open: current main still has a source-level path where the failed iMessage probe can be removed from non-sensitive health output, and an open issue-closing PR is already active, so this should not be closed before that PR lands.

## What This Changes

The issue requests that iMessage health/status report a degraded or error state when `imsg rpc` cannot access the Messages database due to Full Disk Access denial.

## Best Possible Solution

Land a coordinated fix that keeps raw probe details redacted but exposes a canonical iMessage Full Disk Access failure in health/status; use https://github.com/openclaw/openclaw/pull/80068 as the issue-closing PR and coordinate the lower-level imsg banner handling with https://github.com/openclaw/openclaw/pull/80403.

## Reproduction Assessment

Yes, at source level: with iMessage configured and the `imsg rpc` probe failing, the non-sensitive health path can drop the failed probe before formatting, which lets status fall back to `configured`. I did not live-run the macOS FDA-denied path.

## Solution Assessment
