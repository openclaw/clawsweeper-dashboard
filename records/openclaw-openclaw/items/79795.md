## Summary

Keep open: current main still has a source-level path where an iMessage `imsg rpc` probe failure can be removed from non-sensitive health output, allowing `status --deep` to render configured/OK while the transport is unavailable.

## What This Changes

Report that iMessage health/status should surface a degraded/error state when `imsg rpc` fails because Full Disk Access blocks the Messages database.

## Best Possible Solution

Carry a redacted iMessage probe/readiness failure into both Health and Channels status output, with regression tests proving non-sensitive `status --deep` no longer falls back to OK/configured for a failed `imsg rpc` probe.

## Reproduction Assessment

Yes, source-level: configure iMessage so `imsg rpc`/`chats.list` fails with the reported Full Disk Access error, then run `openclaw status --deep`; current health formatting can drop the failed probe before status rendering. I did not live-run the macOS FDA path in this review.

## Solution Assessment

Yes, the narrow maintainable fix is to propagate a redacted failed probe/readiness state into channel health/status instead of treating configured state as OK. It should not add a config option or expose raw probe payloads.
