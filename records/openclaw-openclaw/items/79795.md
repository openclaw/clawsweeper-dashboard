## Summary

Keep open: current main still has a source-level path where a failed iMessage `imsg rpc` probe can be hidden from non-sensitive health/status output, so the user can still see configured/OK while the transport is unavailable.

## What This Changes

The issue requests iMessage health/status to report degraded/error when `imsg rpc` cannot start because macOS Full Disk Access is denied.

## Best Possible Solution

Surface a redacted iMessage probe/readiness failure in both `status --deep` Health and Channels output, while keeping the broader macOS TCC process-identity work in the existing separate issues.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: make `probeIMessage` return `ok:false` for the reported Full Disk Access error and run the non-sensitive `status --deep` health path; the failed probe can be removed before formatting falls back to configured/OK. I did not live-run the macOS TCC path.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to carry a redacted failed probe/readiness state into status/health output, not to add a config option or solve the broader Full Disk Access process-boundary problem here.
