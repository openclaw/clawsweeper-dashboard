## Summary

Keep open: current main still has a source-reproducible path where iMessage `imsg rpc` can fail while non-sensitive health/status output falls back to configured/OK, and the only related thread is a closed duplicate pointing back here.

## What This Changes

The issue asks for iMessage channel health/status to report degraded/error when `imsg rpc` cannot start because Full Disk Access is denied.

## Best Possible Solution

Carry a redacted iMessage probe/readiness failure into the existing health and channel status summaries so `status --deep` reports WARN/failed instead of configured/OK without exposing raw probe payloads.

Yes, at source level: when `probeIMessage` returns `ok:false` for the Full Disk Access failure, current non-sensitive health output can remove the failed probe before formatting and fall back to `configured`. I did not run a live macOS Full Disk Access denial repro.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: propagate a safe failed-probe/readiness status through existing status/health summaries instead of adding a config option or exposing raw probe details.
