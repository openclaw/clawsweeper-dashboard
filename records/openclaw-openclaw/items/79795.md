## Summary

Keep open: current main still has a source-level path where the iMessage live probe can fail but the non-sensitive status/health formatter drops the failed probe payload and reports the configured channel as OK.

## What This Changes

The issue asks status/health output to show degraded/error when iMessage `imsg rpc` fails from missing Full Disk Access instead of reporting OK/configured.

## Best Possible Solution

Propagate a redacted iMessage probe/readiness failure into `status --deep` Health and Channels output, with regression coverage for non-sensitive snapshots and status formatting.

## Reproduction Assessment

Yes, at source level: configure iMessage so `imsg rpc`/`chats.list` fails with the reported Full Disk Access error, then run `openclaw status --deep`; current main can run the probe but remove the failed probe payload before formatting. I did not live-run this on macOS.

## Solution Assessment

Yes, the narrow maintainable fix is to carry a redacted failed probe/readiness state into channel health/status instead of treating configured state as OK. It should not add a config option or expose raw probe payloads.
