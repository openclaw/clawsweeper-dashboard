## Summary

Keep open: current main still has a clear source-level path where a failed iMessage `imsg rpc` probe can be removed from non-admin health output, leaving status to report `configured`/OK while the transport is unavailable.

## What This Changes

The issue asks for iMessage health/status to report degraded or error, with actionable Full Disk Access remediation, when `imsg rpc` cannot access the Messages database.

## Best Possible Solution

Surface a redacted iMessage probe/readiness failure in both `status --deep` Health and Channels output, with regression coverage that proves non-admin status no longer reports configured/OK for a failed `imsg rpc` readiness probe.

## Reproduction Assessment

Yes, at source level: make `probeIMessage` return `ok:false` for the Full Disk Access `chats.list` failure, then request non-admin `health`/`status --deep`; the failed probe can be redacted before formatting. I did not live-run this against macOS Messages permissions.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: carry a redacted failed probe/readiness state into status and health instead of treating configured state as healthy. The implementation should avoid exposing raw probe payloads or adding a new config option.
