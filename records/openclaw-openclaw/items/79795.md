## Summary

Keep open: current main still has a source-level path where a failed iMessage `imsg rpc` probe is removed from non-sensitive health output, allowing `status --deep` to present a configured account as OK.

## What This Changes

The issue asks for iMessage status and health to report degraded/error, with concise Full Disk Access remediation, when `imsg rpc` cannot read the Messages database.

## Best Possible Solution

Carry a redacted probe failure signal into non-sensitive health and status rows so configured iMessage accounts with failed `imsg rpc` read probes show WARN/error with Full Disk Access guidance.

## Reproduction Assessment

Yes, at source level: make the iMessage `chats.list` probe fail with the reported Full Disk Access error, then request non-sensitive `status --deep` health output. I did not live-run the macOS permission-denied setup.

## Solution Assessment

Yes, the narrow maintainable fix is to propagate a redacted failed-probe/readiness state into channel health/status instead of treating configured state as OK. The implementation should avoid exposing raw probe payloads or adding a config option.
