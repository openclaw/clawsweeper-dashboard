## Summary

Keep open: current main still has a plausible source-level path where a failing iMessage `imsg rpc` probe is hidden from non-sensitive health output, letting status fall back to configured/OK despite the transport being unavailable.

## What This Changes

Report iMessage as degraded/error when `imsg rpc` cannot start because macOS denies Full Disk Access to the Messages database.

## Best Possible Solution

Surface a redacted iMessage probe/readiness failure in both `status --deep` Health and Channels output, with regression coverage for non-sensitive health snapshots.

## Reproduction Assessment

Yes, at source level: configure iMessage so `imsg rpc`/`chats.list` fails with the reported Full Disk Access error, then run `openclaw status --deep`; the health path can run the probe but remove the failed probe payload before formatting. I did not live-run this on macOS.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: propagate a redacted failed probe/readiness state into channel health/status instead of treating configured state as OK. A safer implementation should avoid exposing raw probe payloads or adding a config option.
