## Summary

Keep open. Current main still has the reported two-delivery shape: ACP session-mode spawns deliver inline to the bound thread, while the `sessions_spawn` ACP wrapper still registers the same run with completion announce delivery enabled. The related closed report #55568 covered a different duplicate path, and the adjacent open PR #64416 does not clearly suppress this registry announce path.

## What This Changes

Thread-bound Discord ACP `sessions_spawn` sessions should deliver each ACP response once instead of posting both an inline thread reply and a completion announce copy.

## Best Possible Solution

Make ACP thread-bound session mode match native thread-bound session behavior: keep inline delivery as the sole user-visible response path, keep lifecycle tracking for state and cleanup, and suppress completion announce delivery for inline-bound ACP session runs. Run-mode ACP completion delivery and `streamTo: "parent"` behavior should remain unchanged.

## Reproduction Assessment

Yes. The issue has concrete Discord/ACP configuration and steps, and current source inspection shows a high-confidence reproduction path: `mode: "session"` enables inline delivery while the wrapper still tracks the run with completion announce enabled.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the native subagent guard by registering inline-bound ACP session runs with `expectsCompletionMessage: false` or an equivalent announce-suppression signal, instead of relying on prompt wording or broad delivery idempotency.
