## Summary

Keep open. The PR addresses a real current-main bug: the shared dispatch suppress flag is currently bypassed when verbose progress is enabled, which matches the linked Telegram leak. The proposed change is narrow, aligns with the option contract, and I found no blocking functional or security findings.

## What This Changes

The PR makes `suppressDefaultToolProgressMessages` authoritative in shared reply dispatch and updates two dispatch tests so verbose mode no longer overrides suppression.

## Best Possible Solution

Land the focused shared-dispatch fix with the updated regression tests, preserving channel-owned progress callbacks while suppressing default progress bubbles whenever the flag is set.

## Reproduction Assessment

Yes. A high-confidence source reproduction is dispatching with `verboseLevel: "on"` and `replyOptions.suppressDefaultToolProgressMessages: true`; current-main tests expect the leaked `sendToolResult` calls in that exact setup.

## Solution Assessment

Yes. Making the suppress flag unconditional is the narrowest maintainable fix because it matches the option contract and leaves media, error, and exec-approval exceptions in the existing delivery path.
