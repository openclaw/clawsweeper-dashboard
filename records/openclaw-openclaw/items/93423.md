## Summary

Keep open. Current main and v2026.6.8 still forward exec approval prompts without a hold/retract path, but the requested fix is a new approval lifecycle, config, and agent-facing API surface that needs maintainer design rather than cleanup closure.

## What This Changes

Requests an opt-in `approvals.exec` hold window plus an agent-facing retraction action so self-abandoned exec approvals can be canceled before forwarded chat delivery.

## Best Possible Solution

Design a narrow opt-in core approval lifecycle that buffers only forwarded delivery, exposes a typed retract-as-deny path, logs held/retracted requests, and proves default instant forwarding remains unchanged.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug repro, but source inspection confirms the current path has no `holdMs`, no retract RPC, and immediate forwarded prompt delivery.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the approval API and safety model; the safest direction is an opt-in hold that preserves current defaults and proves fail-open-to-human/fail-closed-to-execution behavior.
