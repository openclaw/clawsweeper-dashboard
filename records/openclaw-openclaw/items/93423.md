## Summary

Keep open: current main and v2026.6.8 still support immediate exec approval forwarding with request/list/get/wait/resolve only; the requested hold-and-retract window is a new opt-in approval feature that needs maintainer API, config, and security design.

## What This Changes

The issue requests a configurable approvals.exec hold window plus an agent-accessible retract action for pending forwarded exec approvals.

## Best Possible Solution

Design this in the core exec approval manager/forwarder as an opt-in config plus explicit deny-style retract API, with tests for holdMs=0 compatibility, delivery failure fail-open behavior, execution fail-closed behavior, and audit/log visibility.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs do confirm the current behavior: forwarded exec approvals are delivered through the existing request/resolve flow with no hold or retract surface.

## Solution Assessment

Unclear pending maintainer design. The requested shape is plausible and likely belongs in the core exec approval pipeline, but the permanent config name, gateway method, agent access rules, and audit semantics need product/security agreement first.
