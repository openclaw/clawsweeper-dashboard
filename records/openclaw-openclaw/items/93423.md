## Summary

Keep open: current main and v2026.6.6 still deliver forwarded exec approval prompts immediately and expose no `holdMs` config or retract RPC. The request is useful, but it is a security-sensitive config/protocol feature that needs maintainer API design rather than cleanup closure.

## What This Changes

Request an opt-in `approvals.exec.holdMs` window plus an agent-visible approval retraction action so forwarded exec approval prompts can be buffered and cancelled as denials before human delivery.

## Best Possible Solution

Design a core opt-in hold/retract approval lifecycle with default `holdMs: 0`, additive gateway protocol/config changes, audit logging, and tests for fail-open delivery, retract-as-deny, expiry, and Codex-facing behavior.

## Reproduction Assessment

Not applicable as a feature request; source inspection confirms current forwarded exec approval delivery is immediate and no `holdMs` or `exec.approval.retract` path exists.

## Solution Assessment

Unclear until maintainer design is confirmed: the core approval pipeline is the likely place for this because plugins cannot retract a prompt after the approval is created, but the permanent config and protocol semantics need owner review.
