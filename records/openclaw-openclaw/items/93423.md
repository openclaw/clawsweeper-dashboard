## Summary

Keep open: current main still forwards pending exec approvals immediately and exposes request/wait/resolve only; the requested hold/retract lifecycle is absent and needs maintainer product/API review because it changes command-approval safety and Codex/app-server semantics.

## What This Changes

Request a default-off `approvals.exec.holdMs` window plus agent-accessible retraction so forwarded exec approvals can be cancelled before human delivery.

## Best Possible Solution

Define a default-off approval lifecycle contract with bounded hold timing, explicit retract-as-deny semantics, fail-open delivery recovery, audit logging, and focused Gateway/agent/channel/Codex tests before implementation.

## Reproduction Assessment

No failing-bug repro is applicable because this is a feature request. Source inspection gives a high-confidence current-main check: config, Gateway protocol, manager, forwarder, and agent paths have request/wait/resolve but no `holdMs` or retract operation.

## Solution Assessment

Unclear until maintainers choose the API. The proposed default-off hold window is plausible, but the best solution must first define safety, audit, and Codex/app-server semantics across the approval lifecycle.
