## Summary

Keep open: the fix is aimed at the right completion-state boundary and includes usable runtime proof, but the new progress-only classifier can turn a valid final ACP completion into a blocked task whenever the captured text starts with an interim phrase before the real result.

## What This Changes

The PR adds a shared required-completion contract for ACP/subagent finalization and tests that missing, progress-only, and delivery-exhausted completions become blocked while real final reports remain successful.

## Best Possible Solution

Land the required-completion contract after tightening progress-only detection so missing or pure-progress required completions and exhausted delivery become blocked, while any captured text that includes a real final result remains successful.

## Reproduction Assessment

Yes: source inspection shows current main finalizes parented ACP/subagent successes without a required-final-deliverable check, and the PR body supplies after-fix runtime output for the target scenarios. I did not execute tests because this review was read-only.

## Solution Assessment

No: the owner boundary is right, but the classifier should not block text that starts with progress and later includes a final deliverable. The safer fix is a narrow helper plus regressions for both pure progress and progress-then-final output.
