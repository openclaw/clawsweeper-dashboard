## Summary

Keep this issue open. Current main still has read-only agent avatar resolution/serving and browser-local avatar overrides, but it does not have the requested Agents-tab upload flow, `agents.avatar.upload` RPC, `POST /avatar/<agentId>` upload route, or binary workspace avatar write path. The closed, unmerged #11603 and linked gist remain useful design context rather than shipped behavior.

## What This Changes

The issue requests a Control UI Agents-tab flow that lets an operator pick an agent avatar image, saves it under the agent workspace, updates identity metadata, and renders it through the avatar route.

## Best Possible Solution

Add a maintainer-approved, additive admin-scoped avatar upload path with typed protocol/schema coverage, strict image type and size validation, safe filename normalization, symlink-safe bounded writes under the selected agent workspace, identity updates through existing helpers, Agents-tab picker/refresh behavior, docs, and focused gateway/UI regression tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The negative check is high-confidence from current-main source inspection: there is no upload RPC, no POST avatar route, and no Agents-tab file picker for agent workspace avatars.

## Solution Assessment

Unclear whether the exact `agents.avatar.upload` shape in the issue is the best final API. The user-facing direction is valid, but maintainers should choose the endpoint contract, auth scope, upload validation rules, and identity persistence model before implementation.
