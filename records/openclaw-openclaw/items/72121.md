## Summary

Keep open. Current main still rejects `sk-ant-api03-` input on the Anthropic `paste-token` path before any auth profile is stored, while the Anthropic plugin already exposes a separate API-key auth method and the focused related PR was closed unmerged.

## What This Changes

The issue asks for Anthropic `paste-token` setup to accept standard `sk-ant-api03-` API keys or clearly direct users to the supported API-key auth path.

## Best Possible Solution

Keep Anthropic setup-token and API-key semantics distinct: either detect `sk-ant-api03-` input in `paste-token` and route it into the existing API-key profile/config path, or reject it with a targeted message naming `openclaw models auth login --provider anthropic --method api-key` as the supported API-key setup command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: run `openclaw models auth paste-token --provider anthropic`, paste a value beginning with `sk-ant-api03-`, and the current Anthropic branch calls `validateAnthropicSetupToken`, which rejects anything not starting with `sk-ant-oat01-`.

## Solution Assessment

No. The current implementation is not the best solution because it rejects a standard Anthropic API-key prefix without actionable API-key guidance; widening the setup-token validator alone is also not sufficient unless the credential is stored and configured as API-key auth.
