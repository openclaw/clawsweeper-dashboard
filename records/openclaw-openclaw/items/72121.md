## Summary

Keep open. Current main still rejects `sk-ant-api03-` values in the Anthropic `paste-token` path, the Anthropic plugin already has a separate API-key auth method, and the focused related PR is closed unmerged.

## What This Changes

Accept or clearly redirect standard Anthropic `sk-ant-api03-` API keys when users paste them through the Anthropic `models auth paste-token` CLI path.

## Best Possible Solution

Keep setup-token and API-key semantics distinct by routing `sk-ant-api03-` input to the existing Anthropic API-key credential/config path or by returning a targeted API-key command instead of storing API keys as setup tokens.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sk-ant-api03-` input in `openclaw models auth paste-token --provider anthropic` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and exits before profile storage.

## Solution Assessment

No. The current behavior is not the best solution because it rejects standard Anthropic API-key input without actionable API-key routing; the safer fix is explicit API-key handling or a targeted redirection message, not a broad setup-token validator change.

## Repair Work Prompt

Fix the Anthropic paste-token API-key rejection tracked by https://github.com/openclaw/openclaw/issues/72121. Current main sends `openclaw models auth paste-token --provider anthropic` input through `validateAnthropicSetupToken`, so `sk-ant-api03-` API keys are rejected before storage. Keep setup-token and API-key semantics distinct: route normalized `sk-ant-api03-` input to the existing Anthropic API-key credential/config path (`credential.type: "api_key"`, config mode `api_key`) or, if accepting API keys in `paste-token` is not desired, return an actionable error that names the API-key command (`openclaw models auth login --provider anthropic --method api-key --set-default` or `openclaw onboard --anthropic-api-key`). Do not solve this by simply widening `validateAnthropicSetupToken` so API keys are stored as token profiles. Use closed PR #72124 only as source context, not as the final design. Add focused regression coverage for `sk-ant-api03-` and preserve the existing `sk-ant-oat01-` setup-token path. Add a user-facing changelog entry if behavior changes.
