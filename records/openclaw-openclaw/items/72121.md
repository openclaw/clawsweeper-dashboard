## Summary

Keep open. Current main still routes Anthropic `paste-token` input through the setup-token validator, which rejects `sk-ant-api03-` API keys before storage, while the existing Anthropic plugin has a separate API-key auth method and the focused fix PR was closed unmerged.

## What This Changes

The issue requests that Anthropic CLI auth accept standard `sk-ant-api03-` API keys or clearly route users to the correct API-key path instead of rejecting them as invalid setup tokens.

## Best Possible Solution

Anthropic auth should keep setup-token and API-key semantics distinct: `sk-ant-oat01-` remains the token/setup-token path, while `sk-ant-api03-` is either saved through the existing API-key profile contract or rejected with a targeted message that points to the API-key command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: any non-empty Anthropic pasted value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No for the current implementation. The narrow maintainable fix should reuse the existing Anthropic API-key credential path or provide explicit API-key guidance, rather than widening token-mode storage in a way that mixes credential types.
