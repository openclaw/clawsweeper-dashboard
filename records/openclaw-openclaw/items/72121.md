## Summary

Keep open. Current main still sends Anthropic `paste-token` input through the setup-token-only validator, then persists accepted values as token-mode credentials, while the Anthropic plugin already exposes a separate API-key method for `ANTHROPIC_API_KEY`; the focused related PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to stop rejecting standard Anthropic `sk-ant-api03-` API keys in the `openclaw models auth paste-token --provider anthropic` flow, or to direct users to the separate API-key auth path.

## Best Possible Solution

Keep setup-token and API-key semantics distinct: `sk-ant-oat01-` should remain token-mode setup-token input, while `sk-ant-api03-` should either be routed into the existing Anthropic `api_key` profile/config path or rejected with an actionable message naming `openclaw models auth login --provider anthropic --method api-key`.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: an Anthropic paste-token value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before storage.

## Solution Assessment

Yes, a narrow CLI/auth repair is the best path. The safer implementation is explicit API-key routing or targeted guidance, not only widening the setup-token validator because the current command writes token-mode credentials.
