## Summary

Keep open. Current main still sends Anthropic `paste-token` input through an `sk-ant-oat01-`-only setup-token validator, while the repository already has a separate Anthropic API-key auth method and the focused fix PR was closed unmerged.

## What This Changes

The issue asks for `openclaw models auth paste-token --provider anthropic` to stop rejecting standard Anthropic `sk-ant-api03-` API keys as invalid setup tokens, or to route users clearly to the API-key auth path.

## Best Possible Solution

Anthropic auth should keep setup-token and API-key semantics distinct: `sk-ant-oat01-` should remain the token/setup-token path, while `sk-ant-api03-` should either be persisted through the existing API-key profile contract or rejected with a precise message pointing to the API-key command instead of implying the key is invalid.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: any non-empty Anthropic pasted value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. The current behavior is not the best way to solve the issue because OpenClaw already has a distinct Anthropic API-key path; a safe fix should route or message API keys explicitly rather than broadly treating them as setup tokens.
