## Summary

Keep open. Current main still routes Anthropic `paste-token` input through a setup-token-only validator, while the Anthropic plugin already exposes a separate API-key auth method; the related fix PR #72124 is closed unmerged and did not land on main.

## What This Changes

The issue asks OpenClaw to handle standard Anthropic `sk-ant-api03-` API keys correctly when users try the CLI Anthropic auth paste path.

## Best Possible Solution

Detect Anthropic API-key input in this flow and route it through the existing API-key profile/config writer, or return a precise API-key setup command, while keeping `sk-ant-oat01-` on the token path.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before persistence.

## Solution Assessment

No. Current main is not the best solution because it rejects valid API-key input, and the safe repair should preserve API-key versus setup-token auth semantics instead of treating both as token credentials.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72121. Observable bug: `openclaw models auth paste-token --provider anthropic --profile-id anthropic:default` rejects a standard Anthropic `sk-ant-api03-...` API key with `Expected token starting with sk-ant-oat01-`. Current main routes Anthropic paste-token through `validateAnthropicSetupToken` and then persists all successful input as token credentials. Implement a narrow fix that preserves auth modes: keep `sk-ant-oat01-` as the setup-token/token path; for `sk-ant-api03-`, either write an Anthropic `api_key` credential and config auth profile through the existing API-key helpers or return an actionable API-key login/onboard command if paste-token is intentionally token-only. Do not store API keys as token credentials. Use closed PR #72124 only as prior context; avoid a validator-only fix. Add regression coverage, update relevant docs/help if user-facing behavior changes, and add a changelog fix entry.
