## Summary

Keep open. Current main still sends Anthropic `paste-token` input through the setup-token-only validator, so a standard `sk-ant-api03-` API key still fails before any profile is stored; the related focused PR is closed unmerged.

## What This Changes

The issue asks for Anthropic CLI auth to accept standard `sk-ant-api03-` API keys through the reported setup path, or to clearly route those keys to a separate API-key auth path.

## Best Possible Solution

Keep Anthropic setup-token and API-key contracts distinct: `sk-ant-oat01-` should remain the setup-token path, while `sk-ant-api03-` should either be saved through the existing API-key profile contract or rejected with a targeted message pointing to the API-key auth command, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: an Anthropic pasted value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. Current main is not the best solution because it gives a setup-token-only validation error for API-key material; the safer path is an explicit API-key branch or precise API-key command guidance without weakening token-mode semantics.
