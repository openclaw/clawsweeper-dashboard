## Summary

Keep open. Current main still rejects `sk-ant-api03-` input on the Anthropic `paste-token` path by sending it through a setup-token-only validator, while the provider already has a separate API-key auth method and the focused fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to stop reporting standard Anthropic `sk-ant-api03-` API keys as invalid setup tokens when using `openclaw models auth paste-token --provider anthropic`, either by accepting them correctly or by directing users to the API-key auth path.

## Best Possible Solution

Keep the Anthropic credential families explicit: `sk-ant-oat01-` should remain the setup-token/token path, while `sk-ant-api03-` should either be stored through the existing API-key profile contract or rejected with a precise message that points to the Anthropic API-key setup command instead of saying the key is invalid.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `openclaw models auth paste-token --provider anthropic` passes any pasted `sk-ant-api03-...` value into `validateAnthropicSetupToken`, which fails the `sk-ant-oat01-` prefix check before storage.

## Solution Assessment

No. The current behavior is not the best solution because it treats an API key as an invalid setup token even though OpenClaw already exposes a distinct Anthropic API-key auth method; the safer fix is to route or message API keys explicitly.

Current main rejects `sk-ant-api03-...` values in `openclaw models auth paste-token --provider anthropic` because the command calls the setup-token-only `validateAnthropicSetupToken` helper. Use PR #72124 only as source context; avoid broadly widening the setup-token validator in a way that blurs runtime auth modes.
