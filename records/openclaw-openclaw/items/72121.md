## Summary

Keep open. Current main still routes Anthropic `models auth paste-token` through an `sk-ant-oat01-` setup-token validator, while the codebase already has a distinct Anthropic API-key auth method and tests that treat `sk-ant-api03-` as regular API-key material. The related focused PR was closed unmerged, so the reported workflow remains valid and narrow.

## What This Changes

The issue asks for `openclaw models auth paste-token --provider anthropic` to accept standard `sk-ant-api03-` Anthropic API keys or clearly route users to the separate API-key auth path.

## Best Possible Solution

Anthropic auth should keep setup-token and API-key semantics distinct: `sk-ant-oat01-` should continue through the setup-token/token path, while `sk-ant-api03-` input should either be stored through the existing API-key profile contract or rejected with a direct, actionable API-key command message.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an Anthropic pasted value starting with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. The current behavior is not the best way to solve the issue because OpenClaw already has a separate Anthropic API-key path; the safer fix is an API-key-aware branch or explicit API-key guidance, not treating API keys as setup tokens.

## Repair Work Prompt

Fix the Anthropic `models auth paste-token --provider anthropic` behavior from this issue. Current main rejects standard `sk-ant-api03-` API keys with `Expected token starting with sk-ant-oat01-` because the command always calls `validateAnthropicSetupToken` and then stores Anthropic pasted credentials as token profiles. Preferred fix: detect Anthropic `sk-ant-api03-` input in the paste-token path and store it through the existing API-key credential/config contract (`credential.type: "api_key"`, config mode `"api_key"`) or otherwise emit a direct API-key command message if maintainers choose strict command separation. Add focused tests for `sk-ant-api03-` and `sk-ant-oat01-`, update docs/help if the user-facing command behavior changes, and add a single-line CHANGELOG fix entry crediting the reporter and prior focused PR author where appropriate. Do not alter unrelated provider auth behavior or relax SecretRef/OAuth semantics.
