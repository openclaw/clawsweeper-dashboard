## Summary

Keep open. Current main still rejects `sk-ant-api03-` values in the Anthropic `paste-token` path, while the Anthropic plugin already exposes a separate API-key auth method and the only focused related PR is closed unmerged.

## What This Changes

The issue asks the Anthropic CLI auth flow to accept standard `sk-ant-api03-` API keys or guide users to the API-key auth path instead of rejecting them as invalid setup tokens.

## Best Possible Solution

Keep setup-token and API-key semantics distinct: detect `sk-ant-api03-` input in the Anthropic paste-token flow and route it to the existing API-key profile/config path, or reject it with an actionable command that names the API-key auth method.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a `sk-ant-api03-` value entered through `openclaw models auth paste-token --provider anthropic` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before any profile is written.

## Solution Assessment

No. The current behavior is not the best solution because it rejects standard Anthropic API-key input without clear API-key guidance; the safer fix is explicit API-key routing or a targeted redirection message, not a broad setup-token validator change.
