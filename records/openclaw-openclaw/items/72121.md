## Summary

Keep open. Current main still routes Anthropic `paste-token` input through the setup-token-only `sk-ant-oat01-` validator, while the repo already has a separate Anthropic API-key auth method and the related focused PR was closed unmerged.

## What This Changes

The issue requests that Anthropic `paste-token` accept standard `sk-ant-api03-` API keys or clearly route users to the existing API-key auth path instead of reporting the key as an invalid setup token.

## Best Possible Solution

Anthropic `paste-token` should keep credential modes honest: detect `sk-ant-api03-` as API-key material and persist it through the existing `api_key` profile/config contract, or return a precise message pointing to `openclaw models auth login --provider anthropic` / `openclaw onboard --anthropic-api-key` rather than saying the key itself is invalid.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: any Anthropic pasted value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

Yes, the maintainable fix is narrow: handle API-key-shaped Anthropic input through the existing API-key contract or explicit guidance, not by storing API keys as token-mode setup tokens. The closed PR's broad validator-widening approach is useful context but not the safest final shape.
