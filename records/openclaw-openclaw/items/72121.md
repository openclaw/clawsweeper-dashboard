## Summary

Keep open. Current main still sends Anthropic `paste-token` input through the setup-token validator, so `sk-ant-api03-` API keys are rejected before storage; the existing Anthropic plugin already has a separate API-key auth method, and the focused related PR was closed unmerged.

## What This Changes

The issue requests that Anthropic `sk-ant-api03-` API keys either be accepted by the CLI auth setup path or receive clear guidance to use the separate API-key auth flow.

## Best Possible Solution

Keep setup-token and API-key semantics distinct: either route `sk-ant-api03-` paste-token input into the existing Anthropic API-key profile/config path, or reject it with a targeted message that names `openclaw models auth login --provider anthropic --method api-key` as the supported API-key setup command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: an Anthropic paste-token value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before any profile is written.

## Solution Assessment

No. The current implementation is not the best solution because it rejects a standard Anthropic API key without actionable API-key guidance, while widening the setup-token validator alone would blur credential types; the safer fix is to reuse the existing API-key auth contract or provide explicit API-key redirection.
