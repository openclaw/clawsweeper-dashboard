## Summary

Keep open. Current main still routes Anthropic `paste-token` input through a setup-token-only validator, while the Anthropic plugin already has a separate API-key auth method that should not be bypassed by storing API keys as token credentials.

## What This Changes

The issue asks OpenClaw to accept standard Anthropic `sk-ant-api03-` API keys in the Anthropic CLI auth flow, or clearly direct users to the separate API-key path.

## Best Possible Solution

Make the Anthropic paste-token path API-key-aware without collapsing auth modes: keep `sk-ant-oat01-` as token auth, and either route `sk-ant-api03-` through the existing API-key profile/config writer or return an actionable API-key login command.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before persistence.

## Solution Assessment

No. Simply accepting both prefixes in the setup-token validator is not the safest fix because the caller currently writes token credentials; the maintainable fix should preserve the dedicated API-key auth path or clearly redirect to it.
