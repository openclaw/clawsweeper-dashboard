## Summary

Keep open. Current main still routes Anthropic paste-token input through the setup-token prefix validator and then persists only token-mode credentials, while the Anthropic plugin already exposes a separate API-key auth method for standard `sk-ant-api03-` keys; the focused related PR was closed unmerged.

## What This Changes

Allow or clearly redirect standard Anthropic `sk-ant-api03-...` API keys when users invoke `openclaw models auth paste-token --provider anthropic`.

## Best Possible Solution

Keep setup-token and API-key semantics distinct: detect `sk-ant-api03-` input in the Anthropic paste-token flow and either hand it to the existing API-key profile/config path or reject it with an actionable message naming `openclaw models auth login --provider anthropic --method api-key`. Cover the chosen behavior with focused CLI/auth tests and a user-facing changelog entry if the CLI copy changes.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an Anthropic paste-token value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before any profile is written.

## Solution Assessment

No. The current behavior is not the best solution because it rejects a standard Anthropic API key without clear API-key guidance, and simply widening the setup-token validator would blur credential types; the safer fix is explicit API-key routing or targeted redirection.
