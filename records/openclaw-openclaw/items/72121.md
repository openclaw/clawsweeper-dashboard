## Summary

Keep open. Current main still sends Anthropic `paste-token` input through the setup-token validator, so `sk-ant-api03-` API keys hit the same `sk-ant-oat01-` rejection before any credential can be stored.

## What This Changes

The issue requests that Anthropic `paste-token` either handle `sk-ant-api03-` API keys or clearly point users to the separate API-key auth path.

## Best Possible Solution

Keep API-key material on the existing `api_key` credential path, and make `paste-token` either route `sk-ant-api03-` through that path or reject it with the exact API-key auth command before any token profile write.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects `sk-ant-api03-`; I did not run the interactive CLI in this read-only review.

## Solution Assessment

Yes for keeping this open, but no to applying the closed validator-only PR unchanged. The safer repair is to preserve token semantics and use the existing API-key auth path or an actionable API-key redirect.
