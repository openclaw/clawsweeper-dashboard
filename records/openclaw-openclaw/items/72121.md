## Summary

Keep open. Current main still rejects `sk-ant-api03-` in the Anthropic `paste-token` path, and the existing API-key auth method is separate from the setup-token storage path.

## What This Changes

Request Anthropic `models auth paste-token` to accept standard `sk-ant-api03` API keys or clearly route users to the separate API-key auth path.

## Best Possible Solution

Make the Anthropic paste-token flow prefix-aware while preserving auth modes: keep `sk-ant-oat01-` on token auth and route `sk-ant-api03-` through API-key storage/config or an explicit API-key login instruction before persistence.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before any profile is written.

## Solution Assessment

No. Current main still uses setup-token-only validation, and the narrow maintainable fix must not collapse Anthropic API keys into token credentials.

## Repair Work Prompt

Fix openclaw/openclaw#72121. Current main routes `openclaw models auth paste-token --provider anthropic` through `validateAnthropicSetupToken`, which rejects standard `sk-ant-api03-` Anthropic API keys, then writes token credentials/config mode `token`. Implement a narrow Anthropic paste-token fix that preserves `sk-ant-oat01-` setup-token behavior and handles `sk-ant-api03-` with API-key semantics: either store it as an `api_key` credential with config mode `api_key` using existing auth helpers, or, if that path is not maintainable, fail before persistence with a direct instruction to use `openclaw models auth login --provider anthropic --method api-key --set-default`. Do not merely broaden the setup-token validator and store API keys as token credentials. Add regression tests for both prefixes, update user-facing docs/help as needed, and add a single-line CHANGELOG fix entry.
