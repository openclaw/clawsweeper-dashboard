## Summary

Keep open. Current main still sends Anthropic `paste-token` input through the setup-token validator, so `sk-ant-api03-` keys hit the same `sk-ant-oat01-` rejection before any credential is persisted.

## What This Changes

The issue asks for Anthropic `paste-token` to accept standard `sk-ant-api03-` API keys or clearly direct users to the separate API-key auth path.

## Best Possible Solution

Preserve `sk-ant-oat01-` setup-token behavior while handling `sk-ant-api03-` through `api_key` credentials/config or an explicit actionable redirect to Anthropic API-key auth.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: `openclaw models auth paste-token --provider anthropic` passes Anthropic input to `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before persistence.

## Solution Assessment

No for the closed validator-only approach. The maintainable fix should keep setup tokens in token mode and route standard API keys through the existing API-key auth/profile semantics, or clearly tell users which API-key command to run.
