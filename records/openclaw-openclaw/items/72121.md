## Summary

Keep open. Current main still routes Anthropic `models auth paste-token` input through the setup-token validator, so a standard `sk-ant-api03-` API key is rejected before persistence even though a separate Anthropic API-key auth method exists.

## What This Changes

The issue asks for Anthropic `paste-token` to accept standard `sk-ant-api03-` API keys or clearly direct users to the API-key auth path.

## Best Possible Solution

Preserve `sk-ant-oat01-` setup-token behavior while detecting `sk-ant-api03-` input and routing it through API-key credentials/config, or provide an explicit API-key auth redirect before validation fails.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: `openclaw models auth paste-token --provider anthropic` passes Anthropic input into `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before any credential is persisted.

## Solution Assessment

No. The current setup-token-only validation is not the best solution because the repo already has a distinct Anthropic API-key auth method and credential mode that should be used for standard API keys.
