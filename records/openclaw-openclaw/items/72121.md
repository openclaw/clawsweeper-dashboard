## Summary

Keep open. Current main still routes Anthropic `paste-token` input through the setup-token validator, so a standard `sk-ant-api03-` API key is rejected before any credential can be written, and the related fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to accept Anthropic `sk-ant-api03-` API keys in the auth flow or clearly direct users to the separate API-key path instead of only expecting `sk-ant-oat01-`.

## Best Possible Solution

Preserve `sk-ant-oat01-` setup-token behavior while either routing `sk-ant-api03-` input through the existing Anthropic `api_key` auth/profile semantics or returning an explicit API-key command from the paste-token validation error.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects `sk-ant-api03-` before persistence.

## Solution Assessment

No for current main and for the closed validator-only approach. The maintainable fix should keep setup tokens in token mode and handle API keys through the existing API-key method or an actionable redirect.
