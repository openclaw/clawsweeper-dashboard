## Label Justifications

- `P1`: The issue is a recent regression affecting real Matrix shared-room workflows with privacy-sensitive visible internal progress output.
- `impact:security`: Visible tool/progress messages can expose file paths, command names, search terms, URLs, plugin diagnostics, or other operational context to room participants.
- `impact:message-loss`: Internal progress/tool messages are being delivered to the wrong visible chat surface instead of remaining internal by default.

## Summary

This should stay open: current `main` still has a source-backed dispatch gap where non-Slack `ChatType: "channel"` turns can pass the verbose/tool-summary gate, and Matrix room messages are classified as `channel`. The related history confirms this is adjacent to prior Slack suppression and Discord explicit verbose-delivery work, not already superseded by a merged Matrix fix.

## What This Changes

The issue asks OpenClaw to suppress internal tool/progress messages by default in Matrix multi-person rooms while preserving explicit verbose/message-tool delivery paths.

## Best Possible Solution

Fix the shared dispatch predicate so normalized multi-participant `channel` rooms suppress default text-only tool/progress messages, while preserving forum, media/approval payloads, room-event strictness, and explicit `message_tool_only` verbose progress.

## Reproduction Assessment

Yes for source-level reproduction: current `main` classifies Matrix rooms as `ChatType: "channel"`, and the shared predicate allows non-Slack `channel` tool/progress messages. I did not perform a live Matrix room run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the shared dispatch gate, not in Matrix-specific outbound code. It should use normalized chat-type semantics and preserve the existing explicit verbose/message-tool and room-event contracts.
