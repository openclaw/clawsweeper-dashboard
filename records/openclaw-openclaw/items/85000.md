## Label Justifications

- `P1`: The issue describes a privacy-sensitive regression in a real Matrix multi-person room where internal tool/progress details can become visible chat messages.
- `impact:security`: Visible internal progress summaries can expose file paths, command names, search terms, and operational context to room participants.
- `impact:message-loss`: The defect is a channel delivery routing bug where internal progress messages are sent to a visible chat surface that should remain quiet by default.

## Summary

Keep open: current main and the latest shipped tag still admit the reported Matrix channel path, and https://github.com/openclaw/openclaw/pull/85069 is an open closing PR for this issue.

## What This Changes

The issue reports that default-mode Matrix multi-person rooms can receive internal tool/progress events as visible chat messages because non-Slack `ChatType: "channel"` turns are not suppressed by the shared verbose/tool-summary gate.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85069 if it preserves final replies, explicit message-tool opt-ins, forum behavior, and room-event suppression with focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main still maps Matrix rooms to `ChatType: "channel"`, and the shared dispatch predicate allows non-Slack channel turns through the verbose/tool-summary gate. I did not run a live Matrix room in this read-only review.

## Solution Assessment

Yes on the broad direction: the durable fix belongs in shared auto-reply dispatch with tests for default channel suppression and explicit opt-ins, not in a Matrix-only workaround. The exact exception policy still needs maintainer review because it affects other channel-class surfaces.
