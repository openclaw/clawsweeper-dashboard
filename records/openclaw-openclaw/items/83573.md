## Summary

Keep open: this is useful Mattermost feature work, but the dialog picker still needs a compatibility decision around Mattermost dynamic field refresh plus stronger live proof of the actual dialog open/refresh/apply flow.

## What This Changes

The PR adds a Mattermost native dialog picker for `/oc_model` and `/oc_models`, with signed dialog state, refresh/submit callbacks, slash channel fallback behavior, tests, and docs.

## Best Possible Solution

Keep the plugin-local dialog approach, but preserve or clearly gate the old picker fallback for unsupported Mattermost refresh behavior, document the server-version boundary, and land only after redacted live open/refresh/apply proof.

## Reproduction Assessment

Yes for the review finding from source and dependency inspection: the PR sets dialog field `refresh: true` unconditionally, while Mattermost documents dynamic field refresh as Server 11.1+. I did not run an older Mattermost server in this read-only review.

## Solution Assessment

No, not quite yet. The plugin-local dialog and shared `/model` apply path are the right shape, but the PR should handle or declare the Mattermost refresh-version requirement and add real open/apply proof before merge.
