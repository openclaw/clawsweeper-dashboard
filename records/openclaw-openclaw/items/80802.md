## Best Possible Solution

Migrate the native bridge to the current Talk session RPC contract, make teardown non-restarting, then provide redacted runtime proof for the Talk, Canvas, and iOS/macOS flows.

## Reproduction Assessment

Yes for the blocking review findings: source inspection shows the new bridge calls removed RPCs absent from the PR head method list, and teardown calls a clear path that restarts playback while session is still set. Full runtime reproduction of the broad PR behavior is not established from the supplied proof.

## Solution Assessment

No; the current branch is not the best merge shape while the native bridge targets obsolete Gateway methods and teardown can restart playback after stop. The safer path is a targeted repair against the current Talk RPC contract plus visible runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the current Talk session RPCs in the bridge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/CanvasRealtimeTalkBridge.swift:170`
  - body: `CanvasRealtimeTalkBridge.start` calls `talk.realtime.session`, and the same bridge sends audio/tool/stop traffic through the old `talk.realtime.*` relay methods. PR head only advertises and handles `talk.client.create` plus `talk.session.*`, and current docs mark the `talk.realtime.*` names as removed, so the native realtime bridge will fail before it can start a relay session.
  - confidence: 0.95
- **[P2] Avoid restarting playback during bridge teardown:** `apps/shared/OpenClawKit/Sources/OpenClawKit/CanvasRealtimeTalkBridge.swift:517`
  - body: `stopInternally` calls `clearOutput` before clearing `self.session`, but `clearOutput` starts a new playback stream whenever `session.outputSampleRateHz` is still present. Stopping or erroring the bridge can therefore leave a fresh playback task/continuation alive after teardown; clear the session first or add a non-restarting clear path.
  - confidence: 0.88
