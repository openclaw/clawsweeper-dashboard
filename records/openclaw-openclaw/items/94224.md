## Label Justifications

- `P1`: The issue reports a current regression that marks recurring silent cron jobs as errors and can trigger false failure handling on every scheduled run.
- `impact:other`: The concrete impact is cron reliability and operator observability: false `empty_response` errors mask real scheduled-job failures and spam logs/status.

## Summary

Keep open: current main and v2026.6.9 still route cron runs into the CLI/embedded empty-response handling without passing `allowEmptyAssistantReplyAsSilent`; the issue already has open closing PRs, so maintainer review should pick the canonical fix rather than close or queue another repair.

## What This Changes

The issue reports a regression where intentionally silent cron jobs fail with `empty_response` because the cron execution path never opts into silent empty assistant replies.

## Best Possible Solution

Land or reject one canonical fix that threads the silent-empty policy through the cron executor boundary, covers the intended CLI and embedded cron paths, and updates regression coverage for the chosen policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a cron CLI run with whitespace-only output reaches `runCliAgent` without `allowEmptyAssistantReplyAsSilent`, then the CLI guard throws `empty_response`. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy. Passing an explicit silent-empty flag at the cron executor boundary appears to be the cleanest owner-boundary fix, but all-cron versus delivery-gated versus per-job behavior is still a product decision.
