## Label Justifications

- `P1`: The issue reports a current regression that marks recurring silent cron jobs as errors and can route false failure handling on every scheduled run.
- `impact:other`: The concrete impact is cron reliability and operator observability: false `empty_response` errors mask real scheduled-job failures and spam logs/status.

## Summary

Keep open: current main and the latest release still route cron runs into empty-response failure unless a caller passes `allowEmptyAssistantReplyAsSilent`, and the cron executor does not pass that flag. The issue is already paired with open closing PRs, so the next step is maintainer review of one canonical fix rather than closing or queuing another repair.

## What This Changes

Treat intentionally empty cron assistant replies as silent successful cron runs instead of `empty_response` FailoverError, with a clear policy for delivery and no-deliver cron jobs.

## Best Possible Solution

Land or reject one canonical fix that threads the silent-empty policy through the cron execution boundary, covers both CLI and embedded cron paths as intended, and adds regression coverage for the chosen delivery policy.

## Reproduction Assessment

Yes, from source inspection: current main still sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the runner throws `empty_response` on empty text. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. Passing an explicit silent-empty flag at the cron executor boundary looks like the cleanest owner-boundary fix, but the all-cron versus delivery-gated versus per-job option decision is still open.
