## Summary

Keep open: current main and v2026.6.10 still omit the silent-empty flag on cron runner handoffs, and the issue has open closing PRs that need maintainer review instead of another repair lane.

## What This Changes

The issue reports that isolated cron jobs with intentionally empty assistant replies fail as `empty_response` because cron never passes `allowEmptyAssistantReplyAsSilent` into the runner paths.

## Best Possible Solution

Choose and land one canonical cron-executor-boundary fix that threads the existing silent-empty runner option through both CLI and embedded cron paths with regression coverage for the accepted policy.

## Reproduction Assessment

Yes, at source level: cron calls both runner paths without `allowEmptyAssistantReplyAsSilent`, while the CLI and embedded contracts only accept clean empty replies when that flag is true. I did not run a live scheduled Gateway tick in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. Passing the existing flag at the cron executor boundary appears to be the cleanest owner-boundary fix, but all-cron versus no-deliver-only versus per-job behavior remains a maintainer decision.
