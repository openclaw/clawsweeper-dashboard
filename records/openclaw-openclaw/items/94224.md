## Summary

Current `main` and the latest release still have the reported cron empty-reply gap, and live GitHub data shows open closing PRs already own candidate fixes. This issue should stay open until maintainers choose and land one canonical fix path.

## What This Changes

Cron jobs that intentionally produce no assistant text are reported as `empty_response` failures because the cron execution path does not pass the existing silent-empty option to the agent runners.

## Best Possible Solution

Land one focused canonical fix that threads the existing silent-empty option through cron for intended silent cron contexts while preserving genuine empty-response protection outside that context.

## Reproduction Assessment

Yes. Source inspection on current `main` shows cron can call the CLI runner with cron lane/trigger but without the silent-empty flag, and the runner then throws `empty_response` on empty trimmed output.

## Solution Assessment

Unclear as a final product policy. Passing the existing flag from the cron executor to both runner paths appears to be the cleaner owner boundary, but maintainers still need to choose unconditional cron silence versus narrower delivery/config semantics.
