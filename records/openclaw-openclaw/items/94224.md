## Label Justifications

- `P1`: Silent alert cron jobs can be marked error and log `FailoverError` on every scheduled tick, which breaks a recurring automation workflow for affected users.
- `impact:other`: The concrete impact is cron run status, failure routing, and Gateway log noise rather than a more specific owned impact class.

## Summary

Keep open: current main and v2026.6.8 still route cron agent runs into the empty-response guard without setting `allowEmptyAssistantReplyAsSilent`, and two open closing PRs already own candidate fixes for this issue.

## What This Changes

The issue asks OpenClaw to treat intentionally empty cron assistant replies as silent success instead of marking scheduled runs as `empty_response` failures.

## Best Possible Solution

Review and land one focused PR that threads the existing silent-empty flag through the cron executor for intended silent cron contexts, with regression coverage for both CLI and embedded paths and the broader per-job option tracked separately if maintainers want it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a cron CLI run can pass `trigger: "cron"` and `lane: "cron"` without the silent-empty flag, then `runCliAgent` throws `empty_response` on empty trimmed CLI output.

## Solution Assessment

Unclear as a final product boundary, but the reported code seam is real. The cleaner likely fix is to pass the existing flag from the cron executor for intended silent cron contexts rather than blanket special-casing every cron lane at the runner throw site.
