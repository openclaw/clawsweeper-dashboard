## Label Justifications

- `P1`: The issue reports a current regression that marks recurring silent cron jobs as errors and can route false failure handling on every scheduled run.
- `impact:other`: The concrete impact is cron reliability and operator observability: false `empty_response` errors mask real scheduled-job failures and spam logs/status.

## Summary

Keep open: current main and the latest release still have the cron empty-response failure path, and this issue is already paired with open closing PRs for maintainer review rather than a new repair lane.

## What This Changes

Cron jobs that intentionally produce no assistant text should finish as silent success instead of failing with an `empty_response` FailoverError.

## Best Possible Solution

Review and land or reject one canonical fix that makes the cron execution boundary explicitly opt into silent-empty handling for the chosen scope, with regression coverage for CLI and embedded cron paths.

## Reproduction Assessment

Yes. Source inspection shows current main sends cron CLI runs without `allowEmptyAssistantReplyAsSilent`, and the runner rejects empty text with `empty_response`; I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. Passing an explicit silent-empty flag at the cron executor boundary looks like the clean owner-boundary fix, but all-cron versus delivery-gated versus per-job behavior is still an open decision.
