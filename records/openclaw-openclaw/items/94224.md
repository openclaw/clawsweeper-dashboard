## Label Justifications

- `P1`: Silent alert cron jobs can be marked error and log `FailoverError` on every scheduled tick, breaking a recurring automation workflow for affected users.
- `impact:other`: The concrete impact is cron run status, failure routing, and Gateway log noise rather than one of the more specific owned impact classes.

## Summary

Keep open: current main and v2026.6.8 still route cron CLI runs through `runCliAgent` without `allowEmptyAssistantReplyAsSilent`, so an empty trimmed assistant reply reaches the `empty_response` FailoverError path. The issue is already paired with open closing PRs, so it should stay open until one canonical fix is reviewed and merged.

## What This Changes

The issue asks cron agent runs to treat intentionally empty assistant replies as silent success instead of `FailoverError` failures.

## Best Possible Solution

Review and land one focused PR that threads the existing silent-empty option through cron for intended silent cron contexts, with regression coverage for CLI and embedded paths that preserves genuine empty-response protection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a cron CLI run can pass `trigger: "cron"` and `lane: "cron"` without the silent-empty flag, then `runCliAgent` throws `empty_response` on empty trimmed CLI output.

## Solution Assessment

Unclear as a product boundary, but the reported seam is real. Passing the existing flag from the cron executor is the cleaner likely fix than broad runner lane special-casing, provided tests show genuine empty CLI failures still fail outside intended silent cron contexts.
