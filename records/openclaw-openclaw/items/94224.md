## Summary

Keep open: current main and the latest release still route cron CLI and embedded runs without the silent-empty opt-in, and the issue is paired with open closing PRs that have not merged.

## What This Changes

The issue asks OpenClaw to stop treating intentionally empty cron assistant replies as `empty_response` failures when the cron job is meant to be silent.

## Best Possible Solution

Review and land one canonical cron-executor fix that passes the existing silent-empty flag for maintainer-approved silent cron contexts across both CLI and embedded paths, with regression coverage and live cron proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron can call the CLI runner with `trigger: "cron"` and no silent-empty flag, then whitespace-only CLI output reaches the `empty_response` FailoverError guard.

## Solution Assessment

No for current main. The narrowest maintainable fix is likely to set the existing flag at the cron executor boundary for the agreed silent-cron policy, rather than infer silence from a shared runner lane string.
