## Solution Assessment

No. A plugin-only admin RPC may be a useful experiment, but this branch is not the best mergeable solution until env isolation, sanitizer parity, startup policy, real proof, and the relationship to the open core/SKD work are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Run setup with a restricted base environment:** `extensions/skills-setup/index.ts:553-560`
  - body: `runPluginCommandWithTimeout` forwards this env overlay to `runCommandWithTimeout`, whose command env defaults to `process.env`. That means every skill setup script can read unrelated gateway secrets despite the PR's scoped-env contract; add or use a command runner path that supplies a minimal sanitized base env.
  - confidence: 0.94
- **[P1] Reuse the env security policy for setup overlays:** `extensions/skills-setup/index.ts:424-428`
  - body: `normalizeEnvMap` only filters a few reserved keys before passing config/request env to `bash`, so dangerous startup keys like `BASH_ENV`, `NODE_OPTIONS`, or `SHELL` can still reach the setup process. Reuse the existing skill/host env policy, or expose a setup-specific sanitizer with equivalent blocking coverage.
  - confidence: 0.91
- **[P2] Account for the startup activation baseline:** `extensions/skills-setup/openclaw.plugin.json:3-4`
  - body: The new manifest opts into startup activation, but the bundled metadata test asserts the complete startup plugin id list and this branch does not add `skills-setup` there. Either make the RPC lazy or update the startup baseline and policy intentionally.
  - confidence: 0.84
