## Summary

This PR should stay open for normal review, not cleanup-close: it is a same-day draft feature PR with an open linked feature issue, missing real behavior proof, and two blocking correctness/security findings in the proposed setup runner.

## What This Changes

The PR adds a bundled `skills-setup` plugin that registers an admin-only `skills.setup` gateway RPC to run skill-declared setup scripts with path, env, and timeout checks.

## Best Possible Solution

Keep the feature open, but route it through maintainer review to reuse or expose the shared skills/frontmatter/env sanitizer contracts, fix command failure semantics, and require live `skills.setup` gateway proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows unsafe env keys can flow into `bash` and that non-zero setup results still call `respond(true, result)`. I did not run a live gateway because this review was read-only.

## Solution Assessment

No; the proposed bundled plugin is a plausible direction, but the safest implementation should reuse the shared host/skill env policy, return RPC failure for failed setup scripts, and align with the open SDK helper work before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reuse the host env denylist for setup env:** `extensions/skills-setup/index.ts:425-428`
  - body: This filter only drops `HOME`, `PATH`, `SKILL_DIR`, and two `XDG_*` keys, then passes the rest to `bash` through `runPluginCommandWithTimeout`. That lets `skills.entries.*.env` or request `env` set execution-context variables already blocked by OpenClaw's host policy, such as `BASH_ENV`, `NODE_OPTIONS`, `LD_*`, or `GIT_*`, so a setup call can alter loading or execute code before the intended script runs. Please reuse the host/skill env sanitizer and then allow only the declared credential variables needed by the setup contract.
  - confidence: 0.9
- **[P2] Return failure when the setup script fails:** `extensions/skills-setup/index.ts:566`
  - body: `runPluginCommandWithTimeout` reports non-zero exits and timeouts as a result object with `code !== 0`, but this always acknowledges the RPC as successful. Callers that use `skills.setup` to gate install/update will treat a failed setup as success unless every caller remembers to inspect `code`, so the handler should respond with failure and an error shape when the setup command does not exit cleanly.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds an admin command-execution RPC but sanitizes env overlays less strictly than existing host and skill execution policy.

Concerns:

- **[high] Setup env bypasses host execution policy:** `extensions/skills-setup/index.ts:425`
  - body: `normalizeEnvMap` accepts execution-control variables that current OpenClaw policy blocks for host command execution, and `runPluginCommandWithTimeout` will merge those values into the spawned `bash` environment. This is a concrete command-execution hardening gap for the new RPC.
