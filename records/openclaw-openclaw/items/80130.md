## Summary

Keep open. The PR targets a real cron stall diagnostic gap, but the maintainer-protected label requires explicit handling and the patch appears to regress timeout task-ledger classification when phase-suffixed timeout errors are introduced.

## What This Changes

The PR adds setup and pre-model watchdogs, embedded-runner phase forwarding, focused cron regression tests, docs, and a changelog entry for isolated cron agentTurn stalls.

## Best Possible Solution

Land a watchdog-based cron fix that fails pre-run and pre-model stalls with phase-specific diagnostics while preserving existing timeout status semantics and then reconcile the warning-only follow-up at https://github.com/openclaw/openclaw/pull/74893.

## Reproduction Assessment

Yes, at source level. Current main has no phase callback or pre-model watchdog, and the linked report documents v2026.5.7 isolated cron runs burning the full budget before any model/provider/usage fields appeared; I did not live-run a production stall in this read-only review.

## Solution Assessment

No, not quite. The watchdog direction is the right narrow fix for the reported diagnostic gap, but the implementation should keep canonical timeout classification intact while adding phase-specific user-facing details.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve timed-out task classification for phase errors:** `src/cron/service/timer.ts:280`
  - body: Adding `(last phase: ...)` to `timeoutErrorMessage()` means ordinary job-budget timeouts that have recorded a phase no longer match `tryFinishCronTaskRun()`'s exact `normalizeCronRunErrorText(result.error) === timeoutErrorMessage()` check, so their task ledger status becomes `failed` instead of `timed_out`. Please normalize phase-specific timeout strings, or separate the stable timeout code from the displayed diagnostic text.
  - confidence: 0.82
