## Summary

Keep open: this is a fresh, concrete transport bug report, and current main still has the implicated Discord, Anthropic Vertex, and shared fetch paths without a confirmed gzip/identity-header fix. The new Linux/Vertex comment adds useful evidence that the problem may extend beyond macOS/Node 26, so this should stay with maintainers rather than be cleaned up.

## What This Changes

The issue asks OpenClaw to stop handing gzip-encoded HTTP/SSE response bytes to JSON and text parsers in Discord, Anthropic Vertex, MCP, and related wrapped fetch paths.

## Best Possible Solution

Fix the narrow transport boundary that exposes encoded bytes to JSON or SSE parsers, then prove it with a live Discord probe and an Anthropic Vertex streaming turn plus focused regression tests.

## Reproduction Assessment

No, we do not have a high-confidence current-main live reproduction. The report and follow-up provide concrete affected-system logs, and current source still follows the matching parser paths, but the live credentialed Discord/Vertex flows were not rerun here.

## Solution Assessment

Unclear. The suggested `Accept-Encoding: identity` workaround is plausible for the Vertex SSE path, but the broader Discord/MCP symptoms need root-cause confirmation before choosing a shared fetch fix versus owner-specific identity headers.
