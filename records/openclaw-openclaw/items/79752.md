## Summary

Keep open: this is a fresh, concrete transport bug report, and current main still has the Discord probe/REST paths parse response bodies directly after wrapped/custom undici fetch without repository-side gzip fallback or regression coverage. Upstream undici fetch is documented and implemented to decode gzip, so the exact Node v26/macOS trigger needs live maintainer reproduction before choosing the fix.

## What This Changes

The issue reports that OpenClaw's wrapped/custom fetch path can pass gzip-compressed HTTP bodies through to JSON parsing on Node v26/macOS, breaking Discord probe/replies and possibly other HTTP integrations.

## Best Possible Solution

Fix the shared dispatcher-backed fetch boundary or the Discord-owned REST fetch path so affected responses are decoded consistently, with live Node v26/macOS Discord proof and focused gzip regression coverage.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction on current main. The report has strong concrete output and current source still follows the matching wrapped-fetch-to-JSON path, but the Node v26/macOS decompression failure itself was not reproduced here.

## Solution Assessment

Unclear. The maintainable fix likely belongs at the shared runtime fetch or Discord REST fetch boundary, but dependency source says undici fetch normally decodes gzip with dispatchers, so live diagnostics should pick the narrowest repair before code changes.
