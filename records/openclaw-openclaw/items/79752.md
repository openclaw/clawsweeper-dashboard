Keep open: this is a fresh, concrete transport bug report, and current main still has the matching Discord wrapped-fetch and immediate JSON/text parsing paths without evidence of a gzip/content-encoding fix or a canonical replacement issue/PR.

## What This Changes

The issue reports gzip-encoded Discord and other HTTP response bodies reaching OpenClaw JSON parsing undecoded through the wrapped/custom fetch path on macOS Node v26.

## Best Possible Solution

Fix the dispatcher-backed HTTP transport or Discord REST/probe fetch boundary so decoded bodies are delivered consistently, then prove it with a macOS Node v26 Discord probe and focused regression tests.

## Reproduction Assessment

No, we do not have a high-confidence current-main live reproduction. The report provides concrete v2026.5.7 gzip-byte output and current source still follows the matching wrapped-fetch-to-JSON path, but this review did not run macOS Node v26 with live Discord credentials.

## Solution Assessment

Unclear. The likely maintainable fix belongs at the shared dispatcher-backed fetch seam or the Discord REST fetch boundary, but live proof is needed before choosing an undici/runtime wiring fix, a dependency change, or a defensive decode wrapper.
