## Summary

Keep open: this is a fresh, concrete transport bug report, and current main still routes Discord probe and REST calls through wrapped/custom undici fetch paths that parse JSON/text without an OpenClaw-side guard for raw gzip bytes. I found no source, test, changelog, related PR, or release evidence that the v2026.5.7 behavior is already fixed on main.

## What This Changes

The issue reports gzipped HTTP response bodies reaching JSON parsing undecoded in Discord and other OpenClaw HTTP paths under Node v26 on macOS.

## Best Possible Solution

Fix the shared dispatcher-backed fetch boundary, or the narrow Discord REST/probe fetch adapter if the bug proves Discord-local, so response bodies are decoded consistently before JSON/text parsing and verify it on an affected Node v26 macOS host.

## Reproduction Assessment

No. The report gives concrete gzip magic bytes from `openclaw channels status --probe --json`, and current source still follows the same wrapped-fetch-to-JSON path, but this review did not establish a live failing current-main run on Node v26/macOS with credentials.

## Solution Assessment

Unclear. The likely maintainable fix belongs at the dispatcher-backed fetch boundary so all affected users receive decoded bodies, but live proof is needed before choosing between undici/runtime wiring, a dependency change, or a defensive decode wrapper.
