## Summary

Keep open: this is a fresh, concrete transport bug report, and current main still routes Discord probe and REST calls through wrapped/custom undici fetch paths that immediately parse JSON/text without any repository-side guard for raw gzip bytes. I found no source, test, changelog, or release evidence that the v2026.5.7 behavior has already been fixed on main.

## What This Changes

Ensure OpenClaw's wrapped and dispatcher-backed HTTP fetch paths decompress gzip-encoded responses before JSON parsing under Node v26 on macOS.

## Best Possible Solution

Fix the dispatcher-backed HTTP transport boundary so Discord probe/REST responses and any shared fetch users see decoded bodies consistently, then verify with a Node v26 macOS Discord probe and focused regression tests.

## Reproduction Assessment

No, we do not have a high-confidence current-main live reproduction. The report provides concrete v2026.5.7 output with gzip header bytes, and current source still follows the matching Discord wrapped-fetch-to-JSON path, but I did not reproduce it on Node v26/macOS with live credentials.

## Solution Assessment

Unclear. The likely maintainable fix belongs at the shared dispatcher-backed fetch or Discord REST fetch boundary, but live proof is needed before choosing between an undici/runtime wiring fix, a dependency change, or a defensive decode wrapper.
