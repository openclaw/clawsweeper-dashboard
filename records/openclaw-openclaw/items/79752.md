## Summary

Keep open: current main still routes Discord probe and REST calls through wrapped/custom fetch paths that parse JSON/text without any repository-side evidence of a gzip decoding fix, and the report is concrete even though live current-main reproduction is still missing.

## What This Changes

Investigate a Node v26/macOS transport bug where OpenClaw wrapped fetch paths appear to pass gzip response bytes into JSON parsing for Discord and possibly other HTTP integrations.

## Best Possible Solution

Fix the dispatcher-backed HTTP transport boundary so Discord probe/REST responses and shared fetch users receive decoded bodies consistently, then prove it with a live macOS Node v26 Discord probe and focused regression coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The report provides concrete v2026.5.7 output with gzip header bytes and current source still follows the matching wrapped-fetch-to-JSON path, but I did not run Node v26 on macOS with live Discord credentials.

## Solution Assessment

Unclear. The likely maintainable fix belongs at the shared dispatcher-backed fetch seam or Discord REST fetch boundary, but live proof is needed before choosing an undici/runtime wiring fix, dependency change, or defensive decode wrapper.
