## Summary

Keep open: this is a fresh, concrete transport bug report against v2026.5.7, and current main still has the Discord probe and REST paths that would parse raw gzip bytes as JSON/text if the wrapped fetch returns an undecoded body. I found no source, test, changelog, PR, or release evidence that the reported Node v26/macOS behavior is already fixed; the exact trigger still needs live confirmation because undici 8.2.0 fetch source normally decodes gzip responses.

## What This Changes

The issue requests fixing OpenClaw's wrapped/custom fetch path so gzip-encoded Discord, Slack MCP, and possibly Anthropic HTTP responses are decoded before JSON or stream parsing on macOS Node v26.

## Best Possible Solution

Confirm the affected macOS Node v26 path with redacted live Discord proof, then fix the narrow shared dispatcher or Discord REST fetch boundary and add focused content-encoding regression coverage.

## Reproduction Assessment

No, we do not have a high-confidence current-main live reproduction. The issue provides concrete v2026.5.7 output and source inspection shows the matching parser path if raw gzip reaches it, but the Node v26/macOS wrapped-fetch trigger was not reproduced here.

## Solution Assessment

Unclear. The likely fix belongs at the shared dispatcher-backed fetch or Discord REST fetch boundary, but undici 8.2.0 fetch source normally decodes gzip, so live proof should drive whether this is an undici wiring fix, a dependency/runtime issue, or a defensive decode guard.
