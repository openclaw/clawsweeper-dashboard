## Summary

Keep open. Current main still leaves a source-provable Feishu markdown `write`/`append` failure path: normal-size converted blocks go through the non-batch descendant serializer, where normalized child IDs can still be overwritten, and the manual credential note still omits the reporter-verified conversion scope.

## What This Changes

The issue reports that Feishu document markdown `write` and `append` return 400 responses while `update_block` works.

## Best Possible Solution

Patch the Feishu plugin serializer so normalized descendant `children` survive for all converted block types, align setup guidance with the conversion permission, and cover the markdown write/append path with focused regressions.

## Reproduction Assessment

Yes, via a high-confidence source-level path: a mocked convert result with string-valued `children` sent through `write` or `append` will build a descendant payload that can send `children` as a string. The reporter also live-verified that adding `docx:document.block:convert` made their write and append calls succeed, but this review did not rerun live Feishu credentials.

## Solution Assessment

Yes. A narrow Feishu plugin/docs repair is the maintainable solution: fix the serializer spread order, add regression coverage for string-valued converted children, and update setup guidance without adding core APIs or product policy.
