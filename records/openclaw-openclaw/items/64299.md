## Summary

Keep open. Current main still has a source-provable Feishu markdown `write`/`append` failure path: normal-size converted blocks go through a descendant serializer where normalized `children` arrays can be overwritten by raw string-valued `children`, and the manual setup permission text still omits the reporter-verified conversion scope.

## What This Changes

The issue reports that Feishu document markdown `write` and `append` return 400 Bad Request despite valid content.

## Best Possible Solution

Patch the Feishu plugin serializer so normalized descendant `children` survive for all converted block types, align setup guidance with the conversion permission, and cover markdown write/append with focused regressions.

## Reproduction Assessment

Yes, source-level: a converted Feishu block with string-valued `children` sent through `write` or `append` can reach the descendant payload with `children` as a string because the normalized value is overwritten. The reporter also live-verified that adding `docx:document.block:convert` made their write and append calls succeed.

## Solution Assessment

Yes. A narrow Feishu plugin/docs repair is the maintainable solution: fix the serializer spread order, add regression coverage for string-valued converted children, and update setup permission guidance without adding core APIs or product policy.
