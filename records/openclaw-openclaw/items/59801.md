## Summary

Keep open: the feature is still useful, but this PR is not ready because it targets stale WhatsApp dispatch code on current main and still lacks public docs/generated-baseline handling plus real after-fix WhatsApp proof.

## What This Changes

The PR adds WhatsApp `errorPolicy`/`errorCooldownMs` config fields, account resolution, an in-memory cooldown helper, delivery suppression wiring, generated config metadata, and unit tests.

## Best Possible Solution

Rebase the feature into `dispatchWhatsAppBufferedReply`, resolve policy from merged WhatsApp account config at the active delivery point, document the public keys, regenerate the tracked config hash, and preserve operational delivery-failure logging.

## Reproduction Assessment

Yes for the PR defect; not applicable as a standalone runtime bug. Source inspection on current main shows WhatsApp delivery now flows through `dispatchWhatsAppBufferedReply`, whose helper filters `payload.isError` before the PR's stale `process-message.ts` hook would control delivery.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best implementation because it wires policy into the obsolete dispatch location and updates local generated baselines instead of the tracked hash and public WhatsApp docs.
