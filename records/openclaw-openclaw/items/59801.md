## Summary

Keep open: the feature is still useful and not implemented on main, but the current PR is not mergeable because it targets a stale WhatsApp dispatcher, only gates `isError` payloads while the reported agent-run failures are metadata-only final payloads, and lacks docs/generated-hash/proof coverage.

## What This Changes

This PR adds WhatsApp `errorPolicy` and `errorCooldownMs` config/schema/account fields, an in-memory suppression helper, old monitor delivery wiring, generated config baseline changes, and unit tests.

## Best Possible Solution

Implement the policy at the active WhatsApp dispatch boundary, target actual agent-run failure payloads, update the tracked config hash and public WhatsApp docs, and preserve delivery-failure logging.

## Reproduction Assessment

Yes for the PR defects: source inspection shows current WhatsApp delivery uses `runInboundReplyTurn` and `dispatchWhatsAppBufferedReply`, while the reported agent-run failures are metadata-only final payloads rather than `isError` payloads. I did not run a live WhatsApp setup in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best implementation until it is rebased into the active dispatch path, applies to the real failure payload class, and updates docs/generated artifacts correctly.
