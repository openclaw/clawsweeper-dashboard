## Summary

Keep open: this is a focused Telegram regression fix, but it is not merge-ready because the new ledger write can use the wrong cache scope from automatic dispatch and only records the first delivered Telegram message id.

## What This Changes

The PR adds a sent-message ledger write to Telegram `deliverReplies` and adds regression tests for successful and skipped delivery.

## Best Possible Solution

Record every successful Telegram send with the caller's `cfg` propagated from automatic dispatch, then cover custom-store and multi-message delivery with regression tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current `deliverReplies` emits `message_sent` after successful sends without a ledger write, while reaction filtering reads `wasSentByBot(..., cfg)`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Co-locating a single write beside the hook is plausible, but the best fix needs the same cfg scope and per-message granularity as the canonical `sendMessageTelegram` path.
