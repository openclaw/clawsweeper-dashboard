## Summary

Keep open: current main has input-provenance markers, plugin before_tool_call enforcement, and exec approval routing, but it still lacks the requested verified-human instruction-provenance policy for sensitive targets and autonomous side-effect actions.

## What This Changes

Request a gateway-enforced sensitive-target policy that requires exec and other sensitive side-effect actions to trace back to an explicit verified-human instruction.

## Best Possible Solution

Maintainers should define the verified-human provenance security contract first, then extend the narrowest core/plugin policy surface needed to enforce sensitive side-effect actions consistently.

Not applicable as a bug reproduction: this is a feature/security policy request, not a report that current documented behavior is broken. Source inspection confirms the requested verified-human provenance gate is not present on current main.

## Solution Assessment

Unclear until maintainers decide the security contract. The best implementation should avoid an exec-only matcher and instead define a shared provenance/action-classification policy surface before sensitive-target enforcement.
