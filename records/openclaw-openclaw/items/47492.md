## Summary

Keep open: current main has exec approvals, allowlists, hook approvals, and narrow input provenance, but it does not implement sensitive target policies that require a verified human instruction chain. The request is security-sensitive, cross-cutting feature work rather than stale cleanup or an implemented behavior.

## What This Changes

The issue requests a gateway-enforced policy layer that marks sensitive exec/tool targets and requires verified human instruction provenance before side-effect actions run.

## Best Possible Solution

Keep this open as a focused security/product design item: define a generic direct-human-intent provenance contract, sensitive target/action taxonomy, fallback approval/challenge semantics, config/schema/docs, and tests coordinated with the open provenance and CaMeL work.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is source-verifiable: exec config, approval payloads, hook events, and input provenance do not carry the requested verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the trust model and public API. A narrow exec-only patch would be premature; the maintainable path is a coordinated gateway/plugin-SDK policy design that covers exec, gateway mutation, messaging, cron, and autonomous flows.
