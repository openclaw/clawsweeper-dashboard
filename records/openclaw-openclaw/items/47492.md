## Summary

Keep open: the request describes a security/product feature that current main does not implement, and the related CaMeL work is complementary taint tracking rather than verified-human authorization for sensitive targets.

## What This Changes

Request to add a gateway-enforced verified-human instruction-provenance policy for sensitive side-effect actions such as exec, SSH/SCP targets, gateway mutation, and high-risk messaging.

## Best Possible Solution

Define a generic direct-human-intent provenance contract, sensitive-action taxonomy, fallback approval/challenge semantics, config/schema/docs, and regression coverage across exec, gateway mutation, messaging, autonomous, cron/heartbeat, and subagent flows.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/security design request. The gap is source-verifiable because current exec config, runtime schema, approval payloads, input provenance, and plugin hook events do not carry verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the security trust model and public API. A narrow exec-only policy would be premature unless it fits a shared side-effect provenance model across channels, gateway mutation, messaging, cron/heartbeat, and subagents.
