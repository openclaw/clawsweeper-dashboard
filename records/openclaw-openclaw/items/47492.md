## Summary

Keep open. Current main has exec approvals, chat/input provenance metadata, plugin approval hooks, and adjacent CaMeL/persistent-taint discussions, but it still lacks a gateway-enforced policy that requires sensitive side-effect targets to trace back to a verified human instruction. The related CaMeL PR and persistent-taint issue are adjacent security work, not a merged replacement for this clean-session/autonomous-action gating request.

## What This Changes

Request to add a policy layer that marks sensitive exec/gateway/messaging targets and denies or escalates those actions unless they trace to a verified human instruction.

## Best Possible Solution

Define a generic instruction-provenance contract and a gateway/tool-call policy point for sensitive side-effect targets, then implement it across exec and other high-risk surfaces with config schema, protocol fields, approval UX, tests, and user-facing docs after maintainers agree on the security semantics.

Not applicable as a feature/security design request. The high-confidence verification path is static: current source, schemas, docs, and related issue state show existing approval/provenance pieces but no verified-human sensitive-target enforcement.

## Solution Assessment
