## Summary

Keep open: current main still lacks a verified-human instruction-provenance policy for sensitive exec targets or other side-effect destinations, and the related CaMeL/provenance work remains open design territory rather than conservative cleanup.

## What This Changes

The issue requests a gateway-enforced policy requiring verified-human instruction provenance for sensitive exec targets and similar side-effect actions.

## Best Possible Solution

Keep this open as a focused security/product design item for a generic direct-human-intent provenance contract, sensitive action taxonomy, fallback approval/challenge semantics, config/schema/docs, and tests coordinated with the broader provenance and CaMeL work.

Not applicable as a feature request. The current-main gap is source-verifiable: exec config/schema, approval payloads, gateway allowlist decisions, hook events, and input provenance do not carry the requested verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the trust model and public API. A narrow exec-only patch would be premature; the maintainable path is a coordinated gateway/plugin-SDK policy design that also covers gateway mutation, messaging, cron, heartbeat, and autonomous flows.
