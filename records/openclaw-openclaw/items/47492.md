## Summary

Keep open: current main still has exec allowlists/approvals, plugin hook approvals, ACP/system input provenance, and inter-session provenance markers, but it does not implement sensitive target policies that require a verified-human instruction chain. This remains security-sensitive feature/config/API design work, not conservative cleanup.

## What This Changes

Request to add gateway-enforced verified-human instruction provenance requirements for sensitive exec targets and other sensitive side-effect actions.

## Best Possible Solution

Keep this open as a focused security/product design item for a generic direct-human-intent provenance contract, sensitive action taxonomy, fallback approval/challenge semantics, config/schema/docs, and tests coordinated with the provenance and CaMeL work.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is source-verifiable: exec config/schema, approval payloads, gateway allowlist decisions, hook events, and input provenance do not carry the requested verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the trust model and public API. A narrow exec-only patch would be premature; the maintainable path is a coordinated gateway/plugin-SDK policy design that also covers gateway mutation, messaging, cron, heartbeat, and autonomous flows.
