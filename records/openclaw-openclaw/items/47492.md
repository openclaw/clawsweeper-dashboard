## Summary

Keep open: current main and the latest shipped tag still lack a verified-human direct-instruction provenance contract or sensitive-target enforcement, and the related CaMeL, taint, and hook-provenance work is complementary rather than a merged replacement.

## What This Changes

The issue requests a gateway-enforced verified-human instruction provenance policy for sensitive side-effect actions such as exec, gateway mutation, messaging, and exfiltration targets.

## Best Possible Solution

Define a generic direct-human-intent provenance contract, sensitive-action taxonomy, fallback approval or challenge semantics, config/schema/docs, and regression coverage across exec, gateway mutation, messaging, cron/heartbeat, autonomous, and subagent flows.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a security feature/design request. The current-main gap is source-verifiable because exec config/schema, approval payloads, input provenance, and before_tool_call events do not carry verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the security trust model and public API. A narrow exec-only sensitiveTargets patch would be premature; the maintainable path should cover side-effect actions across exec, gateway mutation, messaging, cron/heartbeat, autonomous, and subagent flows.
