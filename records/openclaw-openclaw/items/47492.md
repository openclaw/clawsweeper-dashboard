## Summary

Keep open. Current main has exec approvals, input/system provenance metadata, and plugin approval hooks, but it still lacks a gateway-enforced policy requiring sensitive side-effect targets to trace back to a verified human instruction; the related CaMeL PR and persistent-taint issue are adjacent work, not merged replacements.

## What This Changes

The issue requests verified-human instruction-provenance enforcement for sensitive side-effect targets such as exec commands to personal hosts, gateway/config mutation, and high-risk messaging destinations.

## Best Possible Solution

Define a generic instruction-provenance contract and gateway policy point for sensitive side-effect targets, then wire it through exec, gateway mutation, messaging, config/schema, approval UX, tests, and docs after maintainer security design.

## Reproduction Assessment

Not applicable as a feature/security design request. The high-confidence verification path is static: current source, protocol schemas, docs, and related issue state show approval/provenance pieces but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for immediate implementation. The requested direction is valid, but the maintainable solution should be a coordinated core security policy and protocol design rather than an exec-only sensitiveTargets matcher.
