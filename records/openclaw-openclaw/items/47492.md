## Summary

Keep open: current main still lacks a verified-human instruction-provenance contract or sensitive-target enforcement, and the related CaMeL/taint work does not cover clean autonomous actions that were never directly requested by a human.

## What This Changes

Request a gateway-enforced policy layer that marks sensitive exec or side-effect targets and requires traceability to a verified human instruction or a stricter approval flow.

## Best Possible Solution

Keep this as a focused security/product design item until maintainers define generic direct-human-intent provenance, sensitive action taxonomy, fallback approval/challenge semantics, config/schema/docs, and regression coverage.

## Reproduction Assessment

Not applicable as a feature/security design request. The current-main gap is source-verifiable: exec config/schema, approval payloads, input provenance, and before_tool_call events do not carry verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the security trust model and public API. A narrow exec-only sensitiveTargets patch would be premature; the maintainable path should cover exec, gateway mutation, messaging, cron/heartbeat, and autonomous or subagent flows.
