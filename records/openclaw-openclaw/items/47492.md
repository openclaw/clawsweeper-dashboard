## Summary

Keep open: current main still lacks a verified-human instruction provenance contract for sensitive targets, and the request remains security-sensitive feature/config/API design rather than conservative cleanup.

## What This Changes

The issue requests a gateway-enforced policy layer that lets operators mark sensitive exec or side-effect targets and require a verified-human instruction chain before execution.

## Best Possible Solution

Keep this as a focused security/product design item until maintainers define a generic direct-human-intent provenance contract, sensitive action taxonomy, fallback approval/challenge semantics, config/schema/docs, and regression coverage coordinated with the provenance and CaMeL work.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is source-verifiable: exec config/schema, approval payloads, gateway allowlist decisions, hook events, and input provenance do not carry verified-human sensitive-target policy data.

## Solution Assessment

Unclear until maintainers choose the trust model and public API. A narrow exec-only patch would be premature; the maintainable path is coordinated gateway/plugin-SDK policy design that also covers gateway mutation, messaging, cron, heartbeat, and autonomous flows.
