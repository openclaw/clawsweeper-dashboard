## Label Justifications

- `P1`: A shipped user workflow can request exec approval from WhatsApp but cannot complete the approval from that same channel, forcing an operator workaround.
- `impact:security`: The affected code controls who can resolve exec approvals, so the fix must preserve approval authorization boundaries.

## Summary

Keep open: current main still has the reported visibility mismatch. Generic chat `/approve` uses a plain gateway call, while gateway approval visibility only lets approval-runtime clients resolve requester-bound approvals across client/device identity boundaries; Telegram callbacks already use that runtime path.

## What This Changes

The issue asks for WhatsApp text `/approve <id> allow-once` to resolve pending exec approvals the same way Telegram approval callbacks do.

## Best Possible Solution

Route authorized generic text `/approve` submissions through the shared approval-runtime resolver, with focused coverage for WhatsApp direct-chat approvals and explicit checks that unauthorized senders still cannot resolve approvals.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: an authorized WhatsApp text `/approve` reaches `handleApproveCommand`, uses plain `callGateway`, and can miss requester-bound approvals that the gateway only exposes to approval-runtime clients. I did not run a live WhatsApp plus Telegram gateway repro in this read-only review.

## Solution Assessment

Unclear as an implementation, but the suggested direction is the maintainable path if it preserves the existing sender/channel authorization checks before using the approval-runtime resolver. A direct visibility bypass or broad gateway relaxation would not be the right fix.
