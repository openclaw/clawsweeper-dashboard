## Label Justifications

- `P1`: A real channel approval workflow is blocked for WhatsApp operators, forcing a cross-channel Telegram workaround for exec approvals.
- `impact:security`: The affected code is the exec approval authorization and visibility boundary, even though the observed failure is fail-closed rather than a bypass.

## Summary

Keep open: current main still sends generic chat `/approve` through a plain gateway client, while Telegram approval callbacks use the approval-runtime client that can see requester-bound approvals. The report matches the current approval visibility rules, and I did not find a canonical duplicate or merged fix.

## What This Changes

This issue asks OpenClaw to make authorized WhatsApp text `/approve <id> allow-once` resolve pending exec approvals the same way Telegram does.

## Best Possible Solution

Use a shared post-authorization approval-runtime resolver for generic channel text approvals, then cover WhatsApp and the requester-bound gateway visibility case with focused regression tests.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a clear current-main path for the reported failure: generic text approval lacks the approval-runtime visibility that Telegram uses.

## Solution Assessment

Likely yes: route authorized generic text `/approve` through the existing approval-runtime resolver after authorization. The safer alternative is not to broaden gateway visibility generally, but to keep the elevated resolver behind the channel’s existing approval auth checks.
