- `P1`: The original issue blocked a real channel approval workflow and forced operators to use Telegram as a workaround for exec approvals.

## Summary

Close: current main now resolves authorized chat `/approve` commands through the trusted approval resolver, which addresses the reported WhatsApp visibility failure without broadening the raw gateway lookup path.

## What This Changes

The issue reports that WhatsApp manual `/approve <id> allow-once` cannot resolve pending exec approvals that Telegram can resolve with the same ID.

## Best Possible Solution

Keep the merged resolver-based implementation; if the separate duplicate WhatsApp prompt or missing prompt-text ID still reproduces on current main, track that as a narrower follow-up.

## Reproduction Assessment

Yes for the original failure path by source inspection and reporter logs: the old shared command path used a plain gateway call while approval visibility depended on the approval-runtime client. I did not run a live WhatsApp repro in this read-only review, and current main now has the fix path.

## Solution Assessment

Yes: reusing `resolveApprovalOverGateway` after the existing sender authorization and gateway-scope checks is the narrow maintainable fix, because it preserves the approval boundary instead of duplicating visibility logic in the chat command handler.
