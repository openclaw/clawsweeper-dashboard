## Summary

Current main already solves the reported PDF hot-path problem: `pdfjs-dist` is behind the document-extractor runtime path and the PDF tool factory is already gated by effective tool policy before `createPdfTool` is called.

## What This Changes

The PR adds a `tools.deny` check inside `createPdfTool` so it returns `null` when `pdf` is explicitly denied.

## Best Possible Solution

Keep the existing lazy document-extractor implementation and close this redundant PR instead of adding another PDF-specific deny guard.

## Reproduction Assessment

No: current main does not reproduce the stated per-turn `pdfjs-dist` load because the only runtime `pdfjs-dist` load is the dynamic import inside the document-extract plugin's extractor execution path.

## Solution Assessment

No: the branch's deny-only guard is not the best fix for the reported overhead; current main already solves the hot path more generally by narrowing the PDF extractor surface and gating optional PDF factory creation through tool policy.
