## Summary

Current main already solves the reported denied-PDF hot path through centralized optional media factory planning, so this PR's narrower `createPdfTool` guard is redundant.

## What This Changes

The PR adds an early return in `createPdfTool` when `options.config.tools.deny` contains `pdf`.

## Best Possible Solution

Keep the current centralized denylist-aware factory planning on main; this PR no longer needs to add a separate PDF-only guard.

## Reproduction Assessment

No current-main failing reproduction: current main routes `tools.deny: ["pdf"]` into optional media planning before `createPdfTool` is called, with source tests covering that path. I did not run the tests because `pnpm` is unavailable in this environment.

## Solution Assessment

Yes, current main is the better solution: centralized factory planning respects the effective merged deny policy and skips all denied optional media factories before construction, instead of adding a narrower PDF-only check inside `createPdfTool`.
