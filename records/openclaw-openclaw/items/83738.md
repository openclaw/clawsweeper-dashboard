## Summary

Keep open: the reported wake-routing bug is real and the branch is pointed at the right seam, but latest head still misroutes explicit sessionKey-only cross-agent wakes and the supplied proof is synthetic rather than real transport/runtime proof.

## What This Changes

The PR adds optional sessionKey/agentId capture and forwarding for cron wake through the agent tool, gateway protocol/handler, cron service, tests, and a proof harness.

## Best Possible Solution

Keep the origin-capture direction, but fix explicit target precedence, make the caller-lane default a deliberate maintainer choice, and land only with redacted real non-main wake-route proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the agent cron tool wake sends only {mode, text}, and the cron wake/service boundary forwards no agentId from that path; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not yet. The seam is the right place to fix wake origin capture, but explicit sessionKey-only targeting must not inherit the caller agentId, and the compatibility-sensitive default change needs real route proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not pair explicit session keys with the caller agent:** `src/agents/tools/cron-tool.ts:847`
  - body: When a caller supplies `sessionKey: "agent:coding:..."` without `agentId`, this line still fills `agentId` from the caller's current session. The gateway target resolver treats that explicit agent as authoritative, so a cross-agent session key can be canonicalized back to the caller agent's main lane instead of the requested lane. Derive the agent from the explicit canonical session key or omit the inferred agentId when an explicit sessionKey is present, and add a regression for that case.
  - confidence: 0.9
