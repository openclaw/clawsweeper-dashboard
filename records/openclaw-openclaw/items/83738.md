## Summary

Keep this PR open: the direction is useful for the cron wake routing bug, but the latest head still has a blocking explicit-target precedence bug and the supplied proof is synthetic rather than real channel/runtime behavior.

## What This Changes

The PR threads optional `sessionKey` and `agentId` through the cron wake tool, gateway wake schema/handler, cron service wake routing, tests, and a local proof harness.

## Best Possible Solution

Keep the gateway/service origin-threading direction, fix explicit target inference so `sessionKey` and `agentId` cannot disagree, cover the cross-agent case, and require redacted real route proof before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because the cron tool `wake` action forwards only `mode` and `text`, while the gateway adapter already needs session/agent target data to route non-main wakes correctly.

## Solution Assessment

No. The gateway/service threading is the right direction, but the current PR still allows `sessionKey` and inferred `agentId` to disagree and does not include real behavior proof.

- **[P1] Derive the agent from an explicit session key:** `src/agents/tools/cron-tool.ts:847`
  - body: When a caller supplies `sessionKey` but omits `agentId`, this falls back to the caller's inferred `agentId`. Please derive `agentId` from the explicit session key, or omit it and let the gateway derive it, and add a cross-agent regression test.
