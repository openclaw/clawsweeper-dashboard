## Summary

Keep open: the branch is aimed at the right cron/gateway routing bug, but it is not merge-ready because an explicit cross-agent sessionKey can still be paired with the caller's inferred agentId, and the supplied proof is a synthetic harness rather than a real wake route.

## What This Changes

The branch threads optional sessionKey and agentId through cron wake tool calls, gateway wake validation/handling, cron wake service routing, tests, and a synthetic proof harness.

## Best Possible Solution

Keep the gateway/service origin-threading direction, fix target inference so an explicit sessionKey derives or omits agentId consistently, cover the cross-agent case, and require redacted real route proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main's cron tool wake omits origin metadata, and the PR's explicit-sessionKey path can still be traced to a mismatched agent/session route through the gateway cron target resolver.

## Solution Assessment

No. The gateway/service seam is the right place to thread origin, but the PR must avoid pairing an explicit cross-agent sessionKey with the caller's inferred agentId and still needs real behavior proof.
