## Summary

Keep open. The branch addresses a real cron wake routing gap, but explicit cross-agent sessionKey-only tool calls can still be paired with the caller's inferred agentId and routed to the wrong lane, and the supplied proof is still synthetic rather than a live route proof.

## What This Changes

The branch threads optional sessionKey and agentId through the cron wake tool, gateway wake schema/handler, cron service wake routing, regression tests, and a synthetic proof script.

## Best Possible Solution

Keep the origin-capture direction, but make explicit target precedence unambiguous, preserve or explicitly accept the no-origin compatibility change, and land only with redacted real route proof.

## Reproduction Assessment

Yes, by source inspection: current main's cron tool wake sends only {mode, text}, and the gateway cron target resolver falls back when no origin is supplied. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not yet. The design is the right seam, but explicit sessionKey-only targeting must not inherit the caller agentId, and the compatibility-sensitive default change needs live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not pair explicit session keys with the caller agent:** `src/agents/tools/cron-tool.ts:847`
  - body: When a caller supplies `sessionKey: "agent:coding:..."` without `agentId`, this line still fills `agentId` from the caller's current session. The gateway target resolver treats that explicit agent as authoritative, so a cross-agent session key can be canonicalized back to the caller agent's main lane instead of the requested lane. Derive the agent from the explicit canonical session key or omit the inferred agentId when an explicit sessionKey is present, and add a regression for that case.
  - confidence: 0.9
