## Summary

Keep this PR open: the direction fixes a real cron wake routing gap, but the current branch can still misroute explicit cross-agent sessionKey-only wake calls and the supplied proof is a shimmed harness rather than a real gateway/channel/CLI wake path.

## What This Changes

The branch threads optional sessionKey and agentId through the cron wake tool, gateway wake schema/handler, cron service wake path, and adds focused tests plus a proof harness.

## Best Possible Solution

Thread the originating session for ordinary wake calls, but let an explicit sessionKey derive its own agent unless agentId is also explicitly provided, then prove the route with a real redacted wake path.

## Reproduction Assessment

Yes by source inspection: current main's cron tool wake path forwards only mode/text, and the PR's explicit-sessionKey path can be traced into the existing agent/session mismatch fallback. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The architectural direction is right, but explicit sessionKey-only calls should not inherit the caller's agentId, and the PR still needs real behavior proof beyond the shimmed harness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not infer agentId for explicit sessionKey overrides:** `src/agents/tools/cron-tool.ts:847`
  - body: When a caller passes an explicit sessionKey for another agent but omits agentId, this line falls back to the caller's inferred agentId. The gateway cron adapter gives an explicit agentId precedence over deriving from the agent-prefixed sessionKey, so a wake from `main` targeting an `agent:ops:...` session can be re-resolved back to the wrong agent/default lane. Only send an inferred agentId when sessionKey is inferred too, or derive the agent from the explicit sessionKey.
  - confidence: 0.91
