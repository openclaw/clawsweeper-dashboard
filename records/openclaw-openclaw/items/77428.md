## Summary

Keep open. Current main still takes an already-loaded `launchctl bootstrap` result to a `launchctl kickstart`, and the existing unit tests explicitly lock in that behavior even when the mocked service is already running.

## What This Changes

The issue asks macOS LaunchAgent bootstrap repair to return `already-loaded` without kickstarting when the LaunchAgent is already loaded and running.

## Best Possible Solution

Gate the already-loaded branch on `readLaunchAgentRuntime`: return `already-loaded` when the job is running, and preserve the existing kickstart fallback for stopped, unknown, or unhealthy states.

## Reproduction Assessment

Yes at source level. Make `bootstrap` return an already-loaded failure while the mocked LaunchAgent is running; current main then calls `kickstart`, and the unit tests currently expect that call.

## Solution Assessment

Yes. The suggested runtime check is the narrowest maintainable fix because `readLaunchAgentRuntime` already exposes running versus stopped/unknown state and avoids changing restart or fresh-bootstrap behavior.

Preserve the current kickstart and typed failure behavior for already-loaded services that are stopped, unknown, or otherwise not running. Add/update focused regression tests in `src/daemon/launchd.test.ts` for running already-loaded no-kickstart and non-running already-loaded still-kickstart behavior. Add a single-line user-facing `CHANGELOG.md` fix entry in the active fixes section. Do not change `restartLaunchAgent`, install fresh-bootstrap behavior, launchd labels, or add a config option; stop if the fix would become a product-policy change.
