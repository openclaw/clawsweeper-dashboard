## Summary

Keep open: current main has thread-bound spawn gates and ACPX permission policy, but it still lacks the requested per-channel ACP spawn deny/warn policy and lacks completion-event coalescing for repeated ACP failures from non-interactive channels. Related #82017 and #91093 are adjacent ACP execution/lifecycle work, but neither owns this channel-spawn-policy problem.

## What This Changes

Request a channel-aware ACP spawn policy and/or repeated completion coalescing so non-interactive channel ACP sessions cannot flood parent chats after permission denial.

## Best Possible Solution

Define one canonical channel-aware ACP spawn policy or extend the existing thread-binding/tool-policy contract so denial happens before ACP runtime creation, then decide whether repeated terminal ACP task events need generic coalescing and cover schema, docs, and focused ACP/Telegram/session-spawn tests.

## Reproduction Assessment

No live Telegram/Codex reproduction was run, but current source and docs show explicit thread-bound ACP spawns remain allowed when threadBindings permits them and no requested acpSpawnPolicy or coalescing exists.

## Solution Assessment

Unclear: the proposed per-channel acpSpawnPolicy is plausible, but the best fix needs maintainer choice on whether to consolidate with existing threadBindings, ACPX permission policy, tool policy, or a generic background-announcement coalescer.
