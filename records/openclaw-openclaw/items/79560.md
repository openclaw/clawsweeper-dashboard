## Summary

Keep open: current main still lacks a general per-channel ACP deny/warn policy for `sessions_spawn(runtime="acp")`, and the remaining mitigation requires a public config/product decision rather than cleanup closure.

## What This Changes

The issue requests a per-channel ACP spawn policy and/or completion coalescing to prevent non-interactive Telegram DM ACP sessions from flooding chats after permission-denial loops.

## Best Possible Solution

Define a small channel-aware ACP safety contract, document its default behavior, then implement the chosen guard and any announcement coalescing with focused coverage for `sessions_spawn(runtime="acp")`, ACP terminal delivery, and Telegram DM/topic contexts.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection confirms the relevant ACP spawn, non-interactive permission, and background task delivery paths, but the exact Telegram/acpx retry loop was not exercised in a real chat setup.

## Solution Assessment

Unclear as proposed: per-channel deny/warn is plausible, but maintainers should choose whether the durable fix belongs in channel policy, announcement coalescing, acpx retry handling, or a combination.
