## Summary

Keep open: current main has thread-binding spawn gates and ACPX non-interactive permission settings, but it still lacks an ACP-specific per-channel deny/warn policy and lacks repeated terminal-event coalescing for this Telegram/non-interactive ACP flood pattern.

## What This Changes

Request a per-channel ACP spawn policy or equivalent mitigation to stop non-interactive channels from spawning runaway ACP sessions and flooding parent chats with repeated completion events.

## Best Possible Solution

Choose one canonical policy boundary, likely extending threadBindings or ACP spawn policy so denial happens before ACP runtime creation, then add focused config/schema/docs and ACP/Telegram/task-notification coverage with live proof before closing fixed.

No live Telegram/Codex repro was run; source inspection shows explicit thread-bound ACP spawns remain allowed when threadBindings.spawnSessions permits them, and task terminal delivery lacks generic identical-failure coalescing.

## Solution Assessment

Unclear until maintainers choose the policy boundary: a new acpSpawnPolicy is plausible, but extending existing threadBindings or adding generic task coalescing may be cleaner.
