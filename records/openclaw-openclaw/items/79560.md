## Summary

Keep open: current main still has ACP enable/allowlist/concurrency controls and thread-bound spawn toggles, but not a general per-channel ACP deny/warn policy or same-shape completion-event coalescing for noisy Telegram-DM ACP failures.

## What This Changes

The issue requests channel-aware controls to prevent ACP child sessions spawned from non-interactive channels such as Telegram DMs from entering noisy retry/failure announcement loops.

## Best Possible Solution

Define a small documented channel-aware ACP safety contract, then implement the chosen guard with focused coverage around `sessions_spawn(runtime="acp")`, terminal completion delivery, and Telegram DM/topic behavior; add coalescing or retry-bound handling only if maintainers choose that layer too.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm the non-interactive ACP path and the absence of a general per-channel ACP spawn guard or same-shape completion coalescer on current main.

## Solution Assessment

Unclear as proposed. Per-channel deny/warn is plausible, but maintainers should choose whether the durable fix belongs in channel policy, announcement coalescing, ACP retry behavior, or a combination.
