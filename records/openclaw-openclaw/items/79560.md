## Summary

Keep open: current main and the latest release still lack a per-channel ACP spawn deny/warn policy, and the current terminal task delivery path suppresses duplicate records for the same ACP run but does not coalesce repeated identical failure messages from a stuck child. The issue is valid, but it adds public policy/config behavior and overlaps task notification, ACP permission, and channel routing decisions, so it needs maintainer product direction before a fix lane.

## What This Changes

The issue requests a per-channel ACP spawn policy and/or completion-event coalescing to prevent runaway ACP child failures from flooding non-interactive channels such as Telegram DMs.

## Best Possible Solution

Keep this as the canonical maintainer design issue for non-interactive ACP spawn safety, then choose one core policy shape that combines a pre-runtime deny/warn decision with bounded terminal-event flood control without duplicating broader task-notify or permission-profile surfaces.

## Reproduction Assessment

No high-confidence live Telegram reproduction was run. Source inspection gives a medium-confidence reproduction path for the feature gap: current main lacks a per-channel ACP spawn policy and lacks identical failure-shape coalescing in terminal task delivery.

## Solution Assessment

Unclear as a final implementation. A per-channel deny/warn policy is plausible, but maintainers should decide whether the durable fix belongs in channel ACP spawn policy, task-notify defaults/coalescing, scoped ACP permissions, ACPX retry behavior, or a combined minimal contract.
