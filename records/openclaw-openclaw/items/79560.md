## Summary

Keep open: current main still has only ACP enable/allowlist controls, thread-bound spawn gating, and bounded announce-delivery retries; it does not have a general per-channel ACP deny/warn policy or same-shape completion-event coalescing for noisy child sessions. The remaining work needs maintainer choice on the policy layer and defaults.

## What This Changes

The issue requests channel-aware controls to prevent ACP children spawned from non-interactive channels such as Telegram DM from flooding the parent chat with repeated failure announcements.

## Best Possible Solution

Define a small channel-aware ACP safety contract, then implement the chosen guard with focused coverage around `sessions_spawn(runtime="acp")`, announcement delivery, and Telegram DM/topic behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source and docs verify the non-interactive ACP path and missing per-channel ACP policy/coalescer, but the reported retry flood volume was not directly reproduced.

## Solution Assessment

Unclear as proposed. Per-channel deny/warn is plausible, but maintainers should choose whether the durable fix belongs in channel policy, announcement coalescing, ACP retry handling, or a combination.
