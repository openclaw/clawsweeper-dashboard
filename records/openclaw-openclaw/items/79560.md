## Summary

Keep open: current main has ACP enablement, tool-policy, thread-bound spawn gates, and announcement queue drop policy, but it still does not provide the requested general per-channel ACP deny/warn policy or identical completion-event coalescing for noisy child sessions.

## What This Changes

Request a channel-aware ACP spawn policy and/or completion coalescing to prevent Telegram DM sessions from flooding chats with repeated ACP permission-failure completions.

## Best Possible Solution

Define a small channel-aware ACP safety contract, then implement the chosen guard with focused coverage around sessions_spawn(runtime="acp"), announcement delivery, and Telegram DM/topic behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source and docs confirm the relevant non-interactive ACP path and completion-announcement queue, but the exact Telegram/acpx retry flood was not exercised against current main.

## Solution Assessment

Unclear as proposed: per-channel deny/warn is plausible, but maintainers should choose whether the durable fix belongs in channel policy, announcement coalescing, acpx retry handling, or a combination.
