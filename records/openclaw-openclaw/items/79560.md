## Summary

Keep open: current main still has ACP enable/allowlist/concurrency controls plus thread-binding spawn controls, but not a general per-channel ACP deny/warn policy or same-child completion-event coalescing for noisy non-interactive ACP sessions.

## What This Changes

Request a channel-aware ACP spawn policy, especially for Telegram DMs, to prevent non-interactive `sessions_spawn(runtime="acp")` children from flooding parent chats after permission-denial loops.

## Best Possible Solution

Define a small channel-aware ACP safety contract, then implement the chosen guard and any coalescing/retry behavior with focused coverage for `sessions_spawn(runtime="acp")`, announcement delivery, and Telegram DM/topic behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection confirms the relevant non-interactive ACP path and the absence of the requested channel-level ACP guard or identical completion coalescer on current main.

## Solution Assessment

Unclear as proposed: per-channel deny/warn is plausible, but maintainers should decide whether the durable fix belongs in channel policy, announcement coalescing, acpx retry handling, or a combination.
