## Summary

Keep open: current main still has ACP availability, agent allowlists, sandbox inheritance checks, and thread-bound `spawnSessions` gates, but not a general per-channel ACP deny/warn policy or same-shape completion coalescing for noisy ACP child failures. The related pluggable execution-backends issue is broader and open, so it does not supersede this channel-safety request.

## What This Changes

Request a channel-aware ACP spawn policy and/or completion coalescing so non-interactive channels such as Telegram DMs cannot flood chat with repeated ACP permission-denial completions.

## Best Possible Solution

Define a small channel-aware ACP safety contract, then implement the chosen guard and any completion coalescing with focused coverage for `sessions_spawn(runtime="acp")`, Telegram DM/topic routing, and task completion delivery.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm that ACP spawns can proceed from channel context without a general per-channel ACP policy, while the exact retry/flood behavior remains report-backed.

## Solution Assessment

Unclear as proposed: a per-channel deny/warn policy is plausible, but maintainers should choose whether the durable fix belongs in channel policy, completion coalescing, acpx retry handling, or a combination.
