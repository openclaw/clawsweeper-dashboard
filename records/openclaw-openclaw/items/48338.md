## Summary

Keep open: current main still gives non-Azure custom/OpenAI-compatible provider models a hard-min fallback context window without setup-time metadata prompts or fallback visibility, and the matching #43493 implementation candidate is still open rather than merged.

## What This Changes

Improve custom-provider setup and runtime UX so users can see, set, or be warned about the effective contextWindow/maxTokens instead of silently inheriting a small fallback.

## Best Possible Solution

Review and land #43493 or an equivalent setup-focused change, then split auto-detection and runtime overflow copy into explicit follow-ups if maintainers want those beyond setup metadata visibility.

## Reproduction Assessment

Yes. Source inspection shows the custom-provider configure path saves non-Azure models with DEFAULT_CONTEXT_WINDOW and DEFAULT_MAX_TOKENS without collecting contextWindow/maxTokens, and the colocated test asserts the hard-min fallback.

## Solution Assessment

Yes for the near-term direction: the maintainable fix belongs in the custom-provider setup/configuration surface, because raising the fallback alone would still hide whether metadata was inferred, defaulted, or manually set.
