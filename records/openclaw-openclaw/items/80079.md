## Summary

Keep open: current main still has source-level evidence for two reported failure modes, including ACP spawn patching the gateway session before ACP runtime metadata is initialized and the ACP control plane rejecting output-bearing streams that end without a terminal `done` event.

## What This Changes

The issue reports Codex ACP Discord thread sessions failing first-turn metadata setup or keeping spawned child replies in the ACP transcript instead of posting them to Discord.

## Best Possible Solution

Fix the ACP owner path so metadata is persisted before any first-turn dispatch can observe the child session, output-bearing ACP stream closure is handled without losing the reply, and inline Discord thread delivery remains covered by focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the core bug: current main still patches the child session before ACP initialization and explicitly rejects a stream that emits output but closes without `done`. I did not live-run the Discord WSL2 scenario in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow ACP control-plane/spawn repair with regression tests, not a new Discord feature or config knob. The owner boundary appears to be the ACP spawn and turn-completion path, with Discord thread delivery tests proving the expected route.
