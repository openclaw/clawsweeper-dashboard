## Summary

Keep open: current main still matches the reported failure modes. The ACP spawn path patches the child session before ACP runtime metadata is initialized, and the ACP manager still rejects an output-bearing stream that ends without a terminal `done`, while the docs define thread-bound ACP output as returning to the same thread.

## What This Changes

The issue reports that Codex ACP Discord thread-bound spawns can start without persisted ACP metadata or finish in the ACP transcript without posting the child reply back to the originating Discord thread.

## Best Possible Solution

Fix the ACP owner path so spawned ACP metadata is persisted before any first-turn dispatch can observe the child session, output-bearing ACP stream closure preserves delivery instead of failing, and Discord thread-bound inline delivery is covered by focused regression and live proof.

## Reproduction Assessment

Yes for the core regression path: current source still performs the spawn patch before ACP initialization and still rejects an output-bearing stream that closes without `done`. I did not independently reproduce the full live Discord/WSL2 scenario.

## Solution Assessment

Yes: the narrow maintainable fix belongs in ACP spawn/control-plane behavior plus targeted dispatch coverage. This should not require a new channel feature, config flag, or plugin API change.
