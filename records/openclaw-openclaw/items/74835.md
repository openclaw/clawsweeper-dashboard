## Summary

Keep open. Current main still only supports model-provider request proxy and `allowPrivateNetwork` settings under each `models.providers.<id>.request` block, while the issue asks for a new global/default inheritance layer that is not present in schema, runtime resolution, docs, or tests. The request is valid, but it changes proxy and SSRF policy semantics, so it needs maintainer/security design before implementation.

## What This Changes

The issue requests a global/default model-provider request policy so proxy and `allowPrivateNetwork` settings can be configured once, inherited by remote model providers, and overridden per provider.

## Best Possible Solution

Design a models-scoped provider request defaults layer that merges before `models.providers.<id>.request`, preserves per-provider overrides, defines explicit behavior for local/self-hosted providers, appears in effective config inspection, and ships with schema, redaction, runtime, docs, and SSRF threat-model coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Configure one of the proposed global/default request-policy paths and omit `models.providers.<id>.request`; current strict schema has no accepted defaults field, and the runtime only reads the selected provider's request block.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A global defaults layer is a plausible narrow solution to the repeated-provider workaround, but the exact config shape and private-network inheritance semantics need maintainer/security design before implementation is the best fix.
