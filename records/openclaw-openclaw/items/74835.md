## Summary

Keep open. Current main still only supports provider request proxy/private-network policy on individual model providers, plus loopback auto-allow; the shipped per-provider opt-in and related Fake-IP reports are useful evidence, but they do not implement global/default inheritance.

## What This Changes

Request a global/default model-provider request policy so explicit proxy and allowPrivateNetwork settings can be inherited by model providers while preserving per-provider overrides.

## Best Possible Solution

Design a models-scoped provider request defaults layer that merges before models.providers.<id>.request, preserves per-provider opt-out and field-level overrides, treats local/self-hosted providers explicitly, appears in effective config inspection, and ships with schema/docs/redaction/runtime coverage plus SSRF threat-model guidance.

## Reproduction Assessment

Yes. Configure one of the proposed global/default paths and omit models.providers.<id>.request; current schema has no accepted default field, and model resolution only reads the selected provider's request block.

## Solution Assessment

Unclear. A models-scoped defaults layer is a plausible narrow solution, but the config shape and private-network inheritance semantics need maintainer/security design before implementation is the best fix.

## Risks / Open Questions

- The public config shape and inheritance precedence are unresolved, including how per-provider overrides disable or partially override a default request policy.
- A global allowPrivateNetwork default could widen SSRF exceptions across remote/custom providers unless scoped, documented, and visible in effective config inspection.
- Local and self-hosted providers need explicit treatment so a remote-provider fleet default does not accidentally change local-provider behavior or vice versa.
- Any new default request path must preserve SecretInput handling, redaction, schema docs, generated schema, and config inspection semantics for headers/auth/TLS/proxy fields.
