## Label Justifications

- `P1`: Multiple user reports show duplicated outbound messages and sessions remaining unusable until restart or manual rotation.
- `impact:session-state`: The affected session can keep a stuck or oversized history state that re-enters the failure after restart.
- `impact:message-loss`: The retry path can duplicate already-delivered text or media and block later replies while the lane is wedged.
- `impact:crash-loop`: The gateway can remain alive while the session repeatedly times out and requires operator restart or rotation.

## Summary

Keep open: current main has partial mitigations for compaction timeout cleanup and unresolved message-send retries, but the core bug remains because a compaction replay can still repeat a previously successful outbound side effect and the poisoned-session recovery semantics are not defined.

## What This Changes

The issue asks for compaction timeout/retry handling that does not wedge sessions or resend completed outbound messages.

## Best Possible Solution

Persist a logical outbound side-effect identity across compaction retries and define a terminal recovery path for oversized sessions that releases locks without replaying completed tool sends.

## Reproduction Assessment

Yes, at source level: timeout compaction still retries the prompt, while the message-tool tests explicitly permit a fresh key after a successful identical send. I did not run a live current-main Telegram or Feishu reproduction in this read-only review.

## Solution Assessment

No. The current partial mitigations are not the complete solution because unresolved-send retry dedupe does not cover successful side effects replayed after compaction and does not define poisoned-session recovery.
