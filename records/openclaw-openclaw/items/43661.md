## Summary

Keep open. Current main has several adjacent compaction-timeout hardening changes, but the timeout-triggered compaction branch still retries the prompt after successful compaction without checking whether the timed-out attempt already performed user-visible message-tool side effects. The discussion includes cross-channel evidence from Telegram and Feishu, and the related diagnostic/compaction reports do not cover this remaining replay/idempotency gap.

## What This Changes

Fix the embedded-runner path where compaction timeout or timeout-recovery retry can wedge a session and resend already-delivered channel messages.

## Best Possible Solution

Add a replay-safety guard to timeout-triggered compaction recovery: after compaction succeeds, retry only when the timed-out attempt is replay-safe. If message or other mutating tool side effects already committed, return one explicit blocked/error result with `replayInvalid` preserved instead of re-running the prompt. Cover that with a focused regression test using a first attempt that timed out with high prompt usage and committed `message` delivery.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can be built from the existing timeout-compaction harness: make the first attempt timed out with high prompt usage and committed message-tool side effects, make compaction succeed, and observe current main calling `runEmbeddedAttempt` a second time.

## Solution Assessment

No. Current main's unconditional retry after successful timeout compaction is not the safest fix for this issue; the narrow maintainable solution is to use the existing replay metadata/side-effect tracking to block unsafe retries, with durable channel-send idempotency as a possible later defense-in-depth layer.

## Repair Work Prompt

Fix the remaining duplicate-send risk in https://github.com/openclaw/openclaw/issues/43661. In `src/agents/pi-embedded-runner/run.ts`, the timeout-triggered compaction branch currently retries the prompt after successful compaction even when the timed-out attempt already committed user-visible message-tool or other mutating side effects. Use the existing replay metadata/side-effect signals (`attempt.replayMetadata`, `didSendViaMessagingTool`, committed messaging text/media/targets, and related helpers) to retry only replay-safe attempts. If timeout compaction succeeds after side effects, surface one explicit blocked/error result and preserve `replayInvalid`/liveness metadata rather than re-running the prompt. Add a regression test in `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts` where the first attempt times out with high prompt usage and committed `message` delivery, compaction succeeds, and `runEmbeddedAttempt` is not called a second time. Keep the existing retry behavior for replay-safe timeout compactions. Do not change channel-specific Telegram or Feishu send code unless the core guard requires a generic test seam. Changelog is not required unless maintainers want a user-facing reliability note.
