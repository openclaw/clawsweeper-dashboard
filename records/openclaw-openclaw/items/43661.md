## Summary

Keep open. Current main still retries after successful timeout-triggered compaction without checking replay-unsafe side effects, so a first attempt that already delivered a message can be replayed and duplicate outbound sends remain possible.

## What This Changes

The issue reports that compaction timeout recovery can wedge a session and repeatedly resend already-delivered messages, especially through Telegram and later Feishu reports.

## Best Possible Solution

Add replay-safety to the timeout-compaction retry path and deterministic outbound idempotency for replayed message-tool sends, while keeping broader oversized-session rotation and channel-specific cleanup as separate follow-ups.

## Reproduction Assessment

Yes, at source level. A focused mocked runner path can make a high-token timeout attempt record committed message-tool delivery, compact successfully, and still hit the unconditional second embedded attempt.

## Solution Assessment

No, current main is not the best solution because it relies on replaying the prompt after compaction even when the prior attempt recorded side effects. The narrow maintainable fix is to reuse existing replay metadata before retrying and add stable outbound idempotency where the message send can be replayed.
