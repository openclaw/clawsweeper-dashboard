## Summary

Keep open. Current main has partial replay/liveness and delivery-recovery guards, but the timeout-compaction retry path can still retry a prompt after prior outbound side effects, and the issue discussion includes current channel reports beyond Telegram.

## What This Changes

Prevent compaction timeout/retry recovery from wedging sessions or resending already-delivered outbound messages.

## Best Possible Solution

Define one replay-safe contract for compaction recovery: after side-effecting tools or outbound sends, either reuse a stable run/tool idempotency key through transport delivery or stop the retry and surface one terminal error with focused regression coverage.

## Reproduction Assessment

No, not as a high-confidence live reproduction in this review. Source inspection and multiple field reports line up: timeout compaction retries the prompt after side effects, but current main does not show a durable per-send idempotency guard for that replay.

## Solution Assessment

Unclear. The current partial guards are useful, but maintainers still need to choose whether the narrow fix is to block replay after side effects, propagate stable message-tool idempotency through retries, or both.
