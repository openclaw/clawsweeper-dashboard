## Summary

Keep open: current main still has a compaction retry path that can replay a prior visible send, and the merged message-tool idempotency mitigation only reuses keys for unresolved failed sends, not successful sends that are replayed after timeout recovery.

## What This Changes

The issue reports timeout/compaction recovery wedging a session and duplicating outbound channel messages when a retried run re-executes a prior send.

## Best Possible Solution

Carry a durable per-logical-message idempotency identity across compaction and timeout retries, then prove it prevents replayed Telegram/Feishu sends while preserving legitimate repeated sends.

## Reproduction Assessment

Yes, source-reproducible: current main still retries the run after timeout compaction and current tests prove successful identical sends intentionally receive fresh autogenerated idempotency keys.

## Solution Assessment

Unclear until maintainers choose the retry contract; the best fix should use a durable logical-send identity rather than broad content dedupe or queue-only retries.
