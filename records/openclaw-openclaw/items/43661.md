## Summary

Keep open. Current main still has a source-visible timeout-compaction replay/idempotency gap, and the live discussion has multiple field reports across Telegram and Feishu rather than a stale single-host symptom.

## What This Changes

The issue reports that timeout-triggered compaction can leave a session wedged and replay outbound/message-tool sends, causing duplicate deliveries until manual recovery.

## Best Possible Solution

Define one replay-safe timeout-compaction contract that either carries stable outbound idempotency through retries or stops replay after side effects, with stuck-session recovery coverage.

## Reproduction Assessment

No high-confidence live current-main reproduction is available from this review. Source inspection shows the retry-after-timeout-compaction path and cleared per-attempt delivery evidence, and the discussion provides production Telegram and Feishu logs.

## Solution Assessment

Unclear. The current retry path is not the best final behavior, but maintainers need to choose whether to block replay after side effects, propagate stable outbound idempotency, or combine both.
