## Summary

Keep open. Current main still retries after successful timeout-triggered compaction without checking whether the timed-out attempt already committed replay-unsafe side effects, matching the Telegram and Feishu duplicate-send reports.

## What This Changes

The issue asks for compaction timeouts to recover or fail cleanly without hanging sessions or replaying already-sent outbound messages.

## Best Possible Solution

Gate timeout-compaction retries on existing replay metadata: retry only replay-safe attempts, otherwise return one blocked replay-invalid result and leave broader poisoned-session recovery to a separate follow-up.

## Reproduction Assessment

Yes, at source level. Mock a high-token timed-out attempt with committed messaging side effects, make timeout compaction succeed, and current main will still call `runEmbeddedAttempt` a second time.

## Solution Assessment

No for current main. The narrow maintainable fix is to reuse existing replay metadata before the successful timeout-compaction `continue`, not add channel-specific dedupe or a new config knob.
