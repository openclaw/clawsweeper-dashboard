## Summary

Keep open. Current main has several partial mitigations, but the central failure is still source-visible: timeout compaction can replay the prompt after success, and the direct core outbound path does not yet carry a stable transport-level idempotency identity for message-tool sends.

## What This Changes

The issue reports compaction timeout and retry behavior that can wedge sessions and replay outbound channel messages, causing duplicate sends and manual recovery.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: stop prompt replay after external side effects or carry a stable outbound operation identity through direct, gateway, and plugin sends, with a separate recovery path for oversized or lock-wedged sessions.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and direct core message sends do not yet carry the generated message-tool idempotency key into transport delivery.

## Solution Assessment

Unclear as a final fix. The maintainable repair should define whether replay stops after side effects or whether outbound sends get durable operation identities across all delivery paths; a narrow content-cache workaround would not cover the reported lock-wedged session recovery.
