## Summary

Keep open. Current main still has an unconditional retry after successful timeout-triggered compaction even when the first attempt recorded replay-unsafe message-tool side effects, and the direct message action path does not synthesize a stable transport idempotency key for replayed sends.

## What This Changes

The issue asks for compaction timeout and retry failures to recover or fail once without replaying already-sent outbound messages or wedging the session.

## Best Possible Solution

Add replay safety to the timeout-compaction retry path and stable outbound send idempotency where needed, keeping channel-specific Telegram or Feishu dedupe and oversized-session rotation as separate follow-ups.

## Reproduction Assessment

Yes, at source level. A mocked high-token timed-out attempt that already records message-tool delivery will still compact successfully and hit the unconditional `continue`, causing a second embedded attempt.

## Solution Assessment

No for current main. The narrow maintainable fix is to reuse the existing replay metadata before retrying after timeout compaction, and to avoid relying on channel-specific duplicate suppression.
