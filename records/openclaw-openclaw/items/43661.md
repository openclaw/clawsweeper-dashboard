## Label Justifications

- `P1`: The issue describes production duplicate sends and stuck agent/channel lanes affecting real Telegram and Feishu users.
- `impact:session-state`: The reports center on compaction timeout state, session locks, and oversized session reuse after restart.
- `impact:message-loss`: The failure duplicates already-successful outbound sends and can later fail delivery with channel message-size errors.
- `impact:crash-loop`: Affected lanes can remain unavailable or repeatedly re-wedge until service restart or manual session rotation.

## Summary

Keep open. Current main still contains the source-visible timeout-compaction prompt replay path, and the newer durable delivery/idempotency work is partial rather than proof that duplicate sends and wedged sessions are fixed.

## What This Changes

The issue asks OpenClaw to recover cleanly when compaction times out and to avoid duplicate outbound messages or wedged sessions after retry.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: after an attempt has external side effects, either stop prompt replay or carry a stable outbound operation identity through retries, with a separate recovery path for oversized or lock-wedged sessions.

## Reproduction Assessment

Yes for a source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and focused tests assert that retry path while production comments report duplicate Telegram and Feishu sends.

## Solution Assessment

Unclear as a final fix. The maintainable direction should be a stable outbound idempotency or replay-stop contract rather than a narrow content-match cache.
