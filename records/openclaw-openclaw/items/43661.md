## Summary

Keep open. Current main still has a source-visible replay gap: timeout compaction can retry the prompt after side-effecting message-tool delivery, while the in-memory message-tool evidence used for duplicate suppression is reset across the compaction retry and the transport path does not prove stable outbound idempotency for Telegram/Feishu-style sends.

## What This Changes

The issue reports that compaction timeout recovery can wedge a session and replay outbound channel messages, producing duplicate sends until manual recovery.

## Best Possible Solution

Define one replay-safe compaction recovery contract: after side-effecting tools or outbound sends, either carry a stable idempotency key through transport delivery or stop retrying and surface one terminal error, with focused regression coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection and multiple field reports line up: timeout compaction still retries the prompt after side effects, while the per-attempt message-tool evidence used for suppression is cleared before retry.

## Solution Assessment

Unclear. The likely fix is narrow in code, but maintainers need to choose the reliability contract: block replay after side effects, propagate stable message-tool idempotency through retries, or combine both.
