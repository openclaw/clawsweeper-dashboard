## Summary

Keep open: current main has shipped partial mitigations for failed message-tool retries and compaction timeout handling, but it still intentionally gives successful identical message-tool sends fresh autogenerated idempotency keys, so replay after compaction can still duplicate a visible send.

## What This Changes

The issue reports compaction timeout leaving an agent session wedged and repeatedly replaying duplicate outbound messages, especially on Telegram.

## Best Possible Solution

Carry a durable per-logical-message idempotency identity across compaction and timeout retries, with focused tests for successful replay versus legitimate repeated sends and real channel proof before closing.

## Reproduction Assessment

Yes, at source level: current main still reuses autogenerated message-tool keys only for unresolved failures while tests require fresh keys after a successful identical send. I did not run a live Telegram compaction reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation; the best fix needs a maintainer-approved logical-send identity that survives compaction replay without collapsing legitimate repeated sends.
