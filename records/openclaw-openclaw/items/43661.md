## Summary

Keep open. Current main still retries after successful timeout-triggered compaction without checking replay-unsafe side effects, and the direct outbound send path still lacks a stable transport idempotency key for replayed message-tool sends.

## What This Changes

The issue asks OpenClaw to avoid compaction-timeout retry loops that resend already-delivered outbound messages and leave sessions wedged until manual recovery.

## Best Possible Solution

Add replay safety to timeout-triggered compaction retry and stable outbound send idempotency for message-tool delivery, while leaving channel-specific Telegram/Feishu dedupe and oversized-session rotation as separate follow-ups.

## Reproduction Assessment

Yes, at source level. A timed-out high-token attempt with committed messaging-tool delivery can compact successfully and still hit the unconditional retry path, and the existing timeout-compaction test already asserts the second attempt after compaction.

## Solution Assessment

No for current main. The narrow maintainable fix is to gate timeout-compaction retry on the existing replay metadata and provide stable idempotency for replayed outbound sends rather than relying on channel-specific duplicate suppression.
