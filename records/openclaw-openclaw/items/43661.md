## Summary

Keep open. Current main still has a source-visible timeout-compaction replay gap: successful timeout-triggered compaction retries the prompt, message-tool evidence is per-attempt and cleared for compaction retry, and outbound send calls still mint fresh idempotency keys when no stable key is supplied.

## What This Changes

The issue reports compaction timeout/retry behavior that can wedge a session and resend duplicate outbound messages.

## Best Possible Solution

Define one replay-safe timeout-compaction recovery contract: after outbound/message-tool side effects, either carry stable send identity through retries or stop replaying and surface one terminal failure, with focused stuck-session recovery coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows the retry-after-timeout-compaction path and cleared per-attempt delivery evidence, and the issue discussion provides multiple production reports across Telegram and Feishu.

## Solution Assessment

Unclear. The current retry path is not the best final behavior, but maintainers need to choose the shared reliability contract before implementation: block replay after side effects, propagate stable outbound idempotency, or combine both.
