## Summary

Keep open. Current main has useful replay-invalid/liveness and durable-delivery mitigations, but the central duplicate-send problem is still not clearly fixed: timeout compaction can retry the prompt after a timed-out attempt, and message-tool gateway sends still get a fresh idempotency key unless one is explicitly supplied.

## What This Changes

Report that compaction timeout recovery can wedge a session and replay outbound message sends instead of failing once or recovering idempotently.

## Best Possible Solution

Define a replay-safe outbound send identity that survives timeout compaction retries, persist successful message-tool delivery evidence across retry attempts, and align session-lane recovery so oversized or replay-invalid sessions fail once instead of re-sending.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The source path is still reproducible by inspection: timeout compaction can retry the prompt while message-tool sends lack a stable retry idempotency contract by default.

## Solution Assessment

Unclear. The existing replay-invalid and durable queue changes are useful mitigations, but the best fix should be a deliberate outbound idempotency/replay contract rather than another content-only or per-attempt suppression guard.
