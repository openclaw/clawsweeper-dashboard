## Summary

Keep open: current main has partial compaction-timeout and failed-message retry mitigations, but it still intentionally gives successful identical message-tool sends fresh autogenerated idempotency keys, so compaction replay can still duplicate a visible Telegram send.

## What This Changes

The issue asks OpenClaw to stop compaction timeouts from wedging a session and repeatedly sending the same outbound Telegram message.

## Best Possible Solution

Define and implement a durable logical-send identity across compaction timeout and replay, backed by gateway/channel dedupe and tests that distinguish replayed sends from legitimate repeated sends.

## Reproduction Assessment

Yes, source-level: current main only reuses autogenerated message-tool keys for unresolved failures and tests require a fresh key after a successful identical send; no live Telegram compaction repro was run.

## Solution Assessment

No, current main is only a partial mitigation; the best fix needs maintainer-approved run/replay/send identity semantics and Telegram/channel proof.
