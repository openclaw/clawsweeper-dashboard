## Summary

Keep open. Current main still has a source-visible replay gap: timeout-triggered compaction can retry the prompt after successful compaction, while the per-attempt message-tool delivery evidence used to suppress duplicate visible sends is cleared at the compaction retry boundary.

## What This Changes

The issue reports timeout-compaction retry behavior that can wedge sessions and resend already-delivered outbound messages, especially on Telegram and Feishu.

## Best Possible Solution

Define one replay-safe timeout-compaction recovery contract: after outbound/message-tool side effects, either carry a stable send idempotency key through retries or stop replaying and surface one terminal failure, with focused stuck-session recovery coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows the compaction retry and cleared message-tool evidence path, and the issue discussion provides multiple production reports across Telegram and Feishu.

## Solution Assessment

Unclear. The current retry path is not the best final behavior, but maintainers need to choose the shared reliability contract before implementation: block replay after side effects, propagate stable outbound idempotency, or combine both.
