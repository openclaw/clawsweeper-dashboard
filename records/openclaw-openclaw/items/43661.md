## Label Justifications

- `P1`: Multiple deployments report duplicated outbound messages and sessions remaining unusable until restart or manual session rotation.
- `impact:message-loss`: The retry path can duplicate already-delivered text or media and can leave later replies undelivered while the lane is wedged.
- `impact:session-state`: The affected session can retain a stuck lock or oversized toxic history and re-enter the failure after restart.
- `impact:crash-loop`: The gateway can remain alive while the session repeatedly times out and requires operator restart or rotation to restore availability.

## Summary

Current main improves compaction wait cleanup and adds retry idempotency for unresolved message sends, but it intentionally generates a fresh key after a successful identical send; therefore a compaction replay can still repeat an already-completed outbound side effect, and the poisoned-session evidence remains a distinct unresolved liveness concern.

## What This Changes

Prevent compaction-timeout retries from wedging sessions or repeating outbound messages that already succeeded.

## Best Possible Solution

Persist a logical message-side-effect identity across compaction attempts, return the prior result when that send already committed, and make every timeout path terminate or safely rotate the affected session without replaying completed tools.

## Reproduction Assessment

Yes, at source level with high confidence: timeout compaction retries the prompt, while the message-tool tests explicitly permit a fresh idempotency key after a successful identical send. A live current-main end-to-end reproduction was not run in this read-only review.

## Solution Assessment

No, the current partial mitigations are not the complete solution because unresolved-send retry dedupe does not cover successful side effects replayed after compaction, and it does not define recovery for a poisoned session.
