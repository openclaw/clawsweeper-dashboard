## Label Justifications

- `P1`: Multiple deployments report duplicated outbound messages and sessions remaining unusable until restart or manual session rotation.
- `impact:session-state`: The affected session can retain a stuck lock or oversized history and re-enter the failure after restart.
- `impact:message-loss`: The retry path can duplicate already-delivered text or media and can leave later replies undelivered while the lane is wedged.
- `impact:crash-loop`: The gateway can remain alive while the session repeatedly times out and requires operator restart or rotation to restore availability.

## Summary

Keep open: current main has partial compaction cleanup and message-tool retry mitigations, but it still allows a completed identical message-tool send to get a fresh idempotency key on later replay, and the poisoned-session recovery semantics remain unresolved.

## What This Changes

The issue asks OpenClaw to stop compaction-timeout runs from wedging sessions and replaying already-completed outbound sends.

## Best Possible Solution

Persist a logical message-side-effect identity across compaction attempts, return the prior result when that send already committed, and make every timeout path terminate or safely rotate the affected session without replaying completed tools.

## Reproduction Assessment

Yes, at source level with high confidence: timeout compaction retries the prompt, while message-tool tests explicitly permit a fresh key after a successful identical send. I did not run a live current-main end-to-end reproduction in this read-only review.

## Solution Assessment

No, the current partial mitigations are not the complete solution because unresolved-send retry dedupe does not cover successful side effects replayed after compaction, and it does not define recovery for a poisoned session.
