## Summary

Keep open. Current main has partial compaction-timeout and outbound-recovery mitigations, but the reported duplicate-send path is still not clearly closed because timeout compaction can retry the prompt and message-tool sends do not carry a stable retry idempotency contract across attempts.

## What This Changes

The issue asks OpenClaw to fail or recover cleanly when compaction times out, without wedging the session or repeatedly sending the same outbound message.

## Best Possible Solution

Define a replay-safe outbound message identity that survives compaction and timeout retries, persist message-tool delivery evidence across retry attempts, and keep the session-lane recovery behavior aligned with the related compaction issues.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection still shows the risky current-main path: timeout compaction can retry the prompt after success while message-tool sends lack a stable idempotency contract across retry attempts.

## Solution Assessment

Unclear. Existing delivery-queue reconciliation is a useful partial mitigation, but the maintainable fix should be a deliberate replay/idempotency contract for side-effecting message sends rather than ad hoc content-only suppression.
