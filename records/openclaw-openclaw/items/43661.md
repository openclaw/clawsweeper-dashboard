## Summary

Keep open. Current main has partial recovery for compaction-wait hangs, but the timeout-triggered compaction path still retries the prompt after successful compaction without first stopping replay when the previous attempt already sent via the message tool, which matches the Telegram and Feishu duplicate-send reports.

## What This Changes

The issue reports compaction timeout or retry paths leaving sessions wedged and replaying outbound message sends, producing duplicate Telegram/Feishu replies until manual recovery.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: after an attempt has external side effects, either do not replay the prompt or carry stable outbound idempotency through retries, with a separate recovery path for oversized wedged sessions.

## Reproduction Assessment

Yes for a source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and tests assert that retry path, matching the production Telegram and Feishu logs.

## Solution Assessment

Unclear. The content-cache workaround from the discussion is plausible, but the maintainable fix should be a stable outbound idempotency or replay-stop contract rather than a narrow content-match dedupe.
