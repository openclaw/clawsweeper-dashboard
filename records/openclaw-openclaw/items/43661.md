## Summary

Keep open. Current main still retries the prompt after successful timeout-triggered compaction without checking whether the timed-out attempt already committed message-tool or other mutating side effects, matching the cross-channel duplicate-delivery reports.

## What This Changes

Make compaction-timeout recovery fail or recover once without replaying successful outbound message side effects or leaving the session wedged.

## Best Possible Solution

Gate timeout-triggered compaction retries on existing replay metadata: retry only replay-safe attempts, otherwise surface one blocked result with replayInvalid/liveness state preserved and leave broader session-rotation policy to a separate follow-up if still needed.

## Reproduction Assessment

Yes, at source level. A focused unit reproduction can mock a first high-token timeout with committed messaging side effects, make timeout compaction succeed, and observe current main calling runEmbeddedAttempt a second time.

## Solution Assessment

No for current main. The unconditional retry is not the safest fix; the narrow maintainable solution is to reuse the existing replay metadata and block unsafe retries instead of adding channel-specific dedupe or a new config knob.

## Repair Work Prompt

Fix the timeout-triggered compaction replay bug for https://github.com/openclaw/openclaw/issues/43661. In `src/agents/pi-embedded-runner/run.ts`, when a high-token timeout triggers compaction and `contextEngine.compact()` succeeds, retry the prompt only if the timed-out attempt is replay-safe. If `attempt.replayMetadata.hadPotentialSideEffects` is true, replay metadata is missing/unsafe, or committed messaging/cron/mutating evidence makes the attempt unsafe, do not call `runEmbeddedAttempt` again; return a single blocked/error result that preserves `replayInvalid` and an appropriate liveness state. Add focused coverage in `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts` for a timed-out first attempt with committed message-tool text/media/target evidence where compaction succeeds but no second prompt attempt occurs. Reuse existing replay-state/incomplete-turn helpers; do not add Telegram/Feishu-specific dedupe, a new config option, or broad session-rotation policy. Add a one-line `CHANGELOG.md` fix entry because this is user-facing reliability behavior.
