## Summary

Keep open: current main and v2026.6.8 still only dedupe channel sends when the same idempotency key is reused, while successful identical message-tool sends intentionally receive fresh keys; related compaction work may reduce the timeout loop but does not settle durable Telegram send replay identity.

## What This Changes

The issue requests clean recovery from compaction timeout so a Telegram session does not hang or repeatedly resend the same outbound message.

## Best Possible Solution

Define a maintainer-approved logical-send identity across compaction timeout and replay, then implement it through the message-tool/gateway/channel delivery path with focused regression tests and Telegram proof while leaving partial compaction-progress repair to the linked compaction work.

## Reproduction Assessment

Yes, source-level but not live: the issue logs show repeated compaction timeouts and Telegram sends, and current main still gives successful identical message-tool sends fresh idempotency keys that can bypass gateway dedupe on replay.

## Solution Assessment

No, current main is only a partial mitigation. The best fix needs maintainer-approved logical-send and replay semantics before changing delivery dedupe broadly.
