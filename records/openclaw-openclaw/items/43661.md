## Summary

Keep open. Current main still has a source-visible timeout-compaction prompt replay path, and the newer message-tool idempotency work does not yet prove durable dedupe for core direct sends or recovery for the wedged oversized-session state reported by multiple operators.

## What This Changes

The issue reports that compaction timeouts can leave a session wedged while retrying or replaying outbound sends, causing duplicate Telegram and Feishu deliveries until manual restart or session rotation.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: stop prompt replay after external side effects or carry a stable outbound operation identity through retries, with a separate recovery path for oversized or lock-wedged sessions.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and focused tests assert that path while production reports show duplicate Telegram and Feishu sends.

## Solution Assessment

Unclear as a final fix. The discussion's content-cache workaround is plausible, but the maintainable solution should be a stable outbound idempotency or replay-stop contract rather than narrow content matching.
