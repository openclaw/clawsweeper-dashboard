## Label Justifications

- `impact:session-state`: The failure is rooted in compaction, replay, and session recovery state drifting after a timeout.

- `impact:crash-loop`: Reports describe lanes that remain hung or immediately re-poison after restart, creating an availability loop.

## Summary

Keep open: current main has partial timeout-compaction and message-tool idempotency mitigations, but the central failure can still replay a successful visible send after timeout compaction because successful identical `message` sends intentionally receive fresh autogenerated keys.

## What This Changes

The issue asks OpenClaw to recover cleanly when compaction times out and to prevent repeated duplicate outbound sends during session retry.

## Best Possible Solution

Carry a durable per-logical-message identity across compaction and timeout retries, and prove it prevents replayed successful sends without collapsing legitimate repeated or parallel sends.

## Reproduction Assessment

Yes, at source level: current main retries after successful timeout compaction, and message-tool tests assert successful identical sends receive different autogenerated idempotency keys. I did not run a live Telegram or Feishu reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation. The best fix needs a maintainer-chosen logical-send identity and recovery contract rather than extending failed-send retry state blindly.
