## Summary

Keep open: current main and v2026.6.6 still only partially mitigate the report. Timeout compaction can still retry the prompt after prior side effects, while successful replayed `message` sends intentionally receive fresh autogenerated idempotency keys, so the duplicate-send and wedged-session concern remains real.

## What This Changes

The issue reports that compaction timeout/retry paths can wedge a session and replay outbound channel sends, producing duplicate visible messages.

## Best Possible Solution

Define a canonical recovery contract that preserves a stable logical outbound-send identity across compaction/timeout retries and cleanly terminates or recovers compaction timeout state without parallel fallback state.

## Reproduction Assessment

Yes, source-level: current main retries after timeout compaction and current tests assert successful identical `message` sends get fresh autogenerated idempotency keys. I did not run a live Telegram or Feishu repro in this read-only pass.

## Solution Assessment

No, current main is only a partial mitigation. The safer fix needs one maintainer-approved logical-send identity and compaction recovery contract instead of only failed-send retry dedupe or diagnostics.
