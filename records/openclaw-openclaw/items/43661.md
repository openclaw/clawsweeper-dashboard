## Summary

Keep open: current main has partial compaction timeout cleanup and unresolved-send idempotency, but it still retries prompts after timeout compaction and still permits a fresh message-tool key after a successful identical send, leaving the central duplicate-send and poisoned-session recovery problem unresolved.

## What This Changes

A large-session compaction timeout can wedge a session and replay already-delivered message-tool sends, causing duplicate Telegram/Feishu deliveries and manual recovery.

## Best Possible Solution

Persist a durable logical message side-effect identity across compaction attempts, return the prior committed result instead of resending, and make compaction-timeout sessions terminate, recover, or rotate through one explicit session-health path.

## Reproduction Assessment

Yes, at source level: timeout compaction retries the prompt, and message-tool tests explicitly allow a fresh idempotency key after a successful identical send. I did not run a live current-main Telegram or Feishu reproduction in this read-only review.

## Solution Assessment

No, current main's partial mitigations are not the complete fix because unresolved-send retry dedupe does not cover successful side effects replayed after compaction and does not define poisoned-session recovery.
