## Summary

Keep open: current main has real compaction and recovery mitigations, but successful replayed `message` tool sends still do not share one durable logical-send identity, so the duplicate-send class remains unresolved.

## What This Changes

Prevent compaction timeout and retry paths from wedging sessions or replaying already-sent channel messages.

## Best Possible Solution

Define one shared durable logical-send identity for source-visible channel sends across compaction retries, then enforce it at the message-tool/outbound boundary with focused recovery tests.

## Reproduction Assessment

Yes, at source level: current main can compact and retry a prompt, while successful replayed message-tool sends can receive fresh idempotency keys. I did not run a live long-context Telegram or Feishu repro in this read-only pass.

## Solution Assessment

No, current main is only a partial mitigation. The best fix is a maintainer-approved logical-send identity and recovery contract rather than relying on bounded compaction waits or failed-retry-only keys.
