## Summary

Keep open. Current main still has the source-visible timeout-compaction replay path, and the newer partial message-tool/durable-delivery work does not prove the reported duplicate-send plus wedged-session failure is fixed.

## What This Changes

Session compaction timeouts can leave sessions wedged and replay outbound sends, causing duplicate Telegram or Feishu messages.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: after an attempt has external side effects, stop prompt replay or carry a stable outbound operation identity through retries, with a separate recovery path for oversized or lock-wedged sessions.

## Reproduction Assessment

Yes for a source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and focused tests assert that path while production reports show duplicate Telegram and Feishu sends.

## Solution Assessment

Unclear as a final fix. The discussion's content-cache workaround is plausible, but the maintainable solution should be a stable outbound idempotency or replay-stop contract rather than narrow content matching.
