## Summary

Keep open. Current main still has a source-visible timeout-compaction replay path, and the issue now has multiple production reports plus a recent discussion pointing to the same replay-with-side-effects/idempotency gap rather than a resolved or stale report.

## What This Changes

The issue reports that timeout-triggered compaction can wedge an agent session and replay outbound message sends, producing duplicate Telegram or Feishu messages until manual recovery.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: after an attempt has external side effects, either stop prompt replay or carry a stable outbound operation identity through retries, with a separate recovery path for oversized wedged sessions.

## Reproduction Assessment

Yes for a source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and focused tests assert that retry path while production reports show duplicate Telegram and Feishu sends.

## Solution Assessment

Unclear. The discussion's content-cache workaround is plausible, but the maintainable fix should be a stable outbound idempotency or replay-stop contract rather than narrow content matching.
