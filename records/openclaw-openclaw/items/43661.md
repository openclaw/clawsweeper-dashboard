## Summary

Keep open. Current main still has a source-visible timeout-compaction replay gap: a successful timeout-triggered compaction retries the prompt, while message-tool delivery evidence is per-attempt and outbound sends still fall back to fresh random idempotency keys.

## What This Changes

The issue reports compaction timeouts leaving sessions wedged and replaying outbound message sends across retries.

## Best Possible Solution

Define one replay-safe timeout-compaction recovery contract: after outbound/message-tool side effects, either carry stable send idempotency through retries or stop replaying and surface one terminal failure, with focused stuck-session recovery coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows the retry-after-compaction path and cleared per-attempt delivery evidence, and the issue discussion provides multiple production reports across Telegram and Feishu.

## Solution Assessment

Unclear. The current retry path is not the best final behavior, but maintainers need to choose the shared reliability contract before implementation: block replay after side effects, propagate stable outbound idempotency, or combine both.
