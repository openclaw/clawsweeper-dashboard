## Summary

Keep open: current main has real mitigations for compaction timeout cleanup and message-tool timeout retries, but the central duplicate-send problem is not fully implemented because successful replayed logical sends can still get fresh delivery identities and outbound queue entries are not keyed by a stable logical send id.

## What This Changes

The issue requests that compaction timeouts fail or recover cleanly without wedging the session or replaying already-sent channel messages.

## Best Possible Solution

Define one durable logical-send identity and recovery contract for source-visible channel sends across compaction retries, then implement it at the shared outbound/message-tool boundary with focused channel and recovery tests.

## Reproduction Assessment

Yes, at source level: current main still has compaction retry/replay paths, tests only prove idempotency reuse for unresolved failed message-tool retries, and outbound queue entries use fresh UUIDs. I did not run a live long-context Telegram or Feishu repro in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation. The best fix is a maintainer-approved logical-send identity and poisoned-session recovery policy rather than relying on bounded compaction wait or failed-retry-only message-tool keys.
