## Summary

Keep open. Current main still has the source-visible timeout-compaction replay gap: a successful timeout compaction retries the prompt, while per-attempt messaging-tool evidence is cleared and no stable outbound idempotency contract is proven for the replayed Telegram/Feishu-style send path.

## What This Changes

The issue requests that compaction timeout recovery stop hanging sessions and stop replaying duplicate outbound message sends after side effects.

## Best Possible Solution

Define one replay-safe timeout-compaction recovery contract: after outbound/message-tool side effects, either carry a stable idempotency key through transport delivery or stop replaying and surface one terminal failure, with focused stuck-session recovery coverage.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection and multiple field reports line up: timeout compaction still retries after compaction while clearing per-attempt messaging evidence used for duplicate suppression.

## Solution Assessment

Unclear. The current retry path is not the best final behavior, but maintainers need to choose the shared reliability contract before implementation: block replay after side effects, propagate stable outbound idempotency through retries, or combine both.
