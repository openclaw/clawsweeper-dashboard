## Summary

Keep open. Current main has partial compaction-timeout recovery and side-effect metadata, but a successful timeout-triggered compaction still retries the prompt, and direct message-tool sends do not carry a stable idempotency key through that replay; the current tests and production reports still match the duplicate-send/stuck-session failure.

## What This Changes

This issue requests that timeout-triggered compaction avoid wedging sessions and replaying already-delivered outbound messages.

## Best Possible Solution

Define one replay-safe timeout-compaction contract: after an attempt has external side effects, either stop prompt replay or carry stable outbound idempotency through retries, with a separate recovery path for oversized wedged sessions.

## Reproduction Assessment

Yes for a source-level reproduction, not for a live end-to-end replay. Current main still retries the prompt after successful timeout compaction, and tests assert that retry path while production logs show duplicate Telegram and Feishu sends.

## Solution Assessment

Unclear. The content-cache workaround from the discussion is plausible, but the maintainable fix should be a stable outbound idempotency or replay-stop contract rather than narrow content-match dedupe.
