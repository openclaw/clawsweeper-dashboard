## Summary

Keep open: current main already has the requested gated `evaluate` action, but the `click` contract still has no opt-in `force` field or forwarding path. The FSB comments are useful prior art for making forced clicks explicit and auditable rather than a default interaction mode.

## What This Changes

The issue requests an opt-in `force` parameter for browser clicks and an `evaluate` action for page JavaScript escape hatches.

## Best Possible Solution

Keep the current gated evaluate implementation, and add explicit opt-in `force?: boolean` click support through the browser action contract, CLI, managed Playwright path, docs/skill guidance, tests, and clear existing-session behavior.

## Reproduction Assessment

Yes, by source inspection: a click payload with `force: true` has no schema, type, normalizer, CLI flag, existing-session handling, or Playwright forwarding path on current main. I did not run a runtime repro because this review is read-only.

## Solution Assessment

Yes for the remaining work: the narrow maintainable solution is an additive opt-in click field that forwards to Playwright while preserving normal actionability checks by default. The separate statement-body evaluate request should stay in its own follow-up.
