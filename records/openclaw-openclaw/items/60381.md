## Summary

Keep open: current main and the latest release already expose gated `act:evaluate`, and https://github.com/openclaw/openclaw/issues/55598 was closed on that basis, but the requested opt-in click `force` field is still absent from the browser action schema, request type, normalizer, CLI/docs, and Playwright forwarding path. The FSB comments are useful prior art for keeping forced clicks explicit and auditable rather than making them a default interaction mode.

## What This Changes

Request to add an opt-in `force` click parameter and expose page JavaScript evaluation through the browser tool.

## Best Possible Solution

Keep the current gated evaluate implementation, add explicit opt-in `force?: boolean` click support through the browser plugin contract and managed Playwright path, and document it as a last-resort fallback with clear existing-session behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a click payload with `force: true` has no schema/type/normalizer field and cannot reach `locator.click()`; no runtime repro was run because this review is read-only.

## Solution Assessment

Yes for the remaining work. The narrow maintainable path is an additive opt-in click field that forwards to Playwright while preserving normal actionability checks by default; statement-style evaluate remains separately tracked by https://github.com/openclaw/openclaw/issues/51393.

## Repair Work Prompt

Implement only explicit opt-in `force?: boolean` support for browser `act` click. Preserve the current default click behavior, keep gated `evaluate` unchanged, and do not implement statement-style evaluate bodies from https://github.com/openclaw/openclaw/issues/51393. Thread `force` through the browser tool schema, `BrowserActRequest`, `normalizeActRequest`, managed Playwright `clickViaPlaywright`, and `locator.click`/`locator.dblclick` options. Add a CLI `openclaw browser click --force` path and update browser docs plus `extensions/browser/skills/browser-automation/SKILL.md` to say normal click remains first choice and force is an explicit fallback after recording why normal click failed. For Chrome MCP existing-session profiles, either implement a tested equivalent or return/document an explicit unsupported response; do not silently emulate with arbitrary JS without maintainer approval. Add focused regression tests for schema/normalization/dispatch/CLI/docs-sensitive behavior and include a maintainer-owned changelog entry if the landing process requires one.
