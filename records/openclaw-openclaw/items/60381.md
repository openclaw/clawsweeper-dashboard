## Summary

Keep open: current main and the latest release already expose the gated `evaluate` action, but the requested opt-in `force` click parameter is still absent from the browser tool schema, normalized action type, CLI, and Playwright click forwarding path. The FSB comments are useful prior art for making forced clicks explicit and auditable rather than a default interaction mode.

## What This Changes

The issue requests an opt-in forced browser click option and a first-class browser evaluate action for page JavaScript.

## Best Possible Solution

Keep the existing gated `evaluate` implementation and add a narrow, documented `force?: boolean` click option that defaults to false, forwards only when explicitly requested, and is covered across schema, normalization, CLI/docs, Playwright execution, and existing-session behavior.

## Reproduction Assessment

Yes by source inspection: current main accepts and executes `evaluate`, but the click schema, normalizer, CLI, and Playwright forwarding path do not carry a `force` property. I did not run a live browser repro because this read-only review could prove the remaining gap from source and release inspection.

## Solution Assessment

Yes for the remaining work: a narrow opt-in click parameter in the bundled browser plugin is the maintainable path, while the evaluate half should be left on the existing gated implementation. The safer shape is explicit `force: true` with guidance/tests, not changing the default click path.
