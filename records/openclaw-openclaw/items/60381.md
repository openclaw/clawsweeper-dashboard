## Summary

Keep open: current main implements the gated `evaluate` action, but the requested opt-in `force` click parameter is still missing from the browser tool schema, normalized action type, CLI, and Playwright click forwarding path. The recent FSB comments are useful prior art for making forced clicks explicit and auditable instead of a default interaction mode.

## What This Changes

Request to add an opt-in browser click `force` parameter and expose a page JavaScript `evaluate` action for browser automation escape hatches.

## Best Possible Solution

Ship an additive, opt-in `force?: boolean` click path that preserves normal click behavior by default, keeps the existing gated evaluate implementation, and documents force as an auditable fallback.

## Reproduction Assessment

Yes, by source inspection: a click payload with `force: true` has no schema, normalized request field, CLI flag, existing-session behavior, or Playwright forwarding path on current main. I did not run a runtime repro because this review is read-only and the missing contract path is explicit in source.

## Solution Assessment

Yes for the remaining work: the narrow maintainable solution is an additive opt-in click field that forwards to Playwright while preserving normal actionability checks by default. The already-implemented evaluate action should stay gated by `browser.evaluateEnabled`.
