## Summary

Keep open. The config-schema gap is real and this PR is the current narrow tracker for it, but the patch is not ready because it accepts `google-vertex` in config without preserving provider-level `google-vertex` through inline model materialization, and the external draft lacks inspectable real behavior proof.

## What This Changes

The PR adds `google-vertex` to configured model/provider API validation and generated schema output, plus config-schema and Google provider-policy tests for native Vertex configs.

## Best Possible Solution

Refresh the branch to current main, add `google-vertex` to the config API enum and inline transport API allowlist with provider-level regression coverage, use the current schema-generation path, and include redacted real config-validation or routing proof.

## Reproduction Assessment

Yes. Source inspection shows current main rejects `api: "google-vertex"` through `ModelApiSchema`, while the Google runtime only selects native Vertex when that API value survives into the resolved model.

## Solution Assessment

No, not as-is. The enum/schema direction is right, but accepting the value without adding it to `normalizeResolvedTransportApi` still lets provider-level native Vertex configs validate and then lose the API before stream selection.
