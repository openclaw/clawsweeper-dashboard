## Summary

Keep open: the PR targets a real current-main Google Vertex config bug, but the patch is incomplete because the inline provider runtime still drops an inherited provider-level `google-vertex` API before the Google plugin can choose the native Vertex stream. It also lacks inspectable real behavior proof from the contributor's setup.

## What This Changes

The PR adds `google-vertex` to the model API config/schema enum and adds config plus Google provider policy tests for native Vertex configs.

## Best Possible Solution

Update this PR or a narrow replacement so `google-vertex` is accepted by config and preserved through inline provider model resolution, with focused regression coverage and redacted real behavior proof, while coordinating with the broader Google Vertex PR.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure `models.providers.google-vertex.api: "google-vertex"` and let a model inherit the provider API. Current main rejects the API value at schema time, and the PR would still drop inherited provider-level Vertex APIs at runtime.

## Solution Assessment

No. Adding the enum/schema value is necessary, but the narrow maintainable fix also needs `google-vertex` in `normalizeResolvedTransportApi` plus inline provider inheritance coverage; the broader provider/auth direction should stay coordinated with https://github.com/openclaw/openclaw/pull/60860.
