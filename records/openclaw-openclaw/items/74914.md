## Summary

Keep open: this PR targets a real current-main Google Vertex config bug, but the patch is incomplete because provider-level inherited `google-vertex` APIs are still dropped before the Google plugin can select the Vertex transport. It is also an external draft PR with missing real behavior proof and overlap with https://github.com/openclaw/openclaw/pull/60860.

## What This Changes

The PR adds `google-vertex` to model API config/schema validation and adds tests for native Vertex config normalization.

## Best Possible Solution

Update this PR or a narrow replacement so config accepts `google-vertex`, inline provider resolution preserves inherited native Vertex APIs, current schema generation stays aligned, and focused regression coverage plus redacted real behavior proof are provided.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure `models.providers.google-vertex.api: "google-vertex"` with a model inheriting the provider API. Current main rejects the API value at schema time, and the submitted patch still leaves inherited provider-level Vertex APIs dropped at runtime.

## Solution Assessment

No, not as submitted. Adding the enum/schema value is necessary, but the narrow maintainable fix also needs `google-vertex` in `normalizeResolvedTransportApi`, inline provider inheritance coverage, current schema workflow alignment, and coordination with https://github.com/openclaw/openclaw/pull/60860.
