## Summary

Keep open. Current main still has the wedged restart behavior, and this PR is a serious implementation candidate, but it carries the protected `maintainer` label plus an explicit author request for maintainer discussion around the force-retire lifecycle tradeoff. The diff also still needs a changelog entry and has a concrete security-sensitive concurrency concern before merge.

## What This Changes

The PR adds a health-monitor-only `forceRetireOnTimeout` stop option, lifecycle guards for stale channel callbacks, task-scoped runtime/native approval cleanup hardening, and focused gateway/infra regression tests.

## Best Possible Solution

Keep the PR open for maintainer review. If maintainers accept automatic force-retire recovery, land it only after adding the changelog entry and getting exact-head checks green; otherwise ask for generation fencing, a hard-stop plugin contract, or an explicit opt-in recovery policy before automatic replacement lifecycles are started.

## Reproduction Assessment

Yes. Current main has a deterministic source/test reproduction: a channel task that ignores abort makes `stopChannel` time out, keeps `store.tasks` registered, and causes the following `startChannel` to no-op; #71412 also provides real Telegram sleep/wake context.

## Solution Assessment

Unclear. The PR is a narrow bookkeeping recovery for the reported wedge and it hardens OpenClaw-owned cleanup paths, but it is not clearly the best final fix unless maintainers accept concurrent old/new plugin-owned transport risk; a generation fence or hard-stop contract is safer.

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, workflows, lockfiles, or publishing changes, but it introduces a concurrency-sensitive force-retire path that can duplicate channel-side effects if plugin code ignores abort.

Concerns:

- **[medium] Force-retire can run old and new lifecycles together:** `src/gateway/server-channels.ts:708`
  - body: The force-retire path deletes `store.aborts` and `store.tasks` for a timed-out lifecycle before the old task has actually settled. A fresh channel account can then start while the old plugin-owned websocket or polling loop may still receive messages or dispatch actions, creating duplicate processing risk.
