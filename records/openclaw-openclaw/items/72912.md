## Summary

Keep open: current main still has the restart wedge, and this active maintainer-labeled PR needs explicit maintainer judgment on the force-retire lifecycle/security tradeoff before merge.

## What This Changes

The PR adds a health-monitor-only force-retire stop option, stale-lifecycle guards, task-scoped runtime/native approval cleanup hardening, and gateway/infra regression tests for wedged channel restarts.

## Best Possible Solution

Keep the PR open until maintainers choose the lifecycle policy explicitly: accept the force-retire boundary with documented plugin-owned transport responsibility, or require a generation fence, hard-stop contract, or opt-in recovery policy before automatic replacement lifecycles start.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`, and related issue #71412 provides observed Telegram sleep/wake context for the same stop-timeout bookkeeping wedge.

## Solution Assessment

Unclear. The PR is a focused bookkeeping recovery with useful cleanup hardening, but it is not clearly the best final fix until maintainers accept or mitigate the concurrent old/new plugin-owned transport risk.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Fence side effects before force-retiring:** `src/gateway/server-channels.ts:708-715`
  - body: When health-monitor recovery uses `forceRetireOnTimeout`, this branch deletes the retiring lifecycle from `store.tasks`/`store.aborts` before the underlying `startAccount()` promise has stopped. A plugin-owned websocket or polling loop that ignored abort can still process events while a fresh lifecycle starts, so automatic force-retire needs a generation fence, hard-stop contract, or explicit opt-in policy before merge.
  - confidence: 0.82
- **[P3] Add the gateway recovery changelog entry:** `src/gateway/channel-health-monitor.ts:166-168`
  - body: This changes user-facing gateway recovery behavior, but the PR does not add an Unreleased `CHANGELOG.md` entry. Repo policy requires release-note coverage for user-facing fixes, so this reliability fix would ship without a changelog line.
  - confidence: 0.9
