## Summary

Keep this PR open. Current main still has the wedged restart behavior the PR targets, but the PR carries a protected `maintainer` label, the author explicitly asked for maintainer discussion before merge, and the force-retire design leaves a real old/new lifecycle concurrency tradeoff that should be accepted or rejected by maintainers rather than swept closed or automated.

## What This Changes

This PR adds a health-monitor-only forced channel lifecycle retirement path, gates stale lifecycle status/cleanup/restart callbacks, hardens task-scoped runtime and native exec approval cleanup, and adds focused gateway/infra regression tests.

## Best Possible Solution

Keep the PR open for maintainer review. If maintainers accept the availability-over-fail-closed tradeoff, land it after adding the required changelog entry and getting exact-head checks green; otherwise request a design that adds generation fencing, a hard-stop plugin contract, or an explicit opt-in recovery policy before automatic force-retire restarts are enabled.

## Reproduction Assessment

Yes. Current main has a deterministic source/test reproduction: a channel task that ignores abort causes `stopChannel` to time out, leaves `store.tasks` registered, and makes the following `startChannel` no-op; related #71412 adds real Telegram sleep/wake reproduction context.

## Solution Assessment

Unclear. The PR is a narrow and well-targeted recovery for OpenClaw-owned bookkeeping, but it may not be the best final solution unless maintainers accept the possibility of concurrent plugin-owned transport side effects; the safer alternative is lifecycle generation fencing or a hard-stop contract before starting the replacement lifecycle.

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, workflows, lockfiles, or publishing changes, but it introduces a concurrency-sensitive force-retire path that can duplicate channel-side effects if a plugin lifecycle ignores abort.

Concerns:

- **[medium] Force-retire can run old and new channel lifecycles together:** `src/gateway/server-channels.ts:708`
  - body: When `forceRetireOnTimeout` is true, the manager removes the old task from bookkeeping before the underlying task has actually stopped. A fresh lifecycle can then start while plugin-owned transport code from the retired lifecycle may still receive messages or dispatch actions if it does not honor abort, creating duplicate processing risk.
  - confidence: 0.8
