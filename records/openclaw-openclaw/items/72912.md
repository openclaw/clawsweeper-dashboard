## Summary

Keep open. The PR fixes a real gateway recovery bug, but it has the protected `maintainer` label and intentionally leaves a core lifecycle policy/security tradeoff for maintainer discussion: health-monitor recovery can start a replacement channel while plugin-owned transport code from the retired lifecycle may still run.

## What This Changes

The PR adds a health-monitor-only forced-retirement option for channel stop timeouts, gates stale lifecycle callbacks, cleans up task-scoped runtime/native approval surfaces, adds regression tests, and records the fix in the changelog.

## Best Possible Solution

Maintainers should either accept the health-monitor-only recovery boundary explicitly or add a generation fence, hard-stop contract, or opt-in policy before replacement lifecycles can process events.

## Reproduction Assessment

Yes. Current main has a focused unit reproduction in `src/gateway/server-channels.test.ts:304`, and related #71412 plus the closed duplicate reports #75480 and #75519 describe observed Slack/Telegram wedges after stop timeouts.

## Solution Assessment

Unclear. The PR is a focused repair for the wedge and now cleans up OpenClaw-owned state, but it is not clearly the best final solution until the old/new plugin-owned transport overlap is accepted or fenced.
