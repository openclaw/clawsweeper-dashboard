## Summary

Keep open. The PR targets a real current-main health-monitor restart wedge, but it carries the protected `maintainer` label, the author explicitly asked for maintainer discussion, and the diff still makes an automatic lifecycle policy tradeoff that can overlap old and new plugin-owned transports.

## What This Changes

The PR adds a health-monitor-only force-retire stop option, stale-lifecycle guards, task-scoped runtime/native approval cleanup hardening, and gateway/infra regression tests for wedged channel restarts.

## Best Possible Solution

Keep the PR open until maintainers choose the lifecycle policy explicitly: accept force-retire with the plugin-owned transport boundary documented and validated, or require generation fencing, a hard-stop contract, or an opt-in recovery policy before automatic replacement lifecycles start.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`, and #71412 provides observed Telegram sleep/wake context for the same stop-timeout bookkeeping wedge.

## Solution Assessment

Unclear. The PR is a focused bookkeeping recovery with useful cleanup hardening, but it is not clearly the best final fix until maintainers accept the concurrent old/new plugin-owned transport risk or require a stronger lifecycle fence.
