## Summary

Keep open: this PR targets the valid bug tracked at https://github.com/openclaw/openclaw/issues/58139, but the patch still misses the reported Linux-container-on-Windows diagnostic path and the supplied proof is macOS tests/code analysis rather than real Windows Docker bind-mount behavior.

## What This Changes

The PR wraps memory-lancedb table initialization in a three-attempt retry loop and appends a Docker volume hint when retries fail on win32.

## Best Possible Solution

Keep the fix plugin-scoped, make the diagnostic apply to the Linux-container-on-Windows bind-mount failure path, add focused retry/docs coverage, and require redacted Windows Docker terminal/log proof before merge.

## Reproduction Assessment

Yes, source-reproducible: https://github.com/openclaw/openclaw/issues/58139 gives Windows 11 Docker Desktop bind-mount steps and matching LanceDB metadata errors, and current main still initializes LanceDB tables immediately. I did not live-reproduce the Windows Docker setup in this review.

## Solution Assessment

No, not as written: memory-lancedb is the right boundary, but process.platform is the wrong diagnostic trigger for a Linux container on a Windows host. The fix also still needs real Windows Docker proof.
