## Summary

Keep open. Current main still flattens remote markdown images to alt text, and open PR #41025 uses closing syntax for this exact request, so the issue should remain open until that PR is refreshed, landed, or explicitly superseded.

## What This Changes

Render remote http/https markdown image URLs inline in Control UI chat instead of reducing them to alt text.

## Best Possible Solution

Refresh and review PR #41025 against the current markdown-it renderer, preserving explicit http/https allowlisting and unsafe-scheme regression tests before closing this after merge.

## Reproduction Assessment

Yes from source inspection: current main routes non-data markdown image URLs through the image renderer branch that returns escaped alt text, and tests assert that https images do not produce img tags. I did not run the UI because this was a read-only review and pnpm was unavailable.

## Solution Assessment

Unclear. The narrow renderer allowlist proposed by the linked PR is plausible, but it must be updated for the current markdown-it code path and the privacy impact of remote browser fetches needs maintainer approval.

## Security Review

Status: needs_attention

Summary: The requested behavior would render arbitrary remote images from chat markdown, causing client-side network requests to third-party origins.

Concerns:

- **[medium] Decide remote-image privacy policy before allowing all URLs:** `ui/src/ui/markdown.ts:417`
  - body: Rendering http/https markdown images would let assistant or user content trigger browser fetches to arbitrary remote hosts, exposing normal browser request metadata such as IP address and user agent. The implementation should be an explicit product/security choice, with tests for blocked schemes and any required opt-in or domain policy.
