## Summary

Keep open. Current main still flattens remote Control UI markdown images to alt text, and an open PR with closing syntax already targets the issue; the remaining decision is whether maintainers want to change the shipped no-remote-fetch policy.

## What This Changes

The issue requests that Control UI chat markdown render remote `http`/`https` image URLs inline instead of flattening them to alt text.

## Best Possible Solution

Decide the Control UI remote-media policy first; if approved, update the renderer, CSP, docs, changelog, and tests together through #41025 or a replacement, otherwise keep the current hardening.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `toSanitizedMarkdownHtml("![Alt text](https://example.com/img.png)")` reaches the branch that returns escaped alt text, and the current test asserts no `<img>` output.

## Solution Assessment

Unclear. The suggested renderer-only change is not the best standalone fix because the current CSP, docs, and tests deliberately block remote image fetches; the maintainable solution needs an explicit policy decision.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because allowing remote markdown images would make operator browsers fetch arbitrary third-party URLs from rendered chat content.

  - body: Allowing `http(s)` markdown image URLs inline would let user- or model-supplied chat content make the Control UI browser request third-party hosts, conflicting with the documented no-remote-fetch policy unless maintainers add a reviewed control model.
  - confidence: 0.94

  - body: Current CSP excludes remote image sources; enabling the renderer alone would leave broken images, while broadening `img-src` needs the same remote-media safety decision.
