## Summary

Keep open: the session-memory duplication report is valid and the PR is close, but it is not merge-ready because real behavior proof is still missing and the dedupe state can now suppress a legitimate repeated assistant reply after a filtered user command.

## What This Changes

The PR changes session-memory transcript extraction to collapse adjacent identical assistant text and adds regression coverage for delivery-mirror rows, thinking-cleaned duplicates, distinct replies, and repeated replies across visible user turns.

## Best Possible Solution

Reset the dedupe boundary on any user transcript row before filtering it from the memory summary, keep the narrow reader-level dedupe, and require redacted real setup proof for a thinking-enabled `/new` or `/reset` memory save before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 append every extracted user/assistant text row, so a raw thinking assistant entry followed by a cleaned text-only entry with the same visible text produces duplicate memory lines. I did not establish a live thinking-enabled model reproduction in this read-only review.

## Solution Assessment

No, not as currently written: the hook-level dedupe is a reasonable narrow fix, but the dedupe boundary must also reset on skipped user turns to avoid losing legitimate repeated assistant replies. Storage-layer cleanup can remain a separate follow-up if maintainers want to remove the duplicate transcript entries at the source.
