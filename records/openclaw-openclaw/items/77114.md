## Summary

Keep open: current main and the latest release still only trim WhatsApp document filenames, while this PR adds a narrow sanitizer, focused tests, and convincing live Baileys proof. I found no blocking code or security-review findings.

## What This Changes

The branch strips C0/DEL control characters from WhatsApp document filenames in the shared helper, adds focused regression tests, updates the lint suppression allowlist, and records a changelog note.

## Best Possible Solution

Land the focused WhatsApp helper sanitizer after normal maintainer merge gates, while keeping broader filename hardening scoped to https://github.com/openclaw/openclaw/pull/72973.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.5.12` only trim the supplied filename before it reaches the Baileys document payload, and Baileys accepts an optional document `fileName` without sanitizing a supplied value.

## Solution Assessment

Yes. Centralizing the strip in `resolveWhatsAppDocumentFileName` is the narrowest maintainable fix for WhatsApp document filenames and reuses the helper across normal document, forced-document, and normalized outbound media paths without changing MIME policy.
