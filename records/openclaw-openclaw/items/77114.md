## What This Changes

The PR strips C0/DEL control characters from WhatsApp outbound document `fileName` values in the shared filename helper, adds focused tests, allowlists the lint suppression, and adds a changelog note.

## Best Possible Solution

Land the helper-level sanitizer after acceptable real behavior proof is added, while leaving broader cross-channel filename policy to the separate work at https://github.com/openclaw/openclaw/pull/72973.

## Reproduction Assessment

Yes, by source inspection. Current main only trims the filename helper result before passing it to Baileys document payloads, so an embedded CR/LF filename would preserve those bytes in the payload.

## Solution Assessment

Yes. Centralizing the strip in `resolveWhatsAppDocumentFileName` covers both normal document sends and forced visual-as-document sends without changing MIME, caption, or broader filename policy.
