## Summary

Keep open: the PR is a focused and correct WhatsApp filename security hardening, but the external contributor proof is still unit/mock-only, so it should not merge until real WhatsApp/Baileys behavior proof is added.

## What This Changes

The PR strips C0/DEL control characters in the WhatsApp document filename helper, adds helper regression tests, updates the lint-suppression allowlist, and records a changelog entry.

## Best Possible Solution

Land the helper-level sanitizer after the contributor adds redacted real WhatsApp/Baileys proof, while leaving broader cross-channel filename policy to https://github.com/openclaw/openclaw/pull/72973.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveWhatsAppDocumentFileName` only trims, and `createWebSendApi` sends that helper result to Baileys; a filename such as `evil.pdf\r\nX-Injected: bad` would preserve embedded CR/LF before this PR.

## Solution Assessment

Yes. The PR moves the strip into the shared WhatsApp document filename helper, so direct document sends and forced-document/default paths use one narrow boundary fix without changing MIME, caption, or broader filename policy.
