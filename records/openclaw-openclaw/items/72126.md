## Summary

Keep this PR open. Current main still passes a literal `keychain:<service>:<account>` env fallback into the OpenAI Realtime browser-session and bridge auth paths, and this PR targets the right provider-local boundary, but the proposed synchronous Keychain lookup needs a bounded timeout and the user-facing fix needs a changelog entry before merge.

## What This Changes

The PR adds an OpenAI Realtime API-key resolver/cache for `keychain:<service>:<account>` values in `extensions/openai/realtime-voice-provider.ts` and uses it for browser client-secret sessions and backend voice bridge creation.

## Best Possible Solution

Land a provider-local OpenAI Realtime API-key resolver that resolves `keychain:<service>:<account>` for both browser client-secret sessions and backend bridges, bounds the synchronous Keychain call, caches successful results without making transient failures permanent, covers env/config paths with focused tests, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. On current main, set `OPENAI_API_KEY=keychain:openclaw:OPENAI_API_KEY` and invoke the OpenAI Realtime browser-session or bridge path; the source shows that literal env value reaches the bearer-auth path.

## Solution Assessment

No, not as-is. The provider-local resolver/cache is the right narrow direction, but the unbounded synchronous Keychain call and missing changelog entry need to be fixed before this is the best maintainable solution.
