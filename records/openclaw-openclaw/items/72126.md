## Summary

Keep this PR open. The reported OpenAI Realtime keychain-ref bug still exists on current main, and the PR is aimed at the right provider-local boundary, but it still needs a bounded Keychain lookup, regression coverage, changelog follow-through, and a rebase before it is mergeable.

## What This Changes

The PR adds a provider-local resolver/cache for `keychain:<service>:<account>` OpenAI Realtime API-key refs and routes browser-session and voice-bridge API-key lookup through it.

## Best Possible Solution

Land a provider-local OpenAI Realtime API-key resolver that resolves keychain refs for both browser client-secret sessions and backend bridges, caches successful resolutions, fails closed locally on unresolved refs, bounds `/usr/bin/security`, covers env/config behavior with focused tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. On current main, setting `OPENAI_API_KEY` to `keychain:openclaw:OPENAI_API_KEY` reaches both OpenAI Realtime API-key use sites as a literal bearer value; the linked report includes the resulting OpenAI 401 and a live hot-patch success path.

## Solution Assessment

No, not as-is. The provider-local resolver/cache is the right narrow direction, but the PR needs a bounded Keychain call, regression tests, changelog entry, and rebase before it is the maintainable fix.
