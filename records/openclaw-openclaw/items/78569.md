## Summary

Keep this open: current main still has the reported token-auth Control UI path where a device-less approval is bound to the original WebSocket connId, but the approved host=node follow-up runs through a different backend gateway client and can hit `APPROVAL_CLIENT_MISMATCH`.

## What This Changes

The issue asks for approved `exec(host=node)` commands from Control UI token-auth sessions to run successfully instead of failing after approval with `APPROVAL_CLIENT_MISMATCH`.

## Best Possible Solution

Fix the host=node post-approval handoff so a trusted gateway-originated follow-up can validate the approved record without depending on a transient Control UI connId, while preserving the existing approval binding and replay protections.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and reporter logs: token-auth Control UI without device identity records only a requester connId, then the approved async host=node follow-up uses a separate backend gateway client and the sanitizer rejects the approval id as a client mismatch. I did not run a live Control UI plus paired-node setup in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No for current main; the current connId-only fallback is too narrow for backend post-approval forwarding. The safer fix is a targeted trusted handoff or binding adjustment, not disabling the approval guard.
