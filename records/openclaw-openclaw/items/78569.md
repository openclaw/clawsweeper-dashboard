## Summary

Keep open. Current main still has a source-proven failure path for token-only/no-device node exec approvals: the approval record is bound to one backend connection, then the approved async node invoke runs through a fresh backend connection and can trip APPROVAL_CLIENT_MISMATCH.

## What This Changes

Allow approved Control UI token-auth node exec requests to execute instead of failing with APPROVAL_CLIENT_MISMATCH after approval.

## Best Possible Solution

Add a narrow approval-binding path for token-only/no-device backend node exec approvals that preserves node, decision, command, cwd, env, and device replay protections, with regression coverage for the Control UI token-auth no-device flow.

## Reproduction Assessment

Yes, from source inspection: a no-device approval record can be created with only requestedByConnId, then the approved async node.invoke uses a fresh backend Gateway client whose connId differs. I did not live-run the Docker/node scenario.

## Solution Assessment

Unclear until patched. The safest direction is to preserve the security binding and add explicit token-only/no-device backend handling and regression tests, not to remove the mismatch guard wholesale.
