## Summary

Keep open. Current main still has a source-proven no-device/backend node exec approval path where the approval record is bound to one connection id, then the approved `node.invoke` runs through a fresh backend connection and can hit `APPROVAL_CLIENT_MISMATCH`.

## What This Changes

Fix token-auth/no-device Control UI approvals for node-backed exec so approved non-allowlisted commands run instead of failing with `APPROVAL_CLIENT_MISMATCH`.

## Best Possible Solution

Add a narrow no-device backend node exec approval binding that preserves node, run id, decision, `systemRunPlan`, cwd, env, agent/session, and allow-once replay checks, with regression coverage for the Control UI token-auth/no-device flow.

## Reproduction Assessment

Yes, from source inspection: a no-device approval record can be registered through backend shared-auth with only `requestedByConnId`, then the approved async `node.invoke` uses another backend connection whose `connId` differs. I did not live-run the Docker/paired-node scenario.

## Solution Assessment

Unclear until patched. The safest direction is a targeted binding for token-only/no-device backend node exec approvals, not removing the mismatch guard or weakening replay checks globally.
