## Summary

Keep open: the PR targets a real Telegram `/models` forum-topic callback bug, but the current patch is not merge-ready because its `groupPolicy: "open"` short-circuit can bypass `/models` command authorization for `commands.ownerAllowFrom` setups, and the contributor proof is still mock/test-only.

## What This Changes

The PR adds a Telegram model-callback group-policy precheck and a focused forum-topic `/models` callback regression test.

## Best Possible Solution

Keep `/models` command authorization authoritative while using the effective group/topic policy only to avoid false group-allowlist drops, then require redacted live Telegram or equivalent transport proof for the forum-topic flow.

## Reproduction Assessment

Yes at source level: current main routes `mdl_*` callbacks through command ingress without a per-group open-policy correction, and the issue/PR comments include failing focused-test output plus an independent v2026.6.8 reproduction. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No as submitted: reusing the group-policy evaluator is the right direction, but returning `true` on `groupPolicy: "open"` before proving `/models` command authorization weakens the existing callback auth boundary. The safer fix is to make command ingress owner-aware or otherwise preserve command authorization after resolving the effective group policy.
