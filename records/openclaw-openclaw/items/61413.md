## Summary

Keep this PR open for now: the networking problem is plausible, but the current draft branch lacks real transparent-proxy proof and introduces a trusted-mode SSRF policy bypass plus an error-path throw.

## What This Changes

The branch skips the guarded-fetch dispatcher for `TRUSTED_ENV_PROXY` requests when no proxy env vars are set, and adds tests plus nested cause-chain error diagnostics.

## Best Possible Solution

Keep the diagnostics idea with defensive cause serialization, and redesign transparent-proxy support so hostname/private-IP policy checks run before any default-fetch fallback.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the diff shows trusted-mode policy validation is skipped and the new formatter can throw while handling an already-caught error. The original transparent-proxy symptom itself still lacks live after-fix proof.

## Solution Assessment

No. The proposed default-fetch fallback is broader than the current guard contract allows; the safer path validates target policy first, then narrowly chooses a transparent-proxy-compatible transport and guards cause serialization.

Full review comments:

- **[P1] Preserve trusted-mode policy checks:** `src/infra/net/fetch-guard.ts:283-293`
  - body: This branch lets `TRUSTED_ENV_PROXY` requests reach fetch without `resolvePinnedHostnameWithPolicy()` or an equivalent hostname policy check. Callers that pass host policies, including web tool endpoints, would lose allowlist/private-host enforcement before dispatch.

  - confidence: 0.9
- **[P3] Clear proxy env vars in the no-env test:** `src/infra/net/fetch-guard.ssrf.test.ts:738-746`
  - body: The new test asserts that no dispatcher is attached, but it never clears `HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`, or lowercase variants. Developer or CI environments with proxy env configured will take the env-proxy branch and fail this test spuriously.
  - confidence: 0.86
