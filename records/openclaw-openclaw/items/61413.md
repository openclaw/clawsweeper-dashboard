## Summary

Keep open, but the PR is not merge-ready: the transparent-proxy goal is useful, yet the current diff bypasses trusted-mode SSRF policy checks, lacks after-fix real proxy proof, and introduces smaller error-formatting and test reliability defects.

## What This Changes

The branch changes guarded fetch to use default fetch for trusted env-proxy requests when no proxy env vars are set, and adds nested cause-chain details to tool errors with tests.

## Best Possible Solution

Land transparent-proxy support only behind a maintainer-approved operator opt-in that preserves pre-dispatch SSRF policy checks, redacts unsafe diagnostics, includes focused tests, and has real proxy proof.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of the diff shows trusted no-env policy validation is skipped and the new formatter can throw. No for the original transparent-proxy runtime symptom because the PR body and comments do not include after-fix Proxifier/V2Ray evidence.

## Solution Assessment

No. The proposed default-fetch fallback is broader than the current guard contract; a safer solution would keep policy validation, make transparent-proxy behavior explicit, and guard or redact cause serialization.
