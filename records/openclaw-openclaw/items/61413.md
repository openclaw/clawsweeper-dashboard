## Summary

Keep this PR open. The transparent-proxy diagnosis is useful, but the proposed fetch-guard change is not mergeable because it drops trusted-mode SSRF policy checks; the diagnostics change also needs a guarded serialization path and the new test needs deterministic proxy env setup.

## What This Changes

The PR changes `fetchWithSsrFGuard` trusted env-proxy handling to skip custom dispatchers when no proxy env vars exist, and adds nested cause-chain tool error diagnostics with tests.

## Best Possible Solution

Preserve the diagnostics improvement, but implement transparent-proxy support only after target hostname and SSRF policy validation still runs; cover env proxy, no-env fallback, `NO_PROXY`, and off-allowlist trusted-mode cases.

## Reproduction Assessment

Yes for the blocking review findings: a focused unit test can call `fetchWithSsrFGuard` in `TRUSTED_ENV_PROXY` mode with an off-allowlist policy and observe that the PR reaches `fetchImpl` instead of blocking. The user's OS-level transparent-proxy setup itself was not live-reproduced here.

## Solution Assessment

No. Dropping the custom dispatcher for all trusted no-env requests is too broad because it also drops target policy checks; the safer path is to validate policy first and then narrowly choose the default fetch transport.
