## Summary

Keep open. The transparent-proxy problem is plausible and current main has not implemented the PR’s requested no-dispatcher fallback, but this draft branch should not merge as-is because it bypasses trusted-mode SSRF policy checks and is now dirty against a substantially newer fetch-guard implementation. The cause-chain diagnostics are useful, but the added serializer also needs a guard before it is safe in the tool-error catch path.

## What This Changes

The branch changes `fetchWithSsrFGuard` so `TRUSTED_ENV_PROXY` uses an env proxy only when proxy env vars are present and otherwise leaves the default fetch dispatcher, and it adds nested cause-chain text to tool execution errors with tests.

## Best Possible Solution

Preserve the useful diagnostics and solve transparent-proxy support with a narrow trusted-web-tools path that still performs target hostname and policy checks before opting out of a custom dispatcher. The fetch-guard tests should cover no-env transparent proxy behavior, env proxy behavior, `NO_PROXY`, and an off-allowlist trusted-mode URL that must not reach fetch.

## Reproduction Assessment

Yes for the blocking review findings: a unit test can call `fetchWithSsrFGuard` in `TRUSTED_ENV_PROXY` mode with a restrictive policy and an off-allowlist URL, and the PR would reach `fetchImpl` instead of blocking. The live transparent-proxy user symptom itself was not reproduced with an OS-level proxy in this read-only review.

## Solution Assessment

No. Dropping the dispatcher for all trusted-mode no-env requests is too broad because it also drops the target policy check; a safer solution keeps policy validation and makes the default-fetch fallback explicit and narrowly covered.

Full review comments:

- **[P1] Preserve trusted-mode SSRF policy checks:** `src/infra/net/fetch-guard.ts:281-293`
  - body: This branch moves `resolvePinnedHostnameWithPolicy()` out of the `TRUSTED_ENV_PROXY` path, so trusted-mode requests no longer enforce caller policy before fetching. Any caller that passes a hostname allowlist or relies on private-IP blocking can now reach an unexpected URL in trusted mode. Keep the transparent-proxy fallback, but run the target hostname/policy checks before dropping the custom dispatcher.
  - confidence: 0.92
- **[P2] Guard cause serialization in the error path:** `src/agents/pi-tool-definition-adapter.ts:69`
  - body: `JSON.stringify(current)` can throw for circular objects or `BigInt` causes while formatting an already-caught tool error. That secondary exception would escape the tool adapter instead of returning the normal structured error result, so this formatter needs the same guarded serialization pattern used elsewhere in the file.
  - confidence: 0.9
- **[P3] Clear proxy env vars in the no-env test:** `src/infra/net/fetch-guard.ssrf.test.ts:738-746`
  - body: The new test asserts that no dispatcher is attached, but it never clears `HTTP_PROXY`, `HTTPS_PROXY`, or lowercase proxy env vars. On a developer machine or CI worker with proxy env configured, this test will take the env-proxy branch and fail for environmental reasons.
  - confidence: 0.86
