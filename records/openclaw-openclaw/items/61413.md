## Summary

Keep open. The transparent-proxy compatibility goal is still not implemented on current main for the no-proxy-env case, but this draft branch is not mergeable as-is because it bypasses trusted-mode SSRF policy checks and lacks after-fix real behavior proof.

## What This Changes

The PR changes guarded fetch transport selection in trusted env-proxy mode and adds nested cause-chain details to tool execution errors.

## Best Possible Solution

Keep the diagnostics improvement only with defensive serialization, and redesign transparent-proxy support so trusted-mode hostname policy validation runs before any default-fetch fallback.

## Reproduction Assessment

Yes for the blocking review findings: source inspection of the PR diff shows trusted-mode requests can reach fetch without the current-main policy validation branch, and the error formatter can throw while handling a caught error. The reporter's Proxifier/V2Ray symptom itself is not live-reproduced here.

## Solution Assessment

No. The proposed default-fetch fallback is broader than the current guard contract allows; the safer design validates hostname policy first, then narrowly chooses a transparent-proxy-compatible transport.

  - confidence: 0.86
