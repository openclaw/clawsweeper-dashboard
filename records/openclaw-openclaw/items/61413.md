## Summary

Keep this PR open because current main does not implement the transparent-proxy fallback or nested cause-chain diagnostics, but the proposed patch is not mergeable as-is: it is draft/conflicting, lacks real transparent-proxy proof, and introduces a trusted-mode SSRF policy bypass.

## What This Changes

The PR changes guarded fetch so `TRUSTED_ENV_PROXY` requests without proxy env vars use default fetch instead of a DNS-pinned dispatcher, and adds nested cause-chain tool error diagnostics plus tests.

## Best Possible Solution

Design an explicit transparent-proxy path that validates hostname/private-IP policy before any default-fetch fallback, keeps cause serialization defensive, and proves the real proxy setup with redacted runtime evidence.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the diff shows trusted-mode policy validation is skipped, and the new formatter can throw while handling an already-caught error. No for the original transparent-proxy runtime symptom, because the PR body and comments do not include after-fix Proxifier/V2Ray proof.

## Solution Assessment

No. The proposed default-fetch fallback is broader than the current guard contract allows; the safer path validates target policy first, then uses a narrowly opted-in transparent-proxy-compatible transport and guarded cause serialization.
