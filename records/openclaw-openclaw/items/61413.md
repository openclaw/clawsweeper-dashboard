The PR changes `fetchWithSsrFGuard` to use default fetch for `TRUSTED_ENV_PROXY` requests without proxy env vars and adds nested error-cause details to tool execution errors.

## Best Possible Solution

Add a transparent-proxy-compatible trusted path only if it preserves pre-dispatch hostname/private-IP policy checks, has explicit maintainer-approved operator semantics, and includes real proxy proof plus regression coverage.

## Solution Assessment

No. The proposed default-fetch fallback is broader than the current guarded-fetch contract; the maintainable path is a guarded, explicit transparent-proxy mode that keeps policy checks and documents the security boundary.
