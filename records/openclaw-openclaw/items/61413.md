## Summary

Keep open: the PR targets a plausible proxy failure, but the current diff bypasses trusted-mode SSRF policy checks and lacks after-fix real behavior proof for the transparent-proxy setup, so it needs maintainer networking/security judgment before merge.

## What This Changes

The PR changes `fetchWithSsrFGuard` to use default fetch in `TRUSTED_ENV_PROXY` mode when no proxy env vars are set, and adds nested cause-chain diagnostics with tests.

## Best Possible Solution

Preserve the diagnostics improvement, but redesign transparent-proxy support so trusted-mode hostname policy validation still runs before any default-fetch fallback, with coverage for env proxy, no-env fallback, `NO_PROXY`, redirects, and off-allowlist cases.

## Reproduction Assessment

Yes for the blocking review findings: source inspection gives a focused test path where `fetchWithSsrFGuard` in `TRUSTED_ENV_PROXY` mode with an off-allowlist policy would reach `fetchImpl` on the PR branch. The user's OS-level transparent-proxy symptom itself was not live-reproduced here.

## Solution Assessment

No. The proposed default-fetch fallback is too broad because it drops trusted-mode hostname policy enforcement; the safer solution is to validate policy first, then narrowly choose the transparent-proxy-compatible transport.
