The PR extracts RuntimePlan embedded-runner helpers, adds neutral embedded-runner barrels/import guardrails, introduces owner-scoped Harness V2 factory registration through the plugin SDK and Codex plugin, updates docs/API baselines, and adds regression tests.

## Best Possible Solution

Yes for the PR regressions. A source comparison shows current main avoids external auth discovery during reply prep and forwards the default plugin-harness profile, while the proposed helper does neither; a focused unit test can mock a Codex-style plugin harness with no explicit authProfileId.

No, not yet. The refactor/API direction is maintainable, but the model/auth extraction must preserve current auth discovery and plugin-harness profile forwarding before this is the best way to land the cleanup.
