## Summary

Keep open: this is a fresh, concrete Lobster workflow bug report with a documented expectation that `condition` gates steps, and current `main` still delegates workflow-file scheduling to the pinned `@clawdbot/lobster` runtime without a focused regression or proven fix.

## What This Changes

The issue reports that a Lobster workflow step with both `condition` and `stdin` can run even when its condition is false if the referenced stdin producer was also conditional and skipped.

## Best Possible Solution

Add a focused embedded-Lobster regression for a skipped conditional upstream feeding `stdin` into a false-conditioned consumer, then fix or bump `@clawdbot/lobster` if it fails; document a limitation only if the runtime behavior is confirmed intentional.

## Reproduction Assessment

No. The issue gives a clear downstream workflow shape, but I do not have a high-confidence current-main run; inspected dependency source appears to evaluate conditions before resolving stdin.

## Solution Assessment

Unclear. Honoring `condition` before `stdin` is the maintainable behavior, but OpenClaw delegates workflow-file execution to `@clawdbot/lobster`, so the next fix should start with a runtime repro and likely an upstream runtime fix or dependency bump.
