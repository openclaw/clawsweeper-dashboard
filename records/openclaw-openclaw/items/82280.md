## Summary

Keep open: this is a fresh, concrete Lobster workflow bug report with a documented expectation that `condition` gates workflow steps, and I could not prove a current-main fix, duplicate, or non-reproducible current behavior strongly enough to close it.

## What This Changes

The issue reports that a Lobster workflow step with `condition` still runs when it also reads `stdin` from a skipped conditional step.

## Best Possible Solution

Add a focused embedded-Lobster regression for a skipped conditional upstream feeding `stdin` into a false-conditioned consumer, then fix or bump the Lobster runtime if it fails.

## Reproduction Assessment

No. The issue gives a clear workflow shape and observed behavior, but I did not get a high-confidence current-main run; current dependency source appears to evaluate conditions before resolving stdin, so the failing path is not pinned yet.

## Solution Assessment

Unclear. Honoring `condition` before `stdin` resolution is the right behavior, but because OpenClaw delegates workflow-file execution to `@clawdbot/lobster`, the maintainable fix should be an upstream runtime fix or dependency bump after a real current-main reproduction.
