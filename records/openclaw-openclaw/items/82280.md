## Summary

Keep open: this is a fresh, concrete Lobster workflow bug report, but current `main` does not provide enough evidence to close it as fixed or non-reproducible. The OpenClaw wrapper still delegates workflow-file scheduling to the pinned Lobster runtime, and the pinned runtime source appears to evaluate a step condition before resolving `stdin`, so a real current-main reproduction is needed before choosing a runtime fix, dependency bump, or docs clarification.

## What This Changes

The issue reports that a Lobster step with `condition:` and `stdin: $conditional_step.stdout` can run even when its own condition is false if the stdin source step was also conditional and skipped.

## Best Possible Solution

Add a focused embedded-Lobster regression for a skipped conditional upstream feeding `stdin` into a false-conditioned consumer, then fix the Lobster runtime or bump the pinned dependency if the regression fails.

## Reproduction Assessment

No. The report gives a clear workflow shape and observed behavior, but I did not establish a current-main run; the pinned `@clawdbot/lobster@2026.4.6` source appears to evaluate conditions before resolving stdin, so source inspection does not yet reproduce the bypass.

## Solution Assessment

Unclear. Honoring `condition` before `stdin` resolution is the right behavior, but the maintainable path should be chosen after a real current-main reproduction because OpenClaw delegates workflow scheduling to the pinned Lobster runtime.
