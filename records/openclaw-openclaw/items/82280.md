## Summary

Keep open: this is a concrete Lobster bug report against shipped behavior, but current-main review did not prove either a failing reproduction or an existing fix. The OpenClaw wrapper delegates workflow-file execution to the pinned `@clawdbot/lobster` runtime, and that runtime source appears to evaluate a step's `condition` before resolving `stdin`, so the next step should be a focused real workflow repro before choosing a fix or documentation change.

## What This Changes

The issue asks for Lobster step `condition` gates to be honored even when the step reads `stdin` from a conditional upstream step.

## Best Possible Solution

Add a focused Lobster workflow regression for a false consumer condition with `stdin` from a skipped conditional upstream step, then fix or bump the embedded runtime so condition evaluation always wins over stdin dependency handling.

## Reproduction Assessment

No. The reporter provides a concrete workflow and environment, but I did not establish a current-main failing run; the pinned runtime source appears to check `condition` before resolving `stdin`, so the failure still needs a focused real repro.

## Solution Assessment

Unclear. Fixing the runtime is the right direction if the repro fails on current main; documenting the workaround should only be considered if maintainers decide the current dependency behavior is intentional.
