## Summary

Keep open. Current main has overlapping message CLI exit safeguards, gateway-owned Telegram routing, request timeouts, and bounded shutdown hooks, but the report is a new Windows/Node 24 subprocess leak against recent releases and I did not find enough current-main Windows live proof to close it as fixed or unreproducible.

## What This Changes

The issue asks for `openclaw message send` to return cleanly on Windows and avoid leaving child `node.exe` processes when the caller times out.

## Best Possible Solution

Add a focused Windows regression path that bounds the CLI/gateway request and teardown path, ensures the spawned Node process exits on success, failure, and timeout, and preserves Telegram delivery result reporting.

## Reproduction Assessment

No high-confidence current-main reproduction yet. The report gives concrete Windows shell and Python steps, but this review only source-traced the CLI, Gateway, and Telegram paths and did not run a Windows live Telegram scenario.

## Solution Assessment

Unclear until the Windows leak is reproduced against current main. The narrowest likely fix belongs in the CLI message and Gateway teardown path using existing timeout/shutdown seams, not in Telegram-specific delivery behavior unless live proof shows the plugin keeps the event loop alive.
