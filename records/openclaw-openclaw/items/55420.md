## Summary

Keep open. Current main and v2026.5.22 still expose only the existing `self | tree | agent | all` session visibility scopes, while the requested allowlist would add a new public session/transcript authorization mode that needs maintainer product and security review.

## What This Changes

The issue requests adding `tools.sessions.visibility: "allowlist"` plus a `tools.sessions.allowlist` array of session-key glob/prefix patterns for selective cross-session visibility.

## Best Possible Solution

Define one maintainer-approved session visibility contract that composes allowlisted session keys with granular/per-agent visibility, sandbox clamps, A2A policy, schema/types/help/docs, and access-control tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source, schema, docs, exact search, and release-tag inspection give high confidence that `visibility="allowlist"` and `tools.sessions.allowlist` do not exist today.

## Solution Assessment

Unclear until maintainers choose the access-control semantics. The allowlist direction is plausible, but it should be designed together with per-capability visibility, per-agent scoping, canonical key matching, sandbox clamps, memory search, and A2A send behavior.
