## Summary

Keep open. Current main still does not implement the requested session-key allowlist mode: runtime visibility, shared access checks, strict config schema, typed config/docs, and session tool call sites still model only `self`, `tree`, `agent`, and `all`. The related #48785 is adjacent permission-model work, but it does not supersede this narrower pattern allowlist request.

## What This Changes

Add a `tools.sessions.visibility: "allowlist"` mode with a companion `tools.sessions.allowlist` array of session-key patterns for selective cross-session access.

## Best Possible Solution

Keep this open and design the feature in the shared session visibility layer: extend runtime/types/schema/generated docs with an explicit `allowlist` mode and `tools.sessions.allowlist: string[]`, match canonical resolved session keys before granting list/history/send/status access, and add focused tests for allowed and denied same-agent, cross-agent, sandboxed, and sessionId-resolution paths.

## Reproduction Assessment

Not applicable as a feature request. Static verification against current main is high confidence: the runtime enum, schema, docs, and shared guard do not expose or honor `allowlist`.

## Solution Assessment

No current solution exists on main. The requested direction is plausible, but the best maintainable path needs a short maintainer policy decision for A2A and sandbox interactions before code changes.
