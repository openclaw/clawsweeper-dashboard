## Summary

Keep open. Current main still exposes only the four existing session visibility scopes and has no `tools.sessions.allowlist` config surface; the related granular-visibility issue is open and adjacent, but it does not supersede this exact session-key allowlist request.

## What This Changes

Add `tools.sessions.visibility="allowlist"` plus a `tools.sessions.allowlist` array of session-key patterns for trusted cross-session access.

## Best Possible Solution

Define a maintainer-approved, fail-closed session-key pattern contract, then implement it centrally across schema/types/help/docs, the shared visibility helper, all session tool surfaces, status, memory search, and focused access-control tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source, schema, docs, and tests give high confidence that current main does not have an `allowlist` visibility mode or `tools.sessions.allowlist` config surface.

## Solution Assessment

Unclear until maintainers choose the access-control semantics. The requested direction is plausible, but the maintainable fix needs explicit decisions for pattern syntax, canonical key resolution, sandbox clamps, A2A interaction, and read versus send surfaces.
