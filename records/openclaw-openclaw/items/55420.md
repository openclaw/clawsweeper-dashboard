## Summary

Keep open. Current main and v2026.6.10 still expose only the four existing session visibility modes with no `tools.sessions.allowlist`, and the related open items cover adjacent per-capability, per-agent, A2A-send, and sandbox axes rather than superseding this session-key allowlist request.

## What This Changes

The issue requests adding `tools.sessions.visibility="allowlist"` plus `tools.sessions.allowlist` session-key patterns for selective cross-session visibility.

## Best Possible Solution

Define one maintainer-approved session authorization model that decides how session-key patterns compose with list, history, send, status, sandbox clamps, and A2A policy, then update schema, runtime, docs, and focused isolation tests together.

## Reproduction Assessment

Yes by source inspection, not as a bug reproduction. Current main rejects or ignores the requested config shape because the schema and runtime only know `self`, `tree`, `agent`, and `all`.

## Solution Assessment

Unclear. A session-key allowlist is plausible for the stated trusted-group use case, but maintainers should first choose whether the durable model is target-pattern based, per-capability, per-agent, A2A-specific, or consolidated across those axes.
