## Summary

Keep open. The stale `modelOverride` writeback remains present on current main and this PR is still the canonical focused fix, but the current branch drops legacy `sessionStartedAt` backfill for reused sessions and needs that narrow repair before merge.

## What This Changes

The PR drops cached `entry` metadata writebacks from the gateway agent session patch and adds regression tests for stale cached-entry versus fresh-store races.

## Best Possible Solution

Land the focused gateway stale-writeback fix after preserving legacy `sessionStartedAt` backfill, while keeping the delivery/channel follow-up in https://github.com/openclaw/openclaw/issues/70575 separate.

## Reproduction Assessment

Yes, at source level. `loadSessionEntry` can return a cached row without `modelOverride`, `sessions.patch` can write the override to the fresh store, and the current writer then merges the stale patch over that fresh row; the PR body also supplies terminal proof through production store merge code.

## Solution Assessment

Mostly, but not as currently written. Dropping cached-entry override and metadata carryovers is the narrow maintainable fix, but reused sessions still need legacy `sessionStartedAt` fallback preserved without restoring stale writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve legacy session start backfill:** `src/gateway/server-methods/agent.ts:1064`
  - body: For reused sessions this now omits `sessionStartedAt`, but `mergeSessionEntry` only preserves `existing.sessionStartedAt` when the `sessionId` is unchanged; it does not rederive the transcript/header fallback. Legacy rows without the field stop getting backfilled and can later fall back to `updatedAt`, delaying daily/idle reset decisions. Derive the fallback from the fresh row in the store update, or omit it only after the fresh row already has a value.
  - confidence: 0.86
