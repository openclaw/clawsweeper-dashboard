## Summary

Keep this PR open, but it still needs one focused correctness fix before merge. The stale cached-entry overwrite bug is real on current main and the PR addresses the right gateway session-store path, but the patch drops legacy `sessionStartedAt` backfill for reused sessions.

## What This Changes

The PR removes stale cached session-entry writebacks from the gateway agent session-store patch and adds regression tests for fresh model/provider override preservation.

## Best Possible Solution

Land a refreshed focused patch that removes stale override writebacks while preserving or deriving `sessionStartedAt` from the fresh row/header for legacy reused sessions.

## Reproduction Assessment

Yes, at source level. Current main builds a cached-entry patch containing stale override fields and merges it over the fresh store row; the PR body also provides terminal proof through production store merge code.

## Solution Assessment

Mostly, but not as currently written. Dropping cached override writebacks is the right narrow fix, but reused sessions still need legacy `sessionStartedAt` derivation/backfill preserved without restoring stale writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve legacy session start backfill:** `src/gateway/server-methods/agent.ts:1064`
  - body: For reused sessions this patch omits `sessionStartedAt`, but `mergeSessionEntry` only preserves `existing.sessionStartedAt` when the `sessionId` is unchanged; it does not rederive the JSONL/header fallback. Legacy rows without the field stop getting backfilled into the persisted entry, which can later make reset freshness fall back to `updatedAt`. Derive the fallback from the fresh row in the store update, or omit it only after the fresh row already has a value.
  - confidence: 0.86
