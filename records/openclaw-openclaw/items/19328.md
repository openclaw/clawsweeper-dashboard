## Summary

Keep open: current main still has the stale cached session-entry writeback path, and this PR is the focused canonical fix for preserving fresh modelOverride/providerOverride fields. I found one actionable regression risk in the patch around legacy sessionStartedAt backfill that should be repaired before merge.

## What This Changes

This PR changes gateway agent session persistence to omit cached metadata fields from nextEntryPatch and adds regression tests for stale-cache/fresh-store races.

## Best Possible Solution

Land this focused gateway fix after preserving legacy sessionStartedAt backfill, while keeping the delivery/channel follow-up in https://github.com/openclaw/openclaw/issues/70575 separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: loadSessionEntry can return a cached entry without modelOverride, sessions.patch can write modelOverride to the fresh store, and updateSessionStore then merges the stale patch over that fresh row.

## Solution Assessment

Mostly, but not as currently written. Dropping cached-entry override and metadata carryovers is the narrow maintainable fix for the modelOverride race, but the implementation should preserve legacy sessionStartedAt fallback without reintroducing stale writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve session start backfill:** `src/gateway/server-methods/agent.ts:1064`
  - body: For reused sessions this now omits sessionStartedAt and relies on mergeSessionEntry to backfill it, but mergeSessionEntry keeps existing.sessionStartedAt when the sessionId is unchanged. Legacy rows that only have a transcript/header fallback will stop being backfilled and can later get reset to now, delaying daily/idle freshness decisions; derive this from the fresh store row inside updateSessionStore or only omit it when the fresh row already has a value.
  - confidence: 0.86
