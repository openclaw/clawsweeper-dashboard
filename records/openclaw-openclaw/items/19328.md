## Summary

Keep open: the stale model/provider override race remains on current main and this branch addresses it, but the current patch should not merge until it preserves legacy session start backfill for reused rows.

## What This Changes

This branch changes `src/gateway/server-methods/agent.ts` to omit stale cached session fields from `nextEntryPatch` and adds gateway agent-handler regression tests.

## Best Possible Solution

Land a refreshed focused gateway fix that preserves fresh override/session fields without stale cached writebacks while keeping legacy `sessionStartedAt` derivation/backfill for reused rows.

## Reproduction Assessment

Yes, at source level. Current main reads a cached session row, builds a patch containing stale override fields, then merges that patch over the fresh store row; the PR body also supplies terminal proof through production store merge code.

## Solution Assessment

Mostly, but not as currently written. Dropping stale cached override writebacks is the right narrow fix, but reused sessions still need legacy `sessionStartedAt` backfill preserved without restoring stale writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve legacy session start backfill:** `src/gateway/server-methods/agent.ts:1064`
  - body: For reused sessions this patch omits `sessionStartedAt`, but `mergeSessionEntry` only preserves an existing value when the `sessionId` is unchanged; it does not rederive the JSONL/header fallback. Legacy rows without the field stop getting backfilled into the persisted entry, which can later make reset freshness fall back to `updatedAt`. Derive the fallback from the fresh row in the store update, or omit it only after the fresh row already has a value.
  - confidence: 0.86
