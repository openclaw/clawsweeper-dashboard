## Summary

Keep this PR open. Current main still has the stale cached-entry writeback path that can clobber fresh model/provider overrides, and this PR addresses the right gateway path, but the latest head still needs a small fix to preserve legacy session start backfill for reused sessions.

## What This Changes

The PR changes gateway agent session persistence to avoid cached session-entry writebacks clobbering fresh store values and adds stale-cache regression tests for model/provider override and adjacent session fields.

## Best Possible Solution

Land a refreshed focused gateway fix that stops stale cached override fields from being written back while deriving or preserving sessionStartedAt from the fresh store row for reused legacy sessions.

## Reproduction Assessment

Yes, at source level. Current main reads a cached session row, builds a patch containing stale override fields, and merges that patch over the fresh store row; the PR body also supplies terminal proof through production session-store merge code.

## Solution Assessment

No, not as currently written. Dropping stale cached override writebacks is the right narrow fix, but the reused-session path must preserve legacy sessionStartedAt derivation/backfill instead of silently omitting it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve legacy session start backfill:** `src/gateway/server-methods/agent.ts:1189`
  - body: For reused sessions this patch omits sessionStartedAt, but mergeSessionEntry only preserves an already-explicit existing.sessionStartedAt; it does not call the transcript-header lifecycle fallback. Legacy rows without the field stop getting that derived start time persisted, which can later make reset freshness fall back to updatedAt. Derive the fallback from the fresh store row inside the writer, or omit it only after the fresh row already has a value.
  - confidence: 0.87
