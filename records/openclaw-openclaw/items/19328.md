## Summary

Keep this PR open. Current main still has the stale cached-entry writeback path for `modelOverride`, and this PR is the right focused direction, but the branch still needs one narrow repair before merge because it drops legacy `sessionStartedAt` backfill for reused sessions.

## What This Changes

The PR changes gateway agent session persistence to omit cached metadata fields from `nextEntryPatch` and adds regression tests for stale-cache/fresh-store races.

## Best Possible Solution

Land this focused gateway fix after preserving legacy `sessionStartedAt` backfill, while keeping the delivery/channel follow-up in https://github.com/openclaw/openclaw/issues/70575 separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `loadSessionEntry` can return a cached entry without `modelOverride`, `sessions.patch` can write `modelOverride` to the fresh store, and `updateSessionStore` then merges the stale patch over that fresh row; the PR body also supplies terminal proof through production store merge code.

## Solution Assessment

Mostly, but not as currently written. Dropping cached-entry override and metadata carryovers is the narrow maintainable fix for the modelOverride race, but the implementation should preserve legacy `sessionStartedAt` fallback without restoring stale writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve session start backfill:** `src/gateway/server-methods/agent.ts:1064`
  - body: For reused sessions this now omits `sessionStartedAt`, but `mergeSessionEntry` only preserves `existing.sessionStartedAt` when the `sessionId` is unchanged; it does not rederive the transcript/header fallback. Legacy rows without the field stop getting backfilled and can later fall back to `updatedAt`, delaying daily/idle reset decisions. Derive the fallback from the fresh row in the store update, or omit it only after the fresh row already has a value.
  - confidence: 0.86
