- `impact:session-state`: The bug can start unintended local embedded agent runs from control-command input, affecting session flow.
- `impact:message-loss`: Typed control commands are misdelivered as user/model messages instead of command responses.

## Summary

Current main and the latest release still have the reported local TUI fallthrough: slash submissions reach the TUI command switch, `/status` and `/compact` are advertised through shared-command help/autocomplete, and unhandled command names still call `sendMessage(raw)`, which the embedded backend passes to the agent ingress path. The issue should stay open because the two directly linked fix PRs are closed unmerged and no current-main commit contains their central fix.

## What This Changes

The issue reports that TUI local embedded mode advertises `/status` and `/compact` but sends them to the local model as ordinary user prompts instead of handling or rejecting them as commands.

## Best Possible Solution

Fix the TUI local command boundary so local help/autocomplete only advertises locally supported shared commands, and known unsupported shared commands such as `/status` and `/compact` produce an explicit local response without reaching the embedded model path.

## Reproduction Assessment

Yes from source: current main routes slash input to `handleCommand`, advertises shared `/status` and `/compact`, and still falls through unhandled names to `sendMessage(raw)`. I did not run the PTY repro in this read-only cleanup pass.

## Solution Assessment

Yes: the best fix is a narrow TUI local-mode dispatch/discovery repair, not a gateway change, because local embedded mode bypasses the gateway recovery path and calls the embedded backend directly.
