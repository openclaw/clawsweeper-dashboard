## Label Justifications

- `P2`: This is a normal-priority TUI local-mode command routing bug with limited blast radius and a clear source-level fix boundary.
- `impact:message-loss`: Advertised slash commands can be misdelivered as ordinary model prompts instead of command/control responses.
- `impact:session-state`: The fallthrough creates local embedded user/model turns from control commands, affecting session transcript and run state.

## Summary

Keep open: current main and the latest release still advertise shared `/status` and `/compact` in the TUI while unhandled commands fall through to `sendMessage(raw)`. An open closing PR now owns the fix path, but it is unmerged, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that TUI local embedded `/status` and `/compact` commands be handled, routed, or explicitly rejected instead of being delivered to the local model as user prompts.

## Best Possible Solution

Land a local-mode-only command handling path, or a shared command adapter, that keeps `/status` and `/compact` out of local model prompts while preserving gateway-connected command forwarding.

## Reproduction Assessment

Yes, this is source-reproducible. Slash input routes to `handleCommand`, TUI autocomplete includes shared `/status` and `/compact`, and current `handleCommand` lacks those cases before the default `sendMessage(raw)` branch.

## Solution Assessment

The requested fix direction is sound, but the current open PR should be corrected before landing. The best fix is local-mode-only handling or shared command execution that does not break gateway-connected TUI forwarding.
