## Summary

This PR is still necessary because current main still derives `talk.realtime.speakerVoice` from deprecated `voice` during normalization, which source-inspection shows can make doctor report a change for the issue shape. The proposed patch is a narrow, plausible fix with no blocking findings, though a dedicated regression test for the reported shape would improve confidence.

## What This Changes

The PR stops `normalizeTalkRealtimeConfig` from deriving `speakerVoice` from `voice`, keeps that seeding only for legacy flat `talk.voice` migration, and adds a stable JSON comparison guard in the doctor talk normalizer.

## Best Possible Solution

Land the narrow normalization fix while keeping runtime voice fallback at consumption time and preserving `speakerVoice` seeding only for actual legacy flat-field migration.

## Reproduction Assessment

Yes by source inspection: current main derives `speakerVoice` from `talk.realtime.voice`, then doctor compares that normalized shape with the raw config and can report a normalization change. I did not run tests because this review is read-only.

## Solution Assessment

Yes, this is a narrow owner-boundary fix: the fallback stays in `talk-shared` where realtime config is consumed, while doctor only seeds `speakerVoice` during true legacy flat-field migration. A dedicated regression test would make the proof stronger but is not a blocking correctness defect.
