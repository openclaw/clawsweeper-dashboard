This PR removes `speakerVoice` derivation from Talk realtime normalization, seeds `speakerVoice` only for legacy flat `talk.voice` migration, adds a stable JSON comparison guard in the doctor talk normalizer, and updates the normalization test expectation.

## Best Possible Solution

Keep the shipped public Talk normalization/response fallback and make the doctor no-op comparison ignore only `speakerVoice` values that are derived from `realtime.voice`, with a focused regression for the reported config shape.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main and v2026.6.1 derive `speakerVoice` from `talk.realtime.voice`, then `normalizeLegacyTalkConfig` compares that derived normalized object against the raw Talk config and can report a change for the issue shape.

## Solution Assessment

No, not as written. The doctor bug is real, but changing the shared normalizer is broader than necessary because that same function feeds the gateway `talk.config` payload; a doctor-local comparison fix is narrower and preserves shipped response compatibility.
