## Summary

Keep open. Current main and v2026.6.6 still do not expose `batchEmbed` from the Ollama memory adapter, but current docs and code already give Ollama inline array batching and document `remote.batch` for OpenAI, Gemini, and Voyage, so this needs a maintainer decision before a fix PR.

## What This Changes

Request to expose an Ollama memory-adapter `batchEmbed` runtime hook so configured remote batch indexing can use the existing Ollama embedding client batch capability.

## Best Possible Solution

Maintainers should choose the intended Ollama semantics, then either add an adapter hook that preserves inline split/retry/timeout/cache behavior with tests and docs, or clarify that Ollama uses inline batching while `remote.batch` stays limited to provider-native batch APIs.

## Reproduction Assessment

Yes for the exact source gate: current main's Ollama memory runtime lacks `batchEmbed`, and memory-core requires that hook before `remote.batch.enabled` becomes effective. The broader claim that Ollama is forced into one-item embedding is not reproduced because current main already sends array inputs through the inline path.

## Solution Assessment

No, not as an unconditional direct wrapper. The best path is to decide whether `remote.batch` should broaden beyond provider-native batch jobs for Ollama, then implement or document that decision consistently.
