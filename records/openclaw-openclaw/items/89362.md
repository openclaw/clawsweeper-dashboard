## Summary

Keep open. The exact `remote.batch.enabled` gate is source-reproducible for Ollama, but current main already sends inline Ollama embedding batches, and the documented `remote.batch` surface is limited to provider-native batch APIs, so this needs a maintainer decision rather than cleanup closure.

## What This Changes

Requests exposing an Ollama `batchEmbed` memory-runtime hook so configured remote batch memory indexing can turn on and forward chunk texts to Ollama batch embeddings.

## Best Possible Solution

Decide whether `remote.batch.enabled` should grow an Ollama-specific synchronous batch meaning; if yes, implement it with safe chunk splitting and regression coverage, otherwise clarify docs/status so Ollama users know inline batching is already active.

## Reproduction Assessment

Yes for the exact gate: source inspection shows the Ollama runtime lacks `batchEmbed`, and memory-core requires that hook before `remote.batch.enabled` becomes effective. The broader single-item claim is not reproduced from source because current main already calls `provider.embedBatch` with chunk arrays in the inline path.

## Solution Assessment

No, not as proposed without a maintainer decision. The current `batchEmbed` contract is used by provider-native batch-job adapters, so adding an Ollama wrapper is only the best fix if maintainers intentionally broaden that semantic and preserve chunk splitting, retry, timeout, and status behavior.
