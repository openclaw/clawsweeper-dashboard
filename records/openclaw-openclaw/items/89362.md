## Summary

Keep open. Current main and v2026.6.8 still lack the Ollama adapter `batchEmbed` hook, but current docs and sibling adapters treat `remote.batch` as a provider batch-job API while Ollama already sends array `/api/embed` requests inline, so this needs maintainer product/API direction before implementation.

## What This Changes

Request to expose Ollama memory adapter `batchEmbed` so configured memory batch indexing can use Ollama's embedding batch client.

## Best Possible Solution

Maintainers should choose and document the intended Ollama semantics: either keep `remote.batch` limited to provider-native batch-job APIs and clarify status/docs, or add a provider-owned Ollama runtime hook that preserves existing inline safety behavior and has focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows the exact gap: memory batch mode requires `providerRuntime.batchEmbed`, and the current Ollama memory adapter does not provide it; I did not run a live Ollama indexing scenario in this read-only review.

## Solution Assessment

Unclear. Adding `batchEmbed` is a plausible narrow implementation only after maintainers decide whether Ollama's synchronous `/api/embed` array batching should be treated as the memory `remote.batch` runtime contract.

- A direct Ollama `batchEmbed` wrapper would make `remote.batch.enabled` mean synchronous `/api/embed` array calls for one provider while sibling adapters use provider-native batch-job semantics with wait, polling, timeout, and job concurrency.
