## Summary

Keep open. Current main and the latest release still do not expose `batchEmbed` from the Ollama memory adapter; the source gap is real, but current docs limit `remote.batch` to OpenAI, Gemini, and Voyage while Ollama already uses inline array `/api/embed` requests, so this needs a maintainer product/API decision rather than cleanup closure.

## What This Changes

The issue requests exposing Ollama batch embedding capability through the memory adapter so configured memory batch indexing can turn on for Ollama-backed memory.

## Best Possible Solution

Maintainers should choose and document the intended Ollama semantics: either keep `remote.batch` limited to provider-native batch-job APIs and clarify status/docs, or add a provider-owned Ollama runtime hook that preserves existing splitting, retry, timeout, cache, and SSRF-guard behavior.

## Reproduction Assessment

Yes. Source inspection shows the exact gate: memory batch mode requires `providerRuntime.batchEmbed`, and the current Ollama memory adapter does not provide it; the broader single-request-per-chunk claim is not reproduced because Ollama already uses inline array requests.

## Solution Assessment

Unclear. Adding `batchEmbed` is a plausible narrow implementation only after maintainers decide whether Ollama's synchronous `/api/embed` array batching should be treated as the memory `remote.batch` runtime contract.

- If implemented without care, the new path could bypass existing inline splitting, retry, timeout, cache, and local-host protection behavior that currently makes Ollama indexing safer on resource-constrained machines.
