## Summary

Keep open. Current main still does not expose `batchEmbed` from the Ollama memory adapter, while Ollama inline indexing already sends array batches through `/api/embed`; the remaining question is whether `remote.batch.enabled` should intentionally grow Ollama-specific synchronous batch semantics.

## What This Changes

The issue requests exposing a `batchEmbed` runtime from the Ollama memory embedding adapter so `remote.batch.enabled` can enable Ollama-backed batch indexing.

## Best Possible Solution

Maintainers should choose the intended Ollama semantics: either keep `remote.batch` limited to provider-native batch APIs and clarify status/docs, or add an Ollama adapter hook with preserved splitting, retry, timeout, and regression coverage.

## Reproduction Assessment

Yes for the exact gate: source inspection shows the Ollama runtime lacks `batchEmbed`, and memory-core requires that hook before `remote.batch.enabled` becomes effective. The broader single-item-embedding claim is not reproduced from source because current main already calls Ollama `embedBatch` with arrays in the inline path.

## Solution Assessment

No, not as a direct wrapper without a maintainer decision. The narrow maintainable path is to first decide whether `remote.batch` should broaden beyond provider-native batch-job APIs for Ollama, then implement or document that decision consistently.
