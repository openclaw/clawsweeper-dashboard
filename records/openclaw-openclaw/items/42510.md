## Label Justifications

- `P1`: This is a confirmed Google Chat channel regression affecting real deployments and routing replies into the wrong chat context.
- `impact:message-loss`: The issue is about Google Chat messages being misrouted into threads despite an explicit off-mode configuration.

## Summary

Keep open: current main still has Google Chat outbound paths that can pass thread values to `sendGoogleChatMessage` without honoring `replyToMode: "off"`, and the overlapping fix remains open rather than merged.

## What This Changes

The issue requests that Google Chat suppress implicit thread replies when `channels.googlechat.replyToMode` is set to `off`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235, or an equivalent Google Chat-owned boundary fix, with off-mode suppression for implicit thread propagation while preserving explicit valid thread resources.

## Reproduction Assessment

Yes, source-level. Configure Google Chat with `replyToMode: "off"`, process an inbound event containing `message.thread.name`, and current main can still pass that thread through typing-placeholder, attached-result, or reply-delivery paths.

## Solution Assessment

Yes in direction, but it is not on main yet. The maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while keeping explicit valid thread resources working.
