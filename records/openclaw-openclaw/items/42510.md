## Summary

Keep open: current main still has Google Chat paths that can attach messages to the inbound thread despite `replyToMode: "off"`, while the active fix path is still an unmerged PR.

## What This Changes

Google Chat should honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235, or an equivalent Google Chat-owned fix, with off-mode suppression for implicit thread propagation and regression coverage for monitor, attached-result, and message-tool paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Google Chat with `replyToMode: "off"`, process an inbound webhook containing `message.thread.name`, and current main can still forward that thread through the typing-placeholder or outbound adapter path.

## Solution Assessment

Yes in direction, but it is not on main yet. The narrow maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while preserving explicit valid `spaces/.../threads/...` targets.
