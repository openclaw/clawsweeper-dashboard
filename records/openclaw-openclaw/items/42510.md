## Label Justifications

- `P1`: An established Google Chat setting is ignored in real deployments, forcing responses into unwanted threads and breaking a primary channel workflow.
- `impact:message-loss`: Replies are delivered to an unintended conversational location, making them misrouted and easy to miss.

## Summary

Current `main` still creates the Google Chat typing message in the inbound thread without checking `replyToMode`; the first final reply then edits that threaded message, so `"off"` remains ineffective. The open closing PR https://github.com/openclaw/openclaw/pull/89790 owns the active repair path, so this issue should remain open until that PR is resolved.

## What This Changes

Make Google Chat suppress implicit inbound-thread targeting when `channels.googlechat.replyToMode` is `"off"`.

## Best Possible Solution

Use one Google Chat-owned thread-selection policy before typing and reply delivery, distinguish implicit inbound context from explicit thread targets, and add focused coverage for `off`, `first`, and `all` across typing, text, media, and durable final paths.

## Reproduction Assessment

Yes at source level: for an inbound Google Chat group thread with typing-message mode enabled, current `main` creates the typing message in that thread before reply-mode filtering, and the final response edits it. Independent deployed reports corroborate this path, although this read-only review did not run a fresh live Google Chat instance.

## Solution Assessment

Yes, the replacement PR's intended boundary is appropriate if it centralizes implicit thread selection and covers every delivery path; scattered checks only in `sendText` or `sendMedia` would leave typing or durable delivery inconsistent.
