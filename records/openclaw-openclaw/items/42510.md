## Summary

Keep this issue open. Current main still accepts Google Chat `replyToMode: "off"`, but the Google Chat monitor, reply delivery helper, and routed outbound adapter still pass thread identifiers into `sendGoogleChatMessage`; the related open PRs are useful context but are not merged into main.

## What This Changes

Google Chat should suppress implicit thread replies when `channels.googlechat.replyToMode` is set to `"off"`.

## Best Possible Solution

Land one reconciled Google Chat fix that gates implicit thread propagation on the resolved account `replyToMode` across monitor context, typing placeholders, natural reply delivery, and routed outbound text/media, while preserving intentional valid thread targets and adding focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an inbound Google Chat event with `message.thread.name` plus `channels.googlechat.replyToMode: "off"`; current main still passes that thread through the typing path, reply delivery, and routed outbound adapters.

## Solution Assessment

No. Current main is not a solution; the maintainable fix should be a narrow cross-path Google Chat boundary change with tests rather than a monitor-only patch that leaves routed outbound sends uncovered.
