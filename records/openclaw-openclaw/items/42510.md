## Summary

Keep this issue open: current main still exposes Google Chat `replyToMode`, but Google Chat monitor, reply-delivery, and routed outbound send paths continue to pass thread IDs to the API without checking whether the resolved mode is `off`.

## What This Changes

The issue requests that Google Chat suppress implicit thread targets for text, media, and reply delivery when `channels.googlechat.replyToMode` is `off`.

## Best Possible Solution

Land one focused Google Chat plugin fix that gates implicit thread propagation on the resolved account `replyToMode` across monitor typing/reply delivery and routed outbound text/media sends, while preserving intentional explicit thread targets and adding regression tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an inbound Google Chat message with `message.thread.name` plus `channels.googlechat.replyToMode: "off"`; current main still passes that thread through typing, reply delivery, and routed outbound send paths.

## Solution Assessment

Yes, the best direction is a narrow Google Chat-owned boundary fix. The current main implementation is not sufficient, and the safest next step is to reconcile the overlapping Google Chat threading PRs into one complete patch.
