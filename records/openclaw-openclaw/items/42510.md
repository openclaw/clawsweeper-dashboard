## Summary

Keep open: current main still has Google Chat paths that can carry an inbound thread into reply context, typing placeholders, final delivery, and outbound sends when `replyToMode` is `off`; active related PRs already cover the overlapping fix area and need maintainer reconciliation rather than a new cleanup close.

## What This Changes

The issue asks Google Chat to honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies instead of forwarding thread IDs to the Chat API.

## Best Possible Solution

Land one Google Chat-owned fix that suppresses only implicit thread propagation when the resolved account mode is `off`, preserves explicit reply targets, and includes monitor/outbound regression coverage.

## Reproduction Assessment

Yes, source-level: configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook with `message.thread.name`; current main can still pass that thread through the monitor, typing placeholder, or outbound delivery paths.

## Solution Assessment

Yes in direction, but it is not shipped yet. The narrow maintainable solution is to filter implicit Google Chat thread sources at the plugin/reply boundary while preserving explicit reply directives and reconciling the active PRs.
