## Label Justifications

- `P1`: This is a confirmed Google Chat channel regression affecting real deployments and routing replies into the wrong chat context.
- `impact:message-loss`: The issue is about Google Chat responses being misrouted into threads despite an explicit off-mode configuration.

## Summary

Keep open: current main and the latest release still source-reproduce Google Chat thread propagation when `replyToMode: "off"` is configured, while the broad overlapping fix path remains open and conflicting rather than merged.

## What This Changes

Google Chat should honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies instead of forwarding inbound thread IDs to outbound sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned boundary fix that suppresses implicit thread propagation under off mode while preserving explicit valid thread resources.

## Reproduction Assessment

Yes, at source level. Configure Google Chat with `replyToMode: "off"`, process an inbound event containing `message.thread.name`, and current main can still pass that thread through typing-placeholder or attached-result delivery paths.

## Solution Assessment

Yes in direction, but it is not on main yet. The maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while keeping explicit valid thread resources working.
