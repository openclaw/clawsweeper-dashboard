## Label Justifications

- `P1`: This is a confirmed Google Chat channel regression affecting real deployments by routing replies into the wrong chat context.
- `impact:message-loss`: The issue is about Google Chat responses being misrouted into threads despite an explicit off-mode configuration.

## Summary

Keep open: current main and latest v2026.5.20 still allow Google Chat thread values to reach outbound sends while `replyToMode: "off"` is configured, and the overlapping fix path at https://github.com/openclaw/openclaw/pull/74235 remains open rather than merged.

## What This Changes

The issue asks Google Chat to honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-boundary fix that suppresses implicit thread propagation when `replyToMode` is `off` while preserving explicit valid thread resources.

## Reproduction Assessment

Yes, source-level. Configure Google Chat with `replyToMode: "off"`, process an inbound event containing `message.thread.name`, and current main can still pass that thread through typing-placeholder or attached-result send paths.

## Solution Assessment

Yes in direction, but it is not on main yet. The maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while keeping explicit valid thread resources working.
