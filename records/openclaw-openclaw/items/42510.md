## Label Justifications

- `P1`: This is a confirmed Google Chat channel regression affecting real deployments and making replies appear in the wrong chat context.
- `impact:message-loss`: The issue is about Google Chat messages being misrouted into threads despite an explicit off-mode configuration.

## Summary

Keep open: current main still has Google Chat paths that can attach outbound messages to an inbound thread despite `replyToMode: "off"`, and the overlapping fix path is still the open https://github.com/openclaw/openclaw/pull/74235 rather than merged code.

## What This Changes

The issue requests that Google Chat honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies for text, media, and related outbound reply paths.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235, or an equivalent Google Chat-owned boundary fix, with off-mode suppression for implicit thread propagation while preserving explicit valid `spaces/.../threads/...` targets.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Google Chat with `replyToMode: "off"`, process an inbound webhook containing `message.thread.name`, and current main can still pass that thread through the typing-placeholder or attached-result outbound path.

## Solution Assessment

Yes in direction, but it is not on main yet. The maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while keeping explicit valid thread resources working.
