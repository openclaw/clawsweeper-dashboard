## Summary

Keep this issue open: current main still exposes Google Chat `replyToMode` but several Google Chat send paths continue to pass thread identifiers into the API without checking the resolved mode, so the reported off-mode suppression bug remains valid.

## What This Changes

Google Chat should suppress implicit thread replies when `channels.googlechat.replyToMode` is set to `"off"`, matching the existing replyToMode contract.

## Best Possible Solution

Land one Google Chat boundary fix that gates implicit thread propagation on the resolved account `replyToMode` across monitor typing/reply delivery and routed outbound text/media sends, while preserving intentional explicit thread targets and adding focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an inbound Google Chat event with `message.thread.name` plus `channels.googlechat.replyToMode: "off"`: current main still passes that thread through the monitor typing path, reply delivery, and routed outbound adapter paths.

## Solution Assessment

No. Current main is not a solution, and the safest path is to review or reconcile the overlapping Google Chat threading PRs into one focused fix that covers both natural replies and routed outbound sends.
