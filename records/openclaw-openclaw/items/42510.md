## Summary

Keep open: current main still exposes Google Chat `replyToMode`, but monitor typing/reply context and routed outbound text/media paths can still pass thread IDs to `sendGoogleChatMessage` without checking whether the resolved mode is `off`.

## What This Changes

The issue requests that Google Chat stop sending implicit thread IDs when `channels.googlechat.replyToMode` is `off`.

## Best Possible Solution

Land one focused Google Chat-owned fix that gates implicit thread propagation on the resolved account `replyToMode` across monitor typing/reply delivery and routed outbound text/media sends, with regression tests for off-mode suppression.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an inbound Google Chat message with `message.thread.name` plus `channels.googlechat.replyToMode: "off"`; current main still has paths that pass that thread value into Google Chat sends.

## Solution Assessment

Yes. The narrowest maintainable direction is a Google Chat plugin boundary fix, but the active overlapping PRs should be reconciled before another repair branch is started.
