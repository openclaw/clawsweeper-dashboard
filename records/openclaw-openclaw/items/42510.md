## Label Justifications

- `P1`: The issue breaks an established Google Chat channel workflow for real deployments that configure off mode to avoid threaded replies.
- `impact:message-loss`: Replies can be delivered into unintended Google Chat threads instead of the configured flat conversation, which is message misrouting.

## Summary

Keep open: current main and the latest release still pass Google Chat thread metadata into sendGoogleChatMessage when off-mode semantics should suppress it, and the linked fix candidate at https://github.com/openclaw/openclaw/pull/89790 is still open.

## What This Changes

Google Chat should honor channels.googlechat.replyToMode: "off" by omitting thread targets from text, media, typing, and final reply delivery.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/89790, or an equivalent Google Chat-owned patch that suppresses implicit thread targets in off mode while preserving explicit threaded modes.

## Reproduction Assessment

Yes. Source inspection on current main shows Google Chat thread metadata still reaches sendGoogleChatMessage despite off-mode semantics, and the issue discussion includes a deployed-user confirmation.

## Solution Assessment

Yes. The best path is to finish the existing focused Google Chat delivery PR or an equivalent maintainer-chosen patch rather than queue another duplicate repair.
