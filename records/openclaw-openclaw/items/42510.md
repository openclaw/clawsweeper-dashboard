## Summary

Keep open: current main still has Google Chat paths that can send a thread to the Chat API when `replyToMode` is `off`, and two active related PRs overlap enough that a maintainer should reconcile them instead of closing this issue.

## What This Changes

The issue asks Google Chat to honor `channels.googlechat.replyToMode: "off"` by suppressing implicit thread replies like Slack does.

## Best Possible Solution

Land one Google Chat-owned fix that suppresses only implicit thread propagation when the resolved account mode is `off`, preserves explicit reply targets, and includes monitor/outbound regression coverage.

## Reproduction Assessment

Yes, source-level. An inbound Google Chat event with `message.thread.name` and `channels.googlechat.replyToMode: "off"` can still reach current code paths that pass a thread value into `sendGoogleChatMessage`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should stay at the Google Chat plugin/reply boundary and be reconciled with https://github.com/openclaw/openclaw/pull/67055 and https://github.com/openclaw/openclaw/pull/74235 before another branch is started.
