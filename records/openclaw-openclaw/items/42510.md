## Label Justifications

- `P1`: This is a confirmed Google Chat channel regression affecting real deployments by routing replies into the wrong conversation context despite an explicit off-mode setting.
- `impact:message-loss`: The issue is about Google Chat responses being misrouted into threads when the configured behavior is to suppress reply threading.

## Summary

Keep open: current main still source-reproduces Google Chat thread propagation while `channels.googlechat.replyToMode: "off"` is configured, and the overlapping fix path at https://github.com/openclaw/openclaw/pull/74235 remains open rather than merged.

## What This Changes

The issue asks Google Chat to honor the existing `channels.googlechat.replyToMode: "off"` setting by suppressing implicit thread IDs for text, media, and typing replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned boundary fix that suppresses implicit thread propagation under `replyToMode: "off"` while preserving explicit valid thread resources.

## Reproduction Assessment

Yes, source-level: configure Google Chat with `replyToMode: "off"`, process an inbound event containing `message.thread.name`, and current main can still pass that thread through typing-placeholder and attached-result send paths. I did not run a live Google Chat transport reproduction in this read-only review.

## Solution Assessment

Yes in direction, but it is not on main yet. The maintainable fix is to filter implicit Google Chat thread sources at the Google Chat boundary while keeping explicit valid thread resources working.
