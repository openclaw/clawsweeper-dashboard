## Summary

Keep open. Current main has partial stuck-session recovery, including `diagnostics.stuckSessionAbortMs` and stale/embedded-run recovery, but active `model_call` and `tool_call` stalls still classify as observe-only and never request recovery; the maintainer note and later 2026.5.7 production confirmations point to the same remaining policy gap.

## What This Changes

The issue asks for processing sessions that stop making progress to be aborted or drained automatically instead of only logging diagnostics until an external gateway restart.

## Best Possible Solution

Extend the existing generation-guarded recovery design with a maintainer-approved policy for model/tool stalls and any remaining stale-queue/orphan-prompt edge cases, with focused diagnostics, recovery-runtime, and embedded-runner regression coverage.

## Reproduction Assessment

Yes, source-level with high confidence. A stale `activeWorkKind: "model_call"` or `"tool_call"` path still classifies as `recoveryEligible: false`, and the heartbeat only calls recovery for eligible classifications or stalled embedded runs.

## Solution Assessment

Unclear as an implementation policy. The current `stuckSessionAbortMs` plus embedded-run recovery is a good base, but broadening abort/drain to model and tool stalls needs maintainer judgment to avoid killing legitimate long-running work.
