## Summary

Keep open. Current main still lacks a public `diagnostics.stuckSessionAbortMs` or equivalent active-abort threshold: the heartbeat calls stuck-session recovery after the warning threshold, but the recovery path remains warning-only for active embedded runs, active reply work, and active lane tasks unless an internal `allowActiveAbort` flag is explicitly supplied. The issue now has strong production evidence plus deterministic Telegram restart-window and cron-driven `agentTurn` reproductions, and open PR #73243 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

Add diagnostics-driven recovery so sessions stuck in `processing` are aborted and their queue lane is released after a configurable threshold instead of requiring an external gateway restart.

## Best Possible Solution

Ship PR #73243 or an equivalent narrow core diagnostics patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after a configured recovery threshold, releases the affected session lane safely, emits structured recovery diagnostics, and updates config schema, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. The issue includes production logs, RayWoo added deterministic v2026.4.25 Telegram restart-window steps, and xxtract added a v2026.4.29 cron-driven `agentTurn` path; current-main source and tests confirm the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate `diagnostics.stuckSessionAbortMs` path is the narrow maintainable solution because warning and termination have different operational risk profiles, and the current warning-only behavior is intentionally conservative.
