## Summary

Keep open. Current main has partial stuck-session recovery for stale bookkeeping and extended stalled embedded runs, but it still lacks the requested configurable abort threshold and leaves model/tool active stalls non-recovery; an open closing PR is already the right implementation path.

## What This Changes

The issue requests diagnostic recovery for sessions stuck in `processing` by aborting eligible in-flight work, releasing the lane, and exposing a configurable abort threshold.

## Best Possible Solution

Review and land PR #73243, PR #76115, or an equivalent narrow patch that separates warning and abort thresholds, aborts only eligible stuck active work, releases/drains lanes safely, and updates config schema, docs, tests, and changelog.

## Reproduction Assessment

Yes. The comments provide deterministic Telegram restart-window and cron `agentTurn` reproductions, and current-main source/tests show the remaining recovery boundary without needing a live gateway run here.

## Solution Assessment

Yes in direction, but not fully on main. A guarded abort threshold is the maintainable path because warning and termination have different operational risk, and the open PRs are the right place to settle the exact policy.
