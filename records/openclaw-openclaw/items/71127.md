## Summary

Current main and v2026.5.3 now include conservative stuck-session recovery: stale session lanes are released, and stalled embedded runs are abort-drained after an extended no-progress window, with regression coverage and docs. The remaining separately tunable abort-threshold idea is a follow-up policy/config question, not the original unrecoverable-stuck-session bug.

## What This Changes

The issue requests an in-gateway recovery path that aborts or releases sessions left in processing after stuck-session diagnostics detect them.

## Best Possible Solution

Keep the shipped v2026.5.3 recovery path and handle any separately tunable abort-threshold policy in a narrower follow-up instead of leaving this unrecoverable-stuck-session report open.

## Reproduction Assessment

Yes. The issue and comments provide production logs plus deterministic Telegram restart-window and cron agentTurn reproductions, and the current regression tests model the stale-lane and stalled embedded-run recovery paths without needing a live gateway run in this read-only review.

## Solution Assessment

Yes. The shipped solution is conservative: it releases stale bookkeeping immediately, abort-drains only stalled embedded runs after an extended no-progress window, and leaves long-running active work and tool/model-specific timeout policy to their existing owners.
