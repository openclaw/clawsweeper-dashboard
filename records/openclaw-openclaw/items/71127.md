## Summary

Keep open. Current main has partial stuck-session recovery for stale/no-active-work and stalled embedded-run cases, but active model-call and native tool-call stalls are still intentionally observe-only, and a maintainer note explicitly keeps this issue open for those recovery-policy follow-ups.

## What This Changes

The issue asks for stuck processing sessions to be aborted or released by the gateway diagnostics path so affected sessions recover without an external process restart.

## Best Possible Solution

Keep this as the canonical core recovery-policy issue and land owner-reviewed fixes for model/tool stalls, queued/no-active drain, orphan prompt handling, and pre-stream hangs once current-main repros and abort semantics are agreed.

## Reproduction Assessment

Yes, at source level with high confidence. A processing diagnostic session with stale model_call or tool_call activity is classified as session.stalled with recoveryEligible false, so the heartbeat logs recovery=none and does not request recovery.

## Solution Assessment

Unclear as a policy. The current stale-state and embedded-run recovery base is maintainable, but broadening aborts to model and native tool stalls needs maintainer-approved cancellation semantics rather than an automated classifier flip.
