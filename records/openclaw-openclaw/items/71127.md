## Summary

Keep open. Current main still lacks the requested public abort threshold, and active embedded/reply/lane work remains observe-only by default; open PR #73243 is already the implementation candidate and should not be treated as a reason to close this issue before merge.

## What This Changes

The issue requests a configurable stuck-session recovery path that aborts long-running processing sessions, releases the affected queue lane, and restores gateway progress without an external restart.

## Best Possible Solution

Ship #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. The issue has production logs plus deterministic Telegram restart-window and cron-driven agentTurn reproductions, and current-main source/tests confirm the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but it is not implemented on main. A separate diagnostics.stuckSessionAbortMs path is the maintainable approach because warning and forced termination have different operational risk profiles.
