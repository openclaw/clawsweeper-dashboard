## Summary

Keep open. Current main has partial recovery for stale/no-active-work bookkeeping and stalled embedded runs, but active `model_call` and `tool_call` stalls are still classified as observe-only, and the issue discussion includes an explicit maintainer note keeping this as the canonical follow-up surface.

## What This Changes

The issue asks OpenClaw to abort or recover sessions that remain stuck in `processing` after diagnostics detect them, instead of requiring an external gateway restart.

## Best Possible Solution

Keep this as the canonical core recovery-policy issue and land owner-reviewed fixes for remaining active model/tool stalls and post-abort edge cases, using the existing diagnostic/provider timeout surfaces where they are sufficient.

## Reproduction Assessment

Yes, at source level with high confidence: a `processing` session whose active work is stale `model_call` or `tool_call` is classified as stalled with `recoveryEligible: false`, so the heartbeat does not request recovery for that path. A live current-main gateway reproduction was not run.

## Solution Assessment

Unclear as a policy decision. The current stale-session and embedded-run recovery base is maintainable, but extending abort/drain behavior to model and tool stalls needs maintainer-approved cancellation semantics and thresholds.
