## Summary

Keep open. Current main has partial stuck-session recovery, including `diagnostics.stuckSessionAbortMs` and delayed embedded-run aborts, but stale `model_call` and `tool_call` work still remains observe-only and recent production comments report the same `recovery=none` failure shape on the latest stable release.

## What This Changes

The issue asks OpenClaw to abort or recover sessions that remain in `processing` after stuck-session diagnostics so the gateway regains forward progress without an external restart.

## Best Possible Solution

Finish the diagnostics recovery design with an owner-approved policy for stale model/tool/embedded work, queued-work drain, and post-abort prompt cleanup, then lock it with focused diagnostics and embedded-runner regression coverage plus docs/config updates.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. Current main classifies stale `model_call` and `tool_call` work as stalled with `recoveryEligible: false`, while the heartbeat only escalates embedded runs after the abort threshold.

## Solution Assessment

Unclear as a complete fix. The current `stuckSessionAbortMs` and embedded-run recovery path is the right base, but extending abort/drain behavior to model calls, tool calls, queue wakeups, and orphan prompts needs an explicit recovery policy.
