## Summary

Keep open. Current main has partial stuck-session recovery, including `diagnostics.stuckSessionAbortMs` and delayed embedded-run aborts, but active `model_call` and `tool_call` stalls still remain observe-only, matching the latest production `recovery=none` report and the maintainer note that this issue should remain open for follow-up surfaces.

## What This Changes

The issue requests self-healing stuck-session recovery so diagnostics can abort or release a processing session instead of requiring an external gateway restart.

## Best Possible Solution

Keep this as the canonical stuck-session recovery follow-up and extend the generation-guarded recovery design to safely handle proven model/tool stalls, queued-lane wakeups, and post-abort prompt repair with focused diagnostics and embedded-runner tests.

Yes, source-reproducible but not live-reproduced in this read-only pass. Current main still maps stale active `model_call` work to `session.stalled` with `recoveryEligible: false`, matching the latest production `recovery=none` log shape.

## Solution Assessment

No, closing this as implemented is not the best resolution. Current main has the right recovery foundation for stale and embedded-run cases, but active model/tool cancellation needs a maintainer policy decision before broadening abort behavior.
