## Summary

Keep open. Current main still lacks a public stuck-session abort threshold and the diagnostics heartbeat does not enable active-run abort recovery; the linked open PR #73243 is the implementation candidate and this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request diagnostics to abort and release stuck `processing` sessions after a configurable recovery threshold instead of only warning until an external gateway restart restores progress.

## Best Possible Solution

Review and land PR #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, tests, and changelog.

## Reproduction Assessment

Yes. The issue has production logs plus deterministic Telegram startup-window and cron `agentTurn` reproductions, and current-main source/tests confirm active stuck runs are still observe-only by default.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate `diagnostics.stuckSessionAbortMs` path is the narrow maintainable fix because warning and termination have different operational risk profiles.
