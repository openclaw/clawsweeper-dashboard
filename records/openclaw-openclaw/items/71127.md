## Summary

Keep open. Current main has partial stuck-session recovery, including `diagnostics.stuckSessionAbortMs` and stale embedded/tool-call recovery, but model-call stalls still classify as stalled active work without a diagnostic abort path, and fresh 2026.5.12 field reports plus an open maintainer PR show remaining core liveness work.

## What This Changes

The issue requests automatic recovery for sessions stuck in `processing` so the gateway can restore forward progress without an external restart.

## Best Possible Solution

Keep this as the canonical core liveness issue and land owner-reviewed fixes for model-call cancellation, post-abort transcript/orphan handling, queued-work wakeups, and pre-stream timeout behavior with focused regression and packaged/live proof.

## Reproduction Assessment

Yes, at source level with high confidence. A stale `model_call` is classified as `session.stalled` with `recoveryEligible: false`, and the heartbeat’s active-abort path currently covers embedded runs and blocked tool calls but not model calls.

## Solution Assessment

Unclear as an automated fix. The current partial implementation is a reasonable conservative baseline, but extending abort behavior to model calls and post-abort transcript handling needs maintainer-approved cancellation semantics, thresholds, and user-visible recovery policy.
