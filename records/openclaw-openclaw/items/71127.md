## Summary

Keep open. Current main has partial stuck-session recovery, but stalled `model_call` and `tool_call` paths still remain observe-only, and the issue discussion includes a maintainer note plus later 2026.5.7 production confirmations that keep the remaining recovery policy live.

## What This Changes

The issue asks for stuck processing sessions detected by diagnostics to abort or drain automatically instead of requiring an external gateway restart.

## Best Possible Solution

Keep this as the canonical core recovery-policy issue and land focused, owner-reviewed fixes for active model/tool stalls and remaining stale/no-active-work/orphan prompt edges on top of the existing `stuckSessionAbortMs` recovery base.

## Reproduction Assessment

Yes, at source level with high confidence. A `processing` session whose active work is `model_call` or `tool_call` and whose last progress exceeds the diagnostic threshold is classified as stalled but remains non-recovery-eligible on current main.

## Solution Assessment

Unclear as a policy. The current embedded-run/stale-session recovery base is maintainable, but extending abort/drain behavior to model and tool stalls needs maintainer-approved thresholds and ownership boundaries before automation should implement it.
