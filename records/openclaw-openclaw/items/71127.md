## Summary

Keep open. Current main has conservative stuck-session recovery for stale bookkeeping, but it still lacks a public abort threshold and the heartbeat does not enable active-run abort recovery; the issue also has a linked open fix PR, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a configurable diagnostics-driven recovery path that aborts stuck processing sessions, releases the affected queue, and emits recovery diagnostics without requiring an external gateway restart.

## Best Possible Solution

Review and land PR #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, tests, and changelog.

## Reproduction Assessment

Yes. The issue has production logs, a deterministic Telegram restart-window reproduction on v2026.4.25, and a cron-driven agentTurn reproduction on v2026.4.29; current-main source and tests confirm the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate diagnostics.stuckSessionAbortMs path is the narrow maintainable design because warning and termination have different operational risk profiles.
