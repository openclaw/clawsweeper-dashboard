## Summary

Keep open. Current main contains partial stuck-session recovery and the new abort threshold, but active `model_call` and `tool_call` stalls still classify as observe-only/recovery-none, and the latest production comment reports the same wedged `model_call` signature on v2026.5.7.

## What This Changes

The issue asks OpenClaw to recover stuck `processing` sessions by aborting stalled in-flight work and releasing session queues without requiring an external gateway restart.

## Best Possible Solution

Finish the remaining current-main recovery policy by making eligible stalled model/tool work recover without cutting off valid slow work, then cover queue drain and post-abort orphan behavior with focused tests and docs.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: a `processing` session with `activeWorkKind: "model_call"` and stale progress is classified as `session.stalled` with `recoveryEligible: false`, so the heartbeat logs `recovery=none` and does not request recovery. Production comments add deterministic Telegram/cron paths and the latest v2026.5.7 Signal/local-model reproduction.

## Solution Assessment

Unclear as an automated fix: the requested recovery direction is sound, but the remaining model/tool abort policy needs maintainer judgment so OpenClaw distinguishes genuinely wedged work from slow but valid model or tool execution.
