## Summary

Keep open. Current main still has only `diagnostics.stuckSessionWarnMs` as the public threshold, calls conservative recovery from the warning path, and keeps active embedded runs, active reply work, and active lane tasks warning-only by default. The issue has strong production evidence plus deterministic Telegram and cron reproductions, and open PR #73243 is the active implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests a configurable diagnostics recovery path that aborts stuck `processing` sessions, releases the session queue, emits recovery diagnostics, and restores progress without an external gateway restart.

## Best Possible Solution

Ship PR #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. The original report includes production logs, RayWoo added deterministic v2026.4.25 Telegram restart-window steps, and xxtract added a v2026.4.29 cron `agentTurn` trigger; current-main source and tests confirm the active abort path is still absent.

## Solution Assessment

Yes in direction, but it is not yet implemented on main. A separate `diagnostics.stuckSessionAbortMs` path wired into the existing embedded-run abort/drain mechanism is the narrow maintainable fix because warning and termination have different operational risk profiles.
