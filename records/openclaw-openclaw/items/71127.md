## Summary

Keep open. Current main has improved stuck-session classification and stale-lane cleanup, but it still does not provide the requested active-run abort threshold, and the issue is already paired with open recovery PRs that should be reviewed or replaced before closure.

## What This Changes

Request diagnostics-driven recovery for sessions stuck in `processing`, including a configurable abort threshold, in-flight run cancellation, lane release, and structured recovery diagnostics.

## Best Possible Solution

Review and land PR #73243, PR #76115, or an equivalent narrow patch that separates warning and abort thresholds, aborts only eligible stuck active work, safely releases the lane, emits structured recovery diagnostics, and updates config schema, docs, tests, and changelog.

## Reproduction Assessment

Yes. The issue has production logs plus deterministic Telegram restart-window and cron `agentTurn` reproductions in comments, and current-main source/tests confirm active stuck work remains warning-only or observe-only; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction, but not yet on main. A separate guarded abort threshold is the narrow maintainable path because warning and termination have different operational risk, and the open PRs are the right place to settle that behavior.
