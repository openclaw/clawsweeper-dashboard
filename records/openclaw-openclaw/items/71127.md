## Summary

Keep open. Current main partially addresses stuck-session recovery, but source still leaves active `model_call`/`tool_call` stalls observe-only, and recent maintainer/user comments identify those remaining cases as the same recovery failure on current stable.

## What This Changes

The issue requests that gateway diagnostics abort or release sessions that stay in `processing` past a configurable stuck-session threshold instead of requiring an external gateway restart.

## Best Possible Solution

A maintainer-reviewed fix should extend recovery only for proven unsafe active model/tool stalls, preserve false-positive protections for healthy long runs, cover stale queue and post-abort prompt behavior with focused tests, and update docs/changelog before closing this issue.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced in this read-only pass. Current main still classifies active `model_call`/`tool_call` no-progress stalls as `recovery=none`, matching the latest 2026.5.7 production log signature.

## Solution Assessment

No, current main is not yet the full solution for this issue. The embedded-run-only recovery and new abort threshold are useful, but the remaining model/tool stall and post-abort behaviors need a scoped maintainer decision and regression proof.
