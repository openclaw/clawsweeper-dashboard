## Summary

Keep open. Current main still only exposes and wires the stuck-session warning threshold, while active embedded runs, reply work, and active lane tasks remain warning-only by default; the linked closing PR #73243 is still the right implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests a configurable stuck-session abort/recovery path so diagnostics can release sessions that remain in `processing` instead of only logging warnings.

## Best Possible Solution

Ship a narrow, config-backed stuck-session recovery path: keep the existing warning threshold, add a separate conservative abort threshold, abort only eligible stuck active work after that threshold, emit structured recovery diagnostics, and update config schema, docs, and focused tests. The current best path is to review and land #73243 or an equivalent patch.

## Reproduction Assessment

Yes. The issue includes production logs, RayWoo added a deterministic v2026.4.25 restart-window reproduction, and current-main source/tests confirm active stuck work is still not aborted by default.

## Solution Assessment

Yes in direction, but it is not solved on main. A separate `diagnostics.stuckSessionAbortMs` path is safer than overloading `stuckSessionWarnMs` because warning and forced recovery have different operational risk.
