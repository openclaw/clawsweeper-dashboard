## Summary

Keep open. Current main has partial stuck-session recovery, including `stuckSessionAbortMs` and delayed embedded-run aborts, but active `model_call` and `tool_call` stalls still remain observe-only, and maintainer discussion explicitly leaves several current-main follow-up surfaces open.

## What This Changes

Request self-healing for sessions that remain in `processing` after diagnostics classify them as stuck or stalled, instead of requiring an external gateway restart.

## Best Possible Solution

Finish the current-main recovery design for model/tool stalls, stale queued work, post-abort orphan prompts, and pre-stream hangs, with focused regression tests and explicit runtime cancellation policy.

## Reproduction Assessment

Yes, source-reproducible with high confidence. A processing session whose diagnostic activity is stale with `activeWorkKind: "model_call"` or `"tool_call"` still classifies as non-recoverable and the heartbeat has no non-embedded abort branch; production comments corroborate the same `recovery=none` symptom, but I did not run a live gateway repro.

## Solution Assessment

No, current main is only a partial solution. The embedded-run threshold and stale-lane recovery are the right direction, but broadening aborts to model/tool work and changing orphan prompt behavior needs a deliberate policy and coverage rather than a blanket eligibility flip.
