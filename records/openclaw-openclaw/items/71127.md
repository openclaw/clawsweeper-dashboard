## Summary

Keep open. Current main still lacks a public stuck-session abort threshold and the heartbeat path does not enable active abort recovery; active embedded runs, reply work, and active lane tasks remain protected by default. The issue also has an open closing PR, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a configurable diagnostics-driven recovery path that aborts sessions stuck in `processing`, releases the affected queue/lane, and emits recovery diagnostics without requiring an external gateway restart.

## Best Possible Solution

Land #73243 or an equivalent narrow fix that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, and focused tests.

## Reproduction Assessment

Yes. The issue includes production logs, and RayWoo added deterministic v2026.4.25 Telegram restart-window steps; current-main source and tests confirm the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate `diagnostics.stuckSessionAbortMs` path is the narrow maintainable approach because warning and termination have different operational risk profiles.
