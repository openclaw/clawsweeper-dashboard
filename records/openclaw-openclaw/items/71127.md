## Summary

Keep open. Current main has partially implemented `diagnostics.stuckSessionAbortMs` and embedded-run recovery, but active `model_call` and `tool_call` stalls still classify as observe-only; the latest production confirmation and maintainer note both point to remaining recovery-policy work.

## What This Changes

The issue requests automatic abort/recovery for stuck `processing` sessions so queued messages can resume without restarting the gateway.

## Best Possible Solution

Finish the current-main recovery design for model/tool stalls, stale queued work, post-abort orphan prompts, and pre-stream hangs with focused regression tests and docs.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. A processing session with stale `activeWorkKind: "model_call"` or `"tool_call"` still classifies with `recoveryEligible: false`, so the heartbeat logs `recovery=none` and does not request recovery.

## Solution Assessment

Yes in direction, but not complete. The existing `stuckSessionAbortMs` plus conservative embedded-run abort is a maintainable base, but expanding it to model/tool stalls needs an explicit recovery policy and regression coverage.
