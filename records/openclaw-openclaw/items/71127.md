## Summary

Keep open. Current main has shipped partial stuck-session recovery for stale/no-active-work and stalled embedded-run cases, but stalled `model_call` and `tool_call` active work still remains observe-only, and the issue discussion includes a maintainer note explicitly keeping this issue open for those follow-up recovery-policy surfaces.

## What This Changes

The issue reports processing sessions that log stuck/stalled diagnostics but remain wedged until an external gateway restart, requesting runtime abort/drain recovery after a configurable threshold.

## Best Possible Solution

Keep this as the canonical core recovery-policy issue and land owner-reviewed fixes for remaining active model/tool stalls and post-abort edge cases after current-main repros and abort thresholds are agreed.

## Reproduction Assessment

Yes, at source level with high confidence. A `processing` session whose active work is `model_call` or `tool_call` and whose last progress exceeds the diagnostic threshold is classified as stalled but remains non-recovery-eligible on current main.

## Solution Assessment

Unclear as a policy. The current embedded-run/stale-session recovery base is maintainable, but extending abort/drain behavior to model and tool stalls needs maintainer-approved cancellation semantics and thresholds rather than an automated classifier flip.
