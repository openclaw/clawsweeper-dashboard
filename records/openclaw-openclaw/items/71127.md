## Summary

Keep open. Current main still lacks a public stuck-session abort threshold and the default recovery path remains conservative for active embedded runs, active reply work, and active lane tasks. The issue has strong production evidence plus a deterministic Telegram restart-window reproduction, and open PR #73243 is the current implementation candidate rather than a reason to close before merge.

## What This Changes

Request a configurable stuck-session abort/recovery path that cancels in-flight processing, releases the affected session queue, and emits recovery diagnostics after a processing session exceeds a threshold.

## Best Possible Solution

Ship #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, and focused tests.

## Reproduction Assessment

Yes. The issue includes production logs, and RayWoo added deterministic v2026.4.25 Telegram restart-window steps that reproduce the stuck-session symptom; current-main source and tests confirm the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate diagnostics.stuckSessionAbortMs path is the maintainable approach because warning and termination have different operational risk profiles.
