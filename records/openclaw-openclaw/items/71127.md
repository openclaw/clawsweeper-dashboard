## Summary

Keep open. Current `main` has partial stuck-session recovery, but source inspection, live issue discussion, and related open PRs still show unresolved recovery paths for model-call stalls, stale embedded-run lanes, orphan prompt handling, and pre-stream timeout behavior.

## What This Changes

The issue asks OpenClaw to abort or recover sessions that remain in `processing` after stuck-session diagnostics instead of only logging until an external gateway restart.

## Best Possible Solution

Keep this as the canonical core runtime recovery issue while maintainers review or split focused fixes for model-call cancellation, stale embedded-run handles, queued-work wakeups, orphan prompt repair, and pre-stream timeout behavior.

## Reproduction Assessment

Yes, at source level with high confidence: current `main` still leaves stalled `model_call` work outside active abort recovery, and the recovery runtime still has a `keep_lane` path for active embedded-run work. The issue discussion also contains multiple fresh production reproductions on released builds through v2026.5.12.

## Solution Assessment

Unclear as an automated fix. The narrowest maintainable path is owner-reviewed recovery policy plus regression coverage, not simply flipping all stalled active work to abortable.
