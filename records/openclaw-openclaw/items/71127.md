## Summary

Keep open. Current main and v2026.5.3 include a conservative partial recovery path, but they still do not add the requested separate abort threshold and do not abort all active model/tool-call stall shapes described in the issue; the issue also remains paired with open recovery PRs #73243 and #76115.

## What This Changes

The issue asks diagnostics to recover stuck `processing` sessions by aborting or draining in-flight work without requiring an external gateway restart.

## Best Possible Solution

Settle the recovery policy in #73243, #76115, or an equivalent narrow patch: add or explicitly reject a second-stage abort threshold, cover eligible active-run stall cases, preserve slow-run safeguards, and keep config, docs, tests, and changelog aligned.

## Reproduction Assessment

Yes. The issue has production logs plus deterministic Telegram restart-window and cron `agentTurn` reproductions in comments, and current-main source/tests still show active model/tool stall classes are not diagnostics-recovery-eligible.

## Solution Assessment

No, not yet. The v2026.5.3/main implementation is a useful partial fix for stale lanes and stalled embedded runs, but it does not implement the requested configurable active-abort policy or cover every in-flight stall shape described here.
