## Summary

Keep open. Current main has implemented `diagnostics.stuckSessionAbortMs` and conservative embedded-run recovery, but active `model_call` and `tool_call` stalls are still observe-only; the latest production confirmation reports the same `activeWorkKind=model_call ... recovery=none` shape on v2026.5.7, and the maintainer discussion explicitly kept this issue open for those remaining surfaces.

## What This Changes

The issue requests automatic recovery for processing sessions that diagnostics classify as stuck or stalled so the gateway can restore forward progress without an external restart.

## Best Possible Solution

Keep the shipped embedded-run recovery and deliberately resolve the remaining model/tool stall, stale queue wake, post-abort orphan prompt, and pre-stream timeout behaviors from current main with focused tests and docs.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. A processing session with active `model_call` progress stale past the warning and abort thresholds still classifies as `active_work_without_progress` with `recoveryEligible: false`, matching the latest production log shape.

## Solution Assessment

Yes in direction, but not complete. A guarded abort/recovery threshold is the right path for wedged sessions, while broadening it from embedded runs to model/tool stalls needs an explicit policy decision and regression coverage.
