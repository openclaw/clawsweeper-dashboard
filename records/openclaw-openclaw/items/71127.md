## Summary

Keep open. Current main still lacks a public stuck-session abort threshold and the heartbeat does not enable active-run aborts; active embedded runs, reply work, and active lane tasks remain warning-only by default. The issue has strong production evidence plus RayWoo's deterministic Telegram restart-window reproduction, and open PR #73243 is the implementation candidate rather than a reason to close before merge.

## What This Changes

The request is to add configurable stuck-session abort/recovery so processing sessions that exceed a threshold are aborted or released without requiring an external gateway restart.

## Best Possible Solution

Land #73243 or an equivalent narrow patch that keeps warning and abort thresholds separate, aborts only eligible stuck active work after the configured recovery threshold, safely releases the affected lane, emits structured recovery diagnostics, and updates config schema, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. The issue includes production logs, and RayWoo added deterministic v2026.4.25 Telegram restart-window steps that reproduce the stuck-session symptom; current-main source and tests verify that the requested active abort behavior is still absent.

## Solution Assessment

Yes in direction, but it is not implemented on main. A separate diagnostics.stuckSessionAbortMs path is the maintainable shape because warning and termination have different operational risk profiles.
