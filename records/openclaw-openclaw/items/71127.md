## Summary

Keep open. Current main still lacks the requested active stuck-session abort path: diagnostics can warn and attempt conservative lane cleanup, but active embedded runs, active reply work, and active lane tasks remain protected by default. RayWoo's deterministic restart-window reproduction and the open closing PR #73243 make this actionable but not yet implemented on main.

## What This Changes

The issue requests a configurable diagnostics recovery path that aborts stuck processing sessions, releases their queue/lane, and emits recovery diagnostics instead of warning indefinitely.

## Best Possible Solution

Ship a narrow, config-backed stuck-session recovery path that keeps the existing warning signal, adds a separate conservative abort threshold, aborts only eligible stuck active work after that threshold, releases the affected lane safely, emits structured recovery diagnostics, and updates config schema/docs/tests. Keep this issue open until #73243 or an equivalent patch lands.

## Reproduction Assessment

Yes. The issue includes production logs, RayWoo added a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source/tests confirm active stuck work is still not aborted by default.

## Solution Assessment

Yes in direction, but not yet on main. A separate diagnostics.stuckSessionAbortMs path is safer than overloading stuckSessionWarnMs because warning and termination have different operational risk profiles.
