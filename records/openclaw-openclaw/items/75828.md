## Summary

Keep open. The PR is draft/protected and introduces a P2 behavior regression: queued-only stale processing sessions become recovery-ineligible before the existing guarded recovery runtime can decide whether the lane is safe to release.

## What This Changes

The PR adds session liveness classifications and run activity tracking for diagnostics, updates OTel/docs/config metadata, emits Codex app-server progress events, and removes a stale WhatsApp diagnostic export.

## Best Possible Solution

Revise the branch so active embedded/model/tool progress is classified separately while stale queued-only bookkeeping still enters the existing guarded recovery path; coordinate with the overlapping #72010 diagnostics-progress work before landing.

## Reproduction Assessment

Yes. A focused fake-timer heartbeat test can set a processing session with queueDepth > 0 and no active run/model/tool activity, advance past diagnostics.stuckSessionWarnMs, and assert recoverStuckSession is called; the PR branch would emit session.long_running and skip recovery.

## Solution Assessment

No. The liveness-classification direction is maintainable, but treating queueDepth alone as active work is too coarse; the safer fix is to leave queued-only stale sessions recovery-eligible and let the recovery runtime decide whether active reply or lane work blocks release.
