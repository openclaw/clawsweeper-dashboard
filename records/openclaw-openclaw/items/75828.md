## Summary

Keep open. This draft, maintainer-labeled PR should not be auto-closed, and the diff introduces a concrete recovery regression: queued stale sessions become non-recovery-eligible before the existing guarded recovery runtime can inspect and release a queued-only lane.

## What This Changes

The PR adds session liveness classifications and run activity tracking for diagnostics, narrows stuck recovery to true stale bookkeeping, updates OTel/docs/config metadata, and removes a stale WhatsApp API export.

## Best Possible Solution

Land a version that preserves the existing recovery runtime as the source of truth: classify active embedded/model/tool progress separately, but still let stale queued sessions call the guarded recovery path so it can decide whether the lane has active work or can be released.

## Reproduction Assessment

Yes. A focused fake-timer heartbeat test can set a processing session with queueDepth > 0 and no active run/model/tool activity, advance past diagnostics.stuckSessionWarnMs, and assert that recoverStuckSession is called; the PR branch would emit session.long_running and skip recovery instead.

## Solution Assessment

No. The active-work classification direction is maintainable, but treating queueDepth alone as non-recoverable bypasses the narrower existing guard that checks active embedded, reply, and lane work before releasing a lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve recovery for queued stale sessions:** `src/logging/diagnostic.ts:219-226`
  - body: When `queueDepth > 0` and there is no tracked active run/model/tool activity, this branch returns `session.long_running` with `recoveryEligible: false`. That skips `recoverStuckSession` entirely, but that recovery runtime is where OpenClaw checks for active reply/lane work and releases queued-only stale lanes. A stale processing session with queued startup work can therefore remain wedged instead of draining.
