## Label Justifications

- `P2`: The issue affects advanced multi-agent CLI JSON workflows but does not make ordinary agent use or all sub-agent completion delivery unusable.
- `impact:session-state`: The reported foreground JSON result can describe the parent run while descendant session work is still active or later synthesizing.
- `impact:message-loss`: The reported CLI JSON output can omit later parent synthesis that operators expected from the multi-agent run.

## Summary

Keep open: current main and v2026.6.6 still return the foreground parent agent RPC result for `openclaw agent --json` while `sessions_spawn` descendants are explicitly non-blocking and complete through later parent-session events, so the core CLI lifecycle/JSON contract gap remains unresolved.

## What This Changes

The issue asks OpenClaw to make `openclaw agent --json` accurately represent multi-agent runs that spawn asynchronous `sessions_spawn` descendants.

## Best Possible Solution

Define the `openclaw agent --json` lifecycle contract, then either wait for descendant completion in the chosen mode or surface a clear descendants-pending/incomplete state without overloading `replayInvalid`.

## Reproduction Assessment

Yes at source level: current main waits for the parent gateway agent RPC result and writes that JSON response, while `sessions_spawn` remains non-blocking and delivers completions through later parent-session events. I did not run the reporter's Docker MAS scenario.

## Solution Assessment

No, not as originally proposed: the discussion correctly retracts changing `finalAssistantVisibleText` extraction or overloading `replayInvalid`. The safer path is a maintainer-selected CLI JSON lifecycle contract.
