## Summary

Keep open: current main and v2026.6.10 still do not fully satisfy the requested Control UI chat cleanup. The broad tool-call toggle and Activity tab exist, but partial stream fragments are still materialized around tool cards, failed activity groups still auto-open, and the requested default/config direction remains a maintainer UX decision.

## What This Changes

Requests Control UI/WebChat to reduce inline tool-activity clutter by collapsing activity/tool blocks by default, suppressing partial stream fragments, de-promoting retry/successful error blocks, and adding a default setting.

## Best Possible Solution

Define the normal-versus-debug Control UI chat visibility contract, then implement it at the chat projection/render boundary using the existing browser-local toggle and Activity tab unless maintainers deliberately choose a gateway config surface.

## Reproduction Assessment

Yes, at source level: current main still materializes stream segments around tool cards, keeps tool calls visible by default, and auto-opens failed activity groups. I did not establish a fresh live browser reproduction because this review was read-only and the screenshot was not available as an attachment.

## Solution Assessment

Unclear. Collapsing noisy tool activity is plausible, but adding gateway.controlUi.activityDefault is not clearly the best layer while a browser-local toggle and Activity tab already exist.
