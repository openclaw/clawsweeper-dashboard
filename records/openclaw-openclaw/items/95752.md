## Summary

Keep open: current main has compact Activity grouping, an Activity tab, and a browser-local show/hide tool-call toggle, but it still defaults tool calls visible, interleaves stream segments with tool cards, auto-expands errored activity, and lacks the requested `gateway.controlUi.activityDefault` decision.

## What This Changes

Request Control UI to collapse or hide tool activity by default, suppress in-progress assistant fragments around tools, and demote transient error activity after successful retry, possibly with a new gateway config option.

## Best Possible Solution

Define the normal versus debug Control UI transcript contract, reuse the Activity tab and browser-local preferences unless maintainers approve a gateway config surface, then land coordinated render-boundary fixes for fragments and non-terminal errors.

## Reproduction Assessment

Yes, at source level for the current behavior: main still defaults tool calls visible, interleaves stream segments with tool messages, and auto-opens errored activity. I did not run a live browser reproduction or inspect the missing screenshot.

## Solution Assessment

Unclear. Collapsing noisy activity is plausible, but adding `gateway.controlUi.activityDefault` is not clearly the best layer until maintainers choose between default behavior, browser-local preference, Activity-tab separation, and server config.
