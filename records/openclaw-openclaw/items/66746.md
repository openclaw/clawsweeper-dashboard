## Summary

Keep this issue open. Current main has addressed the older `allowExtension` and packaged-runtime-dependency gaps, but the actual sqlite-vec loader/readiness path still has no Windows `.dll` retry and no post-load vec0 usability probe before reporting vector readiness, so the reported Windows FTS-only degradation is not clearly implemented or safely superseded.

## What This Changes

The issue asks OpenClaw memory indexing on Windows to create and populate `chunks_vec` instead of falling back to FTS-only when standalone `node:sqlite` plus `sqlite-vec` works on the same machine.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that keeps both loader copies in sync, verifies sqlite-vec is actually usable after loading, handles the Windows DLL path behavior if confirmed, keeps the existing degraded fallback safe, and adds focused regression coverage for load-failure and load-success-without-usable-vec0 cases.

## Reproduction Assessment

Unclear. The issue includes a detailed Windows reproduction and standalone success case, but this review could only verify current-main source behavior from a Linux checkout rather than reproduce the Windows Node 24 runtime path directly.

## Solution Assessment

No current-main solution is sufficient yet. The existing `allowExtension` and packaged-dependency fixes address adjacent failures, while the narrower loader/readiness gap should be solved in the memory-host SDK loader and memory-core readiness path, potentially using #69059 as source context.
