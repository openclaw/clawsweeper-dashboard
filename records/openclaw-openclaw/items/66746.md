## Summary

Keep open: current main still lacks both a Windows sqlite-vec suffix retry and a post-load usability probe, while the related loader PR and ABI/readiness issue remain open rather than merged into a canonical fix.

## What This Changes

Windows memory indexing should create and populate `chunks_vec` instead of degrading to FTS-only when direct `node:sqlite` plus `sqlite-vec` works on the same host.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that handles the Windows load path, probes sqlite-vec usability before marking vectors ready, preserves degraded FTS fallback, and adds focused regression plus Windows smoke proof.

## Reproduction Assessment

No high-confidence live reproduction in this review. The reporter supplied concrete Windows commands and current main still has the same source-level loader/readiness/warning path, but I did not run the Windows runtime path directly.

## Solution Assessment

No current-main solution is sufficient yet. The maintainable fix should be at the sqlite-vec loader/readiness boundary, coordinated with the open Windows retry PR and the open post-load probe issue.
