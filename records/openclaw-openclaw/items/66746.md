## Summary

Keep open: current main fixes adjacent `allowExtension` and packaged-runtime-dependency problems, but the active sqlite-vec loader/readiness path still lacks a Windows DLL retry and a post-load vec0 usability probe, so the reported Windows FTS-only degradation is not implemented or safely superseded. ## What This Changes

The issue reports that Windows memory indexing in OpenClaw 2026.4.14 degrades to FTS-only because sqlite-vec is unavailable in OpenClaw even though direct node:sqlite plus sqlite-vec works on the same machine.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that keeps the source and package loader copies synchronized, handles the Windows DLL path behavior if confirmed, validates sqlite-vec is actually usable before reporting vector readiness, preserves the safe FTS fallback, and adds focused regression coverage plus a changelog entry.
