## Summary

Keep open: current main still has the same Windows-sensitive sqlite-vec loader/readiness shape described in the report, and the related Windows retry PR and ABI/readiness issue are still separate open work rather than a shipped canonical fix.

## What This Changes

The issue asks OpenClaw to make Windows memory indexing create and populate `chunks_vec` instead of degrading to FTS-only when direct `node:sqlite` plus `sqlite-vec` works on the same machine.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that handles the Windows loader behavior, verifies sqlite-vec is actually usable before marking vectors ready, preserves degraded fallback, and includes focused regression coverage plus a changelog entry.

## Reproduction Assessment

No high-confidence live reproduction was established here. The reporter provided concrete Windows commands and current main still has the same degraded source path, but this review did not run the Windows 11 plus Node 24 runtime path.

## Solution Assessment

No current-main solution is sufficient yet. The narrow maintainable fix should be in the memory-host sqlite-vec loader/readiness contract and memory-core vector initialization, coordinated with #69059 and #65156 rather than treating the older allowExtension fix as complete.
