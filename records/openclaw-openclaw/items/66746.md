## Summary

Keep open: current main still lacks a Windows sqlite-vec suffixless retry and a post-load usability probe, while the related loader PR and broader ABI/readiness issue remain open rather than merged into one canonical fix.

## What This Changes

The issue reports that Windows memory indexing in 2026.4.14 falls back to FTS-only and never creates `chunks_vec` even though a standalone `node:sqlite` plus `sqlite-vec` probe succeeds.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that applies Windows suffixless loading across bundled and platform-variant paths, probes sqlite-vec usability before marking vectors ready, preserves degraded FTS fallback, and includes focused regression plus Windows smoke proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The reporter supplied concrete Windows commands and current main still has the same source-level loader/readiness gaps, but a Windows packaged runtime smoke is still needed.

## Solution Assessment

No current-main solution is sufficient yet. The maintainable fix should be at the sqlite-vec loader/readiness boundary and coordinated with https://github.com/openclaw/openclaw/pull/69059 and https://github.com/openclaw/openclaw/issues/65156.
