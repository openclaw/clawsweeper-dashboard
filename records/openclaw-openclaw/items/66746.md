## Summary

Keep open: current main still loads bundled sqlite-vec without a Windows DLL retry or post-load usability probe, and vector readiness still depends on loader success before `chunks_vec` creation. The adjacent open PR and open ABI/readiness issue are useful context, but they have not landed a canonical fix for this Windows report.

## What This Changes

The issue reports that Windows memory indexing in 2026.4.14 degrades to FTS-only even though a standalone `node:sqlite` plus `sqlite-vec` smoke test works on the same machine.

## Best Possible Solution

Land one canonical memory/sqlite-vec fix that keeps the loader package surface synchronized, handles the confirmed Windows load behavior, verifies sqlite-vec is actually usable before marking vectors ready, preserves degraded fallback, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Unclear. The report provides concrete Windows commands and a standalone `node:sqlite` success case, and current main still has the same degraded warning path, but this review did not run the Windows runtime path directly.

## Solution Assessment

No current-main solution is sufficient yet. The narrow maintainable path is to fix the memory-host SDK loader/readiness contract and memory-core vector initialization rather than treating the closed allowExtension fix or package-dependency mirror as complete coverage.
