## Summary

Keep open: current main still has the BR ninth-digit variant problem, but this PR is not mergeable as-is because it rewrites an authorized target to the allowlist form, which downstream reaction, quote, and session code treats as the delivery/chat key; the external PR also lacks real WhatsApp runtime proof.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit equivalence across direct-target normalization, outbound resolution, inbound allowlist checks, monitor routing, heartbeat recipients, SDK exports, and tests.

## Best Possible Solution

Put BR direct-target equivalence into WhatsApp authorization and ingress matching while preserving the requested or observed chat target for delivery, reactions, quote lookup, and session keys, then validate on current plugin paths with redacted live WhatsApp proof.

## Reproduction Assessment

Yes, by source inspection. Current main exact-matches outbound allowlist entries, self-chat identity, and generic ingress identity keys, so the two BR phone forms remain distinct in the affected WhatsApp paths.

## Solution Assessment

No. The equivalence rule is the right direction, but returning the allowlist's canonical target mixes authorization with delivery/session routing; the safer fix keeps equivalence in matching while leaving the real chat target intact.

## Review Findings

- **[P1] Preserve the requested WhatsApp target after authorization:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: When a BR variant matches, returning `matchedAllowTarget` rewrites the caller's chat target to the allowlist entry. Reaction handling sends to the resolved target, so a reaction for a message in the requested chat can be sent with the alternate BR remote JID. Keep authorization equivalence separate from the delivery target.
  - confidence: 0.91

  - body: The same `matchedAllowTarget` rewrite feeds outbound session routing. In per-peer modes, sending to one BR variant while the allowlist stores the other records the turn under the allowlist peer key, splitting history from the active chat and making later implicit routing stale.
  - confidence: 0.88
