## Summary

Keep this PR open: the underlying WhatsApp BR ninth-digit bug is still source-reproducible on current main, but the PR currently has blocking target-rewrite findings, is conflicting with the current plugin layout, and lacks required real WhatsApp behavior proof.

## What This Changes

The PR adds WhatsApp BR ninth-digit equivalence to target normalization, allowlist and inbound checks, heartbeat/session paths, SDK exports, and regression tests.

## Best Possible Solution

Rework the PR so BR equivalence only authorizes and matches variants while preserving the requested or observed chat target for delivery, reactions, quote lookup, and session keys, then rebase it onto the current extensions/whatsapp layout with redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-reproducible: current main exact-matches outbound allowFrom and same-phone inbound identities, so the linked BR variants still miss each other. I did not establish a live WhatsApp reproduction in this read-only review.

## Solution Assessment

No. BR equivalence is the right fix direction, but the current PR rewrites delivery and routing to the allowlist's canonical variant; the safer solution keeps equivalence in matching while returning the requested or observed chat target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the requested WhatsApp target after authorization:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: When a BR variant matches, returning `matchedAllowTarget` rewrites the chat used after authorization. Reaction authorization returns `resolution.to`, and `handleWhatsAppAction` sends the reaction to that value, so a reaction for a message in the requested chat can be sent to the alternate JID instead.
  - confidence: 0.91
- **[P2] Keep session routing keyed to the requested chat target:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: The same `matchedAllowTarget` rewrite feeds gateway send routing. In per-peer modes, sending to one BR variant while the allowlist stores the other records the turn under the allowlist peer key, splitting history from the active chat and making later implicit routing stale.
  - confidence: 0.88
