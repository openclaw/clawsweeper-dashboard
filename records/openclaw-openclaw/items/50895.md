## Summary

Keep open: the reported WhatsApp BR ninth-digit bug is still valid on current main, but this PR has blocking correctness findings because it rewrites a matched BR variant to the allowlist target and still lacks real WhatsApp behavior proof.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit target equivalence across outbound resolution, inbound access control, monitor routing, heartbeat recipients, SDK exports, and regression tests.

## Best Possible Solution

Add BR direct-target equivalence in WhatsApp authorization and ingress matching while preserving the requested or observed chat target for sends, reactions, quote lookup, and session keys, then validate with redacted live WhatsApp proof.

## Reproduction Assessment

Yes, from source inspection rather than a live WhatsApp run. Current main exact-matches outbound allowFrom, self-phone identity, and generic ingress identities, while the PR's target-rewrite regression is traceable through reaction and session-routing callers.

## Solution Assessment

No. BR equivalence is the right direction, but returning the allowlist's canonical target mixes authorization with delivery and routing; the safer fix keeps equivalence in matching while returning the requested or observed chat target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the requested WhatsApp target after authorization:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: When a BR variant matches, returning `matchedAllowTarget` rewrites the caller's chat target to the allowlist entry. Reaction authorization passes `resolution.to` into `sendReactionWhatsApp`, so a reaction for a message in the requested chat can be sent to the alternate JID; keep equivalence as an authorization check and return the requested normalized target for delivery.
  - confidence: 0.91
- **[P2] Keep session routing keyed to the requested chat target:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: The same `matchedAllowTarget` rewrite flows into gateway send routing. In per-peer modes, sending to one BR variant while the allowlist stores the other records the turn under the allowlist peer key, splitting history from the active chat and making later implicit routing stale.
  - confidence: 0.88
