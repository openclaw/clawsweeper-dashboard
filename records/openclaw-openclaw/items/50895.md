## Summary

Keep open: the underlying WhatsApp BR ninth-digit mismatch is still source-reproducible on current main, but this PR is not mergeable as-is because it rewrites matched variants to the allowlist canonical target and still lacks real WhatsApp proof.

## What This Changes

The PR adds BR ninth-digit equivalence across WhatsApp normalization, outbound and inbound allowlist checks, command authorization, main last-route and heartbeat handling, SDK exports, and regression tests.

## Best Possible Solution

Implement BR variant equivalence at the current extensions/whatsapp boundary while preserving the requested or observed chat target for delivery, reactions, quote lookup, and session keys, then land it with focused tests and redacted live WhatsApp proof.

## Reproduction Assessment

Yes. Current main exact-matches outbound allowFrom and self-phone identities and has no BR variant helpers, so the reported variant mismatch is source-reproducible; I did not establish a live WhatsApp reproduction.

## Solution Assessment

No. BR equivalence is the right fix direction, but rewriting the requested target to the allowlist canonical variant is not the safest solution because downstream reaction and session routing use that resolved target.

  - body: When a BR variant matches, returning matchedAllowTarget rewrites the chat used after authorization. Reaction authorization returns resolution.to and sendReactionWhatsApp sends to that value, so a reaction for a message in the requested chat can be sent to the alternate JID instead.

  - confidence: 0.88
