## Summary

Keep open. Current main still exact-matches the relevant WhatsApp direct targets, so the useful BR ninth-digit equivalence is not implemented, but this PR is not merge-ready because it rewrites authorized outbound targets to the allowlist form, lacks real WhatsApp proof, and now conflicts with the current plugin layout.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit equivalence across outbound target resolution, inbound access control, monitor routing, heartbeat recipient authorization, SDK exports, and regression tests.

## Best Possible Solution

Add BR direct-target equivalence in WhatsApp authorization and ingress matching while preserving the requested or observed chat target for sends, reactions, quote lookup, and session keys, then validate with redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-reproducible: current main exact-matches outbound allowFrom and same-phone ingress identities, and the PR's target-rewrite regression is traceable through reaction and session-routing callers. I did not run a live WhatsApp setup in this read-only review.

## Solution Assessment

No. BR equivalence is the right direction, but returning the allowlist's canonical target mixes authorization with delivery and routing; the safer fix keeps equivalence in matching while returning the requested or observed chat target and rebases into the current plugin API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the requested WhatsApp target after authorization:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: When a BR variant matches, returning `matchedAllowTarget` rewrites the caller's chat target to the allowlist entry. Reaction authorization returns `resolution.to`, and `handleWhatsAppAction` passes it to `sendReactionWhatsApp`, so a reaction for a message in the requested chat can be sent to the alternate JID. Keep equivalence as the authorization check and return the requested normalized target for delivery.
  - confidence: 0.91
- **[P2] Keep session routing keyed to the requested chat target:** `src/whatsapp/resolve-outbound-target.ts:50`
  - body: The same `matchedAllowTarget` rewrite flows into gateway send routing. In per-peer modes, sending to one BR variant while the allowlist stores the other records the turn under the allowlist peer key, splitting history from the active chat and making later implicit routing stale.
  - confidence: 0.88
