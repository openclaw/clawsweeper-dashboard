## Summary

This PR should stay open because current main still exact-matches the affected WhatsApp BR target forms, but the branch should not merge as-is: it rewrites the requested chat target to the allowlist variant and still lacks redacted real WhatsApp runtime proof.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit equivalence across target normalization, outbound allowlist resolution, inbound access control, monitor routing, heartbeat recipients, SDK exports, and tests.

## Best Possible Solution

Add BR direct-target equivalence in the current extensions/whatsapp authorization and ingress paths while preserving the requested or observed chat target for delivery, reactions, quote lookup, and session keys, then validate with redacted live WhatsApp proof.

## Reproduction Assessment

Yes. Source inspection on current main shows exact outbound allowFrom matching, exact self-phone matching, and generic exact ingress identity matching, while the PR's target-rewrite regression is traceable through the reaction and session-route callers.

## Solution Assessment

No. BR equivalence is the right direction, but returning the allowlist's canonical target mixes authorization with delivery and session routing; the safer fix keeps equivalence in matching while returning the actual requested chat target.
