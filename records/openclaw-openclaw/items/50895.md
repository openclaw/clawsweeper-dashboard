## Summary

Keep this PR open. The reported WhatsApp BR ninth-digit problem is still real on current main, but the branch is not merge-ready: it is dirty against current main's plugin layout and its outbound allowlist match rewrites explicit chat targets, which can break reactions and per-peer session routing.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit target equivalence across normalization, outbound and inbound allowlist checks, command/self-chat/main-route/heartbeat paths, SDK exports, and regression tests.

## Best Possible Solution

Port the useful equivalence into the current extensions/whatsapp plugin layout as an internal direct-target comparison helper. Use it for authorization and owner matching, but keep delivery, reactions, quote lookup, and per-peer session routing keyed to the requested normalized chat target unless a caller explicitly asks for a canonical allowlist value.

## Reproduction Assessment

Yes. Configure WhatsApp allowFrom with +553598627740 and send or receive from +5535998627740; current main uses exact comparisons, while the PR authorizes the pair but returns the allowlist variant downstream.

## Solution Assessment

No. The equivalence direction is useful, but returning the allowlist target and widening old public SDK exports are not the narrowest maintainable fix; authorization equivalence should be separated from delivery/session target selection in the WhatsApp plugin.
