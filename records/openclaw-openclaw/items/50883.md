## Summary

Keep this PR open. Current main still lacks `openclaw sessions --kind`, and the PR remains a focused CLI feature, but it should not merge as submitted because the default route-first parser would not forward the new option and the public sessions docs are not updated.

## What This Changes

The PR adds a repeatable `openclaw sessions --kind <type>` filter with a shared parser helper, command filtering, JSON metadata, Commander wiring, and focused tests.

## Best Possible Solution

Revise the feature so both CLI execution paths share the same kind parsing contract: parse and forward repeated/comma-separated `--kind` from Commander and `parseSessionsRouteArgs`, apply it in `sessionsCommand` for text and JSON output, document it on the sessions CLI page, and keep the session-kind type shared to avoid drift.

## Reproduction Assessment

Yes. By source inspection, a default route-first invocation such as `openclaw sessions --kind group` matches the exact `sessions` route, but `parseSessionsRouteArgs` does not parse or forward `kind`, so the command would receive no filter unless route-first is disabled.

## Solution Assessment

No. The PR takes the right command/helper direction, but the narrow maintainable solution must also update the route-first parser, route tests, and docs so the new option behaves consistently in normal CLI use.
