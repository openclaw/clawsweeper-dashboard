## Summary

Keep open. Current main still has no session-kind filter, so the PR is not obsolete, but the submitted patch is incomplete because normal route-first `openclaw sessions --kind ...` invocations would bypass Commander and drop the filter.

## What This Changes

The PR adds a repeatable `openclaw sessions --kind <type>` filter, a shared kind parser helper, command filtering, JSON metadata, Commander wiring, and focused tests.

## Best Possible Solution

Land the feature only after both CLI paths share the same kind parsing contract: Commander and `parseSessionsRouteArgs` should forward repeated/comma-separated values into `sessionsCommand`, the command should filter text and JSON rows consistently, route tests should cover the fast path, and the public sessions docs plus changelog should describe the new option.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a normal route-first invocation such as `openclaw sessions --kind group` matches the exact `sessions` route, but `parseSessionsRouteArgs` does not forward `kind`, so `sessionsCommand` receives no filter.

## Solution Assessment

No. The helper and command filtering direction is reasonable, but the submitted patch is not the best complete fix until it updates the route-first parser/tests and user-facing docs/changelog for the new CLI option.
