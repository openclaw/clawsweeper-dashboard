## Summary

Keep open. The report is source-corroborated against current main: the documented OpenAI/Codex route should work, the current Codex plugin still imports a private SDK subpath, the alias guard only recognizes bundled Codex paths, and the OpenAI tool-schema normalizer does not repair missing array `items`; the open draft fix at https://github.com/openclaw/openclaw/pull/81118 covers only part of the runtime-module failure.

## What This Changes

Report requests fixes for beta upgrade regressions where Codex-backed OpenAI routes fall back after a private SDK module-resolution failure and legacy OpenAI-Codex routes fail invalid array tool schemas, plus related update, doctor, and cron diagnostics.

## Best Possible Solution

Land or adapt the focused Codex module-resolution fix, add a separate OpenAI-facing regression that repairs or rejects array schemas without `items`, and split lower-priority cron/doctor diagnostic follow-ups if they remain after the primary route is healthy.

## Reproduction Assessment

No, not as a full live current-main reproduction in this review. The module-resolution failure is source-reproducible from the npm-installed Codex alias gap, while the OpenAI schema failure is only partially source-corroborated because the sanitized memory tool schema is not in the repo.

## Solution Assessment

No single current or proposed change is the best complete solution. The best path is to review the active Codex runtime PR for the module failure, then handle the schema validation and lower-priority diagnostics as narrower follow-ups.
