## Summary

Keep open. Current main still has the Kimi/Moonshot provider-usage gap: the usage registry and public fetch helper exports stop at the existing providers, the bundled Kimi provider registers auth/catalog/runtime behavior without a usage hook, and the prior quota implementation remains closed without merging. The cross-referenced fuel-gauge issue is broader Control UI product work, not a canonical replacement for this provider-specific request.

## What This Changes

Request to add Kimi/Moonshot usage and balance reporting to existing provider-usage surfaces such as app/status usage, chat status, and CLI status output.

## Best Possible Solution

Decide the supported Kimi/Moonshot usage and credential contract first, then implement provider-owned usage auth and snapshot hooks for Kimi/Moonshot with docs and focused provider/status tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence absence check: Kimi, Kimi Coding, and Moonshot do not resolve through the usage registry and the bundled Kimi provider does not provide a usage hook.

## Solution Assessment

Unclear until maintainers choose the Kimi Code billing/quota contract. The existing provider-owned fetchUsageSnapshot and usage-auth hooks are the right seam, but automation should not choose private Kimi billing/web-token behavior versus documented Moonshot balance on its own.
