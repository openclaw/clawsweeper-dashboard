## Summary

Keep this open. Current main still excludes Kimi, Kimi Coding, and Moonshot from the closed provider-usage registry, and the bundled provider plugins do not provide usage-auth or usage-snapshot hooks even though the plugin API supports that boundary.

## What This Changes

Request to add Kimi/Kimi Coding and Moonshot provider usage and balance reporting to OpenClaw's app/status surfaces, using the prior Moonshot balance and Kimi billing endpoint approach as a starting point.

## Best Possible Solution

Implement provider-owned usage auth and snapshot fetching for Kimi Coding and Moonshot, extend the generic usage-id/alias registry only enough for existing status, Control UI, and model-auth surfaces to find them, and document the supported credentials and known endpoint limits.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: resolveUsageProviderId("kimi") and resolveUsageProviderId("moonshot") cannot resolve today, and the status/dashboard paths depend on that resolver before loading usage.

## Solution Assessment

Yes for the provider-usage portion: the maintainable fix is a small registry extension plus Kimi/Moonshot plugin hooks, not a core switchboard special case. The /quota part is unclear because current docs expose /status, /usage, and openclaw status --usage rather than a documented /quota command.
