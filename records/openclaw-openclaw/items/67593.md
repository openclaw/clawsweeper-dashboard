## Summary

Keep this issue open. Current main still excludes Kimi, Kimi Coding, and Moonshot from the provider usage registry and the bundled Kimi/Moonshot provider plugins do not implement provider-owned usage auth or snapshot hooks, while the existing plugin API is sufficient for a focused implementation.

## What This Changes

Request Kimi/Kimi Coding/Moonshot provider usage and balance visibility across the existing app, status, and provider usage surfaces, using the closed #20343 work as implementation context.

## Best Possible Solution

Implement provider-owned usage auth and snapshot fetching for Kimi Coding and Moonshot inside the bundled provider boundary, extend the generic usage provider id/label/discovery path just enough for existing status, Control UI, and usage reporting surfaces to find them, and document supported credentials and limitations. Treat /quota as a separate command-surface decision unless maintainers explicitly want a new alias or command.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: resolveUsageProviderId("kimi") and resolveUsageProviderId("moonshot") cannot return a provider id because the closed registry excludes them, and both bundled provider plugins lack usage hooks.

## Solution Assessment

Yes for the provider-usage part: the narrow maintainable fix is to use the existing provider plugin hooks plus a small registry/alias extension, not a core switchboard special case. The /quota part is unclear because current docs point users to /status, /usage, and openclaw status --usage rather than a documented /quota command.
