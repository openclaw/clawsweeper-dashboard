## Summary

Keep open. Current main still has the Kimi/Moonshot provider-usage gap: Kimi and Moonshot are documented/configured model providers, but neither resolves through the provider-usage registry or implements the provider-owned usage hooks needed by status/UI usage surfaces. The old `/quota` wording is partly stale against current docs, and the linked fuel-gauge issue is broader Control UI product work, not a canonical replacement for this provider-specific request.

## What This Changes

Add Kimi/Moonshot usage and balance reporting to the existing provider-usage/status/UI surfaces.

## Best Possible Solution

Choose the supported Kimi/Moonshot billing endpoint and credential contract, then implement provider-owned usage auth and fetchUsageSnapshot hooks with docs and focused status/provider-usage tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence absence check: Kimi, Kimi Coding, and Moonshot do not resolve through the usage registry and their bundled provider plugins do not provide usage hooks.

## Solution Assessment

Unclear until maintainers choose the Kimi Coding versus Moonshot balance contract. The existing provider-owned resolveUsageAuth and fetchUsageSnapshot hooks are the right seam, but automation should not choose private web-token billing behavior on its own.
