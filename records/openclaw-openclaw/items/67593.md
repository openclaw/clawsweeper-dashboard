## Summary

Keep open. Current main still excludes Kimi, Kimi Coding, and Moonshot from the provider-usage registry, the bundled provider plugins do not implement provider-usage hooks, and the related quota PRs are closed without merging; the remaining question is the supported Kimi Code usage/billing endpoint and credential contract.

## What This Changes

Add Kimi/Moonshot provider usage and balance reporting to OpenClaw’s existing status, usage, and Control UI auth-status surfaces.

## Best Possible Solution

Decide the supported Kimi Code usage contract first, then implement provider-owned Kimi/Moonshot usage auth and snapshot hooks for the existing status/usage surfaces with docs and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence absence check: Kimi, Kimi Coding, and Moonshot do not resolve through the usage registry and the bundled plugins do not provide usage hooks.

## Solution Assessment

Unclear until maintainers choose the Kimi Code billing/quota contract. The documented Moonshot balance piece fits the existing provider-owned hook design, but private Kimi billing/web-token fallback should not be added by automation without an explicit decision.
