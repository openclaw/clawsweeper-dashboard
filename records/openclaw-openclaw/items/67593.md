## Summary

Keep this open. Current main still excludes Kimi, Kimi Coding, and Moonshot from provider-usage resolution, and the bundled provider plugins do not implement the usage-auth or usage-snapshot hooks that existing status and Control UI paths depend on.

## What This Changes

Add Kimi/Moonshot provider usage and balance reporting to the existing status, app, and chat usage surfaces.

## Best Possible Solution

Implement provider-owned usage auth and snapshot fetching for Kimi Coding and Moonshot, extend the generic usage-id/alias registry only enough for existing usage surfaces to discover them, and document the supported credentials and endpoint limits.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: resolveUsageProviderId("kimi") and resolveUsageProviderId("moonshot") cannot resolve today, and status/dashboard usage loading depends on that resolver before fetching usage.

## Solution Assessment

Yes for the provider-usage gap. The narrow maintainable solution is registry support plus Kimi/Moonshot plugin hooks, not a core switchboard special case; the separate /quota command request needs maintainer product judgment because current docs expose /status, /usage, and openclaw status --usage instead.
