## Summary

Keep open: this PR targets a valid source-reproducible retry bug, but the current patch removes the replay-metadata guard too broadly and can retry after same-attempt replay-unsafe tool side effects; it also still lacks real behavior proof from a runner or gateway setup.

## What This Changes

The PR removes the `replayMetadata` side-effect guard from `shouldRetrySilentErrorAssistantTurn` and updates incomplete-turn tests to allow retry after prior session side effects.

## Best Possible Solution

Split accumulated session replay history from current failed-turn side-effect evidence, keep retries blocked for current-attempt replay-unsafe tools and deliveries, and add redacted runner or gateway proof before merge.

## Reproduction Assessment

Yes at source level: current main still blocks the linked empty provider-error retry when accumulated replay metadata reports side effects, and this PR's unsafe-retry regression is visible because `replaySafe:false` tool metadata is not covered by `hasAttemptTerminalState`. I did not run a live OpenRouter 5xx reproduction in this read-only review.

## Solution Assessment

No: removing the guard entirely is too broad because it also removes current-attempt `replaySafe:false` tool protection. The best fix is a current-attempt side-effect split, not an unconditional accumulated-replay bypass.
