## Summary

Keep this issue open. Current main and the latest release still require a Node-hosted OpenClaw runtime, while the new agent-core and harness surfaces remain TypeScript/plugin contracts rather than a stable cross-runtime protocol for bare-metal, C/Rust/WASM, or ESP32-class ports.

## What This Changes

The issue requests runtime-agnostic OpenClaw support for non-Linux and non-Node.js devices such as embedded systems and microcontrollers.

## Best Possible Solution

Keep this as the canonical embedded/no-Node portability discussion; if maintainers accept the direction, design a small versioned serialized compatibility protocol with conformance fixtures and an external reference adapter rather than decoupling the whole core in one sweep.

## Reproduction Assessment

Not applicable. This is a new portability and interoperability capability request, and source/docs checks clearly show current OpenClaw still intentionally runs on Node-hosted platforms.

## Solution Assessment

No, not as a wholesale decoupling of the existing core from Node. The maintainable path would be a maintainer-approved protocol boundary with security semantics and conformance tests before any embedded or no-Node reference runtime.

## Risks / Open Questions

- A portable agent-core protocol would need explicit compatibility and versioning semantics across tool calls, streaming events, sessions, memory, auth/model selection, cancellation, terminal outcomes, and platform adapters.
- No-OS or microcontroller execution changes the security model for storage, credentials, network access, updates, and tool execution instead of inheriting the Node-hosted Gateway assumptions.
- Closing this as a duplicate of the distributed-runtime RFC would lose the embedded/no-Node scope because that RFC is about splitting Node-hosted gateway and runtime processes.
