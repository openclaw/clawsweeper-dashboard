## Summary

Keep open. Current main still ships a Node-hosted TypeScript/Gateway runtime with in-process agent-harness extension points, while this request is for a runtime-agnostic agent-core protocol that could support bare-metal, embedded, C/Rust/WASM, or ESP32-class ports. The existing maintainer comment also explicitly kept this open and correctly treats MimiClaw as useful prior art rather than a reason to close.

## What This Changes

The issue requests a runtime-agnostic OpenClaw core or protocol that can run without Linux or Node.js on bare-metal, embedded, C/Rust/WASM, or ESP32-class environments.

## Best Possible Solution

Keep this open for maintainer product/API design. A strong solution would define a stable runtime-agnostic agent-core protocol with schemas for tool calls, memory/session state, streaming/events, auth/model selection, platform adapters, and conformance tests or a small reference implementation that proves an OpenClaw-compatible loop can run outside the Node-hosted Gateway.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is static and high-confidence: package metadata, runtime guard code/tests, and install docs all still require an OS-hosted Node runtime, while agent-harness APIs alias the embedded runner rather than defining a portable protocol.

## Solution Assessment

Unclear. A runtime-agnostic protocol is a plausible direction, but the issue is too broad for an automatic fix PR; the current plugin harness is not sufficient because it remains an experimental trusted-plugin seam inside OpenClaw.
