## Summary

Keep this issue open as the canonical product/API discussion. Current main remains explicitly Node-hosted, the agent-harness surface is an in-process plugin contract rather than a serialized cross-runtime protocol, and the narrower browser/WASM issue was closed in favor of this broader portability request. MimiClaw is useful external prior art, but supporting compatible embedded runtimes requires an explicit maintainer decision on protocol scope, security, versioning, and ownership.

## What This Changes

Request a runtime-agnostic OpenClaw agent interface and lightweight reference path for bare-metal, C/C++, Rust, WASM, or microcontroller environments.

## Best Possible Solution

Retain TypeScript/Node as the reference implementation and, only if maintainers endorse the direction, design a small versioned serialized compatibility protocol with security semantics, conformance fixtures, and one external reference adapter that proves an OpenClaw-compatible loop can run outside the host process.

## Reproduction Assessment

Not applicable. This issue requests a new runtime and interoperability capability rather than reporting broken existing behavior; source and documentation clearly establish that current OpenClaw intentionally requires Node.

## Solution Assessment

No, not as a wholesale decoupling of the existing core from Node. If the product direction is accepted, a narrow versioned compatibility protocol and conformance suite is safer and more maintainable than creating a parallel core implementation inside this repository.

## Risks / Open Questions

- A portable agent-core protocol would need explicit compatibility and versioning guarantees across tool calls, streaming events, provider/model selection, authentication, sessions, memory, cancellation, terminal outcomes, and platform adapters.
- Supporting constrained devices changes the security and trust model because filesystem, networking, credential storage, tool execution, updates, and local persistence cannot inherit the Node-hosted Gateway assumptions.
- A wholesale core decoupling or second implementation could create competing orchestration paths and substantial long-term conformance drift without a deliberately limited protocol boundary.
