## Summary

Keep open. Current main still does not expose a Swagger/OpenAPI artifact, generator, Swagger UI, or `/openapi.json` route for the Gateway HTTP APIs; the existing `config.schema` RPC surface is adjacent but not a programmatic OpenAPI contract for HTTP clients. The related config OpenAPI PR was closed unmerged and explicitly scoped out non-config Gateway APIs, so this remains valid maintainer API-boundary work.

## What This Changes

The issue requests a Swagger/OpenAPI schema for OpenClaw APIs so users can generate custom clients for programmatic access.

## Best Possible Solution

Define the stable public Gateway HTTP surface, then add a generated or committed OpenAPI artifact with docs and drift checks; optionally serve it behind the existing Gateway auth boundary at an authenticated discovery endpoint such as `/openapi.json`. Keep WebSocket RPC protocol schemas and config schema tooling separate unless maintainers explicitly choose to model them in OpenAPI.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and docs inspection gives high-confidence absence proof: there is no schema artifact, generator, Swagger UI, or `/openapi.json` route for the Gateway HTTP APIs.

## Solution Assessment

Unclear until maintainers decide the public API boundary. A narrow maintainable solution would document/generate only the stable HTTP routes instead of blindly exposing every route or mixing WebSocket/config schema surfaces into one spec.
