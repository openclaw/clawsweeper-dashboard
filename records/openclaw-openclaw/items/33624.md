## Summary

Keep this open: current main and v2026.6.10 expose documented Gateway HTTP surfaces and protocol/config schemas, but they still do not publish an OpenAPI/Swagger document for generating clients. The remaining work needs a maintainer decision on which Gateway, operator, and plugin routes are stable enough to describe as a public API contract.

## What This Changes

The issue requests publishing an OpenAPI/Swagger schema so users can generate programmatic clients for OpenClaw endpoints.

## Best Possible Solution

Maintainers should first define the stable public Gateway HTTP contract, then publish a narrow OpenAPI artifact and tests for that contract without automatically including every operator or plugin route.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Current-main source and docs show programmatic HTTP surfaces exist, but no OpenAPI/Swagger schema is published.

## Solution Assessment

Unclear until maintainers define the public API boundary. A schema is a plausible user-facing solution, but auto-generating every route before that decision would be the wrong layer for the security and stability contract.
