## Summary

Keep open for maintainer product and security review. Current main exposes codegen-ready Gateway JSON schemas and an opt-in admin HTTP RPC endpoint, but it does not publish Swagger/OpenAPI; the remaining decision is which transport and method set OpenClaw will support as a stable external client contract.

## What This Changes

The issue requests a published Swagger/OpenAPI schema so developers can generate programmatic OpenClaw clients.

## Best Possible Solution

Adopt `packages/gateway-protocol` as the primary generated-client contract, then add a separately scoped OpenAPI document for the opt-in admin HTTP RPC allowlist only if maintainers intentionally commit to that HTTP API and its security/versioning guarantees.

## Reproduction Assessment

Not applicable; this requests a new published API artifact, and current-main inspection confirms there is no existing Swagger/OpenAPI contract whose failure could be reproduced.

## Solution Assessment

No, not as broadly proposed. Generating Swagger from all endpoints would conflate distinct transports and ownership boundaries; a protocol-first artifact plus an optional, allowlist-scoped admin HTTP OpenAPI document is narrower and safer.

## Risks / Open Questions
