## Summary

Keep open: this PR targets the real memory-core duplicate light-summary bug, but the proposed cutoff can still re-emit unchanged later-day daily-memory entries because daily reingestion refreshes `lastRecalledAt`; the supplied proof is tests/build/source-grep only, not a real after-patch dreaming run.

## What This Changes

The PR exports a memory-core phase-signal reader and filters light-dreaming candidates whose `lastRecalledAt` is not newer than their previous `lastLightAt`.

## Best Possible Solution

Keep the repair at the memory-core light selection and phase-signal boundary, but base freshness on a signal that advances only for genuinely new source or recall content and cover unchanged multi-day plus newly re-recalled cases.

## Reproduction Assessment

Yes. Source inspection shows current main can reprocess unchanged daily files on later dreaming days and refresh `lastRecalledAt`, which means this PR's cutoff can still treat old emitted content as fresh; I did not run an overnight live reproduction in this read-only review.

## Solution Assessment

No. Filtering before light ranking is the right layer, but `lastRecalledAt > lastLightAt` is not the best freshness contract because daily reingestion can advance `lastRecalledAt` without new memory content.
