## Summary

Keep open: the bug is real and the source-level direction is useful, with production confirmation in the discussion, but the PR currently hides terminal timeout results, skips the existing A2A follow-up on accepted/no-reply compensation, and lacks the required changelog entry.

## What This Changes

The PR adds post-timeout compensation for `sessions_send` by re-reading target session history after `agent.wait` timeouts and returning late replies or `accepted` instead of an immediate timeout.

## Best Possible Solution

Revise this PR into one focused fix that distinguishes soft wait-window expiry from terminal run timeout, preserves the baseline reply guard, starts the existing A2A follow-up for accepted still-running runs, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A focused mock where `agent` succeeds and `agent.wait` returns `timeout` reproduces current main returning a hard timeout; PR-specific regressions are reproducible by mocking a terminal timeout snapshot or accepted compensation with no new reply.

## Solution Assessment

No as written. The compensation boundary is narrow and maintainable, but it must preserve terminal timeout signals and keep background announce delivery alive before this is the best fix.

- **[P1] Preserve terminal timeout results before compensating:** `src/agents/tools/sessions-send-tool.ts:344-370`
  - body: `agent.wait` can return `status: "timeout"` for a wait-window expiry and for an already-ended aborted run. This branch compensates every timeout, and because the helper treats a successful `chat.history` read as acceptance, a terminal timeout with no new reply can be reported as `accepted` instead of the hard timeout callers need for diagnostics.
  - confidence: 0.88

The target run can still finish later, but the existing announce flow will not wait on it or deliver the eventual result, making nonzero-timeout accepted delivery weaker than fire-and-forget.
  - confidence: 0.84
