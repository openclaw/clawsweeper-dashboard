## Summary

Keep open. The PR still targets a real `sessions_send` timeout edge, but it cannot merge as written because it bypasses current main's terminal-timeout and delayed A2A safeguards, and the external contributor proof gate is still missing.

## What This Changes

The PR adds a post-timeout history-read compensation helper and `sessions_send` tests so late replies can convert an initial wait timeout into `ok` or `accepted`.

## Best Possible Solution

Rebase and consolidate the compensation with current main: preserve terminal timeouts, return `ok` only for a real new post-baseline reply, and keep delayed A2A alive when there is no immediate reply.

## Reproduction Assessment

Yes. Source inspection and focused mocks can reproduce the review findings by making agent start succeed, the first `agent.wait` return timeout, then varying terminal metadata, a post-baseline history reply, or no reply; I did not run tests because this was a read-only review.

## Solution Assessment

No as written. The compensation direction is narrow and useful, but this patch must be rebased onto current main and must not turn terminal timeouts into accepted results or drop the delayed A2A wait path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve terminal timeouts before compensating:** `src/agents/tools/sessions-send-tool.ts:344-370`
  - body: Current main treats `agent.wait` timeout metadata such as `endedAt`, `stopReason`, or `livenessState` as a terminal run timeout. This PR compensates every timeout first and then returns `accepted` whenever `chat.history` succeeds, which can hide hard run timeouts from callers unless the history read itself fails.
  - confidence: 0.9
- **[P2] Keep delayed A2A alive after accepted compensation:** `src/agents/tools/sessions-send-tool.ts:364-370`
  - body: When no new reply is found, this accepted branch returns without calling `startA2AFlow(undefined, runId)`. That drops the current delayed announce wait path, so a target run that finishes shortly after the first wait can complete without the existing follow-up delivery.
  - confidence: 0.87
