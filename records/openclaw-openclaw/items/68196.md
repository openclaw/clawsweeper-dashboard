## Summary

Keep this PR open: the late-reply compensation idea is still useful, and another contributor reported the same production lag pattern, but the current branch regresses `sessions_send` delivery metadata and the new proof does not actually show the timeout edge running in a real setup.

## What This Changes

The PR adds a `compensateAfterWaitTimeout` helper, wires it into `sessions_send` timeout handling, and adds focused unit coverage for late replies, accepted/no-reply compensation, compensation failure, and fire-and-forget behavior.

## Best Possible Solution

Keep the compensation helper, but preserve the existing A2A delivery metadata while adding a compensation note and require a redacted real run that exercises the timeout edge before merge.

## Reproduction Assessment

Yes, partially from source inspection: current main already avoids a hard timeout for non-terminal waits by returning accepted and starting delayed A2A, but it still does not do an immediate post-timeout history compensation read for a reply that materializes just after the first wait window. I did not run tests because this was a read-only review.

## Solution Assessment

No as written: post-timeout compensation is a reasonable fix boundary, but replacing the canonical delivery metadata with helper-local metadata is not the safest contract. Preserve the current `pending`/`skipped` announce metadata and add compensation detail without changing caller semantics.
