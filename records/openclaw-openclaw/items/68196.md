## What This Changes

The PR adds `compensateAfterWaitTimeout` plus `sessions_send` timeout handling and tests so late post-timeout replies can return `ok` and no-immediate-reply cases can return `accepted`.

## Best Possible Solution

Rebase or replace the PR with compensation that runs only after a successful start and non-terminal soft wait timeout, returns `ok` only for a real post-baseline reply, preserves hard terminal timeout, and keeps delayed A2A delivery alive.

## Reproduction Assessment

Yes, source inspection gives a high-confidence focused path: start the target run, make the first `agent.wait` return a timeout, then vary terminal metadata, post-baseline reply, or no reply. I did not run tests because this was a read-only cleanup review.

## Solution Assessment

No as written. Post-timeout compensation is the right boundary, but this implementation must be rebased onto current main and must not hide terminal timeouts or drop delayed A2A follow-up.
