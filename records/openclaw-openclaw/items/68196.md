## Summary

Keep this PR open, but do not merge it as written. The linked reports and production comments show a real `sessions_send` timeout problem on current main, and the compensation direction is useful; however, the patch can mask terminal aborted runs as `accepted` and the accepted/no-reply path still skips the existing background A2A announce flow.

## What This Changes

The PR adds post-timeout compensation for `sessions_send` by re-reading target session history after `agent.wait` times out, returning `ok` for a late reply or `accepted` when no new reply is visible, with focused tests.

## Best Possible Solution

Revise this PR into one focused fix that distinguishes soft wait-window expiry from terminal run timeout, preserves the baseline guard, starts the existing A2A flow for accepted still-running runs, and records the user-facing behavior change in the changelog.

## Reproduction Assessment

Yes. A focused mock where `agent` succeeds and `agent.wait` returns `timeout` reproduces current main returning a hard timeout; PR-specific regressions are reproducible by mocking a terminal timeout snapshot or accepted compensation with no new reply.

## Solution Assessment

No as written. The compensation boundary is narrow and useful, but the implementation must keep terminal timeout signals intact and preserve background announce delivery before it is the best maintainable fix.

- **[P1] Preserve terminal timeout results before compensating:** `src/agents/tools/sessions-send-tool.ts:344-370`
  - body: `agent.wait` can return `status: "timeout"` for an already-ended aborted run as well as for wait-window expiry. Keep terminal timeout results, such as those with ended/stop metadata, on the hard-timeout path.

The target run can continue, but no background A2A flow waits on it and announces the eventual result, making nonzero-timeout accepted delivery weaker than fire-and-forget.
