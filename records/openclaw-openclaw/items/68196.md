## Summary

Keep open: this is a focused fix for a real `sessions_send` timeout bug, but the PR cannot merge as-is because it is missing contributor real-behavior proof and regresses current-main terminal-timeout and delayed-A2A behavior.

## What This Changes

Adds a post-timeout `sessions_send` compensation helper and tests so late replies after `agent.wait` timeouts can return `ok` or `accepted` while fire-and-forget sends stay accepted.

## Best Possible Solution

Rebase and consolidate the compensation with current main: preserve terminal timeouts, return `ok` only for a real new post-baseline reply, and keep delayed A2A alive when there is no immediate reply.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection supports focused mocks where `agent.wait` times out after a successful start, then terminal metadata, a post-baseline reply, or no reply determines the correct path; I did not run tests because this was a read-only review.

## Solution Assessment

Is this the best way to solve the issue? No as written: post-timeout compensation is the right boundary, but this patch must be rebased onto current main and must not hide terminal timeouts or drop delayed A2A follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve terminal timeouts before compensating:** `src/agents/tools/sessions-send-tool.ts:344-370`
  - body: Current main treats timeout metadata such as `endedAt`, `stopReason`, or `livenessState` as a terminal run timeout. This PR compensates every timeout first, so a terminal timeout can become `accepted` whenever the history read succeeds instead of preserving the hard timeout signal.
  - confidence: 0.9
- **[P2] Keep delayed A2A alive after accepted compensation:** `src/agents/tools/sessions-send-tool.ts:364-370`
  - body: When no new reply is found, this accepted branch returns without calling `startA2AFlow(undefined, runId)`. That drops the current delayed announce wait path, so a target run that finishes shortly after the first wait can complete without the existing follow-up delivery.
  - confidence: 0.87
