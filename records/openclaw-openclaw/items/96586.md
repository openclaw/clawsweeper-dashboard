## Label Justifications

- `P2`: This is a normal-priority bundled voice-call plugin bug with a clear source-level cause and limited implementation surface.
- `impact:session-state`: The failure is drift between persisted voice-call session state and the active in-memory manager state after completion or restart.
- `impact:message-loss`: A completed call transcript can be persisted while the agent-facing status path reports the call as missing.

## Summary

Keep open: current main and latest release still use active in-memory voice-call lookups for status, and an open linked fix PR is already the implementation candidate. This issue should close only after a corrected persisted-status fallback lands.

## What This Changes

The issue asks for voice-call status lookups to fall back to the persisted call store when completed calls are no longer in the active manager maps.

## Best Possible Solution

Land one corrected status-scoped persisted lookup that returns the newest canonical call record by call id or provider call id, then close this issue after the merged PR is on main.

## Reproduction Assessment

Yes. Source inspection shows terminal calls are persisted and then removed from active indexes, while status paths only check active manager lookups before returning `found: false`; I did not place a live call in this read-only review.

## Solution Assessment

Yes. A status-scoped persisted lookup is the narrow maintainable fix; broadening all manager active-call lookups would need separate ownership and caller proof.
