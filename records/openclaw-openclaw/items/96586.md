## Summary

Keep open: the reported voice-call status gap is still present on current main and in v2026.6.10, and the open candidate PRs have not merged yet. The issue should remain the canonical report until a reviewed persisted-store fallback lands.

## What This Changes

The issue reports that voice-call status lookup misses completed calls after in-memory eviction even though persisted call records and transcripts remain in plugin state.

## Best Possible Solution

Land one status-scoped persisted lookup that returns the newest matching call record for call id and provider call id, then close this issue against the merged commit.

## Reproduction Assessment

Yes from source inspection: complete or restart past a call so active maps no longer contain it, then query get_status while plugin-state history still has the record. I did not place a live call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to keep the existing status surfaces and add a bounded persisted-record fallback by call id and provider call id; a separate get_transcript action is optional product surface and not needed to repair the broken existing behavior.
