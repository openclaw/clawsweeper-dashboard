## Label Justifications

- `P2`: This is a bounded voice-call plugin bug with a clear source-level cause and existing candidate PRs, not a core runtime outage.
- `impact:session-state`: The failure is drift between active in-memory call state and persisted voice-call state after completion or restart.
- `impact:message-loss`: Agent-facing status can report a completed call as missing even though transcript-bearing call state was persisted.

## Summary

Keep open: current main and the latest release still have the reported voice-call status gap, and the issue is now paired with open fix PRs that should be reviewed or consolidated rather than closing the issue before a merge.

## What This Changes

The issue reports that `voice_call`/`voicecall.status` can return `found:false` for completed calls after in-memory eviction even though the call record and transcript remain in persisted plugin state.

## Best Possible Solution

Keep this issue open and land one focused voice-call plugin fix that falls back to persisted call records on status misses, with regression coverage for call id and provider call id lookup after terminal eviction.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path by completing or restarting past a call so active maps no longer contain it, then querying status while plugin-state history still contains the terminal record. I did not run a live telephony call in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing persisted call record store on status misses for both call id and provider call id. A new transcript-specific action is optional product surface and is not needed to repair the existing status contract.
