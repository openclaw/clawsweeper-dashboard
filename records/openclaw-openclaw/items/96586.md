## Label Justifications

- `P2`: This is a normal-priority voice-call plugin bug with clear source evidence and a limited affected surface.
- `impact:session-state`: The issue is about drift between active in-memory call state and persisted voice-call session state after completion or restart.
- `impact:message-loss`: The agent can lose access to the completed call transcript through status even though transcript data was persisted.

## Summary

Keep open: current main and v2026.6.10 still use active-only voice-call status lookups, and the linked candidate fix PRs have not merged, so this issue should remain the canonical tracker until a clean fix lands.

## What This Changes

The issue asks voice-call status surfaces to fall back to persisted call records when active manager lookups miss after completion, eviction, or restart.

## Best Possible Solution

Land a status-scoped persisted lookup that returns the newest matching call record by call id or provider call id, while keeping active manager methods active-only.

## Reproduction Assessment

Yes, source-reproducible: terminal calls are persisted and then removed from active maps, while every status lookup on current main checks only those active maps. I did not place a live telephony call in this read-only review.

## Solution Assessment

Yes for the issue direction: a status-scoped persisted-store fallback is the narrowest maintainable fix. Broadening `CallManager.getCall` itself would be riskier because other callers use those methods as active-call guards.
