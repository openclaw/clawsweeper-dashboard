## Label Justifications

- `P2`: This is a normal-priority voice-call plugin bug that can make completed call transcript content unreachable through the documented status path.
- `impact:message-loss`: The agent-facing status path can report a completed call as missing even when the conversation transcript was persisted.
- `impact:session-state`: The failure is caused by drift between active in-memory call state and persisted voice-call state after finalization or restart.

## Summary

Keep open: current main still has the reported voice-call status gap. The documented `get_status` tool and gateway status method only check active in-memory call maps, while completed calls are persisted and then removed from those maps, so a completed call can have a stored transcript but still return `found: false`.

## What This Changes

The issue asks for voice-call status lookup to find completed persisted call records after the active manager entry is gone.

## Best Possible Solution

Keep the existing voice-call plugin surface and add a bounded persisted-record lookup for status misses, covering both call id and provider call id without adding config or core ownership.

## Reproduction Assessment

Yes from source inspection: finalize a call, then query status after the active maps no longer contain the call while the persisted history still does. I did not place a live call in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: the maintainable fix is to reuse the existing persisted call history/store lookup when active status lookup misses. Adding a new `get_transcript` action may be useful later, but it is not required to fix the broken existing `get_status` behavior.
