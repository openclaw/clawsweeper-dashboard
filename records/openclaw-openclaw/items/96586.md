## Label Justifications

- `P2`: This is a normal-priority voice-call plugin bug with a clear source-level cause and limited surface area.
- `impact:message-loss`: The agent-facing status path can report a completed call as missing even when transcript data was persisted.
- `impact:session-state`: The failure is drift between active in-memory call state and persisted voice-call state after finalization or restart.

## Summary

Keep open: current main and latest release still have the reported voice-call status gap. The agent tool and gateway status method only check active in-memory call maps, while terminal calls are persisted and then removed from those maps, so a completed call can have stored transcript state and still return `found: false`.

## What This Changes

The issue requests that voice-call status lookup fall back to persisted call records when a completed call is no longer present in the in-memory manager.

## Best Possible Solution

Keep the existing voice-call tool and gateway surfaces, and add a bounded persisted-record lookup for status misses by call id or provider call id inside the voice-call plugin.

## Reproduction Assessment

Yes from source inspection: complete or restart past a call so active maps no longer contain it, then query `get_status` while plugin-state history still has the record. I did not place a live call in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: reusing the existing persisted call record store on status misses is the narrowest maintainable fix. A new `get_transcript` action may be useful later, but it is not needed to repair the broken existing status behavior.
