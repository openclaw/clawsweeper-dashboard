## Label Justifications

- `P2`: This is a normal-priority voice-call bug with clear source evidence, bounded plugin scope, and open repair candidates.
- `impact:session-state`: Completed call state can drift between persisted plugin state and the agent-visible status path after in-memory eviction.
- `impact:message-loss`: The status miss can hide a completed call transcript from workflows that need to report what was said.

## Summary

Keep open: current main and v2026.6.10 still have active-only voice-call status lookups, and https://github.com/openclaw/openclaw/pull/96624 is an open closing candidate rather than a merged fix.

## What This Changes

The issue requests a persisted call-record fallback for voice-call status lookups after completed calls leave the in-memory manager.

## Best Possible Solution

Land one corrected candidate PR that keeps active manager lookups active-only and adds a status-scoped newest persisted-record fallback, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes from source inspection: complete or restart past a call so active maps no longer contain it, then query `get_status` while plugin-state history still has the record. I did not place a live call in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable repair is a persisted fallback behind status reads that returns the newest matching record. Broadening all manager lookups or adding a new transcript action is not needed to fix the existing status behavior.
