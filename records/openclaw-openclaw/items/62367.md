## Summary

Keep open. Current main still has Codex OAuth support for `gpt-5.4-mini` but not a first-class `openai-codex/gpt-5.4-nano` route; direct OpenAI nano support is separate, so the reported Codex runtime behavior is not proven fixed.

## What This Changes

Ensure `openai-codex/gpt-5.4-nano` either resolves and reports as nano in the Codex runtime or fails explicitly instead of normalizing or falling through to another provider/model.

## Best Possible Solution

Let the OpenAI plugin own the decision: if Codex OAuth supports `gpt-5.4-nano`, add the Codex nano constant, dynamic resolver, runtime preference, catalog synthesis, docs, changelog, and focused regression tests mirroring the mini path; if it is unsupported, add provider-owned unsupported-model handling so configured nano cannot silently normalize or fall back.

## Reproduction Assessment

Yes for source-level reproduction: current main has no `openai-codex/gpt-5.4-nano` resolver/catalog path and returns `undefined` for unrecognized Codex model ids. The live production fallthrough was not replayed in this read-only sweep.

## Solution Assessment

Unclear until maintainers confirm upstream Codex OAuth nano availability. Adding nano is the narrowest fix only if the backend accepts it; otherwise an explicit unsupported-model guard is the safer provider-owned solution.
