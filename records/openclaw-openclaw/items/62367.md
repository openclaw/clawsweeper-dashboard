## Summary

Keep open. Current main still lacks a first-class `openai-codex/gpt-5.4-nano` route, and the best evidence now points toward adding an explicit unsupported-model guard rather than silently normalizing or falling through.

## What This Changes

The issue asks OpenClaw to ensure `openai-codex/gpt-5.4-nano` either runs and reports as nano or fails explicitly instead of normalizing to `gpt-5.4` or falling through to another provider.

## Best Possible Solution

Add a provider-owned unsupported-model guard for `openai-codex/gpt-5.4-nano` that points users to direct `openai/gpt-5.4-nano` and prevents cross-provider fallback for this unsupported Codex route.

## Reproduction Assessment

Yes for source-level reproduction: current main still has no `openai-codex/gpt-5.4-nano` resolver, catalog, manifest, or suppression path, while model-not-found can enter the configured fallback loop. I did not run the reporter’s live production session.

## Solution Assessment

Yes, explicit provider-owned rejection is the narrowest maintainable fix given the public OpenAI availability statement; adding a Codex OAuth nano route should wait for maintainer or upstream evidence that it is actually supported.
