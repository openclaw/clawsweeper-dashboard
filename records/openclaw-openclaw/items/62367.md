## Summary

Keep open. Current main still does not provide a first-class `openai-codex/gpt-5.4-nano` route, while adjacent direct OpenAI nano support exists, so the reported Codex OAuth model-identity/fallback problem is not proven fixed.

## What This Changes

The issue asks OpenClaw to preserve `openai-codex/gpt-5.4-nano` as a real Codex runtime route or fail explicitly instead of silently normalizing or falling through.

## Best Possible Solution

Let the OpenAI plugin own the policy: add a real Codex OAuth nano route only if upstream supports it, otherwise add an explicit unsupported-model guard so configured nano cannot silently normalize or fall through.

## Reproduction Assessment

Yes for source-level reproduction: current main has no `openai-codex/gpt-5.4-nano` resolver, catalog, manifest, or docs path, and missing model resolution can enter the configured fallback loop. The live production YouTube path was not replayed.

## Solution Assessment

Unclear until maintainers confirm upstream Codex OAuth nano support. Enabling nano is the narrow fix if supported; otherwise explicit provider-owned rejection/suppression is the safer maintainable fix.
