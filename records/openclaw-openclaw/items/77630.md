## Summary

Keep open: the CLI preflight direction is good and the supplied terminal proof covers the main login failure, but the Gateway `channels.start` preflight is currently placed after the registry-normalization exit that produces the opaque error.

## What This Changes

The PR adds shared channel config-block validation to CLI channel login and Gateway channel start, with focused tests and a changelog entry.

## Best Possible Solution

Keep the CLI pre-save failure, but move or duplicate the Gateway config-block check so known raw channel ids are rejected with actionable config guidance before active-registry normalization can short-circuit.

## Reproduction Assessment

Yes at source level: current main saves auth before Gateway reconciliation, and the Gateway start handler can exit with `invalid channels.start channel` before config guidance when the channel is not in the active registry. I did not run a live WhatsApp QR login because this is a read-only review and that path mutates auth state.

## Solution Assessment

No as-is. The CLI half is the right narrow fix, but the Gateway half should run the config-block preflight before active plugin-registry normalization or handle the `!channelId` branch with static/raw channel-id guidance.
