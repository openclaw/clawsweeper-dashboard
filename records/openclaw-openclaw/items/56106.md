## Summary

Keep this issue open. Current main still lacks source, test, changelog, release, or live Windows/QQBot proof that Chinese transcript JSONL corruption is fixed; the QQBot text path is UTF-8-oriented, but the final transcript append still delegates to the pinned Pi SessionManager and there is no end-to-end CJK regression through transcript storage and memory indexing.

## What This Changes

The issue reports corrupted Chinese text in Windows QQBot transcript JSONL/session and memory files and asks that transcript writes preserve UTF-8 bytes.

## Best Possible Solution

Add a focused Windows/CJK regression that feeds a QQBot Chinese message through gateway decode, event dispatch, content assembly, reply-run transcript persistence, SessionManager append, and session-memory indexing, then fix the first boundary where the original text is lost. If the failure is before persistence, keep the fix in QQBot input handling; if it is at persistence, harden the SessionManager append boundary or update the pinned dependency while preserving the parentId chain.

## Reproduction Assessment

No, not yet. The report gives a concrete Windows/Node/QQBot example, but current main has no automated or live reproduction that proves where the text changes from valid Chinese to replacement characters or mojibake.

## Solution Assessment

Unclear. Forcing explicit UTF-8 writes is reasonable hardening, but current Node string writes are already UTF-8 by default, so the narrowest maintainable fix depends on an end-to-end fixture that isolates whether corruption happens in QQBot input decoding or transcript persistence.
