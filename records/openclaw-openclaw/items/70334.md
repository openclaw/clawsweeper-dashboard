## Summary

Keep this open. Current main has important mitigations for compaction retry hangs and stale processing recovery, but I did not find a focused current-main regression proving the exact overflow-compaction-success retry lane always terminalizes and releases the session without relying on watchdog recovery.

## What This Changes

The issue reports that a successful context-overflow auto-compaction retry can leave the session lane stuck in `processing` with `queueDepth=0` until gateway restart.

## Best Possible Solution

Add or land focused coverage for overflow-compaction success followed by retried prompt success, error, and lost-terminal cases, then ensure the active embedded run and session lane clear without requiring gateway restart.

## Reproduction Assessment

Yes for a source-level mocked path: simulate a context-overflow prompt error, successful forced compaction, then a retried attempt that succeeds, errors, or never emits a terminal event, and assert the session lane clears. No live long-transcript reproduction was established against current main.

## Solution Assessment

Unclear for current main. The aggregate retry timeout and stalled-run recovery are useful mitigations, but the maintainable fix should prove the specific overflow-success retry lifecycle and coordinate with the open diagnostic heartbeat PR.
