## Summary

Keep this open. The report is recent, corroborated, and current main still has the same overflow-compaction success retry lane; the Codex app-server idle timeout is a useful mitigation, but I did not find a focused regression proving that this exact compaction-success retry path clears the active session/lane and accepts follow-up channel messages after the retried prompt hangs, times out, or errors.

## What This Changes

Ensure a session returns to idle after context-overflow auto-compaction succeeds and the retried prompt finishes or fails, instead of leaving all channels blocked until a gateway restart.

## Best Possible Solution

Add a targeted regression for overflow-compaction success followed by a retried prompt that succeeds, errors, and fails to produce a terminal event; the implementation should terminalize the retry, clear the active embedded run and session lane, and allow a follow-up channel message without relying on a gateway restart.

## Reproduction Assessment

Yes for an automated path: mock a prompt context-overflow error, successful compaction, and a retried attempt that times out or never emits a terminal event, then assert the active run/lane clears and a follow-up message is accepted. A live 185+ message reproduction was not run in this read-only review.

## Solution Assessment

Unclear for current main. The Codex app-server timeout is a good mitigation, but the maintainable fix should prove the outer overflow-compaction retry lifecycle releases the session lane instead of depending on diagnostics or unrelated timeout coverage.
