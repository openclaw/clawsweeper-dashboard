## Summary

Keep open. Current main has useful retry, cleanup, and watchdog mitigation around overflow compaction, but I did not find high-confidence proof that the reported post-compaction retry lane always terminalizes and releases the processing session without relying on recovery; open PR https://github.com/openclaw/openclaw/pull/76657 also changes an overlapping compaction-retry diagnostic path.

## What This Changes

The issue reports sessions staying in `processing` with `queueDepth=0` after context-overflow auto-compaction succeeds and the prompt is retried.

## Best Possible Solution

Add or land focused coverage for overflow-compaction success, retried prompt error, and lost-terminal cases, then ensure the active run, diagnostic state, and session lane clear without a gateway restart.

## Reproduction Assessment

Yes for a source-level mocked path: simulate a context-overflow prompt error, successful forced compaction, then a retried attempt that succeeds, errors, or loses terminal progress and assert the session lane clears. No live long-transcript reproduction was established against current main.

## Solution Assessment

Unclear for current main. The retry and cleanup surfaces are the right places to fix this, but the maintainable path should prove the specific overflow-success retry lifecycle and coordinate with the open diagnostic heartbeat PR.
