## Summary

Keep open. Current main still has a WhatsApp Baileys auto-reply path that forwards each deliver callback to the provider without a same-content guard, and the existing tests still allow separate block and final callbacks to become separate provider sends.

## What This Changes

The issue requests fixing deterministic duplicate WhatsApp Baileys auto-reply delivery where the same text and correlation id are sent twice for one inbound message.

## Best Possible Solution

Add a focused WhatsApp same-correlation duplicate-delivery regression, then suppress only equivalent repeat delivery at the narrowest boundary that preserves legitimate block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The live report supplies deterministic Baileys steps and logs, and current main exposes a unit path by invoking the captured WhatsApp deliver callback twice with equivalent normalized text and expecting one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard looks safest if the duplicate only reaches the Baileys callback, while a shared dispatcher change needs proof it will not alter valid block/final sequencing.
