## Summary

Keep open. The report is recent and concrete, and current main still forwards each WhatsApp dispatcher `block`/`final` delivery callback to the Baileys send path without a WhatsApp-specific same-correlation/same-content idempotence guard; the older generic block-streaming dedupe does not prove this reported WhatsApp double-fire is fixed.

## What This Changes

Request to stop WhatsApp Baileys auto-replies from sending the same generated reply twice for one inbound correlationId.

## Best Possible Solution

Add a focused WhatsApp same-content block/final duplicate regression, then suppress only the duplicate delivery at the narrowest boundary that preserves intentional multi-block, media, tool, and error/fallback sends.

## Reproduction Assessment

Yes. The issue gives deterministic live steps and logs, and current source provides a focused unit reproduction shape by invoking the WhatsApp captured deliver callback with equivalent block/final content and expecting one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard is the safer first fix if the double-fire is Baileys-specific, while changing the shared dispatcher needs proof that valid block/final and media sequencing is preserved.
