## Summary

Keep open. The report is recent, concrete, and current main still lacks a focused WhatsApp same-correlation/same-content outbound idempotence guard at the Baileys auto-reply delivery boundary; existing shared block-streaming dedupe helps related cases but does not prove this WhatsApp double-delivery path is fixed.

## What This Changes

The issue reports duplicate WhatsApp Baileys auto-reply sends for one inbound DM, with the same correlationId/run/connection delivered twice through the buffered reply dispatcher path.

## Best Possible Solution

Add a focused regression for the reported same-correlation WhatsApp block/final duplicate, then suppress only duplicate same-content delivery at the narrowest proven boundary while preserving intentional separate block, final, and media sends.

## Reproduction Assessment

Yes. The issue provides deterministic live steps/logs, and current source gives a high-confidence unit path by invoking the captured WhatsApp dispatcher `deliver` callback twice with equivalent block/final content and expecting one provider send after the fix.

## Solution Assessment

Unclear until the regression pins the emitting layer. A narrow WhatsApp delivery guard is safest if only Baileys auto-replies double-fire; a shared dispatcher change is acceptable only with proof that valid block/final and media sequencing is preserved.
