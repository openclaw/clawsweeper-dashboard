## Summary

Keep open. Current main still has a WhatsApp auto-reply path that forwards every dispatcher deliver callback to the provider, while existing coverage allows block and final callbacks to become separate sends rather than proving same-content duplicate suppression.

## What This Changes

The issue requests that WhatsApp Baileys auto-replies deliver only once when duplicate same-content payloads are emitted for one inbound message/correlation id.

## Best Possible Solution

Add a focused WhatsApp same-correlation duplicate-delivery regression and suppress only same-content repeat delivery at the narrowest boundary that preserves intentional multi-block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The issue supplies live WhatsApp steps/logs, and current main can be unit-reproduced by invoking the captured WhatsApp dispatcher deliver callback twice with equivalent block/final text and expecting only one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard is safest if the duplicate is only in the Baileys auto-reply callback, while a shared dispatcher change needs proof it will not alter valid block/final sequencing.
