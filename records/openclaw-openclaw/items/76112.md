## Summary

Keep open. The report is recent, concrete, and current main still forwards each WhatsApp dispatcher deliver callback to `deliverReply` without a same-correlation/same-content idempotence guard, while the existing tests do not cover suppressing duplicate block/final text for one inbound WhatsApp message.

## What This Changes

The issue requests that WhatsApp Baileys auto-replies deliver only one outbound message per inbound correlation when block/final dispatch emits equivalent payloads.

## Best Possible Solution

Add a focused WhatsApp duplicate-delivery regression and suppress only same-message, same-sendable-content repeat deliveries while preserving intentional multi-block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The reporter supplied deterministic live Baileys logs, and current main exposes a focused unit path by invoking the captured WhatsApp dispatcher deliver callback twice with equivalent text for the same message and verifying only one provider send after the fix.

## Solution Assessment

Unclear whether the final fix should live in the WhatsApp callback or the shared dispatcher until the regression is written. The narrowest maintainable starting point is a WhatsApp delivery guard unless tests prove the same duplicate can only originate from a generic dispatcher bug.

## Repair Work Prompt

Fix the WhatsApp Baileys duplicate auto-reply delivery reported in https://github.com/openclaw/openclaw/issues/76112. Current main forwards every deliverable `dispatchReplyWithBufferedBlockDispatcher` callback through `dispatchWhatsAppBufferedReply` to `deliverReply`, so duplicate same-content block/final callbacks for the same inbound WhatsApp message can send two provider messages. Add a focused regression in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts` that invokes the captured dispatcher deliver callback twice with equivalent normalized text for the same message/correlation and expects one `deliverReply` call and one `rememberSentText` call. Implement the narrowest idempotence guard, likely in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts`, unless a shared dispatcher regression proves the bug is generic. Preserve intentional multi-block replies, distinct media payloads, tool media delivery, provider-unaccepted retry semantics, and existing block-streaming final/media dedupe behavior. Do not add a config flag. Add a user-facing `CHANGELOG.md` fix entry with appropriate human attribution only if known. Validate with the targeted WhatsApp test plus the shared reply/block-streaming tests, and use the repo changed gate/Testbox policy before handoff.
