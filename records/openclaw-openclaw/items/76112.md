## Summary

Current main now handles the reported block-plus-final duplicate by suppressing final payload delivery after a successful block stream, so the WhatsApp Baileys path should no longer send the same completed reply twice for that root cause.

## What This Changes

The issue requests that WhatsApp auto-replies deliver only once per inbound message/correlation id instead of sending both a block reply and a duplicate queued final reply.

## Best Possible Solution

Keep the shared final-after-block suppression as the canonical fix, since it addresses the duplicate at the common reply-payload boundary while preserving aborted-stream fallback and unsent media delivery.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: the reported old block-plus-final path is clear, but current main drops final payloads after a successful block stream before WhatsApp delivery can see them.

## Solution Assessment

Yes. The shared payload-builder suppression is narrower and safer than a WhatsApp-only duplicate guard because it prevents duplicate final sends for every block-streamed channel while keeping media-only finals and aborted-stream fallbacks intact.
