## Summary

Keep open. The report is recent, concrete, and not superseded by the already-shipped WhatsApp block-streaming work: current main still has a WhatsApp delivery callback that will pass every deliverable block/final payload to `deliverWebReply`, and the focused WhatsApp test currently locks in separate block and final delivery rather than same-content suppression.

## What This Changes

The issue reports deterministic duplicate WhatsApp auto-reply delivery for the same correlationId through the buffered reply dispatcher.

## Best Possible Solution

Add a focused regression for same-correlation WhatsApp block/final duplicate delivery, then suppress same-content duplicate delivery at the narrowest boundary that preserves intentional multi-block and media sends.

## Reproduction Assessment

Yes. The issue provides concrete live reproduction steps and logs, and current source/test inspection gives a unit-level reproduction shape by invoking the WhatsApp deliver callback twice with equivalent block/final content and expecting only one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A generic dispatcher content key may be right if multiple channels can double-fire, but a WhatsApp-local guard is safer if only the Baileys auto-reply callback lacks idempotence.
