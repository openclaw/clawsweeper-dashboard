## Repair Work Prompt

Fix the WhatsApp Baileys auto-reply duplicate delivery reported here: the same inbound turn can deliver equivalent block/final text twice through `dispatchWhatsAppBufferedReply` / `dispatchReplyWithBufferedBlockDispatcher`, producing two `deliverReply` calls for the same correlationId. Add a regression in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts` that captures the dispatcher `deliver` callback and invokes equivalent block and final text for one turn, expecting only one accepted provider send and one sent-text bookkeeping call. Preserve intentional multi-block delivery when text differs, media/tool-media delivery, errors, and fallback behavior. Prefer the narrowest WhatsApp-boundary idempotence guard after WhatsApp payload normalization; only change the shared dispatcher/pipeline if the regression proves the duplicate originates generically and the shared tests cover valid sequencing. Add a user-facing `CHANGELOG.md` fix entry; do not add a config flag unless a maintainer explicitly asks for product policy.
