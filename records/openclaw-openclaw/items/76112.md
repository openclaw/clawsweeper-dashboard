## Summary

Keep open. The report is recent and concrete, and current main still has a WhatsApp delivery boundary that forwards each deliverable block/final callback to the provider without a same-correlation same-content idempotence guard; existing generic block-streaming dedupe does not prove this Baileys auto-reply path is fixed.

## What This Changes

The issue requests a fix for duplicate WhatsApp Baileys auto-replies where the same `deliverWebReply` payload is delivered twice for one inbound message/correlation.

## Best Possible Solution

Add a focused WhatsApp same-message duplicate-delivery regression and suppress only repeat normalized payloads for the same inbound correlation at the narrowest boundary that preserves intentional multi-block, media, and fallback sends.

## Reproduction Assessment

Yes, at source level. The issue supplies concrete Baileys steps/logs, and current `dispatchWhatsAppBufferedReply` can be unit-reproduced by invoking the captured deliver callback twice with equivalent normalized block/final text for the same `msg.id`; I did not run a live WhatsApp session.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard is safer than a broad shared-dispatcher change unless tests show the duplicate originates before all channel-specific delivery boundaries.
