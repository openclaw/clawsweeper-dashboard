## Summary

Keep open. The report is recent, concrete, and still identifies an uncovered WhatsApp delivery-idempotence gap: current main forwards each deliverable block/final callback to the provider, while the existing generic block-streaming suppression does not prove this same-correlation WhatsApp path is fixed.

## What This Changes

The issue asks OpenClaw to stop WhatsApp auto-replies from sending the same generated reply twice for one inbound message and correlationId.

## Best Possible Solution

Add a focused same-correlation WhatsApp block/final duplicate regression, then suppress only same-content duplicate delivery at the narrowest boundary that preserves intentional multi-block, media, and error/fallback sends.

## Reproduction Assessment

Yes. The issue provides live reproduction steps and logs, and current source/tests give a unit-level reproduction shape by invoking the WhatsApp deliver callback with equivalent block/final content and expecting only one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard is safest if only Baileys auto-replies double-fire, while a shared dispatcher change needs proof it will not alter valid block/final sequencing.
