## Summary

Keep open. The report is recent and concrete, and current main still has a WhatsApp delivery boundary that forwards each deliverable block/final callback to the provider without a same-correlation same-content idempotence guard; prior generic block-streaming fixes reduce related duplicates but do not prove this Baileys auto-reply path is covered.

## What This Changes

The issue reports deterministic duplicate WhatsApp Baileys auto-reply delivery for the same inbound message and correlation id.

## Best Possible Solution

Add a focused WhatsApp same-correlation duplicate-delivery regression and suppress only same-content repeat delivery at the narrowest boundary that preserves intentional multi-block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The issue supplies live WhatsApp steps/logs, and current `dispatchWhatsAppBufferedReply` can be unit-reproduced by invoking the captured dispatcher deliver callback twice with equivalent block/final text and expecting only one provider send after the fix; I did not run a live Baileys test.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery guard is safest if the duplicate is only in the Baileys auto-reply callback, while a shared dispatcher change needs proof it will not alter valid block/final sequencing.
