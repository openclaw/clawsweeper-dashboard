## Summary

Keep open. Current main still forwards each deliverable WhatsApp block/final callback to `deliverReply` without a same-correlation same-content guard, and the recent report includes concrete Baileys logs; adjacent duplicate-delivery fixes do not cover this specific callback path.

## What This Changes

The issue requests that WhatsApp Baileys auto-replies deliver once per inbound message/correlation id instead of sending duplicate same-text replies.

## Best Possible Solution

Add a focused WhatsApp same-correlation same-normalized-content regression and suppress repeated delivery at the narrowest WhatsApp boundary unless a shared dispatcher fix is proven safe for all channels.

## Reproduction Assessment

Yes, at source level. The issue supplies live Baileys logs, and the current `dispatchWhatsAppBufferedReply` test seam can reproduce the defect by invoking the captured deliver callback twice with equivalent normalized text for the same `msg.id` and observing two provider sends.

## Solution Assessment

Unclear until the regression is written. A WhatsApp-local duplicate guard is the safest starting point if the duplicate is only in the Baileys auto-reply callback; a shared dispatcher change needs proof that it preserves valid block/final sequencing across channels.
