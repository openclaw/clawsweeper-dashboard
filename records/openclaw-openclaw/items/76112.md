## Summary

Keep open. The report is recent, concrete, and not covered by current main: the WhatsApp buffered reply callback still forwards each deliverable block/final payload to Baileys without same-correlation same-content idempotence, while the focused test still allows block plus final delivery.

## What This Changes

The issue reports deterministic duplicate WhatsApp DM auto-replies where the same Baileys delivery callback sends two identical provider-accepted messages for one correlationId.

## Best Possible Solution

Keep the existing generic block-streaming dedupe, and add a WhatsApp-focused regression plus the narrowest idempotence guard that suppresses only same-correlation same-content repeats while preserving intentional multi-block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The reporter provides deterministic Baileys DM logs, and current main can be unit-reproduced by invoking the captured WhatsApp deliver callback twice with equivalent block/final text and expecting one provider send after the fix; no live Baileys run was performed.

## Solution Assessment

Unclear until the regression is written. A narrow WhatsApp delivery idempotence guard is the safest first fix if the duplicate is local to Baileys auto-reply delivery, while a shared dispatcher change needs proof it will not alter valid block/final sequencing.
