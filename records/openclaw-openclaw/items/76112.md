## Summary

Keep open. Current main still forwards every deliverable WhatsApp dispatcher callback to `deliverWebReply` without a same-content/correlation guard, while the focused tests do not prove duplicate block/final callbacks are suppressed for the Baileys path.

## What This Changes

The issue requests eliminating duplicate WhatsApp Baileys auto-reply delivery when the same inbound correlation receives repeated dispatcher deliver callbacks.

## Best Possible Solution

Add a focused WhatsApp duplicate-delivery regression and suppress only same-content repeat delivery at the narrowest boundary that preserves intentional multi-block, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The issue supplies live WhatsApp steps/logs, and current main can be unit-reproduced by invoking the captured WhatsApp dispatcher deliver callback twice with equivalent text and expecting one provider send after the fix.

## Solution Assessment

Unclear whether dispatcher-wide dedupe is the best fix. A narrow WhatsApp delivery guard plus regression coverage is safer unless tests prove the shared dispatcher can dedupe without altering valid block/final sequencing.
