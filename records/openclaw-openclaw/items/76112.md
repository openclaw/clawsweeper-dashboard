## Summary

Keep open. The report is recent, concrete, and current main still forwards every deliverable WhatsApp dispatcher callback to `deliverWebReply` without a same-correlation same-content idempotence guard; adjacent block-streaming dedupe exists, but it does not prove this Baileys auto-reply double-delivery path is covered.

## What This Changes

Prevent WhatsApp Baileys auto-reply dispatch from sending the same text payload twice for one inbound message or correlation id.

## Best Possible Solution

Add a focused WhatsApp regression and suppress only repeat same-content delivery for the same inbound WhatsApp turn at the narrowest boundary that preserves intentional distinct block, final, media, tool, and fallback sends.

## Reproduction Assessment

Yes, at source level. The issue supplies deterministic live WhatsApp steps/logs, and current main can be unit-reproduced by invoking the captured WhatsApp dispatcher deliver callback twice with equivalent text for one message and expecting only one provider send after the fix.

## Solution Assessment

Unclear whether the final fix should live in the WhatsApp delivery callback or a shared dispatcher seam. A config flag is not the best solution; the maintainable path is a narrow regression plus a guarded dedupe that does not alter valid multi-payload sequencing.
