## Summary

Keep open. The report is recent and concrete, and current main still has a WhatsApp delivery callback that forwards each deliverable block or final payload to `deliverReply`; the existing block/final suppression is upstream of that callback and the focused WhatsApp test does not cover same-content block/final suppression.

## What This Changes

The issue reports deterministic duplicate WhatsApp auto-reply delivery for one inbound DM, with two provider sends for the same correlationId.

## Best Possible Solution

Add a focused regression for same-correlation WhatsApp block/final duplicate delivery, then suppress same-content duplicate provider sends at the narrowest boundary that preserves intentional multi-block, media, and chunked deliveries.

## Reproduction Assessment

Yes. The issue gives concrete live reproduction steps and logs, and current source/test inspection gives a unit-level reproduction shape by invoking the WhatsApp deliver callback with equivalent block and final content and expecting one provider send.

## Solution Assessment

Unclear until the regression is written. A generic dispatcher content key may help multiple channels, but a WhatsApp-local idempotence guard is safer if only the Baileys auto-reply callback lacks protection after upstream block/final suppression.
