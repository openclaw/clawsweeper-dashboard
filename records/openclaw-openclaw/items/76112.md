Keep open. The report is recent, concrete, and current main still has the WhatsApp callback forward each deliverable block/final payload to provider delivery without a same-content guard, while the focused test still permits separate block and final sends rather than locking the duplicate case.

## What This Changes

Report deterministic WhatsApp Baileys auto-reply duplicate delivery where one inbound message produces two `deliverWebReply` sends with the same correlation ID.

## Best Possible Solution

Add a focused same-correlation WhatsApp block/final duplicate regression, then suppress duplicate same-content delivery at the narrowest boundary that preserves intentional multi-block, tool-media, and media-only replies.

Yes. The issue provides concrete live steps and logs, and current source inspection gives a unit-level reproduction shape by invoking the WhatsApp deliver callback twice with equivalent block/final content and expecting one provider send after the fix.

## Solution Assessment

Unclear until the regression is written. A generic `(text/media + message id)` dispatcher key could solve the report, but a WhatsApp-local guard may be safer if only the Baileys auto-reply delivery callback lacks idempotence.
