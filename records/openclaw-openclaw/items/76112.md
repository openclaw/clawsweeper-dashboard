## Summary

Keep open. Current main still forwards each WhatsApp dispatcher deliver callback to `deliverWebReply`, and the focused tests allow block and final callbacks to become separate provider sends rather than protecting against same-content repeats.

## What This Changes

The issue reports duplicate WhatsApp Baileys auto-reply delivery for one inbound message with the same correlationId, runId, and connectionId.

## Best Possible Solution

Add a focused WhatsApp duplicate-delivery regression and suppress only same normalized payload repeats for one inbound message while preserving intentional multipart and media delivery.

## Reproduction Assessment

Yes, at source level: current main's captured WhatsApp dispatcher deliver callback can be invoked twice with equivalent block/final text and both invocations flow to `deliverReply`. The issue also supplies live Baileys logs, but I did not run a live WhatsApp session in this read-only pass.

## Solution Assessment

Unclear at implementation level. A narrow WhatsApp delivery guard plus regression is safest unless a shared dispatcher fix proves it will not alter valid block/final sequencing across channels.
