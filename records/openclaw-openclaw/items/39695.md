## Summary

Keep open. Current main clarifies the tool-profile behavior, but it still does not prove the reported Discord-bound `exec` plus JSONL transcript path is fixed; `exec` remains outside the `messaging` profile unless explicitly granted.

## What This Changes

A Discord-bound agent should execute `exec` tool calls from its SOUL instructions and create the expected JSONL session transcript instead of posting only a text reply.

## Best Possible Solution

Add a narrow maintainer-approved fix that either diagnoses messaging-profile agents that request runtime tools and points to `tools.profile: "coding"` or explicit `tools.alsoAllow`, or ships a guarded legacy-config migration/doctor repair for known local-onboarding defaults. Pair it with a focused Discord-bound regression that proves an inbound Discord turn can execute `exec` and persist the expected transcript JSONL when runtime tools are actually enabled.

## Reproduction Assessment

Partial but not high-confidence end to end. The code and tests give a strong reproduction for the policy precondition (`messaging` excludes `exec`), but there is no current merged live or integration proof for the full Discord inbound turn, shell execution, and transcript JSONL failure.

## Solution Assessment

Unclear. Automatically enabling `exec` for Discord or all messaging-profile agents is not the safest fix; the maintainable path is an explicit diagnostic or narrowly guarded migration plus a Discord regression test.
