## Summary

Keep open: current main documents and enforces the restrictive-profile behavior, but it does not prove the reported Discord-bound `exec` plus session JSONL failure is fixed; the related open PR only partially addresses silent tool-call surfacing.

## What This Changes

The issue requests that Discord-bound agents execute configured shell commands via `exec` and create session JSONL logs instead of producing a text-only reply with no transcript.

## Best Possible Solution

Add a maintainer-approved diagnostic or narrowly guarded migration for agents that expect runtime tools under restrictive profiles, plus a focused Discord-bound regression proving `exec` runs and a transcript JSONL is persisted when runtime tools are explicitly enabled.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this read-only review. Source inspection does reproduce the policy precondition that `messaging` excludes `exec` unless explicitly granted, but it does not reproduce the full Discord inbound turn plus missing JSONL transcript symptom.

## Solution Assessment

Unclear. Automatically enabling `exec` for Discord or all messaging-profile agents is not the safest fix; the narrower maintainable path is explicit diagnostics or a tightly guarded migration with Discord regression coverage.
