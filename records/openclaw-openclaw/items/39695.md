## Summary

Keep open. Current main explains one likely cause: `exec` is a coding-profile runtime tool, while `tools.profile: "messaging"` intentionally excludes runtime tools; however the report also includes a Discord-bound no-session-log symptom that I could not prove fixed or obsolete, and the related visibility fix in #55923 is still open and explicitly only partial.

## What This Changes

A Discord-bound agent on OpenClaw 2026.3.7 is reported to reply in chat while failing to run SOUL.md exec instructions and failing to create a session transcript.

## Best Possible Solution

Keep the issue open until maintainers review or land the focused silent-failure visibility work and confirm whether a current-main Discord transcript gap remains; do not broaden `messaging` to run `exec` without explicit policy approval.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for exec being unavailable when an upgraded or explicit config is on `tools.profile: "messaging"`, but the Discord no-session-log symptom still needs a current-main live or fixture reproduction with model/provider/config details.

## Solution Assessment

Unclear. The narrowest maintainable direction is to keep `exec` behind coding/full policy and make silent tool-call/transcript failures visible; changing the messaging profile or auto-migrating explicit profiles would need maintainer security/product judgment.
