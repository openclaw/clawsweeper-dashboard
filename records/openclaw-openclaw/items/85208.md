## Label Justifications

- `P1`: The PR targets a beta-blocking Telegram replay bug that can duplicate agent turns and visible replies for real users.
- `merge-risk: 🚨 message-delivery`: The patch changes Telegram inbound dispatch idempotency and an incorrect persistence boundary can still allow duplicate dispatch/replies across restarts.
- `merge-risk: 🚨 session-state`: A replayed Telegram message can still create a second agent turn in the same conversation if the gateway restarts before the dedupe commit.

## Summary

Keep open: this is a plausible Telegram-owned fix with solid live proof, but the durable replay key is committed only after message processing returns. Because claimable dedupe keeps pending claims in memory until commit, a restart during dispatch can still replay the same Telegram message before anything is persisted.

## What This Changes

The PR adds a persistent Telegram message-dispatch replay guard, threads dispatch dedupe keys through Telegram buffering/debounce paths, adds an inbound-debounce cancellation callback, and covers the behavior with focused tests and a changelog entry.

## Best Possible Solution

Keep the Telegram-owned persistent guard, but persist the message identity at the real dispatch-start boundary after context creation and release it only for pre-dispatch drops or failures, with a regression test that recreates the handler while the first dispatch is still pending.

## Reproduction Assessment

Yes for source-level reproduction: current main has only update-level Telegram dedupe before dispatch, and the linked issue identifies an at-least-once replay path. I did not run a new local/live repro because this review was read-only.

## Solution Assessment

No: the Telegram-owned claimable guard is the right direction, but committing after `processMessage` completes misses the restart-mid-dispatch window. The safer fix is to persist at dispatch start and release only before dispatch ownership is established.
