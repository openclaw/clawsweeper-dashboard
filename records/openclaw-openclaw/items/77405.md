Keep open: current main still has a source-reproducible Control UI path that splits one active assistant turn into separate assistant stream groups around tool events, so the reported WebChat fragmentation is still actionable.

## What This Changes

Request that Control UI Webchat merge assistant text fragments produced around tool calls into a coherent bubble instead of rendering each fragment separately.

## Best Possible Solution

Coalesce text-only assistant stream fragments for a single active Control UI run while preserving inline tool cards and canonical transcript/history semantics.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77405 in Control UI WebChat. Current main fragments one assistant response during tool-calling turns because `ui/src/ui/app-tool-stream.ts` commits live `chatStream` text into `chatStreamSegments` on tool events, `ui/src/ui/chat/build-chat-items.ts` emits each segment/current stream as separate `kind: "stream"` items interleaved with tool messages, and `ui/src/ui/views/chat.ts` renders each stream item through `renderStreamingGroup` as its own assistant group. Implement a narrow UI-side coalescing/grouping fix for text-only assistant stream fragments within one active run while preserving live tool-card ordering/visibility and final `chat.history` reload semantics. Do not add a new `gateway.webchat` config option or change gateway protocol behavior. Add focused regression coverage and a user-facing CHANGELOG entry.
