## Summary

Keep this PR open: the heap proof now supports the performance claim, but the patch still introduces a source-level compatibility bug by attaching symbol properties to caller-provided alias maps without guarding non-extensible objects.

## What This Changes

This PR changes plugin loader alias/cache memoization to share equal alias maps and jiti-normalized aliases through content-key caches and symbol-attached lazy properties, with a proof script and adjusted tests.

## Best Possible Solution

Keep the memory optimization, but store source-object attachments only when the alias map is extensible and fall back to the existing content cache or a WeakMap-style side cache for immutable caller-owned maps.

## Reproduction Assessment

Yes, source-level: the PR calls Object.defineProperty on the input aliasMap, and JavaScript throws for frozen or non-extensible objects. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No, not yet. The optimization is reasonable, but the safe implementation needs a guarded attachment or side-cache fallback so cache metadata does not become a new input mutability requirement.
