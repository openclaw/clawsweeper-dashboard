## Label Justifications

- `P2`: This is a focused plugin-loader performance refactor with a concrete merge-blocking compatibility defect but limited blast radius.
- `merge-risk: 🚨 compatibility`: The patch changes the alias-map input contract by attaching properties to caller-owned objects, which can break existing non-extensible inputs.
- `merge-risk: 🚨 availability`: A thrown TypeError in loader option construction can stop plugin module loading for affected callers.

## Summary

Keep open: the performance direction is plausible, but the patch is not merge-ready because it mutates caller-owned alias maps in a way that can throw and it lacks real heap/runtime proof for the claimed memory reduction.

## What This Changes

The PR changes plugin loader alias caching to attach derived cache data to alias-map objects and deduplicate alias maps by content.

## Best Possible Solution

Keep the perf direction, but make source-object cache attachment best-effort or WeakMap-backed, use an alias-only dedupe key, restore reference-reuse assertions, and add redacted real OpenClaw heap/runtime proof.

## Reproduction Assessment

Yes, by source inspection: the PR path will call `Object.defineProperty` on the caller-provided alias map, so `buildPluginLoaderJitiOptions(Object.freeze({...}))` would throw after this patch. I did not run the branch because this review is read-only.

## Solution Assessment

No; source-object cache attachment needs a non-throwing fallback, and the alias-map content dedupe key should not retain an unrelated module-cache JSON string.

- **[P2] Guard alias-map cache attachment:** `src/plugins/sdk-alias.ts:793-796`
  - body: `buildPluginLoaderJitiOptions` accepts caller-provided alias maps, but this new attachment writes a symbol property onto that object.   - confidence: 0.88

That undercuts the memory win this patch is trying to prove; use a sorted alias-entry key instead.
  - confidence: 0.78
