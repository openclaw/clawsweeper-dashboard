## Summary

Keep this PR open: the uploaded heap snapshots are useful proof for the memory goal, but the latest head still has a red lint check and a source-level compatibility risk from attaching symbol properties directly to caller-provided alias maps.

## What This Changes

The PR changes plugin loader alias-map memoization to dedupe derived cache data by attaching symbol-backed properties to alias map objects and adds a heap-snapshot proof script.

## Best Possible Solution

Keep the content-keyed memory dedupe, make symbol attachment best-effort or WeakMap-backed for caller-provided alias maps, and land only after the proof script is lint-clean and the alias loader tests cover the compatibility case.

## Reproduction Assessment

Yes for the PR blockers: the head CI log reports four check-lint errors in the new proof script, and source inspection shows the new Object.defineProperty calls can throw for non-extensible alias maps. I did not run local tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The optimization is promising, but the maintainable solution needs a safe attachment fallback and a lint-clean proof script before merge.
