## What This Changes

This PR changes plugin SDK alias memoization by replacing standalone normalized/config cache entries with symbol-attached alias-map properties and content deduplication, plus updates related tests.

## Best Possible Solution

Keep a content-keyed or WeakMap-backed cache that preserves reference reuse without mutating non-extensible inputs, then require redacted runtime or heap output from a real OpenClaw setup before merge.

## Reproduction Assessment

Yes, source-level: current main keeps a content-keyed normalized jiti alias cache and tests equal-but-distinct alias maps by reference, while the PR removes that cache and weakens the test to same-object reuse. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The maintainable fix is to reduce duplicated cache strings while preserving cross-object normalized alias reuse and avoiding hard mutation of caller-provided alias maps.

- **[P2] Preserve cross-object alias normalization reuse:** `src/plugins/sdk-alias.test.ts:1740`
  - body: By changing this call to reuse the same `aliasMap` object, the test no longer covers the case that current main protects: two distinct objects with identical aliases should get the same pre-normalized jiti alias result.   - confidence: 0.88
- **[P2] Avoid mutating non-extensible alias maps:** `src/plugins/sdk-alias.ts:793-796`
  - body: This unconditionally attaches a symbol property to the caller-provided alias map. `buildPluginLoaderJitiOptions` and `getCachedPluginModuleLoader` accept alias maps from callers; frozen or sealed maps that work on main will now throw a TypeError before loader creation. Store this in a WeakMap/content cache or make attachment best-effort with fallback to the computed normalized map.
  - confidence: 0.77
