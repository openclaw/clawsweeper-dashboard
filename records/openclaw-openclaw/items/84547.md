## Summary

Keep open: the heap-snapshot proof supports the memory-dedup goal, but the diff still introduces a compatibility bug by attaching symbol properties to caller-provided alias maps without guarding frozen or non-extensible objects.

## What This Changes

The PR rewrites plugin loader alias/cache memoization to share equal alias maps and jiti-normalized aliases through content-key caches, symbol-attached lazy properties, adjusted tests, and a heap-snapshot proof script.

## Best Possible Solution

Keep the memory optimization, but attach metadata only when the alias map is extensible and use a side-cache fallback for immutable caller-owned maps.

## Reproduction Assessment

Yes, source-level: the PR calls Object.defineProperty on the input aliasMap, and Node throws TypeError for frozen or non-extensible objects. I did not run the PR test suite because this review was constrained to read-only inspection.

## Solution Assessment

No, not yet. The optimization is reasonable, but the safe implementation needs a guarded attachment or side-cache fallback so cache metadata does not become a new input mutability requirement.
