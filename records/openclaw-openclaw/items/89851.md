## Summary

Keep open: current main still only accepts POST on the loopback MCP endpoint, while this PR addresses the missing Streamable HTTP GET/SSE surface and the previous code findings have been resolved. No blocking line-level defect remains; the remaining merge question is whether maintainers accept the stateless loopback session semantics and the new loopback security surface.

## What This Changes

The PR adds auth-gated GET `/mcp` SSE support, auth-gated DELETE `/mcp` teardown acknowledgement, keeps POST stateless, updates the Allow header, and adds focused gateway transport tests.

## Best Possible Solution

Land the gateway-boundary fix after maintainer acceptance of stateless loopback semantics, keeping GET/DELETE auth and Origin behavior covered by focused gateway tests.

## Reproduction Assessment

Yes, source-reproducible: current main emits a `type: "http"` `/mcp` config while the loopback request validator only allows POST and returns 405 for GET/DELETE. I did not run a full Claude Code session in this read-only review, but the failing server branch is clear from source and the PR includes curl proof for the fixed endpoint.

## Solution Assessment

Yes, this is the best fix shape if maintainers accept stateless loopback semantics: the server that emits the loopback Streamable HTTP config owns the missing GET/SSE surface, and avoiding `Mcp-Session-Id` prevents advertising a session lifecycle it does not enforce. A sessionful implementation would be the alternative only if maintainers want real lifecycle management.
