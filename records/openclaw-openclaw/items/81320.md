## Summary

Keep open: the feature is not already on current main and the report design fits the existing Control UI i18n baseline, but the PR still has a concrete scoped-filter correctness bug that can produce falsely clean reports for multi-token surface names.

## What This Changes

The branch adds a read-only `pnpm ui:i18n:report` command, a new `scripts/control-ui-i18n-report.ts` report helper, and focused Vitest coverage for Control UI i18n baseline reporting.

## Best Possible Solution

Keep the read-only baseline report, but tokenize the requested surface once and apply that same token model to raw-copy paths and translation keys with regression coverage for hyphenated, spaced, and camel-case inputs.

## Reproduction Assessment

Not applicable for the original item because this PR adds a new contributor-facing CLI report rather than fixing an existing bug. The blocking review finding is source-reproducible from the PR head: `--surface chat-queue` is normalized as one token while paths and keys are split into separate tokens.

## Solution Assessment

No, not yet: the report-over-baseline approach is a narrow maintainable fit, but the current scope matcher is inconsistent with the tokenizer it relies on. The safer fix is a small helper that tokenizes `--surface` and updates the helper tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Tokenize --surface before filtering:** `scripts/control-ui-i18n-report.ts:147-148`
