## Summary

Keep open: the feature is not on current main and the supplied real behavior proof is sufficient, but the patch still has a blocking scoped-filter bug for multi-word or hyphenated surfaces.

## What This Changes

This PR adds a `pnpm ui:i18n:report` package script, a read-only Control UI i18n baseline report CLI with focused helper tests, and `ui/AGENTS.md` usage guidance.

## Best Possible Solution

Keep the read-only baseline report, but make requested-surface matching use the same token model as paths and translation keys before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug report because this PR adds a contributor-facing feature; the remaining defect is source-reproducible because `--surface chat-queue` stays one normalized string while path and key inputs are split into separate tokens.

## Solution Assessment

Is this the best way to solve the issue? No, not yet: the report-over-baseline approach is narrow and maintainable, but the scope matcher needs consistent tokenization for requested surfaces, paths, and fallback keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Tokenize --surface before filtering:** `scripts/control-ui-i18n-report.ts:147-148`
  - body: The matcher normalizes the whole requested surface but path/key parts are tokenized, so values like `chat-queue`, `chat queue`, or `sessions-view` can miss matching files or fallback keys and produce a falsely clean scoped report. Tokenize the requested surface once and compare token sets for both raw-copy and fallback-key filters.
  - confidence: 0.9
