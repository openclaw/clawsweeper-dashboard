## Summary

Keep this PR open. The new report command is useful and mostly scoped well, but the final head still has a concrete `--surface` filtering bug that can make multi-word or hyphenated scoped reports falsely empty.

## What This Changes

The branch adds a read-only `pnpm ui:i18n:report` CLI over the Control UI raw-copy baseline and locale metadata, plus focused helper tests.

## Best Possible Solution

Keep the read-only baseline report, but make the requested surface use the same token model as paths and translation keys, with regression coverage for hyphenated, spaced, and camel-case inputs.

## Reproduction Assessment

Not applicable: this PR adds a new contributor-facing report command rather than fixing an existing bug. The remaining review finding is source-reproducible from the final patch because the input surface is not tokenized before comparison.

## Solution Assessment

No, not yet: the report-over-baseline approach is a narrow fit, but the current scope matcher is inconsistent with the tokenizers it relies on. The safer fix is a small helper that tokenizes `--surface` once and applies that consistently in both filters.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Tokenize --surface before filtering:** `scripts/control-ui-i18n-report.ts:147-148`
  - body: The report splits path and key names on punctuation/case boundaries, but compares them to the raw `--surface` value after only lowercasing it. Inputs users are likely to copy from module names, such as `chat-queue`, `chat queue`, or `sessions-view`, therefore match nothing and can produce a falsely clean scoped report. Tokenize the requested surface and cover that behavior in the helper tests.
  - confidence: 0.9
