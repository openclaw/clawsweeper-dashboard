## Summary

Keep open: the latest head still compares tokenized path/key parts against an unsplit requested surface, so hyphenated or multi-word scopes can falsely report no in-scope i18n work. The real behavior proof and security posture look acceptable, but this P2 matcher defect should be fixed before merge.

## What This Changes

The PR adds `pnpm ui:i18n:report` backed by a new read-only report script, focused tests, UI guidance, and a changelog entry.

## Best Possible Solution

Keep the report-over-baseline design, but make requested-surface matching use the same token model as paths and translation keys, with regression tests for hyphenated, spaced, and camel-case scopes.

## Reproduction Assessment

Yes for the review finding: source inspection shows `chat-queue` stays one normalized string while path tokens split to `chat` and `queue`, and the checked-in baseline has `chat-queue.ts` entries. I did not execute tests or the CLI because this review was constrained to read-only inspection.

## Solution Assessment

No, not yet: the new CLI is a narrow maintainable approach, but scope filtering must tokenize the requested surface before it can be trusted for multi-word surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Tokenize --surface before matching scopes:** `scripts/control-ui-i18n-report.ts:147-148`
  - body: The matcher tokenizes path/key parts but compares them to `normalizeToken(surface)`, so `--surface chat-queue`, `chat queue`, or similar multi-word scopes never equal the `chat`/`queue` tokens. Current baseline data includes `ui/src/ui/chat/chat-queue.ts`, so the report can falsely say there is no in-scope hardcoded text or locale gap. Tokenize the requested surface once and compare token sets in both raw-copy and fallback-key filters.
  - confidence: 0.91
