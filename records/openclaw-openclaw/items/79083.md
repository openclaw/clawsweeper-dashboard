## Summary

Keep open. The PR targets a real WhatsApp delivery-queue bug that is still present on current main, the runtime fix is narrowly scoped, and the supplied live proof is sufficient; I found no blocking correctness or security issue.

## What This Changes

Adds a WhatsApp monitor interval that drains matching pending deliveries every 30 seconds while connected and documents the fix in the changelog.

## Best Possible Solution

Land the WhatsApp-local periodic drain after maintainer-owned landing cleanup, preserving the existing drain key, account selector, active-claim guard, and backoff behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp queued deliveries can persist after enqueue while the monitor only drains at connection/reconnect setup, and the PR body supplies after-fix live logs from a real WhatsApp setup; I did not run a local live WhatsApp repro in this read-only review.

## Solution Assessment

Yes. A WhatsApp monitor-local timer is the narrowest maintainable runtime fix because it avoids pushing provider connection state into `enqueueDelivery` while reusing the existing drain, claim, and retry-backoff contract.
