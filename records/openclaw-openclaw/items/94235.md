## Summary

Keep open: current main and v2026.6.8 still have the cron empty-response bug, and this PR is a viable candidate fix. It should not be closed as cleanup, but merge needs maintainer acceptance of the unconditional delivery-target policy plus focused regression coverage.

## What This Changes

This PR sets `allowEmptyAssistantReplyAsSilent: true` on both isolated cron `runCliAgent` and `runEmbeddedAgent` calls.

## Best Possible Solution

Land this cron-executor-boundary fix, or an equivalent one, after maintainers accept the unconditional cron empty-output policy and the branch adds focused CLI plus embedded handoff coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main's cron executor does not pass `allowEmptyAssistantReplyAsSilent`, and `runCliAgent` throws `empty_response` on empty trimmed output when the flag is absent.

## Solution Assessment

Yes for the implementation layer, but the product policy still needs maintainer confirmation. Passing the existing flag from the cron executor is cleaner than special-casing cron inside the shared runner because the caller owns when silence is allowed.
