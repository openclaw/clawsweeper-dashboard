## Summary

Keep open: this is a focused QQBot message-delivery fix, but the current patch can flush buffered tool progress after a later visible final answer if an earlier silent/empty block was skipped. The protected maintainer label also rules out cleanup closure.

## What This Changes

The PR changes the QQBot outbound dispatcher to flush buffered non-streaming tool output when a silent block/final reply is skipped, adds dispatcher regression tests, and adds a local plugin-runtime rebuild helper script.

## Best Possible Solution

Keep the QQBot-local dispatcher fix, but gate the flush on the settled turn having no later visible final/block delivery and cover that sequence with a regression test.

## Reproduction Assessment

Yes at source level for the patch defect: shared dispatch can emit block replies before final replies, and the PR flushes solely because a prior silent skip occurred. I did not run tests because this review is read-only.

## Solution Assessment

No. The `onSkip`/`onSettled` boundary is plausible, but the state must account for later visible delivery before it can be the best fix.
