## Summary

Keep open: the due-slot fix is in the right cron service boundary and the proof is strong, but the patch is not merge-ready because the new anchor fallback also changes genuine `every` interval edits to reuse the old phase instead of the current edit-time phase.

## What This Changes

The PR changes `CronService.update()` to compare normalized scheduling inputs before recomputing next-run state, inherit the previous `every` anchor for schedule re-saves, and add regression tests for due `every` and `cron` re-saves.

## Best Possible Solution

Land the cron update fix after the anchor fallback distinguishes idempotent re-saves from genuine `every` interval changes.

## Reproduction Assessment

Yes, source-reproducible: current main recomputes nextRunAtMs on any schedule patch and the PR proof shows a real `CronService.update()` before/after path. I did not run a local test in this read-only review.

## Solution Assessment

No. The service-level due-slot comparison is the right fix shape, but the anchor inheritance is too broad; the safer version only inherits the old anchor for unchanged `every` re-saves and preserves edit-time anchoring for actual interval changes.

- **[P2] Keep edit-time anchoring for real interval changes:** `src/cron/service/ops.ts:539-540`
  - body: This fallback now always prefers the old `every` anchor whenever the previous job was also `every`. The Control UI sends every-schedule edits without `anchorMs`, and current main treats that as an edit-time anchor (`now`) before recomputing. With this patch, changing an interval such as 10 minutes to 1 hour keeps the old phase, so the saved schedule can fire earlier or later than the existing edit semantics. Please inherit the old anchor only for unchanged re-saves and keep the `now` fallback for genuine interval changes.
