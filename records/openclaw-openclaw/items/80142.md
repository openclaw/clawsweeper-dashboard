## Summary

Keep open. The PR addresses a real security-hardening gap in current main and supplies usable live-output proof, but the proposed trust check suppresses the new warning based on spoofable local `.clawhub/origin.json` metadata.

## What This Changes

The PR adds skill trust metadata, `skills.load.trustedDirs`, and untrusted-local-skill warnings across status generation, CLI output, Control UI, docs, tests, generated config checksums, and changelog.

## Best Possible Solution

Keep the additive status/CLI/UI warning direction, but only suppress local-source warnings for ClawHub installs when the marker is tied to trusted installer provenance; otherwise require `skills.load.trustedDirs` for local warning suppression.

## Reproduction Assessment

Yes, source-based. Current main has no trust metadata in `SkillStatusEntry` and no CLI/Control UI untrusted-local-source bucket, while the PR body shows the new after-fix status/check output.

## Solution Assessment

No. The reporting-layer approach is narrow and useful, but clearing the warning from any shape-valid local `.clawhub/origin.json` is not the safest way to solve a security warning request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate ClawHub markers before suppressing warnings:** `src/agents/skills-status.ts:306-307`
  - body: At this point any local skill can add a `.clawhub/origin.json` file with the five expected fields and be classified as `clawhub`, which suppresses the new untrusted-local-source warning. That makes the warning bypassable by the same local source it is meant to flag; require the marker to be tied to trusted installer provenance, such as matching slug/registry/lockfile data, before returning `untrustedLocalSource: false`.
