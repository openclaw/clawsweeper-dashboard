## Summary

Keep open: the PR addresses a real skills-status security UX gap and has after-fix proof, but it should not merge yet because the trust classifier lets any local skill suppress the warning by shipping a user-controlled `.clawhub/origin.json`, and the branch currently conflicts with main.

## What This Changes

Adds skill trust metadata, a `skills.load.trustedDirs` config key, CLI and Control UI untrusted-local-source warnings, tests, docs, generated config-baseline hashes, and a changelog entry.

## Best Possible Solution

Keep the additive status/API/UI direction, but only suppress local warnings for provenance that cannot be supplied by the skill directory itself, or keep a warning for claimed ClawHub origins until verified; then rebase and rerun the touched-surface checks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main builds skill status entries with only `source` and `bundled`, so CLI/UI status cannot render local-source trust warnings. I did not run a live CLI repro because this review is read-only and `pnpm` is unavailable in the checkout.

## Solution Assessment

No as written. A status-layer warning is a maintainable first step, but treating user-controlled origin metadata as trusted is not the safest way to solve this security issue.
