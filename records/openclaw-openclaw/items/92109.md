## Summary

Current main and the latest v2026.6.5 release still require ctimeNs equality for an unchanged embedded-session fingerprint, while the reporter supplied a deterministic Btrfs idle-file probe and successful runtime before/after evidence. The issue remains necessary as a narrow core session-fence regression, distinct from the compaction-owned-write fix in https://github.com/openclaw/openclaw/pull/90775.

## What This Changes

Make embedded-session takeover detection tolerate Btrfs ctimeNs drift when the session file’s identity and content-change signals remain unchanged.

## Best Possible Solution

Remove ctimeNs from the unchanged session-file fingerprint contract, keep device/inode/size/mtime and existing content-aware fence checks, and add focused positive ctime-drift plus negative real-mutation regression coverage.

## Reproduction Assessment

Yes. The reporter demonstrated ctime-only drift on an idle Btrfs file, reproduced 14+ runtime failures on v2026.6.5, and observed successful cron execution after removing only the ctimeNs comparison; current main contains that exact comparison.

## Solution Assessment

Yes. Excluding ctimeNs from this fingerprint is the narrowest maintainable fix because ctime is not a content-change signal, while the remaining identity/content fields and existing content-aware guards preserve actual takeover detection.
