## Summary

This PR should stay open: it is member-authored, has the protected `maintainer` label, and the latest head still has two blocking invalid-identity fallback issues that can reintroduce device ID churn.

## What This Changes

The branch adds dual TypeScript/Swift device identity schema parsing and migration, regression tests, a changelog entry, and small macOS wizard plus CLI assertion updates.

## Best Possible Solution

Keep the dual-schema compatibility and key-pair validation, but handle recognized invalid identity files through a stable fail-closed, quarantine, or explicit repair path instead of returning a fresh unsaved identity.

## Reproduction Assessment

Yes. Current main source shows the CLI and macOS app can share `identity/device.json` with incompatible PEM versus raw-key schemas, and the PR head source shows recognized invalid files still return fresh unsaved identities.

## Solution Assessment

No. Supporting both schemas is the right direction, but the invalid-file fallback should not produce transient device IDs; a maintainer should choose the recovery semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.81

Full review comments:

- **[P2] Avoid returning unsaved IDs for invalid TypeScript identities:** `src/infra/device-identity.ts:239`
  - body: When this branch recognizes an invalid OpenClaw identity file, it preserves the file but returns a newly generated identity without saving it.   - confidence: 0.82
