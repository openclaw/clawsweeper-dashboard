## Summary

Keep this PR open for maintainer review. The latest head still has the same correctness blocker: recognized invalid OpenClaw identity files are preserved on disk, but both loaders return a fresh unsaved generated identity, so repeated launches can still present different device IDs in the error path this fix is meant to stabilize.

## What This Changes

The PR updates TypeScript and Swift device identity loaders to accept each other's stored schemas, adds focused regression tests, aligns macOS gateway protocol call sites, and updates the changelog.

## Best Possible Solution

Keep the dual-schema compatibility and key-pair validation, but preserve invalid identity files without presenting a fresh unsaved identity; choose a fail-closed, quarantine, or explicit repair path before merge.

## Reproduction Assessment

Yes. Source inspection shows current main writes PEM identities from TypeScript and raw-key identities from Swift at the same `OPENCLAW_STATE_DIR` path, and https://github.com/openclaw/openclaw/issues/76815 includes concrete relaunch and pairing churn evidence.

## Solution Assessment

No. Supporting both schemas with validation is the right direction, but returning unsaved generated identities for recognized invalid files reintroduces unstable device IDs in a safety path this PR explicitly changes.
