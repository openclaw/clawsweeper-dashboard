## Summary

Keep open for review/repair: the compatibility direction is right, but the TypeScript raw Swift-schema migration accepts any two 32-byte blobs without confirming the private key derives the public key, so an invalid identity can be normalized and reused with signatures that will fail verification.

## What This Changes

The PR updates TypeScript and Swift device identity loaders to interoperate between PEM and raw-key identity schemas, preserve recognized invalid files, add TS/Swift regression tests, and add a changelog entry.

## Best Possible Solution

Land the dual-schema compatibility fix after both runtimes validate public/private key pairs before migration and invalid recognized identity files are preserved without being normalized into unusable key material.

## Reproduction Assessment

Yes. Current-main source shows the TypeScript loader expects PEM fields while the Swift loader expects raw key fields at the same `OPENCLAW_STATE_DIR` identity path, and the PR-head regression is directly visible in the raw Swift-schema normalization branch.

## Solution Assessment

No, not as currently proposed. The shared-schema migration is the narrow maintainable direction, but the TypeScript raw-key path needs the same key-pair validation already used for PEM and Swift decoding before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate Swift raw key pairs before migrating them:** `src/infra/device-identity.ts:176-180`
  - body: When the TypeScript loader sees a Swift-schema identity, this branch only checks that `publicKey` and `privateKey` decode to 32 bytes. If those raw keys do not belong together, the code still writes a canonical PEM identity and returns it, so later device signatures are made with a private key that cannot verify against the advertised public key. Please reject that case as `recognized-invalid`, matching the PEM branch and the Swift loader.
  - confidence: 0.88
