## What This Changes

The PR teaches the TypeScript and Swift device identity loaders to accept each other's on-disk schemas at `identity/device.json`, preserve recognized invalid identity files, add regression tests, and update the changelog.

## Best Possible Solution

Land the dual-schema compatibility fix after resolving the macos-swift build failure, keeping key-pair validation and invalid-file preservation as the compatibility boundary.

## Reproduction Assessment

Yes. Current main has a source-reproducible schema collision: TypeScript writes PEM identity fields under the state dir while Swift reads raw-key fields from the same `OPENCLAW_STATE_DIR` path, and https://github.com/openclaw/openclaw/issues/76815 includes concrete pairing churn logs.

## Solution Assessment

Yes for the product direction. Accepting both schemas with key-pair validation and normalizing valid raw-key files is narrower than moving identity paths, but the branch is not merge-ready until the Swift build failure is understood and cleared.
