## Summary

Keep open: the PR is aimed at the right bug, but it is not merge-ready because visible render-time UI still reads the local assistant override without an agent id, so per-agent avatars can still be mis-associated. It also has only unit-test proof, not real Control UI behavior proof.

## What This Changes

The PR changes Control UI assistant avatar localStorage from a flat browser-local avatar to an `agents` map keyed by agent id, passes agent id through the assistant identity controller, and adds controller regression tests.

## Best Possible Solution

Thread the active assistant or chat agent id through every local assistant override read used by render and bootstrap paths, add render-level regression coverage for main and worker overrides, and include redacted real Control UI proof before merge.

## Reproduction Assessment

Yes, source-reproducible from current main: the storage helper uses one global key and render/controller paths read it without an agent id, matching the linked issue steps. I did not run a browser flow in this read-only review.

## Solution Assessment

No. The storage/controller direction is the right layer, but the best fix must update every visible reader of the browser-local assistant override and prove the render path, not only the controller helper.
