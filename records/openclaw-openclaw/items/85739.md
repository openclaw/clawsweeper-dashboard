## Summary

Keep open for maintainer review: the patch is narrow and appears correct with native Windows proof, but it intentionally changes opt-in config fallback behavior on Windows and carries compatibility risk that needs maintainer signoff.

## What This Changes

The branch makes the shell-env fallback/probe return an empty success on win32, threads a platform override for tests, adds a Windows regression test, and records a changelog fix.

## Best Possible Solution

Land the narrow win32 no-op if maintainers accept native Windows shellEnv import as unsupported for POSIX shells; otherwise define an explicit Windows import mode separately instead of relying on /bin/sh.

## Reproduction Assessment

Yes. Current main source shows config loading calls loadShellEnvFallback, and that path reaches the /bin/sh probe without a win32 guard when shell env import is enabled with missing expected keys; issue comments also confirm Windows 2026.5.22 behavior.

## Solution Assessment

Yes, with maintainer compatibility signoff. The patch is the narrowest code path fix and preserves POSIX behavior, but maintainers should explicitly accept that native Windows shellEnv import becomes a no-op rather than a Git Bash/MSYS fallback.
