## Summary

Keep open for maintainer merge/sign-off: current main still has the Windows `/bin/sh` probe path, this branch fixes it cleanly with targeted coverage and native Windows proof, and I found no blocking correctness or security findings.

## What This Changes

This PR makes the optional login-shell environment fallback return an empty success on native Windows, adds a platform override for tests, adds a Windows regression test, and records the fix in the changelog.

## Best Possible Solution

Land this focused fix if maintainers accept native Windows shell-env import as POSIX-only/no-op; Windows users should use process env, `.env`, or config env for credentials, then resolve https://github.com/openclaw/openclaw/issues/84795.

## Reproduction Assessment

Yes, source-level: on current main, enabling shell-env import with missing expected keys reaches `loadShellEnvFallback`, whose shared probe has no Windows guard and resolves `/bin/sh`. I did not rerun current main on native Windows in this checkout, but the linked issue and PR proof match that path.

## Solution Assessment

Yes, with maintainer compatibility sign-off: putting the `win32` guard in the shared probe fixes the direct config path and aligns it with the existing `getShellPathFromLoginShell` Windows skip while preserving POSIX behavior.
