## Summary

Keep open: the PR is pointed at the right performance bug, but it currently sends provider-wildcard model visibility through the read-only catalog even though the Gateway path keeps wildcard views on the full catalog to avoid partial listings. It also needs inspectable real behavior proof before merge; the body has timing claims, but no redacted terminal output, logs, screenshot, or artifact to review.

## What This Changes

The PR changes `/models` auto-reply browsing to load the model catalog read-only by default, disables runtime auth discovery for that static visibility pass, and keeps explicit `all` browse on the full catalog path.

## Best Possible Solution

Keep the fast read-only path for ordinary default browse, preserve the full-catalog path for explicit `all` and configured provider wildcards, then attach redacted real timing proof for the LiteLLM-backed `/models` flow.

## Reproduction Assessment

Yes, source-reproducible: current main always calls the full `loadModelCatalog({ config })` path for `/models`, and the linked issue includes timing logs showing that path is the slow step in a large LiteLLM catalog. I did not run the LiteLLM setup locally because this review is read-only.

## Solution Assessment

No, not yet: using read-only catalog for ordinary browse matches the Gateway direction, but the PR should keep provider-wildcard visibility on the full catalog and provide inspectable real timing proof before merge.
