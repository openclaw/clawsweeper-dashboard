## Summary

Keep open: current main still sends the lean `status --json` path through configured-channel presence resolution that can construct the installed plugin manifest registry, matching the reported 2026.5.7 slow path rather than an already-fixed duplicate.

## What This Changes

Make `openclaw status --json --timeout 1000` avoid repeated plugin manifest/registry discovery and finish within a bounded fast-path time.

## Best Possible Solution

Keep the issue open and make the lean JSON status path compute channel presence from cheap config/env or prepared metadata, loading manifest records only for `--all`, `--deep`, or detail paths that truly render plugin-backed channel data.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79129. Current main's lean `openclaw status --json` path goes `scanStatusJsonFast` -> `hasConfiguredChannelsForReadOnlyScope` -> `resolveConfiguredChannelPresencePolicy` -> `loadInstalledChannelManifestRecords` -> plugin manifest registry construction, despite `includeChannelsData: false`. Keep ordinary JSON status on a cheap read-only path: avoid manifest-registry construction unless `--all`, `--deep`, or a path that actually renders manifest-backed channel detail requires it. Prefer cheap config/env/persisted signals or an explicit prepared snapshot; do not add broad persistent plugin metadata caches or weaken installed-index freshness. Add regression coverage proving lean JSON status does not reach the manifest registry/channel manifest loader, and add a user-facing changelog fix entry.
