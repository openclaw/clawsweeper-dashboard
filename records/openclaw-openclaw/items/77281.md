## Summary

Keep open: the PR addresses a real cache-locality problem and has added docs/changelog coverage, but the current head still introduces a runtime-only event regression by applying prompt-preservation before the existing empty-transcript branch.

## What This Changes

The PR adds an opt-in `agents.defaults.runtimeContextPlacement` config value, threads it into embedded-runner runtime-context prompt splitting, and updates schema/help/generated schema, docs, changelog, and focused tests.

## Best Possible Solution

Preserve the opt-in cache-friendly placement only for normal user turns while keeping heartbeat, cron, exec, and other runtime-only events on the existing runtime-only system-context path.

## Reproduction Assessment

Yes. Source inspection shows current main appends per-turn runtime context to the temporary system prompt, and the PR regression is source-reproducible by calling `resolveRuntimeContextPromptParts` with `transcriptPrompt: ""` and prompt preservation enabled.

## Solution Assessment

No, not as written. The config-based direction is narrow, but the early return must happen after runtime-only handling, and maintainers still need to accept the new config policy.
