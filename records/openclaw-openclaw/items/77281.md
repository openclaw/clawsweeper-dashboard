## Summary

Keep open: the PR is a useful, well-scoped direction for local-model cache locality, but the latest head still bypasses the existing runtime-only event path when the new prompt placement mode is enabled.

## What This Changes

The PR adds an optional `agents.defaults.runtimeContextPlacement` config value, wires it into embedded-runner runtime-context prompt handling, and updates schema metadata, docs, changelog, and focused tests.

## Best Possible Solution

Preserve the cache-friendly placement only for real user-turn prompts, keep runtime-only events on the existing system-context path, and let maintainers explicitly accept the new config contract.

## Reproduction Assessment

Yes. A source-level reproduction is `resolveRuntimeContextPromptParts({ effectivePrompt: "internal event", transcriptPrompt: "", preserveRuntimeContextInPrompt: true })` on the PR head, which returns a normal prompt instead of the current-main `runtimeOnly` result.

## Solution Assessment

No, not as written. The approach is plausible for opt-in local cache locality, but the early return must preserve empty-transcript runtime-only behavior and the config surface still needs maintainer product acceptance.
