## Summary

Keep open: the latest head still appears to let the new prompt-preservation branch catch runtime-only events before the existing runtime-only system-context path runs, so the patch would change synthetic event prompting behavior.

## What This Changes

The PR adds an optional `agents.defaults.runtimeContextPlacement` setting, threads it into embedded-runner runtime-context prompt splitting, and updates schema/help metadata, docs, changelog, and focused tests.

## Best Possible Solution

Preserve the opt-in cache-friendly layout only for real user-turn prompts, while keeping empty-transcript runtime-only events on the existing runtime-only system-context path.

## Reproduction Assessment

Yes, for a source-level review finding: the PR’s early return runs before the current empty-transcript branch, and current main shows that branch is how runtime-only events get marker prompting plus runtime-event system context. I did not live-run Ollama cache logs in this read-only review.

## Solution Assessment

No, not as written. The config direction is plausible for local KV-cache locality, but the implementation should exempt runtime-only events before preserving runtime context in the submitted user prompt.
