## Summary

The linked bug is valid on current main and this PR is on the right repair path, but it should stay open because the new web-search-provider auto-enable path can enable a provider plugin even when web search is explicitly disabled.

## What This Changes

The PR removes synthesized channel plugin auto-enable, sequences missing configured official plugin repair before stale cleanup, reruns auto-enable after installs, adds regression and Docker upgrade-survivor coverage, and updates docs/changelog.

## Best Possible Solution

Land the repair after gating web-search-provider auto-enable on enabled search and adding disabled-search regression coverage, so the linked issue closes through the implementation.

## Reproduction Assessment

Yes. Source inspection on current main shows validation warns for a `plugins.allow` id without a manifest while auto-enable can synthesize and allowlist a configured channel id when no manifest claims that channel.

## Solution Assessment

No, not as-is. The PR addresses the root channel/doctor mismatch, but the new web-search-provider candidate path should also respect `tools.web.search.enabled !== false` before it is the narrowest maintainable fix.

## Review Findings
