## Summary

Keep open: current main and the latest release still allow oversized session JSONL files to reach whole-file repair/open before any default size quarantine, and current main still lacks the repeated bootstrap-marker cap. The linked open PR is useful but explicitly covers only the bootstrap-entry amplifier, not the full crash-restart recovery boundary.

## What This Changes

The issue reports a gateway crash-restart loop where oversized session transcripts and repeated bootstrap entries grow until the gateway OOMs.

## Best Possible Solution

Land or replace the bootstrap-marker circuit breaker for the repeated-entry amplifier, then add a focused default recovery boundary that prevents oversized transcripts from being fully repaired/opened during gateway startup or embedded run preparation.

## Reproduction Assessment

Yes for source reproducibility: current main still whole-file reads/parses the session JSONL during repair/open before overflow prechecks, and the bootstrap completion marker append path has no recent-entry cap. I did not run a live OOM reproduction in this read-only review.

## Solution Assessment

No current complete fix is visible on main. The best fix appears to be two focused pieces: cap the bootstrap-marker amplifier and add a default bounded recovery/quarantine boundary before whole-file session repair/open.
