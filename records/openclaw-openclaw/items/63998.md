- `impact:crash-loop`: The reported failure mode is a gateway OOM followed by service restart into the same crashing session.

## Summary

Keep open. Current main and the latest release still allow the reported large-session path to reach whole-file session repair/open, the bootstrap-entry circuit-breaker work is still an open partial PR, and closed related reports explicitly route this remaining crash-restart recovery boundary back to this issue.

## What This Changes

The issue asks for crash-loop recovery when oversized session transcripts are reloaded, repeatedly grow through bootstrap/media replay, and eventually OOM the gateway.

## Best Possible Solution

Keep this open as the canonical crash-restart recovery issue until maintainers land a bounded transcript-load and degraded-session policy, with the bootstrap circuit breaker treated as only one partial mitigation.

## Reproduction Assessment

Yes at source level, but not as a live OOM repro in this read-only review. Current `SessionManager.open()` still reads and parses the same session JSONL before construction, then `setSessionFile()` reads it again; latest release v2026.6.6 has the same shape.

## Solution Assessment

No, the currently open bootstrap-marker PR is only a partial mitigation. The best fix needs a maintainer-owned recovery boundary for large transcript loading, quarantine/skip behavior, and any media/bootstrap growth controls.
