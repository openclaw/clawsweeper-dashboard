## Summary

Keep this open. Current main still has only partial, opt-in mitigations for large active transcripts, while the startup/restart and embedded-run paths can still read or parse a pathological JSONL transcript before any bounded quarantine or recovery decision. The later Linux confirmation and related open memory reports strengthen the issue rather than making it stale or fully covered elsewhere.

## What This Changes

The issue reports that oversized session JSONL transcripts can crash the gateway on restart, then grow through repeated bootstrap replay until manual recovery is required.

## Best Possible Solution

Implement a core startup-safe transcript recovery guard that stats and classifies active JSONL files before repair, restart recovery, readSessionMessages, or SessionManager.open; rotate, quarantine, or compact pathological transcripts into a bounded successor without first loading the whole file; add a total-count circuit breaker for repeated full-bootstrap markers; and cover those paths with focused regression tests. Media deduplication can remain a follow-up unless it is needed for the bounded recovery path.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete sizes, counts, logs, restart cadence, and platform evidence, and the current source shows deterministic unbounded read paths that can be exercised with a synthetic large valid JSONL fixture.

## Solution Assessment

Yes for the recommended repair direction, not for current main. The narrow maintainable fix is startup-safe transcript classification/quarantine plus a bootstrap marker cap; the existing opt-in byte guard and mid-turn precheck are useful but do not protect restart by default.
