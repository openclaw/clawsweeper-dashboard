## Summary

Keep this open: current main and the latest release still leave oversized active session transcripts on whole-file repair/open paths, and the linked bootstrap-marker PR is an open partial mitigation rather than the full crash-restart recovery boundary.

## What This Changes

The issue reports a gateway crash-restart loop where oversized session transcripts and repeated bootstrap-context entries grow until the process OOMs.

## Best Possible Solution

Add a default-safe recovery boundary for oversized active transcripts before whole-file repair/open, land or replace the bootstrap-marker circuit breaker, and route destructive recovery through explicit degraded/quarantine or doctor-backed semantics with regression coverage.

## Reproduction Assessment

No, not as a live current-main repro in this read-only review. The source path is reproducible enough to keep open: current main still reads and repairs the whole active transcript before run setup, while the issue and comments provide real OOM-loop reports.

## Solution Assessment

No, the currently visible solution is only partial. The open bootstrap-marker PR addresses one amplifier, but the best fix also needs an oversized-transcript load/recovery policy that avoids retrying the same crashing active file indefinitely.
