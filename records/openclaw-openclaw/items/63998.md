## Summary

Keep open. Current main still has the active-session path that reads and repairs the whole transcript and then opens it through `SessionManager.open()`, while the related open bootstrap PR only covers the marker-amplifier slice and is not merged.

## What This Changes

The issue asks for gateway recovery guards when oversized session transcripts and repeated bootstrap markers create a crash-restart OOM loop.

## Best Possible Solution

Define a maintainer-approved recovery policy that prevents oversized active transcripts from being fully repaired/opened on restart, then land coordinated coverage for both the transcript-size boundary and the bootstrap-marker circuit breaker.

## Reproduction Assessment

No, not as a live high-confidence reproduction in this read-only pass. Source inspection still shows the reported oversized-transcript path reaches whole-file repair and two full `SessionManager.open()` parses before any default quarantine or skip boundary.

Unclear. The requested guard direction is valid, but the best durable fix needs maintainer choice on whether oversized transcripts should be truncated, skipped, reset, quarantined, or migrated before runtime load.
