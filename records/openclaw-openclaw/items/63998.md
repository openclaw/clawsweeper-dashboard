## Summary

Keep this open because current main still lacks a default startup/restart-safe recovery boundary for pathological active transcripts. Recent changes reduce some gateway hot-path reads, but repair, recovery indexing, embedded SessionManager open, and bootstrap marker persistence still allow the reported crash-restart transcript growth pattern to remain actionable.

## What This Changes

The issue requests safeguards so oversized session JSONL transcripts and repeated full-bootstrap replay cannot drive a gateway restart OOM loop.

## Best Possible Solution

Implement a default core transcript recovery guard that stats and streams active JSONL before repair, restart recovery, or SessionManager.open; quarantines or rotates pathological active transcripts into a bounded successor; and caps repeated full-bootstrap markers with focused regression coverage.

## Reproduction Assessment

Yes. The issue body and Linux follow-up give concrete sizes, counts, logs, restart cadence, and OOM symptoms, and current source still exposes deterministic unbounded parse paths that can be exercised with a synthetic large JSONL transcript.

## Solution Assessment

Yes for the recommended repair direction, not for current main. The narrow maintainable fix is startup-safe transcript classification/quarantine plus a bootstrap marker cap; broader media dedupe and structuredClone cleanup can remain separate follow-ups unless required by that path.
