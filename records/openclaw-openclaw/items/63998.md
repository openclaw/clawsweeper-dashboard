## Summary

Keep this open: current main has useful bounded tail reads and optional compaction-by-size settings, but it still has default paths that can materialize or reopen a pathological active transcript before any quarantine/recovery boundary.

## What This Changes

Prevent oversized crash-replayed session transcripts from causing gateway restart loops and unbounded bootstrap/media transcript growth.

## Best Possible Solution

Add a default startup-safe transcript guard that stats and streams active JSONL before repair/open/recovery, quarantines or rotates pathological active transcripts into a bounded successor, and caps repeated full-bootstrap markers with regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction: a synthetic oversized JSONL can still exercise full-file repair and SessionManager.open paths before a default quarantine boundary. I did not run a live OOM loop, but the issue’s concrete sizes/logs plus current code make the failure path high-confidence.

## Solution Assessment

Yes for the recommended repair direction, not for current main. Optional maxActiveTranscriptBytes helps configured compaction, but the narrow maintainable fix is a default load-before-open recovery guard plus bootstrap-marker capping.
