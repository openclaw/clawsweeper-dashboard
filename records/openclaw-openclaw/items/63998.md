## Summary

Keep this open: current main has partial bounded transcript reads and optional compaction rotation, but the startup/open path can still read or reopen a pathological active transcript before any default quarantine or rotation guard, and there is still no cap on repeated full-bootstrap marker appends.

## What This Changes

The issue requests preventing oversized session transcripts from causing gateway restart/OOM loops by bounding transcript loading, limiting bootstrap replay, deduping repeated media, and degrading safely.

## Best Possible Solution

Add a default load-before-open recovery guard that stats and streams active session JSONL before repair/open/compaction, rotates or quarantines pathological active transcripts into a bounded successor, and caps repeated full-bootstrap markers with regression coverage.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: a synthetic oversized active session JSONL can still drive repairSessionFileIfNeeded, prewarmSessionFile, SessionManager.open, or readTranscriptFileState before any default quarantine boundary. I did not run the live OOM loop in this read-only review.

## Solution Assessment

Yes, the best fix direction is a default guard at the transcript load/open boundary rather than only adding broad maintenance settings. Existing bounded tail reads and opt-in compaction rotation are useful partial mitigations but do not solve startup/open recovery for an already-pathological transcript.
