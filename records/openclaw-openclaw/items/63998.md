## Summary

Keep open. Current main still lets oversized active session JSONL reach whole-file repair, whole-file transcript-state reads, and `SessionManager.open()` before any default quarantine/skip boundary; the open related PR only covers the bootstrap-marker append amplifier, not the full crash-restart recovery problem.

## What This Changes

The issue reports that oversized session transcripts can drive the gateway into a crash-restart loop that keeps growing session/bootstrap state until recovery requires manual intervention.

## Best Possible Solution

Keep this as the canonical focused tracker for a conservative pre-open transcript safety boundary that prevents oversized active JSONL from entering whole-file repair/open loops, with destructive recovery actions gated by explicit maintainer policy and live proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the unbounded pre-open load path: current main still reads and parses the entire session file before and during `SessionManager.open()`, but I did not run a live large-transcript gateway OOM reproduction.

## Solution Assessment

Is this the best way to solve the issue? Not yet as proposed: the gap is real, but default truncate/archive/skip behavior needs maintainer policy and live proof; a safer path is a bounded classifier plus explicit recovery behavior before whole-file repair/open.
