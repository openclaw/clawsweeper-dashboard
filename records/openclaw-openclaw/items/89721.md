## Label Justifications

- `P2`: This is a bounded cron/session retention bug fix with limited blast radius, but it is not an emergency runtime outage.
- `merge-risk: 🚨 compatibility`: The patch changes the retention contract for existing isolated cron base rows and can alter behavior for long-cadence jobs on upgrade.
- `merge-risk: 🚨 session-state`: The new pruning path can delete stored cron session metadata that carries model, auth, and preference state across runs.

## Summary

Keep open: the reported cron retention gap is source-reproducible, but this PR is not merge-ready because it changes retention from pruning isolated run rows to deleting base cron session rows that current code/docs use to preserve per-job session metadata.

## What This Changes

The PR adds an isolated cron session key matcher, uses it in the cron session reaper, and adds a regression test for pruning `agent:<id>:cron:<jobId>` rows.

## Best Possible Solution

Keep base cron session rows preserved by default, prune `:run:` descendants under `cron.sessionRetention`, and handle deleted-job/orphaned base rows through a dedicated cleanup path or explicit owner-approved contract change with upgrade tests.

## Reproduction Assessment

Yes. Current main only prunes keys accepted by `isCronRunSessionKey`, and that helper rejects base `agent:<id>:cron:<jobId>` keys; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR fixes the observed accumulation by pruning base rows, but current code and docs treat those rows as persistent metadata while completed run rows are the retention target.
