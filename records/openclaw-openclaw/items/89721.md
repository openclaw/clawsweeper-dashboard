Keep open: this PR addresses a real source-proven cron cleanup gap, but the submitted fix routes stable isolated cron base rows into age-based retention even though current code and docs preserve those rows as durable session metadata. ## What This Changes

The PR adds a matcher for isolated `agent:<id>:cron:<jobId>` session keys, prunes those rows in the cron session reaper, and adds a regression test.

## Best Possible Solution

Preserve stable isolated cron base metadata by default, then add a narrowly owned cleanup path for true deleted-job/orphan rows or intentionally document and test any maintainer-approved retention contract change.

## Reproduction Assessment

Yes, from source: current main’s reaper only matches `isCronRunSessionKey`, while isolated cron state is persisted at `agent:<id>:cron:<jobId>` without a `:run:` segment. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted; it is a plausible cleanup patch, but it deletes the stable base row that current code and tests use for durable isolated cron metadata. A safer fix preserves base rows and handles orphan/deleted-job cleanup separately unless maintainers approve a contract change.
