The PR changes Control UI chat rendering so a null stream stays null instead of falling back to an empty reading-indicator stream when an abortable run is present.

## Best Possible Solution

Fix the stale post-final state with a narrower lifecycle or render guard that preserves the active-run pending indicator, cover both the stuck-after-final and active-run-before-stream cases, then attach redacted real UI proof.

## Reproduction Assessment

Yes for the PR regression: the PR-head UI CI reproduces a failing active-run/no-stream test, and current source shows why removing the fallback produces the loading skeleton. I did not run a live Control UI session for the original stuck-indicator report.

## Solution Assessment

No. Removing the fallback is too broad because current main intentionally uses it to keep the reading indicator visible for abortable active runs before stream text arrives; the fix should distinguish stale terminal state from real pending state.
