## Summary

Keep open: current `main` still has the reported prompt parity gap in the `expectsCompletionMessage` announce path, and the surrounding delivery path still re-prompts the requester agent for visible completion delivery.

## What This Changes

The issue requests duplicate-suppression handling for the subagent completion announce branch when a parent agent has already delivered the child result.

## Best Possible Solution

Add the missing silent duplicate guard to the completion-mode announce instruction and cover it with a focused regression test, leaving structural fingerprint-based suppression for a separate maintainer decision if needed.

## Reproduction Assessment

Yes, source-reproducible: with a top-level requester and `expectsCompletionMessage: true`, `runSubagentAnnounceFlow` builds and dispatches a completion prompt that lacks the `NO_REPLY` duplicate escape present in the adjacent announce branches. I did not run a live agent/channel scenario in this read-only pass.

## Solution Assessment

Yes for the minimal fix: adding the same silent-token duplicate guard to the `expectsCompletionMessage` branch is the narrowest maintainable repair. The proposed structural fingerprint short-circuit is a larger follow-up and should not block the prompt parity bug fix.
