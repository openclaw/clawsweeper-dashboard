## Summary

Keep open. Current main still lacks a Jina/OpenAI-compatible `task`, `queryTask`, `documentTask`, or `extraBody` path for memory-search embeddings, and the issue is paired with open, unmerged PR #74444, so the issue should stay open until that implementation lands or is replaced.

## What This Changes

The issue requests configurable Jina Embeddings v5 task parameters, preferably separate query/document task settings or an embedding request extra body, so OpenAI-compatible memory search can use task-specific adapters.

## Best Possible Solution

Review and land PR #74444, or a narrow replacement, that adds typed query/document task settings through memory-search config, resolved provider options, OpenAI-compatible query and batch request bodies, embedding cache identity, tests, docs, and changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `memorySearch.queryTask`, `documentTask`, or `extraBody` on current main and the strict schema/provider request path has no accepted field to carry it, while OpenAI-compatible request serialization still emits only `model`, `input`, and optional `input_type`.

## Solution Assessment

Unclear until PR #74444 is reviewed. The typed `queryTask`/`documentTask` direction is likely safer than unrestricted `extraBody` because query and document embeddings use different request paths and task values must participate in cache identity.
