## Summary

Keep open. Current main still has no configurable embeddings `task` field or embedding-request `extraBody` path, and the issue is already paired with open PR #74444 as the implementation candidate, so it should not be closed before that PR lands or is replaced.

## What This Changes

Request support for configurable Jina Embeddings v5 task-specific adapter parameters in OpenAI-compatible memory search embedding requests, ideally with separate query and document task handling or an equivalent extra body pass-through.

## Best Possible Solution

Accept a narrow, documented memorySearch configuration surface that carries the Jina task value through resolved config, provider creation, query embedding requests, document/batch indexing requests, and embedding cache identity, then land the linked PR or a replacement with focused tests and docs.

## Reproduction Assessment

Yes. The current-main reproduction path is source-level: configure or try to pass a Jina task value through memorySearch and it has no accepted schema field or provider request path; the OpenAI-compatible embedding body currently serializes only `model`, `input`, and optional `input_type`.

## Solution Assessment

Unclear until the linked PR is reviewed. A typed query/document task API looks safer than an unrestricted `extraBody` pass-through because memory search has distinct query and document embedding paths and cache identity must change when task values change.
