## Summary

Keep open. Current main still has no configurable Jina embeddings `task` field or embedding-request `extraBody` path, and the issue is paired with open PR #74444 as the implementation candidate, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue requests a memorySearch configuration surface that forwards Jina Embeddings v5 task-specific adapter values into OpenAI-compatible embedding request bodies.

## Best Possible Solution

Land #74444 or a replacement that adds a typed `queryTask`/`documentTask` memorySearch surface, carries it through resolved config and provider creation, forwards it on query and document embedding paths including batch indexing, includes it in embedding cache identity, and documents the Jina v5 behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure or attempt to pass a Jina task value through `memorySearch`, and current main has no accepted schema field or OpenAI-compatible request path for it.

## Solution Assessment

Unclear until #74444 is reviewed. A typed query/document task API appears narrower and safer than unrestricted `extraBody` because memory search has distinct query and document embedding paths and cache identity must reflect task changes.
