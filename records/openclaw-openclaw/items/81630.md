## Summary

Keep open: current main still has separate live tool-event rendering and final chat.history projection paths, so the reported grow-then-replace WebChat behavior remains a plausible UX issue and needs a maintainer UX decision before a narrow repair.

## What This Changes

Requests WebChat to keep tool-heavy streaming messages visually stable by aligning live tool-call rendering with the final chat.history projection or the existing tool-call visibility control.

## Best Possible Solution

Choose one WebChat contract for live tool progress, then keep live rendering and final history refresh visually equivalent through stable collapsed placeholders or the same projection rules.

## Reproduction Assessment

No high-confidence browser reproduction was run. Source inspection shows the central path: live tool-event rows are rendered during the run, then a final chat.history reload replaces them with projected durable history.

## Solution Assessment

Unclear until maintainers choose the desired UX. Applying equivalent collapse/projection rules to live tool output is likely narrower than adding a new setting, while changing defaults or visibility policy is a product decision.
