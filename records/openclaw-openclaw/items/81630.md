## Summary

Keep open: current main still has separate live tool-event rendering and final `chat.history` projection paths, so the reported WebChat grow-then-shrink reflow remains plausible from source and does not meet a close reason.

## What This Changes

The issue requests a stable WebChat streaming display so live tool-call rendering does not visibly inflate and then shrink when final `chat.history` projection replaces it.

## Best Possible Solution

Define a Control UI contract for live tool progress, then make live rendering and final `chat.history` projection preserve a stable visual footprint through completion, preferably reusing existing tool-call visibility/expansion state unless maintainers want a new setting.

## Reproduction Assessment

No, not with high-confidence live browser proof. Source inspection shows the relevant path: live tool messages can include output, final tool completion reloads `chat.history`, and final projection uses a different normalized transcript shape.

## Solution Assessment

Unclear until maintainers choose the desired UX. Reusing the existing tool-call visibility/collapse behavior during streaming is likely narrower than adding another persisted setting, but the default visible behavior is a product decision.
