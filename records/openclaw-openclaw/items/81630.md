## Summary

Close as duplicate/superseded: the same WebChat streaming reflow problem is already tracked by an older open canonical issue, and current main still follows the source path described there.

## What This Changes

The issue asks WebChat to keep one stable assistant reading surface during multi-tool-call streaming instead of growing with intermediate tool content and shrinking after final history projection.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/63956 as the canonical tracker and fix it with a Control UI-only single in-flight assistant group that keeps text segments, tool cards, and the live tail visually stable without changing gateway protocol or transcript persistence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: committed stream segments plus a live tail produce multiple stream items, each rendered as its own assistant group, before final chat.history projection replaces the live state.

## Solution Assessment

Yes. The narrowest maintainable solution is the canonical issue's UI-only grouped streaming render path; adding a new config toggle would be broader than needed for the core reflow bug.
