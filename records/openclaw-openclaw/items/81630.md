## Summary

Keep open: current main has mitigations such as collapsed tool cards and a `chatShowToolCalls` toggle, but source inspection still shows distinct live-stream and final `chat.history` projection paths, so a tool-heavy turn can still change visible content at completion. The remaining fix is UX policy, not a stale cleanup close.

## What This Changes

The issue requests a stable WebChat streaming experience so tool-call details and intermediate output do not visibly inflate a response and then disappear after the final `chat.history` refresh.

## Best Possible Solution

Align live WebChat rendering with the final history projection, or keep equivalent collapsed placeholders through the final refresh, while preserving the existing tool-call visibility toggle semantics.

## Reproduction Assessment

No high-confidence browser reproduction was run. Source inspection gives a source-reproducible path: live stream/tool state is rendered during the run, then final `chat.history` projection reloads and clears that transient state.

## Solution Assessment

Unclear until maintainers choose the desired UX. Applying the same projection/collapse rules to live streaming is likely narrower than adding another setting, but preserving live progress may be intentional.
