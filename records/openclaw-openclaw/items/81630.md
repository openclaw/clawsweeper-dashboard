## Summary

Keep open: current main still has separate live tool-event rendering and final `chat.history` projection paths, so the reported WebChat reflow remains plausible from source, but the preferred UX needs maintainer direction before a narrow repair.

## What This Changes

The issue requests a stable WebChat streaming experience when multi-tool turns transition from live tool output to the final projected transcript.

## Best Possible Solution

Align live WebChat tool rendering with the final projected transcript, or keep equivalent collapsed placeholders through final refresh, so tool-heavy turns do not remove large visible content at completion.

## Reproduction Assessment

No high-confidence browser reproduction was run. Source inspection shows the central path: live tool messages and stream segments can be present during a run, then a final tool run reloads projected `chat.history` and clears the live tool state.

## Solution Assessment

Unclear until maintainers choose the desired UX. Reusing the existing `chatShowToolCalls` and collapsed disclosure semantics for live tool output looks narrower than adding a new setting, but that is a product decision.
