## Summary

Keep open: current main still has separate live tool-event rendering and final `chat.history` projection paths, so the reported WebChat content reflow remains source-plausible. The exact UX fix needs maintainer direction because current main intentionally shows live tool progress and already has a `chatShowToolCalls` control plus shipped auto-scroll modes.

## What This Changes

The issue asks WebChat to avoid visible bubble growth and shrinkage when multi-tool-call turns switch from live tool-event rendering to final `chat.history` rendering.

## Best Possible Solution

Keep live progress useful while making the final transition stable, either by applying equivalent collapsed/projection semantics during streaming or by preserving stable collapsed placeholders until history catches up.

## Reproduction Assessment

No high-confidence browser reproduction was run. Source inspection shows a clear current-main path where live tool-event messages are rendered during a run and then replaced by the projected `chat.history` view after the final reload.

## Solution Assessment

Unclear until maintainers choose the desired UX. Reusing the existing `chatShowToolCalls` boundary and making live tool cards match the final collapsed/projection shape is likely narrower than adding another setting.
