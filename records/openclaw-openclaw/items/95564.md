## Label Justifications

- `merge-risk: 🚨 auth-provider`: The diff changes how OpenAI OAuth credential POSTs are routed when proxy env vars are configured, which is auth-provider behavior CI alone cannot fully prove.

## Summary

Keep open. The patch targets a real current-main gap: the new OpenAI ChatGPT OAuth token helper posts exchange and refresh requests without trusted env-proxy mode, while the fetch guard only uses env proxy routing in trusted mode. I found no blocking code defect, but the PR still needs real after-fix proof and maintainer acceptance that OAuth credential POSTs should honor operator proxy env vars.

## What This Changes

The PR adds existing env-proxy options to the OpenAI Codex OAuth token form request helper used for authorization-code exchange and refresh.

## Best Possible Solution

Land a focused fix that routes the fixed `auth.openai.com` OAuth token POST through the existing trusted env-proxy fetch mode, with redacted after-fix proof and a small assertion covering exchange/refresh proxy options.

## Reproduction Assessment

Source-reproducible, not live-reproduced: current main posts exchange/refresh token forms through strict `fetchWithSsrFGuard` without trusted env-proxy mode, and strict mode ignores ordinary env proxy by default. I did not run a live restricted-region proxy test in this read-only review.

## Solution Assessment

Yes, this is a narrow and maintainable fix for the current helper because the URL is fixed to `auth.openai.com` and `fetchWithSsrFGuard` already supports trusted env-proxy mode with hostname checks. The remaining concern is proof and a regression assertion, not the patch shape.
