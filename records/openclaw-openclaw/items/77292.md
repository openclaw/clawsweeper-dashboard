## Summary

Keep open: current main and the latest release still preserve shared-main Telegram DM behavior unless `session.dmScope` is set to an isolation mode, and the adjacent single-user subagent routing fix does not prove this cross-user Telegram security report is resolved.

## What This Changes

The issue asks OpenClaw to prevent subagent or async Telegram replies for one DM user from using another DM user's shared `deliveryContext` when a single bot serves multiple users.

## Best Possible Solution

Maintainers should decide whether secure-DM configuration plus doctor/onboarding guidance is the supported multi-user answer, or ship a compatibility-aware default/migration/runtime guard with two-user Telegram subagent regression proof.

## Reproduction Assessment

Source-level yes for the shared-main routing risk: current main still routes Telegram DMs to `agent:main:main` by default and multi-user allowlists are not pinned to one route owner. No live current-main two-user Telegram subagent reproduction was run after the adjacent parent-owner fix.

## Solution Assessment

Unclear as a final product path. `session.dmScope: "per-channel-peer"` is the existing isolation mechanism, but deciding whether shared-main multi-user DMs should be rejected, migrated, or guarded requires maintainer security/product judgment.
