## Summary

Keeping this open because the report is plausible but not yet pinned down: current main already has secure DM isolation via `session.dmScope`, and the reported `agent:main:main` state matches the documented shared-DM mode rather than proving a current-main routing bug.

## What This Changes

Request that Telegram DM delivery and subagent completion routing be isolated per `chat_id`/`sender_id` so one bot serving multiple users never sends content to another DM.

## Best Possible Solution

Confirm the reporter's session configuration first; if they were in shared-main mode, route them to secure DM mode, and if leakage reproduces with `per-channel-peer` or `per-account-channel-peer`, add a focused Telegram + subagent announce regression and fix requester-origin preservation.

## Reproduction Assessment

Unclear. Source inspection shows a high-confidence path for documented shared-main DM context sharing, but not a high-confidence current-main reproduction of cross-DM leakage when secure DM isolation is enabled.

## Solution Assessment

Unclear. Scoping only `deliveryContext` by chat while leaving the transcript in `agent:main:main` would be a partial fix; the maintainable path is to use existing DM-scoped sessions and only patch announce delivery if that invariant fails.
