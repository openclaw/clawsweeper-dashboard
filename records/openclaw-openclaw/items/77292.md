## Summary

Keeping this open because the report describes a plausible cross-user Telegram DM privacy boundary, and current main does not yet have a focused proof that subagent announces stay scoped across two secure-DM users. The documented shared-main DM mode explains some context sharing, but it does not rule out a remaining secure-mode routing bug.

## What This Changes

The issue asks OpenClaw to keep Telegram subagent completion and parent replies scoped per `chat_id`/`sender_id` when one bot serves multiple DM users.

## Best Possible Solution

First reproduce with explicit `session.dmScope` values; route shared-main cases to secure DM configuration, and if secure mode leaks, fix Telegram/subagent requester-origin preservation with a two-DM regression test.

## Reproduction Assessment

Unclear. Source inspection shows documented shared-main DM context sharing, but I did not establish a current-main leak when `session.dmScope` is `per-channel-peer` or `per-account-channel-peer`.

## Solution Assessment

Unclear. Scoping only `deliveryContext` by `chat_id` would partially override documented shared-main continuity; the safer path is to preserve secure DM scopes and patch the announce path only if it leaks under those scopes.
