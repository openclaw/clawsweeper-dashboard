## Label Justifications

- `P1`: The report describes one user's Telegram DM content or subagent completion reaching another user's DM, which is an urgent channel/session isolation concern.
- `impact:security`: Cross-user DM context exposure can reveal private conversation content across a trust boundary.

- `impact:session-state`: The suspected root is shared delivery context and transcript/session state on `agent:main:main` across multiple DM users.

## Summary

Keep open: current main and v2026.6.6 have the related subagent completion-owner fix and supported secure-DM routing, but `session.dmScope: "main"` still intentionally shares DMs, so the remaining cross-user Telegram isolation question needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to keep Telegram bot DM delivery and subagent completion context isolated per chat/sender instead of letting `agent:main:main` reuse the last-touched user's delivery context.

## Best Possible Solution

Make a maintainer security/product decision: either treat secure-DM configuration plus doctor/onboarding guidance as the supported fix, or add a compatibility-aware default/migration path with Telegram multi-user subagent regression coverage.

## Reproduction Assessment

Partly. Source and docs show the shared-session premise when `session.dmScope` is `main`, but I did not establish a live current-main Telegram reproduction for the exact async subagent cross-user delivery after the fix for https://github.com/openclaw/openclaw/issues/80201 shipped.

## Solution Assessment

Unclear. The narrow supported mitigation is `session.dmScope: "per-channel-peer"`, while changing defaults or migrating existing shared-main DM setups is compatibility-sensitive and needs maintainer policy judgment.
