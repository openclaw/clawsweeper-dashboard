## Summary

Keeping this open because it is a plausible privacy-boundary report, but current main also intentionally supports shared-main DM mode; the missing `session.dmScope` configuration is the key fact needed before this can be closed or safely automated.

## What This Changes

The issue asks OpenClaw to prevent Telegram subagent and main-session delivery context from crossing between different DM chat IDs when one bot serves multiple users.

## Best Possible Solution

Confirm the reporter's `session.dmScope` first; route shared-main configurations to secure DM mode, and if leakage reproduces under `per-channel-peer` or `per-account-channel-peer`, add a focused Telegram/subagent announce regression and patch requester-origin preservation without changing shared-main semantics accidentally.

## Reproduction Assessment

Unclear for the security bug as reported. Source inspection gives a high-confidence path for documented shared-main DM context sharing, but not a high-confidence reproduction of cross-DM leakage when secure DM isolation is enabled.

## Solution Assessment

Unclear. Scoping only `deliveryContext` on `agent:main:main` would be a product change and could conflict with documented shared-main continuity; the narrow fix is to verify secure DM mode and then patch only the requester-origin or announce fallback that leaks.
