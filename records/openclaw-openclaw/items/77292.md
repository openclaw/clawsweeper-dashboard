## Summary

Keeping this open because it is a plausible cross-user privacy boundary report, but current `main` also documents shared DM context as the default and already has `session.dmScope` isolation, so the issue is not safe to close or automate without maintainer security/config triage.

## What This Changes

The issue asks OpenClaw to keep Telegram subagent and parent delivery context isolated per `chat_id` or sender when one bot serves multiple DM users.

## Best Possible Solution

Confirm the reporter's effective session configuration first; if it was shared-main mode, route them to secure DM mode, and if leakage reproduces with `per-channel-peer` or `per-account-channel-peer`, add a focused Telegram subagent/async-completion regression and fix requester-origin preservation.

## Reproduction Assessment

Unclear for the asserted regression. Source inspection gives a clear path for shared-main DM context sharing, but I did not establish a high-confidence current-main failure when secure DM isolation is enabled.

## Solution Assessment

Unclear. Scoping only `deliveryContext` by chat would be partial if the transcript is still shared; the maintainable fix depends on whether this is misconfiguration or a real secure-mode announce routing leak.
