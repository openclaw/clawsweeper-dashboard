## Summary

Keeping this open because the report is privacy-sensitive and plausible, but current main also documents shared-main Telegram DMs as unsafe by design unless `session.dmScope` is enabled. The missing piece is whether the leak reproduces under `per-channel-peer` or `per-account-channel-peer`, where it would be a real routing bug rather than the documented shared-DM mode.

## What This Changes

The issue asks OpenClaw to prevent subagent completion and parent replies from using another Telegram DM sender's delivery context when one bot serves multiple users.

## Best Possible Solution

Confirm the reporter's session configuration first; if they used shared-main DMs, route them to secure DM mode, and if leakage reproduces with `per-channel-peer` or `per-account-channel-peer`, add a focused Telegram multi-DM plus subagent announce regression and fix requester-origin preservation.

## Reproduction Assessment

Unclear for the claimed bug. Source inspection confirms the documented shared-main path (`dmScope: main` gives `agent:main:main`), but not a high-confidence current-main leak when secure DM isolation is enabled.

## Solution Assessment

Unclear. Scoping `deliveryContext` by chat on `agent:main:main` would conflict with documented shared-main mode; the safer maintainable fix is to verify secure-mode behavior and patch subagent announce routing only if that invariant fails.
