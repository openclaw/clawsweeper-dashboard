## Summary

Keep open: current main still documents shared-main DM mode as a context-sharing risk and implements secure `dmScope` alternatives, but this report is a privacy-sensitive Telegram/subagent routing claim and cannot be closed as fixed or expected behavior without the reporter's effective config and redacted route proof.

## What This Changes

The issue reports Telegram bot DM delivery context leaking subagent or parent replies across separate users when one agent serves multiple Telegram DMs.

## Best Possible Solution

Confirm the reporter's effective Telegram session configuration and redacted routing traces; keep shared-main setups on secure DM guidance, and if peer-scoped mode leaks, add a focused Telegram two-sender plus subagent completion regression that pins async delivery to the originating requester session.

## Reproduction Assessment

No high-confidence runtime reproduction is available. Source inspection does show the shared-main context path and the secure peer-scoped alternative, but not a proven current-main leak when `dmScope` is `per-channel-peer` or `per-account-channel-peer`.

## Solution Assessment

Unclear. Scoping only `deliveryContext` by chat while leaving transcript context in `agent:main:main` would be partial; the maintainable solution is secure DM configuration unless a focused regression proves requester-origin preservation fails in peer-scoped mode.
