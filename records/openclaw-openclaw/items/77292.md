## Summary

Keep open: current main has requester-origin plumbing for subagent announces, but default-account Telegram DMs can still share `agent:main:main` and shared last-route delivery state, and I did not find an exact current-main regression test proving async delivery is pinned per `chat_id`.

## What This Changes

Ensure Telegram bot DM delivery context for subagent completions and parent follow-ups remains scoped to the originating `chat_id`/sender when multiple users share one bot/main agent.

## Best Possible Solution

Add a focused routing invariant and regression test that async subagent and parent delivery always use the captured Telegram origin (`channel`, `to`, `accountId`, `threadId`) for the initiating DM, while leaving intentional `dmScope` transcript policy unchanged.

## Reproduction Assessment

No executed repro; there is a source-backed path. Default-account Telegram DMs can share `agent:main:main`, inbound messages update shared last-route metadata to `telegram:<chatId>`, and the current tests do not cover same Telegram channel/account with different DM targets during subagent completion.

## Solution Assessment

No proposed patch is available to judge. The best fix is a narrow delivery-context pinning test and implementation around Telegram `to`/origin handling, not a broad `dmScope` default change or a docs-only workaround.
