Keep open: current main still has documented shared-main DM behavior that can collapse multiple Telegram DM senders onto one session, while the report does not include the reporter's effective `session.dmScope` or enough redacted routing proof to distinguish documented shared-main risk from a peer-scoped isolation bug. ## What This Changes

The issue reports Telegram bot DM delivery context and subagent completion routing leaking across users when one bot serves multiple DM senders.

## Best Possible Solution

Confirm the reporter's effective Telegram session configuration and redacted routing traces, then either point shared-main deployments to DM isolation or add a focused regression and fix for peer-scoped subagent completion delivery.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. Source inspection shows the shared-main DM context-sharing path clearly, but the issue does not prove leakage when `session.dmScope` is `per-channel-peer` or `per-account-channel-peer`.

## Solution Assessment

Unclear. Scoping only `deliveryContext` by chat could be partial if transcript/session state remains shared; the safer path is to verify DM scope and pin subagent announce delivery to the recorded requester session when isolated DM mode is in use.
