## Summary

Keep open: current main still lets a channel-specific direct-chat provider label flow into the CLI session reuse hash, while `dmScope: "main"` intentionally collapses direct messages to one session key and `resolveCliSessionReuse` invalidates on hash drift. The open group-channel sibling at https://github.com/openclaw/openclaw/issues/69118 is related, but this direct-DM cross-channel trigger is distinct enough to track or fix explicitly.

## What This Changes

The issue requests stable Claude CLI session reuse when one logical direct-message session is shared across delivery channels under `session.dmScope: "main"`.

## Best Possible Solution

Fix the CLI reuse identity so channel-specific direct-message delivery labels do not invalidate a shared logical direct session, while preserving the delivered prompt guidance and keeping related group-hash work tracked separately.

## Reproduction Assessment

Yes for a source-level reproduction: current main deterministically builds different static prompt hashes for Telegram and WebChat direct-context labels while `dmScope: "main"` shares one session key. I did not run a live Claude CLI plus Telegram/WebChat gateway scenario during this read-only review.

## Solution Assessment

Yes, the best fix is a narrow reuse-identity change: keep useful channel text in the prompt that reaches the model, but exclude or normalize channel-variant direct context from the CLI session reuse hash when the session key can be shared. Broad raw-tail continuity work at https://github.com/openclaw/openclaw/issues/58818 is complementary, not a substitute for fixing the false reset.
