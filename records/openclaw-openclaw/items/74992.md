## Summary

Keep open. The PR addresses a valid current-main gap: configured provider models can resolve without `cost` even though Pi model transports expect a complete cost object. The code approach is narrow and security-clean, but this user-facing runtime fix is missing the required changelog entry before merge.

## What This Changes

The PR adds resolver-boundary zero-fill normalization for missing or non-numeric model cost fields, switches the skipPiDiscovery empty registry path to `ModelRegistry.inMemory`, and adds a regression test for configured provider models that omit cost.

## Best Possible Solution

Land the resolver-boundary normalization with its regression test after adding the required changelog entry and passing maintainer validation, keeping the fix local to model resolution rather than changing config schema semantics or the pinned Pi dependency contract.

## Reproduction Assessment

Yes. A configured provider model with `api` but no `cost` follows the inline match path on current main, and OpenAI/Anthropic transports call Pi `calculateCost` with the resolved model; the added test directly exercises that path, though it was not run during this read-only review.

## Solution Assessment

Yes for the code path. Normalizing after provider/runtime/canonical model normalization is the narrowest maintainable boundary, and using `ModelRegistry.inMemory(authStorage)` matches the pinned Pi public API; the remaining issue is merge readiness, not the implementation approach.
