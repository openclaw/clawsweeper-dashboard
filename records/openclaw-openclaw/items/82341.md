## Summary

Keep open: this is a protected external PR with a valid memory-core bug target, but the current patch still has a startup catch-up race and the contributed proof does not exercise the live gateway restart or no-force CLI recovery path.

## What This Changes

The PR adds memory-core session startup and no-force CLI catch-up scanning that compares transcript path, mtime, and size against persisted SQLite file rows, marks stale session files dirty, and adds focused tests.

## Best Possible Solution

Land the mtime/size reconciliation as a bounded memory-core fix, but route startup catch-up through a queued/follow-up sync path that cannot be coalesced away and require live gateway or CLI recovery proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main has no startup session catch-up scan, loses in-process session dirty state across restart, and the `session-start`/`watch` gate skips session syncing when there is no force, target, or full reindex.

## Solution Assessment

No, not as-is. Durable session transcript reconciliation is the right narrow direction, but the startup catch-up sync must survive active-sync coalescing and the PR still needs real gateway or CLI recovery proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Queue startup catch-up after active syncs:** `extensions/memory-core/src/memory/manager-sync-ops.ts:559`
  - body: When startup catch-up marks files dirty and then calls `this.sync`, `MemoryIndexManager.sync` returns any already-running non-targeted sync instead of queuing this catch-up. If that active sync is a `session-start` or `watch` pass, the existing gate skips session syncing even with dirty files, so the newly detected stale transcripts can remain pending with no follow-up sync. Queue the catch-up after the active sync or route it through a queued path so startup reconciliation cannot be swallowed.
  - confidence: 0.84
