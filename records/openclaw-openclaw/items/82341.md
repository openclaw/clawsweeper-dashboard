## Summary

Keep open. The PR targets a real memory-session indexing bug and includes useful before/after proof, but the current patch can still drop the startup catch-up behind an in-flight excluded sync, so it should not merge as-is.

## What This Changes

The PR adds memory-core startup and CLI catch-up logic that compares session transcript file mtime/size against persisted SQLite file rows and marks stale session files dirty for incremental sync.

## Best Possible Solution

Land a startup catch-up scan that marks stale session transcripts and guarantees a follow-up non-excluded sync, with a regression test for the in-flight `session-start` race.

## Reproduction Assessment

Yes. Current main has no startup catch-up scan and still excludes clean `session-start`/`watch` syncs when dirty state was lost; the contributor’s harness also shows stale transcript marking only after the patch.

## Solution Assessment

No. The catch-up direction is right, but the implementation should queue or rerun a non-excluded sync after dirty files remain instead of relying on a `sync()` call that can be coalesced into an excluded in-flight sync.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Queue catch-up after in-flight startup syncs:** `extensions/memory-core/src/memory/manager-sync-ops.ts:557`
  - body: After `markSessionStartupCatchupDirtyFiles()` sets `sessionsDirtyFiles`, this `sync()` call can be coalesced into an already-running `session-start` or `watch` sync. Those reasons are excluded by `shouldSyncSessionsForReindex`, so the in-flight sync can leave the files dirty without indexing them, and this catch-up path never schedules a second pass. Please ensure a follow-up `session-startup-catchup` sync runs when dirty files remain after any in-flight sync completes.
  - confidence: 0.83
