## Summary

Keep this PR open: it targets a real memory-core session indexing bug, but the branch still has a correctness race where the new catch-up sync can be collapsed into an already-running non-targeted sync, and the PR is protected by the `maintainer` label.

## What This Changes

The PR adds memory-core startup and no-force CLI catch-up scanning for persisted `sessions` rows by comparing transcript path, mtime, and size, then marking stale session files dirty and scheduling incremental sync.

## Best Possible Solution

Land a bounded reconciliation that marks stale session files and guarantees a follow-up session sync after active syncs, with live gateway or CLI recovery proof; leave broader status observability to the linked follow-up.

## Reproduction Assessment

Yes, by source inspection. Current main loses in-process session dirty state across restart, and `session-start`/`watch` or no-force CLI paths can skip session syncing when no full reindex or targeted sync is present; the linked issue provides operator reproduction context.

## Solution Assessment

No, not as-is. Durable mtime/size reconciliation is the right narrow direction, but the catch-up sync must survive active sync coalescing and the PR still needs real gateway or CLI recovery proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Queue startup catch-up after active syncs:** `extensions/memory-core/src/memory/manager-sync-ops.ts:559`
  - body: When startup catch-up marks files dirty and then calls `this.sync`, `MemoryIndexManager.sync` returns any already-running non-targeted sync instead of queuing this catch-up. If that active sync is a `session-start` or `watch` pass, the existing gate skips session syncing even with dirty files, so the newly detected stale transcripts can remain pending with no follow-up sync. Schedule the catch-up after the active sync or route it through a queued path so startup reconciliation cannot be swallowed.
  - confidence: 0.82
