## Summary

Keep open. The PR addresses a real current-main memory-core bug, but the proposed startup catch-up can still be swallowed by an already-running non-targeted sync, and the refreshed proof exercises a harness seam rather than a live gateway restart or no-force CLI recovery path.

## What This Changes

The PR adds memory-core startup and no-force CLI reconciliation for stale session transcript files by comparing on-disk mtime/size with persisted index rows and marking changed files dirty for incremental sync.

## Best Possible Solution

Land a bounded reconciliation that marks stale session files and guarantees a follow-up session sync even when startup overlaps other syncs, while leaving status wording and broader observability to the linked follow-up.

## Reproduction Assessment

Yes. Current-main source shows the reproducible state: empty in-process session dirty files plus `session-start`/`watch` or no-force CLI reaches a gate that skips `syncSessionFiles`; https://github.com/openclaw/openclaw/issues/62625 provides operator repro context.

## Solution Assessment

No, not as-is. Durable mtime/size reconciliation is the right narrow direction, but the startup scheduling needs to survive concurrent sync collapse and the PR still needs real gateway or CLI recovery proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Queue startup catch-up after an active sync:** `extensions/memory-core/src/memory/manager-sync-ops.ts:559`
  - body: When startup catch-up marks files dirty and then calls `this.sync`, `MemoryIndexManager.sync` returns any already-running non-targeted sync instead of queuing this catch-up. If that active sync is a `session-start` or `watch` pass, the existing gate skips session syncing even with dirty files, so the newly detected stale transcripts can remain pending with no follow-up sync. Schedule the catch-up after the active sync or route it through a queued/targeted path so startup reconciliation cannot be swallowed.
  - confidence: 0.82
