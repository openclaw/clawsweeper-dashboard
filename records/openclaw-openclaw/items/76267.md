## Summary

Keep open. Current main only preserves legacy `claude-cli/*` allowlist keys when they are still present, but canonical-only Claude CLI configs from the Anthropic CLI migration can still reject legacy isolated cron `payload.model` values before execution.

## What This Changes

The issue asks for isolated cron `payload.model` values using legacy `claude-cli/*` refs to keep working after the Claude CLI runtime and canonical model migration.

## Best Possible Solution

Restore legacy runtime-ref compatibility for isolated cron when the canonical Anthropic model is allowed and Claude CLI runtime execution is selected, while preserving fail-closed behavior for genuinely unallowed models.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a canonical-only Claude CLI config with `agents.defaults.models` containing `anthropic/claude-sonnet-4-6` and `agentRuntime.id: "claude-cli"`, then an isolated cron `payload.model: "claude-cli/claude-sonnet-4-6"`; the current resolver checks the exact legacy key and rejects it.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing legacy runtime alias seam during cron allowlist resolution or migration, not to relax the model allowlist or add a new fallback execution policy.
