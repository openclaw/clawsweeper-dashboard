## Summary

Keep open. Current main still has a source-reproducible compatibility gap: isolated cron validates `payload.model` against the exact resolved allowlist key, while Claude CLI migration and docs now steer users to canonical `anthropic/*` allowlist entries plus `agentRuntime.id: "claude-cli"`.

## What This Changes

The issue requests restoring isolated cron support for legacy `claude-cli/*` model refs, or an equivalent Claude Max CLI-backed scheduled-task path, after the v2026.4.29 regression.

## Best Possible Solution

Restore legacy runtime-ref compatibility for isolated cron by accepting `claude-cli/<claude-model>` when its canonical `anthropic/<model>` is allowed and CLI runtime execution is selected, while preserving fail-closed validation for genuinely unallowed models.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: use a migrated Claude CLI config with `agents.defaults.models` containing canonical `anthropic/claude-sonnet-4-6`, then run an isolated `agentTurn` cron payload with `model: "claude-cli/claude-sonnet-4-6"`; the current resolver rejects the exact `claude-cli/*` key before execution. I did not run a live cron job because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing legacy runtime alias seam during model allowlist resolution for cron payloads, not to relax the allowlist or add a fallback execution path. The fix should keep invalid refs and unrelated unallowed providers rejected.
