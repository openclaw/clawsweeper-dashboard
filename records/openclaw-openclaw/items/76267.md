## Summary

Keep open. Current main still has a source-reproducible compatibility gap: isolated cron validates the raw `payload.model` against exact allowlist keys, while Claude CLI migration/defaulting stores canonical `anthropic/*` allowlist entries, so legacy `claude-cli/*` cron payloads can still be rejected despite the documented compatibility promise.

## What This Changes

The issue reports a regression where isolated cron jobs using legacy `claude-cli/*` model refs are rejected after upgrading from v2026.4.23 to v2026.4.29.

## Best Possible Solution

Restore legacy runtime-ref compatibility so `claude-cli/<model>` is accepted when the canonical `anthropic/<model>` is allowed and Claude CLI runtime execution is selected, while keeping unrelated invalid or unallowlisted models fail-closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a migrated Claude CLI config with `agentRuntime.id: "claude-cli"` and `agents.defaults.models` containing `anthropic/claude-sonnet-4-6` but not `claude-cli/claude-sonnet-4-6`, then run an isolated `agentTurn` cron payload with `model: "claude-cli/claude-sonnet-4-6"`; the exact-match resolver rejects it before execution.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing legacy runtime alias concept during allowlist resolution or migration/default backfill, not to relax the allowlist or silently fall back to the default model.
