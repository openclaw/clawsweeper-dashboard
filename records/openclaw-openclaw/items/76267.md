## Summary

Keep open: current `main` still rejects legacy `claude-cli/*` cron payload models when Anthropic CLI setup has migrated the allowlist to canonical `anthropic/*` entries, and the active related PR only addresses canonical `anthropic/*` runtime stamping, not this preflight rejection.

## What This Changes

The issue reports that isolated cron payloads using `claude-cli/` model refs are rejected on v2026.4.29 instead of running through the Claude CLI backend.

## Best Possible Solution

Preserve legacy runtime-model compatibility in the shared model-selection boundary so `claude-cli/*` cron payloads validate against canonical Anthropic allowlist entries and then execute through the compatible CLI runtime.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.agentRuntime.id: "claude-cli"`, canonical `agents.defaults.models["anthropic/claude-sonnet-4-6"]`, and an isolated cron `payload.model: "claude-cli/claude-sonnet-4-6"`, current `main` parses the payload as `claude-cli/claude-sonnet-4-6` and rejects it before execution.

## Solution Assessment

No patch is proposed here, but the best fix is likely a generic legacy-runtime alias resolution in model selection rather than a cron-only string bypass; it should compose with the provider/runtime-pair-aware fix being discussed in #75840.
