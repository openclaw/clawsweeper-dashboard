## Summary

Keep open. Current main has fixed related pieces, but the exact compatibility gap remains source-reproducible: a cron `payload.model` using `claude-cli/*` is still resolved and allowlist-checked as a literal `claude-cli/*` key, while Anthropic CLI setup and docs steer users to canonical `anthropic/*` entries plus `agentRuntime` policy.

## What This Changes

The issue asks OpenClaw to keep legacy `claude-cli/*` model refs usable in isolated cron jobs, or provide an equivalent Claude Max CLI-backed scheduled-task path after the v2026.4.29 regression.

## Best Possible Solution

Add a narrow compatibility path so `claude-cli/<model>` cron payloads are accepted when the canonical `anthropic/<model>` is allowed and the selected provider/model runtime policy resolves to `claude-cli`, while preserving fail-closed behavior for unallowed and incompatible refs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: use the Anthropic CLI migration shape with canonical `anthropic/claude-sonnet-4-6` in `agents.defaults.models` plus `agentRuntime: { id: "claude-cli" }`, then resolve an isolated cron `payload.model` of `claude-cli/claude-sonnet-4-6`; current main checks the literal `claude-cli/*` key and rejects it. I did not execute a live cron run in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow resolver or cron-selection compatibility seam that maps legacy runtime refs to the allowed canonical ref only when the configured runtime/provider pair is compatible. The fix should not relax the allowlist globally, undo canonical migration, or add a new config knob.
