## What This Changes

Telegram progress mode should keep inter-tool commentary and later tool-progress lines together in the live draft when commentary progress is enabled.

## Best Possible Solution

Resolve the scope question on https://github.com/openclaw/openclaw/pull/90997, refresh it against current main, keep a focused Telegram dispatcher regression, and add live Telegram proof before closing this issue as fixed.

## Reproduction Assessment

Yes at source level: drive the Telegram dispatcher in progress mode through partial commentary, tool progress, more partial commentary, then another tool-progress event. Current main routes the partial answer text through prepareAnswerLaneForText, suppressing the compositor before the later tool line can render.

## Solution Assessment

Unclear until the linked PR's scope question is resolved. The narrow commentary-enabled dispatcher fix is plausible, but maintainers need to decide whether untagged inter-tool text should be rendered as commentary or suppressed by the broader tool-use text policy.
