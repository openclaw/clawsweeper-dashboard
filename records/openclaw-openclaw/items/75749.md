## Summary

Current main still has the reported delivery-path mismatch: Telegram native plugin approvals can resolve a session-bound Telegram target when `turnSourceChannel` is absent, while the forwarding fallback suppression returns false before consulting that same session binding. This is a valid, narrow bug rather than an implemented or duplicate cleanup item.

## What This Changes

The issue requests preventing duplicate Telegram plugin approval prompts when a plugin approval request lacks `turnSourceChannel` but is still session-bound to Telegram.

## Best Possible Solution

Add a regression-covered suppression path that derives the effective approval channel from either explicit turn-source fields or the persisted session binding, then suppresses the forwarding fallback only when Telegram native delivery is actually enabled for that effective route. Keep #74003 focused on forwarding missing metadata from `before_tool_call`.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram route; current source shows the native runtime resolves the session target while fallback suppression does not fire.

## Solution Assessment

Yes, the proposed direction is sound, but the safest implementation should use the existing approval capability/session-binding contracts rather than adding a Telegram-specific core special case. It should include negative tests for non-Telegram, unbound, and native-disabled requests.
