## Summary

Keep open: current main still has the reported mismatch where Telegram fallback forwarding requires an explicit Telegram turn source before checking native delivery, while the Telegram native approval path can still handle session-bound plugin approvals without that field. The related #74003 metadata propagation issue is adjacent but does not fully cover this duplicate-delivery path.

## What This Changes

Fix Telegram plugin approval delivery so forwarding fallback is suppressed when a request lacks `turnSourceChannel` but is session-bound to a Telegram route handled by native approvals.

## Best Possible Solution

Add a regression-covered shared suppression path that derives the effective approval channel/account from explicit turn-source fields or the persisted session binding, then suppresses fallback only when Telegram native delivery is enabled for that effective route.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram route; current source shows fallback suppression does not fire while native delivery can handle it.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery helper/session-binding path, with Telegram using the existing SDK capability seam rather than adding a Telegram-specific core special case.
