## Summary

Current main still has the reported mismatch: the forwarding fallback suppression returns false when Telegram requires a matching `turnSourceChannel` and the request omits it, while Telegram native approval routing can still accept the same plugin approval through the session/account binding. This is a fresh, valid, narrow bug rather than stale cleanup or a duplicate of #74003.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when a request has a Telegram-bound session but omits `turnSourceChannel`, causing both the forwarding fallback and Telegram native approval runtime to deliver.

## Best Possible Solution

Add a regression-covered generic suppression path that derives the effective approval channel/account from explicit turn-source fields or the persisted session binding, then suppresses Telegram forwarding only when native delivery is actually enabled for that effective route. Keep #74003 focused on forwarding missing metadata from `before_tool_call` rather than relying on that separate fix to prevent duplicates.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram session binding, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram route; current source shows native routing accepts the session-bound request while fallback suppression does not fire.

## Solution Assessment

Yes, the proposed direction is the right shape if it uses the existing approval/session-binding contracts instead of a Telegram-specific core special case. The safer implementation should include negative tests for non-Telegram requests, unbound sessions, explicit mismatched turn sources, and disabled native delivery.
