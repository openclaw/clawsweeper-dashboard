## Summary

Current main still has the reported delivery-path mismatch, so this should stay open. Telegram native plugin approvals can resolve a session-bound Telegram target when `turnSourceChannel` is absent, while the forwarding fallback suppression returns before consulting native delivery/account binding, leaving both paths able to send the same prompt.

## What This Changes

The issue asks OpenClaw to prevent duplicate Telegram plugin approval prompts when a request omits `turnSourceChannel` and both forwarding fallback and Telegram native delivery are enabled.

## Best Possible Solution

Add a regression-covered suppression path that derives the effective approval channel from explicit turn-source fields or persisted session binding, then suppresses the forwarding fallback only when Telegram native delivery is enabled for that effective route. Keep the missing `before_tool_call` metadata work tracked separately in #74003 unless the same helper naturally needs shared route metadata plumbing.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram route; current main shows native delivery resolves that session target while fallback suppression does not fire.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix if it uses the existing approval capability/session-binding contracts rather than a Telegram-specific core special case, with negative coverage for non-Telegram, unbound, and native-disabled requests.
