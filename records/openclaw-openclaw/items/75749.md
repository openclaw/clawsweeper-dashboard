## Summary

Current main still has the reported mismatch: Telegram native plugin approvals can resolve a session-bound request without `turnSourceChannel`, while the shared forwarding fallback suppression returns before checking native delivery when that field is absent. This is narrow, current, and actionable.

## What This Changes

Prevent duplicate Telegram plugin approval prompts when a request has a Telegram session binding but no `turnSourceChannel`.

## Best Possible Solution

Add regression-covered suppression that derives the effective Telegram channel/account from explicit turn-source metadata or persisted session binding before allowing the forwarding fallback to send.

## Reproduction Assessment

Yes. A focused test can create a Telegram session-store entry, submit a plugin approval request with only `sessionKey`, and assert that Telegram native handling accepts it while current forwarding fallback suppression does not suppress the Telegram fallback target.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery suppression/session-binding path, with Telegram continuing to supply its channel/account facts through the existing approval capability seams.

## Repair Work Prompt

Fix the Telegram duplicate plugin approval delivery reported in https://github.com/openclaw/openclaw/issues/75749. Current main lets Telegram native plugin approvals handle a session-bound request with only `sessionKey`, but `createApproverRestrictedNativeApprovalCapability(...).delivery.shouldSuppressForwardingFallback` returns false before native delivery/account checks when `turnSourceChannel` is missing. Add a narrow fix in the shared approval delivery/session-binding path so fallback suppression derives the effective channel/account from explicit turn-source fields or the persisted session binding, then suppresses Telegram fallback only when native delivery is enabled for that effective route. Preserve explicit cross-channel forwarding and multi-account scoping. Do not treat #74003 as the whole fix; route metadata propagation is separate. Add regression coverage for a Telegram plugin approval with session-store binding and null/absent `turnSourceChannel`, plus guard coverage for non-Telegram/foreign-channel and disabled-native cases. Update `CHANGELOG.md` under the active Fixes section for this user-visible bug, crediting @choury if attribution is included.
