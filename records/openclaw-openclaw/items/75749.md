## Summary

Current main still has the reported mismatch: plugin approval fallback delivery suppresses Telegram only when `turnSourceChannel` is explicitly `telegram`, while the Telegram native runtime can handle the same session-bound request without that field. The related #74003 is adjacent metadata propagation work, but this duplicate-delivery path remains independently valid.

## What This Changes

Prevent duplicate Telegram plugin approval prompts when approval requests omit `turnSourceChannel` and both fallback forwarding and Telegram native delivery are eligible.

## Best Possible Solution

Add a regression-covered suppression path that derives the effective approval channel and account from explicit turn-source fields or the persisted session binding, then suppresses the forwarding fallback only when Telegram native delivery is actually enabled for that effective route.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and fallback forwarding to Telegram; current source shows native handling remains eligible while fallback suppression does not fire.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval capability/session-binding suppression path, with Telegram supplying channel facts through the existing SDK seam rather than adding a Telegram-specific core special case.

## Repair Work Prompt

Fix openclaw/openclaw#75749. Current main duplicates Telegram plugin approval prompts when a plugin approval has a Telegram-bound `sessionKey` but no `turnSourceChannel`: the forwarding fallback does not suppress because `createApproverRestrictedNativeApprovalCapability` requires an explicit matching turn source, while Telegram native approval handling can still match from the persisted session binding. Implement the fix in the shared approval delivery/session-binding seam, not as a hardcoded core Telegram special case. Preserve fallback delivery when the explicit turn source is another channel, when the target is not Telegram, or when Telegram native delivery is disabled/unconfigured. Add regression coverage for a plugin approval with no `turnSourceChannel` and a Telegram session-bound route so fallback forwarding is suppressed only when native Telegram delivery is enabled. Also keep #74003 separate; do not solve that metadata propagation issue unless needed for the regression. Update `CHANGELOG.md` with a single-line user-facing fix entry if production behavior changes. Likely files: `src/plugin-sdk/approval-delivery-helpers.ts`, `src/infra/approval-request-account-binding.ts` or `src/infra/exec-approval-session-target.ts`, `extensions/telegram/src/approval-native.ts`, `src/plugin-sdk/approval-delivery-helpers.test.ts`, `src/infra/plugin-approval-forwarder.test.ts`, `extensions/telegram/src/approval-native.test.ts`, and possibly `extensions/telegram/src/exec-approvals.test.ts`. Do not add bundled channel ids to generic core beyond existing manifest/capability seams.
