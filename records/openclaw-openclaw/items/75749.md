## Summary

Current main still has the reported mismatch: the forwarding fallback suppression requires an explicit Telegram turn source before it checks native delivery, while Telegram native plugin approvals can still handle the same request through the session-bound route. The related #74003 is adjacent metadata propagation work, not a complete duplicate of this duplicate-delivery path.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when `turnSourceChannel` is absent because fallback forwarding and the Telegram native approval runtime can both deliver the same request.

## Best Possible Solution

Add a regression-covered suppression path that derives the effective approval channel/account from explicit turn-source fields or the persisted session binding, then suppresses the forwarding fallback only when Telegram native delivery is actually enabled for that effective route. Keep #74003 focused on forwarding missing metadata from `before_tool_call`.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram route; current source shows the native runtime handles it while fallback suppression does not fire.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval capability/session-binding suppression path, with Telegram supplying channel facts through the existing SDK seam rather than adding a Telegram-specific core special case.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75749. Current `main` suppresses Telegram approval forwarding only when `turnSourceChannel` explicitly normalizes to `telegram`; when a plugin approval has a Telegram-bound `sessionKey` but no turn-source fields, the fallback forwarder still sends while the Telegram native runtime also delivers. Implement a generic approval suppression path that can derive the effective channel/account from the request's persisted session binding when explicit turn-source metadata is missing, and suppress fallback only when native delivery is enabled for that effective Telegram route. Do not add a core hardcoded Telegram special case if the existing approval capability/session-binding helpers can express it. Add negative coverage for unbound requests, non-Telegram or mismatched session bindings, mismatched accounts, and native-disabled accounts so fallback remains available when native delivery cannot handle the request.
