## Summary

Current main still has the reported mismatch: forwarding fallback suppression exits before native-delivery/account checks when `turnSourceChannel` is absent, while Telegram native plugin approvals can resolve the session-bound request and deliver, so the duplicate-prompt bug remains actionable.

## What This Changes

Fix duplicate Telegram plugin approval prompts when a plugin approval has a Telegram-bound session but no `turnSourceChannel`.

## Best Possible Solution

Add regression-covered suppression that derives the effective approval channel/account from explicit turn-source fields or persisted session binding, then suppresses the forwarder only when Telegram native delivery is enabled for that route.

## Reproduction Assessment

Yes. A focused test can create a Telegram session-store entry, a plugin approval request with only `sessionKey`, and a Telegram plugin forwarding target; current source shows fallback delivery is not suppressed while native Telegram delivery still matches the session-bound request.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery suppression/session-binding path, with Telegram supplying channel facts through existing SDK/runtime helpers; #74003 should remain separate metadata propagation work.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75749. Current main lets Telegram plugin approvals with a Telegram-bound `sessionKey` and no `turnSourceChannel` be delivered twice: the shared forwarder fallback does not suppress because `requireMatchingTurnSourceChannel` short-circuits on the missing source, while Telegram native delivery resolves the same request through session binding. Implement a narrow shared suppression fix that derives an effective channel/account from explicit turn-source metadata or persisted session binding, and suppresses only when the effective route is Telegram and native delivery is enabled for that account. Keep #74003 focused on forwarding missing `before_tool_call` metadata; do not add Telegram-specific core ids beyond the existing capability seams, do not disable generic fallback when native delivery is unavailable, and preserve multi-account behavior. Add regression coverage for plugin approvals with no `turnSourceChannel`, a Telegram session binding, and plugin forwarding to Telegram. Add a user-facing `CHANGELOG.md` fix entry if production behavior changes.
