## Summary

Current main still has the reported mismatch: forwarding fallback suppression returns false when `turnSourceChannel` is absent, while the Telegram native runtime can still handle the same session-bound plugin approval, so duplicate Telegram prompts remain plausible and actionable.

## What This Changes

Prevent Telegram plugin approvals without `turnSourceChannel` from being delivered by both the forwarding fallback and the native Telegram runtime.

## Best Possible Solution

Teach the shared fallback-suppression path to derive the effective channel/account from persisted session binding when explicit turn-source metadata is absent, then add a Telegram plugin-approval regression while leaving #74003 as separate metadata propagation work.

## Reproduction Assessment

Yes. A focused regression can create a Telegram session-store entry, submit a plugin approval with only `sessionKey`, and assert that native Telegram handling matches while forwarding fallback is currently not suppressed.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery/session-binding seam with Telegram using that generic route data; adding a Telegram-only duplicate-send filter would leave the contract drift in place.

## Repair Work Prompt

Fix the Telegram plugin approval duplicate-delivery bug where a request with `turnSourceChannel` null/undefined but a Telegram-bound `sessionKey` is handled by the native Telegram approval runtime and also delivered by the forwarding fallback. Keep the fix generic where possible: update the shared approval delivery suppression/session-binding path so native fallback suppression can derive the effective channel/account from explicit turn-source fields or persisted session binding, and suppress the forwarder only when native delivery is enabled for that effective Telegram route. Do not solve #74003 by forwarding new route metadata from `before_tool_call` in this PR except where a small test fixture needs explicit inputs; treat #74003 as separate. Add regression coverage for a Telegram plugin approval with only `sessionKey` and a persisted Telegram session store entry, plus any focused shared-helper tests needed to lock the null-source suppression behavior. Add a user-facing `CHANGELOG.md` fix entry. Avoid broad approval architecture refactors and do not change unrelated channel approval behavior.
