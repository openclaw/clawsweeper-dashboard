## Summary

Current main still has the reported route mismatch: Telegram fallback suppression requires an explicit `turnSourceChannel`, while the Telegram native approval runtime can still handle the same session-bound plugin approval without that field, so this remains a valid duplicate-delivery bug.

## What This Changes

Fix Telegram plugin approval delivery so the forwarding fallback and native runtime do not both send when `turnSourceChannel` is missing.

## Best Possible Solution

Add regression-covered suppression that derives the effective approval route from explicit turn-source metadata or the persisted session binding, then suppresses Telegram forwarding only when the native approval client is enabled for that route.

## Reproduction Assessment

Yes. A focused test can create a Telegram session-store entry, send a plugin approval request with `sessionKey` but no `turnSourceChannel`, and show that current forwarding suppression returns false while the Telegram native handler still matches the same request.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery/session-binding path, with Telegram continuing to supply channel-specific account and native-delivery facts through its existing capability adapter; #74003 should remain separate routing-metadata propagation work.

## Repair Work Prompt

Fix openclaw/openclaw#75749. Current main lets Telegram plugin approval requests without `turnSourceChannel` be delivered twice: `src/plugin-sdk/approval-delivery-helpers.ts` returns false before account/native checks, while Telegram native approval handling can resolve the same request from the persisted session binding. Implement the smallest shared-route fix so forwarding fallback suppression can derive the effective channel/account from explicit turn-source fields or the persisted session binding, and only suppress when Telegram native delivery is enabled for that route. Keep #74003 as separate metadata propagation work; do not require every caller to populate `turnSourceChannel` to avoid duplication. Add focused regression coverage for a plugin approval request with `sessionKey` and no `turnSourceChannel`, including the Telegram session-bound native handling path and forwarder suppression path. Add a user-facing CHANGELOG.md fix entry. Do not broaden plugin SDK APIs unnecessarily, do not move Telegram-owned auth/delivery defaults into core, and do not touch unrelated approval or channel behavior.
