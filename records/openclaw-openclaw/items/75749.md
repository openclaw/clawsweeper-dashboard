## Summary

Current main still shows the reported duplicate-delivery mismatch: Telegram native plugin approvals can handle a session-bound request with no `turnSourceChannel`, while the forwarding fallback suppression exits before checking native delivery when that field is absent. The related #74003 metadata propagation issue is adjacent, but it does not remove this fallback/native dedupe bug.

## What This Changes

The issue requests preventing duplicate Telegram plugin approval prompts when a session-bound approval lacks `turnSourceChannel` and both forwarding fallback and Telegram native delivery can send it.

## Best Possible Solution

Add a regression-covered shared suppression path that derives the effective approval channel/account from explicit turn-source fields or the persisted session binding, then suppresses the Telegram forwarding fallback only when native delivery is enabled for that effective route.

## Reproduction Assessment

Yes. A focused test can create a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to Telegram; current source shows native handling proceeds while fallback suppression does not fire.

## Solution Assessment

Yes. The maintainable fix belongs in the shared approval capability/session-binding suppression path, with Telegram supplying channel facts through the existing SDK seam rather than adding a hard-coded Telegram special case in core delivery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75749. Telegram plugin approvals with a Telegram-bound `sessionKey` and no `turnSourceChannel` can currently be sent twice: once by the forwarding fallback and once by Telegram native approval delivery. Implement a generic suppression path in the approval delivery/capability layer that can derive the effective channel/account from explicit turn-source fields or the persisted session binding when `turnSourceChannel` is absent. Suppress the forwarding fallback only when the target channel is Telegram, the effective route is Telegram, and native delivery is enabled for the effective account. Preserve forwarding when the explicit source is another channel, no matching session binding exists, native delivery is disabled, or the target account does not match. Keep #74003 scoped to metadata propagation; this fix must not depend on that issue being fixed first. Add focused regression tests for the shared helper and plugin forwarding path, and update `CHANGELOG.md` if production behavior changes.
