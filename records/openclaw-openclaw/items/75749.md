## What This Changes

The issue requests fixing duplicate Telegram plugin approval prompts when a plugin approval lacks `turnSourceChannel` and both forwarding fallback and native Telegram delivery send.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89688, or an equivalent helper-level fix, after proof shows one Telegram approval prompt while preserving foreign-channel and disabled-native fallback delivery.

## Reproduction Assessment

Yes, source-reproducible: current main exits fallback suppression before checking native delivery when `turnSourceChannel` is null, while Telegram native matching can still accept the request via session/account binding. No live Telegram DM run was performed in this read-only review.

## Solution Assessment

Yes, likely: aligning the shared native fallback suppression helper with Telegram's session/account matching is the narrow owner-boundary fix. The linked PR should remain the active path until it lands or maintainers choose a replacement.
