## Summary

Keep open: current main still has the reported Telegram plugin-approval duplicate path, because forwarding fallback suppression requires an explicit Telegram turn source while the native Telegram runtime can handle the same session-bound request without one.

## What This Changes

Fix Telegram plugin approval delivery so a session-bound request with missing `turnSourceChannel` is delivered by only one path, not both the forwarding fallback and native runtime.

## Best Possible Solution

Add regression-covered suppression that derives the effective approval channel/account from explicit turn-source fields or the persisted session binding, then suppresses only when Telegram native delivery is actually enabled for that effective route; keep #74003 focused on metadata propagation.

## Reproduction Assessment

Yes. A focused regression can seed a Telegram session entry, create a plugin approval with that `sessionKey` and no `turnSourceChannel`, enable Telegram native approvals plus plugin forwarding, and observe that current source lets both delivery paths match.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval delivery/session-binding helper path with Telegram-provided channel facts, not as a Telegram-specific core special case or by broadening suppression without account checks.

## Repair Work Prompt

Fix the Telegram duplicate plugin approval delivery reported in https://github.com/openclaw/openclaw/issues/75749. Current main suppresses the forwarding fallback only when `turnSourceChannel` explicitly matches Telegram, while Telegram native approval handling can still match a plugin approval via the persisted session binding when `turnSourceChannel` is missing. Implement a narrow shared approval-delivery fix, likely in `src/plugin-sdk/approval-delivery-helpers.ts` with any needed helper from the existing approval session/account binding seams, so fallback suppression derives an effective channel/account from explicit turn-source fields or the persisted session binding and suppresses only when native delivery is enabled for that effective Telegram route. Add regression coverage for a plugin approval with a Telegram-bound `sessionKey`, no `turnSourceChannel`, Telegram native approvals enabled, and plugin forwarding to the same Telegram destination. Do not add Telegram-specific owner IDs or bundled-channel special cases in core; keep #74003 focused on forwarding missing `before_tool_call` metadata. Add a user-facing `CHANGELOG.md` fix entry unless the final patch is tests-only.
