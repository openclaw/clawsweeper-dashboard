## Summary

Keep this issue open: current main still has the reported suppressor/native-runtime mismatch, and the issue is already paired with an open closing PR rather than being implemented on main.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when `turnSourceChannel` is missing because generic fallback delivery and Telegram native approval delivery can both handle the same request.

## Best Possible Solution

Land a focused suppressor fix that lets Telegram use account/session evidence when `turnSourceChannel` is missing, while preserving fallback delivery for channels that cannot prove native delivery will handle the same target.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: call the shared suppressor for Telegram with `requireMatchingTurnSourceChannel` and a missing `turnSourceChannel`, while the native runtime can still match by account/session binding.

## Solution Assessment

No, closing is not the best way to solve this yet. The best path is to review, repair, or replace the open linked fix PR, then close this issue after the suppressor behavior is on main with Telegram-visible proof.
