## Summary

Keep open: current main still has the reported Telegram plugin-approval delivery mismatch, where native Telegram routing can handle a session-bound request without `turnSourceChannel` but forwarding fallback suppression returns false before checking native delivery.

## What This Changes

The issue asks OpenClaw to stop duplicate Telegram plugin approval prompts when `turnSourceChannel` is absent or null.

## Best Possible Solution

Use the existing session-binding route helpers in shared approval delivery suppression so Telegram fallback is suppressed only when native delivery will handle the same effective route.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a Telegram session-store entry, submit a plugin approval request with `sessionKey` but no `turnSourceChannel`, and native Telegram can handle it while fallback suppression returns false.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared approval suppression/session-binding path; related #74003 should remain the separate metadata propagation fix.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75749. Current main lets Telegram native plugin approvals handle session-bound requests with no `turnSourceChannel`, but forwarding fallback suppression returns false when `requireMatchingTurnSourceChannel` is true and the field is absent. Update the shared approval delivery suppression path to derive the effective channel/account from explicit turn-source fields or persisted session binding, then suppress Telegram forwarding fallback only when native delivery is enabled for that same route. Keep related #74003's before_tool_call metadata propagation separate unless a test fixture needs to document the interaction. Do not add Telegram-specific IDs to core beyond the existing capability seam. Add regression coverage for a plugin approval with Telegram `sessionKey` and null/missing `turnSourceChannel` proving fallback delivery is skipped, while explicit non-Telegram sources and disabled native delivery still forward. Add a user-facing `CHANGELOG.md` fix entry.
