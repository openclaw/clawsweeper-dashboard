## Summary

Keep open: current main still has the reported mismatch. Telegram native approval routing can use the session-bound origin when `turnSourceChannel` is absent, but the forwarding fallback suppressor returns false before it checks the same native/session route, so the duplicate-delivery path remains possible.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when a session-bound approval request omits `turnSourceChannel`.

## Best Possible Solution

Teach the shared fallback suppression path to use the same session/account binding facts as native approval routing before deciding Telegram native delivery will not handle the request, while keeping #74003 as the separate metadata propagation fix.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run: create a Telegram session-store entry, submit a plugin approval request with that `sessionKey` and no `turnSourceChannel`, and enable both Telegram native approvals and plugin approval forwarding. The native path can resolve the session while the fallback suppressor returns false before checking native delivery.

## Solution Assessment

Yes. The narrow maintainable fix is in shared delivery suppression/session-binding logic, not in Telegram transport code and not by relying on #74003 alone.
