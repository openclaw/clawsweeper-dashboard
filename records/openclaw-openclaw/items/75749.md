## Summary

Keep open: current main still has the reported Telegram plugin-approval delivery mismatch. The forwarding fallback suppression path still rejects null `turnSourceChannel` before it can consult session-bound native delivery facts, while Telegram native approval routing can handle the same session-bound request.

## What This Changes

Fix duplicate Telegram plugin approval prompts when a request has a `sessionKey` but no `turnSourceChannel`.

## Best Possible Solution

Use the existing session-binding route helpers in the shared approval suppression path so Telegram forwarding fallback is suppressed only when native delivery will handle the same effective route.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a Telegram session-store entry, submit a plugin approval request with `sessionKey` but no `turnSourceChannel`, and native Telegram can handle it while fallback suppression returns false.

## Solution Assessment

Yes. The narrow maintainable fix is to make forwarding fallback suppression use the same session-bound channel/account facts as native delivery; related #74003 should remain the separate metadata propagation fix.
