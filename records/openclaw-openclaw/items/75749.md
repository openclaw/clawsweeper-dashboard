## Summary

Keep open: current main still has the Telegram plugin-approval delivery mismatch, where fallback suppression rejects a null `turnSourceChannel` before checking native delivery while Telegram native routing can still resolve the session-bound request.

## What This Changes

The issue asks OpenClaw to stop sending duplicate Telegram plugin-approval prompts when an approval request has a session key but no `turnSourceChannel`.

## Best Possible Solution

Make shared fallback suppression use the same effective session-binding and native-delivery route facts as Telegram native handling, while keeping the related turn-source metadata propagation issue separate.

## Reproduction Assessment

Yes, at source level. A Telegram session-store entry plus a plugin approval request with `sessionKey` and no `turnSourceChannel` lets native Telegram resolve the session-bound route while fallback suppression returns false before checking native delivery.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared suppression/session-binding path, with regression coverage for null-channel Telegram requests and guard coverage for explicit non-Telegram sources.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75749 on current main. The bug is duplicate Telegram plugin-approval delivery when a request has a `sessionKey` but no `turnSourceChannel`: Telegram native approval can resolve the session-bound route, while `shouldSuppressForwardingFallback` returns false before native-delivery/account checks, so the forwarder fallback also sends. Keep the fix narrowly in the shared approval fallback suppression/session-binding path, or a small generic helper it uses. Do not solve the separate https://github.com/openclaw/openclaw/issues/74003 metadata propagation issue here, and do not source routing from model-controlled tool params. Preserve forwarding for explicit non-Telegram turn sources, disabled native delivery, and mismatched Telegram accounts. Add regression coverage for a Telegram session-bound plugin approval with null/undefined `turnSourceChannel` suppressing fallback only when native Telegram delivery is enabled for the effective account. Add or update a changelog entry if the repo policy requires one for the user-visible fix.
