## Summary

Keep open: current main and the latest release still have the null `turnSourceChannel` fallback-suppression gap, and the direct fix is the open, unmerged PR https://github.com/openclaw/openclaw/pull/89688.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when a null `turnSourceChannel` lets both the forwarding fallback and Telegram native approval runtime deliver the same request.

## Best Possible Solution

Land or replace the focused fix so Telegram fallback suppression handles missing `turnSourceChannel` only when native delivery can actually handle the same approval request.

## Reproduction Assessment

Yes at source level: current main skips fallback suppression on null `turnSourceChannel`, while Telegram native matching can still accept the request by account/session binding. I did not perform a live Telegram DM run in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix belongs in the shared native/fallback suppression path with Telegram-specific native-delivery proof. The open PR is the right work owner, but it still needs review or replacement before this issue closes.
