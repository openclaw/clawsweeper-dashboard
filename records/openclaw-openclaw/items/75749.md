## Summary

Keep open: current main and the latest release still return before suppressing Telegram forwarding fallback when `turnSourceChannel` is missing, while the direct fix candidate remains open and unmerged.

## What This Changes

The issue reports duplicate Telegram plugin approval prompts when a null `turnSourceChannel` lets both the generic forwarder fallback and Telegram native approval runtime deliver the same request.

## Best Possible Solution

Land or replace the focused fix so Telegram suppresses generic fallback delivery only when native delivery can handle the same null-source request, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: current main skips fallback suppression on null `turnSourceChannel`, while Telegram native matching can still accept a session/account-bound request. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix is to suppress the generic fallback only when Telegram native delivery can handle the same null-source request. The current open PR is the direct candidate, but it needs rebase/proof or a replacement before this issue can close.
