## Summary

Keep open. Current main still lacks the requested agent-visible `[OPENCLAW-REDACTED]` marker, shared helper, and `sessions_history.notice` field, so the PR is not obsolete; however the current PR head still has correctness and rebase blockers around visible redaction gating, current-main forced tool redaction, docs, and changelog credit.

## What This Changes

The PR adds exported redaction marker constants and a `redactSensitiveTextForAgent` helper, emits a `sessions_history` notice when redaction is reported, and updates logging/session-history tests plus the changelog.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #68425, but land it only after the notice condition is based on redaction visible in the final returned payload, the branch is refreshed onto current main while preserving force-on tool redaction, the session-tool docs describe the new marker contract, and the changelog entry carries the required contributor credit.

## Reproduction Assessment

Yes. A high-confidence reproduction is a `sessions_history` response whose surviving message has safe text before the 4,000-character cutoff and a secret only after it; the PR sets `entry.redacted` before truncation, so it emits `notice` even though the returned text contains no redacted placeholder.

## Solution Assessment

No, not as-is. The additive marker/notice direction is the right narrow fix, but the implementation should compute the marker from the post-truncation, post-cap payload and preserve current main's force-on tool redaction contract.
