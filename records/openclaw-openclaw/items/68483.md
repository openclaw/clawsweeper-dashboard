## Summary

Keep this PR open. The additive marker direction is useful and the latest patch preserves current main's force-on session-history redaction, but it still has a concrete false-positive notice case after text truncation plus small docs/changelog policy gaps.

## What This Changes

The PR adds redaction marker constants and a helper, emits a `sessions_history.notice` when redacted content is detected, expands redaction/session-history tests, and adds a changelog entry.

## Best Possible Solution

Land a refreshed patch that emits the marker only when the final returned agent-visible payload visibly contains redacted placeholders or uses wording that exactly matches the semantics, documents the contract, and preserves force-on tool redaction.

## Reproduction Assessment

Yes, from source inspection for the remaining PR bug: place a secret only after the 4,000-character cutoff in a retained message, and the PR can emit `notice` even though the returned text has no redacted placeholder.

## Solution Assessment

No, not as-is. The marker/notice approach is the right narrow hardening, but the notice gate must reflect the actual returned payload and the new contract needs docs and changelog credit.
