## Summary

Keep open. This PR is still the active implementation path for the redacted-output marker issue, but the current head can emit the new notice when per-text truncation removed the only visible redaction, the new stable contract is undocumented, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds `[OPENCLAW-REDACTED]` constants, a `redactSensitiveTextForAgent` helper, a conditional `sessions_history.notice` field, regression tests, and a changelog entry.

## Best Possible Solution

Gate the notice on redaction visible in the final returned payload, document the marker/notice contract, and add real `sessions_history` proof before merge.

## Reproduction Assessment

Yes. Source inspection shows a retained `sessions_history` text block with more than 4,000 safe characters followed by an sk-style token sets `entry.redacted` before `truncateHistoryText` cuts the returned text, so the PR can emit a notice with no visible placeholder.

## Solution Assessment

No, not as-is. The additive marker is the right narrow direction, but the notice must reflect visible post-truncation redaction or use different wording, and the stable response contract needs documentation.

## Review Findings

Overall correctness: patch is incorrect

This branch then adds a notice saying redaction happened in the returned output even though no masked placeholder is visible; compute the notice from the final returned text or adjust the contract wording.

  - confidence: 0.8
