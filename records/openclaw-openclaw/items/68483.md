## Summary

Keep open: the PR is a useful security hardening direction, but it still gates the new notice on a pre-truncation redaction flag and lacks contributor-provided real behavior proof for the after-fix `sessions_history` output.

## What This Changes

The PR adds redaction marker constants, a `redactSensitiveTextForAgent` helper, conditional `sessions_history.notice` output, regression tests, and a changelog entry.

## Best Possible Solution

Gate the notice on redaction visible in the final returned payload, document the `sessions_history.notice` marker contract, and require redacted real-behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible: return a retained `sessions_history` text block with more than 4,000 safe characters followed by an sk-style token. The PR records redaction before the 4,000-character cut, so it can emit `notice` even though the returned text has no redacted placeholder.

## Solution Assessment

No, not as-is. The additive marker is a narrow maintainable direction, but the notice must reflect the final returned payload and the stable response contract needs public documentation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate the notice on visible redaction:** `src/agents/tools/sessions-history-tool.ts:278-280`
  - body: `contentRedacted` now ignores messages dropped by byte and hard caps, but it still uses `entry.redacted`, which is computed before `truncateHistoryText` applies the 4,000-character cut. If a retained message has its only secret beyond that cutoff, the returned `messages` contain no masked placeholder while this branch still adds a marker claiming redaction happened in the output. Recompute the condition from the final returned text or change the contract wording.
  - confidence: 0.91
- **[P3] Document the marker contract:** `src/logging/redact.ts:208-214`
  - body: The new constants describe a stable downstream contract and the PR adds `sessions_history.notice`, but the public session-tool docs still only list `contentRedacted` among summary flags. Please document the `notice` field and `[OPENCLAW-REDACTED]` guidance so agents and tooling have a durable contract to follow.
  - confidence: 0.8
