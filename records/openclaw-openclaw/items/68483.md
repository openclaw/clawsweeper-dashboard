## Summary

Keep open. The marker/notice direction is the right fix for the linked redacted-output bug, but the current PR head is not merge-ready because it is stale against current main's force-on session-history redaction and can emit a notice when truncation removed the redacted text from the returned payload.

## What This Changes

The PR adds redaction marker constants and a `redactSensitiveTextForAgent` helper, emits a `sessions_history.notice` when redaction is detected, expands redaction/session-history tests, and adds a changelog entry.

## Best Possible Solution

Land a refreshed patch that preserves current-main force-on tool payload redaction, emits the marker only when the final returned session-history payload visibly contains redacted text or uses exact wording for pre-return redaction, and documents the contract with proper changelog credit.

## Reproduction Assessment

Yes. A focused reproduction is `sessions_history` with `logging.redactSensitive: "off"` and a secret in recalled text for the force-on regression, plus a long message with safe text before the 4,000-character cutoff and a secret only after it for the false-positive notice.

## Solution Assessment

No, not as-is. The additive marker/notice approach is the narrow maintainable direction, but the PR must be rebased onto the current force-on redaction contract and derive or word the notice from the actual returned payload semantics.
