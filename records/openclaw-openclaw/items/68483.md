## Summary

Keep open: the PR is a useful redaction hardening direction, but the latest head is not merge-ready because the new notice can still be emitted when truncation removed the only visible redacted placeholder, and the external PR only provides tests/CI rather than real after-fix behavior proof.

## What This Changes

The PR adds redaction marker constants, a `redactSensitiveTextForAgent` helper, `sessions_history.notice` output when redaction is detected, tests, and a changelog entry.

## Best Possible Solution

Update the PR so `sessions_history.notice` reflects the final returned payload, document the marker contract, and add redacted real behavior proof showing the after-fix tool response.

## Reproduction Assessment

Yes. From source inspection, put a secret-like token only after the 4,000-character cutoff in a retained `sessions_history` text block; `entry.redacted` remains true while the returned text lacks a redacted placeholder, so the PR emits a misleading notice.

## Solution Assessment

No, not as-is. The marker/notice approach is a narrow hardening path, but the notice gate must reflect visible final output semantics and the public contract needs docs.
