## Summary

Keep open: the PR is still useful and current main does not implement the marker, but PR head has a source-reproducible false-positive notice after per-text truncation, lacks public docs for the stable marker contract, and lacks after-fix real behavior proof.

## What This Changes

The PR adds redaction marker constants, a `redactSensitiveTextForAgent` helper, a conditional `sessions_history.notice` response field, regression tests, and a changelog entry.

## Best Possible Solution

Gate the notice on redaction visible in the final returned payload, document the `sessions_history.notice` marker contract, and keep the helper scoped to agent-visible tool output.

## Reproduction Assessment

Yes, source-reproducible: a retained `sessions_history` text block with more than 4,000 safe characters followed by an sk-style token sets `entry.redacted` before `truncateHistoryText` cuts the returned text. That lets the new notice claim redaction in the output when no placeholder is visible.

## Solution Assessment

No, not as-is. The additive marker is a reasonable direction, but the notice must reflect visible post-truncation redaction or use different wording, and the stable response contract should be documented.
