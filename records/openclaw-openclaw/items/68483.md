## Summary

Keep open: the marker/notice direction addresses the real safety gap tracked in https://github.com/openclaw/openclaw/issues/68425, but this PR still has a source-reproducible false-positive notice path after per-text truncation and lacks contributor real-behavior proof.

## What This Changes

The PR adds stable redaction marker constants, a `redactSensitiveTextForAgent` helper, a conditional `sessions_history.notice` field, regression tests, and a changelog entry.

## Best Possible Solution

Gate the notice on redaction visible in the final returned payload, document the new `sessions_history.notice` marker contract, and require redacted real-behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible: return a retained `sessions_history` text block with more than 4,000 safe characters followed by an sk-style token. `truncateHistoryText` records redaction before the 4,000-character cut, so this PR emits `notice` even though the returned text lacks a redacted placeholder.

## Solution Assessment

No, not as-is. The additive marker is a narrow maintainable hardening path, but the notice must reflect the final returned payload and the stable response contract needs public documentation.

## Review Findings

Overall correctness: patch is incorrect

  - body: `contentRedacted` now ignores messages dropped by byte/hard caps, but it still uses `entry.redacted`, which is computed before `truncateHistoryText` applies the 4,000-character cut.   - confidence: 0.8
