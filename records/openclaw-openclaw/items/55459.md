## Summary

Keep open: current main and the latest release still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, and no canonical open item owns this exact macOS/TCC cache-daemon workaround.

## What This Changes

The issue requests a fix or documentation for 1Password `op` exec secret providers spawning cache daemons and hanging under macOS LaunchAgent use unless `--cache=false` or equivalent cache disabling is applied.

## Best Possible Solution

Update the 1Password SecretRef docs to disable the CLI cache through existing `args` or `env`, then let maintainers separately decide whether any provider-specific runtime behavior belongs in core.

## Reproduction Assessment

Partly: source inspection reproduces the current cache-enabled docs/runtime shape, and the 1Password CLI cache contract matches the reporter's workaround. No live macOS Tahoe plus 1Password CLI run was performed in this review.

## Solution Assessment

Unclear for runtime behavior: documenting `--cache=false` or `OP_CACHE=false` is the narrowest maintainable fix using existing config, while core auto-detection of `op` would be provider-specific policy that needs maintainer/security approval.
