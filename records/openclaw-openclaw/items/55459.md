## Summary

Keep open: current main and the latest release still document a 1Password `op read` exec-provider example without disabling the 1Password CLI cache, and the generic exec resolver only passes configured args/env rather than applying any 1Password-specific safeguard.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password CLI cache-daemon hangs when `op read` is used as an exec SecretRef provider on macOS LaunchAgents.

## Best Possible Solution

Keep the generic exec-provider contract unchanged and update the official 1Password SecretRef docs to disable the 1Password CLI cache for daemon/service-account setups using existing `args` or `env` support.

## Reproduction Assessment

No. Source inspection and upstream 1Password docs show the documented `op read` path runs without disabling the default cache, but no live macOS Tahoe plus 1Password CLI run was performed.

## Solution Assessment

Yes for the narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses existing exec-provider config and avoids a core 1Password special case.
