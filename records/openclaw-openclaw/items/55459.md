## Summary

Keep open: current main and v2026.6.10 still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, so the reported macOS LaunchAgent hang/crash-loop remains actionable but needs maintainer/security direction.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password CLI cache-daemon hangs for exec SecretRef providers on macOS LaunchAgent setups.

## Best Possible Solution

Document `--cache=false` or `OP_CACHE=false` in the 1Password SecretRef docs using the existing exec `args`/`env` surface, then separately decide whether any provider-specific runtime default belongs in core or a plugin.

## Reproduction Assessment

Partly: source inspection confirms current main and v2026.6.10 still use the cache-enabled 1Password exec-provider shape, and 1Password documents the cache opt-out. A live macOS Tahoe plus 1Password CLI run is still missing.

## Solution Assessment

Unclear for runtime behavior: documenting `--cache=false` or `OP_CACHE=false` is the narrowest maintainable fix using existing config, while core auto-detection of `op` needs maintainer/security approval.
