## Label Justifications

- `P2`: The report can block gateway startup for a specific but plausible 1Password exec SecretRef setup on macOS LaunchAgent installs.
- `impact:auth-provider`: The affected path is SecretRef provider resolution for credential material backed by the 1Password CLI.
- `impact:crash-loop`: The reported failure mode repeatedly re-enters startup under launchd and accumulates hung provider processes.

## Summary

Keep open: current main and v2026.6.6 still document and execute the 1Password exec-provider path without disabling the 1Password CLI cache, and the related broader crash-loop/batching issues do not own this narrow `OP_CACHE=false` or `--cache=false` gap.

## What This Changes

The issue asks OpenClaw to prevent 1Password CLI cache-daemon hangs for exec secret providers on macOS Tahoe, preferably by disabling `op` cache or documenting the workaround.

## Best Possible Solution

Update the supported 1Password exec-provider guidance to disable 1Password CLI caching through existing `args` or `env` configuration, and leave broader startup retry/backoff policy to the open crash-loop tracker.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection plus the 1Password CLI contract show current main still follows a cache-enabled `op read` shape unless users add `--cache=false` or `OP_CACHE=false`, but the macOS Tahoe daemon/TCC hang itself remains reporter-backed.

## Solution Assessment

Yes for a narrow docs/config repair: current exec provider config already supports putting global `op` flags first in `args` or setting `OP_CACHE=false` in `env`. A core auto-detect special case for 1Password is not clearly the best fix without maintainer product approval.
