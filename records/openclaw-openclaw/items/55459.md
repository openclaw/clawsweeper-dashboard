## Summary

Keep open: current main and the latest release still document a 1Password exec SecretRef example that runs `op read` without disabling the 1Password CLI cache, while the exec resolver only passes configured args/env and has no 1Password-specific safeguard. A narrow docs and skill update is actionable; adding automatic `op` detection or a new config field would be a separate product decision.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password CLI cache-daemon hangs for exec SecretRef providers on macOS Tahoe by using `--cache=false` or `OP_CACHE=false`.

## Best Possible Solution

Update the official gateway secrets docs and bundled 1Password skill references to recommend `OP_CACHE=false` or a prepended `--cache=false` for 1Password CLI exec providers in LaunchAgent or service-account setups, while leaving generic exec-provider runtime behavior unchanged.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows current main would run the documented 1Password command without disabling the cache, which matches the reported failure path and the upstream 1Password cache contract.

## Solution Assessment

Yes for a narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses existing config fields and avoids a core special case. A broader automatic 1Password workaround or new `globalArgs` field is not the best first fix without maintainer product review.
