## Summary

Keep open: current main and v2026.6.10 still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, and this issue remains the canonical tracker for the exact macOS/TCC cache-daemon workaround.

## What This Changes

The issue asks OpenClaw to prevent 1Password CLI exec SecretRefs from spawning cache daemons on macOS LaunchAgent installs, via runtime cache disablement, a new global/prepended args path, or documentation.

## Best Possible Solution

Document `--cache=false` or `OP_CACHE=false` in the 1Password SecretRef docs using the existing exec `args`/`env` surface, then separately decide whether any generic runtime or plugin API behavior is needed.

## Reproduction Assessment

Partly: source inspection reproduces the current cache-enabled docs/runtime shape, and the 1Password CLI cache contract matches the reporter's workaround. No live macOS Tahoe plus 1Password CLI run was performed.

## Solution Assessment

Unclear for runtime behavior: documenting `--cache=false` or `OP_CACHE=false` is the narrowest maintainable fix using existing config, while core auto-detection of `op` would be provider-specific policy that needs maintainer/security approval.
