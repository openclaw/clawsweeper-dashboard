## Summary

Keep open: current main still documents a 1Password exec provider without `--cache=false` or `OP_CACHE=false`, and the generic exec resolver only passes explicitly configured args/env, so the reported workaround is not reflected in code or docs.

## What This Changes

Request that OpenClaw handle or document 1Password `op` exec providers with cache disabled so macOS LaunchAgent startup does not hang on spawned `op daemon` processes.

## Best Possible Solution

Update the official 1Password exec SecretRef documentation and bundled 1Password skill references to disable `op` caching for LaunchAgent/service-account usage, leaving any core auto-detection special-case to a separate maintainer decision.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection shows current main would invoke the documented 1Password command without the cache-disabling flag/env, which matches the reported failure path.

## Solution Assessment

Yes for a narrow docs repair: update the 1Password exec provider recipe to disable the `op` cache and pass the needed service-account environment. A core `op` autodetect workaround is not the narrowest maintainable fix without separate maintainer review.
