## Summary

Keep open: current main and the latest release still document a 1Password exec SecretRef example that runs `op read` without disabling the 1Password CLI cache, and the generic exec resolver only passes configured args/env with no 1Password-specific safeguard.

## What This Changes

The issue asks OpenClaw to prevent documented 1Password CLI exec SecretRefs from spawning cache daemon processes in macOS LaunchAgent/service-account setups, at least by documenting `--cache=false` or `OP_CACHE=false`.

## Best Possible Solution

Keep the generic exec-provider contract unchanged and update the official 1Password SecretRef docs to disable the 1Password CLI cache for daemon/service-account setups with existing `args` or `env` support.

## Reproduction Assessment

No live high-confidence reproduction was established for macOS Tahoe plus 1Password CLI. Source inspection shows current main would run the documented 1Password command without disabling the cache, matching the reporter's failure path when the upstream cache daemon hangs.

## Solution Assessment

Yes for the narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses existing exec-provider config and avoids a core 1Password special case. Automatic command detection or a new argument field is not the best first fix without maintainer product review.
