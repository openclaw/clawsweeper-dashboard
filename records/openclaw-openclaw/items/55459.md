## What This Changes

The issue asks OpenClaw to prevent documented 1Password `op read` exec-secret setups from spawning cache daemons and hanging under macOS LaunchAgent/TCC conditions.

## Best Possible Solution

Keep the exec provider generic and update the official 1Password SecretRef docs to disable the 1Password CLI cache using existing `args` or `env` support, while leaving startup backoff and external-provider batching to the related open issues.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection plus the 1Password CLI contract show the documented `op read` path leaves caching enabled and OpenClaw forwards only configured args/env, which is enough for a docs repair but not an end-to-end Tahoe TCC proof.

## Solution Assessment

Yes for the narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses the existing exec-provider contract and avoids a core 1Password special case. The runtime auto-detect and new config-field suggestions are not the best immediate fix without maintainer/secops approval.
