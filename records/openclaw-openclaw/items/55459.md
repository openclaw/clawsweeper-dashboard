## Summary

Keep open: current main and the latest release still document a 1Password exec SecretRef example that runs `op read` without disabling the 1Password CLI cache, while the exec resolver only passes configured args/env and has no 1Password-specific safeguard. A narrow docs repair is clear; automatic `op` detection, new argument fields, retry/backoff, and external-provider batching belong to separate maintainer decisions or related issues.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password CLI cache-daemon hangs for exec SecretRef providers on macOS Tahoe.

## Best Possible Solution

Update the official 1Password exec SecretRef docs to disable the 1Password CLI cache with existing `args` or `env` support for daemon/service-account setups, while keeping generic exec-provider runtime behavior unchanged and leaving broader retry/batching work on the related issues.

## Reproduction Assessment

No live high-confidence reproduction was established for macOS Tahoe plus 1Password CLI. Source inspection does show current main would run the documented 1Password command without disabling the cache, matching the reporter's failure path and the upstream 1Password cache contract.

## Solution Assessment

Yes for the narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses existing exec-provider config and avoids a core 1Password special case. Automatic command detection or a new argument field is not the best first fix without maintainer product review.
