## Summary

Keep open: current main and the latest shipped docs still present a 1Password exec provider recipe that runs `op read` without disabling the 1Password CLI cache, while the exec resolver only uses configured args/env and does not special-case `op`.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password `op` CLI cache daemon hangs for exec SecretRef providers on macOS Tahoe.

## Best Possible Solution

Update the official 1Password exec SecretRef documentation and bundled 1Password skill references to disable `op` caching for LaunchAgent/service-account usage, while leaving generic exec-provider behavior unchanged.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows current main would invoke the documented 1Password command without the cache-disabling flag/env, which matches the reported failure path and upstream cache contract.

## Solution Assessment

Yes for the narrow docs repair: document `--cache=false` or `OP_CACHE=false` in the 1Password exec provider recipe. A core `op` autodetect workaround is not the narrowest maintainable solution without separate maintainer review.
