## Summary

Keep open: current main and v2026.6.8 still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, and no merged or viable canonical item owns this exact cache-daemon workaround.

## What This Changes

The issue reports that 1Password CLI exec secret providers can spawn cache daemons and hang on macOS Tahoe unless `op` caching is disabled.

## Best Possible Solution

Update the 1Password SecretRef docs to disable the 1Password CLI cache through existing `args` or `env`, then let maintainers separately decide whether broader startup retry/backoff belongs under the crash-loop policy issue.

## Reproduction Assessment

No, not as a high-confidence live repro: no macOS Tahoe plus 1Password CLI run was performed. Source inspection does confirm current main still follows a cache-enabled `op read` docs/runtime shape unless users add `--cache=false` or `OP_CACHE=false`.

## Solution Assessment

Unclear for runtime behavior: auto-detecting `op` in core is provider-specific policy. A narrow docs/config repair is the lowest-risk path because existing exec-provider `args` and `env` can express the workaround.
