## Summary

Keep open: current main and the latest release still document a 1Password `op read` exec-provider example without disabling the 1Password CLI cache, and the generic exec resolver only forwards configured args/env. A narrow docs repair remains useful while broader startup backoff and external-provider batching stay with the related open issues.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password `op` exec-provider cache daemon hangs on macOS Tahoe, especially by using `--cache=false` or `OP_CACHE=false`.

## Best Possible Solution

Keep the generic exec-provider runtime contract unchanged and update the official 1Password SecretRef docs to disable the 1Password CLI cache for service-account or LaunchAgent setups using existing `args` or `env` support.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection and upstream 1Password docs show the documented `op read` path runs without disabling the default cache, but the macOS Tahoe TCC hang itself was not replayed.

## Solution Assessment

Yes for the narrow repair: documenting `--cache=false` or `OP_CACHE=false` uses existing exec-provider config and avoids a core 1Password special case. The runtime auto-detect and new config-field options need maintainer/secops judgment before they should be pursued.
