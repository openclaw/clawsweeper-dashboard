## Summary

Keep open: current main and v2026.6.8 still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, and the broader crash-loop issue does not fully own this narrow cache-daemon workaround decision.

## What This Changes

The issue requests safer handling or documentation for 1Password `op` CLI cache daemon behavior when used through OpenClaw exec secret providers on macOS LaunchAgents.

## Best Possible Solution

Document and recommend disabling the 1Password CLI cache through existing exec-provider `args` or `env`, then let maintainers separately decide whether any provider-specific runtime policy is warranted.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows current main still follows a cache-enabled `op read` docs/runtime shape unless users add `--cache=false` or `OP_CACHE=false`, but the macOS Tahoe daemon/TCC hang itself remains reporter-backed.

## Solution Assessment

Unclear for runtime behavior: auto-detecting `op` in core is provider-specific policy. A narrow docs/config repair is currently the best low-risk path because existing exec-provider `args` and `env` can express the workaround.
