## Summary

Keep open. Current main still documents the 1Password exec provider as plain `op read` without `--cache=false` or `OP_CACHE=false`, while the exec resolver only passes through configured args/env and does not add any 1Password-specific mitigation. The broader batching issue #72278 and startup retry work around #56217/#56499 are related, but they do not supersede this cache-daemon workaround.

## What This Changes

The issue requests preventing 1Password CLI cache-daemon hangs when `op` is used as an exec SecretRef provider on macOS LaunchAgent setups, at least by documenting or configuring `--cache=false` / `OP_CACHE=false`.

## Best Possible Solution

Land a narrow docs/config guidance fix: update the 1Password exec provider examples to disable the 1Password cache for headless/LaunchAgent service-account use, prefer the existing `env`/ordered `args` surfaces, and keep broader startup retry and external-vault batching work on their canonical issues.

## Reproduction Assessment

Unclear for the full macOS Tahoe/TCC hang because this read-only review did not run 1Password CLI under a LaunchAgent. We do have a high-confidence static reproduction of the repository gap: follow the current 1Password secrets-doc example and OpenClaw will invoke `op read` without disabling the default UNIX cache daemon.

## Solution Assessment

Yes for the immediate repository fix: documenting `OP_CACHE=false` or leading `--cache=false` is the narrowest maintainable path because current config and CLI builder already support it. Core `op` auto-detection or a new `globalArgs` field should be separate maintainer-reviewed work, not the first repair.
