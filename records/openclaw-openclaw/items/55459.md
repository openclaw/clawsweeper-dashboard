## Summary

Keep open: current main and v2026.6.6 still document and execute the 1Password exec-provider path without disabling the 1Password CLI cache daemon, and no merged or open canonical item covers this narrow `OP_CACHE=false` / `--cache=false` gap.

## What This Changes

The issue requests that OpenClaw prevent or document 1Password CLI cache-daemon hangs for exec SecretRef providers on macOS Tahoe, especially by using `OP_CACHE=false` or `--cache=false`.

## Best Possible Solution

Keep this issue open and make a narrow maintainer-approved repair that updates the supported 1Password exec-provider guidance to disable CLI caching, without folding in broader retry/backoff or batching work.

## Reproduction Assessment

No, not with a high-confidence live reproduction in this review. Source and dependency-contract inspection show current main still follows a cached `op read` shape unless users manually add `OP_CACHE=false` or `--cache=false`.

## Solution Assessment

Unclear for the full request because runtime auto-detection and new argument surfaces are product/security decisions. The narrowest maintainable fix appears to be docs/config guidance that disables 1Password CLI caching for the supported example.
