## Summary

Keep open: current main and the latest release still document and execute the 1Password exec-provider path without disabling the 1Password CLI cache daemon, while broader crash-loop and batching follow-ups do not cover this narrow `OP_CACHE=false`/`--cache=false` gap.

## What This Changes

The issue asks OpenClaw to prevent or document 1Password `op read` cache-daemon hangs when used as an exec SecretRef provider on macOS Tahoe LaunchAgents.

## Best Possible Solution

Keep this issue open and land a narrow docs/config repair that makes the supported 1Password exec-provider example disable CLI caching without adding a new config surface or hardcoded provider policy.

## Reproduction Assessment

No live high-confidence macOS Tahoe repro was run. Source inspection plus official 1Password CLI docs show current main still follows the documented `op read` shape that leaves caching enabled unless users add `OP_CACHE=false` or `--cache=false`.

## Solution Assessment

Yes for a narrow docs/config fix: update the 1Password exec-provider guidance to disable caching and add focused coverage only if code examples are encoded in tests. A core auto-detect special case for 1Password is a less maintainable default without maintainer approval.
