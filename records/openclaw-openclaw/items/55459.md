## Summary

Keep open. Current main still documents the 1Password `op read` exec-provider example without disabling the cache daemon, and there is no `op`-specific mitigation in the resolver; existing `args` and `env` support make this a narrow docs/workaround repair rather than a missing config capability. The related #72278 tracks broader external-provider batching, but it does not supersede this macOS Tahoe/1Password cache issue.

## What This Changes

Request to mitigate 1Password CLI exec-provider hangs on macOS Tahoe by disabling `op` caching, adding a global/prepended-args path, or documenting the workaround.

## Best Possible Solution

Update the official 1Password exec-provider docs/example to disable `op` caching via first-position `--cache=false` or `OP_CACHE=false`, include the required service-account `passEnv` guidance, and leave broader provider batching/memoization to #72278.

## Reproduction Assessment

No live high-confidence reproduction was established; I did not run `op` on macOS Tahoe as a LaunchAgent. The source-backed path is clear: current docs configure `op read` without cache disabling, while the resolver will pass that command through and surface the documented timeout if it stalls.

## Solution Assessment

Yes for the docs/workaround path, no for adding a new global-args field. Existing `args` and `env` already support the workaround, while automatic `op` detection would be a separate product-policy choice.
