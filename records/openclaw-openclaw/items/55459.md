## Summary

Keep open: current main and v2026.6.8 still document and execute the 1Password `op read` exec-provider shape without disabling the 1Password CLI cache, and the remaining choice is maintainer/security policy rather than a safe automatic cleanup close.

## What This Changes

The issue requests handling or documentation for 1Password CLI exec secret providers that need `--cache=false` or `OP_CACHE=false` on macOS Tahoe LaunchAgent setups.

## Best Possible Solution

Document and recommend disabling the 1Password CLI cache through existing exec-provider `args` or `env`, then let maintainers separately decide whether any provider-specific runtime policy is warranted.

## Reproduction Assessment

No, not as a high-confidence live repro: no macOS Tahoe plus 1Password CLI run was performed. Source inspection does confirm current main still follows a cache-enabled `op read` docs/runtime shape unless users add `--cache=false` or `OP_CACHE=false`.

## Solution Assessment

Unclear for runtime behavior: auto-detecting `op` in core is provider-specific policy. A narrow docs/config repair is the lowest-risk path because existing exec-provider `args` and `env` can express the workaround.
