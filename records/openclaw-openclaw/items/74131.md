## Summary

This PR still needs author work because the new readiness preflight runs before later deterministic local spawn validation, so a degraded gateway can mask local model/context/session-prep errors that current main reports directly. The useful lifecycle retry work is focused and well-proven, but the spawn readiness placement should be narrowed before merge.

## What This Changes

The branch adds bounded gateway lifecycle readiness/retry handling for subagent spawn, announce finalization, and registry sweep cleanup, plus focused agent tests and one QA-lab WhatsApp test type correction.

## Best Possible Solution

Keep the lifecycle retry hardening, but move the spawn readiness probe to the last point before the first unavoidable gateway operation after deterministic local validation has completed, then rerun focused subagent lifecycle tests against the current merge result.

## Reproduction Assessment

Yes for the remaining blocker from source inspection: a degraded gateway causes the new readiness call to return before later deterministic local spawn checks can run. I did not run tests in this read-only review, but the path ordering is clear in the PR head source.

## Solution Assessment

No. The PR's retry/finalize/sweep direction is a reasonable fix, but the spawn readiness probe is still too early; the best fix preserves local validation error ordering and only probes immediately before the first unavoidable gateway call.
