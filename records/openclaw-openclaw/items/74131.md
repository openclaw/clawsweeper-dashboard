## Summary

Keep open: current main still has the single-shot subagent lifecycle paths, and this PR is still the active implementation path, but latest head leaves a narrow classifier gap in the new lifecycle retry wrappers before it is ready to merge.

## What This Changes

The PR adds an admin-scoped subagent gateway readiness probe, bounded retries for announce finalize and sweep delete lifecycle calls, focused agents tests, and changelog notes.

## Best Possible Solution

Ship the lifecycle hardening after the finalize and sweep retry paths share the same bounded lifecycle-readiness classifier and the exact head passes focused agents validation.

## Reproduction Assessment

Yes for the PR-head blocker: source inspection shows the new finalize/sweep retries use a classifier that omits readiness variants the same PR treats as transient in spawn. I did not live-reproduce the gateway race.

## Solution Assessment

No for the current PR head because the retry direction is narrow but the lifecycle retry classifier is incomplete. A safer fix is to share or mirror the lifecycle-readiness classifier across spawn, announce finalize, and sweep delete retries.

Full review comments:

- **[P2] Retry lifecycle readiness errors in finalize paths:** `src/agents/subagent-announce.ts:174`
  - body: The new finalize wrapper only retries errors accepted by `isTransientAnnounceDeliveryError`, but that classifier does not match `handshake timeout`, `closed before connect`, or `not yet ready to accept connections` even though this PR's spawn preflight treats those as gateway lifecycle readiness errors. A transient `sessions.patch` or `sessions.delete` with one of those messages will still take the best-effort catch path without retrying, leaving label/delete cleanup unfinished. Use a shared lifecycle-readiness classifier for finalize and sweep gateway calls and add coverage for these messages.
