## Summary

Keep open. Current main appears to have fixed the generic CLI-timeout wording via the related timeout issue, but the central Claude CLI native Read/wrapper boundary still is not resolved: native file tools remain outside OpenClaw's MCP loopback policy path, and deciding whether to support wrapper execution through claude-cli or route it through ACP/node exec needs maintainer judgment.

## What This Changes

The issue asks OpenClaw to avoid a long Telegram-visible claude-cli stall when a wrapper path triggers a native file Read, either by executing the approved wrapper or failing fast with a clear diagnostic.

## Best Possible Solution

Define the supported route for Telegram-triggered wrapper execution: either route it through ACP/node exec where OpenClaw owns policy and diagnostics, or make claude-cli native file-tool paths fail fast with sanitized, actionable guidance when OpenClaw cannot control them.

## Reproduction Assessment

Unclear. The issue has a detailed live reproduction and current source still has the same native Read outside-loopback boundary, but I did not establish a high-confidence current-main reproduction without the reporter's Telegram, Claude CLI, auth, wrapper, and node-approval setup.

## Solution Assessment

Unclear. The merged timeout-copy fix is the right narrow fix for the generic CLI timeout symptom, but the remaining Read/wrapper behavior needs a maintainer decision between fail-fast diagnostics, ACP/node exec routing, or broader claude-cli native-tool support.
