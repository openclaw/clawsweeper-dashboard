## Summary

Keep open. Current main still has the same structural failure boundary: the Claude CLI backend runs with native tools always on, OpenClaw's MCP loopback excludes native file tools such as Read from its policy path, minute-scale CLI watchdog timeouts are still possible, and Telegram still maps dispatch failures to the generic error text.

## What This Changes

Investigate and fix Telegram-driven claude-cli wrapper-path turns that stall during a Claude native Read and only deliver a generic Telegram error.

## Best Possible Solution

Define the supported Claude CLI route for Telegram-triggered wrapper execution, then either fail fast with sanitized diagnostics for unsupported native Read paths or route the workflow through ACP/node exec where OpenClaw owns tool policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for live current-main reproduction: the report is detailed and source inspection preserves the failure shape, but reproducing the actual Read stall requires the reporter's Telegram, Claude CLI, macOS wrapper, auth profile, and node approval setup.

## Solution Assessment

Is this the best way to solve the issue? Unclear. There is no proposed patch, and the narrowest maintainable fix depends on whether maintainers want to diagnose Claude native tools, change Telegram fallback disclosure, or declare raw CLI backend wrapper execution out of scope in favor of ACP/node exec.
