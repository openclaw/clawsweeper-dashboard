## Summary

Keep open. Current main fixed the related generic CLI timeout copy, but the central Claude CLI native Read/wrapper boundary is still unresolved and current source still keeps native file tools outside the OpenClaw loopback policy surface.

## What This Changes

The issue asks OpenClaw to avoid a Telegram-visible stall when a claude-cli turn reaches a native Read of an approved wrapper path, either by executing through supported policy or failing fast with clear diagnostics.

## Best Possible Solution

Define the supported Telegram-to-wrapper route, then either send it through ACP/node exec where OpenClaw owns policy and diagnostics or make claude-cli native file reads fail fast with sanitized guidance.

## Reproduction Assessment

No high-confidence full reproduction was established. The issue has detailed live evidence, and current source still shows the native Read path outside OpenClaw's loopback tool policy, but this review did not recreate the full Telegram, Claude CLI OAuth, wrapper, and node-approval setup.

## Solution Assessment

Unclear. The merged timeout-copy fix is the right narrow fix for one symptom, but the remaining wrapper/Read behavior needs a maintainer decision between fail-fast diagnostics, ACP/node exec routing, or broader claude-cli native-tool support.
