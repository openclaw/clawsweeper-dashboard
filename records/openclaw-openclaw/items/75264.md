## Summary

Keep open. The report is recent, detailed, and still lines up with current `main`: claude-cli runs as a live stdio CLI backend with bundled MCP, native file/exec tool names are excluded from OpenClaw's MCP loopback, the CLI watchdog can terminate a no-output turn on the same minute-scale timing, and Telegram still converts dispatch failures into a generic user-facing fallback. I did not find source, docs, tests, or history evidence that current `main` now fixes the wrapper-path `Read` stall or improves the Telegram diagnostic.

## What This Changes

The issue requests that Telegram turns using the claude-cli backend fail fast with a clear file/tool/exec diagnostic, or successfully run the approved wrapper, instead of stalling for about three minutes and sending only a generic Telegram error.

## Best Possible Solution

Keep this open until maintainers define the supported workflow and close the diagnostic gap: claude-cli native-tool stalls should fail fast with sanitized, actionable diagnostics; Telegram should surface a bounded timeout/tool message instead of only a generic fallback; and wrapper execution should be routed or documented through the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
