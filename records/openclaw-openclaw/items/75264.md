## Summary

Keep open. Current main still matches the reported failure boundary: Claude CLI native file tools remain outside OpenClaw's MCP loopback, the watchdog can still turn a silent CLI/tool stall into a minute-scale timeout, and Telegram still sends the generic dispatch fallback; the related #72338 work addresses adjacent gateway CPU/stall causes but does not prove this Claude native Read path is fixed.

## What This Changes

The issue reports that a Telegram-triggered Claude CLI backend turn stalls on a wrapper-path native Read call and only returns a generic Telegram error instead of stdout or a clear tool denial.

## Best Possible Solution

Define the supported route for Telegram-triggered wrapper execution on Claude CLI, then either fail fast with sanitized diagnostics for unsupported native Read paths or document and route the workflow through ACP/node exec where OpenClaw owns the tool policy.
