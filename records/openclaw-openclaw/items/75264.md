## Summary

Keep open. The report is recent, detailed, and still matches current main: claude-cli runs with bundled MCP and native tools, OpenClaw's MCP loopback excludes native file/exec tool names, CLI no-output watchdogs still use minute-scale timeouts, and Telegram still turns dispatch failures into a generic fallback. I did not find source, docs, tests, or post-report history proving the wrapper-path Read stall is fixed or intentionally unsupported with clear diagnostics.

## What This Changes

The issue asks OpenClaw to stop a Telegram-routed claude-cli turn from stalling on a wrapper-path file Read and to either execute the approved wrapper or fail fast with an actionable diagnostic.

## Best Possible Solution

Keep the issue open until maintainers define the supported workflow and close the diagnostic gap: claude-cli native-tool stalls should fail fast with sanitized, actionable logging; Telegram should surface a bounded timeout/tool diagnostic instead of only the generic fallback; and wrapper execution should be routed or documented through the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
