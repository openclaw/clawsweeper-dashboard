## Summary

Keep open. The merged timeout-copy fix addressed one symptom, but current main still leaves Claude native file tools outside OpenClaw's loopback policy/logging path, and wrapper execution through claude-cli versus fail-fast ACP/node guidance needs maintainer judgment.

## What This Changes

The issue reports Telegram turns using claude-cli stalling on a native Claude `Read` of an approved wrapper path and delivering only a generic error instead of wrapper stdout or a clear denial.

## Best Possible Solution

Define the supported Telegram-to-wrapper route: either send this workflow through ACP/node exec where OpenClaw owns policy and diagnostics, or make unsupported claude-cli native file-tool paths fail fast with sanitized actionable guidance.

## Reproduction Assessment

Unclear. The issue has detailed live evidence and current source still has the same native Read outside-loopback boundary, but I did not establish a high-confidence current-main reproduction without the reporter's Telegram, Claude CLI, auth, wrapper, and node-approval setup.

## Solution Assessment

Unclear. The merged timeout-copy classifier is the right narrow fix for the generic timeout symptom, but the remaining wrapper/native Read behavior needs a maintainer decision between fail-fast diagnostics, ACP/node exec routing, or broader claude-cli native-tool support.
