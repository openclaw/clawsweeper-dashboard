## Summary

Keep open. Current main still matches the reported failure shape: claude-cli uses bundled MCP plus Claude native tools, OpenClaw loopback excludes native file and exec tools, the CLI watchdog can produce a minute-scale no-output abort, and Telegram still collapses dispatcher failures to a generic fallback. The related closed CPU-spin report is adjacent but does not cover this out-of-workspace native Read stall or the Telegram diagnostic gap.

## What This Changes

The issue asks for Telegram turns through the claude-cli backend to either execute an approved wrapper path or fail fast with a clear tool error instead of stalling and returning a generic Telegram error.

## Best Possible Solution

Define the supported workflow boundary and implement it consistently: either make claude-cli native Read stalls fail fast with sanitized diagnostics that Telegram can surface, or document and route Telegram-to-wrapper execution through the intended ACP/node-exec path while preserving approval and file-scope policy.
