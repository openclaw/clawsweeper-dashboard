## Summary

Keep open. The report is recent, detailed, and still lines up with current main: the Claude CLI backend runs through bundled MCP plus native Claude tools, OpenClaw's loopback deliberately excludes native file/exec tools, the live Claude session still has a minute-scale no-output watchdog, and Telegram still collapses dispatcher failures to a generic fallback. The closed #72338 CPU-spin report is adjacent evidence but does not prove this out-of-workspace native Read stall or Telegram diagnostic gap is fixed.

## What This Changes

The issue reports a Telegram-to-claude-cli workflow that stalls during a Claude native file Read of an approved wrapper path and only returns Telegram's generic failure message.

## Best Possible Solution

Define and implement the supported workflow boundary: either make claude-cli native file-tool stalls fail fast with sanitized diagnostics and better Telegram surfacing, or clearly route Telegram-to-wrapper execution through the intended ACP/node-exec path while documenting why raw CLI backends are not supported for that workflow.
