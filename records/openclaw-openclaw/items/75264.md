## Summary

Keep open. Current main still matches the reported failure shape: claude-cli uses a live stdio backend with bundle MCP, OpenClaw deliberately excludes native file/exec tool names from the MCP loopback, the Claude live-session watchdog can abort a silent turn on the same minute-scale timing, and Telegram still has generic user-facing failure fallbacks. I did not find current source, docs, tests, or history evidence that the wrapper-path Read stall or Telegram diagnostic gap has been fixed.

## What This Changes

The issue reports a Telegram-to-claude-cli regression where a native Read of an out-of-workspace wrapper path stalls for about three minutes and Telegram only receives a generic failure message.

## Best Possible Solution

Keep this open until maintainers define the supported workflow and close the diagnostic gap: claude-cli native-tool stalls should fail fast with sanitized actionable diagnostics, Telegram should surface a bounded timeout or tool-failure message instead of only generic copy, and wrapper execution should use the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
