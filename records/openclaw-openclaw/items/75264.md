## Summary

Keep open. The issue is recent, detailed, and still matches current main: the Claude CLI backend runs as a live stdio CLI backend with bundled MCP, native file tools are excluded from the OpenClaw MCP loopback, the no-output watchdog still matches the reported roughly three-minute stall shape, and Telegram still collapses dispatch failures to a generic fallback. I did not find source, docs, tests, or related-item evidence that current main fixes the wrapper-path Read stall or the Telegram diagnostic gap.

## What This Changes

The issue asks OpenClaw to make Telegram-routed claude-cli turns that encounter a wrapper-path file Read either execute through the approved wrapper path or fail quickly with an actionable diagnostic instead of timing out and returning a generic Telegram error.

## Best Possible Solution

Keep this open until maintainers define the supported route and close the diagnostic gap: claude-cli native file-tool stalls should fail fast with sanitized actionable diagnostics, Telegram should surface a bounded timeout/tool message instead of only a generic fallback, and wrapper execution should be routed or documented through the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
