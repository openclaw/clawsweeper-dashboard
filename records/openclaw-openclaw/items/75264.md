## Summary

Keep open. The report is recent, detailed, and still lines up with current main: the bundled Claude CLI backend uses a live stdio path with bundled MCP and native tools, OpenClaw’s MCP loopback deliberately excludes native file/exec tools, the no-output watchdog can terminate a silent turn on the same minute-scale timing, and Telegram still turns dispatch failures into a generic fallback. I did not find source, docs, tests, changelog, or related-item evidence that the wrapper-path native Read stall or Telegram diagnostic gap has been fixed.

## What This Changes

The issue requests that Telegram turns through claude-cli either execute the approved wrapper or fail fast with a clear diagnostic instead of stalling on a native file Read and sending only a generic Telegram error.

## Best Possible Solution

Keep this open until maintainers define the supported workflow and close the diagnostic gap: claude-cli native file-tool stalls should fail fast with sanitized, actionable logging; Telegram should surface a bounded timeout/tool diagnostic instead of only a generic fallback; and wrapper execution should be routed or documented through the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
