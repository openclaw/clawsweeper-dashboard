## Summary

Keep open. The report is recent, detailed, and current main still matches the failure shape: claude-cli runs with native tools enabled, OpenClaw’s MCP loopback excludes native file/exec tools, minute-scale no-output watchdogs are still present, and external chat errors still collapse to generic Telegram-visible text unless verbose diagnostics are enabled. The related closed #72338 is adjacent CPU/stall evidence, but it does not prove this claude-cli native Read and Telegram diagnostic gap is fixed.

## What This Changes

The issue asks OpenClaw to avoid minute-scale claude-cli Telegram stalls and generic Telegram errors when a turn hits a native Read of an approved wrapper path, either by executing the approved workflow or failing fast with a clear sanitized diagnostic.

## Best Possible Solution

Define the supported boundary, then either make claude-cli native file-tool stalls fail fast with sanitized diagnostics and useful Telegram surfacing, or route Telegram-to-wrapper execution through the intended ACP/node-exec path and document why raw CLI backends are not supported for that workflow.
