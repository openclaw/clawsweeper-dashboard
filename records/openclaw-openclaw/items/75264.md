## Summary

Keep open. Current main fixes the generic CLI timeout copy, but the central Claude CLI native Read/wrapper boundary is still unresolved and needs maintainer direction before a safe repair can be scoped.

## What This Changes

The issue asks OpenClaw to avoid Telegram-facing stalls and generic errors when claude-cli hits a native Read of an approved wrapper path, either by executing the wrapper or failing fast with clear policy guidance.

## Best Possible Solution

Define the supported route for Telegram-triggered wrapper execution, then either route it through ACP/node exec where OpenClaw owns policy and diagnostics or make claude-cli native file-tool paths fail fast with sanitized, actionable guidance.

## Reproduction Assessment

No high-confidence full reproduction was established. The issue has detailed live evidence and current source still shows the native Read path outside OpenClaw's loopback policy, but this review did not recreate the Telegram, Claude CLI OAuth, wrapper, and node-approval setup on current main.

## Solution Assessment

Unclear. The merged timeout-copy fix is the right narrow fix for the generic timeout symptom, but the remaining wrapper/Read behavior needs a maintainer decision between fail-fast diagnostics, ACP/node exec routing, or broader claude-cli native-tool support.
