## Summary

Keep open. The report is recent, detailed, and still maps to current main: Claude CLI runs through the live stdio backend with bundled MCP enabled, native Read/exec-style tools are excluded from the OpenClaw loopback tool surface, live Claude turns can still end on a minute-scale no-output watchdog, and Telegram still has generic dispatch/run-failure fallbacks unless verbose/control-UI detail paths apply. The related #72338 CPU-spin item was closed for broader hot-path work, but it does not prove this Claude-native Read stall or the Telegram diagnostic gap is fixed.

## What This Changes

The issue asks OpenClaw to stop stalling and returning only a generic Telegram error when a claude-cli Telegram turn hits a native Read against an approved wrapper path, either by completing the intended wrapper workflow or failing fast with a clear sanitized tool/permission diagnostic.

## Best Possible Solution

Keep this open until maintainers define the supported Telegram-to-wrapper workflow and close the observable diagnostic gap: Claude CLI native Read stalls should either produce a bounded, sanitized failure with useful logs or the docs/runtime should route this workflow through ACP or node exec explicitly, without weakening file-read or host-exec approval boundaries.
