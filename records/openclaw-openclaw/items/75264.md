## Summary

Keep open. The report is recent, detailed, and still matches current main: the Claude CLI backend uses bundled MCP plus native Claude tools, native Read is outside the OpenClaw loopback tool path, the live-session watchdog can produce the reported minute-scale stall, and Telegram still collapses dispatcher failures to a generic fallback. The closed #72338 report is adjacent CPU/stall evidence, but it does not cover this out-of-workspace native Read path or the Telegram diagnostic gap.

## What This Changes

The issue asks OpenClaw to avoid a long Claude CLI-backed Telegram stall on an approved wrapper path by either executing the approved wrapper and returning stdout or failing fast with a clear, sanitized tool error.

## Best Possible Solution

Define the supported product boundary, then implement that boundary consistently: either harden claude-cli native tool stalls with fast sanitized diagnostics and Telegram-safe surfacing, or document and route Telegram wrapper execution through the intended ACP/node-host path while making unsupported CLI-backend attempts fail clearly.
