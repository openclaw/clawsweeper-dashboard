## Summary

Keep open; the PR targets a real transcript-attribution bug, but it does not stop the runtime placeholder from being persisted as a normal user message and it adds an unsafe literal-text sanitizer. Real behavior proof is also missing, so this is not merge-ready.

## What This Changes

The PR exports the runtime-event placeholder constant, installs hidden runtime-context custom messages for runtime-only prompts, and filters the placeholder from `sanitizeUserFacingText`.

## Best Possible Solution

Model runtime-only continuation as non-user transcript provenance or suppress the generated user append at persistence time, while preserving model-visible runtime context and proving UI/export/memory consumers no longer show the placeholder as user-authored text.

## Reproduction Assessment

Yes at source level: current main maps an empty runtime transcript prompt to the placeholder and submits it through `AgentSession.prompt()`, which constructs a normal user-role message. No live WebGUI/TUI compaction reproduction was run in this read-only review.

## Solution Assessment

No: adding a hidden runtime-context custom message is useful for model context, but it does not stop the placeholder prompt from being persisted as a user message. A safer fix belongs at the transcript persistence/provenance boundary, not in a global literal text filter.
