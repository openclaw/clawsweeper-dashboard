## Summary

Keep open. The report is source-reproducible on current main: Slack marks dispatch idle but never marks the run complete, while the shared typing controller needs both signals and Discord/main buffered paths already send both; the fix is narrow enough for a repair PR.

## What This Changes

Slack inbound replies should mark the typing run complete before marking dispatch idle so Slack typing/status cleanup is not left to the TTL fallback.

## Best Possible Solution

Update Slack dispatch to call markRunComplete before markDispatchIdle in both settlement paths, and add a focused Slack regression test without changing the shared typing contract or adding config.

## Reproduction Assessment

Yes, at source level: current Slack dispatch calls markDispatchIdle without markRunComplete, while the shared typing controller requires both signals and Discord already follows that order. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is Slack parity with the existing Discord and buffered-dispatch lifecycle: signal markRunComplete before markDispatchIdle and pin it with a Slack test.
