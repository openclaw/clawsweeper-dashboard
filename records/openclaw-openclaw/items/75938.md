## Summary

Keep open. Current main still has Discord thread detection, parent-channel config inheritance, and bot-owned auto-thread mention bypasses, but it does not support a reusable parent-channel `thread.requireMention` override for ordinary Discord threads.

## What This Changes

Request a Discord channel config option that lets thread messages use a different `requireMention` value than their parent channel.

## Best Possible Solution

Add a backward-compatible Discord plugin config surface that evaluates `guilds.<guild>.channels.<parent>.thread.requireMention` for thread messages after parent detection and before mention filtering.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with parent `requireMention: true`, an unmentioned thread message inherits that parent config and is dropped; the proposed nested `thread.requireMention: false` key is not accepted by the strict schema.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Discord plugin config/schema and preflight mention-gating path, preserving existing parent inheritance, per-thread-ID overrides, and bot-owned auto-thread behavior.

## Repair Work Prompt

Implement thread-specific Discord `requireMention` configuration for issue https://github.com/openclaw/openclaw/issues/75938. Add a backward-compatible config shape such as `guilds.<guildId>.channels.<parentChannelId>.thread.requireMention` in the Discord config types/schema, and apply it when handling Discord thread messages that resolve to a parent channel config. The decision must happen in preflight before the reply/typing pipeline starts. Preserve existing behavior for guild-level defaults, channel-level `requireMention`, concrete per-thread channel IDs, sender allowlists, and bot-owned auto-thread mention bypasses. Add focused tests for schema acceptance and runtime behavior where a parent channel requires mention but thread messages do not, plus coverage that existing parent/channel inheritance still works without the new override. Update Discord docs and CHANGELOG.md for the user-facing config behavior. Do not broaden this into the larger access-list redesign tracked by #69748.
