## Summary

Keep open. Current main still lacks the requested operator-configurable cron outbound payload policy; only channel-owned final-text collapse and broader delivery-mode suppression exist, while the linked implementation PR is closed unmerged.

## What This Changes

The issue requests a configurable isolated-cron outbound delivery policy with `full`, `last`, and `none` modes for mapping resolved payloads to channel sends.

## Best Possible Solution

Add a typed cron outbound payload policy with default `full`, apply `last` after existing payload resolution, and make `none` suppress runner outbound delivery without synthesized-text fallback.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current resolver/tests preserve multiple successful text payloads for non-final-preference channels, and dispatch still sends those payloads or synthesized fallback text when runner delivery is requested.

## Solution Assessment

Yes. The requested explicit payload-mapping policy is the narrowest maintainable direction; existing `delivery.mode=none` / `--no-deliver` is a broader runner fallback control and does not provide full/last/none mapping after payload resolution.
