## Summary

Keep open. Current main still lacks the requested operator-configurable cron payload-mapping policy: channel-owned final-text collapse and `delivery.mode=none` cover some cases, but there is no full/last/none policy after payload resolution and the linked implementation PR is closed unmerged.

## What This Changes

Request a cron outbound delivery policy that can send all resolved deliverable segments, only the last segment, or no channel message while preserving run output.

## Best Possible Solution

Add a typed cron outbound payload policy with default `full`, apply `last` after existing payload resolution, and make `none` suppress outbound runner delivery without falling back to synthesized text.

## Reproduction Assessment

Yes, at source level: current main preserves multiple successful payloads for non-final-preference channels and has no accepted `cron.deliveryPayloads` key. I did not run a live channel cron job in this read-only review.

## Solution Assessment

Yes, the requested explicit payload-mapping policy is the narrowest maintainable direction; existing `delivery.mode=none` / `--no-deliver` is broader runner suppression and does not provide last-only mapping after payload resolution.
