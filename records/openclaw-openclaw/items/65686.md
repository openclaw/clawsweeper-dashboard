## Summary

Keep open. Current main still does not provide an operator-configurable cron payload mapping policy; it preserves multi-payload output by default, only collapses for channel-owned final-text policy, and the linked implementation PR is closed unmerged.

## What This Changes

The issue requests a configurable cron outbound delivery policy so isolated cron jobs can send all resolved payloads, only the last resolved payload, or no channel message.

## Best Possible Solution

Add a typed cron payload policy with default `full`, apply `last` after existing payload resolution, and make `none` suppress runner channel delivery without falling back to synthesized text, with schema, docs, and regression tests updated together.

## Reproduction Assessment

Yes, at source level. Current main preserves multiple successful payloads for non-final-preference channels, the config/schema do not accept a payload policy, and the dispatch path can synthesize a send from empty payloads.

## Solution Assessment

Yes. An explicit post-resolution payload-mapping policy is the narrow maintainable direction; existing `delivery.mode=none` / `--no-deliver` suppresses runner fallback broadly but does not provide last-only mapping after payload resolution.
