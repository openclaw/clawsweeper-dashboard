## Summary

Keep open. Current main still lacks an operator-selectable cron payload mapping policy; it preserves multi-payload output for non-final-preference channels, only has channel-owned final-text collapse, and the exact implementation PR linked from the issue is closed unmerged.

## What This Changes

The issue requests an operator policy for isolated cron outbound delivery that can keep all resolved payloads, keep only the last payload, or suppress runner channel sends entirely.

## Best Possible Solution

Add a typed, documented cron payload policy with default `full`, apply `last` after existing payload resolution, and make `none` suppress runner-owned channel delivery without falling back to synthesized text, with schema, docs, generated config metadata, and focused regression tests updated together.

## Reproduction Assessment

Yes, at source level. Current main has no accepted config key for the policy, preserves multiple successful payloads for non-final-preference channels, and has tests that assert multi-send behavior.

## Solution Assessment

Yes, directionally, but maintainers need to approve the public config shape. A post-resolution payload policy is narrower than changing payload selection semantics, while `delivery.mode="none"` only controls runner fallback delivery and does not provide last-only mapping.
