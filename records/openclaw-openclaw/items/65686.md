## Summary

Keep open. Current main still lacks an operator-selectable cron payload mapping policy for full/last/none delivery; it preserves all successful deliverable payloads except for channel-owned final-text collapse, and the exact implementation PR is closed unmerged.

## What This Changes

The issue requests configurable cron outbound payload mapping so isolated cron runs can send all resolved segments, only the last segment, or no runner-owned channel message.

## Best Possible Solution

Add a maintainer-approved typed cron payload policy that defaults to full, applies last after existing payload resolution, preserves structured/media delivery rules, and keeps existing delivery.mode none semantics intact with schema, docs, metadata, and regression tests updated together.

## Reproduction Assessment

Yes, at source level. Current main lacks an accepted config key, preserves multiple successful payloads for non-final-preference channels, and has tests asserting that multi-send behavior.

## Solution Assessment

Unclear as proposed. A post-resolution payload policy is plausible, but maintainers should decide whether this belongs as a global cron default, a per-job delivery option, or documentation around existing delivery.mode none before implementation.
