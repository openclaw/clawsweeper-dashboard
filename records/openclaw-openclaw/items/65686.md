## Summary

Keep open. Current main still has no operator-selectable cron payload mapping policy: it preserves all successful deliverable payloads unless a channel-owned final-text preference applies, and the exact full/last/none implementation PR is closed unmerged.

## What This Changes

Request a configurable cron outbound delivery policy that keeps the default full payload list while allowing last-only or no runner-owned channel send.

## Best Possible Solution

Add a typed, documented cron payload policy with default full, apply last after existing payload resolution, and make none suppress runner-owned channel delivery without synthesized fallback, with schema, docs, generated config metadata, and focused regression tests updated together.

## Reproduction Assessment

Yes, at source level. Current main has no accepted config key for the policy, preserves multiple successful payloads for non-final-preference channels, and has tests that assert multi-send behavior.

## Solution Assessment

Yes, directionally, but maintainers need to approve the public config shape. A post-resolution payload policy is narrower than changing deliverable selection semantics, while delivery.mode none is only a separate no-delivery mode and does not provide last-only mapping.
