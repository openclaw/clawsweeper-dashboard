## Summary

Keep open. Current main and the latest tagged release still lack the requested operator-configurable cron outbound payload policy; only channel-owned final-text collapse exists, and the linked implementation PR is closed unmerged with no open replacement found.

## What This Changes

The issue requests a configurable isolated-cron outbound delivery policy that preserves today's full payload behavior by default, can send only the last resolved payload, or can suppress runner channel sends without synthetic-text fallback.

## Best Possible Solution

Add a typed cron outbound payload policy with default `full`, apply `last` after existing payload resolution, and make `none` suppress runner outbound delivery without falling back to synthesized text. Keep channel plugin final-text preferences compatible, document the new config, and cover resolver/dispatch/schema behavior with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current resolver/tests preserve multiple successful text payloads for non-final-preference channels, and dispatch still sends those payloads or synthesized fallback text when runner delivery is requested.

## Solution Assessment

Yes. The narrow maintainable fix is to add an explicit cron payload-mapping policy in the existing resolver/dispatch flow; existing `--no-deliver` / `delivery.mode=none` controls runner fallback delivery broadly and does not provide the requested full/last/none mapping after payload resolution.
