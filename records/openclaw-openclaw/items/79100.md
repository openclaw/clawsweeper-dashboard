Keep open: current main still has the reported CLI option mismatch, and the only focused fix found was closed unmerged, so there is no implemented or superseding main-branch fix to close against.

## What This Changes

Request `gateway health` and `gateway probe` to target a non-default local gateway port through the same visible `--port` shape accepted by `gateway run`, or clearly document the supported targeting path.

## Best Possible Solution

Choose the gateway query targeting contract, then either add a config-preserving `--port` path for health/probe with regression coverage or tighten CLI help/docs around `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port`.

## Reproduction Assessment

Yes, at source level: current main declares `--port` for gateway run but not for health/probe, and the query RPC option resolver does not inherit `port`. I did not run the Windows packaged install path locally.

## Solution Assessment

Unclear: the closed related PR shows a plausible narrow config-preserving implementation, but maintainers should first decide whether health/probe should grow a `--port` alias or continue to steer users to `--url` and config/env targeting.
