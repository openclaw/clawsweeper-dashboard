## Summary

Keep open: current main still has the reported option mismatch, and the related fix PR was closed without merging. Because gateway query commands already have a documented `--url`/config/env targeting path, adding `--port` is a CLI UX/API decision rather than a conservative auto-close.

## What This Changes

Request that `gateway health` and `gateway probe` either accept a non-default port option consistent with `gateway run` or clearly document the supported targeting path.

## Best Possible Solution

Decide the gateway query targeting contract, then either land a config-preserving `--port` alias with regression coverage or tighten help/docs around `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port`.

## Reproduction Assessment

Yes, source-level: current main exposes `--port` on the run path but neither declares nor inherits it for health/probe. The reporter also supplied released-version Windows CLI output for 2026.5.7 showing the same unknown-option error.

## Solution Assessment

Unclear: the prior closed PR shows a plausible config-preserving alias, but the existing query-command contract already supports `--url` and config/env targeting. A maintainer should choose whether the intended UX is a new alias or clearer documentation/help.
