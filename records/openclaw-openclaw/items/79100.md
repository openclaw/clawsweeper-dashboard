## Summary

Keep open: current main still has the reported gateway CLI mismatch, and the only linked implementation PR was closed unmerged. The remaining choice is a maintainer CLI contract decision: add a config-preserving `--port` path for health/probe or make the existing `--url` and config/env targeting path clearer.

## What This Changes

The issue asks for `gateway health` and `gateway probe` to target a non-default local gateway port consistently with `gateway run --port`, or to document the supported targeting path more clearly.

## Best Possible Solution

Settle the gateway query targeting contract, then either land a config-preserving `--port` alias with regression coverage or tighten CLI help/docs around `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port`.

## Reproduction Assessment

Yes, source-level: current main exposes `--port` for gateway run but health/probe neither declare it nor inherit it into RPC/status options. The reporter also supplied released-version Windows output showing the same unknown-option error.

## Solution Assessment

Unclear: adding a config-backed `--port` alias is plausible and has a prior implementation attempt, but the existing documented query contract already supports `--url` and config/env targeting, so maintainers should choose the intended CLI UX.
