## Summary

Keep open: current main still has the gateway port-targeting mismatch, and the only focused fix path I found is a closed, unmerged pull request rather than shipped code. The remaining decision is CLI UX/product scope, because current docs expose `--url`/config/env for query commands while the broader gateway surface also exposes `--port`.

## What This Changes

The issue requests consistent non-default gateway port targeting for `gateway health` and `gateway probe`, either by accepting `--port` like `gateway run` or by documenting the supported alternative.

## Best Possible Solution

Choose the query-command targeting contract, then either make `--port` feed health/probe through config-backed local targeting with TLS/auth preserved, or clearly document that `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port` are the supported paths.

## Reproduction Assessment

Yes, at source level. Current main registers `--port` on the gateway/run surface, but health/probe do not declare it and the RPC option resolver does not inherit `port`; the reporter also reproduced the rejection on the v2026.5.7 release.

## Solution Assessment

Unclear until maintainers choose the CLI contract. The closed related pull request is a plausible narrow implementation, but the best user-facing shape needs an explicit decision before a replacement fix is queued automatically.
