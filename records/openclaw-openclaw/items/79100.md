## Summary

Keep open: current main and the latest release still lack a `--port` path for `gateway health` and `gateway probe`, while closed unmerged PRs attempted but did not land the fix.

## What This Changes

The issue asks for `gateway health --port` and `gateway probe --port` to target a non-default local gateway port consistently with `gateway run`, or for docs to clarify the supported alternative.

## Best Possible Solution

Add a focused CLI fix that accepts and forwards a strict config-backed `--port` for health/probe without converting it to `--url`, preserving auth, TLS, and env/config precedence.

## Reproduction Assessment

Yes, at source level: current main does not register `--port` on `gateway health` or `gateway probe`, and Commander 14 errors on unregistered options by default. I did not run the CLI because this review was constrained to read-only checkout inspection.

## Solution Assessment

Yes. The narrow maintainable fix is to add/forward a strict `--port` option through the existing gateway CLI option-resolution path and patch `gateway.port` before target resolution, rather than using a synthetic `--url` that bypasses configured credentials.
