## Summary

Keep open: current main still has the reported CLI option mismatch, and the only linked implementation is a closed unmerged PR. The remaining decision is whether gateway query commands should gain a visible `--port` path or whether help/docs should steer users to the existing `--url`/config/env targeting contract.

## What This Changes

The issue asks for `gateway health` and `gateway probe` to target non-default local gateway ports consistently with `gateway run`, either by accepting `--port` or documenting the supported alternative clearly.

## Best Possible Solution

Choose the query-command targeting contract, then either land a narrow config-backed `--port` override like the closed candidate PR or update CLI help/docs so `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port` are unmistakable.

## Reproduction Assessment

Yes, at source level: current main registers `--port` for gateway run, but health/probe neither declare nor inherit a port option. The reporter also supplied released-version CLI output for 2026.5.6 and 2026.5.7 showing `unknown option '--port'`.

## Solution Assessment

Unclear: a config-backed `--port` alias is a plausible and narrow solution, and https://github.com/openclaw/openclaw/pull/79113 sketched it well. Maintainers still need to choose whether query commands should support that visible alias or stay documented around the existing `--url`/config/env path.
