## Summary

Keep open: current main still has the reported option mismatch, but the existing query-command contract is documented around `--url`/config/env rather than `--port`, so this is a CLI UX/product decision rather than a clear existing-contract bug.

## What This Changes

Requests that `gateway health` and `gateway probe` either accept `--port` like `gateway run` or make the non-default-port targeting path clearer.

## Best Possible Solution

Decide the query-command targeting contract and either add a documented local `--port` alias for `gateway health`/`gateway probe` with regression tests, or tighten CLI help/docs so the `--url`/config/env path is unmistakable.

## Reproduction Assessment

Yes, at source level: current main declares `--port` for `gateway run` but not for the `health` or `probe` child commands, so those child command forms have no `--port` option. I did not live-run the Windows npm install path, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear: accepting `--port` is a plausible convenience fix, but the current query-command contract already supports `--url`, `OPENCLAW_GATEWAY_PORT`, and `gateway.port`. A maintainer should choose whether the best solution is a new local alias or clearer help/docs around the existing path.
