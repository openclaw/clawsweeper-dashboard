## Summary

Keep open. Current main still has the deleted-file ENOENT gap, and this PR is a focused candidate fix, but it needs real disappearing-file backup proof before merge because the provided terminal output only shows a normal backup plus mocked retry tests.

## What This Changes

The PR adds ENOENT-with-path detection to the backup tar retry wrapper, retries those errors with the existing backoff/temp cleanup path, and adds unit coverage for the classifier and retry behavior.

## Best Possible Solution

Land a narrow backup archiver fix that treats expected source-file ENOENT during tar creation as non-fatal enough to complete scheduled backups, with real after-fix proof of the disappearing-file path.

## Reproduction Assessment

Yes for source-level reproduction: current main calls tar.c over live state roots, tar 7.5.16 emits lstat ENOENT before filtering when a source disappears, and linked reports show the failure on released versions. I did not run a live current-main race in this read-only review.

## Solution Assessment

Unclear. Extending the existing retry wrapper is the narrowest maintainable patch, but without real race proof it is not proven to be the best solution for repeated live churn.
