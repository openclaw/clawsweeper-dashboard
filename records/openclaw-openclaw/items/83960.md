## Summary

This should stay open: xAI OAuth browser login is a shipped and documented provider-auth path, current main still sends the same OAuth scope string reported against v2026.5.18, and the only new current-main change is the related device-code flow for remote hosts, not proof that `--method oauth` now works.

## What This Changes

The issue reports that `openclaw models auth login --provider xai --method oauth` fails on Docker/v2026.5.18 with an upstream `Unknown scope` error before credentials are stored.

## Best Possible Solution

Live-test the browser OAuth flow with an eligible xAI account, then either fix `extensions/xai/xai-oauth.ts` scope/client parameters with regression coverage or document device-code as the remote-host path while preserving working browser OAuth.

## Reproduction Assessment

No. The source path and scope string are clear, and xAI discovery currently advertises the requested scopes, but the failing browser/account-dependent OAuth step was not reproduced against current main.

## Solution Assessment

Unclear. The likely fix belongs in the xAI plugin OAuth implementation, but changing scopes or client parameters should wait for live xAI OAuth proof because current upstream discovery does not identify an unsupported scope.
