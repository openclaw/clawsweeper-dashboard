Keep open: the hardening direction is plausible and the PR has supplied real runtime proof, but the diff still has two blocking P2 issues around Kubernetes namespace handling and Trivy suppression scope.

## What This Changes

The PR adds Gitleaks/Trivy certification policy, Docker/Kubernetes hardening, SBOM ownership metadata, a Hono override update, and focused regression-test adjustments.

## Best Possible Solution

Keep the certification hardening, but make the Kubernetes manifest namespace-neutral and scope the Trivy acceptance to the single reviewed manifest before maintainer security review.

## Reproduction Assessment

Yes for the review findings from source inspection: a custom `OPENCLAW_NAMESPACE` conflicts with the patched Deployment namespace, and the plain Trivy ignore suppresses the rule repo-wide. I did not run a live Kubernetes cluster or Trivy scan.

## Solution Assessment

No. The maintainable path is to preserve the hardening but remove the hard-coded namespace and replace the broad Trivy ignore with an inline or path-scoped exception.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep the Deployment namespace configurable:** `scripts/k8s/manifests/deployment.yaml:5`
  - body: When users set `OPENCLAW_NAMESPACE` to anything other than `openclaw`, `deploy.sh` still applies the kustomization with `-n "$NS"`, but this hard-coded namespace makes the Deployment target `openclaw` and can fail or split a custom-namespace install. Omit this field and let the deploy script select the namespace.
  - confidence: 0.93
- **[P2] Scope the Trivy suppression to the accepted file:** `.trivyignore:4`
  - body: This bare `KSV-0125` entry suppresses that finding for every Trivy target even though the comment accepts it only for the Kubernetes Deployment. Use an inline ignore on the accepted resource or a path-scoped YAML ignore so future findings are still reported.
  - confidence: 0.9
