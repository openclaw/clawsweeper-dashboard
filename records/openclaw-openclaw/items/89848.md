## Summary

Keep open: current `main` and the latest release still lack a `failureAlert.threadId` route, and source inspection shows cron failure alerts carry `channel`, `to`, `mode`, and `accountId` but not a thread/topic id. The related open issue https://github.com/openclaw/openclaw/issues/89847 covers the runtime-stall silent-drop path; this issue still owns the separate Telegram topic routing gap.

## What This Changes

The issue requests `failureAlert.threadId` support for cron failure alerts, optional inheritance from `delivery`, and docs for the failure-alert schema.

## Best Possible Solution

Make failure-alert announce routing use the same typed thread/topic contract as cron delivery, with persistence, schema, CLI, docs, and regression tests; leave optional sugar as a follow-up unless approved.

## Reproduction Assessment

Yes from source inspection: current main's failure-alert type, schema, persistence codec, resolver, and gateway send target lack a thread/topic field while primary cron delivery has one. I did not run a live Telegram cron, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes for the primary ask: the narrow maintainable fix is additive `failureAlert.threadId` support through cron config, storage, delivery plumbing, CLI/docs, and tests. The optional inheritance flag should be deferred unless maintainers want that exact config API.
