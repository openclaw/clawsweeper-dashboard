## Summary

Keep open: current main and latest release still do not carry a thread/topic id through cron `failureAlert`, and the open draft PR that closes this issue has not merged yet. The newer `failureDestination` surface overlaps with the problem but also lacks `threadId`, so it is not a complete replacement for the reported Telegram topic-routing gap.

## What This Changes

The issue asks for cron `failureAlert` to support Telegram topic routing via `threadId`, optionally inherit the delivery thread, and document the failure-alert shape.

## Best Possible Solution

Land or replace the linked implementation with a narrow additive fix that carries a failure-alert thread id through schema, storage, CLI/tool/docs, and gateway send plumbing without adding optional sugar as a competing config source of truth.

## Reproduction Assessment

Yes, source-reproducible: current main's `CronFailureAlert` type/schema and gateway send target do not carry `threadId`, while primary cron delivery does. I did not run a live Telegram cron in this read-only review.

## Solution Assessment

Yes for the primary direction: additive `failureAlert.threadId` routing through the existing cron delivery plumbing is the maintainable fix. The optional inheritance sugar should remain secondary unless maintainers approve that extra config API.
