## Summary

Keep open: current main and the latest release still lack a `failureAlert.threadId` contract, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests `failureAlert.threadId` and optional delivery-thread inheritance so cron failure alerts can route to the same Telegram topic as successful cron delivery announcements.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90887 or a narrow replacement that carries failure-alert thread ids through schema, storage, CLI/tool input, docs, and gateway delivery.

## Reproduction Assessment

Yes, source-level: current main exposes `delivery.threadId` but no equivalent `failureAlert.threadId` across the type, schema, resolver, storage projection, gateway send path, and docs. I did not run a live Telegram cron.

## Solution Assessment

Yes for the primary direction: mirroring the existing delivery thread contract onto failure alerts is the narrow maintainable solution. The inheritance shortcut is additional public config/API surface and should remain secondary unless maintainers approve it.
