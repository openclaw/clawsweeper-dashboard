## Summary

Keep open: current main and the latest release still do not carry a thread/topic id through cron `failureAlert`, and the open closing PR is the active implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests `failureAlert.threadId`, optional delivery-thread inheritance, and cron docs/schema coverage so Telegram cron failure alerts can route to the same topic as successful delivery messages.

## Best Possible Solution

Review and land a narrow additive fix that carries `failureAlert.threadId` through cron config/protocol, storage, CLI/tool input, docs, and gateway delivery; treat optional inheritance sugar as secondary unless maintainers explicitly want that extra API.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` accept `delivery.threadId` but do not accept, resolve, persist, document, or send a `failureAlert.threadId`; I did not run a live Telegram cron.

## Solution Assessment

Yes for the primary direction: mirroring the existing delivery thread contract onto failure alerts is the narrow maintainable fix. The optional inheritance shortcut should not be required unless maintainers approve the extra config/API surface.
