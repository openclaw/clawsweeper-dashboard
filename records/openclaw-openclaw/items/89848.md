## Summary

Keep open: current main and the latest release still lack a `failureAlert.threadId` contract and the open closing PR is still the active implementation path rather than a merged fix.

## What This Changes

The issue asks for cron failure alerts to support Telegram topic routing via `failureAlert.threadId`, optional inheritance from `delivery.threadId`, and docs for the failure-alert envelope.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90887 or a replacement that carries failure-alert thread IDs through schema, storage, CLI/tool input, docs, and gateway delivery.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.9` accept `delivery.threadId` but do not accept, resolve, persist, document, or send `failureAlert.threadId`. I did not run a live Telegram cron.

## Solution Assessment

Yes for the primary direction: mirroring the existing delivery thread contract onto failure alerts is the narrow maintainable solution. The inheritance shortcut is additional public config/API surface and should stay secondary unless maintainers approve it.
