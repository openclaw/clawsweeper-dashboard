## Summary

Keep open. Current main still short-circuits live-PID runtime-deps locks before the stale-mtime fallback, while this PR is a narrow, mergeable implementation candidate with focused tests, changelog coverage, successful exact-head check suites, and no blocking review findings.

## What This Changes

The branch adds mtime-based expiry for PID-alive runtime-deps locks missing both `starttime` and `createdAtMs`, plus focused regression tests and an Unreleased changelog entry.

## Best Possible Solution

Land this narrow fallback with its regression tests after maintainer approval, then close or link the paired #74948 report. Keep any broader createdAt-bearing same-PID recovery as a separate runtime-deps lock policy decision.

## Reproduction Assessment

Yes. A source-level reproduction on current main is a call like `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: 1000 }, 601001, () => true)`, which returns `false` because the live-PID branch exits before the mtime fallback.

## Solution Assessment

Yes for the true-legacy case. The PR uses existing mtime evidence only when modern owner fields are absent, preserving active start-time and createdAt-bearing locks while repairing the narrower stale-lock path.
