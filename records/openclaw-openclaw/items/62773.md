## Summary

Keep this PR open: current main still lacks a first-class filesystem directory listing tool, and the linked feature issue was explicitly closed in favor of this PR. The patch is not mergeable as-is because it adds a new filesystem enumeration surface without preserving current sandbox/guard semantics, has deterministic output and metadata contract bugs, and lacks contributor real behavior proof.

## What This Changes

This PR adds a new `list` filesystem tool under `group:fs`, wires required params and catalog registration, and returns structured directory entries with name, type, size, and mtime.

## Best Possible Solution

Land one guarded, policy-cataloged filesystem discovery implementation that preserves sandbox bridge boundaries, resolves `list` versus `ls` naming and symlink metadata semantics, updates docs/tests, and then close superseded attempts.

## Reproduction Assessment

Yes for the PR blockers. Source inspection of the PR diff against current main shows the host filesystem registration in sandboxed runs, child `fs.stat` symlink metadata exposure, truncation-before-sort behavior, and stale package import.

## Solution Assessment

No. A first-class listing tool is a reasonable feature direction, but this patch is not the best implementation until it uses sandbox-aware operations, safe metadata semantics, deterministic ordering, current package imports, docs/tests, and real-run proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Route sandbox listings through the sandbox bridge:** `src/agents/pi-tools.ts:549-552`
  - body: In sandboxed sessions, this registers `list` as a host filesystem tool instead of routing directory reads through `SandboxFsBridge` like the existing filesystem tools. With `tools.fs.workspaceOnly` disabled, an agent could enumerate arbitrary host absolute paths outside sandbox mounts.
  - confidence: 0.91
- **[P1] Use the current PI package name:** `src/agents/pi-tools.list.ts:3`
  - body: The new file imports `AgentTool` from the old `@mariozechner/pi-agent-core` package, but current main depends on `@earendil-works/pi-agent-core`; rebasing this branch as-is leaves the import unresolved.
  - confidence: 0.86
- **[P2] Stop following symlink targets for entry metadata:** `src/agents/pi-tools.list.ts:116-118`
  - body: The parent directory is boundary-checked, but each child is inspected with `fs.stat(entryPath)`, which follows symlinks. A workspace symlink to an out-of-workspace target can expose target type, size, and mtime; use no-follow metadata or revalidate each listed child.
  - confidence: 0.88
- **[P2] Sort before applying the entry cap:** `src/agents/pi-tools.list.ts:105-107`
  - body: The loop stops after 1000 entries in raw filesystem order and sorts only that subset. Large directories therefore return an arbitrary page while reporting `truncated: true`; sort the complete dirent list first, then slice.
  - confidence: 0.86
- **[P2] Return null when mtime is unavailable:** `src/agents/pi-tools.list.ts:113-123`
  - body: `ListEntry.mtime` is documented as an ISO-8601 timestamp, but unreadable or broken-symlink entries return an empty string. Make the field nullable and return `null` so callers can distinguish missing metadata from a real timestamp.
  - confidence: 0.82
