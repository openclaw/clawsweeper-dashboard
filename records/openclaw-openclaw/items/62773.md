## Summary

Keep open. Current main still lacks the requested core directory-listing tool, but this PR is not merge-ready because it bypasses the sandbox filesystem bridge, follows symlink targets for metadata, caps entries before sorting, imports the old PI package name, and has no real behavior proof.

## What This Changes

The PR adds a core `list` filesystem tool under `group:fs`, wires it into required params/catalog/tool construction, and returns structured directory entries with metadata.

## Best Possible Solution

Add the core directory-listing tool only with a sandbox bridge list contract, safe symlink metadata semantics, deterministic truncation, docs, and regression coverage for `group:fs`.

## Reproduction Assessment

Yes. Source inspection of the PR diff is enough to reproduce the blockers: the tool is registered as a host filesystem tool in sandboxed construction, and child metadata is gathered with `fs.stat` after only validating the parent directory.

## Solution Assessment

No. A first-class directory-listing tool is a reasonable direction, but this patch should not merge until it uses a bridge-backed sandbox contract, settles symlink metadata behavior, updates the current PI import, and adds focused docs/tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route sandbox listings through the sandbox bridge:** `src/agents/pi-tools.ts:546-550`
  - body: In sandboxed sessions this adds `list` as a host filesystem tool instead of routing through `SandboxFsBridge` like the existing filesystem tools. If `tools.fs.workspaceOnly` is false, an agent can call `list` on host paths such as `/etc` and enumerate outside the sandbox boundary.
  - confidence: 0.91
- **[P2] Stop following symlink targets for entry metadata:** `src/agents/pi-tools.list.ts:113-115`
  - body: The parent directory is boundary-checked, but each child is inspected with `fs.stat(entryPath)`, which follows symlinks. A workspace symlink to an out-of-workspace target can expose the target type, size, and mtime; use no-follow metadata or revalidate each resolved child.
  - confidence: 0.88
- **[P2] Sort before applying the entry cap:** `src/agents/pi-tools.list.ts:102-107`
  - body: The loop stops after 1000 entries in raw filesystem order and sorts only that subset. Large directories therefore return an arbitrary page while reporting `truncated: true`; sort the full dirent list first, then slice.
  - confidence: 0.86
- **[P2] Return null for missing mtimes:** `src/agents/pi-tools.list.ts:25`
  - body: `ListEntry.mtime` is documented as an ISO-8601 string, but broken or unreadable entries return `""`. Make the field nullable and return `null` when metadata is unavailable so callers do not have to special-case an invalid timestamp.
  - confidence: 0.82
- **[P2] Use the current PI package name:** `src/agents/pi-tools.list.ts:3`
  - body: The new file imports `AgentTool` from `@mariozechner/pi-agent-core`, but current main depends on `@earendil-works/pi-agent-core` and current agent code imports from that package. After rebasing onto main, this unresolved import will fail build/type checks.
  - confidence: 0.83
