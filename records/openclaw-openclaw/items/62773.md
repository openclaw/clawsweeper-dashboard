## Summary

Keep open: the feature is useful, but the patch is not mergeable because it bypasses the sandbox filesystem bridge model, exposes unsafe symlink target metadata, has deterministic output contract bugs, and lacks real behavior proof.

## What This Changes

The branch adds a core `list` filesystem tool, registers it in `group:fs`, validates a `path` argument, and returns structured directory entries with metadata.

## Best Possible Solution

Add the directory-listing tool only after defining a sandbox bridge list contract, safe symlink metadata semantics, docs, and regression coverage for `group:fs`.

## Reproduction Assessment

Yes. Source inspection of the PR diff is enough to reproduce the blockers: the tool is registered as a host filesystem tool in sandboxed construction, and child metadata is collected with `fs.stat` after only checking the parent directory.

## Solution Assessment

No. A first-class directory listing tool is a reasonable direction, but this implementation should route through an explicit sandbox bridge contract and define no-follow or revalidated symlink metadata before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route sandbox listings through the sandbox bridge:** `src/agents/pi-tools.ts:546-550`
  - body: In sandboxed sessions this registers `list` as a host filesystem tool instead of routing through `SandboxFsBridge` like the existing filesystem tools. With `tools.fs.workspaceOnly` unset or false, an agent could call `list` on host paths such as `/etc` and enumerate directories outside sandbox mounts.
  - confidence: 0.91
- **[P2] Avoid following symlink targets for entry metadata:** `src/agents/pi-tools.list.ts:113-115`
  - body: The parent directory is boundary-checked, but each child is inspected with `fs.stat(entryPath)`, which follows symlinks. A workspace symlink to an out-of-workspace file can expose the target's type, size, and mtime; use no-follow metadata or revalidate each resolved child before returning metadata.
  - confidence: 0.88
- **[P2] Sort before applying the entry cap:** `src/agents/pi-tools.list.ts:102-107`
  - body: Entries are capped while iterating raw `readdir` order and sorted only afterward. For directories with more than 1000 entries, callers receive an arbitrary subset rather than the first deterministic sorted page, so `truncated: true` is misleading.
  - confidence: 0.86
- **[P2] Return null for missing mtimes:** `src/agents/pi-tools.list.ts:25`
  - body: `ListEntry.mtime` is documented as an ISO-8601 string, but the stat-failure path returns an empty string. Make the field nullable and return `null` when metadata is unavailable so broken symlinks do not violate the structured contract.
  - confidence: 0.82
