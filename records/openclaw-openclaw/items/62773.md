## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks a first-class directory listing tool in `group:fs`, so the contribution addresses a real product gap, but the proposed implementation introduces sandbox/host filesystem boundary risks and several deterministic output/contract bugs.

## What This Changes

This PR adds a built-in `list` filesystem tool, registers it in the `group:fs` coding profile, adds required path-parameter validation, and wires the tool into `createOpenClawCodingTools`.

## Best Possible Solution

Add `list` as a built-in filesystem tool only after making it follow the same sandbox and workspace boundary model as `read`/`write`/`edit`, likely by extending `SandboxFsBridge` with a directory-list operation or a narrowly equivalent sandbox-owned helper. The final version should document the group change and cover policy, sandbox, symlink, truncation, missing-path, file-path, and broken-symlink behavior with focused tests.

## Reproduction Assessment

Yes. The sandbox boundary issue is reproducible from the diff: in a sandboxed run where `tools.fs.workspaceOnly` is unset or false, calling `list` with an absolute host path such as `/etc` would use host `fs.readdir` instead of `SandboxFsBridge`; the symlink metadata leak can be reproduced by listing a workspace directory containing a symlink to an outside file.

## Solution Assessment

No. A first-class directory listing tool is the right direction for the linked feature gap, but this implementation is not the narrowest maintainable fix because it bypasses the sandbox bridge and does not define safe symlink or deterministic truncation semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route sandbox listings through the sandbox bridge:** `src/agents/pi-tools.ts:546-550`
  - body: In sandboxed sessions this adds `list` as a host filesystem tool instead of routing it through `SandboxFsBridge` like the existing fs tools. When `tools.fs.workspaceOnly` is unset or false, a model can call `list` on host paths such as `/etc` and enumerate host directories outside the sandbox mounts. Route listing through a sandbox-aware bridge operation, or withhold the tool until that path exists.
  - confidence: 0.91
- **[P2] Avoid following symlinks for entry metadata:** `src/agents/pi-tools.list.ts:113-115`
  - body: The parent directory is boundary-checked, but each child is inspected with `fs.stat(entryPath)`, which follows symlinks. A workspace symlink to an out-of-workspace target can expose that target's size and mtime through `details.entries`. Use no-follow metadata or revalidate each resolved child target before including target metadata.
  - confidence: 0.87
- **[P2] Sort before applying the entry cap:** `src/agents/pi-tools.list.ts:102-107`
  - body: Entries are capped while iterating raw `readdir` order and only sorted afterward. For directories with more than 1000 entries, callers receive an arbitrary subset rather than the first sorted page, making `truncated: true` misleading. Sort the dirents first, then slice or collect the first `MAX_ENTRIES`.
  - confidence: 0.86
- **[P2] Return null for missing mtimes:** `src/agents/pi-tools.list.ts:110-119`
  - body: The type comment promises an ISO-8601 timestamp, but the stat-failure path returns an empty string for `mtime`. That makes broken-symlink entries violate the structured contract. Type `mtime` as nullable and return `null` when metadata is unavailable.
  - confidence: 0.82
