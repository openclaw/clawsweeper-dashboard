## Summary

Keep open. Current main still lacks a built-in filesystem `list` tool, but this PR is not mergeable as-is because it adds host-side directory enumeration in sandboxed runs and leaves unsafe symlink metadata semantics in a filesystem boundary-sensitive path.

## What This Changes

The PR adds a first-class `list` filesystem tool under `group:fs`, wires required path parameters, registers it in the core tool catalog, and adds it to `createOpenClawCodingTools`.

## Best Possible Solution

Add `list` only after a sandbox-bridge directory listing contract exists, child symlink behavior is explicitly safe, truncation is deterministic, and docs/changelog/tests cover the new public filesystem tool.

## Reproduction Assessment

Yes. The PR diff is enough to reproduce the blockers from source: sandboxed tool creation calls host `createListTool`, and the new implementation follows child symlinks with `fs.stat` after only validating the parent directory.

## Solution Assessment

No. A first-class directory listing tool is a reasonable direction, but the proposed implementation is not the best solution until it follows the same sandbox and workspace-boundary model as the existing filesystem tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route sandbox listings through the sandbox bridge:** `src/agents/pi-tools.ts:546-550`
  - body: In sandboxed sessions this registers `list` as a host filesystem tool instead of routing it through `SandboxFsBridge` like the existing filesystem tools. With `tools.fs.workspaceOnly` unset or false, an agent can call `list` on host paths such as `/etc` and enumerate directories outside sandbox mounts.
  - confidence: 0.91
- **[P2] Avoid following symlink targets for entry metadata:** `src/agents/pi-tools.list.ts:113-115`
  - body: The parent directory is boundary-checked, but each child is inspected with `fs.stat(entryPath)`, which follows symlinks. A workspace symlink to an out-of-workspace file can expose the target's type, size, and mtime; use no-follow metadata or revalidate each resolved child before returning metadata.
  - confidence: 0.88
- **[P2] Sort before applying the entry cap:** `src/agents/pi-tools.list.ts:102-107`
  - body: Entries are capped while iterating raw `readdir` order and sorted only afterward. For directories with more than 1000 entries, callers receive an arbitrary subset rather than the first deterministic sorted page, so `truncated: true` is misleading.
  - confidence: 0.86
- **[P2] Return null for missing mtimes:** `src/agents/pi-tools.list.ts:22-28`
  - body: `ListEntry.mtime` is documented as an ISO-8601 string, but the stat-failure path returns an empty string. Make the field nullable and return `null` when metadata is unavailable so broken symlinks do not violate the structured contract.
  - confidence: 0.82
- **[P3] Document the new filesystem tool:** `src/agents/tool-catalog.ts:72-78`
  - body: This adds a public built-in tool and changes `group:fs`, but the PR does not update the user-facing tool/config/security docs that currently list the filesystem group without `list`. Add docs for availability, policy behavior, sandbox boundaries, output shape, and truncation.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/tool-catalog.ts:72-78`
  - body: This is a user-facing `feat` that adds a built-in tool, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes.
  - confidence: 0.82
