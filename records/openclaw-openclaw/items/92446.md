## Label Justifications

- `P2`: The PR is a bounded CLI improvement, but its public command semantics need correction before merge.
- `merge-risk: 🚨 compatibility`: Adding a parallel public CLI path for edit-mode requests can create lasting command, help, automation, and telemetry ambiguity for users and integrations.

## Summary

The patch is mechanically focused, but current provider contracts already map any request with input images to edit mode, which is exposed by `openclaw infer image edit --file`; adding the same path under `image generate` creates a second public command and mislabels edit-mode execution as `image.generate` without establishing distinct semantics. The external proof is also test-only, so it does not satisfy the real-provider behavior gate.

## What This Changes

This PR adds repeatable `--file` input to `openclaw infer image generate`, includes the flag in capability metadata, forwards loaded files as `inputImages`, and adds a mocked CLI regression test.

## Best Possible Solution

Keep `image edit --file` as the canonical input-image command, or first define and implement a distinct provider-level reference-generation mode with explicit capability metadata, consistent naming, docs, and live proof before exposing it as `image generate --file`.

## Reproduction Assessment

Yes from source: current main rejects `--file` on `image generate`, while `image edit --file` already loads files and reaches the same input-image provider path. The PR’s forwarding behavior is source-verifiable, but no real provider reproduction was supplied.

## Solution Assessment

No. The patch is mechanically plausible but duplicates the existing edit command and labels provider edit-mode execution as generation; the narrower maintainable solution is one canonical command unless maintainers intentionally define a separate reference-generation contract.
