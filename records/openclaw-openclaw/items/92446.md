## Summary

Keep open for maintainer direction: current docs and provider contracts assign file-conditioned requests to image edit, while this route would report image generation for an operation providers treat as editing; the competing https://github.com/openclaw/openclaw/pull/91807 is also mock-only and is not a safe superseding target.

## What This Changes

The PR adds a repeatable `--file` option to `openclaw infer image generate`, forwards loaded files as `inputImages`, advertises the option in capability metadata, and adds a mocked CLI regression test.

## Best Possible Solution

Retain `image edit --file` as the single canonical input-image command, or deliberately introduce a distinct reference-image generation contract with accurate capability attribution, provider support declarations, aligned documentation, and real-provider proof.

## Reproduction Assessment

Yes for the reported parser behavior: current main rejects `--file` on image generate, but source and documentation show that file-conditioned workflows are intentionally exposed through image edit.

## Solution Assessment

No. Adding the flag alone duplicates the supported image-edit route and preserves a misleading generation capability label for requests providers handle as edits.
