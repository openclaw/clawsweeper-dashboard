## Summary

This PR should stay open but is not merge-ready: current main deliberately exposes input files through `image edit`, and the patch duplicates that provider edit/reference-image path under `image generate` while retaining the `image.generate` capability identity. The linked issue remains open, so the right next step is a maintainer decision on the public CLI/capability model rather than closing either item.

## What This Changes

The branch adds repeatable `--file` input to `openclaw infer image generate`, forwards loaded files as `inputImages`, adds the flag to capability metadata, and adds a mocked CLI regression test.

## Best Possible Solution

Keep input-image operations on the documented `image edit --file` route unless maintainers intentionally redesign the capability model; if they choose the latter, align CLI naming, metadata/envelopes, provider capability checks, docs, tests, and real-provider behavior under one coherent contract before merge.

## Reproduction Assessment

Yes: current main rejects `--file` on `image generate`, exposes it on `image edit`, and documents edit as the input-file route. That reproduces a requested feature gap, not broken behavior against an existing generation contract.

## Solution Assessment

No. The patch is mechanically narrow, but duplicating the existing edit/reference-image route while preserving `image.generate` identity is not the narrowest maintainable solution; maintainers should either keep the canonical edit route or redesign the capability contract end to end.
