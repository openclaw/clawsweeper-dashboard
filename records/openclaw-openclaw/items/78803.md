## Summary

Keep open: this is an open draft PR from a MEMBER with the protected `maintainer` label, and the latest diff still has one concrete correctness issue around bounded chat session refreshes dropping the selected older session.

## What This Changes

The branch adjusts Control UI channel/status/config/session refresh paths, adds focused gateway/UI regression tests, and updates the changelog to reduce status and background refresh churn.

## Best Possible Solution

Keep the responsiveness fixes, but preserve or explicitly hydrate the selected chat session when applying active-window session refresh bounds before maintainer landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main can block newer channel refreshes behind slow probes/summaries, and the PR finding is source-reproducible by opening or switching to a non-main session older than the new 120-minute chat refresh window.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the channel/status direction is appropriately scoped, but the bounded chat session refresh should preserve the selected session so the UI does not lose its active option.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve the selected session when bounding chat refreshes:** `ui/src/ui/app-chat.ts:670-671`
  - body: The new chat-side refresh sends `activeMinutes: CHAT_SESSIONS_ACTIVE_MINUTES`, but `sessions.list` filters by `updatedAt` before returning rows. If a user opens or switches to an older non-main session, the refresh can replace `sessionsResult` with a list that omits the active key; `resolveSessionOptionGroups` only re-adds a missing selected key for agent main sessions, so the session selector/model context can drop the active session. Please keep the selected session unfiltered or explicitly preserve/hydrate that row when applying the bound.
  - confidence: 0.84
