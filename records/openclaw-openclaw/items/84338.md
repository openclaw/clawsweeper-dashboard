## Label Justifications

- `P0`: The PR addresses a Gateway shared-secret boundary bypass that can promote a hook ingress token to trusted operator auth.
- `merge-risk: 🚨 compatibility`: Merging this PR intentionally rejects existing configurations where the hook token and Gateway password are the same secret.
- `merge-risk: 🚨 availability`: Those reused-secret deployments can fail startup after upgrade until an operator rotates one of the credentials.

## Summary

Keep open: the branch starts the right startup and audit hardening, but it is not merge-ready because hooks config hot reload still bypasses the new separation check and the external proof is only test/check output claims.

## What This Changes

The PR adds password-mode `hooks.token` versus Gateway password reuse checks to startup auth and security audit coverage.

## Best Possible Solution

Centralize the hook-token versus active Gateway shared-secret validation so startup, audit, and hooks hot reload all fail closed, with regression coverage and release guidance for reused-secret configurations.

## Reproduction Assessment

Yes by source inspection: current main maps Gateway HTTP bearer auth into both token and password, while the existing hook separation guard only checks token-mode auth. I did not execute a runtime repro because this was a read-only review.

## Solution Assessment

No. Startup and audit coverage are necessary, but the best fix must also enforce the same separation during hooks hot reload or force a restart/fail-closed path for that config change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate hook token reuse on hot reload:** `src/gateway/startup-auth.ts:235-245`
  - body: This new check only runs from startup auth, but `hooks` changes are hot-reloaded: `config-reload-plan.ts` marks `hooks` as hot reload and `server-reload-handlers.ts` applies `resolveHooksConfig(nextConfig)` without comparing against the active Gateway password. A running password-mode gateway can still reload `hooks.token` to the Gateway password and leave the hook bearer usable on full Gateway HTTP surfaces until restart, so please enforce the same separation in the reload path and add regression coverage for it.
  - confidence: 0.92
