## Summary

Keep open: current main and the latest release still provide session/agent sandbox activation plus separate sandbox tool allow/deny filtering and exec escape/routing controls, but not the requested per-tool execution-placement contract. The remaining work is a real security/product design decision, especially given the discussion about whether placement should be operator-defined or tool-declared.

## What This Changes

The issue requests configurable per-tool sandbox placement so tools such as web_fetch and browser can run isolated while exec, message, cron, and gateway remain host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between operator policy, tool-declared execution constraints, plugin participation, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue requests a new execution-placement capability rather than broken behavior defined by current docs or tests. Source and docs confirm the requested per-tool placement mode is absent from current main and v2026.6.6.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first decide whether placement is operator-selected, tool-declared, or precedence-controlled before choosing schema and runtime boundaries.
