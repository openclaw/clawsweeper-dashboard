## Summary

Keep open. Current main still exposes session-level sandbox placement and sandbox-only tool filtering, but not the requested per-tool host-vs-sandbox placement policy; the related Docker sandbox PR only unblocked prerequisites, and existing issue discussion already treats this as remaining security/product design work.

## What This Changes

Request a schema-backed per-tool sandbox placement mode so tools like `web_fetch` and `browser` can run in a sandbox while trusted tools like `exec`, `message`, `cron`, and `gateway` stay on host in the same session.

## Best Possible Solution

Add a documented typed tool-placement contract evaluated where tool identity and execution context are both available, with tests separating placement from filtering, browser targets, web_fetch providers, plugin tools, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection on current main gives high-confidence evidence that the requested capability is missing: sandbox status resolves per session, while sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the config schema, plugin/tool context boundary, and mixed host/sandbox enforcement model before implementation.
