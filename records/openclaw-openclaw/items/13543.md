## Summary

Keep open. Current main still has session/agent-level sandbox placement and sandbox-only allow/deny tool filtering, but not a schema-backed per-tool host-vs-sandbox placement policy; the remaining work is security-sensitive product design, not cleanup closure.

## What This Changes

The issue requests per-tool sandbox execution so tools such as `web_fetch` and `browser` can be forced into a sandbox while trusted tools remain host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from sandbox filtering, browser targets, plugin tools, web_fetch providers, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection of current main confirms the missing capability: sandboxing resolves per session/agent and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, security boundary, and runtime enforcement point before implementation.
