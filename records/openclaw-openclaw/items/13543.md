## Summary

Keep open. Current main still has session/agent-level sandbox selection and sandbox tool allow/deny filtering, but not the requested per-tool execution placement policy; the discussion also already frames this as open security-sensitive design work.

## What This Changes

Request a per-tool sandbox placement policy so untrusted web/browser tools can always execute in a sandbox while trusted tools continue to run on the host.

## Best Possible Solution

Add a documented, typed, schema-backed core tool-placement contract evaluated where tool identity and execution context are both available.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspection: current schema, typed config, runtime status, docs, and tool construction do not expose a tool-keyed placement decision.

## Solution Assessment

No. Current main's session sandboxing, sandbox allow/deny policy, browser target controls, and elevated exec are adjacent controls, but they do not solve selective per-tool execution placement.
