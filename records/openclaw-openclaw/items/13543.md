## Summary

Keep open. Current main and the latest release still expose session-level sandbox placement, sandbox tool filtering, browser target selection, and exec elevation, but not the requested schema-backed per-tool host-versus-sandbox placement policy; the latest discussion also shows the exact security contract still needs maintainer choice.

## What This Changes

Request a per-tool sandbox placement policy so tools such as web_fetch and browser can run isolated while exec, message, cron, and gateway remain host-routed in the same session.

## Best Possible Solution

Define one reviewed core placement contract for mixed host/sandbox execution, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime/plugin boundary.

## Reproduction Assessment

Not applicable: this is a feature/security design request rather than a broken existing behavior report. Source, docs, and the latest release show current support for session-level sandbox modes, sandbox tool filtering, browser target selection, and exec elevation, not per-tool placement.

## Solution Assessment

Unclear until maintainers choose the contract. A core placement contract is the right ownership area, but the latest discussion questions the proposed always/never model and suggests a default-sandboxed model with explicit gateway-running tools instead.
