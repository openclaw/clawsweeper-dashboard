## Summary

Keep open. Current main still has session/agent-level sandboxing, sandbox-only allow/deny filtering, browser target selection, and exec elevated escape behavior, but not the requested per-tool host-vs-sandbox placement policy.

## What This Changes

Request a schema-backed per-tool sandbox placement mode so tools such as `web_fetch` and `browser` can be forced into a sandbox while `exec`, `message`, `cron`, and `gateway` remain host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from filtering, browser target policy, provider fallback, plugin tools, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/security request. Source inspection on current main confirms the missing capability: sandboxing resolves per session/agent and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, security boundary, plugin context, and runtime enforcement point before implementation.
