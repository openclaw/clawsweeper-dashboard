## Summary

Keep open: current main and v2026.6.8 still provide session/agent-level sandbox selection, sandbox tool allow/deny filtering, sandboxed browser settings, web_fetch provider restrictions, and exec-specific host routing, but not the requested per-tool execution-placement policy.

## What This Changes

The issue requests schema-backed per-tool sandbox placement so selected tools can be forced into isolation while others remain host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between operator policy, tool-declared execution constraints, plugin participation, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue requests a new execution-placement capability rather than broken behavior under an existing contract. Source and docs confirm the requested per-tool placement mode is absent from current main and v2026.6.8.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first decide the placement contract, owner boundary, and precedence with existing sandbox tool policy and elevated exec before choosing config syntax.
