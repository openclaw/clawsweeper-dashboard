## Summary

Keep open. Current `origin/main` still has session/agent-level sandboxing plus sandbox-only tool allow/deny filtering, but not the requested per-tool host-vs-sandbox placement policy for mixing sandboxed `web_fetch`/`browser` with host-routed `exec`, `message`, `cron`, and `gateway` in one session.

## What This Changes

Request a schema-backed per-tool sandbox placement mode that can force selected tools into a sandbox while leaving trusted tools on the host.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from sandbox filtering, browser host targeting, plugin tools, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection of current `origin/main` confirms the missing capability: sandboxing resolves per session/agent and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, security boundary, and runtime enforcement point before implementation.
