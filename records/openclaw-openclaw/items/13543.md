## Summary

Keep open. Current main and v2026.5.22 still provide session/agent sandbox modes plus sandbox tool allow/deny filtering, not a schema-backed per-tool host-versus-sandbox placement contract; recent discussion is still debating the intended security model.

## What This Changes

Add per-tool sandbox placement so selected tools such as web_fetch or browser can run in a sandbox while trusted tools stay host-routed in the same session.

## Best Possible Solution

Define one reviewed core placement contract for mixed host/sandbox execution, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source and release checks show current main and v2026.5.22 support session/agent sandbox modes plus sandbox allow/deny filtering, not per-tool placement.

## Solution Assessment

Unclear until maintainers choose the contract. The requested selective always/never map is one possible UX, while the latest discussion proposes default sandboxing with explicit gateway exceptions or plugin manifest metadata.
