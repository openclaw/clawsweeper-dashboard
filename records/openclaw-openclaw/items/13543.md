## Summary

Keep open: current main still has session/agent-level sandbox modes and sandbox tool allow/deny filtering, but not the requested per-tool host-vs-sandbox placement policy. The related Docker sandbox PR only unblocked sandbox availability and explicitly treated this work as partial.

## What This Changes

The issue requests schema-backed per-tool sandbox execution so tools such as `web_fetch` and `browser` can be sandbox-routed while trusted tools remain host-routed in the same session.

## Best Possible Solution

Add a documented typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from filtering, browser targets, plugin tools, web_fetch providers, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high-confidence feature-gap evidence: sandbox status resolves per session and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, plugin/tool context boundary, and mixed host/sandbox enforcement model before implementation.
