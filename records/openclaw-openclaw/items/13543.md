## Summary

Keep open. Current main still supports session/agent-level sandbox selection, sandbox tool allow/deny filtering, and browser target defaults, but not a schema-backed per-tool host-vs-sandbox placement policy; maintainer discussion also already frames this as security-sensitive design work.

## What This Changes

Request a per-tool sandbox execution policy so tools such as `web_fetch` and `browser` can be forced into a sandbox while `exec`, `message`, `cron`, and `gateway` remain host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from filtering and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. The missing capability is confirmed by source and docs: current main exposes session-level sandboxing and tool allow/deny filters, not per-tool placement.

## Solution Assessment

Unclear whether the exact proposed JSON shape is best. The direction is valid, but maintainers should choose the schema and runtime placement boundary before implementation.
