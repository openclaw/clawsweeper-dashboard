## Summary

Keep open: current main and latest release still expose session/agent-level sandbox activation, sandbox tool allow/deny filtering, and exec-only elevated escape behavior, but not the requested per-tool execution placement contract. The live discussion remains a security/product design split between operator-selected placement and tool/plugin-declared gateway execution constraints.

## What This Changes

Request to add schema-backed per-tool sandbox placement so selected untrusted tools can run in a sandbox while trusted tools remain host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between sandbox mode, operator policy, tool/plugin-declared gateway execution, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue asks for a new execution-placement capability. Source and docs inspection confirm current main and v2026.6.8 do not have that per-tool placement mode.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first choose the placement owner and precedence model before committing to an always/never map or tool-declared gateway contract.
