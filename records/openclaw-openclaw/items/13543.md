## Summary

Keep open. Current main and the latest release still provide session/agent sandbox placement plus sandbox tool allow/deny filtering, but not a per-tool host-versus-sandbox placement policy; the remaining work is a security-sensitive product/API design.

## What This Changes

The issue requests schema-backed per-tool sandbox execution so tools like `web_fetch` and `browser` can run in Docker while trusted tools like `exec`, `message`, `cron`, and `gateway` stay host-routed in the same session.

## Best Possible Solution

Add a documented core tool-placement contract that routes selected tools to sandbox execution and selected trusted tools to host execution within one session, with tests proving placement remains separate from filtering and elevated exec.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and docs checks show current main only supports session/agent sandbox modes plus tool allow/deny filtering, not per-tool placement.

## Solution Assessment

Unclear until maintainers choose the design. The capability likely belongs in core sandbox/tool routing, but the schema, runtime boundary, and mixed execution security model need product and security review first.
