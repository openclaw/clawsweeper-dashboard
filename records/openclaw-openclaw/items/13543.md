## Summary

Keep open: current main and the latest release still provide session/agent-level sandbox activation plus sandbox tool allow/deny filtering, not the requested per-tool host-vs-sandbox placement contract. The discussion also shows unresolved security/product direction around an operator-selected always/never map versus a tool/plugin-declared gateway execution contract.

## What This Changes

Request a schema-backed feature that can force selected tools such as `web_fetch` and `browser` into a sandbox while trusted tools such as `exec`, `message`, `cron`, and `gateway` remain host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between sandbox mode, tool/plugin declarations, gateway-owned tools, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue asks for a new execution-placement capability. Source and release inspection confirm current main and v2026.6.10 do not have that per-tool placement mode.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first choose the placement owner and precedence model before committing to an always/never map or a tool-declared gateway contract.
