## Summary

Keep open. Current main still has session/agent sandbox placement plus sandbox-only allow/deny filtering, but no per-tool host-vs-sandbox placement contract; related Docker work only unblocked a prerequisite, and the issue discussion already treats this as remaining security/product design work.

## What This Changes

Request a schema-backed per-tool sandbox placement policy so untrusted tools like `web_fetch` and `browser` can run in a sandbox while trusted tools stay host-routed in the same session.

## Best Possible Solution

Add a documented typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from filtering, browser targets, plugin tools, web_fetch providers, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection on current main confirms the missing capability: sandbox status resolves per session/agent and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, plugin/tool context boundary, and runtime enforcement model before implementation.
