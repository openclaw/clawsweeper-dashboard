## Summary

Keep open. Current main still has session/agent-level sandbox selection and sandbox tool allow/deny filtering, but not a schema-backed per-tool execution placement policy that can force web/browser tools into a sandbox while leaving trusted tools host-routed in the same session. Existing maintainer/Codex discussion also explicitly frames this as still-open security-sensitive design work.

## What This Changes

Request a per-tool sandbox placement policy so tools handling untrusted external content can always execute in a sandbox while trusted tools continue to run on the host.

## Best Possible Solution

Add a documented, typed, schema-backed core tool-placement contract evaluated where tool identity and execution context are both available, with regression tests proving web_fetch/browser can be sandbox-routed while trusted tools remain host-routed and existing filtering/elevated behavior stays separate.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspection: current schema, typed config, runtime status, docs, and tool construction do not expose a tool-keyed placement decision.

## Solution Assessment

No. Current main's session sandboxing, sandbox allow/deny policy, browser target controls, and elevated exec are adjacent controls, but they do not solve selective per-tool execution placement.
