## Summary

Keep open. Current main still lacks the requested per-tool host-vs-sandbox placement contract; existing sandbox mode, sandbox tool filtering, browser host-control, and exec host/elevated controls are separate surfaces, and the issue discussion already flags this as security-sensitive maintainer design work.

## What This Changes

Request a schema/config/runtime mode that can force selected tools such as `web_fetch` and `browser` into sandbox execution while keeping tools such as `exec`, `message`, `cron`, and `gateway` host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with tests proving placement stays separate from sandbox allow/deny filtering, browser host targeting, plugin tools, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection of current main confirms the missing capability: sandboxing resolves per session/agent and `tools.sandbox.tools` filters availability rather than choosing host-vs-sandbox placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, security boundary, and runtime enforcement point before implementation.
