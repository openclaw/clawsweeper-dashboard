## Summary

Keep open: current main and v2026.6.8 still provide session/agent sandbox modes, sandbox tool allow/deny filtering, and exec-only elevated escape behavior, but not the requested per-tool execution placement contract. The discussion remains a security/product design split rather than conservative cleanup material.

## What This Changes

The issue requests a configurable per-tool sandbox placement policy so tools such as web_fetch and browser can run isolated while exec, message, cron, and gateway stay host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between sandbox mode, operator policy, tool/plugin-declared gateway execution, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue asks for a new execution-placement capability. Source and release inspection confirm current main and v2026.6.8 do not have that per-tool placement mode.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first choose the placement owner and precedence model before committing to an always/never map or tool-declared gateway contract.
