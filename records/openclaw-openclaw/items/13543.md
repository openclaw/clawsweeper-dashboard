## Summary

Current main still has session-level sandbox activation plus separate tool-availability and exec escape controls, not selective per-tool execution placement. The request remains necessary, but the discussion identifies a real unresolved product and security decision between operator-defined placement rules and tool-declared gateway capabilities.

## What This Changes

Requests a per-tool execution-placement policy that can sandbox selected tools while keeping other tools on the gateway in the same session.

## Best Possible Solution

Define one typed core placement contract with explicit operator policy, tool-declared execution constraints, precedence rules, and plugin participation, then document and test host/sandbox placement independently from tool availability and elevated exec.

## Reproduction Assessment

Not applicable; this issue requests a new execution-placement capability rather than reporting broken behavior defined by current documentation or tests.

## Solution Assessment

No, not in its current proposed form. The security goal is aligned, but maintainers must first decide whether placement is operator-selected, tool-declared, or a precedence-controlled combination before choosing the schema and runtime boundary.
