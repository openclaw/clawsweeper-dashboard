## Summary

Keep open. Current main still has session/agent-level sandbox modes, sandbox tool allow/deny filtering, browser host-control gates, and one sandboxed tool-construction context, but not a schema-backed per-tool host-vs-sandbox placement policy; the existing maintainer discussion also treats this as security-sensitive design work.

## What This Changes

Request a per-tool sandbox placement policy that can force tools such as web_fetch and browser into sandbox execution while leaving trusted tools such as exec, message, cron, and gateway host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage proving selected untrusted tools can be sandbox-routed while trusted tools remain host-routed.

## Reproduction Assessment

Not applicable as a runtime bug. Source and docs confirm current main exposes session/agent sandboxing and sandbox tool allow/deny filters, not per-tool execution placement.

## Solution Assessment

Unclear whether the proposed JSON shape is the best final schema. The direction is valid, but maintainers should choose the config contract and runtime placement boundary before implementation.
