## Summary

Keep open. Current main still has session/agent-level sandboxing plus sandbox-only tool allow/deny filtering, browser target selection, and elevated exec escape behavior, but no schema-backed per-tool execution placement that can force `web_fetch` or `browser` into a sandbox while keeping `exec`, `message`, `cron`, and `gateway` host-routed in the same session.

## What This Changes

Request tool-level sandbox placement so selected untrusted-content tools run in Docker while trusted tools stay on the host in the same session.

## Best Possible Solution

Add a documented, schema-backed tool placement contract in core sandbox/tool routing, with tests proving placement stays separate from sandbox tool filtering and elevated exec.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Current-main source and docs give a high-confidence capability check: sandbox placement remains session-level while `tools.sandbox.tools` only filters availability.

## Solution Assessment

Unclear as proposed; the direction is reasonable, but the exact config shape and runtime placement model need maintainer security/product design before implementation. A narrow local patch should not conflate placement with existing allow/deny filtering or elevated exec.
