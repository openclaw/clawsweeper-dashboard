## Summary

Keep open. Current main and the latest release still have session/agent sandbox modes, sandbox tool allow/deny filtering, and exec elevation, but not a reviewed per-tool host-versus-sandbox placement contract; the discussion is still choosing the security model.

## What This Changes

The issue requests a schema-backed way to force selected tools such as web fetch and browser into a sandbox while keeping trusted tools such as exec, message, cron, and gateway host-routed in the same session.

## Best Possible Solution

Define one reviewed core tool-placement contract, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime boundary across web_fetch, browser, exec, message, cron, and gateway tools.

## Reproduction Assessment

Not applicable: this is a feature request for a new per-tool sandbox placement contract, not a broken existing behavior report. Source and release inspection show the current product supports session/agent sandboxing and allow/deny filtering, not per-tool placement.

## Solution Assessment

Unclear: the selective always/never map is a plausible UX, but the later discussion proposes a different sandbox-by-default contract with explicit gateway-run declarations. Maintainers should choose the security model before code is written.
