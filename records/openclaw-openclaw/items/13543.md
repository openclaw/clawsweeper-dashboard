## Summary

Keep open. Current main still supports session/agent-level sandbox placement and sandbox-only allow/deny filtering, but not the requested schema-backed per-tool host-vs-sandbox placement policy; the live discussion and related Docker PR both treat this as remaining security/product design work.

## What This Changes

Request to add per-tool sandbox placement so tools like `web_fetch` and `browser` can run sandboxed while `exec`, `message`, `cron`, and `gateway` stay host-routed in the same session.

## Best Possible Solution

Add a documented, typed tool-placement contract evaluated where tool identity and execution context are both available, with regression coverage separating placement from filtering, browser targets, plugin tools, web_fetch providers, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection of current main confirms the missing capability: sandbox status resolves per session/agent and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, plugin/tool context boundary, and runtime enforcement model before implementation.
