## Summary

Keep open. Current main still has session-level sandbox selection and sandbox tool allow/deny filtering, but not a schema-backed per-tool host-vs-sandbox placement policy; the related Docker sandbox work only unblocked prerequisites.

## What This Changes

This issue requests per-tool sandbox execution so selected tools run in Docker while trusted tools remain on the host in the same session.

## Best Possible Solution

Add a documented typed tool-placement contract evaluated where tool identity and execution context are both available, with tests separating placement from filtering, browser targets, plugin tools, web_fetch providers, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection on current main gives high-confidence evidence that the requested capability is missing: sandbox status resolves per session and sandbox tool policy filters availability rather than choosing placement per tool.

## Solution Assessment

Unclear whether the proposed JSON shape is the best contract. The direction is valid, but maintainers should choose the schema, plugin/tool context boundary, and mixed host/sandbox enforcement model before implementation.
