## Summary

Current main still provides session-level sandbox selection, sandbox tool visibility policy, browser controls, and exec-specific routing, but no general per-tool execution-placement contract. This remains useful security work, but the discussion exposes an unresolved product decision: whether operators choose placement by tool name or tools declare a gateway/sandbox capability.

**Likely related people**
- `vincentkoc` — high confidence; recent contributor across the current sandbox runtime, schema, and agent tool-construction path.
- `steipete` — medium confidence; contributor to sandbox backends/config and reviewer of this issue's product boundary.
- `ngutman` — medium confidence; contributor to the adjacent sandbox tool-policy implementation that placement must remain distinct from.

## What This Changes

Add selective per-tool execution placement so specified tools run in a sandbox while other tools remain gateway-routed in the same session.

## Best Possible Solution

Define one typed execution-location contract with explicit gateway-only capability metadata, plugin participation rules, and precedence tests across session sandboxing, tool filtering, browser routing, and elevated exec.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. The capability gap is nevertheless high-confidence from current source and documentation, which expose no general per-tool execution-placement contract.

## Solution Assessment

No, the proposed `always` and `never` arrays are not yet proven to be the best solution. Maintainers should first choose between operator-name-based placement and capability metadata, then define precedence with gateway-only tools, plugins, browser targeting, and elevated exec.
