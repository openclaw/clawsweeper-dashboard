## Summary

Keep open: current main and the latest release still provide session/agent-level sandbox activation, sandbox tool allow/deny filtering, browser sandbox settings, and exec-only elevated escape behavior, but not a schema-backed per-tool execution placement contract. The live discussion remains a security/product design split between operator-selected placement and tool/plugin-declared gateway execution constraints.

## What This Changes

Request per-tool sandbox execution placement so selected tools such as web_fetch and browser can run isolated while other tools stay host-routed in the same session.

## Best Possible Solution

Define one documented core placement contract with explicit precedence between sandbox mode, operator policy, tool/plugin-declared gateway execution, browser/web_fetch behavior, and elevated exec, then test placement separately from tool availability filtering.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue asks for a new execution-placement capability. Source and docs inspection confirm current main and v2026.6.8 do not have that per-tool placement mode.

## Solution Assessment

No, not as the proposed schema alone. The security goal is aligned, but maintainers should first choose the placement owner and precedence model before committing to an always/never map or a tool-declared gateway contract.
