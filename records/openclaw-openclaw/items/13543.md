## Summary

Keep open. Current main still has session/agent sandbox placement, sandbox tool allow/deny filtering, browser target selection, and exec-only elevated escape behavior, but not a schema-backed per-tool host-versus-sandbox placement contract; the recent discussion also shows maintainers have not settled the desired security model.

## What This Changes

This issue requests a new configuration and runtime contract for forcing selected tools such as `web_fetch` and `browser` into a sandbox while leaving trusted tools on the host in the same session.

## Best Possible Solution

Define one reviewed core placement contract for mixed host/sandbox execution, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source and release checks show current main and v2026.5.20 support session/agent sandbox modes plus sandbox allow/deny filtering, not per-tool placement.

## Solution Assessment

Unclear until maintainers choose the contract. The capability likely belongs in core sandbox/tool routing, but the schema, plugin manifest boundary, and gateway-running exception model need explicit product and security review first.
