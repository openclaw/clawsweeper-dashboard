## Summary

Keep open. Current main and v2026.6.5 still have session/agent sandbox modes plus sandboxed-session tool allow/deny filtering, not a per-tool host-versus-sandbox placement contract, and the discussion is still a security/product design decision rather than obsolete work.

## What This Changes

Request to add schema-backed per-tool sandbox placement so tools such as `web_fetch` and `browser` can run in a sandbox while trusted tools stay host-routed in the same session.

## Best Possible Solution

Define one reviewed core placement contract for mixed host/sandbox execution, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable: this is a feature request, and source plus release inspection show the requested per-tool placement surface is absent from current main and v2026.6.5.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the contract; the proposed selective always/never map is plausible, while the latest discussion suggests default sandboxing with explicit gateway exceptions or plugin manifest metadata.
