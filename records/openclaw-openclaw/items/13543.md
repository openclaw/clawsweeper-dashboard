## Summary

Keep open. Current main and the latest release still expose session/agent sandbox modes plus sandbox tool allow/deny filtering, not a schema-backed per-tool host-versus-sandbox placement policy; the recent discussion is still debating the intended security contract.

## What This Changes

Request tool-level sandbox placement so selected tools can run in a sandbox while other tools stay host-routed in the same session.

## Best Possible Solution

Define one reviewed core placement contract for mixed host/sandbox execution, keep it separate from sandbox tool filtering and exec elevation, then document and test the chosen schema/runtime boundary.

Source and release checks show current main and v2026.5.22 support session/agent sandbox modes plus sandbox allow/deny filtering, not per-tool placement.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the contract. The requested selective always/never map is one possible UX, while the latest discussion proposes default sandboxing with explicit gateway exceptions or plugin manifest metadata.
