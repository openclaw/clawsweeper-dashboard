## Summary

This should stay open: current main still has a source-reproducible tool-resolution gap where manifest-selected plugin tool owners are only resolved from already-loaded channel/active registries, so config-path tool plugins can disappear when that registry does not already cover them.

## What This Changes

Restore exposure and invocation of path-loaded plugin tools declared in `contracts.tools` after the v2026.5.2 registry reuse regression.

## Best Possible Solution

Add a scoped channel-surface cold-load fallback for the manifest-selected tool plugin ids when the loaded registries do not cover them, with regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a `contracts.tools` plugin can be selected from the manifest, then `resolvePluginTools` returns without loading it when no loaded channel/active registry contains the selected plugin id. I did not perform a live gateway run.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve the loaded-registry fast path, then fall back to a scoped runtime load only for the selected tool plugin ids instead of broadening startup activation or loading all plugins.
