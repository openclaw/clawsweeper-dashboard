## Summary

Keep open: current main still has the reported cold-registry gap for config-path plugin tools, so a manifest-selected tool plugin can be discovered but never runtime-loaded for agent tool materialization.

## What This Changes

Restore request-time exposure of path-based plugin tools declared through `contracts.tools` and loaded from `plugins.load.paths` after `v2026.5.2`.

## Best Possible Solution

Agent tool resolution should lazily materialize only the manifest-selected tool plugin ids when loaded registries do not already contain them.

## Reproduction Assessment

Yes, from source inspection: a config-origin plugin can be selected by `contracts.tools`, but `resolvePluginTools` returns no runtime tools when neither the channel nor active registry already contains that plugin id. I did not run the live gateway reproducer.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve the loaded-registry and descriptor-cache fast paths, then add a scoped channel-surface fallback for the remaining manifest-selected tool plugin ids.

Fix https://github.com/openclaw/openclaw/issues/76598.
