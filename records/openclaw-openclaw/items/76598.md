## Summary

This should stay open: current `main` still resolves plugin tools only from already-loaded channel/active registries, so a manifest-selected config-path plugin can be identified but never runtime-loaded for tool materialization.

## What This Changes

Issue reports that path-loaded plugin tools with manifest origin `config` stopped appearing in agent tool palettes after `v2026.5.2`.

## Best Possible Solution

Keep the startup loaded-registry fast path, then add a scoped channel-surface fallback that loads only the manifest-selected tool plugin ids when the loaded registry lacks them.

## Reproduction Assessment

Yes, from source inspection: a config-path plugin can be selected by `contracts.tools`, but `resolvePluginTools` returns no runtime tools when no loaded registry already contains that plugin id. The reporter’s minimal plugin matches this path, though I did not run the gateway live in this read-only review.

## Solution Assessment

Yes: the maintainable fix is to preserve the current registry reuse path and add the same narrow scoped-load fallback used by adjacent cold-registry repairs, without broadening plugin activation or changing tool policy.

## Repair Work Prompt

Fix the plugin tool cold-registry regression reported in https://github.com/openclaw/openclaw/issues/76598. In `src/plugins/tools.ts`, keep the existing loaded channel/active registry fast path, but when `resolvePluginToolRegistry` cannot find a registry containing the scoped `runtimePluginIds`, use `ensureStandaloneRuntimePluginRegistryLoaded` with `surface: "channel"`, the same `loadOptions`, and `requiredPluginIds: runtimePluginIds`. Do not load all plugins, do not weaken manifest availability/allow/deny checks, and do not change provider/capability fallback behavior. Add a focused regression test in `src/plugins/tools.optional.test.ts` for a manifest-declared tool plugin selected from config/path-style metadata with no compatible loaded registry, verifying the factory is called, the tool is returned, and the load is scoped to the selected plugin id. Preserve existing tests that prove pinned channel registry reuse does not reload. Add a single-line `CHANGELOG.md` fix entry for the user-visible regression, crediting `@minhhoangvn` if attribution is used.
