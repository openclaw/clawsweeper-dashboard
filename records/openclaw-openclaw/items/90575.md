## Summary

Keep open. Current main still lacks the requested first-class cross-profile shared auth-store config; the existing same-state main-agent inheritance and refresh locking solve adjacent multi-agent cases, but not isolated `--profile` state directories, and this needs explicit product and security review.

## What This Changes

Request for a config and CLI surface that lets multiple OpenClaw profiles or instances share one OAuth auth profile store while keeping runtime auth state local.

## Best Possible Solution

Design a core shared-auth-store model around the current SQLite auth-store/state split, explicit security policy, shared refresh-lock ownership, doctor migration, and public docs before adding config or CLI.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for a new shared auth-store surface. Source and docs provide high-confidence evidence that current main only supports per-agent storage plus same-state main-agent read-through, not a configured cross-profile store.

## Solution Assessment

Unclear. The shared-auth need is valid, but the literal `auth.profilesFile` JSON-file shape is probably not the best final form without redesigning it against the current per-agent SQLite store, auth-state split, and lock ownership.
