## Summary

Keep open. Current main and the latest release still lack a first-class cross-profile shared auth-store surface; the existing SQLite auth store, main-agent read-through, and OAuth refresh lock solve adjacent same-state multi-agent cases but not separate `--profile` state directories, and the request needs explicit product and security design.

## What This Changes

Request first-class config and CLI support for multiple OpenClaw profiles or instances to share one OAuth auth store without copying or symlinking credentials.

## Best Possible Solution

Design a core shared-auth-store model around the current per-agent SQLite auth-store/state split, explicit security policy, shared refresh-lock ownership, doctor migration, and public docs before adding config or CLI.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for a new shared auth-store surface. Source and docs provide high-confidence evidence that current main and v2026.6.8 only support per-agent storage plus same-state main-agent read-through, not a configured cross-profile store.

## Solution Assessment

Unclear. The shared-auth need is valid, but the literal auth.profilesFile JSON-file shape is probably not the best final form against the current per-agent SQLite store, auth-state split, and lock ownership.
