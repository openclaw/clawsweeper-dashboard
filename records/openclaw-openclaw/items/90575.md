## Summary

Keep open. Current main and v2026.6.8 still lack a first-class shared auth-store surface across separate `--profile` state roots; existing per-agent SQLite storage, same-profile main-agent inheritance, and OAuth refresh locking solve adjacent cases but not this requested cross-profile credential-sharing design.

## What This Changes

The issue requests first-class config and CLI support for multiple OpenClaw profiles or instances to share one OAuth auth profile store.

## Best Possible Solution

Design a first-class, security-reviewed shared auth-store mechanism that fits the current SQLite auth model, preserves per-agent runtime state, and includes shared refresh locking, doctor migration, and public docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence way to reproduce the missing capability: use a separate `--profile` state root and there is no schema or store resolver hook that points auth profiles at a shared configured path.

## Solution Assessment

No for the literal proposal as written: `auth.profilesFile` targets the old JSON-file model, while current main stores auth profiles in per-agent SQLite. The best solution needs maintainer design for a secure shared SQLite/store owner, refresh locking, doctor migration, and docs.

- Cross-profile refresh safety would need a deliberate shared lock namespace because the current lock path lives under the active state directory.
- Adjacent open auth issues and OAuth migration PRs may affect nearby implementation details, but none currently replaces this product decision.
