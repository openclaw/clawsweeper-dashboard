## Summary

Keep open: current main and v2026.5.22 have mitigated the original two-minute pain with a 30-minute default, but they still do not expose a supported exec-approval timeout config setting. The linked heartbeat report is complementary and already closed; the remaining request needs a security-sensitive public config/API decision.

## What This Changes

Request to let operators configure the exec approval timeout, presumably through openclaw.json, instead of relying on fixed internal timeout constants.

## Best Possible Solution

Define one bounded public approval-timeout contract, preserve the shipped 30-minute default, enforce conservative limits, and thread the resolved value consistently through approval registration, wait, gateway, node, CLI, docs, and tests.

## Reproduction Assessment

Yes, at source level: the strict public approvals schema has no timeout key, and the normal exec approval request path still uses fixed timeout constants. No live approval flow is needed to verify the missing config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the right public contract; maintainers should choose the key, precedence, maximum, and security semantics first.
