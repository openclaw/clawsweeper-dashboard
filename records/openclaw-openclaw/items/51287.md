## Summary

Keep open: current main and v2026.5.22 still do not expose a supported exec-approval timeout setting. The shipped 30-minute default mitigates the original two-minute pain, but the remaining operator-side knob is a security-sensitive public config/API decision.

## What This Changes

Request to add a supported configuration surface for exec approval timeout duration instead of relying on fixed internal constants.

## Best Possible Solution

Define one bounded public approval-timeout contract, preserve the shipped 30-minute default, enforce conservative limits, and thread the resolved value consistently through approval registration and wait paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main has no accepted approval-timeout config key, and normal exec approval paths still use fixed timeout constants unless an internal caller passes timeoutMs directly. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the right public contract; maintainers should choose the key, precedence, maximum, and security semantics first.
