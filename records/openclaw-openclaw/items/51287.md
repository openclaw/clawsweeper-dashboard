## Summary

Keep open: current main and v2026.6.10 still do not expose a supported exec-approval timeout setting; the closed PR attempts did not land, and the remaining work is a security-sensitive public config/API decision.

## What This Changes

Request to let operators configure how long exec approval requests remain pending instead of relying on the fixed default.

## Best Possible Solution

Define one stable, bounded approval-timeout contract after maintainer product/security agreement, preserve the shipped safe default, reject unsafe values, and thread the resolved value consistently through approval registration, wait budgets, transport timeouts, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main has fixed exec approval timeout constants and no public approval-timeout config or environment override in the config schema. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed: configurability is plausible, but exposing internal constant names is not the right public contract. Maintainers still need to choose the key, bounds, precedence, and security semantics.
