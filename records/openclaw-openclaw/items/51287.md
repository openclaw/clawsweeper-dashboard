## Summary

Keep open: current main and the latest release still lack a supported `openclaw.json` key for exec approval expiry. The earlier two-minute pain is mitigated by the shipped 30-minute default, but the remaining request is a security-sensitive config/API decision rather than an implemented feature.

## What This Changes

The issue asks for a public configuration option to control how long exec approval requests remain pending before expiring.

## Best Possible Solution

Define one bounded public approval-timeout contract, preserve the shipped 30-minute default, enforce conservative limits, and thread the resolved value consistently through all approval registration and wait paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main has no accepted approval-timeout config key, and the exec approval paths still use fixed timeout constants unless an internal caller passes `timeoutMs` directly. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the right public contract; maintainers should choose the key, precedence, maximum, and security semantics first.
