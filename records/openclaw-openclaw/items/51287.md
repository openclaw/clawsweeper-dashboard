## Summary

Keep open: current main and the latest release still do not expose a supported exec-approval timeout setting. The original two-minute pain is mitigated by the current 30-minute default, but the remaining request is a security-sensitive public config/API decision for the host-exec authorization window.

## What This Changes

Request to expose a configurable exec approval timeout, likely through OpenClaw config, so operators can tune how long pending host-exec approvals remain valid.

## Best Possible Solution

Define one stable, bounded approval-timeout contract only after maintainer product/security agreement, preserve the shipped 30-minute default, reject unsafe values, and thread the resolved value consistently through approval registration and wait paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main has no public `approvals.timeoutMs` or `tools.exec.approvalTimeoutMs` key, while normal exec approval request and wait paths still use fixed constants. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the right public contract; maintainers still need to choose the key, bounds, precedence, and security semantics.
