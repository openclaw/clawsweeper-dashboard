## Summary

Keep open: current main and v2026.6.6 still do not expose a supported exec-approval timeout setting. The original two-minute pain is mostly mitigated by the current 30-minute default, but the remaining request is a public config/API and host-exec authorization-window decision.

## What This Changes

Request to let operators configure the exec approval timeout, likely through `openclaw.json`, instead of relying on fixed internal timeout constants.

## Best Possible Solution

Define one stable, bounded approval-timeout contract only after maintainer product/security agreement, preserve the shipped 30-minute default, reject unsafe values, and thread the resolved value consistently through approval registration and wait paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main has no accepted approval-timeout config key, and normal exec approval paths still use fixed timeout constants unless an internal caller passes `timeoutMs` directly. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the right public contract; maintainers should choose the key, precedence, maximum, and security semantics first.
