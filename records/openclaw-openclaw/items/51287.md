## Summary

Keep open: current main and the latest release still do not expose a supported exec-approval timeout setting; the old two-minute default is mitigated by the shipped 30-minute default, but the remaining request is a security-sensitive public config/API decision.

## What This Changes

The issue requests a supported configuration surface for the exec approval wait timeout so operators can tune the host-exec approval window instead of relying on the hardcoded default.

## Best Possible Solution

Define one stable, bounded approval-timeout contract only after maintainer product/security agreement, preserve the shipped 30-minute default, reject unsafe values, and thread the resolved value consistently through approval registration, expiry, wait budgets, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main has no public tools.exec.approvalTimeoutMs, approvals.timeoutMs, or OPENCLAW_EXEC_APPROVAL_TIMEOUT_MS surface, while normal exec approval request and wait paths still use fixed constants. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed: configurability is plausible, but exposing internal constant names is not the right public contract. Maintainers still need to choose the key, bounds, precedence, and security semantics.
