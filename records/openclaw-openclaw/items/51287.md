## Summary

Keep open: current main no longer has the two-minute timeout, but the requested configurable exec approval timeout still has no supported config schema, resolver, docs, or end-to-end plumbing. Related implementation PRs are closed unmerged after incomplete and security-sensitive reviews, so this remains a maintainer product/API decision rather than a cleanup close.

## What This Changes

Request to let users override the exec approval timeout in OpenClaw configuration instead of relying only on fixed defaults.

## Best Possible Solution

Add a bounded, documented approval-timeout config surface selected by maintainers, preserve the 30-minute default, propagate it through all approval registration/wait paths, and cover schema, propagation, and expiry tests.

## Reproduction Assessment

Yes, at source level. Current main has no accepted config key in the strict schemas and the runtime approval paths still use fixed constants, though I did not run a live approval flow in this read-only review.

## Solution Assessment

Unclear as currently proposed. A configurable timeout is plausible, but exposing internal constant names is not the best public API; maintainers should choose the stable key, precedence, bounds, and security contract first.

## Security Review

Status: needs_attention

Summary: The requested change affects how long host-exec approvals remain actionable, so any implementation needs bounded validation and explicit security documentation.

Concerns:

- **[medium] Bound configurable host-exec approval windows:**
  - body: A user-configurable approval timeout can extend the period during which a pending host-exec approval may authorize command execution. A landable design should enforce a conservative documented maximum and keep transport waits aligned with that bound.
