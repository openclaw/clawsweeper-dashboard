## Summary

Keep open: current main still has no supported config key for exec approval expiry, even though the old two-minute pain has been mitigated by the shipped 30-minute default. The remaining work is a security-sensitive config/API decision, not a small bug-fix cleanup.

## What This Changes

The issue requests a user-configurable timeout for pending exec approvals, likely via `openclaw.json`.

## Best Possible Solution

Define one bounded public approval-timeout config contract, preserve the 30-minute default, enforce conservative limits, and thread the resolved value consistently through agent, gateway, node, CLI, transport, schema/help/docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main has no accepted approval-timeout config key, and the exec approval paths still use fixed timeout constants unless an internal caller passes `timeoutMs` directly. No live approval flow is required to prove the missing public config surface.

## Solution Assessment

Unclear as proposed. Configurability is plausible, but exposing internal constant names is not the best public contract; maintainers should choose the key, precedence, maximum, and security semantics first.
