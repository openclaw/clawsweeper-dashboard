## Summary

Keep open: current main still has the read-tool-to-Telegram media bridge gap. Telegram media delivery works once a `mediaUrl` exists, but the current read result path still produces pathless image blocks that the shared extractor intentionally drops.

## What This Changes

Request that images read by the built-in `read` tool be attached to Telegram replies instead of only returning image metadata/text.

## Best Possible Solution

Add structured media metadata for successful trusted read-image results using the validated read path, then let the existing pending-media and Telegram delivery paths handle the attachment with regression coverage for pathless Pi results and untrusted/local-path boundary cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Pi read result containing `Read image file [image/png]` plus an image block but no `details.path` is currently asserted to extract no media, so Telegram receives no `mediaUrl` to send.

## Solution Assessment

No, the direct Bot API workaround is not the best repository fix. The narrow maintainable fix is to emit safe structured media metadata from trusted read results and preserve the existing local media gates.
