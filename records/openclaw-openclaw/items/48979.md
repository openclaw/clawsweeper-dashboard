## Summary

Current main still leaves the reported read-tool-to-Telegram image path unimplemented. The built-in read wrapper returns sanitized text/image blocks but does not attach structured media metadata, the shared extractor still returns no media for pathless read image results, and Telegram only sends images once a media URL/path reaches its delivery payload.

## What This Changes

Allow Telegram channel replies to deliver images produced by the built-in read tool as media attachments instead of only text/image metadata.

## Best Possible Solution

Add a narrow read-tool media bridge in the trusted read-result path, likely by annotating validated read image results with structured media metadata or another safe attachment handle, then keep Telegram on the existing pending-media and sendPhoto delivery path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Pi/read-shaped image result with text plus an image block but no MEDIA directive or details.path is currently covered by a test that expects no extracted media.

## Solution Assessment

No current workaround is the best fix. Calling Telegram directly proves transport capability, but the maintainable solution is to emit safe structured media from trusted read results and let the existing channel delivery pipeline handle it.
