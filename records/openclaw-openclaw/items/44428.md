## Summary

Current main still matches the reported cron delivery bug: `deliveryBestEffort` is passed into dispatch, but active descendant subagents still trigger the full-timeout wait and the active-descendant delivery suppression without checking that flag. The latest release tag also contains the same behavior, and the related fix PR #46160 is closed unmerged, so the issue should stay open and is a good narrow repair candidate.

## What This Changes

The issue requests that isolated cron jobs with `delivery.bestEffort: true` deliver the parent run output immediately instead of waiting for or suppressing delivery because fire-and-forget descendant subagents are still active.

## Best Possible Solution

Add a narrow cron delivery fix in `src/cron/isolated-agent/delivery-dispatch.ts`: when `deliveryBestEffort` is true and descendants are still active, do not wait for descendant summaries and do not suppress the parent delivery solely because descendants remain active. Keep existing non-best-effort orchestration behavior and completed-descendant fallback intact, with focused regression coverage and a changelog entry if the user-visible contract changes.

## Reproduction Assessment

Yes. The source path is deterministic: an isolated cron delivery with `deliveryBestEffort: true` and `countActiveDescendantRuns() > 0` still enters the descendant wait with the full timeout and can then suppress delivery; the existing unit harness already has the active-descendant mock path, but lacks the best-effort variant.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the active-descendant wait/suppression paths respect the existing `deliveryBestEffort` flag while leaving non-best-effort descendant follow-up semantics unchanged.
