## Summary

Current main still matches the reported cron delivery bug: best-effort isolated cron delivery reaches the dispatch layer, but active descendant subagents still trigger the full-timeout follow-up wait and the active-descendant delivery suppression without checking `deliveryBestEffort`. The closed related PR proposed the right narrow shape but was not merged.

## What This Changes

The issue asks OpenClaw to let `delivery.bestEffort: true` isolated cron jobs deliver the parent result immediately when fire-and-forget subagents are still active.

## Best Possible Solution

Make the active-descendant wait and suppression paths respect the existing best-effort flag, preserving non-best-effort descendant follow-up semantics and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path: `deliveryBestEffort: true` is passed into dispatch, but active descendants still take the unguarded wait and suppression branches; I did not run a live cron job in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to gate the active-descendant wait and suppression on non-best-effort delivery, with tests proving best-effort still delivers while non-best-effort keeps waiting for descendant summaries.
