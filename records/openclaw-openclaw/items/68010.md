## Summary

Keep this PR open: current main still has the eager compatibility export and the PR contains the right narrow production direction, but the proposed patch remains incorrect because the regression test is ineffective and the provider env-var hardening needs maintainer security review before merge.

## What This Changes

The PR aliases the legacy provider env API-key candidate export to the lazy provider auth proxy, adds a compat lazy-loading test, and changes provider env-var candidate hardening in `src/secrets/provider-env-vars.ts`.

## Best Possible Solution

Keep this PR as the implementation candidate, but land only after the compat export delegates to the lazy proxy with a real post-import regression test and the provider env-var hardening is either split out or explicitly approved with targeted coverage.

## Reproduction Assessment

Yes. Static inspection on current main is enough: importing `src/agents/model-auth-env-vars.ts` evaluates line 39, which calls the manifest-backed resolver path; the PR test defect is visible because the not-called assertion runs before the dynamic import is awaited.

## Solution Assessment

No, not as-is. The lazy alias is the narrow maintainable production fix, but the regression test must assert after module evaluation and the security-sensitive provider env-var changes need a reviewed policy boundary.

## Review Findings

  - body: The not-called assertion runs before `await modPromise`, so the module has not necessarily evaluated yet.   - confidence: 0.86

  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No new dependency or workflow execution path was found, but the diff touches environment-secret discovery and leaves concrete provider env-var trust/cache concerns unresolved.

Concerns:

- **[high] Workspace plugin env vars remain trusted by default:** `src/secrets/provider-env-vars.ts:59`
  - body: The PR adds regex and denylist filtering, but `shouldUsePluginProviderEnvVars` still includes workspace plugin `providerAuthEnvVars` unless callers pass `includeUntrustedWorkspacePlugins: false`; runtime `resolveEnvApiKey` builds lookup params without that flag, so untrusted workspace metadata can still influence which host env vars are read as provider credentials.
  - confidence: 0.8
- **[medium] Descriptor trap still exposes cached arrays:** `src/secrets/provider-env-vars.ts:203`
  - body: The PR copies arrays in the proxy `get` trap, but `getOwnPropertyDescriptor` still returns the original cached array as `value`, leaving a concrete mutation bypass for shared provider env-var candidates.
  - confidence: 0.86
- **[medium] Global lazy cache needs trust-boundary review:** `src/secrets/provider-env-vars.ts:224`
  - body: Aliasing the compat export to the parameterless global lazy proxy means first access can cache candidate maps derived from ambient workspace/config state; maintainers should decide whether that cache is safe across workspaces or needs parameterized/reset behavior.
  - confidence: 0.68
