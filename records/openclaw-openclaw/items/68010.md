## Summary

Keep open and do not merge as-is. The latest head no longer changes the compat export described by the PR body, and the remaining `src/secrets/provider-env-vars.ts` rewrite introduces blocking provider-auth contract regressions plus security-sensitive trust-policy changes without real behavior proof.

## What This Changes

The PR rewrites provider env-var candidate resolution in `src/secrets/provider-env-vars.ts` to use manifest-registry discovery, env-name filtering, null-prototype maps, copied proxy reads, and extra MiniMax auth env names.

## Best Possible Solution

Narrow the fix back to a lazy compat export with a real post-import regression test, then handle provider env-var trust hardening separately with explicit security review while preserving setup env vars, auth evidence, and default lookup caching.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the branch deletes provider-auth evidence exports still imported by current main and skips documented/tested setup provider env vars. I did not run the live OpenRouter flow because this review is read-only and the PR has no real run proof.

## Solution Assessment

No. The latest branch no longer applies the narrow lazy compat-export fix and instead rewrites provider env-secret resolution in a way that breaks existing contracts and needs security-policy judgment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Restore the lazy compat export path:** `src/secrets/provider-env-vars.ts:114-118`
  - body: `src/agents/model-auth-env-vars.ts` still initializes `PROVIDER_ENV_API_KEY_CANDIDATES` by calling this resolver at import time. Since this patch changes the resolver to call `loadPluginManifestRegistry()` directly and no longer changes the compat export, importing the compat module still performs manifest discovery before property access, so the reported lazy-loading bug remains.
  - confidence: 0.9
- **[P1] Restore auth-evidence exports before merging:** `src/secrets/provider-env-vars.ts:24-29`
  - body: This patch deletes `ProviderAuthEvidence` and `resolveProviderAuthEvidence`, but current main still imports and re-exports them from `src/agents/model-auth-env-vars.ts` and types `src/agents/model-auth-env.ts` against `ProviderAuthEvidence`. The branch will fail typechecking and drops manifest auth-evidence support unless these exports remain or all callers are migrated.
  - confidence: 0.96
- **[P2] Keep setup provider env vars in lookups:** `src/secrets/provider-env-vars.ts:125-127`
  - body: The new loop skips every plugin that lacks `providerAuthEnvVars`, so `setup.providers[].envVars` no longer contribute to `getProviderEnvVars()` or known-secret lists. That breaks the documented preferred manifest path for new plugins and the existing dynamic test that expects setup provider env vars to resolve without runtime loading.
  - confidence: 0.94
- **[P2] Copy descriptor values from the lazy cache:** `src/secrets/provider-env-vars.ts:203-210`
  - body: The defensive copy only applies to normal property access. `Object.getOwnPropertyDescriptor(PROVIDER_AUTH_ENV_VAR_CANDIDATES, id)?.value` still returns the cached array from the descriptor trap, so callers can mutate shared candidate state despite this hardening.
  - confidence: 0.88
- **[P3] Preserve the parameterless lookup cache:** `src/secrets/provider-env-vars.ts:251`
  - body: Changing `getProviderEnvVars()` to always call `resolveProviderEnvVars(params)` bypasses the existing lazy `PROVIDER_ENV_VARS` cache for ordinary parameterless calls. Current coverage expects repeated default lookups to reuse the same lazy metadata load; this regresses the hot lookup path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes host env secret discovery and workspace plugin trust behavior, and concrete trust/cache hardening concerns remain.

Concerns:

- **[high] Workspace plugin env vars remain trusted by default:** `src/secrets/provider-env-vars.ts:48`
  - body: `shouldUsePluginProviderEnvVars` still includes workspace plugin-provided env vars unless callers explicitly pass `includeUntrustedWorkspacePlugins: false`, while the new regex still accepts many unrelated uppercase secret names. That does not fully address the exfiltration concern raised in the PR discussion and needs explicit security-policy review.
  - confidence: 0.83
- **[medium] Descriptor trap still exposes cached arrays:** `src/secrets/provider-env-vars.ts:203`
  - body: The PR copies arrays in the proxy `get` trap, but `getOwnPropertyDescriptor` still returns the original cached array as `value`, leaving a mutation bypass for shared provider env-var candidates.
  - confidence: 0.88
- **[medium] Auth-evidence removal changes credential trust behavior:** `src/secrets/provider-env-vars.ts:24`
  - body: Removing provider auth evidence support from the env-var module breaks the established non-secret local-file credential marker path used by auth resolution, or forces callers into an unreviewed alternate credential behavior if the branch is made to compile another way.
  - confidence: 0.78
