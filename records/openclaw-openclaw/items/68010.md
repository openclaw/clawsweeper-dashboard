## Summary

Keep open: the latest PR head is not mergeable because it removes exported auth-evidence contracts still used by current main, drops supported setup-provider env metadata, and lacks after-fix real behavior proof for the OpenRouter path.

## What This Changes

The PR currently rewrites `src/secrets/provider-env-vars.ts` to resolve provider env candidates from manifest registry data with extra env-name filtering, null-prototype maps, and lazy-proxy defensive-copy changes.

## Best Possible Solution

Keep the PR open only as a review candidate, but narrow it to the intended lazy compat fix while preserving metadata-snapshot, setup env-var, and auth-evidence contracts; split any provider env-var security policy hardening into a separately reviewed change with targeted tests and real OpenRouter proof.

## Reproduction Assessment

Yes. Static inspection shows current main still eagerly initializes `PROVIDER_ENV_API_KEY_CANDIDATES`, and CI on the PR head reproduces concrete TS2305 failures from the removed provider auth evidence exports.

## Solution Assessment

No. The narrow production direction is to preserve the current provider metadata contracts and make the compat API-key candidate path lazy; the latest PR head instead rewrites secret-discovery policy, drops supported metadata, and fails typecheck.
