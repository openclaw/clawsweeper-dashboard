## Summary

Keep open for maintainer review; the latest head is not mergeable because it breaks provider-auth exports and setup/env/auth-evidence contracts, and it still lacks real behavior proof for this external bug-fix PR.

## What This Changes

The PR currently rewrites `src/secrets/provider-env-vars.ts` to use manifest-registry lookup, add provider/env-name guards, copy lazy proxy `get` results, remove auth-evidence support, and adjust provider env-var lists.

## Best Possible Solution

Narrow the bug fix back to a lazy compat export with a real post-import regression test, then split or explicitly approve provider env-var trust hardening while preserving setup env-var and auth-evidence contracts.

## Reproduction Assessment

Yes. Source inspection plus the PR CI log gives a high-confidence reproduction of the current PR defects: head `5a34aea` removes exports still imported by current callers, and `check-prod-types` reports TS2305 for those imports.

## Solution Assessment

No. The latest PR head no longer implements the lazy compat-export change described by the title/body and instead rewrites provider env-var resolution in a way that breaks existing contracts and mixes in security-policy changes.
