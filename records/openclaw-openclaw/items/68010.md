## Summary

Keep open for maintainer handling: the latest PR head is not mergeable as-is, no longer changes the compat module described by the title/body, and breaks provider auth/env-var contracts while touching security-sensitive secret discovery.

## What This Changes

The branch rewrites `src/secrets/provider-env-vars.ts` to use manifest-registry lookup, filter env-var names, block prototype-pollution keys, copy lazy proxy `get` results, and adjust known provider auth env names.

## Best Possible Solution

Narrow the branch back to the lazy compat export plus a post-import regression test, and split provider env-var trust hardening into an explicitly reviewed security/policy change.

## Reproduction Assessment

Yes for the PR defects: source inspection shows removed exports still imported by current main, skipped setup env-var metadata, and a descriptor trap that still exposes cached arrays. There is no live after-fix OpenRouter run proving the original user flow.

## Solution Assessment

No. The current diff is not the narrow lazy-loading fix and instead rewrites security-sensitive provider env resolution while breaking existing contracts; the maintainable path is a small compat export change plus focused regression coverage.

  - confidence: 0.94

  - confidence: 0.88
- **[P3] Preserve the parameterless lookup cache:** `src/secrets/provider-env-vars.ts:251`
  - body: Changing `getProviderEnvVars()` to always call `resolveProviderEnvVars(params)` bypasses the existing lazy `PROVIDER_ENV_VARS` cache for ordinary parameterless calls. Current coverage expects repeated default lookups to reuse the same lazy metadata load, so this regresses the hot lookup path.
  - confidence: 0.82
