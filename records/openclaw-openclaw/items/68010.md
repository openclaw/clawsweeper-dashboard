## Summary

Keep this PR open. Current main still eagerly resolves the compatibility export during module import, so the narrow production idea remains relevant, but the proposed patch is not merge-ready because its regression test does not actually prove laziness and the provider env-var hardening leaves security-sensitive trust/cache concerns unresolved.

## What This Changes

The PR aliases `PROVIDER_ENV_API_KEY_CANDIDATES` to the lazy provider auth env-var proxy, adds a compat lazy-loading test, and adds provider env-var candidate hardening in `src/secrets/provider-env-vars.ts`.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but land only a rebased narrow compat-export fix with a post-import, pre-property-access regression test. Provider env-var trust defaults, denylist policy, descriptor defensive copying, and global cache scope should be split into a clearly reviewed hardening patch or explicitly approved here with targeted coverage.

## Reproduction Assessment

Yes. The current-main reproduction is static: importing `src/agents/model-auth-env-vars.ts` evaluates the top-level const at line 39, which calls the provider env resolver before property access; the PR test defect is visible because the negative assertion runs before the dynamic import is awaited.

## Solution Assessment

No, not as-is. Delegating the compatibility export to the existing lazy proxy is the narrow production direction, but the patch should not merge until the lazy-loading test is fixed, descriptor copying is complete, and the security-sensitive provider env-var policy changes are reviewed or split.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes provider environment-secret discovery while leaving concrete trust and shared-cache concerns unresolved.

Concerns:

- **[high] Workspace plugin env vars remain trusted by default:** `src/secrets/provider-env-vars.ts:48`
  - body: The PR adds regex and denylist filtering, but `shouldUsePluginProviderEnvVars` still includes workspace plugin `providerAuthEnvVars` unless callers explicitly pass `includeUntrustedWorkspacePlugins: false`; runtime `resolveEnvApiKey` builds lookup params without that flag, so workspace plugin metadata can still influence which host env vars are read as provider credentials.
  - confidence: 0.8
- **[medium] Descriptor trap still exposes cached arrays:** `src/secrets/provider-env-vars.ts:203`
  - body: The PR copies arrays in the proxy `get` trap, but `getOwnPropertyDescriptor` still returns the original cached array as `value`, leaving a concrete mutation bypass for shared provider env-var candidates.
  - confidence: 0.86
- **[medium] Global lazy cache needs trust-boundary review:** `src/secrets/provider-env-vars.ts:224`
  - body: Aliasing the compat export to the parameterless global lazy proxy means first access can cache candidate maps derived from ambient workspace/config state; maintainers should decide whether that cache is safe across workspace or tenant boundaries or needs parameterized/reset behavior.
  - confidence: 0.68
