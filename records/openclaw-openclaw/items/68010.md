The PR now modifies only `src/secrets/provider-env-vars.ts`, replacing metadata-snapshot env lookup with manifest-registry lookup, adding env-name/prototype-pollution guards and array copying, removing auth-evidence support, and changing env-var list behavior.

## Best Possible Solution

Narrow the bug fix back to a lazy compat export with a real post-import regression test, then handle provider env-var trust hardening as an explicitly reviewed security/policy change without dropping existing setup/env/auth-evidence contracts.

  - confidence: 0.94

  - body: The added defensive copy only applies to normal property access. `Object.getOwnPropertyDescriptor(PROVIDER_AUTH_ENV_VAR_CANDIDATES, id)?.value` still returns the cached array from the descriptor trap, so callers can mutate shared candidate state despite this hardening.
  - confidence: 0.88
- **[P3] Preserve the parameterless lookup cache:** `src/secrets/provider-env-vars.ts:251`
  - body: Changing `getProviderEnvVars()` to always call `resolveProviderEnvVars(params)` bypasses the existing lazy `PROVIDER_ENV_VARS` cache for ordinary parameterless calls. Current coverage expects repeated default lookups to reuse the same lazy metadata load; this regresses the hot lookup path.
  - confidence: 0.82
