## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain automatic URL preloads to image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches every HTTP(S) URL in the prompt, and the new remote-url path fetches it before knowing whether the user meant it as an image. A normal docs or issue link in a prompt would now cause an outbound gateway request, adding privacy, latency, and availability risk; scope this to intentional image refs or keep non-obvious URLs on the existing fallback path.
  - confidence: 0.92
- **[P1] Forward the remote-media SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia` with only `maxBytes`, so configured `tools.web.fetch.ssrfPolicy` values used by the image/PDF/generation media tools are not honored for native prompt URL preloads. Thread the policy from the runner config into `detectAndLoadPromptImages` and `loadImageFromRef` before enabling this fetch path.
  - confidence: 0.9
- **[P2] Keep remote URL dedupe case-sensitive:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef` uses `normalizeRefForDedupe`, which lowercases keys on Windows for filesystem paths. HTTP URL paths can be case-sensitive, so `https://host/Image.png` and `https://host/image.png` collapse into one native image on Windows; use URL-specific dedupe normalization instead.
  - confidence: 0.83
- **[P2] Strip closing delimiters from URL refs:** `src/agents/pi-embedded-runner/run/images.ts:96`
  - body: The trailing-punctuation trim does not remove closing parentheses, but the HTTP URL regex can capture them. Parenthesized or Markdown-style image URLs like `(https://example.com/image.png)` will be fetched as `image.png)`, causing avoidable load failures unless the server happens to host that path.
  - confidence: 0.82
