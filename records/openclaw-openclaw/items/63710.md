## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict URL preload to explicit image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and the later prompt-wide scan sends each match through the native image preload path. A normal prompt containing a docs link or arbitrary webpage would now cause the gateway to fetch it before the model asks for a tool, expanding latency and privacy exposure beyond the stated image-URL fix.
  - confidence: 0.93
- **[P1] Forward the SSRF policy for remote prompt images:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia` with only `maxBytes`, unlike existing media-tool remote loads that forward the configured SSRF policy and remote read-idle timeout. This means operator allowlists, fake-IP/private-range settings, and timeout behavior are silently bypassed for prompt URL image preloads.
  - confidence: 0.91
- **[P2] Do not mark skipped bracket URLs as loaded refs:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch still adds the URL to `seen` even though it does not emit a remote-url ref. If the same URL is explicitly repeated later in the user prompt, the full-prompt scan suppresses it, so the remote image can be lost when the local path is unavailable.
  - confidence: 0.84
- **[P2] Trim closing delimiters from URL matches:** `src/agents/pi-embedded-runner/run/images.ts:96`
  - body: The URL regex can include a closing `)` from parenthesized or Markdown-style text, and the trailing-punctuation sanitizer does not remove it. `new URL()` accepts that as part of the path, so common prompts like `(https://example.com/image.png)` fetch the wrong URL and fail avoidably.
  - confidence: 0.8
