## Summary

Keep this PR open: it targets a real current-main mixed-source image bug, but the patch still introduces security-sensitive outbound fetch behavior, drops the existing SSRF-policy contract, and lacks convincing after-fix real behavior proof.

## What This Changes

The PR adds HTTP(S) prompt image URL detection and preloading so remote image refs can be merged with uploaded or offloaded images in native vision requests.

## Best Possible Solution

Rework the native prompt-image path to preload only clearly intended remote image refs, forward the configured media SSRF policy, preserve current local/sandbox rules, and add focused regression coverage for mixed-source images, non-image links, punctuation, dedupe, and policy propagation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: an inline/uploaded image plus an explicit HTTPS image URL leaves the URL out of `detectImageReferences()`, so it cannot be merged into the first native vision image payload.

## Solution Assessment

No. The PR changes the right surface, but this implementation is not the best fix because it preloads every HTTP(S) URL-like token and omits the established media-fetch SSRF policy; a safer fix should narrow URL detection and preserve the existing security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid fetching every HTTP URL as a prompt image:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and the new preload loop fetches each detected ref before discarding non-images. A normal prompt that mentions a docs page or tracking link would now trigger an outbound gateway request, adding latency and privacy exposure outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy for remote images:** `src/agents/pi-embedded-runner/run/images.ts:458-462`
  - body: The `remote-url` branch calls `loadWebMedia()` with only `maxBytes`, unlike the existing media tool path that passes `remoteMediaSsrfPolicy`. Configured `tools.web.fetch.ssrfPolicy` allowlists or private-network controls would not be honored for native remote image preloads.
  - confidence: 0.94
- **[P2] Do not mark skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it does not emit a `remote-url` ref. If the same URL is repeated explicitly elsewhere in the prompt, the later scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef()` reuses `normalizeRefForDedupe()`, which lowercases keys on Windows for filesystem paths. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected HTTP image URLs:** `src/agents/pi-embedded-runner/run/images.ts:96-118`
  - body: The URL regex allows `)` and the punctuation trimmer does not remove it, so parenthesized or Markdown-style links such as `(https://example.com/image.png)` are fetched as `image.png)`. That causes avoidable native image-load failures for common prompt formatting.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a new prompt-triggered remote fetch path that broadens outbound requests and bypasses the existing configurable media SSRF policy.

Concerns:

- **[medium] Prompt text can trigger unintended outbound fetches:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: Because the new regex treats every HTTP(S) URL as an image candidate, ordinary links in vision-model prompts cause gateway network requests even when they are not image URLs.
  - confidence: 0.95

  - body: The new `remote-url` loader calls `loadWebMedia()` without passing the configured `tools.web.fetch.ssrfPolicy`, so native image preloads do not honor the same policy used by existing media tool paths.
