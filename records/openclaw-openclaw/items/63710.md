## Summary

Keep open: current main and the latest release still have the mixed HTTP(S) prompt-image gap, and this PR is a useful fix candidate, but it is not merge-ready because the branch conflicts with the renamed runner path, the screenshot does not prove the after-fix behavior, and the new URL preload path expands outbound fetch behavior without preserving SSRF policy.

## What This Changes

This PR changes the old `src/agents/pi-embedded-runner/run/images.ts` prompt-image helper to detect HTTP(S) URL refs, load them with `loadWebMedia`, merge them with inline images, and add focused regression tests.

## Best Possible Solution

Port the focused mixed-source image fix to `src/agents/embedded-agent-runner/run/images.ts`, limit automatic remote preloading to explicit image references, thread the established `tools.web.fetch.ssrfPolicy` into that fetch path, and add current-path tests plus redacted WebUI proof showing both images in the first native vision request.

## Reproduction Assessment

Yes at source level: current main still ignores HTTP(S) prompt image URLs before native vision image assembly, and current tests assert that local-only behavior. No live WebUI plus model run was performed in this read-only review.

## Solution Assessment

No, not as currently submitted. Deterministic preloading before the first vision request is the right direction, but the PR needs to be ported to the current runner path and repaired for URL scope and SSRF policy before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass SSRF policy into remote prompt-image fetches:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia` with only `maxBytes`. `loadWebMedia` and the existing image tool both support `ssrfPolicy`, including hostname allowlists and private-network/fake-IP policy, so this new prompt preload path would not honor the configured fetch boundary.
  - confidence: 0.9
- **[P1] Do not fetch every URL in vision prompts:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and `detectImageReferences` then preloads every match before the model call for vision-capable models. Ordinary links in prompts would now trigger gateway network requests even when they are not explicit image inputs, adding privacy and latency risk outside the stated bug fix.
  - confidence: 0.86
- **[P2] Keep remote URL dedupe case-sensitive:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef` reuses the filesystem dedupe normalizer, which lowercases keys on Windows. HTTP URL paths can be case-sensitive, so a Windows gateway would collapse distinct image URLs such as `/Image.png` and `/image.png` and silently drop one from the native vision payload.
  - confidence: 0.82
- **[P2] Only mark bracket URLs seen when emitting them:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` bracket has a local path, this branch still adds the remote URL to `seen` even though it does not emit a `remote-url` ref. If the same URL is intentionally repeated elsewhere in the prompt, the later full-prompt scan suppresses it and the remote image is lost.
  - confidence: 0.78
- **[P2] Trim closing delimiters from detected URLs:** `src/agents/pi-embedded-runner/run/images.ts:96-118`
  - body: The sanitizer trims commas and sentence punctuation but not closing delimiters, while the URL regex can capture `)` in parenthesized or Markdown-style text. That makes the preload fetch the wrong URL path for common prompts like `(https://example.com/image.png)`.
  - confidence: 0.74
