## Summary

Keep this PR open, but it is not merge-ready: it targets a real source-reproducible WebUI mixed-image bug, while the current patch still expands prompt-triggered remote fetching, skips existing SSRF/sandbox media contracts, and lacks after-fix real behavior proof.

## What This Changes

The PR adds HTTP(S) prompt-image URL detection, remote preloading, and regression tests so URL images can be merged with inline images in the native vision payload.

## Best Possible Solution

Rework the native remote-image preload path to load only intended image refs, preserve SSRF and sandbox contracts, and land with redacted after-fix WebUI or runtime proof.

## Reproduction Assessment

Yes, source-level. Current main filters HTTP(S) prompt image refs out of native image detection, and current tests still expect a local path plus HTTPS image URL prompt to detect only the local path.

## Solution Assessment

No. The PR changes the right ingestion surface, but this implementation is not the narrow maintainable fix because it fetches arbitrary links and skips established SSRF and sandbox protections.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restrict prompt URL preloads to image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches every HTTP(S) URL, and the prompt scan preloads each match before knowing whether it is intended as an image. A normal vision prompt that mentions a docs page or tracking link would now trigger an outbound gateway request outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia()` with only `maxBytes`. Existing media tool paths pass the configured web-fetch SSRF policy, so allowlists, private-network controls, and dispatcher policies are not honored for native remote-image preloads.
  - confidence: 0.94
- **[P1] Preserve sandbox remote-URL blocking:** `src/agents/pi-embedded-runner/run/images.ts:458-462`
  - body: The `remote-url` branch runs before sandbox path validation and fetches HTTP(S) prompt refs even when a sandbox is active. That diverges from the existing image tool contract, which rejects remote URLs under sandbox mode.
  - confidence: 0.9
- **[P2] Do not mark skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it emits no `remote-url` ref. If the same URL appears explicitly later, the full-prompt scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef()` reuses filesystem dedupe normalization, which lowercases keys on Windows. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected HTTP URLs:** `src/agents/pi-embedded-runner/run/images.ts:96-118`
  - body: The URL regex allows `)`, but the trailing-punctuation trimmer does not remove it. Parenthesized or Markdown-style links such as `(https://example.com/image.png)` are fetched as `image.png)`, causing avoidable native image-load failures.
  - confidence: 0.82
