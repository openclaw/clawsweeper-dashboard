## Summary

Keep open. The underlying mixed inline-image plus HTTP(S) image URL bug still exists on current main, but this PR is not mergeable as-is because the new remote preload path broadens outbound fetching, misses existing SSRF/sandbox contracts, and lacks convincing after-fix real behavior proof.

## What This Changes

The branch adds HTTP(S) prompt image URL detection, preloads those URLs with `loadWebMedia`, merges them into native vision image payloads, and adds focused `images.test.ts` coverage.

## Best Possible Solution

Rework the native prompt-image path to preload only clearly intended remote image refs, preserve SSRF and sandbox constraints, add focused regression coverage, and provide redacted after-fix WebUI or log proof.

## Reproduction Assessment

Yes, source-level. On current main, an inline image plus an HTTP(S) image URL leaves the URL out of `detectImageReferences()`, so `detectAndLoadPromptImages()` cannot merge that remote image into the first native vision payload.

## Solution Assessment

No. The PR targets the right surface, but this implementation is not the best fix because it preloads too broadly and bypasses established remote media SSRF and sandbox contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Avoid fetching every HTTP URL as a prompt image:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and the new full-prompt scan turns each match into a `remote-url` ref before content-type filtering. A normal prompt that mentions a docs page or tracking link would now trigger an outbound gateway request, adding latency and privacy exposure outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy for remote images:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The `remote-url` branch calls `loadWebMedia()` with only `maxBytes`. Existing media tool paths pass `remoteMediaSsrfPolicy`, so configured allowlists or private-network controls would not be honored for native remote-image preloads.
  - confidence: 0.94
- **[P1] Keep sandboxed runs from preloading remote URLs:** `src/agents/pi-embedded-runner/run/images.ts:458-462`
  - body: This branch executes before the sandbox path-validation block and fetches HTTP(S) refs directly. The existing image tool rejects remote URLs under sandbox mode, so the PR changes sandboxed remote-image behavior for the new native preload path.
  - confidence: 0.9
- **[P2] Do not mark skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though no `remote-url` ref is emitted. If the same URL is repeated explicitly later, the prompt-wide scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef()` reuses `normalizeRefForDedupe()`, which lowercases keys on Windows for filesystem paths. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected HTTP image URLs:** `src/agents/pi-embedded-runner/run/images.ts:96`
  - body: The URL regex allows `)`, but the punctuation trimmer does not remove it. Parenthesized or Markdown-style links such as `(https://example.com/image.png)` are fetched as `image.png)`, causing avoidable native image-load failures.
  - confidence: 0.82

## Security Review
