## Label Justifications

- `P1`: This is a user-facing native image workflow fix, but the proposed patch has high-impact remote fetch and security-boundary blockers before merge.
- `merge-risk: 🚨 message-delivery`: The patch changes which image refs are placed into native model input and has narrower bugs that can still drop or suppress remote image refs.

- `merge-risk: 🚨 availability`: Fetching arbitrary prompt URLs before model submission can add avoidable network stalls, especially without the established remote-media read idle timeout.

## Summary

Keep open: the mixed inline-image plus HTTP(S) image URL bug remains present on current main and the latest release, but this PR is not merge-ready because the branch conflicts with current main and the new automatic URL fetch path bypasses existing remote-media guardrails.

## What This Changes

The PR teaches native prompt-image detection to treat HTTP(S) URLs as remote image refs, preload them with `loadWebMedia`, merge them with inline/offloaded images for vision models, and adds regression tests.

## Best Possible Solution

Port a narrow native remote-image preload to the current `embedded-agent-runner` path, limit it to explicit image URLs, forward the existing remote-media security/timeout options, cover mixed inline+URL behavior and guardrails in tests, and require after-fix WebUI proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` ignore HTTP(S) prompt refs in `detectImageReferences`, and the embedded runner uses that detector to assemble native prompt images. I did not run a live WebUI repro in this read-only review.

## Solution Assessment

No. The fix direction is useful, but this branch is not the best merge shape until it is ported to current main and reuses the existing remote-media security, timeout, and URL-scope contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict automatic URL preloads to image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches every HTTP(S) URL in the prompt, so normal links like docs pages are fetched by the gateway before the model call and then discarded if they are not images. Please only preload URLs that are explicitly image refs, or keep extensionless/ambiguous URLs on the existing tool path.
  - confidence: 0.91
- **[P1] Forward the SSRF policy on remote prompt images:** `src/agents/pi-embedded-runner/run/images.ts:442-444`
  - body: The new `remote-url` branch calls `loadWebMedia` with only `maxBytes`, unlike the existing image tool path that forwards `tools.web.fetch.ssrfPolicy` and the remote-media idle timeout. This bypasses operator-configured remote-media guardrails for automatically fetched prompt URLs.
  - confidence: 0.9
- **[P2] Do not consume bracket URLs that are not emitted:** `src/agents/pi-embedded-runner/run/images.ts:320-329`
  - body: When a `[media attached: path | url]` bracket has a local path, this code records the URL as seen without adding a `remote-url` ref. A later explicit repeat of the same URL in the prompt is then suppressed, so the remote image can still be missing from native input.
  - confidence: 0.82
- **[P2] Preserve case-sensitive remote URL dedupe keys:** `src/agents/pi-embedded-runner/run/images.ts:273-276`
  - body: `addRemoteUrlRef` reuses `normalizeRefForDedupe`, which lowercases keys on Windows for local filesystem paths. HTTP URL paths can be case-sensitive, so two distinct remote images that differ only by path casing can collapse into one native image on Windows.
  - confidence: 0.78
