## Summary

Keep this PR open, but do not merge it as-is: current main still has the mixed inline-plus-HTTP image gap, while the PR introduces security-sensitive remote fetch behavior and still lacks after-fix real behavior proof.

## What This Changes

The PR adds HTTP(S) prompt-image detection, remote preloading through `loadWebMedia()`, and focused regression tests so remote image URLs can be merged with inline images before native vision requests.

## Best Possible Solution

Rework native prompt-image preloading to accept only clearly intended remote image refs, preserve SSRF and sandbox contracts, add regression coverage, and provide redacted after-fix WebUI or runtime proof.

## Reproduction Assessment

Yes, source-level. Current main skips HTTP(S) refs in `detectImageReferences()`, so an inline image plus an HTTPS image URL leaves the URL image out of the first native vision payload.

## Solution Assessment

No. The PR changes the right surface, but this implementation is not the best fix because it preloads arbitrary HTTP(S) links, does not forward configured SSRF policy, and changes sandboxed remote-image behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restrict prompt URL preloads to image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and the full prompt scan preloads each match before discarding non-images. A normal vision prompt that mentions a docs page or tracking link would now trigger an outbound gateway request outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia()` with only `maxBytes`. Existing media tool paths pass the configured web fetch SSRF policy, so allowlists or private-network controls are not honored for native remote-image preloads.
  - confidence: 0.94
- **[P1] Keep sandboxed runs from fetching remote prompt URLs:** `src/agents/pi-embedded-runner/run/images.ts:458-462`
  - body: The `remote-url` branch runs before sandbox path validation and fetches HTTP(S) prompt refs even when a sandbox is active. That diverges from the existing image tool contract, which rejects remote URLs under sandbox mode.
  - confidence: 0.9
- **[P2] Do not mark skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it emits no `remote-url` ref. If the same URL appears explicitly later, the full-prompt scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef()` reuses filesystem dedupe normalization, which lowercases keys on Windows. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected HTTP URLs:** `src/agents/pi-embedded-runner/run/images.ts:96-118`
  - body: The URL regex allows `)`, but the trailing-punctuation trimmer does not remove it. Parenthesized or Markdown-style links such as `(https://example.com/image.png)` are fetched as `image.png)`, causing avoidable native image-load failures.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a prompt-triggered remote fetch path without preserving the existing media SSRF and sandbox protections.

Concerns:

- **[medium] Prompt text can trigger unintended outbound fetches:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: Because the new regex treats every HTTP(S) URL as an image candidate, ordinary links in vision-model prompts cause gateway network requests even when they are not image URLs.
  - confidence: 0.95
- **[medium] Remote image preload omits SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460`
  - body: The new `remote-url` loader calls `loadWebMedia()` without passing the configured web fetch SSRF policy, so native image preloads do not honor the same policy used by existing media tool paths.
  - confidence: 0.94
- **[medium] Sandboxed runs can fetch remote prompt URLs:** `src/agents/pi-embedded-runner/run/images.ts:458`
  - body: The remote-url branch executes before sandbox validation and does not reject HTTP(S) refs when `options.sandbox` is present, unlike the current image tool sandbox behavior.
