## Summary

Keep open. The PR targets a real mixed-image ingestion gap, but the current diff still expands prompt-triggered remote fetching, skips existing SSRF/sandbox contracts, and lacks after-fix real behavior proof.

## What This Changes

The PR adds HTTP(S) prompt-image URL detection/loading in `images.ts` plus regression coverage in `images.test.ts` so remote URLs can be merged with inline images for native vision input.

## Best Possible Solution

Keep the PR open and rework native remote-image preloading to honor existing media security contracts, then verify the WebUI mixed inline plus HTTP(S) flow with redacted after-fix proof.

## Reproduction Assessment

Yes, source-level. Current main explicitly skips `http://`/`https://` prompt refs in `detectImageReferences()`, and current tests still expect a local path plus HTTPS URL prompt to return only the local image ref.

## Solution Assessment

No. The PR changes the right ingestion surface, but the implementation should narrow remote preloads to intended image refs, thread existing SSRF policy, preserve sandbox behavior, and include after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restrict prompt URL preloads to image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches every HTTP(S) URL, and the prompt scan preloads each match before discarding non-images. A normal vision prompt that mentions a docs page or tracking link would now trigger an outbound gateway request outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia()` with only `maxBytes`. Existing media tool paths pass the configured web-fetch SSRF policy, so allowlists or private-network controls are not honored for native remote-image preloads.
  - confidence: 0.94
- **[P1] Preserve sandbox remote URL blocking:** `src/agents/pi-embedded-runner/run/images.ts:458-462`
  - body: The `remote-url` branch runs before sandbox path validation and fetches HTTP(S) prompt refs even when a sandbox is active. That diverges from the existing image tool contract, which rejects remote URLs under sandbox mode.
  - confidence: 0.9
- **[P2] Avoid marking skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it emits no `remote-url` ref. If the same URL appears explicitly later, the full-prompt scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef()` reuses filesystem dedupe normalization, which lowercases keys on Windows. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected HTTP URLs:** `src/agents/pi-embedded-runner/run/images.ts:96-118`
  - body: The URL regex allows `)`, but the trailing-punctuation trimmer does not remove it. Parenthesized or Markdown-style links such as `(https://example.com/image.png)` are fetched as `image.png)`, causing avoidable native image-load failures.
  - confidence: 0.82
