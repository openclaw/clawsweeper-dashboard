## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict URL preload to explicit image refs:** `src/agents/pi-embedded-runner/run/images.ts:42`
  - body: `HTTP_URL_REGEX_SOURCE` matches every HTTP(S) URL, and `detectAndLoadPromptImages` fetches each detected ref before the model call. A prompt that merely mentions a normal link now causes an outbound gateway fetch even when no image URL was intended, adding latency and exposing link reads; narrow detection to explicit image refs or a maintainer-approved marker before calling `loadWebMedia`.
  - confidence: 0.93
- **[P1] Forward the configured SSRF policy:** `src/agents/pi-embedded-runner/run/images.ts:460-462`
  - body: The new `remote-url` branch calls `loadWebMedia` with only `maxBytes`, while sibling media tools pass the configured `tools.web.fetch.ssrfPolicy`. Gateways with narrow SSRF exceptions or dispatcher policy will silently lose that policy for native prompt-image preloads; thread the policy from the runner call site and cover it with a focused test.
  - confidence: 0.94
- **[P2] Do not consume skipped bracket URL keys:** `src/agents/pi-embedded-runner/run/images.ts:340-348`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it emits no `remote-url` ref. If the same URL is also typed elsewhere in the prompt, the later full-prompt scan suppresses it, so the remote image can still be lost when the local path is unavailable.
  - confidence: 0.82
- **[P2] Use URL-specific dedupe for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:290-295`
  - body: `addRemoteUrlRef` reuses `normalizeRefForDedupe`, which lowercases keys on Windows for filesystem paths. HTTP path casing can be significant, so Windows gateways can collapse two distinct remote image URLs and drop one from the native image payload.
  - confidence: 0.78
