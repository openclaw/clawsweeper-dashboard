## Summary

Keep open. Current main still has no supported `tools.exec.shell` contract, and the open implementation PR needs maintainer review because this changes the command-execution policy boundary.

## What This Changes

Request to add a global and per-agent `tools.exec.shell` configuration option so exec can use WSL, bash, or another configured shell instead of only the built-in shell selection.

## Best Possible Solution

Define a supported exec shell override contract covering precedence, executable-plus-argument semantics, validation, Windows/WSL cwd and PATH behavior, approval/safe-bin interactions, docs/schema metadata, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.

## Reproduction Assessment

Yes for the requested capability: current main statically lacks `tools.exec.shell` in the shell helper, exec defaults, config type, strict schema, generated metadata, and docs. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear. The config direction is plausible, but a simple string passthrough is not enough for a command-execution boundary; maintainers should settle the shell argument, validation, WSL, PATH, cwd, and approval semantics before landing.
