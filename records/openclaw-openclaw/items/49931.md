## Summary

Keep this issue open. Current main still has no supported `tools.exec.shell` contract: shell selection is derived internally, the strict config type/schema omit the requested key, docs describe the existing `SHELL`/PowerShell behavior, and the related implementation PR is still open for maintainer review.

## What This Changes

The issue requests a `tools.exec.shell` configuration option so exec can use a configured shell such as WSL bash, Git Bash, or a specific POSIX shell instead of the platform default.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests before landing the existing PR or a replacement design.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `getShellConfig()` has no config input, the runtime calls it directly, strict config schemas omit `shell`, and the public exec docs do not document `tools.exec.shell`.

## Solution Assessment

Unclear. The requested config key is plausible, but the best implementation needs maintainer review because shell selection affects command parsing, policy prompts, PATH/cwd semantics, and Windows/WSL boundaries.
