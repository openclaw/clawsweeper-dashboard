## Summary

Keep this issue open: current main and v2026.6.8 still do not expose a supported `tools.exec.shell` config contract, and the open discussion correctly identifies unresolved config-stability, WSL, PATH, and safe-bin security questions.

## What This Changes

Request a configurable exec shell override, such as `tools.exec.shell`, so Windows users can run exec commands through WSL, bash, or another chosen shell instead of the built-in PowerShell default.

## Best Possible Solution

Define a maintainer-approved exec shell-selection contract, including arguments, validation timing, WSL path/PATH semantics, host/sandbox/node behavior, approval and safe-bin policy, docs, schema, and focused tests.

## Reproduction Assessment

Not applicable as a bug repro; this is a feature/config request. Source inspection gives high confidence that the requested capability is missing because current main has no `tools.exec.shell` schema/type/docs contract and the exec runtime calls `getShellConfig()` without config input.

## Solution Assessment

Unclear. A config-backed shell override is plausible, but the best fix needs maintainer decisions on shell arguments, WSL path handling, validation timing, sandbox/node behavior, and approval/safe-bin policy before implementation.
