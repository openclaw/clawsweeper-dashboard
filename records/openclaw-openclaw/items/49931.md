## Summary

Keep this issue open: current main and v2026.6.6 still do not expose a supported `tools.exec.shell` config contract, and the matching implementation attempts are closed unmerged while this issue remains the canonical tracker for the security-sensitive shell-selection decision.

## What This Changes

The issue requests a configurable `tools.exec.shell` setting so exec tool invocations can use a user-selected shell instead of the current platform default.

## Best Possible Solution

Define one maintainer-approved exec shell-selection contract, including shell arguments, global versus per-agent precedence, validation timing, WSL cwd/PATH expectations, approval/safe-bin policy, node/sandbox/PTY behavior, docs, schema/help metadata, and focused regression coverage.

## Reproduction Assessment

Yes for the missing supported behavior by source inspection, not as a bug repro. Current main has no `tools.exec.shell` type/schema/docs contract and the host/PTY exec paths call `getShellConfig()` without a configured shell value.

## Solution Assessment

Unclear. A config-backed override is plausible, but the best fix needs maintainer decisions on shell arguments, WSL path handling, validation timing, and approval/safe-bin policy before implementation.
