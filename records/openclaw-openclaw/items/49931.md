## Summary

Keep this open. Current main still does not expose `tools.exec.shell` as config, host exec still resolves the shell internally, and the related implementation PRs are closed rather than merged, so the remaining work needs maintainer product and security review instead of stale cleanup.

## What This Changes

Request adding a global or per-agent `tools.exec.shell` override so Windows and other hosts can run exec commands through a chosen shell such as WSL bash, Git Bash, bash, zsh, or PowerShell.

## Best Possible Solution

Define a maintainer-approved exec shell override contract that covers global/per-agent precedence, executable and argument semantics, validation timing, WSL cwd/PATH behavior, approval/safe-bin interactions, docs/schema metadata, and targeted Windows/POSIX tests.

## Reproduction Assessment

Yes, by source inspection: current main's config type and strict schema omit `tools.exec.shell`, while host exec still calls `getShellConfig()` without a configured override. I did not run a Windows/WSL live repro because the missing config contract is already clear in current source and docs.

## Solution Assessment

Unclear as an implementation direction: the user problem is real, but a raw string config passthrough is not enough for this security-sensitive exec boundary. The best path is a maintained contract that defines shell args, WSL path translation, validation, and approval/safe-bin semantics before implementation.
