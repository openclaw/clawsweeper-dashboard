## Summary

Keep open. Current main still lacks a supported `tools.exec.shell` contract: shell selection is derived internally, the exec config type and strict runtime schema do not accept the requested key, docs document only the existing `SHELL`/PowerShell behavior, and the related implementation PR #39413 is still open for maintainer review.

## What This Changes

The issue requests a configurable global and per-agent `tools.exec.shell` override so exec can use WSL bash, Git Bash, bash, zsh, pwsh, or another shell instead of OpenClaw's default shell selection.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable/argument semantics, startup or first-use validation, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, docs/schema metadata, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `tools.exec.shell` is absent from the strict schema/type surfaces, `getShellConfig()` has no config input, and the runtime calls that helper directly.

## Solution Assessment

Unclear. The requested config direction is plausible, but shell selection is a command-execution policy boundary, so the best implementation needs maintainer review rather than a simple string passthrough.
