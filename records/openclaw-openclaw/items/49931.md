## Summary

Keep open: current main and the latest release still lack a supported `tools.exec.shell` config contract, and the related implementation attempts are closed unmerged rather than shipped fixes.

## What This Changes

The issue requests a configurable exec shell override so users, especially on Windows, can route exec commands through WSL, bash, PowerShell variants, or another chosen shell.

## Best Possible Solution

Define one supported exec shell-selection contract covering precedence, executable and argument semantics, validation, Windows/WSL cwd and PATH behavior, approval and safe-bin interactions, gateway/node/sandbox/PTY behavior, schema/help/docs, and targeted Windows plus POSIX proof.

## Reproduction Assessment

Yes for the capability gap: source inspection shows no accepted `tools.exec.shell` key in the type/schema/docs and the host exec runtime still calls the default shell resolver directly. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear. A config-backed override is plausible, but a maintainable fix needs a full command-execution contract rather than a simple string passthrough into one spawn path.
