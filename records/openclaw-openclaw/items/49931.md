## Summary

Keep this issue open: current main and v2026.6.6 still do not expose a supported `tools.exec.shell` config contract, and the same remaining work is not owned by any merged or open viable fix PR.

## What This Changes

The issue requests a global/per-agent `tools.exec.shell` override so Windows users can run exec commands through WSL, Git Bash, or another selected shell instead of the default PowerShell path.

## Best Possible Solution

Define one maintainer-approved exec shell-selection contract, then update config types/schema/help/docs and thread it consistently through gateway, PTY, sandbox, node, approval, PATH, WSL, and safe-bin behavior with focused tests.

## Reproduction Assessment

Not applicable as a bug repro; this is a feature request. Source inspection gives high confidence that the requested capability is missing because current main calls `getShellConfig()` without a config value and the strict exec schema has no `shell` key.

## Solution Assessment

Unclear. A config-backed shell override is plausible, but the best fix needs maintainer decisions on shell arguments, WSL path handling, validation timing, sandbox/node behavior, and approval/safe-bin policy before implementation.
