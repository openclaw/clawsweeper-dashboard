## Summary

Keep open. Current main and the latest release still lack a supported `tools.exec.shell` contract, and the known implementation PRs for this behavior are closed unmerged, so this remains the canonical security-sensitive exec shell configuration request.

## What This Changes

Request adding a configurable `tools.exec.shell` override so exec can use WSL, Git Bash, bash, pwsh, or another shell instead of the current platform default.

## Best Possible Solution

Define one maintained exec shell-selection contract covering precedence, executable-plus-argument semantics, validation, Windows/WSL cwd and PATH behavior, approval and safe-bin interactions, docs/schema/help metadata, and focused Windows plus POSIX regression proof.

## Reproduction Assessment

Yes for the capability gap: source inspection shows current main has no accepted `tools.exec.shell` key and host exec still calls the default shell resolver directly. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear. A config-backed shell override is plausible, but the maintainable solution needs a full exec contract rather than a simple string passthrough into one spawn path.
