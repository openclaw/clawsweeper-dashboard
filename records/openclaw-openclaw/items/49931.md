## Summary

Keep open. Current main and v2026.5.12 still do not expose, accept, thread, validate, or document a `tools.exec.shell` contract, while the related implementation PRs are closed unmerged and the duplicate issue was closed in favor of this canonical tracker.

## What This Changes

Requests a `tools.exec.shell` config option so exec can use a configured shell, especially WSL/Git Bash instead of PowerShell on Windows.

## Best Possible Solution

Define a supported exec shell-selection contract covering precedence, executable-plus-arguments semantics, validation timing, Windows/WSL cwd and PATH behavior, approval and safe-bin policy, docs/schema/help metadata, and targeted Windows plus POSIX regression tests.

## Reproduction Assessment

Yes for the capability gap: current source inspection shows there is no accepted `tools.exec.shell` key and host exec still calls the default shell resolver directly. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear. A config-backed override is plausible, but the maintainable solution needs a complete exec contract rather than a simple string passthrough into one spawn path.
