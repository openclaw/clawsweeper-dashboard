## Summary

Keep open. Current main still lacks a supported `tools.exec.shell` contract, and the related implementation attempts are closed unmerged, so this remains the canonical feature/config request for maintainer design review.

## What This Changes

Request a global or per-agent `tools.exec.shell` config option to override the exec shell, especially for Windows users who want WSL, Git Bash, or another bash-like shell instead of PowerShell.

## Best Possible Solution

Define a supported exec shell override contract covering precedence, executable-plus-argument semantics, validation, Windows/WSL cwd and PATH behavior, approval/safe-bin interactions, docs/schema metadata, and targeted exec tests before landing an implementation.

## Reproduction Assessment

Yes for the requested capability: source tracing shows current main has no accepted `tools.exec.shell` key and host exec still calls the default shell resolver directly. This is a feature/config gap rather than a broken existing contract.

## Solution Assessment

Unclear. The config direction is plausible, but the maintainable solution needs a full command-execution contract rather than a narrow string passthrough.
