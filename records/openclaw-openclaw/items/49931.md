Keep open. Current main and the latest shipped release still do not provide a supported `tools.exec.shell` contract, and the known implementation PRs for this capability are closed unmerged, so this remains the canonical product/security design item rather than cleanup.

## What This Changes

The issue requests a configurable exec shell override, such as `tools.exec.shell`, so users can choose WSL, bash, PowerShell, or another shell for exec invocations.

## Best Possible Solution

Define a supported exec shell-selection contract covering precedence, executable-plus-argument semantics, validation, Windows/WSL cwd and PATH behavior, approval and safe-bin interactions, gateway/node/sandbox/PTY behavior, schema/help/docs, and targeted Windows plus POSIX proof.

## Solution Assessment

Unclear. A config-backed shell override is plausible, but the maintainable solution needs a full exec contract rather than a simple string passthrough into one spawn path.
