## Summary

Keep open. Current main and the latest release still do not expose, validate, thread, or document a `tools.exec.shell` contract, and the known implementation PRs are closed unmerged, so this remains the canonical exec-shell configuration feature request.

## What This Changes

The issue requests a supported `tools.exec.shell` configuration option for overriding the shell used by exec tool invocations.

## Best Possible Solution

Define a maintained exec shell-selection contract covering precedence, executable-plus-argument semantics, validation timing, Windows/WSL cwd and PATH behavior, approval and safe-bin interactions, docs/schema/help metadata, and targeted Windows plus POSIX regression tests.

## Reproduction Assessment

Yes for the capability gap: source inspection shows current main has no accepted `tools.exec.shell` key and host exec still calls the default shell resolver directly. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear. A config-backed shell override is plausible, but the maintainable solution needs a full exec contract rather than a simple string passthrough into one spawn path.
