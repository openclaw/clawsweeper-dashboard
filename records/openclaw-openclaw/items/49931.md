## Summary

Keep this issue open: current main and v2026.6.6 still do not expose a supported `tools.exec.shell` contract, while the earlier implementation attempts were closed unmerged because the exec security and shell semantics need maintainer direction.

## What This Changes

Requests a global or per-agent `tools.exec.shell` config option so exec can use WSL, Git Bash, bash, or pwsh instead of the platform default shell.

## Best Possible Solution

Define one supported exec shell-selection contract, update config types/schema/help/docs, thread it through all host exec paths consistently, and add focused tests for Windows, WSL-like behavior, PATH, approval, and safe-bin interactions.

## Reproduction Assessment

Yes for the missing behavior by source inspection, not as a bug repro. Current main calls `getShellConfig()` without a configured value and the strict exec config schema has no `shell` key.

## Solution Assessment

Unclear. A config-backed override is plausible, but the best fix needs maintainer decisions on shell arguments, WSL path handling, validation timing, and approval/safe-bin policy before implementation.
