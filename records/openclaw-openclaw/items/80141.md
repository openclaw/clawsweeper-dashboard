## Summary

Keep open: current main still has the approval presentation bug, but this PR is not ready because it can double-summarize native prompt text and the supplied proof does not show the actual visible UI or chat prompt.

## What This Changes

The PR adds long exec approval prompt summarization, carries allowed approval decisions into the Control UI parser/modal, updates docs/changelog, and adds focused approval tests.

## Best Possible Solution

Keep the PR open, apply summarization exactly once per human-facing surface, preserve the full sanitized command for matching/storage, honor allowed decisions in UI actions, and require visible modal or native chat proof before merge.

## Reproduction Assessment

Yes, from source. Current main broadcasts allowedDecisions but the Control UI parser drops them, the modal renders all actions, and long request.command text is rendered without a prompt-level summary.

## Solution Assessment

No, not as-is. The direction is maintainable, but the PR should make summarization single-boundary or idempotent and add visible proof from the actual approval surface before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid re-summarizing native approval text:** `src/infra/exec-approval-reply.ts:306-309`
  - body: Matrix and Telegram native adapters pass `view.commandText` into `buildExecApprovalPendingReplyPayload`; after this PR, `view.commandText` is already summarized by the view model, and this helper summarizes it again. Long native prompts can therefore drop the original truncation marker and report a misleading smaller hidden line count, so summarize at only one boundary or make the helper idempotent.
  - confidence: 0.83
