## Summary

Keep open: the PR targets a real approval UX bug, but the current patch can re-summarize already summarized native chat approval text, and the supplied proof does not show the actual Control UI or native chat behavior after the fix.

## What This Changes

Adds display-only exec approval command summarization, carries allowedDecisions through the Control UI approval parser, updates docs/changelog, and adds focused approval/UI tests.

## Best Possible Solution

Apply summarization exactly once per human-facing approval surface, preserve the full sanitized command only for matching/storage, honor gateway-provided allowed decisions in UI actions, and add visible proof for the modal/native prompt path.

## Reproduction Assessment

Yes, from source. Current main broadcasts allowedDecisions but the Control UI parser drops them and the modal always renders all actions, while long request.command text is rendered without any prompt-level summary.

## Solution Assessment

No. The direction is right, but the patch should make summarization single-boundary or idempotent before merge and should include proof from the actual visible approval surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid re-summarizing native approval text:** `src/infra/exec-approval-reply.ts:306-309`
  - body: Matrix and Telegram native adapters pass `view.commandText` into `buildExecApprovalPendingReplyPayload`; after this PR, `view.commandText` is already summarized by the view model, and this helper summarizes it again. Long native prompts can therefore drop the original truncation marker and report a misleading smaller hidden line count, so summarize at only one boundary or make the helper idempotent.
  - confidence: 0.83
