## Summary

Keep this PR open: it addresses a real approval UX bug and has credible terminal proof, but the patch introduces a double-summarization path for native approval delivery that can distort the hidden-command metadata shown to operators.

## What This Changes

The PR adds an exec approval command summarizer, applies it to Control UI and shared approval presentation surfaces, preserves gateway allowed decisions in the Control UI parser, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the presentation-boundary summarizer and allowed-decision propagation, but make summarization single-pass or idempotent across native/shared payload builders while preserving full sanitized pending records for matching.

## Reproduction Assessment

Yes, by source inspection: current main renders the full Control UI command and ignores gateway-provided allowed decisions, while the gateway already supplies them. I did not run a live gateway approval in this read-only review.

## Solution Assessment

No, not as-is. The direction is maintainable, but the native delivery path should avoid re-summarizing already-summarized view text before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid re-summarizing native approval payloads:** `src/infra/exec-approval-reply.ts:306-309`
  - body: `buildPendingApprovalView` now stores a summarized `view.commandText`, and Matrix/Telegram pass that value into `buildExecApprovalPendingReplyPayload`. Summarizing `params.command` again here treats the first truncation marker as another hidden line, so native approval text can report a new `showing first 5 of 6 lines` marker instead of the original hidden-line/char counts. Make this helper idempotent or ensure native payload builders pass unsummarized display text into the shared builder.
  - confidence: 0.86
