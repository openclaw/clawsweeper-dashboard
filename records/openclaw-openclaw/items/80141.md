## Summary

Keep open: the PR targets a valid approval UX bug, but the branch still has a concrete double-summarization defect in native approval prompts and the supplied proof does not show the actual Control UI modal or native chat surface.

## What This Changes

The PR adds display-time summarization for long exec approval prompts, propagates allowed approval decisions into the Control UI modal, and updates related docs, changelog, and approval tests.

## Best Possible Solution

Keep the presentation-boundary summarizer and allowed-decision propagation, but make summarization single-pass or idempotent across native/shared payload builders while preserving the full sanitized pending record for matching.

## Reproduction Assessment

Yes, from source. Current main broadcasts allowed decisions but the Control UI parser drops them, the modal renders all actions, and long command text is rendered without a prompt-level summary.

## Solution Assessment

No, not as-is. The direction is maintainable, but the patch should avoid summarizing already-summarized native view text before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid re-summarizing native approval payloads:** `src/infra/exec-approval-reply.ts:309-311`
  - body: `buildPendingApprovalView` now stores a summarized `view.commandText`, and Matrix/Telegram pass that value into `buildExecApprovalPendingReplyPayload`. Summarizing `params.command` again here can turn the first truncation marker into another hidden-line/char count, so native prompts can understate what was hidden. Make this helper idempotent or ensure it only receives unsummarized display text.
  - confidence: 0.86
