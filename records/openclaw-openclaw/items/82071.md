## Summary

Keep open: current main still has a plausible source path for the reported memory-only write error to escape a memory-flush turn, and the report is fresh rather than stale or superseded. The exact concurrent task-file race was not reproduced, so this needs owner review before changing the intentional memory-flush error-surfacing behavior.

## What This Changes

The issue reports that automatic memory flush can transiently reject writes to unrelated workspace task or skill files with the memory-only write restriction error.

## Best Possible Solution

Preserve the append-only memory-file guard, but ensure a memory-flush attempt cannot misattribute or abort an unrelated normal workspace write; add focused regression coverage for the reported interleaving.

## Reproduction Assessment

No, there is not yet a high-confidence live reproduction of the concurrent task-file write. Source inspection does show the exact restriction error is produced whenever a memory-triggered embedded run writes outside its single `memory/YYYY-MM-DD.md` target.

## Solution Assessment

Unclear. The append-only guard is the right safety boundary for memory-flush turns, but the best fix likely needs to isolate or classify those failures without weakening the guard or regressing the intentional visible-error diagnostics.
