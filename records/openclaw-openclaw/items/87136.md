## Label Justifications

- `P2`: The issue can make mixed-context-window sessions unreliable, but it is a bounded compaction config/runtime improvement rather than an emergency outage.
- `impact:session-state`: Compaction thresholds control when session context is flushed, summarized, and retained.
- `impact:auth-provider`: The reported failure mode is triggered by switching active provider/model choices with different context windows.

## Summary

Keep open: current main and v2026.6.8 still expose the reported compaction thresholds as numeric token counts only, and this issue is already paired with open implementation candidates that need maintainer API-shape review.

## What This Changes

The issue requests context-window-relative compaction thresholds so sessions can switch between models with different context windows without manual JSON retuning.

## Best Possible Solution

Keep this as the canonical issue, choose one backward-compatible compaction config contract, then land one implementation covering schema, types, all runtime readers, docs/metadata, tests, and real behavior proof.

## Reproduction Assessment

Yes at source level: current main validates and consumes the reported thresholds as numeric token counts, and the preflight threshold is derived from the active context window minus absolute budgets. I did not run a live gateway session switching between 200K and 1M-token models.

## Solution Assessment

Unclear: percentage strings and additive share fields are both plausible, but this is a public config/API decision and needs one maintainer-approved shape before implementation is best.
