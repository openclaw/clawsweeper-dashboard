## Summary

Keep open: current main and v2026.6.8 still expose the reported compaction thresholds as numeric token counts only, and open linked PRs propose competing public config shapes that need maintainer selection before this can close.

## What This Changes

Request context-window-relative compaction thresholds so reserve, keep-recent, reserve-floor, and memory-flush token budgets scale when sessions switch between models with different context windows.

## Best Possible Solution

Keep this as the canonical issue, choose one backward-compatible compaction config contract, then land a single implementation covering schema, types, runtime readers, docs/help metadata, tests, and real mixed-context proof.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path to reproduce the missing relative-threshold capability because current schema, types, docs, and runtime readers are numeric-only. I did not run a live gateway session switching between 200K and 1M-token models.

## Solution Assessment

Unclear: percentage strings and additive share fields are both plausible, but this is a public config/API decision and needs one maintainer-approved shape before implementation is best.
