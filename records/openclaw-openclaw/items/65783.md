## Summary

Keep open. The reported memory-indexing bug is valid on current main, and this PR is close, but it still leaves provider-bound text unsanitized in remote batch embedding and structured text parts.

## What This Changes

The PR changes memory chunking to keep UTF-16 surrogate pairs intact, adds an embed-boundary sanitizer for unpaired surrogates, and adds focused tests.

## Best Possible Solution

Land a refreshed patch that guards the coarse chunk boundary and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse split at maxChars=40 with 39 ASCII characters plus a flower emoji yields a chunk ending with high surrogate d83c and the next starting with low surrogate df38.

## Solution Assessment

No, not yet. The chunker guard and inline sanitizer are the right direction, but the PR must also cover remote batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize remote batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes `missingChunks` directly to `providerRuntime.batchEmbed`. OpenAI and Voyage batch adapters serialize `chunk.text` into request bodies, so remote batch mode can still send a lone surrogate to the embedding API.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-295`
  - body: `sanitizedInputs` replaces only `input.text`. When `input.parts` contains a text part, Gemini serializes `part.text`, so a lone surrogate in structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
