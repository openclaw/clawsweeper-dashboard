## Summary

Keep open: the bug is real on current main and this PR is useful, but the patch is incomplete because provider-bound text can still bypass the new sanitizer through remote batch chunks and structured text parts.

## What This Changes

The PR changes memory chunking to avoid splitting UTF-16 surrogate pairs, adds an embedding-input sanitizer, and adds focused regression tests.

## Best Possible Solution

Land a rebased or replacement fix that preserves surrogate pairs in chunking and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and inline sanitizer are the right direction, but the patch must also sanitize remote batch chunks and structured text parts to cover the embedding boundary it claims to protect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize chunks before remote batch dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes missingChunks directly to providerRuntime.batchEmbed. OpenAI and Voyage batch adapters serialize chunk.text into request bodies, so batch-enabled indexing can still send a lone surrogate to the embedding API and fail the same way.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-295`
  - body: sanitizedInputs replaces only input.text. When input.parts contains a text part, Gemini serializes part.text, so a lone surrogate in structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
