## Summary

Keep open. This is a plausible focused memory-indexing bug fix with live proof, but the PR still leaves structured embedding text parts unsanitized, so provider-visible `parts[].text` can still carry lone surrogates into embedding providers.

## What This Changes

The PR updates memory chunking to avoid splitting UTF-16 surrogate pairs, adds an embedding-input lone-surrogate sanitizer, and adds focused chunker and sanitizer tests.

## Best Possible Solution

Land a repaired version that fixes the coarse chunk boundary and sanitizes every text-bearing embedding input, including `parts[].text`, with focused regression coverage for the chunker, sanitizer, and Gemini structured requests.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop slices at fixed UTF-16 offsets, and a focused Node probe shows the reported 39-ASCII-plus-emoji case splits the surrogate pair exactly as described.

## Solution Assessment

No, not as submitted. The chunker fix and top-level sanitizer are the right direction, but the structured embedding path also needs to sanitize text parts before providers serialize them.
