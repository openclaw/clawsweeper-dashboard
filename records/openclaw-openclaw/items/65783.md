## Summary

Keep open: this PR targets a real, source-proven memory indexing bug, but the current diff still leaves provider batch embedding and structured text-part embedding paths able to send lone surrogates.

## What This Changes

The PR updates memory chunking, embedding retry wrappers, and focused tests to preserve UTF-16 surrogate pairs and replace unpaired surrogates before embedding.

## Best Possible Solution

Land a rebased or replacement patch that guards coarse chunk boundaries and sanitizes every provider-bound text field before inline, structured, and batch embedding calls, with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and top-level sanitizer are the right direction, but this PR must also sanitize provider runtime batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batchEmbed chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes missingChunks directly to providerRuntime.batchEmbed. OpenAI and Voyage batch adapters serialize chunk.text into request bodies, so batch mode can still send a lone surrogate and fail the reindex.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:295`
  - body: sanitizedInputs replaces only input.text. If input.parts contains a text part, Gemini serializes part.text and bypasses the sanitizer, leaving the same failure mode for structured embedding inputs.
  - confidence: 0.9
