## Summary

Keep open. The bug is source-reproducible and the PR is useful, but current main still lacks the fix and this PR still leaves remote batch and structured text-part embedding paths able to send unpaired surrogates.

## What This Changes

The PR updates memory chunking to avoid splitting UTF-16 surrogate pairs, adds an embedding-input sanitizer, and adds focused tests for both paths.

## Best Possible Solution

Land a rebased or replacement patch that preserves surrogate pairs in chunking and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and top-level retry-wrapper sanitizer are the right direction, but the PR must also cover remote batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes cached-missing chunks directly to providerRuntime.batchEmbed. OpenAI and Voyage batch adapters serialize chunk.text into request bodies, so remote batch mode can still send a lone surrogate to the embedding API.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-295`
  - body: sanitizedInputs replaces only input.text. When input.parts contains a text part, Gemini serializes part.text, so a lone surrogate in structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
