## Summary

Keep open: this is a valid memory indexing bug and the chunker fix is useful, but the PR still misses sanitizer coverage for provider batch mode and structured text parts, so it does not fully protect the embedding boundary.

## What This Changes

The branch adjusts memory chunk splitting, adds an unpaired-surrogate sanitizer around embedding calls, and adds focused chunker and sanitizer tests.

## Best Possible Solution

Land a repaired current-main patch that preserves surrogate pairs at coarse chunk boundaries and sanitizes every outbound embedding text field across inline, structured, and provider batch paths with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's coarse split can bisect 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and top-level sanitizer are the right direction, but the PR must also sanitize provider batchEmbed chunks and structured text parts to close the embedding-boundary hole.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batchEmbed chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:239-241`
  - body: The new sanitizer only runs in the inline retry wrappers, but remote batch indexing still sends missingChunks directly to providerRuntime.batchEmbed. OpenAI and Voyage batch adapters serialize chunk.text into batch payloads, so batch mode can still send a lone surrogate and fail the reindex.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-296`
  - body: This sanitizes only input.text before calling embedBatchInputs. If input.parts contains a text part, Gemini serializes part.text and bypasses the sanitizer, preserving the same provider failure mode for structured inputs.
  - confidence: 0.9
