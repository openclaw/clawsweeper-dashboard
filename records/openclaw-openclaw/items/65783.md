## Summary

Keep open. Current main still has the surrogate-splitting memory bug, and this PR is useful but still incomplete because two provider-bound embedding paths can bypass the new sanitizer before merge.

## What This Changes

The PR adds a surrogate-pair-aware coarse split in `chunkMarkdown`, a lone-surrogate sanitizer at memory embedding retry entry points, and focused regression tests.

## Best Possible Solution

Land a rebased or replacement patch that guards the coarse chunk boundary and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into chunks ending and starting with lone surrogate halves before embedding.

## Solution Assessment

No, not as-is. The chunker guard and top-level retry-wrapper sanitizer are the right direction, but the patch must also sanitize remote batch chunks and structured `part.text` values before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize remote batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259-269`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes cached-missing chunks directly to `providerRuntime.batchEmbed`. OpenAI and Gemini batch adapters serialize `chunk.text` or `chunk.embeddingInput` into provider request bodies, so remote batch mode can still send a lone surrogate and fail the reindex.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-296`
  - body: `sanitizedInputs` replaces only `input.text`. When `input.parts` contains a text part, Gemini serializes `part.text`, so a lone surrogate in structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
