## Summary

Keep open: current main still has the coarse UTF-16 split and unsanitized embedding handoff, and this PR is useful but incomplete because remote batch and structured text paths can still send lone surrogates.

## What This Changes

The PR updates memory chunking to avoid coarse UTF-16 surrogate splits, adds a lone-surrogate sanitizer around embedding inputs, and adds regression tests.

## Best Possible Solution

Land a refreshed patch that guards the coarse chunk boundary and applies one sanitizer to every provider-bound text field across inline, structured, and remote batch embedding paths.

## Reproduction Assessment

Yes, at source level. Current main's fixed-step coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into chunks ending and starting with lone surrogate halves; I confirmed that with a read-only Node probe matching the current loop.

## Solution Assessment

No, not as currently patched. The chunker guard is the right direction, but the sanitizer must also cover remote batch chunks and structured text parts before this is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize remote batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:256`
  - body: The new sanitizer only wraps the inline provider.embedBatch path. When remote batch mode is enabled, embedChunksWithBatch still passes missingChunks to providerRuntime.batchEmbed, and OpenAI/Voyage adapters serialize chunk.text directly, so a lone surrogate can still reach the provider and fail the indexing run.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:295`
  - body: sanitizedInputs replaces only input.text. Gemini request construction prefers parts[].text when parts exist, so structured memory inputs can bypass the sanitizer and hit the same provider-side Unicode failure.
  - confidence: 0.9
