## Summary

Keep open. The PR addresses a real memory indexing bug and includes useful live proof, but it is not correct as submitted because the structured embedding path still leaves `parts[].text` unsanitized for providers such as Gemini.

## What This Changes

The PR updates memory chunking, embedding input sanitization, and regression tests to preserve UTF-16 surrogate pairs and replace unpaired surrogates before embedding calls.

## Best Possible Solution

Land a current-main patch that fixes the coarse chunk boundary, sanitizes every text-bearing embedding field including structured `parts[].text`, and keeps retry/checkpointing work in the existing follow-up reports.

## Reproduction Assessment

Yes, at source level. Current main still has the fixed-width coarse slice, and the `tokens=10` / `39 ASCII + flower emoji + 39 ASCII` case deterministically places the cut between `\uD83C` and `\uDF38`.

## Solution Assessment

No, not as submitted. The chunker and top-level text sanitizer direction is right, but structured embedding inputs must also sanitize `parts[].text` before providers build requests from those parts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
