## Summary

Keep open: the PR targets a real, source-reproducible memory indexing bug, but the latest diff still sanitizes only top-level structured input text while Gemini consumes text from `parts` when they are present.

## What This Changes

The branch changes memory chunking to avoid splitting UTF-16 surrogate pairs, adds embed-input surrogate sanitization, and adds focused memory-core and memory-host tests.

## Best Possible Solution

Land a current-main patch that preserves surrogate pairs at the coarse chunk boundary and sanitizes every text-bearing embedding field, including structured `parts[].text`, while leaving broader retry and checkpointing policy to the existing follow-up issues.

## Reproduction Assessment

Yes, at source level. With `maxChars=40`, `39` ASCII characters followed by a surrogate-pair emoji makes the current coarse slice boundary land between the high and low surrogate before any guard runs.

## Solution Assessment

No, not as submitted. The chunker fix and embed-boundary sanitizer are the right shape, but the PR must also sanitize structured `parts[].text` before providers such as Gemini build their request bodies.
