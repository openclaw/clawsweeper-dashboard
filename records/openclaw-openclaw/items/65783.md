## Summary

Keep open. The chunker guard is the right root-cause fix, but the PR still leaves provider-bound text unsanitized in remote batch chunks and structured text parts, so it does not fully close the reported memory indexing failure.

## What This Changes

The PR adds a coarse surrogate-pair guard to memory chunking, a lone-surrogate sanitizer in memory embedding retry wrappers, and focused regression tests.

## Best Possible Solution

Land a rebased or replacement patch that guards coarse chunk boundaries and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls, with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's coarse loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and top-level sanitizer are the right direction, but the PR must also sanitize remote batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batch chunks before remote dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrapper, but remote batch indexing still sends missingChunks directly to providerRuntime.batchEmbed. OpenAI and Voyage batch adapters serialize chunk.text into request bodies, so batch mode can still send a lone surrogate to the provider.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:295`
  - body: sanitizedInputs replaces only input.text. When input.parts contains a text part, Gemini serializes part.text instead, so structured memory inputs can still bypass the sanitizer and hit the same provider-side encoding failure.
  - confidence: 0.9
