## Summary

Keep open. This is a valid memory-indexing bug-fix PR with useful live proof, but the current head still leaves two provider-bound embedding paths unsanitized and the branch is currently conflicting with main.

## What This Changes

The PR updates memory chunking to avoid splitting UTF-16 surrogate pairs, adds an unpaired-surrogate sanitizer around embedding calls, and adds focused regression tests.

## Best Possible Solution

Land a refreshed patch that fixes the coarse chunk boundary and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse split at `maxChars=40` with 39 ASCII characters plus a flower emoji yields one chunk ending with high surrogate `d83c` and the next starting with low surrogate `df38`.

## Solution Assessment

No, not yet. The chunker guard and inline sanitizer are the right direction, but the PR must also sanitize remote batch chunks and structured text parts to fully protect the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize remote batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:201`
  - body: The new sanitizer only runs through inline retry wrappers, but remote batch mode still sends `missingChunks` unchanged to `providerRuntime.batchEmbed`. OpenAI and Voyage build request bodies from `chunk.text`, so a lone surrogate can still reach the provider when remote batching is enabled.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-296`
  - body: `sanitizedInputs` replaces only `input.text`. For structured inputs with `parts`, Gemini serializes text parts from `part.text`, so a text part containing a lone surrogate bypasses the new sanitizer and can still fail provider encoding.
  - confidence: 0.9
