## Summary

Keep open: the PR addresses a real memory-indexing bug that is still present on current main, but the patch is not complete because provider-bound remote batch chunks and structured text parts can still bypass the new sanitizer.

## What This Changes

The PR updates memory chunking to preserve UTF-16 surrogate pairs, adds an unpaired-surrogate sanitizer at embedding boundaries, and adds focused regression tests.

## Best Possible Solution

Land a refreshed patch that fixes the coarse chunk boundary and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse split at maxChars=40 with 39 ASCII characters plus a flower emoji yields a chunk ending with high surrogate d83c and the next chunk starting with low surrogate df38.

## Solution Assessment

No, not yet. The chunker guard and inline sanitizer are the right direction, but the PR must also cover remote batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize remote batch chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs inside `embedBatchWithRetry`, but remote batch mode still passes `missingChunks` directly to `providerRuntime.batchEmbed`. OpenAI and Voyage serialize `chunk.text`, so a lone surrogate can still reach the embedding provider.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-295`
  - body: `sanitizedInputs` only replaces `input.text`. When `input.parts` contains a text part, Gemini builds the request from `part.text`, so structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
