## Summary

Keep open: this is a focused fix for a real current-main memory indexing bug, but the PR still leaves provider-bound text unsanitized in remote batch mode and structured text parts.

## What This Changes

The PR adds a UTF-16 surrogate-pair guard to memory chunking, adds an embedding-input sanitizer, and adds focused tests for both paths.

## Best Possible Solution

Land a rebased or replacement patch that preserves surrogate pairs in chunking and sanitizes every provider-bound text field before inline, structured, and remote batch embedding calls.

## Reproduction Assessment

Yes, at source level. Current main's coarse chunk loop can split 39 ASCII characters plus a flower emoji at maxChars=40 into one chunk ending with a lone high surrogate and the next starting with a lone low surrogate.

## Solution Assessment

No, not yet. The chunker guard and top-level retry-wrapper sanitizer are the right direction, but the patch must also sanitize remote batch chunks and structured text parts before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batchEmbed chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:259`
  - body: The new sanitizer only runs through the inline retry wrappers, but remote batch indexing still passes cached-missing chunks directly to `providerRuntime.batchEmbed`. OpenAI and Voyage batch adapters serialize `chunk.text` into provider request bodies, so remote batch mode can still send a lone surrogate and fail the reindex.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:295`
  - body: `sanitizedInputs` replaces only `input.text`. When `input.parts` contains a text part, Gemini serializes `part.text`, so a lone surrogate in structured memory input can bypass the sanitizer and hit the same provider failure.
  - confidence: 0.9
