## Summary

Keep open: the bug is valid and current main still has the coarse chunk split/raw embedding gap, but this PR does not fully close the embedding boundary because remote batch embedding and structured text parts can still send lone surrogates.

## What This Changes

The PR changes memory chunking, embedding retry wrappers, and focused tests to keep UTF-16 surrogate pairs intact and replace unpaired surrogates before embedding.

## Best Possible Solution

Land a repaired current-main patch that guards coarse chunk boundaries and sanitizes all embedding text across inline, structured, and provider batch paths with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's coarse split can bisect 39 ASCII characters plus a flower emoji at maxChars=40 into lone high and low surrogate chunks before embedding.

## Solution Assessment

No, not yet. The chunker guard and top-level sanitizer are the right direction, but this PR must also sanitize structured text parts and provider runtime batchEmbed payloads before it fully fixes the embedding boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Sanitize batchEmbed chunks before dispatch:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:201`
  - body: The new sanitizer only runs in the inline retry wrappers, but remote batch indexing still sends `missingChunks` directly to `providerRuntime.batchEmbed`. OpenAI and Voyage batch adapters serialize `chunk.text` into batch payloads, so a lone surrogate can still reach the provider and fail the reindex when batch mode is enabled.
  - confidence: 0.91
- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-296`
  - body: This maps `stripUnpairedSurrogates` over only `input.text`; if `input.parts` contains a text part, providers such as Gemini serialize `part.text` and bypass the sanitizer. Sanitize text parts as well so structured inputs cannot keep the same failure mode.
  - confidence: 0.9
