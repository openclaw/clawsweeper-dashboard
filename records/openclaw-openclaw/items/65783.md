## Summary

Keep open. The PR is a useful fix for a real memory indexing bug, but it is incomplete because structured `EmbeddingInput.parts[].text` still bypasses the new sanitizer while Gemini serializes `parts` when present; the branch is also currently conflicting against main.

## What This Changes

The PR updates memory chunking and embedding retry wrappers to avoid lone UTF-16 surrogates, adds a `stripUnpairedSurrogates` helper, and adds focused tests.

## Best Possible Solution

Land a current-main patch that preserves surrogate pairs at the coarse chunk boundary and sanitizes every text-bearing embedding field, including structured `parts[].text`, while leaving retry and checkpointing policy to existing follow-ups.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main can split a mostly-ASCII line at `maxChars=40`, so `39` ASCII characters followed by a surrogate-pair emoji puts the coarse boundary between the high and low surrogate before any guard runs.

## Solution Assessment

No, not as submitted. The chunker fix and top-level embed-boundary sanitizer are the right direction, but structured `parts[].text` also needs sanitization before providers such as Gemini serialize it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Sanitize structured text parts before embedding:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-296`
  - body: The new `sanitizedInputs` mapping only rewrites top-level `input.text`. When an input has `parts`, Gemini builds the request from `params.input.parts` and ignores the sanitized top-level text, so a multimodal chunk with a lone surrogate in its text part can still reach the provider and fail indexing.
