## Summary

Keep open: current main still lacks Plugin SDK PID-start disambiguation, but this branch now conflicts with the fs-safe-backed lock wrapper and the external PR only shows unit/CI proof, not real behavior proof.

## What This Changes

The PR adds optional Linux process start-time metadata to Plugin SDK lock payloads, treats live PID/start-time mismatches as stale, and adds unit coverage for persistence, mismatch recovery, legacy fallback, and spy cleanup.

## Best Possible Solution

Port the start-time identity check onto the current fs-safe-backed Plugin SDK lock wrapper using the shared PID helper, preserve legacy and non-Linux fallback behavior, and require redacted runtime proof that a same-PID stale lock recovers.

## Reproduction Assessment

Yes. A high-confidence source/unit reproduction is to precreate a fresh lock with the current PID and a mismatched Linux starttime; current main ignores that field and treats the live PID as non-stale until createdAt age, though I did not run a live container PID-1 repro.

## Solution Assessment

The code direction is sound, but this exact branch is no longer the best mergeable form because current main moved Plugin SDK locking behind @openclaw/fs-safe. The safer path is to port the same starttime contract through the current wrapper and account for fs-safe's fail-closed stale contract.
