## Summary

Keep open: the root lock-recovery bug still appears present on current main, but this PR targets the pre-fs-safe implementation and lacks after-fix real behavior proof, so it needs a rebase/port rather than cleanup closure.

## What This Changes

The PR adds optional Linux process start-time metadata to Plugin SDK lock files, treats live PID/start-time mismatches as stale, and adds unit tests for persistence, legacy fallback, and recovery.

## Best Possible Solution

Port the start-time identity check onto the current fs-safe-backed Plugin SDK lock wrapper, add safe file_lock_stale recovery modeled on session-write-lock, preserve legacy/non-Linux fallback behavior, and require redacted runtime proof for same-PID stale-lock recovery.

## Reproduction Assessment

Yes. A high-confidence source/unit reproduction is a fresh lock payload with the current PID and a mismatched Linux starttime: current main ignores starttime and treats the live PID as non-stale until createdAt ages out, though no live PID-1 container repro was run here.

## Solution Assessment

No as-is. The code direction is sound, but the PR must be ported to the current fs-safe wrapper and handle fs-safe's fail-closed stale contract before it is the narrow maintainable fix.
