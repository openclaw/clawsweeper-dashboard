## Summary

Keep open. Current main still lacks process-start-time disambiguation for the Plugin SDK file lock, and the PR is a narrow implementation that matches adjacent lock hardening with focused tests. I found no functional or security regression in the patch, but the user-visible lock recovery fix is missing the required changelog entry before merge.

## What This Changes

The PR adds optional Linux process starttime persistence and comparison to `src/plugin-sdk/file-lock.ts` and extends `src/plugin-sdk/file-lock.test.ts` with writer, stale-recovery, legacy-fallback, and retry-recovery coverage.

## Best Possible Solution

Add the missing Unreleased changelog entry, then land the narrow Plugin SDK lock hardening with the payload extension kept optional/internal and the raw `/proc` starttime comparison preserved.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to precreate a `.lock` sidecar with the current PID, fresh `createdAt`, and a mismatched Linux `starttime`; current main treats the alive PID as non-stale until age fallback, while the PR reclaims it.

## Solution Assessment

Yes, aside from the missing changelog entry. Reusing the shared raw `getProcessStartTime` helper is the narrowest maintainable fix and avoids the epoch-ms exact-comparison issue separately tracked for runtime-deps locks.
