  - body: The branch wires `compaction.guard.enabled` and `guard.escalation` into the safeguard runtime, but the generated config help still says the guard is config-only and has no runtime effect.   - body: The recovery guide says post-compaction validation checks whether unresolved tasks/promises survived, but the live safeguard details only populate `latestUserGoal`; `unresolvedItems` is passed to validation but never extracted in the runtime path. Either populate unresolved items before validation or narrow the guide to what the current runtime actually validates.
  - confidence: 0.82
- **[P2] Wrap failure signatures as untrusted prompt data:** `src/agents/pi-extensions/compaction-instructions.ts:92-97`
  - body: `failure.signature` comes from tool error text and is appended inside the compaction instruction block without the existing untrusted prompt-data wrapper or sanitizer. A tool error containing instruction-like text can be promoted into higher-priority summarization guidance; wrap or sanitize these signatures before adding them to the instructions.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, secret, install, publish, or network-surface change, but it does introduce an untrusted prompt-data handling concern in compaction instructions.

Concerns:

- **[medium] Untrusted tool errors enter compaction instructions:** `src/agents/pi-extensions/compaction-instructions.ts:96`
  - body: Repeated failure signatures are derived from tool result/error text and interpolated into the guard instruction block without the repo's untrusted prompt-data wrapper. Because compaction summaries persist into later context, instruction-like error text could steer the summary unless it is wrapped, escaped, or omitted.
  - confidence: 0.8
