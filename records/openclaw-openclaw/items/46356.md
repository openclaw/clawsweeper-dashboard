## Summary

Keep open: the WebChat routing problem is real, but this branch is not mergeable as-is because it mixes that fix with unrelated Discord, WhatsApp, Matrix, SDK, dependency, and docs changes and introduces blocking regressions.

## What This Changes

The branch adds WebChat-to-external mirroring, Discord REST proxy and DM fallback changes, WhatsApp proxy configuration/docs, Matrix thread-binding/runtime changes, SDK export adjustments, and related tests/lockfile updates.

## Best Possible Solution

Ship a narrow Gateway/session fix for Control UI/WebChat replies to channel-backed sessions, then handle Discord proxy/DM hardening, WhatsApp proxy config, and Matrix lifecycle/thread-binding work in separately owned PRs.

## Reproduction Assessment

Yes, source-reproducible. Current main explicitly keeps WebChat chat.send delivery on the internal webchat channel, and #45028 gives concrete Telegram-backed Control UI steps; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The direction addresses a real workflow gap, but this PR is too broad and introduces blocking regressions; the safer solution is a narrow Gateway fix plus separate owner-reviewed channel/proxy/Matrix changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Ignore message-type values in channel fallback:** `extensions/discord/src/monitor/message-utils.ts:136-139`
  - body: `rawData.type` on Discord message payloads is a message type, not a channel type. Using it as `ChannelType` can make missing-REST metadata apply the wrong DM/group/guild semantics and break the hardening this path is adding.
  - confidence: 0.9
- **[P2] Trim text before suppressing mirrored slash commands:** `src/gateway/server-methods/chat.ts:1378`
  - body: Command parsing trims input before detecting `/` commands, but this mirror guard checks the raw text. Inputs like ` /help` can execute locally and still be mirrored to the external chat.
  - confidence: 0.88
- **[P2] Skip mirror targets for already-delivered agent runs:** `src/gateway/server-methods/agent.ts:512-520`
  - body: For `agent` RPC calls with `deliver: true`, the agent command delivery path already sends the final reply externally. Registering `webchatMirrorTarget` here makes the lifecycle handler send the same text again.
  - confidence: 0.84
- **[P2] Preserve colon-containing peers in mirror targets:** `src/gateway/server-methods/webchat-mirror.ts:94-107`
  - body: Splitting the session rest on every colon truncates IDs such as Matrix room IDs, so `agent:main:matrix:channel:!room:example.org` rebuilds the target as `channel:!room` instead of the full room.
  - confidence: 0.82
- **[P2] Use sendable targets for grouped mirror sessions:** `src/gateway/server-methods/webchat-mirror.ts:122-123`
  - body: The generic `group:<peer>` target is not accepted by several channel senders; for example Telegram and Discord expect channel-specific target forms, so mirrored group replies can fail.
  - confidence: 0.82
- **[P1] Preserve Matrix sync persistence on shutdown:** `extensions/matrix/src/matrix/monitor/index.ts:142`
  - body: Replacing `releaseSharedClientInstance(client, "persist")` with `stopSharedClientInstance(client)` drops `stopAndPersist()`, so graceful Matrix restarts can behave like cold starts and skip restart backlog replay.
  - confidence: 0.86
- **[P1] Keep Matrix lifecycle setters on the live registry:** `extensions/matrix/src/matrix/thread-bindings.ts:65-66`
  - body: This file creates a new local manager registry while public lifecycle setters still come from `thread-bindings-shared.ts`, so session lifecycle commands can update a stale registry instead of the active Matrix bindings.
  - confidence: 0.84
