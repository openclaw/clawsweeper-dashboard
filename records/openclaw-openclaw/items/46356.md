## Summary

Keep open: current main still has the WebChat/Control UI routing gap, but this PR is not mergeable as-is because the useful Gateway direction is mixed with stale channel, Matrix, SDK, dependency, and proxy changes that introduce concrete regressions.

## What This Changes

The branch adds WebChat-to-external mirroring, Discord REST/proxy and DM fallback handling, WhatsApp proxy config/docs, Matrix thread-binding/runtime changes, SDK export/test adjustments, and lockfile updates.

## Best Possible Solution

Replace or split this into a narrow Gateway/session fix for Control UI replies to channel-backed sessions, then handle Discord proxy/DM hardening, WhatsApp proxy support, and Matrix/SDK cleanup in separately owned PRs.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: WebChat clients are explicitly prevented from inheriting external delivery routes in `chat.send`, and #45028 gives concrete Telegram-backed Control UI steps.

## Solution Assessment

No. The proposed direction addresses a real workflow gap, but this PR is not the best solution because it mixes unrelated channel and dependency work and introduces merge-blocking regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Ignore message-type values in channel fallback:** `extensions/discord/src/monitor/message-utils.ts:133-139`
  - body: `rawData.type` is a Discord message type, not a channel type. Using it as `ChannelType` can make missing-REST metadata apply arbitrary DM/group/guild semantics and break the DM hardening this path is trying to add.
  - confidence: 0.9
- **[P2] Trim text before suppressing mirrored slash commands:** `src/gateway/server-methods/chat.ts:1375`
  - body: Command parsing trims input before detecting `/` commands, but this mirror guard checks the raw text. Inputs like ` /help` can execute locally and still be mirrored to the external chat.
  - confidence: 0.88
- **[P2] Skip mirror targets for already-delivered agent runs:** `src/gateway/server-methods/agent.ts:509-516`
  - body: For `agent` RPC calls with `deliver: true`, the agent command delivery path already sends the final reply externally. Registering `webchatMirrorTarget` here makes the lifecycle handler send the same text again.
  - confidence: 0.84
- **[P2] Preserve colon-containing peers in mirror targets:** `src/gateway/server-methods/webchat-mirror.ts:85-97`
  - body: Splitting the session rest on every colon truncates IDs such as Matrix room IDs, so `agent:main:matrix:channel:!room:example.org` rebuilds the target as `channel:!room` instead of the full room.
  - confidence: 0.82
- **[P2] Use sendable targets for grouped mirror sessions:** `src/gateway/server-methods/webchat-mirror.ts:111-113`
  - body: The generic `group:<peer>` target is not accepted by several channel senders; for example Telegram only strips `group:` after a `telegram:` prefix, and Discord only accepts `user:` or `channel:` forms.
  - confidence: 0.82
- **[P1] Preserve Matrix sync persistence on shutdown:** `extensions/matrix/src/matrix/monitor/index.ts:143`
  - body: Replacing `releaseSharedClientInstance(client, "persist")` with `stopSharedClientInstance(client)` drops `stopAndPersist()`, so graceful Matrix restarts can behave like cold starts and skip restart backlog replay.
  - confidence: 0.86
- **[P1] Keep Matrix lifecycle setters on the live registry:** `extensions/matrix/src/matrix/thread-bindings.ts:65-66`
  - body: This file creates a new local manager registry while the public lifecycle setters still come from `thread-bindings-shared.ts`, so session lifecycle commands can update a stale registry instead of the active Matrix bindings.
  - confidence: 0.84
