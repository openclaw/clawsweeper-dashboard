## Summary

Keep open: current main and v2026.6.9 still load approved local Telegram outbound media into a full Buffer before creating grammY upload inputs, so the reported RAM-backed path remains source-reproducible.

## What This Changes

Avoid buffering whole approved local Telegram outbound media files into memory by preserving a disk-backed path or stream when no preprocessing is needed.

## Best Possible Solution

Add a validated disk-backed Telegram upload path for approved local files when no preprocessing is needed, while keeping buffered handling for remote media, transforms, MIME/probe work, and security checks that require bytes.

## Reproduction Assessment

Yes for a source-level path: current main reads approved local media into a full Buffer before Telegram upload. I did not run a live low-memory large-file Telegram profile, so the runtime RSS ceiling is not directly reproduced here.

## Solution Assessment

Unclear until maintainer design review. The likely best fix is to carry a validated local path or stream to Telegram send time and keep buffers only where OpenClaw must inspect, transform, or cap bytes.
