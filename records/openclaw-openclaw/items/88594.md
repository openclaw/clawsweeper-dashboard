## Label Justifications

- `P2`: The issue describes a bounded Telegram media-send memory/performance bug with limited blast radius but real delivery impact for large local files.
- `impact:message-loss`: Large local Telegram media sends can fail or become impractical before delivery when process RAM becomes the effective upload limit.

## Summary

Keep open: current main and the latest release still route Telegram outbound local media through buffer-only `loadWebMedia` results before creating grammY `InputFile`s, so the reported RAM-backed upload limitation remains source-backed and is not superseded by the related attachment-path or media-limit fixes.

## What This Changes

The issue asks OpenClaw to avoid eagerly buffering approved Telegram outbound local media files and instead preserve a disk-backed path or stream when safe.

## Best Possible Solution

Add a validated Telegram local upload path that can hand grammY a file path or read stream when no preprocessing is needed, while keeping the existing buffered path for remote inputs, image transforms, video probes, and unsafe or unvalidated local files.

## Reproduction Assessment

No live high-confidence memory reproduction was run. Source inspection does show a reproducible buffer path: local media is read into a full `Buffer` and then passed to Telegram `InputFile`.

## Solution Assessment

Unclear until maintainer design review. The maintainable direction is to carry a validated local file reference or stream to Telegram send time and retain buffer loading only where OpenClaw must inspect or transform the bytes.
