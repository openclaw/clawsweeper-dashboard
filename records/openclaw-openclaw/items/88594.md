## Summary

Keep open: current main and the latest release still route approved Telegram local media through buffer-only media loading before creating grammY `InputFile`s, so the reported memory limitation remains plausible and is not covered by the related attachment-path and media-limit fixes.

## What This Changes

The issue asks OpenClaw to avoid fully buffering validated Telegram outbound local files into RAM when they can be sent as disk-backed references or streams.

## Best Possible Solution

Add a disk-backed or streaming Telegram local-media path after local-root and media-type validation, preserving the existing buffer fallback only when preprocessing or probing truly requires it.

## Reproduction Assessment

No live high-confidence memory reproduction was run. Source inspection does show the current local-media path eagerly returns a `Buffer` and passes that buffer into Telegram `InputFile`, matching the reported failure mode.

## Solution Assessment

Unclear until implementation, but the right direction is narrow: carry a validated local file reference or stream to Telegram send time and keep buffer loading for paths that need transformation or full-content probes.
