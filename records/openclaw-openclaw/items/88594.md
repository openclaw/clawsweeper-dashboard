## Label Justifications

- `P2`: The issue is a bounded Telegram media-send performance and reliability problem with real impact for large local files but limited blast radius.
- `impact:message-loss`: Large local Telegram sends can fail or become impractical before delivery when RAM becomes the effective upload limit.

## Summary

Keep open: current main and the latest release still use buffer-only Telegram outbound media paths for approved local files, so the RAM-backed upload limitation remains source-backed and is not owned by the related attachment-path, media-limit, or inbound Local Bot API PRs.

## What This Changes

The issue requests a Telegram outbound local-media path that preserves a disk-backed file reference or stream for approved local files instead of loading the whole file into RAM before upload.

## Best Possible Solution

Introduce a Telegram-owned disk-backed or streaming local upload path for already-validated local files, while keeping the buffered path for remote media, transformed images, video probes, and unsafe or unvalidated inputs.

## Reproduction Assessment

No live high-confidence memory reproduction was run. Source inspection does show the relevant current-main path: local media is read into a full `Buffer` and then passed to Telegram `InputFile`.

## Solution Assessment

Unclear until maintainer design review. The maintainable direction is to carry a validated local file reference or stream to Telegram send time and retain buffer loading only where OpenClaw must inspect or transform bytes.
