## Summary

Keep open: current main and the latest v2026.6.8 release still route approved local Telegram outbound media through the buffer-only web-media result and then into grammY `InputFile` objects from full buffers, so the reported RAM-backed path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to preserve disk-backed file references or streams for approved Telegram outbound local-media sends instead of eagerly buffering whole local files in memory.

## Best Possible Solution

Add a validated disk-backed Telegram upload path for approved local files when no preprocessing is needed, while keeping the buffered path for remote media, transforms, MIME checks, and probes that require bytes.

## Reproduction Assessment

Yes for the source path: current main reads approved local media into a full `Buffer` before Telegram upload. A live low-memory large-file profile was not run, so the runtime memory ceiling is not directly reproduced here.

## Solution Assessment

Unclear until maintainer design review. The likely best fix is to carry a validated local path or stream to Telegram send time and keep buffers only where OpenClaw must inspect, transform, or cap bytes.
