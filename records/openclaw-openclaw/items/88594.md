## Summary

Keep open: current main and v2026.6.8 still load approved local Telegram media into `Buffer`-shaped `WebMediaResult`s before constructing grammY `InputFile`s, so the reported RAM-backed large-file path remains source-backed and is not covered by the related media-root, attachment-alias, or media-limit fixes.

## What This Changes

The issue asks OpenClaw to avoid buffering whole approved local Telegram outbound media files in RAM when a disk-backed path or stream can be preserved safely.

## Best Possible Solution

Add a validated Telegram local upload path that can hand grammY a safe local path or stream when no preprocessing is needed, while keeping the buffered path for remote media, transforms, MIME checks, and probes that require bytes.

## Reproduction Assessment

Yes for the source path, but not for a live memory failure: current main reads approved local media into a full `Buffer` before Telegram upload, while no large-file memory profile was run here.

## Solution Assessment

Unclear until maintainer design review. The likely best fix is to carry a validated local file reference or stream to Telegram send time and retain buffers only where OpenClaw must inspect or transform bytes.
