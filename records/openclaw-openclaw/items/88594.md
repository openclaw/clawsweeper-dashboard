## Summary

Keep open: current main and the latest release still load approved Telegram local media into `Buffer`-backed `WebMediaResult`s before constructing grammY `InputFile`s, so the reported RAM-backed upload limitation remains source-backed and is not owned by a narrower duplicate.

## What This Changes

The issue requests a disk-backed or streaming Telegram outbound local-media path for approved local files when no preprocessing is required.

## Best Possible Solution

Add a Telegram-owned or shared media result shape that can carry a validated local path/stream for unprocessed local files while keeping buffer-based handling for transforms, probes, and security-sensitive validation.

## Reproduction Assessment

No, not as a live low-memory reproduction in this read-only review. Source inspection does show the current local-media path reads the whole file into memory before sending, which makes the reported RAM bottleneck plausible and actionable.

## Solution Assessment

Unclear until maintainers choose the media contract. The direction is sound, but the best fix is likely a closed disk-backed media result or Telegram upload helper rather than threading raw paths around the existing buffer-only `WebMediaResult`.
