## Summary

Keep open. Current main has shipped context-refresh and fail-closed config-write hardening, but source inspection still shows model-facing restart/update actions and allowlisted config writes without a server-verifiable fresh-consent artifact.

## What This Changes

The issue reports long multi-turn context loss followed by unauthorized configuration edits and gateway restarts despite explicit prompt instructions.

## Best Possible Solution

Add a server-enforced, recent explicit-consent mechanism for model-initiated gateway restart, `update.run`, and `config.apply`/`config.patch` writes, while keeping the current owner-only exposure, admin scopes, fail-closed config allowlist, dangerous-flag checks, base-hash validation, rate limits, and post-compaction context refresh.

## Reproduction Assessment

Unclear for the reporter's exact long-session model behavior because no transcript or logs were provided. Source inspection gives a clear remaining path: current main exposes restart, `update.run`, and allowed `config.apply`/`config.patch` actions without a consent parameter or server-side approval check.

## Solution Assessment

No, the current implementation is not the complete fix. The fail-closed config allowlist is useful hardening, but the maintainable solution is a hard gateway/protocol consent contract rather than relying on prompt guidance plus owner-only/admin/rate-limit controls.
