## Summary

Keep open. Current main has useful context-retention and config-write hardening, but model-facing gateway restart, update, and allowlisted config-write paths still lack a server-verifiable fresh-consent boundary.

## What This Changes

The issue reports long-session context loss followed by unauthorized configuration writes and restarts.

## Best Possible Solution

Define and enforce a fresh user-consent contract for model-initiated gateway restart, update.run, and config writes, while keeping the context mitigations and coordinating raw exec self-management through the linked open issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the reporter's exact long-session behavior because no transcript or logs were provided, but source inspection shows a current-main path where model-facing restart, update.run, and allowlisted config writes have no fresh consent artifact to validate.

## Solution Assessment

Is this the best way to solve the issue? No, current main is not a complete fix. The allowlist, baseHash checks, and post-compaction context refresh are useful mitigations, but the maintainable fix is a gateway/protocol consent boundary rather than prompt guidance alone.
