## Summary

Keep open. Current main has meaningful prompt/context and gateway-write hardening, but the sensitive control-plane gap remains: model-callable owner/admin paths can still restart the gateway, apply allowed config changes, or run updates without a server-validated recent user-consent artifact.

## What This Changes

The issue asks OpenClaw to preserve soul/user/memory and red-line instructions across long multi-turn sessions, and to prevent model-initiated config edits or gateway restarts unless the user explicitly consents.

## Best Possible Solution

Ship a server-enforced, recent explicit-consent mechanism for model-initiated gateway restart, config.apply, config.patch, and update.run. The end state should preserve owner-only exposure, admin scopes, allowlists, dangerous-flag blocking, base-hash checks, rate limits, and post-compaction red-line refresh while adding proof that the user approved the specific sensitive action recently enough.

## Reproduction Assessment

Unclear for the reporter's exact long-session context-loss behavior because the issue has no transcript or logs. Yes for the remaining consent gap: current source shows the owner gateway tool and server protocol can perform restart, allowed config writes, and update.run without any approval id or consent token field.

## Solution Assessment

No, the current implementation is not the best complete solution for the reported behavior. Prompt reminders, owner-only exposure, admin scopes, allowlists, and rate limits are useful mitigations, but the safer fix is a hard approval/consent contract enforced at the agent gateway tool and preferably at the RPC/server layer too.
