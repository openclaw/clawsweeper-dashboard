## Label Justifications

- `P2`: This is a normal bounded-response bugfix with limited blast radius in one bundled plugin REST client.
- `merge-risk: 🚨 compatibility`: Existing ClickClack success responses above 16 MiB would now fail early instead of being read through native unbounded JSON parsing.

## Summary

Keep open: the patch is still necessary because current main still parses ClickClack REST success bodies with raw `response.json()`, while this PR moves the shared request helper onto the existing bounded provider JSON reader. I found no blocking code or security findings; the remaining merge question is maintainer acceptance that successful ClickClack JSON bodies over 16 MiB will now fail early.

## What This Changes

The PR routes ClickClack REST success JSON parsing through `readProviderJsonResponse` and adds a loopback HTTP regression for oversized success responses.

## Best Possible Solution

Land this focused patch after maintainers accept the 16 MiB ClickClack success-response boundary and required checks pass; choose a different cap only if ClickClack has legitimate success JSON payloads above that size.

## Reproduction Assessment

Yes. Source inspection shows current main sends every successful ClickClack REST response through raw `response.json()` in the shared request helper, and the PR body provides loopback after-fix output for the same boundary.

## Solution Assessment

Yes. The shared `request<T>()` return path is the right owner boundary because all ClickClack REST success methods flow through it; per-endpoint parsing or a ClickClack-local duplicate cap would create policy drift.
