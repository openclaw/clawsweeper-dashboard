## Summary

This PR remains necessary: current main still parses ClickClack REST success bodies with raw `response.json()`, while the PR moves the single shared REST success path to OpenClaw's bounded provider JSON reader and adds focused loopback proof. I found no blocking code defect, but the 16 MiB fail-closed behavior is compatibility-sensitive and still needs maintainer acceptance before merge.

## What This Changes

The PR routes ClickClack REST success JSON parsing through `readProviderJsonResponse` and adds a loopback oversized-response regression test.

## Best Possible Solution

Land this focused patch after maintainer acceptance of the shared 16 MiB ClickClack success-response cap and normal required checks; if ClickClack legitimately needs larger success JSON, adjust that cap policy before merging.

## Reproduction Assessment

Yes. Source inspection shows every successful ClickClack REST response currently flows through raw `response.json()`, and the PR body plus added loopback test cover the oversized stream path after the fix.

## Solution Assessment

Yes. The single `request<T>()` helper is the right owner boundary because all ClickClack REST success methods flow through it; per-endpoint parsing or a local ClickClack-specific reader would duplicate provider HTTP policy.
