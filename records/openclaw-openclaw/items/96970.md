## Summary

Keep open: current main still has ClickClack REST success responses parsed through raw `response.json()`, and this PR moves that shared success path onto the existing bounded provider JSON reader with focused loopback proof. The patch looks correct; the only maintainer-visible merge risk is the intentional new 16 MiB cap for successful ClickClack JSON bodies.

## What This Changes

The PR routes ClickClack REST success JSON parsing through `readProviderJsonResponse` and adds a loopback oversized-response regression test.

## Best Possible Solution

Land this focused patch after normal CI if maintainers accept that ClickClack success JSON now follows the shared 16 MiB provider cap; otherwise adjust that cap policy before merge.

## Reproduction Assessment

Yes from source inspection: current main sends every successful ClickClack REST response through raw `response.json()` in the shared request helper. I did not rerun the loopback repro locally, but the PR body includes after-fix live loopback output for the same path.

## Solution Assessment

Yes; the single shared `request<T>()` return path is the right owner boundary because all ClickClack REST success methods flow through it and the helper already comes from the plugin SDK surface. A local ClickClack parser or per-endpoint patch would create duplicate policy.
