## Summary

Keep this issue open. Current main has partial hook timeout infrastructure, but legacy `before_agent_start` handlers still have no default/global timeout and can still block the model-resolution and prompt-build paths indefinitely when a plugin does not pass `opts.timeoutMs`.

## What This Changes

The issue requests a gateway-level timeout and diagnostics guard for hanging plugin `before_agent_start` hooks so agent runs cannot block indefinitely.

## Best Possible Solution

Add a fail-open default timeout for legacy `before_agent_start`, or a documented global plugin hook timeout policy that covers it. Keep the boundary generic, preserve per-hook `opts.timeoutMs` overrides, log plugin id, hook name, elapsed budget, and run id when available, and add regression coverage for both model-resolution and prompt-build legacy paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: register a `before_agent_start` handler that returns a never-settling promise without `opts.timeoutMs`; current main reaches `runModifyingHook()` with no default timeout and both legacy call sites await it.

## Solution Assessment

Yes, the right fix layer is the shared hook runner or its global policy. Fixing only the OpenViking plugin would address one trigger but would not protect OpenClaw from any other legacy/external plugin hook that hangs.
