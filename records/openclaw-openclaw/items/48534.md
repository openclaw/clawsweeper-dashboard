## Summary

Keep open: current main has configurable per-plugin typed-hook timeouts, but the legacy `before_agent_start` path still has no default timeout unless the plugin or operator sets `timeoutMs`, so the reported no-config hang remains source-reproducible.

## What This Changes

Request a default timeout and diagnostic guard for plugin `before_agent_start` handlers so a hung plugin cannot leave agent runs permanently stuck.

## Best Possible Solution

Add a default fail-open timeout for legacy `before_agent_start`, include plugin/hook/run context in the diagnostic log, and keep existing per-plugin timeout overrides as the tuning mechanism.

## Reproduction Assessment

Yes, by source inspection: a `before_agent_start` handler that returns a never-settling promise and has no configured `timeoutMs` still reaches `await promise` with no default timeout. I did not run a live Gateway reproduction because this was a read-only review.

## Solution Assessment

No, current main is only a partial solution: configurable hook budgets are useful, but they do not enforce a default guard on legacy `before_agent_start`. The narrow maintainable fix is to add the missing default and regression coverage without adding another config surface.

## Repair Work Prompt

Fix the remaining `before_agent_start` hook hang described in https://github.com/openclaw/openclaw/issues/48534. In `src/plugins/hooks.ts`, add a default fail-open timeout for the legacy modifying hook `before_agent_start` so a handler with no plugin-authored or operator-configured `timeoutMs` cannot block the agent run indefinitely. Preserve the current precedence where `plugins.entries.<id>.hooks.timeouts.<hookName>` overrides `plugins.entries.<id>.hooks.timeoutMs`, which overrides `api.on(..., { timeoutMs })`, which overrides runner defaults. Add focused regression coverage for a never-settling `before_agent_start` handler proving the runner logs the timeout and continues to a later handler/result. Do not add a new config key unless existing per-plugin hook timeout config is insufficient. Update docs/changelog if user-facing default behavior changes.
