## Summary

Keep open: current main still has the Dreaming cleanup shape described in the report, including per-run narrative session keys and a scrubber that only looks at primary `.jsonl` transcripts, so this is real remaining cleanup work rather than an already-shipped fix.

## What This Changes

The issue reports that the managed `memory-core` Dreaming sweep leaves `dreaming-narrative-*` session rows and `.trajectory` sidecars after the 03:00 cleanup on OpenClaw 2026.5.3-1.

## Best Possible Solution

Land or replace #70464 with stable narrative session keys, a separate per-run idempotency key, pre-cleanup, and explicit `sessions.delete`/scrubber trajectory sidecar cleanup with memory-core regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: enable Dreaming, let the managed 03:00 sweep generate narrative subagent sessions, then inspect `sessions.json` and adjacent `.trajectory` sidecars; I did not run the live overnight sweep here.

## Solution Assessment

Is this the best way to solve the issue? Yes: fix the cleanup lifecycle in `memory-core` and the generic session/trajectory deletion seam rather than hiding Dreaming rows in the UI or relying on later maintenance pruning.
