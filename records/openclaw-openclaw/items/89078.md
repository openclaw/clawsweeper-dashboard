## Summary

Keep open: this PR is a useful fix direction for the TUI queue-mode bug, but it is not merge-ready because typed submits still default to steer when the caller omits `queueMode`, Gateway rows expose only the raw session override instead of the effective queue mode, stale TUI session state can leak across refreshed rows, and the supplied proof does not show the changed real TUI plus Gateway behavior.

## What This Changes

The PR adds `queueMode` to Gateway session rows and TUI session state, relaxes TUI busy guards for non-steer modes, preserves active run state on failed queued sends, and adds focused unit coverage.

## Best Possible Solution

Land a refreshed fix that projects the effective queue mode into TUI session state, passes it through both submit gates, clears stale missing values, preserves pending-submit safeguards, and includes redacted real TUI plus Gateway proof.

Yes. Source inspection on current main and v2026.6.11 shows typed-submit and `sendMessage` can block Gateway-mode active-run messages before `client.sendChat`, while the queue docs define followup, collect, and interrupt behavior for active sessions.

## Solution Assessment

No. Propagating queue mode into the TUI is the right direction, but this branch still needs effective queue-mode resolution, caller propagation, stale-state clearing, and real TUI plus Gateway proof before it is the best fix.
