## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Track queued run IDs before clearing optimistic state:** `src/tui/tui-command-handlers.ts:796-798`
  - body: Clearing `pendingOptimisticUserMessage` after a non-steer ACK allows a second queued send before the first queued run registers, but the TUI still stores only one `pendingChatRunId`. A later ACK overwrites the earlier run ID, so `/stop` or Esc can no longer target that earlier queued run until lifecycle events arrive; keep the single-pending guard or track multiple accepted queued IDs.
  - confidence: 0.86
- **[P2] Preserve the active run when enqueue fails:** `src/tui/tui-command-handlers.ts:731-735`
  - body: This guard newly permits non-steer sends while `activeChatRunId` is already set. If `client.sendChat` rejects for that enqueue, the existing catch cleanup clears `activeChatRunId` and forgets the still-running active turn even though only the new enqueue failed; clean up the failed pending send without dropping active state.
  - confidence: 0.9
- **[P2] Rebase tests onto the pending-user send path:** `src/tui/tui-command-handlers.test.ts:985`
  - body: This added assertion still expects `chatLog.addUser`, but current main records outbound sends with `addPendingUser` and `pendingSubmitDraft`. Keeping this branch shape would miss the current optimistic-row behavior and makes conflict resolution likely to regress it; refresh the branch and assert the pending-user path.
  - confidence: 0.85
