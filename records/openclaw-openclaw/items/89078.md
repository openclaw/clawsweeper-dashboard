## Summary

Keep this PR open: the bug is real and the fix direction is useful, but the patch should not merge until queue-mode clearing is handled and the external proof shows the real TUI/Gateway behavior rather than only build output and mocked tests.

## What This Changes

The PR adds Gateway `queueMode` to session rows, propagates it into TUI session state, and gates the TUI busy guard on `queueMode === "steer"`.

## Best Possible Solution

Merge after queue mode is treated as complete session state, including clearing on reset/default, and after the contributor provides redacted real TUI/Gateway behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main blocks TUI sends on `activeChatRunId`/`pendingChatRunId` without reading queue mode, while the documented queue contract says non-steer modes should queue, collect, or interrupt active runs. I did not run a live provider/TUI repro.

## Solution Assessment

No, not yet: adding `queueMode` to the session row is the right layer, but the PR must also clear stale TUI queue mode when the refreshed row omits it after `/queue reset` or `/queue default`.
