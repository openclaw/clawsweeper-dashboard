Full review comments:

- **[P2] Track every accepted queued run before allowing more sends:** `src/tui/tui-command-handlers.ts:796-798`
  - body: Clearing `pendingOptimisticUserMessage` after a non-steer ACK lets another queued send proceed, but the TUI still stores only one `pendingChatRunId`. A second ACK overwrites the first run ID, so `/stop`/Esc can no longer abort the earlier queued run; either track all accepted queued run IDs or keep the single-pending guard until lifecycle registration catches up.
  - confidence: 0.86
- **[P2] Preserve the active run when enqueue fails:** `src/tui/tui-command-handlers.ts:733-735`
  - body: This guard newly permits non-steer sends while an active run is still tracked. If `client.sendChat` rejects, the existing catch path clears `activeChatRunId`, so the TUI loses the still-running active turn when only the enqueue attempt failed.
  - confidence: 0.9
- **[P2] Assert the pending-user row path:** `src/tui/tui-command-handlers.test.ts:985`
  - body: Current main renders outbound normal sends through `chatLog.addPendingUser`, but this added test still expects `addUser`. Refresh the assertion to the pending-user row path so the test matches the current TUI send implementation after merge.
