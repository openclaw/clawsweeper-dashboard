## Summary

Keep open: the branch addresses resolved non-steer queue modes, but the merge result still blocks the `/queue` directive while the TUI is in default steer mode, so users cannot switch to followup/collect/interrupt during an active Gateway run from the TUI.

## What This Changes

The branch adds resolved `queueMode` to Gateway session rows and TUI session state, then changes TUI submit/send guards and tests so non-steer modes can forward active-run messages to Gateway.

## Best Possible Solution

Keep the resolved queue-mode propagation, but before merge let `/queue` directives reach Gateway from the steer-mode active-run path and add redacted live TUI/Gateway proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: slash input calls `handleCommand`, `/queue` falls through to `sendMessage`, and the PR merge ref still blocks it when `queueMode` is `steer` and an active gateway run exists.

## Solution Assessment

No. The branch is a plausible partial fix for already-resolved non-steer modes, but the best fix also preserves the documented `/queue` override path and proves the live TUI/Gateway active-run flow.
