## Summary

## What This Changes

The PR propagates resolved queue mode into TUI session state and makes the TUI submit/send busy guards allow non-steer queue modes to reach Gateway queue handling.

## Best Possible Solution

Land the queue-mode-aware TUI guard once a redacted real TUI/Gateway active-run proof shows non-steer mid-run sends reaching Gateway queue handling.

## Reproduction Assessment

Yes. Current-main source shows both the typed-submit gate and sendMessage guard block active Gateway runs without considering queue mode, while docs define followup, collect, and interrupt as non-steer active-run modes.

## Solution Assessment

Yes, this is likely the right fix shape because the TUI should only enforce the steer-mode busy guard and let Gateway own queue handling for non-steer modes. The remaining gap is proof from the real TUI/Gateway path, not a different code direction.
