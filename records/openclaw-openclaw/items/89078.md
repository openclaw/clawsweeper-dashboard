## Summary

Keep open. The branch improves sessions that already resolve to a non-steer queue mode, but it still blocks the linked default-steer flow where `/queue mode=followup` must reach Gateway before the session can persist `followup`; the supplied proof also does not show a live TUI plus Gateway active-run path.

## What This Changes

The PR adds effective queue mode to Gateway session rows and TUI session state, then changes TUI submit and send busy guards to allow active-run sends when the resolved mode is non-steer.

## Best Possible Solution

Let queue directives reach Gateway even when the current resolved mode is still steer, preserve duplicate-submit protection, and require redacted live TUI/Gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection shows slash submissions bypass the typed-submit gate and the PR head still blocks unknown Gateway slash commands such as `/queue mode=followup` in the command-handler busy guard when the current mode is steer.

## Solution Assessment

No, not yet. The right owner boundary is Gateway queue policy, but this PR only allows messages after TUI already knows a non-steer mode, so the mode-changing directive path remains blocked.
