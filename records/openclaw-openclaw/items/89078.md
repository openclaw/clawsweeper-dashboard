Full review comments:

- **[P2] Track each accepted queued run before allowing another send:** `src/tui/tui-command-handlers.ts:796-798`
  - body: Clearing `pendingOptimisticUserMessage` after a non-steer ACK allows a second queued send before the first queued run registers, but the TUI still stores only one `pendingChatRunId`. A later ACK overwrites the earlier run ID, so `/stop` or Esc cannot abort the earlier accepted queued run; either track all accepted queued run IDs or keep the single-pending guard until lifecycle events catch up.
  - confidence: 0.86

  - body: This guard newly permits non-steer sends while `activeChatRunId` is already set. If `client.sendChat` rejects for that new enqueue, the existing catch path clears `activeChatRunId` and forgets the still-running active turn even though only the enqueue attempt failed; clean up the new pending send without dropping the active run.
  - confidence: 0.9
