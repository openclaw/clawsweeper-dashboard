## Summary

Keep open. The PR is aimed at the right TUI/Gateway boundary, but it still appears to allow only one queued mid-run message because `pendingOptimisticUserMessage` remains true after `sendChat` accepts and blocks later non-steer sends until the queued run emits; the supplied proof is also function/test output rather than a real TUI-to-Gateway mid-run flow.

## What This Changes

The PR adds resolved `queueMode` propagation into Gateway session rows and TUI session state, then makes TUI typed-submit and send busy guards allow non-steer queue modes to reach Gateway queue handling.

## Best Possible Solution

Separate RPC-in-flight protection from queued-run tracking so non-steer modes can forward additional active-run messages after `sendChat` accepts, while keeping steer-mode and rapid double-submit protection intact.

## Reproduction Assessment

Yes from source. Current main blocks Gateway-mode TUI sends during active runs, and the PR's remaining issue is also source-reproducible because `pendingOptimisticUserMessage` remains true after `sendChat` accepts until a later queued-run event clears it.

## Solution Assessment

No. The owner boundary is right, but the proposed guard is still too broad for non-steer modes because it treats queued-run registration as if the previous RPC were still in flight.
