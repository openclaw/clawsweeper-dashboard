## Summary

Keep open: the PR addresses a valid documented TUI/Gateway queue bug, but the merge result still has a concrete active-run state regression on failed non-steer sends and the supplied proof does not exercise a real TUI connected to Gateway mid-run.

## What This Changes

The PR adds resolved queue mode to Gateway session rows and TUI session state, then makes TUI submit and command-handler busy guards allow non-steer queue modes to send during active runs with focused tests.

## Best Possible Solution

Preserve the pre-existing active run when a non-steer enqueue fails, keep the queue-mode propagation, and land only after redacted real TUI/Gateway proof shows mid-run delivery reaches Gateway queue handling.

## Reproduction Assessment

Yes, source-level: current main rejects gateway-mode active-run sends before Gateway queue policy, and the merge result makes failed non-steer sends reach a catch path that clears the active run. I did not run a live TUI/Gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. Queue-mode-aware forwarding is the right boundary, but the failure path needs to preserve the active run state it did not start and the PR still needs live TUI/Gateway proof.
