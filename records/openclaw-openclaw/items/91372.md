## Summary

Keep this issue open: current main still lacks a reserved-literal guard for Telegram targets, and the linked open PR is the active closing fix candidate rather than merged code.

## What This Changes

The issue asks the gateway/outbound session target resolver to reject reserved literal target strings such as `current`, `self`, `this`, and `me` unless they resolve through an explicit safe binding or destination.

## Best Possible Solution

Land a maintainer-approved fix via https://github.com/openclaw/openclaw/pull/94107 or an equivalent replacement that preserves configured Telegram directory hits before failing closed on reserved-literal misses.

## Reproduction Assessment

Yes, by source inspection: current main has no reserved-target guard, Telegram normalizes username-shaped strings like `current` into `telegram:@current`, and gateway send/session-route paths can pass that normalized destination forward. I did not run live Telegram delivery.

## Solution Assessment

Mostly yes. The active linked PR appears to move the guard to the channel-owned target-resolution boundary and preserve directory hits before failing closed; the remaining question is maintainer acceptance of the plugin SDK contract and compatibility policy.
