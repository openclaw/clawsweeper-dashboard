## Summary

Keep open: current main and the latest release still accept bare reserved target strings on the outbound/session target paths, and the two linked fix PRs were closed unmerged after review found incomplete fail-closed behavior.

## What This Changes

The issue asks OpenClaw to reject bare reserved delivery-target meta-strings such as `current`, `self`, `this`, and `me` instead of resolving them as literal chat targets.

## Best Possible Solution

Add one shared fail-closed reserved-target guard before both session delivery resolution and message-action/directory lookup, preserve a rejected-target state through agent delivery planning, and cover it with focused outbound tests plus redacted real Telegram or gateway proof.

## Reproduction Assessment

Yes for the boundary bug by source inspection: current main accepts reserved `explicitTo` values and Telegram can normalize `current` as `@current`. I did not run a live Telegram reproduction, so the exact third-party-channel hit remains deployment-dependent.

## Solution Assessment

Yes, fail-closed reserved-target handling at the shared resolver boundary is the right direction. The prior PR attempts were not the best final shape because one missed message-action directory lookup and the other converted reserved targets into implicit fallback instead of preserving rejection.
