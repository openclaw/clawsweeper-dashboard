## Label Justifications

- `P1`: The issue describes a broken Telegram delivery workflow that can route outbound status/check-in messages toward the wrong public target.
- `impact:security`: The reported failure can expose outbound messages to an unintended third-party Telegram destination before Telegram rejects the send.
- `impact:message-loss`: The resolver can choose the wrong delivery destination, causing intended messages to fail or be misrouted instead of reaching the current session.

## Summary

Keep this issue open: current main and the latest release still lack a reserved-literal guard, and https://github.com/openclaw/openclaw/pull/94107 is the active open closing fix candidate rather than merged code.

## What This Changes

The issue asks OpenClaw to reject bare session meta-strings such as `current`, `self`, `this`, and `me` at the gateway/outbound target boundary instead of treating them as literal Telegram targets.

## Best Possible Solution

Keep the canonical issue open until maintainers land or replace the open candidate with a directory-aware fail-closed resolver that prevents reserved-word misses from falling through to literal Telegram destinations.

## Reproduction Assessment

Yes, source-reproducible. Current main has no reserved-word guard, Telegram accepts username-shaped strings like `current`, and delivery planning can carry the explicit target forward; I did not run a live Telegram send in this read-only pass.

## Solution Assessment

Unclear until maintainer review completes. The likely best fix is the linked directory-first fail-closed resolver path, because a blanket literal-word rejection could break configured Telegram directory destinations named `current`, `self`, `this`, or `me`.
