## Summary

Keep open: the backend quarantine and raw-row preservation look like the right narrow fix, but the PR still leaves a Control UI schedule guard gap that can blank the cron page when a malformed row reaches the browser.

## What This Changes

The PR quarantines malformed persisted cron jobs during store loading, preserves quarantined raw rows on save, normalizes legacy interval strings, adds Control UI guards, regression tests, and a changelog entry.

## Best Possible Solution

Finish the UI boundary so unsupported schedules are rejected or safely rendered before the cron job list paints, while preserving the backend quarantine and raw-row preservation behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence PR-head path: loadCronJobsPage accepts rows with valid payloads, getVisibleCronJobs returns unsupported schedules when the filter is all, and renderJob calls formatCronSchedule. That formatter dereferences job.schedule.kind, so a valid-payload row with no schedule can still blank the page.

## Solution Assessment

No, not yet. The backend quarantine/preservation approach is the right narrow fix, but the proposed client fallback should also drop or safely render unsupported schedules at the load, filter, or presenter boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less rows before rendering:** `ui/src/ui/controllers/cron.ts:370-383`
  - body: The new helper returns null for unsupported schedules, but the default all filter still returns true, while loadCronJobsPage only filters payload validity. A cron.list row with a valid payload and missing schedule can still reach renderJob, where formatCronSchedule dereferences job.schedule.kind and can blank the cron page. Filter unsupported schedules out before returning visible jobs, or make the presenter render them safely.
  - confidence: 0.9
