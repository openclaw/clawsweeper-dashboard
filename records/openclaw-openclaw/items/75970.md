## Summary

Keep open: the backend quarantine direction is sound, but the PR still leaves a schedule-malformed cron row able to reach the default Control UI render path and crash the dashboard; the head is also currently dirty against main.

## What This Changes

The PR adds cron-store quarantine/preservation for malformed persisted jobs, legacy numeric-string interval normalization, Control UI schedule guards, regression tests, and a changelog entry.

## Best Possible Solution

Keep the store-boundary quarantine/preservation approach, add a shared schedule-shape guard at UI ingress or presenter rendering, and rebase onto current main without dropping state-sidecar persistence behavior.

## Reproduction Assessment

Yes. By source inspection, a `cron.list` row with a valid payload but missing `schedule` passes the PR's default `all` filter path and then `formatCronSchedule` dereferences `job.schedule.kind`, matching the linked issue's browser error.

## Solution Assessment

No, not yet. Backend quarantine plus a UI fallback is the right narrow direction, but the UI fallback must either drop unsupported schedules before state/rendering or make `formatCronSchedule` safe, and the dirty branch needs a rebase that preserves current-main cron persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-malformed jobs before rendering:** `ui/src/ui/controllers/cron.ts:370-374`
  - body: The new schedule guard only excludes malformed schedules when a specific schedule filter is selected. With the default `all` filter, a row that has a valid payload but no `schedule` still passes through and `renderJob` calls `formatCronSchedule(job)`, which dereferences `job.schedule.kind`; that keeps the blank-dashboard crash alive if stale or bypassed backend data reaches the browser. Filter unsupported schedules at UI ingress/all-filter time or harden the presenter too.
  - confidence: 0.91
