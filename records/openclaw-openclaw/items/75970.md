## Summary

Keep open. The PR addresses the malformed payload hole and preserves quarantined rows, but the current head still treats any schedule object with a supported `kind` as hydratable, so kind-only schedules can enter runtime instead of being quarantined.

## What This Changes

The PR quarantines malformed persisted cron jobs during store load, preserves raw quarantined rows on later writes, adds Control UI defensive handling, tests, and a changelog entry for the cron dashboard crash.

## Best Possible Solution

Keep the quarantine/preservation design, but make the store-boundary validator check each schedule variant's required body field and add regression coverage for kind-only schedules before merge.

## Reproduction Assessment

Yes. Persist a job with a non-empty id, valid payload, and `schedule: { "kind": "cron" }`; PR head admits it because the schedule predicate checks only `kind`, while runtime schedule computation requires `expr`.

## Solution Assessment

No, not yet. Backend quarantine plus a UI fallback is the right shape, but the backend validator must reject invalid schedule bodies before runtime hydration.

## Review Findings
