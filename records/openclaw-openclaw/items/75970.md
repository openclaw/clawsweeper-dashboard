## Summary

Keep open: the backend quarantine/preservation direction is sound, but the PR head still has a Control UI fallback gap that can leave a schedule-less cron row visible in the default jobs list and crash the renderer.

## What This Changes

The PR filters malformed persisted cron jobs out of runtime/dashboard reads, preserves quarantined rows on later saves, normalizes string interval schedules, adds Control UI guards, tests, and a changelog entry.

## Best Possible Solution

Complete the backend quarantine with a Control UI list boundary that rejects or safely renders unsupported schedules before the cron jobs view paints.

## Reproduction Assessment

Yes. In PR head, a cron.list result with a valid payload but no schedule passes hasCronJobPayload and the default all filter, then renderJob calls formatCronSchedule, which dereferences job.schedule.kind.

## Solution Assessment

No, not yet. The backend quarantine/preservation approach is the right narrow fix, but the client fallback should also drop or safely render unsupported schedules at the list or presenter boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less rows before rendering:** `ui/src/ui/controllers/cron.ts:370-383`
  - body: The new helper returns null for unsupported schedules, but the default all filter still returns true, while loadCronJobsPage only filters payload validity. A cron.list row with a valid payload and missing schedule can still reach renderJob, where formatCronSchedule dereferences job.schedule.kind and can blank the cron page. Filter unsupported schedules out before returning visible jobs, or make the presenter render them safely.
  - confidence: 0.9
