## Summary

Keep open: the PR has a useful backend quarantine, but the Control UI fallback is still incomplete because schedule-less rows can remain visible in the default cron filter and crash the render path.

## What This Changes

The PR quarantines malformed persisted cron jobs during load, preserves raw quarantined rows on save, normalizes legacy interval schedules, adds a Control UI cron guard, tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine/preservation approach, finish the Control UI list boundary so unsupported schedules are dropped or rendered safely, then rebase and land it with the linked regression.

## Reproduction Assessment

Yes. A cron.list response containing a row with a valid payload but missing schedule remains in the default all filter on the PR head, then the unchanged list render path dereferences job.schedule.kind.

## Solution Assessment

No, not yet. Backend quarantine is the right core fix, but the client guard should also prevent unsupported schedules from reaching renderJob or make the presenter handle them safely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less rows before rendering:** `ui/src/ui/controllers/cron.ts:371-375`
  - body: The new helper avoids directly reading `schedule.kind` during filter comparison, but the default `all` filter still keeps rows where `scheduleKind` is null. Filter unsupported schedules out before returning visible jobs, or make the presenter render them safely.
