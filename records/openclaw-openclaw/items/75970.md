## Summary

Keep open. The PR fixes the missing `schedule`/`payload.kind` crash class, but the new hydratability gate still accepts `agentTurn` payloads without a `message`, so a malformed persisted row can still enter runtime/UI consumers and blank the cron dashboard through the existing markdown renderer.

## What This Changes

The PR adds cron store quarantine/preservation for malformed persisted jobs, Control UI cron controller guards, regression coverage, and a changelog entry for the linked dashboard bug.

## Best Possible Solution

Keep the store-boundary quarantine/preservation design, but make hydratability match the actual cron schedule and payload unions before landing.

## Reproduction Assessment

Yes. A persisted job with a valid schedule and `payload: { kind: "agentTurn" }` passes the PR's payload check, then can still hit the Control UI renderer path that calls `.trim()` on a missing message.

## Solution Assessment

No, not yet. The narrow fix is still the right boundary, but the PR must validate required fields for each schedule/payload variant or quarantine those rows too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate payload bodies before hydrating cron jobs:** `src/cron/service/store.ts:45`
  - body: `isSupportedPayload` currently accepts any object whose `kind` is `agentTurn` or `systemEvent`. A persisted row with `payload: { kind: "agentTurn" }` but no `message` therefore bypasses quarantine, enters `state.store.jobs`, and can still crash the cron dashboard when `renderJobPayload` passes `job.payload.message` into `toSanitizedMarkdownHtml`, which calls `.trim()`. Please require the variant's required body field before treating the row as hydratable, and preserve/quarantine the raw row otherwise.
  - confidence: 0.91
