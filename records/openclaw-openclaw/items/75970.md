## Summary

Keep open: the patch addresses the cron store and Control UI crash path, but this external PR still needs after-fix real behavior proof before merge, and I found one low-priority warning dedupe cleanup.

## What This Changes

The PR quarantines malformed persisted cron jobs during store load, preserves raw quarantined rows on later saves, normalizes legacy string interval schedules, adds Control UI guards, tests, and a changelog entry.

## Best Possible Solution

Keep the store-load quarantine and UI defensive filter approach, add or accept the warning-dedupe cleanup, then merge after real behavior proof shows a malformed jobs.json no longer blanks the dashboard.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main can pass a valid-payload, missing-schedule cron row through the Control UI load/filter path until rendering dereferences schedule.kind. I did not run a browser scenario because this review was read-only.

## Solution Assessment

Mostly yes: quarantining malformed persisted rows at the cron store boundary while preserving raw rows, plus a Control UI schedule guard, is the narrow maintainable fix. The remaining improvement is deduping the new repair warning so persistent malformed rows do not spam logs on repeated forceReload ticks.
