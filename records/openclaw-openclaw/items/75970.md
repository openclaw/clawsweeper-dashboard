## Summary

Keep this PR open. The backend/UI direction matches the linked malformed cron-row regression, but the patch currently drops malformed rows from `state.store` without preserving them for later writes, so normal cron persistence can silently remove the rows that the PR says will be left on disk for explicit repair.

## What This Changes

The PR hardens cron store loading to ignore jobs missing valid schedule/payload kinds, adds regression tests for malformed and legacy cron rows, and makes selected Control UI cron controller paths defensive around schedule/payload kinds.

## Best Possible Solution

Preserve unrecoverable raw cron rows across normal store writes while quarantining them from runtime consumers, then add the required Unreleased Fixes changelog entry and land with targeted cron/UI regression coverage.

## Reproduction Assessment

Yes. The linked issue gives a high-confidence malformed `jobs.json` reproduction for the dashboard crash, and the PR's new persistence regression is reproducible by loading a valid-plus-malformed store and then hitting any cron path that persists `state.store.jobs`.

## Solution Assessment

No. Store-boundary validation plus defensive UI code is the right direction, but skipping rows in memory without a write-preservation mechanism is not the narrowest safe fix because it can turn quarantine into deletion.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve skipped rows on later writes:** `src/cron/service/store.ts:101-111`
  - body: `ensureLoaded` now omits malformed rows from `state.store.jobs`, but normal timer/read/start paths later call `persist(state)`, and `saveCronStore` writes exactly the current `store.jobs`. Please keep quarantined raw rows available for persistence, or merge them back when saving.
