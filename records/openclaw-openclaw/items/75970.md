## Summary

Keep this PR open: the quarantine/preservation design is the right direction and the earlier schedule/payload validator holes were addressed, but the current PR head still quarantines legacy interval jobs with string-typed `everyMs` even though current main explicitly supports them.

## What This Changes

The PR adds cron store quarantine/preservation for malformed persisted jobs, Control UI guards around malformed cron rows, regression coverage, and a changelog entry.

## Best Possible Solution

Keep the quarantine/preservation approach, but make the store-boundary validator preserve recoverable legacy interval rows by reusing the existing finite-number coercion or normalizing them before classification.

## Reproduction Assessment

Yes, by source inspection. Current main has an explicit legacy test for string-typed `everyMs`/`anchorMs`, while PR head rejects string `everyMs` in `isSupportedSchedule` before the scheduler can coerce it.

## Solution Assessment

No, not yet. Backend quarantine plus UI fallback is the narrow maintainable direction, but the validator must not classify an already-supported legacy interval shape as malformed.
