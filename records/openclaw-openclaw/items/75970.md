## Summary

Keep open. The PR is a useful narrow fix for malformed persisted cron rows, and the discussion shows the author addressed several backend quarantine/preservation gaps, but the latest head still leaves a Control UI schedule guard hole that can blank the cron page if a schedule-less row reaches the browser.

## What This Changes

The PR quarantines malformed persisted cron jobs from backend runtime reads, preserves their raw rows on later writes, normalizes legacy string interval schedules, adds Control UI guards, backend tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine/preservation approach, but add a client list boundary that rejects unsupported schedules before the cron presenter and cover that path with regression tests.

## Reproduction Assessment

Yes. On PR head, a cron.list row with a valid payload but no supported schedule can remain visible under the default all filter, then render through formatCronSchedule, which dereferences job.schedule.kind.

## Solution Assessment

No, not yet. The backend quarantine is the right narrow direction, but the UI fallback should also drop or safely render unsupported schedules before the jobs view paints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter unsupported schedules before rendering:** `ui/src/ui/controllers/cron.ts:370-383`
  - body: The new helper returns null for unsupported schedules, but the default all filter still returns true. If cron.list returns a row with a valid payload and a missing schedule, it remains visible and renderJob calls formatCronSchedule, which dereferences job.schedule.kind and can blank the cron page. Drop unsupported schedules at the list/filter boundary or render them safely.
  - confidence: 0.9
