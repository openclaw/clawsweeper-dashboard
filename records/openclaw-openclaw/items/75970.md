## Summary

Keep open: the PR is useful and mostly in the right shape, but latest head still accepts kind-only schedules at the store boundary, so malformed persisted rows can still reach runtime/UI consumers instead of being quarantined.

## What This Changes

The PR adds cron store quarantine/preservation for malformed persisted jobs, Control UI guards for malformed cron rows, regression tests, and a changelog entry.

## Best Possible Solution

Keep the quarantine/preservation design, but validate each schedule variant’s required body before hydration and add regression coverage for kind-only schedules.

## Reproduction Assessment

Yes. Source inspection of PR head shows a persisted row with id, a valid payload, and schedule: { kind: "cron" } passes the new validator even though the runtime contract requires expr; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The backend quarantine plus UI fallback is the right maintainable direction, but the store-boundary validator needs to reject invalid schedule bodies before merge.
