## Summary

Keep open. The PR addresses the right backend boundary for malformed persisted cron rows, but it should not merge as-is because the fallback Control UI guard still lets malformed rows through the default render path, and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR drops unhydratable persisted cron jobs during store load, preserves recoverable top-level legacy schedule/payload inference, and adds controller-level cron UI null guards.

## Best Possible Solution

Land a revised fix that keeps the store quarantine, filters or safely renders malformed cron rows across all dashboard paths, and documents the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. Current main has direct `schedule.kind` and `payload.kind` dereferences that match the linked issue's browser error; against this PR, a malformed row in `state.cronJobs` with the default `all` filter still reaches rendering and can reproduce the blank dashboard path.

## Solution Assessment

No. The store-boundary quarantine is the right primary fix, but the PR's UI fallback is incomplete because invalid rows can still reach render helpers, and the required changelog entry is missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter invalid cron rows before rendering:** `ui/src/ui/controllers/cron.ts:371-375`

Filter unrenderable rows here or harden the render path consistently.
