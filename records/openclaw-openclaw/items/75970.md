## Summary

Keep open: the backend quarantine/preservation path is a good fit, but current PR head still leaves a Control UI fallback gap where schedule-less rows can pass the default job filter and reach a renderer that dereferences `job.schedule.kind`.

## What This Changes

The PR quarantines malformed persisted cron jobs during store loading, preserves quarantined raw rows on later writes, accepts legacy string interval schedules, adds backend regression coverage, and hardens parts of the Control UI cron form path.

## Best Possible Solution

Finish the UI boundary by dropping unsupported schedules before list/render state, or by rendering a safe repair placeholder, while keeping the backend quarantine and raw-row preservation behavior.

## Reproduction Assessment

Yes. A `cron.list` result with a valid `systemEvent` payload but no schedule passes the payload-only load filter and the default `all` schedule filter, then the view calls `formatCronSchedule` and dereferences the missing schedule.

## Solution Assessment

No, not yet. The backend quarantine is the right narrow direction, but the PR's Control UI fallback should reject or safely render unsupported schedules at the list/presenter boundary before merge.
