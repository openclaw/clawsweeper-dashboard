## Summary

Keep open: the backend quarantine is on the right path, but current PR head still leaves a Control UI all-filter path where a valid-payload row with no supported schedule can reach rendering and blank the cron page.

## What This Changes

The PR quarantines malformed persisted cron jobs at load time, preserves quarantined rows on saves, normalizes legacy interval schedules, adjusts Control UI cron handling, adds store regression tests, and updates the changelog.

## Best Possible Solution

Keep the backend quarantine/preservation approach, and add a Control UI list/load boundary that rejects unsupported schedules before any cron job row is rendered.

## Reproduction Assessment

Yes. In PR head, a cron.list row with a valid payload and missing schedule passes the payload-only load filter and the default all schedule filter, then renderJob calls formatCronSchedule and dereferences job.schedule.kind.

## Solution Assessment

No, not yet. The backend quarantine is the narrow maintainable fix for persisted jobs, but the client fallback must also drop or safely render unsupported schedules before the list presenter runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less jobs before rendering:** `ui/src/ui/controllers/cron.ts:370-383`
  - body: The new schedule-kind helper only affects non-`all` schedule filters. With the default `all` filter, a `cron.list` row that has a valid payload but no supported `schedule` still remains visible and reaches `renderJob`, where `formatCronSchedule` dereferences `job.schedule.kind` and can blank the page. Filter out `scheduleKind === null` before returning visible jobs, or filter them at the load boundary.
