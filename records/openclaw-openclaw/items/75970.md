## Summary

Keep open. The author addressed the malformed-payload review hole in the follow-up commit, but the latest PR head still lets malformed schedule bodies through by checking only `schedule.kind`, so the quarantine boundary remains incomplete.

## What This Changes

The PR adds cron-store quarantine and preservation for malformed persisted jobs, Control UI cron guards, regression tests, and a changelog entry for #51871.

## Best Possible Solution

Keep the quarantine/preservation design, but make store-boundary hydratability validation match each `CronSchedule` variant's required fields and add malformed-schedule-body regressions.

## Reproduction Assessment

Yes. Add a persisted row with `schedule: { kind: "cron" }` and a valid payload to the PR's malformed-store regression; at PR head it passes `isSupportedSchedule` and reaches runtime scheduling instead of quarantine.

## Solution Assessment

No, not yet. Store-boundary quarantine is the right maintainable fix, but it needs schedule-body validation rather than relying on later runtime schedule-error handling or UI fallbacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate schedule bodies before hydrating persisted jobs:** `src/cron/service/store.ts:37`
  - body: `isSupportedSchedule` accepts any object whose `kind` is `at`, `every`, or `cron`, so a row like `schedule: { kind: "cron" }` with a valid payload bypasses quarantine and enters `state.store.jobs`. That keeps a malformed persisted row in runtime consumers and only fails later during next-run computation; require each schedule variant's body field before treating the job as hydratable.
