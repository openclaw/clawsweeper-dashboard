## Summary

Keep open: the backend quarantine and legacy interval normalization are the right direction, but the UI fallback still lets schedule-less rows reach the default cron list render path if a malformed row reaches the browser.

## What This Changes

The PR quarantines malformed persisted cron jobs during cron store load, preserves quarantined rows on later saves, normalizes legacy interval schedules, adds Control UI guards, tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine/preservation approach, but complete the Control UI boundary so rows without a supported schedule are filtered or rendered safely before the cron list paints.

## Reproduction Assessment

Yes, source-reproducible. A cron.list response containing a row with a valid payload but missing schedule remains visible in the PR head's default all filter, then the render path dereferences job.schedule.kind.

## Solution Assessment

No, not yet. Backend quarantine is the right core fix, but the PR's client fallback should also drop or safely render unsupported schedules at the list boundary or presenter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less rows before rendering:** `ui/src/ui/controllers/cron.ts:370-373`
  - body: The new guard computes `scheduleKind`, but the default `all` filter still keeps rows where that value is null. If a malformed row with a valid payload reaches the browser, `renderJob` still calls `formatCronSchedule(job)`, which dereferences `job.schedule.kind` and can blank the cron page. Filter unsupported schedules out before returning visible jobs, or make the presenter render them safely.
  - confidence: 0.9
