## Summary

Keep this PR open: the backend quarantine/preservation approach is useful, but the current head still leaves a malformed schedule row able to reach the default Control UI cron list and crash rendering.

## What This Changes

The PR quarantines malformed persisted cron jobs, preserves quarantined rows on later cron writes, normalizes string interval schedules, adds Control UI cron guards, tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine/preservation design, then finish the Control UI boundary by dropping unsupported schedules before list rendering or making the presenter schedule-safe, with a regression test for a valid-payload row missing schedule.

## Reproduction Assessment

Yes, source-reproducible. A cron.list response containing a row with a valid payload but missing schedule passes the current payload-only UI ingress and the PR head's default all filter, then renderJob calls formatCronSchedule and dereferences job.schedule.kind.

## Solution Assessment

No, not yet. The backend quarantine is the right core fix, but the PR's client fallback still needs to filter or safely render unsupported schedules in the default list path.
