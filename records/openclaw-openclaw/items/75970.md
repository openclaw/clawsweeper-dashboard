## Summary

Keep this PR open: the backend quarantine/preservation direction is useful, but the latest head still has a source-reproducible Control UI crash path for a row with a valid payload and missing or unsupported schedule in the default cron list view.

## What This Changes

The PR quarantines malformed persisted cron jobs during store load, preserves quarantined raw rows on save, keeps selected legacy schedule coercions, adds Control UI cron guards, tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine and preservation approach, but complete the UI boundary by dropping unsupported schedules before rendering or by making the presenter tolerate them safely.

## Reproduction Assessment

Yes. Source inspection shows a cron.list response row with a valid payload but missing schedule passes the PR head's default all filter, then the render path dereferences `job.schedule.kind`.

## Solution Assessment

No, not yet. Backend quarantine is the right core fix, but the PR's stated client fallback still needs to filter or safely render unsupported schedules before the cron list paints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop schedule-less rows before rendering:** `ui/src/ui/controllers/cron.ts:370-374`
  - body: The new guard computes `scheduleKind`, but the default `all` filter still keeps rows where that value is null. If a malformed row with a valid payload reaches the browser, `renderJob` still calls `formatCronSchedule(job)`, which dereferences `job.schedule.kind` and can blank the cron page. Filter unsupported schedules out before returning visible jobs, or make the presenter render them safely.
  - confidence: 0.9
