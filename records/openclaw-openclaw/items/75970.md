## Summary

Keep open: the backend quarantine and preservation path is aligned with the bug, but PR head still lets a valid-payload row with a missing or unsupported schedule reach the default Control UI job list and crash during schedule rendering.

## What This Changes

The PR filters malformed persisted cron jobs out of cron service loads, preserves quarantined raw rows on later saves, normalizes string interval schedules, adds UI edit guards, tests, and a changelog entry.

## Best Possible Solution

Keep the backend quarantine/preservation design, then finish the Control UI list boundary so unsupported schedules cannot reach rendering in any filter mode.

## Reproduction Assessment

Yes, source-level. On PR head, a `cron.list` row with a valid payload but no `schedule` passes `hasCronJobPayload`, survives the default `all` schedule filter, and reaches `formatCronSchedule`, which dereferences `job.schedule.kind`.

## Solution Assessment

No, not yet. The backend quarantine/preservation approach is the right narrow fix, but the client list or presenter boundary still needs to reject or safely render unsupported schedules.
