## Summary

Keep open: the branch is directionally right, but the store-boundary validator still accepts kind-only schedule objects, so some malformed persisted rows can bypass quarantine and reach runtime state.

## What This Changes

The PR adds cron store quarantine/preservation for malformed persisted jobs, Control UI guards for cron list/edit handling, regression tests, and a changelog entry for #51871.

## Best Possible Solution

Keep the quarantine/preservation design, but make the store-boundary hydratability check validate each schedule variant's required body field and cover kind-only schedules in regression tests.

## Reproduction Assessment

Yes. Add a persisted row with a valid id/payload and `schedule: { kind: "cron" }`; at PR head it passes `isSupportedSchedule` because only `kind` is checked, while the schedule contract and runtime require `expr`.

## Solution Assessment

No, not yet. Backend quarantine plus the UI fallback is the right maintainable shape, but the backend validator must reject invalid schedule bodies before runtime hydration.
