## Summary

Keep open: the PR fixes the earlier payload-body hole and uses the right store-boundary quarantine design, but the final head still hydrates schedule objects that have only a `kind` and no required schedule body fields, so malformed rows can still reach runtime consumers instead of being quarantined.

## What This Changes

The PR adds cron-store quarantine/preservation for malformed persisted jobs, Control UI cron guards, regression tests, and a changelog entry for #51871.

## Best Possible Solution

Keep the quarantine/preservation design, but make the store-boundary hydratability check match each `CronSchedule` variant's required fields before runtime hydration.

## Reproduction Assessment

Yes. Add a persisted row with `schedule: { kind: "cron" }` and a valid payload to the PR's malformed-store regression; the current PR predicate accepts it because `isSupportedSchedule` only checks `kind`.

## Solution Assessment

No, not yet. Store-boundary quarantine is the right maintainable fix, but it needs schedule-body validation rather than relying on later runtime schedule-error handling or UI fallbacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate schedule bodies before hydrating persisted jobs:** `src/cron/service/store.ts:37`
  - body: `isSupportedSchedule` currently accepts any object whose `kind` is `at`, `every`, or `cron`.
