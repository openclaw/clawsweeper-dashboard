## Summary

Keep open: this is a protected-label PR for a real subagent delivery bug, but the proposed fallback currently conflicts with the mediated delivery contract and lacks after-fix live gateway/channel proof.

## What This Changes

The branch adds a `sendMessage` fallback for single subagent completion thread/channel announcements when the requester completion-agent returns no visible payload, plus focused test updates.

## Best Possible Solution

Keep subagent completion delivery mediated; repair the requester-agent handoff or queue/retry path, or approve a narrow mediated fallback with focused tests and live Telegram/channel proof.

## Reproduction Assessment

Yes, at source level: current main returns `completion agent did not produce a visible reply` when an expected completion message has an empty completion-agent payload, and current tests assert that `sendMessage` is not called. I did not run tests because this review is read-only.

## Solution Assessment

No: the proposed direct `sendMessage` fallback is not the safest fix because it bypasses the requester-agent/message-tool delivery contract preserved by current code, docs, and recent merged history. A maintainer should choose a mediated fallback or handoff/retry repair instead.
