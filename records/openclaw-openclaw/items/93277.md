## Summary

Keep open: the structured marker approach is useful, but the PR still stores the marked Discord message IDs only in process memory and the supplied proof is a harness exercise rather than a real Discord/gateway run.

## What This Changes

The PR marks Discord cron outbound payloads with OpenClaw reply-context metadata, records delivered Discord message IDs, and uses those IDs to preserve selected bot-authored reply bodies.

## Best Possible Solution

Persist the marked notification receipt policy in a bounded owner-approved durable state path, then prove a real Discord reply preserves marked notification context while ordinary bot self-replies stay suppressed.

## Reproduction Assessment

Yes, source-level: current main marks Discord bot-authored referenced messages as self and the shared finalizer suppresses their quote bodies. The PR's remaining failure is also source-reproducible because the marked-ID allowlist is a module-local Set that is empty after restart.

## Solution Assessment

No. The structured marker is the right direction, but transient memory is not the best owner for user-visible reply-context policy; it needs durable bounded state or an explicit maintainer decision to accept process-local behavior.
