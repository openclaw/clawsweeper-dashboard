- `merge-risk: 🚨 compatibility`: The PR changes the existing Discord self-reply suppression contract by adding an exception for marked bot-authored notifications.

## Summary

Keep open: the structured marker is a better direction than text matching, but the PR currently stores marked Discord message IDs only in process memory, so replies after a gateway restart can still lose the notification body; the supplied proof is also harness-only rather than real Discord behavior proof.

## What This Changes

The PR marks Discord cron outbound payloads, records delivered Discord message IDs, and uses those IDs to preserve selected bot-authored reply bodies while keeping ordinary self-replies suppressed.

## Best Possible Solution

Persist the marked notification receipt policy in an owner-approved durable store with bounded expiry, then prove a real Discord reply to a marked notification preserves ReplyToBody while ordinary bot self-replies stay suppressed.

## Reproduction Assessment

Yes, source-level: current main marks Discord bot-authored referenced messages as self and the shared finalizer suppresses their quote bodies. The PR's remaining failure is also source-reproducible because the marked-id allowlist is a module-local Set that is empty after restart.

## Solution Assessment

No. The structured marker is the right direction, but using transient memory for a user-visible reply-context policy is not the best fix; the marker needs a durable bounded state owner or an explicit maintainer decision to accept process-local behavior.
