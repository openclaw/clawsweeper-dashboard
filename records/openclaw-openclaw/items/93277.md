## Summary

Keep open: the PR targets a real Discord cron-notification reply-context gap, but the current branch is not merge-ready because the marker state is process-local and the supplied proof is test-harness output rather than a real Discord or gateway run.

## What This Changes

The PR marks Discord cron notification payloads with reply-context metadata, records delivered Discord message IDs, and preserves marked self-authored quote bodies while ordinary bot self-replies stay suppressed.

## Best Possible Solution

Persist marked notification receipt policy in bounded durable Discord/plugin-owned state, then land only after live Discord or gateway proof shows marked notification context is preserved while ordinary bot self-replies stay suppressed.

## Reproduction Assessment

Yes, source-level. Current main marks Discord bot-authored referenced messages as self and the shared finalizer suppresses their body; the PR's remaining restart failure follows from the module-local Set used as the only allowlist.

## Solution Assessment

No. The structured marker is a reasonable direction, but module-local memory is not the right lifetime for a user reply window; the safer solution is durable bounded state plus maintainer acceptance of the self-authored context exception.
