Keep open: current main and the latest release still use a monolithic hosted Unix installer, and no related issue or PR owns the modularization request. The request is a valid maintainability cleanup, but the proposed split is broad public setup-surface work that needs maintainer agreement before implementation.

## What This Changes

The issue requests splitting `scripts/install.sh` into smaller installer modules for UI, download, platform, npm/git install, build-tool, and gum responsibilities.

## Best Possible Solution

Design a maintainer-approved installer source split only if it preserves the hosted single-script artifact and sync workflow, with focused test migration around the extracted boundaries.

## Reproduction Assessment

Not applicable: this is a maintainability refactor request rather than a runtime bug. The source concern itself is verifiable because current main still has a 3,358-line `scripts/install.sh` and no `scripts/install/` module tree.

## Solution Assessment

Unclear. Splitting responsibilities is plausible, but the exact proposed raw module layout is not proven best because the public installer is currently served as a single script; a generated single-file artifact or narrower staged extraction may be safer.
