## Summary

Current and shipped code confirm the maintainability concern remains, but the proposed runtime module tree is not automatically the best fix because `install.sh` is an atomic `curl | bash` artifact; this needs a maintainer-approved generated-source boundary rather than stale cleanup.

## What This Changes

Split `scripts/install.sh` into responsibility-focused shell modules for UI, downloading, platform detection, package installation, and build-tool setup.

## Best Possible Solution

Refactor into responsibility-focused source modules only behind a deterministic generator that emits the single self-contained `scripts/install.sh`, checks that the generated artifact is current, and preserves all smoke, non-root, E2E, npm, git, and platform paths.

## Solution Assessment

No, not exactly as proposed. Responsibility-focused source modules are reasonable, but the reliable solution must generate one atomic published installer rather than make `curl | bash` source a remote module tree.
