## Summary

Keep open: current main and the latest release still ship `scripts/install.sh` as one large public installer script, and the requested `scripts/install/` module tree is not present. The cleanup is plausible, but it needs maintainer direction because the installer is served as a public `openclaw.ai/install.sh` artifact and the sync workflow/tests currently assume that single-file contract.

## What This Changes

The issue requests splitting the monolithic `scripts/install.sh` installer into smaller shell modules for UI, download, platform, npm, git, and build-tool responsibilities.

## Best Possible Solution

Decide the public installer artifact contract first, then refactor behind a preserved self-contained or generated `install.sh` entrypoint with updated sync workflow, docs, and installer smoke coverage.

## Reproduction Assessment

Not applicable. This is a maintainability cleanup request, and source plus release inspection confirm current main and `v2026.5.22` still have one large `scripts/install.sh` with no requested module tree.

## Solution Assessment

Unclear. Modularization may be the right implementation shape, but maintainers need to choose whether the hosted installer stays checked-in standalone, is generated from modules, or uses another public boundary before implementation.
