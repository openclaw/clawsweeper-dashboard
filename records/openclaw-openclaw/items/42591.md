## Summary

Keep open: current main still has the monolithic installer shape described, and modularizing it is valid cleanup but touches the public curl-piped installer, installer docs, Docker smoke, and release-validation surfaces, so it needs maintainer architecture and release/security review before implementation.

## What This Changes

The issue requests splitting `scripts/install.sh` into smaller installer modules for UI, download, platform, npm, git, build-tool, and gum responsibilities.

## Best Possible Solution

Preserve `scripts/install.sh` and `https://openclaw.ai/install.sh` as stable entrypoints while landing a reviewed modularization or bundling design that updates installer docs, unit probes, Docker smoke, and release validation together.

## Reproduction Assessment

Not applicable as a runtime bug; this is a maintainability refactor request. Source inspection is high confidence: current main has no `scripts/install/` module tree and still keeps the requested responsibilities in `scripts/install.sh`.

## Solution Assessment

Unclear whether the exact proposed file tree is the best final shape. A safer solution is a maintainer-approved installer architecture that preserves the public entrypoint and validates curl-piped, npm, git, Docker smoke, and release-check flows.
