## Summary

Keep open. Current main still has the monolithic installer shape described in the report, and modularizing it would change a public curl-piped installer plus docs, tests, Docker smoke, and release-check assumptions, so this needs maintainer architecture/release review rather than conservative cleanup closure.

## What This Changes

The issue requests splitting `scripts/install.sh` into smaller installer modules for UI, download, npm, git, build-tool, and platform logic.

## Best Possible Solution

Preserve `scripts/install.sh` and `https://openclaw.ai/install.sh` as stable public entrypoints, then land a maintainer-approved modularization or build-time bundling plan that updates installer docs, unit probes, Docker smoke coverage, and release validation together.

## Reproduction Assessment

Not applicable as a runtime bug; this is a maintainability refactor request. The source-inspection path is high confidence: current main has no `scripts/install/` module tree and still keeps the requested responsibilities in `scripts/install.sh`.

## Solution Assessment

Unclear whether the exact proposed file tree is the best final shape. The safer solution is a reviewed installer architecture that preserves the public entrypoint and validates curl-piped, npm, git, Docker smoke, and release-check flows.
