## Summary

Keep this PR open. The linked Discord cron report provided a real leaked `to=cron ... {json}` shape, and current `main` still lacks a defensive strip for that exact prefix, but the branch is stale against the split sanitizer module and its matcher can over-strip legitimate leading prose.

## What This Changes

The PR adds a `sanitizeUserFacingText` helper in `src/agents/pi-embedded-helpers/errors.ts` to remove leading `to=<tool> ... {json}` text and adds two sanitizer regression tests.

## Best Possible Solution

Add a conservative prefix sanitizer in `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts`, with tests for the observed Discord `to=cron ... {"action":"create"}` leak and for leading prose examples that must be preserved, plus a single-line changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: a final assistant string like `to=cron ... {"action":"create"}\nSet.` goes through `sanitizeAssistantVisibleText` and `sanitizeUserFacingText` on current main, and neither current stage matches that prefix.

## Solution Assessment

No, not as submitted. The direction is useful, but the branch should be ported to the current split sanitizer module and the prefix detector should be tightened before it removes text from all user-facing delivery paths.
