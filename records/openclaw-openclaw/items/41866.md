## Summary

Keep open. The reported sanitizer gap is still source-reproducible on current main, but this PR is not mergeable as submitted because it patches the stale sanitizer location, can delete legitimate leading prose, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `sanitizeUserFacingText` pass and regression tests to strip leading `to=<tool> ... {json}` prefixes from user-facing agent replies.

## Best Possible Solution

Port a narrow transcript-shaped prefix guard into the active sanitizer or shared reply normalizer, preserve literal prose/code examples, and coordinate the broader envelope-leak remainder with https://github.com/openclaw/openclaw/issues/44905 and https://github.com/openclaw/openclaw/pull/43679.

## Reproduction Assessment

Yes, source-reproducible. Current main routes visible assistant text through `sanitizeUserFacingText`, and the active sanitizer lacks a handler for the leading bare `to=cron ... {"action":"create"}` shape; I did not run a live Discord/TUI reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The fix needs to move to the active split sanitizer or shared normalizer and use a stricter leaked-transcript detector before deleting user-facing text.
