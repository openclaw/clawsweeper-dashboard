## Summary

Keep open. Current main still accepts `audio` and `video` for configured provider model inputs, while current docs describe only text/image behavior and an existing test intentionally preserves those values, so the remaining work needs a product/API contract decision before implementation.

## What This Changes

Add validation guardrails, tests, CI coverage, and docs so unsupported `models.providers.*.models[*].input` modalities are rejected before deployment.

## Best Possible Solution

Define one authoritative configured-model input contract, then align schema, exported types, generated docs/schema, provider-catalog normalization, regression tests, and operator/plugin docs around that contract.

## Reproduction Assessment

Yes for the missing guardrail by source inspection: current schema/type/test paths accept and preserve `audio`/`video` configured inputs. This is not a broken existing-behavior repro; it is a config/API hardening gap.

## Solution Assessment

Unclear as currently scoped. The guardrail is useful, but the best fix is not blanket schema tightening until maintainers decide whether non-text/image values are invalid, future runtime capabilities, or plugin/catalog metadata.
