## Summary

Keep open. Current main still accepts the disputed configured-model `audio` and `video` input modalities through schema, exported config types, generated schema, and configured-provider catalog normalization, while the runtime and docs still describe configured model input behavior in text/image terms. This is not a cleanup-close candidate because resolving it requires a maintainer contract decision: reject those values as unsupported, or formalize what they mean.

## What This Changes

The issue requests stricter validation for `models.providers.*.models[*].input` so unsupported model input modalities fail before deployment.

## Best Possible Solution

Define one explicit configured-model input contract and apply it consistently across source schema, exported types, generated schema, catalog normalization, tests, and docs: either reject unsupported values such as `audio`/`video`, or implement and document their runtime meaning so CI validates them intentionally.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: a configured provider model with `input: ["audio"]` or `input: ["video"]` is accepted by the schema/type/generated-schema path and preserved by configured-provider catalog normalization.

## Solution Assessment

Unclear. The requested schema guardrail may be the right outcome, but it is not safe to treat as a mechanical fix until maintainers decide whether to narrow a public config contract or formalize `audio`/`video` as supported metadata.
