## Summary

Keep open. Current main and the latest release still accept and preserve configured `audio`/`video` model input metadata, while the docs and catalog contracts remain narrower; tightening this now would reverse a merged public config/type change, so the remaining work needs a maintainer product/API decision.

## What This Changes

The issue requests model-input validation guardrails for configured provider models so unsupported modalities fail before deployment.

## Best Possible Solution

Define one configured-model input contract, then align schema, exported types, configured and manifest catalog normalization, runtime capability checks, tests, generated/public docs, and CI around that contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main reproduction path: configure `models.providers.*.models[*].input` with `audio` or `video`, and the schema/type/catalog path accepts and preserves it rather than rejecting it as unsupported.

## Solution Assessment

Unclear. The requested guardrail is plausible, but direct schema rejection is not the best fix until maintainers choose whether shipped `audio`/`video` values are invalid, future-compatible metadata, or a separate capability contract.
