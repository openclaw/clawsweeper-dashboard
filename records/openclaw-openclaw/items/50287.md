## Summary

Keep open. Current main and the latest release still accept `audio`/`video` in configured model inputs while the runtime and docs only define `text`/`image` behavior, so the requested guardrail remains real and needs a maintainer product/API contract decision.

## What This Changes

Add pre-deploy validation guardrails, tests, and docs for unsupported `models.providers.*.models[*].input` modality combinations.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, runtime capability checks, tests, generated docs/schema, and operator docs around that contract.

## Reproduction Assessment

Yes, at source level. Current schema and configured-catalog code accept and preserve `audio`/`video`, while runtime capability checks and docs only define `text`/`image`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not the best fix until maintainers decide whether audio/video are invalid chat inputs, future-compatible metadata, or a separate media capability surface.
