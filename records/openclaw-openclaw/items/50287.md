## Summary

Keep open. Current main and v2026.5.22 still accept and preserve `audio`/`video` in configured model inputs, while the runtime and operator docs still only define text/image behavior for normal model input capability; the requested guardrail remains unresolved and needs a maintainer product/API decision.

## What This Changes

Request to add CI-facing validation for unsupported `models.providers.*.models[*].input` modalities, with invalid/valid tests and docs updates.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, runtime capability checks, tests, generated schema/docs, and operator docs around it.

## Reproduction Assessment

Yes, at source level. Current main's schema, public type, and configured-catalog path accept and preserve `audio`/`video`, while docs and runtime capability checks only define text/image chat input behavior.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not the best fix until maintainers decide whether `audio`/`video` are invalid chat inputs, future-compatible metadata, or a separate media capability surface.
