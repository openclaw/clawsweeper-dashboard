## Summary

Keep open. Current main and v2026.5.22 still accept and preserve `audio`/`video` in configured model inputs, while the runtime and operator docs only give clear chat-input semantics for `text`/`image`; this remains valid work, but it needs a maintainer product/API decision before a fix PR.

## What This Changes

The issue requests early validation guardrails for `models.providers.*.models[*].input` so unsupported model input modality combinations fail before deployment.

## Best Possible Solution

Define one configured-model input contract, then align schema, exported types, catalog normalization, runtime capability handling, tests, generated schema/docs, and operator docs around that decision.

## Reproduction Assessment

Yes, at source level. Current main accepts and preserves `audio`/`video` in the schema/type/catalog path, while the inspected runtime resolver and docs only define normal chat model input behavior for `text` and `image`.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not the best fix until maintainers decide whether `audio`/`video` are invalid chat inputs, future-compatible metadata, or a separate media-capability surface.
