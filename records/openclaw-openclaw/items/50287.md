## What This Changes

The issue requests schema, test, CI, and docs guardrails for unsupported `models.providers.*.models[*].input` modality declarations.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, tests, generated schema/docs, and any needed migration or diagnostic behavior around that contract.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` accept `audio` and `video` in `models.providers.*.models[*].input` through the schema/type path. I did not run a write-producing validation test in this read-only review.

## Solution Assessment

Unclear. The requested guardrail is directionally useful, but the best fix is not a blanket rejection until maintainers decide whether `audio` and `video` are valid future/custom metadata, unsupported chat inputs, or a separate media-understanding contract.
