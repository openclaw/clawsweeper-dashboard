## Summary

Keep open. Current main and the latest release still accept and preserve `audio`/`video` in configured model inputs, while the runtime and operator docs still describe only text/image behavior for normal chat model input capability; this needs a maintainer API/product decision before a guardrail PR.

## What This Changes

The issue requests schema, CI, test, and docs guardrails so unsupported `models.providers.*.models[*].input` modality combinations fail before deployment.

## Best Possible Solution

Define one configured-model input contract, then align schema, exported types, generated schema/docs, catalog normalization, runtime capability checks, tests, and operator docs around that decision.

## Reproduction Assessment

Yes, at source level. Current main's schema, public type, release contents, and configured-catalog test accept/preserve `audio` and `video`, while runtime selection narrows chat model inputs to text/image.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not the best fix until maintainers decide whether `audio`/`video` are invalid chat inputs, future-compatible metadata, or a separate media capability surface.
