## Summary

Keep open. Current main still accepts configured-model `audio` and `video` input values through config validation and catalog preservation, while runtime behavior and docs give those values no clear configured-model contract beyond text/image handling.

## What This Changes

The issue requests stricter validation for `models.providers.*.models[*].input` so unsupported model input modalities fail before deployment.

## Best Possible Solution

Define one explicit configured-model input contract and enforce it consistently across schema, exported types, generated schema, catalog normalization, tests, and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: a configured provider model with `input: ["audio"]` or `input: ["video"]` is accepted by current schema/type/generated-schema paths and preserved by configured-provider catalog normalization.

## Solution Assessment

Unclear. The requested schema guardrail may be right, but it is not safe as a mechanical cleanup until maintainers choose whether to narrow a public config contract or formalize audio/video as supported metadata.
