## Summary

Keep open. Current main and v2026.6.10 still accept `audio`/`video` in configured model inputs and still trust user-declared image capability in media-understanding selection, while related open work argues some `audio`/`video` catalog metadata is valid; the remaining guardrail needs maintainer product/API direction.

## What This Changes

Add validation, tests, CI failure behavior, and docs so unsupported `models.providers.*.models[*].input` modality configurations are rejected before deployment.

## Best Possible Solution

Define one capability-precedence contract for configured model inputs, then enforce it consistently across schema, exported types, provider/catalog normalization, validation, tests, docs, and any needed doctor migration or explicit override path.

## Reproduction Assessment

Yes, at source level. Current main accepts `audio`/`video` configured model input metadata and media-understanding selection trusts user-declared `image` input without a catalog-capability cross-check; I did not run a live deployment failure repro in this read-only review.

## Solution Assessment

Unclear as a concrete implementation. The requested guardrail is aligned with safer validation, but blanket schema tightening is not the best fix until maintainers decide the capability authority, compatibility behavior, and any migration or override path.
