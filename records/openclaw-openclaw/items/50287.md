## Summary

Keep open. Current main still accepts `audio` and `video` for configured model inputs and intentionally preserves those values in configured provider catalogs, while the docs only describe text/image use; changing this needs a maintainer decision on the public config contract.

## What This Changes

The issue requests validation, tests, CI failure, and docs for rejecting unsupported `models.providers.*.models[*].input` modalities before deployment.

## Best Possible Solution

Define one explicit configured-model input contract, then enforce it consistently across schema, exported types, provider-catalog normalization, tests, docs, and CI.

## Reproduction Assessment

Yes. Source inspection on current main shows configured `input: ["audio"]` or `input: ["video"]` is accepted by schema/type paths and preserved by configured-catalog normalization.

## Solution Assessment

Unclear. The requested guardrail is plausible, but the best implementation depends on deciding whether `audio`/`video` are invalid configured model inputs or valid metadata; blanket schema tightening conflicts with current public type and tests.
