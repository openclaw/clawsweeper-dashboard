## Summary

Keep open. Current main and the latest release still accept `audio` and `video` for configured model input metadata, and current tests intentionally preserve those values, so the requested guardrail remains unresolved and needs a maintainer product/API decision before implementation.

## What This Changes

The issue requests schema validation, tests, CI failure, and docs for rejecting unsupported `models.providers.*.models[*].input` modality configurations before deployment.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, generated schema/docs, regression tests, and any required doctor migration around that contract.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` accept `audio` and `video` through the schema/type path, and the provider-catalog test preserves them. I did not run a write-producing validation test in this read-only review.

## Solution Assessment

Unclear. The requested guardrail is directionally aligned with safer config validation, but a blanket enum rejection is not the best fix until maintainers decide how configured audio/video metadata should behave.
