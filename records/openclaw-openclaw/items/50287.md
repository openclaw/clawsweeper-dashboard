## Summary

Keep open. Current main and the latest release still accept and preserve `audio`/`video` configured model input metadata, while docs and LLM runtime checks remain centered on `text`/`image`, so this is still necessary and needs a maintainer product/API decision rather than cleanup closure.

## What This Changes

The issue requests schema, tests, CI, and docs guardrails so unsupported configured model input modalities are rejected before deployment.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, runtime capability checks, tests, generated docs, and operator docs around that contract.

## Reproduction Assessment

Yes. Source inspection shows that configuring `models.providers.*.models[*].input` with `audio` or `video` is accepted by the schema and preserved by configured-catalog code instead of failing before deploy.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not the best fix until maintainers decide whether audio/video are invalid chat inputs, future-compatible metadata, or a different media capability surface.
