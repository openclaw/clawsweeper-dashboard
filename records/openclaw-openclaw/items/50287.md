## Summary

Keep open. Current main and the latest release still accept `audio` and `video` in configured model input metadata, while docs and runtime-facing behavior do not define a capability-aware guardrail for rejecting unsupported modality combinations.

## What This Changes

The issue requests validation guardrails for `models.providers.*.models[*].input`, targeted valid/invalid tests, early CI failure, and fallback/model-tier docs updates.

## Best Possible Solution

Define one authoritative configured-model input contract, then align schema, exported types, catalog normalization, tests, docs, and any needed doctor migration around that contract.

## Reproduction Assessment

Yes, at source level: current main validates and preserves configured `audio` and `video` model input values through the schema/type/catalog path. I did not run a live or write-producing validation test in this read-only review.

## Solution Assessment

Unclear. A guardrail is directionally useful, but a blanket rejection is not the best fix until maintainers decide the intended `audio`/`video` contract and the authoritative capability source.
