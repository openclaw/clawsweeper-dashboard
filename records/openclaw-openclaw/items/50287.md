## Summary

Keep open. Current main still accepts and preserves `audio` and `video` in configured model input metadata, while the contributor/operator docs describe only text and image behavior, so this remains a real config-contract decision rather than a cleanup close.

## What This Changes

Add schema, tests, CI-facing validation, and docs so unsupported `models.providers.*.models[*].input` modalities are rejected before deployment.

## Best Possible Solution

Define one explicit configured-model input contract, then enforce it consistently across schema, exported types, catalog normalization, tests, docs, and CI.

## Reproduction Assessment

Yes. Source inspection on current main shows `input: ["audio"]` or `input: ["video"]` is accepted by the schema/type path and preserved by configured-catalog normalization.

## Solution Assessment

Unclear. The requested guardrail is plausible, but the best fix depends on deciding whether `audio` and `video` are invalid configured inputs or supported metadata; a blanket schema change would fight current public types and tests.
