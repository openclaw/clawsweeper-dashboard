## Summary

Keep open. Current main and the latest release still accept and preserve `audio`/`video` configured model input metadata, while the runtime and operator docs only define text/image chat-model input behavior; the guardrail remains a real product/API contract decision.

## What This Changes

Request to add schema/CI guardrails that reject unsupported `models.providers.*.models[*].input` modalities and document validation expectations.

## Best Possible Solution

Define one explicit configured-model input contract, then align schema, exported types, catalog normalization, runtime capability checks, tests, and docs around either rejecting unsupported modalities or implementing/documenting their runtime meaning.

## Reproduction Assessment

Yes, at source level. Current main's schema, public type, and configured-catalog path accept and preserve `audio`/`video`, while docs and the inline provider runtime only define text/image chat input behavior.

## Solution Assessment

Unclear. A guardrail is appropriate, but schema-only rejection is not safe until maintainers decide whether audio/video are invalid chat inputs, future-compatible metadata, or a separate media capability surface.
