## Summary

Keep open. Current main and v2026.6.10 still accept configured audio/video input metadata and still select media-understanding image models from user-declared `input` capability without a catalog-capability guardrail, so the remaining work needs maintainer product/API direction rather than cleanup closure.

## What This Changes

The issue requests schema, test, CI, and docs guardrails so unsupported `models.providers.*.models[*].input` modality declarations fail before deployment.

## Best Possible Solution

Define one capability-precedence contract for configured model inputs, then enforce it consistently across schema, exported types, provider/catalog normalization, validation, tests, docs, and any needed doctor migration or explicit override path.

## Reproduction Assessment

Yes. Source inspection shows current main accepts audio/video modality tokens in configured model input metadata and image selection trusts user-declared `input` metadata without catalog-capability validation; I did not run a live deployment failure repro in this read-only review.

## Solution Assessment

Unclear as a concrete implementation. The guardrail is aligned with safer validation, but blanket schema tightening is not the best fix until maintainers decide the capability authority, compatibility behavior, and any migration or override path.
