## Summary

Keep open: current main still has a source-level path where legacy/context-engine compaction can carry an `openai-codex:*` auth profile into the embedded PI direct compaction path for `openai/gpt-*`, while the related `/btw` fix only covered side questions.

## What This Changes

The issue reports Codex OAuth-backed compaction routing through public OpenAI Responses and failing with missing `api.responses.write` scope.

## Best Possible Solution

Route legacy/default compaction for Codex-harness OpenAI sessions through the selected Codex harness/native compaction path, or fail early/drop incompatible auth before any public OpenAI Responses summarization.

## Reproduction Assessment

Yes, at source level: use `openai/gpt-5.5` with `authProfileId=openai-codex:*`, no compaction model override, and default legacy context-engine compaction; the runtime context is preserved into `compactEmbeddedPiSessionDirect()` where public `openai` auth/runtime is prepared. I did not run a live OAuth token reproduction.

## Solution Assessment

Yes, the narrow fix should be in shared compaction runtime routing, not channel-specific handling or documentation. The safest direction is to make the legacy delegate/harness path honor selected Codex native compaction before falling back to PI direct summarization.
