## Summary

This should stay open: current main still has a source-level path where built-in compaction preserves an `openai-codex:*` auth profile while resolving the compaction model as public `openai/*`, even though the docs say explicit PI plus Codex OAuth should route through the Codex-auth transport.

## What This Changes

The issue requests fixing compaction auth routing so Codex OAuth sessions do not summarize through public OpenAI Responses with a Codex OAuth token.

## Best Possible Solution

Update compaction auth/runtime resolution so OpenAI PI compaction with `openai-codex:*` uses the Codex-auth provider/transport or fails before public Responses summarization, with focused regression coverage.

## Reproduction Assessment

Yes, source-level: configure explicit PI `openai/gpt-5.5` with an `openai-codex:*` auth profile and trigger built-in compaction; current main preserves that profile while resolving the compaction model as public `openai`. I did not perform a live OAuth run in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make compaction share the same OpenAI PI/Codex auth-provider routing used by normal embedded attempts, or to fail early on incompatible auth before calling public Responses.
