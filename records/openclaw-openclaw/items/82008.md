## Summary

Keep open: current main still has a source-level compaction auth-routing path where `openai/*` compaction can carry an `openai-codex:*` auth profile toward public OpenAI Responses instead of the Codex-compatible transport, matching the reported beta log.

## What This Changes

The issue reports that Codex OAuth-backed session compaction can preserve an `openai-codex:*` auth profile while routing summarization through public OpenAI Responses.

## Best Possible Solution

Unify compaction target/auth routing with normal OpenAI runtime planning so Codex OAuth never reaches public Responses, while preserving explicit compaction model overrides.

## Reproduction Assessment

Yes, at source level: use an OpenAI `openai/gpt-5.5` session with an `openai-codex:*` OAuth profile and no compaction model override, then trigger compaction. I did not run a live OAuth session, but current main preserves that auth profile through compaction target resolution and can resolve the target as public `openai`.

## Solution Assessment

Yes. The narrow maintainable fix is to make compaction share the same OpenAI PI/Codex auth-provider routing used by normal attempts, or fail before calling public Responses with incompatible Codex OAuth credentials.
