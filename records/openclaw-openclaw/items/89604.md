## What This Changes

This PR threads `skipDeferral` through gateway restart intent emission, makes close-stage reply draining return `0` for that intent, and adds scheduler regression tests.

## Best Possible Solution

Refresh the branch and land the narrow intent-threading fix only if maintainers accept that explicit `--skip-deferral` restarts bypass close-stage reply drain while ordinary restarts keep graceful drain behavior.

## Reproduction Assessment

Yes, at source level: current main stores `skipDeferral` for scheduling but does not carry it into the restart intent consumed by close-stage shutdown, so normal pending-reply drain timing still applies.

## Solution Assessment

Yes, if maintainers accept the escape-hatch semantics. Threading the existing restart intent boundary is narrower than adding a side channel; self-reply filtering or a bounded drain is the safer alternative only if skipped reply draining is not acceptable.
