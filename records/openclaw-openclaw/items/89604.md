## Summary

Keep open: this is a focused candidate fix for the linked gateway skip-deferral bug, but it is still maintainer-review only because the branch is currently conflicting and the patch intentionally changes the escape-hatch semantics to abandon close-stage pending replies instead of waiting on them.

## What This Changes

The PR threads `skipDeferral` through gateway SIGUSR1 restart intent handling and makes skip-deferral restarts use a zero close-stage pending-reply drain timeout, with scheduler regression tests.

## Best Possible Solution

Refresh the branch and land the narrow intent-threading approach only after maintainers explicitly accept that `--skip-deferral` bypasses close-stage reply draining while ordinary restarts keep graceful drain behavior.

## Reproduction Assessment

Yes, at source level: current main forwards `skipDeferral` into restart scheduling but does not carry it into the restart intent consumed by close-stage shutdown, so normal pending-reply drain timing still applies. I did not run a live channel-originated restart with pending replies.

## Solution Assessment

Yes, if maintainers accept the escape-hatch semantics. Threading the existing restart intent boundary is narrower than adding a side channel; self-reply filtering or a bounded drain is the safer alternative only if zero close-stage drain is not acceptable.
