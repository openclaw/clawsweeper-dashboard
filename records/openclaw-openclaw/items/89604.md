## What This Changes

This PR adds a `skipDeferral` restart-intent flag, threads it through the timer and active-deferral bypass paths, makes the run loop use a zero close-stage drain timeout for that intent, and adds scheduler regression tests.

## Best Possible Solution

Refresh mergeability if needed, keep the narrow intent-threading fix, preserve ordinary safe-restart reply draining, and land only after maintainers accept the explicit skip-deferral reply-delivery semantics.

## Reproduction Assessment

Yes, at source level. Current main forwards `skipDeferral` into restart scheduling, but the scheduler emits SIGUSR1 without carrying that bit and the run loop still computes a normal close-stage reply-drain timeout; I did not live-run the channel restart path.

## Solution Assessment

Yes, if maintainers accept the escape-hatch tradeoff. Threading the bit through the existing restart intent boundary is narrower than adding a second side channel, while self-reply filtering would be a larger design only needed if skipped reply draining is not acceptable.
