## Summary

The reported current-main gap is real, but this conflict-marked branch is not merge-ready: it predates the canonical file-backed restart-intent contract, and its zero timeout applies to every globally pending reply rather than only the restart-requesting turn. The PR should stay paired with the open bug while maintainers choose the intended delivery-loss boundary and the contributor updates the implementation and proof.

## What This Changes

The PR propagates `skipDeferral` into restart intent handling, selects a zero close-stage pending-reply drain timeout for that intent, and adds scheduler-path regression tests.

## Best Possible Solution

Rebase onto current main, carry one explicit skip-drain policy through every in-process and persisted restart-intent path, and add shutdown-level coverage proving normal restarts preserve delivery while the explicit skip mode has a maintainer-approved abandonment boundary.

## Reproduction Assessment

Yes. Current main keeps `skipDeferral` only long enough to bypass pre-restart polling, then emits an intent that cannot communicate that decision to close-stage reply draining; the exact live channel outcome was not independently run in this review.

## Solution Assessment

No in its current form. Typed intent propagation is the right architectural boundary, but the patch must cover current main's persisted handoff and requires a maintainer decision on whether the explicit mode may abandon all globally pending replies.
