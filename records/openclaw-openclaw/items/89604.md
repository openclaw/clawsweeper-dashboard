## Summary

Keep open: current main and the latest release still lack the skip-deferral intent bit at close-stage shutdown, and this PR remains the focused candidate fix. It is not ready for cleanup close because the branch is currently dirty against main and the zero-drain escape-hatch semantics still need maintainer acceptance before merge.

## What This Changes

The PR threads `skipDeferral` through gateway restart intent handling so explicit safe skip-deferral restarts use a zero close-stage pending-reply drain timeout, with regression coverage for timer and active-deferral bypass paths.

## Best Possible Solution

Rebase onto current main, preserve the newer restart hook/session-guard behavior, keep the narrow intent-threading fix, and land only after maintainers accept the explicit skip-deferral zero-drain semantics.

## Reproduction Assessment

Yes, at source level. Current main forwards `skipDeferral` to restart scheduling, then emits SIGUSR1 without carrying that bit and computes the normal close-stage pending-reply drain timeout.

## Solution Assessment

Yes, if maintainers accept the escape-hatch tradeoff. Carrying the bit through the existing restart intent boundary is narrower than adding a side channel, while self-reply filtering would be a larger design only needed if skipped reply draining is not acceptable.
