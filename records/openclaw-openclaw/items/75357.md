## Summary

Keep open. Current `main` still has a plausible source-level gap for ordinary simple `openai-completions` sessions: the usage-aware OpenClaw completions transport requests and parses streaming usage, but the simple-completion route only switches to that transport when request proxy/TLS overrides are attached. The issue is also already paired with an open implementation PR, so it should not be closed before that PR is reviewed, landed, or explicitly replaced.

## What This Changes

The issue asks OpenClaw to request `stream_options.include_usage=true` and persist non-zero token usage for simple `openai-completions` sessions against usage-capable OpenAI-compatible endpoints such as llama.cpp.

## Best Possible Solution

Land or adapt the linked implementation so every streaming `openai-completions` simple request sends or preserves `stream_options.include_usage=true` without requiring proxy/TLS overrides, while preserving provider-owned wrappers and adding regression coverage for payload injection plus usage-only final chunks.

## Reproduction Assessment

Yes. The issue includes concrete llama.cpp curl checks showing usage appears only when `stream_options.include_usage=true`, and current source shows the no-proxy/no-TLS simple path can avoid the transport that sends and parses that usage.

## Solution Assessment

Yes in direction. The narrow maintainable fix is payload injection or equivalent routing through the existing usage-aware OpenAI completions stream path; session-log repair or asking users to configure fake proxy/TLS overrides would be less direct.
