## Summary

Keep open. Current main still has a source-level gap for simple no-proxy/no-TLS `openai-completions`: the custom OpenClaw transport requests and parses streaming usage, but the simple path can remain on the pinned pi-ai dependency path where `stream_options.include_usage` is omitted when compat is forced false for generic non-native endpoints. An open linked fork PR proposes the narrow payload-wrapper fix, so this issue should wait for that work to be reviewed or ported rather than be closed.

## What This Changes

Request that simple no-proxy/no-TLS `openai-completions` sessions request streaming usage and persist non-zero token usage from compatible OpenAI-style endpoints.

## Best Possible Solution

Land a target-repository fix that ensures every simple streaming `openai-completions` payload includes `stream_options.include_usage=true` while preserving existing provider wrappers and unsupported-provider behavior. A small payload wrapper around the simple pi-ai stream path is likely narrower than routing all no-proxy completions through the custom transport; #73990 can remain the broader fallback-estimation follow-up for endpoints that still do not return usage.

## Reproduction Assessment

Yes. The issue includes concrete llama.cpp curl checks, and source inspection reproduces the route condition: no-proxy/no-TLS simple completions can avoid the OpenClaw transport, while the pinned pi-ai path omits `stream_options` when compat is false.

## Solution Assessment

Yes, with a narrower implementation shape. The maintainable fix is to request streaming usage on the simple `openai-completions` payload path, either by porting the linked wrapper PR or an equivalent generic seam, rather than repairing zero usage after session persistence.
