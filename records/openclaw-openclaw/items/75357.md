## Summary

Keep open. Current main still has a narrow simple-session routing gap: the OpenClaw completions transport requests and parses streaming usage, but no-proxy/no-TLS simple `openai-completions` runs can stay on the pinned pi-ai path where OpenClaw's local-endpoint compat normalization suppresses `stream_options.include_usage`. An open fork PR already references this issue with closing syntax, so this should stay open until that PR is reviewed, landed, or superseded.

## What This Changes

Request that simple `openai-completions` sessions send `stream_options.include_usage=true` for compatible streaming endpoints and persist nonzero token usage in session JSONL.

## Best Possible Solution

Review and land, or consciously supersede, the linked implementation so the simple `openai-completions` path requests streaming usage without requiring proxy/TLS overrides. The final fix should preserve provider-owned wrappers and add regression coverage for merging existing `stream_options` plus usage-only final chunks.

## Reproduction Assessment

Yes. The issue provides a concrete llama.cpp curl reproduction, and current source plus pinned dependency source show a deterministic path where a no-proxy/no-TLS simple run can omit `stream_options.include_usage` and therefore persist zero usage.

## Solution Assessment

Yes. The narrow maintainable direction is to inject or route streaming usage at the simple `openai-completions` request boundary, reusing existing payload-wrapper or OpenClaw transport seams rather than repairing persisted session logs after the fact.
