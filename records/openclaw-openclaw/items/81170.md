## Summary

Keep open: the PR fixes the forced `openai` lookup after the OpenAI adapter is invoked, but source review shows the reported `api: "openai-completions"` custom-provider path can still fail before that adapter is selected.

## What This Changes

The PR changes the OpenAI memory embedding adapter and resolver to pass `options.provider ?? "openai"` into remote-client lookup/cache data and adds focused regression tests.

## Best Possible Solution

Handle the OpenAI-compatible memory adapter selection for custom providers whose `api` is `openai-completions` while preserving the original custom provider id through remote lookup/cache data, with a regression test that starts from `memorySearch.provider`.

## Reproduction Assessment

Yes, at source level. A custom `memorySearch.provider` with `models.providers.<id>.api: "openai-completions"` is resolved through ids that do not match the OpenAI memory adapter id, and current main also forces `provider: "openai"` if the adapter is reached.

## Solution Assessment

No. Preserving `options.provider ?? "openai"` is necessary, but this PR does not yet prove or repair the preceding adapter-owner lookup for the reported `openai-completions` custom-provider config.
