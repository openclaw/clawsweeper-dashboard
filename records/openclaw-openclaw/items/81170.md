## Summary

Keep open. The PR is a focused, source-backed fix for a current-main provider-id bug, has sufficient terminal real-behavior proof, and I found no blocking review findings.

## What This Changes

The PR preserves the caller-supplied OpenAI-compatible memory embedding provider id through adapter creation, remote-client lookup, and cache-key data, with regression tests for custom and default provider paths.

## Best Possible Solution

Land the focused provider-id preservation with regression tests after normal maintainer review, then close https://github.com/openclaw/openclaw/issues/47884 once merged.

## Reproduction Assessment

Yes. Current main source shows a custom `memorySearch.provider` can select the OpenAI adapter through `models.providers.<id>.api`, but the adapter and resolver force `provider: "openai"` before the remote client reads `models.providers[params.provider]`; the PR body also includes pre/post terminal output from a local HTTP listener.

## Solution Assessment

Yes. Preserving `options.provider ?? "openai"` at both OpenAI embedding call sites is the narrowest maintainable fix because adapter identity stays `openai` while only the provider-config lookup key and cache key change.
