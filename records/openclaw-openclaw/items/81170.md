## Summary

Keep open. The PR fixes part of the provider-id plumbing, but it is incomplete for the linked bug because a custom provider with `api: "openai-completions"` still does not select the OpenAI memory embedding adapter, which is registered and declared as `openai`.

## What This Changes

The PR forwards `options.provider ?? "openai"` through the OpenAI memory embedding adapter/resolver and adds regression tests for custom-provider forwarding and default fallback.

## Best Possible Solution

Add a lookup/alias bridge so custom OpenAI-compatible providers with `api: "openai-completions"` select the OpenAI memory embedding adapter, then preserve the custom provider id through endpoint/auth lookup with full regression coverage.

## Reproduction Assessment

Yes, at source level. The reported custom provider lookup walks through `bailian-embedding` and `openai-completions`, while the OpenAI memory adapter is exposed as `openai`, so the normal flow still misses the adapter before the PR's preserved provider id can matter.

## Solution Assessment

No. Preserving `options.provider ?? "openai"` is necessary after adapter selection, but the PR does not solve selection for the reported `api: "openai-completions"` configuration.
