## Summary

Keep open for maintainer handling: current main still has the hardcoded OpenAI provider lookup, and this PR is a focused bug fix with sufficient terminal proof, passing broad CI, and no blocking review findings.

## What This Changes

The PR preserves caller-supplied OpenAI-compatible memory embedding provider IDs through the OpenAI adapter/client lookup and adds regression tests for custom IDs and the OpenAI fallback.

## Best Possible Solution

Land the focused provider-id preservation with regression tests, then close https://github.com/openclaw/openclaw/issues/47884 after the fix reaches main.

## Reproduction Assessment

Yes. Current main source shows a custom `memorySearch.provider` can select the OpenAI adapter through `models.providers.<id>.api`, but the adapter and resolver then force `provider: "openai"` before the remote client reads `models.providers[params.provider]`.

## Solution Assessment

Yes. Preserving `options.provider ?? "openai"` at both OpenAI embedding call sites is the narrowest maintainable fix because it keeps adapter identity stable while changing only the provider-config lookup key and cache key.
