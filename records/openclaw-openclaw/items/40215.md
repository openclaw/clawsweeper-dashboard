## Summary

Keep open: current main and v2026.6.5 still render `/usage` footers from per-response token/cache/cost data only, and the requested inherited `agents.defaults.responseUsage` config is not in the agent-default schema. The open linked implementation PR still owns the work and is unmerged, so this issue should remain the canonical tracker until that PR is repaired, replaced, or merged.

## What This Changes

The issue requests cumulative context-window usage, model metadata, and an inherited response-usage default in reply usage footers.

## Best Possible Solution

Extend the existing usage formatter and reply-session presentation path with model and cumulative context/window values, then add a narrowly specified inherited default with explicit override semantics; keep the broader custom footer API separate unless maintainers approve it.

## Reproduction Assessment

Yes, at source level. Current main’s formatter, command handler, config schema, status renderer, and docs show that context/model data exists for `/status` but is not included in `/usage` footers and that the footer mode is session-scoped.

## Solution Assessment

Unclear for the linked PR: the user problem is valid, but the broad open PR has not proven that a generic `messages.responseFooter` API is the best solution. A narrower formatter/session-default change is the safer baseline unless maintainers approve the broader API.
