## Summary

Keep open. Current main still routes mutating tool errors through a warning branch that does not apply recoverable-error suppression, and the test suite still codifies a recoverable mutating error producing a user-facing failure payload. Existing edit recovery and same-action stale-error clearing are useful partial safeguards, but they do not fully resolve this report; the related PR was closed unmerged.

## What This Changes

The issue requests suppressing false-positive edit/write failure notifications when a mutating tool write has already succeeded or been recovered.

## Best Possible Solution

Fix the embedded runner/tool-handler path so mutating failure notifications are suppressed only when the mutation is proven applied, explicitly recovered, or cleared by a later matching successful mutation. Keep unresolved mutating failures visible, and add focused coverage for edit/write false positives plus real mutating failures that must still render.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a recoverable mutating `lastToolError` into `buildEmbeddedRunPayloads`; current main still emits a `⚠️ ... failed` payload, and the existing payload error suite asserts that behavior.

## Solution Assessment

No, the issue's proposed one-line change is not clearly the safest complete fix. The better path is a targeted recovery/applied-state signal or stale-error clearing rule so genuine mutating failures remain visible.
