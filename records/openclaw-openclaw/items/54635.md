## Summary

Keep open. Current main and v2026.5.7 have partial recovery for same-target edit/write self-heal, but the broader warning-policy gap remains: a recoverable mutating `lastToolError` still renders a user-visible failure warning, and the regression suite explicitly preserves that behavior today.

## What This Changes

The issue requests suppressing false `Edit failed` notifications for mutating tools when the write/edit ultimately succeeded or was recovered.

## Best Possible Solution

Suppress final failure notifications only when a mutating write/edit error is provably recovered or cleared, while preserving visible warnings for unresolved mutating failures.

## Reproduction Assessment

Yes, at source level. Passing a recoverable mutating `lastToolError` into `buildEmbeddedRunPayloads` still reaches the mutating warning branch, and a checked-in test expects that warning today.

## Solution Assessment

No, the proposed one-line recoverable-error suppression is not clearly the safest fix. The maintainable solution should tie suppression to proven recovery or correct stale-error clearing, not to every recoverable-looking mutating error string.
