## Summary

Keep open. Current main and the latest v2026.5.18 release still route recoverable-looking mutating tool errors through the visible warning path, while the related open PR only addresses the `messages.suppressToolErrors` configuration case and does not settle the broader proven-recovery warning policy.

## What This Changes

The issue asks OpenClaw to stop showing false `Edit` or `Write` failed notifications for mutating tools when the file write actually succeeded or the error was recoverable.

## Best Possible Solution

Adopt a maintainer-approved warning policy that clears or suppresses mutating failure notifications only when recovery or configured suppression is proven, while preserving visible warnings for unresolved mutating failures.

## Reproduction Assessment

Yes, at source level. Passing a recoverable-looking mutating `lastToolError` into `buildEmbeddedRunPayloads` still reaches the visible mutating-warning branch, and current tests assert that behavior.

## Solution Assessment

No, the proposed one-line recoverable-keyword suppression is not clearly the safest fix. The maintainable direction is proof-based state clearing or a maintainer-approved delivery policy that does not hide real unresolved mutating failures.
