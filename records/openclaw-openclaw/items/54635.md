## Summary

Keep open. Current main has the narrower same-target edit/write recovery from the merged related PR, but the broader warning-policy gap remains: a recoverable-looking mutating `lastToolError` still produces a user-visible failed payload, and current tests explicitly preserve that behavior.

## What This Changes

The issue reports false positive mutating-tool failure notifications after successful writes and asks for recoverable mutating errors to stop producing user-visible failed payloads.

## Best Possible Solution

Keep this open for a maintainer-owned fix that clears or suppresses mutating failure notifications only when recovery is proven, while preserving visible warnings for unresolved mutating failures.

## Reproduction Assessment

Yes, at source level. Passing a recoverable mutating `lastToolError` into `buildEmbeddedRunPayloads` still reaches the mutating warning branch, and current tests expect a warning for that case.

## Solution Assessment

No, the proposed one-line recoverable-keyword suppression is not clearly the safest fix. The narrower maintainable direction is to clear stale mutating errors when recovery is proven, not to suppress every recoverable-looking mutating error string.
