## Summary

Keep open. Current main still emits mutating-tool failure warnings through a branch that does not apply recoverable-error suppression, and the latest release tag has the same policy; the related PR is closed unmerged, so there is no implemented or canonical merged fix to close against.

## What This Changes

The issue reports false-positive `Edit failed` notifications for mutating tools after successful writes and asks the warning policy to suppress recoverable or proven-applied failures.

## Best Possible Solution

Fix the embedded runner tool-error path so false mutating failure notifications are suppressed only when the mutation is confirmed recovered/applied or a later matching mutation clears the stale error, while genuine unresolved mutating failures still render. Add focused regression coverage for edit/write false positives and for real mutating failures that must remain visible.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: pass a recoverable mutating `lastToolError` into `buildEmbeddedRunPayloads` and current policy still emits a `⚠️ ... failed` payload; the existing test suite already asserts that behavior. I did not run tests because this review was read-only.

## Solution Assessment

No, the issue's one-line proposed fix is not clearly the safest complete solution. The maintainable fix should use a proven-applied or explicit recovery signal, plus stale-error clearing and regression tests, rather than blindly suppressing every recoverable-looking mutating error.
