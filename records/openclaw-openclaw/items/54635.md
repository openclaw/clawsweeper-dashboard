## Summary

Keep open. Current main still has the reported warning-policy behavior for mutating tool errors: the mutating branch suppresses only when the assistant already emitted an error/failure acknowledgement, while the recoverable-error guard remains limited to the later non-mutating path. Related #59868 and #60664 are useful context, but #60664 closed unmerged and the current source/tests still preserve this false-positive warning path.

## What This Changes

Suppress false positive mutating-tool `failed` notifications when edit/write operations are recoverable or have already succeeded.

## Best Possible Solution

Implement a focused runner/tool-error fix that suppresses false mutating `failed` chat payloads only when the mutation is proven recovered/applied or the error is safe internal validation noise, while preserving visible warnings for genuine unresolved mutating failures. Add payload and tool-error-state regression coverage for recovered edit/write failures and for real permission or delivery failures that must still surface.
