## Summary

Keep this PR open. Current `main` still has the Telegram model-picker fallback bug, while the proposed patch is narrow, uses the existing agent-aware resolver, and includes focused regression coverage; the remaining action is maintainer review and current-head validation, not cleanup closure.

## What This Changes

The PR changes Telegram model-picker callback session reconstruction to use `resolveDefaultModelForAgent({ cfg, agentId })` for the no-override fallback and adds a regression test for an agent-level default that differs from the global default.

## Best Possible Solution

Land this patch or an equivalent small change after validation: compute the routed agent's effective default for `resolveTelegramSessionState()` when no override exists, and keep the regression test proving the Telegram picker marks an `agents.list[].model` default when it differs from `agents.defaults.model`. A minor cleanup can reuse the resolved default for both stored-override provider resolution and final fallback.

## Reproduction Assessment

Yes. A high-confidence reproduction is a Telegram `mdl_list_<provider>_1` callback with no stored model override, routed to an agent whose `agents.list[].model` differs from `agents.defaults.model`; current `main` returns the global default as `sessionState.model`, so exact matching does not mark the agent default.

## Solution Assessment

Yes. The PR uses the existing agent-aware resolver at the narrow Telegram callback fallback point and adds focused regression coverage; avoiding the duplicate resolver call would be a cleanup, not a safer alternative to the proposed behavior.
