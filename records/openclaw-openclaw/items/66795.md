## Summary

Keep this PR open: current main still has the Telegram model-picker fallback bug, and the proposed patch is a narrow fix using the existing agent-aware model resolver with focused regression coverage. I found no blocking review or security finding, but the exact PR head still needs maintainer validation because broad CI checks on the old head include unrelated failures.

## What This Changes

The branch changes Telegram model-picker callback session reconstruction to fall back to the routed agent's default model and adds a regression test for an agent-level default that differs from the global default.

## Best Possible Solution

Land this patch or an equivalent one after current-head validation: compute the routed agent default in `resolveTelegramSessionState()`, use it for the no-override fallback, and keep the regression test that proves the Telegram picker marks an `agents.list[].model` default when it differs from `agents.defaults.model`. A small cleanup can store the resolved default once for both stored-override provider resolution and final fallback.

## Reproduction Assessment

Yes. The high-confidence reproduction is a Telegram DM callback with no stored model override, a routed/default agent whose `agents.list[].model` differs from `agents.defaults.model`, and a `mdl_list_<provider>_1` picker callback; current main returns the global default as `sessionState.model`, so exact picker matching does not mark the agent default.

## Solution Assessment

Yes. The PR uses the existing `resolveDefaultModelForAgent({ cfg, agentId })` API at the narrow Telegram callback fallback point and adds focused regression coverage; the only safer refinement is avoiding the duplicate resolver call noted by Greptile, which is cleanup rather than a correctness blocker.
