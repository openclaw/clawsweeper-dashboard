## Summary

Keep this PR open. The archive-first direction is useful and has terminal proof, but the latest head still has a correctness bug in crash-window self-heal when the already-written archive is from a previous day.

## What This Changes

This PR moves detailed short-term memory promotion output from root `MEMORY.md` into quarterly `memory/archived/YYYY-Q#/` dumps, leaves a compact pointer in `MEMORY.md`, updates docs/help/events, and adds migration coverage.

## Best Possible Solution

Fix the cross-day migration dedupe bug, keep the docs/tests/events aligned with archive-first storage, then have memory-core owners decide whether this should replace the shipped compaction behavior.

## Reproduction Assessment

Yes for the PR finding from source inspection: seed `MEMORY.md` with a marked legacy promotion section, seed the same marker in an earlier archive file, then run apply on a later date. The current migration filter checks only the later date's archive markers and would duplicate the legacy section into the later dump.

## Solution Assessment

No, not yet. Archive-first storage is a plausible solution, but migrated sections should be filtered against all archived markers, not only the current dump, before this is ready for owner review and merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Deduplicate migrated sections across all archives:** `extensions/memory-core/src/short-term-promotion.ts:1960-1964`
  - body: This filters migrated root sections only against markers already in the current date's archive file. If the archive write succeeds, the `MEMORY.md` rewrite crashes, and the next apply runs on a later day, the marker is in the earlier dump, so this path appends the same legacy section into the later dump before removing it from `MEMORY.md`. Use the full archived-marker scan for migration filtering and add a cross-day self-heal regression.
  - confidence: 0.88
