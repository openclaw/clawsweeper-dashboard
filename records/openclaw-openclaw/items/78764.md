## Summary

Keep open: the archive-first direction is useful and the supplied real behavior proof is sufficient, but PR head still has a source-level self-heal gap where an already-archived selected candidate can be marked promoted without restoring the compact `MEMORY.md` pointer.

## What This Changes

The PR archives detailed short-term promotion excerpts into quarterly `memory/archived` dumps, keeps `MEMORY.md` as a compact latest-archive pointer, updates promotion event fields and CLI/docs wording, and adjusts memory-core plus adjacent tests.

## Best Possible Solution

Keep the archive-first design, but make `applyShortTermPromotions` self-heal `MEMORY.md` when selected candidates are reconciled from existing archive markers, and lock that crash window with a focused regression test.

## Reproduction Assessment

Yes, at source level: seed an archive marker for a selected candidate, leave `MEMORY.md` without the compact pointer, then run apply with that candidate. The PR classifies it as `alreadyWritten`, skips the pointer rewrite block, and still marks the store promoted.

## Solution Assessment

No, not yet. The archive-first solution is the right direction, but the current implementation only self-heals migrated root sections, not already-archived selected candidates with a missing root pointer.
