## Summary

Keep open: current main and v2026.6.10 still classify overloaded-worded HTTP 429 signals as rate limits, and the open candidate PR https://github.com/openclaw/openclaw/pull/97009 now owns the likely fix path until it merges or is rejected.

## What This Changes

The issue reports that HTTP 429 responses whose body explicitly says the provider is overloaded are classified as rate limits instead of overloaded failover errors.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/97009, then close this issue only after the chosen fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: current main matches bare `429` as rate-limit text before overloaded text, and the explicit 429 status branch returns `rate_limit` after billing checks.

## Solution Assessment

Yes. The best fix is a narrow shared classifier precedence and HTTP-status handling change with regression coverage; a provider-only workaround would leave the same generic 429 overload bug for other providers.
