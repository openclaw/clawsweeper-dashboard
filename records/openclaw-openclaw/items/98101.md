## Summary

Keep open: current main and v2026.6.10 still classify overload-worded HTTP 429 signals as rate limits, and two open candidate PRs now target the same classifier gap. The issue should close only after maintainers choose and land one canonical fix.

## What This Changes

The issue asks OpenClaw to classify HTTP 429 responses whose body says the provider is overloaded as `overloaded`, not `rate_limit`, so user copy and auth/failover behavior match the transient overload condition.

## Best Possible Solution

Land one canonical shared classifier fix with regression coverage and acceptable provider-path proof, close or supersede the duplicate PR, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows current main matches bare `429` as rate-limit text before overload text and the explicit 429 status branch returns `rate_limit` after billing checks; reporter logs cover the real z.ai agent path.

## Solution Assessment

Yes. The best fix is a narrow shared classifier precedence and HTTP-status handling change with regression coverage; a z.ai-only workaround would leave the same generic 429 overload bug for other providers.
