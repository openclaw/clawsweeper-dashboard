## Summary

Keep open: current main and v2026.6.10 still classify overloaded-worded HTTP 429 responses as rate_limit, and open candidate PRs now own the fix path until one lands or is rejected.

## What This Changes

The issue asks OpenClaw to classify HTTP 429 responses whose body explicitly signals provider overload as overloaded instead of rate_limit.

## Best Possible Solution

Review and land one narrow shared classifier fix, then close this issue after main preserves overloaded for 429 overload bodies while keeping genuine rate-limit and billing 429s unchanged.

## Reproduction Assessment

Yes. Current main source gives a high-confidence reproduction: a 429-bearing overloaded message hits the broad 429 rate-limit matcher first, and explicit HTTP 429 status handling returns rate_limit after billing checks.

## Solution Assessment

Yes. The maintainable fix is in the shared failover classifier order/status handling with regression coverage; a z.ai-only workaround would leave the provider-agnostic 429 overload bug in place.
