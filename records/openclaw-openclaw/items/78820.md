## Summary

Keep open: current main still runs the package-manager update inside the live `update.run` handler and only schedules the Gateway restart after the package swap. The merged stale-chunk mitigation covers adjacent `manager-*` and `runtime-*` reports, but not the exact `bash-tools-GSoqmeZd.js` import reported here.

## What This Changes

The issue reports that control-plane `update.run` can leave the installed `dist` tree briefly inconsistent during an in-place package update, causing hashed ESM module imports to fail until the Gateway restarts.

## Best Possible Solution

Package-manager `update.run` should either hand off package replacement to an updater outside the live Gateway process, or fail closed with clear shell `openclaw update` guidance; any targeted stale-chunk alias should be treated as a compatibility mitigation, not the full race fix.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The report gives exact version and missing-module details, and source inspection shows the package swap precedes the restart, but I did not mutate a real install tree to catch the race live.

## Solution Assessment

Unclear. The current merged alias approach is useful for related stale chunks, but the maintainable fix for this issue likely needs an explicit outside-Gateway handoff or fail-closed control-plane update policy.
