## Summary

Keep open: the report describes a plausible current `update.run` package-update race, and current source still swaps the package before scheduling the live Gateway restart rather than handing package-manager updates fully outside the running Gateway process.

## What This Changes

The issue reports that in-place `update.run` can leave the installed `dist` directory temporarily missing hashed modules, causing CLI module-resolution failures until restart.

## Best Possible Solution

Package-manager `update.run` should either hand off to an updater process outside the live Gateway before package replacement, or fail closed with clear guidance to run shell `openclaw update`; staged install verification should remain in place.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The report gives concrete v2026.5.6 failure details, and source inspection shows the package swap occurs before the `update.run` restart is scheduled, but I did not mutate an install tree to catch the race live.

## Solution Assessment

Unclear. A pure temp-directory swap is not enough to settle the live-process problem; the safer maintainable direction is an explicit outside-Gateway handoff or a fail-closed package-update path for control-plane `update.run`.
