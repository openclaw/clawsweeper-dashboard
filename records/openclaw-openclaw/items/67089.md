## Summary

Keep open. Current main covers the admin-scope portion through the reserved `config.*` policy, but it does not implement config HMAC integrity, and this PR is not merge-ready because it has parse/CI blockers plus unresolved security design issues.

## What This Changes

This PR adds config HMAC sidecar signing/verification, config integrity audit docs/findings, explicit admin checks for config mutation handlers, and a CodexBar onboarding choice.

## Best Possible Solution

A maintainer/security-owned config-integrity design should land narrowly with signature bootstrap or migration, injected token/home/env dependencies, consistent write/read failure semantics, focused tamper/reload/upgrade tests, aligned docs, and unrelated onboarding split out.

## Reproduction Assessment

Yes for the PR blockers: the parse error is visible in the raw branch, CI check-runs are failing on the exact PR head, and the upgrade lockout follows from an established config with a gateway token but no `.sig` file. I did not run local tests because this review is read-only and `pnpm` is unavailable in this checkout.

## Solution Assessment

No. The requested HMAC direction is plausible and remains unimplemented on main, but this patch is not the narrowest maintainable solution until it fixes the compile issue, bootstrap behavior, failure semantics, dependency injection, docs, and unrelated onboarding change.

## Review Findings

Overall confidence: 0.93

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed to `io.writeConfigFile` is opened here but the changed line closes only the function call. TypeScript cannot parse this file, so the branch cannot build.
  - confidence: 0.97
- **[P1] Bootstrap signatures before enforcing reads:** `src/config/io.ts:1112-1113`
  - body: This branch rejects established configs when the `.sig` file is missing. Existing installations with a normal `openclaw.json` and a gateway token will not have a sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.94
- **[P2] Surface signature write failures:** `src/config/io.hmac-integrity.ts:88-89`
  - body: The sidecar writer swallows every write error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a later startup failure with no immediate signal.
  - confidence: 0.91
- **[P2] Use injected deps for token lookup:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` reads `os.homedir()` and `process.env` directly while config IO tests and alternate homes use injected dependencies. That can make tests read config from a fake home while resolving the HMAC key from the real home, skipping verification.
  - confidence: 0.88
- **[P2] Remove the unused HMAC writer import:** `src/config/io.ts:34`
  - body: `writeConfigHmacSig` is imported here, but the diff only calls the sync writer. After the syntax error is fixed, the unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the docs with rejection behavior:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The user-facing security doc needs to match the actual enforcement behavior.
  - confidence: 0.86
- **[P3] Separate the security audit log or fix the doc:** `docs/security/config-protection.md:50`
  - body: The new audit helper writes to `config-audit.jsonl`, but the docs describe the security audit log as separate from the existing `config-audit.jsonl`. Either the filename or the documentation needs to change so operators know which log contains which record shape.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and introduces concrete availability and assurance risks in the proposed config-integrity enforcement.

Concerns:

- **[high] Missing signatures can lock out existing configs:** `src/config/io.ts:1112`
  - body: The PR treats missing signatures on established configs as suspicious and rejects them, but existing installations will not have generated `.sig` sidecars before this feature lands.
  - confidence: 0.94
- **[medium] Sidecar write failures fail open before reads fail closed:** `src/config/io.hmac-integrity.ts:88`
  - body: Signature write errors are silently ignored while read paths can reject missing signatures later, so an I/O error during persistence can turn into a later startup failure.
  - confidence: 0.91
- **[low] HMAC key resolution bypasses injected dependencies:** `src/config/io.hmac-integrity.ts:26`
  - body: Direct `os.homedir()` and `process.env` reads make tests and alternate homes use a different token source from the config IO dependency graph, weakening coverage of the new security path.
  - confidence: 0.88
