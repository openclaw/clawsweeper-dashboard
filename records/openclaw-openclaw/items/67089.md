## Summary

Keep open. Current main already enforces admin scope for `config.*` methods, but it does not implement the PR's HMAC config-integrity surface, so the branch is not obsolete; however, the patch is not merge-ready because it has a parse-breaking edit plus security-sensitive migration and failure-semantics problems.

## What This Changes

The PR adds HMAC sidecar signing and verification for gateway config reads/writes, config security audit logging and docs, explicit admin checks for config-mutating methods, and a CodexBar onboarding option.

## Best Possible Solution

A maintainer/security-owned version should keep the existing generic `config.*` admin-scope policy, add config integrity only with an explicit bootstrap or migration path, make signature write/read failure semantics consistent, use injected config IO dependencies, add focused tamper/reload/upgrade tests, align docs, and split unrelated onboarding work.

## Reproduction Assessment

Yes for the current PR blockers: the branch source shows the parse error directly, and the upgrade lockout follows from `no-sig && suspicious` returning `{}` when a gateway token exists. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. The security direction may be valid, but this implementation is not the narrowest maintainable fix until migration/bootstrap, write-failure handling, dependency injection, docs, and unrelated CodexBar changes are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed into `io.writeConfigFile` is opened but this changed line closes only the function call. TypeScript will fail to parse the file, so the branch cannot build.
  - confidence: 0.98
- **[P1] Bootstrap signatures before enforcing reads:** `src/config/io.ts:1112-1113`
  - body: This path rejects established configs when the `.sig` file is missing. Existing installations with a gateway token will not have sidecars yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.95
- **[P2] Propagate sidecar signature write failures:** `src/config/io.hmac-integrity.ts:88-90`
  - body: The signature writer swallows every I/O error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a self-inflicted startup failure.
  - confidence: 0.92
- **[P2] Use injected dependencies for token lookup:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` reads `os.homedir()` and `process.env` directly while config IO supports injected home and env dependencies. Tests or alternate homes can read config from one location while resolving the HMAC key from another, often skipping verification.
  - confidence: 0.89
- **[P2] Remove the unused HMAC writer import:** `src/config/io.ts:33`
  - body: `writeConfigHmacSig` is imported here but the diff only calls the sync writer. After the syntax error is fixed, the unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the docs with rejection behavior:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are still loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The security docs need to match the actual enforcement semantics.
  - confidence: 0.87
- **[P3] Use a distinct audit log file or schema:** `src/config/config-audit.ts:5`
  - body: This new helper writes to `config-audit.jsonl`, the same filename already used by the existing config IO audit log, while the docs describe it as separate. That will mix two unrelated JSONL schemas in one file.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and introduces concrete availability and assurance risks in the proposed config integrity enforcement.

Concerns:

- **[high] Missing signatures can lock out existing configs:** `src/config/io.ts:1112`
  - body: The PR treats missing sidecars on established configs as suspicious and rejects them, but existing installations will not have generated `.sig` files before this feature lands.
  - confidence: 0.95
- **[medium] Signature write failures fail open before reads fail closed:** `src/config/io.hmac-integrity.ts:88`
  - body: Sidecar write errors are silently ignored while read paths can later reject missing signatures, creating a latent startup failure after a successful config write.
  - confidence: 0.92
- **[low] Token lookup bypasses injected dependencies:** `src/config/io.hmac-integrity.ts:26`
  - body: Direct `os.homedir()` and `process.env` reads can make tests or alternate homes skip HMAC verification or verify against a different token source than the config IO path.
  - confidence: 0.89
