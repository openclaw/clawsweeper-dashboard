## Summary

Keep open: current main already covers the admin-scope portion for `config.*`, but the requested HMAC config-integrity work is still not implemented, and the PR remains security-sensitive with compile, migration, and failure-semantics blockers.

## What This Changes

The PR adds HMAC sidecar signing/verification for config reads and writes, config integrity audit docs/findings, explicit admin checks for config mutations, and a CodexBar onboarding option.

## Best Possible Solution

A maintainer/security-owned version should land a narrow config-integrity design with explicit bootstrap/migration behavior, injected token/home/env dependencies, consistent write/read failure semantics, focused tamper/reload/upgrade tests, aligned docs, and unrelated onboarding split out.

## Reproduction Assessment

Yes for the PR blockers: the raw PR branch shows the parse error, missing-signature rejection path, and swallowed sidecar write failures. I did not run tests because this review was read-only, but live check data already shows failing build/check lanes on the PR head.

## Solution Assessment

No. HMAC config integrity remains plausible and unimplemented on main, but this patch is not the best shape until migration/bootstrap, write failure semantics, dependency injection, docs, tests, and unrelated onboarding scope are resolved.

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed to `io.writeConfigFile` is opened here but the changed line closes only the function call. TypeScript cannot parse this file, so the branch cannot build.
  - confidence: 0.97
- **[P1] Bootstrap signatures before enforcing reads:** `src/config/io.ts:1112-1113`
  - body: This branch rejects established configs when the `.sig` file is missing. Existing installations with a normal `openclaw.json` and a gateway token will not have a sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.94
- **[P2] Propagate sidecar signature write failures:** `src/config/io.hmac-integrity.ts:88-89`
  - body: The sidecar writer swallows every write error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a later startup failure with no immediate signal.
  - confidence: 0.91
- **[P2] Use injected deps for token lookup:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` reads `os.homedir()` and `process.env` directly while config IO tests and alternate homes use injected dependencies. That can make tests read config from a fake home while resolving the HMAC key from the real home, skipping verification.
  - confidence: 0.88
- **[P2] Remove the unused HMAC writer import:** `src/config/io.ts:34`
  - body: `writeConfigHmacSig` is imported here, but the diff only calls the sync writer. After the syntax error is fixed, the unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the docs with rejection behavior:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The user-facing security doc needs to match the actual enforcement behavior.
  - confidence: 0.86
- **[P3] Use a distinct audit log name or share the existing schema:** `src/config/config-audit.ts:5`
  - body: The new helper writes to `config-audit.jsonl`, but the docs describe the security audit log as separate from the existing `config-audit.jsonl`. Operators will not know which record shape to expect unless the filename or documentation is corrected.
