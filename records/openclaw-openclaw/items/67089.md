Keep open: the requested config-integrity hardening is not implemented on current main, but this PR is not mergeable because the patch has a parse error, unsafe upgrade/write semantics, missing real behavior proof, and security design questions that need maintainer review.

## What This Changes

This PR adds config HMAC sidecar signing/verification and audit docs/findings, explicit admin checks for config-mutating gateway methods, and a CodexBar onboarding option.

## Best Possible Solution

A maintainer/security-owned version should settle the config-integrity threat model, then land a narrow implementation with explicit bootstrap/migration behavior, injected token/state dependencies, consistent write failure handling, focused tests, and docs, with CodexBar onboarding split out.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR diff shows the parse error, missing-signature rejection path, swallowed sidecar write failures, and docs/code mismatch. No current-main bug reproduction is needed because this is a PR review, and the HMAC feature is absent from main.

## Solution Assessment

No. HMAC config integrity is a plausible security hardening direction, but this patch is not the best implementation until migration/bootstrap, token lookup, write failure semantics, docs, tests, and unrelated onboarding scope are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed to `io.writeConfigFile` is opened here but the patch closes only the function call. TypeScript cannot parse this file, so the branch cannot build.
  - confidence: 0.97
- **[P1] Bootstrap signatures before enforcing reads:** `src/config/io.ts:1112-1113`
  - body: This rejects established configs when the `.sig` file is missing. Existing installations with a normal `openclaw.json` and a gateway token will not have a sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.94
- **[P2] Propagate sidecar signature write failures:** `src/config/io.hmac-integrity.ts:84-86`
  - body: The sidecar writer swallows write errors, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can become a later startup failure with no immediate signal.
  - confidence: 0.91
- **[P2] Resolve HMAC keys through config IO dependencies:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` hardcodes `os.homedir()` and `process.env`, bypassing the injected home/env/state-dir paths used by config IO. Alternate homes and tests can read one config while verifying against a different token source or no token at all.
  - confidence: 0.88
- **[P2] Remove the unused async HMAC writer import:** `src/config/io.ts:34`
  - body: `writeConfigHmacSig` is imported here, but the diff only calls the sync writer. After the syntax error is fixed, the unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the docs with rejection behavior:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The security doc needs to match the actual enforcement behavior.
  - confidence: 0.86
- **[P3] Use a distinct audit log name or existing schema:** `src/config/config-audit.ts:5`
  - body: The new helper writes to `config-audit.jsonl`, but the docs describe the security audit log as separate from the existing `config-audit.jsonl`. Operators will not know which record shape to expect unless the filename or schema is shared intentionally.
  - confidence: 0.78
