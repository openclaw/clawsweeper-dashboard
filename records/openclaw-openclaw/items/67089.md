## Summary

Keep open: current main already reserves `config.*` gateway methods for admin scope, but it still has no config HMAC integrity surface, and this PR's implementation has blocking correctness, migration, proof, and security-review issues.

## What This Changes

This PR adds config HMAC sidecar signing/verification, config integrity audit docs/findings, explicit admin checks for config mutations, and a CodexBar onboarding option.

## Best Possible Solution

A maintainer/security-owned version should settle the config-integrity threat model, split unrelated onboarding, and land a narrow HMAC design with explicit bootstrap/migration behavior, injected token/home/env dependencies, consistent write/read failure semantics, focused upgrade/tamper/reload tests, and docs that match enforcement.

## Reproduction Assessment

Yes for the PR blockers: the live diff shows the parse error, upgrade-time missing-signature rejection path, swallowed sidecar write failures, and dependency-injection mismatch. I did not run tests because this review is read-only, but live checks on the PR head also show failing build/check lanes.

## Solution Assessment

No. HMAC config integrity is a plausible security direction and is not implemented on main, but this patch is not the best merge shape until migration/bootstrap, write failure semantics, dependency injection, docs, tests, and unrelated onboarding scope are resolved.

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed to `io.writeConfigFile` is opened here but the changed line closes only the function call. TypeScript cannot parse this file, so the branch cannot build.
  - confidence: 0.97
- **[P1] Bootstrap missing signatures before rejecting configs:** `src/config/io.ts:1112-1113`
  - body: This branch rejects established configs when the `.sig` file is missing. Existing installations with a normal `openclaw.json` and a gateway token will not have a sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.94
- **[P2] Propagate signature sidecar write failures:** `src/config/io.hmac-integrity.ts:88-89`
  - body: The sidecar writer swallows every write error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a later startup failure with no immediate signal.
  - confidence: 0.91
- **[P2] Resolve the HMAC key through injected config deps:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` reads `os.homedir()` and `process.env` directly while config IO tests and alternate homes use injected dependencies. That can make tests read config from a fake home while resolving the HMAC key from the real home, skipping verification or disabling integrity in non-default state-dir layouts.
  - confidence: 0.88
- **[P2] Remove the unused async HMAC writer import:** `src/config/io.ts:34`
  - body: `writeConfigHmacSig` is imported here, but the diff only calls the sync writer. After the syntax error is fixed, the unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the config protection docs with rejection behavior:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The user-facing security doc needs to match the actual enforcement behavior.
  - confidence: 0.86
