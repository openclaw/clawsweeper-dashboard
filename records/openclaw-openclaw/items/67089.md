The PR adds config HMAC sidecar signing/verification, config integrity audit/docs, explicit config-method admin checks, and a CodexBar onboarding option.

## Best Possible Solution

Land a maintainer/security-owned config-integrity design separately from onboarding changes, with migration-safe bootstrap behavior, injected token/home/env dependencies, explicit sidecar write-failure semantics, focused tests, and matching docs.

## Solution Assessment

No. HMAC config integrity may be a valid security direction, but this patch is not the best merge shape until migration, failure semantics, dependency injection, docs, tests, real-behavior proof, and unrelated onboarding scope are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Close the write options object:** `src/config/io.ts:1912`
  - body: The options object passed to `io.writeConfigFile` is opened above this line but the patch closes only the function call. TypeScript cannot parse this file, so the branch cannot build.
  - confidence: 0.97
- **[P1] Bootstrap unsigned configs before rejecting them:** `src/config/io.ts:1112-1113`
  - body: This rejects established configs when the `.sig` file is missing. Existing installations with a normal `openclaw.json` and a gateway token will not have a sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.
  - confidence: 0.94
- **[P2] Propagate signature sidecar write failures:** `src/config/io.hmac-integrity.ts:88-90`
  - body: The sidecar writer swallows every write error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a later startup failure with no immediate signal.
  - confidence: 0.91
- **[P2] Resolve the HMAC key through injected config deps:** `src/config/io.hmac-integrity.ts:26-35`
  - body: `readGatewayToken` reads the real home directory and process environment directly while config IO accepts injected home/env dependencies. Tests and alternate state dirs can therefore read config from one home while resolving the HMAC key from another.
  - confidence: 0.88
- **[P2] Remove the unused async HMAC writer import:** `src/config/io.ts:34`
  - body: `writeConfigHmacSig` is imported here, but the diff only calls the sync writer. After the syntax error is fixed, this unused import should still fail the lint gate.
  - confidence: 0.9
- **[P2] Align the docs with the rejection path:** `docs/security/config-protection.md:13`
  - body: The docs say mismatched configs are loaded normally, but the sync load path returns `{}` and reload rejects snapshots with `integrityWarning`. The user-facing security doc needs to match the actual enforcement behavior.
  - confidence: 0.86
- **[P3] Use a distinct security audit log path:** `src/config/config-audit.ts:5`
  - body: The new helper writes `config-audit.jsonl`, the same filename used by the existing config write audit log, while the docs describe this as a separate log. That will interleave incompatible record shapes and make the docs misleading.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces config integrity enforcement with concrete availability and assurance risks that need maintainer/security review.

Concerns:

- **[high] Missing signatures can lock out existing configs:** `src/config/io.ts:1112`
  - body: The PR treats missing signatures on established configs as suspicious and rejects them, but existing installations will not have generated `.sig` sidecars before this feature lands.
  - confidence: 0.94
- **[medium] Sidecar write failures fail open before reads fail closed:** `src/config/io.hmac-integrity.ts:88`
  - body: Signature write errors are silently ignored while read paths can reject missing signatures later, so an I/O error during persistence can become a later startup failure.
  - confidence: 0.91

  - body: Direct home-directory and environment reads make tests and alternate homes use a different token source from the config IO dependency graph, weakening coverage and non-default layouts.
