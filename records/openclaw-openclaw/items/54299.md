## Summary

Keep open: current main and the latest v2026.6.6 release still preserve the source-backed behavior under review, and there is no merged or open canonical item that fully owns the remaining security/product decision.

## What This Changes

The issue asks OpenClaw to avoid silently omitting the Google Gemini CLI OAuth `client_secret` during token exchange and to stop relying on the fixed localhost:8085 redirect callback.

## Best Possible Solution

Keep the work in the bundled Google plugin and make the OAuth contract explicit: either document and test intentional public-client mode, or fail/warn when a secret is expected, while preserving a reliable callback path that does not add unnecessary config surface.

## Reproduction Assessment

Yes at source level: current main conditionally omits `client_secret` and still uses the fixed Google OAuth callback URI/port. I did not run a live Google OAuth exchange or occupy port 8085 in this read-only cleanup review.

## Solution Assessment

Unclear. The issue points at a real source behavior, but the best fix depends on whether maintainers want to support public-client PKCE mode, fail closed for ID-only config, warn, or redesign the redirect registration/callback strategy.
