## Summary

Keep open: current main and the latest release still accept a Gemini OAuth client ID without a paired secret and still use a fixed localhost callback port, but Google's installed-app OAuth docs make `client_secret` optional and recommend loopback redirect behavior that needs a compatibility decision rather than a conservative cleanup close. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app))

## What This Changes

The issue asks OpenClaw to stop silently omitting `client_secret` during Gemini CLI OAuth token exchange and to remove or configure the hardcoded localhost:8085 redirect port.

## Best Possible Solution

Define the supported Gemini CLI OAuth credential modes and loopback redirect policy, then enforce that contract in the Google plugin with focused tests and docs.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client ID without a paired secret, omits `client_secret` when absent, and binds the callback listener to port 8085 before falling back to manual paste mode on listener failure.

## Solution Assessment

Unclear. Throwing on a missing `client_secret` conflicts with Google's installed-app OAuth contract where that field is optional, while dynamic or configurable ports need a redirect-URI compatibility plan. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app))
