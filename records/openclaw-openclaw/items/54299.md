## Summary

Keep open: current main and v2026.5.18 still accept an env OAuth client ID without a secret, conditionally omit `client_secret` from Google token requests, and use a fixed `localhost:8085` callback. The right next step is maintainer security/product review because Google's native-app guidance makes `client_secret` optional while recommending a random available loopback port. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app?utm_source=openai))

## What This Changes

The issue asks OpenClaw to stop silently omitting Google OAuth `client_secret` and to avoid the fixed `localhost:8085` OAuth callback port.

## Best Possible Solution

Define OpenClaw's supported Google OAuth credential modes and loopback redirect policy, then enforce and document that contract in the Google plugin with targeted tests.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.5.18 accept an env OAuth client ID without a paired secret, conditionally omit `client_secret`, and bind the local callback helper to port 8085.

## Solution Assessment

Unclear. Throwing on missing `client_secret` is not obviously the best fix because Google documents installed-app secrets as optional, while the fixed callback port conflicts with Google's random-port loopback guidance. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app?utm_source=openai))
