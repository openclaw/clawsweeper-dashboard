## Summary

Keep open: current main still has the reported Google/Gemini OAuth behavior, and the remaining fix is a security/product policy decision rather than an already-implemented cleanup close.

## What This Changes

The issue asks OpenClaw to make Google/Gemini OAuth token exchange fail or warn when an env client ID lacks a client secret, and to avoid or document the fixed localhost:8085 redirect port.

## Best Possible Solution

Keep this open until the Google plugin owner defines the OAuth credential contract in code, docs, and tests, including whether env client IDs require a matching secret and whether the loopback redirect should be fixed, configurable, or dynamically selected with compatible redirect registration.

## Reproduction Assessment

Yes, by source inspection. On current main, an env client ID can resolve with clientSecret undefined, the token request then omits client_secret, and the Google OAuth flow still binds the fixed localhost:8085 callback port.

## Solution Assessment

Unclear. The suggested fixes are plausible, but the maintainable path needs an explicit Google OAuth client-type and redirect-URI policy before changing runtime behavior.
