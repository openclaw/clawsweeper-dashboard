## Summary

Keep open. Current main still matches the reported Google/Gemini CLI OAuth behavior: env-provided client IDs can flow through without a matching client secret, token exchange only sends `client_secret` when present, and the localhost redirect remains fixed to port 8085. The prior ClawSweeper review reached the same conclusion, and the later source/history check did not find a merged fix or paired open PR.

## What This Changes

The issue requests explicit handling for missing Google/Gemini OAuth client secrets and a configurable or dynamic localhost redirect port.

## Best Possible Solution

Keep this issue open for Google plugin owner review. The end state should make the Gemini CLI OAuth contract explicit in code, docs, and tests: either reject or warn on env client ID without matching secret if confidential-client mode is required, or document and regression-test public-client mode as intentional. Separately, support a configured redirect URI/port only if compatible with Google OAuth registration; otherwise document and test fixed localhost:8085 with manual paste fallback as the intended behavior.

## Risks / Open Questions

- The issue's attacker-impact framing is not fully proven from source alone because the flow uses PKCE and changing OAuth env vars generally implies local environment control; the confirmed problem is the ambiguous, untested client-secret contract.
- A dynamic redirect port may be constrained by Google OAuth redirect URI registration, so the final behavior may need explicit configured redirect URIs or a documented fixed-port plus manual-fallback policy.
- Requiring a client secret whenever an env client ID is present could conflict with prior fixes for Gemini CLI OAuth setups and with any intentionally supported public-client/personal-OAuth mode.
