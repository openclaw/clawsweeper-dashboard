## Summary

Keep open: current main and v2026.5.22 still have the reported Gemini CLI OAuth behavior, but Google's installed-app OAuth contract makes the missing-secret path and loopback-port policy a maintainer security/product decision rather than a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to make Gemini CLI OAuth fail or warn when an env client ID lacks a client secret, and to replace the fixed localhost:8085 callback with a configurable or dynamic redirect port.

## Best Possible Solution

Define the supported Gemini CLI OAuth credential modes and loopback redirect policy, then enforce and document that contract in the Google plugin with focused tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client ID without a paired secret, omits client_secret when absent, and binds the local callback listener to port 8085 before falling back to manual paste mode on listener failure.

## Solution Assessment

Unclear. Throwing on missing client_secret conflicts with Google's installed-app OAuth documentation, while dynamic or configurable ports need a redirect-URI compatibility decision before changing provider behavior. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app))
