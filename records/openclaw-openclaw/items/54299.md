## Summary

Keep open: current main still accepts Google OAuth env credentials when the paired secret is absent, conditionally omits `client_secret` in token and refresh exchanges, and keeps a fixed `localhost:8085` redirect URI. The port-conflict path now falls back to manual paste flow, but the OAuth credential-mode and loopback redirect policy still need maintainer security/product direction rather than an automatic repair.

## What This Changes

The issue asks OpenClaw to stop silently proceeding without an OAuth `client_secret` and to avoid a hardcoded Google OAuth redirect callback port.

## Best Possible Solution

Define and document OpenClaw's supported Google OAuth credential modes and loopback redirect policy, then enforce that contract in the Google plugin with focused coverage for env credentials, token exchange, callback binding, and manual fallback.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client ID without a paired secret, conditionally omits `client_secret` in token requests, and binds the local callback listener to port 8085 before falling back to manual mode on listen errors.

## Solution Assessment

Unclear. Rejecting missing `client_secret` may be too strict for installed-app OAuth clients, while dynamic or configurable callback ports need an explicit redirect URI compatibility policy and docs.
