## Summary

Keep open: current main and the latest release still have the reported Google Gemini CLI OAuth behavior, and the safe fix needs an explicit security/product decision because Google’s installed-app OAuth guidance treats native apps as unable to keep secrets and recommends loopback listeners on random available ports. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app?utm_source=openai))

## What This Changes

The issue reports that Google OAuth token exchange can omit `client_secret` without warning and uses a fixed localhost redirect port.

## Best Possible Solution

Define the supported Gemini CLI OAuth credential modes and loopback redirect policy, then enforce and document that contract inside the Google plugin with focused tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client id without a paired secret, omits `client_secret` when absent, and binds the local callback listener to port 8085 before falling back to manual paste mode on listener failure.

## Solution Assessment

Unclear. Throwing on missing `client_secret` or adding a configurable port may be the wrong default unless maintainers first choose the supported Google OAuth credential and redirect compatibility contract.
