## Summary

Keep open: current main and the latest release still have source-backed optional Google OAuth client-secret handling and fixed localhost:8085 callback behavior; the Gemini path has a manual fallback for local listener failures, but the broader auth/security contract and Google Meet callback behavior remain unresolved.

## What This Changes

The issue asks OpenClaw to make Google OAuth client-secret handling explicit and avoid a fixed localhost:8085 callback as the only automatic redirect path.

## Best Possible Solution

Make the Google OAuth contract explicit across Gemini and Google Meet: either document and test intentional public-client PKCE behavior, or warn/fail when a secret is expected, while keeping callback URI behavior and docs aligned.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still conditionally omit `client_secret` and use fixed localhost:8085 redirect/listener values in Google OAuth paths. I did not run a live Google OAuth exchange or occupy port 8085 in this read-only review.

## Solution Assessment

Unclear: the issue identifies real source behavior, but the best fix depends on maintainer choice between public-client PKCE support, fail-closed confidential-client behavior, warning-only behavior, or a callback-port redesign.
