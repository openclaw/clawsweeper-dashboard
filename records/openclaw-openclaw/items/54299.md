## Summary

Keep open: current main still allows Google/Gemini OAuth to proceed with an env client ID but no matching secret, and the local OAuth callback still uses a fixed localhost:8085 listener. The remaining decision is security/product policy, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to stop silently omitting Google OAuth client_secret and to avoid a hardcoded localhost:8085 OAuth redirect port.

## Best Possible Solution

Define the supported Google OAuth credential modes and loopback redirect policy, then enforce them in the Google plugin with tests and docs while preserving PKCE and manual fallback behavior.

## Reproduction Assessment

Yes, by source inspection. Current main still returns an OAuth config without a matching env client secret, omits client_secret from the token body when absent, and binds the Google callback to localhost:8085.

## Solution Assessment

Unclear. Rejecting, warning, documenting public-client mode, using a configurable port, or using a dynamic loopback port are plausible, but the narrow fix depends on maintainer policy for Google OAuth credentials and redirect URIs.
