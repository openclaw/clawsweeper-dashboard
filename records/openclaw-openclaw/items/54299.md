## Summary

Keep open. Current main still has the reported Google/Gemini CLI OAuth behavior: an env-provided client ID can flow through without a matching secret, token exchange omits client_secret when absent, and the local callback remains fixed to localhost:8085. The existing manual fallback reduces port-conflict breakage, but the client-secret mode and redirect-port policy are still implicit in code, docs, and focused tests.

## What This Changes

The issue asks the Google/Gemini CLI OAuth flow to reject or warn on missing client_secret and to make the localhost:8085 redirect callback configurable, dynamic, or explicitly documented.

## Best Possible Solution

Keep this open for the bundled Google plugin owner to make the Gemini CLI OAuth contract explicit in code, docs, and tests. The desired end state is either a rejected or clearly warned env client ID without a matching secret, or a documented and tested public-client mode, plus a supported redirect-port strategy: dynamic/configurable loopback when compatible, or an intentional fixed-port/manual-fallback policy that is documented.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: set an env client ID without the matching secret and resolveOAuthClientConfig can return clientSecret undefined, exchangeCodeForTokens will omit client_secret, and waitForLocalCallback still binds the fixed 8085 callback path.

## Solution Assessment

Unclear. The suggested direction is reasonable, but the best fix needs owner/security review to choose the OAuth client-type contract and redirect-port compatibility policy rather than blindly converting the flow to confidential-only or dynamic-port behavior.
