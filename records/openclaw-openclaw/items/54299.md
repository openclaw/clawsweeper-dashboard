## Summary

Keep open: current main and v2026.5.19 still have the reported Gemini CLI OAuth credential and callback-port behavior, and the remaining question is a security/auth-provider policy decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to stop silently omitting `client_secret` during Google OAuth token exchange and to avoid a fixed localhost callback port.

## Best Possible Solution

Define and document the supported Gemini CLI OAuth credential modes and loopback redirect policy, then enforce that contract in the Google plugin with focused tests for env credentials, token-body construction, callback binding, and manual fallback.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client ID without a paired secret, omits `client_secret` when absent, and binds the callback listener to port 8085 before falling back to manual paste mode on listener failure.

## Solution Assessment

Unclear. Throwing on a missing `client_secret` may conflict with installed-app OAuth semantics, and changing the loopback port needs a redirect URI compatibility plan rather than a one-line cleanup.
