## Summary

Keep open: current main still has the reported Google/Gemini OAuth shape, and the right fix is a security/product decision rather than a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to reject or warn on Google/Gemini OAuth env credentials without a client secret and to replace the fixed localhost:8085 redirect port with a configurable or dynamic callback policy.

## Best Possible Solution

Define the Google/Gemini OAuth credential mode and loopback redirect policy, then enforce it in plugin code, tests, and docs without weakening PKCE or the manual fallback flow.

## Reproduction Assessment

Yes, by source inspection. Setting OPENCLAW_GEMINI_OAUTH_CLIENT_ID without a matching secret still returns an OAuth config, the token request omits client_secret, and the Google OAuth callback path remains fixed to localhost:8085.

## Solution Assessment

Unclear. The suggested fixes are plausible, but the narrow maintainable solution depends on deciding whether env OAuth clients are confidential or public and whether the redirect URI should stay fixed, become configurable, or use a dynamic loopback port.
