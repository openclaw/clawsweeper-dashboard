## Summary

Keep open. Current main still has the reported Google/Gemini CLI OAuth behavior: user-supplied OAuth client IDs can resolve without a matching secret, token exchange only sends client_secret when present, and the local callback remains fixed to localhost:8085. The manual fallback reduces outright failure when the listener cannot bind, but the client-secret contract and redirect-port policy are still not explicit in source, docs, or focused tests.

## What This Changes

The issue asks for the Google/Gemini CLI OAuth flow to reject or warn on missing client_secret and to avoid or document the fixed localhost:8085 redirect callback.

## Best Possible Solution

Keep this open for the bundled Google plugin owner to make the Gemini CLI OAuth contract explicit in plugin code, docs, and tests. The desired end state is either a clearly rejected or warned env client ID without a matching secret, or a documented/tested public-client mode, plus a supported redirect-port strategy: dynamic/configurable loopback when compatible, or a documented fixed-port/manual-fallback policy when it is intentionally constrained.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: set an env client ID without the matching secret and resolveOAuthClientConfig can return clientSecret undefined, exchangeCodeForTokens will omit client_secret, and waitForLocalCallback still binds the fixed 8085 callback path.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best fix needs owner review to choose the OAuth client type contract and redirect-port compatibility policy rather than blindly converting the flow to confidential-only or dynamic-port behavior.
