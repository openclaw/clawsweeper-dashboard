## Summary

Keep open: the OAuth behavior is still present on current main, but the fix depends on maintainer security/product policy because Google documents installed-app client secrets as optional while the fixed loopback port remains a real reliability and policy concern.

## What This Changes

The issue asks OpenClaw to fail or warn when Google OAuth runs without a client secret and to avoid depending on a hardcoded localhost callback port.

## Best Possible Solution

Define OpenClaw's supported Google OAuth credential modes and loopback redirect policy, then enforce that contract in the Google plugin with focused tests and docs.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env client ID without a secret, conditionally omits client_secret during token exchange, and binds the local callback helper to port 8085.

## Solution Assessment

Unclear. The missing-secret behavior is policy-dependent because Google's installed-app docs make client_secret optional, while the fixed localhost port conflicts with loopback guidance; maintainers should choose the intended contract before code changes.
