## Summary

Keep open: current main and v2026.5.22 still carry the optional-secret token exchange and fixed localhost:8085 redirect behavior, and the remaining fix needs maintainer security/product direction because Google desktop OAuth treats client secrets as optional and loopback redirect ports as registration-sensitive. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app))

## What This Changes

The issue asks OpenClaw to require or warn on a missing Gemini CLI OAuth client_secret and avoid a hardcoded localhost:8085 redirect port.

## Best Possible Solution

Define the supported Gemini CLI OAuth credential modes and loopback redirect policy, then enforce and document that contract inside the Google plugin with focused tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts an env Google OAuth client id without a paired secret, omits client_secret when absent, and binds the local callback listener to port 8085 before falling back to manual paste mode on listener failure.

## Solution Assessment

Unclear. Throwing on missing client_secret conflicts with Google's installed-app OAuth contract, while dynamic or configurable ports need a redirect-URI compatibility decision before changing provider behavior. ([developers.google.com](https://developers.google.com/identity/protocols/oauth2/native-app))
