## Summary

Keep open: this is an active protected draft policy PR, current main does not implement the policy runtime/audit surface, and the PR still has concrete read-only regressions plus an unresolved attestation trust-boundary concern.

## What This Changes

The PR adds a bundled policy plugin runtime/audit layer with attestations, expected-attestation enforcement, approval metadata propagation, runtime evidence health checks, policy diff, doctor lint/health SDK plumbing, docs, and tests.

## Best Possible Solution

Settle the policy trust boundary, keep lint/check paths non-mutating, and continue the policy stack only after redacted real runtime output proves stale-attestation blocking.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable for the feature as a whole, but yes for the review findings: PR-head source shows the read-only command paths still call default observing config snapshots.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is; the direction may be viable, but the patch violates its read-only command contract and leaves accepted-attestation authority in ordinary plugin config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep policy checks read-only:** `extensions/policy/src/cli.ts:142`
  - body: `policy check` and `policy watch` build reports through `readConfigFileSnapshot()` without `{ observe: false }`. Snapshot reads observe config state by default, so these documented read-only audit commands can update config observation state; pass the non-observing option and cover the command path.
  - confidence: 0.91

  - body: `doctor --lint` starts with a non-observing snapshot, but the final config validation check re-reads config with default observing behavior. That lets a documented read-only lint run touch config observation state; pass `{ observe: false }` here or reuse the existing lint snapshot.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff needs attention because accepted-attestation enforcement is modeled as ordinary policy plugin config while the PR describes it as supervisor/gateway-owned state.

Concerns:

- **[medium] Enforce accepted-attestation ownership:** `extensions/policy/openclaw.plugin.json:43`

If the governed workspace can update that config, it can update the accepted hash after evidence drift, weakening enforcement unless a trusted supervisor/gateway-owned source is added or the docs explicitly scope this down.
