## Summary

Keep open. Current main has several adjacent restart-recovery mechanisms, including routed restart sentinels, durable one-shot session-delivery recovery, Task Flow durability, and restart-aborted session recovery, but it still does not provide the RFC's first-class task checkpoint lifecycle with paused/resuming/resumed/failed resume states, plan cursor persistence, checkpoint claim semantics, or original/new run linkage.

## What This Changes

Request to add a standardized checkpoint and auto-resume lifecycle for in-progress tasks across gateway restarts by creating a follow-up run seeded from durable checkpoint state.

## Best Possible Solution

Keep this as the design anchor for a deliberate restart checkpoint contract. A maintainable implementation should reuse the restart sentinel/session-delivery queue for wakeup, Task Flow for durable step progress where applicable, and task/session registries for observable lifecycle, then add a secret-safe checkpoint schema with at-most-once claim/resume semantics and clear linkage between the interrupted run and the follow-up run.

## Reproduction Assessment

Not applicable as a feature RFC. The current-main verification path is high-confidence static inspection: the relevant source and docs show adjacent recovery paths but no requested checkpoint statuses, checkpoint schema, or generic startup checkpoint claim/resume flow.

## Solution Assessment

No, current main is not the complete solution. The RFC direction of resuming through a new seeded follow-up run is plausible, but the best fix needs maintainer design review to align with Task Flow, session recovery, restart sentinels, and secret-safe persistence rather than adding a parallel checkpoint mechanism.
