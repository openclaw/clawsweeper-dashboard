## Summary

Keep open. Current main has useful adjacent recovery mechanisms for restart sentinels, session-delivery replay, Task Flow durability, and restart-aborted session recovery, but it still does not implement the RFC's generic checkpoint lifecycle, at-most-once checkpoint claim/resume state model, plan-cursor checkpoint schema, or original-to-resumed run linkage for all in-progress tasks.

## What This Changes

Request a first-class gateway-restart checkpoint and auto-resume mechanism that continues interrupted in-progress tasks as seeded follow-up runs without a new user prompt.

## Best Possible Solution

Define a maintainer-reviewed checkpoint contract that reuses existing Task Flow progress and restart/session delivery mechanisms where appropriate, with secret-safe persistence and explicit claim/resume/failure semantics.

## Reproduction Assessment

Not applicable as a feature RFC. Static inspection gives a high-confidence gap: adjacent recovery paths exist, but the requested generic checkpoint statuses, schema, atomic claim transition, and plan-cursor auto-resume flow do not.

## Solution Assessment

Unclear because no patch is proposed. The seeded follow-up-run direction is plausible, but the maintainable implementation needs design review across task ownership, side-effect safety, and recovery boundaries.

## Risks / Open Questions

- A generic checkpoint schema could persist secrets or sensitive tool context unless the storage contract is explicitly redacted and reviewed.
- At-most-once resume is hard to guarantee across restart loops, concurrent startup paths, and non-idempotent tool calls.
- The design spans Task Flow, restart sentinels, session delivery, subagent/ACP recovery, and config reload behavior, so a narrow autonomous fix would likely encode product decisions prematurely.
