## Summary

Keep open. Current main has several adjacent recovery paths, but it still does not implement the RFC's generic task checkpoint lifecycle, durable claim transition, plan-cursor checkpoint schema, or original-to-resumed run linkage across gateway restarts.

## What This Changes

Request a first-class checkpoint and auto-resume mechanism for in-progress tasks across gateway restarts, creating an at-most-once follow-up run seeded with the original goal, plan progress, routing, and relevant artifacts.

## Best Possible Solution

Define a versioned core restart-checkpoint contract that persists secret-safe task/flow progress, atomically claims resume work on startup, links original and resumed runs, and composes with the existing narrower recovery paths.

## Reproduction Assessment

Not applicable as a feature RFC. Static inspection of current main gives a high-confidence gap because the requested checkpoint statuses, schema, claim transition, and generic startup resume flow are absent while only adjacent recovery paths exist.

## Solution Assessment

Unclear because no patch is proposed. The seeded follow-up-run direction is plausible, but the maintainable solution needs design review across task state, Task Flow progress, restart sentinels, session delivery, subagent recovery, side-effect safety, and secret-safe persistence.
