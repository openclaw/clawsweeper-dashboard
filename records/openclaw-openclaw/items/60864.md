## Summary

Keep open: current main has adjacent restart drain, Task Flow persistence, restart sentinels, durable session-delivery queues, and partial main-session recovery, but it still lacks this RFC's generic checkpoint lifecycle, atomic at-most-once claim, plan cursor, and original-to-resumed run linkage. The broader graceful-restart tracker does not fully supersede this more precise task-checkpoint RFC.

## What This Changes

The issue requests a first-class checkpoint and automatic follow-up-run resume mechanism for in-progress tasks interrupted by gateway restarts.

## Best Possible Solution

Define a versioned restart-checkpoint contract that composes with Task Flow, restart sentinels, session delivery, and main/subagent/cron recovery, then split storage, claim/resume, delivery/UI, and regression testing into focused changes.

## Reproduction Assessment

Not applicable: this is a feature/RFC rather than a broken existing contract. Source inspection on current main gives high confidence that adjacent recovery pieces exist but the requested generic checkpoint/resume lifecycle is not present.

## Solution Assessment

Unclear because there is no patch to assess. The seeded follow-up-run direction is plausible, but the maintainable solution needs maintainer-scoped contracts across task state, restart orchestration, persistence, idempotency, and delivery/UI behavior.
