## Summary

Keep open. Current main has useful restart recovery pieces, but it still does not implement the RFC's generic task checkpoint lifecycle, at-most-once checkpoint claim/resume model, plan-cursor persistence, or original-to-resumed run linkage for all in-progress work.

## What This Changes

The RFC asks for first-class checkpointing and at-most-once automatic continuation of in-progress tasks across gateway restarts.

## Best Possible Solution

Define one restart checkpoint contract that reuses Task Flow progress, restart sentinels/session delivery for wakeup, and session/subagent recovery markers where appropriate, with a secret-safe schema and at-most-once claim semantics.

## Reproduction Assessment

Not applicable as a feature RFC. Static inspection of origin/main gives a high-confidence gap: the adjacent recovery paths exist, but the requested checkpoint statuses, schema, claim transition, and generic startup resume flow do not.

## Solution Assessment

Unclear as an implementation because no patch is proposed. The RFC direction of a new seeded follow-up run is plausible, but the maintainable solution needs design review across Task Flow, session delivery, restart sentinels, subagent recovery, and secret-safe persistence.
