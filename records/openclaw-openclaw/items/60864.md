## Summary

Keep open. Current main has adjacent Task Flow durability, one-shot restart continuations, session-delivery queue recovery, restart drain, and main-session restart recovery, but it still lacks the RFC's generic in-progress task checkpoint lifecycle, plan-cursor checkpoint schema, atomic at-most-once claim, and original-to-resumed run linkage.

## What This Changes

The issue requests a first-class checkpoint and auto-resume mechanism so in-progress tasks can continue across gateway restarts as seeded follow-up runs.

## Best Possible Solution

Define a versioned core restart-checkpoint contract that complements existing restart sentinel, session-delivery, Task Flow, and main-session recovery paths while preserving secret safety and at-most-once resume semantics.

## Reproduction Assessment

Not applicable as a feature RFC. Source inspection gives a high-confidence product gap because current main has adjacent recovery paths but not the requested generic checkpoint statuses, plan-cursor schema, at-most-once claim, or run linkage.

## Solution Assessment

Unclear because there is no patch to assess. The seeded follow-up-run direction is plausible, but the best implementation needs maintainer design review across task state, restart orchestration, session recovery, non-idempotent tool safety, and secret-safe persistence.
