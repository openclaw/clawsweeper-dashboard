## Summary

Keep open. Current main has adjacent durability for Task Flow, restart sentinels, session-delivery replay, and restart-aborted session recovery, but it still lacks the RFC's generic checkpoint lifecycle, checkpoint claim/resume state model, plan-cursor schema, and original-to-resumed run linkage for in-progress tasks.

## What This Changes

The issue requests a first-class gateway-restart checkpoint and auto-resume mechanism that continues interrupted in-progress tasks as seeded follow-up runs without a new user prompt.

## Best Possible Solution

Define a versioned core restart-checkpoint contract that reuses Task Flow progress and existing restart/session-delivery mechanisms where appropriate, with secret-safe persistence and explicit claim, resume, linkage, and failure semantics.

## Reproduction Assessment

Not applicable as a feature RFC. Static inspection gives a high-confidence gap: adjacent recovery paths exist, but the requested generic checkpoint statuses, schema, atomic claim transition, and plan-cursor auto-resume flow do not.

## Solution Assessment

Unclear because no patch is proposed. The seeded follow-up-run direction is plausible, but the maintainable implementation needs maintainer design review across task ownership, side-effect safety, recovery boundaries, and secret-safe persistence.

## Risks / Open Questions

- A generic checkpoint schema could persist secrets or sensitive tool context unless the storage contract is explicitly redacted and reviewed.
- At-most-once resume is difficult across restart loops, concurrent startup paths, and non-idempotent tool calls with side effects.
- The design spans Task Flow, restart sentinels, session delivery, subagent/ACP recovery, and config reload behavior, so a narrow automated fix would likely encode product decisions prematurely.
