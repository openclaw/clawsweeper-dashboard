## Summary

Keep open. Current main has adjacent Task Flow durability, safe restart drain, one-shot restart continuations, session-delivery recovery, main-session recovery, and subagent recovery, but it still lacks the RFC's generic task checkpoint lifecycle, plan-cursor schema, atomic at-most-once claim, and original-to-resumed run linkage.

## What This Changes

Request a first-class gateway restart checkpoint and auto-resume mechanism for in-progress tasks using persisted state and a seeded follow-up run.

## Best Possible Solution

Define a versioned core restart-checkpoint contract that composes with Task Flow, restart sentinels, session delivery, and main/subagent recovery while preserving secret safety and at-most-once resume semantics.

## Reproduction Assessment

Not applicable: this is a feature/RFC rather than a broken existing contract. Source inspection gives high confidence that current main has adjacent recovery paths but not the requested generic checkpoint lifecycle, claim protocol, or run linkage.

## Solution Assessment

Unclear because there is no patch to assess. The seeded follow-up-run direction is plausible, but the implementation needs maintainer design across task state, restart orchestration, idempotency, and persistence boundaries.
