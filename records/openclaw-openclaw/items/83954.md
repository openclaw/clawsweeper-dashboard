## Summary

Current main now handles the concrete Spark mismatch with account-scoped Codex discovery and documents that Spark is available only when the signed-in subscription catalog exposes it, but the broader request still needs a maintainer decision and explicit guidance for `gpt-5.5-pro` and `gpt-5.4-pro`; the separate CLI-bridge proposal is neither necessary nor justified by the current app-server contract.

## What This Changes

Clarify or implement an account-aware supported path for Pro-tier and Spark models through OpenClaw’s Codex app-server, with a CLI bridge only if the existing runtime cannot provide the capability.

## Best Possible Solution

Keep the existing account-scoped Codex app-server catalog as the single source of truth, add concise OpenAI provider documentation stating that `gpt-5.5-pro` and `gpt-5.4-pro` require the route that actually exposes them and are not guaranteed by a ChatGPT Pro subscription, and avoid a separate CLI bridge unless upstream evidence proves app-server cannot preserve a required supported capability.

## Reproduction Assessment

Yes for the core mismatch: current source and tests establish that Spark must follow the signed-in account catalog, and the supplied reports establish consistent `*-pro` rejection through ChatGPT OAuth. A successful `*-pro` turn from an entitled subscription account is not available.

## Solution Assessment

No, the umbrella's CLI-bridge option is not currently the best solution. Account-aware app-server discovery already solves the Spark class of problem; the remaining narrow solution is explicit entitlement and route documentation unless upstream proves the app-server lacks access that the CLI uniquely provides.
