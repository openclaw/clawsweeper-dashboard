## Summary

Keep open: current main has improved the Spark snapshot and suppression wording, but it still does not settle the requested Pro-plan support contract, and the Codex provider continues to expose/synthesize `gpt-5.5-pro` and `gpt-5.4-pro` without account-scoped viability guidance.

## What This Changes

The issue asks OpenClaw to clarify or implement a supported ChatGPT Pro/Codex path for `gpt-5.5-pro`, `gpt-5.4-pro`, and retired Spark-style model refs.

## Best Possible Solution

Maintainers should define the support contract first: either document Pro/Spark as unsupported in the Codex OAuth/app-server path and suppress them from default picker/fallback flows, or add account-scoped viability probing and only surface Pro refs when the selected account can actually run them.

## Reproduction Assessment

No, not for the live backend rejection on current main because that requires a matching account and harness. Source inspection does reproduce the repository-side confusion: Pro refs are still exposed/synthesized while docs do not give a direct supported or unsupported Pro-plan answer.

## Solution Assessment

Unclear; the issue deliberately presents documentation-only, account-probing, and CLI-bridge options. The maintainable path is to choose the product contract before changing code, because each option has different compatibility and support implications.
