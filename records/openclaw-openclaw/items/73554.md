## Summary

Keep open: the bug is real and the PR is aimed at the right CLI boundary, but the latest diff still validates only hard-coded normalized ids against exact plugin ids, which would reject manifest-backed compatibility aliases that current main treats as valid.

## What This Changes

This PR adds pre-write registry validation to `plugins enable` and `plugins disable`, plus policy tests for missing ids and selected legacy compatibility aliases.

## Best Possible Solution

Validate the operator-supplied id through the registry-backed alias contract before mutation, keep true missing ids as hard failures, add command coverage for manifest contribution aliases, and add the required Unreleased fixes changelog entry.

## Reproduction Assessment

Yes. The linked bug has deterministic CLI steps, and current main source shows both enable and disable mutate config before any discovered-plugin validation.

## Solution Assessment

No, not yet. Pre-write validation is the right fix boundary, but this implementation should reuse the manifest/registry alias normalizer rather than only the hard-coded `normalizePluginId` fallback.
