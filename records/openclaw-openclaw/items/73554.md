## Summary

Keep open: the PR fixes the real stale-config bug in the right area, but the latest guard still rejects manifest-driven compatibility aliases before config mutation and the required user-facing changelog entry is missing.

## What This Changes

The PR adds pre-write plugin-registry validation to `plugins enable` and `plugins disable`, with policy tests for unknown ids and selected compatibility aliases.

## Best Possible Solution

Resolve policy command input through the alias-aware registry metadata normalizer, keep the unknown-id no-write guard, cover manifest aliases for both enable and disable, and add the changelog entry.

## Reproduction Assessment

Yes. The linked bug has deterministic CLI steps, and current main source shows both policy commands write config before discovered-plugin validation; the remaining PR regression is reproducible with a registry alias that is not in the static fallback list.

## Solution Assessment

No, not yet. Registry validation is the right boundary, but the implementation should use the manifest-aware registry alias contract rather than static fallback normalization.
