## Summary

Keep open: the underlying CLI bug is real, but this PR is not ready because the new pre-write guard still rejects registry-backed plugin aliases and the user-facing behavior change lacks the required changelog entry.

## What This Changes

The PR adds registry-snapshot validation and CLI policy tests so `plugins enable` and `plugins disable` reject undiscovered plugin ids before writing config.

## Best Possible Solution

Keep the pre-write validation boundary, but resolve command input through the registry-backed alias contract before checking existence and document the CLI fix in the Unreleased changelog.

## Reproduction Assessment

Yes. The linked bug has deterministic CLI steps, and current-main source shows both policy commands mutate config before any discovered-plugin validation; the PR adds focused tests for the missing-id no-write path.

## Solution Assessment

No, not yet. Registry validation before mutation is the right shape, but it must preserve the registry alias contract rather than only the static fallback aliases, and the fix needs the required changelog entry.
