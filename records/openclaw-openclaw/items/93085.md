## Summary

Keep open: the linked performance bug is real, but this patch should not merge as-is because the low-level TTL cache can return pre-mutation task snapshots to non-status callers, and the contributor proof is tests/build/grep only.

## What This Changes

The PR adds a 5 ms module-level cache around `reconcileInspectableTasks()` and clears it from the task-registry maintenance test reset helper.

## Best Possible Solution

Share one reconciled task snapshot inside `getStatusSummary()` for the status hot path, or add explicit dirty invalidation for task upsert/delete/restore and maintenance configuration changes before keeping a module-level cache.

## Reproduction Assessment

Yes by source inspection: current main calls the task registry summary and task audit paths separately during status, and the PR's cache can be traced through a post-mutation `tasks maintenance --apply` read. I did not run a live large-ledger timing benchmark in this read-only review.

## Solution Assessment

No: the performance issue is real, but a 5 ms module-level cache is broader than the status hot path and lacks freshness invalidation. A per-status shared snapshot, or a dirty invalidated cache with tests, is the safer fix.

- **[P2] Invalidate cached task snapshots after mutations:** `src/tasks/task-registry.maintenance.ts:886-887`
  - body: This cache is populated by any caller of `reconcileInspectableTasks()`, but it is only time-based. Please either keep the optimization scoped to the `getStatusSummary()` call or clear the cache on task upsert/delete/restore and `configureTaskRegistryMaintenance()` changes.
