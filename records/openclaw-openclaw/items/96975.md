## Label Justifications

- `P1`: Fresh live-channel subagent workflows can stall or leak completion output when large child results feed parent delivery and context paths.
- `impact:session-state`: The issue targets parent context pressure, compaction, session locks, and child/parent session isolation.
- `impact:message-loss`: The issue reports subagent completion announce failures and stale or raw child output reaching user-facing channels.

## Summary

Keep open: current main and v2026.6.10 still use a result-bearing subagent completion contract that reads child output, stores it as `task_completion.result`, and formats it into parent prompt context; related open items cover narrower caps, leaks, or lock durability but do not replace the metadata-only completion-envelope decision.

## What This Changes

The issue requests isolating subagent completion from the parent by default, returning only a bounded status/session envelope plus an explicit child-session handle instead of raw child output or long result payloads.

## Best Possible Solution

Define and document a bounded subagent completion-envelope contract, update the protocol/runtime/tests/docs around that contract, and keep full child details available only through explicit bounded history or artifact reads.

## Reproduction Assessment

Yes, at source level: current main still reads child assistant output, stores it as `task_completion.result`, and formats it into parent prompt context. I did not rerun the live Feishu workflow that produced the reported parent context pressure.

## Solution Assessment

Unclear until maintainers choose the completion contract; metadata-only completion is directionally safer, but it changes the documented result-bearing handoff behavior and may need an explicit compatibility policy.
