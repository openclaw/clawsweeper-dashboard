## Summary

Keep open: current main and v2026.6.10 still put selected child result text into the subagent task-completion event and parent prompt path, while the newer steering queue cap only bounds one retry/queue path and related open items cover adjacent delivery durability issues.

## What This Changes

Request metadata-only, bounded subagent completion handoffs with explicit child-session/result inspection instead of automatic child result injection into parent and user-channel paths.

## Best Possible Solution

Define one canonical subagent completion contract: metadata/status/session handle by default, at most a bounded summary, explicit bounded child-history or artifact reads for details, and aligned protocol, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main still reads child output, stores it as the subagent task-completion `result`, and formats it into the parent prompt. I did not rerun the live Feishu workflow that produced the reported pressure.

## Solution Assessment

Unclear: the metadata-only default is directionally aligned with isolation, but the permanent envelope, explicit result-reader path, and any compatibility mode need maintainer product/API direction before implementation.
