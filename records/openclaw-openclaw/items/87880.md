## Summary

This should stay open: current main already carries session metadata through the exec/tool runtime for routing and notifications, but it does not expose the requested session identifiers in subprocess environments. The request is a plausible core exec feature, not a stale or ClawHub-only item, but it needs maintainer product/security review before implementation because it would create a new implicit environment contract for every spawned command.

## What This Changes

The issue requests automatic injection of current session identifiers into subprocess environments spawned by the exec tool.

## Best Possible Solution

Define an explicit core exec-env contract, then implement it consistently for gateway, sandbox, and node exec with user-provided env taking precedence, focused tests, and docs if maintainers approve the security/API shape.

## Reproduction Assessment

Not applicable: this is a feature request, not a report of broken documented behavior. Source and release inspection confirm current exec subprocesses only receive the documented OPENCLAW_SHELL marker, not the requested session variables.

## Solution Assessment

Unclear until maintainers decide the env contract: automatic injection is a reasonable implementation direction, but exposing session IDs and session keys to every subprocess should be an explicit product/security decision. If accepted, the maintainable path is one shared env augmentation helper used before gateway, sandbox, and node dispatch.
