## Summary

Keep open: current main still serializes same-session Discord inbound jobs before the reply queue can apply steering, and the PR's bypass is not session-aware enough to honor documented queue precedence.

## What This Changes

The PR bypasses the Discord inbound per-session run queue for config-level `steer`/`steer-backlog` modes and adds same-session concurrency regression coverage.

## Best Possible Solution

Rebase into the active `extensions/discord` code and resolve the effective queue mode with session precedence before deciding whether the outer Discord run queue should serialize the job.

## Reproduction Assessment

Yes, at source level. Same-session Discord jobs share a session-scoped queue key and current main always routes them through a keyed serial queue before reply steering can run.

## Solution Assessment

No. The PR direction is useful, but the patch should use the effective session-aware queue mode and land against the active Discord plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:179-188`
  - body: The bypass decision only resolves config/by-channel mode, so stored per-session `/queue` overrides are ignored. With global Discord `steer`, a session switched to a serialized mode can still skip `runQueue.enqueue(...)`; with global non-steer and a session override to `steer`, it remains serialized. Resolve the effective mode with session precedence before deciding whether to bypass the outer queue.
  - confidence: 0.88
