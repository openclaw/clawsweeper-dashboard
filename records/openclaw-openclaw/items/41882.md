## Summary

Keep open. Current main still serializes Discord accepted jobs through the active plugin run queue before reply steering can run, and the PR is useful but needs a current-main port plus effective queue-mode handling before merge.

## What This Changes

The PR bypasses the Discord inbound worker queue for `steer` and `steer-backlog` modes and adds same-session concurrency regression coverage.

## Best Possible Solution

Land a focused current-main fix in the active Discord plugin path that bypasses the outer run queue only for the effective `steer` or `steer-backlog` mode, while preserving serialized behavior for non-steer session overrides.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main enqueues accepted Discord jobs by session key, `createChannelRunQueue` serializes same-key work, and the active queue test asserts the second same-session run waits for the first.

## Solution Assessment

No. The PR direction is right, but the proposed bypass resolves only cfg-level mode and targets the old `src/discord` layout; the maintainable fix should use the effective queue mode in `extensions/discord`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective mode before bypassing:** `src/discord/monitor/message-handler.ts:183-188`
  - body: The bypass resolves only config-level Discord queue settings. With global/default `steer` but a stored per-session `/queue followup`, `/queue collect`, or `/queue interrupt` override, same-session Discord messages skip `runQueue.enqueue(...)` and can start concurrently even though that session requested serialized behavior. Resolve the effective session mode before deciding to bypass.
  - confidence: 0.86
