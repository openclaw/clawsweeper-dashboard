## Summary

Keep open. The Discord steer bug is still source-reproducible on current main, but this PR should not merge as-is because it targets the retired `src/discord` layout, resolves only config-level queue mode, and lacks real behavior proof.

## What This Changes

The branch bypasses the Discord same-session inbound run queue for config-level `steer`/`steer-backlog` modes and adds mocked concurrency coverage in the old `src/discord` layout.

## Best Possible Solution

Land a rebased Discord plugin fix that bypasses the outer run queue for effective `steer` and `steer-backlog` modes only after honoring per-session overrides, with regression coverage and redacted real Discord or terminal proof.

## Reproduction Assessment

Yes. Current main always enqueues accepted Discord inbound jobs into the session-scoped run queue, and the shared keyed queue serializes same-key work before inner steering can observe a follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should resolve the effective session-aware queue mode and be rebased into `extensions/discord` before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:180-188`
  - body: The bypass decision resolves mode from only `cfg` and `channel`, so stored per-session `/queue` overrides are ignored. With global `steer`, a session switched to a serialized mode can still skip `runQueue.enqueue(...)`; with global non-steer and a session override to `steer`, it remains serialized. Resolve the effective session-aware mode before deciding whether to bypass the outer queue.
