## Summary

This PR should stay open but needs changes before merge: the reported Discord steer-mode bug is real on current main, but the proposed bypass uses only config-level queue settings and can contradict stored per-session `/queue` overrides.

## What This Changes

The PR changes the legacy Discord handler to bypass its per-session inbound worker queue for config-level `steer` and `steer-backlog` modes and adds a same-session concurrency regression test.

## Best Possible Solution

Implement the fix in the active Discord plugin path by resolving the effective per-session queue mode before the outer run queue and bypassing only for `steer` and `steer-backlog`, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: same-session Discord messages share `job.queueKey`, `createChannelRunQueue` serializes that key, and current tests assert the second same-session run waits for the first.

## Solution Assessment

No. The direction is right, but the proposed bypass uses config-level mode instead of the effective queue mode that honors stored per-session overrides; the active `extensions/discord` path also needs the actual landing change.

## Review Findings

Overall correctness: patch is incorrect
