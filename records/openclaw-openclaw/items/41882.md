## Summary

Keep open: the Discord steer-mode bug is still source-reproducible on current main, but this PR still ignores session-level queue-mode precedence and lacks real after-fix behavior proof.

## What This Changes

The PR bypasses the Discord inbound per-session queue for config-level `steer` and `steer-backlog` modes and adds same-session concurrency regression coverage.

## Best Possible Solution

Port the fix to `extensions/discord` through a plugin-safe queue-mode seam, resolve the effective queue mode with session precedence, and keep focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Same-session Discord jobs share a session-scoped queue key and current main routes them through a keyed serial queue before the steer-capable reply path can run.

## Solution Assessment

No. Bypassing the outer queue is the right direction for steer modes, but the patch should use the effective session-aware queue mode and land against the active Discord plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:179-188`
  - body: The bypass decision only resolves config/by-channel mode, so stored per-session `/queue` overrides are ignored. With global Discord `steer`, a session switched to a serialized mode can still skip `runQueue.enqueue(...)`; with global non-steer and a session override to `steer`, it remains serialized. Resolve the effective mode with session precedence before deciding whether to bypass the outer queue.
