## Summary

Keep this PR open because it addresses a real Discord steer-mode bug, but the current branch is not mergeable as-is: it targets the deleted `src/discord` path, bypasses based only on global/channel config instead of the effective per-session queue mode, and still lacks after-fix real Discord behavior proof.

## What This Changes

The PR adds a Discord inbound-worker bypass for steer-style queue modes and a regression test for same-session message concurrency.

## Best Possible Solution

Land a rebased fix in `extensions/discord` that bypasses the outer Discord run queue only when the routed session’s effective queue mode resolves to `steer`, with focused regression coverage and redacted real Discord proof.

## Reproduction Assessment

Yes. By source inspection, current `main` and `v2026.5.12` enqueue accepted Discord jobs into a same-key run queue before reply steering can see a same-session follow-up; a focused test can block the first run and confirm the second cannot start.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this patch needs to target the active Discord plugin path and decide from the effective session-aware queue settings rather than global/channel config only.
