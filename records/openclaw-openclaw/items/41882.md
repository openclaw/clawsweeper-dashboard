## Summary

Keep this PR open: the reported Discord steer-mode bug still exists on current main, but the submitted branch is stale against the current plugin layout and needs a rebased or replacement implementation before it can land.

## What This Changes

The PR changes the legacy Discord message handler to bypass the per-session queue for steer and steer-backlog modes and adds regression coverage for same-session concurrency.

## Best Possible Solution

Implement the fix in the current Discord plugin queue path: determine the effective queue mode early enough to bypass the outer per-session queue for steer and steer-backlog, keep collect/followup/interrupt/queue modes serialized, and add focused tests for steer, steer-backlog, non-steer serialization, session override precedence, replay dedupe, timeout/deactivation, and busy-state behavior.

## Reproduction Assessment

Yes: a source-level reproduction path is clear. Same-session Discord messages share `job.queueKey`, `createChannelRunQueue` serializes that key, and current tests still assert the second same-session run waits for the first.

## Solution Assessment

No for the PR as currently submitted. The proposed bypass is directionally right, but it targets the old file path and still needs a current-main implementation that honors effective queue-mode precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Honor session overrides before bypassing the queue:** `src/discord/monitor/message-handler.ts:180-183`
