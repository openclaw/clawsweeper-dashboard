## Summary

Keep open: current main still serializes Discord inbound jobs before steering can run, so the central bug remains. This branch is not merge-ready because it targets the retired `src/discord` layout, ignores session-level queue override precedence, and lacks real behavior proof.

## What This Changes

The PR edits the old `src/discord` handler to bypass the same-session run queue for config-level `steer`/`steer-backlog` modes and adds mocked concurrency coverage.

## Best Possible Solution

Land a rebased Discord plugin fix in `extensions/discord` that bypasses the outer run queue for effective `steer` and `steer-backlog` modes after resolving session-aware queue settings, with focused regression coverage and redacted real Discord or terminal proof.

## Reproduction Assessment

Yes, source-reproducible. Current main always enqueues accepted Discord inbound jobs into the Discord run queue, and that queue serializes same-key work through `KeyedAsyncQueue` before inner steering can observe a follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but the fix should be rebased into `extensions/discord` and should use the effective session-aware queue mode rather than config-only resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:177-186`
  - body: The bypass decision resolves mode from only `cfg` and `channel`, so stored per-session `/queue` overrides are ignored. With global `steer`, a session switched to a serialized mode can still skip `runQueue.enqueue(...)`; with global non-steer and a session override to `steer`, it remains serialized. Resolve the effective session-aware mode before deciding whether to bypass the outer queue.
