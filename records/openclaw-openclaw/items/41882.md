## Summary

Keep open: the Discord steer bug is still source-reproducible on current main, but this PR is not merge-ready because it targets the old Discord path, bypasses only config-level queue mode, and lacks real behavior proof.

## What This Changes

The PR bypasses Discord's same-session inbound worker queue for config-level `steer`/`steer-backlog` modes and adds mocked concurrency coverage in the old `src/discord` layout.

## Best Possible Solution

Land a rebased Discord plugin fix that decides bypass from the effective session-aware queue mode, covers `steer` and `steer-backlog`, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main always enqueues accepted Discord inbound jobs into the session-scoped run queue, and that queue serializes same-key work before inner steering can observe a follow-up. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should rebase into `extensions/discord` and use the effective session-aware queue mode before deciding to bypass.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:180-188`

Resolve the effective session-aware mode before deciding whether to bypass the outer queue.
