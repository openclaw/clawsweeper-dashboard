## Summary

Keep open: the Discord steer bug is still source-reproducible on current main, and the PR is directionally useful, but the patch is not ready because it targets the old Discord layout, ignores effective per-session queue mode, and lacks real behavior proof.

## What This Changes

The PR changes the old `src/discord` message handler and queue test to bypass the same-session inbound worker queue for config-level `steer` and `steer-backlog` modes.

## Best Possible Solution

Land a rebased Discord plugin fix that resolves the effective queue mode before bypassing the outer run queue, treats `steer` and `steer-backlog` as bypass modes, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. Current main source shows every accepted Discord inbound job enters a session-scoped run queue, and the shared keyed queue serializes same-key tasks before the inner steering path can observe a follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should be rebased into `extensions/discord` and decide from the effective session-aware queue mode rather than config-only mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:180-188`
  - body: The bypass helper resolves only `cfg` plus `channel`, so stored per-session `/queue` overrides are ignored. With global `steer` but a session switched to a serialized mode, Discord skips `runQueue.enqueue(...)`; with global non-steer but stored `steer`, it remains serialized. Resolve the effective session-aware mode before deciding whether to bypass the outer Discord queue.
  - confidence: 0.88
