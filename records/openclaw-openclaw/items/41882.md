## Summary

Keep open: the Discord steer-mode bug is still present on current main/latest release, but this external PR is not merge-ready because it targets the deleted `src/discord` path, lacks real behavior proof, and resolves bypass from config only instead of the effective session-aware queue mode.

## What This Changes

The PR bypasses the Discord inbound per-session queue for config-level `steer`/`steer-backlog` modes and adds same-session concurrency coverage under the old `src/discord` path.

## Best Possible Solution

Land a rebased Discord plugin fix that bypasses the outer run queue only for effective `steer`/`steer-backlog` modes after session and inline override resolution, with focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main and `v2026.5.7` enqueue accepted Discord inbound jobs into a session-keyed run queue that serializes same-key work before inner steering can run. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should target the active `extensions/discord` implementation and base the bypass on the same effective session-aware queue mode used by reply execution.
