## Summary

Keep open: the PR targets a real Discord steer-mode bug, but current main still routes every Discord inbound job through the per-session run queue, and the proposed bypass is incomplete because it ignores stored session queue overrides and the documented legacy `queue` steering mode.

## What This Changes

The PR bypasses the Discord inbound worker queue for config-resolved `steer`/`steer-backlog` modes and adds a same-session concurrency regression test.

## Best Possible Solution

Carry the fix into the current Discord plugin queue path using the effective per-session queue mode, preserve serialization for non-steer modes, and cover config plus session-override cases with regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main enqueues every Discord inbound job through a per-session `KeyedAsyncQueue` before the inner reply pipeline resolves `steer` behavior, so a second same-session Discord message cannot reach active-run steering until the first job finishes.

## Solution Assessment

No. The PR's direction is right, but the bypass should be based on the effective queue mode for the target session in the current `extensions/discord` implementation, not only on global config in the old path.
