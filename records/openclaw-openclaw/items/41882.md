## Summary

Keep open: the Discord steer serialization bug is still source-reproducible on current main, but this PR is not correct as-is because it targets deleted `src/discord` paths, lacks real behavior proof, and bypasses without honoring effective per-session queue settings.

## What This Changes

The PR changes Discord inbound handling so steer-style queue modes bypass the per-session outer run queue, with a regression test for same-session concurrency.

## Best Possible Solution

Rebase the fix into `extensions/discord`, bypass the outer Discord run queue only for the effective session-aware `steer` mode, preserve non-steer serialization, add focused regression coverage, and provide redacted Discord proof.

## Reproduction Assessment

Yes. Source-level inspection shows current main preflights accepted Discord messages and then enqueues each job into a same-key run queue, so a focused test can block the first same-session run and confirm the second cannot start before it finishes.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR must target the active plugin path and resolve the effective session-aware queue mode instead of checking global/channel config only.
