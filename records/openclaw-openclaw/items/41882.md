## Summary

Keep this PR open: it targets a real Discord steer-mode bug that is still source-reproducible on current main, but the proposed patch is not mergeable as-is because it only checks config-level queue mode, ignores per-session `/queue` overrides, targets the retired `src/discord` layout, and lacks real behavior proof.

## What This Changes

The PR bypasses the Discord same-session inbound run queue for config-level `steer`/`steer-backlog` modes and adds mocked concurrency coverage against the old `src/discord` layout.

## Best Possible Solution

Land a session-aware Discord run-queue bypass in `extensions/discord` that treats `steer` and `steer-backlog` as bypass modes only after resolving the effective queue mode, with regression coverage and redacted real Discord or terminal proof.

## Reproduction Assessment

Yes, source-reproducible. Current main always enqueues accepted Discord inbound jobs through a session-scoped run queue, and `KeyedAsyncQueue` serializes same-key work before inner steering can observe a follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should resolve the effective session-aware queue mode and be rebased into the active `extensions/discord` layout before merge.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:179-188`

Resolve the effective session-aware mode before deciding whether to bypass the outer queue.
