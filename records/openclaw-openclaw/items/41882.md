## Summary

Keep open: the reported Discord steer serialization path is real on current main, but this branch is not mergeable as-is because it targets the old core Discord path, misses documented per-session queue precedence, and lacks required real Discord behavior proof.

## What This Changes

The PR changes Discord inbound message handling so config-level steer and steer-backlog queue modes bypass the outer per-session queue, with regression coverage for same-session concurrency.

## Best Possible Solution

Rebase the fix onto extensions/discord, make Discord run admission use the same queue-mode precedence as the reply pipeline, drop retired steer-backlog handling on current main, and add real Discord proof.

## Reproduction Assessment

Yes, source-level. Current main still sends accepted Discord messages through createChannelRunQueue keyed by session before processDiscordMessage can reach the steer-aware reply queue, so same-session steer input cannot be admitted until the prior job leaves the outer queue.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the branch targets old paths, bases bypass on config instead of the documented queue precedence, and carries retired steer-backlog handling that current main no longer accepts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor session queue overrides before bypassing:** `src/discord/monitor/message-handler.ts:182-185`
  - body: The bypass decision only calls `resolveQueueSettings` with `cfg`, but queue mode precedence gives stored per-session `/queue` overrides priority over global/by-channel config. With global `followup` and a session set to `steer`, Discord still goes through the outer queue and cannot steer the active run; with global `steer` and a session set to `followup` or `collect`, the outer queue is skipped against the session setting. Pass the active session entry or otherwise use the same resolved queue facts as the reply pipeline before deciding to bypass.
  - confidence: 0.87
- **[P2] Drop retired steer-backlog handling on rebase:** `src/discord/monitor/message-handler.ts:186-187`
  - body: Current main's queue contract only allows `steer`, `followup`, `collect`, and `interrupt`, and doctor treats `steer-backlog` as retired. Carrying `steer-backlog` in the bypass and test fixture would revive a removed config mode when this branch is rebased; update the branch to current queue modes instead.
  - confidence: 0.82
