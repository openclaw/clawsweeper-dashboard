## Summary

Keep open: the Discord steer serialization bug is still source-reproducible on current main, but this PR is not merge-ready because it targets the retired `src/discord` layout, bypasses using only config-level queue mode, and lacks real behavior proof.

## What This Changes

The PR changes the old Discord message handler and queue test to bypass the same-session inbound worker queue for config-level `steer` and `steer-backlog` modes.

## Best Possible Solution

Land a small Discord plugin change in `extensions/discord` that resolves the effective queue mode before bypassing the outer run queue, covers `steer` and `steer-backlog`, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main always enqueues accepted Discord inbound jobs into the session-scoped channel run queue, and the shared keyed queue serializes same-key work before inner steering can observe a follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should be rebased into `extensions/discord` and decide from the effective session-aware queue mode rather than config-only mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:180-188`
  - body: The bypass helper resolves mode from only `cfg` and `channel`, so stored or inline per-session `/queue` overrides are ignored. With global `steer`, a session switched to a serialized mode can still skip `runQueue.enqueue(...)`; with global non-steer and a session override to `steer`, it remains serialized. Resolve the effective session-aware queue mode before deciding whether to bypass the outer Discord queue.
  - confidence: 0.88
