## Summary

This PR addresses a real Discord queue bug that still exists on current main, but the branch is not merge-ready: it targets the old `src/discord` layout, bypasses based only on config-level mode instead of the effective session-aware mode, and lacks after-fix real behavior proof.

## What This Changes

The PR bypasses the Discord same-session inbound run queue for config-level `steer` and `steer-backlog` modes and adds mocked concurrency coverage in the old `src/discord` path.

## Best Possible Solution

Land a rebased Discord plugin fix that decides the outer-queue bypass from the effective session-aware queue mode, covers `steer` and `steer-backlog`, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main always sends accepted Discord inbound jobs through the session-scoped run queue, which serializes same-key work before inner steering can observe a follow-up. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should rebase into `extensions/discord` and resolve the same effective session-aware queue mode that reply execution uses before deciding to bypass.
