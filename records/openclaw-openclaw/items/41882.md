## Summary

Keep open: the Discord steer serialization bug is still present on current main, but this PR is not merge-ready because the bypass uses config-only queue resolution, ignores stored per-session queue overrides, targets the old Discord path layout, and has no after-fix real behavior proof.

## What This Changes

The PR changes the Discord inbound worker to bypass the same-session run queue for config-level `steer`/`steer-backlog` modes and adds mocked same-session concurrency coverage.

## Best Possible Solution

Land a session-aware Discord run-queue bypass in `extensions/discord` that treats `steer` and `steer-backlog` as bypass modes only after resolving the effective queue mode, with regression coverage and real Discord proof.

## Reproduction Assessment

Yes, source-reproducible. Current main sends accepted Discord inbound jobs through a session-scoped channel run queue, and the keyed queue serializes same-key jobs before reply steering can observe the follow-up.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this patch should use the effective session-aware queue mode and be rebased into the active `extensions/discord` layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the effective queue mode before bypassing:** `src/discord/monitor/message-handler.ts:179-188`
  - body: The bypass decision resolves mode from only `cfg` and `channel`, so stored per-session `/queue` overrides are ignored. Resolve the effective session-aware mode before deciding whether to bypass the outer queue.
