## What This Changes

The PR modifies the old Discord message handler and queue test to bypass the outer per-session run queue for config-level `steer`/`steer-backlog` modes and add same-session concurrency coverage.

## Best Possible Solution

Rebase into `extensions/discord`, decide bypass from the effective session-aware queue mode, preserve serialized modes, add focused regression coverage, and attach redacted live Discord proof.

## Reproduction Assessment

Yes, source-level: current main preflights accepted Discord messages and enqueues each job into `createChannelRunQueue` by session key, whose `KeyedAsyncQueue` serializes same-key tasks. A focused test can block the first same-session run and assert the second cannot enter processing until the first releases.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this branch must target the active Discord plugin path and resolve the effective session-aware queue mode before bypassing.
