## Summary

Keep open: the underlying Discord steer serialization bug still appears present in current main and the latest release, but this branch is not merge-ready because the bypass uses only global/channel config, conflicts with session queue overrides, targets a stale path, and lacks real behavior proof.

## What This Changes

This PR bypasses the Discord inbound run queue for global `steer`/`steer-backlog` modes and adds mocked same-session concurrency coverage under the old `src/discord` path.

## Best Possible Solution

Land a rebased Discord plugin fix that bypasses the outer queue only for the effective `steer`/`steer-backlog` mode after per-session/inline overrides, with focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main and `v2026.5.7` always enqueue accepted Discord inbound jobs through the session-scoped run queue, which serializes same-key work before inner steering can observe a follow-up. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. Bypassing the outer queue is the right direction, but this PR should target the active `extensions/discord` implementation and base the bypass on the same effective session-aware queue mode that reply execution uses.
