## Summary

Keep this open. The report describes a plausible current-main routing bug: when a session has a routable Feishu deliveryContext but the active origin is webchat/control-ui, the reply route code can prefer the webchat origin and suppress external routing instead of using the stored delivery target.

## What This Changes

The issue asks for Feishu group replies to follow the session deliveryContext after a control-ui initiated message(action=send) targets a Feishu group.

## Best Possible Solution

Make reply routing prefer a routable persisted deliveryContext for channel-scoped sessions when the current origin is internal webchat/control-ui, while preserving webchat-only sessions and explicit-deliver semantics.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows a matching failure path when ctx provider/surface are webchat and the session entry has a Feishu deliveryContext, because current routing suppresses the external route unless ExplicitDeliverRoute is true.

## Solution Assessment

Yes, the likely narrow fix belongs in the shared reply-route decision path rather than Feishu send itself. The regression test should cover a Feishu deliveryContext with a webchat/control-ui origin and also keep the existing webchat-only protections green.
