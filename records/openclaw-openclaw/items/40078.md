## Label Justifications

- `P2`: The issue blocks a useful Discord voice-memo workflow, but the fix is an opt-in feature/config decision with limited blast radius.
- `impact:session-state`: The requested behavior changes whether derived audio transcripts are persisted into channel history and replayed into later context.
- `impact:security`: Silent transcription would process and retain speech from messages that did not explicitly invoke the bot, changing privacy expectations.
- `impact:message-loss`: The later mentioned turn receives only audio placeholders instead of usable spoken content.

## Summary

Keep open: current main and v2026.6.8 still do not provide the requested silent Discord transcription mode for unmentioned requireMention audio; source shows skipped messages record placeholder history and return before transcript-backed dispatch context is built.

## What This Changes

The issue requests a default-off per-channel or guild option to silently transcribe Discord audio in requireMention channels so later mentioned prompts can use those transcripts from history.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy with explicit authorization order, privacy/retention/cost behavior, schema/docs, and focused requireMention-history tests before implementing it.

## Reproduction Assessment

Yes, source-level: an unmentioned Discord guild audio message in a requireMention channel records placeholder-based history and returns before admitted-turn transcript context. I did not run live Discord/STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone; the requested direction is plausible, but the best fix needs maintainer-approved privacy, retention, cost, inheritance, observability, schema, docs, and tests before adding the config surface.
