## Label Justifications

- `P2`: The issue blocks a useful Discord voice memo workflow, but the fix is an opt-in feature/config decision with limited blast radius rather than an urgent regression.
- `impact:message-loss`: Unmentioned Discord voice messages are retained as media placeholders, so later mentioned turns cannot use their spoken content.
- `impact:session-state`: The requested behavior changes whether derived speech transcripts are persisted into channel history and replayed into later context.
- `impact:security`: Silent transcription would send and retain speech from messages that did not explicitly invoke the bot, changing privacy expectations.

## Summary

Keep open: current main and v2026.6.6 still only use Discord preflight audio transcription for mention detection, then store skipped unmentioned audio as placeholder history. The requested workflow remains source-reproducible, but it needs maintainer product and security decisions before implementation.

## What This Changes

Request to add a default-off Discord `preflightTranscribe` guild/channel option that silently transcribes unmentioned audio in `requireMention` channels and stores the transcript for later context without triggering a reply.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy with explicit authorization order, transcript retention, provider-cost behavior, operator visibility, schema/docs, and tests before implementation.

## Reproduction Assessment

Yes at source level: current main transcribes Discord audio for mention detection, but the skipped no-mention path records history from placeholder text without the transcript. I did not run a live Discord STT repro in this read-only review.

## Solution Assessment

No as a bare boolean: the requested direction is plausible, but the best fix needs an explicit policy covering privacy, retention, cost, inheritance, and observability semantics rather than only adding `preflightTranscribe: true`.
