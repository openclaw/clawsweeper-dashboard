## Label Justifications

- `P2`: The issue blocks a useful Discord voice memo workflow, but the fix is an opt-in feature/config decision with limited blast radius rather than an urgent regression.
- `impact:message-loss`: Unmentioned Discord voice messages are retained as media placeholders, so later mentioned turns cannot use their spoken content.
- `impact:session-state`: The requested behavior changes whether derived speech transcripts are persisted into channel history and replayed into later context.
- `impact:security`: Silent transcription would process and retain speech from messages that did not explicitly invoke the bot, changing privacy expectations.

## Summary

Keep open: current main and v2026.6.6 still use Discord audio preflight only for mention detection on admitted paths, while skipped unmentioned audio is recorded as media-placeholder history; the requested default-off silent transcription mode still needs maintainer product and security decisions.

## What This Changes

Request a Discord guild/channel option to silently transcribe unmentioned audio in requireMention channels and persist transcripts for later mentioned turns.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy with explicit authorization order, transcript retention, provider-cost behavior, operator visibility, schema/docs, and focused tests before implementation.

## Reproduction Assessment

Yes, source-level: in a requireMention Discord guild channel with historyLimit and audio STT configured, an audio-only unmentioned message reaches preflight, fails the mention decision, records dropped history from placeholder text, and returns null. I did not run live Discord/STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone: the requested behavior is plausible, but the best solution needs an explicit policy for privacy, retention, cost, inheritance, and observability before adding the config surface.
