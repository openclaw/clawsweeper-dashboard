## Label Justifications

- `P2`: The issue blocks a useful Discord voice-memo workflow, but the fix is an opt-in feature/config decision with limited blast radius.
- `impact:session-state`: The requested behavior changes whether derived audio transcripts are persisted into channel history and replayed into later context.
- `impact:security`: Silent transcription would process and retain speech from messages that did not explicitly invoke the bot, changing privacy expectations.
- `impact:message-loss`: The later mentioned turn receives only audio placeholders instead of usable spoken content.

## Summary

Keep open: current main and v2026.6.8 still use Discord audio preflight for mention detection, while unmentioned `requireMention` audio is dropped into history as placeholder text; the requested silent transcription mode needs maintainer product and security decisions before implementation.

## What This Changes

Request a default-off Discord guild/channel option that silently transcribes audio in `requireMention` channels so later mentioned turns can use transcripts from history.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy with explicit authorization order, privacy/retention/cost behavior, schema/docs, and focused requireMention-history tests before implementing it.

## Reproduction Assessment

Yes, source-level: with `requireMention: true`, an audio-only unmentioned Discord guild message can be transcribed for mention detection, but if it is not considered a mention, the no-mention branch records placeholder history and returns null. I did not run live Discord/STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone: the requested behavior is plausible, but the maintainable solution needs explicit policy for privacy, retention, cost, inheritance, and observability before adding the config surface.
