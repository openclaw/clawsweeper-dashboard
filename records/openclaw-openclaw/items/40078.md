## Label Justifications

- `P2`: The issue blocks a useful Discord voice memo workflow, but the fix is an opt-in feature/config decision with limited blast radius rather than an urgent regression.
- `impact:message-loss`: Unmentioned Discord voice messages in requireMention channels are retained as placeholders, so later mentioned turns cannot use their spoken content.
- `impact:session-state`: The requested behavior changes whether derived speech transcripts are persisted into channel history and replayed into later context.
- `impact:security`: Silent transcription would process and retain speech from messages that did not explicitly invoke the bot, changing privacy expectations.

## Summary

Keep open: current main and v2026.6.6 still only use Discord audio preflight for mention detection and admitted turns; unmentioned audio in requireMention channels is still recorded as placeholder history, so the requested silent history transcription mode remains a product and security decision.

## What This Changes

Request a default-off Discord guild/channel `preflightTranscribe` option that silently transcribes unmentioned audio in requireMention channels and stores the transcript for later context.

## Best Possible Solution

Define a Discord-owned, default-off silent audio transcription policy that covers authorization, consent/visibility, retention, cost limits, inheritance, schema/docs, and tests before implementation.

## Reproduction Assessment

Yes, source-level: in a Discord guild channel with `requireMention`, history enabled, and audio STT configured, an audio-only unmentioned message can reach preflight, fail the mention decision, record placeholder history, and return null. I did not run live Discord/STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone: the requested behavior is plausible, but the best solution needs explicit privacy, retention, provider-cost, inheritance, and observability policy before adding the config surface.
