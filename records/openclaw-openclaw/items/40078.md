## Label Justifications

- `P2`: The issue blocks a bounded Discord voice-memo workflow, but the fix is an opt-in feature/config decision rather than a core outage.
- `impact:session-state`: The requested behavior changes whether derived audio transcripts are retained in channel history and replayed into later context.
- `impact:security`: Silent transcription would process speech from messages that did not explicitly invoke OpenClaw.
- `impact:message-loss`: Later mentioned turns receive audio placeholders instead of usable spoken content for earlier voice messages.

## Summary

Keep open: current main and v2026.6.10 still lack the requested default-off silent Discord transcription mode for unmentioned audio in `requireMention` channels, and the remaining work needs maintainer product/security approval before adding a new config surface.

## What This Changes

Request a Discord guild/channel `preflightTranscribe` option that silently transcribes unmentioned audio in `requireMention` channels and stores transcripts for later mentioned turns.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy that stores bounded transcript history without admitting an agent turn, then implement it with schema/docs and focused `requireMention` history tests.

## Reproduction Assessment

Yes at source level: current Discord preflight can transcribe audio for mention detection, but if no mention is found it records pending history from placeholder text and returns before transcript-backed context is built. I did not run live Discord plus STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone. The direction is plausible, but the best fix needs maintainer-approved privacy, retention, cost, inheritance, schema, docs, and tests before adding the config surface.
