## Label Justifications

- `P2`: The issue blocks a useful Discord voice-memo workflow, but the fix is an opt-in feature/config decision with limited blast radius.
- `impact:session-state`: The requested behavior changes whether derived audio transcripts are retained in channel history and replayed into later context.
- `impact:security`: Silent transcription would process speech from messages that did not explicitly invoke the bot, changing privacy expectations.
- `impact:message-loss`: Later mentioned turns receive only audio placeholders instead of usable spoken content for earlier voice messages.

## Summary

Keep open: current main and v2026.6.8 still lack the requested silent Discord transcription mode for unmentioned `requireMention` audio, and the remaining work needs maintainer security/product approval before implementation.

## What This Changes

Request a Discord guild/channel `preflightTranscribe` option that silently transcribes unmentioned audio in `requireMention` channels and stores transcripts for later context without triggering replies.

## Best Possible Solution

Define a Discord-owned, default-off silent transcription policy with explicit authorization order, privacy/retention/cost behavior, schema/docs, and focused `requireMention` history tests before implementing it.

## Reproduction Assessment

Yes, source-level: an unmentioned Discord guild audio message in a `requireMention` channel records placeholder-based pending history and returns before admitted-turn transcript context. I did not run live Discord/STT in this read-only review.

## Solution Assessment

No, not as a bare boolean alone; the direction is plausible, but the best fix needs maintainer-approved privacy, retention, cost, inheritance, schema, docs, and tests before adding the config surface.
