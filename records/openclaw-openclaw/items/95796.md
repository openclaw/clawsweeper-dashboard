## Summary

Keep open: current main still rejects `HEARTBEAT_OK` acknowledgements when the assistant content array includes reasoning/thinking blocks, and the related fixes are open PRs rather than merged implementation.

## What This Changes

The issue reports that heartbeat transcript pruning fails when an assistant `HEARTBEAT_OK` acknowledgement also contains reasoning or thinking content blocks.

## Best Possible Solution

Land one focused heartbeat-filter PR that ignores provider-internal reasoning/thinking/redacted_thinking blocks for `HEARTBEAT_OK` detection while preserving tool/media non-text safety, then close this issue.

## Reproduction Assessment

Yes. Source inspection shows a reasoning/thinking content block sets `hasNonTextContent` in `isHeartbeatOkResponse`, which returns false before `HEARTBEAT_OK` stripping runs; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The best fix is a narrow classifier change in `src/auto-reply/heartbeat-filter.ts` plus focused regression tests, not a new config surface or broader heartbeat policy change.
