## Summary

Keep open: current main and latest release still reject `HEARTBEAT_OK` acknowledgements when the assistant message also has reasoning/thinking blocks, and https://github.com/openclaw/openclaw/pull/95798 is now the open candidate fix path.

## What This Changes

Fix heartbeat transcript filtering so model-internal reasoning/thinking blocks do not prevent short `HEARTBEAT_OK` acknowledgement spans from being removed from model request history.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/95798; if that branch is rejected, keep the fix narrowly at the heartbeat ack text-extraction boundary with regression coverage for `thinking`, `reasoning`, and `redacted_thinking` blocks.

## Reproduction Assessment

Yes, at source level. A current-main assistant message containing `{ type: "thinking" }` or `{ type: "reasoning" }` plus `{ type: "text", text: "HEARTBEAT_OK" }` makes `resolveMessageText` set `hasNonTextContent`, causing `isHeartbeatOkResponse` and then the span filter to miss the heartbeat ack.

## Solution Assessment

Yes. The best fix is the narrow candidate in the heartbeat filter: treat model-internal reasoning blocks as invisible for ack detection while preserving existing guards for visible non-text content and tool calls.
