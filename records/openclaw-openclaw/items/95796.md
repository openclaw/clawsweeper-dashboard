## Summary

Keep open: current main and the latest release still reject heartbeat acknowledgements when the assistant content includes a reasoning/thinking block, and the linked fix work is still in open PRs rather than merged code.

## What This Changes

The issue reports that heartbeat transcript pruning fails when a short `HEARTBEAT_OK` assistant acknowledgement also contains reasoning or thinking content blocks.

## Best Possible Solution

Land one focused heartbeat-filter fix with regression coverage for reasoning, thinking, and redacted-thinking blocks while still rejecting tool calls and meaningful non-text payloads.

## Reproduction Assessment

Yes for source reproduction: current main still returns false for assistant content arrays containing any non-text block before checking whether the text is a short `HEARTBEAT_OK` acknowledgement. I did not run tests because this review is read-only and test commands can create artifacts.

## Solution Assessment

Yes: the best fix is a narrow classifier change in `src/auto-reply/heartbeat-filter.ts` plus focused regression tests, not a new config surface or broader heartbeat policy change. https://github.com/openclaw/openclaw/pull/95798 appears to be one strong existing candidate, but maintainers should pick one canonical PR.
