## Summary

Keep open. Current main and the latest release still enforce the canonical edit shape, `path` plus `edits: [{ oldText, newText }]`, and there is explicit test coverage rejecting legacy alias-style payloads. The linked implementation PRs are closed unmerged, and the later commenter broadened the compatibility request to `oldString`/`newString`, so this needs an explicit maintainer API-compatibility decision rather than cleanup closure.

## What This Changes

The issue requests accepting `old_text` and `new_text` as additional edit-tool parameter aliases for models that emit snake_case tool arguments.

## Best Possible Solution

Keep this open for a deliberate edit-tool compatibility decision. If maintainers want alias tolerance, implement it against the current `edits[]` contract by normalizing accepted aliases into canonical `oldText`/`newText`, exposing the accepted shape in provider schemas, and updating the canonical-only rejection tests; otherwise close manually with the canonical-only rationale.

## Reproduction Assessment

Yes. Current source gives a high-confidence reproduction path: call `edit` without a valid `edits: [{ oldText, newText }]` array, such as the existing `filePath`/`old_text`/`newString` test payload, and validation rejects it before execution.

## Solution Assessment

Unclear. Adding aliases may be the narrowest compatibility fix if maintainers want tolerant tool arguments, but the current code and tests intentionally enforce canonical-only payloads, so the best solution depends on maintainer API policy.
