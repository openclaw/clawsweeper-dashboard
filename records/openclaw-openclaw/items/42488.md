## Summary

Keep open: current main and v2026.5.3 still reject the requested flat alias payloads, the related implementation PRs are closed unmerged, and later maintainer guidance on #64369 points this toward a plugin/adapter seam decision rather than a small core alias patch.

## What This Changes

Accept `old_text` and `new_text` as edit-tool aliases for models that emit snake_case flat edit arguments.

## Best Possible Solution

Maintainers should define the supported compatibility boundary, then either add a pre-validation provider/plugin adapter seam for Claude-Code-style flat edit args or explicitly bless a core normalizer with regression coverage for the current `edits[]` contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call the current edit tool with top-level `old_text`/`new_text` instead of `edits[]`, and the current test contract expects that payload to fail with missing `path, edits`.

## Solution Assessment

Unclear. The issue's proposed alias-list change no longer fits current main by itself because edit validation now centers on `path` plus `edits[]`; the maintainable fix needs a maintainer-owned adapter/plugin seam decision or an explicit core compatibility normalizer.
