## Summary

Keep open: current main has a partial mitigation for durable failure tracking and retry exhaustion, but it still routes visible memory-flush errors into a terminal `run_failed` before the main reply run, and the open follow-up PR owns the remaining recovery behavior.

## What This Changes

The issue requests per-session memoryFlush failure escalation so provider failure during flush cannot leave an over-threshold session wedging future replies.

## Best Possible Solution

Keep this issue open as the canonical tracker until https://github.com/openclaw/openclaw/pull/88968 or an equivalent bounded fix lands and proves failed flush housekeeping no longer blocks normal replies.

## Reproduction Assessment

Yes, source-reproducible: an over-threshold memory flush that produces visible error payloads still causes `runReplyAgent` to fail the reply before the main turn. I did not run a live gateway/provider-crash reproduction in this read-only pass.

## Solution Assessment

No, current main is only a partial mitigation because it records failures and exhausts a compaction cycle while still treating visible flush errors as terminal. The best fix is the linked follow-up shape or equivalent maintainer-approved recovery that lets the real reply proceed and handles exhausted bloated sessions safely.
