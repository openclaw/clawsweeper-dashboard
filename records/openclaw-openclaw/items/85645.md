## Label Justifications

- `P1`: A failed memory-flush turn can leave an agent/channel workflow wedged so subsequent user messages receive no normal response until manual session repair.
- `impact:session-state`: The persisted session transcript can remain over threshold across restarts without recorded failed-flush state or recovery metadata.
- `impact:message-loss`: The user-visible symptom is repeated inbound Slack/channel messages failing to produce a normal assistant reply.

## Summary

Keep open: current main still handles a failed provider-backed memory-flush turn by logging/surfacing the provider error and returning without persistent failure state, lifecycle escalation, or provider-independent transcript recovery. The related configurable-mode request at https://github.com/openclaw/openclaw/issues/30452 is adjacent, but it does not replace this failure-mode escape hatch.

## What This Changes

Request escalation for failed memoryFlush runs so an over-threshold session does not repeatedly wedge when the provider dies mid-flush.

## Best Possible Solution

Add session-scoped memory-flush failure tracking and lifecycle events, then implement a maintainer-approved provider-independent recovery path with explicit default/config semantics for force truncation versus alert-only recovery.

## Reproduction Assessment

Yes, at source level: configure a non-CLI over-threshold session and make runWithModelFallback reject inside runMemoryFlushIfNeeded; the catch path reports/logs the failure and returns without failure metadata or transcript recovery. I did not run a live provider-crash reproduction in this read-only review.

## Solution Assessment

Unclear: failure tracking and lifecycle events are the narrow reliable direction, but automatic provider-independent truncation versus opt-in or alert-only recovery needs maintainer policy because it can drop prior context.
