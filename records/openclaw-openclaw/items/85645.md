## Label Justifications

- `P1`: A broken memory-flush provider can leave an agent/channel workflow wedged across restarts until manual transcript repair.
- `impact:session-state`: The reported failure leaves the persisted session transcript over threshold and lacks session-scoped recovery metadata.
- `impact:message-loss`: Subsequent inbound channel messages can repeatedly enter the failed flush path and fail to produce a normal assistant response.

## Summary

Keep open: current main and the latest release still run memory flush through the provider-backed assistant path, with no persisted failure tracker or provider-independent escalation when that path fails repeatedly.

## What This Changes

The issue requests session-scoped memory-flush failure tracking, lifecycle events, and a provider-independent recovery path for over-threshold sessions when the flush assistant fails.

## Best Possible Solution

Add persisted per-session flush failure metadata and lifecycle observability, then implement a maintainer-approved provider-independent recovery path that clears oversized transcripts without calling the broken session model.

## Reproduction Assessment

Yes at source level: configure a non-CLI over-threshold session and make `runWithModelFallback` reject during `runMemoryFlushIfNeeded`; the catch path does not persist failure state or shrink the oversized transcript. I did not run a live provider-crash reproduction in this read-only review.

## Solution Assessment

Unclear: failure tracking and lifecycle events are straightforward, but automatic provider-independent truncation versus opt-in or alert-only recovery needs maintainer policy because it can drop session context.
