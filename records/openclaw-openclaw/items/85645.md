## Label Justifications

- `impact:session-state`: The reported failure leaves the persisted session transcript and memory-flush state over threshold across restarts.

## Summary

Keep open: current main still has no memory-flush failure tracker or provider-independent escalation path, so an over-threshold session can keep re-entering the same failed flush path when the flush provider remains broken.

## What This Changes

Requests a provider-independent escalation path for repeated memory flush failures so over-threshold sessions do not keep wedging after the flush model crashes.

## Best Possible Solution

Add session-scoped memory-flush failure metadata and lifecycle events, then implement a maintainer-approved provider-independent recovery path that can clear oversized transcripts without calling the broken session model.

## Reproduction Assessment

Yes at source level: configure a non-CLI over-threshold session and make `runWithModelFallback` reject during `runMemoryFlushIfNeeded`; the catch path does not mark a failed attempt or clear the oversized transcript. I did not run a live provider-crash repro in this read-only review.

## Solution Assessment

Unclear: failure tracking and observability are straightforward, but automatic provider-independent truncation versus configurable or alert-only recovery needs maintainer policy before a safe implementation.
