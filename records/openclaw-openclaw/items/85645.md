## Summary

Keep open: current main still catches failed memory-flush runs without recording failure state or clearing an over-threshold transcript, so the reported repeated wedge remains a valid P1 bug. The recovery policy needs maintainer choice because provider-independent truncation can intentionally drop context.

## What This Changes

The issue asks OpenClaw to detect repeated memoryFlush provider failures and escalate provider-independently so oversized session transcripts stop wedging later messages.

## Best Possible Solution

Add session-scoped memory-flush failure metadata and lifecycle events, then implement a maintainer-approved provider-independent recovery path with clear default/config semantics.

## Reproduction Assessment

Yes at source level: configure a non-CLI over-threshold session and make `runWithModelFallback` reject inside `runMemoryFlushIfNeeded`; the catch path logs/surfaces the failure and returns without failure metadata or transcript recovery. I did not run a live provider-crash reproduction in this read-only pass.

## Solution Assessment

Unclear: failure tracking and lifecycle events are the right direction, but automatic force truncation versus opt-in or alert-only recovery needs maintainer policy because it can discard prior context.
