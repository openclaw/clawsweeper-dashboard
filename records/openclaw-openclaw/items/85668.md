## Summary

Keep open: current main and the latest release still synthesize missing tool results as `isError: true`; OpenAI Responses now intentionally matches upstream Codex `aborted` normalization, while Anthropic/Gemini-class replay still uses synthetic error results for strict provider shape. The remaining question is a maintainer-owned replay contract decision, not a stale cleanup close.

## What This Changes

The issue asks OpenClaw to stop treating replayed missing tool results as failed when the persisted result is actually unknown or absent.

## Best Possible Solution

Define the missing-result replay contract per provider family, then implement the chosen behavior with focused replay tests, docs updates, and a diagnostic path for persisted gaps if maintainers want that visibility.

## Reproduction Assessment

Yes, at source level: an assistant tool call with no matching result still flows through `repairToolUseResultPairing` and produces a synthetic `toolResult` with `isError: true`. I did not run a live crash or abort persistence repro in this read-only review.

## Solution Assessment

Unclear: changing the placeholder to `isError: false` addresses the false-failure concern, but current docs, tests, strict-provider repair, and upstream Codex source show this needs an explicit provider-scoped contract decision.
