## Label Justifications

- `P2`: The issue affects replayed tool-result semantics and can cause wrong follow-up reasoning, but it is scoped to incomplete or corrupted session history rather than a universal outage.
- `impact:session-state`: The reported bug is reconstructed transcript state drifting from the actual tool execution state.

## Summary

Keep open: current main and the latest release still synthesize missing tool results as error results, while the closest open PRs address adjacent duplicate/result-loss cases rather than the unknown-vs-failed replay semantics reported here.

## What This Changes

The issue reports that replay backfills for missing tool results mark an unknown persistence gap as a tool failure.

## Best Possible Solution

Define a clear replay contract for genuinely missing persisted tool results, then update the shared repair helper and strict-provider tests so the placeholder preserves turn shape while telling the model the prior tool outcome is unknown rather than a confirmed failure.

## Reproduction Assessment

Yes for the core replay behavior, with medium confidence: a stored assistant tool call followed by a later turn and no matching `toolResult` reaches `repairToolUseResultPairing`, which calls `makeMissingToolResult` and sets `isError: true`. I did not reproduce the crash/abort persistence gap live in this read-only review.

## Solution Assessment

Unclear: flipping the boolean and text is plausible, but current tests and docs intentionally model these placeholders as synthetic repair output. Maintainers should first decide the unknown-result contract, then apply the narrow helper/test update.
