## Summary

Keep open: current main and latest release still synthesize missing strict-provider tool results as error results, and the remaining question is a maintainer-owned replay contract decision rather than already-implemented cleanup.

## What This Changes

The issue asks OpenClaw to treat missing persisted tool results as ambiguous instead of synthesizing definite failed tool results during replay.

## Best Possible Solution

Define a maintainer-approved missing-result replay contract that preserves strict provider adjacency; if unknown outcomes should be non-error, implement it provider-scoped with focused replay tests, docs updates, and live provider proof.

## Reproduction Assessment

Yes, at source level: current main still routes an assistant tool call with no matching result through `repairToolUseResultPairing`, which creates a synthetic `toolResult` with `isError: true`. I did not run a live crash or abort persistence repro in this read-only review.

## Solution Assessment

Unclear: flipping the placeholder to `isError: false` addresses the false-failure concern, but current tests, docs, strict-provider repair, and upstream Codex behavior show this needs an explicit provider-scoped contract decision first.
