- `P2`: This is a normal-priority agent transcript replay correctness issue with limited blast radius but real session-context impact.
- `impact:data-loss`: The issue is triggered by a tool result that was never persisted, leaving model-visible session history without the real tool output.
- `impact:session-state`: The reported behavior changes replayed session state by filling a missing tool result with synthetic failure context.

## Summary

Keep open: current main partially addresses the abort/error replay path and aligns OpenAI Responses-family missing outputs with upstream Codex, but it still intentionally synthesizes missing strict-provider tool results as error results, so the central ambiguous-versus-failed behavior is not implemented and needs a maintainer contract decision.

## What This Changes

The issue asks OpenClaw to represent missing persisted tool results as ambiguous instead of failed, with an optional doctor warning for affected sessions.

## Best Possible Solution

Keep the issue open for an owner decision on the missing-result replay contract; any fix should preserve strict-provider adjacency, OpenAI Responses/Codex `aborted` behavior, and add focused tests or diagnostics for genuinely unknown persisted-result gaps.

## Reproduction Assessment

Yes, source-level: a replayable assistant tool call followed by a non-tool result still passes through `repairToolUseResultPairing` and gets a synthetic `toolResult` with `isError: true`. I did not run a live crash/abort persistence repro in this read-only review.

## Solution Assessment

Unclear: the requested `isError: false` placeholder directly addresses the false-failure concern, but current code and upstream Codex establish provider-specific synthetic repair contracts that need maintainer approval and live proof before changing.
