## Summary

Keep open: current main and the latest release still synthesize missing strict-provider tool results as `isError: true`; OpenAI Responses now uses Codex-style `aborted` and abort/error turns are partly handled, but the requested ambiguous/non-error replay contract is not implemented and needs maintainer direction.

## What This Changes

The issue asks OpenClaw to stop marking synthesized missing persisted tool results as failed during replay, with an optional diagnostic for sessions where results were never recorded.

## Best Possible Solution

Define an owner-approved missing-result replay contract that preserves strict provider adjacency; if unknown states should be non-error, implement it provider-scoped with focused replay tests, live provider proof, and a doctor/session diagnostic for persisted gaps.

## Reproduction Assessment

Yes, source-level: a replayable assistant tool call with no matching result still flows through `repairToolUseResultPairing` and produces a synthetic `toolResult` with `isError: true`. I did not run a live crash/abort persistence repro in this read-only review.

## Solution Assessment

Unclear: changing the placeholder to `isError: false` addresses the false-failure concern, but current tests, docs, strict-provider repair, and upstream Codex behavior show this needs an explicit provider-scoped contract decision.
