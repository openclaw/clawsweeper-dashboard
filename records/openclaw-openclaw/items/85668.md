- `impact:data-loss`: The report starts from a tool result that was never persisted, leaving replay without the original observed output.
- `impact:session-state`: The remaining behavior changes replayed transcript state by adding a synthetic failed result for a missing persisted result.

## Summary

Keep open: current main and latest release still synthesize missing strict-provider tool results as `isError: true`, while OpenAI Responses/Codex uses upstream-style `aborted`; the remaining question is a maintainer replay-contract decision, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to stop replaying missing persisted tool results as failed tool results and instead represent the missing outcome as ambiguous or otherwise repairable.

## Best Possible Solution

Define a maintainer-approved missing-result replay contract that preserves strict-provider adjacency, then update the shared repair helper, transport tests, live replay proof, and transcript hygiene docs to match.

## Reproduction Assessment

Yes, at source level: a strict-provider assistant tool call with no matching stored tool result still reaches `repairToolUseResultPairing` and receives a synthetic `toolResult` with `isError: true`. I did not run a crash/abort persistence repro in this read-only review.

## Solution Assessment

Unclear: the proposed blanket `isError: false` change addresses the false-failure concern, but current tests, docs, and upstream Codex behavior show the durable fix needs an explicit provider-scoped replay contract.
