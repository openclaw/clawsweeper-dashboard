## Summary

Keep open: current main and the latest release still synthesize missing tool results as `isError: true`, and the remaining question is a maintainer contract decision rather than already-implemented behavior.

## What This Changes

The issue asks OpenClaw to stop replaying never-persisted tool results as failed results and instead represent the missing result as ambiguous or diagnostic state.

## Best Possible Solution

Keep this open for an owner decision that preserves strict-provider adjacency and Codex Responses parity while defining how genuinely unknown persisted-result gaps should appear to the model and diagnostics.

## Reproduction Assessment

Yes for the source-level behavior: current main still inserts missing tool results through `makeMissingToolResult`, and that helper sets `isError: true`. I did not establish a live crash or aborted-write repro in this read-only review.

## Solution Assessment

Unclear: the requested non-error placeholder directly addresses the false-failure concern, but current OpenClaw docs/tests and sibling Codex source make this a provider replay contract decision that needs maintainer approval and live proof.
