## Summary

Keep open: current main and latest release v2026.6.9 still synthesize missing strict-provider tool results as `isError: true`; OpenAI Responses now uses Codex-style `aborted`, and related PRs cover durable reordering or compaction triggers, but the requested ambiguous/non-error replay contract remains a maintainer product decision.

## What This Changes

The issue requests changing missing persisted tool-result replay placeholders from failed synthetic results to an ambiguous/non-error status, plus diagnostics for sessions where result persistence failed.

## Best Possible Solution

Define a maintainer-approved missing-result replay contract that preserves strict provider adjacency; if unknown outcomes should be non-error, implement it provider-scoped with focused replay tests, live provider proof, and a doctor/session diagnostic for persisted gaps.

## Reproduction Assessment

Yes, at source level: a replayable assistant tool call with no matching result still flows through `repairToolUseResultPairing` and produces a synthetic `toolResult` with `isError: true`. I did not run a live crash/abort persistence repro in this read-only review.

## Solution Assessment

Unclear: changing the placeholder to `isError: false` addresses the false-failure concern, but current tests, docs, strict-provider repair, and upstream Codex behavior show this needs an explicit provider-scoped contract decision.
