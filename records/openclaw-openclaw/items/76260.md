## Summary

Keep this open: current main still has `web_fetch` on the shared SSRF guard, while gateway host `exec` reaches approval/allowlist handling and then process execution without an equivalent destination guard. The direct fix candidate is closed unmerged, and the related private-network issues do not supersede this security-boundary report.

## What This Changes

The issue reports that `exec` can reach private/internal URLs after `web_fetch` blocks the same SSRF target.

## Best Possible Solution

Keep this open for a maintainer-approved host-exec egress design with regression coverage for `web_fetch` parity or an explicit documented policy distinction.

## Reproduction Assessment

Yes, at source level: `web_fetch` uses the shared SSRF guard while gateway host `exec` proceeds through approval/allowlist handling toward shell execution without a matching destination guard. I did not run a live localhost harness in this read-only review.

## Solution Assessment

Unclear: argv inspection would be only a partial mitigation, while sandbox/network isolation or an explicit host-exec egress policy may be the cleaner security boundary. Maintainers should decide the intended host-exec networking contract before implementation.
