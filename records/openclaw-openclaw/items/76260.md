## Label Justifications

- `P1`: The issue describes a live security-boundary mismatch between guarded web fetch and unrestricted host shell execution in real agent workflows.
- `impact:security`: The affected behavior is SSRF/private-network access enforcement for agent tools.

## Summary

Keep open: current main still enforces the shared SSRF guard for `web_fetch`, while gateway host `exec` proceeds through approval/allowlist handling into shell execution without an equivalent destination guard. The direct fix candidate is closed unmerged, and the related private-LAN diagnostic work is adjacent rather than a replacement.

## What This Changes

The issue reports that host `exec` can use tools such as `curl` to reach loopback/private URLs that `web_fetch` blocks with the SSRF guard.

## Best Possible Solution

Maintainers should choose the host-exec private-network policy explicitly, then implement the chosen boundary at the runtime owner layer with regression coverage for the reported `web_fetch` to `exec` fallback and documented compatibility behavior.

## Reproduction Assessment

Yes. Source inspection shows `web_fetch` blocks private/internal hosts before fetch, while the gateway exec path can still spawn an arbitrary shell command such as `curl` after normal exec policy checks; I did not run the live localhost harness because this read-only review already proves the mismatch.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing argv is a plausible partial mitigation, but repository docs say exec approvals are not hostile isolation and sandboxing/host isolation are the strong boundary, so maintainers need to choose the intended host-exec network contract before implementation.
