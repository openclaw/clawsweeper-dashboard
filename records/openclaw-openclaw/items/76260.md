## Summary

Keep this open: current main still enforces the private/internal host SSRF guard on `web_fetch`, but gateway `exec` proceeds through approval/allowlist handling into host shell execution without an equivalent destination guard. The only linked fix attempt was closed unmerged and reviewed as incomplete, so there is no canonical merged implementation to close against.

## What This Changes

The issue asks OpenClaw to resolve a security-boundary mismatch where `web_fetch` blocks localhost/private targets but allowed `exec` commands can still reach them through tools like `curl`.

## Best Possible Solution

Keep this open for a maintainer-approved security-boundary design that either makes host-exec egress policy explicit or moves the restriction into sandbox/network isolation, with tests for the chosen default and any opt-in escape hatch.

## Reproduction Assessment

Yes from source inspection: `web_fetch` goes through the shared SSRF guard and blocking tests, while gateway `exec` reaches host shell process execution without a matching destination guard. I did not run a live localhost harness because this is a read-only cleanup review and source proof is enough to keep the security issue open.

## Solution Assessment

Unclear: argv inspection would be a plausible partial mitigation, but sandbox/network isolation or an explicit policy error may be the cleaner security boundary. Maintainers should choose the intended host-exec networking contract before implementation.
