## Summary

Keep open. The report describes a real security boundary mismatch: current main blocks localhost/private targets in `web_fetch`, while host `exec` can still spawn arbitrary shell network clients without the same destination policy.

## What This Changes

The issue asks OpenClaw to address `exec` being able to reach localhost/private URLs after `web_fetch` rejects those same URLs with the SSRF guard.

## Best Possible Solution

Keep this open for a security-owner design decision: either give exec a real egress boundary such as sandbox/network isolation, or document and configure clearly that `web_fetch` SSRF protection is not a boundary when host exec is allowed.

## Reproduction Assessment

Yes. The issue provides a concrete localhost `http.server` transcript, and current source confirms `web_fetch` uses the SSRF guard while host exec spawns shell commands without an equivalent destination filter.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing exec argv would reduce common `curl` fallbacks but is brittle; a durable fix likely needs OS/network sandboxing or an explicit product decision that enabled host exec remains full-power.
