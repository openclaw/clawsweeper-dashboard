## Summary

Keep open. Current main still has the reported security boundary mismatch: `web_fetch` uses the shared SSRF guard, while gateway host exec reaches shell execution without an equivalent destination policy; the related fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to prevent or explicitly account for host exec fetching localhost/private URLs after `web_fetch` blocks the same destinations.

## Best Possible Solution

Define the host-exec egress contract explicitly: either enforce private/loopback blocking at a real execution/network boundary, or document and configure host exec as full-power and outside the `web_fetch` SSRF policy.

## Reproduction Assessment

Yes by source inspection. The issue gives a concrete localhost transcript, and current main confirms `web_fetch` uses the SSRF guard while allowed gateway host exec reaches `runExecProcess` without an equivalent destination filter.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing exec commands is a partial mitigation, but a sandbox/proxy/netns egress boundary or explicit full-power exec policy is the more maintainable security decision.
