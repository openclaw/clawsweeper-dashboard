## Summary

Keep open. Current main still has a security boundary mismatch: `web_fetch` routes requests through the SSRF guard, while host `exec` can spawn arbitrary shell network clients without an equivalent destination policy.

## What This Changes

The issue asks OpenClaw to address or explicitly define the boundary where `exec` can reach localhost/private URLs that `web_fetch` blocks.

## Best Possible Solution

Keep this open for a security-owner decision: either enforce a real egress boundary for host exec or document/configure clearly that `web_fetch` SSRF protection is not a shared boundary when host exec is enabled.

## Reproduction Assessment

Yes. The issue includes a concrete localhost `http.server` transcript, and current main source confirms `web_fetch` uses the SSRF guard while host exec reaches the shell/process path without an equivalent destination filter.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing exec commands is a partial mitigation, while the more maintainable solution is likely OS/network isolation or an explicit product decision that allowed host exec remains full-power.
