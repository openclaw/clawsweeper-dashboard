- `P1`: The issue reports a concrete SSRF-style boundary bypass through enabled host exec that can affect real agent workflows.
- `impact:security`: The affected behavior concerns private/internal network access and the boundary between guarded web fetches and shell execution.

## Summary

Keeping this open: current main still applies the private/internal host SSRF guard to `web_fetch`, but gateway `exec` continues through approval/allowlist handling to host process execution without an equivalent destination guard. The linked attempted fix at https://github.com/openclaw/openclaw/pull/76389 was closed unmerged and is not a safe canonical replacement.

## What This Changes

Request that `exec` enforce the same private/internal hostname and IP SSRF boundary that `web_fetch` enforces, or make the intended boundary explicit.

## Best Possible Solution

Define and land a maintainer-approved exec egress boundary, preferably sandbox/proxy/network isolation for host exec, or explicitly document exec as outside the web_fetch SSRF boundary if that is the intended contract.

## Reproduction Assessment

Yes at source level: `web_fetch` goes through the SSRF-guarded fetch path, while gateway `exec` proceeds to host command execution without an equivalent destination filter. I did not run the live localhost probe in this read-only security review.

## Solution Assessment

Unclear: the current issue describes the real gap, but the best fix needs maintainer/security judgment because a parser-only network-tool block is partial and prior related work showed correctness and bypass concerns.
