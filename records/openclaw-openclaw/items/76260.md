## Label Justifications

- `P1`: The issue describes a real agent workflow where a guarded SSRF denial can be bypassed by host shell execution when `exec` is enabled.
- `impact:security`: The affected behavior is the SSRF/private-network access boundary between guarded `web_fetch` and host `exec`.

## Summary

Keep open: current `main` still has the reported security-boundary mismatch. `web_fetch` enforces the shared SSRF guard on private/internal destinations, while gateway-host `exec` can proceed through normal exec policy to host shell execution without the same destination guard; the only direct fix PR is closed unmerged, and the remaining choice is a maintainer security/product decision.

## What This Changes

The issue asks OpenClaw to decide and close the gap where `exec` can reach localhost/private-network URLs that `web_fetch` blocks with SSRF protection.

## Best Possible Solution

Maintainers should define the intended host-exec private-network contract, then implement the boundary in the exec/runtime owner layer with regression coverage for `web_fetch` to `exec` fallback and clear security/upgrade documentation.

## Reproduction Assessment

Yes, at source level. Current `web_fetch` blocks private/internal targets before fetch, while gateway-host `exec` can still spawn an arbitrary shell command such as `curl` after normal exec policy checks; I did not run the live localhost harness in this read-only review.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing argv is a plausible partial mitigation, but the maintainable fix needs a maintainer-approved host-exec egress policy or sandbox/isolation boundary with documented compatibility behavior.
