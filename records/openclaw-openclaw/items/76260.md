## Summary

Keep open. Current main still has a real security boundary mismatch: `web_fetch` applies the SSRF guard to localhost/private targets, while allowed host `exec` can spawn arbitrary shell network clients without the same destination policy.

## What This Changes

The issue reports that when `web_fetch` blocks localhost/private URLs, the model can fall back to `exec` and curl the same target successfully.

## Best Possible Solution

Keep this open for a security-owner decision: either add a real egress boundary for host exec or document/configure clearly that `web_fetch` SSRF protection is not a shared boundary when host exec is enabled.

## Reproduction Assessment

Yes. The issue gives a concrete localhost `http.server` transcript, and current main source confirms `web_fetch` uses the SSRF guard while host exec reaches the shell/process path without an equivalent destination filter.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing exec commands is a partial mitigation, but the maintainable fix is probably OS/network isolation or an explicit product decision that allowed host exec remains full-power.
