## Label Justifications

- `P1`: The issue describes a live security-boundary mismatch between guarded web fetch and unrestricted host shell execution in real agent workflows.
- `impact:security`: The affected behavior is SSRF/private-network access enforcement for agent tools and host execution.

## Summary

Keep open: current main and v2026.6.10 still enforce the shared SSRF guard for `web_fetch`, while gateway-host `exec` reaches shell execution after normal exec approval policy without an equivalent destination guard. The direct fix PR is closed unmerged, and the remaining question is a security/product decision about the intended host-exec network boundary.

## What This Changes

The issue reports that after `web_fetch` blocks private/internal URLs, the model can fall back to `exec` with tools such as `curl` and reach the same destination.

## Best Possible Solution

Maintainers should choose the intended host-exec private-network contract, then implement that boundary in the runtime owner layer with regression coverage for the `web_fetch` to `exec` fallback and clear upgrade/security documentation.

## Reproduction Assessment

Yes, at source level. Current `web_fetch` blocks private/internal targets before fetch, while gateway-host `exec` can still spawn an arbitrary shell command such as `curl` after normal exec policy checks; I did not run the live localhost harness in this read-only review.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing argv is a plausible partial mitigation, but the maintainable fix needs a maintainer-approved host-exec egress policy or sandbox/isolation boundary with documented compatibility behavior.
