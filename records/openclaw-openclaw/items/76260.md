## Summary

Keep open. Current main still has the reported security boundary mismatch: `web_fetch` applies the shared SSRF guard to localhost/private targets, while allowed host `exec` reaches the shell/process path without an equivalent destination policy.

## What This Changes

The issue requests preventing host `exec` from bypassing `web_fetch` SSRF/private-network blocking when models fall back to shell network clients.

## Best Possible Solution

Keep this open for a security-owner decision and implement a real host-exec egress boundary, or clearly document/configure that allowed host exec is full-power and not covered by `web_fetch` SSRF policy.

## Reproduction Assessment

Yes. The issue gives a concrete localhost `http.server` transcript, and current main source confirms `web_fetch` uses the SSRF guard while host exec reaches the shell/process path without an equivalent destination filter.

## Solution Assessment

Unclear. Mirroring the blocklist by parsing exec commands would reduce the obvious fallback but is brittle; an OS/proxy/sandbox egress boundary or explicit full-power exec policy is the more maintainable decision point.
