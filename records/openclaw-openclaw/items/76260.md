## Summary

Keep open. Current main still shows the reported security boundary mismatch: `web_fetch` uses the SSRF guard for localhost/private targets, while host `exec` proceeds through exec policy and then spawns arbitrary shell commands without a comparable destination policy.

## What This Changes

The issue asks OpenClaw to address or explicitly define the behavior where allowed host `exec` can reach localhost/private URLs after `web_fetch` blocks them.

## Best Possible Solution

Keep this open for a security-owner decision: either enforce a real egress boundary for host exec, preferably at the sandbox/network layer, or make the product/docs explicit that `web_fetch` SSRF protection is not a shared boundary when host exec is enabled.

## Reproduction Assessment

Yes. The issue provides a concrete localhost `http.server` transcript, and current main source confirms the two paths diverge: `web_fetch` blocks via the SSRF guard while host `exec` reaches shell/process execution without destination classification.

## Solution Assessment

Unclear. Parsing exec commands for URLs is a partial mitigation, but the maintainable fix is likely OS/network isolation or an explicit product decision that allowed host exec remains full-power.
