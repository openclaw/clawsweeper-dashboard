## Summary

Keep this PR open. Current main still directly rereads client tool descriptors in the agent runtime path, and the PR’s snapshot approach looks like a valid fix, but it is a protected maintainer draft with overlapping descriptor-hardening drafts that should be reconciled before landing.

## What This Changes

The PR snapshots client tool descriptors once per embedded attempt and reuses those snapshots for reserved-name checks, Tool Search planning, allowlist collection, and client-tool registration, with focused regressions for unreadable and stateful descriptors.

## Best Possible Solution

Land this PR or a reconciled successor as the canonical client-descriptor snapshot fix, while keeping broader Tool Search catalog schema quarantine in a separate owner-approved slice if needed.

## Reproduction Assessment

Yes by source inspection. Current main directly reads client descriptor getters in conflict checks, allowlist collection, Tool Search planning, reserved-name setup, and registration, so a throwing or stateful getter can reproduce the crash/divergence path without needing a broad runtime harness.

## Solution Assessment

Yes. Snapshotting once at the embedded-runner boundary and passing the same snapshots through admission, planning, allowlists, and registration is the narrowest maintainable fix because it removes stateful rereads instead of scattering local guards.
