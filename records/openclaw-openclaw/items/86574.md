## Summary

Keep open: current main still supports global, group/channel, request-scoped, and cron-job delivery controls, but I did not find a supported per-agent or per-account opt-out for automatic source replies. The remaining work is a real gateway delivery-policy feature and needs maintainer choice on the canonical config surface.

## What This Changes

Add an opt-in per-agent or per-account policy that prevents plain assistant final text from being auto-delivered to the source channel, leaving only explicit message-tool sends visible.

## Best Possible Solution

Choose one generic delivery-policy surface, reuse the existing `sourceReplyDeliveryMode: "message_tool_only"` machinery behind an opt-in default-preserving scope, and update schema, docs, doctor expectations, and gateway/cron regression tests for that scope.

## Reproduction Assessment

Yes at source level: current main only resolves visible source replies from request mode, global messages config, group messages config, room-event defaults, and cron job delivery mode. I did not run a live Feishu/DARGO cron flow.

## Solution Assessment

Unclear until maintainers choose the product surface. Reusing the existing message-tool-only delivery primitive looks maintainable, but a Feishu account-only `replyToCurrent` key may be the wrong long-term contract compared with a generic agent/run/channel delivery policy.
