## Summary

Keep open: current main and the latest release have global, group/channel, request-scoped, room-event, and cron delivery controls, but I did not find a supported per-agent or per-account opt-out for automatic assistant final-text source replies. The remaining work is a real gateway delivery-policy feature and needs a maintainer choice on the canonical config surface.

## What This Changes

Request to add an opt-in per-agent or per-channel-account setting that keeps plain assistant text private and only lets explicit `message` tool calls produce channel output.

## Best Possible Solution

Define one generic opt-in delivery-policy surface that reuses the existing `sourceReplyDeliveryMode: "message_tool_only"` machinery, preserves current defaults, and updates schema, docs, doctor expectations, and gateway/cron regression coverage.

## Reproduction Assessment

Yes, at source level: current main only resolves visible source replies from request mode, global messages config, group messages config, room-event defaults, and cron source-delivery plans. I did not run a live Feishu/DARGO cron reproduction.

## Solution Assessment

Unclear until maintainers choose the product surface. Reusing the existing message-tool-only delivery primitive looks maintainable, but a channel-account `replyToCurrent` key may be too narrow compared with a generic agent/run/channel delivery policy.
