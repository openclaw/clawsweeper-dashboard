## Summary

Keep open: current main and v2026.6.9 still expose global/group/request-scoped message-tool-only delivery controls, but no supported per-agent or per-account opt-out for automatic assistant final-text source replies. The source gap is real, but the fix needs a maintainer decision on the canonical config surface before automation should implement it.

## What This Changes

Request adds an opt-in per-agent or per-account delivery policy so plain assistant final text stays private and only explicit message-tool sends reach the source conversation.

## Best Possible Solution

Define one default-preserving generic delivery-policy surface that reuses the existing `sourceReplyDeliveryMode: "message_tool_only"` machinery, then align schema, docs, doctor expectations, and gateway/cron regression coverage.

## Reproduction Assessment

Yes, at source level: current main resolves source replies from request mode, global messages config, group messages config, room-event defaults, and cron source-delivery plans, with no per-agent or per-account opt-out. I did not run a live Feishu/DARGO cron reproduction.

## Solution Assessment

Unclear until maintainers choose the product surface. Reusing the existing message-tool-only delivery primitive looks maintainable, but the proposed channel-account `replyToCurrent` key may be too narrow compared with a generic agent/run/channel delivery policy.
