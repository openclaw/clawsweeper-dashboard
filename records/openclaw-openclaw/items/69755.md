## Summary

Keep open. Current main still builds and delivers a direct `Run:` doctor command in restart sentinel notices without checking whether the receiving heartbeat/outbound surface can complete approval-gated execution, and the adjacent doctor-on-restart PR remains open and does not remove that actionability gap.

## What This Changes

The issue requests context-aware post-restart doctor follow-up messaging so constrained heartbeat or approval-limited surfaces present `openclaw doctor --non-interactive` as a terminal/approval-capable follow-up rather than an immediately actionable command.

## Best Possible Solution

Make restart sentinel doctor follow-up text or payload context distinguish terminal/approval-capable execution from outbound/heartbeat delivery. Constrained surfaces should frame the command as a recommended terminal or approvals-capable UI follow-up, while preserving the actual doctor command where it is genuinely actionable.

## Reproduction Assessment

Yes. Current main statically reproduces the reported loop: config/update/restart sentinel payloads carry a direct doctor command, the startup delivery path forwards that text through heartbeat/outbound surfaces, and the exec approval path can deny timed-out approval attempts as reported.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust restart sentinel follow-up copy/context handling, not to weaken exec approvals or change doctor semantics; open PR #64684 is adjacent but does not solve the actionability wording.
