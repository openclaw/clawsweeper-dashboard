## Label Justifications

- `P1`: The report describes a real multi-account channel setup workflow that can overwrite an existing account's credential config.
- `impact:data-loss`: The issue is about existing `openclaw.json` account credentials being overwritten during setup.
- `impact:auth-provider`: The affected data is channel authentication/account credential material used to route and authenticate a plugin account.

## Summary

Keep open: current main and the latest shipped tag still have the setup write-path gap behind the reported credential overwrite, and https://github.com/openclaw/openclaw/pull/79574 is the active unmerged fix path. The reporter's later WeCom plugin analysis matches the published plugin source: its setup callbacks ignore `accountId`, so OpenClaw still needs the shared setup promotion/scoping fix before this issue can close.

## What This Changes

Prevent the channel setup wizard from reusing or overwriting existing credentials when adding a new non-default account for multi-account plugins such as WeCom.

## Best Possible Solution

Rebase, review, and land https://github.com/openclaw/openclaw/pull/79574 or an equivalent core fix that promotes external account credentials before non-default setup writes and scopes WeCom callbacks to the selected account, with focused regressions.

## Reproduction Assessment

Yes, source-reproducible: current main writes selected-account setup patches without first applying contract-backed account credential promotion, while the published WeCom setup callbacks ignore `accountId` and default their writes. I did not run the live wizard because this review is read-only.

## Solution Assessment

Yes, the active PR's direction is the best narrow fix because it repairs the shared setup write path and adds WeCom compatibility scoping instead of adding plugin-specific policy to config storage. The default policy-template idea is separate product work, not required to fix this corruption bug.
