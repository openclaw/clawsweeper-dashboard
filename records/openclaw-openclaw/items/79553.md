## Summary

Keep open: current main still shows the shared setup-helper gap for external account-schema credential promotion, and a live open PR with closing syntax is already tracking the fix. The issue should stay open until that PR lands or maintainers reject that approach.

## What This Changes

The issue reports that adding a non-default account in multi-account channel plugins can reuse or overwrite existing credentials instead of writing the selected new account.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/79574; the desired end state is a shared setup path that reuses contract-backed account promotion before non-default credential writes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No high-confidence live reproduction was established here, but the source path is reproducible: current main documents promotion before non-default writes while the shared setup adapter does not perform contract-backed promotion before patching.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable direction is to fix the shared setup-helper path and cover it with external-plugin-style regression tests; closing this issue before the linked PR lands would be premature.
