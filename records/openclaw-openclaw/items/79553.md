## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79553 by making channel setup account writes honor the documented account-promotion contract for external multi-account plugins. Focus on shared setup helpers under `src/channels/plugins/`: when the wizard/setup adapter writes credentials for a newly selected non-default account, top-level single-account credential values must be promoted into the correct existing/default account without overwriting an unrelated existing account, and plugin-provided `singleAccountKeysToMove`, `namedAccountPromotionKeys`, and `resolveSingleAccountPromotionTarget` must be respected. Do not add WeCom-specific channel ids or owner-specific policy to core; use a synthetic external plugin/account schema in tests, with a credential key like `botId`, an existing `accounts.main`, and a new account id. Add or update regression tests proving the old account is preserved and the new credentials land under the selected account. Update user-facing changelog if behavior changes.
