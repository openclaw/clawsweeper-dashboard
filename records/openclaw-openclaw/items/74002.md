## Summary

Keep open. Current main still has the reported split-state behavior, and this PR is the active focused implementation candidate, but the remaining merge decision needs maintainer security/design review because it changes where host exec approval policy and the default approval socket live.

## What This Changes

The PR changes exec approval file, socket, and effective-policy host-source resolution to honor `OPENCLAW_STATE_DIR`, updates exec-approval docs/changelog, and adds focused regression tests.

## Best Possible Solution

Land this PR or a maintainer-approved replacement that derives the approvals JSON, default socket, effective-policy host-source text, and user-facing docs from the active state-dir contract while explicitly documenting or hardening legacy-policy migration and socket-directory safety semantics.

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, lockfile, or publishing risk, but it moves host exec approval policy and the default approval socket under an env-selected state root and needs maintainer security review.

Concerns:

- **[medium] Review fresh state-dir approval policy semantics:** `src/infra/exec-approvals.ts:189`
  - body: When `OPENCLAW_STATE_DIR` points at a new directory, the PR would create a new approvals file instead of using or warning about an existing stricter `~/.openclaw/exec-approvals.json`. That may be intended state-dir isolation, but it should be explicit because the file controls host exec approvals.
  - confidence: 0.72
- **[medium] Review socket safety for overridden state roots:** `src/infra/exec-approvals.ts:212`
  - body: The default exec-approvals socket would move under `OPENCLAW_STATE_DIR` without an added ownership or mode check for shared/writable directories. If operators point the state root at an unsafe location, a local process may be able to pre-bind or spoof the socket.
  - confidence: 0.64
