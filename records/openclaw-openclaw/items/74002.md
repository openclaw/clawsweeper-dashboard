## Summary

Keep open. Current main still hardcodes exec approvals storage and socket defaults under `~/.openclaw`, while this PR is the active implementation candidate for a real state-dir bug. The remaining blocker is maintainer/security acceptance of moving host exec approval policy and the default approval socket under an env-selected state root, plus normal PR mergeability handling.

## What This Changes

The PR makes exec approval JSON, default socket, and effective-policy host-source resolution honor `OPENCLAW_STATE_DIR`, updates docs/changelog text, and adds focused regression tests.

## Best Possible Solution

Adopt a state-dir-aware exec approvals resolver aligned with the existing state-dir contract, then explicitly document or harden legacy approval-file migration and socket-directory safety so relocated installs avoid split state without weakening host exec approval expectations.

Yes. A high-confidence source-level reproduction is to set `OPENCLAW_STATE_DIR` and inspect current main: `resolveExecApprovalsPath()` and `resolveExecApprovalsSocketPath()` still expand `~/.openclaw` constants, and persistence paths call those helpers.

## Solution Assessment

Unclear as merge-ready. The resolver direction is narrow and matches the bug, but the best final fix needs maintainer/security approval of state-root isolation, legacy policy behavior, and socket safety because this store gates host exec approvals.

Summary: The diff has no dependency, workflow, lockfile, or publishing risk, but it changes host exec approval policy and socket placement under an env-selected state root.

- **[medium] Review fresh state-dir approval policy semantics:** `src/infra/exec-approvals.ts:189`
  - body: When `OPENCLAW_STATE_DIR` points at a new directory, the PR would create/use a new approvals file instead of an existing stricter `~/.openclaw/exec-approvals.json`.
