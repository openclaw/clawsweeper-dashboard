## Summary

Keep open. Current main still ignores OPENCLAW_STATE_DIR for exec approvals file, socket, and default host-source reporting, so this PR is not obsolete; however, the change touches the security-sensitive exec approval state root and needs maintainer security/design review before merge.

## What This Changes

The PR changes exec approval file, socket, and default effective-policy host-source resolution to honor OPENCLAW_STATE_DIR, updates docs/changelog, and adds focused regression tests.

## Best Possible Solution

Land this PR or a maintainer-approved replacement that derives the approvals JSON, default socket, and effective-policy host-source from the active state-dir contract, preserves the unset default, updates all user-facing docs/diagnostics that mention the storage path, and records the accepted security semantics around migration and socket safety.

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, lockfile, or publishing risk, but it moves the exec approval policy file and socket under an operator-controlled state root and needs maintainer security review.

Concerns:

- **[medium] Review fresh state-dir approval policy semantics:** `src/infra/exec-approvals.ts:185`
  - body: When OPENCLAW_STATE_DIR points to a new directory, the PR would create a new approvals file instead of using or warning about an existing stricter ~/.openclaw/exec-approvals.json. That may be intended state-dir behavior, but it should be explicit because these defaults gate host exec.
  - confidence: 0.72
- **[medium] Review socket safety for overridden state roots:** `src/infra/exec-approvals.ts:210`
  - body: The default exec-approvals socket would move under OPENCLAW_STATE_DIR without an added ownership or mode check for shared/writable directories. If operators point the state root at an unsafe location, a local process may be able to pre-bind or spoof the socket.
  - confidence: 0.64
