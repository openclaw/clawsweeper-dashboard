## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep node-role clients out of operator methods:** `src/gateway/role-policy.ts:21-23`
  - body: The new shared allow-list makes methods like `status`, `chat.send`, `config.get`, and `sessions.list` pass role authorization for `role === "node"`. In current `server-methods.ts`, node roles skip operator-scope checks after this test, so node credentials gain operator-style chat/config/session access; existing role-policy coverage expects `node` plus `status` to be denied. Split this into an owner-reviewed gateway auth change or add scoped authorization instead of returning true here.
  - confidence: 0.91
- **[P2] Mirror notts filtering in the shell helper:** `scripts/tts-filter.sh:22-44`
  - body: The helper is added with the TTS tag feature but only strips/unwraps `<tts>`. A message with `<notts>table</notts>` would print raw `<notts>` tags in the text output and send visual-only content to the voice output, contradicting the new tag contract.
  - confidence: 0.88
- **[P2] Trigger tagged TTS for notts-only messages:** `src/tts/tts.ts:872-874`
  - body: In tagged mode this only treats `<tts>` as a tag. A reply that hides visual content with `<notts>` and leaves speakable text outside has no directive and returns before synthesis, so the complementary tag can silently disable audio.
  - confidence: 0.78
- **[P2] Honor processTtsTags when checking tag triggers:** `src/tts/tts.ts:872-874`
  - body: `preprocessTtsText` preserves literal tag markup when `processTtsTags` is false, but tagged mode still fires on `<tts>`. That configuration can synthesize literal `<tts>...</tts>` text even though the flag says tags should not be processed.
  - confidence: 0.86
