## Summary

Keep open: the useful visual-only TTS idea is not implemented on current main, but this branch is not merge-ready because it targets the old TTS runtime shape, lacks real behavior proof, and includes unrelated security-sensitive gateway authorization changes.

## What This Changes

This PR adds HTML-style `<notts>`/`<tts>` TTS preprocessing, docs, tests, a shell helper, config flags, and several unrelated gateway, macOS, media, and audioAsVoice changes.

## Best Possible Solution

Rebase into the current `extensions/speech-core` TTS runtime as a narrow visual-only TTS feature, remove or split unrelated gateway/media/macOS changes, fix tagged-mode semantics, and add focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the PR's tagged-mode check only matches `<tts>`, its helper script omits `<notts>`, and current main routes TTS through `extensions/speech-core`. No live TTS run was established because the PR lacks real behavior proof and this review is read-only.

## Solution Assessment

No. The requested capability may be useful, but the current branch should be narrowed and ported to speech-core rather than merging stale legacy-runtime edits plus unrelated gateway authorization changes.

Full review comments:

- **[P1] Keep node credentials out of operator methods:** `src/gateway/method-scopes.ts:29-45`
  - body: This new allow-list lets `node` role credentials call methods such as `config.get`, `chat.send`, `chat.history`, and `talk.mode`. Current main restricts node credentials to node-role methods and otherwise requires `operator`, so this unrelated TTS PR would broaden data and write access without a dedicated gateway authorization review.
  - confidence: 0.91
- **[P2] Port preprocessing to the active speech-core runtime:** `src/tts/tts.ts:39`
  - body: Current main no longer runs TTS from this legacy `src/tts/tts.ts` implementation; it is a facade that loads `extensions/speech-core/src/tts.ts`. After rebasing, this preprocessing must move to the speech-core runtime and matching tests, otherwise the new tags will not affect actual TTS output.
  - confidence: 0.88
- **[P2] Include visual-only tags in tagged-mode gating:** `src/tts/tts.ts:872-874`
  - body: `hasTtsTags` only checks for `<tts>`, so a tagged-mode reply that uses only `<notts>...</notts>` to mark visual-only content will return before synthesizing speech even though the remaining text should be spoken.
  - confidence: 0.9
- **[P2] Respect disabled tag processing before triggering audio:** `src/tts/tts.ts:872-874`
  - body: When `processTtsTags` is false, `<tts>` and `<notts>` should be treated as literal text. This condition still treats `<tts>` as a tagged-mode trigger, so it can generate audio and speak literal tag markup despite the opt-out.
  - confidence: 0.86
- **[P2] Mirror visual-only filtering in the helper script:** `scripts/tts-filter.sh:24-42`
  - body: The added shell helper claims to mirror the TTS/text split, but it never unwraps `<notts>` for text output or removes `<notts>...</notts>` for voice output. That means raw tags can appear in chat and visual-only content can still be spoken.
  - confidence: 0.89
