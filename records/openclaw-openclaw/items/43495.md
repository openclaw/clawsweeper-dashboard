## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep operator methods out of the node role:** `src/gateway/role-policy.ts:22-23`
  - body: `authorizeGatewayMethod` skips operator scope checks after a node role is accepted, so returning true for these extra methods lets node clients call `chat.send`, `talk.mode`, `config.get`, and similar operator surfaces without read/write scopes. This permission expansion should be separated and designed explicitly, not bundled into the TTS tag PR.
  - confidence: 0.9
- **[P2] Filter `<notts>` in the shell helper:** `scripts/tts-filter.sh:23-40`
  - body: The new helper claims to mirror the TypeScript preprocessing, but `TEXT_CLEAN` leaves `<notts>` tags in the visible output and `VOICE_CLEAN` still speaks visual-only content. Either update both filters for `<notts>` or drop the helper from this PR.
  - confidence: 0.95
- **[P2] Treat `<notts>` as tagged TTS markup:** `src/tts/tts.ts:869-870`
  - body: In `auto: "tagged"`, a reply that uses only `<notts>` plus spoken prose outside it still returns before synthesis because the gate checks only `<tts>`. If `<notts>` is accepted as TTS-aware markup, this check also needs `<notts>` and should respect the tag-processing config.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete Gateway authorization regression by letting node-role clients call selected operator methods without operator scopes.

Concerns:

- **[high] Node role bypasses operator scopes:** `src/gateway/role-policy.ts:23`
  - body: Current request authorization returns early for node callers after `isRoleAuthorizedForMethod` passes. The PR's `NODE_ALSO_ALLOWED` branch would make methods like `chat.send`, `chat.abort`, `talk.mode`, and `config.get` pass that role check, so paired node clients could reach operator read/write surfaces without the normal scope gate.
  - confidence: 0.9
