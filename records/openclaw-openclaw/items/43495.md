## Summary

Keep open: the requested `<notts>` capability is not already implemented on current main, but this branch is not merge-ready because it mixes unrelated runtime/gateway changes, has concrete TTS correctness gaps, broadens gateway node-role authorization, and still lacks real behavior proof.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing, docs, tests, and prompt hints, while also changing gateway node-role permissions, macOS speech hinting, media transcription conversion, MIME mapping, and tool-result `audioAsVoice` propagation.

## Best Possible Solution

Rebase and split this into a narrow speech-core `<notts>` feature with focused tests, docs, and real TTS/chat proof, while moving the gateway node-role expansion to a separate security-reviewed design if it is still wanted.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR diff shows the shell filter and tagged-mode gate do not implement the PR's own `<notts>` semantics. Current main inspection also shows the central feature is not already implemented.

## Solution Assessment

No. The `<notts>` idea is plausible, but this branch is not the best path because it targets stale TTS files, misses companion behavior, includes unrelated runtime changes, and widens gateway authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep node clients out of operator chat methods:** `src/gateway/method-scopes.ts:31-47`
  - body: This allows node-role clients to call config, chat, sessions, talk, and TTS methods that current main keeps behind operator authorization. That changes the gateway trust boundary in a TTS PR and should be split into a separate security-reviewed design with explicit tests.
  - confidence: 0.86
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter only handles `<tts>`. A message with `<notts>visual</notts>` would print the raw tags in text output and still include the visual-only content in `VOICE_CLEAN`, contradicting the feature semantics.
  - confidence: 0.92
- **[P2] Trigger tagged TTS for `<notts>`-only replies:** `src/tts/tts.ts:871-872`
  - body: In `auto: "tagged"` mode this checks only for `<tts>`, so a reply that wraps a table in `<notts>` plus plain spoken summary text outside the block still returns without audio.
  - confidence: 0.86
- **[P2] Respect `processTtsTags` before triggering on bare tags:** `src/tts/tts.ts:871-872`
  - body: The bare-tag trigger runs even when `messages.tts.processTtsTags` is false. In that configuration tags are supposed to remain literal text, but tagged mode would still synthesize audio because `<tts>` appears in the payload.
  - confidence: 0.83
