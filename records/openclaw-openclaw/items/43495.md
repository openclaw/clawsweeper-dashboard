## Summary

This PR should stay open because the proposed `<notts>` feature still has source-visible correctness gaps, lacks after-fix real behavior proof, and bundles an unrelated gateway role-policy expansion that needs explicit maintainer/security review.

## What This Changes

The PR adds TTS `<notts>` preprocessing plus TTS prompt/docs/tests, while also changing gateway role authorization, macOS Talk Mode speech hints, media handling, and tool-result audio delivery.

## Best Possible Solution

Port a focused `<notts>` implementation into the current `speech-core` TTS runtime with tests for visible/spoken text and tagged-mode behavior, and split unrelated gateway, media, and macOS changes into separately reviewed PRs.

## Reproduction Assessment

Yes for the review blockers. Source inspection of the PR diff shows the script and tagged-mode gaps, and current main tests/source show the node role is intentionally separated from operator-scoped methods.

## Solution Assessment

No. The useful TTS idea needs a smaller port to the current speech-core boundary, while the gateway role change needs separate maintainer/security design instead of being bundled into this PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep node clients out of operator RPCs:** `src/gateway/method-scopes.ts:28-45`
  - body: Adding these methods to the node role bypasses the operator scope checks entirely because `authorizeGatewayMethod` returns before scope authorization for `role === "node"`. A paired node would gain `config.get`, `chat.send`, `talk.mode`, and other UI read/write RPCs that current policy and tests keep operator-only; this needs a dedicated scoped client design or separate security review.
  - confidence: 0.9
- **[P2] Mirror `<notts>` handling in the script filter:** `scripts/tts-filter.sh:23-44`
  - body: The new shell filter still only removes or unwraps `<tts>` blocks. With this PR's advertised syntax, chat output would show raw `<notts>` tags and the voice text would speak the visual-only block, so the script contradicts the new preprocessing semantics.
  - confidence: 0.92
- **[P2] Trigger tagged TTS for `<notts>`-only messages:** `src/tts/tts.ts:871-874`
  - body: In tagged mode, a reply that uses only `<notts>` to mark a table/code block as visual-only has no `[[tts:...]]` directive and does not match this `<tts>`-only regex, so OpenClaw returns without generating audio even though the remaining text is the intended spoken content.
  - confidence: 0.88
- **[P2] Honor `processTtsTags` when checking tag triggers:** `src/tts/tts.ts:871-874`
  - body: When `messages.tts.processTtsTags` is false, this new regex can still force tagged-mode audio for literal `<tts>` markup. The tagged-mode trigger should be gated by the same resolved tag-processing setting used by preprocessing.
  - confidence: 0.86
