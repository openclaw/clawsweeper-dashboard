## What This Changes

The PR adds a `<notts>` visual-only TTS tag with preprocessing docs/tests, while also changing gateway role policy, macOS speech capture, media transcription, MIME mapping, and TTS audioAsVoice plumbing.

## Best Possible Solution

Rebase and split this into a focused speech-core TTS feature that implements `<notts>` in the current runtime, fixes the tagged-mode/script gaps or drops the script, documents the contract, and leaves unrelated gateway/media/macOS changes to separate reviewed PRs.

Yes, source-reproducible. The PR branch shows concrete paths where `<notts>` is not honored by the added shell filter and where tagged auto-TTS only detects `<tts>`, while current main has no `<notts>` implementation to compare against.

  - body: This PR grants node-role callers access to methods such as `config.get`, `chat.send`, `chat.history`, and `sessions.preview`. Current main restricts nodes to node-specific methods, so a TTS tag PR should not broaden gateway authority without a scoped authorization design and owner review.

  - confidence: 0.93
- **[P2] Trigger tagged TTS for notts-only messages:** `src/tts/tts.ts:871-872`
  - body: Tagged mode only checks for `<tts>`, so a reply that uses `<notts>` around a table plus ordinary summary text will return without synthesizing audio even though the author used the new TTS-aware tag.
  - confidence: 0.88
- **[P2] Honor processTtsTags when detecting TTS tags:** `src/tts/tts.ts:871-872`
  - body: When `messages.tts.processTtsTags` is false, preprocessing preserves literal tags, but this check still treats `<tts>` as a tagged-mode trigger. That configuration can unexpectedly synthesize and speak literal tag markup.
  - confidence: 0.84
