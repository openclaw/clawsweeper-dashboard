## Summary

Keep open: the central `<notts>` feature is not on current main, but this PR is not mergeable as-is because it mixes unrelated runtime/gateway/macOS/media changes, has source-reproducible correctness and security issues, and lacks real behavior proof.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing, docs/tests/prompt hints, a shell TTS filter, gateway node-role method access, macOS dictation hinting, media conversion, MIME mapping, and tool-result `audioAsVoice` propagation.

## Best Possible Solution

Split this into a narrow speech-core implementation of visual-only TTS tags with aligned docs/tests and real Telegram/TTS proof; handle gateway authorization, macOS speech, media conversion, and voice-bubble delivery as separate reviewed changes.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR diff shows the shell filter, tagged-mode gate, and role-policy expansion contradict the PR semantics and current gateway/TTS contracts. I did not run live TTS or Telegram behavior because this was a read-only review and contributor real behavior proof is still missing.

## Solution Assessment

No. The `<notts>` direction may be useful, but this branch mixes unrelated runtime and authorization changes and implements the TTS portion against stale runtime files instead of the current speech-core surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep node role out of operator methods:** `src/gateway/method-scopes.ts:31-47`
  - body: `NODE_ALSO_ALLOWED` lets node-role clients call config, chat, sessions, talk, status, and TTS methods. Gateway request handling skips operator scope checks after role authorization for node clients, so this broadens the trust boundary in a TTS PR and needs a separate security-reviewed design.
  - confidence: 0.88
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter handles only `<tts>`. A message with `<notts>visual</notts>` would print raw tags in text output and still include the visual-only content in `VOICE_CLEAN`, contradicting the feature semantics.
  - confidence: 0.92
- **[P2] Trigger tagged TTS for `<notts>` replies:** `src/tts/tts.ts:871-872`
  - body: In tagged auto mode this checks only for `<tts>`, so a reply that uses `<notts>` for a table plus plain spoken summary text outside the block returns without audio even though the tag is the explicit TTS-aware signal.
  - confidence: 0.86
- **[P2] Respect `processTtsTags` in tagged-mode gating:** `src/tts/tts.ts:871-872`
  - body: The bare-tag trigger runs even when `messages.tts.processTtsTags` is false. In that configuration tags should remain literal text, but this condition can still force TTS generation and speak the literal markup.
  - confidence: 0.83
