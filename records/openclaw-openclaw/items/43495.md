## Summary

Keep open: the `<notts>` feature is not implemented on current main, but this PR should not merge as-is because source-visible defects, a gateway authorization expansion, stale/conflicting TTS surface, and missing real behavior proof remain.

## What This Changes

The PR adds HTML-like `<notts>`/`<tts>` TTS preprocessing, docs/tests/prompt hints, and also changes gateway node authorization, macOS speech recognition, media transcription conversion, MIME mapping, and tool-result voice delivery.

## Best Possible Solution

Split this into a narrow speech-core `<notts>` feature with aligned docs/tests and real Telegram/TTS proof, and handle gateway authorization, macOS speech, media conversion, and voice-bubble delivery as separate reviewed changes.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR diff shows the shell filter, tagged-mode gate, and tag-processing option do not match the PR's own semantics. I did not run live behavior because this was a read-only review.

## Solution Assessment

No. The `<notts>` direction may be useful, but this branch mixes unrelated surfaces, misses companion behavior, targets a stale TTS layout, and widens gateway authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep node role out of operator methods:** `src/gateway/method-scopes.ts:31-47`
  - body: `NODE_ALSO_ALLOWED` lets node-role clients call config, chat, sessions, talk, status, and TTS methods. Gateway request handling skips operator scope checks after role authorization for node clients, so this broadens the trust boundary in a TTS PR and needs a separate security-reviewed design.
  - confidence: 0.88
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter handles only `<tts>`. A message with `<notts>visual</notts>` would print raw tags in text output and still include the visual-only content in `VOICE_CLEAN`, contradicting the feature semantics.
  - confidence: 0.92
- **[P2] Trigger tagged TTS for `<notts>` replies:** `src/tts/tts.ts:871-872`
  - body: In tagged auto mode this checks only for `<tts>`, so a reply that uses `<notts>` for a table plus plain spoken summary text outside the block returns without audio even though the tag is the explicit TTS-aware signal.
  - confidence: 0.86
- **[P2] Respect `processTtsTags` in tagged-mode gating:** `src/tts/tts.ts:871-872`
  - body: The bare-tag trigger runs even when `messages.tts.processTtsTags` is false. In that configuration tags should remain literal text, but this condition can still force TTS generation and speak the literal markup.
  - confidence: 0.83
