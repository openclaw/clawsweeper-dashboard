## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep node and operator RPC roles separate:** `src/gateway/role-policy.ts:22-24`
  - body: This adds a path where role `node` can call operator-like RPCs such as `chat.send`, `config.get`, `talk.mode`, and `tts.providers`. Current docs and main policy keep node-originated methods separate from operator control-plane methods, so this would let a paired node gain broader read/write control-plane access without the normal operator role/scopes. Split this into an owner-approved gateway change with explicit tests, or drop it from the TTS PR.
  - confidence: 0.86
- **[P2] Strip `<notts>` in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The new `tts-filter.sh` mirrors the TTS preprocessing path but only handles `<tts>`. A message like `<notts>table</notts>` would print raw tags in the text output and would still speak the visual-only content in the voice output, contradicting the feature semantics.
  - confidence: 0.93
- **[P2] Count `<notts>` as tagged-mode intent:** `src/tts/tts.ts:869-874`
  - body: In `messages.tts.auto: "tagged"`, a reply that only uses `<notts>` to hide a visual block from speech will return before TTS runs because `hasTtsTags` only checks `<tts>`. That makes the advertised `<notts>` plus spoken-summary workflow silently fail unless an additional directive is present.
  - confidence: 0.91
- **[P2] Respect `processTtsTags` before triggering tagged TTS:** `src/tts/tts.ts:869-874`
  - body: The branch adds `processTtsTags: false` to preserve literal tag markup, but tagged auto mode still treats `<tts>` as a trigger unconditionally. With tag processing disabled, literal `<tts>` text can unexpectedly force audio generation and be spoken as markup.
  - confidence: 0.86
