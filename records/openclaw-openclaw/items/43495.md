## Summary

Keep open: the `<notts>` idea may be useful, but this branch is not mergeable as-is because it targets the stale TTS runtime path, leaves tag-handling gaps, broadens gateway node authorization, and still lacks real behavior proof.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing with docs/tests, while also changing gateway node method authorization, macOS dictation, media conversion, MIME mapping, and `audioAsVoice` propagation.

## Best Possible Solution

Land a narrower implementation in `extensions/speech-core` for `<notts>` semantics with docs, focused tests, and real TTS/Telegram proof, while splitting gateway authorization, media conversion, macOS dictation, and voice-bubble delivery into separate reviewed PRs.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR head and current main shows the stale TTS runtime target, shell-filter mismatch, tagged-mode gaps, and gateway role-policy regression. I did not run live TTS or Telegram behavior because this is a read-only review and contributor real behavior proof is missing.

## Solution Assessment

No. The feature direction may be reasonable, but the maintainable solution is a smaller speech-core change with separated security-sensitive gateway work and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep node role out of operator methods:** `src/gateway/method-scopes.ts:31-47`
  - body: `NODE_ALSO_ALLOWED` includes operator-scoped methods such as `config.schema`, `chat.send`, `talk.mode`, `sessions.list`, and `tts.providers`; because role authorization returns true for this set before operator scope checks, node clients can bypass the current gateway trust boundary.
  - confidence: 0.9
- **[P2] Port preprocessing into speech-core:** `src/tts/tts.ts:854-857`
  - body: Current main's `src/tts/tts.ts` is only a facade over the bundled speech-core runtime, so adding preprocessing here will not affect active TTS after rebase. Move the behavior into `extensions/speech-core/src/tts.ts` and its tests.
  - confidence: 0.88
- **[P2] Mirror `<notts>` in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter handles only `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in text output and still include visual-only content in `VOICE_CLEAN`, contradicting the new tag semantics.
  - confidence: 0.93
- **[P2] Trigger tagged TTS for `<notts>`:** `src/tts/tts.ts:871-872`
  - body: In tagged auto mode this checks only for `<tts>`, so a reply using `<notts>` for visual-only content plus plain spoken summary text outside the block returns without audio even though the tag is the explicit TTS-aware signal.
  - confidence: 0.87
- **[P2] Gate tag-triggered TTS on tag processing:** `src/tts/tts.ts:871-872`
  - body: When `messages.tts.processTtsTags` is false, tags are supposed to remain literal text, but this bare tag check can still force tagged-mode audio and speak markup. Gate the tag trigger on the resolved tag-processing setting.
  - confidence: 0.84
