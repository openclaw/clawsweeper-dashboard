## Summary

Keep open: current main still has directive-only TTS behavior and no `<notts>` support, while this PR is stale, broad, and has correctness plus security blockers before it can merge.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing, prompt/docs/config/tests, and also changes gateway role authorization, macOS speech recognition, media MIME/transcoding, and tool-result voice delivery.

## Best Possible Solution

Split or replace this with a narrow speech-core PR that ports visual-only tag handling to the active runtime, updates tests/docs/changelog/config metadata, and leaves gateway, media, macOS, and delivery changes to separate owner-reviewed PRs.

## Reproduction Assessment

Yes. Source and PR-diff inspection show current main is directive-only, while the PR adds `<notts>` semantics inconsistently across stale TTS files, the shell filter, tagged-mode gating, and unrelated gateway authorization.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest maintainable solution because it is stale against the active speech-core runtime and bundles unrelated security-sensitive gateway behavior.
