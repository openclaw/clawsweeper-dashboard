## Summary

Keep open. Current main still does not implement `<notts>` visual-only TTS handling, and this PR is stale against the plugin-owned speech-core runtime while also carrying concrete functional and security blockers.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, tests, docs, and prompt hints, while also changing tool-result audio propagation, macOS speech recognition hints, media/MIME handling, a shell TTS filter, and gateway node-role method authorization.

## Best Possible Solution

Implement visual-only TTS content as a narrow current-main speech-core change with contract tests and docs, then leave gateway authorization, media conversion, macOS speech recognition, and tool-result audio propagation to separate owner-reviewed PRs or keep the already-shipped equivalents on main.

## Reproduction Assessment

Yes. Current main has no `<notts>` matches and the active `maybeApplyTtsToPayload` path only processes `[[tts]]` directives, so `<notts>visual</notts>` would remain ordinary visible and spoken text; the PR diff separately reproduces the shell-filter and tagged-mode gaps.

## Solution Assessment

No. The requested feature is plausible, but this PR is not the best way to solve it because it targets an obsolete core TTS location and bundles unrelated security-sensitive gateway behavior with the speech change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This branch lets node-role clients call operator UI RPCs such as config, chat, session, status, and TTS methods. Current main keeps node-role methods separate from operator-scoped RPCs, so this unrelated authorization expansion should not ship with a TTS tag feature.
  - confidence: 0.9
- **[P2] Handle `<notts>` in the shell filter:** `scripts/tts-filter.sh:23-40`
  - body: The added shell filter mirrors the TTS preprocessing path but only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in text output and speak visual-only content in voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on enabled TTS tag semantics:** `src/tts/tts.ts:869-870`
  - body: Tagged auto-TTS now fires only for directive-style input and the existing `<tts>` check, so a reply using only `<notts>` plus normal spoken summary text can skip audio. The trigger also needs to honor disabled tag processing so literal tag markup does not force audio unexpectedly.
  - confidence: 0.88
