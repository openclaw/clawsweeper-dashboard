## Summary

Keep open: the `<notts>` capability is not implemented on current main, and the PR still needs rework before it is mergeable because it mixes the TTS feature with unrelated gateway authorization changes, stale runtime files, missing real behavior proof, and known functional gaps.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing with tests and docs, plus unrelated gateway node allowlist, macOS dictation, media conversion, MIME, and `audioAsVoice` propagation changes.

## Best Possible Solution

Land a smaller implementation in `extensions/speech-core` that defines `<notts>` semantics in the active TTS runtime, docs, and tests, and split gateway authorization, media conversion, and voice-bubble delivery into separate reviewed PRs.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR head shows the shell filter, tagged-mode trigger, and node-role authorization change contradict the stated TTS semantics and current gateway role contract. I did not run live TTS or Telegram behavior because this is a read-only review and contributor real behavior proof is still missing.

## Solution Assessment

No. The `<notts>` direction may be useful, but this branch needs to be narrowed and ported to the current speech-core runtime before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep node role out of operator methods:** `src/gateway/method-scopes.ts:31-47`
  - body: `NODE_ALSO_ALLOWED` lets node clients call config, chat, sessions, talk, status, and TTS methods. Because node-role requests return before operator scope checks, this broadens the gateway trust boundary in a TTS PR and needs a separate security-reviewed design.
  - confidence: 0.9
- **[P2] Port preprocessing into speech-core runtime:** `src/tts/tts.ts:854-857`
  - body: Current main's `src/tts/tts.ts` is only a facade over the bundled speech-core runtime. Implementing the split in the old runtime path means the feature will not affect active TTS after rebase unless it is moved into `extensions/speech-core`.
  - confidence: 0.86
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter handles only `<tts>`. A message with `<notts>visual</notts>` would print raw tags in text output and still include visual-only content in `VOICE_CLEAN`, contradicting the feature semantics.
  - confidence: 0.93
- **[P2] Trigger tagged TTS for `<notts>` replies:** `src/tts/tts.ts:871-872`
  - body: In tagged auto mode this checks only for `<tts>`, so a reply that uses `<notts>` for a table plus plain spoken summary text outside the block returns without audio even though the tag is the explicit TTS-aware signal.
  - confidence: 0.87
- **[P2] Gate tag-triggered TTS on tag processing:** `src/tts/tts.ts:871-872`
  - body: When `messages.tts.processTtsTags` is false, tags should remain literal text, but this bare tag check can still force tagged-mode audio and speak the markup. Gate this trigger on the resolved tag-processing setting.
  - confidence: 0.84
