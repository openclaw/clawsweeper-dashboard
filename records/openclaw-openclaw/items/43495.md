## Summary

Keep open, but this PR is not merge-ready: the visual-only TTS tag idea is useful, while the branch still has source-reproducible TTS gaps, missing real behavior proof, and a security-sensitive gateway role expansion unrelated to the stated feature.

## What This Changes

The branch adds visual-only `<notts>` TTS preprocessing plus docs/tests/prompt hints, a shell filter, gateway node-role allowlist changes, macOS dictation hinting, media conversion, MIME mapping, and `audioAsVoice` propagation.

## Best Possible Solution

Land a smaller speech-core change that defines `<notts>` in the active TTS runtime, docs, and tests; split gateway authorization, media conversion, and voice-bubble delivery into separately reviewed PRs.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR head shows the shell filter, tagged-mode gate, and node-role authorization change contradict the stated TTS semantics and current gateway role contract. I did not run live TTS or Telegram behavior because this is a read-only review and contributor real behavior proof is still missing.

## Solution Assessment

No. The `<notts>` feature may be worth implementing, but this branch mixes unrelated runtime and authorization changes and needs to be rebuilt around the current speech-core TTS surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep node role out of operator methods:** `src/gateway/method-scopes.ts:31-47`
  - body: `NODE_ALSO_ALLOWED` lets node clients call config, chat, sessions, talk, status, and TTS methods. Current request authorization returns before operator scope checks for node clients, so this broadens the trust boundary in a TTS PR and needs a separate security-reviewed design.
  - confidence: 0.9
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter handles only `<tts>`. A message with `<notts>visual</notts>` would print raw tags in text output and still include the visual-only content in `VOICE_CLEAN`, contradicting the feature semantics.
  - confidence: 0.93
- **[P2] Trigger tagged TTS for `<notts>` replies:** `src/tts/tts.ts:871-872`
  - body: In tagged auto mode this checks only for `<tts>`, so a reply that uses `<notts>` for a table plus plain spoken summary text outside the block returns without audio even though the tag is the explicit TTS-aware signal.
  - confidence: 0.87
- **[P2] Honor disabled tag processing before firing tagged TTS:** `src/tts/tts.ts:871-872`
  - body: When `messages.tts.processTtsTags` is false, tags should remain literal text, but this bare `<tts>` check can still force tagged-mode audio and speak the markup. Gate this trigger on the resolved tag-processing setting.
  - confidence: 0.84
