## Best Possible Solution

Yes for the review blockers from source inspection: the patch's shell filter never matches `<notts>`, the tagged-mode trigger only tests `<tts>`, and current main has no `<notts>` implementation. I did not run the PR branch because this is a read-only review and the branch is conflicting.

## Solution Assessment

No. The feature direction is plausible, but the implementation should be rebased onto `extensions/speech-core` and narrowed to the TTS contract instead of bundling unrelated role-policy, macOS, and media behavior.

  - body: This grants every node-role connection access to methods such as `config.get`, `chat.send`, `chat.history`, `talk.mode`, and `sessions.preview`. A message with `<notts>table</notts>` would show raw tags in text and speak the visual-only content, contradicting the feature semantics.
  - confidence: 0.91

  - body: The new tagged-mode trigger only checks `<tts>` and ignores `messages.tts.processTtsTags`. In `auto: "tagged"`, a reply using only `<notts>` will skip TTS, while disabled `<tts>` markup can still force audio generation; resolve the processing flag first and include both tag forms in the trigger.
  - confidence: 0.87
