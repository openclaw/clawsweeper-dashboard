## Summary

Keep open. The visual-only TTS feature is still not implemented on current main, but this PR is not mergeable as-is because it is broad, stale against the plugin-owned speech-core path, and bundles an unrelated security-sensitive gateway authorization expansion with unresolved tag-handling bugs.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, tests, docs, and prompt hints, while also changing tool-result audio propagation, macOS speech-recognition hints, media/MIME handling, a shell TTS filter, and gateway node-role authorization.

## Best Possible Solution

Port the visual-only speech/text split as a narrow speech-core/plugin-runtime change with contract tests, docs, and a changelog entry, and leave gateway authorization, media conversion, macOS recognition, and audio delivery work to separate owner-reviewed PRs.

## Reproduction Assessment

Yes. Current main has no `<notts>` implementation and the active tagged-mode path only triggers on `[[tts]]` directives, while the PR diff reproduces the shell-filter and tagged-mode gaps called out in review comments.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the narrowest maintainable solution because it bundles unrelated security-sensitive gateway behavior and does not fully implement the new tag semantics across its own added surfaces.

Full review comments:

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This branch lets node-role clients call operator UI RPCs such as config, chat, session, status, and TTS methods. Current main keeps node-role methods separate from operator-scoped RPCs, so this unrelated authorization expansion should not ship with a TTS tag feature.
  - confidence: 0.9
- **[P2] Handle `<notts>` in the shell filter:** `scripts/tts-filter.sh:23-40`
  - body: The added shell filter mirrors the TTS preprocessing path but only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in text output and speak visual-only content in voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on enabled tag semantics:** `src/tts/tts.ts:869-874`
  - body: Tagged auto-TTS should not be forced by literal tag markup when tag processing is disabled, and `<notts>`-only messages should be handled consistently if the feature is enabled. As written, tagged mode can skip intended voice output or speak literal markup depending on config.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/tts/tts-core.ts:20`
  - body: This is a user-facing TTS feature, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.86
