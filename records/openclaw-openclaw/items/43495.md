## Summary

Keep this PR open, but not mergeable as-is: the central `<notts>` idea is still useful, yet the branch is outdated, broad, missing real behavior proof, and introduces concrete TTS and gateway authorization problems.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing and docs, plus unrelated gateway node-role permissions, macOS speech recognition, media transcription, MIME, and tool-result audio routing changes.

## Best Possible Solution

Rebase and split this into a narrow speech-core `<notts>` implementation with focused tests/docs, drop the side changes already on main, and handle gateway node-role permissions in a separate security-reviewed design.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows the shell filter and tagged-mode gate do not implement the PR's own `<notts>` semantics. I did not run live TTS because this is a read-only review and the PR has no real setup proof.

## Solution Assessment

No. The requested `<notts>` behavior is plausible, but this branch is not the best path because it mixes unrelated gateway/macOS/media changes, misses matching tag handling in companion paths, and widens gateway authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep node clients out of operator chat methods:** `src/gateway/method-scopes.ts:31-47`
  - body: This list makes node-role clients authorized for `config.get`, `chat.history`, `chat.send`, `talk.mode`, and other operator surfaces. Current main keeps node-role RPCs limited to node methods; widening that boundary in a TTS PR would let paired node clients read config/history and start chat or talk actions without operator scopes.
  - confidence: 0.86
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The new shell filter only removes or unwraps `<tts>` tags. A message containing `<notts>visual</notts>` would print the raw tags in the text output and still include the visual-only content in `VOICE_CLEAN`, contradicting the feature semantics implemented in TypeScript.
  - confidence: 0.92
- **[P2] Trigger tagged TTS for `<notts>`-only replies:** `src/tts/tts.ts:871-872`
  - body: In `auto: "tagged"` mode this check still looks only for `<tts>`. A reply that uses `<notts>` around a table plus plain spoken summary text outside the block would return without audio even though the new tag is meant to mark TTS-aware delivery.
  - confidence: 0.86
- **[P2] Respect processTtsTags before triggering on bare tags:** `src/tts/tts.ts:871-872`
  - body: The new bare-tag trigger runs even when `messages.tts.processTtsTags` is explicitly false. In that configuration the tags are preserved as literal text by preprocessing, but tagged mode would still synthesize audio because `<tts>` appears in the payload.
  - confidence: 0.83
