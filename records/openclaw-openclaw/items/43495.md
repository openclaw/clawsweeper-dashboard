Full review comments:

- **[P1] Keep node UI methods behind scoped authorization:** `src/gateway/role-policy.ts:23`
  - body: This unconditional allow path lets any `node` role token call the new UI-method allowlist, including config, chat, sessions, talk, status, and TTS methods. That bypasses the existing operator-only default for those methods, so the mobile/desktop UI need should be handled with explicit scoped authorization or a narrower audited seam rather than granting all node credentials these capabilities.
  - confidence: 0.87
- **[P2] Mirror `<notts>` filtering in the shell script:** `scripts/tts-filter.sh:24-42`
  - body: The added shell helper only strips/unwraps `<tts>` tags. With this PR's new contract, `<notts>...</notts>` will leak raw tags into text output and the visual-only content will still be sent to voice output, so the script contradicts the feature it ships with.
  - confidence: 0.93
- **[P2] Trigger tagged mode for `<notts>` blocks:** `src/tts/tts.ts:869-874`
  - body: Tagged auto-TTS still checks only the existing TTS tag/directive signal, so a reply that uses only `<notts>` to mark visual-only content can be treated as untagged and skip audio even though the remaining text is intended for speech-aware delivery.
  - confidence: 0.8
- **[P2] Honor disabled tag processing in tagged-mode gating:** `src/tts/tts.ts:869-874`
  - body: The PR introduces `messages.tts.processTtsTags`, but tagged-mode detection still fires on tag text even when that option disables tag processing. In that configuration, tags are meant to be literal content, so they should not force audio generation or be spoken unexpectedly.
  - confidence: 0.82
