## Summary

Keep open: current main still lacks the requested visual-only TTS tag support, but this PR is not merge-ready because it mixes the TTS feature with unrelated stale changes and a security-sensitive gateway authorization expansion.

## What This Changes

The PR adds `<notts>`/`<tts>` TTS preprocessing, docs/tests/config, plus gateway role authorization, macOS Talk Mode, media transcription/MIME, and tool-result voice-delivery changes.

## Best Possible Solution

Split or replace this with a narrow speech-core PR that implements agreed visual-only TTS semantics in the current plugin runtime, docs, schema/tests, and changelog while dropping unrelated gateway, macOS, media, and delivery changes.

## Reproduction Assessment

Yes for the review findings. Source and PR-diff inspection show current main is directive-only, while the PR adds `<notts>` semantics inconsistently across the shell filter, tagged-mode gate, docs target, and gateway role policy.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest maintainable solution because it bundles unrelated stale changes and a node-role authorization expansion with the TTS syntax.

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This branch lets node-role clients call operator UI RPCs such as config, chat, session, status, and TTS methods. Current main keeps node-role methods separate from operator-scoped RPCs, so this unrelated authorization expansion should not ship with a TTS tag feature.
  - confidence: 0.9
- **[P2] Apply `<notts>` semantics in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The added shell filter only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in text output and speak visual-only content in voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on enabled tag semantics:** `src/tts/tts.ts:869-874`
  - body: Tagged auto-TTS should follow the same enabled tag semantics as preprocessing: `<notts>`-only messages should trigger when tag processing is enabled, and literal `<tts>` markup should not force audio when tag processing is disabled.
  - confidence: 0.88
- **[P3] Move TTS docs onto the current page:** `docs/tts.md:391`
  - body: Current main publishes the TTS guide from `docs/tools/tts.md` and keeps `docs/tts.md` as a redirect. Adding the new feature docs to the redirect page will not update the canonical docs page after rebase.
  - confidence: 0.82
