## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep operator RPCs behind operator authorization:** `src/gateway/role-policy.ts:22-24`
  - body: This branch lets any paired `node` role call the new `NODE_ALSO_ALLOWED` methods, including `config.get`, `chat.send`, `chat.history`, and `talk.mode`. Current policy keeps non-node RPCs operator-only, so this grants device-paired clients operator data/action access without operator scopes; this needs a narrower authenticated UI/operator path instead.
  - confidence: 0.9
- **[P2] Mirror `<notts>` handling in the shell filter:** `scripts/tts-filter.sh:23-42`
  - body: The new shell helper is presented as the same dual voice/text filter, but it only handles `<tts>`. A `<notts>table</notts>` block would print raw tags in the text output and would still be spoken in the voice output, contradicting the feature semantics.
  - confidence: 0.95
- **[P2] Treat `<notts>` as a tagged-mode trigger:** `src/tts/tts.ts:869`
  - body: In `messages.tts.auto === "tagged"`, a message that only uses `<notts>` still explicitly opts into TTS-aware delivery, but this check only looks for `<tts>`. Those messages return before synthesis, so the visible-only table plus spoken surrounding summary case described by the PR silently produces no audio.
  - confidence: 0.9
- **[P2] Respect `processTtsTags` when gating tagged mode:** `src/tts/tts.ts:869`
  - body: The new `processTtsTags` flag disables tag processing, but the tagged-mode gate still treats literal `<tts>` text as an auto-TTS trigger. With the flag set to false, markup that should remain plain text can unexpectedly force synthesis and be spoken with the raw tags.
  - confidence: 0.86
