## Summary

Keep open: current main still has no `<notts>` TTS surface, and this PR is not merge-ready because it mixes a plausible TTS feature with unrelated gateway authorization changes and stale runtime/docs surfaces.

## What This Changes

The PR adds visual-only `<notts>` TTS preprocessing, prompt/docs/tests/config updates, plus gateway role authorization, macOS speech recognition, media MIME/transcoding, and tool-result voice-delivery changes.

## Best Possible Solution

Split this into a narrow speech-core TTS change with current docs, config/schema, tests, and changelog alignment, and leave gateway, macOS, media, and delivery changes to separate owner-reviewed PRs.

## Reproduction Assessment

Yes. Source and PR-diff inspection show current main is directive-only for tagged TTS, while the PR adds `<notts>` semantics inconsistently across its shell filter, tagged-mode gate, and docs/runtime surfaces.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest maintainable solution because it bundles unrelated security-sensitive gateway behavior and stale adjacent changes with the TTS syntax.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This branch lets node-role clients call operator UI RPCs such as config, chat, session, status, and TTS methods. Current main keeps node-role methods separate from operator-scoped RPCs, so this unrelated authorization expansion should not ship with a TTS tag feature.
  - confidence: 0.9
- **[P2] Apply `<notts>` semantics in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The added shell filter only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in text output and speak visual-only content in voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on enabled tag semantics:** `src/tts/tts.ts:869-874`
  - body: Tagged auto-TTS should follow the same enabled tag semantics as preprocessing: `<notts>`-only messages should trigger when tag processing is enabled, and literal `<tts>` markup should not force audio when `processTtsTags` is disabled.
  - confidence: 0.88
- **[P3] Rebase the docs onto the current TTS page:** `docs/tts.md:391`
  - body: Current main publishes the TTS guide from `docs/tools/tts.md` and keeps `docs/tts.md` as a redirect. Adding the new feature docs to the old page path will not align with the current docs navigation after rebase.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/tts/tts-core.ts:20`
  - body: This is a user-facing TTS feature, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.86
