## What This Changes

## Best Possible Solution

## Solution Assessment

No. The useful tag support belongs in the current `extensions/speech-core` runtime, while the unrelated gateway, macOS, media, and Telegram delivery changes should be split for separate review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep node-role clients out of operator methods:** `src/gateway/role-policy.ts:21-23`
  - body: The new shared allow-list makes node-role sessions pass role authorization for methods like `status`, `chat.send`, `config.get`, and `sessions.list`. `server-methods.ts` returns before operator-scope checks for `role === "node"`, so this grants node credentials operator-style chat/config/session access.
  - confidence: 0.91
- **[P2] Mirror notts filtering in the shell helper:** `scripts/tts-filter.sh:22-44`
  - body: The helper added by this PR only strips/unwraps `<tts>`. A message with `<notts>table</notts>` would print raw `<notts>` tags in text output and send visual-only content to voice output, contradicting the new tag contract.
  - confidence: 0.88
- **[P2] Trigger tagged TTS for notts-only messages:** `src/tts/tts.ts:872-874`
  - body: In tagged mode this only treats `<tts>` as a tag. A reply that hides visual content with `<notts>` and leaves speakable text outside has no directive, so it returns before synthesis even though the author marked the message as TTS-aware.
  - confidence: 0.78
- **[P2] Honor processTtsTags when checking tag triggers:** `src/tts/tts.ts:872-874`
  - body: `preprocessTtsText` preserves literal tag markup when `processTtsTags` is false, but tagged mode still fires on `<tts>`. That can synthesize literal `<tts>...</tts>` text even though the flag says tags should not be processed.
  - confidence: 0.86
