## Summary

Keep open for maintainer handling: current main still lacks the requested visual-only TTS tag support, but this PR is not merge-ready because it has source-confirmed TTS semantic gaps and an unrelated gateway role expansion that current main explicitly rejects.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, tests, docs, and prompt hints while also changing tool-result audio propagation, macOS speech-recognition hints, media/MIME handling, a shell TTS filter, and gateway node-role authorization.

## Best Possible Solution

Split this into a narrow speech-core/plugin-runtime change with contract tests, docs, and a changelog entry, while leaving gateway authorization, media conversion, macOS recognition, and audio delivery changes to separate owner-reviewed PRs.

## Reproduction Assessment

Yes, by source and diff inspection rather than runtime execution. Current main has directive-only tagged TTS and no `<notts>` surface, while the PR diff shows the shell-filter, tagged-mode, and gateway-role gaps.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the narrowest maintainable solution because it bundles unrelated security-sensitive gateway behavior and does not fully apply the new tag semantics across its own added surfaces.
