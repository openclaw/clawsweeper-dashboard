## Summary

Keep open: this PR targets a real data-loss bug, but the guard is in the wrong layer. The reported corrupt-header resume path can rewrite the transcript during `SessionManager.open()` before `prepareSessionManagerForRun()` reaches the new assertion, and the submitted proof is test-only.

## What This Changes

The PR adds an unreadable-header assertion to `prepareSessionManagerForRun` plus a unit test using a manually constructed session manager and temp JSONL file.

## Best Possible Solution

Move the preservation check to the session load/repair boundary so non-empty invalid-header transcripts are never rewritten, cover the real `SessionManager.open` or embedded resume path, and add redacted real behavior proof before merge.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: a non-empty JSONL file with a malformed first line and valid message rows makes `loadEntriesFromFile` return `[]`, after which `setSessionFile` rewrites the file. I did not run tests in this read-only review.

## Solution Assessment

No, this is not the best fix because it guards `prepareSessionManagerForRun` after `SessionManager.open` can already destroy the original transcript. The safer fix belongs in the session loading or pre-load repair boundary.
