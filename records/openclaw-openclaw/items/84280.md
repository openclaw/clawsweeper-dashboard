## Summary

Keep open: the bug is valid, but the PR still has an availability blocker because the new Windows unmanaged branch calls a shared restart helper whose stale cleanup can stop the active listener before the scheduled-task or fallback handoff is proven; the supplied proof covers the managed Scheduled Task path, not the changed unmanaged path.

## What This Changes

The PR adds a Windows SIGUSR1 guard and reroutes unmanaged Windows gateway restart from cross-process SIGUSR1 to the scheduled-task restart helper with focused unit tests.

## Best Possible Solution

Keep the SIGUSR1 guard, but make the Windows unmanaged path preflight or guarantee a safe handoff before any listener cleanup, or fail clearly while leaving the running gateway alive, then add redacted real Windows proof for that no-service-manager path.

## Reproduction Assessment

Yes by source inspection, not live: current main sends SIGUSR1 to a verified unmanaged gateway PID, and Node's Windows signal contract makes that cross-process signal invalid. I did not run a live unmanaged Windows repro in this read-only review.

## Solution Assessment

No. The SIGUSR1 guard is useful, but the win32 unmanaged restart branch reuses a helper that can clean up the active listener before the detached handoff is proven; the safer fix is to preflight or fail without stopping the running gateway.
