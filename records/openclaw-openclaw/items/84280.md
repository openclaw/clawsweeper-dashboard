## Summary

Keep open: the underlying Windows SIGUSR1 bug is real and the guard is useful, but the revised Windows restart path still has an availability blocker because it can terminate the live unmanaged gateway before proving a scheduled-task or fallback relaunch exists, and the PR still lacks real Windows after-fix proof.

## What This Changes

The PR adds a Windows guard for SIGUSR1 process signaling and routes unmanaged Windows gateway restart through the existing scheduled-task restart helper with focused unit tests.

## Best Possible Solution

Keep the SIGUSR1 guard, but make Windows unmanaged restart either prove a supported relaunch handoff before killing the PID or fail clearly without terminating the process, then back it with real Windows restart proof.

## Reproduction Assessment

Yes, by source inspection and Node's process signal contract: current main sends SIGUSR1 to an unmanaged gateway PID, while Windows does not support that signal path. I did not run a live Windows repro in this read-only review.

## Solution Assessment

No. The guard is good, but unconditionally using triggerOpenClawRestart() from the no-service-manager path is not the safest fix because that helper can kill the listener before proving a Windows relaunch mechanism exists.
