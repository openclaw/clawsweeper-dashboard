## Label Justifications

- `P2`: This is a focused CLI restart bug affecting Windows gateway lifecycle behavior, but it is not proven as an urgent broad regression.
- `merge-risk: 🚨 availability`: The PR’s Windows SIGTERM path can terminate the gateway process instead of restarting it, which is an availability failure for the local runtime.

## Summary

Keep open: the Windows SIGUSR1 bug is real, but the updated fix still relies on external Windows SIGTERM reaching the gateway’s JS handler, while Node’s Windows signal contract says that target process is terminated unconditionally; after-fix Windows runtime proof is also missing.

## What This Changes

The PR guards Windows SIGUSR1 in gateway process signaling and changes unmanaged Windows gateway restart to send SIGTERM instead, with focused unit tests.

## Best Possible Solution

Use a Windows-supported targeted restart path, or fail clearly for truly unmanaged Windows gateways when no supported handoff exists, and back it with live Windows proof plus focused tests.

## Reproduction Assessment

Yes by source inspection: current main sends SIGUSR1 to a verified unmanaged gateway PID on Windows, and Node documents Windows signal limitations for SIGUSR1/SIGTERM. I did not run a live Windows repro in this read-only review.

## Solution Assessment

No. The proposed SIGTERM substitution is not the safest fix because external Windows SIGTERM termination bypasses the JS restart-intent handler; use a supported supervisor handoff, IPC/control request, or a clear unsupported-path failure instead.
