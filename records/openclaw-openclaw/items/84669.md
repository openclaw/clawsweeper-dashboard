## Summary

Keep open: the second commit fixes the earlier hard-coded provider-id problem, but the source tagging is still too coarse and can treat an explicitly configured OpenAI `reasoning:false` row as stale runtime metadata when the runtime catalog is already loaded.

## What This Changes

The PR adds a provider-owned opt-in for OpenAI/OpenAI-Codex thinking profiles to override stale runtime catalog `reasoning:false` rows, source-tags thinking catalogs, and adds targeted resolver/directive regression tests.

## Best Possible Solution

Carry source or configured metadata per catalog entry, or prefer the configured/allowed selected row when it contributes `reasoning:false`, so only genuinely stale runtime catalog rows are overridden by the provider profile.

## Reproduction Assessment

Yes. Current main returns an off-only thinking profile before provider profile resolution when the selected catalog row has `reasoning:false`, matching the linked Telegram `/think xhigh` rejection path.

## Solution Assessment

No. The provider-owned opt-in is the right direction, but the current source tagging is catalog-wide and still fails the configured-opt-out case when the runtime catalog is already loaded.
