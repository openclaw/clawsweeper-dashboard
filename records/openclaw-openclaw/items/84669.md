- `merge-risk: 🚨 compatibility`: The patch changes precedence between persisted/configured model metadata and provider thinking profiles, which can break existing configured opt-out setups if source provenance is wrong.

## Summary

Keep open: the revised PR moved the OpenAI exception into a provider-owned opt-in, but the current source marker is still too coarse because a loaded model catalog can include configured rows that should remain explicit opt-outs. That compatibility issue should be fixed before merge.

## What This Changes

The PR lets OpenAI/OpenAI-Codex provider thinking profiles override stale runtime catalog `reasoning:false` rows, preserves configured opt-outs in direct resolver tests, and adds targeted `/think` regression coverage plus small CI fixture/lint cleanups.

## Best Possible Solution

Keep the provider-owned override, but carry configured/runtime provenance per catalog entry so only stale runtime `reasoning:false` rows are overridden and every explicit configured opt-out remains off-only, with regression coverage through the loaded-catalog model-selection path.

## Reproduction Assessment

Yes. Source inspection on current main shows `reasoning:false` returns an off-only thinking profile before provider profile resolution, and `/think` directive validation uses that resolver to produce the unsupported-level message.

## Solution Assessment

No, not yet. The provider-owned opt-in is the right shape, but the current patch tags an entire loaded catalog as runtime and therefore does not fully preserve explicit configured opt-outs.
