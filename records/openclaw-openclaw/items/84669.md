## Summary

Keep open: the latest branch addresses the previous configured-catalog provenance case for the `/think` path, but the source-gated override still depends on catalogs being stamped by `createModelSelectionState`; raw runtime catalog callers can still pass stale `reasoning:false` rows to thinking validation and fall back to off-only.

## What This Changes

The PR adds source-aware thinking catalog metadata and an OpenAI/OpenAI-Codex provider opt-in so stale runtime `reasoning:false` entries can be overridden while explicit configured opt-outs remain off-only, with resolver and directive regressions.

## Best Possible Solution

Use one canonical provider-thinking override contract that covers every runtime validation caller, preserves explicit configured `reasoning:false` opt-outs, and has shared regressions for `/think`, sessions patch/subagent validation, OpenAI/OpenAI-Codex, and non-OpenAI off-only rows.

## Reproduction Assessment

Yes at source level: raw runtime catalog entries with `reasoning:false` still reach thinking validation without the PR's `source:"runtime"` marker, and the linked issue provides live release repros for the same off-only rejection. I did not run a live Telegram or subagent session in this read-only review.

## Solution Assessment

No. The provider-owned override direction is sound, but this implementation makes the new behavior depend on one model-selection stamping path and leaves the permanent plugin API choice unresolved.
