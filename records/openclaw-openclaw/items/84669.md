## Summary

Keep open: the PR has strong proof for the reported `/think` bug, but the current patch should not merge because it can override explicit OpenAI `reasoning:false` opt-outs and moves provider policy into the shared resolver.

## What This Changes

The PR changes `src/auto-reply/thinking.ts` so OpenAI/OpenAI-Codex provider thinking profiles can override `reasoning:false` catalog rows, with resolver and directive regression tests for `gpt-5.5`.

## Best Possible Solution

Preserve explicit catalog opt-outs, distinguish stale runtime metadata from configured false rows, and express the OpenAI exception through a provider-owned or source-aware thinking-profile contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: `reasoning:false` returns off-only before provider profile resolution, and directive validation emits the reported unsupported-level message with that catalog shape.

## Solution Assessment

No. The patch fixes the targeted stale OpenAI case, but it conflates stale runtime catalog rows with explicit configured opt-outs and hard-codes provider IDs in shared resolver code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve explicit OpenAI thinking opt-outs:** `src/auto-reply/thinking.ts:181-182`
  - body: The new override treats every `reasoning:false` row for `openai` or `openai-codex` as stale. `thinkingCatalog` can come from configured or allowed model catalogs, and the catalog contract plus the merged explicit-opt-out fix make `reasoning:false` an authoritative capability declaration, so existing OpenAI rows intentionally disabling reasoning would now accept `/think xhigh`. Please make the override source-aware or provider-owned so explicit opt-outs remain off-only.
  - confidence: 0.9
- **[P2] Keep OpenAI policy out of the shared resolver:** `src/auto-reply/thinking.ts:155-156`
  - body: This helper hard-codes bundled provider IDs in core even though the provider-thinking hook and docs make thinking profiles provider-owned. Put the opt-in on a provider/source-aware contract so future providers do not need shared resolver edits for the same stale-catalog case.
  - confidence: 0.86
