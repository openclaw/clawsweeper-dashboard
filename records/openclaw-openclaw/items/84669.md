## Summary

Keep open: the latest revision fixes the earlier broad override problem, but it still marks every matching configured catalog row as `configured`, so configured OpenAI rows that omit `reasoning` can retain stale runtime `reasoning:false` and still fail the reported `/think` path. The live PR also currently reports a merge conflict against current `main`.

## What This Changes

The PR adds source-aware thinking catalog metadata so OpenAI/OpenAI-Codex provider profiles can override stale runtime `reasoning:false` rows while preserving configured opt-outs and adding resolver/directive regressions.

## Best Possible Solution

Carry provenance for the `reasoning` value itself, not the whole catalog row, so only explicit configured `reasoning:false` opt-outs stay off-only while runtime false rows can be overridden by the owning OpenAI provider profile.

## Reproduction Assessment

Yes at source level. Current main rejects non-off levels when catalog metadata says `reasoning:false`, and PR-head source inspection shows a remaining configured-row path that can still produce that off-only catalog result; I did not run a live Telegram/Pi OAuth repro in this read-only review.

## Solution Assessment

Not yet. The provider-owned override is the right boundary, but the PR needs per-field reasoning provenance instead of marking the entire merged row as configured, plus a rebase over current `main`.
