## Summary

Keep open. Current main still hard-caps `.clobbered.*` snapshots and suppresses repeat cap warnings per config path; the related stale `plugins.deny` PR is explicitly separate from this retention/visibility problem.

## What This Changes

Request rotation or louder health signaling when the config clobber-snapshot cap is reached so later repair events keep an observable forensic trail.

## Best Possible Solution

Maintainers should choose the retention/visibility policy, then update the clobber snapshot helper, recovery tests, and troubleshooting docs so capped hosts either rotate to preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but rotation versus persistent warning/health output is a maintainer retention-policy choice rather than an obvious single bug-fix path.
