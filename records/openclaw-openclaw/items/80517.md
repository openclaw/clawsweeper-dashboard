## Summary

Keep open. Current `main` and the latest release still hard-cap `.clobbered.*` snapshots at 32, return `null` for later events, and suppress repeat cap warnings per config path in a process; the diagnosability gap remains and needs a maintainer retention/visibility decision rather than conservative cleanup.

## What This Changes

The issue asks for `openclaw doctor` clobber snapshot cap behavior to preserve or more visibly report later forensic events instead of silently skipping snapshots after the cap.

## Best Possible Solution

Maintainers should choose the retention and visibility policy, then update the clobber snapshot helper, focused recovery tests, and troubleshooting docs so capped hosts either preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. Create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but rotation versus persistent warning or doctor health output is a maintainer retention-policy decision rather than an obvious conservative cleanup fix.
