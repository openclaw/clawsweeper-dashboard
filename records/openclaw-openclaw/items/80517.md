## Summary

Keep open. Current main still hard-caps `.clobbered.*` snapshots at 32, skips new snapshots once capped, and suppresses repeat cap warnings per config path, so the later-event forensic trail problem remains.

## What This Changes

The issue asks for doctor/config recovery to avoid losing later `.clobbered.*` forensic evidence after the snapshot cap is reached, either by rotation, a louder durable warning, or both.

## Best Possible Solution

Maintainers should choose the retention and visibility policy, then update the clobber snapshot helper, focused recovery tests, and troubleshooting docs so capped hosts either preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. Create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but rotation versus a persistent warning or doctor health signal is a maintainer retention-policy choice rather than a safe automatic bug fix.
