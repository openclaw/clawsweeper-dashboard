## Summary

Keep open. Current main and the latest stable tag still hard-cap `.clobbered.*` snapshots at 32, emit only a process-local first cap warning per config path, and then skip later forensic snapshots, so the diagnosability gap reported from the beta run remains.

## What This Changes

The issue asks doctor/config recovery to avoid silently losing later `.clobbered.*` forensic snapshots after the per-path snapshot cap is reached.

## Best Possible Solution

Maintainers should choose the retention and visibility policy, then update the snapshot helper, focused recovery tests, and troubleshooting docs so capped hosts either rotate to preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. Create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first process-local cap warning is emitted for that path.

## Solution Assessment

Unclear. The issue identifies a real forensic-trail gap, but rotation versus a durable warning or doctor health signal is a maintainer retention-policy choice rather than a safe autonomous bug fix.
