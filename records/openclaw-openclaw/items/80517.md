## Summary

Keep open. Current main still caps `.clobbered.*` snapshots at 32 and suppresses repeat cap warnings per config path; https://github.com/openclaw/openclaw/pull/76483 bounded duplicate snapshot growth but did not solve the later-event forensic loss reported here.

## What This Changes

Requests changing `openclaw doctor` clobber-snapshot cap behavior so capped hosts either preserve later forensic events or surface a durable warning instead of quietly skipping future `.clobbered.*` snapshots.

## Best Possible Solution

Maintainers should choose the retention/visibility policy, then update the bounded snapshot helper, recovery/write tests, and troubleshooting docs so capped hosts either rotate to keep the newest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. With at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but rotation versus persistent warning or health-output surfacing is a maintainer retention-policy decision rather than a single obvious bug fix.
