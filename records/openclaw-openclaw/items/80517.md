## Summary

Keep open. Current main still enforces a hard `.clobbered.*` snapshot cap and suppresses repeat cap warnings per config path, so the reported loss of later forensic snapshots remains unresolved even though the cap is now 32 rather than the 64 seen in the beta report.

## What This Changes

The issue asks for `openclaw doctor` clobber snapshots to stop silently losing later forensic events once the per-config snapshot cap is reached.

## Best Possible Solution

Maintainers should choose the retention and visibility policy, then update the clobber snapshot helper, focused recovery tests, and troubleshooting docs so capped hosts either preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. Create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but choosing rotation, a persistent doctor health warning, or both is a maintainer retention-policy decision rather than a narrow bug-fix default.
