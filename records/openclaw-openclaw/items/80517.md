## Summary

Keep open. Current main still hard-caps `.clobbered.*` snapshots and suppresses repeat cap warnings per config path, so the central forensic-trail problem remains even though the threshold is now 32 rather than the 64 seen in the beta report.

## What This Changes

The issue asks for `openclaw doctor` clobber-snapshot retention to stop silently losing event-specific forensic snapshots after the cap is reached.

## Best Possible Solution

Maintainers should choose the retention and visibility policy, then update the clobber snapshot helper, focused recovery tests, and troubleshooting docs so capped hosts either rotate to preserve the latest incident or surface a durable warning operators cannot miss.

## Reproduction Assessment

Yes by source inspection. Create at least `CONFIG_CLOBBER_SNAPSHOT_LIMIT` `.clobbered.*` siblings for one config path, then the next snapshot attempt returns `null` and only the first cap warning is emitted for that process/path.

## Solution Assessment

Unclear. The issue identifies a real diagnosability gap, but rotation versus a persistent warning or doctor health signal is a maintainer retention-policy choice rather than an obvious single-line bug fix.
