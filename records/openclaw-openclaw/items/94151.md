## Summary

Keep open: the PR targets the right Slack plugin edit path, but it is not merge-ready because it adds undocumented `chat.update` identity fields without a no-identity retry and the supplied proof is only tests/source inspection, not real Slack behavior.

## What This Changes

This PR threads `SlackSendIdentity` through `editSlackMessage`, draft-stream edits, preview finalization, and dispatch so Slack `chat.update` payloads can include custom username/avatar fields.

## Best Possible Solution

Land one refreshed Slack plugin fix that forwards identity through agent-reply edit paths, retries without identity for rejected authorship fields, adds focused regression coverage, and includes redacted live Slack proof.

## Reproduction Assessment

Yes, source-level. Current main resolves custom Slack identity for sends, but `editSlackMessage`, draft-stream edits, and preview-finalize edits do not pass identity into `chat.update`; I did not run a live Slack workspace reproduction.

## Solution Assessment

No as submitted. Threading identity inside the Slack plugin is the right layer, but the best fix must preserve no-identity edit fallback behavior when Slack rejects custom identity fields and must show live Slack behavior before merge.
