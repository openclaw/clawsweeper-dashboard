## Summary

Keep open: the PR targets the right Slack plugin path, but it is not merge-ready because the new edit identity payload lacks the send path's customize-scope fallback and the supplied proof is only tests/source inspection, not live Slack behavior.

## What This Changes

This PR adds optional Slack send identity forwarding to `editSlackMessage`, draft-stream edits, and preview-finalize dispatch call sites so edited Slack messages can include custom username/avatar fields.

## Best Possible Solution

Land one Slack-plugin fix that forwards identity through agent-reply edit paths, mirrors the send path's no-identity retry behavior, adds focused edit/fallback regression tests, and includes redacted live Slack proof.

## Reproduction Assessment

Yes. Source inspection on current main shows initial Slack sends receive `SlackSendIdentity`, while `editSlackMessage`, draft-stream edits, and preview-finalize edits do not pass identity into `chat.update`; I did not run a live Slack workspace reproduction.

## Solution Assessment

No as submitted. Threading identity through the Slack plugin is the right layer, but the best fix must preserve no-identity fallback behavior and prove the real Slack edited-message result before merge.
