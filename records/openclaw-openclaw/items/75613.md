## Summary

Keep open. This PR is maintainer-authored, draft, and protected by the `maintainer` label, so it should not be closed by cleanup. The code direction is narrow and consistent with the plugin runtime boundary, but it leaves a small user-facing documentation gap around `--deep` now being the place where plugin runtime security collectors run.

## What This Changes

The PR changes security-audit context defaults so plugin security-audit collectors load only for deep audits unless explicitly requested, adds a regression test for plain audits, and adds a changelog entry.

## Best Possible Solution

Keep this PR open, update the user-facing `security audit` docs/help to describe plugin runtime collectors as deep-only, then let maintainers finish review and land it if the stated targeted tests and changed gate remain green. Leave the broader cold snapshot and backup-directory scanner work as a separate follow-up.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: plain `security audit` passes `deep: false` without a collector override, and the audit context currently defaults collector loading to true before reaching plugin runtime/metadata discovery.

## Solution Assessment

Mostly yes. Defaulting collector loading to `deep` is a narrow maintainable fix that preserves explicit callers, but the behavior change should be paired with docs/CLI copy so operators know plugin runtime collectors now require `--deep`.
