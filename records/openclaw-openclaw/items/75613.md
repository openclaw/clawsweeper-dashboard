## Summary

Keep open. This PR is open, ready for review, authored by a MEMBER, and labeled `maintainer`, so it should not be closed by cleanup; the proposed runtime change is narrow and directionally correct, but the user-facing CLI/docs copy still needs to describe that plugin runtime security collectors are now deep-only before a clean automerge path.

## What This Changes

The PR changes security-audit context defaults so plugin security-audit collectors load only for deep audits unless explicitly requested, adds a regression test for plain audits, and adds a changelog entry.

## Best Possible Solution

Keep the narrow runtime default change, add matching CLI/help and security-doc wording that plain audit avoids plugin runtime collectors while `--deep` includes them, then let the automerge path rely on the targeted tests and changed gate.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: plain `security audit` passes `deep: false` without a collector-loading override, and the audit context currently defaults collector loading on before reaching active plugin registry or metadata snapshot discovery.

## Solution Assessment

Mostly yes. Defaulting collector loading to `deep` is the narrowest maintainable fix because it preserves explicit internal callers and deep audits, but it should be paired with docs/CLI copy so the audit coverage boundary is visible.
