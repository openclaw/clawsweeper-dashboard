## Summary

Keep open. This is an automerge-opted maintainer PR with one narrow docs/help repair remaining: the runtime change is source-supported and covered by a focused regression test, but the CLI help and security docs still describe `--deep` only as a live Gateway probe while plugin-owned security collectors move to deep-only by default.

## What This Changes

This PR changes security-audit defaults so plugin security collectors load only for deep audits or explicit callers, adds a plain-audit regression test, and adds a changelog entry.

## Best Possible Solution

Keep the runtime default and regression test, update CLI/help and security docs so plain audit is documented as the cold config/filesystem path while `--deep` includes plugin-owned security collectors, then let the automerge lane gate the exact head with targeted checks and the changed gate.

## Reproduction Assessment

Yes. Current main’s plain CLI audit passes no collector-loading override, current main defaults collector loading on, and collector discovery reaches plugin runtime/metadata paths; the PR’s new regression test covers the intended non-deep behavior.

## Solution Assessment

Mostly yes. Defaulting collector loading to `opts.loadPluginSecurityCollectors ?? deep` is the narrowest maintainable runtime fix because it preserves explicit internal callers and deep audits; the safer complete solution is to pair it with docs/help updates for the changed audit boundary.
