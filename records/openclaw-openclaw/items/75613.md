## Summary

Keep open. This is a protected, maintainer-labeled, automerge-opted PR, and the runtime change is a narrow improvement, but it still needs one mechanical docs/CLI-copy repair so users can see that plugin-owned security collectors move from plain audit to deep audit by default.

## What This Changes

The PR changes security-audit context defaults so plugin security-audit collectors run only for deep audits or explicit callers, adds a plain-audit regression test, and adds a changelog entry.

## Best Possible Solution

Keep the runtime default change, add matching CLI/help and security-doc wording that plain audit avoids plugin runtime collectors while `--deep` includes plugin-owned collectors, then let the automerge path rely on the targeted regression test and changed gate.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main’s plain CLI audit passes no collector-loading override, current main defaults collector loading on, and collector discovery reaches plugin runtime/metadata paths; the PR’s added regression test covers the intended non-deep behavior.

## Solution Assessment

Mostly yes. Defaulting collector loading to `opts.loadPluginSecurityCollectors ?? deep` is the narrowest maintainable runtime fix because it preserves explicit internal callers and deep audits; the safer complete solution is to pair it with docs/help updates for the changed audit boundary.
