## Summary

Keep open. This is a protected, member-authored automerge PR, so it should not be cleanup-closed; the diff is directionally correct, but it still needs a narrow docs/help update before the automerge lane should treat it as complete.

## What This Changes

The PR changes security-audit context defaults so plugin security-audit collectors run only for deep audits or explicit callers, adds a plain-audit regression test, and adds a changelog entry.

## Best Possible Solution

Keep the runtime default and regression test, add matching CLI/help and security-doc wording that plain audit stays on cold config/filesystem paths while `--deep` includes plugin-owned security collectors, then let the automerge lane gate the exact head with targeted docs/test checks.

## Reproduction Assessment

Yes. Current main’s plain CLI audit passes no collector-loading override, current main defaults collector loading on, and collector discovery reaches plugin runtime/metadata paths; the PR’s new test covers the intended non-deep behavior.

## Solution Assessment

Mostly yes. Defaulting collector loading to `opts.loadPluginSecurityCollectors ?? deep` is the narrowest maintainable runtime fix because it preserves explicit internal callers and deep audits; the safer complete solution is to pair it with docs/help updates for the changed audit boundary.
