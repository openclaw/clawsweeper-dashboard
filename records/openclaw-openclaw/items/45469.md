## Summary

Keep open. Current main and the latest release still have an unbounded `GatewayClient.scheduleReconnect()` path, and https://github.com/openclaw/openclaw/pull/77961 is still the open implementation candidate with unresolved proof/retry-policy review state.

## What This Changes

The issue requests bounding `GatewayClient.scheduleReconnect()` retries to prevent unbounded reconnect loops when the gateway stays unreachable.

## Best Possible Solution

Settle GatewayClient retry semantics, then land a caller-aware reconnect cap with coverage and proof, or explicitly document unlimited retry as the intended contract.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. The long-running zombie-process and CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs caller-aware opt-in semantics or an explicit documented unlimited-retry contract.
