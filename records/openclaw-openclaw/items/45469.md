## Summary

Keep open. Current main still has unbounded ordinary `GatewayClient` reconnects, while existing code and tests intentionally preserve managed recovery for node-host/UI clients and only pause selected auth-failure loops, so the remaining change needs maintainer reconnect-policy selection rather than conservative cleanup.

## What This Changes

Add a maximum retry or exhausted-state policy to `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not retry forever.

## Best Possible Solution

Define a caller-aware reconnect policy that preserves managed node-host recovery while giving foreground clients a bounded terminal state or threshold callback, then cover it with focused gateway-client and TUI tests.

## Reproduction Assessment

Yes for the unbounded reconnect path: source inspection shows ordinary close events repeatedly reach `scheduleReconnect()`, and fake timers could drive unlimited replacement sockets. No for the separate zombie-process/listener accumulation claim, which is not proven against current main.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but a blanket max-retry default is not clearly the best fix because existing node-host recovery depends on continued reconnects; caller-specific policy or an exhausted/threshold callback is safer.
