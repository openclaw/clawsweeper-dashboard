## Summary

Keep open: current main and v2026.6.8 still leave the active package GatewayClient reconnect scheduler unbounded, and https://github.com/openclaw/openclaw/pull/77961 is the linked candidate but remains open, conflicting, and still needs caller/default policy plus real reconnect proof.

## What This Changes

The issue requests a bounded retry policy for GatewayClient.scheduleReconnect() to prevent indefinite reconnect loops when the gateway stays unreachable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77961 with a package-level opt-in reconnect cap, wire only the short-lived callers that should stop retrying, preserve node-host recovery, and add focused tests plus redacted real reconnect proof.

## Reproduction Assessment

Yes at source level: current packages/gateway-client close handling reaches scheduleReconnect(), which schedules another start without a max-attempt counter. I did not run a live gateway reproduction of the reported zombie-process accumulation.

## Solution Assessment

No, the issue's blanket max-retry suggestion is not the best final shape because node-host depends on long-lived recovery. The maintainable fix is an opt-in package-level policy wired to short-lived callers with explicit defaults and proof.
