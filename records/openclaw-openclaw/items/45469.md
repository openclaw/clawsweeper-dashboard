## Summary

Keep open. Current main still has the source-level unbounded reconnect path, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR is landed or deliberately closed.

## What This Changes

The issue requests a bounded retry policy for `GatewayClient.scheduleReconnect()` to avoid unbounded reconnect attempts when the gateway remains unreachable.

## Best Possible Solution

Keep this issue as the canonical thread while maintainers review the linked PR, decide the caller-specific retry policy, and land coverage plus any required production opt-ins.

Yes for the source-level loop: current `main` routes ordinary WebSocket closes into `scheduleReconnect()`, which schedules another `start()` without an attempt budget. No live long-running zombie-process reproduction was performed in this read-only review.
