## Summary

Keep open: current main still has an unbounded GatewayClient reconnect path for ordinary closes, and an open linked PR with closing syntax is already attempting the fix. The discussion also flags that unlimited reconnect is intentional for node-host recovery, so this needs caller-scoped retry policy review rather than a blanket cleanup close.

## What This Changes

Request to bound `GatewayClient.scheduleReconnect()` so unreachable gateways do not cause an infinite reconnect loop.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/77961 with caller-scoped bounded reconnect semantics that preserve node-host recovery.

## Reproduction Assessment

Yes for the source-level loop: current main's ordinary close path reaches `scheduleReconnect()`, which schedules `start()` again without an attempt counter. No live Railway/macOS reproduction was established for the reported zombie-process and CPU impact.

## Solution Assessment

Unclear. A blanket cap is probably not the best fix because node-host recovery intentionally relies on persistent reconnects; an opt-in or caller-scoped policy is safer but needs maintainer review.
