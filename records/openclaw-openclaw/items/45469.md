## Summary

Keep open. Current main and v2026.5.20 still have an unbounded GatewayClient reconnect scheduler, and the open closing PR should be reviewed or revised rather than closing this issue before merge.

## What This Changes

The issue requests a bounded retry policy for GatewayClient scheduleReconnect() to avoid endless reconnect attempts when the gateway stays unreachable.

## Best Possible Solution

Settle caller-aware GatewayClient retry semantics, then land https://github.com/openclaw/openclaw/pull/77961 or a replacement that gives short-lived clients a bounded retry option while preserving documented long-running node-host recovery.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches scheduleReconnect(), which schedules another start() without an attempt counter. The long-running zombie-process and CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs caller-aware opt-in semantics or an explicitly documented unlimited-retry contract.
