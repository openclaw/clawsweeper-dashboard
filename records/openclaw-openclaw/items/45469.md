## Summary

Keep open. Current main and v2026.5.12 still leave `GatewayClient.scheduleReconnect()` without an attempt ceiling, and the linked implementation candidate at https://github.com/openclaw/openclaw/pull/77961 is still open, so the issue should wait for that PR or an explicit retry-policy decision.

## What This Changes

The issue requests a maximum reconnect-attempt policy for `GatewayClient.scheduleReconnect()` to avoid unbounded WebSocket reconnect cycles when the gateway stays unreachable.

## Best Possible Solution

Keep this issue as the canonical thread until GatewayClient retry semantics are settled and a caller-aware cap or explicitly documented unlimited-retry contract lands.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. The long-running zombie-process and CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs caller-aware opt-in semantics or an explicit documented unlimited-retry contract.
