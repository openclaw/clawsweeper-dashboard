## What This Changes

The issue requests a bounded GatewayClient reconnect policy because unbounded retries can leave clients looping when the gateway is permanently unavailable.

## Best Possible Solution

Land a package-level, caller-specific reconnect policy with live reconnect proof, preserving long-lived node-host recovery while giving short-lived clients a bounded retry mode.

## Reproduction Assessment

## Solution Assessment

No, not as a hard global cap. The safer solution is an explicit bounded-reconnect policy in the package client, wired only to callers that should fail fast and proven against a real reconnect scenario.
