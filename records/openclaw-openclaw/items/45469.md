## Summary

Keep open. Current main and latest release still have unbounded GatewayClient reconnect scheduling, and the open linked implementation PR should be reviewed or replaced rather than closing this issue before merge.

## What This Changes

The issue requests a bounded retry policy for GatewayClient scheduleReconnect() to avoid infinite reconnect loops when the gateway remains unreachable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77961 or a replacement that gives short-lived clients bounded reconnect support while preserving the documented long-running node-host recovery contract.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches scheduleReconnect(), which schedules another start() without any attempt counter. The long-running zombie-process and idle CPU claims were not live-reproduced in this review.

## Solution Assessment

Unclear. A blanket default cap is not obviously the best fix because node-host currently relies on GatewayClient retrying indefinitely; the maintainable direction is caller-aware opt-in retry limiting or a clearly documented unlimited-retry contract.
