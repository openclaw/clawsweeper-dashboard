## Summary

Keep open: current main and v2026.6.6 still have no GatewayClient reconnect attempt ceiling, and the open closing implementation path remains conflicting, proof-blocked, and dependent on a reconnect policy decision.

## What This Changes

This issue asks for GatewayClient reconnect scheduling to stop after a bounded number of attempts instead of retrying forever.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/77961 is rebased or replaced with a package-level bounded reconnect policy, explicit caller behavior, focused tests, and real runtime reconnect proof.

## Reproduction Assessment

Yes for the core source path: current scheduleReconnect always schedules another start after close with max backoff but no attempt ceiling. I did not reproduce the reported zombie-process or CPU accumulation in a live setup.

## Solution Assessment

Unclear until maintainers decide the reconnect policy. An opt-in or caller-specific cap looks safer than a global default because node-host currently relies on long-running reconnects.
