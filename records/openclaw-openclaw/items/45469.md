## Summary

Keep open. Current main still has an unbounded ordinary GatewayClient reconnect scheduler; adjacent fixes bound request timeouts and selected auth-failure loops, but not normal WebSocket close retries. The remaining fix needs maintainer reconnect-policy selection because node-host clients currently rely on long-lived recovery while TUI/UI callers may need a finite terminal path or threshold callback.

## What This Changes

The issue asks for a retry ceiling or terminal reconnect policy in `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not retry forever.

## Best Possible Solution

Define a documented GatewayClient reconnect policy that preserves intentional indefinite recovery where needed, gives supervisor-backed or interactive clients a finite terminal path or threshold callback, resets any retry budget after hello-ok, keeps existing auth-pause behavior intact, and adds focused regression tests plus protocol documentation.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to use the existing GatewayClient WebSocket stub and fake timers, emit repeated ordinary close events such as 1006 or 1012, and observe replacement sockets being scheduled indefinitely with no retry counter or exhaustion state.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but the suggested blanket max-retry default is not clearly the best maintainable fix because node-host recovery currently depends on continued reconnects; a caller-specific policy or threshold callback is safer.
