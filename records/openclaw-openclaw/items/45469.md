## Summary

Keep open. Current main still has an unbounded GatewayClient reconnect scheduler for ordinary WebSocket closes, while the adjacent fixes only bound request timeouts and selected terminal auth reconnect loops. The remaining work needs a maintainer decision on reconnect policy because node-host clients currently expect long-lived recovery.

## What This Changes

The issue requests a maximum retry limit or terminal reconnect path for GatewayClient.scheduleReconnect() to avoid endless reconnect attempts when the gateway remains unavailable.

## Best Possible Solution

Define a documented GatewayClient reconnect policy that preserves intentional indefinite recovery for long-lived node-host or SDK clients, gives supervisor-backed or UI clients a finite terminal path or threshold callback, resets any retry budget after hello-ok, keeps auth-pause behavior intact, and adds focused regression tests plus protocol docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to use the existing GatewayClient WebSocket stub and fake timers, emit repeated ordinary close events such as 1006 or 1012, and observe replacement sockets being created after each scheduled delay with no retry counter or exhaustion state.

## Solution Assessment

Unclear. The report identifies a real unbounded path, but the suggested blanket max-retry default is not clearly the best fix because node-host recovery intentionally relies on continued reconnects; a caller-specific policy or threshold callback is safer.
