Keep open. Current main and the latest shipped release still have unbounded GatewayClient reconnect scheduling, and the live closing PR is open/unmerged, so this issue should wait for that PR or a maintainer retry-policy decision.

## What This Changes

The issue requests a maximum retry limit or pause mechanism for GatewayClient reconnects when the gateway remains unreachable.

## Best Possible Solution

Keep this issue as the canonical thread until maintainers decide GatewayClient retry semantics and either land a caller-aware retry-cap change or document that unlimited reconnect is intentional for the relevant clients.

## Reproduction Assessment

Yes for the source-level reconnect loop: an ordinary WebSocket close reaches scheduleReconnect(), which schedules start() again without an attempt counter. The long-running zombie-process and sustained CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A caller-aware opt-in cap may be maintainable, but default retry behavior for node-host versus short-lived clients is still a product/runtime policy choice and the open PR has not merged.
