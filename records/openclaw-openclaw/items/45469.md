## Summary

Keep open: current main and the latest release still leave the active GatewayClient reconnect scheduler unbounded, and the linked implementation candidate is open, conflicting, and needs maintainer disposition rather than a duplicate repair job.

## What This Changes

The issue reports that `GatewayClient.scheduleReconnect()` retries forever without a retry ceiling, potentially causing an infinite reconnect loop when the gateway stays unreachable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77961 with a package-level opt-in reconnect cap, maintainer-approved short-lived caller wiring, unchanged node-host recovery by default, focused tests, and redacted real reconnect proof.

## Reproduction Assessment

Yes at source level: current package-client close handling reaches `scheduleReconnect()`, which schedules another `start()` without a max-attempt counter. I did not run a live long-duration reproduction of the reported zombie-process accumulation.

## Solution Assessment

No, the blanket max-retry suggestion is not the best final shape because node-host depends on long-lived recovery. The maintainable fix is an opt-in package-level policy wired only to maintainer-approved short-lived callers, with proof that the unlimited default remains intact.
