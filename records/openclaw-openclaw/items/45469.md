- `impact:crash-loop`: Transient gateway failure can leave affected clients indefinitely cycling through reconnect attempts.

## Summary

This issue remains necessary: current `main` intentionally retries transient gateway disconnects without a ceiling, while the claimed resource leak is not yet proven and the safer caller-specific policy remains unresolved. The open implementation PR https://github.com/openclaw/openclaw/pull/77961 explicitly owns the work, so the issue should remain open until that PR is updated, proven, and merged or closed.

## What This Changes

Add an opt-in maximum reconnect-attempt policy for short-lived GatewayClient callers while preserving indefinite recovery for long-running hosts.

## Best Possible Solution

Keep indefinite retries for long-running hosts, prove and repair any socket or timer resource leak independently, and expose bounded lifetime only in the shared package for concrete short-lived callers with a defined terminal action.

## Reproduction Assessment

Partly yes: current source gives a high-confidence path for indefinite transient retries, but there is no high-confidence current-main reproduction of listener accumulation or zombie processes.

## Solution Assessment

No, an unconditional retry ceiling is not the best solution because long-running node clients require recovery; the maintainable path is caller-specific bounded lifetime plus separate cleanup proof.
