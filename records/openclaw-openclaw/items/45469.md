## Summary

Keep open: current main still leaves the active GatewayClient reconnect scheduler without a retry ceiling, and the open linked implementation candidate remains unmerged and needs maintainer disposition.

## What This Changes

The issue reports that GatewayClient reconnect scheduling can retry forever without a maximum attempt policy, causing runaway reconnect behavior when the gateway stays unreachable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77961 with a package-level opt-in reconnect cap, maintainer-approved caller wiring, unchanged node-host recovery by default, focused tests, and redacted real reconnect proof.

## Reproduction Assessment

Yes at source level: current package-client close handling reaches scheduleReconnect(), which schedules another start without a max-attempt counter. I did not run a live long-duration reproduction of the reported zombie-process accumulation.

## Solution Assessment

No, the blanket max-retry suggestion is not the best final shape because node-host depends on long-lived recovery. The maintainable fix is an opt-in package-level policy wired only to maintainer-approved short-lived callers, with proof that the unlimited default remains intact.
