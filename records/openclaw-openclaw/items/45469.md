## Summary

Keep open. Current main still has an unbounded `GatewayClient` reconnect scheduler, and an open implementation PR references this issue as its resolution, so cleanup should wait for that PR to land or be deliberately closed.

## What This Changes

The issue requests a bounded retry policy for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not cause unbounded reconnect attempts.

## Best Possible Solution

Keep this issue as the canonical thread while maintainers review the linked opt-in retry-limit PR, decide caller-specific retry semantics, and land tests plus any required production opt-ins.

## Reproduction Assessment

Yes for the source-level loop: an ordinary close reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. I did not reproduce the long-running zombie-process or CPU claim in a live container.

## Solution Assessment

No close action is appropriate yet. The active PR is the current implementation path, but the maintainable fix still needs caller-specific retry policy review before this issue is resolved.
