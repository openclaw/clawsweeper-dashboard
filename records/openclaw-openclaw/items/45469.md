## Summary

Keep open. Current main and the latest release still have unbounded `GatewayClient` reconnect scheduling, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/77961 has not merged, so this should wait for the linked PR or an explicit maintainer retry-policy decision.

## What This Changes

Request to add a maximum retry limit or caller-visible pause path for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not retry forever.

## Best Possible Solution

Keep this issue as the canonical thread until GatewayClient retry semantics are settled and a caller-aware cap or explicit unlimited-retry contract is landed.

## Reproduction Assessment

Yes for the source-level reconnect loop: an ordinary WebSocket close reaches `scheduleReconnect()`, which schedules `start()` again without an attempt counter. The long-running zombie-process and sustained CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs a caller-aware opt-in cap or an explicit documented unlimited-retry contract.
