## Summary

Keep open: current main and the latest release still have no generic `GatewayClient` reconnect attempt ceiling, and the open closing implementation path at https://github.com/openclaw/openclaw/pull/77961 is still conflicting and proof-blocked.

## What This Changes

The issue requests bounding `GatewayClient.scheduleReconnect()` so a permanently unreachable gateway cannot retry forever without a caller-visible stop condition.

## Best Possible Solution

Keep this issue open until the linked PR is rebased/ported or replaced with a package-level bounded reconnect policy, focused tests, and real reconnect proof.

## Reproduction Assessment

Yes, source-level: current `packages/gateway-client/src/client.ts` schedules another `start()` after every close with backoff capped at 30 seconds and no attempt counter. I did not establish a live reproduction of the reported zombie-process accumulation.

## Solution Assessment

Unclear until maintainer product judgment: an opt-in `maxReconnectAttempts` policy is safer than a default cap because node-host currently relies on long-running reconnects. The existing open PR is the right path to rebase, port, prove, or close.
