## Summary

Keep open: current main and the latest release still have no `GatewayClient` reconnect attempt ceiling, and the open linked implementation PR remains the canonical path rather than a merged fix.

## What This Changes

The issue requests a bounded retry or pause policy for `GatewayClient` reconnects so permanently unreachable gateways do not retry forever.

## Best Possible Solution

Land or replace the linked PR with a package-level retry policy, focused tests, explicit caller/default behavior, and redacted real reconnect proof.

## Reproduction Assessment

Yes, source-level: current `packages/gateway-client/src/client.ts` schedules another `start()` after every close with no max-attempt counter. I did not run a live reproduction of the reported zombie-process accumulation.

## Solution Assessment

Unclear as a final product path: an opt-in cap is safer than a global cap because some node-host flows intentionally recover after long outages, but maintainers still need to decide caller policy and wire it in the package client.
