## Summary

Keep open. Current main still has unbounded `GatewayClient` reconnect scheduling, and an open implementation PR references this issue as its resolution, so cleanup should wait for that PR to merge or be deliberately closed.

## What This Changes

The issue requests a bounded `GatewayClient` reconnect policy so permanent gateway outages do not retry forever.

## Best Possible Solution

Keep this issue as the canonical thread while maintainers review the linked opt-in retry-limit PR, decide caller-specific retry semantics, and land tests plus any required production opt-ins.

## Reproduction Assessment

Yes for the source-level loop: ordinary close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. I did not reproduce the long-running zombie-process or CPU claim in a live container.

## Solution Assessment

No close action is appropriate yet. An opt-in cap is a plausible direction, but the maintainable solution still needs maintainer agreement on retry semantics and call-site opt-ins before this issue is resolved.
