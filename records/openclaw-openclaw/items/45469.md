## Summary

Keep open: current main and latest release still leave the active GatewayClient reconnect scheduler unbounded, and the linked fix candidate is open but unmerged, conflicting, and still needs policy/proof work.

## What This Changes

The issue requests a bounded reconnect retry policy for GatewayClient so permanently unreachable gateways do not cause unbounded reconnect loops.

## Best Possible Solution

Keep the issue open and land or replace the linked PR with a package-level, compatibility-aware retry ceiling, explicit opt-in caller policy, focused tests, and redacted real reconnect proof.

## Reproduction Assessment

Yes at source level: current `packages/gateway-client/src/client.ts` reaches `scheduleReconnect()` from close paths and schedules another `start()` without a max-attempt counter. I did not run a live gateway reproduction of the reported zombie-process accumulation.

## Solution Assessment

No, the blanket max-retry suggestion is not the best final shape because node-host relies on long-lived recovery. The best fix should live in `packages/gateway-client`, preserve unlimited recovery by default where needed, and explicitly opt in short-lived callers that should stop retrying.
