## Summary

Keep open. Current main still has unbounded ordinary GatewayClient reconnects with no retry budget or exhausted callback, while node-host clients intentionally rely on continued recovery, so the remaining fix needs maintainer reconnect-policy selection rather than automatic cleanup.

## What This Changes

The issue requests a maximum retry or exhausted-state policy for `GatewayClient.scheduleReconnect()` to prevent endless WebSocket reconnect attempts.

## Best Possible Solution

Define and document a caller-aware reconnect policy that preserves intentional node-host recovery, gives foreground UI/TUI callers a finite terminal path or threshold callback, and adds focused regression tests.

## Reproduction Assessment

Yes for the unbounded reconnect path: source inspection shows ordinary close events repeatedly reach `scheduleReconnect()` and fake timers can drive unlimited replacement sockets. The separate zombie-process impact is not proven against current main.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but a blanket max-retry default is not clearly the best fix because node-host recovery currently depends on continued reconnects; caller-specific policy or an exhausted/threshold callback is safer.
