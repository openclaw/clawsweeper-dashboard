## Summary

Keep open. Current `main` and the latest release still have an unbounded `GatewayClient.scheduleReconnect()` path, and the linked closing PR remains open with unresolved retry-policy/proof review state.

## What This Changes

The issue requests a bounded retry policy for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways cannot reconnect forever.

## Best Possible Solution

Settle GatewayClient retry semantics, then land a caller-aware reconnect cap or explicitly document unlimited retry as the intended contract with coverage for node-host recovery.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. The long-running zombie-process and CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs caller-aware opt-in semantics or an explicit documented unlimited-retry contract.
