## Summary

Keep open. Current main still has unbounded GatewayClient reconnect scheduling, and https://github.com/openclaw/openclaw/pull/77961 is an open closing PR for the same behavior, so this issue should wait for that PR to merge or for maintainers to choose another retry policy.

## What This Changes

The issue requests a maximum retry limit for GatewayClient reconnect scheduling to avoid unbounded reconnect loops.

## Best Possible Solution

Keep this issue as the canonical thread while maintainers review the linked opt-in retry-limit PR, settle retry semantics, and land tests plus any intended caller opt-ins.

## Reproduction Assessment

Yes for the source-level loop: an ordinary WebSocket close on current main reaches scheduleReconnect(), which schedules start() again without an attempt counter. The long-running zombie-process and sustained CPU claims were not live-reproduced here.

## Solution Assessment

No close action is appropriate yet. An opt-in cap may be reasonable, but the maintainable solution still needs maintainer agreement on retry semantics and caller opt-ins before this issue is resolved.
