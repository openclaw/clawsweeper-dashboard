## Summary

Keep open: current main and the latest release still have no GatewayClient reconnect attempt ceiling, and the linked implementation PR remains open, conflicting, and proof-blocked.

## What This Changes

The issue requests a bounded retry policy for GatewayClient reconnects so permanently unreachable gateways do not retry forever.

## Best Possible Solution

Keep this issue open until the linked PR is rebased or replaced with a maintainer-approved bounded reconnect policy, focused tests, and real reconnect proof.

## Reproduction Assessment

Source-reproducible: scheduleReconnect() on current main schedules another start() after every close with backoff capped at 30s and no attempt counter. I did not establish a live reproduction of the reported zombie-process accumulation.

## Solution Assessment

Unclear until maintainer product judgment: an opt-in maxReconnectAttempts policy looks safer than a default cap because node-host currently relies on long-running reconnects. The existing open PR is the right path to rebase, port, prove, or close.
