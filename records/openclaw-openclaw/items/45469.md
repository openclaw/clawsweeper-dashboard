## Summary

Keep open. Current main and the latest release still expose an unbounded GatewayClient reconnect scheduler, and the issue is paired with an open closing PR that needs review rather than pre-merge closure.

## What This Changes

The issue requests a maximum retry budget for `GatewayClient.scheduleReconnect()` to avoid indefinite reconnect loops when the gateway remains unreachable.

## Best Possible Solution

Settle caller-aware GatewayClient retry semantics, then land the linked PR or a replacement that adds a bounded retry option where needed while preserving intentional long-running node-host recovery.

## Reproduction Assessment

Yes for the source-level reconnect loop: ordinary WebSocket close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter. The long-running zombie-process and CPU claims were not live-reproduced here.

## Solution Assessment

Unclear. A blanket default cap could break node-host recovery, so the maintainable fix likely needs caller-aware opt-in semantics or an explicitly documented unlimited-retry contract.
