## Summary

Keep open. The underlying duplicate reset-hook problem is still source-reproducible on current main, but this PR cannot merge as-is because it edits the old reset-hook location and its dedupe can both miss concurrent duplicates and suppress later valid resets.

## What This Changes

The PR adds reset-hook deduplication and focused tests for repeated `/new` or `/reset` hook emissions from the same prior session.

## Best Possible Solution

Implement reset-hook duplicate suppression in `commands-reset-hooks.ts` with a pre-await claim scoped to one reset attempt or other short-lived boundary, plus regression coverage for duplicate and later-valid reset cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: hold `triggerInternalHook` pending, call `emitResetCommandHooks` twice for the same action/session/prior session, and current main has no pre-await duplicate claim to stop both emissions.

## Solution Assessment

No. The PR targets a valid bug, but the current implementation is not the best fix because it marks the key after the await, keeps duplicate state process-wide, and edits the old module instead of current `commands-reset-hooks.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Claim reset-hook duplicates before awaiting:** `src/auto-reply/reply/commands-core.ts:104-107`
  - body: The guard checks the dedupe set before firing the hook, but the key is not added until after `await triggerInternalHook`. Two re-entrant callers with the same prior session can both pass the guard and both emit `command:new` or `command:reset`, so the duplicate memory artifacts remain possible.
  - confidence: 0.9
- **[P2] Scope duplicate state to one reset attempt:** `src/auto-reply/reply/commands-core.ts:32-33`
  - body: The module-level Set is keyed only by action, normalized session key, and previous session id. A later valid reset with the same stable prior session can return early here and skip both the internal command hook and `before_reset`; keep the claim scoped to the current reset attempt or another short-lived boundary.
  - confidence: 0.86
