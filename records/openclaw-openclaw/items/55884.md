## Summary

Keep this PR open because the reported duplicate reset-hook problem still exists on current main, but the proposed patch is not merge-ready: it claims duplicates only after an await, uses long-lived process-wide suppression, and targets the pre-refactor `commands-core.ts` path instead of the current reset-hook module.

## What This Changes

This PR adds a bounded reset-hook dedupe set keyed by action, session key, and previous session id, plus tests for duplicate suppression and changed prior-session allow-through.

## Best Possible Solution

Implement reset-hook duplicate suppression in `commands-reset-hooks.ts` with a pre-await, per-reset-attempt claim that prevents true re-entry duplicates while preserving valid later `/new`, `/reset`, and `/reset soft` hook emissions.

## Reproduction Assessment

Yes. A focused unit test can hold `triggerInternalHook` pending, call `emitResetCommandHooks` twice with the same action/session key/previous session, and assert only one command hook and `before_reset` run; another test should prove a later independent reset is still emitted.

## Solution Assessment

No. The PR targets the right bug class, but the dedupe is registered after the first await, is too long-lived, and needs to be moved to the current reset-hook module on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Claim reset-hook duplicates before awaiting:** `src/auto-reply/reply/commands-core.ts:105-107`
  - body: The new guard checks `recentResetHookEmissions` before `await triggerInternalHook`, but the key is not added until after that await. Two callers with the same prior session can both pass the guard and both fire `command:new` or `command:reset`, so the duplicate memory artifacts this PR tries to prevent remain possible.
  - confidence: 0.9
- **[P2] Scope duplicate state to one reset attempt:** `src/auto-reply/reply/commands-core.ts:32-33`
  - body: The module-level `recentResetHookEmissions` set persists across unrelated commands and is keyed only by action, session key, and previous session id. A later valid reset with the same stable prior session id can return early and skip both internal command hooks and `before_reset`; keep the claim scoped to a single reset attempt or another short-lived boundary.
  - confidence: 0.86
