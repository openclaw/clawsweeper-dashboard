## Summary

Keep open: this PR targets a real reset-hook duplicate-emission bug, but the proposed process-wide dedupe races before it claims the key, can suppress valid later resets, and now targets an emitter that current main has moved into `commands-reset-hooks.ts`.

## What This Changes

The PR adds a bounded reset-hook dedupe set and focused tests in `commands-core` to suppress repeated `/new` or `/reset` hook emissions for the same prior session.

## Best Possible Solution

Implement reset-hook duplicate suppression in current `commands-reset-hooks.ts` with a pre-await, reset-attempt-scoped claim that blocks true re-entry duplicates while preserving later valid reset hook emissions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: hold `triggerInternalHook` pending, call `emitResetCommandHooks` twice for the same action/session/prior session, and assert only one command hook and `before_reset` run.

## Solution Assessment

No. The PR targets the right bug, but the key is claimed after the first await, the process-wide state is too broad, and the fix needs to move to the current reset-hook module.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Claim reset-hook duplicates before awaiting:** `src/auto-reply/reply/commands-core.ts:104-107`
  - body: The new guard checks `recentResetHookEmissions` before the hook fires, but the key is not added until after `await triggerInternalHook`. Two re-entrant callers with the same prior session can both pass the guard and both run `command:new` or `command:reset`, so the duplicate memory artifacts remain possible.
  - confidence: 0.9
- **[P2] Scope duplicate state to one reset attempt:** `src/auto-reply/reply/commands-core.ts:32-33`
  - body: The module-level `recentResetHookEmissions` set is keyed only by action, session key, and previous session id, so a later valid reset with the same stable prior session can return early and skip both internal command hooks and `before_reset`. Keep the claim scoped to the current reset attempt or another short-lived boundary.
  - confidence: 0.86
