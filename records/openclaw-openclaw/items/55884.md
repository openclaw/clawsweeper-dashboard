## Summary

Keep this PR open. Current main still emits `/new` and `/reset` hooks from `commands-reset-hooks.ts` without any keyed prior-session dedupe, so the PR is not obsolete; however, the proposed patch is not merge-ready because it targets the pre-split `commands-core.ts` implementation, registers the duplicate key after an await, and uses process-wide session-id memory that can suppress later valid reset hooks.

## What This Changes

The PR adds a module-level bounded Set to dedupe reset-hook emission by action, normalized session key, and previous session id, plus focused tests for duplicate suppression and new-prior-session allow-through.

## Best Possible Solution

Implement a current-main reset-hook dedupe in `src/auto-reply/reply/commands-reset-hooks.ts` that claims a duplicate before the first await and scopes that claim to one reset attempt or another safe short-lived boundary, while preserving valid later `/new` and `/reset` emissions and adding regression tests for both duplicate suppression and valid subsequent resets.

## Reproduction Assessment

Yes. A focused unit test can mock `triggerInternalHook` with a held promise and call `emitResetCommandHooks` twice with the same prior session to show the PR's post-await claim still double-fires; another test can repeat a valid later reset with the same process-wide key to show over-suppression.

## Solution Assessment

No. The PR targets the right bug class, but this is not the best implementation because it patches an old file location, records the dedupe key after an await, and keeps the dedupe in process-wide session-id state instead of a reset-attempt scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Register duplicate claims before awaiting hooks:** `src/auto-reply/reply/commands-core.ts:105-107`
  - body: The new guard checks `recentResetHookEmissions` before `await triggerInternalHook`, but the key is not added until after that await. Two concurrent calls with the same prior session can both pass the check and both fire the hook, so duplicate session-memory artifacts remain possible. Claim the key before the first await or use an equivalent pre-await per-attempt claim.
  - confidence: 0.9
- **[P2] Limit duplicate state to one reset attempt:** `src/auto-reply/reply/commands-core.ts:33`
  - body: The module-level `recentResetHookEmissions` Set persists across unrelated commands and is keyed only by action, session key, and prior session id. If a later valid `/new` or `/reset` sees the same key, this returns early and skips `command:new`/`before_reset` hooks. Scope the dedupe to the command/request or another short-lived reset attempt.
  - confidence: 0.86
