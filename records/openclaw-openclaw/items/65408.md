## Summary

Keep open. Current main still intentionally supports trusted cached device-token retry after `AUTH_TOKEN_MISMATCH` and keeps paired device-token sessions alive across shared-token rotation, while related work only narrows endpoint trust or handles the reconnect/login-gate UX separately.

## What This Changes

Request shared-token rotation to invalidate Control UI browser sessions, add an auth logout that clears persisted state, and avoid login-gate flashes during session restore.

## Best Possible Solution

Define the shared-token rotation and paired-device-token invalidation contract, then update gateway server/client behavior, docs, regression tests, and Control UI logout/bootstrap UI together.

## Reproduction Assessment

Yes, at source level. A browser with an explicit shared token, device identity, and cached paired device token can hit `AUTH_TOKEN_MISMATCH`, receive retry advice, and reconnect with `auth.deviceToken` on trusted endpoints.

## Solution Assessment

No current fix is present. Disabling the retry path alone is not clearly the best solution because current docs and tests support it, so maintainers should first decide the security and compatibility contract.
