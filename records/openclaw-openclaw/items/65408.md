## Summary

Keep open. Current main still contains the Control UI shared-token mismatch recovery path through cached device tokens, and current docs/tests describe that retry and device-token survival after shared-token rotation as expected behavior; the logout and initial login-gate flash concerns also remain unresolved on main, with related open PRs covering only part of the reconnect UX.

## What This Changes

The issue asks to invalidate Control UI browser auth after gateway token rotation, add a logout that clears persisted auth state, and avoid flashing the login gate during session restore.

## Best Possible Solution

Define the shared-token rotation contract first, then enforce it consistently in gateway and browser auth, add a Control UI logout that clears both scoped session token and device auth state, and introduce an explicit restoring state before rendering the login gate.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: authenticate with a shared gateway token, keep a stored paired device token, rotate the shared token, then the client/server retry path can recover with the cached device token.

## Solution Assessment

No current fix is present. A narrow code removal is not clearly the best solution because current docs/tests define the retry as supported behavior, so maintainers need to decide the auth/security compatibility contract before implementation.
