- `impact:security`: The report is about whether gateway token rotation reliably revokes browser access through the Control UI auth boundary.
- `impact:session-state`: The reported browser session can persist or recover across refresh/restart, and the requested logout must clear persisted session state.
- `impact:auth-provider`: The affected path is gateway token and device-token authentication, including retry behavior after token mismatch.

## Summary

Keep open. Current main still documents and tests trusted cached device-token retry after `AUTH_TOKEN_MISMATCH`, and paired device-token sessions are still preserved across shared-token rotation, so this needs a security/product contract decision rather than conservative cleanup.

## What This Changes

The issue asks for gateway token rotation to invalidate Control UI browser sessions and device-token retry recovery, add a first-class auth logout, and avoid login-gate flashes during session restore.

## Best Possible Solution

Define the token-rotation and paired-device invalidation policy first, then update the gateway server/client behavior, an explicit Control UI auth logout, bootstrap rendering, docs, and regression tests consistently.

## Reproduction Assessment

Yes, at source level. A browser with an explicit shared token, device identity, and cached paired device token can receive `AUTH_TOKEN_MISMATCH` retry advice and reconnect with `auth.deviceToken` on trusted endpoints.

## Solution Assessment

No current implementation path is clearly best yet. Disabling the retry path alone conflicts with documented and tested compatibility behavior, so maintainers should decide whether rotation revokes paired device tokens globally, via migration, or through an explicit strict mode.
