## Summary

Keep open. Current main still contains the reported Control UI shared-token mismatch recovery path through cached device tokens, and the docs/tests still describe that retry as expected behavior. The logout and bootstrap/login-flash parts also remain unresolved as first-class Control UI auth/session behavior, although open UI PRs appear related to the transient reconnect UX portion.

## What This Changes

The issue asks OpenClaw to invalidate Control UI browser sessions on gateway token rotation, add a logout action that clears persisted auth state, and avoid flashing the login form during session restore.

## Best Possible Solution

Define the shared-token rotation contract first, then enforce it consistently: either bind/revoke browser device tokens when the shared gateway token changes or remove the shared-token mismatch fallback for Control UI shared-token sessions. Add a visible Control UI logout that clears the scoped sessionStorage gateway token, local device auth token, and active GatewayBrowserClient state, and add an explicit restoring/reconnect state before showing the login gate.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: authenticate with a shared gateway token, keep a stored paired device token, rotate the shared token, and the client/server retry path can still recover with the cached device token.

## Solution Assessment

No current fix is present. A one-line removal of the retry would be risky because the fallback is documented and tested; the best path needs maintainer approval for the auth/security compatibility contract, with the UX-only reconnect PRs reviewed separately.
