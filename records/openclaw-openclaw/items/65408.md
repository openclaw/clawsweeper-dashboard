## Summary

Keep open. Current main still supports cached device-token recovery after a shared-token mismatch and keeps paired device-token sessions alive across shared-token rotation; changing that requires a security/product contract decision plus coordinated Control UI logout and bootstrap UX work.

## What This Changes

The issue requests gateway-token rotation to invalidate Control UI browser sessions, plus a first-class logout action and a bootstrap state that avoids login-form flashing on refresh.

## Best Possible Solution

Define the token-rotation and paired-device invalidation policy, then update the gateway server/client behavior, Control UI logout/bootstrap UX, docs, and regression tests consistently.

## Reproduction Assessment

Yes, at source level. A browser with an explicit shared token, device identity, and cached paired device token can receive AUTH_TOKEN_MISMATCH retry advice and reconnect with auth.deviceToken on trusted endpoints.

## Solution Assessment

No current fix is present. Disabling the retry path alone is not clearly best because current docs and tests support it, so maintainers should first decide the security and compatibility contract.
