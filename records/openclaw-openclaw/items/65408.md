## Summary

Keep this issue open. Current main still intentionally allows the reported Control UI shared-token mismatch recovery path through cached device tokens, the behavior is documented and covered by tests, and the requested logout/bootstrap UX changes are not implemented as first-class Control UI session controls.

## What This Changes

The issue requests stronger Control UI auth invalidation after gateway token rotation, a logout action that clears persisted auth state, and a restoring state that prevents a login-form flash during session restore.

## Best Possible Solution

Define and document the shared-token rotation contract, then enforce it consistently by binding or revoking browser device tokens when the shared token changes, or by removing the shared-token mismatch fallback for Control UI shared-token sessions. Add a Control UI logout that clears the scoped sessionStorage gateway token, persisted device auth entry, and active GatewayBrowserClient state, plus an explicit restoring state before rendering the login form.

## Reproduction Assessment

Yes. Current source and focused tests provide a high-confidence reproduction path: authenticate with a gateway token, retain a stored paired device token, rotate the shared token, then the retry path can authenticate through the cached device token.

## Solution Assessment

No current fix is present. Disabling the fallback alone may be too broad because the behavior is documented and tested; the safer path is a maintainer-approved auth contract plus targeted logout and bootstrap UI changes.
