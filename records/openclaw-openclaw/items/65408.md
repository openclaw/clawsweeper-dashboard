## Summary

Keep open. Current main and the latest release still allow and document trusted cached device-token retry after shared-token mismatch; the requested invalidation/logout/bootstrap behavior conflicts with existing tests/docs and needs maintainer security/product direction rather than cleanup closure.

## What This Changes

The issue asks for shared-token rotation to invalidate Control UI browser/device-token auth, add a logout that clears persisted auth state, and avoid showing the login gate while session restore is pending.

## Best Possible Solution

Define the shared-token rotation and browser-session invalidation contract, then update gateway/client/docs/tests together and add a Control UI logout plus explicit restoring state.

## Reproduction Assessment

Yes, at source level. A browser with an explicit shared token, device identity, and cached paired device token can hit AUTH_TOKEN_MISMATCH, receive retry hints, and reconnect with auth.deviceToken on trusted endpoints.

## Solution Assessment

No current fix is present. Deleting the retry path alone is not clearly the best solution because the retry is documented and tested as supported behavior, so maintainers should decide the security/compatibility contract first.
