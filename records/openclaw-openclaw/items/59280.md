## Summary

Keep open: current main still has no native restore command, and this PR is useful, but the latest head still has destructive restore-safety bugs, no contributor-supplied real behavior proof artifacts, and an unresolved product overlap with the open encrypted snapshot restore work.

## What This Changes

The PR adds a native `openclaw backup restore <archive>` command with dry-run, force, JSON output, archive path decoding, shared manifest reading, and restore tests.

## Best Possible Solution

Harden native archive restore with execution-time overwrite refusal, rollback-safe state replacement, docs and trust-model coverage, and an explicit maintainer choice on whether it lands separately from the encrypted snapshot restore flow.

## Reproduction Assessment

Yes for the PR findings: latest-head source inspection shows the no-force execution-time overwrite path and the forced directory rollback failure path. This is not a current-main bug reproduction because main has no native `backup restore` command.

## Solution Assessment

No: the proposed direction is plausible, but this patch is not yet the best safe implementation. The safer path is to fail closed on execution-time conflicts, restore from a staging/swap model that can roll back directories, document the trust model, and decide how it relates to the encrypted snapshot restore PR.

- **[P1] Reject raced-in targets without --force:** `src/infra/backup-restore.ts:130-132`
  - body: `restoreSingleAsset` recomputes `conflictNow`, but when `--force` is false it only leaves `snapshotPath` undefined and then continues into `copyFile` or directory merge.   - confidence: 0.87
