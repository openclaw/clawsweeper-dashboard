## Summary

Keep open. Current main still exposes only backup create/verify, and although restore belongs in core rather than only in the linked OpenClaw Snapshot workaround, this PR has blocking restore-target integrity, file-asset, duplicate-entry, and overwrite-safety defects.

## What This Changes

The PR adds `openclaw backup restore <archive>` with `--dry-run`, `--force`, and `--json`, factors shared archive-reading helpers out of backup verification, adds path decoding, and adds restore tests.

## Best Possible Solution

Land a hardened core restore command in the existing backup CLI, with destination integrity checks, file and directory asset handling, duplicate-entry parity with verify, execution-time conflict checks, docs, and a changelog entry; keep OpenClaw Snapshot as a useful external workaround.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config-only archive from `openclaw backup create --only-config`, plus a tampered manifest whose `sourcePath` differs from its encoded payload path; the current PR diff handles those paths unsafely.

## Solution Assessment

No. Adding restore to the core backup command is the right direction, but this patch is not the best implementation until it validates restore destinations, handles file assets, preserves verify’s duplicate checks, and re-checks overwrite conflicts at execution time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive restore targets from archive paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the write destination even though the protected payload location is `archivePath`. A crafted archive can keep a valid payload path but rewrite `sourcePath` to an arbitrary local path, then restore or rename that path with `--force`; derive the destination from the encoded payload path or fail unless the manifest matches it exactly.
  - confidence: 0.91
- **[P1] Restore file assets without directory coercion:** `src/infra/backup-restore.ts:182-183`
  - body: Supported backups can contain a single file asset, especially `openclaw backup create --only-config`. This code creates `asset.restorePath` as a directory before merging staged content, so a config-file restore can fail or create a directory where the JSON file should be restored.
  - confidence: 0.87
- **[P2] Re-check conflicts at restore time:** `src/infra/backup-restore.ts:132-139`
  - body: `executeRestore` relies on `asset.conflict` computed during planning. If a target appears after planning but before extraction, restore proceeds without `--force` and without taking a `.pre-restore-*` snapshot, violating the overwrite-safety contract.
  - confidence: 0.78
- **[P2] Reject duplicate archive entries before restore:** `src/infra/backup-restore.ts:74-80`
  - body: Restore normalizes archive entries into a `Set` before validation, dropping duplicate-path evidence that `backup verify` rejects. Preserve the duplicate-entry check before planning so restore cannot accept last-entry-wins archive content that verify would reject.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention: the patch adds a state-writing restore command that can trust tampered manifest data and weaker archive validation than `backup verify`.

Concerns:

- **[high] Manifest-controlled restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses manifest `sourcePath` directly as the destination instead of deriving or validating it from the encoded payload path, so a crafted archive can redirect restore writes or forced renames.
  - confidence: 0.91
- **[medium] Duplicate archive entries accepted by restore:** `src/infra/backup-restore.ts:74`
  - body: Restore drops duplicate normalized entry paths by converting them to a `Set`, weakening restore relative to `backup verify` and allowing non-deterministic restored content.
  - confidence: 0.82
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:132`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created path can be overwritten without `--force` or a snapshot.
  - confidence: 0.78
