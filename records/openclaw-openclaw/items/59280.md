## Summary

Keep open: the latest head fixes some earlier archive-integrity gaps, but the destructive restore path can still write into a target created after planning when `--force` is not set, and contributor-supplied real behavior proof is still absent.

## What This Changes

This PR adds a native `openclaw backup restore <archive>` subcommand with dry-run, force, JSON output, shared manifest reading, archive path decoding, and restore tests.

## Best Possible Solution

Harden native archive restore with execution-time overwrite refusal, rollback-safe snapshots, docs and trust-model coverage, plus redacted CLI proof, or fold the useful archive-restore tests into https://github.com/openclaw/openclaw/pull/44111 if maintainers prefer one restore design.

## Reproduction Assessment

Yes for the PR findings: source inspection of the latest PR head shows the no-force execution-time overwrite window and the pre-try forced-rename rollback gap. This is not a current-main bug reproduction because current main has no native `backup restore` command.

## Solution Assessment

No, not yet. The feature direction is useful, but the current patch still needs overwrite-safety hardening, rollback coverage, docs/trust-model coverage, real CLI proof, and maintainer direction on whether to land separately from the broader snapshot restore work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject fresh restore conflicts without --force:** `src/infra/backup-restore.ts:130-132`
  - body: The command only rejects conflicts found by `planRestore`. If a target appears before `restoreSingleAsset` runs and `--force` is false, this fresh conflict check does not throw, so restore continues into an existing destination without taking a safety snapshot, violating the advertised overwrite refusal.
  - confidence: 0.92
- **[P2] Move the force snapshot inside rollback coverage:** `src/infra/backup-restore.ts:134-138`
  - body: On forced restores, the existing target is renamed before the staging directory is created and before the try/catch begins. If staging setup throws after the rename, the rollback block never runs and the user's active state remains moved to the `.pre-restore-*` path.
  - confidence: 0.84
