## Summary

Keep this PR open: current main still exposes only backup create/verify, but the proposed native restore command is a new destructive CLI feature with unresolved restore-safety findings and missing inspectable real behavior proof. The linked OpenClaw Snapshot workaround is useful context, but it does not replace a hardened first-party restore command.

## What This Changes

The PR adds `openclaw backup restore <archive>` with dry-run, force, JSON output, shared backup archive-reading helpers, path decoding, CLI registration, and restore tests.

## Best Possible Solution

Land a hardened native restore flow only after archive destination validation, file and empty-directory handling, duplicate-entry checks, execution-time conflict checks, docs, and redacted real CLI proof are in place.

## Reproduction Assessment

Yes for the PR review findings: the unsafe destination trust, duplicate-entry validation gap, stale conflict decision, file-asset handling, empty-directory handling, and zero-asset behavior are visible by source inspection of the PR head. This is not a current-main bug reproduction because current main has no native restore subcommand.

## Solution Assessment

No. A native restore command is a reasonable direction, but this branch is not the best mergeable implementation until the destructive restore path is hardened and the contributor supplies inspectable real CLI proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive restore targets from archive paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the write target even though the payload path is encoded in `archivePath`. A tampered archive can keep a valid payload entry and redirect `sourcePath`, so restore should derive the target from the encoded archive path or reject mismatches before writing or forced renaming.
  - confidence: 0.9
- **[P1] Restore file assets without directory coercion:** `src/infra/backup-restore.ts:181-183`
  - body: The restore loop always creates `asset.restorePath` as a directory before merging staged content. Backups can contain file assets such as `openclaw backup create --only-config`, so this can fail or recreate the config file path as a directory instead of restoring the file.
  - confidence: 0.86
- **[P2] Reject duplicate archive entries before planning restore:** `src/infra/backup-restore.ts:74-80`
  - body: `readManifestFromArchive` normalizes entries into a `Set` before validation, which can hide duplicate tar paths that current verify rejects. Restore should reject duplicates before planning so malformed archives cannot restore last-entry-wins content while appearing valid.
  - confidence: 0.8
- **[P2] Re-check target conflicts before writing:** `src/infra/backup-restore.ts:132-143`
  - body: `executeRestore` relies on the plan-time `asset.conflict` bit to decide overwrite behavior. If a target appears after planning, the command can write into it without `--force` or a safety snapshot, so existence needs to be checked immediately before each restore.
  - confidence: 0.8
- **[P2] Allow valid empty directory assets:** `src/infra/backup-restore.ts:173-178`
  - body: A valid empty directory asset can extract to an empty staging directory after the full asset path is stripped, but this branch treats that as corruption. Restore should recreate the empty directory instead of failing on a valid backup.
  - confidence: 0.76
- **[P2] Reject zero-asset manifests before restoring:** `src/infra/backup-restore.ts:97-99`
  - body: `planRestore` accepts an empty `assets` array and reports a normal no-op plan. Backup create does not intentionally emit zero-asset archives, so restore should fail malformed or truncated manifests before a recovery command claims success.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff adds a destructive local restore command with concrete archive-trust and overwrite-safety concerns.

Concerns:

- **[high] Manifest controls restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses the manifest `sourcePath` as the filesystem destination instead of deriving or validating it against the encoded payload path, so a crafted archive can redirect restore writes or forced renames.
  - confidence: 0.9
- **[medium] Duplicate archive entries bypass restore validation:** `src/infra/backup-restore.ts:74`
  - body: Restore collapses normalized archive entries into a `Set` before validation, weakening restore relative to verify and allowing duplicate-path archive content to be accepted.
  - confidence: 0.8
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:132`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created path can be overwritten without `--force` or a safety snapshot.
  - confidence: 0.8
