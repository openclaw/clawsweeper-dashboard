## Summary

Keep open: current main still lacks a native backup restore command, but the latest PR head still has destructive restore safety defects and no attached real behavior proof.

## What This Changes

The PR adds `openclaw backup restore <archive>` with `--dry-run`, `--force`, and `--json`, refactors backup manifest reading for reuse, adds archive path decoding, and adds restore tests.

## Best Possible Solution

Harden native archive restore with execution-time overwrite refusal, rollback-safe snapshots, docs and trust-model coverage, plus redacted CLI proof, or fold the useful archive tests into the broader snapshot restore design.

## Reproduction Assessment

Yes for the PR findings: source inspection of the latest PR head shows the no-force execution-time overwrite window and the pre-try forced-rename rollback gap. This is not a current-main bug reproduction because current main has no native `backup restore` command.

## Solution Assessment

No, not yet. The feature direction is useful, but the current patch still needs overwrite-safety hardening, rollback coverage, docs/trust-model coverage, real CLI proof, and maintainer direction on whether to land separately from the broader snapshot restore work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fail on execution-time conflicts without --force:** `src/infra/backup-restore.ts:130-132`
  - body: `conflictNow` is recomputed, but when `force` is false the result is ignored. If a restore target is created after `planRestore` and before execution, this path proceeds to extract and copy into it without `--force` or a `.pre-restore-*` snapshot, violating the command's advertised overwrite refusal.
  - confidence: 0.88
- **[P2] Keep snapshot rename inside rollback handling:** `src/infra/backup-restore.ts:134-138`
  - body: The forced-restore rename happens before staging allocation enters the guarded block. If `fs.mkdtemp` fails after the rename, rollback is skipped and the user's active state remains moved away from its expected path; allocate staging first or put the rename inside the `try` block.
  - confidence: 0.78
