## Summary

Keep open. Current main still exposes only backup create/verify, and this PR is a relevant core implementation candidate, with the author’s linked OpenClaw Snapshot skill as a useful workaround in the meantime. The branch should not merge as-is because the restore implementation still trusts mutable manifest destinations, mishandles file assets such as `--only-config` backups, and the PR head is currently dirty against base.

## What This Changes

This PR adds `openclaw backup restore <archive>` with `--dry-run`, `--force`, and `--json`, factors archive-reading helpers out of backup verification, and adds restore command/infra tests.

## Best Possible Solution

A hardened core restore command should land in the existing backup CLI, deriving or validating destinations from the encoded archive payload path, handling both file and directory assets, rejecting malformed/tampered archives consistently with verify, documenting the new command, and preserving the linked OpenClaw Snapshot skill as an external workaround rather than the core answer.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config-only archive from `openclaw backup create --only-config`: the PR’s restore path treats the config file target as a directory, and a tampered manifest can redirect restore writes because `planRestore` trusts `manifest.assets[].sourcePath`.

## Solution Assessment

No. Adding restore to the core backup command is the right product direction, but this implementation needs destination integrity checks, file-asset support, conflict rechecks, docs/changelog, and a clean rebase before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive restore targets from archive paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the write destination even though the protected payload location is `archivePath`. A tampered archive can keep a valid payload path but rewrite `sourcePath` to any filesystem location, then restore or rename that path with `--force`. Decode the destination from the encoded payload path, or fail unless the manifest source matches it exactly.
  - confidence: 0.91
- **[P1] Restore file assets without creating target directories:** `src/infra/backup-restore.ts:182-183`
  - body: Supported backups can contain a single file asset, especially `openclaw backup create --only-config`. This code always creates `asset.restorePath` as a directory before merging staged content, so a config-file restore can fail or create a directory where the JSON file should be restored.
  - confidence: 0.87
- **[P2] Re-check conflicts before restoring each asset:** `src/infra/backup-restore.ts:132-139`
  - body: `executeRestore` relies on `asset.conflict` computed during planning. If a target appears after planning but before extraction, restore proceeds without `--force` and without taking a `.pre-restore-*` snapshot, violating the command’s overwrite-safety contract.
  - confidence: 0.78
- **[P2] Reject duplicate archive entries during restore:** `src/infra/backup-restore.ts:74-80`
  - body: Restore builds a `Set` of normalized entry paths before verification, which drops duplicates. `backup verify` rejects duplicate normalized entries, but restore can accept the same archive and then extract with last-entry-wins behavior. Preserve the duplicate-entry check before calling shared verification.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces a local restore command that can write or rename arbitrary local paths if a backup manifest is tampered with.

Concerns:

- **[high] Manifest-controlled restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses the manifest `sourcePath` directly as the destination instead of deriving or validating it from the encoded payload path. A crafted archive can pass structural payload checks while redirecting restore writes to an attacker-chosen path.
  - confidence: 0.91
- **[medium] Duplicate archive entries accepted by restore:** `src/infra/backup-restore.ts:74`
  - body: Restore converts normalized archive entries into a `Set`, losing duplicate-path evidence before verification. That weakens restore relative to `backup verify` and can allow non-deterministic restored content from duplicate payload entries.
  - confidence: 0.82
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:132`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created target can be overwritten without `--force` and without a safety snapshot.
  - confidence: 0.78
