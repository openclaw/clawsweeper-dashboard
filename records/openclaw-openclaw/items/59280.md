## Summary

Keep this PR open: current main still lacks a native backup restore command, but the proposed restore path remains unsafe for merge because it writes user state based on manifest-controlled destinations and has several restore-validation gaps.

## What This Changes

The PR adds `openclaw backup restore <archive>` with dry-run, force, JSON output, shared archive-reading utilities, path decoding, and restore tests.

## Best Possible Solution

Land a hardened first-party restore command only after maintainers settle the native backup scope and the implementation validates destinations, preserves verify parity, handles file and empty-directory assets, documents the command, and includes redacted real CLI proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the latest PR head shows the unsafe restore paths and validation gaps directly. This is not a current-main bug reproduction because main does not yet have the restore command.

## Solution Assessment

No. A native restore command is a valid direction, but this patch is not the best mergeable implementation until the restore trust model, file/empty-dir handling, duplicate checks, conflict timing, docs, and real proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive restore targets from archive paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the destination even though the protected payload location is `archivePath`. A crafted archive can keep a valid payload path but rewrite `sourcePath` to any local path, then restore or rename that target with `--force`; derive the destination from the encoded payload path or fail unless the manifest matches it exactly.
  - confidence: 0.91
- **[P1] Restore file assets without directory coercion:** `src/infra/backup-restore.ts:181-183`
  - body: Backups can contain a single file asset, especially `openclaw backup create --only-config`. This code always creates `asset.restorePath` as a directory and then merges staged content into it, so config-file restores can fail or create a directory where the JSON file should be.
  - confidence: 0.87
- **[P2] Re-check target conflicts before writing:** `src/infra/backup-restore.ts:132-143`
  - body: `executeRestore` relies on the `asset.conflict` value computed during planning. If a target appears after planning but before extraction, restore can write into it without `--force` and without taking a `.pre-restore-*` snapshot, violating the overwrite-safety contract.
  - confidence: 0.8
- **[P2] Reject duplicate archive entries before planning restore:** `src/infra/backup-restore.ts:74-80`
  - body: `readManifestFromArchive` normalizes entries into a `Set` before validation, which silently drops duplicate paths. That lets restore accept archives that current `backup verify` rejects and can produce last-entry-wins restored content from a malformed archive.
  - confidence: 0.82
- **[P2] Allow valid empty directory assets:** `src/infra/backup-restore.ts:173-178`
  - body: A valid backup can include an empty directory asset such as an empty workspace. After stripping the exact directory entry, staging can be empty; throwing here makes restore fail on a valid archive instead of recreating the empty directory.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The patch adds a state-writing restore command and still has concrete archive-trust and overwrite-safety weaknesses.

Concerns:

- **[high] Manifest controls restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses manifest `sourcePath` directly as the write target instead of deriving or validating it from the encoded payload path, so a crafted archive can redirect restore writes or forced renames.
  - confidence: 0.91
- **[medium] Duplicate archive entries can bypass restore validation:** `src/infra/backup-restore.ts:74`
  - body: Restore converts normalized entries to a `Set` before validation, weakening restore relative to `backup verify` and allowing duplicate-path archive content to be accepted.
  - confidence: 0.82
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:132`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created path can be overwritten without `--force` or a safety snapshot.
  - confidence: 0.8
