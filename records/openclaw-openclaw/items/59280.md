## Summary

Keep open: current main still has create/verify only, while this external restore PR remains unsafe to merge because source inspection shows destructive restore trust and overwrite gaps and the PR lacks inspectable real behavior proof.

## What This Changes

The PR adds `openclaw backup restore <archive>` with dry-run, force, JSON output, shared archive-reading helpers, path decoding, CLI registration, and restore tests.

## Best Possible Solution

Land a hardened native restore flow only after archive target validation, verify parity, file and empty-directory handling, execution-time conflict checks, docs, and redacted CLI proof are in place.

## Reproduction Assessment

Yes for the PR review findings: the unsafe destination trust, duplicate-entry validation gap, stale conflict decision, file-asset handling, empty-directory handling, and zero-asset behavior are visible by source inspection on the PR head. This is not a current-main bug reproduction because main has no restore subcommand.

## Solution Assessment

No. A native restore command is a reasonable direction, but this patch is not the best mergeable implementation until the destructive restore path is hardened and the contributor supplies redacted real CLI proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive restore targets from archive paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the write target even though the payload path is encoded in `archivePath`. A crafted archive can point a valid payload at another local path, so derive the destination from the encoded archive path or reject mismatches before any writes or forced renames.
  - confidence: 0.91
- **[P1] Restore file assets without directory coercion:** `src/infra/backup-restore.ts:181-183`
  - body: The restore loop always creates `asset.restorePath` as a directory before merging staged content. Archives created with `openclaw backup create --only-config` contain a file asset, so restore can fail or replace the intended JSON file with a directory.
  - confidence: 0.87
- **[P2] Reject duplicate archive entries before planning restore:** `src/infra/backup-restore.ts:74-80`
  - body: `readManifestFromArchive` collapses normalized entries into a `Set` before validation. That accepts duplicate tar entries that current verify rejects and can restore last-entry-wins content from malformed archives.
  - confidence: 0.82
- **[P2] Re-check target conflicts before writing:** `src/infra/backup-restore.ts:132-143`
  - body: `executeRestore` uses the plan-time `asset.conflict` bit to decide overwrite behavior. If a target appears after planning, the command can write into it without `--force` or a safety snapshot.
  - confidence: 0.8
- **[P2] Allow valid empty directory assets:** `src/infra/backup-restore.ts:173-178`
  - body: After stripping the asset path, a valid empty directory asset leaves staging empty, but this branch treats that as corruption. Restore should recreate the empty directory instead of failing on a valid backup.
  - confidence: 0.76
- **[P2] Reject zero-asset manifests before restoring:** `src/infra/backup-restore.ts:97-99`
  - body: `planRestore` accepts an empty `assets` array and reports a successful no-op plan. Backup create does not intentionally emit zero-asset archives, so treat this as malformed before a disaster-recovery command claims success.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff adds a destructive state restore command with concrete archive-trust and overwrite-safety issues.

Concerns:

- **[high] Manifest controls restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses manifest `sourcePath` directly as the write target instead of deriving or validating it from the encoded payload path, so a crafted archive can redirect restore writes or forced renames.
  - confidence: 0.91
- **[medium] Duplicate archive entries bypass restore validation:** `src/infra/backup-restore.ts:74`
  - body: Restore converts normalized entries to a `Set` before validation, weakening restore relative to `backup verify` and allowing duplicate-path archive content to be accepted.
  - confidence: 0.82
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:132`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created path can be overwritten without `--force` or a safety snapshot.
  - confidence: 0.8
