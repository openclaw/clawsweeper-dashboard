## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive restore targets from payload paths:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the filesystem destination even though the payload path already encodes the backed-up path. A tampered archive can keep a valid payload entry but redirect restore writes or forced renames, so restore should derive the destination from `archivePath` or reject mismatches before writing.
  - confidence: 0.9
- **[P1] Restore file assets without making them directories:** `src/infra/backup-restore.ts:181-183`
  - body: The restore loop always creates `asset.restorePath` as a directory before merging staged content. Current `backup create --only-config` produces a config-file asset, so restoring that archive can fail or recreate the file path as a directory instead of restoring the file.
  - confidence: 0.86
- **[P2] Reject duplicate archive entries before validation:** `src/infra/backup-restore.ts:74-80`
  - body: `readManifestFromArchive` normalizes entries into a `Set` before validation, which can hide duplicate tar paths that current `backup verify` rejects. Restore should reject duplicates before planning so malformed archives cannot restore last-entry-wins content while appearing valid.
  - confidence: 0.82
- **[P2] Recheck conflicts immediately before writing:** `src/infra/backup-restore.ts:131-143`
  - body: `executeRestore` relies on the plan-time `asset.conflict` bit to decide overwrite behavior. If a target appears after planning, the command can write into it without `--force` or a safety snapshot, so existence needs to be checked immediately before each restore.
  - confidence: 0.8
- **[P2] Preserve valid empty directory assets:** `src/infra/backup-restore.ts:173-178`
  - body: A valid empty directory asset can extract to an empty staging directory after the asset prefix is stripped, but this branch treats that as corruption. Restore should recreate the empty directory instead of failing on a valid backup.
  - confidence: 0.76
- **[P2] Reject empty restore manifests:** `src/infra/backup-restore.ts:97-99`
  - body: `planRestore` accepts an empty `assets` array and reports a normal no-op plan. `backup create` does not intentionally emit zero-asset archives, so restore should fail malformed or truncated manifests before a recovery command claims success.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds a destructive local restore command with concrete archive-trust and overwrite-safety concerns.

Concerns:

- **[high] Manifest controls restore destination:** `src/infra/backup-restore.ts:99`
  - body: `planRestore` uses manifest `sourcePath` as the filesystem destination instead of deriving or validating it against the encoded payload path, so a crafted archive can redirect restore writes or forced renames.
  - confidence: 0.9
- **[medium] Duplicate archive entries bypass restore validation:** `src/infra/backup-restore.ts:74`
  - body: Restore collapses normalized archive entries into a `Set` before validation, weakening restore relative to verify and allowing duplicate-path archive content to be accepted.
  - confidence: 0.82
- **[medium] Plan-time overwrite decision can go stale:** `src/infra/backup-restore.ts:131`
  - body: The write loop uses the conflict bit from planning instead of checking the target immediately before restore, so a newly-created path can be overwritten without `--force` or a safety snapshot.
