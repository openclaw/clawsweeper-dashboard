- `impact:session-state`: The central failure is stale active session bookkeeping preventing the direct lane from draining.
- `impact:message-loss`: The stuck lane delayed the user-visible response path and made queued direct messages effectively silent until recovery completed.

## Summary

Keep open: current `main` still lets the stuck-session warning backoff return before the heartbeat can request recovery, matching the reported failure mode. The central fix is narrow; the QQ fast-ack and hook watchdog ideas should stay separate follow-ups.

## What This Changes

The issue asks OpenClaw to separate stuck-session diagnostic warning throttling from recovery scheduling so recovery-eligible stale sessions are not delayed by log backoff.

## Best Possible Solution

Fix `logSessionAttention`/heartbeat recovery so warning throttling suppresses repeated warnings/events only, while recovery-eligible stuck or stalled classifications still reach `requestStuckSessionRecovery`; track QQ fast-ack and hook process watchdogs separately.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current `main` can return from `logSessionAttention` during stuck or long-running warning backoff before the heartbeat reaches the recovery request branch. I did not run a live QQ gateway reproduction in this read-only review.

## Solution Assessment

Yes for the central bug: separating warning/event throttling from recovery scheduling is the narrowest maintainable fix. The QQ fast-ack and hook watchdog ideas are useful follow-ups, but they are separate from this regression boundary.
