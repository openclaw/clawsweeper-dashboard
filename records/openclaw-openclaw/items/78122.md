## Summary

Keep open: source inspection supports the reported workspace-default bug and the patch direction, but this external PR still lacks required after-fix real behavior proof and has a small changelog-credit issue.

## What This Changes

The PR changes non-default agent workspace resolution to use sibling state `workspace-<id>` fallbacks, adds resolver and `agents add` wizard regression coverage, and adds a changelog entry.

## Best Possible Solution

Land a narrow workspace resolver/add-wizard fix that keeps new non-default agents on documented sibling workspaces, with regression coverage, changelog credit, and real CLI proof from the contributor.

## Reproduction Assessment

Yes from source inspection: current main resolves a non-default agent without an explicit workspace to `agents.defaults.workspace/<id>`, and `agents add` uses that resolver for the wizard initial value. I did not run a live wizard because this review is read-only.

## Solution Assessment

Mostly yes: removing the non-default fallback aligns the suggested workspace with documented `workspace-<agentId>` siblings. The one caveat is compatibility with the previously tested #59789 fallback, which maintainers should consciously accept.
