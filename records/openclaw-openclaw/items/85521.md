## Summary

Current main and the latest release still show the reported auth-profile snapshot bug: `saveAuthProfileStore` persists a disk-filtered store and then installs that same filtered store into the active runtime snapshot. This issue should stay open because https://github.com/openclaw/openclaw/pull/85558 is the open fixing PR with closing syntax and should own the fix path until it lands or is replaced.

## What This Changes

The issue reports `saveAuthProfileStore` replacing active runtime auth snapshots with a disk-filtered store, causing external CLI OAuth profiles to disappear from in-process auth.

## Best Possible Solution

Land the focused auth-profile store fix from https://github.com/openclaw/openclaw/pull/85558 or an equivalent replacement that preserves filtered disk persistence while keeping active runtime snapshots hydrated.

## Reproduction Assessment

Yes, at source level: current main filters runtime-only external OAuth profiles for persistence and then writes that filtered store into active runtime snapshots. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs at the auth-profile snapshot boundary: keep disk persistence filtered while refreshing active runtime snapshots with a runtime-safe store or reattached external profiles.
