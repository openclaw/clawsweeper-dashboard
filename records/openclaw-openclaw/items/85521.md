## Label Justifications

- `P1`: A valid source-backed auth-provider state bug can break embedded-agent runs for external-CLI OAuth users while the gateway remains healthy.
- `impact:session-state`: The failure is caused by in-process runtime auth-profile snapshot state diverging from the recoverable on-disk auth store.
- `impact:auth-provider`: The reported breakage is specifically loss of OAuth/external-CLI auth-profile material used for provider credential resolution.

## Summary

Keep open: current main still has the source-backed snapshot boundary bug where `saveAuthProfileStore` can replace an active runtime auth-profile snapshot with the disk-filtered `localStore`, and embedded-agent auth paths can later read snapshot-backed stores without re-overlaying external CLI profiles.

## What This Changes

The issue reports that auth-profile saves can drop external CLI OAuth credentials from the in-process runtime snapshot while leaving disk state recoverable.

## Best Possible Solution

Keep persistence filtering for disk writes, but update active runtime snapshots with a runtime-safe full store or re-overlay external CLI profiles before snapshot-backed embedded-agent auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the filtered `localStore` into an active runtime snapshot, and embedded-agent auth can consume snapshot-backed stores without external profiles. I did not run a live gateway reproduction because this was a byte-for-byte clean read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is at the auth-profile store snapshot boundary rather than another config reload workaround. The fix should separate disk persistence shape from runtime snapshot shape and add focused regression coverage.
