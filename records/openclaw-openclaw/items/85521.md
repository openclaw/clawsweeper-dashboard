## Summary

Current main still has a source-backed auth-profile snapshot bug: `saveAuthProfileStore` writes a persistence-filtered `localStore` back into the active runtime snapshot, and embedded-agent auth paths can consume snapshot-backed stores without external CLI profiles. This should stay open as a narrow fix candidate rather than being closed as stale or already implemented.

## What This Changes

The issue reports that `saveAuthProfileStore` can replace the in-process auth-profile snapshot with a disk-filtered view and drop external CLI OAuth credentials such as `anthropic:claude-cli`.

## Best Possible Solution

Keep the disk-filtered persistence behavior, but update runtime snapshots with a runtime-safe full store or consistently re-overlay external CLI profiles before snapshot-backed auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the filtered `localStore` into the runtime snapshot, and embedded auth paths can consume runtime snapshots without external profiles. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix belongs at the auth-profile store snapshot boundary: preserve persistence filtering for disk while keeping runtime snapshots hydrated with external-CLI OAuth material or re-overlaying it before snapshot-backed reads.
