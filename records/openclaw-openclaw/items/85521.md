## Summary

Current main still has a source-backed auth-profile runtime snapshot bug, so this should stay open as a narrow P1 auth-provider state repair rather than maintainer cleanup.

## What This Changes

The issue reports that `saveAuthProfileStore` can replace an active runtime auth-profile snapshot with a disk-filtered store and drop external-CLI OAuth credentials from embedded-agent auth resolution.

## Best Possible Solution

Keep external-CLI credentials out of local persisted auth files, but update the runtime snapshot with a runtime-safe full store or consistently re-overlay external profiles before snapshot-backed embedded auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the filtered `localStore` into an active runtime snapshot, and the embedded runner can pass a no-external-profiles store into model auth. I did not run the live gateway sequence in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow auth-profile store boundary repair: preserve persistence filtering for disk while keeping runtime snapshots hydrated with external-CLI OAuth material or re-overlaying it before snapshot-backed embedded auth reads.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85521. Current main's `saveAuthProfileStore` builds a disk-filtered `localStore` and installs it as the runtime snapshot; embedded auth can then use `ensureAuthProfileStoreWithoutExternalProfiles` and pass that stripped store into model auth, causing external-CLI OAuth profiles such as `anthropic:claude-cli` to disappear from in-process state. Implement the narrow fix at the auth-profile store snapshot or runtime overlay boundary so disk persistence still excludes externally managed credentials when appropriate, but active runtime snapshots/readers retain or re-attach usable external-CLI OAuth material. Add a regression that sets an active runtime snapshot with an external-CLI OAuth profile, calls `saveAuthProfileStore` through a local/config-write-style save, verifies the persisted file remains correctly filtered when expected, and verifies embedded/secrets-runtime auth resolution can still resolve the external CLI profile afterward. Do not broaden plugin/provider APIs, do not persist runtime-only external secrets incorrectly, and do not change provider fallback policy. Add a `CHANGELOG.md` fix entry if preparing a repair PR.
