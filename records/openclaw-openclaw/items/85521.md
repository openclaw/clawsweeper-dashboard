## Label Justifications

- `P1`: The issue can break embedded-agent authentication for external-CLI OAuth users while the gateway remains healthy.
- `impact:session-state`: The failure is caused by in-process runtime auth-profile snapshot state diverging from the usable auth state.
- `impact:auth-provider`: The affected path is provider OAuth profile resolution for external-CLI credentials such as `anthropic:claude-cli`.

## Summary

Current main still has the reported auth-profile snapshot boundary problem: `saveAuthProfileStore` builds a persistence-safe `localStore` and installs that same store into the active runtime snapshot, while embedded-agent auth paths can consume snapshot-backed stores without re-overlaying external CLI profiles. This remains a narrow, source-backed auth-provider/session-state bug rather than a cleanup close.

## What This Changes

The issue asks for `saveAuthProfileStore` to preserve or reattach external-CLI OAuth profiles in runtime auth-profile snapshots after writing persistence-filtered auth state.

## Best Possible Solution

Keep external-CLI credentials out of persisted local auth files, but update active runtime snapshots with a runtime-safe full store or consistently re-overlay external profiles before snapshot-backed auth reads return, with focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the persistence-local store into an active runtime snapshot, and embedded auth paths can consume no-external-profile snapshot-backed stores. I did not run a live gateway/config-reload reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is at the auth-profile store snapshot boundary: keep disk filtering intact while ensuring runtime snapshots retain or reattach external-CLI OAuth profiles before embedded auth resolution uses them.
