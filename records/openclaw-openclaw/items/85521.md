## Label Justifications

- `P1`: The issue can break embedded-agent authentication for external-CLI OAuth users while the gateway remains healthy.
- `impact:session-state`: The failure is an in-process runtime auth-profile snapshot diverging from the persisted auth-profile state.
- `impact:auth-provider`: The affected state is OAuth/external-CLI provider credential resolution for an auth profile.

## Summary

Current main still has a narrow auth-profile runtime snapshot hazard: the save path builds a filtered local store for persistence and installs that same filtered store as the active runtime snapshot, while embedded auth paths can consume snapshot-backed stores before external auth is reattached. This should stay open as a focused P1 auth-provider/session-state bug.

## What This Changes

The issue asks OpenClaw to keep external-CLI OAuth credentials available in runtime auth-profile snapshots after `saveAuthProfileStore` writes a filtered persisted store.

## Best Possible Solution

Preserve the disk-filtered persisted store, but keep active runtime snapshots hydrated or consistently re-overlay external auth before snapshot-backed embedded auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the filtered `localStore` into an active runtime snapshot, and embedded auth paths can consume snapshot-backed stores without external profiles. I did not run the live gateway scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is at the auth-profile store snapshot boundary. Keep persistence filtering for disk, but avoid installing a stripped persistence view as the runtime view or reattach external profiles before snapshot-backed runtime reads return.
