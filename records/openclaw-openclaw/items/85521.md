## Summary

Current main still has the reported auth-profile snapshot bug: the save path persists a filtered local store and then installs that same filtered store as the active runtime snapshot, while embedded-agent auth paths can consume snapshot-backed stores without external CLI profiles. This is a valid narrow auth-provider state bug, not a cleanup close.

## What This Changes

The issue requests that `saveAuthProfileStore` preserve or reattach external-CLI OAuth profiles in the runtime auth-profile snapshot after auth/config writes.

## Best Possible Solution

Keep external-CLI credentials out of persisted local auth files, but update the runtime snapshot with a runtime-safe full store or consistently re-overlay external profiles before snapshot-backed auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still writes the filtered `localStore` into the runtime snapshot, and embedded auth paths can consume runtime snapshots without external profiles. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix belongs at the auth-profile store snapshot boundary: preserve persistence filtering for disk while keeping runtime snapshots hydrated with external-CLI OAuth material or consistently overlaying it before snapshot-backed reads.
