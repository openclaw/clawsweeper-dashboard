## Label Justifications

- `P1`: The reported state bug can break embedded-agent authentication for external-CLI OAuth users while the gateway remains healthy.
- `impact:session-state`: The failure is caused by the in-process runtime auth-profile snapshot diverging from the usable credential state.
- `impact:auth-provider`: The affected path drops external-CLI OAuth profile material needed for provider credential resolution.

## Summary

Keep open: current main still writes the persistence-filtered `localStore` into an active runtime auth-profile snapshot, and snapshot-backed auth reads can then bypass external-CLI OAuth reattachment. This is a narrow, source-reproducible P1 auth-provider/session-state bug suitable for a focused fix PR.

## What This Changes

The issue requests preserving external-CLI OAuth profiles in runtime auth-profile snapshots after `saveAuthProfileStore` writes filtered local auth data.

## Best Possible Solution

Keep external-CLI credentials out of persisted local auth files, but update the active runtime snapshot from a runtime-safe full store or consistently re-overlay external profiles before snapshot-backed auth reads return.

## Reproduction Assessment

Yes for source-level reproduction: current main still filters a save store and then installs that filtered value as the active runtime auth-profile snapshot. I did not run a live gateway reproduction because this review was required to remain read-only.

## Solution Assessment

Yes, the narrowest maintainable solution is at the auth-profile store snapshot boundary. Preserve persistence filtering for disk, but do not let that filtered view become the runtime credential snapshot used by embedded-agent auth.
