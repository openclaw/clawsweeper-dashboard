## Label Justifications

- `P2`: This is a normal-priority memory recall bug with silent indexed-content loss for a narrow but realistic archive transcript pattern.
- `impact:data-loss`: The transcript remains on disk, but its searchable archived session content is silently dropped from the memory index.
- `impact:session-state`: The bug misclassifies ordinary archived session text as cron-generated state and makes memory_search drift from the real transcript history.

## Summary

Keep open. Current main and the latest release still allow a usage-counted reset/deleted archive to flip `generatedByCronRun` from a user-role `[cron:...]` text prefix, clear collected content, and skip the rest of the archive before it reaches the session memory index.

## What This Changes

The issue asks to stop reset/deleted session archives from being emptied in `memory_search` when an ordinary user message starts with `[cron:`.

## Best Possible Solution

Use trusted session metadata/provenance for archive-wide cron suppression, but keep ordinary user-authored `[cron:]` text and following archive content indexed for session memory.

## Reproduction Assessment

Yes at source level: current `buildSessionEntry` clears archive content and skips later messages after a user-role `[cron:]` prefix on usage-counted archive paths. I did not run a live end-to-end `memory_search` query in this read-only review.

## Solution Assessment

Yes, if kept narrow: the best fix is in `buildSessionEntry`, removing content-only archive-wide cron classification while preserving trusted metadata/store classification for genuine cron runs.
