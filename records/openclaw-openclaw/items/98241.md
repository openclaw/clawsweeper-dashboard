## Label Justifications

- `P2`: This is a normal-priority memory recall bug with silent indexed-content loss for a narrow but realistic transcript pattern.
- `impact:data-loss`: The archive remains on disk but its searchable session content is silently dropped from the memory index.
- `impact:session-state`: The bug misclassifies ordinary archived session transcripts as cron-generated state and drifts memory_search from the real transcript history.

## Summary

Keep open. Current main and the latest release still allow a usage-counted reset/deleted archive to flip `generatedByCronRun` from a user-role `[cron:...]` text prefix, clear already-collected content, and skip the remaining archive, which matches the reported `memory_search` loss mode.

## What This Changes

Fix `memory_search` archive indexing so a human-authored `[cron:...]` message in a reset/deleted session archive does not make the whole archive invisible.

## Best Possible Solution

Constrain archive-wide cron classification to trusted transcript/session metadata or store classification, keep only the single generated `[cron:]` line sanitized when appropriate, and add a regression proving later unrelated archive content remains indexed.

## Reproduction Assessment

Yes at source level: current `buildSessionEntry` clears archive content and skips all later messages after a user-role `[cron:]` prefix on usage-counted archive paths. I did not run a live end-to-end memory_search query in this read-only review.

## Solution Assessment

Yes, if kept narrow: the best fix is in `buildSessionEntry`, removing content-only archive-wide cron classification while preserving trusted metadata/store classification for genuine cron runs. A broader config or Dreaming/QMD rewrite is not needed for this bug.
