## Summary

Keep open: the underlying lint bug is real and the supplied proof is usable, but the patch lowercases path targets and can hide broken case-sensitive markdown links, so it needs a small correction before merge.

## What This Changes

The branch updates memory-wiki lint normalization to accept page titles, aliases, fragments, query strings, leading `./`, and `.md` variants, with a regression test.

## Best Possible Solution

Keep the title/alias support, but split lint matching so path-derived targets are normalized without case folding while title and alias targets can be matched case-insensitively.

## Reproduction Assessment

Yes, from source inspection: bridge pages store titles separately from generated paths, `extractWikiLinks` returns title/path targets, and current lint only accepts path-derived targets. I did not run a local repro because this review is read-only.

## Solution Assessment

No, not as written. The direction is right, but the implementation should preserve case-sensitive path matching and only case-fold title/alias matches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve path casing for path link targets:** `extensions/memory-wiki/src/lint.ts:55`
  - body: The new normalization lowercases every valid target and every link target. That means a path link such as `[Alpha](sources/Alpha.md)` will match an existing `sources/alpha.md` page, so the linter will stop reporting a broken link on case-sensitive filesystems and Git checkouts. Keep title/alias matching case-folded, but compare path-derived targets with case preserved.
  - confidence: 0.91
