## Summary

Keep open: the PR addresses a real memory-wiki lint bug, but the current patch has two blocking correctness issues and the later bridge self-heal change still lacks after-fix real behavior proof.

## What This Changes

The PR expands memory-wiki lint target normalization to page titles and aliases and adds a bridge imported-source repair path for malformed pages with current sync state.

## Best Possible Solution

Land a memory-wiki-local fix that keeps path-derived link checks case-sensitive, case-folds only title or alias metadata keys, and verifies group-specific imported-source provenance before skipping repair.

## Reproduction Assessment

Yes. Source inspection shows current main only validates path-shaped lint targets, and the PR body provides real terminal proof that title-style links stopped producing broken-wikilink warnings after the patch.

## Solution Assessment

No. The direction is right, but this implementation should preserve path casing, require bridge or unsafe-local provenance fields before skipping repair, and add real proof for the bridge self-heal path before merge.

- **[P2] Preserve path casing during lint normalization:** `extensions/memory-wiki/src/lint.ts:55`
  - body: Lowercasing every normalized target makes path-based links case-insensitive. In a native Markdown vault on a case-sensitive filesystem or Git checkout, `[Alpha](sources/Alpha.md)` can pass when the actual file is `sources/alpha.md`, suppressing a real broken-link warning; keep path-derived keys case-preserving and only case-fold title or alias keys.

When sync state is current, those pages will be skipped and existing `missing-import-provenance` warnings can remain until the source changes; include the group-specific fields before returning unchanged.
