## Summary

Keep open: this PR fixes a real memory-wiki lint false positive and includes sufficient real-vault proof, but the current patch lowercases every lint target and can mask wrong-case relative path links that should still be reported broken.

## What This Changes

The branch updates memory-wiki lint normalization to accept page titles, aliases, fragments, query strings, markdown extensions, and leading `./` targets, with an Obsidian title/alias regression test.

## Best Possible Solution

Merge a narrow lint resolver fix that accepts case-folded title and alias targets while keeping path-based targets case-preserving, with regression coverage for wrong-case path links.

## Reproduction Assessment

Yes. Current main’s lint resolver only accepts exact path/basename targets while Obsidian title targets are extracted as link targets, and the PR body supplies after-fix output from a real vault showing the false positives disappeared.

## Solution Assessment

No. The direction is right, but the implementation should separate case-preserving path keys from case-folded title/alias keys instead of lowercasing every target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve path casing during lint normalization:** `extensions/memory-wiki/src/lint.ts:55`
  - body: Lowercasing every normalized target makes relative path links case-insensitive. A native markdown link like `[Alpha](sources/Alpha.md)` would pass when the file is `sources/alpha.md`, hiding a link that can fail on case-sensitive filesystems or Git checkouts; keep path keys case-preserving and case-fold only titles and aliases.
  - confidence: 0.9
