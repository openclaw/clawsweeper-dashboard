## Summary

Keep open. The PR addresses a real `memory-wiki` lint false positive and includes sufficient real-vault proof, but the current patch should not merge as-is because it lowercases relative path targets and can hide broken path links on case-sensitive vaults.

## What This Changes

The branch adds memory-wiki lint normalization for title, alias, fragment, query, markdown extension, and leading `./` link targets, plus a regression test.

## Best Possible Solution

Merge a narrow lint resolver change that adds title/alias and fragment/query normalization while preserving case-sensitive path validation and adding regression coverage for wrong-case path links.

## Reproduction Assessment

Yes. Current main's lint resolver only accepts exact path/basename targets, and the PR body provides a real-vault after-fix command showing the title-link false positives disappeared.

## Solution Assessment

No. The requested direction is right, but the implementation should separate case-folded title/alias keys from case-preserving path keys instead of lowercasing every link target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve path casing during lint normalization:** `extensions/memory-wiki/src/lint.ts:55`
  - body: This helper lowercases every link target before lookup, including relative path links. That makes `[Alpha](sources/Alpha.md)` pass when the actual page is `sources/alpha.md`, so native markdown vaults on case-sensitive filesystems or Git checkouts no longer get a `broken-wikilink` warning for a link that will not resolve. Keep path keys case-preserving and case-fold only titles/aliases.
  - confidence: 0.9
