## Summary

Keep open: the reported memory-wiki lint bug is real and this PR is relevant, but the current head should not merge because it case-folds path links, leaves a provenance repair gap, and lacks real behavior proof for the added bridge self-heal path.

## What This Changes

The PR updates memory-wiki lint target normalization for title, alias, fragment, query, and relative Markdown targets, adds lint coverage, and adds bridge imported-source self-repair logic with a bridge test.

## Best Possible Solution

Merge a narrow memory-wiki fix that preserves case-sensitive path validation, case-folds only title/alias metadata, verifies group-specific imported-source provenance before cache skips, and includes real bridge self-heal proof.

## Reproduction Assessment

Yes. Current main's lint resolver indexes only path-shaped targets, Obsidian title/fragment targets reach lint unresolved, and shouldSkipImportedSourceWrite can skip an existing generated page based on sync metadata and file existence before rendered content is rechecked.

## Solution Assessment

No. The direction is right, but the implementation should split case-preserving path keys from case-folded title/alias keys and require bridge or unsafe-local provenance fields before treating an imported source page as complete.

  - body: Lowercasing every normalized target makes path-based links case-insensitive. In a native Markdown vault on a case-sensitive filesystem or Git checkout, `[Alpha](sources/Alpha.md)` can pass when the actual file is `sources/alpha.md`, suppressing a real broken-link warning; keep path-derived targets case-preserving and only case-fold title or alias metadata keys.
