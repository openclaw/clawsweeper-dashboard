## Best Possible Solution

Keep the exec-event `nextDueMs` deferral bypass and retryable min-spacing behavior, but preserve normal schedule advancement after successful exec-event attempts and add stale-due regression coverage before merge.

## Solution Assessment

No. The source-aware exec-event bypass is the right layer, but skipping schedule advancement after exec-event runs is broader than needed and can leave stale interval due slots; the best fix preserves advancement while bypassing only the exec-event deferral gate.

Full review comments:

- **[P2] Advance the schedule after exec-event runs:** `src/infra/heartbeat-runner.ts:2167-2168`
  - body: This early return means a successful exec-event run leaves `agent.nextDueMs` unchanged. If that due slot is already at or before `now`, the runner's `finally` path calls `scheduleNext()`, which arms a 0 ms interval wake; scheduled wakes do not go through the min-spacing gate, so a background exec completion can be followed by an immediate extra heartbeat. The exec-event delivery fix only needs the source-aware bypass in `shouldDeferWake`, so keep normal schedule advancement and add a stale-due regression test.
  - confidence: 0.86
- **[P3] Include min-spacing in the retryable reason type:** `src/infra/heartbeat-wake.ts:14`
  - body: The patch adds `HEARTBEAT_SKIP_MIN_SPACING` to the runtime retryable set, but the exported `RetryableHeartbeatBusySkipReason` union still excludes it. Any typed helper or test using that contract cannot represent a reason that `isRetryableHeartbeatBusySkipReason()` now accepts, so the type should be updated with the new constant.
  - confidence: 0.78
