## Summary

Keep open, but the latest patch is not merge-ready: it addresses a real background-exec notification loss path and includes usable real behavior proof, but the shared schedule change is broader than the exec-event path and can leave other wake sources on stale due slots.

## What This Changes

The PR changes heartbeat cooldown behavior so `exec-event` wakes bypass interval `nextDueMs`, min-spacing deferrals become retryable, and scheduler tests expect background exec completions to retry after min-spacing.

## Best Possible Solution

Keep the exec-event delivery fix, but make schedule advancement source-aware so only the background exec completion path stops pushing `nextDueMs` forward while non-exec wake sources keep the existing cooldown advancement and regression coverage.

## Reproduction Assessment

Yes for the review finding: on this branch, a non-exec wake that runs with `nextDueMs <= now` returns from `advanceAgentSchedule` without moving the due slot, and `scheduleNext()` then re-arms from that stale value. I did not run tests because this review was read-only; the original exec-event bug also has contributor-supplied terminal proof in the PR body.

## Solution Assessment

No. The exec-event-specific cooldown bypass is a plausible narrow fix, but making schedule advancement a no-op for every non-interval wake is broader than the reported bug and should be narrowed before merge.
