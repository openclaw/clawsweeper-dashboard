## What This Changes

The PR adds a shared shell-command heuristic, non-blocking stderr warnings for main-session systemEvent cron add/edit payloads, focused cron CLI tests, and unrelated formatting-only churn across several existing files.

## Best Possible Solution

Land the narrow CLI warning without changing cron runtime semantics: warn before the gateway RPC for add and edit, cover existing-main edits when --session is omitted, use normalized session target handling for explicit edit sessions, add focused tests and a changelog entry, then close #63107 after merge.

## Reproduction Assessment

Yes. Current main can be reproduced from source and docs: main-session cron enqueues systemEvent text and records success, while cron add/edit do not warn when shell-looking text is used. The PR adds focused tests for the intended warning paths, but I did not execute them in this read-only review.

## Solution Assessment

Yes, mostly. A non-blocking CLI warning is the narrowest maintainable fix because it preserves current cron execution semantics; the safer finish is to base the edit warning on the already-normalized sessionTarget and add the required changelog entry.
