## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, while the underlying Gateway session RPC methods and schemas already exist. The related closed PRs do not resolve the requested HTTP surface, and the recent maintainer comment explicitly keeps this as the focused tracker rather than folding it into the broader read-only REST request.

## What This Changes

The issue requests authenticated REST session-management endpoints for HTTP API clients using `/v1/chat/completions`, covering reset/status/compact/model/reasoning without re-enabling slash-command dispatch.

## Best Possible Solution

Add a focused Gateway HTTP session-management surface that maps the requested REST endpoints onto existing session RPC/shared-helper semantics, documents the API, and keeps command authorization hardening unchanged for chat completions.
