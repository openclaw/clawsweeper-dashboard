## Summary

Keep open. Current main and v2026.6.5 still lack REST-native reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related plugin-SDK reset work, closed unmerged REST PR, broader read-only REST tracker, and ACP delegated-session PR do not resolve this focused Gateway HTTP request.

## What This Changes

Expose authenticated REST endpoints so HTTP API integrations can reset, inspect, compact, and change model/reasoning state for a stable OpenClaw session key.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface over existing `sessions.*` semantics, preserve slash-command default-deny behavior on chat completions, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no dedicated `/v1/sessions/*` REST management routes, `/v1/chat/completions` builds normal agent input without command authorization, and the default-off admin HTTP RPC plugin does not allowlist the relevant `sessions.*` methods.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations, not re-enabling slash-command interception or duplicating session business logic.
