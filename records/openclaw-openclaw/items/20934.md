## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints, while the existing Gateway session RPC methods and schemas provide the underlying operations; the prior REST PR was closed unmerged and the plugin-SDK reset work did not add HTTP endpoints.

## What This Changes

The issue requests authenticated REST endpoints for HTTP clients to reset, inspect, compact, and update model/reasoning settings for sessions.

## Best Possible Solution

Add a focused authenticated Gateway HTTP module that maps session reset, status, compact, model, and reasoning requests onto existing session RPC/schema semantics and documents the public contract.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main does not register the requested /v1/sessions/* paths, unmatched HTTP requests fall through to 404, and /v1/chat/completions still sends slash-like text through normal agent ingress without CommandAuthorized.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema behavior is the narrowest maintainable solution; re-enabling slash-command interception in /v1/chat/completions would weaken the current security boundary and is not the requested fix.
