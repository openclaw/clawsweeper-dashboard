## Summary

Keep open. Current main still lacks a supported HTTP route for session management: chat completions intentionally treat unauthorized slash commands as text, and the newer admin HTTP RPC plugin explicitly excludes all `sessions.*` methods. The prior REST PR was closed unmerged, the plugin reset proposal did not provide HTTP access, and the broader read-only REST issue does not own these mutation operations.

## What This Changes

Requests authenticated HTTP endpoints for integrations to reset, inspect, compact, and change per-session model or reasoning state.

## Best Possible Solution

Expose a deliberately reviewed subset of canonical Gateway session methods through an opt-in HTTP control plane, reusing existing schemas, handlers, scope checks, and error shapes while preserving default-deny slash-command behavior in `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. The current generic HTTP RPC plugin rejects all `sessions.*` methods, while the OpenAI-compatible chat route intentionally does not authorize slash-command dispatch.

## Solution Assessment

No, five independent REST handlers are not clearly the best implementation. A scoped HTTP mapping to the existing strict Gateway session methods would avoid duplicating business logic, but maintainers must choose whether that belongs in the full-operator admin plugin or a narrower contract.

- Exposing these methods through the existing admin plugin would deliberately broaden its destructive full-operator surface and must remain opt-in and private.
