## Summary

Keep open. Current `main` still lacks the requested REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, while the underlying Gateway session RPC methods and schemas already exist. The closed REST PR was not merged, the plugin-SDK reset discussion did not add this HTTP surface, and recent maintainer/bot comments explicitly kept this as the focused tracker rather than folding it into the broader read-only REST request.

## What This Changes

The issue requests authenticated REST endpoints for HTTP integrations to reset, inspect, compact, and adjust OpenClaw sessions by key.

## Best Possible Solution

Add a focused Gateway HTTP session-management module that maps authenticated REST requests onto existing session RPC/helper semantics for reset, status, compact, model, and reasoning, with route/auth/request/response tests and public docs. Keep `/v1/chat/completions` slash-command handling default-deny, and coordinate with but do not collapse this work into #47826.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current HTTP routing never registers the requested `/v1/sessions/*` management paths, unmatched requests fall through to 404, and `/v1/chat/completions` still sends slash-like text through the normal agent path without `CommandAuthorized`.

## Solution Assessment

Yes. The narrowest maintainable solution is a small authenticated REST mapping over the existing session methods and schemas; re-enabling command interception in the OpenAI-compatible chat endpoint would weaken the current security boundary and is not the requested fix.
