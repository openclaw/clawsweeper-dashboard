## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; existing WebSocket/RPC methods, adjacent session kill/history HTTP helpers, and the opt-in admin HTTP RPC plugin do not cover this focused request.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update sessions without re-enabling slash-command handling on `/v1/chat/completions`.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface over existing `sessions.*` semantics, preserve slash-command default-deny behavior, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no dedicated REST session-management routes, `/v1/chat/completions` builds normal agent input without command authorization, and the default-off admin HTTP RPC plugin excludes the relevant `sessions.*` methods.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations, not re-enabling slash-command interception or duplicating session business logic.
