## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; related work is adjacent or closed unmerged, and the exact authenticated control-plane API shape needs maintainer approval.

## What This Changes

Request authenticated REST endpoints so HTTP API clients can reset, inspect, compact, and update Gateway sessions without re-enabling slash-command interception.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations, while preserving default-deny slash-command behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested `/v1/sessions/*` routes and handler are absent, while the HTTP router falls through to 404 for unmatched paths.

## Solution Assessment

Yes directionally, but the exact public API contract needs maintainer approval. The maintainable path is an authenticated REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`.
