## Summary

Keep open. Current main still has no REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the remaining work is a public Gateway control-plane feature that needs maintainer product and security approval.

## What This Changes

Requests authenticated REST session-management endpoints for HTTP API clients to reset, inspect, compact, and change model/reasoning for keyed Gateway sessions.

## Best Possible Solution

Define and document a minimal authenticated Gateway HTTP session-control surface that wraps existing session RPC/schema behavior, preserves default-deny slash-command handling, and includes route, auth, schema, and docs coverage.

## Reproduction Assessment

Yes, at source level. Current main has no requested /v1/sessions/* route helper or request stage, and unmatched HTTP paths fall through to 404 while the corresponding operations remain RPC-only.

## Solution Assessment

Yes directionally, but not as an automatic repair. The maintainable path is a REST wrapper over existing session RPC/schema behavior, not command interception in /v1/chat/completions, with maintainer approval for the public API and auth contract first.
