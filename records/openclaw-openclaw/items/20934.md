## Summary

Keep this issue open. Current `main` and latest release still lack REST-native session reset/status/compact/model/reasoning endpoints, and the related plugin-SDK work plus the closed unmerged REST attempt do not resolve the requested HTTP API surface.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update OpenClaw sessions without re-enabling slash-command handling in `/v1/chat/completions`.

## Best Possible Solution

A maintainer-approved, authenticated Gateway HTTP session-control API should map narrowly to the existing session RPC semantics, keep `/v1/chat/completions` slash commands default-denied, add route/auth/request/response coverage, and update the public OpenAI-compatible HTTP docs.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: the HTTP router has no `/v1/sessions/*` management handlers, while the requested session operations exist only as Gateway RPC methods and adjacent non-`/v1` HTTP helpers.

## Solution Assessment

Unclear until maintainers choose the public API shape. The likely maintainable direction is a narrow authenticated HTTP mapping to existing session RPC/helper semantics, but this adds operator-control surface and should get product/security approval first.
