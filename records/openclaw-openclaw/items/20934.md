## Summary

Keep open. Current main still lacks the requested authenticated REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, while the existing Gateway RPC/session schemas already cover the underlying operations. The prior REST PR was closed unmerged, the related plugin-SDK discussion did not add this HTTP surface, and the issue remains a focused Gateway API gap rather than a duplicate of the broader read-only REST request.

## What This Changes

Request authenticated REST session-management endpoints so HTTP API clients using `/v1/chat/completions` can reset, inspect, compact, and adjust stable sessions without sending slash commands as chat text.

## Best Possible Solution

Add a focused Gateway HTTP session-management module that maps authenticated REST requests onto existing session RPC/helper semantics for reset, status, compact, model, and reasoning, with route/auth/request/response tests and public docs. Keep `/v1/chat/completions` slash-command handling default-deny, and coordinate with #47826 without collapsing this narrower write-capable session-control tracker into that broader read-only REST request.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current HTTP routing never registers the requested `/v1/sessions/*` management paths, unmatched requests fall through to 404, and `/v1/chat/completions` still sends slash-like text through the normal agent path without `CommandAuthorized`.

## Solution Assessment

Yes. The narrowest maintainable solution is a small authenticated REST mapping over existing session methods and schemas; re-enabling command interception in the OpenAI-compatible chat endpoint would weaken the current security boundary and is not the requested fix.
