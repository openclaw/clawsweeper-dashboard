## Summary

Keep open. Current main and v2026.5.20 still lack REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; the closed REST PR was not merged, the plugin-SDK work did not add REST endpoints, and the broader open HTTP-query issue does not supersede this focused session-management API request.

## What This Changes

Request REST-native session reset, status, compact, model, and reasoning endpoints for clients using `/v1/chat/completions`.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface wrapping existing `sessions.*` semantics, preserve slash-command default-deny behavior on chat completions, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no `/v1/sessions/*` management route, and `/v1/chat/completions` still builds normal agent chat input without command authorization, so command-like text is not a session-control path.

## Solution Assessment

Unclear until maintainers choose the API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations rather than re-enabling slash-command interception or duplicating session business logic.
