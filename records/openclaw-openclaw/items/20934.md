## Summary

Keep open. Current main and v2026.5.20 still lack REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; the existing WebSocket/RPC methods, adjacent session kill/history helpers, plugin-SDK reset work, and broader read-only HTTP-query issue do not resolve this focused HTTP session-management API request.

## What This Changes

Requests authenticated REST endpoints for HTTP integrations to reset, inspect, compact, and change model/reasoning state for sessions by key instead of sending slash-command text through `/v1/chat/completions`.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface wrapping existing `sessions.*` semantics, preserve slash-command default-deny behavior on chat completions, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no `/v1/sessions/*` management route, and `/v1/chat/completions` still builds normal agent chat input without command authorization.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations rather than re-enabling slash-command interception or duplicating session business logic.
