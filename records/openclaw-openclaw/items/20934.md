## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related closed PRs did not merge a usable HTTP surface.

## What This Changes

The issue requests authenticated REST endpoints that let `/v1/chat/completions` HTTP integrations reset, inspect, compact, and update session model/reasoning state by session key.

## Best Possible Solution

Add a focused, documented, bearer-authenticated Gateway HTTP session-management surface that reuses existing session RPC/schema semantics and keeps `/v1/chat/completions` slash-command default-deny behavior intact.

## Reproduction Assessment

Yes for the reported capability gap. Source inspection on current main shows the requested `/v1/sessions/*` routes are not registered and would fall through to the HTTP 404 tail, while `/v1/chat/completions` still does not authorize slash-command dispatch.

## Solution Assessment

Yes, directionally. A small authenticated REST mapping over existing session RPC/schema operations is the narrow maintainable direction; re-enabling slash-command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
