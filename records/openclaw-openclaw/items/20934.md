## Summary

Keep open. Current main and the latest release still do not expose REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related plugin-SDK/reset and read-only REST items do not replace this tracker.

## What This Changes

The issue requests authenticated REST endpoints for HTTP clients to manage OpenClaw sessions without re-enabling slash-command interpretation on `/v1/chat/completions`.

## Best Possible Solution

Define a focused authenticated Gateway HTTP session-management surface that reuses canonical session RPC schemas/handlers, preserves default-deny slash-command behavior for chat completions, and adds route/auth/request/response tests plus public docs.

## Reproduction Assessment

Yes, at source level. Current main has `/v1/chat/completions` session-key routing and canonical session RPC methods, but the HTTP router, file list, docs, and latest release still lack the requested `/v1/sessions/*` management endpoints.

## Solution Assessment

Yes on direction, but only with a reviewed HTTP control-plane contract. The maintainable implementation should reuse existing Gateway session schemas and handlers rather than interpreting slash commands or duplicating session business logic.
