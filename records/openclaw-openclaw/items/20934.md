## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing Gateway RPC methods and adjacent session kill/history HTTP helpers do not satisfy the requested HTTP control-plane surface.

## What This Changes

Add authenticated REST session-management endpoints for clients using the OpenAI-compatible HTTP API.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations while preserving default-deny slash-command behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. Current main registers no requested `/v1/sessions/*` route or handler and unmatched HTTP paths fall through to 404, while the corresponding session operations are only exposed through Gateway RPC.

## Solution Assessment

Yes directionally, but not as an automatic repair yet. The maintainable path is a REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`, with maintainer approval for the public API and auth contract first.
