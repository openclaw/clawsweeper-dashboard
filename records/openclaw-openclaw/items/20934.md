## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing implementation has WebSocket/RPC session controls and adjacent session kill/history HTTP helpers, but not the requested `/v1/sessions/*` management surface.

## What This Changes

Requested authenticated REST endpoints for HTTP API integrations to reset, inspect, compact, and update session-level model or reasoning state.

## Best Possible Solution

Define and implement a focused authenticated Gateway HTTP session-management surface that reuses existing session RPC/helper semantics and enforces the right operator scopes while preserving default-deny slash-command behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes. Source inspection on current main confirms there is no `/v1/sessions/*` management router or handler, while the OpenAI chat path builds agent commands without `CommandAuthorized`, so `/new` and `/status` over `/v1/chat/completions` remain chat text rather than control commands.

## Solution Assessment

Yes, the requested direction is the right class of solution, but it needs product and security decisions. The clean path is a focused authenticated Gateway HTTP mapping to existing session RPC/helpers, not re-enabling slash command interception in chat completions or collapsing this into read-only REST query work.
