## Summary

Keep open. Current main and v2026.6.10 still do not provide REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; adjacent Gateway RPC methods, session kill/history HTTP helpers, admin HTTP RPC, and plugin reset work do not satisfy this `/v1/sessions/*` request.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update session-level model/reasoning state by key without re-enabling slash-command dispatch in `/v1/chat/completions`.

## Best Possible Solution

Keep this issue open for a focused maintainer-approved Gateway HTTP session-management design that reuses existing typed `sessions.*` handlers and schemas, preserves default-deny slash-command handling, and documents/tests the auth and route contract.

## Reproduction Assessment

Yes, at source level: `/v1/chat/completions` dispatches as normal ingress input without trusted command authorization, and current main has no `/v1/sessions/*` REST alternative for reset/status/compact/model/reasoning.

## Solution Assessment

Unclear until maintainers confirm the product/security contract. The narrow maintainable shape is a thin authenticated adapter over existing Gateway session methods and schemas, not slash-command interpretation or duplicated session logic.
