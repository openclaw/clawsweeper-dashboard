## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP integrations, while the existing slash-command hardening and Gateway RPC/session schemas confirm this is a real HTTP transport/API gap rather than a broken command path.

## What This Changes

The issue requests authenticated REST session-management endpoints for `/v1/chat/completions` clients that need reset, status, compaction, model, and reasoning controls by session key.

## Best Possible Solution

Define a focused authenticated Gateway HTTP session-management surface that maps to existing `sessions.*` schemas/handlers, preserves default-deny slash-command handling for chat completions, and ships with auth/route tests plus public docs.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` have no `/v1/sessions/*` management routes, while `/v1/chat/completions` still sends slash-command-looking input through the normal agent path without setting `CommandAuthorized`.

## Solution Assessment

Yes, with narrowing. A REST-native mapping to existing Gateway session methods is the right layer, but it should be a reviewed control-plane contract rather than five ad hoc handlers.
