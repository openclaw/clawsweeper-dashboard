## Summary

Keep open: current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP integrations, while the existing RPC/session primitives only cover WebSocket/control-plane callers. The closed REST attempt and closed Plugin SDK reset work overlap with the request but do not deliver the requested HTTP API surface, and the broader REST query tracker is read-only rather than a replacement.

## What This Changes

The issue requests authenticated REST session-management endpoints for clients using the OpenAI-compatible `/v1/chat/completions` HTTP API.

## Best Possible Solution

Define a focused authenticated Gateway HTTP session-management API that maps to the existing session RPC/shared helper semantics, documents auth and session-key behavior, adds route/auth/request/response coverage, and keeps slash-command hardening unchanged.

## Reproduction Assessment

Yes. Source inspection of current `main` shows the HTTP router and docs have no `/v1/sessions/*` management endpoints, while the equivalent operations exist only as Gateway RPC/session primitives.

## Solution Assessment

Unclear as a final API design, but the broad direction is correct: add a REST-native authenticated transport mapping rather than re-enabling slash-command interception on `/v1/chat/completions`. Maintainers still need to approve the exact endpoint shape and security model.
