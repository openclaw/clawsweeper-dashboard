## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing session operations are available through Gateway RPC and adjacent HTTP helpers only. The closed unmerged REST attempt at https://github.com/openclaw/openclaw/pull/23941, the closed plugin-SDK reset work at https://github.com/openclaw/openclaw/pull/29985, and the broader read-only REST tracker at https://github.com/openclaw/openclaw/issues/47826 do not supersede this mutation-focused request.

## What This Changes

The issue requests authenticated REST session-management endpoints for HTTP API integrations, covering reset, status, compact, model, and reasoning controls.

## Best Possible Solution

Define and document a minimal authenticated Gateway HTTP session-control surface that wraps the existing session RPC/schema behavior, preserves default-deny slash-command handling, and includes route, auth, schema, docs, and upgrade-safety coverage.

## Reproduction Assessment

Yes, at source level. Current main has no requested /v1/sessions/* route helper or request stage, and unmatched HTTP paths fall through to 404 while the corresponding operations remain RPC-only.

## Solution Assessment

Yes directionally, but not as an automatic repair. The maintainable path is a REST wrapper over existing session RPC/schema behavior, not command interception in /v1/chat/completions, with maintainer approval for the public API and auth contract first.
