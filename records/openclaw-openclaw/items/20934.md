## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints, and the related closed PRs either were not merged or covered a different plugin-SDK/API direction rather than this HTTP control-plane surface.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update session model/reasoning state.

## Best Possible Solution

Define and document a minimal authenticated Gateway HTTP session-control surface that reuses existing session RPC/schema operations with explicit operator scopes, route/auth/schema tests, and default-deny slash-command behavior preserved.

## Reproduction Assessment

Yes, at source level. Current main has no requested `/v1/sessions/*` route helper or request stage, and unmatched HTTP paths fall through to the 404 tail while the corresponding operations are RPC-only.

## Solution Assessment

Yes directionally, but not as an automatic repair. The maintainable path is a REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`, with maintainer approval for the public API and auth contract first.
