## Summary

Keep open. Current main still lacks the requested REST-native session control endpoints for HTTP API integrations, while the existing code only exposes WebSocket/RPC session methods plus adjacent session kill/history HTTP helpers.

## What This Changes

The issue requests authenticated REST Gateway endpoints for HTTP clients to reset, inspect, compact, and update session settings.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations while preserving slash-command default-deny behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested `/v1/sessions/*` routes and handler are absent, while unmatched HTTP paths fall through to 404.

## Solution Assessment

Yes directionally, but the exact public API needs maintainer approval. The narrow maintainable path is an authenticated REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`.
