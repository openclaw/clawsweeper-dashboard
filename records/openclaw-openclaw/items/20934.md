## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, while the related plugin-SDK and REST attempts do not resolve this specific authenticated HTTP control-plane gap.

## What This Changes

The issue requests authenticated REST endpoints so `/v1/chat/completions` HTTP integrations can reset, inspect, compact, and update session model/reasoning settings without re-enabling slash-command dispatch.

## Best Possible Solution

Define one authenticated Gateway HTTP session-management surface that reuses the canonical `sessions.*` handlers and schemas, preserves `/v1/chat/completions` slash-command default-deny behavior, documents the operator-security boundary, and coordinates with the broader read-only REST tracker without collapsing this mutation-focused issue into it.

## Reproduction Assessment

Yes, at source level. Current main registers no requested `/v1/sessions/*` management routes while the underlying Gateway session RPC handlers and schemas exist.

## Solution Assessment

Yes as a tracker, but the exact API shape still needs maintainer product/security judgment. The maintainable fix should map HTTP to canonical `sessions.*` handlers and schemas rather than treating command-like chat text as authorized controls.
