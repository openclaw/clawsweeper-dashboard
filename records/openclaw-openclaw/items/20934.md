## Summary

Keep open. Current main and v2026.6.10 still do not provide REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing Gateway RPC methods, adjacent history/kill HTTP helpers, default-off admin HTTP RPC plugin, and related plugin reset work do not satisfy the requested `/v1/sessions/*` surface.

## What This Changes

The issue requests authenticated REST-native session management endpoints for HTTP API integrations using `/v1/chat/completions`, including reset, status, compact, model, and reasoning controls.

## Best Possible Solution

Define a thin authenticated Gateway HTTP adapter over existing `sessions.*` schemas and handlers, document the route contract, test auth/routes/responses, and preserve default-deny slash-command handling in `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 expose the underlying Gateway session RPC methods but have no `/v1/sessions/*` REST management routes, handler, allowlisted admin RPC alternative, or docs.

## Solution Assessment

Unclear until maintainers confirm the product/security contract. The maintainable shape appears to be a thin authenticated adapter over existing typed Gateway methods, not slash-command interpretation or duplicated session logic.
