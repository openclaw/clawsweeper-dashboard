Keep open. Current main and v2026.6.6 still lack the requested REST-native `/v1/sessions/*` session-management endpoints for HTTP API integrations; existing Gateway RPC methods and the plugin-SDK reset work cover adjacent control paths but do not solve the HTTP transport gap.

Request REST endpoints so `/v1/chat/completions` HTTP clients can reset, inspect, compact, and update model or reasoning state for a stable session key.

## Best Possible Solution

Implement a focused authenticated Gateway HTTP session-management adapter over existing session RPC schemas and handlers, document it, and preserve default-deny slash-command handling for `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 register no `/v1/sessions/*` management routes while the underlying Gateway RPC methods for reset, patch/status-like listing, and compaction exist.

## Solution Assessment

Unclear until maintainers confirm the HTTP control-plane contract. The maintainable direction is a thin authenticated adapter over existing typed Gateway methods, not slash-command interpretation or duplicated session business logic.
