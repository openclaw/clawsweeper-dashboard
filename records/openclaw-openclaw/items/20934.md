## Summary

Keep open. Current main and v2026.5.20 still do not expose REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; the related plugin-SDK reset work and closed REST PR do not resolve this HTTP API surface, and existing admin HTTP RPC is default-off and does not allow session methods.

## What This Changes

The issue requests authenticated REST endpoints for HTTP integrations to reset, inspect, compact, and update session-level model/reasoning state.

## Best Possible Solution

Keep this as the focused tracker for an authenticated Gateway HTTP session-management surface that wraps existing sessions.* semantics, preserves slash-command default-deny behavior on chat completions, and ships with route/auth/schema/docs coverage.

## Reproduction Assessment

Yes, at source level. Current main has no /v1/sessions/* management route, and /v1/chat/completions still builds normal agent chat input without command authorization, so command-like text is not a session-control path.

## Solution Assessment

Unclear until maintainers choose the API boundary. The maintainable direction is a focused authenticated HTTP mapping over existing sessions.* operations rather than re-enabling slash-command interception or duplicating session business logic.
