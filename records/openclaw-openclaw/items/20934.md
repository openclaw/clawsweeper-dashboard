## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints for HTTP integrations; the existing Gateway session RPC/schema support and adjacent HTTP helpers do not expose reset/status/compact/model/reasoning under `/v1/sessions/*`, and the prior REST PR is closed unmerged.

## What This Changes

Request REST endpoints for HTTP API clients to reset, inspect, compact, and update model or reasoning state for sessions keyed through `/v1/chat/completions`.

## Best Possible Solution

Expose a documented `/v1/sessions` management surface that reuses existing session RPC/schema behavior and scope checks while leaving `/v1/chat/completions` slash-command hardening unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows no `/v1/sessions/*` route predicate or stage, so those paths fall to the HTTP 404 tail while `/v1/chat/completions` continues to send slash-like text as normal agent input without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema operations is the narrow maintainable direction; re-enabling slash-command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
