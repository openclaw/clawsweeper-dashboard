## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; related work is adjacent or closed unmerged, and the authenticated control-plane API shape still needs maintainer approval.

## What This Changes

The issue requests authenticated REST endpoints for HTTP clients to reset, inspect, compact, and adjust sessions without re-enabling slash-command interception in `/v1/chat/completions`.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations while preserving default-deny slash-command behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested `/v1/sessions/*` route helpers and handler are absent, and unmatched HTTP paths fall through to 404; I did not start the gateway for this read-only review.

## Solution Assessment

Yes directionally, but not as an automatic repair yet. The maintainable path is an authenticated REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`, and maintainers should approve the public API/auth contract first.
