## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; related plugin-SDK reset and dashboard/session helper work are useful context but do not expose the requested HTTP API surface.

## What This Changes

The request is to add authenticated REST endpoints so HTTP API clients can reset, inspect, compact, and update session model/reasoning state by session key.

## Best Possible Solution

Expose a documented authenticated `/v1/sessions` management surface over the existing session RPC/schema operations while keeping `/v1/chat/completions` slash-command default-deny intact and coordinating read-only overlap with https://github.com/openclaw/openclaw/issues/47826.

## Reproduction Assessment

Yes. Source inspection on current main shows no REST session-management route or handler, so the requested `/v1/sessions/*` management paths fall through to the HTTP 404 tail while chat completions still do not authorize slash commands.

## Solution Assessment

Yes. A small authenticated HTTP wrapper over existing session RPC/schema behavior is the narrow maintainable direction; re-enabling slash-command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
