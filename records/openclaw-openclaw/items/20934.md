## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related plugin-SDK reset work plus the closed unmerged REST PR do not resolve this specific HTTP control-plane surface.

## What This Changes

The issue requests authenticated REST endpoints that let `/v1/chat/completions` HTTP clients reset, inspect, compact, and update session model/reasoning state by session key.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations while preserving default-deny slash-command behavior for `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. Current main registers no requested `/v1/sessions/*` route or handler and unmatched HTTP paths fall through to 404, while the corresponding session RPC operations exist only behind the Gateway method surface.

## Solution Assessment

Yes directionally, but not as an automatic repair yet. The maintainable approach is a REST wrapper over existing session RPC/schema behavior, not command interception in `/v1/chat/completions`, with maintainer approval for the public API and auth contract first.
