## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related closed PRs either were not merged or covered a different plugin-SDK path rather than this HTTP control-plane surface.

## What This Changes

Request to add authenticated REST session-management endpoints for HTTP API integrations to reset, inspect, compact, and change model/reasoning state by session key.

## Best Possible Solution

Define a small documented authenticated Gateway HTTP session-management API over the existing session RPC/schema operations while preserving default-deny slash-command behavior for /v1/chat/completions.

## Reproduction Assessment

Yes, at source level. Current main registers no requested /v1/sessions/* route or handler, so those paths fall through to 404 while the corresponding session operations are only exposed through Gateway RPC.

## Solution Assessment

Yes directionally, but not as an automatic repair yet. The maintainable path is a REST wrapper over existing session RPC/schema behavior, not command interception in /v1/chat/completions, with explicit maintainer approval for the public API and auth contract first.
