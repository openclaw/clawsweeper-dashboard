## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; related plugin-SDK and closed REST PR work does not resolve this focused HTTP API request, and exposing these controls over HTTP needs maintainer product and security approval.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update session settings.

## Best Possible Solution

Build a narrow authenticated Gateway HTTP session-management surface over existing sessions.* operations, with explicit scope/auth semantics, strict request/response schemas, docs, and tests, while keeping /v1/chat/completions slash-command handling default-deny.

## Reproduction Assessment

Yes, at source level. Current main has no /v1/sessions/* management route, and /v1/chat/completions still builds normal agent chat input without command authorization, so command-like text is not a session-control path.

## Solution Assessment

Unclear until maintainers choose the API boundary. The maintainable direction is a focused authenticated HTTP mapping over existing sessions.* operations rather than re-enabling slash-command interception or duplicating session business logic.
