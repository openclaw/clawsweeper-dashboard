## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, and the related closed PRs did not land that HTTP surface.

## What This Changes

Requests authenticated REST endpoints that let HTTP API clients manage OpenClaw sessions by key.

## Best Possible Solution

Expose a documented authenticated `/v1/sessions` management surface over the existing session RPC/schema behavior while keeping `/v1/chat/completions` slash-command default-deny intact and coordinating status/read-only shape with https://github.com/openclaw/openclaw/issues/47826.

## Reproduction Assessment

Yes. Source inspection on current main shows no REST session-management route or handler, so `/v1/sessions/*` management paths fall through to 404 while `/v1/chat/completions` still builds agent input without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated HTTP transport over the existing session RPC/schema operations is the narrow maintainable direction; re-enabling command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
