## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints for HTTP integrations, while the underlying Gateway session RPC methods and schemas already exist; the prior REST PR was closed unmerged and the broader REST-query tracker does not replace this mutation-focused request.

## What This Changes

Add REST-native HTTP session-management endpoints for `/v1/chat/completions` integrations to reset, inspect, compact, and update per-session model/reasoning state.

## Best Possible Solution

Add a focused authenticated Gateway HTTP module that maps reset, status, compact, model, and reasoning requests onto existing session RPC/schema behavior, with route/auth tests and public docs.

## Reproduction Assessment

Yes. Source inspection on current main shows no `/v1/sessions/*` route predicate or stage, so those paths fall to the 404 tail; `/v1/chat/completions` still sends slash-like text as normal agent input without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema operations is the narrowest maintainable direction; re-enabling slash-command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
