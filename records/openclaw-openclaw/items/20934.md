## Summary

Keep open. Current main and v2026.6.10 still do not expose REST-native reset/status/compact/model/reasoning endpoints for HTTP API integrations; the available pieces are WebSocket/RPC session methods, narrower session history/kill HTTP helpers, and a default-off admin HTTP RPC plugin whose allow-list omits `sessions.*`.

## What This Changes

The issue requests authenticated REST endpoints so `/v1/chat/completions` integrations can manage stable OpenClaw sessions without re-enabling slash-command interpretation.

## Best Possible Solution

Define a thin authenticated Gateway HTTP adapter over existing `sessions.*` schemas and handlers, document the route contract, test auth/routes/responses, and keep `/v1/chat/completions` slash-command handling default-deny.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 have the underlying Gateway session RPC methods but no `/v1/sessions/*` REST management routes, handler, allowlisted admin RPC alternative, or docs.

## Solution Assessment

Unclear until maintainers confirm the product and security contract. The maintainable direction is a thin authenticated adapter over existing typed Gateway methods, not slash-command interpretation or duplicated session business logic.
