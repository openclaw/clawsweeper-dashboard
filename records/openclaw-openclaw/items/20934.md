## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; existing WebSocket/RPC session methods, adjacent session history/kill HTTP helpers, the default-off admin HTTP RPC plugin, and the broader read-only REST issue do not resolve this focused `/v1` session-management request.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API integrations to reset, inspect, compact, and adjust sessions without re-enabling slash-command dispatch on chat completions.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface wrapping existing `sessions.*` semantics, preserve slash-command default-deny behavior on chat completions, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no dedicated REST session-management routes, `/v1/chat/completions` still builds normal agent input without command authorization, and the existing admin HTTP RPC plugin is default-off and excludes the relevant session methods.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations, not re-enabling slash-command interception or duplicating session business logic.
