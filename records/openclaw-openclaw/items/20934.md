## Summary

Keep open. Current `main` still lacks the requested REST-native session-management surface, while the Gateway already has the underlying session RPC methods and schemas needed to implement it. The related REST PR was closed without merging, the plugin-SDK reset discussion did not add HTTP endpoints, and recent maintainer/bot review context explicitly kept this as the focused tracker rather than folding it into the broader read-only REST request.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API integrations to reset, inspect, compact, and update OpenClaw sessions without re-enabling slash-command handling in `/v1/chat/completions`.

## Best Possible Solution

Add a focused Gateway HTTP session-management module that maps authenticated REST requests onto existing `sessions.*` method/helper semantics for reset, status, compact, model, and reasoning, updates the public OpenAI-compatible HTTP docs, and keeps `/v1/chat/completions` slash-command handling default-deny.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current HTTP routing never registers the requested `/v1/sessions/*` paths, unmatched paths return 404, and `/v1/chat/completions` still sends slash-like text through normal agent ingress without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema behavior is the narrowest maintainable fix; re-enabling command interception in the OpenAI-compatible chat endpoint would weaken the current security boundary and is not the requested path.
