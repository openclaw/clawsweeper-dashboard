## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; the existing session RPCs and plugin SDK reset path are useful adjacent work but do not provide the requested HTTP API surface.

## What This Changes

The issue requests authenticated REST endpoints so `/v1/chat/completions` HTTP integrations can reset, inspect, compact, and adjust session-level model/reasoning state by session key.

## Best Possible Solution

Keep this as the focused tracker for an authenticated Gateway HTTP session-management surface that wraps existing `sessions.*` semantics, preserves slash-command default-deny behavior on chat completions, and ships with route, auth, schema, and docs coverage.

## Reproduction Assessment

Yes, at source level. Current main has no `/v1/sessions/*` management route, and `/v1/chat/completions` still builds normal agent chat input without command authorization.

## Solution Assessment

Unclear until maintainers choose the API boundary. The maintainable direction is a focused authenticated HTTP mapping over existing `sessions.*` operations rather than re-enabling slash-command interception or duplicating session business logic.
