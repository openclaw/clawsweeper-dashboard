## Summary

Current main and v2026.6.5 still lack HTTP-accessible session reset, compact, keyed status, model, and reasoning controls. The newer default-off admin HTTP RPC plugin provides a narrower transport seam, but session methods are intentionally absent from its allowlist; the closed unmerged REST PR and unrelated ACP work do not resolve this request. Likely related people include clay-datacurve, Vincent Koc, and steipete based on adjacent session-management and Gateway HTTP history.

## What This Changes

Add authenticated HTTP session-management operations for reset, status, compaction, model selection, and reasoning settings without enabling slash-command dispatch in chat completions.

## Best Possible Solution

Approve a trusted-operator session-management contract that reuses existing Gateway schemas and handlers—preferably by deliberately allowlisting narrowly selected session RPC methods in the default-off admin HTTP RPC plugin—while defining keyed status output, authorization scope, audit behavior, and public documentation.

## Reproduction Assessment

Yes. Current main and v2026.6.5 expose the underlying session RPC methods but neither the core HTTP router nor the admin HTTP RPC allowlist exposes the requested HTTP session controls.

## Solution Assessment

No, the original proposal of five parallel core routes is not yet established as the best design. The narrower maintainable direction is to reuse the default-off trusted admin HTTP RPC seam and existing strict session handlers unless maintainers deliberately choose a separate resource-style REST contract.
