## Summary

Keep open. Current main and v2026.6.8 still do not provide REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing Gateway RPC methods, adjacent session history/kill HTTP helpers, and default-off admin HTTP RPC plugin do not satisfy the requested `/v1/sessions/*` surface.

## What This Changes

The issue requests authenticated REST session-management endpoints for HTTP clients using `/v1/chat/completions` with stable session keys.

## Best Possible Solution

Define a thin authenticated Gateway HTTP adapter over the existing `sessions.*` schemas and handlers, document the route contract, test auth/routes/responses, and preserve default-deny slash-command behavior in `/v1/chat/completions`.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 have the underlying Gateway session RPC methods but no `/v1/sessions/*` REST management routes, handler, allowlisted admin RPC alternative, or docs.

## Solution Assessment

Unclear until maintainers confirm the product/security contract. The maintainable solution is a thin authenticated adapter over existing typed Gateway methods, not slash-command interpretation or duplicated session business logic.
