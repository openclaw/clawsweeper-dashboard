## Summary

Keep open. Current main and v2026.6.6 still do not expose REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; existing Gateway RPC methods, adjacent HTTP helpers, and closed related PRs do not solve the requested authenticated HTTP surface.

## What This Changes

Add bearer-authenticated REST session-management endpoints for HTTP clients using stable `/v1/chat/completions` session keys.

## Best Possible Solution

Define a thin authenticated Gateway HTTP adapter over existing `sessions.*` schemas and handlers, document it, test auth/routes/responses, and keep slash commands default-denied in chat completions.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 have the underlying Gateway session RPC methods but no `/v1/sessions/*` REST management routes or docs.

## Solution Assessment

Unclear until maintainers confirm the HTTP control-plane contract. The maintainable direction is a thin authenticated adapter over existing typed Gateway methods, not slash-command interpretation or duplicated session business logic.
