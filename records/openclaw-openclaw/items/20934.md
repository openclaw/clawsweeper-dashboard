## Summary

Keep open. Current main still lacks REST-native reset/status/compact/model/reasoning endpoints for HTTP integrations; existing WebSocket/RPC session methods, adjacent session history/kill helpers, the default-off admin HTTP RPC plugin, and the broader read-only REST tracker do not resolve this focused request.

## What This Changes

Request a REST-native Gateway HTTP surface for HTTP integrations to reset, inspect, compact, and update session model/reasoning state without re-enabling slash-command interception.

## Best Possible Solution

Add a focused authenticated Gateway HTTP session-management surface over existing sessions.* semantics, preserve slash-command default-deny behavior on chat completions, and ship route/auth/schema/docs coverage after maintainer product and security approval.

## Reproduction Assessment

Yes, at source level. Current main has no dedicated REST session-management routes, /v1/chat/completions builds normal agent input without command authorization, and the default-off admin HTTP RPC plugin excludes the relevant sessions.* methods.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a focused authenticated HTTP mapping over existing sessions.* operations, not re-enabling slash-command interception or duplicating session business logic.
