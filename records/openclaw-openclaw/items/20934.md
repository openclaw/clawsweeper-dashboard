## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints for HTTP integrations, while the underlying Gateway session RPC methods and schemas already exist; the related REST PR was closed unmerged and the plugin-SDK reset work did not add the HTTP surface.

## What This Changes

Add authenticated REST endpoints so HTTP API clients can reset, inspect, compact, and update model or reasoning settings for sessions without re-enabling slash-command handling in /v1/chat/completions.

## Best Possible Solution

Add a focused, authenticated Gateway HTTP module that maps reset, status, compact, model, and reasoning requests onto existing session RPC/schema behavior, with route/auth/request/response tests and public OpenAI-compatible HTTP docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main does not register the requested /v1/sessions/* routes, those unmatched requests fall through to 404, and /v1/chat/completions still treats slash-like text as ordinary agent input without CommandAuthorized.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema operations is the narrowest maintainable direction; re-enabling slash-command interception in /v1/chat/completions would cross the current security boundary and is not the requested fix.
