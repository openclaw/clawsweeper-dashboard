## Summary

Keep open. Current main and the latest release still lack REST-native session reset/status/compact/model/reasoning endpoints for `/v1/chat/completions` integrations; the existing Gateway session RPC handlers and schemas cover the operations, but only WebSocket/RPC and adjacent history/kill HTTP helpers are exposed today.

## What This Changes

The issue requests authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update model/reasoning state for sessions by key.

## Best Possible Solution

Add a maintainer-approved authenticated Gateway HTTP session-management surface that reuses the existing session RPC handlers/schemas, preserves `/v1/chat/completions` slash-command default-deny behavior, and updates focused route/auth/docs tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `/v1/chat/completions` accepts a stable session key, but the router has no `/v1/sessions/*` management endpoints and the HTTP command path still defaults slash-command authorization to false.

## Solution Assessment

Yes on direction, with maintainer approval still needed. The maintainable fix is a focused HTTP adapter over existing Gateway session methods and schemas, not re-enabling slash-command interpretation or duplicating session business logic.
