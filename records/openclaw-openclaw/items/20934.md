## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations; the existing session RPC/schema primitives and adjacent session history/kill HTTP helpers do not expose the requested `/v1/sessions/*` management surface.

## What This Changes

Add authenticated REST endpoints for HTTP API clients to reset, inspect, compact, and update model/reasoning settings for a session key.

## Best Possible Solution

Expose a documented authenticated `/v1/sessions` management surface over existing session RPC/schema semantics while preserving `/v1/chat/completions` slash-command default-deny behavior and coordinating naming with the broader read-only REST tracker.

## Reproduction Assessment

Yes. Source inspection on current main shows no REST session-management route or handler, so the requested `/v1/sessions/*` paths fall through to the HTTP 404 tail while `/v1/chat/completions` still builds agent input without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated HTTP transport over the existing session RPC/schema operations is the narrow maintainable direction; re-enabling command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/20934 as a focused Gateway HTTP API change. Add authenticated REST-native session-management endpoints for HTTP API integrations, covering reset, status, compact, model override, and reasoning override for a session key, preferably under `/v1/sessions/*` as requested. Reuse existing session RPC/shared semantics (`sessions.reset`, `sessions.compact`, `sessions.patch`, and an appropriate status/describe/list helper) instead of duplicating business logic. Preserve `/v1/chat/completions` slash-command default-deny behavior; do not set `CommandAuthorized` or make `/new`/`/status` text dispatch as commands in the chat endpoint. Enforce Gateway HTTP auth and existing operator scopes: read-only scope for status, admin/mutating scope for reset/compact/model/reasoning. Add strict path matching so `/v1/sessionsXYZ` is not handled, validate reset `reason` to allowed values, and do not pass a non-schema `key` field into `sessions.list`. Update public OpenAI-compatible HTTP docs and add a changelog entry because this is a user-facing API feature. Do not change unrelated session lifecycle policy or add a broad REST API beyond this issue; coordinate naming with https://github.com/openclaw/openclaw/issues/47826 but keep this implementation focused.
