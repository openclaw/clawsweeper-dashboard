## Summary

Keep open. Current main still lacks the requested REST-native session reset/status/compact/model/reasoning endpoints for HTTP API integrations, while the underlying Gateway session RPC methods and schemas already exist. The linked REST PR was closed unmerged, the plugin-SDK discussion did not add this HTTP surface, and the recent maintainer comment explicitly kept this as the focused tracker rather than folding it into the broader read-only REST issue.

## What This Changes

The issue requests authenticated REST session-management endpoints for HTTP API integrations, covering reset, status, compaction, model selection, and reasoning controls for sessions used by `/v1/chat/completions`.

## Best Possible Solution

Add a focused Gateway HTTP session-management module that maps authenticated REST requests onto the existing session RPC/helper semantics for reset, status, compact, model, and reasoning, with public docs and tests. Keep slash-command handling on `/v1/chat/completions` default-deny and coordinate with, but do not collapse into, the broader read-only REST endpoint request.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current HTTP routing never registers the requested `/v1/sessions/*` paths and unmatched requests fall through to 404, while `/v1/chat/completions` still sends slash-like text through the agent path without `CommandAuthorized`.

## Solution Assessment

Yes. The narrowest maintainable solution is a small authenticated REST mapping over the existing session methods and schemas, not re-enabling command interception in the OpenAI-compatible chat endpoint. The implementation should reuse existing HTTP auth/scope helpers and existing session handlers where their schemas fit.
