## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints, while the related plugin-SDK reset work and closed REST PR do not expose reset/status/compact/model/reasoning for HTTP API integrations.

## What This Changes

Add authenticated REST endpoints for HTTP integrations to reset, inspect, compact, and update model/reasoning settings for stable Gateway sessions.

## Best Possible Solution

Expose a documented authenticated `/v1/sessions` management surface that reuses existing Gateway session RPC/schema behavior, keeps `/v1/chat/completions` slash-command default-deny intact, and coordinates the status/read-only shape with https://github.com/openclaw/openclaw/issues/47826.

## Reproduction Assessment

Yes. Source inspection on current main shows no `/v1/sessions/*` route predicate or request stage, so those paths fall through to the HTTP 404 tail while `/v1/chat/completions` still builds agent input without `CommandAuthorized`.

## Solution Assessment

Yes. A small authenticated REST mapping over existing session RPC/schema operations is the narrow maintainable direction; re-enabling slash-command interception in `/v1/chat/completions` would cross the current security boundary and is not the requested fix.
