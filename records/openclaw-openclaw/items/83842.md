## Summary

Keep open: the Discord opt-out is focused and has sufficient live-output proof, but the patch drops `replyTo` only inside the Discord handler after the shared send path has already resolved and consumed first-reply state.

## What This Changes

The PR adds a Discord-only `useReply` message-tool schema option and makes Discord send handlers omit native reply references when `useReply: false` is passed.

## Best Possible Solution

Make the reply opt-out visible before shared reply resolution consumes first-mode state, keep default Discord replies unchanged, and add a regression covering an opt-out send followed by a normal first-mode send.

## Reproduction Assessment

Yes. Current main forwards resolved `replyTo` unconditionally in the Discord send handlers, and the remaining PR blocker is source-reproducible because shared reply resolution consumes first-mode state before plugin dispatch.

## Solution Assessment

No. The Discord-local suppression is the right send boundary, but it needs a companion change before shared reply-state consumption so the new option does not corrupt first-mode reply behavior.

- **[P2] Do not consume first-mode replies for opt-out sends:** `extensions/discord/src/actions/handle-action.ts:111-112`
