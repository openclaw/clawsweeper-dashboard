## Summary

Keep open. The PR targets a valid cron `timeoutSeconds` watchdog bug, but the diff appears to ignore configured CLI watchdog `maxMs` caps for cron runs and is missing the required user-facing changelog entry.

## What This Changes

The PR passes cron trigger context into CLI watchdog resolution, lets cron timeouts raise CLI and embedded idle watchdog budgets, and adds focused regression tests.

## Best Possible Solution

Keep the PR open, preserve explicit backend watchdog caps while lifting only the default cron watchdog budget, add a changelog entry, then rerun the focused agent and cron tests plus the changed gate.

## Reproduction Assessment

Yes, from source inspection. A cron CLI resume run with `timeoutMs: 600000` currently resolves through the 180s resume ceiling, and an embedded cron run with a longer explicit timeout currently caps the idle watchdog at the default idle timeout.

## Solution Assessment

No, not as written. The direction is right, but the CLI change should only lift default watchdog ceilings and must continue honoring explicit `reliability.watchdog.*.maxMs` configuration; it also needs the required changelog entry.
