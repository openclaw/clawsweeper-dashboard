## Summary

Current main accepts the documented Telegram `richMessages` key in the strict config schema and covers both top-level and account override parsing; the matching fix came from https://github.com/openclaw/openclaw/pull/93279 and is not included in `v2026.6.8`.

## What This Changes

The issue reports that `channels.telegram.richMessages` is documented but rejected by the installed `v2026.6.8` schema, causing gateway startup failure after adding the key.

## Best Possible Solution

Keep the current main schema/test/runtime behavior and ship it in the next release so users on `v2026.6.8` can move to a build where the documented key is accepted.

## Reproduction Assessment

Yes, from source: `v2026.6.8` has a strict Telegram schema without `richMessages`, while current main includes the key and account-override coverage. I did not run the gateway because this review is read-only, and current main no longer has the failing path.

## Solution Assessment

Yes: adding `richMessages` to the shared Telegram account schema is the narrow owner-boundary fix because both top-level and account schemas derive from that base. The remaining practical step is release uptake, not another code change on main.
