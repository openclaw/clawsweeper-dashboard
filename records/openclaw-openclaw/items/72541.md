## Summary

Keep open. Current main still has the reported false-negative shape: the incomplete-turn classifier can see that `sessions_spawn` was a mutating tool, but it cannot see that the tool returned `status: "accepted"` with a child session, so a pure relay that spawns and stops with no visible payload still reaches the warning path.

## What This Changes

The issue asks the gateway incomplete-turn check to treat a successful terminal `sessions_spawn` call from a pure-relay agent as completion instead of surfacing an empty-response warning.

## Best Possible Solution

Carry structured terminal tool-success metadata from tool execution into the embedded runner, recognize a non-error `sessions_spawn` result with `status: "accepted"` and a non-empty `childSessionKey` as completion, and keep the existing warning behavior for failed spawns, malformed results, and truly empty turns.

## Reproduction Assessment

Yes. A focused unit test can construct `payloadCount: 0`, `assistantTexts: []`, `lastAssistant.stopReason: "stop"`, and `toolMetas: [{ toolName: "sessions_spawn" }]` with current replay metadata; current main returns the side-effect incomplete-turn warning because no accepted-spawn result is available to classify as terminal success.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the successful spawn result into runner completion metadata and classify that specific accepted terminal action as success, rather than disabling empty-turn detection globally or requiring relay agents to emit acknowledgement text.
