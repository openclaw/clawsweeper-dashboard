## Summary

Keep open. Current main still has the same classifier gap: accepted `sessions_spawn` results are recorded only as generic tool metadata, so a zero-payload terminal assistant turn can still fall through to the incomplete-turn warning path.

## What This Changes

The issue asks the gateway to treat a zero-payload turn ending in a successful `sessions_spawn` as complete instead of surfacing the “Agent couldn't generate a response” warning.

## Best Possible Solution

Preserve a structured terminal-success marker for accepted `sessions_spawn` calls and suppress the incomplete-turn warning only when that marker proves a child run was created.

## Reproduction Assessment

Yes. A focused unit test can model payload count `0`, no visible assistant text, `stopReason: "stop"`, and a completed accepted `sessions_spawn`; current main has no accepted-spawn signal available to the classifier, so it can still return the warning.

## Solution Assessment

Yes. A narrow accepted-spawn exemption backed by `status`, `runId`, and `childSessionKey` is more maintainable than a relay-agent acknowledgement workaround or a broad empty-turn bypass.
