## Summary

Keep open. Current main still lacks a structured accepted-`sessions_spawn` success signal in the incomplete-turn classifier path, so the reported zero-payload pure-relay false-negative remains valid.

## What This Changes

The issue asks the gateway completeness check to treat a successful terminal `sessions_spawn` acceptance as a complete pure-relay turn even when no text payload is emitted.

## Best Possible Solution

Preserve an accepted-spawn terminal-success record in attempt metadata and suppress the incomplete-turn warning only when `status: "accepted"`, `childSessionKey`, and `runId` prove a child run was created.

## Reproduction Assessment

Yes. A focused unit test can build a zero-payload terminal assistant turn after an accepted `sessions_spawn`; current main has no accepted-spawn signal in the classifier path, so it can still reach the warning branch.

## Solution Assessment

Yes. The narrow accepted-spawn exemption is more maintainable than a relay-agent acknowledgement workaround or a broad empty-turn bypass because it relies on structured tool-result proof.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72541. Current main treats a zero-payload terminal turn after a successful `sessions_spawn` as incomplete because the accepted spawn result is not preserved for `resolveIncompleteTurnPayloadText`. Add a narrow structured terminal-success path: when `sessions_spawn` completes successfully with `status: "accepted"`, a non-empty `childSessionKey`, and a non-empty `runId`, preserve that fact in attempt metadata and suppress the incomplete-turn warning for that attempt. Do not add a broad empty-turn bypass, do not rely on assistant acknowledgement text, and do not make this core logic depend on a bundled agent id. Add focused coverage in the incomplete-turn tests and tool handling/spawn tests as needed. Add a single-line `CHANGELOG.md` fix entry for the user-facing gateway warning behavior.
