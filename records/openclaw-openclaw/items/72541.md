## Summary

Keep open. Current main still has the reported false-negative shape: an accepted `sessions_spawn` is recorded only as mutating tool activity, while the incomplete-turn classifier has no structured success exemption for `status: "accepted"` plus `childSessionKey`, so a pure relay can still produce the side-effect incomplete-turn warning.

## What This Changes

The issue requests that the gateway treat a zero-payload pure relay turn ending in a successful `sessions_spawn` as complete instead of surfacing an incomplete-response warning.

## Best Possible Solution

Carry structured successful terminal tool metadata from `sessions_spawn` through the subscriber attempt state into the runner classifier, accept only `status: "accepted"` with a non-empty `childSessionKey` as terminal completion, and add focused coverage for accepted, failed, and malformed spawn results while preserving existing empty-turn warnings.

## Reproduction Assessment

Yes. A focused unit test can construct `payloadCount: 0`, `assistantTexts: []`, `lastAssistant.stopReason: "stop"`, and replay metadata from a current-attempt `sessions_spawn`; current main still resolves that to the side-effect incomplete-turn warning instead of `null`.

## Solution Assessment

Yes. The issue's first-class behavior direction is the narrow maintainable fix: propagate the actual accepted spawn result to the classifier, rather than disabling empty-turn detection globally or requiring relay agents to emit acknowledgement text.
