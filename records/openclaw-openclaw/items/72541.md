## Summary

Keep open. Current main still classifies zero-payload terminal stops using payload, silent-reply, and committed messaging-delivery evidence only; successful `sessions_spawn` result fields are not preserved for that classifier, so the reported false-negative remains actionable.

## What This Changes

The issue asks the gateway to treat a successful terminal `sessions_spawn` as a completed zero-payload turn rather than warning that the agent produced no response.

## Best Possible Solution

Preserve a structured terminal-success signal for accepted `sessions_spawn` calls and suppress the incomplete-turn warning only when that signal proves a child run was created.

## Reproduction Assessment

Yes. A focused unit test can construct a zero-payload `stop` turn after a successful `sessions_spawn`; current main only carries `toolName/meta`, so the classifier still reaches the empty-response warning path.

## Solution Assessment

Yes. The narrow maintainable fix is an accepted-spawn exemption backed by structured status, `runId`, and `childSessionKey`, not a broad empty-turn bypass or a relay-agent acknowledgement requirement.

## Repair Work Prompt

Fix the gateway/embedded-runner false incomplete-turn warning for pure relay agents that successfully call `sessions_spawn` and then stop with no visible text. Preserve structured evidence from successful `sessions_spawn` tool results, requiring `status: "accepted"` plus a non-empty `childSessionKey` and `runId`, and have `resolveIncompleteTurnPayloadText` suppress the warning only for that proven terminal success. Do not broadly suppress zero-payload turns, do not treat failed/forbidden spawn results as success, and do not add owner-specific relay-agent policy. Add focused regression coverage for accepted spawn, failed spawn, generic mutating tool, and existing committed messaging delivery behavior. Add a user-facing changelog fix entry; do not change dependency pins or unrelated subagent completion semantics.
