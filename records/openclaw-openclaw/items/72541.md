## Summary

Keep open. Current main still lacks a successful `sessions_spawn` terminal-success signal in the incomplete-turn classifier; the accepted spawn is reduced to generic tool metadata, so a zero-payload `stop` turn after a child run starts can still surface the false incomplete-turn warning.

## What This Changes

The issue requests that a zero-payload parent turn ending after an accepted `sessions_spawn` be treated as successful instead of incomplete.

## Best Possible Solution

Preserve accepted child-run facts through tool completion and suppress incomplete-turn output only when a successful `sessions_spawn` proves a child run was created.

## Reproduction Assessment

Yes, at source level. A current-main attempt with `payloadCount: 0`, no assistant text, `stopReason: "stop"`, and a successful `sessions_spawn` only reaches the classifier as generic `toolMetas`, so the empty-response warning path can still fire.

## Solution Assessment

Yes. The narrow maintainable fix is to carry structured accepted-spawn metadata into attempt classification and bypass only that proven terminal-success case, rather than adding a broad empty-turn bypass or a new config option.
