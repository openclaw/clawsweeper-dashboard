## Summary

Keep open. Current main still lacks a structured accepted-`sessions_spawn` terminal-success signal in the incomplete-turn classifier, so the source-level false-negative described by the issue remains plausible and narrowly fixable.

## What This Changes

The issue asks OpenClaw to treat a successful terminal `sessions_spawn` relay as a completed turn even when the parent emits no visible text.

## Best Possible Solution

Preserve a structured accepted-spawn terminal-success marker through tool completion and the attempt result, then suppress the incomplete-turn warning only when that marker proves a child run was created.

## Reproduction Assessment

Yes, at source level: model a zero-payload stop turn after an accepted `sessions_spawn`; current main has no accepted-spawn result field in the attempt shape the classifier receives, so the empty-response/incomplete-turn path can still fire. I did not run live ACP dispatch or tests because this is a read-only cleanup review.

## Solution Assessment

Yes. The narrow maintainable fix is to carry accepted spawn facts into the attempt result and exempt only proven child-run creation, rather than adding a broad empty-turn bypass or a new config flag.

## Repair Work Prompt

Fix the gateway/embedded-runner false incomplete-turn warning for pure relay agents that successfully call `sessions_spawn` and then emit no visible text. Preserve a structured terminal-success marker only when `sessions_spawn` returns `status: "accepted"` with non-empty `childSessionKey` and `runId`; carry that fact from tool completion through the subscribe state and `EmbeddedRunAttemptResult` into `resolveIncompleteTurnPayloadText`. Suppress the incomplete-turn warning only for that proven accepted-spawn terminal action; do not broadly silence empty assistant turns, do not add a config flag, and keep existing guards for tool errors, aborted/timed-out runs, silent replies, committed messaging delivery, cron silent tool results, and normal empty-response retries. Add focused regression coverage for accepted spawn success and for rejected/error or malformed spawn results still surfacing the warning. Include a changelog entry if the user-visible warning behavior changes.
