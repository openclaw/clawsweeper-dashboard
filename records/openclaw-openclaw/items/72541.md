## Summary

Keep open. Current main still has the reported false-negative shape: a zero-payload, non-error assistant stop after `sessions_spawn` reaches the incomplete-turn classifier with only generic mutating-tool metadata, so an accepted child session cannot be recognized as terminal success.

## What This Changes

The issue asks OpenClaw to treat a successful `sessions_spawn` result as terminal completion for pure-relay agents instead of surfacing the incomplete-turn warning when no text payload is emitted.

## Best Possible Solution

Add structured current-attempt metadata for successful terminal tool actions, starting with `sessions_spawn` accepted results, and make the incomplete-turn classifier accept `status: "accepted"` with a non-empty `childSessionKey` as completion while preserving warnings for failed spawns and truly empty turns.

## Reproduction Assessment

Yes. A focused unit test can construct `payloadCount: 0`, `assistantTexts: []`, a non-error `stop`, `toolMetas: [{ toolName: "sessions_spawn" }]`, and replay metadata from `buildAttemptReplayMetadata`; current main returns the side-effect incomplete-turn warning.

## Solution Assessment

Yes. The narrowest maintainable solution is to propagate the actual successful `sessions_spawn` result into runner completion metadata and classify that as terminal success, rather than disabling empty-turn detection globally or requiring relay agents to emit acknowledgement text.

Current main flags a pure-relay turn as incomplete when it ends with zero text after a successful `sessions_spawn`, because `toolMetas` only preserve `{ toolName, meta }` and `resolveIncompleteTurnPayloadText` cannot see the accepted spawn result.
