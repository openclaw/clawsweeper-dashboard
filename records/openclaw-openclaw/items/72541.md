## Summary

Keep open. Current main still has the classifier gap: a zero-payload terminal assistant turn can be classified as an empty response, while the successful `sessions_spawn` result is not preserved in the attempt shape that the completeness check can inspect.

## What This Changes

The issue requests that the gateway treat a successful zero-payload `sessions_spawn` terminal turn as complete instead of surfacing the incomplete-turn warning.

## Best Possible Solution

Preserve a structured accepted-spawn terminal-success marker through tool completion and suppress the incomplete-turn warning only when that marker proves a child run was created.

## Reproduction Assessment

Yes, at source level: model `payloadCount: 0`, no visible assistant text, `stopReason: "stop"`, and a successful accepted `sessions_spawn`; current main has no accepted-spawn fields available to the classifier, so the empty-response warning path can still fire. I did not run tests or a live ACP dispatch because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to carry `sessions_spawn` success facts into the attempt result and exempt only accepted child-run creation, rather than adding a broad empty-turn bypass or a new config flag.
