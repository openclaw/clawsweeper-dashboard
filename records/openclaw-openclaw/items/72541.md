## Summary

Keep open. Current `main` still has the reported false-negative shape: a zero-payload, non-error assistant stop after an accepted `sessions_spawn` is classified as an incomplete turn because the classifier only receives generic tool metadata and replay side-effect state, not the spawn result fields that would prove terminal success.

## What This Changes

The issue asks for the gateway incomplete-turn check to treat a successful `sessions_spawn` result as a valid terminal action for pure-relay agents, instead of surfacing the empty-response warning.

## Best Possible Solution

Propagate structured current-attempt metadata for successful terminal tool actions, starting with accepted `sessions_spawn` results, and make the incomplete-turn classifier accept `status: "accepted"` with a non-empty `childSessionKey` as completion while preserving warnings for failed spawns and truly empty turns.

## Reproduction Assessment

Yes. A focused unit test can construct a zero-payload non-error assistant stop with `toolMetas: [{ toolName: "sessions_spawn" }]` and replay metadata from `buildAttemptReplayMetadata`; current main will return the side-effect incomplete-turn warning because no accepted-spawn result metadata reaches the classifier.

## Solution Assessment

Yes. The narrowest maintainable fix is to retain the successful `sessions_spawn` result shape in runner attempt metadata and classify that as terminal success, rather than disabling empty-turn detection globally or requiring relay agents to emit acknowledgement text.
