## Summary

Keep open. Current main still classifies a zero-payload terminal turn after `sessions_spawn` through the generic incomplete-response path, and the runner does not preserve accepted spawn status or child-session identity for the classifier.

## What This Changes

The issue asks the gateway completeness check to treat a terminal successful `sessions_spawn` as a completed turn even when no assistant text payload is emitted.

## Best Possible Solution

Preserve a typed terminal-success marker for accepted `sessions_spawn` calls and have the incomplete-turn classifier suppress the warning only when that marker proves a child session was created.

## Reproduction Assessment

Yes. A focused unit test can construct a zero-payload stop turn after a successful `sessions_spawn`; current main still lacks the result fields needed for the classifier to distinguish that from an empty failed turn.

## Solution Assessment

Yes. The narrow maintainable fix is to carry accepted-spawn result metadata into the runner and exempt only proven child-session creation, rather than broadly allowing empty tool-use stops or relying on relay agents to emit acknowledgements.
