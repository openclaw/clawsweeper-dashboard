## Summary

Keep open. The report is fresh and plausible, and current main still has multiple gateway status/channel probe paths where local token-auth behavior depends on documented credential precedence and device-token fallback; I did not find a shipped or main-only fix that clearly eliminates this `gateway:channels.status` token-mismatch case.

## What This Changes

Report asks OpenClaw to stop recurring `gateway:channels.status` WebSocket `token_mismatch` auth failures during loopback token-auth status/channel probes.

## Best Possible Solution

Add a targeted regression or diagnostic for status/channel live probes under local token auth when ambient env, configured token, and cached device-token state disagree, then either surface auth drift clearly or make the internal follow-up probe reuse the proven-safe auth path without relaxing shared-secret security.

Unclear. The issue gives concrete logs and setup steps, but current source review suggests a high-confidence reproduction needs the missing effective env/service auth state or cached device-token state that would cause `channels.status` to send a mismatched token.

## Solution Assessment
