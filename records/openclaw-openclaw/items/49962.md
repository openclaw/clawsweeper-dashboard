## Summary

Keep this PR open: current main still has the reported group-policy bug, and the proposed resolver change is narrow, but the PR still lacks required real Telegram behavior proof and leaves user-facing group allowlist docs inconsistent with the new access-control semantics.

## What This Changes

The PR changes group-policy resolution so explicit `groupPolicy: "open"` keeps group entries as per-group overrides instead of treating them as an allowlist, and adds resolver regression tests.

## Best Possible Solution

Land the resolver fix with focused regression coverage, update the group-policy docs to distinguish explicit `open` from implicit allowlist behavior, and require redacted real Telegram or terminal proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: explicit `groupPolicy: "open"` plus any configured `groups` entry makes the resolver reject an unlisted group, and Telegram maps that rejection to `This group is not allowed.`

## Solution Assessment

Partially. The shared resolver condition is the narrow maintainable fix, but the PR should also align the user-facing docs and include after-fix real behavior proof.
