## Summary

Keep open. The patch targets a real, source-reproducible access-control bug and matches the documented `groupPolicy: "open"` contract, but this external PR still needs real Telegram/gateway behavior proof and fresh validation before merge.

## What This Changes

This PR changes group-policy resolution so explicit `groupPolicy: "open"` prevents `groups` entries from enabling a chat allowlist, with regression tests for channel-level and account-level configs.

## Best Possible Solution

Land the narrow resolver fix with the added regression coverage after real runtime proof and fresh green validation; clarify docs only if maintainers want to call out explicit open policy with per-group overrides.

## Reproduction Assessment

Yes, source-level. On current main, an explicit `groupPolicy: "open"` plus nonempty `groups` makes `allowlistEnabled` true, and an unlisted group is rejected because it has no matching `groupConfig`.

## Solution Assessment

Yes. The proposed condition is the narrowest maintainable fix because it preserves explicit `allowlist`, `disabled`, and backward-compatible implicit allowlist behavior when `groupPolicy` is unset.
