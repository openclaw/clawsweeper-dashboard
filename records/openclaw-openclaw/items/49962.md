## Summary

Keep open. The PR addresses behavior that still exists on current main, but it changes group admission semantics in an access-control path and still needs the public docs/security contract reconciled before merge.

## What This Changes

The PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents nonempty `groups` entries from enabling chat allowlist mode, and adds channel-level and account-level regression tests.

## Best Possible Solution

Align code, tests, docs, and security guidance around one explicit contract: either accept explicit-open groups as overrides-only and update the public group-policy guidance, or keep the current fail-closed interpretation and document wildcard groups as the allow-all path.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: current main sets `allowlistEnabled` from `hasGroups`, then Telegram turns an unlisted chat denial into the reported `This group is not allowed.` response.

## Solution Assessment

No, not as submitted. The predicate change is the narrow code fix if maintainers choose overrides-only semantics, but the best complete fix also updates the operator-facing docs and security guidance for the widened admission behavior.
