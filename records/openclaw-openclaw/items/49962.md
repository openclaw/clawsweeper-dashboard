## Summary

Keep open: the PR targets a real source-reproducible group-policy bug, but it still lacks contributor real behavior proof and leaves user-facing group-policy docs inconsistent with the access-control change.

## What This Changes

The PR changes `resolveChannelGroupPolicy` so explicit `groupPolicy: "open"` with configured `groups` does not enable the implicit group allowlist, and adds two resolver tests.

## Best Possible Solution

Land the resolver fix with docs updates that clearly distinguish explicit `open` from implicit allowlist behavior, plus redacted real Telegram or terminal proof.

## Reproduction Assessment

Yes from source inspection, not a live run. Current main enables the group allowlist whenever `groups` exists, so explicit `groupPolicy: "open"` plus an unlisted group reaches Telegram's `This group is not allowed.` rejection path.

## Solution Assessment

Partially. The shared resolver condition is the narrow maintainable fix, but this PR should also update the user-facing group-policy docs and include after-fix real behavior proof before merge.
