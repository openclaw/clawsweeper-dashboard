## Summary

Keep open. Current main still reproduces the reported behavior, and the PR is a useful narrow code change, but it changes shared access-control semantics without reconciling the current Telegram/WhatsApp/group/security guidance that treats configured group entries as admission allowlists.

## What This Changes

The PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents non-empty `groups` entries from enabling allowlist mode, and adds channel-level and account-level regression tests.

## Best Possible Solution

Make an explicit maintainer contract decision, then land the code only with matching shared docs, Telegram/WhatsApp group-policy guidance, security/audit wording, and a required changelog entry; if fail-closed groups-as-admission is intentional, reject the code change and clarify that contract instead.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: with explicit `groupPolicy: "open"` and a non-empty `groups` map, current main sets `allowlistEnabled` true, leaves an unlisted group without `groupConfig`, and Telegram maps that denial to `This group is not allowed.`

## Solution Assessment

No, not as submitted. The predicate change is the narrow code fix if maintainers want explicit `open` to override group admission, but the complete solution must reconcile operator-facing access-control docs, security guidance, and changelog expectations.
