## Summary

Keep open: the current-main bug path is source-reproducible, but this PR is not ready as-is because it changes shared group admission semantics while channel docs and security warnings still describe configured groups as allowlists.

## What This Changes

The PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents non-empty `groups` entries from enabling chat allowlist mode, and adds channel-level and account-level resolver tests.

## Best Possible Solution

Make the explicit-open contract consistent across the resolver, shared/channel docs, security warnings, and changelog, or deliberately retain the current fail-closed groups-as-allowlist behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `groupPolicy: "open"` plus a non-empty `groups` map sets `allowlistEnabled=true`; an unlisted group resolves `allowed=false`, and Telegram maps that denial to the reported message.

## Solution Assessment

No, not as submitted. The predicate is the narrow code change if maintainers want explicit `open` to override route allowlists, but the complete fix needs matching docs, security-warning wording, and changelog coverage before merge.
