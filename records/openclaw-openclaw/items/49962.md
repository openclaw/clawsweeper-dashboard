## Summary

Keep this PR open, but not merge-ready: the code change is a plausible narrow fix for the reported behavior, yet it changes shared access-control semantics while current Telegram, WhatsApp, security-warning, and shared group docs still describe configured `groups` entries as route allowlists.

## What This Changes

The PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents non-empty `groups` entries from enabling allowlist mode, and adds channel-level and account-level regression tests.

## Best Possible Solution

Make an explicit maintainer contract decision, then either land this predicate with matching shared docs, Telegram/WhatsApp guidance, security-warning wording, and changelog coverage, or keep the fail-closed behavior and clarify the docs accordingly.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: explicit `groupPolicy: "open"` plus a non-empty `groups` map sets `allowlistEnabled` true, an unlisted group resolves `allowed=false`, and Telegram maps that denial to `This group is not allowed.`

## Solution Assessment

No, not as submitted. The predicate is the narrow code change if maintainers want explicit `open` to override route allowlists, but the complete fix must update the shared operator contract, security guidance, and changelog before merge.
