## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap: the Telegram plugin computes the configured cap and the direct send path passes it to media loading, but final reply and native-command reply delivery still call `loadWebMedia` without a configured `maxBytes`; the two direct fix PRs are closed unmerged.

## What This Changes

Telegram final replies with outbound media should honor `channels.telegram.mediaMaxMb` by passing the computed media byte cap into reply media loading.

## Best Possible Solution

Thread the already-computed Telegram media cap through message processing and native-command delivery into `deliverReplies` and `deliverMediaReply`, then pass it as `maxBytes` with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: configure Telegram `mediaMaxMb`, trigger a final model or native-command reply with `mediaUrl` or `mediaUrls`, and the reply path reaches `loadWebMedia` without `maxBytes`.

## Solution Assessment

Yes. Threading the existing computed `mediaMaxBytes` through the Telegram reply pipeline is the narrowest maintainable fix; it does not need a new config option, default change, or product decision.
