## Summary

Keep open. Current main still has the reported Telegram reply-media cap propagation gap: the configured cap is computed and direct sends honor it, but reply delivery paths still call media loading without passing `maxBytes`, and both related implementation PRs are closed unmerged.

## What This Changes

The issue requests that Telegram final replies with outbound media honor the configured `channels.telegram.mediaMaxMb` limit.

## Best Possible Solution

Land a narrow Telegram plugin fix that carries the computed `mediaMaxBytes` into message and native-command reply delivery, passes it as `maxBytes` to `buildOutboundMediaLoadOptions`, and adds focused regression tests for each reply path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: configure Telegram `mediaMaxMb`, trigger a reply-media path that reaches `deliverReplies`, and the source shows `loadWebMedia` is called through options that omit `maxBytes`.

## Solution Assessment

Yes. Threading the already-computed `mediaMaxBytes` value through the Telegram reply pipeline is the narrowest maintainable fix; no new config key or core API decision is needed.

Implement the narrow parameter threading only: add optional `mediaMaxBytes` where needed, pass it from bot setup through normal reply dispatch and native-command reply delivery, and call `buildOutboundMediaLoadOptions({ mediaLocalRoots, maxBytes: mediaMaxBytes })` for reply media. Do not change Telegram defaults, direct-send behavior, media loading internals, durable delivery policy, or unrelated transport logic. Include a one-line `CHANGELOG.md` fix entry if this user-visible fix still requires one under current policy.
