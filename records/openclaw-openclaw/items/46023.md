## Label Justifications

- `P2`: This is a bounded Telegram plugin delivery bug with a clear source path and limited blast radius to outbound media replies.
- `impact:message-loss`: The bug can prevent Telegram final media replies from being delivered even when the configured media cap should allow them.

## Summary

Keep open. Current main and the latest release still have the Telegram reply-media cap gap, and https://github.com/openclaw/openclaw/pull/83478 is an open closing PR for the same fix, so this issue should remain open until that PR lands or is closed.

## What This Changes

Telegram final and native replies with media should pass the configured `channels.telegram.mediaMaxMb` cap into outbound media loading.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83478 or an equivalent narrow Telegram plugin fix that threads `mediaMaxBytes` through final and native reply delivery into `buildOutboundMediaLoadOptions({ maxBytes })`, with regression coverage and preserved proxy option plumbing.

## Reproduction Assessment

Yes, at source level. Configure `channels.telegram.mediaMaxMb`, trigger a Telegram final or native reply with `mediaUrl`/`mediaUrls`, and current main reaches `loadWebMedia` through `buildOutboundMediaLoadOptions` without `maxBytes`.

## Solution Assessment

Yes. Threading the already-computed `mediaMaxBytes` through the existing Telegram reply delivery options is the narrowest maintainable fix, and the open closing PR is the right place to review it.
