## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap: the configured cap is computed and direct sends use it, but final reply and native-command reply delivery still call media loading without `maxBytes`; the two related fix PRs are closed unmerged.

## What This Changes

Telegram final replies containing media should honor configured `channels.telegram.mediaMaxMb` by passing the computed byte cap into outbound media loading.

## Best Possible Solution

Land a narrow Telegram plugin fix that carries the computed media cap through message processing and native-command delivery into `deliverReplies`/`deliverMediaReply`, passes it as `maxBytes`, and locks the behavior with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure Telegram `mediaMaxMb`, trigger a final model or native-command reply with `mediaUrl`/`mediaUrls`, and the reply path reaches `loadWebMedia` without `maxBytes`. I did not run a live Telegram bot smoke in this read-only review.

## Solution Assessment

Yes. Threading the already-computed `mediaMaxBytes` through the existing Telegram reply pipeline is the narrowest maintainable fix; no new config, default change, or product policy is needed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/46023 with a narrow Telegram plugin change. Current main computes `mediaMaxBytes` in `extensions/telegram/src/bot-core.ts`, but final model replies and native-command replies do not pass that cap into `deliverReplies`, `deliverMediaReply`, or `buildOutboundMediaLoadOptions`, so `loadWebMedia` runs without `maxBytes`. Thread the computed cap through `createTelegramMessageProcessor`, `dispatchTelegramMessage`, `registerTelegramNativeCommands`/command delivery base options, `deliverReplies`, and `deliverMediaReply`, then pass `maxBytes: params.mediaMaxBytes` when building outbound media load options. Add focused regression tests proving configured `mediaMaxMb` reaches `loadWebMedia` for final media replies and native-command reply delivery; preserve current defaults when no cap is configured. Do not change config keys/defaults, direct send behavior, media loader internals, or unrelated Telegram delivery policy. Add a one-line `CHANGELOG.md` Fixes entry because this is user-visible.
