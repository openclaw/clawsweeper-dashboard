## Summary

Keep open. Current main and the latest release still show the Telegram reply-media cap gap, and an open implementation PR now explicitly targets this issue with closing syntax, so the issue should stay open until that PR is reviewed and resolved.

## What This Changes

The issue asks for Telegram final and native-command media replies to honor the configured `mediaMaxMb` limit in outbound media loading.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/83478 against the current Telegram reply delivery paths and close this issue through that PR if the parameter-threading fix and regression tests are accepted.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure `channels.telegram.mediaMaxMb`, trigger a final or native-command Telegram reply with `mediaUrl` or `mediaUrls`, and current main reaches `loadWebMedia` through `buildOutboundMediaLoadOptions` without `maxBytes`.

## Solution Assessment

Yes. Threading the already-computed `mediaMaxBytes` through the existing Telegram reply delivery option chain is the narrowest maintainable fix, and the open linked PR is the right place to review that implementation.
