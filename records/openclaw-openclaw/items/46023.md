## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap: the configured cap is computed and direct sends pass it to media loading, but final reply delivery still reaches `loadWebMedia` without `maxBytes`, and both related implementation PRs are closed unmerged.

## What This Changes

Telegram final replies with media should honor the configured `channels.telegram.mediaMaxMb` limit in the outbound delivery path.

## Best Possible Solution

Land a narrow Telegram plugin fix that carries `mediaMaxBytes` through bot setup, message dispatch, native-command delivery, `deliverReplies`, and `deliverMediaReply`, then passes it as `maxBytes` to `buildOutboundMediaLoadOptions` with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure Telegram `mediaMaxMb`, trigger a final reply with `mediaUrl` or `mediaUrls`, and the current reply path reaches `loadWebMedia` without a `maxBytes` option. I did not run a live Telegram bot smoke in this read-only review.

## Solution Assessment

Yes. Threading the already-computed byte cap through the existing Telegram reply pipeline is the narrowest maintainable fix; no new config or product policy is needed.

## Repair Work Prompt

Fix the Telegram reply-media `mediaMaxMb` bug from https://github.com/openclaw/openclaw/issues/46023. Current main computes `mediaMaxBytes` in `extensions/telegram/src/bot-core.ts`, but normal final replies and native-command reply delivery do not pass it into `deliverReplies`, `deliverMediaReply`, or `loadWebMedia`. Thread the existing computed cap through `createTelegramMessageProcessor`, `dispatchTelegramMessage` delivery base options, `registerTelegramNativeCommands`, `deliverReplies`, and `deliverMediaReply`, then pass `maxBytes: params.mediaMaxBytes` into `buildOutboundMediaLoadOptions` while preserving existing defaults when unset. Use https://github.com/openclaw/openclaw/pull/74490 and https://github.com/openclaw/openclaw/pull/46425 as references, but port carefully to current main instead of applying either patch mechanically. Add or update focused tests in the Telegram delivery, dispatch, native-command, and bot creation suites proving configured `mediaMaxMb` reaches `loadWebMedia` for final media replies and native-command replies. Do not change the Telegram direct send path except if a type update requires it; it already passes `maxBytes`. Add a one-line `CHANGELOG.md` fix entry because this is user-visible.
