## Summary

Keep open. Current main and the latest release still document and compute Telegram mediaMaxMb, but the final-reply and native-command reply delivery paths do not pass any media cap into outbound media loading; the two direct fix PRs are closed unmerged.

## What This Changes

The issue asks for Telegram final/native media replies to honor the configured mediaMaxMb limit during outbound delivery.

## Best Possible Solution

Land a narrow Telegram plugin fix that carries the existing mediaMaxBytes value through final-reply and native-command delivery into deliverMediaReply and buildOutboundMediaLoadOptions, with regression tests for the affected reply paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure Telegram mediaMaxMb, trigger a final or native-command reply with mediaUrl/mediaUrls, and the current reply path reaches loadWebMedia through buildOutboundMediaLoadOptions without maxBytes.

## Solution Assessment

Yes. Threading the already-computed mediaMaxBytes through the existing reply delivery option chain is the narrowest maintainable fix and does not require a new setting, default, or product policy.
