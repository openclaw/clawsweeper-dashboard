## Summary

Keep open. Current main now routes ordinary final Telegram media replies through durable outbound delivery when that path is supported, but the underlying `deliverReplies` media path still builds outbound media load options without `maxBytes`, and native-command reply delivery still uses that path directly.

## What This Changes

The issue reports that Telegram final reply media delivery does not consistently honor configured `mediaMaxMb` because the computed byte cap is not propagated into outbound reply media loading.

## Best Possible Solution

Keep the durable final-reply path, but also pass the configured Telegram media byte cap into the remaining `deliverReplies` media loader paths so every outbound Telegram reply surface follows the documented limit.

## Reproduction Assessment

Yes. Source inspection on current main shows a clear remaining path: native-command replies and durable-final fallback routes can still call `deliverReplies`, which reaches `loadWebMedia` through `buildOutboundMediaLoadOptions` without `maxBytes`.

## Solution Assessment

Yes. Threading the already-computed `mediaMaxBytes` through the Telegram reply-delivery options is the narrowest maintainable fix and does not require a new setting or policy change.
