## Summary

Keep open. Current main has the intended per-channel Slack session-key plumbing and session-keyed active-run lookup, but there is still no deterministic current-main proof that a slow Slack Socket Mode plus Bedrock `modelByChannel` turn cannot starve another Slack channel or the shared event loop.

## What This Changes

The issue reports that a long Bedrock-backed Slack Socket Mode turn in one channel leaves the shared `main` session processing and blocks other Slack channels until restart.

## Best Possible Solution

Add a deterministic Slack cross-channel isolation regression, then fix any missing session-key propagation or active-run lookup it exposes while leaving provider timeout policy to the separate tracker.

## Reproduction Assessment

Unclear for current main. The report has concrete v2026.4.27 steps, logs, and rollback evidence, but I did not find a current-main Slack Socket Mode plus Bedrock/modelByChannel test or live proof that reproduces or disproves cross-channel starvation.

## Solution Assessment

Unclear. The narrow maintainable path is to prove and preserve per-channel session/lane isolation; adding broad Slack timeout knobs or stuck-session recovery would not directly prove the reported channel-blocking regression.
