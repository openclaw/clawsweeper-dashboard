## What This Changes

The issue reports that a long-running Bedrock-backed Slack Socket Mode turn in one channel can leave the shared `main` session processing and stop other Slack channels from responding in v2026.4.27 through v2026.4.29.

## Best Possible Solution

Keep the issue open until current main has a focused regression proving that a slow Slack channel turn cannot block another Slack channel, then fix the narrow generic boundary that fails: session-key/lane isolation if both messages share `main`, or snapshot/threading reuse if provider/plugin preparation still blocks the gateway event loop.

Yes for the reported v2026.4.27-v2026.4.29 range because the issue gives setup, steps, logs, and a rollback comparison; unclear for current main because I found partial fixes but no live Slack/Bedrock run or deterministic cross-channel starvation test.

## Solution Assessment
