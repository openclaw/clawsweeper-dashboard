## Summary

Keep open. The report is concrete and current `main` has the intended Slack session-key and reply-run isolation plumbing, but I did not find a deterministic Slack Socket Mode plus Bedrock `modelByChannel` regression test or live proof showing that one slow channel run cannot still starve another.

## What This Changes

The issue reports that a long Bedrock call in one Slack channel leaves the shared `main` session processing and blocks other Slack channels until restart.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix any missing session-key propagation or active-run lookup exposed by that test.

## Reproduction Assessment

Yes for the reported v2026.4.27 to v2026.4.29 setup because the issue includes concrete steps, logs, and rollback evidence. Unclear for current `main` because there is no Slack/Bedrock cross-channel regression test or live proof in the inspected tree.

## Solution Assessment

Unclear until current `main` is covered by a deterministic regression. The narrow maintainable fix should stay in Slack session-key propagation or reply-run lane isolation rather than broad stuck-session recovery or provider-timeout redesign.
