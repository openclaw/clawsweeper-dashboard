## Summary

Keep open. The report is concrete and still not safely closeable: current main has the intended per-Slack-channel session-key and queue plumbing plus post-release plugin metadata snapshot reuse, but I did not find direct proof that the reported Slack Socket Mode plus Bedrock `modelByChannel` starvation no longer reproduces, and the overlapping performance fixes are not in the latest shipped v2026.4.29 release.

## What This Changes

A long Bedrock-backed turn in one Slack channel should not leave a shared processing session stuck or stop other Slack channels from responding when `modelByChannel` is configured.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix any missing session-key propagation or metadata-snapshot threading uncovered by that test. Keep timeout and provider error-surfacing work separate unless the reproduction proves it is the same root cause.
