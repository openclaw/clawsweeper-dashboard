## Summary

Keep open. Current main has per-channel Slack session-key plumbing and newer Socket Mode transport tuning, but I did not find a deterministic current-main proof or merged fix showing that a slow Bedrock `modelByChannel` turn can no longer starve other Slack channels.

## What This Changes

The issue reports that a long Bedrock call in one Slack channel on v2026.4.27 leaves the shared main session processing and blocks other Slack channels until gateway restart.

## Best Possible Solution

Add a focused current-main Slack cross-channel isolation regression around Socket Mode, `modelByChannel`, and long-running provider calls, then fix any missing session-key or active-run lane propagation it exposes.
