## Summary

Keep open. The report is concrete and recent; current main has Slack per-channel session-key propagation and reply-queue lookup plumbing, but I did not find deterministic or live current-main proof that a long Slack Socket Mode Bedrock run with `modelByChannel` cannot starve another Slack channel.

## What This Changes

A Slack Socket Mode Bedrock run in one channel should not leave shared `sessionId=main` stuck in `processing` or block replies in other Slack channels.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix the narrow Slack session-key or reply-queue propagation gap exposed by that test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported v2026.4.27 to v2026.4.29 setup because the issue gives concrete steps, logs, and rollback evidence; unclear for current main because there is no Slack Socket Mode plus Bedrock regression or live proof in the tree.

## Solution Assessment

Is this the best way to solve the issue? Unclear until current main has a deterministic regression; the maintainable path is channel-scoped session/lane isolation, not a broad gateway restart, stuck-session recovery, or upstream timeout redesign.
