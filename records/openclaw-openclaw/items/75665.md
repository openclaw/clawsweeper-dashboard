## Summary

Keep open. The report is concrete and current main has relevant per-channel session-key and runtime-cache plumbing, but there is still no deterministic Slack Socket Mode plus Bedrock `modelByChannel` regression proof showing that one slow channel cannot starve another; the newest related fixes also have not shipped in `v2026.4.29`.

## What This Changes

A long Bedrock turn in one Slack channel should not leave shared `sessionId=main` processing or block other Slack channels when `modelByChannel` overrides are used.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix any missing session-key propagation or metadata-snapshot reuse that the test exposes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported `v2026.4.27` to `v2026.4.29` range because the issue includes setup, steps, logs, and rollback evidence; unclear for current main because the repo lacks a Slack/Bedrock cross-channel regression test or live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until a current-main reproduction exists; the narrow maintainable path is session-key/lane isolation plus request-scoped metadata reuse, not a broad stuck-session recovery or upstream-timeout policy change.
