## Summary

Keep open. Current main has the intended Slack per-channel session-key plumbing, but the embedded runner still nests every normal session lane under the singleton default global lane, so a long active run can still serialize unrelated channel sessions before they reach provider execution. The report remains concrete and actionable rather than stale or implemented.

## What This Changes

The issue reports that in v2026.4.27-v2026.4.29 a long Bedrock run in one Slack channel leaves the shared `main` session processing and blocks responses in other Slack channels.

## Best Possible Solution

Add a deterministic regression proving a slow active run in one Slack channel session does not prevent another Slack channel session from starting or replying, then narrow the embedded-runner lane policy so normal channel/user runs serialize by session key without being gated by a singleton default global lane. Keep provider timeout/error-surfacing work in #75648 separate unless the regression test proves the same root cause.

## Reproduction Assessment

Yes. The issue gives a live reproduction for the reported release range, and current main has a high-confidence source-level reproduction path: two embedded runs with different `sessionKey` values still nest under the same default global lane, whose default concurrency is one.

## Solution Assessment

No, current main is not the best solution yet. The maintainable fix is a generic embedded-runner/session-lane isolation change with focused Slack/channel regression coverage, not a Slack-specific transport workaround or broad stuck-session recovery policy.

Start with the generic embedded-runner/command-queue boundary: `runEmbeddedPiAgent` currently serializes per-session work through `resolveGlobalLane(params.lane)`, which defaults to the singleton main lane.
