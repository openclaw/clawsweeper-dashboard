## Summary

Keep open. The report is concrete and current main has relevant per-channel session-key, active-run queue, and plugin-runtime hot-path fixes, but I did not find a deterministic Slack Socket Mode plus Bedrock `modelByChannel` regression proof showing that one slow channel cannot still starve another.

## What This Changes

The issue reports that a long Bedrock-backed Slack channel run can leave the shared `main` session processing and block other Slack channels.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix the smallest missing session-key propagation or runtime-cache reuse path exposed by that test.

## Reproduction Assessment

Yes for the reported v2026.4.27 to v2026.4.29 range because the issue includes setup, steps, logs, and rollback evidence. Unclear for current main because there is no Slack/Bedrock cross-channel regression test or live proof in the repository.

## Solution Assessment

Unclear until current main has a deterministic reproduction. The narrow maintainable direction is per-channel session/lane isolation plus request-scoped metadata/runtime reuse, not a broad stuck-session recovery or provider-timeout policy change.
