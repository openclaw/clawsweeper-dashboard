## Summary

Keep open. The report is concrete and current main still is not safely closeable: Slack now has per-channel session-key plumbing, but there is no live Slack Socket Mode plus Bedrock proof or regression test showing a slow run in one channel cannot hold another channel behind shared active-session state.

## What This Changes

The issue reports a regression where a long Bedrock-backed Slack Socket Mode run in one channel leaves `sessionId=main` processing and stops replies in other Slack channels when `modelByChannel` is used.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix any missing session-key propagation or active-run lookup boundary uncovered by that test. Keep provider timeout and Retry-After handling in #75648 unless the reproduction proves it is the same root cause.

## Reproduction Assessment

Unclear for current main. The original report gives a high-confidence release-range reproduction, but current main lacks a live Slack/Bedrock run or deterministic unit test proving whether the two-channel stall still reproduces.

## Solution Assessment

Unclear until a current-main regression exists. The narrow maintainable direction is to keep queue and active-run ownership keyed by resolved session key for channel sessions, not to add Slack-specific recovery policy or a broad stuck-session abort change.
