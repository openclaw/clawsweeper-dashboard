## Summary

Keep open. Current main has per-channel Slack session-key plumbing and adjacent event-loop-blocking fixes, but I did not find deterministic current-main proof that a slow Bedrock/modelByChannel Slack turn cannot starve another Slack channel or the shared event loop.

The issue reports that a long-running Bedrock turn in one Slack channel leaves the shared session stuck and blocks responses in other Slack channels.

## Best Possible Solution

Add a deterministic Slack cross-channel isolation regression, then fix any missing session-key propagation or active-run lookup it exposes while leaving provider timeout policy to #75648.
