## Summary

Keep open. Current main has the intended per-channel Slack session-key plumbing and session-keyed active-run lookup, but there is still no current-main proof that a slow Slack Socket Mode plus Bedrock `modelByChannel` turn cannot starve another Slack channel or the shared event loop.

## What This Changes

The issue reports a v2026.4.27 Slack Socket Mode regression where one long Bedrock/modelByChannel turn leaves `sessionId=main` processing and stops other Slack channels from responding.

## Best Possible Solution

Add a deterministic Slack two-channel isolation regression, then fix any missing session-key propagation or active-run lookup it exposes while leaving Slack keepalive and provider timeout policy to their existing focused trackers.

## Reproduction Assessment

Unclear. The v2026.4.27 report is concrete and source inspection shows the intended current-main paths, but there is no current-main Slack Socket Mode plus Bedrock/modelByChannel replay or test that reproduces or disproves the channel-blocking behavior.

## Solution Assessment

Unclear. The narrow maintainable path is to prove per-channel session/lane isolation with a regression test; broad timeout knobs or Slack worker-thread keepalive changes would not directly prove this reported session-blocking bug.
