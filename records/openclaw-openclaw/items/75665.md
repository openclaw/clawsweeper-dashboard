## Summary

Keep open. The report is concrete and plausible, and it matches the post-v2026.4.23 event-loop/plugin hot-path regression cluster. Current main has related session isolation and post-v2026.4.29 plugin/gateway performance fixes, but I did not find direct proof that the Slack Socket Mode plus Bedrock `modelByChannel` starvation no longer reproduces, and the likely fixes are not in the latest shipped release tag.

## What This Changes

The issue requests a fix for a v2026.4.27-v2026.4.29 regression where a long Bedrock-backed Slack channel turn leaves `sessionId=main` processing, blocks other Slack channels, and causes event-loop delay plus Socket Mode pong timeouts.

## Best Possible Solution

Preserve the per-channel session-key and lane model, then add a focused regression proving a slow turn in one Slack channel cannot block another channel, while ensuring embedded-run plugin/provider preparation reuses existing metadata snapshots instead of doing synchronous broad discovery on each turn.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported v2026.4.27-v2026.4.29 range because the issue includes setup, steps, rollback, and matching logs; unclear for current main because this review did not run the live Slack/Bedrock scenario after the main-only fixes.

## Solution Assessment

Is this the best way to solve the issue? Unclear until a targeted current-main reproduction exists; the maintainable direction is generic session-key/lane isolation plus plugin metadata snapshot reuse, not Slack-specific recovery policy or a broader stuck-session abort change.
