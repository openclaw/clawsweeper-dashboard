## Summary

Keep open. The report is concrete and current main has per-channel Slack session-key and active-run lookup plumbing, but I did not find a current-main Slack Socket Mode plus Bedrock `modelByChannel` regression test or live proof showing that one long channel run cannot starve another.

## What This Changes

A Slack Socket Mode deployment with per-channel `modelByChannel` should not let one long Bedrock-backed channel run leave shared `main` processing state that prevents other Slack channels from responding.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same active session or lane, then fix any missing session-key propagation or active-run lookup exposed by that test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported `v2026.4.27` to `v2026.4.29` setup because the issue includes concrete steps, logs, and rollback evidence; unclear for current main because no cross-channel Slack/Bedrock regression or live proof is present.

## Solution Assessment

Is this the best way to solve the issue? Unclear until current main is covered by a deterministic regression; the narrow maintainable path is session/lane isolation for Slack channel keys, not a broad stuck-session recovery or upstream-timeout redesign.
