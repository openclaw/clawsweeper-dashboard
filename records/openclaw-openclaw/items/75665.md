## Summary

Keep open. The report is concrete and current main shows intended per-channel Slack session-key and session-keyed active-run plumbing, but there is still no current-main Slack Socket Mode plus Bedrock `modelByChannel` regression or live proof showing that one long channel run cannot starve another channel.

## What This Changes

The issue asks OpenClaw to prevent a long Bedrock run in one Slack Socket Mode channel from leaving shared `main` processing state and blocking other Slack channels.

## Best Possible Solution

Add a focused regression proving a slow active run in one Slack channel does not make another Slack channel wait on the same session or lane, then fix only the missing session-key propagation or active-run lookup exposed by that test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported v2026.4.27 setup because the issue gives concrete steps, logs, and rollback evidence; unclear for current main because no cross-channel Slack/Bedrock regression or live proof is present.

## Solution Assessment

Is this the best way to solve the issue? Unclear until current main is covered by a deterministic regression; the narrow maintainable path is Slack channel session/lane isolation, while #75648 should separately track provider timeout semantics.
