## Summary

Keep open. The report is fresh, concrete, and not covered by a shipped or clearly canonical fix: current main routes ordinary Slack channel turns through per-channel session keys, but the reported all-channel stall also involves event-loop starvation during a long Bedrock embedded run, and the adjacent upstream-timeout work remains open rather than resolved.

## What This Changes

The issue reports that a long-running Bedrock call in one Slack channel can leave a shared processing session stuck and stop responses in other Slack channels until the gateway restarts.

## Best Possible Solution

Keep the issue open until a focused fix or proof covers Slack multi-channel isolation under a long or hung Bedrock embedded run: preserve route-specific session keys through reply-run lanes, keep Socket Mode responsive under provider pressure, and link any remaining upstream timeout configurability to #75648 instead of treating that as full coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for triage: configure multiple Slack channels, set a heavier Bedrock model via `modelByChannel` for one channel, trigger a long turn there, then send messages in other channels while watching stuck-session, liveness, and Slack pong-timeout logs. This review did not run the live Slack/Bedrock path, so confirmation still needs operator or maintainer reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear; current main's per-channel Slack routing and unreleased plugin-prep improvements are the right direction, but they do not prove the all-channel stall is fixed. The narrow maintainable path is a regression test or live proof around Slack route-session/lane isolation plus a bounded/yielding provider or plugin-prep fix for the blocking path that reproduces the event-loop starvation.
