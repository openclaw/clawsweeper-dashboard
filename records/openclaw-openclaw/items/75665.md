## Summary

Keep open. Current main has important Slack session-key and session-keyed reply-run plumbing plus broad v2026.5.2 performance fixes, but there is still no high-confidence current-main proof that a slow Slack Socket Mode Bedrock turn with `modelByChannel` cannot starve another Slack channel or the shared event loop.

## What This Changes

The issue reports that a long Bedrock turn in one Slack Socket Mode channel leaves a shared processing session stuck and stops other Slack channels from responding.

## Best Possible Solution

Add a deterministic Slack cross-channel isolation regression for two Slack room session keys with one long active Bedrock-style run and `modelByChannel`, then fix any missing session-key or lane propagation it exposes while leaving upstream timeout policy to #75648.

## Reproduction Assessment

Unclear for current main. The v2026.4.27 report is concrete and source inspection shows the relevant Slack/session/reply-run paths, but I did not find or run a deterministic current-main Slack Socket Mode plus Bedrock/modelByChannel test that reproduces or disproves cross-channel starvation.

## Solution Assessment

Unclear. Current main has the right session-key plumbing and broad performance fixes, but the narrow maintainable solution is still to prove per-channel isolation under a blocked run before changing shared lane or provider-timeout policy.
