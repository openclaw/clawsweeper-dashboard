## Summary

Keep this PR open for maintainer review: it points at a real ACP timeout failure mode, but the branch changes the core ACP timeout contract globally while current main already has targeted Codex/Claude ACPX timeout suppression and the documented/tested `/acp timeout` behavior still forwards backend-advertised timeout config.

## What This Changes

The PR removes `timeoutSeconds` from ACP session config option emission and from ACP spawn runtime initialization while keeping dispatch-level timeout handling.

## Best Possible Solution

Adopt one explicit ACP timeout contract: either narrow suppression to the adapters/rejections that reject timeout, or update core tests, docs, and live proof to make runtime timeouts OpenClaw-watchdog-only for all ACP backends.

## Reproduction Assessment

I did not establish a fresh live current-main Claude/Codex failure because main already has targeted ACPX suppression for those adapters.

## Solution Assessment

No, not yet: the patch is a plausible repair for the failing adapter class, but it changes the generic ACP timeout contract without aligning docs, manager tests, or upgrade proof. A narrower adapter-specific suppression remains the safer default unless maintainers intentionally adopt watchdog-only ACP timeouts.
