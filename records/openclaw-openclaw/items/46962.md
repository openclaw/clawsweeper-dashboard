## Summary

Keep open. The underlying blockquote-prefix bug still exists on current main and this PR targets the right Markdown IR surface, but the branch is stale against current main's loose-list paragraph fix and would regress token-aware list spacing if merged as-is. The Greptile discussion and author follow-up are useful: the early hardbreak, multi-paragraph, and nested quote gaps were addressed, leaving the newer list-separator interaction as the concrete blocker.

## What This Changes

The PR adds blockquote-depth tracking in `markdownToIR` and regression tests so configured blockquote prefixes are re-emitted on soft breaks, hard breaks, multi-paragraph blockquotes, and nested blockquote continuation lines.

## Best Possible Solution

Integrate the blockquote-depth prefix restoration into current main's shared Markdown IR renderer while preserving the existing token/open-level loose-list separator behavior, then land it with focused blockquote and nested-list coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The current-main reproduction is rendering `> Line 1\n> Line 2` through `markdownToIR` with `blockquotePrefix: "> "`; the source shows soft/hard breaks append a bare newline while Slack and Signal pass that prefix option, so continuation lines lose the quote marker.

## Solution Assessment

No. The PR's direction is the right narrow fix, but the current branch is not the best merge shape because it overwrites current main's token-aware list separator; the safer solution is to add blockquote-depth prefixing to that existing separator path instead of returning for every non-empty list stack.
