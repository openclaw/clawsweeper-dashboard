## Summary

Keep open: current main still has the reported blockquote-prefix loss, but this stale PR is not merge-ready because it would overwrite current loose-list paragraph separator behavior and has no after-fix real behavior proof.

## What This Changes

The PR changes shared Markdown IR rendering to track blockquote depth, re-emit configured blockquote prefixes after line breaks and paragraph transitions, and adds blockquote spacing regression tests.

## Best Possible Solution

Refresh or replace the branch against current main so blockquote prefixes are restored on line and paragraph continuations while preserving the token/open-level loose-list guard, then validate with focused IR tests and redacted Slack or Signal style runtime output.

## Reproduction Assessment

Yes, by source inspection. Current main appends only a bare newline for soft/hard breaks while Slack and Signal set blockquotePrefix, so a two-line quoted Markdown input loses the quote marker on the continuation line.

## Solution Assessment

No, not as submitted. The blockquote-depth approach is narrow, but the branch must be rebased onto current appendParagraphSeparator(state, token) behavior from https://github.com/openclaw/openclaw/pull/74474 instead of replacing it with the older listStack-only guard.
