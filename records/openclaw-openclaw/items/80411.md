## Summary

Keep open. Current main does not contain the reported `qa parity-report --runtime-axis --token-efficiency` surface, and the still-open related PR contains a source-confirmable path where a failed live 0-token runtime pair can render as a passing token-efficiency report.

## What This Changes

The issue asks QA-lab to avoid marking failed live runtime-pair runs with zero captured assistant usage as passing token-efficiency comparisons.

## Best Possible Solution

Update https://github.com/openclaw/openclaw/pull/80323 so live token-efficiency rows fail or skip when either runtime cell failed or no assistant-message usage was captured, with regression coverage before that QA-lab expansion merges.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks the reported runtime-axis token-efficiency command. The issue is source-reproducible on the linked PR branch because failed 0/0 live rows are evaluated only by token delta and can therefore pass.

## Solution Assessment

Yes. The narrow maintainable fix is to have the token-efficiency report consult runtime parity status, runtime cell failure state, and captured assistant-message usage before allowing a live row to pass.
