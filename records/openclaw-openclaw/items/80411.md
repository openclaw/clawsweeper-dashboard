## What This Changes

The issue asks QA-lab runtime token-efficiency reporting to fail, skip, or mark unavailable live rows where runtime cells failed before assistant-message usage instead of passing 0 vs 0.

## Best Possible Solution

Update https://github.com/openclaw/openclaw/pull/80323 so live token-efficiency rows consult runtime parity status, cell failure state, and captured assistant-message usage before a live row can pass.

No high-confidence current-main reproduction: current main and v2026.5.7 do not expose the reported --runtime-axis --token-efficiency command. The bug is source-reproducible on the linked PR head because failed 0/0 live rows are evaluated only by token delta and can pass.

## Solution Assessment

Yes for the linked PR branch. The narrow maintainable fix is to make token-efficiency reporting reject, skip, or mark unavailable live rows where either runtime cell failed before assistant-message usage, with regression coverage before the QA-lab expansion merges.
