## Summary

Keep open. Current main still lacks the runtime-axis token-efficiency report, and the open QA-lab PR that adds it still has a source-confirmable path where live 0/0 usage can pass because the token report only fails on token-delta threshold rows.

## What This Changes

Requests that QA-lab runtime token-efficiency reports fail or skip live rows when either runtime cell fails before producing assistant-message usage, instead of reporting a passing live 0/0 usage result.

## Best Possible Solution

Fix the token-efficiency guard in the open QA-lab runtime parity PR so live rows can pass only when both runtime cells completed and captured assistant-message usage, with a regression test for failed zero-usage cells.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks the reported `--runtime-axis --token-efficiency` command. The issue is source-reproducible on the linked PR head because the token report can pass a live 0/0 row while runtime parity records failure separately.

## Solution Assessment

Yes. The narrow maintainable solution is to make the PR's token-efficiency report consult runtime parity failure state and assistant-message usage presence before allowing a live row to pass.
