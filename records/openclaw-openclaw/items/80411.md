## What This Changes

The issue requests that the QA-lab runtime token-efficiency report not mark failed live runtime-pair runs with zero captured usage as a passing live-usage result.

## Best Possible Solution

Fix https://github.com/openclaw/openclaw/pull/80323 so live token-efficiency rows fail or skip when runtime parity failed or no assistant-message usage was captured, with focused regression coverage before the harness lands.

## Reproduction Assessment

No high-confidence current-main reproduction: current main only has the model-axis `qa parity-report` command. The failure is source-reproducible on the linked PR branch because live token-efficiency pass/fail is based on flagged token deltas, so a failed 0/0 row yields no failure.

## Solution Assessment

Yes. The narrow maintainable fix is to have the token-efficiency report consult runtime parity status, cell failure state, and captured assistant-message usage before allowing a live row to pass.
