## Summary

Keep open. Current main still lacks the reported runtime-axis token-efficiency command, while the same-author open PR that adds it still has source-confirmed logic where a failed live 0/0 usage row can render as a passing token-efficiency report.

## What This Changes

The issue asks QA-lab runtime token-efficiency reports to treat failed live runtime pairs with zero assistant usage as unavailable or failing instead of a passing 0/0 live-usage result.

## Best Possible Solution

Update the open runtime-parity PR so live token-efficiency rows fail or skip when either runtime cell failed or no assistant-message usage was captured, with regression coverage before merge.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main does not expose the reported runtime-axis token-efficiency command. The issue is source-reproducible on the linked PR branch because failed 0/0 live rows are evaluated only by token delta and can therefore pass.

## Solution Assessment

Yes. The narrow maintainable fix is to have the token-efficiency report consult runtime parity status, runtime cell failure state, and captured assistant-message usage before allowing a live row to pass.
