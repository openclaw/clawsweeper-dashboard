## Summary

Keep open. Current main and the latest release do not expose the reported runtime-axis token-efficiency command, while the same-author open implementation PR still contains a source-confirmable path where a failed live 0/0 usage row can be reported as a pass.

## What This Changes

The issue asks for QA Lab runtime token-efficiency reporting to avoid showing `pass` with `live-usage` and 0 vs 0 tokens when a live runtime-pair run failed before assistant usage was captured.

## Best Possible Solution

Fix https://github.com/openclaw/openclaw/pull/80323 so live token-efficiency rows skip or fail when either runtime cell failed or no assistant-message usage was captured, add regression coverage, and close this issue only after that implementation merges or is superseded.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main and `v2026.5.7` do not expose `qa parity-report --runtime-axis --token-efficiency`. The issue is source-reproducible on https://github.com/openclaw/openclaw/pull/80323 because live token rows fail only on token-delta flags, so a failed 0/0 usage pair can pass.

## Solution Assessment

Yes for the linked branch: the narrow maintainable fix is for the token-efficiency report to consult runtime parity status, runtime cell error state, and captured assistant-message usage before allowing a live row to pass. A separate main-branch repair would duplicate the still-open implementation PR.
