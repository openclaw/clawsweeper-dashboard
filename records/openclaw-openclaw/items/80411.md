## Summary

Keep open. Current main still does not contain the reported runtime-axis token-efficiency command, and the linked open PR still has a source-confirmable path where a failed live 0-token runtime pair can render as a passing token-efficiency report.

## What This Changes

Ensure QA-lab live token-efficiency reports do not mark failed runtime-pair runs with zero captured assistant usage as passing live usage.

## Best Possible Solution

Fix https://github.com/openclaw/openclaw/pull/80323 so live token-efficiency rows fail, skip, or become unavailable when either runtime cell failed or no assistant-message usage was captured, with regression coverage before the harness lands.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks `qa parity-report --runtime-axis --token-efficiency`. The failure is source-reproducible on the linked PR head because live failures are derived only from token-delta threshold flags, so a failed 0/0 row can pass.

## Solution Assessment

Yes. The narrow maintainable fix is to make the token-efficiency report consult runtime parity status, runtime cell failure state, and captured assistant-message usage before allowing a live row to pass.
