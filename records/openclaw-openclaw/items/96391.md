## Summary

Keep this PR open: the central fix is still missing from current main, the patch is a focused candidate for the linked WebChat session-context bug, and I found no discrete code-review defect. The remaining blocker is maintainer acceptance of the session-policy change: a successful history load now arms the private one-shot resume marker for the next send, so a matching displayed session can bypass idle/daily rollover once.

## What This Changes

The PR updates WebChat history loading to arm `reconnectResumeSessionId` after `chat.history` returns an existing session, updates the UI send expectation, and adds session reset regression coverage for ordinary stale sends.

## Best Possible Solution

Land the current focused patch only after maintainers explicitly accept loaded-history first-send resume as the intended WebChat session policy, while keeping the marker private and preserving ordinary stale-send rollover coverage.

## Reproduction Assessment

Yes at source level: current main loads the history-backed session id but does not arm the resume marker, while the backend only suppresses stale rollover when that marker is present. I did not rerun a live WebChat restart locally in this read-only review, but I inspected the contributor's screenshots and terminal proof.

## Solution Assessment

Yes, with maintainer policy acceptance: reusing the existing private Control UI resume path is narrower than adding a public Gateway field or rebuilding transcript context in a new path. The unresolved question is whether loaded-history continuity should intentionally beat idle/daily rollover for one send.
