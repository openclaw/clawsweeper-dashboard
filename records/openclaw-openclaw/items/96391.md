## Summary

Keep open, but this PR is not merge-ready: it targets a real WebChat session-context bug, yet the implementation bypasses daily/idle freshness for ordinary WebChat sends and exposes a session freshness override through exported `MsgContext`.

## What This Changes

The PR adds `CallerSessionId` to `MsgContext`, sets it from Gateway `chat.send`, and teaches `initSessionState` plus new tests to reuse matching stale sessions.

## Best Possible Solution

Preserve WebChat restored-session continuity through a bounded gateway-owned resume signal while keeping ordinary stale sends and plugin contexts under documented daily/idle reset semantics.

## Reproduction Assessment

Yes for the PR regression path: source inspection shows every WebChat send with a stored entry would set the new override, and `callerVerifiedSession` is ORed into `freshEntry` before freshness evaluation. I did not rerun the original MiniMax/WebChat restart scenario live.

## Solution Assessment

No; the caller-verified idea is plausible for the restore path but is too broad and in the wrong contract layer. The safer fix is to keep resume intent on a gateway-private, one-shot signal and preserve ordinary stale rollover.
