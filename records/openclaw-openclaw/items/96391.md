## Summary

Keep open: this is a focused fix for a real WebChat session-state bug, but the patch is not merge-ready because it bypasses configured daily/idle session rollover for ordinary `chat.send` calls.

## What This Changes

The PR adds `CallerSessionId` to `MsgContext`, sets it from the webchat `chat.send` backing session, and lets session initialization reuse a matching stored entry with new unit coverage.

## Best Possible Solution

Preserve WebChat restored-session continuity through a bounded verified resume path while keeping ordinary sends and no-sessionId clients subject to configured reset policy, with gateway/session regression coverage for both paths.

## Reproduction Assessment

Yes for the merge blocker: source inspection shows `CallerSessionId` is set from any existing stored entry, and current tests define ordinary asserted sends as still rolling over when stale. I did not rerun a live current-main WebChat/MiniMax reproduction in this read-only review.

## Solution Assessment

No; the caller-verified idea is plausible for the linked bug, but setting it from the server-loaded entry on every send is too broad. The safer fix needs a bounded verified resume path that does not replace configured reset policy for ordinary sends.
