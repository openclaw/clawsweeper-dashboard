## Summary

Keep open; this branch should not merge as-is because it can bypass configured stale-session rollover and exposes a session-resume control through `MsgContext`, while current `main` already has a private one-shot resume boundary for the related Control UI reconnect path.

## What This Changes

The PR adds a `CallerSessionId` context field, sets it from WebChat `chat.send`, and uses it in reply session initialization to reuse an existing stored session after gateway restart.

## Best Possible Solution

Preserve WebChat restored-session continuity through a bounded gateway-owned resume or restore signal while keeping ordinary stale sends and plugin-built contexts under the documented daily/idle reset semantics.

## Reproduction Assessment

Yes for the merge blocker: source inspection shows every stored-entry WebChat send would receive `CallerSessionId`, and current docs/tests require ordinary stale sends to roll over. I did not rerun a live WebChat/MiniMax reproduction of the original issue in this read-only review.

## Solution Assessment

No; the caller-verified idea is plausible for the linked bug, but setting it from the server-loaded entry on every send is too broad. The safer solution keeps resume controls internal and bounded to the specific WebChat restore semantics maintainers choose.
