## Summary

Keep this PR open: it addresses a real WebChat session-context bug with a focused UI-side signal and sufficient screenshot proof, but the current head still has a concrete lint failure before it can be considered merge-ready. ## What This Changes

The PR changes WebChat history loading so a loaded session id arms the existing private one-shot reconnect-resume marker for the next `chat.send`, and updates UI/session regression tests.

## Best Possible Solution

Fix the lint failure, keep the resume signal private to Control UI/Gateway internals, and land only if maintainers accept loaded-history first-send continuity as the intended WebChat session policy.

## Reproduction Assessment

Yes at source level: current main loads and displays a stored `sessionId` from `chat.history` without arming the private resume marker, while the backend only suppresses stale rollover when that marker is present. The contributor screenshots also show an after-restart WebChat flow where the model recalls prior context.

## Solution Assessment

Yes, after the lint repair and maintainer policy acceptance: reusing the existing private Control UI resume path is narrower than adding a public Gateway or Plugin SDK resume field. The remaining tradeoff is whether loaded-session continuity should intentionally beat idle/daily rollover for one send.
