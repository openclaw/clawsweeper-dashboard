## Summary

This PR should stay open but needs changes before merge: it targets a real WebChat session-state bug, but the current implementation can bypass configured daily/idle rollover for ordinary WebChat sends and exposes that bypass through exported message context.

## What This Changes

The PR adds `CallerSessionId` to `MsgContext`, sets it from WebChat `chat.send`, and reuses a matching stored session entry in `initSessionState` with new session tests.

## Best Possible Solution

Preserve WebChat restart continuity through a bounded gateway-owned resume signal while keeping ordinary stale sends and plugin contexts under documented daily/idle reset semantics.

## Reproduction Assessment

Yes for the PR-introduced regression path: source inspection shows ordinary WebChat sends can carry the new matching override while current docs and tests require ordinary stale sends to roll over. I did not run the original MiniMax/WebChat restart scenario live in this read-only review.

## Solution Assessment

No; preserving restart context is the right goal, but exporting a general `MsgContext` freshness override is broader than the existing private Control UI resume boundary.
