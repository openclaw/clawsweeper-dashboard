## Summary

Keep this PR open. Current main still does not implement the central hardening: attachment-cache local reads use the current custom readLocalBuffer path rather than the shared readLocalFileSafely/openLocalFileSafely identity-checked path, and generated temp attachment files are still written without an explicit 0o600 mode. The PR is also dirty against current main, so it needs a port/rebase rather than cleanup closure.

## What This Changes

This PR changes the media attachment cache to read local attachments through readLocalFileSafely with SafeOpenError("too-large") mapping and to write generated temp attachment files with mode 0o600.

## Best Possible Solution

Port the hardening onto current main by refactoring or replacing readLocalBuffer with the shared safe-read/open helper while preserving allowed-root validation and MediaUnderstandingSkipError/fallback semantics, add { mode: 0o600 } to the temp write, add focused regression tests for oversized safe-read mapping, path identity/race behavior, and temp-file permissions, then run the focused media-understanding tests plus the normal changed gate before merge.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the remaining gap: getBuffer still reaches readLocalBuffer and getPath still writes the temp file without mode 0o600; a focused test can make the identity-check and permission differences executable.

## Solution Assessment

Unclear as currently submitted. The direction is the right narrow fix, and the author addressed the prior SafeOpenError("too-large") review concern, but the PR must be ported to the current attachment-cache shape before maintainers can judge the final implementation.
