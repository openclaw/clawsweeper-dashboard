## Summary

Keep this PR open. Current main still does not use the shared safe-read helper for the attachment cache and still writes generated temp media without an explicit private mode, but the PR head is dirty against main and needs maintainer review or a port before it can merge safely.

## What This Changes

The PR changes `src/media-understanding/attachments.cache.ts` to read local attachment buffers through `readLocalFileSafely`, map oversized `SafeOpenError` to `MediaUnderstandingSkipError`, update the resolved path, and write generated temp files with mode `0o600`.

## Best Possible Solution

Port the hardening onto current main by replacing or refactoring `readLocalBuffer` around the shared safe-read/open helper while preserving allowed-root checks and `MediaUnderstandingSkipError` fallback behavior, add `{ mode: 0o600 }` to generated temp writes, and cover the behavior with focused media-understanding tests before the changed gate.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the remaining gap: local attachment reads still go through `readLocalBuffer`, and temp writes still omit `0o600`; focused tests can make the identity-check and permission differences executable.

## Solution Assessment

Unclear as submitted. The direction is the narrow maintainable fix and the prior oversized-file mapping concern is addressed, but the PR is based on an older attachment-cache shape and must be ported to current main before the final implementation can be judged.
