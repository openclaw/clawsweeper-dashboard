## Summary

Keep open: current main already routes local attachment reads through `openLocalFileSafely`, which provides the same opened-file identity checks used by `readLocalFileSafely`, but the temp-file `0o600` hardening is still unique and the external PR still lacks real behavior proof.

## What This Changes

The PR replaces attachment-cache local reads with `readLocalFileSafely`, maps oversized safe-open errors back to `MediaUnderstandingSkipError`, and writes generated media temp files with mode `0o600`.

## Best Possible Solution

Port the remaining hardening onto current main by preserving the current `readLocalBuffer` root/fallback/maxBytes semantics, adding explicit `0o600` temp-file mode, and adding focused media-cache tests for the permission guarantee.

## Reproduction Assessment

Yes for the source-level hardening check: current main shows the local-read path is already identity-checked through `openLocalFileSafely`, while generated temp media files are still written without an explicit mode. This was not run as a live macOS/iMessage reproduction in this read-only review.

## Solution Assessment

No as submitted against current main. The direction is good, but the best mergeable solution is a small port that keeps current `readLocalBuffer` behavior and adds the remaining temp-mode hardening with targeted coverage.
