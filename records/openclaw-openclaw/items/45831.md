## Summary

Keep open. The security-hardening direction is still relevant on current main, but the source PR is dirty against the current attachment-cache implementation and needs maintainer review or a careful port rather than cleanup closure.

## What This Changes

The PR changes the attachment cache to read local files through `readLocalFileSafely`, map oversized safe-open errors to media skip errors, update the cached canonical path, and write generated temp files with `0o600` mode.

## Best Possible Solution

Port the hardening onto current main by using the shared identity-checked safe-open/read path or an equivalent helper inside the current cache flow, add explicit `0o600` temp writes, preserve media skip/fallback semantics, and cover the behavior with focused media tests.

## Reproduction Assessment

Yes. Source inspection of current main shows local attachment reads still go through `readLocalBuffer` instead of `readLocalFileSafely`, and generated temp attachment files still omit an explicit mode.

## Solution Assessment

Unclear as submitted. The direction is narrow and maintainable, and the prior oversized-file mapping concern is addressed, but the PR targets an older attachment-cache shape and needs a current-main port before merge.
