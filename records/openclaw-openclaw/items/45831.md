## Summary

Keep open: current main already supersedes the raw-read part with handle-based fs-safe reads, but this PR still contains a meaningful `0o600` temp-file permission hardening that is not on main and should not be swept away.

## What This Changes

The PR replaces the attachment cache's raw local read with fs-safe reading, maps oversized safe-open errors to media skip errors, and writes attachment temp copies with mode `0o600`.

## Best Possible Solution

Rebase or replace the PR with a narrow temp-file `0o600` patch, keep current main's `openLocalFileSafely`/`readLocalBuffer` implementation, and add focused mode coverage plus real filesystem proof.

## Reproduction Assessment

Not applicable: this is proactive security hardening rather than a user-visible bug report. Source inspection gives a high-confidence current-main comparison: safe reads are already present, while explicit temp-file mode is still missing.

## Solution Assessment

No: replaying the old safe-read diff is no longer the best solution because current main has a stronger handle-based safe-read path. The maintainable solution is to extract only the remaining temp-file permission hardening and verify it.
