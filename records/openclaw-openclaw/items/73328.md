## Summary

Keep this issue open: current main still falls through to the 32K Bedrock discovery default for reported unlisted Claude 3/3.5 IDs, and open PRs #73382 and #73959 are already the active implementation candidates, so closing or queueing a duplicate repair would be premature.

## What This Changes

Request per-model Amazon Bedrock discovery context-window handling so Claude 3.x/4.x Bedrock model IDs do not inherit the 32K default.

## Best Possible Solution

Land one canonical Amazon Bedrock plugin-owned fix with regression coverage for reported Claude 3/3.5 foundation IDs and region/profile-prefixed variants. If maintainers choose a public override knob instead of a narrow family resolver, keep the manifest schema, exported config type, docs, UI hints, and discovery cache key behavior aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: mock `ListFoundationModels` with `anthropic.claude-3-opus-20240229-v1:0` or `anthropic.claude-3-5-sonnet-20241022-v2:0`, and discovery falls through to `defaultContextWindow` because those IDs are absent from `KNOWN_CONTEXT_WINDOWS`.

## Solution Assessment

Unclear. The requested outcome is valid, but the issue proposes two solution shapes; a narrow family/known-model resolver is the smaller behavioral fix unless maintainers explicitly want a new user-facing override surface.
