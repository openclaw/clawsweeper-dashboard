## Summary

Keep this open. Current main still lets unlisted Bedrock Claude 3.x/3.5 model IDs fall through to the Amazon Bedrock discovery default context window, and two open PRs already use closing syntax for this issue, so the issue should stay open until maintainers choose and land one canonical fix.

## What This Changes

The issue asks Amazon Bedrock discovery to assign correct per-model context windows for Claude 3.x/4.x models, or expose a per-model override, instead of applying the global 32K fallback to every unlisted discovered model.

## Best Possible Solution

Land one plugin-owned Bedrock discovery fix with regression coverage for the reported Claude 3 Opus/Sonnet, Claude 3.5 Sonnet, and region-prefixed inference-profile variants. If maintainers choose a public override knob instead of a known-model/family resolver, keep the plugin manifest schema, exported config type, docs, UI hints, cache key, and tests aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a stubbed Bedrock discovery response for an unlisted reported Claude ID will miss `KNOWN_CONTEXT_WINDOWS` and fall through to the 32000 default path used by `toModelDefinition` and inference-profile fallback handling.

## Solution Assessment

Unclear. The requested outcome is valid, but the open PRs choose different contracts; the narrowest maintainable fix appears to be a provider-owned known-model/family resolver unless maintainers explicitly want a new regex override config surface.
