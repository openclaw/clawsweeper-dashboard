## Summary

Keep this open. Current main still resolves unlisted Bedrock Claude 3.x/3.5 IDs through the Amazon Bedrock discovery fallback, the plugin exposes only a global default context-window knob, and two open PRs already use closing syntax for this issue; maintainers should choose and land one canonical fix before the issue is closed.

## What This Changes

The issue requests per-model Amazon Bedrock discovery context-window handling for Claude 3.x/4.x models instead of applying the 32000-token default to every unmatched discovered model.

## Best Possible Solution

Land one plugin-owned Bedrock discovery fix with regression coverage for the reported Claude 3/3.5 IDs and region-prefixed inference-profile variants. If maintainers choose a public override knob instead of a family resolver, keep the plugin manifest schema, exported config type, docs, UI hints, cache key, and tests aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an unlisted Claude 3.5 Sonnet or Claude 3 Opus Bedrock model ID is absent from `KNOWN_CONTEXT_WINDOWS`, so discovery falls through to `defaults.contextWindow`, which defaults to 32000.

## Solution Assessment

Unclear. The requested outcome is valid, but the open PRs propose different contracts; the narrowest maintainable fix appears to be a provider-owned known-model/family resolver unless maintainers explicitly want a new regex override config surface.
