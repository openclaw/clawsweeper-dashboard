## Summary

Keep open: the Feishu object-format streaming config is not on current main, but this branch is unsafe to merge as-is because it regresses default streaming, leaves public config surfaces stale, conflicts with later streaming-card work, and lacks real Feishu/Lark proof.

## What This Changes

The branch adds boolean-or-object Feishu streaming config parsing, auto-mode streaming startup from partial replies, configurable throttle/title plumbing for Card Kit sessions, and updated dispatcher tests.

## Best Possible Solution

Define one Feishu-owned streaming config contract across schema, manifest, docs, dispatcher, runtime, and tests while preserving current defaults, identity metadata, and a real Feishu/Lark verification path.

## Reproduction Assessment

Yes for source-level PR review: comparing the PR diff with current main shows the default-config regression, async auto-mode partial drop, public metadata gap, throttle mismatch, and identity-header bypass. No live Feishu/Lark setup was proven by the contributor.

## Solution Assessment

No. The object-config direction may be useful, but the patch should preserve default semantics, publish one config contract across public surfaces, route title/header behavior through identity metadata, and include real Feishu/Lark proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve default-enabled streaming:** `extensions/feishu/src/reply-dispatcher.ts:143-146`
  - body: The new predicate only enables streaming for explicit `true` or object config, so omitted `streaming` no longer follows current main and docs' default-true behavior. Existing Feishu configs using card mode or auto-detected card replies would stop streaming unless users add new config.
  - confidence: 0.97
- **[P2] Queue the partial that starts auto streaming:** `extensions/feishu/src/reply-dispatcher.ts:408-416`
  - body: `startStreaming()` begins Card Kit creation asynchronously, but this partial is queued only if the session is already active. The first auto-mode chunk can be dropped while startup is in flight, so short replies may show no incremental update.
  - confidence: 0.91
- **[P2] Publish object streaming config in public surfaces:** `extensions/feishu/src/config-schema.ts:46-53`
  - body: The runtime schema accepts an object for `streaming`, but the plugin manifest and Feishu docs still advertise only a boolean. Setup/config discovery and users following docs will not see or validate `enabled`, `throttleMs`, or `title`.
  - confidence: 0.88
- **[P2] Preserve the current throttle cadence:** `extensions/feishu/src/reply-dispatcher.ts:200`
  - body: Current main uses a tested 160 ms Feishu streaming-card throttle, but this branch falls back to 150 ms in the dispatcher and 100 ms in the session constructor. Keep the established cadence unless maintainers intentionally choose a new API-rate tradeoff.
  - confidence: 0.84
- **[P2] Merge custom titles with identity metadata:** `extensions/feishu/src/reply-dispatcher.ts:215`
  - body: Passing a hard-coded header for every streaming session bypasses current main's identity-based card header and note path. A title option should flow through the existing resolver instead of replacing configured agent identity metadata.
  - confidence: 0.83
