## Summary

Keep open: the Feishu streaming configuration direction may be useful, but this branch is still not merge-ready against current main. It regresses the default-enabled streaming contract, misses public config surfaces, conflicts with later identity/header/note and streaming-card work, and still lacks contributor-provided Feishu/Lark runtime proof.

## What This Changes

The PR changes Feishu streaming config to accept boolean-or-object values, uses object throttle/title when starting Card Kit streaming, starts auto-mode streaming from partial replies, and updates dispatcher expectations.

## Best Possible Solution

Define one Feishu-owned streaming config contract across schema, manifest, docs, dispatcher, and tests, preserve shipped defaults unless maintainers choose a new rate tradeoff, and route title/header behavior through the existing identity header/note path.

## Reproduction Assessment

Yes for source-level PR review: comparing current main, the PR diff, and related Feishu streaming surfaces shows the default-config regression, async auto-mode partial drop, public metadata gap, throttle mismatch, and identity-header bypass. No live Feishu/Lark setup was proven by the contributor.

## Solution Assessment

No. The object-config direction may be useful, but this patch is not the best implementation because it changes established defaults, can lose early streaming text, bypasses identity metadata, and does not update the public config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve default-enabled streaming:** `extensions/feishu/src/reply-dispatcher.ts:142-146`
  - body: The new predicate only enables streaming for explicit `true` or object config, so omitted `streaming` no longer follows current main and docs' default-true behavior. Existing Feishu configs using card mode or auto-detected card replies would stop streaming unless users add new config.
  - confidence: 0.97
- **[P2] Queue the partial that starts auto streaming:** `extensions/feishu/src/reply-dispatcher.ts:408-416`
  - body: `startStreaming()` begins Card Kit creation asynchronously, but this partial is queued only if the session is already active. The first auto-mode chunk can be dropped while startup is in flight, so short replies may show no incremental update.
  - confidence: 0.91
- **[P2] Publish object streaming config in public surfaces:** `extensions/feishu/src/config-schema.ts:46-53`
  - body: The runtime schema accepts an object for `streaming`, but the plugin manifest and Feishu docs still advertise only a boolean. Setup/config discovery and users following docs will not see or validate `enabled`, `throttleMs`, or `title`.
  - confidence: 0.88
- **[P2] Preserve the current throttle cadence:** `extensions/feishu/src/reply-dispatcher.ts:200`
  - body: Current main uses a tested 160 ms Feishu streaming-card throttle, but this branch falls back to 150 ms in the dispatcher and 100 ms in the session constructor. Keep the established cadence unless maintainers intentionally choose a new API-rate tradeoff.
  - confidence: 0.84
- **[P2] Merge custom titles with identity metadata:** `extensions/feishu/src/reply-dispatcher.ts:215`
  - body: Passing a hard-coded header for every streaming session bypasses current main's identity-based card header and note path. A title option should flow through the existing resolver instead of replacing configured agent identity metadata.
  - confidence: 0.83
