## Summary

Keep open. Current main still does not implement boolean-or-object Feishu streaming config or config-driven throttle/title, and the PR head is not mergeable because it regresses the existing default-enabled streaming behavior and can drop the first partial while streaming startup is pending.

## What This Changes

The PR changes Feishu streaming config parsing to accept boolean or object forms, passes throttle/title/header options into the Feishu streaming session, and adjusts dispatcher tests for the new streaming card header.

## Best Possible Solution

Preserve the existing `undefined` means enabled and `false` means disabled contract, add a typed Feishu streaming config resolver for boolean and object forms, pass bounded `throttleMs` and validated optional title/header settings through `FeishuStreamingSession`, update manifest/docs, and cover default, disabled, object-config, custom throttle/title, auto plain-text, and auto card streaming regressions.

## Reproduction Assessment

Yes. The regression is reproducible by evaluating the PR head `streamingEnabled` predicate with `streaming` omitted and `renderMode` set to `card` or auto markdown: it returns false where current main returns true. The partial-drop issue follows from the PR diff because the first partial calls asynchronous `startStreaming()` and is only queued if `streaming?.isActive()` is already true.

## Solution Assessment

No. The direction is reasonable, but the current patch is not the best implementation because it changes established defaults, skips pending-start partials, and leaves the user-facing config contract undocumented in the manifest and Feishu docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve default-enabled Feishu streaming:** `extensions/feishu/src/reply-dispatcher.ts:142-146`
  - body: The new predicate only enables streaming when `streaming` is explicitly `true` or an object, so an omitted setting now disables streaming. Existing configs rely on the previous `!== false` behavior, so card mode and auto-detected card replies stop streaming unless users add new config.
  - confidence: 0.97
- **[P2] Queue the partial that starts streaming:** `extensions/feishu/src/reply-dispatcher.ts:414-418`
  - body: In auto mode, the first partial can call `startStreaming()`, but it is only queued when `streaming?.isActive()` is already true. Since startup is asynchronous, the chunk that triggers Card Kit creation is dropped, which weakens or removes streaming for short responses.
  - confidence: 0.89
- **[P2] Expose the object config in plugin metadata:** `extensions/feishu/src/config-schema.ts:43-53`
  - body: This schema accepts an object form for `streaming`, but the Feishu plugin manifest still declares the field as boolean at the account and top-level config surfaces. Manifest-driven setup or config UI will reject or misrepresent the new supported shape unless the metadata is updated with the runtime schema.
  - confidence: 0.82
