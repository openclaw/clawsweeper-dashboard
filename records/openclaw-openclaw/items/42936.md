## Summary

Keep open: the PR still has merge-blocking Feishu streaming default, delivery, and public-config-surface problems, and the external contributor has not provided real Feishu/Lark proof. Current main covers part of the auto-mode streaming path, but the remaining object/throttle/title config contract needs maintainer judgment rather than cleanup closure.

## What This Changes

The PR changes Feishu streaming from boolean-only config to boolean/object config, passes throttle/title into streaming card startup, starts streaming from auto-mode partials, and updates tests for a default card header.

## Best Possible Solution

Define one Feishu-owned streaming config contract across schema, manifest, docs, runtime, and focused tests while preserving default-enabled streaming, identity metadata, and the current throttle cadence unless maintainers explicitly choose a new rate-limit tradeoff.

## Reproduction Assessment

Yes for source-level PR review: the diff against current main shows the default-config regression, async auto-mode partial drop, public metadata gap, throttle mismatch, and identity-header bypass. No live Feishu/Lark runtime reproduction was provided by the contributor.

## Solution Assessment

No. The object-config direction may be useful, but this patch should preserve current defaults, publish one public config contract, route title/header behavior through identity metadata, and include real Feishu/Lark proof before merge.

Full review comments:

- **[P1] Preserve default-enabled streaming:** `extensions/feishu/src/reply-dispatcher.ts:143-146`
  - body: The new predicate enables streaming only for explicit `true` or object config, so omitted `streaming` no longer follows current main's `!== false` default. Existing Feishu card or auto-card setups would stop streaming unless users add a new config key.
  - confidence: 0.97
- **[P2] Queue the partial that starts auto streaming:** `extensions/feishu/src/reply-dispatcher.ts:405-416`
  - body: `startStreaming()` creates the CardKit session asynchronously, but this partial is queued only if `streaming?.isActive()` is already true. Early auto-mode chunks emitted while startup is in flight can be dropped, so short replies may show little or no incremental update.
  - confidence: 0.91
- **[P2] Publish object streaming config in public surfaces:** `extensions/feishu/src/config-schema.ts:46-53`
  - body: The runtime schema accepts `streaming: { enabled, throttleMs, title }`, but the plugin manifest and Feishu docs still advertise only a boolean. Setup/config discovery and users following docs would not see the new fields, leaving two competing contracts.
  - confidence: 0.88
- **[P2] Preserve the current throttle cadence:** `extensions/feishu/src/reply-dispatcher.ts:199-200`
  - body: Current main centralizes Feishu streaming-card updates at 160 ms, but this branch falls back to 150 ms in the dispatcher and 100 ms in the session constructor. Keep the established cadence unless maintainers intentionally choose a new API-rate tradeoff.
  - confidence: 0.84
- **[P2] Merge custom titles with identity metadata:** `extensions/feishu/src/reply-dispatcher.ts:214-215`
  - body: Passing a hard-coded header for every streaming session bypasses current main's identity-derived card header and note path. A title option should flow through the existing resolver or be a documented override so configured agent identity is preserved.
  - confidence: 0.83
