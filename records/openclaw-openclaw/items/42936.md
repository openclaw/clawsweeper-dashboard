## Summary

Keep open, but this PR is not mergeable as-is: the object-format Feishu streaming config is still unique work, while the branch regresses default-enabled streaming and lacks real Feishu behavior proof.

## What This Changes

The PR changes the Feishu plugin to accept boolean or object `streaming` config, pass throttle/title options into streaming cards, start streaming for auto-detected card replies, and update dispatcher expectations.

## Best Possible Solution

Rework this on current main only if maintainers want the new object config shape: keep default streaming enabled, preserve `streaming: false`, preserve current identity header/note behavior, add schema and dispatcher regression tests, and provide redacted live Feishu proof.

## Reproduction Assessment

Yes, from source inspection: a config with `renderMode: "card"` or auto-detected markdown and no `streaming` key is default-enabled on current main, but the PR predicate disables it. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is plausible, but the patch must preserve the existing default/opt-out contract, queue or await auto-mode first partials correctly, and rebase onto current main's streaming card header/note behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve default-enabled Feishu streaming:** `extensions/feishu/src/reply-dispatcher.ts:143-146`
  - body: The new predicate only enables streaming for explicit `true` or an object config, so the documented/default `streaming` value of `undefined` now disables card streaming. Existing configs that relied on the default in `renderMode: "card"` or auto markdown replies would stop streaming; include the `undefined` case while still honoring `streaming: false`.
  - confidence: 0.93
- **[P2] Queue the first partial after starting auto streaming:** `extensions/feishu/src/reply-dispatcher.ts:409-416`
  - body: In auto mode this starts the async streaming session and then immediately checks `streaming?.isActive()`, which remains false until `start()` finishes. The first card-like partial is dropped, and short responses can lose the intended incremental update; await the start promise or queue the update after scheduling the session.
  - confidence: 0.86
