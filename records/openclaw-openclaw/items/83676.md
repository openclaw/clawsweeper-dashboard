## Label Justifications

- `P2`: The report concerns a real but scoped Telegram startup failure mode for proxy users, already covered by current code and release behavior.
- `impact:message-loss`: A false Telegram startup health-check failure can keep the channel from receiving or sending Telegram messages.

## Summary

Current main and the shipped v2026.5.12 release already route Telegram startup getMe through a 15s floor and allow higher `channels.telegram.timeoutSeconds`, so the reported 10s false failure is obsolete.

## What This Changes

The issue requests a proxy-friendly Telegram startup health-check timeout, either configurable or longer than the reported 10 seconds.

## Best Possible Solution

Keep the shipped startup-probe timeout behavior and use `channels.telegram.timeoutSeconds` for slower proxy environments; a new report should include fresh logs only if that documented setting still cannot cover the proxy handshake.

## Reproduction Assessment

No. Current main and v2026.5.12 route startup getMe through a 15s floor plus `channels.telegram.timeoutSeconds`, so the reported 10s timeout path does not reproduce from source inspection.

## Solution Assessment

Yes. Reusing the existing Telegram `timeoutSeconds` contract avoids adding a second health-check-only knob while still letting proxy users raise the startup probe budget.
