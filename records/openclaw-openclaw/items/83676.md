## Summary

Current main and the latest release already cover the reported 10s Telegram startup getMe timeout problem: startup getMe now uses a 15s floor, honors higher `channels.telegram.timeoutSeconds`, and has tests plus docs for that behavior.

## What This Changes

Request to make the Telegram startup getMe health-check timeout proxy-friendly by adding a configurable timeout or increasing the default.

## Best Possible Solution

Keep the shipped 15s startup guard and existing `channels.telegram.timeoutSeconds` override; open a new focused request only if a separate per-operation timeout knob is still needed beyond that existing config.

## Reproduction Assessment

No current-main failure reproduces from source inspection: the reported `elapsedMs=11802` would be below the current 15s startup getMe floor, and higher values can use `channels.telegram.timeoutSeconds`.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution because it avoids a new Telegram-only config key while still allowing proxy-heavy installs to raise the startup probe timeout.
