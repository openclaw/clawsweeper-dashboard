## Label Justifications

- `P2`: The report affects Telegram channel startup behind proxies, but current release evidence shows the requested timeout configurability is already available.
- `impact:message-loss`: A false Telegram startup failure can prevent the channel from receiving or sending messages until the timeout/configuration path is corrected.

## Summary

Current main and v2026.5.12 already route Telegram startup getMe through the Telegram request-timeout guard, defaulting to 15s and honoring higher channel or per-account timeoutSeconds values, which covers the reported 11.8s proxy startup case.

## What This Changes

The issue requests a configurable or longer Telegram startup getMe health-check timeout for proxy-heavy environments.

## Best Possible Solution

Keep the shipped timeoutSeconds startup-probe behavior and route any remaining current-release Telegram proxy transport failures to the narrower live-diagnostics thread.

## Reproduction Assessment

No live proxy repro was run. The reported 10s timeout matches the old/observed failure shape, and current source plus tests show startup getMe now uses a 15s default and can be raised with timeoutSeconds.

## Solution Assessment

Yes. Reusing the existing Telegram timeoutSeconds contract is the narrowest maintainable solution because it avoids a second health-check-only config knob while covering slow proxy startup.
