## Summary

Keep open: current main still uses strict provider-string comparison for live model switch detection, while the existing runtime-alias helper already treats `openai-codex` and `openai` as equivalent model-route providers. That makes the reported stale-switch path plausible and narrowly fixable.

## What This Changes

The issue asks OpenClaw to avoid aborting an active Codex turn when live model switch state differs only by the `openai-codex` versus `openai` provider alias for the same model.

## Best Possible Solution

Keep this open and update live-switch provider/model comparison to reuse the existing runtime-model alias equivalence helper while preserving auth-profile/source checks and stale-flag clearing.

## Reproduction Assessment

Yes, at source level: a session with `liveModelSwitchPending`, default `openai/gpt-5.5`, and active `openai-codex/gpt-5.5` will resolve a different provider string and return a switch request. I did not run a live beta Codex session in this read-only pass.

## Solution Assessment

Yes: the narrowest maintainable fix is to apply the existing runtime-model alias comparison in `src/agents/live-model-switch.ts` and add a focused regression test. A new alias table or broad provider rewrite would be unnecessary duplication.
