## What This Changes

The PR changes the webhook in-flight limiter to reject brand-new keys when the active-key table is full and adds regression tests for active-key capacity behavior.

## Best Possible Solution

## Reproduction Assessment

Yes. Source inspection on current main shows acquiring a second key with `maxTrackedKeys: 1` can prune the first active counter, and a later `release` for that first key is a no-op.

## Solution Assessment

Yes for the runtime code shape: checking capacity before inserting a brand-new key is the narrowest way to preserve active counters. As submitted, the public SDK contract text, branch state, real proof, and maintainer acceptance still need follow-up.
