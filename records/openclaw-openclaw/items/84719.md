## Summary

Keep open: the patch appears to fix a real source-reproducible limiter bug without introducing a discrete code defect, but this external PR still has only test/mock proof and intentionally changes overload behavior so new webhook keys get rejected at active-key capacity.

## What This Changes

The PR changes `createWebhookInFlightLimiter` to treat `maxTrackedKeys` as an active-key capacity cap, rejecting new keys at capacity while preserving existing counters, and adds focused regression tests.

## Best Possible Solution

Land the hard active-key cap with the focused regression tests, redacted real webhook/runtime proof, and explicit maintainer acceptance that new keys fail closed with 429 at capacity.

## Reproduction Assessment

Yes from source inspection: current main sets the new active key and then prunes the map, so the `maxTrackedKeys: 1` sequence from the linked issue can evict an active counter. I did not run tests because this review is constrained to read-only inspection.

## Solution Assessment

Yes for the code shape: checking capacity before inserting a brand-new key is the narrowest maintainable fix for preserving active counters. Merge should still wait for real behavior proof and maintainer acceptance of the fail-closed overload semantics.
