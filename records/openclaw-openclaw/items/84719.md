## Summary

## What This Changes

The PR changes `createWebhookInFlightLimiter` to reject brand-new keys when the active-key table is full and adds regression tests for full-capacity and same-key reacquire behavior.

## Best Possible Solution

Rebase over current main, preserve current SDK option normalization, keep the hard active-key cap with regression coverage, add redacted real webhook/runtime proof, and merge only after maintainer acceptance of the 429-at-capacity contract.

## Reproduction Assessment

Yes, from source inspection. On current main, acquiring one key and then a second key with `maxTrackedKeys: 1` stores the second key and prunes the first active counter, after which `release` for the first key is a no-op.

## Solution Assessment

Yes for the code shape: checking capacity before inserting a brand-new key is the narrowest way to preserve active counters. Merge readiness still depends on real runtime proof, branch update, and maintainer acceptance of the changed overload semantics.
