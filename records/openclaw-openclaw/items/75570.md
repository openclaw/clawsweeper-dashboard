## Summary

Keep this PR open. The code change is a plausible narrow fix for the linked Discord status-reaction bug: current main only removes the immediately previous tracked emoji on terminal transitions, while the PR adds best-effort terminal cleanup for the rest of the known status emojis and covers set-only adapters. I found no functional or security blocker, but the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds a terminal-state cleanup loop in the shared status reaction controller to remove every non-terminal known emoji through the optional `removeReaction` capability, plus unit tests for removal-capable and set-only adapters.

## Best Possible Solution

Land the shared-controller cleanup after adding a single-line Unreleased/Fixes changelog entry and letting CI finish, then close the linked Discord bug through the PR merge.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a controller sequence where earlier `removeReaction` calls fail or are dropped: current main only removes the last tracked emoji on terminal `setDone`, so a previously visible Discord reaction can remain stale. The PR's new unit test models that by walking queued to thinking to coding to done and checking terminal cleanup removals.

## Solution Assessment

Yes, with the changelog fix. The shared controller is the narrow maintainable boundary because Discord and Slack use `removeReaction` while Telegram's set-only adapter is guarded by the optional capability check; a Discord-only workaround would duplicate lifecycle policy.
