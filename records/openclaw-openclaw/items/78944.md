## Summary

Keep open: current main still has a credible cron run-history routing defect. The dashboard links each run to the per-run session key, but cron persists the session store entry under the base cron key, so chat history lookup for the per-run key can resolve no transcript or the wrong visible session.

## What This Changes

Fix cron run history’s Open run chat action so it opens the specific historical session for the selected cron run.

## Best Possible Solution

Make run-history chat navigation resolve the actual run transcript consistently, either by persisting a resolvable per-run session entry or by teaching chat-history/session resolution to map cron run keys to the recorded transcript reference.

## Reproduction Assessment

Yes from source inspection, but not from a live running Gateway. A cron isolated run log can contain agent:<agent>:cron:<job>:run:<sessionId>, while the session store is written under the base agent:<agent>:cron:<job> key, so opening the run key cannot reliably load that run’s chat history.

## Solution Assessment

Yes, the narrow maintainable fix is in cron session/run-history resolution rather than changing scheduler behavior. The fix should preserve per-run isolation and add regression coverage for opening a historical run from the dashboard.
