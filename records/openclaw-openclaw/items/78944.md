## Summary

Keep open: current main still has a credible cron run-history chat routing defect. The UI links run history to the per-run session key, but the cron session store persists the transcript entry under the stable cron key, so chat history lookup for the per-run key can return no transcript or the wrong visible session.

## What This Changes

Fix cron run-history Open Chat navigation so it opens the correct historical cron conversation for OpenClaw 2026.5.6.

## Best Possible Solution

Make the run-history session key resolve to the actual run transcript, either by persisting a safe per-run session snapshot or by teaching session history lookup to resolve cron run keys to the recorded transcript without weakening isolation.

## Reproduction Assessment

Yes from current source inspection, though not from a live Gateway run. A run log can link to agent:<agent>:cron:<job>:run:<sessionId>, while the session store currently writes only agent:<agent>:cron:<job>, so sessions.get cannot reliably load that run key.

## Solution Assessment

Yes, the narrow maintainable fix is in cron session/run-history resolution rather than scheduler behavior. The fix should keep per-run isolation and add regression coverage for opening a historical run from the dashboard.
