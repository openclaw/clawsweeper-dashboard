## Summary

Keep open. Current main has already solved the generated Linux systemd unit default by emitting `KillMode=control-group`, and that line is present in the latest release tag, but the central diagnostics request remains unimplemented: status/doctor still do not collect, model, print, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`.

## What This Changes

The issue requests Linux systemd gateway hygiene visibility in doctor/status, including KillMode/task/memory counters and safer generated or repaired unit defaults.

## Best Possible Solution

Add a shared read-only Linux systemd hygiene collector used by gateway status and doctor/service audit: parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; include the values in deep status/JSON; warn on suspicious `KillMode=process` plus high task or memory counts; and print concrete `systemctl --user show`, `systemd-cgls --user-unit`, restart, and explicit repair guidance without silently migrating existing units.

## Reproduction Assessment

Yes for the missing diagnostic behavior: mocked `systemctl show` output containing `KillMode=process`, high `TasksCurrent`, and high `MemoryCurrent` would be ignored today because current main neither requests nor models those fields. The original production cgroup accumulation was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the remaining work: a narrow read-only doctor/status observability PR is the best next step because the generated-unit default is already `KillMode=control-group`. A separate maintainer decision can later revisit `mixed` versus `control-group` or migration policy.
