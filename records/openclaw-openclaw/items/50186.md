## Summary

Keep open: current main and v2026.5.19 already use `KillMode=control-group` for generated Linux gateway units, but doctor/status still do not collect or expose `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the requested hygiene visibility remains missing.

## What This Changes

Request adds Linux systemd gateway hygiene diagnostics to doctor/status and considers safer KillMode defaults for generated or repaired gateway units.

## Best Possible Solution

Add shared Linux systemd hygiene collection for `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`, surface it in doctor/status human and JSON output, and warn conservatively while keeping unit-default and migration policy separate.

## Reproduction Assessment

Yes for the remaining diagnostic gap: current source only queries and carries basic systemd runtime state, so `KillMode`, `TasksCurrent`, and `MemoryCurrent` cannot appear in doctor/status. The original production cgroup growth was not replayed live.

## Solution Assessment

Yes: the observability-first path is the narrow maintainable fix because generated units already default to `control-group`; any `mixed` versus `control-group` migration policy should stay separate.
