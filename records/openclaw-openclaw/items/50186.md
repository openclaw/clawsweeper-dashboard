## Summary

Keep open. Current main and v2026.5.18 already generate Linux gateway units with `KillMode=control-group`, but `openclaw doctor` and `openclaw gateway status --deep` still do not query, parse, carry, or print `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains valid.

## What This Changes

Request to expose Linux systemd gateway hygiene fields in doctor/status and reconsider safer systemd KillMode defaults.

## Best Possible Solution

Add shared read-only Linux systemd hygiene collection for gateway status and doctor, expose the raw fields, and warn conservatively while leaving generated-unit defaults unchanged unless maintainers explicitly revisit the policy.

## Reproduction Assessment

Yes for the remaining diagnostic gap: current source only queries and carries basic systemd runtime state, so `KillMode`, `TasksCurrent`, and `MemoryCurrent` cannot appear in doctor/status. The original production cgroup growth was not replayed live.

## Solution Assessment

Yes. Observability-first is the narrow maintainable next step because generated units already default to `control-group`; any `mixed` versus `control-group` migration policy should stay separate.
