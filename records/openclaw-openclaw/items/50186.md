## Summary

Keep open. Current main and the latest release already use `KillMode=control-group` for generated Linux systemd gateway units, but the core diagnostics request remains: doctor/status still do not request, parse, expose, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the reported hygiene failure would remain hard to see.

## What This Changes

Request to add Linux systemd gateway hygiene diagnostics to doctor/status and reconsider safer `KillMode` defaults for generated or repaired gateway units.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector used by gateway status and doctor/service audit, expose the values in deep status/JSON, warn on suspicious `KillMode=process` plus high cgroup task or memory counts, and keep migration/default policy changes separate.

## Reproduction Assessment

Yes for the remaining diagnostic gap: mocked `systemctl show` output containing `KillMode=process`, `TasksCurrent`, and `MemoryCurrent` would be ignored because current main neither requests nor models those fields. The original host-level cgroup growth was not live-reproduced here.

## Solution Assessment

Yes for the next step: observability first is the narrow maintainable fix because the generated-unit default is already `control-group`. Any `mixed` versus `control-group` policy change should be reviewed separately.

## Repair Work Prompt

Implement the first-stage Linux systemd gateway hygiene diagnostics for this issue. Add read-only collection of `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` from `systemctl --user show <gateway>.service`; extend the systemd parser/runtime types; expose the values in `openclaw gateway status --deep` / JSON where service runtime is shown; add doctor/service-audit warnings when `KillMode=process` combines with suspiciously high task or memory counts; print concrete next-step commands including `systemctl --user show ... -p KillMode -p TasksCurrent -p MemoryCurrent`, `systemd-cgls --user-unit openclaw-gateway.service`, and restart guidance. Do not silently migrate existing units and do not change the current `KillMode=control-group` default in this PR unless maintainers explicitly request that policy change. Add focused Vitest coverage with mocked `systemctl` output, update user-facing docs if output changes, and add a `CHANGELOG.md` entry because this is user-visible diagnostics behavior.
