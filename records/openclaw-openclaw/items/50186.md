## Summary

Keep open. Current main and v2026.5.3 already generate Linux gateway units with `KillMode=control-group`, but doctor/status still do not collect or expose `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the main observability request remains valid.

## What This Changes

Request to add Linux systemd gateway hygiene visibility to doctor/status and reconsider generated gateway unit KillMode behavior.

## Best Possible Solution

Add read-only Linux systemd hygiene collection shared by gateway status and doctor, expose the raw fields, and warn conservatively while leaving KillMode policy or migration changes separate.

## Reproduction Assessment

Yes for the remaining diagnostic gap: source inspection shows `systemctl show` output containing `KillMode=process`, `TasksCurrent`, and `MemoryCurrent` would be ignored because current main neither requests nor models those fields. The original production cgroup growth was not live-reproduced here.

## Solution Assessment

Yes for the next step: observability-first is the narrow maintainable path because generated units already default to `control-group`. Any `mixed` versus `control-group` default-policy change should be reviewed separately.

## Repair Work Prompt

Implement a focused fix for https://github.com/openclaw/openclaw/issues/50186. Add read-only Linux user-systemd gateway hygiene diagnostics to `openclaw gateway status --deep`/JSON and `openclaw doctor`: collect `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` via `systemctl --user show <unit> --property ...`, parse them into typed fields, and warn when `KillMode=process` combines with suspicious task or memory counts. Include actionable next-step commands from the issue. Keep generated unit `KillMode=control-group` unchanged, do not silently migrate existing units, and leave `mixed` versus `control-group` policy out of scope. Add regression coverage and a user-facing `CHANGELOG.md` entry.
