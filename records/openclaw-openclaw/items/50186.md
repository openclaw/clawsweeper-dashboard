## Summary

Keep open: current main already uses the safer `KillMode=control-group` for generated Linux systemd units, but doctor/status still do not collect, carry, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`. The related Codex sidecar cleanup merged separately and does not cover this diagnostic surface.

## What This Changes

Request Linux gateway doctor/status visibility for systemd service hygiene fields and conservative warnings around unhealthy cgroup state.

## Best Possible Solution

Add shared Linux systemd hygiene collection for `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`, surface it in doctor/status human and JSON output, and warn conservatively while leaving process reaping and default migration policy separate.

## Reproduction Assessment

Yes for the remaining diagnostic gap: current source only queries and carries basic systemd runtime state, so the requested `KillMode`, `TasksCurrent`, and `MemoryCurrent` values cannot appear in doctor/status. The original production cgroup growth was not replayed live.

## Solution Assessment

Yes: the narrow maintainable path is observability first, because generated units already default to `control-group`; any `mixed` versus `control-group` migration policy should remain separate.
