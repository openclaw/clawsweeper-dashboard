## Summary

Keep open: current main and the latest release already use `KillMode=control-group` for generated Linux systemd units, but doctor/status still do not collect or display `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains actionable.

## What This Changes

Add Linux systemd gateway hygiene diagnostics to `openclaw doctor` and `openclaw status --deep`, and evaluate safer KillMode behavior for generated or repaired gateway units.

## Best Possible Solution

Add shared Linux systemd hygiene fields and conservative doctor/status warnings, while keeping default/migration policy separate unless maintainers explicitly choose to change it.

## Reproduction Assessment

Yes for the remaining diagnostic gap: source inspection shows systemd runtime only requests active/substate/main PID/last-exit fields, and the runtime DTO/formatters have no `KillMode`, `TasksCurrent`, or `MemoryCurrent` slots. I did not live-replay the original production cgroup growth.

## Solution Assessment

Yes: observability first is the narrow maintainable path because generated units already default to `control-group`. Default migration or `mixed` versus `control-group` policy can remain a separate maintainer decision.
