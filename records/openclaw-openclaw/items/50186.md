## Summary

Keep open. Current main and v2026.5.7 already use `KillMode=control-group` for generated Linux gateway units, but doctor/status still do not collect or expose `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains valid.

## What This Changes

Request to expose Linux systemd gateway hygiene fields in doctor/status and consider safer generated-unit KillMode defaults.

## Best Possible Solution

Add read-only Linux systemd hygiene collection shared by gateway status and doctor, expose the raw fields, and warn conservatively while leaving generated-unit defaults unchanged.

## Reproduction Assessment

Yes for the remaining diagnostic gap: current main only requests/parses active state, substate, main PID, and exit fields, so `KillMode`, `TasksCurrent`, and `MemoryCurrent` would be ignored. The original production cgroup growth itself was not live-reproduced.

## Solution Assessment

Yes for the next step: observability-first is the narrow maintainable path because generated units already default to `control-group`. Any `mixed` versus `control-group` migration policy should stay separate unless maintainers explicitly reopen that decision.
