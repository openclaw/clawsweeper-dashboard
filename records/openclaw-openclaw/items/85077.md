## Summary

Current main and shipped releases already provide the requested provider-aware escape hatch: `compat.supportsPromptCacheKey: true` disables prompt-cache stripping for known-compatible OpenAI Responses proxy routes while preserving the default safety behavior for endpoints that reject those fields.

## What This Changes

The issue requests preserving `prompt_cache_key` and `prompt_cache_retention` for compatible third-party `openai-responses` providers, preferably through provider-aware configuration.

## Best Possible Solution

Keep the shipped provider-aware compat flag and default safety behavior; operators with known-compatible Responses proxies should set `compat.supportsPromptCacheKey: true` instead of no-oping the strip globally.

## Reproduction Assessment

Yes for the source path: current main still strips prompt-cache fields for proxy-like `openai-responses` endpoints when `compat.supportsPromptCacheKey` is unset, and preserves them when the flag is true. I did not run a live third-party provider because the repository review was read-only.

## Solution Assessment

Yes. The shipped compat flag is the narrowest maintainable solution because a global no-op would regress endpoints already known to reject `prompt_cache_key`, while the flag lets compatible providers opt in explicitly.
