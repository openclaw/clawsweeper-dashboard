## Summary

Current main and shipped releases already provide the requested provider-aware opt-in: `compat.supportsPromptCacheKey: true` preserves the Responses prompt-cache fields for compatible proxy/custom providers, while the default strip remains for unknown proxy-like endpoints that may reject those fields.

## What This Changes

Request to preserve or make configurable `prompt_cache_key` and `prompt_cache_retention` forwarding for OpenAI-compatible `/v1/responses` providers that support provider-side prompt caching.

## Best Possible Solution

Keep the conservative default strip for unknown proxy-like Responses endpoints and use the shipped per-model compat flag for providers known to forward the OpenAI prompt-cache fields.

## Reproduction Assessment

Yes. Source inspection shows an unflagged custom/proxy `openai-responses` route still resolves `shouldStripResponsesPromptCache: true` and deletes the fields, while `compat.supportsPromptCacheKey: true` disables that strip on current main.

## Solution Assessment

Yes. The shipped compat flag is the maintainable solution because it restores cache fields for known-compatible providers without breaking providers that reject unknown prompt-cache parameters.
