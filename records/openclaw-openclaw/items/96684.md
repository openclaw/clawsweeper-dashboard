## Summary

Keep open: the PR targets a real active-hours scheduling gap, but the patch removes the only explicit loop bound, so millisecond heartbeat configs can synchronously scan a seven-day horizon and stall gateway scheduling; it also still lacks real heartbeat-runner proof.

## What This Changes

The PR removes the `MAX_SEEK_ITERATIONS` guard from `seekNextActivePhaseDueMs` and changes the fallback test from a 1 ms pathological case to a one-hour horizon case.

## Best Possible Solution

Preserve a finite seek budget or minimum seek step that covers legitimate second/sub-minute active-hours schedules over the seven-day horizon while bounding millisecond schedules, then add real runner proof.

## Reproduction Assessment

Yes by source inspection: `heartbeat.every` can parse `1ms`, the runner passes that interval into `seekNextActivePhaseDueMs`, and the PR's horizon-only loop would scan seven days when `isActive` never returns true. I did not execute the hanging path.

## Solution Assessment

No. Removing the static cap is a plausible way to cover more sub-minute slots, but the best fix is to extend supported horizon coverage while preserving a finite safety bound for pathological intervals.
