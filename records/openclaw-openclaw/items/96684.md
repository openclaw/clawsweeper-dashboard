## Summary

Keep this PR open, but the current patch should not merge because it removes the only synchronous loop guard in active-hours heartbeat seeking while config still accepts millisecond and second intervals.

## What This Changes

The PR removes the static iteration cap from `seekNextActivePhaseDueMs` and updates the heartbeat schedule test to assert a seven-day horizon fallback with a one-hour interval.

## Best Possible Solution

Preserve a finite seek budget or minimum seek step that covers legitimate second/sub-minute active-hours schedules over the seven-day horizon while still bounding pathological millisecond intervals.

## Reproduction Assessment

Yes by source inspection: config accepts explicit `ms`/`s` heartbeat durations, the runner passes that interval into the seek helper, and the PR's horizon-only loop would scan 604,800,001 slots for a 1 ms never-active seven-day window. I did not execute the hanging path.

## Solution Assessment

No. Removing the static cap is a plausible way to cover more sub-minute slots, but the best fix is to extend supported horizon coverage while preserving a finite safety bound for pathological intervals.
