## Summary

Keep open. Current main still routes `/voiceclaw/realtime` to Gemini only, while the issue is paired with open draft PR #73032 using closing language; that PR is the right implementation candidate to review rather than closing this issue early or starting a duplicate repair.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, preferably through an OpenAI-Realtime-compatible adapter that can share protocol work with future OpenAI Realtime support.

## Best Possible Solution

Use open draft PR #73032 as the canonical implementation path, with maintainer review deciding the gateway/plugin boundary and requiring mocked WebSocket coverage, provider selection tests, docs for `XAI_API_KEY`, and API-key/error sanitization before closing this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for static/current-main verification: `provider: "xai"` is not accepted by the public type, session startup overwrites the provider to Gemini, and the endpoint requires `GEMINI_API_KEY` rather than `XAI_API_KEY`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review. A shared OpenAI-Realtime-compatible adapter is plausible, but OpenClaw also has a realtime voice plugin seam, so the durable shape should be decided while reviewing the paired draft PR.
