## Summary

Keep open. Current main still exposes `/voiceclaw/realtime` as a Gemini-only gateway path, and open draft PR #73032 is already the paired implementation candidate with closing language, so this issue should not be closed or duplicated before that PR is reviewed.

## What This Changes

The issue requests xAI Realtime Voice Agent support for `/voiceclaw/realtime`, ideally through a shared OpenAI-Realtime-protocol adapter with provider-specific defaults, tests, and docs.

## Best Possible Solution

Use open draft PR #73032 as the canonical implementation path, with maintainer review deciding the gateway-versus-plugin boundary and requiring mocked WebSocket coverage, provider selection tests, docs for `XAI_API_KEY`, and explicit credential/error sanitization before this issue closes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for static current-main verification: `provider: "xai"` is not in the VoiceClaw config type, session startup forces Gemini, and the endpoint checks `GEMINI_API_KEY` rather than an xAI realtime provider path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review. A shared OpenAI-Realtime-compatible adapter is plausible, but current docs and the plugin capability model also point toward provider-owned realtime voice registration, so the paired PR is the right place to settle that shape.
