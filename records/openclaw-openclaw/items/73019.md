## Summary

Keep open. Current main still cannot select xAI for `/voiceclaw/realtime`; it forces Gemini and documents xAI realtime voice as not exposed, while draft PR #73032 is the linked implementation candidate, so this issue should remain open until that PR is reviewed and resolved.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, preferably through a shared OpenAI-Realtime-compatible adapter reusable for xAI and future OpenAI Realtime work.

## Best Possible Solution

Use PR #73032 as the canonical implementation path; settle the gateway/plugin boundary, then require provider dispatch, mocked WebSocket protocol coverage, docs for `XAI_API_KEY`, and secret/error sanitization before closing this issue.

## Reproduction Assessment

Yes. Static current-main verification is enough: `provider: "xai"` is not accepted by the public type, startup forces Gemini and `GEMINI_API_KEY`, and docs say xAI realtime voice is not exposed.

## Solution Assessment

Unclear until maintainer review. A shared OpenAI-Realtime-compatible adapter is plausible, but the existing realtime-voice provider SDK creates a real boundary choice that should be resolved in the paired PR.
