## Summary

Keep open. Current main still lacks xAI support on `/voiceclaw/realtime`, and the supplied timeline shows open draft PR #73032 is already the paired implementation candidate with closing language, so this issue should remain open until that PR is reviewed, merged, or redirected.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, preferably through an OpenAI-Realtime-compatible adapter that could also support future OpenAI Realtime work.

## Best Possible Solution

Use #73032 as the canonical implementation path, with maintainer review deciding the gateway-vs-plugin boundary and requiring focused mocked protocol tests plus docs for provider selection and `XAI_API_KEY` before this issue closes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for current-main verification: static inspection shows `"xai"` is absent from the provider type, session startup forces Gemini, and docs/plugin metadata say xAI realtime voice is not exposed.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A shared OpenAI-Realtime-compatible adapter is plausible, but the maintainable shape depends on whether maintainers want this specific VoiceClaw gateway surface extended directly or routed through the existing realtime-voice plugin capability.
