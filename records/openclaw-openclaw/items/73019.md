## Summary

Keep open. Current main still has a Gemini-only `/voiceclaw/realtime` path and documents xAI realtime voice as not exposed, while open draft PR #73032 explicitly tracks this request, so this issue should remain open until that implementation is reviewed and resolved.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime` through an OpenAI-Realtime-compatible adapter with provider selection, docs, and mocked protocol coverage.

## Best Possible Solution

Use PR #73032 as the canonical path: settle the gateway/plugin boundary, then land provider dispatch, xAI adapter behavior, docs, mocked WebSocket tests, and secret/error sanitization together.

## Reproduction Assessment

Yes. Static current-main verification is enough: the public type does not accept `provider: "xai"`, session startup forces Gemini plus `GEMINI_API_KEY`, and docs say xAI realtime voice is not exposed.

## Solution Assessment

Unclear until maintainer review. A shared OpenAI-Realtime-compatible adapter is plausible, but the repository already has a separate realtime-voice provider plugin seam, so the narrowest maintainable shape should be decided on the paired PR.
