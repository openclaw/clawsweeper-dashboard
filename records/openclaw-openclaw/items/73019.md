## Summary

Keep open. Current main still has a Gemini-only `/voiceclaw/realtime` implementation and the xAI docs still mark realtime voice as not exposed, while open draft PR #73032 is already the paired implementation candidate with closing language.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, preferably through an OpenAI-Realtime-compatible adapter shape that can share protocol work with future OpenAI Realtime support.

## Best Possible Solution

Use the paired draft PR as the canonical implementation path, with maintainer review deciding the gateway-vs-plugin boundary and requiring focused mocked protocol coverage plus docs for `XAI_API_KEY` and provider selection before the issue is closed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for static verification: current main excludes `"xai"` from the provider type, forces session startup to Gemini, and documents xAI realtime voice as unavailable.

## Solution Assessment

Is this the best way to solve the issue? Unclear. An OpenAI-Realtime-compatible adapter is plausible, but the maintainable shape depends on maintainer review of the existing plugin realtime voice capability versus the direct VoiceClaw gateway adapter path.
