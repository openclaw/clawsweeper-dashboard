## Summary

Keep open. Current main still wires `/voiceclaw/realtime` to Gemini only and documents xAI realtime voice as unavailable, while open draft PR #73032 is the paired implementation candidate with closing language.

## What This Changes

Add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, ideally through an OpenAI-Realtime-compatible adapter that can share protocol code with future OpenAI Realtime support.

## Best Possible Solution

Use PR #73032 as the canonical implementation path, with maintainer review settling the gateway-vs-plugin boundary and requiring focused mocked protocol coverage plus docs for provider selection and `XAI_API_KEY` before closing this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for static verification: current main excludes `"xai"` from the provider type, forces session startup to Gemini, and documents xAI realtime voice as unavailable.

## Solution Assessment

Is this the best way to solve the issue? Unclear. An OpenAI-Realtime-compatible adapter is plausible, but the maintainable shape depends on maintainer review of PR #73032 and the existing realtime voice plugin capability boundary.
