## Summary

Keep open. Current main still lacks an xAI provider path for `/voiceclaw/realtime`, the xAI docs explicitly say realtime voice is not exposed, and open draft PR #73032 is already linked as the implementation candidate for this issue, so this should remain open until that PR is reviewed and either merged or closed.

## What This Changes

Request to add xAI Realtime Voice Agent support to `/voiceclaw/realtime`, preferably through a shared OpenAI Realtime-compatible adapter with xAI defaults, provider dispatch, tests, and docs for `XAI_API_KEY`.

## Best Possible Solution

Use open draft PR #73032 as the canonical implementation path. Maintainer review should settle the gateway/plugin boundary, then require provider dispatch, mocked WebSocket protocol coverage, docs for `XAI_API_KEY`, and secret/error sanitization before this issue is closed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for static current-main verification: `provider: "xai"` is not accepted by the public type, `startSession` forces Gemini, and the docs say xAI realtime voice is not exposed.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review. A shared OpenAI-Realtime-compatible adapter is plausible, but current main also has a realtime voice plugin seam and the paired PR should be reviewed against that boundary before this issue is resolved.
