## Summary

Keep open: current main still has source-visible fallback/status logic that treats `anthropic/<model>` and `claude-cli/<model>` as different active routes, and the issue is already paired with an open fixing PR that uses closing syntax.

## What This Changes

The issue asks `/status` to treat Anthropic models executed through the `claude-cli` runtime as the same physical route for fallback, auth-label, and context display.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79082 or an equivalent alias-aware status fix on main, while preserving real fallback notices when the selected and active physical routes truly differ.

## Reproduction Assessment

Yes, source-level: a selected `anthropic/<model>` plus active `claude-cli/<model>` makes current main compare raw provider-qualified refs and persist an active fallback. I did not run live Anthropic Max OAuth, but the failing path is clear in source and independently confirmed on release `2026.5.7`.

## Solution Assessment

Yes: the narrow maintainable fix is to centralize runtime-alias equivalence for fallback/status display and mark CLI usage as a fresh context snapshot when it is the prompt-size signal, without changing provider selection or adding configuration.
