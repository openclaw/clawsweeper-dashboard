## Summary

Keep open: current main still has a source-visible display bug when an Anthropic-selected model is executed through the `claude-cli` runtime alias, and the context freshness path can leave the displayed context numerator unknown while token/cache counters are present.

## What This Changes

The issue asks `/status` to treat `claude-cli/<model>` as the same physical Anthropic model/auth route for fallback, auth-label, and context-usage display.

## Best Possible Solution

Make status/fallback accounting compare canonical physical model/auth routes for known runtime aliases, while keeping real fallback chains visible and persisting a fresh prompt-context snapshot for CLI-backed runs when usage data is available.

## Reproduction Assessment

Yes, source-level: an Anthropic selected model plus `agentRuntime` `claude-cli` becomes a `claude-cli` active provider, and the runner/status paths compare those provider-qualified labels directly. I did not establish a live Anthropic Max OAuth reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing runtime-alias contract for fallback/status equivalence and to pass or derive a fresh context prompt-token snapshot for CLI runs, without changing provider selection or adding configuration.
