## Summary

Keep open: current main still has source-visible `/status` fallback/auth/context logic that treats `anthropic/<model>` and `claude-cli/<model>` as different active routes, and an open linked fixing PR uses closing syntax for this issue.

## What This Changes

The issue requests alias-aware `/status` display for Anthropic models routed through the `claude-cli` runtime, avoiding a spurious fallback line, `unknown` auth, and an unknown context numerator.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79082 or an equivalent alias-aware status fix on main while preserving real fallback notices when selected and active physical routes truly differ.

## Reproduction Assessment

Yes, source-level: a selected `anthropic/<model>` plus active `claude-cli/<model>` makes current main compare raw provider-qualified refs and persist an active fallback. I did not run live Anthropic Max OAuth, but the failing path is clear and independently confirmed by a second reporter on release `2026.5.7`.

## Solution Assessment

Yes: the narrow maintainable fix is to centralize runtime-alias equivalence for fallback/status display and carry the active CLI auth/context snapshot into the displayed selected route without changing provider selection or adding configuration.
