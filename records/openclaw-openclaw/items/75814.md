## Summary

Keep open. Current main still eagerly includes auth-profile SecretRefs in startup secret activation, so a stale provider reference in a stored profile can propagate as a fatal startup-secret error even when normal model-auth selection would catch and skip that profile.

## What This Changes

The issue requests that gateway startup stay available when a stale SecretRef exists only in a non-default or dormant model auth profile, degrading just the affected profile/provider instead of taking the whole gateway offline.

## Best Possible Solution

Define auth-profile active-surface semantics for startup activation: keep eager fail-closed resolution for credentials needed by active gateway/config/runtime surfaces, but mark dormant or non-selected auth-profile refs unresolved/unavailable without aborting startup. Add regression coverage for stale provider aliases and update the secrets/auth credential docs to explain the boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call startup secret activation with an auth store containing an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers`; current main collects that stored profile ref and turns the missing-provider error into a fatal startup wrapper.

## Solution Assessment

Unclear pending owner approval. The requested behavior is directionally right, but the best fix should be a narrow active-profile/startup-scope rule rather than a blanket lazy-resolution or retry change that weakens required-secret handling.
