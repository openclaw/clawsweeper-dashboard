## Summary

Keep open. Current main still resolves all auth-profile SecretRefs during startup and wraps a missing provider as a fatal required-secrets startup error; the referenced timeline commit only updates contributor-status notes, not the implementation.

## What This Changes

The issue asks gateway startup to stay online when a non-default generated model auth profile references a removed SecretRef provider.

## Best Possible Solution

Add explicit active auth-profile SecretRef semantics so startup fails closed for active/default routes while marking dormant stale profiles unavailable with targeted diagnostics and doctor repair guidance.

## Reproduction Assessment

Yes. A source-level reproduction is to call `prepareSecretsRuntimeSnapshot` with an auth store containing an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers`; current tests already expect that default path to reject.

## Solution Assessment

Unclear pending owner approval. The maintainable path is a narrow active-profile startup filter plus request/status degradation for dormant stale profiles, not a broad relaxation of required-secret resolution.
