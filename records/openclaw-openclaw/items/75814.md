## Summary

Keep open. Current main still batches auth-store SecretRefs into startup-wide secret activation, so a stale provider reference in any stored auth profile can still fail gateway startup; the related fix PR is closed unmerged and its head is not present on main.

## What This Changes

Gateway startup should degrade only the affected auth profile/provider when a non-default generated auth profile contains a stale SecretRef provider reference.

## Best Possible Solution

Split config SecretRef resolution from auth-store profile SecretRef resolution so config refs remain startup-fatal, while stale auth-profile refs produce a targeted warning and leave that credential unusable without taking the gateway offline.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load an auth store with an `api_key.keyRef` or `token.tokenRef` whose provider is absent from `secrets.providers`, then call default startup secret activation; current main collects that ref, throws the missing-provider error, and wraps it as a startup failure.

## Solution Assessment

Yes. The narrow maintainable fix is to keep strict config-level SecretRef startup checks, but resolve auth-store profile refs separately so only the affected profile degrades.
