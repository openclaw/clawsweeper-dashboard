## Summary

Keep open. Current main still resolves every stored auth-profile SecretRef during startup preflight and wraps any resolution failure as a fatal gateway startup error; the linked fix PR is open and unmerged, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue asks for stale SecretRefs in non-default generated model auth profiles to degrade only the affected profile instead of aborting gateway startup.

## Best Possible Solution

Land a narrow fix like #76118 that keeps config-required SecretRefs fatal while degrading only dormant auth-profile SecretRef failures into targeted warnings and unavailable profiles.

## Reproduction Assessment

Yes. Load an auth store with an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers`, then call startup secret activation with default auth-store inclusion; current code resolves that ref and converts the missing provider error into a fatal startup failure.

## Solution Assessment

Unclear until the linked PR is reviewed, but the proposed boundary is the right shape: preserve fatal startup checks for required config secrets and handle stale non-default auth profiles as profile-scoped degraded state.
