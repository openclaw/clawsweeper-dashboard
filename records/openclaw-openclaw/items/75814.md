## Summary

Keep open. Current main still batches auth-profile SecretRefs into startup-wide secret activation, and the linked open PR with closing syntax is the active implementation candidate, so this issue should remain open until that PR lands or is closed.

## What This Changes

Gateway startup should degrade only the affected non-default auth profile when an auth-profiles SecretRef points at a removed provider.

## Best Possible Solution

Land a reviewed fix that keeps config SecretRefs strict but handles auth-store `keyRef`/`tokenRef` failures per profile with a warning/status signal and regression coverage.

## Reproduction Assessment

Yes. A focused unit reproduction is to load an auth store with an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers`, then call startup secret activation or default `prepareSecretsRuntimeSnapshot`; current source and tests show that path rejects and startup wraps it as fatal.

## Solution Assessment

Unclear until PR #76118 is reviewed. Its described approach, splitting config assignment resolution from auth-store assignment resolution, appears to be the narrow maintainable direction if it preserves fail-closed startup behavior for required credentials.
