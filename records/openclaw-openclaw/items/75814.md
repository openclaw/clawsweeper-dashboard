## Summary

Keep open. Current main still resolves auth-store SecretRefs as part of startup-wide secret activation, so a stale non-default auth profile reference can still abort gateway startup; the linked fix PR is closed unmerged and is not present on main.

## What This Changes

The issue requests that stale SecretRefs in generated, non-default model auth profiles degrade only the affected profile instead of taking the gateway offline.

## Best Possible Solution

Split config SecretRef resolution from auth-store profile SecretRef resolution so config remains startup-fatal, while stale auth-profile `keyRef`/`tokenRef` failures produce a targeted warning/status and leave only that profile unavailable with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load an auth store containing an `api_key.keyRef` or `token.tokenRef` whose provider is absent from `secrets.providers`, then call default `prepareSecretsRuntimeSnapshot` or startup secret activation; current main collects that ref, throws the missing-provider error, and wraps it as a startup failure.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve strict startup behavior for config-level SecretRefs but handle auth-store profile ref failures per profile, matching existing profile-resolution behavior that treats unresolved refs as unavailable credentials.
