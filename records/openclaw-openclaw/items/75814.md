## Summary

Keep open. Current main still resolves stored auth-profile SecretRefs in the startup-wide strict secret-resolution path, and the active closing implementation is open and unmerged.

## What This Changes

Gateway startup should stay online when a stale SecretRef exists only in a non-default stored model auth profile, degrading that profile instead of aborting the gateway.

## Best Possible Solution

Review and land the open fix PR or an equivalent targeted patch that keeps config/reload SecretRefs strict while degrading only stale stored auth-profile refs with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load a stored auth profile with an `api_key.keyRef` or `token.tokenRef` pointing to an absent provider, then run startup secret activation; main batches that ref into strict resolution and wraps the provider error as a startup failure.

## Solution Assessment

Yes. The narrow maintainable fix is to split config SecretRef resolution from auth-store profile ref handling so only the stale profile becomes unavailable while required config secrets remain startup-fatal.
