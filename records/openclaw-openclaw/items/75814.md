## Summary

Keep open. Current main still resolves auth-profile SecretRefs as part of startup-wide secret activation and fails startup on the first unresolved ref; #76118 is open with closing syntax and should be reviewed or landed before closing this issue.

## What This Changes

The issue asks the gateway to stay up when a non-default generated model auth profile contains a stale SecretRef provider, degrading only that profile/provider.

## Best Possible Solution

Review and land or close #76118, preserving strict config SecretRef startup failures while degrading stale auth-store refs only for the affected auth profile with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load an auth store with an `api_key.keyRef` or `tokenRef` whose provider is absent, then call default `prepareSecretsRuntimeSnapshot`; startup activation wraps that resolver failure as fatal.

## Solution Assessment

Yes for the issue direction. The narrow maintainable fix is to split required config SecretRef resolution from auth-store profile resolution, keeping config failures strict while warning and marking only stale auth profiles unavailable.
