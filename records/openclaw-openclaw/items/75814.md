## Summary

Keep open. Current main still resolves every auth-profile `keyRef`/`tokenRef` during startup activation and wraps any missing provider as a fatal startup error, so the reported v2026.4.29 gateway abort path remains valid.

## What This Changes

Request gateway startup to degrade stale SecretRefs in non-default generated auth profiles instead of aborting the whole gateway.

## Best Possible Solution

Define startup active-profile SecretRef semantics: fail closed for credentials needed by active/default routes, but mark dormant stale profiles unavailable without taking the gateway offline.

## Reproduction Assessment

Yes. A source-level reproduction is to load an auth store with an `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent from `secrets.providers` and call startup secret activation; current main resolves that ref and wraps the missing-provider error as fatal.

## Solution Assessment

Unclear pending maintainer approval. A narrow active-profile filter is likely maintainable, while broad lazy resolution would risk weakening required credential failures.
