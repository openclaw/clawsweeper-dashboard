## Summary

Keep open. Current main still treats auth-profile SecretRefs as part of startup-wide secret activation, so a stale non-default profile can still turn a missing provider into a fatal gateway startup error.

## What This Changes

The issue asks for gateway startup to stay available when a stale SecretRef appears only in a non-default model auth profile.

## Best Possible Solution

Define active auth-profile SecretRef semantics for startup, keeping fail-closed behavior for credentials needed by active routes while marking dormant stale profiles unavailable for status/probe and request-time selection.

## Reproduction Assessment

Yes. A source-level reproduction is to load an auth store with an `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent from `secrets.providers` and call startup secret activation; current code resolves that ref during startup and wraps the missing-provider error as fatal.

## Solution Assessment

Unclear pending maintainer approval. The requested direction is maintainable if implemented as a narrow active-profile filter, but a broad lazy-resolution change would risk weakening required credential failures.
