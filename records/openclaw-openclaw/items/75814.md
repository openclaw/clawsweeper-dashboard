## Summary

Keep open. Current main still eagerly resolves loaded auth-profile SecretRefs during gateway startup, so a stale provider reference in a dormant profile can become a fatal startup error. The request-time model auth path already catches per-profile resolution failures, which supports the report; the remaining fix needs maintainer approval because it changes SecretRef fail-closed activation semantics.

## What This Changes

The issue requests that gateway startup tolerate stale SecretRefs in non-default model auth profiles and degrade only the affected profile/provider instead of taking the gateway offline.

## Best Possible Solution

Define auth-profile active-surface semantics for gateway startup: resolve and fail closed for profiles that can be selected by the active/default provider routing path, but keep dormant or explicitly non-selected stale profiles as unavailable with status/probe diagnostics instead of aborting the gateway. Add focused tests and update the secrets/auth credential docs to make the startup boundary explicit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to prepare a secrets runtime snapshot or gateway startup activation with an auth store containing an `api_key.keyRef` or `tokenRef` whose SecretRef provider is missing from `secrets.providers`; current main resolves all auth-store refs and wraps the missing-provider error as a fatal startup failure.

## Solution Assessment

Unclear pending owner approval. The requested behavior is directionally right, but the maintainable fix should be a narrow active-profile filtering rule for auth-profile refs, not a generic retry or blanket downgrade of all startup SecretRef failures.
