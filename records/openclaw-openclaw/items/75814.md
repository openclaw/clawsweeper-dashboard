## Summary

Keep open. Current main still resolves every stored auth-profile SecretRef during startup secret activation, and any unresolved provider is wrapped into a fatal gateway startup error even when the profile is unrelated to the default route.

## What This Changes

The issue asks the gateway to stay available when a stale SecretRef exists only in a non-default model auth profile, degrading that profile instead of aborting startup.

## Best Possible Solution

Define an active auth-profile SecretRef boundary for gateway startup, then keep required active-route credentials fail-closed while marking stale dormant profiles unavailable for status and request-time selection.

## Reproduction Assessment

Yes. A source-level reproduction is to load an auth store with an `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent from `secrets.providers` and call startup secret activation; current code resolves that ref and wraps the missing-provider error as fatal.

## Solution Assessment

Unclear until maintainers approve the credential boundary. The requested direction is maintainable as a narrow active-profile filter, but a broad skip of auth-profile SecretRefs would be too risky.
