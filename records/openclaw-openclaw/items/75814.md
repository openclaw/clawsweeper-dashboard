## Summary

Keep open. Current main still resolves all auth-store SecretRefs in the startup-wide batch and wraps unresolved provider failures as fatal startup errors; open PR #76118 already uses closing syntax and is the active fix candidate.

## What This Changes

The issue requests that a stale SecretRef in a non-default auth profile degrade only that profile/provider instead of aborting gateway startup.

## Best Possible Solution

Land a reviewed fix that separates config-required SecretRef failures from auth-store profile failures, preserving strict startup behavior for required config credentials while warning and disabling only the affected profile.

## Reproduction Assessment

Yes, source-level. Load an auth store with an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers`, then call the default startup snapshot/activation path; current main routes the provider-resolution error into the fatal startup wrapper.

## Solution Assessment

Yes, the narrow maintainable fix is to split config assignment resolution from auth-store assignment resolution and downgrade only auth-profile failures. PR #76118 appears to target that path, so this issue should not close before that implementation is reviewed and merged.
