## Summary

Keep open. Current main still eagerly resolves auth-profile SecretRefs during startup secret activation and still treats any activation failure as fatal, so a stale non-default profile can still abort gateway startup.

## What This Changes

The issue asks for gateway startup to stay available when a stale SecretRef appears only in a non-default/generated model auth profile.

## Best Possible Solution

Define active auth-profile SecretRef semantics for startup, filter or defer only dormant profile refs, keep active route credentials fail-fast, and add focused regression coverage around gateway startup plus auth-store snapshot preparation.

## Reproduction Assessment

Yes. Load an auth store containing an api_key.keyRef or tokenRef whose SecretRef provider is absent, then call startup secret activation with the default includeAuthStoreRefs behavior; current main collects that ref and wraps the missing-provider error as a fatal startup failure.

## Solution Assessment

Unclear pending maintainer approval. The requested degradation is directionally sound if limited to dormant profiles, but broad lazy resolution would weaken current required-secret startup guarantees.
