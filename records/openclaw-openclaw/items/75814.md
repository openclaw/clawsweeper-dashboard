## Summary

Keep open. Current main still resolves auth-profile SecretRefs during startup-wide secret activation and converts a missing provider into a fatal required-secrets startup error, matching the reported failure mode.

## What This Changes

A stale SecretRef provider in a non-default generated auth profile should degrade only that profile instead of aborting gateway startup.

## Best Possible Solution

Add an explicit active-profile startup boundary so required route credentials still fail closed, while dormant stale auth profiles are marked unavailable and resolved again when selected.

## Reproduction Assessment

Yes. Load an auth store containing an `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent from `secrets.providers`, then call startup secret activation or default `prepareSecretsRuntimeSnapshot`; current main resolves that profile and throws the missing-provider error.

## Solution Assessment

Unclear pending maintainer approval. A narrow active-profile filter is maintainable, but a broad soft-fail for all auth-profile SecretRefs could hide credentials that really are required for startup or default routing.
