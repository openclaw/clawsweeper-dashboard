## Summary

Keep open. Current main still eagerly resolves every loaded auth-profile SecretRef during gateway startup, so a stale non-default auth profile can bubble a missing-provider error into the fatal startup wrapper. The normal provider selection path is more tolerant and catches failed profile resolution, which supports the report that startup is treating dormant auth-profile refs as globally required.

## What This Changes

The issue reports that gateway startup aborts when a generated non-default model auth profile still references a removed SecretRef provider.

## Best Possible Solution

Define and implement auth-profile active-surface semantics for startup: keep fail-closed eager resolution for active gateway/config credentials, but do not let dormant or non-selected auth-profile SecretRefs abort gateway startup. The affected profile should remain present with an unavailable/unresolved status so `models status --probe` and request-time model selection can report or skip it, with docs and regression tests covering stale provider aliases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to load an auth store containing an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers` and call startup activation; current code resolves all auth-store refs by default and wraps the provider error as a fatal startup failure.

## Solution Assessment

Unclear pending owner approval. The requested behavior is directionally right, but the safest fix is not generic retry; it is a narrow active-surface/lazy-resolution rule for auth-profile refs that preserves fail-closed behavior for credentials actually needed at startup.
