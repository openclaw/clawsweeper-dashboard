## Summary

Keep open. Current main still batches stored auth-profile SecretRefs into startup-wide secrets resolution, and the related fix PR was closed unmerged, so a stale non-default auth profile reference can still abort gateway startup.

## What This Changes

The issue asks the gateway to stay online when a stale SecretRef exists only in a non-default model auth profile, while marking only that profile/provider unavailable.

## Best Possible Solution

Keep config-level and effectively active SecretRefs startup-fatal, but resolve auth-store profile SecretRefs separately so stale non-selected profiles become profile-level unavailable state with a targeted warning/status and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load an auth store containing an `api_key.keyRef` or `token.tokenRef` whose provider is absent from `secrets.providers`, then call startup secret activation; current main collects that ref and routes the missing-provider error into the startup-fatal path.

## Solution Assessment

Yes. The narrow maintainable fix is to split config SecretRef resolution from auth-store profile SecretRef resolution, preserving strict startup behavior for required config refs while degrading only the affected auth profile.
