## Summary

Keep open. Current main still resolves every loaded auth-profile SecretRef during gateway startup and wraps any resolution failure as a fatal startup error, while the request-time model-auth path already catches failed profile resolution and continues to other profiles or credentials.

## What This Changes

The issue asks for gateway startup to stay available when a stale SecretRef appears only in a non-default generated model auth profile, degrading only that affected profile/provider.

## Best Possible Solution

Define an explicit active-surface rule for auth-profile startup SecretRefs: keep fail-closed eager resolution for gateway/config credentials and actively selected/default model credentials, but do not let dormant auth-profile refs abort startup. The affected profile should remain visible as unavailable or unresolved for model status/probe and request-time selection, with docs and regression coverage for stale provider aliases.

## Reproduction Assessment

Yes. A source-level reproduction is to load an auth store containing an api_key.keyRef or tokenRef whose provider is absent from secrets.providers and call startup activation; current code collects all auth-store refs by default and wraps the missing-provider error as a fatal startup failure.

## Solution Assessment

Unclear pending owner approval. The requested behavior is directionally right, but the maintainable fix should be a narrow active-profile/startup-surface rule, not a generic retry or blanket skip of auth-store SecretRefs.
