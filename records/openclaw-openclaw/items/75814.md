## Summary

Keep open. Current main still eagerly includes auth-profile SecretRefs in startup-wide secret activation, so a stale non-default profile can still become a fatal gateway startup error instead of a profile-scoped degraded state.

## What This Changes

Stale SecretRefs in non-default generated auth profiles should not bring down gateway startup when unrelated default routing remains valid.

## Best Possible Solution

Define active auth-profile SecretRef semantics for startup: fail fast for credentials needed by active routes, but keep dormant stale profiles unavailable until status/probe or request-time selection.

## Reproduction Assessment

Yes. Load an auth store containing an `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent from `secrets.providers`, then call startup secret activation with default `includeAuthStoreRefs`; current code resolves that ref and wraps the missing-provider error as fatal.

## Solution Assessment

Unclear pending maintainer approval. The requested direction is maintainable as a narrow active-profile filter, while broad lazy resolution would risk weakening required credential failures.
