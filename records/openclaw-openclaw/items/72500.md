## Summary

Keep open. Current main still conflates transient Control UI WebSocket disconnects with the first-connect/auth gate state, and PR #72522 is the active unmerged implementation path rather than a reason to close this issue early.

## What This Changes

Prevent Control UI WebChat from flashing the credentials gate when an already-authenticated browser tab transiently disconnects and reconnects.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI change that remembers a successful session, keeps the chat UI mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth/pairing/device-identity failure states, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The code path is high confidence: a recoverable WebSocket close invokes onClose, clears connected, renderApp immediately selects the login gate, and reconnect scheduling happens after that callback.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable direction; a timing-only debounce would mask the flash while still conflating transient transport loss with first-connect and auth-failure states.
