## Summary

Current main still conflates a transient Control UI WebSocket disconnect with the first-connect credentials gate, and the active open PR with closing syntax should be reviewed or replaced before this issue is closed.

## What This Changes

The issue requests that Control UI webchat distinguish transient reconnects from first-connect or auth-failure states so the credentials gate does not flash after an already authenticated tab regains focus.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72522 or an equivalent narrow Control UI change that records a successful session, keeps the chat UI mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth/pairing/device-identity failures, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The current source path is high confidence: after a successful session, a recoverable WebSocket close invokes onClose, clears connected, renderApp immediately selects the login gate, and reconnect scheduling happens afterward.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable direction; a timing-only debounce would mask the flash while still conflating transient transport loss with first-connect and auth-failure states.
