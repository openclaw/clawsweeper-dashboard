## Summary

Keep open. Current main still routes any disconnected Control UI state to the credentials gate, and the gateway close path clears `connected` before recoverable reconnect scheduling; the linked open PR #72522 remains the implementation candidate and is not merged.

## What This Changes

The issue requests a Control UI WebChat reconnect state that keeps chat visible after a prior authenticated session unless authentication, pairing, or device identity explicitly fails.

## Best Possible Solution

Land #72522 or an equivalent narrow Control UI fix that records a successful session, keeps chat mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth/pairing/device-identity failures, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The source path is high confidence: after a successful hello, a recoverable WebSocket close calls `onClose`, clears `connected`, and `renderApp` immediately returns the full login gate before reconnect scheduling completes.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable fix direction; a timing-only debounce would mask the flash while still conflating transient transport loss with first-connect and auth-failure states.
