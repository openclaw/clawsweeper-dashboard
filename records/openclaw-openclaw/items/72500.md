## Summary

Keep open. Current main still renders the full Control UI credentials gate whenever `connected` is false, and #72522 remains the open unmerged implementation candidate for the same issue.

## What This Changes

The issue asks Control UI WebChat to keep the chat UI mounted, or show a reconnecting state, during transient reconnects after a previously authenticated session.

## Best Possible Solution

Review and land #72522, or an equivalent narrow Control UI fix, so first-connect/auth-failure remains distinct from transient transport reconnects and the behavior is covered by focused tests.

## Reproduction Assessment

Yes. The source path is high confidence: after a successful connection, a recoverable close sets `connected = false`, and `renderApp` immediately returns the credentials login gate before reconnect restores chat.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable fix direction; a timing-only debounce would still conflate transient transport loss with first-connect and auth-failure states.
