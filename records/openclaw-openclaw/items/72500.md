## Summary

Keep open. Current main still conflates transient reconnects with first-connect/auth-failure disconnected states, and the linked open PR #72522 is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

Avoid flashing the Control UI webchat credentials gate during transient reconnects after a previously authenticated session.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that keeps chat mounted after a prior successful session during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth/pairing/device failures, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. On current main, a recoverable WebSocket close reaches the active onClose handler, sets connected=false, and renderApp immediately selects the credentials gate because there is no sticky prior-success or reconnecting state.

## Solution Assessment

Yes, as a direction. A prior-successful-session or explicit reconnecting state is the narrow maintainable solution; a timing-only debounce would hide the flash while still conflating transient transport loss with real disconnected/auth states.
