## Summary

Keep open. Current main and the latest release still render the full credentials gate for any disconnected Control UI state, and both linked fix PRs are closed unmerged, so the central reconnect/login-gate bug is not implemented or safely superseded.

## What This Changes

The issue asks OpenClaw to keep the authenticated Control UI shell mounted during transient webchat or Settings reconnects instead of flashing the credentials gate after a successful session.

## Best Possible Solution

Add a Control UI reconnect/prior-success state that keeps the authenticated shell mounted for transient closes, returns to the credentials gate for first connect and explicit auth/pairing/revocation failures, and covers webchat plus the related Settings/macOS surfaces with focused regression tests.

## Reproduction Assessment

Yes, at source level: a successful session followed by `connected=false` reaches `renderApp`, which currently returns the credentials gate without checking prior `hello` or reconnect state. I did not run a live browser/Pi repro in this read-only pass.

## Solution Assessment

The issue’s requested direction is right, but the best fix is not a simple `!state.hello` gate because `connectGateway` clears `hello` during reconnect startup. The maintainable path needs a dedicated prior-success/reconnecting state plus explicit auth and revocation reset coverage.
