## Summary

Keep open. Current main still has the reported Control UI reconnect behavior: disconnect paths immediately set `connected` false, and the renderer still maps that state to the full credentials gate; #72522 is the open closing PR candidate and should resolve this before the issue closes.

## What This Changes

The issue requests that Control UI WebChat preserve the authenticated chat UI during transient reconnects instead of briefly rendering the credentials gate.

## Best Possible Solution

Land #72522 or an equivalent narrow fix that tracks prior successful authentication and keeps chat mounted across transient transport closes while returning to the credentials gate only before first connection or after explicit auth/pairing failure.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after a successful session, a recoverable close sets `connected` false and `renderApp` immediately returns the credentials gate before reconnect restores chat.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the narrow maintainable direction; a timing-only debounce would still conflate transient transport loss with first-connect and auth-failure states.
