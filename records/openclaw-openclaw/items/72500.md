- `impact:session-state`: The issue concerns authenticated Control UI session presentation being replaced by unauthenticated credentials state during transient reconnects.

## Summary

Keep open: current main and the latest release still route every disconnected Control UI render to the credentials gate, and the related fix PRs are closed unmerged with auth-boundary or proof blockers rather than a safe canonical landing path.

## What This Changes

The issue requests keeping an already-authenticated Control UI webchat/settings/dashboard view mounted during transient Gateway reconnects instead of briefly flashing the credentials gate.

## Best Possible Solution

Add an auth-safe prior-success or reconnecting state in the Control UI that preserves the shell only for transient transport reconnects and resets to the credentials gate for first connect and explicit auth, pairing, scope, device, or revocation failures.

## Reproduction Assessment

Yes, at source level: reconnect startup and socket close set connected=false, and renderApp returns the credentials gate for every disconnected state before the dashboard shell can render. I did not run live browser or Pi proof in this read-only pass.

## Solution Assessment

Unclear as a final patch: the sticky prior-success/reconnecting direction is narrow and plausible, but previous PRs show the auth and revocation reset contract needs maintainer-confirmed behavior before another fix lands.
