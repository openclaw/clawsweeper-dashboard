## Summary

Keep open. Current main still has the reported Control UI reconnect behavior, and the linked closing PR remains the active unmerged implementation path rather than a reason to close this issue early.

## What This Changes

Prevent Control UI WebChat from showing the credentials/login gate during transient reconnects after a previously successful authenticated session.

## Best Possible Solution

Review and land #72522 or an equivalent narrow Control UI fix that keeps authenticated chat mounted across transient disconnects and returns to the credentials gate only before first success or after explicit auth/pairing failure.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: after a successful session, a recoverable close sets connected to false and renderApp immediately returns the credentials gate before reconnect restores chat.

## Solution Assessment

Yes. A sticky prior-success or explicit reconnecting state is the narrow maintainable direction; a timing-only debounce would still conflate transient transport loss with first-connect and auth-failure states.
