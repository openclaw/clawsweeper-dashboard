## Summary

Keep open. Current main still renders the Control UI credentials gate for every disconnected state, and the open linked PR #72522 is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests that Control UI WebChat distinguish transient reconnects after a successful session from first-connect or explicit authentication failures so the credentials gate does not flash on tab refocus.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that remembers a successful session, keeps the chat UI mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth failure states, and covers those state transitions with focused tests.

## Reproduction Assessment

Yes. The deterministic reproduction path is visible in current main: after a successful connection, a recoverable WebSocket close sets connected=false and renderApp immediately selects the credentials-gate branch before reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or reconnecting state is the narrow maintainable solution; a timing-only debounce would mask the flash but still conflate transient transport disconnects with first-connect and auth-failure states.
