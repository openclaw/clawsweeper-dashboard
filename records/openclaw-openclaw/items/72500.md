## Summary

Keep open. Current main still has the reported two-state Control UI reconnect behavior, and the linked implementation PR is open and unmerged with closing syntax for this issue.

## What This Changes

The issue requests preserving the authenticated Control UI webchat across transient WebSocket reconnects instead of rendering the credentials gate whenever `connected` is false.

## Best Possible Solution

Land a corrected version of the linked PR, or a replacement, that preserves prior authenticated chat UI across transient reconnects while returning to the credentials gate on first connect and explicit auth, pairing, or revocation failure.

## Reproduction Assessment

Yes. Current main clears `connected` on connect and close, and `renderApp` immediately renders the credentials gate for any disconnected state, so a transient close after an authenticated session follows the reported path.

## Solution Assessment

Yes, directionally. A sticky prior-successful-session or explicit reconnecting state is the narrow maintainable fix; the linked PR is the active candidate, but the issue should remain open until that work lands or is replaced.
