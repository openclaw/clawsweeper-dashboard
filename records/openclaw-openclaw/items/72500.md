## Summary

Keep open. Current main still renders the Control UI credentials gate for every disconnected state, the latest shipped release contains the same behavior, and open PR #72522 is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The request is to keep authenticated Control UI webchat mounted across transient WebSocket reconnects instead of flashing the credentials gate.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that records a successful authenticated session, keeps chat content mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth-failure states, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The source-level path is deterministic on current main: after a successful connection, a recoverable WebSocket close calls `onClose`, clears `connected`, and `renderApp` immediately selects the credentials-gate branch before reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or reconnecting state is the narrow maintainable solution; a timing-only debounce would reduce the flash but still conflate transient transport disconnects with first-connect and auth-failure states.
