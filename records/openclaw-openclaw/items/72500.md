## Summary

Keep open. Current main still has the reported two-state Control UI behavior, and #72522 is an open, unmerged PR that explicitly fixes this issue.

## What This Changes

Prevent transient Control UI WebChat reconnects from briefly rendering the full credentials gate after a session has already authenticated.

## Best Possible Solution

Land #72522 or an equivalent narrow Control UI fix that preserves authenticated chat UI across transient reconnects while still showing the credentials gate for first connect and explicit auth or pairing failures.

## Reproduction Assessment

Yes. The source path is high confidence: after a successful connection, a transient close sets `connected` false and `renderApp` immediately returns the login gate before reconnect can restore chat.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable fix; a timing-only debounce would still conflate transient transport loss with first-connect and auth-failure states.
