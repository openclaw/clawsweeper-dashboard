## Summary

Keep open. Current main still has the reported Control UI state collapse from transient reconnect to credentials gate, and the linked closing PR #72522 is still open and unmerged, so this issue should not be closed before that implementation is reviewed, revised if needed, and landed or replaced.

## What This Changes

The issue requests that Control UI WebChat distinguish first-connect or explicit auth failure from transient reconnects so tab refocus does not briefly render the full credentials gate after a prior successful session.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that keeps chat mounted during recoverable transport reconnects, restores the credentials gate for first-connect and explicit auth/device/pairing revocation states, and includes focused gateway/client tests for both transient closes and code-only auth revocation closes.

## Reproduction Assessment

Yes. The deterministic source-level reproduction is current main after a successful connection: a recoverable WebSocket close sets connected=false, GatewayBrowserClient schedules reconnect afterward, and renderApp immediately selects the credentials-gate branch.

## Solution Assessment

Yes for the requested direction, but not finished in main. A sticky prior-success or reconnecting state is the narrow maintainable fix; a timing-only debounce would mask the flash while leaving transient disconnects conflated with first-connect and auth-failure states.
