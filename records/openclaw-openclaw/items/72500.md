## Summary

Keep open. Current main still renders the credentials gate for every disconnected Control UI state, and the open paired PR is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests a Control UI/WebChat reconnect state that keeps an already-authenticated chat view mounted across transient tab-refocus or WebSocket reconnects instead of flashing the credentials gate.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that remembers a successful session, keeps the chat UI mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth failure states, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The reproduction path is high confidence from current main: after a successful connection, a recoverable WebSocket close calls onClose, sets connected=false, and renderApp immediately selects the credentials-gate branch before reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or reconnecting state is the narrow maintainable direction; a pure timing debounce would only mask the flash while still conflating transient transport loss with first-connect and auth-failure states.
