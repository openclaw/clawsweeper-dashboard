## Summary

Keep open: current main and the latest release still render the credentials gate for every disconnected Control UI state, while the related fix PRs are closed unmerged and no newer open replacement owns the work.

## What This Changes

The issue requests an auth-safe Control UI reconnect state so transient WebSocket disconnects do not replace an already authenticated chat or settings view with the credentials gate.

## Best Possible Solution

Add an auth-safe prior-success or reconnecting state in Control UI that keeps the shell mounted for transient transport closes and returns to the credentials gate for first connect and explicit auth, pairing, scope, or revocation failures.

## Reproduction Assessment

Yes, at source level: after a previously successful Control UI session, reconnect startup and socket close set `connected=false`, and current `renderApp` returns the credentials gate for every disconnected state.

## Solution Assessment

Unclear as a final patch: the direction is right, but the fix needs an auth-safe prior-success or reconnecting state rather than a broad `hello` check because reconnect startup clears `hello` before the new connection succeeds.
