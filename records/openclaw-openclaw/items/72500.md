## Summary

Keep open: current main still renders the credentials gate for every disconnected Control UI state, the latest release has the same behavior, and the prior fix PRs were closed unmerged rather than landing the requested auth-safe reconnect state.

## What This Changes

The issue requests that Control UI keep an already-authenticated chat/dashboard mounted during transient WebSocket reconnects instead of flashing the credentials gate.

## Best Possible Solution

Add an auth-safe prior-success or reconnecting state in Control UI that keeps the shell mounted for transient transport closes, returns to the credentials gate for first connect and nonrecoverable auth/pairing/scope/revocation failures, and proves both paths with focused tests plus redacted browser proof.

## Reproduction Assessment

Yes, at source level: a successful Control UI session followed by a transient close sets `connected=false`, and current `renderApp` immediately returns the credentials gate. I did not run a live browser or Pi repro in this read-only review.

## Solution Assessment

Yes on direction, but not as a simple `hello` check because `connectGateway` clears `hello` during reconnect startup. The maintainable fix is a dedicated prior-success/reconnecting state aligned with the gateway nonrecoverable-auth contract.
