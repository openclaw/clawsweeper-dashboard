## Summary

Current main does not solve the automatic-delivery case: its newer terminal-send hook explicitly excludes automatic mode, and the only linked implementation was closed unmerged after review found two message-loss regressions. The issue should remain the canonical focused report while maintainers define a safe terminal-reply signal with source-thread identity.

## What This Changes

Ensure a successful terminal message-tool reply to the current source conversation is recorded so a later private final cannot replace or duplicate it.

## Best Possible Solution

Introduce an explicit terminal-source-reply fact at the outbound/runner boundary that requires positive delivery, terminal intent, and exact channel plus thread identity, then use that fact to suppress only later private finals in automatic mode.

## Reproduction Assessment

Yes, at source level with high confidence: current main explicitly does not install terminal-send handling for automatic delivery mode, matching the reported path. The discussion also provides independent Telegram direct and group observations, though this review did not run a live reproduction.

## Solution Assessment

No, the closed linked PR was not the best solution because delivery success alone did not distinguish terminal replies from interim sends and source matching omitted thread identity. The best fix needs a narrower typed terminal-source-reply contract shared by outbound delivery and final selection.
