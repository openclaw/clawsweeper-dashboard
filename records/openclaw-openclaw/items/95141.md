## Summary

Keep open: current main still has a WebChat-specific restart-recovery gap. The prior merged restart-marker work helps adjacent restart paths, but WebChat unresumable notices still do not have a native delivery path and the reporter reproduced the empty failed session on the latest release.

## What This Changes

The issue asks OpenClaw to stop silently losing an in-flight WebChat turn when a config-triggered Gateway restart interrupts it, by deferring restart, persisting the user turn, or showing a WebChat recovery notice.

## Best Possible Solution

Add a WebChat-native restart-recovery path that either preserves the inbound user turn before restart can kill the process or persists a terminal recovery notice into the transcript read by chat.history, while keeping external-channel recovery on message.action.

## Reproduction Assessment

Yes at source level, plus a concrete latest-release repro from the reporter. I did not run the live macOS restart scenario, but current source still sends unresumable notices only through deliverable message.action channels while WebChat is internal-only.

## Solution Assessment

No, current main is not the best solved state yet. The narrow repair should use the WebChat transcript/chat.history path or earlier durable user-turn persistence, not make message.action a general WebChat delivery API.
