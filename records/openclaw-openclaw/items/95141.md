## Label Justifications

- `P1`: The report describes a real user-facing WebChat workflow where an active turn silently disappears during a config-triggered restart.
- `impact:data-loss`: The user's own prompt is reported missing from the persisted transcript after restart.
- `impact:message-loss`: The active WebChat turn is neither recovered nor replaced by a visible interruption message.
- `impact:session-state`: The session is left failed with aborted restart state and recovery cannot resume from the stored transcript.

## Summary

Keep open. Current main still has a plausible WebChat-specific message-loss path: config restarts can force/abort active runs, empty or non-resumable transcripts fail recovery, and the unresumable notice path only targets deliverable channels while WebChat is internal-only.

## What This Changes

The issue reports that a config-triggered gateway restart can drop an in-flight WebChat dashboard turn, leaving an empty failed session with no visible recovery notice.

## Best Possible Solution

Fix the core WebChat restart path so an interrupted dashboard turn either preserves the user message before runtime persistence can be lost or records a WebChat-native terminal/retry notice, while keeping WebChat internal rather than making message.action deliver to it.

## Reproduction Assessment

Partially: the source path supports the reported empty-transcript/no-WebChat-notice failure, and the issue includes concrete v2026.6.8 repro evidence. I did not run the full WebChat plus config-restart scenario against current main.

## Solution Assessment

Unclear. The safest fix is not to make WebChat a generic outbound channel; maintainers should choose between earlier user-turn persistence, stronger restart deferral, and a WebChat-native failed/retry notice, or combine them where needed.
