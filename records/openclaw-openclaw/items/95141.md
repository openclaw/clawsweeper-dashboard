## Summary

Keep open: current main still has a WebChat-specific restart-recovery gap. The merged restart-marker work helps adjacent shutdown paths, but unresumable WebChat sessions still do not get a native transcript/chat.history notice and the reporter reproduced the empty failed session on the latest release.

## What This Changes

The issue asks OpenClaw to stop silently losing an in-flight WebChat turn when a config-triggered Gateway restart interrupts it, either by deferring restart, preserving the user turn, or showing a recovery notice.

## Best Possible Solution

Add a WebChat-native restart-recovery path that either persists the inbound user turn before restart can kill the process or writes the existing unresumable recovery notice into the transcript read by chat.history, while leaving external channels on message.action.

## Reproduction Assessment

Yes at source level, plus a concrete latest-release reproduction from the reporter. I did not run the live macOS restart scenario, but current main still sends unresumable notices only through deliverable message.action channels while WebChat is internal-only.

## Solution Assessment

No, current main is not the best solved state yet. The narrow repair should use the WebChat transcript/chat.history path or earlier durable user-turn persistence, not make message.action a general WebChat delivery API.
