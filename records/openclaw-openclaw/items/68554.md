## Summary

Keep open. The production symptom and root cause are credible, but this PR is now stale against current main and the one-line `loadSessionStore(storePath)` replacement still routes locked writers through a full-store clone/parse path instead of a true locked-writer cache fast path.

## What This Changes

The PR removes cache-bypassing session-store loads inside `updateSessionStore` and `updateSessionStoreEntry` and updates the nearby comment.

## Best Possible Solution

Introduce a narrow transactional locked-writer load/cache path that avoids redundant disk rereads and full-store clone/parse work while preserving public `loadSessionStore` isolation and cache invalidation guarantees.

## Reproduction Assessment

Yes. Use a large `sessions.json`, trigger queued `updateSessionStore` or `updateSessionStoreEntry` writes, and current main will synchronously bypass cache and parse the file while holding the write lock; #70857 and #75386 provide matching production logs.

## Solution Assessment

No. Removing `skipCache` alone avoids one disk reread path but leaves locked writers on the default cache-hit clone path and loses newer current-main `clone: false` behavior, so it is not the narrowest maintainable fix.
