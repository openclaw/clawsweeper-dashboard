## Summary

Keep open. The reported session-store lock stall is plausible and current main still performs a synchronous disk read/parse in the locked update paths, but this PR is not merge-ready as-is because replacing the current locked load with the default `loadSessionStore` path would still clone/parse the full cached store under the lock.

## What This Changes

The PR removes `skipCache: true` from the locked `updateSessionStore` and `updateSessionStoreEntry` loads in `src/config/sessions/store.ts` and updates the nearby comment to describe cache freshness inside the lock.

## Best Possible Solution

Rework the PR into a narrow session-store cache API change: locked writers should avoid the disk re-read and avoid full-store clone/parse work, while normal `loadSessionStore` callers keep defensive clone semantics. Add a focused regression or performance guard and a changelog entry for the user-visible outage fix.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to create a large `sessions.json` and trigger queued `updateSessionStore` or `updateSessionStoreEntry` writes; code inspection confirms current main synchronously reads and parses that file inside the lock, and #70857 corroborates the long-held-lock symptom.

## Solution Assessment

No. Removing `skipCache` alone is not the best fix on current main because the default cache-hit path still clones/parses the full store; the safer solution is an explicit no-clone cache read for locked writers with tests covering cache freshness and large-store cost.
