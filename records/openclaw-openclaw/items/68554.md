## Summary

Keep open. The PR addresses a real session-store write-lock bottleneck, but the current patch is dirty against main and would replace newer clone-free locked loads with default cached loads that still parse/clone large stores while the write lock is held.

## What This Changes

The PR removes cache-bypassing `loadSessionStore` options from `updateSessionStore` and `updateSessionStoreEntry` and updates the nearby locked-write comment.

## Best Possible Solution

Replace this with a narrow locked-writer cache/read path that avoids disk rereads and full-store clone/parse work under the lock while preserving public load isolation, stale-cache protection, and concurrent writer correctness.

## Reproduction Assessment

Yes. Use a large `sessions.json`, trigger queued `updateSessionStore` or `updateSessionStoreEntry` writes, and current main reaches synchronous cache-bypassing reads inside the session-store write lock; #70857 and #75386 provide matching production logs.

## Solution Assessment

No. Dropping `skipCache` alone routes locked writers through the normal cache-hit clone path, so it does not fully remove the large synchronous parse work from the lock; the safer fix is an explicit clone-free locked read/cache API with coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid cloning cached stores inside the lock:** `src/config/sessions/store.ts:429`

That cache path clones via `JSON.parse` of the serialized full store, which keeps large `sessions.json` parse work inside the write lock and can still cause the reported event-loop stalls; preserve a clone-free locked read/cache path in both helpers.
