## Summary

Keep open. The original session-store performance bug is valid and the PR now addresses the previous clone-on-cache-hit concern, but the current diff also removes the file-backed session-store lock from public mutation paths, which can drop updates when a separate OpenClaw process such as `openclaw sessions cleanup` writes the same store while the gateway is active.

## What This Changes

The PR replaces session-store file-lock based writers with an in-process writer queue, adds a mutable cache-borrow path, updates voice/heartbeat callers to use mutation helpers, exposes those helpers to plugins, and adds docs, changelog, and focused tests.

## Best Possible Solution

Keep the clone-free hot writer/cache path, but preserve cross-process exclusion for public session-store mutation paths or route separate-process maintenance through the gateway before removing `.lock` coordination.

## Reproduction Assessment

Yes. The original bug reproduces with a large `sessions.json` and queued session-store writes that hit the synchronous cache-bypassing read/parse path; the PR discussion and #70857/#75386 provide matching production logs. The introduced regression has a code-level reproduction with two OpenClaw processes mutating the same store, such as gateway writes racing `openclaw sessions cleanup --enforce`.

## Solution Assessment

No. The mutable cache-borrow path is the right direction for avoiding disk rereads and JSON clones, but replacing the file-backed lock with only an in-process queue is broader than the observed bottleneck and loses existing cross-process write safety.
