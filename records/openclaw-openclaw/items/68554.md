## Summary

Keep open. The PR targets a real current-main session-store lock bottleneck, but the submitted patch is not merge-ready because default cache hits still clone/parse the full store inside the lock and the branch is dirty against newer session-store changes.

## What This Changes

The PR removes cache-bypassing `loadSessionStore(..., { skipCache: true })` calls from the two locked session-store update helpers and updates the nearby cache-freshness comment.

## Best Possible Solution

Rework this into a narrow transactional locked-writer cache path that avoids both redundant disk rereads and full-store clone/parse work, while preserving normal public-load isolation, stale-cache protection, and concurrent update correctness.

## Reproduction Assessment

Yes. A high-confidence path is to use a large `sessions.json`, trigger queued `updateSessionStore` or `updateSessionStoreEntry` writes, and observe current main doing synchronous cache-bypassing reads/parses while the session-store lock is held; #70857 and #75386 provide matching production logs.

## Solution Assessment

No. Removing `skipCache` alone avoids the disk read but routes locked writers through the default cache-hit clone path, so large stores can still block the event loop under the lock; the safer fix needs an explicit locked read/cache API with tests for cache freshness and mutation isolation.
