## Summary

Keep open: current main still aborts auto provider selection when an adapter omits `shouldContinueAutoSelection`, and this PR addresses that real bug, but the proposed `adapter.id !== "local"` guard can incorrectly soft-skip local third-party adapters and the external PR lacks real after-fix behavior proof.

## What This Changes

The PR changes memory-core auto embedding provider selection to soft-skip common auth/config errors when an adapter omits explicit auto-selection guidance.

## Best Possible Solution

Land the behavior after changing the heuristic to use the public transport contract, adding focused regression coverage for omitted adapter guidance, and letting maintainers handle release-note/changelog work during landing.

## Reproduction Assessment

Yes from source inspection. On current main, registering an auto-select remote adapter that omits `shouldContinueAutoSelection` and throws a missing-key error makes the helper return false and the auto loop throw before FTS-only fallback.

## Solution Assessment

No as proposed. The direction is the narrow maintainable fix, but the soft-skip predicate should use `adapter.transport !== "local"` and the PR needs focused coverage plus real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Check adapter transport before soft-skipping auth errors:** `extensions/memory-core/src/memory/embeddings.ts:78`
  - body: This branch treats every adapter id except `local` as remote. A third-party adapter can declare `transport: "local"` with its own id; if it omits `shouldContinueAutoSelection`, an auth/config-looking setup error would now be swallowed instead of surfacing the local adapter failure. Gate the heuristic on `adapter.transport !== "local"`.
  - confidence: 0.88
