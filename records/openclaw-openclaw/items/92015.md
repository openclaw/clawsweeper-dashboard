## Summary

Keep open: current `main` and the latest release still resolve a non-default agent without an explicit workspace to a derived workspace under `agents.defaults.workspace`, then normal runtime paths can seed bootstrap files and run `git init` there. The related preseeded-bootstrap issue at https://github.com/openclaw/openclaw/issues/91931 overlaps the workspace state machine but does not cover ACP runtime-managed agents or nested git initialization.

## What This Changes

The issue requests skipping default workspace seeding and nested git initialization for ACP/runtime-managed agents that omit an explicit workspace.

## Best Possible Solution

Keep this open and make a bounded core fix: runtime-managed ACP agents without an explicit workspace should avoid default bootstrap seeding, explicit workspaces should keep current behavior, and `git init` should be skipped when an ancestor git repository already exists.

## Reproduction Assessment

Yes from source inspection: a non-default ACP config agent without `workspace` still resolves to `<agents.defaults.workspace>/<agentId>`, and normal runtime reply paths still call `ensureAgentWorkspace()` with bootstrap enabled. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear as a final implementation plan: skipping scaffold for no-explicit-workspace ACP agents is likely the right direction, while maintainers should decide the compatibility boundary for OpenClaw-side workspace uses. The nested `git init` parent-repo guard is a clear hardening subfix.
