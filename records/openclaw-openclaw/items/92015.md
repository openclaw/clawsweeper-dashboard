## Summary

Current main and the latest release still synthesize per-agent workspaces and initialize Git when those workspaces are materialized, so the hardening request remains necessary. The report is directionally valid, but its startup-only trigger is not fully proven and its proposed ACP-wide workspace skip conflicts with the documented target-workspace cwd fallback, requiring a narrower owner-boundary decision.

## What This Changes

Prevent ACP runtime agents without explicit workspaces from receiving unused bootstrap scaffolding, and avoid automatic nested Git repositories inside an existing repository.

## Best Possible Solution

Preserve explicit workspaces and ACP target-workspace cwd inheritance, stop only the proven eager caller from materializing an unused implicit runtime-owned workspace, and make brand-new workspace Git initialization ancestor-aware so an existing parent repository remains the sole repository owner.

## Reproduction Assessment

No, not yet at high confidence. Source proves the synthesized path and the bootstrap/Git side effects once workspace setup runs, but plain gateway startup skips missing `BOOT.md`, so the exact automatic caller in the minimal reproduction remains unidentified.

## Solution Assessment

No, the proposed blanket ACP workspace skip is not yet the best fix because ACP currently documents target-workspace cwd inheritance. The maintainable path is to repair the specific eager materialization boundary and separately make Git initialization respect an ancestor repository.
