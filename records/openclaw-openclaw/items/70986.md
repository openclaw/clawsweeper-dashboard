## Summary

Keep open: current main still sends raw Markdown through Memory Wiki wikilink extraction and the lint resolver still checks only path-shaped target keys. Related resolver PRs are open, but the Markdown-aware extractor bug remains a narrow, source-backed repair candidate.

## What This Changes

The issue asks Memory Wiki lint to ignore non-link `[[...]]` text in Markdown/code regions and resolve Obsidian wikilinks by frontmatter titles or normalized title variants.

## Best Possible Solution

Land a Memory Wiki-local Markdown-aware extractor fix and reconcile resolver/title normalization through one canonical resolver PR such as https://github.com/openclaw/openclaw/pull/80915 or https://github.com/openclaw/openclaw/pull/73722, then close this issue after both behaviors are on main and shipped.

Yes. Source inspection shows fenced or inline-code `[[...]]` reaches `extractWikiLinks`, and title-style targets miss because lint indexes only path-shaped keys; I did not execute a live lint run because this review is read-only.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized metadata-backed resolver keys is the narrow maintainable fix; tag-specific reply filtering would leave fenced shell/code examples and title links broken.
