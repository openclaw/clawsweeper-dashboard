## Summary

Keep open: current `main` and `v2026.5.12` still expose the central Memory Wiki lint bugs. The narrow `.md` relative-target fix is present, and resolver normalization has active PRs, but raw Markdown extraction still scans code-like regions and title/alias resolution has not landed on `main`.

## What This Changes

The issue reports Memory Wiki `wiki lint` false broken-wikilink warnings because raw `[[...]]` tokens inside Markdown regions are extracted and title-style frontmatter links are not accepted by the resolver.

## Best Possible Solution

Land a Memory Wiki-local extractor that skips Markdown frontmatter, fenced code, inline code, and comments, then land or replace one resolver-normalization PR for title, alias, slug, and fragment matching before closing this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: raw `[[...]]` in fenced or inline code still reaches `extractWikiLinks`, and title-style targets still miss because lint indexes only path-shaped keys.

## Solution Assessment

Yes. The maintainable direction is a generic Markdown-aware extractor plus normalized metadata-backed resolver keys; tag-specific reply filtering or content rewrites would leave code examples and title links broken.
