## Summary

Keep open: current main and the latest release still expose the two central Memory Wiki lint bugs. The `.md` relative-target false positive is fixed on main, but wikilink extraction still scans raw Markdown and broken-link lint still does not accept frontmatter titles or aliases; the related resolver PRs are open/partial rather than a terminal fix.

## What This Changes

Make `openclaw wiki lint` ignore wikilink-looking text inside Markdown structure and resolve Obsidian wikilinks against frontmatter-backed title metadata.

## Best Possible Solution

Land a Memory Wiki-local fix that strips or scans Markdown structure before extracting wikilinks and uses metadata-backed normalized target keys for lint resolution, coordinating the resolver half with the open title-normalization PRs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: raw `[[...]]` in fenced or inline code still reaches `extractWikiLinks`, and title-style targets still miss because `collectBrokenLinkIssues` indexes only path-shaped keys.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized metadata-backed resolver keys is the narrow maintainable fix; tag-specific reply filtering or content rewrites would leave fenced shell/code examples and title links broken.
