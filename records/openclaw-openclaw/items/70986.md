## Summary

Keep open: current `main` and the latest release still expose both central Memory Wiki lint problems. The `.md` relative-target expansion is present, but the extractor still scans raw Markdown and the broken-link resolver still indexes only path-shaped targets, while title/normalization work remains in open resolver PRs.

## What This Changes

The issue requests that Memory Wiki lint ignore `[[...]]` text in Markdown non-link regions and resolve wikilinks against frontmatter titles, aliases, and normalized title variants.

## Best Possible Solution

Land a Memory Wiki-local Markdown-aware extractor fix, choose or land one canonical resolver-normalization path for titles and aliases, then close this issue after both behaviors are on `main` and shipped.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: put `[[...]]` in a fenced or inline-code region, or link to an existing page by frontmatter title, and current `wiki lint` still feeds those raw targets into a path-only broken-link check.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus one normalized metadata-backed resolver is the narrow maintainable fix; tag-specific allowlists would miss bash bracket syntax, documentation examples, and natural title links.
