## Summary

Keep open: current main and latest release still have the raw Markdown wikilink extraction and path-only lint resolver that cause the reported false `broken-wikilink` warnings. The `.md` relative-target slice has landed, and active resolver PRs are useful, but no merged change covers both Markdown-aware extraction and title/alias normalization yet.

## What This Changes

The issue asks Memory Wiki lint to ignore wikilink-shaped text inside Markdown structure and to resolve wikilinks by frontmatter titles or normalized title variants.

## Best Possible Solution

Land a Memory Wiki-local fix that strips or parses Markdown structure before extracting wikilinks, then land one coordinated resolver fix that accepts title, alias, slug, fragment, and imported-source variants without weakening path validation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: fenced or inline-code `[[...]]` reaches `extractWikiLinks`, and title-style targets miss because lint only indexes path-shaped targets. The issue comments add independent live-vault confirmations, including a May 18 reproduction against a current bundled install.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus metadata-backed, normalized lint target keys is the narrow maintainable direction; tag-specific filtering or content rewrites would leave bash/code examples and title links broken.
