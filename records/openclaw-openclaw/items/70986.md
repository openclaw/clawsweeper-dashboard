## Summary

Keep open: current main still has both reported memory-wiki lint gaps. The extractor scans raw Markdown outside only the managed related block, and the lint resolver still validates only extensionless paths and basenames; related PR #73722 covers only the resolver half while PR #68852 was closed unmerged for the extractor half.

## What This Changes

The issue requests memory-wiki lint to ignore literal wikilink syntax in Markdown code/comment/frontmatter contexts and to resolve valid pages by frontmatter title and normalized link variants.

## Best Possible Solution

Keep this as the canonical combined memory-wiki lint bug until both halves are fixed: land or replace the resolver normalization from #73722, then add a Markdown-aware extraction pass that skips fenced blocks, inline code, frontmatter, and HTML comments without losing real prose links. Add focused regressions in `extensions/memory-wiki/src/markdown.test.ts` and `extensions/memory-wiki/src/lint.test.ts`, with a short docs note if accepted link forms stay intentionally constrained.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: a page containing fenced or inline literal `[[reply_to_current]]` still flows through raw `extractWikiLinks`, and a page linked as `[[Example 配置清单]]` still misses the path-only `validTargets` set unless the filename or basename exactly matches.

## Solution Assessment

Yes, the requested generic extractor-plus-normalized-resolver direction is the best maintainable fix. Tag-specific sanitization would only hide one bridge symptom, and #73722 is useful but only covers the resolver side.
