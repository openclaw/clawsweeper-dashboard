## Summary

Keep open: current main still has both reported memory-wiki lint behaviors. The extractor scans raw Markdown for `[[...]]`, and lint still validates links against path and basename targets only; related open work covers the resolver half, but the Markdown-aware extractor half remains unmerged.

## What This Changes

The issue requests that memory-wiki lint ignore literal wikilink syntax inside Markdown code/comment/frontmatter contexts and resolve Obsidian links by normalized page titles, slugs, aliases, and path variants instead of path-only targets.

## Best Possible Solution

Keep this as the canonical combined lint-noise bug until both sides land: reuse or finish the resolver normalization from #73722, then add a generic Markdown-aware extraction pass that skips fenced code, inline code, frontmatter, and HTML comments without losing real prose links. Lock the behavior with focused `markdown.test.ts` and `lint.test.ts` regressions, and update docs only if accepted link forms remain intentionally narrower than Obsidian expectations.

## Reproduction Assessment

Yes. A minimal vault page containing fenced or inline literal `[[reply_to_current]]` still flows through raw `extractWikiLinks`, and a page linked as `[[Example 配置清单]]` still misses the path-only `validTargets` set unless the filename or basename exactly matches.

## Solution Assessment

Yes. The generic Markdown-aware extractor plus normalized resolver is the narrow maintainable solution; tag-specific sanitization like the related #64696 direction would miss bash bracket syntax and other literal `[[...]]` examples, while #73722 only covers the resolver side.
