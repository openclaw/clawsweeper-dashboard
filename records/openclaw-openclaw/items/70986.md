## Summary

Keep open: current main still has the reported memory-wiki lint root causes, with raw Markdown wikilink extraction and path-only broken-link validation still in place. The related #73722 PR covers the resolver half, but the Markdown-aware extractor half remains unmerged.

## What This Changes

The issue asks memory-wiki lint to ignore literal `[[...]]` in Markdown/code/comment contexts and to resolve wikilinks by title or normalized page keys instead of only path and basename.

## Best Possible Solution

Land resolver normalization through #73722 or an equivalent maintainer-approved patch, then add a focused Markdown-aware extractor that skips frontmatter, fenced and inline code, and HTML comments with regression coverage.

## Reproduction Assessment

Yes. The current source path is enough to reproduce: a page containing fenced or inline literal `[[reply_to_current]]` still feeds raw text into `extractWikiLinks`, and a page linked by `[[Example 配置清单]]` still misses the path-only `validTargets` set unless the path or basename matches exactly.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized resolver is the narrow maintainable fix; tag-specific sanitization like the related reply-tag direction would miss bash bracket syntax and other literal `[[...]]` examples.
