Keep open: current main still has the reported memory-wiki lint false positives, and the later comments add independent vault evidence rather than making the report stale or duplicative.

## What This Changes

`openclaw wiki lint` should ignore literal `[[...]]` inside Markdown code/comment/frontmatter contexts and resolve existing pages by frontmatter title or normalized wiki target keys.

## Best Possible Solution

Keep this as the canonical combined lint-noise issue until a Markdown-aware extractor and the resolver normalization are both landed with focused regressions.

## Reproduction Assessment

Yes. A page containing fenced or inline literal `[[reply_to_current]]` still flows through raw `extractWikiLinks`, and `[[Example 配置清单]]` still misses a page whose title matches but whose relative path is slugified.

## Solution Assessment

Yes. A generic Markdown-aware extraction pass plus normalized title/path resolver is the maintainable fix; tag-specific sanitization would miss bash bracket syntax and other literal `[[...]]` examples.
