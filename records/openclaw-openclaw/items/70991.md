## Summary

Keep open. Current main still only partially addresses this regression: session-scoped `chat.abort` support exists, but the Control UI still hides Stop after local run state is lost on reconnect/reload, and Gateway WebSocket protocol ping/connection-health work is still represented by open overlapping PRs rather than landed code.

## What This Changes

The issue reports that long OpenClaw Control UI tasks can drop the Gateway connection, lose the Stop button after reload, and fail to stop the active model run via `/stop`.

## Best Possible Solution

Keep this as the canonical user-facing regression until one Gateway WebSocket keepalive/connection-health implementation lands, then update the Control UI so abort availability after reconnect or reload comes from server/session active-run state or a session-scoped Stop path, using `chat.abort` without `runId` when local run id is unavailable.

## Reproduction Assessment

Unclear for the full connection-drop path because the issue has plausible long-task steps but no logs or live Windows/Kimi proof. Yes for the Stop disappearance path: source inspection shows reconnect clears `chatRunId`, Stop rendering is gated on that local value, and tests cover the disconnected no-run-id abort no-op.

## Solution Assessment

No, current main is not the full solution. The backend session-scoped abort implementation is a useful foundation, but the UI still does not expose a session-level Stop control after local state loss, and the connection stability fix should land through one canonical Gateway ping/health owner.
