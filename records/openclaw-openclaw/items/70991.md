## Summary

Keep open. Current main still only partially addresses the report: session-scoped `chat.abort` exists, but the Control UI still hides Stop after local run state is lost and the Gateway has no landed protocol-level WebSocket ping/pong or connection-health owner on main.

## What This Changes

The issue reports long-running Control UI/Gateway connection drops where reload loses the Stop control and `/stop` fails to interrupt the active model run.

## Best Possible Solution

Land one Gateway WebSocket ping/health implementation, then make Control UI recover server/session active-run abort availability after reconnect or reload and expose Stop/session-scoped abort even when local `chatRunId` is gone.

## Reproduction Assessment

Unclear for the full live disconnect path because the report has no logs and I did not run a Windows/Kimi long-task smoke. Yes for the Stop disappearance path: current source clears `chatRunId` on reconnect and renders Stop only when that local value exists.

## Solution Assessment

No, current main is not the full solution. The backend session-scoped abort path is the right primitive, but the UI still depends on local run state and the Gateway connection-stability work needs one canonical ping/health owner.
