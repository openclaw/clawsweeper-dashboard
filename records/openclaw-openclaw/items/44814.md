## Summary

Keep open. Current main partially supersedes the old hardcoded 5 MB call sites with a shared resolver, but this PR is stale against the current upload contract and the larger-image problem still remains behind the separate 6 MiB image cap tracked by #71142.

## What This Changes

The PR adds `gateway.webui.attachmentMaxBytes` to config types/schema and uses it for Web UI `chat.send` plus node-event `agent.request` attachment parsing.

## Best Possible Solution

Resolve the remaining work through #71142 by choosing the maintained upload policy surface, routing server parsing and image hydration through one bounded effective limit, exposing browser prevalidation data in the Control UI bootstrap contract, and documenting the behavior with focused Gateway/UI tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static current-main reproduction exists: configure `agents.defaults.mediaMaxMb` above 6 MiB and submit a Control UI image larger than `MAX_IMAGE_BYTES`; `chat.send` reaches the shared configured max path, then `parseMessageWithAttachments` rejects the fixed image cap.

## Solution Assessment

Is this the best way to solve the issue? No. The PR was reasonable for the older hardcoded call sites, but current main now has a shared resolver and this PR would add a stale `gateway.webui` namespace while leaving the image cap and browser upload policy unresolved.
