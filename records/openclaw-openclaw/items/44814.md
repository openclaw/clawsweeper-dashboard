## Summary

Keep open: the PR addresses a real upload-limit problem, but current main has moved the relevant call sites to a shared attachment-limit resolver and the remaining Control UI upload-size work needs maintainer API/product direction through #71142 before this branch can be refreshed safely.

## What This Changes

The PR adds `gateway.webui.attachmentMaxBytes` to Gateway config types/schema and uses it for Control UI `chat.send` plus node-event `agent.request` attachment parsing.

## Best Possible Solution

Resolve the remaining work through #71142 by choosing one maintained Control UI upload policy, keeping server parsing on a shared effective limit, exposing browser prevalidation data where needed, and covering the Gateway/UI paths with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: configure an upload limit above 6 MiB, submit a larger image through Control UI, and `parseMessageWithAttachments` still rejects it at the `MAX_IMAGE_BYTES` guard after the shared max check.

## Solution Assessment

No. The PR was reasonable for the older hardcoded 5 MB call sites, but on current main it would add a stale `gateway.webui` namespace and bypass the shared resolver instead of solving the maintained Control UI upload policy.
