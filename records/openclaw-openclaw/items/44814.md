## Summary

Keep open. Current main has partly superseded the old hardcoded 5 MB paths with a shared resolver, but this PR is stale against the current upload contract and would add a separate `gateway.webui` namespace while the larger-image failure remains tracked by the open Control UI upload-limit issue.

## What This Changes

The PR adds `gateway.webui.attachmentMaxBytes` to Gateway config types/schema and uses it in Web UI `chat.send` plus node-event `agent.request` attachment parsing.

## Best Possible Solution

Resolve the remaining work through #71142 by choosing the maintained upload policy surface, routing server parsing and image hydration through one bounded effective limit, exposing browser prevalidation data in the Control UI bootstrap contract, and documenting the behavior with focused Gateway/UI tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.mediaMaxMb` above 6 MiB and submit a Control UI image larger than `MAX_IMAGE_BYTES`; `chat.send` reaches the shared configured max path, then `parseMessageWithAttachments` rejects the fixed image cap.

## Solution Assessment

No. The PR was reasonable for the older hardcoded call sites, but current main now has a shared resolver and this PR would add a stale `gateway.webui` namespace while leaving the image cap and browser upload policy unresolved.
