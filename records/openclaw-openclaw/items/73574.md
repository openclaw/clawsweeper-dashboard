## Summary

Keep open. Current main still has the reported memory-wiki lint resolver gap, and the issue is already paired with open closing PR #73722, so the right cleanup action is to review or land that PR rather than close this issue early or queue duplicate repair work.

## What This Changes

Make `openclaw wiki lint` resolve existing memory-wiki pages when Obsidian links use titles, slug variants, fragment/query suffixes, or imported source-path suffixes.

## Best Possible Solution

Land PR #73722 or an equivalent memory-wiki-owned change that expands lint lookup keys, strips fragments and query strings before comparison, preserves imported provenance, and adds focused regression coverage for the reported link variants while leaving broader markdown parsing work to #70986.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a page stored as `syntheses/alpha-db.md` with title `Alpha Database` linked as `[[Alpha Database#Evidence]]` will not match the current `syntheses/alpha-db` or `alpha-db` target set.

## Solution Assessment

Yes. The narrow maintainable solution is to normalize targets inside the memory-wiki lint resolver and cover it with colocated lint tests; rewriting imported evidence would damage provenance and is not the right fix.
