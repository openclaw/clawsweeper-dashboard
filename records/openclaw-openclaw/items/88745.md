## What This Changes

The PR makes `resolveConfiguredProviderFallback` model-aware and passes `defaultModel` from the model-selection and status-summary call sites.

## Best Possible Solution

Keep the model-aware fallback direction, but first preserve `defaultProvider/defaultModel` when that provider is already configured, add duplicate-model regression coverage, and require redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the PR regression: configure an earlier custom provider and `openai` with the same `gpt-5.5` id, then resolve the default `openai/gpt-5.5`; the new scan returns the earlier provider before checking `openai`. I did not run the test because this review is read-only.

## Solution Assessment

No. The proposed boundary is plausible, but this implementation scans across providers before preserving a configured default provider, so the safer fix is to check the default provider first and only then perform any cross-provider fallback.
