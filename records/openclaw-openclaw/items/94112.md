## Summary

This PR remains useful, but it should not merge as-is because the new bare `(no output)` early return can drop otherwise sendable non-text reply payloads. Current source shows media, presentation, interactive, and channelData content are deliberately preserved by the normalizer after text cleanup, and the PR bypasses that content gate.

## What This Changes

The PR extends shared reply sanitization to strip copied current-message/history scaffolding and standalone `(no output)` lines, and adds an `empty-tool-output` skip reason for bare `(no output)` reply text.

## Best Possible Solution

Keep the sanitizer improvements, but change the normalizer so `(no output)` only clears or skips the text portion when no other deliverable reply content remains, with a regression test covering mixed-content payloads.

## Reproduction Assessment

Yes at source level. Current main lacks the reported sanitizer rules, and the PR-introduced mixed-content drop is source-reproducible because the new early return bypasses the existing reply-content preservation path.

## Solution Assessment

No, not as written. The shared sanitizer and normalizer are the right layer, but the normalizer should clear placeholder text and then use the existing content gate instead of returning null unconditionally.
