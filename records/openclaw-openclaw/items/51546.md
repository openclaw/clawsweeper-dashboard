## Summary

Keep open: current main and the latest release do not implement assistant-message disclosure, while this PR is conflicting and still has availability, toggle, CSS, and real-proof blockers.

## What This Changes

The PR adds a Control UI long-message Show more/Show less wrapper, lowers JSON auto-parse size, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Ship a revised bounded Control UI folding implementation that defines the UX policy, renders a capped preview until expansion, preserves oversized and streaming fallbacks, keeps state reactive, and proves the behavior in a real browser.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows full markdown parse/render before CSS collapse, widened parse limits, stale closed-over toggle state, and missing expanded-state positioning. I did not run a browser session in this read-only review.

## Solution Assessment

No. Progressive disclosure is a plausible direction, but this patch should bound preview rendering, preserve the cheap oversized and streaming fallback, and resolve state/CSS defects before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bound collapsed markdown rendering:** `ui/src/ui/chat/grouped-render.ts:650-656`
  - body: Completed messages over the disclosure threshold still call `toSanitizedMarkdownHtml(markdown)` and insert the full rendered result before CSS collapses it. A 150k-500k payload can still force the parse, sanitize, and DOM work this feature is intended to avoid; render a capped preview or cheap fallback until expansion.
  - confidence: 0.93
- **[P2] Keep the streaming fallback cheap:** `ui/src/ui/markdown.ts:60`
  - body: Raising `MARKDOWN_PARSE_LIMIT` to 100k makes long in-progress replies keep parsing and sanitizing every growing chunk from 40k-100k characters, and the cache misses because each chunk differs. Preserve the existing cheap fallback for streaming content unless the user explicitly expands a completed message.
  - confidence: 0.88
- **[P2] Drive the toggle from current state:** `ui/src/ui/chat/grouped-render.ts:663-675`
  - body: The click handler closes over the render-time `isExpanded` value. After the first click, later clicks on the same button keep taking the original branch until another render replaces the DOM, so Show less can fail to collapse the message and the label can stay stale.
  - confidence: 0.9
- **[P2] Anchor the expanded toggle to the message:** `ui/src/styles/chat/text.css:153-157`
  - body: `.chat-text__show-more` is absolutely positioned, but `.chat-text--expanded` does not provide a positioning context. When expanded, the Show less button can be positioned against an ancestor instead of the message container.
  - confidence: 0.86
- **[P3] Keep the control out of the fade mask:** `ui/src/styles/chat/text.css:149-150`
  - body: The fade mask is applied to the same `.chat-text` element that contains the absolutely positioned Show more button. That masks the control along with the message tail, making the affordance less visible and harder to click on long replies.
  - confidence: 0.78
