## Summary

Keep open: current main still lacks long-message progressive disclosure, and this useful PR is not merge-ready because it reopens expensive markdown parse/render work and still has toggle and CSS positioning defects.

## What This Changes

The PR adds Control UI Show more/Show less rendering for long markdown messages, lowers JSON auto-parse size, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Ship a revised Control UI implementation that keeps markdown/JSON guards cheap, renders only a bounded preview until expansion, stores expansion state in stable reactive state, keeps controls outside masked content, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main still hard-limits/falls back, while the PR diff raises parsing limits and renders full long markdown before CSS collapse; the toggle and positioning defects are visible in the added handler and CSS.

## Solution Assessment

No. Progressive disclosure is a reasonable direction, but this implementation should use bounded preview rendering, cheap parse guards, reactive expansion state, and controls outside the masked content before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep the large-message parse guard cheap:** `ui/src/ui/markdown.ts:67`
  - body: Raising MARKDOWN_PARSE_LIMIT to 100k makes 40k-100k replies hit full markdown parse/sanitize again, and the new renderer still mounts the entire long message before CSS hides it. That reopens the freeze path this PR is trying to avoid; keep the cheap fallback or render only a bounded preview until expansion.
  - confidence: 0.91
- **[P2] Read the toggle state at click time:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The click handler closes over isExpanded from the render that created the button. After the first click, later clicks on the same DOM node can keep taking the same branch, leaving the label stale and preventing reliable Show less behavior until an unrelated rerender replaces the template.
  - confidence: 0.9
- **[P2] Keep the expanded button positioned in its container:** `ui/src/styles/chat/text.css:153-157`
  - body: .chat-text__show-more is absolutely positioned, but .chat-text--expanded removes the positioning context supplied by .chat-text--long. When expanded, the Show less button can be positioned relative to an ancestor instead of the message container.
  - confidence: 0.86
- **[P3] Move the disclosure control outside the fade mask:** `ui/src/styles/chat/text.css:146-147`
  - body: The fade mask is applied to the same .chat-text--long element that contains the button, so the reveal control can be faded with the message tail. Mask only the content body or render the control outside the masked element.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `ui/src/ui/chat/grouped-render.ts:638`
  - body: This is a user-facing Control UI feature, but the diff only changes UI source and CSS files. Add an Unreleased changelog entry before merge.
  - confidence: 0.82
