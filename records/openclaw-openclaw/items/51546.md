## Summary

Keep this PR open for Web UI maintainer follow-up: current main still does not implement long-message progressive disclosure, but the proposed patch is not merge-ready because it still performs expensive markdown work before collapse and has concrete toggle/layout defects called out in review.

## What This Changes

The PR adds a long-message Show more/Show less renderer and CSS for chat markdown, lowers the JSON auto-parse cap, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Keep the PR open, but require a revised implementation that renders only a bounded preview by default, preserves the cheap large-text fallback for streaming and oversized content, stores expansion state in stable reactive message state, keeps controls outside masked content, and adds focused long-message regression coverage before merge.

## Reproduction Assessment

Yes. The current-main behavior is reproducible by inspecting the direct markdown render paths and existing 40,000-character fallback tests; the PR defects are also high-confidence from the diff because long completed messages still call toSanitizedMarkdownHtml before collapse and the click handler closes over the original isExpanded value.

## Solution Assessment

No. Progressive disclosure is the right direction, but this implementation is not the narrowest maintainable fix because it hides large DOM/parse work with CSS instead of avoiding it and leaves the toggle state partly outside Lit/reactive ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep the long-message parse guard cheap:** `ui/src/ui/markdown.ts:67`
  - body: The collapse UI still calls `toSanitizedMarkdownHtml(markdown)` before hiding the tail, so raising `MARKDOWN_PARSE_LIMIT` from 40k to 100k makes 40k-100k messages hit full markdown parse/sanitize during render. This regresses the large-payload freeze path the PR is intended to fix; keep the cheap fallback or render a bounded preview until expansion.
  - confidence: 0.91
- **[P2] Base the toggle on current expanded state:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The click handler closes over the `isExpanded` value from the render that created the button. After the first click, later clicks on the same DOM node can keep taking the same branch, leaving the label stale and preventing reliable Show less behavior until an unrelated rerender replaces the template.
  - confidence: 0.9
- **[P2] Keep the expanded button positioned in its container:** `ui/src/styles/chat/text.css:153-157`
  - body: `.chat-text__show-more` is absolutely positioned, but the expanded state removes the only positioning context supplied by `.chat-text--long`. When the user expands a message, the Show less button can be positioned relative to an ancestor instead of the message container.
  - confidence: 0.86
- **[P3] Move the toggle outside the fade mask:** `ui/src/styles/chat/text.css:146-147`
  - body: The fade mask is applied to `.chat-text--long`, and the button is rendered as a child of that same element at the bottom edge. The new affordance can be faded with the content it is supposed to reveal, so put the control outside the masked content or mask only the text body.
  - confidence: 0.78
