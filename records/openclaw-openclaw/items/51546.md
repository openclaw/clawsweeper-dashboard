## Summary

Keep open: the PR addresses a real Control UI pain point, but the current branch still has blocking long-message rendering and disclosure-state defects, and the external contributor has not attached observable after-fix real behavior proof.

## What This Changes

The PR adds a Control UI Show more/Show less renderer for long markdown messages, lowers the JSON auto-parse cap, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Ship a revised Control UI implementation that renders a bounded preview until expansion, preserves cheap markdown fallback behavior during streaming, stores disclosure state reactively, keeps the control outside masked content, and adds focused regression coverage.

## Reproduction Assessment

Yes for the review blockers: source inspection gives a high-confidence path because current main lacks disclosure UI and the PR diff shows the expensive parse path, stale toggle state, and CSS positioning issue. I did not run a live browser session in this read-only review.

## Solution Assessment

No. Progressive disclosure is a reasonable direction, but this implementation should bound preview rendering and use stable reactive state/CSS before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep long-message parsing bounded:** `ui/src/ui/markdown.ts:67`
  - body: Raising MARKDOWN_PARSE_LIMIT to 100k makes 40k-100k replies hit full markdown parse/sanitize again, while completed long messages still render the full markdown before CSS hides the tail. That reopens the large-payload freeze path this feature is meant to avoid; keep the cheap fallback or render only a bounded preview until expansion.
  - confidence: 0.91
- **[P2] Read disclosure state at click time:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The handler closes over isExpanded from the render that created the button. After the first click, later clicks on that same DOM node keep taking the same branch until an unrelated rerender replaces the template, so Show less can become unreliable and the label/classes can stay stale.
  - confidence: 0.9
- **[P2] Keep the expanded control positioned inside the message:** `ui/src/styles/chat/text.css:153-157`
  - body: .chat-text__show-more is absolutely positioned, but .chat-text--expanded removes the positioning context supplied by .chat-text--long. When expanded, the Show less button can be positioned relative to an ancestor instead of the message container.
  - confidence: 0.86
- **[P3] Move the disclosure control outside the fade mask:** `ui/src/styles/chat/text.css:146-147`
  - body: The fade mask is applied to the same .chat-text--long element that contains the button, so the reveal control can be faded with the message tail. Mask the content body or render the control as an unmasked sibling.
  - confidence: 0.78
