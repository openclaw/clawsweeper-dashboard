## Summary

Keep open: current main still has the hard markdown limits and no progressive disclosure, while this PR remains a useful but incorrect implementation. The diff still does expensive parse/mount work before hiding long content, has stale toggle-state behavior, and has CSS placement/fade issues already called out in prior Greptile and ClawSweeper review comments.

## What This Changes

The PR adds a Control UI Show more/Show less renderer and CSS for long markdown messages, lowers JSON auto-parse size, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Ship a revised Control UI implementation that keeps the protective markdown/JSON guards cheap, renders only a bounded preview until expansion, stores expansion state in stable reactive message state, keeps controls outside masked content, and adds focused completed-message, streaming, and toggle regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main still hard-limits/falls back, and the PR diff shows full markdown parse/mount before CSS collapse plus stale click-state and CSS positioning defects.

## Solution Assessment

No. Progressive disclosure is a reasonable direction, but this implementation hides expensive DOM/parse work with CSS and keeps toggle state partly outside Lit’s reactive ownership; a bounded-preview renderer with reactive expansion state is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the large-message parse guard cheap:** `ui/src/ui/markdown.ts:67`
  - body: Raising `MARKDOWN_PARSE_LIMIT` to 100k makes 40k-100k chat replies hit markdown parse/sanitize again, and the new renderer still mounts the whole long message before CSS hides it. That reopens the freeze path this PR is meant to avoid; keep the cheap fallback or render only a bounded preview until expansion.
  - confidence: 0.91
- **[P2] Read the toggle state at click time:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The handler closes over the `isExpanded` value from the render that created the button. After the first click, later clicks on the same DOM node can keep taking the same branch, leaving the label stale and preventing reliable Show less behavior until an unrelated rerender replaces the template.
  - confidence: 0.9
- **[P2] Keep the expanded button positioned in its container:** `ui/src/styles/chat/text.css:153-157`
  - body: `.chat-text__show-more` is absolutely positioned, but `.chat-text--expanded` removes the only positioning context supplied by `.chat-text--long`. When expanded, the Show less button can be positioned relative to an ancestor instead of the message container.
  - confidence: 0.86
- **[P3] Move the disclosure control outside the fade mask:** `ui/src/styles/chat/text.css:146-147`
  - body: The fade mask is applied to the same `.chat-text--long` element that contains the button, so the reveal control can be faded with the text tail. Mask only the content body or render the control outside the masked element.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `ui/src/ui/chat/grouped-render.ts:638`
  - body: This is a user-facing Control UI feature, but the diff only changes UI source/CSS files. Add an Unreleased changelog entry with contributor credit before merge.
