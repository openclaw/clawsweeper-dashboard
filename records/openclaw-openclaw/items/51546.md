## Summary

Keep open: progressive disclosure for large Control UI messages is a useful direction, but this PR is not merge-ready because it still renders large markdown before collapse, widens the expensive streaming parse path, has toggle/CSS bugs, lacks inspectable real behavior proof, and is currently conflicting with main.

## What This Changes

This PR adds Control UI Show more/Show less disclosure for markdown messages over 5,000 characters, lowers JSON auto-parse size, and raises markdown character, parse, and cache limits.

## Best Possible Solution

Ship a revised Control UI implementation that renders only a bounded preview until expansion, preserves the existing cheap oversized and streaming fallback path, keeps disclosure state reactive, positions the control reliably, and includes focused tests plus real UI proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows full markdown rendering before collapse, widened streaming parse limits, stale toggle state, and missing expanded-state positioning. I did not run a live browser session in this read-only review.

## Solution Assessment

No. Progressive disclosure is a reasonable UI direction, but this implementation should bound preview rendering, avoid widening the streaming parse path, and fix the state and CSS issues before merge.

Full review comments:

- **[P1] Bound the collapsed markdown render:** `ui/src/ui/chat/grouped-render.ts:650`
  - body: Completed messages above the threshold still call toSanitizedMarkdownHtml(markdown) and insert the full rendered result before CSS collapses it.   - confidence: 0.92
- **[P2] Preserve the cheap streaming fallback:** `ui/src/ui/markdown.ts:60`
  - body: Raising MARKDOWN_PARSE_LIMIT to 100k makes long streaming replies parse and sanitize every growing chunk from 40k-100k characters, and the cache misses because the input changes on each update. That regresses the large-stream jank path this PR is trying to improve.
  - confidence: 0.88
- **[P2] Read expansion state at click time:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The click handler closes over the render-time isExpanded value. After the first click, later clicks on the same button keep taking the original branch until another render replaces the DOM, so Show less can fail to collapse the message.
  - confidence: 0.9
- **[P2] Keep the expanded control positioned in the message:** `ui/src/styles/chat/text.css:153-157`
  - body: .chat-text__show-more is absolutely positioned, but .chat-text--expanded drops the position: relative context provided by .chat-text--long. When expanded, the Show less button can be positioned against an ancestor instead of the message.
  - confidence: 0.86
- **[P3] Move the reveal control outside the fade mask:** `ui/src/styles/chat/text.css:149-150`
  - body: The fade mask is applied to the same .chat-text--long element that contains the button, so the Show more control is faded with the message tail. Mask the content body or render the control as an unmasked sibling.
  - confidence: 0.78
