## What This Changes

The PR adds a Show more/Show less wrapper for long Control UI markdown messages, lowers JSON auto-parse size, and raises markdown truncation, parse, and cache limits.

## Best Possible Solution

Revise or replace the branch so collapsed long messages render a capped preview until expansion, preserve the current oversized and streaming fallbacks, use stable reactive expand/collapse state, and prove the behavior in the real Control UI.

## Reproduction Assessment

Yes for the PR blockers via source review: the PR head still parses and inserts full long markdown before CSS collapse, raises parse limits, and uses stale render-time toggle state. I did not run a browser session in this read-only review.

## Solution Assessment

No. Progressive disclosure is a plausible direction, but this implementation should first bound parse/render work, preserve the oversized fallback, and align with the open UX policy choices.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Render a bounded preview before raising parse limits:** `ui/src/ui/chat/grouped-render.ts:650`
  - body: For completed messages above the 5k threshold, this still computes and inserts `toSanitizedMarkdownHtml(markdown)` for the entire payload before CSS hides the tail. Since the same PR raises the hard and parse limits, the large messages this feature targets can still freeze the chat view with larger inputs; render only a capped preview until expansion and keep the cheap fallback for oversized content.
  - confidence: 0.93
- **[P2] Keep the streaming oversized fallback cheap:** `ui/src/ui/markdown.ts:60`
  - body: Raising `MARKDOWN_PARSE_LIMIT` from 40k to 100k makes in-progress long replies keep going through markdown parsing for an extra 60k characters, and cache hits do not help while the stream changes on every chunk. Preserve the current streaming/oversized fallback unless expansion is explicitly requested.
  - confidence: 0.88
- **[P2] Derive toggle behavior from current state:** `ui/src/ui/chat/grouped-render.ts:663-675`
  - body: The click handler closes over the render-time `isExpanded`, so later clicks can keep taking the same branch and the label remains bound to the stale value until another render. Store this in reactive owner state or read the current state and request a re-render/text update on each toggle.
  - confidence: 0.9
- **[P2] Keep the disclosure button anchored and visible:** `ui/src/styles/chat/text.css:153-160`
  - body: `.chat-text__show-more` is absolutely positioned, but `.chat-text--expanded` lacks a positioned containing block, so Show less can escape the message. The collapsed control is also inside the masked element, which can fade the button with the message tail; keep the control outside the masked content or use a stable positioning wrapper.
  - confidence: 0.86
