## What This Changes

The PR adds a Show more/Show less wrapper for long Control UI markdown messages, lowers JSON auto-parse size, and raises markdown parse/cache/truncation limits.

## Best Possible Solution

Revise or replace this branch with a bounded Control UI implementation that keeps collapsed previews cheap, preserves the oversized fallback for streaming, and follows the UX policy settled in the related feature issue.

## Reproduction Assessment

Yes for the review blockers: the PR diff shows full markdown parse/render before CSS collapse, widened parse limits, stale closed-over toggle state, and missing expanded-state positioning. I did not run a browser session in this read-only review.

## Solution Assessment

No as proposed. Progressive disclosure is a plausible solution, but this implementation hides already-rendered content rather than bounding work and still needs the UX policy decisions tracked by the related feature issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Render a bounded preview before collapsing:** `ui/src/ui/chat/grouped-render.ts:647`
  - body: For completed long messages, this still calls `toSanitizedMarkdownHtml(markdown)` and inserts the full sanitized content before CSS hides it. With the same PR raising the hard limit to 500k, the browser can still pay the full parse/sanitize/DOM cost that this feature is meant to avoid; render a bounded preview until the user expands.
  - confidence: 0.94
- **[P2] Preserve the streaming oversized fallback:** `ui/src/ui/markdown.ts:58`
  - body: Raising `MARKDOWN_PARSE_LIMIT` from 40k to 100k makes long streaming replies keep hitting markdown parsing for an extra 60k characters while each chunk is a cache miss. That regresses the current cheap fallback path for the live auto-scroll case the PR is trying to protect.
  - confidence: 0.86
- **[P2] Derive the toggle from current state:** `ui/src/ui/chat/grouped-render.ts:658-673`
  - body: The click handler closes over the `isExpanded` value from the render that created the button. A message first rendered collapsed always takes the expand branch on later clicks, so the control cannot reliably collapse again until an unrelated rerender replaces it.
  - confidence: 0.9
- **[P2] Anchor the expanded toggle button:** `ui/src/styles/chat/text.css:153-157`
  - body: When a rerender produces the expanded template, `.chat-text--expanded` removes the `.chat-text--long` class that provided `position: relative`, so the absolutely positioned Show less button can anchor to an ancestor instead of the message container.
  - confidence: 0.82
