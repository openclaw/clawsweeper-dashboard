## Summary

Keep this PR open, but it should not merge yet: the progressive-disclosure direction is useful, but the current patch still parses/renders full long messages before collapse, has toggle/CSS defects, and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds Show more/Show less rendering for long Control UI markdown messages and changes markdown/JSON size limits.

## Best Possible Solution

Ship a revised Control UI implementation that renders only a bounded preview until expansion, preserves cheap streaming fallback behavior, stores disclosure state reactively, positions the control reliably, and adds focused regression coverage plus real UI proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows full markdown parsing before collapse, stale closure state in the toggle handler, and a missing positioning context. I did not run a live browser session in this read-only review.

## Solution Assessment

No. Progressive disclosure is a reasonable direction, but this implementation should bound preview rendering, keep the streaming parse guard cheap, and fix the state/CSS issues before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Bound the collapsed markdown render:** `ui/src/ui/chat/grouped-render.ts:650`
  - body: Completed long messages still call `toSanitizedMarkdownHtml(markdown)` before collapse, so a 150k-500k dump is fully parsed, sanitized, and inserted into the DOM before CSS hides it. Render only a capped preview or cheap fallback until the user expands the message.
  - confidence: 0.92
- **[P2] Keep the streaming parse guard cheap:** `ui/src/ui/markdown.ts:60`
  - body: Raising `MARKDOWN_PARSE_LIMIT` to 100k makes streaming replies parse and sanitize every growing chunk from 40k-100k characters, and the cache will miss because the input changes on each update. That regresses the large-stream jank path this PR is trying to fix.
  - confidence: 0.88
- **[P2] Read expansion state when clicked:** `ui/src/ui/chat/grouped-render.ts:659-673`
  - body: The click handler closes over the render-time `isExpanded` value. After the first click, later clicks on the same button keep taking the original branch until another render replaces the DOM, so Show less can fail to collapse the message.
  - confidence: 0.9
- **[P2] Keep the expanded button positioned in the message:** `ui/src/styles/chat/text.css:153-157`
  - body: `.chat-text__show-more` is absolutely positioned, but `.chat-text--expanded` does not keep the positioning context supplied by `.chat-text--long`. When expanded, the Show less button can be positioned against an ancestor instead of the message.
  - confidence: 0.86
- **[P3] Move the reveal control outside the fade mask:** `ui/src/styles/chat/text.css:149-150`
  - body: The fade mask is applied to the same `.chat-text--long` element that contains the button, so the Show more control is faded with the message tail. Render the control as an unmasked sibling or mask only the content body.
  - confidence: 0.78
