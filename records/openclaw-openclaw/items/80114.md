## Summary

Keep open. The report is fresh and current main still has real tension between the documented host-config hook opt-in, the agent-facing `gateway config.patch` safety allowlist, and upgrade/doctor/plugin repair paths for private plugin config; I did not find strong evidence that this is already fixed or safely closeable.

## What This Changes

The issue asks OpenClaw to stop losing private plugin `hooks.allowConversationAccess` config during upgrades and to provide a safe way to restore or preserve that hook policy without manual JSON edits.

## Best Possible Solution

Preserve existing `plugins.entries.<id>.hooks` for local/private plugins across update, doctor, and plugin repair flows, add focused regression coverage and a clear warning/docs path, and coordinate any manifest-request metadata with the open RFC without letting plugins self-grant raw conversation access.

## Reproduction Assessment

Partially. The agent-facing `gateway config.patch` rejection is source-reproducible from the gateway tool allowlist, but I did not establish a high-confidence current-main reproduction for the upgrade/doctor step that drops the private plugin `hooks` block.

## Solution Assessment

Unclear. Allowing agent-facing config.patch to write this trust policy is not obviously the best fix; a narrower maintainable solution is to preserve existing host config during upgrade/repair and handle manifest-level request metadata through the linked RFC and PR.
