## Summary

Keep open. Current main confirms the documented `agent_end` block and the agent-facing `gateway config.patch` protected-path error, but I did not find enough evidence to close this as fixed, duplicate, or not-in-repo because the reported upgrade/doctor config-drop path remains unresolved and adjacent issues only cover parts of the problem.

## What This Changes

The issue asks OpenClaw to stop local/community plugin `hooks.allowConversationAccess` from being lost or hard to restore after upgrades so `agent_end` cleanup hooks keep running.

## Best Possible Solution

Keep host approval for raw conversation hooks, then either prove and fix any install/update/doctor path that drops existing plugin hook policy or document the supported operator-owned repair path; broader manifest or state-only permission work should stay with the linked canonical items.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for `agent_end` being blocked without `allowConversationAccess` and for the agent-facing gateway tool rejecting that protected config edit, but the reported upgrade/doctor loss of the `hooks` block was not reproduced on current main.

## Solution Assessment

No for simply allowing agent-driven `gateway config.patch` to set the field. The safer path is to preserve existing operator-approved hook policy across repair/update flows and use direct operator config commands, while maintainers decide any security-sensitive expansion of the agent-facing allowlist.
