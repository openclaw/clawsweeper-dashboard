## Summary

Keep open. Current main already accepts `plugins.entries.<id>.hooks.allowConversationAccess` through the normal config schema, but the reported protected-path error still matches the agent-facing Gateway tool guard, and allowing that path would change a raw-conversation-access security boundary.

## What This Changes

The issue asks for local/community plugins to retain or set `plugins.entries.<id>.hooks.allowConversationAccess` so conversation-sensitive hooks such as `agent_end` are not blocked after upgrades.

## Best Possible Solution

Preserve or restore local plugin hook policy through explicit operator-controlled config, upgrade, or doctor flows without making the raw-conversation-access gate agent-mutable unless maintainers deliberately approve that boundary change.

## Reproduction Assessment

Yes for the protected `gateway config.patch` failure by source inspection: the agent-facing Gateway tool allowlist omits plugin hook policy paths and throws the reported protected-path error. The claimed upgrade reset of `hooks` blocks still needs a focused current-main reproduction.

## Solution Assessment

No for the previously proposed allowlist-only fix. The safer direction is an operator-controlled preservation/repair path, or an explicit maintainer decision to expand the agent-facing mutation boundary.
