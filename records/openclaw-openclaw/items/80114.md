## Summary

Keep open: current main and v2026.6.8 still enforce non-bundled conversation hooks from host config and still keep that raw-conversation grant out of the agent-facing Gateway config mutation allowlist. The old schema rejection is fixed, and the linked upstream rl-training-headers source now appears to avoid `agent_end`, but the OpenClaw-side policy/preservation question remains security-sensitive and needs maintainer direction.

## What This Changes

The issue asks OpenClaw to stop losing a local plugin's `hooks.allowConversationAccess` policy on upgrade and to provide a safe way to grant non-bundled `agent_end` access.

## Best Possible Solution

Preserve operator-authored hook policy through doctor/update/install flows or add an explicit operator approval path, while keeping raw conversation access out of model-mutable config unless maintainers approve a new trust model.

## Reproduction Assessment

Partly yes. Source inspection confirms the current runtime gate, the protected agent-facing `config.patch` path, and doctor cleanup that can remove unknown plugin entries, but the exact local plugin upgrade/reset was not live-reproduced and the linked upstream plugin now avoids `agent_end`.

## Solution Assessment

No for the broad `config.patch` allowance as stated. The safer end state is preserving explicit operator policy or adding an operator-approved grant path before raw conversation access can be changed by agent-facing tools.
