## Summary

Keep open. Current main still has an unresolved boundary between the documented host-config opt-in for raw conversation hooks, the agent-facing config.patch safety guard, and doctor/update cleanup paths that can remove plugin entries when a local plugin is not discoverable.

## What This Changes

The issue asks OpenClaw to preserve local plugin hooks.allowConversationAccess across upgrade/doctor flows or provide a safe way to restore it without manual config edits.

## Best Possible Solution

Preserve existing operator-approved hooks blocks for discoverable local plugins across update, doctor, and plugin repair flows, warn before any cleanup removes them, and keep manifest-level conversation access work tied to the open RFC without making plugins self-authorize raw transcript access.

## Reproduction Assessment

Partially. The agent-facing gateway config.patch rejection is source-reproducible from the gateway tool allowlist, and doctor stale cleanup can delete unknown plugin entries, but I did not establish a high-confidence live reproduction of the exact upgrade reset shape on current main.

## Solution Assessment

Unclear. Allowing the agent-facing gateway tool to write hooks.allowConversationAccess is not obviously the best fix; the safer first path is preserving existing host-approved config during upgrade/doctor repair and coordinating any manifest request metadata through https://github.com/openclaw/openclaw/issues/71428.
