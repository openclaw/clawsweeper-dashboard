## What This Changes

The issue asks for local/community plugin `hooks.allowConversationAccess` policy to be configurable or preserved so `agent_end` hooks keep working after upgrade/config flows.

## Best Possible Solution

Preserve already user-approved local plugin hook policy through upgrade/doctor/plugin-management paths, or add a clearly human-operated approval path, without making raw conversation access agent-mutable unless maintainers explicitly choose that boundary.

## Reproduction Assessment

Yes for the `config.patch` symptom by source inspection: the agent Gateway tool allowlist omits this path and rejects changed protected paths before forwarding the write. The upgrade/drop mechanism still needs a live or targeted test reproduction.

## Solution Assessment

No, the closed allowlist-only PR is not the best fix because it changes the model-to-operator trust boundary for a raw-conversation hook gate. A safer direction is preserving explicit human-approved config or designing a reviewed human approval path.
