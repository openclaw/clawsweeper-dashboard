## Summary

Keep open: current main still requires explicit host config for non-bundled conversation hooks, and the agent-facing `gateway config.patch` guard still blocks this raw-conversation access path. The related PR that tried to allow the path was closed unmerged after a security-boundary review, so maintainers still need to choose the safer fix shape.

## What This Changes

The issue asks OpenClaw to keep local-plugin `hooks.allowConversationAccess` usable across upgrades and provide a safe way to set it when `agent_end` is blocked.

## Best Possible Solution

Keep this open until maintainers choose a fix that preserves explicit operator approval for local plugin conversation hooks without making the agent-facing config mutation boundary broader by default.

## Reproduction Assessment

Partly. The protected `config.patch` rejection is source-reproducible through the agent-facing Gateway tool guard, while the upgrade/doctor reset portion still needs a live or focused test reproduction.

## Solution Assessment

No. Adding `allowConversationAccess` to the agent Gateway tool allowlist is not the best fix without maintainer security approval; safer directions are preserving operator-set hook policy through update/doctor or resolving the manifest-permission design without self-authorization.
