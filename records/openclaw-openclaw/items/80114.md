## Summary

Keep open. Current main still blocks the agent-facing `gateway config.patch` path for this hook-policy field, and https://github.com/openclaw/openclaw/pull/80190 is an open closing PR with a security-boundary review finding rather than a merged fix.

## What This Changes

The issue asks OpenClaw to keep or restore `plugins.entries.<id>.hooks.allowConversationAccess` for local/community plugins so `agent_end` cleanup hooks are not blocked after upgrades.

## Best Possible Solution

Preserve or restore operator-approved plugin hook policy through upgrade/doctor/config-management paths, or add an explicitly human-operated approval path; keep the agent-facing `config.patch` allowlist closed unless maintainers deliberately change that security boundary.

## Reproduction Assessment

Yes for the `gateway config.patch` rejection: current source lacks this path in the agent Gateway tool allowlist and throws on disallowed changed paths. The upgrade reset portion is still based on reporter evidence rather than a current-main execution reproduction in this read-only pass.

## Solution Assessment

No for the open PR's direct allowlist approach as the default fix. It addresses the symptom, but the safer maintainable path is to preserve operator-approved hook policy or require a clearly human-operated approval flow before granting raw conversation access.
