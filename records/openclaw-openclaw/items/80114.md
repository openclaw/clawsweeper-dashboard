## Summary

Keep this open: current main still enforces the conversation-hook gate and still keeps plugin hook policy out of the agent-facing Gateway config mutation allowlist, while doctor/update-side stale plugin cleanup can remove unknown plugin entries when discovery says they are stale.

## What This Changes

The issue asks OpenClaw to preserve or provide a safe way to set per-plugin `hooks.allowConversationAccess` for a local/community plugin whose `agent_end` hook is blocked after upgrades.

## Best Possible Solution

Preserve or restore operator-authored plugin hook policy through upgrade/doctor/install flows, or add an explicit human-operated approval path, while keeping raw conversation access out of agent-mutable config unless maintainers intentionally change that security model.

## Reproduction Assessment

Partly yes: source inspection confirms the runtime `agent_end` block without `allowConversationAccess` and the agent Gateway protected-path rejection for this field. The specific upgrade/doctor reset still needs live validation against the reported local plugin setup.

## Solution Assessment

No for the broad `config.patch` allowance as stated: making this agent-mutable is not the safest default fix because it grants raw conversation access to plugins. The safer end state is preserving existing operator policy during repair/update or adding an explicit operator approval flow.
