## Summary

Keep open. Current main still requires `plugins.entries.<id>.hooks.allowConversationAccess=true` for non-bundled `agent_end` hooks, direct config accepts that field, but the agent-facing `gateway config.patch` guard does not allow plugin hook-policy paths; the linked open fix PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks OpenClaw to preserve or restore `plugins.entries.<id>.hooks.allowConversationAccess` for local/community plugins after upgrades and make that existing hook-policy field configurable through `gateway config.patch`.

## Best Possible Solution

Keep explicit host approval for raw conversation hooks, then either land the linked narrow allowlist fix after security review or implement an upgrade-preservation path that stops dropping trusted local plugin hook policy.

## Reproduction Assessment

Yes, from source inspection. A patch that changes `plugins.entries.rl-training-headers.hooks.allowConversationAccess` reaches the gateway config mutation guard, and current main has no matching allowed path, so it is rejected before the existing valid config field can restore the non-bundled `agent_end` hook permission.

## Solution Assessment

Unclear. Allowing this exact path through `gateway config.patch` is narrow and matches the linked PR, but maintainers should explicitly approve the security tradeoff or choose an upgrade-preservation fix that avoids expanding the agent-facing config write surface.
