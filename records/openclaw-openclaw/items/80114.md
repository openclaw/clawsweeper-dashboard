## What This Changes

The issue asks OpenClaw to keep or make settable `plugins.entries.rl-training-headers.hooks.allowConversationAccess` so the local plugin's `agent_end` cleanup hook is not lost after upgrades.

## Best Possible Solution

Keep explicit host approval for raw conversation hooks, fix any confirmed upgrade path that drops existing plugin hook policy, and resolve manifest request metadata through the existing RFC/PR without making plugin manifests self-authorizing.

## Reproduction Assessment

Partial. Source confirms that a non-bundled `agent_end` hook is blocked without `plugins.entries.<id>.hooks.allowConversationAccess=true` and that the agent-facing gateway tool protects plugin trust paths, but I did not establish a current-main path where upgrade or doctor drops an existing hooks block.

## Solution Assessment

Unclear. Preserving existing hook policy during a confirmed upgrade rewrite would be narrow, but allowing agent-driven `config.patch` or manifest self-authorization for raw conversation hooks needs maintainer security/product judgment.
