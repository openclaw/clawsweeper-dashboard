## Summary

Keep open. Current main still rejects this through the agent-facing Gateway `config.patch` guard, while the underlying hook gate and schema/docs confirm `allowConversationAccess` is a real operator trust setting; the linked quick fix was closed unmerged after security review flagged the raw-conversation access boundary.

## What This Changes

The issue requests a supported way for local/community plugin hook policy, especially `hooks.allowConversationAccess` for `agent_end`, to survive upgrade and be configurable without repeated manual base-config edits.

## Best Possible Solution

Preserve explicit operator-approved plugin hook policy across upgrade/install/doctor flows or provide a clearly operator-only config path, while keeping manifest permission work as request metadata and not making the agent Gateway tool grant raw conversation access without maintainer security approval.

## Reproduction Assessment

Yes for the `gateway config.patch` half by source inspection: current main lacks this path in the agent Gateway allowlist, so changing `plugins.entries.<id>.hooks.allowConversationAccess` is rejected before the RPC write. The upgrade reset half still needs a live or focused test reproduction.

## Solution Assessment

No for the closed allowlist-only PR: it fixes the symptom by making a raw-conversation trust gate agent-mutable. A safer solution should preserve operator config or add an explicit maintainer-approved consent path.
