## Summary

Keep open. Current main still has a source-reproducible agent-facing `gateway config.patch` rejection for `plugins.entries.<id>.hooks.allowConversationAccess`, and the open linked implementation PR uses closing syntax but is blocked on a security/product boundary decision.

## What This Changes

The issue asks OpenClaw to preserve or make configurable the local plugin `hooks.allowConversationAccess` policy so non-bundled `agent_end` hooks keep working after upgrades.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/80190 or replace it with a safer narrow fix that preserves local plugin hook policy through upgrade/config management without weakening the agent-to-operator security boundary.

## Reproduction Assessment

Yes. Source inspection shows current main blocks the agent-facing `config.patch` before forwarding when the patch changes `plugins.entries.<id>.hooks.allowConversationAccess`, while the plugin registry still requires that flag for non-bundled `agent_end` hooks.

## Solution Assessment

No for the current linked PR as written. Adding the field to the agent allowlist fixes the symptom, but the safer maintainable path is an explicit maintainer decision on whether this raw-conversation-access gate may be agent-mutable, or a fix in upgrade/config preservation instead.
