## Summary

Keep open: current main still blocks agent-facing `gateway config.patch` from changing plugin hook trust policy, while non-bundled `agent_end` remains gated on `plugins.entries.<id>.hooks.allowConversationAccess=true`. The closed linked PR addressed the symptom by making this raw-conversation access gate agent-mutable, but that was rejected as a security-boundary concern and did not land.

## What This Changes

The issue reports that a local `rl-training-headers` plugin loses `hooks.allowConversationAccess` during upgrades and cannot restore it through `gateway config.patch`, causing its `agent_end` cleanup hook to be blocked.

## Best Possible Solution

Keep raw conversation hook approval explicit and operator-controlled, then fix the upgrade/doctor/config-management path to preserve or clearly warn about local plugin hook policy instead of making the agent Gateway tool grant that trust.

## Reproduction Assessment

Yes, by source inspection for the `gateway config.patch` part: current main lacks the plugin hook policy path in the agent Gateway tool allowlist, so changing `plugins.entries.<id>.hooks.allowConversationAccess` is rejected before the RPC write is forwarded. I did not run a live upgrade/doctor reproduction for the reported hook-block reset.

## Solution Assessment

No for the previously proposed direct allowlist fix: it addresses the symptom but changes the model-to-operator trust boundary for a raw-conversation access gate. The safer solution is preserving explicit operator-set hook policy through update/doctor/config flows or adding a reviewed consent/manifest workflow that does not self-grant access.
