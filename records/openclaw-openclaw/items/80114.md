## Summary

Keep open: current main still blocks non-bundled `agent_end` without host hook policy and still keeps that policy out of the agent-facing Gateway `config.patch` allowlist; the older schema rejection is fixed, but the upgrade/preservation and security-model question remains.

## What This Changes

The issue asks OpenClaw to stop local/community plugins such as `rl-training-headers` from losing or being unable to restore `hooks.allowConversationAccess` after upgrade/doctor flows.

## Best Possible Solution

Preserve operator-authored plugin hook policy through update/doctor/install flows or add an explicit operator approval path, while keeping raw conversation access out of agent-mutable config unless maintainers intentionally change that trust model.

## Reproduction Assessment

Partly yes. Source inspection confirms the runtime `agent_end` gate, the protected agent-facing `config.patch` path, and doctor cleanup that can remove unknown plugin entries; the exact local plugin upgrade/reset still needs live validation.

## Solution Assessment

No for the broad `config.patch` allowance as stated. The safer fix is to preserve existing operator policy through repair/update or require an explicit operator approval path before raw conversation access can be granted.
