## Summary

Keep open. Current main still rejects this path through the agent-facing Gateway tool while `agent_end` remains blocked for non-bundled plugins without the host-config trust gate; the linked open fix PR uses closing syntax but has a security-boundary review concern that needs maintainer judgment before this issue can close.

## What This Changes

The issue requests a reliable way for a local community plugin to keep or set `plugins.entries.<id>.hooks.allowConversationAccess` so its `agent_end` cleanup hook is not blocked after upgrade/config changes.

## Best Possible Solution

Review or replace https://github.com/openclaw/openclaw/pull/80190 with a fix that preserves explicit host approval for raw conversation hooks while addressing the local plugin config-management failure.

## Reproduction Assessment

Yes. By source inspection, the agent-facing Gateway tool rejects `plugins.entries.<id>.hooks.allowConversationAccess` as a protected path, and the plugin registry blocks non-bundled `agent_end` unless that policy is true.

## Solution Assessment

No for the linked PR as currently written. Allowing the agent-facing Gateway tool to grant raw conversation access is a security-boundary change; a safer fix should preserve operator approval and target upgrade/config preservation or an explicitly maintainer-approved operator path.
