## Summary

Keep open. Current main fixes the older schema-validator problem and appears to preserve existing plugin entry subfields in install/auto-enable paths, but this report still raises a security-sensitive policy question about local plugin conversation-hook approval after upgrade and the agent-facing gateway tool intentionally blocks that config mutation.

## What This Changes

Request that local/community plugin hook policy for `rl-training-headers` survive upgrades or become settable/discoverable without manual JSON edits.

## Best Possible Solution

Keep explicit operator approval for raw conversation hooks, then either preserve that approval for still-installed local plugins during update/doctor or emit a clear repair warning when discovery cannot find the plugin.

## Reproduction Assessment

Partial yes from source: non-bundled `agent_end` is blocked without `allowConversationAccess`, and the agent-facing gateway tool rejects that config mutation as protected. No high-confidence current-main reproduction was established for the claimed upgrade path dropping an existing `hooks` block.

## Solution Assessment

Unclear. The safest path is not to let plugin manifests or agent-facing `config.patch` self-grant raw conversation access; maintainers should decide between preserving explicit local operator config, improving diagnostics/docs, and the related manifest-metadata RFC.
