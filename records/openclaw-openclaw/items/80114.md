## Summary

Keep open. Current main still blocks this exact agent-facing `gateway config.patch` mutation by design, direct human-operated config paths already accept `allowConversationAccess`, and the remaining direction is a security/product decision rather than a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to keep or restore `plugins.entries.<id>.hooks.allowConversationAccess` for a local community plugin so its `agent_end` cleanup hook is not blocked after upgrade/config workflows.

## Best Possible Solution

Maintain explicit human approval for raw conversation hooks, then have maintainers choose a safe path: preserve existing local plugin hook policy through update/doctor, add a reviewed manifest/consent workflow, or document the supported human-operated config command and direct config edit path.

## Reproduction Assessment

Yes for the protected `gateway config.patch` path by source inspection: current main rejects non-allowlisted config paths before forwarding the RPC. The separate claim that update/doctor drops the hooks block still needs a focused runtime reproduction.

## Solution Assessment

No for the closed PR direction of simply allowlisting this field in the agent-facing gateway tool, because it weakens a raw conversation-access trust boundary. A safer solution should target human-approved config persistence, manifest request/consent, or clearer docs without making the agent mutate the gate by default.
