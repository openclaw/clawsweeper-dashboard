## Summary

Keep open: current main still blocks the agent-facing `gateway config.patch` path from changing this raw-conversation hook gate, and the directly related fix PR was closed unmerged after a security-boundary review. Operator-owned config paths now accept the field, so the remaining decision is how to preserve or restore local plugin hook policy without making the agent tool able to grant conversation access.

## What This Changes

The issue asks OpenClaw to stop local/community plugins from losing `plugins.entries.<id>.hooks.allowConversationAccess` after upgrades and to provide an operator-usable way to set it for `agent_end`.

## Best Possible Solution

Preserve or restore operator-approved local plugin hook policy through upgrade/doctor/operator config paths, and keep the agent-facing gateway tool restricted unless maintainers approve a security-boundary change.

## Reproduction Assessment

Yes for the protected-path error: source inspection shows `plugins.entries.*.hooks.allowConversationAccess` is absent from the agent gateway config mutation allowlist, so that path still throws before forwarding the write. The upgrade-reset part still needs a live or fixture-backed current-main reproduction.

## Solution Assessment

No if the solution is to add this field to the agent-facing allowlist; that was already reviewed as a raw-conversation access boundary expansion. A safer solution is operator-owned config persistence, targeted upgrade preservation, or a reviewed manifest/request workflow that keeps explicit host approval.
