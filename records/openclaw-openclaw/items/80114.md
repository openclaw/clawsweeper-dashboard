## Label Justifications

- `P2`: This is a real plugin hook/config preservation problem with security-sensitive impact, but it is limited to trusted non-bundled plugin setups rather than core runtime availability.
- `impact:security`: The affected flag controls whether non-bundled plugins can read raw conversation content through typed hooks.
- `impact:session-state`: The reported `agent_end` block prevents plugin cleanup and can leave stale session identifiers in plugin state across runs.

## Summary

Keep this open: current main documents and enforces the host config gate, but the reported recovery path through the agent-facing Gateway `config.patch` remains intentionally protected and the upgrade/doctor preservation problem is not proven fixed.

## What This Changes

A local/community plugin can lose or fail to restore `plugins.entries.<id>.hooks.allowConversationAccess`, causing its `agent_end` cleanup hook to be blocked after upgrade or config repair.

## Best Possible Solution

Preserve or restore local plugin hook policy through upgrade/doctor/install/config-management, or add an explicit human-operated approval flow, while keeping raw conversation access out of the agent-mutable Gateway config path unless maintainers intentionally change that security model.

## Reproduction Assessment

Partly: source inspection confirms the runtime `agent_end` block without `allowConversationAccess` and the agent Gateway protected-path rejection for this field. A live upgrade/doctor reproduction of the hook-policy reset was not run.

## Solution Assessment

No, allowing the agent-facing Gateway `config.patch` tool to set this field is not the best default fix because it changes a raw-conversation-access trust boundary. The safer direction is preserving existing host policy through upgrade/doctor flows or adding a human-operated approval path.
