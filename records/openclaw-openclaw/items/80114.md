## Summary

Keep open. Current main still has the reported conflict between explicit host-approved conversation hook policy, the protected agent-facing config patch surface, and doctor stale-plugin repair paths that can remove unknown plugin entries; this is not safely closeable as implemented, duplicate, or external-only for https://github.com/openclaw/openclaw/issues/80114.

## What This Changes

The issue asks OpenClaw to stop losing `plugins.entries.<id>.hooks.allowConversationAccess` for a local plugin during upgrade/doctor repair and to provide an actionable way to restore or warn about that protected hook policy.

## Best Possible Solution

Preserve existing operator-approved `plugins.entries.<id>.hooks` for local/private plugins across update, doctor, and plugin repair flows, add targeted warnings/docs, and keep raw conversation-hook access behind explicit host approval.

## Reproduction Assessment

No full high-confidence live reproduction yet. Source inspection proves the protected `gateway config.patch` rejection and a doctor stale-plugin cleanup path that can delete unknown plugin entries, but the exact `rl-training-headers` upgrade reset still needs a focused current-main repro.

## Solution Assessment

Unclear. Letting agent-facing `config.patch` write conversation-hook trust policy is not the safest fix; preserving existing host-authored hook policy during update/repair while coordinating manifest request metadata with https://github.com/openclaw/openclaw/issues/71428 is the narrower maintainable direction.
