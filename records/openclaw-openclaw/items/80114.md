## Summary

Keep open. Current main still blocks the agent-facing Gateway tool from changing this raw-conversation hook gate, while direct operator config paths already support the field; the remaining work needs maintainer judgment around upgrade/config preservation without weakening the model-to-operator trust boundary.

## What This Changes

The issue asks for local/community plugin `hooks.allowConversationAccess` to survive upgrade/config-management flows so `agent_end` hooks can run without manual `openclaw.json` edits.

## Best Possible Solution

Preserve existing per-plugin hook policy through install/update/doctor config flows and document a human-operated recovery path; do not make raw-conversation access agent-mutable without explicit maintainer approval.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the agent-facing `gateway config.patch` rejection and the `agent_end` hook gate, but the reported upgrade reset still needs a live `doctor --fix` or `openclaw update` reproduction for this local plugin.

## Solution Assessment

No for the closed allowlist PR path. The safer solution is to preserve or restore operator-owned plugin hook config during config-management flows, while keeping manifest-level permission work in https://github.com/openclaw/openclaw/issues/71428.
