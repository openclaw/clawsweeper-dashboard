## Summary

Keep this open. Current main still blocks the reported agent-facing `gateway config.patch` path for `plugins.entries.<id>.hooks.allowConversationAccess`, while the schema and runtime hook policy already recognize that field; the remaining question is whether granting the agent Gateway tool this sensitive config mutation is acceptable.

## What This Changes

The issue requests a way for local/community plugins to retain or restore `plugins.entries.<id>.hooks.allowConversationAccess` so `agent_end` hooks are not blocked after upgrades.

## Best Possible Solution

Keep the runtime hook gate, then add a maintainer-approved operator path to restore `allowConversationAccess` without accidentally broadening model-driven config writes; the manifest permission discussion can remain in the existing RFC thread.

## Reproduction Assessment

Yes from source inspection: the agent-facing Gateway tool computes the plugin hook policy leaf path and rejects it because the current allowlist has no `plugins.entries.*.hooks.allowConversationAccess` pattern. I did not run a live Gateway invocation in this read-only review.

## Solution Assessment

Unclear. The unmerged referenced commits are a narrow code change, but the change grants the agent Gateway tool permission to enable raw-conversation plugin hooks, so maintainers should decide whether that is the right trust boundary or whether a separate operator-only repair path is safer.
