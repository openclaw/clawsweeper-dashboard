Full review comments:

- **[P2] Register auto-recall even when startup config disables it:** `extensions/memory-core/src/auto-recall.ts:80-82`
  - body: This return means the `before_prompt_build` hook is never registered unless some autoRecall setting is already enabled at plugin startup. Since the handler reads `api.runtime.config.loadConfig()` per turn, enabling `agents.defaults.memorySearch.autoRecall` later will still do nothing until restart; register the hook unconditionally and let the handler no-op from effective config.
  - confidence: 0.93
- **[P2] Warn from the effective memory search config:** `extensions/memory-core/src/auto-recall.ts:63-64`
  - body: This suppresses the recall-only warning whenever defaults say `autoRecall.enabled`, but `resolveMemorySearchConfig` still returns `null` if effective `memorySearch.enabled` is false and agent overrides can disable autoRecall. In those cases recall-only removes `MEMORY.md` and no per-turn recall runs, leaving the agent with no automatic memory context and no startup warning.
  - confidence: 0.88
- **[P2] Let channel plugins normalize thread targets first:** `src/utils/delivery-context.ts:119-126`
  - body: This fallback now runs before `messaging.resolveDeliveryTarget` for Telegram, so topic conversations skip Telegram's parser and return the raw conversation id as `threadId`. That bypasses the plugin-owned normalization path and can send isolated or cron replies to an inconsistent topic target.
  - confidence: 0.91
- **[P2] Remove the messaging temp-dir guard allowlist expansion:** `scripts/check-no-random-messaging-tmp.mjs:20-33`
  - body: This PR is about memory config, but it also exempts many messaging and plugin runtime paths from the `os.tmpdir()` guard. That weakens a security-adjacent boundary check without fixing the underlying temp-directory ownership, so split or remove these allowlist additions before merging the memory feature.
  - confidence: 0.86
