## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Register auto-recall even when initially disabled:** `extensions/memory-core/src/auto-recall.ts:80-81`
  - body: The registrar returns before installing the `before_prompt_build` hook when startup config has no autoRecall enabled. Since the handler is supposed to consult live config per turn, turning autoRecall on later has no effect until restart.
  - confidence: 0.9
- **[P2] Use the runtime config snapshot API:** `extensions/memory-core/src/auto-recall.ts:90`
  - body: Bundled plugin production code must not call deprecated `api.runtime.config.loadConfig()`. Use `api.runtime.config.current()` or passed config here so the plugin follows the SDK runtime contract and the config-boundary guard remains green.
  - confidence: 0.88
- **[P2] Resolve warnings from effective recall availability:** `extensions/memory-core/src/auto-recall.ts:63-67`
  - body: This warning resolver treats raw default or agent `autoRecall.enabled` as enough, but `resolveMemorySearchConfig` can still return null when memory search is disabled or an agent override disables recall. In `recall-only` mode those agents lose automatic memory context without the intended warning.
  - confidence: 0.84
- **[P2] Let channel plugins normalize thread delivery first:** `src/utils/delivery-context.ts:118-126`
  - body: This new early return runs before `messaging.resolveDeliveryTarget` for Telegram, Slack, and Mattermost thread children. Telegram topic delivery relies on the plugin resolver to normalize the target and thread id, so this can route with raw conversation ids.
  - confidence: 0.89
