## Summary

Keep open: the PR is a focused fix for a real Gemini replay bug and now has adequate live proof, but the patch still has a blocking edge where foreign provider thought signatures can be replayed to Gemini.

## What This Changes

The branch updates the native Google transport to merge streamed tool-call thoughtSignature parts, collect and replay prior Gemini tool-call signatures, sanitize outbound signatures, and add regression tests.

## Best Possible Solution

Keep the native Google fix, but gate collected/replayed tool-call signatures to real same-route Gemini history and add regression coverage for foreign signatures being omitted.

## Reproduction Assessment

Yes: source inspection and the linked live Gemini API matrix give a high-confidence reproduction path for the missing thoughtSignature replay 400, although I did not rerun the live API myself in this read-only review.

## Solution Assessment

No: the overall direction is appropriate, but the implementation should not trust sanitized-looking signatures from other providers or models because Gemini requires returning its own signatures exactly as received.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate replayed signatures to the same Gemini route:** `extensions/google/transport-stream.ts:218-222`
  - body: `collectToolCallThoughtSignatures` now records signatures from every assistant toolCall, and the serializer then uses `block.thoughtSignature` or the collected value after only a charset/length filter. Cross-provider values such as `msg_abc1234` or `foreign_sig` can pass that sanitizer, so switching a mixed history to Gemini can send a non-Gemini signature instead of omitting it, reintroducing provider rejection. Please collect/use signatures only from the same Google provider/API/model route, and add a Gemini 3 cross-provider regression test.
  - confidence: 0.9
