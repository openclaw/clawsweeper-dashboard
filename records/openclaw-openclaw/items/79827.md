## Summary

Keep open: this is a valid focused Gemini replay bug fix with useful parser/replay work and live API proof, but exact-head CI still fails and source inspection shows the patch still drops Google’s documented dummy fallback and trusts stored signatures without checking `api`. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/thought-signatures?utm_source=openai))

## What This Changes

The PR updates the native Google transport to merge streamed tool-call thought signatures, replay/sanitize Gemini tool-call signatures, and add Google transport regression tests.

## Best Possible Solution

Keep the Google-owner-module repair, but preserve exact-route real signatures, keep Google’s dummy fallback only for unsigned current-turn replay, and repair the test narrowing/lint failures.

## Reproduction Assessment

Yes: the linked issue has a direct Gemini API matrix for the replay 400, and current-main source/tests show the native Google replay shape. I did not rerun the paid Gemini API in this read-only review.

## Solution Assessment

No: the direction is right, but this patch is not yet the safest fix because it removes the documented fallback, misses api in one trust guard, and fails exact-head CI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep the Gemini dummy fallback reachable:** `extensions/google/transport-stream.ts:537-541`
  - body: `sanitizeGeminiToolCallThoughtSignature` rejects `skip_thought_signature_validator`, so the fallback directly below is always erased. Google documents that dummy value for unavoidable custom/cross-model function-call blocks without signatures, and current main has a regression test for that replay path, so omitting it can reintroduce current-turn 400s. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/thought-signatures?utm_source=openai))
  - confidence: 0.9
- **[P2] Guard stored signatures by api too:** `extensions/google/transport-stream.ts:536`
  - body: The new collector checks provider/api/model, but `ownSignature` still uses `isSameProviderAndModel`, which ignores `api`. A Google OpenAI-compatible history entry with the same model id can still pass a non-native signature into the native Gemini serializer.
  - confidence: 0.88
- **[P2] Use typed find helpers for model turns:** `extensions/google/transport-stream.test.ts:764`
  - body: Exact-head CI still reports `No overload matches this call` on this `filter((c: { role: string }) => ...)` pattern, and lint also flags the same lookup style at line 823. Use a typed predicate or `find`/narrowing helper so the changed test file compiles and passes lint.
  - confidence: 0.96
