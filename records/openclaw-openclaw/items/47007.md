## Summary

Keep open, but the patch is not mergeable as-is: source review confirms multiple PR-introduced Feishu delivery regressions, current main has diverged around agent delivery, and external real behavior proof is missing.

## What This Changes

The PR adds a ChannelPlugin outbound-dispatcher hook and a Feishu streaming bridge/status UI so subagent runs can stream into Feishu cards.

## Best Possible Solution

Rework this against the current agent delivery/message lifecycle, keep block streaming opt-in, preserve Feishu thread/root metadata, avoid a second bridge for main delivered runs, and add Feishu regression coverage plus redacted live proof.

## Reproduction Assessment

Yes, at source level. The PR head directly shows the default block-streaming inversion, the main-run bridge condition, and the dropped thread identifier; I did not run a live Feishu workspace.

## Solution Assessment

No. The safest direction is a current-main rework that wires only subagent-visible streaming through the existing delivery lifecycle, preserves routing metadata, and reuses the throttled streaming update path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep block streaming opt-in by default:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: With `account.config?.blockStreaming !== false`, accounts that do not set the new option now enable block streaming. That lets internal block chunks fall through to Feishu delivery by default, changing the previous default-off behavior and risking fragmented partial replies in normal chats.
  - confidence: 0.9
- **[P1] Limit the streaming bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: This condition also creates the bridge for normal delivered main-agent runs. Those runs still go through the regular final delivery path after `runAgentAttempt`, so Feishu can receive both a streamed card and the final delivered reply for the same command.
  - confidence: 0.86
- **[P2] Forward the actual Feishu thread identifier:** `extensions/feishu/src/channel.ts:499`
  - body: Reducing `threadId` to `replyInThread: Boolean(...)` drops the ID needed when there is no `replyToId`. In thread-only subagent contexts the streaming session has neither `replyToMessageId` nor `rootId`, so it falls back to creating a top-level card outside the originating thread.
  - confidence: 0.88
- **[P2] Throttle replacement streaming updates:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent` bypasses the session's existing 100 ms update throttle and coalescing logic. Since the PR routes forced status/model updates through this method, rapid updates can enqueue many card update requests instead of being collapsed like normal streaming updates.
  - confidence: 0.78
