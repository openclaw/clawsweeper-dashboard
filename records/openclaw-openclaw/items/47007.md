## Summary

Keep this PR open for human/contributor follow-up: the Feishu subagent streaming feature is not already implemented on current main, but this branch is stale, conflicting, missing inspectable real Feishu proof, and still introduces source-visible delivery/threading/block-streaming regressions.

## What This Changes

The PR adds a channel outbound dispatcher hook plus Feishu reply-dispatcher and streaming-card changes intended to stream subagent output, reasoning, and tool status in Feishu cards.

## Best Possible Solution

Land a refreshed Feishu subagent streaming path through an SDK-safe additive delivery/preview seam that preserves topic/root routing, keeps block streaming opt-in, throttles card updates, adds focused tests/docs, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: the PR builds the bridge for delivered main-agent runs, drops Feishu thread ids, flips block-streaming defaults, imports core internals, and bypasses update throttling. No high-confidence real reproduction of the intended Feishu improvement is available because no inspectable Feishu proof was attached.

## Solution Assessment

No. The current patch is broader than the subagent-only problem, stale against current main, and weakens Feishu delivery contracts; the safer solution is a narrower SDK-safe streaming/delivery seam with preserved routing and proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The new bridge is created for `opts.deliver === true` as well as subagent runs, but delivered main-agent commands still go through normal final delivery. On Feishu this can stream a card and then send the final reply again for an ordinary delivered run.
  - confidence: 0.92
- **[P1] Preserve Feishu thread ids in the outbound dispatcher:** `extensions/feishu/src/channel.ts:499`
  - body: The dispatcher converts `threadId` to a boolean and never passes the actual topic/root id. When there is no `replyToId`, the streaming session has no routing identifier and can create the card outside the originating thread.
  - confidence: 0.9
- **[P2] Keep Feishu block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-434`
  - body: `account.config?.blockStreaming !== false` enables block streaming whenever the option is unset, and the PR also forces `disableBlockStreaming: false`. Default Feishu accounts could receive fragmented block messages that current main keeps disabled unless explicitly enabled.
  - confidence: 0.9
- **[P2] Use SDK-safe types at the plugin boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: This production Feishu plugin file imports a type from `../../../src/auto-reply/types.js`, crossing the extension boundary. The type should come from the plugin SDK or a Feishu-owned runtime API before adding a plugin-facing seam.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent()` queues a CardKit update every call, while the new model-name animation reaches it once per character at 30 ms intervals. That bypasses the existing streaming throttle and can burst Feishu API calls or sequence updates.
  - confidence: 0.82
