## Summary

Keep this PR open but do not merge as-is: the Feishu subagent streaming direction is useful, but the branch introduces delivery regressions, violates the plugin boundary, is stale against the moved agent-command code, and lacks real Feishu proof artifacts.

## What This Changes

The PR adds a Feishu outbound streaming bridge through a new channel plugin hook, expands Feishu streaming-card updates for subagent/tool/reasoning progress, and adds block-streaming config handling.

## Best Possible Solution

Rework this as an SDK-safe, subagent-only outbound streaming path on current main, preserve Feishu thread anchors and block-streaming defaults, throttle card updates, add focused regression tests, and attach real redacted Feishu proof before merge.

## Reproduction Assessment

Yes for the source-level blockers: the PR head shows duplicate-delivery, block-streaming default, thread-anchor, and throttle problems directly in the changed code. No live Feishu reproduction artifact was provided for the claimed improved behavior.

## Solution Assessment

No. The product direction may be reasonable, but this diff is not the narrowest maintainable solution because it widens a core channel seam, applies to main delivered runs, and regresses documented Feishu defaults; a rebased subagent-only bridge with SDK-facing types is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict outbound streaming to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for every `opts.deliver === true` run, its callbacks are passed into `runAgentAttempt`, and the same turn still reaches `deliverAgentCommandResult` afterward. On Feishu this can stream a card during a normal delivered main-agent run and then send the final reply again, so the bridge should be limited to subagent runs or made to own final delivery explicitly.
  - confidence: 0.9
- **[P1] Keep Feishu block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: `account.config?.blockStreaming !== false` makes an unset Feishu config behave as enabled, and the returned `disableBlockStreaming: false` asks core to emit block replies by default. Current Feishu docs/tests keep block streaming off unless explicitly true, so this would send internal block fragments to existing accounts.
  - confidence: 0.92
- **[P2] Preserve Feishu thread anchors in the bridge:** `extensions/feishu/src/channel.ts:498-500`
  - body: The outbound dispatcher converts `threadId` into a boolean and drops the actual thread/root identifier. When no `replyToId` is present, `FeishuStreamingSession.start()` has neither `replyToMessageId` nor `rootId`, so it falls back to top-level card creation instead of streaming inside the originating thread.
  - confidence: 0.86
- **[P2] Keep Feishu plugin imports on the SDK boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: The Feishu plugin now imports a type from `../../../src/auto-reply/types.js`, which crosses out of the extension package into core internals. Bundled plugins are meant to use `openclaw/plugin-sdk/*` or local barrels, so this seam should expose the needed type through the SDK/runtime API instead.
  - confidence: 0.84
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent()` clears pending throttled updates and immediately enqueues an API update, while the model-name animation calls it every 30 ms per character. This bypasses the existing 100 ms streaming-card throttle and can create Feishu rate-limit or sequence-ordering failures under normal model-name animation.
  - confidence: 0.8
