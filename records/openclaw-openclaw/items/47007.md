## Summary

Keep this PR open, but it is not merge-ready: the requested Feishu subagent streaming feature is still useful and current main does not expose the proposed outbound-dispatcher hook, yet the branch has source-level P1 routing/duplication blockers and is stale against the current agent-command split. PR URL: https://github.com/openclaw/openclaw/pull/47007

## What This Changes

The PR adds a channel outbound-dispatcher hook and Feishu streaming-card wiring so delivered and subagent runs can stream partial text, tool status, reasoning, and status updates.

## Best Possible Solution

Revise the feature as a subagent-only, SDK-documented streaming path that preserves Feishu reply/thread/root routing end to end, keeps block streaming opt-in, coalesces card updates, and adds focused Feishu plus agent-command contract coverage.

## Reproduction Assessment

Yes. The PR blockers are source-reproducible from the diff and current main: the bridge is enabled for delivered runs that still final-send, and the Feishu dispatcher drops the concrete thread/root id before streaming-card routing chooses between reply, root, and top-level sends.

## Solution Assessment

No. The proposed approach expands a public channel contract while reusing Feishu reply streaming without preserving delivery, thread, block-streaming, and update-throttling semantics; a subagent-only bridge integrated with current routing is safer.

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:735-771`
  - body: The new guard also builds a streaming bridge for ordinary delivered main-agent turns. Those turns still go through normal outbound delivery after the run, so Feishu can stream a card and then send the final reply again for the same execution. Gate this bridge to lanes that lack an inbound dispatcher, starting with the subagent lane.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces threadId to a boolean and drops the actual topic/root id. When replyToId is absent, FeishuStreamingSession.start has no concrete thread target and falls back to top-level message creation, so streamed subagent output can leave the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: account.config?.blockStreaming !== false enables block delivery whenever the setting is unset, reversing the documented/current behavior that non-Telegram channels require explicit blockStreaming: true. Default Feishu configs would start receiving fragmented internal block chunks as chat messages.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: replaceContent bypasses the existing streaming-card throttle, while the new status/model-name paths can call it repeatedly, including per-character animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK-facing type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports ModelSelectedContext through a relative ../../../src/** path, crossing the bundled plugin boundary. Expose the needed callback type through the SDK or a local plugin facade instead.
  - confidence: 0.78
