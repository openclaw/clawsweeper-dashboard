## Summary

Keep open: current main has Feishu subagent delivery-target hooks, but it does not implement this PR's real-time Feishu subagent streaming bridge. The proposed patch is not mergeable as-is because it introduces source-level delivery/threading/block-streaming regressions and lacks after-fix real behavior proof.

## What This Changes

The branch adds a ChannelPlugin outbound dispatcher hook, wires Feishu delivered/subagent runs into FeishuStreamingSession with tool/reasoning card updates, and adds a Feishu blockStreaming config key.

## Best Possible Solution

Rework this as a subagent-only channel/agent streaming seam that preserves Feishu thread roots, keeps block streaming opt-in, throttles card updates, and includes focused regression coverage plus real Feishu proof.

## Reproduction Assessment

Yes, at source level. The PR diff creates the streaming bridge for delivered main-agent runs and changes unset Feishu blockStreaming semantics, both of which can be traced directly against current delivery and Feishu dispatcher behavior.

## Solution Assessment

No. The direction is useful, but the submitted bridge is too broad and should be narrowed to subagents while reusing current Feishu thread-binding and delivery contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict outbound streaming bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: `opts.deliver === true` creates a Feishu streaming bridge for normal delivered main-agent commands, but the command still runs the standard final delivery path afterward. That can produce a streaming card plus a second final reply for every delivered Feishu run; keep this bridge subagent-only or make it exclusively own final delivery.
  - confidence: 0.9
- **[P1] Pass the Feishu thread root into streaming starts:** `extensions/feishu/src/channel.ts:498-500`
  - body: The dispatcher converts `threadId` to a boolean and drops the actual id. In thread-only subagent contexts with no `replyToId`, `FeishuStreamingSession.start()` has no `replyToMessageId` or `rootId`, so the streaming card falls back to a top-level message outside the originating thread.
  - confidence: 0.88
- **[P1] Keep Feishu block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: `account.config?.blockStreaming !== false` enables block streaming when the key is unset. Current Feishu behavior and docs make completed-block streaming opt-in, so this can deliver internal block chunks as fragmented messages for existing accounts.
  - confidence: 0.9
- **[P2] Throttle snapshot card replacements:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent()` sends an update immediately for each forced refresh, while the model-name animation forces refreshes every 30 ms. That bypasses the existing 100 ms card update throttle and can hit Feishu rate limits or sequence ordering failures.
  - confidence: 0.78
