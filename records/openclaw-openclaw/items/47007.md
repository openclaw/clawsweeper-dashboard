## Summary

Keep open. The PR still has source-reproducible blockers around duplicate delivery, Feishu thread routing, block-streaming defaults, update throttling, and plugin boundary rules; current main has adjacent Feishu streaming work but does not already implement this outbound subagent streaming bridge.

## What This Changes

The PR adds a channel outbound-dispatcher hook and Feishu streaming-card callbacks so delivered subagent runs can stream partial, reasoning, and tool-status output.

## Best Possible Solution

Revise this into a narrow subagent-only Feishu streaming path that preserves reply/root/thread routing, keeps block streaming opt-in, uses SDK-backed channel contracts, coalesces card updates, and adds focused Feishu plus agent-command coverage.

## Reproduction Assessment

Yes. The blockers are reproducible by source inspection from the PR diff plus current main: delivered runs still final-send, Feishu streaming cards need concrete reply/root routing, and current block streaming is explicitly opt-in.

## Solution Assessment

No. The proposed path expands a channel contract and reuses Feishu streaming without preserving main-run delivery, thread routing, block-streaming defaults, and plugin boundary constraints; a rebased subagent-only bridge is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:735-771`
  - body: The new bridge is also built for ordinary delivered main-agent turns. Those runs still go through normal outbound delivery after the attempt, so Feishu can stream a card and then send the final reply again for the same run; gate the bridge to lanes that lack the inbound dispatcher path.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces threadId to a boolean and drops the actual topic/root id. When replyToId is absent, the streaming card has no concrete reply/root target and can fall back to top-level message creation outside the originating Feishu topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: The new blockStreaming check enables block delivery whenever the setting is unset, reversing the current Feishu behavior where internal block chunks are dropped unless blockStreaming is explicitly true. Default accounts could receive fragmented internal block messages.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: replaceContent bypasses the existing streaming-card throttle, while the new status/model-name paths can call it repeatedly. This can burst Feishu CardKit updates and trigger rate or sequence failures; route replacement updates through the same coalescing throttle.
  - confidence: 0.82
- **[P2] Use an SDK-facing model callback type:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports ModelSelectedContext through a relative ../../../src/** path, crossing the bundled plugin boundary. Expose the needed type through the plugin SDK or a local plugin facade before using it from the extension.
  - confidence: 0.78
