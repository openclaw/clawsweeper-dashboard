## Summary

Keep this PR open, but it is not mergeable as-is: the feature is still plausibly useful, current main does not provide an equivalent outbound Feishu subagent streaming bridge, and the proposed patch introduces delivery, thread-routing, block-streaming, SDK-boundary, and rate-limit risks.

## What This Changes

The PR adds a ChannelPlugin outbound dispatcher hook and Feishu reply-dispatcher/card changes so agent and subagent runs can stream partial output, reasoning, and tool status through Feishu cards.

## Best Possible Solution

Land a refreshed, SDK-safe Feishu subagent streaming implementation through the existing message/live delivery direction or a narrowly documented additive SDK seam, preserving thread/root routing, block-streaming defaults, throttling, tests, and redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR-introduced regressions: the diff shows the delivered-run bridge, dropped thread id, opt-out block-streaming default, core-internal import, and unthrottled replacement updates. The requested Feishu streaming improvement itself was not live-reproduced and has no inspectable artifact.

## Solution Assessment

No. The current patch is broader than the subagent-only problem, crosses the plugin/core boundary, and weakens existing Feishu delivery contracts; a narrower SDK-safe live delivery seam is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict the bridge to subagent runs:** `src/commands/agent.ts:735`
  - body: The bridge is created for `opts.deliver === true`, but delivered main-agent runs still go through normal final delivery after the attempt. On Feishu this can create a streaming card and then send the same final reply again through standard delivery for every delivered main-agent command.
  - confidence: 0.92
- **[P1] Preserve the Feishu thread target:** `extensions/feishu/src/channel.ts:487-499`
  - body: `createOutboundDispatcher` reduces `threadId` to a boolean and never passes the actual topic/root id to the reply dispatcher. When `replyToId` is absent, the streaming session has no target thread id and can post the subagent stream as a top-level card instead of staying in the originating topic.
  - confidence: 0.9
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:425-433`
  - body: `account.config?.blockStreaming !== false` enables block streaming whenever the option is unset, and the PR also forces `disableBlockStreaming: false`. Current Feishu behavior keeps internal block chunks disabled unless `blockStreaming` is explicitly true, so default accounts could receive fragmented block messages.
  - confidence: 0.9
- **[P2] Use SDK-safe types at the plugin boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: This production plugin file imports `../../../src/auto-reply/types.js`, which crosses the extension boundary and ties Feishu to core source layout. Expose or reuse the needed type through the plugin SDK or Feishu runtime API instead.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:357-371`
  - body: `replaceContent()` queues a Cardkit update every time it is called, while the new model-name animation calls the replace path once per character at 30 ms intervals. That bypasses the existing 100 ms throttle used by normal streaming updates and can burst Feishu API calls.
  - confidence: 0.82
