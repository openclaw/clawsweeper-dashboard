## Summary

Keep open: current main does not already implement this PR's Feishu subagent streaming bridge, and the proposed patch still has source-reproducible routing, duplicate-delivery, and block-streaming regressions that should block merge.

## What This Changes

The PR adds a ChannelPlugin outbound-dispatcher hook, wires Feishu reply streaming callbacks into agent/subagent runs, and changes Feishu streaming-card, reasoning, tool-status, and block-streaming behavior.

## Best Possible Solution

Revise this into an SDK-safe, subagent-only outbound preview-streaming path that preserves Feishu reply/thread/root routing, keeps block streaming opt-in, coalesces card updates, and has focused Feishu plus agent-command coverage.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current delivery code reproduces the blockers: the bridge attaches to ordinary delivered runs, threadId is reduced to a boolean, and unset Feishu blockStreaming becomes enabled.

## Solution Assessment

No. The current PR expands a public channel contract and reuses Feishu reply streaming without preserving current delivery, thread, block-streaming, and final-send semantics; a narrower subagent-only seam is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:735-771`
  - body: The PR creates the streaming bridge for ordinary delivered main-agent turns, but those runs still final-send through normal outbound delivery. On Feishu this can stream a card and then send the final reply again for the same run, so gate the bridge to lanes that lack the normal inbound dispatcher path.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher converts threadId to replyInThread: Boolean(params.threadId) and drops the actual topic/root id. When replyToId is absent, the streaming card has no concrete thread target and falls back to top-level message creation, so streamed subagent output can leave the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: Using account.config?.blockStreaming !== false enables block delivery whenever the setting is unset, reversing the current documented Feishu behavior where non-Telegram block streaming is off unless explicitly enabled. Default accounts would start receiving internal block chunks as chat messages.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: replaceContent bypasses the existing streaming-card throttle, while the new status/model-name paths can call it repeatedly, including per-character animation. Coalesce or throttle replacement updates so Feishu CardKit is not hit with burst updates and sequence/rate failures.
  - confidence: 0.82
- **[P2] Use an SDK-facing model selection type:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports ModelSelectedContext through a relative ../../../src/** core path, which crosses the bundled plugin boundary. Expose the callback shape through the plugin SDK or a local Feishu facade instead of deep-importing core internals.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI execution surface, but it introduces a concrete Feishu routing/privacy risk by dropping the actual thread target for streamed output.

Concerns:

- **[medium] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
