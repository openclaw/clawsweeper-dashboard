## Summary

Keep open: the Feishu subagent streaming direction is useful, but the PR is not mergeable as-is because it lacks real behavior proof and source review shows duplicate main-run delivery, lost Feishu thread routing, a block-streaming default regression, unthrottled card replacements, and plugin-boundary drift.

## What This Changes

The branch adds a ChannelPlugin outbound dispatcher hook and Feishu streaming-card handlers so delivered/subagent agent output can stream with tool, reasoning, and block-streaming UI updates.

## Best Possible Solution

Rebase onto current src/agents/agent-command.ts and implement a documented subagent-only channel streaming seam that preserves Feishu thread/root IDs, keeps block streaming opt-in, uses throttled card updates, and includes focused regression tests plus real Feishu proof.

## Reproduction Assessment

Yes, at source level for the review findings. The PR diff shows the main-run bridge condition, lost thread id, unset blockStreaming semantics, and unthrottled replacement path directly against current main's delivery and Feishu streaming contracts.

## Solution Assessment

No. The direction is plausible, but the submitted patch is too broad and should be narrowed to a subagent-only, SDK-safe channel seam that preserves existing Feishu delivery defaults and thread binding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict the outbound streaming bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: `opts.deliver === true` builds a channel streaming bridge for normal delivered main-agent runs, and the command still sends final payloads through `deliverAgentCommandResult`. On Feishu this can stream a card and then send the final reply again; create this bridge only for subagent runs or make it exclusively own final delivery.
  - confidence: 0.9
- **[P1] Preserve the Feishu thread id for outbound streaming:** `extensions/feishu/src/channel.ts:499`
  - body: This converts `threadId` to a boolean and drops the actual id. In thread-only subagent contexts where `replyToId` is absent, `FeishuStreamingSession.start()` receives no root id or reply target and can stream a top-level card outside the originating thread.
  - confidence: 0.88
- **[P1] Keep Feishu block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: `account.config?.blockStreaming !== false` enables block streaming when the key is unset. Current Feishu docs/tests define completed-block streaming as off by default, so existing accounts can receive internal block chunks as fragmented messages unless this stays `=== true`.
  - confidence: 0.91
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent()` bypasses the existing `updateThrottleMs` and pending-flush path. The model/tool animation path can force replacements every 30 ms, which can burst Feishu Cardkit updates and hit rate or sequence-ordering failures.
  - confidence: 0.82
- **[P2] Keep Feishu on the plugin SDK boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: The plugin imports `ModelSelectedContext` from `../../../src/auto-reply/types.js`, but extension production code must use `openclaw/plugin-sdk/*` or local barrels. Expose the type through an SDK/runtime API instead of adding a deep core import.
  - confidence: 0.86
