## Summary

Keep open: the feature is still plausible and not superseded, but the PR branch is not mergeable as-is because the diff introduces source-level Feishu delivery regressions and the external contributor has not provided inspectable real Feishu proof.

## What This Changes

The PR adds a ChannelPlugin outbound dispatcher hook and Feishu reply-dispatcher/card updates so subagent runs can stream partial text, reasoning, and tool status in Feishu.

## Best Possible Solution

Implement a subagent-only Feishu streaming path through an SDK-safe channel lifecycle or streaming seam, preserving thread/root routing and block-streaming defaults, with focused regression tests and redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR-introduced regressions at source level: the diff shows the broad opts.deliver bridge, dropped Feishu thread id, opt-out blockStreaming check, and unthrottled replacement updates. The intended Feishu streaming improvement itself was not live-tested here.

## Solution Assessment

No. The current PR is too broad for main-agent delivery and crosses plugin/core boundaries; a safer fix should be subagent-only, preserve existing Feishu delivery contracts, and avoid new internal imports from bundled plugin production code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for `opts.deliver === true`, but delivered main-agent runs still go through normal final delivery after the attempt. On Feishu that can create a streaming card and then send the same final reply again through standard delivery for every delivered main-agent command.
  - confidence: 0.92
- **[P1] Preserve the Feishu thread target:** `extensions/feishu/src/channel.ts:499`
  - body: `createOutboundDispatcher` turns `threadId` into a boolean and drops the actual topic/root id. When `replyToId` is absent, the streaming session has no reply target or root id, so topic subagent streams can be posted as top-level cards instead of staying in the originating thread.
  - confidence: 0.9
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: `account.config?.blockStreaming !== false` enables block streaming whenever the option is unset. Current Feishu behavior keeps internal block chunks disabled unless `blockStreaming` is explicitly true, so default accounts would start receiving fragmented block messages.
  - confidence: 0.9
- **[P2] Use SDK-safe types at the plugin boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: This production plugin file imports `../../../src/auto-reply/types.js`, which crosses the extension boundary and ties Feishu to core source layout. Expose or reuse the needed type through the plugin SDK or Feishu runtime API instead.
  - confidence: 0.86
- **[P2] Keep replacement updates under the existing throttle:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent()` queues a Cardkit content update every time it is called, while the new model-name animation calls the replace path once per character at 30 ms intervals. That bypasses the existing `updateThrottleMs`/pending flush guard used by normal streaming updates.
  - confidence: 0.82
