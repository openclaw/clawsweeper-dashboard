## Summary

Keep open: this Feishu feature is not implemented on current main, and the PR is not mergeable as-is because source review shows duplicate delivery risk, lost topic routing, a block-streaming default regression, a plugin/core boundary violation, unthrottled card updates, and missing after-fix real behavior proof.

## What This Changes

The PR adds a ChannelPlugin outbound-dispatcher hook and Feishu streaming-card callbacks so subagent runs can stream text, reasoning, and tool-status updates in Feishu.

## Best Possible Solution

Rebase onto current main and implement a subagent-only Feishu streaming path that reuses the existing channel delivery/threading contracts, preserves topic routing, keeps block streaming opt-in, avoids plugin imports from core internals, and includes focused regression tests plus redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR-introduced regressions at source level: the PR head shows the duplicate bridge condition, dropped thread target, and opt-out blockStreaming check directly in changed code. The feature itself was not live-tested in Feishu during this read-only review.

## Solution Assessment

No. The proposed direction needs to be narrowed to subagent runs and routed through the existing Feishu threading/delivery contracts instead of adding a broad main-agent bridge and plugin/core boundary leak.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict the streaming bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is also created for `opts.deliver === true`, but delivered main-agent runs still go through the normal final delivery path after `runAgentAttempt`. On Feishu that can stream one card through the bridge and then send the same final payload through standard delivery for every delivered main-agent command.
  - confidence: 0.92
- **[P1] Preserve the Feishu thread target:** `extensions/feishu/src/channel.ts:499`
  - body: `createOutboundDispatcher` turns `threadId` into a boolean and drops the actual id. When `replyToId` is absent, `FeishuStreamingSession.start()` receives neither `replyToMessageId` nor `rootId`, so a subagent stream from a topic falls back to a top-level card instead of the originating thread.
  - confidence: 0.9
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: `account.config?.blockStreaming !== false` enables block streaming whenever the option is unset. Current Feishu behavior keeps block chunks internal unless `blockStreaming` is explicitly true, so default accounts would start receiving fragmented internal block messages.
  - confidence: 0.9
- **[P2] Use SDK types at the Feishu plugin boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: This production plugin file imports `../../../src/auto-reply/types.js`, crossing the extension/core boundary and tying the plugin to internal source layout. Please expose or reuse the needed model-selected type through the plugin SDK or the Feishu runtime API instead.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent()` queues a Cardkit update every time it is called, while the new model-name animation calls it once per character at 30 ms intervals. That bypasses the existing streaming update throttle and can cause Feishu rate-limit or sequence-order failures during ordinary model selection.
  - confidence: 0.82
