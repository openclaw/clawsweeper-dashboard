## Summary

Keep open; the PR is not mergeable as-is because it introduces duplicate delivery for delivered main-agent turns, drops Feishu thread routing for streamed output, and flips block-streaming defaults while current main has moved the agent-command path.

## What This Changes

The PR adds a channel-plugin outbound dispatcher hook, wires Feishu agent/subagent callbacks into streaming cards, and changes Feishu streaming-card status, reasoning, and block handling.

## Best Possible Solution

Revise this into a narrow SDK-safe subagent streaming path that attaches only where no inbound dispatcher already exists, preserves Feishu reply/thread/root routing end to end, keeps block streaming opt-in, coalesces card updates, and adds focused Feishu plus agent-command contract tests.

## Reproduction Assessment

Yes. The main blockers are reproducible by code inspection: the PR bridge is attached to delivered main-agent turns that still final-deliver, and the Feishu bridge discards the concrete thread/root id before streaming-card routing decides where to post.

## Solution Assessment

No. The proposed path is not the best fix as-is because it expands a public channel contract and reuses Feishu reply streaming without preserving delivery, thread, block-streaming, and final-send semantics; a subagent-only bridge integrated with the existing delivery route is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:735`
  - body: The new guard also builds `streamingBridge` for ordinary delivered main-agent turns. Those turns still go through normal outbound delivery, so Feishu can stream a card and then send the final reply again for the same run. Gate this to lanes that lack an inbound dispatcher, starting with the subagent lane.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces `threadId` to `replyInThread: Boolean(params.threadId)` and drops the actual topic/root id. When there is no `replyToId`, `FeishuStreamingSession.start` has no concrete thread target and falls back to top-level message creation, so streamed subagent output can leave the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: `account.config?.blockStreaming !== false` enables block delivery whenever the setting is unset, reversing the current documented behavior that non-Telegram channels require explicit `blockStreaming: true`. Default Feishu configs would start receiving fragmented internal block chunks as chat messages.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent` bypasses the existing streaming-card throttle, while the new status/model-name paths can call it repeatedly, including per-character animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK-facing type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports `ModelSelectedContext` through a relative `../../../src/**` path, crossing the bundled plugin boundary. Expose the needed callback type through the SDK or a local plugin facade instead.
  - confidence: 0.78
