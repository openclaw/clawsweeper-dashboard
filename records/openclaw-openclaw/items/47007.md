## Summary

Keep open. The Feishu subagent streaming feature has not landed on current main, and the PR branch is conflicting, lacks inspectable real Feishu proof, and still contains source-provable regressions in delivery scope, thread routing, block streaming defaults, plugin boundaries, and streaming-card update pacing.

## What This Changes

The PR adds a ChannelPlugin outbound dispatcher hook and Feishu reply-dispatcher/card changes to stream subagent output, reasoning, and tool status in real time.

## Best Possible Solution

Land a refreshed, narrow Feishu subagent streaming implementation through an SDK-safe delivery/preview seam that preserves thread/root routing, keeps block streaming opt-in, throttles card updates, and includes focused tests/docs plus redacted Feishu proof.

## Reproduction Assessment

Yes for the PR-introduced regressions by source inspection: the bridge scope, thread-id loss, block-streaming default flip, core-internal import, and unthrottled replacement updates are visible in the diff. No high-confidence real reproduction of the intended Feishu improvement is available because no inspectable after-fix proof was attached.

## Solution Assessment

No. The current patch is broader than the subagent-only problem, is stale against current main, and weakens Feishu delivery contracts; a narrower SDK-safe streaming seam with preserved routing and proof is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for `opts.deliver === true` as well as subagent runs, but delivered main-agent commands still go through normal final delivery. On Feishu this can stream a card and then send the final reply again for an ordinary delivered run.
  - confidence: 0.92
- **[P1] Preserve Feishu thread ids in the outbound dispatcher:** `extensions/feishu/src/channel.ts:499`
  - body: The dispatcher converts `threadId` to a boolean and never passes the actual topic/root id. When there is no `replyToId`, the streaming session has no routing identifier and can create the card outside the originating thread.
  - confidence: 0.9
- **[P2] Keep Feishu block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-434`
  - body: `account.config?.blockStreaming !== false` enables block streaming whenever the option is unset, and the PR also forces `disableBlockStreaming: false`. Default Feishu accounts could receive fragmented block messages that current main keeps disabled unless explicitly enabled.
  - confidence: 0.9
- **[P2] Use SDK-safe types at the plugin boundary:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: This production Feishu plugin file imports a type from `../../../src/auto-reply/types.js`, crossing the extension boundary. The type should come from the plugin SDK or a Feishu-owned runtime API before adding a plugin-facing seam.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-371`
  - body: `replaceContent()` queues a CardKit update every call, while the new model-name animation reaches it once per character at 30 ms intervals. That bypasses the existing streaming throttle and can burst Feishu API calls or sequence updates.
  - confidence: 0.82
