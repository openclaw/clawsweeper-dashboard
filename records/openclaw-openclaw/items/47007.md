## Summary

This PR should stay open but is not mergeable as-is. Current main does not already implement the proposed channel-level outbound streaming bridge, and the PR diff still introduces concrete Feishu routing, duplicate-delivery, block-streaming, rate-limit, and plugin-boundary regressions that need author or maintainer revision.

## What This Changes

The PR adds a channel plugin outbound dispatcher hook, wires Feishu outbound agent/subagent callbacks into streaming cards, and changes Feishu reply-dispatcher, streaming-card, block-streaming, and status/reasoning UI behavior.

## Best Possible Solution

Revise the feature into a narrow, SDK-safe subagent streaming path that only attaches where no inbound dispatcher already exists, preserves Feishu replyToId/threadId/rootId routing end to end, coordinates with final delivery to avoid duplicate replies, keeps block streaming backwards-compatible, coalesces card replacement updates, and adds focused Feishu plus agent-command tests and plugin-contract docs.

## Reproduction Assessment

Yes. The main blockers are reproducible by code inspection against the PR diff and current main: the PR creates a bridge for delivered main-agent turns while current main still final-delivers through `deliverOutboundPayloads`, and the Feishu bridge reduces `threadId` to a boolean before streaming-card routing chooses its target.

## Solution Assessment

No. The proposed implementation is not the best way to solve the issue as-is because it broadens a public channel contract and reuses Feishu reply streaming without preserving delivery, thread, and final-send semantics; a subagent-only bridge integrated with the existing delivery route is safer.

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:735`
  - body: The new guard also creates the streaming bridge for ordinary delivered main-agent turns. Those turns still go through final outbound delivery, so Feishu can show a streamed card and then send the final reply again for the same run. Gate this to lanes that lack an inbound dispatcher, starting with the subagent lane.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher turns `threadId` into `replyInThread: Boolean(params.threadId)` and drops the actual topic/root id. In thread-only contexts without `replyToId`, the streaming card has no concrete Feishu thread target and can be posted outside the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: `account.config?.blockStreaming !== false` enables block delivery when the setting is unset, reversing the current Feishu behavior that suppresses internal block chunks unless the streaming-card path can safely consume them. Default configs would start receiving fragmented block messages.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent` bypasses the existing streaming-card update throttle while the new status/model-name paths can call it repeatedly, including per-character animation. That can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK-facing type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports `ModelSelectedContext` through a relative `../../../src/**` path, crossing the bundled plugin boundary. Expose the needed callback type through the SDK or a local plugin facade instead.
  - confidence: 0.78
