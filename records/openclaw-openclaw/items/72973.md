## Summary

Keep open. Current main still lacks outbound filename control/bidi stripping, so the PR targets a real media-boundary hardening gap, but the proposed helper is not mergeable because its RegExp pattern throws at module import time; the PR also needs the required user-facing changelog entry.

## What This Changes

The PR adds a shared outbound filename control-character stripping helper, wires it into remote fetch and web/local media filename extraction, and adds media-focused regression tests.

## Best Possible Solution

Fix the helper so the selected Unicode strip set is represented with valid explicit ranges, keep the sanitization centralized in `src/media`, add the required changelog entry, and then have a maintainer make the final policy call on directional marks and coordination with #71517.

## Reproduction Assessment

Yes. Current main can be reproduced by feeding `fetchRemoteMedia` a mocked `Content-Disposition` filename containing percent-escaped RLO/zero-width characters; the PR-specific blocker reproduces with a direct Node `new RegExp(...)` probe using the helper's pattern source.

## Solution Assessment

No, not as currently implemented. The media-boundary approach is the right shape, but the proposed regex must be corrected before it can safely solve the issue; a safer implementation is an explicit range such as `\u0000-\u001F\u007F-\u009F...` with tests proving module import and replacement behavior.
