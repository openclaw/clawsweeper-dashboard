## Summary

Keep open. The media-boundary hardening request is still valid on current main, but this PR is not mergeable as written because the new helper's RegExp construction throws at module import; it also lacks the required changelog entry for a user-visible media fix.

## What This Changes

The PR adds a shared outbound filename helper to strip selected invisible/control Unicode characters from filenames and wires it into remote fetch and web media extraction with focused tests.

## Best Possible Solution

Fix the helper to use valid explicit C0/C1 escape ranges, add the required changelog entry, then land the centralized media-boundary sanitization after maintainers approve the Unicode strip policy and reconcile the overlap with #71517.

## Reproduction Assessment

Yes. Current main can be reproduced with a mocked `fetchRemoteMedia` response using `Content-Disposition: filename*=UTF-8''report%E2%80%AEgpj%E2%80%8B.exe`; the PR branch failure is reproducible by constructing the proposed helper RegExp, which throws before import completes.

## Solution Assessment

No, not as currently written. Centralizing the filename filter in `fetch.ts` and `web-media.ts` is the right boundary, but the regex must be corrected before the approach can work.
