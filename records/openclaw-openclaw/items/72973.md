## Summary

This PR should stay open because the central hardening is still useful, but the current patch is not mergeable: the new filename sanitizer regex throws during module initialization, so imports of the media fetch/web-media path would fail before any sanitized filename is produced. The related filename decoder work remains open and does not supersede this narrower control-character hardening.

## What This Changes

The PR adds a shared outbound filename control-character stripping helper, applies it to Content-Disposition and URL-derived media filenames, and adds focused media tests.

## Best Possible Solution

Land a repaired shared media-boundary sanitizer that compiles, strips the agreed invisible/control filename characters at the extraction points, keeps focused regression tests, and coordinates cleanly with the open filename decoder work.

## Reproduction Assessment

Yes for the PR blocker: the exact RegExp source added by this PR throws before the helper can run. The underlying main-branch issue is source-reproducible from the current basename/decode paths that pass extracted filenames through without filtering.

## Solution Assessment

No, not as currently implemented. Centralizing this at the media filename boundary is the right shape, but the regex must be corrected before the patch can be considered mergeable, and maintainers should decide whether the narrower bidi-mark policy is acceptable.
