## Summary

Keep open. Current main still lacks ACP child-session cancellation in the subagent stop cascade, but this PR is dirty against current main and needs repair before merge because it can hang reset/delete on an unbounded ACP cancel and adds an unsafe fallback from an explicit stop target to the wrapper session.

## What This Changes

The PR makes subagent stop propagation async, awaits stop/reset callers, cancels ACP-backed child sessions, persists child aborted state, and adds fast-abort regression coverage.

## Best Possible Solution

Land a rebased repair that preserves current-main subagent dedupe and explicit target semantics, cancels ACP-backed child sessions through a bounded cleanup path, persists intentional child abort state, and documents the user-visible stop/reset fix in the active changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows the subagent stop cascade never cancels ACP child sessions, and the PR diff shows reset/delete would await an unbounded child ACP cancel once that cascade becomes async.

## Solution Assessment

No, not as submitted. The direction is right, but the mergeable solution needs bounded child ACP cancellation, a rebase that preserves current target/dedupe behavior, and a changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bound ACP child cancellation during stop cascade:** `src/auto-reply/reply/abort.ts:199-203`
  - body: The new child-stop path awaits `acpManager.cancelSession` before continuing. `AcpSessionManager.cancelSession` can wait on `runtime.cancel` or an active turn cancel promise without a local timeout, and session reset now awaits this cascade, so one stuck ACP backend can hang reset/delete instead of following the bounded cleanup flow already used by gateway ACP reset.
  - confidence: 0.9
- **[P2] Do not fall back from an explicit stop target to the wrapper session:** `src/auto-reply/reply/commands-session-abort.ts:153-160`
  - body: When the stop target is distinct from `params.sessionKey`, retrying `stopSubagentsForRequester` with the wrapper session after the target stops zero subagents can terminate subagents owned by the slash/native wrapper session rather than the requested target. Keep the fallback out of explicit-target cases or prove the two keys describe the same controlled session.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/auto-reply/reply/abort.ts:139-142`
  - body: This changes user-visible `/stop` and session reset behavior for active subagents, but the PR does not update `CHANGELOG.md`. Add a single-line `Unreleased` Fixes entry before merge.
  - confidence: 0.82
