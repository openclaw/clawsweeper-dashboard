## Summary

Keep open: the reported bug is plausible from current source, but this external PR still needs real behavior proof and has blocking correctness issues before it can merge.

## What This Changes

The PR makes subagent stop propagation async, awaits it from fast-abort and gateway reset paths, cancels ACP-backed child sessions, persists child abort state, adds /stop fallback behavior, and adds regression coverage.

## Best Possible Solution

Land a rebased bounded ACP child-session cleanup path that preserves target scoping and child abort state, with focused regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main stops child queues/embedded runs and recurses but never resolves or cancels ACP child sessions, and gateway reset reaches this helper before bounded direct-target ACP cleanup. I did not run a live ACP setup in this read-only review.

## Solution Assessment

No, not as submitted. The direction is right, but direct awaited ACP child cancellation needs a bound and the /stop fallback must stay inside the resolved target boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Bound ACP child cancellation in the stop cascade:** `src/auto-reply/reply/abort.ts:199-203`
  - body: This awaits acpManager.cancelSession directly for each child. cancelSession can await the active turn's cancelPromise without a timeout, and reset/delete now awaits this cascade before the existing bounded target ACP cleanup, so a stalled ACP backend can hang reset/delete.
  - confidence: 0.91
- **[P2] Keep stop fallback scoped to the resolved target:** `src/auto-reply/reply/commands-session-abort.ts:153-157`
  - body: When the resolved target stops zero children, this retries with params.sessionKey. That can stop unrelated subagents for the command wrapper after the user targeted a distinct session, conflicting with current target-boundary coverage.
  - confidence: 0.86
