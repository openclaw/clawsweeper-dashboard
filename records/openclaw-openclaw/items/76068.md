## Summary

Keep open. The latest PR head appears to have addressed the earlier async Dir double-close finding and CI is green, but the branch is dirty against current main and the claimed plugin tool cache-key fix is still only a comment-level change that leaves the hashing path in place.

## What This Changes

The PR routes gateway agent startup through an async workspace skill snapshot builder, adds skills snapshot hydration caching, updates plugin tool-factory cache-key comments, adjusts mocks/exports, and adds a changelog entry for the gateway performance regression.

## Best Possible Solution

Rebase the branch onto current main, keep the async skill startup direction only with parity coverage, and either implement a real non-hashing cache-key path with invalidation proof or narrow the changelog/body to the behavior actually changed.

## Reproduction Assessment

Yes for the review finding: a non-runtime config passed through the PR head still reaches resolveRuntimeConfigCacheKey, whose current-main implementation hashes that object. The original Windows freeze has strong issue-log evidence but was not live-reproduced in this read-only review.

## Solution Assessment

No. Async startup scanning may be a useful part of the fix, but the PR currently overstates the tool-cache change and needs rebase plus cache-key or changelog alignment before it is the narrowest maintainable solution.
