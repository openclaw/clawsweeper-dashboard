## Summary

Keep open: this is useful, well-scoped cron/agent observability work with real terminal proof, but the new drain-continuation path still misses the first embedded attempt when Gateway drain begins after the outer run is admitted and before the first session/global enqueue.

## What This Changes

The PR adds cron fallback progress reporting, queue-pressure observation fields, task-progress sanitization, Gateway-drain continuation plumbing for fallback attempts, focused tests/proof script, and related docs.

## Best Possible Solution

Land after the continuation flag covers the first already-admitted attempt as well as later fallback and live-switch attempts, while ordinary new queue enqueues still reject during Gateway drain and cron task progress remains sanitized.

## Reproduction Assessment

Yes, from source inspection: if `markGatewayDraining()` happens before the first `runEmbeddedAgent` enqueue, `enqueueCommandInLane` rejects because `runWithModelFallback` does not pass the continuation option for `i === 0`. I did not run a local repro because this review is read-only.

## Solution Assessment

No; the PR is close, but the new guard is too narrow for the admitted-run boundary it is trying to protect. The best fix is to pass continuation for the first already-admitted attempt, or deliberately abort on the drain error, while preserving fail-closed behavior for unrelated new work.
