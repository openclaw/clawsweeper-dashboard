## Summary

Keep open: the latest head fixes the earlier watermark, pagination, docs, and changelog blockers, but two Matrix read-path gaps still break the promised thread isolation.

## What This Changes

The PR scopes Matrix room history and message reads by thread, changes Matrix thread send/edit relation metadata, forwards thread IDs in lower-level message actions, updates Matrix docs/changelog, and adds focused tests.

## Best Possible Solution

Land a Matrix-only fix after forwarding threadId through the exposed read action and making main-room filtering recognize threaded edit events, with focused tests and the existing docs/changelog kept aligned.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: thread sends always include fallback metadata, threaded edits write reply context on the outer replace relation, room-history calls do not pass thread scope, and Matrix reads have no threadId path.

## Solution Assessment

No. The PR direction is narrow and appropriate, but the current patch is not yet the best implementation because the exposed read action still drops threadId and main-room filtering misses threaded edit events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Forward threadId through the exposed read action:** `extensions/matrix/src/actions.ts:232-238`
  - body: The plugin exposes the Matrix message read surface as action `read`, but this dispatch builds `readMessages` without passing `params.threadId`. A caller can provide `threadId` and still hit the main-room path, so the new thread relations endpoint is unreachable from the normal Matrix message tool.
  - confidence: 0.9
- **[P2] Filter threaded edit events from main reads:** `extensions/matrix/src/matrix/actions/messages.ts:18-25`
  - body: This only recognizes top-level `m.relates_to.rel_type === "m.thread"`. The same patch emits threaded edits with the thread relation under `m.new_content["m.relates_to"]` and an outer `m.replace`, so edited thread messages still leak into main-room reads.
  - confidence: 0.84
