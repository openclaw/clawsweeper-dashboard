## Summary

Keep this PR open. Current main still has the Matrix thread-history and reply-placement problems the PR targets, and the branch is scoped to the Matrix plugin, but the proposed patch still has blocking pagination and thread-watermark correctness bugs plus required docs/changelog follow-up before merge.

## What This Changes

The PR changes Matrix room-history tracking, message reads, send/edit relation metadata, tool-action threadId forwarding, and focused Matrix tests to isolate thread history and keep thread replies out of the main room timeline.

## Best Possible Solution

Land a Matrix-only implementation that preserves thread scope through room history, reads, sends, and edits while keeping pagination cursors aligned with returned events, rejecting stale thread watermark writes after subqueue eviction, clearing legacy room watermarks, and updating Matrix docs plus the changelog.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: Matrix thread sends always include fallback reply metadata, threaded edits write reply context on the outer replace relation, room-history APIs lack thread scope, and read actions have no thread endpoint path.

## Solution Assessment

No, not yet. The PR direction is narrow and maintainable, but the current patch needs pagination cursor handling, thread-watermark liveness, legacy watermark cleanup, docs, and changelog fixes before it is the safest solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve unread events when over-fetching main timeline:** `extensions/matrix/src/matrix/actions/messages.ts:188`
  - body: When main-room reads filter thread events, this path can fetch extra pages, collect more non-thread events than `limit`, then drop the surplus while returning the cursor from the last fetched page. Callers paginating with that token skip readable main-room history, so keep the cursor aligned to the returned events or stop fetching past the returned window.
  - confidence: 0.88
- **[P2] Reject stale consumes for evicted thread queues:** `extensions/matrix/src/matrix/monitor/room-history.ts:410`
  - body: `consumeHistory` validates the room generation but not the thread subqueue snapshot. If a thread subqueue is evicted and recreated while an older trigger is in flight, the late consume can advance the new thread watermark past fresh messages and hide them from the agent.
  - confidence: 0.86
- **[P3] Clear legacy room watermarks during room eviction:** `extensions/matrix/src/matrix/monitor/room-history.ts:133`
  - body: `clearRoomWatermarks` only recognizes the new JSON watermark keys, while `computePendingHistory` still reads legacy `agentId:roomId` keys. After upgrade, evicted rooms can leave legacy entries in the capped watermark map and reduce capacity for active rooms.
  - confidence: 0.8
- **[P3] Update the Matrix history docs for thread scope:** `extensions/matrix/src/matrix/monitor/room-history.ts:46`
  - body: This changes Matrix room history from room-only to thread-scoped, but `docs/channels/matrix.md` still states that Matrix room history is room-only. Update the docs so operators understand main-room versus thread history behavior after the change.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `extensions/matrix/src/matrix/send/formatting.ts:144`
  - body: This PR changes user-visible Matrix thread reply placement and context behavior, but it does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing feat/fix changes to add a single-line Unreleased entry with contributor credit.
  - confidence: 0.78
