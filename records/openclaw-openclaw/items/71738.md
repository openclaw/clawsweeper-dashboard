## Summary

Keep open: the Matrix thread-isolation direction is useful and the contributor supplied live proof, but PR head still drops pagination tokens for threaded reads by treating the Matrix relations response as `/messages`-style `start`/`end`.

## What This Changes

The PR updates the Matrix plugin, docs, tests, and changelog so native Matrix threads scope room history and reads, thread sends/edits avoid fallback reply metadata by default, and action handlers pass thread IDs.

## Best Possible Solution

Keep the Matrix-plugin-scoped thread isolation, but map relation `next_batch`/`prev_batch` into OpenClaw pagination fields and add focused regression coverage before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main has room-flat Matrix history and unconditional fallback thread relations, while the PR head pagination defect follows directly from the Matrix relations response contract. No live current-main failure run was performed in this read-only review.

## Solution Assessment

No as-is: the architecture is mostly the right plugin-scoped fix, but threaded reads must preserve Matrix relation pagination tokens before merge. The narrow repair is to parse `next_batch`/`prev_batch` while preserving the new hydration behavior.
