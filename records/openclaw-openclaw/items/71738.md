## Summary

Keep open: the Matrix thread isolation direction is well-scoped and the contributor supplied current-head live Matrix proof, but the patch still breaks thread-read pagination by treating the relations endpoint response as `start`/`end` instead of `next_batch`/`prev_batch`.

## What This Changes

The PR updates the Matrix plugin so native thread roots scope room history and read actions, thread sends/edits avoid fallback reply metadata by default, and related tests, docs, and changelog are updated.

## Best Possible Solution

Map Matrix relation `next_batch`/`prev_batch` into OpenClaw `nextBatch`/`prevBatch`, keep the Matrix-plugin-scoped thread behavior, and land after focused Matrix tests cover the token shape.

## Reproduction Assessment

Yes for source-level reproduction: current main has flat room queues and unconditional fallback thread relations, and the PR pagination regression is visible by comparing the PR code with the Matrix relations response contract. No live current-main failure run was performed in this read-only review.

## Solution Assessment

No as-is: the architecture is mostly the right Matrix-plugin-scoped fix, but thread reads must preserve the relations endpoint pagination tokens before merge. The narrow repair is to parse `next_batch`/`prev_batch` or reuse the existing `MatrixClient.getRelations` token shape while preserving hydration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Preserve relation pagination tokens:** `extensions/matrix/src/matrix/actions/messages.ts:216-217`
