## Summary

Keep open: current main still has the Matrix flat-history and fallback-relation problem, and this PR is the right Matrix-owned direction, but the latest thread-read endpoint filters out encrypted and poll thread events before hydration, so it should not merge as-is.

## What This Changes

The PR scopes Matrix room history, watermarks, message reads, send/edit relation payloads, tests, docs, and changelog around native Matrix thread roots.

## Best Possible Solution

Land the Matrix thread isolation direction after thread reads preserve all relation event types through hydration and focused Matrix tests/proof are green.

## Reproduction Assessment

Yes. Source inspection shows current main keeps Matrix history room-flat and emits fallback reply metadata; the PR finding is also source-reproducible because the thread-read URL filters server-side to m.room.message before hydration can recover encrypted replies or poll events.

## Solution Assessment

No. The overall PR direction is maintainable, but the thread-read endpoint should omit the event-type segment, hydrate all m.thread relation events, and then apply OpenClaw's local message/poll filtering.
