## Summary

Keep open: the Matrix-thread direction is useful and the earlier pagination-token concern appears repaired, but source review found remaining Matrix thread-membership contract gaps in the read path.

## What This Changes

The PR scopes Matrix room history and read actions by native thread, removes default fallback reply metadata from thread sends/edits, and updates Matrix docs, tests, and changelog.

## Best Possible Solution

Update Matrix message reads to follow the Matrix thread-membership model for both main-room filtering and thread reads, then land the otherwise useful Matrix-plugin-scoped thread isolation work with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: PR head can be compared directly with the Matrix spec, which treats indirect child relations of thread events as threaded while this code only handles direct m.thread relations. No live Matrix run was performed in this read-only review.

## Solution Assessment

No as-is: the room-history scoping, fallback-metadata removal, and pagination-token repair are directionally right, but the read implementation should use the Matrix thread membership contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter indirect thread relations from main reads:** `extensions/matrix/src/matrix/actions/messages.ts:39-47`
  - body: `isThreadEvent` only checks the event's own `m.relates_to` or `m.new_content` for `rel_type: "m.thread"`. Matrix also treats events whose parent is in a thread as threaded, so poll votes, reactions, and ordinary replacements under a thread message can still pass through main-room reads and pollute the main timeline history.
  - confidence: 0.9
- **[P2] Read threads without dropping child relations:** `extensions/matrix/src/matrix/actions/messages.ts:145`
  - body: The thread read path calls `/relations/{threadRootId}/m.thread`, but the Matrix relationships spec warns that filtering by `rel_type=m.thread` is not appropriate for fetching all events in a thread because relations to threaded events are filtered out. Use an unfiltered/recurse relation query with client-side thread filtering, or another bounded approach that preserves indirect thread events.
  - confidence: 0.88
