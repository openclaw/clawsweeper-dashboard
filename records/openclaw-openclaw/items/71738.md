## Summary

Keep this PR open. Current main still has the Matrix thread-history and reply-placement problems the PR targets, but the proposed patch is not merge-ready because the new main-room over-fetch path can skip readable events and the thread-history watermark logic can accept stale thread-queue snapshots after subqueue eviction.

## What This Changes

The PR scopes Matrix room history and read actions by thread, adjusts Matrix thread send/edit relation metadata, forwards `threadId` through Matrix actions, and adds focused Matrix tests.

## Best Possible Solution

Land a Matrix-only fix that preserves thread scope through history, reads, sends, and edits while keeping pagination cursors aligned with returned events, rejecting stale thread-subqueue watermark writes, cleaning up legacy watermarks, and updating Matrix docs plus changelog.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: thread sends always include fallback reply metadata, threaded edits write reply context on the outer replace relation, room-history calls do not pass thread scope, and message reads have no thread-specific path.

## Solution Assessment

No. The PR direction is the right narrow area, but this patch is not yet the best implementation because it can lose main-room pagination results and accept stale thread watermark writes; fixing those defects is the safer path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve unread events when over-fetching main timeline:** `extensions/matrix/src/matrix/actions/messages.ts:183`
  - body: When main-room reads filter thread events, this path can fetch extra pages, collect more non-thread events than `limit`, then drop the surplus while returning `nextBatch` from the last fetched page. Callers paginating with that token skip readable main-room history, so keep the cursor aligned to the returned events or stop fetching past the returned window.
  - confidence: 0.88
- **[P2] Reject stale consumes for evicted thread queues:** `extensions/matrix/src/matrix/monitor/room-history.ts:411`
  - body: `consumeHistory` validates the room generation but not the thread subqueue snapshot. If a thread subqueue is evicted and recreated while an older trigger is in flight, the late consume can advance the new thread watermark past fresh messages and hide them from the agent.
  - confidence: 0.86
- **[P3] Clear legacy room watermarks during room eviction:** `extensions/matrix/src/matrix/monitor/room-history.ts:135`
  - body: `clearRoomWatermarks` only recognizes the new JSON watermark keys, while `computePendingHistory` still reads legacy `agentId:roomId` keys. After upgrade, evicted rooms can leave legacy entries in the capped watermark map and reduce capacity for active rooms.
  - confidence: 0.8
- **[P3] Update Matrix history docs for thread scope:** `extensions/matrix/src/matrix/monitor/room-history.ts:46`
  - body: This changes Matrix room history from room-only to thread-scoped, but `docs/channels/matrix.md` still says Matrix room history is room-only. Update the docs so operators understand main-room versus thread history behavior after the change.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `extensions/matrix/src/matrix/send/formatting.ts:144`
  - body: This PR changes user-visible Matrix thread reply placement and context behavior, but it does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing feat/fix changes to add a single-line active-version entry.
  - confidence: 0.78
