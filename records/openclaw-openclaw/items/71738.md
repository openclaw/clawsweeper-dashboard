## Summary

Keep open: the Matrix-thread direction is useful and mostly well-scoped, but the PR still drops Matrix relations pagination tokens in the new thread-read path, which blocks merge for multi-page thread history.

## What This Changes

The PR changes the Matrix plugin to scope room history and read actions by Matrix thread roots, avoid default fallback reply metadata for thread sends/edits, and adds Matrix tests, docs, and changelog coverage.

## Best Possible Solution

Keep the Matrix-plugin-scoped thread-isolation design, map relations next_batch/prev_batch into the action result, and land with focused pagination regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main is room-flat for Matrix history and always emits fallback thread relations, while the PR-head pagination regression is visible from the relations endpoint response shape. No live current-main run was performed in this read-only review.

## Solution Assessment

No as-is: the architecture is the right plugin-local direction, but the thread read path must preserve Matrix relations pagination tokens before merge. The narrow repair is to parse next_batch/prev_batch or reuse MatrixClient.getRelations while keeping the current hydration and poll handling.
