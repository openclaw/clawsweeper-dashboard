## Summary

Keep this PR open: it fixes a real Matrix thread-isolation problem, but the current head still drops pagination tokens for threaded reads by reading Matrix relation responses as message-history responses.

## What This Changes

The PR scopes Matrix room history and read actions by native thread, removes default fallback reply metadata from thread sends/edits, and updates Matrix docs, changelog, and focused tests.

## Best Possible Solution

Keep the Matrix-plugin-scoped thread design, but map relation next_batch/prev_batch into OpenClaw nextBatch/prevBatch and add focused coverage for the token shape before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main has flat room queues and unconditional fallback thread relations, and the PR-head pagination regression is visible by comparing its source with the Matrix relations response contract. No live current-main failure run was performed in this read-only review.

## Solution Assessment

No as-is: the architecture is mostly the right Matrix-plugin-local fix, but threaded reads must preserve relation pagination tokens before merge. The narrow repair is to parse next_batch/prev_batch or reuse the existing MatrixClient.getRelations token shape while preserving hydration behavior.
