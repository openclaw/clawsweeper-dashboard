## Summary

Keep open: the Matrix thread-isolation direction is useful and the supplied live proof covers the main thread-read and reply-placement behavior, but the current PR still has a spec-backed thread-read bug that would drop indirect thread relations such as poll responses.

## What This Changes

The PR scopes Matrix room history and message reads by native Matrix thread, removes default fallback reply metadata for thread sends, adjusts threaded edits, and updates Matrix tests/docs/changelog.

## Best Possible Solution

Keep the Matrix thread-isolation approach, but fetch thread history with a spec-compatible unfiltered recursive relations traversal or equivalent collection so direct thread messages and indirect poll/edit relation events are available before summarization.

## Reproduction Assessment

Yes for the blocking review finding: PR head builds a `m.thread`-filtered relations request while the new poll response fixture is an `m.reference` relation, and the Matrix spec says that filter is not appropriate for fetching all thread events. I did not run live Matrix locally during this read-only review.

## Solution Assessment

No. The broader fix direction is maintainable, but the thread-read endpoint needs to omit the `rel_type` filter and collect indirect relations, or use an equivalent server/client traversal, before this is the right implementation.

## Review Findings
