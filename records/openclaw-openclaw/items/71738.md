## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Keep the first page from skipping a thread event:** `extensions/matrix/src/matrix/actions/messages.ts:291`
  - body: On the first thread read, this requests `limit` child events from `/relations`, prepends the root event, later slices the combined list back to `limit`, and still returns the server's `next_batch`; Matrix relations pages define `limit` as the maximum chunk size with pagination tokens for that page. When the server returns a full page, one child event is dropped locally while the returned token points past it, so the caller can never read that event. Fetch one fewer child when prepending the root, or append the root outside the paginated window without advancing past an undisplayed child. ([spec.matrix.org](https://spec.matrix.org/latest/client-server-api/index.html?utm_source=openai))
  - confidence: 0.88
- **[P1] Preserve Matrix thread fallback compatibility by default:** `extensions/matrix/src/matrix/send/formatting.ts:151-153`
  - body: Normal bot replies in threads pass `threadId` and suppress `replyToId`, so after this change they no longer include `is_falling_back`/`m.in_reply_to` at all. Current main emitted that fallback metadata, and the Matrix spec recommends it for unthreaded clients; removing it by default can make existing clients or workflows stop seeing bot replies in the main timeline. Please preserve the current fallback behavior by default, add an explicit strict thread-only mode, or get a maintainer decision documenting the compatibility break. ([spec.matrix.org](https://spec.matrix.org/latest/client-server-api/index.html))
  - confidence: 0.82
