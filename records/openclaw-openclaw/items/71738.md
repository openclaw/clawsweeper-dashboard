## Summary

Keep open. The PR addresses a real Matrix thread context/reply-placement gap with Matrix-owned code, tests, CI, and live proof, but it has one concrete thread-read regression around Matrix poll events that should be fixed before merge.

## What This Changes

This PR scopes Matrix room history, read actions, send relations, threaded edits, docs, tests, and changelog by native Matrix thread.

## Best Possible Solution

Keep the Matrix-owned thread isolation approach, but fix thread reads to preserve existing poll-summary behavior before normal owner review and merge handling.

## Reproduction Assessment

Yes, source-level. Current main leaves Matrix room-history calls room-flat and emits unconditional thread fallback metadata; the PR's new thread-read regression is also source-reproducible because Matrix poll event types are not m.room.message while the new thread endpoint filters to m.room.message.

## Solution Assessment

No as submitted. The overall direction is narrow and Matrix-owned, but the thread read endpoint should fetch all m.thread relation event types, or otherwise include Matrix poll events, so the existing readMessages poll-summary behavior is not lost in threads.

## Review Findings
