## Summary

Keep open: current main covers the doctor repair pieces for missing configured plugins and legacy Codex model refs, but the reported startup-loop path still hard-fails when an installable web_search provider is configured but inactive.

## What This Changes

The issue asks OpenClaw to recover safely after update when `tools.web.search.provider` points at an unavailable Brave plugin, and to have doctor fully repair legacy `openai-codex/*` defaults.

## Best Possible Solution

Keep the doctor/install repairs and add a deliberate supervisor-safe startup policy for unavailable optional providers that avoids silent provider fallback and is covered by config/startup tests.

## Reproduction Assessment

Yes, by source inspection: a config with `tools.web.search.provider: "brave"` and no active Brave plugin fails validation, and gateway startup asserts config validity before runtime auto-enable. I did not run a live LaunchAgent loop in this read-only review.

## Solution Assessment

Unclear: the current doctor repairs are a good recovery path, but the startup-loop fix should wait for a maintainer decision on fail-closed versus degraded startup for explicitly selected providers.
