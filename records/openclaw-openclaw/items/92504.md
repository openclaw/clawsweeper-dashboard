## Summary

Current main and v2026.6.6 still accept explicit well-formed agent selectors without checking the configured roster, then construct agent-scoped request and session context from that value. This is a shipped gateway routing bug, but the issue must remain open because two open PRs use closing syntax and maintainers need to select one implementation and confirm the canonical explicit-agent validity boundary.

## What This Changes

Reject explicit unknown gateway agent selectors instead of silently executing them with default agent configuration.

## Best Possible Solution

Select one canonical PR, validate only explicit header/model selectors through the owner-approved configured-agent validity seam across chat completions, responses, and embeddings, preserve omitted/default routing, and prove configured secondary-agent success plus unknown-agent rejection.

## Reproduction Assessment

Yes. Current main and v2026.6.6 provide a direct source-level path from an explicit unknown selector to agent-scoped request and session context without roster validation; endpoint tests also encode successful non-default selector routing.

## Solution Assessment

Partly. Shared explicit-selector validation is the narrowest maintainable direction, but a new fallback config option is not presently justified and maintainers must confirm whether `agents.list` or another existing agent-scope contract defines valid explicit addressing.
