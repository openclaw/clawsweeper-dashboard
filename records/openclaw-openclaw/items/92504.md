## Summary

Current main still accepts explicit unknown agent selectors, and two mergeable open PRs specifically claim this fix. Keep the issue open until maintainers select or consolidate one implementation and merge it.

## What This Changes

Reject well-formed explicit agent selectors that are absent from the configured roster instead of allowing agent-scoped execution to continue.

## Best Possible Solution

Land one shared request-context fix that distinguishes explicit from implicit selection, validates explicit IDs against the canonical configured roster, returns a consistent OpenAI-compatible 4xx across all three endpoint families, and preserves omitted/default routing.

## Reproduction Assessment

Yes. On current main, a syntactically valid unknown ID supplied through `x-openclaw-agent-id` or an OpenClaw model selector is returned by the shared resolver without any configured-roster membership check.

## Solution Assessment

Yes, with a narrow boundary: validate only explicit selectors in the shared request-context path and preserve implicit default routing. A new fallback config flag is not warranted unless maintainers identify an intentional shipped contract for unknown-ID fallback.
