## Label Justifications

- `P1`: A production integration can receive a successful response generated under unintended default configuration for an explicit nonexistent agent target.

## Summary

This is a real, shipped gateway routing bug: explicit unknown agent IDs are accepted and passed into runtime context instead of producing a client error. The fix should stay shared across all OpenAI-compatible endpoint callers and preserve only intentional default aliases.

## What This Changes

Validate explicit agent targets from OpenAI-compatible model aliases and agent headers against the configured roster, returning a structured 4xx for unknown IDs.

## Best Possible Solution

Return a typed shared target-resolution result that validates only explicit IDs with canonical roster semantics, then map unknown targets to the standard structured 4xx across Chat Completions, Responses, and Embeddings while retaining documented default aliases.

## Reproduction Assessment

Yes at source level. Configure a roster without a valid slug such as `missing`, send it through `model: openclaw/missing` or `x-openclaw-agent-id: missing`, and current source shows the ID reaches request context without a roster lookup.

## Solution Assessment

Yes, with one adjustment: validation belongs in the shared resolution contract used by all affected endpoints, not only the chat-completions handler. An opt-in fallback flag should not be added unless maintainers identify a shipped contract requiring explicit unknown IDs to fall back.
