## Summary

Current main and v2026.6.6 accept syntactically valid explicit agent selectors without checking the configured roster, then use the unchecked ID in agent-scoped request/session context across the OpenAI-compatible endpoints. This remains a shipped routing bug, but two open closing PRs now own the repair; https://github.com/openclaw/openclaw/pull/92510 is the stronger canonical candidate because it is cleanly mergeable and includes live configured-gateway proof across unknown, configured, default, and omitted selectors.

## What This Changes

Reject explicit unknown agent selectors across OpenAI-compatible gateway endpoints instead of continuing with unconfigured or default-backed agent behavior.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92510 as the canonical shared-resolver fix, then close https://github.com/openclaw/openclaw/pull/92505 as superseded and let the merged PR close this issue.

## Reproduction Assessment

Yes. Current and released source provide a high-confidence reproduction path: a syntactically valid unknown header or model selector bypasses roster validation and reaches agent-scoped request context; the canonical linked PR also supplies a live configured-gateway selector matrix.

## Solution Assessment

Yes. Shared validation of explicit selectors is the narrowest maintainable solution, and https://github.com/openclaw/openclaw/pull/92510 best demonstrates it across chat, responses, and embeddings while preserving omitted, default, and configured-agent routing.
