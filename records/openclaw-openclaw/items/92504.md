## Summary

Current main still accepts explicit well-formed agent IDs without checking the configured roster, then uses that ID for OpenAI-compatible request and session routing. This is a valid shipped gateway bug, but the issue must remain open because two unmerged PRs currently claim it; https://github.com/openclaw/openclaw/pull/92510 has the stronger live configured-gateway proof and broader endpoint coverage, so maintainers should select it or an equivalent canonical patch and close the competing PR.

## What This Changes

Reject explicit unknown agent selectors across OpenAI-compatible gateway endpoints while preserving omitted, default-alias, and configured-agent routing.

## Best Possible Solution

Land one canonical shared-boundary fix that distinguishes explicit selectors from implicit/default routing, validates explicit IDs against `listAgentIds`, returns a consistent OpenAI-compatible 400 error across chat completions, Responses, and Embeddings, and proves configured secondary-agent plus default/omitted success paths.

## Reproduction Assessment

Yes at source level: current main has a direct, high-confidence path from an explicit normalized selector through `resolveAgentIdForRequest` into agent-scoped request/session context without roster validation. This read-only review did not launch a gateway, but the issue and PR 92510 also provide concrete live HTTP reproduction evidence.

## Solution Assessment

Yes in direction: shared explicit-selector validation is the narrowest maintainable fix, and no new fallback config is needed because omitted and `openclaw/default` routing can remain unchanged. The strongest implementation should cover every direct/shared caller and preserve configured secondary-agent tests.
