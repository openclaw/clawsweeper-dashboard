## Label Justifications

- `P1`: A real integration can receive a successful response from an unintended persona when it routes to a nonexistent agent.
- `impact:session-state`: The unvalidated target is embedded in the session key and can create state under a phantom agent scope.
- `impact:other`: The gateway reports success for an invalid routing target, making integration failures indistinguishable from valid agent execution.

## Summary

Current main and v2026.6.6 accept well-formed explicit agent selectors without checking the configured roster, and both OpenAI-compatible endpoints consume that shared result. The earlier matching implementation closed unmerged, so this remains a current, shipped gateway routing bug with a bounded repair path.

## What This Changes

Reject explicitly requested but unconfigured agent IDs on the OpenAI-compatible chat and responses endpoints instead of executing them with default agent settings.

## Best Possible Solution

Validate explicit non-default agent selections against the configured roster in the shared request-context path, preserve default routing when no explicit target is supplied, and return the existing OpenAI-compatible invalid-request response without adding a fallback configuration option.

## Reproduction Assessment

Yes from current and released source: well-formed explicit header and model selections bypass listAgentIds and flow into both HTTP handlers. Focused endpoint tests can verify the current successful response and the corrected 4xx behavior.

## Solution Assessment

Yes, shared roster validation is the narrowest maintainable solution, but the suggested fallback config flag is unnecessary; omitted selection should keep existing default behavior while explicit unknown selection fails.
