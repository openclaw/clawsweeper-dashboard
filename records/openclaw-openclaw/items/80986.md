## Summary

Keep open. The PR targets a credible Telegram startup fanout bug, but it introduces a queued-abort cancellation issue and the supplied proof is only controlled-runtime Vitest output, while Telegram startup/transport changes need real Telegram proof before merge.

## What This Changes

The PR adds a Telegram startup `getMe` probe limiter capped at two concurrent probes, regression tests for multi-account fanout and queued aborts, and Telegram docs/changelog updates.

## Best Possible Solution

Keep the fix scoped to Telegram startup probing, make queued/manual abort resolve as cancellation, require redacted live multi-account Telegram startup proof, and leave https://github.com/openclaw/openclaw/issues/80695 open for the remaining transport/outbound cascade pieces.

## Reproduction Assessment

No for a high-confidence live reproduction in this review. Source inspection confirms current main can fan out Telegram `getMe` probes beyond the intended slot lifetime, and the linked issue provides concrete v2026.5.7 Windows logs, but I did not run a live Telegram multi-account gateway.

## Solution Assessment

Not yet. A Telegram-local probe limiter is a plausible narrow fix, but queued aborts should complete as cancellation instead of surfacing as a channel error, and the PR still needs real Telegram proof before merge.
