## Summary

Keep open: current main and latest release v2026.6.10 still skip Control UI browser device identity when `crypto.subtle` is unavailable, so the central HTTP LAN/reverse-proxy request remains unresolved. The bypass-scope and implied-scope subissues are now implemented, but the remaining change needs maintainer security/product direction because it would move persistent browser device identity into plaintext remote HTTP contexts.

## What This Changes

The issue requests browser Control UI device identity generation without depending on Secure Context/WebCrypto, plus related fixes for device-auth bypass scope preservation and implied operator scopes.

## Best Possible Solution

Keep the issue open until maintainers decide the plaintext browser identity policy; if approved, land a narrow UI identity/hash change with docs, dependency review, focused auth tests, and real LAN/reverse-proxy browser proof.

## Reproduction Assessment

Yes. Source inspection on current `main` and v2026.6.10 shows `ui/src/ui/gateway.ts` skips browser device identity when `crypto.subtle` is absent, and `ui/src/ui/device-identity.ts` still uses `crypto.subtle.digest` for fingerprinting.

## Solution Assessment

Unclear. The proposed pure-JS hash and always-attempt-identity implementation is mechanically plausible, but it is only the best solution if maintainers intentionally support persistent Control UI device identity over plaintext browser contexts.
