## Summary

Current main still cannot create the required Control UI device identity on ordinary non-localhost HTTP origins because fingerprinting uses WebCrypto. The separate device-auth-bypass scope regression is already covered by current policy code, so the remaining issue is whether OpenClaw should intentionally extend device authentication to plaintext HTTP—a security and product decision that still needs maintainer review.

Likely related people: Vincent Koc (high confidence, introduced and recently maintained the gateway scope policy) and elve.xu (medium confidence, authored the focused prior implementation for insecure-context identity).

## What This Changes

Support Control UI device identity on non-localhost HTTP deployments while preserving authorized scopes when the explicit device-auth bypass is enabled.

## Best Possible Solution

Choose and document one security model: either retain HTTPS/localhost as the device-identity boundary with clearer diagnostics, or explicitly support plaintext-LAN identity using the pure-JS cryptography path plus focused pairing, interception-risk, upgrade, and browser behavior coverage.

## Reproduction Assessment

Yes at source level: current main calls `crypto.subtle.digest` while loading or creating the required browser identity, and the gateway rejects a remote Control UI connection without that identity unless the explicit break-glass bypass is enabled. This read-only review did not rerun the reporter's live reverse-proxy setup.

## Solution Assessment

Unclear. Replacing WebCrypto SHA-256 with the existing pure-JS dependency is the narrow technical implementation, but it is only the best solution if maintainers intentionally support persistent device authentication over plaintext HTTP; otherwise better diagnostics and secure transport remain the safer path.
