## Summary

Keep open. Current main still skips browser Control UI device identity outside Secure Context/WebCrypto and the gateway still documents and tests remote plain-HTTP Control UI rejection without device identity. The bypass-scope and implied operator-scope parts are already implemented, but the remaining LAN HTTP identity request is a security-sensitive product boundary rather than conservative cleanup.

## What This Changes

The issue requests decoupling browser Control UI device identity from Secure Context/WebCrypto so LAN HTTP reverse-proxy deployments can generate device identity without using the device-auth bypass.

## Best Possible Solution

Maintainers should make an explicit threat-model decision for remote plain-HTTP Control UI. If supported, add a browser-safe SHA-256 path with the correct UI dependency, narrow or remove the Secure Context identity guard, update gateway/UI tests and public security docs, and add a user-facing changelog entry; if not supported, keep the current boundary and resolve the remaining UX through the active messaging/warning items.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: simulate an insecure browser context so the UI skips loadOrCreateDeviceIdentity(), then connect as remote Control UI without device identity and hit the reject-control-ui-insecure-auth policy path. I did not run tests because this review was required to stay read-only.

## Solution Assessment

Unclear. Replacing the WebCrypto digest with a pure-JS hash is plausible for the client-side coupling, but always attempting device identity over remote non-secure HTTP changes the documented security posture and needs maintainer approval before it is the best fix.
