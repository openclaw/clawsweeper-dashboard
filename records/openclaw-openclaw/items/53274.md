## Summary

Keep open. Current main and v2026.5.18 still skip browser device identity when `crypto.subtle` is unavailable and still reject remote device-less Control UI sessions; the bypass-scope and implied-scope subparts are already implemented, leaving a real security/product policy decision rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to generate browser device identity without depending on Secure Context/WebCrypto so HTTP LAN reverse-proxy Control UI deployments can authenticate without disabling device auth.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if support is approved, add a browser-safe SHA-256 path, attempt device identity with a safe fallback, and update gateway policy, tests, docs, and UI dependency ownership together.

## Reproduction Assessment

Yes, source-backed. In an insecure browser context the UI skips device identity, and a remote device-less Control UI connect reaches `reject-control-ui-insecure-auth`; I did not run a live browser or Docker repro in this read-only review.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but allowing device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
