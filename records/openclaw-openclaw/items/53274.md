## Summary

Keep open. Current main still preserves the insecure-browser-context skip and remote device-less Control UI rejection, while two narrower scope-related subparts are already fixed; the remaining request is a security/product decision, not stale cleanup.

## What This Changes

The issue asks OpenClaw to support Control UI device identity in non-secure browser contexts and fix related device-auth scope handling for HTTP LAN/reverse-proxy deployments.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if support is approved, update the browser hash path, connection policy, tests, docs, and dependency ownership together.

## Reproduction Assessment

Yes, source-backed. In an insecure browser context the UI skips device identity because crypto.subtle is absent, and a remote device-less Control UI connect reaches the documented reject-control-ui-insecure-auth path.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but allowing or accepting device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
