## Summary

Keeping this open: current main and the latest release still skip Control UI browser device identity on non-secure origins, while the scope-bypass subproblem is now partly covered; the remaining work is a security/product decision about whether plaintext HTTP should support persistent device authentication.

## What This Changes

The issue asks OpenClaw to decouple Control UI device identity generation from browser secure-context/WebCrypto availability and resolve related device-less auth-scope behavior for HTTP reverse-proxy deployments.

## Best Possible Solution

Choose and document one security model: keep HTTPS/localhost as the device-identity boundary with clearer diagnostics, or explicitly support plaintext-LAN identity with focused auth, pairing, interception-risk, upgrade, and browser behavior coverage.

## Reproduction Assessment

Yes, at source level: current main still calls `crypto.subtle.digest` while loading or creating the browser identity and skips identity creation when `crypto.subtle` is unavailable. I did not run the reporter's live HTTP reverse-proxy setup in this read-only review.

## Solution Assessment

Unclear. Replacing WebCrypto SHA-256 with a pure-JS hash is the narrow technical path, but it is only the best solution if maintainers intentionally support persistent Control UI device authentication over plaintext HTTP.
