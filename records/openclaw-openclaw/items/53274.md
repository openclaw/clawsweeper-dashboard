## Summary

Keep open. Current main and v2026.5.22 still skip browser device identity when `crypto.subtle` is unavailable and still document/enforce remote plain-HTTP Control UI as a device-identity security boundary; the scope-bypass and implied-scope subparts are already fixed, but the central secure-context decoupling remains a product/security decision.

## What This Changes

The issue requests browser Control UI device identity generation without depending on Secure Context/WebCrypto, plus related scope preservation and operator-scope normalization fixes.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if approved, implement the pure-JS fingerprint path, attempt browser identity outside Secure Context, and update gateway policy, docs, tests, and dependency declarations together.

## Reproduction Assessment

Yes, source-backed. In an insecure browser context the UI skips identity because `crypto.subtle` is absent, and the gateway rejects remote device-less Control UI unless trusted-proxy, localhost compatibility, or break-glass bypass applies.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but accepting device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
