## Summary

Keep open: current main and the latest release still skip Control UI browser device identity when `crypto.subtle` is unavailable, so the central HTTP LAN/reverse-proxy request remains unresolved. The scope-preservation subissues are implemented, but removing the remaining gate would relax an auth boundary and needs maintainer security/product direction.

## What This Changes

The issue asks OpenClaw to generate Control UI browser device identity without depending on browser secure context and to preserve operator scopes when device-auth bypass is configured.

## Best Possible Solution

Keep the implemented scope fixes, then have a maintainer decide whether remote plain-HTTP browser identity should remain rejected, become an explicit opt-in mode, or be handled through HTTPS/trusted-proxy/Tailscale deployment guidance.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.10` shows the browser client skips device identity unless `crypto.subtle` exists, and the device-identity fingerprint still uses `crypto.subtle.digest`.

## Solution Assessment

Unclear. The proposed pure-JS hash and always-attempt identity path is mechanically plausible, but it is only the best solution if maintainers intentionally support persistent Control UI device identity in plaintext remote browser contexts.
