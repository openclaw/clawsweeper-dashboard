## Summary

Keep open. Current main still skips browser device identity when `crypto.subtle` is absent and still documents/enforces remote non-localhost HTTP as a Control UI device-identity security boundary; the linked implementation PR was closed as stale/conflicted while explicitly leaving this issue canonical.

## What This Changes

Request to let Control UI browser device identity work over non-secure LAN or reverse-proxy HTTP by removing the `crypto.subtle`/Secure Context gate, alongside already-addressed scope-bypass cleanup.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if approved, implement the pure-JS fingerprint path, remove the browser gate, and update connection policy, docs, tests, and dependency ownership together.

## Reproduction Assessment

Yes, source-backed. In an insecure browser context the UI skips device identity because `crypto.subtle` is absent, and current gateway policy rejects remote device-less Control UI connects unless trusted-proxy or break-glass bypass applies.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but accepting device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
