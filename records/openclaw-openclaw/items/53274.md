## Summary

Keep open. Current main and the latest release still tie browser device identity to `crypto.subtle`/Secure Context, while the bypass-scope and implied operator-scope subparts are already implemented; the closed companion PR discussion explicitly kept this issue as the canonical place for the remaining identity-policy work.

## What This Changes

The issue asks to decouple Control UI device identity from browser Secure Context so LAN HTTP and reverse-proxy deployments can authenticate with device identity, plus fix related bypass/scope handling.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if supported, add a direct browser-safe SHA-256 path, always attempt browser device identity with safe fallback, and update gateway policy, tests, docs, and package ownership together.

## Reproduction Assessment

Yes, source-backed. An insecure browser context makes the UI skip device identity, then a remote device-less Control UI connect reaches `reject-control-ui-insecure-auth`; I did not run a live browser or Docker repro in this read-only review.

## Solution Assessment

Unclear. The pure-JS hash direction is technically plausible, but allowing device identity over remote plain HTTP changes a documented security boundary, while the bypass-scope and implied-scope fixes are already on main.
