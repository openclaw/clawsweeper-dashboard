## Summary

Keep open. Current main still gates browser Control UI device identity on Secure Context/WebCrypto and documents remote plain-HTTP Control UI as rejected without device identity; only the bypass-scope and implied operator-scope parts are already implemented. The remaining request is a real but security-sensitive product boundary, and the related open items cover symptom/messaging rather than resolving the core non-secure LAN HTTP identity decision.

## What This Changes

Request to support browser Control UI device identity over non-secure LAN HTTP by replacing WebCrypto SHA-256, removing the Secure Context gate, and fixing related device-auth scope/bypass behavior.

## Best Possible Solution

Maintainers should explicitly decide the non-secure LAN HTTP threat model. If it is supported, add a browser-safe hashing path with the correct direct UI dependency, narrow or remove the Secure Context identity-generation guard, update gateway/UI tests, and revise the public Control UI and security docs; if it is not supported, keep the existing bypass/scope fixes and close or redirect the remaining UX work through the active messaging/warning issues.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: simulate an insecure browser context so the UI skips loadOrCreateDeviceIdentity(), then connect as remote Control UI without device identity and hit the reject-control-ui-insecure-auth policy path. I did not execute tests because this review was read-only.

## Solution Assessment

Unclear. The pure-JS hash direction is plausible for removing the client-side WebCrypto coupling, but always attempting browser device identity in remote non-secure HTTP would change the documented security posture; the narrow maintainable solution depends on maintainer threat-model approval.
