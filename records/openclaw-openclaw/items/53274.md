## Summary

Keep open. Current main still keeps browser Control UI device identity tied to Secure Context/WebCrypto and rejects remote device-less Control UI sessions; the bypass-scope and implied operator-scope subparts are already implemented, but the remaining HTTP LAN identity behavior needs maintainer security/product judgment.

## What This Changes

Allow Control UI device identity to be created on plain-HTTP LAN/reverse-proxy browser contexts, while preserving scopes for the explicit device-auth bypass.

## Best Possible Solution

Maintainers should explicitly decide the remote plain-HTTP Control UI threat model; if supported, add a declared browser-safe SHA-256 path, adjust the UI/gateway policy, and update tests/docs, otherwise keep the secure-context policy and route users to HTTPS, trusted-proxy auth, or the documented break-glass path.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence path: simulate an insecure browser context so the UI skips device identity, then connect as remote Control UI without a device and hit reject-control-ui-insecure-auth; I did not execute tests because this review was read-only.

## Solution Assessment

Unclear. The pure-JS hash change is technically plausible, but removing or narrowing the Secure Context gate changes the documented security boundary, so the best maintainable fix depends on maintainer threat-model approval.
