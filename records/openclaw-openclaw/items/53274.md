## Summary

Keep open. Current main and v2026.5.19 still skip browser device identity when `crypto.subtle` is unavailable and still reject remote device-less Control UI sessions; the bypass-scope and implied-scope parts are already implemented, so the remaining request is a security/product decision rather than cleanup.

## What This Changes

Request to support browser Control UI device identity over non-secure LAN or reverse-proxy HTTP contexts, with related scope-preservation fixes for device-auth bypass flows.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if approved, add browser-safe fingerprinting, keep the security policy explicit, and update gateway policy, docs, tests, and UI dependency ownership together.

## Reproduction Assessment

Yes, source-backed. In an insecure browser context the UI skips device identity because `crypto.subtle` is unavailable, and a remote device-less Control UI connect reaches `reject-control-ui-insecure-auth`; I did not run a live browser or Docker repro in this read-only review.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint is technically plausible, but removing the secure-context gate unconditionally would change a documented security boundary and needs maintainer approval first.
