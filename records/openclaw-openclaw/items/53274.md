## Summary

Keep open. Current main and the latest shipped tag still keep Control UI browser device identity tied to `crypto.subtle`/Secure Context, while the bypass-scope and implied operator-scope subparts are already implemented; a later maintainer review on the closed companion PR also says this issue remains the canonical place for the remaining narrow identity work.

## What This Changes

Request to let Control UI generate device identity without depending on browser Secure Context/WebCrypto, plus related scope handling fixes for device-auth bypass and operator scope hierarchy.

## Best Possible Solution

Maintainers should decide the remote plain-HTTP Control UI threat model; if supported, add a direct browser-safe SHA-256 dependency/path, remove or narrow the Secure Context gate, and update gateway policy, tests, and docs together.

## Reproduction Assessment

Yes, source-backed. An insecure browser context makes the UI skip device identity, then a remote device-less Control UI connect hits `reject-control-ui-insecure-auth`; I did not run a live browser or Docker repro in this read-only review.

## Solution Assessment

Unclear. The pure-JS hash direction is technically plausible, but allowing device identity over remote plain HTTP changes a documented security boundary, so the maintainable fix needs maintainer threat-model approval first.
