## Summary

Keep open. Current main still leaves the browser Control UI device identity path behind the Secure Context/WebCrypto gate, while the server-side bypass and implied operator-scope portions of the report have already landed. A maintainer also closed the companion PR as stale while explicitly saying this issue remains valid and should be handled with a narrower replacement.

## What This Changes

The issue requests generating browser Control UI device identity without depending on Secure Context/WebCrypto, plus related bypass-scope and operator-scope normalization fixes.

## Best Possible Solution

Maintainers should decide whether remote non-secure LAN HTTP is supported for Control UI device identity. If yes, narrow the implementation to a pure-JS browser fingerprint hash, always attempt identity creation before falling back to token-only auth, update the insecure-context tests, and revise the public Control UI docs; if no, keep the existing boundary and route UX improvements through #32473, #56830, and #71669.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: simulate an insecure browser context so `GatewayBrowserClient` does not call `loadOrCreateDeviceIdentity()`, then connect from a remote Control UI client without device identity and hit the gateway's `reject-control-ui-insecure-auth` policy.

## Solution Assessment

Unclear. The pure-JS hash plus always-attempt identity direction is the narrowest code shape for the client-side coupling, but it also changes the documented remote HTTP posture, so the final solution depends on maintainer security/product approval.
