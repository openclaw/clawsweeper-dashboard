## What This Changes

The PR changes the retry-after cap check from `>` to `>=` in `shouldBypassLongSdkRetry` and adds a regression test for a 429 response with `Retry-After: 60`.

## Best Possible Solution

Align code and public docs around an explicit inclusive retry-cap policy, then merge only after redacted real fallback-chain proof shows the 60-second OpenRouter stall is gone.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main leaves `Retry-After: 60` retryable at the default cap, and the upstream SDKs sleep parsed retry-after values unless `x-should-retry: false` is present.

## Solution Assessment

No, not as-is: the predicate change is the likely narrow code fix, but the PR should also align public retry/failover docs and supply real fallback-chain proof before merge.
