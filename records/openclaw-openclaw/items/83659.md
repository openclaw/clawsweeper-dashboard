## What This Changes

The PR changes SDK retry-cap handling so `Retry-After` values equal to the configured max wait bypass SDK-internal retry sleeps and adds a regression test for the 60-second boundary.

## Best Possible Solution

Align code and docs around an explicit inclusive cap policy, then merge after real fallback-chain proof shows the reported OpenRouter 60-second stall is gone.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main leaves `Retry-After: 60` retryable at the default 60-second cap, and upstream SDKs then sleep the retry-after value unless `x-should-retry: false` is injected.
