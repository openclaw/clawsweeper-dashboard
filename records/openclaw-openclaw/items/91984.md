## Summary

Keep open: the PR addresses a real Telegram local Bot API attachment materialization gap, but the new trusted-root candidate loop is not merge-ready because it treats every local read failure as a candidate miss instead of preserving size, symlink, permission, or fs-safe policy failures.

## What This Changes

The PR normalizes Telegram local Bot API container-absolute and relative `getFile().file_path` values into candidates under configured `trustedLocalFileRoots`, with focused resolver regression tests.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but make mapped candidate misses continue only on not-found and surface all other trusted-root read failures like the existing absolute trusted-root branch.

## Reproduction Assessment

Yes, at source level: current main rejects container-absolute local Bot API paths outside the host trusted root, while the PR maps them under configured roots and includes production proof of a materialized document. I did not run a live Telegram container in this read-only review.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the mapped candidate loop must distinguish not-found misses from real trusted-root read failures before merge.
