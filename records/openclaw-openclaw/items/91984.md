## Summary

Keep open: the PR addresses a real Telegram local Bot API attachment materialization bug, but the current patch still treats every trusted-root candidate read failure as a missing candidate, which can hide size, permission, or fs-safe policy failures before merge.

## What This Changes

The branch maps Telegram local Bot API container and relative file paths through configured `trustedLocalFileRoots` before HTTP fallback and adds resolver regression tests.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but preserve local read failure semantics and add focused regression coverage before merge.

## Reproduction Assessment

Yes, at source level. Current main rejects container-absolute local Bot API paths outside the host trusted root, and upstream local mode returns absolute `file_path` values; I did not run a live Telegram container in this read-only review.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the new candidate loop must distinguish not-found misses from real trusted-root read failures before merge.
