## Summary

Keep open: this targets a real Telegram local Bot API attachment bug and the PR body supplies usable production proof, but the patch should not merge as-is because the new trusted-relative candidate loop swallows all local read failures and can hide media materialization errors or fall back incorrectly.

## What This Changes

The PR maps Telegram local Bot API container and relative file paths through configured `trustedLocalFileRoots` before falling back to Bot API HTTP media download.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but make the new candidate loop continue only for not-found misses and surface other local read failures with focused regression coverage.

## Reproduction Assessment

Yes from source, but not from a live Telegram container in this read-only review. Current main rejects container-absolute local Bot API paths outside the host trusted root, and upstream local mode returns absolute `file_path` values.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the catch-all local candidate failure handling should distinguish not-found misses from real read failures before merge.
