## Summary

Keep open: this is a useful Telegram local Bot API media fix, but the PR still treats every trusted-root read failure in the new candidate loop as a miss, which can hide size, symlink, permission, or fs-safe policy failures before merge.

## What This Changes

The PR maps Telegram local Bot API container-absolute and relative `file_path` values under configured `trustedLocalFileRoots`, with focused media resolver regression tests.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but continue only on not-found candidate misses and preserve non-not-found trusted-root read failures with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects container-absolute local Bot API paths unless they are already under a host trusted root, and the PR maps those paths under configured roots. I did not run a live Telegram container in this read-only review.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the new candidate loop must distinguish not-found misses from real trusted-root read failures before this is the best fix.
