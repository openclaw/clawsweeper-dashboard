## Summary

Keep open: the PR is a useful Telegram local Bot API materialization fix, but the current patch has a merge-blocking resolver bug because the new trusted-root candidate loop treats every local read failure as a candidate miss.

## What This Changes

The PR maps Telegram local Bot API container and relative `file_path` values through configured `trustedLocalFileRoots` before falling back to HTTP downloads, with resolver regression tests.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but preserve local read failure semantics and add focused regression coverage before merge.

## Reproduction Assessment

Yes, at source level: current main rejects container-absolute local Bot API paths that are outside the configured host trusted root, and the PR body includes production proof that mapped paths materialized documents. I did not run a live Telegram container in this read-only review.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the new mapped candidate loop needs to distinguish path misses from real local read failures before this is the best fix.
