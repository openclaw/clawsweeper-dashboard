## Summary

Keep open: the change addresses a real Telegram local Bot API media materialization bug, but the PR is not merge-ready because the new mapped trusted-root loop treats every local read failure as a miss and can hide real attachment failures.

## What This Changes

The PR maps Telegram local Bot API container-absolute and relative `getFile().file_path` values into candidates under configured `trustedLocalFileRoots`, with resolver regression tests.

## Best Possible Solution

Keep the Telegram plugin-owned local Bot API mapping, but make mapped candidate misses continue only on typed not-found failures and surface other trusted-root read failures like the existing absolute trusted-root branch.

## Reproduction Assessment

Yes, at source level: current main rejects a container-absolute local Bot API file path unless that exact path is inside a configured host trusted root. The PR body also provides redacted production proof that the mapped path materialized a document, but I did not run a live Telegram container in this read-only pass.

## Solution Assessment

No, not as currently written. The resolver is the right owner boundary, but the mapped candidate loop must distinguish typed not-found misses from real trusted-root read failures before merge.
