## Summary

Keep this PR open, but it is not merge-ready: the fix targets the right Feishu webhook deadlock, yet the new unsigned encrypted path validates `earlyPayload.token` before decrypting the `{encrypt}` challenge body, so the reported real Feishu envelope still returns 401 instead of completing setup.

## What This Changes

The PR adds a pre-signature Feishu webhook challenge path for unsigned encrypted URL verification requests plus e2e coverage for token validation and malformed encrypted payloads.

## Best Possible Solution

Keep the fix inside the Feishu plugin, decrypt or parse the unsigned `{encrypt}` challenge enough to identify `url_verification`, validate the decrypted challenge token with timing-safe comparison, keep unsigned non-challenge events rejected, and prove the real callback shape.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.6.8` reject missing-signature requests before challenge handling, and the PR's own changed branch rejects the reported encrypted `{encrypt}` envelope before the SDK can decrypt it.

## Solution Assessment

No. The direction is right, but checking `earlyPayload.token` before decrypting the encrypted challenge means this patch does not solve the official/reported setup callback shape.
