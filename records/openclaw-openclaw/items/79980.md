## Summary

Keep open: current main still documents xAI realtime voice as not exposed, and the xAI plugin does not register a realtime voice provider. This is a new provider capability for Voice Call, not a stale cleanup item.

## What This Changes

Request first-class xAI Grok Voice realtime support for Voice Call, plus direct realtime send_dtmf and end_call handlers.

## Best Possible Solution

Add a provider-owned xAI realtime voice bridge through the existing realtime voice plugin seam, with docs, focused tests, and live xAI/Twilio proof; decide the DTMF/end-call realtime handlers as separate Voice Call scope if needed.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug. Source inspection verifies current main lacks an xAI realtime voice provider while the generic provider seam exists.

## Solution Assessment

Yes for the provider direction: a dedicated xAI realtime voice provider fits the plugin/provider boundary better than branching on xAI hosts inside the OpenAI bridge. The DTMF and end-call handlers should be reviewed as a separate Voice Call behavior decision.
