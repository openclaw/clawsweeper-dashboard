## Summary

Keep open. Current main still lacks an xAI/Grok realtime voice provider for the Voice Call realtime path, and the related open xAI realtime items target adjacent `/voiceclaw/realtime` or `/chat` surfaces rather than this Twilio/voice-call request. The remaining work is feature/product design, not conservative cleanup.

## What This Changes

Request native xAI Grok Voice Agent support for Voice Call realtime, including xAI-compatible session setup, barge-in behavior, and optional direct DTMF/end-call realtime tool handling.

## Best Possible Solution

Keep the OpenAI bridge provider-specific code clean, and resolve this through an xAI-owned realtime voice provider plus a separate maintainer decision on default Voice Call realtime DTMF/end-call handlers.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks xAI/Grok realtime voice registration for Voice Call and only exposes OpenAI/Google realtime voice providers there.

## Solution Assessment

Unclear until maintainer direction. The requested dedicated xAI provider-plugin direction is more maintainable than host-branching inside the OpenAI bridge, but bundling versus ClawHub and the generic DTMF/end-call realtime handler behavior need product/API review.
