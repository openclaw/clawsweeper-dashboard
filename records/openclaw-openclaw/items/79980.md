## Summary

Keep open: current main explicitly lacks an xAI realtime voice provider for Voice Call, and the related open xAI realtime work targets different `/voiceclaw/realtime` or `/chat` surfaces. This is a new bundled provider capability with live-vendor behavior and a separate phone-control tool question, so it needs maintainer/provider review rather than cleanup closure.

## What This Changes

Request first-class xAI/Grok realtime voice support for the Voice Call plugin, plus possible `send_dtmf` and `end_call` realtime tool handlers.

## Best Possible Solution

Add a provider-owned xAI realtime voice bridge through the existing realtime voice SDK seam, with docs, focused tests, and redacted xAI/Twilio Voice Call proof; split the DTMF/hangup handlers unless maintainers approve them as part of the same Voice Call feature.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high-confidence verification that current main does not register or document xAI as a Voice Call realtime voice provider.

## Solution Assessment

Yes for the report's Option B direction: a dedicated xAI realtime voice provider matches the provider/plugin boundary better than branching on xAI hosts inside the OpenAI bridge. The DTMF and hangup handlers should be reviewed as a separate Voice Call tool scope unless maintainers intentionally bundle them.
