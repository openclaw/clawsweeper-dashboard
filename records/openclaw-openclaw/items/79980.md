## Summary

Keep open: current main still documents and implements xAI realtime voice as unavailable, and this is a new provider capability for Voice Call rather than stale cleanup. Related xAI realtime threads are adjacent but target different surfaces, so they do not supersede the telephony-specific request or its DTMF/hangup scope.

## What This Changes

The issue requests first-class xAI Grok Voice Agent API support for Voice Call realtime, preferably through a dedicated xAI realtime voice provider plus optional realtime DTMF and hangup tool handlers.

## Best Possible Solution

Add an xAI-owned RealtimeVoiceProviderPlugin in the xAI plugin for Voice Call and Talk-compatible realtime voice, with docs, focused tests, and redacted live Twilio/xAI proof; decide the DTMF and hangup realtime tools as a separate Voice Call scope if needed.

Not applicable as a bug reproduction: this is a feature request for a new provider capability. Source inspection gives high confidence that current main lacks an xAI realtime voice provider while already supporting other xAI audio surfaces.

## Solution Assessment

Yes for the report's dedicated-provider direction: it matches the plugin/provider boundary better than branching on xAI hosts inside the OpenAI bridge. The phone-control realtime tools should be reviewed as separate Voice Call behavior rather than silently bundled into the provider change.
