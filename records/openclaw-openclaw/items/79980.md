## Summary

Keep open. Current main has the generic realtime voice provider surface that could support this, but xAI/Grok is not registered as a realtime voice provider and the remaining work needs maintainer direction on bundled-provider scope, shared protocol handling, and voice-call tool behavior.

## What This Changes

Request native xAI Grok Voice Agent support for Voice Call realtime sessions, including xAI-specific session schema, barge-in behavior, and optional DTMF/hangup realtime tools.

## Best Possible Solution

Add a maintainer-approved xAI `RealtimeVoiceProviderPlugin` on the provider-owned xAI surface, with focused Voice Call coverage and docs, reusing shared OpenAI-Realtime protocol helpers only where that avoids duplicating provider semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main cannot select `realtime.provider: "xai"` because xAI registers no realtime voice provider.

## Solution Assessment

Unclear until maintainer review. A dedicated xAI realtime voice provider is the cleanest code boundary, but bundling policy, shared adapter reuse, and Voice Call realtime tool exposure need product/API ownership decisions first.
