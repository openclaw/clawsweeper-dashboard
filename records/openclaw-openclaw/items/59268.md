- `P2`: This is a real macOS remote-mode session-routing bug with limited blast radius to paired multi-Mac setups.
- `impact:message-loss`: The report describes messages from one Mac appearing in the other Mac’s app, which is a concrete misrouting/delivery-isolation problem.

## Summary

Keep open. Current main and the latest shipped tag still route macOS WebChat startup and picker defaults through the gateway main session, while existing device/node identity can distinguish paired Macs but is not used for the local paired-Mac default.

## What This Changes

The issue requests per-Mac node-scoped chat picker defaults and isolated sessions when multiple macOS apps are paired to one gateway.

## Best Possible Solution

Implement a remote-mode macOS WebChat default that resolves to the local paired node session and local Mac display name, while preserving local-mode WebChat on the selected agent’s main session.

## Reproduction Assessment

Yes, at source level. Current main still opens macOS WebChat and rewrites picker defaults through the shared gateway main session instead of a local paired-node session key; I did not run the live two-Mac lab.

## Solution Assessment

Yes directionally, but not by changing only the static client id. The maintainable fix should use the existing paired device/node identity and node-scoped session convention for remote-mode macOS WebChat defaults.
