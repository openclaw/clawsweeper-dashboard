## Label Justifications

- `P1`: The report describes a real multi-Mac workflow where chat labels and visible history can cross paired nodes with no workaround.
- `impact:message-loss`: The issue is about messages appearing in the wrong paired Mac chat session rather than staying scoped to the sending node.

## Summary

Keep open. Current main and v2026.5.18 still drive macOS WebChat startup and picker defaults through the shared gateway main session instead of the local paired-node session, so the multi-Mac label/session collision remains source-reproducible.

## What This Changes

The issue asks for each paired macOS app to default WebChat to its own node session and display the local node label instead of sharing or showing another Mac’s session.

## Best Possible Solution

Use the existing device/node identity and node-session convention to make remote-mode macOS WebChat default to the local paired node session and label, while keeping local-mode WebChat on the selected agent main session.

## Reproduction Assessment

Yes, at source level. Current main still opens macOS WebChat through WebChatManager.preferredSessionKey() and rewrites picker defaults through the shared gateway main session while existing node-originated event plumbing defaults to node-scoped sessions.

## Solution Assessment

Yes directionally, but not by changing only the static client id. GatewayChannel already sends device identity when enabled, so the maintainable fix should resolve remote-mode macOS WebChat defaults from the paired local node identity and preserve local-mode main-session behavior.
