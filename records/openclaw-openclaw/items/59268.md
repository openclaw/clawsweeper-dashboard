## Summary

Keep open. Current main and v2026.5.18 still route macOS WebChat startup and picker defaults through the shared gateway main session instead of the local paired-node session, while node-originated events already use node-scoped sessions.

## What This Changes

The issue reports that multiple paired macOS apps can display the other Mac's node label and share a chat session instead of using isolated per-node sessions.

## Best Possible Solution

Use the existing device/node identity and node-session convention to make remote-mode macOS WebChat default to the local paired node session and label, while preserving local-mode WebChat on the selected agent main session.

## Reproduction Assessment

Yes, at source level. Current main still opens macOS WebChat through WebChatManager.preferredSessionKey() and rewrites picker defaults through the shared gateway main session while node-originated event plumbing defaults to node-scoped sessions.

## Solution Assessment

Yes directionally, but not by changing only the static client id. GatewayChannel already sends device identity when enabled, so the maintainable fix should resolve remote-mode macOS WebChat defaults from the paired local node identity and preserve local-mode main-session behavior.
