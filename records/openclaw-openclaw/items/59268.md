## Label Justifications

- `P1`: The report describes a real multi-Mac workflow where chat sessions and visible history are shared across paired nodes with no workaround.
- `impact:message-loss`: The issue is about messages appearing in the wrong paired Mac chat session rather than staying scoped to the sending node.

## Summary

Keep open. Current main and the latest stable release still route macOS WebChat startup and picker defaults through the shared gateway main session rather than a local paired-node session, so the reported multi-Mac label/session collision remains source-reproducible.

## What This Changes

The issue asks for each paired macOS app to default to its own node-scoped chat session and display the local node label when multiple Macs share one gateway.

## Best Possible Solution

Implement a remote-mode macOS WebChat default that resolves to the local paired node session and local Mac display name, while keeping local-mode WebChat on the selected agent main session.

## Reproduction Assessment

Yes, at source level. Current main still starts macOS WebChat and rewrites picker defaults through the shared gateway main session while node-scoped session plumbing exists elsewhere.

## Solution Assessment

Yes directionally, but not by changing only the static client id. The maintainable fix should use existing device/node identity and the gateway node-session convention for remote-mode macOS defaults.
