## Label Justifications

- `P1`: The issue breaks a real macOS remote multi-node chat workflow by sharing session history and picker state across paired Macs.
- `impact:message-loss`: Messages from one paired Mac can appear in the other Mac’s chat because both apps target the same session.
- `impact:session-state`: The core failure is incorrect session selection and shared chat history state for distinct paired nodes.

## Summary

Keep open. Current main and the latest release still route macOS WebChat startup, picker defaults, history, and send calls through the shared gateway main session while node-scoped session plumbing exists elsewhere but is not used for the local paired Mac default.

## What This Changes

The issue asks macOS WebChat to default each paired Mac to its own node chat session and label instead of sharing one gateway-main session across Macs.

## Best Possible Solution

Implement a remote-mode macOS WebChat default that resolves to the local paired device/node session and display name, while preserving local-mode WebChat on the selected agent’s main session.

## Reproduction Assessment

Yes, at source level. Current main still starts macOS WebChat from GatewayConnection.shared.mainSessionKey(), rewrites session-list defaults to cachedMainSessionKey(), and sends history/messages through that canonicalized shared key.

## Solution Assessment

Yes directionally, but not by changing only the static client id. The maintainable fix should use existing paired device/node identity and the gateway’s node-scoped session convention for remote-mode macOS WebChat defaults.
