## Summary

Current main does not yet satisfy the reported per-Mac chat isolation behavior. The macOS app still opens WebChat on the gateway main session, while the current node/device identity plumbing is separate and does not drive the picker’s preferred session.

## What This Changes

The issue requests that each remotely paired macOS app default to its own node-scoped chat session and show its own node label when multiple Macs are paired to one gateway.

## Best Possible Solution

Implement a remote-mode macOS WebChat default that resolves to the local paired node’s session and labels it with the local Mac display name, while keeping local-mode WebChat on the selected agent’s main session and updating the macOS WebChat docs/tests to describe the split.

## Reproduction Assessment

Yes. The issue gives a concrete two-Mac remote pairing path, and current source inspection confirms both app instances choose the same gateway main session key instead of a local device/node-derived key.

## Solution Assessment

No. Current main has the right device identity plumbing for unique nodes, but the macOS WebChat picker still uses the shared main-session default; the narrower maintainable fix is to use a node-scoped default only for remote macOS node mode and preserve the existing local-mode behavior.
