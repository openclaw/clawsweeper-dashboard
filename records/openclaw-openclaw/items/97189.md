## Summary

Keep open: the durable restart-audit direction is useful, but this PR is not merge-ready because audit persistence can block restart scheduling, raw session keys can reach restart warnings, and the contributor has not supplied real setup proof.

## What This Changes

The PR adds a durable `gateway_restart_audit` SQLite table and threads restart source/session/preflight audit metadata through gateway restart request paths.

## Best Possible Solution

Land the durable audit approach only after audit persistence is non-blocking or tightly bounded, raw session identifiers are redacted or explicitly approved for logs, fresh/upgrade DB coverage exists, and redacted real restart output proves an inserted audit row.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR calls a synchronous shared-state write before the scheduled restart state is armed, and it adds raw `sessionKey=` output to restart warnings. I did not run tests or live restart proof because this review is read-only.

## Solution Assessment

No. Shared SQLite is a plausible owner for durable restart audit events, but this patch should not let audit storage delay restart scheduling and should not add raw session identifiers to warnings without redaction or explicit maintainer approval.

Full review comments:

- **[P2] Keep audit writes off the restart scheduling path:** `src/infra/restart.ts:1138-1149`
  - body: This writes the durable audit row before the pending restart state is armed. `writeGatewayRestartAuditEventSync()` uses the shared state DB's synchronous immediate transaction path, and that connection has a 30s busy timeout, so a locked state DB can delay `/restart`, config, or update restart scheduling even though audit recording is best-effort. Arm the restart first or make the audit write non-blocking/short-timeout.
  - confidence: 0.9
- **[P2] Redact session keys before writing restart warnings:** `src/infra/restart.ts:419-420`
  - body: Adding `sessionKey=` to restart warning text can expose channel or user routing identifiers in logs that are often copied into support/debug reports. Keep the durable DB attribution if maintainers want it, but warning logs should hash/drop the raw key or carry an explicit maintainer-approved diagnostic contract.
  - confidence: 0.82
