## Summary

Keep open: the PR targets a real source-reproducible Codex allowlist bug, but the patch is not mergeable as-is because it turns any explicit native-owned tool name into the full Codex native shell/file code-mode surface, contradicting the documented `--tools` restriction and the current fail-closed guard.

## What This Changes

The PR changes the Codex app-server native-surface gate so explicit `toolsAllow` entries matching native-owned tools enable the native surface, and adds a unit test for those allowlists.

## Best Possible Solution

Keep wildcard or absent allowlists as the full native-surface path, but make explicit native-owned allowlists enforce only the requested tools through an OpenClaw dynamic surface or fail with a clear diagnostic when exact enforcement is unavailable.

## Reproduction Assessment

Yes. Source inspection and the linked issue show that explicit non-wildcard `toolsAllow` entries containing native names are excluded from dynamic tools and do not enable the native surface on current main; I did not run a live cron in this read-only review.

## Solution Assessment

No. Enabling the full native surface for any native-owned name is not the safest repair because current code treats that surface as broad code mode, so the fix should preserve exact allowlist semantics or fail loudly when exact enforcement is impossible.
