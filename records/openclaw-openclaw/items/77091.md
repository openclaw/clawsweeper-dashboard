## Summary

Keep open: the Telegram status/probe stall is still plausible on current main, but this PR is not merge-ready because it adds an opt-in timeout without settling the default freeze path, leaves generated config metadata and changelog coverage incomplete, has timeout tests that can hang, and bundles unrelated fixes.

## What This Changes

The PR adds a Telegram `network.httpTimeoutMs` probe timeout setting and also changes transcript migration, plugin context-engine cache restoration, and Control UI tool-card rendering.

## Best Possible Solution

Split this into a narrow Telegram status/probe fix with default-safe behavior, regenerated config metadata/docs/changelog, and focused tests; handle the transcript, plugin cache, and Control UI fixes separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows Telegram status/startup probes can wait on the existing 15s/full budgets, but the full dashboard menu-freeze path was not live-run against current main.

## Solution Assessment

Is this the best way to solve the issue? No: an opt-in `network.httpTimeoutMs` may be useful, but the reported default menu freeze should be fixed with a default-safe probe/status path and complete generated metadata, docs, changelog, and tests.
