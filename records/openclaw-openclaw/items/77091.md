Overall confidence: 0.9

Full review comments:

- **[P1] Cap the whole probe, not each retry:** `extensions/telegram/src/probe.ts:187`
  - body: This only applies `httpTimeoutMs` to each individual `fetchWithTimeout` call. After a configured timeout, the retry loop still sleeps and retries until the original probe budget is exhausted, so `httpTimeoutMs: 5000` under a larger `channels.status` budget can still hold the health probe near the old delay instead of fixing the reported freeze.
  - confidence: 0.9
- **[P2] Make the timeout tests settle on abort:** `extensions/telegram/src/probe.test.ts:312-320`
  - body: The added mock returns a promise that never resolves and only checks `signal.aborted` when the fetch is first called. `fetchWithTimeout` aborts the signal later but still awaits the fetch promise, so these tests can hang instead of proving the configured timeout behavior.
  - confidence: 0.93
- **[P1] Split the unrelated state and UI changes out:** `src/config/sessions/transcript-append.ts:120-122`
  - body: This Telegram timeout PR also changes transcript migration, context-engine cache restore, and tool-card rendering. Those are separate session-state/plugin-cache/UI behaviors, and one tool-card piece is already present on current main, so merging them here would bypass the focused ownership and proof this timeout change needs.
  - confidence: 0.88
