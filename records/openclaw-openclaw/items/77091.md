## Summary

Keep open: current main does not implement the Telegram timeout knob, and this PR is not ready as-is because the proposed fix is opt-in, the added timeout tests can hang, and the config/schema surface is incomplete for the Control UI.

## What This Changes

The PR adds a Telegram `network.httpTimeoutMs` probe timeout setting and also bundles unrelated transcript migration, plugin context-engine cache, and Control UI tool-card changes.

## Best Possible Solution

Split this into a narrow Telegram fix that prevents the default health/status probe path from freezing the Control UI, then update the config schema, generated metadata, docs, changelog, and focused regression tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source shows Telegram startup/status probes can await the existing 15-30 second budgets, but I did not run the full dashboard menu-freeze path against current main.

## Solution Assessment

Is this the best way to solve the issue? No: the PR adds an opt-in config knob but leaves the default freeze path and generated Control UI config metadata incomplete; a narrower default-safe probe/status fix would be more maintainable.
