## Summary

Keep open: current docs/tests define account-scoped bindings as a valid route selector, and source inspection shows a credible reload boundary where binding changes can be classified as no-op while long-running channel tasks keep using a captured config snapshot.

## What This Changes

The issue reports that a Feishu account binding added after Gateway startup continues routing messages to the main agent instead of the newly configured agent.

## Best Possible Solution

Make binding and agent-list changes reliably visible to inbound channel routing, either by restarting affected channel account tasks on those config changes or by resolving route-critical config from the current runtime snapshot at dispatch time, with Feishu/account-scoped regression coverage.

## Reproduction Assessment

Unclear for a full live reproduction: I did not run a Feishu/macOS Gateway, but source inspection shows a credible current-main path where binding changes are treated as no-op while channel workers use a captured config object. The documented account-binding behavior and routing tests make the reported fallback to main actionable enough to keep open.

## Solution Assessment

Unclear, but the narrowest maintainable direction is to fix the generic reload/config freshness boundary rather than special-case Feishu. A focused fix should prove that adding agents.list plus bindings plus a Feishu account routes the next inbound message to the new agent.
