## Label Justifications

- `P1`: The reported bug misroutes Feishu messages to the wrong agent session, which is a broken channel workflow for affected users.
- `impact:message-loss`: Messages intended for the newly bound Feishu agent were delivered to `main` instead of the configured target agent.
- `impact:session-state`: The wrong route also creates or continues the wrong per-agent session key, mixing conversation state with `main`.

## Summary

Current main already addresses the reported Feishu stale-config routing failure, and the fix shipped in beta v2026.6.8-beta.1, so this no longer needs to stay open as separate work.

## What This Changes

The issue reports that a Feishu account binding added after Gateway startup still routes direct messages to `main` instead of the newly bound agent.

## Best Possible Solution

Keep the Feishu runtime-config refresh and route re-resolution shipped in v2026.6.8-beta.1; reopen or file a new report only if the same account binding still falls back to `main` on that version or newer.

## Reproduction Assessment

Yes. Source inspection shows the old failure mode as a stale Feishu request config defaulting before runtime-current bindings were checked; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes. The fix is in the Feishu plugin owner boundary, refreshes runtime config before dispatch, and reuses the shared account-scoped route resolver instead of adding a second routing path.
