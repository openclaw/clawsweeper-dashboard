## Summary

Keep open: current main still leaves `bindings` as a no-op reload path while Feishu inbound routing uses the config object captured by the running channel account, so a newly added account binding can still be invisible to the next message route.

## What This Changes

A newly added Feishu account binding after Gateway startup should route messages to that agent instead of falling back to `main`.

## Best Possible Solution

Make route-critical binding and agent-list changes visible to Feishu and other long-lived channel routing decisions, with regression coverage proving a newly added account binding routes the next inbound message to the new agent.

No, not as a live Feishu/macOS reproduction in this read-only review. Source inspection does show a concrete current-main path where `bindings` reloads are no-op plans while Feishu routes from a captured channel-start config object.

## Solution Assessment

Yes, the narrow maintainable fix is config freshness at the Gateway/channel routing boundary, not a Feishu-only workaround or a new user setting. The patch should prove the next inbound Feishu message sees the newly added binding.
