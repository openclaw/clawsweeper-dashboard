## Summary

Keep open: current main still treats `bindings` as a no-op for Gateway reload while Feishu workers route with a config object captured at channel start. The docs and resolver both say account bindings should select the configured agent before falling back to `main`, so the reported fallback remains actionable.

## What This Changes

The issue asks OpenClaw to apply newly added agent/account bindings to Gateway routing after config reload or restart instead of routing Feishu messages to the default `main` agent.

## Best Possible Solution

Make route-critical `bindings` and agent-list changes refresh the generic Gateway/channel routing snapshot, with regression coverage proving a newly added account binding routes the next inbound message to the new agent.

## Reproduction Assessment

No high-confidence live reproduction: I did not run Feishu on macOS. Source inspection shows a concrete current-main path where binding changes are no-op reload paths while Feishu routing uses a config captured by the channel worker.

## Solution Assessment

Yes, the best fix is generic Gateway config freshness for routing rather than a Feishu-only workaround. The patch should make binding changes visible to channel routing and prove the next inbound message selects the newly bound agent.
