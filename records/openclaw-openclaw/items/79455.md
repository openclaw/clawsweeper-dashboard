## Summary

Keep open: current main still maps Telegram direct/private topic sends to `message_thread_id` and falls back by stripping that field, while the reporter's live Bot API probe and current Telegram docs indicate `direct_messages_topic_id` is now needed for direct messages chat topics. The related open work at https://github.com/openclaw/openclaw/issues/79448 and https://github.com/openclaw/openclaw/pull/79410 covers adjacent fallback/status behavior, not this field-selection mismatch.

## What This Changes

Request Telegram outbound delivery to use `direct_messages_topic_id` for direct-message topics while preserving `message_thread_id` for forum/private-chat topic modes that still require it.

## Best Possible Solution

Carry a typed Telegram thread kind from inbound/direct-message metadata through outbound planning, then emit `direct_messages_topic_id` only for direct messages chat topics and keep existing forum/private-chat `message_thread_id` behavior where Telegram still requires it.

## Reproduction Assessment

Yes, with high confidence from source plus external API evidence: current main sends DM-scoped topic ids as `message_thread_id`, the issue supplies live Bot API success/failure probes, and the retry path strips the field into a main-DM send. I did not rerun a live Telegram bot send in this read-only review.

## Solution Assessment

No, not as a blanket `scope === "dm"` replacement. The maintainable fix is to distinguish Telegram direct messages chat topics from private-chat forum-topic mode and choose the API field from that prepared runtime fact.
