## Summary

Keep open: current main still sends DM-scoped Telegram topic IDs as `message_thread_id` and falls back to a bare DM on `message thread not found`; the report and the related live-probe comments show that affected Telegram DM topic sends now succeed with `direct_messages_topic_id` instead.

## What This Changes

The issue asks OpenClaw to route Telegram private DM topic sends with `direct_messages_topic_id` while preserving `message_thread_id` for forum/group topics.

## Best Possible Solution

Add a guarded Telegram plugin compatibility path that keeps existing forum behavior, preserves current `message_thread_id` sends where they still work, and uses or retries `direct_messages_topic_id` for affected DM/direct-message topic sends before falling back to a bare DM.

## Reproduction Assessment

Yes from source inspection for OpenClaw's current behavior: DM topic sends are built with `message_thread_id`, and the fallback removes that field after `message thread not found`. The external Telegram rejection path is backed by the reporter's raw Bot API probes and a related operator comment, not by a local tokened repro in this review.

## Solution Assessment

Yes, a narrow Telegram plugin change is the best repair, but it should be compatibility-oriented rather than a blanket field swap. The safer path is to add `direct_messages_topic_id` support or retry for DM/direct-message topics while preserving existing forum and private-topic behavior where Telegram still accepts `message_thread_id`.
