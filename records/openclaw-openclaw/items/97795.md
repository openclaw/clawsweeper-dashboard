## Summary

Keep open: current main still has rejected terminal-persistence promises that clean up or broadcast fallback state without recording the failure, so the report identifies a real gateway observability gap.

## What This Changes

The issue requests logging for failed chat-abort terminal persistence so operators can diagnose session-store write failures.

## Best Possible Solution

Add one focused gateway-side diagnostic for rejected terminal session persistence, keep the current cleanup and restart-recovery behavior intact, and cover the rejected-persistence path with a regression test.

## Reproduction Assessment

Yes from source, though I did not run a failing test in this read-only review. A rejected `projectSessionTerminalPersistence` promise reaches catch blocks that delete or project fallback state without any logger call.

## Solution Assessment

The requested direction is right, but the exact proposed patch is only a sketch because `chat-abort.ts` does not currently own a logger. The best repair should log once at the gateway persistence/tracking boundary and preserve cleanup semantics.
