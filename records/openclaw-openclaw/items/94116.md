## Summary

Keep this PR open, but it needs changes before merge: the finalizer is the right narrow layer for the linked cron status bug, but the branch reclassifies every non-aborted delivery `status: "error"` as execution `ok`, which also catches deliberate `errorKind: "delivery-target"` refusals that current code treats as errors to avoid unsafe implicit last-target delivery outcomes.

## What This Changes

The PR changes isolated cron finalization so successful agent execution stays `ok` when post-run delivery fails, while retaining delivery failure metadata and adding a regression test.

## Best Possible Solution

Keep the focused finalizer fix, but narrow it so `errorKind: "delivery-target"` remains an error, add regression coverage for that guard, and require redacted real cron or persisted run-log proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main produces `errorKind: "delivery-target"` refusal results with `status: "error"`, and the PR's broad post-dispatch reclassification would convert that same shape to `ok`.

## Solution Assessment

No as submitted. `finalizeCronRun` is the right narrow layer, but the best fix must preserve deliberate delivery-target refusals as errors while only decoupling genuine post-run delivery failures from successful execution.
