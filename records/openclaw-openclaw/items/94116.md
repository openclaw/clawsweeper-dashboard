## Summary

Keep open: the patch targets the right cron finalization path and I found no blocking code defect, but this external PR’s after-fix proof is mock/test-only and the merge intentionally changes how delivery failures affect persisted cron status.

## What This Changes

The PR changes isolated cron finalization so delivery-dispatch errors preserve a successful isolated agent result as `ok` while carrying delivery error metadata and adds a regression test for that case.

## Best Possible Solution

Land the focused finalization fix only after contributor-provided real cron proof and maintainer acceptance of `ok` plus delivery metadata as the interim partial-success model while the broader `warning` status contract remains separate.

## Reproduction Assessment

## Solution Assessment

Yes as an acceptable narrow mitigation, but not the full product solution. The best long-term model may be the separate public `warning` status tracked by https://github.com/openclaw/openclaw/issues/49190; this PR is a smaller fix that preserves existing status values.
