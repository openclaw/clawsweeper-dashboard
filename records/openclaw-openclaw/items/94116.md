## Summary

Keep open: current main still has the post-dispatch delivery-error propagation path this draft PR targets, and the related recovered-tool-warning reports are adjacent rather than a safe superseding replacement. The PR remains blocked on external real cron/delivery proof and maintainer acceptance of changing delivery-only failures from primary cron `error` to `ok` plus delivery metadata.

## What This Changes

The PR changes isolated cron finalization so a successful agent run stays `ok` when post-run delivery returns an error, while preserving delivery error metadata and adding focused regression coverage.

## Best Possible Solution

Land the focused finalization fix only after contributor-provided real cron proof and maintainer acceptance of `ok` plus delivery metadata as the interim partial-success model, while keeping broader warning-status work separate.

## Reproduction Assessment

Yes, source-reproducible: current main can return a dispatcher `status: "error"` after non-fatal isolated payload resolution, and the finalizer currently propagates that status. I did not reproduce the original WSL2/Feishu scheduled setup live.

## Solution Assessment

Yes as an acceptable narrow mitigation, but not the full product solution. Maintainers still need to accept the interim `ok` plus `not-delivered` semantics or route the behavior through the broader warning-status contract.
