## Summary

Keep this PR open with a blocking security review finding. Current main preserves the fail-closed WhatsApp `allowFrom` behavior for explicit direct-message sends, the PR would reverse the shipped #20108 fix, and the open #49549 `allowTo` proposal is the safer opt-in path for outbound-only authorization.

## What This Changes

The PR changes WhatsApp outbound target resolution and regression coverage so `mode: "explicit"` direct targets can resolve outside `allowFrom` while implicit and other gated modes remain on the allowlist path.

## Best Possible Solution

Do not merge the unconditional explicit-mode bypass. Preserve the shipped fail-closed `allowFrom` behavior unless maintainers land an explicit outbound policy such as `allowTo` with config/schema/docs/tests and clear security semantics.

## Reproduction Assessment

Yes. A focused code-level reproduction is `resolveWhatsAppOutboundTarget({ to: <valid E.164>, allowFrom: [<different E.164>], mode: "explicit" })`: current main returns an allowFrom policy error, while this PR changes that case to succeed; the main outbound send path uses the same explicit mode.

## Solution Assessment

No. The PR is not the best solution because it silently broadens a documented authorization boundary; an opt-in outbound allowlist such as #49549 is the narrower maintainable path for outbound-only use cases.
