## Summary

Keep this PR open for maintainer judgment. Current main still intentionally enforces WhatsApp `allowFrom` for explicit direct-message outbound targets, and this PR would reverse a documented security fix while also targeting the old `src/whatsapp` layout; the open #49549 `allowTo` proposal is the safer opt-in direction but is not merged.

## What This Changes

The PR changes the WhatsApp outbound target resolver and regression test so `mode: "explicit"` direct targets bypass `allowFrom` membership checks while implicit and custom modes remain allowlist-gated.

## Best Possible Solution

Do not merge the unconditional explicit-mode bypass. Maintainers should choose the outbound authorization model explicitly: either keep the shipped fail-closed `allowFrom` behavior, or land an opt-in `allowTo`/outbound policy with WhatsApp plugin config types, schema, docs, and regression tests before closing this PR as superseded.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `resolveWhatsAppOutboundTarget({ to: <valid E.164>, allowFrom: [<different E.164>], mode: "explicit" })`: current main fails closed, while this PR changes that case to succeed; the main message send path uses the same explicit mode.

## Solution Assessment

No. The PR’s unconditional bypass is not the best way to solve outbound-only WhatsApp use cases because it silently broadens a documented security boundary; an explicit `allowTo` or equivalent opt-in outbound policy is the safer maintainable alternative.
