## Summary

Keep open: current `main` still omits the queued message id from the followup delivery threading path, and this PR is the canonical small fix for that handoff. It should not be closed as stale or duplicate, but it still needs contributor real behavior proof and refreshed merge/check status before merge.

## What This Changes

The PR adds an optional `currentMessageId` handoff through `resolveFollowupDeliveryPayloads` and passes `queued.messageId` from the followup runner into `applyReplyThreading`.

## Best Possible Solution

Land the minimal current-message handoff with focused regression coverage and contributor real behavior proof, while keeping the false-default implicit-threading work tracked separately at https://github.com/openclaw/openclaw/issues/66540.

## Reproduction Assessment

Yes, at source level. Current `main` stores `FollowupRun.messageId` and `applyReplyThreading` can consume `currentMessageId`, but the followup delivery path never passes that value through.

## Solution Assessment

Yes for the missing-id handoff. The proposed optional-parameter plumbing is the narrowest maintainable fix for this path; the separate `replyToCurrent: false` default issue should remain a separate follow-up.
