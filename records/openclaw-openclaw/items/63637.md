## Summary

Keep this PR open. Current main still does not forward the queued followup message id into reply threading, while the PR's three-line plumbing change matches the existing helper contract and appears correct. It should get maintainer review, preferably with a focused regression test for the followup handoff before merge.

## What This Changes

The PR adds an optional currentMessageId parameter to followup delivery, passes queued.messageId from the followup runner, and forwards it to applyReplyThreading so followup replies can populate replyToId.

## Best Possible Solution

Land the minimal currentMessageId handoff with a focused regression test, unless a maintainer intentionally folds the same behavior into a broader reply-threading change. The end state should be that queued followup replies configured with replyToMode all receive the triggering message id as replyToId.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: queued followups store messageId, but the followup runner and resolveFollowupDeliveryPayloads never pass it to applyReplyThreading, so replyToMode all cannot populate replyToId for the drained followup payload.

## Solution Assessment

Yes. The proposed optional-parameter plumbing is the narrowest maintainable fix for the observed gap because the data already exists on FollowupRun and the downstream helper already accepts it; the safer merge path is to add one focused regression test around that handoff.
