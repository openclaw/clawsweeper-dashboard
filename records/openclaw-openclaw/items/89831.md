## Summary

No blocking findings. The patch matches the source-level failure path from the linked isolated-cron report, keeps stale or missing progress on the existing recoverable stuck path, and is scoped to diagnostic classification plus focused regression coverage.

## What This Changes

The PR adds a fresh-progress branch to `classifySessionAttention` so sessions without `activeWorkKind` are treated as long-running instead of stuck when `lastProgressAgeMs` is within `staleMs`, with two regression cases.

## Best Possible Solution

Land the bounded classifier fix with the focused regression tests, then keep any broader CLI-runtime active-work registration redesign as a separate follow-up if maintainers want stronger diagnostics taxonomy.

## Reproduction Assessment

Yes at source level: current main ignores fresh `lastProgressAgeMs` when `activeWorkKind` is absent, which yields a recoverable stuck classification for the reported classifier input. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the classifier is the narrowest owner of the bad decision because it already receives `lastProgressAgeMs` and `staleMs`, and stale/missing progress still falls through to the existing stuck recovery path.
