## Summary

Keep open. The product direction is useful and current main does not fully replace the separate chat agent selector, but this PR is still not mergeable because it would regress agent session key synthesis and session-advertised thinking choices.

## What This Changes

This PR adds a Control UI chat agent selector, moves model and thinking selectors into the shared chat controls row, and adds CSS sizing for the new selector.

## Best Possible Solution

Rebase or rewrite the PR so the agent selector is implemented through the current shared chat session-control path, derives agent main-session targets from a plain main key, preserves the existing model/thinking metadata contract, and adds focused tests for legacy non-agent sessions plus dynamic thinking options.

## Reproduction Assessment

Yes. A static source reproduction is enough: a non-agent active key plus sessionDefaults.mainSessionKey can produce a nested agent session key, and session/default thinking metadata such as adaptive, xhigh, or max is dropped by the PR's rebuilt thinking selector.

## Solution Assessment

No. The requested UI split is reasonable, but the current PR duplicates older model/thinking logic and builds agent target keys from the wrong fallback; extending the shared session-control helper is the narrower maintainable fix.
