## Summary

Keep open. Current main still has only adjacent mitigations for this failure: session-scoped nested lanes, Telegram polling watchdog/status diagnostics, channel health monitoring, and conservative stuck-session recovery. Those paths depend on timers or intentionally avoid aborting active work, while the report describes all Telegram pollers and timer-driven logs going silent together after nested subagent activity.

## What This Changes

The issue reports that nested `sessions_send` subagent activity can leave the gateway process alive while all configured Telegram polling and health-monitor logging stop until restart.

## Best Possible Solution

Reproduce the scenario on current main with multiple Telegram bot accounts and a coordinator-to-subagent `sessions_send`, capture logs plus a diagnostics/stability export, then land a focused liveness or recovery change that proves Telegram polling and diagnostic timers continue after nested A2A completion or escalates to a bounded restart when they do not.

## Solution Assessment
