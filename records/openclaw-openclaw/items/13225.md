## Summary

Current main still lacks the requested routing capability, and the closest implementation is an open PR with unresolved auth, fallback, context-window, security, and proof gaps. This issue should remain open as the product and plugin-API design anchor.

## What This Changes

Requests an additive pre-tool-call hook contract that can inspect assistant context and redirect pending tool execution to another model or session.

## Best Possible Solution

Add one generic, typed routing decision at the model-attempt or agent-loop boundary with explicit trust, audit, provider/auth, context, fallback, and session-continuation semantics; keep task classification and cost policy in plugins.

## Reproduction Assessment

Not applicable: this is a new capability request, and current source inspection conclusively shows that the existing hook cannot redirect a pending tool turn.

## Solution Assessment

No, adding `redirect` directly to the existing tool-policy result is not yet the best demonstrated design; model/session continuation belongs at the agent-loop attempt boundary and needs explicit cross-provider, auth, fallback, and transcript invariants.

## Risks / Open Questions

- A redirect contract could let installed plugins change provider credentials, context budgeting, fallback chains, permissions, and session ownership after the assistant has emitted a tool call.
- Cross-model continuation can corrupt or reject transcript and tool-call state when providers have different tool schemas, reasoning metadata, context windows, or replay requirements.
- Redirecting to an existing or isolated session needs explicit audit, cancellation, approval, error, and delivery semantics to avoid mis-associated session state.
