## Label Justifications

- `P2`: This is a concrete enhancement to plugin/runtime model routing with real user demand, but it is not an urgent regression or outage.
- `impact:session-state`: The requested redirect can move a pending tool call across sessions, so transcript and session association semantics are central to the issue.
- `impact:security`: Pre-tool routing and policy enforcement affect who may redirect execution and how sensitive tool actions are audited or blocked.
- `impact:auth-provider`: Routing a tool call to another model/provider can change model choice, auth profile resolution, and provider credential boundaries.

## Summary

Keep open: current main still has a pre-tool hook for parameter rewrite, blocking, and approval, but not the requested contract for inspecting pending assistant context and redirecting a tool call to another model or session. The closest implementation candidate is still open and does not supersede this plugin/runtime API design request.

## What This Changes

Request to extend `before_tool_call` or add a pre-tool hook that can inspect preceding model output and route a pending tool call to another model or session.

## Best Possible Solution

Keep this as a plugin/runtime API design tracker for an additive typed pre-tool routing contract with explicit context, redirect, audit, permission, and failure behavior while preserving current params/block/approval semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source inspection gives high confidence that current main lacks the requested redirect/session-routing fields and runtime behavior.

## Solution Assessment

Unclear pending maintainer decision. The maintainable direction is likely an additive typed routing hook or carefully scoped extension to `before_tool_call`, not conflating it with prompt-time `before_model_resolve` or the separate internal-HOOK.md bridge.
