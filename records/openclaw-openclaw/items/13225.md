## Summary

Keep open: current main and v2026.6.6 still expose `before_tool_call` as a params/block/approval hook, not a typed pre-tool-call model/session routing contract. Related PRs and issues cover adjacent context fields, prompt-time routing, or per-tool config ideas, but none supersede this plugin/runtime API design request.

## What This Changes

The issue requests a pre-tool-call hook that can inspect preceding assistant output and redirect a pending tool call to another model or session.

## Best Possible Solution

Define an additive typed pre-continuation routing contract after maintainer product and security review, while preserving the existing `before_tool_call` params/block/approval behavior for policy and argument-injection use cases.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main and v2026.6.6 lack the requested redirect contract.

## Solution Assessment

No for the literal `before_tool_call` extension as sketched. The safer maintainable path is a typed routing phase with explicit audit, permission, provider/auth, fallback, context-budget, and session-state semantics.
