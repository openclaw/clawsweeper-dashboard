## Summary

Keep this PR open: the core normalization fix is useful, but the current head introduces a user-visible count regression in `openclaw tasks maintenance` by using the alias-preservation set as the reported running-job count.

## What This Changes

The PR changes task maintenance to preserve running cron-run session rows for non-slug cron job ids by matching both slugified main-session and raw lowercased isolated session-key job segments, with regression tests and a contract-comment update.

## Best Possible Solution

Keep the dual preservation keys, but track and report the actual number of running cron jobs separately from the preservation-key aliases, with a regression assertion for non-slug ids.

## Reproduction Assessment

Yes for source-level reproduction. Current main can miss non-slug cron-run session rows, and the PR-introduced count bug follows directly because one running non-slug job contributes two aliases while `runningCronJobs` reports set size.

## Solution Assessment

No, not on the current head. The normalization location is the right boundary, but the implementation should separate preservation aliases from the actual running-job count before merge.
