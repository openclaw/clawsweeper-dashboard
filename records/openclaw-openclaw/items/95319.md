## Summary

Keep this PR open: current main still has the lowercase-only running cron id set, while cron-run session keys use the slugified session-key segment; the branch applies the narrow producer-side fix and I found no blocking correctness or security finding.

## What This Changes

The PR changes task maintenance to build running cron job ids with `normalizeCronLaneSegment(job.id, "job")`, updates the session-maintenance contract comment, and adds a regression test for a non-slug cron job id.

## Best Possible Solution

Land the narrow normalization fix after normal CI so session-registry pruning uses the existing cron-run session-key contract instead of a second lowercase-only contract.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a running persisted cron job id `Daily Report` enters the running set as `daily report`, while the cron-run session key is parsed as `daily-report`, so stale-session pruning fails to preserve it.

## Solution Assessment

Yes. Reusing `normalizeCronLaneSegment` at the producer side is the narrowest maintainable fix; changing the parser or adding a fallback would create a second normalization contract.
