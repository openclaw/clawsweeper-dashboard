## Summary

Keep open. The branch fixes the renderer only when it is handed `activeModelAuth: "oauth"`, but the normal `/status` path can still resolve both selected and active `claude-cli` labels as the aliased env API key before the renderer runs.

## What This Changes

The branch changes `/status` rendering to prefer active OAuth over selected API-key for equivalent runtime aliases, hides cost in that case, adds a renderer regression test, and adds a changelog entry.

## Best Possible Solution

Land one canonical fix that makes the full `/status` path report active CLI OAuth when an env API key is present, with coverage through `buildStatusText` rather than only the lower-level renderer.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve `/status` for an Anthropic model routed through `claude-cli` while `ANTHROPIC_API_KEY` is present, and the auth-label resolver can return API-key before cached Claude CLI OAuth is considered.

## Solution Assessment

No. The renderer override is too late when `buildStatusText` has already resolved both selected and active auth labels as API-key; the fix should adjust the upstream auth choice or pass the active CLI OAuth label into the renderer with high-level coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Fix the upstream auth-label path:** `src/status/status-message.ts:870-879`

With both args arriving as API-key, the reported label can remain wrong; cover the `buildStatusText` path or fix the resolver/selection upstream.
