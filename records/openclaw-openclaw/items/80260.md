## Summary

Keep open. The PR is a focused fix for a source-backed status-label bug and the supplied live renderer output is enough real behavior proof, but the added regression test does not actually force the selected auth path to be api-key, so it should be tightened before merge.

## What This Changes

The branch updates `/status` rendering to prefer active OAuth over selected env API-key for equivalent runtime aliases, suppresses cost display in that case, adds a status regression, and adds a changelog entry.

## Best Possible Solution

Land the renderer fix after tightening the regression so it fails on current main for selected api-key plus active claude-cli OAuth, while keeping the fix scoped to status display and cost visibility.

## Reproduction Assessment

Yes. Source inspection on current main shows a selected Anthropic env api-key label is treated as concrete and can outrank the active OAuth label for an equivalent claude-cli runtime alias; I did not run a live gateway OAuth session in this read-only review.

## Solution Assessment

Mostly yes. Updating `buildStatusMessage` is the narrow rendering fix for label and cost output, but the regression should explicitly create the api-key-selected/OAuth-active state instead of relying on the existing unknown-auth fallback.
