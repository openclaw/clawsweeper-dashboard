## Summary

Keep open: the renderer fix is narrow and points at the right status path, but the submitted cost-regression/proof setup does not actually exercise `claude-cli` pricing because it uses the unmatched provider key `claudecli`. ## What This Changes

The PR updates `/status` rendering, a focused status test, and the changelog so equivalent `claude-cli` runtime aliases prefer the active OAuth label and do not show API-key cost when the selected auth was only an env key.

## Best Possible Solution

Keep the narrow renderer fix, but update the regression/proof to configure real `claude-cli` pricing and show that direct API-key status still emits cost while the alias OAuth case does not.

## Reproduction Assessment

Yes, from source inspection: call `buildStatusMessage` with selected `anthropic/<model>`, active `claude-cli/<model>`, `modelAuth: "api-key"`, and `activeModelAuth: "oauth"`; current main prefers the selected auth label and cost mode. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes: changing the status renderer at the final selected-vs-active display decision is the narrow maintainable path, but the test/proof should use the real `claude-cli` provider key so the cost assertion proves the intended behavior.
