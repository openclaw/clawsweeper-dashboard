## Summary

Keep open. The branch improves the renderer for a synthetic selected-api-key plus active-OAuth case, but the reported `/status` path can still resolve the active `claude-cli` auth label as `api-key` before this renderer fallback sees OAuth, so the linked bug is not fully fixed yet.

## What This Changes

The PR changes status message rendering and a regression test so equivalent runtime aliases can prefer an active OAuth auth label over a selected env API-key label, with a changelog entry.

## Best Possible Solution

Fix the full `/status` auth-resolution path so `claude-cli` OAuth wins over aliased env API keys before rendering, keep cost suppression consistent, and cover it with a high-level status regression plus redacted real proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run the full status path with selected `anthropic/<model>`, active `claude-cli/<model>`, valid Claude CLI OAuth, and `ANTHROPIC_API_KEY` present. The resolver can return `api-key (...)` for the active alias before the PR's renderer condition ever sees OAuth.

## Solution Assessment

No. The renderer fallback is useful but not sufficient because it assumes `activeModelAuth` is already OAuth; the narrower maintainable fix is to correct the resolver/status-text path and then let the renderer hide cost from the effective active auth.
