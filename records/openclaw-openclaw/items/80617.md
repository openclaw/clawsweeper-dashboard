## Summary

Keep open: current main still has a source-visible drift between the primary reply path and queued follow-up delivery. The Slack routed-thread fallback may already protect some threaded Slack follow-ups, but `resolveFollowupDeliveryPayloads` still cannot apply implicit `currentMessageId` threading itself.

## What This Changes

The issue requests consistent implicit reply-to-current threading for Slack/channel mention follow-up deliveries so replies stay in the originating thread without requiring an explicit directive.

## Best Possible Solution

Carry the existing inbound current-message and reply-threading policy through queued follow-up delivery, reuse `applyReplyThreading`, and leave any new top-level opt-out directive for a separate product decision.

## Reproduction Assessment

No live high-confidence Slack reproduction is available from this review. Source inspection does reproduce the delivery-path inconsistency: follow-up delivery cannot set implicit `replyToId` because it never receives `currentMessageId` or `replyThreading`.

## Solution Assessment

Yes, the narrowest maintainable fix is to pass the existing threading context into the follow-up path rather than adding a new directive or changing Slack routing policy.
