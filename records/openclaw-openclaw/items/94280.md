## Label Justifications

- `P2`: This is a concrete screenshot-delivery bug with limited blast radius in the nodes/agent tool-result path.
- `impact:message-loss`: The reported truncation corrupts screenshot media payloads so downstream send/write/save operations cannot deliver the intended image.

## Summary

Keep open: the shipped release still has the reported raw `nodes.invoke screen.snapshot` truncation shape, while current main only partially routes the default agent path to a dedicated file-output `screen_snapshot` action and the issue is paired with open closing PRs that need review or intentional supersession.

## What This Changes

The issue asks OpenClaw to prevent inline base64 screenshot payloads from being truncated when `nodes invoke screen.snapshot` is used, so screenshots remain usable for saving or sending downstream.

## Best Possible Solution

Resolve this through one maintainer-approved path: either land a compatible raw-invoke sanitizer or declare the dedicated `screen_snapshot` file-output action the supported fix, then close the linked PRs or this issue accordingly.

## Reproduction Assessment

Yes at source/release level, though I did not run a live Windows node. The latest release still returns raw generic invoke JSON with a 16K live tool-result cap, and current main preserves that raw path when media invoke is explicitly enabled.

## Solution Assessment

Unclear as a product choice. The dedicated `screen_snapshot` file-output path is the safer existing direction, while changing generic raw invoke output needs an explicit compatibility decision.
