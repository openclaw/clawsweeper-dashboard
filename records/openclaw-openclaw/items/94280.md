## Summary

Keep open: current main and v2026.6.9 provide the safer file-backed `screen_snapshot` action, but the raw `nodes invoke screen.snapshot` behavior that the report targets is still an explicit compatibility question and is paired with an open same-author closing PR.

## What This Changes

The issue asks for `nodes invoke screen.snapshot` screenshot output to remain usable instead of truncating inline base64 before downstream send, write, or save operations.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/94281 first by either landing a compatible, proven raw-invoke fix or explicitly closing it as superseded by the shipped `screen_snapshot` path, then close this issue with that rationale.

## Reproduction Assessment

Yes at source level, but not from a live Windows node in this read-only review. Current main still has a 16,000-character default tool-result cap and still preserves raw JSON for explicitly enabled media invoke, while the default path redirects users to `screen_snapshot`.

## Solution Assessment

No final solution decision yet. The dedicated `screen_snapshot` action is the safest shipped default path, but changing generic raw invoke output needs an explicit compatibility decision through the linked PR.
