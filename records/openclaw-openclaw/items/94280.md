## Summary

Current main already fixes the reported screenshot delivery problem by routing `screen.snapshot` through the file-backed `screen_snapshot` action and blocking raw media invoke output by default; the fix is on main but not in `v2026.6.8`.

## What This Changes

The issue reports that `nodes invoke screen.snapshot` returns inline base64 that gets truncated before it can be saved or sent as a screenshot.

## Best Possible Solution

Keep the current main behavior: screenshots should go through the dedicated file-backed `screen_snapshot` action, while any broader generic invoke sanitizer should be reviewed separately from this already-fixed report.

## Reproduction Assessment

No against current main: raw `screen.snapshot` media invokes are blocked before Gateway output, and the dedicated `screen_snapshot` path writes the image to a file. The older release behavior in the report is credible, but it is not the current-main behavior.

## Solution Assessment

Yes. The merged file-backed `screen_snapshot` action is the narrow owner-boundary fix for screenshot delivery; a generic invoke sanitizer would be broader and belongs in a separate PR review if maintainers still want it.
