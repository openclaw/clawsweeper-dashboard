## Summary

Keep open: current main has a dedicated file-backed `screen_snapshot` action and blocks raw media invoke by default, but the explicitly enabled raw invoke path still returns raw JSON and the latest release still has the 16K live tool-result cap. The issue is also paired with the open closing PR at https://github.com/openclaw/openclaw/pull/94281, so it should stay open until that PR lands or maintainers intentionally choose a different fix path.

## What This Changes

The issue asks for `nodes invoke screen.snapshot` screenshot data to remain accessible instead of being truncated, likely through file-backed output or another bounded media-delivery path.

## Best Possible Solution

Resolve this through the linked PR at https://github.com/openclaw/openclaw/pull/94281 or an explicit maintainer decision that `screen_snapshot` is the supported path, then close this issue and clean up duplicate fix attempts.

## Reproduction Assessment

Yes at source level, though this review did not run a live Windows node. Current main preserves a raw JSON media invoke path behind `allowMediaInvokeCommands`, and the latest release/default cap remains 16K, matching the reported truncated base64 payload.

## Solution Assessment

Unclear as a product choice. The dedicated `screen_snapshot` file-output action is the safest existing direction, while changing explicitly enabled raw invoke output needs maintainer approval or the linked PR repaired around compatibility.
