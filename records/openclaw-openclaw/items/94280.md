## Summary

Keep open: current main and v2026.6.10 provide the safer file-backed `screen_snapshot` path and block raw media invoke by default, but explicitly enabled raw invoke still returns raw JSON under the 16,000-character cap and the same-author closing PR remains open for that compatibility-sensitive behavior.

## What This Changes

The issue reports that `nodes invoke screen.snapshot` can return inline base64 screenshot data that is truncated before it can be reused or delivered.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/94281 first: either land a compatible, proven raw-invoke fix or close that PR as superseded by the shipped `screen_snapshot` file-output path, then close this issue with that rationale.

## Reproduction Assessment

Yes at source level: explicitly enabled raw `screen.snapshot` invoke still returns raw JSON while the default live result cap is 16,000 characters. I did not run a live Windows node in this read-only review.

## Solution Assessment

No final solution decision yet. The dedicated `screen_snapshot` action is the safer shipped path, but changing or replacing generic raw invoke output needs the linked PR or a maintainer compatibility decision.
