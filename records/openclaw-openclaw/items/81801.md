## Summary

Keep open: the report is recent, detailed, and points at a real supported `bundle-mcp` streamable-http path that current main still runs without a transport-owned undici dispatcher or pool lifecycle, while the cited Telegram and Discord fixes were subsystem-specific.

## What This Changes

The issue reports intermittent `bundle-mcp` streamable-http remote MCP failures with `Unexpected content type: null`, suspected from stale undici dispatcher pool reuse after NAT or idle connection drops.

## Best Possible Solution

Add a narrow bundle-mcp streamable-http transport-owned undici dispatcher lifecycle using existing HTTP/1 bounded helper patterns, preserve the current redirect/header hardening, and cover it with focused regression tests plus a changelog fix entry.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The report gives a credible NAT/Node/undici reproduction path, and source inspection confirms the implicated streamable-http path currently lacks an MCP-owned dispatcher or pool lifecycle.

## Solution Assessment

Yes, the likely best fix is a narrow transport-owned dispatcher/lifecycle change through the SDK fetch hook rather than a broad global dispatcher reset or a new user-facing config knob. It should reuse existing OpenClaw undici helper patterns and avoid disturbing SSE, stdio, OAuth, or redirect-scrubbing behavior.
