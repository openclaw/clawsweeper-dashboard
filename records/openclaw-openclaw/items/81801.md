## Summary

Current main now routes bundle MCP `streamable-http` through OpenClaw's package-undici fetch path with `undici@8.2.0`, and the same fix is present in the current beta release; the reporter's follow-up also verified that moving the same runtime path to Undici 8.2.0 eliminated the production failures.

## What This Changes

The issue asks OpenClaw to prevent bundle MCP `streamable-http` reconnect failures caused by stale Undici dispatcher/pool reuse behind NAT.

## Best Possible Solution

Keep the current package-undici-backed Streamable HTTP transport path and only add per-server dispatcher/keepalive config if fresh failures appear on the fixed current-main or beta line.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. The original production logs were detailed, but current main no longer uses the affected SDK-default/global fetch path for `streamable-http`, and the reporter's same-environment Undici 8.2.0 run saw zero failures.

## Solution Assessment

Yes. Routing the SDK transport through OpenClaw's package Undici 8.2 fetch is the narrowest maintainable fix for the observed stale-pool failure, while a new dispatcher/keepalive config knob would be broader fallback work only if the bug recurs on the fixed line.
