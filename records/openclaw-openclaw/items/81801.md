## Summary

Keep open: current main still routes bundle-mcp Streamable HTTP through the MCP SDK without an owned dispatcher lifecycle, while the documented transport is expected to work for remote MCP servers and the related Telegram/Discord hardening was subsystem-specific.

## What This Changes

Report requests hardening bundle-mcp Streamable HTTP so remote MCP servers do not cascade-fail on stale undici dispatcher pools after NAT or HTTP/2 connection teardown.

## Best Possible Solution

Add bundle-mcp-owned Streamable HTTP dispatcher hardening with bounded HTTP/1 pool behavior and lifecycle cleanup tied to the session runtime; add a config knob only if the safe default proves incompatible.

## Reproduction Assessment

No, not a high-confidence live reproduction. The production logs and source path are detailed, but this review did not recreate the 16-22 minute NAT/gateway runtime failure with many remote MCP servers on current main.

## Solution Assessment

Yes, the narrow maintainable direction is to harden the bundle-mcp Streamable HTTP transport itself, not to rely on Telegram/Discord-specific fixes or a config-first workaround.
