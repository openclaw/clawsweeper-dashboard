## Summary

Keep open: current main still routes bundle-mcp Streamable HTTP through the MCP SDK with an undici fetch wrapper but no bundle-mcp-owned dispatcher or keep-alive lifecycle, so the documented remote transport lacks the subsystem hardening used by Telegram and Discord paths.

## What This Changes

The issue reports intermittent bundle-mcp Streamable HTTP remote MCP failures where long-running gateway processes receive `Unexpected content type: null` after apparent stale undici dispatcher reuse.

## Best Possible Solution

Add bundle-mcp-owned Streamable HTTP dispatcher hardening with bounded HTTP/1 pool behavior and lifecycle cleanup tied to session disposal; keep a config knob out unless compatibility testing proves it is needed.

## Reproduction Assessment

No, not at high confidence. The report has detailed production logs and current source still shows the unowned dispatcher path, but I did not recreate the 16-22 minute NAT/gateway failure on current main.

## Solution Assessment

Yes, the narrow maintainable direction is to harden the bundle-mcp Streamable HTTP transport itself and close its dispatcher during existing session cleanup, rather than relying on Telegram/Discord-specific fixes or adding a config-first workaround.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81801. Current main builds `StreamableHTTPClientTransport` in `src/agents/mcp-transport.ts` with a custom redirect-scrubbing fetch but no bundle-mcp-owned undici dispatcher or keep-alive policy. Add streamable-http dispatcher hardening in the bundle-mcp path: create a per-server direct/env-proxy-aware HTTP/1 dispatcher using existing undici runtime helpers or a narrow helper, attach it through `requestInit.dispatcher` so SDK GET/POST/DELETE calls use it, preserve configured headers and redirect scrub behavior, and expose cleanup so `src/agents/pi-bundle-mcp-runtime.ts` closes/destroys the dispatcher when a server fails to start or the session runtime disposes. Add focused regression tests proving the dispatcher is passed, bounded pool options are applied, and cleanup runs.
