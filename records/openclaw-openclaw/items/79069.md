## Summary

Keep open: current main still drops the no-delivery cron state before the embedded runner call, so the reported empty-assistant false-positive path remains source-reproducible.

## What This Changes

The issue asks for `delivery.mode: "none"` cron agent-turn runs to pass `allowEmptyAssistantReplyAsSilent` into `runEmbeddedPiAgent`.

## Best Possible Solution

Thread an already-resolved `deliveryPlan.mode === "none"` boolean through the cron executor into `runEmbeddedPiAgent`, and cover the cron call-site propagation without relaxing incomplete-turn handling globally.

## Reproduction Assessment

Yes, source-level. Current main resolves `delivery.mode: "none"` before execution, but the embedded cron call receives no silent-empty allowance and the embedded classifier treats the omitted option as false.

## Solution Assessment

Yes. The narrow maintainable solution is to propagate the already-resolved no-delivery state to the embedded runner, not to change the embedded incomplete-turn classifier for all callers.

Fix https://github.com/openclaw/openclaw/issues/79069. Current main resolves cron `delivery.mode: "none"`, but `createCronPromptExecutor` drops that state before calling `runEmbeddedPiAgent`, so clean empty assistant stops still hit `incomplete turn detected`. Thread a boolean based on the resolved cron delivery plan, preferably `deliveryPlan.mode === "none"`, from `run.ts`/`executeCronRun` into `createCronPromptExecutor` and pass `allowEmptyAssistantReplyAsSilent` to `runEmbeddedPiAgent` only for no-delivery cron jobs. Keep `src/agents/pi-embedded-runner/run/incomplete-turn.ts` strict for callers that do not opt in; do not relax global incomplete-turn classification, do not change delivery target resolution policy, and do not edit generated bundles.
