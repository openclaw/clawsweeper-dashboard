## Summary

Keep open: current main and v2026.6.11 still omit the cron silent-empty opt-in at both isolated cron runner handoffs, and the direct closing PR remains open and conflicting rather than merged.

## What This Changes

The issue reports that isolated no-delivery cron jobs can classify intentionally empty assistant replies as incomplete or failed because the cron embedded-runner path does not pass `allowEmptyAssistantReplyAsSilent`.

## Best Possible Solution

Land one canonical cron executor-boundary fix that covers the no-delivery embedded path and explicitly tests the post-tool empty-final case, while separately deciding the broader all-cron policy tracked by the related issue.

## Reproduction Assessment

Yes, source-level: current main sends isolated cron runs into `runCliAgent` and `runEmbeddedAgent` without `allowEmptyAssistantReplyAsSilent`, while both runner contracts require that opt-in for empty assistant output. I did not run a live scheduled cron tick in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. Passing the existing flag at the cron executor boundary is the right layer, but the open branches differ on no-delivery-only versus all-cron behavior and on the post-tool empty-final case.
