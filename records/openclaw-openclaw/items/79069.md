## Summary

Keep open. Current main and v2026.6.6 still omit the silent-reply flag at the cron embedded-run handoff, while the specific open fixing PR is unmerged and currently conflicting, so this issue should continue tracking the fix until a refreshed PR lands.

## What This Changes

No-delivery isolated cron agent turns should forward silent-reply allowance into the embedded runner so empty final assistant replies can be treated as silent success instead of terminal errors.

## Best Possible Solution

Refresh or replace the linked fix with a current-main-compatible cron executor change plus regression coverage for `delivery.mode: "none"` empty-final success, while keeping the broader accept-silent-stop feature separate in https://github.com/openclaw/openclaw/issues/76159.

## Reproduction Assessment

Yes, at source level: current main routes `delivery.mode: "none"` cron embedded-provider runs through `run-executor.ts` without the silent-reply flag, while the embedded runner has a flag-gated silent-empty path. I did not execute a live model repro in this read-only sweep.

## Solution Assessment

Mostly yes: forwarding the existing silent-reply flag at the cron executor boundary is the right owner layer for this narrow bug. The final PR should also prove the reported post-tool empty-final case because current incomplete-turn logic treats some post-tool empty stops as ambiguous.
