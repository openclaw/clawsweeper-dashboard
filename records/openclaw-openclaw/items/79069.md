## Summary

Keep open: the reported bug still exists on current main. The cron path computes no-delivery state but still does not pass the embedded-runner silent-empty option, and the only matching fix PR is closed unmerged.

## What This Changes

Silent cron jobs with `delivery.mode: "none"` should pass `allowEmptyAssistantReplyAsSilent` into `runEmbeddedPiAgent` so clean empty assistant stops are treated as silent success.

## Best Possible Solution

Thread the resolved no-delivery state into the existing embedded-runner option and cover the cron call site, while keeping incomplete-turn classification strict for visible delivery modes.

## Reproduction Assessment

Yes, source-level. `delivery.mode: "none"` is resolved before execution, but the cron embedded-runner call drops that state, and the embedded classifier treats an omitted `allowEmptyAssistantReplyAsSilent` as false.

## Solution Assessment

Yes. The narrow maintainable fix is to pass a resolved no-delivery boolean to the existing `runEmbeddedPiAgent` option instead of relaxing incomplete-turn handling globally.
