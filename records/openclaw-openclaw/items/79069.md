## Summary

Keep open: current main still drops the no-delivery cron state before the embedded runner, so empty assistant stops for `delivery.mode: "none"` can still be classified as incomplete turns. The closed related PR matches the right fix shape but was not merged.

## What This Changes

Cron `agentTurn` jobs with `delivery.mode: "none"` should pass the embedded-runner silent-empty flag so intentionally empty final assistant replies complete quietly.

## Best Possible Solution

Thread the already-resolved no-delivery state into the embedded runner option and add focused regression coverage for `none` and visible delivery modes without weakening global incomplete-turn detection.

## Reproduction Assessment

Yes, source-level. Current main resolves `delivery.mode: "none"`, but the embedded cron call omits the flag and the embedded classifier treats an omitted `allowEmptyAssistantReplyAsSilent` as false.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the existing no-delivery state to `runEmbeddedPiAgent`, rather than changing the incomplete-turn classifier globally.
