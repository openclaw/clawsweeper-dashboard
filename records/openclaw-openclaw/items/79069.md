## What This Changes

The issue requests threading `allowEmptyAssistantReplyAsSilent` from silent cron delivery mode into the embedded agent runner.

## Best Possible Solution

Thread a resolved `deliveryPlan.mode === "none"` boolean through the cron executor into `runEmbeddedPiAgent`, and cover the call-site behavior with a focused cron regression test.

Yes, source-level. `delivery.mode: "none"` is resolved before execution, but that mode is dropped before the embedded runner call, and the embedded classifier treats an omitted `allowEmptyAssistantReplyAsSilent` as false.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the already-resolved cron no-delivery state to the embedded runner instead of relaxing incomplete-turn classification globally.

For isolated cron `agentTurn` jobs whose resolved delivery mode is `none`, pass `allowEmptyAssistantReplyAsSilent: true` into the embedded `runEmbeddedPiAgent` call so a clean empty assistant final is treated as silent success. Prefer threading the already-resolved `deliveryPlan.mode === "none"` state from `runCronIsolatedAgentTurn` through `executeCronRun`/`createCronPromptExecutor`; do not loosen the generic embedded incomplete-turn classifier and do not change announce/webhook behavior.
