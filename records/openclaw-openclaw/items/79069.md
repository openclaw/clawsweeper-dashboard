## What This Changes

The issue asks for isolated cron `delivery.mode: "none"` agent jobs to pass the existing silent-empty runner option so intentionally empty assistant replies do not become cron errors.

## Best Possible Solution

Land one canonical cron fix that passes the existing silent-empty option from the cron executor into both CLI and embedded runner paths for intended silent cron contexts, with regression coverage that preserves genuine empty-response failures.

## Reproduction Assessment

Yes. Source inspection shows a no-delivery isolated cron run reaches the CLI or embedded runner without the silent-empty option, while the runners only accept empty replies when that option is explicitly true; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear as currently proposed. Passing the existing flag from the cron executor is the right owner boundary, but the final implementation must prove the reported post-tool empty-final shape without weakening real incomplete-turn detection.
