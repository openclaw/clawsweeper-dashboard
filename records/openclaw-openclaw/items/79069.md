## Summary

Keep open: current main still does not thread the existing silent-empty runner option from isolated cron into `runEmbeddedAgent`, and the only exact linked fix PR is still open and conflicting. The report is source-reproducible, but the proposed one-line fix should be validated against the reported post-tool empty-final shape because the embedded runner separately guards some side-effect/tool cases.

## What This Changes

The issue requests that no-delivery isolated cron jobs pass `allowEmptyAssistantReplyAsSilent` into the embedded agent runner so empty final assistant replies are treated as silent success.

## Best Possible Solution

Land a focused cron regression fix that threads the existing silent-empty contract for `delivery.mode: "none"` where safe and proves the reported no-delivery empty terminal run records success without weakening real incomplete-turn detection.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run an isolated cron `agentTurn` with `delivery.mode: "none"` and an empty final assistant reply, and current main will not pass the silent-empty option into the embedded runner. I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. Passing the flag through is the right seam for clean no-delivery empty replies, but the linked fix needs validation against the reported post-tool side-effecting shape because current runner code intentionally excludes some side-effect/tool empty turns from silent handling.
