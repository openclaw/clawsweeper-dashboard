## Summary

Keep open: current main still drops the silent-empty flag before the embedded cron runner call, while the embedded runner only accepts empty assistant stops when that flag is explicitly set. The closed related PR attempted the right narrow fix but was not merged, so the bug remains present on current main.

## What This Changes

Cron jobs with `delivery.mode === "none"` should pass `allowEmptyAssistantReplyAsSilent` into `runEmbeddedPiAgent` so intentional empty assistant replies are treated as silent success.

## Best Possible Solution

Thread a resolved `deliveryPlan.mode === "none"` boolean through `executeCronRun` and `createCronPromptExecutor` into `runEmbeddedPiAgent`, then cover `none` and visible delivery modes with focused cron regression tests.

## Reproduction Assessment

Yes, source-level. Current main resolves no-delivery mode before execution, drops that mode before the embedded runner call, and the embedded classifier treats an omitted `allowEmptyAssistantReplyAsSilent` as false.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the already-resolved cron no-delivery state to the existing embedded-runner option rather than relaxing incomplete-turn handling globally or adding a new public config surface.
