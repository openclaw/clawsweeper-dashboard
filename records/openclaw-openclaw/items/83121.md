## Summary

Current main already implements the reported Telegram `/models` path: authenticated CLI runtime providers such as `claude-cli` are now surfaced as selectable provider rows backed by the provider-owned catalog, and the merged fix has release provenance in the beta line. The reporter’s 2026.5.12 observation is consistent with an older stable release that predates the verified fixed release line.

## What This Changes

Telegram `/models` should let users discover and select Claude CLI runtime-backed model lanes after the provider/runtime split.

## Best Possible Solution

Keep the merged `/models` picker/catalog implementation; users on 2026.5.12 should move to a fixed build, while broader catalog-refresh automation remains separately tracked at https://github.com/openclaw/openclaw/issues/81216.

## Reproduction Assessment

No for current main. The old failure is explainable on the 2026.5.12 release line, but current main source now routes authenticated `claude-cli` catalog rows into `/models` and Telegram renders the shared provider list.

## Solution Assessment

Yes. The landed fix is a narrow maintainable solution for the reported discoverability break: `/models` can surface authenticated CLI runtime provider rows without reopening global legacy-alias visibility, while richer runtime-selector UX can remain a separate product discussion.
