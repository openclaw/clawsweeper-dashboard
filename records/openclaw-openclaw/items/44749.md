## Label Justifications

- `P1`: Normal parallel agent tool calls can silently lose durable `allow-always` decisions and repeatedly break a core approval workflow.
- `impact:data-loss`: Concurrent whole-file writes can silently discard user-approved allowlist entries.
- `impact:security`: The corrupted file is the persistent authorization boundary for future command execution.

## Summary

Current main still writes the complete approvals file from request-local snapshots, and both gateway and node-host approval flows use this path. The only focused repair PR closed without merging, so the reported lost-update race remains actionable.

## What This Changes

Fix concurrent `allow-always` approval persistence so every approved command remains in the exec allowlist.

## Best Possible Solution

Put every allowlist mutation behind one cross-process serialized transaction that reloads the latest state under the existing file lock, applies all entries for one approval decision, atomically saves once, and has a deterministic competing-writer regression test.

## Reproduction Assessment

Yes at source level. The report supplies repeatable concurrent approval steps, and current main still allows independently loaded request snapshots to reach whole-file saves; a focused two-snapshot test can reproduce the lost update deterministically.

## Solution Assessment

No, an unlocked re-read only narrows the race, and an in-process mutex does not cover separate writers. The best fix is a shared cross-process lock around fresh read, batched mutation, and atomic write.
