## Label Justifications

- `impact:data-loss`: The reported failure silently drops persisted allowlist entries from `exec-approvals.json`.
- `impact:security`: The affected persisted file controls whether future host exec commands require operator approval.

## Summary

Keep open: current `main` still writes allow-always and allowlist-use updates from caller-held `ExecApprovalsFile` snapshots, so concurrent approval handlers can clobber each other's persisted trust entries. The earlier same-scope repair PR is closed unmerged, and the current adjacent command-planner PRs do not replace the missing serialized read-modify-write fix.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can silently drop earlier entries from `exec-approvals.json` through last-write-wins persistence.

## Best Possible Solution

Adopt a store-level serialized read-modify-write helper, likely using the existing `withFileLock` surface, so gateway and node-host allow-always and usage updates reload the latest approvals file before saving and preserve all concurrent entries.

## Reproduction Assessment

Yes for source reproduction: the current gateway and node-host paths can each hold an old `approvals.file`, then `addAllowlistEntry` or `recordAllowlistUse` writes the whole stale object. I did not execute a live concurrent approval run in this read-only review.

## Solution Assessment

Yes, the best fix direction is the reporter's store-boundary lock/reload shape rather than a caller-side workaround or external backup routine. The closed unmerged repair PR attempted that shape, but current `main` still needs an adopted implementation and security review.
