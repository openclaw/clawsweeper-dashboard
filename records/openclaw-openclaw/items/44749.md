## Summary

Keep open: current `main` and the latest release still persist `allow-always` and allowlist-use updates from stale exec approval snapshots. The same-scope repair at https://github.com/openclaw/openclaw/pull/52403 is closed unmerged, so no implemented or viable superseding path currently owns the fix.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can overwrite each other in `exec-approvals.json`, silently losing allowlist entries.

## Best Possible Solution

Adopt a store-level serialized read-modify-write helper, likely using the existing `withFileLock` surface, so both gateway and node-host approval paths merge `allow-always` and usage updates against the latest approvals file before saving.

## Reproduction Assessment

Yes for source reproduction: each concurrent handler can hold an old `approvals.file` object, then `addAllowlistEntry` mutates that object and `saveExecApprovals` writes the whole file. I did not run a live concurrent approval scenario in this read-only review.

## Solution Assessment

Yes, the right fix direction is the reporter's lock/reload-at-store-boundary shape, not another caller-side workaround. The closed unmerged repair PR attempted that shape, but current `main` still needs an adopted implementation and security review.
