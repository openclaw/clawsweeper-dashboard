## Label Justifications

- `P1`: Normal concurrent agent approval workflows can silently lose persisted allow-always trust entries, breaking a user-facing exec approval workflow.
- `impact:data-loss`: The issue is about silently dropped entries in the persisted exec approvals artifact.
- `impact:security`: The affected artifact controls durable command allowlisting, which is part of OpenClaw's exec security boundary.

## Summary

Keep open: current main and the latest release still mutate and save caller-owned `ExecApprovalsFile` snapshots for allow-always and allowlist-use persistence, so the reported last-write-wins loss remains source-reproducible. The direct fix attempt is closed unmerged, and the adjacent Telegram/node approval report is a distinct issue.

## What This Changes

Report requests fixing a race where concurrent allow-always exec approvals overwrite `exec-approvals.json` so earlier allowlist entries are silently dropped.

## Best Possible Solution

Centralize exec approvals file mutations behind a serialized fresh-read/merge/write helper, then update gateway and node-host callers plus focused concurrent persistence tests.

## Reproduction Assessment

Yes, by source inspection: concurrent handlers can each hold a separately loaded `approvals.file`, then `persistAllowAlwaysPatterns` or `recordAllowlistMatchesUse` saves that stale object back over newer entries. I did not run the live Telegram reproduction in this read-only pass.

## Solution Assessment

Yes on the issue direction, but the safest fix is a central serialized fresh-read/merge/write path rather than a re-read-only helper. The existing closed PR sketched that shape, but it is not merged and should be re-reviewed for the current code.
