## Summary

Keep open: current main still mutates and saves caller-held `ExecApprovalsFile` snapshots for `allow-always` and allowlist usage, so concurrent approval handlers can still overwrite earlier durable allowlist entries. The direct fix attempt is closed unmerged, and the overlapping open security PR is not landed and still has review/proof blockers.

## What This Changes

The issue reports a race where concurrent `allow-always` exec approvals can overwrite each other in `exec-approvals.json`, losing durable allowlist entries.

## Best Possible Solution

Add a single serialized, freshness-guarded exec approvals mutation path that reloads and merges current JSON under a lock or CAS guard, then route gateway and node-host allowlist persistence through it without weakening approval checks.

## Reproduction Assessment

Yes, at source level: two approval handlers can each hold an old approvals object, then the later `addAllowlistEntry` or `recordAllowlistUse` call saves its stale whole-file snapshot over an earlier persisted entry. I did not run a live Telegram approval race in this read-only review.

## Solution Assessment

Yes for the requested direction, but it is not implemented on current main. A locked reload-and-merge or CAS-guarded helper is the narrow maintainable fix; external backup tooling can reduce damage but cannot make approval persistence correct.
