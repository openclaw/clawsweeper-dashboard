## Label Justifications

- `P1`: The issue affects a security-sensitive exec approval workflow and can silently lose persisted user allowlist state during normal concurrent approval use.
- `impact:data-loss`: Concurrent writes can silently drop persisted allowlist entries from `exec-approvals.json`.

## Summary

Keep open: current main and the latest release still persist allow-always and allowlist-use updates by saving a previously resolved approvals object, so concurrent approvals can still overwrite each other; the related fix PR was closed unmerged and the broader open writeback PR is not a clean merged fix path.

## What This Changes

The issue reports concurrent `allow-always` exec approvals silently losing persisted allowlist entries in `exec-approvals.json`.

## Best Possible Solution

Land a security-reviewed core fix that makes allow-always additions and allowlist-use metadata use one locked, fresh merge or CAS write path across gateway and node-host, with regression coverage for concurrent approvals and stale writeback.

## Reproduction Assessment

Yes. Source inspection shows gateway and node-host approval handlers can hold an approvals snapshot loaded before another approval write, then save that stale full object through `persistAllowAlwaysPatterns`, `addAllowlistEntry`, or `recordAllowlistMatchesUse`.

## Solution Assessment

Yes for the direction: this belongs in core exec approval persistence, not in an external backup workaround. The narrowest maintainable fix should serialize or freshness-guard a central read-merge-write helper instead of adding a partial re-read-only mitigation.
