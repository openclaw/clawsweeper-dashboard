## Label Justifications

- `P1`: The issue affects a user-facing exec approval workflow and can silently lose security-state persistence during concurrent approvals.
- `impact:data-loss`: The reported race drops previously approved allowlist entries from persisted exec approval state.
- `impact:security`: The affected file controls allow-always exec approval policy and trusted command persistence.

## Summary

Keep this issue open: current main and the latest release still persist allow-always and allowlist-use updates by mutating a previously resolved approvals object and saving the whole file, so the reported last-write-wins race remains source-reproducible.

## What This Changes

Concurrent allow-always exec approvals can overwrite persisted exec-approvals.json allowlist entries and silently lose earlier approvals.

## Best Possible Solution

Keep the issue open until a maintainer-reviewed security fix serializes exec approval mutations under one fresh-read/hash-guarded store contract, covering allow-always additions, allowlist-use metadata, and initialization writes with focused regression tests.

## Reproduction Assessment

Yes at source level: current main still passes captured approval snapshots into addAllowlistEntry and recordAllowlistUse, then saves the entire file. I did not run a live concurrent approval flow in this read-only review.

## Solution Assessment

Yes, the right boundary is the shared exec approvals persistence path, not an external backup or channel-specific workaround. The implementation should reload and serialize or hash-guard file mutations before writing policy data.
