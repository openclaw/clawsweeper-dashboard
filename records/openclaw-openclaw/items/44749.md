## Summary

Keep open: current main and v2026.6.10 still persist allow-always updates from a previously loaded approvals object, so concurrent stale snapshots can still overwrite each other. The focused fix PRs for this exact race are closed unmerged, and the open overlapping security PR is not merged and is currently conflicting.

## What This Changes

The issue reports that concurrent allow-always exec approvals can silently lose persisted allowlist entries in exec-approvals.json.

## Best Possible Solution

Merge an owner-reviewed persistence-boundary fix that reloads, locks, or CAS-merges exec approval allowlist writes for both gateway and node-host paths, with regression coverage for concurrent allow-always approvals and stale usage writeback.

## Reproduction Assessment

Yes. Source inspection shows both gateway and node-host allow-always paths persist entries from a previously loaded approvals object, and addAllowlistEntry saves that object wholesale; I did not run a live concurrent approval scenario in this read-only review.

## Solution Assessment

No. The current implementation uses atomic replacement for file integrity, but it does not merge concurrent stale snapshots; the best fix belongs at the shared exec-approvals persistence boundary rather than only at one caller.
