## Summary

Keep open: current main still persists allow-always and allowlist-use updates from caller-held `ExecApprovalsFile` snapshots, so concurrent approval handlers can overwrite earlier durable entries. The focused fix PR was closed unmerged, and the broad open security PR touches only approvals read-error handling, not stale-snapshot merging.

## What This Changes

The issue asks OpenClaw to prevent concurrent `allow-always` exec approvals from silently losing durable allowlist entries in `exec-approvals.json`.

## Best Possible Solution

Add one serialized, freshness-guarded exec approvals mutation path that reloads and merges current JSON under a lock or CAS guard, then route gateway and node-host allow-always plus usage-metadata writes through it with focused stale-snapshot regression coverage.

## Reproduction Assessment

Yes, at source level: two pending approval handlers can each hold an old `approvals.file`, then the later `addAllowlistEntry` or `recordAllowlistUse` call saves its stale whole-file snapshot over a prior persisted entry. I did not run the live Telegram race in this read-only review.

## Solution Assessment

Yes, the requested direction is the right shape, but it is not implemented on current main. The narrow maintainable fix is a locked reload-and-merge or CAS-guarded helper; backup tooling can reduce damage but cannot make approval persistence correct.
