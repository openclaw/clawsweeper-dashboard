## Summary

Keep this issue open: current main and the latest release still persist allow-always and allowlist usage updates by saving a previously resolved exec-approvals object, so concurrent approvals can still clobber entries.

## What This Changes

Make concurrent allow-always exec approvals persist all intended allowlist entries instead of losing earlier writes in exec-approvals.json.

## Best Possible Solution

Land one reviewed core exec-approvals persistence fix that serializes or hash-guards all allow-always and allowlist metadata writes across gateway and node paths, with regression coverage for concurrent approvals and stale policy writeback.

## Reproduction Assessment

Yes at source level: current gateway and node allow-always paths pass an earlier approvals.file object into helpers that save the full object after approval resolution. I did not run a live concurrent Telegram or node-host reproduction in this read-only review.

## Solution Assessment

Yes on owner boundary: the fix belongs in core exec approval persistence, not in an external backup skill or Telegram-specific workaround. The exact lock/CAS shape should be reviewed together with the overlapping security writeback issue.
