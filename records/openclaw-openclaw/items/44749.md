## Label Justifications

- `P1`: The issue can silently drop user-approved exec allowlist entries during normal concurrent agent/tool workflows.
- `impact:data-loss`: The reported race loses persisted allowlist entries from `exec-approvals.json` without notifying the user.
- `impact:security`: The affected file is security-relevant host command trust state used to decide whether future exec commands may run without approval.

## Summary

Keep open: current main still persists allow-always and allowlist-use updates from the caller-provided `ExecApprovalsFile` snapshot, so concurrent approvals can still overwrite earlier entries. The only direct fix PR I found, https://github.com/openclaw/openclaw/pull/52403, was closed unmerged, and the latest release tag has the same stale-snapshot helper shape.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can silently clobber entries in `exec-approvals.json` because the last writer saves an older in-memory allowlist snapshot.

## Best Possible Solution

Move allowlist add/update persistence to one locked read-modify-write helper that reloads the latest approvals file under the shared file-lock/store boundary, use it from both gateway and node-host allow-always paths, and add a focused concurrent persistence regression test.

## Reproduction Assessment

Yes, at source level: concurrent approval handlers can each carry an older `approvals.file` snapshot into `addAllowlistEntry`/`persistAllowAlwaysPatterns`, then save the whole file. I did not run a live Telegram repro because this review is read-only and should not mutate runtime state.

## Solution Assessment

Yes, a core persistence fix is the right layer; an external backup/merge workaround cannot make OpenClaw's own read-modify-write path atomic. The best fix is a shared locked reload-and-save helper, not separate caller-level patches.
