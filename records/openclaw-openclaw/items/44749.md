## Summary

Keep this open: current main and the latest release still mutate a caller-held `ExecApprovalsFile` and save the whole approval file from that stale snapshot, so concurrent `allow-always` approvals can still clobber each other. The direct fix PR was closed unmerged, and the related open security PR is not yet a safe canonical landing path.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can overwrite each other in `exec-approvals.json`, silently dropping earlier allowlist entries.

## Best Possible Solution

Move allowlist additions and usage metadata updates behind one store-owned locked/freshness-checked mutation path used by gateway and node-host callers, with regression coverage for concurrent stale-snapshot adds and revocation writeback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two approval handlers can hold separate `approvals.file` snapshots, then each writes the full file through `saveExecApprovals`, so the later write can omit entries added by the earlier one.

## Solution Assessment

Yes on direction: fixing the core approval store boundary is the narrow maintainable path. An external backup workaround is useful for recovery but cannot make internal exec-approval writes atomic.
