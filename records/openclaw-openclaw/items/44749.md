## Label Justifications

- `P1`: Concurrent allow-always approvals can silently lose persisted trust entries in a normal parallel exec workflow, affecting real agent/channel use.
- `impact:data-loss`: The reported failure drops user-approved allowlist entries from `exec-approvals.json` without warning.
- `impact:security`: The affected file is the exec approval allowlist, a persisted command-trust boundary that must be updated atomically.

## Summary

Keep open: current main and the latest release still persist exec allowlist mutations from caller-held snapshots, so concurrent allow-always handlers can still clobber earlier entries. The related lock-based repair PR is closed unmerged, and the open command-planner PRs do not clearly replace the missing serialized persistence fix.

## What This Changes

The issue reports that concurrent allow-always exec approval handlers write `exec-approvals.json` from stale snapshots and silently drop earlier allowlist entries.

## Best Possible Solution

Fix the shared exec approvals persistence helper so allowlist additions and usage metadata reload and write under one file lock, then cover gateway and node allow-always concurrency with focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: two approval handlers can each hold an older `approvals.file` snapshot and later write the whole file through `addAllowlistEntry` or `recordAllowlistUse`. I did not run a live concurrent Telegram approval flow in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to serialize the shared approvals-file read-modify-write in the store helper, not to patch only one caller. The existing `withFileLock` helper looks like the right primitive, but the final shape needs security review because this is persisted command trust.
