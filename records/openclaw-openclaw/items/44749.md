## Summary

Keep this issue open. Current main still has the reported stale-snapshot persistence path: allow-always and allowlist-use mutations save a caller-provided `ExecApprovalsFile` object wholesale, while gateway and node-host approval paths pass already-loaded snapshots into those helpers. The backup-plugin comment is useful mitigation context, but it does not replace a source-level fix for the internal race, and the directly related fix PR #52403 is closed unmerged.

## What This Changes

Fix concurrent `allow-always` exec approvals so all approved allowlist entries persist in `exec-approvals.json` instead of later writers overwriting earlier entries.

## Best Possible Solution

Add a write-side mutation helper in `src/infra/exec-approvals.ts` that serializes approval-file mutations, reloads the latest `exec-approvals.json` inside the critical section, applies additive allowlist, durable approval, and metadata changes without dropping unrelated fields, and saves once. Then route gateway and node-host allow-always and allowlist-use paths through that helper with regression coverage for two stale snapshots adding different entries to the same agent.

## Reproduction Assessment

Yes. The issue body provides concrete concurrent approval examples, and current main has a source-level reproduction: load two `ExecApprovalsFile` snapshots, add different allowlist entries through each snapshot, and the later `saveExecApprovals` writes its stale whole-file view over the earlier entry.

## Solution Assessment

Yes for a serialized reload-before-mutate store helper; it is the narrowest durable fix for the race. A backup or external merge workaround can reduce data loss impact, but it cannot make OpenClaw’s internal approval persistence correct.
