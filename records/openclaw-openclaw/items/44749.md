## Summary

Keep open: current main still writes caller-held exec approval snapshots for `allow-always`, so concurrent approval handlers can overwrite each other's allowlist entries. The keepmyclaw comment is useful backup-mitigation context and https://github.com/openclaw/openclaw/pull/52403 was a relevant attempted fix, but that PR is closed unmerged and the internal persistence race remains.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can silently lose earlier `exec-approvals.json` allowlist entries through last-write-wins persistence.

## Best Possible Solution

Add a serialized exec-approvals mutation path that reloads current JSON inside the write queue and file lock, then route gateway and node-host `allow-always` persistence through it without new config or weaker approval behavior.

## Reproduction Assessment

Yes, at source level. Two handlers can hold separate old `ExecApprovalsFile` objects, then the later `addAllowlistEntry` or `recordAllowlistUse` call writes its stale whole-file snapshot over the earlier persisted entry.

## Solution Assessment

Yes. A queued read-modify-write helper using the existing file-lock surface is the narrow maintainable fix; external backup tooling can reduce loss impact but cannot make approval persistence correct.
