## Summary

Keep open: current main and the latest release still persist allow-always and allowlist-usage updates from request-local exec approval snapshots, so concurrent writes can still silently replace earlier allowlist entries. Closed prior PRs did not merge, and the related open stale-writeback PR is conflicting and still needs proof/review rather than serving as implemented evidence.

## What This Changes

The issue reports concurrent allow-always exec approvals overwriting each other in exec-approvals.json.

## Best Possible Solution

Route all exec approval allowlist mutations through one concurrency-safe store update that reloads/merges under a lock or CAS guard, covers gateway and node callers, and has regression coverage for concurrent allow-always approvals.

## Reproduction Assessment

Yes from source inspection: two approval phases can hold separate ExecApprovalsFile snapshots and later save whole-file mutations through addAllowlistEntry. I did not run a live Telegram concurrency reproduction during this read-only review.

## Solution Assessment

No current-main solution is sufficient. The issue’s simple re-read suggestion would narrow but not eliminate the race; the best fix is a single locked or CAS-backed mutation path for approval state.
