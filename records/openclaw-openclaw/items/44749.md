## Summary

Keep open: current main and the latest release still save `exec-approvals.json` from an earlier in-memory approvals snapshot, so concurrent `allow-always` approvals can still clobber allowlist entries. The exact prior fix PR was closed unmerged, and the later open stale-writeback PR only partially overlaps because concurrent additions need merge semantics, not merely stale-write skipping.

## What This Changes

The issue reports concurrent `allow-always` exec approvals overwriting one another in `exec-approvals.json`, silently losing persisted allowlist entries.

## Best Possible Solution

Centralize exec approvals mutations behind one locked fresh-read merge path that preserves compatible concurrent allow-always additions, fails closed on conflicting admin or revocation updates, and covers both gateway and node-host paths with focused regression tests.

## Reproduction Assessment

Yes, source-level: two approval handlers can resolve approvals before the async approval wait, then each later saves its own stale `approvals.file` after `allow-always`. I did not run a live concurrent channel scenario in this read-only review.

## Solution Assessment

No, current main is not the best solution because it still writes the whole approvals file from stale snapshots. The best repair is a shared locked merge contract that distinguishes compatible additions from conflicting admin-policy changes.
