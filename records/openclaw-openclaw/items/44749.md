## Label Justifications

- `P1`: The report describes a broken exec approval workflow that can silently lose persisted security-policy state during normal concurrent agent work.
- `impact:data-loss`: The affected behavior silently drops persisted allowlist entries from `exec-approvals.json`.
- `impact:security`: The lost state controls which host commands can run without future approval.

## Summary

Keep this issue open: current main and the latest release still persist allowlist mutations from a caller-held approvals snapshot, so concurrent `allow-always` approvals can still lose earlier entries. The linked keepmyclaw backup suggestion is a mitigation for recovery, but it does not fix the core read-modify-write race, and the direct fix PR was closed unmerged.

## What This Changes

The issue reports that concurrent `allow-always` exec approvals can overwrite each other in `exec-approvals.json`, silently losing allowlist entries.

## Best Possible Solution

Keep this issue open until the exec approvals store performs allowlist mutations by re-reading and merging fresh state under serialization or a hash/CAS guard, with regression coverage for concurrent gateway and node-host allow-always writes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: two callers can hold separate `ExecApprovalsFile` snapshots, each call `addAllowlistEntry`, and the later `saveExecApprovals` writes its stale whole-file view over the earlier update.

## Solution Assessment

Yes, the best fix direction is at the exec approvals store boundary: serialize or hash-guard a fresh read-and-merge before saving. Fixing only one gateway, Telegram, or node caller would leave sibling stale-write paths behind.
