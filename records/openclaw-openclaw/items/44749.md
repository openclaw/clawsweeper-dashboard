## Label Justifications

- `P1`: Concurrent approvals can silently lose persisted allow-always entries in a real exec approval workflow, causing repeated approval prompts and undermining a security-sensitive control path.
- `impact:data-loss`: The issue is about silently dropped user approval allowlist entries in exec-approvals.json.
- `impact:security`: The affected file controls durable exec authorization, so the race sits on a security boundary even though the reported symptom is lost entries.

## Summary

Keep open: current main and the latest release still have the stale read-modify-write path for exec approval allowlist updates, so concurrent allow-always approvals can still clobber entries. The directly related fix PR was closed unmerged, and no searched canonical open item clearly owns this exact storage race.

## What This Changes

The issue asks OpenClaw to preserve every concurrent allow-always exec approval entry instead of letting stale exec-approvals.json snapshots overwrite each other.

## Best Possible Solution

Centralize exec approval allowlist mutations behind a serialized reload-mutate-save helper used by allow-always and allowlist-use updates in both gateway and node-host paths, with focused concurrent stale-snapshot regression coverage.

## Reproduction Assessment

Yes. The source path is high-confidence: concurrent handlers can pass stale ExecApprovalsFile snapshots into addAllowlistEntry or recordAllowlistUse, and each writes the whole file. I did not run a live concurrent Telegram approval in this read-only review.

## Solution Assessment

No. Current main still writes stale snapshots; the best fix is a serialized reload-mutate-save store helper, not another caller-local write or a backup workaround.
