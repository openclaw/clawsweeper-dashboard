## Summary

Keep open: current main still persists `allow-always` and allowlist-use mutations from caller-held `ExecApprovalsFile` snapshots, so concurrent approval handlers can still overwrite entries with stale whole-file writes. The earlier direct fix attempt at https://github.com/openclaw/openclaw/pull/52403 is closed unmerged, and the overlapping open security PR at https://github.com/openclaw/openclaw/pull/78226 is not merged and has review blockers.

## What This Changes

Concurrent `allow-always` exec approvals should persist every approved allowlist entry instead of losing earlier entries to stale last-write-wins saves.

## Best Possible Solution

Add a single serialized exec-approvals mutation path that reloads current JSON under a file lock or equivalent queue, merges allowlist additions and usage metadata, and routes gateway and node-host persistence through it without broadening approval trust.

## Reproduction Assessment

Yes, at source level. Two approval handlers can each hold an old approvals object, then the later `addAllowlistEntry` or `recordAllowlistUse` call saves its stale whole-file snapshot over an earlier persisted entry.

## Solution Assessment

Yes for the proposed direction, but it is not implemented on current main. A locked reload-and-merge helper is the narrow maintainable fix; external backup tooling can reduce damage but cannot make approval persistence correct.
