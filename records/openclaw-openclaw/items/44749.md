## Label Justifications

- `P1`: Silent loss of allow-always approval state can break a real security-sensitive exec approval workflow for users approving concurrent commands.
- `impact:data-loss`: The reported race silently drops persisted allowlist entries from `exec-approvals.json`.

## Summary

Keep open: current main and the latest release still persist allow-always allowlist changes from request-local approval snapshots, so concurrent approvals can still clobber earlier entries. The issue is now paired with an open closing PR, so it should stay open until that fix is reviewed, corrected if needed, and merged.

## What This Changes

The issue reports silent loss of persisted exec allowlist entries when concurrent allow-always approvals overwrite `exec-approvals.json` with stale snapshots.

## Best Possible Solution

Make exec approval persistence merge or reload the latest approvals state inside one shared critical section, with regression coverage for concurrent allow-always approvals across the gateway and node-host paths.

## Reproduction Assessment

Yes from source inspection: two concurrent approval handlers can each hold a different loaded `ExecApprovalsFile` snapshot and later save the whole stale object. I did not run a live concurrent Telegram approval scenario in this read-only review.

## Solution Assessment

Yes for the issue boundary: the fix should live in the shared exec approvals persistence path so both gateway and node-host callers get the same atomic behavior. The linked PR should be checked for merge-under-lock semantics, not only write serialization.
