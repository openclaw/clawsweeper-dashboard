## Label Justifications

- `impact:security`: The lost allowlist state governs future host exec approval and trust decisions.

## Summary

Keep open: current main and the latest release still write allowlist updates from the caller's in-memory approvals snapshot, so concurrent allow-always approvals can still overwrite each other; the linked repair PR was closed unmerged and is not a canonical fix path.

## What This Changes

This issue requests a fix for concurrent allow-always exec approvals so all approved allowlist entries persist in exec-approvals.json instead of the last writer overwriting earlier entries.

## Best Possible Solution

Serialize the exec approvals read-modify-write path under the shared file-lock/store boundary for addAllowlistEntry, recordAllowlistUse, gateway, and node-host allow-always paths, with a focused concurrent persistence regression test and no config or UX changes.

## Reproduction Assessment

Yes at source level: current main still passes stale approvals.file snapshots into addAllowlistEntry/recordAllowlistUse and writes the whole file without locking the read-modify-write. I did not run a live concurrent approval scenario in this read-only pass.

## Solution Assessment

Yes: a locked store helper that reloads and merges inside the serialized critical section is the narrow owner-boundary fix. A re-read-only patch would reduce the race window but would not fully solve concurrent writers.
