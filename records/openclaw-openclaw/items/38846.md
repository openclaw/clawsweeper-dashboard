## Summary

The PR should not be merged as-is: current main validates both the resolved Windows wrapper command and argv, and this diff would reject ordinary Windows paths/flags while exposing rejected values in error messages.

## What This Changes

The PR broadens Windows cmd.exe argument validation in `src/process/exec.ts`, adds PowerShell/null/control-character checks, and exports `validateWindowsCommandArgument()`.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening change rejects only true wrapper metacharacters, redacts values, and adds regression coverage for `.cmd` paths, slash switches, equals flags, and rejected injection characters.

## Reproduction Assessment

Yes for the PR regressions by source inspection. On win32, the wrapper command line validates both the resolved command path and argv, so the PR regex would throw on backslashes, slash switches, and equals flags before spawn.

## Solution Assessment

No. A broad blacklist that mixes cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags is not the narrowest maintainable hardening; validation should stay scoped to the wrapper parser and redact error values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow Windows path separators in cmd validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\\` and `/` for every value passed through `escapeForCmdExe()`. That includes resolved `.cmd`/`.bat` paths and valid slash-prefixed switches, so normal Windows wrapper calls can throw before spawn.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` blocks valid arguments this repo emits, including npm flags like `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself.
  - confidence: 0.91
- **[P2] Redact rejected argument values from errors:** `src/process/exec.ts:73-78`
  - body: The new validation errors include `JSON.stringify(arg)`, so rejected arguments containing tokens, passwords, or API keys can be copied into exception messages and logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
