## Summary

Keep open. Current main does not implement the broader Windows command-argument validation proposed here, and the PR diff is not safe to merge because it over-blocks valid Windows cmd wrapper syntax and exposes raw arguments in errors on a security-sensitive execution path.

## What This Changes

The PR modifies `src/process/exec.ts` to broaden Windows cmd.exe argument validation, add null/control-character and PowerShell-pattern checks, and export a Windows command-argument validator.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening change is designed. A replacement should reject only real cmd-wrapper separators or expansion hazards, preserve paths, slash switches, equals flags, and scoped packages, avoid applying PowerShell-specific policy to non-PowerShell command paths, redact argument values in errors, and add targeted Windows regression tests.

## Reproduction Assessment

Yes for the PR regressions. On `win32`, `.cmd` and `.bat` commands go through `buildCmdExeCommandLine`, which validates the resolved command path and each argument; the PR regex would reject normal backslashes, slash switches/package specs, and equals-bearing npm flags before execution.

## Solution Assessment

No. The broad blacklist is not the best fix because it conflates cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags; the safer direction is a narrow shell-wrapper validation policy with redacted errors and regression tests for known Windows call sites.

Full review comments:

- **[P1] Allow path separators in cmd wrapper validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\\` and `/` for every value passed through `escapeForCmdExe`. That includes resolved `.cmd`/`.bat` paths and valid arguments such as slash-prefixed switches or scoped package specs, so normal Windows wrapper calls can throw before spawn instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags like `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself, so this turns normal flows into hard failures.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-78`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key can be copied into exception messages, stack traces, or logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
