## Summary

Keep open, but do not merge as-is: the PR touches Windows command execution and the diff would reject legitimate .cmd/.bat paths, slash switches, and equals-bearing npm flags while also echoing raw argument values in validation errors.

## What This Changes

The PR expands Windows cmd.exe argument validation in `src/process/exec.ts`, adds PowerShell/null/control-character checks, exports a validator, and adds explanatory security comments.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening change rejects only real wrapper metacharacters, redacts values in errors, and adds regression coverage for .cmd paths, slash switches, equals flags, and any PowerShell-specific path.

## Reproduction Assessment

Yes for the PR regressions by source inspection. On `win32`, `.cmd` and `.bat` commands go through `buildCmdExeCommandLine()`, which validates the resolved command path and every argument; the PR regex would throw on normal backslashes, slash switches, and equals flags before spawn.

## Solution Assessment

No. The broad blacklist is not the best fix because it conflates cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags; the safer direction is a narrow shell-wrapper validation policy with redacted errors and targeted Windows tests.

## Review Findings

Full review comments:

- **[P1] Allow path separators in cmd wrapper validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\\` and `/` for every value passed through `escapeForCmdExe()`. That includes resolved `.cmd`/`.bat` paths and valid slash-prefixed arguments, so normal Windows wrapper calls can throw before spawn instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags like `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself, so this turns normal flows into hard failures.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-78`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key can be copied into exception messages, stack traces, or logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
