## Summary

Keep open, but do not merge as written. Current main does not implement the PR's exported validator or added checks, so it is not obsolete; however the branch introduces blocking Windows command-execution regressions and raw argument leakage in validation errors.

## What This Changes

The PR broadens `src/process/exec.ts` Windows cmd.exe argument validation by expanding the unsafe-character regex, adding PowerShell-pattern and null/control-character checks, and exporting a Windows command-argument validator.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening change is designed. A replacement should reject only real cmd wrapper separators/expansion hazards, preserve paths, slash switches, equals flags, and scoped packages, avoid PowerShell policy in non-PowerShell paths, redact argument values in errors, and add targeted Windows regression tests.

## Reproduction Assessment

Yes for the PR regressions: on `win32`, `.cmd`/`.bat` commands go through `buildCmdExeCommandLine`, which validates the command path and every argument before spawn. The PR regex then rejects normal backslashes, `/` package/switch arguments, and `=` npm flags before execution.

## Solution Assessment

No. The broad blacklist is not the best fix for command injection hardening because it conflates cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags; the safer direction is a narrower shell-wrapper validation policy with redacted errors and tests for known Windows call sites.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow path separators in cmd wrapper validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\\` and `/` for every value passed through `escapeForCmdExe`. That includes resolved `.cmd`/`.bat` paths and scoped package specs passed as argv, so valid Windows wrapper calls can throw before `spawn` instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments that this repo emits, such as npm update flags `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself, so this turns normal update/install flows into hard failures.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-78`
  - body: The new errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key will be copied into exception messages and logs. Report the rejected character, context, and length without echoing the full argument value.
  - confidence: 0.9
