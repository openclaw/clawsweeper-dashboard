## Summary

Keep this PR open, but it is not safe to merge as written: the patch changes the shared Windows cmd.exe wrapper path and the new blacklist blocks normal Windows paths and flags while also exposing rejected argument values in errors.

## What This Changes

The PR expands Windows cmd.exe argument validation in `src/process/exec.ts`, adds PowerShell/null/control-character checks, and exports a validator for external callers.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening patch rejects only real wrapper metacharacters, redacts rejected values, and adds regression coverage for `.cmd` paths, slash switches, equals flags, and any PowerShell-specific path.

## Reproduction Assessment

Yes. By source inspection, on `win32` the PR validates the resolved `.cmd`/`.bat` command and every wrapper argument before spawn, so normal backslashes, slash switches, and equals flags can throw before execution.

## Solution Assessment

No. The proposed broad blacklist conflates cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags; the safer solution is a narrow wrapper policy with redacted errors and targeted Windows tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow path separators in cmd wrapper validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\` and `/` for every value passed through `escapeForCmdExe()`. That includes resolved `.cmd`/`.bat` paths and valid slash-prefixed arguments, so normal Windows wrapper calls can throw before spawn instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags like `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself, so this turns normal flows into hard failures.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-84`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key can be copied into exception messages, stack traces, or logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
