## Summary

Keep open: the proposed hardening currently over-rejects valid Windows wrapper syntax and leaks raw rejected values in errors, and the PR still lacks real Windows behavior proof.

## What This Changes

The branch expands Windows cmd.exe argument validation in `src/process/exec.ts`, adds PowerShell/null/control-character checks, and exports a validation helper.

## Best Possible Solution

Rework this as narrow cmd-wrapper hardening: keep path separators, slash switches, and equals flags valid; reject actual cmd metacharacters; redact validation errors; add wrapper regression tests; and provide redacted Windows runtime proof.

## Reproduction Assessment

Yes, for the PR regression by source inspection: on `win32`, `.cmd` and `.bat` commands flow through `buildCmdExeCommandLine(resolvedCommand, ...)`, and the PR validates that resolved path plus normal args with a blocklist that rejects path separators and common flags. I did not run live Windows proof in this read-only review.

## Solution Assessment

No. The proposed blacklist conflates cmd.exe metacharacters, PowerShell policy, Windows path syntax, and ordinary CLI flags; a narrower wrapper-specific policy with redacted errors is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow Windows path separators in cmd validation:** `src/process/exec.ts:45`
  - body: The expanded blocklist rejects `\` and `/` for every value passed through `escapeForCmdExe()`. `buildCmdExeCommandLine()` passes the resolved `.cmd`/`.bat` path itself plus normal CLI arguments, so Windows wrapper calls with absolute paths or slash switches can throw before spawn.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags such as `--loglevel=error` and `--omit=optional` when Windows falls back to command shims. `=` is not a cmd.exe command separator by itself.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:74-92`
  - body: The new errors include `JSON.stringify(arg)` for dangerous-character, PowerShell-pattern, and null-byte failures. Rejected arguments can contain tokens or passwords, so full values can leak into stack traces or logs; report class, context, and length instead.
  - confidence: 0.9
