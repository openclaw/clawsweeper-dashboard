## Summary

Keep open: this security-sensitive Windows command-execution PR still has source-reproducible correctness and security regressions, lacks after-fix real Windows proof, and is still tracked in the open Microsoft/Windows backlog.

## What This Changes

The PR modifies `src/process/exec.ts` to expand Windows cmd.exe argument validation, add PowerShell/null/control-character checks, and export a validation helper.

## Best Possible Solution

Replace the broad blacklist with focused cmd-wrapper hardening that rejects true cmd.exe metacharacters, redacts validation errors, and adds Windows regression coverage for `.cmd` paths, slash switches, equals flags, and any separate PowerShell path.

## Reproduction Assessment

Yes, for the PR regression by source inspection. On `win32`, `.cmd` and `.bat` commands flow through `buildCmdExeCommandLine()`, so the proposed validator would reject resolved paths and common flags before spawn.

## Solution Assessment

No. The broad blacklist conflates cmd.exe metacharacters, PowerShell policy, Windows path syntax, and ordinary CLI flags; a narrower wrapper policy with redacted errors and targeted Windows tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow Windows path separators in cmd validation:** `src/process/exec.ts:45`
  - body: The expanded blocklist rejects `\` and `/` for every value passed through `escapeForCmdExe()`. `buildCmdExeCommandLine()` passes the resolved `.cmd`/`.bat` path itself plus normal CLI arguments, so Windows wrapper calls with absolute paths or slash switches can throw before spawn.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags such as `--loglevel=error` and `--omit=optional` when Windows falls back to command shims. `=` is not a cmd.exe command separator by itself.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:74-92`
  - body: The new errors include `JSON.stringify(arg)` for dangerous-character, PowerShell-pattern, and null-byte failures. Rejected arguments can contain tokens or passwords, so full values can leak into stack traces or logs; report class, context, and length instead.
  - confidence: 0.9
