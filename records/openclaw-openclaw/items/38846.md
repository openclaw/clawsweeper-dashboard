## Summary

Keep open. The PR targets a real security-sensitive command-execution surface, but the current patch is not safe to merge because it rejects normal Windows paths and command flags and adds new raw-argument error paths that can expose secrets.

## What This Changes

The PR modifies `src/process/exec.ts` to broaden Windows cmd.exe argument validation, add PowerShell/null/control-character checks, and export a Windows command-argument validator.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused Windows hardening change is designed. A replacement should reject only real cmd-wrapper separators or expansion hazards, preserve paths, slash switches, equals flags, and scoped packages, avoid applying PowerShell-specific policy to non-PowerShell command paths, redact argument values in errors, and add targeted Windows regression tests.

## Reproduction Assessment

Yes for the PR regressions. On `win32`, `.cmd` and `.bat` commands go through `buildCmdExeCommandLine`, which validates the resolved command path and each argument; the PR regex would reject normal backslashes, `/` package/switch arguments, and `=` npm flags before execution.

## Solution Assessment

No. The broad blacklist is not the best fix for command injection hardening because it conflates cmd.exe metacharacters, PowerShell policy, path syntax, and ordinary CLI flags; the safer alternative is a narrower shell-wrapper validation policy with redacted errors and regression tests for known Windows call sites.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow path separators in cmd wrapper validation:** `src/process/exec.ts:45`
  - body: The new blocklist rejects `\\` and `/` for every value passed through `escapeForCmdExe`. That includes resolved `.cmd`/`.bat` paths and scoped package specs passed as argv, so valid Windows wrapper calls can throw before `spawn` instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments that this repo emits, such as npm update flags `--loglevel=error` and `--omit=optional` when Windows falls back through `npm.cmd`. `=` is not a cmd.exe command separator by itself, so this turns normal update/install flows into hard failures.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-78`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key can be copied into exception messages and logs. Report the rejected class, context, and length without echoing the full argument value.
  - confidence: 0.9
