## Summary

Close as mostly implemented on current main: the useful Windows cmd.exe hardening now exists in the main wrapper and tests, while this older branch’s remaining changes are over-broad, security-sensitive, and already reviewed as breaking valid Windows command execution.

## What This Changes

The PR modifies `src/process/exec.ts` to broaden Windows cmd.exe argument validation, add PowerShell/null/control-character checks, and export a Windows command argument validation helper.

## Best Possible Solution

Keep the current-main wrapper implementation; any further Windows or PowerShell hardening should be a new focused PR against current main with redacted errors, clear cmd.exe versus PowerShell boundaries, and targeted Windows regression coverage.

## Reproduction Assessment

Yes, for the PR regression by source inspection. On `win32`, `.cmd` and `.bat` commands flow through `buildCmdExeCommandLine()`, so the proposed validator would reject resolved Windows paths and common flags before spawn.

## Solution Assessment

No. Current main’s narrower cmd.exe wrapper guard is the safer implementation; this branch’s broad blacklist conflates cmd.exe metacharacters, PowerShell policy, Windows path syntax, and ordinary CLI flags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow Windows path separators in cmd validation:** `src/process/exec.ts:45`
  - body: The expanded blocklist rejects `\` and `/` for every value passed through `escapeForCmdExe()`. That includes resolved `.cmd`/`.bat` paths and valid slash-prefixed arguments, so normal Windows wrapper calls can throw before spawn instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm update/install flags such as `--loglevel=error` and `--omit=optional` when Windows falls back through command shims. `=` is not a cmd.exe command separator by itself.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:72-92`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected argument containing a token, password, or API key can be copied into exception messages, stack traces, or logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
