## Summary

Keep open: this PR is a plausible security-hardening proposal, but the current patch is not mergeable because it breaks the Windows cmd wrapper path, leaks raw rejected arguments in errors, and still lacks after-fix real Windows proof.

## What This Changes

The PR broadens Windows cmd.exe argument validation in `src/process/exec.ts`, adds PowerShell/null/control-character checks, and exports a validator.

## Best Possible Solution

Keep the current narrow cmd.exe wrapper guard until a focused hardening change rejects only true wrapper metacharacters, redacts rejected values, and adds Windows regression coverage for `.cmd` paths, slash switches, equals flags, and any separately scoped PowerShell behavior.

## Reproduction Assessment

Yes, for the PR regression by source inspection. On `win32`, `.cmd` and `.bat` invocations pass the resolved command path and arguments through `escapeForCmdExe()`, so the proposed validator would reject normal paths and common flags before spawn.

## Solution Assessment

No. The broad blacklist conflates cmd.exe metacharacters, PowerShell policy, Windows path syntax, and ordinary CLI flags; a narrower wrapper policy with redacted errors and targeted Windows tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow Windows path separators in cmd validation:** `src/process/exec.ts:45`
  - body: The expanded regex rejects backslash and `/` for every value passed through `escapeForCmdExe()`. `buildCmdExeCommandLine()` passes resolved `.cmd`/`.bat` command paths through that function, so normal Windows wrapper calls can throw before spawn instead of executing.
  - confidence: 0.94
- **[P2] Permit equals signs in cmd wrapper arguments:** `src/process/exec.ts:45`
  - body: Rejecting `=` breaks valid arguments this repo emits, including npm flags such as `--loglevel=error` and `--omit=optional` when Windows falls back through command shims. `=` is not a cmd.exe command separator by itself.
  - confidence: 0.91
- **[P2] Redact argument values in validation errors:** `src/process/exec.ts:73-92`
  - body: The new validation errors include `JSON.stringify(arg)`, so a rejected token, password, or API key can be copied into exception messages, stack traces, or logs. Report the rejected class, context, and length without echoing the full value.
  - confidence: 0.9
