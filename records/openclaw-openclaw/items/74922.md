## Summary

Keep open: current main still suppresses Discord automatic ack/status reactions for guild/channel source turns that resolve to `message_tool_only`, including the reported `ackReactionScope: "all"` case. The channel-resolve fallback does not appear to drop ID-keyed guild config by itself, and the linked #73133 reaction-listener fix is a separate inbound reaction-event path; the remaining decision is whether configured ack reactions are allowed visible output in tool-only guild rooms.

## What This Changes

The issue asks OpenClaw to restore Discord automatic ackReaction behavior in guild channels after v2026.4.27 when `ackReactionScope` is configured as `all`.

## Best Possible Solution

Define the Discord/group-channel contract explicitly. If configured ack reactions are allowed in tool-only rooms, decouple the initial ack reaction from lifecycle status/final-reply suppression and update the matching tests/docs; if they are intentionally private output, document the `ackReaction`, `ackReactionScope`, `statusReactions`, and `messages.groupChat.visibleReplies` interaction and add clearer suppression logging.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists on current main: a guild/channel route with `ackReactionScope: "all"` resolves to `message_tool_only`, and the current Discord process test asserts no reaction is sent.

## Solution Assessment

Unclear. Re-enabling `ackReactionScope: "all"` for tool-only rooms would address the report, but a naive change could bypass the shipped group/channel visible-output privacy policy; the safer path is a maintainer decision followed by a narrow Discord reaction-gate/docs update.
