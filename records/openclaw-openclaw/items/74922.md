## Summary

Keep open: current main still suppresses automatic Discord ack/status reactions for guild/channel turns that resolve to `message_tool_only`, including the reported `ackReactionScope: "all"` shape. The channel-resolution fallback preserves configured guild entries, and the linked #73133 path is a separate inbound reaction-event listener fix, so this is not implemented or safely closeable as stale cleanup.

## What This Changes

The issue reports that Discord automatic ack reactions stopped firing in guild channels after v2026.4.27 when `ackReaction` and `ackReactionScope: "all"` are configured.

## Best Possible Solution

Define the Discord guild/channel contract explicitly. If configured ack reactions are allowed in tool-only rooms, decouple the initial ack from lifecycle status/final-reply suppression and update focused tests/docs; if they are intentionally private output, document the `ackReaction`, `ackReactionScope`, `statusReactions`, and `messages.groupChat.visibleReplies` interaction and add clearer suppression logging.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists on current main: a guild/channel route with `ackReactionScope: "all"` resolves to `message_tool_only`, and the current Discord process test asserts no reaction is sent.

## Solution Assessment

Unclear. Re-enabling `ackReactionScope: "all"` for tool-only rooms would address the report, but a naive change could bypass the shipped group/channel visible-output privacy policy; the safer path is a maintainer decision followed by a narrow Discord reaction-gate/docs update.
