## Summary

Keep open: current main still suppresses automatic Discord ack/status reactions for guild/channel turns that resolve to `message_tool_only`, including the reported `ackReactionScope: "all"` shape. The reported channel-resolution error is real context but not the direct config-loss cause, and the #73133 listener fix is a separate inbound reaction-event path.

## What This Changes

The issue asks for Discord automatic `ackReaction` to fire in guild/channel messages configured with `ackReactionScope: "all"` despite channel-resolution fallback errors and per-channel guild allowlists.

## Best Possible Solution

Define the Discord guild/channel visibility contract explicitly. If configured ack reactions are allowed in tool-only rooms, decouple the initial ack from lifecycle status/final-reply suppression and update focused tests, docs, and changelog; if they are intentionally private output, document the `ackReaction`, `ackReactionScope`, `statusReactions`, and `messages.groupChat.visibleReplies` interaction and add clearer suppression logging.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists on current main: a guild/channel route with `ackReactionScope: "all"` resolves to `message_tool_only`, and the current Discord process test asserts no reaction is sent.

## Solution Assessment

Unclear. Re-enabling `ackReactionScope: "all"` for tool-only rooms would address the report, but deciding whether a visible emoji acknowledgement is allowed under the shipped guild privacy policy needs maintainer/product judgment first.
