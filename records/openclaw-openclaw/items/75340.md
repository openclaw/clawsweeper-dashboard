## Summary

Keep open. Current main still routes plain text status, including `--usage`, through the fast channel-table path with setup/runtime fallback disabled, while the bundled channel loaders intentionally skip runtime-deps registration on those discovery/setup loads. The issue discussion correctly identifies a real packaged CLI regression rather than a stale or out-of-scope request.

## What This Changes

The issue requests that packaged `openclaw status --usage` register or otherwise use bundled runtime dependencies before building bundled channel status rows, so healthy configured channels do not disappear behind false missing-module warnings.

## Best Possible Solution

Keep plain status read-only and reasonably fast, but make its configured channel table reliable for bundled packaged installs by using manifest/setup metadata that does not require runtime imports or by preparing the active staged runtime-deps root before any bundled setup/channel import that can require external packages. Avoid turning plain status into a broad live probe or full plugin preload.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: run packaged `openclaw status --usage` with configured bundled channels whose dependencies are staged only under the managed runtime-deps root; current main keeps the fast path off setup fallback and the bundled discovery/setup loaders skip runtime-deps registration.

## Solution Assessment

No current-main solution is sufficient yet. The narrowest maintainable fix is status/read-only-channel specific: either build configured bundled channel rows from descriptor metadata or register the staged packaged runtime-deps root before the existing bundled import path that can require those packages.
