## Summary

Keep open. Current main still matches the packaged CLI/status regression: plain text status, including `--usage`, stays on the fast metadata path with setup/runtime fallback disabled, while bundled channel discovery still avoids staged runtime-deps registration before importing generated channel/setup entry modules. Related #75685 is open and broader, not a merged superseding fix.

## What This Changes

The issue requests that packaged `openclaw status --usage` register or otherwise account for managed bundled runtime dependencies before loading bundled channel status/setup modules so configured channel rows render without false missing-module warnings.

## Best Possible Solution

Keep status read-only and fast, but make configured bundled channel table construction reliable for packaged installs by using metadata that avoids runtime imports where possible, or by preparing only the active staged bundled runtime-deps root before the existing configured bundled channel setup/table fallback imports.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is a packaged `openclaw status --usage` with configured bundled channels whose dependencies exist only under the managed runtime-deps root; current main keeps that path off setup fallback and skips staged runtime-deps preparation during bundled channel discovery.

## Solution Assessment

No current-main solution is sufficient yet. The safest fix is a narrow status/read-only-channel change, not an unconditional full plugin preload, package-manager repair, or live `channels.status` probe for plain status.

## Repair Work Prompt

Fix openclaw/openclaw#75340. Current main routes plain text `openclaw status --usage` through `scanStatus` without letting `usage` affect the scan mode; plain status sets `includeChannelSetupRuntimeFallback` false, and bundled channel discovery imports generated channel/setup entries with `installRuntimeDeps: false`. In packaged installs, configured bundled channels whose deps only exist under the managed `plugin-runtime-deps` root can therefore emit missing-module warnings and render an empty Channels table. Implement a narrow fix so configured bundled channel rows render without missing-module warnings in packaged `status --usage`. Prefer manifest/setup metadata that avoids runtime imports; if runtime import is still required, prepare/register only the active staged bundled runtime-deps root before the existing configured bundled channel setup/table fallback imports. Do not turn plain status into a live `channels.status` probe, do not broadly preload all plugins, and do not install or repair deps from normal status. Add focused regression coverage for the packaged fast-status path and update `CHANGELOG.md` under `2026.4.30` Fixes with `Thanks @tasi333`; also credit @lamkan0210 if the PR intentionally covers related #75685.
