## Summary

Keep open. Current main still routes plain text status, including `--usage`, through the fast channel-table path with setup/runtime fallback disabled, while the bundled channel loaders have runtime-deps preparation available but explicitly skip it for discovery/setup loads. The report remains a valid packaged CLI regression, not a stale or out-of-scope cleanup item.

## What This Changes

The issue requests that packaged `openclaw status --usage` register or use the active managed bundled runtime-deps root before loading bundled channel status/setup modules so configured channels render correctly without manual `NODE_PATH` or `--deep`.

## Best Possible Solution

Keep plain status read-only and reasonably fast, but make the configured channel table reliable for packaged bundled installs by using dependency-light manifest/setup metadata where possible or by registering the already staged active runtime-deps root before any bundled setup/channel import that can require external packages. Avoid turning plain status into a broad live probe or package-manager repair path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: run packaged `openclaw status --usage` with configured bundled channels whose dependencies are present only under the managed runtime-deps root; current main leaves setup fallback disabled and skips runtime-deps registration on the bundled import path.

## Solution Assessment

No current-main solution is sufficient yet. The narrow maintainable fix should be status/channel-table specific and should preserve the hot-path goal without returning an empty or misleading Channels table.
