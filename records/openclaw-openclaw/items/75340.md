## Summary

Keep open. Current main still leaves plain text `openclaw status --usage` on the fast status scan that disables setup/runtime fallback, and the bundled channel loaders still skip staged runtime-deps registration when loading discovery/setup entries. The post-release runtime-deps fixes address adjacent upgrade and WhatsApp staging paths, but I found no current-main source, test, or changelog evidence that this status path now prepares staged dependencies or reliably handles setup-runtime channel rows.

## What This Changes

Ensure packaged `openclaw status --usage` can render configured bundled channel rows without false missing-module warnings when bundled channel dependencies exist only under the managed runtime-deps root.

## Best Possible Solution

Keep plain status read-only and fast, but make its configured channel table reliable for packaged installs by using manifest/setup metadata that does not require runtime imports, or by registering the active staged runtime-deps root before any necessary bundled setup/status import. Avoid turning plain status into a broad live probe or full plugin preload.

## Reproduction Assessment

Yes. The high-confidence source reproduction is a packaged install with configured bundled channels whose dependencies are staged only under the managed runtime-deps root, then running `openclaw status --usage`; current main still keeps that path off setup fallback and the bundled discovery/setup loaders skip runtime-deps registration.

## Solution Assessment

No. Current main's metadata-only fast path is not sufficient for the reported setup-runtime dependency case; the narrowest maintainable fix is status/read-only-channel specific rather than making every status invocation run full live channel probes.
