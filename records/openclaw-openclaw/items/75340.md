## Summary

Keep open. Current main still matches the reported packaged CLI/status regression: plain text `openclaw status --usage` takes the fast status path with channel setup fallback disabled, while bundled channel entry/setup discovery continues to suppress staged runtime-deps registration before importing generated modules. Related #75685 is open and adjacent, but it is broader packaged channel runtime-deps fallout rather than a merged superseding fix for this status table path.

## What This Changes

The issue requests that packaged `openclaw status --usage` register or otherwise use the active managed bundled runtime-deps path before loading bundled channel status/setup modules so configured channels render correctly without manual `NODE_PATH` or `--deep`.

## Best Possible Solution

Keep status read-only and fast, but make configured bundled channel table construction reliable for packaged installs by using metadata that avoids runtime imports where possible, or by preparing only the active staged bundled runtime-deps root before the existing configured bundled channel setup/table fallback imports.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is a packaged `openclaw status --usage` with configured bundled channels whose dependencies exist only under the managed runtime-deps root; current main keeps that command off setup fallback and skips staged runtime-deps preparation during bundled channel discovery.

## Solution Assessment

Unclear for the reporter's exact implementation, but the direction is right. The safest solution is a narrow status/read-only channel change, not an unconditional full plugin preload, package-manager repair, or live `channels.status` probe for plain status.
