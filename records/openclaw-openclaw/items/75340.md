## Summary

Keep open. Current main still has the status/runtime-deps split described in the report: plain text status, including `--usage`, stays on the fast path with setup fallback disabled, while configured bundled channels can still require setup/runtime or staged dependencies before their status rows can be built. Related #75685 is broader and open, so it is useful context but not a merged superseding fix.

## What This Changes

The issue reports that packaged `openclaw status --usage` loads bundled channel status modules before registering managed bundled runtime dependencies, causing false missing-module warnings and an empty channel table.

## Best Possible Solution

Keep status read-only and fast while making configured bundled channel rows reliable for packaged installs: either use sufficient manifest/setup metadata without importing runtime modules, or prepare only the active staged bundled runtime-deps root before the existing setup/channel import path that needs external packages.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: run packaged `openclaw status --usage` with configured bundled channels whose dependencies are only under the managed runtime-deps root; current main keeps setup fallback disabled and imports generated bundled channel/setup entries with runtime-deps installation disabled. The issue also supplies a successful `NODE_PATH` workaround that matches this path.

## Solution Assessment

No current-main solution is sufficient yet. The best fix is a narrow status/read-only channel change, not an unconditional full plugin preload or live `channels.status` probe for plain status.

## Repair Work Prompt

Fix the packaged CLI/status regression reported in https://github.com/openclaw/openclaw/issues/75340. Current main routes plain `openclaw status --usage` through the text status scan with `includeChannelSetupRuntimeFallback` disabled, then builds the configured channel table through `listReadOnlyChannelPluginsForConfig`; bundled generated channel/setup entry loads pass `installRuntimeDeps: false`, so packaged installs whose channel deps live only under `~/.openclaw/plugin-runtime-deps/openclaw-<version>-*/node_modules` can emit false missing-module warnings and show an empty Channels table. Keep the fix narrow and read-only: do not enable live `channels.status` probes for plain status, do not full-preload all plugins, and preserve the lean `status --json` behavior unless a test proves it must change. Prefer manifest/setup metadata that avoids runtime imports; if runtime import is still required for configured bundled channel rows, prepare/register only the active staged bundled runtime-deps root before that import path. Coordinate with related #75685 and note the closed related #75394 symptom, but preserve this issue's status-specific acceptance criteria. Add or update focused tests around `status --usage`/plain text status, `includeSetupFallbackPlugins`, bundled setup/runtime deps registration, and read-only channel table behavior. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter with `Thanks @tasi333` and include `Thanks @lamkan0210` if the repair also covers #75685. Do not change dependency versions, release metadata, or broad plugin loading policy.
