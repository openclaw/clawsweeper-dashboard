## Summary

Keep open. Current main still matches the report: plain text `openclaw status --usage` keeps the channel table on the fast metadata path, while configured bundled channels that need setup/runtime fallback can be skipped or warn because the bundled channel loaders deliberately avoid registering staged runtime deps on discovery/setup reads.

## What This Changes

The issue asks for packaged `openclaw status --usage` to locate bundled channel runtime dependencies before building the fast status Channels table.

## Best Possible Solution

Keep plain status non-live, but make configured bundled channel rows reliable in packaged installs by either using complete manifest/static metadata for the status table or narrowly registering the active staged runtime-deps root before the existing bundled setup/status imports that need it. Avoid turning plain status into a live channel probe or broad plugin preload.

## Reproduction Assessment

Yes. A packaged install with configured bundled channels whose dependencies exist only under the managed runtime-deps stage should reproduce the source-level path: `openclaw status --usage` leaves setup fallback off, then bundled discovery/setup imports skip runtime-deps registration.

## Solution Assessment

No current-main solution is sufficient. The narrow maintainable fix is status/read-only-channel specific: provide accurate configured channel rows without full runtime loading, or register the packaged runtime-deps stage before the minimal bundled imports that are already required.

## Repair Work Prompt

Fix `openclaw status --usage` for packaged installs where bundled channel runtime dependencies live only under the managed runtime-deps stage. Current main keeps plain status on the metadata-only path and disables setup runtime fallback unless `--deep`/`--all`, so configured bundled channels like Discord, Slack, Nostr, and Telegram can emit missing-module warnings and disappear from the Channels table. Keep plain status fast and read-only: do not turn it into a live `channels.status` probe and do not broadly preload all plugin runtimes. Prefer either complete manifest/static metadata for configured bundled channel rows or narrowly register the active packaged bundled runtime-deps root before the existing bundled setup/status imports that require staged dependencies. Add regression coverage around `status --usage`/fast status with configured bundled channels and staged-only dependencies, update existing read-only/bundled loader tests as needed, and add a one-line user-facing `CHANGELOG.md` fix entry crediting `Thanks @tasi333`. Do not change unrelated provider/channel behavior or dependency versions.
