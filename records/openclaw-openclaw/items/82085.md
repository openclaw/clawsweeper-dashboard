## Summary

The PR should stay open: the core compaction guard direction is valid, but the current patch still changes timeout-payload synthesis for user-aborted prompt timeouts and lacks real Stop-button proof.

## What This Changes

The PR adds an `externalAbort` guard to timeout-triggered compaction and `timedOutDuringPrompt`, adds a regression case for user-aborted timeout compaction, and adds a changelog entry.

## Best Possible Solution

Keep the `externalAbort` exclusion scoped to timeout-triggered compaction, preserve timeout error payload synthesis for externally aborted prompt timeouts, and require redacted real Stop-button proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main can reach `timedOut=true`, `aborted=true`, `externalAbort=true`, high prompt usage, and no delivered message-tool payload, which makes the compaction guard fire without checking `externalAbort`. I did not run a live gateway Stop-button reproduction in this read-only review.

## Solution Assessment

No. Guarding timeout-triggered compaction is the right narrow direction, but changing `timedOutDuringPrompt` also alters terminal timeout error synthesis and can remove the expected user-facing error payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep timeout payload synthesis independent:** `src/agents/pi-embedded-runner/run.ts:2512`
  - body: The new `!externalAbort` on `timedOutDuringPrompt` disables the explicit timeout-error branch for the same scenario the test adds. With `timedOut=true`, `aborted=true`, `externalAbort=true`, and no messaging-tool delivery, aborted payload construction drops assistant text and the incomplete-turn fallback returns null, so the run can finish without the expected timeout payload. Keep `!externalAbort` on the compaction trigger only, or split the predicates.
  - confidence: 0.93
