## Summary

Keep open. Current main has useful partial hardening for live logs/session files and installed plugin extension dependencies, but normal backups still start from the whole state directory and lack a complete generated-state, size, cleanup, or retention policy; the overlapping exclude-pattern PR is still open, conflicting, and has unresolved review concerns.

## What This Changes

The issue asks OpenClaw backup/upgrade flows to avoid copying the entire large `~/.openclaw` generated state and multiplying disk usage.

## Best Possible Solution

Define and document a default backup payload taxonomy for critical versus rebuildable state, then land a reviewed narrow exclusion/retention policy or a revised exclude-pattern PR.

## Reproduction Assessment

Yes, from source: current main still seeds normal backups with the full state directory and only applies narrow hard-coded filters. The exact legacy `~/.openclaw_backup_YYYYMMDD` upgrade path was not reproduced in current main.

## Solution Assessment

Unclear pending maintainer product policy. The current partial volatile-file filter helps, but the maintainable fix needs an approved state taxonomy or revised smart-exclude design rather than another one-off skip.
