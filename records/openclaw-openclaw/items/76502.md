## Summary

Keep open. Current main still treats the resolved OpenClaw state directory as a default backup source and only filters plugin dependency trees; related log-rotation and opt-in exclude-pattern work does not yet decide the default upgrade/migration backup payload or retention policy for generated state.

## What This Changes

Requests changing upgrade and backup behavior so OpenClaw preserves critical state without duplicating large generated runtime data such as logs, browser caches, workspaces, and old backups.

## Best Possible Solution

Define and document a conservative backup payload policy for upgrade/migration backups, preserving credentials, auth profiles, config, sessions, and other critical state while excluding or truncating generated runtime data by default where safe.

## Reproduction Assessment

Yes, partially from source: current main's backup planner includes the full state root and the archive writer walks that tree with only a plugin `node_modules` filter. The exact legacy upgrade directory creation path was not reproduced on current main.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable solution is likely a documented state taxonomy plus coordinated backup defaults, not a one-off exclusion that might drop restore-critical data.
