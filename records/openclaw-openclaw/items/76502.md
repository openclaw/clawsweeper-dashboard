## Summary

Keep open. Current main still has a broad backup payload for `openclaw backup create`, and #41676 only covers the active log-growth side; this issue raises a separate backup-retention/payload-policy problem that is not implemented or superseded.

## What This Changes

Request to prevent upgrade and backup flows from duplicating large generated OpenClaw state such as logs, browser data, workspaces, and caches.

## Best Possible Solution

Define and document a conservative backup payload policy that preserves config, credentials, sessions, auth profiles, and other critical state while excluding or truncating generated runtime data by default, coordinated with #41676 for active gateway log retention.

## Reproduction Assessment

Yes, partially from source: current main's backup planner includes the whole state directory and the archive writer walks that source tree. The exact legacy upgrade directory creation path was not reproduced or traced in current main.

## Solution Assessment

Unclear pending maintainer policy. The safest fix is not a one-line exclusion; it needs a documented state taxonomy so generated data can be skipped without losing user-critical local state.
