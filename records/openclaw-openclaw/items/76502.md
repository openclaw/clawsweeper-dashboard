## Summary

Keep open: current main still plans the OpenClaw state root as a backup source and writes every included asset tree, while the related #41676 only covers active gateway log growth, not backup payload or retention policy.

## What This Changes

The issue asks OpenClaw backup and upgrade flows to stop duplicating generated runtime data such as logs, browser data, and workspaces when preserving user-critical state.

## Best Possible Solution

Define and document a conservative backup payload policy that preserves critical config, credentials, sessions, auth profiles, and provider state while excluding or truncating generated runtime data by default, coordinated with #41676 for active log retention.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path for `openclaw backup create` copying the whole state tree, including logs/browser/workspaces under it. The exact legacy `~/.openclaw_backup_YYYYMMDD` upgrade-directory path from 2026.2.26-beta.1 was not traced on current main.

## Solution Assessment

Unclear pending maintainer policy. The narrow maintainable direction is a documented state taxonomy plus targeted defaults, not an ad hoc one-line exclusion that might drop recoverable state.
