## Summary

Keep open. Current main still backs up the whole OpenClaw state tree by default and only filters installed plugin `node_modules`; related #41676 tracks active log growth, not the separate backup payload and retention policy problem.

## What This Changes

Request to prevent upgrade and backup flows from duplicating large generated OpenClaw state such as logs, browser data, workspaces, and caches.

## Best Possible Solution

Define and document a conservative backup payload taxonomy that preserves critical state while excluding or truncating generated runtime data by default, coordinated with #41676 for active gateway log retention.

## Reproduction Assessment

Partially: source inspection is high confidence for `openclaw backup create` including the whole state directory and walking it into the archive. I did not reproduce the exact legacy `~/.openclaw_backup_YYYYMMDD` upgrade-directory creation path on current main.

## Solution Assessment

Unclear pending maintainer policy. The safest fix is not a one-line exclusion; it needs a documented state taxonomy so generated data can be skipped without losing user-critical local state.
