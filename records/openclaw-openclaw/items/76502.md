## Summary

Keep open. Current main has partial backup-size hardening for volatile logs/session files and installed plugin extension dependencies, but the default backup scope still starts from the whole state directory and does not define a complete generated-state policy for `plugin-runtime-deps/`, `browser/`, `cache/`, in-state workspaces, old backup directories, or backup retention. The related log-growth and manifest-duplication issues only cover adjacent root causes, and the broader exclude-pattern PR is still open review work rather than an implemented fix.

## What This Changes

The issue asks OpenClaw to avoid multiplying disk usage by backing up large generated state under `~/.openclaw`, including logs, browser/cache data, temporary files, workspaces, and plugin runtime dependencies.

## Best Possible Solution

Define and document a default backup payload policy that preserves critical user state while excluding or separately retaining rebuildable generated data, then land either a reviewed narrow policy patch or the relevant parts of the open exclude-pattern PR.

## Reproduction Assessment

Yes, from source: normal backups still include the whole state directory and current filters only skip targeted volatile files and extension `node_modules`. The exact legacy upgrade-backup directory naming was not reproduced in current main.

## Solution Assessment

Unclear pending maintainer product policy. The current partial volatile-file filter is useful, but the maintainable fix needs a state taxonomy or reviewed smart-exclude design rather than another ad hoc single-directory skip.
