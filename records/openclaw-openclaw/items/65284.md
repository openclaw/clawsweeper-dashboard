## Summary

Keep open: the PR targets a real source-reproducible context-window collision, but the latest head still makes persisted context authoritative in reply/follow-up paths, can hide the mutating-tool retry warning, lacks real behavior proof, and is currently conflicting.

## What This Changes

The branch removes orphaned generated providers, switches several context-window lookup paths to provider-qualified resolution, adds regressions, and adds embedded-runner/cron error and rebuild-diagnostic changes.

## Best Possible Solution

Rebase and narrow the PR to provider-aware context resolution plus a documented generated-provider merge contract, with regression coverage and real after-fix proof.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id but different context windows, then leave stale generated provider state or hit the current-main gateway default path that still calls `lookupContextTokens(resolved.model)`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but persisted session context must stay a fallback, the mutating-tool warning must be preserved, and the generated-provider merge contract needs maintainer confirmation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep stored reply context as fallback:** `src/auto-reply/reply/agent-runner.ts:524`
  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before provider/model lookup, so adding `activeSessionEntry?.contextTokens` here keeps a poisoned persisted limit authoritative. Pass only the configured agent cap as the override and use the stored session value as the fallback.
  - confidence: 0.93
- **[P1] Keep stored follow-up context as fallback:** `src/auto-reply/reply/followup-runner.ts:282`
  - body: Passing `sessionEntry?.contextTokens` as the override makes any stale persisted context window bypass provider-aware lookup. In the collision scenario, follow-up accounting will keep writing the old limit instead of correcting it from the known provider/model.
  - confidence: 0.91
- **[P2] Preserve the mutating-tool warning:** `src/agents/pi-embedded-runner/run.ts:1097`
  - body: This prefers `formattedAssistantErrorText` over `genericEmptyTurnError`, so after mutating tools run a friendly disconnect/transport message can hide the warning that tool actions may already have executed. Keep that caution when `hadMutatingTools` is true.
  - confidence: 0.8
