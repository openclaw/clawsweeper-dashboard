## Summary

Keep this PR open: it targets a real provider/context-window collision, but the branch is not mergeable as written because it changes the documented models.json merge contract, keeps stale stored context authoritative in reply paths, removes transient transport failover handling, and still lacks real behavior proof.

## What This Changes

The PR changes provider merge behavior, provider-aware context resolution in session/status/runtime paths, embedded-runner error diagnostics, cron terminal payload fallback, and failover timeout classification.

## Best Possible Solution

Land a narrowed, rebased fix that uses provider-qualified lookup wherever provider and model are known, treats persisted context as fallback, preserves merge-mode provider retention unless explicitly changed, and splits unrelated diagnostics or failover taxonomy work.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave stale generated-provider or session context. Source inspection shows bare lookups and authoritative stored overrides can keep the wrong window; no live setup was run in this read-only review.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this patch also changes provider retention semantics, keeps stale context authoritative in two paths, and removes transport failover matches without a focused proof path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve merge-mode providers unless pruning is explicit:** `src/agents/models-config.merge.ts:222-223`
  - body: Starting `mergedProviders` empty deletes any existing provider absent from the next generated plan. That conflicts with the documented merge-by-default behavior and can drop unrelated custom providers during models.json regeneration; stale-provider cleanup needs a narrower explicit prune or replace contract.
  - confidence: 0.9
- **[P1] Use stored reply context only as fallback:** `src/auto-reply/reply/agent-runner.ts:524-525`
  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before provider/model lookup, so passing `activeSessionEntry?.contextTokens` here preserves the stale value this fix is meant to clear. Keep only the configured agent cap as the override and pass stored session context as fallback.
  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted `sessionEntry.contextTokens` authoritative for follow-up turns, bypassing provider-qualified lookup even when provider and model are known. A poisoned session can keep writing the old limit back instead of self-correcting after a run.
  - confidence: 0.91
- **[P2] Preserve mutating-tool retry warnings:** `src/agents/pi-embedded-runner/run.ts:1097`
  - body: Preferring the formatted provider error over the generic empty-turn message drops the warning that mutating tool actions may already have run. Users should still see that side-effect caution before retrying after an incomplete turn with mutating tool calls.
  - confidence: 0.8
- **[P2] Keep transient network failures eligible for failover:** `src/agents/failover-error.ts:242`
  - body: Narrowing this to only timeout errno codes and removing generic network/fetch patterns means common provider transport failures such as `fetch failed` or `ECONNRESET` no longer trigger fallback classification. Current behavior treats these as transient failover cases, so this can strand users on a failed provider.
  - confidence: 0.82
