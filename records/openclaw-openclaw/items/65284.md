## Summary

Keep open: the PR is aimed at a real stale context-window bug, but the current branch still has blocking correctness findings, lacks required real behavior proof, and reverses the generated-provider merge contract that current tests still assert.

## What This Changes

The branch changes generated models.json provider merging, provider-aware context lookup in gateway/session/reply/cron paths, embedded-runner empty-turn and failover diagnostics, and related regression tests.

## Best Possible Solution

Rebase and narrow the branch to provider-aware context resolution plus an explicit generated-provider merge contract, keep persisted session context only as fallback, preserve mutating-tool warnings, and split unrelated terminal-error/failover changes.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave a stale generated provider or use a bare lookup path. Current main still preserves absent existing providers in merge mode, and the resolver contract shows why an override can keep stale context authoritative.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this branch still treats persisted session context as an override, can hide mutating-tool side-effect warnings, and needs maintainer confirmation for the generated-provider merge semantics.

## Review Findings

  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before provider/model lookup, so passing `activeSessionEntry?.contextTokens` here preserves the stale value this fix is meant to clear.   - confidence: 0.91

  - confidence: 0.8
