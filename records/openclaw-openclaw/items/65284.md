## Summary

Keep open. The provider-context collision is source-reproducible and current main still has some bare lookup paths, but this PR is not mergeable as-is because it changes the documented models.json merge contract, keeps stale stored context authoritative in two reply paths, is currently conflicting, and lacks external real-behavior proof.

## What This Changes

The branch changes models.json provider merging, provider-aware context resolution in session/status/reply/cron paths, embedded-runner error and rebuild diagnostics, failover timeout taxonomy, and regression tests.

## Best Possible Solution

Land a narrowed, rebased fix that keeps documented merge-mode provider retention, resolves known-provider context windows without stale stored values as overrides, and separates unrelated runner diagnostics or taxonomy changes unless they receive their own review.

## Reproduction Assessment

Yes, for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave a stale generated provider or persisted session context. The current-main and PR-head source show how bare lookups or authoritative stored overrides can keep the wrong context window in display/accounting paths.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but the patch drops merge-mode providers, keeps stale persisted context authoritative in reply paths, and needs contributor proof plus a rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep merge-mode providers unless replace is explicit:** `src/agents/models-config.merge.ts:222-223`
  - body: Starting `mergedProviders` empty means any existing provider absent from `nextProviders` is deleted on the next models.json regeneration. That conflicts with the documented merge-by-default behavior and can drop unrelated custom provider entries; stale-provider cleanup needs a narrower contract or an explicit replace/prune mode.
  - confidence: 0.9
- **[P1] Use stored reply context only as fallback:** `src/auto-reply/reply/agent-runner.ts:524-525`
  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before provider/model lookup, so passing `activeSessionEntry?.contextTokens` here preserves the stale value this fix is meant to clear. Keep only the configured agent cap as the override and pass stored session context as the fallback.
  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted `sessionEntry.contextTokens` authoritative for follow-up turns, bypassing provider-qualified lookup even when the provider and model are known. A poisoned session can keep writing the old limit back instead of self-correcting after the run.
  - confidence: 0.91
- **[P2] Preserve the mutating-tool retry warning:** `src/agents/pi-embedded-runner/run.ts:1097`
  - body: When a turn already ran likely mutating tools, `formattedAssistantErrorText` can replace the warning that actions may have executed before retrying. Users should still see the side-effect caution even when the provider also returned a friendly transport error.
  - confidence: 0.8
