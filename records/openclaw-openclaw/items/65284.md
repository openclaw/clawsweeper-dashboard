## Summary

Keep open: the reported stale context-window collision is source-reproducible and parts of the branch are useful, but the current PR head is conflicting, missing real behavior proof, and still has blocking correctness issues around merge-mode provider retention, persisted context overrides, and mutating-tool warnings.

## What This Changes

The branch changes generated models.json merging, provider-aware context lookup in gateway/session/reply/cron persistence paths, embedded-runner diagnostics, terminal error handling, and regression tests.

## Best Possible Solution

Narrow and rebase the branch around provider-qualified context resolution, preserve stored session context only as fallback, keep merge-mode provider retention unless a deliberate pruning contract is approved, and retain user-visible side-effect warnings.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave a stale generated provider or stored session context. Current main and PR-head source show how bare lookups and authoritative overrides can keep stale context values in accounting paths.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this implementation drops merge-mode providers, treats persisted session context as an override in reply paths, and needs a rebase plus maintainer decision on provider pruning.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep merge-mode providers unless replace is explicit:** `src/agents/models-config.merge.ts:222-223`
  - body: Starting `mergedProviders` empty means any existing provider absent from `nextProviders` is deleted on the next models.json regeneration. That conflicts with the documented merge-by-default behavior and can drop unrelated custom provider entries; stale-provider cleanup needs a narrower contract or replace mode.
  - confidence: 0.9
- **[P1] Use stored reply context only as fallback:** `src/auto-reply/reply/agent-runner.ts:524-525`
  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before provider/model lookup, so passing `activeSessionEntry?.contextTokens` here preserves the stale value this fix is meant to clear. Keep only the configured agent cap as the override and pass stored session context as the fallback.
  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted `sessionEntry.contextTokens` authoritative for follow-up turns, bypassing provider-qualified lookup even when the provider and model are known. A poisoned session can keep writing the old limit back instead of self-correcting after the run.
  - confidence: 0.91
- **[P2] Preserve the mutating-tool retry warning:** `src/agents/pi-embedded-runner/run.ts:1097`
  - body: When a turn already ran likely mutating tools, `formattedAssistantErrorText` can replace the warning that actions may have executed before retrying. Users should still see the side-effect caution even when the provider also returned a friendly transport error.
  - confidence: 0.8
