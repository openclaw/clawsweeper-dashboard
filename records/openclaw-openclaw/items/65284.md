## Summary

Keep open, but this PR is not merge-ready: the stale provider context-window bug is real, yet the branch is conflicting, lacks required real behavior proof, and still introduces compatibility, session-state, and failover regressions.

## What This Changes

The branch changes models.json provider merging, provider-aware context-window resolution in session/status/reply/cron paths, embedded-runner diagnostics, failover classification, and related regression tests.

## Best Possible Solution

Land a narrowed, rebased patch that uses provider-qualified context lookup where provider and model are known, treats stored session context only as a fallback, and preserves merge-mode provider retention unless maintainers approve a documented pruning contract.

## Reproduction Assessment

Yes, at source level. Configure two providers with the same bare model id and different context windows, then leave stale generated provider or session context; source inspection shows bare lookup or authoritative stored overrides can preserve the wrong denominator.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this branch needs to preserve merge-mode compatibility, treat persisted context as fallback, avoid unrelated failover regressions, and provide real after-fix proof.
