## Summary

Keep this PR open: the stale-provider context-window bug is real and still partially visible on current main, but this branch is not merge-ready because it preserves stale session context as an authoritative override in two changed paths, weakens the mutating-tool retry warning, mixes unrelated terminal-error behavior into the context fix, and lacks external real behavior proof.

## What This Changes

The branch removes existing-provider seeding in generated `models.json` merging, switches several session/status/runtime context-window lookups to provider-qualified resolution, adds regressions, and changes embedded-runner and cron empty-turn error handling.

## Best Possible Solution

Rebase and narrow the branch to provider-aware context resolution plus an explicit generated-provider merge contract, keep persisted session context only as fallback, preserve the mutating-tool warning, and split terminal-error behavior into separate work.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave a stale generated provider or use the remaining bare lookup paths on current main. I did not run tests because this review is read-only, but the code paths and existing assertions make the failure mode clear.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but the PR should not treat persisted session context as an override, should not hide mutating-tool side-effect warnings, and needs maintainer confirmation for the generated-provider merge contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use stored reply context only as fallback:** `src/auto-reply/reply/agent-runner.ts:524-525`
  - body: `resolveContextTokensForModel` returns `contextTokensOverride` before it does any provider/model lookup, so passing `activeSessionEntry?.contextTokens` here preserves exactly the stale value this fix is trying to clear. Keep only the configured agent cap as the override and pass the stored session value as fallback after provider-aware lookup fails.
  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted `sessionEntry.contextTokens` authoritative for follow-up turns, bypassing the provider-qualified lookup even when `provider` and `model` are known. A poisoned session will keep writing the old limit back instead of self-correcting after the run.
  - confidence: 0.91
- **[P2] Preserve the mutating-tool retry warning:** `src/agents/pi-embedded-runner/run.ts:1078`
  - body: When a turn has already run likely mutating tools, `formattedAssistantErrorText` can now replace the existing warning that actions may have executed before retrying. Users should still see the side-effect caution, even if the provider also returned a friendly transport error.
  - confidence: 0.8
