## What This Changes

## Best Possible Solution

Land a narrowed, rebased patch that uses provider-qualified lookup where provider and model are known, treats stored session context only as fallback, preserves merge-mode provider retention, and splits unrelated failover/diagnostic changes.

## Reproduction Assessment

Yes, at source level. Configure two providers with the same bare model id and different context windows, then leave stale generated provider or session context; source inspection shows bare lookup or authoritative stored overrides can preserve the wrong denominator.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this branch needs to preserve merge-mode compatibility, treat persisted context as fallback, avoid unrelated failover regressions, and provide real after-fix proof.

  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted `sessionEntry.contextTokens` authoritative for follow-up turns, bypassing provider-qualified lookup even when provider and model are known. A poisoned session can keep writing the old limit back instead of self-correcting after a run.
  - confidence: 0.91

  - body: Removing generic network and errno matches means common provider transport failures like `fetch failed` or `ECONNRESET` no longer trigger fallback classification. Current code treats these as transient failover cases, so the taxonomy change can strand users on a failed provider.
  - confidence: 0.84
