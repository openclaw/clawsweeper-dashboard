## Summary

Keep this PR open for revision: the stale-provider/context-window bug is still visible on current main, but the branch is not merge-ready because it can keep poisoned session context values authoritative and drops an existing side-effect warning in the embedded runner error path.

## What This Changes

The PR removes existing-provider seeding from generated `models.json` merging, switches several session/status/runtime context-token paths toward provider-qualified lookup, adds focused regressions, and also changes embedded-runner and cron empty-turn error handling.

## Best Possible Solution

Rebase and narrow the branch to provider-aware context resolution plus an explicit generated-provider merge contract, keep persisted session context as fallback rather than override, preserve the mutating-tool warning, split terminal-error behavior into separate work, and add the required changelog entry.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers exposing the same bare model id with different context windows, leave a stale generated provider in merge mode, and the current main bare lookup paths can select the wrong cap. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but the branch needs rebase/revision so it does not preserve poisoned session context as authoritative or mix unrelated terminal-error behavior into the context fix.
