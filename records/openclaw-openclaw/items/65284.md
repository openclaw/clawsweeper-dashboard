## Summary

Keep open: the provider-qualified lookup direction is valid, but this PR head still conflicts with the documented merge-mode provider contract, keeps stale stored context authoritative in reply paths, removes transient network failover matches, lacks real behavior proof, and overlaps current main fixes.

## What This Changes

This PR changes models.json provider merging, provider-qualified context-token lookup in session/reply/cron/gateway paths, embedded-runner diagnostics, transport failover classification, and related tests.

## Best Possible Solution

Land a narrowed, rebased fix that uses provider-qualified lookup where provider and model are known, treats persisted context as fallback, and preserves merge-mode provider retention unless an explicit pruning contract is approved.

## Reproduction Assessment

Yes for source-level reproduction: configure two providers with the same bare model id and different context windows, then leave stale generated provider or session context. Source inspection shows bare lookups or authoritative stored overrides can keep the wrong window in session/status accounting paths; no live setup was run in this read-only review.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this patch drops merge-mode providers, keeps stale persisted context authoritative in reply paths, removes network failover matches, and needs proof plus a rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve merge-mode providers unless pruning is explicit:** `src/agents/models-config.merge.ts:222-223`
  - body: Starting mergedProviders empty deletes any existing provider absent from the next generated plan. That conflicts with the documented merge-by-default behavior and can drop unrelated custom providers when models.json regenerates; stale-provider cleanup needs a narrower explicit prune/replace contract.
  - confidence: 0.9
- **[P1] Use stored reply context only as fallback:** `src/auto-reply/reply/agent-runner.ts:524-525`
  - body: resolveContextTokensForModel returns contextTokensOverride before provider/model lookup, so passing activeSessionEntry?.contextTokens here preserves the stale value this fix is meant to clear. Keep only the configured agent cap as the override and pass stored session context as fallback.
  - confidence: 0.93
- **[P1] Use stored follow-up context only as fallback:** `src/auto-reply/reply/followup-runner.ts:282-283`
  - body: This makes any persisted sessionEntry.contextTokens authoritative for follow-up turns, bypassing provider-qualified lookup even when provider and model are known. A poisoned session can keep writing the old limit back instead of self-correcting after a run.
  - confidence: 0.91
- **[P2] Preserve mutating-tool retry warnings:** `src/agents/pi-embedded-runner/run.ts:1097`
  - body: When a provider returns a recognizable transport error, formattedAssistantErrorText replaces the warning that mutating tools may already have run. Users should still see the side-effect caution before retrying after an incomplete turn with mutating tool calls.
  - confidence: 0.8
- **[P2] Keep transient network failures eligible for failover:** `src/agents/pi-embedded-helpers/failover-matches.ts:34-36`
  - body: Removing generic network and errno matches means common provider transport failures like fetch failed or ECONNRESET no longer trigger fallback classification. Current code treats these as transient failover cases, so the taxonomy change can strand users on a failed provider.
  - confidence: 0.82
