## What This Changes

The branch changes generated models.json merging, provider-qualified context-token lookup in session/reply/cron paths, embedded-runner diagnostics, failover classification, and regression tests.

## Best Possible Solution

Land a narrowed, rebased fix that uses provider-qualified context lookup where provider and model are known, treats stored session context only as fallback, and preserves merge-mode provider retention unless maintainers explicitly approve a documented prune contract.

## Reproduction Assessment

Yes, at source level. Configure two providers with the same bare model id and different context windows, then leave stale generated provider or session context; source inspection shows bare or authoritative stored lookups can preserve the wrong denominator.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but the branch needs to preserve merge-mode compatibility, treat persisted context as fallback, avoid unrelated failover regressions, and add real after-fix proof.
