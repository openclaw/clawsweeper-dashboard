## Summary

Keep open: the context-window bug is plausible and source-reproducible, but this PR is still merge-conflicting, lacks required real behavior proof, and introduces blocking compatibility, session-state, and failover regressions.

## What This Changes

The branch changes provider-aware context-token resolution and models.json merge behavior across gateway/session/commands/reply/cron paths, plus embedded-runner diagnostics, failover classification, and regression tests.

## Best Possible Solution

Land a narrowed, rebased fix that uses provider-qualified context lookup when provider and model are known, keeps persisted session context as fallback, and preserves merge-mode provider retention unless maintainers explicitly approve a documented prune path.

## Reproduction Assessment

Yes, at source level. Configure two providers with the same bare model id and different context windows, then leave stale generated provider or session context; current bare lookups and authoritative stored-context overrides show how the wrong denominator can persist.

## Solution Assessment

No, not as currently proposed. Provider-qualified lookup is the right direction, but this branch needs a narrower rebase, stored context as fallback, merge-mode compatibility, and no unrelated failover regression.
