## Summary

Keep open: the PR is a useful UI-side mitigation, but current main still allows stale session rows to arrive without `totalTokens`, so the changed helper can still return `null` in the reported browser path. The existing focused test also still expects stale totals to hide the notice, and the supplied proof only exercises a synthetic helper call rather than the real Control UI send-message/session-row flow.

## What This Changes

The branch updates the Control UI context notice and CSS so stale token totals render with a `~` prefix, stale title text, and muted italic styling.

## Best Possible Solution

Define and test the stale-token display contract at the session-row boundary, render stale usage only when a usable stale total is present, preserve the hidden state for missing totals, and keep the separate `/status` mismatch tracked outside this PR.

## Reproduction Assessment

Yes from source: current main hides the notice on `totalTokensFresh === false`, and current session-row tests show stale rows may omit `totalTokens`. I did not run the browser flow because this review is read-only.

## Solution Assessment

No. The PR is a plausible helper-level mitigation, but the best fix must prove or preserve the stale total on the real `GatewaySessionRow` path that `renderContextNotice` receives.
