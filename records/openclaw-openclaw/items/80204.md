## Summary

Keep open: the TLS bootstrap bug is valid and the narrow shared-resolver direction is maintainable, but this PR is still conflicting and carries unrelated release, dependency, generated protocol, plugin-version, agent, and workflow churn that should not merge with the gateway fix.

## What This Changes

The branch adds shared gateway TLS fingerprint resolution, forwards bootstrap TLS fingerprints into ACP, MCP, and operator approval GatewayClient construction, and adds regression tests, while also changing unrelated release, dependency, generated, and plugin-version surfaces.

## Best Possible Solution

Land a narrow rebased TLS bootstrap patch on current main that shares the existing callGateway fingerprint precedence, forwards the resolved value through ACP/MCP/operator approval clients, preserves protocol models, and keeps release/dependency churn out of the bug fix.

## Reproduction Assessment

Yes, at source level. Current main's bootstrap helper does not return tlsFingerprint, while ACP/MCP/operator approval clients construct GatewayClient from it and GatewayClient only trusts a self-signed WSS certificate when tlsFingerprint is supplied.

## Solution Assessment

Yes for the shared TLS resolver and pass-through; no for the branch as submitted. The maintainable solution is a narrow current-main patch that excludes release, dependency, generated protocol, and unrelated agent/provider churn.
