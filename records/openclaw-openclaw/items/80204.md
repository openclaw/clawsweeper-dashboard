## Summary

Keep open: this is a recent, focused bug-fix PR, and current main still lacks TLS fingerprint propagation for bootstrap clients. The patch uses the existing GatewayClient pinning contract, includes targeted coverage and terminal proof, and I found no blocking correctness or security issue.

## What This Changes

The PR adds a shared gateway TLS fingerprint resolver, wires it into client bootstrap and callGateway, forwards the resolved fingerprint to ACP, MCP, and operator approval GatewayClients, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land the focused bootstrap propagation after normal maintainer review, keeping TLS fingerprint precedence centralized and locked by the targeted regression tests.

## Reproduction Assessment

Yes. This is source-reproducible on current main because local TLS fingerprint selection exists in callGateway while resolveGatewayClientBootstrap and its ACP/MCP/approval callers do not carry tlsFingerprint; the PR body also includes after-fix terminal proof, but I did not run a live repro in this read-only review.

## Solution Assessment

Yes. Centralizing the fingerprint selection helper and forwarding bootstrap.tlsFingerprint into each GatewayClient is the narrowest maintainable fix because it reuses the existing GatewayClient pinning contract and avoids duplicate precedence rules.
