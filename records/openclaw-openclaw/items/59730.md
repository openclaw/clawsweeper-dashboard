## What This Changes

The issue requests consolidating OpenClaw's explicit Vitest shard/config layout into one `vitest.config.ts` API controlled by `VITEST_MODE` or a factory.

## Best Possible Solution

Have test-infrastructure owners choose the lane API first, then either reduce duplication behind shared helpers while preserving explicit shard configs or intentionally migrate scripts, CI, docs, and tests to a documented mode-based contract.

## Reproduction Assessment

## Solution Assessment

Unclear; the proposed single `VITEST_MODE` config is not obviously the best solution because current scripts, docs, CI routing, shard weights, changed-test routing, and tests depend on explicit config paths.
