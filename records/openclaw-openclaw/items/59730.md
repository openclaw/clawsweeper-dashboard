## Summary

Keep open. Current main still uses a multi-config, shard-oriented Vitest model, and the package scripts, CI plan, tests, and docs all encode that model, so this remains an unresolved test-infrastructure design decision rather than a cleanup-close candidate.

## What This Changes

The issue requests consolidating the repo’s multiple Vitest configs into one mode-driven config and migrating scripts, CI, docs, and old config files accordingly.

## Best Possible Solution

Define the desired test-lane contract first, then reduce duplication behind shared helpers or generated metadata while preserving scoped lane routing unless maintainers intentionally choose a new mode-based API.

## Solution Assessment
