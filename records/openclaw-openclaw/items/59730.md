## Summary

Keep open. Current main still intentionally uses a multi-config, shard-oriented Vitest test-lane model, and replacing it with one `VITEST_MODE`-driven config would be a test-infrastructure design decision rather than a cleanup close candidate.

## What This Changes

The issue requests consolidating the repository’s multiple Vitest config files into a single mode-aware configuration and updating scripts, CI, and docs around that contract.

## Best Possible Solution

Have the test-infrastructure owners define the desired lane contract first, then either reduce duplication behind shared helpers while preserving explicit shard routing or intentionally migrate scripts, CI, docs, and tests to a new mode-based API.

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? Unclear; a single `VITEST_MODE` switch is not obviously the narrowest maintainable fix because current scripts, CI, docs, tests, and scheduling code rely on explicit config and shard contracts.
