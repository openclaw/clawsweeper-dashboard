## Summary

Keep open. The reported cache-invalidation problem is plausible, but this PR is not merge-ready: it moves every Project Context file below the cache boundary, which would make stable workspace context uncached, and it leaves prompt-order tests/docs inconsistent with the new ordering.

## What This Changes

The PR moves stable project context file rendering in `buildAgentSystemPrompt` from before to after `SYSTEM_PROMPT_CACHE_BOUNDARY`.

## Best Possible Solution

Keep the cacheable stable workspace files above the boundary, move only genuinely volatile context such as auto-written `MEMORY.md` or explicitly volatility-marked plugin context below it, and update the prompt-order tests, docs, and changelog to lock that behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: build a prompt with `MEMORY.md`, inspect its position relative to `SYSTEM_PROMPT_CACHE_BOUNDARY`, and compare that with `applyAnthropicCacheControlToSystem`, which only cache-controls the prefix before the boundary.

## Solution Assessment

No. Moving all Project Context below the boundary solves invalidation by making that whole section uncached; the narrower maintainable fix is to classify volatile context separately while preserving cache reuse for stable workspace files.
