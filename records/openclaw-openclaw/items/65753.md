## Summary

Keep this PR open. Current main still only skips triple-backtick fenced blocks in the docs link audit, so the PR addresses a real gap, but the proposed parser uses the same prefix match for opening and closing fences and can resume scanning inside a valid code fence when a content line starts with a same-length marker plus trailing text.

## What This Changes

The PR updates the docs link-audit script to parse backtick and tilde fence delimiters with marker-length tracking and adds unit tests for delimiter parsing.

## Best Possible Solution

Merge a small version of this docs tooling fix after the close-fence logic distinguishes opening and closing lines, requires same marker plus sufficient length with only trailing whitespace for closes, and adds audit-level regressions for tilde and longer fences.

## Reproduction Assessment

Yes. Current main can be reproduced with a tilde-fenced Markdown block containing an invalid internal route: the audit loop scans the link because it only toggles on triple backticks. The proposed edge is also reproducible with a same-marker content line like `~~~not-a-close` before a link inside the fence.

## Solution Assessment

No. The direction is right, but using one prefix parser for both opening and closing is not the narrowest correct fix; the close path should validate that a closing fence has no non-whitespace trailing content.
