## Summary

Keep this PR open. The reported docs-link-audit bug is real, but the proposed patch still exits a fence on content like `~~~not-a-close`, so it can report links that remain inside the same code block.

## What This Changes

The PR modifies `scripts/docs-link-audit.mjs` to parse backtick and tilde fenced-code delimiters with marker length and adds parser unit tests in `src/scripts/docs-link-audit.test.ts`.

## Best Possible Solution

Land a narrow docs tooling fix that tracks marker and length, parses opener and closer rules separately, and adds audit-loop regressions for tilde fences and content lines beginning with fence markers.

## Reproduction Assessment

Yes. A tilde-fenced Markdown block containing `[bad](/missing-openclaw-route)` is still scanned by current main because the audit loop only toggles on backtick fences; the PR edge case also reproduces when `~~~not-a-close` prematurely clears the proposed fence state.

## Solution Assessment

No. The PR direction is right, but the close path must distinguish opening or info-string lines from true closing fence lines that have only trailing whitespace after the marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Require real closing fence lines:** `scripts/docs-link-audit.mjs:351-355`

A content line like `~~~not-a-close` inside a `~~~md` block clears `codeFence`, so the following `[bad](/route)` is audited even though it is still code; closing fences should require the marker/length match plus only trailing whitespace.
