## Summary

Keep open: the docs-audit bug is real, but this PR still closes fences on any matching marker prefix, so content like `~~~not-a-close` can resume link auditing inside code; it also lacks inspectable after-fix proof.

## What This Changes

The branch modifies `scripts/docs-link-audit.mjs` and `src/scripts/docs-link-audit.test.ts` to track fenced-code markers and add parser unit coverage.

## Best Possible Solution

Land a narrow docs tooling fix that separates opener and closer parsing, requires closing fences to match marker and length with only trailing whitespace, and adds audit-loop regressions for tilde fences and marker-prefixed content.

## Reproduction Assessment

Yes. A read-only in-memory run of the current audit loop reports `/missing-openclaw-route` inside a `~~~md` block, and a simulation of the PR logic reports `/still-inside-code` after `~~~not-a-close`.

## Solution Assessment

No. The PR is aimed at the right script, but using one prefix parser for both openers and closers is not sufficient because closers must reject non-whitespace suffix text.
