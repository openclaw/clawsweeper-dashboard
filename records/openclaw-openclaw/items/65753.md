## Summary

Keep this PR open. Current main still has the docs-link audit bug for tilde fences, but the proposed branch has a narrow correctness issue: it uses the same prefix parser for both fence openers and closers, so content like `~~~not-a-close` inside a fence can prematurely re-enable link auditing.

## What This Changes

The PR adds a `parseFenceDelimiter` helper to support backtick and tilde fenced-code markers with marker-length tracking in the docs link audit, and adds parser-level unit tests.

## Best Possible Solution

Land a narrow docs tooling fix that parses opening and closing fence lines separately, matches marker type and opening length, requires only trailing whitespace on close lines, and covers the audit loop with tilde-fence and longer-fence regressions.

## Reproduction Assessment

Yes. Current main can be reproduced with a tilde-fenced Markdown block containing an invalid internal route; the audit loop scans the link because it only toggles on backtick fences. The proposed patch edge also reproduces with `~~~not-a-close` before a link inside the same fence.

## Solution Assessment

No. The PR direction is right, but using one prefix parser for both openers and closers is not the narrowest correct fix; the close path should validate a real closing fence line per CommonMark.
