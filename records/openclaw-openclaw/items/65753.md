## Summary

Keep open and request changes: current main still has the tilde-fence false-positive bug, but this PR still treats content lines like `~~~not-a-close` as closing fences, so the proposed patch is not ready to merge.

## What This Changes

The PR updates the docs link audit to parse backtick and tilde fence delimiters with marker length and adds unit tests for the parser helper.

## Best Possible Solution

Land a narrow docs tooling fix that separates opener and closer parsing, requires closing fences to have matching marker/length plus only trailing whitespace, and adds audit-loop regressions for tilde fences and marker-prefixed content inside fences.

## Reproduction Assessment

Yes. A read-only in-memory run of the current audit loop reports `/missing-openclaw-route` inside a `~~~md` block, and a second simulation of the PR logic reports `/still-inside-code` after `~~~not-a-close`.

## Solution Assessment

No. The PR direction is the right owner boundary, but using one prefix parser for both openers and closers is not sufficient because closers must reject non-whitespace suffix text.
