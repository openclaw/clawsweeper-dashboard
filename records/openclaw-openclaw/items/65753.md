## Summary

Keep this PR open: current main still has the tilde-fence audit bug, but the proposed patch still treats marker-prefixed content as a closing fence, so it needs a focused repair before merge.

## What This Changes

The PR updates the docs link audit to track backtick and tilde code fences by marker and length and adds parser unit tests.

## Best Possible Solution

Land a narrow docs tooling fix that follows opener and closer fence rules for backtick and tilde markers, with regressions for tilde fences and marker-prefixed content lines.

## Reproduction Assessment

Yes. A minimal read-only check against the current-main loop shows the audit still collects links inside `~~~` fences because the source only toggles on backtick fences.

## Solution Assessment

No. The PR direction is right, but the close path must distinguish true closing fence lines from content or info-string lines such as `~~~not-a-close`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Require real closing fence lines:** `scripts/docs-link-audit.mjs:351-355`

A content line like `~~~not-a-close` inside a `~~~md` block clears `codeFence`, so the following link is audited even though it is still code; closing fences should require the marker/length match plus only trailing whitespace.
