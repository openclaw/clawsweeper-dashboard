## Summary

Keep open: this is a fresh, concrete Telegram message-loss report, and current main still has source-supported Telegram streaming/finalization paths where final text can be treated as delivered without a durable Telegram send. The older open Telegram partial-stream issue is related, but this report adds a group-chat plus non-adaptive-thinking matrix that is not yet covered closely enough to close as a duplicate.

## What This Changes

The issue asks OpenClaw to deliver Telegram group replies reliably when Telegram streaming is enabled for models that do not support adaptive thinking.

## Best Possible Solution

Make Telegram streaming finalization for group replies end in a confirmed preview final, a durable sendMessage fallback, or an explicit delivery failure for affected reasoning/answer lane shapes.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows current-main paths where Telegram streaming finalization can return without a durable final send or can mark a preview delivered without a message id, but the exact non-adaptive group-chat trigger still needs a failing regression or live proof.

## Solution Assessment

Unclear until the exact matrix is reproduced. The best direction is a narrow Telegram dispatch and preview-finalization fix, not disabling streaming or changing model thinking policy.
