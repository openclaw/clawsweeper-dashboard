## Summary

Keep this PR open: current main still lets unset IRC `blockStreaming` inherit the global block-streaming default, so the proposed IRC-local final-only default remains a valid focused bug fix. It should not merge yet because the external PR lacks after-fix real IRC proof and its tests target an old test file layout.

## What This Changes

The PR changes IRC inbound reply options so block streaming is disabled unless the IRC account explicitly sets `blockStreaming: true`, and adds regression coverage for unset, false, and true account config states.

## Best Possible Solution

Refresh the IRC-local resolver onto current main, place regression coverage in the current IRC inbound behavior tests, and merge after redacted real IRC or terminal/log proof is added.

## Reproduction Assessment

Yes, at source level. Current main passes `disableBlockStreaming: undefined` for unset IRC `blockStreaming`, and the shared resolver then inherits `agents.defaults.blockStreamingDefault` despite docs requiring explicit channel opt-in.

## Solution Assessment

Yes functionally. A channel-local resolver that disables block streaming unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix, but this branch still needs refresh and real behavior proof.
