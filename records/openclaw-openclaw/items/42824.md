## Summary

Keep open: current main and v2026.5.12 still let unset IRC `blockStreaming` inherit the global default despite the documented opt-in contract. The branch is the right small fix direction, but it is currently merge-conflicting, its tests target a removed file, and the external PR still lacks real behavior proof.

## What This Changes

The branch changes IRC inbound reply options so unset or false account `blockStreaming` passes `disableBlockStreaming: true`, with regression coverage for unset, false, and true config states.

## Best Possible Solution

Rebase the IRC-local opt-in resolver onto current main, move the regression coverage into the current IRC inbound behavior tests, and merge only after redacted real IRC or terminal/log proof shows full final replies when IRC `blockStreaming` is unset.

## Reproduction Assessment

Yes, at source level. With `agents.defaults.blockStreamingDefault` on and IRC account `blockStreaming` unset, current main passes `disableBlockStreaming: undefined`, so the shared resolver enables block streaming despite the documented IRC opt-in contract.

## Solution Assessment

Yes, functionally. Passing a concrete IRC-local disable boolean unless `blockStreaming: true` is explicitly set is the narrow maintainable fix, but this branch needs a rebase, test relocation, and real behavior proof before merge.
