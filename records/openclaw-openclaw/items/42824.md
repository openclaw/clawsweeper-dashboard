## Summary

Keep this PR open: current main still lets unset IRC `blockStreaming` inherit the global block-streaming default, and the PR’s IRC-local boolean override is the right narrow fix. ## What This Changes

The PR changes IRC inbound reply options to disable block streaming unless the account sets `blockStreaming: true`, with regression coverage for unset, false, and true config states.

## Best Possible Solution

Refresh the IRC-local resolver onto current main, put regression coverage in `extensions/irc/src/inbound.behavior.test.ts`, and merge after redacted real IRC or terminal/log proof confirms full final replies.

## Reproduction Assessment

Yes, at source level. Current main passes `disableBlockStreaming: undefined` for unset IRC config, and the shared resolver then inherits `agents.defaults.blockStreamingDefault` despite docs requiring explicit channel opt-in.

## Solution Assessment

Yes functionally. A channel-local resolver that sends a concrete disable boolean unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix; the branch just needs refresh and real behavior proof.
