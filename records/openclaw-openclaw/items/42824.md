## What This Changes

The PR changes IRC inbound reply options so block streaming is disabled unless an IRC account explicitly sets `blockStreaming: true`, with regression coverage for unset, false, and true config states.

## Best Possible Solution

Refresh the IRC-local resolver onto current main, place regression coverage in `extensions/irc/src/inbound.behavior.test.ts`, and merge after redacted real IRC or terminal/log proof confirms full final replies when IRC `blockStreaming` is unset.

## Reproduction Assessment

Yes, at source level. Current main passes `disableBlockStreaming: undefined` for unset IRC config, and the shared resolver then inherits `agents.defaults.blockStreamingDefault` despite the documented channel opt-in contract.

## Solution Assessment

Yes functionally. A channel-local resolver that sends a concrete disable boolean unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix; the branch still needs a current-main refresh and real behavior proof.
