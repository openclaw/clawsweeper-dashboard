## Summary

Keep open: current main still lets unset IRC `blockStreaming` inherit the global block-streaming default despite the documented channel opt-in contract, and this PR is a narrow fix. It still needs real behavior proof and a branch refresh before merge.

## What This Changes

The PR changes IRC inbound reply options so block streaming is disabled unless the IRC account explicitly sets `blockStreaming: true`, and adds config-state regression tests.

## Best Possible Solution

Refresh the IRC-local opt-in resolver onto current main, cover unset/false/true config states in the current inbound behavior tests, and merge after redacted real IRC or terminal/log proof confirms full final replies when IRC `blockStreaming` is unset.

## Reproduction Assessment

Yes, at source level. Set `agents.defaults.blockStreamingDefault` on and leave IRC account `blockStreaming` unset: current main passes `disableBlockStreaming: undefined`, and the shared resolver enables block streaming despite the documented IRC opt-in contract.

## Solution Assessment

Yes, functionally. A channel-local resolver that passes a concrete disable boolean unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix, but the branch needs refresh and real behavior proof before merge.
