## Summary

Keep open: current `main` and the latest release still let IRC inherit the global block-streaming default when `channels.irc.blockStreaming` is unset, while the docs say non-Telegram channels require explicit opt-in. The PR’s central fix is still useful, but it needs a current-main refresh and real behavior proof before merge.

## What This Changes

This PR changes IRC inbound replies so unset or false `channels.irc.blockStreaming` sends `disableBlockStreaming: true`, with regression coverage for the opt-in behavior.

## Best Possible Solution

Land a refreshed IRC-scoped fix that disables block streaming unless the IRC account explicitly sets `blockStreaming: true`, with focused regression coverage in current test files and redacted live IRC proof.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.blockStreamingDefault: "on"` with IRC `blockStreaming` unset and current `handleIrcInbound` passes `undefined`, which the shared reply directive resolver treats as permission to use the global default.

## Solution Assessment

Yes. The IRC-scoped override is the narrowest maintainable fix for the documented non-Telegram opt-in contract; the broader streaming-boundary issue at https://github.com/openclaw/openclaw/issues/42106 is separate.
