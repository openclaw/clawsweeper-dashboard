## Summary

Keep open: current main and v2026.5.12 still let unset IRC `blockStreaming` inherit the global block-streaming default, while this PR carries the narrow IRC opt-in fix. The code direction looks correct, but the external PR still lacks after-fix real behavior proof and needs a branch refresh for the current test layout.

## What This Changes

The branch changes IRC inbound reply options so unset or false account `blockStreaming` passes `disableBlockStreaming: true`, with regression tests for unset, false, and true config states.

## Best Possible Solution

Rebase the IRC-local opt-in resolver onto current main, adapt the regression coverage to the current inbound behavior test surface, and merge after redacted real IRC or terminal/log proof shows full final replies when IRC `blockStreaming` is unset.

## Reproduction Assessment

Yes, at source level. With `agents.defaults.blockStreamingDefault` on and IRC account `blockStreaming` unset, current main passes `disableBlockStreaming: undefined`, so the shared resolver enables block streaming despite the documented IRC opt-in contract.

## Solution Assessment

Yes, functionally. A channel-local resolver that passes a concrete disable boolean unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix; the branch still needs refresh and real behavior proof before merge.
