## Summary

Keep open: current main and v2026.5.7 still let IRC inherit the global block-streaming default when IRC account config is unset, while this PR’s IRC-local override is a plausible narrow fix; merge remains blocked by missing real behavior proof and a stale test target.

## What This Changes

The PR changes IRC inbound reply options to pass a concrete `disableBlockStreaming` unless IRC `blockStreaming` is explicitly true, and adds regression tests for unset, false, and true account config states.

## Best Possible Solution

Refresh the IRC-local resolver onto current main, move the regression coverage into the current IRC inbound behavior tests, and merge only after redacted real IRC or terminal/log proof is added.

## Reproduction Assessment

Yes, at source level. Current main passes `disableBlockStreaming: undefined` for unset IRC `blockStreaming`, and the shared resolver then inherits `agents.defaults.blockStreamingDefault` despite docs requiring explicit channel opt-in.

## Solution Assessment

Yes functionally. A channel-local resolver that disables block streaming unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix, but the PR still needs a current-main refresh and real behavior proof.
