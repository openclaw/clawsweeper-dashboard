## Summary

Keep open: current main still lets IRC inherit the global block-streaming default when the IRC account does not set `blockStreaming`, and the PR is a narrow functional fix, but it still needs contributor real-behavior proof and a branch refresh before merge.

## What This Changes

The PR changes IRC inbound reply options so block streaming is disabled unless the IRC account explicitly sets `blockStreaming: true`, and adds regression coverage for unset, false, and true config states.

## Best Possible Solution

Refresh the IRC-local resolver onto current main, move the regression coverage into the current inbound behavior test file, and merge after redacted real IRC or terminal/log proof confirms full final replies when IRC `blockStreaming` is unset.

## Reproduction Assessment

Yes, at source level. Current main passes `disableBlockStreaming: undefined` for unset IRC config, and the shared resolver then inherits `agents.defaults.blockStreamingDefault` despite the documented channel opt-in contract.

## Solution Assessment

Yes functionally. A channel-local resolver that passes a concrete disable boolean unless IRC explicitly sets `blockStreaming: true` is the narrow maintainable fix, but the branch still needs refresh and real behavior proof.
