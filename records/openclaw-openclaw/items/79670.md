## Summary

Keep open. Current main still sets Google API-key onboarding to `google/gemini-3.1-pro-preview`, the docs and tests still reflect that preview default, and the related stable-default PR was closed unmerged, so this remains a provider default-policy issue rather than cleanup closure.

## What This Changes

The issue requests that Google onboarding stop recommending Gemini preview defaults without quota awareness, either by using stable defaults or warning/displaying per-model quota limits.

## Best Possible Solution

Define the Google API-key onboarding policy explicitly: use a production-safe stable default or keep the preview default only with clear quota/rate-limit warnings and synchronized docs/tests, while handling failover cooldown behavior in its own tracked issues.

## Reproduction Assessment

Yes for the source-level onboarding path: current main deterministically wires Google API-key onboarding to `google/gemini-3.1-pro-preview`. No for the full Tier 1 quota and cost cascade, because that requires a quota-limited Google account and private billing/runtime evidence.

## Solution Assessment

No for an automated one-line stable-default change right now. The stable-default direction is plausible, but the maintainable fix needs provider-owner policy alignment plus docs, tests, and possibly warning/quota UX.
