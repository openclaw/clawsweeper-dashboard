## Summary

Keep this PR open: the reported SecretRef hydration problem is real and the Codex context metadata update is useful, but the proposed SecretRef fix erases configured refs into `undefined`, which can fall through to ambient/provider credentials instead of resolving through the runtime snapshot or failing closed. The related merged Discord fix in #76987 is the better pattern to copy for these paths.

## What This Changes

This PR replaces strict SecretRef normalization with non-throwing normalization in skill, web-search, memory-host, and Ollama embedding credential paths, and updates Codex fallback model context metadata.

## Best Possible Solution

Preserve the Codex catalog metadata update, but repair SecretRef handling with active runtime snapshot selection or explicit configured-unavailable status so configured refs resolve when available and fail closed otherwise, with focused tests and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main shows strict SecretRef readers can throw on raw refs, and existing tests define the expected fail-closed behavior for unresolved memory/web credential refs.

## Solution Assessment

No. Switching to `normalizeSecretInputString` avoids the throw by treating raw SecretRefs as absent; the narrower maintainable fix is to select the active runtime snapshot or carry configured-unavailable status without ambient fallback.
