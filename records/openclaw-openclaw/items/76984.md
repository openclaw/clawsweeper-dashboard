## Summary

Keep open: the PR targets a real current-main SecretRef hydration problem and a useful Codex catalog correction, but the proposed SecretRef fix erases configured raw SecretRef state in several credential selectors, which can fall through to ambient/provider credentials instead of resolving from the runtime snapshot or failing closed. The related merged #76987 Discord fix shows the safer pattern: select the active runtime snapshot and carry `configured_unavailable` status explicitly.

## What This Changes

The branch replaces strict SecretRef normalization with non-throwing normalization in skill, web-search, memory, and Ollama credential readers, exports a memory-host helper, and updates Codex fallback context metadata.

## Best Possible Solution

Keep the Codex metadata update, but repair SecretRef handling with active runtime snapshot selection or inspect-mode `configured_unavailable` state so configured credentials resolve when available and fail closed when not, with focused tests and a changelog entry.

## Reproduction Assessment

Yes, source inspection and existing tests give a high-confidence reproduction path: current strict readers throw on raw SecretRefs before runtime snapshots, and the PR diff changes those raw refs into ordinary `undefined` in fallback-sensitive credential paths.

## Solution Assessment

No. Returning `undefined` for raw SecretRefs avoids the immediate throw but loses configured-ref state; the narrower maintainable fix is runtime-snapshot selection or explicit configured-unavailable status, matching the related Discord fix.
