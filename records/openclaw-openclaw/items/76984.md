## Summary

Keep open: the PR targets a real SecretRef regression and the Codex context update is directionally supported, but the patch currently treats configured-but-unavailable SecretRefs as absent, which can trigger fallback credentials, and it leaves an existing fail-fast test stale.

## What This Changes

The PR switches several SecretRef readers from throwing normalization to non-throwing string normalization and updates the Codex provider catalog to expose a 400K context window plus a 272K input-token cap.

## Best Possible Solution

Preserve a distinct configured-unavailable SecretRef state so hot paths can defer or diagnose without falling back to ambient credentials, then add regression coverage for the skill, web-search, memory remote, Ollama embedding, and Codex context-cap paths.

## Reproduction Assessment

Yes, source-backed: current main's strict SecretRef normalizer throws on raw apiKey SecretRefs, and the affected callers pass those values through before relying on a resolved runtime snapshot. The existing Ollama embedding test also encodes one current fail-fast path.

## Solution Assessment

No. The direction is plausible, but using normalizeSecretInputString collapses configured-but-unavailable and not-configured into the same undefined result; an inspect/tri-state path would be safer and more consistent with the SecretRef contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve configured-unavailable SecretRefs:** `src/agents/tools/web-search-provider-common.ts:76`
  - body: Returning `undefined` for a raw SecretRef makes downstream callers treat an explicit but unresolved credential as absent. Web-search providers then fall back to ambient env keys, and memory remote falls back to provider auth, so a configured SecretRef can be bypassed instead of remaining configured-unavailable. Please keep a separate inspect/tri-state result and only allow fallback when no credential was configured.
  - confidence: 0.88
- **[P2] Update the stale embedding SecretRef test:** `extensions/ollama/src/embedding-provider.ts:150`
  - body: This behavior change means `createOllamaEmbeddingProvider` should no longer reject for the remote apiKey SecretRef case, but `extensions/ollama/src/embedding-provider.test.ts` still asserts that rejection. As written, the targeted extension test suite will fail once this patch is applied, so the PR needs the test changed to the intended deferred-resolution behavior.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff touches credential resolution and can let unresolved explicit SecretRefs fall through to unrelated fallback credentials.

Concerns:

- **[medium] Unresolved SecretRefs can fall back to ambient credentials:** `src/agents/tools/web-search-provider-common.ts:76`
  - body: The proposed non-throwing readers return `undefined` for raw SecretRefs, while existing web-search and memory code interprets `undefined` as no configured credential and tries env/provider fallback. That can use a different credential than the explicitly configured SecretRef and weakens the fail-closed SecretRef model.
  - confidence: 0.88

  - body: For memory remote embeddings, a configured-but-unavailable `remote.apiKey` SecretRef would become `undefined`, causing `resolveRemoteEmbeddingBearerClient` to use provider auth for the remote base URL instead of preserving the explicit remote credential state.
  - confidence: 0.82
