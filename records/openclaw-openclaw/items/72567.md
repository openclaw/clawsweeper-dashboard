## Summary

Keep open: this is a plausible feature request, but the proposed patch still misses provider-profiled models, conflicts with the current provider-owned thinking contract, has a rank/docs mismatch, and lacks required real behavior proof.

## What This Changes

The PR expands backend and Control UI thinking-level defaults, ranks, labels, tests, and docs to expose `xhigh`, `adaptive`, and `max` for non-binary providers.

## Best Possible Solution

Decide the provider thinking menu contract first, then either keep exact provider-owned profiles or implement a consistent profile-normalization/provider-profile update with docs, tests, and real UI proof.

## Reproduction Assessment

Yes. Source inspection shows provider profiles return before fallback levels are used, and Gateway plus the Sessions UI render those provider options directly; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR changes fallback lists and UI defaults without resolving the authoritative provider-profile contract, and its rank change conflicts with the documented/tested adaptive fallback.
