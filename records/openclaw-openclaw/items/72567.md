## Summary

Keep open for maintainer review: the PR is not mergeable as-is because expanding fallback lists does not affect real non-empty provider profiles that Gateway and the Control UI already use, and the adaptive rank change conflicts with the documented fallback behavior when `xhigh` is present.

## What This Changes

The PR expands backend/UI fallback thinking-level lists, labels, ranks, tests, and docs so non-binary provider dropdowns advertise `xhigh`, `adaptive`, and `max` with graceful downgrade.

## Best Possible Solution

Decide the thinking-level contract first; if all-level display is intended, apply that policy after provider-profile normalization or update provider-owned profiles consistently with Gateway/UI and directive coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a non-empty provider profile such as OpenAI returns before fallback levels are used, while Gateway rows pass that exact list to the Control UI.

## Solution Assessment

No. The patch changes fallback lists and documentation, but the authoritative provider-profile path remains exact and bypasses the new fallback levels, so the stated UI behavior is not achieved for real profiled providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply all-level policy after provider profiles:** `src/auto-reply/thinking.shared.ts:34-43`
  - body: Expanding `BASE_THINKING_LEVELS` only affects fallback profiles. `resolveThinkingProfile` returns non-empty provider profiles before this list is used, and real providers like OpenAI still declare exact levels without `adaptive`/`max`, so Gateway rows and the Control UI will not show all eight options as the PR claims. Apply the policy after provider-profile normalization or update provider-owned profiles and tests consistently.
  - confidence: 0.9
- **[P2] Align adaptive rank with the documented fallback:** `src/auto-reply/thinking.shared.ts:50-51`
  - body: With `xhigh` ranked 50 and `adaptive` ranked 55, the existing fallback resolver maps unsupported `adaptive` to `xhigh` on an xhigh-capable profile, not to `high` as the updated docs and test name claim. Put `adaptive` below `xhigh` if `high` is the intended fallback, or update the docs/tests for the xhigh-capable case.
  - confidence: 0.82
