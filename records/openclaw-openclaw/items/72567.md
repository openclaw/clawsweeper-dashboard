## Summary

Keep open. The PR targets a real UI limitation, but the proposed patch does not implement its stated all-level dropdown behavior for the main provider-profile path: current code returns non-empty provider profiles before the expanded base fallback list, while Gateway rows and the Control UI consume those exact provider-owned options.

## What This Changes

The PR expands shared/backend and Control UI fallback thinking-level lists to include `xhigh`, `adaptive`, and `max`, adjusts ranks and labels, and updates thinking tests and docs for all-level dropdown behavior.

## Best Possible Solution

Maintainers should decide the intended contract first. If all-level display is desired, implement it in one central post-processing seam after provider-profile normalization, or update provider-owned profiles consistently, with coverage for at least one real profiled provider plus Gateway row and Control UI selector paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: with any active non-empty provider profile such as OpenAI, `resolveThinkingProfile` returns before the expanded base profile, and Gateway rows pass that exact list to the Control UI.

## Solution Assessment

No. Expanding only fallback lists and docs is not the narrowest maintainable solution because it misses the provider-profile path that current docs and tests treat as authoritative.
