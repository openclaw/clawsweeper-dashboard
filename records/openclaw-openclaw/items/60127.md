## Summary

Keep open. Current main and the latest shipped release still document one trusted operator boundary per Gateway and lack tenant identity propagation, tenant RBAC, tenant-scoped plugin/config/memory/quotas, or audit attribution; related issues cover narrower workspace routing, embeddable API, or closed out-of-scope SaaS/RBAC discussions rather than fully superseding this shared-Gateway tenant-isolation request.

## What This Changes

Request secure tenant-isolated multi-tenancy in one shared OpenClaw Gateway, with RBAC, scoped resources, strict data boundaries, and operational controls instead of one instance per user.

## Best Possible Solution

Keep this as a maintainer-led product/security design issue; either explicitly keep shared-Gateway tenancy out of scope, or define a tenant context, RBAC, state, plugin/config, memory, quota, and audit contract with cross-tenant denial tests before implementation.
