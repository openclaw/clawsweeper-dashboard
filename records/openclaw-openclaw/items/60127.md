## Summary

Keep open. Current main explicitly documents OpenClaw as one trusted operator boundary per Gateway and still lacks tenant identity, tenant RBAC, tenant-scoped plugin/config/memory/quotas, or audit attribution; related threads such as https://github.com/openclaw/openclaw/issues/51451 and https://github.com/openclaw/openclaw/issues/78811 cover narrower or different problems rather than superseding this request.

## What This Changes

The issue requests native shared-instance multi-tenancy with tenant isolation, RBAC, scoped resources, quotas, and strict data-boundary guarantees.

## Best Possible Solution

Keep this as a product/security design issue; if shared-Gateway tenancy is in scope, specify tenant context through auth/protocol plus tenant-scoped state, config, plugins, memory, quotas, audit logs, and cross-tenant denial tests before implementation.
