## Summary

Keep open. Current main has adjacent multi-agent routing, per-agent state/auth/tool settings, and global plugin controls, but it still explicitly documents one trusted operator boundary per Gateway and lacks the tenant identity propagation, tenant RBAC, per-tenant quotas, tenant-scoped plugin enablement, and tenant-attributed audit boundaries requested here. The KinthAI comments add useful external implementation evidence, and the request needs maintainer product/security triage rather than automated cleanup.

## What This Changes

Request native shared-server multi-tenancy in one OpenClaw instance, including tenant isolation, RBAC, scoped resources, quotas/plugin config, and audit boundaries instead of one deployment per user.

## Best Possible Solution

Keep this as a product/security design issue. If shared-server multi-tenancy is in scope, design tenant context through Gateway auth/request context/protocol, tenant-scoped state/config/plugins/skills/model auth, per-tenant quotas, tenant-attributed audit logs, and cross-tenant denial tests/docs. If it is out of scope, document that hostile shared-Gateway tenancy is unsupported and split narrower issues only for specific plugin/API seams external managed-service implementations need.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source/docs inspection: the security guide disclaims hostile multi-tenancy, ConnectParamsSchema has no tenant identity field, and plugin configuration is global rather than tenant-scoped.

## Solution Assessment

Unclear, and not currently solved by the existing multi-agent model. The maintainable path is design-first product/security work or an explicit out-of-scope decision; a narrow automated fix PR would be unsafe because tenant isolation spans protocol, auth, storage, plugins, memory, quotas, audit, and docs.
