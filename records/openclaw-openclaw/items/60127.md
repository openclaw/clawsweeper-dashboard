## Summary

Keep open. Current main still defines OpenClaw as a one-trusted-operator Gateway, while this request asks for a new shared-tenant security boundary with tenant identity, RBAC, quotas, tenant-scoped plugin/config/memory behavior, audit attribution, and hard data-isolation guarantees.

## What This Changes

Request native shared-Gateway multi-tenancy with tenant isolation, RBAC, scoped resources, and strict data boundaries instead of per-user OpenClaw instances.

## Best Possible Solution

Keep this as a product/security design issue; if shared-Gateway tenancy is in scope, specify tenant context through auth/protocol plus tenant-scoped state, config, plugins, memory, quotas, audit logs, and cross-tenant denial tests before implementation.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source/docs inspection showing the single-operator Gateway model, no tenant identity in the Gateway handshake, and no tenant-scoped plugin/config/RBAC contract.
