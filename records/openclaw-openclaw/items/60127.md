## Summary

Keep open. Current main has adjacent multi-agent routing, per-agent state, and global plugin controls, but it still documents one trusted operator boundary per Gateway and lacks the tenant identity, RBAC, quotas, tenant-scoped plugin enablement, tenant-attributed audit, and hard data-boundary guarantees requested here.

## What This Changes

Request native shared-Gateway multi-tenancy with tenant isolation, RBAC, scoped resources, and strict data boundaries instead of one OpenClaw instance per user.

## Best Possible Solution

Keep this as a product/security design issue; if shared-Gateway tenancy is in scope, specify tenant context through auth/protocol, tenant-scoped state/config/plugins/memory, quotas, audit logs, and cross-tenant denial tests, otherwise document it as unsupported and split only narrow plugin/API seam requests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source/docs inspection showing the single-operator Gateway model, no tenant identity in ConnectParamsSchema, and no tenant-scoped plugin/config/RBAC contract.

## Solution Assessment

Unclear. The existing multi-agent model is useful adjacent infrastructure but is not the requested hostile shared-tenant boundary, and the maintainable path is design-first product/security work or an explicit out-of-scope decision.
