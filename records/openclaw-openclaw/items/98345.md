## Summary

Keep open: current main still has the reported fail-open existing-page reads in both memory-wiki source-page writers, so this is a real data-loss bug rather than stale cleanup.

## What This Changes

The issue reports that memory-wiki can erase a user-authored Notes block when re-ingest treats a transient existing-page read failure as an empty page.

## Best Possible Solution

Add a narrow fail-closed or bounded-retry existing-page read helper for both memory-wiki source-page writers, then cover both paths with transient read-failure regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the existing page unreadable at the read site, and the catch returns an empty string so the writer stores the regenerated page without the prior Notes block; I did not execute a fault-injection repro in this read-only review.

## Solution Assessment

Yes. The best fix is a narrow fail-closed or bounded-retry change at the two existing-page read sites, plus regression tests for both source-page writers; adding a config knob or changing unrelated unsafe-local prune behavior would be the wrong boundary.
