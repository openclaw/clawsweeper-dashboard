## Summary

Keep open: current main still lacks a Volcengine CodingPlan object-shaped thinking payload path, and the open closing PR is the active implementation candidate. The issue should close only after that PR lands or is replaced by an equivalent fix.

## What This Changes

The issue reports that Volcengine CodingPlan OpenAI-compatible requests cannot use OpenClaw thinking levels because the API expects an object-shaped `thinking` payload instead of the generic reasoning payload.

## Best Possible Solution

Review and land or replace the provider-owned Volcengine compat fix so affected CodingPlan models are reasoning-capable, map OpenClaw thinking levels to object budgets, and include focused tests, docs, and redacted live provider proof.

## Reproduction Assessment

No high-confidence live reproduction was established because this review did not run a credentialed Volcengine request. Source inspection gives a medium-confidence path: current main has no way to emit the requested object-shaped Volcengine thinking payload for OpenAI-compatible completions.

## Solution Assessment

Yes, the narrow provider-owned compat direction is the best maintainable path. Broad user configuration should not be required unless maintainers decide this is a cross-provider contract rather than a Volcengine-specific compatibility rule.
