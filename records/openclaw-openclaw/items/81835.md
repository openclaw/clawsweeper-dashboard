## Summary

Keep open: the updated report is now concrete, and current main still lacks a Volcengine-owned thinking payload adapter for the Coding Plan OpenAI-compatible route, so the reported 400 is plausible and narrowly actionable.

## What This Changes

Requests Volcengine CodingPlan models to send provider-compatible object-shaped `thinking` payloads instead of raw string thinking levels.

## Best Possible Solution

Add a Volcengine-owned stream wrapper or manifest/provider policy that maps supported Coding Plan thinking levels to the vendor object payload, with focused tests for `/think` and `thinkingDefault` on `volcengine-plan` models.

## Reproduction Assessment

Yes from source inspection, but not from a live credentialed call. The current `volcengine-plan` route is OpenAI-compatible with no thinking adapter, while the runtime does pass `/think` and session thinking levels into provider wrapping.

## Solution Assessment

Yes, a provider-local Volcengine payload wrapper is the narrowest maintainable fix. A generic core `thinking` format switch would be broader than needed and risks changing other OpenAI-compatible providers.

Add a Volcengine-owned stream wrapper or equivalent provider-local policy under `extensions/volcengine` that maps non-off OpenClaw thinking levels to `thinking: { type: "enabled", budget_tokens: N }` for `volcengine-plan` models and disables/omits thinking for `off` according to the confirmed Volcengine contract.
