## Summary

Keep open: this is a focused fix for a real Browser tab-selection race, but the PR still needs a small fallback hardening change and stronger live local-managed Chrome/CDP proof before merge.

## What This Changes

The PR changes the bundled Browser plugin's `ensureTabAvailable()` resolver to preserve newly opened tab snapshots, poll for delayed `wsUrl` discovery, fall back to unfiltered local tabs, and add regression tests.

## Best Possible Solution

Harden the polling fallback to keep the latest non-empty tab snapshot, tolerate transient polling failures, and add redacted live local-managed Chrome/CDP proof before merge.

## Reproduction Assessment

Yes, at source level: current main still discards the opened tab, immediately re-lists, filters local-managed candidates without `wsUrl`, and throws when no candidate remains. I did not run a live Chrome timing repro in this read-only review.

## Solution Assessment

No, not yet: the resolver is the right layer, but the polling fallback should preserve the latest non-empty tab snapshot and tolerate transient list failures before maintainers accept it as the best fix.
