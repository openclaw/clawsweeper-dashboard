## Summary

Keep open. Current main has mitigations for the startup regression, but the remaining hung-process/shutdown path is still source-backed: active OpenRouter/LiteLLM pricing fetches are not tied to the Gateway lifecycle, and refresh completion can still schedule later work after stop.

## What This Changes

The issue asks for `openclaw gateway` to continue starting and shut down cleanly when OpenRouter/LiteLLM pricing-catalog fetches time out after upgrading to 2026.4.24 on Windows npm-global installs.

## Best Possible Solution

Make the model-pricing refresh lifecycle-aware while keeping the existing startup mitigations: abort active catalog fetches on Gateway stop, avoid cache mutation and refresh scheduling after stop, unref long background timers, and add focused shutdown tests around the pricing refresh path.

## Reproduction Assessment

Yes. The issue gives concrete released-package steps, and current main supports a focused source-level reproduction by starting `startGatewayModelPricingRefresh` with hanging fetches, calling the returned stop callback during the in-flight refresh, and asserting the fetches abort with no post-stop timer scheduled.

## Solution Assessment

No. Current main is a partial mitigation; the narrow maintainable fix is to thread a lifecycle abort signal through pricing refresh and scheduling, not only to defer startup work or provide an operator opt-out.
