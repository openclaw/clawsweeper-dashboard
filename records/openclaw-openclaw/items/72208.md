## Summary

Keep open. Current main has useful startup mitigations and a documented pricing opt-out, but the remaining shutdown lifecycle gap is still present: in-flight pricing fetches are not tied to Gateway stop, and refresh completion can still schedule future work after stop.

## What This Changes

The issue reports that `openclaw gateway` hangs on Windows after a global install of `openclaw@2026.4.24` when OpenRouter and LiteLLM pricing catalog requests time out, leaving the dashboard unreachable and Ctrl+C ineffective.

## Best Possible Solution

Make the model-pricing refresh lifecycle-aware while preserving the existing startup mitigations: pass a Gateway-owned abort signal into both catalog fetches, abort in-flight refresh work on stop, suppress cache mutation and refresh scheduling after stop, unref long background timers, and cover those behaviors with focused Gateway pricing tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue has concrete released-package steps, and current main has a high-confidence source-level reproduction path: start pricing refresh with hanging fetches, call the returned stop callback while fetches are in flight, then assert the fetches abort and no post-stop timer is scheduled.

## Solution Assessment

No. Current main is only a partial mitigation; the narrow maintainable fix is lifecycle-aware cancellation in `src/gateway/model-pricing-cache.ts` and its runtime service wiring, not relying only on longer/shorter request timeouts or the operator opt-out.
