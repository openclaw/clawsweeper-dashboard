## Summary

Keep open. Current main has startup mitigations and a documented pricing opt-out, but the pricing refresh is still not shutdown-aware: in-flight fetches cannot be aborted by Gateway stop, and refresh completion can schedule another timer after stop.

## What This Changes

The issue asks for the 2026.4.24 npm-global Gateway pricing-timeout regression on Windows to stop hanging startup/shutdown and keep the dashboard reachable.

## Best Possible Solution

Keep the existing startup mitigations, then make model-pricing refresh lifecycle-aware: pass a Gateway abort signal into both catalog fetches, abort in-flight work on stop, avoid cache mutation and refresh scheduling after stop, unref long background timers, and add focused lifecycle tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue has concrete released-package steps, and current main has a high-confidence source-level reproduction path: start a pricing refresh with hanging fetches, call the returned stop callback during the in-flight refresh, then assert the fetches abort and no post-stop timer is scheduled.

## Solution Assessment

No. Current main is only a partial mitigation; the narrow maintainable fix is a lifecycle repair in model-pricing refresh rather than relying only on a longer timeout, shorter timeout, or operator opt-out.
