## Summary

Keep open. Current main has partial startup mitigations for the pricing-refresh regression, including deferred bootstrap and a config opt-out, but the source still does not make the in-flight LiteLLM/OpenRouter refresh shutdown-aware; the reported Ctrl+C/hung-process path remains actionable.

## What This Changes

The issue requests a fix for OpenClaw 2026.4.24 Windows/npm-global Gateway startup hangs when LiteLLM and OpenRouter pricing refreshes time out, including dashboard availability and shutdown responsiveness.

## Best Possible Solution

Keep the existing startup mitigations and config opt-out, then make model-pricing refresh lifecycle-aware: pass a Gateway-owned abort signal into both pricing catalog fetches, abort in-flight work on stop, prevent cache mutation and rescheduling after stop, unref any long background timer, and add focused tests for in-flight stop behavior.

## Reproduction Assessment

Yes. The report has concrete released-package steps, and current main has a source-level reproduction path: start the pricing refresh with hanging fetches, call the returned stop callback during the in-flight refresh, and assert the fetches abort and no post-stop refresh timer is scheduled.

## Solution Assessment

No. Current main is only a partial mitigation because it defers/bootstrap-optimizes the refresh and documents an opt-out, but the default refresh still cannot be canceled once started; the narrow maintainable fix is to wire pricing refresh into the existing Gateway stop lifecycle.
