## Summary

Keep open: the report describes a fresh, plausible Telegram message-loss regression, and current main still has source paths where Telegram delivery mirrors are transcript events but not consistently filtered before user-visible delivery/finalization paths.

## What This Changes

Telegram should stop transcript-only delivery-mirror assistant entries from being delivered back to Telegram and replacing the real assistant reply.

## Best Possible Solution

Keep the Telegram transcript mirror for transcript hygiene, but make every Telegram-visible finalization/redelivery path ignore transcript-only OpenClaw assistant messages by provider/model before it can edit or send a Telegram reply.

## Reproduction Assessment

No runnable high-confidence reproducer is available yet; the issue has detailed production timing evidence, and source inspection shows the likely leak path. A fix should add a focused regression that simulates a real assistant reply followed by a later delivery-mirror with different earlier-turn text.

## Solution Assessment

Yes, the suggested structural filter is the right direction. Same-text/latest dedupe is only a safety net and cannot protect the reported earlier-turn mirror entries.
