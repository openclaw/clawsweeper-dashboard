## Repair Work Prompt

Fix the user-visible dispatch-error fallback gap tracked by https://github.com/openclaw/openclaw/issues/60812. Current main has Telegram-specific fallback behavior, but Discord, Slack, Signal, WhatsApp, and the related LINE path still need a shared delivery-fallback path.

Implement a small shared helper exposed through an appropriate plugin SDK/reply-runtime or inbound-reply-dispatch seam. Do not make bundled plugins deep-import core internals. The helper should track the last dispatch/delivery error, non-silent skips, failed delivery counts, and actual delivered state, then choose a user-visible fallback for provider/run failures, empty/no-visible output, and generic delivery failures without duplicating messages that were already delivered successfully.

Wire the helper into Discord, Slack, Signal, WhatsApp, and LINE dispatch paths. Preserve existing abort/stop/silent-success behavior and existing streaming/draft-preview cleanup. For WhatsApp, handle the current isError payload suppression so provider errors can still produce a user-visible fallback. For Slack and Discord, do not treat merely queued counts as proof the user saw a reply when delivery failed.

Add focused tests covering delivery throw, dispatcher throw or rejected dispatch, no visible queued/delivered output, provider error payload, and no duplicate fallback after a successful visible reply. Use existing channel test harnesses where possible. Add a changelog fix entry because this is user-visible; credit @garnetlyx and any source PR work used. Do not change channel auth, allowlists, routing, or unrelated streaming behavior.
