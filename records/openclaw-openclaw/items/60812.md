## Summary

Keep open. Current main still has the Telegram fallback pattern, but Discord, Slack, Signal, and WhatsApp either log, mark status reactions as error, or return false after failed/no-visible dispatch without sending a user-visible fallback reply; the linked Telegram/model PR is open and does not implement this cross-channel work.

## What This Changes

The issue requests user-visible fallback replies for Discord, Slack, Signal, and WhatsApp when auto-reply dispatch fails or produces no visible delivery.

## Best Possible Solution

Add an extension-safe shared fallback helper through an existing SDK/runtime facade, then invoke it from each affected channel when dispatch ends with no visible reply plus an error, failed final count, or provider rejection, preserving each channel's status-reaction and suppression guards.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main's Discord, Slack, Signal, and WhatsApp dispatch paths expose failure/no-visible-delivery branches that do not send a fallback reply. I did not perform live transport reproduction.

## Solution Assessment

Yes in direction, with one boundary adjustment: the helper should be exposed through an extension-safe SDK/runtime surface rather than imported from core internals by bundled plugins.
