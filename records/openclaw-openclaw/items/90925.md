## Summary

Keep open: current main still carries the Codex-owned auto-compaction skip plus CLI throw paths and subagent retry-limit suspension that match the reported delivery failure. The live comments report reproduction on 2026.6.1 and 2026.6.6, and the adjacent open compaction PR does not prove the exact high-context Codex/OAuth requester announce path.

## What This Changes

A high-context Codex/OAuth parent session can lose a completed native subagent announce because requester transcript compaction resolves to an API-key-only OpenAI Responses path or treats Codex-owned compaction as a fatal delivery failure.

## Best Possible Solution

Land a maintainer-approved repair that makes requester announce compaction use the effective Codex/OAuth runtime boundary or preserve delivery non-fatally, with focused regression tests and live Codex/OAuth subagent proof.

## Reproduction Assessment

Yes at source level, with medium confidence: reporter logs and current code show completed subagent delivery can be marked failed/blocked when requester-agent handoff throws during compaction. I did not run a live current-main Discord/Codex/OAuth canary.

## Solution Assessment

Unclear: making post-turn compaction failures non-fatal is a plausible mitigation, but the best fix needs maintainer confirmation of the Codex auto-compaction ownership and OAuth routing contract. The final repair should prove that the requester announce path keeps the effective runtime/auth facts or safely preserves delivery without masking real session-state failures.
