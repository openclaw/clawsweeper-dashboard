## Summary

Keep open: current main still sends requester-side subagent completion announce through a normal agent handoff that does not carry the parent session's prepared Codex runtime/auth facts, and the latest issue comments report the same delivery boundary still failing on v2026.6.6 by both OpenAI Responses API-key routing and Codex app-server compaction ownership.

## What This Changes

Requester-side subagent completion announce should preserve the parent session's effective Codex runtime/OAuth compaction route instead of dropping delivery through OpenAI Responses API-key compaction or Codex-owned auto-compaction failure paths.

## Best Possible Solution

Define the requester announce compaction contract, preserve or re-resolve the parent session's effective Codex runtime/OAuth facts at that boundary, and prove high-context Discord Codex/OAuth subagent completion is visibly delivered without API-key-route or ownership-skip failures.

## Reproduction Assessment

Yes for source-level reproduction: the live reports give repeatable high-context Discord Codex/OAuth steps, and current source maps direct announce into requester compaction/auth paths; I did not run a live current-main Discord scenario.

## Solution Assessment

The requested direction is likely the right boundary fix, but maintainers still need to choose whether announce should carry prepared runtime/auth facts, treat Codex-owned automatic compaction as a non-fatal wake path, or combine both with durable delivery recovery.
