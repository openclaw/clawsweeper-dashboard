## Summary

Keep this PR open: the user-facing bug is real and the patch direction is narrow, but the latest head still leaves one Gateway shared-auth test double returning the old undefined write result while the new code dereferences `persistedConfig`.

## What This Changes

The PR changes config write helpers and Gateway `config.set`/`config.apply`/`config.patch` responses to carry the persisted config result, updates restart follow-up data, test doubles, regression coverage, and the changelog.

## Best Possible Solution

Keep the persisted-config write boundary, update the remaining shared-auth Gateway mock to return the new write-result shape, then let focused Gateway/config checks and normal PR review continue.

## Reproduction Assessment

Yes. Current `main` is source-reproducible because the handlers return parsed or validated input config while the IO writer can stamp and return a different persisted payload; the PR blocker is source-reproducible from the shared-auth mock returning `undefined`.

## Solution Assessment

No for the current PR head as-is. Carrying `persistedConfig` through the shared write boundary is the narrow maintainable fix, but the branch must update every affected Gateway method mock to honor that result contract.
