## Summary

Keep open: the auth-alias direction is useful, but the runtime patch hard-codes a Gemini model list instead of using the existing wildcard/provider runtime policy, so dynamic Google models can still miss Gemini CLI routing. The PR also has only test-output proof, not a live Gemini CLI OAuth/chat run.

## What This Changes

The PR adds a `google-gemini-cli` to `google` auth alias in the Google plugin manifest and expands Gemini CLI OAuth setup to write runtime bindings for multiple Gemini models.

## Best Possible Solution

Keep the manifest auth alias, replace the exact model entries with a wildcard or provider-scoped runtime policy that covers dynamic Google chat models while preserving exact overrides, add regression coverage, and require redacted live Gemini CLI proof before merge.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection shows current main only writes a Gemini CLI runtime policy for the default model and can return no CLI runtime from auth fallback when profile matching is ambiguous.

## Solution Assessment

No. The manifest alias is the right owner boundary, but the exact model list is not the best fix because OpenClaw already has wildcard/provider runtime policy for provider-wide dynamic model coverage.
