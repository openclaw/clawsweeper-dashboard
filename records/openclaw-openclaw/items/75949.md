## Summary

Keep open. Current main still has a concrete lock-contention path: same-key gateway and heartbeat lanes mitigate some ordinary cases, but transcript writers that reach the same session file outside that lane can still fail after the 10s session write-lock timeout instead of being queued or retried.

## What This Changes

Ensure peer-agent and system-event deliveries to an active session wait or retry instead of dropping after a session JSONL write-lock timeout.

## Best Possible Solution

Add transcript-file-aware serialization or retry for peer-agent/internal event delivery so same-transcript writes preserve ordering and wait behind the active turn while keeping stale-lock cleanup intact.

## Reproduction Assessment

Yes for the underlying failure mode: current unit coverage shows a held session write lock makes a second default acquisition reject with `session file locked`. The full Wren-to-Telegram workflow is config-specific, but the reported contention is source-backed.

## Solution Assessment

No for simply increasing the timeout; that would only make drops less frequent. The maintainable fix is to queue or retry by normalized session file identity before transcript writes while preserving existing session-lane behavior.
