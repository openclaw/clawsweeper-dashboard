## Summary

Keep open. Current main still has a real same-transcript lock-contention gap: same-key agent turns are serialized and heartbeat has a busy-lane skip, but writers that reach the same JSONL through another key/process can still hit the 10s session write-lock timeout and surface an error instead of being queued for ordered delivery.

## What This Changes

The issue requests that peer/system events arriving during long agent turns be queued or retried instead of being dropped after `SessionWriteLockTimeoutError`.

## Best Possible Solution

Add normalized transcript-file-aware serialization or retry for peer-agent messages and system-event wakes so they wait behind the active writer and preserve transcript order without turning this into a broader durable-delivery redesign.

## Reproduction Assessment

Yes. The underlying failure mode is reproducible from current unit coverage: one held session write lock makes a second default acquisition against the same file reject with `session file locked`; the exact multi-agent Telegram incident is environment-specific but source-backed.

## Solution Assessment

No for the timeout-only workaround. The maintainable fix is to queue or retry by resolved transcript file identity before write-lock acquisition, while preserving stale-lock cleanup and existing same-session lane behavior.
