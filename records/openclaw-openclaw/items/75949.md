## Summary

Keep open. Current main still has a 10s transcript write-lock timeout and queueing is keyed by session key, so same-key heartbeats are mitigated but same-transcript delivery through another key or writer path can still fail instead of being serialized.

## What This Changes

The issue asks for incoming system or peer-agent events to queue or retry instead of being lost while a long agent turn holds the session transcript write lock.

## Best Possible Solution

Add transcript-file-aware serialization or retry for incoming agent/system-event delivery, preserving transcript ordering and stale-lock cleanup without expanding into durable restart-time delivery.

## Reproduction Assessment

Yes for the underlying failure mode: current unit coverage proves a held session write lock makes a second acquisition reject with `session file locked`. The full Wren-to-Telegram incident is environment-specific, but current source supports the reported contention shape.

## Solution Assessment

No, increasing the timeout alone is not the best fix. The maintainable path is to queue or retry by normalized transcript file identity before writing, while keeping existing same-key lane and stale-lock behavior intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75949. Current main queues embedded runs by session key, but the transcript write lock is by normalized session file, so the same JSONL transcript can still be contended through different session keys or writer paths and fail after the 10s `SessionWriteLockTimeoutError`. Add a generic transcript-file-aware queue or retry path so incoming agent/system-event delivery waits for the active writer and preserves transcript ordering. Do not solve this by only raising the timeout, do not add Telegram/Wren/plugin-specific logic, and do not broaden the scope into durable delivery across gateway restarts unless maintainers ask. Preserve stale-lock cleanup and existing same-session-key lane behavior. Add a regression that proves same-transcript/different-key or equivalent file-lock contention is serialized instead of dropped, update `CHANGELOG.md` as a user-facing fix, and keep the implementation in core generic session/agent/gateway surfaces.
