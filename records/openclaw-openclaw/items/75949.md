## Summary

Keep open: current main mitigates ordinary same-key heartbeat wakeups, but the underlying session file lock still fails after a 10s wait and there is no normalized transcript-file queue/retry path for peer writers that share the same transcript through aliases or a different writer path.

## What This Changes

Incoming system or peer-agent events should be queued or retried instead of being dropped when a long agent turn holds the session JSONL write lock.

## Best Possible Solution

Add a normalized transcript-file keyed serialization or retry path so peer-agent messages and system-event wakeups wait for the active turn and preserve transcript ordering, while keeping restart-durable event storage out of scope.

## Reproduction Assessment

Yes. The low-level failure is reproducible from current unit coverage by holding a session write lock and attempting a second default acquisition, while the full Wren-to-Telegram route remains environment-specific.

## Solution Assessment

Partly. Queueing or retrying by normalized transcript file identity is the maintainable direction; increasing the timeout alone would reduce symptoms without preventing dropped peer/system delivery.
