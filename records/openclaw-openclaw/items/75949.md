## Summary

Keep open. Current main has partial protection through same-session command lanes and heartbeat busy retries, but the underlying session file lock still times out after 10s and there is no clear file-identity queue/retry path for peer/system writers that reach the same transcript through a different session key or writer path.

## What This Changes

The issue asks OpenClaw to stop dropping peer-agent and system-event updates when long active agent turns hold the session JSONL write lock.

## Best Possible Solution

Add a transcript-file-aware queue or retry path so peer-agent messages and system-event wakes wait for the active turn and preserve transcript ordering instead of surfacing a lock-timeout failure.

## Reproduction Assessment

Yes for the underlying failure mode: current unit coverage shows a held session write lock makes a second default acquisition reject with `session file locked`. The full Wren-to-Telegram incident path is config-specific, but the reported lock contention is concrete and source-backed.

## Solution Assessment

No, increasing the timeout alone is not the best fix. The maintainable fix is to serialize or retry incoming delivery by session file identity before writing, while preserving stale-lock cleanup and existing same-key lane behavior.
