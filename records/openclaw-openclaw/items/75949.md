## Summary

Keep open. Current main still has the reported lock-contention failure mode: embedded turns hold the transcript write lock through the active run, the default acquisition timeout is still 10s, and existing same-session lane/heartbeat mitigations do not prove serialization for every writer that reaches the same transcript through a different key or path.

## What This Changes

The issue requests that peer/system events arriving during long agent turns be queued or retried instead of being dropped after the session transcript write-lock timeout.

## Best Possible Solution

Add a transcript-file-aware serialization or retry path so peer-agent messages and system-event wakes wait behind the active turn and preserve transcript ordering without relying on a longer timeout.

## Reproduction Assessment

Yes for the underlying issue: current unit coverage proves a held session write lock makes a second default same-file acquisition fail with `session file locked`. The full external incident route is configuration-specific, but the lock contention and drop-prone writer path are source-backed.

## Solution Assessment

No, increasing the timeout alone is not the best fix. The maintainable direction is to serialize or retry incoming delivery by normalized transcript file identity while preserving stale-lock cleanup and existing same-key lane behavior.
