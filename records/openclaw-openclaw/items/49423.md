## Summary

Keep this PR open. Current main still uses synchronous filesystem APIs in the user-facing export-session command, so the cleanup is not implemented, but the proposed patch should not merge as-is because it does not update the sync-shaped tests and it adds an unnecessary access-before-recursive-mkdir filesystem race. https://github.com/openclaw/openclaw/pull/49423

## What This Changes

The PR converts the export-session command from sync `fs` calls to `fs.promises` calls and loads export HTML templates in parallel with `Promise.all`.

## Best Possible Solution

Revise this PR in place, or open a narrow replacement if the source branch cannot be updated: keep the async `fs.promises` direction and parallel template loading, call recursive `mkdir` directly, remove the redundant `return await`, and update `commands-export-session.test.ts` to mock and assert the async fs path.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to apply the PR diff and run `pnpm test src/auto-reply/reply/commands-export-session.test.ts`; static inspection shows those tests still mock sync `fs` methods while the patch calls real `fs.promises` methods for the mocked `/tmp` paths.

## Solution Assessment

No, not as currently proposed. The async migration is the right narrow direction, but the maintainable fix also updates the test harness and uses direct recursive `mkdir` instead of an `access` guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mock the async fs path before switching calls:** `src/auto-reply/reply/commands-export-session.ts:142`
  - body: The existing tests mock only sync `fs` methods. After this line switches session existence to `fs.promises.access`, the test harness will hit the real filesystem for `/tmp/target-store/session.jsonl`, return the not-found path, and stop exercising the HTML export assertions. Update the colocated test mocks/assertions for `fs.promises` in the same PR.
  - confidence: 0.88
- **[P2] Create the output directory directly:** `src/auto-reply/reply/commands-export-session.ts:187-191`
  - body: `mkdir(outputDir, { recursive: true })` is already safe when the directory exists, so checking `access(outputDir)` first adds an avoidable TOCTOU window between the check and use. Call recursive `mkdir` directly before writing the export file.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: No supply-chain or secrets-handling surface is changed, but the patch introduces a low-severity filesystem TOCTOU concern in output-directory creation.

Concerns:

- **[low] Access-before-mkdir race:** `src/auto-reply/reply/commands-export-session.ts:187`
  - body: The proposed output-directory flow checks `fs.promises.access(outputDir)` before recursive `mkdir`; another process can change the path between those operations. Calling recursive `mkdir` directly avoids that race.
  - confidence: 0.86
