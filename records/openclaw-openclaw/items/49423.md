## Summary

Keep open. Current main still uses synchronous filesystem calls in the export-session command, so the PR targets a real remaining cleanup, but the patch is not merge-ready because it switches the command to real `fs.promises` calls without updating the colocated fs mocks and adds an unnecessary access-before-mkdir race.

## What This Changes

The PR changes `src/auto-reply/reply/commands-export-session.ts` to replace synchronous fs calls with `fs.promises` and load export HTML templates in parallel.

## Best Possible Solution

Revise this PR in place, or open a narrow replacement if the source branch cannot be updated: keep the async `fs.promises` direction and parallel template loading, call recursive `mkdir` directly, remove the redundant `return await`, and update `commands-export-session.test.ts` so the async fs paths are mocked and asserted.

## Reproduction Assessment

Yes. Apply the PR diff and run `pnpm test src/auto-reply/reply/commands-export-session.test.ts`; static inspection shows the tests still mock sync `fs` methods while the patch calls real `fs.promises` methods for the synthetic `/tmp` paths.

## Solution Assessment

No, not as currently proposed. The async migration is a narrow maintainable cleanup, but it needs matching async fs test mocks and direct recursive `mkdir` to avoid the race introduced by the access guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mock the async fs path before switching calls:** `src/auto-reply/reply/commands-export-session.ts:142`
  - body: The existing tests mock only sync `fs` methods. After this line switches session existence to `fs.promises.access`, the test harness will call the real filesystem for `/tmp/target-store/session.jsonl`, return the not-found path, and stop exercising the HTML export assertions. Update the colocated test mocks and assertions for `fs.promises` in the same PR.
  - confidence: 0.88
- **[P2] Create the output directory directly:** `src/auto-reply/reply/commands-export-session.ts:187-191`
  - body: `mkdir(outputDir, { recursive: true })` is already safe when the directory exists, so checking `access(outputDir)` first adds an avoidable TOCTOU window between the check and use. Call recursive `mkdir` directly before writing the export file.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: No supply-chain, workflow, dependency, or secrets-handling surface is changed, but the patch introduces a low-severity filesystem TOCTOU concern in output-directory creation.

Concerns:

- **[low] Access-before-mkdir race:** `src/auto-reply/reply/commands-export-session.ts:187`
  - body: The proposed output-directory flow checks `fs.promises.access(outputDir)` before recursive `mkdir`; another process can change the path between those operations. Calling recursive `mkdir` directly avoids that race.
  - confidence: 0.86
