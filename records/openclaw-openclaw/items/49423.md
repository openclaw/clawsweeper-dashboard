## Summary

Keep this PR open. The async filesystem migration is a useful narrow cleanup and current main still uses synchronous filesystem calls, but the proposed patch is not merge-ready because it does not update the colocated fs mocks and it adds an avoidable access-before-mkdir race.

## What This Changes

The PR converts `src/auto-reply/reply/commands-export-session.ts` from sync `fs` calls to `fs.promises` calls and loads export HTML templates in parallel without changing tests or docs.

## Best Possible Solution

Revise the PR in place, or replace it narrowly if the branch cannot be updated: keep the async `fs.promises` direction and parallel template loading, update `commands-export-session.test.ts` to mock/assert the async filesystem path, call recursive `mkdir` directly, and remove the redundant `return await` cleanup without changing command semantics.

## Reproduction Assessment

Yes. Apply the PR diff and run `pnpm test src/auto-reply/reply/commands-export-session.test.ts`; static inspection shows the patched command would call real `fs.promises.access` for the mocked `/tmp/target-store/session.jsonl` path and stop before the HTML export assertions.

## Solution Assessment

No, not as currently proposed. The async migration is the right narrow direction, but the maintainable fix must include async fs test mocks/assertions and direct recursive directory creation instead of an `access` guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mock the async fs path before switching calls:** `src/auto-reply/reply/commands-export-session.ts:142`
  - body: The existing tests mock only sync `fs` methods. After this line switches session existence to `fs.promises.access`, the test harness will hit the real filesystem for `/tmp/target-store/session.jsonl`, return the not-found path, and stop exercising the HTML export assertions. Update the colocated test mocks/assertions for `fs.promises` in the same PR.
  - confidence: 0.9
- **[P2] Create the output directory directly:** `src/auto-reply/reply/commands-export-session.ts:187-191`
  - body: `mkdir(outputDir, { recursive: true })` already succeeds when the directory exists, so checking `access(outputDir)` first adds an avoidable TOCTOU window between the check and use. Call recursive `mkdir` directly before writing the export file.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: No workflow, dependency, package, or secrets surface changes were found, but the patch introduces a low-severity filesystem TOCTOU concern in output-directory creation.

Concerns:

- **[low] Access-before-mkdir race:** `src/auto-reply/reply/commands-export-session.ts:187`
  - body: The proposed output-directory flow checks `fs.promises.access(outputDir)` before recursive `mkdir`; another process can change the path between those operations. Calling recursive `mkdir` directly avoids that race.
  - confidence: 0.86
