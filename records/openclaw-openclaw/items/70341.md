## Summary

Keep open. The PR targets a real Windows PowerShell cmdlet allowlist gap, and the later commits appear to address the minor caching/test-pattern review notes, but the current diff still changes exec allowlist semantics without preserving the approved PowerShell host through node-host execution.

## What This Changes

The PR adds a Windows-only PowerShell cmdlet fallback for exec allowlist matching and allow-always persistence, moves PowerShell path resolution into infra, and adds cmdlet allowlist tests.

## Best Possible Solution

Revise the design so the allowlist decision and execution plan bind the same command: either preserve a PowerShell-hosted argv for cmdlet matches all the way through node-host execution, or avoid reusable cmdlet fallback semantics and rely on exact-command durable approvals. The final PR should include Windows node-host execution coverage and the required changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: the PR makes a bare cmdlet allowlist-satisfied via a synthetic PowerShell match, while current node-host execution rewrites allowlisted Windows shell payloads to the bare segment argv before spawning.

## Solution Assessment

No. The PR updates allowlist evaluation and persistence, but not the execution plan, so the approved target can diverge from what the node host attempts to execute; carrying cmdlet-match metadata or an exact PowerShell argv into execution is the safer fix.
