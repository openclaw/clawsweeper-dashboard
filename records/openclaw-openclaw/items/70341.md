## Summary

Keep open. Current main does not implement the PR's reusable PowerShell-cmdlet-to-PowerShell-path allowlist semantics, and the proposed patch is not merge-ready because it can approve a synthetic PowerShell executable match while the Windows node-host execution path still runs the bare cmdlet argv.

## What This Changes

The PR adds a Windows-only fallback that lets unresolved bare PowerShell cmdlets match PowerShell executable allowlist entries, persists cmdlet-specific allow-always patterns, moves PowerShell path resolution into infra with caching, and adds cmdlet allowlist tests.

## Best Possible Solution

Either close this PR as superseded by the exact-command durable approval direction for #52952, or revise it so cmdlet-backed matches carry an explicit execution plan that preserves the PowerShell host through allowlist evaluation, allow-always persistence, and Windows node-host execution tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: at PR head, Get-ChildItem can become allowlist-satisfied through a synthetic PowerShell match, then resolveSystemRunExecArgv rewrites the Windows shell payload to the bare cmdlet argv, which is not a spawnable executable.

## Solution Assessment

No. Matching the synthetic PowerShell argv only inside allowlist evaluation is incomplete; the safer approach is to bind the approved PowerShell host to the execution plan or rely on the existing exact-command durable approval path.

## Review Findings
