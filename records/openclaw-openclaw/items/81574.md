## Summary

Keep open: the additive lookup metadata is useful and the PR includes real Gateway proof, but the latest head still misreports legacy plugin install timestamp paths as restart-required even though current reload planning treats them as no-op.

## What This Changes

The PR adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, wires the Gateway handler to a reload resolver, updates protocol docs/schema/Swift model, adds lookup coverage, and includes an unrelated live-agent test type cleanup.

## Best Possible Solution

Land the additive `reloadKind` contract after the resolver uses the same effective path classification as `buildGatewayReloadPlan`, including legacy plugin timestamp no-ops, with regression coverage for that path.

## Reproduction Assessment

Yes. The linked issue gives release-level steps for the missing metadata, and the PR defect is source-reproducible by comparing the new resolver for `plugins.installs.<id>.resolvedAt` with the current planner's no-op path handling.

## Solution Assessment

No as written. Optional resolver-backed `reloadKind` is a narrow maintainable direction, but it must share the planner's no-op classification before the public API field can be trusted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mirror timestamp no-op reload metadata:** `src/gateway/config-reload-plan.ts:228-229`
  - body: The new resolver reports `plugins.installs.<id>.resolvedAt` and `.installedAt` as `restart` because they match the broad `plugins.installs` rule here. `buildGatewayReloadPlan` handles those timestamp-only paths before rule matching and puts them in `noopPaths`, so `config.schema.lookup` would expose inaccurate restart metadata for paths that config apply treats as no-op. Please share that planner classification or special-case these paths and add lookup coverage.
  - confidence: 0.9
