## Summary

Keep open: the docs follow-up is addressed and the additive API shape is reasonable, but the new resolver does not fully mirror the existing reload planner for plugin install timestamp metadata.

## What This Changes

The PR adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, wires the Gateway handler to reload rules, updates the Swift protocol model, documents the field, and adds lookup coverage.

## Best Possible Solution

Land the additive `reloadKind` contract after the resolver matches `buildGatewayReloadPlan` for special no-op paths and regression coverage includes both normal Gateway fields and plugin install timestamp metadata.

## Reproduction Assessment

Yes. The linked report at https://github.com/openclaw/openclaw/issues/81409 gives release-level steps showing schema lookup lacks reload metadata while config patch planning knows restart behavior, and current main source confirms lookup returns only schema, hint, and children.

## Solution Assessment

No as written. Adding optional resolver-backed `reloadKind` is the right direction, but the resolver must share the planner semantics for no-op timestamp metadata before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Mirror plugin timestamp no-op reloads:** `src/gateway/config-reload-plan.ts:228-229`
  - body: The new resolver promises to mirror the Gateway reload planner, but it only reads `matchRule(path)?.kind` and defaults to `restart`. Current reload planning checks `plugins.installs.<id>.resolvedAt` and `plugins.installs.<id>.installedAt` before rule matching and treats timestamp-only changes as no-op, with existing coverage for that behavior. As written, schema lookup will report `restart` for those fields even though applying that timestamp-only change would not restart; share the planner decision or special-case those paths and add coverage.
  - confidence: 0.89
