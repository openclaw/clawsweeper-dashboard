## Summary

Keep this PR open for fixes. The approach is useful and the manual gateway lookup proof covers the main behavior, but the patch currently leaves generated protocol artifacts stale and misreports legacy plugin install timestamp metadata as restart-required even though the reload planner treats those paths as no-op.

## What This Changes

The branch adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, maps it from gateway reload planning, and extends config/protocol tests.

## Best Possible Solution

Keep the additive `reloadKind` lookup API, but make the resolver share the planner's path classification semantics and commit regenerated protocol artifacts before merge.

## Reproduction Assessment

Yes. The linked issue has a Docker reproduction for the shipped behavior, and current source inspection confirms `config.schema.lookup` returns schema/hint/children without reload metadata while reload planning separately classifies the same paths.

## Solution Assessment

No, not as submitted. The API shape is a reasonable narrow solution, but the implementation needs generated protocol artifacts and a shared or equivalent reload metadata classifier so it does not diverge from current no-op planning behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Regenerate protocol artifacts for reloadKind:** `src/gateway/protocol/schema/config.ts:110-112`
  - body: Adding `reloadKind` to `ConfigSchemaLookupResultSchema` changes the Gateway protocol contract, but the generated client artifacts were not committed. CI's `pnpm protocol:check` regenerated `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift` with a new `reloadkind` property, so this patch cannot merge until the protocol outputs are regenerated and included.
  - confidence: 0.98
- **[P2] Mirror timestamp no-op reload planning:** `src/gateway/config-reload-plan.ts:225-227`
  - body: `resolveConfigReloadMetadata` goes straight through `matchRule`, so `plugins.installs.<id>.installedAt` and `resolvedAt` report `restart` via the `plugins.installs` rule. `buildGatewayReloadPlan` checks those timestamp paths first and treats them as no-op, so the new lookup metadata would give clients the wrong reload policy for an existing no-op path.
  - confidence: 0.93
