## Summary

Keep open. The PR detects the right compaction marker, but it still collapses a pending compaction window into a terminal successful completion with `(no output)` because the announce flow cannot distinguish “defer” from “no reply.”

## What This Changes

The PR adds compaction-system-message handling to subagent output selection and unit coverage to suppress pre-compaction tool output while preserving timeout partial-progress text.

## Best Possible Solution

Keep the compaction marker detection, but return a typed pending/defer result through the output capture boundary and have `runSubagentAnnounceFlow` leave completion unannounced until post-compaction assistant text or true timeout partial progress exists.

## Reproduction Assessment

Yes. A focused code-path reproduction uses an assistant tool call, a toolResult, a synthetic compaction system marker, and no later assistant turn; the PR returns undefined, and the current caller can still emit a successful `(no output)` completion after retry.

## Solution Assessment

No. The low-level detector is useful, but `undefined` is already the no-content value; the maintainable fix needs a distinct pending-compaction state or equivalent defer result.
