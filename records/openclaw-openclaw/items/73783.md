## Summary

Keep open. The PR targets a valid subagent completion race and adds a useful compaction-marker detector, but it still collapses the pending-compaction state into `undefined`, which the current announce caller can turn into a successful `(no output)` completion before the real post-compaction answer arrives.

## What This Changes

The PR adds compaction-marker detection to subagent output selection and unit tests so pre-compaction raw tool output is suppressed while timeout partial-progress remains available.

## Best Possible Solution

Carry an explicit pending-compaction/defer result through output capture and announce construction so successful announcements wait for post-compaction assistant text or a true timeout partial-progress result, then consolidate the overlapping open PRs into one covered fix.

## Reproduction Assessment

Yes, at source level. A history with an assistant tool call, a toolResult, a synthetic compaction system marker, and no later assistant turn causes the PR branch to return `undefined`, which the current caller can still announce as successful `(no output)` after retry.

## Solution Assessment

No. The detector is in the right low-level place, but returning the same `undefined` used for ordinary no-content output is not enough; the announce flow needs a distinct pending/defer state or equivalent skip-terminal-announcement behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate pending compaction instead of returning no output:** `src/agents/subagent-announce-output.ts:324`
  - body: When the newest child history is a compaction marker, this branch now makes `readSubagentOutput` return `undefined`. `runSubagentAnnounceFlow` already treats `undefined` as missing content, retries through the same `string | undefined` API, and then builds a successful `(no output)` completion. That can still finalize the parent announcement before the post-compaction assistant answer arrives, so carry a distinct pending/defer state through capture and announce construction instead of reusing the no-reply sentinel.
  - confidence: 0.9
