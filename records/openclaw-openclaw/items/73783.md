## Summary

Keep open. The PR correctly targets real compaction markers and prevents one raw-tool-output fallback, but it still maps the pending-compaction state to the same undefined value used for ordinary missing output, so the announce flow can emit a successful `(no output)` completion before the child produces its real final answer.

## What This Changes

The PR adds compaction-system-message detection to subagent output selection and unit tests that suppress pre-compaction tool output while preserving timeout partial-progress text.

## Best Possible Solution

Keep the compaction marker detection, but carry a distinct pending/defer result through output capture and announce flow so completion is not announced until post-compaction assistant text or true timeout partial-progress exists.

## Reproduction Assessment

Yes. A focused code-path reproduction uses an assistant tool call, a toolResult, a synthetic compaction system marker, and no later assistant turn; the PR returns undefined, and the current caller can still turn that into a successful `(no output)` completion after retry.

## Solution Assessment

No. The detector is the right low-level signal, but `undefined` is already the no-content value; the maintainable fix needs a typed pending-compaction state or equivalent defer result across the caller boundary.
