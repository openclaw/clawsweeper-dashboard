## Summary

Keep open. The reported auto-compaction announce bug is credible, but this PR is not yet sufficient: it suppresses stale raw tool text in the output reader without preventing the caller from emitting a terminal successful announcement with `(no output)` while the child is still waiting for its post-compaction continuation.

## What This Changes

The PR adds compaction-marker detection to `src/agents/subagent-announce-output.ts` and unit tests so pre-compaction raw tool output is not selected as a subagent completion reply.

## Best Possible Solution

Carry the compaction-continuation state through the announce boundary as a typed defer result, and have `runSubagentAnnounceFlow` keep the run pending until a post-compaction assistant turn is available or a real timeout should emit partial progress. Preserve the useful compaction marker detection, but do not convert the limbo window into a completed empty result.

## Reproduction Assessment

Yes. A focused code-path reproduction is available with assistant tool use, a tool result, a synthetic compaction system marker, and no later assistant turn: the PR makes the reader return undefined, while the caller still converts that into a successful `(no output)` completion after retry.

## Solution Assessment

No. Detecting the compaction marker is the right lower-level signal, but returning undefined from the reader is not enough because the caller treats undefined as empty terminal content rather than a reason to defer and retry later.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Defer completion while compaction is still pending:** `src/agents/subagent-announce-output.ts:324-326`
  - body: When the newest child history is the compaction marker, the new logic suppresses raw tool text by returning `undefined`, but the caller treats that as an empty final result and sends a successful completion with `(no output)`. In the production timeline, the real assistant answer arrives minutes later, so the parent would still get a terminal completion before the actual result and no later announce would fire. Propagate a distinct continuation/defer state to the announce flow, or otherwise keep the run pending until a post-compaction assistant turn or true timeout is available.
