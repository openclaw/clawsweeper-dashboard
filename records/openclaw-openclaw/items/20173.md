## Summary

Keep open. Current main still has no Discord MESSAGE_UPDATE listener, monitor registration, edit-processing config/docs, or edit-aware replay semantics, and the related broader item remains open rather than superseding this edit-only tracker.

## What This Changes

Request opt-in Discord handling for edited user messages so MESSAGE_UPDATE can re-trigger agent processing with debounce and edit-aware behavior.

## Best Possible Solution

Implement Discord edit handling in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, changelog, and focused monitor tests; keep delete-to-cancel separate unless maintainers intentionally broaden scope.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Discord gateway dispatch uses exact listener types, no registered listener handles MESSAGE_UPDATE payloads, and same-message replay currently has no edit-aware semantics.

## Solution Assessment

Yes on direction, unclear on final policy. A Discord-plugin edit pipeline is the maintainable place to solve this, but maintainers should decide defaults, partial update handling, and replay/abort semantics before automation changes behavior.
