## Summary

Keep open. Current main still has no Discord `MESSAGE_UPDATE` listener or registration path, no edit-specific Discord config/docs surface, and the same-message replay guard would drop a naive edit reprocess without explicit edit-aware semantics.

## What This Changes

Request Discord edited messages to trigger a new agent turn, optionally gated by `handleEdits` and per-message debounce configuration.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed `MESSAGE_UPDATE` dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Discord `MESSAGE_UPDATE` dispatch has no matching registered monitor listener, and same-message edits have no edit-aware replay key.

## Solution Assessment

Yes on the owner boundary, but not as an automatic bug fix. The Discord plugin is the right place to solve it, while config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment.
