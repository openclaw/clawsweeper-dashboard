## Summary

Keep open. Current main and latest release v2026.6.5 still route Discord inbound messages through MESSAGE_CREATE only; there is no MessageUpdate listener, requested edit config/docs surface, or edit-aware replay semantics, and the related duplicate edit reports point back here rather than superseding it.

## What This Changes

The issue requests Discord MESSAGE_UPDATE support so edited user messages can trigger a fresh agent turn with optional opt-in/debounce behavior.

## Best Possible Solution

Keep this as the canonical Discord edit-to-reprocess tracker and implement the feature in the bundled Discord plugin with typed MessageUpdate dispatch, opt-in policy/config/docs, content-change filtering, per-message debounce, edit-aware replay/session handling, and focused monitor tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches exact listener types, has no Discord MessageUpdate listener or requested edit config surface, and the same-message-id replay guard would treat naive edit handling as a duplicate.

## Solution Assessment

Unclear as an automatic repair. The bundled Discord plugin is the right owner boundary, but maintainers still need to choose edit policy, default/config shape, debounce/rate limits, partial update hydration, and replay or abort semantics.
