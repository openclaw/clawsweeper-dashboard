## Summary

Keep open. Current main still lacks the Discord MESSAGE_UPDATE listener, edit-aware normalization/replay handling, opt-in config/docs, and focused tests, while the recent issue comments correctly keep this as the canonical Discord edit-to-reprocess tracker.

## What This Changes

Request opt-in Discord MESSAGE_UPDATE handling so edited user messages can trigger a fresh agent turn with debounce/config semantics.

## Best Possible Solution

Implement the feature in the bundled Discord plugin with MESSAGE_UPDATE registration, edited-payload normalization, content-change filtering, per-message debounce, edit-aware replay keys, docs/config metadata, changelog, and focused monitor tests while leaving delete-to-cancel separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord exposes MESSAGE_UPDATE, OpenClaw dispatches listeners by exact event type, and the Discord monitor currently registers only the create-message inbound listener.

## Solution Assessment

Yes for the implementation direction. The narrow maintainable path is an opt-in Discord plugin edit pipeline with debounce and replay-aware tests; delete-to-cancel and in-flight abort behavior should remain separate unless maintainers explicitly broaden scope.
