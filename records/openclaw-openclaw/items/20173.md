## Summary

Keep open. Current main and the latest release still lack Discord MESSAGE_UPDATE listener wiring, edit config/docs, and edit-aware replay semantics; related closed duplicates point back to this issue as the narrower canonical edit-to-reprocess tracker.

## What This Changes

Request to add opt-in Discord MESSAGE_UPDATE handling so edited user messages can be debounced and reprocessed as updated inbound agent turns.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay/session handling, mention-gating coverage, and focused monitor tests; keep the broader delete-to-cancel work coordinated separately.

## Reproduction Assessment

Yes, at source level. Discord exposes MESSAGE_UPDATE, but current main has no matching listener registration, and same-message edits would collide with replay semantics keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and abort/replay semantics need maintainer judgment before implementation.
