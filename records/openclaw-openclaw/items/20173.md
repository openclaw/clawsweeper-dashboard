## Summary

Keep open. Current main and v2026.5.12 still route Discord inbound turns through MESSAGE_CREATE only, with no MESSAGE_UPDATE listener, edit config surface, or edit-aware replay semantics; the prior duplicate reports point back to this issue while https://github.com/openclaw/openclaw/issues/53654 remains broader because it also covers delete-to-cancel.

## What This Changes

The issue requests opt-in Discord MESSAGE_UPDATE handling so edited user messages can trigger a fresh agent turn with the updated content.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, mention-gating coverage, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord exposes MESSAGE_UPDATE, but current main has no matching listener registration and same-message edits would hit replay semantics based on the original account/channel/message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment before implementation.
