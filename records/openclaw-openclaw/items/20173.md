## Summary

Keep open. Current main and v2026.5.12 still do not register or hydrate Discord MESSAGE_UPDATE events, and the requested opt-in edit reprocessing needs maintainer decisions for config, debounce/rate-limit, partial update filtering, and replay semantics before it is a safe automated fix.

## What This Changes

Request opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger agent processing with debouncing.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests; keep delete-to-cancel separate unless maintainers choose to combine it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: MESSAGE_UPDATE has no exact listener registered, and a same-message edit would need explicit replay semantics beyond the current account/channel/message key.

## Solution Assessment

Yes on owner boundary, but not as an automatic bug fix. The Discord plugin is the right place, while config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment.
