## Summary

Keep open. Current main and v2026.5.12 still route Discord inbound text through create-message handling only, with no MESSAGE_UPDATE listener, no edit config surface, and no edit-aware replay semantics; the related duplicate was closed in favor of this tracker while the broader delete-to-cancel request remains separate.

## What This Changes

Request opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger agent processing with debounce and edit-aware behavior.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests; keep delete-to-cancel separate unless maintainers choose to combine it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord MESSAGE_UPDATE exists in the locked gateway types, but current main has no matching listener registration and same-message edits would need replay semantics beyond the current account/channel/message key.

## Solution Assessment

Yes on owner boundary, but not as an automatic repair. The Discord plugin is the right place, while config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment.
