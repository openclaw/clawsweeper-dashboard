## Summary

Keep open. Current main still has no Discord MESSAGE_UPDATE inbound path: the Discord monitor only registers create-message handling, the internal Discord listener/dispatch layer has no MessageUpdate listener or normalization, and the requested handleEdits/editDebounceMs config/docs surface is absent. The prior maintainer-side keep-open comment remains directionally correct, with the implementation target now being the internal Discord client rather than Carbon after the latest refactor.

## What This Changes

Enable the bundled Discord plugin to detect Discord MESSAGE_UPDATE events and optionally reprocess edited user messages with debounce and backward-compatible config.

## Best Possible Solution

Keep this issue open as the canonical Discord edit-to-reprocess tracker. Implement the feature inside the bundled Discord plugin by adding internal MessageUpdate event support, an opt-in config/schema/docs surface, content-change filtering, per-message debounce, edit-aware replay semantics, and focused tests for disabled-by-default behavior plus edits that add a required bot mention.

## Risks / Open Questions

- A naive edit handler could retrigger on embed-only updates, bot-authored edits, or rapid edit streams; the implementation needs content-change filtering and per-message debounce.
- The existing same-id replay guard intentionally suppresses duplicate create deliveries, so edit-aware dedupe must be added without regressing duplicate MESSAGE_CREATE suppression.
- Abort/supersede behavior for edits during an in-flight turn needs a scoped product decision; delete-to-cancel from #53654 should stay out of the first fix unless maintainers explicitly broaden scope.
