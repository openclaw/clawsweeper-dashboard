## Summary

Keep open. Current main still lacks Discord MESSAGE_UPDATE listener wiring, edit-specific config/docs, and edit-aware replay semantics; the closed Discord edit duplicates point back to this issue, while the broader delete-to-cancel tracker does not replace this edit-only request.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger an agent turn with debounce and edit-aware replay semantics.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, changelog, and focused monitor tests; keep delete-to-cancel separate unless maintainers explicitly fold in the broader tracker.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Discord gateway dispatch uses exact listener types, only MESSAGE_CREATE is registered for inbound text, and same-message replay has no edit-aware path.

## Solution Assessment

Yes on the owner boundary, but not as an auto-close or automatic bug-fix. The bundled Discord plugin is the right place to solve it, while config defaults, debounce, partial-update handling, and abort/replay policy need maintainer judgment.
