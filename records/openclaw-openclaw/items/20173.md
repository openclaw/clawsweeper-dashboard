## Summary

Keep open. Current main still lacks Discord edited-message inbound handling: the monitor only registers create-message handling, Discord config/docs have no opt-in edit surface, and the replay guard is keyed in a way that would need an edit-aware policy. The April and May issue comments remain accurate, and this remains the canonical Discord edit-to-reprocess tracker.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger agent processing with debounce and edit-aware replay semantics.

## Best Possible Solution

Implement the Discord edit path inside the bundled Discord plugin with opt-in config, MessageUpdate listener support, payload normalization, content-change filtering, per-message debounce, edit-aware replay handling, docs/schema/changelog updates, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Discord MESSAGE_UPDATE dispatch would only reach a matching update listener, but current main defines and registers only the create-message listener path for inbound message handling.

## Solution Assessment

Yes. The narrow maintainable solution is an opt-in Discord plugin implementation for edited-message reprocessing; delete-to-cancel and in-flight abort behavior should stay separate unless maintainers explicitly broaden the feature.
