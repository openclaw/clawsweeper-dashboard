## Summary

Keep open. Current main and the latest release still lack a Discord MESSAGE_UPDATE listener/registration path, no edit-specific Discord config or docs surface exists, and the current same-message replay guard would treat a naive edited-message reprocess as a duplicate without explicit edit-aware semantics.

## What This Changes

Request to add opt-in Discord MESSAGE_UPDATE handling so edited Discord user messages can trigger a new agent turn with the updated content.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests while keeping delete-to-cancel scope separate unless maintainers choose to combine it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Discord MESSAGE_UPDATE dispatch has no exact registered monitor listener, and a same-message edit has no edit-aware replay key.

## Solution Assessment

Yes on the owner boundary, but not as an automatic bug fix. The Discord plugin is the right place to solve it, while config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment.
