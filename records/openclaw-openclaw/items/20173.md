## Summary

Keep open. Current main and v2026.5.12 still handle Discord inbound turns through message-create only, with no MESSAGE_UPDATE listener, no edit config surface, and no edit-aware replay semantics; the closed duplicates point back here while the broader delete-to-cancel tracker remains separate.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can trigger a new agent turn with updated content.

## Best Possible Solution

Implement the feature in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests; keep delete-to-cancel separate unless maintainers choose to combine it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: discord-api-types exposes MESSAGE_UPDATE, but current main has no matching listener registration and same-message edits would need replay semantics beyond the current account/channel/message key.

## Solution Assessment

Yes on owner boundary, but not as an automatic repair. The Discord plugin is the right place, while config defaults, debounce/rate-limit behavior, partial update handling, and abort/replay semantics need maintainer judgment.
