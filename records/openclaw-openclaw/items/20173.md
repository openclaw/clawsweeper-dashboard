## Summary

Keep open. Current main still only wires Discord inbound text through the create-message listener path; there is no MESSAGE_UPDATE listener, edit-aware gateway normalization, opt-in edit config, docs, or test coverage, and the existing comments correctly keep this as the canonical Discord edit-to-reprocess tracker.

## What This Changes

Request to add opt-in Discord MESSAGE_UPDATE handling so edited user messages can be reprocessed after debounce.

## Best Possible Solution

Implement an opt-in Discord edit pipeline in the bundled plugin that registers MESSAGE_UPDATE, normalizes edited payloads, filters real content changes, debounces per message, and replays edited messages without folding in delete-to-cancel unless maintainers choose that separately.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: MESSAGE_UPDATE is a distinct Discord gateway event, Client.dispatchGatewayEvent only calls listeners of the same type, and the Discord monitor currently registers only the create-message inbound listener.

## Solution Assessment

Yes. The narrow maintainable direction is an opt-in Discord plugin implementation with config, docs, debounce, edit-aware replay handling, and focused monitor tests; delete-to-cancel and in-flight abort behavior should remain separate unless the scope is explicitly broadened.
