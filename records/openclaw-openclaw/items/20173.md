## Summary

Keep open. Current main still lacks a Discord MESSAGE_UPDATE inbound path, edit-processing config/docs, and edit-aware replay semantics; existing discussion correctly treats this as the canonical edit-to-reprocess tracker while the broader edit/delete request remains separate.

## What This Changes

Add optional Discord handling for edited user messages so updated content can trigger a new agent turn after a MESSAGE_UPDATE event.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE support, content-change filtering, per-message debounce, edit-aware replay handling, config/docs metadata, changelog, and focused monitor tests; keep delete-to-cancel separate unless maintainers explicitly broaden scope.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Discord gateway dispatch uses exact listener types, no registered listener handles MESSAGE_UPDATE payloads, and same-id replay currently needs explicit edit-aware semantics.

## Solution Assessment

Yes for the implementation direction, but not yet for automation. The narrow maintainable path is a Discord-plugin edit pipeline; the opt-in config/defaults and edit replay semantics should be settled by maintainers before a repair lane changes behavior.
