- `impact:message-loss`: Edited Discord messages are currently not delivered into the inbound Discord agent workflow.
- `impact:session-state`: Corrected same-message content can leave the agent session based on stale original message text.

## Summary

Keep open. Current main still wires Discord inbound user text through MESSAGE_CREATE only; MESSAGE_UPDATE listener registration, opt-in edit config/debounce, and edit-aware replay semantics remain missing, while the related closed duplicates point back to this issue as the canonical edit-to-reprocess tracker.

## What This Changes

The issue requests opt-in Discord MESSAGE_UPDATE handling so edited user messages can trigger a fresh agent turn after debounce.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, opt-in config/docs, content-change filtering, per-message debounce, edit-aware replay/session handling, mention-gating coverage, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Discord defines MESSAGE_UPDATE, but current main has no matching listener registration, and same-message edits would collide with replay semantics keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and abort/replay semantics need maintainer judgment before implementation.
