## Summary

Keep open. Current main still lacks Discord edited-message inbound handling: the monitor registers only create-message handling, the config/docs have no opt-in edit surface, and the existing replay guard would suppress same-message edit semantics without a dedicated policy. The April maintainer review remains accurate and this issue is still the canonical Discord edit-to-reprocess tracker.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can be debounced and reprocessed as updated inbound turns.

## Best Possible Solution

Implement the Discord edit path inside the bundled Discord plugin with an opt-in account config, gateway update listener, payload normalization, content-change filtering, per-message debounce, edit-aware replay semantics, docs/schema updates, and focused monitor tests. Carry the add-mention-on-edit behavior from #48894 and leave delete-to-cancel as separate work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a Discord MESSAGE_UPDATE dispatch would only reach listeners registered for that event type, but current main registers no update listener and only normalizes MESSAGE_CREATE into the internal message dispatch shape.

## Solution Assessment

Yes, the requested opt-in Discord plugin implementation is the narrow maintainable direction. The safer boundary is to handle edited-message reprocessing first and avoid folding in in-flight abort or delete-to-cancel semantics unless maintainers explicitly choose that broader behavior.
