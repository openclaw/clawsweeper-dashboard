## Summary

Keep open as the canonical Discord edit-to-reprocess tracker. Current main and release v2026.6.5 still receive inbound user messages only through the create-message listener, while the closed duplicate reports direct their edit-specific requirements here; the broader open report also requests delete-to-cancel and does not safely supersede this focused item. Likely related people: @vincentkoc (high confidence, substantial prior Discord monitor/config history), @steipete (medium confidence, recent Discord contributor and reviewer), and Tak Hoffman (medium confidence, introduced the related inbound replay guard).

## What This Changes

Add Discord edited-message handling that can reprocess updated user content, with proposed opt-in configuration, debounce, and edit-aware conversation behavior.

## Best Possible Solution

Keep this issue as the focused canonical tracker and implement Discord-owned content-change and mention filtering, bounded per-message debounce, edit-aware replay keys, and explicit transcript/active-run semantics; leave delete-to-cancel separate unless maintainers deliberately expand scope.

Yes at source level with high confidence: current main and v2026.6.5 register only the Discord create-message inbound path, so a user edit cannot trigger processing. This read-only review did not execute a live Discord edit.

## Solution Assessment
