## Summary

Keep open. Current main and v2026.6.5 still lack Discord MESSAGE_UPDATE listener wiring, requested edit config/docs, and edit-aware replay semantics; https://github.com/openclaw/openclaw/issues/53654 overlaps but is broader because it also asks for delete-to-cancel.

## What This Changes

Requested opt-in Discord handling for MESSAGE_UPDATE events so edited user messages can re-trigger agent processing with debounce and edit-aware semantics.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay/session handling, mention-gating coverage, and focused monitor tests while coordinating delete-to-cancel separately.

## Reproduction Assessment

Yes, at source level. Current main has no Discord message-update listener registration, no requested edit config surface, and a same-message-id replay guard that would treat naive edits as duplicates.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but maintainers still need to choose the edit policy, default/config shape, debounce/rate-limit behavior, partial update hydration, and replay/abort semantics.
