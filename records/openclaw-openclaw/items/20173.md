## Summary

Keep open. Current main and v2026.5.19 still only route Discord inbound user messages through MESSAGE_CREATE, while the requested MESSAGE_UPDATE listener, edit config, debounce, and edit-aware replay semantics remain missing.

## What This Changes

Request to add opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger agent processing with updated content.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, mention-gating coverage, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Discord exposes MESSAGE_UPDATE, but current main has no matching listener registration and same-message edits would collide with replay semantics keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and abort/replay semantics need maintainer judgment before implementation.
