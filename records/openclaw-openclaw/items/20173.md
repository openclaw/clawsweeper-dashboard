## Summary

Keep open. Current main still has no Discord MESSAGE_UPDATE inbound path, no edit-processing config/docs surface, and no edit-aware replay semantics; the prior discussion correctly treats this issue as the canonical Discord edit-to-reprocess tracker while https://github.com/openclaw/openclaw/issues/53654 remains the broader edit/delete scope.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger an agent turn with the updated content.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE support, content-change filtering, per-message debounce, edit-aware replay handling, config/docs metadata, changelog, and focused monitor tests; keep delete-to-cancel separate unless maintainers explicitly broaden scope.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Discord gateway dispatch matches exact listener types, and no registered listener handles MESSAGE_UPDATE payloads.

## Solution Assessment

Yes for the implementation direction. The narrow maintainable path is a Discord-plugin edit pipeline, but the opt-in config/defaults and edit replay semantics need maintainer scoping before implementation.
