## Summary

Keep open. Current main still handles Discord inbound text through MESSAGE_CREATE only and lacks the requested MESSAGE_UPDATE listener, edit-aware replay semantics, opt-in config/docs, and tests; the recent comments correctly keep this as the canonical Discord edit-to-reprocess tracker while #53654 remains broader delete-to-cancel scope.

## What This Changes

Add opt-in Discord MESSAGE_UPDATE handling so edited user messages can re-trigger agent processing with debounce/edit semantics.

## Best Possible Solution

Implement this in the bundled Discord plugin with MESSAGE_UPDATE registration, edited-payload normalization, content-change filtering, per-message debounce, edit-aware replay handling, config/docs metadata, changelog, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord gateway dispatch is exact-type listener based, and the monitor currently registers only a MessageCreate-backed inbound message listener.

## Solution Assessment

Yes for the implementation direction. The narrow maintainable path is an opt-in Discord plugin edit pipeline with debounce and replay-aware tests; delete-to-cancel and in-flight abort should remain separate unless maintainers explicitly broaden scope.
