## Summary

Keep open. Current main still has no Discord `MESSAGE_UPDATE` listener path, no edit-specific Discord config/docs surface, and no edit-aware replay semantics, while the closed duplicate reports point back to this issue as the canonical edit-to-reprocess tracker.

## What This Changes

Add opt-in Discord edited-message handling so edited user messages can re-trigger agent processing with debounce and edit-aware behavior.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed `MESSAGE_UPDATE` dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, and focused monitor tests; keep delete-to-cancel in the broader tracker unless maintainers explicitly combine them.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Discord `MESSAGE_UPDATE` event would not match any registered monitor listener, and same-message edits have no edit-aware replay key. I did not run a live Discord E2E, but the current dispatch/listener wiring is exact enough to prove the gap from source.

## Solution Assessment

Yes on the owner boundary, but not as an automatic bug-fix. The Discord plugin is the right place to solve it, while config defaults, debounce behavior, partial update handling, and abort/replay semantics need maintainer judgment.
