## Summary

Keep open. Current main still has no Discord MESSAGE_UPDATE listener, monitor registration, edit-processing config/docs, or edit-aware replay semantics; the overlapping broader Discord update/delete item remains coordination context rather than a replacement for this edit-only tracker.

## What This Changes

The issue requests opt-in Discord MESSAGE_UPDATE handling so edited user messages can trigger a new agent turn with updated content.

## Best Possible Solution

Implement Discord edit handling in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, changelog, and focused monitor tests; keep delete-to-cancel separate unless maintainers intentionally broaden scope.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: gateway dispatch uses exact listener types, no registered listener handles MESSAGE_UPDATE payloads, and the same-message replay guard has no edit-aware semantics.

## Solution Assessment

Yes on direction, unclear on final policy. A Discord-plugin edit pipeline is the maintainable place to solve this, but maintainers should decide defaults, partial update handling, debounce/rate-limit behavior, and replay/abort semantics before automation changes behavior.
