## Summary

Keep open. Current main still lacks Discord MESSAGE_UPDATE listener wiring, edit-specific config/docs, and edit-aware replay semantics; the closed duplicate threads point back here, while the broader delete-to-cancel tracker does not replace this edit-only request.

## What This Changes

Request to add opt-in Discord MESSAGE_UPDATE handling so edited user messages can retrigger agent processing with updated content.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay handling, changelog, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: MESSAGE_UPDATE has no registered Discord monitor listener, and same-message replay currently has no edit-aware path.

## Solution Assessment

Yes on the owner boundary, but final policy is still a maintainer decision. The Discord plugin is the right place to solve it, while defaults, partial update handling, debounce, and abort/replay semantics should be settled before implementation.
