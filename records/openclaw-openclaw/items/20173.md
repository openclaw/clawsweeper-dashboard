- `impact:message-loss`: Edited Discord messages are currently not delivered into the inbound Discord agent workflow.
- `impact:session-state`: Corrected same-message content can leave the agent session based on stale original message text.

## Summary

Keep open. Current main still has a create-only Discord inbound path, while the requested MESSAGE_UPDATE listener, opt-in edit config, debounce, and edit-aware replay behavior remain missing; closed duplicates already point back to this issue as the narrower canonical edit tracker.

## What This Changes

Request that the Discord plugin optionally handle MESSAGE_UPDATE events so edited user messages can be debounced and reprocessed as updated inbound turns.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay/session handling, mention-gating coverage, and focused monitor tests; keep https://github.com/openclaw/openclaw/issues/53654 coordinated for the broader delete-to-cancel scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Discord exposes MESSAGE_UPDATE, but current main has no matching listener registration, and same-message edits would collide with replay semantics keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and abort/replay semantics need maintainer judgment before implementation.
