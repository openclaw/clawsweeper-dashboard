## Summary

Keep open. Current main and the latest release still have a create-only Discord inbound path, while the requested MESSAGE_UPDATE listener, opt-in edit config, debounce, and edit-aware replay semantics remain missing; closed duplicates already point back to this issue as the narrower canonical edit tracker.

## What This Changes

Request re-processing edited Discord user messages via MESSAGE_UPDATE, with opt-in edit handling and debounce configuration.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed MESSAGE_UPDATE dispatch, content-change filtering, opt-in config/docs, per-message debounce, edit-aware replay/session handling, mention-gating coverage, and focused monitor tests; keep https://github.com/openclaw/openclaw/issues/53654 coordinated for broader delete-to-cancel scope.

## Reproduction Assessment

Yes, at source level. Discord exposes MESSAGE_UPDATE, but current main has no matching listener registration, and same-message edits would collide with replay semantics keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and abort/replay semantics need maintainer judgment before implementation.
