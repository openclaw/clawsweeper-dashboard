## Summary

Keep open. Current main still has a source-level clobber path: configure asks to preserve the existing primary, but provider auth can merge xAI/OpenRouter-style patches that set `agents.defaults.model.primary` before the preservation helper reads the previous primary; onboard also enters provider auth with unconditional default-setting behavior.

## What This Changes

The issue asks provider setup to preserve an existing primary model when adding another provider unless the user explicitly chooses to change it.

## Best Possible Solution

Carry the pre-auth primary model through provider-auth application and use that original value for preservation decisions, while allowing explicit default-setting commands and fresh setup paths to keep changing the primary deliberately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start with a config whose primary is a Bedrock model, run configure/provider auth with `preserveExistingDefaultModel: true`, and use an xAI/OpenRouter-style auth method that returns both a primary-setting `configPatch` and `defaultModel`; current code reads the patched primary as the previous primary.

## Solution Assessment

Yes. A narrow fix in the provider-auth application path is the best approach because it aligns the implementation with existing docs and PR #70793 without changing provider preset helpers or explicit `--set-default` semantics.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug for https://github.com/openclaw/openclaw/issues/50268. Current behavior: `openclaw configure` passes `preserveExistingDefaultModel: true`, but provider auth merges `configPatch` before `applyDefaultModelFromAuthChoice` reads the previous primary, so xAI/OpenRouter-style patches that already set `agents.defaults.model.primary` make the new provider look like the old primary. Carry the pre-auth config or pre-auth primary into the preservation decision and ensure provider patches can still add auth profiles, provider config, aliases, model allowlist entries, and notes. Also review classic onboarding: rerunning onboarding to add a provider should not silently clobber an existing primary unless the flow is fresh setup or the user explicitly chooses a new default. Do not change explicit default-setting behavior such as `openclaw models auth login --provider <id> --set-default`, do not remove provider `primaryModelRef` presets, and do not fold in the broader migration/billing policy from #74395. Add focused regression coverage for a provider auth method that returns both a `configPatch` containing `agents.defaults.model.primary` and a `defaultModel`, including xAI/OpenRouter-shaped behavior. Add a single-line user-facing changelog fix entry. Validate with the targeted auth/provider tests and `pnpm check:changed` in Testbox by default.
