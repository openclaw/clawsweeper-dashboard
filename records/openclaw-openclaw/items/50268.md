## Summary

Keep open. Current main still has the source-level clobber path: provider auth can merge a provider-owned config patch that already rewrites `agents.defaults.model.primary`, then the preservation helper treats that mutated primary as the previous value. PR #70793 fixed the adjacent configure preservation contract, and the Keep My Claw comment is a useful recovery workaround, but neither removes this OpenClaw config mutation bug.

## What This Changes

Adding a second provider through onboard/configure should preserve an existing `agents.defaults.model.primary` unless the user explicitly opts to make the new provider the default.

## Best Possible Solution

Use the pre-auth config as the source of truth for preservation decisions. Provider-owned patches should still add auth profiles, provider config, aliases, and allowlist entries, but configure/onboard should not silently replace an existing primary unless setup is fresh or the caller is an explicit default-setting command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by starting with a config whose primary is a Bedrock model, then applying an xAI/OpenRouter-style API-key auth method that returns both a primary-setting `configPatch` and `defaultModel` through the configure path with `preserveExistingDefaultModel: true`.

## Solution Assessment

Yes, a narrow code fix is the best way to solve it. Carry the original primary/pre-auth config into `applyDefaultModelFromAuthChoice` or strip provider-patch primary changes only for preservation paths, then add regression tests with real provider-style patches while leaving explicit set-default flows authoritative.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug described in this issue. In configure/onboard preservation paths, keep the pre-auth `agents.defaults.model.primary` when a provider auth method returns a `configPatch` that already sets a new primary plus a `defaultModel`. The fix should be in the generic provider-auth application path, not in xAI/OpenRouter-specific policy, and it must not change explicit default-setting commands such as `openclaw models auth login --provider <id> --set-default` or intentional default-model migrations. Add regression coverage using a provider auth method whose `configPatch` includes `agents.defaults.model.primary` and `defaultModel`, plus coverage for xAI/OpenRouter-style behavior if practical. Add a single-line changelog Fixes entry with `Thanks @rixcorp-oc` if the maintainer wants user-facing bugfix notes. Do not alter provider catalogs, auth storage semantics, or unrelated onboarding UX.
