## Summary

Keep open: current source still has a Telegram payload-send path that derives send text only from `text` or `interactive`, while the existing presentation contract says presentation payloads should degrade safely to text. The exact cron/announce failure was not live-reproduced, but the gap is narrow and source-backed enough for a focused fix candidate.

## What This Changes

The issue reports that Telegram delivery can reject replies whose content is represented only as a presentation payload without a top-level text field.

## Best Possible Solution

Add a focused Telegram-side fallback that guarantees presentation-derived payloads produce safe send text before the non-empty Telegram guard, with regression coverage for presentation-only Telegram payload delivery.

## Reproduction Assessment

No high-confidence end-to-end reproduction is established. Source inspection does show that a Telegram payload reaching `sendTelegramPayloadMessages` with only `presentation` and no rendered `text` or `interactive` fallback will pass an empty string into Telegram send validation.

## Solution Assessment

Partly: the issue points at the right boundary, but the safer fix must account for core stripping `presentation` after render. A narrow Telegram fallback plus tests for both direct payload sends and the delivery render path is the maintainable approach.
