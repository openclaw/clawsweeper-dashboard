## Summary

Keep open. Current main still routes reset-triggered active turns through the configured queue mode, and open PR #74144 is already the targeted implementation candidate for the same queue/reset path. The reporter’s later Feishu timing notes add useful adjacent evidence, but the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that reset-triggered turns such as `/new` and `/reset` bypass normal `steer` queueing and interrupt active runs immediately.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that threads reset state through the active-run queue policy, disables steer/followup behavior for reset turns, clears or aborts active work, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. Any remaining Feishu startup readiness delays should be tracked separately after the queue repair.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main marks the turn as reset-triggered but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes. Making reset-triggered turns use interrupt semantics through the existing queue-policy and runner seams is the narrowest maintainable solution for the reported delay; the later Feishu startup blockers should remain separate unless they still reproduce after that fix.
