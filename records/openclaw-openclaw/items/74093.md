## Summary

Keep open. Current main still detects reset-triggered turns, but the active-run queue policy and reply execution paths continue deriving steer/followup behavior from the configured queue mode; open PR #74144 explicitly targets this bug and should remain the implementation path.

## What This Changes

Reset-triggered `/new` and `/reset` turns should interrupt active same-session work instead of waiting behind normal steer/followup queue behavior.

## Best Possible Solution

Land an amended #74144 or equivalent core fix that carries reset-triggered interrupt intent through active-run policy, prepared queue handling, and agent-runner behavior with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode: "steer"`, followed by an authorized `/new` or `/reset`; current main records reset state but still resolves active-run queue behavior from the configured mode.

## Solution Assessment

Yes for the original queue bug. Threading reset state into active-run queue policy, suppressing steer/followup for reset turns, and making prepared queue handling abort active work is the narrow maintainable direction; the linked PR should be reviewed or amended before closure.
