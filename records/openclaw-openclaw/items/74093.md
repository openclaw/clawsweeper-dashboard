## Summary

Keep open. Current main still matches the reported queue/reset bug, and open PR #74144 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Reset-triggered chat turns such as `/new` and `/reset` should bypass active-run steer/followup queueing and interrupt or clear active work immediately.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that threads reset state through the existing active-run queue policy, disables steer/followup behavior for reset turns, clears/aborts active work, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. If Feishu startup health delays persist after that fix, track them separately.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main marks the turn as reset-triggered but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes. Making reset-triggered turns use interrupt semantics through the existing queue-policy and runner seams is the narrowest maintainable fix for the reported delay; the Feishu startup blockers should remain separate unless they still reproduce after the queue repair.
