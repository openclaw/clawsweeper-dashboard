## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy, and open PR #74144 is the explicit unmerged implementation candidate for the core reset/queue bug.

## What This Changes

Reset commands such as `/new` and `/reset` should interrupt active runs instead of being queued or steered under active-run queue modes.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that treats reset-triggered turns as interrupt semantics before steering/followup; split the Feishu startup or health-hang symptoms into a narrower follow-up if they still reproduce afterward.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current main records reset state but computes active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the core queue bug. The narrow maintainable fix is to thread reset state into active-run queue policy, suppress steer/followup for reset turns, and abort/clear active work before starting the fresh session.
