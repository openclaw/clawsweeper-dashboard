## Summary

Keep open. Current `main` still routes reset-triggered turns through the configured active-run queue policy, and open PR #74144 is the explicit implementation candidate for the core reset/queue bug.

## What This Changes

The issue asks for `/new` and `/reset` turns to interrupt active runs instead of being steered or queued behind them when queue mode is `steer`, `followup`, `collect`, or `steer-backlog`.

## Best Possible Solution

Review and land an updated #74144 or equivalent core fix for reset-triggered interrupt semantics, then split any remaining Feishu startup or health-hang behavior into a narrower report if it still reproduces.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current code records reset state but still computes active-run behavior from the configured queue mode. No live Windows/Feishu run was attempted in this read-only review.

## Solution Assessment

Yes for the core queue bug. Threading reset state into active-run queue policy, suppressing steer/followup for reset turns, and aborting/clearing active work is the narrow maintainable fix; #74144 is the current implementation path.
