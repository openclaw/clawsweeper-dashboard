## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy instead of giving them interrupt semantics, and open PR #74144 explicitly targets and closes the same queue/reset bug. The later Feishu startup notes may need a separate follow-up, but the central reset queue defect remains unresolved on main.

## What This Changes

The issue asks for `/new` and `/reset` turns to interrupt or clear active work instead of being steered or queued behind an active run when `messages.queue.mode` is `steer` or a related queue mode.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that threads reset-triggered state into active-run queue handling, suppresses steer/followup behavior for reset turns, clears/aborts the active run when needed, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. Any remaining Feishu startup readiness delay should be tracked separately after the reset queue behavior is fixed.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode: "steer"`, then an authorized `/new` or `/reset`; current main marks the turn as reset-triggered but still derives active-run queue behavior from `steer`, which returns `enqueue-followup`.

## Solution Assessment

Yes, the linked fix direction is the narrowest maintainable solution: make reset-triggered turns bypass steering/followup queue semantics and use interrupt-style active-run cleanup. The startup/model warmup observations should not be bundled into the same core queue fix unless they still reproduce after that change lands.
