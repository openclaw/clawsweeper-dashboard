## Summary

Keep open. Current `main` still has the reset intent available before reply execution, but active-run queue handling still derives steer/followup behavior from the configured queue mode, and open PR #74144 explicitly targets this bug with closing syntax.

## What This Changes

Reset-triggered `/new` and `/reset` turns should interrupt active same-session work instead of waiting behind normal steer/followup queue behavior.

## Best Possible Solution

Land an updated #74144 or equivalent core fix that makes reset-triggered turns use interrupt semantics across active-run policy and prepared queue handling, with focused regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active same-session run with `messages.queue.mode: "steer"`, followed by an authorized `/new` or `/reset`; current main records reset state but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the original queue bug. Threading reset state into active-run queue policy, suppressing steer/followup for reset turns, and clearing/aborting active work is the narrow maintainable direction; the linked PR should remain the implementation path unless maintainers replace it.
