## Summary

Keep open. Current main still matches the reported queue/reset bug, and open PR #74144 explicitly targets this issue with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt active work under steer/followup queue modes instead of being queued behind the current run.

## Best Possible Solution

Review and land PR #74144, or an equivalent core fix that threads reset state through the active-run queue policy, disables steer/followup handling for reset turns, clears or aborts active work, and adds regression coverage for the affected queue modes. Any remaining Feishu startup delay should be split into a narrower follow-up after the queue behavior is fixed.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main marks the turn as reset-triggered but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes. Making reset-triggered turns use interrupt semantics through the existing queue policy and runner seams is the narrowest maintainable fix for the reported delay; the Feishu startup observations should remain separate unless they still reproduce afterward.
