## Summary

Keep open. Current main still has reset state available before queue handling, but the active-run queue policy and both call sites still derive steer/followup behavior from the configured queue mode with no reset exception. Open PR #74144 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Reset-triggered `/new` and `/reset` turns should bypass configured steer/followup active-run queuing and interrupt or clear active work immediately.

## Best Possible Solution

Land #74144 or an equivalent core fix that treats reset-triggered turns as interrupt semantics through the existing active-run queue seams, clears queued lane work, aborts active work, suppresses steer/followup handling for reset turns, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. Preserve the later Feishu startup/session-index evidence as a follow-up if it still reproduces after the reset scheduling fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main sets `resetTriggered` but still resolves active-run behavior from `resolvedQueue.mode`, so the reset can be steered or enqueued as followup work.

## Solution Assessment

Yes. Threading reset state into the active-run queue decision and disabling steer/followup behavior for reset turns is the narrowest maintainable fix; the linked PR appears to follow that direction, but the issue should stay open until it merges or is replaced.
