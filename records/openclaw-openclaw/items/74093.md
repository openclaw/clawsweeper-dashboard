## Summary

Keep open. Current main still records reset-triggered turns but routes active-run behavior through the configured queue mode, and open PR #74144 is already the explicit implementation candidate for this core queue/reset bug.

## What This Changes

Reset commands such as `/new` and `/reset` should bypass steer/followup queueing and interrupt any active same-session run immediately.

## Best Possible Solution

Review and land #74144 or an equivalent core fix, then split the later Feishu startup/health observations into a narrower issue if they still reproduce.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current main sets `resetTriggered` but still derives active-run behavior from the configured queue mode.

## Solution Assessment

Yes. The narrow maintainable fix is to treat reset-triggered turns as interrupt semantics before steer/followup decisions, clear queued work, and abort/wait the active run; #74144 is the active implementation path.
