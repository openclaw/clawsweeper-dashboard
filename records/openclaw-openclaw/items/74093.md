## Summary

Keep open. Current main still has the reset intent available before reply execution, but the active-run queue path still derives steer/followup behavior from the configured queue mode, and open PR #74144 already targets the central bug with closing syntax.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt or clear active same-session work instead of waiting behind `steer`, `followup`, `collect`, or `steer-backlog` queue policy.

## Best Possible Solution

Review and land #74144 or an equivalent core fix for reset-as-interrupt semantics, then split any still-reproducible Feishu startup or gateway health symptoms into narrower follow-up issues.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active same-session run with `messages.queue.mode: "steer"` followed by an authorized `/new` or `/reset`; current main records reset state but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the central queue bug. Threading reset state into active-run queue policy, suppressing steer/followup for reset turns, and clearing or aborting active work is the narrow maintainable direction; the later startup/hang notes need separate validation if they remain after that fix.
