## Summary

Keep open. Current main still lets reset-triggered turns use the configured active-run queue policy, and open PR #74144 is the explicit unmerged implementation candidate for the core reset/queue bug.

## What This Changes

Reset commands such as `/new` and `/reset` should interrupt active runs instead of being queued or steered under active-run queue modes.

## Best Possible Solution

Review and land an updated #74144 or equivalent core fix that treats reset-triggered turns as interrupt semantics before steering/followup, then split any remaining Feishu startup or health-hang behavior into a narrower report if it still reproduces.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current main records reset state but computes active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the core bug direction, but the implementation should carry reset-triggered interrupt intent through both active-run policy and prepared queue handling so it cannot wait behind an active run.
