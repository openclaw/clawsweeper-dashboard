## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy, and the linked open PR #74144 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `/new` and `/reset` turns to interrupt and clear active work under steer/followup queue modes instead of waiting behind the active run.

## Best Possible Solution

Land #74144 or an equivalent core fix that makes reset-triggered turns use interrupt semantics through the existing active-run queue seams, clears/aborts active work, suppresses steer/followup handling for reset turns, and adds regression coverage across `steer`, `steer-backlog`, `followup`, and `collect`. Treat any remaining Feishu startup or session-index inconsistency as a narrower follow-up after the reset scheduling bug is fixed.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main records the reset but still resolves active-run behavior from the configured queue mode, which can enqueue the reset as followup work.

## Solution Assessment

Yes. Threading `resetTriggered` into the active-run queue policy and disabling steer/followup for reset turns is the narrowest maintainable fix for the reported delay; the open PR appears to follow that shape, but the issue should stay open until it merges or is replaced.
