## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy, and open PR #74144 explicitly targets the same reset queue bug with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Issue reports that reset commands such as `/new` and `/reset` can be delayed behind an active run when `messages.queue.mode` is `steer`, instead of interrupting and starting a fresh session immediately.

## Best Possible Solution

Land #74144 or an equivalent core fix that makes reset-triggered turns use interrupt semantics through the existing active-run queue seams, clears queued/active work for reset turns, suppresses steer/followup handling for those turns, and keeps regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. If Feishu startup or session-index inconsistency still reproduces afterward, split that into a narrower follow-up.

## Reproduction Assessment

Yes for the original queue bug. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main records reset state but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the original bug. Threading reset state into the active-run queue policy, disabling steer/followup for reset turns, and aborting/clearing active work is the narrowest maintainable fix; the later Feishu startup symptoms should be handled separately if they remain after that lands.
