## Summary

Keep open because current main still lets reset-triggered turns flow through the configured active-run queue policy, and open PR #74144 is the explicit unmerged closing PR for the core reset/queue bug.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt or clear active work instead of being steered or queued behind an active run.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that carries reset-triggered interrupt semantics through queue policy, lane clearing, prepared queue handling, and agent-runner steering, then split any remaining Feishu startup hang into a narrower report if it still reproduces.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current main records reset state but computes active-run behavior from the configured queue mode without reset input.

## Solution Assessment

Yes for the core direction. The narrow maintainable fix is to make reset-triggered turns override steer/followup semantics before active-run policy and prepared queue waiting; current main does not yet do that, while #74144 is the explicit implementation candidate.
