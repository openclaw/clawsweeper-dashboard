## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue mode, and open PR #74144 already targets this issue with closing syntax.

## What This Changes

Reset commands such as `/new` and `/reset` should interrupt active same-session runs instead of waiting behind `steer` or followup queue behavior.

## Best Possible Solution

Land #74144 or an equivalent core fix that makes reset-triggered turns use interrupt semantics in both prepared reply handling and agent execution, with focused regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active same-session run with `messages.queue.mode: "steer"`, followed by an authorized `/new` or `/reset`; current main records reset state but still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the original queue bug. The narrow maintainable fix is to thread reset state into active-run queue policy, suppress steer/followup for reset turns, and abort or clear active work before starting the fresh session.
