## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy, and open PR #74144 is the explicit unmerged implementation candidate for the same reset/queue bug.

## What This Changes

Reset commands such as `/new` and `/reset` should bypass steer/followup active-run queues and interrupt or clear the current run immediately.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that carries reset-triggered interrupt semantics through active-run policy and prepared queue handling, then track any remaining Feishu startup readiness problem separately.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an active same-session run with `messages.queue.mode` set to `steer` and an authorized `/new` or `/reset`; current main records reset state but computes active-run behavior from the configured queue mode.

## Solution Assessment

Yes for the core fix direction. The narrow maintainable solution is to force reset-triggered turns into interrupt semantics before steer/followup handling, while keeping unrelated Feishu startup blockers out of this patch.
