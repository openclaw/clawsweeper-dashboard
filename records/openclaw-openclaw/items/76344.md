## Summary

Keep open. Current main still routes stateful `openclaw googlemeet` CLI session commands through a CLI-process-local `GoogleMeetRuntime`, while the #72441 gateway-owned fix covers agent tool actions, so the empty session registry after CLI exit remains source-backed.

## What This Changes

Make Google Meet CLI session lifecycle commands keep using the gateway-owned session registry after the starting CLI process exits.

## Best Possible Solution

Route only the stateful Google Meet CLI session lifecycle commands through the existing `googlemeet.*` gateway RPC methods, while leaving OAuth/API/export helpers on their current local paths.

## Reproduction Assessment

Yes from source inspection, though not from a live Meet run here. The current CLI path creates or queries a per-process `GoogleMeetRuntime`, whose session registry is private in-memory state, so a later CLI process cannot see the prior session.

## Solution Assessment

Yes. The narrow maintainable fix is to make stateful CLI commands reuse the existing gateway methods that already own the shared Google Meet runtime; no new config or product policy is needed.
