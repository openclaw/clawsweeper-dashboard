## Summary

Keep open. Current main still routes stateful `openclaw googlemeet` CLI session commands through a CLI-process-local `GoogleMeetRuntime`, while the #72441 gateway-owned fix covers agent tool actions through gateway RPC, so the reported empty registry after CLI exit remains source-backed.

## What This Changes

The issue asks for Google Meet CLI realtime sessions created by `join` or `test-speech` to remain addressable by later `status`, `speak`, and `leave` commands after the starting CLI process exits.

## Best Possible Solution

Move stateful Google Meet CLI session lifecycle commands onto the existing `googlemeet.*` gateway RPC methods while leaving OAuth, artifact, and non-session API helpers on their current local paths.

## Reproduction Assessment

Yes from source inspection. Current CLI `join`/`test-speech` and later `status`/`speak` calls use per-process `GoogleMeetRuntime` maps, so separate CLI invocations cannot share the session registry unless they go through the gateway runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the already-registered `googlemeet.*` gateway methods for stateful CLI commands, matching the #72441 tool-action route without adding a new product surface.
