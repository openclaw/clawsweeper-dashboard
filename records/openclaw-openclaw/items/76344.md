## Summary

Keep open. Current main still routes stateful `openclaw googlemeet` CLI session commands through a CLI-process-local `GoogleMeetRuntime`, while the #72441 gateway-owned fix only covers agent tool actions, so the reported empty registry after CLI exit remains source-backed.

## What This Changes

The issue reports that Google Meet realtime sessions started from `openclaw googlemeet` CLI commands are lost after that CLI process exits, leaving `googlemeet status` empty and `speak` unable to target the session.

## Best Possible Solution

Move stateful Google Meet CLI session lifecycle commands onto existing `googlemeet.*` gateway RPC methods so CLI, tool, and gateway paths share one runtime-owned session registry.

## Reproduction Assessment

Yes from source inspection, though not from a live Meet run here. The current CLI path creates or queries a per-process `GoogleMeetRuntime`, and that runtime keeps sessions in private in-memory maps, so a later CLI process cannot see a session created by `join` or `test-speech`.

## Solution Assessment

Yes, the maintainable fix is to reuse the already-registered `googlemeet.*` gateway methods for stateful CLI commands, matching the #72441 tool-action route. Non-session OAuth/API/export helpers should stay local unless they join or query live session state.

## Repair Work Prompt

Fix the Google Meet CLI session ownership bug reported in openclaw/openclaw#76344. Stateful `openclaw googlemeet` CLI lifecycle commands that create, list, inspect, leave, or speak to live sessions should call the registered `googlemeet.*` gateway methods so sessions are owned by the gateway runtime across CLI process exits. Preserve local-only OAuth/API/export/calendar/artifact helpers where they do not depend on live session registry state. Likely commands include `join`, `create` when it joins, `test-speech`, `test-listen` if it creates a session, `status`, session `doctor`, `leave`, and `speak`; do not add a new config option or move owner-specific Google Meet behavior into core. Add focused regression coverage in `extensions/google-meet/src/cli.test.ts` proving stateful CLI commands use gateway RPC rather than an injected local runtime, and update plugin/gateway tests only where needed. Add a single-line `CHANGELOG.md` fix entry because this is user-facing. Validate with targeted Google Meet plugin tests, extension test typecheck, targeted oxfmt check, and `git diff --check`. Related context: #72441 fixed the same ownership class for agent tool actions but not CLI commands.
