## Summary

Keep open. Current main still routes stateful `openclaw googlemeet` CLI session commands through a CLI-process-local `GoogleMeetRuntime`; #72441 fixed the agent tool path by calling gateway methods, but the CLI path still loses its in-memory session registry when the command exits.

## What This Changes

Make Google Meet realtime sessions started from the `googlemeet` CLI survive CLI process exit so later status, speak, and leave commands can target the gateway-owned session.

## Best Possible Solution

Route stateful Google Meet CLI lifecycle commands through the existing `googlemeet.*` gateway methods, while keeping OAuth/API/export-only helpers local unless they need gateway-owned session state.

## Reproduction Assessment

Yes from source inspection, though not from a live Meet run. A CLI `join` or `test-speech` creates the session in an in-process `GoogleMeetRuntime` map, and a later CLI `status` or `speak` command creates a fresh runtime with an empty map.

## Solution Assessment

No, the current CLI-local runtime path is not the best way to solve the issue. The narrow maintainable fix is to reuse the already registered gateway methods for stateful Google Meet CLI commands, matching the #72441 tool-action route.
