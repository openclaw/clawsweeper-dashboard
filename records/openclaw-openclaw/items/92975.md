## Summary

Keep open: the PR targets a real Feishu inbound-message regression, but it is not mergeable because the reply dispatcher still throws when the Feishu runtime store is empty, so the central user reply path can still be logged and dropped. The supplied proof shows the runtime-store helper behavior, not an after-fix Feishu inbound message reaching the agent.

## What This Changes

The PR exposes an optional Feishu runtime getter and changes several Feishu inbound/comment/dedup call sites to use null guards when the plugin runtime store is uninitialized.

## Best Possible Solution

Fix the owner boundary so Feishu inbound reply dispatch has a real runtime available, either by correcting runtime initialization for config-selected plugins or by threading the already available runtime/core into the dispatcher, then prove a real Feishu DM reaches the agent.

## Reproduction Assessment

Yes, from source inspection: with `channelRuntime` supplied but the global Feishu runtime store empty, the PR can pass the earlier optional checks and still throw in `createFeishuReplyDispatcher()` before sending the reply. I did not run a live Feishu account, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The optional getter is a plausible partial mitigation, but the best fix must remove the remaining global-runtime dependency from the reply path or fix runtime initialization at plugin setup, then cover that path with regression proof.
