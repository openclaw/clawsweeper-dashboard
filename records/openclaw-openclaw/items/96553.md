## Summary

Keep open: current main and v2026.6.10 detect `reason=system-prompt` CLI session invalidation and replay history, but the reset is still only host-log visible on the reviewed path. The related open issues are broader or adjacent, not a canonical replacement for this active-conversation notice request.

## What This Changes

The issue requests an in-band notice or runtime event when a CLI backend session reset replays prior transcript history after a system-prompt change.

## Best Possible Solution

Design a narrow runtime-owned CLI session reset event that records the reason and replay fact, then map it to safe active-conversation/channel notice delivery without changing the load-bearing history replay prompt.

## Reproduction Assessment

Yes, by source inspection: current main returns `invalidatedReason: "system-prompt"`, logs the reset, and prepares a replay prompt, while the active conversation receives no reset notice on that path. I did not run a live gateway restart repro in this read-only review.

## Solution Assessment

Unclear pending maintainer product direction. An in-band runtime notice is plausible and maintainable, but maintainers need to decide the durable event shape, channel visibility, and session-id redaction policy first.
