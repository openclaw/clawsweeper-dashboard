## Summary

Keep open: current main and v2026.6.10 still make `system-prompt` CLI reset/reseed visible through backend logs and execution metadata, but not as a durable in-band conversation or runtime event.

## What This Changes

Request to emit a visible structured notice/event when a `claude-cli` session reset with `reason=system-prompt` replays transcript history into a fresh CLI session.

## Best Possible Solution

Define a core CLI reset-notice/event contract, then implement it through the runner/session surface with reason, safe prior/new identifiers, `historyReplayed`, and focused reset/reseed coverage.

## Reproduction Assessment

Yes at source level: current main invalidates on prompt-sensitive reuse drift, logs the reset, and builds a fresh-session reseed prompt, while returned agent metadata lacks a reset notice. I did not run a live `claude-cli` gateway reset in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. The requested structured in-band notice is plausible, but it should use a durable session/runtime event surface rather than a one-off log-to-chat shortcut.
