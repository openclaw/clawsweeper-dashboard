## Label Justifications

- `P2`: The request improves session-state legibility and operator trust for a real Claude CLI reset path, but it is not an emergency data-loss, outage, or security-bypass report.
- `impact:session-state`: The issue is about CLI session identity, transcript replay, and active-conversation state being indistinguishable from a separate session without host logs.

## Summary

Keep open: current main still treats `reason=system-prompt` CLI-session invalidation as a host-log-only reset plus reseed prompt, and the requested in-band/channel/queryable notice remains a real feature/product-design gap rather than stale cleanup.

## What This Changes

Request to surface `claude-cli` session resets caused by system-prompt changes inside the active conversation, including structured reset metadata and replay status.

## Best Possible Solution

Define a typed CLI-session-reset lifecycle event that preserves transcript ordering, gives the active agent and channel a concise reset explanation, and exposes safe correlation metadata for diagnostics.

## Reproduction Assessment

Yes at source level. Current main invalidates CLI reuse with `reason=system-prompt`, logs that reset, and can build a reseed prompt, while no inspected path emits the requested active-conversation or queryable reset notice.

## Solution Assessment

Unclear as an implementation shape. The requested direction is valid, but maintainers should choose the durable event/transcript/channel contract and metadata redaction before a fix PR hard-codes one surface.
