## Label Justifications

- `P2`: The issue reports a normal-priority session-state visibility gap that can cause false incident escalation but does not stop agent execution.
- `impact:session-state`: The affected behavior is CLI session reset and transcript replay state being invisible to the active agent/conversation.

## Summary

Keep open: current main still treats `system-prompt` CLI session resets as backend log/execution metadata only, and no exact duplicate or merged fix owns the requested in-band notice contract.

## What This Changes

The issue requests a visible, structured notice when `claude-cli` invalidates a CLI session for `reason=system-prompt` and reseeds a fresh session from transcript history.

## Best Possible Solution

Define a durable CLI reset-notice contract, then implement it through the CLI runner/session event path with reason, safe prior/new CLI identifiers, `historyReplayed`, and focused reset/reseed regression coverage.

## Reproduction Assessment

Yes, at source level: current main invalidates on system-prompt hash drift, logs the reset, and prepares a history-reseed prompt while returning no in-band reset notice metadata. I did not run a live claude-cli reset scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the event contract. The requested structured in-band notice is plausible, but it should use an existing session/runtime event or transcript notice surface rather than an ad hoc log-to-chat shortcut.
