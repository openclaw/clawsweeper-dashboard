## Summary

Keep open: current main still makes `system-prompt` CLI-session invalidation observable only through backend logs and execution metadata, while the requested active conversation/runtime reset notice remains missing.

## What This Changes

Request to surface `claude-cli` session resets and transcript replay to the active conversation with structured reset metadata.

## Best Possible Solution

Define a durable CLI-session-reset event/notice contract, then thread it from the CLI runner into active conversation and runtime query surfaces with safe identifiers, `reason`, and `historyReplayed` coverage.

## Reproduction Assessment

Yes, at source level: current main invalidates CLI reuse on system-prompt-sensitive hash drift, logs the reset, and reseeds transcript history without an active conversation reset notice. I did not run a live `claude-cli` reset scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the event contract. The requested in-band structured notice is plausible, but it should use an existing session/runtime notice surface rather than an ad hoc log-to-chat shortcut.
