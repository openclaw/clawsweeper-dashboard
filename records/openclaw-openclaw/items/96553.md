## Summary

Keep open: current main still invalidates and reseeds CLI sessions after system-prompt-sensitive changes, but the reset is only evidenced through backend logs and execution metadata, not a durable in-band conversation/runtime notice.

## What This Changes

Request to surface claude-cli session resets caused by system prompt changes to the active conversation with structured reset metadata.

## Best Possible Solution

Choose a durable reset-notice contract, then implement it through the CLI runner/session event surface with reason, prior/new CLI ids when safe, and focused reset/reseed regression coverage.

## Reproduction Assessment

Yes, at source level: current main invalidates on system-prompt hash drift, logs the reset, and reseeds with prior transcript history, while no reset notice is present in the active-result metadata or documented channel surface. I did not run a live claude-cli reset scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the event contract. The requested in-band structured notice is plausible, but it should land through an existing session/runtime event or transcript notice surface rather than an ad hoc log-to-chat shortcut.
