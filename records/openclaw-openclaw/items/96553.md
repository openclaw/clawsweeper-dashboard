## Label Justifications

- `P2`: The issue describes a normal-priority session-state visibility gap that can trigger false incident investigations but does not stop the agent from continuing.
- `impact:session-state`: The reported problem is that CLI session reset and transcript replay state is invisible to the active agent/session and can be mistaken for a second session.

## Summary

Keep open: current main still resets and reseeds CLI sessions on system-prompt-sensitive drift, but the reset is only surfaced through backend logs/reseed metadata rather than a durable in-band session or channel notice.

## What This Changes

Request to emit a visible structured notice/event when a CLI backend resets a session for `reason=system-prompt` and replays transcript history.

## Best Possible Solution

Define a durable CLI reset notice contract, then implement it through the existing session/runtime event surface with focused reset and reseed regression coverage.

## Reproduction Assessment

Yes, at source level: current main invalidates CLI session reuse for system-prompt-sensitive hash drift, logs the reset, and reseeds the fresh CLI session from history. I did not run a live Claude CLI reset scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the event contract. The requested in-band structured notice is plausible, but it should use an existing session/runtime event or transcript notice surface rather than an ad hoc log-to-chat shortcut.
