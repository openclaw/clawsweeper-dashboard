## Summary

Keep open: current main and v2026.6.10 still make system-prompt CLI session resets observable through host logs and reseed prompts, but not as an in-band active-conversation or queryable runtime event. The request is useful and source-reproducible, but the exact event surface and session-id visibility policy need maintainer product direction before an implementation lane should run.

## What This Changes

The issue asks OpenClaw to emit an in-band, structured notice when a CLI backend session is reset because the system prompt changed and transcript history is replayed.

## Best Possible Solution

Define one structured CLI-session-reset runtime event at the CLI reuse boundary, with maintainer-approved redaction and projection rules for transcript, channel, and queryable runtime surfaces.

## Reproduction Assessment

Yes, at source level: changing the stored CLI prompt/tool-policy fingerprints makes `resolveCliSessionReuse` return `reason=system-prompt`, after which `prepareCliRunContext` logs the reset and `executePreparedCliRun` can run with a reseed history prompt. I did not run a live gateway/webchat reproduction.

## Solution Assessment

Unclear as a final design. Surfacing a structured reset event is the right direction, but the best implementation should use an existing runtime-context or session-event seam rather than ad hoc log parsing, with maintainer direction on channel visibility and session-id redaction.
