## Summary

Keep open: current main and v2026.6.10 still log and reseed CLI session resets without a durable in-band notice; the source gap is real, but choosing the conversation/runtime event contract needs maintainer product judgment.

## What This Changes

The issue requests a visible, structured reset event or notice when claude-cli invalidates a reusable CLI session for reason system-prompt and replays prior transcript into a fresh CLI session.

## Best Possible Solution

Define a durable CLI reset-notice contract, attach it to the existing runtime/result or status-notice surface, and render it consistently without replacing the load-bearing transcript reseed prompt.

## Reproduction Assessment

Yes, at source level: current main invalidates on system-prompt-sensitive reuse changes, logs the reset, and can reseed transcript history, but does not attach a reset notice to active conversation metadata. I did not run a live claude-cli reset scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the event contract. The requested structured in-band notice is plausible, but it should land through an existing session/runtime event or status-notice surface rather than an ad hoc log-to-chat shortcut.
