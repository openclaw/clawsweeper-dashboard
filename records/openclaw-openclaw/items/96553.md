## Summary

Keep open. Current main still treats `system-prompt` CLI session invalidation as host-side logging plus transcript reseeding, so the requested in-band/session-visible reset notice remains meaningful and needs maintainer direction on event shape and identifier exposure.

## What This Changes

Request to emit an in-band and ideally queryable event when the `claude-cli` backend resets a CLI session for `reason=system-prompt` and replays prior history.

## Best Possible Solution

Define a typed CLI-session-reset event with data minimization and ordered delivery, then render a short in-band notice only on maintainer-approved channel/prompt surfaces while keeping transcript replay intact.

## Reproduction Assessment

Yes at source level. Current main has a `system-prompt` invalidation path, logs the reset host-side, and builds a transcript replay prompt without any visible reset notice in the active conversation; I did not run a live gateway reproduction.

## Solution Assessment

Unclear until maintainers pick the surface. The narrow code direction is plausible, but the best fix needs a product decision on whether this is prompt-visible, channel-visible, queryable runtime metadata, or a combination.
