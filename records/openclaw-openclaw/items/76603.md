## Summary

Keep open. Current main still has the reported control-plane/runtime mismatch: bundled manifest tool contracts can add disabled bundled plugins to the runtime tool-discovery scope, and registry reuse then rejects the otherwise usable registry because every required id is not loaded.

## What This Changes

The issue requests that enabled plugin-backed tools such as `browser`, `memory_search`, and `memory_get` remain available when unrelated bundled tool plugins are disabled.

## Best Possible Solution

Filter runtime tool-discovery required plugin ids to the actually enabled/loadable plugin owners, while preserving bundled manifest visibility for control-plane catalog and setup flows.

## Reproduction Assessment

Yes, at source level: the current `resolvePluginToolRuntimePluginIds()` can include bundled disabled tool-contract plugins, and `getLoadedRuntimePluginRegistry()` rejects registries unless all required ids are loaded. I did not execute a live inventory run, so this is not marked as a live reproduced run.

## Solution Assessment

Yes, the issue's option to exclude disabled plugins from the runtime tool-discovery required-id set is the narrowest maintainable fix. Changing global control-plane availability would be riskier because catalog, setup, and doctor flows intentionally need bundled manifest metadata without loading runtime code.

## Repair Work Prompt

Fix the plugin tool discovery bug from https://github.com/openclaw/openclaw/issues/76603. When resolving plugin-backed tools, disabled bundled plugins that merely declare `contracts.tools` must not become required runtime plugin ids and must not cause enabled plugin tools such as `browser`, `memory_search`, or `memory_get` to disappear. Keep the fix in the shared plugin tool-discovery/runtime-scope path; do not change global control-plane manifest availability for bundled plugins, do not load disabled plugin runtimes, and do not add a new config option. Add a regression test in `src/plugins/tools.optional.test.ts` or the closest existing plugin-tool test that sets up an enabled browser/memory-style plugin plus a disabled bundled tool-contract plugin, then asserts the enabled plugin tools still resolve and the disabled plugin id is not required for registry reuse. Add a single-line `CHANGELOG.md` fix entry because this is user-facing. Validate with the focused plugin-tool tests and the changed gate via Testbox if promoted.
