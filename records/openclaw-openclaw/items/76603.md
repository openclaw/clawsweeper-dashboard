## Summary

Keep open. Current main still has the source-level failure path described in the report: bundled plugin manifests with tool contracts are considered available for control-plane discovery, then the resulting runtime plugin id set is passed as required loaded ids, so disabled bundled plugins can make the runtime registry lookup fail before enabled plugin tools are appended.

## What This Changes

Ensure disabled unrelated bundled tool plugins do not cause enabled plugin-backed browser and memory tools to disappear from the agent tool surface.

## Best Possible Solution

Filter the runtime tool discovery scope to actually enabled/loadable plugin ids, or pass only that subset as `requiredPluginIds`, with regression coverage for browser and memory-core enabled while unrelated bundled tool plugins are disabled.

## Reproduction Assessment

Yes, we have a high-confidence source reproduction path: current `resolvePluginTools` can require disabled bundled plugin ids as loaded runtime plugins before returning any enabled plugin tools. I did not execute a live agent session, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, the narrowest maintainable fix is to keep disabled bundled plugins out of the runtime required-id set for tool discovery, rather than treating disabled plugins as globally loaded. The existing draft PR for #76507 addresses a related full-profile symptom but not this registry-scope root cause.
