## Summary

Keep this issue open: independent macOS and Linux reports provide strong evidence of a real long-running gateway availability problem, but the proposed diagnostic spread/Map changes cannot fully explain failures observed on v2026.6.1 because those source files were not present in that release. Current main now bounds the ACP session registry, yet no evidence establishes that this registry owns the large serialized session strings seen in the heap snapshot. Likely related people include @steipete for the diagnostic/session architecture, @openperf for recent diagnostic activity cleanup, and @osolmaz for model-stream progress work.

## What This Changes

Investigate and fix long-running gateway memory and CPU growth that culminates in OOM-driven restart loops.

## Best Possible Solution

Build a focused current-main retention harness that repeatedly creates, streams, closes, and resumes Codex/ACPX sessions, compare heap dominators and retained instances between batches, then release or bound the earliest OpenClaw-owned cache or lifecycle object retaining duplicate session payloads; optimize the newer diagnostic chunk-copy path separately if profiling confirms material cost.

## Reproduction Assessment

Yes at the reported-behavior level: independent hosts provide consistent RSS growth, restart, CPU, native-stack, and heap-snapshot evidence, including a run with the original custom plugins disabled. The exact dominant retaining edge on current main still needs a focused heap harness before implementation.

## Solution Assessment

No. The suggested spread rewrites and Map cleanup are plausible optimizations, but they do not yet explain the v2026.6.1 failure or identify the owner retaining duplicate session payloads; automatic restarts and heap limits are safeguards rather than the best root fix.
