## Summary

Keep this issue open: two independent deployments report severe uptime-correlated gateway memory growth, and current main contains plausible process-lifetime diagnostic registries, but the proposed object-spread explanation is not yet supported by heap-retainer evidence. Existing cleanup PRs cover adjacent stale markers rather than proving or fully resolving the reported RSS growth.

## What This Changes

Investigate and fix sustained gateway memory growth and eventual OOM/restart behavior during long-running model and session activity.

## Best Possible Solution

Define the production lifetime of a diagnostic SessionActivity, reclaim every session/run alias once no recoverable owner or work remains, and validate bounded registry and heap growth with repeated completed runs plus heap-snapshot dominator deltas; treat heap limits or automatic restarts as a separate product safeguard.

## Reproduction Assessment

Yes at the symptom level: two independent hosts report uptime-correlated RSS growth that resets after restart, including a report with the custom plugins disabled. No high-confidence root-cause reproduction exists yet because no current-main heap comparison or bounded-registry harness identifies the dominant retained owner.

## Solution Assessment

No. The requested heap cap and scheduled or RSS-triggered restarts require product and operational-policy decisions, while replacing spreads is not supported as the root fix; lifecycle-correct reclamation of the proven retained structure is the narrower solution.
