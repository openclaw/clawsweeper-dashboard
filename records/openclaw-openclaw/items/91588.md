## Summary

Keep this open as a credible core gateway stability incident: multiple independent hosts report uptime-correlated RSS growth and event-loop failure on v2026.6.1, while current main still has the relevant diagnostics and process-lifetime state surfaces and no canonical fix PR owns the regression. The discussion materially improves the report by isolating custom plugins and contributing native-stack and heap evidence, but it does not yet prove that object spread is the retaining cause; current main already bounds several suspected caches, so the next step is a dominator-backed owner investigation rather than a syntax-only patch or restart workaround.

## What This Changes

Investigate and fix long-running gateway memory growth and OOM/restart behavior, including retained session/run state and per-stream diagnostic overhead.

## Best Possible Solution

Capture comparable current-main heap snapshots and allocation profiles before and after sustained model/session activity, identify the dominant retaining path and owner, then release or bound that canonical state and add a focused leak harness plus per-stream performance coverage; treat heap limits or restarts only as a separately approved operational policy, not the fix.

## Reproduction Assessment

Yes for the production symptom at medium confidence: several hosts on v2026.6.1 show uptime-correlated RSS growth that resets on gateway restart, including reproduction with the original custom plugins disabled. No for a high-confidence single-cause reproduction: a controlled current-main harness or heap-dominator comparison has not yet tied the growth to one owner.

## Solution Assessment

Keeping the issue open for root-cause work is the best course, but the proposed spread-to-assignment changes are not yet proven sufficient and the restart/heap-limit proposals are operational mitigations. The narrow maintainable solution is to fix the proven retaining lifecycle owner and separately benchmark the per-stream sizing path.
