## Summary

Keep open. Current main has meaningful `@openclaw/sdk` progress, but it still does not satisfy the requested reusable, public, platform-agnostic Gateway WebSocket client because the package is private and Node-built, the default transport wraps the internal Node client, and Control UI still carries its own browser client. Brani5's linked `openclaw-gateway-client` reference is useful workaround context, not a core substitute.

## What This Changes

The issue requests a reusable public Gateway WebSocket client SDK package for third-party apps and eventual CLI/UI convergence.

## Best Possible Solution

Define and ship a supported App SDK or Gateway client contract with clear Node/browser/RN transport and device-identity storage adapters, generated Gateway protocol coverage, docs, parity tests, and a migration path for CLI and Control UI clients.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution, but the reusable Gateway client still needs maintainer decisions on publication, generated protocol coverage, runtime adapters, and CLI/Control UI migration.
