## Summary

Keep open. Current main and the latest release have meaningful App SDK progress, but they still do not provide the requested public, platform-agnostic low-level Gateway WebSocket client; the existing package is private, Node-built, wraps the internal Node GatewayClient, and Control UI still keeps a separate browser client. The Brani5 reference client is useful workaround context, and https://github.com/openclaw/openclaw/issues/74704 is related SDK stabilization work, but neither supersedes this broader SDK/API decision.

## What This Changes

Request to extract a reusable Gateway WebSocket client SDK package with platform adapters, typed protocol coverage, and eventual CLI/Control UI reuse.

## Best Possible Solution

Define and ship a supported App SDK/Gateway client contract with generated low-level protocol coverage, adapterized WebSocket and device-identity storage boundaries, public docs, parity tests, and a staged CLI/Control UI migration plan.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source, docs, release, and test inspection give high confidence that current main only partially implements the requested public, platform-agnostic Gateway client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is useful partial progress, but the best final shape needs maintainer decisions on package naming, generated low-level API coverage, adapter support, auth/storage boundaries, and migration scope.
