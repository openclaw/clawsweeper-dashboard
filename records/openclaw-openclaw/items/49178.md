## Summary

Keep open. Current main and v2026.5.22 include useful `@openclaw/sdk` progress, but they still do not provide the requested public, platform-agnostic low-level Gateway WebSocket client: the package is private and Node-built, the SDK transport wraps the internal Node `GatewayClient`, and Control UI still maintains a separate browser client.

## What This Changes

The issue requests extracting a reusable platform-agnostic Gateway WebSocket client SDK with shared handshake, auth, request/response, events, reconnect, and adapter boundaries for CLI, Control UI, and third-party clients.

## Best Possible Solution

Define and ship a supported App SDK/Gateway client contract with generated low-level protocol coverage, adapterized WebSocket and device-identity storage boundaries, public docs, parity tests, and a staged CLI/Control UI migration plan.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source, docs, release, and related-item inspection give high confidence that current main only partially implements the requested public, platform-agnostic Gateway client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is useful partial progress, but the best final shape needs maintainer decisions on package naming, generated low-level API coverage, adapter support, auth/storage boundaries, and migration scope.
