## Summary

Keep open. Current main and the latest release include meaningful `@openclaw/sdk` progress, but they still do not satisfy the requested reusable, platform-agnostic Gateway WebSocket client SDK because the package remains private and Node-built, its default transport wraps the internal Node Gateway client, and Control UI still uses its own browser client. Brani5's linked reference implementation is useful workaround context, not a core substitute.

## What This Changes

Request a reusable, platform-agnostic Gateway WebSocket client SDK package with transport and device-identity adapters plus shared protocol handling for CLI, Control UI, and third-party apps.

## Best Possible Solution

Define and ship a supported App SDK or Gateway client contract with clear Node/browser/RN transport and device-identity storage adapters, generated Gateway protocol coverage, docs, parity tests, and a migration path for CLI and Control UI clients.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution, but the reusable Gateway client still needs maintainer decisions on package naming, publication, generated protocol coverage, and runtime adapter support.
