## Summary

Keep open: current main and v2026.6.10 have useful internal gateway-client and SDK groundwork, but the requested public, platform-agnostic Gateway WebSocket SDK is still private, Node-oriented, and not the Control UI browser client path. This remains a maintainer API/security product decision, with external demand and an existing third-party reference implementation already linked in the discussion.

## What This Changes

The issue requests a reusable, platform-agnostic `@openclaw/gateway-client` SDK package for external Gateway WebSocket clients and eventual CLI/UI convergence.

## Best Possible Solution

A maintainer should sponsor a versioned, documented, security-reviewed public Gateway SDK with clear Node/browser/mobile adapter and storage boundaries, or explicitly keep the package private and continue directing external apps to the documented Gateway protocol/RPC surfaces.
