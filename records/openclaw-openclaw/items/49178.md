## Summary

Keep open. Current main has useful `@openclaw/sdk` progress, but it still does not satisfy the requested reusable, platform-agnostic Gateway WebSocket client SDK: the package is private and Node-built, its transport wraps the internal Node Gateway client, and Control UI still has a separate browser client implementation.

## What This Changes

The issue requests a public reusable Gateway WebSocket client SDK package with shared protocol/auth/reconnect/event handling for CLI, Control UI, and third-party apps.

## Best Possible Solution

Define and ship a supported App SDK or gateway-client contract with clear Node/browser/RN boundaries, auth and device-storage adapters, generated low-level Gateway method coverage, docs, and parity tests for existing Node and Control UI clients.

## Reproduction Assessment

Not applicable; this is a feature/API request, not a broken existing behavior report. Source inspection gives high confidence that current main is only a partial implementation of the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution for Node/TypeScript consumers, but the remaining publication, adapter, generated-client, and Control UI migration decisions need maintainer product/API judgment.
