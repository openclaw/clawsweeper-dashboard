## Summary

Keep open. Current main has meaningful `@openclaw/sdk` progress, but it still does not satisfy the requested public, platform-agnostic Gateway WebSocket client SDK because the package is private and Node-built, its default transport wraps the internal Node Gateway client, and Control UI still maintains a separate browser client.

## What This Changes

Request to add a reusable public Gateway WebSocket client SDK with shared handshake, request/response, events, reconnect, device auth, and runtime adapter support.

## Best Possible Solution

Define and ship a supported App SDK or gateway-client contract with clear Node/browser/RN adapter boundaries, generated Gateway protocol coverage, docs, parity tests, and a migration path for CLI and Control UI clients.

## Reproduction Assessment

Not applicable; this is a feature/API request, not a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution, but the reusable Gateway client still needs maintainer decisions on naming, publication, generated protocol coverage, and runtime adapter support.
