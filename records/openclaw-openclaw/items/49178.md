## Summary

Keep open. Current `main` has meaningful App SDK progress, but it still does not satisfy the requested reusable, public, platform-agnostic Gateway WebSocket client: the SDK package is private and Node-built, its default transport wraps the internal Node `GatewayClient`, and Control UI still maintains a separate browser client. The linked Brani5 implementation is useful workaround context, and https://github.com/openclaw/openclaw/issues/74704 is related SDK stabilization work, but neither replaces this broader SDK/API decision.

## What This Changes

Extract a public, reusable, platform-agnostic Gateway WebSocket client SDK package for CLI, Control UI, and third-party app consumers.

## Best Possible Solution

Define and ship a supported `@openclaw/sdk` or `@openclaw/gateway-client` contract with generated Gateway method/event coverage, adapterized transports and device storage, public docs, parity tests, and a deliberate CLI/Control UI migration plan.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic Gateway client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial path, but the best final shape needs maintainer decisions on public package naming, generated low-level API, adapter support, and migration scope.
