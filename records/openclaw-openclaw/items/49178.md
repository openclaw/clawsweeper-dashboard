## Summary

Keep open. Current main and v2026.5.22 include substantial `@openclaw/sdk` work, but they still do not provide the requested public, platform-agnostic low-level Gateway WebSocket client package; the existing SDK package is private, Node-backed, and Control UI still carries its own browser Gateway client implementation. Brani5's external reference implementation is useful context, but it does not replace a supported OpenClaw contract.

## What This Changes

The issue requests a reusable Gateway WebSocket client SDK package for third-party consumers, with platform adapters, protocol/auth handling, events, reconnects, and eventual CLI/Control UI reuse.

## Best Possible Solution

Have maintainers choose the public SDK boundary, likely by making `@openclaw/sdk` publishable with a generated low-level Gateway client and platform adapters or explicitly approving a separate Gateway client package, then migrate CLI/UI only after that contract is stable.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a reported broken behavior. Source inspection confirms current main lacks the requested publishable, platform-agnostic low-level client because `@openclaw/sdk` is private and Node-backed while Control UI keeps its own browser client.

## Solution Assessment

No, the exact proposed `@openclaw/gateway-client` package is not yet proven to be the best shape. The safer path is maintainer API design first, deciding whether the low-level client belongs inside publishable `@openclaw/sdk` or in a separate package.
