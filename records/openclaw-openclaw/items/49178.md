## Summary

Keep open. Current main and v2026.5.22 include substantial `@openclaw/sdk` work, tests, and docs, but they still do not provide the requested public, platform-agnostic low-level Gateway WebSocket client package; the current package is private, Node-built, wraps the internal Gateway client, and Control UI still carries its own browser client implementation.

## What This Changes

Requested extraction of a reusable, public Gateway WebSocket client SDK package for external OpenClaw apps, tooling, dashboards, and mobile/browser consumers.

## Best Possible Solution

Keep the current App SDK direction, then make an explicit maintainer decision on whether the low-level generated Gateway client lives inside `@openclaw/sdk` or a separate `@openclaw/gateway-client`, with clear adapter, auth/storage, docs, tests, and migration boundaries.

## Reproduction Assessment

Not applicable; this is a feature/API design request rather than a bug with a failing runtime reproduction path. Source inspection shows the current SDK is only a partial overlap, not a full reproduction target.

## Solution Assessment

No, not as an autonomous cleanup fix: the requested package crosses public API naming, generated-client strategy, auth/storage, browser/mobile adapters, and migration decisions. The maintainable path is a maintainer-owned SDK contract decision followed by focused implementation PRs.
