## Summary

Keep open. Current main has meaningful App SDK progress, but it still does not fully satisfy the requested reusable gateway WebSocket client SDK: the package is private, the provided transport wraps the internal Node GatewayClient, no generated low-level protocol client is present, and Control UI still maintains its own browser WebSocket client. The discussion also points to real third-party demand and an external reference implementation, so this remains a valid maintainer API/product-scope item rather than conservative cleanup.

## What This Changes

The issue requests a reusable, platform-agnostic gateway WebSocket client SDK package for OpenClaw gateway consumers.

## Best Possible Solution

Turn the existing App SDK work into an explicit supported contract: either publish and document @openclaw/sdk as the gateway-client surface, or split a lower-level adapterized Gateway client from it; keep protocol schema/types in lockstep, define Node/browser/RN support boundaries, and add parity coverage for the Node GatewayClient and Control UI browser client.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is source and docs inspection: the SDK exists, but the requested public, platform-agnostic reusable gateway client surface is only partially present.

## Solution Assessment

Unclear. The current @openclaw/sdk direction may be the right product shape, but closing this as solved would skip the unresolved publication, generated low-level client, platform adapter, and Control UI drift concerns raised in the request.
