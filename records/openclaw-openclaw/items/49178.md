## Summary

Keep open. Current main has meaningful `@openclaw/sdk` progress, but it still does not fully satisfy the requested public, platform-agnostic reusable Gateway WebSocket client package: the package is private/node-built, the transport wraps the internal Node Gateway client, and Control UI still maintains its own browser client path.

## What This Changes

Request a reusable public Gateway WebSocket client SDK package for third-party apps and eventual CLI/Control UI protocol sharing.

## Best Possible Solution

Define and ship an explicit supported App SDK or gateway-client contract with publication naming, Node/browser/RN support boundaries, generated low-level Gateway client strategy, auth/device storage adapters, docs, and parity coverage for existing Node and Control UI clients.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection verifies current main has partial SDK work, but not the requested platform-agnostic public Gateway client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution for Node/TypeScript consumers, but closing this as solved would skip the package publication/name, low-level generated client, adapter, and Control UI drift decisions raised by the request.
