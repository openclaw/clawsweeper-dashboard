## Summary

Keep open. Current main has meaningful App SDK progress, including WebSocket e2e coverage, but it still does not provide the requested reusable public platform-agnostic Gateway WebSocket client: the package remains private and Node-built, its transport wraps the internal Node GatewayClient, and Control UI still owns a separate browser client. The Brani5 reference client is useful workaround context, and https://github.com/openclaw/openclaw/issues/74704 is related SDK stabilization work, but neither replaces this broader SDK/API decision.

## What This Changes

Request to extract a reusable public Gateway WebSocket client SDK with platform-neutral transports, storage adapters, typed protocol support, reconnect/request/event handling, and eventual CLI/Control UI migration.

## Best Possible Solution

Define and ship a supported App SDK/Gateway client contract with generated Gateway method/event coverage, adapterized transport and device storage boundaries, public docs, parity tests, and an explicit CLI/Control UI migration plan.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic Gateway client contract.

## Solution Assessment

Unclear. The current @openclaw/sdk direction is a strong partial path, but the best final shape needs maintainer decisions on package naming, generated low-level API, adapter support, auth/storage boundaries, and migration scope.
