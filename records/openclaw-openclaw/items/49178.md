## Summary

Keep open. Current main has useful `@openclaw/sdk` progress, but it still does not satisfy the requested public, platform-agnostic Gateway WebSocket client SDK because the package remains private and Node-built, its default transport wraps the internal Node Gateway client, and Control UI still owns a separate browser client implementation.

## What This Changes

Request to extract a reusable, platform-agnostic WebSocket Gateway client SDK package for third-party consumers and reduce CLI/Control UI protocol duplication over time.

## Best Possible Solution

Define and ship a supported App SDK or gateway-client contract with clear Node/browser/RN adapter boundaries, generated low-level Gateway coverage, docs, and parity tests before migrating CLI or Control UI clients.

## Reproduction Assessment

Not applicable; this is a feature/API request, not a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial solution, but the requested reusable Gateway client needs maintainer decisions on package naming, publication, generated protocol coverage, and runtime adapter support.
