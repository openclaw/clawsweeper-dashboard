## Summary

Keep open. Current main has meaningful App SDK progress, but it still does not satisfy the requested reusable, platform-agnostic Gateway WebSocket client SDK because the SDK package is private and Node-built, the default transport wraps the internal Node Gateway client, and Control UI still maintains its own browser client; Brani5's linked reference implementation is useful workaround context rather than an OpenClaw core substitute.

## What This Changes

The issue requests a reusable public Gateway WebSocket client SDK package for OpenClaw gateway consumers, with shared protocol, auth, request/event, reconnect, and adapter support.

## Best Possible Solution

Define and ship a supported `@openclaw/sdk` or `@openclaw/gateway-client` contract with adapterized transports and storage, generated Gateway method/event coverage, docs, parity tests, and a migration plan for CLI and Control UI consumers.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that current main only partially implements the requested public, platform-agnostic client contract.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong partial path, but the remaining Gateway client scope needs maintainer decisions on package naming, publication, generated protocol coverage, and adapter support.
