## Summary

Keep open: current main still drops WhatsApp final error payloads before delivery, and the linked open PR remains the narrow implementation path for this bug.

## What This Changes

The issue reports WhatsApp sessions silently losing the user-visible fallback reply when a long model call ends in an incomplete turn with no payload delivered.

## Best Possible Solution

Land a focused WhatsApp delivery fix that lets sanitized final isError fallback payloads reach the user while keeping reasoning, compaction notices, and non-final tool/block errors suppressed.

## Reproduction Assessment

Yes for the delivery-loss path from source inspection: current main creates a final isError fallback payload, then WhatsApp filters it in both inbound dispatch and outbound sendPayload. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to open the final user-facing isError path at both WhatsApp delivery gates while preserving non-final error suppression. The broader observability suggestions should be evaluated after the delivery bug is fixed.
