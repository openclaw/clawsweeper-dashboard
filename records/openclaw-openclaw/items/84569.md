## Summary

Keep open: current main still has the WhatsApp delivery-loss path, and the issue is paired with an open closing PR that should be reviewed or rejected before this issue is closed.

## What This Changes

WhatsApp should deliver a visible fallback reply when a long or incomplete agent turn produces a final error payload instead of leaving the user with silence.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578; the desired end state is for WhatsApp to deliver sanitized final error payloads while continuing to suppress reasoning, compaction notices, and non-final tool noise.

## Reproduction Assessment

Yes for the delivery-loss path from source inspection: current main creates an `isError` fallback for incomplete turns, then WhatsApp filters it in both inbound dispatch and outbound sendPayload. I did not run a live long-running WhatsApp session.

## Solution Assessment

Yes for the narrow delivery bug: the current linked PR changes the two implicated WhatsApp gates while preserving suppression for non-final/tool noise. Broader proactive session rotation or timeout policy should be handled as follow-up only if it remains after final error delivery is fixed.
