## What This Changes

The issue reports that long WhatsApp-originated model calls can end in an incomplete turn whose fallback/error reply is not delivered to the user.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578; after that, split any remaining proactive session recovery or observability work into a narrower follow-up such as https://github.com/openclaw/openclaw/issues/85532 if production still shows silence.

## Reproduction Assessment

Yes for the delivery-loss path: current main creates an incomplete-turn isError payload and then both WhatsApp delivery gates suppress it. I did not rerun a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to allow sanitized final user-facing error payloads through WhatsApp delivery while preserving suppression for reasoning, compaction notices, and non-final tool noise. The open linked PR is the branch to review before closing this issue.
