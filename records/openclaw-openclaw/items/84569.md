## Summary

Keep open: current main still has the WhatsApp delivery suppression path for final `isError` fallback payloads, and an open linked PR with closing syntax already targets that narrow delivery bug.

## What This Changes

Request that long WhatsApp agent turns and incomplete-turn fallbacks produce a visible channel reply instead of leaving the user with silence.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578; after the delivery path is fixed, split any remaining proactive session recovery or observability work into a narrower follow-up if production still shows silence.

## Reproduction Assessment

Yes for the delivery-loss path: current main creates an incomplete-turn `isError` payload and then the WhatsApp dispatcher and outbound adapter suppress that payload before it can reach the user. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to allow sanitized final user-facing error payloads through WhatsApp delivery while preserving suppression for reasoning, compaction notices, and non-final tool noise. The open linked PR is the implementation path to review before closing this issue.
