## Summary

Keep open: current main still suppresses the WhatsApp final error payload path, and the closing-linked fix PR is open, so this issue should stay open until that PR lands or is rejected.

## What This Changes

The issue requests that WhatsApp deliver a user-visible fallback when a long or stalled model call ends in an incomplete turn instead of leaving the user with no reply.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84578; the desired end state is for WhatsApp to deliver sanitized final incomplete-turn error payloads while preserving suppression for reasoning, compaction notices, and non-final tool noise.

## Reproduction Assessment

Yes for the delivery-loss path from source inspection: the runner emits an isError fallback for incomplete turns, and WhatsApp filters that payload at both inbound-dispatch and outbound sendPayload gates. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to allow final user-facing WhatsApp isError payloads through while preserving suppression for reasoning, compaction notices, and non-final tool/block noise; the linked open PR is the current implementation path.
