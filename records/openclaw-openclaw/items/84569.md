## Summary

Keep open: current main still filters WhatsApp `isError` reply payloads before delivery, even though the embedded runner returns a user-facing error payload for incomplete turns. This is a valid message-loss bug with a narrow fix path.

## What This Changes

A production WhatsApp DM report asks for long model calls and incomplete turns to produce a visible fallback reply instead of silently losing the response.

## Best Possible Solution

Align WhatsApp with the shared reply pipeline by delivering sanitized final error payloads for incomplete turns, while continuing to hide reasoning, compaction notices, and non-final tool noise.

## Reproduction Assessment

Yes for the delivery drop from source inspection: current main produces `isError` fallback payloads for incomplete turns and then WhatsApp filters those payloads before send. I did not run a live WhatsApp reproduction.

## Solution Assessment

Yes, the best fix is narrow: preserve sanitized final `isError` payload delivery in WhatsApp inbound and outbound paths, with regression tests. The broader long-call timeout/recovery behavior should be rechecked after the visible fallback is no longer suppressed.
