## Label Justifications

- `P1`: A real WhatsApp channel workflow can silently lose the user-visible fallback reply after a stalled or incomplete agent turn.
- `impact:message-loss`: The reported failure is a generated fallback/error reply never reaching the WhatsApp user.
- `impact:session-state`: The trigger involves queued work behind a long active model call and stalled session recovery state.

## Summary

Keep open: current main and the latest shipped release still suppress WhatsApp final error payloads before delivery, and the linked open fix path should be reviewed or rejected before this issue closes.

## What This Changes

The issue asks for WhatsApp users to receive a visible fallback/error reply when a long model call stalls or completes as an incomplete turn instead of silently losing the response.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84578; the end state should deliver sanitized final WhatsApp error payloads while preserving suppression for reasoning, compaction notices, and non-final tool noise.

## Reproduction Assessment

Yes from source for the delivery loss: current main creates an isError fallback payload for incomplete turns, then WhatsApp filters that payload in both inbound dispatch and sendPayload. I did not run a live WhatsApp long-call reproduction.

## Solution Assessment

Yes for the narrow delivery bug: the best path is to allow final user-facing isError payloads through the WhatsApp delivery gates while preserving non-final suppression. Broader still-working or timeout UX should be rechecked after the linked delivery fix is resolved.
