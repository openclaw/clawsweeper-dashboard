## Summary

Keep open: current main still creates a user-facing incomplete-turn error payload, but the WhatsApp dispatcher and outbound adapter both suppress `isError` payloads before they can reach the user. The linked open pull request https://github.com/openclaw/openclaw/pull/84578 uses closing syntax for this issue, so the issue should stay open until that fix or an equivalent lands.

## What This Changes

The issue reports that long WhatsApp-originated model calls can end in an incomplete turn whose fallback/error reply is silently dropped instead of being delivered to the user.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578; if silence remains after the delivery fix, track proactive session-recovery work separately, such as the related https://github.com/openclaw/openclaw/issues/85532 area.

## Reproduction Assessment

Yes for a source-level reproduction: current main creates an incomplete-turn `isError` payload, then WhatsApp drops that payload in both the inbound dispatcher and outbound `sendPayload` path. I did not run a live long-running WhatsApp session in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to allow sanitized final user-facing error payloads through the two WhatsApp delivery gates while keeping non-final tool/block noise suppressed. The open linked pull request is the implementation path to review before closing this issue.
