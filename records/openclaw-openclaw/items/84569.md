## What This Changes

The issue reports WhatsApp users receiving no visible fallback when a long agent turn stalls or produces an incomplete turn with no deliverable payloads.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578 first; if production still shows silence afterward, split proactive session recovery or observability into a focused follow-up.

## Reproduction Assessment

Yes for the delivery-loss path: current main creates an incomplete-turn isError payload and then WhatsApp suppresses it in both the inbound dispatcher and outbound sendPayload path. I did not run a live stalled WhatsApp session in this read-only review.

## Solution Assessment

Yes, the best immediate solution is the narrow linked WhatsApp delivery fix that lets sanitized final error payloads reach users while keeping reasoning, compaction, and non-final tool noise suppressed. Broader timeout recovery or session-rotation ideas should be follow-up work if the delivery fix does not cover the remaining production symptom.
