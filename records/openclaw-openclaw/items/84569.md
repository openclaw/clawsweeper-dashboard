## Summary

Keep open: current main still suppresses the user-facing WhatsApp error payload generated for incomplete turns, and the open linked fix PR remains the narrow implementation path.

## What This Changes

The issue reports WhatsApp losing fallback/error replies when a long model call stalls and an incomplete turn produces a user-facing error payload.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84578; the desired end state is for WhatsApp to deliver sanitized final user-facing isError payloads while still filtering reasoning, compaction, and non-final tool noise.

## Reproduction Assessment

Yes for the delivery loss from source inspection: current main produces an isError fallback payload for incomplete turns, then WhatsApp filters that payload in both the inbound dispatcher and outbound sendPayload path. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes for the narrow delivery bug: allow final user-facing isError payloads through WhatsApp delivery while preserving filters for reasoning, compaction notices, and non-final tool noise. Broader delivery receipts or heartbeat metadata should be handled separately if maintainers want them.
