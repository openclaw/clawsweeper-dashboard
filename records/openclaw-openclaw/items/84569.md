## Summary

Keep open: current main still has a source-reproducible WhatsApp delivery gap for incomplete-turn final error replies, and the linked open pull request with closing syntax is the active implementation path to review before this issue closes.

## What This Changes

The issue reports that WhatsApp users receive no fallback reply when a long model call stalls and the embedded runner ends an incomplete turn with no delivered payload.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84578; the desired end state is for WhatsApp to deliver sanitized final incomplete-turn fallback replies while continuing to suppress non-final tool, reasoning, and internal error noise.

## Reproduction Assessment

Yes for the delivery loss from source inspection: current main creates an isError fallback payload for incomplete turns, and WhatsApp filters isError payloads before both inbound dispatch delivery and outbound sendPayload delivery. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes, the best narrow solution is to let final user-facing isError fallback text through WhatsApp delivery while preserving filters for non-final tool, reasoning, and internal noise; the linked open PR is the active path to review.
