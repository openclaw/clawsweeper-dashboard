## Label Justifications

- `P1`: A real WhatsApp channel workflow can silently lose the user-visible fallback reply after a stalled or incomplete agent turn.
- `impact:message-loss`: The issue is about an internally produced fallback/error reply never reaching the WhatsApp user.
- `impact:session-state`: The trigger involves queued session work behind long active model calls and stalled-session recovery state.

## Summary

Keep open: current main and the latest release still suppress the final WhatsApp `isError` fallback path, and an open closing PR already targets the narrow delivery bug while a maintainer-owned issue tracks the broader cross-channel contract.

## What This Changes

The issue asks OpenClaw to avoid silent WhatsApp loss when long or incomplete model turns produce a final fallback/error reply.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578 for the WhatsApp final-error delivery path, then keep cross-channel fallback semantics in https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes for the delivery-loss path: current main creates a final `isError` payload for incomplete turns, and the WhatsApp dispatcher plus outbound adapter suppress that payload before it can reach the user. I did not run a live long-running WhatsApp session in this read-only review.

## Solution Assessment

Yes for the current issue path: the narrow maintainable fix is to deliver sanitized final error payloads through WhatsApp while preserving suppression for reasoning, compaction notices, and non-final tool noise. The open linked PR is the right immediate implementation candidate before this issue closes.
