## Label Justifications

- `P1`: A real WhatsApp workflow can silently lose the user-visible fallback reply after a stalled or incomplete agent turn.
- `impact:message-loss`: The reported and source-proven failure is that the generated fallback/error reply is never delivered to WhatsApp.
- `impact:session-state`: The issue is triggered by queued session work behind a long active model call and stalled agent-run state.

## Summary

Keep open: current main still produces a user-facing incomplete-turn error payload and then suppresses that `isError` payload on both WhatsApp auto-reply and routed outbound paths, which matches the reported silent message-loss failure.

## What This Changes

A WhatsApp direct session that stalls during a long model call should deliver the incomplete-turn fallback/error reply instead of dropping it silently.

## Best Possible Solution

Align WhatsApp with the shared reply pipeline by delivering sanitized final incomplete-turn error payloads while continuing to suppress reasoning, compaction notices, non-final tool noise, and raw internal error details.

## Reproduction Assessment

Yes for the delivery drop from source inspection: current main creates a final `isError` fallback for incomplete turns and then WhatsApp filters `isError` payloads before send. I did not run a live WhatsApp reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve sanitized final `isError` delivery in WhatsApp inbound and outbound paths with regression tests. Broader long-call timeout or configurable error-policy behavior should remain separate from this bug fix.
