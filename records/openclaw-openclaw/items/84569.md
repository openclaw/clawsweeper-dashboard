## Summary

Keep open: current main still has the WhatsApp delivery suppression path reported here, and https://github.com/openclaw/openclaw/pull/84578 is an open linked implementation candidate with closing syntax rather than a reason to close this issue early.

## What This Changes

The issue reports that a long WhatsApp session can end in an incomplete turn whose user-facing fallback/error payload is not delivered to the originating chat.

## Best Possible Solution

WhatsApp should deliver sanitized final user-facing error payloads from incomplete turns while continuing to suppress reasoning, compaction notices, tool noise, and non-final internal errors.

## Reproduction Assessment

Yes for the delivery-loss path: current main emits an isError fallback for incomplete turns and the WhatsApp delivery paths filter isError payloads before sending. I did not establish a live reproduction of the long-running stalled-session sequence.

## Solution Assessment

Yes: the narrow maintainable fix is to allow final user-facing error payloads through WhatsApp delivery while preserving suppression for non-final tool and internal noise. The later adapter-level comment is relevant for truly empty payloads, but the linked PR is the current concrete path for the proven dropped isError payload.
