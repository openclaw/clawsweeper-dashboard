## Summary

Keep open: current main and the latest release still suppress the WhatsApp final `isError` fallback path, and the open linked PR with closing syntax is still the right near-term implementation candidate.

## What This Changes

The issue reports that long WhatsApp-originated agent turns can stall and end without a delivered user-visible fallback/error reply.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84578 for the narrow WhatsApp final-error delivery bug, while keeping the cross-channel durable fallback contract in https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes for the narrow delivery-loss path: current main returns a final `isError` payload from the embedded runner, then WhatsApp suppresses it in both dispatcher and outbound adapter code. I did not rerun a live WhatsApp long-model-call stall in this read-only review.

## Solution Assessment

Yes for the immediate bug: the narrow maintainable fix is to let sanitized final user-facing error payloads through WhatsApp delivery while preserving suppression for reasoning, compaction notices, and non-final tool noise. Broader timeout/recovery observability should stay in the existing cross-channel follow-up.
