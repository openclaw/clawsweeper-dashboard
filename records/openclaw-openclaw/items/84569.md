## Summary

Keep open: current main still drops WhatsApp final error payloads before delivery, and an open linked fix PR already targets the narrow delivery bug, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks for WhatsApp to surface user-visible fallback/error replies when long model calls produce incomplete turns instead of silently dropping them.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/84578; the desired end state is for WhatsApp to deliver sanitized final isError payloads while preserving filters for reasoning, compaction notices, and non-final tool noise.

## Reproduction Assessment

Yes for the delivery loss from source inspection: current main produces an isError fallback payload for incomplete turns, then WhatsApp filters that payload in both the inbound dispatcher and outbound sendPayload path. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes, the best narrow fix is to allow final user-facing isError payloads through WhatsApp delivery while continuing to suppress reasoning, compaction, and non-final tool noise. The open linked PR is the current implementation path to review.
