## Label Justifications

- `P1`: A real WhatsApp channel workflow can silently lose the user-visible fallback reply after a stalled or incomplete agent turn.
- `impact:message-loss`: The issue is about a generated fallback or error reply never reaching the WhatsApp user.
- `impact:session-state`: The trigger involves queued session work behind a long active model call and stalled session recovery state.

## Summary

Keep open: current main still suppresses final WhatsApp `isError` fallback payloads before delivery and again in `sendPayload`, and an open linked PR with closing syntax targets that source-proven delivery bug.

## What This Changes

The issue asks OpenClaw to deliver a WhatsApp-visible fallback or error reply when long or incomplete agent turns would otherwise end with no payload delivered to the user.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84578, then split any remaining proactive session recovery or observability work into a narrower follow-up if production still shows silence.

## Reproduction Assessment

Yes for the delivery-loss path: current main creates an incomplete-turn `isError` payload and then the WhatsApp dispatcher and outbound adapter suppress that payload before it can reach the user. I did not run a live WhatsApp stalled-session reproduction.

## Solution Assessment

Yes for the narrow delivery bug: the maintainable fix is to allow sanitized final user-facing `isError` payloads through WhatsApp delivery while preserving suppression for reasoning, compaction notices, and non-final tool noise. Broader proactive recovery should be handled separately if it remains after the delivery fix.
