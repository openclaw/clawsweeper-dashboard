## Summary

Keep open. The stuck ACP lane problem is real and current main still has no session-lane deadline, but the latest PR head still introduces blocking lifecycle races: timed-out prompt turns can skip terminal bookkeeping, and timed-out idle closes can leave a cached handle reusable while close is still pending. https://github.com/openclaw/openclaw/pull/52747

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wraps ACP session actor work with a timeout and stale-operation fence, updates config schema/help/generated metadata, and adds manager tests for hung initialize and idle-eviction cases.

## Best Possible Solution

Keep the lane-deadline feature direction, but revise the implementation so timed-out operations have explicit terminal bookkeeping, same-session work cannot reuse handles in a closing/timed-out lifecycle state, and cancellation or stale-operation fencing is applied only after cleanup invariants are preserved.

## Reproduction Assessment

Yes. A mocked ACP runtime whose `runTurn()` or `close()` remains pending past `acp.sessionLane.taskTimeoutMs` is enough to exercise the two remaining races: prompt-turn bookkeeping is skipped by the stale guard, and idle close leaves the cached handle visible while the queue advances.

## Solution Assessment

No. The latest stale-fence approach is closer than the original patch, but it still releases the queue before preserving session/task state and without a closing state for cached runtimes; the safer solution needs lifecycle accounting before stale suppression or same-session reuse.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve timeout bookkeeping before marking turns stale:** `src/acp/control-plane/manager.core.ts:923-925`
  - body: When the lane timeout fires, `onTimeout` marks the actor operation stale. A prompt turn has already created a background task and set the session to `running`, but this catch returns before `markBackgroundTaskTerminal()` and the error-state write, so a timed-out prompt can remain visible as running after the caller receives the timeout.
  - confidence: 0.91
- **[P1] Prevent reuse while idle close is pending:** `src/acp/control-plane/manager.core.ts:1927-1935`
  - body: Idle eviction now keeps the cache entry until `runtime.close()` returns, but the timeout on this queued close lets later same-session work run while that close is still pending. Add a closing state or otherwise hide/block the cached handle until close completes or fails, otherwise the next turn can reuse a runtime that is being closed.
  - confidence: 0.88
