## Summary

Keep open. The stuck ACP lane problem is valid and current main still has no session-lane deadline, but the latest PR head still releases the lane before all runtime lifecycle and durable-state cleanup is safe, so it needs maintainer review before merge.

## What This Changes

The PR adds configurable ACP session-lane task timeouts, wires the new config through schema/help surfaces, and adds ACP manager regression tests for hung lane behavior.

## Best Possible Solution

Revise the session-lane timeout design so timed-out turns still record terminal state, stale operations cannot mutate or erase newer state, idle-close handles are quarantined or kept lane-held until retired, and the new config ships with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows runtime.ensureSession can run inside withSessionActor without a cancellation signal, and KeyedAsyncQueue only advances same-key work after the current promise settles; I did not run tests because this was a read-only review and pnpm is unavailable.

## Solution Assessment

No. A lane deadline is the right direction, but this patch releases the queue before terminal bookkeeping and runtime-cache lifecycle are safe; the narrower maintainable design needs cancellation, fencing, and handle quarantine/accounting.

  - body: When the lane timeout marks the operation stale, a prompt turn that already created a background task and set the session to `running` can return here before `recordTurnCompletion()`, `markBackgroundTaskTerminal()`, and the error-state write run. If close exceeds that timeout, the queue releases while the cache entry remains visible until close resolves, so later same-session work can reuse a handle that is already terminating; keep the handle quarantined or hold the lane until retirement is confirmed.
  - confidence: 0.87
