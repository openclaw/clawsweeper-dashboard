## Summary

Keep open. Current main still has the ACP session-lane liveness problem and this PR has useful real-behavior proof, but the current patch still drops counted closing-runtime accounting when a stale close later rejects, which can undermine `maxConcurrentSessions`.

## What This Changes

The branch adds configurable ACP session-lane task timeouts with cooperative abort/stale fencing, runtime-cache closing accounting, ACPX signal passthrough, config/docs/changelog wiring, and regression tests.

## Best Possible Solution

Keep the session-lane timeout direction, but preserve closing runtime accounting until close succeeds or a confirmed termination/retry path retires it, with regression coverage for late close rejection.

## Reproduction Assessment

Yes. Current main has a clear source path where same-session ACP work is serialized through `KeyedAsyncQueue` and `initializeSession()` can await a non-settling `ensureSession()` without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, cooperative abort, and stale-fencing approach is close, but the rejected stale-close path should not release runtime accounting until the old runtime is confirmed closed or handled by a retry/termination path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep rejected stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When `runtime.close()` rejects after the lane timeout has marked this operation stale, this branch clears the closing cache entry even though the failed close is not proof that the runtime is gone. Since `RuntimeCache.size()` now includes closing entries for `maxConcurrentSessions`, a timed-out close can leak an ACP runtime while allowing more sessions; keep the closing entry counted until close succeeds or a confirmed termination/retry path retires it.
  - confidence: 0.88
