## Summary

Keep this PR open, but it still needs changes before merge: the core ACP lane-timeout fix is useful and current main still has the stuck-lane failure mode, yet the current head still drops runtime accounting when a timed-out close later rejects, and the real behavior proof predates the latest head.

## What This Changes

Adds configurable ACP session-lane task timeouts, cooperative abort signals, stale-operation fencing, runtime-cache closing accounting, ACPX/config/docs wiring, and ACP manager regression tests.

## Best Possible Solution

Keep the session-lane timeout design, but preserve runtime accounting on rejected timed-out closes and add focused regression coverage before merge.

## Reproduction Assessment

Yes. Current main's same-key queue waits on the previous promise and initializeSession() can await a non-settling runtime.ensureSession() without a lane deadline; the PR body also supplies terminal proof for the central stuck-before/unblocked-after path on an older head.

## Solution Assessment

No. The timeout, cooperative abort, and stale-fencing direction is appropriate, but the stale close rejection branch should not retire counted runtime accounting until close succeeds or termination/retry is confirmed.
