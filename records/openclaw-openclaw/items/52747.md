## Summary

Keep open: current main still has the ACP session-lane wedge the PR targets, but the PR still has a resource-accounting bug in the stale close path that should block merge.

## What This Changes

Adds configurable ACP session-lane task timeouts with cooperative abort/stale fencing, runtime-cache closing accounting, ACPX AbortSignal plumbing, config/docs/changelog updates, and ACP manager regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but retain closing runtime handles in accounting until close succeeds or a hard termination/retry path confirms retirement, with a regression covering aborted close rejection.

## Reproduction Assessment

Yes. Source inspection shows current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession inside that lane with no deadline, so a never-settling ensureSession can keep later same-session tasks queued.

## Solution Assessment

No. The timeout, cooperative abort, and stale-fencing direction is appropriate, but the current stale close error path can drop runtime accounting before close or process termination is confirmed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed closes counted after timeout:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When a close is aborted by the session-lane timeout and `runtime.close()` rejects before the runtime is confirmed retired, this stale branch deletes the counted closing entry. `RuntimeCache.size()` then stops counting a possibly still-live process/session, so `maxConcurrentSessions` can admit another runtime and leak resources; keep the entry counted until close/kill success or a retry path confirms retirement.
  - confidence: 0.88
