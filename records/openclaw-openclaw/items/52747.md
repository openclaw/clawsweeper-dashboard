## Summary

Keep open: the stuck ACP lane problem is valid on current main, but this PR still has a blocking runtime-close accounting defect that can drop `maxConcurrentSessions` protection before a timed-out runtime is confirmed closed.

## What This Changes

This PR adds configurable ACP session-lane task timeouts, AbortSignal plumbing through ACP manager/runtime/ACPX paths, stale-operation fencing, runtime-cache closing accounting, docs/schema/changelog updates, and ACP manager regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but retain closing runtime handles in counted accounting until close succeeds or a hard termination/retry state confirms retirement, then refresh current-head terminal proof.

## Reproduction Assessment

Yes. Source inspection of current main shows `initializeSession()` awaits `runtime.ensureSession()` inside the per-session `KeyedAsyncQueue`, so a never-settling backend load can hold the lane indefinitely.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is close, but the current stale close rejection path clears resource accounting before runtime retirement is confirmed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep aborted closes counted until they are confirmed closed:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When a lane timeout aborts `runtime.close()`, this stale catch branch clears the closing entry even though a rejected/aborted close does not prove the runtime process exited. That drops `RuntimeCache.size()` and can let new ACP sessions bypass `maxConcurrentSessions` while the old runtime is still alive; keep it counted or move it to a retry/terminated state until close is confirmed.
  - confidence: 0.88
