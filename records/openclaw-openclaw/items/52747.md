## Summary

This PR is still useful and addresses a real ACP control-plane liveness bug, but it should not merge yet because one stale close failure path can still drop runtime accounting without proof that the runtime was closed.

## What This Changes

The PR adds a configurable ACP session-lane task timeout with cooperative abort/stale fencing, runtime-cache closing accounting, ACPX signal passthrough, config/docs/changelog wiring, and regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but preserve closing runtime accounting until close succeeds or a confirmed retry/termination path retires the runtime, with focused regression coverage for late close rejection.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through `KeyedAsyncQueue` and `initializeSession()` can await a non-settling `ensureSession()` without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, abort, and stale-fencing approach is close, but the rejected stale-close path should keep runtime accounting until the old runtime is confirmed closed or handled by a retry/termination path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When `runtime.close()` rejects after the lane timeout has made this operation stale, this branch clears the counted closing entry even though the failed close is not proof that the runtime is gone. Since `RuntimeCache.size()` includes closing entries for `maxConcurrentSessions`, a timed-out close can leak an ACP runtime while allowing more sessions; keep the closing entry counted until close succeeds or a confirmed termination/retry path retires it.
  - confidence: 0.88
