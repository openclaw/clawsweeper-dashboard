## Summary

Keep this PR open. The timeout and stale-fencing direction is useful, but the latest head still clears a counted closing runtime entry when a stale `runtime.close()` later rejects, so a failed close can undercount live ACP runtime resources before merge.

## What This Changes

The branch adds a configurable ACP session-lane task timeout with cooperative runtime abort signals, stale-operation fencing, runtime-cache closing accounting, config/docs wiring, and ACP manager regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but preserve runtime cache accounting until close succeeds or a confirmed termination/retry path retires the runtime, with rejection-path regression coverage.

## Reproduction Assessment

Yes. Current main's same-key queue and `initializeSession()` source path can hold the lane forever on a non-settling `ensureSession`, and the PR body supplies terminal before/after proof for that path; the remaining close-accounting finding is source-reproducible from the stale rejection branch.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is appropriate, but clearing the counted closing entry on stale close rejection can leak or undercount runtime resources and should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P1] Keep rejected stale closes counted:** `src/acp/control-plane/manager.core.ts:2472`
