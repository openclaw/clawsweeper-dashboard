## Summary

Keep open: the PR addresses a real current-main ACP lane deadlock, but the current head still has a blocking resource-accounting bug when a timed-out close later rejects.

## What This Changes

The PR adds configurable ACP session-lane task timeouts with cooperative abort/stale fencing, runtime-cache closing accounting, acpx signal passthrough, config/docs/schema/changelog wiring, and regression tests.

## Best Possible Solution

Keep the timeout, cooperative abort, and stale-fencing direction, but preserve counted closing runtime state until close succeeds or a confirmed termination/retry path retires it, with rejection-path regression coverage.

## Reproduction Assessment

Yes. Current main has a clear source path where same-session ACP work is serialized through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The design is close, but clearing the counted closing entry after a stale close rejection is not safe because RuntimeCache.size() backs maxConcurrentSessions accounting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When runtime.close() rejects after the lane timeout has marked this operation stale, this branch clears the counted closing entry even though the runtime may still be alive. Since RuntimeCache.size() now backs maxConcurrentSessions, this can leak an ACP runtime while allowing new sessions; keep the closing entry counted until close succeeds or a confirmed termination/retry path retires it.
