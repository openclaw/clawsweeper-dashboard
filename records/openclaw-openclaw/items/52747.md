## Summary

Keep this PR open: current main still has the ACP session-lane wedge the PR targets, and the supplied terminal proof is relevant, but the current head is not merge-ready because a timed-out stale close rejection can still drop runtime accounting and the branch now conflicts with main.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, passes abort signals through ACP runtime/session operations, fences stale lane work, updates config/docs/changelog, and adds ACP manager regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but preserve runtime accounting until close succeeds or a confirmed termination/retry path retires the runtime, add rejection-path regression coverage, and then rebase alongside the complementary queued-abort work.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is appropriate, but clearing the counted closing entry after a stale close rejection can undercount live runtime resources and should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect
