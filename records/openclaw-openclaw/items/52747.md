## Summary

Keep open: this is a plausible source-reproducible ACP bug, but the latest external PR head is conflicting, lacks real behavior proof, and still has a blocking close-accounting issue in the timeout path.

## What This Changes

The PR adds a configurable ACP session-lane task timeout with stale-operation fencing, cooperative abort signals, runtime-cache closing accounting, schema/docs/changelog wiring, and regression tests.

## Best Possible Solution

Keep the ACP session-lane deadline direction, but keep timed-out or aborted closes counted until close succeeds or hard cleanup confirms termination, then require real ACP stuck-lane recovery proof before maintainer merge review.

## Reproduction Assessment

Yes from source inspection: current main serializes same-session ACP work through a promise tail, and initializeSession() awaits runtime.ensureSession() inside that lane without a deadline. I did not execute tests because this review is read-only.

## Solution Assessment

No. The PR improves the timeout design, but the stale close-rejection path can still drop runtime accounting before closure is confirmed; the branch also needs real behavior proof and conflict resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep timed-out close entries counted:** `src/acp/control-plane/manager.core.ts:2340`
  - body: When the lane timeout marks this operation stale and `runtime.close()` rejects, this branch clears the `closing` entry. That makes `RuntimeCache.size()` stop counting the old handle even though close did not complete, so `maxConcurrentSessions` can admit new sessions while the old runtime is still alive; keep the closing entry counted until close succeeds or a hard cleanup path confirms termination.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow or dependency supply-chain change was found, but the ACP close timeout path can still drop runtime resource accounting after a rejected close.

Concerns:

  - body: A lane timeout can abort `runtime.close()`; if it rejects while the actor operation is stale, the PR clears the counted closing entry without confirmed closure, which can undermine `maxConcurrentSessions` and leak ACP runtime resources.
