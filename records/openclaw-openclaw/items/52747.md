## Summary

Keep open: the PR addresses a real current-main ACP lane liveness bug and has sufficient terminal proof, but the latest head still has a resource-accounting bug in the timed-out close path that should block merge.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, propagates AbortSignals through ACP runtime operations, fences stale timed-out side effects, updates config/docs/changelog surfaces, and adds ACP manager regression tests.

## Best Possible Solution

Keep the timeout, abort, and stale-fencing design, but preserve counted closing runtime state until close succeeds or a confirmed hard termination/retry path retires it, with regression coverage for close rejection after a lane timeout.

## Reproduction Assessment

Yes. Source inspection on current main shows same-session ACP work is serialized through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession in that lane without a deadline; the PR body also supplies terminal proof of the stuck-before and unblocked-after behavior.

## Solution Assessment

No. The direction is close, but clearing the counted closing entry on stale close rejection is not the safest fix because maxConcurrentSessions uses RuntimeCache.size(); the narrower maintainable repair is to keep the old runtime accounted until closure is confirmed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
