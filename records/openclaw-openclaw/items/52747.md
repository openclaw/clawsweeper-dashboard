## Summary

Keep open: current main still has the ACP stuck-lane failure mode, but the latest PR remains blocked by missing real behavior proof and a concrete runtime-cache accounting bug in the close-timeout path.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, propagates AbortSignal and stale-operation fencing through ACP runtime paths, updates config/docs/schema/changelog, and adds ACP manager regression tests.

## Best Possible Solution

Keep the ACP lane-deadline design, but keep closing runtime handles counted until close is confirmed or hard termination succeeds, then require redacted real ACP stuck-session recovery proof before merge.

## Reproduction Assessment

Yes from source inspection: current main serializes same-session ACP work through a promise tail, and initializeSession awaits runtime.ensureSession inside that lane without a deadline. I did not run live tests in this read-only review.

## Solution Assessment

No. The direction is appropriate, but the latest diff still drops close accounting on stale close rejection and lacks real behavior proof from a real ACP setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted until confirmed closed:** `src/acp/control-plane/manager.core.ts:2340`
  - body: When the lane timeout aborts a cooperative runtime.close() and that close rejects later, this stale-operation branch clears the closing cache entry. That drops the handle from RuntimeCache.size() without proving the runtime process/session is gone, so maxConcurrentSessions can admit more sessions while the old runtime still consumes resources; keep the closing entry counted until close succeeds or a hard-kill path confirms cleanup.
  - confidence: 0.88
