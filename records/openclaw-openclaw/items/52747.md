## Summary

Keep this PR open: the ACP lane timeout direction is useful and current main still has the stuck-lane failure mode, but the current patch still has a resource-accounting bug in the stale close path that should block merge.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, AbortSignal propagation, stale-operation fencing, runtime close accounting, config/docs/changelog updates, and ACP manager regression coverage.

## Best Possible Solution

Keep the session-lane deadline design, but keep or restore closing runtime handles in accounting until close succeeds or a confirmed hard termination/retry path retires them, with a regression for rejected stale closes.

## Reproduction Assessment

Yes. Source inspection shows current main serializes same-session ACP work through KeyedAsyncQueue and awaits runtime.ensureSession in that lane without a deadline, so a never-settling backend load can keep later same-session work queued.

## Solution Assessment

No. The timeout, abort, and stale-fencing approach is close, but the current close error path can drop runtime accounting before the old runtime is confirmed gone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep aborted closes counted after timeout:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When runtime.close() rejects after the lane timeout aborts a stale operation, this branch clears the closing cache entry even though the runtime may still be alive. Because RuntimeCache.size() is the maxConcurrentSessions accounting source, dropping the closing entry can allow new ACP sessions past the configured limit while the old runtime is still consuming resources; keep or restore it until close succeeds or hard termination is confirmed.
  - confidence: 0.87
