## Summary

Keep open. Current main still has the ACP session-lane wedge this PR is trying to fix, but the current PR head still has a close-accounting bug that can drop live runtime resource accounting after a timed-out close rejects, and the real behavior proof names an older head SHA rather than the current head.

## What This Changes

The PR adds configurable ACP session-lane task timeouts, stale-operation fencing, AbortSignal plumbing through ACP runtime paths and ACPX, runtime-cache closing accounting, config/docs/schema/changelog updates, and ACP manager regression tests.

## Best Possible Solution

Keep the ACP session-lane deadline design, but retain closing runtime handles in accounting until close succeeds or a hard termination/retry path confirms retirement, then refresh current-head real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession inside that lane with no deadline, so a never-settling ensureSession can keep later work queued.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is close, but the current close error path clears runtime accounting before confirming that the old runtime is gone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep rejected closes counted after timeout:** `src/acp/control-plane/manager.core.ts:2353`

  - confidence: 0.88
