## Summary

Keep open: current main still has the reported ACP stuck-lane path, but the latest PR still has a blocking runtime-cache accounting bug around timed-out closes and lacks after-fix real behavior proof.

## What This Changes

The PR adds `acp.sessionLane.taskTimeoutMs`, cooperative abort/stale fencing, runtime-cache closing tracking, config/docs/schema wiring, a changelog entry, and ACP manager regression tests.

## Best Possible Solution

Keep the ACP lane-deadline design, but keep closing runtime handles counted until close is confirmed or a hard termination path completes, then add targeted tests for aborted/rejected close plus real ACP stuck-session recovery proof.

## Reproduction Assessment

Yes from source inspection: current main serializes same-session ACP work through a promise tail, and `initializeSession()` awaits `runtime.ensureSession()` inside that lane without a deadline. I did not execute tests because this review is read-only.

## Solution Assessment

No. The latest PR improves the earlier timeout design with abort signals and stale fencing, but the close path can still drop runtime accounting after a cooperative abort, and the PR has no real behavior proof yet.
