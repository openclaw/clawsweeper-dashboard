## Summary

This is a real ACP control-plane bug and the PR is actively improved, but it still has blocking correctness and resource-accounting issues in the latest head. The supplied terminal proof is now enough for the control-plane stuck-lane path, and CI is green, but the patch needs maintainer review before merge.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, cooperative abort and stale-operation fencing, runtime cache closing accounting, config/docs/schema wiring, tests, and a changelog entry.

## Best Possible Solution

Keep the ACP lane-deadline direction for stuck initialize/status/cancel/close work, but preserve configured turn timeout semantics and keep runtime handles counted until close succeeds or a hard termination/retry path resolves them.

## Reproduction Assessment

Yes from source inspection: current main chains ACP session actor tasks per key and `initializeSession()` awaits `runtime.ensureSession()` without a deadline, so a never-settling backend load holds the lane. I did not execute tests because this review is read-only.

## Solution Assessment

No. The proposed design is close for stuck control-plane tasks, but the current implementation applies the new lane timeout too broadly and still has an unsafe stale-close rejection path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not let lane timeout override ACP turn timeouts:** `src/acp/control-plane/manager.core.ts:2134`
  - body: `withSessionActor()` now applies the default 600000ms lane timeout to every caller, including `runTurn()`. That means a long ACP turn configured through `/acp timeout`, `runTimeoutSeconds`, or the 48-hour agent default can be aborted after 10 minutes with the generic lane-timeout error; keep this timeout off active turns or ensure it cannot fire before the existing turn timeout contract.
  - confidence: 0.9
- **[P1] Keep rejected stale closes counted until closure is confirmed:** `src/acp/control-plane/manager.core.ts:2353`
  - body: If a timed-out `runtime.close()` rejects after the operation has been marked stale, this branch clears the closing entry even though the runtime may still be alive. That drops it from `RuntimeCache.size()` and can let `maxConcurrentSessions` admit new work before the old backend process/session is actually closed; keep the closing entry or use a hard-kill/retry path before releasing accounting.
  - confidence: 0.88

## Security Review

Status: needs_attention
