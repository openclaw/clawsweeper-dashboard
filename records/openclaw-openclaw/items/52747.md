## Summary

Keep open: this PR targets a real ACP lane liveness bug and has sufficient terminal proof, but the current head still drops counted runtime accounting after a stale close rejection, which can undercount active ACP runtimes. GitHub also reports the branch is now conflicting with current main, so it needs repair before merge.

## What This Changes

The PR adds configurable ACP session-lane task timeouts with abort/stale fencing, runtime-cache closing accounting, ACPX signal passthrough, config/docs/changelog wiring, and regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but preserve runtime accounting until close succeeds or a confirmed retry/termination path retires the runtime, then rebase cleanly with the complementary queue-abort work in mind.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is close, but the stale close rejection branch should keep runtime accounting until the runtime is confirmed closed or retired by a retry/termination path.

## Review Findings
