## Summary

Keep open: the stuck ACP lane bug is real and the PR now has sufficient after-fix proof, but the current diff still drops closing-runtime accounting when a timed-out close rejects, so it should not merge as-is.

## What This Changes

The PR adds configurable ACP session-lane task timeouts with abort/stale fencing, runtime-cache closing accounting, config/docs/schema wiring, ACPX signal propagation, tests, and a changelog entry.

## Best Possible Solution

Keep the session-lane timeout design, but retain closing runtime handles in accounting until close succeeds or a hard termination/retry path conclusively retires them, with regression coverage for aborted and rejected close.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession inside that lane with no deadline, so a never-settling ensureSession can keep later work queued.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is close, but the current close error path clears runtime accounting before confirming that the old runtime is gone.

## Review Findings

Overall correctness: patch is correct

- **[P1] Keep rejected closes counted after timeout:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When a lane timeout aborts runtime.close(), this stale catch branch removes the closing entry even if the runtime rejects on abort and may still be alive.
