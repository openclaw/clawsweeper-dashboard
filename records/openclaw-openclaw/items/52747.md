## Summary

Keep open: the PR addresses a real ACP lane liveness bug and includes sufficient terminal proof, but the current head still has a blocking stale-close rejection path that can undercount live runtime resources.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, cooperative abort and stale-operation fencing through ACP manager/runtime contracts, runtime-cache closing accounting, config/docs wiring, and regression tests.

## Best Possible Solution

Keep the session-lane deadline and stale-fencing design, but preserve runtime accounting until close succeeds or a confirmed termination/retry path retires the runtime.

## Reproduction Assessment

Yes. Source inspection shows current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, cooperative abort, and stale-fencing direction is appropriate, but clearing a counted closing entry after a stale close rejection can undercount live runtime resources and should be fixed before merge.
