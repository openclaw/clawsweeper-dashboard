## Summary

Keep open. The stuck ACP session-lane bug is valid, but the current patch still releases timed-out lane work while underlying operations can continue, leaving prompt-turn bookkeeping and idle-close runtime lifecycle state inconsistent before merge.

## What This Changes

The PR adds a configurable ACP session-lane task timeout, wires it through manager/config schema/help surfaces, and adds regression coverage for hung lane tasks.

## Best Possible Solution

Keep the session-lane deadline direction, but revise it so timeout reporting cannot release same-session work until cancellation or stale fencing preserves terminal state, and quarantine runtime handles until close is confirmed.

## Reproduction Assessment

Yes. Source inspection shows a never-settling `runtime.ensureSession()` can run inside `withSessionActor()`, and `KeyedAsyncQueue` only advances same-key work when that promise settles; I did not execute tests because `pnpm` is unavailable in this read-only shell.

## Solution Assessment

No. The PR's timer race rejects the queued promise before the underlying ACP operation necessarily stops, and the added stale fencing avoids some writes by skipping terminal bookkeeping while idle close leaves handles reusable while closing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve terminal bookkeeping on lane timeout:** `src/acp/control-plane/manager.core.ts:923-925`
  - body: When a prompt turn times out after creating a background task and setting the session to `running`, this stale-operation return exits before `recordTurnCompletion()`, `markBackgroundTaskTerminal()`, and the error-state write. The caller gets a timeout, but durable task/session state can remain running for later status and recovery paths.
  - confidence: 0.91
- **[P1] Quarantine handles while idle close is pending:** `src/acp/control-plane/manager.core.ts:1927-1930`
  - body: Idle eviction now lets the lane timeout while `runtime.close()` is still pending and keeps the cached handle visible until close resolves. A later same-session task can reuse or re-close a runtime that is already terminating; keep the lane held or move the handle into a closing/quarantined state until it is retired.
  - confidence: 0.88
