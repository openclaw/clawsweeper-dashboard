## Summary

Keep this PR open. The stuck ACP lane problem is real and the PR has useful timeout, abort, config, docs, and test work, but the current head still drops runtime-cache accounting when a stale close later rejects, which can leak an ACP runtime while allowing more sessions past the configured cap.

## What This Changes

The PR adds configurable ACP session-lane task timeouts with cooperative abort/stale fencing, runtime-cache closing accounting, ACPX signal passthrough, config/docs/schema wiring, changelog, and regression tests.

## Best Possible Solution

Keep the session-lane timeout approach, but preserve runtime accounting until close succeeds or a confirmed retry/termination path retires the runtime, then rebase and validate the touched ACP/config surfaces before merge.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is close, but the stale close rejection branch should keep runtime accounting until the runtime is confirmed closed or retired by a retry/termination path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When runtime.close() rejects after the lane timeout has made this operation stale, this branch clears the counted closing entry even though the failed close is not proof that the runtime is gone. Because RuntimeCache.size() includes closing entries for maxConcurrentSessions, this can leak an ACP runtime while allowing more sessions; keep the entry counted until close succeeds or a confirmed termination/retry path retires it.
