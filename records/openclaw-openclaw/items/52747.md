## Summary

Keep open: current main has a valid ACP lane-stall failure mode, but this PR still releases the per-session queue while the underlying ACP operation can continue, and it lacks after-fix real behavior proof.

## What This Changes

Adds `acp.sessionLane.taskTimeoutMs`, wraps `SessionActorQueue` tasks with timeout/stale-fencing behavior, updates ACP manager/config schema/help surfaces, and adds manager regression tests.

## Best Possible Solution

Keep the lane-deadline goal, but redesign it around cooperative cancellation or a caller-visible timeout that preserves per-session serialization, terminalizes task/session state, and quarantines runtime handles until close is confirmed.

## Reproduction Assessment

Yes from source inspection: make `runtime.ensureSession()` never settle inside `initializeSession()`, and current main keeps later same-key work chained behind that promise. I did not execute tests because this review is read-only.

## Solution Assessment

No. The PR's timer race settles the queue promise before the ACP operation necessarily stops, and the added stale fencing does not cover every post-await side effect or runtime-cache lifecycle race.
