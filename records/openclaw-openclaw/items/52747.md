## Summary

Keep open: the bug is real and the PR proves the main timeout path, but the current head still drops runtime accounting when a timed-out close later rejects, which can leak or over-admit ACP runtimes.

## What This Changes

The PR adds a configurable ACP session-lane task timeout with abort/stale fencing, ACPX signal passthrough, config/schema/docs wiring, a changelog entry, and regression tests.

## Best Possible Solution

Keep the session-lane deadline and stale-fencing direction, but preserve counted closing runtime state until close succeeds or a confirmed termination/retry path retires the runtime.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession without a deadline; the PR body also supplies terminal proof of the stuck-before and unblocked-after behavior.

## Solution Assessment

No. The timeout, abort, and stale-fencing approach is close, but clearing the counted closing entry after a stale close rejection is not the safest fix because RuntimeCache.size() backs maxConcurrentSessions accounting.

## Review Findings
