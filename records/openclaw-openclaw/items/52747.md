## Summary

Keep open. Current main still has no ACP session-lane timeout, so the PR addresses a real stuck-lane gap, but the latest head still releases the per-session queue before preserving cleanup invariants; timed-out prompt turns can leave durable state running, and timed-out idle closes can leave a cached handle reusable while close is still pending.

## What This Changes

The branch adds `acp.sessionLane.taskTimeoutMs`, threads it through ACP session actor execution and config schema/help metadata, and adds manager/config regression coverage for hung lane tasks.

## Best Possible Solution

Keep the session-lane deadline direction, but revise it so a deadline cannot release same-session work until terminal bookkeeping is durable or the runtime handle is quarantined, preserve runtime-cache accounting until close is confirmed, and add focused tests for hung initialize, prompt-turn lane timeout, and idle-close timeout/reuse behavior.

## Reproduction Assessment

Yes. A manager-test style mocked ACP runtime whose `runTurn()` or `close()` remains pending past `acp.sessionLane.taskTimeoutMs` can reproduce the two remaining races: prompt state remains running after the caller receives the timeout, and a cached runtime handle can be reused or closed again while its idle close is still pending.

## Solution Assessment

No. The PR is closer than the original timeout race because it adds unref, clamping, and stale fencing, but releasing the queue before cleanup and without a closing/quarantine cache state is not the narrowest maintainable fix for ACP runtime lifecycle safety.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve terminal state before marking timed-out turns stale:** `src/acp/control-plane/manager.core.ts:923-925`
  - body: When the session-lane timeout fires, `withSessionActor()` marks the operation stale. A prompt turn may already have created a background task and set the session to `running`, but this catch returns before `markBackgroundTaskTerminal()` and the error-state write, so the caller gets a timeout while durable state can remain running.
  - confidence: 0.91
- **[P1] Block reuse while idle eviction close is still pending:** `src/acp/control-plane/manager.core.ts:1947-1948`
  - body: The idle-eviction close is now wrapped in the lane timeout, but the cache entry remains visible until `runtime.close()` resolves. If that close times out, later same-session work can run against or re-close the same cached handle while the original close is still pending; add a closing/quarantine state or keep the lane held until the handle is retired.
  - confidence: 0.88
