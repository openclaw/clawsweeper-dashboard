## Summary

Keep open: the PR addresses a real ACP session-lane deadlock, but the latest head still has a blocking runtime-accounting bug when a timed-out close later rejects.

## What This Changes

The branch adds a configurable ACP session-lane task timeout, cooperative runtime abort signals, stale-operation fencing, runtime-cache closing accounting, config/docs wiring, and ACP manager regression tests.

## Best Possible Solution

Keep the session-lane deadline design, but preserve runtime accounting until close succeeds or a confirmed termination/retry path retires the runtime, with rejection-path regression coverage.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession can await a non-settling ensureSession without a lane deadline; the PR body also supplies terminal before/after proof for that path.

## Solution Assessment

No. The timeout, abort, and stale-fencing direction is appropriate, but clearing the counted closing entry after a stale close rejection can undercount live runtime resources and should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep rejected stale closes counted:** `src/acp/control-plane/manager.core.ts:2472`
  - body: When `runtime.close()` rejects after the lane timeout has marked this operation stale, this branch clears the counted closing entry even though the failed close is not proof that the runtime is gone. Because `RuntimeCache.size()` includes closing entries for `maxConcurrentSessions`, this can leak an ACP runtime while allowing extra sessions; keep the entry counted until close succeeds or a confirmed termination/retry path retires it.
  - confidence: 0.88
