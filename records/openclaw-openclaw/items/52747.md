## Summary

Keep open: the PR addresses a real ACP lane liveness bug, but current head still has a concrete runtime-accounting bug when a stale timed-out close rejects.

## What This Changes

The PR adds configurable ACP session-lane timeouts with abort/stale fencing, acpx signal passthrough, config/docs/schema/changelog wiring, and regression tests.

## Best Possible Solution

Keep the timeout, abort, and stale-fencing design, but preserve counted closing runtime state until close succeeds or a confirmed termination/retry path retires it.

## Reproduction Assessment

Yes. Current main serializes same-session ACP work through KeyedAsyncQueue and initializeSession awaits runtime.ensureSession in that lane without a deadline; the PR body also supplies terminal proof of the stuck-before and unblocked-after behavior.

## Solution Assessment

No. The direction is close, but clearing the counted closing entry on stale close rejection is not the safest fix because RuntimeCache.size() backs maxConcurrentSessions accounting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep failed stale closes counted:** `src/acp/control-plane/manager.core.ts:2353`
  - body: When runtime.close() rejects after the lane timeout marks this operation stale, this branch clears the counted closing entry even though the runtime may still be alive. Since RuntimeCache.size() backs maxConcurrentSessions, a failed or aborted close can leak a runtime while admitting new ACP sessions; keep the closing entry counted until close succeeds or a confirmed hard termination/retry retires it.
  - confidence: 0.86
