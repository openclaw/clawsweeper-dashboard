## Summary

Keep open: the PR targets a real Telegram context-duplication bug, but it introduces one actionable metadata-leak issue and the supplied external proof is still test/build output rather than real Telegram behavior proof.

## What This Changes

The PR adds Telegram message-cache session key metadata and direct-chat prompt-context dedupe, with focused persisted-cache and private-chat tests.

## Best Possible Solution

Keep the Telegram-local dedupe direction, but strip internal session key metadata from reply-chain/public context and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, at source level: current main selects recent Telegram cached messages into `Conversation context` while the shared turn flow records the same inbound context into session history and Codex prepends current-turn context to the prompt. I did not run a live Telegram session.

## Solution Assessment

No, not yet: the direct-chat session-bound dedupe approach is narrow and plausible, but the patch should first prevent `sessionKeys` from leaking into reply-chain context and supply real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Strip session keys before building reply-chain context:** `extensions/telegram/src/message-cache.ts:19`
  - body: Adding `sessionKeys` to cached message nodes means the existing `toReplyChainEntry` spread will copy that internal array into `ctxPayload.ReplyChain` for reply targets. That raw context is recorded with the turn, so messages bound to another Telegram session can expose routing metadata; drop `sessionKeys` alongside `sourceMessage` before returning reply-chain entries.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff stays local to Telegram runtime/cache/tests, but the new session key metadata can leak into recorded reply-chain context unless stripped.
