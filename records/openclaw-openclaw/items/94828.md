## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Require origin slug match before deleting:** `src/skills/lifecycle/clawhub.ts:1585`
  - body: `planSkillUninstall` treats any normalized origin metadata under the requested directory as ClawHub provenance, but it never checks that `trackedOrigin.slug` normalizes to `resolvedSlug`. A copied or corrupt `.clawhub/origin.json` for another skill would set `isClawHubInstall` and let `executeSkillUninstall` recursively remove this directory, despite the docs promise that non-ClawHub directories are left untouched. Please require the origin slug, owner, and lock data to match the requested install before setting this true, and add a mismatched-origin regression test.
  - confidence: 0.93
- **[P3] Report owner-qualified misses as not installed:** `src/skills/lifecycle/clawhub.ts:1592`
  - body: For an untracked request like `@owner/missing`, `ownerRequiredButMissing` becomes true solely because a requested owner exists and no tracked owner was found. The CLI checks that before the no-dir/no-lock branch, so a missing skill reports an owner mismatch instead of the documented not-installed outcome. Gate this flag on an actually tracked install, or check the not-installed case first.
  - confidence: 0.88
- **[P2] Make dry-run reflect actual removals:** `src/cli/skills-cli.ts:441-445`
  - body: The dry-run output always says the workspace directory and lockfile entry would be removed, even when `isClawHubInstall` is false or `lockfileEntryExists` is false. For a manual non-ClawHub skill or stale state, `--dry-run` now reports removals that execution will not perform, which is especially misleading for a destructive command. Please render the dry-run plan from the same conditions used by `executeSkillUninstall`.
  - confidence: 0.84
