## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve valid URL whitespace semantics:** `src/agents/tools/web-fetch.ts:439`
  - body: `replace(/\s+/g, "")` deletes whitespace everywhere before parsing. That fixes `https:// docs...`, but it also changes already-valid URLs like `https://example.com/a b` or `?q=a b`, which `new URL()` currently serializes with `%20`; after this patch they fetch `/ab` or `?q=ab`. Normalize only the malformed scheme/host boundary or parse and use the URL serializer so valid path/query spaces are preserved.
  - confidence: 0.92
- **[P2] Keep fallback results on the normalized URL:** `src/agents/tools/web-fetch.ts:510`
  - body: When the guarded fetch falls back to a provider, the PR passes `cleanedUrl` as `urlToFetch` but `maybeFetchProviderWebFetchPayload` still reports `requestedUrl: params.url`. For an input like `https:// docs.example.com/page`, fallback can fetch the cleaned URL while returning the raw invalid URL to downstream tool chaining, so normalize the request URL once before all direct and provider payload paths.
  - confidence: 0.82
