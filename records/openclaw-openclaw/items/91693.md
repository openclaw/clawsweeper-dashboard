## Summary

Keep this PR open: the revised branch fixes the earlier provider-response defect and preserves the guarded-fetch boundary, but its recovery fallback still removes whitespace from every part of an otherwise invalid URL, which can silently change the intended hostname or path instead of repairing only the reported scheme-adjacent typo. ## What This Changes

The PR retries failed web_fetch URL parsing after whitespace removal, propagates the resulting canonical URL through direct and provider fetch paths, and adds two URL-normalization regression tests.

## Best Possible Solution

Restrict recovery to the documented typo class—such as whitespace directly following an HTTP(S) scheme—while retaining normal URL parsing for every other input, then prove the latest head through an actual OpenClaw web_fetch invocation plus focused direct-fetch, provider-fallback, and SSRF tests.

## Reproduction Assessment

Yes. Current main deterministically rejects the reported whitespace-corrupted scheme at `new URL(params.url)`, and the proposed tests exercise that tool entry point without requiring the original model to reproduce the typo.

## Solution Assessment

No, not yet. Parsing normally first is correct, but globally deleting whitespace after any parse failure is broader than necessary; scheme-specific recovery is safer and more closely matches the reported failure.
