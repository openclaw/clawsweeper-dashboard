## Summary

Keep open: this is a plausible fix for the queued same-session delivery-loss bug, but the current patch is not merge-ready because lazy fence creation can also let an older same-key reply deliver after a newer visible reply has already won.

## What This Changes

The branch makes foreground reply fence creation lazy in buffered auto-reply dispatch and adds freshness tests for settled-only and queued same-session delivery.

## Best Possible Solution

The end state should align foreground fence creation with real delivery/admission while retaining current newer-visible winner ordering for stale same-key replies.

## Reproduction Assessment

Yes, at source level: beforeDeliver runs immediately before transport delivery, and this patch delays fence creation until that point. The existing freshness test for newer-visible stale suppression demonstrates the invariant the patch can break.

## Solution Assessment

No as submitted. Deferring fence creation is the right layer for queued-before-visible turns, but the best fix must also preserve newer-visible stale suppression for same-key replies that lose the visible delivery race.
