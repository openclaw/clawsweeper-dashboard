## Summary

Keep this PR open, but it still needs changes before merge: the lazy fence approach targets the right queued-before-visible bug, but as submitted it can let an older same-key dispatch create a fresh fence and deliver after a newer visible reply has already won.

## What This Changes

The PR defers foreground reply fence creation in `dispatchInboundMessageWithBufferedDispatcher` until `beforeDeliver` or `deliver`, and adds freshness tests for settled-only and queued same-session delivery cases.

## Best Possible Solution

Keep lazy fence creation for queued-before-visible turns, but preserve the existing newer-visible stale suppression invariant with focused freshness coverage for both cases.

## Reproduction Assessment

Yes at source level: `beforeDeliver` runs immediately before transport delivery, and this patch delays fence creation until that point. The existing freshness test that suppresses an older final after a newer visible reply demonstrates the invariant the patch can break.

## Solution Assessment

No as submitted. Deferring fence creation is the right layer for queued-before-visible turns, but the best fix must also preserve newer-visible stale suppression for same-key replies that lose the visible delivery race.
