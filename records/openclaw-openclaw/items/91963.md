## Summary

Keep open: the lazy foreground-fence direction is relevant to the linked message-loss bug, but the submitted patch can let an older same-key dispatch create a fresh fence after a newer visible reply already completed, allowing stale or out-of-order external-channel delivery.

## What This Changes

The PR defers foreground reply fence creation in `dispatchInboundMessageWithBufferedDispatcher` until delivery time and adds freshness tests for settled-only and queued same-session delivery cases.

## Best Possible Solution

Preserve lazy fence creation for queued-before-visible turns while carrying enough ordering state to keep newer-visible stale suppression intact, with focused freshness coverage for both cases.

## Reproduction Assessment

Yes at source level: the dispatcher queue calls `beforeDeliver` immediately before delivery, and the PR delays fence creation until that point. I did not run tests because this is a read-only review, but the current freshness test suite encodes the visible-winner suppression path that the patch can break.

## Solution Assessment

No as submitted. Deferring fence creation is the right layer for queued-before-visible turns, but the best fix must also preserve newer-visible stale suppression for turns that lose the visible delivery race.
