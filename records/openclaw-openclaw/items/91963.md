## Summary

The central race fix is directionally correct, but this PR changes an exported plugin SDK callback contract: `onFreshSettledDelivery` no longer runs when no ordinary delivery created a fence snapshot. That regression should be repaired before merge, and the claimed live validation needs inspectable evidence.

## What This Changes

The PR lazily creates the buffered auto-reply foreground freshness fence on the first delivery attempt and adds a regression test for a queued newer same-session dispatch.

## Best Possible Solution

Keep lazy fence creation for ordinary delivery, but always invoke `onFreshSettledDelivery` with the optional snapshot so existing plugin callers retain settled fallback delivery; add a no-ordinary-delivery regression test and attach redacted live channel proof.

## Reproduction Assessment

Yes. Current main opens the newer freshness generation at buffered-dispatch entry before the queued same-session turn emits a reply, and the added deterministic test reproduces that ordering from source.

## Solution Assessment

No, not as written. Lazy fence creation is the narrow correct direction, but the implementation should preserve unconditional settled-callback execution because freshness filtering already supports an undefined snapshot.
