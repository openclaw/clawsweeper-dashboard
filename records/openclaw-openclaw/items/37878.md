## Summary

Keep open. Current `main` and latest release `v2026.5.20` still use raw Python numeric checks for model-usage costs, so numeric-string costs are dropped while booleans and non-finite floats can pass through; the linked fix PR is closed unmerged and the open token-summary PR does not own this normalization.

## What This Changes

Normalize model-usage cost values by accepting finite numeric strings, rejecting booleans and non-finite values, and adding regression coverage.

## Best Possible Solution

Add one shared finite cost-normalization helper and route `aggregate_costs`, `pick_current_model`, and `latest_day_cost` through it with focused regression tests.

## Reproduction Assessment

Yes. A bytecode-disabled Python probe against current `main` reproduces dropped numeric-string cost values, accepted boolean cost values, and propagated `NaN`/`Infinity` values in the affected helper paths.

## Solution Assessment

Yes. A shared finite cost parser is the narrowest maintainable fix because all three cost consumers need the same input-normalization contract without changing plugin APIs or product policy.
