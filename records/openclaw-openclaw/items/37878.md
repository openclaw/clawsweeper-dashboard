## Summary

Keep open. Current main and the latest release still use raw Python numeric checks for model-usage costs, so numeric-string costs are dropped while booleans and non-finite floats can affect totals; the directly linked implementation PR was closed unmerged, and the open token-summary PR does not supersede this cost-normalization bug.

## What This Changes

The issue requests that the model-usage helper accept finite numeric-string costs while ignoring booleans, NaN, and Infinity values.

## Best Possible Solution

Add one shared finite cost-normalization helper and route aggregate_costs, pick_current_model, and latest_day_cost through it with focused edge-case regression tests.

## Reproduction Assessment

Yes. A current-main Python -B probe against aggregate_costs, pick_current_model, and latest_day_cost reproduces the dropped numeric string, accepted boolean, and propagated NaN/Infinity behavior.

## Solution Assessment

Yes. A shared parse_cost-style helper is the narrowest maintainable fix because all three affected paths need the same finite-cost contract.

Fix the model-usage cost normalization bug reported in https://github.com/openclaw/openclaw/issues/37878. In skills/model-usage/scripts/model_usage.py, add a shared helper that accepts finite numeric costs from ints/floats and numeric strings, rejects booleans, rejects NaN/Infinity, and returns a normalized float/None. Use it in aggregate_costs, pick_current_model, and latest_day_cost without changing CLI behavior otherwise. Add focused tests in skills/model-usage/scripts/test_model_usage.py for numeric-string aggregation, boolean rejection, NaN/Infinity rejection, current-model selection, and latest-day numeric-string cost.
