## Label Justifications

- `P2`: This is a normal-priority bug in a bundled skill helper with limited blast radius but real cost-summary correctness impact.
- `impact:data-loss`: Valid cost values can be silently omitted from summaries and non-finite values can corrupt totals, which is dropped or unusable usage data.

## Summary

Keep open because current main and latest release still use native `int`/`float` checks for model-usage costs, while https://github.com/openclaw/openclaw/pull/87861 is an open clean PR that closes this issue. Closing now would leave the source-reproducible cost undercount and NaN-poisoning path unresolved.

## What This Changes

The issue asks for `model-usage` cost parsing to accept numeric-string cost values while ignoring booleans and non-finite values across aggregation, current-model selection, and latest-day cost output.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/87861, or an equivalent narrow fix that centralizes finite cost coercion across all three cost readers, then close this issue after merge.

## Reproduction Assessment

Yes at source level: current main's three cost-reading sites use `isinstance(cost, (int, float))`, so numeric strings are skipped while booleans and NaN/Infinity pass through the native-number path. I did not run a command repro because this review is read-only.

## Solution Assessment

Yes, a single finite-cost coercion helper reused by `aggregate_costs`, `pick_current_model`, and `latest_day_cost` is the narrowest maintainable fix. The open linked PR implements that shape, so the issue should remain open until that PR lands or is replaced.
