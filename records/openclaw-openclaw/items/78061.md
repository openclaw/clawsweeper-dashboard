## Summary

Keep open: this remains a credible Slack thread delivery bug with multiple live reports, a source-reproducible native-stream finalization drop path on current main, and an open PR that explicitly closes this issue but has not merged.

## What This Changes

Slack thread sessions should deliver generated assistant replies back to the originating Slack thread instead of only recording them in the session transcript.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78536 or an equivalent focused Slack native-stream pending-text fallback, validate the threaded delivery path, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes at the source level, though not as a fresh live Slack run in this review. The reproducible path is a short ChatStreamer-buffered reply followed by an unexpected stopSlackStream failure, which current main logs without routing pendingText through threaded fallback delivery.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix because it reuses the existing deliverReplies fallback and preserves the original Slack thread timestamp without changing Slack routing, config, or streaming defaults.
