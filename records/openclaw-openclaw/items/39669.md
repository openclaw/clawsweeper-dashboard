## Summary

Keep open. Current main still has the reported busy-followup behavior: the active-run path enqueues the message, cleans up typing, and returns without a generic queued-feedback hook or a Feishu/Lark queued receipt; related implementation work is still open or closed unmerged.

## What This Changes

The issue asks OpenClaw to immediately show user-visible acknowledgement when a message is queued behind a busy agent run.

## Best Possible Solution

Add a generic, non-blocking queued-followup feedback seam at the active-run enqueue-followup boundary, then wire Feishu/Lark and any channel without existing initial status reactions to an immediate queued receipt or reaction. The hook should fire only after a followup is actually enqueued, swallow or log notification failures, avoid duplicate status signals on channels that already set initial queued reactions, and include focused queue and Feishu regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with an active run and a followup/steer fallback queue mode, resolveActiveRunQueueAction reaches enqueue-followup, runReplyAgent enqueues and returns undefined, and existing tests already assert that enqueue-return shape without any feedback callback.

## Solution Assessment

Yes. The narrow maintainable direction is a generic queued-receipt callback or typed hook plus channel-specific wiring, rather than a Feishu-only special case; #39117 may be the active implementation vehicle but still needs review against this behavior.
