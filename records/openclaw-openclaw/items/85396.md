## Summary

Keep open. The performance direction is sound and the PR body includes copied before/after benchmark output, but the new no-config shortcut does not mirror the existing config dotenv contract and can skip legacy runtime dotenv plugin overrides before deciding static help is safe.

## What This Changes

The PR adds a filesystem precheck before root help imports the config stack, plus a regression test for the no-config root help fast path.

## Best Possible Solution

Keep the fast path, but make its dotenv probe match the config loader's runtime dotenv sources before skipping the config import.

## Reproduction Assessment

Yes, by source inspection: current config reads load runtime dotenv before the config existence check, while the PR precheck only looks for CWD and state .env files. I did not run a runtime repro because this review was read-only.

## Solution Assessment

No. The fast-path approach is maintainable, but it should mirror the existing dotenv source contract before returning null and printing precomputed help.

- **[P2] Check the legacy runtime dotenv before returning null:** `src/cli/root-help-live-config.ts:43`
  - body: `readConfigFileSnapshot()` currently calls `loadDotEnv()` before checking for a config file, and that runtime path also reads `~/.config/openclaw/gateway.env`. Mirror the same runtime dotenv probes, or load only the relevant plugin-help env before returning null.
