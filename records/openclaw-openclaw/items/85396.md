## Summary

Keep this PR open: the performance direction is useful and has plausible timing proof, but the new preflight misses one existing dotenv source and can make root help ignore plugin-help-affecting env overrides from the Ubuntu gateway.env compatibility path.

## What This Changes

The PR adds a fast path that returns precomputed root help without importing config when there is no config file, no checked dotenv file, and no current plugin-help-affecting env override, plus a focused test for the no-config case.

## Best Possible Solution

Keep the fast path, but make its dotenv/config preflight match the existing config dotenv contract before it skips the config import.

## Reproduction Assessment

Yes. Source inspection shows a gateway.env-only plugin-help-affecting env override is loaded by current config reads but is invisible to the PR's new preflight before it returns precomputed help.

## Solution Assessment

No, not yet. The optimization is the right shape, but the preflight needs to reuse or mirror the existing dotenv discovery contract before it is a safe compatibility-preserving fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep gateway.env in the dotenv preflight:** `src/cli/root-help-live-config.ts:43`
  - body: Current config reads call `loadDotEnv()`, and global dotenv loading also reads `~/.config/openclaw/gateway.env` when the default state dir is in use. With this preflight, a user whose only plugin-help-affecting setting is in `gateway.env`, such as `OPENCLAW_BUNDLED_PLUGINS_DIR`, returns `null` before config loading hydrates it, so root help uses stale precomputed commands. Include the compatibility path, or share the real dotenv discovery helper, before skipping the config import.
