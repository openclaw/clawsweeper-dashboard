## Summary

Codex review finds the issue still necessary: current main can dynamically execute newly introduced OpenRouter model IDs, but general OpenRouter text discovery and picker/list population still depend on a static catalog. The protected `maintainer` label and unresolved filtering, safety, caching, and offline semantics require explicit maintainer product judgment.

## What This Changes

Requests runtime discovery and selection of current provider model catalogs while preserving model allowlist, safety, and usability controls.

## Best Possible Solution

Add OpenRouter text discovery through the existing provider-owned unified catalog registration, reuse its capability endpoint and SQLite cache, and explicitly define filtered picker, refresh/offline, and allowlist behavior before implementation.

## Reproduction Assessment

Yes for the remaining gap: current source shows unknown OpenRouter IDs can be resolved dynamically while general text listing remains backed by a static catalog. This is a high-confidence source reproduction, not a live-provider run.

## Solution Assessment

No to the proposed second core `ModelCatalogSource` architecture. Current main already has provider-owned dynamic resolution and unified catalog APIs, so the narrowest maintainable solution is to complete OpenRouter text live-catalog support on those seams after maintainers choose filtering and safety semantics.
