## Summary

Keep open. The issue has a protected `maintainer` label, and current `main` only partially addresses the request: OpenRouter can resolve unknown text model IDs at runtime, but OpenRouter text model listing and picker/catalog discovery are still static or explicitly configured.

## What This Changes

Request fully dynamic runtime and listing model discovery for OpenRouter and other fast-moving providers while preserving model allowlist and cost-control semantics.

## Best Possible Solution

Keep this as the canonical maintainer-tracked feature and implement provider-owned live text catalog discovery for OpenRouter, using the existing provider catalog seams with bounded picker defaults, cache behavior, allowlist wildcard semantics, docs, and focused Gateway/CLI/UI coverage.

## Reproduction Assessment

Yes, from source inspection: current `models.list` and picker paths can only show OpenRouter text models present in static/configured catalogs, while live OpenRouter text catalog data is used for runtime dynamic resolution rather than listing. I did not need a live OpenRouter key to establish the missing listing path.

## Solution Assessment

Unclear as a fully specified implementation: provider-owned live text catalog discovery through the existing catalog seams looks like the maintainable direction, but picker defaults, cache TTL, and allowlist/cost-control behavior need maintainer product/API decisions first.
