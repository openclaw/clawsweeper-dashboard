## Summary

Current main fixes arbitrary OpenRouter model execution through provider-owned dynamic resolution, but it still does not expose the fetched provider catalog as a complete live browse list. The issue remains useful as the protected cross-provider product decision for picker scope, refresh/offline policy, and safety semantics.

## What This Changes

Add provider-backed runtime model discovery so current models can be listed and selected without static catalog lag while preserving allowlist, safety, and cost controls.

## Best Possible Solution

Extend the existing unified provider catalog contract so provider plugins can publish bounded refreshable browse entries while retaining per-ID runtime resolution, provider-wildcard allowlisting, SQLite caching, and explicit offline failure behavior.

## Reproduction Assessment

Yes for the remaining gap: source inspection shows OpenRouter fetches and caches live metadata for a requested ID but does not enumerate the full fetched catalog into browse and picker surfaces. The original arbitrary-ID `Unknown model` failure is no longer a high-confidence current-main reproduction.

## Solution Assessment

No, the issue's original new `ModelCatalogSource` plus `modelsMode` proposal is not the best current shape because unified catalog types, provider hooks, and provider wildcard scopes now exist; the narrow solution is to extend those existing seams.
