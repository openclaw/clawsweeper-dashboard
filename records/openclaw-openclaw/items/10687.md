## Summary

Current main materially fixes the original runtime failure by resolving arbitrary OpenRouter model IDs with live capability metadata, but the broader discovery request remains necessary: browse and list surfaces intentionally use persisted or static catalogs, and the protected maintainer signal plus unresolved picker-size, caching, offline, metadata, and allowlist semantics require a product decision.

## What This Changes

Add runtime discovery and selection of current models for fast-moving providers while preserving usable model pickers and allowlist and cost controls.

## Best Possible Solution

Extend the existing provider-owned catalog contract with an explicit read-only, bounded live-discovery result for browse and list surfaces, reuse the SQLite-backed metadata cache, keep arbitrary-ID runtime resolution separate, and define consistent wildcard allowlist, offline, freshness, and picker-filtering semantics before implementation.

## Reproduction Assessment

Yes. Current source provides a high-confidence reproduction of the remaining gap: arbitrary OpenRouter IDs can resolve for execution, while read-only browse and list loading stays static or persisted and therefore does not enumerate the current provider catalog.

## Solution Assessment

No, not as a parallel core ModelCatalogSource stack or a new mode setting by default. The narrowest maintainable direction is to extend the existing provider catalog and runtime hooks, reuse wildcard allowlists, and have maintainers decide bounded listing and cache behavior.
