## Summary

Keep open for maintainer product review. Current main resolves arbitrary OpenRouter model IDs and loads their capabilities on demand, but its text catalog still exposes only three static rows rather than the provider’s current catalog, so the requested list and picker discovery remains missing. Likely related people include @steipete for the unified catalog architecture, @shakkernerd for static provider-listing behavior, and @vincentkoc for recent model-catalog browsing work.

## What This Changes

Add provider-backed dynamic model discovery for current model listing and selection, initially for OpenRouter, while preserving manageable visibility and safety controls.

## Best Possible Solution

Extend the existing OpenRouter provider catalog hook to fetch and normalize current text models with bounded shared caching, preserve its static fallback and arbitrary-ID resolver, use provider wildcards for opt-in broad visibility, and track other providers through focused sibling issues using the same generic contract.

## Reproduction Assessment

Yes at source level. Current main deterministically exposes only three static OpenRouter text catalog entries while the live provider endpoint currently returns 339 models; this is a missing feature rather than a failure of an already documented dynamic-discovery contract.

## Solution Assessment

No, the original parallel core abstraction and new global modelsMode option are broader than necessary. The narrow maintainable path is provider-owned discovery through the existing generic catalog hook, shared cache infrastructure, and wildcard visibility controls.
