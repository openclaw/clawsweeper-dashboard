## Summary

Keep open: current main has partial OpenRouter runtime support for explicit dynamic model refs and a guarded live-catalog SDK, but OpenRouter catalog/list/picker discovery still uses a static three-row text catalog while the live provider endpoint exposes a much larger model set. The protected maintainer/product-decision context and unresolved allowlist, cache, offline, picker-size, and credential-safety semantics make this a maintainer design item rather than conservative cleanup.

## What This Changes

Request full dynamic model discovery for OpenRouter and similar providers so model lists, pickers, configured allowlists, and custom model refs can track provider `/models` endpoints instead of hardcoded catalogs.

## Best Possible Solution

Keep this as the canonical maintainer design thread and wire OpenRouter text catalog discovery through the guarded provider-catalog SDK after maintainers settle picker defaults, `provider/*` allowlist semantics, cache/offline behavior, and auth/SSRF policy.

## Reproduction Assessment

Yes, source-level: OpenRouter explicit runtime refs have dynamic hooks, but catalog/listing still use `buildOpenrouterProvider()` static rows; a read-only OpenRouter `/models` check returned 339 rows. I did not run an authenticated OpenClaw picker or agent flow.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable fix is provider-owned live catalog discovery via the guarded SDK plus maintainer-approved allowlist/cache policy, not an ad hoc unknown-model fallback or standalone raw fetch helper.
