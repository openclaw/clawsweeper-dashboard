## Summary

Keep open. Current `main` partially fixes the OpenRouter stale-catalog failure by resolving unknown OpenRouter model ids dynamically at runtime, but the catalog/listing surfaces are still backed by a static OpenRouter catalog. The issue’s stated goal of fully dynamic model discovery for `/model list` and related pickers is not implemented yet.

## Evidence

- **Runtime fallback for unknown OpenRouter models exists:** The bundled OpenRouter plugin now resolves unknown model ids via `resolveDynamicModel` and preloads OpenRouter capabilities in `prepareDynamicModel`, which addresses the direct runtime `Unknown model` path.
  - file: [extensions/openrouter/index.ts:50](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/openrouter/index.ts#L50)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **OpenRouter provider catalog is still static:** `buildOpenrouterProvider()` still returns a fixed list of four models (`auto`, `hunter-alpha`, `healer-alpha`, `moonshotai/kimi-k2.6`) rather than a live provider catalog.
  - file: [extensions/openrouter/provider-catalog.ts:36](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/openrouter/provider-catalog.ts#L36)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Model listing still reads the shared catalog path:** TUI model listing calls `loadGatewayModelCatalog()`, which delegates to `loadModelCatalog()`. That catalog loader reads the registry and only appends plugin `augmentModelCatalog` rows; it does not consume OpenRouter capability fetches directly.
  - file: [src/tui/embedded-backend.ts:363](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/tui/embedded-backend.ts#L363)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Shared catalog loader only adds explicit plugin augment rows:** `loadModelCatalog()` builds the catalog from the registry and then merges `augmentModelCatalogWithProviderPlugins(...)` results. There is no OpenRouter-specific live catalog fetch in this path.
  - file: [src/agents/model-catalog.ts:81](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/model-catalog.ts#L81)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **No OpenRouter catalog augmentation hook is present:** A repo search finds `augmentModelCatalog` referenced in `src/agents/model-catalog.ts`, but not implemented in `extensions/openrouter/index.ts`, so dynamic OpenRouter rows are not merged into picker/list surfaces.
  - command: `rg -n "augmentModelCatalog" extensions/openrouter/index.ts src/agents/model-catalog.ts`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Tests cover runtime resolution, not live catalog listing:** Current tests assert that unknown OpenRouter models resolve from cached/fetched capabilities, including nested model ids like `google/gemini-3.1-flash-image-preview`, but there is no corresponding catalog/listing proof here.
  - file: [src/agents/pi-embedded-runner/model.test.ts:581](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/pi-embedded-runner/model.test.ts#L581)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)

## Risks / Open Questions

- Closing now would lose the remaining work for dynamic `/model list` and picker discovery even though only the runtime unknown-model path is implemented.
- The partial fix appears on current `main`, but the repository metadata says the latest release is `v2026.4.22`, so shipped behavior may still lag behind `main`.
