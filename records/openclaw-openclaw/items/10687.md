## Summary

Keep open. Current main and the latest release partially support executing explicit OpenRouter model IDs through runtime capability lookup, but the OpenRouter text catalog, model-list, and picker path still use a curated provider catalog and only the video catalog is registered as live; the protected maintainer/product-decision context leaves discovery scope, picker scale, cache/offline behavior, and allowlist semantics unresolved.

## What This Changes

The issue requests fully dynamic model discovery so OpenRouter and other fast-moving providers can expose current model IDs without waiting for static catalog updates.

## Best Possible Solution

Keep this as the canonical maintainer design item for provider-owned live text catalog discovery through the existing plugin catalog seam after default picker/list scope, cache freshness/offline behavior, and allowlist semantics are decided.

## Reproduction Assessment

Yes, at source level. Current main can synthesize OpenRouter runtime models from cached /api/v1/models capabilities, but OpenRouter text list/picker rows still come from a small curated catalog and live catalog registration is video-only.

## Solution Assessment

Unclear as a fully specified implementation. Existing plugin/provider catalog seams look like the right boundary, but full OpenRouter and generic-provider discovery needs maintainer decisions on UX scale, freshness, offline fallback, and safety/cost controls.
