## Summary

Current main fixes the worst explicit OpenRouter `Unknown model` failure through provider-owned dynamic resolution, but it still does not expose the full live OpenRouter text catalog in normal list and picker surfaces; the remaining UX, filtering, refresh, and safety semantics need the protected maintainer/product decision.

## What This Changes

Request fully dynamic provider model discovery so newly available models can be listed and selected without waiting for a static catalog refresh.

## Best Possible Solution

Use the existing provider plugin catalog and dynamic-resolution contracts: add an authenticated, cached, filtered OpenRouter text live catalog consumed by list and picker surfaces, preserve `provider/*` allowlist policy, and define refresh/offline behavior without adding another config mode unless existing wildcard semantics prove insufficient.

## Reproduction Assessment

Yes, at source level: current main dynamically resolves an explicitly named OpenRouter model, but the catalog used for normal text-model listing remains curated to three entries, so the full discovery acceptance criterion is still reproducibly absent.

## Solution Assessment

No, the original core-wide `ModelCatalogSource` proposal is no longer the best shape because current main has generic provider plugin hooks; the narrow maintainable solution is an OpenRouter-owned live text catalog plus a maintainer-approved filtering and cache policy.
