## Summary

Keep open. Current main and the latest release partly address the original OpenRouter runtime failure by synthesizing and preparing arbitrary OpenRouter model IDs, but the full request is still not implemented because OpenRouter browse/list catalog rows remain a small static set and the issue has a protected maintainer label plus unresolved product questions.

## What This Changes

The issue requests fully dynamic model discovery for OpenRouter and other fast-moving provider catalogs without relying on static catalogs or disabling model safety controls.

## Best Possible Solution

Extend the existing provider-owned catalog contract so OpenRouter and similar providers can expose bounded live browse rows, reuse the existing capability/cache path where appropriate, and keep wildcard allowlists as the safety-control mechanism after maintainers decide default scope and offline behavior.

## Reproduction Assessment

Yes. Source inspection shows arbitrary OpenRouter IDs can be prepared for execution, while OpenRouter's provider catalog hook still returns a small static set and the Gateway browse path avoids fresh provider discovery per request.

## Solution Assessment

No, not as the issue's parallel core `ModelCatalogSource` plus a new mode setting by default. The better fit is likely to extend the existing provider/plugin catalog contract and settle picker size, cache freshness, offline, auth, and wildcard allowlist semantics first.
