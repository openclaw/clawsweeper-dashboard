## Summary

Keep open. The issue has a protected `maintainer` label and current `main` still only partially addresses the user problem: OpenRouter unknown-model runtime resolution is dynamic, but text model listing and picker/catalog discovery remain static or explicitly configured.

## What This Changes

The issue requests fully dynamic runtime model discovery so OpenRouter and other fast-moving provider catalogs can populate selection/listing surfaces while preserving allowlist and cost controls.

## Best Possible Solution

Keep this as the canonical maintainer-tracked feature and implement provider-owned live text catalog discovery, likely through `registerModelCatalogProvider`, with bounded picker defaults, allowlist/provider-wildcard semantics, cache behavior, and focused Gateway/CLI/UI tests.

## Reproduction Assessment

Yes, from source inspection: current `models.list` and picker paths do not have an OpenRouter live text catalog source, while runtime unknown-model resolution is handled separately. I did not need a live OpenRouter key to establish the missing listing path.

## Solution Assessment

No, closing this as fixed or treating the runtime fallback as sufficient would miss the issue's main listing/discovery goal. The maintainable direction is a provider-owned live catalog surface with explicit UX and allowlist policy.
