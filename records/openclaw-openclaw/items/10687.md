## Summary

Current main now resolves and runs previously unknown OpenRouter model IDs dynamically, addressing the original execution failure, but the broader request remains necessary because OpenRouter list and picker surfaces still use a small static catalog rather than the current provider catalog. This protected umbrella issue also retains unresolved product choices around catalog size, refresh behavior, offline fallback, and safety controls.

## What This Changes

Requests provider-backed runtime model discovery so current models can be listed and selected without waiting for static catalog updates.

## Best Possible Solution

Use the existing provider-plugin catalog augmentation seam for provider-owned live discovery, backed by bounded shared SQLite caching and deliberate picker filtering, while preserving current wildcard allowlist and configured-model safety semantics.

## Reproduction Assessment

Yes, at source level. Current main dynamically resolves an explicitly selected unknown OpenRouter model, while its provider catalog remains a small static list and therefore cannot populate list and picker surfaces with the current catalog.

## Solution Assessment

No, the proposed new core-wide `ModelCatalogSource` hierarchy is broader than necessary. The narrower maintainable direction is to implement live discovery through the existing provider-owned catalog augmentation hooks and shared cache infrastructure.
