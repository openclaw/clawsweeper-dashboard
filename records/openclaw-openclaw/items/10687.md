## Summary

Keep open. Current main partially addresses the OpenRouter execution failure with provider-owned dynamic model resolution and provider wildcard allowlists, but full dynamic browse/list discovery remains a maintainer-scoped product/API decision and this issue has the protected `maintainer` label.

## What This Changes

Request fully dynamic model discovery for fast-moving model providers, starting with OpenRouter, while preserving picker usability and allowlist safety controls.

## Best Possible Solution

Extend the existing provider-owned live catalog/plugin SDK path so OpenRouter and similar providers can expose bounded live rows, with maintainer-approved wildcard allowlist, cache, offline, and picker-default semantics.

## Reproduction Assessment

Yes. Source inspection shows current main can prepare and execute some arbitrary OpenRouter IDs, while the OpenRouter browse catalog still exposes only a small static row set and generic live catalog semantics remain unresolved.

## Solution Assessment

No, not as the issue's proposed parallel core `ModelCatalogSource` plus new mode setting by default. The more maintainable direction is to extend the existing provider/plugin live catalog contract after maintainers settle picker size, cache freshness, offline behavior, auth scope, and allowlist semantics.
