## Summary

Keep open: current main still has a source-proven WebChat terminal-error gap. The gateway can suppress the delayed active `chat.send` lifecycle error, and the Control UI `error` branch clears run state without reliably appending or reloading the persisted failed assistant turn; both related fix PRs are closed unmerged.

## What This Changes

WebChat should stop polling and show a failed assistant turn immediately when a provider 429 or idle timeout ends a chat run before content.

## Best Possible Solution

Deliver exactly one terminal failed chat turn after the retry grace and have Control UI display or reload the persisted fallback turn without provider-specific handling.

## Reproduction Assessment

Yes, at source level: current main has an active no-link `chat.send` lifecycle-error path that suppresses the terminal chat error after the grace timer, and the UI `error` branch does not fetch or append the persisted failed turn. I did not run a live provider 429 replay in this read-only review.

## Solution Assessment

Yes, the maintainable fix belongs in the gateway/UI terminalization path. The repair should cover final error delivery and failed-turn reconciliation rather than adding OpenRouter-specific handling.
