## Summary

Keep open: current main still preserves the suppressing `skipChatErrorFinal` option through the delayed lifecycle-error finalizer, and the adjacent test still asserts that no terminal chat error is emitted for the active no-link `chat.send` case.

## What This Changes

WebChat should stop its active polling state and show the failed assistant turn immediately when provider 429 or idle-timeout failures occur before content.

## Best Possible Solution

Make gateway lifecycle error finalization emit or reconcile a terminal WebChat error after the retry grace expires, with tests for active no-link `chat.send` failures and no provider-specific logic.

## Reproduction Assessment

Yes, at source level: current main schedules an active no-link `chat.send` lifecycle error with `skipChatErrorFinal` and the delayed finalizer still receives that suppressing option. I did not rerun a live OpenRouter 429 in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is in the gateway/WebChat terminalization path, not in OpenRouter-specific provider handling. The closed related PRs point at the right area, but a fresh current-main fix should preserve retry/fallback behavior while ensuring terminal failure reconciliation.
