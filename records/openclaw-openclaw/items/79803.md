## Summary

Keep open: current main still has a concrete WebChat terminal-error reconciliation gap. Source and adjacent tests show active `chat.send` lifecycle errors can be suppressed after the grace timer, and the UI `error` branch clears local state without pulling the persisted failed assistant turn into `chatMessages` unless a fragile deferred `session.message` reload happens.

## What This Changes

WebChat should stop the active polling state and show the failed assistant turn when a provider 429 or idle timeout fails before producing content.

## Best Possible Solution

Emit or reconcile exactly one terminal failed chat turn after the retry grace expires, then reload or display the persisted failed assistant turn so WebChat replaces the spinner without provider-specific logic.

## Reproduction Assessment

Yes, at source level: current main has an active chat.send lifecycle-error path that suppresses the terminal chat error after the grace timer, and the UI terminal `error` branch does not fetch the persisted failure turn unless a deferred session-message reload was already pending. I did not run a live provider 429 replay in this read-only review.

## Solution Assessment

Yes, the maintainable fix belongs in the gateway/UI terminalization path. The repair should cover both final error delivery and failed-turn history reconciliation rather than adding provider-specific OpenRouter handling.
