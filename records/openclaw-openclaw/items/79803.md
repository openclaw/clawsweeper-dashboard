## Summary

Keep open: current main still has a source-reproducible WebChat/gateway terminalization gap. The active chat-send/no-chat-link lifecycle-error path preserves `skipChatErrorFinal` through the grace timer and the current regression test expects no `state: "error"` chat broadcast; the related fix PR https://github.com/openclaw/openclaw/pull/79817 is closed and unmerged.

## What This Changes

WebChat should clear the live assistant polling state and show a failed assistant turn immediately when a provider 429 or idle timeout ends before producing content.

## Best Possible Solution

Update gateway chat lifecycle finalization so a delayed no-content lifecycle error emits a terminal WebChat error after retry grace and triggers the existing deferred history reload, with focused regression coverage.

## Reproduction Assessment

Yes at source level, though I did not rerun a live provider 429. Current main has an active chat-send/no-chat-link lifecycle-error path that suppresses the delayed `state: "error"` event after the grace timer, while the UI only reloads the persisted failed turn after receiving such a terminal chat event.

## Solution Assessment

Yes: the narrow maintainable fix is in gateway chat lifecycle terminalization, either by dropping or re-evaluating stale `skipChatErrorFinal` after the grace window. A provider-specific workaround would miss the generic lost-terminal-event failure mode.
