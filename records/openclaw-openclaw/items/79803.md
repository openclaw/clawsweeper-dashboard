## Summary

Keep open: the report is a concrete WebChat live-state bug, and current source still has a plausible gap where persisted failed assistant turns can remain hidden behind an active blank stream unless a matching terminal chat event triggers the deferred history reload.

## What This Changes

WebChat should stop the active assistant polling indicator and show the persisted failed assistant turn when provider 429 or idle-timeout failure happens before content.

## Best Possible Solution

Make failed agent-run completion produce a reliable WebChat terminal update that clears the active run and reloads or displays the persisted failed assistant turn, with coverage for provider 429/idle-timeout before content.

## Reproduction Assessment

Yes, at source level: an active blank stream renders the polling indicator, same-session transcript updates are deferred during `chatRunId`, and failed-turn display depends on a terminal event plus history reload. The reporter supplied live logs/screenshots, but I did not rerun a provider 429 locally.

## Solution Assessment

Yes: the narrow maintainable fix is in the WebChat/Gateway chat terminalization path, not a provider-specific workaround. The fix should preserve the existing deferred-history behavior for successful final payloads while forcing failed no-content turns to reconcile promptly.
