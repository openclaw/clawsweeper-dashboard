## Summary

Keep open: current main still contains the gateway/UI state path that can leave WebChat waiting for a terminal chat event, and an open linked fix PR explicitly claims to close this issue, so the issue should remain open until that PR is reviewed, landed, or closed.

## What This Changes

WebChat should stop the live assistant polling state and show the failed assistant turn when a provider 429 or idle-timeout failure occurs before content.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/79858, keeping the fix in the gateway/WebChat terminalization path with regression coverage and real behavior proof.

## Reproduction Assessment

Yes at source level: current main preserves skipChatErrorFinal through the delayed lifecycle-error finalizer, the adjacent test expects no terminal chat error for an active no-link chat.send run, and the reporter supplied live logs/screenshots. I did not independently run a live provider 429 flow.

## Solution Assessment

Yes: the narrow maintainable direction is to make the gateway emit a reliable terminal chat error and let WebChat reconcile the persisted failed turn, not add provider-specific OpenRouter handling.
