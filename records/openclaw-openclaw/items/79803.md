## Summary

Keep open: current main still has a source-level WebChat/gateway path where an active chat.send lifecycle error can be cleaned up without emitting a terminal chat error, leaving the UI dependent on refresh/history reload to show the persisted failed assistant turn.

## What This Changes

The issue asks WebChat to stop polling and show a failed assistant turn immediately when a provider 429 or idle-timeout fails before content.

## Best Possible Solution

Add a narrow gateway/WebChat terminalization fix so a failed no-content chat.send run reliably clears the active WebChat run and reconciles the persisted failed turn, while preserving fallback retry behavior from the lifecycle-error grace path.

## Reproduction Assessment

Yes, at source level: current main has a no-link active chat.send lifecycle-error test that expects no terminal chat error after the grace timer, matching the reported stuck polling path. The reporter also supplied live logs and screenshots, but I did not rerun a real provider 429.

## Solution Assessment

Yes: the best direction is a narrow terminal chat-event/reconciliation fix in the gateway/WebChat lifecycle path, not a provider-specific OpenRouter workaround. The fix must keep per-attempt fallback suppression intact while forcing true terminal failures to resolve the UI.
