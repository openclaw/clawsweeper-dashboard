## Summary

Keep open: current main still contains the source-level failure mode, and an open linked PR explicitly tracks the implementation candidate for this issue; the issue should not be closed before that PR lands or is replaced.

## What This Changes

WebChat should stop polling and surface the failed assistant turn live when a provider 429 or idle-timeout ends an assistant run before content.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79858, or an equivalent narrow gateway lifecycle fix, so failed no-content assistant runs emit a terminal WebChat update and reconcile the persisted failure without refresh.

## Reproduction Assessment

Yes, at source level: current main carries `skipChatErrorFinal` into the delayed lifecycle-error finalization path, and the current test asserts no chat error after the grace window for an active `chat.send` run without a chat link. I did not perform a live provider 429 rerun.

## Solution Assessment

Yes: the narrow maintainable solution is in the gateway/WebChat terminalization path, preserving retry-grace suppression during the retry window but guaranteeing a final error once the run is terminal. The open linked PR is the current implementation candidate.
