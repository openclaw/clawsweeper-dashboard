## Summary

Keep open: current main still has a source-level WebChat/gateway terminal-error gap, and the only linked fix PRs are closed unmerged. The issue has concrete logs/screenshots, a second report after v2026.5.12, and a recent code-path analysis pointing at the same live-state boundary.

## What This Changes

WebChat should stop the active polling indicator and show the persisted failed assistant turn when a provider 429 or idle-timeout fails before producing content.

## Best Possible Solution

Make provider-failure terminalization deterministic: WebChat should receive a terminal failure, clear the active run, and display or reload the persisted failure sentinel without relying on a manual refresh or provider-specific behavior.

## Reproduction Assessment

Yes, at source level: a send creates `chatRunId` with blank `chatStream`, current gateway code can suppress the delayed terminal error for active no-link `chat.send` runs, and the UI error branch does not append or reload the persisted failure turn. The reporter supplied logs/screenshots, and another user reported the same symptom after v2026.5.12.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the gateway/WebChat terminalization path, not in an OpenRouter-specific workaround. The fix should cover both terminal error delivery and the UI’s failed-turn reconciliation, with focused regression tests.
