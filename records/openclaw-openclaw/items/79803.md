## Summary

Keep open: current main still has a source-proven WebChat terminalization gap where a provider lifecycle error can be delayed, then suppressed after the grace timer, leaving the active UI run uncleared until history is reloaded.

## What This Changes

The issue asks WebChat to stop polling and surface the persisted failed assistant turn immediately after provider 429 or idle-timeout failures.

## Best Possible Solution

Fix the gateway/WebChat terminalization contract so a failed no-content assistant run always produces a terminal UI update or history reconciliation after the lifecycle grace window, with no provider-specific workaround.

## Reproduction Assessment

Yes, at source level: current main preserves `skipChatErrorFinal` through the delayed lifecycle-error timer and the existing test asserts that no chat error is emitted for an active chat.send run without a chat link. The reporter also supplied live gateway logs and before/after screenshots, but this review did not rerun a real provider 429.

## Solution Assessment

Yes: the narrow maintainable fix is in the gateway chat lifecycle grace path, either dropping or re-evaluating the stale suppression flag when the retry window expires. It should preserve the existing grace period and successful final reconciliation behavior.

Keep the fix in the generic gateway/WebChat lifecycle path, not in OpenRouter/provider-specific code. Add a single-line `CHANGELOG.md` fix entry because this is user-visible. Do not add config flags, provider-specific handling, broad registry caches, or unrelated WebChat refactors.
