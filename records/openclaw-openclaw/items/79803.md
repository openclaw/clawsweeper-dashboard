## Summary

Keep open: current main still carries the WebChat live-state gap, where delayed lifecycle errors for active `chat.send` runs without a chat link suppress the terminal chat error and leave the UI dependent on refresh/history reconciliation. The two related fix PRs were closed unmerged, and a later user comment reports the symptom after upgrading to `2026.5.12`.

## What This Changes

WebChat should stop polling and show the failed assistant turn when a provider 429 or idle-timeout ends an assistant run before content is produced.

## Best Possible Solution

Patch the gateway lifecycle finalization so terminal no-content failures for active `chat.send` runs emit a terminal chat error or otherwise force WebChat reconciliation after the retry grace expires, while preserving suppression for true per-attempt fallback errors and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main still carries a path where an active `chat.send` lifecycle error without a chat link schedules `skipChatErrorFinal`, and the adjacent test expects no terminal chat error after the grace timer. The reporter supplied live logs/screenshots, but this review did not rerun a live provider 429.

## Solution Assessment

Yes, the narrow maintainable direction is to fix gateway/WebChat terminalization rather than add provider-specific handling. The closed PR direction of dropping the stale suppressing option after the retry grace is a strong starting point, but it should be rebased onto current main and verified with focused tests and real behavior proof.
