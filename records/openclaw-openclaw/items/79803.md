## Summary

Keep open: current main and v2026.5.12 still preserve the delayed `skipChatErrorFinal` suppression path, and the adjacent regression test still expects no terminal WebChat error for the active no-link `chat.send` case. A newer user comment also reports the same failure after upgrading to 2026.5.12, so this remains an actionable WebChat/gateway bug.

## What This Changes

The issue asks WebChat to stop polling and surface the persisted failed assistant turn immediately when a provider 429 or idle timeout ends a run before content.

## Best Possible Solution

Emit a reliable terminal WebChat error after the lifecycle-error retry grace expires for failed `chat.send` runs, then let the existing UI clear the active run and reload the persisted failed assistant turn.

## Reproduction Assessment

Yes at source level: current main simulates the active no-link `chat.send` lifecycle-error path and asserts that no terminal `state: "error"` chat event is emitted after the grace timer. The reporter supplied live logs/screenshots, but I did not run a live provider 429 locally.

## Solution Assessment

Yes: the narrow maintainable solution is in the gateway lifecycle finalizer/test path, not an OpenRouter-specific workaround or a new UI polling mode. The closed PRs point at the right code direction, but a fresh focused repair with verifiable proof is needed.
