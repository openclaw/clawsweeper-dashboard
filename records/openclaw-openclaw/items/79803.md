## Summary

Keep open: current main still has source-backed WebChat/gateway gaps that can leave a failed no-content assistant turn hidden until history is reloaded, and both linked fix PRs were closed unmerged.

## What This Changes

WebChat should stop the live polling indicator and display the failed-assistant turn immediately when a provider 429 or idle timeout fails before producing content.

## Best Possible Solution

Make failed agent-run completion deterministically emit and reconcile a terminal WebChat error, then display the persisted failure sentinel without relying on a later refresh or lucky `session.message` ordering.

## Reproduction Assessment

Yes at source level: current main still has an active `chat.send` lifecycle-error path that suppresses the final chat error, and the UI terminal error branch does not append or force-load the persisted failure turn. The reporter also supplied live logs/screenshots, but I did not rerun a provider 429 locally.

## Solution Assessment

Yes: the narrow maintainable fix is in WebChat/gateway chat terminalization, not provider-specific retry policy. The earlier closed server-side PRs are useful context but incomplete unless the failed-turn display path is made deterministic too.
