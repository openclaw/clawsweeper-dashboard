## Summary

Keep open for maintainer review: current main does not have a process-persistent restart throttle before sidecar startup, and this PR is a focused attempt to address the linked gateway restart-loop class. I did not find a blocking correctness or security issue, but the full Ubuntu npm/systemd OOM loop was not reproduced, and the backoff policy is a gateway maintainer decision.

## What This Changes

Adds a gateway startup-throttle module, invokes it before sidecar startup, schedules a stable-start counter clear, and covers the sentinel/backoff behavior with tests and a changelog entry.

## Best Possible Solution

Land a maintainer-approved first-rung process-level startup throttle if the gateway owners accept the backoff policy, while keeping deeper OOM root-cause and graduated recovery work tracked separately.

## Reproduction Assessment

No for the full reporter scenario: I did not reproduce the Ubuntu npm/systemd OOM loop on current main. Source inspection does show current main lacks a persisted pre-sidecar process-restart throttle, and the linked reporter attachment supports the external-restart failure mode.

## Solution Assessment

Unclear but reasonable: this is a narrow mitigation at the right startup boundary and has focused coverage, but it does not identify or fix the OOMing sidecar itself. Maintainers should decide whether this policy belongs in core as the first recovery rung.
