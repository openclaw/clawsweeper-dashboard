## Summary

Keep open: current main still lacks a process-level startup throttle, the linked report has real systemd-oomd restart evidence, and this active PR provides a narrow tested mitigation with sufficient live-output proof and no blocking correctness or security finding. Maintainer review is still needed for the operational backoff tuning before merge.

## What This Changes

Adds a stateDir-backed gateway startup restart sentinel with exponential backoff before sidecar startup, plus regression tests and a changelog entry.

## Best Possible Solution

Land the narrow startup-throttle mitigation if maintainers accept the tuning, and keep the broader graduated recovery design tracked in https://github.com/openclaw/openclaw/issues/79165.

## Reproduction Assessment

No high-confidence local reproduction of the full OOM loop was established. The reporter attachment proves real systemd-oomd kills and restarts, and source inspection confirms current main has no process-level sentinel before sidecar loading.

## Solution Assessment

Yes, as a narrow mitigation: a persistent counter before sidecar startup covers kernel SIGKILL plus external respawn cases that in-process channel restart guards cannot observe. It should not be treated as the full crash-recovery ladder tracked separately in https://github.com/openclaw/openclaw/issues/79165.
