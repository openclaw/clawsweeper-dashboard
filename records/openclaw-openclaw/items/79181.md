## Summary

Keep open: this is a plausible gateway bug fix, but the patch can clear its restart counter at the same ~10 minute cadence the PR says the linked OOM loop used, so the backoff may never fire for the reported case.

## What This Changes

The PR adds a gateway startup throttle sentinel, wires it before sidecar startup, adds unit tests, and adds a changelog entry for the Ubuntu npm/systemd OOM restart-loop bug.

## Best Possible Solution

Keep the process-level sentinel approach, but make the counter-clear semantics last beyond the failure cadence the rapid window is meant to cover and add a regression test for a crash after the clear timer but still inside `RAPID_WINDOW_MS`.

## Reproduction Assessment

No for a high-confidence full reproduction of the original Ubuntu OOM loop; the reporter attachment and PR comments show a real environment but not enough to recreate it here. The PR-specific timer mismatch is source-reproducible from `STABLE_CLEAR_MS` and `scheduleStartupThrottleClear`.

## Solution Assessment

No. A startup sentinel is a reasonable narrow fix direction, but clearing `rapidCount` after 10 minutes can erase the count for the reported ~10 minute OOM cycle before the 30-minute rapid window can trigger backoff.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the restart count through the covered OOM window:** `src/gateway/server-startup-throttle.ts:8`
  - body: `STABLE_CLEAR_MS` is 10 minutes, but the PR discussion says the linked OOM restarts were about 10 minutes apart while the rapid window is 30 minutes. Once sidecars start, the scheduled clear writes `rapidCount: 0`; if the gateway is OOM-killed at or after that point, the next process start becomes count 1 and backoff never reaches the threshold. Please make the stable clear exceed the failure window, or clear only after a truly clean/stable period, and add a regression for a restart after the clear time but within `RAPID_WINDOW_MS`.
  - confidence: 0.89
