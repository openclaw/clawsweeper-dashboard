## Summary

Keep open: this is a recent active PR that adds a focused gateway startup throttle, and current main still lacks that process-level sentinel. I found no blocking correctness or security finding, but the full Ubuntu/npm OOM loop was not reproduced outside the reporter's environment.

## What This Changes

This PR adds a stateDir-backed gateway startup throttle before sidecar loading, covers the counter/backoff/reset behavior with tests, and adds a changelog entry.

## Best Possible Solution

Land the narrow startup-throttle mitigation if maintainers accept the operational tuning, then keep broader graduated recovery design tracked separately in https://github.com/openclaw/openclaw/issues/79165.

## Reproduction Assessment

No high-confidence local reproduction path was established. The reporter's sanitized log proves real systemd-oomd kills and restarts, and current main lacks a process-level sentinel, but the exact sidecar/config trigger was not reproduced.

## Solution Assessment

Yes, as a narrow mitigation: persisting a process-level counter before sidecars covers kernel SIGKILL plus external respawn cases that in-process channel health monitors cannot observe. It should not be treated as the full recovery ladder requested in https://github.com/openclaw/openclaw/issues/79165.
