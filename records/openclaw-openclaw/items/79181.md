## Summary

Keep open: this is a focused gateway bug-fix PR with supplied terminal proof and green CI, but the proposed stable-clear timing can still reset the crash counter at the same roughly 10-minute cadence described for the OOM loop.

## What This Changes

The PR adds a gateway startup throttle sentinel before sidecar startup, unit tests for the sentinel behavior, and a changelog entry for the Linux OOM restart-loop fix.

## Best Possible Solution

Keep the process-level sentinel approach, but make the stable reset safely outlive the reported OOM cadence or depend on a stronger healthy signal, with focused tests for a kill after the current clear boundary.

## Reproduction Assessment

No high-confidence live reproduction of the full Ubuntu OOM loop is available. The source gap is credible because a SIGKILL loses all in-process restart state, and current main has no process-level startup throttle before sidecars.

## Solution Assessment

No, not as currently implemented. The sentinel belongs in the right gateway startup boundary, but clearing the counter after exactly 10 minutes is too close to the reported roughly 10-minute OOM restart cadence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the stable clear past the OOM cadence:** `src/gateway/server-startup-throttle.ts:10`
  - body: The linked OOM loop is now described as restarts roughly 10 minutes apart, but this timer clears `rapidCount` after exactly 10 minutes once sidecars come up.
