## Summary

Keep open. Current main still documents and implements the Gateway as a single long-running control/process boundary with embedded agent execution; the recent comments add useful Phase 0/1 and multi-gateway operator context, but they do not replace the broader distributed runtime architecture RFC.

## What This Changes

The RFC requests splitting `openclaw gateway` into a control plane and separately managed per-agent runtimes for lifecycle isolation, scaling, and secret boundaries.

## Best Possible Solution

Keep this as the architecture umbrella while maintainers choose reviewed slices such as per-agent lifecycle, runtime registration/protocol, and secret/auth isolation before container or Kubernetes orchestration.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new runtime architecture. Source and docs inspection give high-confidence negative capability evidence that current main still uses a single Gateway process and embedded agent runner.

## Solution Assessment

Unclear until maintainers choose the architecture boundary. The narrowest maintainable path appears to be a reviewed Phase 0/1 slice around per-agent lifecycle, runtime registration, and auth/secret contracts before distributed container or Kubernetes backends.
