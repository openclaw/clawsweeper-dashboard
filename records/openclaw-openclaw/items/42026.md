## Summary

This should stay open: current main still treats the Gateway as the single long-running control plane and does not implement the proposed per-agent runtime/process boundary. The discussion is also recent and useful, especially the Phase 0/1 comments about per-agent lifecycle commands and the macOS multi-user workaround.

## What This Changes

The issue proposes splitting `openclaw gateway` into a lightweight control plane plus separate per-agent runtimes for embedded, local-split, and distributed deployments.

## Best Possible Solution

Keep this as the architecture discussion and let maintainers decide a first slice, likely Phase 0/1 per-agent lifecycle plus a versioned runtime protocol before distributed orchestration.

## Reproduction Assessment

Not applicable. This is an RFC for a new runtime architecture, and current-main source/docs show the requested runtime/control-plane split is not implemented.

## Solution Assessment

Unclear until maintainers choose the architecture boundary. The narrowest maintainable direction appears to be designing per-agent lifecycle and runtime registration first, then layering containers, Kubernetes, or microVM backends later.
