## Label Justifications

- `P2`: The issue is a significant architecture improvement request with real operator impact, but it is not a current core-runtime emergency.
- `impact:session-state`: The RFC is explicitly about isolating per-agent runtime state and avoiding shared in-flight session disruption.
- `impact:security`: The RFC explicitly proposes stronger per-agent process and secret isolation.
- `impact:message-loss`: The RFC calls out Gateway restarts and single-process failures dropping channel connections and in-flight sessions.

## Summary

Keep open: current main and the latest release still use a single Gateway process with embedded-agent drain/restart bookkeeping, and the narrower per-agent restart request was already closed as superseded by this RFC, so this remains the canonical product/design item.

## What This Changes

The issue proposes splitting `openclaw gateway` into a lightweight control plane and separate per-agent compute runtimes with embedded, local-split, and distributed deployment modes.

## Best Possible Solution

Keep this as the canonical maintainer design issue for deciding a narrow Phase 0/1 runtime-control-plane boundary, then split approved work into smaller protocol, lifecycle, security, docs, and validation issues.

## Reproduction Assessment

Not applicable: this is an RFC/feature request rather than a concrete current-main bug. Source and docs do verify that the requested distributed runtime and per-agent lifecycle surface do not exist today.

## Solution Assessment

No, not as a single automatic implementation item. The durable solution needs maintainer agreement on a scoped runtime/control-plane contract before code is split into smaller protocol, lifecycle, security, and deployment tasks.
