## Summary

Keep open: current main and v2026.6.10 still describe and implement a single Gateway/control-plane process with embedded or in-process runtime surfaces, while related items are narrower open design slices, concrete stall bugs, or closed satellites rather than a merged distributed runtime implementation.

## What This Changes

The issue requests a distributed agent runtime that splits Gateway control-plane duties from per-agent compute/runtime duties.

## Best Possible Solution

Keep this RFC open as the architecture umbrella, then decompose any maintainer-approved direction into narrower protocol, lifecycle, session-state, secret-isolation, and deployment milestones.

## Reproduction Assessment

Not applicable: this is an RFC/feature request rather than a concrete bug report. Source, docs, related-item, and release inspection confirm the requested distributed per-agent runtime split is not implemented on current main or v2026.6.10.

## Solution Assessment

Unclear as a final implementation: the proposed split is plausible and has real operator demand, but maintainers still need to choose the product boundary, trust model, protocol, and first milestone. A narrower per-agent lifecycle phase or the open execution-backend proposal may be the first implementation slice.
