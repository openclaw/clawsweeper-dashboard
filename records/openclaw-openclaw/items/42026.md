## Summary

Keep open: current main still documents and implements the Gateway as the single control-plane process with embedded/local agent execution, while this RFC remains a broad product and security architecture decision rather than an implemented cleanup target.

## What This Changes

The issue proposes splitting the Gateway control plane from per-agent compute runtimes so agents can run, restart, scale, and isolate resources independently.

## Best Possible Solution

Keep this as the broad canonical RFC, and split implementation into maintainer-approved milestones for worker/runtime protocol, trust boundaries, lifecycle commands, migration/compatibility, and deployment backend ownership.

## Reproduction Assessment

Not applicable as a feature RFC; the missing capability is source-confirmed by current docs and CLI/runtime code that still use a single Gateway-owned runtime. I did not run live multi-host proof in this read-only review.

## Solution Assessment

Unclear: the control-plane/runtime split is a plausible end state, but maintainers still need to choose the first milestone, protocol boundary, trust model, and relationship to the worker-runtime and subagent-backend slices.
