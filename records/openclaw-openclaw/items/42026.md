## Summary

Keep open. Current main has adjacent per-agent state/config, sandbox policy, and selectable embedded/CLI agent runtimes, but it still routes agent turns through the Gateway-side embedded path and exposes no Gateway-to-per-agent-runtime protocol or per-agent start/stop/restart lifecycle. The related comments usefully narrow a Phase 0 lifecycle slice, but this remains a broad runtime/protocol/security architecture item rather than a cleanup closure.

## What This Changes

The issue requests splitting the Gateway into a lightweight control plane and separately managed per-agent runtime processes, preserving embedded mode while adding local-split and distributed deployment modes.

## Best Possible Solution

Keep this as an architecture tracker and split implementation into reviewable milestones: preserve embedded mode, define the Gateway-to-runtime contract, implement local per-agent process lifecycle first, then add container/Kubernetes orchestration and secret/resource isolation after explicit threat-model review.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The capability gap is high-confidence because current docs and source still show Gateway-side embedded execution and no per-agent runtime lifecycle/protocol surface.

## Solution Assessment

Unclear as a direct implementation plan. The direction is plausible, but the maintainable path is a phased maintainer-approved protocol and security design rather than one broad autonomous patch.
