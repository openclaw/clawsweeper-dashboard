## Summary

Keep open: current main still documents and implements the Gateway as a single control-plane process with embedded/CLI agent runtimes and node peripherals, while the requested per-agent runtime split remains a broad architecture RFC that needs maintainer design decisions.

## What This Changes

The issue requests splitting `openclaw gateway` into a lightweight control plane plus separately managed per-agent runtime processes for isolation, scaling, and independent lifecycle control.

## Best Possible Solution

Keep this as an architecture RFC and break it into maintainer-approved follow-ups, starting with a scoped Phase 0/1 design for per-agent lifecycle commands and a versioned gateway-runtime protocol.

## Reproduction Assessment

Not applicable. This is a feature/RFC, and source/docs verification shows current main still lacks the requested separate per-agent runtime lifecycle and protocol.

## Solution Assessment

No, not as an automated cleanup or single fix PR. The maintainable path is maintainer architecture review plus narrower implementation issues for lifecycle, routing, runtime protocol, and deployment backends.
