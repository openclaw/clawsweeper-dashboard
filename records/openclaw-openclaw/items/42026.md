## Summary

Keep open: current main still documents and implements a single Gateway-centered runtime with embedded or ACP runtime choices, while related open or draft work covers narrower slices rather than replacing this broader distributed-runtime RFC.

## What This Changes

The issue requests an RFC-level split of the monolithic Gateway into a control plane plus separately deployable per-agent runtime processes for isolation, scaling, lifecycle, and security.

## Best Possible Solution

Keep this RFC open for maintainer architecture and security review, then split accepted work into narrower protocol, lifecycle, session-state, secret-isolation, worker-placement, and deployment milestones.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because distributed runtimes require explicit process, worker-auth, authorization, secret-scoping, channel/session routing, and revocation boundaries.

Concerns:

- **[high] Define runtime trust and secret boundaries:**
  - body: A distributed runtime protocol would need fail-closed worker identity, authorization, scoped secret delivery, audit, and revocation semantics before any runtime can safely execute outside the Gateway process.
  - confidence: 0.9
- **[medium] Preserve session and channel delivery ownership:**
  - body: Moving agent compute out of the Gateway changes who owns transcript writes, in-flight replies, retries, health, and recovery, so the design must prevent message loss or split-brain session state.
  - confidence: 0.86
