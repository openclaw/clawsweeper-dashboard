## Summary

Keep open: current main still implements and documents a Gateway-owned embedded agent runtime, while recent comments add active operator data for Phase 0/1 rather than making the RFC obsolete.

## What This Changes

Request to split `openclaw gateway` into a control-plane gateway plus per-agent runtime processes with local and distributed deployment modes, lifecycle commands, and runtime protocol work.

## Best Possible Solution

Keep this as an architecture RFC and split it into maintainer-approved phases: per-agent lifecycle/registry first, then a versioned runtime protocol, trust model, health/reconnect behavior, and deployment docs.

## Reproduction Assessment

Not applicable. This is an RFC/feature request, and source inspection of current main shows no first-class `openclaw-runtime`, local-split runtime protocol, or `agents start/stop/restart` behavior to exercise.

## Solution Assessment

Unclear because there is no patch to validate. The requested direction is plausible, but the maintainable path should be phased design work rather than one broad implementation PR.

## Risks / Open Questions

- The RFC spans process topology, runtime registration, reconnect semantics, health checks, secret routing, tool permissions, deployment, and migration, so a code-first change could lock in the wrong architecture.
- The security boundary for cross-process or cross-host runtimes is undefined; secrets, workspace access, runtime identity, and tool execution need explicit review before implementation.
- Current multi-Gateway workflows solve part of the operator pain, so maintainers should decide whether Phase 0 lifecycle commands are a separate smaller feature or part of the full distributed-runtime design.
