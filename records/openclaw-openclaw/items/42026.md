## Summary

Keep open: current main still documents a single Gateway process and dispatches agent turns through in-process command/embedded-harness paths, while the recent Phase 0/1 and operator comments make this an active architecture RFC rather than stale cleanup.

## What This Changes

The issue requests splitting `openclaw gateway` into a control plane and separate per-agent runtime compute processes with lifecycle, protocol, deployment, scaling, and isolation support.

## Best Possible Solution

Keep the RFC open and split it into maintainer-approved phases: per-agent lifecycle/registration first, then a versioned runtime protocol, secret isolation model, health/reconnect behavior, and deployment story.

## Reproduction Assessment

Not applicable. This is an architecture RFC/feature request, and source inspection shows no first-class `openclaw-runtime`, local-split mode, distributed runtime protocol, or per-agent lifecycle CLI to exercise.

## Solution Assessment

Unclear because there is no patch to validate. The requested direction is plausible, but the maintainable path is a phased design that separates per-agent lifecycle from the later distributed runtime and security protocol.
