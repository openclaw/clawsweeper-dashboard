## Label Justifications

- `P2`: This is a real multi-agent scaling and operability request, but it is broad architecture work rather than an urgent regression.
- `impact:session-state`: The requested split changes where session metadata, transcripts, active runs, and ownership state live.
- `impact:security`: The RFC explicitly targets per-agent process, tool, filesystem, and secret isolation across a new runtime boundary.
- `impact:message-loss`: The motivating failures include gateway restarts and routing gaps that can interrupt active channel messages.

## Summary

Keep open. Current main still implements gateway-owned embedded agent execution, while the issue discussion and related open work only narrow possible Phase 0/1 slices rather than resolving the distributed runtime/control-plane split.

## What This Changes

Request to split the gateway into a control plane plus separately deployable per-agent runtime processes with lifecycle, session, routing, and secret isolation.

## Best Possible Solution

Keep this as the distributed-runtime architecture umbrella while maintainers split focused milestones for local worker isolation, runtime registration/protocol, per-agent lifecycle, shared-channel routing, session/state migration, and secret/auth contracts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new runtime architecture. Source and docs give high-confidence negative capability evidence that current main still uses gateway-owned embedded agent execution for the central path.

## Solution Assessment

Unclear until maintainers choose the architecture boundary. The safer path is to review the existing worker-runtime experiment and split Phase 0/1 issues around lifecycle, runtime registration, session/channel ownership, and secret contracts before container or Kubernetes backends.
