## Summary

Keep open: current main still documents and implements a single embedded agent runtime per Gateway plus multiple-gateway workarounds, while this RFC asks for a new versioned control-plane/runtime split; the recent Phase 0/1 operator comments make it active architecture work, not stale cleanup.

## What This Changes

Request to split `openclaw gateway` into a lightweight control plane and separately managed per-agent runtimes with local-split and distributed deployment modes.

## Best Possible Solution

Keep this as an architecture RFC and split it into maintainer-approved phases: first per-agent lifecycle and routing semantics, then a versioned runtime protocol, isolation/security model, health/reconnect behavior, and deployment backends.

## Reproduction Assessment

Not applicable. This is a feature/RFC rather than a broken current behavior report, and source inspection shows the requested `openclaw-runtime`, local-split mode, distributed runtime protocol, and per-agent lifecycle commands are not current-main behavior to reproduce.

## Solution Assessment

Unclear because there is no patch to validate. The direction is plausible, but the maintainable solution should be phased and versioned instead of implemented as one broad architecture change.

## Risks / Open Questions

- The RFC spans protocol, process lifecycle, secret boundaries, health/reconnect semantics, config migration, deployment packaging, and channel routing, so a one-shot automated fix would likely encode product and security policy too early.
- Existing multiple-Gateway setups and `agents.list` routing solve part of the operator pain but do not answer the shared inbound channel, runtime registration, and orchestration questions raised by the comments.
- The overlapping open K8s worker-pairing and gateway-stall reports should be coordinated so this RFC becomes the architecture umbrella rather than competing duplicate implementation tracks.
