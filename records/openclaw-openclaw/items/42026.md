- `P2`: This is meaningful multi-agent scaling and operability work, but it is broad architecture work rather than an urgent regression.
- `impact:session-state`: The requested control-plane/runtime split changes where session metadata, transcripts, active runs, and runtime ownership live.

- `impact:message-loss`: The motivating failures include Gateway restarts and routing gaps that can interrupt active channel messages.

## Summary

Keep open. Current main and the latest release still document and implement gateway-owned embedded agent execution, while the live discussion and related open issues are narrower milestones rather than a replacement for this distributed-runtime RFC.

## What This Changes

The issue requests splitting the gateway into a control-plane process and separate per-agent runtimes for isolation, scaling, lifecycle control, and deployment flexibility.

## Best Possible Solution

Keep this as the distributed-runtime architecture umbrella and split maintainer-approved milestones for runtime registration/protocol, per-agent lifecycle, session/channel routing, secret/auth isolation, local worker mode, and deployment backends.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new runtime architecture. Source, docs, latest-release, and related-item checks give high-confidence negative capability evidence that current main still uses gateway-owned embedded agent execution for the central path.

## Solution Assessment

Unclear until maintainers choose the architecture boundary. The RFC is directionally plausible, but the maintainable path is to split Phase 0/1 milestones around lifecycle, runtime registration, session/channel ownership, and secret contracts before container or Kubernetes backends.
