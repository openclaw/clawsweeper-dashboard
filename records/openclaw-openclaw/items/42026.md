## Summary

Keep open. Current main still documents and implements the Gateway as the single long-running process boundary with embedded agent execution; the newer narrower per-agent restart issue is useful Phase 0 input, but it does not replace this broader control-plane/runtime architecture RFC.

## What This Changes

The issue requests splitting `openclaw gateway` into a control plane plus per-agent runtime processes for isolation, lifecycle control, scaling, and deployment flexibility.

## Best Possible Solution

Keep this as the architecture umbrella, then let maintainers choose smaller first slices such as per-agent lifecycle, runtime registration/protocol, and explicit secret/auth isolation before container or Kubernetes orchestration backends.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new runtime architecture. Current-main source and docs provide high-confidence negative capability evidence: Gateway remains one process and agent turns still flow through `runEmbeddedPiAgent()`.

## Solution Assessment

Unclear until maintainers choose the architecture boundary. The narrowest maintainable path appears to be designing per-agent lifecycle and runtime registration first, then layering distributed containers, Kubernetes, or microVM backends later.
