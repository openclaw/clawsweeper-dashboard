## Summary

Keep open: current main and v2026.6.10 still document and implement a single gateway-centered runtime, with no per-agent runtime process, runtime registration protocol, distributed worker deployment model, or per-agent lifecycle CLI; the remaining work is broad security-sensitive product architecture rather than cleanup.

## What This Changes

Request to split `openclaw gateway` into a control plane plus per-agent runtime processes or containers with embedded, local-split, and distributed deployment modes.

## Best Possible Solution

Keep this as the umbrella architecture RFC, then promote maintainer-approved Phase 0/1 milestones for per-agent lifecycle, runtime protocol, security boundaries, and upgrade behavior before implementation.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a bug report; source and release inspection confirm the requested runtime split is absent.

## Solution Assessment

Unclear as a final implementation: local-split and per-agent lifecycle are plausible first milestones, but protocol, security model, config, session routing, and upgrade boundaries need maintainer design approval.

## Security Review

Summary: The issue is security-sensitive because it proposes new process, filesystem, credential, network, and runtime trust boundaries; no patch was reviewed.

Concerns:

- **[high] Define runtime authentication and authorization:**
  - body: A gateway-to-runtime protocol must specify fail-closed registration, agent identity binding, revocation, audit, and authorization before remote or container runtimes can execute trusted work.
  - confidence: 0.9
