- `impact:security`: The proposal directly concerns process isolation, secret scoping, filesystem boundaries, and authorization between agent runtimes and the control plane.

## Summary

Current main still uses one Gateway-owned messaging/control plane with agent turns executed through the in-process embedded runner, and nodes remain peripheral tool executors rather than agent runtimes. The proposal is therefore still necessary as the canonical architecture discussion, but it is a large security-, delivery-, and state-sensitive product program requiring explicit maintainer design rather than an autonomous fix.

## What This Changes

Define embedded, local-split, and distributed deployment modes that keep channel routing in a control plane while moving each agent’s model loop, tools, workspace, memory, and conversation state into an independently managed runtime.

## Best Possible Solution

Keep this as the canonical architecture decision record, first specifying a minimal authenticated local-split protocol with single-owner session and delivery semantics while preserving embedded mode, then prove lifecycle and failure recovery before adding containers, Kubernetes, or dynamic scaling.

## Reproduction Assessment

Not applicable: this is an architecture RFC, not a bounded report of broken behavior; current source and operator comments do establish the existing single-process limitation.

## Solution Assessment

Unclear as a complete solution: separating control and compute is directionally credible, but the proposed full program is too broad until maintainers define protocol, session ownership, security, and delivery guarantees; a local-split proof should precede orchestration backends.

## Risks / Open Questions

- A runtime protocol without durable ownership and acknowledgement rules could duplicate, lose, or misroute inbound/outbound messages during reconnects and retries.
- Splitting session metadata from conversation state creates split-brain, cancellation, compaction, and migration risks unless one canonical database owner and transaction boundary are defined.
- Per-runtime credentials, plugin hooks, MCP processes, workspace access, node invocation, and authorization materially change the security boundary and need a threat model before implementation.
- The proposed CLI, config, runtime image, protocol, and deployment modes are public compatibility surfaces; implementing all phases together would be difficult to review, migrate, or safely roll back.
