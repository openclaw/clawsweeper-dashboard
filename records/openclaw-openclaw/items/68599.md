## Summary

Keep open: the diagnostic idea is useful, but the patch currently keys the new warning off a cumulative overflow counter, so later unrelated overflows can receive a misleading post-compaction hint; the external PR also still lacks real behavior proof.

## What This Changes

The PR adds a `[context-overflow-hint]` warning, compaction troubleshooting docs, a changelog note, and two overflow-loop unit tests around `agents.defaults.contextLimits.toolResultMaxChars`.

## Best Possible Solution

Keep the operator hint and docs, but gate the warning on a dedicated immediate-post-compaction retry marker that is cleared by any intervening non-overflow retry path, with a regression for that sequence.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a compacted overflow can be followed by planning-only, reasoning-only, empty-response, or silent-error retries before a later overflow sees the stale nonzero counter. I did not run tests because this is a read-only review.

## Solution Assessment

No. The hint is a reasonable direction, but `overflowCompactionAttempts > 0` is not the narrowest maintainable signal for an overflow that immediately persisted after compaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Gate hint to direct post-compaction retries:** `src/agents/pi-embedded-runner/run.ts:1169-1171`
