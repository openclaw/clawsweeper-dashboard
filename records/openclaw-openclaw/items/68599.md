## Summary

Keep open. The PR addresses a real operator-observability gap, but the proposed warning is keyed to a cumulative run-loop counter, so it can emit after a later independent overflow rather than only when an overflow directly persists after compaction.

## What This Changes

The PR adds a `[context-overflow-hint]` warning, compaction troubleshooting docs, a changelog entry, and two overflow-loop regression tests for the `toolResultMaxChars` hint.

## Best Possible Solution

Keep the operator hint, but drive it from a separate immediate-post-compaction retry marker that is set only when compaction directly schedules a retry and cleared by any intervening non-overflow attempt or other retry path. Preserve the cumulative overflow-attempt counter as the safety guard, and add a regression for a later independent overflow that must not log the hint.

## Reproduction Assessment

Yes for the PR regression. The existing overflow-loop harness can mock overflow, successful compaction, an intervening non-overflow retry path such as planning-only or empty-response retry, then a later overflow; the current PR condition would still see `overflowCompactionAttempts > 0`.

## Solution Assessment

No. The additive log and docs direction are useful, but `overflowCompactionAttempts > 0` is not the narrowest maintainable signal for “immediately persisted after compaction”; a separate direct-retry flag is safer and preserves the existing max-attempt guard.
