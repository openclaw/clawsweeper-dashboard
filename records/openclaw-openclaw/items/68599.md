## Summary

Keep open: the diagnostic idea is useful, but the current patch still uses a cumulative overflow-compaction counter that can emit a misleading hint on a later independent overflow, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a context-overflow warning, compaction troubleshooting docs/changelog text, and two overflow-loop unit tests for overflow after compaction.

## Best Possible Solution

Keep the additive hint/docs, but gate the warning on a dedicated immediate-post-compaction retry marker that is cleared by intervening non-overflow retry paths, then show redacted real-run proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the run-scoped counter is incremented during overflow recovery and later reasoning-only or empty-response retries can continue without clearing it. I did not run tests because this review is read-only.

## Solution Assessment

No. The operator hint is a good direction, but `overflowCompactionAttempts > 0` is not the narrowest maintainable signal for an overflow that directly persisted after compaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate overflow hint to immediate post-compaction retries:** `src/agents/pi-embedded-runner/run.ts:1169-1171`
  - body: `overflowCompactionAttempts` is cumulative for the whole run loop, so after any compaction a later independent overflow after planning-only, reasoning-only, or empty-response retries can emit `[context-overflow-hint]` even though it did not directly persist after compaction. Track a dedicated post-compaction retry marker and clear it on intervening non-overflow retry paths before warning.
  - confidence: 0.9
