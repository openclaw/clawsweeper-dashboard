## Summary

Keep open. Current main still has the reported failure shape in the memory chunker and embedding path, and open PR #65783 is already paired to this issue with closing syntax, so the issue should remain open until that PR or an equivalent replacement lands.

## What This Changes

The issue asks OpenClaw to stop memory indexing from permanently stalling when `chunkMarkdown` splits an emoji or other supplementary-plane character into lone UTF-16 surrogate halves and those halves reach the embedding provider.

## Best Possible Solution

Land or replace #65783 with the same narrow two-layer fix: preserve surrogate pairs in the coarse chunk boundary, sanitize lone surrogates before embedding provider calls, and keep focused regression tests for both paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `tokens=10`, `maxChars` is 40, and `39 ASCII + 🌸 + 39 ASCII` makes the current outer loop split between `\uD83C` and `\uDF38`.

## Solution Assessment

Yes. The linked fix direction is the narrowest maintainable solution because it removes the chunker root cause and adds an embed-boundary guard against future invalid strings; retry-policy changes alone would still send invalid payloads.
