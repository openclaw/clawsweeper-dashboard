## Summary

Keep open: current main and the latest release still leave the Discord outbound upload path without a MIME type, and the open linked fix PR at https://github.com/openclaw/openclaw/pull/41265 is still unresolved and conflicting.

## What This Changes

The issue reports that Discord outbound media loses MIME content type and that WebP or animated image optimization can silently change format or strip animation frames.

## Best Possible Solution

Land the linked PR or a narrow equivalent that forwards contentType through every Discord upload path and preserves WebP/animated images under caps, returning clear cap errors instead of lossy optimization.

## Reproduction Assessment

Yes, source reproduction is high confidence: current main passes only media.buffer into an untyped Blob for Discord uploads while the REST body layer only preserves MIME type when file.contentType is supplied. I did not run a live Discord bot upload in this read-only review.

## Solution Assessment

Yes, the right repair is in the Discord upload call sites plus the shared web-media optimizer boundary. The best next step is to update/review the existing linked PR rather than start another fix branch while that implementation candidate is open.
