## Label Justifications

- `P2`: This is a real media fidelity bug with limited Discord/shared-media blast radius and an open implementation path.
- `impact:data-loss`: Animated frames and original image format can be silently lost when WebP media is optimized into still PNG/JPEG output.
- `impact:message-loss`: Discord attachments can be delivered with missing or generic MIME metadata, which can cause recipients or bots to lose the intended media interpretation.

## Summary

Keep open: current main and the latest release still show the Discord upload and shared media paths described in the report, and https://github.com/openclaw/openclaw/pull/41265 is an open closing PR that has not landed.

## What This Changes

Discord media sends should preserve upload MIME content type and avoid converting WebP or animated image uploads in a way that changes format or drops frames.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/41265 or an equivalent current-main patch that threads MIME metadata through standard, component, and native Discord upload paths while preserving WebP/animated media within existing size limits.

## Reproduction Assessment

Yes. The current source path is high-confidence: Discord upload builders still construct typeless blobs, and WebP/non-GIF images still reach an optimizer path that emits PNG or JPEG.

## Solution Assessment

Yes in direction, but not complete yet. The maintainable fix is to thread existing MIME metadata through the Discord payload contract and preserve animated/WebP bytes without weakening size limits; the open linked PR still needs repair/proof before it can close this issue.
