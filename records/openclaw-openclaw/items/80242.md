## Summary

Keep open: the reported parent-session misroute is source-reproducible and the PR fixes one plausible routing order, but the latest patch still broadens the spawn context key in a way that can change sandbox policy checks, and the PR still lacks after-fix real Telegram or equivalent runtime proof. The author removed the unrelated mediated-delivery change and left that behavior to the separate related PR, which is the right split.

## What This Changes

The PR changes `sessions_spawn` construction to prefer `runSessionKey`, reorders subagent completion-origin binding lookup to check requester bindings before child bindings, and adds focused regression tests.

## Best Possible Solution

Split requester/completion ownership from sandbox and policy context, keep the requester-first completion-origin behavior if the focused tests prove it, and require redacted live Telegram or equivalent runtime proof before merge.

## Reproduction Assessment

Yes, at source level: current main passes the sandbox/channel peer key into `sessions_spawn` and checks the child binding before the requester binding, which matches the related issue’s reported wrong-recipient path. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as written: requester-first binding lookup is a reasonable part of the fix, but replacing the whole spawn context key with `runSessionKey` is broader than completion ownership and affects sandbox checks. A separate completion requester key or policy-key split is the safer direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep sandbox policy separate from requester ownership:** `src/agents/openclaw-tools.ts:408`
  - body: This passes `runSessionKey` as the full `sessions_spawn` context key. `spawnSubagentDirect` derives the requester key from `ctx.agentSessionKey` and uses it for sandbox runtime checks before allowing an unsandboxed child, so an embedded Telegram run whose sandbox key differs from `agent:main:main` can be evaluated as the unsandboxed main session. Split completion requester ownership from the policy/runtime session key instead.
  - confidence: 0.86
