## Summary

Keep open: the reported routing bug is source-reproducible on current main and this PR is aimed at the right code path, but the patch is not ready because it lacks after-fix real Telegram/runtime proof and the `runSessionKey` swap conflates requester ownership with sandbox/policy evaluation.

## What This Changes

The PR changes `sessions_spawn` construction to prefer `runSessionKey`, changes subagent completion routing to check requester bindings before child bindings, and adds focused regression tests.

## Best Possible Solution

Separate requester ownership from runtime policy context so subagent registration and completion delivery use the originating parent session while sandbox and spawn policy still evaluate against the sandbox/policy session.

## Reproduction Assessment

Yes, at source level: current main passes the sandbox/channel peer key into `sessions_spawn` and checks child bindings before requester bindings. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as written: requester-binding priority is the right direction, but replacing the whole spawn context key with `runSessionKey` is broader than requester ownership and can change sandbox/policy behavior. A separate requester-owner field or policy-key split is the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Keep sandbox policy separate from requester ownership:** `src/agents/openclaw-tools.ts:408`
  - body: This changes the whole `sessions_spawn` context key to `runSessionKey`. In embedded runs, `runSessionKey` is set only when the sandbox/policy key differs from the live session key, while `spawnSubagentDirect` uses `ctx.agentSessionKey` to decide whether a requester is sandboxed and should be barred from spawning an unsandboxed child. With a sandboxed channel policy key and an unsandboxed `agent:main:main` run key, this bypasses the inherited sandbox guard; please split requester ownership from the policy/runtime key instead.
  - confidence: 0.82
