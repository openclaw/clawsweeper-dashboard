## Summary

Keep open: the requester-session routing fix is plausible against current main, but the branch also carries an unrelated mediated-delivery change that can break group/channel subagent completions without message-tool guidance, and the external PR still lacks after-fix real Telegram behavior proof.

## What This Changes

The PR changes sessions_spawn requester-key wiring, subagent completion binding precedence, adds subagent_announce to the mediated completion source set, and adds focused regression tests.

## Best Possible Solution

Land a focused requester-session routing fix after dropping or completing the mediated-delivery change, then add redacted live Telegram proof that the originating parent session observes the announce.

## Reproduction Assessment

Yes, source-level: current main passes the sandbox/peer key into sessions_spawn ownership and resolves child bindings before requester bindings, matching the linked Telegram misroute report. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No as submitted: the runSessionKey and requester-first routing changes are likely the narrow fix, but the extra mediated-delivery change needs to be removed or paired with message-tool-only guidance, and merge still needs real behavior proof.

## Review Findings
