## Summary

Keep open: the PR addresses a plausible source-reproducible subagent completion routing bug, but the current diff still conflates completion ownership with sandbox/policy context and lacks required real Telegram or equivalent runtime proof.

## What This Changes

This PR changes subagent spawn tool construction to prefer `runSessionKey`, changes completion-origin resolution to check requester bindings before child bindings, and adds focused regression tests.

## Best Possible Solution

Split completion requester ownership from sandbox/policy context, keep sandbox checks keyed to the sandbox session, keep the requester-first binding direction, and verify with focused tests plus redacted live Telegram or equivalent runtime proof.

## Reproduction Assessment

Yes, at source level: current main passes the sandbox/channel peer key into `sessions_spawn` and checks child bindings before requester bindings. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as written: requester-first binding lookup is the right direction, but replacing the whole spawn context key with `runSessionKey` is broader than completion ownership. A separate owner key or policy-key split is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep sandbox policy separate from requester ownership:** `src/agents/openclaw-tools.ts:408`
  - body: This passes `runSessionKey` as the full `sessions_spawn` context key. `spawnSubagentDirect` derives the requester key from `ctx.agentSessionKey` and uses it for sandbox runtime checks before allowing an unsandboxed child, so an embedded Telegram run whose sandbox key differs from `agent:main:main` can be evaluated as the unsandboxed main session. Please split completion requester ownership from the policy/runtime session key instead.
  - confidence: 0.86
