## Summary

Keep open: this targets a valid source-reproducible subagent announce routing bug, but the current diff also changes spawn policy/sandbox context and still lacks required after-fix real Telegram proof.

## What This Changes

The PR changes `sessions_spawn` construction to prefer `runSessionKey`, resolves subagent completion bindings requester-first, and adds focused regression tests.

## Best Possible Solution

Split requester ownership from sandbox/policy identity so registration and completion delivery use the originating parent session while spawn policy, sandbox inheritance, depth limits, and allow rules continue to evaluate against the sandbox/policy session.

## Reproduction Assessment

Yes, at source level: current main passes the sandbox/channel peer key into `sessions_spawn` and resolves child bindings before requester bindings. The linked issue has field evidence from v2026.5.7, but I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as written. Requester-first completion routing is the right direction, but replacing the whole spawn context key with `runSessionKey` is broader than requester ownership and should be replaced by a narrow owner/policy split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep sandbox policy separate from requester ownership:** `src/agents/openclaw-tools.ts:408`
  - body: This changes the entire `sessions_spawn` context key to `runSessionKey`. `spawnSubagentDirect` uses `ctx.agentSessionKey` for requester identity, depth/active-child accounting, target policy, and the sandboxed-requester guard, so a channel-sandboxed embedded run can be evaluated as the unsandboxed parent session; please split requester completion ownership from the policy/runtime key instead.
  - confidence: 0.84

## Security Review
