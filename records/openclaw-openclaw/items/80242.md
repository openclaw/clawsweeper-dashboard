## Summary

Keep open: the PR targets a real source-reproducible requester-session routing bug and the code direction looks sound, but it is an external Telegram-visible fix without after-fix real behavior proof; the only failing check is the proof gate.

## What This Changes

The branch passes `runSessionKey` into `sessions_spawn` requester wiring, prioritizes requester bindings before child bindings, treats `subagent_announce` as agent-mediated completion, and adds routing regression tests.

## Best Possible Solution

Land the requester-session routing fix after redacted live Telegram proof shows completions return to the originating parent session, and reconcile the overlapping mediated-delivery hunk with https://github.com/openclaw/openclaw/pull/80223 if that lands first.

## Reproduction Assessment

No live current-main reproduction was run in this read-only review. Source inspection gives a high-confidence path: embedded runs can build tools with the Telegram sandbox peer key, `sessions_spawn` records that key as the requester, and completion origin currently checks child bindings before requester bindings.

## Solution Assessment

Yes for the central bug: using the already-threaded `runSessionKey` for spawn ownership and checking requester bindings before child fallback is the narrow maintainable fix. The only code-scope caveat is the overlapping `subagent_announce` mediated-delivery hunk, which should be reconciled with https://github.com/openclaw/openclaw/pull/80223 if needed.
