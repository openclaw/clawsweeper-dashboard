## Summary

Keep open: current main and the latest release still do not provide a persistent config default for the per-response `/usage` footer, and the issue is already paired with open implementation work that needs maintainer review before the issue can close.

## What This Changes

Request to add a config-level default for the `/usage` footer mode so new sessions can inherit `off`, `tokens`, or `full` without running `/usage` in each session.

## Best Possible Solution

Keep this issue open as the canonical tracker until maintainers choose the stable config API and land or reject an implementation that adds the default, preserves explicit session overrides, updates schema/help/docs, and covers reply, gateway, and TUI behavior.

## Reproduction Assessment

Yes, at source level: current main has no config key for a usage-footer default and the reply runner falls back only to session `responseUsage`. I did not run a live chat path because this is a feature gap rather than a failing existing contract.

## Solution Assessment

Unclear. A persistent default is the right product area to consider, but maintainers still need to choose the public config owner and explicit-off versus inherit semantics before an implementation is merge-ready.
