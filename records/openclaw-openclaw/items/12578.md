## Summary

Keep open: current main still does not implement a persistent response-usage footer default. The schema, types, docs, reply runner, slash command, TUI, and gateway patch paths still treat `responseUsage` as session-local, and the issue is already linked by an open closing PR candidate.

## What This Changes

The issue requests a persistent config default for per-response usage footer display so new sessions can inherit off, tokens, or full usage output.

## Best Possible Solution

Add one stable public `agents.defaults` usage-footer key, wire reply/TUI/gateway session resolution so unset inherits while explicit off remains explicit, update schema/types/generated config help/docs, and land focused regression coverage. The open implementation candidates should be reviewed or closed before a separate repair lane starts.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a configured `agents.defaults.responseUsage`-style key is not accepted by the strict defaults schema, and reply formatting only reads session-local `responseUsage`.

## Solution Assessment

Yes for the issue direction: a typed inherited default under `agents.defaults` is the narrowest maintainable core fix. The exact public key and inheritance semantics should be chosen before accepting the broader footer PR.
