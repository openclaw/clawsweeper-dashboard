## What This Changes

The issue requests a persistent config default for new sessions so `/usage` token or full footers do not need to be enabled manually per session.

## Best Possible Solution

Review, narrow, land, or replace the linked implementation with one validated `agents.defaults.responseUsage` contract that updates schema/help/docs and preserves per-session overrides including explicit `off`.

## Reproduction Assessment

Yes, for the source-level capability gap: current main has no inherited usage-footer key under `agents.defaults`, and runtime/docs still treat `responseUsage` as session-local. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Unclear which exact public key and inheritance semantics are best until maintainers decide the config contract. The maintainable path is a single canonical default with explicit override semantics, not another parallel session-state path.
