## Summary

Keep open: current main still lacks a persistent `agents.defaults` usage-footer default, and an open PR already references this issue with closing syntax, so the issue should stay open until an accepted implementation lands or maintainers reject the feature.

## What This Changes

Add a config default for `/usage off|tokens|full` so new sessions can inherit per-response usage footer behavior without running `/usage` manually.

## Best Possible Solution

Land or replace one accepted typed `agents.defaults` usage-footer contract with schema/types/help/docs, runtime inheritance, explicit-off versus clear-to-inherit behavior, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection for the capability gap: current main has no `agents.defaults` usage-footer key, and the reply, command, and gateway paths still use session-local `responseUsage`. No live channel run is needed to verify that the requested config surface is absent.

## Solution Assessment

Yes, a typed inherited agent-defaults key is the narrow maintainable direction, but the exact public key and override semantics need maintainer choice. The best path is to review and land or replace the existing open implementation PR rather than close this issue early.
