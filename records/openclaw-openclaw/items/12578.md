## Summary

Keep open: current main still lacks a persistent agent-defaults usage-footer setting, and an open closing PR is already the implementation path rather than a reason to close the issue before merge.

## What This Changes

The issue requests a typed config default for the `/usage` reply-footer mode so new sessions can inherit `off`, `tokens`, or `full` without running `/usage` manually.

## Best Possible Solution

Land one accepted typed `agents.defaults` usage-footer contract with schema, types, generated help/docs, runtime inheritance, explicit-off semantics, and focused regression coverage, or close the linked implementation PRs if maintainers reject the feature.

## Reproduction Assessment

Yes for the capability gap by source inspection: current main has no `agents.defaults` usage-footer key, and the reply, command, and gateway paths still rely on session-local `responseUsage`. No live run is needed to see that the requested config surface is absent.

## Solution Assessment

Yes, a typed inherited agent-defaults key is the narrow maintainable direction, but the exact public key and override semantics need maintainer choice. The best path is to review and land or replace the existing open implementation candidate rather than close this issue early.
