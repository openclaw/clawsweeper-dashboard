## Summary

Keep this issue open: current main and the latest release still only support session-local `responseUsage`, while open implementation candidates are still awaiting maintainer decisions on the public config key and explicit-off versus inherit semantics.

## What This Changes

The issue requests a persistent config default for response usage footers so new sessions can start with `off`, `tokens`, or `full` without running `/usage` each time.

## Best Possible Solution

Keep this issue open until maintainers choose the stable config surface and land a focused implementation with schema/help/docs updates plus regression coverage for inherited default, explicit off, clear-to-inherit, and per-session override behavior.

## Reproduction Assessment

Yes, at source level: current main lacks any usage-footer default in the agent defaults schema/types, and reply rendering reads only session `responseUsage`. The existing `/usage` and gateway patch paths are enough to verify the missing inherited-default behavior without a live channel run.

## Solution Assessment

The requested direction is plausible, but the best fix is not yet settled. Maintainers should first choose the public config owner and explicit-off semantics, then land one focused implementation rather than leaving competing fallback shapes open.
