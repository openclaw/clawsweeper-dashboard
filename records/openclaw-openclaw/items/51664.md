## Summary

Keep open: current main still emits WebChat code-block Copy buttons but handles them with a direct `navigator.clipboard.writeText` call and an empty rejection handler, while the two related fallback PRs were closed unmerged.

## What This Changes

The issue requests that WebChat code-block Copy work in Microsoft Edge when the Clipboard API write path fails.

## Best Possible Solution

Add a small shared Control UI clipboard helper that tries Clipboard API first, falls back to a hidden textarea plus `document.execCommand('copy')`, and wires WebChat code-block Copy through it with focused regression coverage.

## Reproduction Assessment

Yes, source-level. When `navigator.clipboard.writeText` rejects on the `.code-block-copy` click path, current main ignores the rejection and leaves the button as a no-op, matching the reported Edge failure mode.

## Solution Assessment

Yes. A narrow shared Control UI clipboard helper is the best maintainable fix because the failure is in browser copy behavior, not markdown parsing, gateway protocol, or a product setting.
