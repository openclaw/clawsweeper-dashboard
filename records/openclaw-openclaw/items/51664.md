## Summary

Keep open: current main still routes WebChat code-block Copy through a direct Clipboard API write with an empty rejection handler, while the renderer still emits the affected buttons and the related fallback PRs were closed unmerged.

## What This Changes

Fix WebChat code-block Copy in Microsoft Edge by adding a fallback when Clipboard API writes fail.

## Best Possible Solution

Add a small shared Control UI clipboard helper that tries `navigator.clipboard.writeText`, falls back to a temporary textarea plus `document.execCommand('copy')`, returns a success boolean, and is wired into WebChat code-block Copy with focused regression tests.

## Reproduction Assessment

Yes, source-level. When `navigator.clipboard.writeText` rejects on the `.code-block-copy` path, current main has no fallback or user-visible failure handling, matching the reported Edge behavior.

## Solution Assessment

Yes. A narrow Control UI clipboard helper is the best maintainable fix because the failure is in the browser UI copy path, not in markdown rendering, gateway protocol, or a product setting.
