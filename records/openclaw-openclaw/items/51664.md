## Summary

Keep open: current main and the latest release still route WebChat code-block Copy through a direct Clipboard API call with an ignored rejection path, while the two Edge fallback PRs linked from the discussion were closed unmerged.

## What This Changes

Reports that WebChat code-block Copy in Microsoft Edge does not write code to the clipboard, forcing users to select the code manually.

## Best Possible Solution

Add a shared Control UI clipboard helper that tries the Clipboard API first, falls back to a hidden-textarea copy path, and wire WebChat code-block Copy through it with focused regression coverage.

## Reproduction Assessment

Yes, at source level: forcing `navigator.clipboard.writeText` to reject on the `.code-block-copy` path exercises the current no-op rejection handler. A live Edge smoke was not run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is a shared Control UI clipboard helper plus regression coverage, not a markdown renderer redesign, gateway change, or new configuration option.

## Repair Work Prompt

Fix the WebChat code-block Copy button failure when `navigator.clipboard.writeText` rejects, especially in Microsoft Edge. Add or reuse a small Control UI clipboard helper that first tries `navigator.clipboard.writeText(text)` and then falls back to a hidden textarea plus `document.execCommand('copy')`, cleaning up the DOM in all cases. Wire `ui/src/ui/views/chat.ts` code-block Copy through that helper and consider reusing it from `ui/src/ui/chat/copy-as-markdown.ts` if it keeps the API simple. Preserve the current markdown DOM shape, i18n labels, sidebar handler binding, and any active changes from https://github.com/openclaw/openclaw/pull/73633 if that lands first; do not redesign markdown rendering or add config. Add focused regression coverage for rejected Clipboard API fallback and keep existing rendering tests green. Add a single user-facing Fixes changelog entry if the repair PR changes behavior.
