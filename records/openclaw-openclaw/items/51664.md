## Summary

This should stay open: current main and the latest release still route WebChat code-block Copy through `navigator.clipboard.writeText` with no fallback and a swallowed rejection, while the two Edge-specific fallback PRs are closed unmerged. The open related PR targets a different payload-selection bug, so it is useful context but not a replacement for this Edge clipboard failure.

## What This Changes

The issue reports that the Control UI/WebChat code-block Copy button silently fails in Microsoft Edge on Windows 11.

## Best Possible Solution

Add a narrow, tested Control UI clipboard helper that tries `navigator.clipboard.writeText` first and falls back when the write is rejected, then use it from the WebChat code-block Copy handler without changing config, docs, or unrelated copy surfaces unless reuse is smaller.

## Reproduction Assessment

No live high-confidence Edge reproduction was run in this read-only review. Source inspection is still strong: current main and `v2026.6.6` use the Clipboard API directly and swallow rejection in the code-block Copy handler, matching the reported silent failure mode.

## Solution Assessment

Yes, the best repair appears to be a focused clipboard-write helper/fallback used by the existing code-block Copy handler. The open rendered-text PR is a good adjacent fix for copied payload selection, but it does not solve this browser API rejection path.
