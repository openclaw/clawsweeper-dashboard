## Summary

Keep open: current main and the latest release still route WebChat code-block Copy through `navigator.clipboard.writeText` with an ignored rejection path, while the two Edge fallback PRs were closed unmerged and the newer open textContent PR addresses a different payload-selection bug.

## What This Changes

The issue requests that WebChat code-block Copy work in Microsoft Edge instead of silently failing when the Clipboard API write path does not copy text.

## Best Possible Solution

Add a shared Control UI clipboard helper that tries the Clipboard API first, falls back to a hidden-textarea copy path, and wire WebChat code-block Copy through it with focused regression coverage.

## Reproduction Assessment

Yes, at source level: forcing `navigator.clipboard.writeText` to reject on the `.code-block-copy` path exercises the current no-op rejection handler. A live Edge smoke was not run during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a reusable Control UI clipboard helper with a fallback path plus regression coverage; the open rendered-text PR is useful adjacent work but does not solve rejected Clipboard API writes.
