## Summary

Keep this issue open. Current main still routes WebChat code-block Copy through a direct Clipboard API call that silently ignores rejected writes, the renderer still emits that affected button path, the latest release still contains the same behavior, and the two related fallback PRs were closed unmerged.

## What This Changes

WebChat code-block Copy should work in Microsoft Edge by handling Clipboard API failures with a fallback and clear success or failure feedback.

## Best Possible Solution

Add a small shared Control UI clipboard helper that tries `navigator.clipboard.writeText`, falls back to a temporary textarea plus `document.execCommand('copy')` when the primary path is unavailable or rejected, returns a success boolean, and is wired into WebChat code-block Copy with focused tests and an Edge-on-Windows smoke check.

## Reproduction Assessment

Yes. The issue is source-level reproducible by making `navigator.clipboard.writeText` reject on the `.code-block-copy` path: current main has no fallback and silently ignores the failure; a live Edge/Windows smoke test is still needed for final validation.

## Solution Assessment

Yes. A narrow Control UI clipboard helper is the best maintainable direction because the bug is in the browser UI copy path, not the gateway protocol or markdown renderer contract, and it can also cleanly cover adjacent Control UI copy buttons without core changes.
