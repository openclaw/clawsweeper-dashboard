## Summary

Keep open: current main and v2026.5.20 still route WebChat code-block Copy through a bare `navigator.clipboard.writeText` call with an ignored rejection, so Edge/permission failures remain a no-op. The two direct fallback PRs were closed unmerged, and the open rendered-text PR is useful adjacent work but does not add a Clipboard API fallback.

## What This Changes

The issue reports that WebChat code-block Copy does nothing in Microsoft Edge on Windows when users click the code-block Copy button.

## Best Possible Solution

Add a shared Control UI clipboard helper that tries `navigator.clipboard.writeText` first, falls back to a cleaned-up hidden textarea plus `document.execCommand("copy")`, and wire WebChat code-block Copy through it with focused regression coverage.

## Reproduction Assessment

Yes at source level: stubbing `navigator.clipboard.writeText` to reject on a `.code-block-copy` click reaches the current empty rejection callback with no fallback. I did not live-reproduce it in Edge on Windows during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a reusable browser clipboard helper with a fallback path plus tests for rejected or missing Clipboard API writes; the adjacent rendered-text PR solves a different payload-selection problem.
