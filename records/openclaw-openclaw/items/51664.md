## Summary

Keep open: current main and v2026.5.22 still send WebChat code-block Copy through `navigator.clipboard.writeText` with an empty rejection handler, so a rejected Clipboard API write remains a no-op. The direct fallback PRs were closed unmerged, and the open rendered-text PR is adjacent but does not add a Clipboard API fallback.

## What This Changes

Fix the Control UI WebChat code-block Copy button so Edge users can copy code blocks without manually selecting text.

## Best Possible Solution

Add a shared Control UI clipboard helper that tries `navigator.clipboard.writeText` first, falls back to a cleaned-up hidden textarea plus `document.execCommand("copy")`, and wire WebChat code-block Copy through it with regression coverage.

## Reproduction Assessment

Yes at source level: stubbing `navigator.clipboard.writeText` to reject on a `.code-block-copy` click reaches the current empty rejection callback with no fallback. I did not live-reproduce it in Edge on Windows during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a reusable browser clipboard helper with a fallback path plus tests for rejected or missing Clipboard API writes; https://github.com/openclaw/openclaw/pull/73633 solves a different payload-selection problem.
