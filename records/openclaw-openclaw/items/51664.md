## Summary

Keep open: current main and v2026.5.22 still route WebChat code-block Copy through `navigator.clipboard.writeText` with an empty rejection handler, so an Edge Clipboard API rejection remains a no-op. The direct fallback PRs were closed unmerged, and the open rendered-text PR addresses a different copy-payload bug.

## What This Changes

Fix WebChat code-block Copy in Microsoft Edge so a rejected Clipboard API write still copies the code-block text.

## Best Possible Solution

Add a shared Control UI clipboard helper that tries `navigator.clipboard.writeText` first, falls back to a hidden textarea plus `document.execCommand("copy")`, and wire WebChat code-block Copy through it with regression coverage.

## Reproduction Assessment

Yes at source level: a rejected `navigator.clipboard.writeText` call on `.code-block-copy` reaches the current empty rejection callback with no fallback. I did not live-reproduce it in Edge on Windows during this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a reusable browser clipboard helper with a fallback path plus tests for rejected or missing Clipboard API writes; the adjacent rendered-text PR solves a different payload-selection problem.
