## Label Justifications

- `P1`: The issue can exhaust sandbox process slots in long-running agent workloads and break normal tool execution.
- `impact:crash-loop`: The reported zombie accumulation can lead to process-table exhaustion and sandbox command failures, which is a process-level availability risk.

## Summary

Current main and the latest release still create Docker sandbox containers without `--init` and run `sleep infinity` as PID 1, so the reported PID-1 zombie accumulation remains valid. The direct init fix and the create-args hash rollout are already covered by open PRs, so this should stay open for maintainer coordination rather than another independent fix PR.

## What This Changes

The issue asks OpenClaw to make Docker sandbox PID 1 reap orphaned zombie processes so long-running sandbox sessions do not accumulate defunct children toward `pids.max`.

## Best Possible Solution

Land a coordinated sandbox fix that runs Docker sandbox containers under an init/reaper process and includes a compatible recreate path for existing long-running containers.

## Reproduction Assessment

Yes for source reproduction: current main still runs Docker sandbox containers as `sleep infinity` without Docker `--init`, matching the reported unreaped `PPID=1` zombie condition. I did not run a mutable long-duration Docker workload in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix belongs at the sandbox container creation and reuse boundary, using Docker init plus create-args hash invalidation. Userland killing is not the best fix because the reporter observed it can increase zombies when PID 1 does not reap them.
