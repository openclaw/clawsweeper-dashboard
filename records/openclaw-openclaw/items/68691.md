## Label Justifications

- `P1`: The report describes a real sandbox process-table exhaustion path that can make shell/tool execution fail under normal agent workloads.
- `impact:crash-loop`: The issue is about process-level availability risk from unreaped zombies accumulating until sandbox PID limits can block new commands.

## Summary

Keep this open: current main and v2026.5.18 still create Docker sandbox containers without Docker `--init`, still run the sandbox command as `sleep infinity`, and still omit a create-args epoch from the normal/browser sandbox hash inputs while the focused rollout PRs remain open.

## What This Changes

The issue reports long-lived Docker sandbox containers accumulating orphaned PPID=1 zombie processes because sandbox PID 1 is `sleep infinity` rather than an init/reaper.

## Best Possible Solution

Land or consolidate Docker `--init` with a create-args epoch bump so new and reused long-lived sandbox containers run an init-backed PID 1, then verify with redacted Docker proof.

## Reproduction Assessment

Yes, source-level: the report provides concrete PPID=1 zombie snapshots, Docker documents `--init` as reaping processes, and current main creates sandboxes as `sleep infinity` without `--init`. I did not run a live stress workload in this read-only review.

## Solution Assessment

Yes. Docker `--init` plus a create-args epoch is the narrow maintainable fix; the shipped main-runtime Tini change and a create-args-only flag change are each insufficient for existing long-lived sandbox containers.
