## Summary

Keep this open: current main and the latest release still create sandbox containers without Docker `--init`, still run the normal sandbox command as `sleep infinity`, and still lack a create-args epoch that would roll long-lived containers forward when the flag changes.

## What This Changes

The issue reports orphaned zombie processes adopted by PID 1 inside Docker sandbox sessions, accumulating until sandbox PID limits can be exhausted.

## Best Possible Solution

Land or consolidate Docker `--init` with a create-args epoch bump so new and existing long-lived sandbox containers use an init-backed PID 1, with redacted Docker proof of the recreate path.

## Reproduction Assessment

Yes, source-level: the issue includes concrete PPID=1 zombie snapshots, and current main still creates sandbox containers as `sleep infinity` without Docker `--init`. I did not run a live stress workload in this read-only review.

## Solution Assessment

Yes. Docker `--init` is the narrow mechanism for orphaned children already adopted by PID 1, and the create-args epoch is needed so existing long-lived containers are recreated instead of keeping the old non-init PID 1.
