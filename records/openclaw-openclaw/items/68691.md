## Label Justifications

- `P1`: The report describes sandbox process-table pressure that can make normal agent command execution fail under real shell-heavy workloads.
- `impact:crash-loop`: Unreaped zombies can exhaust process slots inside the sandbox and cause process-level availability failures.

## Summary

Keep open: current main and the latest release still create Docker sandbox containers without Docker init and run `sleep infinity` as PID 1, while the two related sandbox PRs remain open and not a landed fix path.

## What This Changes

The issue reports unreaped zombie processes adopted by PID 1 in Docker sandbox sessions, risking sandbox process-limit exhaustion under shell-heavy workloads.

## Best Possible Solution

Land or replace a narrow sandbox fix that adds Docker `--init`, bumps a Docker-create-args hash epoch so long-lived containers recreate, and includes real Docker proof that orphaned children are reaped.

## Reproduction Assessment

Yes for a source-level path: current main creates Docker sandboxes with `sleep infinity` as PID 1 and no Docker `--init`, and it can skip pruning for long-lived containers. I did not run a live Docker workload that accumulates zombies in this review.

## Solution Assessment

Yes, the best fix direction is Docker-level init-backed reaping plus a create-args hash epoch so both new and long-lived containers pick it up. A userland zombie killer would be the wrong layer because already-adopted zombies need PID 1 to reap them.
