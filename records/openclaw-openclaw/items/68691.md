## Label Justifications

- `P1`: The report describes sandbox process-table pressure that can make normal agent command execution fail under real shell-heavy workloads.
- `impact:crash-loop`: Unreaped zombies can exhaust process slots inside the sandbox and cause process-level availability failures.

## Summary

Keep open: current main and v2026.6.6 still create Docker sandboxes without Docker init, still run `sleep infinity` as PID 1, and the related sandbox fix PRs remain open/conflicting rather than landed.

## What This Changes

The issue reports Docker sandbox zombies adopted by PID 1 and accumulating until the sandbox can run out of process slots.

## Best Possible Solution

Land or replace a narrow sandbox fix that adds Docker init-backed reaping, ensures long-lived containers recreate for the new create args, and includes real Docker proof.

## Reproduction Assessment

Yes for a source-level path: current main creates Docker sandboxes without `--init`, starts PID 1 as `sleep infinity`, and can keep no-prune containers alive indefinitely. Live Docker workload reproduction has not been run in this read-only review.

## Solution Assessment

Yes: the best fix direction is Docker-level init-backed reaping plus recreate/hash coverage for existing long-lived containers. A userland zombie killer would be the wrong layer because already-adopted zombies need PID 1 to reap them.
