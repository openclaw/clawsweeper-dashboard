## Label Justifications

- `P1`: The issue describes sandbox PID-1 zombies that can exhaust process slots and break normal agent command execution in long-running containers.
- `impact:crash-loop`: The concrete impact is process-level availability failure when zombie accumulation contributes to hitting the sandbox process limit.

## Summary

The issue is still necessary: current main and the latest release still create sandbox containers without Docker init and run `sleep infinity` as PID 1, while the direct init and hash-rollout PRs remain open and unmerged.

## What This Changes

The issue reports unreaped orphan zombie processes in Docker sandbox sessions that can accumulate under PID 1 and exhaust the sandbox process budget.

## Best Possible Solution

Land or replace the open work with one coordinated sandbox bugfix that adds Docker `--init`, invalidates Docker create-args hashes for existing containers, covers fresh and upgrade paths, and includes redacted real Docker proof.

## Reproduction Assessment

Yes, source-level: current main creates Docker sandboxes without `--init`, runs `sleep infinity` as the container command, and can keep prune-disabled containers alive indefinitely. I did not run a live zombie-producing Docker workload in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix belongs at the sandbox container creation and reuse boundary, using Docker init plus create-args hash invalidation. Userland killing is not the best fix because the reporter observed it can increase zombies when PID 1 does not reap them.
