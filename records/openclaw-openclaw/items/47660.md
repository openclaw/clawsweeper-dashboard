## Summary

Keep open. Current main has not implemented this Dockerfile simplification, and the PR still introduces Docker supply-chain and documented workflow regressions: mutable base tags, unverified Bun installation, removed selective plugin dependency staging, and missing slim-runtime CA setup.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build that copies all extensions, uses mutable Node image tags, installs Bun via `curl | bash`, and removes some build args and retry logic.

## Best Possible Solution

Keep the PR open only if it is reworked to reduce Dockerfile complexity while preserving pinned Node/Bun sources, `OPENCLAW_EXTENSIONS` staging, slim-runtime CA setup, and release/docs workflow contracts.

## Reproduction Assessment

Yes. Source inspection of the provided PR diff against current main shows the regressions: mutable base images, remote Bun installer execution, removed selective plugin dependency staging, and missing slim-runtime CA setup.

## Solution Assessment

No. A broad two-stage rewrite that removes maintained Docker build and supply-chain contracts is not the narrowest maintainable way to simplify the Dockerfile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep Docker base images digest-pinned:** `Dockerfile:29`
  - body: The branch replaces the digest-pinned Node base image ARGs with mutable `node:24-bookworm` tags. Future builds would silently consume changed upstream base contents outside the reviewed digest update flow.
  - confidence: 0.93
- **[P1] Keep Bun sourced from the pinned image:** `Dockerfile:33`
  - body: Current main copies Bun from the digest-pinned `OPENCLAW_BUN_IMAGE`. This branch runs `curl -fsSL https://bun.sh/install | bash`, reintroducing unverified build-time code execution in the Docker image path.
  - confidence: 0.92
- **[P1] Install CA certificates in the slim runtime:** `Dockerfile:106-107`
  - body: The runtime is hardcoded to `bookworm-slim`, but the package list omits `ca-certificates` and never runs `update-ca-certificates`. Current main installs both because HTTPS outbound calls need a populated trust store in the slim image.
  - confidence: 0.92
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying the full `extensions/` tree before `pnpm install` makes every bundled plugin workspace participate in the install, while docs, setup, and smoke workflows still use `OPENCLAW_EXTENSIONS` to bake only selected plugin dependencies.
  - confidence: 0.94
- **[P3] Refresh stale Dockerfile comments:** `Dockerfile:3-14`
  - body: The header still describes removed `ext-deps` behavior, and the Bun comment still promises retry behavior after the retry loop was removed. That leaves misleading instructions in the file this PR is simplifying.
  - confidence: 0.88
