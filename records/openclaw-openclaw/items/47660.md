## Summary

Keep open: this Docker refactor is not ready to merge because current main now preserves selective plugin dependency baking and pinned Docker/Bun inputs that this stale branch bypasses. The PR is also merge-conflicting and labeled as risky infrastructure work, so maintainer direction is needed before any replacement approach.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build, copying all extensions before install, using mutable Node base tags and curl-installed Bun, and removing the ext-deps/runtime-assets/base-runtime stages and retry loops.

## Best Possible Solution

Preserve main's pinned, documented Docker build contract; any Dockerfile simplification should be rebased and scoped to reduce incidental staging complexity without removing selective plugin dependency baking or supply-chain pins.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head shows mutable Node tags, curl-installed Bun, and full `extensions` copying before `pnpm install`, while current main/docs/setup still define the pinned and selective build contract. I did not run a Docker build because this is a read-only review and the source-level break is direct.

## Solution Assessment

No. The proposed simplification is not the best way to solve Dockerfile complexity as written because it weakens current supply-chain hardening and breaks the documented `OPENCLAW_EXTENSIONS` behavior; a safer simplification would keep those contracts intact.
