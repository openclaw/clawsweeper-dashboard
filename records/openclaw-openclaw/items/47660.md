## Summary

Keep open. The PR has real Docker build proof, but the patch is not merge-ready because it regresses current Docker build contracts and supply-chain hardening that now exist on main.

## What This Changes

The branch simplifies the root Dockerfile to a two-stage build while removing variant switching, selected plugin dependency staging, retry loops, and digest-pinned base image references.

## Best Possible Solution

Preserve the current main Docker security and plugin dependency contracts, and only pursue simplification as a smaller rework that keeps digest-pinned sources, `OPENCLAW_EXTENSIONS`, release metadata, docs, and smoke workflows aligned.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR head and current main shows the branch copies all `extensions/*` before install, ignores the documented selective plugin dependency path, and reintroduces unpinned Docker build inputs.

## Solution Assessment

No. The simplification is not the best current fix because it removes documented Docker behavior and weakens supply-chain controls; a safer path is a targeted cleanup on current main that preserves those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch replaces digest-pinned Node image ARGs with moving `node:24-*` tags and installs Bun by piping `https://bun.sh/install` to `bash`. Current main already fixed that supply-chain path with a pinned `oven/bun` stage, so merging this would reintroduce the Docker build exposure.
  - confidence: 0.95
- **[P2] Preserve the OPENCLAW_EXTENSIONS install contract:** `Dockerfile:43`
  - body: Copying the whole `extensions/` tree before `pnpm install` makes every `extensions/*` workspace participate in dependency installation. The setup script, docs, release workflow, and install smoke still use `OPENCLAW_EXTENSIONS` to select bundled plugin dependencies, so this breaks a documented Docker build contract.
  - confidence: 0.93
- **[P2] Re-declare the APT upgrade build arg in runtime:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this `RUN` executes in the runtime stage. Without re-declaring the ARG in that stage, `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` is ignored and the upgrade path still runs.
  - confidence: 0.86

## Security Review
