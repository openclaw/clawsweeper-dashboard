## Summary

Keep open for human Docker/security review: the branch is a broad Dockerfile cleanup, but current PR head conflicts with main and removes pinned build inputs, selective plugin staging, runtime packaging checks, TLS/Python/init runtime guarantees, and Docker apt key hardening now encoded in current main.

## What This Changes

The PR branch rewrites the root Dockerfile into a two-stage build with rolling Node tags, direct Bun installer use, unconditional plugin copying, and simplified runtime install/copy steps.

## Best Possible Solution

Split any Dockerfile readability cleanup into small owner-reviewed changes that preserve pinned images, selective plugin staging, runtime asset pruning/validation, CA/python/tini runtime setup, OCI metadata, and the Docker apt key guard.

## Reproduction Assessment

Yes. Source inspection of PR head f918dda against current main shows the regressions directly, and current Docker tests/docs/workflows encode the expected contracts without needing another Docker run.

## Solution Assessment

No. The broad two-stage rewrite removes established security, packaging, and runtime guarantees; the maintainable path is targeted cleanup that keeps the Docker contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch uses rolling node:24-* tags and installs Bun with `curl | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned oven/bun image, so this reopens supply-chain and reproducibility risk.
  - confidence: 0.97
- **[P2] Keep the root Dockerfile frontend implicit:** `Dockerfile:1`
  - body: Adding `# syntax=docker/dockerfile:1.7` makes BuildKit resolve an external Dockerfile frontend before the build. Current main has coverage rejecting syntax directives for the root Dockerfile.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes all plugin workspaces participate and invalidates the install layer on source-only plugin changes. Docs and CI still rely on `OPENCLAW_EXTENSIONS` to select plugin dependencies.
  - confidence: 0.95
- **[P2] Re-declare the apt-upgrade build arg in runtime:** `Dockerfile:103-105`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before `FROM`, but this runtime-stage `RUN` reads it without a stage-local `ARG`. Docker expands it empty here, so callers cannot disable the upgrade branch.
  - confidence: 0.9
- **[P2] Restore runtime certificates, Python, and tini:** `Dockerfile:106-108`
  - body: The slim runtime install omits current main's `ca-certificates`, `python3`, `tini`, `update-ca-certificates`, and final `ENTRYPOINT`. That can break HTTPS egress, Python-backed skills, and signal/zombie handling in long-running containers.
  - confidence: 0.9
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying from `build` skips current main's runtime pruning, bundled-plugin postinstall, plugin dist pruning, and dist import check. It also omits `pnpm-workspace.yaml`, `patches`, and `qa`, which current smoke coverage expects in the runtime image.
  - confidence: 0.9
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` reintroduces a runtime discovery override that current main explicitly avoids and tests reject for Docker images.
  - confidence: 0.82
- **[P2] Keep the Docker apt key multi-key guard:** `Dockerfile:172`
  - body: The optional Docker CLI path compares only the first fingerprint from the downloaded key file, but `gpg --dearmor` would trust every key in that file. Current main first requires exactly one primary public key before trusting the keyring.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain controls by adding an external frontend, executing an unpinned Bun installer, using rolling Node tags, and dropping the Docker apt multi-key guard.

Concerns:

- **[high] Unpinned Bun installer executes during build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash`, executing remote code during image construction and undoing current main's pinned Bun image copy.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use literal node:24-bookworm and node:24-bookworm-slim tags instead of digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] Dockerfile frontend is fetched externally:** `Dockerfile:1`
  - body: The syntax directive causes BuildKit to resolve an external Dockerfile frontend before the build, which current main intentionally rejects for the root Dockerfile.
  - confidence: 0.93
- **[medium] Docker apt key guard trusts extra primary keys:** `Dockerfile:172`
  - body: The optional Docker CLI path checks only the first fingerprint from the downloaded key file, while apt would trust every key imported from that file.
  - confidence: 0.86
