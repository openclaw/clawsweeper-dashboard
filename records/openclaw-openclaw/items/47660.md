## Summary

Keep this PR open as patch-incorrect: the branch is a broad Dockerfile rewrite that removes several current-main supply-chain, runtime-hardening, plugin-packaging, and cache contracts already covered by source and tests.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build and removes digest-pinned base/Bun setup, manifest-only plugin staging, runtime-assets validation, runtime package/entrypoint/state handling, and several cache/retry guards.

## Best Possible Solution

Keep current main's Docker security/runtime contract and consider only a smaller Docker-owner-reviewed readability cleanup that preserves pinned inputs, manifest staging, runtime-assets validation, TLS/Python/tini, Docker GPG validation, and state-directory permissions.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main and the Dockerfile tests gives a high-confidence path for the regressions; I did not run the optional Docker builds locally.

## Solution Assessment

No. The broad two-stage rewrite removes tested Docker supply-chain, runtime, and packaging guarantees; the safer path is a narrower readability cleanup that preserves those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and installs Bun with `curl | bash`; current main pins Node and Bun image inputs and copies Bun from the pinned Bun image, so this reopens Docker build supply-chain and reproducibility risk.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit resolve an external frontend before the build. Current main has coverage rejecting syntax directives in the root Dockerfile to avoid that extra supply-chain input.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Keep native-addon verification in the build:** `Dockerfile:47-48`
  - body: The simplified install path drops current main's supported-architecture install flags and Matrix native-addon verification/retry block, so Docker builds can miss the guard for absent `matrix-sdk-crypto` native binaries.
  - confidence: 0.88
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, plus `update-ca-certificates` and the final `ENTRYPOINT`; that can break HTTPS egress, Python-backed skills, and long-running container process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses current main's production prune, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import validation, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Do not force source plugin discovery in runtime images:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` overrides the packaged runtime discovery path; current main explicitly tests that Docker images do not force this source-tree override.
  - confidence: 0.86
- **[P2] Re-declare the apt-upgrade arg in the runtime stage:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the first `FROM`, but not re-declared in the runtime stage, so callers cannot use `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` to skip the upgrade branch.
  - confidence: 0.84
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node 700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening in multiple concrete ways.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main explicitly rejects.
  - confidence: 0.93
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Bundled plugin postinstall hardening is skipped:** `Dockerfile:77`
  - body: Copying from `build` bypasses `scripts/postinstall-bundled-plugins.mjs`, including current bundled-plugin runtime hotfixes applied during runtime asset preparation.
  - confidence: 0.86
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
