## Summary

Keep open. Current main has explicit Dockerfile tests, docs, and workflow usage for digest-pinned image inputs, pinned Bun copying, selective plugin dependency staging, runtime workspace metadata, and slim-runtime support; this PR removes or regresses those contracts and is also marked conflicting/risky infra.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build with rolling Node image tags, Bun installed by `curl | bash`, unconditional bundled plugin copying, and simplified runtime setup.

## Best Possible Solution

Preserve current main's Docker supply-chain and selective plugin-build contracts, and split any readability cleanup into narrower owner-reviewed changes with tests, docs, and workflows kept aligned.

## Reproduction Assessment

Yes. PR-head source inspection at `f918dda76d3fdbb24f575029df510ad3ee952900` plus current-main Dockerfile tests gives a high-confidence reproduction path for the incompatible contracts, even though I did not run Docker in this read-only review.

## Solution Assessment

No. The broad simplification is not the safest maintainable solution because it intentionally removes existing supply-chain, plugin-build, and runtime packaging guarantees; a narrower Docker-owner-guided cleanup should preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This adds `# syntax=docker/dockerfile:1.7`, which makes BuildKit resolve an external Dockerfile frontend before the build. Current main has coverage rejecting syntax directives for the root Dockerfile, so merging this would undo that hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The build and runtime stages use rolling `node:24-*` tags and install Bun with `curl | bash`. Current main pins the Node and Bun image inputs and copies Bun from the pinned image, so this regresses reproducibility and reintroduces unverified remote code execution during image construction.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, workflows, and tests still use that build arg to control which plugin dependencies are included.
  - confidence: 0.95
- **[P2] Redeclare the apt upgrade build arg in the runtime stage:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs in the runtime stage where the arg is not in scope unless redeclared. The opt-out expands empty, so `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` still runs the upgrade path.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies only selected app files from `build`, but current main copies `pnpm-workspace.yaml` and `patches` after pruning so container-local pnpm workflows can resolve patched dependencies. Dropping those files breaks the runtime packaging contract covered by current tests.
  - confidence: 0.84
- **[P2] Keep runtime certificates, Python, and init behavior:** `Dockerfile:106-107`
  - body: The runtime install omits current main's explicit `ca-certificates`, `python3`, and `tini` setup and the final `ENTRYPOINT ["tini", "-s", "--"]`. That can regress HTTPS trust, Python-backed workflows, and signal/zombie handling in long-running containers.
  - confidence: 0.82
- **[P2] Do not override bundled plugin discovery in runtime images:** `Dockerfile:122`
  - body: This sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions`, while current main has a regression test asserting the Dockerfile does not override bundled plugin discovery with that legacy environment variable.
  - confidence: 0.78
