## Summary

Keep open for human review; the branch conflicts with current main and would regress established Docker supply-chain and compatibility contracts despite sufficient default-build proof.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removing digest-pinned inputs, retry loops, selective plugin staging, runtime variant handling, and simplifying APT/Corepack paths.

## Best Possible Solution

Preserve current Docker contracts, and pursue any simplification as narrower maintainer-guided changes that keep pinned inputs, selective plugin staging, runtime metadata, and docs/tests/workflows aligned.

## Reproduction Assessment

Yes for the review findings. PR-head source inspection shows the external frontend directive, rolling base tags, `curl | bash` Bun install, unconditional plugin copying, and stage-scoped ARG issue while current-main tests/docs/workflows encode the opposite contracts.

## Solution Assessment

No. The proposed broad simplification removes established supply-chain and compatibility guarantees; the safer path is a maintainer-directed cleanup that preserves digest-pinned inputs, pinned Bun copy, `OPENCLAW_EXTENSIONS`, runtime metadata, and current validation contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This reintroduces `# syntax=docker/dockerfile:1.7`, which forces an external Dockerfile frontend pull during builds. Current main has a regression test rejecting that directive, so this would undo the Docker hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch switches the build/runtime bases to rolling `node:24-*` tags and runs `curl -fsSL https://bun.sh/install | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned image, so this regresses reproducibility and supply-chain hardening.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, release workflows, and tests still use that build arg to control which plugin dependencies are baked into the image.
  - confidence: 0.95
- **[P2] Redeclare the apt upgrade build arg:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the first `FROM`, but this check runs in the runtime stage where that arg is out of scope unless redeclared. The opt-out expands empty, so the upgrade still runs when callers pass `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0`.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies selected app files from `build` but drops `pnpm-workspace.yaml` and `patches`. Current main keeps those files after pruning so container-local pnpm workflows can resolve patched dependencies, and current smoke coverage checks for that contract.
  - confidence: 0.84
