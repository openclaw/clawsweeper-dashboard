## Summary

Keep open and block merge as-is. The branch is a risky Docker infra refactor that removes current-main supply-chain pinning, runtime hardening, runtime-asset validation, and documented Docker build contracts; the existing default-build proof does not overcome those source-level regressions.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build and removes several Docker build/runtime stages, image pinning, and selective plugin/runtime packaging behavior.

## Best Possible Solution

Keep current main’s Docker security and runtime contracts, and consider only a smaller Docker readability refactor that preserves pinned inputs, manifest staging, runtime-assets validation, runtime packages/tini, GPG guard, state-dir permissions, docs, and tests.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main shows the regressions directly, and current-main Dockerfile tests encode the affected contracts; I did not run tests because this was a read-only review.

## Solution Assessment

No. A broad two-stage rewrite that removes supply-chain and runtime-validation contracts is not the best way to simplify the Dockerfile; the safer path is a narrower refactor that keeps those contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and installs Bun with `curl | bash`. Current main pins the Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the `tini` entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.95
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit resolve an external frontend image before the build. Current main has coverage rejecting this for the root Dockerfile to avoid adding another supply-chain input.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and removes the documented selective dependency staging behavior driven by `OPENCLAW_EXTENSIONS`.
  - confidence: 0.95
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI install path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node 700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening contracts that current main explicitly preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main rejects.
  - confidence: 0.93
- **[high] Runtime TLS, Python, and init hardening are omitted:** `Dockerfile:107`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from `build` skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor the whole key file; current main rejects key files with more than one primary public key before trusting them.
  - confidence: 0.88
- **[medium] State directory secrecy guard is removed:** `Dockerfile:197`
  - body: The final image no longer creates `/home/node/.openclaw` as `node:node` mode `0700` before switching users, weakening first-run named-volume behavior for credential and auth-profile state.
  - confidence: 0.86
