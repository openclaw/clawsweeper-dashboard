## Summary

Keep open for maintainer handling: the PR’s readability goal is understandable, but the branch currently regresses Docker supply-chain, packaging, runtime, and documented plugin-build contracts that current main now protects with source and test coverage.

## What This Changes

The PR rewrites the root Dockerfile from a multi-stage, digest-pinned build to a two-stage build with direct Bun install, unconditional plugin copying, and simplified runtime install/copy steps.

## Best Possible Solution

Pursue any Dockerfile readability cleanup as smaller owner-reviewed changes that preserve the current pinned inputs, plugin staging, runtime pruning/validation, package list, OCI metadata, and key-verification contracts.

## Reproduction Assessment

Yes. Source inspection of PR head `f918dda` against current main shows the regressions directly, and current Docker tests/docs encode the expected contracts without needing another Docker run.

## Solution Assessment

No. The broad rewrite removes established Docker security and runtime guarantees; a narrow cleanup that keeps the current contracts is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch uses rolling `node:24-*` tags and installs Bun with `curl | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned image, so this reopens supply-chain and reproducibility risk.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: Adding `# syntax=docker/dockerfile:1.7` makes BuildKit resolve an external Dockerfile frontend before the build. Current main has coverage rejecting syntax directives for the root Dockerfile.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes all plugin workspaces participate and invalidates the install layer on source-only plugin changes. Docs and current tests still rely on `OPENCLAW_EXTENSIONS` selecting plugin dependencies.
  - confidence: 0.95
- **[P2] Re-declare the apt-upgrade build arg in runtime:** `Dockerfile:103-105`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before `FROM`, but this runtime-stage `RUN` reads it without a stage-local `ARG`. Docker does not carry that arg into the stage, so callers cannot control the upgrade branch as the comment claims.
  - confidence: 0.9
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-108`
  - body: The slim runtime install omits current main's `ca-certificates`, `python3`, `tini`, `update-ca-certificates`, and final `ENTRYPOINT`. That can break HTTPS egress, Python-backed skills, and signal/zombie handling in long-running containers.
  - confidence: 0.92
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying from `build` skips current main's runtime pruning, bundled-plugin postinstall, plugin dist pruning, and dist import check, and it omits runtime `pnpm-workspace.yaml`, `patches`, and `qa` paths expected by current coverage.
  - confidence: 0.9
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` reintroduces a runtime discovery override that current main explicitly avoids and tests reject for Docker images.
  - confidence: 0.82
- **[P2] Keep the Docker apt key multi-key guard:** `Dockerfile:170-177`
  - body: The optional Docker CLI path compares only the first fingerprint from the downloaded key file, but `gpg --dearmor` would trust every key in that file. Current main first requires exactly one primary public key before trusting the keyring.
  - confidence: 0.86
- **[P2] Pre-create the OpenClaw home before USER node:** `Dockerfile:197`
  - body: The branch drops the `install -d -m 0700 -o node -g node /home/node/.openclaw` guard before switching to `USER node`, so first-run named volumes can become root-owned or lose the intended credential-directory secrecy.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening by removing pinned inputs, guarded key import, runtime TLS/init setup, and state-directory protection.

Concerns:

- **[high] Unpinned Bun installer executes during build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash`, executing remote code during image construction and undoing current main's pinned Bun image copy.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use literal `node:24-bookworm` and `node:24-bookworm-slim` tags instead of digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] Dockerfile frontend is fetched externally:** `Dockerfile:1`
  - body: The syntax directive causes BuildKit to resolve an external Dockerfile frontend before the build, which current main intentionally rejects for the root Dockerfile.
  - confidence: 0.93
- **[medium] Docker apt key guard trusts extra primary keys:** `Dockerfile:172`
  - body: The optional Docker CLI path checks only the first fingerprint from the downloaded key file, while apt would trust every key imported from that file.
  - confidence: 0.86
- **[medium] Runtime postinstall and patch safeguards are skipped:** `Dockerfile:112`
  - body: Copying runtime files from `build` skips the bundled-plugin postinstall and omits patch metadata that current main keeps for runtime integrity and in-container package-manager operations.
  - confidence: 0.82
- **[medium] Credential state directory protection is removed:** `Dockerfile:197`
  - body: Dropping the pre-created `0700` node-owned OpenClaw home can leave named-volume state unwritable or less private for credentials and auth profiles.
  - confidence: 0.84
