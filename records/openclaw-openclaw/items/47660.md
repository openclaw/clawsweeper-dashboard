## Summary

Keep open: the PR has default-path Docker proof, but the patch would regress current Docker supply-chain hardening and documented plugin dependency behavior, so it needs maintainer Docker/security judgment before merge.

## What This Changes

This PR collapses the root Dockerfile into fewer stages while removing digest-pinned image inputs, Bun/Corepack retry logic, runtime variant selection, and selective plugin dependency staging.

## Best Possible Solution

Preserve main's Docker security and plugin dependency contracts, and pursue any simplification as a smaller maintainer-directed change that keeps pinned inputs, `OPENCLAW_EXTENSIONS`, docs, tests, and workflows aligned.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff against current main shows the branch drops pinned inputs, bypasses selective plugin dependency staging, and uses a global Docker ARG inside a later stage. I did not run a local Docker build in this read-only review, but the contributor's attached log covers the default-path after-change build.

## Solution Assessment

No. The proposed broad simplification is not the narrowest maintainable solution because it removes documented Docker behavior and weakens supply-chain controls; targeted cleanup that preserves the current contracts would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch replaces digest-pinned Node and Bun image inputs with moving `node:24-*` tags and reinstates `curl -fsSL https://bun.sh/install | bash`. Current main now uses digest-pinned Node and Bun image inputs, so this regresses Docker build reproducibility and supply-chain hardening.
  - confidence: 0.95
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying the whole plugin tree before `pnpm install` makes every plugin workspace participate in dependency installation. Docker docs, setup scripts, install smoke, and release workflows still use `OPENCLAW_EXTENSIONS` to select plugin dependencies, so this breaks a documented build contract.
  - confidence: 0.93
- **[P2] Re-declare the APT upgrade arg in the runtime stage:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs inside the runtime stage. Without re-declaring the ARG in that stage, `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` is ignored and the upgrade path still runs.
  - confidence: 0.86
- **[P3] Update stale Dockerfile comments:** `Dockerfile:12-19`
  - body: The header and nearby comments still describe removed behavior such as selective extension args, runtime variants, digest pinning, and retry logic. If the refactor proceeds, those comments need to match the new Dockerfile so contributors do not follow removed build options.
  - confidence: 0.82
