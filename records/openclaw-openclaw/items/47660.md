## Summary

Keep this PR open but do not merge as-is: the simplification still removes current Docker supply-chain, runtime, and packaging guardrails that are encoded in main and covered by tests/docs. The attached proof shows the default image can build and run `gateway --help`, but it does not address the security-sensitive and optional Docker paths affected by the diff.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build with rolling Node tags, curl-installed Bun, unconditional plugin source copying, simplified apt/cache handling, and direct runtime copies from the build stage.

## Best Possible Solution

Keep current main's Docker security/runtime contract and accept only Docker-owner-reviewed readability changes that preserve pinned inputs, selective plugin staging, runtime asset validation, TLS/Python/tini, GPG validation, and state-directory permissions.

## Reproduction Assessment

Yes. Source inspection of PR head plus current main Dockerfile tests/docs gives a high-confidence reproduction path for the regressions; I did not rerun a full Docker build locally.

## Solution Assessment

No. The broad rewrite removes established Docker supply-chain, runtime, and packaging guarantees; the maintainable path is a narrower cleanup that leaves those contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-34`
  - body: The branch builds from rolling Node tags and installs Bun with `curl | bash`, while current main pins Node and Bun image inputs and copies Bun from a pinned image. This reopens supply-chain and reproducibility risk for Docker builds.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: Adding the syntax directive makes BuildKit resolve an external frontend before the build. Current main has coverage rejecting syntax directives in the root Dockerfile to avoid that extra supply-chain input.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes all plugin source participate in the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` dependency selection contract.
  - confidence: 0.95
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, plus `update-ca-certificates` and the final `ENTRYPOINT`; that can break HTTPS egress, Python-backed skills, and long-running container process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` skips current main's production prune, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import validation, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Remove or re-scope the apt upgrade arg:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before `FROM`, so the runtime-stage shell does not see the caller's build arg. The check therefore cannot preserve the intended opt-out and reintroduces distro-upgrade behavior that main no longer uses.
  - confidence: 0.86
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: The branch drops the `node:node 700` `/home/node/.openclaw` creation and assertion before switching to `USER node`. Named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening in multiple concrete ways.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main explicitly rejects.
  - confidence: 0.93
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:106`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Bundled plugin postinstall hardening is skipped:** `Dockerfile:112`
  - body: Copying from `build` bypasses `scripts/postinstall-bundled-plugins.mjs`, including current bundled-plugin runtime hotfixes applied during runtime asset preparation.
  - confidence: 0.86
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
