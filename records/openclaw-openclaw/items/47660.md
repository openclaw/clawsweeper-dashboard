## Summary

Keep open: this is an active risky Docker infra PR with blocking correctness and supply-chain regressions against current main's Dockerfile contracts, not a conservative cleanup close candidate.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes digest-pinned Node/Bun inputs and selective plugin staging, and copies runtime assets directly from the build stage.

## Best Possible Solution

Pursue Dockerfile readability as smaller owner-reviewed changes that preserve pinned inputs, selective plugin staging, runtime pruning and validation, TLS/Python/tini, the Docker key guard, and state-directory permissions.

## Reproduction Assessment

Yes. Source inspection of PR head f918dda76d3fdbb24f575029df510ad3ee952900 against current main and the focused Dockerfile tests show the regressions directly.

## Solution Assessment

No. The broad two-stage rewrite removes established Docker security and runtime guarantees; a narrower cleanup preserving those contracts is the maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch switches build/runtime stages to rolling node tags and installs Bun with `curl | bash`. Current main pins Node and Bun image inputs, so this reopens supply-chain and reproducibility risk.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: Adding `# syntax=docker/dockerfile:1.7` makes BuildKit resolve an external Dockerfile frontend before the build. Current main has coverage rejecting syntax directives for the root Dockerfile.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes all plugin workspaces participate and invalidates the install layer on source-only plugin changes. Current docs/setup still rely on `OPENCLAW_EXTENSIONS` selecting plugin dependencies.
  - confidence: 0.95
- **[P2] Re-declare the apt-upgrade build arg in runtime:** `Dockerfile:103-105`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before `FROM`, so this runtime-stage condition cannot see caller values. The documented opt-out path no longer works unless the ARG is re-declared in the runtime stage.
  - confidence: 0.9
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-108`
  - body: The slim runtime install omits current main's `ca-certificates`, `python3`, `tini`, `update-ca-certificates`, and final `ENTRYPOINT`. That can break HTTPS egress, Python-backed skills, and signal/zombie handling in long-running containers.
  - confidence: 0.92
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying from `build` skips current main's runtime pruning, bundled-plugin postinstall, plugin dist pruning, and dist import check, and it omits runtime `pnpm-workspace.yaml`, `patches`, and `qa` paths expected by current coverage.
  - confidence: 0.9
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` reintroduces a runtime discovery override that current main explicitly avoids and tests reject for Docker images.
  - confidence: 0.82
- **[P2] Keep the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The optional Docker CLI path compares only the first fingerprint from the downloaded key file, but `gpg --dearmor` would trust every key in that file. Current main first requires exactly one primary public key before trusting the keyring.
  - confidence: 0.86
- **[P2] Pre-create the OpenClaw state directory before USER node:** `Dockerfile:197`
  - body: The branch drops the `install -d -m 0700 -o node -g node /home/node/.openclaw` guard before switching to `USER node`, so first-run named volumes can become root-owned or lose the intended credential-directory secrecy.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening by removing pinned inputs, guarded key import, runtime TLS/init setup, runtime postinstall safeguards, and state-directory protection.

Concerns:

- **[high] Unpinned Bun installer executes during build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash`, executing remote code during image construction and undoing current main's pinned Bun image copy.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use literal `node:24-bookworm` and `node:24-bookworm-slim` tags instead of digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] Dockerfile frontend is fetched externally:** `Dockerfile:1`
  - body: The syntax directive causes BuildKit to resolve an external Dockerfile frontend before the build, which current main intentionally rejects for the root Dockerfile.
  - confidence: 0.93
- **[high] Runtime TLS and init packages are omitted:** `Dockerfile:106`
  - body: The runtime apt list omits `ca-certificates`, `python3`, and `tini`, so HTTPS egress, Python-backed skills, and long-running process handling can regress.
  - confidence: 0.92
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The optional Docker CLI path no longer checks that the downloaded key file contains exactly one primary public key before dearmoring it into an apt trust root.
  - confidence: 0.86
- **[medium] State directory secrecy guard is removed:** `Dockerfile:197`
  - body: Dropping the pre-created `0700` `/home/node/.openclaw` directory can make credential and auth-profile storage fail or inherit weaker mount-point permissions.
  - confidence: 0.84
