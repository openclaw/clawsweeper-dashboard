## Summary

Keep this PR open for human Docker/security review: the branch has real default-build proof, but it still regresses current main's Docker supply-chain, packaging, and runtime contracts and is now conflicting with later Docker hardening work.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes selective plugin dependency staging and runtime variant/digest-pinning/retry behavior, and uses rolling Node tags plus a remote Bun installer.

## Best Possible Solution

Preserve current main's pinned inputs, selective plugin manifest staging, runtime metadata/certificates/init behavior, and split readability cleanup into smaller Docker-owner-reviewed changes with tests and docs kept aligned.

## Reproduction Assessment

Yes. Source inspection of PR head f918dda against current main shows the regressions directly, and current Docker tests/docs/scripts encode the expected contracts without needing a Docker run.

## Solution Assessment

No. The broad simplification removes established security, packaging, and runtime guarantees; the safer path is narrower cleanup that preserves those Docker contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch uses rolling `node:24-*` tags and installs Bun via `curl | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned image, so this regresses reproducibility and reintroduces unverified remote code execution during image construction.
  - confidence: 0.97
- **[P2] Keep the root Dockerfile frontend implicit:** `Dockerfile:1`
  - body: Adding `# syntax=docker/dockerfile:1.7` makes BuildKit resolve an external Dockerfile frontend before the build. Current main has regression coverage rejecting syntax directives for the root Dockerfile, so this undoes that hardening.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all plugin sources before `pnpm install` makes every plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, scripts, and tests still use that build arg to control which plugin dependencies are baked into the image.
  - confidence: 0.95
- **[P2] Re-declare the apt-upgrade arg in the runtime stage:** `Dockerfile:103-105`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this runtime-stage `RUN` reads it without re-declaring the ARG. Docker will expand it empty here, so callers cannot opt out of the upgrade layer.
  - confidence: 0.9
- **[P2] Keep runtime certificates, Python, and init behavior:** `Dockerfile:106-108`
  - body: The runtime install omits current main's explicit `ca-certificates`, `python3`, and `tini` setup and the final `ENTRYPOINT ["tini", "-s", "--"]`. That can regress HTTPS trust, Python-backed workflows, and signal/zombie handling in long-running containers.
  - confidence: 0.88
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies selected app files but drops `pnpm-workspace.yaml` and `patches`. Current main copies both after pruning, and install-smoke checks that patched dependencies resolve inside the container.
  - confidence: 0.87
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:120-122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` reintroduces a legacy discovery override. Current main has regression coverage asserting the Dockerfile does not set this environment variable.
  - confidence: 0.8
- **[P2] Preserve the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The optional Docker CLI block accepts the first fingerprint from the downloaded key file. Current main first requires exactly one primary public key so apt cannot trust extra keys bundled in the same file.
  - confidence: 0.84
