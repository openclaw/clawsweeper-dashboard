## Summary

Keep open: this PR is a broad Dockerfile rewrite that conflicts with current main's Docker supply-chain, runtime, plugin packaging, and state-directory contracts, with concrete blockers already reinforced by current tests/docs and recent Docker review comments.

## What This Changes

The PR simplifies the root Dockerfile into a two-stage build while removing digest-pinned inputs, selective plugin staging, retry/validation stages, and several runtime image paths.

## Best Possible Solution

Keep current main's Docker security/runtime contract and accept only narrow Docker-owner-reviewed readability cleanup that preserves pinned inputs, selective plugin staging, runtime-assets validation, TLS/Python/tini, Docker GPG validation, and state-directory permissions.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main, plus current Dockerfile tests, gives high-confidence reproduction paths for the regressions; I did not run the full optional Docker builds locally.

## Solution Assessment

No. The broad simplification is not the narrowest maintainable solution because it removes established Docker supply-chain, runtime, packaging, and cache guarantees; a safer path is a smaller cleanup that leaves those contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and installs Bun with `curl | bash`; current main pins Node and Bun image inputs and copies Bun from a pinned image, so this reopens supply-chain and reproducibility risk for Docker builds.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: Adding the syntax directive makes BuildKit resolve an external frontend before the build. Current main has coverage rejecting syntax directives in the root Dockerfile to avoid that extra supply-chain input.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Keep native-addon verification in the build stage:** `Dockerfile:47-48`
  - body: The simplified install path drops current main's supported-architecture install/prune flags and Matrix native-addon verification/retry block, so Docker builds can miss the guard that catches absent `matrix-sdk-crypto` native binaries.
  - confidence: 0.88
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, plus `update-ca-certificates` and the final `ENTRYPOINT`; that can break HTTPS egress, Python-backed skills, and long-running container process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` skips current main's production prune, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import validation, and runtime copies for `patches` and `qa`. That can ship unvalidated bytes and break in-container package operations.
  - confidence: 0.92
- **[P2] Do not force source plugin discovery in runtime images:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` overrides the packaged runtime discovery path; current main explicitly tests that Docker images do not force this source-tree override.
  - confidence: 0.86
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node 700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86
- **[P2] Make the apt-upgrade build arg stage-local:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the `FROM`, but not re-declared in the runtime stage, so `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` cannot control this `RUN` block as written.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening in multiple concrete ways.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main explicitly rejects.
  - confidence: 0.93
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:106`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Bundled plugin postinstall hardening is skipped:** `Dockerfile:112`
  - body: Copying from `build` bypasses `scripts/postinstall-bundled-plugins.mjs` and the runtime-assets validation/pruning stage, including current bundled-plugin runtime fixes applied before packaging.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path no longer checks that the downloaded armored key contains exactly one primary public key before dearmoring it into the trusted apt keyring.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
