## Summary

Keep open for maintainer review: this risky-infra PR still has blocking Docker regressions against current main, including unpinned build inputs, removed runtime hardening, bypassed runtime-assets validation, and dropped GPG/state-dir protections.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes selective extension staging, variant switching, retry/digest pinning, and copies runtime assets directly from the build stage.

## Best Possible Solution

Keep current main's Docker security/runtime contract; if readability is still desired, pursue a smaller Docker-owner-reviewed cleanup that preserves pinned inputs, manifest staging, runtime-assets validation, TLS/Python/tini, GPG guard, and state-dir permissions.

## Reproduction Assessment

Yes. Source inspection of the PR head Dockerfile against current main and the Dockerfile guard tests gives a high-confidence path for the regressions; I did not run new Docker builds locally.

## Solution Assessment

No. The branch trades established Docker supply-chain and runtime contracts for readability; a narrower cleanup that preserves those contracts is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and runs `curl -fsSL https://bun.sh/install | bash`. Current main pins Node/Bun image refs and copies Bun from the pinned Bun image, so this reopens supply-chain and reproducibility risk.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit resolve an external frontend before the build. Current main has coverage rejecting syntax directives in the root Dockerfile to avoid that extra supply-chain input.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Keep native-addon verification in the build:** `Dockerfile:48`
  - body: The simplified install step drops current main's supported-architecture install flags and Matrix native-addon verification/retry block, so Docker builds can miss the guard for absent `matrix-sdk-crypto` native binaries.
  - confidence: 0.88
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the `tini` ENTRYPOINT. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Do not force source plugin discovery in runtime images:** `Dockerfile:122`
  - body: Setting `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` overrides the packaged runtime discovery path; current main explicitly tests that Docker images do not force this source-tree override.
  - confidence: 0.86
- **[P2] Re-declare the apt-upgrade arg in the runtime stage:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the first `FROM`, but this `RUN` executes in the runtime stage where the arg is not in scope. Callers cannot use `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` to skip the upgrade branch.
  - confidence: 0.84
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:172-177`
  - body: The Docker CLI path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node 700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces concrete Docker supply-chain and runtime-hardening regressions that need Docker/security review before any merge.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main explicitly rejects.
  - confidence: 0.93
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Bundled plugin postinstall hardening is skipped:** `Dockerfile:112`
  - body: Copying from `build` bypasses `scripts/postinstall-bundled-plugins.mjs`, including current bundled-plugin runtime hotfixes applied during runtime asset preparation.
  - confidence: 0.86
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
