## Summary

Keep open for maintainer handling: the PR is a risky Dockerfile cleanup that removes or bypasses several current-main Docker supply-chain, runtime, and validation contracts.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build using rolling Node tags, curl-installed Bun, full extension copies, and direct runtime copies from the build stage.

## Best Possible Solution

Preserve the current Docker hardening and runtime contracts; any readability cleanup should be a smaller owner-reviewed refactor that keeps pinned inputs, manifest staging, runtime-assets validation, docs, and guard tests intact.

## Reproduction Assessment

Yes. The PR-head Dockerfile source at `f918dda` directly shows the removed pins, runtime packages, validation stage, key guard, and state-dir guard, and current-main tests document those contracts.

## Solution Assessment

No. Collapsing the Dockerfile by deleting supply-chain and runtime safeguards is not the narrowest maintainable cleanup; a safer path would preserve those contracts and refactor only readability concerns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The PR head builds from rolling Node tags and runs the Bun installer with `curl | bash`. Current main pins Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime TLS, Python, and tini support:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the tini entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Restore the Docker apt key primary-key guard:** `Dockerfile:170-177`
  - body: The Docker CLI install path verifies only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node` mode `0700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch an external Dockerfile frontend before the build. Current main has coverage rejecting this for the root Dockerfile to avoid adding another supply-chain input.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening controls that current main intentionally preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying Bun from a pinned image, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images and frontend become moving inputs:** `Dockerfile:29`
  - body: The PR uses rolling Node image tags and an external Dockerfile frontend tag instead of current main's digest-pinned image args and no-frontend policy, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the tini entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from `build` skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] State directory secrecy guard is removed:** `Dockerfile:197`
  - body: The image no longer pre-creates `/home/node/.openclaw` as `node:node` mode `0700`, weakening the credential/state directory assumptions for named-volume deployments.
  - confidence: 0.86
