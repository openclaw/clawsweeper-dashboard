## Summary

Keep open for maintainer Docker/security review. The PR has real default-build proof, but it regresses current main's Docker supply-chain hardening, selective plugin build contract, and runtime packaging expectations.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build and removes digest-pinned Node/Bun inputs, selective `OPENCLAW_EXTENSIONS` staging, variant handling, and several retry/cache/runtime packaging paths.

## Best Possible Solution

Preserve current main's Docker supply-chain and plugin-build contracts, and split any simplification into narrower maintainer-guided changes that keep pinned inputs, selective plugin staging, runtime metadata, and aligned tests/docs/workflows.

## Reproduction Assessment

Yes. PR-head source inspection reproduces the review findings against `Dockerfile`, and current-main tests, docs, setup scripts, and workflows encode the opposite Docker contracts.

## Solution Assessment

No. The broad simplification removes established compatibility and supply-chain guarantees; a safer path would preserve the contracts and simplify only isolated implementation details after Docker/security owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This reintroduces `# syntax=docker/dockerfile:1.7`, which forces an external Dockerfile frontend pull during builds. Current main removed that directive and added coverage rejecting it, so this branch would undo that hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch switches the build/runtime bases to rolling `node:24-*` tags and runs the Bun installer via `curl | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned image, so this regresses reproducibility and supply-chain hardening.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, workflows, and tests still use that build arg to control which plugin dependencies are baked into the image.
  - confidence: 0.95
- **[P2] Redeclare the apt upgrade build arg:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the first `FROM`, but this check runs in the runtime stage where the arg is not in scope unless redeclared. The opt-out expands empty, so the upgrade still runs when callers pass `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0`.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage only copies selected app files from `build`, but current main also copies `pnpm-workspace.yaml` and `patches` after pruning so container-local pnpm workflows can resolve patched dependencies. Dropping those files regresses the runtime packaging contract covered by current tests.
  - confidence: 0.84
