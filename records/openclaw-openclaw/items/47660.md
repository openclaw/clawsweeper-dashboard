## Summary

Keep this PR open for maintainer handling: the branch is a broad Dockerfile rewrite with current blocking review feedback, merge conflicts, and security/runtime regressions against current main rather than an obsolete or safely closeable cleanup.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build and removes several current build/runtime controls, including selective plugin dependency staging, digest-pinned image inputs, pinned Bun copy, retry loops, and runtime staging.

## Best Possible Solution

Keep the Dockerfile readability work scoped to smaller owner-reviewed patches that preserve pinned inputs, selective plugin dependency staging, runtime-assets pruning/validation, runtime packages, tini/state-dir behavior, OCI metadata, and aligned docs/tests.

## Reproduction Assessment

Yes. Source inspection of PR head f918dda against current main shows the regressions directly, and current Docker tests/docs/scripts encode the expected contracts without requiring a Docker run.

## Solution Assessment

No. The proposed two-stage simplification removes established Docker security, packaging, and runtime guarantees; the maintainable path is narrower cleanup that keeps those contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Restore runtime TLS, Python, and init support:** `Dockerfile:107`
  - body: The runtime apt list drops `ca-certificates`, `python3`, and `tini`/ENTRYPOINT behavior from current main. On `node:24-bookworm-slim`, that can break HTTPS egress, Python-backed skills, and long-running container signal/reaping behavior.
  - confidence: 0.95
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: Using rolling `node:24-*` tags and `curl | bash` for Bun reverts current main's digest-pinned Node/Bun image inputs. Release builds lose reviewed reproducible inputs and execute unpinned remote code during image construction.
  - confidence: 0.97
- **[P2] Preserve the runtime-assets validation stage:** `Dockerfile:77-78`
  - body: Collapsing runtime pruning into the build stage and copying runtime bytes from `build` skips current main's bundled-plugin postinstall, plugin-dist pruning, package export validation, and later runtime `pnpm-workspace.yaml`/`patches`/`qa` copies.
  - confidence: 0.92
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all `extensions` before `pnpm install` makes every plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`, regressing the documented selective dependency baking and install-layer cache behavior.
  - confidence: 0.94
- **[P2] Remove the external Dockerfile frontend directive:** `Dockerfile:1`
  - body: The root Dockerfile now asks BuildKit to fetch `docker/dockerfile:1.7` before building. Current main has regression coverage requiring root Dockerfiles to avoid external frontend pulls.
  - confidence: 0.92
- **[P2] Restore the Docker apt key primary-key guard:** `Dockerfile:172`
  - body: The optional Docker CLI block verifies only the first fingerprint and then trusts the whole dearmored keyring. Current main first rejects key files with more than one primary public key.
  - confidence: 0.88
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: The runtime image sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions`, reintroducing a legacy discovery override that current main explicitly tests against.
  - confidence: 0.81
- **[P2] Keep the runtime state directory pre-created:** `Dockerfile:197`
  - body: The branch drops current main's pre-created `/home/node/.openclaw` directory before `USER node`, so named volumes can again inherit root-owned state instead of the expected node-owned 0700 directory.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain controls and apt-key validation in a security-sensitive build path.

Concerns:

- **[high] Unpinned Bun installer executes during build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash`, executing remote code as part of the Docker build and undoing current main's pinned Bun image copy.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use literal `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive causes BuildKit to resolve `docker/dockerfile:1.7` before the build, which current main specifically guards against for the root Dockerfile.
  - confidence: 0.92
- **[medium] Docker apt key guard trusts extra primary keys:** `Dockerfile:172`
  - body: The optional Docker CLI block checks the first fingerprint but no longer requires exactly one primary public key before dearmoring, so extra keys in the same file would also be trusted by apt.
  - confidence: 0.88
