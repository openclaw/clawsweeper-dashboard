## Summary

Keep open. The PR is a broad Docker infra refactor that now conflicts with current main and would regress protected Docker build contracts around pinned inputs, selective plugin dependency staging, runtime package metadata, and container supply-chain hardening.

## What This Changes

The PR rewrites the root Dockerfile from a multi-stage, pinned, selective plugin build into a simpler two-stage Docker build with rolling image tags and unconditional plugin source copying.

## Best Possible Solution

Preserve current main's Docker supply-chain and plugin-build contracts, and split any simplification into narrower owner-reviewed changes that keep pinned inputs, selective plugin staging, runtime metadata, docs, tests, and workflows aligned.

## Reproduction Assessment

Yes. PR-head source inspection at `f918dda76d3fdbb24f575029df510ad3ee952900` shows the regressing Dockerfile paths, and current-main Dockerfile tests encode the incompatible expected contracts.

## Solution Assessment

No. The broad refactor is not the safest maintainable solution because it removes established compatibility and supply-chain guarantees; a narrower Docker-owner-guided cleanup should preserve those contracts.

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This branch adds `# syntax=docker/dockerfile:1.7`, which forces an external Dockerfile frontend pull during builds. Current main has coverage rejecting syntax directives for this Dockerfile, so merging this would undo that hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The build and runtime stages use rolling `node:24-*` tags and install Bun with `curl | bash`. Current main pins the Node and Bun image inputs and copies Bun from the pinned image, so this regresses reproducibility and reintroduces unverified remote code execution during image construction.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, workflows, and tests still use that build arg to control which plugin dependencies are included.
  - confidence: 0.95
- **[P2] Redeclare the apt upgrade build arg in the runtime stage:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs in the runtime stage where the arg is not in scope unless redeclared. The opt-out expands empty, so `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` still runs the upgrade path.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies only selected app files from `build`, but current main copies `pnpm-workspace.yaml` and `patches` after pruning so container-local pnpm workflows can resolve patched dependencies. Dropping those files breaks the runtime packaging contract covered by current tests.
  - confidence: 0.84
- **[P2] Keep slim-runtime utilities and init behavior:** `Dockerfile:106-108`
  - body: The runtime install omits current main's explicit `ca-certificates`, `python3`, and `tini` setup and the final `ENTRYPOINT ["tini", "-s", "--"]`. That can regress HTTPS trust, Python-backed workflows, and signal/zombie handling in the container.
  - confidence: 0.82
- **[P2] Do not override bundled plugin discovery in runtime images:** `Dockerfile:120-122`
  - body: This branch sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions`, while current main has a regression test asserting the Dockerfile does not override bundled plugin discovery with that legacy environment variable.
  - confidence: 0.78
