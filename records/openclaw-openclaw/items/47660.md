## Summary

Keep open. Current main does not implement the proposed Dockerfile simplification, and the PR branch still conflicts with maintained Docker build contracts around digest-pinned base images, selective plugin dependency baking, slim-runtime TLS support, and Docker build-arg behavior.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes the ext-deps/runtime variant stages and retry loops, copies all plugins before dependency install, and switches the Node base images from pinned digests to mutable tags.

## Best Possible Solution

If Dockerfile simplification remains desirable, rework it on current main while preserving the maintained contracts: digest-pinned Node bases and base OCI labels, selective OPENCLAW_EXTENSIONS dependency baking, current slim-runtime CA/python/runtime utility setup, bundled plugin runtime-dependency staging, and aligned Docker docs/workflows.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: inspect the PR head Dockerfile lines for mutable FROM tags, full extensions COPY, missing runtime CA install, and the out-of-scope OPENCLAW_DOCKER_APT_UPGRADE check; focused builds with OPENCLAW_EXTENSIONS=matrix, OPENCLAW_DOCKER_APT_UPGRADE=0, and an HTTPS runtime smoke would exercise the same paths.

## Solution Assessment

No. A broad two-stage rewrite that removes digest pinning and selective plugin dependency behavior is not the narrowest maintainable fix; a safer simplification would reduce Dockerfile complexity only after keeping the existing build, release, and security contracts intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Install CA certificates in the slim runtime:** `Dockerfile:106-107`
  - body: The patch hardcodes `node:24-bookworm-slim` but the runtime package install omits `ca-certificates` and `update-ca-certificates`. Current main already fixed this slim-runtime regression because HTTPS outbound calls fail without a populated CA bundle, so channel/plugin startup can break in the produced image.
  - confidence: 0.92
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying the whole `extensions/` tree before `pnpm install` makes every plugin workspace participate in the install. Docs, setup, CI, and release builds still pass `OPENCLAW_EXTENSIONS` to preinstall only selected plugin dependencies, so this regresses the documented build arg and can bloat or invalidate Docker dependency layers unexpectedly.
  - confidence: 0.94
- **[P2] Keep Node base images digest-pinned:** `Dockerfile:29`
  - body: Switching the build stage to `node:24-bookworm` and the runtime stage to `node:24-bookworm-slim` makes Docker builds consume mutable tags and drops the runtime base digest metadata. That weakens the existing reproducible-build and release-review contract.
  - confidence: 0.91
- **[P2] Re-declare OPENCLAW_DOCKER_APT_UPGRADE in runtime:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but the runtime-stage `RUN` reads it after a later `FROM`. Docker does not keep that global ARG in stage scope unless it is re-declared, so the opt-out build arg is ignored and apt upgrades always run.
  - confidence: 0.88
- **[P3] Refresh stale Dockerfile comments:** `Dockerfile:31-33`
  - body: The Bun install comment still promises retry behavior even though the retry loop was removed, and the header still describes removed variant/ext-deps behavior. Leaving those instructions stale will mislead future Docker maintainers.
  - confidence: 0.86
