## Label Justifications

- `P1`: The PR touches the root Docker image and would regress security-sensitive and availability-sensitive runtime behavior for self-hosted users.
- `merge-risk: 🚨 compatibility`: The diff removes documented Docker build args, base-image metadata, selective plugin dependency behavior, runtime copies, and state-volume setup used by existing deployments.
- `merge-risk: 🚨 security-boundary`: The diff reintroduces unpinned remote installer execution, removes digest-pinned image inputs, bypasses Docker apt key hardening, and weakens state-directory secrecy guarantees.
- `merge-risk: 🚨 availability`: The runtime image can lose CA certificates, Python, tini, browser-install TLS, plugin hotfixes, and named-volume writability, all of which can break container startup or runtime flows.

## Summary

Keep open for Docker/security owner judgment: the branch simplifies the Dockerfile by removing current-main supply-chain, runtime-hardening, plugin-staging, and documented Docker contracts, and the supplied proof only covers a default build plus CLI smoke.

## What This Changes

This PR rewrites the root Dockerfile into a two-stage build while removing digest-pinned/staged inputs, selective plugin dependency staging, and several runtime setup paths.

## Best Possible Solution

Replace this branch with a narrower Docker readability cleanup that preserves current pinned inputs, manifest-only plugin staging, runtime-assets validation, runtime package hardening, docs contracts, and focused Docker proof.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main shows the removed Docker contracts, and reviewer comments include a focused TLS repro for the PR's runtime package set.

## Solution Assessment

No. Dockerfile readability is a valid goal, but this patch is not the best solution because it removes deliberate hardening and compatibility behavior instead of refactoring around it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: This branch switches the build stage to rolling `node:24-bookworm` and runs the Bun installer with `curl | bash`. Current main pins Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime TLS, Python, and tini support:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the tini entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.95
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Restore the Docker apt key primary-key guard:** `Dockerfile:170-177`
  - body: The Docker CLI install path verifies only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.9
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node` mode `0700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.88
- **[P2] Keep the apt-upgrade control in scope:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before `FROM`, so the runtime `RUN` sees it as empty and the opt-in upgrade lever no longer behaves as the Dockerfile comments imply.
  - confidence: 0.82
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch an external Dockerfile frontend before the build. Current main has coverage rejecting this for the root Dockerfile to avoid adding another supply-chain input.
  - confidence: 0.9

  - confidence: 0.94
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from `build` skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88

  - body: The image no longer pre-creates `/home/node/.openclaw` as `node:node` mode `0700`, weakening the expected credential-directory ownership and permission boundary for first-run named volumes.
