## Summary

Keep open: the branch removes deliberate Docker supply-chain, runtime validation, and hardening contracts that current main preserves in source, docs, and guard tests.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build using rolling Node image tags, curl-installed Bun, unconditional extension copying, and direct runtime copies from the build stage.

## Best Possible Solution

Preserve current main's pinned inputs, manifest-only dependency staging, runtime-assets validation, runtime packages, entrypoint, GPG guard, and state-directory ownership; any readability cleanup should be a smaller Docker-owner-reviewed refactor that keeps those contracts intact.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main plus Dockerfile guard tests gives a high-confidence reproduction path for the regressions without rebuilding the image.

## Solution Assessment

No. Collapsing the Dockerfile by removing pinned inputs, runtime validation, and hardening is not the narrowest maintainable solution; a safer cleanup would preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and installs Bun with curl piped to bash. Current main pins the Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits ca-certificates, python3, and tini, and the final image no longer uses the tini entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.95
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from build bypasses the runtime-assets prune, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for pnpm-workspace.yaml, patches, and qa.
  - confidence: 0.94
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full extensions tree before pnpm install makes every plugin source change invalidate the install layer and bypasses the documented OPENCLAW_EXTENSIONS manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI install path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.9
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the node:node 0700 /home/node/.openclaw creation before USER node can make first-run named volumes unwritable or lose the credential-directory secrecy assumed by Docker deployments.
  - confidence: 0.88
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch an external Dockerfile frontend before the build. Current main has coverage rejecting this for the root Dockerfile to avoid another supply-chain input.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening protections that current main explicitly preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs curl piped to bash instead of copying Bun from a pinned image, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use node:24-bookworm and node:24-bookworm-slim tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[high] Runtime TLS, Python, and init hardening are omitted:** `Dockerfile:107`
  - body: The runtime omits ca-certificates, update-ca-certificates, python3, tini, and the tini entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.95
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from build skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.91
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.9
- **[medium] Credential state directory hardening is removed:** `Dockerfile:197`
  - body: The final image no longer pre-creates /home/node/.openclaw as node:node mode 0700 before USER node, which can break named-volume writes or weaken credential-directory permissions.
  - confidence: 0.88
- **[medium] External Dockerfile frontend is added:** `Dockerfile:1`
  - body: The syntax directive causes BuildKit to resolve docker/dockerfile:1.7 before the build, adding another remote build input that current main rejects.
  - confidence: 0.92
