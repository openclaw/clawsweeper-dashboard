## Summary

This PR should stay open for normal maintainer handling, but it is not mergeable as-is: the branch removes current main's Docker supply-chain, runtime-validation, and runtime-hardening contracts while the submitted proof only covers the default smoke path.

## What This Changes

The PR branch rewrites the root Dockerfile into a two-stage build and removes digest-pinned base images, selective plugin dependency staging, Bun image copying, runtime-assets validation, and several optional runtime/build guards.

## Best Possible Solution

Keep current main's Docker security and runtime contracts; any readability cleanup should preserve pinned inputs, manifest staging, runtime validation, TLS/Python/tini, GPG guard, and state-dir permissions.

## Reproduction Assessment

Yes. Source inspection against current main and Dockerfile guard tests gives a high-confidence way to reproduce the regressions without a full image build.

## Solution Assessment

No. The branch trades established Docker supply-chain and runtime contracts for readability, while a smaller Docker-owner-reviewed cleanup would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and runs the Bun installer with `curl | bash`. Current main pins Node and Bun image refs and copies Bun from the pinned image, so this reopens build reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the `tini` entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.94
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The `# syntax=docker/dockerfile:1.7` directive makes BuildKit resolve an external frontend before the build. Current main has coverage rejecting root Dockerfile syntax directives to avoid that extra supply-chain input.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying runtime files directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-177`
  - body: The Docker CLI install path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node 700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening contracts that current main now explicitly preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch a Dockerfile frontend image before the build, adding an external build input that current main explicitly rejects.
  - confidence: 0.93
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
