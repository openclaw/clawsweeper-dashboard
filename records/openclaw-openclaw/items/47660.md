## Summary

Keep open: the PR is not safe cleanup as written. Current main explicitly preserves Docker supply-chain, runtime, and validation contracts that this branch removes, and the existing review discussion identifies concrete security and runtime regressions rather than a small obsolete diff.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, copies all extensions before install, uses rolling Node tags plus curl-installed Bun, and copies runtime files directly from the build stage.

## Best Possible Solution

Preserve the current Docker hardening and runtime contracts; if readability cleanup is still desired, split it into a smaller Docker-owner-reviewed refactor that keeps pins, manifest staging, runtime-assets validation, docs, and guard tests intact.

## Reproduction Assessment

Yes. Source inspection of the PR head against current main and the Dockerfile guard tests gives a high-confidence reproduction path for the regressions without rebuilding the image.

## Solution Assessment

No. Collapsing the Dockerfile by deleting pinned inputs, runtime validation, and hardening is not the narrowest maintainable solution; a safer cleanup would preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The PR switches the build to rolling `node:24-bookworm` and `curl | bash` for Bun. Current main pins Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and remote-code execution risk.
  - confidence: 0.97
- **[P1] Restore runtime TLS, Python, and tini support:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, and the final image no longer uses the tini entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes every plugin source change invalidate the install layer and bypasses the documented `OPENCLAW_EXTENSIONS` manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Restore the Docker apt key primary-key guard:** `Dockerfile:170-177`
  - body: The Docker CLI install path verifies only the first fingerprint before dearmoring the whole key file. Current main rejects multi-key files before trusting the downloaded apt keyring.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the `node:node` mode `0700` `/home/node/.openclaw` creation before `USER node` means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch an external Dockerfile frontend before the build. Current main has coverage rejecting this for the root Dockerfile to avoid another supply-chain input.
  - confidence: 0.93
- **[P3] Re-declare the apt-upgrade arg in the runtime stage:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before `FROM` but not in the runtime stage, so the shell sees it as empty and the opt-out condition always runs the upgrade path.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening controls that current main explicitly preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying Bun from a pinned image, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of digest-pinned image args and OCI base-image labels, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the tini entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from `build` skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:172`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
- **[medium] External Dockerfile frontend is introduced:** `Dockerfile:1`
  - body: The `# syntax=docker/dockerfile:1.7` directive makes BuildKit fetch an external frontend before the Docker build, adding a supply-chain input that current main tests reject.
  - confidence: 0.93
