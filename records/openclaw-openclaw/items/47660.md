## Summary

Keep open, but do not merge as-is. The branch is a broad Docker infrastructure refactor that conflicts with current main and removes or regresses established Docker security and compatibility contracts around pinned inputs, selective plugin dependency staging, and current runtime packaging checks.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes digest-pinned base/Bun inputs, removes selective plugin staging and runtime variant handling, and simplifies APT/Corepack install paths.

## Best Possible Solution

Preserve current main's Docker contracts, and if simplification is still wanted, split it into narrower maintainer-guided changes that keep pinned inputs, selective plugin staging, runtime metadata, no external frontend pull, and docs/tests/workflows aligned.

## Reproduction Assessment

Yes for the review findings. PR-head source inspection shows the unpinned inputs, `curl | bash` Bun install, external frontend directive, unconditional plugin copying, and stage-scoped ARG issue; current-main docs, workflows, and tests encode the opposite contracts.

## Solution Assessment

No. The broad simplification removes established supply-chain and compatibility guarantees; the safer approach is a maintainer-directed cleanup that preserves digest-pinned images, pinned Bun copy, `OPENCLAW_EXTENSIONS`, and current runtime packaging checks.

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This reintroduces `# syntax=docker/dockerfile:1.7`, which forces an external Dockerfile frontend pull during builds. Current main removed that directive and added a test rejecting it, so this branch would undo that Docker hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch switches the build/runtime bases to rolling `node:24-*` tags and runs `curl -fsSL https://bun.sh/install | bash`. Current main pins Node and Bun image inputs and copies Bun from a pinned image, so this regresses reproducibility and supply-chain hardening.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, release workflows, and tests still use that build arg to control which plugin dependencies are baked into the image.
  - confidence: 0.95
- **[P2] Redeclare the apt upgrade build arg:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared before the first `FROM`, but this check runs in the runtime stage where the arg is not in scope unless redeclared. The opt-out expands empty, so the upgrade still runs when callers pass `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0`.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage only copies selected app files from `build`, but current main also copies `pnpm-workspace.yaml` and `patches` after pruning so container-local pnpm workflows can resolve patched dependencies. Dropping those files regresses the runtime packaging contract covered by current tests.
  - confidence: 0.84
