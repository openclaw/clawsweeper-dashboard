## Summary

Keep open: this PR is a broad Dockerfile simplification, but current main now explicitly preserves digest-pinned build inputs, manifest-only plugin staging, runtime-assets validation, TLS/Python/tini runtime support, Docker GPG guard, and state-directory ownership. The branch removes those contracts and needs Docker/security owner judgment rather than cleanup closure.

## What This Changes

The branch rewrites the root Dockerfile into a two-stage build, removes digest-pinned image stages and selective plugin manifest staging, and copies runtime assets directly from the build stage.

## Best Possible Solution

Keep current main's Docker security and runtime contracts; any readability cleanup should be a smaller Docker-owner-reviewed refactor that preserves those contracts.

## Reproduction Assessment

Yes. Source inspection of the PR head compared with current main and Dockerfile guard tests gives a high-confidence reproduction path for the regressions without rebuilding the image. The attached log only proves the default build and CLI smoke path.

## Solution Assessment

No. Collapsing the Dockerfile into two stages by removing pinned inputs, runtime validation, and hardening is not the narrowest maintainable solution; a safer cleanup would preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch builds from rolling Node tags and runs the Bun installer with curl piped to bash. Current main pins Node and Bun image refs and copies Bun from the pinned image, so this reopens reproducibility and supply-chain risk.
  - confidence: 0.97
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits ca-certificates, python3, and tini, and the final image no longer uses the tini entrypoint. That can break HTTPS egress, Python-backed skills, and long-running process handling.
  - confidence: 0.94
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The syntax directive makes BuildKit fetch an external Dockerfile frontend before the build. Current main has coverage rejecting this for the root Dockerfile to avoid another supply-chain input.
  - confidence: 0.93
- **[P2] Preserve manifest-only plugin staging:** `Dockerfile:43`
  - body: Copying the full extensions tree before pnpm install makes every plugin source change invalidate the install layer and bypasses the documented OPENCLAW_EXTENSIONS manifest-only dependency contract.
  - confidence: 0.95
- **[P2] Copy runtime files from the validated assets stage:** `Dockerfile:112-118`
  - body: Copying directly from build bypasses production prune validation, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import checks, and runtime copies for pnpm-workspace.yaml, patches, and qa.
  - confidence: 0.92
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI install path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory:** `Dockerfile:197`
  - body: Dropping the node:node 700 /home/node/.openclaw creation before USER node means named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86
- **[P2] Re-declare the apt-upgrade build arg in the runtime stage:** `Dockerfile:103`
  - body: OPENCLAW_DOCKER_APT_UPGRADE is declared before the first FROM but used in the runtime stage without re-declaration, so build-arg overrides are not visible there and the opt-out path cannot work as written.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff removes Docker supply-chain and runtime-hardening contracts that current main explicitly preserves.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs curl piped to bash instead of copying Bun from a pinned image, executing remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use node:24-bookworm and node:24-bookworm-slim tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:107`
  - body: The runtime omits ca-certificates, update-ca-certificates, python3, tini, and the tini entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Runtime hotfix and package validation are bypassed:** `Dockerfile:112`
  - body: Copying from build skips the runtime-assets postinstall and validation stage, including bundled-plugin hotfixes, plugin dist pruning, dist import checks, and patch-file runtime preservation.
  - confidence: 0.9
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:197`
  - body: Without pre-creating /home/node/.openclaw as node:node 700, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
