## Summary

This PR should stay open for maintainer Docker/security judgment: the branch is a broad infrastructure simplification that now conflicts with current `main` Docker contracts and would regress several maintained release-image safeguards.

## What This Changes

The PR rewrites the root `Dockerfile` into a two-stage Docker build, removes the `ext-deps` and runtime-variant stages, removes retry loops, copies all plugins before dependency install, and switches pinned base images to mutable tags.

## Best Possible Solution

Rework the Dockerfile simplification on current `main` only if it preserves the maintained release-image contracts: pinned Node and Bun sources, selective `OPENCLAW_EXTENSIONS` dependency baking, slim-runtime CA/runtime utilities, plugin runtime-dependency staging, and aligned Docker docs/workflows.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the regressions: mutable base images, full plugin workspace install, slim runtime without CA certificates, stale ARG scoping, and a return to `curl | bash` Bun install.

## Solution Assessment

No. A broad two-stage rewrite that removes or regresses current Docker build and supply-chain contracts is not the best path; a safer simplification would preserve those contracts first and reduce only proven incidental complexity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep Bun sourced from the pinned image:** `Dockerfile:33`
  - body: Current `main` copies Bun from the digest-pinned `OPENCLAW_BUN_IMAGE`. This branch goes back to `curl -fsSL https://bun.sh/install | bash`, so merging it would reintroduce unverified build-time code execution that the Dockerfile has already hardened away.
  - confidence: 0.91
- **[P1] Install CA certificates in the slim runtime:** `Dockerfile:106-107`
  - body: The branch hardcodes `bookworm-slim` but installs only `procps hostname curl git lsof openssl`. Current `main` installs `ca-certificates` and runs `update-ca-certificates` because HTTPS outbound calls fail in the slim image without a populated trust store.
  - confidence: 0.92
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying all of `extensions/` before `pnpm install` makes every plugin workspace participate in the install. Docs, setup, release, and smoke workflows still pass `OPENCLAW_EXTENSIONS` to bake only selected plugin dependencies, so this breaks the documented Docker build arg.
  - confidence: 0.94
- **[P2] Keep Docker base images digest-pinned:** `Dockerfile:29`
  - body: Switching the build stage to `node:24-bookworm` and the runtime stage to `node:24-bookworm-slim` makes release builds consume mutable tags instead of the reviewed digest-pinned base images used by current `main`.
  - confidence: 0.93
- **[P2] Redeclare the apt-upgrade ARG before reading it:** `Dockerfile:81`
  - body: The file declares `OPENCLAW_DOCKER_APT_UPGRADE` before the first `FROM`, then reads it in the runtime stage after a later `FROM`. Docker requires redeclaring a global ARG in each stage, so `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` is ignored here.
  - confidence: 0.86
- **[P3] Refresh stale Dockerfile comments:** `Dockerfile:31-32`
  - body: The header still describes removed variant and ext-deps behavior, and the Bun install comment still promises retry behavior after the retry loop was removed. That leaves misleading instructions in the file this PR is trying to simplify.
  - confidence: 0.86
