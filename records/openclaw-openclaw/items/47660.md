## Summary

Keep open. Current main still treats digest-pinned Docker inputs and `OPENCLAW_EXTENSIONS` as supported Docker contracts, while this PR removes those contracts and reintroduces unverified Bun installer execution, so it needs maintainer infra/security judgment before merge.

## What This Changes

This PR collapses the root Dockerfile to two stages and removes digest-pinned build inputs, retry loops, runtime variant selection, and selective plugin dependency staging.

## Best Possible Solution

Preserve main's Docker security and plugin dependency contracts, and pursue any simplification as a smaller maintainer-directed refactor that keeps pinned inputs, `OPENCLAW_EXTENSIONS`, docs, tests, and workflows aligned.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff against current main shows the branch drops pinned inputs, bypasses selective plugin dependency staging, and loses the runtime-stage APT upgrade arg. The contributor's attached log proves only the default after-change build path.

## Solution Assessment

No. The broad simplification is not the narrowest maintainable path because it removes documented Docker behavior and rolls back supply-chain hardening; targeted cleanup that preserves those contracts would be safer.

  - body: The branch replaces digest-pinned Node and Bun image inputs with moving `node:24-*` tags and reinstates `curl -fsSL https://bun.sh/install | bash`. Current main uses digest-pinned Node and Bun image inputs, so this regresses Docker build reproducibility and supply-chain hardening.

  - body: Copying the whole plugin tree before `pnpm install` makes every plugin workspace participate in dependency installation.   - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs inside the runtime stage. Without re-declaring the ARG in that stage, `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` is ignored and the upgrade path still runs.
  - confidence: 0.86
- **[P3] Update stale Dockerfile comments:** `Dockerfile:12-19`
  - body: The header and nearby comments still describe removed behavior such as selective extension args, runtime variants, digest pinning, and retry logic. If the refactor proceeds, those comments need to match the new Dockerfile so contributors do not follow removed build options.
  - confidence: 0.82
