## Summary

Keep open for maintainer judgment: the branch conflicts with current Dockerfile direction by removing documented plugin dependency staging and weakening Docker supply-chain controls that main now preserves explicitly.

## What This Changes

The PR simplifies the root Dockerfile into two stages while removing digest-pinned base references, Bun/Corepack retry/bootstrap logic, runtime variant switching, and selective plugin dependency staging.

## Best Possible Solution

Preserve main's Docker security and plugin dependency contracts, and pursue any simplification as a smaller maintainer-directed rework that keeps digests, `OPENCLAW_EXTENSIONS`, docs, tests, and release workflows aligned.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR head and current main shows the branch copies all plugin workspaces before install, drops digest-pinned Docker inputs, and executes the Bun installer during the Docker build.

## Solution Assessment

No. The proposed simplification is not the best current solution because it removes documented Docker behavior and weakens supply-chain controls; targeted cleanup on current main would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch replaces digest-pinned Node image ARGs with moving `node:24-*` tags and installs Bun by piping `https://bun.sh/install` to `bash`. Current main now uses digest-pinned Node and Bun image inputs, so this would regress Docker build reproducibility and supply-chain hardening.
  - confidence: 0.95
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying the whole plugin tree before `pnpm install` makes every plugin workspace participate in dependency installation. The Docker docs, setup script, install smoke, and release workflow still use `OPENCLAW_EXTENSIONS` to select bundled plugin dependencies, so this breaks a documented Docker build contract.
  - confidence: 0.93
- **[P2] Re-declare the APT upgrade arg in the runtime stage:** `Dockerfile:103`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs inside the runtime stage. Without re-declaring the ARG in that stage, `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` is ignored and the upgrade path still runs.
  - confidence: 0.86
