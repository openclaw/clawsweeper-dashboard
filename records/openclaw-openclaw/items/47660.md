## Summary

Keep open. The PR is a broad Docker refactor that regresses current main's digest-pinned build inputs, pinned Bun copy, selective plugin dependency staging, runtime metadata, and slim-runtime init/certificate contracts; approving that needs human Docker/security compatibility review, not cleanup close.

## What This Changes

The PR rewrites the root Dockerfile from a multi-stage pinned build into a two-stage build with rolling Node tags, Bun installed by `curl | bash`, unconditional plugin copying, and simplified runtime setup.

## Best Possible Solution

Preserve current main's pinned Docker inputs, selective plugin staging, runtime metadata, and init/certificate behavior; split any readability cleanup into narrower Docker-owner-reviewed changes with tests and docs kept aligned.

## Reproduction Assessment

Yes. PR-head source inspection at `f918dda76d3fdbb24f575029df510ad3ee952900` plus current-main Dockerfile tests/docs gives a high-confidence reproduction path for the incompatible contracts, though I did not run Docker in this read-only review.

## Solution Assessment

No. The broad simplification is not the safest maintainable solution because it removes established Docker supply-chain, plugin-build, and runtime packaging guarantees; a narrower cleanup should preserve those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Keep the Dockerfile frontend implicit:** `Dockerfile:1`
  - body: This adds a `# syntax=docker/dockerfile:1.7` directive, which makes BuildKit resolve an external Dockerfile frontend before the build. Current main has coverage rejecting syntax directives for the root Dockerfile, so merging this would undo that hardening.
  - confidence: 0.9
- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The build and runtime stages use rolling `node:24-*` tags and install Bun with `curl | bash`. Current main pins the Node and Bun image inputs and copies Bun from the pinned image, so this regresses reproducibility and reintroduces unverified remote code execution during image construction.
  - confidence: 0.97
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all of `extensions` before `pnpm install` makes every bundled plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, workflows, and tests still use that build arg to control which plugin dependencies are included.
  - confidence: 0.95
- **[P2] Redeclare runtime-stage build args before use:** `Dockerfile:103-104`
  - body: `OPENCLAW_DOCKER_APT_UPGRADE` is declared only before the first `FROM`, but this check runs in the runtime stage where that arg is not in scope unless redeclared. The opt-out expands empty, so `--build-arg OPENCLAW_DOCKER_APT_UPGRADE=0` still runs the upgrade path.
  - confidence: 0.86
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies selected app files from `build` but drops `pnpm-workspace.yaml` and `patches`. Current main copies those after pruning so container-local pnpm workflows can resolve patched dependencies, and install-smoke checks for that contract.
  - confidence: 0.84
- **[P2] Keep runtime certificates, Python, and init behavior:** `Dockerfile:106-107`
  - body: The slim runtime install omits current main's explicit `ca-certificates`, `python3`, and `tini` setup and the final `ENTRYPOINT ["tini", "-s", "--"]`. That can regress HTTPS trust, Python-backed workflows, and signal/zombie handling in long-running containers.
  - confidence: 0.82
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: This sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions`, while current main has a regression test asserting the Dockerfile does not override bundled plugin discovery with that legacy environment variable.
  - confidence: 0.78
