## Summary

Keep this PR open for human Docker/security review rather than cleanup-close: the branch is a broad infra rewrite that removes current main's Docker supply-chain, runtime, and packaging contracts, and recent reviewer comments identify concrete regressions that still apply.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build while removing digest-pinned image inputs, selective plugin staging, runtime asset pruning, variant handling, retry logic, and several runtime hardening steps.

## Best Possible Solution

Keep current main's Docker security/runtime contracts and pursue readability through narrow Docker-owner-reviewed cleanups that preserve pinned inputs, selective plugin staging, runtime-assets pruning, TLS/Python/tini, Docker GPG validation, and state-directory permissions.

## Reproduction Assessment

Yes. Source inspection of PR head f918dda76d3fdbb24f575029df510ad3ee952900 against current main and `src/dockerfile.test.ts` gives a high-confidence reproduction path for the regressions; the attached log only proves the default smoke path.

## Solution Assessment

No. The broad two-stage simplification removes established Docker supply-chain, runtime, and packaging guarantees; a safer solution would split readability-only edits that keep those contracts intact.

Overall confidence: 0.96

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-34`
  - body: The branch builds from rolling Node tags and installs Bun with `curl | bash`, while current main pins Node and Bun image inputs and copies Bun from a pinned image. This reopens supply-chain and reproducibility risk for every Docker build.
  - confidence: 0.97
- **[P2] Remove the external Dockerfile frontend:** `Dockerfile:1`
  - body: The build log shows this branch resolving `docker/dockerfile:1.7` before the build. Current main has coverage rejecting root Dockerfile syntax directives to avoid adding that extra supply-chain input.
  - confidence: 0.9
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying the full `extensions` tree before `pnpm install` makes all plugin source participate in the install layer and ignores the documented `OPENCLAW_EXTENSIONS` dependency contract.
  - confidence: 0.95
- **[P1] Restore runtime certificates, Python, and tini:** `Dockerfile:106-107`
  - body: The slim runtime package list omits `ca-certificates`, `python3`, and `tini`, plus `update-ca-certificates` and the `tini` entrypoint. That can break HTTPS egress, Python-backed skills, and long-running container signal/zombie handling.
  - confidence: 0.94
- **[P2] Copy runtime files from the pruned assets stage:** `Dockerfile:112-118`
  - body: Copying directly from `build` skips current main's production prune, bundled-plugin postinstall hotfixes, plugin dist pruning, dist import validation, and runtime copies for `pnpm-workspace.yaml`, `patches`, and `qa`.
  - confidence: 0.92
- **[P2] Restore the Docker apt key multi-key guard:** `Dockerfile:170-172`
  - body: The Docker CLI path compares only the first fingerprint before dearmoring the whole key file. Current main first requires exactly one primary public key so a multi-key file cannot pass while adding extra trusted keys.
  - confidence: 0.88
- **[P2] Pre-create the OpenClaw state directory before USER node:** `Dockerfile:213`
  - body: The branch drops the `node:node 700` `/home/node/.openclaw` creation and assertion before switching to the non-root user. Named-volume first runs can become unwritable or lose the intended credential-directory secrecy.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff weakens Docker supply-chain and runtime hardening by removing pinned inputs, runtime trust/bootstrap packages, postinstall safeguards, Docker apt key validation, and state-directory protections.

Concerns:

- **[high] Unpinned Bun installer executes during image build:** `Dockerfile:33`
  - body: The build stage runs `curl -fsSL https://bun.sh/install | bash` instead of copying the pinned Bun binary from current main, executing unaudited remote code during image construction.
  - confidence: 0.97
- **[medium] Base images move to rolling tags:** `Dockerfile:29`
  - body: The build and runtime stages use `node:24-bookworm` and `node:24-bookworm-slim` tags instead of current main's digest-pinned image args, reducing reproducibility for release and downstream builds.
  - confidence: 0.95
- **[medium] External Dockerfile frontend is fetched:** `Dockerfile:1`
  - body: The proof log shows BuildKit resolving `docker/dockerfile:1.7`, adding a new external build input that current main's Dockerfile tests reject.
  - confidence: 0.9
- **[high] Runtime TLS and init hardening are omitted:** `Dockerfile:106`
  - body: The slim runtime omits `ca-certificates`, `update-ca-certificates`, `python3`, `tini`, and the `tini` entrypoint, which can break secure egress, Python-backed skills, and process handling.
  - confidence: 0.94
- **[medium] Bundled plugin postinstall hardening is skipped:** `Dockerfile:112`
  - body: Copying from `build` bypasses `scripts/postinstall-bundled-plugins.mjs`, including the current Baileys encrypted-stream flush hotfix applied during runtime asset preparation.
  - confidence: 0.86
- **[medium] Docker apt key multi-key guard is removed:** `Dockerfile:170`
  - body: The Docker CLI install path verifies the first fingerprint but would still dearmor and trust every key in the downloaded file; current main rejects files with more than one primary public key first.
  - confidence: 0.88
- **[medium] Credential state directory protection is dropped:** `Dockerfile:213`
  - body: Without pre-creating `/home/node/.openclaw` as `node:node 700`, named-volume deployments can start with root-owned or permissive state paths for credentials and auth profiles.
  - confidence: 0.86
