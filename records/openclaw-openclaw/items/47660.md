## Summary

Keep open for maintainer review: current main has not adopted the two-stage Dockerfile, and the proposed branch is stale relative to current Docker hardening. The patch would regress selective plugin dependency baking, base-image pinning, and the slim-runtime certificate fix, so it is not ready to merge as-is.

## What This Changes

The PR branch rewrites the root Dockerfile into a two-stage build, copies all plugins before install, removes retry loops, uses mutable Node base image tags, and hardcodes a slim runtime image.

## Best Possible Solution

If Dockerfile simplification remains desirable, it should be rebased onto current main and keep the maintained contracts: digest-pinned Node bases and OCI base labels, selective OPENCLAW_EXTENSIONS dependency baking, current slim-runtime CA setup, current bundled runtime dependency staging, and aligned Docker docs/workflows.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to build the PR branch with OPENCLAW_EXTENSIONS=matrix and compare installed plugin workspaces/deps against current main, then run an HTTPS fetch in the slim runtime; the code and current-main regression history already show why those paths are affected.

## Solution Assessment

No. A two-stage rewrite that removes dependency selection and digest pinning is not the narrowest maintainable solution; a safer refactor would reduce comments/stages only after preserving the current Docker contracts and security posture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Install CA certificates in the slim runtime:** `Dockerfile:106-107`
  - body: The patch hardcodes `node:24-bookworm-slim` but the runtime package list does not install `ca-certificates` or run `update-ca-certificates`. Current main already fixed this exact slim-runtime regression because HTTPS outbound calls fail without a populated CA bundle, so channel/plugin startup can break in the produced image.
  - confidence: 0.92
- **[P2] Preserve selective plugin dependency baking:** `Dockerfile:43`
  - body: Copying the whole `extensions/` tree before `pnpm install` makes every plugin workspace participate in the install, while docs, setup, CI, and release builds still pass `OPENCLAW_EXTENSIONS` to preinstall only selected plugin dependencies. This regresses the documented build arg and can bloat or invalidate Docker dependency layers unexpectedly.
  - confidence: 0.94
- **[P2] Keep Node base images digest-pinned:** `Dockerfile:29`
  - body: Switching the build stage from the digest-pinned ARG to `node:24-bookworm` makes Docker builds consume a mutable tag; the runtime stage does the same with `node:24-bookworm-slim` and drops base digest metadata. That weakens the existing reproducible-build and release-review contract.
  - confidence: 0.9
- **[P3] Refresh stale Dockerfile comments:** `Dockerfile:31-33`
  - body: The Bun comment still promises retry behavior even though the retry loop was removed, and nearby header/runtime comments still describe removed ext-deps and full-bookworm variant behavior. Leaving those instructions stale will mislead future Docker maintainers.
  - confidence: 0.86
