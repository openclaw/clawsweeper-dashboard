## Summary

Keep open. The PR is a broad Dockerfile simplification, but against current main it would regress established Docker supply-chain, plugin dependency staging, and runtime image contracts; the attached build log proves a default smoke only, not the risky optional and security-sensitive paths.

## What This Changes

The PR rewrites the root Dockerfile into a two-stage build, removes selective plugin staging/runtime variant/digest pinning/retry behavior, and uses rolling Node tags plus a remote Bun installer.

## Best Possible Solution

Preserve current main's pinned inputs, selective plugin staging, runtime metadata/certificates/init behavior, and split any readability cleanup into smaller Docker-owner-reviewed changes with tests and docs kept aligned.

## Reproduction Assessment

Yes. Source inspection of PR head against current main shows the regressions directly, and current Docker tests/docs/workflows encode the expected contracts without needing a Docker run.

## Solution Assessment

No. The broad simplification removes established security, packaging, and runtime guarantees; the maintainable path is narrower cleanup that preserves those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep Docker build inputs pinned:** `Dockerfile:29-33`
  - body: The branch uses rolling `node:24-*` tags and installs Bun with `curl | bash`. Current main pins Node and Bun image inputs and copies Bun from the pinned image, so this would regress reproducibility and reintroduce unverified remote code execution during image construction.
  - confidence: 0.97
- **[P2] Keep the root Dockerfile frontend implicit:** `Dockerfile:1`
  - body: Adding `# syntax=docker/dockerfile:1.7` makes BuildKit resolve an external Dockerfile frontend before the build. Current main has a regression test rejecting syntax directives for the root Dockerfile, so this undoes that hardening.
  - confidence: 0.93
- **[P2] Preserve selective plugin dependency staging:** `Dockerfile:43`
  - body: Copying all plugin sources before `pnpm install` makes every plugin workspace participate regardless of `OPENCLAW_EXTENSIONS`. Current docs, setup scripts, workflows, and tests still use that build arg to control which plugin dependencies are baked into the image.
  - confidence: 0.95
- **[P2] Keep runtime certificates, Python, and init behavior:** `Dockerfile:106-108`
  - body: The slim runtime install omits current main's explicit `ca-certificates`, `python3`, and `tini` setup and the final `ENTRYPOINT ["tini", "-s", "--"]`. That can regress HTTPS trust, Python-backed workflows, and signal/zombie handling in long-running containers.
  - confidence: 0.88
- **[P2] Copy runtime workspace metadata and patches:** `Dockerfile:112-118`
  - body: The runtime stage copies selected app files but drops `pnpm-workspace.yaml` and `patches`. Current main copies both after pruning, and install-smoke checks that patched dependencies resolve inside the container.
  - confidence: 0.87
- **[P2] Do not override bundled plugin discovery:** `Dockerfile:122`
  - body: This sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions`, while current main has a regression test asserting the Dockerfile does not override bundled plugin discovery with that legacy environment variable.
  - confidence: 0.8
- **[P2] Preserve the Docker apt key multi-key guard:** `Dockerfile:172`
  - body: The Docker CLI optional block accepts the first fingerprint from the downloaded key file. Current main first requires exactly one primary public key so apt cannot trust extra keys bundled in the same file.
  - confidence: 0.84
