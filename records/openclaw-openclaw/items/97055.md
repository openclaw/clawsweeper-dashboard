## Summary

Keep open: current main still drops the configured custom Ollama memory provider id before endpoint lookup, and the issue is already paired with an open candidate fix PR that should be reviewed rather than closing the issue early.

## What This Changes

The issue asks memory search embeddings to honor a custom Ollama-backed `memorySearch.provider` id and route to that provider's configured `baseUrl` instead of the base `ollama` endpoint.

## Best Possible Solution

Review and land a focused fix that preserves the requested provider id through gateway memory-specific adapter creation and the Ollama memory adapter, with regression coverage and explicit cache identity upgrade handling.

## Reproduction Assessment

Yes. Source inspection shows current main resolves a custom provider id to the Ollama memory adapter, then drops or overwrites that id before `resolveConfiguredProvider` can use its configured `baseUrl`.

## Solution Assessment

Yes for the issue direction: the narrow maintainable fix is to pass the existing provider id through the gateway memory-specific path and stop the Ollama adapter from hardcoding the base provider. The open linked PR should be reviewed and landed or replaced before closing this issue.
