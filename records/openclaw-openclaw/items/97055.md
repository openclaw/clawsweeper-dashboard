## Summary

Keep open: current main still drops the configured custom Ollama memory provider id before endpoint lookup, and the issue is already paired with the open candidate fix at https://github.com/openclaw/openclaw/pull/97059.

## What This Changes

This issue reports that `memorySearch.provider` can name a custom Ollama provider, but memory embedding requests are routed through the base `ollama` provider instead of that provider's configured `baseUrl`.

## Best Possible Solution

Review and land or replace the focused linked PR so custom Ollama provider ids are preserved through gateway memory-specific adapter creation and the Ollama memory adapter, with regression coverage and explicit cache identity upgrade handling.

## Reproduction Assessment

Yes. Source inspection on current main shows the custom provider id can select the Ollama memory adapter, then the gateway path and Ollama adapter drop that id before endpoint lookup.

## Solution Assessment

Yes for the issue direction. The narrow maintainable fix is to pass the existing provider id through the gateway memory-specific path and stop the Ollama adapter from hardcoding the base provider, with the open linked PR reviewed for cache identity upgrade safety.
