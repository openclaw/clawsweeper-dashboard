## Summary

Keep open: current `main` and `v2026.6.10` still drop the configured custom provider ID on the Ollama memory-specific embedding path, so the documented `memorySearch.provider` custom-provider contract remains broken for this report.

## What This Changes

The issue reports that `memorySearch.provider` with a custom Ollama provider ID is not preserved through embedding provider resolution, causing memory embeddings to use the wrong configured endpoint.

## Best Possible Solution

Preserve the requested custom provider ID through memory-specific adapter creation and the Ollama adapter, with regression coverage proving the custom provider's `baseUrl`, headers, auth, and index identity behavior are honored without adding a new config surface.

## Reproduction Assessment

Yes at source level: current `main` resolves the custom provider ID to the Ollama adapter, then the gateway/Ollama adapter path drops that ID before endpoint lookup. I did not run a live two-Ollama setup in this read-only sweep.

## Solution Assessment

Yes: the narrow maintainable fix is to pass the existing provider option through the gateway memory-specific path and stop the Ollama adapter from hardcoding the base provider where endpoint lookup depends on the configured ID.
