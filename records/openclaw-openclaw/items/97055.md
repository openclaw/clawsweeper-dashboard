## Summary

Keep open: current `main` still appears to violate the documented custom-provider contract because the Ollama memory embedding adapter overwrites the requested custom provider ID before endpoint lookup, so this report remains actionable.

## What This Changes

The issue reports that `agents.defaults.memorySearch.provider` with a custom Ollama provider ID ignores the configured `baseUrl` and routes embeddings through the base Ollama provider.

## Best Possible Solution

Preserve the requested custom provider ID through the Ollama memory embedding adapter and sibling gateway memory-specific path, with regression coverage for custom `baseUrl` routing and explicit handling of memory index identity compatibility.

## Reproduction Assessment

Yes at source level. The current adapter lookup can resolve a custom provider ID to the Ollama memory adapter, but that adapter overwrites the ID before `resolveConfiguredProvider` reads the configured `baseUrl`.

## Solution Assessment

Yes. The narrow maintainable solution is to thread the existing `provider` option through the Ollama-owned adapter and sibling gateway path, not add another config key or fallback route.

## Repair Work Prompt

Fix the custom Ollama memory provider routing bug reported in https://github.com/openclaw/openclaw/issues/97055. Current `memorySearch.provider` may be a custom `models.providers.<id>` with `api: "ollama"` and a custom `baseUrl`; the memory provider lookup resolves that ID to the Ollama adapter, but `extensions/ollama/src/memory-embedding-adapter.ts` overwrites `options.provider` with `"ollama"`, so `resolveConfiguredProvider` reads the base Ollama provider instead of the custom provider config. Preserve the requested provider ID through the Ollama memory adapter and review the sibling `src/gateway/embeddings-http.ts` memory-specific adapter path, which currently omits `provider: providerId` while the generic path includes it. Add focused regression coverage showing a custom provider ID such as `ollama-cpu` or `ollama-5080` uses its configured `baseUrl`; account for memory index identity/cacheKeyData compatibility if adding provider/baseUrl to runtime identity. Do not add a new config key or broad provider fallback. Related refs: https://github.com/openclaw/openclaw/issues/96742 and https://github.com/openclaw/openclaw/pull/96764. No `CHANGELOG.md` edit.
