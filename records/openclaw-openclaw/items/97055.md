## Label Justifications

- `P2`: The report is a real provider-routing bug with a narrow Ollama memory-embedding surface and no evidence of core runtime outage.
- `impact:auth-provider`: The affected behavior is provider selection, endpoint routing, headers, and auth-source resolution for custom Ollama memory embeddings.

## Summary

Keep open: current main and v2026.6.10 still drop the configured custom Ollama provider ID on the memory-specific embedding path, and the open linked PR is a candidate fix rather than a landed implementation.

## What This Changes

The issue reports that `memorySearch.provider` can resolve a custom Ollama provider but then lose that custom provider ID before endpoint lookup, sending memory embeddings to the wrong Ollama instance.

## Best Possible Solution

Review and land, revise, or replace https://github.com/openclaw/openclaw/pull/97059 so custom Ollama memory provider IDs reach endpoint lookup without adding a new config surface.

## Reproduction Assessment

Yes at source level: current main resolves the custom provider ID to the Ollama adapter, then the gateway and Ollama memory adapter path drop that ID before endpoint lookup. I did not run a live two-Ollama setup in this read-only review.

## Solution Assessment

Yes for the issue boundary: the narrow maintainable fix is to pass the existing provider option through the gateway memory-specific path and stop the Ollama adapter from hardcoding the base provider. The already-linked PR is the active candidate, but it still needs maintainer review before the issue can close.
