## Summary

Keep open: current main and the latest release still drop the configured custom Ollama memory provider id before endpoint lookup, and an open closing PR already owns the candidate fix path.

## What This Changes

The issue reports that `memorySearch.provider` can name a custom Ollama-backed provider, but memory embedding requests route through the base Ollama endpoint instead of the configured provider `baseUrl`.

## Best Possible Solution

Land a focused fix that preserves the configured provider id through both the gateway memory adapter path and the Ollama memory adapter, with regression coverage and explicit handling of any memory index rekeying.

## Reproduction Assessment

Yes. Source inspection shows a documented custom `memorySearch.provider` id can resolve to the Ollama memory adapter, but the current gateway memory path and Ollama adapter drop that id before provider config lookup; I did not run a live two-Ollama deployment in this read-only review.

## Solution Assessment

Yes. The best fix direction is to preserve the existing provider id through the current adapter boundary rather than adding a new config knob or fallback path; the linked PR is the active candidate, pending maintainer review of proof and cache identity.
