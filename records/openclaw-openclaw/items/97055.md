## Summary

Keep open: current main and v2026.6.10 still drop the custom Ollama memory provider id before endpoint lookup, and the issue already has an open closing PR that should be reviewed rather than creating or closing around a duplicate path.

## What This Changes

The issue reports that `memorySearch.provider` with a custom Ollama provider id and `baseUrl` routes embedding requests through the base Ollama provider instead of the configured endpoint.

## Best Possible Solution

Keep the issue open and use the linked PR as the implementation path; close this issue only after current main contains a proved provider-id propagation fix.

## Reproduction Assessment

Yes at source level: current main resolves a custom provider id to the Ollama memory adapter, then the gateway and Ollama adapter paths drop or overwrite that id before endpoint lookup. I did not run a live two-Ollama setup in this read-only review.

## Solution Assessment

Yes for the issue direction: the narrow maintainable fix is to pass the existing provider id through the gateway memory-specific path and stop the Ollama adapter from hardcoding the base provider. The open linked PR should be reviewed and landed or replaced before closing this issue.
