## What This Changes

The PR adds an opt-in process-supervisor survival boundary using Linux systemd user scopes, threads the option through child spawning, adds boundary/adapter/systemd coverage, and documents the design plus macOS gap.

## Best Possible Solution

Fix launcher resolution so the availability probe and spawned command use the same command path/env contract, then land the boundary only with inspectable systemd proof and a clear maintainer decision about the first production opt-in caller.

## Reproduction Assessment

No live current-main repro was run in this read-only review. Source inspection shows current child runs are direct supervisor children and service-managed mode keeps them attached, so the restart-survival problem is source-reproducible but not locally proven here.

## Solution Assessment

No, not yet. The systemd scope boundary is a reasonable direction, but the launcher env/path mismatch and absent production opt-in decision need resolution before this is the best merge shape.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Probe the same env used to launch systemd-run:** `src/process/supervisor/adapters/child.ts:96`
  - body: `resolveSupervisorBoundary()` decides that `systemd-run` is available from the gateway's `process.env`, but the later spawn runs `systemd-run` with `launchEnv`, which can come from the worker override. If that override sets `PATH` without `systemd-run`, this path chooses `systemd-scope` and then fails with `ENOENT` instead of falling back inline. Resolve the launcher command/path against the same env used for spawn, or use an absolute gateway-resolved `systemd-run` path and cover the PATH-override case.
  - confidence: 0.84
