The opt-in systemd path currently checks availability with `process.env` but then spawns `systemd-run` using `launchEnv`, which may come from a worker env override with a PATH that cannot resolve `systemd-run`. Update the implementation so the availability probe and actual launch use the same launcher command/env contract, or resolve `systemd-run` to an absolute gateway-visible path before spawning while preserving the worker env override and bus-var sourcing behavior. Add focused unit coverage in `src/process/supervisor/adapters/child.test.ts` or `src/process/supervisor/boundary.test.ts` for a worker env override whose PATH excludes `systemd-run`. Do not add a production caller for `surviveSupervisorRestart`; keep the behavior opt-in/default-off. Validate with the focused supervisor tests through the repo-approved Vitest wrapper and note that live systemd proof remains the PR body proof.
