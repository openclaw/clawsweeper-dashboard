## Summary

Keep open. Current main still has the locally verifiable parser/classifier shape behind the report, and I did not find a merged implementation that recognizes the reported Claude synthetic placeholder or surfaces a dedicated Telegram-visible outcome for it.

## What This Changes

The issue reports that claude-cli synthetic placeholder output can be parsed as an empty turn and leave Telegram users with no visible reply or retry signal.

## Best Possible Solution

Keep this open and add a parser-level synthetic-placeholder classification that flows through existing empty-response/silent status handling with focused Telegram and cli-output regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live claude-cli run. The reported assistant text shape is ignored by the Claude assistant-record parser, and an empty terminal result becomes empty CLI output that can be converted into silence under current policy.

## Solution Assessment

Unclear as a product decision. The narrow parser/classifier fix is the right owner boundary, but maintainers should decide whether the placeholder should trigger retry, deliberate-silent handling, or a Telegram-visible status/error.
