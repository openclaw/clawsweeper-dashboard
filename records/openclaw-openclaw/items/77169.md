## Summary

Keep open: current main already supports native provider-side video understanding, including Moonshot/Kimi `video_url`, but it does not expose the requested agent-facing `video` analysis tool analogous to `image`.

## What This Changes

Request to add an agent-facing native video analysis tool that accepts video paths or URLs and routes them to multimodal models such as Kimi K2.6.

## Best Possible Solution

Define a generic video-analysis surface that reuses the existing media-understanding runtime and provider registry, with explicit path/URL policy and docs for Kimi/Moonshot video input.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main gap: video understanding exists internally and through CLI, but no agent tool named `video` or equivalent video-analysis tool is registered.

## Solution Assessment

Unclear until maintainers choose the public surface. The narrow maintainable implementation would reuse `api.runtime.mediaUnderstanding.describeVideoFile`/provider selection rather than adding Kimi-specific logic to core.
