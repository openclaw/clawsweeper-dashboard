## Summary

Keep open: current main already has Moonshot/Kimi video-understanding internals and `openclaw infer video describe`, but it does not expose the requested agent-facing video analysis tool analogous to `image`.

## What This Changes

Request adds a native agent video-analysis tool that accepts video paths or URLs and routes them to models with native video input such as Kimi K2.6.

## Best Possible Solution

Define a generic agent-facing video analysis surface, likely a `video` tool reusing the existing media-understanding runtime, with explicit path/URL limits and long-video behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new feature request. Source inspection gives high-confidence current-main behavior: video understanding exists in provider and CLI paths, but no agent-facing video analysis tool is registered.

## Solution Assessment

Unclear until maintainers choose the user-facing surface. The narrowest maintainable direction appears to reuse `describeVideoFile` and media-understanding provider selection behind a generic tool rather than adding Kimi-specific logic to core.
