## Summary

Keep open: current main has native video-understanding internals, Moonshot/Kimi `video_url` request construction, and `openclaw infer video describe`, but it still lacks the requested agent-facing `video` analysis tool analogous to `image`. Related open work tracks broader direct video upload/native forwarding, so this needs maintainer API-surface direction before implementation.

## What This Changes

The issue requests an agent-facing native video analysis tool, similar to `image`, that accepts video paths or URLs and sends provider-native video inputs such as Kimi `video_url`.

## Best Possible Solution

Add a generic agent-facing video analysis surface or explicitly document the intended alternative, reusing the existing media-understanding provider contracts and coordinating with the native video upload/forwarding trackers.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence gap: current main can describe videos through internals and CLI, but there is no agent tool named `video` or equivalent video-analysis tool.

## Solution Assessment

Unclear until maintainers choose the user-facing surface. The narrow maintainable direction is a generic video-analysis tool or documented media-understanding path that reuses provider contracts, not Kimi-specific logic in core.
