## Summary

Keep open: current main already has Moonshot/Kimi video-understanding internals and `openclaw infer video describe`, including native `video_url` payload construction, but the requested agent-facing `video` analysis tool analogous to `image` is not present.

## What This Changes

The issue requests a native video-analysis input surface that accepts video paths or URLs and sends them to multimodal providers such as Kimi K2.6 without frame extraction.

## Best Possible Solution

Add a generic agent-facing video analysis tool, patterned after `image` and backed by the existing media-understanding runtime, with clear URL/path policy and docs for Kimi/Moonshot video input.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence gap: current main has native video-understanding internals and CLI description, but no agent tool named `video` or equivalent video-analysis tool.

## Solution Assessment

Unclear until maintainers choose the user-facing surface. The narrowest maintainable direction is likely a `video` tool that reuses media-understanding provider selection and the existing Moonshot `video_url` implementation rather than adding Kimi-specific logic to core.
