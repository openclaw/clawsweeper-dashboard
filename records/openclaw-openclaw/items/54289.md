## Summary

Keep open: current main and v2026.5.12 still implement the automatic Gemini CLI OAuth client-config extraction and fallback, while the trust-model/security classification needs maintainer policy review. The related diagnostics PR and protect-mcp comment are useful context, but neither removes nor gates the OpenClaw Google plugin behavior.

## What This Changes

The issue asks to remove or explicitly gate the Google plugin's automatic extraction and fallback use of Gemini CLI OAuth client credentials from local Gemini CLI installations.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic credential extraction by default or keep it behind explicit opt-in/disclosure with matching tests and provider docs.

## Reproduction Assessment

Yes. Source inspection on current main shows that with no env OAuth client id, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should decide the intended policy before changing runtime behavior.
