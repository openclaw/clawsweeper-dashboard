## Summary

Keep open: current main and v2026.5.18 still automatically extract and cache Gemini CLI OAuth client config from local Gemini CLI files, then use it as the fallback when explicit env credentials are absent. The protect-mcp comment is useful credential-isolation context, and the related diagnostics/layout PRs may improve failure handling, but none remove, gate, or document the extraction behavior, so this still needs maintainer security/product review.

## What This Changes

The issue asks OpenClaw to remove, gate, or explicitly disclose automatic Gemini CLI OAuth client ID and secret extraction from local Gemini CLI installations.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in/disclosure with matching provider docs and regression tests.

## Reproduction Assessment

Yes, by source inspection. With no env OAuth client ID, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should decide the intended policy before changing runtime behavior.
