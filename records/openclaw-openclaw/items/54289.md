## Summary

Keep open: current main and the latest release still implement the reported Gemini CLI OAuth credential extraction fallback, and the related open PRs only improve diagnostics or broaden layout support rather than resolving the security-policy question.

## What This Changes

The issue asks OpenClaw to stop, gate, or explicitly disclose automatic extraction of Gemini CLI OAuth client credentials from local Gemini CLI installation files.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in or disclosure with provider docs and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client ID, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials returns cached values and resolveOAuthClientConfig uses them.

## Solution Assessment

Unclear until maintainers decide the intended trust boundary. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB.
