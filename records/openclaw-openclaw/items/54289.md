## Summary

Keep open: current main and v2026.5.18 still automatically extract Gemini CLI OAuth client config and use it as the fallback when explicit env credentials are absent. The protect-mcp comment and the open diagnostics PR are useful context, but neither removes, gates, or documents the extraction behavior, so this still needs maintainer security/product review.

## What This Changes

The issue asks OpenClaw to remove or explicitly gate automatic extraction of Gemini CLI OAuth client credentials from a local Gemini CLI installation.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in/disclosure with matching tests and provider docs.

## Reproduction Assessment

Yes. Source inspection shows that with no env OAuth client id, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should decide the intended policy before changing runtime behavior.
