## Summary

Keep open: current main and the latest release still implement the reported Gemini CLI OAuth credential extraction fallback, and the related open PRs only improve diagnostics or install-layout coverage rather than resolving the security-policy question. The protect-mcp comment usefully points at credential isolation as an ecosystem mitigation, but it does not change OpenClaw core behavior.

## What This Changes

The issue asks OpenClaw to remove, gate, or warn before automatically extracting Gemini CLI OAuth client credentials from a local Gemini CLI installation.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in or disclosure with provider docs and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client ID, a `gemini` executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, `extractGeminiCliCredentials()` returns cached values and `resolveOAuthClientConfig()` uses them.

## Solution Assessment

Unclear until maintainers decide the intended trust boundary. Removing the fallback is the safest narrow behavior, while preserving compatibility likely needs an explicit opt-in or disclosure path plus docs and tests.
