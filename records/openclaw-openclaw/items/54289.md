## Summary

Keep open: current main and the latest shipped release still implement the reported Gemini CLI OAuth client-config extraction and fallback path, and resolving whether that is acceptable under the trusted bundled-plugin model requires maintainer security/product judgment.

## What This Changes

Review whether OpenClaw should stop or explicitly gate automatic extraction of Gemini CLI OAuth client credentials from a local Gemini CLI installation.

## Best Possible Solution

Keep this open for Google plugin security/product review, then either remove the automatic Gemini CLI client-config fallback by default or add extraction-specific opt-in/disclosure, with matching OAuth tests and provider docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client id, a `gemini` executable on `PATH`, and Gemini CLI OAuth source or bundle files containing the expected patterns, `extractGeminiCliCredentials()` returns cached values and `resolveOAuthClientConfig()` uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly the safer posture, but `SECURITY.md` treats plugin file/env reads as inside the trusted-plugin boundary, so maintainers should make the policy call before changing runtime behavior.
