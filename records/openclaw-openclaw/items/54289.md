## Summary

Keep open: current main and v2026.5.18 still automatically search local Gemini CLI files for OAuth client config, cache extracted values, and use them as the fallback when explicit env credentials are absent. The related protect-mcp comment is useful credential-isolation context, and the open diagnostics PR may improve failure messages, but neither removes, gates, or documents the extraction behavior.

## What This Changes

The issue asks OpenClaw to stop, gate, or disclose automatic extraction of Gemini CLI OAuth client ID and secret values from another local tool installation.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in/disclosure with matching provider docs and regression tests.

## Reproduction Assessment

Yes, by source inspection. With no env OAuth client ID, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should decide the intended policy before changing runtime behavior.
