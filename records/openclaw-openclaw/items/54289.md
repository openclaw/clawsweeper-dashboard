## Summary

Keep open: current main and the latest release still implement the reported Gemini CLI OAuth credential extraction fallback, and the related open PRs improve diagnostics or broaden extraction layouts rather than resolving the security-policy question.

## What This Changes

The issue asks OpenClaw to stop silently extracting Gemini CLI OAuth client credentials from another application's installation and require explicit consent or configuration.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or retain it behind explicit opt-in or disclosure with provider docs and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client ID, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials returns cached values and resolveOAuthClientConfig uses them.

## Solution Assessment

Unclear until maintainers decide the intended trust boundary. Removing or gating the fallback is plausibly safer, but the current SECURITY.md trust model treats trusted plugin file/env reads as inside the gateway TCB.
