## Summary

Keep open: current `main` and `v2026.5.20` still read Gemini CLI installation files, parse OAuth client ID/secret values, cache them, and use them when explicit OpenClaw env credentials are absent. The protect-mcp comment and the related diagnostics/layout PRs are useful context, but they do not resolve the Google OAuth trust-boundary policy question.

## What This Changes

The issue asks OpenClaw to remove or gate automatic Gemini CLI OAuth client ID/secret extraction and require explicit credential configuration or user consent.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or keep it behind explicit opt-in/disclosure with provider docs and regression tests.

## Reproduction Assessment

Yes. By source inspection, with no env OAuth client ID, a `gemini` executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, `extractGeminiCliCredentials()` returns cached values and `resolveOAuthClientConfig()` uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but `SECURITY.md` currently treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should decide the intended policy before changing runtime behavior.
