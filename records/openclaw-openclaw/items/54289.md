## Summary

Keep open: current main and the latest release still implement the Gemini CLI OAuth client-config extraction and fallback path, and resolving whether that is acceptable needs security/product judgment. The related diagnostics PR at https://github.com/openclaw/openclaw/pull/41991 does not remove or gate the behavior, and the protect-mcp comment is useful credential-isolation context rather than a core fix.

## What This Changes

The issue asks OpenClaw to remove or gate automatic extraction of Gemini CLI OAuth client config from local Gemini CLI source or bundle files and rely on explicit credentials or consent.

## Best Possible Solution

Keep this open for Google plugin security/product review, then either remove the automatic Gemini CLI client-config fallback by default or add extraction-specific opt-in/disclosure with matching OAuth tests and provider docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client id, a `gemini` executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, `extractGeminiCliCredentials()` returns cached values and `resolveOAuthClientConfig()` uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but `SECURITY.md` treats plugin file/env reads as inside the trusted-plugin boundary, so maintainers should make the policy call before changing runtime behavior.
