## Summary

Keep open: current main and the latest stable release still implement automatic Gemini CLI OAuth client-config extraction and fallback, while the docs only warn that the OAuth integration is unofficial. The related diagnostics PR at https://github.com/openclaw/openclaw/pull/41991 does not remove or gate extraction, and the protect-mcp comment is useful credential-isolation context rather than a core fix.

## What This Changes

The issue asks OpenClaw to remove or gate automatic extraction of Gemini CLI OAuth client ID and secret values from local CLI installation files when explicit env credentials are absent.

## Best Possible Solution

Keep this open for Google plugin security/product review, then either remove the automatic Gemini CLI client-config fallback by default or add explicit opt-in/disclosure with matching OAuth tests and provider docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env OAuth client id, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md treats trusted plugin file/env reads as inside the gateway TCB, so maintainers should define the intended policy before changing runtime behavior.
