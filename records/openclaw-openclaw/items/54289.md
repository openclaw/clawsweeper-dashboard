- `impact:security`: The issue concerns credential handling, trust boundaries, and whether a bundled plugin may read another local tool's OAuth client config.

## Summary

Keep open: current main and the latest release still implement automatic Gemini CLI OAuth client-config extraction and env-missing fallback, while the vulnerability classification needs maintainer security/product judgment under the documented trusted-plugin model.

## What This Changes

The issue requests removing, gating, or explicitly disclosing automatic Google Gemini CLI OAuth client credential extraction from local Gemini CLI installations.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth client-config contract, then either remove automatic extraction by default or keep it only behind extraction-specific opt-in/disclosure with matching tests and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows that with no env OAuth client id, a gemini executable on PATH, and Gemini CLI OAuth source or bundle files containing the expected patterns, extractGeminiCliCredentials() returns cached values and resolveOAuthClientConfig() uses them.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats enabled plugin file/env reads as inside the gateway trusted computing base, so maintainers should decide the intended policy before changing runtime behavior.
