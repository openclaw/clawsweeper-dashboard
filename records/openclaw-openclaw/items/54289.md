## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction and fallback path, and the existing prompt/docs warn about unofficial OAuth risk without explicitly gating or disclosing local Gemini CLI source/bundle reads.

## What This Changes

The issue asks OpenClaw to stop automatically extracting and reusing Gemini CLI OAuth client ID/secret data unless the user explicitly configures or consents to that behavior.

## Best Possible Solution

Keep this open for maintainer security/product review, then make an owner-local Google plugin change that removes automatic Gemini CLI client-config extraction by default or gates it behind explicit extraction-specific consent, with matching tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no env credentials, a `gemini` executable on `PATH`, and a Gemini CLI OAuth file or bundle containing the expected patterns, `extractGeminiCliCredentials()` returns cached values and `resolveOAuthClientConfig()` uses them.

## Solution Assessment

Unclear. The current implementation is maintainable for onboarding convenience, but the safer security posture likely requires a maintainer-owned policy choice before changing OAuth discovery semantics.
