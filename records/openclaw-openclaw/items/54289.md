## Summary

Keep open: current main and v2026.5.18 still extract Gemini CLI OAuth client config from local CLI source/bundle files and use it as the fallback when explicit OAuth env credentials are absent; the linked diagnostics PR does not remove, gate, or disclose that behavior.

## What This Changes

The issue asks OpenClaw to stop or explicitly gate automatic extraction of Gemini CLI OAuth client ID and secret values from local Gemini CLI installation files.

## Best Possible Solution

Define the supported Google Gemini CLI OAuth credential contract, then either remove automatic extraction by default or retain it only behind explicit opt-in/disclosure with matching tests and provider docs.

## Reproduction Assessment

Yes. Source inspection shows that with no explicit OAuth env client id, a gemini executable on PATH, and Gemini CLI source or bundle files containing the accepted patterns, extractGeminiCliCredentials() returns cached values that resolveOAuthClientConfig() uses for auth URL and token exchange paths.

## Solution Assessment

Unclear. Removing or gating the fallback is plausibly safer, but SECURITY.md currently treats trusted plugin file/env reads as inside the gateway TCB and the provider already has a general OAuth confirmation, so maintainers should decide the intended policy before changing runtime behavior.
