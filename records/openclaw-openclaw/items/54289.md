## Summary

Keep open. Current main and the latest release still implement the reported Gemini CLI OAuth client-config extraction and fallback path, while the existing prompt/docs only warn about unofficial Gemini CLI OAuth risk rather than explicitly disclosing or gating reads of Gemini CLI installation files. The related diagnostics PR does not resolve the security/product question.

## What This Changes

The issue asks OpenClaw to remove or explicitly gate automatic extraction of Gemini CLI OAuth client credentials from local Gemini CLI installation files.

## Best Possible Solution

Keep this open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI client-config extraction by default, or gate it behind an explicit extraction-specific opt-in/confirmation, then update the OAuth flow, tests, provider prompt, and public Google/Gemini provider docs. Treat the protect-mcp comment as useful credential-vault design input, not as a substitute for hardening this bundled behavior.

## Reproduction Assessment

Yes. The source path and tests provide a high-confidence reproduction: with a gemini executable on PATH and a Gemini CLI oauth2.js or bundle file containing client id and secret patterns, extractGeminiCliCredentials() returns and caches those values, and resolveOAuthClientConfig() uses them when env credentials are absent.

## Solution Assessment

Unclear. The current implementation is maintainable for onboarding convenience but is not clearly the best solution for explicit-consent security posture; the narrower safer direction is an owner-local Google plugin change that requires explicit env config or extraction-specific opt-in before scanning local Gemini CLI files.
