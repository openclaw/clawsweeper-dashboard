## Summary

Keep open: current main still has the documented Signal UUID allowlist regression, and this PR targets the right Signal-owned path, but it needs compact UUID handling and contributor real behavior proof before merge.

## What This Changes

The PR adds a Signal allowlist normalizer in `extensions/signal/src/shared.ts` and tests in `extensions/signal/src/core.test.ts` so `uuid:` `allowFrom` entries survive config and DM-policy normalization while phone numbers stay E.164.

## Best Possible Solution

Land this PR or an equivalent Signal-owned fix after compact UUID handling is consistent across command auth, DM policy, and approval auth, with focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Signal run. Current main delegates command auth to Signal `formatAllowFrom`, while Signal `formatAllowFrom` and DM policy still normalize through `normalizeE164` despite documented `uuid:<id>` allowlist support.

## Solution Assessment

No, not yet. The owner-local helper is the right seam, but it should canonicalize or reject compact UUIDs consistently and the contributor still needs real after-fix Signal proof.

## Review Findings

Overall correctness: patch is incorrect
