## Summary

Keep this PR open because current main still normalizes Signal shared command allowlists and shared DM-policy entries through phone-only E.164 logic, while Signal docs and pairing support `uuid:<id>` senders. The PR remains a focused Signal-owned fix with regression coverage and no concrete security or supply-chain concern in the provided diff.

## What This Changes

The PR adds a Signal-specific allowlist normalizer in `extensions/signal/src/shared.ts` that preserves `uuid:` senders while still normalizing phone numbers, and adds focused tests for shared allowlist formatting and DM-policy normalization.

## Best Possible Solution

Land this PR or an equivalent Signal-owned normalizer used by both `signalConfigAdapter.formatAllowFrom` and `signalSecurityAdapter.normalizeDmEntry`. Prefer sharing the same normalization shape with Signal approval auth to avoid drift, and decide whether compact UUIDs should canonicalize to the runtime sender form before landing.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a Signal `allowFrom` entry such as `uuid:abcdef12-3456-7890-abcd-ef1234567890` is routed through `normalizeE164` in both shared formatting and DM-policy normalization, while pairing and docs use `uuid:<id>` sender IDs.

## Solution Assessment

Yes, the proposed direction is a narrow maintainable fix because it stays inside the Signal plugin and updates both shared normalization sites. A small refinement would be to share or extract the existing Signal approval-auth normalizer and address the compact UUID canonicalization edge.
