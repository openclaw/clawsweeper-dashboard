## What This Changes

The PR adds a Signal-specific allowlist normalizer in `extensions/signal/src/shared.ts` and focused tests in `extensions/signal/src/core.test.ts` for UUID preservation and E.164 phone normalization.

## Best Possible Solution

Land this PR or an equivalent Signal-owned fix after canonicalizing UUID allowlist entries consistently across command auth, DM policy, and approval auth, while preserving wildcard and E.164 behavior and documenting the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Signal `allowFrom` entry like `uuid:abcdef12-3456-7890-abcd-ef1234567890` is documented and produced by pairing, but current main routes it through phone-only normalization before command authorization and DM-policy matching.

## Solution Assessment

No, not yet. The proposed direction is narrow and owner-local, but the best version should canonicalize or reject compact UUIDs consistently, avoid drift with `signalApprovalAuth`, and add the required release note.
