## Summary

Keep open. The PR targets a real Signal regression still present on current main, but the proposed helper leaves compact UUID allowlist entries unmatched and the external PR still lacks real Signal behavior proof.

## What This Changes

The PR adds a Signal-specific allowlist normalizer and tests so `uuid:` `allowFrom` entries survive command/DM policy normalization while phone numbers remain E.164.

## Best Possible Solution

Land this PR or an equivalent Signal-owned fix after compact UUID handling is made consistent across command auth, DM policy, and approval auth, with focused regression tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows Signal command auth uses the plugin formatter, while Signal `formatAllowFrom` and `normalizeDmEntry` still normalize through phone-only `normalizeE164` despite documented `uuid:<id>` allowlist support.

## Solution Assessment

No, not yet. The owner-local helper is the right seam, but it should canonicalize or reject compact UUIDs consistently and the contributor still needs real after-fix Signal proof.
