## Summary

Keep open: current main still has the Signal phone-only allowlist normalization path, and this PR is a relevant narrow fix, but the branch still leaves compact UUID entries noncanonical and lacks real after-fix behavior proof.

## What This Changes

The PR adds a Signal allowlist normalizer in `extensions/signal/src/shared.ts` plus tests in `extensions/signal/src/core.test.ts` to preserve `uuid:` senders while normalizing phone numbers.

## Best Possible Solution

Land this PR or an equivalent Signal-owned fix after canonicalizing or rejecting compact UUID allowlist entries consistently across command auth, DM policy, and approval auth while preserving wildcard and E.164 behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a documented Signal `allowFrom` value like `uuid:abcdef12-3456-7890-abcd-ef1234567890` is routed through phone-only normalization before command authorization and DM-policy matching on current main.

## Solution Assessment

No, not yet. The PR’s owner-local helper is the right direction, but it should canonicalize or reject compact UUIDs consistently and needs real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect
