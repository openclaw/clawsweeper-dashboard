## Summary

Keep open; the PR targets a real source-reproducible agent retry gap, but the new sleep classifier is broad enough to misclassify side-effectful shell groups as no-progress activity, so it needs a narrow repair before merge.

## What This Changes

The PR changes `resolvePlanningOnlyRetryInstruction()` and focused tests so strict-agentic `toolUse` turns with sleep-only bash/exec metas can still trigger the planning-only retry while ordinary bash remains progress.

## Best Possible Solution

Keep the focused strict-agentic fix, but restrict no-progress classification to a single sleep command plus non-shell display suffixes and preserve side-effect protection for every other bash/exec meta.

## Reproduction Assessment

Yes. The linked bug is source-reproducible on current main because non-`update_plan` tool metas and `toolUse` stop reasons block planning-only retry before text inspection; the PR-introduced shell-group risk is also source-reproducible from the new regex.

## Solution Assessment

No. The direction is right, but the current regex and side-effect bypass are too broad; the maintainable fix is a narrow single-sleep classifier with regression coverage for shell-group and ordinary-bash boundaries.
