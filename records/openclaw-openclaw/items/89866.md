## Summary

Keep open: this PR targets a real source-reproducible strict-agentic planning-only gap, but the new sleep classifier can treat a sleep-prefixed shell group as no-progress and bypass retry safety for side-effectful bash/exec commands.

## What This Changes

This PR changes the embedded agent planning-only retry helper and focused tests so `update_plan` and sleep-only bash/exec tool-use turns are treated as no-progress while ordinary bash still counts as progress.

## Best Possible Solution

Keep the focused strict-agentic fix, but restrict no-progress classification to a single sleep command/display-only suffix that cannot be shell syntax and preserve replay side-effect metadata for every other bash/exec case.

## Reproduction Assessment

Yes. The linked bug is source-reproducible from current main because tool-use stop reasons and non-`update_plan` tool metas block planning-only retry before text inspection; the introduced regex issue is also source-reproducible by inspecting the PR head pattern.

## Solution Assessment

No. The direction is right, but the current regex and side-effect bypass are too broad; the safer solution is a narrow single-sleep classifier with regression coverage for shell-group and ordinary-bash boundaries.

Full review comments:

- **[P1] Do not treat sleep plus a shell group as no progress:** `src/agents/embedded-agent-runner/run/incomplete-turn.ts:843`
  - body: The new regex also matches `sleep 0.1; (touch /tmp/x)`: the optional semicolon consumes the command separator and the arbitrary parenthesized suffix matches the shell group. Please restrict this heuristic to a single sleep command/display-only suffix and add a regression for sleep followed by a shell group.
  - confidence: 0.9
