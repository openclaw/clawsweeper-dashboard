## Summary

Keep open. Current `main` still has the unguarded checked-continuation bridge in the live WebSocket keepalive path, and related duplicate PRs were closed in favor of this PR as the canonical focused fix; the remaining blocker is missing after-fix real behavior proof from the contributor.

## What This Changes

The PR adds a private `NSLock`-guarded one-shot continuation wrapper around `WebSocketTaskBox.sendPing()` so duplicate pong callbacks cannot resume the same checked continuation twice.

## Best Possible Solution

Land this focused `GatewayChannel.swift` at-most-once guard after contributor real-behavior proof or an explicit maintainer proof override, preferably with a small duplicate-callback regression test added during landing.

## Reproduction Assessment

Yes. A `WebSocketTasking` test double that invokes `pongReceiveHandler` twice would drive current `main` into two resumes of the same checked continuation; related closed reports also include matching crash stacks, though I did not run a live device repro in this read-only review.

## Solution Assessment

Yes. The PR targets the implicated bridge directly, preserves existing keepalive semantics, and uses a local lock-guarded one-shot wrapper instead of broad reconnect or policy changes; the safer landing path is to pair it with proof and focused regression coverage.
