## Summary

Keep this PR open. Current main and the latest release still use the unguarded `CheckedContinuation` resume in `WebSocketTaskBox.sendPing()`, while this PR is a narrow one-file crash fix with no security or supply-chain concern. The remaining cleanup is mechanical: add the required changelog entry, and preferably focused duplicate-callback regression coverage before landing.

## What This Changes

The PR adds an `NSLock`-guarded one-shot wrapper around `WebSocketTaskBox.sendPing()` so duplicate `URLSessionWebSocketTask` ping callbacks cannot resume the same `CheckedContinuation` twice.

## Best Possible Solution

Land one narrow at-most-once guard in `WebSocketTaskBox.sendPing()`, add the required `Unreleased` changelog credit, add focused duplicate-callback coverage if feasible, and then close the already-superseded duplicate PRs/issues against the landed implementation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a `WebSocketTasking` test double that invokes `pongReceiveHandler` twice; current main would attempt to resume the same checked continuation twice, matching the crash logs in this PR and related report #55878.

## Solution Assessment

Yes. A one-shot guard local to `WebSocketTaskBox.sendPing()` is the narrow maintainable fix because the duplicate callback is specific to the ping bridge; a closure-local lock or this private wrapper are both acceptable as long as only the first callback resumes the continuation.
