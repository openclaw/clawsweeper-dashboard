## Summary

Keep open. Current main and the latest release still resume the WebSocket ping continuation directly, so this PR is still useful; the one-shot guard is a narrow crash fix, but it still needs the required changelog entry and final validation on an updated branch before merge.

## What This Changes

The PR adds an `NSLock`-guarded `OneShotContinuation` around `WebSocketTaskBox.sendPing()` so duplicate ping callbacks only resume the checked continuation once.

## Best Possible Solution

Land one local at-most-once guard in `WebSocketTaskBox.sendPing()`, add the required `Unreleased` changelog credit, add focused duplicate-callback coverage if feasible, and then retire the superseded duplicate reports against the landed fix.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a `WebSocketTasking` test double that invokes `pongReceiveHandler` twice; current main would attempt to resume the same checked continuation twice, matching the crash logs in this PR and related report #55878.

## Solution Assessment

Yes. A one-shot guard local to `WebSocketTaskBox.sendPing()` is the narrowest maintainable fix because the duplicate callback is specific to the ping bridge; the proposed wrapper is acceptable once the branch is updated and release bookkeeping is added.
