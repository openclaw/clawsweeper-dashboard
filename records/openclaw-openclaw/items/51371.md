## Summary

Keep open. Current main and the latest release still have the unguarded `WebSocketTaskBox.sendPing()` checked-continuation bridge, and this PR remains the only open focused fix path; it is blocked on after-fix real native gateway proof, not on a clear patch defect.

## What This Changes

The PR adds a private `OneShotContinuation` in `GatewayChannel.swift` and routes `WebSocketTaskBox.sendPing()` through it so duplicate pong callbacks only resume the checked continuation once.

## Best Possible Solution

Land this or an equivalent one-shot guard for `WebSocketTaskBox.sendPing()` after real native gateway proof, with focused regression coverage if maintainers can add it without overbuilding the test harness.

## Reproduction Assessment

Yes at source level: a `WebSocketTasking` double that invokes `pongReceiveHandler` twice would drive current main into two resumes of the same checked continuation. I did not run a live iOS or macOS device repro in this read-only review.

## Solution Assessment

Yes, pending proof. A lock-guarded one-shot wrapper is a narrow maintainable fix for this callback-to-async bridge; merge should wait for after-fix real behavior proof or a maintainer proof override.
