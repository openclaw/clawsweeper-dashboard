Keep open. Current main still has the reported provider-entry auth-routing gap, and the live linked fix PR has closing syntax for this issue but is still open, so the issue should stay open until that PR or a successor merges.

## What This Changes

The issue requests provider-entry credential selection to respect `models.providers.<id>.apiKey` for split provider entries instead of selecting a canonical provider profile from `auth.order`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79447 or a successor that defines the provider-entry credential binding across runtime auth, session auto-selection, status labels, docs/schema, and focused regression tests.

## Reproduction Assessment

Yes. The split-provider setup is source-reproducible on current main because runtime auth, session auto-selection, and status labels still use auth-profile order before the selected provider entry's late `apiKey` fallback, while non-marker `apiKey` strings remain literal credentials.
