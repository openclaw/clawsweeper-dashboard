## Summary

Keep open. Current main still has the same split-provider auth gap: runtime auth, session auto-selection, and status labels use provider auth-profile order before the selected provider entry's configured credential except in explicit `auth: "api-key"` mode, while `models.providers.*.apiKey` remains a credential/SecretRef field rather than a profile binding.

## What This Changes

The issue asks auth resolution to respect split provider-entry credential/profile bindings instead of selecting an interchangeable canonical OpenRouter auth profile.

## Best Possible Solution

Define a documented provider-entry auth-profile binding or model-aware profile mapping, then apply it consistently in runtime auth resolution, session auto-selection, status labels, schema/docs, and split-provider regression tests.

## Reproduction Assessment

Yes. The issue gives a concrete split-OpenRouter setup, and current-main source inspection shows the profile-order path that can select a shared canonical provider profile before the selected provider entry's configured credential on the normal path.

## Solution Assessment

No for the narrow `apiKey` overload as written. The safer maintainable fix is an explicit auth-profile binding that preserves existing `SecretInput` credential semantics and is reused by all auth display/runtime paths.
