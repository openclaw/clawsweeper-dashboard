## Summary

Keep open. Current main still has the source-reproducible provider-entry credential routing gap, and https://github.com/openclaw/openclaw/pull/79447 is an open closing PR for this issue, so the issue should remain open until that PR or a successor lands.

## What This Changes

The issue asks OpenClaw auth routing to respect a split provider entry's declared `apiKey` credential/profile binding instead of selecting a different profile through canonical provider auth order.

## Best Possible Solution

Define one explicit provider-entry auth-profile binding rule and apply it consistently across runtime auth, session auto-selection, status labels, docs/schema, and focused regression tests.

## Reproduction Assessment

Yes. The current-main source path is clear: profile-looking provider `apiKey` strings still flow through literal credential fallback, while runtime/session/status paths continue to consult auth-profile order without a provider-entry binding rule.

## Solution Assessment

No for closing now. The linked PR is the appropriate implementation path to review, but this issue should stay open until that PR or an accepted successor merges and resolves the credential-binding contract consistently.
