## Summary

Keep open. Current main still resolves the normal runtime, session, and status paths through canonical auth-profile order before provider-entry credentials, while `models.providers.*.apiKey` remains a SecretInput credential surface rather than a profile binding.

## What This Changes

The issue asks OpenClaw to select the credential declared on a split provider entry instead of rotating through the canonical provider's auth profiles.

## Best Possible Solution

Define a documented provider-entry auth-profile binding or model-aware profile mapping, preserve existing SecretInput `apiKey` semantics, and apply the rule consistently across runtime auth, session auto-selection, status, schema/docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure split provider entries plus shared canonical profiles, then the normal auth path can select from `auth.order` before the selected provider entry credential. I did not run a live OpenRouter request in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. Overloading `apiKey` as an auth-profile id is not the safest fix because the field is currently a credential/SecretRef surface; an explicit binding is more maintainable.
