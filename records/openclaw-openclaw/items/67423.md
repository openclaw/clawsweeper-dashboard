## Summary

Keep open. Current main still lacks a provider-entry-aware auth-profile binding for split providers, and https://github.com/openclaw/openclaw/pull/79447 is an open linked fix PR with closing syntax, so this issue should remain open until that PR or a successor merges.

## What This Changes

The issue asks OpenClaw to use a split provider entry's declared `apiKey`/profile binding instead of canonical `auth.order` when selecting credentials for same-endpoint provider entries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79447 or a successor after the provider-entry auth-profile binding contract is explicit across runtime auth, session auto-selection, status labels, docs/schema, and regression tests.

## Reproduction Assessment

Yes. The split-provider setup is source-reproducible on current main: auth-profile order and session/status paths can be selected before the late provider-entry `apiKey` fallback, and non-marker `apiKey` strings are still returned as literal `models.json` credentials.

## Solution Assessment

No for closing now. The linked PR is the right implementation path to evaluate, but the final fix needs maintainer judgment around the `apiKey`-as-profile-reference contract and provider compatibility before this issue can be closed.
