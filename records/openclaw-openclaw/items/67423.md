## Summary

Keep open. Current main still resolves the normal split-provider auth path through canonical provider auth-profile order before the selected provider entry's configured credential, and the visible session/status paths use the same canonical ordering; the remaining fix needs an explicit config/API contract because `models.providers.*.apiKey` is still documented and typed as credential material.

## What This Changes

The issue requests provider-entry-aware auth selection for split OpenRouter-style provider entries so each entry uses its intended credential/profile instead of canonical provider auth rotation.

## Best Possible Solution

Define a documented provider-entry auth-profile binding, such as a dedicated profile reference or model-aware profile mapping, then centralize that rule across runtime auth resolution, automatic session profile selection, status display, schema/docs, and focused split-provider regression tests.

## Reproduction Assessment

Yes. The issue gives concrete split-OpenRouter steps, and current-main source inspection shows the canonical profile-order path that can select a shared upstream provider profile before provider-entry credentials unless the entry is forced into explicit API-key mode.

## Solution Assessment

No for the issue's narrow `apiKey` overload as written. The safer maintainable solution is an explicit documented auth-profile binding that preserves existing `SecretInput` credential semantics and is reused by runtime, session, and status code.
