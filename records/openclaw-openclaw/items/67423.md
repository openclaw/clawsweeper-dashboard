## Summary

Keep open. Current main still matches the reported split-provider auth problem: the runtime resolver, session auto-selection, and status label path select from canonical auth-profile order before the selected provider entry's configured `apiKey`, while the config contract still defines that field as credential material.

## What This Changes

The issue requests provider-entry-aware auth resolution so split provider entries sharing one upstream OpenRouter auth provider use the key/profile declared on the selected provider entry instead of a canonical provider rotation pick.

## Best Possible Solution

Define a documented provider-entry auth-profile binding, or an equivalent explicit split-provider contract that preserves `apiKey` as credential material, then wire that rule through runtime auth resolution, session auto-selection, status labels, schema/docs, and focused regression tests for multiple OpenRouter-style entries sharing one auth provider.

## Reproduction Assessment

Yes. The issue provides a concrete split-OpenRouter configuration and observed `/status`/session state, and current-main source inspection shows the deterministic path that selects canonical provider auth-profile order before provider-entry credentials on the normal path.

## Solution Assessment

No for the narrow `apiKey` overload as stated. The maintainable fix is an explicit documented binding or contract, because the current `apiKey` field is a sensitive credential surface rather than a profile-reference field.
