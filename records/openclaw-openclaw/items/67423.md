## Summary

Keep open. Current main still lacks provider-entry-aware dereferencing for a split provider entry's `apiKey`, and the live PR with closing syntax is still open, so this issue should remain paired with that fix path until it merges or is replaced.

## What This Changes

The issue asks OpenClaw to use the credential/profile intended by a split provider entry instead of selecting a different key through canonical provider auth order.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79447 or a successor that defines the provider-entry credential binding consistently across runtime auth, session override, status labels, docs/schema, and regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main still routes through auth-profile order and treats non-marker provider-entry `apiKey` strings as literal `models.json` credentials, which matches the reported wrong-profile and literal-bearer failure modes.

## Solution Assessment

No for closing now. The linked PR is the active fix path, but this issue should remain open until the provider-entry credential contract is accepted and merged on main.
