## Summary

Keep open. Current main still lacks a provider-entry-aware binding for split-provider auth, and the live linked fix PR is open with closing syntax, so this issue should remain open until that PR or a successor merges.

## What This Changes

The issue requests that split provider entries use the credential declared on the selected provider entry instead of selecting an auth profile through canonical provider `auth.order`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79447 or a successor after the provider-entry auth-profile binding contract is explicit across runtime auth, session auto-selection, status labels, docs/schema, and regression tests.

## Reproduction Assessment

Yes. The split-provider setup is source-reproducible on current main: runtime auth, session override, and status labels still use canonical auth-profile order before the selected provider entry's late `apiKey` fallback, while non-marker `apiKey` strings remain literal credentials.

## Solution Assessment

No for closing now. The linked PR is the right fix path to review, but the issue should stay open until that PR or a successor merges and the provider-entry auth contract is accepted.
