## Summary

Keep open: the sender-context API direction matches a real channel-plugin gap and the linked DMWork use case, but the current patch regresses tool-family construction guards and still lacks real behavior proof from the external contributor.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards sender context from `createOpenClawCodingTools` into channel-owned agent-tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow patch after preserving the existing `includeChannelTools` and `includeOpenClawTools` guards, adding focused regression coverage plus SDK docs/changelog, and posting redacted real behavior proof.

## Reproduction Assessment

Yes for the PR regression. Source inspection shows current main keeps `includeChannelTools` and `includeOpenClawTools` separate, while the PR patch gates both branches on `includeCoreTools`; the existing construction-plan tests cover plugin/channel-only allowlists.

## Solution Assessment

No as written. The sender-context API addition is narrow and consistent with adjacent trusted requester context, but the implementation should preserve per-family guards and add test/docs/changelog/proof before merge.
