## Summary

Keep this PR open: the sender-context API direction is useful, but the current patch regresses tool-family gating and still lacks required real behavior proof for an external contributor PR.

## What This Changes

The PR widens `ChannelAgentToolFactory` and `listChannelAgentTools` params with trusted sender context and forwards `createOpenClawCodingTools` sender fields into channel agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, restore `includeChannelTools` and `includeOpenClawTools`, add focused propagation coverage, and merge only after real behavior proof or a maintainer proof override.

## Reproduction Assessment

Yes for the PR regression: the diff source-reproduces the gate change, and live CI fails `check-prod-types` because `includeChannelTools` becomes unused. The requested sender-context API is a feature gap, not a current-main bug reproduction.

## Solution Assessment

No, not as written. Passing trusted sender params is a reasonable narrow API direction, but the patch must preserve the family-specific gates and prove the runtime propagation path.
