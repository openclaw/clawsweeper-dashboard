## Summary

Keep open: this is the canonical PR for the sender-context feature tracked from https://github.com/openclaw/openclaw/issues/58806, but the current head regresses existing tool-family gates and still lacks after-fix real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` parameters with trusted sender fields and passes `senderId`/`senderIsOwner` from coding-tool construction into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and SDK docs/changelog updates if maintainers treat the fields as user-facing.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the PR regression: current main supports `includeCoreTools: false` with `includeChannelTools: true`, while PR head gates channel tools and OpenClaw tools on `includeCoreTools`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must preserve the existing planned tool-family gates and provide proof that the new context reaches a real channel factory.
