The PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factory params and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Keep the PR open and land a narrow version that preserves the existing per-family gates, proves sender context reaches a real channel factory, and updates SDK-facing docs/changelog if maintainers accept the contract.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows channel-only construction can have `includeCoreTools: false` while `includeChannelTools: true`, and CI confirms `includeChannelTools` becomes unused. I did not run local tests because this is a read-only review.

## Solution Assessment

No, not as written. The sender-context propagation is a reasonable direction, but the implementation must keep `includeChannelTools` and `includeOpenClawTools` as separate gates and add focused propagation proof.
