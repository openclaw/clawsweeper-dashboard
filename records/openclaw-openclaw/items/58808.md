## Summary

Keep open: the sender-context SDK seam is useful, but the current PR breaks existing tool-family gates, fails CI, and still lacks contributor real behavior proof for the external setup.

## What This Changes

The PR adds optional trusted requester sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel agent tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after preserving `includeChannelTools`/`includeOpenClawTools`, adding propagation coverage, and updating SDK docs/changelog if maintainers accept the contract.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection and CI show the PR makes `includeChannelTools` unused and skips channel-owned tools when a construction plan has `includeCoreTools: false` but `includeChannelTools: true`. I did not run local tests because this review is read-only.

## Solution Assessment

No, not as written. Passing trusted sender context is a reasonable SDK seam, but the patch must preserve the existing per-family construction gates and prove the fields reach a real channel factory.
