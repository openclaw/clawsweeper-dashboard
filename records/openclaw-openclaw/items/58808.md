## Summary

Keep this PR open: it addresses a real channel-agent SDK gap that is still present on current main, but the current head is not mergeable because it collapses the channel/OpenClaw tool-family gates onto `includeCoreTools` and still lacks real behavior proof from a real plugin setup.

## What This Changes

The PR widens `ChannelAgentToolFactory` and `listChannelAgentTools` params with trusted sender context and forwards `createOpenClawCodingTools` sender options into channel agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, pass it through under `includeChannelTools`, leave `createOpenClawTools` under `includeOpenClawTools`, add focused propagation coverage, then merge after real proof or maintainer override.

## Reproduction Assessment

Yes. The PR diff source-reproduces the blocker by replacing family-specific gates with `includeCoreTools`, while current tests define valid plans where `includeCoreTools` is false and `includeChannelTools` is true.

## Solution Assessment

No, not as written. Passing trusted sender params is the right seam, but the implementation must preserve `includeChannelTools` and `includeOpenClawTools` separately and prove the runtime propagation path.
