## Summary

Keep open: the requested sender-context SDK seam is still absent on current main and appears useful, but this PR currently introduces a tool-family construction gate regression and lacks required real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` to receive trusted sender context, forwards it from `createOpenClawCodingTools`, and changes the channel/OpenClaw tool construction guards in `pi-tools.ts`.

## Best Possible Solution

Expose sender context through the channel-agent-tool SDK seam while preserving the existing family-specific construction gates and adding focused regression coverage.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current tests and source allow channel tools with `includeCoreTools: false`, while the PR gates channel tools on `includeCoreTools`. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep `listChannelAgentTools` behind `includeChannelTools` and `createOpenClawTools` behind `includeOpenClawTools`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve the family-specific construction gates:** `src/agents/pi-tools.ts:871-878`
  - body: The patch gates both channel-agent-tool and OpenClaw-tool construction on `includeCoreTools`. Narrow plans can set `includeCoreTools: false` while `includeChannelTools` remains true, so allowed channel-owned tools disappear; OpenClaw tools can also be built when the plan disabled that family. Keep these calls behind `includeChannelTools` and `includeOpenClawTools` while forwarding sender context.
