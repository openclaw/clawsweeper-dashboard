## Summary

This PR should stay open: the sender-context API direction is plausible, but the current diff still regresses tool-family construction and the external contributor has not provided real behavior proof from a channel-plugin setup.

## What This Changes

The PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` with optional trusted sender context, forwards existing sender fields from `createOpenClawCodingTools`, and changes the channel/OpenClaw tool construction guards in `pi-tools.ts`.

## Best Possible Solution

Land this as a narrow additive ChannelAgentToolFactory context extension only after preserving the existing per-family tool guards, adding focused coverage, and documenting the SDK surface with maintainer-owned changelog handling.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main has separate `includeChannelTools` and `includeOpenClawTools` guards, and the PR patch gates both branches on `includeCoreTools`. Existing construction-plan tests also cover channel-only allowlists with `includeCoreTools: false`.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the implementation must keep the existing per-family guards intact and needs real behavior proof before merge.
