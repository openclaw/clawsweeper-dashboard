## Summary

Keep open: the sender-context seam is useful and current main still has the factory gap, but this PR currently regresses the existing tool-family gates and lacks contributor-owned real behavior proof for the access-control use case.

## What This Changes

The branch adds optional trusted sender fields to channel agent-tool factories and changes `createOpenClawCodingTools()` to pass `options.senderId` and `options.senderIsOwner` while also gating channel and OpenClaw tool construction on `includeCoreTools`.

## Best Possible Solution

Forward trusted sender context through channel agent-tool factories while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then prove it with focused regression tests and redacted real channel output.

## Reproduction Assessment

Yes for source-level behavior: current main only passes `cfg` to channel agent-tool factories while adjacent OpenClaw tool construction receives trusted sender context. The PR regression is also source-reproducible from the diff because it replaces distinct construction-plan gates with `includeCoreTools`.

## Solution Assessment

No. The API shape is reasonable, but this implementation should keep the existing channel/OpenClaw tool-family gates and add focused proof before merge.
