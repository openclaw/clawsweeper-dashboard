## Summary

Keep open: the API direction addresses a source-visible channel plugin gap, but the latest head collapses the separate channel/OpenClaw tool gates to `includeCoreTools`, which fails prod types and can break restricted tool construction; contributor-owned real behavior proof is also still missing.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factory params and forwards sender context from `createOpenClawCodingTools()` into `listChannelAgentTools()`.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and any maintainer-required SDK docs/changelog updates.

## Reproduction Assessment

Yes for source-level behavior: current main only passes `cfg` into channel agent-tool factories while adjacent OpenClaw tools receive trusted sender context. The PR regression is also source-reproducible from the latest diff and construction-plan tests; I did not run a live channel scenario in this read-only review.

## Solution Assessment

No. The requested API shape is reasonable, but the current patch should preserve the distinct channel and OpenClaw construction gates instead of using `includeCoreTools` for both.

## Review Findings
