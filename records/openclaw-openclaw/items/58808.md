## Summary

Keep open: the additive sender-context SDK change is useful, but the current PR head regresses separate tool-family allowlist gates and still lacks external real behavior proof.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then add focused regression coverage and SDK docs/changelog if maintainers treat the fields as user-facing.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main supports plugin/channel-only construction with `includeCoreTools: false`, while PR head gates channel tools and OpenClaw tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the separate `includeChannelTools` and `includeOpenClawTools` gates.
