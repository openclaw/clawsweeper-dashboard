## Summary

Keep open: the feature is still useful and not implemented on current main, but this patch should not merge as written because it drops channel-only tool construction paths and still lacks external real behavior proof.

## What This Changes

The PR extends `ChannelAgentToolFactory`/`listChannelAgentTools` with requester sender context and forwards `createOpenClawCodingTools` sender options while changing the channel/OpenClaw tool guards at the call site.

## Best Possible Solution

Keep the additive sender-context API, preserve the existing per-family tool construction guards, add focused propagation coverage, and land after SDK review plus real plugin proof.

## Reproduction Assessment

Yes, source-reproducible. Current main lacks sender context on `ChannelAgentToolFactory`, and the PR diff's `includeCoreTools` guard would drop channel-only allowlists such as `whatsapp_login` that current tests expect to keep channel tools available.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the PR should preserve `includeChannelTools`/`includeOpenClawTools` and add focused propagation coverage before merge.
