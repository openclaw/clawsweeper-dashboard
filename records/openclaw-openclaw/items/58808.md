## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land the additive sender-context API while preserving `includeChannelTools` and `includeOpenClawTools` as separate construction guards, with focused regression tests and redacted sender-aware channel proof.

## Reproduction Assessment

Yes, for the PR regression by source inspection: the live diff gates both channel and OpenClaw tool construction on `includeCoreTools`, while current main and existing tests support channel/plugin-only plans with `includeCoreTools: false` and OpenClaw-disabled plans with `includeCoreTools: true`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The maintainable path is to add and forward the optional sender fields without changing the existing tool-family construction gates, then prove the API with focused tests and a redacted external-channel run.
