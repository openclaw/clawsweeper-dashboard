## Summary

Keep open: the sender-context contract addition is useful for the linked DMWork/channel-plugin use case, but the current patch is not mergeable because it replaces the channel/OpenClaw tool-family gates with `includeCoreTools`, fails current CI, and still lacks required real behavior proof.

## What This Changes

The branch adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel agent-tool factories.

## Best Possible Solution

Keep the narrow sender-context threading, restore the `includeChannelTools` and `includeOpenClawTools` gates, add focused regression coverage for factory params, then merge after maintainer SDK review and real behavior proof or override.

## Reproduction Assessment

Yes. The live PR diff and current CI reproduce the patch defect: `includeChannelTools` becomes unused and planned channel/OpenClaw tool-family gates are ignored.

## Solution Assessment

No, not as currently implemented. The best fix is the same additive sender-context contract, but without changing tool-family gate conditions, plus a regression test proving the trusted sender fields reach a channel agent-tool factory.
