## Summary

Keep open: the requested API gap still exists on current main, but this branch cannot merge as-is because it changes tool-family gates and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factories and forwards the existing sender options from `createOpenClawCodingTools()`.

## Best Possible Solution

Forward the trusted sender fields through the channel agent-tool factory API while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then cover both sender forwarding and allowlist-driven construction plans.

## Reproduction Assessment

Yes for source-level behavior: current main only passes `cfg` to channel agent-tool factories while adjacent OpenClaw tool construction receives the trusted sender context. The PR regression is also source-reproducible from the diff and the construction-plan tests.

## Solution Assessment

No. The API direction is reasonable, but the implementation should preserve the distinct channel/OpenClaw tool-family gates and include real behavior proof before merge.
