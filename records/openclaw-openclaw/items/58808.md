## Summary

Keep open: the sender-context API addition is plausible, but the current PR head still regresses tool-family gates by using `includeCoreTools` for channel and OpenClaw tool construction, and external real behavior proof is still missing.

## What This Changes

The PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` params with `requesterSenderId`/`senderIsOwner`, then forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Preserve the sender-context API addition, restore `includeChannelTools` and `includeOpenClawTools`, add focused propagation coverage, and merge only after real behavior proof or a maintainer proof override.

## Reproduction Assessment

Yes for the review finding: the live PR diff source-reproduces the gate regression against current-main construction-plan behavior. The requested sender-context API is a new capability, so there is no current-main bug reproduction beyond confirming the API is absent.

## Solution Assessment

No, not as written. Passing trusted sender params is a reasonable narrow API direction, but the patch must preserve the existing family-specific gates and prove the runtime propagation path.
