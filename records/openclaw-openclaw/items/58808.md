## Summary

Keep this PR open: the requested sender-context seam is still useful and not on current main, but the latest branch cannot merge because it regresses the existing channel/OpenClaw tool-family gates and still lacks real behavior proof.

## What This Changes

The PR adds optional requesterSenderId and senderIsOwner parameters to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context to channel agent tool factories.

## Best Possible Solution

Land the additive sender-context API only after restoring the existing tool-family gates, adding focused regression coverage for factory context propagation, and collecting redacted real setup proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main and tests support includeCoreTools false with includeChannelTools true, while the latest PR gates channel tools on includeCoreTools. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The narrow maintainable solution is to pass options?.senderId and options?.senderIsOwner into listChannelAgentTools() while preserving includeChannelTools and includeOpenClawTools separately, then add proof and coverage.
