## Summary

This PR is useful but should stay open: current main still lacks sender context for channel agent-tool factories, while the live PR head still changes the channel and OpenClaw tool-family gates to `includeCoreTools`, fails `check-prod-types`, and lacks required real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with `requesterSenderId` and `senderIsOwner`, then forwards those values from `createOpenClawCodingTools`.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving `includeChannelTools` and `includeOpenClawTools`, add focused propagation coverage, and merge only after maintainer review plus real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main allows `includeCoreTools`, `includeChannelTools`, and `includeOpenClawTools` to diverge, while the live patch uses `includeCoreTools` for both channel and OpenClaw construction. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The sender-context seam is narrow and plausible, but the implementation must preserve family-specific gates and include real channel-plugin proof before merge.
