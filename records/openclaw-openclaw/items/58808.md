## Summary

Keep open: the sender-context addition is still useful, but the latest PR head regresses tool-family gating and still lacks required real behavior proof from a real channel setup.

## What This Changes

The PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` with `requesterSenderId`/`senderIsOwner`, then forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and SDK docs if maintainers accept the public contract.

## Reproduction Assessment

Yes, for the PR regression by source inspection: current main supports `includeCoreTools: false` with `includeChannelTools: true`, while the PR gates channel tools and OpenClaw tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the separate channel and OpenClaw tool-family gates.
