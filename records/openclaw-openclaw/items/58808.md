## Summary

Keep open: current main still lacks the useful sender-context seam, but the latest PR head is not mergeable because it regresses family-specific tool construction gates and lacks after-fix real behavior proof.

## What This Changes

The branch widens `ChannelAgentToolFactory`/`listChannelAgentTools` for `requesterSenderId` and `senderIsOwner`, forwards those values from `createOpenClawCodingTools`, and changes channel/OpenClaw tool construction guards to `includeCoreTools`.

## Best Possible Solution

Land an equivalent narrow SDK propagation that preserves `includeChannelTools` and `includeOpenClawTools`, adds focused sender-context coverage, and updates docs/changelog if maintainers treat the new fields as public API.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: construction plans can set `includeCoreTools: false` while `includeChannelTools: true`, but the patch gates channel tools on `includeCoreTools`. I did not run tests in this read-only review.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep `listChannelAgentTools` behind `includeChannelTools` and `createOpenClawTools` behind `includeOpenClawTools`.
