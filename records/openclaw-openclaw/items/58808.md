## Summary

Keep open: the API direction is useful, but the current patch regresses the tool-family construction gates and still lacks contributor-owned real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with trusted sender context and forwards `senderId`/`senderIsOwner` from tool construction.

## Best Possible Solution

Keep the sender-context API addition, but pass the new fields under `includeChannelTools`, leave `createOpenClawTools` under `includeOpenClawTools`, and add focused regression coverage before merge.

## Reproduction Assessment

Yes. The PR-introduced regression is reproducible from source: current main allows channel/plugin-only plans with `includeCoreTools: false`, while the PR would skip channel tools by gating them on `includeCoreTools`.

## Solution Assessment

No, not as written. The sender-context threading is the right narrow direction, but the implementation should preserve the planned tool-family gates instead of collapsing them to `includeCoreTools`.
