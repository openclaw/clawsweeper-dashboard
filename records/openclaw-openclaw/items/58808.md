## Summary

Keep this PR open: the sender-context API direction is useful, but the latest diff still regresses existing tool-family construction guards and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The branch adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, preserve the separate channel/OpenClaw tool-family guards, add focused propagation coverage, and land after SDK review with redacted real plugin proof.

## Reproduction Assessment

Yes, source-reproducible. Current main lacks sender context on ChannelAgentToolFactory, and current tests show channel-only allowlists intentionally have includeCoreTools:false while includeChannelTools remains true.

## Solution Assessment

No as written. Passing trusted sender context into the channel factory is the narrow maintainable direction, but this patch should preserve includeChannelTools/includeOpenClawTools and add focused propagation coverage.
