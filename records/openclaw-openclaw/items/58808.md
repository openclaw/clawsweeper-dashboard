## Summary

Keep open: current main still lacks the requested sender context for channel agent-tool factories, but this PR head regresses tool-family construction guards and still needs real behavior proof.

## What This Changes

The branch adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context, while also switching channel and OpenClaw tool construction to the broader includeCoreTools guard.

## Best Possible Solution

Expose trusted sender context through the channel-agent-tool factory while preserving per-family construction guards, then add focused regression coverage and SDK docs/changelog if maintainers treat the fields as user-facing.

## Reproduction Assessment

Yes. Source inspection on current main shows the factory/list function accept only cfg while the adjacent OpenClaw tool path receives sender context, and the PR diff shows the guard regression against existing allowlist construction tests.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the patch must keep includeChannelTools/includeOpenClawTools separate and add proof plus focused coverage before merge.
