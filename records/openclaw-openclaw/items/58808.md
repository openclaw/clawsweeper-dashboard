## Summary

Keep open: current main still lacks the requested sender context for channel agent-tool factories, but the PR as written regresses planned tool-family gating and has no after-fix real behavior proof.

## What This Changes

The branch extends `ChannelAgentToolFactory`/`listChannelAgentTools` with `requesterSenderId` and `senderIsOwner`, then threads `options.senderId` and `options.senderIsOwner` from `createOpenClawCodingTools`.

## Best Possible Solution

Keep the sender-context API addition, but pass the fields under `includeChannelTools`, preserve the separate `includeOpenClawTools` gate, add focused regression coverage, and require contributor-owned real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main supports channel/plugin-only construction with `includeCoreTools: false` and `includeChannelTools: true`, while the PR would skip channel tools by gating them on `includeCoreTools`.

## Solution Assessment

No, not as written. Threading sender context is the right narrow direction, but the implementation should preserve the planned tool-family gates instead of collapsing channel and OpenClaw tool construction under `includeCoreTools`.
