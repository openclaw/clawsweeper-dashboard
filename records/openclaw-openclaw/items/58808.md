## Summary

Keep open: the requested sender-context API is useful and source-backed, but the current PR head changes tool-construction gate semantics, lacks the required changelog entry, and still has no after-fix real behavior proof.

## What This Changes

The PR extends `ChannelAgentToolFactory` params with `requesterSenderId` and `senderIsOwner` and forwards those values from `createOpenClawCodingTools` into channel agent-tool factories.

## Best Possible Solution

Add the optional trusted sender fields through the existing `includeChannelTools` path, leave `createOpenClawTools` behind `includeOpenClawTools`, add changelog/regression coverage, and merge only after real sender-aware channel-tool proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an additive public plugin SDK feature. Source inspection confirms current main only passes `cfg` to channel agent-tool factories, and the PR behavior is reviewable from the diff plus current construction-plan tests.

## Solution Assessment

No. The API direction is narrow and plausible, but the current implementation should preserve `includeChannelTools` and `includeOpenClawTools` instead of switching those call sites to `includeCoreTools`.
