## Summary

Keep open: current main still lacks the requested sender context in channel agent-tool factories, and this PR is the canonical open implementation path, but the current patch is not mergeable because it changes the family-specific construction gates and lacks required real behavior proof.

## What This Changes

The PR adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow additive patch after preserving the separate channel/OpenClaw tool gates, adding focused regression coverage, and providing redacted proof from a real channel plugin.

## Reproduction Assessment

Yes for the PR-introduced regression: source and existing tests show channel-only plans can have includeCoreTools false while includeChannelTools is true, and the PR head CI also fails because includeChannelTools becomes unused. I did not run local tests in this read-only review.

## Solution Assessment

No, not as written. The additive sender-context seam is reasonable, but the implementation must keep includeChannelTools and includeOpenClawTools as separate gates and prove the real runtime path.
