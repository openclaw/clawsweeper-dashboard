## Summary

Keep open: the sender-context gap is real and current main still lacks the requested channel agent-tool factory parameters, but the latest PR head breaks existing tool-family gates, fails production typecheck, and still lacks external real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` parameters with `requesterSenderId` and `senderIsOwner`, then forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Add the sender fields while preserving `includeChannelTools` and `includeOpenClawTools`, add focused regression coverage, update SDK docs if maintainers want the contract publicized, and collect redacted real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection plus CI show the branch gates channel and OpenClaw factories on `includeCoreTools`, while current main intentionally keeps those tool families separately controlled. I did not run local tests because this review is read-only.

## Solution Assessment

No, not as written. The API seam is plausible, but the maintainable patch must keep channel/OpenClaw gates separate and prove the real plugin path works.
