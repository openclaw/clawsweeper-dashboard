## Summary

This PR should stay open: the requested sender-context seam is still useful and absent on current main, but the current patch regresses existing tool-family guards and still lacks contributor-supplied real behavior proof.

## What This Changes

The PR widens ChannelAgentToolFactory/listChannelAgentTools params with trusted sender fields and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land this seam or an equivalent narrow patch after preserving the family-specific guards, adding regression coverage for sender-context propagation and allowlist behavior, and updating SDK docs/changelog if maintainers treat it as public.

## Reproduction Assessment

Yes. Source inspection shows current main's ChannelAgentToolFactory/listChannelAgentTools path is cfg-only, and existing construction-plan coverage defines channel-only allowlists where the PR's includeCoreTools guard would drop valid channel tools.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the implementation must keep includeChannelTools and includeOpenClawTools as separate guards.
