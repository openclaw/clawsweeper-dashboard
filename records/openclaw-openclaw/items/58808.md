## What This Changes

The PR widens `ChannelAgentToolFactory` and `listChannelAgentTools` with `requesterSenderId`/`senderIsOwner` and forwards `createOpenClawCodingTools` sender context into channel agent-tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after preserving the existing per-family gates, adding focused propagation coverage, and documenting the SDK-facing contract if accepted.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection and CI show `includeChannelTools` becomes unused and channel-only allowlists can be skipped. For the requested feature gap, current main source shows sender context is not passed to channel agent-tool factories.

## Solution Assessment

No, not as written. Passing trusted sender context is a reasonable direction, but the implementation must keep `includeChannelTools` and `includeOpenClawTools` as separate guards and provide real behavior proof.
