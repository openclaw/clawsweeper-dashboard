## Summary

Keep open: the sender-context seam is useful and current main still lacks it, but this PR currently regresses tool-family gating and has no required real behavior proof from a real channel-plugin setup.

## What This Changes

The PR adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Expose trusted sender context through the channel-agent-tool SDK seam while preserving includeChannelTools and includeOpenClawTools as separate gates, with focused regression coverage and SDK docs/baseline follow-through if maintainers treat the fields as public.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main and tests preserve separate channel/OpenClaw construction gates, while the PR gates both families on includeCoreTools. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep listChannelAgentTools behind includeChannelTools and createOpenClawTools behind includeOpenClawTools while forwarding the new context.
