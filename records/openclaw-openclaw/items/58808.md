## Summary

Keep open: the DMWork sender-aware channel-tool use case is valid and not implemented on current main, but this PR still breaks the existing tool-family gates and lacks required real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` parameters and forwards `senderId`/`senderIsOwner` from `createOpenClawCodingTools` into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel-agent-tool factory path while preserving `includeChannelTools`/`includeOpenClawTools`, then add focused propagation coverage and SDK docs/changelog if maintainers accept the public contract.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a `whatsapp_login` allowlist plans `includeCoreTools: false` with `includeChannelTools: true`, while the PR uses `includeCoreTools` to guard channel factories. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The API seam is reasonable, but the patch must preserve the existing per-family construction gates and include proof that sender context reaches a real channel factory.
