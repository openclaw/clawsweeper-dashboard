## Summary

Keep open: the sender-context seam is still missing on current main and the PR is useful, but the current head is not mergeable as written because it collapses separate tool-family gates and lacks real behavior proof.

## What This Changes

The branch extends ChannelAgentToolFactory/listChannelAgentTools with optional requesterSenderId and senderIsOwner fields and forwards createOpenClawCodingTools sender context into channel agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through ChannelAgentToolFactory/listChannelAgentTools while preserving includeChannelTools and includeOpenClawTools, then add focused regression coverage and SDK docs if maintainers accept the contract change.

## Reproduction Assessment

Yes for the PR regression by source inspection: current construction plans can have includeCoreTools false while includeChannelTools is true, and current pi-tools uses includeChannelTools independently. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The sender-context plumbing is the right narrow direction, but replacing the separate channel/OpenClaw tool-family guards with includeCoreTools is not behavior-preserving.
