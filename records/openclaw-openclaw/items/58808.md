## Summary

Keep open. Current main still lacks the requested channel agent-tool sender context and the linked issue was closed in favor of this PR, but the PR head has a blocking tool-family gate regression, a failing type check, and no real behavior proof.

## What This Changes

The PR adds optional trusted requester sender fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context to channel-owned agent-tool factories.

## Best Possible Solution

Land a narrow version that widens the ChannelAgentToolFactory/listChannelAgentTools contract and threads sender fields while preserving includeChannelTools/includeOpenClawTools, with focused regression coverage and maintainer SDK docs/changelog handling.

## Reproduction Assessment

Yes for the PR regression: the PR-head diff and CI show includeChannelTools becomes unused when channel/OpenClaw tools are gated by includeCoreTools, while current main tests require channel-only plans with includeCoreTools false and includeChannelTools true. The underlying requested SDK feature gap is source-proven on main, not runtime-reproduced here.

## Solution Assessment

No. Sender-context threading is the right narrow direction, but the current diff should preserve the per-family gates and add regression coverage before merge.
