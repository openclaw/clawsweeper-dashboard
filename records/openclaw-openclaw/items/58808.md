## Summary

Keep open. The sender-context API is still useful and current main does not implement it, but the latest PR head still replaces family-specific tool construction gates with `includeCoreTools`, which breaks plugin-only allowlists and leaves CI failing; real behavior proof is also missing.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards sender context from `createOpenClawCodingTools` into channel-owned agent-tool factories.

## Best Possible Solution

Preserve the existing family-specific gates, keep the narrow sender-context extension, add focused propagation coverage, and merge after maintainer API review plus real plugin proof or a maintainer proof override.

## Reproduction Assessment

Yes for the review finding: the live PR diff and latest `check-prod-types` failure source-reproduce the gate regression. The requested sender-context API itself is a new capability, so there is no current-main bug reproduction beyond confirming the API is absent.

## Solution Assessment

No, not as written. Passing trusted sender params is a reasonable narrow API direction, but the patch must preserve `includeChannelTools`/`includeOpenClawTools` and prove the runtime propagation path.
