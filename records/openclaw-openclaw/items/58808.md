## Summary

Keep open: the requested sender-aware channel-tool seam is useful and is not on current main, but this PR head still replaces the separate channel/OpenClaw tool-family gates with includeCoreTools and lacks required real behavior proof.

## What This Changes

This PR extends ChannelAgentToolFactory/listChannelAgentTools with optional requesterSenderId and senderIsOwner fields and attempts to forward createOpenClawCodingTools sender context to channel-owned agent-tool factories.

## Best Possible Solution

Merge a revised narrow patch that forwards sender context through the existing includeChannelTools gate, preserves includeOpenClawTools, adds focused propagation and gate coverage, and includes redacted real plugin proof or a maintainer proof override.

## Reproduction Assessment

Yes. The PR-introduced regression is source-reproducible from the live diff and current-main tests: plugin-only allowlists can enable channel tools while includeCoreTools is false, and OpenClaw tools can be disabled independently.

## Solution Assessment

No, not as written. Adding trusted sender params is a reasonable narrow direction, but the patch must preserve the family-specific includeChannelTools and includeOpenClawTools gates instead of replacing both with includeCoreTools.
