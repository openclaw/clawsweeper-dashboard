## Summary

Keep open: current main still has the requested SDK gap, but the latest PR head is not mergeable because it collapses separate channel/OpenClaw tool-family gates and fails prod typecheck; it also still lacks real behavior proof for the external sender-aware channel factory use case.

## What This Changes

The PR extends channel agent-tool factory parameters with trusted sender context and forwards that context from `createOpenClawCodingTools()` into `listChannelAgentTools()`.

## Best Possible Solution

Forward `options?.senderId` and `options?.senderIsOwner` into `listChannelAgentTools()` while keeping `listChannelAgentTools()` behind `includeChannelTools` and `createOpenClawTools()` behind `includeOpenClawTools`, then add focused regression coverage for propagation and channel-only allowlists.

## Reproduction Assessment

Yes for the PR-introduced regression: the diff gates channel and OpenClaw factories on `includeCoreTools`, and CI reproduces the resulting `TS6133` typecheck failure because `includeChannelTools` is no longer read. Current-main tests also show channel-only allowlists intentionally set `includeCoreTools: false` with `includeChannelTools: true`; I did not run local tests because this review is read-only.

## Solution Assessment

No, not as written. The narrow maintainable fix is to add the optional sender fields and pass them through the existing `includeChannelTools` branch while leaving OpenClaw tool construction on `includeOpenClawTools`.
