## Summary

Keep open: the sender-context direction is useful, but the current PR regresses per-family tool construction by gating channel and OpenClaw tools on `includeCoreTools`, and it still lacks real behavior proof from a channel-plugin setup.

## What This Changes

The branch extends `ChannelAgentToolFactory` and `listChannelAgentTools` with `requesterSenderId`/`senderIsOwner`, then forwards those values from `createOpenClawCodingTools`.

## Best Possible Solution

Land the additive sender-context seam only after preserving the existing per-family tool guards, adding focused channel-only allowlist and factory-param coverage, and documenting/reviewing the new public SDK fields.

## Reproduction Assessment

Yes. Source inspection of the PR diff against current main shows channel-only construction plans can have `includeCoreTools: false` with `includeChannelTools: true`, while the PR gates channel tools on `includeCoreTools`.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the implementation must preserve the existing `includeChannelTools` and `includeOpenClawTools` guards and include real behavior proof before merge.
