## Summary

Keep open: the sender-context API addition is useful, but the current diff is not merge-ready because it replaces family-specific tool-construction gates with `includeCoreTools`, fails prod type checking, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `requesterSenderId` and `senderIsOwner` to `ChannelAgentToolFactory` params and forwards them from `createOpenClawCodingTools` into channel-owned agent-tool factories.

## Best Possible Solution

Keep the additive sender-context fields, but restore `includeChannelTools` for channel agent tools and `includeOpenClawTools` for core OpenClaw tools, add focused regression coverage, and land only after maintainer SDK review plus redacted real behavior proof.

Yes, source inspection shows the current-main feature gap: channel agent-tool factories receive only `cfg` while adjacent OpenClaw tools receive trusted sender context. The PR regression is also source-reproducible and CI-confirmed through the unused `includeChannelTools` type error.
