## Summary

Keep open: this is a useful channel SDK feature, but the current branch still regresses channel-only tool construction and lacks after-fix real behavior proof from an external setup.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel agent-tool factories.

## Best Possible Solution

Land a narrow sender-context SDK addition that preserves per-family construction guards, adds focused forwarding coverage, and includes docs/changelog treatment if maintainers consider the new fields user-facing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main drops sender context before channel agent-tool factories, and the PR-head regression is visible because channel-only allowlists can set `includeCoreTools: false` while `includeChannelTools` remains true.

## Solution Assessment

No as written. Passing trusted sender context into the channel factory is the right narrow direction, but replacing `includeChannelTools` and `includeOpenClawTools` with `includeCoreTools` is not maintainable because it breaks existing allowlist construction semantics.
