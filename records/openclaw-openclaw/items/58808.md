## Summary

Keep open: current main still lacks sender context in the channel agent-tool factory path, and this PR is the canonical implementation, but the current head still has a blocking tool-family gate regression, a typecheck failure, and missing real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools()` and forwards `createOpenClawCodingTools()` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then add focused regression coverage and public SDK docs if maintainers want the new fields documented.

## Reproduction Assessment

Yes for source-level behavior: current main omits sender context from channel agent-tool factories, and PR head visibly gates channel/OpenClaw tools on the wrong boolean. I did not run local tests because this review is read-only, but CI also confirms a typecheck failure from the same regression.

## Solution Assessment

No. The API direction is plausible, but this patch should keep the distinct tool-family gates and needs a regression test plus contributor-owned real behavior proof before merge.
