## Summary

Keep this PR open: the sender-context API seam is reasonable, but the current branch still changes the channel/OpenClaw tool-family guards and has no after-fix real behavior proof from a real setup.

## What This Changes

The PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` with requester sender fields, then passes `options.senderId` and `options.senderIsOwner` from `createOpenClawCodingTools` while routing channel and OpenClaw tool construction through `includeCoreTools`.

## Best Possible Solution

Thread `requesterSenderId` and `senderIsOwner` through the existing channel factory params while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and real setup proof for a sender-aware channel tool.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR replaces family-specific guards with `includeCoreTools`, while current main and tests deliberately support channel/plugin-only construction with `includeCoreTools: false`. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The API addition is a narrow, maintainable direction, but the implementation should only thread the new sender fields and should preserve the existing per-family construction guards.
