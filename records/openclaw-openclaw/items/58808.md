## Summary

Keep open: the sender-context plumbing is useful, but the current PR head regresses tool-family gating in `pi-tools` and still lacks after-fix real behavior proof, so it should not merge as written.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and SDK docs/changelog updates if maintainers accept the public contract change.

## Reproduction Assessment

Yes, for the PR regression by source inspection: pass `includeCoreTools: false` with a plan where `includeChannelTools` is true, or pass a plan where `includeOpenClawTools` is false while core tools are otherwise enabled. Current main preserves those separate gates; the PR gates both factories on `includeCoreTools`.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but replacing the separate channel/OpenClaw tool-family gates with `includeCoreTools` is not maintainable or behavior-preserving.
