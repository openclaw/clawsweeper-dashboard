## Summary

Keep open: the DMWork sender-aware channel-tool use case and API gap are valid, but this PR is not merge-ready because it breaks the planned tool-family gates, currently fails prod typecheck, and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds optional sender context fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land a narrow version that only widens the factory params and forwards sender context while preserving the existing family gates, with focused regression coverage and contributor-owned real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main and existing tests allow channel/plugin tools when `includeCoreTools` is false, while the PR gates channel/OpenClaw tool creation on `includeCoreTools` and CI reports TS6133. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The API direction is narrow, but the implementation should keep channel tools under `includeChannelTools` and OpenClaw tools under `includeOpenClawTools` while adding sender params.
