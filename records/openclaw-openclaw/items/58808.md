## Summary

Keep open: the requested sender-context seam is a real public plugin/channel SDK gap, but the latest PR diff still regresses family-specific tool construction gates and lacks contributor real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factory params and forwards them from `createOpenClawCodingTools`.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused propagation coverage and maintainer-approved SDK docs/changelog if the fields are accepted.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main has construction-plan states where `includeCoreTools` intentionally diverges from `includeChannelTools` and `includeOpenClawTools`, while the PR gates both constructions on `includeCoreTools`. No tests were run because this review is read-only.

## Solution Assessment

No, not as written. The sender-context seam is narrow and plausible, but the patch should preserve the existing family-specific gates and include focused coverage before the public SDK contract change lands.
