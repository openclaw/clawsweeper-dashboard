## Summary

Keep open: current main does not implement the requested sender context, and the active PR still has a tool-family guard regression plus no required real behavior proof from a channel-plugin setup.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params to include trusted `requesterSenderId` and `senderIsOwner`, then forwards `createOpenClawCodingTools` sender context to channel agent tool factories.

## Best Possible Solution

Land a narrow sender-context seam only after restoring the existing per-family guards, adding focused regression coverage for channel factory context forwarding, and documenting the public SDK contract if maintainers treat it as user-facing.

## Reproduction Assessment

Yes for the blocking PR regression: source inspection shows the live PR head gates channel and OpenClaw tool construction on `includeCoreTools`, while current tests define channel-only allowlists with `includeCoreTools: false`. The original feature request is source-verifiable on current main because the factory type and call still pass only `cfg`.

## Solution Assessment

No as written. Passing trusted sender context into `ChannelAgentToolFactory` is the right narrow direction, but the patch must preserve `includeChannelTools`/`includeOpenClawTools` and include real behavior proof plus regression coverage before merge.
