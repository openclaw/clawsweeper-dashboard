## Summary

Keep https://github.com/openclaw/openclaw/pull/58808 open. The sender-context API addition is useful, but the current patch still changes the channel/OpenClaw tool guards in a way that conflicts with current main's per-family construction plan, and the external PR still lacks real behavior proof.

## What This Changes

This PR adds optional `requesterSenderId` and `senderIsOwner` params to channel agent-tool factories and forwards sender context from `createOpenClawCodingTools`, while also gating channel and OpenClaw tool construction on `includeCoreTools`.

## Best Possible Solution

Land an equivalent narrow SDK addition that forwards trusted sender context through channel agent-tool factories while preserving per-family construction guards, with regression coverage, changelog coverage, and redacted real setup proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR gates channel and OpenClaw tool construction on `includeCoreTools`, while current main and tests intentionally support plugin/channel-only construction with `includeCoreTools: false`. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The sender-context API direction is narrow and maintainable, but the implementation should keep `includeChannelTools` and `includeOpenClawTools` as the guards and add changelog, regression coverage, and real behavior proof.
