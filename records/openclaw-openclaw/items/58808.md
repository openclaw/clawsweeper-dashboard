## Summary

Keep this PR open: the sender-context seam is useful and still missing on current main, but the current patch regresses tool-family construction guards and lacks external contributor real behavior proof.

## What This Changes

The PR adds optional requester sender fields to channel agent-tool factory parameters and forwards existing sender context from `createOpenClawCodingTools`, while also changing two tool-family guards to `includeCoreTools`.

## Best Possible Solution

Land this as a narrow SDK-compatible change only after preserving the existing per-family guards, adding focused coverage plus docs/changelog, and posting redacted real behavior proof from a channel-plugin setup.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main keeps channel and OpenClaw tool-family guards separate, while the PR diff gates both branches on `includeCoreTools`; existing construction-plan tests cover channel-only allowlists.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the implementation must keep `includeChannelTools` and `includeOpenClawTools` guards intact and needs proof/coverage before merge.
