## Summary

Keep open: the sender-context plumbing is useful, but the latest branch still changes the tool-family gates and lacks after-fix real behavior proof.

## What This Changes

The branch adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and SDK docs/changelog updates if maintainers treat the fields as public contract.

## Reproduction Assessment

Yes, for the PR regression by source inspection: current main supports plugin/channel-only construction with `includeCoreTools: false`, while the PR gates channel tools and OpenClaw tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the separate channel and OpenClaw tool-family gates.
