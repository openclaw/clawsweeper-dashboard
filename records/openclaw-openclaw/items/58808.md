## Summary

Keep open: the sender-context API gap is real and the PR is useful, but the current diff still collapses channel/OpenClaw tool-family gates under includeCoreTools and lacks required after-fix real behavior proof.

## What This Changes

The PR extends ChannelAgentToolFactory/listChannelAgentTools with optional requesterSenderId and senderIsOwner fields and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Keep the additive sender-context seam only after preserving includeChannelTools/includeOpenClawTools exactly, adding focused regression coverage, and providing redacted real behavior proof from a channel plugin that uses the new fields.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a plugin/channel-only plan has includeCoreTools false while includeChannelTools remains true, but the PR gates channel tools on includeCoreTools. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. Passing trusted sender fields through ChannelAgentToolFactory is the narrow direction, but replacing the separate family gates with includeCoreTools is not behavior-preserving.
