## Summary

Keep open: the API direction is plausible, but the patch currently collapses separate tool-construction gates and lacks contributor-owned real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds trusted `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory` and threads them from `createOpenClawCodingTools` into channel agent-tool factories.

## Best Possible Solution

Preserve the existing `includeChannelTools` and `includeOpenClawTools` gates while passing the new trusted sender fields into `listChannelAgentTools`, add focused coverage, and require redacted real channel-plugin proof before merge.

## Reproduction Assessment

Yes for the PR regression by source inspection: the construction planner can set `includeCoreTools` false while `includeChannelTools` is true, but the PR gates channel tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. Threading sender context is the right narrow direction, but the implementation should keep the planned channel/OpenClaw tool-family gates separate and add proof for the public SDK behavior.
