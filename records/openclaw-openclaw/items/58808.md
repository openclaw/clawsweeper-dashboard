## Summary

Keep open: the requested sender-context seam is plausible and still missing on current main, but the latest PR diff regresses tool-family gating and still lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` parameters to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context to channel agent-tool factories.

## Best Possible Solution

Add the trusted sender fields through the channel-agent-tool factory path while preserving `includeChannelTools`/`includeOpenClawTools` gates, with focused propagation tests and SDK docs/changelog if maintainers accept the public contract.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows current main supports channel-only construction plans such as `whatsapp_login`, while the PR gates channel factories on `includeCoreTools` and would skip that path. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The requested seam is reasonable, but the patch must preserve the existing per-family gates and prove the sender-aware channel factory path works in a real setup.
