Keep this PR open, but it should not merge as written: the requested sender-context gap is real, yet the current diff also changes tool-family gates in a way that can break plugin-only and narrow allowlists, and the external contributor still needs real behavior proof.

## What This Changes

The PR adds optional trusted sender fields to ChannelAgentToolFactory/listChannelAgentTools, forwards createOpenClawCodingTools sender context, and changes the channel/OpenClaw tool guards in pi-tools to use includeCoreTools.

## Best Possible Solution

Thread requesterSenderId and senderIsOwner through ChannelAgentToolFactory/listChannelAgentTools while preserving the existing tool-family gates, then add focused regression coverage and SDK docs/changelog if maintainers accept the public contract change.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main keeps channel and OpenClaw tool-family gates separate, and construction-plan tests encode plugin-only allowlists with includeCoreTools false but includeChannelTools true. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The sender-context plumbing is the right narrow direction, but replacing includeChannelTools/includeOpenClawTools with includeCoreTools is not behavior-preserving.
