## Summary

Keep this PR open, but do not merge the current head. The sender-context addition is useful and not already on main, but the latest diff still replaces the existing channel/OpenClaw tool-family gates with `includeCoreTools`, and the PR still lacks required real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` parameters to `ChannelAgentToolFactory` and threads `createOpenClawCodingTools` sender options into `listChannelAgentTools`.

## Best Possible Solution

Keep the additive sender-context threading, but preserve `includeChannelTools` and `includeOpenClawTools`, add focused regression coverage, and merge only after real behavior proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes. The latest PR diff is enough to reproduce the review finding from source: construction plans intentionally allow channel/plugin-only paths with `includeCoreTools: false`, but the patch gates channel and OpenClaw tools on `includeCoreTools`; the latest CI also reports `check-prod-types` failure.

## Solution Assessment

No. Passing trusted sender context to channel agent-tool factories is the right narrow direction, but the implementation should keep channel tools under `includeChannelTools` and OpenClaw tools under `includeOpenClawTools`.
