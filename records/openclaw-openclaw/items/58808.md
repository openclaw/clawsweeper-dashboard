## Summary

Keep this PR open, but the current patch is not merge-ready: the sender-context API direction is plausible, yet the diff replaces the existing channel/OpenClaw tool-family gates with `includeCoreTools`, which drops channel tools for channel-only plans and can construct OpenClaw tools for plans that explicitly disabled them. The PR also still lacks after-fix real behavior proof for the sender-aware channel-tool path.

## What This Changes

The branch extends `ChannelAgentToolFactory`/`listChannelAgentTools` with optional trusted sender fields and forwards them from `createOpenClawCodingTools`, while changing the channel/OpenClaw construction guards to `includeCoreTools`.

## Best Possible Solution

Preserve the existing per-family guards, forward the new sender fields through only the channel-agent-tool factory path, add focused regression coverage for sender params and construction plans, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced regression by source inspection and CI: current construction plans can have `includeCoreTools:false` with `includeChannelTools:true`, and the latest `check-prod-types` run reports `includeChannelTools` unused after the PR replaces its guard.

## Solution Assessment

No. Forwarding trusted sender fields is the right direction, but the maintainable fix must preserve `includeChannelTools` and `includeOpenClawTools` rather than replacing them with `includeCoreTools`.
