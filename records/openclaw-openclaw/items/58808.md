## Summary

Keep open: the sender-context seam is useful and matches a real DMWork permission use case, but the latest PR head still regresses tool-family allowlist gates and lacks after-fix real behavior proof.

## What This Changes

The branch extends ChannelAgentToolFactory/listChannelAgentTools with optional trusted sender fields and passes createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving the existing includeChannelTools and includeOpenClawTools gates, then add focused regression coverage and SDK docs/changelog only if maintainers want this public contract documented now.

## Reproduction Assessment

Yes, for the PR regression by source inspection: current main supports includeCoreTools false with includeChannelTools true, while the PR gates channel tools on includeCoreTools. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the existing separate channel and OpenClaw tool-family gates.
