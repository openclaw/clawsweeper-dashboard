## Summary

Keep open: the sender-context feature gap is real and linked to a concrete DMWork access-control use case, but the current PR head changes family-specific tool construction gates to `includeCoreTools`, causing a source- and CI-proven regression, and it still lacks contributor real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with `requesterSenderId` and `senderIsOwner` and forwards `createOpenClawCodingTools` sender options into channel-owned agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, restore `includeChannelTools` and `includeOpenClawTools`, add focused propagation coverage, and merge only after real behavior proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes for the blocking PR regression: the live diff and CI show channel-agent/OpenClaw construction now keyed off `includeCoreTools` instead of the family-specific gates. The original sender-context request is a feature gap on current main, not a failing current-main bug.

## Solution Assessment

No, not as written. Passing trusted sender params is a narrow maintainable API direction, but the implementation must preserve the existing tool-family construction gates and prove the runtime propagation path.
