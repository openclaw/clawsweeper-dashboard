## Summary

Keep open: the sender-context API direction is useful, including for the linked DMWork channel use case, but the latest diff still changes channel and OpenClaw tool materialization to use `includeCoreTools`, which breaks current narrow allowlist behavior. The external PR also still lacks contributor-owned real behavior proof.

## What This Changes

This PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` with `requesterSenderId` and `senderIsOwner`, then forwards sender context from `createOpenClawCodingTools` to channel agent tool factories.

## Best Possible Solution

Keep the sender-context API addition, but pass the new fields under the existing `includeChannelTools` gate and leave `createOpenClawTools` under `includeOpenClawTools`; add focused regression coverage before merge.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main and existing tests show channel/plugin-only allowlists can require channel tools while `includeCoreTools` is false. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The API direction is narrow, but the implementation should preserve the family-specific gates instead of using `includeCoreTools` for channel and OpenClaw tool materialization.
