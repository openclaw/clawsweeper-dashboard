## Summary

Keep open: the requested sender-context API gap is real on current main, but this PR is not mergeable as written because it changes the channel and OpenClaw tool-family gates to `includeCoreTools`, breaking narrow allowlist construction and leaving real behavior proof missing.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with `requesterSenderId` and `senderIsOwner`, then forwards `createOpenClawCodingTools()` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land an equivalent narrow additive propagation that preserves `includeChannelTools` and `includeOpenClawTools`, adds focused regression coverage, and documents or changelogs the public SDK fields if maintainers treat them as user-facing.

## Reproduction Assessment

Yes for the PR-introduced regression from source: current main intentionally allows channel/plugin construction when `includeCoreTools` is false, while the PR gates those factories on `includeCoreTools`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The right shape is the additive sender-field propagation while preserving the existing channel and OpenClaw tool-family gates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep the channel and OpenClaw tool gates separate:** `src/agents/pi-tools.ts:840-847`
  - body: This hunk switches both `listChannelAgentTools()` and `createOpenClawTools()` to `includeCoreTools`. Narrow channel/plugin allowlists intentionally produce `includeCoreTools: false` with `includeChannelTools: true`, so channel-owned tools such as `whatsapp_login` would be skipped; plans with `includeOpenClawTools: false` can also materialize the wrong core tool family. Keep the existing gates and only add the new sender fields to the channel-tools call.
