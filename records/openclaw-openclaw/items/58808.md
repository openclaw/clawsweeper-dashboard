## Summary

Keep open. The linked DMWork sender-aware factory use case is legitimate and current main still lacks that factory context, but this PR head also changes the planned tool-family gates so channel/plugin-only plans can lose channel tools and disabled OpenClaw tools can be constructed; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds optional trusted requester sender fields to `ChannelAgentToolFactory` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving the existing tool-family gates, then add focused regression coverage and SDK docs or changelog updates if maintainers treat the contract as user-facing.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main supports `includeCoreTools: false` with `includeChannelTools: true` for plugin/channel-only allowlists, while the PR diff gates channel and OpenClaw tool construction on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The narrow maintainable fix is to add the sender fields to the existing `listChannelAgentTools` call while keeping the separate `includeChannelTools` and `includeOpenClawTools` gates intact.
