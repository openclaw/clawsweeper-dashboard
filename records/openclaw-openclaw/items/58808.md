## Summary

Keep open. The requested sender-context seam is useful, but the current PR diff regresses planned tool-family materialization by gating channel and OpenClaw factories on `includeCoreTools`, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch extends `ChannelAgentToolFactory` and `listChannelAgentTools` with optional sender context fields, and changes tool composition to call channel/OpenClaw factories under the broad `includeCoreTools` gate.

## Best Possible Solution

Preserve the existing tool-family gates, pass the sender context through `listChannelAgentTools`, add focused regression coverage for the factory params, then merge after maintainer contract review and real behavior proof.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the PR gates channel/OpenClaw factories on `includeCoreTools` even though current planner cases intentionally separate those booleans. I did not run tests because this was a read-only review.

## Solution Assessment

No. The narrow fix is to keep `includeChannelTools` and `includeOpenClawTools` as the materialization gates and only add `requesterSenderId`/`senderIsOwner` to the `listChannelAgentTools` params.
