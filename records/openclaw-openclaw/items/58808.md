## Summary

Keep open: current main still lacks sender context on channel agent-tool factories and this PR is the canonical implementation path, but the current diff would regress channel-only tool allowlists and still lacks contributor real behavior proof.

## What This Changes

This PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with `requesterSenderId` and `senderIsOwner`, then forwards `createOpenClawCodingTools` sender context into channel agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, preserve the existing `includeChannelTools`/`includeOpenClawTools` guards, add focused propagation coverage, and land after SDK review plus real plugin proof.

## Reproduction Assessment

Yes, source-reproducible from current main and the PR diff. The construction plan can return `includeCoreTools: false` with `includeChannelTools: true` for `whatsapp_login`, while the PR gates channel tools on `includeCoreTools`.

## Solution Assessment

No as written. Passing trusted sender context into the factory is the right narrow direction, but the PR should preserve the per-family guards and add propagation coverage before merge.
