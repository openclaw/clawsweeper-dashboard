## Summary

Keep open: the requested sender-context API gap is real, but the current PR head is not mergeable because it replaces the separate channel/OpenClaw tool-family gates with `includeCoreTools`, causing a least-privilege allowlist regression and a prod type failure; it also lacks external real behavior proof.

## What This Changes

The PR extends `ChannelAgentToolFactory` and `listChannelAgentTools` with optional sender fields and forwards `createOpenClawCodingTools` sender context into channel-owned agent tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after preserving the existing tool-family gates, adding focused regression coverage, handling SDK docs/changelog expectations, and posting redacted real behavior proof from a channel plugin.

## Reproduction Assessment

Yes, by source inspection and CI: current main supports channel/plugin-only construction with `includeCoreTools: false`, while the PR gates channel tools on `includeCoreTools` and the live prod type check reports `includeChannelTools` is unused. I did not run local tests because this was a read-only review.

## Solution Assessment

No, not as written. Passing trusted sender fields is the narrow direction, but the patch must keep `includeChannelTools` for channel tools and `includeOpenClawTools` for OpenClaw tools instead of collapsing them into `includeCoreTools`.
