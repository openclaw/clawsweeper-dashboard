## Summary

Keep open: the SDK gap is real, but the current PR head regresses the existing per-family tool construction gates and lacks required contributor real-behavior proof and changelog coverage.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` params to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards coding-tool sender options into channel agent-tool factories.

## Best Possible Solution

Add the optional trusted sender fields through the existing `includeChannelTools` path, preserve `includeOpenClawTools`, add regression and changelog coverage, and land only after real DMWork/OpenClaw proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a public plugin SDK feature request. Source inspection confirms current main only passes `cfg` to channel tool factories, while current tests define the construction-gate behavior the PR must preserve.

## Solution Assessment

No. The API direction is plausible, but the patch should thread sender context under the existing `includeChannelTools` gate and leave `createOpenClawTools` behind `includeOpenClawTools`.
