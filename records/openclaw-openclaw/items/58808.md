## Summary

Keep open: the sender-context seam is useful, but this PR currently regresses tool-family construction gates and still lacks after-fix real behavior proof.

## What This Changes

The PR extends `ChannelAgentToolFactory`/`listChannelAgentTools` with optional `requesterSenderId` and `senderIsOwner` and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Land a narrow additive SDK propagation that preserves `includeChannelTools` and `includeOpenClawTools`, adds focused regression coverage, and includes redacted proof from a real channel plugin.

## Reproduction Assessment

Yes. Source inspection plus existing construction-plan tests show a channel-only allowlist can have `includeCoreTools: false` and `includeChannelTools: true`, so the PR’s `includeCoreTools` guard would skip allowed channel tools.

## Solution Assessment

No, not as written. The added sender fields are the right shape, but the implementation should keep the separate family gates and prove the real plugin runtime path.
