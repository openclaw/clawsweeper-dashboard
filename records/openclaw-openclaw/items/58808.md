## Summary

This PR should stay open: current main still lacks the requested sender context for channel agent-tool factories, but the current branch regresses the existing tool-family construction gates and still lacks real behavior proof for the external setup described in the PR.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory` parameters and forwards `senderId`/`senderIsOwner` from `createOpenClawCodingTools` into `listChannelAgentTools`.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving the existing tool-family gates, then add focused regression coverage for the propagation and allowlist split.

## Reproduction Assessment

Yes, for the PR regression by source inspection: current main supports plugin/channel-only construction with `includeCoreTools` false, while the PR gates channel tools and OpenClaw tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep `includeChannelTools` and `includeOpenClawTools` as separate construction gates.
