## Summary

Keep this PR open for revision: the sender-context API direction is useful, but the branch currently replaces the existing per-family tool-construction gates with `includeCoreTools`, which can both expose OpenClaw tools in narrow core-only plans and drop channel tools in plugin/channel-only plans. It also still lacks after-fix real behavior proof from an external setup.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory` params and forwards existing sender context from `createOpenClawCodingTools` into channel agent-tool discovery.

## Best Possible Solution

Thread the optional sender context through `listChannelAgentTools` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, with focused regression coverage for sender params and narrow construction plans.

## Reproduction Assessment

Yes, by source inspection. Current main can produce `includeCoreTools: true` with `includeOpenClawTools: false` and `includeCoreTools: false` with `includeChannelTools: true`; the PR's `includeCoreTools` guards bypass both distinctions.

## Solution Assessment

No. The sender-context seam is the right place to solve the plugin access-control gap, but this patch should preserve the existing per-family construction gates instead of widening them.
