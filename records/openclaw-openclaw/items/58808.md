## Summary

Keep open: current main still lacks the requested sender-aware channel tool factory context, and the DMWork access-control use case is valid, but this branch currently regresses tool-family gating and still lacks real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and SDK docs/changelog updates if maintainers treat the fields as user-facing.

## Reproduction Assessment

Yes, for the PR regression by source inspection: current main supports channel/plugin-only construction with `includeCoreTools: false`, while the PR gates channel tools and OpenClaw tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the separate channel and OpenClaw tool-family gates.
