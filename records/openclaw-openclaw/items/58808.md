## Summary

Keep open: the requested sender-context seam is plausible and still missing from current main, but the current PR regresses tool-family allowlist gating and lacks real behavior proof for the proposed channel factory path.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent tool factories.

## Best Possible Solution

Add the sender fields through the channel-agent-tool factory path while preserving the separate channel/OpenClaw construction gates, then cover the propagation with focused tests and document the public SDK contract if maintainers accept it.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the diff gates channel and OpenClaw factories on `includeCoreTools`, while current main intentionally keeps `includeChannelTools` and `includeOpenClawTools` separate. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The requested seam is reasonable, but the implementation must preserve the existing per-family gates and prove the real sender-aware channel factory path works.
