## Summary

Keep open: the linked DMWork use case is a plausible plugin-side need and current main still lacks sender context in channel agent-tool factories, but this PR is not mergeable as written because it changes family-specific tool construction gates and still lacks contributor real behavior proof.

## What This Changes

This PR adds optional trusted sender fields to channel agent-tool factory params and forwards sender context from `createOpenClawCodingTools` through `listChannelAgentTools`.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then add focused propagation coverage before accepting the SDK contract change.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main has construction-plan states where `includeCoreTools` intentionally diverges from `includeChannelTools` and `includeOpenClawTools`, while the PR gates both constructions on `includeCoreTools`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The sender-context seam is narrow and plausible, but the patch should preserve family-specific gates and add focused coverage before maintainer SDK-contract approval.
