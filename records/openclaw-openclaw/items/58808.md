## Summary

Keep open: the sender-context seam is still useful and not on current main, but this PR is not mergeable as written because it replaces family-specific tool gates with `includeCoreTools`, fails prod typecheck, and still lacks required real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards them from `createOpenClawCodingTools`.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving the existing per-family construction gates, adding focused propagation coverage, and documenting the SDK seam if maintainers accept it.

## Reproduction Assessment

Yes for the PR-introduced regression: the diff replaces family-specific gates with `includeCoreTools`, current source/tests show those gates are intentionally independent, and the latest prod typecheck fails from the now-unused `includeChannelTools`. I did not run local tests because this review is read-only.

## Solution Assessment

No, not as written. The optional sender fields are a narrow plausible seam, but the patch should keep `includeChannelTools` and `includeOpenClawTools` as separate guards and add regression coverage plus real setup proof.
