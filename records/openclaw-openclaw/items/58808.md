## Summary

Keep open: the requested sender-context seam is useful, but the PR still collapses existing channel/OpenClaw tool-family gates and the external contributor has not provided real behavior proof.

## What This Changes

The PR widens ChannelAgentToolFactory/listChannelAgentTools params with requesterSenderId and senderIsOwner, then forwards createOpenClawCodingTools sender context into channel agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through ChannelAgentToolFactory/listChannelAgentTools while preserving includeChannelTools and includeOpenClawTools, then add focused regression coverage and SDK documentation if maintainers accept the contract change.

## Reproduction Assessment

Yes for the PR regression by source inspection: current construction-plan tests define includeCoreTools false with includeChannelTools true for plugin/channel allowlists, while the PR gates channel tools on includeCoreTools. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. Threading sender fields is the right narrow direction, but the patch must keep includeChannelTools/includeOpenClawTools separate and add regression coverage plus real behavior proof.
