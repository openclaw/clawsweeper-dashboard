## Summary

Keep open: the requested sender-context seam is useful, but the current PR still regresses the per-family tool construction gates and lacks required after-fix real behavior proof from a real channel plugin setup.

## What This Changes

The branch adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards sender context from pi-tools while changing the channel/OpenClaw tool construction guards to includeCoreTools.

## Best Possible Solution

Expose trusted sender context through the channel-agent-tool SDK seam while preserving includeChannelTools and includeOpenClawTools as separate gates, with focused regression coverage and redacted real plugin proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main and tests preserve separate channel/OpenClaw construction gates, while the PR gates both families on includeCoreTools. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep listChannelAgentTools behind includeChannelTools and createOpenClawTools behind includeOpenClawTools.
