## Summary

Keep open. The requested public channel-tool seam is still useful and not on current main, but the current PR head regresses existing tool-family allowlist gates and lacks required real behavior proof for an external PR.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then add focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main can plan `includeCoreTools: false` with `includeChannelTools: true`, while the PR gates channel tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must preserve the separate channel and OpenClaw tool-family gates and the PR needs proof for the public SDK behavior.
