## Summary

Keep open: the sender-aware channel-tool seam is useful, but this PR should not merge as written because it regresses the tool-family construction gates and still lacks real behavior proof.

## What This Changes

The branch widens `ChannelAgentToolFactory`/`listChannelAgentTools` to accept sender context and forwards it from `createOpenClawCodingTools`, while also changing channel/OpenClaw tool construction guards to `includeCoreTools`.

## Best Possible Solution

Land an equivalent narrow SDK propagation that preserves the family-specific construction gates, adds focused sender-context coverage, and includes docs/changelog updates if maintainers treat the added fields as user-facing.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current construction-plan tests define channel-only allowlists with `includeCoreTools: false` and `includeChannelTools: true`, while the patch gates channel tools on `includeCoreTools`. I did not run tests in this read-only review.

## Solution Assessment

No, not as written. The added sender fields are the right narrow seam, but the implementation should keep `listChannelAgentTools` behind `includeChannelTools` and `createOpenClawTools` behind `includeOpenClawTools`.

## Review Findings

Overall correctness: patch is incorrect
