## Summary

Keep this PR open: the requested sender-context API is useful and current main still lacks it, but the current diff has a blocking gate regression in `createOpenClawCodingTools` and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with trusted sender fields and forwards `options.senderId`/`options.senderIsOwner` from `createOpenClawCodingTools`.

## Best Possible Solution

Expose the sender context through the channel agent-tool factory only while preserving the existing channel/OpenClaw construction gates, then add focused regression coverage and contributor-owned real behavior proof.

## Reproduction Assessment

Yes for the source-level gap: current main only passes `cfg` into channel `agentTools` factories while the adjacent core-tool path receives sender context. I did not run a live channel plugin proof because this was a read-only review.

## Solution Assessment

No. Adding optional sender fields is a plausible API direction, but the current patch is not the best implementation because it regresses the existing tool-family gates and still lacks real behavior proof.
