## Summary

Keep open: the SDK seam is plausible, but the current PR head still regresses tool-family construction by using `includeCoreTools` for channel/OpenClaw factories and it lacks required real behavior proof.

## What This Changes

The branch extends `ChannelAgentToolFactory`/`listChannelAgentTools` with optional trusted sender fields and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Keep the sender-context addition, restore family-specific construction guards, add focused forwarding and allowlist regression coverage, then land after SDK review and redacted real channel-plugin proof.

## Reproduction Assessment

Yes. Source inspection against current `origin/main` shows channel agent-tool factories receive only config while adjacent OpenClaw tools already receive trusted sender context; existing construction-plan tests also give a high-confidence path for the guard regression.

## Solution Assessment

No as written. Passing trusted sender context into the channel factory is the right narrow direction, but replacing `includeChannelTools`/`includeOpenClawTools` with `includeCoreTools` is not the maintainable fix because it breaks existing allowlist construction semantics.
