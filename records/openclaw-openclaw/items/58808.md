## Summary

Keep open: the requested sender-context API gap still exists on current main and the PR is useful in direction, but the latest head still regresses tool-family materialization gates and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Keep the sender-context API addition, preserve the family-specific materialization gates, add focused coverage for factory params, and land after maintainer SDK review plus real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: latest PR head gates channel and OpenClaw tool construction on `includeCoreTools`, while current main and planner tests keep `includeChannelTools` and `includeOpenClawTools` separate. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The API direction is narrow, but the implementation should pass the new fields under the existing `includeChannelTools` gate and leave `createOpenClawTools` under `includeOpenClawTools`.
