## Summary

Keep open: current main still lacks the requested sender context and the paired issue was closed in favor of this PR, but the latest head regresses tool-family gating and lacks real behavior proof.

## What This Changes

The PR adds optional requesterSenderId/senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools and forwards them from createOpenClawCodingTools, while changing the channel and OpenClaw tool factory gates to includeCoreTools.

## Best Possible Solution

Thread the trusted sender fields through the channel agent-tool factory while preserving includeChannelTools and includeOpenClawTools, then add focused regression coverage and SDK docs/changelog updates if maintainers accept the public contract change.

## Reproduction Assessment

Yes for the PR regression by source inspection: plugin-only plans can set includeCoreTools false while includeChannelTools is true, and core-only plans can set includeCoreTools true while includeOpenClawTools is false. The PR gates both factories on includeCoreTools, so those existing combinations change behavior.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but replacing the separate channel/OpenClaw tool-family gates with includeCoreTools is not behavior-preserving.
