## Summary

Keep open: the sender-context API direction is useful and matches the linked DMWork use case, but the live head still swaps family-specific tool gates to `includeCoreTools`, fails prod typecheck, and lacks after-fix real behavior proof.

## What This Changes

This PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factory params and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after restoring `includeChannelTools`/`includeOpenClawTools`, adding focused sender-context propagation coverage, and providing contributor proof or a maintainer proof override.

## Reproduction Assessment

Yes for the PR regression: the diff source-reproduces it by replacing the family-specific gates with `includeCoreTools`, while current-main tests define plugin/channel-only construction plans. The sender-context request itself is a feature gap, not a current-main bug reproduction.

## Solution Assessment

No, not as written. Passing trusted sender params is a reasonable narrow direction, but the patch must preserve the existing tool-family gates and prove the runtime propagation path.
