## Summary

Keep open: current main still lacks sender context for channel agent-tool factories, and this PR is the canonical implementation path, but the current diff regresses tool-family gates and still lacks contributor-owned real behavior proof.

## What This Changes

The branch adds optional requester sender fields to ChannelAgentToolFactory/listChannelAgentTools and forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving includeChannelTools/includeOpenClawTools, then add focused regression coverage and real channel proof before merge.

## Reproduction Assessment

Yes for source-level behavior: current main only passes cfg to channel agent-tool factories while adjacent OpenClaw tools receive sender context, and the PR diff shows the gate regression. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No. The additive API direction is reasonable, but the patch should keep the distinct channel and OpenClaw construction gates instead of using includeCoreTools for both.
