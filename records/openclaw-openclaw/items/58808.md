## Summary

Keep open: the sender-context SDK seam is useful, but this PR currently regresses tool-family construction gates and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to channel agent-tool factory params and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Land a narrow additive SDK propagation that preserves `includeChannelTools` and `includeOpenClawTools`, adds focused regression coverage, and documents or changelogs the new plugin API if maintainers treat it as user-facing.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection. A channel-only allowlist can produce `includeCoreTools: false` with `includeChannelTools: true`, so the PR's `includeCoreTools` guard would skip allowed channel-owned tools.

## Solution Assessment

No, not as written. The added sender fields are the right shape, but the implementation should keep the separate family gates and prove the real plugin runtime path.
