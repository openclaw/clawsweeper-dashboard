## Summary

Keep open: the PR is a plausible narrow SDK seam for sender-aware channel agent tools, and linked issue https://github.com/openclaw/openclaw/issues/58806 was already closed in favor of this PR, but the current branch introduces a tool-family gate regression and still lacks real behavior proof from a channel plugin setup.

## What This Changes

The branch adds optional requesterSenderId and senderIsOwner fields to ChannelAgentToolFactory/listChannelAgentTools, forwards createOpenClawCodingTools sender context into channel-owned agent-tool factories, and changes the pi-tools construction guards.

## Best Possible Solution

Expose trusted sender context through the channel-agent-tool SDK seam while preserving the existing family-specific construction gates and adding focused regression coverage, with docs/changelog if maintainers treat the new fields as user-facing.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current source and tests preserve separate channel/OpenClaw construction gates, while the PR gates both families on includeCoreTools. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep listChannelAgentTools behind includeChannelTools and createOpenClawTools behind includeOpenClawTools.
