## Summary

Keep open. The requested SDK addition is useful, but the current PR head still replaces the existing split `includeChannelTools` and `includeOpenClawTools` gates with `includeCoreTools`, which drops valid channel/plugin-only tool construction and can materialize OpenClaw tools for plans that intentionally disabled them; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards sender context from `createOpenClawCodingTools`.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, add focused regression coverage, and land only after redacted real channel-plugin proof is provided.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main can plan `includeCoreTools: false` with `includeChannelTools: true`, while the PR gates channel tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must preserve the planned channel/OpenClaw tool-family gates and add proof for the public SDK behavior.
