## Summary

Keep open: the sender-context API addition is plausible and tied to a concrete DMWork plugin use case, but the current diff still changes tool-family construction guards in a way source/tests show would break narrow allowlist plans and it has no real behavior proof.

## What This Changes

The PR adds optional trusted sender context fields to channel agent tool factories and forwards sender data from `createOpenClawCodingTools`.

## Best Possible Solution

Land the sender-context API as a narrow SDK change that keeps the existing per-family guards, adds focused regression tests, and includes redacted proof from a sender-aware channel tool path.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: a plan with `includeCoreTools: false` and `includeChannelTools: true` would now skip channel tools, and a plan with `includeCoreTools: true` but `includeOpenClawTools: false` would still call `createOpenClawTools`. I did not run tests because this was a read-only review.

## Solution Assessment

No; the maintainable fix is to add the optional sender params without changing the existing `includeChannelTools` and `includeOpenClawTools` guards. The trusted-context API also needs focused regression coverage and real sender-aware plugin proof before merge.
