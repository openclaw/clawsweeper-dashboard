## Summary

Keep this PR open: the requested sender-context seam is useful, but the latest diff regresses existing tool-family construction gates and still lacks contributor real behavior proof.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` parameters to carry `requesterSenderId` and `senderIsOwner`, then forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through the channel-agent-tool factory path while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused propagation coverage and SDK docs if maintainers accept the public contract.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current main intentionally permits `includeChannelTools: true` with `includeCoreTools: false`, while the PR guards channel factories with `includeCoreTools`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The API seam is reasonable, but the patch must preserve the existing per-family construction gates and include proof that sender context reaches a real channel factory.
