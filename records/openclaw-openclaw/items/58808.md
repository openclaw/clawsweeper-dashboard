## Summary

Keep open: the PR is useful in direction, but it is not mergeable as written because it still changes the per-family tool construction guards and the external PR lacks real behavior proof for the new sender-context path.

## What This Changes

The PR widens ChannelAgentToolFactory/listChannelAgentTools params with optional requesterSenderId and senderIsOwner and forwards createOpenClawCodingTools sender context into channel agent-tool factories.

## Best Possible Solution

Keep the additive sender-context API, preserve the existing includeChannelTools/includeOpenClawTools guards, add focused propagation coverage, and land after SDK review with redacted real plugin proof.

## Reproduction Assessment

Yes, source-reproducible. Current main has a cfg-only ChannelAgentToolFactory and current tests show channel-only allowlists can set includeCoreTools:false while includeChannelTools remains true.

## Solution Assessment

No as written. Passing trusted sender context into the channel factory is the narrow maintainable direction, but the patch should keep listChannelAgentTools gated by includeChannelTools and createOpenClawTools gated by includeOpenClawTools.
