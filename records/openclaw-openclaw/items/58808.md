## Summary

Keep open: the sender-context API direction is valid and current main still lacks it for channel agent-tool factories, but the latest PR head regresses tool-family gating and still has no contributor-owned real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender options into channel agent-tool factories.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and redacted real setup proof.

## Reproduction Assessment

Yes for source-level review: current main passes only `cfg` into channel agent-tool factories while adjacent OpenClaw tools receive trusted sender context. The PR regression is also source-reproducible from the diff and existing construction-plan tests that separate channel and OpenClaw tool families.

## Solution Assessment

No. The sender-context API direction is reasonable, but this patch should preserve the distinct `includeChannelTools` and `includeOpenClawTools` gates instead of using `includeCoreTools` for both.
