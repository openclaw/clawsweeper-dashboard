## Summary

Keep open: the sender-context API direction is source-backed, but the latest PR head still replaces the plan-specific channel/OpenClaw tool gates with `includeCoreTools` and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds optional trusted sender fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `senderId`/`senderIsOwner` from coding-tool creation.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving `includeChannelTools` and `includeOpenClawTools`, then cover both sender forwarding and allowlist-driven construction behavior.

## Reproduction Assessment

Yes for source-level behavior: current main only passes config into channel agent-tool factories while adjacent OpenClaw tools receive trusted sender context. The PR regression is also source-reproducible from the latest diff and construction-plan tests.

## Solution Assessment

No. The API direction is reasonable, but this patch should preserve the distinct channel and OpenClaw tool-family gates instead of using `includeCoreTools` for both.
