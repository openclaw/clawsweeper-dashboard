## Summary

Keep open: the sender-context API gap is real, but the current diff regresses per-family tool construction guards and still lacks real behavior proof for an external PR.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Land the narrow sender-context SDK addition after preserving the existing per-family guards, adding focused regression coverage and changelog/docs as required, and posting redacted real behavior proof.

## Reproduction Assessment

Yes for the PR regression: current main source and tests show `includeCoreTools: false` can still require `includeChannelTools: true` for plugin/channel-only allowlists such as `whatsapp_login`. The PR diff would gate channel tools and OpenClaw tools on `includeCoreTools` instead.

## Solution Assessment

No as written. The sender-context API direction is narrow, but the implementation should retain `includeChannelTools` and `includeOpenClawTools`, add regression coverage/changelog, and include real behavior proof before merge.
