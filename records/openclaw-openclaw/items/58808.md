## Summary

Keep open. The API gap is real on current main and the PR adds the useful contract extension, but the latest head still regresses tool-family construction gates and lacks contributor-owned real behavior proof.

## What This Changes

The PR adds optional trusted sender context fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender options into channel-owned agent-tool factories.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving `includeChannelTools` and `includeOpenClawTools`, then add regression coverage and redacted real setup proof.

## Reproduction Assessment

Yes for source-level review: current main passes only `cfg` into channel agent-tool factories while adjacent OpenClaw tools receive trusted sender context. The PR regression is also source-reproducible by comparing the latest head with the construction-plan tests that separate channel and OpenClaw tool families.

## Solution Assessment

No. The sender-context API direction is reasonable, but this patch should preserve the distinct `includeChannelTools` and `includeOpenClawTools` gates instead of using `includeCoreTools` for both.
