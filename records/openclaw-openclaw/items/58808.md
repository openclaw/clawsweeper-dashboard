## Summary

Keep this PR open: the sender-context seam is plausible, but the current branch still has a blocking regression because it replaces the family-specific channel/OpenClaw tool gates with `includeCoreTools`, and it still lacks required after-fix real behavior proof.

## What This Changes

The branch adds optional `requesterSenderId` and `senderIsOwner` fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards them from `createOpenClawCodingTools`, while also changing the channel/OpenClaw tool construction gates.

## Best Possible Solution

Forward trusted sender context through `ChannelAgentToolFactory` while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused propagation coverage and SDK docs/changelog follow-through if maintainers accept the contract.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: current `main` uses separate family gates, and existing construction-plan tests encode plugin-only plans where `includeCoreTools` is false but channel tools remain enabled. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as written. The narrow maintainable fix is to add the trusted sender params without changing tool-family allowlist gates, then cover the propagation path if the SDK contract is accepted.
