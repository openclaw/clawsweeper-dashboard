## Summary

Keep open: the requested sender-context plumbing is still useful, but the current PR head regresses the tool-family construction gates and still lacks real behavior proof for the external channel use case.

## What This Changes

The PR widens `ChannelAgentToolFactory`/`listChannelAgentTools` params with trusted sender fields and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Thread the trusted sender fields through `ChannelAgentToolFactory` while preserving the existing `includeChannelTools` and `includeOpenClawTools` gates, then add focused regression coverage and SDK docs/changelog if maintainers want the new fields documented.

## Reproduction Assessment

Yes for the PR regression by source inspection: current main supports `includeCoreTools: false` with `includeChannelTools: true`, while PR head gates channel tools on `includeCoreTools`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not as written. The additive sender-context plumbing is the right narrow direction, but the call site must keep the separate planned channel and OpenClaw tool-family gates.
