## Summary

Keep this PR open: the requested sender-aware channel-agent-tool seam is useful and still absent on current main, with the linked DMWork adapter issue showing a real plugin use case, but the current patch introduces a tool-family gate regression and still lacks real behavior proof.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` params to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards `createOpenClawCodingTools` sender context into channel-owned agent-tool factories.

## Best Possible Solution

Expose sender context through the channel-agent-tool SDK seam while preserving `includeChannelTools` and `includeOpenClawTools` as separate gates, with focused regression coverage and any maintainer-required SDK docs/changelog updates.

## Reproduction Assessment

Yes. By source inspection, `resolveEmbeddedAttemptToolConstructionPlan` can produce `includeCoreTools: false` with `includeChannelTools: true` for `whatsapp_login`, while the PR would gate channel tool construction on `includeCoreTools`.

## Solution Assessment

No, not as written. The sender fields are the right narrow seam, but the implementation should keep `listChannelAgentTools` behind `includeChannelTools` and `createOpenClawTools` behind `includeOpenClawTools`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve the family-specific tool gates:** `src/agents/pi-tools.ts:871-878`
  - body: The patch gates both channel-agent-tool and OpenClaw-tool construction on `includeCoreTools`. Keep these calls behind `includeChannelTools` and `includeOpenClawTools` while forwarding sender context.
