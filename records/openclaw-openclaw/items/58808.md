## Summary

Keep open: the sender-context API extension is useful, but the latest PR head still replaces the distinct channel/OpenClaw tool gates with `includeCoreTools`, leaves CI failing, and has no real channel proof.

## What This Changes

The PR adds optional trusted sender context fields to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards sender data from `createOpenClawCodingTools`.

## Best Possible Solution

Forward the trusted sender fields through the channel factory API while preserving `includeChannelTools` and `includeOpenClawTools`, then add focused regression coverage and contributor-owned real behavior proof before merge.

## Reproduction Assessment

Yes for source-level behavior: current main omits sender context for channel `agentTools`, and the PR regression is visible in the diff plus the failing `check-prod-types` job. No live channel reproduction proof was provided.

## Solution Assessment

No. The API direction is reasonable, but the patch should preserve the existing tool-family gates instead of using `includeCoreTools` for channel-agent-tool and OpenClaw-tool construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve distinct tool-family gates:** `src/agents/pi-tools.ts:913-920`
  - body: The patch replaces both `includeChannelTools` and `includeOpenClawTools` with `includeCoreTools`. Construction plans can set `includeCoreTools: false` while `includeChannelTools: true`, and can set `includeCoreTools: true` while `includeOpenClawTools: false`, so this can drop planned channel tools or materialize OpenClaw tools outside the plan; it also leaves `includeChannelTools` unused and is currently failing `check-prod-types`.
  - confidence: 0.96
