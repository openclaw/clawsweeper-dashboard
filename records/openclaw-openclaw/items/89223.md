## Label Justifications

- `P1`: The report affects a real Windows credential workflow and can block SecretRef-backed config writes for users.
- `impact:security`: The affected path is secret-file ACL validation and may pressure users to store credentials in plaintext or bypass permission checks.

## Summary

Keep open. The config-write `allowInsecurePath` half is already addressed in current main and the latest release, but both still pin `@openclaw/fs-safe@0.3.0`, whose Windows ACL inspector runs `icacls.exe <path> /sid`; the default SecretRef file-provider path still needs a security-reviewed Windows ACL dependency repair and live proof.

## What This Changes

The issue reports that Windows file-backed SecretRefs fail because ACL inspection uses unsupported `icacls /sid` and runtime config-write preflight did not consistently honor `allowInsecurePath`.

## Best Possible Solution

Repair `@openclaw/fs-safe` Windows ACL inspection to use a supported `icacls` invocation or narrow fallback, keep `allowInsecurePath` as an explicit trusted-path bypass, bump OpenClaw to the fixed dependency, and prove both `secrets audit` and `config.patch` on Windows.

## Reproduction Assessment

No live Windows repro was established in this review. Source inspection does show the current pinned dependency invokes `icacls.exe` with `/sid` and fails closed to `unknown` ACL source unless `allowInsecurePath` bypasses the check.

## Solution Assessment

No, closing is not the best way to solve the issue. The current implementation fixes the inconsistent preflight workaround, but the maintainable fix is a security-reviewed dependency repair plus Windows regression proof for the default ACL path.
