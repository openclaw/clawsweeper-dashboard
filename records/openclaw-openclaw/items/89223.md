## Summary

Keep open. Current main and the latest release still pin `@openclaw/fs-safe@0.3.0`, whose Windows ACL inspector invokes `icacls <path> /sid`; current source appears to pass `allowInsecurePath` into both static and runtime SecretRef resolution, so the remaining actionable work is to repair/prove the Windows ACL path and add regression coverage for the runtime preflight path.

## What This Changes

The issue reports that Windows file-backed SecretRefs fail because the ACL inspector uses unsupported `icacls /sid` and runtime config-write preflight does not consistently honor `allowInsecurePath`.

## Best Possible Solution

Preserve fail-closed SecretRef defaults, repair the Windows ACL inspector to use supported command behavior or a safe fallback, and prove `allowInsecurePath` works during active runtime config-write preflight without forcing users back to plaintext credentials.

## Reproduction Assessment

No high-confidence live Windows reproduction was run. Source inspection does show the current and shipped ACL inspector still uses `icacls /sid`, and the runtime write path resolves active SecretRefs during preflight.

## Solution Assessment

The issue direction is partly right: repairing the unsupported `/sid` dependency behavior is the narrow fix. The optional idea to downgrade unknown ACLs to warnings is not the best default without separate security/product approval.
