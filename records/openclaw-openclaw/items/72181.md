## Summary

Keep open. Current main still lacks a BlueBubbles trainer/supervised approval mode: the public config/docs surfaces have no requested fields, and the DM reply delivery path still sends the agent's final text directly through BlueBubbles. The feature is actionable, but owner-gated private-message delivery, correction logging, and AI-send tagging need BlueBubbles maintainer/product/privacy review before automation should implement it.

## What This Changes

Add optional BlueBubbles trainer/supervised modes that draft replies, route them to an owner for approval or correction, and prevent DM sends without owner sign-off.

## Best Possible Solution

Keep this BlueBubbles-owned unless implementation proves a reusable channel approval seam is needed: add schema-backed trainerMode/trainerNotifyNumber/agentTag only after maintainer approval, intercept DM final delivery before sendMessageBlueBubbles, store pending drafts with TTL, parse owner approve/correct commands, deliver only approved supervised replies, log corrections with explicit privacy semantics, and update BlueBubbles docs plus focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path. Source/docs searches show the requested config/API is absent, and the BlueBubbles final delivery path still sends assistant replies directly to the resolved target.

## Solution Assessment

Unclear until BlueBubbles owner/product review. The proposed plugin-local boundary is the right starting point, but command grammar, correction logging, AI tagging, and privacy defaults should be decided before a fix PR is queued.
