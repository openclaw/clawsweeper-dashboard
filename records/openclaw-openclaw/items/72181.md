## Summary

Keep open. Current main still lacks the requested BlueBubbles trainer/supervised owner-approval mode, and the change affects private-message delivery, owner command parsing, correction logging, and AI-origin tagging, so it needs BlueBubbles maintainer/product/privacy review before implementation.

## What This Changes

Request to add optional BlueBubbles `training` and `supervised` modes that draft DM replies and require owner approval or correction before anything is sent to the original sender.

## Best Possible Solution

Keep the work BlueBubbles-owned unless implementation proves a reusable channel approval seam is needed; after maintainer approval, add schema-backed trainer settings, intercept final DM delivery before send, manage pending drafts with TTL, parse owner approve/correct commands, define privacy-aware correction logging, update docs, and cover config plus delivery paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path. Exact source/docs searches show the requested config surface is absent, and the final BlueBubbles delivery path still sends assistant replies directly to the resolved target.

## Solution Assessment

Unclear until BlueBubbles maintainer/product review. The proposed plugin-local boundary is the right starting point, but owner command semantics, correction logging, AI tagging, media/chunk behavior, and private-message safety need explicit decisions before a fix PR is safe to queue.
