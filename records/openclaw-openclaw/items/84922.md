## Summary

Keep open: current main still filters unsupported cron payload kinds out of the loaded cron store, and a later full persist can rewrite `jobs.json` without those rows. The warning path now makes this partly visible, but the durable-job data-loss path remains source-reproducible.

## What This Changes

The issue reports that persisted cron jobs with legacy or unsupported payload kinds are filtered out by gateway cron validation and can be lost on later full writes.

## Best Possible Solution

Make cron persistence lossless for unsupported persisted rows while keeping runtime execution limited to documented payload kinds, with regression coverage for `command` and `agentmessage` rows surviving a full persist.

## Reproduction Assessment

Yes. Seed `jobs.json` with an unsupported `payload.kind` plus any valid job path that triggers full persistence; source inspection shows the unsupported row is filtered out before `saveCronStore` serializes the store.

## Solution Assessment

Yes for the lossless-persistence boundary: preserve raw unsupported rows and warn, while leaving execution and direct command support to the existing narrow cron payload contract or separate trackers.
