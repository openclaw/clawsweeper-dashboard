## Summary

Keep open: current main still rejects unsupported persisted cron `payload.kind` values, drops those rows from the loaded store, and can later rewrite `jobs.json` from that filtered in-memory state. The issue is already paired with an open closing PR, so the right next step is review and validation of that fix path rather than closing this issue early.

## What This Changes

The issue reports that gateway cron persistence can drop jobs whose stored payload kind is not `systemEvent` or `agentTurn`.

## Best Possible Solution

Review and land or replace the linked preservation fix so unsupported persisted cron rows survive reload and full persist, with regression coverage; handle runnable legacy-kind migration separately if maintainers want it.

## Reproduction Assessment

Yes. Seed `jobs.json` with an unsupported string `payload.kind`; current main marks it invalid, omits it from `state.store.jobs`, and a later full persist can serialize only the filtered list.

## Solution Assessment

Yes for the lossless-persistence part: preserve unsupported stored rows and warn without broadening runtime execution semantics. A separate doctor migration should define any conversion from `command` or `agentmessage` to canonical runnable payloads.
