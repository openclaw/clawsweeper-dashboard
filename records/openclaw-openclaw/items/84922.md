## Label Justifications

- `P0`: The issue reports durable cron job definitions being deleted from `jobs.json`, which is configuration data loss.
- `impact:data-loss`: The affected data is persisted cron job configuration, not only a failed or skipped cron run.

## Summary

Keep open: current main and the latest release still reject persisted cron jobs whose `payload.kind` is not `systemEvent` or `agentTurn`, then omit those rows from the loaded store before later full persistence can rewrite `jobs.json` without them. The linked fix PR is open and uses closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Issue reports that gateway cron persistence can silently lose jobs with legacy or unknown `payload.kind` values such as `command` and `agentmessage`.

## Best Possible Solution

Land a lossless cron-store persistence fix, preferably through https://github.com/openclaw/openclaw/pull/84971 or a replacement, with regression coverage for unsupported string payload kinds surviving load and full persist while runtime execution remains limited to canonical kinds.

## Reproduction Assessment

Yes. Seed `jobs.json` with a valid cron job whose payload has an unsupported string kind, then trigger a force reload and a full persist; source inspection shows the row is filtered from `state.store.jobs` before serialization.

## Solution Assessment

Yes for the lossless-persistence part: preserve unsupported persisted rows and warn without broadening runtime execution. Migration or execution support for specific legacy kinds should be a separate maintainer-approved follow-up if desired.
