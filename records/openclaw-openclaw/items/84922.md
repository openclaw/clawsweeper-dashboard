## Label Justifications

- `P0`: The issue reports persisted cron job definitions being deleted, and current source inspection shows a plausible config-data loss path.
- `impact:data-loss`: The affected data is durable cron job configuration in `jobs.json`, not only a transient failed run.

## Summary

Keep open: current main still rejects cron jobs whose persisted payload kind is not `systemEvent` or `agentTurn`, filters them out of the loaded store, and later full persistence paths can rewrite `jobs.json` without those rows. The warning path is present now, so the “silent” part is partly mitigated, but the reported data-loss risk remains.

## What This Changes

The issue asks for cron-store validation to preserve jobs with unsupported or legacy payload kinds instead of dropping them during gateway persistence.

## Best Possible Solution

Make routine cron-store writes lossless for unsupported persisted rows while keeping runtime execution limited to documented payload kinds, and add regression coverage for `command`/`agentmessage` rows surviving a full persist.

## Reproduction Assessment

Yes. Seed `jobs.json` with an unsupported `payload.kind` plus any valid job that causes a full persist; current source rejects the unsupported row, omits it from `state.store.jobs`, and serializes only the filtered store on full write.

## Solution Assessment

Yes for the lossless-persistence part: preserve raw unsupported rows and warn, while leaving execution and doctor migration policy narrow. It would not be best to broaden runtime cron payload execution to arbitrary unknown kinds as part of this bug fix.
