## Label Justifications

- `P1`: The issue describes a current cron persistence path that can silently remove scheduled jobs for real users.
- `impact:data-loss`: The affected data is the user's persisted cron job definitions in `jobs.json`.

## Summary

This should stay open: current `main` still has a source-level path where unsupported cron payload kinds are filtered out of the loaded in-memory store and later persistence can rewrite `jobs.json` without those jobs. The report is credible and narrow enough for a focused repair PR.

## What This Changes

The issue reports that cron jobs with legacy or unsupported `payload.kind` values such as `command` or `agentmessage` can be silently removed from the persisted cron store.

## Best Possible Solution

Keep runtime execution strict, but preserve unsupported raw cron records on disk until `openclaw doctor --fix` or an explicit user action migrates or removes them; add regression coverage for force-reload plus persist.

## Reproduction Assessment

Yes, at source level. A persisted job with `payload.kind: "command"` or `"agentmessage"` reaches `getInvalidPersistedCronJobReason`, is skipped by `ensureLoaded`, and the timer has later `persist(state)` paths that write the filtered in-memory job list.

## Solution Assessment

Yes, the best fix is a narrow persistence-safety change rather than broad support for arbitrary payload kinds. Runtime can still skip unsupported jobs for execution, but runtime maintenance should not destructively rewrite them out of the user's store.

Current `main` filters persisted cron jobs whose `payload.kind` is not `systemEvent` or `agentTurn` during `ensureLoaded`, and later timer persistence can rewrite `jobs.json` without those records.
