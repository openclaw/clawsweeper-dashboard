## Label Justifications

- `P0`: The issue describes persisted cron job definitions being removed from durable scheduler configuration, which is configuration data loss.
- `impact:data-loss`: The affected artifact is durable cron job data in `jobs.json`, not only a transient failed cron run.

## Summary

Keep open: current main still rejects unsupported cron `payload.kind` values, skips those persisted rows during load, and later full persistence paths can serialize only the filtered in-memory store. The warning path partly mitigates the “silent” part, but the data-loss bug remains, and the linked fix at https://github.com/openclaw/openclaw/pull/84971 is still open.

## What This Changes

The issue reports that gateway cron-store validation drops persisted cron jobs whose `payload.kind` is not `systemEvent` or `agentTurn`, including legacy `command` and `agentmessage` rows.

## Best Possible Solution

Make routine cron-store writes lossless for unsupported persisted rows, keep execution limited to canonical payload kinds, and add regression coverage for legacy-kind rows surviving a full load and persist cycle.

## Reproduction Assessment

Yes. Seed `jobs.json` with an unsupported string `payload.kind`; current source marks it `invalid-payload`, omits it from `state.store.jobs`, and full persistence can write back only the filtered list.

## Solution Assessment

Yes for the lossless-persistence part: preserving unsupported persisted rows and warning is the narrow maintainable fix. It would not be best to broaden runtime execution to arbitrary unknown payload kinds inside this bug fix.
