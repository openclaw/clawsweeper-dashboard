## Label Justifications

- `P0`: The issue reports durable cron job definitions being silently removed from persisted configuration, and current source inspection supports that data-loss path.
- `impact:data-loss`: The affected data is stored cron job configuration in `jobs.json`, not only a transient failed run or skipped message.

## Summary

Keep open: current main still rejects unsupported cron payload kinds during persisted-store validation, omits those jobs from loaded state, and full persist paths can rewrite `jobs.json` without them; https://github.com/openclaw/openclaw/pull/84971 is an open implementation path but has not merged.

## What This Changes

The issue asks for cron persisted-store validation to preserve unsupported string `payload.kind` rows instead of dropping them during reload and persist cycles.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84971 or an equivalent lossless persistence fix with store-cycle regression coverage, while keeping legacy-kind execution/migration as a separate explicit decision.

## Reproduction Assessment

Yes, at source level: seed `jobs.json` with an unsupported string `payload.kind`, then current main marks it invalid, omits it from `state.store.jobs`, and a later full persist can serialize only the filtered list. I did not run a live write-cycle test because this review is read-only.

## Solution Assessment

Yes for the lossless-persistence fix: preserve unsupported persisted rows through normal reload/write paths and keep warning or doctor migration policy separate. It would not be the best bug fix to broaden runtime cron execution to arbitrary unknown payload kinds in the same change.
