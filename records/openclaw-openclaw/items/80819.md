## Summary

Keep open: current main and the latest shipped tag still start Claude live sessions with stdio permission prompts but have no `control_request` / `control_response` handling path, so the reported stall remains a valid source-proven bug.

## What This Changes

The issue asks OpenClaw's Claude CLI live-session bridge to answer Claude `can_use_tool` control requests using the effective OpenClaw exec policy instead of stalling until timeout.

## Best Possible Solution

Add a tested handler in the Claude live-session backend that responds to `can_use_tool` control requests from the session-start exec policy, denies native tools by default with MCP guidance, and only permits native Bash when OpenClaw's effective policy is fully permissive.

## Reproduction Assessment

Yes, from source: a Claude live session that emits a non-result `control_request` will pass through `handleClaudeLiveLine` and return without any matching stdin response. I did not run a live Claude session because this environment lacks the `claude` binary and auth material.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if the allow/deny mapping stays tied to the existing OpenClaw exec policy. The implementation should live in `src/agents/cli-runner/claude-live-session.ts` with regression coverage for stdin `control_response` writes and timeout avoidance.
