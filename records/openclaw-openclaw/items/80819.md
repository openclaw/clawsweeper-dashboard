## Summary

Keep open: current main still starts Claude live sessions in stdio permission-prompt mode but has no `control_request` handling or `control_response` write path, so the reported stall remains source-proven on main and on the latest shipped tag.

## What This Changes

The issue asks OpenClaw's Claude CLI live-session bridge to answer Claude `can_use_tool` control requests with policy-based allow or deny responses instead of waiting for the global timeout.

## Best Possible Solution

Add focused Claude live-session handling that writes tested `control_response` frames for `can_use_tool`, denies native tools by default with MCP guidance, and only allows native Bash when the resolved OpenClaw exec policy is fully permissive.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a Claude stream-json `control_request` reaches `handleClaudeLiveLine`, is appended to the turn, and then returns because it is not a `result`. I did not run a live Claude CLI session because this checkout does not include the Claude binary or auth material.

## Solution Assessment

Mostly yes: handling `can_use_tool` inside `src/agents/cli-runner/claude-live-session.ts` is the narrow fix. A maintainer should confirm the native Bash allow/deny boundary before implementation because this touches Claude's native permission surface.
