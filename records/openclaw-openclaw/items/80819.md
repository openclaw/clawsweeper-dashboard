## Summary

Keep open: current main and v2026.5.7 still start Claude live sessions with stdio permission prompts but drop non-result JSONL records, so the reported stall remains source-proven. An open fix PR already references this issue with closing syntax, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests policy-aware `control_response` handling for Claude CLI live-session `control_request` / `can_use_tool` frames so sessions do not stall until timeout.

## Best Possible Solution

Review and land a tested Claude live-session `control_request` handler, coordinating native tool policy with the broader Claude CLI exec-policy bridge rather than closing this issue before the linked PR merges.

## Reproduction Assessment

Yes, at source level. Current main forces Claude stream-json stdio permission prompts and then returns for every non-`result` JSONL record, so a Claude `control_request` has no `control_response` path; I did not run a live Claude CLI session.

## Solution Assessment

Mostly yes. The narrow fix belongs in `src/agents/cli-runner/claude-live-session.ts`, but maintainers should decide whether the first shipped behavior is deny-only native tools or an exec-policy-mapped native Bash allow path.
