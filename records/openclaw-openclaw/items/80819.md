## Summary

Keep open: the reported Claude live-session stall remains source-proven on current main and latest release, and the open fixing PR should be reviewed or superseded before this issue is closed.

## What This Changes

The issue asks OpenClaw to answer Claude CLI stream-json `control_request` / `can_use_tool` prompts with `control_response` frames instead of stalling until timeout.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/80973 with a tested Claude live-session handler that answers `can_use_tool` prompts and preserves OpenClaw's MCP-first permission boundary.

## Reproduction Assessment

Yes, at source level: during an active Claude live turn, a stream-json `control_request` reaches `handleClaudeLiveLine`, is stored, and returns at the non-`result` branch without writing to stdin. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Mostly yes: handling `control_request` in `src/agents/cli-runner/claude-live-session.ts` is the narrow repair. Maintainers should confirm the native Bash policy boundary before landing the linked PR because that affects command-execution permissions.
