## Summary

Keep open: current main still runs Claude live sessions with stdio permission prompts but drops non-result JSONL frames, so `control_request` frames still have no `control_response` path. The issue also has an active closing PR, so it should stay open until that PR, or an equivalent fix, lands.

## What This Changes

The issue asks OpenClaw to answer Claude CLI live-session `control_request` permission frames instead of letting sessions stall until timeout.

## Best Possible Solution

Land a narrow Claude live-session bridge fix that writes SDK-shaped `control_response` frames for `can_use_tool`, adds regression coverage, and records the intended native-tool policy boundary.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main starts Claude with stdio permission prompting, then `handleClaudeLiveLine` returns for non-`result` frames before any `control_response` can be written.

## Solution Assessment

Mostly yes. Handling the frame in `src/agents/cli-runner/claude-live-session.ts` is the narrow maintainable fix, but maintainers should decide whether to deny native tools only or allow native Bash when OpenClaw exec policy is fully permissive.
