## Summary

Keep open: current main and the latest release still ignore Claude `control_request` frames, and an open PR with closing syntax is now the active fix path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Respond to Claude CLI live-session `control_request` / `can_use_tool` events with `control_response` decisions instead of letting sessions hang until the no-output timeout.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80973 with a narrow Claude live-session `control_response` handler, regression coverage, real behavior proof, and an explicit native-tool policy decision.

## Reproduction Assessment

Yes. Source inspection on current main shows Claude live sessions start with stdio permission prompts and then drop every non-`result` JSONL frame, so a `control_request` has no matching `control_response` path.

## Solution Assessment

Mostly yes. Handling this in `src/agents/cli-runner/claude-live-session.ts` is the narrow maintainable fix, but the linked PR still needs real behavior proof and maintainer confirmation of the native Bash allow/deny boundary.
