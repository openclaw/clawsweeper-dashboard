## Summary

Keep open: current main still starts Claude live sessions with stdio permission prompts but has no handler that writes `control_response` frames, so a Claude native tool approval can still stall until the no-output watchdog fires.

## What This Changes

The issue asks OpenClaw to handle Claude CLI stream-json `control_request` permission prompts in live sessions and map `can_use_tool` decisions to OpenClaw exec policy.

## Best Possible Solution

Add a focused Claude live-session permission bridge with regression coverage: respond to `can_use_tool`, deny native tools by default with OpenClaw MCP guidance, and allow native Bash only under the maintainer-approved fully permissive exec policy.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a Claude `control_request` record reaches `handleClaudeLiveLine` and returns as non-`result` without writing anything to stdin. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Mostly yes: handling `can_use_tool` inside `src/agents/cli-runner/claude-live-session.ts` is the narrow technical fix. The native Bash allow/deny boundary should be confirmed by maintainers because it affects command-execution permissions.
