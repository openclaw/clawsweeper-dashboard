## Summary

Keep open: current main and the latest v2026.5.7 release still start Claude live sessions with stdio permission prompts and then drop non-result JSONL records, and an open closing PR already owns the fix path.

## What This Changes

The issue asks OpenClaw's Claude CLI live-session bridge to answer `can_use_tool` `control_request` frames with a policy-aware `control_response` instead of hanging until timeout.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/80973; the accepted fix should write Claude `control_response` frames, cover the stdio seam with tests, and make the native-tool permission policy explicit.

## Reproduction Assessment

Yes. Source inspection shows a Claude stream-json `control_request` reaches `handleClaudeLiveLine` and returns as a non-`result` record without any stdin `control_response` path; I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Mostly yes. Handling `control_request` inside `src/agents/cli-runner/claude-live-session.ts` is the narrow fix location, but maintainers should settle the native Bash allow/deny boundary before landing the linked PR.
