## Summary

Keep open: current main and the latest shipped release still lack a Claude live-session `control_request` response path, and an open fix PR already uses closing syntax for this issue, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue asks OpenClaw's Claude CLI live-session bridge to answer Claude `control_request`/`can_use_tool` permission prompts with `control_response` frames instead of stalling until timeout.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/80973; the final fix should add tested Claude live `control_request` handling and settle the native-tool permission policy explicitly.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: current main starts Claude with stdio permission prompts and then returns on every non-`result` JSONL record, so a `control_request` has no `control_response` write path. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Mostly yes: the narrow maintainable fix belongs in `src/agents/cli-runner/claude-live-session.ts` with regression coverage. Maintainers still need to confirm whether native Bash should ever be allowed under fully permissive exec policy or whether denial-with-guidance is the intended scope.
