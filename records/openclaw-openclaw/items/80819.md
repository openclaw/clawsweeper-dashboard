## Summary

Keep open: current main still enables Claude stdio permission prompts for live sessions, but the stream-json line handler still has no `control_request` or `can_use_tool` branch and no `control_response` write path, so the reported stall remains source-proven and is not already fixed by the latest release.

## What This Changes

The issue asks OpenClaw's Claude CLI live-session bridge to answer Claude `control_request` permission prompts, especially `can_use_tool`, instead of letting sessions stall until timeout.

## Best Possible Solution

Add focused Claude live-session permission-prompt handling that writes tested `control_response` frames, denies native tools by default with MCP guidance, and only allows native Bash when OpenClaw's resolved exec policy is fully permissive.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: a Claude stream-json `control_request` reaches `handleClaudeLiveLine`, is stored, and then returns because it is not a `result`. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Mostly yes: emitting `control_response` frames from `src/agents/cli-runner/claude-live-session.ts` is the narrow technical fix. Maintainer policy confirmation is still needed for native Bash allowance because this touches Claude's native permission surface.
