## What This Changes

The PR adds a Discord plugin shared on-disk instance-claim registry, periodic provider refresh, message/native command ownership gates, and focused ownership tests.

## Best Possible Solution

Keep the fix in the Discord plugin, but derive claims from effective allowed ownership for the active bot and resolved channel/thread/guild identity, use lossless shared storage, then add focused regression coverage and redacted two-instance Discord proof.

## Solution Assessment

No. The Discord plugin is the right layer, but this patch is not the best mergeable solution until effective allowlist semantics, bot scoping, resolved channel identity, shared-write safety, and real proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build claims from effective channel access:** `extensions/discord/src/monitor/instance-claims.ts:197-218`
  - body: `buildClaimEntries` records raw numeric channel keys and a narrow guild-wide case before running the same channel policy resolver used by ingress. Disabled or denied channels can be claimed and blackholed by peers, while name/slug or wildcard-with-overrides allowed channels are not claimed and still hit the old local deny path.
  - confidence: 0.9
- **[P2] Pass the active bot id to native claim checks:** `extensions/discord/src/monitor/native-command.ts:462-469`
  - body: The new native autocomplete and dispatch lookups call `resolveDiscordClaimOwnership` without `botId`, so the resolver scans every bot bucket in the shared claims file. On a multi-bot host, a different bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:778`
  - body: Preflight already normalizes payload fallbacks into `messageChannelId`, but the new claim lookup uses `message.channelId` directly. Payloads that rely on `channel_id` or raw-data fallbacks can miss ownership claims and fall through to the old local allowlist path.
  - confidence: 0.87
- **[P2] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:237-247`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, causing temporary `no-entry` results and incorrect routing decisions.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes Discord access-control routing and introduces concrete authorization/availability risks that should be fixed before merge.

Concerns:

- **[medium] Claims can blackhole denied channels:** `extensions/discord/src/monitor/instance-claims.ts:214`
  - body: The registry claims raw numeric channel keys without proving the instance will actually process that channel, so peers can skip a channel that the claiming instance later rejects.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:237`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
