## Summary

Keep open: the linked Discord multi-instance problem is still source-reproducible on current main, but the PR is not merge-safe because the ownership registry can over-claim channels, miss valid allowlist forms, lose peer claims, and suppress the wrong bot.

## What This Changes

The PR adds a Discord instance-claim registry, refreshes it from provider startup, and skips Discord message/native-command handling when another instance appears to own the guild or channel.

## Best Possible Solution

Keep the fix in the Discord plugin, but publish effective bot-scoped ownership claims from the resolved allowed channel set and store them without lost peer writes.

## Reproduction Assessment

Yes. The linked two-instance same-token setup is concrete, and current main still has the local native-command deny path with no shared ownership registry; I did not live-run Discord in this read-only review.

## Solution Assessment

No. The Discord-plugin ownership gate is the right layer, but this implementation needs allowlist-aware claim publication, bot-scoped lookups, resolved channel IDs, and serialized or per-instance storage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Claim only channels this instance can process:** `extensions/discord/src/monitor/instance-claims.ts:213-221`
  - body: buildClaimEntries publishes every numeric channel key without checking whether that channel entry is disabled or otherwise denied. Because peers skip claimed channels before their own allowlist checks, one instance can claim a channel it later drops and blackhole traffic for sibling instances.
  - confidence: 0.9
- **[P2] Pass the active bot id into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:466-472`
  - body: The native autocomplete and dispatch claim checks omit botId, so resolveDiscordClaimOwnership can scan claims for unrelated bots in the shared file. Multi-bot or multi-account deployments can therefore silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: preflightDiscordMessage already resolves messageChannelId from the message and event fallback, but the new claim lookup uses message.channelId. Payloads that only carry the channel id through the event wrapper will miss ownership claims and fall back to the local deny path this PR is meant to avoid.
  - confidence: 0.86
- **[P2] Serialize shared claims updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: refreshDiscordClaims reads, modifies, and rewrites one shared JSON file without a cross-process lock or compare-and-swap. Two instances refreshing together can overwrite each other's fresh claims, temporarily returning no-entry and routing authorization decisions incorrectly.
  - confidence: 0.84
- **[P2] Include named and wildcard channel claims:** `extensions/discord/src/monitor/instance-claims.ts:203-217`
  - body: Discord channel allowlists can match names, slugs, and wildcard fallback entries, but the registry discards nonnumeric keys and only marks a guild-wide claim for an empty or lone wildcard map. Name-configured and wildcard-plus-override channels can remain unclaimed, so siblings still take the old deny path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes Discord authorization routing and introduces concrete availability/access-control risks before merge.

Concerns:

- **[medium] Claims can blackhole disabled channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The registry claims raw numeric channel keys before checking whether the claiming instance will process that channel, so sibling instances can skip traffic that the owner later denies or disables.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command claim lookups omit the active bot id, allowing another Discord bot's claim in the same shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
