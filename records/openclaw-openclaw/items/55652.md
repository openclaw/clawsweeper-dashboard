## Summary

Keep open. The PR addresses a valid Discord multi-instance routing bug, but the current patch still has merge-blocking correctness and security issues in ownership claim publication, bot scoping, channel identity resolution, and shared-file coordination.

## What This Changes

The PR adds a Discord instance-claim registry, refreshes claims from provider startup, and gates Discord message preflight, autocomplete, and native slash-command dispatch on claim ownership.

## Best Possible Solution

Keep the fix in the Discord plugin, but derive ownership from effective Discord channel config for the active bot and resolved channel/thread/guild identity, then update claims through per-instance files or serialized writes before suppressing local allowlist denials.

## Reproduction Assessment

Yes. The linked report gives a concrete two-instance setup sharing one Discord bot token, and current main still has the native-command local deny path with no sibling ownership check.

## Solution Assessment

No. A Discord-plugin-owned ownership gate is the right direction, but this implementation is not safe until claim publication is allowlist-aware, bot-scoped, name/slug-aware, and robust against concurrent writers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Claim only channels this instance will process:** `extensions/discord/src/monitor/instance-claims.ts:195-200`
  - body: `buildClaimEntries` publishes every numeric channel key as owned without checking whether the entry is disabled or otherwise denied. Because peers skip claimed channels before their own allowlist checks, one instance can claim a channel it later rejects and blackhole traffic for all siblings.
  - confidence: 0.9
- **[P2] Pass the active bot id into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:466-472`
  - body: Native autocomplete and command dispatch call `resolveDiscordClaimOwnership` without `botId`, so the resolver can scan claims for any bot in the shared file. In multi-bot deployments, another bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:779-782`
  - body: Preflight already resolves `messageChannelId` from the message and event fallback, but the new claim check uses `message.channelId`. Payloads that only carry the channel id through the event wrapper will miss ownership claims and fall back to the local deny path this PR is meant to avoid.
  - confidence: 0.86
- **[P2] Serialize shared claims file updates:** `extensions/discord/src/monitor/instance-claims.ts:219-228`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, temporarily returning `no-entry` and letting the wrong instance handle or deny the channel.
  - confidence: 0.84
- **[P2] Preserve name-based Discord channel allowlists:** `extensions/discord/src/monitor/instance-claims.ts:195-199`
  - body: Discord channel config can match by channel name or slug, but this registry discards every nonnumeric key and the lookup API has no channel-name input. Name-configured channels therefore never get claimed, so sibling instances can still take the old incorrect deny path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes Discord authorization routing and introduces concrete availability/access-control concerns that need maintainer attention before merge.

Concerns:

- **[medium] Claims can blackhole denied channels:** `extensions/discord/src/monitor/instance-claims.ts:200`
  - body: The registry claims raw numeric channel keys without checking whether the owner will actually process that channel. Since peers skip claimed channels before local allowlist checks, denied or disabled claims can make all instances drop the channel.
  - confidence: 0.9
- **[medium] Native ownership checks are not isolated per bot:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:219`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
