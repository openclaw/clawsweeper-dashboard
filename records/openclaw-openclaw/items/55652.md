## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Serialize shared claim refreshes:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` loads the whole shared file, updates this instance, then rewrites it. Two OpenClaw instances refreshing at startup or on the 30s timer can both read the same old file and the later rename drops the other instance's fresh claims, so peer lookups fall back to `no-entry` and the wrong instance can handle or deny the channel again. Use a per-instance file, lock, or CAS-style write before making this a routing input.
  - confidence: 0.93
- **[P1] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:775-783`
  - body: The preflight path already resolves `messageChannelId` from the message and event payload, then uses that value for channel config. The new claim lookup passes `message.channelId` instead, so payloads where only `data.channel_id` is populated query an empty or stale id, return `no-entry`, and can still run the local deny path this PR is meant to avoid.
  - confidence: 0.9
- **[P1] Do not claim disabled channels:** `extensions/discord/src/monitor/instance-claims.ts:213-221`
  - body: The claim builder records every numeric channel key as owned without checking whether the channel entry is actually enabled. Because the new claim gate runs before the existing local channel checks, a sibling instance will skip a disabled channel as `claimed-by-other` while the claiming instance later drops it, blackholing traffic that another instance might otherwise handle.
  - confidence: 0.86
- **[P1] Preserve name-keyed channel allowlists:** `extensions/discord/src/monitor/instance-claims.ts:213-217`
  - body: Discord channel config can match by channel id, name, slug, parent, or wildcard, and the docs show name keys like `general` and `help`. This claim builder discards every non-numeric channel key, so name-keyed deployments never publish ownership claims and sibling instances keep falling through to their local allowlist denial path.
  - confidence: 0.88
- **[P2] Scope native command claims to the active bot:** `extensions/discord/src/monitor/native-command.ts:859-866`
  - body: Both native command claim lookups omit `botId`, so `resolveDiscordClaimOwnership` searches every bot bucket in the shared claims file. On hosts running multiple Discord accounts or bots, a channel claimed for another bot can make this bot's autocomplete or command dispatch return `claimed-by-other` and silently drop a valid interaction.
  - confidence: 0.84
