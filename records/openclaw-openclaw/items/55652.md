## Summary

Keep open: current main still lacks a Discord instance-ownership gate for the linked two-instance bug, but the PR still has blocking authorization-routing and shared-state defects.

## What This Changes

The PR adds a shared Discord instance-claim registry, refreshes it from provider startup, and suppresses Discord message/native-command handling when another instance claims the guild or channel.

## Best Possible Solution

Keep the fix in the Discord plugin, but derive claims from effective allowed channel config for the active bot and resolved channel/thread/guild identity, then publish claims through lossless serialized or per-instance storage with focused regression coverage.

## Reproduction Assessment

Yes. The linked report gives a concrete two-instance setup sharing one Discord bot token, and current main still has native-command local allowlist denial with no sibling ownership registry.

## Solution Assessment

No. The Discord-plugin ownership gate is the right layer, but this patch is not merge-safe until claim publication is allowlist-aware, bot-scoped, based on resolved channel IDs, and robust against lost peer claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Claim only channels this instance will process:** `extensions/discord/src/monitor/instance-claims.ts:219-227`
  - body: `buildClaimEntries` publishes every numeric channel key without checking whether that channel entry is disabled or denied. Because peers skip claimed channels before their own allowlist checks, one instance can claim a channel it later rejects and blackhole traffic for all sibling instances.
  - confidence: 0.9
- **[P2] Pass the active bot id into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:462-469`
  - body: The native autocomplete and dispatch claim checks omit `botId`, so `resolveDiscordClaimOwnership` can scan claims for any bot in the shared file. In multi-bot or multi-account deployments, another bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:778`
  - body: Preflight already resolves `messageChannelId` from the message and event fallback, but the new claim lookup uses `message.channelId`. Payloads that only carry the channel id through the event wrapper will miss ownership claims and fall back to the local deny path this PR is meant to avoid.
  - confidence: 0.86
- **[P2] Serialize shared claims file updates:** `extensions/discord/src/monitor/instance-claims.ts:244-253`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, temporarily returning `no-entry` and letting the wrong instance handle or deny the channel.
  - confidence: 0.84
- **[P2] Preserve name-based Discord channel allowlists:** `extensions/discord/src/monitor/instance-claims.ts:219-223`
  - body: Discord channel config can match by channel name or slug, but this registry discards every nonnumeric key and the lookup API has no channel-name input. Name-configured channels therefore never get claimed, so sibling instances can still take the old incorrect deny path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch changes Discord authorization routing and introduces concrete availability/access-control risks that should be fixed before merge.

Concerns:

- **[medium] Claims can blackhole denied channels:** `extensions/discord/src/monitor/instance-claims.ts:224`
  - body: The registry claims raw numeric channel keys without checking whether the owner will actually process that channel, so peers can skip a channel that the claiming instance later denies or disables.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:462`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:244`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
