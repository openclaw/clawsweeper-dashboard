## Summary

Keep this PR open. Current main still lacks a Discord cross-instance ownership registry, so the reported multi-instance slash-command denial remains real, but the proposed patch is not ready because it introduces authorization-routing edge cases around denied channels, bot scoping, resolved channel IDs, and shared-file writes.

## What This Changes

The PR adds a Discord instance-claims file under `extensions/discord/src/monitor`, refreshes claims from provider startup, and consults those claims in Discord message preflight, autocomplete, and native slash-command dispatch.

## Best Possible Solution

Keep the Discord-plugin-owned fix, but make the ownership registry bot-scoped, claim only channels this instance will actually process, use the resolved channel/thread identity consistently, and make shared claim updates robust against concurrent OpenClaw instances before local allowlist denial is bypassed.

## Reproduction Assessment

Yes. The linked report gives a concrete two-instance setup sharing one Discord bot token, and current main still has the local native-command deny path that can reply `This channel is not allowed.` before any sibling ownership check.

## Solution Assessment

No. A Discord-plugin-owned ownership gate is the right area, but this PR's current implementation is not the best fix until it closes the bot-scoping, denied-channel, channel-id fallback, and multi-writer registry gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Filter denied channels before publishing claims:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: `buildClaimEntries` publishes every numeric channel key as owned, including entries configured with `allow: false` or `enabled: false`. Because the new ownership gates run before local allowlist checks, sibling instances will skip these channels while the claiming instance later rejects them, blackholing otherwise valid traffic.
  - confidence: 0.9
- **[P2] Pass the active bot ID into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:860-866`
  - body: Native command dispatch and autocomplete call `resolveDiscordClaimOwnership` without a `botId`, so the resolver can match claims for any bot stored in the shared file. In multi-bot or multi-account deployments, another bot's claim can make this bot silently drop valid interactions; pass the interaction/application bot id just like message preflight does.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel ID for claim checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: Preflight already computes `messageChannelId` from the message and event fallback, but the new claim lookup uses `message.channelId`. When Discord payloads only provide the channel through the event wrapper, the claim lookup sees no channel entry and falls through to the local deny path this PR is trying to avoid.
  - confidence: 0.86
- **[P2] Serialize shared claims file updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` still performs an unlocked read-modify-write of one shared claims file. Two instances refreshing at the same time can overwrite each other's fresh claims, briefly returning `no-entry` and letting both instances handle or deny the same channel; use an advisory lock/CAS or per-instance claim files.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes Discord authorization routing and introduces concrete availability/access-control concerns that need maintainer attention before merge.

Concerns:

- **[medium] Claims can blackhole denied or disabled channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The new registry claims numeric channel keys without checking whether this instance will actually process those channels. Since peers skip claimed channels before their own allowlist checks, a denied or disabled claim can make all instances drop the channel.
  - confidence: 0.9
- **[medium] Native claims are not isolated per bot:** `extensions/discord/src/monitor/native-command.ts:860`
  - body: Native command claim lookups omit `botId`, allowing claims for another Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The single claims file is updated with an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and temporarily route authorization decisions incorrectly.
  - confidence: 0.84
