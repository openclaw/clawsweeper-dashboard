Overall confidence: 0.91

Full review comments:

- **[P1] Port claim checks into the current native auth helpers:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: Current main moved native autocomplete and auth decisions into `native-command-auth.ts`, but this branch adds the claim lookup to an older in-file copy in `native-command.ts`. Rebase and port the gate to the current helper structure so the fix does not bypass or revert the recent native-command refactor.
  - confidence: 0.87
- **[P1] Build claims only from effectively allowed channels:** `extensions/discord/src/monitor/instance-claims.ts:213-218`
  - body: `buildClaimEntries` records every numeric channel key and ignores name/slug keys, without checking `allow: false` or `enabled: false`. A sibling can skip a channel that this instance later denies, while name-based allowlists still fail to claim ownership; derive claims from the same effective allowlist semantics used by runtime routing.
  - confidence: 0.9
- **[P1] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` reads the shared JSON file, mutates one bucket, and writes the whole file back without locking or compare-and-swap. Two instances refreshing together can overwrite each other's fresh claims, causing `no-entry` fallthrough and duplicate/wrong-instance handling.
  - confidence: 0.88
- **[P2] Pass the active bot id to native claim lookups:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete/dispatch claim checks omit `botId`, so `resolveDiscordClaimOwnership` scans every bot bucket in the shared claims file. In multi-bot or multi-account setups, a different bot's claim for the same channel can suppress this bot's valid interaction.
  - confidence: 0.86
- **[P2] Use the resolved message channel id for preflight claims:** `extensions/discord/src/monitor/message-handler.preflight.ts:775-782`
  - body: Preflight already resolves `messageChannelId` from the Discord payload, but the new claim check queries `message.channelId`. Payloads that only populate the resolved fallback will miss peer claims and continue to the local deny path this PR is trying to avoid.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the diff changes Discord channel authorization/routing before existing allowlist checks and uses a shared local claim file whose semantics are not yet safe enough.

Concerns:

- **[medium] Claim registry can suppress authorized Discord traffic:** `extensions/discord/src/monitor/instance-claims.ts:213`
  - body: The gate trusts shared claim state before local allowlist checks, while the builder can claim disabled/denied channels and the native lookup can match other bots. That can turn channel allowlist state into silent message loss or wrong-instance routing.

  - body: The registry is updated through whole-file read-modify-write without cross-process coordination, so a sibling instance can accidentally erase fresh ownership state and weaken the intended routing boundary.
  - confidence: 0.86
