## Summary

Keep open: current main still lacks a Discord ownership gate for this multi-instance denial path, but this PR is not merge-safe because it changes Discord access-control routing with over/under-broad claims, native bot-scope gaps, and an unlocked shared-file update path, and it still lacks inspectable real behavior proof.

## What This Changes

The PR adds a Discord shared on-disk instance-claim registry, provider refresh, guild message and native command ownership gates, and focused ownership tests under the Discord plugin.

## Best Possible Solution

Keep the fix in the Discord plugin, but publish only effective allowed ownership for the active bot and resolved channel/thread/guild identity through lossless shared storage, with focused regression coverage and redacted two-instance Discord proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still has no shared Discord ownership registry and native slash commands can still hit the local `This channel is not allowed.` path. I did not run a live two-instance Discord setup here.

## Solution Assessment

No. The Discord plugin is the right layer, but this patch is not the best mergeable solution until claims are derived from effective allowlist semantics, scoped to the active bot, based on resolved channel identity, and written without losing peer state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build claims from the effective channel allowlist:** `extensions/discord/src/monitor/instance-claims.ts:203-218`
  - body: `buildClaimEntries` treats the raw channel map as ownership: it claims every numeric key, drops name/slug keys, and does not model wildcard entries with overrides. Because peers skip before running their own channel checks, disabled or denied channels can be blackholed while name-based or wildcard-allowed channels still miss ownership suppression.
  - confidence: 0.9
- **[P2] Pass the active bot id to native claim checks:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete and dispatch claim lookups omit `botId`, so the resolver can scan claims for every bot in the shared file. On multi-bot hosts, another bot's claim can silently suppress valid interactions for this bot; pass the interaction/application bot id like preflight does.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: Preflight already normalizes payload fallbacks into `messageChannelId`, but the new claim lookup uses `message.channelId` directly. Payloads that rely on `channel_id` or raw-data fallbacks can miss ownership claims and fall back to the old local deny path.
  - confidence: 0.87
- **[P2] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, causing temporary `no-entry` results and incorrect routing decisions.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes Discord access-control routing and introduces concrete authorization/availability risks that should be fixed before merge.

Concerns:

- **[medium] Claims can blackhole denied channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The registry claims raw numeric channel keys without checking whether the instance will actually process that channel, so peers can skip a channel that the claiming instance later rejects.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
