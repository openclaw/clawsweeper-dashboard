## Summary

Keep open: current main does not implement Discord instance ownership claims, and this PR still targets a real unresolved multi-instance routing bug. The branch is not merge-ready because the shared claim store can lose peer claims, the claim builder can overclaim channels that current auth would deny, native lookup is not yet aligned with current-main helper structure, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a shared Discord channel-claim registry and gates Discord guild messages plus native slash interactions so sibling OpenClaw instances sharing a bot token can skip channels owned by another instance.

## Best Possible Solution

Rebase onto current main, implement a lifecycle-owned serialized or per-instance Discord ownership store, derive claims from effective allowed channel configs, pass bot and resolved channel IDs through the current auth helpers, then add focused regression tests plus live redacted two-instance proof.

## Solution Assessment

Is this the best way to solve the issue? Not yet; a shared ownership registry may be viable, but it must be serialized, derived from effective auth, and integrated with current native auth helpers before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` still reads, prunes, mutates, and rewrites the whole shared claims file without any cross-process lock, CAS, or per-instance file split. Two instances refreshing at startup or on the same timer tick can drop the peer bucket, so lookups return `no-entry` and the wrong instance can process or deny the channel again.
  - confidence: 0.9
- **[P1] Claim only effectively allowed channels:** `extensions/discord/src/monitor/instance-claims.ts:213-218`
  - body: `buildClaimEntries` records every numeric channel key as owned before checking whether that entry is effectively allowed. A channel configured with `enabled: false` can be claimed by one instance, make siblings skip it as `claimed-by-other`, and then be dropped by the owner, blackholing valid traffic that another instance might otherwise handle.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id for claims:** `extensions/discord/src/monitor/message-handler.preflight.ts:781-782`
  - body: The new preflight gate queries ownership with `message.channelId`, but this function already normalizes the usable channel id into `messageChannelId` for payloads where the message object is incomplete. In those cases the claim lookup sees an empty id and falls back to the local allowlist path, reintroducing the wrong-instance denial.
  - confidence: 0.82
- **[P2] Scope native claim checks to the active bot:** `extensions/discord/src/monitor/native-command.ts:859-866`
  - body: Native autocomplete and dispatch call `resolveDiscordClaimOwnership` without passing the interaction's bot/application id, so the lookup scans all bot keys in the shared file. In multi-bot or multi-account hosts, one bot's claim can make another bot silently skip a valid interaction.
  - confidence: 0.8
