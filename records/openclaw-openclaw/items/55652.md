## Summary

Keep open: current main still lacks shared Discord ownership gating and the PR targets a real multi-instance Discord bug, but the latest branch still has access-control/message-delivery blockers and no inspectable real behavior proof.

## What This Changes

This PR adds a Discord shared instance-claim registry, provider refresh wiring, slash/autocomplete/message preflight gates, and focused claim tests to skip channels owned by sibling instances.

## Best Possible Solution

Rework the Discord plugin implementation so claims are derived from effective per-bot channel access, stored without losing peer claims, queried with resolved channel and bot identity, and proven with redacted two-instance Discord evidence.

## Reproduction Assessment

Yes for a source-level path: current main still has the local Discord native-command deny response and no shared claim gate. I did not run the live two-instance Discord setup, so the proof is not a high-confidence real-environment reproduction.

## Solution Assessment

No. The Discord plugin is the right layer, but this implementation is not the best mergeable solution until effective access semantics, bot scoping, resolved channel identity, shared-write safety, and real behavior proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build claims from effective channel access:** `extensions/discord/src/monitor/instance-claims.ts:213-218`
  - body: `buildClaimEntries` records every numeric channel key before applying the same effective access policy used by ingress. Disabled or denied channels can be claimed and blackholed by peers, while name/slug keys and wildcard-with-overrides allowed channels are missed and can still fall through to the old local deny path.
  - confidence: 0.9
- **[P2] Pass the active bot id into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete lookup calls `resolveDiscordClaimOwnership` without a `botId`, and the dispatch path has the same pattern. When the requested bot id is absent the resolver scans every bot bucket, so a different bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:775-783`
  - body: Preflight already normalizes payload fallbacks into `messageChannelId`, but the new ownership lookup queries `message.channelId`. Payloads that rely on `channel_id` or raw-data fallbacks can miss the claim and fall through to the local allowlist denial.
  - confidence: 0.87
- **[P2] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` does an unlocked read-modify-write of one shared JSON file, so two OpenClaw instances can overwrite each other's fresh claims. The fixed temp filename also leaves same-process refreshes vulnerable to rename collisions when multiple accounts refresh together.
  - confidence: 0.84
