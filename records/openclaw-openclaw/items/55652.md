## Summary

Keep open: current main still lacks the Discord multi-instance ownership gate, but this PR is not merge-ready because it changes Discord access-control routing without artifact-backed real behavior proof and still has claim-scoping/storage correctness issues.

## What This Changes

This PR adds a Discord shared instance-claim registry, provider refresh wiring, slash/message ownership gates, and focused claim tests.

## Best Possible Solution

Rework this in the current Discord plugin tree so ownership claims are derived from effective per-bot channel access, stored without losing peer claims, scoped by stable instance/config and bot identity, and proven with redacted two-instance Discord evidence.

## Reproduction Assessment

Yes, for a source-level path: current main still has the local Discord native-command deny response and no shared claim gate. I did not run the live two-instance Discord setup described by the reporter.

## Solution Assessment

No. The Discord plugin is the right layer, but this branch is not the best mergeable solution until effective access semantics, bot scoping, resolved channel identity, shared-write safety, current-main rebase, and real behavior proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build claims from effective allowed channels:** `extensions/discord/src/monitor/instance-claims.ts:213-218`
  - body: `buildClaimEntries` records every numeric channel key before applying the same effective access policy used by ingress. Disabled or denied channels can be claimed and blackholed by peers, while name/slug and wildcard-with-overrides allowed channels are not claimed and can still fall through to the old local deny path.
  - confidence: 0.9
- **[P2] Scope native claim lookups to the active bot:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete and dispatch ownership lookups omit the active bot/application id, so the resolver can scan unrelated bot buckets in the shared file. On a multi-bot host, another bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:775-783`
  - body: Preflight already normalizes payload fallbacks into `messageChannelId`, but the new ownership lookup queries `message.channelId`. Payloads that rely on `channel_id` or raw-data fallbacks can miss the claim and fall through to the old local allowlist denial.
  - confidence: 0.87
- **[P2] Serialize shared claim file updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances or multiple accounts refreshing together can overwrite each other's fresh claims, producing temporary `no-entry` ownership decisions.
  - confidence: 0.84
- **[P2] Pass config-specific instance keys into refreshes:** `extensions/discord/src/monitor/provider.ts:1038-1044`
  - body: The provider refresh calls only pass `cfg` and `accountId`, so launches without `OPENCLAW_CONFIG_PATH` or explicit `gateway.instanceId` can collapse common accounts such as `default` to the same `openclaw-default` key. That makes sibling instances overwrite each other's claims instead of isolating ownership.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch changes Discord access-control routing and has concrete blackhole/suppression risks from overbroad claims, unscoped bot lookups, and lossy shared-file writes.

Concerns:

- **[medium] Claims can blackhole disabled channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The registry can claim raw numeric channel keys without proving the instance will process that channel, so peers may skip traffic that the claiming instance later rejects.
  - confidence: 0.9
- **[medium] Native claim checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:465`
  - body: Native command ownership checks omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
- **[low] Instance keys can collapse without config path:** `extensions/discord/src/monitor/provider.ts:1038`
  - body: Provider refreshes do not pass the active config path, so starts without the expected environment can reuse the same fallback instance key and overwrite sibling ownership.
  - confidence: 0.78
