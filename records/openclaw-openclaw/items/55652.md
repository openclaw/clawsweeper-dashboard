## Summary

Keep open: current main still lacks Discord instance-claim routing and the linked bug remains plausible, but the current PR head has blocking routing/access-control issues and lacks inspectable real behavior proof.

## What This Changes

The PR adds a Discord shared instance-claim registry, provider refresh, message preflight/native command ownership gates, and focused tests under extensions/discord/src/monitor.

## Best Possible Solution

Keep the fix in the Discord plugin, but derive claims from effective allowed channel config for the active bot and resolved channel/thread/guild identity, then publish claims through lossless serialized or per-instance storage with focused regression coverage and real two-instance proof.

## Reproduction Assessment

Yes. The linked report gives a concrete two-instance setup sharing one Discord bot token, and current main still has the local deny path with no sibling ownership registry; I did not run the live setup here.

## Solution Assessment

No. The Discord plugin is the right layer, but this patch is not merge-safe until claim publication is allowlist-aware, bot-scoped, based on resolved IDs/names, and robust against lost peer claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Claim only channels this instance will process:** `extensions/discord/src/monitor/instance-claims.ts:213-221`
  - body: `buildClaimEntries` records every numeric channel key as owned without checking whether that entry is disabled or denied. Because peers skip claimed channels before their own allowlist checks, one instance can claim a channel it later rejects and blackhole traffic for all sibling instances.
  - confidence: 0.9
- **[P2] Pass the active bot id into native claim lookups:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete and dispatch claim checks omit `botId`, so `resolveDiscordClaimOwnership` can scan claims for every bot key in the shared file. In multi-bot deployments, another bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: Preflight already resolves `messageChannelId` from the message and event fallback, but the new claim lookup uses `message.channelId`. Payloads that only carry the channel id through the event wrapper will miss ownership claims and can fall back to the local deny path this PR is meant to avoid.
  - confidence: 0.86
- **[P2] Serialize shared claims file updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, temporarily returning `no-entry` and letting the wrong instance handle or deny the channel.
  - confidence: 0.84
- **[P2] Preserve name-based Discord channel allowlists:** `extensions/discord/src/monitor/instance-claims.ts:213-216`
  - body: Discord channel config can match by channel name or slug, but this registry discards every nonnumeric key and the lookup API has no channel-name input. Name-configured channels therefore never get claimed, so sibling instances can still take the old incorrect deny path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes Discord authorization routing and introduces concrete availability/access-control risks that should be fixed before merge.

Concerns:

- **[medium] Claims can blackhole denied channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The registry claims raw numeric channel keys without checking whether the owner will actually process that channel, so peers can skip a channel that the claiming instance later denies or disables.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
