## Summary

Keep open: current main still lacks Discord sibling ownership claims, and this PR targets a real multi-instance Discord bug, but the current patch still has access-control/shared-state blockers and lacks inspectable real two-instance proof.

## What This Changes

The PR adds a Discord shared instance-claim registry, periodic provider refresh, guild message and native command ownership gates, and focused tests under `extensions/discord/src/monitor`.

## Best Possible Solution

Keep the fix in the Discord plugin, but publish only effective allowed ownership for the active bot and resolved channel/thread/guild identity through lossless shared storage, with focused regression coverage and redacted two-instance Discord proof.

## Reproduction Assessment

Yes, this is source-reproducible from the linked two-instance setup plus current-main code: a non-owner Discord instance still reaches the local allowlist deny path with no sibling ownership registry. I did not run the live Discord setup here.

## Solution Assessment

No. The Discord plugin is the right layer, but this patch is not merge-safe until claims are allowlist-aware, bot-scoped, based on resolved channel identity, and protected against lost peer writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Claim only channels this instance will process:** `extensions/discord/src/monitor/instance-claims.ts:213-221`
  - body: `buildClaimEntries` records every numeric channel key without checking the entry value. Because peers skip claimed channels before local channel checks, a channel configured as disabled or denied can be skipped by siblings and then rejected by the claiming instance, blackholing traffic.
  - confidence: 0.9
- **[P2] Scope native claim checks to the active bot:** `extensions/discord/src/monitor/native-command.ts:465-472`
  - body: The native autocomplete and dispatch claim checks call `resolveDiscordClaimOwnership` without `botId`, so the resolver can scan claims for every bot key in the shared file. In multi-bot or multi-account hosts, another bot's claim can silently suppress valid interactions for this bot.
  - confidence: 0.88
- **[P2] Use the resolved preflight channel id:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: Preflight already normalizes the channel into `messageChannelId` from `channelId`, `channel_id`, `rawData.channel_id`, and the event fallback, but the new claim lookup uses only `message.channelId`. Payloads that rely on those fallbacks miss ownership claims and can fall back to the old local deny path.
  - confidence: 0.87
- **[P2] Serialize shared claim updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` performs an unlocked read-modify-write of one shared JSON file. Two instances refreshing together can overwrite each other's fresh claims, causing temporary `no-entry` results and incorrect routing decisions.
  - confidence: 0.84
- **[P2] Preserve name-based channel claims:** `extensions/discord/src/monitor/instance-claims.ts:213-216`
  - body: Discord channel allowlists can match by channel name or slug, but this registry discards every nonnumeric channel key. Name-configured channels never publish ownership, so sibling instances can still take the incorrect local deny path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes Discord access-control routing and introduces concrete availability/authorization risks that should be fixed before merge.

Concerns:

- **[medium] Claims can blackhole disabled channels:** `extensions/discord/src/monitor/instance-claims.ts:218`
  - body: The registry claims raw numeric channel keys without checking whether the owner will actually process that channel, so peers can skip a channel that the claiming instance later rejects.
  - confidence: 0.9
- **[medium] Native ownership checks are not bot-scoped:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership lookups omit the active bot id, allowing claims for a different Discord bot in the shared file to suppress this bot's autocomplete or slash-command dispatch.
  - confidence: 0.88
- **[medium] Shared registry writes can lose peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The shared claims file is updated through an unlocked read-modify-write cycle, so concurrent OpenClaw instances can overwrite each other's ownership data and route authorization decisions incorrectly.
  - confidence: 0.84
