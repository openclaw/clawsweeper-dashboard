## Summary

Keep open: the latest head removed the explicit raw screenshot `MEDIA` marker and `details.media`, but the new vision-success path still returns untrusted vision text from a trusted browser tool without neutralizing `MEDIA:` directive-shaped lines, so channel media extraction can still treat model/page text as deliverable media.

## Best Possible Solution

Land screenshot vision only after untrusted vision descriptions cannot synthesize channel media delivery, ideally by making external-content-wrapped text non-media-parsable or escaping directive-shaped lines before returning the tool result.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR returns wrapped vision output as a browser text result, while current main scans browser text results for `MEDIA:` lines and trusts browser local media. I did not run a live forged-page scenario in this read-only review.

## Solution Assessment

No. The feature shape is plausible and the explicit raw screenshot markers were removed, but the text path still needs `MEDIA:` neutralization or extractor awareness before this is the safest implementation.
