## Summary

This PR should stay open: the screenshot-vision feature is not already on current main, and the branch still has merge-blocking security and compatibility defects in the new vision path. The supplied recording is useful proof for the happy path, but current-head proof still needs to cover directive-shaped text and the failed-vision fallback after those fixes.

## What This Changes

The PR adds opt-in `browser.models` screenshot vision for text-only main models, updates browser config/schema/docs/tests, and forwards media-understanding auth profile fields.

## Best Possible Solution

Land this feature only after browser vision text cannot synthesize delivery directives, the failure fallback preserves existing image sanitization, and final proof covers both the happy path and the negative delivery path.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR returns wrapped vision text as a browser text content block, while current main parses line-start `MEDIA:` from all text blocks and trusts browser media. The sanitization regression is also source-reproducible because the PR catch fallback omits the option that current main passes on the normal screenshot path.

## Solution Assessment

No, not yet: the feature direction is reasonable, but the implementation needs directive neutralization and the existing screenshot sanitization guard on all fallback image paths. A safer solution is to treat browser vision output like other untrusted transcript text that must be directive-safe before it reaches tool-result media parsing.
