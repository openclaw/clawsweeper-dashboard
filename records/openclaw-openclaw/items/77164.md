## Summary

Keep open: current main still has Telegram conversation-binding infrastructure but no Telegram `subagent_spawning` or delivery hook registration, and the core `sessions_spawn(thread=true)` path requires those hooks before a thread-bound session can bind.

## What This Changes

Request Telegram to either support `sessions_spawn(thread=true, mode="session")` through subagent lifecycle hooks or clearly document that native Telegram thread-bound subagent sessions are unsupported.

## Best Possible Solution

Add Telegram-owned subagent lifecycle hooks that use the existing Telegram conversation binding manager for native thread-bound spawns, then reconcile docs and config help with the supported Telegram scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram `sessions_spawn(thread=true)` call needs `subagent_spawning`, but Telegram registers no such hook while the core spawn path rejects without it.

## Solution Assessment

Yes. The narrow maintainable fix is a Telegram-owned hook bridge analogous to Discord, Matrix, and Feishu; a docs-only change would leave current Telegram config help and binding infrastructure inconsistent.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77164. Telegram native thread-bound subagent sessions currently fail because `spawnSubagentDirect` requires `subagent_spawning` hooks, but Telegram only registers conversation binding infrastructure. Add a Telegram-owned lazy subagent hook registration path, likely `extensions/telegram/subagent-hooks-api.ts` plus `extensions/telegram/src/subagent-hooks.ts`, and register it from the bundled Telegram entry without adding Telegram-specific logic to core. Implement `subagent_spawning`, `subagent_delivery_target`, and `subagent_ended` using the existing Telegram thread binding manager/session binding service; support direct chats and forum topics conservatively, and return actionable errors when the requester context cannot be bound. Add focused regression tests, update Telegram/subagent docs and config help to match the implemented scope, and add a user-facing `CHANGELOG.md` fix entry.
