## Summary

Keep open, but the patch is not merge-ready: it targets a real system-event display leak at a plausible shared sanitizer, yet the new regex also drops the current `System (untrusted):` marker used for sanitized user-controlled text, causing visible message and history loss.

## What This Changes

The PR adds shared display-sanitizer logic and tests to drop line-leading `System:` and `System (untrusted):` event lines from inbound user-visible text while preserving prose such as `Operating System:`.

## Best Possible Solution

Keep the shared sanitizer from deleting `System (untrusted):` user text, and strip generated system-event lines only through trusted event provenance or a narrower generated-event pattern with regression coverage.

## Reproduction Assessment

Yes for the PR-introduced regression at source level: current inbound finalization rewrites user-controlled `System:` lines to `System (untrusted):`, and the patched sanitizer would drop those lines. The original leak has linked-report and source-path evidence, but I did not establish a live current-main reproduction during this read-only review.

## Solution Assessment

No. The shared sanitizer is a plausible defense-in-depth location, but the submitted regex is too broad; the maintainable fix must preserve sanitized user text and hide only generated internal events.
