## Summary

Keep open. Current main still gates raw conversation typed hooks from per-plugin host config only, the native manifest contract does not preserve `permissions.conversationAccess`, and open PR #72690 is an unmerged closing implementation candidate that should be reviewed or replaced before this issue closes.

## What This Changes

Request to add a native plugin manifest field for declaring conversation-hook access needs while keeping the existing `plugins.entries.<id>.hooks.allowConversationAccess` config path backward compatible.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request/disclosure metadata through manifest parsing, registry/status/docs surfaces, and focused tests, while preserving explicit host config as the effective approval or deny gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` still follows `hookPolicy: entry?.hooks` and is blocked on current main unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The safer maintainable direction is manifest request metadata plus explicit host approval, which the linked PR describes but has not merged.
