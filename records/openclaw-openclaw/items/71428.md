## Summary

Keep open: current main still has no native manifest `permissions.conversationAccess` contract, and the linked implementation PR remains open, conflicting, and not merged. The issue should continue tracking the product/security decision until that PR lands or is replaced.

## What This Changes

The issue requests an additive native plugin manifest field for declaring conversation-hook access instead of requiring third-party plugins to patch host config schema surfaces.

## Best Possible Solution

Review and land or replace the linked implementation with a narrow request-only manifest metadata contract that keeps explicit host approval and documents the trust model.

## Reproduction Assessment

Yes for the current gap by source inspection: a non-bundled plugin with only manifest `permissions.conversationAccess` would still receive hook policy from host config and be blocked from `agent_end` without `allowConversationAccess=true`. No runtime command was run because this was a read-only review.

## Solution Assessment

Unclear. Request-only manifest metadata appears safer than the original self-authorizing sketch, but maintainers still need to accept the trust model and land or replace the open implementation PR.
