## Summary

Keep open. Current main still lacks a native manifest `permissions.conversationAccess` contract and still grants raw conversation typed-hook access only from per-plugin host config; open PR #72690 is a linked implementation candidate, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

Request to add a native plugin manifest field for declaring conversation-content hook access needs while preserving the existing host-config opt-in path.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata through manifest parsing, registry/status/docs surfaces, and focused loader/registry tests while keeping explicit host config as the effective approval or denial gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` will still be blocked on current main unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The safer maintainable solution is manifest request/disclosure metadata plus explicit host approval, which the linked PR describes but has not merged.
