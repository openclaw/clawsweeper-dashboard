## Summary

Keep open. Current main still enforces raw conversation typed hooks from per-plugin host config only, the native plugin manifest contract still has no `permissions.conversationAccess` field, and open PR #72690 is a linked implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request adding a plugin manifest `permissions.conversationAccess` field so plugins can declare raw conversation hook access needs without downstream config-schema patches while preserving backward-compatible config behavior.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata, carries it through manifest registry/status/docs, keeps explicit host config as the effective grant or deny, and adds focused loader/registry tests for manifest request preservation plus default rejection.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end`; current main ignores that manifest field and blocks the hook unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original RFC sketch if manifest `true` grants access by itself. The safer maintainable direction is manifest request/disclosure metadata plus explicit host approval, which is what the linked implementation PR claims to preserve.
