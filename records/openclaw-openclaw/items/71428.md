## Summary

Keep open: current main still lacks a native manifest `permissions.conversationAccess` contract and continues to enforce raw conversation hooks from host config only; https://github.com/openclaw/openclaw/pull/72690 is open with closing syntax and should be reviewed, landed, replaced, or closed before this issue is resolved.

## What This Changes

Request to add a plugin manifest field for declaring conversation-hook access needs while preserving the existing host-config flag for compatibility.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/72690 with a documented manifest request field that preserves explicit host approval for actual raw conversation-hook access.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-bundled plugin that only declares manifest `permissions.conversationAccess: true` would still receive `hookPolicy: entry?.hooks` and be blocked unless host config grants `allowConversationAccess`.

## Solution Assessment

No for the original self-authorizing manifest sketch; the safer solution is request metadata in the manifest plus explicit host approval or a reviewed deny-precedence policy. The open implementation PR is the right review target before closing this issue.
