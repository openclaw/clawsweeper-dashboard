## Summary

Keep open. Current main still models conversation-hook access as host config only, and open PR #72690 is an unmerged implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests an additive `permissions.conversationAccess` native plugin manifest field to replace local schema patches while preserving the existing host-config opt-in path.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request metadata through manifest parsing, registry/status/docs surfaces, and focused tests while keeping host config as the effective approval gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` still reaches `hookPolicy: entry?.hooks` and is blocked unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The maintainable direction is manifest request/disclosure metadata plus explicit host approval, which the open linked PR describes but has not merged.
