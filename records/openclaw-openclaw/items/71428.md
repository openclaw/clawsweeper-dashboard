## Summary

Keep open: current main still has host-config-only authorization for non-bundled conversation hooks, no native manifest `permissions.conversationAccess` contract, and the closing implementation PR is still open and conflicting. The issue remains the right product/security tracker until maintainers land or replace that PR.

## What This Changes

Request adding a plugin manifest `permissions.conversationAccess` field so plugins can declare conversation-hook access instead of relying only on host config schema patches.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/72690 with a narrow manifest metadata contract, documented trust model, and tests that preserve host approval unless maintainers explicitly choose a stronger policy.

## Reproduction Assessment

Yes, by source inspection: a non-bundled plugin with only manifest `permissions.conversationAccess` would still receive hook policy from host config and be blocked from `agent_end` without `allowConversationAccess=true`. No runtime command was run because this was a read-only review.

## Solution Assessment

Unclear. Request-only manifest metadata appears safer than manifest self-authorization, but maintainers need to decide whether that narrower trust model satisfies the RFC.
