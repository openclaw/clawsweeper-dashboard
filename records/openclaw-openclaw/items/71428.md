## Summary

Keep open: current main and v2026.6.8 still gate raw conversation hooks only through host config, while the native plugin manifest has no `permissions.conversationAccess` contract or registry carrier. The linked implementation PR was closed unmerged, so the remaining work is a maintainer product and security decision, not conservative cleanup.

## What This Changes

Request to add a plugin manifest `permissions.conversationAccess` field so non-bundled plugins can declare raw conversation-hook access instead of patching host config schemas.

## Best Possible Solution

Adopt a maintainer-approved additive manifest request contract with explicit host allow or deny precedence, then carry it through manifest loading, manifest registry records, loader hook policy, public docs, and focused tests.

## Reproduction Assessment

Yes, source-based. Current main and v2026.6.8 have no manifest `permissions.conversationAccess` field or manifest-record carrier, and the loader/registry path still uses only `plugins.entries.<id>.hooks.allowConversationAccess`.

## Solution Assessment

Unclear as written. Manifest metadata is a plausible plugin API direction, but self-granting raw conversation access is not obviously safe; the best fix needs maintainer-approved request/grant semantics and host override or deny precedence.
