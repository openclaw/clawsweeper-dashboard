## Summary

Keep open: current main still enforces conversation-content hooks from host config only, and open PR #72690 is the unmerged implementation candidate with closing syntax.

## What This Changes

Request to declare conversation hook access in `openclaw.plugin.json` as `permissions.conversationAccess` while keeping the existing host config flag backward-compatible.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level request metadata, registry/status/docs coverage, and focused tests while preserving explicit host config as the effective approval gate.

## Reproduction Assessment

Yes. On current main, a non-bundled plugin declaring only `permissions.conversationAccess: true` and registering `agent_end` reaches `hookPolicy: entry?.hooks` and is blocked unless host config sets `allowConversationAccess=true`.

## Solution Assessment

No, not if the manifest field grants access by itself. The maintainable solution is manifest request/disclosure metadata plus explicit operator approval through host config, which the open PR describes but has not merged.
