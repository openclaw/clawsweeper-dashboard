## Summary

Keep open. Current main still enforces raw conversation-content typed hooks from per-plugin host config only, and the native plugin manifest/registry/docs/test surfaces do not yet carry `permissions.conversationAccess`; open PR #72690 is a linked implementation candidate, not a reason to close before merge.

## What This Changes

The issue proposes adding a `permissions.conversationAccess` field to native plugin manifests so plugins can declare raw conversation hook needs while preserving backward compatibility with `plugins.entries.<id>.hooks.allowConversationAccess`.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata, preserves it through manifest parsing/registry/status/docs, keeps explicit host config as the effective grant or deny, and adds focused tests for manifest preservation plus default rejection.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end`; current main ignores that manifest field and blocks the hook unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original RFC if manifest `true` grants access by itself. The safer maintainable solution is manifest request/disclosure metadata plus explicit host approval, which the linked implementation PR claims to preserve but has not merged.
