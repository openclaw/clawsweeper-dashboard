## Summary

Keep open: current main and the latest release still model raw conversation-hook access as host-config policy only, while the manifest/registry contract has no `permissions.conversationAccess` path. The linked implementation PR is still open, so this issue should remain the tracking item until that PR lands or is replaced.

## What This Changes

Add a plugin manifest declaration for conversation-hook access needs so plugin authors can disclose the requirement without patching host config schemas.

## Best Possible Solution

Review and land or replace `https://github.com/openclaw/openclaw/pull/72690` with a narrow manifest metadata contract that documents requested access while preserving explicit host approval and clear precedence tests.

## Reproduction Assessment

Yes. Source inspection shows a non-bundled plugin that only declares manifest `permissions.conversationAccess: true` would still receive `hookPolicy: entry?.hooks`, and `registerTypedHook` blocks `agent_end` unless host config grants `allowConversationAccess`.

## Solution Assessment

No for the original self-authorizing sketch. The safer path is request/disclosure metadata in the manifest plus explicit host-config approval or a clearly reviewed deny-precedence policy.
