## Summary

Keep open: current main and the latest release accept the existing host-config flag, but they still do not provide a native manifest `permissions.conversationAccess` contract or a loader/registry path that uses manifest metadata for conversation-hook access.

## What This Changes

The issue requests a plugin manifest permission for conversation-content typed hooks so non-bundled plugins do not need per-install schema patches or host-config-only opt-in.

## Best Possible Solution

Keep this issue open until maintainers choose the trust model, then implement a documented manifest contract with explicit host-config precedence and tests for manifest-only, config-only, both-present, and default-deny behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows no manifest permission field, no manifest-record carrier, and a loader/registry path that still uses only `plugins.entries.<id>.hooks.allowConversationAccess`.

## Solution Assessment

Unclear as written. Manifest metadata is a plausible plugin API direction, but self-granting raw conversation access is not obviously the safest solution; the best path needs maintainer-approved request/grant semantics and host override or deny precedence.
