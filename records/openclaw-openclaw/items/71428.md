## Summary

Keep open: current main still grants raw conversation typed hooks only through `plugins.entries.<id>.hooks.allowConversationAccess`, the native manifest path still lacks `permissions.conversationAccess`, and https://github.com/openclaw/openclaw/pull/72690 is the unmerged implementation candidate.

## What This Changes

Add a native plugin manifest `permissions.conversationAccess` field so plugins can declare raw conversation-hook needs instead of carrying host schema patches.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/72690 with a narrow manifest metadata path that preserves explicit host approval for raw conversation hooks.

## Reproduction Assessment

Yes. A non-bundled plugin that only declares manifest `permissions.conversationAccess: true` still gets `hookPolicy: entry?.hooks`, so `registerTypedHook` blocks `agent_end` unless host config grants `allowConversationAccess`.

## Solution Assessment

No as originally sketched if manifest `true` grants access by itself. The maintainable direction is request/disclosure metadata in the manifest plus explicit host-config approval or a clearly reviewed deny-precedence policy.
