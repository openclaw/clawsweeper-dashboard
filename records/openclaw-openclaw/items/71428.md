## Summary

Keep open: current main and v2026.6.8 still have the host-config `allowConversationAccess` gate but no native manifest `permissions.conversationAccess` contract, and the only linked implementation PR was closed unmerged as an unmergeable candidate.

## What This Changes

The issue requests a plugin manifest permission field for conversation-access hook declarations so plugins do not need host config schema patches for that policy surface.

## Best Possible Solution

Keep the issue open until maintainers choose the trust model, then implement a documented manifest contract with explicit request/grant semantics, host allow or deny precedence, docs, and focused loader/manifest-registry tests.

## Reproduction Assessment

Yes, source-based. Current main and v2026.6.8 have no manifest `permissions.conversationAccess` field or manifest-record carrier, and the loader/registry path still uses only `plugins.entries.<id>.hooks.allowConversationAccess`.

## Solution Assessment

Unclear as written. Manifest metadata is a plausible plugin API direction, but self-granting raw conversation access is not obviously safe; the best fix needs maintainer-approved request/grant semantics and explicit host override or deny precedence.
