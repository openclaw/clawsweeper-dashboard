## Summary

Keep open: current main and v2026.6.10 still gate raw conversation hooks from per-plugin host config only, while the native plugin manifest has no `permissions.conversationAccess` field or loader/registry merge path. The request remains a real plugin API gap, but it needs maintainer product and security direction because plugin-authored manifest metadata could change raw conversation-content trust semantics.

## What This Changes

The issue requests moving raw conversation-hook access from per-plugin host config into an additive `permissions.conversationAccess` plugin manifest field, with config compatibility retained.

## Best Possible Solution

Adopt a maintainer-approved additive manifest contract with explicit request/grant and host override semantics, then carry it through manifest loading, manifest registry records, loader hook policy, docs, and focused tests.

## Reproduction Assessment

Yes for source-level reproduction: current `main` blocks non-bundled conversation hooks unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`, and the manifest path still has no `permissions.conversationAccess` carrier.

## Solution Assessment

Unclear as written. Manifest metadata is a plausible plugin API direction, but maintainers need to approve whether it can grant raw conversation access or only declare a request that host config must still approve.
