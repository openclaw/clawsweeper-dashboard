## Label Justifications

- `P2`: This is a normal-priority plugin manifest and hook-policy improvement with limited blast radius but real product and security-model implications.
- `impact:security`: The issue concerns declaring or granting access to raw conversation-content hooks, which is a sensitive data boundary.

## Summary

Keep open: current main still authorizes non-bundled conversation hooks from host config only, the native plugin manifest has no `permissions.conversationAccess` contract, and https://github.com/openclaw/openclaw/pull/72690 is an open conflicting implementation candidate rather than a merged fix.

## What This Changes

Request an additive plugin manifest field for declaring conversation-hook access needs, replacing external schema patches while preserving backward compatibility with the existing host config flag.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/72690 with a narrow manifest permission/request contract, documented trust model, and tests that preserve host approval unless maintainers explicitly choose manifest-granted access.

## Reproduction Assessment

Yes, by source inspection: a non-bundled plugin with only manifest `permissions.conversationAccess` would still receive hook policy from host config and be blocked from `agent_end` without `plugins.entries.<id>.hooks.allowConversationAccess=true`. No runtime command was run because this was a read-only review.

## Solution Assessment

Unclear. Request-only manifest metadata is safer than manifest self-authorization, but maintainers need to decide whether that narrower trust model satisfies the RFC.
