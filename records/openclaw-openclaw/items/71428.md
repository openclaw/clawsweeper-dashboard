## Summary

Keep open: current main still models raw conversation-hook access as host-config policy only, and the manifest/registry contract has no `permissions.conversationAccess` path. The linked implementation PR is still open with closing syntax, so this issue should remain the tracker until that PR lands or is replaced.

## What This Changes

The issue requests a native plugin manifest field for declaring conversation-access requirements instead of relying only on per-plugin host config schema entries.

## Best Possible Solution

Review and land or replace `https://github.com/openclaw/openclaw/pull/72690` with a narrow manifest metadata contract that documents requested access while preserving explicit host approval and precedence tests.

## Reproduction Assessment

Yes. Source inspection shows a non-bundled plugin that only declares manifest `permissions.conversationAccess: true` would still receive `hookPolicy: entry?.hooks`, and `registerTypedHook` blocks `agent_end` unless host config grants `allowConversationAccess`.

## Solution Assessment

No for the original self-authorizing sketch. The safer direction is manifest request/disclosure metadata plus explicit host-config approval or a clearly reviewed deny-precedence policy.
