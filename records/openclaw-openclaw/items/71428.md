## Summary

Keep open: current main and v2026.6.8 still lack a native manifest `permissions.conversationAccess` field or registry/loader merge path, while raw conversation-hook access remains host-config gated. The request is a real plugin API gap, but it needs maintainer product and security direction because plugin-authored manifest metadata could change raw conversation-content trust semantics.

## What This Changes

Request an additive plugin manifest permission field for conversation-hook access instead of requiring host config schema patches.

## Best Possible Solution

Adopt a maintainer-approved additive manifest contract with explicit request/grant and host override semantics, then carry it through manifest loading, manifest registry records, loader hook policy, docs, and focused tests.

## Reproduction Assessment

Yes for the current-state check. Source inspection shows current main and v2026.6.8 still have no manifest `permissions.conversationAccess` field or registry carrier, and hook enforcement uses only host config.

## Solution Assessment

Unclear as written. Manifest metadata is a plausible plugin API direction, but maintainers need to approve whether it can grant raw conversation access or only declare a request that host config must still approve.
