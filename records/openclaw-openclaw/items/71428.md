## Summary

Keep open. Current main still grants raw conversation typed hooks only through host config, the native manifest path does not model `permissions.conversationAccess`, and open PR #72690 is the unmerged implementation candidate that should be reviewed or replaced before this issue closes.

## What This Changes

Request an additive `permissions.conversationAccess` native plugin manifest field so plugin authors can declare raw conversation-hook access needs while preserving existing config compatibility.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request metadata through manifest parsing, registry/status/docs surfaces, and focused tests while preserving explicit host config as the effective approval gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` still reaches `hookPolicy: entry?.hooks` and is blocked unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The best maintainable direction is request/disclosure metadata in the manifest plus explicit host approval, which the linked PR describes but has not merged.
