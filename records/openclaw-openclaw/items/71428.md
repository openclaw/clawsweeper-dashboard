## Summary

Keep open. Current main still has only host-config-based conversation hook approval, and open PR #72690 is the unmerged implementation candidate that specifically targets this issue.

## What This Changes

The issue requests a native plugin-manifest `permissions.conversationAccess` field for conversation hook access metadata while preserving the existing host config path.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, as manifest request metadata plus explicit host-config approval rather than manifest-only grant authority.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` and registers `agent_end` would still reach `hookPolicy: entry?.hooks` and be blocked unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original manifest-grants-access sketch. The safer maintainable path is manifest disclosure/preservation with host config remaining the effective approval gate, which is what PR #72690 describes.
