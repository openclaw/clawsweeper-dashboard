## Summary

Keep open. Current main still does not implement manifest-declared conversation access: the manifest, manifest registry, and loader do not carry `permissions.conversationAccess`, and hook enforcement still reads only host config policy. The linked open PR #72690 is an implementation candidate, not closure evidence before merge, and the remaining design is security-sensitive because plugin-owned manifest data must not self-grant raw conversation-hook access.

## What This Changes

Add an additive native plugin manifest field for conversation-access metadata while preserving existing host-config compatibility and precedence.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata, preserves it through manifest registry/status/docs, keeps explicit host config as the effective grant or deny, and adds focused loader/registry tests for default rejection and compatibility.

## Reproduction Assessment

Yes. A non-bundled plugin that only declares `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` will still be blocked on current main, because the loader passes only `plugins.entries.<id>.hooks` into hook policy.

## Solution Assessment

No for the original manifest-as-grant sketch. The safer maintainable solution is manifest request/disclosure metadata plus explicit operator approval or denial in host config, with tests documenting that manifest-only access does not grant raw conversation hooks.
