## Summary

Keep open. Current main still only grants raw conversation typed-hook access from per-plugin host config, the native plugin manifest/manifest registry do not model `permissions.conversationAccess`, and open PR #72690 is a linked implementation candidate rather than a reason to close before merge.

## What This Changes

Request to add a native plugin manifest `permissions.conversationAccess` declaration for conversation-content hook access metadata while keeping the existing host config opt-in compatible.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata, preserves it through manifest/registry/status/docs surfaces, and keeps explicit host config as the effective approval or denial gate with focused loader/registry tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end`; current main ignores that manifest field and blocks registration unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original RFC sketch if manifest `true` grants access by itself. The safer maintainable solution is manifest request/disclosure metadata plus explicit host approval, which the linked implementation PR claims to preserve.
