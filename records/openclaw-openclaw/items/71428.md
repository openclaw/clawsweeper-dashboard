## Summary

Keep open. Current main still only grants non-bundled conversation-content hooks through host config, the native plugin manifest contract still has no `permissions.conversationAccess` field or registry preservation path, and open PR #72690 is the linked implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request adding manifest-level `permissions.conversationAccess` metadata for plugins while preserving the existing host-config `hooks.allowConversationAccess` compatibility path.

## Best Possible Solution

Land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request metadata, preserves it through manifest loading and registry/status surfaces, documents the semantics, and keeps explicit host config as the effective approval or denial gate for raw conversation hooks.

## Reproduction Assessment

Yes. A code-level reproduction is a non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end`; current main ignores that manifest field and still blocks the hook unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the RFC sketch if the manifest value itself grants access. The maintainable approach is to treat the manifest field as request/disclosure metadata and preserve operator-controlled approval through host config, which is the direction described by PR #72690.
