## Summary

Keep open. Current main still has only the host-config `plugins.entries.<id>.hooks.allowConversationAccess` approval path, the native plugin manifest does not model `permissions.conversationAccess`, and open PR #72690 is a linked implementation candidate that should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests an additive native plugin manifest field for declaring conversation-hook access needs, replacing external schema patches while keeping existing host config compatibility.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request metadata through manifest parsing, registry/status/docs surfaces, and focused tests, while preserving explicit host config as the effective approval or deny gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` still follows `hookPolicy: entry?.hooks` and is blocked on current main unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The safer solution is manifest request/disclosure metadata plus explicit host approval, which the linked PR describes but has not merged.
