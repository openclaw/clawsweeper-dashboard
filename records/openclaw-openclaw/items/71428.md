## Summary

Keep open. Current main still enforces conversation-content typed hooks from per-plugin host config only, the manifest/registry path does not model `permissions.conversationAccess`, and open PR #72690 is the unmerged closing implementation candidate.

## What This Changes

Add a plugin manifest `permissions.conversationAccess` declaration for plugins that need conversation-content typed hooks, replacing the host-config-only schema opt-in while preserving backward compatibility.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that stores manifest-level request metadata, updates docs/tests/status surfaces, and preserves explicit host config as the effective approval gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registers `agent_end` still reaches `hookPolicy: entry?.hooks` and is blocked unless host config sets `allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself. The safer maintainable solution is manifest request/disclosure metadata plus explicit host approval, which is the direction described by PR #72690.
