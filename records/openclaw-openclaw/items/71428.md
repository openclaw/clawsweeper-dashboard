## Summary

Keep open: current main still gates conversation-content typed hooks from host config only, lacks manifest-level conversation-access metadata, and open PR #72690 remains the unmerged implementation candidate.

## What This Changes

The issue requests plugin manifest metadata for `permissions.conversationAccess` while preserving the existing host-config conversation-hook approval path.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest request metadata, preserves it in registry/status/docs/tests, and keeps explicit host configuration as the effective approval gate.

## Reproduction Assessment

Yes. A non-bundled plugin that declares only `permissions.conversationAccess: true` in its manifest and registers `agent_end` still reaches `hookPolicy: entry?.hooks`, so the registry blocks it unless host config sets `allowConversationAccess=true`.

## Solution Assessment

No for the original sketch if manifest `true` grants access by itself; the safer solution is manifest disclosure/request metadata plus explicit host approval, which the open linked PR appears to target but has not merged.
