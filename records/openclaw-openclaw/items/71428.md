## Summary

Keep open: current main and v2026.6.10 still gate raw conversation hooks from per-plugin host config only, and the native manifest path has no `permissions.conversationAccess` carrier. The request remains valid, but it needs maintainer product and security direction because plugin-authored manifest metadata would affect raw conversation-content trust semantics.

## What This Changes

The issue requests a native plugin manifest permission, likely `permissions.conversationAccess`, to replace per-install schema patches for `plugins.entries.<id>.hooks.allowConversationAccess`.

## Best Possible Solution

Adopt a maintainer-approved additive manifest contract with explicit request/grant and host override semantics, then carry it through manifest loading, registry records, loader hook policy, docs, and focused tests.

## Reproduction Assessment

Yes for source-level reproduction: current main still blocks non-bundled conversation hooks unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`, and the manifest path has no `permissions.conversationAccess` carrier. I did not run a live plugin because the source and existing tests directly cover the gate.

## Solution Assessment

Unclear as written. Manifest metadata fits the plugin owner boundary, but maintainers need to approve whether it can grant raw conversation access or only declare a request that host config must approve.
