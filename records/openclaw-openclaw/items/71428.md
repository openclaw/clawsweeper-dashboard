## Summary

Keep open: current main and v2026.6.10 still gate raw conversation hooks from per-plugin host config only, the native manifest has no `permissions.conversationAccess` carrier, and the direct implementation attempt closed unmerged. The request remains valid but needs maintainer product and security direction because plugin-authored manifest metadata would change raw conversation-content trust semantics.

## What This Changes

The issue requests adding `permissions.conversationAccess` to native plugin manifests so plugins can declare or request raw conversation hook access instead of relying on host config schema patches.

## Best Possible Solution

Define an additive manifest permission contract with explicit request/grant and host override semantics, then carry it through manifest loading, registry records, loader hook policy, docs, and focused tests.

## Reproduction Assessment

Yes, source-level: a non-bundled plugin that declares `permissions.conversationAccess` but omits host config still reaches `registerTypedHook` with only `entry?.hooks`, and current tests show host config is the only allow path. I did not run a live plugin in this read-only pass.

## Solution Assessment

Unclear as written. Manifest metadata fits the plugin owner boundary, but the best solution needs maintainer-approved request/grant and host override semantics so plugins cannot self-grant raw conversation access accidentally.
