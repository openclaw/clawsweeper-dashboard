## Summary

Keep open. Current main has the host-config `allowConversationAccess` schema fix, but it still has no native plugin-manifest `permissions.conversationAccess` contract, and https://github.com/openclaw/openclaw/pull/72690 is the open unmerged implementation candidate that references this issue with closing syntax.

## What This Changes

Request to add a native plugin manifest field such as `permissions.conversationAccess` so plugins can declare raw conversation hook needs without patching host config schemas.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72690 or an equivalent narrow patch that adds request-only manifest metadata, docs, and tests while preserving explicit host config as the effective grant.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a broken documented behavior. Source inspection still gives high confidence that current main lacks the requested manifest path: the manifest has no permissions field and the hook gate only receives `entry?.hooks`.

## Solution Assessment

No for the original sketch if manifest `true` grants raw conversation access by itself. The safer maintainable direction is manifest request/disclosure metadata plus explicit operator approval through host config, which the open implementation candidate attempts but has not merged.
