## Summary

Keep open: current main still has host-config-only conversation-hook authorization and no native manifest `permissions.conversationAccess` contract. The linked implementation PR is open and conflicting, so this should remain the product/security tracking issue until maintainers land or replace that work.

## What This Changes

Requests adding plugin manifest `permissions.conversationAccess` so plugins can declare conversation-hook access without maintaining parallel schema patches, while keeping backward compatibility for the existing host config flag.

## Best Possible Solution

Review and land or replace the linked PR with a narrow request-only manifest metadata contract that preserves explicit host approval unless maintainers deliberately approve manifest-granted access.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-bundled plugin with only manifest `permissions.conversationAccess` would still receive hook policy from host config and be blocked from `agent_end` without `allowConversationAccess=true`.

## Solution Assessment

Unclear. Request-only manifest metadata is safer than manifest self-authorization, but maintainers still need to decide whether that narrower trust model satisfies this RFC.
