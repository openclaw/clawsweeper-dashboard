Continue keeping this open: current main still has the source-backed live heartbeat tool-event leak, and an open PR with closing syntax is already the implementation path rather than a reason to close the issue early.

## What This Changes

The issue reports webchat heartbeat polls leaking visible tool cards and the literal `HEARTBEAT_OK` into chat.

## Best Possible Solution

Review, update if needed, and land the open Gateway fix PR so heartbeat-originated tool events stay out of WebChat while normal tool cards and actionable heartbeat alert text remain visible.

## Reproduction Assessment

Yes, by source inspection: register a Control-UI-visible heartbeat run with a session subscriber or run-scoped tool recipient, emit a tool event, and current main broadcasts it into the same stream that the UI converts into chat tool cards. I did not run a live 30-minute WebChat heartbeat in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is a Gateway projection guard for heartbeat-originated live tool events, paired with regression coverage for both run-scoped and session-scoped subscribers. The already-open fixing PR is the right route to resolve this issue after normal PR review/proof handling.
