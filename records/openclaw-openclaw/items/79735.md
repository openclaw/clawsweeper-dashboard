## Summary

The issue should stay open: current main still has a source-visible path that can broadcast heartbeat-originated tool events into WebChat, and an open linked fix PR already uses closing syntax for this issue.

## What This Changes

Heartbeat polling in WebChat should run silently without visible tool cards or `HEARTBEAT_OK` chat messages.

## Best Possible Solution

Land or reject the linked Gateway fix after real behavior proof, keeping heartbeat tools internal to WebChat while preserving normal tool cards and actionable heartbeat alert text.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register a heartbeat run for a WebChat-visible session, emit a tool event, and the Gateway can still broadcast `agent` or `session.tool` payloads that the Control UI renders as tool cards.

## Solution Assessment

Yes. The narrow maintainable solution is a Gateway projection guard for heartbeat-originated tool events, and the currently open linked PR is the right place to review that fix before closing this issue.
