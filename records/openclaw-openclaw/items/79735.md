## Summary

Keep open: current main still lets webchat heartbeat output escape when `showOk` is true and still forwards heartbeat tool events to Control UI; an open closing PR exists but is not merged and appears focused on the `HEARTBEAT_OK` text path.

## What This Changes

The issue asks for webchat heartbeat polls to stay silent instead of showing tool calls or rendering `HEARTBEAT_OK` as chat content.

## Best Possible Solution

Land a narrow Gateway/Control UI fix that suppresses heartbeat-originated live chat text and tool-card projections for webchat while preserving external-channel `showOk` delivery semantics.

## Reproduction Assessment

Yes, by source inspection: set `channels.defaults.heartbeat.showOk: true`, run a webchat/internal heartbeat, and current main returns false from the chat-output suppression helper while still broadcasting heartbeat tool events to session subscribers. I did not run a live browser reproduction in this read-only sweep.

## Solution Assessment

Yes, keeping this issue open is the right state while https://github.com/openclaw/openclaw/pull/79786 is reviewed or expanded. The final fix should cover both the `HEARTBEAT_OK` text path and the live tool-card path rather than only changing history rendering.
