## Summary

Keep open for maintainer review: the patch addresses a real wrong-room keyless cron delivery path and has sufficient Matrix proof, but it intentionally changes shipped fallback behavior by refusing keyless or channel-only implicit crons and the broader restart replay issue remains open.

## What This Changes

The PR makes implicit cron delivery return `ok:false` when the target is inherited only from the shared agent-main `lastTo`, threads `sessionTarget` into delivery resolution, and cleans `deleteAfterRun` sessions on refused delivery.

## Best Possible Solution

Land the resolver-level refusal only after maintainers accept the fail-closed compatibility policy, while keeping broader startup delivery-queue and Matrix replay hardening on the canonical issue.

## Reproduction Assessment

Yes for the source path: current main and v2026.6.6 fall back to the main session entry for implicit last-target delivery, and the PR body supplies after-fix Matrix/Synapse proof. I did not rerun the live Matrix scenario in this read-only review.

## Solution Assessment

Yes as a narrow mitigation if maintainers accept fail-closed behavior; the resolver is the shared boundary for dispatch, preview, and failure notifications. If compatibility for existing keyless crons must remain the default, the safer alternative is persisted creator-session provenance plus a migration design.
