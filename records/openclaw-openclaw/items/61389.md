## Summary

Keep open. Current main and the latest release already solve the stale-threshold half with interval-relative validation, but `channelMaxRestartsPerHour: 0` is still rejected and its intended meaning remains a maintainer-owned gateway config contract decision.

## What This Changes

The issue asks to tighten gateway config validation by rejecting stale thresholds below the health-check interval and allowing `channelMaxRestartsPerHour: 0` as a no-limit or disabled value.

## Best Possible Solution

Keep the interval-relative stale-threshold rule, then define the zero-value restart-cap contract and update schema, runtime behavior, docs/help, generated metadata, and focused tests together.

## Reproduction Assessment

Yes for source-level reproduction of the remaining behavior: current main rejects `gateway.channelMaxRestartsPerHour: 0`, while the runtime would treat an accepted zero as zero allowed restarts. The stale-threshold part no longer reproduces because current validation rejects thresholds below the effective health-check interval.

## Solution Assessment

No, the literal schema-only suggestion is not the best fix. A static `.min(5)` is less precise than the current cross-field stale-threshold rule, and `.min(0)` needs a maintainer-approved zero-value contract plus runtime/docs/test updates.
