## Summary

Keep open. Current main already handles the stale-threshold half with interval-relative validation, but `channelMaxRestartsPerHour: 0` is still rejected and its intended public meaning needs a maintainer config-contract decision.

## What This Changes

The issue asks OpenClaw to tighten gateway config validation for `channelStaleEventThresholdMinutes` and allow `channelMaxRestartsPerHour: 0`.

## Best Possible Solution

Keep the interval-relative stale-threshold rule, then define the zero-value restart-cap contract and update schema, generated metadata, docs/help, runtime behavior, and focused tests together.

## Reproduction Assessment

Yes for source-level reproduction of the remaining behavior: current main rejects `gateway.channelMaxRestartsPerHour: 0`. The stale-threshold part no longer reproduces because current validation rejects thresholds below the effective health-check interval.

## Solution Assessment

No; the literal schema-only suggestion is not the best fix. A static `.min(5)` is less precise than the current cross-field stale-threshold rule, and `.min(0)` needs maintainer-approved zero-value semantics plus runtime/docs/test updates.
