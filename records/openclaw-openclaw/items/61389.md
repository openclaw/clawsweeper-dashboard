## Summary

Keep open. Current main already handles the stale-threshold safety concern with cross-field validation against the effective health-check interval, but `gateway.channelMaxRestartsPerHour: 0` is still rejected and the requested zero semantics conflict with the runtime cap behavior.

## What This Changes

The issue asks to change gateway config validation so stale-event thresholds cannot be lower than the health-check cadence and the max-restarts-per-hour setting can accept `0`.

## Best Possible Solution

Keep this issue as the canonical tracker, preserve the interval-relative stale-threshold validation, and make an explicit maintainer decision for `channelMaxRestartsPerHour: 0` before updating the Zod schema, generated schema, docs/help text, and focused config/runtime tests.

## Reproduction Assessment

Yes for the remaining behavior: source inspection of current main shows `gateway.channelMaxRestartsPerHour: 0` is rejected by both Zod and generated schema minimums. The stale-threshold half no longer reproduces as originally framed because current validation already rejects thresholds below the effective health-check interval.

## Solution Assessment

No, the literal suggested fix is not the best way to solve the full issue. The maintainable path is to keep the cross-field stale-threshold rule and separately define whether a zero restart cap means zero allowed restarts or unlimited restarts, then align schema, runtime, generated artifacts, docs, and tests.
