## Summary

Keep open. Current main already handles the stale-threshold side with interval-relative validation, but the restart-cap side remains a config-contract decision: `channelMaxRestartsPerHour: 0` is still rejected, and accepting it without runtime/docs changes would mean zero allowed restarts, not unlimited restarts.

## What This Changes

The issue requests changing gateway config validation so stale-event thresholds cannot be below the health-check interval and restart caps can use zero as an unlimited value.

## Best Possible Solution

Preserve the current interval-relative stale-threshold rule, then make an explicit maintainer decision for zero restart-cap semantics before updating schema, runtime behavior, docs/help, generated config metadata, and focused tests together.

## Reproduction Assessment

Yes for source-level reproduction of the remaining issue: current main rejects `gateway.channelMaxRestartsPerHour: 0`, and the runtime would treat an accepted zero as zero allowed restarts. The stale-threshold part no longer reproduces because current validation rejects thresholds below the effective health-check interval.

## Solution Assessment

No, the literal suggested schema-only fix is incomplete. A static `.min(5)` is less precise than the current cross-field stale-threshold rule, and `.min(0)` needs matching runtime, docs/help, generated config metadata, and tests after the zero-value contract is decided.
