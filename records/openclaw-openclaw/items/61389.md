## Summary

Keep open. Current main already solves the stale-threshold floor with interval-relative validation, but the restart-cap half remains unresolved because `channelMaxRestartsPerHour: 0` is still rejected and the requested zero-value meaning conflicts with current runtime behavior.

## What This Changes

The issue requests gateway config validation changes so stale thresholds cannot be shorter than the health-check interval and `channelMaxRestartsPerHour` accepts zero.

## Best Possible Solution

Preserve the current interval-relative stale-threshold rule, then make an explicit maintainer decision for zero restart-cap semantics before updating schema, runtime behavior, docs/help, generated config metadata, and focused tests together.

## Reproduction Assessment

Yes for source-level reproduction of the remaining issue: current main rejects `gateway.channelMaxRestartsPerHour: 0` in Zod, and runtime would treat an accepted zero as zero allowed restarts. The stale-threshold half no longer reproduces because current validation rejects thresholds below the effective health-check interval.

## Solution Assessment

No, the literal suggested fix is not the best complete solution. A static `.min(5)` is less precise than the current cross-field stale-threshold rule, and `.min(0)` needs a matching runtime and documentation contract.
