## Summary

Keep open. Current main has already solved the stale-threshold floor with interval-relative validation, but the restart-cap half remains unresolved because `channelMaxRestartsPerHour: 0` is still rejected and the requested zero semantics do not match current runtime behavior.

## What This Changes

The issue asks to change gateway config validation so stale-event thresholds cannot be shorter than health checks and zero is accepted for `channelMaxRestartsPerHour`.

## Best Possible Solution

Keep this as the canonical tracker, preserve the interval-relative stale-threshold validation, and make an explicit maintainer decision for zero restart-cap semantics before updating schema, generated docs, runtime behavior, and focused tests.

## Reproduction Assessment

Yes for the remaining issue: current source inspection shows `gateway.channelMaxRestartsPerHour: 0` is still rejected by Zod and generated schema. The stale-threshold part no longer reproduces as originally described because current validation already rejects thresholds below the effective health-check interval.

## Solution Assessment

No, the literal suggested fix is not the best complete solution. A static `.min(5)` is less precise than the current cross-field stale-threshold rule, and accepting `.min(0)` needs a matching runtime contract for whether zero means unlimited or disabled.
