## Summary

Keep open: the linked memory-wiki zero-artifact bug is real, but this PR is not merge-ready because the fallback bypasses the registered memory capability/activation boundary, reaches into another plugin's private source, has a strict type-contract issue, and only supplies unit-test proof.

## What This Changes

The branch adds a memory-wiki helper that falls back from the registered memory public-artifacts capability to a direct memory-core artifact import, then routes bridge import and status through that helper.

## Best Possible Solution

Make bridge status/import see the active public-artifacts provider through the existing memory capability or a public SDK lifecycle seam, preserving explicit disablement and other memory-plugin behavior, then prove it with a redacted real CLI run.

## Reproduction Assessment

Yes at source level: current main's bridge/status paths depend on `listActiveMemoryPublicArtifacts`, which returns empty when no memory capability provider is registered, and snapshot loads restore prior memory state. I did not run the live Gateway CLI flow in this read-only review.

## Solution Assessment

No. The proposed direct memory-core fallback is not the best fix because it bypasses plugin activation and private-extension boundaries; the narrower maintainable path is a public active-memory artifact lifecycle or SDK seam that preserves disablement semantics.
