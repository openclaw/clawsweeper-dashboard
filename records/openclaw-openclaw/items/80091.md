## Summary

Keep open. Current main still has the Slack outbound route split described here, and the issue is now paired with an open fixing PR, so it should remain open until that PR lands or is closed.

## What This Changes

The issue reports that Slack outbound delivery mirrors can create `slack:channel:<d...>` sessions for native IM channel IDs instead of canonical `slack:direct:<u...>` sessions.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80111, or an equivalent Slack-owned resolver fix, then close this issue after merge; keep legacy session repair as a separate follow-up if it grows broader.

## Reproduction Assessment

Yes. From source inspection, a `message.send` target like `D...` reaches Slack outbound session resolution, is parsed as a channel, skips the `G...` metadata reclassification branch, and can build a channel-scoped mirror route; I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Slack plugin outbound route resolver and Slack conversation metadata helper, with tests proving `D...` and `channel:D...` targets cannot create `slack:channel:d...` mirror sessions while preserving channel and MPIM behavior.
