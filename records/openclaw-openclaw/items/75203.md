## Label Justifications

- `P1`: The issue can route real WebChat user input into a heartbeat-isolated background session and make the intended conversation appear clobbered or silent.
- `impact:session-state`: The report is about selecting and writing to the wrong session identity when a heartbeat-isolated row is present.
- `impact:message-loss`: Messages sent through the heartbeat row can disappear from the user's expected primary WebChat conversation.

## Summary

Keep open: current main and the latest release still allow a heartbeat-isolated `:heartbeat` row to flow through the normal session list/picker path, and WebChat sends still target the selected session key and current session id. No merged related item owns this exact Control UI write-into-heartbeat failure.

## What This Changes

The issue reports Control UI webchat selecting and sending user input into a `:heartbeat` session when the primary `agent:main:main` session has been pruned.

## Best Possible Solution

Define heartbeat-isolated rows as background/read-only session state, ensure the primary WebChat session is present or preferred, and add guards so ordinary WebChat sends cannot write into `:heartbeat` rows.

## Reproduction Assessment

Yes, source-level: current main creates marked `:heartbeat` session rows, does not filter them from `sessions.list` or the Control UI picker, and sends WebChat input to the selected key. I did not run a live browser/pruned-store reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the background-session contract. A maintainable fix likely needs both selection behavior and a server-side `chat.send` guard or redirect, not only a cosmetic picker label change.
