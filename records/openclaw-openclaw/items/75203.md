## Label Justifications

- `P1`: A real Control UI workflow can route active WebChat user input into an unintended background session and disrupt the conversation.
- `impact:session-state`: The selected Control UI chat state can drift from the intended primary session into a synthetic heartbeat-isolated session.
- `impact:message-loss`: Messages sent to the heartbeat session can appear missing from the primary conversation and be superseded by later heartbeat state changes.

## Summary

Keep open: current main and the latest release still allow heartbeat-isolated session rows through listing, UI selection, and WebChat send routing, and no merged or viable related PR supersedes the Control UI routing problem.

## What This Changes

The issue requests preventing Control UI webchat from routing user messages into `:heartbeat`-suffixed heartbeat-isolated sessions when the primary main session row has been pruned.

## Best Possible Solution

Define heartbeat-isolated sessions as background state, keep or recreate the canonical primary WebChat session, and prevent ordinary WebChat sends from targeting heartbeat rows.

## Reproduction Assessment

Yes: current main source shows a heartbeat-isolated row can be persisted, listed, selected, preserved as the current non-alias session, and sent to unchanged by WebChat.

## Solution Assessment

Unclear: the durable fix should cover picker/default behavior and send-routing protection, but maintainers need to choose the permanent UI contract for background heartbeat sessions first.
