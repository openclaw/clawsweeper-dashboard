## Summary

Keep open: current main still appears to expose heartbeat-isolated session rows as selectable chat sessions and sends WebChat input to the selected session key, so the reported wrong-session routing path remains source-reproducible. The existing maintainer-review/product-decision labels are appropriate because the fix needs a policy choice about whether heartbeat rows should be hidden, read-only, redirected, or paired with primary-session recreation.

## What This Changes

The issue reports that Control UI webchat can select and send user input to a heartbeat-isolated session after the primary main session is pruned.

## Best Possible Solution

Define heartbeat-isolated sessions as background state in the Control UI/session contract, keep or recreate the canonical primary WebChat row, and prevent ordinary user sends from targeting `:heartbeat` rows.

## Reproduction Assessment

Yes from source inspection: a surviving heartbeat-isolated row can be listed and selected, and WebChat sends to the selected `sessionKey`. I did not perform a live pruned-session browser reproduction.

## Solution Assessment

Unclear until maintainers choose the heartbeat-session UI contract. The maintainable end state should keep the primary session writable by default and stop background heartbeat sessions from receiving normal WebChat input.
