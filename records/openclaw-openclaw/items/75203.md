## Label Justifications

- `P1`: The issue can misroute real webchat user messages into a background heartbeat session and disrupt the visible chat workflow.
- `impact:message-loss`: User-visible chat input can be delivered to the wrong session target instead of the intended main webchat conversation.
- `impact:session-state`: Conversation history and active UI selection can drift into an isolated heartbeat session after pruning removes the main row.

## Summary

Keep open: current main still appears to expose isolated heartbeat sessions as normal selectable chat sessions and routes `chat.send` to the selected key, while the closed Slack PR that mentioned this issue did not touch this surface.

## What This Changes

Issue #75203 asks OpenClaw to keep Control UI webchat user input on the main session rather than an isolated `:heartbeat` session after pruning removes the main row.

## Best Possible Solution

Define isolated heartbeat sessions as background/read-only for Control UI, make webchat fall back to the base main session when selected or missing, and harden send routing so ordinary user input cannot accidentally target the heartbeat row.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: current main still creates heartbeat-isolated `:heartbeat` rows, lists/preserves them as selectable active-agent sessions, and sends to the requested session key; no live browser run was performed.

## Solution Assessment

No current fix is already present; the best solution needs a maintainer decision on the UI/gateway boundary before implementing a narrow repair.
