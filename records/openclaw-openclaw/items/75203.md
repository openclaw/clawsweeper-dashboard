## Label Justifications

- `P1`: The issue can route real WebChat user input into a heartbeat-isolated background session and make the primary conversation appear clobbered or silent.
- `impact:session-state`: The report is about selecting and writing to the wrong session identity when a heartbeat-isolated row exists.
- `impact:message-loss`: Messages sent through the wrong heartbeat session can disappear from the user's expected primary WebChat conversation.

## Summary

Keep open: current main and the latest inspected release still expose heartbeat-isolated session rows through the normal session path, and WebChat send routing can target the selected `:heartbeat` session key/session id, so the reported route-into-heartbeat state remains possible.

## What This Changes

Control UI should keep ordinary WebChat input bound to the primary session and prevent heartbeat-isolated sessions from being selected or written as normal chat sessions.

## Best Possible Solution

Define heartbeat-isolated sessions as background state, ensure the primary WebChat session is present/preferred, and reject or redirect ordinary WebChat sends away from `:heartbeat` rows with regression coverage.

## Reproduction Assessment

Yes, source-level: a stored `agent:main:main:heartbeat` row is not filtered from `sessions.list` or the picker, and `chat.send` routes to the selected key and current session id; I did not run a live browser repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the background-session contract; the maintainable fix should make heartbeat-isolated sessions non-writable from ordinary WebChat and restore or prefer the canonical primary session.
