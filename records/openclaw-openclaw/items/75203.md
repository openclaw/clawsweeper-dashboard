## Summary

Keep open: current main and the latest release still expose heartbeat-isolated session rows to the Control UI chat picker and send WebChat input to the selected session key. The report is source-reproducible, but the durable fix needs a maintainer policy decision for background heartbeat rows and primary-session recreation.

## What This Changes

The issue reports that after pruning removes the primary WebChat session, the Control UI can select a `:heartbeat` session and route user input into that background session.

## Best Possible Solution

Define heartbeat-isolated sessions as background state in the session/UI contract, keep or recreate the canonical primary WebChat session, and prevent ordinary WebChat sends from targeting `:heartbeat` rows.

## Reproduction Assessment

Yes for source reproduction: a surviving heartbeat-isolated row can be listed and selected, explicit non-alias selections are preserved, and WebChat sends to the selected `sessionKey`. I did not perform a live pruned-session browser reproduction.

## Solution Assessment

Unclear until maintainers choose the heartbeat-session UI contract. The maintainable solution should keep the primary session writable by default and stop background heartbeat sessions from receiving normal WebChat input.
