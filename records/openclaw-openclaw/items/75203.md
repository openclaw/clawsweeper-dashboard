## Summary

Keep open. Current main still lets heartbeat-isolated session rows flow into the Control UI picker and sends WebChat input to the selected session key directly, so the reported routing bug remains. The issue is also paired with open closing PR #75356, which should be reviewed, landed, or replaced before this issue is closed.

## What This Changes

The issue asks Control UI/WebChat to avoid selecting heartbeat-isolated sessions and to route stale heartbeat-session sends back to the primary base session after pruning.

## Best Possible Solution

Land or replace the narrow marker-backed fix: project `heartbeatIsolatedBaseSessionKey` or an equivalent synthetic-session flag through `sessions.list`, hide or normalize heartbeat-isolated rows in the Control UI picker, and guard `chat.send` so stale heartbeat selections route to the base session while preserving real user sessions that happen to end in `:heartbeat`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by listing only `agent:main:main:heartbeat` after the base row is pruned: current main adds that row to the picker and sends WebChat input with `sessionKey: state.sessionKey`.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix if review confirms the details: use the stored heartbeat-isolated marker rather than suffix-only matching, normalize stale WebChat sends back to the base session, and add focused UI/gateway regression tests.
