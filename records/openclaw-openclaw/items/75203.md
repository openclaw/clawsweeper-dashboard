## Label Justifications

- `P1`: The reported Control UI workflow can route real user input into a background heartbeat session and disrupt the visible conversation.
- `impact:session-state`: The issue is about chat state drifting from the intended primary session into a heartbeat-isolated session.
- `impact:message-loss`: User messages can be misrouted to the wrong session target and appear lost when the visible session changes.

## Summary

Keep open: current main still lets heartbeat-isolated session rows appear as normal chat session options and routes WebChat sends to the selected key, so the reported wrong-session input path remains possible.

## What This Changes

The issue requests that Control UI WebChat keep normal user input bound to the primary `agent:main:main` session and prevent `:heartbeat` isolated sessions from becoming writable chat targets.

## Best Possible Solution

Define heartbeat-isolated sessions as background state in the Control UI contract, keep or recreate the canonical primary WebChat row, and prevent ordinary user sends from targeting `:heartbeat` rows.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: a surviving heartbeat-isolated row can be listed and selected, and WebChat sends to the selected `sessionKey`; I did not perform a live pruned-session browser reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the heartbeat-session UI contract; the narrow fix should still keep the primary session writable by default and stop background heartbeat sessions from receiving normal WebChat input.
