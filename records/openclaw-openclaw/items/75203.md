## Label Justifications

- `P1`: A real Control UI WebChat workflow can route active user input into an unintended background session and disrupt the conversation.
- `impact:session-state`: The selected WebChat session can drift from the intended primary session into a heartbeat-isolated session row.
- `impact:message-loss`: Messages sent to the heartbeat row can appear missing from the primary conversation and later be hidden or superseded by heartbeat state churn.

## Summary

Keep open: current main and the latest release still let heartbeat-isolated rows be persisted, listed, selected, preserved as a non-alias Control UI session, and sent to unchanged by WebChat, with no merged related item superseding the reported routing bug.

## What This Changes

The issue requests that Control UI WebChat avoid selecting and sending user input to heartbeat-isolated `:heartbeat` sessions when the primary session has been pruned.

## Best Possible Solution

Define heartbeat-isolated sessions as background state, keep or recreate the canonical primary WebChat session, and prevent ordinary WebChat sends from targeting heartbeat rows.

## Reproduction Assessment

Yes, at source level: current main can persist, list, select, preserve, and send WebChat input to a heartbeat-suffixed session key; I did not run a live pruned-store browser reproduction.

## Solution Assessment

Unclear: the narrow fix should cover picker/default behavior and send-routing protection, but maintainers need to choose whether heartbeat rows are hidden, read-only, redirected, rejected, or otherwise combined.
