## Summary

Keep open. Current main still exposes heartbeat-isolated session rows as ordinary selectable Control UI sessions and sends WebChat input to the active session key directly; open PR #75356 is the linked implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that Control UI/WebChat avoid selecting or sending user input into heartbeat-isolated `:heartbeat` sessions when the base `main:main` session has been pruned.

## Best Possible Solution

Land or replace the narrow marker-backed fix: project `heartbeatIsolatedBaseSessionKey` or an equivalent synthetic-session flag through the gateway session-list contract, hide or normalize heartbeat-isolated rows in the Control UI picker, and guard `chat.send` so stale heartbeat selections route to the base session while preserving real user sessions that merely end in `:heartbeat`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by listing only `agent:main:main:heartbeat` after the base row is pruned: current main adds that row to the picker and sends WebChat input with `sessionKey: state.sessionKey`.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because it uses the stored heartbeat-isolated marker rather than suffix-only matching, normalizes stale sends to the base session, and can be covered with focused UI/gateway tests.
