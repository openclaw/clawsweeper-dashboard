## Label Justifications

- `P1`: The reported workflow can route real WebChat user input into a background heartbeat session and disrupt the visible conversation.
- `impact:session-state`: The issue is about selected conversation state drifting from the intended main session into a heartbeat-isolated session.
- `impact:message-loss`: User messages can be misdelivered to the wrong session target from the chat UI.

## Summary

Keep open: current main still lets heartbeat-isolated session rows reach the Control UI chat picker and routes WebChat sends to the selected key, so the reported misrouting remains a real session-state/message-delivery risk.

## What This Changes

The issue requests that WebChat avoid selecting or sending user input to `:heartbeat` isolated sessions when the primary main session has been pruned.

## Best Possible Solution

Define heartbeat-isolated sessions as background/read-only, keep or recreate the canonical main WebChat row, and prevent ordinary user sends from targeting the suffixed heartbeat row.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully: the current source path supports the report, but no live Control UI run with aggressive pruning was performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the issue identifies the right boundary problem, but maintainers still need to choose whether the fix should live in UI filtering, Gateway send policy, session maintenance, or a combination.
