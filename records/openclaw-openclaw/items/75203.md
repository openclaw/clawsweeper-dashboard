Keep open: current main still exposes the reported source-level path, and the previously linked pull request is now closed unmerged without changing the webchat heartbeat-session surface.

## What This Changes

The issue reports that Control UI webchat can select and send into a synthetic isolated heartbeat session after the primary main session is pruned.

## Best Possible Solution

Project the synthetic heartbeat base marker through the gateway session-list contract, keep heartbeat-owned rows out of normal writable picker choices, and normalize stale heartbeat sends back to the base main session without treating arbitrary `:heartbeat` keys as synthetic.

## Reproduction Assessment

Yes, at source level: seed the session list with only `agent:main:main:heartbeat`, and current main will offer that row in the picker and pass it unchanged to `chat.send`. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes. A marker-backed fix is the narrow maintainable path because storage already records `heartbeatIsolatedBaseSessionKey`, avoiding brittle suffix-only handling for legitimate session keys that happen to end in `:heartbeat`.
