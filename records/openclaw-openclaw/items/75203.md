## Summary

Keep open: current main still exposes the reported source-level path, and the only closing-looking reference is an unrelated closed Slack PR/commit that did not change the webchat heartbeat-session picker or send binding.

## What This Changes

The issue requests that Control UI keep webchat input bound to the primary main session and keep isolated heartbeat sessions out of writable session selection when pruning removes the base row.

## Best Possible Solution

Project the stored heartbeat-isolated base marker through the gateway session-list contract, keep marked heartbeat-owned rows out of normal writable picker choices, and normalize stale marked heartbeat sends back to the base main session while preserving real user sessions whose keys happen to end in ':heartbeat'.

## Reproduction Assessment

Yes, at source level: seed the session list with only a row keyed `agent:main:main:heartbeat`, and current main will offer that row in the picker and pass it unchanged to `chat.send`. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes. A marker-backed fix is the narrow maintainable path because storage already records `heartbeatIsolatedBaseSessionKey`, avoiding brittle suffix-only handling for legitimate session keys that happen to end in `:heartbeat`.
