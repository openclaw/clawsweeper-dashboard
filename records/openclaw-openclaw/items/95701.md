## Summary

Keep open: current main still keeps generated media tasks in a blocking `running` state while completion delivery awaits the requester turn, and the open candidate PR has not merged yet.

## What This Changes

The issue reports sequential `video_generate` work being delayed because generated media tasks remain `running` until the completion wake returns.

## Best Possible Solution

Land a focused fix that makes generated-but-delivering media non-blocking for distinct follow-up video requests while preserving delivery-confirmed terminal closeout and same-request duplicate protection.

## Reproduction Assessment

Yes. The source path shows a session-backed `video_generate` task can stay `running` while `wakeTaskCompletion` awaits an `expectFinal` requester turn, and the session-wide prompt guard treats that task as active.

## Solution Assessment

Yes, the best direction is a typed generated/delivering phase or equivalent state boundary, not completing the task before delivery confirmation or relying on progress-summary string matching. Current main lacks that fix, and the open candidate PR should be reviewed as the implementation path.
