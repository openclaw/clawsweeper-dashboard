## Summary

Keep open: current main and v2026.6.9 still keep generated media tasks active until the requester wake returns, while the active-task guard treats queued/running media tasks as blocking. The report is source-backed, but the fix should preserve the documented same-session video concurrency contract while removing the generated-but-still-running stall.

## What This Changes

The issue reports that completed `video_generate` background tasks remain marked `running` during completion delivery, delaying sequential video-production workflows in the same session.

## Best Possible Solution

Add an explicit generated/delivering phase or equivalent media-ready marker that stops reporting provider work as `running` and unblocks only the post-generation delivery window, while preserving delivery-failure safety and the documented active-task guard before media is ready.

## Reproduction Assessment

Yes, by source inspection: current main records generated media progress, awaits an `expectFinal` requester wake, and only then marks the task terminal, so the task stays active during delivery. I did not run a live provider reproduction in this read-only cleanup pass.

## Solution Assessment

Unclear. The intermediate-state direction is plausible, but maintainers should decide the durable task-state semantics because current docs intentionally block same-session video generation while a task is active.
