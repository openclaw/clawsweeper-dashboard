## Summary

Keep open: current main and the latest release still keep generated media tasks in `running` until the completion wake finishes, and no merged or open related PR owns this exact delayed terminal-state bug.

## What This Changes

The issue requests fixing session-backed `video_generate` completion ordering so generated-but-delivering media stops blocking subsequent distinct video requests while delivery confirmation still determines the final task outcome.

## Best Possible Solution

Add a typed generated-but-delivering phase or equivalent maintainable state so provider-finished media no longer blocks distinct follow-up video work, while preserving delivery-confirmation terminal outcome handling and same-request duplicate protection.

## Reproduction Assessment

Yes, by source inspection rather than a live provider rerun: session-backed `video_generate` can remain `running` while `wakeTaskCompletion` awaits an `expectFinal` requester turn, and both prompt/tool guards treat that active session task as blocking.

## Solution Assessment

The suggested direction is mostly right: do not move `completeTaskRun` before delivery confirmation, but introduce a real non-blocking generated/delivery phase instead of relying on the existing `running` state or a fragile progress-summary string match.

## Repair Work Prompt

Fix the session-backed `video_generate` delayed terminal-state bug described in https://github.com/openclaw/openclaw/issues/95701. Current main records `Generated media; delivering completion`, awaits the requester completion wake, and only then completes the task; during that window prompt/tool guards still treat the task as active `running`. Implement a maintainable generated-but-delivering/nonblocking phase or equivalent typed state so provider-finished media no longer suppresses a distinct next `video_generate` request in the same session, while same-request duplicate guards and genuinely queued/provider-running work remain blocked. Preserve the existing rule that final `succeeded` versus blocked terminal outcome is derived after delivery confirmation; do not simply complete the task before the wake. Avoid adding a config option or changelog entry. Update focused tests and docs only if user-visible task/guard semantics change.
