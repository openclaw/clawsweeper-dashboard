## Summary

Keep open: current `main` and the latest release still keep generated media tasks blocking as `running` while completion delivery awaits an `expectFinal` requester turn, and no related PR clearly owns this exact video-generation deadlock.

## What This Changes

The issue reports that sequential session-backed `video_generate` calls are blocked for minutes after media is ready because task completion waits for the requester wake before marking the task terminal.

## Best Possible Solution

Add a typed generated-but-delivering phase or equivalent non-blocking state so finished media no longer blocks distinct follow-up video work while same-request duplicate protection and delivery-confirmed terminal outcomes remain intact.

## Reproduction Assessment

Yes by source inspection, not by a live provider rerun: current `main` leaves the media task `running` while awaiting an `expectFinal` completion wake, and the active-task prompt/tool guards treat that session task as blocking.

## Solution Assessment

Yes in direction: the fix should preserve delivery-confirmation terminal handling and introduce a real non-blocking generated/delivery phase rather than moving `completeTaskRun` early or matching progress text.
