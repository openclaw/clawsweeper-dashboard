## What This Changes

The issue requests fixing session-backed `video_generate` tasks that stay `running` during completion delivery and block sequential video generation in the same session.

## Best Possible Solution

Add a maintainable generated-but-delivering phase or equivalent non-blocking task state so media-ready video tasks stop blocking distinct follow-up requests while same-request duplicate protection and delivery-confirmed terminal outcomes remain intact.

## Reproduction Assessment

Yes, by source inspection: current main keeps the task `running` until the `expectFinal` completion wake returns, while prompt/tool guards treat any queued/running session video task as active. I did not run a live provider reproduction in this read-only pass.

## Solution Assessment

Yes with one caveat: the best fix is a typed generated/delivery phase or equivalent state, not moving `completeTaskRun` before delivery confirmation and not relying on a fragile progress-summary string hack.
