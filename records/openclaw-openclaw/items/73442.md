## Summary

Keep open: this protected draft maintainer PR is still useful, but the proposed rename helper has a P2 symlink-alias correctness and security blocker, and current main does not already provide the new helper surface.

## What This Changes

The PR extends fs-safe and the pinned Python helper with root-scoped stat, readdir, and rename operations, adds focused fs-safe tests, and updates the changelog/plugin SDK baseline hash.

## Best Possible Solution

Keep the fs-safe helper direction, but make rename operate on the requested final directory entries or fail closed on final symlinks, then rerun targeted tests/typechecks before maintainer merge review.

## Reproduction Assessment

Not applicable for current main: this is a feature PR adding helper APIs. The blocker is source-reproducible from the proposed rename path because strict resolution follows final symlinks before the pinned rename helper runs.

## Solution Assessment

No: centralizing stat, readdir, and rename in fs-safe is maintainable, but the proposed rename should preserve or reject final symlink directory entries instead of resolving them to targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve final symlinks during rename:** `src/infra/fs-safe.ts:907-914`
  - body: Both rename endpoints are resolved with the strict boundary policy before the pinned helper runs. Strict resolution follows an in-root final symlink, so renaming `link` moves the target while leaving `link` behind, and overwrite destinations can clobber a symlink target instead of replacing the requested entry. Preserve final symlink endpoints or reject them before calling the helper.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive filesystem helpers, and rename currently follows final symlink aliases inside the root.

Concerns:

- **[medium] Rename follows final symlink aliases:** `src/infra/fs-safe.ts:907`
  - body: `renamePathWithinRoot` pre-resolves source and destination with strict boundary resolution, so final symlinks that point inside the root are followed before the pinned rename runs. That can redirect a rename or overwrite to the symlink target rather than the requested directory entry.
  - confidence: 0.82
