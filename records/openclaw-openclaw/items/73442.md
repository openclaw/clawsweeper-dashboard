## Summary

Keep open: the latest draft head fixes the earlier pinned stat identity and Stats predicate issues, but the new bounded helpers still have string-path fallback paths that can escape the root after a race when the Python helper is unavailable or on Windows.

## What This Changes

The PR adds root-scoped fs-safe stat, directory listing, and same-root rename helpers, extends the pinned Python helper, adds fs-safe coverage, updates plugin SDK baseline hashes, and adds a changelog entry.

## Best Possible Solution

Keep the pinned-helper design, but make stat, readdir, and rename fallback paths as bounded as the primary helper or fail closed, add fallback race regression coverage, then rerun the focused fs-safe and type gates.

## Reproduction Assessment

Yes, at source level. Mock `runPinnedPathHelper` to swap an in-root stat/list/rename target or parent to an outside symlink and reject with a spawn error; the fallback paths then operate on stale pre-resolved path strings.

## Solution Assessment

No. The pinned-helper direction and latest primary stat hardening are good, but the helper-unavailable and Windows paths need equivalent root pinning or a fail-closed behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Revalidate fallback stat results:** `src/infra/fs-safe.ts:491-497`
  - body: `statPathWithinRoot` compares identity only for the pinned-helper result. If the helper cannot spawn, a final target can be swapped to an outside symlink after the earlier `realpath`; the fallback `fs.stat(resolved.canonicalPath)` can then return outside metadata while the stale in-root `realPath` still passes the final root check.
  - confidence: 0.86
- **[P2] Keep fallback directory listings pinned:** `src/infra/fs-safe.ts:561-562`
  - body: `readdirWithinRoot` falls back to `fs.readdir(resolved.resolved)` after the bounded precheck. In helper-unavailable paths, a directory component can be swapped to an outside symlink between resolution and fallback, so the new bounded list helper can return entries from outside root.
  - confidence: 0.84
- **[P2] Avoid legacy rename after bounded precheck:** `src/infra/fs-safe.ts:948-950`
  - body: `renamePathWithinRoot` can fall back to the legacy `fs.rename` implementation after resolving `from` and `to`. A raced source or destination parent symlink after that precheck can make the fallback rename operate outside the root; keep rename fd-pinned or fail closed when no safe helper is available.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff still has concrete filesystem-boundary concerns in the new helper-unavailable and Windows fallback paths.

Concerns:

- **[medium] Stat fallback can leak outside metadata:** `src/infra/fs-safe.ts:491`
  - body: After an in-root followed stat target is prechecked, the helper-spawn fallback can follow a raced final symlink to an outside file and return that outside file's metadata without the identity check used by the pinned-helper path.
  - confidence: 0.86
- **[medium] List and rename fallbacks can cross the root:** `src/infra/fs-safe.ts:561`
  - body: The new list and rename helpers fall back to legacy path-string operations after resolution, so a raced symlink parent can redirect those fallback operations outside the root when the pinned helper is unavailable or on Windows.
  - confidence: 0.83
