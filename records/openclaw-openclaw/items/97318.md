## Summary

Keep open: current main preserves ordinary nonzero exec output at the exec runtime layer, but shared downstream error classification still treats a completed nonzero exec result as a tool error, so the reported cron/message-delivery failure is not resolved.

## What This Changes

The issue reports that semantic nonzero subprocess exits from the exec tool are surfaced as framework-level failures, hiding stdout/stderr/exit-code semantics from agent and cron workflows.

## Best Possible Solution

Maintainers should choose the exec contract and then align the shared tool-result classifier, cron fatal-error handling, and docs so ordinary completed exits remain observable without masking real spawn, timeout, signal, and shell 126/127 failures.

## Reproduction Assessment

Yes from source inspection: current main still classifies `{status: "completed", exitCode: 1}` as a tool-result error, while exec runtime itself returns ordinary nonzero exits as completed output. I did not run a live cron reproduction in the read-only checkout.

## Solution Assessment

Unclear as a product contract: the narrow technical fix is to stop treating ordinary completed exec exits as observer errors, but maintainers should decide whether this is the default behavior or an opt-in accepted-exit-code policy.
