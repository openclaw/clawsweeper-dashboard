## Summary

Keep open: current main still has a downstream mismatch where the exec runtime treats ordinary nonzero exits as completed, but the shared tool-result classifier converts any finite nonzero exit code back into tool-error state, matching the reported cron/message-delivery failure mode.

## What This Changes

The issue asks OpenClaw to preserve stdout, stderr, and exit-code semantics for ordinary nonzero `exec` subprocess exits instead of surfacing them as opaque framework tool failures.

## Best Possible Solution

Maintainers should choose the exec contract, then align the shared classifier, embedded observer state, cron handling, and docs so ordinary completed exits remain inspectable while spawn errors, timeouts, signals, and shell 126/127 failures still surface as real failures.

## Reproduction Assessment

Yes, at source level: current main returns completed exec results with `exitCode: 1`, then `isToolResultError` reclassifies that completed result as an error and the embedded handler records it as tool-error state. I did not run a live cron reproduction in this read-only review.

## Solution Assessment

Unclear as a product contract: the narrow technical fix is to stop treating ordinary completed exec exits as observer errors, but maintainers should decide whether this is the default behavior or an opt-in accepted-exit-code policy.
