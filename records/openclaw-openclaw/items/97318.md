## Summary

Keep open: current main still has a source-level mismatch where exec normal nonzero exits are shaped as completed results, but the shared tool-result classifier reclassifies any nonzero exitCode as an error, which can drive lastToolError and cron failure handling.

## What This Changes

The issue requests that exec nonzero subprocess exits be returned as normal stdout/stderr/exit-code results instead of framework-level tool failures, especially for cron agentTurn workflows.

## Best Possible Solution

Align the tool-result error classifier with exec's normal-exit contract while preserving spawn errors, signals, timeouts, and shell 126/127 failures, then add focused regression coverage for exec lifecycle and cron error state.

## Reproduction Assessment

Yes at source level: current main returns exec exitCode 1 as a completed outcome, then isToolResultError reclassifies that completed result as an error. I did not execute the reporter's live agent/cron reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug direction: current runtime and historical PR context already define ordinary nonzero exec exits as completed results, so the maintainable fix is to align lifecycle error classification instead of adding a new config option.
