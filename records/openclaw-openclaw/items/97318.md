## Summary

Keep open: current main only partially solves this. The exec runtime records ordinary nonzero exits as completed, but the shared tool-result classifier still turns the completed nonzero result into observer error state, and isolated cron can still treat that error payload as fatal before delivery.

## What This Changes

The issue requests that exec subprocess nonzero exits remain observable to agents as stdout/stderr plus exit-code semantics instead of being masked as framework-level tool failures.

## Best Possible Solution

Maintainers should choose the exec contract, then align the shared tool-result classifier, cron finalization, and docs so ordinary completed exits remain observable without masking real spawn, timeout, signal, and shell 126/127 failures.

## Reproduction Assessment

Yes, from source inspection: current main can produce `{status: "completed", exitCode: 1}` for exec, then `isToolResultError` converts that same result into observer error state. I did not run a live cron reproduction in this read-only checkout.

## Solution Assessment

Unclear as a product contract: the narrow technical fix is to stop treating ordinary completed exec exits as observer errors, but maintainers should decide whether that is the default behavior or an opt-in accepted-exit-code policy.
