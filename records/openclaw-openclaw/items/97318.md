## Summary

Keep open: current main still turns completed nonzero exec results into tool-error state downstream, so the reported cron/message-delivery problem is not fully solved by the existing exec runtime change.

## What This Changes

The issue asks OpenClaw to preserve stdout/stderr and exit-code semantics for exec subprocesses that complete with ordinary nonzero exit codes, especially inside isolated cron agent turns.

## Best Possible Solution

Maintainers should choose the exec contract and then align the shared tool-result classifier, cron fatal-error handling, and docs so ordinary completed exits remain observable without masking real spawn, timeout, signal, and shell 126/127 failures.

## Reproduction Assessment

Yes from source inspection: current main still classifies `{status: "completed", exitCode: 1}` as a tool-result error, and isolated cron treats unrecovered structured error payloads as fatal. I did not run a live cron reproduction in the read-only checkout.

## Solution Assessment

Unclear as a product contract: the narrow technical fix is to stop treating ordinary completed exec exits as observer errors, but maintainers should decide whether this is the default behavior or an opt-in accepted-exit-code policy.
