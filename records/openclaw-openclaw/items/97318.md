## Summary

Keep open: current main partly preserves nonzero exec exits as completed process outcomes, but the shared tool-result classifier still turns completed nonzero exit codes into error payloads, which isolated cron can record as a failed run and increment error state.

## What This Changes

The issue asks for exec subprocess nonzero exits to be returned with stdout, stderr, and exit-code semantics instead of being treated as framework-level tool failures.

## Best Possible Solution

Align exec error classification with the completed-exit contract, preserving real failures for spawn errors, timeouts, signals, and shell 126/127 while documenting the chosen default or opt-in semantics.

## Reproduction Assessment

Yes from source inspection: completed exec exitCode 1 is still classified as a tool error, and isolated cron converts fatal error payloads into error run state.

## Solution Assessment

Unclear until maintainers choose the compatibility contract; the narrow technical fix is to stop treating ordinary completed exec exits as observer errors while retaining hard-failure paths.
