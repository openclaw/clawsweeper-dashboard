## Label Justifications

- `impact:message-loss`: The reported Slack alert was not posted because the exec result became an error payload instead of usable stdout-driven delivery.

## Summary

Keep open: current main still converts completed nonzero exec results into observer/tool-error state downstream, so the reported cron delivery loss and missing exit-code semantics remain unresolved.

## What This Changes

The issue requests normal stdout/stderr/exit-code visibility for completed nonzero `exec` subprocess exits, or an explicit documented/configured contract for the current behavior.

## Best Possible Solution

Maintainers should choose the exec nonzero-exit contract, then align the shared tool-result classifier, cron fatal-error handling, and docs so semantic exits remain observable without masking spawn, timeout, signal, and shell 126/127 failures.

## Reproduction Assessment

Yes, at source level: current main still classifies `{ status: "completed", exitCode: 1 }` as a tool-result error and propagates that to cron-facing error payloads. I did not run a live cron reproduction in this read-only checkout.

## Solution Assessment

Unclear until maintainers choose the product contract: the narrow technical fix is to stop treating ordinary completed exec exits as observer errors, but the default-versus-opt-in compatibility decision remains open.
