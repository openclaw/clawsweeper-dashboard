## Summary

Keep open. Current main still has OpenAI catalog discovery marked static and has no OpenAI `/models` discovery helper wired into provider catalog resolution; this PR adds a standalone primitive and tests, but it intentionally leaves the linked dynamic-catalog contract and Control UI plumbing to #74481.

## What This Changes

The PR adds an OpenAI discovery module and tests that fetch an OpenAI-compatible `/models` endpoint, map responses into `ModelDefinitionConfig[]`, cache results, and export the helper from the OpenAI plugin barrel.

## Best Possible Solution

Review this alongside #74481, settle the dynamic provider-catalog contract first, then either merge this primitive after small cleanup or fold it into the eventual OpenAI provider hook with SSRF/timeout handling, docs, focused tests, and Control UI/model-list follow-through.

## Reproduction Assessment

Not applicable for the feature request itself; this is an implementation PR, not a bug report. The P3 cache finding has a clear reproduction path by calling the helper with the same key and equivalent base URLs that differ only by trailing slash and observing two fetches.

## Solution Assessment

Unclear as the final product solution: the helper is a plausible first slice, but the best maintainable fix depends on the provider-catalog API decisions still open in #74481. The patch should at least normalize the cache key before merge if maintainers accept this split.
