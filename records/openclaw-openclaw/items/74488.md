## Summary

Keep open. Current main still has no OpenAI `/models` discovery primitive or dynamic OpenAI catalog path, and the linked design issue remains open; this PR is useful source work but needs maintainer catalog-contract review plus a few small pre-merge fixes.

## What This Changes

The branch adds an OpenAI `discoverOpenAIModels` helper and tests, maps OpenAI-compatible `/models` responses into `ModelDefinitionConfig[]`, caches results for one hour, and exports the helper through the OpenAI plugin barrel.

## Best Possible Solution

Settle the provider dynamic-catalog contract in #74481, then land this OpenAI discovery primitive either as a cleaned-up standalone helper or folded into the chosen provider hook with normalized cache keys, no public test reset export, changelog coverage, focused tests, and follow-through for model-list/Control UI behavior.

## Reproduction Assessment

Not applicable for the feature request itself; this is an implementation PR, not a bug report. The cache-key finding has a high-confidence reproduction path by calling the helper with equivalent base URLs that differ only by trailing slash and observing two fetches because the key is computed before normalization.

## Solution Assessment

Unclear as the final product solution. The helper is a plausible first slice, but the best maintainable fix depends on the provider-catalog API decisions still open in #74481; if accepted as a slice, the patch should first address the public API and cache cleanup findings.
