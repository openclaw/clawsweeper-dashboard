## Summary

Keep open. Current main still does not implement OpenAI upstream `/v1/models` discovery, and this PR remains a plausible first slice, but it depends on the open provider-catalog design in #74481 and has one small cache-normalization issue to address before merge.

## What This Changes

The branch adds an OpenAI `/models` discovery helper with cached fail-soft response mapping tests and exports it from the OpenAI plugin API without wiring it into provider catalog resolution.

## Best Possible Solution

Settle #74481's provider catalog API first, then land this discovery logic either behind the chosen OpenAI provider catalog hook or as a private helper used by that hook, with guarded fetch behavior, cache normalization, focused tests, and follow-through for `models list` and the Control UI picker.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection gives high-confidence proof of the current gap: main has static OpenAI catalog metadata and provider augmentation, but no OpenAI `/v1/models` discovery helper or wiring.

## Solution Assessment

Unclear. The primitive is a reasonable shape and mirrors Mantle, but it is not yet the complete maintainable solution because the provider hook, merge semantics, and UI refresh path remain undecided; the cache-key normalization should be fixed if maintainers proceed with this slice.
