## Summary

Keep open. Current main still exposes the OpenCode Claude thinking profile only through the loaded provider runtime, while the directive/session validation path can use the lightweight provider-policy surface where OpenCode has no artifact, matching the reported downgrade to `high`.

## What This Changes

Ensure `opencode/claude-opus-4-7` accepts the Claude Opus 4.7 extended thinking levels instead of downgrading `xhigh` to `high`.

## Best Possible Solution

Add an OpenCode-owned lightweight provider-policy artifact that delegates Claude thinking profile resolution to the shared Claude helper, with regression coverage for the OpenCode route and no core provider-id special case.

## Reproduction Assessment

No executable live reproduction was run in this read-only review. Source inspection gives a high-confidence reproduction path where `opencode/claude-opus-4-7` can fall back to base thinking levels when the OpenCode provider runtime is not active and no policy artifact supplies its Claude profile.

## Solution Assessment

Yes. The narrow maintainable fix is to expose OpenCode's Claude thinking profile through the existing provider-policy surface, rather than adding another core hardcoded OpenCode exception or relying on eager provider activation.
