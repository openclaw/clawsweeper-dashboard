## Summary

Keep open: current main still hardcodes Slack assistant thread status and exposes no config for custom `status` or `loading_messages`. The remaining blocker is a Slack product/config contract decision, especially with adjacent open and closed PRs proposing different status surfaces.

## What This Changes

The issue requests Slack config for custom `assistant.threads.setStatus` `status` and `loading_messages` so assistant thread indicators can match a bot persona.

## Best Possible Solution

Choose one Slack assistant-status contract, then implement it in the Slack plugin with typed config, schema validation, Slack docs, focused tests for configured start/clear behavior, and redacted live Slack proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main and v2026.6.8 verifies the feature gap: Slack still sends hardcoded status text and no config/schema field accepts custom status or loading messages.

## Solution Assessment

Unclear until maintainers choose the public contract. A Slack-owned config option is plausible, but landing it before reconciling account-level, per-agent, built-in loading-message, and tool-progress surfaces would create drift.
