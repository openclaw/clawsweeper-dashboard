## Summary

Keep this open. Current main still hardcodes Slack assistant thread status and has no config/schema/docs path for custom `status` or `loading_messages`; the feature is valid but needs maintainer choice because #33413 and closed, unmerged #42555 propose overlapping status semantics.

## What This Changes

Expose Slack assistant thread status configuration for custom status text and `loading_messages`.

## Best Possible Solution

Choose one Slack assistant-status contract, then add typed and validated Slack config, pass `status` and optional `loading_messages` through the Slack status wrapper, and update Slack tests plus https://docs.openclaw.ai/channels/slack.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: current main has a static Slack status call, no accepted config field, and docs/tests for the existing fallback behavior.

## Solution Assessment

Unclear until maintainers choose the public Slack status contract. A Slack-owned config passthrough is technically narrow and matches the Slack API, but #33413 and closed #42555 show adjacent semantics that should not diverge.
