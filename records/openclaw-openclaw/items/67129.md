## Summary

Keep this open. Current main still hardcodes Slack assistant-thread status text and has no typed Slack config/schema/docs path for custom `status` or `loading_messages`; the request is valid but needs maintainer API/product choice because related work proposes overlapping status semantics.

## What This Changes

Expose Slack assistant-thread status customization so users can configure the displayed status text and optional loading messages.

## Best Possible Solution

Keep this as the Slack status-customization request, choose the public contract, then add validated Slack config, pass `status` and optional `loading_messages` through the Slack status wrapper, and update focused Slack tests plus Slack docs.

## Reproduction Assessment

Not applicable as a feature request, but source inspection gives a high-confidence verification path. Current main has a static Slack status call, no accepted config field, and docs/tests that describe the existing fallback behavior.

## Solution Assessment

Unclear until maintainers choose the public Slack status contract. A Slack-owned config passthrough is narrow and plausible, but the closed identity-status PR and the open tool-progress request show competing or adjacent semantics.
