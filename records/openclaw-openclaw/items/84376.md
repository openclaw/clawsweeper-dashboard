## Summary

Keep open: the prior manifest fix covers bundled/source-checkout Codex, but the reported package-install path still appears valid because current marker recognition only trusts bundled manifest markers while doctor installs Codex from `@openclaw/codex` as an external/global plugin.

## What This Changes

The issue asks `openclaw secrets audit` to stop reporting `providers.codex.apiKey = "codex-app-server"` as `PLAINTEXT_FOUND` when that value is Codex's non-secret app-server auth marker.

## Best Possible Solution

Use the existing trusted-official plugin boundary, or an equivalently narrow Codex-owned contract, so `codex-app-server` from installed `@openclaw/codex` is treated as non-secret without broadening audit suppression to untrusted plugins.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an installed `@openclaw/codex` plugin is discovered as global/external, while `isNonSecretApiKeyMarker` only consumes bundled manifest markers, so `codex-app-server` falls through to `PLAINTEXT_FOUND` in `models.json` auditing.

## Solution Assessment

The requested direction is right, but the safest implementation is not a broad allowlist for all installed plugins; it should include bundled plus trusted official plugin-owned markers, or another narrow Codex-owned compatibility path.
