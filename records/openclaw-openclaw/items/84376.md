## Label Justifications

- `impact:security`: The affected surface is the secrets-audit signal, where false positives can hide real credential findings.
- `impact:auth-provider`: The false positive is caused by provider auth marker classification for the Codex runtime marker.

## Summary

Keep open: current main still only harvests `nonSecretAuthMarkers` from bundled plugin manifests, so the reported package-installed/global Codex path can still classify `codex-app-server` as `PLAINTEXT_FOUND`. The directly linked fix PR was closed unmerged, and the open broader SecretRef PR explicitly does not change secrets-audit marker classification.

## What This Changes

Treat the Codex `codex-app-server` runtime auth marker as non-secret so `openclaw secrets audit` does not report false-positive `PLAINTEXT_FOUND` findings for `providers.codex.apiKey`.

## Best Possible Solution

Recognize the exact Codex-owned `codex-app-server` marker for installed Codex without trusting arbitrary third-party/global manifest markers, with regression coverage for the package-installed or bundled-disabled path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `models.json` provider keys flow through `isNonSecretApiKeyMarker`, while installed Codex manifests are scanned as `global` and excluded from the bundled-only marker harvest.

## Solution Assessment

Yes on the direction, but the safest implementation is an exact Codex-owned marker or trusted-official plugin contract, not broad acceptance of all global plugin markers.
