## Label Justifications

- `P2`: The issue is a repeatable CLI audit false positive with CI/operator impact but no crash or direct credential exposure.
- `impact:security`: The affected surface is the secrets audit signal, where noisy false positives can hide real credential findings.
- `impact:auth-provider`: The false positive is caused by provider auth marker classification for the Codex runtime plugin.

## Summary

Keep open: source inspection confirms current main still only trusts bundled `nonSecretAuthMarkers`, so npm/global Codex installs can still fall through to `PLAINTEXT_FOUND`; an open fix PR already references this issue and should be reviewed instead of closing early.

## What This Changes

Treat Codex's `codex-app-server` synthetic auth marker as non-secret so `openclaw secrets audit` does not flag it as plaintext when Codex is installed externally.

## Best Possible Solution

Land or replace the narrow fix that recognizes the Codex-owned marker across installed official Codex topologies while keeping third-party marker suppression out of the audit trust boundary.

## Reproduction Assessment

Yes. Source inspection shows installed Codex manifests are discovered as `global` while marker recognition only consumes `bundled` origins; in that topology `codex-app-server` is not recognized and the audit path emits `PLAINTEXT_FOUND`.

## Solution Assessment

Yes. The maintainable fix is a narrow Codex-owned/core-trusted marker path with regression coverage for the external-install topology, not a broad trust rule for all global plugin markers.
