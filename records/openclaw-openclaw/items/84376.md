## Summary

Keep open. The earlier Codex marker false-positive was fixed for bundled-manifest source checkouts, but this fresh report includes v2026.5.18 npm evidence, and current/release source still excludes `dist/extensions/codex/**` from the core package while `isNonSecretApiKeyMarker` only imports `nonSecretAuthMarkers` from `origin === "bundled"` manifests; that makes the packaged external `@openclaw/codex` path plausibly still flag `codex-app-server`.

## What This Changes

The issue reports that `openclaw secrets audit` treats the Codex provider's `codex-app-server` synthetic auth marker as a `PLAINTEXT_FOUND` credential finding.

## Best Possible Solution

Preserve the marker-owned contract, but make the packaged official Codex path visible to secrets audit through a trusted core/official-plugin metadata path or a narrow legacy core marker, with regression coverage for an npm-style install where Codex is external rather than bundled.

## Reproduction Assessment

Yes, at source level: the release package excludes Codex extension metadata, the marker registry only accepts bundled-origin manifest markers, and the audit guard flags any non-empty `models.json` apiKey that is not recognized. The reporter also provided live v2026.5.18 CLI output for the same path.

## Solution Assessment

No, the earlier manifest-only fix is not enough for the reported packaged external Codex path. The maintainable fix should either expose trusted official external plugin markers to core audit or add a narrow legacy marker without opening a third-party suppression hole.
