## Label Justifications

- `P2`: The issue is a repeatable CLI audit false positive with CI/operator impact but no runtime crash or direct credential exposure.
- `impact:security`: The affected surface is the secrets audit signal, where noisy false positives can hide real credential findings.
- `impact:auth-provider`: The false positive is caused by provider auth marker classification for the Codex runtime plugin.

## Summary

Keep open: current `main` still only recognizes `codex-app-server` through the bundled Codex manifest, while the reported npm/global installed Codex path is discovered as `global` and excluded from marker harvesting. The exact follow-up PR that added a core Codex marker was closed unmerged, so there is no merged fix on `main`.

## What This Changes

The issue asks `openclaw secrets audit` to treat the Codex `codex-app-server` auth marker as non-secret instead of reporting `providers.codex.apiKey` as `PLAINTEXT_FOUND`.

## Best Possible Solution

Add a narrow Codex-owned marker path, preferably the exact `codex-app-server` core marker with a regression test where bundled plugin scanning is disabled, without expanding trust to arbitrary global plugin manifests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: package-installed Codex manifests are discovered as `global`, while `isNonSecretApiKeyMarker` only consumes bundled manifest markers, so `codex-app-server` falls through to `PLAINTEXT_FOUND` when no bundled Codex manifest supplies it.

## Solution Assessment

Yes, with a narrow trust boundary. The best fix is to recognize this exact Codex-owned marker, or an equivalently trusted official Codex install path, rather than accepting marker suppressions from arbitrary global plugins.
