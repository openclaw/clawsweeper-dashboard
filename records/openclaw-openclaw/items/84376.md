## Summary

Current main still only trusts `nonSecretAuthMarkers` from manifests with `origin === "bundled"`; the reported npm/global Codex install path is discovered as `origin: "global"`, so the `codex-app-server` marker can still fall through to `PLAINTEXT_FOUND`. The earlier merged manifest-only fix helps bundled/source checkouts, but the focused fix for this issue was a later closed, unmerged PR.

## What This Changes

The issue requests that `openclaw secrets audit` recognize Codex's `codex-app-server` synthetic auth marker as non-secret in installed/global Codex provider deployments.

## Best Possible Solution

Add a narrow Codex-owned marker recognition path, with regression coverage for the global/non-bundled install topology, while keeping arbitrary global plugin markers out of the secrets-audit allowlist.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: when Codex is installed under the state `extensions` directory, its manifest origin is `global`, so current marker harvesting excludes its `codex-app-server` marker and the audit guard can emit `PLAINTEXT_FOUND`. I did not run a live audit because this review is read-only and the source path is direct.

## Solution Assessment

Yes, the best fix is narrow marker recognition for the Codex-owned synthetic value plus a regression test with bundled plugin discovery disabled. A broad global-plugin marker allowlist would be less safe because it would expand the audit suppression boundary.
