## Summary

Keep open: current main still only recognizes `codex-app-server` through bundled Codex manifest metadata, so the package-installed/global Codex topology can still fall through to `PLAINTEXT_FOUND`; the focused fix PR was closed unmerged, and the open SecretRef PR is related but does not solve this marker-classification bug.

## What This Changes

The issue requests that `openclaw secrets audit` stop reporting Codex `providers.codex.apiKey: "codex-app-server"` as a plaintext credential.

## Best Possible Solution

Add a narrow trusted Codex-owned marker path, with regression coverage for the package-installed or bundled-disabled topology, while leaving broader third-party marker trust for a separate security design.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: installed/global Codex manifests are scanned with origin `global`, while `isNonSecretApiKeyMarker` only harvests bundled manifest markers, so `codex-app-server` is not trusted when no bundled Codex manifest contributes it.

## Solution Assessment

Yes. The narrowest maintainable fix is to recognize the exact Codex-owned marker through a trusted core or official-plugin path; broadly accepting global plugin markers would widen the audit suppression boundary unnecessarily.
