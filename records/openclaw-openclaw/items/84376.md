## Summary

Keep open: current main still appears to miss the reported package-install topology because `codex-app-server` is only recognized through bundled manifest markers, while installed `@openclaw/codex` manifests are classified as global and excluded from the marker harvest. The direct fix PR at https://github.com/openclaw/openclaw/pull/84391 was closed unmerged, and the related open PR at https://github.com/openclaw/openclaw/pull/84658 targets SecretRef source snapshots rather than this marker classifier.

## What This Changes

The issue asks for `openclaw secrets audit` to treat the Codex `codex-app-server` synthetic auth marker as non-secret instead of reporting `PLAINTEXT_FOUND`.

## Best Possible Solution

Land a narrow Codex-owned marker recognition path, preferably the exact literal or an equivalently trusted official-plugin boundary, with regression coverage for the installed/global topology while leaving the broader SecretRef persistence work to its own issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: installed `@openclaw/codex` is represented as a global plugin, but `isNonSecretApiKeyMarker` only consumes bundled manifest markers, so `codex-app-server` can fall through to `PLAINTEXT_FOUND` in `models.json` auditing.

## Solution Assessment

Yes. The narrow maintainable fix is to recognize the exact Codex-owned marker or a trusted official Codex marker path; accepting arbitrary global plugin markers would be the wrong security boundary.
