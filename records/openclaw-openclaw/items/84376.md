## Summary

Keep open: current main still recognizes `codex-app-server` only through the bundled Codex manifest, while the reported npm/global install path discovers Codex as a global plugin whose `nonSecretAuthMarkers` are ignored. The exact follow-up PR for this issue was closed unmerged, and the separate open SecretRef PR does not change audit marker classification.

## What This Changes

The issue asks for `openclaw secrets audit` to treat Codex's `codex-app-server` synthetic auth marker as non-secret in package-installed deployments.

## Best Possible Solution

Add a narrow Codex-owned marker path, such as the exact core marker from the closed fix PR or an equivalent trusted-official plugin contract, with regression coverage for package-installed Codex and no trust expansion for arbitrary global plugins.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `models.json` provider apiKeys go through `isNonSecretApiKeyMarker`, installed Codex manifests are `global`, and current marker harvesting only accepts `bundled` manifest markers.

## Solution Assessment

Yes, the requested direction is the right narrow fix if it stays scoped to the Codex-owned literal. A broad allowlist for all global plugin markers would be the unsafe variant.
