## Summary

Keep open: current `main` still only trusts bundled plugin manifest markers, so the reported package-installed Codex path can still classify the official `codex-app-server` marker as `PLAINTEXT_FOUND`.

## What This Changes

The issue requests that `openclaw secrets audit` stop reporting the Codex `codex-app-server` synthetic auth marker as a plaintext API key.

## Best Possible Solution

Fix marker recognition through the existing trusted-official plugin boundary, or an equivalently narrow Codex-owned contract, so installed `@openclaw/codex` suppresses only its own synthetic marker without broadening the audit allowlist.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: installed/global Codex manifests are filtered out of `listKnownNonSecretApiKeyMarkers()`, so `codex-app-server` falls through to the `PLAINTEXT_FOUND` branch in `openclaw secrets audit`.

## Solution Assessment

Yes, with a trust-boundary caveat. The narrow fix should extend marker recognition only to bundled or trusted official Codex/plugin-owned markers, not to arbitrary external plugin manifests.
