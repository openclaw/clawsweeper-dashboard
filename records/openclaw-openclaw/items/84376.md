## Summary

Keep open: current main and v2026.5.20 still only recognize `codex-app-server` through bundled plugin manifests, while installed Codex plugins are discovered as `global`. The focused fix in https://github.com/openclaw/openclaw/pull/84391 closed unmerged, and the open SecretRef PR is related but explicitly does not change marker classification.

## What This Changes

The issue asks for `openclaw secrets audit` to treat the Codex `codex-app-server` auth marker as non-secret instead of reporting `providers.codex.apiKey` as `PLAINTEXT_FOUND` in installed/global Codex deployments.

## Best Possible Solution

Add a narrow Codex-owned marker recognition path for `codex-app-server` plus regression coverage for the disabled-bundled/external-install topology, without broadening audit suppression to arbitrary global plugin manifests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: installed Codex plugins are discovered as `global`, but `isNonSecretApiKeyMarker` only consumes bundled manifest markers, so `codex-app-server` can still fall through to `PLAINTEXT_FOUND`.

## Solution Assessment

Yes, the requested direction is the right fix if it stays exact-marker and Codex-owned. A broad trust of all global plugin `nonSecretAuthMarkers` would be a weaker security boundary.
