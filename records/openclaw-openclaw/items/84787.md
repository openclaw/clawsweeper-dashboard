Keep the current status contract but refactor summary collection to count all sessions with lightweight metadata and hydrate model/runtime/context details only for the recent rows that human and JSON output return.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84787. Current `getStatusSummary` builds full hydrated session rows for every stored session, then slices to the newest 10, which makes `openclaw status` and `openclaw status --json` slow on larger stores. Preserve the existing output contract: `sessions.count` and `byAgent.count` must count all sessions, `sessions.recent` and each `byAgent.recent` must still expose the newest 10 rows with the same fields, and pinned model warnings must still work for recent rows. Refactor the status summary path to read each store once per command, compute lightweight counts/top-N by `updatedAt`, and hydrate model/runtime/context only for rows that will be returned; cache hydrated rows when the same session appears in aggregate and per-agent recent lists. Do not add a config flag and do not fold in the separate task-registry reconciliation issue. Add focused Vitest coverage with more than 10 sessions that asserts non-recent rows do not call `resolveSessionRuntimeLabel`, `resolveSessionModelRef`, or `resolveContextTokensForModel`, while total counts remain correct. Add a user-facing changelog fix entry if required by the repair lane.
