## Summary

## What This Changes

Keep `openclaw status` fast by avoiding full session-summary/model-runtime hydration across all stored sessions.

Keep the released bounded status-summary implementation; handle any distinct task-registry or artifact-footprint latency in narrower follow-up issues such as https://github.com/openclaw/openclaw/issues/73531.

## Reproduction Assessment

No for current main: the old source path built full hydrated rows before slicing, but current main selects recent candidates before hydration and has regression coverage for that boundary.

## Solution Assessment

Yes. The current candidate-first summary is the narrow maintainable fix because it preserves total/per-agent counts and the existing status output while avoiding model/runtime/context hydration for non-rendered rows.
