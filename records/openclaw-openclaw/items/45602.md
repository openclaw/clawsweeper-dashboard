## Summary

Keep open: the underlying one-Enter `/status` bug is valid on current main, but this PR fixes it by moving `/status` onto a separate local UI status path, which risks diverging from the canonical gateway/agent status renderer.

## What This Changes

This PR marks `/status` as a local web UI slash command, adds UI-side status execution/default plumbing, and registers slash-command regression tests.

## Best Possible Solution

Fix the slash-menu selection/send behavior so no-argument agent commands such as `/status` submit with one Enter while preserving the existing gateway/agent status renderer.

## Reproduction Assessment

Yes. Source inspection shows current main keeps `/status` off the local-command list, and the menu handler only sends selected no-argument commands when `executeLocal` is true, so the first Enter completes the draft instead of submitting it.

## Solution Assessment

No. Making `/status` local fixes the keypress symptom but duplicates the canonical status path; the narrower maintainable fix is to submit the existing agent-routed command from the menu with one Enter.

  - body: The local status path only receives the global thinking default plus catalog data, so it misses `agents.defaults.models.<provider>/<model>.params.thinking` when no session override is stored. That can make `/status` report a lower level than runtime actually uses.
  - confidence: 0.87
