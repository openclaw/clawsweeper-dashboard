## Summary

Keep open: current main still has the reported one-Enter `/status` slash-menu bug, but this PR changes `/status` into a duplicate local status path that can diverge from the canonical status output and still has only test output as proof.

## What This Changes

The PR marks `/status` as a local web UI slash command, adds local status formatting plus session default fields/tests, and registers those tests in Vitest.

## Best Possible Solution

Fix the slash-menu selection/send flow so no-argument agent commands like `/status` submit on the first Enter while continuing to use `buildStatusText`/`buildStatusReply`, with focused regression coverage for that menu flow.

## Reproduction Assessment

Yes, from source inspection. `status` is not local on current main, and the menu only sends immediately for no-arg `executeLocal` commands, so selecting `/status` fills the draft and a second Enter sends it.

## Solution Assessment

No. The PR fixes the keypress symptom by moving `/status` into a new local formatter, but the narrower maintainable fix is to submit the selected agent command or call the canonical status path without duplicating status logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep `/status` on the canonical status path:** `ui/src/ui/chat/slash-command-executor.ts:72-73`
  - body: Routing `/status` to this new local executor changes more than the Enter behavior: it returns a small session summary instead of the canonical `buildStatusText` output, dropping runtime, queue, task, plugin, auth, and other status context. Fix the menu-submit behavior or call the canonical status path instead of rebuilding `/status` in the UI.
  - confidence: 0.88
- **[P2] Compute fast mode for the active session model:** `src/gateway/session-utils.ts:750-757`
  - body: `fastModeDefault` is computed from the globally resolved default model and then reused by `/status` and `/fast status` for the active session. If the session uses a different model with different per-model `params.fastMode`, the UI can display the opposite of runtime behavior.
  - confidence: 0.9
- **[P2] Honor per-model thinking defaults in local status:** `ui/src/ui/chat/slash-command-executor.ts:304-307`
  - body: The local status path only receives a global `thinkingDefault` plus catalog data, so it cannot apply `agents.defaults.models.<provider>/<model>.params.thinking` when no session override is stored. That can make `/status` report a catalog/default level while runtime uses the configured per-model thinking level.
  - confidence: 0.87
