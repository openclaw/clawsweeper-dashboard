## Summary

Keep this PR open but do not merge it as-is: current main still has the one-Enter `/status` bug, while this branch fixes it by moving `/status` onto a new local status implementation that is stale against current main and can misreport effective session defaults.

## What This Changes

The PR marks `/status` as a local web UI slash command, adds a client-side status executor using session/model RPC state, extends session-default types for thinking/fast-mode display, and registers regression tests.

## Best Possible Solution

Preserve the canonical `/status` command behavior and fix the web slash menu so selecting a no-argument command submits it with one Enter instead of reimplementing status formatting in the UI.

## Reproduction Assessment

Yes. Source inspection shows `/status` is not local, and the Enter handler only sends no-arg commands when `executeLocal` is true, so the first Enter completes the draft and the second sends it.

## Solution Assessment

No. Moving `/status` to a separate local executor fixes the keypress symptom but duplicates the canonical status path and leaves effective thinking/fast-mode defaults vulnerable to mismatch; a narrower menu-submit fix is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep `/status` on the canonical status path:** `ui/src/ui/chat/slash-command-executor.ts:72-73`
  - body: Routing `/status` to this new local executor changes more than the Enter behavior: it returns a minimal session summary instead of the existing status text built by the agent/gateway status path, dropping runtime, queue, task, plugin, auth, and other status details. Fix the slash-menu submission behavior or call a canonical status RPC instead of rebuilding `/status` in the UI.
  - confidence: 0.88
- **[P2] Resolve fast mode for the active session model:** `src/gateway/session-utils.ts:750-757`
  - body: `fastModeDefault` is computed from the globally resolved default model and then reused by `/status` and `/fast status` for the active session. If the session is using another model with a different per-model `params.fastMode`, the UI can display the opposite of runtime behavior; compute this per session/model or avoid the local status path.
  - confidence: 0.91
- **[P2] Honor per-model thinking defaults in local status:** `ui/src/ui/chat/slash-command-executor.ts:304-307`
  - body: The local status path only receives a global `thinkingDefault` plus catalog data, so it misses `agents.defaults.models.<provider>/<model>.params.thinking` when no session override is stored. That can make `/status` report `off` or a catalog default while runtime uses a configured per-model thinking level.
  - confidence: 0.87
