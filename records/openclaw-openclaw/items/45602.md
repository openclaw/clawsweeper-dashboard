## Summary

Keep open: the underlying bug is still source-reproducible on current main, but this PR is not merge-ready because it moves `/status` off the canonical status path, leaves status/default correctness regressions, is conflicting with main, and lacks external real behavior proof.

## What This Changes

The PR changes the web UI `/status` slash command to execute locally, adds local status formatting/default propagation, and registers focused Vitest coverage.

## Best Possible Solution

Preserve canonical `/status` behavior and fix the slash-menu selection path so choosing `/status` submits once without duplicating status formatting in the UI.

## Reproduction Assessment

Yes. By source inspection, current main keeps `/status` out of `LOCAL_COMMANDS`, and `selectSlashCommand` only submits on the first Enter for no-argument `executeLocal` commands, so `/status` is selected into the draft instead of sent.

## Solution Assessment

No. The PR fixes the keypress symptom by making `/status` local, but the narrower maintainable fix is to submit the selected no-argument agent command or call the canonical status path without rebuilding status output in the UI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep `/status` on the canonical status path:** `ui/src/ui/chat/slash-command-executor.ts:73-74`
  - body: Routing `/status` to this new local executor changes more than the Enter behavior: it returns a small session summary instead of the canonical `buildStatusText`/`buildStatusReply` output, dropping runtime, task, plugin, auth, and other status context. Fix the menu-submit behavior or call the canonical status path instead of rebuilding `/status` in the UI.
  - confidence: 0.88
- **[P2] Compute fast mode for the active session model:** `src/gateway/session-utils.ts:750-757`
  - body: `fastModeDefault` is computed from the globally resolved default model and then reused by `/status` and `/fast status` for the active session. If the session uses a different model with different per-model `params.fastMode`, the UI can display the opposite of runtime behavior.
  - confidence: 0.9
- **[P2] Honor per-model thinking defaults in local status:** `ui/src/ui/chat/slash-command-executor.ts:307-308`
  - body: The local status path only receives a global `thinkingDefault` plus catalog data, so it cannot apply configured per-model thinking defaults when no session override is stored. That can make `/status` report a catalog/default level while runtime uses the configured per-model thinking level.
  - confidence: 0.87
