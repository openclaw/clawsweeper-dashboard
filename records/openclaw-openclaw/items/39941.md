## Summary

Keep open: current main and the latest release still have the first-object-variant bug, and this PR looks like a narrow valid fix, but it still needs contributor real behavior proof and fresh validation before merge.

## What This Changes

The PR changes Control UI config form coercion to prefer `anyOf`/`oneOf` object variants whose top-level `const` property constraints match the form value, with regression tests for `anyOf`, `oneOf`, and unconstrained-variant ordering.

## Best Possible Solution

Land this local const-discriminator selection fix, or an equivalent maintainer-carried patch, with the regression tests once redacted real behavior proof and current validation are present.

## Reproduction Assessment

Yes, at source level. Current main returns from `coerceFormValues()` on the first object-valued `anyOf`/`oneOf` variant, so a later const-matching variant can be skipped before nested string fields are coerced.

## Solution Assessment

Yes for the implementation direction. The PR keeps primitive and array union behavior intact, prefers object variants only when declared `const` constraints match, and falls back to the existing first-object behavior when no discriminator matches.
