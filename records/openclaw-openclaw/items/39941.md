## Summary

Keep open. Current main still chooses the first object-valued `anyOf`/`oneOf` variant during Control UI config form coercion, so the PR addresses a real remaining bug; the latest PR patch also addresses the earlier review concern about unconstrained object variants outranking constrained matches.

## What This Changes

The PR adds discriminator-aware object variant selection to `coerceFormValues()` for config-schema `anyOf`/`oneOf` unions and adds focused regression tests for matching constrained variants and the unconstrained-variant edge case.

## Best Possible Solution

Review and land the narrow discriminator-aware selection in the Control UI config coercion helper, preserving existing primitive-union and array fallback behavior while covering both `anyOf` and `oneOf` object unions in tests.

## Reproduction Assessment

Yes. A unit-level reproduction is clear from current main: pass `coerceFormValues()` an object value whose discriminator matches a later object variant while the first object variant gives a nested field a numeric schema, and current main will recurse into the first object variant and coerce the nested string.

## Solution Assessment

Yes. The PR keeps the fix local to the form coercion helper, prefers object variants with matching top-level property `const` constraints, falls back to the first object variant when no constrained match exists, and adds the relevant regression tests without changing broader config schema contracts.
