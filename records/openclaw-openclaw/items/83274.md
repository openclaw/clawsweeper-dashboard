## Summary

Keep open: the PR is aimed at a real safety area, but the branch has blocking regressions in channel ownership/routing and still leaves an unsafe announce-output fallback path; it also lacks required after-fix real behavior proof.

## What This Changes

The branch filters some unsafe subagent announce output, rejects persistent thread-bound subagent session spawns without a routable deliveryOrigin, and special-cases Telegram as an automatic thread-bound spawn channel.

## Best Possible Solution

Keep the safety goal, but apply unsafe-output filtering to every fallback, preserve existing indirect routing, and express any Telegram automatic-spawn behavior through plugin-owned metadata or an explicitly approved contract.

## Reproduction Assessment

Yes. Source inspection shows the proposed deliveryOrigin guard would reject current-main tests that intentionally accept threadBindingReady without deliveryOrigin; a focused verification would run `node scripts/run-vitest.mjs run --config test/vitest/vitest.agents.config.ts src/agents/subagent-spawn.thread-binding.test.ts` on the PR branch.

## Solution Assessment

No. The safer solution is to keep channel policy in plugin-owned metadata, preserve indirect completion routing where current tests cover it, and run the unsafe-output guard after fallback assistant reads too.
