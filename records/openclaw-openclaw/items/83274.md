## Summary

This PR should stay open, not close: the safety intent is valid, but the current diff is not merge-ready. It hard-codes Telegram policy in core, rejects existing indirect thread-bound session paths, leaves an unsafe assistant-output fallback unfiltered, and lacks required real Telegram/runtime behavior proof.

## What This Changes

The PR adds unsafe subagent announce-output filtering, fails persistent thread-bound session spawns without a routable deliveryOrigin, and special-cases Telegram as an automatic thread-bound spawn channel.

## Best Possible Solution

Keep the safety goal, but move Telegram opt-in into a plugin-owned contract, preserve indirect completion routing when a generic binding can resolve delivery, and filter every announce-output fallback before surfacing it.

## Reproduction Assessment

Yes. Source inspection shows the new deliveryOrigin guard would return an error for existing main-branch tests that currently accept ready-without-origin thread-bound session paths; I did not run tests because this review is read-only.

## Solution Assessment

No. The safer solution is to keep channel policy in plugin metadata, only fail closed when no valid indirect route exists, and apply the unsafe-output guard after fallback assistant reads too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move Telegram spawn policy out of core:** `src/channels/thread-bindings-policy.ts:52-55`
  - body: This hard-codes a bundled channel id in core policy even though the helper already reads channel plugin metadata. It also conflicts with the Telegram plugin's own `defaultTopLevelPlacement: "current"`, creating two sources of truth for when automatic thread-bound spawns are exposed. Please move the Telegram opt-in into a plugin-owned contract or metadata field instead of special-casing `telegram` here.
  - confidence: 0.9
- **[P1] Preserve indirect thread-bound session routing:** `src/agents/subagent-spawn.ts:1011-1027`
  - body: Rejecting every `mode="session"` bind without a routable `deliveryOrigin` breaks existing supported paths where the hook marks the binding ready but completion is intentionally routed back through the owner/generic binding resolver. Current coverage in `src/agents/subagent-spawn.thread-binding.test.ts` accepts both the Telegram controller-owner case and the generic binding case with no `deliveryOrigin`, so this guard would turn those spawns into errors.
  - confidence: 0.94
- **[P2] Filter fallback assistant output too:** `src/agents/subagent-announce-output.ts:356-366`
  - body: When `latestAssistantText` is unsafe, `selectSubagentOutputText` returns `undefined`, but `readSubagentOutput` then falls back to `readLatestAssistantReply` and returns that assistant text without the new guard. That leaves unsafe assistant-shaped output able to reach the announce path, so the safety check needs to apply to the fallback result as well.
  - confidence: 0.86
