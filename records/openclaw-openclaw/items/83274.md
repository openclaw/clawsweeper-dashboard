## Summary

Keep open. The safety goal is valid, but the patch still has blocking correctness and security issues: it hard-codes Telegram policy in core, rejects existing indirect thread-bound routing paths, and leaves the unsafe-output fallback unfiltered; it also lacks after-fix real behavior proof.

## What This Changes

This PR filters unsafe subagent announce output, rejects thread-bound session spawns without a routable deliveryOrigin, and special-cases Telegram as automatic thread-bound spawn-capable.

## Best Possible Solution

Keep the safety goal, but move Telegram opt-in into plugin-owned metadata or a typed contract, preserve indirect completion routing when a valid owner/generic route exists, and filter every announce-output fallback before surfacing it.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the new `deliveryOrigin` guard would reject current-main ready-without-origin thread-binding paths covered by existing tests. I did not run tests because this review is read-only.

## Solution Assessment

No. The narrow maintainable fix is to keep channel policy in plugin-owned metadata, fail closed only when no valid indirect route exists, and apply unsafe-output filtering to the fallback assistant read too.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Move Telegram spawn policy out of core:** `src/channels/thread-bindings-policy.ts:52-55`
  - body: This hard-codes a bundled channel id in core policy even though the helper already reads channel plugin metadata. It also conflicts with the Telegram plugin's own `defaultTopLevelPlacement: "current"`, creating two sources of truth for automatic thread-bound spawn exposure.
  - confidence: 0.9
- **[P1] Preserve indirect thread-bound session routing:** `src/agents/subagent-spawn.ts:1011-1027`
  - body: Rejecting every `mode="session"` bind without a routable `deliveryOrigin` breaks supported paths where the hook marks the binding ready but completion is intentionally routed back through the owner or generic binding resolver. Current coverage accepts both the Telegram controller-owner case and a generic binding case with no `deliveryOrigin`.
  - confidence: 0.94
- **[P2] Filter fallback assistant output too:** `src/agents/subagent-announce-output.ts:356-366`
  - body: When unsafe assistant text is suppressed by the new selector, `readSubagentOutput` falls back to `readLatestAssistantReply` and returns that assistant text without the same guard. That leaves unsafe assistant-shaped output able to reach completion or timeout announces.
  - confidence: 0.86
