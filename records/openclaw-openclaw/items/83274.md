## Summary

This PR should stay open and is not merge-ready: the patch hard-codes Telegram policy in core, rejects existing indirect thread-bound routing paths, leaves an unsafe-output fallback unfiltered, and still lacks real behavior proof.

## What This Changes

The PR filters subagent announce output, rejects thread-bound session spawns without hook delivery origins, updates related tests, and makes Telegram report automatic thread-bound spawn support in core policy.

## Best Possible Solution

Keep the safety goal, but implement Telegram opt-in through a plugin-owned contract, preserve owner/generic completion routing, and filter every announce-output fallback before surfacing it.

## Reproduction Assessment

Yes. Source inspection shows the added deliveryOrigin guard would reject current-main ready-without-origin thread-binding paths, and the unsafe-output filter falls through to an unguarded `readLatestAssistantReply` fallback; I did not run tests because this review is read-only.

## Solution Assessment

No. The safer path is plugin-owned Telegram policy, a narrower fail-closed check that preserves indirect completion routing, and applying the unsafe-output guard to fallback assistant text too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move Telegram spawn policy out of core:** `src/channels/thread-bindings-policy.ts:52-55`
  - body: Core already resolves automatic thread-spawn support through channel plugin `conversationBindings.defaultTopLevelPlacement`, and Telegram currently declares `defaultTopLevelPlacement: "current"`. Hard-coding `telegram` here creates a second source of truth in core; please move the opt-in into a typed plugin contract or metadata field instead.
  - confidence: 0.9
- **[P1] Preserve indirect thread-bound session routing:** `src/agents/subagent-spawn.ts:1011-1027`
  - body: These added lines reject every `mode="session"` thread bind that lacks a hook `deliveryOrigin`. Current main explicitly supports ready-without-origin paths where completion is routed through the controller/owner or generic binding resolver, so valid spawns would turn into errors.
  - confidence: 0.94
- **[P2] Filter fallback assistant output too:** `src/agents/subagent-announce-output.ts:356-366`
  - body: When an unsafe latest assistant message is skipped by this selector, `readSubagentOutput` falls through to `readLatestAssistantReply` and returns that assistant text without calling the new guard. Apply the unsafe-output predicate to the fallback result as well.
  - confidence: 0.86
