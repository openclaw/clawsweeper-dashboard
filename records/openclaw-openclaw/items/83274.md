## Summary

Keep open: the PR has useful safety intent, but the patch has blocking correctness/security issues around channel ownership, indirect thread routing, and an unfiltered assistant fallback, and it still lacks after-fix real behavior proof.

## What This Changes

The PR filters subagent announce output, rejects thread-bound session spawns without a routable deliveryOrigin, and hard-codes Telegram as eligible for automatic thread-bound session spawn.

## Best Possible Solution

Keep the safety goal, but move Telegram opt-in into a plugin-owned contract, preserve indirect completion routing when a generic binding can resolve delivery, and filter every announce-output fallback before surfacing it.

## Reproduction Assessment

Yes, by source inspection. The new deliveryOrigin guard would reject existing current-main paths that intentionally accept ready-without-origin bindings, and the unsafe-output fallback path is visible in the PR head without running tests.

## Solution Assessment

No. The safer solution is to keep channel policy in plugin metadata or a typed channel seam, fail closed only when no valid route exists, and apply unsafe-output filtering after the fallback assistant read as well.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move Telegram spawn policy out of core:** `src/channels/thread-bindings-policy.ts:52-54`
  - body: This hard-codes a bundled channel id in core policy even though the helper already reads channel plugin metadata. It also conflicts with Telegram's plugin-owned `defaultTopLevelPlacement: "current"`, creating two sources of truth for automatic thread-bound spawns.
  - confidence: 0.9
- **[P1] Preserve indirect thread-bound session routing:** `src/agents/subagent-spawn.ts:1011-1027`
  - body: Rejecting every `mode="session"` bind without a routable `deliveryOrigin` breaks supported paths where the hook marks the binding ready but completion is intentionally routed through the owner or generic binding resolver. Current main has coverage for both cases.
  - confidence: 0.94
- **[P2] Guard the fallback assistant read:** `src/agents/subagent-announce-output.ts:356-366`
  - body: When the selector suppresses unsafe assistant text by returning `undefined`, `readSubagentOutput` falls back to `readLatestAssistantReply` and returns the same assistant-shaped text without the new guard. Apply the unsafe-output predicate to that fallback before announcing it.
  - confidence: 0.86
