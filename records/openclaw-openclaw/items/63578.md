## Summary

Keep this PR open. Current main still does not hydrate media/file hints for `message.send`, while the latest PR shape addresses the prior unconditional-preload concern by making send-media hydration opt-in for plugin handlers; the remaining work is maintainer review of the public plugin contract plus required release/docs validation.

## What This Changes

The branch adds an optional `preloadSendMedia` channel action flag, threads it into send-action attachment hydration, and adds unit/runner tests for default no-preload and opt-in media hydration behavior.

## Best Possible Solution

Land a reviewed opt-in plugin send-media preloading contract, or an equivalent generic seam, with SDK/plugin documentation, changelog coverage, focused tests, and unchanged default URL/path forwarding for handlers that already load media in their own runtime.

## Reproduction Assessment

Yes. Current main provides a source-level reproduction: `message.send` can advertise media/attachment fields, but the hydrator exits before `send` can materialize media/file hints into shared `buffer`, `contentType`, and `filename` fields.

## Solution Assessment

Unclear pending maintainer API review. The opt-in approach is safer than unconditional preloading because existing handlers such as Google Chat and Matrix load media themselves, but maintainers should confirm this is the preferred public contract and require docs/changelog/API proof.
