## Summary

Keep open. Current main still does not hydrate ordinary plugin `send` media into `buffer`/`contentType`/`filename`, and this PR is a plausible focused fix, but it is not ready to merge because the public SDK API baseline is missing and the contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds a `preloadSendMedia` opt-in to plugin channel actions, threads it into send-action attachment hydration, and adds unit/runner coverage.

## Best Possible Solution

Land one approved opt-in plugin send-media preload seam with the SDK baseline updated, default no-preload behavior preserved, focused outbound tests passing, and redacted real behavior proof attached.

## Reproduction Assessment

Yes, source-level. On current main, a plugin `send` action with media/file hints reaches the shared runner, but `send` is excluded from the hydrator gate and the runner has no plugin opt-in for materializing `buffer`, `contentType`, or `filename`.

## Solution Assessment

Unclear pending maintainer API review. The opt-in `preloadSendMedia` design is safer than unconditional send hydration, but the public SDK baseline and real behavior proof gaps should be resolved before merge.
