## Summary

Keep open. Current main still does not hydrate media/file hints for `message(action="send")`, and the PR is a plausible opt-in fix, but it changes the exported channel-plugin contract without the SDK API baseline update and lacks contributor real-behavior proof.

## What This Changes

The PR adds an opt-in `preloadSendMedia` channel action flag, threads it into send-action attachment hydration, and adds outbound unit/runner coverage.

## Best Possible Solution

Land an approved opt-in send-media preload seam with the SDK API baseline updated, default no-preload behavior preserved, focused outbound tests passing, and maintainer-owned release notes handled at landing.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: on current main, a plugin `send` action with `media` or `filePath` reaches the hydrator, which returns before materializing `buffer`, `contentType`, or `filename`. I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer API review. The opt-in design is safer than unconditional send-media preloading, but it should not merge until the SDK baseline and real behavior proof are present.
