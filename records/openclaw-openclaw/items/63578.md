## Summary

Keep open. Current main still skips attachment hydration for `send`, and this PR is directionally useful after moving to an explicit plugin opt-in, but the branch still changes the exported channel-plugin SDK contract without updating the API baseline or changelog.

## What This Changes

The PR adds opt-in `preloadSendMedia` support for plugin `send` handlers, threads that option into attachment hydration, and adds regression coverage for default no-preload and opt-in media hydration.

## Best Possible Solution

Land an opt-in generic send-media preload seam only after SDK API baseline, changelog, focused outbound tests, changed-gate validation, and maintainer approval of the public channel-plugin contract.

## Reproduction Assessment

Yes. Current main provides a source-level reproduction: a `send` action with media/file hints reaches `hydrateAttachmentParamsForAction`, which returns before populating `buffer`, `contentType`, or `filename`.

## Solution Assessment

Unclear pending API review. The opt-in approach is safer than unconditional preloading because existing Google Chat and Matrix handlers load media themselves, but the public SDK metadata and release notes must be completed first.
