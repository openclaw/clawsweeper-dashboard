## Summary

Keep open. Current main still does not hydrate media hints for plugin-handled `message.send`, and this PR remains a plausible implementation path, but the branch changes a public plugin SDK/channel contract and is not ready until the SDK baseline, changelog, validation, and maintainer API review are handled.

## What This Changes

The PR adds an optional `preloadSendMedia` channel action flag, threads opt-in send-action media hydration into the outbound runner, and adds unit/runner coverage for default no-preload and opt-in hydrated media behavior.

## Best Possible Solution

Land an opt-in generic send-media preload seam only after the public SDK contract is approved, the generated Plugin SDK API hash and changelog are updated, plugin docs cover the new flag if maintainers want it documented, focused outbound tests pass, and default URL/path forwarding remains unchanged for handlers that load media in their own runtime.

## Reproduction Assessment

Yes. Current main provides a source-level reproduction: call `hydrateAttachmentParamsForAction` with `action: "send"` and a media/file hint, or run a plugin `send` path with media, and the hydrator returns before populating `buffer`, `contentType`, or `filename`.

## Solution Assessment

Unclear pending maintainer API review. The opt-in approach is narrower and safer than unconditional shared preloading because existing handlers such as Google Chat and Matrix load media themselves, but the public contract needs baseline, changelog, and validation follow-through before merge.
