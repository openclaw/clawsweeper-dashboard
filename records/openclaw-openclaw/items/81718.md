## Summary

Close as duplicate/superseded. The same gateway base64 stack-safety bug is already tracked by the open canonical PR at https://github.com/openclaw/openclaw/pull/80708, which keeps the max-size guard before decoding and adds regression coverage; this branch also introduces an unbounded decode-before-size-check regression and lacks inspectable real behavior proof.

## What This Changes

The PR replaces `isValidBase64` in `src/gateway/chat-attachments.ts` with a whitespace-stripping Buffer round-trip validator.

## Best Possible Solution

Close this duplicate branch and route the stack-safe base64 fix through the canonical open PR, preserving max-size-before-decode behavior and regression tests.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The linked report includes exact `v2026.5.7` gateway logs and current source still has the same full-payload regex validator, so the suspected path is source-reproducible.

## Solution Assessment

No. Removing the unbounded regex scan is the right direction, but this branch is superseded by https://github.com/openclaw/openclaw/pull/80708 and regresses the existing size-before-decode guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep size checks before base64 decoding:** `src/gateway/chat-attachments.ts:128`
  - body: This moves `Buffer.from(..., "base64")` into validation, so oversized attachment payloads are fully decoded before the existing `estimateBase64DecodedBytes` max-size guard can reject them. Current tests explicitly assert oversized inputs are rejected without any base64 decode; keep the size estimate before any Buffer decode and use a linear validator/canonicalizer instead.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it introduces an availability/resource-control concern by decoding unchecked-size gateway input before the size guard.

Concerns:

- **[medium] Avoid decoding unchecked attachment payloads:** `src/gateway/chat-attachments.ts:128`
  - body: Decoding user-provided base64 before max-size rejection can allocate memory for oversized gateway inputs and bypasses the current resource-control invariant tested on main.
  - confidence: 0.88
