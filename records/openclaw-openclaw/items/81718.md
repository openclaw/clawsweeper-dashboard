## Summary

Keep this PR open but block merge: it targets a plausible gateway attachment crash, but the proposed round-trip validator decodes user payloads before existing size guards and no after-fix real-behavior proof is attached.

## What This Changes

The PR replaces `src/gateway/chat-attachments.ts` base64 regex validation with a `Buffer` decode/re-encode round-trip in `isValidBase64`.

## Best Possible Solution

Use allocation-conscious base64 validation that preserves the estimate-before-decode guard, add focused regressions for large valid payloads and oversize rejection, then attach redacted real webchat proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection plus the linked issue logs give a clear path through `chat.send` with a large mobile image, but I did not execute the iOS Safari webchat flow in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. A full `Buffer` round-trip removes the stack-prone regex but decodes and re-encodes user input before size checks; a loop/canonicalization validator with preserved pre-decode size guards is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the pre-decode size guard:** `src/gateway/chat-attachments.ts:128`
  - body: This validation now decodes and re-encodes the entire attachment before `estimateBase64DecodedBytes`, `maxBytes`, or `MAX_IMAGE_BYTES` can reject it. That breaks the existing contract covered by the gateway test that oversized attachments must fail without any `Buffer.from(..., "base64")` call, and it can turn rejected user input into avoidable memory pressure. Please validate characters/canonical form without decoding, or move the size guard ahead of any round-trip decode.
  - confidence: 0.9
