## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not scope mixed alias requests to only the resolvable plugin:** `src/plugins/capability-provider-runtime.ts:660-664`
  - body: With no active registry, a request set like `openai-realtime` plus `xai` resolves only the xAI manifest contract, so the cold load is scoped to xAI and the later explicit OpenAI alias lookup fails even though OpenAI is available. Treat unresolved requested IDs as unscoped, or resolve provider aliases before narrowing the load.
  - confidence: 0.88
- **[P2] Keep talk.catalog from narrowing cold provider lists:** `src/gateway/server-methods/talk.ts:235`
  - body: When no realtime-transcription registry is active, `listTalkTranscriptionProviders` passes the configured Voice Call provider IDs as a load scope, so a config that selects xAI can return only xAI even if OpenAI is also installed. The protocol describes this group as active provider plus all candidates, so provider-picker clients lose valid options until some other path activates the registry.
  - confidence: 0.9
- **[P2] Keep scoped realtime catalog providers in sync:** `src/gateway/server-methods/talk-shared.ts:90`
  - body: This helper now reads `@openclaw/voice-call` for both streaming and realtime config, but the realtime catalog still calls `listRealtimeVoiceProviders(config)` without passing the scoped realtime provider. With an active OpenAI-only registry and scoped `realtime.provider: "google"`, the catalog can advertise Google as active while listing only OpenAI, so either pass the requested realtime provider through or keep the scoped read limited to the streaming path changed by this PR.
  - confidence: 0.84
