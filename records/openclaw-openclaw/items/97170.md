## Summary

Keep open. The reported bug is source-reproducible and the PR is focused, but the patch puts voice-call-specific config parsing into the generic provider runtime and the external PR still lacks after-fix real Gateway proof.

## What This Changes

The PR teaches realtime transcription capability resolution to cold-load providers from voice-call streaming config and adds a regression test for an xAI streaming provider when only OpenAI is active.

## Best Possible Solution

Keep the bug fix direction, but route configured streaming provider IDs through Talk/voice-call-owned selection into generic capability resolution and require real Gateway proof before merge.

## Reproduction Assessment

Yes from source inspection: current main can return the active OpenAI realtime transcription registry while talk.catalog still reports voice-call streaming activeProvider as xAI. I did not run a live Gateway repro, so this is source-reproducible rather than executed locally.

## Solution Assessment

No: the PR targets the right failure, but the implementation is not the best boundary because it duplicates voice-call config parsing inside the generic plugin resolver. A better fix keeps provider-request ownership in Talk/voice-call code and passes generic requested provider IDs into capability resolution.
