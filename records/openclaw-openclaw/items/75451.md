## Summary

Keep this PR open: the source-level performance issue is real and the patch is directionally aligned, but the external PR still needs qualifying real startup/discovery proof before merge. I also found one small perf follow-up in the new channel catalog discovery path.

## What This Changes

The PR adds optional precomputed plugin discovery parameters to bundled plugin registry helpers and reuses one discovery result inside the scoped contracts registry retry path.

## Best Possible Solution

Land the explicit discovery threading after real runtime proof is added, with the channel catalog helper avoiding eager install-index reads when a caller supplies `discovery`.

## Reproduction Assessment

Yes, at source level: current main's scoped contracts retry loop can call a helper that performs `discoverOpenClawPlugins` internally on each retry. I did not establish measured startup timing in this read-only review.

## Solution Assessment

Mostly yes: explicit function-scoped discovery reuse fits the plugin cache boundary and avoids hidden global caching. The patch should also avoid the eager channel install-record read and needs real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Avoid the eager install-index read:** `src/plugins/channel-catalog-registry.ts:38-39`
  - body: When `params.discovery` is supplied, `listChannelCatalogEntries` still calls `resolveInstallRecords(params)` before the nullish-coalescing fallback. For non-bundled callers that can perform the synchronous installed-index read even though the supplied discovery result is used directly; move install-record resolution into the fallback branch so the new reuse path actually skips that internal I/O.
