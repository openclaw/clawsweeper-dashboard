## Summary

Keep open. The PR is a plausible bounded perf cleanup for https://github.com/openclaw/openclaw/issues/82308, but it still needs real startup/discovery-path proof and has one small perf gap in the new catalog fast path.

## What This Changes

The PR adds optional `PluginDiscoveryResult` parameters to bundled plugin helpers, reuses one discovery result across the scoped contracts registry retry loop, and adds a changelog entry.

## Best Possible Solution

Land the explicit function-scoped discovery reuse after the contributor supplies redacted real startup/discovery proof and the catalog fast path avoids the needless installed-index read.

## Reproduction Assessment

Yes, at source level: current main's scoped contracts retry loop can call `loadBundledCapabilityRuntimeRegistry` twice, and that helper independently calls `discoverOpenClawPlugins({ env })`. I did not run a benchmark or runtime startup trace in this read-only review.

## Solution Assessment

Mostly yes: function-scoped explicit discovery threading matches the plugin cache policy and avoids a hidden global cache. The PR should also skip install-record loading when `listChannelCatalogEntries` receives a supplied discovery result.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Avoid loading install records when discovery is supplied:** `src/plugins/channel-catalog-registry.ts:36`
  - body: With `params.discovery` set and no bundled-only origin, this still calls `resolveInstallRecords(params)` before deciding to use the supplied discovery, so the new fast path can still do the synchronous installed-index disk read it was meant to avoid. Gate the install-record lookup behind the fallback discovery call or only run it when `params.discovery` is absent.
  - confidence: 0.86
