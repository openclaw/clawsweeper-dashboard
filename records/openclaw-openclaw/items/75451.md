## Summary

Keep open; the PR introduces a blocking freshness and security regression by adding a hidden process-wide plugin discovery cache that can return stale candidates after plugin install/removal, manifest edits, or permission changes, contrary to current plugin metadata cache boundaries.

## What This Changes

The PR adds a process-local `discoverOpenClawPlugins()` result cache keyed by roots/load paths/ownership/env, exports `clearPluginDiscoveryCache()`, updates discovery tests to clear the cache, and adds a changelog entry.

## Best Possible Solution

Keep discovery reads fresh and move the startup performance win into explicit `PluginMetadataSnapshot`, `PluginLookUpTable`, or manifest-registry reuse through the startup call chain, with snapshot replacement when config, plugin inventory, or installed-index state changes.

## Reproduction Assessment

Yes. Current main already has a focused freshness test that discovers a plugin, removes it, and expects the next discovery call to drop it; with the PR cache hit, that same path returns stale data unless `clearPluginDiscoveryCache()` is manually called.

## Solution Assessment

No. Caching inside `discoverOpenClawPlugins()` is not the best fix because it hides metadata state behind a global process cache; the maintainable path is explicit snapshot ownership that callers replace when inputs change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep plugin discovery reads fresh:** `src/plugins/discovery.ts:880-881`
  - body: The new cache returns the previous discovery result whenever the roots/env key matches, so a plugin installed, removed, edited, or made unsafe under the same roots is invisible until some caller knows to call `clearPluginDiscoveryCache()`. Current main explicitly tests that root changes are reflected on the next discovery call, and the plugin cache-boundary docs require installs, manifest edits, and load-path changes to be visible on the next metadata read.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff can bypass plugin path safety rechecks by serving stale cached candidates after filesystem permissions or ownership change.

Concerns:

- **[medium] Cached discovery can bypass path safety rechecks:** `src/plugins/discovery.ts:880`
  - body: Discovery currently re-stats each candidate root/source and blocks world-writable or suspicious-ownership paths. Returning cached candidates before rescanning means a later chmod/chown/removal under the same roots can remain trusted until an explicit cache clear, which is a concrete plugin loading security regression.
  - confidence: 0.86
