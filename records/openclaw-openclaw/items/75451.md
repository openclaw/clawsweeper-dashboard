## Summary

Keep this PR open for normal maintainer review. Current main still repeats plugin discovery inside the scoped capability retry path, and the proposed diff is a narrow explicit-object reuse path that matches the documented plugin cache boundary; I found no discrete correctness or security blocker in the provided final diff.

## What This Changes

The PR adds optional `PluginDiscoveryResult` parameters to bundled plugin source, channel catalog, and bundled capability runtime helpers, then reuses one function-scoped discovery result across scoped capability registry retry attempts with a changelog entry.

## Best Possible Solution

Land a narrow implementation that keeps `discoverOpenClawPlugins()` stateless, reuses only caller-owned discovery objects for the current flow, and lets CI or focused plugin contract tests validate the retry path before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: force the first scoped capability load to miss or fail, and the two-attempt retry loop calls the bundled runtime loader twice, with each loader call performing its own discovery scan.

## Solution Assessment

Yes. The explicit optional parameter and function-scoped reuse is the narrow maintainable solution here; a hidden module-level discovery cache would conflict with the plugin cache boundary.
