## Summary

Keep this PR open. The revision moved away from the earlier hidden `discoverOpenClawPlugins()` cache, but the new contracts registry helper still stores discovery in process-level module state, so later independent contract reads can reuse stale candidates instead of performing the fresh metadata read required by the plugin cache-boundary docs.

## What This Changes

The PR adds optional `PluginDiscoveryResult` plumbing to bundled capability/source/channel catalog helpers, reuses discovery across bundled contract registry capability loads, and updates the changelog.

## Best Possible Solution

Keep `discoverOpenClawPlugins()` stateless and make reuse local to one contract-registry materialization, or thread an existing `PluginMetadataSnapshot`/manifest registry through startup-owned flows with replacement on config, inventory, or installed-index changes. Add a focused regression test showing two independent contract-registry reads do not pin stale discovery after a plugin root removal or permission change.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to access one contract-registry export, change a discovered plugin root or its permissions, then access another export in the same process; the PR's module-level `contractsDiscovery` prevents a second discovery scan. Current main's direct discovery freshness test establishes the expected baseline behavior.

## Solution Assessment

No. The optional `discovery` parameter is a reasonable building block, but storing it in a module-scoped lazy variable is not the best boundary; the safer implementation is a call-scoped discovery object or an explicit startup-owned metadata snapshot that is replaced when inputs change.
