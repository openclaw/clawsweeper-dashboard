## Summary

Keep open. Current main does not implement the optional discovery threading, and the latest PR state appears to address the earlier ClawSweeper blocker by keeping reuse function-scoped instead of module-scoped. I did not find a concrete correctness or security regression in the final diff; it should proceed through normal review and CI rather than cleanup closure.

## What This Changes

The PR threads an optional `PluginDiscoveryResult` through bundled capability runtime, bundled source, and channel catalog helpers, then reuses one function-scoped discovery result across contract-registry retry attempts with a changelog entry.

## Best Possible Solution

Land a narrow implementation that keeps `discoverOpenClawPlugins()` stateless, shares only explicit caller-owned discovery results for the current flow, and validates the retry path plus discovery freshness with focused plugin tests and the changed gate.

## Reproduction Assessment

Yes. For the perf issue, a high-confidence static reproduction is the current retry path: force the first scoped capability load to miss and current main calls the bundled runtime loader twice, with each loader call performing its own discovery scan.

## Solution Assessment

Yes. The final PR's explicit optional parameter and function-scoped retry reuse is the narrow maintainable direction; a hidden module-level cache would not be, and the provided final diff no longer uses that shape.
