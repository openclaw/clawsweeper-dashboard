## Summary

Keep open: the per-flow discovery threading is plausible, but this external PR still lacks after-fix real behavior proof, and the new channel catalog fast path still performs an ignored installed-ledger read when `discovery` is supplied.

## What This Changes

The branch adds optional `PluginDiscoveryResult` parameters to bundled plugin registry/catalog helpers and reuses one discovery result across scoped contract registry retry attempts.

## Best Possible Solution

Keep the explicit per-flow discovery threading, skip ledger loading when channel catalog callers supply discovery, then require redacted real behavior proof and exact-head plugin registry validation before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the scoped capability retry loop can call the bundled capability registry twice, and that helper performs plugin discovery on each load; I did not run a real startup benchmark.

## Solution Assessment

Mostly yes. Function-scoped discovery ownership matches the plugin cache boundary and avoids a hidden global cache, but `listChannelCatalogEntries` should not load install records when a supplied discovery result will be used.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Skip ledger loading when discovery is supplied:** `src/plugins/channel-catalog-registry.ts:37`
  - body: With the new `discovery` option, this helper still calls `resolveInstallRecords(params)` before deciding whether to use the supplied discovery result. Callers that already did discovery can therefore still synchronously read the installed-plugin ledger even though those records are ignored, leaving part of the new fast path doing avoidable disk work. Move the ledger lookup into the fallback discovery branch or skip it when `params.discovery` is present.
  - confidence: 0.86
