## Summary

Keep this PR open. Current main still repeats plugin discovery in the scoped capability retry path, and the proposed explicit, caller-owned discovery object matches the documented plugin cache boundary without introducing a hidden cache. I found no blocking review finding in the supplied final diff; it should proceed through normal maintainer review and CI.

## What This Changes

The PR adds optional `PluginDiscoveryResult` parameters to bundled plugin helper APIs, reuses one discovery result across scoped capability retry attempts, and adds a changelog entry for the plugin performance improvement.

## Best Possible Solution

Land the explicit current-flow discovery reuse approach after CI, preferably with a small focused assertion for retry reuse if the maintainer wants regression coverage, while keeping `discoverOpenClawPlugins()` stateless and avoiding any module-level discovery cache.

## Reproduction Assessment

Yes. The static reproduction is current main's scoped retry helper: force the first bundled capability load to miss or fail, and the loop calls `loadBundledCapabilityRuntimeRegistry` twice; that helper performs its own discovery scan on each call.

## Solution Assessment

Yes. The PR's explicit optional parameter and function-scoped reuse is the narrow maintainable fix for the repeated scan; a hidden cache would conflict with the documented plugin cache boundary.
