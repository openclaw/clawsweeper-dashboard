## Summary

Keep this PR open. Current main still repeats plugin discovery through the scoped capability retry path, and the PR addresses that with explicit caller-owned discovery rather than a hidden cache; I found no discrete correctness or security blocker in the provided final diff, so the remaining step is maintainer review and CI.

## What This Changes

The PR adds optional `PluginDiscoveryResult` plumbing to bundled runtime, bundled source, and channel catalog helpers, reuses one function-scoped discovery result across scoped capability retry loads, and records the performance change in `CHANGELOG.md`.

## Best Possible Solution

Land a narrow explicit-discovery threading change if required checks pass, keeping `discoverOpenClawPlugins()` stateless and limiting discovery reuse to caller-owned current-flow objects rather than persistent process caches.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the scoped capability retry helper can invoke `loadBundledCapabilityRuntimeRegistry` twice, and each invocation performs its own `discoverOpenClawPlugins` scan.

## Solution Assessment

Yes. The proposed explicit optional parameter and function-scoped retry reuse is the narrow maintainable fix; a hidden module-level cache would be less safe because plugin installs, manifest edits, and path-safety checks must remain fresh on the next explicit metadata read.
