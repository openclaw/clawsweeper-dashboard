## Summary

Keep open: current main has the cron update contract and runtime pieces for real JSON null, but the tool-facing schema intentionally advertises sessionKey as a plain string for provider compatibility and the update path still accepts a literal "null" string as a real stored session key. That matches the reported symptom closely enough to warrant a focused fix rather than closure.

## What This Changes

The issue requests that cron tool update patches with patch.sessionKey set to null clear the stored cron job sessionKey instead of leaving the literal string "null".

## Best Possible Solution

Preserve the provider-compatible plain-string tool schema, but normalize the cron tool's nullable clear fields before cron.update so stringified null from strict tool surfaces clears sessionKey, with regression coverage and no nullable type-array schema regression.

## Reproduction Assessment

Yes from source for the persisted-string case: normalizeCronJobPatch preserves actual null but treats string "null" as a normal string, and applyJobPatch stores non-empty strings into job.sessionKey. I did not run a live webchat/tool call because this review is read-only.

## Solution Assessment

No, current main is not the best final state because the provider-compatible schema can cause or permit stringified null while runtime still stores it. The safer solution is a narrow normalization fix plus tests, not reverting the OpenAPI-compatible schema from https://github.com/openclaw/openclaw/pull/61221.
