## Summary

Keep this open. Current main still has a source-backed path where the agent-facing cron schema can produce the literal string "null" and normalization preserves it as a session key; the linked fixing PR is open but not merged.

## What This Changes

The issue asks for cron update patches with `sessionKey: null` to clear the persisted session key instead of saving the literal string "null".

## Best Possible Solution

Review and land a narrow fix that clears `sessionKey` when the agent-facing cron path produces the string "null", while preserving the existing OpenAPI-compatible schema and avoiding unrelated `agentId` compatibility changes unless explicitly approved.

## Reproduction Assessment

Yes by source inspection: current main exposes a string-only agent cron field, preserves the literal string "null" during normalization, and stores non-empty string session keys in the cron service. I did not run a live gateway round-trip in this read-only review.

## Solution Assessment

Yes, the maintainable fix belongs in the cron tool/sessionKey normalization boundary because the Gateway protocol and CLI already support real null clears. The linked PR should be reviewed and either narrowed or revised before this issue closes.
