## Summary

Keep this open. Current `main` clears a real JavaScript `null` at the service layer, but the agent-facing cron tool still exposes `sessionKey` as a string-only schema while telling the model that null clears it, and the normalization path preserves the string `"null"` as a real session key.

## What This Changes

The issue reports that updating a cron job with `patch.sessionKey: null` through the cron tool stores the literal string `"null"` instead of clearing the session binding.

## Best Possible Solution

Align the cron tool schema with the Gateway nullable contract and normalize `sessionKey` clear requests so both real null and model-produced `"null"` clear the field without changing `sessionTarget` routing.

## Reproduction Assessment

Yes by source inspection, but not by a live run. A real null clears today, while the cron tool schema can steer callers toward a string and current normalization preserves `"null"` as a session key.

## Solution Assessment

Yes, the narrow maintainable fix is in cron tool schema and cron patch normalization. The Gateway protocol already supports null, so this does not need a new feature or protocol redesign.
