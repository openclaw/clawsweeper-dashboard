## Label Justifications

- `P2`: This is a normal-priority channel/session orchestration improvement with real workflow impact but no current outage or emergency.
- `impact:session-state`: The request changes whether spawned subagents become persistent thread-bound sessions and how follow-up state is associated.
- `impact:message-loss`: Incorrect automatic thread/session binding can misroute child replies or follow-up messages away from the requester.

## Summary

Keep open. Current main and v2026.6.8 still require explicit `thread: true` for `mode: "session"`, and the existing channel threading adapter does not expose the requested subagent auto-injection policy; this needs maintainer product/API direction rather than cleanup closure.

## What This Changes

Request to add a channel-plugin threading policy so `sessions_spawn` can automatically supply thread/session parameters for channels that require them, without hardcoded channel names or prompt instructions.

## Best Possible Solution

Choose one explicit channel-owned session/thread contract, then update the SDK type, `sessions_spawn` schema/execution, channel metadata, docs, and focused tests together.

## Reproduction Assessment

Yes. Source inspection shows the no-technical-params path still fails or stays one-shot: `mode: "session"` requires explicit `thread: true`, and the current channel plugin contract has no auto-injection policy.

## Solution Assessment

Unclear. A plugin-owned policy seam fits the architecture, but maintainers first need to decide how it should coexist with the current fail-closed thread contract and the open decoupled persistent-session direction.
