## Label Justifications

- `P2`: This is a normal-priority plugin SDK/runtime feature with limited blast radius but real impact on subagent threading workflows.
- `impact:session-state`: The requested behavior controls whether spawned subagents become persistent session-bound work or one-shot runs.
- `impact:message-loss`: Missing or incorrect thread binding can make channel follow-up delivery unavailable or misrouted for spawned sessions.

## Summary

Keep open: current main still lacks the requested channel-owned spawn-default contract. `sessions_spawn` still derives `thread` and `mode` from explicit tool input, Feishu's hook only runs after a thread-bound spawn was requested, and the related open session/threading issues do not supersede this plugin SDK design question.

## What This Changes

The issue requests a channel plugin threading policy that can automatically default subagent spawn parameters such as `thread: true` and `mode: "session"`.

## Best Possible Solution

Define a backwards-compatible plugin SDK spawn-default contract resolved before `sessions_spawn` validation, opt in only channels with safe current-conversation binding support, and document/test the resulting defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source inspection is high-confidence that current main still requires explicit `thread`/`mode` input and lacks channel-owned spawn-default metadata.

## Solution Assessment

Unclear. The ownership boundary is sound, but the exact SDK field and defaulting semantics need maintainer design so they do not conflict with existing headless-session and channel-binding contracts.
