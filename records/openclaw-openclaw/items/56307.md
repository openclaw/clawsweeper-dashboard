## Summary

Keep open. Current main has channel-owned subagent thread-binding hooks, including Feishu support, but `sessions_spawn` still requires the caller to pass `thread: true`; there is no channel/plugin-declared policy that auto-injects `thread: true` or `mode: "session"` for native subagent spawns.

## What This Changes

Request a channel/plugin-declared threading policy so subagent spawns can automatically become thread-bound persistent sessions on channels that require that behavior.

## Best Possible Solution

Add a single generic, backward-compatible threading policy for native subagent spawns that reuses existing conversation binding and thread-binding concepts where possible, then let each channel opt in through its owned plugin/config surface with tests for Feishu and at least one existing thread-bound channel.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: call `sessions_spawn` with `mode: "session"` and no `thread` and current `spawnSubagentDirect` rejects it; call without both params and it stays a non-threaded run even on Feishu because the Feishu hook only runs after `threadRequested` is true.

## Solution Assessment

Unclear. The requested direction is valid, but the exact proposed new `threading` schema may duplicate existing `conversationBindings`, subagent hooks, and `threadBindings.spawnSubagentSessions`; maintainers should choose the smallest generic policy surface before a fix PR.
