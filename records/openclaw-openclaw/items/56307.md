## Summary

Keep open. Current main and the latest release still require explicit `thread: true` for thread-bound `sessions_spawn`, the tool does not auto-inject `thread` or `mode` from channel metadata, and the remaining request is a plugin SDK/session-routing product decision rather than cleanup-closeable.

## What This Changes

The issue requests a channel plugin threading policy that lets channel plugins declare automatic `sessions_spawn` thread/session parameter injection instead of relying on model-supplied parameters or channel-specific behavior.

## Best Possible Solution

Define one maintainer-approved channel-owned session/thread policy contract, then update the SDK surface, `sessions_spawn` schema/execution, channel metadata, docs, and focused tests in coordination with the non-thread persistent-session and channel-specific trackers.

## Reproduction Assessment

Yes, at source level. Current main derives `thread` only from explicit tool input, rejects `mode="session"` without that flag, and lacks the requested channel auto-injection contract.

## Solution Assessment

Unclear. A plugin-owned policy seam fits OpenClaw's architecture, but the exact fields, defaults, and interaction with https://github.com/openclaw/openclaw/issues/53548 should be chosen before implementation rather than added as a one-off boolean pair.
