## Summary

Keep open: current main has generic thread-binding policy and Feishu binding hooks, but it still depends on explicit `sessions_spawn` `thread`/`mode` inputs and lacks a channel-owned spawn-default contract; the related Feishu binding PR solved delivery/binding, not automatic injection.

## What This Changes

Request adds a channel/plugin-declared threading policy so `sessions_spawn` can auto-apply thread-bound session defaults for subagents without channel-name hardcoding.

## Best Possible Solution

Add a backwards-compatible plugin SDK spawn-default contract resolved before `sessions_spawn` mode/thread validation, then opt in Feishu only where current-conversation binding is supported and document the contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source inspection is high-confidence that current main still derives thread from `params.thread` and rejects `mode: "session"` without explicit `thread: true`.

## Solution Assessment

Unclear. The requested flags identify the right ownership boundary, but the maintainable design should probably reuse or extend the conversation-binding/spawn-default contract rather than overloading reply-threading adapter semantics.
