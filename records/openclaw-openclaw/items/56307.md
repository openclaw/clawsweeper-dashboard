## Label Justifications

- `P2`: This is a normal-priority plugin SDK/runtime feature with real subagent workflow impact but no current outage or urgent regression.
- `impact:session-state`: The request changes whether spawned subagents become persistent thread-bound sessions or one-shot runs based on channel-owned policy.
- `impact:message-loss`: Incorrect or missing thread binding can make spawned-session follow-ups invisible, misrouted, or unavailable to the requester.

## Summary

Keep open: current main and v2026.6.6 still require explicit `thread: true` for persistent thread-bound subagent sessions, and there is no plugin SDK/threading metadata contract that auto-injects `thread` or `mode`. Adjacent open items are narrower Telegram, safety, or session/thread decoupling work and do not supersede this API decision.

## What This Changes

Request to add channel/plugin-owned threading policy so `sessions_spawn` can default or inject thread-bound subagent parameters instead of relying on model-supplied `thread` and `mode` arguments.

## Best Possible Solution

Define an explicit backward-compatible channel/plugin spawn-default capability, preferably reusing existing `threadBindings` and `conversationBindings` where possible, then update `sessions_spawn`, SDK docs, channel implementations, and focused tests after maintainer approval.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives high confidence current main still makes `thread` an explicit tool input and rejects `mode: "session"` without it.

## Solution Assessment

Unclear pending maintainer design review. Moving ownership out of prompts and channel-name conditionals is directionally sound, but the exact SDK/API shape should likely reuse existing thread-binding metadata instead of adding parallel booleans without approval.
