## Summary

Keep open. Current main has removed the narrow Feishu-in-core smell by using generic subagent lifecycle hooks and Feishu-owned binding code, but it has not implemented the requested automatic `thread: true` / `mode: "session"` injection: native `sessions_spawn` still derives threading only from the caller's explicit tool input, and Feishu's hook no-ops unless `threadRequested` is already true.

## What This Changes

Request adding a channel-declared threading policy so `sessions_spawn` automatically applies `thread: true` and `mode: "session"` for channels such as Feishu without channel-specific core checks or prompt instructions.

## Best Possible Solution

Add a backwards-compatible, channel-owned subagent spawn-default contract that is resolved before `sessions_spawn` mode/thread validation, then opt Feishu in for supported conversations and document how third-party channel plugins should declare the same behavior.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: on current main, `sessions_spawn({ task })` enters native spawn with `thread` false and `mode` unset, while Feishu's binding hook only runs when `threadRequested` is already true; `sessions_spawn({ task, mode: "session" })` returns the existing `requires thread=true` error.

## Solution Assessment

Unclear. The proposed `ChannelPlugin.threading` flags are plausible, but current main already has `conversationBindings` and subagent lifecycle hooks, so the best maintainable fix may be a narrower generic spawn-default seam rather than the exact schema sketched in the issue.
