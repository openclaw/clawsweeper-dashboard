- `impact:session-state`: The request changes whether spawned subagents become persistent thread-bound sessions and how follow-up state is associated.

## Summary

Keep open. Current main and v2026.6.10 still require explicit `thread: true` for session-mode spawning, no channel plugin auto-injection contract exists, and the remaining work is a plugin API/session-routing product decision rather than cleanup-closeable.

## What This Changes

Request adds a channel plugin threading policy so channels can declare when `sessions_spawn` should auto-apply `thread: true` and `mode: "session"` for subagents.

## Best Possible Solution

Define one channel-owned session/thread contract, then update the SDK type, `sessions_spawn` schema/execution, channel metadata, docs, and focused tests while coordinating with the non-thread persistent-session discussion.

## Reproduction Assessment

Yes, at source level. Current main derives `thread` only from explicit tool input, rejects session mode without it, and lacks the requested channel auto-injection contract.

## Solution Assessment

Unclear. A plugin-owned policy seam fits the architecture, but the exact fields and defaults should be chosen with the broader persistent-session/thread-binding contract rather than added as a one-off boolean pair.
