## Summary

Keep open. Current main and v2026.6.10 still require explicit `thread: true` for thread-bound `sessions_spawn`, do not auto-inject `thread` or `mode` from channel metadata, and the remaining request is a plugin SDK/session-routing product decision rather than cleanup-closeable.

## What This Changes

The issue requests channel-plugin threading policy metadata so `sessions_spawn` can automatically choose thread-bound subagent parameters without hardcoded channel names.

## Best Possible Solution

Define one maintainer-approved channel-owned session/thread policy contract, then update the SDK surface, `sessions_spawn` schema/execution, channel metadata, docs, and focused tests alongside the broader persistent-session decision.

## Reproduction Assessment

Yes, at source level. Current main derives `thread` only from explicit tool input, rejects `mode="session"` without that flag, and lacks the requested channel auto-injection contract.

## Solution Assessment

Unclear. A plugin-owned policy seam fits OpenClaw's architecture, but the exact fields, defaults, and interaction with https://github.com/openclaw/openclaw/issues/53548 should be chosen before implementation rather than added as a one-off boolean pair.
