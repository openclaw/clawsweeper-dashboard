## Label Justifications

- `P2`: This is a normal-priority plugin SDK/runtime feature with limited blast radius but real impact on subagent threading workflows.
- `impact:session-state`: The requested policy controls whether spawned subagents become persistent session-bound work or one-shot runs.
- `impact:message-loss`: Incorrect or missing thread binding can make spawned-session follow-up delivery unavailable, hidden, or routed to the wrong conversation.

## Summary

Keep open: current main still lacks a channel-owned spawn-default contract for automatically setting `thread` or `mode` in `sessions_spawn`, and the adjacent open threading issues do not supersede this plugin SDK design question.

## What This Changes

Request to add channel/plugin-owned threading metadata so `sessions_spawn` can automatically default subagent spawns to `thread: true` and `mode: "session"` for channels that require it.

## Best Possible Solution

Define a backward-compatible plugin SDK spawn-default contract resolved from lightweight channel metadata before tool schema/execution, opt in only channels with safe binding support, and document/test explicit and default behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a plugin SDK/runtime feature request. Source inspection confirms current main lacks a channel-owned auto-default contract and still relies on explicit `thread`/`mode` input.

## Solution Assessment

Unclear until maintainer design review. The ownership boundary is sound, but the exact SDK metadata, schema exposure timing, and defaulting semantics need product/API judgment so they do not conflict with existing headless-session and channel-binding contracts.
