- `P2`: This is a normal-priority channel/session orchestration improvement with real workflow impact but no current outage or emergency.
- `impact:session-state`: The request changes whether spawned subagents become persistent thread-bound sessions and how follow-up state is associated.

## Summary

Keep open. Current main and v2026.6.8 still require explicit `thread: true` for session-mode subagent spawning, and the channel plugin threading surface does not expose the requested subagent auto-injection policy; this remains a product/API decision, not cleanup-closeable.

## What This Changes

The issue requests a channel-declared threading policy so subagent spawning can auto-inject thread/session parameters instead of hardcoding channel names.

## Best Possible Solution

Choose one explicit channel-owned session/thread contract, then update the SDK type, `sessions_spawn` schema/execution, channel metadata, docs, and focused tests together while coordinating with the decoupled persistent-session discussion.

## Reproduction Assessment

Yes. Source inspection shows the no-technical-params path still fails or stays one-shot: `mode: "session"` requires explicit `thread: true`, and the current channel plugin contract has no auto-injection policy.

## Solution Assessment

Unclear. A plugin-owned policy seam fits OpenClaw’s channel architecture, but maintainers need to decide how it should coexist with the current fail-closed thread contract and the open decoupled persistent-session direction.
