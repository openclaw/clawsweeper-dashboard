## Summary

Keep open: current main and v2026.6.6 still lack an exec-level known-heavy validation guard before Gateway host execution reaches the process supervisor. The merged Linux OOM-score work and local script heavy-check locks are useful adjacent mitigations, but they do not prevent unisolated heavy exec commands from starting; the direct implementation PR is closed unmerged and left policy/proof work behind.

## What This Changes

The issue asks OpenClaw to reject or isolate known-heavy validation commands before local Gateway exec runs them in the live Gateway resource domain.

## Best Possible Solution

Keep this issue open for a core exec resource-policy decision, then land a narrow guard or isolation contract with finite resource limits, docs, focused tests, and live Gateway refusal/allowed-path proof.

## Reproduction Assessment

Yes at source level: current main routes Gateway exec commands to the normal host process supervisor without a known-heavy command guard. No live OOM/restart reproduction was attempted in this read-only review.

## Solution Assessment

Unclear until maintainers decide the resource policy: an exec-entry guard is a plausible narrow fix, but the closed implementation PR showed the design needs finite isolation checks, docs, and live Gateway proof.
