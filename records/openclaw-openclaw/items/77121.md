## Summary

Keep open: current main still accepts unisolated Gateway exec commands and the related OOM-score, tsgo-pressure, heavy-check-lock, and broad-find work are complementary rather than canonical fixes for this exec-level resource policy gap.

## What This Changes

Request an exec-tool guard that rejects or reroutes known-heavy validation commands before they run unisolated in the live Gateway resource domain.

## Best Possible Solution

Define a Gateway exec resource policy that either rejects known-heavy validation commands with a clear isolation instruction or routes them through an approved resource-limited scope, with tests for direct, chained, shell-wrapped, light, and already-isolated commands.

## Reproduction Assessment

No high-confidence live reproduction was established for the full systemd OOM incident. Source inspection does show the relevant acceptance path: current main has no heavy-command preflight before Gateway exec spawns the command.

## Solution Assessment

Unclear whether the exact proposed blacklist, wrapper name, and OPENCLAW_EXEC_HEAVY_ENFORCEMENT bypass are the best long-term interface. The narrowest maintainable direction is likely an exec/resource-policy seam with maintainer-approved defaults and bypass semantics.
