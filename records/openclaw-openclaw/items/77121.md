## Summary

Keep open: current main still allows local Gateway exec to proceed to process launch without a heavy-command preflight, and the only matching guard PR was closed unmerged. The related OOM-score and script-heavy-check work is useful but complementary, not an exec-entry resource policy.

## What This Changes

The issue requests an exec-tool guard that blocks or resource-isolates known-heavy validation commands before they run in the live Gateway resource domain.

## Best Possible Solution

Define a maintained Gateway exec resource policy that rejects known-heavy unisolated validation commands or routes them through an approved finite resource-limited scope, with docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction of the full systemd OOM or host hang was established. Source inspection does give a clear current-main proof that local Gateway exec can reach process launch without the requested heavy-command preflight.

## Solution Assessment

Unclear as proposed. An exec-entry guard is a plausible narrow direction, but the command classifier, wrapper trust model, finite resource-limit validation, and bypass naming need maintainer/security policy decisions.
