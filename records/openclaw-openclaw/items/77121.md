## Label Justifications

- `P1`: The report describes production Gateway restarts and a host hang from agent-launched validation commands, which is an urgent availability risk needing maintainer policy review.
- `impact:crash-loop`: The affected behavior can mark the Gateway systemd unit oom-killed, restart the service, or leave the host unresponsive under memory pressure.

## Summary

Keep open: current main and v2026.6.6 still let local Gateway exec reach host process launch without a known-heavy validation guard, and the direct implementation attempt is closed unmerged. The remaining work is real, but it needs maintainer/security policy decisions on enforcement scope, isolation wrapper semantics, and any bypass/config surface.

## What This Changes

Request an exec-level safeguard that rejects or isolates known-heavy validation commands launched through host=gateway before they run in the live Gateway resource domain.

## Best Possible Solution

Define a core exec resource policy that either runs known-heavy host=gateway commands in a bounded resource scope or rejects them with clear isolation guidance, while preserving sandbox/node semantics and documenting any maintainer-approved bypass.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source and docs show local Gateway exec can still reach host process launch without the requested heavy-command guard, but the OOM restart/hard-hang path requires a memory-pressure Gateway setup that was not reproduced here.

## Solution Assessment

Unclear. An exec-entry guard is plausible, but maintainers still need to choose between rejection, resource-scoped execution, classifier scope, wrapper trust, and bypass semantics before this is the best fix.
