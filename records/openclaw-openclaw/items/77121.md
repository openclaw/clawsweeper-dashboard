## Summary

Keep open: current main still has no Gateway exec heavy-command preflight before process launch, and the only directly matching guard PR is closed unmerged. The merged Linux OOM-score mitigation and the open local-tsgo script guard are useful adjacent work, but they do not implement an exec-entry resource policy for arbitrary agent-generated validation commands.

## What This Changes

The issue requests a Gateway exec resource policy that blocks or isolates known-heavy validation commands before launching them in host=gateway.

## Best Possible Solution

Define a maintained Gateway exec resource policy that rejects known-heavy unisolated validation commands or routes them through an approved finite resource-limited scope, with docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction of the full systemd OOM or host hang was established. Source inspection does show current main can reach local Gateway process launch without the requested heavy-command preflight.

## Solution Assessment

Unclear as proposed. An exec-entry guard is plausible, but classifier scope, wrapper trust, finite limit checks, and bypass/config naming need maintainer/security policy decisions before implementation.
