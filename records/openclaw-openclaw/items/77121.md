## Summary

Keep open: current main and the latest shipped release still route local Gateway exec toward host process launch without a known-heavy command guard, and the exact implementation PR is closed unmerged. The adjacent OOM-score and local script heavy-check work are useful mitigations, but they do not enforce a resource policy for arbitrary agent-generated exec commands.

## What This Changes

Add a Gateway exec preflight/resource policy that blocks or isolates known-heavy validation commands before they run inside the live Gateway resource domain.

## Best Possible Solution

Define and document a maintained Gateway exec resource policy that rejects known-heavy unisolated validation commands or routes them through a verified finite resource-limited scope, with regression coverage and live Gateway proof.

## Reproduction Assessment

Yes at source level: current main can route local Gateway exec through existing policy/preflight checks to host process launch without a heavy-command classifier. I did not live-reproduce the systemd OOM restart or host hang.

## Solution Assessment

Unclear as a specific implementation: an exec-entry guard is plausible, but the maintainable fix needs an agreed classifier, finite isolation contract, docs, and bypass policy.
