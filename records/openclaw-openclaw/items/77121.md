## Summary

Keep open: current main and the latest release still allow local Gateway exec to start known-heavy validation commands without a hard exec-level guard or resource scope, while the merged OOM-score and systemd OOMPolicy work only reduce adjacent failure modes after a heavy child has already started.

## What This Changes

The issue requests a core exec policy or resource boundary to block or isolate known-heavy validation commands run through local Gateway exec.

## Best Possible Solution

Land a maintainer-approved core execution boundary that resource-scopes known-heavy local Gateway exec commands or rejects them with documented bypass semantics and focused regression coverage.

No high-confidence safe live repro was run; intentionally triggering OOM in the live Gateway resource domain can disrupt the host. Source inspection and the June 21 production incident show the current exec path still lacks the requested guard.

## Solution Assessment

Unclear: an exec-level preflight is a plausible mitigation, but maintainers need to choose between fail-closed heuristics, resource-scoped execution, and documented bypass/config behavior.
