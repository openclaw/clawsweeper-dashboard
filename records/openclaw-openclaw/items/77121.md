## Label Justifications

- `P1`: The report describes live Gateway restarts and host hangs from agent-launched validation commands, which is an urgent availability risk even though the exact policy still needs maintainer design.
- `impact:crash-loop`: The affected behavior can mark the Gateway unit oom-killed, restart the service, or leave the host unresponsive under memory pressure.

## Summary

Keep open: current main and the latest release still lack an exec-level guard for known-heavy validation commands, while the direct implementation attempt at https://github.com/openclaw/openclaw/pull/77140 is closed unmerged. The merged Linux OOM-score work at https://github.com/openclaw/openclaw/pull/70419 and the open local-script guard at https://github.com/openclaw/openclaw/pull/71652 are useful adjacent mitigations, but they do not prevent arbitrary agent-generated Gateway exec commands from starting unisolated.

## What This Changes

The issue asks OpenClaw to block or isolate known-heavy validation commands launched through the Gateway exec tool so they cannot destabilize the live Gateway resource domain.

## Best Possible Solution

Design a core exec resource policy that either runs known-heavy host=gateway commands in a bounded resource scope or rejects them with clear isolation guidance and a maintainer-approved escape hatch, while preserving sandbox, node-host, and approval semantics.

## Reproduction Assessment

No, not as a live failure in this review. Source and docs do show that current main can route local Gateway exec commands to process launch without a known-heavy validation guard, but the production OOM/hang path was not reproduced here.

## Solution Assessment

Unclear. The requested guard is plausible and aligned with the incident, but maintainers need to choose between exec-level rejection, resource-scoped execution, and the exact wrapper or bypass contract before this is a safe implementation task.
