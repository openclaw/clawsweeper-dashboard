## Label Justifications

- `P1`: The report now has multiple real incidents where agent-launched heavy commands disrupted or hung Linux Gateway hosts.
- `impact:security`: The issue concerns the safety boundary for arbitrary agent shell execution in the trusted Gateway process context.
- `impact:crash-loop`: The observed failure mode includes systemd marking openclaw-gateway.service OOM-killed, restarting it, and in one case hanging the host.

## Summary

Keep open: current main and the latest release still lack the requested exec-level heavy-command guard or resource-scope requirement, while the merged OOM-score mitigation and the open systemd OOMPolicy work only address adjacent failure modes.

## What This Changes

The issue requests a core exec policy that blocks or isolates known-heavy Gateway-host validation commands such as package-manager install/test/lint/typecheck/build runs before they execute in the live Gateway resource domain.

## Best Possible Solution

Keep the issue open for maintainer/security review, then land a core-owned execution boundary that either resource-scopes known-heavy local Gateway exec commands or rejects them with a documented, deliberate bypass path and regression coverage.

## Reproduction Assessment

No high-confidence safe live repro was run against current main; intentionally reproducing the OOM path would risk disrupting a Gateway host. Source inspection plus the June 21 production incident show the current exec path still lacks the requested guard.

## Solution Assessment

Unclear: an exec-level preflight is a plausible mitigation, but the best permanent fix needs maintainer choice between heuristic blocking, resource-scoped execution, and documented bypass/config semantics.
