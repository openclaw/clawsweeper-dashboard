## Summary

Keep open: current main still has the exec-level resource-policy gap, and open draft PR #77140 explicitly tracks this issue with closing syntax. The adjacent OOM-score and local heavy-check work reduces related failure modes but does not prevent unisolated heavy validation commands from launching through Gateway exec.

## What This Changes

The issue requests an exec-tool guard that rejects or isolates known-heavy validation commands before they run in the live Gateway service resource domain.

## Best Possible Solution

Review #77140 or a successor design and land a maintainer-approved exec resource policy that blocks or resource-isolates known-heavy validation commands while preserving explicit operator escapes and focused coverage.

## Reproduction Assessment

No high-confidence live reproduction was established for the full systemd OOM incident. Source inspection does show the relevant acceptance path: current main has no heavy-command preflight before local Gateway exec can launch a requested command.

## Solution Assessment

Unclear whether the exact proposed blacklist, wrapper name, and OPENCLAW_EXEC_HEAVY_ENFORCEMENT bypass are the best long-term interface. The maintainable direction is an exec/resource-policy seam with maintainer-approved defaults and bypass semantics, and the open draft PR is the right place to review that.
