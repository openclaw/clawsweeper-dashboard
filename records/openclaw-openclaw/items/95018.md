## Summary

Keep open for now: the PR targets a real reasoning message-loss bug, but the branch is not merge-ready because it removes the shared `isReasoning` guards for every generic dispatch channel, changes WhatsApp/generic expectations to deliver raw thinking text, and leaves reasoning exposed to shared TTS paths. The Discord-linked part is already fixed on current `main` by https://github.com/openclaw/openclaw/pull/95029, while the remaining Telegram work is better handled by a narrow Telegram-owned pass-through that preserves generic suppression.

## What This Changes

The PR removes both shared `isReasoning` suppression guards in `dispatch-from-config` and updates dispatch tests to expect WhatsApp/generic delivery of reasoning payloads.

## Best Possible Solution

Preserve generic-channel suppression, pass durable reasoning only to channel-owned reasoning handlers, exclude reasoning from TTS, and let the merged Discord fix plus the chosen Telegram PR own the linked issues.

## Reproduction Assessment

Yes, source-level: current main emits `isReasoning` block/final payloads for enabled reasoning, then shared dispatch suppresses those payloads before Telegram durable delivery. I did not run a live Telegram reasoning-model session in this read-only review.

## Solution Assessment

No. Removing the shared guards is a plausible part of the repair, but this PR is too broad because it makes generic dispatch deliver raw reasoning and does not protect shared TTS or account for the already-merged Discord fix.
