## Summary

Keep open: the underlying cron run-log OOM bug is still valid on current main, but this stale PR cannot merge as-is because it conflicts with current main, regresses current run-log behavior/security hardening, and lacks real behavior proof.

## What This Changes

This PR adds read-side cron run-log pruning, gateway prune-limit plumbing, cron/gateway tests, and a pi embedded harness mock sync.

## Best Possible Solution

Land or replace this with a current-main fix that bounds per-job and all-jobs run-log reads, honors cron.runLog settings, preserves fs-safe access and current search/response fields, and adds focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. By source inspection, creating an oversized runs/<jobId>.jsonl and calling readCronRunLogEntriesPage or cron.runs exercises the current-main path that reads the full JSONL file before pagination.

## Solution Assessment

No. The direction is useful, but this PR head is stale and has correctness/security regressions; the safer solution is a bounded tail read or serialized best-effort prune that preserves current fs-safe and query contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Grow tail reads until enough complete lines are found:** `src/cron/run-log.ts:166`
  - body: The prune window is capped at maxBytes or keepLines * 2048, but run-log entries can exceed 2 KB. That can rewrite fewer than keepLines recent records, so expand the tail in bounded chunks until enough complete JSONL lines are available or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when partial:** `src/cron/run-log.ts:184-185`
  - body: The prune path drops the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the line is complete, so pruning loses one valid recent entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery channel fields in cron run-log pagination, but this head narrows query text to summary, error, and job identifiers. Rebasing this as-is would make existing cron.runs searches stop matching those fields.
  - confidence: 0.84
