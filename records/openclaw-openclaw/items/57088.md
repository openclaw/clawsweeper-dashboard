Keep open; the underlying cron run-log OOM path still exists on current main, and this PR is a plausible fix path, but the current head has blocking correctness, stale-head security, and real-behavior-proof gaps.

## What This Changes

The PR adds read-side defensive pruning for cron JSONL run logs, threads prune limits through gateway request handling, and adds cron run-log/gateway tests plus one pi embedded harness mock update.

## Best Possible Solution

Repair or replace this PR with a current-main branch that bounds cron run-log reads for per-job and all-jobs pagination, preserves fs-safe writes and current search fields, and includes focused regression tests plus real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source path is to create an oversized `runs/<jobId>.jsonl` and call `readCronRunLogEntriesPage` or `cron.runs`; current main reads the complete file text before pagination.

## Solution Assessment

No. The direction is useful, but the current PR head should not land until tail pruning handles large records correctly, only drops partial first lines, and is rebased without losing current fs-safe and query behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Expand the tail until enough complete lines are available:** `src/cron/run-log.ts:166`
  - body: `tailBytes` assumes each JSONL entry is under 2 KB. Cron run-log records can exceed that, so pruning may see fewer than `keepLines` complete records and rewrite less history than configured; grow the tail window in bounded chunks until enough complete lines are collected or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This shifts the first streamed line whenever `startPos > 0`, but `startPos` can land immediately after a newline. In that case the first streamed line is already complete, so pruning loses a valid recent run-log entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields when rebasing:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this stale PR head narrows query text to summary, error, and job id. Rebasing this way would make existing `cron.runs` searches stop matching diagnostics and delivery channels.
  - confidence: 0.84
