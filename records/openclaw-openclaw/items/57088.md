## Summary

Keep open: the linked issue remains valid on current main and this PR is the active closing implementation, but the head is conflicting, lacks real behavior proof, and would regress newer run-log safety/search behavior if rebased mechanically.

## What This Changes

The PR adds read-side cron run-log pruning, runtime-configured prune limits in gateway request context, cron run-log tests, and a pi embedded harness mock update.

## Best Possible Solution

Rebase or replace this PR with a current-main fix that bounds per-job and all-jobs cron run-log read memory, honors cron.runLog settings, preserves fs-safe access and search fields, and includes focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an oversized runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the full file before pagination.

## Solution Assessment

No. The direction is useful, but this head is stale and has bounded-tail/data-loss issues; a safer repair should use bounded reads or race-safe pruning while preserving current fs-safe and query contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Grow tail reads until keepLines are complete:** `src/cron/run-log.ts:166`
  - body: The prune window is capped at maxBytes or keepLines * 2048, so recent JSONL entries larger than 2KB can leave fewer than keepLines complete records available before rewriting. Expand the tail in bounded chunks until enough complete lines are collected or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when partial:** `src/cron/run-log.ts:184-185`
  - body: The prune path drops the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the line is complete, so pruning loses a valid recent entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this head narrows query text to summary, error, and job identifiers. Rebasing this as-is would make existing cron.runs searches stop matching diagnostics and delivery channels.
  - confidence: 0.84
