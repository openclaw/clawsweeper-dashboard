## Summary

Keep open. Current main still has the unbounded cron run-log read paths this PR targets, but the PR head is not mergeable as-is because its streaming prune can discard valid retained history and it is dirty against current main.

## What This Changes

The PR adds read-side defensive pruning for cron JSONL run logs, threads configured prune limits through gateway request handling, expands cron run-log tests, and adjusts a pi embedded-runner test harness mock.

## Best Possible Solution

Update this PR or create a narrow replacement that keeps read-side pruning serialized, best-effort, config-driven, streaming-safe, and compatible with current main run-log filtering, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. On current main, an oversized cron/runs/<jobId>.jsonl followed by readCronRunLogEntriesPage or readCronRunLogEntriesPageAll reaches full-file fs.readFile; the PR’s remaining defects are reproducible with recent entries larger than the fixed tail estimate or a tail start exactly after a newline.

## Solution Assessment

No. The direction is right, but this PR head is not the best mergeable solution until the tail reader expands until enough complete lines are available, only drops a truly partial first line, and is reconciled with current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expand the tail until enough complete lines are available:** `src/cron/run-log.ts:166`
  - body: tailBytes assumes each run-log JSONL entry is under 2 KB. Cron summaries and errors can produce longer records, so pruning can stream too small a suffix, drop the first partial line, and retain fewer than keepLines recent records. Grow the tail window in bounded chunks until keepLines complete lines are available or BOF is reached.
  - confidence: 0.91
- **[P2] Only drop the first streamed line when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This removes the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the first streamed line is already complete, and shifting it loses a valid recent run-log entry during prune. Check the byte before startPos before discarding.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/cron/run-log.ts:423-426`
  - body: This is a user-facing cron reliability fix, but the PR head does not add a CHANGELOG.md entry. Add an active-version Fixes entry and keep contributor credit policy compliant.
  - confidence: 0.86
