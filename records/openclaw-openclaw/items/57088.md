## Summary

Keep open. The linked cron run-log OOM bug remains source-reproducible on current main, but this PR head is conflicting, missing after-fix real behavior proof, and would regress current run-log search and fs-safe write behavior if merged as-is.

## What This Changes

The PR adds read-side cron run-log pruning before pagination, threads prune limits through gateway request handling, adds cron run-log/gateway tests, and updates one pi embedded test harness mock.

## Best Possible Solution

Repair or replace this PR with a current-main branch that bounds cron run-log read memory for per-job and all-jobs pagination, honors cron.runLog settings, preserves fs-safe writes and search fields, and includes focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source-level path is to create an oversized runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the complete file text before pagination.

## Solution Assessment

No. The direction is useful, but this PR head is stale and regresses current behavior; the maintainable fix should rebase onto current main and use a bounded read or race-safe prune while preserving existing query and fs-safe contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Grow the tail window until enough complete lines are kept:** `src/cron/run-log.ts:166`
  - body: The tail window assumes each JSONL entry fits under 2KB. Cron run-log records can exceed that, so pruning can see fewer than keepLines complete records and rewrite less history than configured; grow the window in bounded chunks until enough complete lines are collected or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This drops the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the line is already complete, so pruning loses a valid recent run-log entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields during rebase:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this head narrows query text to summary, error, and job identifiers. Rebasing this way would make existing cron.runs searches stop matching diagnostics and delivery channels.
  - confidence: 0.84
