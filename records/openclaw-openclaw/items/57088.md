## Summary

Keep open. The underlying cron run-log OOM bug is valid on current main and this PR is moving in the right direction, but the current PR head is not mergeable as-is because its bounded-tail prune can still discard valid recent history, it can drop a complete first line at a tail boundary, and the stale branch needs to preserve newer current-main run-log search behavior plus the required changelog entry.

## What This Changes

The PR adds serialized best-effort read-side pruning before cron run-log reads, threads configured prune limits through Gateway request handling, expands focused cron run-log tests, and updates a pi embedded-runner test harness mock.

## Best Possible Solution

Update this PR or create a narrow replacement that keeps the read-side prune serialized, best-effort, and config-driven; replaces the fixed tail estimate with a bounded expanding tail scan until enough complete lines are available or BOF is reached; only drops an actually partial first line; preserves current-main run-log filtering; and adds focused regression coverage plus the active changelog entry.

## Reproduction Assessment

Yes. On current main, creating an oversized `cron/runs/<jobId>.jsonl` and calling `readCronRunLogEntriesPage` or `readCronRunLogEntriesPageAll` exercises the unbounded read path; the PR’s remaining data-loss cases are reproducible with recent JSONL entries larger than the fixed tail estimate and with a tail start positioned immediately after a newline.

## Solution Assessment

No. The direction is maintainable, but the current PR head is not the best mergeable fix because the prune algorithm can drop valid retained history and the stale branch needs reconciliation with shipped current-main behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Expand the tail until enough complete lines are available:** `src/cron/run-log.ts:166`
  - body: `tailBytes` assumes each run-log JSONL entry is under 2 KB. Cron summaries or errors can exceed that, so pruning can stream too small a suffix, drop the first partial line, and rewrite fewer than `keepLines` recent records. Grow the tail window in bounded chunks until `keepLines` complete lines are available or BOF is reached.
  - confidence: 0.91
- **[P2] Only drop the first streamed line when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This removes the first streamed line whenever `startPos > 0`, but `startPos` can land immediately after a newline. In that case the first line is already complete, and shifting it loses a valid run-log entry during prune. Check the byte before `startPos` before discarding.
  - confidence: 0.88
- **[P2] Preserve delivery-channel run-log search:** `src/cron/run-log.ts:437`
  - body: Current main lets `cron.runs` query match delivery intended/resolved/message-tool channels, but this PR head's reader only searches summary, error, and job fields. A rebase or merge resolution needs to keep the current delivery-field search so run-history filtering does not regress.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cron/run-log.ts:423-426`
  - body: This is a user-facing cron fix, but the PR head does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix changes to add an active-version changelog entry with contributor credit.
  - confidence: 0.86
