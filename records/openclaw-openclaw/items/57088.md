## What This Changes

This PR adds read-side defensive pruning for cron JSONL run logs, threads prune limits through gateway request handling, updates tests, and adjusts one pi embedded-runner test harness mock.

## Best Possible Solution

Repair or replace this PR with a current-main branch that bounds cron run-log reads without OOM risk, preserves fs-safe and filtering behavior, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create an oversized `runs/<jobId>.jsonl`, then call `readCronRunLogEntriesPage` or `readCronRunLogEntriesPageAll`; current main reads the whole file before pagination. ## Solution Assessment

No. The direction is right, but the current PR head should not land until pruning expands its tail window until enough complete lines are available, drops only a truly partial first line, and is reconciled with current main’s fs-safe and filtering changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Expand the tail until enough complete lines are available:** `src/cron/run-log.ts:166`
  - body: `tailBytes` assumes each run-log JSONL entry is under 2 KB. Cron summaries and errors can produce longer records, so pruning can stream too small a suffix, drop the first partial line, and retain fewer than `keepLines` recent records. Grow the tail window in bounded chunks until `keepLines` complete lines are available or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This removes the first streamed line whenever `startPos > 0`, but `startPos` can land immediately after a newline. In that case the first streamed line is already complete, and shifting it loses a valid recent run-log entry during prune. Check the byte before `startPos` before discarding.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields when rebasing:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this stale PR head narrows the query text to summary, error, and job id. Rebasing or resolving conflicts this way would make existing `cron.runs` searches stop matching diagnostics and delivery targets.
  - confidence: 0.84
