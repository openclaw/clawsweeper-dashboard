## Summary

Keep open. The linked cron run-log OOM bug is still present on current main, and this PR is a plausible implementation path, but the current head is dirty, lacks after-fix real behavior proof, and would regress current main’s run-log search and fs-safe behavior if rebased as-is.

## What This Changes

This PR adds read-side cron run-log pruning, threads prune limits through gateway request handling, and adds cron/gateway tests plus a pi embedded test harness mock update.

## Best Possible Solution

Rebase or replace this PR with a current-main implementation that bounds cron run-log read memory for per-job and all-jobs pagination, preserves fs-safe access and current search fields, and includes focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Create an oversized runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the full JSONL text before pagination.

## Solution Assessment

No. The direction is useful, but the current PR head is stale and regresses current-main behavior; a safer fix should use bounded reads or race-safe pruning while preserving fs-safe and query contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Grow tail reads until keepLines are complete:** `src/cron/run-log.ts:166`
  - body: The prune window is capped at maxBytes or keepLines * 2048, so recent JSONL entries larger than 2KB can leave fewer than keepLines complete records available before rewriting. Expand the tail in bounded chunks until enough complete lines are collected or BOF is reached.
  - confidence: 0.91
- **[P2] Drop the first streamed line only when partial:** `src/cron/run-log.ts:184-185`
  - body: The prune path drops the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the line is complete, so pruning loses a valid recent entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this head narrows query text to summary, error, and job identifiers. Rebasing this as-is would make existing cron.runs searches stop matching diagnostics and delivery channels.
  - confidence: 0.84
