Repair or replace this PR with a current-main branch that bounds cron run-log read memory for per-job and all-jobs pagination, honors existing cron.runLog settings, preserves fs-safe writes and search fields, and includes focused regression tests plus redacted real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Grow the tail window until enough complete lines are kept:** `src/cron/run-log.ts:166`
  - body: tailBytes assumes each JSONL entry is under 2 KB. Cron run-log records can exceed that, so pruning may see fewer than keepLines complete records and rewrite less history than configured; grow the tail window in bounded chunks until enough complete lines are collected or BOF is reached.
  - confidence: 0.91
- **[P2] Only drop the first streamed line when it is partial:** `src/cron/run-log.ts:184-185`
  - body: This shifts the first streamed line whenever startPos > 0, but startPos can land immediately after a newline. In that case the first streamed line is already complete, so pruning loses a valid recent run-log entry.
  - confidence: 0.88
- **[P2] Preserve current run-log search fields during rebase:** `src/cron/run-log.ts:437`
  - body: Current main searches diagnostics and delivery fields in cron run-log pagination, but this stale head narrows query text to summary, error, and job id. Rebasing this way would make existing cron.runs searches stop matching diagnostics and delivery channels.
  - confidence: 0.84
