## Summary

Keep this issue open. Current main still has the reported docs/schema mismatch: the docs, help text, TypeScript config type, and runtime resolution all expose `tools.exec.approvalRunningNoticeMs`, but the strict global `tools.exec` Zod schema still omits it, so the documented global key remains rejected. The linked fix PR #57293 is open and unmerged, so the issue should stay open until that PR or an equivalent narrow fix lands.

## What This Changes

The issue asks OpenClaw to align the documented global `tools.exec.approvalRunningNoticeMs` config key with the runtime schema, either by accepting the key or by removing the documented path.

## Best Possible Solution

Accept the documented global key by moving `approvalRunningNoticeMs` into `ToolExecBaseShape`, regenerating `src/config/schema.base.generated.ts`, and adding regression coverage that global `ToolsSchema` accepts `exec.approvalRunningNoticeMs` while still rejecting unknown `tools.exec` keys. Keep the docs and runtime behavior aligned rather than removing the documented knob.

## Reproduction Assessment

Yes. The high-confidence reproduction path is `ToolsSchema.parse({ exec: { approvalRunningNoticeMs: 3000 } })` or any root config load with that value: current main routes global `tools.exec` through a strict schema that lacks the field, while the docs and runtime resolution still reference it.

## Solution Assessment

Yes for the config portion: the narrow maintainable fix is to move the field into the shared exec base shape and regenerate schema output. The linked PR implements that core direction, but maintainers should ensure it is reviewed as a narrow config/schema change or split from unrelated systemd edits.
