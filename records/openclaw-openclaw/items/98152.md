Full review comments:

- **[P1] Preserve lifecycle lookups for scoped install records:** `src/plugins/installs.ts:90`
  - body: This stores agent-scoped installs under keys like `pluginId:agentId`, but current index and update code still look up records by the manifest plugin id. As a result, scoped installs can lose install metadata in the installed index and be treated as a different plugin id during update/repair flows. Use a record shape that keeps the map key compatible, or update every lifecycle consumer before writing compound keys.
  - confidence: 0.91
- **[P2] Expose scoped uninstall before writing scoped installs:** `src/cli/plugins-cli.ts:191`
  - body: The PR adds `plugins install --agent`, but `plugins uninstall` still has no matching `--agent` option or runtime wiring. Users who create `pluginId:agentId` records will not have a supported way to remove that scoped install or its agent workspace files.
  - confidence: 0.9
- **[P1] Do not remove shared plugin state during scoped uninstall:** `src/plugins/uninstall.ts:393`
  - body: Even when `agentId` is supplied, this function still removes `plugins.entries`, `plugins.allow`, `plugins.deny`, slots, and sometimes channel config by plain `pluginId`. A scoped uninstall would therefore disable or deconfigure global and sibling-agent installs of the same plugin instead of removing only the selected agent's record/files.
  - confidence: 0.88
- **[P2] Validate the requested agent before choosing an install root:** `src/cli/plugins-install-command.ts:1011`
  - body: `resolveAgentWorkspaceDir` falls back to a synthesized workspace for unknown non-default ids, so this line does not fail closed for a typo or unconfigured agent. The command should normalize and validate the agent id against configured agents before installing plugin code and persisting the scoped record.
  - confidence: 0.86
