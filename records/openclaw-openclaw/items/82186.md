## Summary

Keep open: the PR is a plausible follow-up to the merged gateway lifecycle hooks, but it currently has blocking config/docs contract defects and still lacks required real behavior proof from a real gateway restart setup.

## What This Changes

The PR raises gateway lifecycle hook timeout defaults, adds configurable timeout fields, updates hook documentation, and expands gateway close-handler tests.

## Best Possible Solution

Keep the larger timeout direction only if maintainers agree, then wire the config keys through schema/help/label coverage with finite bounded integer validation, correct the docs to the actual event-object handler API, and require real restart-hook proof before merge.

## Reproduction Assessment

Yes for the blocking review defects: source inspection shows the strict config schema rejects the new documented keys, and the hook runner invokes imported handler functions with an event object rather than shell environment variables. I did not run tests or a live restart because this review is read-only.

## Solution Assessment

No. The direction is plausible, but the proposed patch is not yet the narrow maintainable solution because the public config surface does not validate and the docs describe an unsupported shell handler contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Wire the timeout keys into config validation:** `src/gateway/server.impl.ts:965-970`
  - body: The PR reads `hooks.internal.gatewayShutdown.timeoutMs` and `gatewayPreRestart.timeoutMs` from runtime config, but `InternalHooksSchema` is strict and still only accepts `enabled`, `handlers`, `entries`, `load`, and `installs`. A config using the documented keys will fail validation before these overrides can be used.
  - confidence: 0.91
- **[P2] Remove the unsupported shell env-var contract:** `docs/automation/hooks.md:139-150`
  - body: Current hook discovery imports `handler.ts`/`handler.js` modules and calls them with the `InternalHookEvent` object; there is no shell `HOOK.md` runner or per-event injection for these `OPENCLAW_*` variables. Documenting them would lead users to write hooks that never receive the advertised values.
  - confidence: 0.88
