## Summary

Keep open. The PR addresses a plausible gateway lifecycle timeout need, but review found blocking gaps: the new config keys are not accepted by the strict config schema, the docs advertise shell env vars that the hook runner does not provide, and the external PR has only test claims rather than real behavior proof.

## What This Changes

The PR raises gateway lifecycle hook default timeouts, adds hooks.internal gateway timeout config fields, passes them into gateway shutdown, updates hook docs, and expands close-handler tests.

## Best Possible Solution

Keep the larger default budgets if maintainers agree, but make the config surface complete with schema/help/label coverage, validate finite bounded integer timeouts, correct the docs to the actual handler API, and require real restart-hook proof before merge.

## Reproduction Assessment

Yes for the blocking review defects: source inspection shows the strict config schema lacks the new gateway timeout keys while the PR reads them from runtime config, and the hook loader invokes JS/TS handlers with event objects rather than shell env vars. I did not run tests or a live gateway restart because this was a read-only review.

## Solution Assessment

No. The direction is plausible, but the proposed implementation is not yet the narrow maintainable solution because the public config keys must validate and the docs must match the actual hook runner contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Wire the timeout keys into config validation:** `src/gateway/server.impl.ts:965-970`
  - body: The PR reads `hooks.internal.gatewayShutdown.timeoutMs` and `gatewayPreRestart.timeoutMs` from runtime config, but `InternalHooksSchema` is strict and still only accepts `enabled`, `handlers`, `entries`, `load`, and `installs`. A config using the documented keys will fail validation before these overrides can be used, so the configurable timeout feature does not work.
  - confidence: 0.91
- **[P2] Remove the unsupported shell env-var contract:** `docs/automation/hooks.md:139-150`
  - body: Current hook discovery imports `handler.ts`/`handler.js` modules and calls them with the `InternalHookEvent` object; there is no shell `HOOK.md` runner or per-event injection for `OPENCLAW_EVENT`, `OPENCLAW_GATEWAY_SHUTDOWN_REASON`, or `OPENCLAW_RESTART_EXPECTED_MS`. Documenting these variables would send users to write hooks that never receive the advertised values.
  - confidence: 0.88
