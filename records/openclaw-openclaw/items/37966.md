## Summary

Keep this issue open: current `main` still routes bundled LiteLLM through the `openai-completions` transport, while the current resolver and Anthropic `cache_control` injector do not cover that default LiteLLM Anthropic proxy path.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic prompt-cache markers on the LiteLLM OpenAI-compatible path.

## Best Possible Solution

Add a LiteLLM-owned `wrapStreamFn` payload wrapper that injects Anthropic `cache_control` markers only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, while skipping absent config, `none`, and non-Anthropic LiteLLM models.

## Reproduction Assessment

Yes, at source level: current main routes LiteLLM through `openai-completions`, the resolver excludes that provider/API family, and the only Anthropic system-message `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the maintainable fix belongs in LiteLLM provider request shaping through `wrapStreamFn`, not in a broad resolver change for every `openai-completions` provider. The implementation should reuse the existing SDK payload-patch helpers where possible and add focused regression coverage for enabled, disabled, and non-Anthropic cases.

Current main still routes bundled LiteLLM through `api: "openai-completions"`, so explicit `agents.defaults.models["litellm/claude-..."] .params.cacheRetention` does not produce Anthropic `cache_control` markers. Implement the fix at the LiteLLM provider boundary, preferably with a `wrapStreamFn` in `extensions/litellm/index.ts` using public `openclaw/plugin-sdk/provider-stream` helpers rather than importing core internals directly. Inject Anthropic cache markers only when the selected LiteLLM model ref looks Anthropic/Claude-like and effective `cacheRetention` is `"short"` or `"long"`; do not inject for absent config, `cacheRetention: "none"`, invalid values, or non-Anthropic LiteLLM models. Keep core plugin-agnostic, avoid broad resolver changes for all `openai-completions` providers unless required by the implementation, and coordinate with https://github.com/openclaw/openclaw/pull/82973 if it lands first. Add focused regression coverage for LiteLLM string/array system message payloads, non-Anthropic models, no config, and explicit `none`; add a user-facing changelog fix entry if the repair PR is prepared for landing.
