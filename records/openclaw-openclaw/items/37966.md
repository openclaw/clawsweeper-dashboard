## Summary

Keep this issue open. Current main still does not honor explicit `cacheRetention` for the bundled LiteLLM OpenAI-compatible Claude route: the LiteLLM provider remains `openai-completions`, the resolver only passes explicit custom-provider cache settings for `anthropic-messages`, and the only payload-level Anthropic `cache_control` injection is still OpenRouter-scoped. The linked fix PR is closed unmerged and would not have added the needed LiteLLM payload marker path.

## What This Changes

The issue requests honoring explicit `cacheRetention`/`cacheControlTtl` for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers to LiteLLM OpenAI-compatible request payloads.

## Best Possible Solution

Add a narrow LiteLLM Anthropic prompt-cache pass-through for the bundled `litellm` `openai-completions` route: when explicit `cacheRetention` or `cacheControlTtl` resolves to `short` or `long` for a Claude/Anthropic-looking LiteLLM model, inject Anthropic `cache_control` markers into system/developer message content blocks before the proxy request, with no injection for `none`, missing config, non-Claude LiteLLM models, or arbitrary custom OpenAI-compatible providers.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `litellm/claude-opus-4-6` with explicit `cacheRetention`, then the bundled LiteLLM provider stays on `openai-completions`, `resolveCacheRetention()` returns `undefined`, and the only cache marker wrapper exits unless the route is OpenRouter.

## Solution Assessment

Yes, the narrowest maintainable fix is a LiteLLM-specific Anthropic payload wrapper or equivalent resolver path gated by explicit cache config and Claude-like model identity. A broad “honor cacheRetention for every provider” change would be riskier because OpenAI-compatible proxies differ in accepted payload fields.
