## Summary

Keep open: current main still leaves the bundled LiteLLM provider on the OpenAI-compatible route where explicit `cacheRetention` does not resolve to an Anthropic cache-retention family and no LiteLLM-specific `cache_control` payload wrapper exists. The earlier fix PR is closed unmerged and the currently open related PR is scoped to OpenRouter, not LiteLLM.

## What This Changes

The issue requests that explicit `cacheRetention` for LiteLLM-proxied Anthropic models produce Anthropic-compatible prompt-cache markers on the OpenAI-compatible LiteLLM request path.

## Best Possible Solution

Add a narrow LiteLLM Anthropic prompt-cache pass-through for explicit cache config, using existing payload marker helpers and tests/docs, while leaving arbitrary OpenAI-compatible proxies unchanged.

## Reproduction Assessment

Yes, by source inspection. Configure `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention`, and current main keeps bundled LiteLLM on `openai-completions`, returns no cache retention for that route, and only injects Anthropic `cache_control` markers for OpenRouter.

## Solution Assessment

Yes. The narrow maintainable repair is a LiteLLM-specific Anthropic/Claude-gated resolver and payload-marker path; the closed PR's generic retention pass-through alone is insufficient because the OpenAI-compatible payload still needs `cache_control` blocks.
