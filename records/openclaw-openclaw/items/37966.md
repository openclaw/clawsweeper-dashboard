## Summary

Keep open: current main still leaves bundled LiteLLM on the proxy-style OpenAI-compatible route with no prompt-cache hints, while cache retention and payload marker logic only cover Anthropic-family APIs, Bedrock Claude, Google, or OpenRouter-specific Anthropic markers. ## What This Changes

The issue requests that explicit `cacheRetention` be honored for LiteLLM-proxied Anthropic models and translated into Anthropic-compatible `cache_control` payload markers.

## Best Possible Solution

Add a narrow LiteLLM Anthropic-compatible prompt-cache pass-through in the agent runtime, gated by explicit cache config and Claude/Anthropic-looking LiteLLM model identity, with docs and changelog updates.

Fix openclaw/openclaw#37966 on current main. The bug is that explicit `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention` is ignored because bundled LiteLLM uses `openai-completions`; `resolveCacheRetention()` only accepts Anthropic-family APIs, Bedrock Claude, or Google, and `cache_control` payload injection is OpenRouter-only. Implement a narrow LiteLLM Anthropic pass-through: honor explicit `cacheRetention`/`cacheControlTtl` only for Claude/Anthropic-looking LiteLLM model refs on the OpenAI-compatible route, and inject Anthropic-compatible `cache_control` markers into system/developer prompt blocks. Do not enable markers for arbitrary OpenAI-compatible providers or non-Claude LiteLLM models, and keep direct Anthropic, Bedrock, OpenRouter, and Gemini behavior unchanged. Add regression coverage for resolver behavior and outbound payload mutation, then update LiteLLM/prompt-caching docs and `CHANGELOG.md` because this changes user-visible behavior.
