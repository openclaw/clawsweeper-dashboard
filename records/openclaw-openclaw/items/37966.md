## Summary

Keep this issue open: current main and latest release v2026.5.22 still leave the bundled LiteLLM text provider on the OpenAI-compatible route, while the cache-retention resolver and Anthropic cache-control payload wrapper only cover other provider families.

## What This Changes

Explicit `cacheRetention` should activate Anthropic prompt-cache markers for LiteLLM-proxied Claude models instead of being ignored.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper, or a small public SDK helper plus LiteLLM wrapper, that injects Anthropic `cache_control` only for explicit `short` or `long` retention on Claude-like LiteLLM refs and leaves absent, `none`, and non-Anthropic LiteLLM traffic unchanged.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM uses `openai-completions`, the resolver excludes that route, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is provider-owned LiteLLM request shaping, not a broad resolver change that would send Anthropic cache markers to every OpenAI-compatible provider.

Current main leaves bundled LiteLLM on `api: "openai-completions"`; explicit `cacheRetention` for `litellm/claude-*` is not resolved for that route and the only Anthropic `cache_control` injector is OpenRouter-scoped. Implement a provider-owned LiteLLM Anthropic cache-control wrapper, preferably in `extensions/litellm/stream.ts` wired via `wrapStreamFn` from `extensions/litellm/index.ts`, using public `openclaw/plugin-sdk/provider-stream-shared` helpers or adding a narrow public SDK helper only if needed.
