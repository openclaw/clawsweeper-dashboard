## Summary

Keep this issue open: current `main` and latest release `v2026.5.22` still route bundled LiteLLM through `openai-completions`, do not resolve explicit cache retention for that route, and only apply Anthropic `cache_control` markers on OpenRouter. The closed related PRs either did not merge or cover custom `anthropic-messages`/OpenRouter behavior, not the LiteLLM default path.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers to the OpenAI-compatible LiteLLM payload.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper or equivalent provider-owned seam that injects Anthropic `cache_control` only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, skips `none` and absent config, and covers non-Anthropic LiteLLM models with regression tests.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM uses `openai-completions`, the resolver excludes that route, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is provider-owned LiteLLM request shaping, not a broad resolver change that would send Anthropic cache markers to every OpenAI-compatible provider.
