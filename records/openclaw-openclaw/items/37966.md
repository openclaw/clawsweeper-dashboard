## Summary

Keep this issue open: current main and the latest release still route bundled LiteLLM through the OpenAI-compatible transport, while cache-retention resolution and Anthropic `cache_control` payload injection remain scoped to `anthropic-messages`, direct Anthropic/Bedrock/Vertex, Google, and OpenRouter-specific paths. The merged custom-Anthropic fixes help a different LiteLLM configuration but do not fix the reported default LiteLLM proxy path.

## What This Changes

The issue requests that explicit `cacheRetention` be honored for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers to the OpenAI-compatible LiteLLM payload.

## Best Possible Solution

Add a LiteLLM-owned or shared payload wrapper that injects Anthropic cache markers only for Anthropic-like LiteLLM refs with resolved retention `short` or `long`, while preserving absent config, `none`, and non-Anthropic LiteLLM models.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM remains `openai-completions`, cache retention resolution excludes that API family, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is a LiteLLM/provider-owned payload wrapper or shared helper reuse that activates only for explicit `short` or `long` retention on Anthropic-backed LiteLLM refs. It should leave absent config, `none`, and non-Anthropic LiteLLM models untouched.
