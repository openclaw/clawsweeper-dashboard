## Summary

Keep this issue open: current main still keeps LiteLLM on the `openai-completions` path, cache retention resolution only honors explicit custom Anthropic retention for `anthropic-messages`, and payload `cache_control` injection remains OpenRouter-scoped.

## What This Changes

Honor explicitly configured `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic prompt-cache markers on the LiteLLM OpenAI-compatible payload path.

## Best Possible Solution

Add LiteLLM-owned prompt-cache payload handling through the provider stream wrapper or a shared helper, with docs updated to describe explicit LiteLLM Anthropic retention support.

## Reproduction Assessment

Yes, at source level: LiteLLM remains `openai-completions`, the resolver excludes that API family, and the only Anthropic `cache_control` payload injector is OpenRouter-scoped. I did not perform a live LiteLLM backend cache-hit run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is a LiteLLM/provider-owned payload wrapper or shared helper reuse that activates only for explicit `short` or `long` retention on Anthropic-backed LiteLLM refs. It should leave absent config, `none`, and non-Anthropic LiteLLM models untouched.
