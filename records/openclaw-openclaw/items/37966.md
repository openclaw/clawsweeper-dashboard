## Summary

Current `main` still does not honor `cacheRetention` for the documented LiteLLM OpenAI-compatible Claude route: retention resolution excludes that route, and Anthropic payload markers require compatibility metadata the LiteLLM catalog does not provide. The directly linked fix PR is closed unmerged and only addressed retention propagation, not the required payload transformation.

## What This Changes

Honor explicit `cacheRetention` settings and emit Anthropic-compatible cache markers when LiteLLM routes an OpenAI-compatible model to Anthropic.

## Best Possible Solution

Add provider-owned LiteLLM model compatibility detection for known Anthropic routes, reuse the existing OpenAI-completions Anthropic cache-marker implementation, and cover the serialized payload plus non-Anthropic negative cases.

## Reproduction Assessment

Yes at source level: the documented LiteLLM Claude route uses `openai-completions`, while current retention resolution and Anthropic marker generation omit that default route. This read-only review did not execute a live LiteLLM-to-Anthropic request.

## Solution Assessment

No, the closed resolver-only proposal is not the best complete fix because it does not produce payload-level markers. The narrow maintainable solution is to use the existing model compatibility and OpenAI-completions payload contracts from the LiteLLM provider boundary.
