## Label Justifications

- `P2`: This is a source-reproducible provider request-shaping bug with cost impact for a bounded LiteLLM Anthropic setup, but it does not crash or block core runtime use.
- `impact:auth-provider`: The issue concerns provider/model routing and configured model parameters being ignored on a specific provider path.

## Summary

Keep this issue open: current main and the latest release still route bundled LiteLLM through the OpenAI-compatible transport, while the Anthropic cache-retention resolver and payload `cache_control` injector remain scoped to `anthropic-messages`/OpenRouter paths. The merged custom-Anthropic fixes help a different LiteLLM configuration but do not fix the reported default LiteLLM proxy path.

## What This Changes

Honor explicit `cacheRetention` for LiteLLM models proxying Anthropic by adding Anthropic `cache_control` markers on the OpenAI-compatible LiteLLM request path.

## Best Possible Solution

Add a LiteLLM-owned or shared payload wrapper that injects Anthropic cache markers only for Anthropic-like LiteLLM refs with resolved retention `short` or `long`, while preserving absent config, `none`, and non-Anthropic LiteLLM models.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM remains `openai-completions`, cache retention resolution excludes that API family, and the only Anthropic `cache_control` payload injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is a LiteLLM/provider-owned payload wrapper or shared helper reuse that activates only for explicit `short` or `long` retention on Anthropic-backed LiteLLM refs. It should leave absent config, `none`, and non-Anthropic LiteLLM models untouched.
