## Summary

Keep this issue open: current main still routes the bundled LiteLLM provider through `openai-completions`, and the current cache-retention resolver plus Anthropic `cache_control` injector still do not cover the reported LiteLLM Anthropic proxy path.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding payload-level Anthropic `cache_control` markers on the LiteLLM request path.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper through the provider `wrapStreamFn` seam that injects Anthropic cache markers only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, and skips absent config, `none`, and non-Anthropic LiteLLM models.

## Reproduction Assessment

Yes, at source level: current main routes LiteLLM through `openai-completions`, the resolver excludes that provider/API family, and the only Anthropic system-message `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: a provider-owned LiteLLM payload wrapper is the narrow maintainable fix because the default LiteLLM path needs Anthropic wire markers, not a broad core resolver change for every `openai-completions` provider.
