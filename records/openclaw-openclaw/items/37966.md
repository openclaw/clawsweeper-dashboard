## Summary

Keep this issue open: current main and v2026.5.12 still route the bundled LiteLLM provider through `openai-completions`, where explicit LiteLLM Anthropic `cacheRetention` neither resolves to stream options nor triggers payload-level `cache_control` injection. The related custom `anthropic-messages` fixes are merged, but the LiteLLM OpenAI-compatible proxy path reported here remains unhandled and the prior LiteLLM-specific PRs are closed unmerged.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers on the LiteLLM OpenAI-compatible request path.

## Best Possible Solution

Add LiteLLM-owned prompt-cache payload handling, preferably via the provider plugin using public provider-stream helpers, that injects Anthropic `cache_control` only for explicit short/long retention on Anthropic-backed LiteLLM refs and leaves absent config or `none` untouched.

## Reproduction Assessment

Yes, at source level: LiteLLM is still `openai-completions`, cache-retention resolution only accepts the custom `anthropic-messages` family, and the only payload `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit reproduction in this read-only review.

## Solution Assessment

Yes: the maintainable fix is a narrow LiteLLM/provider-owned wrapper or shared helper reuse, not broadening OpenRouter-specific behavior to every proxy. The fix should include payload-level regression coverage for short, long, none, absent config, and non-Anthropic LiteLLM models.
