## Summary

Keep this issue open: current main still routes LiteLLM through `openai-completions` without a LiteLLM-specific Anthropic `cache_control` payload wrapper, so explicit cache retention remains ineffective for the reported proxy path.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic-compatible cache markers on the LiteLLM OpenAI-compatible path.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper through the provider wrapper seam that injects Anthropic `cache_control` only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, while preserving OpenRouter and Bedrock semantics.

## Reproduction Assessment

Yes, at source level: LiteLLM is still `openai-completions`, the resolver excludes that provider/API family, and the only Anthropic system-message `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: a provider-owned LiteLLM payload wrapper is the narrow maintainable fix because this path needs Anthropic wire markers, not only a broad core resolver change for every OpenAI-compatible provider.
