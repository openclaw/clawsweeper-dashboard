## Summary

Current main still does not satisfy the reported LiteLLM openai-completions path: cacheRetention is only resolved for direct/custom anthropic-message APIs, and the only payload-level cache_control injector is scoped to OpenRouter routes. The closed related PR was useful context but did not land and would not have added the required LiteLLM payload wrapper.

## What This Changes

The issue requests honoring explicit cacheRetention for LiteLLM-proxied Anthropic models by injecting Anthropic cache_control markers into OpenAI-compatible payloads.

## Best Possible Solution

Add a provider-owned LiteLLM stream wrapper that honors explicit cacheRetention for Anthropic-backed LiteLLM models on the openai-completions path, with focused payload tests and docs for the supported opt-in boundary.

## Reproduction Assessment

Yes, at source level: LiteLLM models use openai-completions, resolveCacheRetention returns undefined for that API, and the only cache_control injector is OpenRouter-scoped. I did not establish live backend cache-hit reproduction.

## Solution Assessment

Yes, the maintainable direction is a LiteLLM provider-owned stream wrapper or helper reuse, not broadening core OpenRouter-specific logic to arbitrary proxies.
