## Summary

Keep this issue open: current main still does not apply Anthropic `cache_control` markers on the bundled LiteLLM `openai-completions` path, and the related merged fixes only cover custom `anthropic-messages` providers or OpenAI-style prompt-cache keys.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers to LiteLLM OpenAI-compatible payloads.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper that injects Anthropic `cache_control` only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, with tests for enabled, `none`, absent config, and non-Anthropic LiteLLM models.

## Reproduction Assessment

Yes, at source level: current main routes bundled LiteLLM through `openai-completions`, the resolver excludes that route, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is LiteLLM provider request shaping, not a broad resolver change for every OpenAI-compatible provider. The fix should reuse the existing payload-patch helpers and preserve explicit `cacheRetention: "none"` as an opt-out.
