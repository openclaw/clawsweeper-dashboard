## Summary

Keep open: current main still does not honor the reported LiteLLM OpenAI-compatible Anthropic path. The source now supports explicit cache retention for direct/custom Anthropic-message transports and OpenRouter-specific marker injection, but LiteLLM catalog models remain `openai-completions` and have no provider-owned `cache_control` wrapper.

## What This Changes

The issue requests that explicit `cacheRetention` on LiteLLM-proxied Anthropic models inject Anthropic `cache_control` markers on the LiteLLM OpenAI-compatible path.

## Best Possible Solution

Add a LiteLLM-owned prompt-cache wrapper or shared Anthropic-proxy helper that only activates for explicit Anthropic-backed LiteLLM cache config and covers the outbound payload shape with focused tests and docs.

## Reproduction Assessment

Yes, at source level: LiteLLM catalog models use `api: "openai-completions"`, current `resolveCacheRetention` does not return for that provider/API pair, and the only `cache_control` injector is OpenRouter-scoped. I did not establish live backend cache-hit reproduction.

## Solution Assessment

Yes, the maintainable direction is provider-owned LiteLLM handling or a shared Anthropic-proxy helper with an explicit opt-in boundary, not generic cache marker injection for all OpenAI-compatible proxies.
