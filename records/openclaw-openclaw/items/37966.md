## Summary

Keep this issue open: current main still drops the reported LiteLLM `openai-completions` path before any Anthropic `cache_control` payload markers can be injected, and the linked fix PR is closed without merge.

## What This Changes

Honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models so repeated requests can use Anthropic prompt caching through LiteLLM.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper or shared provider-stream helper that applies Anthropic `cache_control` markers only for explicit `cacheRetention` on Anthropic-backed LiteLLM model refs, with payload regression tests and prompt-caching docs.

## Reproduction Assessment

Yes, at source level: LiteLLM uses `openai-completions`, `resolveCacheRetention` excludes that API family for explicit non-Bedrock cache config, and the only Anthropic `cache_control` payload injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow provider-owned LiteLLM payload wrapper or shared helper reuse, not broadening OpenRouter-specific behavior to all proxies. The repair should assert outgoing payload markers and preserve the explicit-config boundary.

Add regression coverage that inspects the outgoing payload on `litellm/claude-*` over `openai-completions`, preserves no-op behavior without explicit config, and keeps OpenRouter/Bedrock/custom `anthropic-messages` behavior intact.
