## Summary

Keep this issue open: current `main` still does not cover the reported LiteLLM `openai-completions` path. The custom `anthropic-messages` cache-retention fix has landed, but LiteLLM is still registered as OpenAI-compatible, the cache resolver returns no retention for that path, and the only Anthropic `cache_control` payload injector remains OpenRouter-scoped.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by injecting Anthropic `cache_control` markers on the LiteLLM OpenAI-compatible request path.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper or shared helper use that applies Anthropic `cache_control` markers only for explicit `cacheRetention` on Anthropic-backed LiteLLM model refs, with focused payload tests and docs for the supported opt-in boundary.

## Reproduction Assessment

Yes, at source level: LiteLLM models still use `openai-completions`, the cache resolver returns no retention for that API, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not establish a live LiteLLM backend cache-hit reproduction.

## Solution Assessment

Yes, the narrow maintainable path is provider-owned LiteLLM payload handling or helper reuse, not broadening OpenRouter-specific cache-marker policy to arbitrary proxies.
