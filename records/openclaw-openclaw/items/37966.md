## Summary

Keep this issue open: current main still does not cover the reported LiteLLM `openai-completions` path, and the related fix attempt at https://github.com/openclaw/openclaw/pull/38221 is closed and unmerged. The source path is still narrow and repairable: LiteLLM is registered as an OpenAI-compatible provider, cache-retention resolution excludes that API family, and Anthropic `cache_control` injection remains scoped to OpenRouter routes.

## What This Changes

Explicit `cacheRetention` on LiteLLM-proxied Anthropic models should add Anthropic `cache_control` markers so prompt caching works through the proxy.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper, or shared provider-stream helper reuse, that applies Anthropic `cache_control` markers only when explicit `cacheRetention` targets an Anthropic-backed LiteLLM model ref, with regression tests and docs for the opt-in boundary.

## Reproduction Assessment

Yes, at source level: LiteLLM models still use `openai-completions`, `resolveCacheRetention` returns no value for that API family, and the only current Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit reproduction in this read-only review.

## Solution Assessment

Yes, the best fix shape is narrow provider-owned LiteLLM payload handling or shared helper reuse, not broadening OpenRouter-specific policy to every proxy. The fix should keep the explicit-config boundary and add tests that inspect the outgoing payload, not only whether a wrapper exists.

The observable bug is that explicit `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention` is ignored for LiteLLM Anthropic models because LiteLLM uses `api: "openai-completions"`, `resolveCacheRetention` does not classify that path as Anthropic-family, and no LiteLLM stream wrapper injects Anthropic `cache_control` payload blocks. Implement the narrow provider-owned fix: add or reuse a LiteLLM `wrapStreamFn`/provider-stream helper that patches outgoing OpenAI-compatible LiteLLM payloads only for explicit `cacheRetention` on Anthropic-backed LiteLLM model refs, preserving non-Anthropic LiteLLM behavior and avoiding OpenRouter policy drift. Add focused payload tests that prove `cache_control` appears for explicit LiteLLM Anthropic `long`/`short` retention and is absent for no config, `cacheRetention: "none"`, and non-Anthropic LiteLLM models. Update `docs/providers/litellm.md` and `docs/reference/prompt-caching.md` if the supported behavior changes. Add a single user-facing fix changelog entry if this repair lane owns changelog updates. Do not add a new config option, do not move LiteLLM policy into core hardcoded provider tables unless a shared SDK helper is the established local pattern, and do not broaden OpenRouter-specific cache-marker guards to arbitrary proxies.
