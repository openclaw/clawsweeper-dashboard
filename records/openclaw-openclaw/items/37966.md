## Summary

Keep this issue open: current main still leaves bundled LiteLLM text models on the OpenAI-compatible route, while the cache-retention resolver and Anthropic cache-control payload injection only cover direct/custom Anthropic-family routes, Bedrock, Google, and OpenRouter-specific paths. The related LiteLLM fix PRs are closed unmerged, and the merged custom-provider fixes do not cover bundled LiteLLM's default `openai-completions` provider route.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic/Claude models by adding Anthropic `cache_control` markers on the LiteLLM OpenAI-compatible request path.

## Best Possible Solution

Add a LiteLLM-owned Anthropic cache-control stream wrapper, or a small SDK helper plus LiteLLM wrapper, gated only on explicit `short` or `long` retention for Claude-like LiteLLM refs; keep absent, `none`, and non-Anthropic LiteLLM traffic unchanged and update the prompt-caching docs.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM uses `openai-completions`, the resolver excludes that route, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check in this read-only review.

## Solution Assessment

Yes: the maintainable fix is provider-owned LiteLLM request shaping for explicit Anthropic cache retention, not a broad resolver change that would send Anthropic markers to every OpenAI-compatible provider.
