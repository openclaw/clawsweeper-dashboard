## Summary

This should stay open: current `main` still excludes the LiteLLM OpenAI-compatible Claude route from both explicit retention resolution and Anthropic `cache_control` payload injection. The exact LiteLLM fix PRs are closed unmerged, while merged related work covers custom Anthropic APIs or prompt-cache-key opt-ins rather than this path.

## What This Changes

The issue asks OpenClaw to honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by resolving the setting and adding Anthropic-compatible `cache_control` payload markers on the OpenAI-compatible LiteLLM path.

## Best Possible Solution

Add a bounded LiteLLM/proxy prompt-cache path that resolves explicit retention for Anthropic-backed LiteLLM routes and injects Anthropic `cache_control` markers only when the route is known to support them.

## Reproduction Assessment

Yes at source level: the current LiteLLM Claude preset is `openai-completions`, while retention resolution and Anthropic payload-marker injection exclude that provider/API combination. I did not run a live LiteLLM-to-Anthropic request in this read-only review.

## Solution Assessment

No; the closed related PR's resolution-only approach is not the best complete fix because the OpenAI-compatible LiteLLM transport also needs payload-level `cache_control` markers and negative coverage for non-Anthropic LiteLLM routes.
