## Summary

Keep this issue open: current main still leaves the default LiteLLM OpenAI-compatible path without Anthropic `cache_control` injection, so the reported `cacheRetention` no-op remains source-reproducible.

## What This Changes

Honor explicit `cacheRetention` for LiteLLM-proxied Anthropic models by adding Anthropic `cache_control` markers to the LiteLLM OpenAI-compatible payload path.

## Best Possible Solution

Add a LiteLLM-owned stream wrapper that injects Anthropic `cache_control` only for explicit `short` or `long` retention on Anthropic-like LiteLLM model refs, with focused regression tests for enabled, disabled, and non-Anthropic cases.

## Reproduction Assessment

Yes, at source level: bundled LiteLLM still uses `openai-completions`, the cache-retention resolver excludes that provider/API family, and the only Anthropic `cache_control` injector is OpenRouter-scoped. I did not run a live LiteLLM backend cache-hit check.

## Solution Assessment

Yes, the narrow maintainable fix is a LiteLLM provider-owned payload wrapper, not a broad resolver change for every OpenAI-compatible provider. It should reuse the existing payload-patch helper shape and explicitly skip `none`, absent config, and non-Anthropic models.
