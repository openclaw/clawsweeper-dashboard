## Summary

Keep this open. Current main still logs configured bundled-MCP startup failures, disposes the failed server session, and continues with only the successful MCP tools, so the agent/channel still has no structured failed-server notification or `required` fail-fast behavior.

## What This Changes

The issue requests agent/user-visible diagnostics and optional required-server fail-fast behavior when a configured MCP server fails to start.

## Best Possible Solution

Implement a generic bundled-MCP startup diagnostics path: preserve redacted failed-server records in the session MCP catalog/runtime, surface a concise failure notice into the agent run context and active channel, support `mcp.servers.<name>.required: true` as a run-blocking config contract, and document/test the behavior without making it Telegram- or shell-server-specific.

## Reproduction Assessment

Yes. Configure a broken `mcp.servers` entry and start an embedded agent run; current code catches the startup/list failure, logs it, deletes that server session, and returns only successful tools to the agent.

## Solution Assessment

Yes for keeping this open and routing it as a generic runtime/config repair. The narrow maintainable solution is to add redacted bundled-MCP diagnostics and required-server semantics at the runtime/materialization/runner boundary, not a channel-specific workaround.

## Risks / Open Questions

- Startup diagnostics may contain secrets in command arguments, environment values, headers, URLs, stderr, or dependency error text; any prompt/channel payload must use stricter redaction than the current URL-only log redaction.
- Exact expected-vs-available tool names are not always knowable when a server fails before `listTools()`, so the first fix should avoid inventing missing tool names unless a safe previous catalog exists.
- `required` behavior needs a clear transport-agnostic contract so optional failures can degrade while required failures block the run cleanly across stdio, SSE, and streamable HTTP.
