## Label Justifications

- `P2`: The issue affects real agent capability and user trust when MCP servers fail, but the blast radius is a bounded runtime/config improvement rather than a core outage.
- `impact:session-state`: The agent session can proceed with a degraded tool context that is not represented in agent-visible state.

## Summary

Keep open: current main still logs bundled MCP startup/list failures, drops the failed server from the session catalog, and gives the agent only the successful tool subset. A related open PR explores degraded/required startup semantics, but nothing is merged on main and the public config, redaction, channel notification, and upgrade behavior still need maintainer direction.

## What This Changes

Request that bundled MCP startup failures be surfaced to the agent or user, with optional required-server fail-closed and tool-diff semantics.

## Best Possible Solution

Add a generic, redacted bundled-MCP failure diagnostic into runtime materialization and prompt/channel assembly, then define, document, and test required-server behavior with backward-compatible defaults.

## Reproduction Assessment

Yes, at source level. A broken configured MCP server follows the current `getCatalog()` catch path: log the startup/list failure, dispose and delete the failed session, and return only successful tools to the run.

## Solution Assessment

Unclear for the exact suggested config. The notification direction is maintainable, but `required`, `onStartFailure`, missing-tool wording, redaction, and upgrade behavior need maintainer-approved semantics before implementation.

## Risks / Open Questions

- MCP startup errors can include sensitive URLs, headers, command arguments, stderr, or env-derived text, so any agent/channel diagnostic needs strict redaction before user-visible delivery.
- Exact unavailable tool names may be unknowable when a server fails before `listTools()`, unless OpenClaw has a prior safe catalog or explicit configured expectations to compare against.
- Unconditional `required` fail-closed behavior could break existing workflows on upgrade, so maintainers need explicit default and opt-in semantics.
