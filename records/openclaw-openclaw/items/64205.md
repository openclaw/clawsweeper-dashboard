## Summary

Keep open: current main still logs and drops failed bundled-MCP servers, then starts the agent with only the remaining tools and no structured failed-server notice, required-server fail-fast path, or documented config semantics.

## What This Changes

Request notification and fail-fast semantics when configured MCP servers fail to start, so agents and users know MCP tools are unavailable.

## Best Possible Solution

Add a generic, redacted bundled-MCP startup diagnostic contract in the runtime/catalog/materialization/runner path, with documented `mcp.servers.<name>.required` behavior and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Configure a broken `mcp.servers` entry and start an embedded agent run; current code logs, disposes, and removes the failed server while returning only successful `catalog.tools`.

## Solution Assessment

Yes for keeping this open and fixing it generically. The maintainable direction is a redacted bundled-MCP diagnostics and required-server contract at the runtime/runner boundary, not a Telegram-specific workaround.

## Repair Work Prompt

Fix openclaw/openclaw#64205 by adding generic bundled-MCP startup failure reporting. Preserve redacted per-server startup/list failures from `createSessionMcpRuntime().getCatalog()` in a typed catalog/runtime diagnostic path, surface a concise notice into embedded agent context and the active channel, and implement `mcp.servers.<name>.required: true` so required server failure blocks the run with a user-visible error while optional failures degrade with notification. Update config types/schema/help/docs and add a user-facing CHANGELOG.md entry. Do not add Telegram-specific logic, do not expose raw command args/env/headers/URLs/stderr, and do not invent missing tool names when startup fails before `listTools()`.
