## Summary

Keep open: current main still logs and drops failed bundled MCP servers, so the agent receives only the reduced successful tool set. The requested notification and `required`/`onStartFailure` behavior is useful but introduces user-visible config and prompt/channel semantics that need maintainer direction before a repair PR.

## What This Changes

The issue requests agent-visible notification when a configured MCP server fails to start, plus optional required-server behavior and tool-diff awareness.

## Best Possible Solution

Add a generic, redacted, transport-agnostic bundled-MCP startup failure diagnostic that can reach the run, and document a minimal required-server behavior if maintainers accept that config contract.

## Reproduction Assessment

Yes, from source inspection. Configure a broken `mcp.servers` entry and current main catches the startup failure, logs it, removes that server session, and returns only successfully listed tools to the run.

## Solution Assessment

Yes for keeping this open, but the best implementation is a generic bundled-MCP runtime/catalog/materialization/runner diagnostic path rather than Telegram-specific notification code.
