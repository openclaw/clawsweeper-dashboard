## Summary

Keep open: current main still logs MCP startup failures, drops the failed server from the session catalog, and gives the agent only the successful tools. The requested user/agent notification is valid, but the `required` and `onStartFailure` semantics need maintainer product direction before implementation.

## What This Changes

The issue requests agent-visible MCP startup-failure diagnostics, optional required-server behavior, and tool-diff awareness when configured MCP tools are unavailable.

## Best Possible Solution

Add a generic, redacted bundled-MCP startup-failure diagnostic path into runtime materialization and prompt/channel assembly, then document and test required-server behavior only after maintainers accept that config contract.

## Reproduction Assessment

Yes, at source level. Configure a broken `mcp.servers` entry and current main catches startup/list failures, logs them, removes the failed session, and returns only successful tools to the run.

## Solution Assessment

Unclear for the exact suggested config. The notification direction is maintainable, but `required` and `onStartFailure` need maintainer-approved semantics, redaction rules, and upgrade behavior before implementation.
