## Summary

Keep open: current main still logs and drops failed bundled MCP servers, then starts with the reduced tool catalog and no agent-visible startup-failure diagnostic, required-server contract, or documented fail-fast behavior. The adjacent open MCP PRs touch related runtime lifecycle/performance areas but do not implement this issue.

## What This Changes

The issue requests agent-visible bundled MCP startup failure diagnostics, optional fail-fast required server semantics, and tool-gap awareness when configured MCP servers are unavailable.

## Best Possible Solution

Add a generic, redacted bundled-MCP startup diagnostic contract through runtime catalog/materialization and the embedded runner, document required-server behavior, and keep channel-specific delivery out of the MCP runtime itself.

## Reproduction Assessment

Yes, from source inspection. A broken `mcp.servers` entry reaches the runtime catch path, which logs and removes that server while returning only successful `catalog.tools` to the runner.

## Solution Assessment

Yes for keeping this open. The maintainable fix is a generic redacted runtime/config/docs contract, not a Telegram-specific workaround or a precise tool diff when discovery never completed.
