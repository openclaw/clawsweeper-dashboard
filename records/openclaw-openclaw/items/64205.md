## What This Changes

The issue requests agent/user-visible notification when a configured MCP server fails to start, with optional required-server semantics and tool-diff awareness.

## Best Possible Solution

Add a generic, redacted bundled-MCP startup diagnostic contract through the runtime/catalog/materialization/runner path, document `mcp.servers.<name>.required`, and avoid Telegram-specific handling.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64205. Current main catches bundled-MCP startup/list failures in `createSessionMcpRuntime().getCatalog()`, logs the failure, disposes the failed server, deletes it from the session map, and returns a catalog containing only successful servers/tools; the agent/channel only sees the reduced tool set. Add a generic, transport-agnostic startup-failure diagnostic path from the bundled-MCP runtime catalog through materialization and embedded runner prompt/channel handling. Implement and document `mcp.servers.<name>.required: true` as fail-fast agent-start behavior with a user-visible notification through the active channel; avoid adding `onStartFailure` unless maintainers explicitly choose that extra policy knob. Do not add Telegram-specific logic. Redact before any prompt/channel/log diagnostic payload: command args, env values, headers, URLs, stderr, dependency error text, tokens, and private paths where appropriate. If startup fails before `listTools()`, report the server-level failure rather than inventing missing tool names; only use a previous tool diff if there is a safe existing catalog contract. Update config types, schema/help/docs, tests, and changelog. Coordinate with open MCP runtime PRs https://github.com/openclaw/openclaw/pull/73792 and https://github.com/openclaw/openclaw/pull/79882 if they land first.
