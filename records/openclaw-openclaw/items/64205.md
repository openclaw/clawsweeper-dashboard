## Summary

Keep open: current main still logs and drops a failed bundled MCP server, then passes only successful tools into the run. The requested notification and required-server behavior are useful, but they add public config plus prompt/channel semantics that need maintainer direction.

## What This Changes

The issue asks OpenClaw to notify the agent or user when a configured MCP server fails to start and optionally block startup for required servers.

## Best Possible Solution

Add a generic, redacted bundled-MCP startup-failure diagnostic path into runtime/materialization/prompt assembly, then document required-server behavior if maintainers accept that public config contract.

## Reproduction Assessment

Yes, from source inspection. Configure a broken `mcp.servers` entry and current main catches the startup failure, logs it, removes the failed session, and returns only successfully listed tools to the run.

## Solution Assessment

Unclear for the exact suggested config. The notification direction is right, but `required` and `onStartFailure` need maintainer-approved semantics, redaction rules, and upgrade behavior before implementation.
