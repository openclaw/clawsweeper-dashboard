## Summary

Keep this open. Current main still logs bundled-MCP startup/list failures and then continues with only successful tools, while the catalog, materializer, runner prompt path, config schema, and docs do not carry failed-server diagnostics or `required` fail-fast semantics.

## What This Changes

The issue asks OpenClaw to notify the agent or block startup when a configured MCP server fails to start.

## Best Possible Solution

Implement generic bundled-MCP startup diagnostics with redacted failed-server records, agent-visible degraded-tool context, `mcp.servers.<name>.required: true` fail-fast behavior, and docs/tests without making it Telegram- or shell-server-specific.

## Reproduction Assessment

Yes. Configure a broken `mcp.servers` entry and start an embedded agent run; the current source path catches the startup/list failure, logs it, removes that server session, and presents only the reduced successful tool set to the agent.

## Solution Assessment

Yes for keeping this open and repairing it generically. The narrow maintainable fix belongs at the bundled-MCP runtime/materialization/attempt-runner/config boundary, not in a channel-specific workaround.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/64205. Current bundled MCP startup silently degrades after logging when a configured server cannot connect or list tools. Add a generic, redacted diagnostics path so embedded Pi agent runs know when configured MCP servers failed, and support `mcp.servers.<name>.required: true` as a run-blocking config contract. Expected boundary: extend MCP server config types/schema/help/docs; preserve failed startup/list records in the session MCP runtime/catalog or an adjacent typed runtime result; surface a concise redacted notice into the agent run context before the model call; for `required: true`, fail the run before model I/O and deliver a channel-visible error through the normal run failure path. Do not hardcode Telegram, shell, npm, or any bundled server id. Do not expose raw env values, headers, URLs with credentials/query tokens, command-expanded secrets, or stderr blobs. Add focused tests for failed optional server notification, failed required server blocking, redaction, and unchanged successful tool materialization. Add a user-facing `CHANGELOG.md` fix entry.
