## Summary

Keep this open. Current main still logs configured MCP startup failures and then silently returns only the successful MCP tools to the embedded agent path, with no failed-server payload, no `required` fail-fast semantics, and no agent/channel notification path.

## What This Changes

The issue requests user-visible/agent-visible diagnostics when configured MCP servers fail to start, plus per-server `required` behavior and missing-tool awareness.

## Best Possible Solution

Implement a generic bundled-MCP diagnostics path: preserve redacted failed-server records in the session MCP catalog/runtime, inject optional failure notices into the run context, block runs for `mcp.servers.<name>.required: true` with a user-visible channel error, and document/test the config contract across supported MCP transports.

## Reproduction Assessment

Yes. Configure a broken `mcp.servers` entry and start an embedded agent run; current code catches the connect/list failure, logs it, removes that server from the runtime, and returns only the remaining tools to the agent.

## Solution Assessment

Yes for keeping this open. The maintainable fix is a generic bundled-MCP runtime/config/prompt diagnostic path, not a Telegram-only or shell-server-specific workaround; tool diff reporting should be limited to data OpenClaw can know safely.

## Repair Work Prompt

Fix openclaw/openclaw#64205. Implement bundled MCP startup failure diagnostics for embedded Pi configured MCP servers. Preserve per-server startup/listTools failures in the session MCP catalog/runtime with redacted error text; optional failures should be surfaced to the agent prompt or first-run context so the model can tell the user that tools from that server are unavailable. Add typed `mcp.servers.<name>.required?: boolean` semantics: when true, a connect/list failure should abort/block the agent run with a user-visible delivery error through the existing run/channel error path. If exact expected tool names are unknown because the server never listed tools, report the server-level outage rather than inventing tool names. Update config types/schema/help/docs and add focused tests for stdio failure, HTTP/SSE or streamable-HTTP failure, optional vs required behavior, prompt diagnostics, and redaction.
