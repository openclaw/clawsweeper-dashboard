## Summary

Keep this issue open. Current main still lacks the requested WhatsApp pre-built link preview pass-through, has no opt-in high-quality preview config/socket wiring, and exposes only the single-message Gateway `send` method.

## What This Changes

The request asks for WhatsApp pre-built `linkPreview` pass-through, an opt-in high-quality preview setting, and a Gateway batch send RPC for bulk outbound messages.

## Best Possible Solution

Split the work: land a typed WhatsApp-owned pre-built `linkPreview` pass-through first, then separately design automatic preview fetching and Gateway batch sending with security and protocol docs.

## Reproduction Assessment

Yes. Source inspection on current main shows the WhatsApp send path drops preview metadata, the WhatsApp config/socket path has no preview-generation option, and Gateway exposes only single-message `send`.

## Solution Assessment

No, not as one combined change. The pre-built preview pass-through is narrow, but automatic URL fetching and a new batch RPC should be staged as separate maintainer decisions.

## Risks / Open Questions

- Automatic high-quality preview generation would make the Gateway-hosted WhatsApp session fetch arbitrary outbound URLs, so SSRF, redirect, DNS/private-network, timeout, response-size, proxy, and privacy behavior need explicit design.
- A batch Gateway send method needs protocol semantics for idempotency, partial failure, ordering, pacing/rate limits, mirroring, and per-message results before implementation.
- The issue combines three separable surfaces, so a single automated fix would risk mixing a narrow WhatsApp pass-through with security-sensitive fetching and a new protocol feature.
