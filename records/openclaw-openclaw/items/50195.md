## Summary

Keep this open: current main and the latest shipped release still lack the requested WhatsApp pre-built link-preview pass-through, opt-in automatic preview config, and public batch send RPC, and the related prior PRs are either reverted or unmerged.

## What This Changes

The issue requests WhatsApp pre-built `linkPreview` payload support, an opt-in `generateHighQualityLinkPreview` config, and a batch outbound send RPC for bulk WhatsApp/digest workflows.

## Best Possible Solution

Stage the work: add a typed WhatsApp-owned pre-built `linkPreview` pass-through first, then handle automatic URL fetching and batch Gateway sending as separate security/protocol decisions.

## Reproduction Assessment

Yes for the capability gap. Source inspection on current main shows no WhatsApp preview metadata path, no high-quality preview config/socket option, and only a single-message Gateway `send` RPC.

## Solution Assessment

No as one combined change. The pre-built `linkPreview` pass-through is a narrow first step, while automatic URL fetching and batch RPC behavior need separate security and protocol decisions.

## Risks / Open Questions

- Automatic high-quality preview fetching would make the Gateway host fetch URLs found in outbound WhatsApp text, so SSRF, private-network access, redirects, proxy behavior, timeouts, response-size limits, and logging/privacy need explicit design.
- A public batch send RPC changes Gateway protocol, idempotency, partial-failure reporting, transcript mirroring, and rate-limit semantics; it should not be bundled into the narrow WhatsApp preview pass-through without maintainer approval.
- The earlier per-message preview implementation was reverted, so reintroducing it should use current typed plugin/Gateway contracts rather than reviving the old web path directly.
