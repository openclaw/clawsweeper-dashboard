## Summary

Keep open. Current main still has the documented bug, and this PR is a plausible narrow fix, but it is not merge-ready because the branch has a portable test blocker and lacks after-fix real behavior proof for the interactive CLI flow.

## What This Changes

Adds a peer-workspace suggestion helper for new `agents add` prompts, wires it into the wizard, re-exports/tests it, and adds a changelog entry.

## Best Possible Solution

Land one compatibility-preserving fix that gives brand-new `agents add` prompts the documented peer-level workspace default while preserving explicit workspaces, existing-agent updates, state-dir isolation, and intentional shared-base configurations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: setup stores `agents.defaults.workspace`, the wizard uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver appends a non-default agent id under that workspace.

## Solution Assessment

No as submitted. The wizard-only strategy is a narrow maintainable direction, but this branch needs the portable test environment stub and after-fix real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stub the home environment in the wizard prefill test:** `src/commands/agents.add.test.ts:80-109`
  - body: `suggestPeerAgentWorkspaceDir` compares the configured default against `resolveDefaultAgentWorkspaceDir(process.env)`, but this test hardcodes `/Users/me/.openclaw/workspace` without stubbing `HOME` or `OPENCLAW_HOME`. On Linux CI the helper will see a different documented default and fall back to `/Users/me/.openclaw/workspace/freshbug`, so the peer-level assertion fails.
