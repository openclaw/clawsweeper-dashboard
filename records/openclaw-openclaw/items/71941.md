## Summary

Keep this PR open: current main still has the documented workspace-prefill bug, and the proposed wizard-only fix is plausible, but the submitted patch has a portable-test blocker and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a peer-workspace suggestion helper, uses it for brand-new `agents add` prompts, re-exports and tests it, and adds a changelog entry for the workspace default fix.

## Best Possible Solution

Land one compatibility-preserving fix that makes the add-agent wizard default match the documented peer workspace layout while preserving explicit workspaces, existing-agent updates, state-dir isolation, and intentional shared-base setups.

## Reproduction Assessment

Yes, by source inspection. Setup stores `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver appends a non-default id while docs expect a sibling workspace.

## Solution Assessment

No, not as submitted. The wizard-only production boundary is plausible, but the PR needs the home environment stubbed in the wizard test and redacted after-fix CLI proof before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stub the home environment in the wizard prefill test:** `src/commands/agents.add.test.ts:80-83`
  - body: `suggestPeerAgentWorkspaceDir` compares the configured default against `resolveDefaultAgentWorkspaceDir(process.env)`, but this test hardcodes `/Users/me/.openclaw/workspace` without stubbing `HOME` or `OPENCLAW_HOME`. On Linux CI the helper sees the runner home instead, falls back to `/Users/me/.openclaw/workspace/freshbug`, and fails the peer-level assertion.
