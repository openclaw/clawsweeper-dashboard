## Summary

Keep this PR open. Current main still has the external streaming-steer transcript-continuity problem, but the proposed patch is not mergeable yet because it misresolves external channel identity, misses metadata for timestamp-prefixed session files, is conflicting with current main, and lacks required live Telegram proof.

## What This Changes

The PR routes external-channel streaming steer turns through the follow-up queue, preserves internal webchat embedded steering, and adds Telegram stale-parent transcript doctor repair logic and tests.

## Best Possible Solution

Rework the PR on current main so external channel identity comes from Provider/messageProvider, doctor metadata resolves through stored session identity, and the contributor supplies redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `runReplyAgent` enters embedded steering whenever `effectiveShouldSteer && isStreaming`, before the follow-up queue path can run for Telegram-style external turns.

## Solution Assessment

No, not yet. The direction is right, but this patch should prefer Provider/messageProvider over Surface for channel identity and resolve doctor metadata through stored session identity before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Prefer Provider over Surface when resolving steer channel:** `src/auto-reply/reply/agent-runner.ts:900-904`
  - body: When `OriginatingChannel` is absent, this fallback checks `Surface` before `Provider` or `followupRun.run.messageProvider`. A relayed external context with `Surface="webchat"` and `Provider="telegram"` will still be treated as internal and routed into live embedded steering, recreating the stale in-flight transcript branch this PR is meant to avoid.
  - confidence: 0.92
- **[P2] Resolve metadata from the stored session file:** `src/commands/doctor-session-transcripts.ts:409`
  - body: `loadSessionMetadata` keys entries by `sessionId`, but this lookup uses the JSONL basename. Forked transcripts are written as timestamp-prefixed session files and persisted in `sessionFile`, so `doctor --fix` misses metadata for those Telegram DM transcripts and skips the stale-parent repair path.
  - confidence: 0.9
