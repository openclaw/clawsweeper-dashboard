## Summary

Keep open: source inspection supports the bug, but this PR still misclassifies some relayed external contexts, misses stored transcript-file metadata in the doctor repair path, and lacks required live Telegram proof.

## What This Changes

The branch routes streaming external-channel steer turns through the follow-up queue, preserves internal webchat embedded steering, and adds Telegram stale-parent transcript doctor repair tests.

## Best Possible Solution

Keep the PR open for a focused rework that resolves channel identity from the actual delivery provider, fixes doctor metadata lookup through stored session identity, rebases onto current main, and adds redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `runReplyAgent` tries embedded steering whenever `effectiveShouldSteer && isStreaming`, before the follow-up queue path, so an active Telegram steer turn can mutate the in-flight session instead of queueing.

## Solution Assessment

No, not yet. The direction is right, but the PR should prefer Provider/messageProvider over Surface for channel identity and resolve doctor metadata through the stored session file or transcript identity before it is mergeable.
