## Summary

Keep open: current main still has the source-reproducible external-channel steering bug, but this PR is not mergeable as-is because it mis-resolves relayed channel identity, misses stored session-file metadata for doctor repair, and has no live Telegram proof.

## What This Changes

The PR routes streaming external-channel steer turns through the follow-up queue, preserves internal webchat steering, and adds Telegram stale-parent transcript doctor repair coverage.

## Best Possible Solution

Rework this PR or a replacement branch so external streaming turns queue based on actual Provider/messageProvider identity, webchat keeps embedded steering, and doctor repair resolves metadata from stored session identity before live Telegram validation.

## Reproduction Assessment

Yes, from source inspection. Current main runs embedded steering for any `effectiveShouldSteer && isStreaming` before queue policy, so an active Telegram streaming turn can mutate the in-flight embedded session instead of queueing behind the latest persisted transcript leaf.

## Solution Assessment

No, not yet. The direction is right, but the PR should resolve channel identity from Provider/messageProvider before Surface and resolve doctor metadata through stored session identity before it is mergeable.
