## Summary

The request describes a real, intentionally visible behavior that current main does not offer a suppression control for, but changing it requires maintainer judgment because fallback notices were deliberately made unconditional to preserve model-routing transparency. The issue should remain open while maintainers decide whether automated runs may suppress that signal and which single policy scope should own the behavior.

## What This Changes

Requests a per-session or per-agent option to suppress model fallback notices for non-interactive workflows.

## Best Possible Solution

If maintainers decide automated workflows may hide fallback transitions, add one canonical source-aware suppression policy at the existing fallback-notice delivery boundary, preserve notices by default, and cover both suppressed automated runs and unchanged interactive runs in focused tests and documentation.

## Reproduction Assessment

Yes, from source with high confidence: current main explicitly creates and persists model fallback notices, while the documentation states that their delivery bypasses ordinary source-reply suppression. A live provider timeout was not needed to establish the missing configuration capability.

## Solution Assessment

No, the per-session-or-per-agent proposal is not yet the best-shaped solution because it leaves ownership and precedence ambiguous. If suppression is approved, one typed automated-source policy with notices enabled by default is narrower and more maintainable.
