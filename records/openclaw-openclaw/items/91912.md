## Summary

Keep open: current main and the latest release intentionally deliver model fallback notices and expose no `suppressFallbackNotice` config key. The request adds a new config/default surface and changes user-visible fallback behavior, so it needs maintainer product judgment before implementation.

## What This Changes

The issue asks for a per-session or per-agent `suppressFallbackNotice` setting to opt out of model fallback notices in chat surfaces.

## Best Possible Solution

Keep fallback notices visible by default; if maintainers accept the ergonomics request, add a narrow opt-in suppression policy with aligned config schema/help/docs and regression coverage around fallback state and source-reply suppression.

## Reproduction Assessment

Yes, from source inspection rather than a live DeepSeek timeout: current runtime, docs, and tests show fallback notices are intentionally delivered when fallback state changes and bypass normal source-reply suppression. The missing behavior is the absence of any current `suppressFallbackNotice` config key.

## Solution Assessment

Unclear until maintainers decide the product contract. A suppression setting is plausible, but adding a new per-session or per-agent config option for fallback delivery should be weighed against keeping fallback state visible and the repository's high bar for new config surfaces.
