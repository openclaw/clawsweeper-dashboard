## Label Justifications

- `P3`: This is a useful but low-urgency ergonomics and notification-policy enhancement rather than a regression in documented behavior.
- `impact:other`: Always-delivered fallback notices can create unwanted repeated chat traffic in automated sessions without fitting a more specific impact category.

## Summary

Current and shipped behavior deliberately sends model fallback and recovery notices once per state change even when ordinary source replies are suppressed. This remains a valid feature request because the existing send policy cannot retain normal assistant replies while suppressing only operational notices, but the public policy shape and precedence need maintainer product judgment.

## What This Changes

Add a per-session or per-agent setting that suppresses model fallback and recovery notices without suppressing normal assistant replies.

## Best Possible Solution

Define a canonical operational-notice delivery policy with agent defaults and explicit session override precedence, apply it to both fallback and recovery notices, and preserve lifecycle events and status metadata regardless of chat suppression.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.5 explicitly deliver fallback notices despite ordinary source-reply suppression, and the existing send policy cannot selectively suppress only those notices.

## Solution Assessment

No, the proposed standalone `suppressFallbackNotice` boolean is not yet proven to be the best shape. A canonical operational-notice policy with clear agent/session inheritance would avoid accumulating one-off suppression flags while satisfying the use case.
