## Summary

Keep this PR open: it is a focused candidate for the Terminal Hatch startup-loader bug, but the patch still has a TUI activity-state regression and the external real-behavior proof is not strong enough for merge.

## What This Changes

The PR adds a busy `starting up` TUI activity state during `client.onConnected` post-connect initialization and adds unit plus fake-backend PTY regression coverage.

## Best Possible Solution

Keep the narrow startup-loader approach, but make `starting up` and its cleanup own only the startup idle state so in-flight and reconnect streaming survive, then verify with real TUI or PTY proof.

## Reproduction Assessment

Yes, by source inspection: current main sets ready before awaited post-connect initialization and only renders the loader for busy activity statuses. ## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by in-flight runs and reconnect recovery.
