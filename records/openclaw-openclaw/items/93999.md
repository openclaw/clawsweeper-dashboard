## Summary

Keep this PR open, but not merge-ready: the startup-loader direction is sound, yet the current patch can overwrite active TUI run state and still lacks real after-fix TUI/PTTY/Hatch proof from a real setup.

## What This Changes

The PR marks TUI post-connect initialization as a busy `starting up` activity, clears it on success or startup failure, and adds unit plus fake-backend PTY coverage.

## Best Possible Solution

Keep the narrow startup-loader fix, but make the startup status own only the startup-idle window, preserve active-run states, and require redacted real TUI/PTTY/Hatch proof before merge.

## Reproduction Assessment

Yes, source-level: current main sets a ready connection status before awaited post-connect work, while the loader only renders for activity statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery or in-flight run adoption.
