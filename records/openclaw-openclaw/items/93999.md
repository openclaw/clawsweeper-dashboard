## Summary

Keep this PR open, but it is not merge-ready: the missing startup loader is a real source-level TUI bug, yet the patch can overwrite reconnect/history-owned active-run state and the supplied proof is only a helper simulation rather than a real TUI/PTTY/Hatch run.

## What This Changes

The PR adds a `starting up` busy activity status during `runTui` post-connect initialization and adds unit plus fake-backend PTY coverage for that loader.

## Best Possible Solution

Keep the narrow startup-loader approach, but make the startup status own only the startup-idle window, preserve reconnect/history busy states, and require redacted real TUI/PTTY/Hatch proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main marks the connection ready before awaited post-connect initialization, while the loader only renders for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery or in-flight run adoption.
