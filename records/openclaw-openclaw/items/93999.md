## Summary

Keep this PR open but do not pass it as submitted: the startup-loader direction is source-reproducible and useful, but the patch can overwrite active run status restored during reconnect/history loading, and the external-contributor proof is still not a real TUI/PTTY/Hatch run.

## What This Changes

The PR changes the TUI post-connect path to mark startup initialization as a busy activity status and adds unit plus fake-backend PTY harness coverage.

## Best Possible Solution

Keep the narrow startup-loader approach, but make the startup status own only the startup-idle window, preserve reconnect/history active-run states, and require redacted real TUI/PTTY/Hatch proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main sets a ready connection status before awaited post-connect work, while the loader only renders for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery or in-flight run adoption.
