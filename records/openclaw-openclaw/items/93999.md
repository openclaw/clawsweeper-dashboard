## Summary

Keep this PR open, but not merge-ready: the startup-loader direction is useful and addresses the linked Hatch issue, but the patch unconditionally overwrites reconnect-restored busy state and the supplied proof is a simulation rather than a real TUI/onConnected run.

## What This Changes

The PR adds `starting up` to TUI busy activity statuses, sets it during `client.onConnected` post-connect initialization, and adds unit plus PTY regression coverage.

## Best Possible Solution

Keep the narrow startup-loader fix, but gate it so it only takes over idle/non-busy activity and require real TUI or PTY proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main renders the loader only for busy activity statuses and sets ready before awaited post-connect initialization. The PR finding is also source-reproducible because reconnect restores `streaming` before the new unconditional `starting up` assignment.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right narrow layer, but it must preserve existing busy statuses restored by reconnect and needs real TUI/PTY proof.
