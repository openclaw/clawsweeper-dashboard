## Summary

Keep this PR open, but it is not merge-ready: the startup-loader direction is valid, while the patch can overwrite TUI activity state owned by reconnect/history recovery and the supplied proof is a helper simulation rather than a real after-fix TUI/PTTY flow.

## What This Changes

The PR adds `starting up` to TUI busy activity statuses, sets it during `client.onConnected` post-connect initialization, clears it after startup/failure, and adds unit plus fake-backend PTY coverage.

## Best Possible Solution

Keep the narrow `onConnected` startup-loader approach, but make the startup status own only the startup-idle window and preserve reconnect/history-owned busy states, then add real TUI or PTY evidence with private details redacted.

## Reproduction Assessment

Yes, by source inspection: current main sets ready before awaited post-connect initialization, and the renderer only shows a loader for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. The patch regression is also source-reproducible because reconnect recovery and `loadHistory()` can set `streaming` before the new startup cleanup overwrites it.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by in-flight runs and reconnect recovery.
