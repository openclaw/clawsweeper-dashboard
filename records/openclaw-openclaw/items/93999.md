## Summary

Keep open: current main still has the missing TUI startup-loader gap, but this PR is not merge-ready because it can overwrite active run/session activity state and the supplied proof is a helper simulation rather than a real TUI/PTTY/Hatch run.

## What This Changes

The PR adds `starting up` to the TUI busy-status set, sets that activity during `client.onConnected` post-connect initialization, and adds unit plus fake-backend PTY regression coverage.

## Best Possible Solution

Keep the narrow startup-loader approach, but make the startup status own only the startup-idle window, preserve reconnect/history busy states, and require redacted real TUI/PTTY/Hatch proof before merge.

## Reproduction Assessment

Yes, by source inspection against current main: `client.onConnected` marks the connection ready before awaited post-connect initialization, while the renderer only shows a loader for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery, history in-flight adoption, or active run events.
