## Summary

Keep this PR open: it targets a real source-reproducible TUI startup-loader bug, but the submitted patch can overwrite active run/reconnect activity state and the contributor proof is still a simulation rather than a real TUI or PTY run.

## What This Changes

The PR adds a `starting up` busy activity state during TUI post-connect initialization and adds unit plus PTY harness regression coverage.

## Best Possible Solution

Keep the narrow startup-loader approach, but make `starting up` and its cleanup own only the startup idle state so in-flight and reconnect streaming survive, then verify with real TUI or PTY proof.

## Reproduction Assessment

Yes. Current main sets ready before awaited post-connect initialization and only renders the loader for busy activity statuses; the PR regression is also source-reproducible because `loadHistory()` and reconnect recovery can set `streaming` before the new cleanup resets activity to `idle`.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by in-flight runs and reconnect recovery.
