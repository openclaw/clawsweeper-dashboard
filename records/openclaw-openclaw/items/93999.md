## Summary

Keep open: current main still has the missing startup-loader behavior this PR targets, but the PR is not merge-ready because it can overwrite reconnect/history-owned activity state and the supplied proof is a helper simulation rather than real TUI/PTTY/Hatch behavior.

## What This Changes

The PR adds a `starting up` busy TUI activity status during `client.onConnected` post-connect initialization and adds unit plus fake-backend PTY coverage for the slow-startup window.

## Best Possible Solution

Keep the narrow startup-loader fix, but make startup status changes affect only the startup-owned idle window, preserve reconnect/history busy states, add focused regression coverage for that invariant, and require redacted real TUI/PTTY/Hatch proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main marks the TUI connection ready before awaited post-connect initialization, while the loader only renders for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery or in-flight run adoption.
