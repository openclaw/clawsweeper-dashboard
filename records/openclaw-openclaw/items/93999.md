## Summary

Keep this PR open for changes: the startup-loader direction is right, but the patch can still overwrite active TUI run state after `loadHistory()` adopts an in-flight run, and the supplied proof is a render simulation plus fake-backend coverage rather than a real TUI/Hatch run.

## What This Changes

This PR adds `starting up` to TUI busy statuses, sets it during `client.onConnected` post-connect initialization, clears it on success/failure, and adds unit plus fake-backend PTY coverage.

## Best Possible Solution

Keep the narrow startup-loader approach, but make startup status own only the startup-idle window, preserve active run statuses, and require redacted real TUI/Hatch proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main sets a ready connection status before awaited post-connect work, while the renderer only shows a loader for statuses in `TUI_BUSY_ACTIVITY_STATUSES`. I did not run an interactive macOS Hatch session in this read-only review.

## Solution Assessment

No, not as submitted. Marking post-connect initialization busy is the right layer, but the implementation must preserve activity state owned by reconnect recovery or in-flight run adoption.
