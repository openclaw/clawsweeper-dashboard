## Summary

Close as implemented: current main and the v2026.5.5 source/package both route registered Telegram plugin commands to local handlers before agent dispatch, including the `commands.native: false` case; the new report does not include plugin registration logs showing a distinct remaining core path.

## What This Changes

The issue reports that custom plugin slash commands are still sent to the LLM instead of being intercepted and executed locally in v2026.5.5.

## Best Possible Solution

Keep the shipped v2026.5.5/current-main routing implementation; any remaining failure should come back as a new plugin-specific report with `api.registerCommand()` details, startup registration logs, and Telegram command config.

## Reproduction Assessment

No. The current source, v2026.5.5 tag source, npm package, docs, and regression tests all show the Telegram plugin command path is registered and executed locally before agent dispatch; the issue lacks logs or a minimal plugin that bypasses those paths.

## Solution Assessment

Yes. The current Telegram-owned handler registration plus the shared plugin-command-first handler is the narrow maintainable solution; further reports should diagnose registration/config/plugin specifics rather than add another core routing layer.
