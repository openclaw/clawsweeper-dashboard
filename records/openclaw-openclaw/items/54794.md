## Summary

Keep open. Current main and the latest release still lack a Telegram inline-query runtime path; the only inline-query-related implementation on main reports BotFather account capability, while the prior implementation PR is closed and unmerged with unresolved correctness and security concerns.

## What This Changes

The issue requests Telegram inline query support so users can invoke an OpenClaw bot from any Telegram chat via `@botname query`.

## Best Possible Solution

Keep this as the canonical tracker and implement a reviewed Telegram plugin feature with config/schema/docs, conservative authorization/session semantics, bounded inline results, focused tests, and live Telegram proof.

## Solution Assessment

Unclear for a direct automated fix. The Telegram plugin is the right owner boundary, but the feature needs maintainer decisions on authorization, privacy, session scope, deadlines, and result semantics before implementation.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram contexts, so sender authorization, privacy boundaries, session scoping, rate limiting, and result deadlines need explicit design before routing into agent turns.
- Inline-result UX and any `chosen_inline_result` correlation need careful handling to avoid leaking, losing, or misattributing work.
- The current `inlineQueries=true/false` status flag reports Telegram account capability only, which can be mistaken for OpenClaw runtime support until docs or implementation clarify it.
