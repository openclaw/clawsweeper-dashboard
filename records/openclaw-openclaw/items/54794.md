## Summary

Keep open. Current main and the latest shipped release still lack Telegram inline-query handling, response delivery, config/docs, and tests; the closed inline-query PR is only reference material, while the open Guest Mode work is related but does not implement inline queries.

## What This Changes

Add opt-in Telegram inline-query support so typing `@botname query` in any Telegram chat can route through OpenClaw and return Telegram inline results.

## Best Possible Solution

Implement this in the Telegram plugin with explicit opt-in config/schema/docs, conservative sender authorization, isolated session/result semantics, bounded `answerInlineQuery` and chosen-result handling, focused tests, and live Telegram proof while coordinating with the separate Guest Mode tracker.

## Reproduction Assessment

Yes as a source-level feature gap, not as a broken-behavior repro: current main has no inline-query handler, no `answerInlineQuery` path, and no public config/docs surface. I did not run a live Telegram inline-query probe because the missing runtime path is clear from source.

## Solution Assessment

No for an autonomous repair lane. The Telegram plugin is the right owner boundary, but the feature needs maintainer decisions on auth, privacy, session scope, result semantics, rate limits, deadlines, and coordination with Guest Mode before code lands.
