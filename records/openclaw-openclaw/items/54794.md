## Summary

Keep open. Current main and v2026.5.18 still lack Telegram inline-query handling, response delivery, config/docs, and tests; the closed implementation attempt at https://github.com/openclaw/openclaw/pull/51963 is useful reference material but was not merged and had unresolved security/correctness concerns.

## What This Changes

Add Telegram inline-query support so users can invoke an OpenClaw Telegram bot from any chat with `@botname query`.

## Best Possible Solution

Keep this as the canonical Telegram inline-query tracker and design a Telegram plugin implementation with explicit opt-in config, conservative sender authorization, isolated session/result semantics, bounded `answerInlineQuery` handling, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes as a source-level feature gap, not as a broken existing-behavior repro: current main has no inline-query handler, no `answerInlineQuery` path, and no public config/docs surface. I did not run a live Telegram inline-query probe because the missing runtime path is clear from source.

## Solution Assessment

Unclear as an implementation plan. The Telegram plugin is the right owner boundary, but maintainers still need to decide auth, privacy, session scope, result semantics, deadlines, and coordination with Guest Mode before code lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram chats, so authorization, privacy boundaries, session scope, result limits, rate limits, and deadlines need explicit design before routing them into agent turns.
- The existing `inlineQueries=true/false` status flag can be mistaken for runtime support even though it only reports the Telegram bot account capability.
- Guest Mode overlaps the broader user goal, but it is a separate Telegram update/response model and does not close the inline-result UX requested here.
