## Summary

Keep open. Current main still has the reported status-path bug, and this PR targets the right surface, but the proposed fix should not merge as-is because it adds a weaker duplicate global-install detector instead of reusing the existing detector that already handles real package roots and custom Bun install roots.

## What This Changes

The PR adds a path-based global install manager detector, uses it for package-install update status, and adds unit tests for pnpm, Bun, and npm global path examples.

## Best Possible Solution

Route package-root status detection through the existing global-install ownership logic, preserve repo package-manager detection for git checkouts, and add focused regressions for npm-global OpenClaw metadata declaring pnpm plus custom Bun install roots.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a non-git OpenClaw package root whose `package.json` declares pnpm and current `checkUpdateStatus` returns that metadata-derived manager, which `openclaw update status` then displays as the install method.

## Solution Assessment

No. The PR fixes the correct user-visible problem, but it is not the best implementation because it duplicates a weaker path heuristic instead of reusing or adapting `detectGlobalInstallManagerForRoot`, which already handles real roots, symlinks, npm prefixes, and Bun install root configuration.
