## Summary

Keep open: current main and the latest release still route package-install status through package metadata, so the reported bug remains real, but this PR needs changes before merge because its new detector misses supported custom Bun global roots and lacks after-fix real install proof.

## What This Changes

This PR adds a path-based global install-manager detector for package update status and detector tests for npm, pnpm, and Bun global paths.

## Best Possible Solution

Reuse or share the existing global install-manager detection for update status, preserve current `update-check` behavior, add focused regression coverage, and attach redacted real global-install proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: package installs flow through `checkUpdateStatus()` to `detectPackageManager(root)`, and status renders that value as `Install`. A published npm global root whose package metadata says `pnpm` can still display `pnpm`; I did not run a live global install in this read-only review.

## Solution Assessment

No. The PR targets the right status path, but the maintainable fix should align with `detectGlobalInstallManagerForRoot()` or equivalent shared behavior and handle configured Bun roots while preserving current-main update-check changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Honor configured Bun global roots:** `src/infra/detect-package-manager.ts:51-52`
  - body: `detectGlobalInstallManager()` only reports `bun` when the package path contains `/.bun/`, then falls back to `npm`. The updater already supports custom Bun roots via `BUN_INSTALL`, so a Bun global install under a supported custom root would still show the wrong install manager in `openclaw update status`.
  - confidence: 0.92
