## Summary

Keep open: the PR targets a real status bug that still exists on current main, but the branch is not merge-ready because it uses a narrower path heuristic than the existing global-install detector, lacks real global-install proof, and is currently conflicting with main.

## What This Changes

The PR adds a path-based global install package-manager detector, uses it for package-install update status, and adds detector tests for npm, pnpm, and Bun global paths.

## Best Possible Solution

Use `detectGlobalInstallManagerForRoot()` or a shared equivalent for package-install status, preserve current `checkUpdateStatus()` realpath and registry-channel behavior, add focused regression coverage for npm Homebrew, configured Bun, and custom pnpm roots, then attach real installed-CLI proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a package root flows through `checkUpdateStatus()` to `detectPackageManager()`, which reads `package.json.packageManager`, and status renders that value as `Install`. I did not run a live global install in this read-only review.

## Solution Assessment

No. The PR fixes the right status boundary, but the maintainable fix should reuse or share the existing global-install detector instead of adding a narrower path-only detector, and it needs real global-install proof before merge.
