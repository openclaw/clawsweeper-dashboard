## Summary

Keep open: the reported status bug is still present on current main, but this PR needs changes because its Bun detection misses supported custom `BUN_INSTALL` roots and the external PR only provides unit-test output, not real installed-CLI proof.

## What This Changes

The PR changes update status package-install detection to infer the global installer from the resolved package root path instead of the installed package's `packageManager` metadata.

## Best Possible Solution

Rebase the PR and wire package-install status reporting through the existing global-install detection contract, including custom `BUN_INSTALL` and pnpm roots, with focused tests and redacted real `openclaw update status` proof from actual global installs.

## Reproduction Assessment

Yes. Source inspection shows current main reads `packageManager` first, the root package metadata is pnpm, and status renders that value directly for package installs; I did not run a live npm-global install because this pass is read-only.

## Solution Assessment

No. The PR addresses the npm Homebrew case, but the path-only detector is narrower than the existing global-install detector and misses supported custom Bun roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Detect Bun installs from custom BUN_INSTALL roots:** `src/infra/detect-package-manager.ts:51-52`
  - body: `detectGlobalInstallManager` only returns `bun` when the path contains `/.bun/`, then falls back to `npm`. Bun supports custom `BUN_INSTALL` roots such as `/opt/bun/install/global/node_modules/openclaw`, and current update code already resolves that layout via `BUN_INSTALL`; this PR would still report those installs as `npm`. Please reuse the existing global-install detector or cover Bun's configured global root.
  - confidence: 0.9
