## Summary

Keep this PR open. The bug is valid and the PR targets the right status/update path, but the proposed detector regresses supported Bun installs with custom `BUN_INSTALL`, and the external PR still has only test output rather than real after-fix behavior proof.

## What This Changes

The PR adds path-based global install manager detection for package installs and uses it in update status instead of reading the package root `packageManager` field.

## Best Possible Solution

Keep the status fix, but base global manager detection on the existing update-global contract or extend the new helper to preserve custom Bun and pnpm global roots before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main renders package installs from `update.packageManager`, and `checkUpdateStatus()` still derives that value from package root metadata that declares pnpm.

## Solution Assessment

No, not as written. The direction is right for npm global installs, but the new heuristic is narrower than existing updater detection and should preserve supported custom Bun roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve custom Bun install roots:** `src/infra/detect-package-manager.ts:49-51`
  - body: This only recognizes Bun when the package root contains `/.bun/`. A supported custom `BUN_INSTALL` such as `/opt/bun/install/global/node_modules/openclaw` has the same Bun global layout but no `/.bun/`, so `checkUpdateStatus` reports `npm` and uses npm dependency markers for a Bun install. Match the Bun global layout or reuse the existing `BUN_INSTALL`-aware detection.
  - confidence: 0.91
