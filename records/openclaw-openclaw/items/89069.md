## Summary

Keep open: current main and v2026.6.8 still keep childless Codex-native task mirrors live for 30 minutes and only audit running tasks as stale after 30 minutes, while the reporter’s stuck-initializing shape remains possible from the mirror source and Codex’s bounded unified exec pool makes the resource-exhaustion concern concrete. The already-merged related fixes cover indefinite childless recovery and blocked-spawn terminalization, but not the earlier detection/recovery semantics requested here.

## What This Changes

The issue asks OpenClaw to detect or surface childless Codex-native subagents stuck in initializing/running before the current 30-minute stale reconciliation window can exhaust unified exec capacity.

## Best Possible Solution

Add a maintainer-approved earlier warning or recovery rule for childless Codex-native mirrors stuck initializing well before the 30-minute lost fallback, with focused task-registry/status coverage and live Codex proof before changing terminalization semantics.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection reproduces the remaining gap: current main and v2026.6.8 still wait 30 minutes before audit/reconciliation for a childless Codex-native initializing task shape.

## Solution Assessment

Unclear for automation: a shorter fail-closed threshold is plausible, but maintainers need to choose warning-first versus lost/cancelled recovery semantics for quiet live Codex threads. A new config knob should not be the default answer unless maintainers decide the threshold cannot be safely fixed by product behavior.
