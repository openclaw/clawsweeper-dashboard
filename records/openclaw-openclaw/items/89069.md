## Summary

Keep open: current main and v2026.6.8 still keep childless Codex-native initializing task mirrors live for 30 minutes before lost/audit handling, while related merged fixes only cover indefinite stale recovery and blocked-spawn terminalization. The remaining request is an earlier warning or recovery rule before bounded Codex app-server or unified exec capacity is exhausted.

## What This Changes

The issue asks OpenClaw to detect or surface childless Codex-native subagents stuck initializing before the current 30-minute reconciliation window can exhaust execution capacity.

## Best Possible Solution

Add a maintainer-approved earlier warning or recovery rule for childless Codex-native mirrors stuck initializing well before 30 minutes, with focused task-registry/status coverage and live Codex proof before changing terminalization semantics.

## Reproduction Assessment

No for the full live failure: I did not reproduce unified exec capacity exhaustion on current main. Source inspection does reproduce the remaining gap: current main and v2026.6.8 still wait 30 minutes before audit/reconciliation for the reported childless Codex-native initializing shape.

## Solution Assessment

Unclear for automation: a shorter fail-closed threshold is plausible, but maintainers need to choose warning-first versus lost/cancelled recovery semantics for quiet live Codex threads. A new config knob should not be the default answer unless maintainers decide the threshold cannot be safely fixed by product behavior.
