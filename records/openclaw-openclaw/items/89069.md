## Summary

Keep open: current main and v2026.6.6 still use the 30-minute childless Codex-native reconciliation window, while OpenClaw mirrors Codex `pendingInit` as an initializing task and Codex source confirms these child threads consume limited execution capacity.

## What This Changes

The issue asks OpenClaw to detect or recover childless Codex-native subagents stuck initializing before they exhaust Codex app-server or unified exec capacity.

## Best Possible Solution

Add a targeted Codex-native initializing liveness policy with earlier warning/audit surfacing and a maintainer-approved recovery threshold, while avoiding a broad config knob unless maintainers explicitly want one.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence path for the stale-threshold mismatch because current tests keep the exact task shape running at 10 minutes and only mark it lost after 31 minutes; the resource-exhaustion timing itself was not live-reproduced.

## Solution Assessment

Unclear until maintainers pick the liveness policy. The current 30-minute-only recovery is not enough for the reported stuck-initializing case, but an earlier terminal threshold should be chosen deliberately to avoid killing valid slow subagents.
