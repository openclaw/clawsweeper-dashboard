## Summary

Keep open: current main and v2026.6.6 still use the 30-minute childless Codex-native reconciliation/audit window, while the Codex-native mirror can keep a child thread in `pendingInit`/initializing and Codex unified exec has a bounded process pool. The remaining question is the maintainer-owned warning/recovery threshold, not whether the report is already fixed.

## What This Changes

The issue asks OpenClaw to detect or surface childless Codex-native subagents stuck initializing before they can exhaust Codex unified exec capacity.

## Best Possible Solution

Add a maintainer-approved warning or recovery rule for childless Codex-native mirrors stuck initializing well before the 30-minute lost fallback, with focused task-registry/status coverage and live Codex proof before changing terminalization semantics.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection does reproduce the remaining gap: current main and v2026.6.6 still wait 30 minutes before audit/reconciliation for this childless Codex-native initializing shape.

## Solution Assessment

Unclear for automation: a shorter fail-closed threshold is plausible, but maintainers need to choose warning-first versus lost/cancelled recovery semantics for quiet live Codex threads. A new config knob should not be the default answer unless maintainers decide the threshold cannot be safely fixed by product behavior.
