## Label Justifications

- `P1`: The report describes a live agent/runtime workflow where one stuck native subagent can exhaust unified exec capacity before existing recovery acts.
- `impact:crash-loop`: The failure mode affects process-level availability by exhausting app-server or unified exec capacity even when OS process counts and session counts look normal.
- `impact:session-state`: The affected record is stuck runtime task state for a childless Codex-native subagent that remains running/initializing until manual cancellation or late reconciliation.

## Summary

Keep open: current main and the latest release still keep childless `codex-native` subagent task records alive for the 30-minute reconcile window, while the report shows a real stuck-initializing case exhausting execution capacity well before that.

## What This Changes

The issue asks OpenClaw to detect or recover childless `codex-native` subagents that stay initializing long enough to exhaust unified exec capacity.

## Best Possible Solution

Add a Codex-native initializing-specific liveness policy that surfaces a warning before 30 minutes and only marks tasks lost earlier if maintainers choose that recovery threshold, without adding a broad config knob by default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence path for the stale-threshold mismatch: current tests keep this exact task shape running at 10 minutes and only mark it lost after 31 minutes, but the unified exec capacity exhaustion itself was not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the recovery policy; the safest path is a targeted childless Codex-native initializing warning plus a deliberate earlier recovery threshold rather than a broad new config surface.
