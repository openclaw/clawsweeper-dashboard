## Summary

Keep open: current main still has a clear source-level path where stale transcript usage can dominate the lower active replay estimate and trigger early preflight compaction after a compaction event.

## What This Changes

The issue reports repeated safeguard preflight compactions after a successful compaction because token estimation may reuse stale pre-compaction transcript usage.

## Best Possible Solution

Bound stale transcript usage against the recent replay estimate, avoid falling back to whole-transcript byte estimates unless the explicit transcript-byte policy is configured, and add a regression test for the post-compaction stale-usage case.

## Reproduction Assessment

Yes, at source level: a stale session with a high last usage record and a much smaller recent replay estimate will still use the high transcript usage for preflight compaction. I did not run the live Discord/Pi session loop in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing preflight transcript estimator and should preserve conservative tail-pressure handling while rejecting clearly stale pre-compaction usage.
