## Summary

Keep this PR open. Current `main` still has the ACL audit/reset command-resolution problem, while this PR is a focused security hardening patch with no definite correctness finding; because it changes Windows command execution and overlaps related Windows env hardening, it should go through maintainer/security review rather than cleanup closure.

## What This Changes

The PR moves `SYSTEMROOT`/`WINDIR` into the shared host-env override policy and generated macOS policy, exports the shared default Windows system root, resolves Windows ACL helpers through validated install roots, uses resolved `icacls.exe` paths for reset commands/display, and adds tests plus a changelog entry.

## Best Possible Solution

Land one maintainer-approved Windows hardening that keeps workspace dotenv and host-env policy artifacts consistent, validates Windows system roots before resolving security audit/fix helper binaries, executes ACL helpers by validated `System32` paths, preserves regression coverage and changelog credit, and coordinates the remaining `reg.exe` thread into the same product direction.

## Reproduction Assessment

Yes. On current `main`, a focused seam reproduction can pass `{ SystemRoot: ".\\fake-root" }` into `inspectWindowsAcl()` and observe command construction under `.\fake-root\System32`, while `createIcaclsResetCommand()` still returns bare `icacls`; the PR adds cross-platform tests for those paths.

## Solution Assessment

Yes, directionally. Reusing `getWindowsInstallRoots()` and the existing install-root validator is the narrow maintainable fix for this ACL path; the safer merge path is maintainer/security review with the related `reg.exe` hardening coordinated.
