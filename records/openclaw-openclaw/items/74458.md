## Summary

Keep open. Current main still leaves the Windows ACL audit/fix helper path env- and PATH-dependent, while this PR is a narrow security hardening that should go through maintainer/security review and CI rather than cleanup closure.

## What This Changes

The PR validates Windows system roots for `icacls.exe`/`whoami.exe`, uses resolved System32 paths for ACL reset execution/display, updates host-env policy artifacts, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land one maintainer-approved Windows hardening that keeps workspace dotenv and host-env override policy aligned, validates system roots through the shared install-root helper, and resolves ACL audit/fix binaries to trusted System32 paths.

## Reproduction Assessment

Yes. Current main can be exercised through `inspectWindowsAcl()` and `createIcaclsResetCommand()` with `env: { SystemRoot: ".\\fake-root" }` or an empty env to observe attacker-controlled or bare command construction.

## Solution Assessment

Yes, directionally. Reusing the existing Windows install-root normalizer through `getWindowsInstallRoots()` is the narrow maintainable fix; the remaining gate is maintainer/security approval and validation, not a different implementation.
