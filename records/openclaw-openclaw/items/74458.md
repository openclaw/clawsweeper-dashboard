## Summary

This PR still carries useful, unmerged security hardening: current main has the workspace dotenv blocklist portion from related merged work, but the Windows ACL audit/fix helpers still resolve `icacls.exe`/`whoami.exe` from unvalidated env roots or bare command names. The patch is focused, CI on the exact head is green, and the remaining action is maintainer/security review plus coordination with the related `reg.exe` hardening PR.

## What This Changes

The PR adds a changelog entry, moves `SYSTEMROOT`/`WINDIR` into host-env override policy/generated artifacts, exports the default Windows root, and changes Windows ACL audit/reset helpers and tests to use validated System32 executable paths.

## Best Possible Solution

Land one maintainer-approved hardening that keeps host-env policy, generated artifacts, and tests aligned while resolving Windows ACL helper binaries through validated System32 paths, then coordinate the remaining `reg.exe` follow-up separately.

## Reproduction Assessment

Yes, from source inspection. On current main, calling `inspectWindowsAcl()` or `createIcaclsResetCommand()` with a relative `SystemRoot` or with no root env reaches attacker-controlled or bare command construction without needing a live Windows host to see the path selection.

## Solution Assessment

Yes. Reusing the existing Windows install-root validator and moving `SYSTEMROOT`/`WINDIR` into the shared host-env override policy is the narrow maintainable fix for this ACL helper path; the remaining work is maintainer/security approval and coordination with #74454.
