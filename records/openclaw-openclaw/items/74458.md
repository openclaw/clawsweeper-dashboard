## Summary

Keep open. Current main still leaves the Windows ACL audit/fix helpers partly env- and PATH-dependent, while this PR is a focused security hardening for that remaining surface. It should go through maintainer/security review and exact-head validation rather than cleanup closure, especially with related Windows env-hardening PRs already merged or still open.

## What This Changes

The PR adds SYSTEMROOT/WINDIR to host-env policy artifacts, routes Windows ACL icacls.exe/whoami.exe resolution through validated Windows install roots, updates icacls reset/display command construction, and adds tests plus a changelog entry.

## Best Possible Solution

Land one maintainer-approved hardening that keeps workspace dotenv and host-env policy consistent, validates Windows system roots before resolving security audit/fix helper binaries, executes ACL helpers by validated System32 paths, and coordinates the related Windows env hardening PRs into one coherent state.

## Reproduction Assessment

Yes. On current main, a focused seam can pass env values such as { SystemRoot: ".\\fake-root" } to the Windows ACL helpers and observe attacker-controlled System32 command construction, while createIcaclsResetCommand still returns bare icacls.

## Solution Assessment

Yes, directionally. Reusing getWindowsInstallRoots and the existing Windows install-root validator is the narrow maintainable fix for this ACL path; the safer maintainer path is to review and land it with the related reg.exe/env-policy work coordinated.
