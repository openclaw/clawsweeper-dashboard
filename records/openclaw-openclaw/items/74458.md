## Summary

Keep open. Current main still leaves the Windows ACL audit/fix helpers dependent on unvalidated Windows root env values and a bare `icacls` reset command, while the latest PR head directly hardens that remaining path; related merged PRs cover workspace dotenv/taskkill/shell trust-root pieces but do not make this PR obsolete.

## What This Changes

The PR updates host-env policy artifacts for `SYSTEMROOT`/`WINDIR`, exports the shared Windows system-root default, resolves Windows ACL helper commands through validated Windows install roots, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Land a maintainer-approved version of this hardening, or consolidate it with #74454, so all Windows ACL helpers execute from validated `System32` paths while workspace dotenv and host-env policy artifacts remain aligned.

## Reproduction Assessment

Yes. On current main, a focused seam reproduction can call `inspectWindowsAcl()` with `env: { SystemRoot: ".\\fake-root" }` and observe fake-root helper paths, while `createIcaclsResetCommand()` still returns bare `icacls`.

## Solution Assessment

Yes. Reusing `getWindowsInstallRoots()` and the existing install-root validator is the narrow maintainable fix for this ACL path; the only safer alternative is coordinating this PR with #74454 before merge.
