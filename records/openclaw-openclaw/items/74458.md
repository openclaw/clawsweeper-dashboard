## Summary

Keep open. Current main has the shared workspace dotenv block for Windows root variables from #74456, but the Windows ACL audit/fix path still resolves `icacls.exe` and `whoami.exe` from unvalidated env roots or bare command names. The PR targets a real security surface and I did not find a definite correctness bug in the patch, but it needs maintainer/security review and coordination with overlapping Windows env-hardening PRs.

## What This Changes

The PR updates workspace dotenv and host-env policy artifacts for `SYSTEMROOT`/`WINDIR`, exports the shared Windows system-root default, resolves Windows ACL audit/reset helpers through validated `System32` paths, and adds dotenv, host-env, and Windows ACL regression tests.

## Best Possible Solution

Land one maintainer-approved Windows hardening that keeps the merged workspace dotenv blocklist, validates Windows trust-root env values before any System32 helper resolution, executes ACL audit/fix helpers by validated absolute path, and leaves related `reg.exe`, `taskkill.exe`, shell, and host-env policy changes consistent across the codebase.

## Reproduction Assessment

Yes for the remaining ACL helper behavior: on current main, passing `env: { SystemRoot: ".\\fake-root" }` to the ACL path resolves a relative `fake-root\System32\icacls.exe`, and an empty env falls back to bare `icacls.exe`. The exact workspace `.env` source is now partially blocked on main by #74456, so the PR is no longer the only blocklist fix.

## Solution Assessment

Yes, directionally: using the shared Windows install-root normalizer and validated `System32` paths is a narrow maintainable fix for the ACL path. The safer merge plan is to consolidate this with the overlapping Windows env-hardening PRs rather than landing contradictory partial policies.
