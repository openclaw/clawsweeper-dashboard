## Summary

Keep open. This maintainer-labeled member PR is a plausible gateway/node runtime pinning change, but the review found a blocking behavior gap: node services can persist a runtime pin yet a later forced reinstall does not read the existing node service environment, so the stated preservation behavior is incomplete.

## What This Changes

The PR adds `--runtime-path` for gateway and node service installs, persists `OPENCLAW_DAEMON_RUNTIME_PATH`, updates doctor/repair/audit handling, and documents pinned runtime usage.

## Best Possible Solution

Land the runtime-path contract only after node forced reinstall preserves existing service pins like the gateway path, with docs matching the final preservation scope.

## Reproduction Assessment

No high-confidence live supervisor reproduction was established in this read-only pass. The issue is source-reproducible on current main because runtime selection only carries a family and `resolvePreferredNodePath` can substitute system Node for a version-managed Node.

## Solution Assessment

No. The gateway-side direction is reasonable, but the PR should also preserve node service runtime pins during forced reinstall or narrow the node-facing claim before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve node runtime pins on forced reinstall:** `src/cli/node-cli/daemon.ts:161`
  - body: `runNodeDaemonInstall` validates an explicit `--runtime-path`, but it never reads the existing node service command/environment before rebuilding the plan. After installing a node service with a pinned runtime, a later `openclaw node install --force` without the flag uses only `process.env` and can drop `OPENCLAW_DAEMON_RUNTIME_PATH`, recreating the runtime drift this PR says it preserves. Mirror the gateway read/merge path for node services, or narrow the node behavior/docs.
