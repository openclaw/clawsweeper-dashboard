## Summary

Keep open: the PR’s direction matches the gateway restart-drain ownership model, but it should not merge until the upgrade audit also detects stale supervisor stop timeouts. Without that, some existing launchd or manually adjusted systemd installs can still hard-kill the gateway before the intended drain window finishes.

## What This Changes

The PR changes generated systemd and LaunchAgent service defaults to use a 330s stop budget and systemd `KillMode=mixed`, then updates daemon tests, service audit behavior, and Linux gateway docs around that policy.

## Best Possible Solution

Land the mixed/330s supervisor policy with audit and doctor coverage for stale stop-timeout drift, while keeping custom drain-budget derivation as a separate maintainer-owned follow-up.

## Reproduction Assessment

Yes. Current source renders a 30s/control-group systemd unit while the gateway restart path waits up to 300s for active work, and the systemd/Codex stdio contracts explain why child processes can be cut off before the gateway drain finishes.

## Solution Assessment

No, not quite. The generated `KillMode=mixed` and 330s budget are the right layer, but the best fix also audits stale systemd and launchd stop-timeout values so upgrades get a reliable repair signal.
