## Label Justifications

- `P1`: The PR targets active gateway/channel turns being interrupted during supervised restarts, a user-facing agent workflow failure.
- `merge-risk: 🚨 compatibility`: Generated service defaults and service-audit expectations change for existing systemd and launchd installs, so upgrade behavior depends on reinstall or repair.
- `merge-risk: 🚨 availability`: The new supervisor stop budget intentionally changes how long systemd and launchd wait before hard-killing a stuck gateway.

## Summary

Keep open for maintainer review: current main still renders short supervisor stop budgets that can cut across the existing gateway restart drain, and this PR fixes that path without an obvious blocking code defect. The longer supervisor hard-kill budget is still a compatibility and availability tradeoff maintainers should consciously accept.

## What This Changes

The PR changes generated systemd and launchd gateway service stop policy, service-audit drift checks, tests, and Linux/gateway docs to preserve gateway restart drain before supervisor cleanup.

## Best Possible Solution

Land a focused supervisor-default and audit fix if maintainers accept the longer hard-kill budget, while tracking dynamic timeout derivation as separate follow-up work.

## Reproduction Assessment

Yes, source-reproducible: current main’s generated systemd unit and launchd plist stop budgets are shorter than the gateway’s existing 300s restart drain, and systemd `control-group` can signal child runtimes during that drain. I did not run a fresh live active-turn restart myself.

## Solution Assessment

Yes, this is an acceptable best fix for the supervisor layer because it preserves cgroup cleanup while letting the gateway main process own graceful drain. The separate completed-reply recovery path is already handled by https://github.com/openclaw/openclaw/pull/90790, so duplicating that work here would be the wrong boundary.
