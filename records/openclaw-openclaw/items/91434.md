## Label Justifications

- `P1`: A configured provider/model route can silently remove tools from a tool-dependent agent workflow and lead to false execution claims.
- `impact:auth-provider`: The affected behavior is tied to provider/model capability routing and configured fallback expectations for LM Studio versus OpenAI-backed sessions.
- `impact:session-state`: The transcript can contain assistant claims about tool work while the actual session/tool ledger has no matching tool calls.

## Summary

Keep open: current main and v2026.6.6 still support a `compat.supportsTools: false` local-model route that proceeds as an ordinary agent turn with no model-visible tools, and the remaining request is the missing user/operator guard for tool-dependent sessions. The related issues cover different payload/parser failures, so this is not implemented or cleanly duplicated.

## What This Changes

Request that LM Studio/local-model sessions configured without tool support warn, block, or fall back instead of silently answering tool-dependent requests with no model-visible tools.

## Best Possible Solution

Add a non-breaking capability guard for tool-dependent sessions first, with a clear user/operator warning before the assistant response and in diagnostics; make automatic fallback or blocking an explicit maintainer-approved policy.

## Reproduction Assessment

Yes for the source-level no-tools/no-guard path: a `supportsTools:false` model suppresses model-visible tools, and a visible assistant response does not trigger fallback. I did not run a live LM Studio replay in this read-only review.

## Solution Assessment

Unclear until maintainers choose policy. A warning/status guard is the narrowest safe first step; automatic fallback or fail-closed blocking needs an explicit product decision because no-tools local models are documented as usable.
