## Label Justifications

- `P1`: The report describes a real tool-dependent agent workflow where provider/model selection silently removes tools and can produce false execution claims.
- `impact:auth-provider`: The affected behavior is tied to provider/model capability routing and configured fallback expectations for LM Studio versus OpenAI-backed sessions.

## Summary

Keep open: current main and the latest release still allow a `compat.supportsTools: false` model path to run with no model-visible tools, and the remaining request is the missing user/operator guard for tool-dependent sessions. The related LM Studio reports cover different failures, so this issue is not already implemented or cleanly duplicated.

## What This Changes

The issue requests a clear warning, fallback, block, or prominent UI/log state when an LM Studio/local-model session compiles with zero tools for a tool-dependent agent profile.

## Best Possible Solution

Add a non-breaking no-tools capability guard for tool-dependent profiles first: warn visibly before the assistant response and in session/status surfaces when the selected model disables tools, while leaving fallback or fail-closed behavior to an explicit maintainer decision.

## Reproduction Assessment

Yes for the zero-tool/no-guard path: current source drops runtime tools when `compat.supportsTools` is false, and the latest release contains the same behavior. I did not run the live LM Studio setup, so the model's exact false artifact claims are supported by the issue evidence rather than local replay.

## Solution Assessment

Unclear: the requested direction is valid, but maintainers should choose the product behavior before implementation. A warning/status guard is the narrowest safe first step; automatic fallback or blocking should be intentional because it changes existing no-tools local-model workflows.
