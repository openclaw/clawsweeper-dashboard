## Label Justifications

- `P1`: A tool-dependent agent workflow can silently lose all tools and still produce plausible execution claims, which is an urgent agent/provider routing failure for affected users.
- `impact:auth-provider`: The issue is tied to provider/model capability routing and configured fallback expectations for LM Studio versus tool-capable OpenAI models.

## Summary

Keep open: current main and v2026.6.6 still treat `compat.supportsTools: false` as a no-tools runtime path, and I did not find a pre-response warning, automatic tool-capable fallback, or fail-closed guard for tool-dependent sessions.

## What This Changes

The issue asks OpenClaw to guard LM Studio/local-model sessions that compile with zero tools so tool-dependent agent turns warn, block, or fall back instead of silently producing ordinary responses.

## Best Possible Solution

Add a generic, visible no-tools guard for tool-dependent runs first, then only add automatic fallback or fail-closed behavior if maintainers explicitly choose that upgrade policy.

## Reproduction Assessment

Yes for the source-level failure path: a `compat.supportsTools: false` model disables runtime tools, `context.compiled` records the empty tool set, and visible plain-text output is accepted without automatic fallback. I did not replay the exact LM Studio conversation live.

## Solution Assessment

Unclear until maintainers choose the policy. A visible warning/status guard is the narrowest non-breaking fix; automatic fallback or blocking may be correct but needs an explicit product decision because it changes no-tools local-model behavior.
