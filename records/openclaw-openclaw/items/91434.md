## Label Justifications

- `P1`: The report describes a broken tool-dependent agent workflow where provider/model capability selection silently removes tools and can produce false execution claims.
- `impact:auth-provider`: The failure is tied to provider/model capability routing and configured LM Studio versus OpenAI fallback behavior.
- `impact:session-state`: The assistant transcript can claim tool-backed work while the compiled context and tool-call ledger contain no tools or tool calls.

## Summary

Keep open: current main and v2026.6.6 still intentionally suppress tools when `compat.supportsTools` is false, but there is still no visible no-tools guard, fallback, or block for tool-dependent LM Studio sessions. The related issues cover different tool-call failures, and no linked PR now owns this exact missing warning/fallback/block policy.

## What This Changes

The issue asks OpenClaw to warn, fall back, block, or otherwise surface when an LM Studio/local-model session compiles with zero tools for a tool-dependent agent turn.

## Best Possible Solution

Add a non-breaking, visible no-tools capability guard for tool-dependent profiles before the first assistant response, then let maintainers explicitly decide whether automatic fallback or blocking should become default behavior.

## Reproduction Assessment

Yes for the zero-tool/no-guard path: current source disables runtime tools when `compat.supportsTools` is false and records that state in `context.compiled`. I did not live-replay LM Studio, so the exact false artifact claims rely on the reporter's logs.

## Solution Assessment

Unclear: the requested direction is valid, but maintainers need to choose the product policy. A warning/status guard is the safest first fix; automatic fallback or fail-closed behavior changes existing no-tools local-model workflows.
