## Label Justifications

- `P1`: A provider/model route can silently remove tools from a tool-dependent agent workflow and lead to false execution claims in real user sessions.
- `impact:session-state`: The transcript can show assistant claims about tool work while the session/tool ledger has no matching tool calls.
- `impact:auth-provider`: The affected behavior is tied to model capability metadata, provider routing, and configured fallback expectations for LM Studio versus OpenAI-backed sessions.

## Summary

Keep open: current main and v2026.6.8 still allow a `compat.supportsTools: false` LM Studio/OpenAI-compatible route to run with zero model-visible tools, and current fallback policy accepts a visible assistant reply as terminal instead of warning, blocking, or falling back. The related issues cover adjacent payload/parser or text-tool-call failures, not this zero-tool capability guard.

## What This Changes

The issue asks OpenClaw to protect tool-dependent LM Studio/local-model sessions when the selected model compiles with zero tools by warning, blocking, falling back, or prominently surfacing the no-tools state.

## Best Possible Solution

Add a non-breaking zero-tool capability guard first, with clear user/operator diagnostics before the first assistant response; make automatic fallback or blocking an explicit maintainer-approved policy with upgrade-safe tests.

## Reproduction Assessment

Yes for the source-level no-tools/no-guard path: a `supportsTools:false` model suppresses model-visible tools, and a visible assistant reply is not classified as fallback-worthy. I did not run a live LM Studio replay in this read-only review.

## Solution Assessment

Unclear until maintainers choose policy. A warning/status guard is the narrowest maintainable first step; automatic fallback or fail-closed blocking needs explicit product approval because no-tools local models are documented as usable.
