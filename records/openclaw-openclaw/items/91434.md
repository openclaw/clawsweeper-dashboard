## Label Justifications

- `P1`: A provider/model route can silently remove tools from a tool-dependent agent workflow and lead to false execution claims in real user sessions.
- `impact:session-state`: The session can continue with compiled zero-tool capability while the transcript contains ordinary assistant claims about tool-dependent work.
- `impact:auth-provider`: The affected behavior is tied to model capability metadata, provider routing, and configured fallback expectations.

## Summary

Keep open: current main still has a source-reproducible zero-tool route for `compat.supportsTools: false` models, and a visible assistant answer is still accepted rather than warning, blocking, or triggering model fallback. The adjacent open performance PR does not cover the user-visible guard requested here.

## What This Changes

The issue asks OpenClaw to warn, block, or fall back when a tool-dependent LM Studio/local-model session compiles with zero tools and would otherwise let the assistant answer as if tools ran.

## Best Possible Solution

Add a non-breaking zero-tool capability guard first, with clear user/operator diagnostics before the first assistant response; make automatic fallback or blocking an explicit maintainer-approved policy with upgrade-safe tests.

## Reproduction Assessment

Yes, at source level. A `supportsTools:false` model suppresses model-visible tools, `context.compiled` records the resulting empty tool list, and visible assistant text is not classified as fallback-worthy.

## Solution Assessment

Unclear until maintainers choose policy. A warning/status guard is the narrowest maintainable first step; automatic fallback or fail-closed blocking needs explicit product approval because no-tools local models are documented as usable.
