## Summary

Keep open: current main and the latest shipped tag still use the 2 second Chrome `--version` probe and collapse timeout/null into the reported doctor warning. The issue is now paired with an open fixing PR, so this issue should stay open until that PR is reviewed, proven on macOS, and merged or replaced.

## What This Changes

Report that `openclaw doctor --deep` can falsely warn that Chrome's version is unknown on macOS when Gatekeeper cold-cache latency makes the 2 second Chrome `--version` probe time out.

## Best Possible Solution

Land or replace the canonical plist-first macOS version probe in https://github.com/openclaw/openclaw/pull/85460, keeping a bounded fallback and focused coverage, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows any Chrome `--version` probe that exceeds 2000 ms makes `execFileSync` throw, `execText` return null, and doctor emit the exact reported warning; I did not reproduce the macOS cold-cache delay live here.

## Solution Assessment

Yes. The right fix belongs in the browser plugin host-inspection path, with macOS bundle metadata read before spawning Chrome and a bounded fallback for non-bundle paths.
