## Summary

Keep open: current main and v2026.5.20 still use a 2 second synchronous Chrome `--version` probe that turns timeouts into the reported unknown-version doctor warning. The issue is source-reproducible and is already tied to the open fixing PR https://github.com/openclaw/openclaw/pull/85460, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks browser doctor to avoid false Chrome MCP version warnings on macOS when cold Gatekeeper verification makes Chrome `--version` exceed the current 2 second timeout.

## Best Possible Solution

Review and land, replace, or close https://github.com/openclaw/openclaw/pull/85460; the desired end state is plist-first macOS Chrome bundle version detection with a bounded executable fallback and clear doctor behavior for true probe failures.

## Reproduction Assessment

Yes. By source inspection, any Chrome `--version` run that exceeds 2000 ms makes `execFileSync` throw, `execText` return null, and browser doctor emit the reported unknown-version warning; I did not live-reproduce macOS Gatekeeper latency in this Linux checkout.

## Solution Assessment

Yes for the plist-first direction in the open fixing PR. Reading macOS bundle metadata before spawning Chrome is narrower than a global timeout-only change, while the issue itself should remain open until the fix lands.
