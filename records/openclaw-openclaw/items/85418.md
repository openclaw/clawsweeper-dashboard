## Summary

Current `main` still uses a 2 second synchronous `--version` spawn for Chrome version detection, and doctor collapses a null/timeout result into the same warning as an unknown version. The issue is recent, concrete, and narrow, so it should stay open and is suitable for a focused fix PR rather than cleanup closure.

## What This Changes

The issue requests that doctor make Chrome version detection on macOS reliable when a cold Gatekeeper cache makes `Google Chrome --version` exceed the current 2 second timeout.

## Best Possible Solution

Update the browser plugin’s host inspection path to avoid or tolerate cold macOS Chrome spawns, ideally by reading macOS bundle metadata first and preserving a slower fallback with a timeout-specific doctor diagnostic.

## Reproduction Assessment

Yes, by source inspection: a Chrome `--version` invocation that runs longer than 2000ms makes `execFileSync` throw, `execText` returns null, and doctor emits the reported warning. I did not reproduce the macOS Gatekeeper cold-cache delay live in this Linux checkout.

## Solution Assessment

Yes, a bounded fix in the browser plugin’s version probe is the right shape; plist-first macOS detection plus a timeout-aware fallback/diagnostic would solve the reported false warning without changing core plugin policy.

Prefer reading `Contents/Info.plist` / `CFBundleShortVersionString` for macOS app-bundle executables before falling back to `--version`; keep Linux/Windows behavior compatible, avoid changing public SDK names unless necessary, and do not add a user config knob.
