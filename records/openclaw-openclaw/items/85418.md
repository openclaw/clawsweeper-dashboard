## Summary

Current main and the latest shipped release still use a 2 second Chrome `--version` probe and collapse timeout/null into the reported doctor warning. A linked open PR already targets this issue with closing syntax, so the issue should stay open until that PR lands or maintainers choose a different fix path.

## What This Changes

The issue reports that `openclaw doctor --deep` can falsely warn that Chrome's version is unknown on macOS when a cold Gatekeeper/amfid check makes Chrome's `--version` call exceed the current 2 second timeout.

## Best Possible Solution

Review and land, replace, or close the linked fix PR at https://github.com/openclaw/openclaw/pull/85460; the durable fix belongs in the browser plugin version probe, likely by reading macOS app bundle metadata before a longer bounded executable fallback.

## Reproduction Assessment

Yes, by source inspection: a Chrome `--version` invocation that runs longer than 2000 ms makes `execFileSync` throw, `execText` return null, and doctor emit the reported unknown-version warning. I did not live-reproduce the macOS cold Gatekeeper delay in this Linux checkout.

## Solution Assessment

Yes, the narrow maintainable solution is to change the browser plugin's version probe rather than core policy; plist-first macOS detection plus a longer bounded fallback matches the reported failure mode.
