Keep open: current main and the latest shipped release still use the 2 second Chrome `--version` probe and still collapse a null probe result into the reported doctor warning. ## What This Changes

The issue reports that `openclaw doctor --deep` can falsely warn about an unknown Chrome version on macOS when Gatekeeper cold-cache latency makes Chrome `--version` exceed the 2 second timeout.

## Best Possible Solution

Review and land or close the linked fixing PR; once a macOS-proven fix reaches main, close this issue as implemented with the merge commit or release evidence.

## Reproduction Assessment

Yes, by source inspection: on current main, a Chrome `--version` probe that exceeds 2000 ms returns null and `noteChromeMcpBrowserReadiness` emits the reported warning. I did not reproduce the macOS Gatekeeper cold-cache delay live in this Linux checkout.

## Solution Assessment

Yes, the narrow maintainable fix is in the browser plugin version probe: read macOS bundle metadata before spawning Chrome and keep a bounded slower fallback. The existing linked PR follows that shape but still needs review/proof before this issue can close.
