## What This Changes

The issue reports flaky Chrome version detection in `openclaw doctor --deep` on macOS because the browser plugin uses a 2 second Chrome `--version` timeout that can expire during cold Gatekeeper verification.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/85460 or an equivalent browser-plugin fix that reads macOS bundle metadata first, keeps a slower fallback probe, and adds focused regression coverage; close this issue after the fix merges.

## Reproduction Assessment

Yes, by source inspection: a Chrome `--version` call taking longer than 2000 ms returns null through `execText`, and doctor turns that null into the reported unknown-version warning. I did not reproduce the macOS cold-cache delay live in this Linux checkout.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the browser plugin’s host-inspection/version probe path. The linked PR’s plist-first macOS approach with a longer fallback timeout matches the issue’s root cause, subject to normal PR review.
