## Summary

Keep open. The PR targets a real gap that current main still does not cover: background exec sessions are only marked backgrounded and exposed through `process`, not linked into task records. The PR branch is not merge-ready because its CLI task maintenance and session-row mapping are too broad for existing `runtime:"cli"` producers, and one progress path lets active output stop refreshing task activity.

## What This Changes

The PR adds linked `runtime:"cli"` task records for background exec/process runs, changes process remove/kill visibility, derives session rows from linked task state, adds a CLI liveness hook to task maintenance, and expands regression tests.

## Best Possible Solution

Implement the exec-specific task linkage on top of current main: create linked CLI tasks only for background exec/process sessions, preserve the newer runtime-aware maintenance model for other CLI producers, refresh task activity on every output chunk, scope session-row overrides to the linked exec task, and document the user-visible fix in the changelog.

## Reproduction Assessment

Yes. There is a high-confidence code-level reproduction for the review findings: create or inspect an existing non-exec `runtime:"cli"` task such as a gateway agent/media task and the PR's global bash-process liveness hook can mark it lost after the CLI grace; a long-running exec with output after the first summarized chunk also leaves task `lastEventAt` stale.

## Solution Assessment

No. The PR's product direction is reasonable, but this implementation is not the best way to solve it because it makes all CLI task liveness depend on bash-process-registry and maps all owner tasks into session-row state instead of adding an exec-specific linkage within the existing task runtime contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict CLI liveness checks to exec-backed tasks:** `src/tasks/task-registry.maintenance.ts:75`
  - body: The new maintenance path sends every `runtime:"cli"` task through the bash-process `isCliRunActive` observer. Existing gateway-agent and media tasks also use `runtime:"cli"` with run ids that are not process session ids, so they can be marked `lost` after the 5s grace while their real work is still active. Scope this hook to exec-created task records or keep the existing runtime-specific liveness path for other CLI task kinds.
  - confidence: 0.91
- **[P2] Refresh CLI task activity for every output chunk:** `src/agents/bash-tools.exec-runtime.ts:678-696`
  - body: `reportCliTaskProgress` only writes `lastEventAt` when a summary is present, but normal ongoing output after the first chunk passes `undefined`. A command that keeps producing output can therefore have stale task activity, which affects active session filtering and status freshness. Record the heartbeat timestamp on every accepted output chunk and only gate the human summary text.
  - confidence: 0.86
- **[P2] Limit session-row overrides to linked exec CLI tasks:** `src/gateway/session-utils.ts:1216-1221`
  - body: `buildGatewaySessionRow` builds its task-driven status from all tasks owned by the session. Any active or recent non-exec owner task can override the session row's status/timing even though this PR is only trying to surface the linked background exec task. Filter this path to the specific exec-linked CLI task shape before overriding the row.
  - confidence: 0.82
