## What This Changes

The PR wires background exec/process sessions into runtime:"cli" task records, changes process remove/kill visibility, updates task maintenance, and derives session rows from task state.

## Best Possible Solution

Rebase onto current main, preserve the existing runtime-aware maintenance behavior, and scope the new background-exec task linkage to exec-owned CLI task records with proof from a real run.

## Reproduction Assessment

Yes. Source inspection on current main shows the background exec yield path only calls markBackgrounded and resolveRunning, with no linked task record creation; I did not run a live failing scenario.

## Solution Assessment

No, not as proposed. The useful background-exec task linkage should be kept, but the patch needs to preserve current main's broader task-maintenance contract and avoid driving session state from unrelated task records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope CLI orphan checks to exec-owned tasks:** `src/tasks/task-registry.maintenance.ts:75`
  - body: This treats every runtime:"cli" row as backed by isCliRunActive(runId), but the wired observer only knows bash process session IDs. Existing gateway and media CLI task producers use their own run IDs, so active work can be marked lost after the 5s CLI grace.
  - confidence: 0.93
- **[P2] Restrict session status overrides to linked exec tasks:** `src/gateway/session-utils.ts:1217`
  - body: The new row mapping reads all tasks for the owner key, so unrelated active or recent tasks can override a completed session's status, timing, and activeMinutes visibility. Filter this to the specific background exec task records introduced by this PR.
  - confidence: 0.9
- **[P2] Refresh CLI task heartbeats for every output chunk:** `src/agents/bash-tools.exec-runtime.ts:682`
  - body: recordTaskRunProgressByRunId only runs when summary is present, but normal post-first output passes undefined. A long-running command that keeps emitting output can therefore have stale lastEventAt and stale session/status visibility.
  - confidence: 0.86
- **[P2] Resolve background yield before task writes:** `src/agents/bash-tools.exec.ts:1651`
  - body: onYieldNow awaits the dynamic import and task creation before resolving the background response. That can push the user-visible yield past the configured window; start task creation asynchronously and let the exit path await the stored promise.
  - confidence: 0.82
