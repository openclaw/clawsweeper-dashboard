## Best Possible Solution

Rebase on current main, keep the existing runtime-aware task maintenance, add exec-owned CLI task linkage with scoped session-row overrides and output heartbeat refreshes, then provide redacted real behavior proof.

- **[P1] Scope CLI orphan checks to exec-owned tasks:** `src/tasks/task-registry.maintenance.ts:75`
  - body: This branch treats every `runtime:"cli"` task as backed only by `isCliRunActive(runId)`, but the configured observer only knows bash process sessions.   - confidence: 0.91

  - body: The new row mapping reads all tasks for the owner key and applies the first active/focused task regardless of whether it is the background exec task. An unrelated owner task can make a completed session appear running or failed and affect active session filtering.
  - confidence: 0.88
- **[P2] Refresh CLI task heartbeat on every output chunk:** `src/agents/bash-tools.exec-runtime.ts:682`
  - body: `reportCliTaskProgress` only records `lastEventAt` when a summary is present, but normal output after the first chunk passes no summary. Long-running commands that keep producing output can stop refreshing the task heartbeat and look stale in status surfaces.
  - confidence: 0.86
- **[P2] Preserve the configured background yield latency:** `src/agents/bash-tools.exec.ts:1651`
  - body: `onYieldNow` awaits dynamic import and task creation before returning the background response, so a slow registry path can extend the configured yield window. Start task creation asynchronously and keep the promise for terminal-state synchronization.
  - confidence: 0.78
