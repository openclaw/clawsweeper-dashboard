## Summary

Keep open. The PR fixes the parser/sequential-key path and adds useful guards, but the receive handler still rejects payloads with missing or empty `message.content` before those new guards can run, so the linked Feishu bug is not fully fixed.

## What This Changes

The PR defaults missing Feishu message content in `parseFeishuMessageEvent`, guards Feishu preview/debounce string operations, adds a sequential-key regression test, and updates `CHANGELOG.md`.

## Best Possible Solution

Accept missing Feishu content as empty text at the receive/parser boundary, preserve mention gating and empty-message skipping, and add receive-handler regression coverage for missing-content mention events.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and mention metadata, then exercise `getFeishuSequentialKey()` or the receive/debounce path. I did not run a live Feishu replay.

## Solution Assessment

No. Defaulting in `parseFeishuMessageEvent()` and guarding `.trim()`/`.replace()` are the right direction, but the receive payload parser still requires truthy content and can drop the event before those guards run.
