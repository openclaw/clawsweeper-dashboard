## Summary

Keep open: the PR is aimed at the right Feishu empty-content bug, but the receive payload parser still requires truthy `message.content`, so missing-content events can be rejected as malformed before the new normalization, debounce guard, or empty-message skip can run.

## What This Changes

The PR normalizes missing Feishu message content, guards preview/debounce string operations, adds a sequential-key regression test, and updates the changelog.

## Best Possible Solution

Accept missing Feishu content as empty at the receive/parser boundary, preserve mention gating and empty-message skipping, and add receive-handler regression coverage for missing-content mention events.

## Reproduction Assessment

Yes, at source level: construct a Feishu receive payload with missing `message.content` and mention metadata, then exercise the receive handler or sequential-key path. I did not live-replay Feishu traffic.

## Solution Assessment

No. Defaulting in `parseFeishuMessageEvent()` and guarding string methods are useful, but the fix should also make `parseFeishuMessageEventPayload()` accept and normalize missing content before debounce/dispatch.
