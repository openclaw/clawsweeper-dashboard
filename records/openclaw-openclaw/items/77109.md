## Summary

Keep this PR open: the change is aimed at a real Feishu crash, but it only guards two downstream trim sites and leaves earlier parser, dispatch-preview, and debounce paths able to read missing content unsafely.

## What This Changes

The PR adds optional chaining around parsed Feishu content trimming in `sequential-key.ts` and empty-message handling in `bot.ts`, plus a unit test for missing `message.content`.

## Best Possible Solution

Normalize Feishu inbound message content to a string at the parser or receive boundary, cover missing-content mention/debounce/dispatch cases, and add an Unreleased Feishu fix entry.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and mention metadata, then call `parseFeishuMessageEvent()`, `getFeishuSequentialKey()`, or the receive handler path. The current code can reach `.replace()` or `.trim()` on a missing parsed content value.

## Solution Assessment

No. Optional chaining at two downstream consumers is narrower than the failure surface; the maintainable fix is to normalize content once at the Feishu parser/receive boundary and keep downstream routing behavior unchanged.
