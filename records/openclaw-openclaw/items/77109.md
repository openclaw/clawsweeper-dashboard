## Summary

Keep open: the PR fixes two downstream trim sites, but current source review shows missing Feishu content can still throw before those guards in mention normalization, handler preview logging, and debounce trimming; it also lacks the required changelog entry.

## What This Changes

This PR optional-chains Feishu parsed-content trimming in sequential-key and empty-message handling, and adds a sequential-key regression test for events without `message.content`.

## Best Possible Solution

Normalize Feishu inbound content to a string at the parser or receive boundary, then cover missing-content events through sequential key, mention normalization, debounce, and dispatch with an Unreleased changelog fix entry.

## Reproduction Assessment

Yes, source-reproducible: construct a Feishu text event with missing `message.content` and mention metadata, then call `parseFeishuMessageEvent()` or `getFeishuSequentialKey()`. The parser can throw in mention normalization before the PR's sequential-key optional chain, and no-mention events can still reach `ctx.content.replace(...)` in dispatch.

## Solution Assessment

No. Guarding two downstream trim sites is narrower than the failure surface; the maintainable fix is to normalize content once at the Feishu parser or receive boundary and keep downstream consumers typed as string-only.
