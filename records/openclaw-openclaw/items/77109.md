## Summary

Keep this PR open for changes: it fixes one unsafe Feishu trim call, but the same missing-content shape can still fail before the new guards run, and the user-facing fix also needs a changelog entry.

## What This Changes

The PR adds optional content-trim guards in Feishu sequential-key and empty-message skip logic, plus a sequential-key regression test for missing `message.content`.

## Best Possible Solution

Normalize Feishu inbound message content to a string at the parser or receive-handler boundary, preserve existing mention/debounce/sequential behavior, cover missing-content mention events in tests, and add the active changelog fix entry.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and a mention array, then exercise `parseFeishuMessageEvent`, `getFeishuSequentialKey`, or the receive path. I did not live-replay Feishu traffic in this read-only review.

## Solution Assessment

No. Optional chaining at two leaf trim sites is too narrow because missing content needs to be normalized before mention parsing, preview logging, debounce/sequential classification, and empty-message skipping.
