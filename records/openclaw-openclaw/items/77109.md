## Summary

Keep open. The PR addresses the direct parsed-content trim crash, but it still appears incomplete because receive payload parsing rejects missing or empty `message.content` before the new normalization runs, and the external real-behavior proof gate is still unmet.

## What This Changes

The PR normalizes missing Feishu message content before mention parsing, guards handler preview/debounce/empty-message trimming, adds a sequential-key regression test, and adds a changelog entry.

## Best Possible Solution

Accept and normalize missing Feishu content at the receive/parser boundary, preserve existing mention gating and empty-message behavior, cover sequential key plus receive/debounce paths, then land with redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and mention metadata, then exercise `getFeishuSequentialKey()` or the receive/debounce path. I did not run tests because this was a read-only review.

No. Parser-side normalization is the right direction, but the PR still leaves `parseFeishuMessageEventPayload()` requiring truthy content, so the live receive path can drop the event before the fix is exercised.
