## Summary

Keep open. The PR addresses several unsafe Feishu string operations, but the live receive parser still rejects missing or empty `message.content` before those new guards can run, and the external real-behavior proof gate remains unmet.

## What This Changes

The PR defaults missing Feishu message content in `parseFeishuMessageEvent`, guards Feishu preview/debounce string operations, adds a sequential-key regression test, and adds a changelog entry.

## Best Possible Solution

Normalize missing Feishu `message.content` at the receive/parser boundary, preserve mention gating and empty-message skip behavior, add receive-handler coverage, then land with redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and mention metadata, then exercise `getFeishuSequentialKey()` or the receive handler. I did not run tests because this was a read-only review.

## Solution Assessment

No. Parser-side defaulting is the right direction, but the current patch leaves `parseFeishuMessageEventPayload()` requiring truthy content, so the live receive path can drop the event before the new guards execute.
