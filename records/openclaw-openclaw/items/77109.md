## Summary

Keep open. The PR heads in the right direction for the Feishu empty-content crash, but the leaf optional-chaining changes do not normalize the parser/receive boundary, so @mention-only or missing-content payloads can still be dropped or hit earlier unsafe content use; it also needs the required changelog entry.

## What This Changes

The PR adds optional chaining around two Feishu parsed-content trim sites and a sequential-key regression test for an event without `message.content`.

## Best Possible Solution

Normalize Feishu inbound message content to a string at the parser or receive-handler boundary, preserve existing mention/debounce/sequential behavior, add regression coverage for mention-bearing missing-content events, and include the active changelog fix entry.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with missing `message.content` and a mention array, then exercise `parseFeishuMessageEvent`/`getFeishuSequentialKey` or the receive path. I did not live-replay Feishu traffic in this read-only review.

## Solution Assessment

No. Optional chaining at the leaf trim sites is too narrow; the safer maintainable fix is to normalize missing content once at the Feishu parser/receive boundary and cover the sequential, debounce, and dispatch paths.
