## Summary

The PR targets a real Feishu empty-content crash, but the current patch is incomplete: it guards two later `.trim()` calls while leaving earlier receive/handler dereferences that can still throw for the same missing `message.content` case.

## What This Changes

The PR adds optional chaining around Feishu parsed-content trimming in `bot.ts` and `sequential-key.ts`, plus a `sequential-key` regression test for events without `message.content`.

## Best Possible Solution

Normalize Feishu parsed message content to a string at the parser/context boundary, then add receive-handler and dispatch regression coverage plus a single-line changelog entry for the user-visible Feishu fix.

## Reproduction Assessment

Yes, from source inspection: an event whose runtime `message.content` is missing can make `parseMessageContent()` return undefined, and current main then reaches unguarded `.trim()`/`.replace()` calls in the Feishu receive and dispatch paths. I did not run the Vitest reproduction because this review is read-only and `pnpm` is unavailable in this shell.

## Solution Assessment

No. Optional chaining at the two edited call sites is not the narrowest maintainable fix because the same value is used as a string earlier in receive/dispatch; coercing once in the parser or `parseFeishuMessageEvent()` would cover all consumers consistently.
