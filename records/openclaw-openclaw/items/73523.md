## Summary

Keep open. Current main still has the source-reproducible passive Chrome MCP spawn path, but this PR head introduces regressions that block explicit cold-cache attach and narrow documented existing-session support.

## What This Changes

The PR adds Chrome MCP cache/file/port/HTTP health probes, rewires browser status/profile/tab/start availability paths, and updates browser tests and changelog entries.

## Best Possible Solution

Merge a focused browser-plugin fix that keeps passive status/profile/tab probes non-spawning while preserving explicit start attach and documented Brave/Edge/userDataDir/non-macOS existing-session behavior.

## Reproduction Assessment

Yes, source-reproducible. On current main, passive Chrome MCP status/profile paths can reach ensureChromeMcpAvailable(), getSession(), and the MCP SDK transport spawn path on a cold session cache.

## Solution Assessment

No. Cache-only passive probing is the right direction, but this PR also blocks deliberate cold-cache start recovery and narrows the existing documented browser contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Allow explicit start to warm a cold MCP cache:** `extensions/browser/src/browser/server-context.availability.ts:333-337`
  - body: When the probe returns HIGH from live file/port/HTTP signals after a gateway restart, `cacheAttached` is false and this branch throws before a deliberate `POST /start` can create the MCP client. That contradicts the PR contract that explicit start still attaches when the cache is empty.
  - confidence: 0.9
- **[P2] Recognize documented Chromium port owners:** `extensions/browser/src/browser/chrome-mcp.ts:833-842`
  - body: The new owner classifier only trusts Chrome/Chromium process names. Documented Brave and Edge existing-session profiles use different listener process names, so a healthy remote-debugging port is classified as `other` and refused instead of attaching.
  - confidence: 0.86
- **[P2] Do not block non-macOS existing-session probes:** `extensions/browser/src/browser/chrome-mcp.ts:1020-1021`
  - body: On non-darwin hosts this probe always returns `unknown`; with a valid `DevToolsActivePort`, that becomes MEDIUM and `decideStartGate()` refuses explicit start. Existing-session `userDataDir` support is documented as host-local, not macOS-only.
  - confidence: 0.82
