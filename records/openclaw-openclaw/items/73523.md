## Summary

Keep open. Current main still has the passive Chrome MCP spawn path, but this PR’s latest head has blocking browser-plugin regressions in explicit start recovery and documented Chromium existing-session support.

## What This Changes

The PR adds Chrome MCP cache/file/port/HTTP health probes, rewires existing-session browser status/profile/tab/start availability paths, updates browser tests and changelog, and carries merged-main changes across several other areas.

## Best Possible Solution

Merge a focused browser-plugin fix that keeps passive status/profile/tab probes non-spawning while preserving explicit start attach, documented Brave/Edge/userDataDir support, and host-platform behavior.

## Reproduction Assessment

Yes, source-reproducible. On current main, passive Chrome MCP status/profile/tab paths can reach `ensureChromeMcpAvailable()` and `getSession()`, and the pinned MCP SDK starts a stdio child during connect.

## Solution Assessment

No. Cache-only passive probing is the right direction, but the current PR also refuses explicit cold-cache attach and narrows documented existing-session support below the current browser contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Allow explicit start to warm cold MCP cache:** `extensions/browser/src/browser/server-context.availability.ts:333-337`
  - body: When remote debugging is already healthy after a gateway restart, `probeChromeMcpHealth()` can return HIGH with `cacheAttached: false`. This branch then throws before `ensureChromeMcpAvailable()` can create the MCP client, so explicit start/browser actions cannot recover the cold gateway cache even though the PR body says `POST /start` should still attach.
  - confidence: 0.9
- **[P2] Recognize documented Chromium port owners:** `extensions/browser/src/browser/chrome-mcp.ts:833-842`
  - body: The new owner classifier only treats Chrome/Chromium process names as trusted. Documented Brave and Edge existing-session profiles use different listener process names, so a healthy remote-debugging port is classified as `other` and refused instead of attaching.
  - confidence: 0.86
- **[P2] Do not block non-macOS existing-session probes:** `extensions/browser/src/browser/chrome-mcp.ts:1020-1021`
  - body: On non-darwin hosts this probe always returns `unknown`; with a valid `DevToolsActivePort` that becomes MEDIUM and `decideStartGate()` refuses explicit start. That breaks existing-session hosts configured with `userDataDir` outside macOS even though the docs describe the mode as host-based rather than macOS-only.
  - confidence: 0.82
