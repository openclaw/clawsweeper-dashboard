## Summary

Keep open: current main still has a source-reproducible passive Chrome MCP spawn path, but the latest PR branch is not mergeable as-is because it treats live Chrome remote-debugging signals as a ready MCP client session.

## What This Changes

The PR adds Chrome MCP health/preflight probing, wires browser status/profile/tab/start paths through it, expands browser plugin tests, and adds changelog entries.

## Best Possible Solution

Land the narrow cache-only passive probe so status, profile listing, and tab listing never create a Chrome MCP session, and review any live-signal auth/start policy separately.

## Reproduction Assessment

Yes, source-reproducible: current main routes passive Chrome MCP status/profile/tab checks through helpers that can create `StdioClientTransport`, and the MCP SDK starts a subprocess during transport start. I did not live-trigger the macOS consent dialog.

## Solution Assessment

No, not in the latest branch shape: a cache-only probe is the right narrow fix, but this PR conflates live Chrome remote-debugging signals with a ready MCP session and changes explicit start policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep live Chrome signals separate from MCP attachment:** `extensions/browser/src/browser/chrome-mcp.ts:1328-1334`
  - body: When the cache is cold, this HIGH branch sets `attached: true` even though OpenClaw has no ready `chrome-devtools-mcp` client. `/tabs` and `listProfiles()` then treat the profile as safe to list tabs, which falls through to ephemeral session creation and can spawn from passive endpoints again.
  - confidence: 0.9
- **[P2] Require a cached MCP session before short-circuiting start:** `extensions/browser/src/browser/server-context.availability.ts:327-328`
  - body: For a cold cache with HIGH file/port signals, `probeChromeMcpHealth()` can return `attached: true` with `cacheAttached: false`, so `POST /start` returns before `ensureChromeMcpAvailable()` creates a usable MCP session. The next browser action still has no client session and can fail or hit the gate instead of using the explicit attach path.
  - confidence: 0.84
