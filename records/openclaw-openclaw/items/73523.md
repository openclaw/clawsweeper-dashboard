## Summary

Keep open: the branch fixes the previously flagged passive cache-attached holes, but latest head is still not merge-ready because the new start gate refuses a healthy live Chrome/no-cache state and breaks the documented explicit existing-session start workflow.

## What This Changes

The PR adds Chrome MCP cache/live health probes, rewires browser status/profile/tab/start availability paths, updates browser plugin tests, adds changelog entries, and includes small formatting-only changes in two agent files.

## Best Possible Solution

Keep passive Chrome MCP status/profile/tab probes cache-only, but let an explicit start attach and warm the MCP client when the user intentionally starts a healthy remote-debugging browser, with regression coverage for both paths.

## Reproduction Assessment

Yes from source inspection: current main passive Chrome MCP status/profile/tab paths can reach session creation, and the pinned MCP SDK starts a stdio child when connected. I did not run the macOS consent prompt live in this read-only pass.

## Solution Assessment

No. Cache-only passive probes are the right direction, but the current PR also blocks documented explicit existing-session start when Chrome is already ready but the gateway cache is cold; separating passive probes from explicit attach intent is safer.
