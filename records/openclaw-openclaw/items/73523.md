## Summary

Keep open: the current-main bug is source-reproducible, but the latest PR head is not merge-ready because live Chrome file/port signals are treated as MCP attachment, so passive paths can still spawn and explicit start can return without a usable MCP client.

## What This Changes

The PR adds Chrome MCP health/preflight probing for existing-session browser profiles, rewires status/profile/tab/start paths, expands browser plugin tests, and updates the changelog.

## Best Possible Solution

Land the narrow cache-only Chrome MCP readiness fix: passive status/profile/tabs should require a ready cached MCP session before listing, while explicit start remains the path that creates or reuses a session.

## Reproduction Assessment

Yes, source-reproducible: current main routes passive Chrome MCP status/profile/tab checks through helpers that can create `StdioClientTransport`, and the MCP SDK starts a subprocess during transport start. I did not trigger the macOS consent dialog live in this read-only review.

## Solution Assessment

No. The cache-only probe is the right shape, but this PR uses live Chrome signals as `attached`, which reopens passive spawning and changes explicit start semantics; the safer fix is to keep passive readiness tied to `cacheAttached` or an equivalent ready cached session.
