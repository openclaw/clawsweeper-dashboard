## Summary

Keep open: the original passive-spawn bug is source-reproducible on current main, but the latest PR still lets live-signal-only health drive passive tab/profile listing into `listChromeMcpTabs()`, which creates `chrome-devtools-mcp` on a cold cache.

## What This Changes

The PR adds Chrome MCP health and browser-auth preflight logic for existing-session browser profiles, rewires status/profile/tab/start paths, adds browser plugin tests, and updates the changelog.

## Best Possible Solution

Keep passive status/profile/tab checks cache-only, count tabs only when a ready cached MCP session exists, and make explicit start either reuse a cached session or perform the intentional attach path when the start gate allows it.

## Reproduction Assessment

Yes. Current main source shows passive Chrome MCP status/list/tab paths can reach session creation, and the MCP SDK transport starts a child process; I did not need the macOS UI prompt to prove the spawning path.

## Solution Assessment

No. A cache-only passive probe is the right direction, but the PR currently conflates live Chrome auth signals with a ready cached MCP client, so passive routes and start can still take the wrong path.

## Review Findings

Full review comments:

- **[P1] Gate passive tab listing on a cached MCP session:** `extensions/browser/src/browser/routes/tabs.ts:94`
  - body: `health.attached` can now be true from file/port/HTTP live Chrome signals while `cacheAttached` is false. In that cold-cache state `/tabs` proceeds to `profileCtx.listTabs()`, which calls `listChromeMcpTabs()` without `ephemeral` and creates a new `chrome-devtools-mcp` session, so a passive poll can still spawn.
  - confidence: 0.91
- **[P1] Avoid tab-count probes without a cached MCP session:** `extensions/browser/src/browser/server-context.ts:184-186`
  - body: `listProfiles()` also treats live-signal-only `attached` as running and then calls `listChromeMcpTabs(..., { ephemeral: true })`. On a cold cache `leaseSession()` falls through to `createEphemeralSession()`, so `GET /profiles` can still spawn while trying to count tabs.
  - confidence: 0.9
- **[P2] Require a cached MCP session before skipping start:** `extensions/browser/src/browser/server-context.availability.ts:327-328`
  - body: `ensureBrowserAvailableOnce()` returns whenever `health.attached` is true, but HIGH can come from live Chrome signals with no cached MCP child. That lets `POST /start` report success without establishing the MCP client the next browser action needs.
  - confidence: 0.86
