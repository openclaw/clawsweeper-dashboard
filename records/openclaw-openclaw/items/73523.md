## Summary

Keep open: the PR targets a valid current-main Chrome MCP spawning bug, but the latest head still treats live Chrome auth signals as equivalent to a ready cached MCP client in passive routes and start handling, so cold-cache paths can still spawn or return without establishing the client.

## What This Changes

The PR adds Chrome MCP health and browser-auth preflight logic for existing-session browser profiles, rewires status/profile/tab/start paths, adds browser plugin tests, and updates the changelog.

## Best Possible Solution

Keep passive status/profile/tab checks cache-only, count or list tabs only when a ready cached MCP session exists, and make explicit start no-op only for that cached-ready state or otherwise follow the intentional attach/error path.

## Reproduction Assessment

Yes. Source inspection of current main shows passive Chrome MCP status/list/tab paths can reach `ensureChromeMcpAvailable()` or `listChromeMcpTabs()`, and the SDK transport starts a subprocess on session creation; I did not need the macOS prompt to prove the spawning path.

## Solution Assessment

No. A cache-only passive probe is the right direction, but the PR currently uses `health.attached` where it needs a ready cached MCP session, conflating live Chrome CDP signals with an in-process MCP client.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require cached MCP health before listing tabs:** `extensions/browser/src/browser/routes/tabs.ts:94`
  - body: `health.attached` can be true from live Chrome file/port/HTTP signals while `cacheAttached` is false. In that cold-cache state this route proceeds to `profileCtx.listTabs()`, which calls `listChromeMcpTabs()` without `ephemeral` and can create a new `chrome-devtools-mcp` session from a passive `GET /tabs` poll.
  - confidence: 0.91
- **[P1] Skip tab counts without cached MCP health:** `extensions/browser/src/browser/server-context.ts:184`
  - body: `listProfiles()` also treats live-signal-only `attached` as running, then calls `listChromeMcpTabs(..., { ephemeral: true })` to count tabs. With an empty cache, that falls through to `createEphemeralSession()`, so `GET /profiles` can still spawn while trying to count tabs.
  - confidence: 0.9
- **[P2] Do not skip start without a cached MCP client:** `extensions/browser/src/browser/server-context.availability.ts:327-328`
  - body: `ensureBrowserAvailableOnce()` returns whenever `health.attached` is true, but HIGH can come from live Chrome signals with no cached MCP child. That lets explicit start report success without establishing the MCP client the next browser action needs, leaving the next action to spawn or fail later.
  - confidence: 0.86
