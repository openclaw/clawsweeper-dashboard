## Summary

Keep open: the branch targets a real source-reproducible Chrome MCP passive-spawn bug, but the latest head now treats cache-cold browser-auth signals as an attached MCP session, which can reintroduce passive spawning and make start report success without a usable session.

## What This Changes

The PR adds a Chrome MCP health/preflight probe, wires browser status, profile listing, tab listing, and start readiness through it, and updates browser-plugin tests plus the changelog.

## Best Possible Solution

Land the narrow cache-only passive probe so status, profile, and tab listing never create a Chrome MCP session, and split any confidence-gated start/auth policy into a separately reviewed browser behavior change.

## Reproduction Assessment

Yes, source-reproducible: current main routes passive Chrome MCP status/profile/tab checks through availability or tab-list helpers that can create a `StdioClientTransport`, and the MCP SDK starts a subprocess during transport start. I did not live-trigger the macOS consent dialog.

## Solution Assessment

No, not in the latest branch shape: a cache-only probe is the right narrow fix, but this PR conflates live Chrome remote-debugging signals with a ready MCP session. Passive routes should depend on a ready cached MCP child, and explicit start/auth gating should be handled separately.
