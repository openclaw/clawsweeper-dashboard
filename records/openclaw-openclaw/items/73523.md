## Summary

Keep open: the reported main-branch bug is real, but this PR head still treats live Chrome auth signals as a usable Chrome MCP client, so cold-cache profile and tab paths can still spawn and explicit start can report success without a ready MCP session.

## What This Changes

The PR adds Chrome MCP health/preflight probes, rewires browser status/profile/tab/start handling, adds browser plugin tests and changelog entries, and includes incidental formatting in two agent files.

## Best Possible Solution

Keep passive Chrome MCP status/profile/tab probes cache-only: only a ready cached MCP session should permit tab enumeration, while live Chrome auth signals may inform status or block unsafe starts but must not be treated as a usable MCP client.

## Reproduction Assessment

Yes. Current main source shows the passive Chrome MCP status/profile/tab paths can reach session creation, and the pinned MCP SDK starts a subprocess when the client connects to a stdio transport.

## Solution Assessment

No. A cache-only passive probe is the right direction, but this PR currently conflates live Chrome auth evidence with a ready cached MCP client, leaving spawning paths and a false-success start path in place.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a cached MCP client before listing tabs:** `extensions/browser/src/browser/routes/tabs.ts:94`
  - body: `health.attached` can be true from file/port/HTTP live Chrome signals while `cacheAttached` is false. In that cold-cache state, `GET /tabs` continues to `profileCtx.listTabs()`, whose Chrome MCP implementation calls `listChromeMcpTabs()` without `ephemeral` and can create a new `chrome-devtools-mcp` session from a passive request.
  - confidence: 0.91
- **[P1] Do not count tabs without a cached MCP client:** `extensions/browser/src/browser/server-context.ts:184-186`
  - body: `listProfiles()` also treats live-signal-only health as running and then calls `listChromeMcpTabs(..., { ephemeral: true })`. On a cold cache the ephemeral lease has no existing session to reuse, so it falls through to creating an ephemeral Chrome MCP session while answering a passive profile list.
  - confidence: 0.9
- **[P2] Attach the MCP client before treating start as complete:** `extensions/browser/src/browser/server-context.availability.ts:327-328`
  - body: `ensureBrowserAvailableOnce()` returns whenever `health.attached` is true, but HIGH can come from live Chrome signals with no cached MCP child. `POST /start` can then report success without establishing the MCP client the next browser action needs.
  - confidence: 0.86
