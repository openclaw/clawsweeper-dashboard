## Summary

Keep this PR open, but do not merge the current head: the main-branch bug is source-reproducible, and the latest patch still lets live-only Chrome signals masquerade as a ready MCP session on passive paths.

## What This Changes

The PR adds Chrome MCP health/preflight handling for existing-session browser profiles, rewires status/profile/tab/start checks, expands browser plugin tests, and updates the changelog.

## Best Possible Solution

Land a narrow cache-only passive probe: passive status/profile/tab routes should only treat a ready cached MCP session as MCP-attached, while explicit start remains the path that creates or reattaches the MCP client.

## Reproduction Assessment

Yes, source-reproducible. Current main routes passive Chrome MCP status/profile checks through ensureChromeMcpAvailable or listChromeMcpTabs, and the MCP SDK transport starts a subprocess when a session is created.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch conflates live Chrome auth confidence with an attached MCP client, leaving passive spawn paths and an explicit-start no-op path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit attached to cached MCP sessions:** `extensions/browser/src/browser/chrome-mcp.ts:1325-1328`
  - body: `computeBrowserAuthConfidence()` returns `attached: true` for a live Chrome port even though `probeChromeMcpHealth()` returns `cacheAttached: false` and `mcpPid: null`. `listProfiles()` and `/tabs` use `attached` to decide whether to call `listChromeMcpTabs`, so a cold cache with healthy Chrome still falls into the session-creation path and can spawn `chrome-devtools-mcp` during passive polling.
  - confidence: 0.91
- **[P2] Require a cached MCP client before skipping start:** `extensions/browser/src/browser/server-context.availability.ts:326-328`
  - body: `ensureBrowserAvailableOnce()` returns whenever `health.attached` is true, but that can now mean only file/port/HTTP Chrome signals with no cached MCP session. In that state explicit start reports success without creating the MCP client that later browser actions need; gate this on a cached session signal such as `cacheAttached` or a non-null MCP pid.
  - confidence: 0.87
