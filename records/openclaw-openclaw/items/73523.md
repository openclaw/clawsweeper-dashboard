## Summary

Keep open: the narrow cache-only passive probe is the right direction, but the current branch still lets passive paths spawn on HIGH live Chrome signals and changes explicit start to return without a ready MCP session.

## What This Changes

The PR adds Chrome MCP health/preflight probing, wires browser status/profile/tab/start paths through it, expands browser plugin tests, and adds changelog entries.

## Best Possible Solution

Land the cache-only ready-session probe for passive status, profile listing, and tab listing, and split or explicitly approve the broader live-signal auth/start gate separately.

## Reproduction Assessment

Yes, source-reproducible: current main routes passive Chrome MCP status/profile/tab checks through helpers that can create `StdioClientTransport`, and the MCP SDK starts a subprocess during transport start. I did not live-trigger the macOS consent dialog.

## Solution Assessment

No. A cache-only probe is the right narrow fix, but this branch conflates live remote-debugging signals with a ready MCP client and changes explicit start semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep live Chrome signals separate from MCP attachment:** `extensions/browser/src/browser/chrome-mcp.ts:1328-1334`
  - body: This HIGH branch can set `attached: true` from Local State/port-owner/HTTP evidence even when `cacheAttached` is false, so callers treat a cold cache as a usable MCP client. `GET /tabs` and profile listing then call `listChromeMcpTabs(..., { ephemeral: true })`, which falls through to `createEphemeralSession()` and can spawn from passive endpoints again.
  - confidence: 0.9

Explicit `start` no longer establishes the client session that later browser actions need.
  - confidence: 0.84
