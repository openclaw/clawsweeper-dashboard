## Summary

Needs changes before merge: the passive non-spawning direction is valid, but the PR now blocks documented explicit existing-session attach flows and rejects supported Chromium variants in the new owner gate.

## What This Changes

This PR adds Chrome MCP cache/file/port/HTTP health probes, rewires existing-session status/profile/tab/start paths to avoid passive spawns, updates browser tests and changelog, and includes small formatting-only agent changes.

## Best Possible Solution

Merge a focused browser-plugin fix that keeps passive status/profile/tab probes non-spawning while preserving explicit attach warm-up and documented Chromium profile support.

## Reproduction Assessment

Yes, source-reproducible. Current main passive Chrome MCP status/profile/tab paths can reach MCP session creation, and the pinned MCP SDK starts a stdio child on connect; the PR regressions are also visible directly in the changed start gate and owner classifier.

## Solution Assessment

No. Cache-only passive probing is the right shape, but the current PR also turns explicit start/action paths into refusals and narrows supported browser owners below the documented existing-session contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Allow explicit start to warm cold MCP cache:** `extensions/browser/src/browser/server-context.availability.ts:333-337`
  - body: When Chrome remote debugging is already healthy after a gateway restart, `probeChromeMcpHealth()` can return high confidence with `cacheAttached: false`. `POST /start` reaches this branch through `ensureBrowserAvailable()`, then throws before `ensureChromeMcpAvailable()` can create the MCP client, so the documented `start -> status -> tabs -> snapshot` smoke path cannot recover a cold gateway cache.
  - confidence: 0.9
- **[P2] Recognize documented Chromium port owners:** `extensions/browser/src/browser/chrome-mcp.ts:833-842`
  - body: The new owner classifier only treats Chrome/Chromium process names as trusted. Documented Brave and Edge existing-session profiles use different listener process names, so a healthy remote-debugging port is classified as `other`, converted to a LOW conflict, and then status/start refuse the profile instead of attaching.
  - confidence: 0.86
- **[P2] Do not block non-macOS existing-session probes:** `extensions/browser/src/browser/chrome-mcp.ts:1020-1021`
  - body: On non-darwin hosts this probe always returns `unknown`; with a valid `DevToolsActivePort` that becomes MEDIUM and `decideStartGate()` refuses explicit start. That breaks custom existing-session/browser-node hosts where `userDataDir` is configured outside macOS even though the attach path is documented as host-based, not macOS-only.
  - confidence: 0.82
