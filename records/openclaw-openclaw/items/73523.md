## Summary

Keep open: the PR targets a real, source-reproducible Chrome MCP passive-spawn bug, but the latest head still blocks explicit cold-cache start recovery and documented Chromium-profile cases.

## What This Changes

The PR adds Chrome MCP cache/file/port/HTTP health probes, rewires browser status/profile/tab/start availability paths, increases browser start timeout, and updates browser tests plus changelog entries.

## Best Possible Solution

Merge a focused browser-plugin fix that keeps passive status/profile/tab probes non-spawning while preserving explicit start recovery and documented Chromium-based existing-session profiles.

## Reproduction Assessment

Yes, source-reproducible. On current main, passive Chrome MCP status/profile/tab paths can reach getSession(), and a cold cache creates a StdioClientTransport-backed MCP session.

## Solution Assessment

No. Cache-only passive probing is the right direction, but this patch also prevents deliberate cold-cache attach and documented Brave/Edge/cross-OS existing-session starts; the safer fix separates passive non-spawning probes from explicit start recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Allow explicit start to warm a cold MCP cache:** `extensions/browser/src/browser/server-context.availability.ts:347-351`
  - body: When Chrome is already in remote-debugging mode after a Gateway restart, the probe can return HIGH with `cacheAttached=false`. This branch throws before `ensureChromeMcpAvailable()` can create the MCP client, so a deliberate `start` cannot reattach the cold cache even though the PR body and docs require that recovery path.
  - confidence: 0.92
- **[P2] Recognize documented Chromium browser owners:** `extensions/browser/src/browser/chrome-mcp.ts:900-909`
  - body: The owner whitelist only accepts Chrome/Chromium process names. Documented Brave and Edge existing-session profiles use different listener process names, so a healthy remote-debugging port is classified as `other` and blocked instead of attaching.
  - confidence: 0.88
- **[P2] Keep userDataDir attach startable off macOS:** `extensions/browser/src/browser/chrome-mcp.ts:1087-1088`
  - body: `probeChromePortOwner()` returns `unknown` on non-darwin hosts, and a configured `userDataDir` with a listening `DevToolsActivePort` then becomes MEDIUM and fails the start gate. Existing-session `userDataDir` profiles are documented as host-local Chromium profiles, not macOS-only.
  - confidence: 0.8
- **[P3] Trim unrelated changelog carry-over:** `CHANGELOG.md:95-120`
  - body: This PR adds many unrelated active-release bullets alongside the browser notes. Merging as-is would duplicate or reorder other fixes in the release notes, so keep only the browser entries for this change.
  - confidence: 0.78
