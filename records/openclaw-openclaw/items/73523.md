## Summary

Keep open: the underlying bug is valid and source-reproducible on current main, but PR head 63edf858 still conflates live Chrome remote-debugging state with a cached MCP transport, so passive consumers and start can still take spawning or no-op paths.

## What This Changes

This PR adds Chrome MCP health/preflight probes, rewires existing-session status/profile/tab/start paths, and updates browser plugin tests plus the changelog.

## Best Possible Solution

Land a narrow cache-only passive probe and keep live Chrome auth signals separate from cached MCP readiness, with route/start tests covering `attached:true`, `cacheAttached:false`, and `mcpPid:null`.

## Reproduction Assessment

Yes, source-reproducible. Current main routes passive Chrome MCP status through `ensureChromeMcpAvailable(..., { ephemeral: true })`, and the MCP stdio transport starts a subprocess; I did not live-trigger the macOS consent prompt.

## Solution Assessment

No. The cache-only direction is right, but this PR uses `attached` for both a ready MCP client and live Chrome remote-debugging signals, so the cold-cache passive and start paths are still not reliably non-spawning or ready.
