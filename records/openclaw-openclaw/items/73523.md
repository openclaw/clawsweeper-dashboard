## Summary

Keep open. The PR addresses a real current-main browser plugin bug where passive Chrome MCP status/profile/tab reads can still reach session creation and spawn `chrome-devtools-mcp`; the proposed cache-only probe is a narrow fit, but the branch still needs the required changelog entry before merge.

## What This Changes

The PR adds a cache-only Chrome MCP health probe and rewires browser status, profile listing, tab listing, and related tests so passive reads do not spawn `chrome-devtools-mcp` when no cached session is ready.

## Best Possible Solution

Keep this PR as the active implementation candidate, add a single-line `CHANGELOG.md` entry under `Unreleased` / `Fixes` with appropriate contributor credit, then land after maintainer review while leaving broader subprocess cleanup to #53399.

Yes. The high-confidence reproduction is code-level on current main: passive `GET /`, `GET /profiles`, and `GET /tabs` for a Chrome MCP profile can reach `isTransportAvailable` or `isReachable`, and an empty cache falls through to `createEphemeralSession` / `createRealSession`.

## Solution Assessment

Yes, with one process follow-up. A cache-only Chrome MCP health probe owned in `chrome-mcp.ts` and used only by passive route/listing paths is the narrow maintainable fix; explicit attach/start behavior remains unchanged.
