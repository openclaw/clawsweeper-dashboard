## Summary

Keep open: the bug is real, but the latest branch still conflates live Chrome remote-debugging signals with a ready Chrome MCP client, leaving passive list/tab paths and explicit start behavior unsafe.

## What This Changes

The PR adds a Chrome MCP health/preflight layer for existing-session browser profiles, rewires status/profile/tab/start paths, adds browser plugin tests, and updates the changelog.

## Best Possible Solution

Land a narrow cache-only passive probe: passive status/list/tab checks must never create an MCP session, and explicit start should only short-circuit when a ready cached MCP session exists.

## Reproduction Assessment

Yes, source-reproducible. Current main routes passive Chrome MCP status through `ensureChromeMcpAvailable(..., { ephemeral: true })`, and the MCP SDK transport starts a subprocess; I did not live-trigger the macOS consent dialog.

## Solution Assessment

No. A cache-only passive probe is the right narrow direction, but the latest patch reports live Chrome signals as `attached` even when no MCP client is cached, so callers still take spawning or no-op paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not mark live-only Chrome signals as MCP-attached:** `extensions/browser/src/browser/chrome-mcp.ts:1325-1328`
  - body: This HIGH branch returns `attached: true` even when `cacheAttached` is false and `mcpPid` is null. Callers use `attached` to mean a ready MCP client exists; on a cold cache, `/tabs` and `listProfiles()` then call `listChromeMcpTabs`, whose ephemeral path creates a new session, so passive polling can still spawn `chrome-devtools-mcp`.
  - confidence: 0.91
- **[P2] Require a cached MCP session before skipping start:** `extensions/browser/src/browser/server-context.availability.ts:327-328`
  - body: `ensureBrowserAvailableOnce()` returns whenever `health.attached` is true. Because that can now come from file/port/HTTP probes with `cacheAttached: false`, `POST /start` can report success without creating a Chrome MCP client, leaving the next browser action without the explicit attach this PR says should remain available.
  - confidence: 0.86
