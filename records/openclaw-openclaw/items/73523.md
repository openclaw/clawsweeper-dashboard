## Summary

Keep open: the reported main-branch bug is valid and the PR includes useful real behavior proof, but the latest head still blocks deliberate cold-cache start recovery and narrows documented existing-session support.

## What This Changes

The PR adds Chrome MCP cache/file/port/HTTP health probes, rewires browser status/profile/tab/start availability paths, and updates browser tests and changelog entries.

## Best Possible Solution

Merge a focused browser-plugin fix that keeps passive status/profile/tab probes non-spawning while allowing explicit start to warm a cold MCP cache and preserving documented Brave, Edge, userDataDir, and non-macOS existing-session behavior.

## Reproduction Assessment

Yes, source-reproducible. On current main, passive Chrome MCP status/profile paths can reach ensureChromeMcpAvailable(), getSession(), and the MCP SDK transport spawn path on a cold session cache.

## Solution Assessment

No. Cache-only passive probing is the right direction, but this PR also gates deliberate attach and supported browser/OS cases too aggressively; the safer fix separates passive probes from explicit start recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Allow explicit start to warm a cold MCP cache:** `extensions/browser/src/browser/server-context.availability.ts:333-337`
  - body: When the probe returns HIGH from live file/port/HTTP signals after a gateway restart, cacheAttached is false and this branch throws before a deliberate start can create the MCP client. That contradicts the PR contract that explicit start still attaches when the cache is empty.
  - confidence: 0.91
- **[P2] Recognize documented Brave and Edge port owners:** `extensions/browser/src/browser/chrome-mcp.ts:833-842`
  - body: The new owner classifier only trusts Chrome/Chromium process names. Documented Brave and Edge existing-session profiles use different listener process names, so a healthy remote-debugging port is classified as other and refused instead of attaching.
  - confidence: 0.88
- **[P2] Keep non-macOS userDataDir attach startable:** `extensions/browser/src/browser/chrome-mcp.ts:1020-1021`
  - body: On non-darwin hosts this probe always returns unknown; with a valid DevToolsActivePort, that becomes MEDIUM and decideStartGate() refuses explicit start. Existing-session userDataDir support is documented as host-local, not macOS-only.
  - confidence: 0.84
- **[P3] Remove duplicated changelog carry-over:** `CHANGELOG.md:74-80`
  - body: The branch adds many unrelated changelog bullets that are already present later in current main, so merging as-is would duplicate other fixes in the release notes. Keep only the browser entries for this PR.
  - confidence: 0.78
