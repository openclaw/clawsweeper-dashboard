## Summary

Keep this PR open: current main’s passive Chrome MCP status paths can spawn `chrome-devtools-mcp`, so the reported bug is real, but the PR head now blocks the documented explicit existing-session `start` path when Chrome is live and the MCP cache is cold.

## What This Changes

This PR adds Chrome MCP cache/file/port/HTTP health probes, wires browser status/profile/tab and existing-session availability paths to those probes, updates browser tests and changelog, and includes small formatting-only agent test/runtime changes.

## Best Possible Solution

Keep passive Chrome MCP status/profile/tab probes cache-only for the MCP session, while letting an explicit existing-session start attach and warm the MCP client when Chrome remote debugging is already healthy.

## Reproduction Assessment

Yes from source inspection: current main passive Chrome MCP status/profile/tab paths can reach MCP session creation, and the pinned MCP SDK starts a stdio child on connect. I did not live-trigger the macOS consent prompt in this read-only pass.

## Solution Assessment

No. The passive probe direction is right, but the current PR also blocks the documented explicit start path from warming the MCP cache in a healthy live Chrome state; that intent needs a separate allowed attach path.
