## Summary

Keep open; the underlying bug is valid and source-reproducible, but this PR head still conflates live Chrome readiness with a cached MCP transport, leaving passive spawn paths and a documented Chromium-profile regression.

## What This Changes

The PR adds Chrome MCP health/confidence probes, rewires browser status/profile/tab/start paths, updates browser plugin tests, and adds changelog entries.

## Best Possible Solution

Land a cache-only passive Chrome MCP probe where cached MCP attachment is distinct from live Chrome auth signals, explicit start only skips when a ready cached MCP client exists, and documented Brave/Edge `userDataDir` profiles remain attachable.

## Reproduction Assessment

Yes. Current main can be traced from passive browser status/profile/tab routes to `StdioClientTransport.start()` spawning `npx`, and the PR head still has live-only health paths that can lead passive list routes back into `listChromeMcpTabs()`.

## Solution Assessment

No. A non-spawning cache-only probe is the right direction, but this patch is not mergeable while `attached` can mean live Chrome rather than a cached MCP client and while the new owner gate rejects documented Chromium browsers.
