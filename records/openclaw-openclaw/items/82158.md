## Summary

Keep open with changes requested: the PR is aimed at the right regression, but it strips the empty field before the OpenRouter DeepSeek V4 wrapper runs, and that wrapper currently re-adds `reasoning_content: ""` to assistant tool-call replay messages. The external PR also lacks after-fix real behavior proof.

## What This Changes

This PR strips empty-string `reasoning` and `reasoning_content` in `sanitizeOpenRouterReasoningReplayFields` and adds a unit test for `buildOpenAICompletionsParams`.

## Best Possible Solution

Move the fix to the OpenRouter DeepSeek V4 wrapper or shared helper so verified OpenRouter routes do not add blank `reasoning_content` to assistant tool-call messages, while preserving direct DeepSeek and proxy behavior that still needs the placeholder.

## Reproduction Assessment

Yes, source-reproducible: current main composes the OpenRouter DeepSeek V4 wrapper after base payload creation, and the existing OpenRouter test expects `reasoning_content: ""` on an assistant `tool_calls` message. I did not run a live OpenRouter request.

## Solution Assessment

No. Stripping the field in `buildOpenAICompletionsParams` is too early in the pipeline because the OpenRouter wrapper re-injects it afterward; the maintainable fix belongs in the wrapper/shared helper with OpenRouter-specific regression coverage.
