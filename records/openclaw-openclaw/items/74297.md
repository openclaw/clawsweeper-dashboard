## Summary

Keep open. Current main appears to have fixed the narrow Anthropic transport header mistake for `sk-ant-api` values, but it still accepts Anthropic `token` profiles, can prefer them ahead of `api_key` profiles, and does not surface an API-key-shaped token profile as a doctor/auth-health warning. That leaves the requested diagnostic and documentation cleanup actionable.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic auth profiles where API-key credentials are stored as `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve Anthropic setup-token and generic token-profile support, but add a provider-aware doctor/auth-health warning when an Anthropic token profile contains an API-key-shaped `sk-ant-api` credential, and update user docs to show Enterprise/API-only accounts should use `type: "api_key"` with `key`.

## Reproduction Assessment

Yes for the source-level misconfiguration path: current main accepts an Anthropic `type: "token"` profile, can select it before `api_key`, and marks it static when no expiry exists. No for the live 529 loop without real Anthropic Enterprise credentials, and current transport suggests the original wrong-header route is already mitigated.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing `token` from the schema is not the best solution because Anthropic setup-token auth remains an intentional provider feature.
