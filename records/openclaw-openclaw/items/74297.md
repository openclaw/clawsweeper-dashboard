## Summary

Keep open: current main appears to avoid the original Anthropic wire-auth failure for API-key-shaped `sk-ant-api...` values, but the requested diagnostic and docs cleanup is still incomplete because static token profiles remain accepted and docs still conflict on current SQLite auth-profile storage versus legacy `auth-profiles.json`.

## What This Changes

The issue requests Anthropic auth-profile validation, diagnostics, and documentation so Enterprise API keys are not misconfigured as static token profiles that can fail provider auth.

## Best Possible Solution

Keep the current prefix-based Anthropic wire-auth split, and add a narrow doctor/docs diagnostic that makes Anthropic Enterprise API keys canonical `api_key` profiles while explaining that `token` is for bearer/OAuth-style static tokens.

## Reproduction Assessment

No high-confidence live reproduction was run because this read-only review had no Anthropic Enterprise credential. Source inspection does reproduce the remaining gap: `token` profiles are still accepted, current Anthropic wire routing is key-shape based, and docs still conflict about the active auth-profile store.

## Solution Assessment

The best fix is not to remove token support globally. A narrow Anthropic-specific doctor/docs warning plus docs alignment around SQLite profile storage and API-key versus bearer-token semantics is the maintainable path.
