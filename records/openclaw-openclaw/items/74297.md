Keep open: current main and v2026.6.8 appear to avoid the original Anthropic API-key-as-bearer failure, but the requested diagnostic and docs cleanup remains incomplete and the closed same-repo PR was not merged.

## What This Changes

The issue asks OpenClaw to handle Anthropic Enterprise API keys stored as `type: "token"` without 529 auth loops, and to add validation, diagnostics, or documentation for `token` versus `api_key` profiles.

## Best Possible Solution

Keep the current prefix-based Anthropic transport split, then add a narrow doctor/status diagnostic and docs update that steer `sk-ant-api` credentials to `api_key` without breaking `sk-ant-oat01` setup-token support.

## Reproduction Assessment

No high-confidence live reproduction is available because this review has no Anthropic Enterprise credential. Source inspection does show the remaining current-main gap: token profiles are still accepted and the docs do not clearly warn API-key users away from Anthropic token profiles.

## Solution Assessment

Yes: a narrow diagnostic/docs repair is the best current path. Auto-removing `token` support or silently rewriting all Anthropic token profiles would be too broad because setup-token and other static bearer-token flows remain supported.
