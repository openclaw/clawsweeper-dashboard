## Summary

Keep open. Current main appears to have fixed the original API-key-shaped Anthropic request header path, but the remaining source-backed gap is that manually stored Anthropic `token` profiles are still accepted, can sort ahead of `api_key`, and are not surfaced by doctor/status as a likely API-key misconfiguration.

## What This Changes

Request validation, diagnostics, and documentation for Anthropic Enterprise/API-key accounts accidentally stored in `auth-profiles.json` as `type: "token"`.

## Best Possible Solution

Keep Anthropic setup-token support, but add targeted doctor/status guidance and docs that flag `sk-ant-api*` Anthropic credentials stored as `token` and direct API-only users to canonical `api_key` profiles.

## Reproduction Assessment

Yes for the remaining current-main source path: a non-expiring Anthropic `token` profile is accepted, resolves as a runtime credential, can sort ahead of `api_key`, and is not a doctor issue. No live Enterprise 529 reproduction was established without real Anthropic Enterprise credentials.

## Solution Assessment

No, removing `token` support or trying to auto-detect Enterprise accounts is not the best fix because setup-token remains supported. The narrow maintainable path is diagnostics and docs for API-key-shaped Anthropic tokens while preserving setup-token behavior.
