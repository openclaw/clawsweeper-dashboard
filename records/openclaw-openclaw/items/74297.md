## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic values, but the remaining diagnostic/docs gap is still real: Anthropic `token` profiles are accepted, can be selected ahead of `api_key` profiles, and static token profiles do not get a doctor warning telling Enterprise/API-key users to canonicalize to `api_key`.

## What This Changes

The issue asks OpenClaw to validate or warn about Anthropic Enterprise API keys stored as `type: "token"` in `auth-profiles.json` and to document the `token` versus `api_key` distinction.

## Best Possible Solution

Preserve Anthropic setup-token support and the current prefix-based transport behavior, then add focused doctor/status guidance and docs that direct Anthropic Enterprise/API-key users to the canonical `api_key` profile shape.

## Reproduction Assessment

Yes for the remaining source-level issue: current main still accepts, resolves, and prioritizes Anthropic token profiles without warning when they are static. The original live 529 loop is not reproduced without Enterprise credentials, and current transport now sends non-`sk-ant-oat` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics and documentation; removing token support would regress the intentional Anthropic setup-token path.
