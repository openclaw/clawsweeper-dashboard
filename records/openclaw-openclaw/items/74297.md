## Summary

Keep open. Current main mitigates the original Anthropic wrong-header route for API-key-shaped values, but the reported `type: "token"` profile shape is still accepted, can still be selected ahead of an API-key profile, and is not surfaced by auth health or doctor as a repairable Anthropic API-key misconfiguration.

## What This Changes

The issue asks OpenClaw to validate, warn about, and document Anthropic Enterprise/API-only credentials stored as `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve supported Anthropic setup-token and generic token-profile behavior, but add an Anthropic-specific auth-health/doctor warning when a stored Anthropic token profile contains an API-key-shaped `sk-ant-api` credential, guide users to `type: "api_key"` with `key`, and update the public model auth/Anthropic docs accordingly.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer it before an API-key profile, and does not warn when the token is static. No for the exact live 529 cascade without Anthropic Enterprise credentials, and current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing supported token auth or rejecting all Anthropic token profiles would be too broad because setup-token support is intentional.
