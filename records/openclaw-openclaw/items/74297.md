## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic values, but it still accepts Anthropic static `token` profiles, can prefer them over `api_key` profiles, and lacks a doctor/status warning or explicit Enterprise API-key profile guidance.

## What This Changes

The issue asks OpenClaw to prevent or warn about Anthropic Enterprise API keys stored as `type: "token"` in `auth-profiles.json`, and to document the supported `api_key` shape.

## Best Possible Solution

Preserve Anthropic setup-token support, add Anthropic-specific doctor/status guidance for API-key-shaped static token profiles, and document the canonical Enterprise `api_key` auth-profile shape.

## Reproduction Assessment

Yes for the source-level footgun: current main loads Anthropic `token` profiles, can rank them before `api_key`, and doctor/status treats non-expiring tokens as static. The original live 529 loop was not re-run, and current transport now sends non-`sk-ant-oat` values as `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing Anthropic token support would regress the supported setup-token path.
