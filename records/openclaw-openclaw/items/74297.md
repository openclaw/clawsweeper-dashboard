## Summary

Keep open. Current main appears to mitigate the original wrong-header path for `sk-ant-api` values, but the remaining requested cleanup is still actionable: Anthropic `token` profiles are accepted, can sort before API-key profiles, and are reported as static/healthy when non-expiring, while the docs do not explicitly warn Enterprise/API-only users to store `sk-ant-api` credentials as `api_key` profiles.

## What This Changes

The issue requests safer handling, diagnostics, and documentation for Anthropic `type: "token"` auth profiles that contain Enterprise/API-key credentials and should be stored as `api_key` profiles.

## Best Possible Solution

Preserve supported Anthropic setup-token behavior, but add a provider-aware diagnostic in auth health/doctor/models status for Anthropic token profiles whose credential shape looks like an API key, and update user docs to show Enterprise/API-only accounts should use `type: "api_key"` with `key`.

## Reproduction Assessment

Yes for the source-level diagnostic gap: current main accepts an Anthropic `type: "token"` profile, can select it before an API-key profile, and reports a non-expiring token as static/healthy. No for the live 529 cascade without real Anthropic Enterprise credentials.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing `token` from the schema would break the intentionally supported Anthropic setup-token path.
