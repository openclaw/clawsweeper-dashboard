## Summary

Keep open. Current main mitigates the original Anthropic wrong-header failure for API-key-shaped values, but it still accepts and can prefer static Anthropic `token` profiles without an Enterprise/API-key-specific doctor/status warning or explicit auth-profile documentation.

## What This Changes

Request validation, diagnostics, and documentation for Anthropic Enterprise API keys accidentally stored as `type: "token"` in `auth-profiles.json`.

## Best Possible Solution

Keep Anthropic setup-token support, preserve the current `sk-ant-oat` transport distinction, add Anthropic-specific diagnostics for static token profiles that look like API keys, and update public auth docs with the canonical Enterprise `api_key` profile shape.

## Reproduction Assessment

Yes for the source-level issue: current main accepts Anthropic `token` profiles, can rank them ahead of `api_key`, and treats non-expiring token profiles as static/healthy. The exact live 529 loop was not reproduced because it requires an Anthropic Enterprise API key.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing Anthropic token support would regress the supported setup-token path.
