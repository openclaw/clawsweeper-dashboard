## Summary

Keep open. Current main has mitigated the original wrong-header transport path for API-key-shaped Anthropic values and has better docs, but static Anthropic `token` profiles are still accepted, can be selected ahead of `api_key`, and do not trigger doctor guidance to canonicalize Enterprise/API-key credentials.

## What This Changes

The issue asks OpenClaw to avoid or clearly warn about Anthropic `type: "token"` auth profiles for Enterprise/API-key accounts and document the `token` versus `api_key` distinction.

## Best Possible Solution

Keep Anthropic setup-token support and the current prefix-based transport behavior, then add focused doctor/status guidance and docs that steer Anthropic Enterprise/API-key users to canonical `api_key` profiles.

## Reproduction Assessment

Yes for the remaining source-level issue: current main still accepts, resolves, and prioritizes static Anthropic `token` profiles without a doctor warning. No for the original live 529 cascade, which needs Anthropic Enterprise credentials to exercise end to end.

## Solution Assessment

No, the current main state is only a partial solution: the transport now sends API-key-shaped values via `x-api-key`, but users can still carry a stale Anthropic `token` profile without clear diagnostics. The narrow maintainable fix is a doctor/status warning plus sharper docs, not removing supported setup-token behavior.
