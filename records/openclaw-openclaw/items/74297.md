## Summary

Current main resolves the reported Anthropic Enterprise failure mode: API-key-shaped Anthropic credentials are no longer forced through the bearer-token route, while Anthropic setup-token support remains intentional and documented as separate from the recommended production API-key path.

## What This Changes

The issue requests validation, diagnostics, and documentation so Anthropic Enterprise API keys are not configured or sent as unsupported `token` auth profiles that trigger 529 errors.

## Best Possible Solution

Keep the current split: preserve supported Anthropic setup-token and Claude CLI token paths, route only OAuth-shaped `sk-ant-oat` values as bearer tokens, send API-key-shaped Anthropic credentials with `x-api-key`, and keep production docs pointed at Anthropic API keys.

## Reproduction Assessment

Yes for the reported configuration path at source level: a `type: "token"` Anthropic profile resolves to a provider credential, and current main routes `sk-ant-api` values to `x-api-key` rather than bearer auth. Live 529 reproduction was not attempted without Enterprise credentials.

## Solution Assessment

Yes. Prefix-based Anthropic transport routing plus clearer provider docs is the narrow maintainable fix because removing `token` from the schema would break supported setup-token and Claude CLI compatibility.
