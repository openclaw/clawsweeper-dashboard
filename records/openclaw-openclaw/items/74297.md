## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic values, but it still accepts and can prioritize static Anthropic `token` profiles without a doctor/status warning or explicit Enterprise API-key auth-profile guidance.

## What This Changes

Request validation, diagnostics, and docs for Anthropic `auth-profiles.json` `token` versus `api_key` credentials after Enterprise API-key migrations hit 529 errors.

## Best Possible Solution

Preserve Anthropic setup-token support, add Anthropic-specific doctor/status guidance for API-key-shaped static token profiles, and document the canonical Enterprise `api_key` profile shape.

## Reproduction Assessment

The original live 529 loop needs Enterprise credentials, while current transport now sends non-`sk-ant-oat` values as `x-api-key`.
