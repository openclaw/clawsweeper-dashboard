## Summary

Keep open. Current main appears to have mitigated the exact wrong-header transport path for `sk-ant-api` values, but it still accepts Anthropic `token` profiles, can order them ahead of API-key profiles, and does not surface an API-key-shaped token profile as a doctor/auth-health problem, so the requested diagnostic/docs cleanup remains actionable.

## What This Changes

Request to prevent, diagnose, and document Anthropic Enterprise API keys being stored as `type: "token"` instead of `type: "api_key"` in `auth-profiles.json`.

## Best Possible Solution

Preserve Anthropic setup-token and generic token-profile support, but add a provider-aware warning in doctor/auth health or the Anthropic auth-doctor hint path when an Anthropic token profile contains an API-key-shaped `sk-ant-api` credential, and update the user docs to show Enterprise/API-only accounts should use `api_key` with `key`.

## Reproduction Assessment

Yes for the source-level misconfiguration path: current main accepts an Anthropic `type: "token"` profile, can select it before `api_key`, and marks it static/healthy when no expiry is present. No for the live 529 cascade without real Anthropic Enterprise credentials.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable direction is diagnostics plus documentation; removing `token` from the schema would be too broad because Anthropic setup-token auth is still supported.
