## Summary

Keep open. Current main mitigates the original wrong-header path for `sk-ant-api` values, but the reported `type: "token"` auth-profile shape is still accepted, can still outrank an API-key profile, and is not surfaced as an Anthropic API-key misconfiguration by auth health, doctor, or the relevant docs.

## What This Changes

The issue asks OpenClaw to validate, warn, and document that Anthropic Enterprise API keys should use `type: "api_key"` with `key`, not `type: "token"` with `token`.

## Best Possible Solution

Add an Anthropic-specific auth-health/doctor diagnostic for API-key-shaped static token profiles, guide users to `type: "api_key"` with `key`, and update the public Anthropic/model-auth docs while leaving legitimate setup-token and Claude CLI token flows intact.

## Reproduction Assessment

Yes for the remaining source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can rank it before an API-key profile, and doctor does not warn for the static-token case. The exact live Enterprise 529 loop is unclear without live Enterprise credentials, and current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing token auth or rejecting all Anthropic token profiles would be too broad because supported setup-token behavior exists.
