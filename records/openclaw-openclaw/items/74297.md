## Summary

Keep open. Current main mitigates the original wrong-header path for `sk-ant-api` values, but the remaining diagnostics/docs request is still valid: Anthropic `type: "token"` profiles are accepted, treated as static/healthy when they have no expiry, can outrank API-key profiles, and the docs do not show the canonical `api_key` auth-profile shape for Enterprise/API-key accounts.

## What This Changes

Request validation, diagnostics, and documentation for Anthropic auth profiles where `type: "token"` is used with API-key-shaped Enterprise credentials.

## Best Possible Solution

Preserve supported Anthropic setup-token behavior, but add an Anthropic-specific doctor/model-auth warning for API-key-shaped static token profiles and document the canonical `api_key` auth-profile shape for Enterprise/API-key accounts.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves Anthropic `type: "token"` profiles, can prefer token profiles ahead of API-key profiles, and does not warn on static token health. The live 529 cascade still requires Anthropic Enterprise credentials, while current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing token support or rejecting setup-token profiles would be too broad because Anthropic setup-token remains an intentional supported path.
