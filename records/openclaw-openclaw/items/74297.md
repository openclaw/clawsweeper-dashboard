## Summary

Current main solves the reported Anthropic Enterprise failure path: stored token profiles still resolve to a credential value, but Anthropic transports choose bearer auth only for `sk-ant-oat` setup-token/OAuth-shaped credentials and send non-setup-token `sk-ant-*` values as `x-api-key`.

## What This Changes

The issue requests safer handling and documentation for Anthropic auth profiles where `type: "token"` is used with API-key-style Enterprise credentials.

## Best Possible Solution

Keep the current transport-level split that sends API-key-shaped Anthropic credentials via `x-api-key`, while leaving broader setup-token propagation and UX work to the separate open Anthropic setup-token thread.

## Reproduction Assessment

No for current main: source and tests show non-setup-token `sk-ant-*` values go through the API-key route, so the reported bearer-auth 529 loop is not reproducible from the current code path.

## Solution Assessment

Yes: routing at the Anthropic transport boundary is narrower than rewriting persisted credential types, because it preserves valid token/setup-token profiles while sending API-key-shaped credentials through the documented API-key header.
