## Summary

Keep open: current main mitigates the original Anthropic wire-auth failure by routing `sk-ant-api...` credentials through API-key auth, but the requested diagnostic and docs cleanup remains incomplete because static Anthropic `token` profiles are still accepted and user docs still blur legacy `auth-profiles.json` storage with current SQLite auth storage.

## What This Changes

The issue asks OpenClaw to validate or warn on Anthropic Enterprise API keys stored as `type: "token"` and to document `token` versus `api_key` auth profile semantics.

## Best Possible Solution

Keep the current prefix-based Anthropic wire-auth split, then add a narrow doctor/models-auth diagnostic or repair for Anthropic `token` profiles containing `sk-ant-api...` material and align docs around SQLite auth storage plus `paste-api-key` versus `paste-token` semantics.

## Reproduction Assessment

No for the original 529 loop: this review did not have an Anthropic Enterprise credential for a live reproduction. The remaining gap is source-reproducible because current main still accepts static `token` profiles and docs still do not clearly distinguish Anthropic API keys from setup/OAuth tokens.

## Solution Assessment

Yes for the proposed narrow direction: a doctor/docs diagnostic is the safest fix. Removing token support globally would be too broad because `sk-ant-oat` setup-token/OAuth-style paths remain supported.
