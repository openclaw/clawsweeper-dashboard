## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic credentials, but it still accepts static Anthropic token profiles, can prefer them over API-key profiles, and does not provide the requested Enterprise/API-key warning in doctor/status/docs.

## What This Changes

The issue asks OpenClaw to validate or warn about Anthropic Enterprise API keys stored as `type: "token"` profiles and document the `token` versus `api_key` distinction.

## Best Possible Solution

Preserve Anthropic setup-token support, but add Anthropic-specific doctor/status guidance for API-key-shaped static token profiles and document the canonical Enterprise API-key profile shape.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves Anthropic token profiles, can rank them before API-key profiles, and reports non-expiring tokens as static/healthy. The original live 529 cascade still requires Anthropic Enterprise credentials to reproduce end to end.

## Solution Assessment

Yes: diagnostics plus documentation is the narrowest maintainable direction. Removing Anthropic token support would break the intentional setup-token route, while the transport path already handles non-`sk-ant-oat` values as API keys.
