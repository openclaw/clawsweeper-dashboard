## Summary

Keep open. Current main appears to have mitigated the original wrong-header transport route for `sk-ant-api` values, but the reported stored shape is still accepted, can still participate in Anthropic profile selection, and is not surfaced by auth health or doctor as an API-key-shaped Anthropic token misconfiguration.

## What This Changes

The issue asks OpenClaw to validate, warn about, or document Anthropic Enterprise/API-only credentials stored as `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve supported Anthropic setup-token and generic token-profile behavior, but add an Anthropic-specific diagnostic for token profiles containing API-key-shaped `sk-ant-api` credentials, guide users to `type: "api_key"` with `key`, and update the model auth/provider docs accordingly.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer token before API key, and does not warn when the token is static. No for the exact live 529 cascade without Anthropic Enterprise credentials, and current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The best fix is a narrow diagnostics/docs repair; removing token support or rejecting all Anthropic token profiles would be too broad because setup-token support is intentional.
