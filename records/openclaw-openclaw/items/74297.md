## Summary

Keep open: current main and v2026.6.8 appear to avoid the original Anthropic API-key-as-bearer failure, but OpenClaw still accepts `token` auth profiles and does not clearly diagnose or document Anthropic API keys stored in that shape.

## What This Changes

The issue asks OpenClaw to prevent, diagnose, or document Anthropic Enterprise API keys stored as `type: "token"` in auth profiles.

## Best Possible Solution

Keep the shipped `sk-ant-oat` bearer split, then add a redacted doctor/status/docs warning that steers Anthropic API-key-shaped token profiles to `api_key` without breaking setup-token support.

## Reproduction Assessment

No high-confidence live reproduction is available because this review has no Anthropic Enterprise credential. Source inspection does show the remaining current-main gap: token profiles are still accepted and the docs/status path does not clearly warn API-key users away from Anthropic token profiles.

## Solution Assessment

Yes for a narrow diagnostic/docs repair: it addresses the remaining user confusion without changing supported token semantics. Removing token support or silently rewriting every `sk-ant-*` token would be too broad because setup-token remains supported.
