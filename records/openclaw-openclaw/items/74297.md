## Summary

Keep open. Current main mitigates the specific wrong-header path for `sk-ant-api` values, but Anthropic `type: "token"` profiles are still accepted, can outrank `api_key`, and are not warned about in auth health or doctor when they contain a static API-key-shaped value.

## What This Changes

The issue asks OpenClaw to validate, warn, and document that Anthropic Enterprise API credentials should use `type: "api_key"` rather than `type: "token"`.

## Best Possible Solution

Keep token/setup-token support, add an Anthropic-specific health/doctor warning for static API-key-shaped token profiles, and document the canonical Anthropic `api_key` profile shape.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can rank it ahead of `api_key`, and does not warn when it is static. The exact live 529 loop needs Anthropic Enterprise credentials, and the current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing token support entirely would be too broad because Anthropic setup-token and other provider token profiles remain intentional.
