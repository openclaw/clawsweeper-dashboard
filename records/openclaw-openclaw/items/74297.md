## Summary

Current main and the latest stable release already route Anthropic Enterprise API keys through the API-key header even when the stored credential came from a token-profile path, so the reported 529-causing auth-route mismatch is no longer necessary to track as an open core issue.

## What This Changes

The issue asks OpenClaw to prevent or clearly diagnose Anthropic `auth-profiles.json` entries that use `type: "token"` with Enterprise API keys where `type: "api_key"` is required.

## Best Possible Solution

Keep the shipped prefix-based Anthropic transport behavior and let the broader setup-token propagation work remain tracked separately in https://github.com/openclaw/openclaw/issues/51911.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains. Source and shipped tests show that token-profile credentials carrying `sk-ant-api` values now reach the Anthropic transport as a credential string and are sent with `x-api-key`, while only `sk-ant-oat` uses Bearer auth.

## Solution Assessment

Yes. The current transport-level split is the narrowest maintainable fix for the reported 529 loop because it preserves supported setup-token/OAuth behavior while sending Enterprise API keys through Anthropic's API-key route.
