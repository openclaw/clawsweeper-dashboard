## Summary

Keep open. Current main mitigates the original wrong-header path for `sk-ant-api` values, but persisted Anthropic `type: "token"` profiles are still accepted, can resolve as static credentials, can be ordered before API-key profiles, and are not surfaced by doctor/docs with guidance to use `type: "api_key"` for Anthropic Enterprise API keys.

## What This Changes

Warn, document, or validate Anthropic auth profiles so Enterprise API keys use `type: "api_key"` instead of `type: "token"`.

## Best Possible Solution

Preserve Anthropic setup-token support, add an Anthropic-specific doctor/auth-health diagnostic for API-key-shaped token profiles, and document the canonical `type: "api_key"`/`key` profile shape for Enterprise API accounts.

## Reproduction Assessment

Yes for the remaining source-level issue: current main accepts an Anthropic `type: "token"` profile, can select it before an API-key profile, and reports it as healthy static token state. The exact live 529 cascade is not fully reproducible without Anthropic Enterprise credentials, and current transport code should no longer send `sk-ant-api` as bearer auth.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing token support entirely would be too broad because Anthropic setup-token profiles are still an intentional supported path.
