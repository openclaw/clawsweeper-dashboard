## Summary

Keep open. Current main and v2026.4.29 mitigate the original Anthropic wrong-header path for `sk-ant-api` values, but the reported configuration shape is still accepted, can still be preferred over an API-key profile, and is not surfaced by doctor/auth health as a repairable Anthropic API-key misconfiguration.

## What This Changes

The issue asks OpenClaw to validate or warn on Anthropic Enterprise API keys stored as `type: "token"` in `auth-profiles.json` and to document the `token` versus `api_key` distinction.

## Best Possible Solution

Preserve Anthropic setup-token and generic token-profile support, but add an Anthropic-specific doctor/auth-health warning when a token profile contains an API-key-shaped `sk-ant-api` credential, and update the auth/provider docs to tell Enterprise/API-only users to store those credentials as `type: "api_key"` with `key`.

## Reproduction Assessment

Yes for the source-level misconfiguration path: current main accepts an Anthropic `type: "token"` profile, resolves it as a provider credential, may order it before `api_key`, and marks it static/healthy when no expiry is present. No for the live 529 cascade without real Anthropic Enterprise credentials, and current transport now mitigates the original wrong-header route for `sk-ant-api` values.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing `token` from the schema or rejecting all Anthropic token profiles would be too broad because setup-token support is intentional.
