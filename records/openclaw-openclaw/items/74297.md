## Summary

Keep open. Current main mitigates the original wrong-header route for `sk-ant-api` values, but the requested diagnostics/docs cleanup remains valid: Anthropic `type: "token"` profiles are still accepted, can be selected before API-key profiles, and static API-key-shaped token profiles are not warned on by auth health or doctor.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic Enterprise API keys that are stored in `auth-profiles.json` as `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve supported Anthropic setup-token behavior, but add an Anthropic-specific doctor/status warning for API-key-shaped static token profiles and document the canonical `api_key` profile shape for Enterprise/API accounts.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer it before an API-key profile, and does not warn when the token is static. The exact live 529 cascade needs Anthropic Enterprise credentials, and current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; rejecting all Anthropic token profiles would be too broad because setup-token support is intentional.
