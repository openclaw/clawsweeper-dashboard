## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic values, but persisted Anthropic `type: "token"` profiles are still accepted, can be selected ahead of `api_key` profiles, and are reported as healthy/static without guidance to use `type: "api_key"` for Enterprise API keys.

## What This Changes

The issue asks OpenClaw to validate, warn, or document that Anthropic Enterprise API keys should use `type: "api_key"` rather than persisted `type: "token"` profiles.

## Best Possible Solution

Preserve Anthropic setup-token support, add Anthropic-specific doctor/status guidance for API-key-shaped static token profiles, and document the canonical Enterprise API-key profile shape.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves Anthropic token profiles, can rank them before API-key profiles, and reports non-expiring tokens as static/healthy. The original live 529 cascade needs Enterprise credentials, while current transport now sends non-`sk-ant-oat` values with `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing Anthropic token support would break the intentional setup-token route.
