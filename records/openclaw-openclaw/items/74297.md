## Summary

Keep open. Current main mitigates the original wrong-header path for `sk-ant-api` values, but persisted Anthropic `type: "token"` profiles are still accepted, can resolve as usable static credentials, can outrank API-key profiles, and are not surfaced by doctor/auth health with guidance to use `type: "api_key"`.

## What This Changes

Validate, diagnose, or document Anthropic `type: "token"` auth profiles that contain Enterprise API keys and should use the API-key profile shape.

## Best Possible Solution

Preserve Anthropic setup-token support, add an Anthropic-specific diagnostic for API-key-shaped static token profiles, and document the canonical `type: "api_key"` profile shape and migration guidance.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer it before an API-key profile, and reports it as a healthy static token. The exact live 529 cascade needs Anthropic Enterprise credentials, while current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; rejecting all Anthropic token profiles would be too broad because setup-token support is intentional.
