## Summary

Keep open. Current main mitigates the reported wrong-header failure for API-key-shaped Anthropic values, but it still accepts and prefers persisted Anthropic `token` profiles, reports non-expiring tokens as healthy/static, and lacks a clear doctor/status warning for Enterprise API-key profiles stored with the wrong credential type.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic Enterprise API keys that were stored in `auth-profiles.json` as `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve Anthropic setup-token support, add Anthropic-specific doctor/status guidance for API-key-shaped static token profiles, and document the canonical Enterprise API-key profile shape without changing the transport mitigation.

## Reproduction Assessment

Yes for the remaining source-level issue: current main accepts Anthropic token profiles, can choose them before API-key profiles, and treats non-expiring tokens as static/healthy. The original live 529 loop needs Enterprise credentials, and current transport now sends non-`sk-ant-oat` values through `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is diagnostics plus documentation; removing Anthropic token support would break the intentional setup-token route.
