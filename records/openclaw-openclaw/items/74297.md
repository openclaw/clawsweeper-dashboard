## Summary

Keep open. Current main mitigates the original wrong-header route for API-key-shaped Anthropic credentials, but the remaining diagnostics/docs gap is still present: stored Anthropic `type: "token"` profiles are accepted, can be preferred over API-key profiles, and are not surfaced by auth health or doctor when they contain a static API-key-shaped value.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic Enterprise API keys accidentally stored as `type: "token"` auth profiles.

## Best Possible Solution

Preserve supported Anthropic setup-token and generic token-profile behavior, but add an Anthropic-specific health/doctor warning for API-key-shaped static token profiles and document the canonical API-key profile shape.

## Reproduction Assessment

Yes for the source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer it before an API-key profile, and does not warn when the token is static. The exact live 529 cascade needs Anthropic Enterprise credentials, and current transport now sends `sk-ant-api` values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing all token support or rejecting Anthropic setup-token profiles would be too broad because setup-token support is intentional.
