## Summary

Keep open. Current main mitigates the original wrong-header path for API-key-shaped Anthropic values, but it still accepts and prioritizes static Anthropic `token` profiles without doctor/status guidance that steers Enterprise API-key users to canonical `api_key` profiles.

## What This Changes

The issue asks OpenClaw to prevent or clearly diagnose Anthropic Enterprise API keys being stored as `type: "token"` in `auth-profiles.json`.

## Best Possible Solution

Keep Anthropic setup-token support, but add focused doctor/status guidance and docs that warn when an Anthropic API-key-shaped credential is stored or selected as `token` and direct Enterprise/API-only users to `api_key`.

## Reproduction Assessment

Yes for the remaining current-main source path: static Anthropic `token` profiles are still accepted, selectable before `api_key`, and not flagged by doctor when non-expiring. No live reproduction was established for the original Enterprise 529 cascade without real Anthropic Enterprise credentials.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is diagnostic/status guidance plus docs while preserving supported setup-token behavior, not removing the token credential type outright.
