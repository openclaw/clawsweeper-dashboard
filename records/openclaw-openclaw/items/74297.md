## Summary

Keep open. Current main appears to have mitigated the original wrong-header path by sending resolved Anthropic credentials through `x-api-key`, but the remaining reported gap is still present: Anthropic `type: "token"` profiles with static API-key-shaped values are accepted, can be considered healthy, and are not called out by doctor or docs as a misconfiguration for Enterprise API-key accounts.

## What This Changes

The issue asks OpenClaw to validate, warn about, and document Anthropic Enterprise API-key credentials that are stored as auth-profile `type: "token"` instead of `type: "api_key"`.

## Best Possible Solution

Preserve supported Anthropic setup-token and generic token-profile behavior, but add an Anthropic-specific warning for static `sk-ant-api...` token profiles and document the canonical `api_key` auth-profile shape.

## Reproduction Assessment

Yes for the remaining source-level issue: current main accepts and resolves an Anthropic `type: "token"` profile, can prefer it over an API-key profile, and reports a static token as healthy. The exact live 529 cascade is unclear on current main because the transport now sends resolved Anthropic API-key values via `x-api-key`.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable solution is diagnostics plus documentation; removing all Anthropic token support would be too broad because setup-token support is intentional.
