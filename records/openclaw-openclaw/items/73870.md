## Summary

Keep open. Current main still has a global subagent lane plus reject-only per-requester active-child caps; it does not implement the requested queued per-parent spawn throttle, and the linked open concurrency issues cover different scheduling axes.

## What This Changes

Request a configurable per-parent concurrency limit for subagent spawns so same-parent fanout queues past the limit instead of saturating gateway resources.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from the global subagent lane and the active-child cap, covers native and ACP spawns, and defines queue visibility plus lifecycle cleanup semantics.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane scheduling plus reject-only active-child caps, not queued per-parent throttling. The reported 30-agent Slack timeout symptom itself remains unverified without a live load run.

## Solution Assessment

Unclear. A per-parent queued throttle plausibly targets same-parent fanout pressure, but the default cap and lifecycle/fairness behavior need maintainer policy before a safe fix PR.
