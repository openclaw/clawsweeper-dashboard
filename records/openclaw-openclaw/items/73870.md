## Summary

Keep open. Current main still has a global subagent execution lane plus reject-only per-session child caps; it does not implement the requested queued per-parent spawn throttle, and the related open scheduler issues cover adjacent but distinct policies.

## What This Changes

Request a configurable per-parent queued concurrency cap for subagent spawns so one fanout-heavy parent cannot saturate gateway/WebSocket work.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer across native and ACP spawns that composes with the existing global subagent lane and preserves child lifecycle cleanup.

## Reproduction Assessment

Yes for the missing product behavior: source, docs, and tests on current main show global lane scheduling plus reject-only active-child caps, not queued per-parent throttling. No live reproduction was attempted for the large-fleet Slack timeout symptom.

## Solution Assessment

Unclear. A queued per-parent cap is plausible, but the default limit, parent identity, fairness model, queue visibility, and abort semantics should be decided before encoding this policy.
