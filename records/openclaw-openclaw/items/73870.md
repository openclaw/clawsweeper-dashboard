## Summary

Keep open. Current main still has one global `subagent` command lane plus reject-only active-child caps; it does not implement queued requester/parent-keyed spawn admission, and related open issues cover adjacent per-target, lane, lifetime-budget, and queue-visibility scopes rather than superseding this request.

## What This Changes

The issue requests a configurable per-parent concurrency limit for subagent spawns, with excess spawns queued instead of letting one parent saturate gateway resources.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer for native and ACP spawns, with explicit fairness, status, cancellation, cleanup, and compatibility semantics alongside the existing global lane and active-child limits.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new scheduler and config behavior. Source inspection gives high-confidence proof that current main still lacks queued per-parent throttling, but no live 30-agent Slack reproduction was attempted.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but maintainers need to choose the parent key, defaults, fairness model, queue UX, and interaction with related open concurrency work.
