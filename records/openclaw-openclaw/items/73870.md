## Summary

Keep open. Current main still has one global subagent execution lane plus reject-only per-session child caps; it does not implement a queued parent-keyed spawn throttle, and the adjacent open scheduler issues cover per-agent caps and multi-lane routing rather than this exact policy.

## What This Changes

Request a configurable per-parent concurrency cap that queues subagent spawns above the limit to reduce gateway event-loop and WebSocket pressure during large fanout.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer for native and ACP spawns, with explicit fairness, status, cancellation, and cleanup semantics that compose with the existing global lane and maxChildrenPerAgent caps.

## Reproduction Assessment

Yes for the missing scheduler behavior: current source, docs, and tests show global lane scheduling plus reject-only active-child caps, not queued per-parent throttling. No live reproduction was attempted for the reported large-fleet CPU and Slack timeout symptom.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but the cap default, parent key, fairness model, queue UX, and lifecycle cleanup need maintainer policy before a safe implementation.
