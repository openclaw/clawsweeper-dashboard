## Summary

Keep open. Current main still lacks queued parent-keyed subagent throttling: it has one global subagent lane plus reject-only active-child caps, and the related open concurrency issues cover different per-agent or multi-lane designs rather than this requester/parent burst-control behavior.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess child spawns queued instead of allowing one parent’s fanout to saturate shared gateway/WebSocket work.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from both the global `subagent` lane cap and `maxChildrenPerAgent`. Queued native and ACP child runs should be visible in task/subagent state and have defined behavior for start, stop, restart, cleanup, and completion delivery.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the scheduler defaults, fairness model, task-state representation, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
