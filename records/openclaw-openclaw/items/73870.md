## Summary

Keep open. Current main still has a lane-wide subagent concurrency cap plus reject-only per-requester child limits; it does not implement a queued, per-parent spawn throttle, and the related open issues track different concurrency dimensions rather than replacing this request.

## What This Changes

Request a configurable per-parent concurrency cap for subagent spawns so excess child runs queue instead of all starting or being rejected.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn throttle that is separate from the global subagent lane cap and the active-child rejection cap. It should cover native and ACP spawns, preserve capacity for unrelated parents, expose queued/running state consistently, and define lifecycle behavior for completion, kill/cancel, restart, and cleanup.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane scheduling plus reject-only child caps, not queued per-parent throttling. The reported large-fleet Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent throttle directly matches the reported fanout problem, but the scheduler policy and lifecycle semantics are broad enough that maintainer design should precede an automated repair.
