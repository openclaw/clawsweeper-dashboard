## Summary

Keep open. Current main still has global subagent-lane queueing plus reject-only requester child caps; the requested queued parent-keyed spawn throttle remains a new scheduler/config policy and is not covered exactly by the adjacent open scheduler requests.

## What This Changes

Add a configurable per-parent queued concurrency cap for subagent spawns so excess same-parent fanout waits instead of saturating the gateway or being rejected.

## Best Possible Solution

Define and implement a documented requester/parent-keyed queued spawn admission layer for native and ACP sessions_spawn that composes with the global subagent lane, maxChildrenPerAgent, status visibility, cancellation, and cleanup semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the missing current-main behavior: source and tests show global lane queueing plus reject-only requester child caps, and the depth-limit test asserts maxConcurrent is not a per-parent gate. No live large-fleet CPU/Slack timeout reproduction was attempted.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A queued per-parent cap plausibly addresses same-parent fanout pressure, but maintainers still need to choose the default, parent key, fairness model, queue visibility, cancellation behavior, and interaction with existing caps.
