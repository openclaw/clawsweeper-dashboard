## Summary

Keep open. Current main still has a lane-wide subagent queue plus a reject-only per-session active-child cap; the requested queued per-parent spawn throttle remains missing, and the related open concurrency items track different axes rather than subsuming this parent-keyed burst-control behavior.

## What This Changes

The issue requests a configurable per-parent subagent spawn concurrency limit with queued excess spawns to reduce fanout CPU and shared WebSocket contention.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from both the global subagent lane and the active-child reject cap, with visible queued state and consistent native/ACP start, stop, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane queueing plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the scheduler defaults, fairness policy, state model, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
