## Summary

Keep open. Current main still has only a global subagent lane plus a reject-only per-session child cap; the requested queued per-parent throttle remains missing, and the related open concurrency issues track different axes rather than subsuming this parent-keyed burst-control behavior.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns so excess child runs queue instead of saturating gateway event-loop and WebSocket resources.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from both the global subagent lane and the active-child reject cap. Queued native and ACP child runs should be visible in task/subagent state, preserve capacity for unrelated parents, and define start, stop, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show a global lane plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but scheduler defaults, fairness policy, state representation, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
