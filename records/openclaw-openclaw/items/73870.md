## Summary

Keep open. Current main still exposes a single global subagent execution lane plus a reject-only active-child limit; it does not implement queued requester/parent-keyed spawn throttling, and the related open concurrency requests cover different dimensions.

## What This Changes

Request a configurable per-parent concurrency cap for subagent fanout so excess `sessions_spawn` children queue instead of all contending immediately.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from the global subagent lane and the active-child reject cap, covering native and ACP spawns plus task/subagent state, restart cleanup, stop/kill, and announce behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but scheduler defaults, fairness policy, task-state representation, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
