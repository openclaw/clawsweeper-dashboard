## Summary

Keep open. Current main still has a single global `subagent` execution lane plus reject-only per-requester child caps; it does not implement the requested queued per-parent spawn throttle, and related open issues cover adjacent per-agent or multi-lane scheduling rather than this parent-keyed policy.

## What This Changes

Requested a configurable per-parent queued concurrency limit for subagent spawns to reduce large fanout event-loop and WebSocket contention.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer across native and ACP spawns, with clear fairness, status, cancellation, and cleanup semantics aligned with existing global lane behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane scheduling plus reject-only active-child caps, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but the default limit, fairness model, queue UX, and lifecycle cleanup should be decided before changing scheduler behavior.
