## Summary

Keep open. Current main still routes native and ACP child runs through one global subagent lane and uses per-session active-child caps that reject new spawns; it does not implement a queued, parent-keyed spawn throttle. The open related reports cover per-target-agent and named-lane scheduling, which are adjacent but do not replace this parent-fanout control.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns so excess child runs queue instead of one parent saturating shared gateway/WebSocket work.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn throttle that is separate from the global subagent lane cap and the active-child reject cap. It should cover native and ACP child runs, preserve capacity for unrelated parents, and define queue visibility plus stop, cleanup, restart, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show a global lane plus reject-only child caps, and a current test explicitly asserts that subagent `maxConcurrent` is not a per-parent spawn gate. The large-fleet Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent throttle fits the reported fanout problem, but the implementation should not simply repurpose `maxChildrenPerAgent`; maintainers should settle defaults, fairness, queued task state, and lifecycle semantics before a repair lane writes code.
