## Summary

Keep open. Current main still implements a single global subagent execution lane plus a reject-only per-session child cap; it does not implement queued per-parent throttling, and the related open concurrency requests do not subsume this parent-keyed burst-control behavior.

## What This Changes

The issue requests a configurable per-parent subagent spawn concurrency cap, with excess child spawns queued instead of all children contending through the current shared path.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from the global subagent lane and the active-child reject cap. Queued native and ACP child runs should be represented in task/subagent state, preserve capacity for unrelated parents, and define start, stop, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only review.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the scheduler defaults, fairness policy, task-state representation, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
