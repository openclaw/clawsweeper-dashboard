## Summary

Keep open. Current main still has one global subagent execution lane plus reject-only per-session child caps; it does not implement queued per-parent spawn admission, and the related open items cover per-target caps, caller-selected lanes, or the broader event-loop symptom rather than this exact scheduler policy.

## What This Changes

Request a configurable per-parent queued concurrency cap for subagent spawns to reduce gateway event-loop and announce-path pressure during multi-wave fanout.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer for native and ACP spawns, with explicit fairness, status, cancellation, and cleanup semantics that compose with the existing global lane and `maxChildrenPerAgent` caps.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new scheduler/config behavior. Source inspection gives high-confidence proof that current main still lacks queued per-parent throttling, but no live 30-agent Slack reproduction was attempted.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but maintainers need to choose the parent key, default limit, fairness model, queue UX, and interaction with https://github.com/openclaw/openclaw/issues/75378, https://github.com/openclaw/openclaw/issues/8061, and https://github.com/openclaw/openclaw/issues/10467.
