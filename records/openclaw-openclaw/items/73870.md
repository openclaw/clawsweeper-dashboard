## Summary

Keep open. Current main still models subagent concurrency as a global subagent lane plus reject-only active-child caps, and it does not implement the requested queued per-parent/requester spawn throttle; related open issues #8061 and #10467 cover different scheduling axes.

## What This Changes

Request a configurable queued concurrency cap keyed by the parent/requester session for subagent spawns to reduce same-parent fanout pressure on the gateway.

## Best Possible Solution

Add a documented parent/requester-keyed queued spawn throttle that is distinct from the global subagent lane and the active-child admission cap, with matching native/ACP behavior, status visibility, lifecycle handling, docs, and regression tests.

## Reproduction Assessment

Yes for the missing product behavior: source, docs, and tests show global lane scheduling plus reject-only per-requester child caps, not queued per-parent throttling. The production Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued parent-keyed throttle is plausible, but maintainers should first choose the default, fairness behavior, queue visibility, and cleanup semantics across native and ACP spawns.
