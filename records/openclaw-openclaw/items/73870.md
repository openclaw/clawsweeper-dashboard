## Summary

Keep open. Current main still has a global subagent lane plus reject-only per-requester child caps; it does not implement the requested queued per-parent spawn throttle, and related open issues #8061 and #10467 track different scheduling axes.

## What This Changes

Request a configurable per-parent concurrency limit that queues excess subagent spawns to prevent same-parent fanout from saturating the gateway and Slack delivery.

## Best Possible Solution

Add a documented requester/session-keyed spawn throttle separate from the global subagent lane and `maxChildrenPerAgent`, covering native and ACP spawns with coherent queued/running state, completion, cancellation, restart, and cleanup semantics.

## Reproduction Assessment

Yes for the missing product behavior: source, docs, and tests show global lane scheduling plus reject-only per-requester child caps, not queued per-parent throttling. The reported 30-agent Slack timeout scenario was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent throttle plausibly addresses same-parent fanout pressure, but the defaults, fairness policy, lifecycle semantics, and user-visible queued state need maintainer design first.
