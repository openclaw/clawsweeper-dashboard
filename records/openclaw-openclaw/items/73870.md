## Summary

Keep open. Current main still lacks queued per-parent subagent spawn admission; it only has a lane-wide subagent queue plus reject-only active-child limits, so the reported fanout pressure remains an unresolved scheduler/config feature request.

## What This Changes

The issue requests a configurable per-parent concurrency cap that queues excess subagent spawns to reduce event-loop and WebSocket contention during multi-wave fanout.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer for native and ACP spawns that composes with the existing global lane queue, active-child caps, cancellation, cleanup, diagnostics, and related per-agent/per-lane proposals.

## Reproduction Assessment

No, not as a strict bug reproduction because this is a scheduler feature request. Source inspection gives high-confidence proof that current main lacks queued per-parent throttling, but the reported 30-agent Slack timeout scenario was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but maintainers need to choose the public config shape, parent identity, defaults, fairness model, and interaction with existing global lane and child-limit behavior.
