## Summary

Keep open. Current main still has one global `subagent` lane plus reject-only per-requester active-child caps; it does not implement queued per-parent spawn admission, and the related open items cover adjacent scheduler scopes rather than superseding this request.

## What This Changes

The issue requests a configurable per-parent concurrency cap that queues subagent spawns beyond the cap to reduce gateway event-loop saturation during fanout.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn admission layer for native and ACP spawns, with explicit fairness, status, cancellation, and cleanup semantics that compose with the existing global lane and active-child caps.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new scheduler/config behavior. Source inspection gives high-confidence proof that current main still lacks queued per-parent throttling, but no live 30-agent Slack reproduction was attempted.

## Solution Assessment

Unclear. A queued per-parent cap plausibly targets same-parent fanout pressure, but maintainers need to choose the parent key, default limit, fairness model, queue UX, and how it composes with related open concurrency work.
