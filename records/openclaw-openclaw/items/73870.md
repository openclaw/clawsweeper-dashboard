## Summary

Keep open. Current main still implements subagent concurrency as a global subagent lane plus reject-only per-requester child caps; it does not provide the requested queued per-parent spawn throttle, and the related open concurrency issues track different scheduling axes rather than superseding this one.

## What This Changes

Request a configurable per-parent concurrency cap for subagent spawns so excess children queue instead of all contending through the same gateway/WebSocket path.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is distinct from both the global subagent lane and the active-child admission cap. It should cover native and ACP child runs, expose queued/running state coherently, preserve capacity for unrelated parents, and define lifecycle behavior for completion, cancellation, restart, and cleanup.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane scheduling plus reject-only per-requester child caps, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle plausibly addresses same-parent fanout pressure, but defaults, fairness policy, queued-state UX, and lifecycle semantics need maintainer design before a safe implementation.
