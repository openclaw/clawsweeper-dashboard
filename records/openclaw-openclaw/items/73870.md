## Summary

Keep open. Current main still has a single shared `subagent` execution lane plus a reject-only active-child cap per requester session; it does not provide queued per-parent spawn throttling, and the related open concurrency issues cover adjacent but not identical behavior.

## What This Changes

Request a configurable per-parent concurrency cap for `sessions_spawn`, queueing child spawns beyond the cap to prevent large fan-out from saturating gateway resources and causing Slack timeouts.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from the global `subagent` lane and from `maxChildrenPerAgent`. Queued native and ACP child runs should be visible in task/subagent state and define start, stop, restart, cleanup, and completion semantics without starving unrelated parents.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global-lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the scheduler defaults, fairness policy, task-state representation, and lifecycle semantics are broad enough that maintainer design should come before an automated repair.
