## Summary

Keep open. Current main still has global subagent-lane concurrency plus reject-only per-session active-child caps, not the requested queued requester/parent-keyed spawn throttle; related open issues track adjacent but different scheduling axes.

## What This Changes

The issue requests a configurable queued per-parent concurrency cap for subagent spawns to prevent same-parent fanout from saturating the gateway/WebSocket path.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn throttle that remains distinct from the global subagent lane and `maxChildrenPerAgent`, and make it coherent for native and ACP child runs across completion, cancellation, restart, and cleanup.

## Reproduction Assessment

Yes for the missing requested behavior: current source, docs, and tests show lane-wide scheduling plus active-child rejection, not queued per-parent throttling. The production CPU/Slack timeout symptom itself still needs live workload proof.

## Solution Assessment

Unclear. A queued per-parent throttle plausibly addresses same-parent fanout pressure, but the default cap and lifecycle/fairness semantics need maintainer policy before an automated implementation is safe.
