## Summary

Keep open. Current main still has a single global subagent execution lane plus reject-only per-session child caps; it does not implement the requested queued per-parent spawn throttle, and the related open concurrency reports cover different scheduling axes rather than replacing this one.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess child spawns queued instead of letting one parent saturate the global subagent lane or hit a reject-only active-child cap.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is separate from both the global subagent lane and the active-child cap. It should cover native and ACP child runs, expose queued/running state coherently, preserve capacity for unrelated parents, and define start, stop, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show a global lane plus reject-only per-session child caps, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the reported fanout problem, but scheduler defaults, fairness policy, queued task visibility, and lifecycle semantics are broad enough that maintainer design should come before an automated repair.
