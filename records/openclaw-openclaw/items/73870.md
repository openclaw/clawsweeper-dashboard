## Summary

Keep open. Current main still has global subagent lane concurrency, reject-only per-requester active-child limits, and an announce-delivery queue, but it does not implement the requested queued per-parent spawn throttle. The related open issues #8061 and #10467 cover different concurrency axes and do not subsume requester-session keyed spawn queueing.

## What This Changes

The issue requests a configurable per-parent subagent spawn concurrency cap, defaulting around 2-3, with excess spawns queued to reduce fanout CPU and WebSocket contention.

## Best Possible Solution

Add a documented requester-session keyed spawn throttle that is distinct from the global `subagent` lane and `maxChildrenPerAgent`. Queued native and ACP child runs should be visible in task/subagent state, start when that parent has capacity, preserve fairness for unrelated parents, and define cancellation, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet CPU and Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the issue, but the scheduler default, fairness rule, task-state representation, and lifecycle semantics are broad enough that maintainer design should come before a narrow automated repair.
