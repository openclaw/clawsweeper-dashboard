## Summary

Keep this PR open. Current main still has the outbound hook/session-key regression and the PR is pointed at the right narrow runtime paths, but the current patch is not merge-ready because its hook JSDoc promises outbound `message_sent`/`runId` fields that the patch does not actually deliver, and it is missing the required user-facing changelog entry.

## What This Changes

The PR updates routed outbound reply session-key selection, passes the computed outbound session key into `message_sending`, adds outbound delivery tests, and expands hook/session context JSDoc.

## Best Possible Solution

Land the narrow runtime fix after the public contract is made exact: either implement and test `message_sent` session-key and outbound `runId` propagation, or narrow the new JSDoc to only promise the `message_sending` `sessionKey` behavior this PR actually provides, then add the changelog entry and a redirect regression test.

## Reproduction Assessment

Yes. Current main can be reproduced by invoking outbound delivery with a `message_sending` hook and `session.key`; the hook context is built without `sessionKey`. The native redirect mismatch is also visible by comparing `get-reply.ts` agent session resolution with the routed delivery parameters in `dispatch-from-config.ts`.

## Solution Assessment

No, not as written. The runtime plumbing is the right narrow direction, but the documentation advertises fields the patch does not deliver and the PR lacks the required changelog entry; the safer alternative is to narrow the docs or implement the missing field propagation with tests.
