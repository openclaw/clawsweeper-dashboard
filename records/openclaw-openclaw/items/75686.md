## Summary

Keep open. The PR targets a real packaged-runtime-dependency class, but the current diff is not safe to merge because it treats built plugin manifest dependencies as root package dependency requirements, which conflicts with the staged runtime-deps architecture now on main.

## What This Changes

The PR changes release validation to include built `dist/extensions/*/package.json` manifests in the root dependency mirror check and adds a Telegram dependency regression assertion.

## Best Possible Solution

Keep Discord and Telegram runtime dependencies extension-owned, and validate the built package by checking staged plugin payloads and activation/repair behavior before built channel imports run. If #75685 still exposes a release gap, add coverage in that staged package or post-publish verification path rather than root-mirroring every built plugin dependency.

## Reproduction Assessment

Yes. The paired issue gives a concrete v2026.4.29 packaged install reproduction, and this PR's introduced bug is statically reproducible from the diff by applying the root mirror check to staged Telegram or Discord deps that are intentionally not root deps.

## Solution Assessment

No. The proposed root dependency mirror expansion is not the best fix because current main separates root-owned dist imports from plugin-local staged runtime dependencies; validating staged payloads is narrower and matches the dependency contract.
