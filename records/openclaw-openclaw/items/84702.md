## Label Justifications

- `P1`: The PR changes the default for a production Feishu inbound transport path where failure can stop real messages from arriving.
- `merge-risk: 🚨 compatibility`: Existing Feishu setups that rely on ambient proxy env will change from proxied WebSocket routing to direct routing unless a new env var is configured.
- `merge-risk: 🚨 message-delivery`: If a proxy-only deployment loses WebSocket connectivity after upgrade, Feishu inbound messages stop arriving.
- `merge-risk: 🚨 security-boundary`: The diff changes whether Feishu WebSocket traffic follows inherited process proxy settings, which can affect operator egress-control assumptions.

## Summary

This is a valid Feishu WebSocket proxy bug fix with strong transport proof, but it should stay open because the default proxy-policy flip still needs an approved upgrade path and the new test env var is not restored.

## What This Changes

The PR changes Feishu WebSocket client creation to ignore ambient proxy env unless `OPENCLAW_FEISHU_WS_USE_PROXY=1` and updates proxy handling tests.

## Best Possible Solution

Land a Feishu-owned proxy policy that keeps the intended direct default only after maintainer approval, documents the opt-in and upgrade path, and restores every env mutation in the tests.

Yes, at source level. Current main unconditionally resolves an ambient proxy agent for Feishu WS, the linked issue gives a concrete failing proxy setup, and the PR body supplies local WSS/proxy proof; I did not run live Feishu production credentials in this read-only review.

## Solution Assessment

No, not yet. The code touches the right Feishu-owned call path, but the default flip needs an explicit upgrade/security-policy decision and the tests need to clean up the new env var before this is the safest merge path.

## Risks / Open Questions

- Existing Feishu installs that intentionally rely on `HTTPS_PROXY`, `HTTP_PROXY`, or `ALL_PROXY` for WebSocket egress will switch to direct connections after upgrade unless operators discover and set the new opt-in env var.
- Proxy-only Feishu deployments can lose inbound message delivery if the WebSocket can no longer reach Feishu directly after this default change.
- Operators using inherited process proxy env as an egress-control boundary may have Feishu WebSocket traffic bypass that proxy unless maintainers explicitly approve and document the new contract.
- The Feishu client tests can leave `OPENCLAW_FEISHU_WS_USE_PROXY=1` in a non-isolated Vitest worker after the new opt-in tests run.
