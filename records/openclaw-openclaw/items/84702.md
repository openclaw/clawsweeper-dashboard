## Label Justifications

- `P1`: The PR targets a Feishu WebSocket default that can stop real inbound channel delivery for users with broken ambient proxy env.
- `merge-risk: 🚨 compatibility`: Merging changes the default for existing Feishu deployments that relied on inherited proxy env for WebSocket egress.
- `merge-risk: 🚨 message-delivery`: A wrong proxy default in either direction can prevent Feishu WebSocket connection and stop inbound messages.
- `merge-risk: 🚨 security-boundary`: The diff changes whether Feishu WebSocket traffic follows process proxy env by default, which is an operator egress-policy boundary.

## Summary

Keep open: the root-cause and supplied real transport proof support the Feishu-local fix, but the branch should not merge until the new opt-in env is restored in tests, the opt-in is documented, and maintainers explicitly accept the ambient-proxy compatibility and egress-policy change.

## What This Changes

The PR makes Feishu WebSocket proxy-agent creation opt in through `OPENCLAW_FEISHU_WS_USE_PROXY=1` and updates Feishu client tests for default-direct and opt-in proxy behavior.

## Best Possible Solution

Land the Feishu-local default-direct gate after restoring test env cleanup, documenting the opt-in and upgrade behavior, and explicitly accepting the ambient proxy compatibility change while preserving managed proxy routing semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows Feishu WebSocket creation resolves an ambient proxy agent from process proxy env and passes it to the Lark WS client; the PR body also includes after-fix local WSS/proxy transport proof, though I did not run a live Feishu app in this read-only review.

## Solution Assessment

Yes, with completion work. The Feishu-owned `getWsProxyAgent()` gate is the narrow maintainable fix for the reported root cause, but it needs test env cleanup, docs for the new opt-in, and maintainer acceptance of the proxy default change before merge.
