## Summary

Keep open. The report is credible for an existing env-backed provider-auth path, but current source still leaves the failing boundary split between native subagent dispatch, ACP process env filtering, and the reporter's custom provider mapping, so this is not a safe cleanup close or automated fix candidate yet.

## What This Changes

The issue asks for `sessions_spawn` subagent or ACP sessions to preserve gateway-process environment variables needed by env-backed provider API keys, or to document an equivalent handoff mechanism.

## Best Possible Solution

Prove native subagent and ACP `sessions_spawn` auth separately, then preserve env SecretRef resolution through the active gateway secrets snapshot or a narrow explicit env handoff instead of forwarding the full host environment.

## Reproduction Assessment

Unclear. The issue gives a detailed systemd/env/custom-provider path and an observed 401, but I did not run a live gateway/systemd/Alibaba setup and source inspection leaves the failing boundary split between native subagent dispatch, ACP process spawn, and custom provider auth.

## Solution Assessment

Unclear. Full parent-environment inheritance is not the best default because ACP currently strips provider auth env vars deliberately; the safer fix is to maintain SecretRef semantics through the gateway runtime snapshot and only add constrained env propagation where a live repro proves it is needed.
