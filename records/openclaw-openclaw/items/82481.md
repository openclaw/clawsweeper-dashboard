## Summary

Keep open. The report is credible for an existing env-backed custom-provider auth path, but current main does not isolate the failing boundary well enough to close or queue an automated fix; the prior related discussion also rejected broad gateway-env copying as a secret-leak risk.

## What This Changes

The issue requests that `sessions_spawn` subagent/ACP runs preserve gateway process env-backed provider credentials, or provide a documented mechanism for passing them safely.

## Best Possible Solution

Prove native subagent model auth and ACP spawn auth separately, then preserve env SecretRef resolution through the active gateway secrets snapshot or a narrow explicit env contract rather than forwarding the full host environment.

## Reproduction Assessment

Unclear. The issue provides concrete systemd/custom-provider/subagent steps and an observed 401, but I did not run a live gateway/systemd/Alibaba setup and source inspection leaves native subagent auth, ACP process env filtering, and custom provider mapping as separate possible boundaries.

## Solution Assessment

Unclear. Broad parent-environment inheritance is not the best default because ACP intentionally strips provider auth env vars; the safer direction is to preserve resolved SecretRef/auth state through the gateway runtime snapshot or add a constrained env handoff only where a repro proves it is needed.
