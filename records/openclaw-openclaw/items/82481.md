## Summary

Keep open. The report is credible and affects an existing env-backed provider-auth path, but current source does not yet give a high-confidence failing path for native subagents because they dispatch back through the gateway agent runtime; ACP env stripping and custom-provider auth make the exact boundary ambiguous.

## What This Changes

Ensure `sessions_spawn` subagent/ACP runs can resolve env-backed provider API keys from the gateway process, or document the supported handoff mechanism.

## Best Possible Solution

Prove the native and ACP `sessions_spawn` auth paths separately, then fix env SecretRef resolution through the active secret runtime snapshot or a constrained handoff rather than broadly forwarding all host environment variables.

## Reproduction Assessment

Unclear. The issue gives concrete systemd/env steps and an observed 401, but I did not run a live gateway/systemd setup and current source leaves the failing boundary ambiguous between native subagent, ACP process spawn, and custom provider auth.

## Solution Assessment

Unclear. Inheriting the full parent process environment would be risky; the safer direction is to preserve documented SecretRef/provider auth semantics through the gateway runtime snapshot and only add explicit env handoff where ACP actually requires it.
