## Summary

Keep open: current main still uses nullish fallback for Telegram account groups, so an explicit empty account-level groups map suppresses root groups, and the raw group-policy path can reject the same chat before routing. The linked fix attempts are both closed unmerged, so no canonical implementation has landed.

## What This Changes

The issue asks Telegram single-account configs to treat explicit empty `accounts.<id>.groups: {}` as absent so root `channels.telegram.groups` still routes allowed group updates.

## Best Possible Solution

Update Telegram group resolution so empty account-level group maps count as absent only for single-account inheritance, make merge and raw policy resolution agree, and add focused regression coverage for both paths.

## Reproduction Assessment

Yes, at source level: the reported config reaches both `mergeTelegramAccountConfig` and `resolveChannelGroupPolicy` with `account.groups` set to `{}`, and current nullish fallback semantics prevent root groups from being considered. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No, not as a merge-only one-line fix: the raw group-policy resolver also needs aligned empty-as-absent semantics. The maintainable fix should cover both paths while keeping multi-account root-group suppression intact.
