## Summary

Keep open: current main still treats an explicit empty Telegram account-level `groups` map as a real override, so the single-account root `channels.telegram.groups` allowlist is skipped and the runtime policy path can reject the configured group as not allowed. The two linked fix PRs are closed unmerged, and I did not find an open replacement.

## What This Changes

Treat explicit empty Telegram account-level `groups` as absent in single-account configs so root `channels.telegram.groups` continues to authorize configured groups.

## Best Possible Solution

Implement Telegram-owned empty-groups-as-absent semantics for single-account group resolution across the merge helper and raw policy resolver, preserve multi-account isolation, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level: the reported config reaches `mergeTelegramAccountConfig` and `resolveChannelGroups` with `account.groups` set to `{}`, and both current paths skip root `channels.telegram.groups`. I did not replay a live Telegram update in this read-only pass.

## Solution Assessment

Yes on the direction, but not as a merge-helper-only change. The maintainable fix should make Telegram merge behavior and raw group-policy authorization agree while preserving the existing multi-account isolation contract.
