## Summary

Keep open: current main still allows an explicit empty Telegram account-level `groups` map to shadow the root group allowlist, and the same nullish fallback pattern exists in the runtime group-policy resolver used before the `not-allowed` log.

## What This Changes

The issue requests that single-account Telegram configs treat `accounts.<id>.groups: {}` like an absent account-level groups override so root `channels.telegram.groups` continues to route allowed group updates.

## Best Possible Solution

Treat an empty Telegram account-level groups object as no groups override for single-account inheritance, keep multi-account root groups suppressed, and cover both account merging and runtime group-policy access with regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure one Telegram account with root groups plus `accounts.default.groups: {}`, then the nullish group fallback in both merge and group-policy resolution selects the empty object instead of the root allowlist.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust Telegram group inheritance semantics for empty account-level maps while preserving the multi-account behavior shipped for https://github.com/openclaw/openclaw/issues/30673; adding only a warning would leave the configured route broken.
