## Summary

Keep open: current main still has the empty-object fallback bug in both Telegram account merging and runtime group-policy resolution, and an open linked PR already targets the fix with closing syntax.

## What This Changes

Treat explicit empty `accounts.<id>.groups: {}` as absent for single-account Telegram root group inheritance so root `channels.telegram.groups` continues to allow configured groups.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81030 once its live Telegram proof and maintainer review requirements are satisfied.

## Reproduction Assessment

Yes, at source level: the reported config shape leaves `account.groups` as `{}` in both the Telegram merge path and the generic group-policy resolver, so root groups are not considered and an allowlisted chat is rejected. I did not run a live Telegram gateway scenario in this read-only pass.

## Solution Assessment

Yes: the maintainable fix is to make the Telegram merge path and generic group-policy resolution share the same empty-as-absent single-account semantics while preserving multi-account isolation. The linked open PR is the appropriate path to review rather than closing the issue early.
