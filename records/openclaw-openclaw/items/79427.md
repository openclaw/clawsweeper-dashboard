## Summary

Keep open: current main still has the reported Telegram empty-account-groups behavior, and an open closing PR already targets this issue, so the issue should wait for that PR to land or be closed.

## What This Changes

Fix Telegram single-account group resolution so an explicit empty `accounts.<id>.groups: {}` does not override root `channels.telegram.groups` and drop allowlisted group updates.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81030, or an equivalent fix that updates both Telegram account merge and generic group-policy resolution with regression coverage and live Telegram proof.

## Reproduction Assessment

Yes, at source level: the reported config reaches both `mergeTelegramAccountConfig` and `resolveChannelGroupPolicy` with `account.groups` set to `{}`, which blocks root group fallback and produces the not-allowed path. I did not run a live Telegram gateway repro in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow maintainable fix because it updates both the merge helper and the runtime policy resolver while preserving multi-account isolation. A merge-only change would leave the generic group-policy path inconsistent.
