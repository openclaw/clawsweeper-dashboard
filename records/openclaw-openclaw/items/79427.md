## Summary

Keep open: checked-out main still uses nullish account-groups fallbacks in both Telegram account merging and the raw runtime group-policy resolver, so an explicit empty account groups map can still suppress root Telegram groups. Two linked fix PRs were closed unmerged, so the issue is still actionable.

## What This Changes

The issue asks for Telegram single-account configs with explicit empty `accounts.<id>.groups: {}` to inherit root `channels.telegram.groups` instead of dropping all allowlisted group updates.

## Best Possible Solution

Implement a Telegram-owned group resolution rule that treats empty account-level group maps as absent for single-account inheritance, preserves multi-account isolation, and makes the merged account config and raw gateway policy path agree.

## Reproduction Assessment

Yes, at source level: the reported config reaches `mergeTelegramAccountConfig` with `account.groups` set to `{}`, and the runtime group-policy resolver independently uses the same nullish fallback behavior. I did not run a live Telegram gateway scenario in this read-only pass.

## Solution Assessment

No, not as a merge-only fix. The narrow fix direction is right, but it also needs to update the raw `resolveChannelGroupPolicy` path while preserving the multi-account contract from https://github.com/openclaw/openclaw/issues/30673.
