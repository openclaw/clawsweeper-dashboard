## Summary

Keep open: current main still treats `accounts.default.groups: {}` as an account-level value before falling back to root Telegram groups, and the group-policy path also sees that empty account map and rejects the configured chat allowlist, matching the reported `not-allowed` drop.

## What This Changes

The issue asks Telegram single-account configs with an explicit empty account-level `groups` map to keep inheriting root `channels.telegram.groups` instead of blocking group updates.

## Best Possible Solution

Implement a Telegram-owned group resolution path that treats empty account-level group maps as absent for single-account inheritance, preserve multi-account isolation from https://github.com/openclaw/openclaw/issues/30673, and cover the merge plus runtime policy path with regression tests.

## Reproduction Assessment

Yes, at source level: the reported config reaches `mergeTelegramAccountConfig` with `account.groups` set to `{}`, and both merge and group-policy resolution use nullish fallback semantics that prevent root `channels.telegram.groups` from being considered. I did not run a live Telegram gateway scenario in this read-only pass.

## Solution Assessment

No, the issue's one-line merge-only fix is necessary but likely incomplete because `bot-core` also checks group policy through raw config via `resolveChannelGroupPolicy`. The best fix should make Telegram merge, group-policy, and mention/tool group lookup agree on the same empty-as-absent single-account semantics without changing unrelated channels.
