## Summary

Current main still preserves separate Slack DM and channel sessions, mirrors message-tool sends into the destination session, and blocks unrelated sibling-session access under the default tree visibility. The issue remains necessary, but it is an enhancement requiring a deliberate privacy and authorization contract rather than a narrow bug fix; the closed DM-history report addresses hydration within a DM, not recall of the agent’s channel activity from a DM.

## What This Changes

Add a safe way for an agent operating in a Slack DM to recall its own outbound messages and conversations from other Slack channels or threads.

## Best Possible Solution

Introduce a generic, permission-aware query for the agent’s own outbound activity, keyed by agent, Slack account/team, destination, thread, and provider message identifier, backed by the canonical SQLite state owner and without merging full transcripts or widening general session-tool visibility.

## Reproduction Assessment

Yes, from source with high confidence: send a message from a DM/main turn to a Slack channel, observe that it is mirrored to the destination channel session, then query from the DM where default tree visibility cannot access that sibling session. A live Slack run was not needed to establish the current structural limitation.

## Solution Assessment

No, the proposed unified-session and unrestricted cross-session-history options are not the best solution because they weaken isolation and audience boundaries. A narrow, authorized self-authored activity query is the safest maintainable direction.
