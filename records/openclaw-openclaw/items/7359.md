## Label Justifications

- `P2`: The issue causes a meaningful but bounded Slack continuity failure without making the core runtime or channel unusable.
- `impact:session-state`: The agent’s outbound channel message is recorded in a sibling destination session that the active DM session cannot recall.
- `impact:security`: Cross-session recall could disclose private channel activity unless Slack identity and destination authorization are enforced.

## Summary

Keep open. Current main still isolates Slack DM/main sessions from channel sessions, mirrors message-tool sends only into the destination session, and hides sibling sessions under the default visibility policy; resolving that gap requires an explicit privacy and authorization design.

## What This Changes

Add privacy-preserving recall of the agent’s own outbound Slack channel messages from a separate Slack DM session.

## Best Possible Solution

Add a bounded per-agent SQLite record of the agent’s own outbound Slack activity, keyed by Slack account, destination, thread, and message identifier, with authorization-aware lookup from DMs; do not merge full transcripts or relax general session visibility.

## Reproduction Assessment

Yes, from current-main source with high confidence: outbound sends are mirrored to the destination session, while default `tree` visibility excludes that sibling channel session from the requesting DM. A live Slack workspace reproduction was not run during this read-only review.

## Solution Assessment

No, unified Slack sessions or globally broader session visibility are not the best solution because they weaken isolation and expose unrelated transcript content. A purpose-built, authorization-aware self-activity query is narrower and safer.
