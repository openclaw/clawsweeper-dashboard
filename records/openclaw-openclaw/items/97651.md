## Summary

Keep open: current main and v2026.6.10 still replay ordinary tool-result text into OpenAI-compatible model messages, while existing pruning and guard paths only trim old or oversized results rather than solving the reported recent heartbeat/mcporter cache-break path.

## What This Changes

The issue asks OpenClaw to prevent heartbeat/tool-call output from destabilizing DeepSeek prefix-cache reuse in long-running agent sessions.

## Best Possible Solution

Define a cache-safe lifecycle for heartbeat or otherwise ephemeral tool results that preserves required provider tool-call pairing while keeping low-value outputs out of DeepSeek's reusable prefix, then prove it with focused prompt-shape tests and redacted live DeepSeek cache telemetry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection confirms current main and v2026.6.10 replay tool-result text into OpenAI-compatible model messages, and the issue includes cacheRead logs, but no live DeepSeek repro was run in this review.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the report identifies a real model-visible tool-result path, but excluding tool output entirely is too broad; maintainers should choose a narrower heartbeat or ephemeral-result policy that preserves tool-call contracts.
