## Summary

Keep open: current main and v2026.6.10 still replay normal tool-result text into OpenAI-compatible model messages, and the existing heartbeat isolation/pruning knobs are optional mitigations rather than a default fix for the reported DeepSeek cache-collapse path.

## What This Changes

The issue reports that heartbeat and mcporter tool outputs are appended to the model-visible session transcript, reducing DeepSeek prefix-cache reuse and increasing long-session cost.

## Best Possible Solution

Define a cache-safe lifecycle for heartbeat or ephemeral tool results that preserves required provider tool-call pairing while keeping low-value repetitive outputs out of DeepSeek's reusable prefix, then prove it with source tests and live or redacted DeepSeek telemetry.

## Reproduction Assessment

No, not for the full live provider failure: source inspection confirms current main replays normal tool-result text into OpenAI-compatible prompts, and the issue includes cacheRead logs, but this review did not run a live DeepSeek session.

## Solution Assessment

Unclear: the report identifies a real model-visible tool-result path, but excluding all tool output is too broad; the safer solution is a narrow heartbeat/ephemeral-result policy or isolated-session default discussion that preserves tool-call contracts.
