## Summary

Keep open: current main and v2026.6.10 still send non-empty toolResult text into OpenAI-compatible role tool messages, while heartbeat isolation and context pruning are optional or pressure-based mitigations rather than a default fix. A second user has now confirmed the same DeepSeek v2026.6.10 cache/cost pattern from transcript analysis, so this remains actionable but needs maintainer product and live-provider validation.

## What This Changes

The issue reports that normal tool and mcporter heartbeat outputs are replayed into the OpenAI-compatible DeepSeek prompt, disrupting prefix-cache hits and increasing session cost/context size.

## Best Possible Solution

Define a cache-safe lifecycle for heartbeat or otherwise ephemeral tool results that preserves required tool-call pairing while keeping repetitive low-value outputs out of DeepSeek's reusable prefix, then prove it with focused source tests and live or redacted DeepSeek telemetry.

## Reproduction Assessment

No, not for the full live provider failure. Source inspection confirms current main still replays normal tool-result text into OpenAI-compatible prompts and the issue includes cacheRead logs, but this review did not run a live DeepSeek session.

## Solution Assessment

Unclear: excluding all tool output is too broad because provider tool-call pairs and follow-up reasoning need some result visibility. The safer path is a narrow heartbeat or ephemeral-result policy, or provider/cache-aware pruning, chosen by maintainers and proven against DeepSeek telemetry.
