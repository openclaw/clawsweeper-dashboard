## Summary

Keep open: current main and v2026.6.10 still replay normal tool-result text into OpenAI-compatible model messages, while existing pruning/guard paths only mitigate old or oversized results rather than the reported recent heartbeat/mcporter cache-break mechanism.

## What This Changes

The issue reports that model-visible tool-result messages in long DeepSeek sessions break provider prefix-cache reuse and inflate heartbeat costs.

## Best Possible Solution

Define a cache-safe tool-result lifecycle for heartbeat/ephemeral tool calls that preserves required provider tool-call pairing while keeping stale or low-value outputs out of DeepSeek's reusable prefix, then prove it with focused source tests and live or redacted DeepSeek cache telemetry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection confirms current main and v2026.6.10 replay tool-result text into OpenAI-compatible model messages, and the issue includes detailed cacheRead logs, but no live DeepSeek repro was run in this review.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the report correctly identifies a risky model-visible tool-result path, but excluding tool output entirely is too broad; maintainers should choose a narrower heartbeat/ephemeral-result policy that preserves tool-call contracts.
