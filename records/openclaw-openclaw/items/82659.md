## Summary

Keep open: the updated branch fixes the earlier wrong-CODEX_HOME review finding, but it still lowers Codex command approval to `never` without preserving managed approval-policy requirements, and the supplied proof does not show the patched gateway/app-server path running end to end.

## What This Changes

This PR adds Codex app-server approval-policy downgrading when the bridged Codex home disables hooks, skips re-promotion for that downgrade, emits warnings, and adds focused config/run-attempt coverage.

## Best Possible Solution

Keep the bridged Codex-home detection, but preserve managed approval requirements and fail clearly instead of downgrading when `never` is not an allowed policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: combine `allowed_approval_policies = ["on-request"]` with bridged Codex-home `features.hooks = false`, and the branch changes the effective policy to `never` before promotion/requirements are considered.

## Solution Assessment

No. The CODEX_HOME placement fix is good, but the downgrade must be constrained by the same approval-policy requirements that current main and upstream Codex treat as admin-enforced.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor approval requirements before downgrading:** `extensions/codex/src/app-server/config.ts:1095-1096`
  - body: The downgrade unconditionally changes a non-`never` policy to `never` when the bridged Codex home has `features.hooks=false`. In a managed install with `allowed_approval_policies = ["on-request"]`, current main deliberately selects `on-request`; this branch bypasses that local requirement check and can either violate the required approval posture or make Codex reject the run. Gate the downgrade on `never` being allowed, or fail with a clear diagnostic when hooks are disabled but approvals are required.
  - confidence: 0.86
