## Summary

Keep this PR open: the reported Codex exec-decline failure is plausible and tied to recent native-tool-policy work, but the patch is not currently mergeable because it reads hook state from a different Codex config location than the spawned app-server uses, and the supplied real-behavior proof exercises the existing env workaround rather than the patched resolver.

## What This Changes

The PR adds Codex app-server approval-policy downgrading when user Codex `features.hooks` is disabled, prevents re-promotion to `untrusted`, emits relay/downgrade warnings, and adds resolver/run-attempt tests.

## Best Possible Solution

Resolve hook availability from the same effective Codex home/config layer the app-server will actually use, then warn clearly without widening command execution based on an unrelated host config.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: put `features.hooks=false` in the app-server CODEX_HOME that `auth-bridge.ts` injects, leave process HOME/CODEX_HOME different, and the new resolver will inspect the wrong file before spawning Codex.

## Solution Assessment

No, not as currently implemented. The fix should be moved to a point that knows the bridged app-server CODEX_HOME, or use Codex/app-server state directly, so the policy downgrade matches the real runtime config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve hooks state from the spawned Codex home:** `extensions/codex/src/app-server/config.ts:386-393`
  - body: The downgrade reads `params.env` / `process.env` before `bridgeCodexAppServerStartOptions` injects the `CODEX_HOME` actually passed to the stdio app-server. Normal local launches set `CODEX_HOME` to the agent-owned `codex-home`, and spawn env overlays `startOptions.env`, so this can miss disabled hooks in the real Codex home or downgrade to `never` based on an unrelated host config. Move the detection after start options are bridged, or resolve against the same CODEX_HOME that will be spawned.
  - confidence: 0.92
