## Summary

Keep open: the PR targets a real Codex app-server failure mode, but the current patch can turn a fail-closed OpenClaw tool-policy path into unguarded Codex-native command execution when user Codex hooks are disabled.

## What This Changes

The PR reads user Codex hook settings, downgrades non-never Codex app-server approval policies to never when hooks are disabled, adds one-shot warnings, and covers the downgrade and relay-miss paths with tests.

## Best Possible Solution

Preserve OpenClaw tool-policy enforcement: only downgrade to never when no OpenClaw policy hook would be bypassed, and otherwise fail closed with an actionable configuration warning or require an explicit OpenClaw approval-policy override.

## Reproduction Assessment

Yes at source level: Codex 0.130 gates hooks behind the CodexHooks feature and current OpenClaw routes native policy through those hooks or app-server approvals. I did not run a live current-main gateway reproduction in this read-only review.

## Solution Assessment

No: downgrading to never and then skipping OpenClaw's promotion path is not the safest fix because it can bypass configured before_tool_call policy. The narrower maintainable solution is to keep fail-closed policy behavior when OpenClaw policy hooks are present.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve OpenClaw tool policy when hooks are disabled:** `extensions/codex/src/app-server/run-attempt.ts:479-480`
  - body: When a before_tool_call policy exists, current main promotes implicit approvalPolicy="never" to "untrusted" so Codex-native shell/file requests pass through OpenClaw policy. This early return leaves the downgraded policy at "never", so a user with features.hooks=false gets native commands executed without the configured policy instead of failing closed or surfacing a configuration error. Gate the downgrade on no policy hooks, or keep a fail-closed path when policy is present.
  - confidence: 0.88
