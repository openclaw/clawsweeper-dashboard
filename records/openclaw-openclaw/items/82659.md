## Summary

Keep open: the PR addresses a valid high-impact Codex app-server failure and the latest diff looks source-level plausible, but it still needs after-patch real gateway/app-server proof of the new downgrade path before merge.

## What This Changes

Adds Codex app-server logic to downgrade non-never approval policies to `never` when the bridged Codex home disables hooks, preserve managed requirements, warn once, and add regression coverage.

## Best Possible Solution

Land the guarded bridged-CODEX_HOME downgrade after redacted patched-runtime proof is supplied and a maintainer accepts the approval-policy tradeoff.

## Reproduction Assessment

Yes at source/log level: stdio Codex app-server with a non-`never` approval policy and bridged Codex-home `features.hooks=false` can leave exec approvals unable to reach the native hook relay, yielding declined bash/tool results. I did not run a live current-main gateway repro in this read-only review.

## Solution Assessment

Yes at the source level, pending proof and maintainer acceptance. The patch checks the same bridged Codex home the child process reads, preserves managed `allowed_approval_policies`, and prevents policy promotion from undoing the downgrade.
