## Summary

Keep open. The PR addresses the right TUI UX mismatch, but the current diff only checks `activeChatRunId`, so it misses the fast submit window where a run is pending locally but not yet bound to a gateway run id; it also needs the required user-facing changelog entry.

## What This Changes

This PR appends a Ctrl+C status hint in `src/tui/tui.ts` when `activeChatRunId` is set, leaving abort behavior and keybindings unchanged.

## Best Possible Solution

TUI should present one consistent cancellation policy: either keep Ctrl+C as clear/exit and show the run-active hint for both pending optimistic sends and bound runs, or land the broader abort-first behavior from #44223 with tests and docs aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit a TUI message and press Ctrl+C before the first chat event arrives: `pendingOptimisticUserMessage` is true, `activeChatRunId` is null, and this PR's hint condition does not fire while the run can continue.

## Solution Assessment

No as written. A text-only hint is a narrow maintainable fix if Ctrl+C remains clear/exit, but the predicate must include pending submissions and the user-visible change needs changelog coverage.
