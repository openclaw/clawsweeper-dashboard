## Summary

Keep open. The PR targets a valid TUI UX mismatch, but the current diff only checks `activeChatRunId`, so it misses the fast submit window where a run is pending locally but not yet bound to a gateway run id; it also needs the required user-facing changelog entry.

## What This Changes

The PR appends a run-still-active hint to TUI Ctrl+C clear/warn status messages when `activeChatRunId` is set, without changing abort behavior or keybindings.

## Best Possible Solution

TUI should have one documented cancellation policy: if Ctrl+C remains clear/exit, the status hint should cover both pending optimistic sends and bound active runs; if Ctrl+C should abort active runs, #44223 should be reviewed as the policy-changing path.

## Reproduction Assessment

Yes. Submit a TUI message and press Ctrl+C before the first chat event arrives; source shows `pendingOptimisticUserMessage` is true while `activeChatRunId` is still null, so this PR's hint condition does not fire while the run can continue.

## Solution Assessment

No as written. A text-only hint is maintainable for the current Ctrl+C policy, but the predicate must include pending submissions and the user-visible change needs changelog coverage.
