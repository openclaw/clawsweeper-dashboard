## Summary

Keep open: the current main source still admits the reported Discord heartbeat leak, and an open focused PR already references this issue with closing syntax, so the issue should stay open until that PR lands or is rejected.

## What This Changes

Discord scheduled heartbeat/channel turns can show failed internal file-read progress such as `print lines ... failed` in visible chat despite group/channel message-tool-only delivery being configured.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/85357, while keeping broader Discord internal-output leak tracking separate.

## Reproduction Assessment

Yes at source level: current main can bypass heartbeat `message_tool_only` delivery because heartbeat prompt selection ignores `messages.groupChat.visibleReplies` while dispatch honors it. I did not live-replay the Discord scheduled turn.

## Solution Assessment

Yes: the best fix is to make heartbeat response-tool selection reuse the existing group/channel visibility and runtime-policy contracts instead of adding Discord-specific output filtering. The open linked PR follows that direction, but it has not merged yet.
