- `impact:security`: The visible failed-read messages can expose local file paths and operational context to chat participants.
- `impact:message-loss`: The issue is about internal runtime messages being delivered to the wrong user-visible channel surface.

## Summary

Keep open: the report is source-plausible on current main because scheduled heartbeat/cron-style Discord runs do not fully reuse the group/channel visible-reply policy or model-specific runtime policy that normally keeps message-tool-only output private.

## What This Changes

The issue asks OpenClaw to stop showing failed internal file-read/tool-progress messages as visible Discord chat output.

## Best Possible Solution

Align heartbeat scheduled runs with the same source-reply visibility and runtime policy used for normal group/channel turns, including groupChat visibleReplies and model-specific Codex runtime overrides, then add regression coverage for Discord/channel scheduled turns with failed text-only tool progress.

## Reproduction Assessment

Yes at source level: the reported config can bypass heartbeat message-tool-only source delivery even though normal group/channel dispatch honors it. I did not live-replay the Discord scheduled turn.

## Solution Assessment

No patch is proposed yet; the maintainable fix is to share or mirror the existing source visibility/runtime resolution for heartbeat runs rather than adding a Discord-only sanitizer special case.
