## Label Justifications

- `P2`: This is a persistent user-visible TUI reporting bug with limited functional blast radius and no data loss or availability impact.
- `impact:session-state`: The footer inaccurately represents the active conversation's context-usage state.

## Summary

Keep open: current main and release v2026.6.5 still render `tokens ?/<context>` when lightweight TUI session metadata has a known context window but no exact total. The earlier zero-default fixes were correctly closed because missing or stale usage is not equivalent to zero; the merged context-budget reporting work now provides the basis for a truthful estimated fallback.

## What This Changes

Make the TUI footer show meaningful context usage instead of `?/<context window>` when exact usage is unavailable.

## Best Possible Solution

Carry the minimum freshness and estimated context-budget state into the TUI, prefer fresh exact totals, render a current estimate with explicit estimated notation when exact usage is unavailable, and retain `?` only when neither source is trustworthy.

## Reproduction Assessment

Yes, from current source with high confidence: lightweight session metadata can contain `contextTokens` without `totalTokens`, and the footer deterministically renders that state as `tokens ?/<context>`. A live provider-backed TUI run was not performed.

## Solution Assessment

Yes in direction, but not via the closed zero-default approach. The narrowest maintainable fix is to reuse the existing fresh-total and estimated-context semantics rather than inventing a conflicting token value.
