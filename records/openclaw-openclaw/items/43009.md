## Summary

Keep open: current main and v2026.6.6 still have the exact-total-only TUI footer path, while the gateway/status layer now exposes freshness and estimated context-budget data that the TUI does not carry. The issue remains a narrow source-reproducible TUI display bug, not an implemented-on-main cleanup close.

## What This Changes

The issue asks for the TUI status bar to show real context-token usage instead of `?/200k` when usable session context data exists.

## Best Possible Solution

Carry `totalTokensFresh` and `contextBudgetStatus` through the TUI session metadata, prefer fresh exact totals, render estimated context usage with explicit estimated notation when exact usage is unavailable, and keep `?` only when usage is genuinely unknown or stale.

## Reproduction Assessment

Yes, at source level: current main can hold `contextTokens` without a fresh `totalTokens`, and the TUI footer deterministically renders that state as `tokens ?/<context>`. I did not run a live provider-backed TUI session during this read-only review.

## Solution Assessment

Yes, the best fix is freshness-aware TUI rendering, not the closed zero-default approach. The repair should reuse the gateway/status `contextBudgetStatus` and `totalTokensFresh` semantics so the footer does not invent an exact token count.
