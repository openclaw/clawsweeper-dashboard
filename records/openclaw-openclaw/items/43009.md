## Summary

Keep this open. Current main still has the reported TUI footer path: a known context window with a missing/unknown total reaches the formatter as `null` and renders `tokens ?/<ctx>`, while the related implementation PR is closed and unmerged. The closed PR also received a valid review concern that blindly coercing unknown totals to zero can underreport usage, so the remaining fix should preserve the unknown-vs-zero distinction.

## What This Changes

The issue requests that the TUI footer show actual context token usage instead of `tokens ?/<context>` when the Gateway/status path can report a concrete `0/<context> (0%)` value.

## Best Possible Solution

Keep unknown totals visibly unknown, but make the TUI receive or derive an explicit fresh zero only when zero usage is known. The most maintainable end state is aligned session/status semantics plus focused regression coverage for both `unknown/<ctx>` and `0/<ctx> (0%)` so the footer no longer disagrees with the Gateway/status view.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `formatTokens(null, 200000)` follows the inspected formatter branch and renders `tokens ?/200k`, and the footer passes that nullable state through from `sessionInfo`.

## Solution Assessment

No, the closed formatter-only PR is not the best fix as written because it converts every unknown total with a known context into zero. A safer repair should preserve the unknown state and only display zero when the session/list/status path has a concrete fresh zero or equivalent derivation.
