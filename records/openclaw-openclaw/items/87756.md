## Label Justifications

- `P1`: The issue reports a regression that hangs an agent-launched workflow and blocks a real chat-driven Lobster use case.
- `impact:crash-loop`: The affected behavior is a prompt-launched workflow hang where no response headers are returned from the nested invoke.
- `impact:session-state`: The likely root cause is same-session nested execution contending with per-session lane or lock ownership.

## Summary

Keep open: current main and latest release still document embedded Lobster nested `openclaw.invoke` as unreliable, and source still lacks a child-session, fail-fast, or in-process `llmAdapters` bridge for the prompt-launched hang path.

## What This Changes

The issue asks OpenClaw to stop prompt-launched Lobster workflows from hanging when a nested `/tools/invoke` calls `llm-task`, either by supporting the nested call or failing fast clearly.

## Best Possible Solution

Add a regression guard and choose one supported model: fail fast for same-session nested HTTP invokes, or implement the no-credential in-process Lobster LLM adapter bridge tracked in #90909.

## Reproduction Assessment

Yes from source, not live: a prompt-launched parent run can occupy `session:<main>`, while nested `/tools/invoke` without `sessionKey` resolves to the same main session and `llm-task` queues another embedded run there.

## Solution Assessment

Unclear until maintainer decision; fail-fast is the safest narrow mitigation, while the `ctx.llmAdapters` bridge is the likely long-term fix and credential env injection is the wrong security boundary.
