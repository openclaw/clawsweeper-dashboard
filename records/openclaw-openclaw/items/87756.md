## Summary

Keep open: current main and v2026.6.9 still leave the prompt-launched embedded Lobster nested `/tools/invoke` path unsupported for this hang case, and the related in-process LLM bridge work does not settle the same-session HTTP invoke fail-fast or routing behavior.

## What This Changes

The issue reports a regression where a prompt-launched embedded Lobster workflow hangs when a shell step calls nested `/tools/invoke` for `llm-task`, while the same workflow works when launched directly with curl.

## Best Possible Solution

Add regression coverage and a maintainer-approved behavior: at minimum fail fast for nested same-session `/tools/invoke`, while handling the long-term embedded LLM bridge through the related bridge issue or its active PR.

## Reproduction Assessment

Yes, from source rather than a live run: a prompt-launched parent embedded run can occupy the default session/global lanes while a nested `openclaw.invoke` posts to `/tools/invoke` and waits for nested tool execution before HTTP response headers are sent.

## Solution Assessment

Unclear until maintainers choose the supported model. A fail-fast guard is the narrowest bug mitigation, while child-session routing or an in-process adapter bridge is broader and needs product/security approval.
