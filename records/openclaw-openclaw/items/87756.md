## Summary

Keep open: current main and latest release still leave prompt-launched embedded Lobster nested `openclaw.invoke` unsupported, and the source still lacks a child-session route, fail-fast reentrant guard, or in-process `llmAdapters` bridge for the hang path.

## What This Changes

The issue asks OpenClaw to fix a regression where an agent-launched embedded Lobster workflow hangs when a nested `openclaw.invoke`/`/tools/invoke` call invokes `llm-task`, while direct curl-launched invocation works.

## Best Possible Solution

Add a regression guard and maintainer-approved behavior: at minimum fail fast for same-session nested `/tools/invoke`, and preferably route embedded Lobster LLM steps through a safe in-process `ctx.llmAdapters` bridge without exposing Gateway credentials.

## Reproduction Assessment

Yes from source, not live: omitted `sessionKey` resolves to the main session, embedded Lobster has no in-process LLM bridge, and nested `llm-task` re-enters serialized session execution while the parent prompt-launched run can still hold that session.

## Solution Assessment

Unclear pending maintainer decision: fail-fast same-session HTTP invoke is the narrowest bug mitigation, while the in-process adapter bridge is the cleaner long-term LLM workflow path but needs security-boundary approval.
