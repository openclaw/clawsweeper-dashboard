## Summary

Keep open: current main and v2026.6.8 still leave prompt-launched embedded Lobster nested `/tools/invoke` unsupported for this hang path, and the remaining fix needs a maintainer decision on fail-fast behavior versus a safe child/in-process bridge.

## What This Changes

The issue reports that a prompt-launched embedded Lobster workflow hangs when a shell step calls back into Gateway `/tools/invoke`, while direct curl-launched execution works.

## Best Possible Solution

Add regression coverage and a maintainer-approved behavior: at minimum fail fast for nested same-lane `/tools/invoke`, while handling the long-term embedded LLM bridge through https://github.com/openclaw/openclaw/issues/90909 or its replacement.

## Reproduction Assessment

Yes, from source rather than a live run: a prompt-launched parent embedded run can hold the default global lane while a nested `openclaw.invoke` posts to `/tools/invoke` and the nested `llm-task` path needs embedded-runner execution before the HTTP response can be sent.

## Solution Assessment

Unclear until maintainers choose the supported model. A fail-fast guard is the narrowest bug mitigation, while child-session routing or an in-process adapter bridge is broader and needs product/security approval.
