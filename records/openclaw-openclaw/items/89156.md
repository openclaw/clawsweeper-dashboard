## Summary

Keep this PR open: it targets a real source-backed Feishu message-delivery bug, but the current patch detaches bot identity recovery from the channel lifecycle and still has only mock/source-level proof for an external channel fix.

## What This Changes

The PR changes Feishu monitor startup to always schedule bot identity recovery when `botOpenId` is missing, passing no abort signal if the startup signal is already aborted, and adds a focused regression test.

## Best Possible Solution

Keep Feishu bot identity recovery lifecycle-owned: recover from the abort race without allowing stale account starts to keep probing or write identity after stop, reload, or credential changes.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` both gate `startBotIdentityRecovery` on `!abortSignal?.aborted` after an unknown identity, and the linked issue includes recent reporter logs. I did not run a live Feishu/Lark tenant reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the skipped scheduling branch, but passing `undefined` for an already-aborted signal makes the retry unowned; the safer fix is a cancellable or replaceable recovery path tied to the active account lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep identity recovery tied to the active lifecycle:** `extensions/feishu/src/monitor.account.ts:465`
  - body: Passing `undefined` here lets the retry worker outlive the stopped or reloaded account. `retryBotIdentityProbe` waits and later calls `fetchBotIdentityForMonitor` and `applyBotIdentityState` with the old account object, so repeated reloads or credential changes can leave stale probes writing `botOpenIds` and `botNames` after the active lifecycle has moved on. Please cancel, replace, or guard recovery through the current account lifecycle instead of detaching it.
  - confidence: 0.86
