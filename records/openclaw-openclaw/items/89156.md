## Label Justifications

- `P1`: The PR targets a Feishu channel workflow that can suppress message handling until restart and currently has a blocking lifecycle-safety concern.
- `merge-risk: 🚨 message-delivery`: A stale detached recovery can overwrite or clear bot identity used by Feishu mention-gated message routing.
- `merge-risk: 🚨 security-boundary`: The diff can continue Feishu credential-backed bot-info probes after the account lifecycle has already been aborted.

## Summary

Keep open. The PR targets a real, source-backed Feishu message-delivery bug, but the proposed fix detaches bot-identity recovery from an already-aborted lifecycle, which can let stale recovery work continue with old account credentials and later write global bot identity state after a replacement monitor exists.

## What This Changes

The PR changes Feishu account monitoring to start bot identity recovery whenever `botOpenId` is missing, passing no abort signal when the lifecycle signal is already aborted, and adds a focused regression test.

## Best Possible Solution

Keep the fix Feishu-local, but make identity recovery owned by the active account lifecycle or a generation-checked recovery owner so aborted probes cannot start unbounded retries or overwrite newer identity state.

## Reproduction Assessment

Yes, at source level. Current main can return an empty identity after an abort, write unknown bot identity, and skip `startBotIdentityRecovery` because the same abort signal is already aborted.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix needs lifecycle or generation ownership so recovery remains cancellable and stale aborted work cannot write identity for a replacement monitor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep recovery tied to the current lifecycle:** `extensions/feishu/src/monitor.account.ts:467`
  - body: Passing `undefined` here means an already-aborted monitor starts a retry worker with no cancellation owner. That worker waits and probes with the old Feishu account credentials, then calls the normal identity writer, so a stopped or replaced account can later overwrite `botOpenIds`/`botNames` for the active monitor. Keep recovery tied to a current lifecycle, or add generation ownership so stale aborted probes cannot write identity after abort.
  - confidence: 0.88
