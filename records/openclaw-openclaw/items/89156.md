## Label Justifications

- `P1`: The linked bug and this PR affect Feishu channel recovery and can leave real message handling gated after a restart race.
- `merge-risk: 🚨 message-delivery`: A stale or uncancelled identity recovery can overwrite Feishu bot identity state and keep mention-gated message handling broken or inconsistent.
- `merge-risk: 🚨 security-boundary`: The patch can keep using old Feishu account credentials for background probe calls after the owning channel lifecycle has been aborted.

## Summary

Keep open: the Feishu abort-race is source-reproducible, but this patch detaches identity recovery from lifecycle cancellation, so a retry can outlive channel stop/reload and mutate shared bot identity state from stale account data. External real behavior proof is also mock-only.

## What This Changes

The PR changes Feishu monitor startup to schedule bot identity recovery whenever `botOpenId` is missing, drops an already-aborted signal for that worker, and adds a mocked lifecycle regression test.

## Best Possible Solution

Keep bot identity recovery lifecycle-owned: recover after the abort race without allowing stale account starts to keep probing or overwrite identity state after stop/reload.

## Reproduction Assessment

Yes, source-reproducible: current main skips `startBotIdentityRecovery` when `botOpenId` is missing and the lifecycle signal is already aborted. I did not establish a live Feishu/Lark tenant reproduction.

## Solution Assessment

No. The PR fixes the skipped call, but dropping the abort signal makes recovery unowned; the safer fix is a cancellable or replaceable recovery worker tied to the active account lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep identity recovery tied to the channel lifecycle:** `extensions/feishu/src/monitor.account.ts:465`
  - body: Passing `undefined` here makes the retry worker unowned when this path is reached from an aborted start. `retryBotIdentityProbe` waits and later calls `applyBotIdentityState`, so after a stop or reload it can keep using the old account object and overwrite the shared bot identity maps, or stack several retries across repeated abort races. Please cancel, replace, or guard recovery through the current account lifecycle instead of dropping the signal.
