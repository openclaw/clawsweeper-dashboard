## Summary

Keep open. The PR targets a real Feishu bot identity recovery bug, but the patch detaches recovery from the channel lifecycle when the startup signal is already aborted, and the external proof is still source-level only.

## What This Changes

The PR changes Feishu monitor startup to start bot identity recovery whenever identity is missing, and adds a Vitest regression for the already-aborted startup signal path.

## Best Possible Solution

Keep the candidate PR open, but make identity recovery owned by an active account lifecycle or guard stale retries before merge, then add redacted Feishu/Lark restart proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 skip `startBotIdentityRecovery` when bot identity is missing and the lifecycle signal is already aborted. I did not run a live Feishu tenant restart.

## Solution Assessment

No. Scheduling recovery is the right Feishu-local area, but dropping the abort signal creates unowned retry work; a lifecycle-owned retry or stale-write guard would address the race without detached network calls.
