## Summary

Keep open. Current main and the latest release still forward generic Telegram callback fallback text as the raw callback payload, while the public Telegram docs promise the `callback_data: <value>` text form; the focused fix at https://github.com/openclaw/openclaw/pull/54962 remains open and conflicting, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

Telegram inline button callback_query events should reach the agent in the documented callback text form so agent instructions can trigger tool calls instead of fabricating confirmations.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962, or apply an equivalent targeted change that prefixes only generic Telegram callback fallback text, adds regression coverage, and includes real Telegram proof before closing this issue.

## Reproduction Assessment

Yes for the source-level routing contract: current main subscribes to `callback_query` but routes the generic fallback as raw `data`, while the docs require `callback_data: <value>`. I did not run a live Telegram webhook reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow generic fallback prefix while preserving structured internal callback paths. Directly forcing callback clicks into webhook calls would be the wrong layer because tool selection remains agent/model behavior.
