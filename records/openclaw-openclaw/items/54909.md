## Summary

Keep open: current main and v2026.5.22 still route generic Telegram callback fallback text as the raw payload, while the docs promise `callback_data: <value>`. The open linked pull request at https://github.com/openclaw/openclaw/pull/54962 owns the narrow fix path and should be reviewed, rebased, landed, or replaced before this issue closes.

## What This Changes

The issue reports Telegram inline button callbacks reaching the agent as raw callback payloads instead of documented `callback_data: <value>` text, causing tool-call instructions to be missed.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/54962 or an equivalent targeted patch that prefixes only generic callback fallback text, preserves structured callback handlers, adds prefix regression coverage, and includes real Telegram proof.

## Reproduction Assessment

Yes, source-level: current main subscribes to `callback_query` but sends generic fallback callback data as raw `data`, while the Telegram docs require `callback_data: <value>`. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prefix only generic fallback callback text and preserve native/approval/plugin/select callback routes. Directly invoking webhooks from Telegram callbacks would be the wrong layer because tool selection remains agent/model behavior.
