## Summary

Keep open: current main and the latest release still forward generic Telegram callback_query fallback text as the raw callback payload, while the Telegram docs promise `callback_data: <value>`. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/54962 is still open and conflicting, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to route Telegram inline button callback_query payloads to agents in the documented `callback_data: <value>` text form so agent instructions can trigger real tool calls.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/54962 or an equivalent targeted patch that prefixes only generic callback fallback text, preserves structured callback handlers, adds a regression assertion for the prefix, and includes real Telegram proof.

## Reproduction Assessment

Yes for source-level reproduction: current main subscribes to `callback_query` and routes the generic fallback as raw `data`, while the docs require `callback_data: <value>`. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prefix only the generic fallback message and preserve the structured callback paths. Forcing webhook calls directly would be the wrong layer because tool choice remains agent/model behavior.
