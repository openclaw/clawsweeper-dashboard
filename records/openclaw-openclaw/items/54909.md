## Summary

Keep open. Current main still routes generic Telegram callback_query fallback payloads to the agent as raw text, while the Telegram docs promise the `callback_data: <value>` form, and the linked implementation PR is still open and unmerged.

## What This Changes

Route Telegram inline button callback clicks to agents in the documented prefixed text form so instructions can recognize callback payloads and trigger tools.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962, or apply an equivalent current-main fix that prefixes only generic callback fallback text, preserves structured callback paths, adds regression coverage, and includes real Telegram proof before closing this issue.

## Reproduction Assessment

Yes for source-level reproduction: current main subscribes to `callback_query` but routes generic fallback data as raw text while the docs require `callback_data: <value>`. I did not run a live Telegram webhook reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow generic fallback prefix while preserving structured internal callback handling. Directly forcing callback clicks into webhook calls would be the wrong layer because tool selection remains agent/model behavior.
