## Summary

Keep open. Current main still sends generic Telegram callback fallback text to the agent as the raw payload, while the public Telegram docs promise the `callback_data: <value>` text form, and the linked implementation at https://github.com/openclaw/openclaw/pull/54962 is still open and unmerged.

## What This Changes

The issue asks for Telegram inline button callback payloads to reach the agent in the documented `callback_data: <value>` form so agent instructions can trigger real tool calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962 or an equivalent targeted change with regression coverage and real Telegram proof, then close this issue after the fix merges.

## Reproduction Assessment

Yes for the core routing contract: current main routes generic Telegram callback fallback text as raw `data` while the docs require `callback_data: <value>`. I did not live-reproduce the final webhook call in this read-only review, but the source-level text-shape mismatch is clear.

## Solution Assessment

Yes, the narrow maintainable fix is to prefix only generic callback fallback text while preserving the structured internal callback paths. Forcing callback clicks to call HTTP directly would be the wrong layer because tool selection remains agent/model behavior.
