## Summary

Keep open because current origin/main and the latest stable release still route generic Telegram callback fallback text as the raw payload while the docs promise `callback_data: <value>`. The linked PR at https://github.com/openclaw/openclaw/pull/54962 still references this issue with closing syntax, so the issue should stay open until that PR or an equivalent targeted fix lands.

## What This Changes

This issue asks OpenClaw to route Telegram inline button `callback_query` payloads to agents as documented callback text so configured HTTP tool workflows can run.

## Best Possible Solution

Land the linked PR or an equivalent replacement that forwards only generic Telegram callback fallback text as `callback_data: ${data}`, preserves native/internal callback paths, adds regression coverage, and then closes this issue after merge and proof.

## Reproduction Assessment

Yes for the core routing contract: origin/main and the latest release route generic callback fallback text as raw data while the docs promise `callback_data: <value>`. I did not live-reproduce the final HTTP tool call, but that is downstream of the source-level text-shape bug.

## Solution Assessment

Yes, the best fix is a narrow Telegram handler change plus regression coverage for the documented callback text shape. Forcing callback clicks to call HTTP directly would be the wrong layer because tool choice remains agent/model behavior.
