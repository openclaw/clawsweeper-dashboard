## Summary

Keep open. Current `main` still routes generic Telegram inline-button callback fallback data to the agent as the raw payload, while the Telegram docs promise the `callback_data: <value>` text shape; the linked PR #54962 is still the open implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that Telegram inline button callback data be routed to the agent in the documented `callback_data: <value>` form so configured tool instructions can trigger real webhook calls.

## Best Possible Solution

Review and land PR #54962, or apply an equivalent targeted change in `extensions/telegram/src/bot-handlers.runtime.ts` that forwards generic callback fallback text as `callback_data: ${data}` while preserving native/internal callback handling, then add a regression test for that documented contract.

## Reproduction Assessment

Yes for the routing contract: current source and tests show a callback payload is forwarded as raw `data`, which contradicts the documented `callback_data: <value>` form. The exact downstream hallucinated HTTP-tool behavior would still require live model/provider reproduction, but it is not needed to verify this core routing mismatch.

## Solution Assessment

Yes, the narrow maintainable fix is to change only the generic callback fallback text shape and test that contract. Forcing Telegram callbacks to invoke HTTP directly would be the wrong layer because tool choice remains agent/model behavior.
