## Summary

Keep open. Current main still forwards generic Telegram callback_query fallback data as the raw callback payload even though the Telegram docs promise the callback_data text form, and the linked implementation candidate is still open and unmerged.

## What This Changes

The issue asks for Telegram inline button callback_query payloads to reach the agent in the documented callback_data text form so callback instructions can trigger real tool calls instead of model-only confirmations.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962, or apply an equivalent focused fix that prefixes only generic Telegram callback fallback text, preserves structured callback paths, adds regression coverage, and includes real Telegram proof before closing this issue.

## Reproduction Assessment

Yes for source-level reproduction: current main subscribes to callback_query but routes generic fallback data as raw text while the docs require callback_data: <value>. I did not run a live Telegram webhook reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow generic fallback prefix while preserving structured internal callback handling. Directly forcing callback clicks into webhook calls would be the wrong layer because tool selection remains agent/model behavior.
