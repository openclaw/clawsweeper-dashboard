## Summary

Keep open because current main still forwards generic Telegram callback_query fallback data as the raw callback payload, while the Telegram docs promise the `callback_data: <value>` text form. PR #54962 is still the open implementation candidate and references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for Telegram inline button callback clicks to reach the agent in the documented `callback_data: <value>` form so configured instructions can trigger real HTTP tool calls.

## Best Possible Solution

Land PR #54962 or an equivalent targeted change that prefixes only generic Telegram callback fallback text, then add regression coverage for the documented callback contract and close the issue after merge.

## Reproduction Assessment

Yes for the routing contract: current source and tests show the generic fallback forwards raw callback data, contradicting the documented `callback_data: <value>` form. The downstream model hallucination and missing HTTP call were not live-reproduced, but they are not needed to verify the core text-shape bug.

## Solution Assessment

Yes, the narrow maintainable fix is to change only the generic callback fallback text shape and test that documented contract. Forcing Telegram callbacks to invoke HTTP directly would be the wrong layer because tool choice remains agent/model behavior.
