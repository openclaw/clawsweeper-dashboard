## Summary

Keep open: current main and the latest release still route generic Telegram callback fallback text as the raw payload, while the docs promise `callback_data: <value>`. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/54962 is still open and conflicting, so this issue should stay open until that fix or an equivalent patch lands.

## What This Changes

The issue requests Telegram inline button callback payloads be delivered to the agent in the documented prefixed text form so agent tool instructions can trigger real webhook calls.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/54962 or an equivalent targeted patch that prefixes only generic callback fallback text, adds a regression assertion for the prefix, and includes real Telegram proof before closing this issue.

## Reproduction Assessment

Yes for source-level reproduction: current main subscribes to `callback_query` but forwards generic callback fallback text as raw `data` while docs require `callback_data: <value>`. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

Yes: the maintainable fix is to prefix only the generic fallback message and preserve the structured callback paths called out in the Telegram maintainer note. Directly forcing webhook calls would be the wrong layer because tool choice remains agent/model behavior.
