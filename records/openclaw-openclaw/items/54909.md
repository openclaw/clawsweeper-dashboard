## Summary

Keep open: current main still forwards generic Telegram callback_query fallback data to the agent as the raw callback payload, while the Telegram docs promise the `callback_data: <value>` text form. The linked implementation PR is still open, conflicting, and unmerged, so this issue should stay open until that fix or an equivalent current-main patch lands.

## What This Changes

The issue reports Telegram inline button callback_query payloads reaching the agent in the wrong form, causing configured HTTP-tool workflows to be skipped while the model replies as if the action succeeded.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962, or apply an equivalent current-main fix that prefixes only generic callback fallback text, preserves structured callback paths, adds regression coverage, and includes real Telegram proof before closing this issue.

## Reproduction Assessment

Yes for source-level reproduction: current main subscribes to `callback_query` but routes generic fallback data as raw text while the docs require `callback_data: <value>`. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to prefix only the generic callback fallback while preserving structured internal callback handling. Directly forcing callback clicks into webhook calls would be the wrong layer because tool selection remains agent/model behavior.
