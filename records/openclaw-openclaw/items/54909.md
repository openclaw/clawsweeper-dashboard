## Summary

Keep open: current main and the latest release still route generic Telegram callback fallback text as the raw callback payload, while the Telegram docs require `callback_data: <value>`. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/54962 is still open and conflicting, so this issue remains necessary until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Telegram inline button `callback_query` events reach the agent in the documented `callback_data: <value>` form so HTTP-tool instructions can trigger real webhook actions.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/54962, or apply an equivalent targeted patch that prefixes only generic callback fallback text, preserves structured callbacks, adds a regression assertion, and includes real Telegram proof.

## Reproduction Assessment

Yes, source-level: current main subscribes to `callback_query` and routes generic fallback text as raw `data`, while the docs require `callback_data: <value>`. I did not run a live Telegram/webhook reproduction in this read-only review.

## Solution Assessment

Yes: prefixing only the generic fallback text is the narrow maintainable fix. Direct webhook bypass would be the wrong layer because tool choice remains agent/model behavior.
