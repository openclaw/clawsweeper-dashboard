## Label Justifications

- `P2`: This is a real memory vector-search regression for affected memorySearch provider configurations, but the blast radius is narrower than a core runtime outage.
- `impact:session-state`: The dirty index state blocks semantic recall from persisted memory index data and can leave agent memory behavior stale or degraded.
- `impact:auth-provider`: The failure centers on memory embedding provider selection across OpenAI, Google, and Gemini provider/auth routing.

## Summary

Keep open: current main and v2026.6.6 still contain the source-level path that can produce a blank expected memory index model, while the auto-to-OpenAI upgrade behavior remains an explicit compatibility/product decision rather than cleanup-safe closure.

## What This Changes

The issue reports an upgrade regression where memorySearch auto/Google provider state resolves to OpenAI or an empty expected model, leaving the memory index permanently dirty and vector search unavailable.

## Best Possible Solution

Define one canonical upgrade contract for legacy auto/Google/Gemini memory indexes, prevent blank expected models, and pair the runtime behavior with doctor/status recovery plus regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows unset/auto resolves to OpenAI, raw `google` is not the canonical memory embedding provider id, and an unregistered adapter can still produce an empty expected model for index identity comparison.

## Solution Assessment

Unclear. The narrow empty-string guard in https://github.com/openclaw/openclaw/pull/91691 is plausible for one symptom, but maintainers still need to choose the upgrade/provider compatibility contract before this issue is fully solved.
