## Summary

Keep open: current main fixes the empty expected-model symptom, but the central upgrade/provider contract remains unresolved because unset or legacy `auto` memorySearch still resolves to OpenAI rather than preserving the reporter's prior Gemini-backed behavior.

## What This Changes

The issue asks OpenClaw to preserve or safely migrate memorySearch provider selection across upgrades and prevent dirty vector indexes caused by empty expected model identity checks.

## Best Possible Solution

Define the supported upgrade contract for legacy auto/Google/Gemini memory indexes, pair it with doctor or runtime migration guidance, and keep the current empty-model guard as part of that end state.

## Reproduction Assessment

Yes. Source inspection shows unset and legacy `auto` memorySearch provider values still resolve to OpenAI, while the latest release still contains the blank expected-model identity path reported by users.

## Solution Assessment

Unclear. The merged empty-model guard is a good partial repair, but the full fix needs a maintainer decision on whether legacy auto should stay OpenAI, migrate to the prior provider, or surface a doctor/provider-id migration path.
