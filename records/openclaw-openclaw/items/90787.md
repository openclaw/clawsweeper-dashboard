## Summary

Keep open: current main and latest release still define unset or legacy `auto` memorySearch as OpenAI, while Google memory embeddings are still registered under the canonical `gemini` adapter. The merged status fix only covered adapter-default comparison, and the open empty-model PR explicitly leaves the auto-to-OpenAI upgrade decision out of scope.

## What This Changes

The issue reports an upgrade regression where memorySearch provider/default handling can leave Google/Gemini vector search permanently dirty with an empty expected model.

## Best Possible Solution

Maintainers should choose one upgrade contract: migrate or repair legacy auto/Google memorySearch state to canonical `gemini` with clear reindex guidance, or keep the OpenAI default and add diagnostics/recovery so vector search is not left permanently dirty.

## Reproduction Assessment

Yes, by source inspection: unset or legacy `auto` still resolves to OpenAI, `google` is not the memory embedding adapter id, and identity comparison can still use an empty fallback model before provider initialization. I did not run the full Rocky Linux upgrade flow.

## Solution Assessment

Unclear as an implementation direction: the empty expected-model guard is narrow, but the best fix still depends on a maintainer decision about preserving or migrating legacy auto/Google memory provider behavior.
