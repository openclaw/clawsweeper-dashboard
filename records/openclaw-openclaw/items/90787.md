## Label Justifications

- `P2`: The issue reports a real memory vector-search regression, but the blast radius is limited to users on affected memorySearch provider configurations.
- `impact:session-state`: The dirty index state can block semantic memory recall from persisted memory/session index data.
- `impact:auth-provider`: The failure centers on memory embedding provider selection and Google/Gemini versus OpenAI credential routing.

## Summary

Keep open: current main and v2026.6.1 still make unset or legacy `auto` memorySearch resolve to OpenAI, while Google memory embeddings are registered under the canonical `gemini` provider id; this leaves the reported upgrade behavior as a real maintainer policy/repair question, not cleanup-safe closure.

## What This Changes

The issue asks OpenClaw to preserve or safely migrate Google/Gemini memorySearch provider state across upgrade so existing vector indexes do not remain permanently dirty.

## Best Possible Solution

Maintainers should choose one canonical upgrade path: migrate/repair legacy auto or Google/Gemini memorySearch state to the supported `gemini` provider with rebuild proof, or keep the OpenAI default and add explicit diagnostics/recovery so users are not stuck with a permanently dirty index.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: unset/auto resolves to OpenAI, `google` is not the registered memory embedding provider id, and stale Gemini index metadata mismatches the expected provider/model. I did not run the full Rocky Linux upgrade flow.

## Solution Assessment

Unclear as an implementation direction: a code fix is likely straightforward, but the best policy must decide whether to preserve legacy auto/Gemini upgrade behavior or keep the documented OpenAI default and improve migration/diagnostics.
