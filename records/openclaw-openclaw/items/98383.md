## Summary

Keep open: current main and the latest release still only support exact `gateway.controlUi.allowedOrigins` entries or `*`, while the reported browser/WebView node-client case needs a new constrained loopback dynamic-port origin policy. The request is valid from source inspection, but it is a security-sensitive config/product decision rather than a strict bug-fix candidate.

## What This Changes

The issue requests constrained browser WebSocket origin pattern support for dynamic loopback ports used by browser-hosted node clients.

## Best Possible Solution

Add a maintainer-approved, constrained Gateway browser-origin policy that supports only safe loopback dynamic-port patterns, logs pattern matches distinctly, preserves exact allowlist behavior, keeps auth/pairing checks intact, and documents it as the safer alternative to wildcard origins.

## Reproduction Assessment

Yes from source inspection: a non-local socket peer with `Origin: http://127.0.0.1:<random-port>` and a Tailnet `Host` misses exact allowlist, same-host, and local-client loopback acceptance. I did not run a live Gateway/WebView repro in this read-only cleanup pass.

## Solution Assessment

Unclear until maintainer direction is set. A constrained policy belongs in the shared Gateway origin-check path, but maintainers need to decide whether the permanent config extends `gateway.controlUi` or introduces a role/client-scoped browser-origin policy.
