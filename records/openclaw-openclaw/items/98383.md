## Summary

Keep open: current main and the latest release still only support exact browser origins or the global `*` allowlist, while the requested dynamic loopback-port policy would add a new security-sensitive Gateway config surface that needs maintainer policy review before implementation.

## What This Changes

Request support for constrained loopback origin patterns so browser/WebView-hosted node clients with ephemeral local ports can connect without using `gateway.controlUi.allowedOrigins: ["*"]`.

## Best Possible Solution

Define a narrow Gateway browser-origin policy for dynamic loopback ports, then implement it in the origin-check/config/docs path with explicit allow and reject coverage for loopback-only dynamic ports, non-loopback wildcards, client/role scoping if accepted, and unchanged auth/pairing checks.

## Reproduction Assessment

Yes, source-level reproduction is clear: a remote WebView request with `Origin: http://127.0.0.1:<port>` and a non-loopback Gateway host will fail unless that exact origin is listed or `*` is used. I did not run a live phone/WebView setup.

## Solution Assessment

Unclear as proposed: constrained loopback patterns are a plausible direction, but the best fix needs maintainer-approved security semantics and config ownership before implementation.
