## Summary

Keep open. Current main has bundle-backed stdio LSP startup plus generic Gateway diagnostics/log viewing, but it still does not collect or expose LSP `textDocument/publishDiagnostics` events, server health, request latency, searchable LSP logs, or an editor diagnostics dashboard. The same-author LSP setup proposal #65251 remains open, so this should be scoped with that work rather than closed independently.

## What This Changes

Add real-time LSP diagnostics visualization and troubleshooting in the editor, including server status, diagnostic feeds, performance metrics, logs, smart fixes, and code-quality insights.

## Best Possible Solution

Define a generic LSP diagnostics contract first: capture server lifecycle, stderr/errors, request timing, and `publishDiagnostics` events, expose that through a documented plugin/gateway API, then decide whether the first UI belongs in Control UI, a plugin-served view, or an external editor integration.

## Reproduction Assessment

Not applicable; this is a feature proposal rather than a bug report. The current-main verification path is source/docs/UI inspection showing that LSP diagnostics are not captured or surfaced today.

## Solution Assessment

No, a monolithic dashboard is not the best first fix. The maintainable path is a narrow diagnostics/event contract with focused tests before adding UI, smart repair, history, or CI integrations.
