## Summary

Keep open. Current main still only starts configured bundle LSP servers and exposes hover/definition/references plus generic gateway logs; it does not collect or surface LSP publishDiagnostics events, server health, request latency, searchable LSP logs, or an editor diagnostics dashboard, and the related LSP setup proposal remains open.

## What This Changes

The issue requests real-time LSP diagnostics, server health, troubleshooting, and dashboard-style code quality feedback in the editor.

## Best Possible Solution

Define a focused LSP lifecycle/diagnostics event contract first, then expose it through the appropriate gateway/plugin/UI surface before adding dashboard, smart repair, history, or CI integrations.

## Reproduction Assessment

Not applicable; this is a feature proposal rather than a bug report. The current-main verification path is source, docs, UI, and issue-context inspection showing the requested LSP diagnostics surface is not implemented.

## Solution Assessment

No; the broad dashboard and smart-repair suite is not the narrowest maintainable first step. A safer path is a versioned LSP diagnostics/lifecycle event contract, followed by scoped UI or plugin-served views.
