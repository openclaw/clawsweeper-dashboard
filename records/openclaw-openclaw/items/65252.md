## Summary

Keep open. Current main has bundle-backed stdio LSP startup plus generic Gateway diagnostics/log viewing, but it still does not collect or expose LSP publishDiagnostics events, server health, request latency, searchable LSP logs, or an editor diagnostics dashboard; the paired LSP setup proposal #65251 also remains open.

## What This Changes

Add real-time LSP server status, diagnostic feeds, performance/log views, troubleshooting suggestions, and editor-facing code-quality feedback.

## Best Possible Solution

Define and test a generic LSP lifecycle/diagnostics event contract first, then expose it through the appropriate gateway/plugin/UI surface before adding dashboard, smart repair, history, or CI integrations.

## Reproduction Assessment

Not applicable; this is a feature proposal rather than a bug report. The current-main verification path is source/docs/UI inspection showing LSP notifications are only logged and generic Gateway logs are the only current UI diagnostic surface.

## Solution Assessment

No, a monolithic dashboard is not the narrowest maintainable first step. The better path is a focused LSP event/API contract with runtime tests, followed by a scoped UI or plugin-served experience.
