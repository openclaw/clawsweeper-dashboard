## Summary

Keep open. Current main has bundle-backed stdio LSP startup and generic Gateway logs/diagnostic stability, but it still does not capture or expose LSP publishDiagnostics, server health, request latency, searchable LSP-specific logs, or an editor diagnostics dashboard. The same-author LSP setup proposal #65251 is still open and overlaps the dashboard/status work, so this should be scoped with that item instead of closed independently.

## What This Changes

Request real-time LSP diagnostics, server-health monitoring, troubleshooting, code-quality insights, and an editor-facing diagnostics dashboard.

## Best Possible Solution

Define a generic LSP diagnostics contract first: capture server lifecycle, stderr/errors, request timing, and publishDiagnostics events, expose that through a documented plugin/gateway API, then decide whether the first UI belongs in Control UI, a plugin-served view, or an external editor integration.

## Reproduction Assessment

Not applicable; this is a feature proposal, not a bug report. The current-main verification path is source/docs/UI inspection showing that LSP diagnostics are not captured or surfaced today.

## Solution Assessment

No, the proposed all-in dashboard is not the best first implementation step. The maintainable path is a narrow diagnostics/event contract with tests before adding UI, smart repair, history, or CI integrations.
