## Summary

Keep open: current main and v2026.6.10 still silently invalidate the writer-owned session-store cache on stat mismatch, and the only directly related PR is closed unmerged. The request is a narrow session-state observability feature, but it still needs maintainer confirmation because it defines a new structured operator warning on a hot storage path.

## What This Changes

Request a structured warn log when the session-store writer cache is invalidated because the on-disk store changed outside the gateway writer queue.

## Best Possible Solution

Confirm the operator diagnostic contract, then add a fail-open structured signal in the writer-cache invalidation path with focused coverage for stat mismatch, missing file, and normal queue serialization.

## Reproduction Assessment

Yes, source-level: current main invalidates the writer cache on `mtimeMs` or `sizeBytes` mismatch and reloads without a structured warning. I did not run a filesystem repro because this was a read-only review, but the source path is direct.

## Solution Assessment

Unclear until maintainers confirm the product contract. The proposed warn-only shape is narrow, but the repository should decide whether this belongs as structured logging, an audit event, or no operator-facing signal.
