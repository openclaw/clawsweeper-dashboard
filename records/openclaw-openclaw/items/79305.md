## Summary

Keep open: current main still silently invalidates the writer cache on store mtime/size mismatch, but the requested warn payload is a new operator-facing observability contract that needs maintainer direction before implementation.

## What This Changes

Request a structured `sessions/store` warn log when an external on-disk session-store change invalidates the writer cache.

## Best Possible Solution

Approve or revise the log contract, then add a narrow structured warn in the cache mismatch path with regression coverage for external invalidation and in-process queued writes.

## Reproduction Assessment

Yes, by source inspection: a writer-cache stat mismatch invalidates and returns `null`, and the writer reloads from disk without logging. I did not run a live write scenario because this review is read-only.

## Solution Assessment

Unclear: the proposed structured warn on `sessions/store` is the narrowest visible implementation path, but its payload and frequency should be approved as a new observability contract first.
