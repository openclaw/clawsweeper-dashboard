## Summary

Keep open: current main still silently drops the writer cache on session-store stat mismatch, but adding a structured warn is a new operator-facing observability contract that needs maintainer direction.

## What This Changes

Add an operator-facing structured warn when the session-store writer cache is invalidated by on-disk store changes between writer cache loads.

## Best Possible Solution

Have a maintainer approve or adjust the sessions/store warn contract, then implement a narrow fail-open log with regression coverage for external invalidation and no false-positive in-process queued writes.

## Reproduction Assessment

Yes, by source inspection: the writer path borrows `takeMutableSessionStoreCache`, which invalidates and returns `null` on stat mismatch, then reloads from disk without logging. I did not run a live write scenario because this review is read-only.

## Solution Assessment

Unclear: the proposed structured warn is narrow and fits the existing subsystem logger metadata surface, but payload shape and warn-level frequency should be approved as an observability contract before implementation.
