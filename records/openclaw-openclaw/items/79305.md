## Summary

Keep open: current main and the latest release still silently reload the session store after writer-cache stat mismatches, but adding a warn-level operator signal is a new observability contract that needs maintainer direction before implementation.

## What This Changes

The issue requests a structured warning when the session writer cache detects that the on-disk session store changed between cache loads.

## Best Possible Solution

Approve or adjust the logging contract, then add a narrow structured warn in the writer-cache invalidation path with focused regression coverage for external invalidation and no false positives for queued in-process writes.

## Reproduction Assessment

Yes, by source inspection: a writer-cache stat mismatch invalidates and returns `null`, and the writer reloads from disk without logging on current main and `v2026.5.12`. I did not run a live write scenario because this review is read-only.

## Solution Assessment

Unclear: a structured `sessions/store` warn is the narrowest visible implementation path and fits the logger API, but the warn level, payload, and privacy/noise tradeoffs need maintainer approval first.
