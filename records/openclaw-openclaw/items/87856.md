## What This Changes

The PR changes model-call diagnostic byte accounting to count high-frequency stream deltas from incremental `delta` strings, omit growing `partial` snapshots from fallback accounting, add regression coverage, and update logging docs.

## Best Possible Solution

Land this as a documented diagnostic-semantics change after maintainer compatibility acceptance, while keeping https://github.com/openclaw/openclaw/issues/86599 open for the broader Windows local-model starvation investigation.

## Reproduction Assessment

Yes. Current main's `observeResponseChunk` JSON-serializes every stream chunk for byte accounting, so delta chunks carrying growing `partial` snapshots will repeatedly walk accumulated assistant content; the PR adds a focused regression test for that source path.

## Solution Assessment

Yes for the narrow diagnostic hot path: counting incremental delta bytes and omitting `partial` snapshots preserves bounded telemetry without repeatedly walking the accumulated assistant message. It is not the complete solution for the broader Windows local-model starvation issue.
