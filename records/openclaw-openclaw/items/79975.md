## Summary

Keep open: current main still has a source-visible gap where iMessage validates and forwards configured attachment paths, but the image tool reads local files only from default media roots plus the workspace, so configured Messages attachment roots are not honored.

## What This Changes

The issue requests that image and other media tools trust paths covered by `channels.imessage.attachmentRoots` when iMessage attachments are enabled.

## Best Possible Solution

Thread the current channel/account inbound attachment roots into the shared media-tool local-root resolver and apply it to image/media tools without re-enabling arbitrary parent-directory expansion.

## Reproduction Assessment

Yes, from source inspection: iMessage accepts and forwards configured attachment paths, while the image tool passes only default media roots plus workspace roots to `loadWebMedia`. A targeted test can reproduce current rejection with an image under a configured Messages attachment root.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing channel inbound-root contract for the current channel/account. The safer alternative to avoid is appending arbitrary media-source parent directories, which existing tests deliberately reject.
