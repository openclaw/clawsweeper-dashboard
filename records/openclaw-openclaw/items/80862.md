## Summary

Keep open: current main and the latest release still rotate split Telegram reasoning previews with `forceNewMessage()` before the old preview is stopped or cleared, matching the stale-message path in the report; the related fix PRs are closed unmerged.

## What This Changes

The issue asks Telegram `/reasoning stream` to keep one editable and deletable reasoning preview instead of accumulating stale preview messages.

## Best Possible Solution

Repair the Telegram split-reasoning draft lifecycle so each split stops, preserves, or clears the previous reasoning preview before rotation, then add regression coverage and Telegram behavior proof when feasible.

## Reproduction Assessment

Yes, as a high-confidence source-level path: after a reasoning segment ends, the next reasoning segment enters `onReasoningStream`, current main calls `forceNewMessage()` before `stop()` or `clear()`, and `forceNewMessage()` clears the retained preview id. I did not run a live Telegram DM.

## Solution Assessment

Yes, the narrow maintainable fix is to repair the existing Telegram draft-lane lifecycle using `rotateLaneForNewMessage` or an equivalent clear-before-rotate path, not to add a new setting or separate streaming mode.

Fix the Telegram `/reasoning stream` stale preview bug tracked at https://github.com/openclaw/openclaw/issues/80862. Current main rotates split reasoning previews by calling `reasoningLane.stream?.forceNewMessage()` after `onReasoningEnd` and before the old preview is stopped or cleared, which can drop the previous `streamMessageId` before cleanup.
