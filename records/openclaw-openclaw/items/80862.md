## What This Changes

The issue reports that Telegram `/reasoning stream` creates multiple stale reasoning preview messages instead of editing and deleting one live preview.

## Best Possible Solution

Repair Telegram reasoning split rotation so the previous reasoning preview is stopped or cleared before a new preview is forced, then add focused regression coverage and live Telegram proof before landing.

## Repair Work Prompt

Fix the Telegram `/reasoning stream` stale preview bug from https://github.com/openclaw/openclaw/issues/80862. On current main, `extensions/telegram/src/bot-message-dispatch.ts` sets `splitReasoningOnNextStream` on `onReasoningEnd`, then the next `onReasoningStream` calls `reasoningLane.stream?.forceNewMessage()` and resets lane state before the prior reasoning preview is stopped or cleared; `extensions/telegram/src/draft-stream.ts` then drops `streamMessageId`, so final cleanup cannot delete the old preview. Keep the fix scoped to Telegram draft lifecycle: prefer reusing `rotateLaneForNewMessage(reasoningLane)` if it preserves the intended answer-lane semantics, or otherwise clear/stop the prior reasoning preview before forcing the next message. Do not add config, a new send path, or change answer streaming, tool-progress drafts, long-preview page retention, or final answer delivery. Add a regression test in `extensions/telegram/src/bot-message-dispatch.test.ts` that simulates `onReasoningStream` A, `onReasoningEnd`, `onReasoningStream` B, and final delivery, asserting the prior reasoning stream is stopped or cleared before `forceNewMessage()`/second update and that final cleanup still handles the active preview. Touch `extensions/telegram/src/draft-stream.test.ts` only if the draft-stream contract itself changes. Add a single-line `CHANGELOG.md` fix entry because this is user-visible Telegram behavior.
