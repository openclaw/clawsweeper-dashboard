## Summary

Keep open: current main still has the source-level Telegram reasoning preview orphan path, and an open linked PR is the active fix path. This issue should remain open until that PR lands or is replaced.

## What This Changes

Telegram reasoning streaming should update and delete a single preview message instead of creating multiple stale `sendMessage` previews.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80910, or an equivalent narrow fix that stops or clears the active reasoning preview before rotating to the next split segment, with appropriate Telegram proof.

## Reproduction Assessment

Yes, at source level rather than from a live current-main run. The reporter's logs show repeated Telegram reasoning `sendMessage` calls without edit/delete on 2026.5.7, and current main still sets `splitReasoningOnNextStream` then calls `forceNewMessage()` directly before cleanup.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing stop-then-rotate preview lifecycle for split reasoning segments rather than adding a new send path or configuration surface.
