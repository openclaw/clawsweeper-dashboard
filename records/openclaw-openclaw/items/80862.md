## Summary

Keep open: current main still has the Telegram split-reasoning path that rotates the reasoning draft with `forceNewMessage()` without first stopping or clearing the active preview, so the prior preview id can be dropped before cleanup. The linked fix attempts are all closed unmerged, and the documented `/reasoning stream` behavior still expects one live reasoning preview that is deleted after final delivery.

## What This Changes

The issue reports that Telegram `/reasoning stream` creates multiple reasoning preview messages instead of editing and deleting a single live preview.

## Best Possible Solution

Repair the Telegram reasoning split path so the previous reasoning preview is stopped, preserved, or cleared before starting the next preview, then add a focused regression test and Telegram behavior proof.

## Reproduction Assessment

Yes, as a source-level reproduction: trigger a reasoning segment, `onReasoningEnd`, then a second reasoning segment, and current main rotates with `forceNewMessage()` before preserving or deleting the prior preview id. I did not run a live Telegram DM in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to repair the Telegram draft-lifecycle rotation for split reasoning, using the existing lane rotation contract or an equivalent clear-before-rotate path without adding config or changing unrelated delivery modes.
