## Summary

Keep open: current main still has the Telegram split-reasoning path that rotates the reasoning draft with `forceNewMessage()` without first preserving or clearing the prior preview id, while the docs promise the reasoning preview is deleted after final delivery. The three linked fix PRs are closed unmerged, and the live issue has no merged closing pull request.

## What This Changes

The issue reports that Telegram `/reasoning stream` leaves stale reasoning preview messages because split reasoning updates create new preview messages instead of editing and cleaning up one live preview.

## Best Possible Solution

Repair the Telegram split-reasoning draft lifecycle so the previous reasoning preview is stopped, preserved, or cleared before the next preview is forced, then add focused regression coverage and Telegram behavior proof.

## Reproduction Assessment

Yes, as a source-level reproduction: trigger a reasoning segment, `onReasoningEnd`, then a second reasoning segment; current main rotates the reasoning stream with `forceNewMessage()` before preserving or deleting the prior preview id. I did not run a live Telegram DM in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Telegram draft-lifecycle repair using the existing lane-rotation contract or an equivalent clear-before-rotate path. A new config option or separate send path would add unnecessary product surface.
