## Summary

Keep open: the report still maps to a current Telegram reasoning-preview lifecycle gap. Current main can rotate the reasoning draft after `onReasoningEnd` with `forceNewMessage()` and drop the previous preview id before final cleanup, while the docs require the reasoning preview to be deleted after final delivery.

## What This Changes

The issue asks Telegram `/reasoning stream` to keep one editable reasoning preview and delete it, instead of creating multiple stale `sendMessage` previews.

## Best Possible Solution

Repair reasoning-lane rotation so any existing reasoning preview is cleared or otherwise preserved before a new reasoning draft message starts, then add a regression test for multi-segment reasoning streams.

## Reproduction Assessment

Yes, by source inspection with high confidence: a reasoning stream, `onReasoningEnd`, and a later reasoning stream can rotate the draft with `forceNewMessage()` and lose the previous message id. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to keep this inside the existing Telegram draft-stream lifecycle by clearing or tracking the superseded reasoning preview before rotation, not by adding a new config option or separate send path.
