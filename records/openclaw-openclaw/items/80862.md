## Summary

Keep open: current main still has the Telegram split-reasoning path that can rotate a reasoning preview with `forceNewMessage()` before preserving or deleting the previous preview id, and the related fix PRs are closed unmerged.

## What This Changes

The issue reports that Telegram `/reasoning stream` can leave stale reasoning preview messages because split reasoning updates create new messages instead of editing and cleaning up one preview.

## Best Possible Solution

Repair the Telegram reasoning split lifecycle so the prior reasoning preview is stopped or cleared before forcing the next preview, then add regression coverage for the split `onReasoningEnd` to next `onReasoningStream` sequence.

## Reproduction Assessment

Yes, as a source-level reproduction: stream a reasoning segment, fire `onReasoningEnd`, then stream a second reasoning segment; current main rotates with `forceNewMessage()` before preserving or deleting the prior preview id. I did not run a live Telegram DM in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Telegram draft-lifecycle repair using the existing stop-before-rotate behavior or an equivalent clear-before-rotate path. A new config option or core API change is not needed.

Current main calls `reasoningLane.stream?.forceNewMessage()` directly when `splitReasoningOnNextStream` is set in `extensions/telegram/src/bot-message-dispatch.ts`, which can drop the previous reasoning preview message id before cleanup. Keep the change inside the Telegram plugin draft lifecycle: stop or clear the existing reasoning preview before forcing the next reasoning preview, preserve answer streaming/tool-progress behavior, and do not add a config option or core plugin API. Add focused regression coverage for `onReasoningStream` -> `onReasoningEnd` -> second `onReasoningStream`, proving the prior preview is stopped or cleared before the new preview is forced and final cleanup still handles the active preview.
