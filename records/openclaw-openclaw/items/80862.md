## Summary

Keep open: the report matches a plausible current Telegram reasoning-lane lifecycle gap. Current code edits and clears the active draft stream, but the reasoning stream can rotate to a new message after `onReasoningEnd` before the prior reasoning preview is cleared, which can orphan the earlier preview id.

## What This Changes

The issue reports that Telegram `/reasoning stream` sends multiple stale reasoning preview messages instead of editing and deleting one transient preview.

## Best Possible Solution

Keep reasoning streaming inside the existing Telegram draft lifecycle, preserving or clearing the previous preview receipt before starting any new reasoning draft, then cover the multi-segment reasoning case with a focused regression test.

## Reproduction Assessment

Yes from source inspection with medium confidence: a reasoning stream followed by `onReasoningEnd` and another reasoning stream can rotate the lane with `forceNewMessage()` before final cleanup, losing the prior message id. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to repair reasoning-lane rotation/cleanup around the existing draft stream rather than adding a separate send path or new configuration.
