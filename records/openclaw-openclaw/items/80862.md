## Summary

Keep open: current main still rotates split Telegram reasoning streams with `forceNewMessage()` before preserving or deleting the prior preview id, and all three linked fix PRs are closed without merging.

## What This Changes

The issue asks Telegram `/reasoning stream` to keep one live preview by editing and deleting it instead of accumulating stale preview messages.

## Best Possible Solution

Repair the Telegram reasoning-lane lifecycle so a split reasoning segment clears or archives the prior reasoning preview before starting the next preview, then add focused regression coverage and live Telegram proof.

## Reproduction Assessment

Yes, source-level with high confidence: on current main, `onReasoningEnd` sets `splitReasoningOnNextStream`, and the next `onReasoningStream` forces a new draft message before the prior preview id can be cleared. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in the Telegram reasoning preview lifecycle: clear or archive the active reasoning preview before `forceNewMessage()` starts the next split preview, without adding config or changing unrelated lanes.
