## What This Changes

The issue reports that Telegram `/reasoning stream` creates repeated reasoning preview messages instead of editing and deleting one transient preview.

## Best Possible Solution

Fix the Telegram reasoning preview lifecycle so a split reasoning segment deletes or preserves the previous preview id before a new preview is created, with regression coverage and real Telegram proof.

## Reproduction Assessment

Yes, from source rather than a live run: current main sets `splitReasoningOnNextStream` on reasoning end, then calls `forceNewMessage()` before cleanup can delete the old preview id, matching the reporter's repeated `sendMessage` with no `editMessageText` or `deleteMessage` logs.

## Solution Assessment

Yes, the best direction is a narrow Telegram draft lifecycle fix, not a config or product change. The fix must prove the prior reasoning preview id is deleted or retained before `forceNewMessage()` can reset it.

Fix https://github.com/openclaw/openclaw/issues/80862 as a narrow Telegram reasoning-preview lifecycle bug. Current main sets `splitReasoningOnNextStream` in `onReasoningEnd`, then the next `onReasoningStream` calls `reasoningLane.stream?.forceNewMessage()` and `resetDraftLaneState(reasoningLane)` before the previous preview id is deleted; `createTelegramDraftStream.forceNewMessage()` resets `streamMessageId`, so the old Telegram preview can be orphaned. Implement the fix in the Telegram plugin boundary, likely around `extensions/telegram/src/bot-message-dispatch.ts` and/or `extensions/telegram/src/draft-stream.ts`, so a previous reasoning preview is actually deleted or archived before a new reasoning preview is created. Do not add config, change `/reasoning` semantics, edit `CHANGELOG.md`, or broaden unrelated progress-mode behavior. Add focused regression coverage for `onReasoningStream` -> `onReasoningEnd` -> second `onReasoningStream`, proving the first preview is cleared/deleted or otherwise not orphaned and the next preview still streams correctly. Be careful: simply replacing the direct call with `rotateLaneForNewMessage(reasoningLane)` is not sufficient unless the real draft stream still preserves or deletes the previous `streamMessageId` before `forceNewMessage()` resets it.
