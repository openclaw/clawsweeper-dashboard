## Summary

Keep open: current main still has the Telegram split-reasoning rotation path that can drop the previous preview message id before cleanup, and the three linked fix PRs were closed unmerged. The issue is narrow and source-reproducible, so it is a good focused repair candidate with Telegram proof required before landing.

## What This Changes

The issue reports stale Telegram reasoning preview messages accumulating because split reasoning streams create new Telegram messages instead of editing and deleting a single preview.

## Best Possible Solution

Repair the reasoning split rotation so the prior reasoning preview is stopped or cleared before creating the next preview, then add a regression test and real Telegram proof for the visible cleanup behavior.

## Reproduction Assessment

Yes at source level: simulate `onReasoningStream`, `onReasoningEnd`, then another `onReasoningStream` and current main rotates with `forceNewMessage()` before preserving or deleting the prior preview id. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or mirror the existing stop-before-rotate draft-lane lifecycle for reasoning, not add a new send path or configuration.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80862. Current main's Telegram `onReasoningStream` split path calls `reasoningLane.stream?.forceNewMessage()` and resets the lane after `onReasoningEnd`, which can discard the prior reasoning preview id before final cleanup can call `deleteMessage`. Keep the fix inside the Telegram plugin draft lifecycle, preferably by reusing `rotateLaneForNewMessage(reasoningLane)` or an equivalent stop/clear-before-force-new sequence. Do not add config or reintroduce Telegram native drafts. Add focused regression coverage for two reasoning segments separated by `onReasoningEnd`, asserting the prior preview is stopped/cleared before the next `forceNewMessage()`/update and final cleanup still runs. Preserve answer streaming, tool-progress draft behavior, long-preview retention, and existing `onSupersededPreview` semantics. Add a changelog entry because the fix is user-visible. Validate with `node scripts/run-vitest.mjs extensions/telegram/src/bot-message-dispatch.test.ts extensions/telegram/src/draft-stream.test.ts`; if possible, also run a real Telegram proof lane showing one live reasoning preview is edited and cleaned up, with private chat ids/tokens redacted.
