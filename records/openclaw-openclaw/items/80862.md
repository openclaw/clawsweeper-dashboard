## Summary

Keep open: current main still has the Telegram split-reasoning path that calls `forceNewMessage()` before preserving or deleting the previous reasoning preview id, and the three linked fix PRs are closed unmerged.

## What This Changes

The issue reports that Telegram `/reasoning stream` creates multiple stale preview messages instead of updating and deleting one live preview.

## Best Possible Solution

Patch the split reasoning rotation to stop, preserve, or clear the current reasoning preview before forcing a new message, then add focused regression coverage and live Telegram proof.

## Reproduction Assessment

Yes, as a source-level reproduction: emit a reasoning segment, call `onReasoningEnd`, then emit another reasoning segment; current main rotates the reasoning stream with `forceNewMessage()` before preserving or deleting the prior preview id. I did not run a live Telegram DM in this read-only review.

## Solution Assessment

Yes: the narrow maintainable repair is to reuse the existing stop-before-rotate lane contract, or an equivalent clear-before-rotate path, and avoid adding config or a separate send path.
