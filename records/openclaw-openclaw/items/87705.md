## Summary

Keep open: current source does have a hardcoded 30s control-send wait, but this PR changes an existing spawn-run timeout setting into a synchronous control-send wait, lacks real slow-subagent proof, and has not confirmed that the linked memory-core failure is on this code path.

## What This Changes

The PR makes `sendControlledSubagentMessage` derive its `agent.wait` timeout from subagent run-timeout configuration or the run entry, keeps a 30s fallback, renames testing exports, and adds unit coverage.

## Best Possible Solution

Keep control-send waits separately bounded from spawn run budgets, then land the chosen timeout behavior only with maintainer-approved semantics and redacted real-run proof from the slow local-LLM scenario.

## Reproduction Assessment

Unclear as a full issue reproduction: current main source shows the hardcoded 30s `sendControlledSubagentMessage` wait, but the linked memory-core narrative path currently calls plugin `waitForRun` with 60s. No after-fix slow local-LLM run proves the reported failure path.

## Solution Assessment

No: reusing `runTimeoutSeconds` is not clearly the best fix because docs define it as the spawned run abort budget and examples use long values. A separate bounded control wait or proven caller-budget path is safer after maintainer confirmation.
