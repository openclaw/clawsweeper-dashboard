## Summary

Keep open: the fix direction is plausible, but this PR is not merge-ready because the added test calls an unimported helper name, the timeout resolver ignores the current run record's timeout, and the external PR only provides unit-test proof.

## What This Changes

The branch changes `sendControlledSubagentMessage` to compute its `agent.wait` timeout from `agents.defaults.subagents.runTimeoutSeconds` with a 5s buffer and adds unit coverage for configured and default cases.

## Best Possible Solution

Use a single subagent-control timeout resolver that preserves the 30s fallback, honors the active run/config budget, has focused tests, and includes real slow-run proof or redacted logs.

## Reproduction Assessment

Yes from source, not from a live run: current main hardcodes `timeoutMs: 30_000` in the affected control path while the surrounding subagent configuration already has a longer run-timeout surface.

## Solution Assessment

No, not yet: reusing the existing config is plausible, but the patch currently breaks test compilation and should pass through the active run timeout contract before maintainers decide whether the shared knob is the right UX.
