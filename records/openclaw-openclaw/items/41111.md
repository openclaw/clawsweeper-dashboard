## Summary

Keep open. Current main and the latest release still expose only `openclaw backup create` and `openclaw backup verify`, while the requested first-party restore command remains unmerged; the keepmyclaw ClawHub plugin is useful prior art and PR #59280 is the direct implementation candidate, but neither closes the native CLI gap yet.

## What This Changes

Add a first-party `openclaw backup restore <archive>` CLI command to restore local OpenClaw state from backup archives, complementing the existing create and verify commands.

## Best Possible Solution

Keep this issue open until OpenClaw ships a safe native `openclaw backup restore <archive>` workflow or maintainers explicitly consolidate it into a canonical backup/restore tracker. The implementation should stay in the backup CLI surface, reuse the archive validation contract, include restore registration and behavior tests, and update `docs/cli/backup.md` plus the generated CLI index.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inspect current main's backup command registration, command tests, backup docs, generated CLI index, and backup-related files; all show create/verify only and no native archive restore command.

## Solution Assessment

Yes. The best direction is a first-party restore command with conservative safety behavior because core already owns backup archive creation and verification; the keepmyclaw plugin is useful prior art, but it does not complete the built-in backup lifecycle.
