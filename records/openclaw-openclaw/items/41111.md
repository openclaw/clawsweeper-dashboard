## Summary

Keep open: current main and the latest release still expose only `backup create` and `backup verify`, while the native restore command remains unmerged. The linked keepmyclaw and OpenClaw Snapshot tools are useful workarounds, and PR #59280 is the direct implementation candidate, but that PR is still open, dirty, and has restore-safety findings that need maintainer handling.

## What This Changes

Add a native `openclaw backup restore <archive>` command for restoring from backup archives.

## Best Possible Solution

Ship a hardened core restore workflow in the existing backup CLI, with conservative archive validation, conflict handling, rollback behavior, docs, and changelog coverage while keeping external restore tools as workarounds.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The missing command is still high-confidence source-verifiable: current main and `v2026.5.2` register and document only `backup create` and `backup verify`.

## Solution Assessment

Yes, a first-party restore command is the right product direction because core already owns local backup creation and verification. The current open PR is not ready as-is until restore safety and security findings are resolved.
