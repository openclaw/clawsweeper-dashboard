## Summary

Keep open: current main and v2026.5.3 still expose only backup create/verify, while the requested native restore command remains unshipped. The keepmyclaw ClawHub plugin is useful prior art and a workaround, and open PR #59280 is the direct implementation candidate, but the first-party CLI gap remains.

## What This Changes

Add a native `openclaw backup restore <archive>` command to restore archives created by the existing backup CLI.

## Best Possible Solution

Ship a safe first-party `openclaw backup restore <archive>` workflow for core backup archives, or explicitly consolidate the remaining native restore work into the broader #13616 backup/restore tracker.

## Reproduction Assessment

Yes. A high-confidence source-level check on current main shows `registerBackupCommand`, its tests, the backup docs, and the generated CLI index all expose create/verify only, and the v2026.5.3 tag has the same surface.

## Solution Assessment

Yes, a native restore command is the right end state for the built-in backup lifecycle if it is guarded by conservative validation and overwrite behavior. The keepmyclaw plugin is useful prior art and a workaround, but it does not complete restoration for archives produced by `openclaw backup create`.
