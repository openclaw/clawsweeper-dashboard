## Summary

Keep open: current main still has privacy-gated WhatsApp `message_received` hooks and outbound reaction sending, but no WhatsApp reaction listener or public `reaction_received` / `reaction_removed` hook contract. The request is useful and specific, but it adds a public plugin/config/privacy surface that should be aligned with the broader WhatsApp reaction-read and listen-only work.

## What This Changes

Request to add WhatsApp `reaction_received` and optional `reaction_removed` plugin hooks for inbound reaction events without creating agent runs or outbound WhatsApp activity.

## Best Possible Solution

Define an additive, privacy-gated reaction event contract, implement it in the WhatsApp plugin-owned inbound monitor, and ship it with focused tests, docs, and live add/remove/reconnect proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through a plugin reaction hook today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned reaction hook is the plausible boundary given the socket-safety evidence, but final names, payload, opt-in config, removal behavior, and relationship to related reaction trackers need product direction.
