## Summary

Keep open: current `main` still has opt-in WhatsApp `message_received` hooks and outbound WhatsApp reaction sending, but no WhatsApp reaction add/remove listener or public reaction hook contract. The issue is useful, well-scoped product/API input, and it needs maintainer decisions on hook naming, payload, privacy opt-in, and overlap with the broader WhatsApp reaction trackers and open PRs.

## What This Changes

Add WhatsApp `reaction_received` and optionally `reaction_removed` plugin hooks so plugins can observe reaction add/remove events without opening a second Baileys socket or triggering agent or outbound activity.

## Best Possible Solution

Design an additive, privacy-gated reaction event contract, then implement it inside the WhatsApp-owned inbound monitor with focused tests, docs, and live add/remove/reconnect proof that no agent run or outbound WhatsApp activity occurs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug because this is a new feature/API request; source inspection on current `main` does show with high confidence that WhatsApp reactions are not exposed through a plugin reaction hook today.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public API contract. The in-process WhatsApp-owned hook boundary matches the socket-safety concern, but names, payload, privacy gating, and relation to the broader reaction tracker need product direction.
