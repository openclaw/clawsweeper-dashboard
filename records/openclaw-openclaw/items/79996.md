## Summary

Keep open: current main still has an opt-in WhatsApp message_received hook and outbound/status reaction support, but no WhatsApp reaction listener or public reaction hook contract. The related open reaction/query/listen-only issues and implementation PRs are useful input, but none has merged or supersedes this privacy-sensitive API request.

## What This Changes

Request to add WhatsApp reaction_received and reaction_removed plugin hooks for inbound reaction add/remove events with structured payloads and no agent-run or outbound side effects.

## Best Possible Solution

Define an additive, privacy-gated WhatsApp-owned reaction hook contract, emit add/remove events from the official inbound listener without creating agent runs or outbound activity, align it with cross-channel reaction hooks and listen-only mode, then ship tests, docs, and live proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through plugin hooks today: there is no reaction listener, typed hook name, handler map entry, or documented WhatsApp reaction hook opt-in.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the reporter's socket-safety evidence, but names, payload, privacy gating, and overlap with the broader reaction/listen-only trackers need product direction.
