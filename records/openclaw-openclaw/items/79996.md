## Summary

Keep open: current main still exposes outbound/status WhatsApp reactions and opt-in inbound `message_received` hooks, but it has no WhatsApp reaction listener, public reaction hook name, handler payload type, or reaction hook opt-in. The request is a useful feature/API expansion with concrete unsafe-workaround evidence and overlaps open WhatsApp reaction/query/listen-only work that needs maintainer API and privacy design.

## What This Changes

The issue requests WhatsApp `reaction_received` and `reaction_removed` plugin hooks so plugins can observe reaction add/remove events without unsafe Baileys socket workarounds.

## Best Possible Solution

Define an additive, privacy-gated WhatsApp-owned reaction hook contract, implement it in the official inbound listener, align it with cross-channel hooks and listen-only behavior, and ship tests, docs, and live proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through plugin hooks today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the reporter's socket-safety evidence, but names, payload, privacy gating, and relation to broader reaction trackers need product direction.
