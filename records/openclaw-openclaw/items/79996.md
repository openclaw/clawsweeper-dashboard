## Summary

Keep open: current main still has outbound/status WhatsApp reactions and an opt-in `message_received` hook, but no WhatsApp reaction listener or public reaction hook contract. The request is a useful feature/API expansion with concrete workaround evidence, and it overlaps active reaction-read, Discord hook, and listen-only work that needs maintainer API/privacy design.

## What This Changes

Add privacy-gated WhatsApp `reaction_received` and `reaction_removed` plugin hooks for reaction add/remove events without creating agent runs or outbound activity.

## Best Possible Solution

Define an additive, privacy-gated WhatsApp-owned reaction event hook contract, implement it in the official inbound listener, align it with cross-channel hooks and listen-only behavior, and ship tests, docs, and live proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through plugin hooks today: there is no reaction listener, typed hook name, handler map entry, or documented WhatsApp reaction hook opt-in.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the reporter's socket-safety evidence and the Baileys event contract, but names, payload, privacy gating, and relation to the broader reaction trackers need product direction.
