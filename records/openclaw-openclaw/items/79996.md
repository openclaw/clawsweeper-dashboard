## Summary

Keep open. Current main and the latest release still lack a public WhatsApp reaction add/remove plugin hook; WhatsApp only exposes opt-in `message_received` hooks and approval-specific reaction handling, while related open items cover broader querying, hooks-only mode, or sibling-channel parity rather than superseding this request.

## What This Changes

The issue requests first-class WhatsApp plugin hooks for reaction add/remove events without agent runs or outbound WhatsApp activity.

## Best Possible Solution

Approve a generic observation-only reaction hook contract, then implement WhatsApp add/remove emission inside the existing WhatsApp socket path with privacy gating, tests, and docs aligned with hooks-only mode.

## Reproduction Assessment

Yes, at source level: current main has no public reaction hook name, WhatsApp exposes only opt-in `message_received` hooks, and `reactionMessage` is consumed only for persisted approval targets. I did not run a live WhatsApp account reproduction in this read-only review.

## Solution Assessment

Yes directionally, but not at final contract detail. Emitting from the existing channel-owned socket path is the maintainable boundary, while maintainers still need to decide the stable plugin API name, payload, privacy gate, and relationship to hooks-only mode.
