## Summary

Keep open: current main still has opt-in WhatsApp `message_received` hooks and outbound WhatsApp reaction sending, but no WhatsApp reaction listener or public reaction hook contract. The request is useful and well scoped, but it expands the plugin/API surface and needs maintainer decisions before implementation.

## What This Changes

The issue requests WhatsApp `reaction_received` and `reaction_removed` plugin hooks for reaction add/remove events without agent runs or outbound activity.

## Best Possible Solution

Define an additive, privacy-gated reaction event contract, then implement it in the WhatsApp plugin-owned inbound monitor with tests, docs, no agent-run side effects, and live add/remove/reconnect proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through a plugin reaction hook today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned reaction hook is the plausible boundary given the report's socket-safety evidence, but final names, payload, opt-in config, and relation to the broader reaction trackers need product direction.
