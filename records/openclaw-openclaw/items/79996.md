- `impact:message-loss`: WhatsApp reaction feedback is currently received only for narrow approval handling and is otherwise absent from the plugin event stream needed by reaction-driven automations.

## Summary

Keep open. Current main and the latest release still lack a public WhatsApp reaction plugin hook: the WhatsApp monitor only wires normal inbound messages plus approval-specific reaction handling, and the public hook inventory has message hooks but no reaction add/remove event. The related open issues for broad WhatsApp reaction querying, Discord reaction hooks, and WhatsApp hooks-only mode are useful context, but none supersedes this narrower WhatsApp observation-hook request.

## What This Changes

Request to add observation-only WhatsApp reaction_received and reaction_removed plugin hooks for Baileys reaction events without creating agent runs or outbound WhatsApp activity.

## Best Possible Solution

Approve a generic observation-only reaction hook contract in the plugin SDK, then implement WhatsApp add/remove emission inside the existing WhatsApp monitor with opt-in privacy gating and tests/docs proving no agent run or outbound activity.

## Reproduction Assessment

Yes, at source level. Current main lacks a reaction hook in the public plugin hook names, does not wire a WhatsApp reaction event listener, and only handles reactionMessage for persisted approval targets; no live WhatsApp account was exercised in this read-only review.

## Solution Assessment

Yes directionally, but not at final contract detail. A channel-owned in-process observation hook is safer than plugins attaching to Baileys directly, while maintainers still need to align names, payload, and privacy gating with the broader reaction-hook and hooks-only work.
