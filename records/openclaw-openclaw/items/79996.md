## Summary

Keep open: current `main` still has opt-in WhatsApp `message_received` hooks and outbound WhatsApp reaction sending, but no WhatsApp reaction listener or public reaction hook contract. This is useful, well-scoped product input, but it expands the plugin/API surface and needs maintainer decisions on naming, payload, privacy opt-in, and overlap with the broader reaction-read tracker.

## What This Changes

Request to add WhatsApp plugin hooks for reaction add/remove events without creating agent runs or outbound WhatsApp activity.

## Best Possible Solution

Design an additive, privacy-gated reaction event contract, then implement it in the WhatsApp plugin-owned inbound monitor with tests, docs, no agent-run side effects, and live add/remove/reconnect proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current `main` gives high confidence that WhatsApp reactions are not exposed through a reaction hook today.

## Solution Assessment

Unclear until maintainers choose the API contract. An in-process WhatsApp-owned reaction hook is the right boundary suggested by the report’s socket-safety evidence, but final names, payload, opt-in config, and relation to the broader reaction tracker need product direction.
