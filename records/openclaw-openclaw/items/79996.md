## Summary

Keep open: current main still lacks a WhatsApp reaction listener, typed `reaction_received`/`reaction_removed` hook contract, and reaction hook opt-in. The linked workaround evidence and related PRs are useful, but this remains a new privacy-sensitive plugin API/config design that needs maintainer alignment with the broader WhatsApp reaction and listen-only work.

## What This Changes

The issue requests a WhatsApp `reaction_received` and optional `reaction_removed` plugin hook for inbound reaction add/remove events.

## Best Possible Solution

Define an additive, privacy-gated WhatsApp-owned reaction hook contract, implement it inside the official inbound listener, align it with listen-only and cross-channel reaction designs, then ship tests, docs, and live proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection gives high confidence that current main lacks a WhatsApp reaction listener, typed reaction hook name, handler payload, and documented opt-in.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the Baileys event contract and the reporter's socket-safety evidence, but names, payload, privacy gating, and relation to the broader reaction trackers need product direction.
