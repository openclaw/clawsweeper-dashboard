## Summary

Keep open: current main still has the opt-in WhatsApp `message_received` hook and outbound/status reaction support, but no WhatsApp reaction listener or public reaction hook contract. The related open PRs are useful implementation input, not merged fixes, and this needs maintainer API/privacy design before implementation.

## What This Changes

The issue asks for WhatsApp `reaction_received` and optional `reaction_removed` plugin hooks emitted from the official WhatsApp channel without creating agent runs or outbound activity.

## Best Possible Solution

Define an additive, privacy-gated WhatsApp-owned reaction hook contract, implement it in the official inbound listener, align it with cross-channel hooks and listen-only behavior, and ship tests, docs, and live proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection gives high confidence that WhatsApp reactions are not exposed through plugin hooks today: there is no reaction listener, typed hook name, handler map entry, or documented WhatsApp reaction hook opt-in.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the reporter's socket-safety evidence and the Baileys event contract, but names, payload, privacy gating, and relation to the broader reaction trackers need product direction.
