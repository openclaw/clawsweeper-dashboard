## Summary

Keep open. Current main still exposes WhatsApp `message_received` hooks and approval-specific WhatsApp reaction handling, but it does not expose ordinary WhatsApp reaction add/remove events through the public plugin hook API; the nearby open observe-only PR is partial and does not cover this reaction-hook request.

## What This Changes

The issue requests observation-only WhatsApp `reaction_received` and optional `reaction_removed` plugin hooks with privacy gating, no agent run, and no outbound WhatsApp activity.

## Best Possible Solution

Approve a generic observation-only reaction hook contract, then implement WhatsApp add/remove emission inside the existing channel monitor with privacy gating, focused tests, docs, and proof that the hook creates no agent run or outbound WhatsApp activity.

## Reproduction Assessment

Yes, at source level: current main lacks public reaction hook names and only emits WhatsApp `message_received` hooks, while `reactionMessage` handling is approval-specific. I did not run a live WhatsApp soak test in this read-only review.

## Solution Assessment

Yes directionally: a channel-owned in-process observation hook is safer than plugins attaching to Baileys directly. The exact public API shape and privacy gate need maintainer product/API approval, so this is not a strict automated bug fix.
