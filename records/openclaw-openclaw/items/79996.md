## Summary

Keep open: current main still has only opt-in WhatsApp `message_received` hooks and outbound WhatsApp reaction sending; it does not expose WhatsApp reaction add/remove events through the public plugin hook contract. The request is coherent and source-backed, but it adds a privacy-sensitive hook/config/API surface that should be aligned with the broader reaction and listen-only trackers before implementation.

## What This Changes

The issue requests WhatsApp `reaction_received` and `reaction_removed` plugin hooks so plugins can observe reaction add/remove events without agent runs or outbound activity.

## Best Possible Solution

Define a privacy-gated, additive reaction event contract and implement it inside the WhatsApp plugin-owned inbound monitor with focused tests, docs, and redacted live add/remove/reconnect proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through a plugin reaction hook today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the plausible boundary given the socket-safety evidence, but names, payload, removal semantics, opt-in config, and cross-channel alignment need product/API direction.
