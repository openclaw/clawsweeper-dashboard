## Summary

Keep open: current main has opt-in WhatsApp `message_received` hooks and outbound `react`, but it has no WhatsApp reaction add/remove listener or public plugin hook contract. The reporter's production workaround evidence is useful, but this is a new privacy-sensitive plugin/API feature that needs maintainer scope selection.

## What This Changes

Add WhatsApp plugin hooks for reaction add/remove events so trusted plugins can observe reactions without starting agent runs or opening another Baileys socket.

## Best Possible Solution

Design an additive reaction add/remove hook contract, implement it inside the WhatsApp plugin's owned inbound monitor behind explicit privacy opt-in, then add docs, focused tests, and live add/remove plus reconnect proof while coordinating with the broader reaction trackers.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current main gives high confidence that the requested reaction listener and hook surface do not exist.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned hook is the right boundary for avoiding unsafe second-socket workarounds, but the final names, payload, opt-in gate, and cross-channel relationship need product/API direction.
