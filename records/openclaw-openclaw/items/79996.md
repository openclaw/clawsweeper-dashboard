## Summary

Keep open: current main still exposes WhatsApp `message_received` hooks and outbound WhatsApp reaction sending, but it does not expose WhatsApp reaction add/remove events through the public plugin hook system. The request is useful and source-backed, but it is a new public hook/config surface with privacy and cross-channel API choices that need maintainer direction.

## What This Changes

Request to add WhatsApp `reaction_received` and optional `reaction_removed` plugin hooks for reaction add/remove events without agent runs or outbound WhatsApp activity.

## Best Possible Solution

Design an additive, privacy-gated reaction event contract, then implement it in the WhatsApp plugin-owned inbound monitor with focused tests, docs, no agent-run side effects, and live add/remove/reconnect proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through a reaction hook today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process WhatsApp-owned reaction hook is the right boundary for the socket-safety problem described in the report, but names, payload, opt-in config, and relation to broader reaction work need product direction.
