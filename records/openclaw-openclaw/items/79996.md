## Summary

Keep open: current main has outbound WhatsApp reaction controls and an opt-in WhatsApp message_received hook, but still lacks a WhatsApp reaction listener and public reaction hook contract. This is a concrete feature/API request with useful workaround evidence, and it needs maintainer design because it touches privacy-gated plugin payloads and overlaps active reaction/listen-only work.

## What This Changes

The issue requests additive WhatsApp reaction_received and reaction_removed plugin hooks so plugins can observe reactions without creating agent runs or outbound activity.

## Best Possible Solution

Define a privacy-gated WhatsApp-owned reaction event hook contract, align it with cross-channel hooks and listen-only behavior, then ship focused tests, docs, and live WhatsApp proof.

## Reproduction Assessment

Not applicable: this is a feature/API request, not broken documented behavior. Source inspection on current main gives high confidence that WhatsApp reactions are not exposed through plugin hooks today.

## Solution Assessment

Unclear until maintainers choose the public API contract. An in-process official WhatsApp hook is the plausible boundary, but names, payload, removal semantics, privacy gating, and relation to broader reaction/listen-only work need maintainer direction.
