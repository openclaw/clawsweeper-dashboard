## Summary

Keep open: the legacy JSON-state and over-limit recall-store paths are partly fixed by SQLite-backed memory-core state, but current main and v2026.6.8 still let memory-core log per-workspace dreaming failures while returning a handled cron result, so the remaining silent `NO_REPLY`/status-visibility problem is not proven gone.

## What This Changes

The issue asks OpenClaw to stop silently accepting a managed Memory Dreaming Promotion cron that produces `NO_REPLY`, leaves run tracking stale, and stops writing fresh Dreaming output.

## Best Possible Solution

Keep this issue as the tracker for the remaining managed Dreaming `NO_REPLY` and cron-status visibility gap until maintainers either land a status/diagnostic fix or collect fresh latest-release proof that the workflow no longer reproduces.

## Reproduction Assessment

Yes at source level for the remaining status-semantics concern: current memory-core logs per-workspace Dreaming failures but still returns a handled result to the cron-triggered run. I did not establish a live failing current-release deployment reproduction of the reporter's exact `jobs.json` symptom.

## Solution Assessment

Unclear. The SQLite migration is the right fix for the legacy JSON-state failures, but the remaining best fix depends on maintainer direction about how failed or no-op Dreaming sweeps should surface in cron status, doctor, logs, or diagnostics.
