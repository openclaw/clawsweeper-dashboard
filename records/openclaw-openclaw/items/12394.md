## Summary

Keep open: current main and v2026.6.6 still expose channel heartbeat as visibility booleans plus one resolved heartbeat delivery target, not a per-user WhatsApp OK-recipient contract. The request is useful, but it adds compatibility-sensitive config and needs maintainer product direction.

## What This Changes

The issue requests optional WhatsApp channel-level heartbeat `target` or `targets` fields so `HEARTBEAT_OK` messages can go only to admins while alert routing stays unchanged.

## Best Possible Solution

Define one canonical WhatsApp/channel heartbeat OK-recipient contract, validate recipients through existing account and allowlist policy, apply it only to HEARTBEAT_OK delivery, and preserve current alert routing plus omitted-config behavior.

## Reproduction Assessment

Yes, at source level: current main rejects target/targets in strict channel heartbeat config, resolves only visibility booleans, and sends OK acknowledgments to the single resolved delivery target. I did not run a live WhatsApp heartbeat.

## Solution Assessment

Unclear pending maintainer product direction. Splitting OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the final config shape and validation rules need maintainer approval.
