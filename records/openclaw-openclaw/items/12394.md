## Summary

Keep this issue open for a product decision. Current main removed the former WhatsApp `allowFrom` broadcast path, but it still resolves one destination for the entire heartbeat run, so an admin cannot receive OK acknowledgments separately while alert content follows its normal user route.

## What This Changes

Requests separate per-user recipients for channel heartbeat OK acknowledgments while preserving normal alert routing.

## Best Possible Solution

Define one generic OK-acknowledgment recipient contract at the heartbeat owner boundary, normalize and authorize it through existing channel outbound-target APIs, and prove that alert content continues using its existing delivery route.

## Reproduction Assessment

Yes, at source level. Current main resolves one `delivery.to` and uses it for both OK acknowledgments and non-OK alert payloads, while channel heartbeat configuration exposes only visibility controls; no live WhatsApp run was performed.

## Solution Assessment

No, the proposed WhatsApp-only `target` or `targets` field is not yet established as the best solution. The routing distinction is valid, but the maintainable implementation should use a generic heartbeat contract and existing channel target normalization rather than introduce parallel WhatsApp policy.
