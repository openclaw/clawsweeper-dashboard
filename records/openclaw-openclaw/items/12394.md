## Label Justifications

- `P3`: This is a useful but non-urgent WhatsApp heartbeat ergonomics feature with an existing workaround of disabling visible OK acknowledgments.
- `impact:message-loss`: The issue is about heartbeat OK messages being routed to every allowed WhatsApp recipient while preserving relevant alert delivery.

## Summary

Keep open: current main still has visibility-only channel heartbeat config and one shared heartbeat delivery target, so per-user WhatsApp HEARTBEAT_OK routing remains unimplemented and needs a maintainer product/config decision.

## What This Changes

Request to add optional channel-level WhatsApp heartbeat recipient targeting so visible HEARTBEAT_OK messages can go only to selected admin users while alerts keep their normal routing.

## Best Possible Solution

Define one additive OK-acknowledgment recipient contract, validate recipients through WhatsApp account and allowlist policy, apply it only to visible HEARTBEAT_OK delivery, and preserve existing alert/content routing plus omitted-config compatibility.

## Reproduction Assessment

Yes, at source level: current main rejects target/targets in strict channel heartbeat config, resolves only visibility booleans, and sends OK acknowledgments through the same resolved heartbeat delivery target. I did not run a live WhatsApp heartbeat.

## Solution Assessment

Unclear pending maintainer product direction. Splitting OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the final config shape and validation rules need owner approval.
