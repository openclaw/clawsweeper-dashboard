## Summary

Keep open: current main and v2026.6.5 still provide visibility controls but no separate recipient contract for routine heartbeat acknowledgments, and no canonical issue or PR owns the remaining product decision. The earlier maintainer-side Codex review correctly identified that existing agent-level delivery settings route the whole heartbeat rather than preserving alert recipients while selecting different OK recipients.

## What This Changes

Add account-aware recipient selection for WhatsApp heartbeat OK acknowledgments without changing the existing recipients for alerts and other heartbeat content.

## Best Possible Solution

Approve one additive, account-aware acknowledgment-recipient contract at the channel plugin boundary, keep alert and content delivery on the existing route, validate recipients against the selected WhatsApp account’s outbound policy, and document and test the precedence rules.

## Reproduction Assessment

Yes, at source level. Current main’s strict channel heartbeat schema cannot represent separate OK recipients, and the runner has one delivery route for the outbound heartbeat batch.

## Solution Assessment

Unclear. Separate acknowledgment routing is the right behavioral direction, but target versus targets, per-account precedence, authorization, and interaction with existing whole-heartbeat delivery settings require maintainer approval before implementation.
