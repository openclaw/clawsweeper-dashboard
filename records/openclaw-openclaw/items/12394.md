## Label Justifications

- `impact:message-loss`: The issue is about heartbeat OK messages being misrouted to every allowed WhatsApp recipient while preserving relevant alert delivery.

## Summary

Keep open: current main and v2026.6.10 still expose channel heartbeat as visibility-only plus one shared heartbeat delivery target, so OK-only per-user WhatsApp routing remains unimplemented and needs a maintainer product/config decision.

## What This Changes

Request to add optional channel-level WhatsApp heartbeat recipient targeting so HEARTBEAT_OK acknowledgments go only to configured admin target(s) while alert routing stays unchanged.

## Best Possible Solution

Define one additive OK-acknowledgment recipient contract, validate recipients through WhatsApp account and allowlist policy, apply it only to visible HEARTBEAT_OK delivery, and preserve existing alert/content routing plus omitted-config compatibility.

## Reproduction Assessment

Yes, at source level: current main rejects target/targets in strict channel heartbeat config, resolves only visibility booleans, and sends OK acknowledgments through the same resolved heartbeat delivery target. I did not run a live WhatsApp heartbeat.

## Solution Assessment

Unclear pending maintainer product direction. Splitting OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the final config shape and validation rules need owner approval.
