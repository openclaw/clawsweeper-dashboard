## Summary

Keep open. Current main still lacks channel-level WhatsApp heartbeat OK recipient routing: the channel heartbeat config only models visibility flags, WhatsApp inherits that shape, and heartbeat delivery still resolves one agent-level `target`/`to` before distinguishing OK acknowledgments from alert content. The two existing review comments on this issue already identified the same gap, and current main still matches that assessment.

## What This Changes

The issue requests optional `channels.whatsapp.heartbeat.target` or `targets` recipient routing so `HEARTBEAT_OK` acknowledgments can go only to admins while alert routing remains unchanged.

## Best Possible Solution

Design and implement a backward-compatible channel heartbeat recipient contract that validates recipients, applies only to OK acknowledgments, preserves alert/content delivery, and updates schema, types, docs, and regression coverage.

## Reproduction Assessment

Yes, by source inspection: `channels.whatsapp.heartbeat.target` or `targets` is outside the current strict channel heartbeat schema, and OK delivery never consults channel heartbeat recipients. This was not live-tested against WhatsApp because the review was read-only.

## Solution Assessment

Unclear until maintainers choose the public config shape and validation rules. The OK-only recipient-routing direction is plausible, but the existing agent-level `target`/`to` is not a full substitute because it routes the whole heartbeat delivery path.
