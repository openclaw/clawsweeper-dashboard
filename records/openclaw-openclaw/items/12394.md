## Summary

Keep this issue open for a product decision: current main sends heartbeat acknowledgments and alerts through one resolved delivery target, so the earlier WhatsApp allowlist broadcast path is gone, but admins still cannot receive OK acknowledgments separately while alerts retain their normal recipient routing.

## What This Changes

Requests per-user routing for channel-level heartbeat acknowledgments while preserving recipient-specific alert delivery.

## Best Possible Solution

Define one canonical, preferably channel-owned acknowledgment-recipient field with account-aware validation; apply it only to OK delivery while alerts continue using the existing resolved heartbeat destination, and cover omitted-field compatibility, account precedence, and invalid recipients.

## Reproduction Assessment

Yes for the remaining capability gap: current source clearly routes OK acknowledgments and alerts through the same `delivery.to` value and exposes no channel-level recipient field. No live WhatsApp run established that current main still broadcasts to every allowlisted user.

## Solution Assessment

Unclear pending maintainer policy: independent OK routing is the right behavior boundary, but adding both `target` and `targets` would create unnecessary parallel config; one canonical field and explicit account/authorization semantics would be safer.
