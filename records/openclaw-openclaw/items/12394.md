## Summary

Keep this issue open: current main and the latest release still route heartbeat acknowledgments through the heartbeat’s single resolved delivery target, while channel heartbeat configuration controls visibility only and cannot select separate OK-message recipients.

## What This Changes

Requests per-user recipient selection for channel-level heartbeat OK messages without changing alert routing.

## Best Possible Solution

Define one canonical, account-aware acknowledgment-recipient contract at the correct channel-owner boundary, validate recipients through existing WhatsApp target policy, apply it only to HEARTBEAT_OK delivery, and preserve current alert/content routing plus omitted-config behavior.

## Reproduction Assessment

Yes, at source level: current main's strict channel heartbeat schema has no recipient field, and the runner sends visible OK acknowledgments only through the heartbeat's already-resolved single delivery target. No live WhatsApp account was used, so this is source reproduction rather than an executed runtime reproduction.

## Solution Assessment

Unclear. Separating OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the proposed target versus targets syntax and its generic-versus-WhatsApp ownership require maintainer product judgment before implementation.
