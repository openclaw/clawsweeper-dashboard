## Summary

Keep open: current main and the latest release still expose channel heartbeat as visibility booleans plus one resolved heartbeat delivery target, so admin-only WhatsApp `HEARTBEAT_OK` routing remains missing and needs a product/config decision.

## What This Changes

The issue requests per-user WhatsApp/channel heartbeat OK-recipient routing so `HEARTBEAT_OK` can go only to selected admins while alert delivery keeps its existing recipient behavior.

## Best Possible Solution

Define one additive OK-acknowledgment recipient contract, validate recipients through WhatsApp account and allowlist policy, apply it only to visible `HEARTBEAT_OK` delivery, and preserve current alert/content routing plus omitted-config compatibility.

## Reproduction Assessment

Yes, at source level: current main rejects `target`/`targets` in strict channel heartbeat config, resolves only visibility booleans, and sends OK acknowledgments to the single resolved heartbeat delivery target. I did not run a live WhatsApp heartbeat.

## Solution Assessment

Unclear pending maintainer product direction. Splitting OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the final config shape and validation rules need owner approval.
