## Summary

Keep this issue open: current main and the latest release still lack a separate channel-level recipient contract for visible heartbeat OK acknowledgments, and implementing it would add compatibility-sensitive config/routing behavior that needs maintainer product direction.

## What This Changes

Add optional per-user WhatsApp/channel heartbeat OK routing so visible HEARTBEAT_OK acknowledgments can go only to selected admin recipients while alert/content routing stays unchanged.

## Best Possible Solution

Define one canonical, account-aware acknowledgment-recipient contract at the channel-owner boundary, validate recipients through existing WhatsApp target policy, apply it only to HEARTBEAT_OK delivery, and preserve current alert/content routing plus omitted-config behavior.

## Reproduction Assessment

Yes, at source level: current main's strict channel heartbeat schema has no recipient field, and the runner sends OK acknowledgments only through the already-resolved single heartbeat delivery target. I did not run a live WhatsApp heartbeat, so this is source-reproducible rather than an executed runtime repro.

## Solution Assessment

Unclear. Separating OK acknowledgment recipients from alert/content routing is the right behavioral boundary, but the target versus targets syntax and generic-versus-WhatsApp ownership need maintainer product judgment before implementation.
