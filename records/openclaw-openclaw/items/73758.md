## Summary

Keep open. Current main still has the reported short explicit tagged TTS failure path, and the issue is paired with open closing PR #73911, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Fix the TTS auto-synthesis path so explicit `[[audio_as_voice]]` plus `[[tts:text]]...[[/tts:text]]` output from the claude-cli route produces a synthesized voice reply instead of an empty skipped Telegram payload.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final tagged-mode payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to exempt explicit non-empty tagged `ttsText` from the short-text guard, not to change Telegram's empty-payload skip behavior or provider credential handling.
