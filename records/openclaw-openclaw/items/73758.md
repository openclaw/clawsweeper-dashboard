## Summary

Keep this open. Current main still has the source-visible short explicit tagged-TTS failure, and open PR #73911 already uses closing syntax for the central bug, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Tagged TTS directives emitted through the claude-cli reply path should synthesize and deliver a Telegram voice note instead of being stripped and skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow change that lets non-empty explicit `directives.ttsText` bypass the short-text guard while preserving untagged short auto-TTS suppression; track the SecretRef runtime-snapshot variant separately through #75420.

## Reproduction Assessment

Yes. A payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current-main path where hidden text is captured, visible text is stripped, synthesis returns early because the text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure. The narrowest maintainable fix is to exempt explicit non-empty tagged TTS text from the short-text guard, not to change Telegram's empty-payload skip behavior or provider credential handling.
