## Summary

Keep open. Current main still returns before synthesis for non-empty explicit tagged TTS text shorter than 10 characters, and an open closing PR, #73911, already targets this narrow bug, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that explicit tagged TTS output like `[[tts:text]]hello[[/tts:text]]` with `audioAsVoice` is parsed but not synthesized on the `claude-cli` route, leaving Telegram with an empty voice-only payload.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesize even below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` in tagged mode follows a current source path where hidden text is captured, visible text is stripped, the TTS runtime returns at the under-10-character guard before synthesis, and Telegram skips the empty voice-only payload.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to exempt non-empty explicit tagged speech text from the short-text guards while leaving provider config, parsing, Telegram delivery policy, and untagged short-text suppression unchanged.
