## Summary

Keep open. Current main still short-circuits explicit short tagged TTS before synthesis, and open PR #73911 already uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent change lands. The later SecretRef reproduction is useful context but appears to need separate confirmation after the narrow tagged-text fix.

## What This Changes

The issue reports that tagged TTS directives emitted through the claude-cli route are parsed but not synthesized, causing Telegram voice replies to be dropped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that lets explicit non-empty tagged ttsText bypass the short-text guards while preserving untagged short auto-TTS suppression; then verify the SecretRef runtime-snapshot report separately if it still reproduces.

## Reproduction Assessment

Yes. A payload like [[audio_as_voice]] [[tts:text]]hello[[/tts:text]] follows a source-visible path where hidden text is captured, the visible block is stripped, maybeApplyTtsToPayload returns before synthesis because the text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure. The narrow maintainable fix is to exempt explicit non-empty tagged TTS text from the short-text guard, not to change Telegram's empty-payload skip behavior or broaden provider credential handling in the same patch.
