## Summary

Keep open. Current main still preserves the reported failure path for short explicit tagged TTS text, and an open closing PR already targets the narrow fix, so this issue should not be closed before that PR lands or is replaced.

## What This Changes

The issue reports that explicit tagged TTS output such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` is parsed on the claude-cli backend but never synthesized, leaving Telegram with no voice delivery.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Is this the best way to solve the issue? Yes. The open PR direction is the narrowest maintainable fix because it changes explicit tagged-speech short-text semantics while leaving provider config, directive parsing, Telegram delivery, and untagged short-text suppression intact.
