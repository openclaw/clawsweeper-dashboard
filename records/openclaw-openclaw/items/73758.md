## Summary

Keep this open. Current main still contains the short explicit tagged-TTS failure for the original `hello` reproduction, and open PR #73911 already targets this issue with a narrow fix, so the issue should remain open until that PR or an equivalent change lands.

## What This Changes

The issue requests that `[[audio_as_voice]]` plus `[[tts:text]]...[[/tts:text]]` replies synthesize and deliver TTS voice output on the claude-cli-backed Telegram path instead of being silently dropped.

## Best Possible Solution

Land #73911 or an equivalent narrow change that treats non-empty explicit `directives.ttsText` as an opt-in to synthesize below the generic short-text threshold while preserving untagged short auto-TTS suppression; then separately verify whether the SecretRef runtime-snapshot report still reproduces on current main.

## Reproduction Assessment

Yes. A payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden text is captured, visible text is stripped, `maybeApplyTtsToPayload` returns before synthesis because the text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure. The narrowest maintainable fix is to exempt explicit non-empty tagged TTS text from the short-text guard, not to change Telegram's empty-payload skip behavior or provider credential handling.
