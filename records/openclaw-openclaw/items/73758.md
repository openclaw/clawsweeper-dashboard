## Summary

Keep open. Current main still short-circuits explicit tagged TTS text under 10 characters before synthesis, and the open closing PR #73911 is the active narrow fix candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that explicit `[[tts:text]]...[[/tts:text]]` voice directives from the claude-cli route are parsed but not synthesized, causing Telegram voice replies to be skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and covering `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Yes. A payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes. The open PR direction is the narrowest maintainable fix because it changes explicit tagged-speech short-text semantics while leaving provider config, directive parsing, Telegram delivery, and untagged short-text suppression intact.
