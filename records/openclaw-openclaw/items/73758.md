## Summary

Keep open. Current main still preserves the reported short tagged-TTS failure path, and open PR #73911 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix tagged TTS directives emitted through the claude-cli reply path so explicit hidden speech text is synthesized and delivered as voice instead of being stripped and skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding regression coverage for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Is this the best way to solve the issue? Yes. The open PR direction is the narrowest maintainable fix because it changes explicit tagged-speech short-text semantics while leaving provider config, directive parsing, Telegram delivery, and untagged short-text suppression intact.
