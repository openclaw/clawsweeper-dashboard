## What This Changes

The issue reports that tagged TTS directives emitted through the claude-cli route are parsed but not synthesized, leaving Telegram with an empty voice-only reply and no delivered voice message.

## Best Possible Solution

Land #73911 or an equivalent narrow change that treats non-empty explicit `directives.ttsText` as an opt-in below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`. Keep SecretRef-only runtime snapshot behavior tied to #75420 unless the TTS path still fails on current main after the short-tag fix.

## Reproduction Assessment

Yes. A tagged-mode final payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, synthesis returns early because the spoken text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure. The narrow maintainable fix is to exempt explicit non-empty tagged `ttsText` from the short-text guard, not to change Telegram's empty-payload skip behavior or provider credential handling.
