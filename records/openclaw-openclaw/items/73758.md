## Summary

Keep open. Current main still preserves the reported failure path for short explicit tagged TTS text, and open PR #73911 already targets the narrow runtime/test fix with closing syntax for this issue, so cleanup should wait for that PR to land or be replaced.

## What This Changes

The issue requests that explicitly tagged TTS output such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` synthesize and deliver voice audio on the claude-cli auto-reply path instead of being stripped and skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow change that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while keeping untagged short auto-TTS suppression and adding a regression case for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. On current main, a final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` captures hidden TTS text, strips visible text, returns before synthesis because the spoken text is under 10 characters, and then Telegram skips the resulting empty audioAsVoice-only reply.

## Solution Assessment

Is this the best way to solve the issue? Yes. The current open PR direction is the narrowest maintainable fix because it changes only explicit tagged short-text synthesis semantics while leaving directive parsing, provider configuration, Telegram delivery, and untagged short-reply suppression intact.
