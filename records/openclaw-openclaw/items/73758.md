## Summary

Current `main` still has the short explicit tagged-TTS failure: the parser stores hidden `[[tts:text]]...[[/tts:text]]` text and strips it from visible output, but `maybeApplyTtsToPayload` returns before synthesis when the spoken text is under 10 characters. Because open PR #73911 uses closing syntax for this issue, the issue should remain open until that PR or an equivalent fix lands; the newer SecretRef comment looks relevant but belongs to the same TTS/secrets triage rather than making this safe to close.

## What This Changes

The issue asks OpenClaw to synthesize and deliver voice output for parsed TTS directives such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` on the claude-cli/Telegram path instead of silently dropping the reply.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`. Keep any remaining SecretRef runtime-snapshot failure tied to the related secrets issue unless the same TTS path still fails on current main after the short-tag fix.

## Reproduction Assessment

Yes. A final tagged-mode payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure. The narrow maintainable fix is to exempt explicit non-empty tagged `ttsText` from the short-text guard, not to change Telegram's empty-payload skip behavior or provider credential handling.
