## Summary

Keep open. Current main still has the original short explicit tagged-TTS failure: hidden `[[tts:text]]hello[[/tts:text]]` is captured and stripped from visible text, then synthesis returns before `textToSpeech` because the spoken text is under 10 characters. The open closing PR #73911 is the right review path for that root cause, while the newer SecretRef-backed TTS report appears related to the broader runtime-snapshot issue tracked by #75420 rather than evidence that this issue is fixed.

## What This Changes

The issue reports that inline tagged TTS directives from claude-cli output are parsed but not synthesized, causing Telegram voice replies to be skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`. Keep any remaining SecretRef runtime-snapshot failure tied to #75420 unless the same TTS path still fails on current main after the short-tag fix.

## Reproduction Assessment

Yes. A final tagged-mode payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, synthesis returns because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes for the original reported failure. The narrow maintainable fix is to exempt explicit non-empty tagged `ttsText` from the short-text guard, not to change Telegram's empty-payload skip behavior or broaden provider credential handling in the same patch.
