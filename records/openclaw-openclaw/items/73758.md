## Summary

Keep this open. Current main still has the source-visible short explicit tagged-TTS skip, and open PR #73911 already targets the central bug with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `[[audio_as_voice]]` plus `[[tts:text]]...[[/tts:text]]` output is parsed but not synthesized before Telegram delivery on `claude-cli`, causing the expected voice reply to disappear.

## Best Possible Solution

Land #73911 or an equivalent narrow speech-core fix for explicit short tagged TTS, then split or route the SecretRef runtime-snapshot reproduction to the secrets/runtime follow-up if it still reproduces.

## Reproduction Assessment

Yes. The original reproduction is visible in current source with `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`: the hidden text is captured, the short-text guard returns before synthesis, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes for the original short-tagged failure: exempting non-empty explicit `directives.ttsText` from the short auto-TTS suppression is the narrowest maintainable fix while preserving untagged short-reply noise control. The later SecretRef evidence likely needs a separate runtime-secret fix path if it remains reproducible.
