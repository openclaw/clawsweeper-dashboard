## Summary

Keep open. Current main still short-circuits explicit hidden `[[tts:text]]hello[[/tts:text]]` content under the 10-character TTS guard before synthesis, and Telegram still skips the resulting empty `audioAsVoice` payload. The issue is also paired with open closing PR #73911, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to synthesize explicitly tagged TTS text, including short hidden `[[tts:text]]...[[/tts:text]]` content with `[[audio_as_voice]]`, when replies are routed through the claude-cli backend.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in below the generic short-text threshold, preserves untagged short auto-TTS suppression, and adds a regression case for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`. Keep adjacent message-tool work such as #74955 separate unless maintainers decide to combine follow-up validation.

## Reproduction Assessment

Yes. A final tagged-mode payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current-main path where hidden text is captured, visible text is stripped, TTS returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes. The narrow maintainable solution is to exempt explicit non-empty tagged `ttsText` from the short-text guard while leaving Telegram's empty-payload skip and untagged short-reply suppression intact.
