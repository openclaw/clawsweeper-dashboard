## Summary

Keep this open. Current main still has the short explicit tagged-TTS failure path, and open PR #73911 already uses closing syntax for the central bug, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that tagged TTS directives such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` can be parsed but not synthesized, leaving Telegram with no voice message.

## Best Possible Solution

Land #73911 or an equivalent narrow fix for explicit short tagged TTS, then track the SecretRef runtime-snapshot behavior through #75420 if it still reproduces after that fix.

## Reproduction Assessment

Yes. A payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden TTS text is captured, visible text is stripped, the short-text guard returns before synthesis, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes for the central short-tagged failure. The narrowest maintainable fix is to exempt explicit non-empty `directives.ttsText` from the short-text guard while preserving untagged short auto-TTS suppression.
