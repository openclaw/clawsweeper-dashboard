Keep open: the PR is the active fix path for the transcript append redaction gap, but the latest head is still conflicting, paused for human review, and has a concrete config-off fallback bug plus a changelog credit issue.

## What This Changes

The PR centralizes transcript message redaction in `appendSessionTranscriptMessage`, returns the redacted appended message to inline update emitters, updates Telegram/gateway/session callers, adds redaction tests, and refreshes the SDK API baseline and changelog.

## Best Possible Solution

Land a maintainer-reviewed central append-layer redaction fix that returns and emits the redacted message, honors logging redaction settings for every payload shape, and preserves human source credit.

## Reproduction Assessment

Yes. Source inspection on current main shows the JSONL append helper persists a redacted payload but inline session, gateway, and Telegram update paths still emit the original secret-bearing message; the PR finding is also source-reproducible by passing a non-role payload with `logging.redactSensitive: "off"` to the exported append helper.

## Solution Assessment

No, not as-is. Centralizing redaction at the append sink is the right maintainable direction, but the latest head should first make the non-role fallback config-aware, fix the source contributor credit, resolve conflicts, and get maintainer acceptance for the SDK surface change.
