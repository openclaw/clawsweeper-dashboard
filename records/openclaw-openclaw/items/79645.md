## Summary

Keep open: current main still has the inline transcript redaction gap, and this bot repair is useful, but the exact PR head still has a security-sensitive redaction bypass for non-role append payloads and automation has been explicitly paused for human review.

## What This Changes

This PR centralizes transcript redaction in `appendSessionTranscriptMessage`, returns the sanitized message to gateway and Telegram transcript emitters, updates tests and the SDK API baseline, and adds a changelog entry.

## Best Possible Solution

Land an append-layer redaction fix that returns and emits sanitized messages while preserving broad redaction for every append payload shape, with the SDK contract change accepted and documented in the PR summary.

## Reproduction Assessment

Yes, by source inspection: current main redacts the persisted JSONL entry but gateway and Telegram inline transcript updates still use the original payload. The remaining PR-head regression is also source-reproducible at `appendSessionTranscriptMessage` because non-role payloads bypass redaction.

## Solution Assessment

No, not yet. Centralizing redaction at the append sink is the right direction, but the implementation should either keep redacting unknown payload shapes or explicitly narrow the exported SDK contract so unsupported shapes cannot reach this path.

## Review Findings

Overall correctness: patch is incorrect

## Security Review

Status: needs_attention

Summary: The patch improves inline transcript redaction but introduces a concrete redaction bypass for non-role payloads passed to the exported append helper.

Concerns:

- **[medium] Non-role append payloads bypass transcript redaction:** `src/config/sessions/transcript-append.ts:297`
  - body: The changed append sink skips redaction unless the payload has a string `role`, whereas current main redacts the whole message value before writing it. This can leak secrets from SDK or internal callers that pass raw strings or structured payloads without `role`.
  - confidence: 0.86
