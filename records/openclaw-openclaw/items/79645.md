The PR extracts transcript redaction into a shared helper, redacts appended session transcript messages before JSONL writes, returns the redacted message to gateway and Telegram inline transcript emitters, adds focused tests, and refreshes generated API baseline/changelog data.

## Best Possible Solution

Land a refreshed human-reviewed branch that keeps centralized append-layer redaction and redacted inline transcript events, preserves built-in redaction when custom patterns are configured or explicitly updates/tests a different contract, and clears exact-head checks.

## Reproduction Assessment

Yes, by source inspection. Current main writes a redacted disk entry but gateway and Telegram inline transcript events still use the original message object; the PR head changes those emitters to use the appended redacted message, and the source PR supplied live JSONL proof.

## Solution Assessment

No, not as this exact merge candidate. The append-layer redaction contract is the right direction, but the branch must preserve default redaction with custom patterns and clear the stale test-type failure before merge.
