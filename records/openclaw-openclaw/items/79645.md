## Summary

Keep open: the current PR head is still a useful security fix path, but it narrows the current append-sink redaction coverage and a member has explicitly paused automation for human review.

## What This Changes

The PR centralizes session transcript message redaction at the JSONL append path, threads `OpenClawConfig` through transcript injection/update paths, adds regression coverage, and updates the changelog.

## Best Possible Solution

Keep the append-layer redaction centralization, but make it preserve recursive structured coverage for every plain message/content field by reusing or generalizing the current sanitizer with config-aware options, then add a regression for arbitrary gateway content-block fields.

## Reproduction Assessment

Yes, source-level: pass a gateway-injected assistant message with a custom content block containing a secret in `source.url`, `data`, or another non-selected field. Current main recurses through the whole message via `redactSecrets`, while this PR leaves those fields unchanged.

## Solution Assessment

No, not as-is. Centralizing transcript redaction at the append sink is the right direction, but the helper must preserve current main's recursive structured coverage before it is safer than the existing sink.
