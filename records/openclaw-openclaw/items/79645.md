## Summary

Keep open: this is a useful security repair, but the current head still narrows transcript redaction compared with current main and a member explicitly paused ClawSweeper automation for human review.

## What This Changes

This PR moves transcript redaction into appendSessionTranscriptMessage, threads OpenClawConfig through transcript and gateway writes, adds redaction tests, and carries forward the source contribution from https://github.com/openclaw/openclaw/pull/73563.

## Best Possible Solution

Keep the centralized append-layer redaction, but reuse or generalize the current recursive tool-payload redaction so configured custom patterns and built-in defaults apply to every message/content field, then correct the changelog credit before re-review.

## Reproduction Assessment

Yes, source-reproducible. At PR head, a transcript message with logging.redactPatterns set to a custom-only pattern can leave an sk-token unmasked, and a gateway-injected custom content block with source/url/data secrets bypasses the helper where current main redacts the whole message recursively.

## Solution Assessment

No, not as-is. Centralizing redaction at appendSessionTranscriptMessage is the right direction, but the helper must preserve current main's recursive coverage and default-pattern merge before replacing the existing sink.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

  - body: `redactTranscriptOptions` passes `logging.redactPatterns` through as the complete pattern set. Once a user configures a custom pattern, `redactSensitiveText` no longer includes the built-in token patterns, so an `sk-...` token in a transcript can be written unmasked; current tool-payload redaction merges custom patterns with defaults.

Current main redacts the whole message recursively before persistence, so this narrows disk redaction coverage.
