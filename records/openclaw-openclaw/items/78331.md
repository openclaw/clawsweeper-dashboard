## Summary

Keep open: the patch targets a real stream leak, but it currently has a blocking ordering bug at content-to-tool/reasoning boundaries, and the external PR proof is unit-fixture/check output rather than after-fix real behavior proof.

## What This Changes

The PR adds a DeepSeek-scoped streaming text filter in `processOpenAICompletionsStream` plus regression tests to suppress DSML `tool_use_error` blocks in OpenAI-compatible completions streams.

## Best Possible Solution

Keep the provider-scoped stream-filter approach, but repair boundary flushing, cover the content-prefix-followed-by-tool/reasoning case, and require redacted real OpenClaw/DeepSeek stream proof before merge.

## Reproduction Assessment

Yes, at source level: current main appends every `choice.delta.content` into assistant text, so a DeepSeek content delta containing the DSML block leaks. I did not execute a live DeepSeek run in this read-only review.

## Solution Assessment

No. Filtering at the stream seam is likely the right boundary, but this implementation must flush non-suppressed buffered text before non-content deltas or block transitions, and the contributor still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Flush DSML buffer before non-content deltas:** `src/agents/openai-transport-stream.ts:1498`
  - body: When a filtered content chunk ends with a prefix of an open DSML token, for example `see <`, and the next chunk carries `tool_calls` or reasoning, `push()` leaves the `<` in the filter buffer while the loop processes the non-content delta. The final `flush()` then emits that visible text after the tool/reasoning block, changing transcript order. Flush pending non-suppressed filter text before handling non-content deltas or switching block types.
  - confidence: 0.92
