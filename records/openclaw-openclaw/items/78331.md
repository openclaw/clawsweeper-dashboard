## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Flush pending text before handling non-content deltas:** `src/agents/openai-transport-stream.ts:1530-1531`
  - body: The new filter can retain a possible DSML-token prefix from a content chunk. Because the loop then processes reasoning or `tool_calls` before flushing that retained visible text, a stream like content `see <` followed by a native tool call records `<` after the tool block instead of before it.
  - confidence: 0.9
- **[P2] Accept singular DSML close tokens:** `src/agents/deepseek-text-filter.ts:37-44`
  - body: The filter opens on singular `<|DSML|tool_call>` but `DSML_CLOSE_TOKENS` only includes plural `tool_calls`. A complete singular block followed by normal text is treated as unterminated and the trailing visible text is dropped on final flush.
  - confidence: 0.94
- **[P2] Gate inline thinking on a real inline-thinking contract:** `src/agents/openai-transport-stream.ts:1654`
  - body: `hasEnabledThinkingPayload()` returns true for any `reasoning_effort`, but DeepSeek V4 thinking streams are documented and handled through `delta.reasoning_content`. Short normal answers can be buffered until stream end instead of emitting `text_delta` as they arrive.
  - confidence: 0.88
- **[P2] Respect thinkingFormat opt-outs for DSML filtering:** `src/agents/openai-transport-stream.ts:1636`
  - body: A DeepSeek model that explicitly sets `compat.thinkingFormat` away from `deepseek` still gets filtered because `model.provider === "deepseek"` wins. That bypasses the existing generic compat contract and can drop literal assistant text containing these markers on updated routes.
  - confidence: 0.9
