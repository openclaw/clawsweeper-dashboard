## Summary

Keep open: the DeepSeek leak appears real and the supplied parser replay proof is useful, but the latest patch still has blocking stream-ordering and parsing regressions plus a security-sensitive DSML-to-tool-call boundary that needs maintainer review.

## What This Changes

The PR adds a DeepSeek-scoped OpenAI-completions stream text processor, regression tests, and a changelog entry to suppress DSML leaks, parse DSML tool-call text, and route inline thinking output.

## Best Possible Solution

Keep the DeepSeek-scoped stream filter at the transport seam, fix the ordering, close-token, and inline-thinking gating bugs, and merge only after maintainers approve or narrow the DSML-to-tool-call conversion.

## Reproduction Assessment

Yes, at source level: current main appends `choice.delta.content` directly, so a captured DSML content delta will be emitted as assistant text. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

No. Filtering provider-private DeepSeek text at the OpenAI-completions stream seam is the right direction, but this patch still needs parser fixes and an explicit maintainer-approved tool-call boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Flush buffered text before non-content deltas:** `src/agents/openai-transport-stream.ts:1575-1577`
  - body: When a DeepSeek content chunk ends with a prefix of a DSML open token, `push()` can keep that visible suffix buffered. If the next chunk carries only `tool_calls` or reasoning, the loop handles that non-content block first and emits the buffered visible text only at final flush, changing stream order; flush pending visible text before switching to non-content handling.
  - confidence: 0.9
- **[P2] Accept singular DSML close tokens:** `src/agents/openai-transport-stream.ts:1717-1719`
  - body: The singular `<...tool_call>` definition only accepts plural `</...tool_calls>` close tokens. A complete singular block followed by normal assistant text can be treated as unterminated until `flush()`, parsing the tool call while dropping or delaying trailing visible text; include singular close tokens here too.
  - confidence: 0.94
- **[P2] Do not infer inline thinking from reasoning_effort:** `src/agents/openai-transport-stream.ts:1979`
  - body: Returning true for any `reasoning_effort` enables the inline `</think>` scanner on ordinary DeepSeek thinking requests. DeepSeek's OpenAI-compatible API returns thinking through `delta.reasoning_content`, already handled below, so normal short content chunks can stop emitting text deltas until completion; gate the inline marker path on a narrower endpoint signal.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive path that converts provider text into OpenClaw tool calls and should receive maintainer approval before merge.

Concerns:

- **[medium] Inline DSML can become executable tool calls:** `src/agents/openai-transport-stream.ts:1513`
  - body: `applyDeepSeekTextEvent` appends parsed DSML text as `toolCall` blocks whenever structured `delta.tool_calls` have not appeared. That can turn provider text into OpenClaw tool execution requests, so the exact DeepSeek contract and scoping need maintainer approval or narrower gating.
  - confidence: 0.82
