## Summary

Keep open: current `main` still has the gap the report describes. The Codex app-server path builds thread config from native hook relay plus native plugin `apps` patches, while user-defined `mcp.servers` are only projected by the CLI bundle-MCP path into Codex `mcp_servers`.

## What This Changes

The issue requests that Codex app-server agents receive user-defined `mcp.servers` from `openclaw.json` the same way `codex-cli` receives them.

## Best Possible Solution

Add a boundary-safe Codex app-server config projection that maps normalized OpenClaw `mcp.servers` into Codex `mcp_servers` alongside existing hook and native plugin `apps` config, with focused start/resume regression coverage.

## Reproduction Assessment

Yes, source-level. Current `main` shows the app-server thread config path never reads `config.mcp.servers`, while the reporter supplied a concrete beta reproduction and the CLI-only code path proves where `mcp_servers` is currently injected.

## Solution Assessment

Yes, the proposed app-server config projection is the narrow maintainable fix. It should be implemented through an extension-local or SDK-facing helper rather than patching built `dist` files or deep-importing core internals from the plugin.
