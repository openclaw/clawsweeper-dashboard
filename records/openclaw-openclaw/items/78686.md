## Summary

Keep this PR open, but it is not merge-ready: the happy-path proof is useful, yet the proposed skill-creator generator has concrete CLI parsing and code-generation bugs plus a bundled-skill scope decision for maintainers.

## What This Changes

The PR adds `--tool` generation to the bundled `skill-creator` skill, including `toolgen.py`, documentation, and regression tests for dash-prefixed tool args and Node syntax.

## Best Possible Solution

If maintainers want this in the bundled skill, fix the parser and code-generation issues, add the required changelog entry, and validate generated Python, Node, and Bash tools with targeted unit and lint checks.

## Reproduction Assessment

Yes for the review findings: the PR diff itself shows the unconditional `parse_known_args()` path, the Node value-consumption bug, and unescaped generated Python source. I did not run write-producing repro commands in the read-only checkout.

## Solution Assessment

No. A maintainable version should preserve strict CLI validation, use the generated argument schema when emitting templates, escape generated source safely, and first confirm this new bundled skill capability belongs in core rather than ClawHub.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject unknown args outside tool-arg forwarding:** `skills/skill-creator/scripts/init_skill.py:389-392`
  - body: Switching the whole CLI to `parse_known_args()` means misspelled or unsupported options no longer fail. For example, a basic init command with an unknown flag would now create the skill and silently ignore the bad flag when `--tool` is absent, or fold it into generated tool args when `--tool` is present. Please keep strict parsing except for values intentionally following `--tool-args`.
  - confidence: 0.91
- **[P2] Use the Node schema to consume flag values:** `skills/skill-creator/scripts/toolgen.py:166-174`
  - body: The generated Node parser only consumes a value when the runtime argument contains a colon, but the generated tool flags are normal flags like `--url`. A generated tool run as `fetcher.js --url https://example.com` records `url: true` and leaves the URL unused, so the advertised argument parsing is broken for Node tools.
  - confidence: 0.88
- **[P2] Escape strings before emitting Python source:** `skills/skill-creator/scripts/toolgen.py:88-90`
  - body: `toolgen.py` interpolates `description`, flags, and help text directly into quoted Python source. A normal apostrophe in a tool description or help string can make the generated Python file fail to compile, and crafted text can alter the generated script. Please emit these values with safe quoting such as `repr`/JSON-style escaping and validate generated identifiers/types.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `skills/skill-creator/SKILL.md:299-306`
  - body: This adds a user-facing `skill-creator` feature and documented CLI behavior, so repo policy requires a `CHANGELOG.md` entry before merge. The PR currently touches only the skill files and tests.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds local code-generation for executable skill scripts, and unescaped template inputs can emit unintended generated code if the generator is driven by untrusted text.
