## What This Changes

Add WhatsApp Web call activity detection and expose authorized, privacy-safe call information to agents.

## Best Possible Solution

Add a typed, WhatsApp-plugin-owned call-activity path only after maintainers define which statuses count as contact, the account/DM/group/self and allowlist rules, the opt-in consent surface, and whether events are ephemeral hooks or canonical SQLite-backed history; then sanitize and deduplicate Baileys events and document the supported agent-facing contract.

## Risks / Open Questions

- A raw call stream exposes communication metadata including contact identifiers, timestamps, group context, call type, and status; forwarding or storing it without explicit consent and authorization rules could reveal sensitive relationship and activity information.
- The issue proposes several materially different product surfaces—session metadata, persistent daily history, a message-tool query, or a plugin hook—and current code does not define which one should own non-message channel activity.
- Persisting `callsToday` or `lastCallWith` would introduce a stored data model and retention policy; that is substantially broader than subscribing to an ephemeral Baileys event and must not be implemented as ad hoc sidecar state.
- Baileys call status events may be repeated transitions for one call, so counting contact requires a defined deduplication and terminal-state policy rather than treating every event as a distinct interaction.
