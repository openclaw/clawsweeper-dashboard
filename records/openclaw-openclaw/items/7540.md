## Summary

Keep open: current main still lacks WhatsApp/Baileys call-event handling, and the live maintainer discussion explicitly scopes this as valid WhatsApp-plugin-owned, privacy-sensitive feature work rather than a PSTN voice-call or ClawHub-only item.

## What This Changes

The issue requests WhatsApp Web support for Baileys call events so call activity can count as user contact/activity and be exposed to agents through an opt-in, documented surface.

## Best Possible Solution

Implement this inside `extensions/whatsapp` with opt-in call tracking, sanitized WACallEvent normalization, existing account/DM/group/self/allowlist boundaries, a documented exposure surface, focused tests, and docs.

## Reproduction Assessment

Yes, for source-level reproduction: current main registers creds.update, connection.update, and messages.upsert but no call listener, while the pinned Baileys package exposes a call event. I did not generate a live WhatsApp call event in this read-only review.

## Solution Assessment

Partly: WhatsApp plugin ownership is the right boundary, but maintainers still need to choose the opt-in setting and runtime exposure surface before implementation. The PSTN voice-call plugin is not the right replacement because WhatsApp calls are a separate Web transport event stream.
