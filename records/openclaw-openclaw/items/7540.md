## Summary

Current main and v2026.6.5 still ignore Baileys call events, with no canonical issue or PR replacing this work; the enhancement remains valid but needs maintainer product and security decisions because it would expose sensitive WhatsApp activity metadata.

## What This Changes

Add optional WhatsApp call-event tracking from Baileys and expose sanitized, access-controlled call activity to agents.

## Best Possible Solution

Implement one WhatsApp-plugin-owned, opt-in activity contract that normalizes and minimizes Baileys call data, applies existing account and access-control boundaries, defines deduplication and retention explicitly, and exposes only the approved event or query surface with focused live and unit proof.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.5 consume WhatsApp message and connection events but not the typed Baileys call event. A live WhatsApp call was not run, but it is unnecessary to establish that the requested capability is absent.

## Solution Assessment

No final proposed shape is ready to implement autonomously. Keeping the work in the WhatsApp plugin is the correct boundary, but maintainers must first choose the privacy, retention, opt-in, deduplication, and agent-facing API semantics.
