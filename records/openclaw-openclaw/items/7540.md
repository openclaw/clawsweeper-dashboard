## Summary

Keep open: current main and the latest release still ignore Baileys WhatsApp call events, while the pinned Baileys package exposes them and the remaining work needs a maintainer decision on the privacy, access-control, and exposed activity surface.

## What This Changes

Request to detect WhatsApp voice/video call activity from Baileys call events and expose sanitized contact activity to agents or tools.

## Best Possible Solution

Implement this inside the WhatsApp plugin as an opt-in sanitized call-activity path that subscribes to Baileys call events, applies the existing account/DM/group/self/allowlist boundaries, and documents/tests the chosen session or hook surface.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 subscribe to WhatsApp message and connection events, while the exact Baileys package exposes a call event that OpenClaw does not consume. I did not run a live WhatsApp call flow.

## Solution Assessment

Unclear as a final product shape: the WhatsApp plugin is the right owner boundary, but maintainers should choose one sanitized opt-in activity surface instead of adding ad hoc session metadata and a parallel tool API.
