## Summary

Current main still ignores Baileys WhatsApp call events, and focused related-item searches found no merged or viable canonical replacement. The request remains useful and vision-aligned, but implementation needs explicit maintainer decisions for consent, sanitization, access control, retention, persistence, and the agent-facing contract.

## What This Changes

Add WhatsApp voice/video call-event ingestion so agent workflows can include calls in contact-activity tracking.

## Best Possible Solution

Keep this WhatsApp-plugin-owned and define one opt-in sanitized contact-activity contract that reuses inbound identity and access checks, stores only necessary canonical SQLite data with retention rules, and exposes a documented typed event or query rather than raw Baileys payloads.

## Reproduction Assessment

Yes at source level: current main registers credential, connection, and message events but no Baileys `call` event, while the installed dependency exposes that event. No live WhatsApp call was placed, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

No, the raw-listener sketch is not the complete best solution. The maintainable solution must first define sanitization, consent, access control, retention, canonical storage, and one agent-facing event or query contract inside the WhatsApp plugin.
