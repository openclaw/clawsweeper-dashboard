## Summary

Keep open: current main and the latest v2026.6.8 release still ignore Baileys call events, while the pinned Baileys contract exposes them. This remains a valid WhatsApp-plugin feature request, but the call metadata/privacy surface needs maintainer product and security direction before implementation.

## What This Changes

Add WhatsApp/Baileys voice and video call-event tracking so calls can count as contact activity or be exposed to agents.

## Best Possible Solution

Ship a WhatsApp-plugin-owned opt-in call-event surface that normalizes and sanitizes Baileys calls, reuses inbound admission/access boundaries, and documents/tests the privacy contract.

## Reproduction Assessment

Yes for the source-level gap: current main subscribes to message and connection events only, while the pinned Baileys contract exposes `call`. I did not run a live WhatsApp call scenario, so runtime E2E behavior remains unproven.

## Solution Assessment

Unclear until maintainers choose the permanent event/config/tool surface. The best direction is a plugin-owned opt-in implementation with the same privacy and access-control rigor as inbound messages.
