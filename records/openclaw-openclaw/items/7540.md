## Summary

Keep open. Current main still handles WhatsApp Web inbound activity as message-only, Baileys exposes a typed call-event stream, and the issue discussion already scopes this as valid WhatsApp-plugin-owned work that needs privacy/API decisions.

## What This Changes

Add opt-in support for consuming Baileys WhatsApp voice/video call events and exposing sanitized call activity to agents.

## Best Possible Solution

Implement this in extensions/whatsapp as an opt-in, sanitized call-activity surface that reuses the existing account, DM/group, self-chat, allowlist, and plugin-hook privacy boundaries, with focused tests and docs.

## Reproduction Assessment

Yes, as a source-level feature gap. Current main registers credential, connection, and message listeners but no Baileys call listener, while the pinned Baileys source exposes a typed call event.

## Solution Assessment

Unclear on the exact final API. The maintainable direction is WhatsApp-plugin-owned, default-off call tracking through a sanitized documented surface rather than folding WhatsApp Web calls into the PSTN voice-call plugin.
