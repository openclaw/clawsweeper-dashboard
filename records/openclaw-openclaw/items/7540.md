- `impact:session-state`: The requested behavior affects whether agent/session contact state can account for WhatsApp calls as activity.

## Summary

This should stay open: current `main` and the latest release still handle WhatsApp messages and connection state only, while the pinned Baileys package exposes a `call` event. The remaining work is valid but needs maintainer product/security review because call metadata includes contact identifiers, timing, status, and group/direct context.

## What This Changes

Request to add opt-in WhatsApp Web/Baileys call-event tracking so calls can count as contact activity alongside messages.

## Best Possible Solution

Implement this in the official WhatsApp plugin as an opt-in, sanitized call-activity surface that subscribes to Baileys `call`, applies the same account and sender-access boundaries as inbound messages, and documents/tests the chosen hook/tool/session contract.

## Reproduction Assessment

Yes, as a source-level feature gap: current `main` subscribes to WhatsApp message and connection events but has no Baileys `call` listener, call config, or call hook/tool/session surface. I did not run a live WhatsApp call because the missing subscription is directly visible in source and dependency types.

## Solution Assessment

Unclear as a final product shape: subscribing to Baileys `call` inside the WhatsApp plugin is the right boundary, but maintainers should choose the opt-in config and exposed activity surface before implementation. An unconditional default-on listener would be the wrong fit for the existing privacy model.
