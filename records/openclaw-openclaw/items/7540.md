## Summary

Current main still does not consume Baileys WhatsApp call events, and repository searches found no canonical issue or viable PR that supersedes this request. The enhancement fits the messaging-channel roadmap, but it needs maintainer decisions about its generic event API, authorization, privacy, consent, and retention before implementation.

## What This Changes

Add WhatsApp/Baileys call-event tracking so voice and video call activity can count as user contact.

## Best Possible Solution

Define a typed, sanitized contact-activity event with explicit access-control and retention semantics, then have the WhatsApp plugin map Baileys call events into it without coupling the feature to PSTN voice calls or ad hoc session fields.

## Reproduction Assessment

Yes at source level: current main has no WhatsApp `call` listener, normalization path, hook, tool, or documented call-activity surface, while the pinned dependency supplies the event. This is a missing capability rather than a regression in an existing contract.

## Solution Assessment

No, the issue’s proposed session metadata, tool action, and configuration paths are alternatives rather than a settled best fix. The narrowest maintainable direction is a typed generic activity event with plugin-owned Baileys mapping and an explicit privacy and persistence policy.

## Risks / Open Questions

- Call events expose contact identifiers, timing, status, direction, and group context, so implementation without explicit authorization, sanitization, and retention rules could leak sensitive activity metadata.
- Storing `lastCallWith` or `callsToday` directly in session metadata could create a WhatsApp-specific parallel source of truth; the generic activity-event and persistence boundary is not yet decided.
- A new configuration flag is not automatically justified; maintainers should first decide whether existing account or plugin-hook enablement can express operator consent without expanding the public config surface.
