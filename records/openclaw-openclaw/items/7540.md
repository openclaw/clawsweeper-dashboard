## Summary

Keep open: current main still lacks a WhatsApp/Baileys `call` event listener or agent-facing call activity surface, and the live discussion explicitly scopes this as valid WhatsApp-plugin-owned feature work that needs privacy and product decisions.

## What This Changes

Add WhatsApp Web call-event tracking from Baileys and expose sanitized call activity to agents, likely behind an opt-in setting.

## Best Possible Solution

Implement this inside the official WhatsApp plugin as an opt-in call-event path that normalizes and sanitizes `WACallEvent`, applies existing account/access boundaries, and documents/tests the agent-facing surface.

## Reproduction Assessment

Yes, at source level: current main subscribes to WhatsApp message and connection events only, while the pinned Baileys contract exposes a `call` event that OpenClaw does not consume. I did not run a live WhatsApp call.

## Solution Assessment

Yes on direction, but not as an automated repair yet: the work belongs in `extensions/whatsapp`, with maintainers deciding opt-in defaults, privacy boundaries, and the exposed call-activity contract before implementation.
