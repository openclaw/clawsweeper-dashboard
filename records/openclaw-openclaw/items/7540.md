## Summary

Current main and v2026.6.5 still ignore Baileys call events, and no canonical issue or pull request owns this work. The enhancement remains valid, but its privacy, retention, deduplication, and agent-facing contract require explicit maintainer product and security decisions.

## What This Changes

Add opt-in WhatsApp/Baileys voice and video call-event tracking and expose sanitized contact activity to agents.

## Best Possible Solution

Define one WhatsApp-plugin-owned normalized call-activity event with explicit opt-in, privacy filtering, lifecycle deduplication, SQLite-backed retention only if persistence is approved, and one documented agent/plugin consumption surface before attaching the Baileys listener.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.5 have no Baileys `call` consumer despite the dependency exposing that event. This is a missing capability rather than a failure of existing documented behavior.

## Solution Assessment

No, subscribing to `call` alone is not the best complete solution; the maintainable path must also define normalization, terminal-state semantics, privacy and access enforcement, deduplication, retention ownership, and one documented consumer contract.
