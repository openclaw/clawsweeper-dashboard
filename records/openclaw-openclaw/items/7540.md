## Summary

Current main still does not consume Baileys call events, the exact pinned dependency supports them, and no related issue or pull request supersedes this request. The feature remains necessary, but its privacy, retention, access-control, and agent-facing contract require maintainer product judgment before implementation.

## What This Changes

Add WhatsApp Web call-event tracking so call activity can contribute to contact history and agent-visible state.

## Best Possible Solution

Implement a WhatsApp-plugin-owned normalized call-activity pipeline after maintainers define eligible statuses, identity/access rules, retention, deduplication, and one agent-facing contract; default to no persistence or external hook emission unless explicitly enabled.

## Reproduction Assessment

Yes at source level: current main has no WhatsApp `call` event listener, while its exact pinned Baileys version exposes that event. This is a missing capability rather than a failing existing workflow requiring live reproduction.

## Solution Assessment

Unclear. Keeping ownership in the WhatsApp plugin is the right boundary, but session metadata, persistence, a tool query, and a new plugin hook have materially different security and compatibility costs that maintainers must decide between.
