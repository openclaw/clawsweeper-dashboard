## Summary

Keep open. Current main still lacks Baileys call-event handling in the bundled WhatsApp plugin, and maintainer context says this remains valid WhatsApp-plugin-owned work that needs privacy and API-surface decisions.

## What This Changes

Request to have the bundled WhatsApp Web plugin subscribe to Baileys `call` events and expose WhatsApp voice/video call activity to agents.

## Best Possible Solution

Implement this inside `extensions/whatsapp` with sanitized Baileys call normalization, message-equivalent access controls, an explicit opt-in surface, docs, and focused WhatsApp plugin tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inspect current main and confirm WhatsApp registers `creds.update`, `connection.update`, and `messages.upsert` but no `call` handler, while the pinned Baileys API exposes a typed `call` event.

## Solution Assessment

Unclear on the exact user-facing API, but the ownership direction is clear. The maintainable path is WhatsApp-plugin-owned handling with explicit privacy opt-in, not the PSTN `voice-call` plugin or a separate outbound WhatsApp-call channel.
