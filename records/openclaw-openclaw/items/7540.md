## Summary

Keep this open: current main still lacks a WhatsApp/Baileys call-event subscription or exposure path, and the live discussion/product labels continue to scope it as valid WhatsApp-plugin-owned work that needs privacy and surface decisions.

## What This Changes

Add support for detecting WhatsApp voice/video call activity from Baileys call events and exposing that activity to agents or session state.

## Best Possible Solution

Implement this inside the WhatsApp plugin by subscribing to Baileys call events, normalizing sanitized call activity, applying the same account/DM/group/self-chat/allowlist boundaries as inbound messages, and documenting the chosen opt-in hook/tool/session surface with focused tests.

## Reproduction Assessment

Yes, at source level: current main subscribes to WhatsApp message and connection events only, while the pinned Baileys contract exposes a call event that OpenClaw does not consume. I did not run a live WhatsApp call.

## Solution Assessment

Yes on direction, but not as an automated patch yet: the work belongs in extensions/whatsapp rather than the PSTN voice-call plugin, and maintainers need to decide the opt-in/privacy/exposure surface first.
