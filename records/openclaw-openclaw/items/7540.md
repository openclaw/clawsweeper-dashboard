## Summary

Current main and the latest release still do not consume Baileys WhatsApp call events. The request remains valid WhatsApp-plugin-owned feature work, but the public event/query surface, consent, authorization, deduplication, and retention rules need maintainer product and security review first.

## What This Changes

Add WhatsApp/Baileys voice and video call event tracking so calls can count as contact activity, not only chat messages.

## Best Possible Solution

Add a typed WhatsApp-plugin-owned call-activity path after maintainers define counted statuses, consent, authorization, deduplication, storage/retention, and the documented agent-facing event or query contract.

## Reproduction Assessment

Yes, at source level: current main has no Baileys call listener or call-event normalization path, while the exact pinned dependency exposes call: WACallEvent[]. I did not exercise a live WhatsApp call, so this is source-reproducible rather than a live reproduction.

## Solution Assessment

No, the listener-only proposal is not the best complete solution because it leaves authorization, consent, deduplication, retention, and the public event/query surface undefined. The safer path is a typed WhatsApp-plugin-owned activity contract first, then transport wiring.
