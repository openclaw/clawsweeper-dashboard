## Summary

Keep open: current main and the latest release still lack WhatsApp/Baileys call-event handling, while the pinned Baileys contract exposes that event and the issue discussion scopes this as valid WhatsApp-plugin work pending privacy, security, and product decisions.

## What This Changes

The issue requests subscribing to Baileys WhatsApp call events and exposing call-contact activity to agents.

## Best Possible Solution

Implement this inside `extensions/whatsapp` with opt-in call tracking, sanitized `WACallEvent` normalization, existing account/DM/group/self/allowlist checks, a documented exposure contract, focused tests, and redacted live or fixture-backed proof.

## Reproduction Assessment

Yes, for source-level reproduction. Current main registers `creds.update`, `connection.update`, and `messages.upsert` but no `call` listener, while the pinned Baileys version exposes `Events.call` as `WACallEvent[]`.

## Solution Assessment

Unclear. The WhatsApp plugin boundary is right, but maintainers still need to choose the opt-in config and whether call activity is exposed as session metadata, a tool query, a plugin hook, or another event surface.
