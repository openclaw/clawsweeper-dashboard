## Summary

Keep this open: current main still lacks a WhatsApp/Baileys call listener and any call activity config, hook, or session surface, while the dependency contract and existing discussion show this is valid WhatsApp-plugin-owned feature work that needs product/privacy scoping.

## What This Changes

Add WhatsApp/Baileys voice/video call-event detection so calls can count as contact activity.

## Best Possible Solution

Implement a WhatsApp-plugin-owned call-event path that normalizes and sanitizes Baileys call events, applies existing inbound access boundaries, and exposes them through an explicit documented opt-in surface.

## Reproduction Assessment

Yes, at source level: current main subscribes to WhatsApp message and connection events only, while the pinned Baileys contract exposes a call event that OpenClaw does not consume. I did not run a live WhatsApp call.

## Solution Assessment

Yes on direction, but not as an automated repair yet: the work belongs in extensions/whatsapp, with maintainers deciding the opt-in, privacy, and exposure surface before implementation.
