## Summary

Keep open: current main and the latest release still consume WhatsApp message and connection events only, while the pinned Baileys package exposes typed call events; the remaining work is valid but needs a maintainer privacy/product decision before automation.

## What This Changes

Add WhatsApp Web/Baileys call-event subscription and expose sanitized voice/video call activity so contact-aware workflows can count calls as contact.

## Best Possible Solution

Implement this inside the WhatsApp plugin as one documented opt-in sanitized call-activity surface, reusing inbound account, DM/group, self-chat, allowlist, and privacy boundaries with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current WhatsApp code subscribes to Baileys message and connection events, while the exact pinned Baileys package exposes a call event that OpenClaw does not consume. I did not run a live WhatsApp call flow.

## Solution Assessment

Unclear as a final product shape: subscribing inside the WhatsApp plugin is the right owner boundary, but maintainers should choose one sanitized opt-in activity surface rather than adding ad hoc session metadata and a separate tool API in parallel.
