## Summary

Current main and v2026.6.5 still consume WhatsApp messages and connection updates but not Baileys call events, and no related issue or PR now owns the same capability. This remains valid bundled WhatsApp plugin work, but its sensitive activity data, persistence, configuration, and agent-facing semantics require maintainer product and security decisions before implementation.

## What This Changes

Add opt-in WhatsApp call-event tracking through the bundled Baileys channel and expose sanitized call activity to agents.

## Best Possible Solution

Define one WhatsApp-plugin-owned call-activity contract that sanitizes and authorizes events, stores only approved durable facts in SQLite when necessary, and exposes a documented query or typed event surface without treating calls as ordinary messages.

## Reproduction Assessment

Yes at source level: current main directly shows message and connection subscriptions without a Baileys call listener or call-activity surface. A live WhatsApp voice/video call was not exercised, so this is source reproduction rather than an end-to-end runtime reproduction.

## Solution Assessment

No, adding only `sock.ev.on("call", ...)` is not the best complete solution. The narrow maintainable fix must jointly define sanitization, access control, deduplication, retention, and the supported agent-facing query or typed event contract.
