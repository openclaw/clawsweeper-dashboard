## Summary

Keep open: current main still ignores Baileys WhatsApp call events, while the pinned Baileys package exposes them and the remaining work needs explicit WhatsApp privacy, access-control, and product-surface decisions.

## What This Changes

The issue requests WhatsApp Web support for Baileys voice/video call events so call activity can count as contact activity.

## Best Possible Solution

Keep this as WhatsApp-plugin-owned work: add a narrowly documented opt-in for sanitized call activity, subscribe to Baileys call events, apply the same account/DM/group/self/allowlist boundaries as inbound messages, and cover the behavior with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current WhatsApp code subscribes to Baileys message and connection events, while the exact Baileys package exposes a call event that OpenClaw does not consume. I did not run a live WhatsApp call flow.

## Solution Assessment

Unclear as a final product shape: subscribing inside the WhatsApp plugin is the right owner boundary, but maintainers should choose one sanitized opt-in activity surface instead of adding ad hoc session metadata and a separate tool API in parallel.
