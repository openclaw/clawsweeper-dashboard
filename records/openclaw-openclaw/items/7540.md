## Summary

Keep open: current main still does not consume Baileys WhatsApp call events, and this remains a valid WhatsApp-plugin feature that needs product/security direction before implementation.

## What This Changes

The issue requests subscribing to Baileys WhatsApp call events and exposing sanitized call activity so calls can count as contact activity.

## Best Possible Solution

Implement a WhatsApp-plugin-owned, opt-in sanitized call-activity path that applies existing account, DM/group, self-chat, allowlist, and privacy boundaries before updating contact/session activity or emitting a documented hook.

## Reproduction Assessment

Yes. Source inspection shows current main subscribes to WhatsApp message and connection events, while the declared Baileys package exposes a separate call event that OpenClaw does not consume; no live WhatsApp call scenario was run.

## Solution Assessment

Unclear as a final product shape. The WhatsApp plugin is the right owner boundary, but maintainers should choose one sanitized opt-in activity surface rather than adding ad hoc session metadata plus a parallel tool API.
