## Summary

Keep open. Current main still lacks WhatsApp/Baileys call-event support, the bundled WhatsApp plugin remains message-focused, and maintainer discussion explicitly scoped this as valid WhatsApp-plugin-owned work rather than PSTN voice-call or ClawHub cleanup.

## What This Changes

The issue requests support for detecting WhatsApp voice/video call activity from Baileys `call` events so contact tracking can count calls as well as messages.

## Best Possible Solution

Implement this in `extensions/whatsapp`: subscribe to Baileys `call`, normalize and sanitize `WACallEvent` payloads, apply the same account, DM/group, self-chat, allowlist, and privacy boundaries as inbound messages, add an explicit opt-in if maintainers want one, document the exposed call-activity surface, and cover it with focused WhatsApp plugin tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inspect current main and confirm the WhatsApp socket/inbound monitor register `creds.update`, `connection.update`, and `messages.upsert`, but no `call` handler, while the pinned Baileys API exposes a typed `call` event.

## Solution Assessment

Unclear on the exact user-facing API, but the ownership direction is clear. The narrow maintainable path is WhatsApp-plugin-owned implementation; folding this into the PSTN voice-call plugin or generic core without a documented privacy surface would be the wrong boundary.
