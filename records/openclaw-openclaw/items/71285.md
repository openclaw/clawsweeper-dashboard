## Summary

Keep open. Current main and v2026.6.8 still lack a memory-search exclusion contract: builtin scanning and QMD default collections include `memory/**/*.md`, while default dreaming storage writes separate reports under `memory/dreaming/<phase>/...`; the community `memory-lancedb-dreaming` plugin is a useful workaround, but it does not replace the missing bundled memory-core/builtin/QMD product decision.

## What This Changes

The issue requests configurable memory-search exclusion paths so generated dreaming artifacts can be kept out of indexing and watcher scope.

## Best Possible Solution

Keep this issue as the canonical tracker and have maintainers define one documented exclusion or generated-artifact filtering contract across schema/types, builtin indexing, QMD collections, watcher scope, status/docs, and reindex cleanup.

## Reproduction Assessment

Yes at source level: current scanning and QMD default collection code include `memory/**/*.md`, while dreaming defaults write separate reports under `memory/dreaming/<phase>/...`. I did not run a live `memory_search` scenario.

## Solution Assessment

Unclear. A config-based exclusion knob is plausible, but the best solution needs maintainer agreement on API naming, matcher semantics, builtin/QMD parity, watcher behavior, and cleanup/reindex behavior.
