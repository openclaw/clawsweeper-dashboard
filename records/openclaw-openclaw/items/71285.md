## Summary

Keep open. Current main and v2026.6.8 still lack a memory-search exclusion config, and both builtin and QMD indexing can include generated `memory/dreaming/**/*.md` reports; the `memory-lancedb-dreaming` ClawHub plugin is useful context for LanceDB users, but it does not resolve the core memory-search contract requested here.

## What This Changes

The issue requests a config-based memory-search exclusion path so users can keep `memory/dreaming/` generated reports out of indexing, search results, and watcher scope.

## Best Possible Solution

Define one documented generated-artifact exclusion or classification contract for memory search, then implement it consistently across schema/types, builtin scanning, QMD collections, watcher behavior, docs, status output, tests, and reindex guidance.

## Reproduction Assessment

Yes, at source level: current builtin indexing recurses through `memory/`, and QMD default collections include `memory/**/*.md`, so Markdown reports under `memory/dreaming/` are eligible for the memory corpus. I did not run a live memory-search query in this read-only review.

## Solution Assessment

Unclear. A config-based exclusion is plausible, but the best fix needs maintainer agreement on the public contract, QMD/builtin parity, default behavior, and stale-index cleanup before implementation.
