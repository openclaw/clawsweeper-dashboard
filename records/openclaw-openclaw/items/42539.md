## Summary

Keep this issue open: current main still bundles auto-TTS text into the Telegram voice caption, and no supported delivery-shape setting provides the requested split. The request is valid but needs maintainer product judgment because it adds configuration and must define shared ordering, streaming, failure-fallback, and cross-channel semantics.

## What This Changes

Requests a configurable Telegram TTS mode that sends readable text and the voice note as separate messages instead of one captioned voice message.

## Best Possible Solution

Keep bundled delivery as the default, define a closed shared TTS delivery mode after maintainer approval, and have Telegram map the separate mode to a normal text message followed by an uncaptioned voice note while preserving text on synthesis or delivery failure.

## Reproduction Assessment

Yes, at source level: auto-TTS builds one text-plus-audio payload, and Telegram uses that payload's text as the voice caption, with separate text reserved for overflow or error fallback. No live Telegram run was performed during this read-only review.

## Solution Assessment

No, the proposed config sketch alone is underspecified. The maintainable solution needs a shared delivery contract and explicit streaming, ordering, cross-channel, and failure semantics before Telegram implements its transport mapping.
