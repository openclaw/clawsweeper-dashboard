## Summary

Keep open. Current main still delivers normal Telegram auto-TTS as one media payload whose text becomes a voice caption except for the existing caption-overflow fallback, and the requested separate text plus voice delivery mode is a new config/product surface rather than an implemented behavior.

## What This Changes

Request a configurable Telegram TTS delivery mode that sends readable text and the voice note as separate messages instead of a single voice message with a caption.

## Best Possible Solution

Add a backward-compatible shared auto-TTS delivery policy that defaults to current bundled/caption behavior and supports a Telegram-safe separate text plus voice mode with schema, docs, focused unit coverage, and real Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence in the current behavior: enable auto-TTS for a normal Telegram reply shorter than the caption limit and the current path sends one voice/media message with text as caption.

## Solution Assessment

Unclear. The proposed config direction is plausible, but the maintainable fix should settle the shared TTS delivery policy and related voice-only/async overlap before adding a Telegram-only mode.
