## Summary

Keep this open: current main still has no Telegram TTS delivery selector, and the implemented path remains attach-audio-to-reply plus Telegram caption delivery, with separate text only for overflow or voice-error fallback cases. This is a real feature request, but the config/product semantics need maintainer choice before an automated fix should add a public option.

## What This Changes

The issue requests a Telegram TTS delivery option that sends readable text and the voice note as separate Telegram messages instead of a single voice-with-caption message.

## Best Possible Solution

Settle one canonical auto-TTS text/audio delivery contract, then update config schema/docs and Telegram plus speech-core coverage if separate delivery is accepted.

## Reproduction Assessment

Yes, from source inspection: current main has no delivery-mode config key, auto-TTS attaches audio to the existing reply payload, and Telegram captions first-media replies unless overflow or voice-error fallback splits the text. I did not run a live Telegram client scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A delivery-mode config option is plausible, but it should be reconciled with voice-only/text-suppression, captioned-voice, and final-mode churn work before adding a public config surface.
