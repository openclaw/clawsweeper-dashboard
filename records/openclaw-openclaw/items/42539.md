## Summary

Keep this open: current main and v2026.6.6 still lack a Telegram TTS delivery selector, and the Telegram path remains caption-first except for overflow/fallback cases. The related open items do not implement this request; one explores the opposite caption-combining behavior and another is a separate voice-button feature.

## What This Changes

The issue requests a configurable Telegram TTS delivery mode that sends readable normal text and a voice note as separate messages instead of using the voice-note caption by default.

## Best Possible Solution

Settle one canonical auto-TTS text/audio delivery contract, then update the config schema/docs and Telegram plus speech-core coverage if separate delivery is accepted.

## Reproduction Assessment

Yes, from source inspection: current main has no delivery-mode key, auto-TTS keeps visible text on the audio-bearing reply, and Telegram uses that text as a voice/media caption unless overflow or fallback handling splits it. I did not run a live Telegram client scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A delivery-mode config option is plausible, but it should be reconciled with voice-only/text-suppression, captioned-voice, and final-mode churn work before adding a public config surface.
