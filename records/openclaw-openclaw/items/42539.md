## Summary

Keep open: current main still lacks a configurable always-separate Telegram TTS delivery mode. Normal auto-TTS success keeps visible text on the media-bearing payload, and Telegram sends normal-length voice/audio text as a caption, so this remains a feature/config product decision rather than cleanup.

## What This Changes

The issue requests a Telegram TTS delivery mode that sends full-width text and a separate voice message instead of a captioned voice message.

## Best Possible Solution

Define a backward-compatible TTS delivery policy that keeps the current bundled/captioned default while allowing Telegram to send full-width text and a separate voice note, with schema, docs, and channel tests covering the chosen ordering semantics.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection gives high confidence for the current behavior: normal Telegram auto-TTS keeps text on the media payload and the Telegram sender turns normal-length text into a voice/audio caption.

## Solution Assessment

Unclear until maintainers choose the product contract. The proposed `messages.tts.delivery: "separate"` direction is plausible, but it should be aligned with async/background TTS and voice-only text-delivery semantics before implementation.
