## Summary

Keep this open: current main and the latest release still lack a Telegram TTS separate-delivery selector, while the open related captioned-final-TTS PR solves a different, mostly opposite delivery policy.

## What This Changes

The issue requests a `messages.tts` delivery option that sends Telegram TTS text as a normal message and the voice note as a separate message instead of using the text as a voice caption.

## Best Possible Solution

Maintainers should choose the canonical TTS text/audio delivery policy first; if separate Telegram delivery is accepted, implement it through the typed config/schema/docs plus speech-core and Telegram delivery tests while preserving the current bundled default.

## Reproduction Assessment

Yes. Source inspection shows current main has no delivery-mode config key, auto-TTS attaches audio to the existing reply payload, and Telegram captions that text unless the overflow/error fallback splits it.

## Solution Assessment

Unclear. A delivery-mode option is plausible, but maintainers should first reconcile it with captioned final-mode TTS and voice-only/text-suppression proposals before adding public config.
