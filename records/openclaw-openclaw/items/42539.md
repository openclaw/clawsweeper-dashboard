## Summary

Keep open: current main still lacks a TTS delivery-mode option, auto-TTS preserves visible reply text while adding audio to the same payload, and Telegram still turns that text into the first voice/media caption except for overflow or failure fallback paths. This is a clear user-facing feature/config request that needs maintainer product/API judgment, not conservative cleanup.

## What This Changes

Request a configurable Telegram TTS mode that sends readable text and a separate voice note instead of captioning the voice payload.

## Best Possible Solution

Define a backward-compatible, typed TTS delivery policy that defaults to current bundled behavior and adds a Telegram-safe separate text/voice mode with ordering, fallback, docs, and tests.

## Reproduction Assessment

Yes for current behavior from source: enable auto-TTS for a normal Telegram final reply, and the code preserves reply text while adding audio to the same payload, then Telegram uses that text as the voice/media caption. This is a feature request rather than a broken existing contract, so no live failure reproduction is required to keep it open.

## Solution Assessment

Unclear. The requested separate mode is plausible, but maintainers should choose the policy scope and its relationship to async TTS and voice-only delivery before implementation.
