## Summary

Keep open: current main and v2026.6.6 still do not provide `messages.tts.delivery` or an equivalent text/voice split mode, and the remaining work is a Telegram/TTS product and config-contract decision rather than cleanup.

## What This Changes

The issue requests a configurable Telegram TTS delivery mode that sends text and voice as separate messages instead of using a voice-caption bundle.

## Best Possible Solution

Decide one canonical auto-TTS text/voice delivery policy across Telegram and other voice-note channels, then implement the accepted mode through config/schema, channel capability contracts where needed, docs, and focused speech-core plus Telegram tests.

## Reproduction Assessment

Yes. Source inspection shows current main has no delivery-mode config key, Auto-TTS attaches generated audio to the visible reply, and Telegram captions voice/media except for overflow or voice-send fallback.

## Solution Assessment

Unclear. A delivery-mode option is plausible, but the best fix needs a maintainer decision across separate text, captioned voice, and voice-only behavior before adding another public TTS config surface.
