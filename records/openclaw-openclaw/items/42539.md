## Summary

Keep this open: current main still lacks a Telegram TTS separate-delivery selector, and the active related PR is solving captioned final-mode churn rather than this issue's requested readable text-plus-separate-voice mode.

## What This Changes

Add a Telegram TTS delivery mode option so users can send readable text as a normal message plus a separate voice note instead of always captioning the voice note.

## Best Possible Solution

Maintainers should choose the canonical TTS text/audio delivery policy first; if separate Telegram delivery is accepted, implement it through typed config/schema/docs plus speech-core and Telegram delivery tests while preserving the current bundled default.

## Reproduction Assessment

Yes for the current behavior, but not as a broken-contract bug: source inspection shows current main has no delivery-mode config key, auto-TTS attaches audio to the existing reply payload, and Telegram captions that text unless overflow or sendVoice fallback splits it.

## Solution Assessment

Unclear. A delivery-mode option is plausible, but maintainers should first reconcile it with captioned final-mode TTS and voice-only/text-suppression proposals before adding public config.
