## Summary

Keep this open: current main and the latest release still lack a Telegram TTS delivery selector, and the active captioned-final-TTS work moves toward a single voice-with-caption message rather than the requested full-width text plus separate voice note.

## What This Changes

The issue requests a `messages.tts` delivery mode that sends Telegram auto-TTS text and voice as separate messages instead of a voice message caption.

## Best Possible Solution

Choose one canonical TTS text/audio delivery policy first; if separate delivery is accepted, update the config schema/types, speech-core payload shaping, Telegram delivery behavior, docs, and focused tests together.

## Reproduction Assessment

Yes at source level. Current main has no TTS delivery-mode config key, auto-TTS attaches audio to the same reply payload, and Telegram captions short media text unless the overflow fallback splits it; I did not run a live Telegram client scenario.

## Solution Assessment

Unclear. A delivery-mode option is plausible, but maintainers should first reconcile it with captioned final-mode TTS and voice-only/text-suppression proposals before adding public config.
