## Summary

Keep this open: current main and v2026.6.10 still lack a Telegram TTS separate-delivery selector, and the active captioned-final work solves a different single-captioned-message policy.

## What This Changes

The issue requests a Telegram TTS delivery option that sends readable text as a normal message and voice as a separate voice message instead of bundling text as a voice caption.

## Best Possible Solution

Keep this as the canonical product/config decision for Telegram separate text-plus-voice delivery and reconcile it with captioned-final and voice-only TTS direction before implementation.

## Reproduction Assessment

Yes for the current behavior, but as a source-level feature gap rather than a broken existing contract. Current main has no delivery-mode config key, speech-core emits one media-bearing payload, and Telegram captions that text unless overflow or sendVoice fallback splits it.

## Solution Assessment

Unclear as a product direction. A delivery-mode option is plausible, but maintainers should first reconcile it with captioned final-mode TTS and voice-only/text-suppression proposals before adding public config.
