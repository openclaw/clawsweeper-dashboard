## Summary

Keep this open: current main and v2026.6.6 still lack a Telegram TTS delivery selector, and the code continues to attach generated audio to the existing reply so Telegram uses captions except for overflow/error fallback. The request is valid, but adding a public `messages.tts` delivery mode needs maintainer product direction because related TTS work points in different directions.

## What This Changes

Request a `messages.tts` delivery-mode option so Telegram can send full-width text and a voice note as separate messages instead of one captioned voice message.

## Best Possible Solution

Choose one canonical auto-TTS text/audio delivery policy, then update the config schema, docs, speech-core behavior, and Telegram delivery tests if separate delivery is accepted.

## Reproduction Assessment

Yes, from source inspection: current main has no delivery-mode config key, auto-TTS attaches audio to the existing reply payload, and Telegram captions that text unless overflow or voice-error fallback splits it. I did not run a live Telegram client scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A delivery-mode option is plausible, but it should be reconciled with voice-only/text-suppression, captioned-voice, and final-mode churn work before adding public config.
