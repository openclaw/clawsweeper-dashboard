## Summary

Keep open: current main still lacks an always-separate Telegram TTS delivery mode; normal-length Telegram voice replies still carry the visible text as a media caption, and only overflow/error fallbacks split text from voice.

## What This Changes

Request a configurable Telegram TTS delivery mode that sends readable text as a normal message and the voice note as a separate follow-up message.

## Best Possible Solution

Define a backward-compatible TTS delivery policy that preserves the current bundled default while allowing Telegram to send full-width text and a separate voice note, ideally in a shape that can also support future async TTS delivery.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source and tests give a high-confidence current-behavior path: auto-TTS attaches audio to one payload, and Telegram sends normal-length voice replies with that text as the caption.

## Solution Assessment

Unclear until maintainers choose the product contract. The requested `messages.tts.delivery` direction is plausible, but it should be weighed against a generic TTS delivery policy, a Telegram channel override, and #69315's async/text-first work.
