## Summary

Keep open. Current main still attaches automatic TTS audio to the same reply payload and Telegram still sends that text as a media caption unless the caption overflows, so the requested separate-delivery mode remains unimplemented and needs a product/config decision.

## What This Changes

Requested behavior: add a configurable Telegram TTS delivery mode that sends readable text and voice as separate messages instead of a single voice caption.

## Best Possible Solution

Design a backward-compatible TTS delivery policy, defaulting to today’s bundled caption behavior, with a Telegram implementation, schema/docs updates, and tests for ordering, threading, buttons, and overflow fallback.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still attaches auto-TTS audio to one payload and Telegram captions the media unless caption overflow triggers the existing fallback.

## Solution Assessment

Unclear. A delivery mode is a plausible direction, but maintainers should choose the shared versus Telegram-scoped config surface and message ordering before implementation.
