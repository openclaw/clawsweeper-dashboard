## Summary

Keep this open: current main and v2026.6.6 still lack a Telegram TTS delivery selector, and the open captioned-final-TTS work solves a related churn problem in the opposite direction rather than offering separate text-plus-voice delivery.

## What This Changes

Request a `messages.tts` delivery option so Telegram can send readable full-width text and a separate voice note instead of using a voice caption.

## Best Possible Solution

Choose one canonical TTS text/audio delivery policy first; if separate delivery is accepted, update the config schema/types, docs, speech-core payload shaping, and Telegram delivery tests together.

## Reproduction Assessment

Yes. Source inspection shows current main has no delivery-mode config key, auto-TTS attaches audio to the existing reply payload, and Telegram captions that text unless overflow or voice-error fallback splits it; I did not run a live Telegram client scenario.

## Solution Assessment

Unclear. A delivery-mode option is plausible, but maintainers should first reconcile it with captioned final-mode TTS and voice-only/text-suppression proposals before adding public config.
