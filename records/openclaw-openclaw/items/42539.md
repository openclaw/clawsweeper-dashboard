## Summary

Keep open. Current main still preserves visible text on successful auto-TTS, sends Telegram voice media with that text as a caption unless it exceeds Telegram's caption limit, and has no config/schema/docs surface for a default separate text-plus-voice delivery mode; the closed duplicate points back to this issue while the broader async and voice-only issues do not fully cover this exact UX request.

## What This Changes

The issue requests a configurable Telegram TTS delivery mode that sends full-width text and the generated voice note as separate messages instead of one captioned voice message.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy, defaulting to current captioned behavior, that can make Telegram send readable text and the voice note as separate messages while documenting its interaction with async and voice-only TTS work.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source and existing tests give high confidence that current main preserves text on the auto-TTS payload and Telegram captions short voice sends, splitting only oversized captions.

## Solution Assessment

Unclear. The requested separate-delivery direction is plausible, but the maintainable fix should be a shared delivery policy coordinated with async and voice-only TTS work rather than a Telegram-only one-off.
