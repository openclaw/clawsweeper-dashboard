## Summary

Keep open. Current main still lacks a configurable Telegram TTS delivery mode: the validated TTS config has no delivery policy, speech-core still returns one media-bearing reply payload, and Telegram reply delivery still uses the visible reply text as the voice caption except for overflow or sendVoice error fallbacks.

## What This Changes

The issue requests a Telegram TTS delivery option that sends readable text and the voice note as separate messages instead of one voice message with a caption.

## Best Possible Solution

Add a backwards-compatible TTS delivery policy after maintainer product review. The default should preserve bundled voice-caption behavior, while a separate mode should send the visible Telegram text as a normal message first and then send the voice note without a caption, with schema/types/docs and focused Telegram plus speech-core tests.

## Reproduction Assessment

Yes. A read-only current-main trace is enough: enable auto-TTS for Telegram with a reply under Telegram's caption limit, and speech-core produces one media reply while Telegram sends reply.text as the sendVoice caption.

## Solution Assessment

Unclear until maintainers choose the public config boundary. The requested text-first and captionless-voice behavior is maintainable, but the narrowest durable fix should account for Telegram-specific UX and the broader open async TTS delivery discussion in #69315.
