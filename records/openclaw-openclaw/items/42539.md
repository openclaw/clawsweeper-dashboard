## Summary

This should stay open: current main and v2026.6.6 still have no configurable Telegram TTS text/voice delivery mode, and source inspection confirms the current behavior remains caption-first except for overflow or fallback cases.

## What This Changes

Request to add a Telegram TTS delivery mode that sends readable text and a separate voice note instead of one captioned voice message.

## Best Possible Solution

Choose one canonical auto-TTS text/voice delivery policy across Telegram and other voice-note channels, then implement the accepted behavior through config/schema/docs and focused Telegram plus speech-core coverage.

## Reproduction Assessment

Yes, from source inspection: current main has no `messages.tts.delivery`-style key, Auto-TTS keeps visible text on the audio-bearing reply, and Telegram uses that text as the voice/media caption unless overflow or sendVoice fallback kicks in. I did not run a live Telegram client scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A delivery-mode config option is plausible, but it should be reconciled with open captioned-voice churn fixes and voice-only/text-suppression requests before adding a new public config surface.
