## Summary

This remains necessary: current `main` still percent-encodes a provider-qualified Google model ID into the Vertex model segment, while the PR reuses the established Google prefix normalizer at the correct provider-owned boundary. The patch is focused, cleanly mergeable, consistent with the sibling Gemini path and Google’s Vertex REST resource format, and backed by direct live before/after proof; no actionable correctness or security finding remains.

## What This Changes

The PR strips a redundant `google/` prefix before encoding Google Vertex model IDs and adds a regression test for the generated request URL.

## Best Possible Solution

Land the focused normalization at the Vertex URL boundary and keep the regression test alongside the existing Gemini path test so both Google REST transports enforce the same model-ID invariant.

## Reproduction Assessment

Yes. Current `main` deterministically builds the encoded `google%2F...` path, and the supplied live Vertex REST plus real OpenClaw transport comparison shows the failure before and successful generation after the patch.

## Solution Assessment

Yes. Reusing the existing provider-owned prefix normalizer immediately before Vertex URL encoding is the narrowest maintainable fix and avoids changing canonical model IDs, configuration, routing, or public APIs.
