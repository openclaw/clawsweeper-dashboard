## Summary

Keep open: this is a focused Moonshot/Kimi bug-fix PR, but it broadens provider fallback for every custom Moonshot chat base URL and has only unit-test proof, so compatibility direction and real behavior proof need to land before merge.

## What This Changes

The PR removes the native-host guard when Kimi web search falls back to `models.providers.moonshot.baseUrl` and updates the regression test to expect custom proxy inheritance.

## Best Possible Solution

Preserve native-host inheritance as the default and use the existing Kimi webSearch baseUrl override for custom endpoints unless maintainers explicitly approve broader custom-chat inheritance with upgrade proof.

## Reproduction Assessment

Yes, source-level reproduction is clear: on current main, a custom `models.providers.moonshot.baseUrl` falls back to the default Kimi host because the native-host guard rejects it. I did not establish a live Kimi/proxy repro with real credentials.

## Solution Assessment

No. The code change is a plausible narrow fix for custom proxies, but the best mergeable solution needs either explicit custom Kimi baseUrl use or maintainer-approved upgrade behavior for inheriting every custom chat baseUrl.
