## What This Changes

The issue reports that isolated cron jobs using the `claude-cli` backend can be marked `queued_work_without_active_run` while actively streaming, leading to ineffective recovery and later stalled isolated cron runs.

## Best Possible Solution

Land a focused diagnostic classifier or active-work ownership fix for fresh CLI-live progress, then use live isolated cron proof to decide whether any separate `runtime-plugins` stall remains.

## Reproduction Assessment

Yes for the core misclassification: current source can produce fresh `cli_live:stream_progress` without `activeWorkKind`, then classify queued work as recoverable stuck state. The full multi-hour lane/capacity cascade still needs live gateway proof after the classifier fix.

## Solution Assessment

Yes for the near-term direction: a narrow fresh-progress gate or equivalent CLI active-work registration is the maintainable fix for the source-confirmed false recovery. Broader cron capacity work should wait until proof shows a separate failure remains.
