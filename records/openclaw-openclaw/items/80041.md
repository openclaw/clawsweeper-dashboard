## Summary

Keep open. Current main still reuses the CLI pre-read config snapshot for every in-process gateway restart, while this PR applies the narrow first-start-only fix with regression coverage and credible Docker before/after proof. There is an earlier duplicate fix PR at https://github.com/openclaw/openclaw/pull/79951, but this PR is the more complete candidate because it includes tests, changelog, and real behavior proof.

## What This Changes

The branch consumes `startupConfigSnapshotRead` only on the first gateway loop start, adds a regression test, and records a changelog entry for the in-process restart config-loss bug.

## Best Possible Solution

Land one canonical narrow fix that consumes the pre-read snapshot only on cold start, lets restart iterations reread disk through the existing startup loader, and closes the linked config-loss bug.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: pre-read config once, write config before SIGUSR1, then the next in-process restart calls the same start callback with the stale snapshot; the PR body also provides Docker before/after hashes and logs.

## Solution Assessment

Yes. Consuming the snapshot in `runGatewayCommand` is the narrowest maintainable fix because it preserves the first-start optimization and avoids changing standalone callers that intentionally pass an initial snapshot.
