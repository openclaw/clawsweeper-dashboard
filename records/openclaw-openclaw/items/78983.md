## Summary

Keep open: current main partially improves subagent labeling, but the agent switcher can still auto-select a target agent's newest subagent session because preferred-session selection does not exclude subagent keys.

## What This Changes

The issue asks the Control UI session picker to avoid exposing unmarked subagent sessions and to switch agents back to an appropriate main/default session instead of a prior or transient subagent session.

## Best Possible Solution

Land a focused Control UI fix that prevents agent switching from auto-selecting subagent sessions while preserving the existing recent-dashboard preference and leaving broader picker grouping/filtering to the related UI work.

## Reproduction Assessment

Yes, from source inspection: a newer `agent:<id>:subagent:*` row for the target agent can be returned by `resolvePreferredSessionForAgent`, because the helper chooses the newest target-agent row without excluding subagent keys. I did not run a live browser scenario in this read-only review.

## Solution Assessment

Yes for a narrow repair: skip subagent keys when computing the preferred session on agent changes, keep the existing recent-dashboard behavior, and add regression tests for subagent-newer-than-main and subagent-newer-than-dashboard cases.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78983 by preventing the Control UI agent dropdown from automatically switching into subagent sessions. In `ui/src/ui/chat/session-controls.ts`, update `resolvePreferredSessionForAgent` so it ignores subagent session keys when choosing the target agent's preferred row, while preserving the current same-agent no-op and the existing preference for a recent dashboard session over main when a dashboard session exists. Do not implement a broad session-kind filter, hierarchy/grouping UI, or clear-completed workflow; those are covered by related picker/filter work. Add focused regression coverage in `ui/src/ui/views/chat.test.ts` or the nearest existing session-control tests for: target agent has a newer subagent and older dashboard, choose dashboard; target agent has a newer subagent and no dashboard row, choose `agent:<id>:main`; normal recent-dashboard behavior still passes. Use the existing `isSubagentSessionKey` helper rather than ad hoc string parsing if practical. No changelog is needed unless the maintainer wants a user-visible fix note.
