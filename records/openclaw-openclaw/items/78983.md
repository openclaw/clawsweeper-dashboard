## Summary

Keep open: current main still has a source-level path where the Control UI chat session picker includes sub-agent rows and agent switching can prefer the newest session for the target agent, which can be a sub-agent.

## What This Changes

Request that the Control UI avoid surfacing sub-agent worker sessions indistinguishably in the chat session picker and avoid landing on them when switching agents.

## Best Possible Solution

Treat sub-agent worker sessions as non-default or clearly segregated in chat session controls, while preserving current saved dashboard-session behavior and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: put a newer agent:<id>:subagent:* row in sessionsResult and the current picker logic will include it and may choose it as the preferred target-agent session.

## Solution Assessment

No, not as a blanket reset-to-main change. The safer fix is to keep current dashboard-session fallback behavior but exclude or clearly segregate sub-agent worker sessions and never auto-select one during an agent switch.
