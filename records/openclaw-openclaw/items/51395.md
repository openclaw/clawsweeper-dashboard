## Summary

Keep this open: current main and v2026.6.6 still only expose file-input/file-chooser upload mechanics, while the requested ByteUploader/drop-style path would need an explicit Browser plugin API and security/product decision.

## What This Changes

Request an explicit browser upload fallback or mode for custom non-input uploaders such as ByteUploader using DataTransfer, clipboard, or native drag simulation.

## Best Possible Solution

Add an explicit Browser plugin drop-file or paste-file upload mode, keep standard upload behavior unchanged, document the security boundary, and require live custom-uploader proof before landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live ByteUploader reproduction was run; source and dependency inspection show current main lacks the requested drop/paste/native-drag upload mode.

## Solution Assessment

Is this the best way to solve the issue? No for a silent fallback; the safer solution is an explicit browser file-drop or paste action with maintainer-approved security semantics and live proof.
