## Summary

Keep open. Current main still supports standard file chooser and direct file-input paths only; the requested ByteUploader/drop/paste/native-drag behavior is a new browser-tool capability, and the latest FSB reference supports making that path explicit instead of a silent fallback.

## What This Changes

Add a browser upload strategy for non-standard uploader widgets that do not expose a normal file input.

## Best Possible Solution

Introduce an explicit opt-in custom upload/drop-file browser action that preserves upload-root validation, declares target/strategy semantics, and has synthetic dropzone coverage plus docs.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path because the report names service-specific ByteDance surfaces without a public fixture. Source inspection does show current main lacks DataTransfer, paste, or native drag upload support.

## Solution Assessment

Unclear. The requested direction is plausible, but the FSB reference and current API shape point toward an explicit drop-file/custom-uploader tool rather than silently trying fallback mechanisms after ordinary upload calls.
