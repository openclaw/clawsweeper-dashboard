## Summary

Keep open. Current main still supports browser uploads through file chooser arming or direct file-input setting only; it does not define a DataTransfer, paste, or native drag fallback for custom upload widgets. The related https://github.com/openclaw/openclaw/issues/38844 is adjacent stale-ref/timeout work, not a canonical tracker for ByteUploader-style non-standard inputs.

## What This Changes

Request a browser upload fallback for custom/non-standard upload components such as ByteUploader that bypass normal `<input type="file">` handling.

## Best Possible Solution

Add a generic, opt-in browser upload fallback with explicit target and strategy semantics, preserve upload-root validation, cover it with synthetic custom-uploader tests, and document when agents should use it.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path because the report names service-specific ByteDance surfaces without a public fixture. Source inspection does show current main lacks the requested fallback path and remains input/file-chooser oriented.

## Solution Assessment

Unclear. The requested capability is plausible, but the maintainable solution needs a maintainer decision on an explicit upload strategy/target contract rather than silently trying unsafe fallbacks after every file-input upload.
