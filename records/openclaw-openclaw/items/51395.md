## Summary

Keep open: current main and the latest release still expose browser upload as a file chooser or direct file-input path, with no DataTransfer, clipboard paste, native drag, or ByteUploader-style custom uploader support. The request is a plausible browser automation feature, but it needs maintainer product/API and file-injection security decisions before implementation.

## What This Changes

The issue requests browser upload support for non-standard custom uploader widgets such as ByteUploader, using a DataTransfer drop, clipboard paste, or native drag-style path instead of only standard file inputs.

## Best Possible Solution

Add an explicit custom-upload or drop-file browser action in the browser plugin, with target semantics, upload-root validation, synthetic custom-uploader tests, docs, and a security-reviewed boundary instead of a silent fallback behind ordinary upload.
