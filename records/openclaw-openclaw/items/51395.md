## Summary

Keep open: current main still supports only file chooser or direct file-input upload paths, while the requested ByteUploader/drop/paste/native-drag behavior is a new browser automation capability with unresolved API and file-injection safety decisions. The FSB reference linked in the discussion is useful evidence for an explicit drop-file style tool rather than a silent fallback.

## What This Changes

The issue requests browser upload support for custom ByteUploader-style components through DataTransfer, clipboard paste, or native drag simulation when normal file input upload does not attach files.

## Best Possible Solution

Add an explicit opt-in custom upload or drop-file browser action that preserves upload-root validation, declares target and strategy semantics, and includes synthetic dropzone tests plus docs.

## Reproduction Assessment

No high-confidence live reproduction is available because the issue lacks a public ByteUploader fixture or ByteDance test URL. Source inspection does confirm current main lacks DataTransfer, paste, or native drag upload support.

## Solution Assessment

Unclear. The proposed mechanisms are plausible, but the safer maintainable direction is an explicit custom-upload action rather than a silent fallback after ordinary upload calls.
