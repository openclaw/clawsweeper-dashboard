## Summary

Keep open. Current main still limits browser upload to standard file chooser/direct file-input paths, while the requested ByteUploader/DataTransfer/clipboard/native-drag fallback is a new browser-plugin capability with security and product tradeoffs, not a stale or duplicate cleanup item.

## What This Changes

Request adding a browser upload fallback for ByteUploader-style non-standard upload widgets that do not populate `<input type=file>` through the existing file-input path.

## Best Possible Solution

Define a supported browser-plugin upload fallback contract, then implement only the chosen fallback path with path confinement, permission semantics, docs, and regression coverage for both standard inputs and non-standard upload widgets.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a new fallback request. Source inspection confirms current main only arms a Playwright file chooser or calls `setInputFiles`, but I did not live-test ByteUploader.

## Solution Assessment

Unclear. DataTransfer, clipboard, and native CDP drag are plausible directions, but maintainers should first choose the supported semantics rather than silently adding every fallback to the existing upload path.
