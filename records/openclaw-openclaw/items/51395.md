- `impact:security`: The requested behavior would inject local file contents into browser pages through alternate drop/paste events, which is a security-sensitive automation boundary.

## Summary

Keep this open: current main and the latest release still only support standard file input/file chooser upload paths, while the requested custom uploader/drop-style behavior needs an explicit browser-tool API and security/product decision.

## What This Changes

The issue requests a browser upload fallback for custom uploader components such as ByteUploader when standard file input upload does not attach files.

## Best Possible Solution

Define an explicit browser file-drop or paste upload mode in the Browser plugin, keep standard upload behavior unchanged, document the security boundary, and validate it with live custom-uploader proof before landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run against a ByteUploader page; source and upstream dependency inspection show the current upload path is limited to standard file input/file chooser mechanics.

## Solution Assessment

Is this the best way to solve the issue? No for a silent fallback; the safer direction is an explicit drop-file or paste-file browser action with maintainer-approved security semantics and live proof.
