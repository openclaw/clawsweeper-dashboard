## Summary

Keep open: current main and the latest release still expose standard file input/file chooser upload paths only, while the requested drop/paste/native-drag behavior needs an explicit Browser API plus security/product review rather than a silent fallback.

## What This Changes

The issue requests a Browser upload capability for non-standard custom uploaders by simulating DataTransfer/drop, clipboard paste, or native drag behavior instead of relying only on standard file inputs.

## Best Possible Solution

Define an explicit Browser drop-file or paste-file upload mode, keep standard upload semantics unchanged, document the local-file injection boundary, and require live custom-uploader proof before implementation lands.

## Reproduction Assessment

No high-confidence live reproduction was run against a ByteUploader page. Source and dependency inspection do show the current Browser upload path is limited to standard file input/file chooser mechanics and lacks the requested drop, paste, or native-drag mode.

## Solution Assessment

No for a silent fallback: that would blur a security-sensitive local-file injection boundary. The maintainable direction is an explicit drop-file or paste-file browser action with maintainer-approved semantics and live proof.
