## Summary

Keep open: current main and the latest release still expose browser upload as a file chooser or direct file-input path, with no DataTransfer, clipboard paste, native drag, or ByteUploader-style custom uploader support. The FSB comment is useful evidence for an explicit drop-file API shape, but this needs maintainer API/product and file-injection security review before implementation.

## What This Changes

The issue requests a browser upload fallback for ByteUploader-style custom upload widgets that do not accept CDP file-input attachment.

## Best Possible Solution

Add an explicit custom-upload or drop-file browser action in the browser plugin, with target semantics, upload-root validation, synthetic custom-uploader tests, docs, and a security-reviewed boundary instead of a silent fallback behind ordinary upload.

## Reproduction Assessment

No high-confidence live reproduction is available because the issue does not include a public ByteUploader fixture or redacted target page. Source inspection does confirm current main has no DataTransfer, paste, native drag, or custom-uploader upload path.

## Solution Assessment

Unclear: the requested mechanisms are plausible and the FSB reference is useful, but the safer maintainable direction appears to be an explicit custom-upload or drop action rather than a silent fallback inside ordinary upload.
