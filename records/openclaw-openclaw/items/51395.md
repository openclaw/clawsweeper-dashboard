## Summary

Keep open. Current main still supports browser uploads only through file chooser arming, direct file-input setting, or Chrome MCP upload_file; there is no generic DataTransfer, paste, or native drag fallback for custom uploader widgets, and related #38844 tracks stale refs/timeout messaging rather than this missing upload mode.

## What This Changes

Request a browser upload fallback for custom non-file-input uploaders such as ByteUploader using DataTransfer, paste, or native drag simulation.

## Best Possible Solution

Add a generic, explicit browser-plugin upload fallback contract with target/mode semantics, keep existing upload-root validation, cover it with synthetic custom-uploader tests, and document when agents should use it.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this asks for a new upload capability beyond the documented file chooser/input behavior. Source inspection does confirm current main lacks the requested fallback path, but there is no public ByteUploader fixture for live reproduction.

## Solution Assessment

Unclear. The fallback direction is plausible, but the best maintainable fix needs maintainer agreement on automatic fallback versus explicit mode/target semantics and the safety constraints for synthetic file payloads.
