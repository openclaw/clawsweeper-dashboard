## Summary

Keep open. Current main still implements browser upload as file chooser arming or direct file-input setting, and the ByteUploader-style DataTransfer, paste, or native drag fallback remains a new browser-tool capability that needs an explicit API and safety decision.

## What This Changes

Request a browser upload fallback for custom non-standard uploader widgets such as ByteUploader, using DataTransfer, clipboard paste, or native drag simulation when standard file inputs do not work.

## Best Possible Solution

Add a generic, opt-in browser upload fallback with explicit target and strategy semantics, preserve upload-root validation, cover it with synthetic custom-uploader tests, and document when agents should use it.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path because the report names service-specific ByteDance surfaces without a public fixture. Source inspection does show current main lacks the requested fallback path and remains input/file-chooser oriented.

## Solution Assessment

Unclear. The requested capability is plausible, but the maintainable solution needs a maintainer decision on an explicit upload strategy and target contract rather than silently trying fallbacks after every file-input upload.
