## Summary

Keep open. Current main still only supports standard file chooser/file-input upload paths, while the requested ByteUploader-style drop, paste, or native drag fallback is a new browser-tool capability that needs maintainer API and safety decisions.

## What This Changes

Request a browser upload fallback for custom ByteUploader-style uploaders that do not respond to standard file-input or file-chooser handling.

## Best Possible Solution

Add an explicit, opt-in browser upload strategy for custom uploader targets, preserve upload-root validation, cover it with synthetic custom-uploader tests, and document when agents should use it.

## Reproduction Assessment

No, there is not a high-confidence live reproduction path because the report names service-specific ByteDance surfaces without a public fixture. Source inspection does show current main lacks DataTransfer, paste, or native drag upload fallback support.

## Solution Assessment

Unclear. The requested capability is plausible, but the maintainable solution needs an explicit target and strategy contract rather than silently trying fallback mechanisms after every file-input upload.
