## Summary

Keep open: the PR targets a real Control UI upgrade lockout, but the current patch is not merge-ready because the service-worker broadcast can run outside the activate lifetime and before `clients.claim()` has made stale tabs visible to the new worker.

## What This Changes

The PR adds a Control UI service-worker activation message and a browser listener that reloads the page when a new service-worker version reports activation.

## Best Possible Solution

Chain claim, cache cleanup, and update notification inside one activate `waitUntil` path, guard it to real prior-version upgrades, add focused service-worker regression coverage, and provide redacted browser upgrade proof.

## Reproduction Assessment

Yes at source level: stale Control UI bundles can still hit the strict protocol-mismatch path, and the linked issue supplies real cloud symptoms. I did not run a live Zeabur/Chrome stale-service-worker upgrade reproduction in this read-only review.

## Solution Assessment

No, this is only a plausible fix as written. The broadcast should be sequenced inside activation after client claiming and backed by focused coverage plus real browser upgrade proof.
