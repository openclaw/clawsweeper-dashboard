## Best Possible Solution

Repair the activation path with valid `Client.postMessage` usage, include the notification in activation `waitUntil` after claiming or include uncontrolled clients intentionally, add focused coverage, and prove a stale token-authenticated browser reloads once and reconnects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the service-worker Client.postMessage signature:** `ui/public/sw.js:48`
  - body: `Client.postMessage` in a service worker accepts the message plus a transfer list or options object, not a Window-style target origin. Passing `"*"` here can throw during activation before stale tabs receive `sw-updated`, so the recovery this PR adds can fail at the exact upgrade point it is meant to fix.
  - confidence: 0.95
- **[P1] Wait for claimed clients before broadcasting:** `ui/public/sw.js:46-50`
  - body: The new `matchAll`/`postMessage` work is not part of `event.waitUntil` and can run before `clients.claim()` resolves. Since `includeUncontrolled` defaults to false, the stale windows this fix is meant to reload can be missed; sequence the broadcast inside the activation promise after claiming, or deliberately include uncontrolled windows.
  - confidence: 0.91
