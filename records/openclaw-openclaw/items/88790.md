## Summary

Keep open: current main and the latest shipped release still require an explicit Discord `guildId` for message search before any current-channel fallback can help, while the trusted tool context carries channel/provider data but no guild id. The related reports are adjacent Discord message-search/read issues, not a canonical fix for this root cause.

## What This Changes

The issue reports that Discord guild-channel message search fails with `guildId required` when the agent calls `message(action="search")` from an in-channel session without a model-visible guild id.

## Best Possible Solution

Implement a focused Discord plugin fix that derives the current guild safely for in-channel search, keeps the existing read-target authorization checks, and adds regression tests for `message(action="search", query=...)` without an explicit `guildId` from a Discord guild-channel context.

## Reproduction Assessment

Yes for source-level reproduction: on current main, `message(action="search")` without `guildId` reaches a required `guildId` reader before any current-session guild fallback exists, producing the reported `ToolInputError`. I did not run a live Discord session in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow Discord message-action/context repair, not a new config option or a relaxation of guild-scoped Discord search. The adapter already maps `query` to runtime `content`, so the main repair is safe current-guild resolution plus regression coverage.
