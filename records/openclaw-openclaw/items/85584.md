## Summary

Keep open. The updated PR appears to address the prior `toolsAllow: []` review finding and I did not find a blocking correctness issue, but it intentionally changes isolated cron delivery ownership and should get maintainer compatibility sign-off before merge.

## What This Changes

The PR changes isolated cron `none` and `announce` delivery planning so the message tool remains available but is no longer forced, announce uses direct fallback ownership, and prompt guidance honors explicit `toolsAllow` cases with focused tests.

## Best Possible Solution

Keep the direct-fallback-owned cron delivery fix if maintainers agree that routine cron final delivery belongs to the cron runner, with the message tool remaining optional for explicit sends.

## Reproduction Assessment

Yes at source level. Current main constructs cron none/announce source delivery with `messageToolForced: true`, and the executor forwards that policy into the embedded runner; I did not run a live current-main cron in this read-only review.

## Solution Assessment

Yes, with maintainer compatibility sign-off. The PR changes the source-delivery plan at the ownership boundary and keeps same-target message-tool fallback suppression, which is narrower than adding another delivery flag or runtime workaround.
