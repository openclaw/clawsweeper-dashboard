## Summary

Keep open: the delivery-owner direction is plausible, and the PR includes targeted tests plus a live no-delivery cron smoke, but the patch leaves explicit no-tools cron runs with a prompt/tool-surface mismatch after removing forced message-tool insertion.

## What This Changes

The PR changes isolated cron `delivery.mode: none` and `delivery.mode: announce` source delivery so the message tool is no longer forced, and makes announce delivery direct-fallback-owned with updated policy tests.

## Best Possible Solution

Keep cron fallback as the routine delivery owner, but align empty-allowlist prompt/tool behavior and cover that upgrade-sensitive case before merging.

## Reproduction Assessment

Yes at source level: an isolated announce cron with `payload.toolsAllow: []` now loses forced message-tool insertion while `canPromptForMessageTool` still treats the empty allowlist as message-capable. I did not run a live cron in this read-only review.

## Solution Assessment

No, not yet: making cron fallback the delivery owner is a maintainable direction, but the implementation must also align prompt guidance with explicit no-tools allowlists before merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Handle explicit no-tools cron runs before dropping the force:** `src/cron/isolated-agent/run.ts:316`
  - body: This flips announce cron runs to `forceMessageTool: false`, but `canPromptForMessageTool` still treats `toolsAllow: []` as allowed because an empty array satisfies `!params.toolsAllow?.length`. Please align the prompt/tool policy and add a regression for `toolsAllow: []`.
