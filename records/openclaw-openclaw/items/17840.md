- `P2`: The request is a normal-priority cross-channel workflow improvement with real friction but no core-runtime outage.
- `impact:message-loss`: Accepted reactions can be queued as context without waking the agent, so intended operator inputs remain unprocessed until another trigger arrives.

## Summary

Keep open. Current main and v2026.6.5 still expose reaction notifications as passive queued system events, while the remaining implementation attempts are channel-specific and not merged as a cross-channel contract.

## What This Changes

The issue requests an opt-in mechanism for accepted reaction events to wake an agent turn instead of waiting as passive system-event context.

## Best Possible Solution

Add one default-passive reaction or notification wake policy through the chosen plugin/config contract, then wire reaction-capable bundled plugins to request a routed heartbeat only after existing filters and successful enqueue.

## Reproduction Assessment

Yes, at source level. Current Slack, Discord, Telegram, Signal, and Matrix reaction handlers enqueue accepted reactions without requesting a heartbeat wake, and the searched opt-in wake contract names are absent from current main.

## Solution Assessment

No, not as independent channel patches. The maintainable path is a shared default-passive opt-in policy, with each channel preserving its existing authorization and notification filters before waking.
