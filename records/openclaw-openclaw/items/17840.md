## Summary

Keep open: current main and the latest release still only queue accepted reaction notifications as passive system events, while the open implementation work is unmerged and still needs a maintainer product/API decision.

## What This Changes

Request an opt-in contract that lets accepted reaction notifications wake the routed agent session instead of staying passive until a later turn.

## Best Possible Solution

Keep this as the canonical cross-channel product/API request, choose one default-off reaction wake contract, then land a single coherent implementation with schema, docs, and focused reaction-capable plugin tests.

## Reproduction Assessment

Yes. Source inspection on current main shows reaction handlers enqueue accepted reaction system events but do not request a heartbeat wake; no live transport run was needed to prove the missing opt-in contract.

## Solution Assessment

Unclear until maintainers choose the product contract. A shared default-off queue/wake/off policy looks safer than an unconditional one-channel wake, but the final API shape is still a maintainer decision.
