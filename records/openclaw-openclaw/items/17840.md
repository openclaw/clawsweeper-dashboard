## Summary

Keep open: current main and the latest release still treat accepted reactions as queued system events, while the requested opt-in reaction wake policy remains unmerged and needs a maintainer-approved public config/API shape.

## What This Changes

The issue requests a default-off mechanism for reaction events to wake an agent turn for reaction-driven workflows.

## Best Possible Solution

Land one maintainer-approved, default-off shared reaction wake policy that preserves queue-by-default behavior, applies only after existing channel filters, updates schemas/docs/tests, and reconciles the open implementation PRs.

## Reproduction Assessment

Yes, at source level: current reaction handlers enqueue accepted reaction system events, and targeted search found no reaction wake policy or heartbeat request in the inspected paths. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A shared default-off queue/wake/off policy after existing filters looks safer than unconditional or one-channel wake behavior, but that direction still needs product approval.
