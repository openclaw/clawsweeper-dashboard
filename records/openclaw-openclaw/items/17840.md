## Summary

Keep open. Current main still has notification-only reaction handling and no shared opt-in reaction-triggered turn contract, while the open Discord and Telegram implementation attempts are narrower, unmerged paths rather than a settled cross-channel product/API decision.

## What This Changes

Request an opt-in mechanism so accepted reaction events can wake an agent turn instead of remaining passive queued system events.

## Best Possible Solution

Define a default-off shared reaction-trigger contract, then wake only the routed session after existing reaction filters accept and enqueue the event, with schema, generated metadata, public docs, and focused channel tests updated consistently.

## Reproduction Assessment

Yes, by source inspection: accepted reaction handlers enqueue system events while the wake API remains separate on current main. This proves the requested opt-in behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public API. The opt-in direction is coherent, but the maintainable fix should settle shared versus channel-specific semantics before wiring reaction handlers to heartbeat wakes.
