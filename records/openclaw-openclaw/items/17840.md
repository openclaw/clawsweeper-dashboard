## Summary

Keep open. Current main and v2026.6.6 still provide notification-only reaction handling: accepted reactions are filtered and queued as system events, but there is no documented opt-in reaction wake contract, and the open implementation candidates are not merged or settled as the cross-channel product direction.

## What This Changes

The issue requests an opt-in mechanism for accepted reaction events to trigger agent turns instead of only being queued as passive system events.

## Best Possible Solution

Keep this as the canonical cross-channel product/API tracker until maintainers choose and land a default-safe opt-in reaction wake contract that preserves passive reaction notifications by default and reconciles the open implementation candidates.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted Discord, Slack, Telegram, Signal, Matrix, and iMessage reactions are filtered and queued as system events, while heartbeat wake remains a separate API with no reaction-triggered opt-in contract.

## Solution Assessment

Unclear until maintainers choose the product/API shape. A shared queue/wake/off notification policy looks cleaner than one-off channel wake calls, but that direction needs explicit approval because it adds config and possibly SDK surface.
