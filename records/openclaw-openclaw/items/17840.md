## Summary

Keep open. Current main still has passive reaction notification/system-event handling, while the requested default-off reaction-triggered wake/turn contract is not implemented and recent discussion is asking for maintainer API direction.

## What This Changes

Request a default-off opt-in mechanism so accepted reaction events can wake/start agent turns for lightweight confirm/cancel, polling, and workflow actions.

## Best Possible Solution

Add a default-off shared reaction turn trigger contract for reaction-capable channels, then wake only the routed session after existing authorization and reaction notification filters accept the event.

## Reproduction Assessment

Yes, by source inspection: enable an existing `reactionNotifications` mode and deliver an accepted reaction; current handlers enqueue a routed system event but do not call `requestHeartbeat` or `requestHeartbeatNow`. I did not run a live channel event.

## Solution Assessment

Unclear. The default-off opt-in direction is reasonable, but maintainers still need to choose the public contract and whether this lands cross-channel first or starts with a narrower channel implementation.
