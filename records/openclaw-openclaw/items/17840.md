## Summary

Keep open. Current main still treats accepted reactions as notification-only system events, with no opt-in config, per-message flag, or bundled cross-channel wake path, and the latest discussion is still asking maintainers to choose the product/API contract.

## What This Changes

Request an opt-in reaction-triggered turn contract so accepted channel reactions can wake an agent for lightweight confirm, cancel, and polling workflows.

## Best Possible Solution

Define a default-off shared reaction-trigger contract, then wake only the routed session after existing reaction filters accept the event, with schema, generated metadata, public docs, and focused channel tests updated consistently.

## Reproduction Assessment

Yes, by source inspection: accepted reaction handlers enqueue system events and the wake API remains separate. This proves the requested opt-in behavior is absent on current main, though the item is a feature/API request rather than a broken existing public contract.

## Solution Assessment

Unclear until maintainers choose the public API. The opt-in direction is coherent, but the maintainable solution should settle shared versus channel-specific semantics before wiring reaction handlers to heartbeat wakes.
