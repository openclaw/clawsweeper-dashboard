## Summary

Keep open. Current main and v2026.5.20 still expose reaction notifications as passive queued system events, while the opt-in reaction wake contract is only present in open PR work and still needs maintainer product/API selection.

## What This Changes

The issue requests a default-off mechanism for reaction events to wake the agent and trigger turns for reaction-driven workflows.

## Best Possible Solution

Adopt or supersede the open shared policy work with one default-off reaction wake contract that wakes only authorized, accepted reactions after successful enqueue and updates schemas, docs, generated metadata, SDK contracts, and focused channel tests.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, and Slack reaction handlers enqueue system events while the heartbeat wake API remains separate and no wake config exists on current main. This proves the requested behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but it still needs review, replacement, or explicit product approval before this issue can close.
