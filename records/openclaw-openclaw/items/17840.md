## Summary

Keep open. Current main still has reaction notification/system-event delivery only, with no opt-in wake/turn contract, and the discussion already treats this as the canonical cross-channel product/API request.

## What This Changes

Add a default-off opt-in mechanism that lets accepted channel reaction events wake an agent turn instead of remaining passive system events.

## Best Possible Solution

Add an additive, default-off reaction turn trigger contract in shared config or the message API, then wire accepted bundled-channel reactions to requestHeartbeat only after existing access and reaction-notification filters pass.

## Reproduction Assessment

Yes, by source inspection: enable an existing reactionNotifications mode and deliver an accepted reaction; handlers enqueue a system event and do not call requestHeartbeat/requestHeartbeatNow. I did not run a live channel reproduction, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. The default-off opt-in direction is sound, but maintainers still need to choose the public contract; unconditional wake-on-reaction would not be the narrowest maintainable fix.
