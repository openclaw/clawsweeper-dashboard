- `impact:message-loss`: Reaction-only user inputs are queued as context without waking the agent, so actionable reaction signals can be missed until a later text message or heartbeat.

## Summary

Keep open. Current main and the latest release still expose reaction notifications as passive queued system events, while the opt-in wake policy remains only in open PR work and needs a maintainer API/product decision before this issue can close.

## What This Changes

Request an opt-in mechanism for reaction events to wake the agent and start a turn after existing channel reaction filters accept the event.

## Best Possible Solution

Land or supersede the open PRs with one default-off shared reaction wake policy that wakes only authorized, accepted reaction inputs after successful enqueue and updates schemas, docs, generated metadata, and focused channel tests.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, and Slack reaction handlers enqueue system events while the wake API remains separate on current main. This proves the requested opt-in behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The default-off shared policy direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but it still needs review or replacement before this issue can close.
