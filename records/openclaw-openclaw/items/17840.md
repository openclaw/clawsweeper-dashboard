## Label Justifications

- `impact:message-loss`: Actionable reaction-only user input can remain queued as passive context without waking the agent until another message or heartbeat occurs.

## Summary

Keep open. Current main and v2026.6.5 still treat accepted reaction notifications as queued system events, while the opt-in wake policy exists only in open PR work that still needs maintainer product/API selection.

## What This Changes

Request a default-off opt-in mechanism that lets accepted reaction events wake an agent turn instead of remaining passive system events.

## Best Possible Solution

Adopt or supersede the open reaction-wake PR work with one shared default-off contract that wakes only authorized, accepted reaction events after successful enqueue, with schemas, generated metadata, docs, SDK/runtime contracts if needed, and focused channel tests kept in sync.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, Slack, Matrix, Signal, and iMessage reaction paths enqueue system events while the heartbeat wake API remains separate and no wake config exists on current main. This proves the requested behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 and https://github.com/openclaw/openclaw/pull/83169 looks maintainable, while a one-channel wake without shared config semantics would be too narrow.
