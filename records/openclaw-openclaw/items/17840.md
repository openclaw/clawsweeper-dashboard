## Summary

Keep open. Current main still treats accepted reactions as notification/system-event inputs, while wake APIs exist separately; this remains a cross-channel opt-in feature/API request and maintainer discussion already treats it as canonical rather than superseded.

## What This Changes

Request an opt-in mechanism, such as a reactionTriggersTurn config mode or awaitReaction message flag, so accepted user reactions can start agent turns.

## Best Possible Solution

Add a default-off reaction turn trigger contract shared by reaction-capable channels, then call requestHeartbeat only after existing authorization and reaction-notification filters accept the event; update schemas, docs, and focused channel tests.

## Reproduction Assessment

Yes, by source inspection: enable an existing reactionNotifications mode and deliver an accepted reaction; the handlers enqueue a system event and do not call requestHeartbeat/requestHeartbeatNow. I did not run a live channel event, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. The default-off opt-in direction is reasonable, but maintainers still need to choose the public contract and scope; unconditional wake-on-reaction would not be the narrowest maintainable solution.
