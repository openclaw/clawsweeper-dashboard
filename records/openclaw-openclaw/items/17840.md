## Summary

Keep open: current main and v2026.6.6 still expose accepted reactions as queued system events without a documented opt-in wake/turn contract, while the overlapping implementation PRs are still open and require a maintainer product choice.

## What This Changes

Request a default-safe opt-in mechanism that lets accepted reaction events wake the routed agent turn instead of only enqueueing passive system events.

## Best Possible Solution

Land one maintainer-approved, default-safe notification-wake contract that preserves passive behavior by default, wakes only after accepted reaction enqueue succeeds, reconciles the open PRs, and updates config schemas, generated metadata, channel docs, and focused reaction-capable channel tests.

## Reproduction Assessment

Yes, at source level. Current main reaction handlers for Discord, Telegram, Slack, Signal, and Matrix enqueue accepted reaction system events, while the inspected config/docs/runtime paths do not expose a reaction-triggered wake contract.

## Solution Assessment

Unclear until maintainers choose the product/API shape. A shared default-safe queue/wake/off policy appears more maintainable than one-off channel wake behavior, but that needs explicit maintainer approval because it adds config and agent-triggering semantics.
