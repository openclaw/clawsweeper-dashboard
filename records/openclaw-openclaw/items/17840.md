## Summary

Keep open. Current main still has passive reaction notification/system-event handling, no opt-in wake/turn contract, and the latest human comment is explicitly waiting on maintainer API direction.

## What This Changes

Request an opt-in per-message or config-level mechanism that lets accepted reaction events wake the routed agent turn instead of waiting for the next text message or heartbeat.

## Best Possible Solution

Define a default-off reaction turn trigger contract, then wake only the routed session after existing reaction filters accept the event, with config schema, generated metadata, channel docs, and focused tests updated consistently.

## Reproduction Assessment

Yes for the current passive-behavior gap by source inspection: accepted reaction paths enqueue system events and docs describe system-event delivery, but current main has no trigger config or heartbeat call. A live multi-channel reproduction was not run, and the requested opt-in wake mode is a new feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the public API. The opt-in direction is reasonable, but an automated one-channel shape would risk fragmenting the cross-channel contract while narrower Telegram work is already tracked separately.
