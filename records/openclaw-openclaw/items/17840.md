## Summary

Keep open: current main and v2026.6.10 still expose reaction notifications as passive queued system events, while the requested opt-in wake policy remains unmerged product/API work. The broad candidate at https://github.com/openclaw/openclaw/pull/83611 and the narrower Discord candidate at https://github.com/openclaw/openclaw/pull/83169 should be reviewed rather than closing this canonical issue early.

## What This Changes

The issue requests an opt-in mechanism for reactions to wake an agent turn instead of remaining passive queued system events.

## Best Possible Solution

Land one maintainer-approved, default-off shared reaction wake policy that preserves queue-by-default behavior, applies only after existing channel filters, and updates schemas, generated metadata, docs, and focused channel tests.

## Reproduction Assessment

Yes, at source level. Current Discord, Telegram, Slack, and Matrix reaction paths enqueue system events, and targeted search found no current-main reaction wake policy or heartbeat request in those inspected reaction handlers; no live channel repro was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A shared default-off queue/wake/off policy after existing filters looks safer than unconditional or one-channel wake behavior, and https://github.com/openclaw/openclaw/pull/83611 is the broad candidate to review.
