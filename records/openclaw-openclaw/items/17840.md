## Summary

Keep open. Current main still has notification-only reaction handling for the main bundled reaction paths, while the requested opt-in wake contract remains a cross-channel product/API decision rather than a stale or implemented cleanup item.

## What This Changes

The issue requests an opt-in way for emoji reactions to wake an agent turn instead of remaining passive system events.

## Best Possible Solution

Add an additive, default-off reaction turn trigger contract in shared config/metadata or the message API, then wire accepted bundled-channel reactions to the heartbeat wake primitive only after existing access and notification checks pass.

## Reproduction Assessment

Yes. A high-confidence source-level path is to enable an existing `reactionNotifications` mode and deliver an accepted reaction: the inspected Slack, Discord, Telegram, Signal, Matrix, and BlueBubbles handlers enqueue a system event but do not request a heartbeat wake.

## Solution Assessment

Unclear. The default-off opt-in direction is sound, but maintainers still need to choose the contract shape; an unconditional wake on every reaction would not be the safest or narrowest solution.
