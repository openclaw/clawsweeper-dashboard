## Summary

Keep open. Current main and the latest release still expose reaction notifications as passive queued system events, while the requested default-off reaction wake/turn policy remains unmerged and tied to maintainer product/API selection.

## What This Changes

The issue requests an opt-in mechanism so accepted reaction events can wake an agent turn for quick confirm/cancel, emoji-choice, and reaction-driven workflow patterns.

## Best Possible Solution

Adopt or supersede a shared default-off reaction wake contract, then wire accepted reaction events through it after existing channel filters while keeping schemas, generated metadata, docs, and focused channel tests aligned.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, Slack, and Matrix reaction paths enqueue system events while current main has no opt-in reaction wake config or handler-level heartbeat call. This proves the requested behavior is absent, although it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but this issue should stay open until that path is reviewed, landed, or explicitly superseded.
