## Label Justifications

- `impact:message-loss`: Reaction-only user inputs can be queued as context without waking the agent, so actionable reaction signals may be missed until another turn or heartbeat occurs.

## Summary

Keep open. Current main still has passive reaction notification paths with no default-off shared reaction wake policy, while the active implementation work remains in open PRs that need maintainer API/product review before this issue can close.

## What This Changes

The issue requests an opt-in cross-channel mechanism so accepted reaction events can wake an agent turn instead of only being queued as passive system events.

## Best Possible Solution

Land or supersede the open work with one default-off shared reaction wake policy that runs only after existing reaction notification and authorization gates pass, then update schemas, docs, generated metadata, and focused tests for the reaction-capable bundled plugins.

## Reproduction Assessment

Yes, at source level. Current main shows accepted Discord, Telegram, Slack, Matrix, iMessage, and Teams reaction paths enqueue system events while the wake API remains separate, so the requested opt-in wake behavior is absent.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off `notificationWake.reactions` direction in https://github.com/openclaw/openclaw/pull/83611 looks like the most maintainable shape, but it is still open and needs review/proof before this issue can close.
