## Label Justifications

- `P2`: This is a normal-priority cross-channel enhancement/API gap with clear workflow friction but no core runtime outage.
- `impact:message-loss`: Reaction-only user inputs are queued as context without waking the agent, so actionable reaction signals can be effectively missed until a later text message or heartbeat.

## Summary

Keep open. Current main still treats accepted reaction notifications as queued system events, and the shared opt-in wake policy is only present in open PR work rather than in `main`, so this remains the canonical cross-channel product/API request.

## What This Changes

Request an opt-in mechanism that lets accepted reaction events trigger agent turns instead of remaining passive system events.

## Best Possible Solution

Land a default-off shared reaction wake policy, or an equivalent maintainer-approved contract, that wakes only the routed session after accepted reaction notification enqueue succeeds and updates schema metadata, docs, and focused channel tests.

## Reproduction Assessment

Yes, by source inspection: accepted reaction handlers enqueue system events while the wake API remains separate on current main. This proves the requested opt-in behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The default-off opt-in direction is maintainable, and https://github.com/openclaw/openclaw/pull/83611 appears to be the closest current implementation path, but it still needs review or landing before this issue can close.
