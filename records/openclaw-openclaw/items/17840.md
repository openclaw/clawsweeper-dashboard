## Summary

Keep open: current main and the latest release still only queue accepted reaction notifications as passive system events, while the requested opt-in wake/turn policy is unmerged product/API work with open implementation candidates.

## What This Changes

The issue requests an opt-in policy that lets accepted reaction events wake the agent and start a turn for low-friction reaction-driven workflows.

## Best Possible Solution

Land one maintainer-approved, default-off shared reaction wake policy that preserves passive defaults, applies only after existing channel filters, and updates schemas, docs, and tests consistently.

## Reproduction Assessment

Yes at source level: current Discord, Slack, Telegram, Signal, Matrix, and iMessage reaction paths enqueue system events, and targeted search found no merged reaction wake policy or heartbeat call in those reaction handlers. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A shared default-off policy after existing filters looks safer than an unconditional or one-channel wake behavior, but that direction still needs product approval.
