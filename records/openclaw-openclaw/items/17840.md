- `impact:message-loss`: Actionable reaction-only inputs are queued as context without waking the agent, so the signal can be missed until another text message or heartbeat.

## Summary

Keep open. Current main still exposes reaction notifications as passive queued system events, while the requested opt-in reaction wake contract is only present in open PR work and needs a maintainer product/API decision.

## What This Changes

Request an opt-in per-message or config-level mechanism that lets accepted reaction events wake the routed agent session for reaction-driven workflows.

## Best Possible Solution

Adopt or supersede the open policy PRs with one default-off shared reaction wake contract that wakes only authorized, accepted reactions after successful enqueue and updates schemas, docs, generated metadata, and focused channel tests.

## Reproduction Assessment

Yes. By source inspection, accepted Discord, Telegram, Slack, Signal, Matrix, and iMessage reaction paths enqueue system events while the heartbeat wake API remains separate; this proves the requested opt-in behavior is absent on current main.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but it still needs review, replacement, or explicit product approval before this issue can close.
