## Summary

Keep open. Current main still has passive reaction notification/system-event behavior and no merged opt-in wake contract; the live related PRs are implementation candidates, not merged replacements for this cross-channel product/API request.

## What This Changes

The issue requests a default-off mechanism for accepted reaction events to trigger agent turns after existing channel authorization and reaction-notification filters pass.

## Best Possible Solution

Keep this issue as the canonical cross-channel tracker, review the shared default-off notification wake direction, and land or revise it only after maintainers accept the config/API contract and compatibility story.

## Reproduction Assessment

Yes at source level: current main accepts and enqueues reaction events in representative Slack, Discord, Telegram, Matrix, Signal, and iMessage paths, but the searched wake option names are absent and inspected reaction handlers do not request heartbeat. I did not run live channel E2E because this is a feature/product contract request rather than a broken existing documented behavior.

## Solution Assessment

Unclear until maintainers choose the product/API shape. A shared default-off notification wake policy that preserves current passive defaults looks safer than scattered channel-specific options, but the current candidates still need maintainer review before this issue can close.
