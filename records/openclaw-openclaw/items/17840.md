## Summary

Keep open. Current main still only records accepted reactions as passive system events, while the wake APIs are separate and no opt-in reaction-triggered turn contract exists; recent discussion is asking for maintainer API direction, not cleanup.

## What This Changes

Request an opt-in mechanism so accepted message reactions can wake the routed agent session for lightweight confirm/cancel, polling, and emoji-command workflows.

## Best Possible Solution

Add a default-off reaction turn trigger contract shared by reaction-capable channels, then wake the routed session only after existing authorization and notification filters accept the reaction.

## Reproduction Assessment

Yes, by source inspection: enable an existing reactionNotifications mode and deliver an accepted reaction; current handlers enqueue a routed system event but do not call requestHeartbeat/requestHeartbeatNow. I did not run a live channel event.

## Solution Assessment

Unclear. The default-off opt-in direction is reasonable, but maintainers still need to choose the public contract and rollout scope before implementation.
