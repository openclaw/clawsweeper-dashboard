## Summary

Keep open. Current main still treats most bundled reaction ingress as notification-only system events; the heartbeat wake primitive exists, but there is no opt-in config or per-message contract that turns accepted reactions into agent turns across the requested channels. This is not a ClawHub close because the missing piece is a shared bundled-channel/config/runtime behavior decision, not an optional external integration.

## What This Changes

Request an opt-in mechanism so accepted user reactions can wake the agent and start an agent turn for lightweight confirm/cancel, polling, and reaction-driven workflows.

## Best Possible Solution

Keep this as the canonical cross-channel request. Maintainers should choose a backwards-compatible opt-in contract, preserve passive reaction notifications by default, and wire accepted reaction notifications to runtime.system.requestHeartbeatNow only after existing access-control and reactionNotification filters pass, with schema, generated metadata, docs, and focused channel tests updated together.
