## Summary

Keep open: current main and the latest release still treat accepted reactions as queued system events, while the requested opt-in wake/turn policy remains unmerged product/API work with open implementation candidates.

## What This Changes

Request an opt-in config or per-message mechanism that lets accepted reaction notifications wake a routed agent turn instead of remaining passive system events.

## Best Possible Solution

Land one maintainer-approved, default-off shared reaction wake policy that preserves queue-by-default behavior, applies only after existing channel filters, updates schemas/docs/tests, and reconciles the open implementation PRs.

## Reproduction Assessment

Yes at source level: current Discord, Slack, Telegram, Signal, and Matrix reaction paths enqueue system events, and targeted search found no reaction wake policy or heartbeat request in the inspected reaction handlers. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A shared default-off queue/wake/off policy after existing filters looks safer than an unconditional or one-channel wake behavior, but that direction still needs product approval.
