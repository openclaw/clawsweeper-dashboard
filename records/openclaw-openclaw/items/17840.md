## Label Justifications

- `P2`: This is a normal-priority enhancement/API gap with clear user friction across reaction-capable channels but no core runtime outage.
- `impact:message-loss`: The issue is about reaction-only user inputs being queued as context without waking the agent, so the actionable message signal can be effectively missed until a later text message or heartbeat.

## Summary

Keep open. Current main still has notification-only reaction handling with no shared opt-in reaction-triggered turn contract, and the open Discord/Telegram PRs are narrower unmerged attempts rather than a settled cross-channel API decision.

## What This Changes

The issue requests a default-off mechanism for accepted reaction events to wake the agent and start a turn across reaction-capable channels.

## Best Possible Solution

Define a default-off shared reaction-trigger contract, then wake only the routed session after existing reaction filters accept and enqueue the event, with schema, generated metadata, public docs, and focused channel tests updated consistently.

## Reproduction Assessment

Yes, by source inspection: accepted reaction handlers enqueue system events while the wake API remains separate on current main. This proves the requested opt-in behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public API. The default-off opt-in direction is coherent, but the maintainable fix should settle shared versus channel-specific semantics before wiring reaction handlers to heartbeat wakes.
