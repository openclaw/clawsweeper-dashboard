## Summary

Keep open: current main and v2026.6.6 still only enqueue accepted reaction notifications as system events, while the requested opt-in wake/turn contract is being handled by open implementation candidates that are not merged yet.

## What This Changes

The issue requests an opt-in per-message or configuration-level mechanism for accepted reactions to wake the routed agent turn.

## Best Possible Solution

Land or reject one maintainer-approved shared opt-in reaction wake policy that preserves queued-only behavior by default, wakes only after existing reaction filters and successful enqueue, updates generated config/docs/tests, and then closes this issue with that provenance.

## Reproduction Assessment

Yes at source level. Current main accepts reaction events in multiple channel handlers, queues system events for the routed session, and has no reaction wake policy or heartbeat request in those inspected paths; I did not run a live channel account in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. A shared queue/wake/off notification policy looks more maintainable than unconditional channel-specific wake behavior, but the shared policy is still an open PR and not current main.
