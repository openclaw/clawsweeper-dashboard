## Summary

Keep open. Current main and the latest release still expose reaction notifications as passive queued system events, while the default-off reaction wake contract exists only in open PR work that still needs maintainer product/API selection.

## What This Changes

Request an opt-in cross-channel mechanism that lets accepted reaction events wake the agent and start a turn for reaction-driven workflows.

## Best Possible Solution

Adopt or supersede the open reaction-wake PR work with one default-off shared contract that wakes only authorized, accepted reaction events after successful enqueue, with schemas, generated metadata, docs, SDK/runtime contracts if needed, and focused channel tests kept in sync.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, and Slack reaction handlers enqueue system events while the heartbeat wake API remains separate and no wake config exists on current main. This proves the requested behavior is absent, though it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but it still needs review, replacement, or explicit product approval before this issue can close.
