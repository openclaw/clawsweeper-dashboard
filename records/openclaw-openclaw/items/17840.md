## Summary

Keep open: current main and v2026.6.6 still expose reaction notifications as queued system events without an opt-in wake/turn contract, and the related implementation PRs are still open, conflicting, and not a complete landed cross-channel answer.

## What This Changes

The issue requests an opt-in mechanism for reaction events to wake an agent turn across reaction-capable channels while preserving passive reaction notifications by default.

## Best Possible Solution

Choose one maintainer-approved opt-in reaction wake contract, preserve queued-only behavior by default, and land it with docs, generated config metadata, and focused tests across the intended reaction-capable channels.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: accepted reaction handlers enqueue routed system events and targeted current-main searches show no opt-in wake contract or heartbeat call in those reaction paths.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the API; a shared opt-in queue/wake/off policy or narrowly scoped per-message flag is safer than unconditional or one-channel wake behavior.
