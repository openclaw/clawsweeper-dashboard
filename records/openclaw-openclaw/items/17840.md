## Summary

Keep open. Current main still treats accepted reaction notifications as queued system events and does not provide an opt-in config, per-message flag, or bundled channel path that wakes the agent for a new turn; the recent maintainer discussion also identifies this as the canonical cross-channel request rather than a duplicate of narrower reaction issues.

## What This Changes

The issue requests an opt-in configuration or per-message contract that lets accepted reaction events wake the agent and start a turn instead of remaining passive queued system events.

## Best Possible Solution

Add an additive, default-off reaction turn trigger contract in shared config/metadata or the message API, then wire accepted bundled-channel reaction events to `runtime.system.requestHeartbeatNow` only after authorization and notification-mode checks pass, with docs and focused tests across the reaction-capable plugins.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to enable an existing `reactionNotifications` mode, deliver an accepted reaction, and observe that the handler enqueues a system event but has no wake call; the current tests assert enqueue behavior only. I did not run live channel tests because this was a read-only review.

## Solution Assessment

Unclear. The opt-in direction is sound, but maintainers still need to choose the narrow contract shape; an unconditional wake on every reaction would not be the safest or most maintainable fix.
