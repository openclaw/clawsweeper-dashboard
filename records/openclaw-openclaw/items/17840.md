## Summary

Keep open: current main and the latest release still expose accepted reactions as passive system events, while the requested opt-in wake/turn contract remains unmerged and needs maintainer product/API direction.

## What This Changes

Request an opt-in mechanism for accepted reaction events to wake the agent and start a turn for lightweight reaction-driven workflows.

## Best Possible Solution

Review and land one maintainer-approved shared opt-in reaction wake policy, with default passive behavior preserved and each channel waking only after existing reaction notification and access filters pass.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current reaction handlers enqueue accepted reaction system events, and focused searches over those handlers show no heartbeat wake call or documented opt-in turn trigger.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the API. A shared queue/wake/off notification policy looks safer than unconditional channel-specific wake behavior, but the exact config and per-message contract are product decisions.
