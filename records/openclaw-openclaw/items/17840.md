- `impact:message-loss`: A user's reaction intent reaches passive session context but is effectively suppressed until another event wakes the agent.

## Summary

The feature is still missing on current main and the latest release: supported channels expose reaction notification filters and enqueue accepted reactions as system events, but no opt-in wake/turn policy connects those events to the separate heartbeat API. The open Telegram-only PR is conflicting and does not supersede this cross-channel product decision.

## What This Changes

Add an opt-in mechanism for accepted message reactions to wake the relevant agent session and start a turn.

## Best Possible Solution

Define one default-off, plugin-owned reaction wake policy with shared semantics, then implement supported bundled channels only after their existing authorization and notification filters accept the reaction and resolve the canonical session.

## Reproduction Assessment

Yes from current source: accepted reactions are enqueued as system events while heartbeat wake is a separate API, so a reaction alone has no defined immediate-turn behavior. This is source-reproducible rather than a failing current-main bug because passive delivery is the established behavior.

## Solution Assessment

No proposed variant is yet established as the best fix. A shared, default-off plugin contract is safer than landing Discord- or Telegram-specific policy that would create config and behavior drift.
