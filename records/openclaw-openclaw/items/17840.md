## Summary

Keep open. Current main still has notification-only reaction handling: accepted reactions enqueue system events, but there is no opt-in config, per-message flag, or bundled channel path that wakes an agent turn, and recent discussion is still asking for a product/API direction.

## What This Changes

Request a default-off mechanism for accepted reaction events to wake the appropriate agent session for reaction-driven workflows.

## Best Possible Solution

Define a default-off shared reaction-trigger contract, then wake only the routed session after existing reaction filters accept the event, with schema, generated metadata, public docs, and focused channel tests updated consistently.

## Reproduction Assessment

Yes, by source inspection: accepted reaction handlers enqueue system events and the wake API remains separate. This proves the requested opt-in behavior is not present, but the item is a feature/API request rather than a broken existing public contract.

## Solution Assessment

Unclear until maintainers choose the public API. The opt-in direction is reasonable, but the maintainable solution should settle shared versus channel-specific semantics before wiring reaction handlers to heartbeat wakes.
