## Summary

Keep open: current main and v2026.6.6 still expose reaction notifications as passive queued system events, with no opt-in config, per-message flag, or channel reaction path that wakes an agent turn.

## What This Changes

Request an opt-in cross-channel mechanism, such as a config scope or per-message flag, that lets accepted reaction events trigger agent turns.

## Best Possible Solution

Choose a default-off shared reaction wake contract, then implement it after existing reaction filters across the relevant bundled channels with aligned schemas, docs, generated config metadata, and focused tests.

## Reproduction Assessment

Yes, at source level: current reaction handlers enqueue accepted reactions as system events, and the config/docs surface lacks an opt-in wake policy. I did not run a live channel reaction repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A default-off shared wake policy appears directionally best, while the Discord-first contributor proposal needs a product answer because this issue is explicitly cross-channel.
