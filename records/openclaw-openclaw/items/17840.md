## Summary

Keep open. Current main and the latest release still treat accepted reaction notifications as queued system events, while the opt-in wake/turn policy remains unmerged in related PRs and still needs maintainer product/API selection.

## What This Changes

Request an opt-in mechanism for reaction events to wake the agent and start a turn after existing channel reaction filters accept the event.

## Best Possible Solution

Adopt or supersede the shared default-off notification wake contract so accepted reaction events queue by default, wake only when explicitly configured, and keep schemas, generated metadata, docs, and focused channel tests aligned.

## Reproduction Assessment

Yes, by source inspection: accepted Discord, Telegram, Slack, and Signal reaction paths enqueue system events while current main has no opt-in reaction wake config or handler-level heartbeat call. This proves the requested behavior is absent, although it is a feature/API gap rather than a broken documented default.

## Solution Assessment

Unclear until maintainers choose the public contract. The shared default-off notification wake direction in https://github.com/openclaw/openclaw/pull/83611 looks maintainable, but the issue should stay open until that path is reviewed, landed, or explicitly superseded.
