## Label Justifications

- `P2`: This is a recurring cross-channel interaction limitation with meaningful workflow impact but no core-runtime outage.
- `impact:message-loss`: Accepted reaction inputs can remain dormant until another message or heartbeat, so intended operator actions are effectively not delivered for processing.

## Summary

Current main and the latest release still treat accepted reactions as passive queued system events rather than opt-in turn triggers. The cross-channel request remains necessary and is not superseded: the shared-policy proposal closed unmerged, while the surviving implementation PR is Telegram-only and depends on an unresolved product contract.

## What This Changes

Add an opt-in contract so accepted reaction events can wake the relevant agent session and start an agent turn.

## Best Possible Solution

Add one default-passive notification wake policy through the plugin and config contracts, then have reaction-capable bundled plugins request a routed heartbeat only after authorization, reaction filtering, and successful event enqueue.

## Reproduction Assessment

Yes, at source level. Current Telegram, Slack, and Discord handlers enqueue accepted reaction events for a resolved session but do not request a wake, so processing waits for another trigger.

## Solution Assessment

Not yet. The desired opt-in behavior is sound, but a shared default-passive notification wake contract is narrower and more maintainable than independent channel flags, synthetic message turns, or a per-message API added before cross-channel semantics are chosen.
