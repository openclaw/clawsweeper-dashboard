## Summary

Keep open. Current main and the latest release still expose reaction notifications as passive queued system events, while the requested default-off reaction wake/turn contract remains a product/API decision.

## What This Changes

The issue requests an opt-in mechanism for reaction events to wake the agent and start a turn.

## Best Possible Solution

Keep this as the canonical cross-channel request; a maintainer should choose a default-off opt-in contract, then wire accepted reaction events to `requestHeartbeat` after existing access and notification filters with schema, docs, and focused channel tests.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: enable reaction notifications, send an accepted reaction, and the channel handler queues a system event without requesting a heartbeat. I did not run a live channel repro because the code path is direct across multiple bundled reaction handlers.

## Solution Assessment

Unclear until maintainers choose the API shape. The best implementation should be a deliberate default-off contract shared across reaction-capable channels, not an ad hoc wake call in one handler.
