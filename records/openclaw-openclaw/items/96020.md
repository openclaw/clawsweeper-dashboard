- `P2`: The issue affects cross-session delivery semantics for agent workflows, but it is not a crash, security issue, or release blocker.

## Summary

Keep open: current main and the latest release still return static `delivery.status: "pending"` for normal `sessions_send` A2A announce delivery, but the right fix is a contract decision because tests currently assert that shape and true delivered/read receipts would expand session-delivery semantics.

## What This Changes

The issue asks for `sessions_send` delivery metadata to stop reporting `pending` after an inter-session message has actually been delivered.

## Best Possible Solution

Define the `sessions_send` delivery-result contract first: either return explicit async/accepted metadata instead of a misleading status, or add a real tracked receipt/read-state path with focused tests and docs.

## Reproduction Assessment

Yes at source level for the static pending result: current main always returns `delivery.status: "pending"` for non-skipped A2A announce flow. The claimed retry-loop impact still needs a targeted runtime or regression-test reproduction.

## Solution Assessment

Unclear; simply marking the field `delivered` would not be the best fix unless a real receipt exists. The safer path is to choose between less misleading async metadata and a real receipt/read-state contract.
