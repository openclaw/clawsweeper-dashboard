## Summary

Keep open: current main and the latest release still return stale `pending` delivery metadata on normal async `sessions_send` paths, and two open closing PRs already own candidate fixes with different response-contract choices.

## What This Changes

The issue asks OpenClaw to stop reporting `sessions_send` async inter-session delivery as `delivery.status: "pending"` after the message has already been delivered or accepted by the A2A flow.

## Best Possible Solution

Keep the issue open until one linked PR lands a narrow `sessions_send` metadata contract that no longer exposes misleading async `pending` state, with real behavior proof and maintainer agreement on omit-versus-status semantics.

## Reproduction Assessment

Yes, by source inspection: current main constructs non-skipped `sessions_send` delivery metadata as `pending` before launching async A2A and returns the same object. I did not run a live A2A scenario in this read-only review.

## Solution Assessment

Unclear which linked PR is the best final shape: both are narrow candidate fixes, but maintainers need to choose whether async A2A responses should omit delivery metadata or expose a more precise status contract.
