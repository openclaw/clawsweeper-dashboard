## Summary

Keep open: current main and the latest release still return stale `pending` delivery metadata on normal async `sessions_send` A2A paths, and two open closing PRs already own competing fixes that need maintainer contract review.

## What This Changes

`sessions_send` should stop reporting `delivery.status: "pending"` after async inter-session A2A delivery has proceeded without any path that can update that returned status.

## Best Possible Solution

Keep the issue open until one linked PR lands a chosen `sessions_send` response contract that removes the stale async `pending` signal while preserving the useful `skipped` case.

## Reproduction Assessment

Yes. Current main source constructs `delivery.status: "pending"` before launching the void async A2A flow and returns that object from normal `sessions_send` responses; focused tests assert the same behavior.

## Solution Assessment

Unclear which linked PR is the best final contract. Omitting async delivery metadata is likely safer than reporting terminal delivery before announce completion, but maintainers should choose the response contract.
