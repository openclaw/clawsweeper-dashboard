The PR adds a topbar error-pill dismiss button, threads a `realtimeTalkProviderAvailable` UI state, disables the Talk button when that state is false, and updates chat test fixtures.

## Best Possible Solution

Keep the current-main dismiss flow; if proactive Talk disabling is still desired, add it separately from explicit provider capability/config state rather than caching a negative result from an error string.

## Reproduction Assessment

No live browser reproduction was run, but current-main source and tests show the reported persistent error is now dismissible via `renderChat` and `dismissChatError`. I did not establish a remaining failing current-main path for the linked bug.

No, this PR is not the best way to solve the issue now. Current main already fixes the user-visible dismissal problem, and the PR’s added disable behavior misses the `Realtime voice provider "openai" is not configured` case from the linked report.
