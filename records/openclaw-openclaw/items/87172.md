## Summary

Keep open. The PR targets a real Tool Search performance bug, but the current patch still reports cross-run rebuilds as reused, leaves stale-catalog equality gaps, adds an unbounded session fingerprint cache, and only supplies helper/build proof rather than a real repeated embedded-agent run.

## What This Changes

This PR adds Tool Search catalog fingerprint/reuse flags, suppresses repeated embedded catalog log lines when reuse is reported, and adds helper tests for unchanged and changed tool sets.

## Best Possible Solution

Make reuse truth-preserving: only mark reused when rebuild/register work is actually skipped or a valid cached catalog is restored, compare every catalog-affecting field, bound the fingerprint lifecycle, and prove it with a repeated embedded-agent run.

## Solution Assessment

No. The proposed cross-run path still rebuilds and registers the catalog while reporting reuse, and the equality/cache lifetime design is too narrow for the catalog fields Tool Search actually consumes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not report rebuilt catalogs as reused:** `src/agents/tool-search.ts:804`
  - body: After `clearToolSearchCatalog`, `existingCatalog` is absent, so the fingerprint path still builds a fresh `catalog`, calls `registerToolSearchCatalog`, and then returns `catalogReused: true`. The attempt runner uses that flag to suppress the catalog log, which hides the repeated embedded-turn rebuild/register work instead of skipping it.
  - confidence: 0.92
- **[P2] Compare every catalog field before reusing:** `src/agents/tool-search.ts:507-508`
  - body: `catalogMatchesEntries` only checks id and description, but catalog entries also carry label, source/sourceName, parameters, and the executable tool. If a tool schema or implementation changes while name and description stay stable, Tool Search can keep describing or calling the stale entry.
  - confidence: 0.87
- **[P2] Bound the fingerprint cache lifecycle:** `src/agents/tool-search.ts:472`
  - body: `sessionCatalogFingerprints` is a module-level map that intentionally survives `clearToolSearchCatalog` and is never deleted or bounded. A long-running gateway can accumulate one fingerprint per session/sessionKey, so the cache needs an owner lifecycle, LRU/TTL, or explicit session-close cleanup.
  - confidence: 0.82
