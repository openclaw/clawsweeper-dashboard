## Summary

Keep open. Current main still has a source-level path matching the report: an `alsoAllow`-only tool policy can be forwarded as a restrictive optional-media factory allowlist, blocking generation factories even when explicit defaults are configured; `models.authStatus` omitting API-key-only providers is expected and not the root cause.

## What This Changes

The issue reports generation tools listed in the catalog but omitted from effective tools on v2026.5.3 despite explicit OpenAI/Google generation defaults and API keys.

## Best Possible Solution

Adjust optional media factory policy plumbing so additive `tools.alsoAllow` and plugin-discovery entries do not suppress core generation tools, while preserving explicit restrictive allow/deny behavior and adding regression coverage for the reported config shape.

## Reproduction Assessment

Yes, source-reproducible. With explicit generation defaults plus `tools.alsoAllow: ["group:memory"]`, current source can derive a non-media factory allowlist and block `image_generate`, `video_generate`, and `music_generate` before the explicit-default checks take effect.

## Solution Assessment

Yes. The narrow maintainable fix is to separate optional core media factory gating from plugin-discovery `alsoAllow` plumbing; changing `models.authStatus` would not fix liveness because API-key-only providers are intentionally omitted there.

## Repair Work Prompt

Fix openclaw/openclaw#77244. On v2026.5.3, a config with explicit `agents.defaults.imageGenerationModel`, `videoGenerationModel`, and `musicGenerationModel`, provider env keys, no `tools.allow`/`tools.deny`, and `tools.alsoAllow: ["group:memory"]` leaves `image_generate`, `video_generate`, and `music_generate` out of `tools.effective`.
