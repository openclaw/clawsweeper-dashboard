## Summary

Keep open. The report is fresh, matches documented media-tool availability expectations, and the latest comment supplies a plausible root cause that current main still supports: configured OpenAI/Google generation model refs can enable the tool factory, but the owning provider plugin may still be absent from the gateway startup/runtime provider registry when its chat harness is not otherwise selected.

## What This Changes

The issue reports that media generation tools remain absent from effective tools on v2026.5.3 despite explicit OpenAI/Google generation defaults and provider API keys.

## Best Possible Solution

Add a generic startup/runtime capability path for configured image, video, and music generation model providers, preserving plugin allow/deny semantics and avoiding broad eager loading, then cover it with focused startup/effective-tool regression tests.

## Reproduction Assessment

Yes at source level, but not live-executed here. With a configured generation model whose owning plugin is not selected by channel, chat harness, speech, hook, onStartup, or manifest onConfigPaths, current startup/runtime discovery can omit that provider even though docs say the tool should be live.

## Solution Assessment

Unclear whether the per-manifest onConfigPaths patch is the best complete fix. It is a plausible narrow repair, but the more maintainable solution may be a generic configured-generation-provider activation/discovery path that also covers third-party plugin contracts.
