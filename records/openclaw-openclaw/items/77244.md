## Summary

Keep open. The report is fresh and matches documented behavior that configured generation providers should expose tools; current source also intends explicit default generation models to create those tools, so the v2026.5.3 runtime output points to an unresolved liveness, config, or runtime-state path. `models.authStatus` not listing OpenAI/Google API-key env providers is not by itself proof of missing credentials because current source intentionally excludes plain API-key providers from refreshable auth-status synthesis.

## What This Changes

Generation tools should appear as live/effective when OpenAI/Google API keys and image/video/music generation defaults are configured.

## Best Possible Solution

Add a focused regression or diagnostic for `tools.effective` with explicit OpenAI/Google generation defaults and env-backed provider manifests, then fix the layer that suppresses the generation factories or emit a clear policy/config notice.

## Reproduction Assessment

No. The issue includes concrete v2026.5.3 gateway calls showing a catalog/effective mismatch, but I did not establish the same failure on current main; current source and tests say explicit generation defaults should enable the factories.

## Solution Assessment

Unclear. Changing `models.authStatus` alone would not solve the liveness bug; the narrow maintainable fix should first identify whether tool policy, runtime config snapshots, stale plugin shadow state, or session loading is suppressing the media factories.
