## Summary

Keep open: current main still advertises ACP MCP HTTP/SSE capability as unsupported and rejects ACP per-session MCP servers, while separate OpenClaw-managed `mcp.servers` HTTP/SSE support already exists for embedded Pi/bundle runtimes. Deciding whether to expose this through ACP or a new opt-in flag is a product and security-scope decision, not a conservative cleanup close.

## What This Changes

Request asks OpenClaw to lift the ACP MCP HTTP/SSE capability guard behind a config flag or publish a roadmap/timeline for native HTTP/SSE MCP client support.

## Best Possible Solution

Make the intended scope explicit: either document that ACP per-session MCP transports remain unsupported while directing users to `mcp.servers`, or design a guarded ACP/client transport path with config, tests, and security review.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the ACP guard and rejection path in current main; the remaining question is requested feature and roadmap direction.

## Solution Assessment

Unclear until maintainers decide the product/security boundary. The current `mcp.servers` path is narrower and documented, while lifting ACP `mcpCapabilities` would be a separate user-facing capability with new contract and lifecycle implications.
