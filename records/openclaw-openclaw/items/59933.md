## Summary

Keep open. Current main and the latest release still apply session-tool visibility before `kinds` filtering, so same-agent group/channel rows outside the requester session tree remain hidden unless broader visibility is configured; changing that default or adding a diagnostic escape hatch needs product and security review.

## What This Changes

The issue asks for `sessions_list` to surface active same-agent group/channel sessions, or provide a safe filter/diagnostic path that makes those sessions discoverable.

## Best Possible Solution

Define the session-tool diagnostic visibility contract first, then implement the approved safe path across shared visibility enforcement, session tools, docs, and tests without weakening default privacy.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, `sessions_list` receives gateway rows but rejects a separate same-agent group/channel row before the `group` kind filter can include it.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is the documented broader same-agent scope, but changing defaults or adding a diagnostic escape hatch needs maintainer approval because it changes privacy semantics.
