## Label Justifications

- `P2`: This is a real user-facing session visibility and diagnostics problem, but the blast radius is bounded to session tools and needs product/security direction.
- `impact:session-state`: The issue is about group/channel session state existing in storage and CLI output while being hidden from the agent-facing session tool.

## Summary

Keep open. Current main and the latest release still apply session-tool visibility before `kinds` filtering, so a same-agent group/channel row outside the requester session tree is hidden unless broader visibility is explicitly configured; changing that default or adding a diagnostic escape hatch needs product and security review.

## What This Changes

The issue asks `sessions_list` to surface active same-agent group/channel sessions, or to provide a safe group-session filter or diagnostic path.

## Best Possible Solution

Define the session-tool diagnostic visibility contract first, then add focused coverage and either improve the existing `agent` visibility guidance, add a safe diagnostic mode, or implement an allowlist without weakening default privacy.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, `sessions_list` receives gateway rows but rejects a separate same-agent group/channel row before the `group` kind filter can include it.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is a documented broader scope, but changing defaults or adding diagnostic behavior needs maintainer approval because it changes privacy semantics.
