## Label Justifications

- `P2`: This is a real user-facing session visibility and diagnostics problem, but the blast radius is bounded to session tools and needs product/security direction.
- `impact:session-state`: The issue is about active group/channel session state existing in storage and CLI output while being hidden from the agent-facing session tool.

## Summary

Keep open. Current main and v2026.5.18 still apply the session-tool visibility guard before `kinds` filtering, so same-agent group/channel rows outside the current session tree remain hidden unless the operator opts into broader visibility; changing that default or adding a diagnostic escape hatch needs product and security direction.

## What This Changes

Request that the agent-facing `sessions_list` tool surface active Telegram/Discord group or channel sessions for the same agent, or provide an explicit group diagnostic/filter path.

## Best Possible Solution

Define the session-tool diagnostic visibility contract first, then add focused coverage and either improve the existing `agent` visibility guidance or add an explicit safe diagnostic mode without weakening default privacy.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, `sessions_list` receives gateway rows but rejects a separate same-agent group/channel row before the `group` kind filter can include it.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is a documented broader scope, but changing defaults or adding diagnostic behavior needs maintainer approval because it changes privacy semantics.
