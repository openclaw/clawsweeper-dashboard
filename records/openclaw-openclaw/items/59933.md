## Summary

Keep open. Current main still applies the default `tree` session-tool visibility boundary before `kinds` filtering, so same-agent group/channel rows can exist in the gateway/CLI view but be skipped from `sessions_list`; changing that behavior needs a session-privacy product decision.

## What This Changes

The issue requests that `sessions_list` surface same-agent Telegram group and Discord channel sessions that exist in the session store but are hidden from the tool output.

## Best Possible Solution

Define the safe diagnostic visibility contract, then add focused coverage and either expose same-agent group/channel sessions through an explicit safe mode or report hidden-by-visibility diagnostics without weakening the default `tree` boundary.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent group/channel row can both be returned by the gateway, but the row checker denies the non-requester, non-owned row before `kinds` filtering.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is the explicit workaround, but changing the default or adding diagnostic escape behavior needs maintainer approval because it changes privacy semantics.
