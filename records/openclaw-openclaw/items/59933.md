## Summary

Keep open. Current `main` still matches the reported failure mode: `sessions_list` asks the gateway for sessions, then silently drops same-agent group/channel sessions that are outside the current `tree` visibility before `kinds` filtering can reveal them. The documented `tools.sessions.visibility="agent"` setting is a workaround, but the setup-diagnostic mismatch described in the issue and Discord follow-up remains unresolved.

## What This Changes

The issue requests that `sessions_list` surface active same-agent Telegram group and Discord channel sessions, or provide a filter/mode that makes those sessions visible without manual transcript inspection.

## Best Possible Solution

Define the intended session-tool diagnostic contract, then add focused regression coverage. A maintainable end state would either expose same-agent group/channel sessions through a safe explicit mode/filter or keep `tree` as default while making hidden-by-visibility results discoverable with clear guidance to `tools.sessions.visibility="agent"`.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent Telegram group or Discord channel session will be enumerated by the gateway but rejected by `createSessionVisibilityGuard` unless the target is the requester or a spawned child.

## Solution Assessment

Unclear. The current opt-in `agent` visibility is a valid workaround but not a complete setup diagnostic fix; widening defaults is security-sensitive, so the best code path needs maintainer approval of the visibility contract first.
