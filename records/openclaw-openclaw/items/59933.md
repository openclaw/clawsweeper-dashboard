## Summary

Keep open. Current main still filters `sessions_list` rows through the default `tree` visibility boundary before `kinds` filtering, so separate same-agent group/channel sessions remain hidden unless visibility is widened; changing that default or adding a diagnostic escape hatch is a privacy-sensitive product decision.

## What This Changes

The issue asks for `sessions_list` to surface same-agent Telegram group sessions, or provide an explicit filter/mode that makes those sessions discoverable without manual transcript inspection.

## Best Possible Solution

Define the session-tool diagnostic visibility contract, then add focused coverage and either expose same-agent group/channel sessions through a safe explicit mode or return hidden-by-visibility diagnostics without weakening default privacy unexpectedly.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent Telegram group or Discord channel row are listed by the gateway but rejected unless the target is the requester or is marked owner/spawned/parent-owned.

## Solution Assessment

Unclear. The documented `tools.sessions.visibility="agent"` setting is a workaround, but changing the default or making `kinds:["group"]` bypass visibility would alter privacy semantics and needs maintainer approval.
