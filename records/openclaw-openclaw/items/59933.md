## Summary

Keep open. Current main still filters `sessions_list` rows through the default `tree` session visibility before kind filtering, so separate same-agent group/channel sessions remain hidden unless visibility is widened; changing that default or adding an explicit diagnostic mode is a privacy-sensitive product decision, not cleanup-close material.

## What This Changes

The issue requests that `sessions_list` surface active same-agent group or channel sessions, or provide a filter/mode that makes those sessions discoverable for diagnostics.

## Best Possible Solution

Define the session-tool diagnostic visibility contract, then add focused coverage and either expose same-agent group/channel sessions through a safe explicit mode/filter or report hidden-by-visibility metadata without weakening default privacy unexpectedly.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent Telegram group or Discord channel session are enumerated by the gateway but rejected unless the target is the requester or a spawned/owned child.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is a workaround, but changing defaults or making `kinds` bypass visibility would alter privacy semantics and needs maintainer approval first.
