## Summary

Keep open. Current main still filters `sessions_list` rows through the default `tree` visibility boundary before `kinds` filtering, so separate same-agent group/channel sessions remain hidden unless visibility is widened; changing that default or adding a diagnostic escape hatch needs maintainer product/security judgment.

## What This Changes

Request that `sessions_list` surface active same-agent Telegram or Discord group/channel sessions, or provide an explicit group-session diagnostic path, instead of making them invisible from another same-agent session.

## Best Possible Solution

Define the session-tool diagnostic visibility contract, then add focused coverage and either expose same-agent group/channel sessions through a safe explicit mode/filter or return hidden-by-visibility diagnostics without weakening default privacy unexpectedly.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent Telegram group or Discord channel session are enumerated by the gateway but rejected unless the target is the requester or a spawned/owned child.

## Solution Assessment

Unclear. The existing `tools.sessions.visibility="agent"` setting is a documented workaround, but changing the default or making `kinds:["group"]` bypass visibility would alter privacy semantics and needs maintainer approval.
