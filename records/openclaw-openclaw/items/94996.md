Keep open: the patch touches the right CLI skills-status surface, but it does not solve the live Gateway-owned remote-state boundary, so the reported remote macOS eligibility bug can still remain after merge.

## What This Changes

The PR changes `src/cli/skills-cli.ts` so `loadSkillsStatusReport` lazily imports `getRemoteSkillEligibility` and passes it into `buildWorkspaceSkillStatus`.

## Best Possible Solution

Make the CLI skills status commands prefer Gateway `skills.status` when reachable, keep the local builder as the offline fallback, and prove the live Linux Gateway plus macOS node flow.

## Reproduction Assessment

Yes, from source inspection: the CLI status path can run in a process that has no Gateway-populated remote node cache, while the linked issue supplies matching released CLI output. I did not run a live paired macOS node setup in this read-only review.

## Solution Assessment

No: direct use of `getRemoteSkillEligibility()` in the CLI is only a plausible local refactor, not the best fix for Gateway-owned live state. The safer implementation is to consume Gateway `skills.status` when available and retain local/offline fallback behavior.
