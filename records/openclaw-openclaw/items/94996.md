Route skills status CLI commands through Gateway `skills.status` when reachable, with the existing local workspace status path preserved as the offline fallback.

## Reproduction Assessment

Yes from source inspection. Current main's CLI status path builds local status directly, while the connected remote macOS node state lives in the Gateway/runtime path; I did not run a live paired macOS node here.

## Solution Assessment

No. Passing the existing eligibility option is plausible, but the best fix is to consume the Gateway's live `skills.status` result because that process owns the connected remote-node state.
