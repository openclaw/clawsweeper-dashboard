## Summary

Keep open: this targets a real Feishu message-delivery regression, but the diff does not fix the actual gateway startup path because that path passes `RuntimeEnv`, so the new `channel` guard never initializes the Feishu plugin runtime in production. The supplied proof is also mock-only.

## What This Changes

This PR adds Feishu monitor-side runtime seeding when `monitorFeishuProvider` receives a runtime with a `channel` object and adds mocked startup regression coverage.

## Best Possible Solution

Seed the Feishu plugin runtime through the actual gateway/plugin runtime startup contract before account handlers run, cover the `channel.ts` gateway start path, and require redacted real Feishu WebSocket proof before merge.

## Reproduction Assessment

Yes from source, though not live-tested here: the gateway `startAccount` path passes `RuntimeEnv` into `monitorFeishuProvider`, while Feishu inbound handling requires `getFeishuRuntime()` to have been initialized.

## Solution Assessment

No. The PR seeds the runtime only for a synthetic `PluginRuntime`-shaped monitor argument; the maintainable fix should initialize the existing Feishu runtime store on the real gateway/plugin startup path and test that caller.
