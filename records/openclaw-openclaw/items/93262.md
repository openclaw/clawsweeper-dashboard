## Summary

Keep this PR open, but it is not merge-ready: it targets a valid source-reproduced external channel startup bug, while the current patch short-circuits later startup predicates for all non-bundled channel owners and still lacks real behavior proof from an installed external plugin.

## What This Changes

The PR adds a gateway-startup planner branch and regression tests so non-bundled channel plugins enabled only through `plugins.entries.<id>.enabled` can be included without a `channels.*` config entry.

## Best Possible Solution

Narrow the startup-planner branch to the explicitly enabled external-channel case without blocking sibling provider/harness/hook predicates, then require real installed-plugin gateway boot proof before merge.

## Reproduction Assessment

Yes at source level: current main drives gateway loading from `startup.pluginIds`, and the linked issue shows plugin-entry-only external channel plugins are omitted from that plan. I did not run a live packaged external-plugin gateway reproduction in this read-only review.

## Solution Assessment

No as written: the PR is a plausible channel-only fix, but the broad non-bundled channel branch blocks later startup predicates for mixed-surface plugins. The best fix is a narrower explicit-external-channel predicate or a fall-through path when the channel gate does not pass.
