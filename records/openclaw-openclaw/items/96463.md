## Summary

Keep open: current main still has an unresolved generic custom OpenAI-compatible streaming-usage boundary. The embedded runner now propagates assistant usage when a usage chunk reaches the assistant message, but generic custom endpoints only request `stream_options.include_usage` when compatibility is declared or explicitly configured, which matches this report’s likely failure mode.

## What This Changes

The issue reports zero usage fields for a custom-named OpenAI-compatible provider during `openclaw agent --local --json` embedded runs.

## Best Possible Solution

Choose the maintained compatibility boundary for generic custom OpenAI-compatible streaming usage, then either document/diagnose the existing `compat.supportsUsageInStreaming` opt-in or add a narrow safe default for provider classes known to honor `stream_options.include_usage`.

## Reproduction Assessment

Partially: the reporter gives detailed release steps, and current-main source shows generic custom providers do not send streaming usage unless compatibility is detected or explicitly configured. No live MaaS endpoint was exercised in this read-only review.

## Solution Assessment

Unclear: automatically sending `stream_options.include_usage` for every custom endpoint would undo a recent compatibility boundary. Maintainers should choose between explicit opt-in, docs/diagnostics, or a narrower safe default.
