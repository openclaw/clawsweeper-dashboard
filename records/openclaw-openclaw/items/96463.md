## Summary

Keep open: current main still has a source-reproducible hosted custom OpenAI-compatible streaming-usage gap. The embedded runner now propagates assistant-message usage when it reaches the transcript, but the reporter's generic hosted custom provider config does not cause OpenClaw to request `stream_options.include_usage` unless compat metadata or an explicit opt-in is present.

## What This Changes

The issue reports zero token usage in embedded local JSON/session/trajectory output for custom OpenAI-compatible providers using `api: "openai-completions"`.

## Best Possible Solution

Choose the canonical generic hosted custom-provider streaming-usage policy, then either document the explicit opt-in or broaden defaults for verified endpoint classes without regressing strict providers.

## Reproduction Assessment

Partially: the reporter gives detailed release steps, and current-main source shows the provided generic hosted custom-provider config would not request streaming usage without an explicit compatibility opt-in. I did not run a live custom MaaS endpoint.

## Solution Assessment

Unclear: sending `stream_options.include_usage` for every hosted custom endpoint may fix this provider but would change compatibility defaults for endpoints that reject that field. The safer path is a maintainer-approved compat policy plus focused tests and docs.
