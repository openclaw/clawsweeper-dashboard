## Summary

Keep open: current main still leaves a source-visible usage-accounting gap, and the open linked PR is a candidate fix for only the embedded-runner accounting subset while the broader custom-provider streaming-usage policy remains unresolved.

## What This Changes

The issue reports zero token usage for custom OpenAI-compatible providers when `openclaw agent --local --json` uses the embedded runner.

## Best Possible Solution

Land or close the linked accounting PR on its merits, then make a maintainer decision for generic custom OpenAI-compatible streaming usage: explicit compat opt-in, clearer docs/diagnostics, or a narrower safe default for endpoints known to honor `stream_options.include_usage`.

## Reproduction Assessment

Partially: source inspection shows the embedded runner can treat a present zeroed assistant usage snapshot as authoritative, and generic custom endpoints do not request streaming usage unless compat declares it. I did not run a live MaaS endpoint repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose scope: https://github.com/openclaw/openclaw/pull/96523 appears to be the narrow best fix for zero-snapshot accounting, while broadening default streaming usage for every custom provider is a separate compatibility decision.
