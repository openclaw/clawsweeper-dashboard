## Summary

Keep open: current main still sends WhatsApp auto-replies through the inbound message delivery closures, while the existing main-DM owner pin only protects persisted lastRoute metadata. The requested safeguard is a security-sensitive routing policy/config change that still needs maintainer choice on block versus reroute.

## What This Changes

Request an opt-in WhatsApp routing safeguard that blocks or reroutes plain replies when a DM turn came from an admitted non-owner sender.

## Best Possible Solution

Add a WhatsApp-owned opt-in delivery guard before visible auto-reply delivery, with schema/help/docs and regression coverage for owner DMs, non-owner DMs, media replies, and explicit message-tool sends.

## Reproduction Assessment

Yes, from source inspection. An admitted non-owner WhatsApp DM under main dmScope with a single-owner allowFrom pin reaches dispatchWhatsAppBufferedReply, which passes the inbound msg to deliverWebReply, whose reply/sendMedia closures deliver to the inbound chatJid.

## Solution Assessment

Unclear for the exact requested block-versus-reroute behavior until maintainers decide the policy. The maintainable direction is a WhatsApp plugin delivery guard, not another prompt rule or a hidden core-wide routing override.
