## Summary

Keep open: current main still intentionally routes WhatsApp DM auto-replies to the inbound sender, while the existing main-DM pinning only protects lastRoute and follow-up metadata. The requested fail-closed third-party DM guard is useful but needs a new security-sensitive config and product decision.

## What This Changes

Request a WhatsApp DM routing safeguard that blocks or reroutes implicit plain-text replies when the latest inbound sender is not the session owner.

## Best Possible Solution

Add a plugin-owned, documented WhatsApp DM safety policy that fail-closes implicit replies for non-owner direct senders, with owner resolution aligned to allowFrom or explicit owner routes and focused regression coverage.

## Reproduction Assessment

Yes for the reported current behavior: source inspection shows WhatsApp direct replies set the origin to msg.from and deliver text through msg.reply(...). I did not run a live WhatsApp conversation, and the requested guard is a new behavior rather than an existing contract failure.

## Solution Assessment

Unclear until maintainers choose the policy shape. A narrow WhatsApp plugin config is likely the maintainable direction, but block vs owner-route behavior and owner identity rules need product/security review first.

## Risks / Open Questions

- Owner identity is not fully defined for multi-owner allowFrom lists, wildcard allowlists, pairing approvals, and commands.ownerAllowFrom routes.
- Blocking versus auto-routing to owner is a product/security choice because auto-routing may disclose third-party content to the owner while blocking may break legitimate agent-to-agent DM workflows.
- The change crosses WhatsApp plugin behavior and shared session routing expectations, so it should be designed with the existing lastRoute pinning and private command routing work rather than patched only at one send call.
