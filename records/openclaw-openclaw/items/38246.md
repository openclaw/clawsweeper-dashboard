## Summary

Keep this open. Current main still routes WhatsApp visible auto-replies through the inbound message's reply/sendMedia closures, so an admitted non-owner DM can receive plain text or media replies that the agent meant for the owner; the existing main-DM pinning only protects persisted lastRoute metadata.

## What This Changes

The issue requests an opt-in WhatsApp routing safeguard that blocks or forces explicit targets for plain auto-replies when the last inbound DM sender is not the session owner.

## Best Possible Solution

Add an opt-in, typed WhatsApp root/account policy in the WhatsApp plugin delivery path that refuses visible implicit auto-replies for non-owner DMs when the main DM owner is pinned, while preserving owner DMs and explicit message-tool sends; update schema/help/docs and cover owner, non-owner, media, and explicit-target cases.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction path is an admitted non-owner WhatsApp DM under main dmScope with a single-owner allowFrom pin: the reply payload reaches dispatchWhatsAppBufferedReply and deliverWebReply calls the msg.reply closure bound to the inbound chatJid.

## Solution Assessment

Unclear for the exact product behavior until maintainers choose block versus reroute. The narrowest maintainable direction is a WhatsApp-owned opt-in guard before visible delivery, not another prompt rule or a core-wide hidden routing override.
