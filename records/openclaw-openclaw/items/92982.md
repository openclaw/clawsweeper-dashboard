## Summary

Keep open. The reported double-wrap bug is valid, but this PR is not merge-ready because the guard returns an already wrapped tool and skips the fresh per-run hook context that `createOpenClawCodingTools` builds for isolated and cron runs.

## What This Changes

Adds an `isToolWrappedWithBeforeToolCallHook` import and guard in `createOpenClawCodingTools` to avoid wrapping tools that already carry the `before_tool_call` marker.

## Best Possible Solution

Preserve before_tool_call source/context metadata through normalization, then rebuild or dedupe wrappers so the hook fires once with the current `createOpenClawCodingTools` context and focused regression coverage.

## Reproduction Assessment

Yes, from source. Current main normalizes tools and then unconditionally wraps them, so an already wrapped OpenClaw tool can be wrapped again; I did not execute a live cron or isolated-agent reproduction in this read-only review.

## Solution Assessment

No. The PR touches the right wrap boundary, but returning the existing wrapper is not the best fix because it skips the current per-run hook context; a context-preserving rewrap or metadata-preserving dedupe is safer.
