## Summary

Keep open: the report overlaps active heartbeat/session-contamination work, and current main still has shared-history heartbeat behavior by default plus no active reply-run guard in the heartbeat runner, so there is not enough evidence to close this as fixed or duplicate-only.

## What This Changes

The issue reports a 2026.4.29–5.4 regression where heartbeat turns in multi-agent setups can deliver replies to the wrong agent/session and replay stale conversation content before `HEARTBEAT_OK`.

## Best Possible Solution

Resolve the heartbeat/session-contamination cluster by landing the active reply-run guard, making an explicit maintainer decision on default heartbeat isolation, and then confirming this reporter’s Zai multi-agent setup with a redacted transcript before closing the remaining branch.

## Reproduction Assessment

No. Source inspection shows plausible current-main paths for shared heartbeat history and active-reply preemption, but the exact cross-agent Zai heartbeat contamination has not been reproduced without a transcript or logs.

## Solution Assessment

Unclear. `isolatedSession: true` plus `lightContext: true` is a strong mitigation, but the best repo fix depends on maintainer approval for default isolation and on whether the remaining cross-agent branch is reply-run, subagent delivery, or provider replay related.
