## Summary

Keep open: current main still leaves heartbeat shared-history behavior as the default and still lacks the active reply-run guard that related reports identify as one branch of heartbeat preemption, while the exact multi-agent Zai cross-contamination path has not been proven fixed or fully reproduced.

## What This Changes

The issue reports a regression where heartbeat runs in multi-agent setups can reply in the wrong agent/session and include stale prior conversation content before `HEARTBEAT_OK`.

## Best Possible Solution

Resolve the cluster by landing or replacing the active reply-run heartbeat guard, making an explicit maintainer decision on default heartbeat isolation, and then validating this reporter’s multi-agent Zai setup with redacted logs or transcript evidence.

## Reproduction Assessment

No for the full reported issue: the exact cross-agent Zai contamination was not reproduced. Source inspection does show current-main paths that can still produce shared-history heartbeat context and active-reply preemption, so this remains a plausible current bug cluster.

## Solution Assessment

Unclear. `isolatedSession: true` plus `lightContext: true` is a strong operator mitigation, but the repo fix needs both the narrow active-reply guard and a maintainer decision on whether heartbeat isolation should become the default.
