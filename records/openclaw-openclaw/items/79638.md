## Summary

Keep open: current main still leaves heartbeat shared-session behavior as the default, lacks the active reply-run guard proposed for the heartbeat preemption branch, and related open PRs only cover slices of the stale replay/no-op leak cluster.

## What This Changes

The issue reports a regression where heartbeat turns in multi-agent Telegram/Zai setups can deliver replies to the wrong agent/session or include stale conversation content before `HEARTBEAT_OK`.

## Best Possible Solution

Resolve the cluster by landing or replacing the focused active-reply, stale-replay, and no-op suppression fixes, making an explicit default-isolation policy decision, and then verifying the reporter's redacted multi-agent Zai/Telegram setup.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available for the exact multi-agent Zai cross-contamination. Source inspection does show current-main paths for shared heartbeat history, active-reply preemption, stale pending-final heartbeat replay, and Telegram context duplication.

## Solution Assessment

Unclear as a single fix. The best path is a set of narrow runtime guards plus a maintainer decision on default heartbeat isolation, rather than closing this under any one open PR before merge and verification.
