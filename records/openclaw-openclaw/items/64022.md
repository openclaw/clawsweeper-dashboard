## Summary

Keep open: the Slack attachment bug is still source-reproducible on current main, but this PR currently drops the guard-created dispatcher from the final media request without an approved SSRF enforcement contract, carries a stale raw-fetch allowlist update, and lacks inspectable real behavior proof.

## What This Changes

The PR strips `dispatcher` from Slack media fetch init before delegating to runtime fetch, updates the focused Slack media test expectation, and changes Slack raw-fetch allowlist line numbers.

## Best Possible Solution

Land a refreshed Slack media fix that removes the Undici dispatcher conflict while preserving Slack URL validation, auth headers, redirect behavior, and the intended SSRF enforcement contract.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The linked issue gives Slack Socket Mode/container steps and `invalid onRequestStart method` logs, and current main still forwards the guard-created dispatcher through `createSlackMediaFetch`; I did not run a live Slack workspace.

## Solution Assessment

No, not as currently proposed. Stripping the forwarded dispatcher matches the observed failure mode, but this patch removes dispatcher-level SSRF enforcement unless maintainers approve a Slack preflight-only contract or choose an enforcement-preserving replacement.
