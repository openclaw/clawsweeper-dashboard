## Summary

Keep open: the Slack attachment failure is credible, but the patch still needs maintainer/security approval because it changes Slack media from pinned-dispatcher enforcement to preflight-only validation, and the branch carries stale raw-fetch allowlist updates.

## What This Changes

The PR strips the SSRF guard dispatcher from Slack media fetch init before calling runtime fetch, updates the Slack media test expectation, and shifts raw-fetch allowlist line numbers.

## Best Possible Solution

Fix Slack attachment downloads with an explicit reviewed transport contract: either preserve pinned enforcement while avoiding the Undici mismatch, or opt Slack media into a documented preflight-only path with focused tests and a fresh branch.

## Reproduction Assessment

Yes. The linked report gives Slack Socket Mode/container steps and `invalid onRequestStart method` logs, and current main still routes Slack inbound media through `createSlackMediaFetch` with a guard-created dispatcher; I did not run a live Slack workspace.

## Solution Assessment

No, not as currently proposed. The branch removes the pinned dispatcher from the final Slack media request without an explicit approved Slack transport contract and includes stale script line updates; a reviewed `pinDns: false`/preflight path or pinned-enforcement-compatible fix would be safer.
