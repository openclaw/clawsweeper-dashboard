## Summary

Keep open. The Slack attachment failure appears real and current main still forwards the SSRF guard dispatcher through the Slack media fetch path, but this PR changes a security-sensitive pinned-DNS contract and its raw-fetch allowlist update is stale against current main.

## What This Changes

The PR strips `dispatcher` from Slack media fetch init before delegating, updates the Slack media regression assertion, and adjusts raw-fetch allowlist line numbers.

## Best Possible Solution

Make an explicit maintainer decision for Slack media transport: either add a reviewed preflight-only or `pinDns: false` contract with regression coverage and fresh allowlist lines, or preserve pinned dispatcher enforcement while fixing the Undici version incompatibility.

## Reproduction Assessment

Yes. The linked Slack report gives concrete container/sandbox steps and `invalid onRequestStart method` logs, and current main still routes Slack inbound media through `createSlackMediaFetch` with a guard-created dispatcher; I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No, not as currently proposed. The source/test change is narrow and has a BlueBubbles precedent, but it silently removes the pinned dispatcher from the final Slack request and the branch carries a stale allowlist update; the safer path is an explicit reviewed transport contract or a dispatcher-compatible fix.
