## Summary

Keep open: current main has staged package-update, plugin-sync, and macOS LaunchAgent recovery safeguards, but I did not find end-to-end proof that the reported v2026.5.2 macOS dev/source-to-stable cutover incident is fully fixed. The existing update-channel Docker lane uses `--no-restart`, so it does not exercise the LaunchAgent wrapper and restart state central to this report.

## What This Changes

The issue asks OpenClaw to keep macOS dev/source-to-stable package cutovers from leaving service wrapper, bundled/plugin loading, plugin install metadata, and context-engine state partially inconsistent.

## Best Possible Solution

Add or run a targeted macOS package-cutover validation that proves wrapper/plist state, package root, plugin install records/load paths, Discord readiness, and the selected context engine stay consistent or fail closed with actionable recovery guidance.

## Reproduction Assessment

No. The issue preserves concrete v2026.5.2 incident evidence, but there is no current-main reproduction path, and the existing update-channel smoke does not exercise macOS LaunchAgent wrapper state.

## Solution Assessment

Unclear. The current safeguards are directionally correct, but the narrowest maintainable fix should be driven by an integrated macOS cutover validation run before changing update, plugin, or restart behavior.
