## Summary

Keep this PR open. Current main still commits embedded message bookkeeping from the stale `tool_execution_start` arguments before consuming the adjusted `before_tool_call` params, while the wrapper already stores and executes those adjusted params. The PR is still a plausible implementation path for the bookkeeping gap, even though the linked direct-interception issue was later closed as implemented on main.

## What This Changes

The PR changes embedded message and cron bookkeeping to use adjusted `before_tool_call` params, adds adjusted-message/cron regression tests, and broadens transcript repair to recognize snake_case raw tool-call block types.

## Best Possible Solution

Rebase this PR onto current main, keep the adjusted-param bookkeeping change in `handleToolExecutionEnd`, add the focused message/cron regressions, and either split or explicitly approve the transcript-repair change before landing.

## Reproduction Assessment

Yes. A high-confidence reproduction is a `before_tool_call` hook that rewrites a `message` send after `tool_execution_start`; current main records the original text/target/media at start and commits that state at end before reading the adjusted params.

## Solution Assessment

Yes, with cleanup. Consuming adjusted params at the start of `handleToolExecutionEnd` and using them for message and cron bookkeeping is the narrowest maintainable fix; the safer landing path is to separate or deliberately review the unrelated transcript repair expansion.
