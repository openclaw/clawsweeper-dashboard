## Summary

Keep open: current main handles the documented `MEDIA:` directive and already-structured media fields, but it does not implement the broader channel-neutral artifact guard requested for raw path claims, allowed-root policy, send-success-gated stripping, and failure fallback semantics. The related open PR is adjacent transcript hygiene work and explicitly excludes outbound delivery changes, so it is not a superseding fix.

## What This Changes

Add a generic outbound artifact delivery guard that turns validated generated-artifact references into channel attachments and removes unsafe raw delivery prose only after successful delivery.

## Best Possible Solution

Keep this as a design issue and, if accepted, add an additive core outbound artifact-normalization layer that reuses existing media staging/delivery while defining conservative parsing, allowed roots, and failure semantics before channel-specific patches.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an already-defined expected behavior. Source inspection gives a high-confidence current-main boundary: documented `MEDIA:` and structured media fields are handled, while broader raw path claims are not.

## Solution Assessment

Unclear: the requested direction is plausible, but the best implementation depends on maintainer decisions about hook location, path grammar, configurable roots, and delivery-failure payload behavior. A safer direction is an additive shared helper covered by synthetic channel tests, reusing existing `ReplyPayload.mediaUrls` staging and outbound delivery paths.
