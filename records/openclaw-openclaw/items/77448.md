## Summary

Keep open: current main has partial MEDIA/local media handling, but it does not implement the broader delivery-result-aware, channel-neutral artifact guard requested here. The related PR is adjacent transcript-hygiene work and explicitly leaves outbound delivery out of scope.

## What This Changes

Request a channel-neutral final-delivery guard that validates generated artifact references, sends them as attachments, and strips or replaces raw path/directive text based on delivery outcome.

## Best Possible Solution

Define a core outbound artifact-delivery contract around ReplyPayload/structured metadata, narrow allowed roots, delivery-result-aware text projection, and synthetic channel coverage before per-channel rollout.

## Reproduction Assessment

No; this is a feature/design request, not a concrete public current-main bug reproduction. Source inspection confirms only partial MEDIA and message-action media handling, while the private deployment examples are not directly reproducible here.

## Solution Assessment

Unclear; a channel-neutral guard is a plausible direction, but maintainers should first choose the hook location, allowed-root contract, metadata inputs, and failure semantics. A Telegram-only or skill-only patch would not solve the requested invariant.
