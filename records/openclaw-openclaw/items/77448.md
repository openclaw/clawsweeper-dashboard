## Summary

Keep open: current main handles documented `MEDIA:` directives and structured media payloads, but this asks for a broader channel-neutral guard for raw artifact path claims, allowed roots, send-success-gated stripping, and failure semantics. Related #77445 is adjacent transcript hygiene work and explicitly excludes outbound delivery.

## What This Changes

Add a channel-neutral outbound guard that validates generated-artifact references, sends them as attachments, strips raw path/directive prose only after send success, and defines failure fallback.

## Best Possible Solution

Keep this as a design issue and, if accepted, add an additive core artifact-normalization layer that reuses existing media staging/delivery and is proven with synthetic-channel success and failure tests.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an already-defined expected behavior. Current source and docs provide a high-confidence gap check: documented attachment handling is `MEDIA:` or structured payload media, while raw `File:`/localized path prose is not an attachment contract.

## Solution Assessment

Unclear: the requested direction is plausible, but the best implementation depends on maintainer choices about hook location, path grammar, allowed roots, and failure payload behavior. The narrowest maintainable route would reuse `ReplyPayload.mediaUrls`, existing media access checks, and synthetic-channel coverage rather than adding a Telegram-specific patch.
