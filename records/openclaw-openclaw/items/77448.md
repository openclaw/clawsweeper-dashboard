- `impact:security`: The issue is about preventing raw local filesystem paths or delivery directives from leaking into human-visible chat.

## Summary

Keep open: current main and v2026.6.8 have structured media fields, legacy `MEDIA:` parsing, and bounded local media reads, but they still do not implement the requested success-gated artifact guard for generic local-path prose, structured/metadata sources, and sanitized attachment-failure behavior.

## What This Changes

The issue requests a channel-neutral guard that turns validated generated-artifact references into real outbound attachments and prevents unsafe raw path/directive text from reaching users.

## Best Possible Solution

Define a core-owned outbound artifact delivery contract that reuses existing media access policy, covers final text plus structured payload sources, strips unsafe directives only after proven delivery, and sends sanitized failure responses when attachment delivery fails.

## Reproduction Assessment

Yes, at source level: current main parses explicit `MEDIA:` directives and structured media fields while leaving ordinary local-path prose as text. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear. A channel-neutral final-mile guard is plausible, but maintainers still need to choose the hook location, artifact source scope, trusted-root policy, and failure UX before implementation.
