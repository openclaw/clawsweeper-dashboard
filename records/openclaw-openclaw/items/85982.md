## Summary

Keep this PR open for maintainer approval rather than cleanup-close: the patch looks mechanically sound and now has usable proof, but it intentionally changes the default `/models` provider picker from showing implicit CLI alias providers to a canonical-provider-first view.

## What This Changes

The PR deduplicates implicit CLI runtime alias provider rows in `/models`, derives alias ownership from runtime/setup CLI backend bindings, and adds regression coverage for preserved explicit legacy and `all` browse paths.

## Best Possible Solution

Merge only if maintainers explicitly accept canonical-provider-first picker behavior; otherwise preserve alias provider rows by default or introduce a narrower opt-in/deprecation path.

## Reproduction Assessment

Yes, from source and current tests: current main adds both canonical providers and authenticated CLI runtime providers into `/models`, and the existing fixture expects both `anthropic` and `claude-cli` rows for the same style of catalog data. I did not run tests because this review is read-only.

## Solution Assessment

The implementation is in the right shared owner path, `buildModelsProviderData`, and uses the CLI backend registry rather than hard-coded alias tables. The unresolved part is not a code defect but whether canonical-first default browse is the product-compatible behavior maintainers want.
