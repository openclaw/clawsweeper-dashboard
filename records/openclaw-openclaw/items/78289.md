## Summary

Keep open: current main still omits Feishu's `card_msg_content_type` query parameter on both single-message reads and thread-history reads, so interactive cards can still arrive in the downgraded shape that the existing parser can only turn into `[Interactive Card]`.

## What This Changes

The issue asks Feishu message reads and thread-history fetches to request full interactive-card JSON so card markdown/body content can be parsed instead of returning `[Interactive Card]`.

## Best Possible Solution

Patch the Feishu plugin's read/history API calls to request `user_card_content`, preserve existing fallback parsing, and add focused request-parameter regression coverage plus a changelog entry.

## Reproduction Assessment

Yes from source and upstream API contract, but not from a live tenant: current main calls Feishu get/list without `card_msg_content_type`, while Feishu documents `user_card_content` as required for original card JSON. That is enough for source reproduction, but not a live `message(action=read)` reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to add `card_msg_content_type: "user_card_content"` to the two Feishu API calls and assert those request params in `send.test.ts`. No core plugin API or user-facing config change is needed.
