## Summary

Keep open: current main still omits Feishu's card-content return-format query parameter on both single-message reads and thread-history reads, so interactive cards can still reach the parser as fallback content. The direct fix PR was closed unmerged, and the remaining repair is narrow but should verify the exact Feishu parameter value and SDK boundary before landing.

## What This Changes

The issue requests that Feishu message reads and thread-history reads return actual interactive-card content instead of `[Interactive Card]`.

## Best Possible Solution

Patch the Feishu plugin read/history API calls to request full card JSON through a verified SDK-safe or raw-request boundary, preserve parser fallbacks, add request-parameter regression tests, and coordinate with the open parser work at https://github.com/openclaw/openclaw/issues/60380.

## Reproduction Assessment

Yes, at source level: `message(action=read)` calls `getMessageFeishu`, and both affected Feishu get/list calls currently omit the card-content return-format parameter. I did not establish a live tenant reproduction in this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the Feishu plugin API request layer plus focused tests. The implementation should first verify the current Feishu parameter value and use a raw request adapter if the typed SDK still lacks the field.
