## Summary

Keep open: current main still reads Feishu single messages and thread history without requesting Feishu's full interactive-card content format, so `message(action=read)` can still receive only fallback card content. The direct fix PR is closed unmerged; the remaining work is a narrow Feishu plugin bug fix with an SDK/API-contract boundary to handle carefully.

## What This Changes

The issue asks Feishu message reads and thread history to request original interactive-card JSON instead of returning only the `[Interactive Card]` fallback.

## Best Possible Solution

Land a narrow Feishu-plugin read/history fix that requests full card JSON through a verified SDK-safe or raw request adapter, preserves parser fallbacks, adds request-parameter regression tests, and coordinates broader parser fallback work through the existing Feishu parser issue.

## Reproduction Assessment

Yes, at source level. `message(action=read)` reaches `getMessageFeishu`, and both current Feishu get/list calls omit the official card-content return-format parameter needed for original card JSON; I did not run a live tenant reproduction.

## Solution Assessment

Yes in direction: the fix belongs in the Feishu plugin fetch layer with focused tests. The implementation should first verify and encapsulate the Feishu query parameter through a typed/raw adapter because the current SDK type surface does not expose it.
