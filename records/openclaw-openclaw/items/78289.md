## Summary

Keep open: current main still reads Feishu single messages and thread history without Feishu's documented full-card query parameter, so interactive cards can still arrive as fallback text. The direct fix PR is closed unmerged, and the broader open Feishu parser PR does not close this exact fetch-layer bug.

## What This Changes

The issue asks the Feishu plugin to return actual interactive-card content when reading messages, instead of `[Interactive Card]`.

## Best Possible Solution

Land a narrow Feishu-plugin read/history fix that requests `card_msg_content_type=user_card_content` through an SDK-safe request boundary, preserves parser fallbacks, adds request-parameter regression tests, and coordinates parser-only leftovers with https://github.com/openclaw/openclaw/issues/60380.

## Reproduction Assessment

Yes, at source level: `message(action=read)` calls `getMessageFeishu`, and both Feishu get/list calls still omit the documented full-card return-format parameter. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes in direction: the narrow fix belongs in the Feishu plugin fetch layer with focused tests. The implementation should use a typed/raw request adapter because the current SDK types do not expose the documented parameter.
