## Summary

Keep open: current main still routes Feishu `message(action=read)` and thread-history reads through fetch calls that omit Feishu's documented full-card return-format parameter, so interactive-card reads can still degrade to `[Interactive Card]`. The direct fix PR is closed unmerged, and the remaining work is a narrow Feishu plugin bug fix rather than stale cleanup.

## What This Changes

The issue requests that Feishu single-message reads and thread-history reads return actual interactive-card JSON/text instead of the `[Interactive Card]` placeholder.

## Best Possible Solution

Land a narrow Feishu-plugin fetch-layer fix that requests `card_msg_content_type: "user_card_content"` for single-message and thread-history reads, preserves existing parser fallbacks, adds request-shape regression tests, and leaves broader parser fallback work to the related parser issue.

## Reproduction Assessment

Yes, at source level. `message(action=read)` reaches `getMessageFeishu`, and both current-main Feishu get/list calls omit the documented full-card return-format parameter; I did not run a live tenant reproduction.

## Solution Assessment

Yes in direction: the narrow fix belongs in the Feishu plugin fetch layer with focused tests. The implementation should verify the SDK/raw-request boundary because the current SDK types do not expose the documented parameter.
