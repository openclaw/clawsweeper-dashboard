## Summary

Keep open: current main still routes Feishu `message(action=read)` through helpers that omit any card-content return-format parameter, so interactive card reads can still fall back to `[Interactive Card]`. The direct fix attempt at https://github.com/openclaw/openclaw/pull/78302 is closed unmerged, and https://github.com/openclaw/openclaw/issues/54972 was closed as duplicate/superseded in favor of this issue, so this remains the canonical open tracker.

## What This Changes

The issue asks the Feishu plugin to return actual interactive card content for single-message reads and thread-history reads instead of `[Interactive Card]`.

## Best Possible Solution

Land one Feishu-plugin read/history fix that requests full card JSON through a validated SDK-safe or raw-request boundary, preserves existing parser fallbacks, adds request-parameter regression tests, and coordinates with the overlapping open Feishu interactive-card PR.

## Reproduction Assessment

Yes as a source reproduction: current main's Feishu read path reaches `getMessageFeishu`, and both single-message and thread-history fetches omit any card-content query parameter before parsing interactive card content. No live Feishu tenant run was performed.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the Feishu plugin's message get/list request boundary, not core or configuration. The implementation should first verify the exact Feishu/Lark parameter value and route it through an SDK-safe or raw request helper because the typed SDK surface omits the field.
