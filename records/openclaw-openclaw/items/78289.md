## Summary

Keep open: current main still omits Feishu's documented `card_msg_content_type` query parameter on both single-message reads and thread-history reads, so interactive cards can still arrive in a downgraded shape that the existing parser turns into `[Interactive Card]`.

## What This Changes

The issue requests Feishu read and thread-history message fetches to request `card_msg_content_type: "user_card_content"` so interactive-card body text is available to the existing parser.

## Best Possible Solution

Patch the Feishu plugin read/history API calls to request `user_card_content`, preserve existing parser fallbacks, add request-parameter regression tests and a changelog entry, and coordinate with #60383/#73203/#56795 to avoid divergent card parsers.

## Reproduction Assessment

Yes, at source/API-contract level but not from a live tenant. `message(action=read)` reaches `getMessageFeishu`, current main omits `card_msg_content_type` on both Feishu get/list calls, and Feishu documents `user_card_content` as required for original card JSON.

## Solution Assessment

Yes. The narrow maintainable fix is to add the documented query parameter to the two Feishu plugin API calls and assert the request shape in `send.test.ts`; no core plugin API or user-facing config is needed.
