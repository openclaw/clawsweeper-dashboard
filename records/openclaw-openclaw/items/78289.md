## Summary

Keep open: current main still routes Feishu single-message reads and thread-history reads through API calls that omit Feishu's documented `card_msg_content_type=user_card_content` query parameter, so interactive cards can still degrade to `[Interactive Card]`. The direct fix PR is closed unmerged, while the remaining work is a narrow Feishu plugin bug fix.

## What This Changes

The issue requests that Feishu `message(action=read)` and thread-history reads return actual interactive-card content instead of `[Interactive Card]`.

## Best Possible Solution

Land a narrow Feishu-plugin fetch-layer fix that requests `card_msg_content_type: "user_card_content"` for single-message and thread-history reads through an SDK-safe or raw-request boundary, preserves parser fallbacks, and leaves fallback-parser work to https://github.com/openclaw/openclaw/issues/60380.

## Reproduction Assessment

Yes, at source level. `message(action=read)` reaches `getMessageFeishu`, and both current-main Feishu get/list calls omit the officially documented full-card return-format parameter; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in the Feishu plugin request layer with focused request-shape tests, but it must account for the SDK typing gap rather than adding an unchecked broad `any` change.
