## Summary

Keep open: current main and the latest shipped release still read Feishu single messages and thread history without requesting Feishu's full interactive-card content format, so the reported `[Interactive Card]` result remains source-reproducible. The direct fix PR is closed unmerged, and the remaining work is a narrow Feishu plugin bug fix rather than a cleanup close.

## What This Changes

The issue asks Feishu `message(action=read)` and thread-history reads to return actual interactive-card content instead of only `[Interactive Card]`.

## Best Possible Solution

Land one narrow Feishu-plugin read/history fix that requests full card JSON through a verified SDK-safe or raw request adapter, preserves existing parser fallbacks, and adds request-parameter regression coverage.

## Reproduction Assessment

Yes, at source level. `message(action=read)` reaches `getMessageFeishu`, and both current Feishu get/list calls omit Feishu's documented card-content return-format parameter needed for original interactive-card JSON.

## Solution Assessment

Yes in direction: the fix belongs in the Feishu plugin fetch layer with focused tests. The implementation should encapsulate the documented query parameter through a typed/raw adapter because the current SDK type surface does not expose it.
