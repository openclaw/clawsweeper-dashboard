## Summary

Keep open. Current main still builds the embedded-run tool set inside the `core-plugin-tools` stage and still invokes upstream `createCodingTools()` before runtime allow/profile filtering, while the reporter provided fresh Windows/WSL2 + Node 24 timing data that should not be collapsed into the broader plugin-runtime-deps tracker.

## What This Changes

The issue asks OpenClaw to investigate and reduce a 30-40s Windows/WSL2 + Node 24 cold-start block in `core-plugin-tools`, potentially by caching or deferring Pi coding-tool construction.

## Best Possible Solution

Split `core-plugin-tools` into Pi base tools, OpenClaw built-ins, and plugin factories; remove avoidable upstream base-tool construction when behavior stays identical; then ask the reporter to validate a Windows/WSL2 + Node 24 build.

## Reproduction Assessment

Yes for the reported stage-level issue: the reporter provided repeated Windows/WSL2 + Node 24 timings and current main maps that bucket to `createOpenClawCodingTools`. The exact subcall attribution is not yet live-proven and should be confirmed with narrower instrumentation.

## Solution Assessment

No complete solution is in place yet. The narrowest maintainable repair is to instrument the hot substages and eliminate the discarded upstream `createCodingTools()` work where OpenClaw already constructs equivalent wrapped tools, rather than adding broad caching first.
