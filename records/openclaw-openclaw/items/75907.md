Keep open. Current main still builds the embedded-run tool set inside the `core-plugin-tools` stage and still invokes upstream `createCodingTools()` before tool-policy filtering, while the fresh Windows + Node 24 traces and follow-up comment give enough evidence for focused performance work rather than cleanup closure.

## What This Changes

The issue reports a Windows WSL2 + Node 24 cold-start delay in `core-plugin-tools` and asks for investigation, caching, or lazy loading of coding-tool initialization.

## Best Possible Solution

Split `core-plugin-tools` timing into Pi base tools, OpenClaw built-ins, and plugin tools, remove avoidable upstream coding-tool construction if behavior stays identical, and only then consider narrowly keyed caching or lazy loading.

## Reproduction Assessment

Yes. The issue has a high-confidence reported reproduction path from repeated stage timings, and current main maps that stage directly to `createOpenClawCodingTools`; this review did not live-test Windows WSL2 + Node 24.

## Solution Assessment
