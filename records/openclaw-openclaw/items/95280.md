## Summary

Keep open: current main and v2026.6.9 still emit OpenAI-completions answer text before the native reasoning block is closed, and open linked PRs already own candidate fixes.

## What This Changes

DeepSeek-style OpenAI-completions reasoning should close before final answer content so `/reasoning on` delivers the thought and answer as separate blocks.

## Best Possible Solution

Land one focused provider-boundary fix that emits exactly one `thinking_end` before text or tool-call lanes, then close this issue and supersede duplicate candidate PRs.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: `reasoning_content` opens a thinking block, later `content` emits text, and `thinking_end` is delayed until stream finalization on current main.

## Solution Assessment

Yes for the provider-boundary fix shape. The provider owns the `reasoning_content` to text/tool event order before subscribers and channels consume it; a subscriber-only workaround is less complete.
