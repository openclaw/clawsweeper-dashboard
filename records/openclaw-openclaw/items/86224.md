## Summary

Keep this PR open but do not merge it yet: the central fix is well-motivated, but the new `agentRuntimeOverride` shortcut is broader than the provider-bound runtime contract and can make the memory gates skip embedded maintenance for sessions the dispatcher would still run embedded.

## What This Changes

The PR updates the preflight-compaction and memory-flush gates to resolve the effective CLI runtime before skipping embedded maintenance, with regression coverage for configured CLI runtimes, auth-profile pins, and persisted runtime overrides.

## Best Possible Solution

Use one provider-bound effective-runtime helper for dispatch, preflight compaction, and memory flush so the gates skip embedded maintenance only when the actual run would execute through a compatible CLI runtime.

## Reproduction Assessment

Yes for the reported bug: current main gates on the raw canonical provider, while the PR discussion supplies live logs showing the wrong `isCli=false` path before the patch and normal Telegram replies after a resolver-based gate. The new review finding is source-reproducible from the PR line that accepts any CLI backend id in `agentRuntimeOverride` without binding it to the active provider.

## Solution Assessment

No, not yet: resolving the effective CLI runtime is the right direction, but this implementation should use the provider-bound override contract already present in session-runtime compatibility and dispatch. The safer fix is to share that resolution shape instead of duplicating a broader `isCliProvider(runtime)` shortcut.
