## Summary

Keep open: the underlying bug is still real on current main, but this PR still has a blocking resolver-parity issue. The memory gates now consult a broader shared session-runtime helper than the dispatch path uses, so they can skip embedded maintenance for a persisted CLI override while the actual reply dispatch remains embedded.

## What This Changes

The PR changes auto-reply preflight-compaction and memory-flush gates to resolve an effective CLI runtime before deciding whether to skip embedded maintenance, with regression tests for configured runtimes, auth-profile pins, and persisted overrides.

## Best Possible Solution

Use one authoritative runtime-resolution helper for dispatch and both memory gates, then prove parity for configured runtime policy, auth-profile routing, persisted CLI overrides, stale or mismatched overrides, and `openclaw`/`pi` cases.

## Reproduction Assessment

Yes. By source inspection, the PR's memory gates use the shared helper that returns provider-compatible CLI overrides, while both checked dispatch paths import the narrower local helper from `agent-runner-execution`; the PR tests also assert the wider override behavior.

## Solution Assessment

No. The requested direction is right, but this branch is not the best fix until the gates and dispatch use the same resolver or the persisted-override semantics are intentionally changed together.
