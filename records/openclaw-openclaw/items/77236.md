## Summary

Keep open: the PR addresses a real Telegram reasoning-default bug, but the latest diff still lets explicit `off` values fall through to a lower-precedence global default and can re-enable reasoning contrary to the documented precedence.

## What This Changes

The PR adds Telegram `reasoningDefault` fallback from session state to per-agent/global config, auth-gates configured defaults, adds dispatch tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Land a narrow Telegram resolver update that resolves any valid `off`/`on`/`stream` value by precedence under the configured-default auth gate, with regression tests for explicit `off`.

## Reproduction Assessment

Yes. Source inspection of the latest PR diff shows `reasoningLevel: "off"` and per-agent `reasoningDefault: "off"` fall through to lower-precedence global defaults; existing shared code and tests confirm `off` is a valid override.

## Solution Assessment

No. The fallback direction and auth gate are appropriate, but the implementation must treat `off` as a valid higher-precedence setting before checking lower defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Honor explicit off before lower defaults:** `extensions/telegram/src/bot-message-dispatch.ts:224-243`
  - body: The resolver only returns session and per-agent values when they are `on` or `stream`. With this fallback added, a stored `reasoningLevel: "off"` or per-agent `reasoningDefault: "off"` is skipped, so `agents.defaults.reasoningDefault: "stream"` can re-enable reasoning even though session and per-agent settings are documented to win. Return any valid `off`/`on`/`stream` higher-precedence value before checking lower defaults.
  - confidence: 0.95
