## Summary

Keep open: the PR fixes the main Telegram fallback path and the earlier explicit-`off` precedence blocker, but it still gates configured defaults on raw Telegram `CommandAuthorized` instead of the resolved authorization contract used by the shared resolver.

## What This Changes

The PR adds Telegram fallback from session reasoning state to per-agent and global `reasoningDefault`, with regression tests and a changelog entry.

## Best Possible Solution

Use the existing Plugin SDK command-auth surface to compute the same configured-default authorization gate as the shared resolver, then land the PR after conflict resolution and focused Telegram dispatch validation.

## Reproduction Assessment

Yes. Source inspection of current main shows a fresh Telegram session with configured `reasoningDefault` falls through to `off` because the resolver only checks stored session `on`/`stream` values.

## Solution Assessment

No. The fallback order and explicit-`off` handling are right, but the authorization gate should use resolved command authorization rather than raw `ctxPayload.CommandAuthorized`.
