## Summary

Keep open: the PR addresses the original Telegram fallback and the prior auth/changelog concerns, but it still has a precedence bug where explicit `off` values can be bypassed by a global `stream` default.

## What This Changes

The PR adds Telegram reasoning default fallback from session state to per-agent and global config, gates configured defaults by Telegram command authorization, adds dispatch tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow Telegram resolver fix that mirrors shared precedence for `off`/`on`/`stream` under the existing configured-default auth gate, with regression tests for explicit `off` overrides.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows `reasoningLevel: "off"` and per-agent `reasoningDefault: "off"` do not return before lower-precedence global defaults, so a global `stream` default can re-enable reasoning.

## Solution Assessment

No. The fallback direction and auth gate are right, but the implementation must treat `off` as a valid higher-precedence setting instead of only returning `on` and `stream`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor explicit off before falling through to global defaults:** `extensions/telegram/src/bot-message-dispatch.ts:223-240`
  - body: The resolver only returns session and per-agent values when they are `on` or `stream`. With this fallback added, a stored `reasoningLevel: "off"` or per-agent `reasoningDefault: "off"` is skipped, so `agents.defaults.reasoningDefault: "stream"` can re-enable reasoning even though session and per-agent settings are documented to win. Treat `off` as a valid resolved value before checking lower-precedence defaults.
  - confidence: 0.95
