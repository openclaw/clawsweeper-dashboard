## Summary

Keep open. Current main still ignores configured Telegram reasoning defaults, and the PR is pointed at the right bug, but the latest patch still gates defaults only on `ctxPayload.CommandAuthorized` and therefore misses owner-authorized Telegram senders; it also lacks after-fix real Telegram behavior proof.

## What This Changes

The PR adds Telegram fallback from session reasoning state to per-agent and global `reasoningDefault`, updates Telegram dispatch tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow Telegram fix after it uses the same resolved owner/authorized-sender reasoning gate as the shared resolver and the contributor supplies redacted real Telegram runtime proof; leave broader Feishu/helper extraction to the separate related PR if maintainers want that scope.

## Reproduction Assessment

Yes, from source inspection. On current main, a fresh Telegram session with `agents.list[].reasoningDefault` or `agents.defaults.reasoningDefault` set falls through to `off` because `resolveTelegramReasoningLevel` only reads stored session state.

## Solution Assessment

No. The fallback order is close, but the PR should use resolved owner/authorized-sender semantics instead of only `ctxPayload.CommandAuthorized`, then add a regression case for an owner-authorized Telegram sender.
