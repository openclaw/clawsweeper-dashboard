## Summary

Keep open. Current main still preserves `thinking` blocks for the reported custom `qiniu` Anthropic Messages route: there is no bundled `qiniu` provider, the unowned Anthropic fallback only enables `dropThinkingBlocks` for Claude model IDs, and the linked fix PR was closed unmerged. The report is valid, but the remaining fix needs maintainer choice on the public provider/model replay-policy boundary.

## What This Changes

The issue requests that OpenClaw stop sending Anthropic `thinking` content blocks for a custom `qiniu/moonshotai/kimi-k2.5` `anthropic-messages` route when the model is configured not to support reasoning.

## Best Possible Solution

Add a documented provider/model replay-policy path for Anthropic-compatible providers that reject `thinking` blocks, with regression coverage proving `qiniu/moonshotai/kimi-k2.5` drops or converts historical and outbound thinking while Claude signed-thinking replay remains intact.

## Reproduction Assessment

Yes. Resolve transcript policy for provider `qiniu`, API `anthropic-messages`, and model id `moonshotai/kimi-k2.5`: current main leaves `dropThinkingBlocks` false, so replay and outbound request sanitization do not remove `thinking` blocks; the reporter supplied the matching HTTP 400.

## Solution Assessment

Unclear. The old PR’s `dropThinkingBlockModelHints` approach is stale because legacy provider capabilities are deprecated and no longer read at runtime; maintainers should choose the supported provider hook, generic compat field, or plugin route first.
