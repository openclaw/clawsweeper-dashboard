## Summary

Keep open. Current main still leaves `dropThinkingBlocks` false for the reported custom `qiniu` Anthropic Messages route, the old fix PR is closed unmerged, and the latest maintainer live smoke only covered bundled Moonshot/OpenRouter paths while explicitly leaving the custom-route behavior unresolved.

## What This Changes

The issue asks OpenClaw to avoid replaying Anthropic `thinking` blocks to the custom `qiniu/moonshotai/kimi-k2.5` route when that Kimi-compatible API rejects them.

## Best Possible Solution

Add a provider-owned or documented generic replay policy for Anthropic-compatible Kimi routes that reject `thinking` blocks, with coverage that preserves signed Claude replay behavior.

## Reproduction Assessment

Yes, at source level. Resolving policy for provider `qiniu`, API `anthropic-messages`, and model `moonshotai/kimi-k2.5` leaves `dropThinkingBlocks` false, and both replay-history and outbound-request cleanup are gated on that flag.

## Solution Assessment

Unclear. The old PR's legacy `provider-capabilities` approach is not the right current path because runtime policy now comes from provider replay hooks or the unowned fallback; maintainers should choose the public config/plugin boundary first.
