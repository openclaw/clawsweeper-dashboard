## Summary

Keep open. Current main still preserves `thinking` blocks for the reported custom `qiniu` Anthropic Messages route, and the only linked fix PR was closed unmerged; the remaining decision is the provider/plugin replay-policy boundary, not an already-shipped fix.

## What This Changes

The issue requests that Anthropic-compatible Kimi routes such as `qiniu/moonshotai/kimi-k2.5` stop replaying or sending `thinking` content blocks when the upstream API rejects them.

## Best Possible Solution

Ship a provider-owned Qiniu/Kimi replay policy, or a documented generic replay-policy configuration, that drops unsupported thinking blocks for this route while preserving Claude signed-thinking behavior.

## Reproduction Assessment

Yes. Source inspection against current main shows the reported `qiniu` + `anthropic-messages` + `moonshotai/kimi-k2.5` path resolves no `dropThinkingBlocks` policy, so replay and outbound request sanitization leave `thinking` blocks intact.

## Solution Assessment

Unclear. The closed PR’s provider-capability hint approach is stale against the current provider hook model; a provider-owned `buildReplayPolicy` or explicit generic replay config is safer than a broad Kimi heuristic in core.
