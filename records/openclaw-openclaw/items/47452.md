## Summary

Keep open. Current main still preserves `thinking` blocks for the reported custom `qiniu` Anthropic Messages route, the old fix PR is closed unmerged, and the remaining fix needs a maintainer choice on the public provider/model replay-policy boundary.

## What This Changes

The issue asks OpenClaw to avoid sending Anthropic `thinking` content blocks to `qiniu/moonshotai/kimi-k2.5` when the configured model says reasoning is unsupported.

## Best Possible Solution

Add a documented replay-policy path for Anthropic-compatible providers that reject `thinking` blocks, with regression coverage for `qiniu/moonshotai/kimi-k2.5` and preservation of Claude signed-thinking replay behavior.

## Reproduction Assessment

Yes. A source-level reproduction is available by resolving transcript policy for provider `qiniu`, API `anthropic-messages`, and model `moonshotai/kimi-k2.5`: current main leaves `dropThinkingBlocks` false, so replay and outbound request sanitization do not remove `thinking` blocks.

## Solution Assessment

Unclear. The old PR's capability-hint approach is stale against the current provider hook model; maintainers should choose whether this belongs in a Qiniu/provider plugin, a generic model compat field, or documented plugin configuration before implementation.
