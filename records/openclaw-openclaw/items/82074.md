Keep open: current main and the latest shipped tag still build Discord voice-message ffmpeg args without an explicit OGG muxer while the atomic writer passes a `.ogg.part` staging path, so the report matches the source, tests, docs contract, and FFmpeg behavior.

## What This Changes

Fix Discord voice-message TTS delivery so non-OGG audio can be transcoded to OGG/Opus even when the safe temporary output path ends in `.part`.

## Best Possible Solution

Land a focused Discord plugin fix that keeps atomic fs-safe staging but makes the ffmpeg OGG/Opus conversion format-explicit, with regression coverage for the staged `.ogg.part` output path.

## Solution Assessment
