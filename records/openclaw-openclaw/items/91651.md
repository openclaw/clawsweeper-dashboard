## Summary

Keep open: current main still rejects the reported malformed URL, and three open pull requests are competing to implement the repair. The issue should remain the canonical problem record until maintainers select and land one narrowly scoped fix.

## What This Changes

Make `web_fetch` tolerate model-generated whitespace between an HTTP(S) scheme separator and the URL authority.

## Best Possible Solution

Land one canonical patch that trims the argument and removes whitespace only immediately after an `http://` or `https://` separator, while preserving path/query semantics and proving the repaired destination still passes through the existing URL and SSRF guards.

## Reproduction Assessment

Yes at source level: current main forwards the exact malformed argument to `new URL()` unchanged, which deterministically takes the reported error path. The intermittent model behavior that generates the malformed argument was not reproduced live.

## Solution Assessment

A narrow scheme-authority repair is the best maintainable direction; broad whitespace deletion or a fallback that repairs every parse failure is not, because it can rewrite destination semantics at a guarded network boundary.
