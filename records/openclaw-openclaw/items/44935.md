Keep open: current main has request-time cooldown probes, sticky auto-fallback state, and verbose-gated fallback notices, but it still lacks the requested background health-probe recovery loop, configurable recovery interval, and default recovery notification behavior.

## What This Changes

The issue requests health-probe-based automatic recovery from fallback models back to the primary model for new sessions, subagents, and cron work while keeping existing sessions sticky.

## Best Possible Solution

Keep this as the canonical recovery-policy issue and extend the existing cooldown-probe, auto-fallback state, and fallback-notice substrate so recovered primaries are used for new work without blind-switching active fallback sessions.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an existing current-main contract. Source and docs give high confidence that current main only implements request-time cooldown probing and sticky fallback state, not the requested scheduler/config/default-notification behavior.

## Solution Assessment

Unclear until maintainer product design is done. The requested direction is plausible, but the maintainable path is to extend the existing fallback state and notice mechanisms rather than add a blind switchback or ad hoc timer.
