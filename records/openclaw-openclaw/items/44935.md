## Summary

Keep open. Current main has per-run fallback, cooldown probing, sticky auto-fallback session overrides, and fallback lifecycle/status notices, but it still does not provide the requested background recovery loop, durable recovered-primary state for future work, configurable recovery interval, or recovery notification behavior tied to the open companion notification request.

## What This Changes

Requested health-probe-based model fallback recovery that returns new sessions, subagents, and scheduled work to the primary model after it recovers while keeping existing fallback sessions sticky.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, records primary availability for future routing, moves only new sessions/subagents/cron runs back to the recovered primary, keeps active fallback sessions pinned, documents config defaults, and coordinates recovery notifications with #44941.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request involving provider outage timing. Static inspection gives high confidence that current main lacks the requested background recovery loop, future-work routing state, configurable interval, and recovery notification surface.

## Solution Assessment

Unclear until maintainer design is done. The health-probe direction is plausible, but the maintainable fix should add an explicit recovery state/routing contract rather than clearing sticky session overrides or blindly switching active sessions back.
