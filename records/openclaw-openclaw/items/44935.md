## Summary

Keep open: this is the canonical recovery-policy request, and current main still only has sticky fallback plus per-candidate cooldown probing, not health-probe-based auto-return to the primary for new work.

## What This Changes

The issue requests health-probe-based automatic recovery from a fallback model to the primary model for new sessions, subagents, and scheduled work while keeping existing sessions sticky.

## Best Possible Solution

Define a maintainer-approved recovery state machine that records fallback origin, probes the primary on a controlled cadence, routes only new work back after a successful probe, keeps existing fallback sessions sticky, and coordinates notification/config docs with the open notification issue.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks the requested recovery state machine and configurable health-probe loop.

## Solution Assessment

Unclear until maintainers choose the product contract. The proposed direction is plausible, but a narrow automated fix would be risky without decisions on probe cadence, session inheritance, notification defaults, and config shape.
