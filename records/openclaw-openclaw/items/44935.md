## Summary

Keep open: current main has request-time cooldown probes, sticky auto-fallback state, primary-as-last-candidate recovery, and verbose-gated fallback-cleared notices, but it still lacks the requested background health-probe loop, configurable recovery interval/policy, and default recovery notification behavior. The narrower persisted-fallback report was closed as a duplicate of this issue, while notification behavior remains tracked separately in https://github.com/openclaw/openclaw/issues/44941.

## What This Changes

The issue requests health-probe-based automatic recovery from a fallback model back to the primary model for new work, while keeping existing fallback sessions sticky.

## Best Possible Solution

Keep this as the canonical recovery-policy thread and extend the existing fallback state, cooldown-probe, and lifecycle-notice substrate after maintainers define scheduler, config, and notification policy.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an existing current-main contract. Source and docs give high confidence that current main implements request-time cooldown probing and sticky fallback state, not the requested scheduler/config/default-notification behavior.
