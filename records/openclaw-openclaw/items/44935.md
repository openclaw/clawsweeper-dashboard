## Summary

Keep open: current `main` has request-time cooldown probes and sticky auto-fallback state, but it does not implement the requested background health-probe recovery loop, configurable recovery interval, or default recovery notification. The notification layer is still tracked separately at https://github.com/openclaw/openclaw/issues/44941, and https://github.com/openclaw/openclaw/issues/76108 was already closed back to this issue as the canonical recovery-policy thread.

## What This Changes

The issue requests health-probe-based auto-recovery from fallback to the primary model for new work, with existing sessions kept sticky, a configurable probe interval, and recovery notifications.

## Best Possible Solution

Keep this as the canonical design issue and extend the existing cooldown-probe, auto-fallback state, and fallback notice substrate so recovered primaries are used for new work without switching active fallback sessions; coordinate visible notification defaults with https://github.com/openclaw/openclaw/issues/44941.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an existing current-main contract. Source and tests give high confidence that current `main` only has request-time cooldown probing and sticky auto-fallback behavior, not the requested scheduler/config/notification pieces.

## Solution Assessment

Unclear until maintainer product design is done. The existing request-time probe and sticky-session mechanisms are the right substrate, but adding periodic health probing, recovery state, and default notification behavior is broader than a narrow bug fix.
