## Summary

Keep open: current main has request-time cooldown probes, sticky auto-fallback state, primary-as-last-candidate recovery, and verbose-gated fallback-cleared notices, but it still lacks the requested background health-probe loop, configurable recovery interval/policy, and default recovery notification behavior. The narrower persisted-fallback report was closed as a duplicate of this issue, while the notification layer remains separately tracked in https://github.com/openclaw/openclaw/issues/44941.

## What This Changes

The issue requests health-probe-based auto-recovery from fallback models back to the primary model for new work, while keeping existing sessions sticky on fallback and notifying users on recovery.

## Best Possible Solution

Keep this as the canonical recovery-policy issue and extend the existing cooldown-probe, auto-fallback state, and fallback-notice substrate so recovered primaries are used for new work without blind-switching active fallback sessions, coordinating notification defaults with https://github.com/openclaw/openclaw/issues/44941.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a bug with an existing current-main contract. Source and docs give high confidence that current main implements request-time cooldown probing and sticky fallback state, not the requested scheduler/config/default-notification behavior.

## Solution Assessment

Unclear until maintainer product design is done. The requested health-probe direction is plausible, but the maintainable path is to extend the existing fallback state/probe/notice mechanisms rather than add a blind switchback or ad hoc timer.
