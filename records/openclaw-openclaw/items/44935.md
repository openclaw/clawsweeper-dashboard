## Summary

Keep open: current main has partial cooldown-aware request-time probing and sticky auto fallback behavior, but it does not implement the requested background health-probe recovery loop, configurable probe interval, or recovery notification contract. The related notification request remains open, and a later duplicate explicitly points back to this issue as the canonical recovery-policy thread.

## What This Changes

The issue requests health-probe-based recovery from fallback models back to the primary model for new sessions, subagents, and scheduled work while keeping existing fallback sessions sticky.

## Best Possible Solution

Keep this as the canonical design issue and extend the existing cooldown-probe and auto-fallback state so recovered primaries are used for new work without switching active fallback sessions; coordinate any visible notification/config contract with https://github.com/openclaw/openclaw/issues/44941.

Not applicable: this is a feature/design request, not a broken behavior with an existing current-main contract. Source inspection gives high confidence that current main only has request-time cooldown probing and sticky auto fallback, while the requested scheduler/config/notification pieces are absent.

## Solution Assessment

Unclear until maintainer product design is done. The existing cooldown-probe and sticky-session mechanisms are the right substrate, but adding a periodic health probe, recovery state, and default notification behavior is broader than a narrow bug fix.
