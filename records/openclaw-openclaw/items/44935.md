## Summary

Keep open: current main has per-run fallback, cooldown probing, sticky auto-fallback session state, and fallback lifecycle notices, but it still lacks the requested background recovery loop that probes the primary and routes only new work back after recovery.

## What This Changes

Request a configurable health-probe-based fallback recovery mechanism that returns new sessions, subagents, and scheduled work to the primary model while keeping active fallback sessions sticky.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, records recovered primary availability for future routing, keeps active fallback sessions pinned, and coordinates recovery notices with #44941.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request involving provider outage timing. Source inspection gives high confidence that current main lacks the requested background recovery loop, durable recovered-primary routing state, configurable interval, and recovery notification surface.

## Solution Assessment

Unclear until maintainer design is done. The health-probe direction is plausible, but the maintainable fix should add an explicit recovery state and routing contract rather than clearing sticky session overrides or blindly switching active sessions back.
