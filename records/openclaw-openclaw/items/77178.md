## Summary

This should stay open: current main still has a Claude CLI live-session stdin path that can surface raw `write EPIPE`, and the existing EPIPE hardening covers other subprocess paths rather than this live Claude stdio path.

## What This Changes

The issue reports immediate `write EPIPE` when starting a Claude CLI OAuth live session from Docker/Unraid.

## Best Possible Solution

Handle broken stdin in the Claude CLI live-session/supervisor path so a closed Claude child pipe is converted into a controlled live-session failure or failover with useful stderr/exit context, not a raw uncaught or user-visible `write EPIPE`.

## Reproduction Assessment

Yes from source inspection, though not from a live container run: start the `claude-cli` live-session path and have the child close stdin or emit EPIPE before the first turn write. Current main writes through `writeTurnInput` and propagates the write error into the run, matching the report.

## Solution Assessment

Yes, a narrow fix in `src/process/supervisor/adapters/child.ts` and/or `src/agents/cli-runner/claude-live-session.ts` is the best direction. The repair should follow the existing EPIPE precedent for subprocess stdin while preserving Claude CLI exit/stderr reporting.
