## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77178. The observable bug is that Claude CLI OAuth on Docker/Unraid can fail immediately with `write EPIPE` when OpenClaw starts the `claude-stdio` live session, even though `claude --print` works in the container. Keep the fix in the Claude CLI runner/backend boundary; do not add a new config option, change unrelated provider routing, or relax the existing Claude auth env-clearing policy. Add a regression in `src/agents/cli-runner.spawn.test.ts` or the nearest focused helper test that simulates a first-turn Claude live-session stdin write failure such as `Object.assign(new Error("write EPIPE"), { code: "EPIPE" })` and/or an immediate child exit with stderr. Implement a narrow recovery in `src/agents/cli-runner/claude-live-session.ts` / `src/agents/cli-runner/execute.ts`: preserve warm stdio reuse on success, but on first-turn live-session stdin EPIPE or startup close either retry once through the normal one-shot Claude CLI `-p` path when safe or surface a `FailoverError` with captured Claude startup/auth stderr instead of a raw uncaught-looking EPIPE. Update `CHANGELOG.md` if the user-facing behavior changes. Stop and ask for maintainer review if the root cause requires changing Claude CLI runtime policy, Docker image packaging, or auth product behavior.
