## Summary

Keep this open: Claude CLI reuse is a documented supported path, and current main still routes Claude live-session turns through a long-lived child stdin pipe where an early child exit can surface as `write EPIPE` instead of a graceful CLI backend failure.

## What This Changes

The issue asks OpenClaw to stop failing with `write EPIPE` when a Docker/Unraid install uses the Claude CLI OAuth live-session backend.

## Best Possible Solution

Harden the Claude live-session managed stdin path so an exited or closed child produces a structured CLI backend failure or restart path instead of surfacing an uncaught `EPIPE`, then cover it with focused unit tests and the Docker Claude subscription smoke when credentials are available.

## Reproduction Assessment

No. The issue has clear Docker/OAuth steps and a source-level path through Claude live-session stdin, but this read-only review did not run a Claude OAuth Docker/Unraid session against current main.

## Solution Assessment

Yes. The narrow maintainable fix is in the generic managed child stdin or Claude live-session write/exit race, not in Docker or Unraid-specific configuration.

## Repair Work Prompt

Fix the Claude CLI live-session `write EPIPE` failure reported in https://github.com/openclaw/openclaw/issues/77178. Keep the boundary narrow: harden the managed child stdin path used by `claude-cli` live sessions so a child that exits before or during a turn write does not escape as an uncaught/non-fatal EPIPE and instead rejects the turn with a structured CLI backend failure or restarts/cleans up consistently. Start in `src/agents/cli-runner/claude-live-session.ts` and `src/process/supervisor/adapters/child.ts`; add regression coverage in `src/agents/cli-runner.spawn.test.ts` and/or `src/process/supervisor/adapters/child.test.ts` for async stdin EPIPE/closed-pipe behavior. Use the existing `src/process/exec.ts` and MCP stdio EPIPE handling as precedent, but do not add new config options, Docker-specific policy, or broad provider behavior changes. Add a user-facing `CHANGELOG.md` fix entry if production behavior changes.
