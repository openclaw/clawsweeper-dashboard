## Summary

Keep this open: current main still has a plausible Claude live-session EPIPE path where an early-closing Claude child can surface a raw stdin write error and even hit the non-fatal uncaught-exception path instead of a useful Claude startup/auth diagnostic.

## What This Changes

The issue reports `write EPIPE` when OpenClaw spawns a Claude CLI OAuth live session in Docker/Unraid.

## Best Possible Solution

Harden managed child stdin and Claude live-session error mapping so early Claude CLI exits never produce uncaught EPIPE and instead surface actionable Claude stderr/auth/startup diagnostics, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a `claude-cli` live child that exits before consuming stream-json stdin will reject `stdin.write` with EPIPE, and the managed child adapter currently lacks a stdin error listener. I did not establish a live Docker/Unraid OAuth run.

## Solution Assessment

Yes, the narrowest maintainable fix is to handle `pipe-open` stdin errors in the process supervisor and make Claude live-session EPIPE prefer captured Claude stderr or a closed-before-turn diagnostic. Avoid changing Claude CLI default flags or Docker auth policy until the startup cause is known.
