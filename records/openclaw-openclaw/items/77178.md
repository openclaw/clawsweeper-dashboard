## Summary

Keep open: current main still has a clear Claude CLI live-session path that can surface `write EPIPE` when the child process exits before or during the first stdin write, and the supervisor child adapter does not have the EPIPE guard already present in the older one-shot exec helper.

## What This Changes

The issue reports an immediate `write EPIPE` when Docker/Unraid runs a Claude CLI OAuth live session and expects the bot conversation to start.

## Best Possible Solution

Handle premature Claude CLI child exit in the supervisor/live-session stdin path so EPIPE is not an uncaught user-facing run error, while preserving warm stdio sessions and reporting the real Claude CLI failure when the child exits.

## Reproduction Assessment

No live reproduction was established. Source inspection gives a clear current-main path: the Claude live session writes the first turn to an open child stdin pipe, and the child adapter lacks the EPIPE handling already present in the one-shot exec helper.

## Solution Assessment

Yes, a narrow fix in `src/process/supervisor/adapters/child.ts` and/or the Claude live-session write path is the best direction. The repair should convert premature-pipe closure into a controlled session failure or retry path with useful stderr, not a new config option.
