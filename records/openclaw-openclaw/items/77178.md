## Summary

Current main still has a Claude live-session stdin-write path that can surface raw `EPIPE` before any Claude result; the existing generic EPIPE guard covers one-shot exec, not this supervisor-backed live-session path.

## What This Changes

The issue reports an immediate `write EPIPE` when starting a Claude CLI OAuth-backed live session in the Docker image on Unraid.

## Best Possible Solution

Add a narrow live-session stdin error handling fix and regression test so a closed Claude child produces a controlled CLI startup/failover error and cleanup, without changing Claude CLI auth, model selection, or adding configuration.

## Reproduction Assessment

Yes from source inspection, but not from a live Docker run. A Claude live-session child that exits before the first JSONL turn can make the managed stdin write fail with `EPIPE`, which currently flows through the live-session abort/error path rather than a dedicated controlled startup error.

## Solution Assessment

Yes. The narrowest maintainable fix is on the Claude live-session stdin/supervisor write path, because the existing one-shot process EPIPE suppression does not cover `stdinMode: "pipe-open"` live sessions.
