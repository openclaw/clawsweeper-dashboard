## Summary

Keep open: the report matches a current OpenClaw-side EPIPE path in Claude CLI live sessions, where managed child stdin can emit an async error without a stream error listener.

## What This Changes

Handle EPIPE during Claude CLI live-session startup in Docker/OAuth setups so the conversation starts or fails with a useful Claude CLI error instead of an immediate broken-pipe run error.

## Best Possible Solution

Add generic managed-child stdin error handling in the process supervisor so async stdin EPIPE is routed through the current write/run failure path, with focused regression coverage and no new config or Claude-specific policy change.

Fix https://github.com/openclaw/openclaw/issues/77178. Current main's Claude CLI live session writes prompt JSONL to `ManagedRun.stdin`, but `src/process/supervisor/adapters/child.ts` does not listen for async `child.stdin` error events, so `write EPIPE` can surface as an uncaught benign exception and fail the turn. Add generic supervisor-level stdin error handling that prevents async EPIPE from escaping and routes it to the pending write callback/current run failure path. Do not add Claude-specific special cases, new config, dependency changes, or provider policy changes. Add focused regression coverage in `src/process/supervisor/adapters/child.test.ts` simulating a pipe-open child stdin EPIPE after a managed write; add or update `src/agents/cli-runner.spawn.test.ts` only if needed to prove Claude live-session propagation. Add a one-line user-facing `CHANGELOG.md` fix entry. Validate with targeted tests before any broad gate.
