## Summary

Keep open. The report matches a real current-main risk in the Claude CLI live-session path: OpenClaw keeps a `claude` stdio process open and writes turns to child stdin, but the process-supervisor stdin wrapper does not currently guard the underlying stream's async `error` event, so a fast-exiting Claude process can still surface `write EPIPE` instead of a controlled Claude CLI failure.

## What This Changes

The issue requests that Claude CLI OAuth-backed Docker sessions start a conversation instead of failing immediately with `write EPIPE`.

## Best Possible Solution

Harden the supervised `pipe-open` stdin boundary and Claude live-session startup so early Claude exits produce a controlled failure with captured stderr or retryable auth guidance rather than an uncaught/raw EPIPE.

## Reproduction Assessment

No, not as an end-to-end Docker/Unraid OAuth run. Source inspection gives a credible reproduction path: start a Claude live session whose child process exits before the first turn write, then the current managed stdin wrapper can surface `write EPIPE`.

## Solution Assessment

Yes. The narrow maintainable fix is to handle child stdin `error` events and write-callback failures at the process-supervisor/live-session boundary while preserving the existing Claude CLI env hardening and stderr-based failure reporting.
