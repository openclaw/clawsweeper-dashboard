## What This Changes

This PR adds one core rule to `skills/healthcheck/SKILL.md` requiring healthcheck alerts, reports, and logs to use the user's local timezone from the cron `Current time:` line or `agents.defaults.userTimezone` instead of raw UTC.

## Best Possible Solution

Merge a narrow healthcheck-skill instruction that makes user-local time the primary timestamp for scheduled healthcheck output, preferably with wording that permits UTC only as secondary diagnostic context if maintainers want that behavior.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current isolated cron prompts still include a mixed local/UTC `Current time:` line, and the current healthcheck skill has no rule telling the model which timestamp to display in user-facing alerts.

## Solution Assessment

Yes. For this focused PR, a bundled skill rule is the smallest maintainable fix because the cron runner and config already expose the needed local-time context; a structured formatter would be stronger but is a broader product change.
