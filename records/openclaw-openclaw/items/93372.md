## Summary

Keep open: this is a plausible emergency mitigation for the Telegram rich-message outage, but it is too narrow because the default partial-streaming path can still finalize visible replies through raw rich send/edit, and the external PR supplies only local tests rather than real Telegram proof.

## What This Changes

The PR changes Telegram default text sends and bot delivery helper sends from raw Bot API rich messages to legacy `sendMessage` with HTML rendering, 4096-character chunking, and updated tests.

## Best Possible Solution

Land a single Telegram rollback that covers normal final sends, bot delivery, outbound chunking expectations, and default draft preview finalization, then prove readable replies in a real Telegram client before merge.

## Reproduction Assessment

Yes at source/report level: current main still sends normal text and draft previews through rich-message paths, and the linked issue includes cross-client unsupported-message evidence. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No as submitted: moving `sendMessageTelegram` and `sendTelegramText` back to legacy `sendMessage` is the right direction, but the default streaming preview/finalization path remains rich and external real behavior proof is missing.
