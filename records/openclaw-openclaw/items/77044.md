## Summary

Keep open: current main has packaging guardrails for the next Discord plugin publish, but npm `latest` still resolves to the broken `@openclaw/discord@2026.5.2` artifact, so the user-visible update path remains broken until a fixed package is published and verified.

## What This Changes

The issue reports that `openclaw update` reinstalls a Discord plugin package whose metadata points at TypeScript source entries instead of loadable built JavaScript.

## Best Possible Solution

Publish a fixed `@openclaw/discord` package from current main or a release branch, verify the tarball contains built `dist` runtime files plus runtime metadata, then close this after the npm `latest` update is proven.

## Reproduction Assessment

Yes for source/artifact triage: the published `@openclaw/discord@2026.5.2` tarball has TypeScript-only entries and current install validation requires a built runtime entry for that shape. I did not run a live gateway update, but the registry artifact and loader contract are enough to reproduce the package-level failure path.

## Solution Assessment

Yes, the main-branch direction is the right fix path: build package-local runtime files, overlay runtime entry metadata during publish, and verify the published tarball. The issue should stay open until that fixed artifact is actually published and verified on npm.
