## Summary

Keep open: current main still has the confusing disable/list behavior, and the PR is a narrow, mergeable UX fix with sufficient terminal proof and no blocking code or security findings.

## What This Changes

The PR adds a disable-only stderr hint to the cron CLI and a service regression test for disable/list/enable visibility.

## Best Possible Solution

Keep the existing cron list semantics and land the stderr-only hint with focused regression coverage so users can find and re-enable disabled cron jobs without changing storage or filtering behavior.

## Reproduction Assessment

Yes. Source inspection of current main shows `cron disable` updates and prints JSON, default `cron list` sends `includeDisabled: false`, and the service filters disabled jobs unless `includeDisabled` is true; the PR body also supplies a live terminal round trip.

## Solution Assessment

Yes. A disable-only stderr hint is the narrowest maintainable solution because it points to `cron list --all` and `cron enable <id>` without changing list defaults, storage semantics, or stdout JSON.
