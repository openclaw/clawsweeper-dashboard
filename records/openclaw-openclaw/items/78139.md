## Summary

Keep open. Current main still has the disable/list confusion, and this PR addresses it narrowly with a disable-only stderr hint, focused service coverage, supplied terminal proof, and no blocking review findings.

## What This Changes

The PR adds a stderr note after `openclaw cron disable <id>` explaining `cron list --all` and `cron enable <id>`, plus a service regression test for the disable/list/enable round trip.

## Best Possible Solution

Keep the existing list semantics and land the stderr-only hint with regression coverage so users can find and re-enable disabled cron jobs without changing cron storage or filtering behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows disable updates the job, default `cron list` sends `includeDisabled: false`, and the service filters disabled jobs unless `includeDisabled` is true; the PR body also supplies a live terminal round trip.

## Solution Assessment

Yes. A disable-only stderr hint is the narrowest maintainable fix because it explains `cron list --all` and `cron enable <id>` without changing list defaults, storage semantics, or stdout JSON.
