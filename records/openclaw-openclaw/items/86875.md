## Summary

Keep open for changes before merge: the central fix matches the documented hook contract and the supplied terminal proof is adequate, but the newly added delimiter scanner recurses over inline marker false positives and can stack-overflow on large prompt or hook text. The hook-placement behavior also changes where plugin prompt context lands, so maintainers should consciously accept that compatibility shift before landing.

## What This Changes

The PR adds a `modelPrompt` path to embedded runtime-context splitting so `before_prompt_build` prepend/append text stays in the model prompt while marked internal runtime context is hidden, with helper and runner regression tests.

## Best Possible Solution

Land the hook-context separation after replacing the recursive delimiter scan with an iterative/shared parser and explicitly accepting the plugin-hook compatibility change.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main strips any prompt additions around `transcriptPrompt` into runtime context. The PR's helper and runner tests cover the intended fixed path, though I did not run tests in this read-only review.

## Solution Assessment

No, not quite yet. Keeping hook context prompt-local is the right direction, but the new delimiter scan should be iterative or shared with the existing internal runtime-context parser before merge.
