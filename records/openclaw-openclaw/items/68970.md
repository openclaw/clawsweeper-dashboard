## Summary

Keep open. Current main still has a TUI-specific reconnect/backfill gap: Web UI subscribes to transcript/session events and reloads history on relevant `session.message` updates and sequence gaps, while TUI only does a one-time reconnect history load and reports gaps without backfill.

## What This Changes

The issue asks for TUI reconnect/backfill recovery so a post-gateway-restart heartbeat reply appears in the active TUI session as it does in Web UI.

## Best Possible Solution

Add TUI-owned transcript/gap backfill using the existing gateway session event APIs, with active-run deferral matching the Web UI pattern.

## Reproduction Assessment

Yes, at source level: simulate a reconnect or websocket sequence gap where the heartbeat reply is appended after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI. Current main has no TUI `session.message` handler or gap-triggered history reload to recover that append.

## Solution Assessment

Yes. The narrow maintainable fix is TUI-side transcript/gap backfill through the existing gateway session-message path, not a heartbeat, delivery-context, or routing rewrite.
