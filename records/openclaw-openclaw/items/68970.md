## Summary

Keep open. Current main still lacks TUI subscription or handling for `session.message` transcript updates, while the Gateway and Web UI already have that path; the active related PR is the right implementation candidate but is still open and not proven for this restart-heartbeat timing case.

## What This Changes

The issue reports that gateway-mode TUI misses a recovery reply generated after a gateway restart while Web UI shows the same session reply.

## Best Possible Solution

Land or replace the active TUI live transcript subscription fix so TUI subscribes to the active session across reconnects and session switches, with regression coverage for post-restart heartbeat reply visibility.

## Reproduction Assessment

Yes at source level, but not live-reproduced in this read-only review. Gateway/Web UI support subscribed `session.message` backfill, while TUI currently relies on gateway `chat` events plus a single reconnect `loadHistory()` and does not subscribe to `session.message`.

## Solution Assessment

Yes, the best direction is the existing active-session transcript subscription path, not changing heartbeat delivery. https://github.com/openclaw/openclaw/pull/73402 appears to be the right owner path, but it should add or prove this restart-heartbeat acceptance case before the issue closes.
