## Summary

Keep open. Current main and v2026.5.18 still lack a TUI `session.message` subscription or post-gap backfill path, while https://github.com/openclaw/openclaw/pull/73402 remains the active maintainer-labeled implementation candidate and has not yet proven this post-restart heartbeat timing case.

## What This Changes

The issue asks TUI to show the same post-restart heartbeat recovery reply that is already visible in Web UI for the active session.

## Best Possible Solution

Fold this scenario into the active TUI live-transcript work: TUI should subscribe to `session.message` for the active session, resubscribe on reconnect/session switch, and perform bounded post-gap backfill without clobbering active streams.

## Reproduction Assessment

Yes, at source level. Current TUI reconnect performs a single `loadHistory()` and has no `session.message` subscription or gap backfill, so a heartbeat reply appended after that load with no TUI-routed chat event remains invisible until another manual reload.

## Solution Assessment

Yes. The maintainable fix is TUI-owned use of the existing `sessions.messages.subscribe` / `session.message` contract plus bounded reconnect/gap backfill, not a heartbeat or delivery-context rewrite.
