## Summary

Keep open. Current main still has the reported TUI recovery gap: TUI reloads history once on reconnect and only handles live chat/agent events, while Control UI also recovers from sequence gaps and subscribed transcript updates.

## What This Changes

TUI should show the same post-restart recovery reply that Web UI shows for the active session.

## Best Possible Solution

Add TUI-owned reconnect/backfill parity for the watched session using the existing transcript subscription or a debounced idle history reload on gaps, with active-run deferral and focused tests.

## Reproduction Assessment

Yes. A focused test can simulate a reconnect or sequence gap where a heartbeat reply is persisted after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is to add TUI transcript/gap backfill behavior rather than relying solely on the active-run watchdog.

## Repair Work Prompt

Fix the TUI post-restart/reconnect backfill gap from https://github.com/openclaw/openclaw/issues/68970. Current main handles only live `chat`, `chat.side_result`, and `agent` events in `src/tui/tui.ts`; `onGap` only updates status, and reconnect performs one immediate `loadHistory()`. Add TUI-owned recovery so the currently watched session backfills persisted replies that arrive after reconnect, preferably by using `sessions.messages.subscribe`/`session.message` or an equivalent debounced idle history reload. Defer reloads while `state.activeChatRunId` is active and flush after terminal chat/error/abort, following the Control UI pattern. Keep the fix TUI-owned; do not add core owner-specific routing policy or change heartbeat semantics. Add a user-facing `CHANGELOG.md` fix entry. Validate with targeted TUI tests and formatter checks; run changed gate via Testbox if it expands broadly.
