## Summary

Keep open. Current main still leaves the reported TUI recovery gap: TUI only consumes live chat/side-result/agent events, does one reconnect history load, and only uses reconnect watchdog recovery when a tracked active run exists; it does not backfill idle post-reconnect transcript updates the way Control UI does.

## What This Changes

The issue asks for TUI to show the same post-restart heartbeat recovery reply that is already visible in Web UI.

## Best Possible Solution

Add TUI-owned reconnect/gap backfill for the watched session using the existing `sessions.messages.subscribe` path or a debounced idle history reload, with active-run deferral and focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: simulate a reconnect or sequence gap where a heartbeat reply is appended after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI. I did not run the live TUI/gateway restart flow in this read-only review.

## Solution Assessment

No, current main is still only a partial solution. The narrow maintainable fix is to add TUI transcript/gap backfill behavior with active-run deferral, reusing the existing Gateway API rather than relying solely on reconnect watchdog state.
