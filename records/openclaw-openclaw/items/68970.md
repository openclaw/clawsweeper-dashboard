## Summary

Keep open. Current main still has the same TUI reconnect/backfill gap: TUI does one reconnect history load and handles live chat/agent events, but it does not handle `session.message` transcript events or reload on websocket sequence gaps the way Web UI does.

## What This Changes

TUI should show the post-restart heartbeat recovery reply for the watched session when the same reply is visible in Web UI.

## Best Possible Solution

Add TUI-owned transcript/gap backfill for the watched session using the existing gateway `session.message` APIs, with active-run deferral matching the Web UI pattern.

## Reproduction Assessment

Yes, at source level: simulate a reconnect or websocket sequence gap where the heartbeat reply is appended after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI. Current main has no TUI transcript-event handler or gap-triggered reload to recover that append.

## Solution Assessment

Yes. The narrow maintainable fix is TUI-side transcript/gap backfill through the existing gateway session-message path, not a heartbeat, delivery-context, or routing rewrite.
