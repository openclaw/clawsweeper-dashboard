## Summary

Keep open. Current main still has a TUI reconnect/backfill gap: TUI does one reconnect history load and handles live chat/agent events, but it still has no transcript-event or event-gap recovery path for a post-restart reply that lands after that load.

## What This Changes

Ensure `openclaw-tui` shows the same post-restart heartbeat recovery reply that is already visible in Web UI.

## Best Possible Solution

Add TUI-owned transcript/gap backfill for the watched session using the existing gateway transcript event APIs, with active-run deferral and focused regression tests.

Yes, at source level: simulate a reconnect or sequence gap where the heartbeat reply is appended after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI. Current main has no TUI transcript-event handler or gap-triggered reload to recover that append.

## Solution Assessment

Yes. The narrow maintainable fix is TUI-side transcript/gap backfill through the existing gateway session-message path, not a heartbeat, delivery-context, or routing rewrite.
