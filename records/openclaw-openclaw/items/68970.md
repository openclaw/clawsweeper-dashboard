## Summary

Keep open. Current main still has the TUI recovery gap: TUI handles live `chat`, `chat.side_result`, and `agent` events, performs one reconnect history load, and reports sequence gaps without forcing reconnect or history backfill; it still does not consume persisted `session.message` transcript updates like Web UI does.

## What This Changes

The issue requests that TUI show the same post-restart heartbeat recovery reply that Web UI shows for the watched active session.

## Best Possible Solution

Add TUI-owned reconnect/gap transcript backfill for the watched session, preferably through the existing `sessions.messages.subscribe` path with active-run deferral and focused regression tests.

Yes, at source level: simulate a reconnect or sequence gap where a heartbeat reply is appended after TUI's immediate reconnect `loadHistory()` and no matching live `chat` event reaches TUI. The current TUI code has no transcript-event or gap-reload path to recover that append.

## Solution Assessment
