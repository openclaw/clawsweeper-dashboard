## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Port Mermaid rendering to the current fence renderer:** `ui/src/ui/markdown.ts:247-260`
  - body: This hunk targets the old `htmlEscapeRenderer.code` path, but current main now uses `md.renderer.rules.fence` with highlight.js, task-list, URL, sanitizer, and cache behavior. Resolving the conflict by carrying this old renderer forward would either fail to apply or risk dropping current markdown behavior, so the feature needs to be reimplemented on the current markdown-it fence renderer.
  - confidence: 0.91
- **[P2] Only emit Mermaid placeholders where rendering is scheduled:** `ui/src/ui/markdown.ts:260-263`
  - body: `toSanitizedMarkdownHtml()` is shared by Markdown sidebar, agent file preview, cron, and side-result views, but this PR only schedules Mermaid rendering from the chat lifecycle. Those other views would receive an empty `.mermaid-block__render` with the source hidden in `<details>`, so Mermaid fences can become blank panels instead of readable code.
  - confidence: 0.88
- **[P2] Let failed Mermaid blocks render again after transient errors:** `ui/src/ui/mermaid.ts:141-143`
  - body: The renderer only selects `.mermaid-block:not([data-mermaid-status])`, but streamed Mermaid output and temporary chunk/render failures can set `data-mermaid-status="error"` before the definition is complete. Later updates then skip the same DOM node forever, leaving a stale fallback until the transcript is rebuilt.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after inserting the SVG:** `ui/src/ui/mermaid.ts:183-185`
  - body: Mermaid 11.13.0 documents `bindFunctions` on the render result for attaching diagram event handlers after insertion. This code only inserts `svg`, so diagrams that depend on Mermaid interactions such as tooltips or click bindings render as static content with silently broken behavior.
  - confidence: 0.82
