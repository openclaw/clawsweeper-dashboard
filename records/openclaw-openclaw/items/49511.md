## Summary

Keep open: this is a plausible WebChat feature and current main still does not render Mermaid fences, but the PR cannot merge as-is because it is stale against the current markdown-it renderer and adds a security-sensitive SVG/dependency boundary with unresolved rendering gaps.

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat markdown fences, Mermaid-specific styles/tests, a Mermaid dependency, and package/lockfile updates.

## Best Possible Solution

Land a focused current-main Mermaid implementation in the markdown-it fence renderer or a clearly scoped browser render pass, preserving sanitizer, task-list, code-block, i18n, and shared-markdown contracts.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review blockers are source-reproducible by comparing current main's markdown-it fence renderer with the PR head's marked renderer, lifecycle scheduling, and Mermaid render-error paths.

## Solution Assessment

No. Mermaid rendering is a reasonable feature direction, but this branch is not the best merge shape until it is ported to current main's renderer and the Mermaid SVG/dependency boundary is accepted by maintainers.

  - body: Current main renders shared UI markdown through `md.renderer.rules.fence` after the markdown-it security migration, but this branch adds Mermaid handling to the stale marked renderer path.   - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but the branch rewrites every Mermaid fence into an empty render target while only chat lifecycle updates call `scheduleMermaidRender()`. Non-chat markdown consumers can show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule Mermaid rendering after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this scheduler. Those diagrams stay blank until an unrelated chat update or tab switch occurs.
  - confidence: 0.87
- **[P2] Allow transient Mermaid failures to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects blocks without `data-mermaid-status`, then marks transient import, parse, streaming, or render failures as `error`. A partial streamed diagram or temporary chunk failure can permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted so links, tooltips, and click handlers work. This branch only uses `svg`, so interactive diagrams silently degrade.
  - confidence: 0.84
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. Users see an empty panel with the source collapsed instead of a normal code fence or actionable render failure.
  - confidence: 0.82
