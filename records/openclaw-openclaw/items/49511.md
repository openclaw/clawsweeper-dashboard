## Summary

Keep open: this is the canonical active PR for built-in WebChat Mermaid rendering, but the current branch should not merge as-is because it is stale against the current markdown-it renderer and still has source-reproducible rendering and security-boundary blockers.

## What This Changes

The PR adds lazy-loaded Mermaid rendering, styling, tests, a UI dependency, and lockfile changes for Mermaid code fences in Control UI WebChat.

## Best Possible Solution

Land a narrower current-main implementation that preserves markdown-it, sanitizer/task-list/code-block contracts, scopes Mermaid rendering to consumers that schedule it, and calls Mermaid's full render contract after dependency/security approval.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head and current main: the branch uses the older marked renderer, rewrites shared markdown without shared scheduling, and drops Mermaid bindFunctions.

## Solution Assessment

No. Mermaid rendering may be a valid feature, but this branch should be ported to the current markdown-it path or implemented as an explicitly scoped post-render pass before merge.

Full review comments:

- **[P1] Port Mermaid support to markdown-it:** `ui/src/ui/markdown.ts:2`
  - body: Current main moved shared UI markdown rendering to markdown-it for parser hardening, task lists, sanitizer behavior, cache, and localized code-block copy labels. This branch still imports `marked` and calls `marked.parse`, so merging it would replace that current renderer instead of extending it.
  - confidence: 0.94
- **[P2] Render Mermaid only where the scheduler runs:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but this branch rewrites every Mermaid fence into an empty `.mermaid-block__render` while only the chat lifecycle schedules rendering. Sidebar and other shared markdown consumers can show a blank panel with the source hidden under details.
  - confidence: 0.9
- **[P2] Retry failed Mermaid blocks on later renders:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer selects only blocks without `data-mermaid-status`, then marks import, streaming, parse, or render failures as `error`. A partial streamed diagram or transient chunk failure can permanently skip the same DOM node after the definition becomes valid.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that must run after the SVG is inserted so links, tooltips, and click handlers work. This branch's API type drops that property and only uses `svg`, so interactive diagrams silently degrade.
  - confidence: 0.86
- **[P2] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid source is empty, this branch marks the block as `error` but never writes fallback text into the render target. Users then see a blank panel with the source collapsed instead of an actionable render failure.
  - confidence: 0.82
