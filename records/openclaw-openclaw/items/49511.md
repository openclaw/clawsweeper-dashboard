## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Render Mermaid wherever the shared helper rewrites fences:** `ui/src/ui/markdown.ts:263-264`
  - body: This changes the shared markdown renderer so every ```mermaid fence becomes a `.mermaid-block` with an empty render slot, but the only render trigger added is the chat lifecycle call to `scheduleMermaidRender()`. Other current consumers of `toSanitizedMarkdownHtml()` such as markdown sidebars, cron details, skills, and side results can insert these placeholders without any local render pass, leaving users with a blank panel and collapsed source. Scope the Mermaid rewrite to surfaces that schedule rendering, or make the renderer install/schedule from every consumer that can create the block.
  - confidence: 0.91
- **[P2] Call Mermaid bindFunctions after inserting the SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11.13.0 returns optional `bindFunctions` from `render()` and its own `run()` path calls that callback after assigning `innerHTML`. This branch destructures only `{ svg }`, so diagrams that rely on Mermaid's post-insert bindings, such as tooltips or supported click handlers, render as inert SVG even though Mermaid produced the binding callback. Preserve and invoke the callback safely after inserting the sanitized SVG, or explicitly disable/document unsupported interactions with tests.
  - confidence: 0.86
- **[P3] Show an error for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When a Mermaid fence has no definition, this branch marks the block as `error` but never calls `setMermaidRenderError()`. The render slot stays empty while the source remains tucked behind details, so malformed or empty Mermaid output looks like missing UI instead of an actionable fallback. Set the same visible fallback text used for render failures before marking the block errored.
  - confidence: 0.78
