Full review comments:

- **[P1] Port Mermaid handling to the current renderer:** `ui/src/ui/markdown.ts:241-264`
  - body: This PR still adds Mermaid handling to the old `marked` renderer, but current main now uses `markdown-it` and `md.renderer.rules.fence` for code blocks. Keeping this path during conflict resolution would drop current markdown behavior such as ReDoS hardening, syntax highlighting, and localized copy labels.
  - confidence: 0.91
- **[P1] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-151`
  - body: `renderMermaidBlocks()` only scans blocks without `data-mermaid-status`, but transient import/render failures and incomplete streamed fences are marked `error`. Once the message finishes or the chunk load recovers, the same DOM node is permanently skipped and the diagram never renders.
  - confidence: 0.88
- **[P2] Do not rewrite Mermaid fences where rendering is unscheduled:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by chat, sidebar markdown, and side-result rendering, but this branch rewrites every Mermaid fence into an empty render target while scheduling Mermaid only from the chat lifecycle. Shared panes can therefore show a blank Mermaid box with the source hidden instead of rendered content or a normal code fence.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:185-187`
  - body: Mermaid 11.13 can return `bindFunctions` from `render()` for diagram event listeners, but the PR narrows the API to `{ svg }` and only inserts the SVG. Diagrams that rely on Mermaid interactions such as clicks or tooltips will silently render as static output.
  - confidence: 0.78
