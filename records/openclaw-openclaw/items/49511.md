## Summary

Keep open for maintainer review. Current main still does not render Mermaid fences in WebChat, and the related issue/alternate PR chain now points here as the canonical implementation, but this branch is not merge-ready because it is based on the old marked renderer, leaves blank Mermaid placeholders in shared non-chat markdown surfaces, misses rerender paths, and adds a new SVG/dependency security boundary.

## What This Changes

The PR adds Mermaid rendering for Control UI WebChat markdown fences with a lazy Mermaid module, chat styling, tests, a Mermaid dependency, lockfile/root override changes, and lifecycle scheduling.

## Best Possible Solution

If maintainers want this feature, land a focused current-main implementation in the markdown-it renderer or an explicit adjacent render pass, covering every surface that emits Mermaid blocks and preserving sanitizer, task-list, code-block, cache, locale, retry, theme, and Mermaid bindFunctions behavior.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review blockers are source-reproducible from the PR head against current main's markdown-it renderer, shared markdown consumers, manual-refresh lifecycle, and Mermaid render contract.

## Solution Assessment

No. Mermaid rendering is a reasonable feature direction, but this branch is not the best merge shape because it bypasses the current hardened markdown-it renderer and does not consistently render or retry the blocks it emits.

- **[P1] Port Mermaid support to the markdown-it renderer:** `ui/src/ui/markdown.ts:202-206`
  - body: Current main renders shared markdown through markdown-it after the ReDoS-hardening migration, but this branch routes the same user-controlled surface back through marked.parse(). Implement the Mermaid branch in the current md.renderer.rules.fence path so the sanitizer, task-list, cache, locale, and parser-hardening contracts stay intact.
  - confidence: 0.94
- **[P2] Render Mermaid blocks in every shared markdown surface:** `ui/src/ui/markdown.ts:263-264`
  - body: toSanitizedMarkdownHtml() is used outside the chat transcript, but this branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle schedules rendering. Non-chat markdown previews can therefore show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule rendering after manual transcript refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: handleUpdated() still returns while chatManualRefreshInFlight is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this scheduler. Those diagrams stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects .mermaid-block elements without data-mermaid-status, then marks transient import, parse, streaming, or render failures as error. A partial streamed diagram or temporary chunk failure can therefore permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:185-187`
  - body: Mermaid render results can include bindFunctions that must run after the SVG is inserted for links, tooltips, and click handlers. This branch destructures only svg and discards that hook, so interactive diagrams silently degrade.
  - confidence: 0.84
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as error but never calls setMermaidRenderError. The user sees an empty panel with the source collapsed instead of the normal code fence or an actionable render failure message.
  - confidence: 0.82
