## Summary

Keep open. Current main still does not render Mermaid fences in Control UI chat, so the PR is not obsolete, but the branch is not merge-ready: it targets an older markdown renderer shape and still has concrete rendering lifecycle, retry, and Mermaid API integration bugs plus a sanitizer profile that needs maintainer security review.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI webchat fenced code blocks, with markdown output changes, chat lifecycle scheduling, Mermaid styles/tests, and pinned Mermaid-related dependency updates.

## Best Possible Solution

Refresh the implementation against current main's markdown-it renderer, decide whether Mermaid rewriting is chat-only or a shared markdown capability, then land a narrow version with complete render scheduling, retry/error fallback behavior, Mermaid bindFunctions support, theme handling, sanitizer tests, and focused UI/unit validation.

## Reproduction Assessment

Yes. On current main, a Control UI chat message containing a fenced ```mermaid block is reproducible as a generic code block because chat calls toSanitizedMarkdownHtml() and the active fenced-code renderer has no Mermaid branch; the PR findings are also reproducible by code-path inspection of shared sanitizer consumers and lifecycle scheduling.

## Solution Assessment

No. The feature direction is useful, but this PR is not yet the best final solution because it rewrites shared markdown while rendering only from chat lifecycle hooks, leaves retry/error paths incomplete, and omits part of Mermaid's render contract; a rebased, scoped implementation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Render Mermaid wherever markdown is rewritten:** `ui/src/ui/markdown.ts:261`
  - body: `toSanitizedMarkdownHtml()` is used by sidebar, cron, and side-result views too, but this branch turns every Mermaid fence into an empty render placeholder while only chat updates call `scheduleMermaidRender()`. Those consumers will show a blank diagram area with the source collapsed, so either scope the rewrite to chat or install/render Mermaid for every sanitizer consumer.
  - confidence: 0.88
- **[P2] Schedule rendering after chat-local rerenders:** `ui/src/ui/app-lifecycle.ts:121`
  - body: The only scheduler call is tied to `handleUpdated()` chat properties, but manual refreshes return early while `chatManualRefreshInFlight` is true and transcript search uses local `requestUpdate()` state. Both paths can insert fresh Mermaid blocks without this scheduler, leaving blank placeholders until an unrelated chat update or tab switch occurs.
  - confidence: 0.84
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:150`
  - body: The render pass only selects `.mermaid-block:not([data-mermaid-status])`, but module-load and render failures mark blocks as `data-mermaid-status="error"`. Even though the lazy import promise is reset after a load failure, the existing DOM blocks are skipped forever on later passes unless the DOM is rebuilt.
  - confidence: 0.86
- **[P3] Run Mermaid bind functions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's `render()` result can include `bindFunctions` that must run after the SVG is inserted for diagram event handlers such as clicks/tooltips. This code narrows the API to `{ svg }` and never calls the binder, so interactive Mermaid diagrams silently render as static diagrams.
  - confidence: 0.79
- **[P3] Show fallback text for empty definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When a Mermaid fence has no body, the code marks the block as `error` and continues without calling `setMermaidRenderError()`. Users get an empty panel with the source hidden instead of an actionable failure message, which regresses from the old visible code fence behavior.
  - confidence: 0.77
