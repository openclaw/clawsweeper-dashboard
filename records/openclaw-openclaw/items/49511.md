## Summary

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat markdown fences, Mermaid chat styling/tests, a Mermaid dependency, and root lockfile/override changes.

## Best Possible Solution

Land a narrower current-main Mermaid implementation that preserves markdown-it, sanitizer/task-list/code-block contracts, scopes rendering to consumers that schedule it, and calls Mermaid's full render contract after maintainers approve the SVG dependency.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head and current main: the branch uses the older marked renderer while main uses markdown-it, and the Mermaid render failure/bindFunctions issues are visible in the added source and Mermaid 11.13.0 types.

## Solution Assessment

No. Mermaid rendering may be a valid feature direction, but this branch should be ported to the current markdown-it path or an explicitly scoped post-render pass before merge.

  - body: Current main moved shared UI markdown rendering to markdown-it for parser hardening, task lists, sanitizer behavior, cache, and localized code-block copy labels.   - confidence: 0.9
- **[P2] Retry failed Mermaid blocks on later renders:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer selects only blocks without `data-mermaid-status`, then marks import, streaming, parse, or render failures as `error`. A partial streamed diagram or transient chunk failure can permanently skip the same DOM node after the definition becomes valid.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that must run after the SVG is inserted so links, tooltips, and click handlers work. This branch's API type drops that property and only uses `svg`, so interactive diagrams silently degrade.
  - confidence: 0.86
