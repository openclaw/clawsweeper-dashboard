Full review comments:

- **[P1] Port Mermaid rendering to the current markdown-it path:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch adds Mermaid handling inside the old marked-based renderer, but current main now uses markdown-it for this file and the PR is live-conflicting. Resolving the conflict by keeping this block would risk dropping the current ReDoS-hardened renderer, code highlighting, task-list handling, and sanitizer behavior, so the feature needs to be rewritten into the current `md.renderer.rules.fence` path before merge.
  - confidence: 0.93
- **[P2] Render Mermaid placeholders only where post-render runs:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat lifecycle, but this change turns every Mermaid fence into an empty render target while only `handleUpdated()` schedules `scheduleMermaidRender()`. Markdown sidebar and other shared markdown surfaces can therefore show a blank panel with the source hidden instead of a rendered diagram or a normal code fence.
  - confidence: 0.88
- **[P2] Retry Mermaid blocks after transient failures:** `ui/src/ui/mermaid.ts:149-150`
  - body: `renderMermaidBlocks()` only queries blocks without `data-mermaid-status`, but both the lazy-import failure path and render failure path mark existing blocks as `error`. Even though the import promise is reset, those DOM nodes are skipped on later passes, so a temporary chunk or parse failure can become a persistent broken diagram until the DOM is rebuilt.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11.13.0's render result can include `bindFunctions`, and upstream types document calling it after the SVG is inserted so diagram interactions work. This code discards that callback and inserts only sanitized SVG, so diagrams that rely on Mermaid tooltips or click bindings render as static output.
  - confidence: 0.84
