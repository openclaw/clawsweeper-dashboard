- **[P1] Port Mermaid rendering to the current markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch still inserts Mermaid handling into the old marked renderer, but current main's WebChat renderer is markdown-it with code highlighting, task-list handling, i18n copy labels, and ReDoS hardening. Resolving the conflict by carrying this path forward would replace those current behaviors, so the feature needs to be rewritten inside the current fence renderer before merge.
  - confidence: 0.94
- **[P2] Only emit Mermaid placeholders where rendering is scheduled:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by sidebar and other markdown surfaces, but the PR only schedules Mermaid rendering from chat lifecycle updates. Any non-chat consumer that receives a Mermaid fence will get an empty render pane with the source hidden, so the renderer either needs a scoped option/fallback or every consumer must schedule rendering.
  - confidence: 0.84
- **[P2] Let failed Mermaid blocks retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The scheduler only selects blocks with no `data-mermaid-status`, then marks load/render failures as `error`. A transient chunk, load, or render failure leaves the same DOM block permanently skipped even though the import promise can reset, so later updates should clear or retry failed blocks.
  - confidence: 0.87
- **[P2] Run Mermaid post-render bindings:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` to attach diagram interactions after the SVG is inserted. This code only consumes `svg`, so diagrams that rely on Mermaid click or tooltip bindings render as static or broken; include the post-render binding call after insertion.
  - confidence: 0.8
