## Summary

Keep open: the feature is not implemented on current main, but the current PR branch needs maintainer review and repair before merge because it is stale behind the current markdown renderer and still has source-reproducible rendering gaps.

## What This Changes

The PR adds lazy Mermaid rendering for fenced code blocks in Control UI webchat, with styling, SVG sanitization, tests, and Mermaid-related dependency updates.

## Best Possible Solution

Port Mermaid support onto current main's markdown-it fence renderer, keep current sanitizer and i18n behavior, decide whether built-in Mermaid is acceptable for Control UI, then schedule rendering for every insertion path and honor Mermaid's render API.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review findings are source-reproducible from the PR branch and current main: the branch uses the old renderer, rewrites a shared sanitizer while only chat schedules Mermaid, skips manual refreshes, and drops Mermaid `bindFunctions`.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to rebase onto the current markdown-it renderer, preserve the security migration, make the scheduling boundary match every consumer that can receive a Mermaid block, and follow the Mermaid render result contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Port Mermaid rendering to the current markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main replaced the old `marked` renderer with markdown-it to fix a Control UI markdown ReDoS freeze, but this branch still implements Mermaid inside the obsolete `htmlEscapeRenderer.code` path. Rebasing by taking this file would drop the current sanitizer/i18n/cache behavior and risk reintroducing the old parser; implement the Mermaid case in the current `md.renderer.rules.fence` path instead.
  - confidence: 0.9
- **[P2] Schedule rendering for every markdown consumer:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by markdown sidebar, side-result, cron, dreaming, and chat renderers, but this change turns every Mermaid fence into an empty render target while only the chat lifecycle schedules Mermaid. Non-chat markdown views can therefore show a blank panel with the source hidden unless an unrelated chat update runs the scheduler.
  - confidence: 0.88
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while refreshed history replaces `chatMessages`, so `handleUpdated()` returns before it reaches this new scheduler call. Mermaid blocks loaded by the refresh button can stay blank until another chat update or tab change happens.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include `bindFunctions` that must run after inserting the SVG so diagram callbacks and tooltips are wired. This code discards that function, so interactive Mermaid diagrams render as static SVGs with their interactions silently missing.
  - confidence: 0.84
