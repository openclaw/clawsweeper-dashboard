## Summary

Keep open: current main still renders Mermaid fences as ordinary code and has no Mermaid dependency, while this PR remains a useful but not-ready feature patch with source-reproducible rendering lifecycle, retry, and sanitizer concerns.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI webchat markdown fences, dedicated styling/interactions/tests, and Mermaid dependency/lockfile updates.

## Best Possible Solution

Refresh a narrow Mermaid implementation against current main's markdown-it renderer, render or preserve fallback consistently across every markdown surface, harden retry/fallback/sanitizer behavior, and add the required changelog plus focused UI coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no Mermaid renderer or dependency, and the PR defects are reproducible from the shared markdown renderer plus chat-only scheduling and retry selectors.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best final form because it targets stale renderer assumptions, rewrites shared markdown without rendering every consumer, leaves retry/fallback gaps, and needs explicit SVG sanitizer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope Mermaid rewriting to rendered surfaces:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside the chat transcript, but this patch rewrites every Mermaid fence into an empty `.mermaid-block__render` placeholder while only scheduling rendering from chat updates. Sidebar, cron, side-result, and preview markdown can therefore show a blank panel with source hidden under `<details>` instead of the previous visible code block.
  - confidence: 0.9
- **[P2] Schedule Mermaid after local chat rerenders:** `ui/src/ui/app-lifecycle.ts:121`
  - body: The scheduler is reached only from selected Lit `changed` fields and this function returns while `chatManualRefreshInFlight` is true. Manual refreshes and local search/toggle rerenders can insert fresh Mermaid placeholders without a render pass until an unrelated chat update happens.
  - confidence: 0.86
- **[P2] Let failed Mermaid blocks retry:** `ui/src/ui/mermaid.ts:150`
  - body: This selector skips any block that already has `data-mermaid-status`, but transient load/render failures and partial streamed definitions mark the existing node as `error`. Once the source or chunk load becomes valid, later render passes still skip the same block and leave it broken.
  - confidence: 0.88
- **[P3] Call Mermaid bind functions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11's `render()` result can include `bindFunctions` that callers must run after injecting the SVG. This code keeps only `svg`, so diagrams that rely on Mermaid click or tooltip bindings silently render as static SVG.
  - confidence: 0.8
- **[P3] Show fallback text for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid fence has no body, this branch marks the block as `error` without calling the error renderer. Users see an empty placeholder with collapsed source instead of an actionable failure message, which regresses from the old visible code fence.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `ui/src/ui/markdown.ts:263`
  - body: This is a user-facing Control UI feature, but the PR file list does not include `CHANGELOG.md`. Repo policy requires a changelog entry before merging user-facing `feat` changes.
  - confidence: 0.85
