## Summary

Keep open for maintainer review: current main still lacks WebChat Mermaid rendering, but this PR is unmergeable and should not merge as-is because it is stale behind the current markdown-it markdown renderer and misses rendering lifecycle contracts.

## What This Changes

The PR adds a Mermaid dependency and lazy renderer for Control UI WebChat markdown fences, with chat styling, lifecycle scheduling, tests, and package/lockfile updates.

## Best Possible Solution

If maintainers want built-in Mermaid, land a narrow port on current main's markdown-it fence renderer that preserves sanitizer/cache/i18n behavior, schedules rendering for every markdown surface that emits Mermaid blocks, and honors Mermaid's full render result contract.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review blockers are source-reproducible by comparing the PR head's marked-based renderer and scheduler with current main's markdown-it renderer and shared markdown call sites.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to port the feature onto the current markdown-it renderer and shared rendering lifecycle rather than merging the stale marked-based branch.

  - body: Current main moved this user-controlled markdown path to markdown-it, but this branch adds Mermaid inside the old `marked` renderer. Those views can show a blank diagram panel with the source hidden behind details.
  - confidence: 0.88
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: The refresh button replaces chat history while `chatManualRefreshInFlight` is true, and `handleUpdated()` returns before reaching this new scheduler call. Mermaid blocks loaded by manual refresh can stay blank until another chat update or tab switch happens.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that callers must run after inserting the SVG. This code only consumes `svg`, so diagrams that rely on Mermaid callbacks or tooltips render as static SVGs with interactions missing.
  - confidence: 0.84
