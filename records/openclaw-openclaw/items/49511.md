## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope Mermaid rendering to surfaces that schedule it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but this branch rewrites every Mermaid fence into a `.mermaid-block` placeholder while only `handleUpdated()` calls `scheduleMermaidRender()`. Markdown previews such as the sidebar can therefore show an empty render box with the source hidden instead of either a diagram or a normal code fence; make the Mermaid wrapper opt-in or schedule rendering wherever this wrapper is emitted.
  - confidence: 0.86
- **[P2] Allow Mermaid blocks to retry after load failures:** `ui/src/ui/mermaid.ts:149-150`
  - body: `loadMermaidApi()` resets the promise after a failed dynamic import, but the catch path marks each current block `data-mermaid-status="error"` and future passes only query blocks without a status. A transient chunk or network failure becomes permanent for those diagrams until the DOM is rebuilt, which defeats the advertised retry behavior.
  - confidence: 0.82
- **[P2] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the code block has an empty Mermaid definition, this branch sets the block status to `error` and continues without calling `setMermaidRenderError()`. Users see an empty diagram panel with the source collapsed, which regresses from the previous plain code-fence behavior for malformed or partial Mermaid blocks.
  - confidence: 0.84
