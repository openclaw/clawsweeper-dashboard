## Summary

Keep open: current main still does not implement Mermaid rendering, and the related closed issue/PRs point to this PR as the canonical remaining implementation. The branch should not be cleanup-closed, but it needs maintainer review because the head is conflicting/stale against current Control UI markdown code and still has source-review blockers.

## What This Changes

The PR adds lazy Mermaid rendering for Control UI WebChat code fences, collapsible source/preview styling, tests, and Mermaid dependency/lockfile updates.

## Best Possible Solution

Land a current-main implementation only if maintainers want built-in Mermaid support: port it onto the existing markdown-it renderer, preserve sanitizer/i18n/code-fence behavior, render every surface that emits Mermaid blocks or scope the feature to chat, and add focused UI/security regression coverage.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection gives high confidence that current main lacks the feature: Mermaid fences still go through the generic fenced-code renderer and `ui/package.json` has no Mermaid dependency.

## Solution Assessment

No, not as-is. The feature direction is plausible, but this branch is stale/conflicting and currently emits Mermaid placeholders from a shared renderer while scheduling rendering only from chat updates; a current-main, scoped rewrite is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Render emitted Mermaid blocks outside chat:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch changes the shared markdown renderer to emit `.mermaid-block` for every Mermaid fence, but the only scheduler call is in the chat lifecycle. `toSanitizedMarkdownHtml()` is also used by non-chat preview surfaces such as the markdown sidebar, so those views can insert an empty render target with the source hidden under `<details>` and never render it until an unrelated chat update. Scope the rewrite to chat or schedule rendering for each view that can emit these blocks.
  - confidence: 0.86
- **[P2] Let transient Mermaid load failures retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The load-failure path resets `mermaidApiPromise`, but the scheduler only selects blocks without `data-mermaid-status`. Once the catch path marks the current blocks as `error`, a later scheduler pass skips them even though the module promise is retryable, so a temporary chunk/network failure leaves existing diagrams stuck until the DOM is rebuilt.
  - confidence: 0.82
- **[P3] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When a Mermaid fence has no definition, this branch marks the block as `error` and continues without calling `setMermaidRenderError()`. Users then see an empty diagram panel with the source collapsed, which is a regression from the previous visible code fence for malformed or partial content.
  - confidence: 0.8
