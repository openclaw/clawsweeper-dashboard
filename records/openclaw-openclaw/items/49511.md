## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope Mermaid rendering to scheduled surfaces:** `ui/src/ui/markdown.ts:263-264`
  - body: This changes the shared markdown renderer to emit `.mermaid-block` for every Mermaid fence, but the only scheduler call is in the chat lifecycle. Non-chat consumers such as the markdown sidebar can now insert an empty render slot with the source hidden under `<details>` and never render it; either scope the rewrite to chat or schedule rendering for every surface that can emit these blocks.
  - confidence: 0.86
- **[P2] Schedule Mermaid after manual chat refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while `refreshChat()` replaces messages, so `handleUpdated()` returns before this new scheduler call. The later flag-clear update does not include the chat fields checked here, leaving refreshed transcripts with blank Mermaid render slots until another unrelated chat update or tab switch happens.
  - confidence: 0.83
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The loader resets `mermaidApiPromise` after an import failure, but this selector only revisits blocks with no `data-mermaid-status`. Once the catch path marks current blocks as `error`, a transient chunk or network failure leaves existing diagrams permanently skipped until the DOM is rebuilt.
  - confidence: 0.82
- **[P3] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When a Mermaid fence has no definition, this branch marks the block as `error` and continues without calling `setMermaidRenderError()`. Users see an empty panel with the source collapsed, which regresses from the previous visible code fence for malformed or partial content.
  - confidence: 0.8
