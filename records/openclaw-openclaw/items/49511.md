## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Port Mermaid rendering to the current fence renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch adds the Mermaid branch inside `htmlEscapeRenderer.code`, but current main no longer uses that marked renderer; it uses `md.renderer.rules.fence`. Port this into the markdown-it fence renderer so conflict resolution does not drop the feature or bypass current highlighting, sanitizer, task-list, i18n, and URL behavior.
  - confidence: 0.93
- **[P2] Schedule rendering wherever Mermaid blocks are emitted:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by sidebar, cron, dreaming, side-result, and chat surfaces, but this PR only schedules Mermaid rendering from the chat lifecycle. Any non-chat consumer can receive a `.mermaid-block` with an empty render target and collapsed source, so either scope the rewrite to chat or schedule/render Mermaid blocks everywhere the shared sanitizer output is used.
  - confidence: 0.88
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: `renderMermaidBlocks()` only queries blocks without `data-mermaid-status`, while the load/render catch paths mark transient failures as `error`. Because the module loader already resets after import failure, later render passes should be able to retry errored blocks instead of leaving the same diagram stuck until the DOM is rebuilt.
  - confidence: 0.86
- **[P3] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the source code block is empty, this path marks the block as `error` but never calls `setMermaidRenderError`, leaving an empty placeholder with the source collapsed. Empty or streamed-empty Mermaid fences should show the same actionable fallback text as other render failures.
  - confidence: 0.78
