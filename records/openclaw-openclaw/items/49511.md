## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Port Mermaid rendering to the current markdown-it fence renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This patch modifies an older marked-style renderer, but current main’s `toSanitizedMarkdownHtml()` is built around markdown-it with task-list handling, highlighting, sanitizer policy, and ReDoS hardening. Merging the feature in this shape would conflict with or replace current renderer behavior, so it needs to be ported onto the current fence pipeline.
  - confidence: 0.9
- **[P2] Only emit Mermaid placeholders where a render pass will run:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by non-chat markdown views, but the PR only schedules rendering from the chat lifecycle. Those views would receive an empty `.mermaid-block__render` with source hidden under `<details>` and no render pass, so Mermaid content becomes blank instead of staying readable as code or rendering in place.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to render again:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer queries only blocks without `data-mermaid-status`, then marks failures as `error`. A transient lazy-import failure or streamed partial Mermaid definition can therefore permanently skip the completed diagram until the DOM is rebuilt, leaving a stale fallback in the transcript.
  - confidence: 0.88
- **[P2] Run Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:177-178`
  - body: Mermaid 11.13.0’s render contract can return `bindFunctions` that callers must run after inserting the SVG. This code drops that callback, so diagrams that rely on Mermaid interactions, tooltips, or click bindings render as static output with behavior silently missing.
  - confidence: 0.8
