## Summary

Keep open: current main does not implement Mermaid rendering, and this PR still has blocking Control UI integration issues around shared markdown surfaces, manual refresh rendering, and Mermaid's render contract.

## What This Changes

The PR adds lazy Mermaid rendering for fenced code blocks in Control UI webchat, with styling, tests, SVG sanitization, and package lock updates.

## Best Possible Solution

Land a rebased Control UI implementation that either scopes Mermaid rendering to WebChat or explicitly supports every shared markdown consumer, schedules rendering after every DOM insertion path, preserves Mermaid bindFunctions, keeps SVG sanitization/source-copy UX, and adds focused UI tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. The happy path has screenshot proof, and the blocking review findings are source-reproducible from the PR branch and current-main renderer paths.

## Solution Assessment

No, not as currently implemented. The narrow maintainable solution is to integrate Mermaid at the actual shared markdown/rendering boundary, cover manual refresh and local rerender paths, and follow Mermaid's render API contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Render Mermaid only where a render pass is scheduled:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is also used by the markdown sidebar, but this change rewrites every mermaid fence into an empty `.mermaid-block__render` while only the chat lifecycle calls `scheduleMermaidRender()`. Opening tool output with Mermaid can therefore show a blank panel with the source hidden until an unrelated chat update happens; either scope this rewrite to rendered chat surfaces or schedule Mermaid from every consumer that inserts the block.
  - confidence: 0.89
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual chat refresh sets `chatManualRefreshInFlight` while `refreshChat()` replaces `chatMessages`, so `handleUpdated()` returns before reaching this new scheduler call. The later flag clear does not change `chatMessages` or `tab`, leaving Mermaid blocks loaded by Refresh blank until some other update occurs.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after inserting the SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result includes optional `bindFunctions` that must run after the SVG is inserted for diagram event listeners such as tooltips and click callbacks. This code destructures only `svg`, so valid interactive Mermaid diagrams render as static SVGs with their interactions silently dropped.
  - confidence: 0.82
