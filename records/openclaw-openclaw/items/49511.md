  - body: This branch adds the Mermaid special case inside the old `marked.Renderer().code` path, but current main renders fences through `markdown-it` and `toStreamingMarkdownHtml`.   - confidence: 0.93

  - body: Adding `lodash-es` to root pnpm overrides changes dependency resolution for the whole workspace even though this is a Control UI feature. Please remove the override unless a maintainer explicitly approves that supply-chain change with regenerated current-main lockfile proof.
  - confidence: 0.9
- **[P1] Let Mermaid blocks retry after transient errors:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer only queries `.mermaid-block:not([data-mermaid-status])`, but streamed or transiently invalid Mermaid can be marked `error` before the definition is complete. Later chat updates then skip the same block permanently, so completed streamed diagrams can stay stuck on the fallback text.
  - confidence: 0.87
- **[P2] Render Mermaid wherever the shared renderer emits it:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `toSanitizedMarkdownHtml()` is shared by sidebar and side-result views, but this PR only schedules Mermaid rendering from chat lifecycle updates. Those non-chat consumers can receive empty `.mermaid-block__render` placeholders with the source hidden under `<details>` unless the output is scoped or each root schedules rendering.
  - confidence: 0.88
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that callers should run after the SVG is inserted. This implementation only uses `svg`, so diagrams that rely on Mermaid tooltips or click bindings would render as static images with interactions silently broken.
  - confidence: 0.82
