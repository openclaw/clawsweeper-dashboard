## Summary

Keep open: current main still lacks built-in Mermaid rendering and this PR is the canonical remaining tracker after the older Mermaid issue and competing PR were closed. The branch should not merge as-is because it is stale against the current markdown-it renderer and has source-reproducible lifecycle, retry, and SVG-boundary issues.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat markdown fences, Mermaid-specific styling/tests, a Mermaid dependency and lockfile changes, plus a root lodash-es override.

## Best Possible Solution

Land a focused current-main implementation in the markdown-it fence renderer or a scoped post-render pass after maintainers accept the Mermaid/SVG dependency boundary, preserving sanitizer, task-list, code-block, cache, i18n, and shared markdown contracts.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head, current main's markdown-it renderer and shared markdown consumers, and Mermaid 11.13.0's render contract.

## Solution Assessment

No. Mermaid rendering may be a valid feature direction, but this branch is not the best merge shape until it is ported to current main and the SVG/dependency boundary is explicitly accepted.

Full review comments:

- **[P1] Port Mermaid handling to the markdown-it renderer:** `ui/src/ui/markdown.ts:2`
  - body: Current main moved shared UI markdown rendering to markdown-it for parser hardening, task lists, sanitizer behavior, cache, and localized code-block copy labels. This branch still imports `marked` and calls `marked.parse`, so merging it would replace that current renderer instead of extending it.
  - confidence: 0.94
- **[P2] Scope Mermaid placeholders to views that render them:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but this branch rewrites every Mermaid fence into an empty `.mermaid-block__render` while only the chat lifecycle schedules rendering. Sidebar and other shared markdown consumers can show a blank panel with the source hidden under details.
  - confidence: 0.9
- **[P2] Schedule Mermaid after chat-only rerenders:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` only schedules Mermaid for selected reactive fields and still returns while manual refresh is in flight. Manual refreshes and local chat search can insert fresh `.mermaid-block` nodes without hitting this scheduler, leaving diagrams blank until an unrelated update.
  - confidence: 0.88
- **[P2] Allow transient Mermaid failures to retry:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer selects only blocks without `data-mermaid-status`, then marks import, streaming, parse, or render failures as `error`. A partial streamed diagram or transient chunk failure can permanently skip the same DOM node after the definition becomes valid.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that must run after the SVG is inserted so links, tooltips, and click handlers work. This branch's API type drops that property and only uses `svg`, so interactive diagrams silently degrade.
  - confidence: 0.86
- **[P2] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. Users see an empty panel with the source collapsed instead of a normal code fence or actionable render failure.
  - confidence: 0.82
