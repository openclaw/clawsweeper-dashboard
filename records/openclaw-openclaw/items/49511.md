## Summary

Keep this PR open for maintainer review: current main still lacks built-in WebChat Mermaid rendering, but the branch is not merge-ready because it is stale against the current markdown-it renderer and still has source-reproducible lifecycle, dependency, and SVG-sanitization blockers.

## What This Changes

The PR adds Mermaid rendering for Control UI webchat markdown fences with lazy Mermaid loading, dedicated styles, tests, and package/lockfile dependency changes.

## Best Possible Solution

If maintainers want built-in Mermaid support, replace this with a focused current-main implementation in the markdown-it fence renderer and shared render lifecycle, preserving sanitizer, cache, i18n, task-list, link, retry, and Mermaid bindFunctions behavior.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head against current main's markdown-it renderer, shared markdown consumers, manual refresh lifecycle, and Mermaid render contract.

## Solution Assessment

No. The branch demonstrates the intended visible feature, but the maintainable solution is a current-main rewrite that keeps markdown-it hardening and treats Mermaid SVG rendering as an explicit reviewed browser boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:202-203`
  - body: Current main renders user-controlled markdown through markdown-it after the ReDoS hardening change, but this branch sends the shared renderer back through `marked.parse()`. That drops the current parser-hardening, sanitizer, task-list, cache, locale, and renderer contracts, so the Mermaid branch needs to live in the current markdown-it fence renderer instead.
  - confidence: 0.94
- **[P2] Render Mermaid wherever shared markdown emits it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside the chat transcript, but this branch converts every Mermaid fence into an empty render target while only chat updates schedule Mermaid rendering. Non-chat markdown previews can therefore show blank Mermaid panels unless the feature is scoped to chat or all shared consumers get a render pass.
  - confidence: 0.9
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` still returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching the new scheduler. Those diagrams stay blank until an unrelated chat update or tab change runs the scheduler.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block:not([data-mermaid-status])`, then records transient import/render failures as `error`. Streamed partial diagrams or chunk-load failures can become permanently failed for the same DOM node instead of retrying when the definition completes or the import succeeds later.
  - confidence: 0.91
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include `bindFunctions` that must run after the SVG is inserted to attach diagram interactions. This branch narrows the API type to `{ svg }` and discards the hook, so valid Mermaid links, tooltips, or click handlers can silently degrade.
  - confidence: 0.86
