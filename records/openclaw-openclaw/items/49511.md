## Summary

Keep open: current main does not implement Mermaid rendering, so the PR is still the active feature attempt, but it is not merge-ready because it is stale behind the markdown-it security migration and has concrete rendering lifecycle/API defects.

## What This Changes

The PR adds lazy Mermaid rendering for fenced code blocks in Control UI webchat, with styling, SVG sanitization, tests, and Mermaid-related dependency updates.

## Best Possible Solution

If maintainers want built-in Mermaid, port it onto current main's markdown-it fence renderer, keep the existing sanitizer/cache/i18n behavior, schedule rendering wherever Mermaid blocks can be inserted, and honor Mermaid's render result contract.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review findings are source-reproducible from the PR head and current main: the branch uses the old marked renderer, rewrites a shared sanitizer while only chat schedules Mermaid, skips manual refreshes, and drops Mermaid bindFunctions.

## Solution Assessment

No, not as currently implemented. The maintainable path is to rebase onto current main's markdown-it renderer, preserve the ReDoS hardening, and make the scheduling/API behavior match every rendered surface.

- **[P1] Port Mermaid support to the markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main replaced the old marked renderer with markdown-it to fix a Control UI markdown ReDoS freeze, but this branch still implements Mermaid in the obsolete htmlEscapeRenderer.code path.   - confidence: 0.9

  - body: toSanitizedMarkdownHtml is shared by non-chat views such as the markdown sidebar and side-result renderer, but this change turns every Mermaid fence into an empty render target while only chat lifecycle updates call scheduleMermaidRender. Those other views can show a blank Mermaid panel with the source hidden unless some unrelated chat update runs the scheduler.
  - confidence: 0.88
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps chatManualRefreshInFlight true while refreshed history replaces chatMessages, so handleUpdated returns before it reaches this new scheduler call. Mermaid blocks loaded by the refresh button can stay blank until another chat update or tab switch happens.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include bindFunctions that must run after the SVG is inserted so diagram callbacks and tooltips are wired. This code only uses svg, so interactive Mermaid diagrams render as static SVGs with their interactions silently missing.
  - confidence: 0.84
