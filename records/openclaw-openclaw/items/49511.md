## Summary

Keep open. Current main still renders Mermaid fences as ordinary code blocks, so the feature remains valid, but this PR head is not merge-ready: it is stale against the current markdown-it renderer and still has concrete rendering lifecycle, retry, and SVG-sanitization review blockers.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat code fences, adds Mermaid styles/tests/interactions, and updates UI dependency metadata plus the lockfile.

## Best Possible Solution

Land a refreshed, narrow Mermaid implementation against current main's markdown-it renderer after maintainer security review. The implementation should either scope Mermaid rewriting to chat or render all shared markdown consumers, recover from streamed/transient failures, preserve Mermaid's render contract, handle theme/fallback states, and include focused sanitizer, lifecycle, and UI tests.

## Reproduction Assessment

Yes. On current main, a message containing a fenced mermaid block follows the generic fenced-code renderer in `ui/src/ui/markdown.ts`, so the requested feature is absent. The PR regressions are reproducible by code-path inspection of the shared sanitizer consumers, chat lifecycle scheduling, and Mermaid retry selector.

## Solution Assessment

No. The feature direction is reasonable, but this PR head is not the best final solution because it rewrites shared markdown while rendering only from chat lifecycle hooks, does not retry existing failed blocks, omits Mermaid bindFunctions, and requires SVG sanitizer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope Mermaid rewriting to rendered surfaces:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside the chat transcript, including sidebar markdown, cron views, side-result rendering, and file previews, but this branch only schedules Mermaid rendering from the chat lifecycle. Those consumers will get an empty `.mermaid-block__render` with the source hidden behind `<details>`, so Mermaid content outside chat becomes a blank panel instead of a visible code block.
  - confidence: 0.9
- **[P2] Schedule Mermaid after local chat rerenders:** `ui/src/ui/app-lifecycle.ts:124`
  - body: The only render pass is queued here from selected `handleUpdated()` properties. Manual refresh returns before this call while `chatManualRefreshInFlight` is true, and transcript search/local toggles use `requestUpdate()` without those changed properties, so fresh Mermaid blocks can be inserted with no render pass until an unrelated chat update occurs.
  - confidence: 0.86
- **[P2] Let failed Mermaid blocks rerender:** `ui/src/ui/mermaid.ts:150`
  - body: The render pass only selects `.mermaid-block:not([data-mermaid-status])`, but load failures, render failures, and empty definitions mark existing nodes as `data-mermaid-status="error"`. A streamed partial diagram or transient chunk failure can therefore stay broken after the definition/module becomes valid unless the DOM is rebuilt.
  - confidence: 0.88
- **[P3] Call Mermaid bind functions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11's `render()` result can include `bindFunctions` that must run after inserting the SVG for interactions such as clicks/tooltips. This code destructures only `svg` and never calls the binder, so interactive diagrams silently degrade to static SVGs.
  - confidence: 0.8
- **[P3] Show fallback text for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When a Mermaid fence has no body, this branch marks the block as `error` and continues without calling `setMermaidRenderError()`. Users see an empty placeholder with collapsed source instead of an actionable failure message, which regresses from the old visible code fence.
  - confidence: 0.78
