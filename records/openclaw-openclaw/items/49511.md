Full review comments:

- **[P1] Port Mermaid rendering to the current markdown-it fence renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch adds Mermaid handling to the old marked `htmlEscapeRenderer.code` path, but current main's chat renderer is `md.renderer.rules.fence` from markdown-it. Resolving the conflict by reviving the old path would drop current highlighting, i18n, task-list, sanitizer, and ReDoS hardening behavior, so the feature needs to be rebased onto the current fence renderer.
  - confidence: 0.93
- **[P2] Only emit Mermaid placeholders where a render pass will run:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat lifecycle, but this change rewrites every Mermaid fence into an empty `.mermaid-block__render` placeholder while only `handleUpdated()` schedules rendering. Mermaid content in sidebar/tool/preview markdown views can therefore show a blank panel with source hidden behind details until an unrelated chat update happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to render again:** `ui/src/ui/mermaid.ts:149-150`
  - body: `renderMermaidBlocks()` only selects blocks with no `data-mermaid-status`, then marks import, render, and empty-definition failures as `error`. Streamed Mermaid output can be parsed while incomplete, and transient import failures are possible, so the completed diagram is skipped forever until the DOM is rebuilt.
  - confidence: 0.88
- **[P2] Call Mermaid bindFunctions after injecting the SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that callers must run after inserting the SVG for diagram listeners to work. This code destructures only `svg`, inserts it, and marks the block ready, so diagrams that rely on Mermaid click or tooltip bindings render as inert output.
  - confidence: 0.84
