## Summary

Keep open. The Mermaid WebChat feature is still absent from current main, but this PR needs maintainer review before merge because the branch is stale against main's current markdown-it renderer and still has source-reproducible rendering lifecycle and sanitizer concerns.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat code fences, dedicated styling/interactions/tests, UI dependency and lockfile updates, and chat lifecycle scheduling.

## Best Possible Solution

Land a refreshed, narrow Mermaid implementation against current main's markdown-it renderer after maintainer security review, with rendering scheduled for every surface that emits Mermaid placeholders and focused retry, fallback, sanitizer, lifecycle, and UI coverage.

## Reproduction Assessment

Yes. On current main, a fenced `mermaid` block follows the generic fenced-code renderer and there is no Mermaid dependency or scheduler; the PR regressions are source-reproducible from the shared renderer, chat lifecycle scheduler, and retry selector.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best final form because it targets the old renderer, rewrites shared markdown without rendering every consumer, has retry/fallback gaps, and needs explicit SVG sanitizer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope Mermaid rewriting to rendered surfaces:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside the chat transcript, but this patch rewrites every Mermaid fence into an empty `.mermaid-block__render` placeholder while only scheduling rendering from chat updates. Sidebar, cron, side-result, and preview markdown can therefore show a blank panel with source hidden under `<details>` instead of the previous visible code block.
  - confidence: 0.9
- **[P2] Schedule Mermaid after local chat rerenders:** `ui/src/ui/app-lifecycle.ts:124`
  - body: The scheduler is reached only from selected Lit `changed` fields and this function returns while `chatManualRefreshInFlight` is true. Manual refreshes and local search/toggle rerenders can insert fresh Mermaid placeholders without a render pass until an unrelated chat update happens.
  - confidence: 0.86
- **[P2] Let failed Mermaid blocks retry:** `ui/src/ui/mermaid.ts:150`
  - body: This selector skips any block that already has `data-mermaid-status`, but transient load/render failures and partial streamed definitions mark the existing node as `error`. Once the source or chunk load becomes valid, later render passes still skip the same block and leave it broken.
  - confidence: 0.88
- **[P3] Call Mermaid bind functions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11's `render()` result can include `bindFunctions` that callers must run after injecting the SVG. This code destructures only `svg`, so diagrams that rely on Mermaid click or tooltip bindings silently render as static SVG.
  - confidence: 0.8
- **[P3] Show fallback text for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid fence has no body, this branch marks the block as `error` without calling `setMermaidRenderError()`. Users see an empty placeholder with collapsed source instead of an actionable failure message, which regresses from the old visible code fence.
  - confidence: 0.78
