## Summary

Keep open: the WebChat Mermaid feature is not on current main, but this PR is not merge-ready because it is stale against the current markdown-it renderer and has source-reproducible rendering lifecycle issues.

## What This Changes

Adds lazy-loaded Mermaid rendering for Control UI WebChat Mermaid fences, diagram styling/interactions/tests, and Mermaid dependency and lockfile updates.

## Best Possible Solution

If maintainers want built-in Mermaid, land a narrow current-main implementation in the markdown-it renderer/shared rendering lifecycle that preserves sanitizer, cache, i18n, task-list, and code-block behavior with focused security review and tests.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from PR head and current main: stale marked.parse usage, shared markdown consumers without render scheduling, skipped failed blocks, and Mermaid bindFunctions handling.

## Solution Assessment

No, not as currently implemented. The maintainable path is to port the feature onto current main's markdown-it renderer and shared render lifecycle rather than merging this stale marked-based branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:202-206`
  - body: Current main moved this user-controlled markdown path to markdown-it for renderer behavior and ReDoS hardening, but this branch calls `marked.parse()` and adds Mermaid inside the old marked renderer. Merging this file as-is would drop the current renderer, sanitizer/cache, task-list, and i18n behavior, so the Mermaid branch needs to live in the current `md.renderer.rules.fence` implementation.
  - confidence: 0.93
- **[P2] Render Mermaid wherever shared markdown emits it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by chat, side results, and markdown sidebar surfaces, but this branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle calls `scheduleMermaidRender()`. Non-chat views can therefore show a blank diagram panel with the source hidden behind details.
  - confidence: 0.89
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while replacing chat history, and `handleUpdated()` returns before reaching this new scheduler call. Mermaid blocks loaded by a manual refresh can therefore stay blank until another unrelated chat update or tab switch happens.
  - confidence: 0.86
- **[P2] Allow transient Mermaid failures to rerender:** `ui/src/ui/mermaid.ts:149-151`
  - body: The render pass only selects `.mermaid-block:not([data-mermaid-status])`, then marks failed blocks as `error`. A streamed or temporarily invalid Mermaid block that fails once is skipped forever even after the message completes or the lazy import can be retried.
  - confidence: 0.9
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include `bindFunctions` that must run after the SVG is inserted for callbacks, links, and tooltips to work. This code destructures only `svg`, so supported Mermaid interactions silently degrade to a static image.
  - confidence: 0.84
- **[P2] Show an error for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid fence is empty, this branch marks the block as `error` but never writes fallback text into the render target. The transcript then shows a blank placeholder with the source collapsed instead of actionable feedback.
  - confidence: 0.85
