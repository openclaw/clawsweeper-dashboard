- **[P1] Port Mermaid handling to the active renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This adds Mermaid markup to the old `htmlEscapeRenderer.code` path, but current main renders fenced code through `markdown-it` and streaming-safe markdown. The branch needs a real rebase and port before it can prove Mermaid support on the active renderer.
  - confidence: 0.95
- **[P1] Remove the workspace-wide lodash override:** `package.json:1253`
  - body: A Control UI Mermaid feature should not add a root `lodash-es` override that changes dependency resolution for the whole workspace. Root policy treats overrides and lockfiles as security-sensitive, so this needs removal or explicit maintainer approval with regenerated lockfile proof.
  - confidence: 0.9
- **[P2] Scope Mermaid placeholders to rendered surfaces:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml` is shared outside chat, but this PR only schedules Mermaid rendering from the chat lifecycle. Shared callers would get an empty render slot with the source hidden under details instead of a rendered diagram or normal code block.
  - confidence: 0.86
- **[P2] Run Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid 11.13.0 documents that `render()` can return `bindFunctions` that must run after the SVG is inserted. This PR's local API type drops that return field and only assigns `innerHTML`, so tooltips or click bindings silently become static.
  - confidence: 0.84
- **[P2] Let failed Mermaid blocks retry:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer only queries blocks without `data-mermaid-status`, but load and render failures mark blocks as `error`. A transient chunk, network, or first-pass render failure permanently skips the same DOM block even though the module promise can reset.
  - confidence: 0.8
