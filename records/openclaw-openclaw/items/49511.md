## Summary

Keep open: current main still lacks built-in WebChat Mermaid rendering, and this PR is the canonical remaining implementation after the later duplicate PR was closed. The patch is not mergeable as-is because it targets the obsolete marked renderer, risks dropping current markdown-it/ReDoS hardening, only schedules chat lifecycle renders while the markdown renderer is shared, and misses part of Mermaid's render contract.

## What This Changes

The PR adds lazy Mermaid rendering for Control UI/WebChat fenced Mermaid code blocks, with Mermaid dependency and lockfile updates, rendering helpers, SVG sanitization, chat styling, lifecycle scheduling, and tests.

## Best Possible Solution

If maintainers want built-in Mermaid, land a narrow port on current main's markdown-it fence renderer that preserves sanitizer/cache/i18n behavior, schedules rendering for every markdown surface, and honors Mermaid's full render result contract.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review blockers are source-reproducible from the PR head and current main: the branch uses the old marked renderer, rewrites shared markdown output while only chat schedules Mermaid, skips manual refreshes, and ignores Mermaid bindFunctions.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to port the feature onto current main's markdown-it renderer and shared markdown-rendering lifecycle instead of resolving conflicts by taking the stale marked-based implementation.

## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid support to markdown-it:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main replaced the Control UI markdown renderer with markdown-it for ReDoS hardening, but this branch adds Mermaid inside the old marked htmlEscapeRenderer.code path. Merging or conflict-resolving this file as-is would drop current sanitizer/cache/i18n behavior, so the Mermaid case needs to live in the current md.renderer.rules.fence path.
  - confidence: 0.91
- **[P2] Schedule Mermaid wherever shared markdown renders:** `ui/src/ui/markdown.ts:263-264`
  - body: toSanitizedMarkdownHtml is used by non-chat surfaces such as the markdown sidebar and side-result renderer, but this branch converts every Mermaid fence into an empty render target while only chat lifecycle updates call scheduleMermaidRender. Those views can show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.88
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps chatManualRefreshInFlight true while refreshed history replaces chatMessages, so handleUpdated returns before reaching this new scheduler call. Mermaid blocks loaded by the refresh button can remain blank until another chat update or tab switch happens.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render can return bindFunctions that must run after the SVG is inserted so callbacks and tooltips are wired. This code only uses svg, so interactive Mermaid diagrams render as static SVGs with interactions silently missing.
  - confidence: 0.84
- **[P2] Let transient Mermaid failures rerender:** `ui/src/ui/mermaid.ts:149-150`
  - body: The scheduler only selects .mermaid-block elements without data-mermaid-status, but render and load failures mark blocks as error. During streamed replies or transient chunk failures, the same DOM node can stay permanently failed even after the Mermaid source becomes complete or the import could retry.
  - confidence: 0.82
