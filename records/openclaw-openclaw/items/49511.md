## Summary

Keep open: current main still lacks WebChat Mermaid rendering, and the related issue/alternate PR history now points back to this PR as the canonical active implementation. The branch is not ready to merge because it targets the old marked renderer and leaves rendering lifecycle, retry, Mermaid API, and SVG/dependency security questions unresolved.

## What This Changes

The PR adds lazy-loaded Mermaid rendering for Control UI WebChat markdown fences, Mermaid-specific styles and tests, a new Mermaid dependency, and package/lockfile updates.

## Best Possible Solution

Land a focused current-main Mermaid implementation in the markdown-it fence renderer or a clearly scoped adjacent render pass, with sanitizer/dependency review and coverage for chat plus other shared markdown consumers.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from current main and the PR head: main uses markdown-it, while the branch rewrites Mermaid fences in the old marked renderer and only schedules rendering through part of the chat lifecycle.

## Solution Assessment

No. Mermaid rendering is a plausible feature direction, but this branch is not the narrowest maintainable merge shape until it is ported to current main's markdown-it path and the SVG/dependency boundary is accepted by maintainers.

Full review comments:

- **[P1] Port Mermaid support to the markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main renders shared markdown through `md.renderer.rules.fence` after the markdown-it security migration, but this branch adds Mermaid handling to the old marked `htmlEscapeRenderer.code` path. Rebase the feature into the current fence renderer or an explicit adjacent pass so the parser, sanitizer, cache, i18n, and task-list contracts stay intact.
  - confidence: 0.94
- **[P2] Render every emitted Mermaid block:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but the branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle schedules rendering. Non-chat markdown consumers can therefore show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule rendering after manual transcript refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this new scheduler. Those diagrams stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to rerender:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block` elements without `data-mermaid-status`, then marks transient import, parse, streaming, or render failures as `error`. A partial streamed diagram or temporary chunk failure can therefore permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that callers must run after the SVG is inserted for links, tooltips, and click handlers. This branch destructures only `svg`, so interactive diagrams silently degrade.
  - confidence: 0.84
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. Users see an empty panel with the source collapsed instead of the normal code fence or an actionable render failure message.
  - confidence: 0.82
