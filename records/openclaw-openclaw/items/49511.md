## Summary

Keep open: current main still does not implement WebChat Mermaid rendering, and this PR remains the canonical open feature branch, but the current branch is not merge-ready because it is stale against the hardened markdown-it renderer and has source-reproducible rendering/security blockers.

## What This Changes

The PR adds Mermaid rendering for Control UI webchat markdown fences with lazy loading, dedicated styling, sanitizer tests, lifecycle scheduling, and Mermaid-related dependency/lockfile changes.

## Best Possible Solution

Land a focused current-main implementation only if maintainers want built-in Mermaid support, extending the markdown-it renderer or an explicit adjacent render pass while preserving sanitizer, task-list, code-block, cache, locale, retry, theme, and Mermaid `bindFunctions` behavior across every emitted surface.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head against current main's markdown-it renderer, shared markdown consumers, manual-refresh lifecycle, and Mermaid render contract.

## Solution Assessment

No. Mermaid rendering is a reasonable feature direction, but this branch is not the best merge shape because it bypasses the current hardened markdown-it renderer and does not consistently render, retry, or wire the blocks it emits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid support to the current markdown-it renderer:** `ui/src/ui/markdown.ts:202-206`
  - body: Current main renders shared markdown through markdown-it after the ReDoS-hardening migration, but this branch routes the same user-controlled surface back through `marked.parse()`. Implement the Mermaid branch in the current `md.renderer.rules.fence` path or an explicit adjacent render pass so sanitizer, task-list, cache, locale, and parser-hardening contracts stay intact.
  - confidence: 0.94
- **[P2] Render Mermaid blocks wherever they are emitted:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside the chat transcript, but this branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle schedules rendering. Non-chat markdown previews can therefore show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule Mermaid rendering after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` still returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this scheduler. Those diagrams stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to rerender:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block` elements without `data-mermaid-status`, then marks transient import, parse, streaming, or render failures as `error`. A partial streamed diagram or temporary chunk failure can therefore permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted for links, tooltips, and click handlers. This branch destructures only `svg` and discards that hook, so interactive diagrams silently degrade.
  - confidence: 0.86
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error-rendering helper. The user sees an empty panel with the source collapsed instead of the normal code fence or an actionable render failure message.
  - confidence: 0.82
