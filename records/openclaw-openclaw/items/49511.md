## Summary

Keep open: Mermaid rendering is still not on current main, and the earlier issue/alternate PR are closed, but this branch still has source-reproducible correctness and security blockers against the current Control UI markdown path.

## What This Changes

The branch adds lazy-loaded Mermaid rendering for Control UI WebChat markdown fences, Mermaid-specific styles and tests, a new Mermaid dependency, and package/lockfile updates.

## Best Possible Solution

Land a focused current-main Mermaid implementation in the markdown-it fence renderer or an explicit adjacent render pass, with sanitizer/dependency review and coverage for chat plus other markdown consumers.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from current main and the PR head: the branch targets the old marked renderer and leaves shared rendering, retry, refresh, and Mermaid API-contract gaps.

## Solution Assessment

No. Mermaid rendering is a reasonable feature direction, but this branch is not the best merge shape until maintainers accept the dependency/SVG boundary and the implementation is ported to current main's hardened markdown-it path.

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid support to the markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main renders shared markdown through `md.renderer.rules.fence` after the markdown-it migration, but this branch adds Mermaid handling to the older marked `htmlEscapeRenderer.code` path. Rebase the feature into the current fence renderer or an explicit adjacent pass so sanitizer, cache, locale, task-list, and parser-hardening contracts stay intact.
  - confidence: 0.94
- **[P2] Render every emitted Mermaid block:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but the branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle schedules rendering. Non-chat markdown previews can therefore show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule rendering after manual transcript refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this scheduler. Those diagrams stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to rerender:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block` elements without `data-mermaid-status`, then marks transient import, parse, streaming, or render failures as `error`. A partial streamed diagram or temporary chunk failure can therefore permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted for links, tooltips, and click handlers. This branch destructures only `svg` and discards that hook, so interactive diagrams silently degrade.
  - confidence: 0.84
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. The user sees an empty panel with the source collapsed instead of the normal code fence or an actionable render failure message.
  - confidence: 0.82
