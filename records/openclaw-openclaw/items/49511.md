## Summary

Keep open. Current main still lacks Mermaid webchat rendering and the related Mermaid feature attempts are closed unmerged, but this PR is not merge-ready because it is conflicting, stale against the current markdown-it hardening boundary, has source-reproducible lifecycle/rendering gaps, and needs explicit Mermaid/SVG security review.

## What This Changes

The PR adds lazy-loaded Mermaid rendering and styling/tests for Control UI WebChat markdown fences, plus a Mermaid dependency, lockfile updates, and a root lodash-es override.

## Best Possible Solution

Land a focused current-main Mermaid implementation after maintainer/security acceptance, using the markdown-it fence renderer or a scoped post-render pass while preserving shared markdown sanitization and chat lifecycle contracts.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head files, current main's markdown-it renderer, shared markdown consumers, and Mermaid 11.13.0's render contract.

## Solution Assessment

No. Mermaid rendering may be a valid feature direction, but this branch should not merge until it is ported to current main, the lifecycle gaps are fixed, and the Mermaid/SVG dependency boundary is explicitly accepted.

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid handling to the markdown-it renderer:** `ui/src/ui/markdown.ts:2`
  - body: Current main moved shared UI markdown rendering to markdown-it for parser hardening, but this branch still adds Mermaid handling to the older marked renderer path. Rebase the feature into `md.renderer.rules.fence` or a scoped post-render pass so sanitizer, task-list, cache, i18n, and code-block behavior stay intact.
  - confidence: 0.94
- **[P2] Scope Mermaid placeholders to render-capable views:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but the branch rewrites every Mermaid fence into an empty `.mermaid-block__render` placeholder while only chat lifecycle updates call `scheduleMermaidRender()`. Non-chat markdown panes can show a blank panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule Mermaid after transcript refresh and search rerenders:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` still misses DOM replacements from manual refreshes and local chat search rerenders, so those flows can insert fresh `.mermaid-block` nodes without scheduling rendering. Diagrams can stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.88
- **[P2] Allow transient Mermaid failures to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects blocks without `data-mermaid-status` and then marks import, streaming, parse, or render failures as `error`. A partial streamed diagram or temporary chunk failure can permanently skip the same DOM node even after the definition becomes valid.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include `bindFunctions` that must run after the SVG is inserted so links, tooltips, and click handlers work. This branch only uses `svg`, so interactive diagrams silently degrade.
  - confidence: 0.86
- **[P2] Show a fallback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. Users see an empty panel with the source collapsed instead of a normal code fence or actionable render failure.
  - confidence: 0.82
