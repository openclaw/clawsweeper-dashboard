## Summary

Keep this PR open: it is the canonical active implementation for WebChat Mermaid rendering, and current main still lacks the feature. The branch should not merge as-is because it is stale against the current markdown-it renderer, has source-reproducible rendering/lifecycle bugs, and introduces a security-sensitive SVG/dependency path that needs maintainer review.

## What This Changes

The PR adds lazy Mermaid rendering for Control UI WebChat markdown fences, source/preview UI, chat styles, tests, a Mermaid dependency, lockfile changes, and lifecycle scheduling.

## Best Possible Solution

Land a focused current-main implementation in markdown-it's fence renderer or an explicit adjacent render pass, preserving sanitizer, task-list, cache, locale, retry, theme, and Mermaid bindFunctions behavior across every surface that emits Mermaid blocks.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The review blockers are source-reproducible from current main and the PR diff: main has no Mermaid support, while the PR emits placeholders that are not reliably rendered or retried.

## Solution Assessment

No. Mermaid rendering is a reasonable feature direction, but this branch is not the best merge shape because it is stale against the current hardened markdown-it path and leaves shared-surface, retry, and Mermaid API-contract gaps.

- **[P1] Port Mermaid support to the markdown-it renderer:** `ui/src/ui/markdown.ts:260-264`
  - body: Current main renders shared markdown through markdown-it after the ReDoS-hardening migration, but this branch changes the older renderer shape instead. Rebase the Mermaid branch into `md.renderer.rules.fence` or an explicit adjacent render pass so the current sanitizer, task-list, cache, locale, and parser-hardening contracts stay intact.
  - confidence: 0.94
- **[P2] Render every emitted Mermaid block:** `ui/src/ui/markdown.ts:260-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat transcript, but the branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle schedules rendering. Non-chat markdown previews can therefore show a blank Mermaid panel with the source hidden behind details.
  - confidence: 0.9
- **[P2] Schedule rendering after manual transcript refreshes:** `ui/src/ui/app-lifecycle.ts:121`
  - body: `handleUpdated()` still returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching this scheduler. Those diagrams stay blank until an unrelated chat update or tab switch happens.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to rerender:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block` elements without `data-mermaid-status`, then marks transient import, parse, streaming, or render failures as `error`. A partial streamed diagram or temporary chunk failure can therefore permanently skip the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:185-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted for links, tooltips, and click handlers. This branch destructures only `svg` and discards that hook, so interactive diagrams silently degrade.
  - confidence: 0.84
- **[P2] Show feedback for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:177-180`
  - body: When the Mermaid code block is empty, the branch marks the block as `error` but never calls the error renderer. The user sees an empty panel with the source collapsed instead of the normal code fence or an actionable render failure message.
  - confidence: 0.82
