## Summary

This PR should stay open because it proposes an unmerged WebChat Mermaid feature that current main still lacks, and the branch has source-reproducible blockers against the current markdown-it renderer, shared markdown consumers, lifecycle scheduling, retry handling, Mermaid interactions, and SVG/security boundary.

## What This Changes

The PR adds lazy Mermaid rendering for Control UI WebChat markdown fences, Mermaid styles/interactions/tests, lifecycle scheduling, and package/lockfile updates for Mermaid.

## Best Possible Solution

If maintainers want built-in Mermaid support, rework it as a narrow current-main implementation in the markdown-it fence renderer and shared render lifecycle while preserving sanitizer, cache, task-list, i18n, retry, and Mermaid bindFunctions behavior.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head against current main's markdown-it renderer, shared markdown consumers, chat lifecycle, and Mermaid 11.13.0 render contract.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to port the feature onto current main's markdown-it renderer and schedule rendering wherever shared sanitized markdown output can insert Mermaid blocks, with explicit review of the SVG/dependency boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main renders user-controlled markdown through markdown-it, but this branch adds Mermaid handling inside the old marked renderer path. Merging that approach would bypass current renderer behavior such as sanitizer, cache, task-list, i18n, and ReDoS hardening, so the Mermaid branch should live in the current markdown-it fence renderer.
  - confidence: 0.93
- **[P2] Render Mermaid wherever shared markdown emits it:** `ui/src/ui/markdown.ts:263-264`
  - body: toSanitizedMarkdownHtml() is shared outside chat, but the branch rewrites Mermaid fences into empty render targets while only the chat lifecycle schedules Mermaid rendering. Non-chat markdown consumers can therefore show blank Mermaid panels unless the feature is scoped or scheduled for all consumers.
  - confidence: 0.89
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: handleUpdated() returns while chatManualRefreshInFlight is true, so a manual refresh that replaces the transcript can insert Mermaid blocks without calling scheduleMermaidRender(). Those diagrams stay blank until an unrelated chat update or tab switch triggers rendering.
  - confidence: 0.86
- **[P2] Allow transient Mermaid failures to rerender:** `ui/src/ui/mermaid.ts:149-151`
  - body: The renderer only selects Mermaid blocks without data-mermaid-status, then marks import/render failures as error. Since loadMermaidApi() can retry after transient import failures, the same DOM blocks should also be eligible for a later render attempt.
  - confidence: 0.9
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render results can include bindFunctions that must run after the SVG is inserted to attach diagram interactions. This branch only uses svg, so valid links, tooltips, or click handlers silently degrade.
  - confidence: 0.84
- **[P2] Show fallback text for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the fenced Mermaid body is empty, the branch marks the block as error but does not call the fallback helper. Users see an empty panel with collapsed source instead of a clear render-failed message.
  - confidence: 0.86
