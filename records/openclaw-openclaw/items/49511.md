  - body: This branch adds the Mermaid branch to the old `marked` renderer, but current main now renders fenced code through `MarkdownIt` and `md.renderer.rules.fence`. Rebase and port this logic to the active renderer before this can land.

  - body: Adding a root `lodash-es` override for this UI feature changes dependency resolution across the whole workspace. Repository policy treats dependency overrides as a supply-chain surface, so this needs explicit maintainer approval or should be removed in favor of Mermaid's own dependency range.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's `render()` result can include `bindFunctions`, which the upstream type docs say must run after the SVG is inserted. This code drops it, so diagrams with Mermaid interactions will render without their event bindings.
  - confidence: 0.88
- **[P2] Render or preserve Mermaid fences outside chat:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `toSanitizedMarkdownHtml()` is used by non-chat views too, but this PR only schedules Mermaid rendering from the chat lifecycle. Those other surfaces would get an empty render placeholder with source hidden under details instead of a rendered diagram or the original code fence.
  - confidence: 0.82
- **[P2] Allow transient Mermaid failures to rerender:** `ui/src/ui/mermaid.ts:150`
  - body: The selector only revisits blocks without `data-mermaid-status`, but module-load errors, streamed partial diagrams, and parse failures set `error`. Once a streamed Mermaid block fails before the definition is complete, later updates can leave it stuck in the fallback state until the DOM is rebuilt.
  - confidence: 0.78
