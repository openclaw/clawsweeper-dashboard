This PR adds lazy Mermaid rendering for Control UI WebChat markdown fences, source disclosure/dialog styling, Mermaid dependency and lockfile updates, and focused UI tests.

## Best Possible Solution

Port Mermaid support onto the current markdown-it fence renderer, preserve existing code-block behavior, render or gracefully fall back in every markdown surface that emits Mermaid blocks, and land only after maintainers accept the built-in Mermaid/SVG dependency surface.

## Solution Assessment

No. The feature direction is reasonable, but the branch is built on the old renderer and needs a current-main rewrite with complete render scheduling, retry behavior, and maintainer acceptance of the Mermaid/SVG surface.

Full review comments:

- **[P1] Port Mermaid fences to the current markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch still adds the Mermaid special case to the old marked renderer, while current main now renders chat markdown through `md.renderer.rules.fence` with highlight.js, localized copy buttons, and JSON collapse behavior. Resolve the conflict by adding Mermaid support to that current fence renderer; carrying this shape forward would replace existing markdown behavior instead of extending it.
  - confidence: 0.9
- **[P2] Render Mermaid in every markdown surface you rewrite:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by the chat transcript, sidebar, side-result, cron, and other views, but this PR only schedules Mermaid rendering from the chat lifecycle. Any Mermaid fence inserted into those other surfaces can become an empty render panel with the source hidden, so either scope the rewrite to surfaces that schedule rendering or schedule/render per surface when the HTML is inserted.
  - confidence: 0.87
- **[P2] Retry Mermaid blocks after transient failures:** `ui/src/ui/mermaid.ts:150`
  - body: The renderer only selects `.mermaid-block:not([data-mermaid-status])`, then marks module-load and render failures as `error`. A temporary chunk/load/render failure, or a partial block that later becomes valid, is skipped by later scheduler passes unless the whole DOM is rebuilt; clear the status or key it to the source so the same block can recover.
  - confidence: 0.83
- **[P2] Schedule Mermaid rendering after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while `refreshChat()` swaps messages with scrolling disabled, so `handleUpdated()` returns before this new scheduler runs. The later frame only clears the flag and does not change a watched render field, leaving refreshed transcripts with blank Mermaid slots until an unrelated chat update occurs.
  - confidence: 0.82
