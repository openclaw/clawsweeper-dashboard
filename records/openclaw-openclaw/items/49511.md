## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope Mermaid rewriting to rendered surfaces:** `ui/src/ui/markdown.ts:263`
  - body: `toSanitizedMarkdownHtml()` is used outside chat, but this patch rewrites every Mermaid fence into an empty render placeholder while only chat lifecycle updates schedule rendering. Sidebar, side-result, cron, and preview markdown can therefore show blank Mermaid panels instead of the previous visible code block.
  - confidence: 0.9
- **[P2] Schedule Mermaid after local chat rerenders:** `ui/src/ui/app-lifecycle.ts:121`
  - body: This scheduler only runs for selected Lit changed fields and returns during manual refresh. Local search or manual refresh can insert fresh Mermaid placeholders without another render pass, leaving diagrams blank until an unrelated chat update happens.
  - confidence: 0.86
- **[P2] Let failed Mermaid blocks retry:** `ui/src/ui/mermaid.ts:150`
  - body: The selector skips any block with `data-mermaid-status`, so transient load failures, render failures, or partial streamed definitions mark the node as error and prevent later passes from rendering the now-valid source.
  - confidence: 0.88
- **[P3] Call Mermaid bind functions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render contract can return `bindFunctions` that should be called after the SVG is inserted. Keeping only `svg` silently drops diagram click and tooltip bindings for Mermaid diagrams that rely on them.
  - confidence: 0.8
- **[P3] Show fallback text for empty definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid fence has no body, this branch marks the block as error without setting any fallback text. Users see an empty placeholder with collapsed source instead of an actionable render-failure message.
  - confidence: 0.78
