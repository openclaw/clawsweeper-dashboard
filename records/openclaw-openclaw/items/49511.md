## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks Mermaid rendering, but the branch is stale behind the markdown-it security migration and still has concrete rendering-lifecycle defects that need maintainer direction before repair.

## What This Changes

The PR adds lazy Mermaid rendering, styling, tests, and dependency updates for Mermaid fenced code blocks in Control UI webchat.

## Best Possible Solution

Port Mermaid support onto current main's markdown-it fence renderer, preserve the current sanitizer and i18n behavior, then make rendering scheduling match every view that can receive Mermaid blocks.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The blocking findings are source-reproducible from the PR head and current main by comparing the stale marked renderer, shared markdown consumers, chat-only scheduling, and Mermaid render API contract.

## Solution Assessment

No, not as currently implemented. The safer solution is to rebase onto current main's markdown-it renderer and repair the lifecycle/API issues before reconsidering the built-in Mermaid dependency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Port Mermaid rendering to the markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: Current main moved chat markdown rendering from marked to markdown-it for ReDoS hardening, but this branch adds Mermaid support inside the old `htmlEscapeRenderer.code` path. Rebasing this file as-is would discard the current sanitizer/i18n/cache behavior, so implement the Mermaid fence case in `md.renderer.rules.fence` instead.
  - confidence: 0.92
- **[P2] Scope Mermaid rewriting to views that render it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by chat, side-result, markdown sidebar, cron, and dreaming views, but the PR converts every Mermaid fence into an empty render target while only chat lifecycle updates call the scheduler. Non-chat consumers can show a blank panel with the source hidden under details, so either schedule rendering for all consumers or keep the rewrite scoped to chat.
  - confidence: 0.88
- **[P2] Let transient Mermaid failures render again:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer only selects `.mermaid-block:not([data-mermaid-status])`, then marks load, parse, streamed-incomplete, and empty-definition failures as `error`. Once the streamed message completes or a transient import/render failure clears, the same DOM node is never retried, leaving a permanent fallback until the transcript is rebuilt.
  - confidence: 0.86
- **[P2] Call Mermaid bindFunctions after SVG insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render API can return `bindFunctions` that must be invoked after the SVG is inserted. This code only uses `svg`, so diagrams with Mermaid click handlers or tooltips render without their interactions.
  - confidence: 0.84
- **[P2] Schedule Mermaid rendering after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while history replaces `chatMessages`, and `handleUpdated()` returns before this scheduler call. A refreshed transcript containing Mermaid blocks can stay blank until another chat update or tab change happens.
  - confidence: 0.84
- **[P2] Show a fallback for empty Mermaid fences:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the Mermaid definition is empty or trimmed away, this branch sets the block status to `error` but does not call the error-rendering helper. Users see an empty placeholder with collapsed source instead of an actionable fallback message.
  - confidence: 0.8
