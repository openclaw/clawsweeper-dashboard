## Summary

Keep open: current main still does not implement WebChat Mermaid rendering, and this PR remains the canonical open implementation, but the current branch should not merge without a current-main rewrite and focused maintainer/security review.

## What This Changes

The PR adds built-in Mermaid rendering for Control UI WebChat markdown fences, including a Mermaid dependency, markdown renderer changes, chat lifecycle scheduling, styles, tests, and lockfile/package updates.

## Best Possible Solution

If maintainers want built-in Mermaid support, rework it as a narrow current-main implementation in the markdown-it fence renderer and shared render lifecycle while preserving sanitizer, cache, task-list, i18n, retry, and Mermaid bindFunctions behavior.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible from the PR head against current main's markdown-it renderer, shared markdown consumers, chat lifecycle, and Mermaid render contract.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to port the feature onto current main's markdown-it renderer and schedule rendering wherever shared sanitized markdown output can insert Mermaid blocks, with explicit review of the SVG/dependency boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:202-203`
  - body: Current main renders user-controlled markdown through markdown-it after the ReDoS hardening change, but this branch runs the shared renderer through `marked.parse()` and adds Mermaid inside that older renderer path. That would drop current sanitizer, cache, task-list, locale, and parser-hardening behavior, so the Mermaid branch should live in the current markdown-it fence renderer instead.
  - confidence: 0.93
- **[P2] Render Mermaid wherever shared markdown emits it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside chat, but this branch rewrites Mermaid fences into empty render targets while only the chat lifecycle schedules Mermaid rendering. Non-chat markdown consumers can therefore show blank Mermaid panels unless the feature is scoped to chat or scheduled for all consumers that can emit these blocks.
  - confidence: 0.89
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces the transcript can insert Mermaid blocks without calling `scheduleMermaidRender()`. Those diagrams stay blank until an unrelated chat update or tab switch triggers rendering.
  - confidence: 0.86
- **[P2] Allow failed Mermaid blocks to rerender:** `ui/src/ui/mermaid.ts:149-151`
  - body: The renderer only selects `.mermaid-block:not([data-mermaid-status])`, then marks import/render failures as `error`. Since load/import failures and streamed partial diagrams can be transient, the same DOM blocks should be eligible for a later render attempt instead of staying permanently failed.
  - confidence: 0.9
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:185-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted to attach diagram interactions. This branch only keeps `svg`, so valid links, tooltips, or click handlers can silently degrade. ([mermaid.js.org](https://mermaid.js.org/config/setup/mermaid/interfaces/RenderResult.html?utm_source=openai))
  - confidence: 0.84
- **[P2] Show fallback text for empty Mermaid definitions:** `ui/src/ui/mermaid.ts:178-180`
  - body: When the fenced Mermaid body is empty, the branch marks the block as `error` but never calls the fallback helper. Users see an empty panel with collapsed source instead of a clear render-failed message.
  - confidence: 0.86
