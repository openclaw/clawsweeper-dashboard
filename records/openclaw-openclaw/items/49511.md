## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:202-206`
  - body: Current main renders this surface through `markdown-it` after the ReDoS hardening work, but this branch calls `marked.parse()` and adds the Mermaid branch to the older marked renderer. The feature should live in the current markdown-it fence renderer so sanitizer, task-list, cache, locale, and renderer contracts stay intact.
  - confidence: 0.94
- **[P2] Render Mermaid blocks in every shared markdown surface:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is used outside chat, but the branch converts every Mermaid fence into an empty render target while only chat lifecycle updates call `scheduleMermaidRender()`. Non-chat markdown previews can therefore show blank Mermaid panels with the source collapsed.
  - confidence: 0.9
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: `handleUpdated()` still returns while `chatManualRefreshInFlight` is true, so a manual refresh that replaces transcript HTML can insert Mermaid blocks without reaching the new scheduler. Those diagrams stay blank until an unrelated chat update or tab change runs the scheduler.
  - confidence: 0.87
- **[P2] Allow failed Mermaid blocks to retry:** `ui/src/ui/mermaid.ts:149-150`
  - body: The renderer selects only `.mermaid-block:not([data-mermaid-status])`, then records transient import, parse, or streaming failures as `error`. A streamed partial diagram or temporary chunk-load failure can therefore become permanently failed for the same DOM node.
  - confidence: 0.89
- **[P2] Call Mermaid bindFunctions after insertion:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid render results can include `bindFunctions` that must run after the SVG is inserted for links, tooltips, or click handlers. This branch destructures only `svg` and discards that hook, so interactive diagrams silently degrade.
  - confidence: 0.84
