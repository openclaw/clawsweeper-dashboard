## Summary

Keep open: current main still does not implement WebChat Mermaid rendering, and this PR is the canonical remaining tracker after the related issue and alternate PR were closed, but the branch should not merge as-is because it is stale behind the current markdown-it renderer and has source-reproducible rendering lifecycle and Mermaid API contract bugs.

## What This Changes

The PR adds lazy Mermaid rendering for Control UI WebChat markdown fences, dedicated Mermaid UI styling/interactions/tests, and Mermaid-related package and lockfile updates.

## Best Possible Solution

If maintainers want built-in Mermaid, land a narrow current-main implementation in the markdown-it fence renderer that preserves sanitizer, cache, i18n, task-list, and code-block behavior; schedules rendering for every surface that emits Mermaid blocks; and honors Mermaid retries and bindFunctions.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The merge blockers are source-reproducible by comparing this PR head with current main's markdown-it renderer, shared markdown consumers, lifecycle update path, and Mermaid 11.13.0 render contract.

## Solution Assessment

No, not as currently implemented. The maintainable solution is to port the feature onto the current markdown-it renderer and shared rendering lifecycle rather than merging the stale marked-based branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Port Mermaid rendering to markdown-it:** `ui/src/ui/markdown.ts:202-206`
  - body: Current main moved this user-controlled markdown path to markdown-it for renderer behavior and ReDoS hardening, but this branch calls `marked.parse()` and adds Mermaid inside the old marked renderer. Merging this file as-is would drop the current renderer, sanitizer/cache, task-list, and i18n behavior, so the Mermaid branch needs to live in the current `md.renderer.rules.fence` implementation.
  - confidence: 0.93
- **[P2] Render Mermaid wherever shared markdown emits it:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared by chat, side results, and markdown sidebar surfaces, but this branch rewrites every Mermaid fence into an empty render target while only the chat lifecycle calls `scheduleMermaidRender()`. Those non-chat views can show a blank diagram panel with the source hidden behind details.
  - confidence: 0.89
- **[P2] Schedule Mermaid after manual refreshes:** `ui/src/ui/app-lifecycle.ts:124`
  - body: Manual refresh keeps `chatManualRefreshInFlight` true while replacing chat history, and `handleUpdated()` returns before reaching this new scheduler call. Mermaid blocks loaded by a manual refresh can therefore stay blank until another unrelated chat update or tab switch happens.
  - confidence: 0.86
- **[P2] Allow transient Mermaid failures to rerender:** `ui/src/ui/mermaid.ts:149-151`
  - body: The render pass only selects `.mermaid-block:not([data-mermaid-status])`, then marks failed blocks as `error`. A streamed or temporarily invalid Mermaid block that fails once is skipped forever even after the message completes or the lazy import can be retried.
  - confidence: 0.9
- **[P2] Call Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:186-187`
  - body: Mermaid's render result can include `bindFunctions` that must run after the SVG is inserted for callbacks, links, and tooltips to work. This code destructures only `svg`, so supported Mermaid interactions silently degrade to a static image.
  - confidence: 0.84
