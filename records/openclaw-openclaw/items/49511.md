- **[P1] Port Mermaid rendering to the current markdown-it renderer:** `ui/src/ui/markdown.ts:263-264`
  - body: This branch adds the Mermaid branch to the old marked `htmlEscapeRenderer.code` path, but current main's chat renderer is `md.renderer.rules.fence` from markdown-it. Resolving the conflict by reviving this path would drop current markdown-it behavior and the ReDoS/security hardening that came with it, so the feature needs to be rebased onto the current fence renderer instead.
  - confidence: 0.93
- **[P2] Only emit Mermaid placeholders where rendering is scheduled:** `ui/src/ui/markdown.ts:263-264`
  - body: `toSanitizedMarkdownHtml()` is shared outside the chat lifecycle, but this change rewrites every Mermaid fence into an empty `.mermaid-block__render` placeholder while only `handleUpdated()` schedules rendering. Mermaid content in sidebar/tool/preview markdown views can therefore show a blank panel with source hidden behind details until an unrelated chat update happens.
  - confidence: 0.87
- **[P2] Let transient Mermaid failures render again:** `ui/src/ui/mermaid.ts:149-150`
  - body: `renderMermaidBlocks()` only selects blocks with no `data-mermaid-status`, then marks import/render/empty-definition failures as `error`. Streamed Mermaid output can be parsed while incomplete, and transient import failures are also possible, so the completed diagram is skipped forever until the DOM is rebuilt.
  - confidence: 0.88
- **[P2] Run Mermaid bindFunctions after inserting SVG:** `ui/src/ui/mermaid.ts:185-187`
  - body: Mermaid 11's documented `render()` result can include `bindFunctions` that must be called after the SVG is inserted for diagram interactions to work. This implementation types and reads only `svg`, so diagrams relying on Mermaid callbacks, clicks, or tooltips render as static output with interactions silently missing.
  - confidence: 0.78
