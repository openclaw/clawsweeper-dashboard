## Summary

Keep open: the feature is not implemented on current main, and this PR still has merge-blocking correctness issues around stream lifetime limiting, chat-completions coverage, invalid cap parsing, queued aborts, formatting, and missing real behavior proof.

## What This Changes

The branch adds a provider limiter helper and uses it around OpenAI and Azure Responses stream creation, controlled by `LLM_MAX_CONCURRENT`.

## Best Possible Solution

Land a tested stream-aware limiter at the OpenAI/Azure transport boundary that covers Responses and OpenAI-compatible chat completions, holds slots until streams finish or abort, parses the cap safely, and documents the new knob.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows the proposed limiter wraps stream creation rather than stream consumption, and the chat-completions path remains a direct SDK call.

## Solution Assessment

No. The direction is reasonable, but the current implementation should be replaced with a tested async-iterable-aware limiter with abort handling, finite positive cap parsing, completions coverage, docs, and changelog handling during landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hold limiter slots until streams finish:** `src/agents/openai-transport-stream.ts:905-914`
  - body: The limiter wraps only `client.responses.create(...)`, whose promise resolves when the async stream object is created. The slot is released before chunks are consumed, so long-running streams can still overlap even with `LLM_MAX_CONCURRENT=1`.
  - confidence: 0.95
- **[P2] Wrap the chat-completions transport too:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR says `chat.completions.create` is limited, but that call remains direct. Users on the OpenAI-compatible completions path bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back on invalid concurrency values:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. That makes `active >= max` false and silently disables throttling.
  - confidence: 0.93
- **[P2] Remove aborted tasks before launch:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry an `AbortSignal`, so a request aborted while waiting for a slot can still start later. Pass the signal into the limiter and reject or remove queued work before launch.
  - confidence: 0.88
- **[P2] Format the limiter helper:** `src/utils/providerLimiters.ts:4-23`
  - body: The helper body and wrapped Responses calls are visibly malformed, which will fail the repo formatting gate before the PR can merge.
  - confidence: 0.86
