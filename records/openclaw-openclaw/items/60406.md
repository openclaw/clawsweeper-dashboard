## Summary

This PR should stay open but cannot merge as-is: the proposed limiter releases before streamed responses are consumed, misses the chat-completions path it claims to cover, and adds an undocumented env knob that needs maintainer product approval plus real behavior proof.

## What This Changes

The PR adds `src/utils/providerLimiters.ts` and wraps the OpenAI and Azure Responses `client.responses.create` calls with an `LLM_MAX_CONCURRENT`-controlled per-provider limiter.

## Best Possible Solution

Land a tested stream-aware and abort-aware limiter at the OpenAI/Azure transport boundary that covers Responses and OpenAI-compatible chat completions, parses the cap safely, and documents the approved config surface.

## Reproduction Assessment

Yes for the PR defects via source inspection. The proposed wrapper only awaits SDK stream creation while current code consumes the stream later, and the chat-completions path remains a direct SDK call.

## Solution Assessment

No. The direction may be useful, but this patch should be replaced with a stream-lifetime limiter with abort handling, completions coverage, finite-positive cap parsing, tests, and maintainer-approved docs/config wording.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hold limiter slots until streamed output finishes:** `src/agents/openai-transport-stream.ts:905-912`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streamed variant resolves to a `Stream` before chunks are consumed. The slot is released before `processResponsesStream(...)` runs, so long-running streams can still overlap even with a cap of 1.
  - confidence: 0.95
- **[P2] Wrap the chat-completions transport too:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR says `chat.completions.create` is limited, but that SDK call is still direct. Users routed through the OpenAI-compatible completions transport bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the concurrency value is invalid:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` stays `NaN`. That makes `active >= max` false and silently disables throttling instead of using the default cap.
  - confidence: 0.93
- **[P2] Remove aborted requests before launching queued work:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can still start later when a slot opens. Pass the signal through the limiter and reject or remove aborted entries before launch.
  - confidence: 0.88
- **[P2] Fix malformed formatting before the gate runs:** `src/utils/providerLimiters.ts:4-23`
  - body: The helper body and wrapped call sites are flush to the wrong indentation level, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
