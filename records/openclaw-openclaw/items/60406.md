## What This Changes

The PR adds a new provider limiter helper and wraps OpenAI/Azure Responses stream creation behind an `LLM_MAX_CONCURRENT` cap.

## Best Possible Solution

Land only a maintainer-approved, documented, stream-lifetime-aware limiter that covers OpenAI/Azure Responses and Chat Completions, handles aborts and invalid caps, and includes focused tests plus redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch wraps stream creation instead of stream consumption, leaves chat completions direct, parses invalid caps to `NaN`, and queues work without abort state. I did not run tests because this review is read-only.

## Solution Assessment

No. The current approach needs maintainer approval for the new config/default behavior and should be revised to cover the full streaming lifetime, all OpenAI-compatible paths, safe cap parsing, abort handling, current ESM imports, formatting, docs, and proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Hold limiter slots until streaming finishes:** `src/agents/openai-transport-stream.ts:908-913`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streaming overload resolves to a stream object. The limiter slot is released before `processResponsesStream(...)` consumes chunks, so `LLM_MAX_CONCURRENT=1` can still allow overlapping long-running streams.
  - confidence: 0.96
- **[P2] Wrap the chat-completions transport:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR body says `chat.completions.create` is covered, but the direct chat-completions streaming call remains outside the limiter. Users routed through OpenAI-compatible chat completions would bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the cap value is invalid:** `src/utils/providerLimiters.ts:26`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. Then `active >= max` is always false, silently disabling throttling instead of using the default cap.
  - confidence: 0.94
- **[P2] Reject queued work when callers abort:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can remain pending until a slot opens and then attempt to run stale work. The limiter should accept a signal and reject or remove aborted entries before launch.
  - confidence: 0.88
- **[P2] Use the emitted .js helper import:** `src/agents/openai-transport-stream.ts:2`
  - body: This repo uses NodeNext ESM and emitted `.js` specifiers for local runtime imports in this area. The extensionless `../utils/providerLimiters` import will not resolve consistently after build.
  - confidence: 0.88
- **[P2] Fix the limiter formatting:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body and wrapped call sites are indented incorrectly, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
