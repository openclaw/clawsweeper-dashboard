## Summary

Keep this PR open. Current main still lacks a provider-level OpenAI concurrency limiter, and the PR targets a real transport gap, but the patch is not mergeable as-is because it releases limiter slots before stream consumption, misses the advertised chat-completions path, and needs config, abort, format, test, docs, and changelog hardening.

## What This Changes

The PR adds a provider limiter helper and wraps OpenAI/Azure Responses SDK calls with an `LLM_MAX_CONCURRENT` per-provider queue.

## Best Possible Solution

Land a stream-aware provider limiter at the OpenAI/Azure transport boundary that covers OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, holds slots until each async stream finishes or aborts, safely parses configuration, and documents the new knob.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR wraps only stream creation while current transport code consumes async iterables afterward, and the chat-completions path remains a direct SDK call in both current main and the PR context.

## Solution Assessment

No. The direction is reasonable, but the proposed implementation is not the best fix yet; the safer path is a tested async-iterable-aware limiter with abort handling, finite positive cap parsing, completions coverage, docs, and changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hold limiter slots until streams finish:** `src/agents/openai-transport-stream.ts:905-914`
  - body: The limiter wraps only `client.responses.create(...)`, whose promise resolves once the async iterable is created. The slot is released before chunks are consumed, so long-running streams can still overlap even with `LLM_MAX_CONCURRENT=1`; wrap the iterable or stream processing lifecycle instead.
  - confidence: 0.95
- **[P2] Wrap the chat-completions transport too:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR says `chat.completions.create` is limited, but that call remains direct. Users on the OpenAI-compatible completions path bypass the new cap entirely, so this transport needs the same stream-lifetime limiter coverage.
  - confidence: 0.96
- **[P2] Fall back on invalid concurrency values:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` stays `NaN`. Since `active >= max` is then always false, a typo silently disables throttling instead of falling back to the default.
  - confidence: 0.93
- **[P2] Remove aborted tasks before launch:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry an `AbortSignal`, so a request aborted while waiting for a slot can still start later. Pass the caller signal into the limiter and reject or remove queued work before launch when it aborts.
  - confidence: 0.88
- **[P2] Format the limiter helper:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body is flush-left, and the wrapped Responses call bodies in the transport file have the same issue. The repo formatter/check gate will reject the patch before it can merge.
  - confidence: 0.86
