## Summary

Keep this PR open. Current main still has no OpenAI/Azure provider request limiter, so the contribution targets a real gap, but the submitted patch is not mergeable because it misses the chat-completions path and the limiter does not actually hold slots for streamed response lifetimes.

## What This Changes

This PR adds a new provider limiter helper and wraps two OpenAI/Azure Responses SDK calls with an `LLM_MAX_CONCURRENT`-controlled per-provider queue.

## Best Possible Solution

Land a stream-aware provider limiter at the OpenAI/Azure transport boundary that covers OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, holds slots until each async stream finishes or aborts, safely handles invalid configuration, and includes focused tests plus docs/changelog for the chosen knob.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has direct SDK calls on all three affected transport paths, and the PR patch can be read to show the limiter returns after stream creation rather than after stream consumption.

## Solution Assessment

No. The direction is reasonable, but this PR is not the best implementation yet because it misses one advertised transport path and limits only SDK stream creation; a safer fix is a tested async-iterable-aware limiter with abort and config hardening.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Hold limiter slots until streams finish:** `src/agents/openai-transport-stream.ts:905-914`
  - body: The limiter wraps only `client.responses.create(...)`, whose promise resolves once an async iterable is created. The slot is released before chunks are consumed, so long-running streams can still overlap even with `LLM_MAX_CONCURRENT=1`; wrap the returned iterable or the stream processing so the slot is held for the full response lifetime.
  - confidence: 0.95
- **[P2] Wrap the chat-completions transport too:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR says `chat.completions.create` is limited, but that call remains direct. Users on the OpenAI-compatible completions path bypass the new cap entirely, so the limiter needs to cover this stream path as well.
  - confidence: 0.96
- **[P2] Reject invalid concurrency values safely:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` stays `NaN`. With `max` as `NaN`, `active >= max` is always false and throttling is silently disabled; parse a finite positive integer and fall back to the default.
  - confidence: 0.93
- **[P2] Remove aborted tasks from the queue:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry an `AbortSignal`, so a request aborted while waiting for a slot can still start later. Pass the caller signal into the limiter and reject/remove queued work before launch when it aborts.
  - confidence: 0.88
- **[P2] Format the limiter and wrapped call sites:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body and wrapped Responses call bodies are flush-left, so the repo formatter/pre-commit gate will reject the patch before it can merge. Re-run the repo formatter after the behavior fixes.
  - confidence: 0.86
