## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Hold the limiter slot until the stream finishes:** `src/agents/openai-transport-stream.ts:905-910`
  - body: `client.responses.create(...)` resolves once the SDK has produced a `Stream`, but OpenClaw consumes that stream later in `processResponsesStream`. Releasing the limiter in `finally` immediately after `create` means `LLM_MAX_CONCURRENT=1` can still allow many long-running response streams at once.
  - confidence: 0.93
- **[P1] Cover the chat completions transport too:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR description says Chat Completions are limited, but the `client.chat.completions.create(...)` path is still outside the limiter. Any model using `openai-completions` bypasses the new cap entirely.
  - confidence: 0.91
- **[P1] Do not default existing users into a new global cap:** `src/utils/providerLimiters.ts:25`
  - body: `LLM_MAX_CONCURRENT` defaults to `2`, so upgrading would silently serialize existing OpenAI/Azure workloads that previously ran with higher concurrency. Keep the current behavior unless users opt in, or add maintainer-approved docs, tests, and upgrade proof for the default change; also reject invalid values instead of letting `NaN` disable the limiter.
  - confidence: 0.88
- **[P2] Reject queued work when the caller aborts:** `src/utils/providerLimiters.ts:19-22`
  - body: Queued entries do not carry the caller's `AbortSignal`, so an aborted request can sit in the queue and start only after the caller has already given up. Pass the signal into the limiter and reject/remove queued tasks before starting them.
  - confidence: 0.82
