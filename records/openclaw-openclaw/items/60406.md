## Summary

Keep open: current main does not implement an OpenAI concurrency cap, and this PR introduces a new config surface with blocking correctness issues plus missing real behavior proof.

## What This Changes

The PR adds a provider limiter helper and wraps OpenAI/Azure Responses stream creation behind an `LLM_MAX_CONCURRENT` cap.

## Best Possible Solution

Land an approved, documented, stream-lifetime and abort-aware limiter that covers OpenAI/Azure Responses and Chat Completions, with focused tests and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch wraps only stream creation, leaves chat completions direct, parses invalid caps to `NaN`, and does not model queued aborts. I did not run tests because this is a read-only review.

## Solution Assessment

No. The proposed limiter needs maintainer approval and should be changed to hold slots through stream consumption, cover chat completions, handle aborts/config parsing, and document the new config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Hold limiter slots until streaming finishes:** `src/agents/openai-transport-stream.ts:908-913`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streaming overload resolves to a stream object. The slot is released before `processResponsesStream(...)` consumes chunks, so `LLM_MAX_CONCURRENT=1` can still allow overlapping long-running streams.
  - confidence: 0.96
- **[P2] Wrap the chat-completions transport:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR body says `chat.completions.create` is covered, but this direct SDK call remains outside the limiter. Users routed through OpenAI-compatible chat completions would bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the concurrency value is invalid:** `src/utils/providerLimiters.ts:26`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. That makes `active >= max` false and silently disables throttling instead of using the default cap.
  - confidence: 0.94
- **[P2] Reject queued work when its abort signal fires:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can still start later when a slot opens. The limiter should accept the signal and remove or reject aborted entries before launching them.
  - confidence: 0.88
- **[P2] Use the emitted .js extension for the helper import:** `src/agents/openai-transport-stream.ts:2`
  - body: This repo uses Node ESM-style emitted `.js` specifiers for local runtime imports in this area. The extensionless `../utils/providerLimiters` import will not resolve consistently after build.
  - confidence: 0.88
- **[P2] Fix formatting before the gate runs:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body and wrapped call sites are indented incorrectly, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
