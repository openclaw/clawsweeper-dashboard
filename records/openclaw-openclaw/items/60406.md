## Summary

Keep open. The PR is a plausible provider-throttling direction, but the branch is not merge-ready because it limits stream creation instead of stream lifetime, omits the chat-completions path it claims to cover, adds an undocumented global env/default, is conflicting, and has no real OpenAI/Azure proof.

## What This Changes

The PR adds `src/utils/providerLimiters.ts` and wraps OpenAI/Azure Responses stream creation with an `LLM_MAX_CONCURRENT` provider cap.

## Best Possible Solution

Land only an approved, documented, stream-lifetime and abort-aware limiter that covers Responses and Chat Completions, has focused tests, and includes redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the diff wraps only `client.responses.create`, current main consumes streams later, chat completions remains direct, invalid env parsing can yield `NaN`, and queued work has no abort signal. I did not run tests because this review is read-only.

## Solution Assessment

No. The branch should be revised to enforce stream-lifetime limits, cover chat completions, handle abort/config parsing safely, and document the new config contract before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Hold limiter slots until streaming finishes:** `src/agents/openai-transport-stream.ts:905-912`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streaming overload resolves to a stream object. The slot is released before `processResponsesStream(...)` consumes chunks, so `LLM_MAX_CONCURRENT=1` can still allow overlapping long-running streams.
  - confidence: 0.96
- **[P2] Wrap the chat-completions transport:** `src/agents/openai-transport-stream.ts:2081-2084`
  - body: The PR body says `chat.completions.create` is covered, but the separate chat-completions stream path still calls the SDK directly. Users routed through OpenAI-compatible chat completions would bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the cap value is invalid:** `src/utils/providerLimiters.ts:26`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. Then `active >= max` is always false, silently disabling throttling instead of using the default cap.
  - confidence: 0.94
- **[P2] Reject queued work when callers abort:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can remain pending until a slot opens and then attempt stale provider work. The limiter should accept a signal and reject or remove aborted entries before launch.
  - confidence: 0.88
- **[P2] Use the emitted .js helper import:** `src/agents/openai-transport-stream.ts:2`
  - body: This repo uses NodeNext ESM and emitted `.js` specifiers for local runtime imports in this area. The extensionless `../utils/providerLimiters` import will not resolve consistently after build.
  - confidence: 0.88
- **[P2] Fix the limiter formatting:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body and wrapped call sites are indented incorrectly, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
