## Summary

Keep open, but this PR is not mergeable as-is: the limiter releases before streamed output is consumed, misses the chat-completions path, has config parsing/abort/format/import issues, and lacks real behavior proof.

## What This Changes

The PR adds a provider-keyed limiter helper and wraps the OpenAI and Azure Responses `client.responses.create` calls with an `LLM_MAX_CONCURRENT` cap.

## Best Possible Solution

Land a stream-lifetime, abort-aware limiter at the OpenAI/Azure transport boundary only after the config surface is approved, documented, and covered by focused tests plus real behavior proof.

## Reproduction Assessment

Yes for the PR defects via source inspection: the diff wraps only stream creation, leaves chat completions direct, parses invalid caps as `NaN`, ignores queued aborts, and adds an extensionless NodeNext import. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction may be useful, but the patch should be replaced with a stream-aware limiter that covers both Responses and Chat Completions, handles aborts and invalid caps, uses valid ESM imports, and documents the approved config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Hold limiter slots until streamed output finishes:** `src/agents/openai-transport-stream.ts:905-912`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streamed variant resolves to a `Stream` before chunks are consumed. The slot is released before `processResponsesStream(...)`, so long-running streams can still overlap even with a cap of 1.
  - confidence: 0.96
- **[P2] Wrap the chat-completions transport too:** `src/agents/openai-transport-stream.ts:1683-1686`
  - body: The PR says chat completions are limited, but `client.chat.completions.create(...)` remains a direct SDK call. Users routed through the OpenAI-compatible completions transport bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the concurrency value is invalid:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` stays `NaN`. That makes `active >= max` false and silently disables throttling instead of using the default cap.
  - confidence: 0.94
- **[P2] Remove aborted requests before launching queued work:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can still start later when a slot opens. Pass the signal through the limiter and reject or remove aborted entries before launch.
  - confidence: 0.88
- **[P2] Use the emitted .js extension for the local import:** `src/agents/openai-transport-stream.ts:2`
  - body: This repo uses NodeNext ESM, and local runtime imports in this file use emitted `.js` specifiers. `../utils/providerLimiters` will not resolve after build; import `../utils/providerLimiters.js` instead.
  - confidence: 0.88
- **[P2] Fix malformed formatting before the gate runs:** `src/utils/providerLimiters.ts:4-23`
  - body: The helper body and wrapped call sites are flush to the wrong indentation level, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
