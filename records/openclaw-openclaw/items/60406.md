## Summary

Keep open. The PR is a plausible direction for provider throttling, but the patch is incorrect, lacks after-fix real behavior proof, and introduces a new global env/config policy that needs maintainer approval before it can merge.

## What This Changes

The PR adds a new provider-keyed limiter helper and wraps the OpenAI/Azure Responses stream creation sites behind an `LLM_MAX_CONCURRENT` cap.

## Best Possible Solution

Replace the patch with an approved, documented, stream-lifetime and abort-aware limiter at the OpenAI/Azure transport boundary that covers both Responses and Chat Completions with focused tests and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects through source inspection: the diff wraps only stream creation, misses chat completions, parses invalid caps as `NaN`, ignores queued aborts, and uses an extensionless NodeNext import. I did not run tests because this is a read-only review.

## Solution Assessment

No. The limiter direction may be useful, but this implementation releases slots before streams finish, leaves a claimed transport uncovered, lacks abort-safe queue behavior, and adds an undocumented config knob.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Hold limiter slots for the full stream lifetime:** `src/agents/openai-transport-stream.ts:905-912`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streaming overload resolves when the stream object is created. The slot is released before `processResponsesStream(...)` consumes chunks, so long-running streams can still overlap even with `LLM_MAX_CONCURRENT=1`.
  - confidence: 0.96
- **[P2] Wrap the chat-completions transport:** `src/agents/openai-transport-stream.ts:1692-1695`
  - body: The PR body says `chat.completions.create` is covered, but the completions path remains a direct SDK call. Users routed through OpenAI-compatible chat completions bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the concurrency value is invalid:** `src/utils/providerLimiters.ts:25`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. That makes `active >= max` false and disables throttling instead of falling back to the default cap.
  - confidence: 0.94
- **[P2] Reject queued work when its abort signal fires:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can still start later when a slot opens. Pass the signal through the limiter and remove or reject aborted entries before launching them.
  - confidence: 0.88
- **[P2] Use the emitted .js extension for the helper import:** `src/agents/openai-transport-stream.ts:2`
  - body: This repo uses NodeNext ESM and local runtime imports in this area use emitted `.js` specifiers. `../utils/providerLimiters` will not resolve consistently after build; import the emitted `../utils/providerLimiters.js` path.
  - confidence: 0.88
- **[P2] Fix formatting before the gate runs:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper body and wrapped call sites are flush to the wrong indentation level, which will fail the repository formatting gate before the PR can merge.
  - confidence: 0.86
