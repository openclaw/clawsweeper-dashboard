## Summary

Keep open, but this PR is not merge-ready: it is a plausible provider-runtime improvement, yet the current diff changes existing OpenAI/Azure concurrency behavior by default, releases limiter slots before streaming completes, misses the chat-completions path, lacks real behavior proof, and conflicts with current main.

## What This Changes

Adds a new provider limiter helper and wraps OpenAI/Azure Responses stream creation with an `LLM_MAX_CONCURRENT` per-provider cap.

## Best Possible Solution

Land only an approved, documented, stream-lifetime and abort-aware limiter that covers OpenAI/Azure Responses and Chat Completions with upgrade-safe defaults, focused tests, and redacted real provider proof.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the diff wraps stream creation instead of stream consumption, leaves chat completions direct, and parses invalid caps to `NaN`. I did not run tests because this review is read-only.

## Solution Assessment

No. The current approach needs maintainer approval and should be reworked to cover stream lifetime, abort handling, chat completions, config validation, documentation, upgrade behavior, and real provider proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve existing concurrency unless the cap is approved:** `src/utils/providerLimiters.ts:26`
  - body: `LLM_MAX_CONCURRENT` defaults to 2 as soon as this lands, so existing OpenAI/Azure users with more parallel model work would be throttled on upgrade without schema/docs or upgrade proof. Make the cap opt-in or get explicit maintainer approval plus tests/proof for the new default.
  - confidence: 0.91
- **[P1] Hold limiter slots until streaming finishes:** `src/agents/openai-transport-stream.ts:905-910`
  - body: `withProviderLimiter` only awaits `client.responses.create(...)`, whose streaming overload resolves to a stream object. The slot is released before `processResponsesStream(...)` consumes chunks, so `LLM_MAX_CONCURRENT=1` can still allow overlapping long-running streams.
  - confidence: 0.97
- **[P2] Wrap the chat-completions transport:** `src/agents/openai-transport-stream.ts:1170-1172`
  - body: The PR body says `chat.completions.create` is covered, but the OpenAI-compatible chat completions path remains a direct SDK call. Users routed through that transport would bypass the new cap entirely.
  - confidence: 0.96
- **[P2] Fall back when the cap value is invalid:** `src/utils/providerLimiters.ts:26`
  - body: `Number(process.env.LLM_MAX_CONCURRENT || '2')` can produce `NaN`, and `Math.max(1, NaN)` remains `NaN`. Then `active >= max` is always false, silently disabling throttling instead of using the default cap.
  - confidence: 0.95
- **[P2] Reject queued work when callers abort:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry the caller's `AbortSignal`, so a request aborted while waiting can remain pending until a slot opens and then attempt stale work. The limiter should accept a signal and remove or reject aborted entries before launch.
  - confidence: 0.88
- **[P2] Use the emitted .js helper import:** `src/agents/openai-transport-stream.ts:2`
  - body: This package is NodeNext ESM and local runtime imports use emitted `.js` specifiers. The extensionless `../utils/providerLimiters` import will not resolve consistently after build/typecheck.
  - confidence: 0.86
- **[P3] Restore formatter-compliant indentation:** `src/utils/providerLimiters.ts:4-23`
  - body: The new helper and the wrapped Responses call bodies are flush-left in several places, which does not match the repo formatter and should fail the format gate before runtime behavior can be reviewed cleanly.
  - confidence: 0.98
