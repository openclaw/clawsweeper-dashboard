## Summary

Keep open. Current main still does not implement the broad Anthropic native server-tools request: the accepted config/schema lacks first-class Anthropic server-tool and PTC fields, the Anthropic Messages transport still sends ordinary client tool definitions, streamed server result blocks are not preserved, and the only open related implementation is scoped to native web search rather than web_fetch, code_execution/PTC, allowed_callers, and rendering/security policy.

## What This Changes

Support opt-in Anthropic server-side web_search, web_fetch, code_execution, and Programmatic Tool Calling behavior as alternatives to OpenClaw client-side web and tool execution.

## Best Possible Solution

Keep this as the broad Anthropic server-tools tracker, with implementation owned at the Anthropic plugin/transport boundary and split deliberately across config/schema, payload injection, server block preservation/rendering, docs, tests, and security defaults while https://github.com/openclaw/openclaw/issues/49949 and https://github.com/openclaw/openclaw/pull/50692 cover the narrower web_search slice.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a provider feature request. Current-main source inspection gives a high-confidence absence check for the requested config, payload injection, and server content-block handling.

## Solution Assessment

Unclear until maintainer design is settled. The maintainable direction appears to be Anthropic plugin/transport-owned support, while the current related work only covers a narrower native web_search provider slice.
