Keep this issue open as the broad Anthropic native server-tools tracker. Current main and v2026.6.6 still lack an Anthropic-owned configuration/provider contract, native tool request injection, and end-to-end preservation of server-tool results; all three related implementation PRs closed unmerged, while the narrower web-search issue remains open.

## What This Changes

Add opt-in Anthropic server-side web search, web fetch, and code execution/PTC support, including caller metadata and native response-block handling.

## Best Possible Solution

Keep this as the broad canonical tracker and design an Anthropic-plugin implementation with explicit opt-in, independent tool authorization, retention disclosure, native result/citation and continuation preservation, container lifecycle ownership, and credentialed fresh-session plus multi-turn proof.

## Reproduction Assessment

Yes for source verification: current main and v2026.6.6 expose no Anthropic native-tool configuration or provider contract, and the direct Messages transport lacks end-to-end native server-block handling. A credentialed API call was not needed to establish the missing capability.

## Solution Assessment

No, not exactly as proposed: adding raw tool IDs to a generic configuration array and rendering a few block types is insufficient. The maintainable solution is a provider-owned typed contract covering authorization, retention, continuation, result preservation, and container state.

## Risks / Open Questions

- `allowed_callers` is not a hard authorization boundary, so OpenClaw must independently restrict which sensitive tools provider-side code may invoke.

- A new provider/config surface needs explicit defaults and client-tool deduplication rules so existing search and fetch behavior is not silently replaced.
