## Summary

Keep open. Current main still lacks the broad Anthropic native server-tools support requested here: there is no config/schema surface for Anthropic serverTools/PTC/allowed_callers, Anthropic Messages payload construction still emits only client-side tool definitions, streamed server-tool/result blocks are not preserved, and the bundled Anthropic plugin does not declare web search or web fetch provider contracts. The related prior PRs are closed unmerged, while open #50692 only targets the narrower #49949 native web_search provider request.

## What This Changes

The issue requests Anthropic native server-side tools for web_search, web_fetch, and code_execution/PTC, including opt-in config, request-tool injection, response block handling, and Anthropic-only behavior.

## Best Possible Solution

Keep this as the broad Anthropic native server-tools tracker. Review or land the narrower #49949/#50692 web_search work if it passes, then separately design the Anthropic plugin/transport changes for web_fetch, code_execution/PTC, allowed_callers, server content-block preservation/rendering, docs, tests, and security defaults.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path: source inspection shows the requested config keys, payload injection, and streamed block handling are absent.

## Solution Assessment

No, closing or treating the narrow web_search PR as sufficient would not solve the issue. The maintainable path is phased provider-owned implementation with any necessary generic transport seams and explicit security policy for PTC/tool invocation.
