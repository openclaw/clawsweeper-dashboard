## Summary

Keep open. Current main still has the reported Discord voice reliability gap: autoJoin is wired only to READY, RESUMED is dispatched through exact listener matching without a voice listener, and voice cleanup still calls `VoiceConnection.destroy()` on paths that can encounter an already-destroyed connection. The issue is also paired with open PR #40676, so cleanup should wait for that PR to be reviewed, updated, replaced, merged, or closed.

## What This Changes

The issue requests reliable Discord voice autoJoin recovery after health-monitor account restarts, including RESUMED handling and idempotent stale VoiceConnection cleanup.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The end state should be a current-main `extensions/discord` patch that triggers autoJoin on RESUMED or an equivalent account-resume path, makes stale/destroyed voice connection cleanup idempotent with `@discordjs/voice`, and adds focused regression coverage for RESUMED autoJoin plus already-destroyed cleanup.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger a Discord account-level RESUMED path via health-monitor restart, and observe that current main dispatches RESUMED without any voice autoJoin listener; the double-destroy failure is also supported by current direct destroy calls plus the pinned dependency’s throw-on-destroyed contract.

## Solution Assessment

Yes for the owner boundary: this should be solved narrowly inside the Discord plugin rather than core. The paired PR is useful evidence and may be a starting point, but it needs maintainer review, update, or replacement for current `extensions/discord` paths before this issue can close.
