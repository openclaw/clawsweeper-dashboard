Reuse the existing `exec.approval.list` and `exec.approval.resolve` RPCs through the operator-approval runtime client or equivalent authority path, keep output sanitized, fix CI, add focused tests, and require redacted live gateway proof before merge.

## Solution Assessment

Full review comments:

- **[P1] Use the approval runtime client for pending approvals:** `src/cli/exec-approvals-cli.ts:656`
  - body: Both new subcommands call the generic gateway RPC helper, but pending approval visibility is requester-bound unless the client has approval-runtime authority. Existing e2e coverage shows a local-source pending approval resolves through `withOperatorApprovalsGatewayClient`, while a loopback client without that runtime token gets `APPROVAL_NOT_FOUND`; in the local smoke path from the PR body, this CLI can therefore list nothing or fail to resolve the pending approval. Route these commands through the operator approval runtime client or an equivalent helper that preserves that authority.
  - confidence: 0.89

  - body: `check-lint` fails because the new decision normalizer is declared to return `string` but the invalid-decision path calls `exitWithError(...)` without returning it. Return the `never` call or throw after it so the lint lane passes.
  - confidence: 0.98
- **[P2] Widen the pending approval mock return type:** `src/cli/exec-approvals-cli.test.ts:226`
  - body: `check-test-types` fails because the new test overrides `callGatewayFromCli` with a bare pending-approval array that is not assignable to the mock's inferred resolved type. Widen or type the mock return for this case so the test compiles.
  - confidence: 0.97
