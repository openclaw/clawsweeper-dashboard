## Summary

Keep open: the PR is a focused doctor-layer fix for the linked auth-provider upgrade regression, but the branch currently fails production type/lint/guard checks because the new helper reads legacy credential fields through the metadata-only `AuthProfileConfig` type.

## What This Changes

The branch extends doctor auth migration to import legacy default-agent credentials from global `auth.profiles` into the per-agent SQLite auth store, strip migrated config credentials to metadata, and add regression coverage.

## Best Possible Solution

Keep the fix in doctor, make the legacy config-profile import use an explicit raw legacy credential shape that compiles, then rerun the auth migration tests and exact-head type/lint checks before merge.

## Reproduction Assessment

Yes by source inspection: current main detects only per-agent legacy JSON files for this migration while runtime auth reads the per-agent persisted SQLite store. I did not run a full live v2026.6.1-to-v2026.6.6 upgrade.

## Solution Assessment

Yes, the doctor migration layer is the right place to repair shipped legacy config credentials without adding runtime config fallback. The current PR shape is not mergeable until the raw legacy credential typing is fixed.
