## Summary

Keep open. The underlying Discord blockquote chunking bug still appears valid on current main, but this PR is stale against the current Discord chunker and should not merge as-is because it uses a deprecated SDK test barrel and would lose current Unicode/CJK split protections; the broader #63460 overlaps but remains open and dirty rather than a shipped superseder.

## What This Changes

The PR adds Discord blockquote-prefix tracking and tests so mid-line chunk continuations keep quote rendering while line-limit splits avoid an empty quoted paragraph.

## Best Possible Solution

Transplant only the blockquote continuation behavior onto the current Discord chunker, keep the existing code-point and CJK split helpers, use `openclaw/plugin-sdk/test-fixtures` in tests, and then decide whether this narrow path or the broader #63460 should carry the markdown-structure work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a long single-line Discord blockquote split by `maxChars` will create continuation segments, and current `flush()` only re-seeds fenced code state, so the next chunk is not re-prefixed with `> `.

## Solution Assessment

No, not as-is. The blockquote idea is narrow and maintainable, but the PR must be rebased onto current main while preserving current Unicode/CJK split contracts and SDK test-fixture imports, or be superseded deliberately by #63460 after maintainer review.
