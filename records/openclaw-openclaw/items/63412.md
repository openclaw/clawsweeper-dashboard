## What This Changes

The branch adds Discord blockquote-prefix tracking in `chunkDiscordText`, blockquote regression tests, and a changelog entry.

## Best Possible Solution

Land a rebased, narrow Discord chunker fix that carries quote prefixes only for mid-line continuations, reserves message budget, adds max-length regression coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A long single-line Discord blockquote split by `maxChars` creates continuation segments, and current main only reseeds open code fences, so continuation messages lose the `> ` prefix.

## Solution Assessment

No, not yet. The PR is on the right owner surface, but it must reserve room for restored quote prefixes, rebase against current main, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reserve room for restored quote prefixes:** `extensions/discord/src/chunk.ts:187-189`
  - body: When a long quoted line is split by `maxChars`, a continuation segment can already be sized to `charLimit`. After `flush({ preserveBlockquote: true })` seeds `current` with `openBlockquote`, the full segment is appended without reserving that prefix or rechecking length, so the emitted chunk can exceed Discord's 2000-character cap. Reserve the quote prefix in the continuation budget or split again after reseeding.
  - confidence: 0.87
