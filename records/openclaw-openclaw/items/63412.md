## Summary

Keep open: the Discord blockquote chunking bug is source-reproducible and the PR is focused, but the latest head can still emit over-limit chunks after restoring the quote prefix and it lacks after-fix real behavior proof.

## What This Changes

The PR adds Discord chunker blockquote-state carryover, two regression tests, and a changelog entry for quoted text split across chunk boundaries.

## Best Possible Solution

Land a rebased, narrow Discord chunker fix that reserves restored blockquote-prefix space, preserves the existing Unicode/CJK guards, adds a max-size regression test, and includes redacted real Discord or terminal proof.

## Reproduction Assessment

Yes, by source inspection. Current main splits long single-line Discord blockquotes into continuation segments but only reopens fenced code blocks after a flush, so continuation chunks can start without `> `.

## Solution Assessment

No, not yet. Keeping the fix in the Discord chunker is the right boundary, but the current PR must reserve budget for the restored prefix before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reserve space for restored blockquote prefixes:** `extensions/discord/src/chunk.ts:217-219`
  - body: When a long quoted line is split, `splitLongLine` can create a continuation segment already sized to `charLimit`. After `flush({ preserveBlockquote: true })` seeds `current` with the quote prefix, the full segment is appended without subtracting that prefix or rechecking the size, so a chunk can exceed the configured Discord message limit. Reserve the prefix in the continuation budget or re-split after reseeding.
  - confidence: 0.87
