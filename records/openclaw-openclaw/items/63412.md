## Summary

Keep open: this is still a focused Discord bug-fix PR, but the latest head has a limit regression in quote-prefix reseeding, remains dirty against current main, and lacks required real behavior proof.

## What This Changes

The branch adds blockquote-prefix tracking to the Discord chunker, two blockquote regression tests, and a changelog entry.

## Best Possible Solution

Land a rebased, narrow Discord chunker change that preserves blockquote prefixes without exceeding Discord's message limit, with focused tests and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A long single-line Discord blockquote split by `maxChars` creates continuation segments, and current main only reseeds open code fences, so the continuation message loses the `> ` prefix.

## Solution Assessment

No, not yet. The direction is narrow and mostly aligned with current main, but it must reserve space for restored quote prefixes, rebase cleanly, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reserve room for the restored quote prefix:** `extensions/discord/src/chunk.ts:187-189`
  - body: When a long quoted line is split by `maxChars`, `splitLongLine` can return a continuation segment already sized to `charLimit`. After `flush()` seeds `current` with `openBlockquote`, the full segment is appended without another size check, so the emitted chunk can exceed Discord's 2000-character cap. Reserve the quote prefix in the continuation budget or split again after reseeding.
  - confidence: 0.87
