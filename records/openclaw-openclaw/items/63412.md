## Summary

Keep open. Current main still lacks blockquote context propagation in the Discord chunker, so the PR is not obsolete, but the branch is not merge-ready against current main because it introduces a guardrail-failing test import and carries an older splitLongLine implementation that would regress Unicode-safe chunking. It also overlaps the broader open #63460, so a maintainer should decide whether to rebase this narrow fix or continue only with the broader patch.

## What This Changes

The PR adds Discord chunking logic and tests that preserve blockquote prefixes across mid-line continuation chunks while avoiding empty blockquote lines on line-limit splits.

## Best Possible Solution

Transplant the blockquote continuation behavior onto current extensions/discord/src/chunk.ts, preserve the existing Unicode/CJK split helpers, update tests to use openclaw/plugin-sdk/test-fixtures, and then decide whether this narrow rebased PR should land or be closed in favor of the broader #63460 path.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: a long single-line Discord blockquote split by maxChars will split the line into continuation segments, and flush() only re-seeds fenced code state, not the leading > prefix. The PR's added tests cover that character-split case and the earlier line-limit empty-blockquote regression.

## Solution Assessment

No, not as-is. The core blockquote idea is narrow and maintainable, but the branch should be rebased onto current main and fixed for the current SDK test import and Unicode chunking contracts before merge; otherwise #63460 is the safer canonical review target.
