## Label Justifications

- `P2`: This is a normal bounded install/update UX feature, but it is not a current regression or outage.
- `impact:security`: The requested behavior would install and enable executable hooks from downloaded skill archives, affecting the operator trust and hook execution boundary.

## Summary

Keep open: current `main` still installs or updates ClawHub skills as skill files plus ClawHub metadata only, while the requested auto-install and auto-enable hook behavior crosses an executable-hook trust boundary that needs maintainer product/security direction.

## What This Changes

Request to have ClawHub skill install/update detect bundled `hooks/` directories, copy them into managed hooks, enable them, and report the installed hooks.

## Best Possible Solution

Define a security-gated skill hook sidecar contract only if maintainers accept the trust model; otherwise keep executable hooks on the existing plugin/hook-pack route and document the supported packaging path.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection gives high-confidence current behavior: ClawHub skill install/update copies skill content and metadata without hook installation or enablement.

## Solution Assessment

Unclear as proposed: the UX goal is valid, but automatic install-and-enable is not the safest default without a maintainer-approved trust, naming, conflict, update, and uninstall contract. The safer end state is an explicit security-gated sidecar contract or the existing plugin/hook-pack route.

## Risks / Open Questions

- Auto-installing and enabling hooks from skill archives would let a skill install mutate executable hook files and hook config, so maintainers need an explicit trust prompt or policy contract before implementation.
- The proposal's overwrite/re-enable behavior can collide with bundled, plugin-managed, managed, workspace, or manually installed hooks unless ownership, naming, conflict, update, and rollback rules are defined.
- The current explicit hook-pack path already installs and enables hook packages through the plugin installer, so a skill sidecar path must avoid creating a second competing source of truth.
