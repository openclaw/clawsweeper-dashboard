## Summary

Keep open: the reported live runtime-pair failure is plausible and not implemented on current main, but the exact `--runtime-pair` flow is still tied to the open runtime-parity PR rather than a shipped/current-main command.

## What This Changes

The issue asks QA-lab live runtime-pair runs to either stage usable OpenAI/OpenAI Codex auth into the isolated QA agent or fail fast with an actionable auth preflight.

## Best Possible Solution

Resolve this in the runtime-parity work by staging only portable static credentials and adding a live-auth preflight that stops before launching runtime cells when the selected model only has non-portable Codex OAuth.

## Reproduction Assessment

No high-confidence exact current-main reproduction: current main does not expose the reported `--runtime-pair` command. The source path is still reproducible enough to show the auth gap: live QA staging writes env API-key profiles but does not stage Codex OAuth, and current auth contracts intentionally treat OAuth refresh credentials as non-portable by default.

## Solution Assessment

Yes for the fail-fast direction, not for blind OAuth copying. The narrow maintainable fix is a runtime-pair/live-frontier preflight plus portable static profile staging, keeping OAuth refresh-token copying behind explicit provider-owned opt-in if it is ever allowed.
