## Summary

Keep open. Current main still lacks the reported runtime-pair CLI surface, and the shared live QA staging path only provisions env API-key profiles into isolated QA agents; the related open runtime-parity PR and live-proof tracker still treat this as active credential-provisioning work.

## What This Changes

Request that live QA runtime-pair runs either stage usable OpenAI/OpenAI Codex auth into the isolated QA agent or fail fast with an actionable auth preflight instead of launching cells that later fail or time out.

## Best Possible Solution

Fold a narrow credential preflight or safe staging contract into the active runtime-parity work: require a portable env API key for paired live runs or fail before launching cells with a clear remediation message.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks the reported `--runtime-pair` command. Source inspection still shows the shared isolated live-auth path only stages env API-key profiles, which matches the linked live/OAuth failure mode.

## Solution Assessment

Yes, the narrow maintainable direction is a fail-fast auth preflight or explicitly portable credential-staging contract in the runtime-parity QA path. Blindly copying OAuth profile material into temporary QA stores would be riskier than the documented auth portability model.
