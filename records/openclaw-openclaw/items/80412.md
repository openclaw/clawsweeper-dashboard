## Summary

Keep open. Current main still does not expose the reported runtime-pair CLI path, and the shared live QA auth staging code only stages portable env API-key profiles, while the same-author open runtime-parity PR and proof cluster still own this auth/preflight gap.

## What This Changes

Make live runtime-pair QA either provision usable OpenAI/OpenAI Codex auth for isolated QA agents or fail fast before launching paired runtime cells when auth is unavailable.

## Best Possible Solution

Fold a safe credential preflight or portable env API-key staging contract into the active runtime-parity work so live runtime-pair lanes fail before launch when usable auth is unavailable.

## Reproduction Assessment

No high-confidence exact current-main reproduction exists because current main and `v2026.5.7` do not expose `--runtime-pair`. Source inspection still confirms the shared live QA staging path only writes env API-key profiles into isolated agent stores, which matches the linked OAuth-only failure mode.

## Solution Assessment

Yes, with the narrower fail-fast or portable API-key variant. Copying non-portable OAuth store material into temp QA agents should not be the default without explicit owner and security review.
