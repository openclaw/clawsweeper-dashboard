## Summary

Keep open. The report describes a real QA-lab live auth/proof blocker: current main does not have the reported runtime-pair CLI surface, and the live auth staging code still only stages env API-key/setup-token credentials into the isolated QA state, while the same-author runtime-pair PR and live proof tracker remain open.

## What This Changes

The issue asks QA-lab live runtime-pair runs to stage usable OpenAI Codex auth into isolated QA agents or fail fast before launching paired cells.

## Best Possible Solution

Keep this open until the runtime-pair PR or its replacement either adds a safe live-auth preflight or an owner-approved credential staging path, with regression coverage and live proof showing the QA run no longer burns full cell time on missing auth.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does show the isolated QA gateway only stages env API-key/setup-token credentials, not host OpenAI Codex OAuth, and current main lacks the exact `--runtime-pair` command used in the report.

## Solution Assessment

Unclear. The safest maintainable fix is likely a preflight that refuses live OpenAI/OpenAI Codex runtime-pair runs without a portable credential source, unless maintainers explicitly choose to stage OAuth into the isolated QA agent.
