## Summary

Keep open: this is an active QA-lab/runtime-parity blocker, the related runtime-pair PR and live-proof tracker are still open, and current main does not yet contain the reported `--runtime-pair` surface. Source inspection also supports the reported auth gap: live QA staging only creates env API-key profiles in the isolated state, while the issue is about host OpenAI Codex OAuth not being available to the isolated `qa` agent.

## What This Changes

Request that live QA runtime-pair runs using OpenAI Codex either stage usable auth into the isolated QA agent or fail fast before launching paired cells when auth/plugin setup is unavailable.

## Best Possible Solution

Handle this in the runtime-parity PR or a narrow follow-up by adding a live auth/plugin preflight, and only stage OpenAI Codex auth when it follows the existing OAuth portability policy.

## Reproduction Assessment

No high-confidence current-main reproduction: current main does not expose the reported `--runtime-pair` flag. Source inspection and the reporter's artifacts do show the relevant isolated live QA auth path only stages env API-key profiles for the `qa` agent, so the OAuth-only failure is credible.

## Solution Assessment

The direction is right, but blind OAuth copying is not the best fix. The safer maintainable path is a fail-fast preflight or provider-owned safe staging in the runtime-parity harness before the live token-efficiency lane treats the run as usable.
