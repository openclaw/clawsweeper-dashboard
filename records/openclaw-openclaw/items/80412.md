## Summary

Keep open. The reported runtime-pair command is supplied by the open QA-lab harness PR, but the underlying live QA auth-staging path on current main already shows the blocker: isolated QA state stages live env API-key profiles and an Anthropic setup token, not a usable OpenAI Codex OAuth profile or preflight for missing Codex auth.

## What This Changes

The issue asks QA-lab live runtime-pair runs to stage usable OpenAI/OpenAI Codex auth into isolated QA agents or fail fast before launching the paired cells.

## Best Possible Solution

Add a QA live auth preflight/staging path that follows existing auth portability rules: stage portable API-key/token profiles and explicitly portable Codex OAuth profiles into the temp main/qa stores, otherwise fail before launching runtime cells with an actionable auth message.

## Reproduction Assessment

No high-confidence current-main runtime-pair reproduction exists because `--runtime-pair` is only in the open related PR. The auth failure is source-reproducible on current main's live QA staging path and on the PR's runtime-pair path because selected live providers get env API-key staging only, not Codex OAuth staging or a preflight.

## Solution Assessment

Yes, the requested direction is valid, but the safest implementation is the fail-fast or portable-profile path rather than blindly copying OAuth refresh tokens. The fix should land in the QA-lab gateway/auth staging surface and be coordinated with the open runtime-parity PR.
