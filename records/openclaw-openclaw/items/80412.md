## Summary

Keep open. Current main still does not expose the reported runtime-pair QA command, while the shared live QA auth staging code only stages portable env API-key profiles and the related open runtime-parity PR plus live proof tracker still own the remaining auth/proof work.

## What This Changes

The issue asks QA-lab live runtime-pair runs to stage usable OpenAI/OpenAI Codex auth into isolated QA agents or fail fast before launching paired runtime cells.

## Best Possible Solution

Keep this tied to the active runtime-parity work: live runtime-pair lanes should use portable env/leased QA credentials or fail fast with an actionable preflight, without implicitly copying non-portable OAuth state into preserved QA artifacts.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks the reported `--runtime-pair` command. Source inspection does show the shared live auth staging helper skips providers without an env API key, which matches the local OAuth-only missing-auth failure mode.

## Solution Assessment

Yes as a direction: a narrow preflight or portable credential-staging contract is safer than silently launching cells with an empty isolated QA auth store. Copying Codex OAuth material by default would need explicit maintainer security/product approval.
