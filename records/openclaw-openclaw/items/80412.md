## Summary

Keep open. The exact live runtime-pair surface is still in the open related PR, current main does not yet implement or fix this auth/preflight path, and current auth docs make copying OpenAI Codex OAuth into isolated agents a policy-sensitive decision.

## What This Changes

The issue requests that live QA runtime-pair runs either stage usable OpenAI Codex auth into isolated QA agents or fail fast before launching runtime cells when auth/plugin setup is unavailable.

## Best Possible Solution

Fix the runtime-pair PR so live frontier preflights selected provider auth for each isolated cell, stages only portable env/static credentials, and fails fast with an actionable message when OpenAI Codex OAuth is only present in the host/main store.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main lacks the reported --runtime-pair option. Source and discussion still make the auth gap clear for the linked PR/live lane because isolated QA staging covers env API keys but not host OpenAI Codex OAuth profiles.

## Solution Assessment

The requested fail-fast direction is the best conservative fix. Copying OAuth profiles by default is not the best default unless a provider-owned policy explicitly marks that credential shape portable.
