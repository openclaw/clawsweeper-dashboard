## Summary

This should stay open because the reported live runtime-pair behavior is not implemented on current main and is still tied to open QA-lab runtime-parity PR work. Current main stages live env API-key profiles for isolated QA agents and preserves CODEX_HOME, but it does not provide a runtime-pair CLI path or a Codex OAuth staging/preflight path for that new lane.

## What This Changes

The issue asks the QA-lab live runtime-pair suite to either provide usable OpenAI Codex auth to isolated QA agents or fail fast with an actionable preflight error.

## Best Possible Solution

Land the runtime-pair harness with an explicit live-frontier Codex auth preflight that fails before cell launch unless a provider-owned, safe credential staging path is available.

## Reproduction Assessment

No high-confidence current-main reproduction exists because --runtime-pair is not on main. The underlying source path is clear: isolated QA live-frontier staging currently handles env API keys, while Codex OAuth profiles are not copied into the temporary QA agent store.

## Solution Assessment

Unclear until maintainers choose the auth policy for the new runtime-pair lane. The safer maintainable direction is fail-fast preflight first, with any OAuth staging kept provider-owned and explicit.
