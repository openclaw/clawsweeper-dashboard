## Summary

Keep open: current main and v2026.6.8 still only pass through Bedrock `requestMetadata` when options already contain it, and they do not derive per-agent metadata. The target-repo PR that attempted this was closed unmerged, while the remaining fork PR is conflicting and not a safe canonical closure path.

## What This Changes

The issue requests automatic per-agent Amazon Bedrock `requestMetadata` injection, or an equivalent configuration surface, so multi-agent deployments can attribute Bedrock spend per agent.

## Best Possible Solution

Keep this issue open until maintainers choose a Bedrock-provider-owned metadata policy, then land a narrow implementation with metadata minimization, merge semantics, focused tests, and redacted live Bedrock or CloudTrail proof.

## Reproduction Assessment

Yes, source-level. Current main has agent context at the embedded-runner wrapper and a Bedrock provider that forwards `requestMetadata`, but no current-main or v2026.6.8 code injects agent-derived metadata; I did not run live AWS traffic.

## Solution Assessment

Unclear until maintainers choose the metadata policy. Automatic injection is plausible, but the best fix should live in the Bedrock/provider-runtime boundary, minimize exposed values, and merge with any explicit request metadata rather than overwriting it.
