## Summary

Keep open: current main and v2026.6.8 can forward Bedrock requestMetadata when it is already supplied, but neither derives per-agent metadata nor exposes a supported config path for it. The same-repo implementation PR was closed unmerged, and the remaining fork PR is conflicting and not a safe canonical closure path.

## What This Changes

The issue requests automatic per-agent Bedrock requestMetadata so shared-role multi-agent deployments can attribute AWS Bedrock spend by agent.

## Best Possible Solution

Keep this issue open until maintainers choose the Bedrock metadata policy, then land a narrow provider/runtime implementation with metadata minimization, merge semantics, focused tests, and redacted live Bedrock or CloudTrail proof.

## Reproduction Assessment

Yes, source-level. Current main has agent context at the embedded-runner boundary and a Bedrock provider that forwards requestMetadata, but no current-main or v2026.6.8 code derives per-agent metadata or exposes a supported config path; I did not run live AWS traffic.

## Solution Assessment

Unclear until maintainers choose the metadata policy. Automatic injection is plausible, but the best fix should live in the Bedrock/provider-runtime boundary, minimize exposed values, and merge with any explicit request metadata rather than overwriting it.
