## Summary

Keep open: current main and v2026.6.10 can forward Bedrock `requestMetadata` only when it is already supplied, but OpenClaw still does not automatically derive or configure per-agent metadata for Bedrock cost attribution. The remaining work needs maintainer product/security review before a fix PR because agent IDs or names would leave OpenClaw and enter AWS request metadata/logging surfaces.

## What This Changes

Request automatic or configurable per-agent Bedrock `requestMetadata` so multi-agent deployments can attribute Bedrock costs per agent.

## Best Possible Solution

Keep this issue open until maintainers choose the Bedrock metadata policy, then land a narrow Bedrock provider/runtime implementation with minimized metadata, explicit merge semantics, focused tests, docs, and redacted live Bedrock or invocation-log proof.

## Reproduction Assessment

Yes, source-level: current main passes active `agentId` into extra-param resolution and Bedrock can forward `requestMetadata`, but no code derives per-agent metadata or exposes a supported Bedrock metadata config path. I did not run live AWS traffic.

## Solution Assessment

Unclear until maintainers choose the metadata policy. Automatic injection is plausible, but the best fix should live in the Bedrock provider/runtime boundary, minimize exposed values, and merge with explicit metadata rather than overwriting it.
