## Summary

Keep open: current main and v2026.6.10 can forward Bedrock requestMetadata only when supplied, but they do not derive per-agent Bedrock metadata or expose a supported metadata config path. The remaining work needs maintainer product/security judgment because AWS records requestMetadata in invocation logs, not directly as Cost Explorer/CUR cost allocation tags, and agent identifiers or names could become externally logged metadata.

## What This Changes

Request automatic or configurable per-agent Bedrock requestMetadata injection so multi-agent deployments can attribute Bedrock usage by agent.

## Best Possible Solution

Keep this issue open until maintainers choose the Bedrock metadata policy, then land a narrow provider/runtime change with minimized metadata, deterministic merge semantics, focused tests, and redacted live Bedrock invocation-log proof.

## Reproduction Assessment

Yes, source-level. Current main has active agent context at the embedded-runner boundary and a Bedrock provider that forwards requestMetadata, but no current-main or v2026.6.10 code derives per-agent metadata or exposes a supported config path; I did not run live AWS traffic.

## Solution Assessment

Unclear until maintainers choose the metadata policy. Automatic injection is plausible, but the best fix should live in the Bedrock/provider-runtime boundary, avoid raw sensitive values, merge with explicit metadata, and describe that attribution comes from invocation logs rather than native Cost Explorer tags.
