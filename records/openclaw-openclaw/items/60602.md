Keep open: current main and v2026.6.10 can forward Bedrock requestMetadata only when it is already supplied, but they do not derive per-agent Bedrock metadata or expose a supported requestMetadata config path. ## What This Changes

Request automatic per-agent Bedrock requestMetadata injection, or a supported config path, so multi-agent Bedrock costs can be attributed per agent.

## Best Possible Solution

Keep this issue open until maintainers choose the Bedrock metadata policy, then land a narrow provider/runtime implementation with minimized metadata, deterministic merge semantics, focused tests, and redacted live Bedrock or CloudTrail proof.

## Reproduction Assessment

Yes, source-level. Current main has active agent context at the embedded-runner boundary and a Bedrock provider that forwards requestMetadata, but no current-main or v2026.6.10 code derives per-agent metadata or exposes a supported config path; I did not run live AWS traffic.

## Solution Assessment

Unclear until maintainers choose the metadata policy. Automatic injection is plausible, but the best fix should live in the Bedrock/provider-runtime boundary, minimize exposed values, and merge with explicit request metadata rather than overwriting it.
