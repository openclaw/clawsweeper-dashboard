## Summary

Keep this issue open. Current main still does not inject or expose Bedrock `requestMetadata`, and #60698 remains open and unmerged with closing syntax for this issue, so the maintainer path is to review, land, or replace that implementation candidate.

## What This Changes

Request automatic per-agent Amazon Bedrock `requestMetadata` injection, or a config surface for it, so Bedrock costs can be attributed by agent in multi-agent deployments.

## Best Possible Solution

Finish a Bedrock-owned implementation through #60698 or a successor: thread verified session agent identity into the provider request path, apply Bedrock metadata validation, avoid sensitive values, preserve non-Bedrock behavior, coordinate configurable metadata with #57360, and add focused tests around the final stream options or payload.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no `requestMetadata` matches, and the active stream option builder omits it. A live AWS run would only confirm downstream CloudTrail display, not the OpenClaw gap.

## Solution Assessment

Yes on direction, with implementation details still to review. The narrow maintainable shape is provider-owned Bedrock metadata injection using an explicit agent-identity seam, not a broad unvalidated core pass-through.
