## Summary

Keep open: current main and the latest release still expose only three static OpenRouter image-generation shortcuts, while OpenRouter's live image catalog currently includes many more image-output models, including the reporter's example. Source inspection also shows explicit `openrouter/<model>` overrides route to the OpenRouter provider already, so the remaining work is a maintainer catalog/support-policy decision rather than a narrow broken-runtime bug.

## What This Changes

The issue requests broader OpenRouter image-generation model support, either by curating more built-in shortcuts or dynamically discovering OpenRouter image-output models.

## Best Possible Solution

Keep this provider-local and choose a maintained OpenRouter image catalog strategy: either curated shortcuts with live-contract proof, or an auth-gated dynamic `image_generation` catalog from OpenRouter's image-output API with conservative per-model capability mapping.

## Reproduction Assessment

Yes for the remaining source-level catalog/listing gap: current main exposes three static OpenRouter image models, while the live OpenRouter image catalog reports 34 image-output models including `microsoft/mai-image-2.5`. The exact paid generation call was not run; source shows explicit provider/model overrides already route to OpenRouter.

## Solution Assessment

Unclear until maintainers choose the policy. Adding one or two curated IDs is mechanically small, but the more maintainable solution may be a provider-local dynamic catalog with conservative capability mapping and docs/tests.
