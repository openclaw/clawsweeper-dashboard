## Summary

Keep open: current main and the latest release still expose only three static OpenRouter image model shortcuts, while OpenRouter's live image-output catalog includes the requested `microsoft/mai-image-2.5` model. The runtime can pass explicit `openrouter/<model>` refs to the provider, so the remaining work is a provider-owned catalog/listing and support-policy decision rather than an already-fixed bug.

## What This Changes

The issue requests broader OpenRouter image-generation model support, either through curated model IDs or dynamic detection from OpenRouter's model catalog.

## Best Possible Solution

Keep the work provider-local and choose either a curated OpenRouter image model catalog with docs/tests or a maintained dynamic catalog based on OpenRouter image output modalities, without adding OpenRouter-specific policy to core media generation.

## Reproduction Assessment

Yes for the catalog/listing gap: current source and the latest release expose only a static three-model OpenRouter image list, while OpenRouter's live image-output catalog includes the requested model. I did not run a paid live image-generation call with `microsoft/mai-image-2.5`.

## Solution Assessment

Unclear as an implementation path: adding one curated ID is mechanically small, but choosing curated shortcuts versus dynamic discovery requires maintainer product judgment around catalog freshness, compatibility, and docs.
