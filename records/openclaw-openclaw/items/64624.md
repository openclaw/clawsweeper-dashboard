## Summary

Keep open: current main and v2026.6.10 still enqueue WhatsApp connected/disconnected lifecycle notices into prompt-visible session system events, and the remaining fix is a product/config policy decision rather than an already-implemented behavior.

## What This Changes

Request to suppress, throttle, or surface-filter transient channel connection-status system events so short WhatsApp reconnects do not clutter unrelated agent sessions.

## Best Possible Solution

Define a WhatsApp-owned lifecycle notification policy that coalesces short reconnects, keeps terminal outage/action notices visible, and only promotes a shared SDK helper if multiple channel plugins need the same contract.

## Reproduction Assessment

Yes, at source level: current main still enqueues WhatsApp connected/disconnected notices into the resolved session and drains queued events into System prompt lines. I did not run a live WhatsApp reconnect, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as proposed: thresholding, verbosity config, and surface-aware filtering are all plausible, but the best fix needs a maintainer decision on the channel lifecycle notification contract. The closed shared prompt-filter PR is not the best layer because it can hide important disconnect context.
