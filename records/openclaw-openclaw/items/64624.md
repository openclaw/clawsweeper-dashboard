## Summary

Keep open: current main and v2026.6.10 still enqueue WhatsApp connected/disconnected lifecycle notices into the resolved session and drain retained system events as prompt-visible System lines, with no threshold, verbosity, coalescing, or surface-aware policy. The earlier PR that tried a shared prompt filter was closed as too broad, so the remaining work needs maintainer policy direction rather than cleanup closure.

## What This Changes

The issue requests suppression, throttling, or surface-aware filtering for transient channel connection status system events that clutter active sessions.

## Best Possible Solution

Adopt a WhatsApp/channel-owned lifecycle notification policy that coalesces short reconnects and keeps durable operator-action failures visible, with any generic SDK or config surface added only after maintainer approval.

## Reproduction Assessment

Yes, at source level: current main enqueues WhatsApp lifecycle notices into the resolved session key and the reply path drains queued events into prompt-visible System lines. I did not run a live WhatsApp reconnect scenario in this read-only review.

## Solution Assessment

Unclear as proposed: thresholding, verbosity, and surface-aware filtering are plausible, but the best fix needs a maintainer decision on defaults and which lifecycle states must remain visible. The earlier shared prompt-filter PR was not the best shape because it would hide all matching disconnect context.
