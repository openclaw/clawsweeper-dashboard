## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the session system-event path with no threshold, verbosity option, or reconnect coalescing; the matching open PR is not merged and has a prior blocking review for suppressing too broadly.

## What This Changes

Request to suppress or throttle transient channel connection status system events, especially WhatsApp reconnect banners, through a threshold, verbosity, or surface-aware policy.

## Best Possible Solution

Define a WhatsApp-owned lifecycle notification policy that suppresses routine short flaps while preserving terminal recovery/action notices, then land a narrowed version of https://github.com/openclaw/openclaw/pull/49579 or a replacement with tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the WhatsApp monitor immediately queues connected/disconnected lifecycle events and the next prompt path formats retained system events as prompt-visible `System:` lines.

## Solution Assessment

No for the currently open PR as written. A plain prefix filter is too broad; the maintainable fix should classify routine versus critical disconnects at the WhatsApp source or with typed lifecycle metadata/config.
