## Summary

Keep open. Current main and v2026.6.8 still enqueue WhatsApp connected/disconnected lifecycle notices and drain retained queue entries into prompt-visible `System:` lines; the earlier broad prefix-filter PRs were closed unmerged and superseded by this issue, so a maintainer policy decision is still needed.

## What This Changes

The issue asks for suppressing, throttling, or surface-filtering transient channel lifecycle system events, especially WhatsApp reconnect banners, so brief disconnects do not clutter active sessions.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy with default-preserving behavior, reconnect coalescing for short outages, and focused docs/config coverage; promote a generic SDK/helper only after multiple channel plugins need the same contract.

## Reproduction Assessment

Yes, by source inspection. Current main still enqueues WhatsApp connected/disconnected lifecycle notices and drains retained queue entries into prompt-visible `System:` lines, with no threshold, verbosity, or surface-aware filtering.

## Solution Assessment

Unclear until maintainers choose the product policy. The previous shared prefix-filter PR is not the best fix because it can hide terminal disconnect context; a channel-owned lifecycle policy with explicit defaults is safer.
