## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the resolved session and drains retained generic events into prompt-visible System lines; the requested threshold, verbosity, reconnect coalescing, or surface-aware policy is still absent, and the related banner-filter PRs were closed rather than merged.

## What This Changes

Request suppression or throttling of transient channel connection status system events, especially short WhatsApp disconnect/reconnect notices that clutter unrelated sessions.

## Best Possible Solution

Implement a WhatsApp-owned lifecycle notification policy with backward-compatible channel/account config and reconnect coalescing, then promote a generic SDK/helper contract only if multiple channel plugins need the same behavior.

## Reproduction Assessment

Yes by source inspection. Current main enqueues WhatsApp connected/disconnected notices in the monitor path and the shared prompt drain retains them as System lines unless they match existing heartbeat/periodic/node filters.

## Solution Assessment

Yes for the source-owned policy direction, but not for the older shared prefix-filter PRs. A WhatsApp-owned lifecycle policy can suppress transient reconnect churn while preserving terminal outage context.
