## Summary

Keep open. Current main and v2026.6.10 still enqueue WhatsApp connected/disconnected lifecycle notices into prompt-visible system events, and the requested threshold, verbosity, or surface-aware filtering remains a maintainer product/config decision.

## What This Changes

The issue requests suppression or throttling of transient channel connection status system events so short reconnects or irrelevant channel status do not clutter active session/model context.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy with default-preserving behavior, short-outage coalescing, and docs/config coverage; generalize only if multiple channel plugins need the same contract.

## Reproduction Assessment

Yes, by source inspection. Current main still queues WhatsApp connected/disconnected lifecycle notices and drains queued system events into prompt-visible System lines, with no threshold, verbosity setting, reconnect coalescing, or surface-aware filter.

## Solution Assessment

Unclear until maintainers choose the product policy. A shared prefix filter was already closed as too broad; a channel-owned lifecycle policy with explicit defaults is the safer path.
