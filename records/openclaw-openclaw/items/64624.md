Keep open. Current main and the latest release still enqueue WhatsApp lifecycle connected/disconnected notices into prompt-visible system events, while the requested threshold, verbosity, or surface-aware policy remains a product/config decision rather than an implemented behavior.

## What This Changes

The issue requests suppressing or throttling transient channel connection status system events so brief reconnects do not clutter active sessions or agent context.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy with default-preserving behavior, short-outage coalescing, and docs/config coverage; generalize only if multiple channel plugins need the same contract.

## Reproduction Assessment

Yes, by source inspection. Current main still enqueues WhatsApp connected/disconnected lifecycle notices and drains retained queue entries into prompt-visible `System:` lines, with no threshold, verbosity, or surface-aware filtering.

## Solution Assessment

Unclear until maintainers choose the product policy. A shared prefix filter is too broad because it can hide terminal disconnect context; a channel-owned lifecycle policy with explicit defaults is safer.
