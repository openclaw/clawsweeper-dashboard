## Summary

Keep open. Current main still queues WhatsApp connected/disconnected lifecycle notices into the resolved session and drains generic queued system events into prompt-visible `System:` lines; there is no threshold, verbosity setting, reconnect coalescing, or surface-aware policy yet. The related banner-filter PR was closed unmerged, and the merged WhatsApp delivery-drain PR addresses delayed outbound delivery rather than lifecycle-status suppression.

## What This Changes

Request configurable, threshold-based, or surface-aware suppression of transient channel connection status events so short WhatsApp reconnects do not clutter active session prompt context.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy with backward-compatible channel/account config and reconnect coalescing, then promote a generic SDK/helper contract only if multiple channel plugins need the same behavior.

## Reproduction Assessment

Yes by source inspection. Current main enqueues WhatsApp connected/disconnected notices in the monitor path and the shared prompt drain retains generic queued events as `System:` lines unless they match the existing heartbeat/periodic/node filters.

## Solution Assessment

Unclear until maintainers choose the user-facing policy. The safest direction is a WhatsApp-owned lifecycle policy with explicit defaults, not the older shared prefix filter that would hide all matching disconnect context.
