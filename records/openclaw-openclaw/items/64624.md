## Summary

Keep open. Current main and the latest release still enqueue WhatsApp connected/disconnected lifecycle notices into the resolved session and drain generic queued events into prompt-visible `System:` lines; the closed banner-filter PR was explicitly superseded by this issue rather than merged.

## What This Changes

Request suppression, throttling, or surface-aware filtering for transient channel connection status system events so short reconnect flaps do not clutter active sessions.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy with explicit defaults and reconnect coalescing, then promote a generic SDK/helper contract only if more channel plugins need the same behavior.

## Reproduction Assessment

Yes, by source inspection. Current main enqueues WhatsApp connected/disconnected notices and the shared prompt drain formats generic queued events as `System:` lines unless they match existing unrelated filters.

## Solution Assessment

Unclear until maintainers choose the user-facing policy. The narrow maintainable direction is a WhatsApp-owned lifecycle policy with explicit defaults, not the older shared prefix filter that would hide all matching disconnect context.
