## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the resolved session, and the exposed health-monitor config only supports enablement, not thresholding, verbosity, coalescing, or surface-aware filtering.

## What This Changes

The issue requests suppression, throttling, or surface-aware filtering for transient channel connection status system events.

## Best Possible Solution

Add a WhatsApp-owned lifecycle notification policy with reconnect coalescing and documented channel/account config, without changing the shared system-event queue semantics for unrelated durable notices.

## Solution Assessment

Yes. A channel-owned policy is the narrowest maintainable fix because the noise originates in the WhatsApp lifecycle enqueue path, while the shared queue also carries unrelated durable system notices.
