## Summary

Keep open. Current main still queues WhatsApp connected/disconnected lifecycle notices immediately into the resolved session system-event queue, and the available config/docs surface still lacks a threshold, verbosity mode, reconnect coalescing, or surface-aware filtering.

## What This Changes

The issue requests suppressing, throttling, or filtering transient channel connection status system events so short reconnects do not clutter active sessions.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy that coalesces short outages before they become session system events, with documented channel/account config for any user-facing verbosity or notify-after setting.

## Reproduction Assessment

Yes. Source inspection on current main shows any WhatsApp monitor connect followed by a close decision enqueues connected and disconnected system events, and the next prompt drains generic queued events into System lines.

## Solution Assessment

Yes. A WhatsApp-owned lifecycle notification policy with backward-compatible config is the narrowest maintainable path; changing the shared system-event queue would risk hiding unrelated durable system notices.

Add a WhatsApp-owned notification policy that can suppress/coalesce short reconnect outages before connected/disconnected lifecycle notices are enqueued as session system events.
