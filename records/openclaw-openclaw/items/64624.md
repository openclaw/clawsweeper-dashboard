## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the resolved session, and the generic system-event queue only drops exact consecutive duplicate text. I found no implemented threshold, verbosity setting, reconnect coalescing, or surface-aware filter, and the related tool-warning and Feishu health-monitor items do not supersede this request.

## What This Changes

The issue requests suppressing, throttling, or filtering transient channel connection-status system events so short reconnects do not clutter unrelated active sessions.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy that coalesces short disconnect/reconnect cycles before they become session system events, with explicit channel/account config and docs for any user-facing verbosity or notify-after setting. Keep the shared system-event queue generic unless multiple channel plugins need the same contract.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: any WhatsApp monitor connect followed by a close decision queues connected and disconnected system events for the resolved session, and the shared queue only suppresses exact consecutive duplicate text. I did not run live Baileys or tests because this review was read-only.

## Solution Assessment

Yes. A narrow WhatsApp-owned policy is the best current direction because changing the shared system-event queue could hide unrelated durable notices, while a channel-local gate can target the reported reconnect churn and keep existing defaults compatible.
