## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the resolved session and drains generic system events into prompts, while the exposed config/docs surface still lacks a threshold, verbosity mode, reconnect coalescing, or surface-aware policy; the closed banner-filter PR was explicitly superseded by this issue rather than fixing it.

## What This Changes

Request configurable suppression, throttling, or surface-aware filtering for transient channel connection status system events, especially WhatsApp reconnect notices.

## Best Possible Solution

Implement a WhatsApp-owned lifecycle notification policy with backward-compatible channel/account config and reconnect coalescing, then promote a generic plugin SDK helper only if more channels need the same contract.

## Reproduction Assessment

Yes. Source inspection on current main shows a WhatsApp connect followed by a close decision enqueues connected and disconnected system events, and the prompt drain keeps those generic events as System lines.

## Solution Assessment

Yes, the requested direction is better solved at the WhatsApp/channel lifecycle source than by a shared prompt-string filter. The exact threshold/verbosity/surface policy still needs a maintainer product decision.
