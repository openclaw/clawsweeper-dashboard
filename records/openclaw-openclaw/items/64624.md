## Summary

Keep open. Current main still immediately enqueues WhatsApp connected/disconnected lifecycle notices into the resolved session system-event queue, and the shared queue only suppresses exact consecutive duplicate text; there is no threshold, verbosity setting, reconnect coalescing, or surface-aware filter.

## What This Changes

The issue asks for configurable suppression, throttling, or surface-aware filtering of short-lived channel connection status system events.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy that coalesces short outages before they become session system events, with documented config for any user-facing verbosity or notify-after setting.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: any WhatsApp monitor connect followed by a close decision enqueues connected and disconnected system events, and the next prompt drains those queued lines.

## Solution Assessment

Yes. A narrow WhatsApp-owned policy is the best current direction; changing the shared system-event queue would risk hiding unrelated durable system notices.
