## Summary

Keep open. Current main still enqueues WhatsApp connected/disconnected lifecycle notices immediately into the resolved session system-event queue, and the exposed config/docs surface still lacks a threshold, verbosity mode, reconnect coalescing, or surface-aware filter; related #39406 and #41940 cover different warning and health-monitor surfaces.

## What This Changes

The issue requests a configurable policy to suppress, throttle, or scope transient channel lifecycle system events so short reconnect churn does not enter unrelated sessions.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy first, with backward-compatible channel/account config and reconnect coalescing for outages that recover before the notification threshold.

## Reproduction Assessment

Yes. Source inspection on current main shows any WhatsApp monitor connect followed by a close decision enqueues connected and disconnected system events, and the next prompt drains generic queued events into System lines.

## Solution Assessment

Yes. A WhatsApp-owned lifecycle notification policy is narrower and safer than changing the shared system-event queue; a generic SDK helper should wait until more channel plugins need the same contract.
