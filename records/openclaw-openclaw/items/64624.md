## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle system events into the resolved session, and there is no threshold, verbosity, reconnect coalescing, or surface-aware policy on the exposed config/docs surface; the closed banner-filter PR was explicitly superseded by this issue, not a fix for it.

## What This Changes

Request to suppress or throttle transient channel connection status system events, especially short WhatsApp reconnect banners that clutter unrelated active sessions.

## Best Possible Solution

Implement a WhatsApp-owned lifecycle notification policy with backward-compatible channel/account config and reconnect coalescing, then promote a generic plugin SDK helper only if multiple channel plugins need the same contract.

## Reproduction Assessment

Yes. Source inspection on current main shows a WhatsApp connect followed by any close decision enqueues connected and disconnected system events, and the next prompt drains retained generic events into System lines.

## Solution Assessment

Yes. A WhatsApp-owned lifecycle notification policy is narrower and safer than a shared prompt-string filter, because it can preserve terminal outage context while coalescing transient reconnect churn at the source.
