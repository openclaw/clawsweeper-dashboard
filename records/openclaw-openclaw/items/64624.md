## Summary

Keep open. Current main and v2026.6.10 still enqueue WhatsApp connected/disconnected lifecycle notices into prompt-visible system events, and the requested suppression/throttling/surface filtering remains a product/config policy decision.

## What This Changes

Request to suppress, throttle, or surface-filter transient channel connection status system events so short reconnects do not clutter active sessions.

## Best Possible Solution

Implement a default-compatible WhatsApp-owned lifecycle notification policy with short-outage coalescing and focused tests, then generalize only if multiple channel plugins need the same contract.

## Reproduction Assessment

Yes, by source inspection: current main still queues WhatsApp connected/disconnected lifecycle notices and drains queued system events into prompt-visible `System:` lines. I did not run a live WhatsApp reconnect scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. A broad shared prefix filter was already closed as too broad, so the safer direction is a channel-owned lifecycle policy with explicit defaults and preservation tests.
