## Label Justifications

- `P2`: The issue is a bounded media-task behavior problem that can suppress a requested output without causing a core outage.
- `impact:session-state`: The task ledger and agent-visible status can diverge from the user's multi-image request when the second request is treated as the first task's status.

## Summary

Keep open: current main still returns the active session image task as a duplicate-guard status before the second request's prompt, filename, or options are parsed, so distinct second image requests can be suppressed without a clear rejection.

## What This Changes

The issue asks OpenClaw to avoid silently treating a distinct second `image_generate` request as a status check for the first active session task.

## Best Possible Solution

Preserve protection against same-request retries, but make distinct active-session image requests explicit: either queue/start them with their own request metadata or return a structured rejection that clearly says no new generation was started.

## Reproduction Assessment

Yes by source inspection: in a session with any queued/running image task, a second generate action returns duplicate-guard status before its distinct prompt or filename can be represented. I did not run a live provider repro in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch and the product choice matters: queueing distinct requests is broader, while an explicit active-task rejection is the narrowest way to stop false progress reports.
