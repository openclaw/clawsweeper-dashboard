## Label Justifications

- `P1`: Linux memory watcher descriptor fan-out can exhaust watcher resources and degrade long-running memory_search or gateway workflows.
- `merge-risk: 🚨 session-state`: Merging changes Linux memory change detection, so missed watcher events could leave the memory index stale.
- `merge-risk: 🚨 availability`: Watcher setup, fallback, or descriptor-scaling mistakes could recreate ENOSPC pressure or degraded gateway availability on Linux.

## Summary

Keep open for maintainer review. I found no blocking correctness or security finding: current main still routes Linux memory directories through chokidar, the upstream Node and Chokidar sources support the fan-out concern, and the PR's directory-only Linux watcher is in the right memory-core owner boundary; the main remaining question is how much Linux live-gateway proof maintainers want before merge.

## What This Changes

The PR adds a Linux-specific directory-only native fs.watch tree watcher for memory-core directories, keeps chokidar for file paths and fallback paths, and expands watcher configuration tests.

## Best Possible Solution

Land the directory-only Linux watcher after maintainer review confirms the Linux proof and watcher-fallback risk are acceptable, while keeping chokidar fallback and macOS/Windows behavior intact.

## Reproduction Assessment

Yes, source and dependency proof give a high-confidence reproduction path: current main routes Linux directories through chokidar, while Node and Chokidar source show per-entry watcher allocation paths. I did not rerun the Linux probe in this read-only review.

## Solution Assessment

Yes, this appears to be the best owner-boundary fix: keep memory-core responsible for its watcher backend, avoid Node's recursive Linux path, and preserve chokidar for file paths and degraded fallback. The safer pre-merge addition is live Linux gateway proof, not a different code layer.
