## Summary

Keep open for normal maintainer review. I found no blocking correctness or security issue; the source path supports the reported Gemini ACP timeout failure, but the fix is not on current main and should be sequenced with the overlapping ACP timeout-control PRs.

## What This Changes

The PR adds Gemini ACP command detection in the ACPX runtime so `timeout` and `timeout_seconds` config options are skipped for `gemini --acp` and `gemini --experimental-acp`, with regression tests preserving forwarding for other ACP agents.

## Best Possible Solution

Land this narrow Gemini timeout no-op, or fold the same behavior into the chosen broader ACP timeout-control fix while preserving non-timeout config forwarding.

## Reproduction Assessment

Yes, by source inspection. Persisted `timeoutSeconds` maps to `timeout`/`timeout_seconds`, current main forwards those through ACPX for Gemini, and upstream Gemini ACP does not expose a config-option control.

## Solution Assessment

Mostly yes. The adapter-local no-op matches the existing Codex precedent and keeps unrelated Gemini ACP config forwarding intact; maintainers should sequence it against the broader open timeout-control PRs.
