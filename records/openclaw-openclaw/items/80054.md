## Summary

Keep this PR open: it is a plausible narrow bug fix for ACPX/Gemini timeout forwarding, but it still needs real Gemini ACP behavior proof before merge and has one small compatibility gap around Gemini's older experimental ACP flag.

## What This Changes

The PR changes the ACPX runtime to treat Gemini `timeout` and `timeout_seconds` config controls as no-ops and adds focused unit coverage for `gemini --acp`.

## Best Possible Solution

Land a narrow ACPX adapter fix that suppresses unsupported Gemini timeout controls while preserving other config controls, after adding real Gemini ACP proof and covering the experimental-flag command form if supported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist `timeoutSeconds` on a Gemini ACP session, then run a turn so the ACP control plane sends `timeout` through the current ACPX adapter before prompt execution.

## Solution Assessment

Mostly yes. Handling this as a no-op at the ACPX adapter boundary matches the existing Codex special case and avoids changing generic ACP control-plane behavior; the final patch should also cover `--experimental-acp` and include real behavior proof.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Recognize Gemini's experimental ACP flag:** `extensions/acpx/src/runtime.ts:402`
  - body: ACPX 0.7.0 still treats `gemini --experimental-acp` as a Gemini ACP command for older or custom Gemini CLI setups, but this predicate suppresses timeout controls only when `--acp` is present. Those sessions would still forward the unsupported timeout config and hit the same failure path this PR is trying to avoid.
