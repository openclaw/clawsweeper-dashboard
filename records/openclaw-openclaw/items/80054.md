## Summary

Keep open: this is a focused bug fix that is not on current main, and source plus upstream ACP inspection supports the reported failure path. The patch looks correct, but it should be reconciled with the overlapping broader Claude ACP timeout PR and current CI state before landing.

## What This Changes

The branch adds Gemini ACP command detection in the ACPX runtime, no-ops Gemini timeout config forwarding with focused tests, and refreshes the generated Swift gateway model field already present on current main.

## Best Possible Solution

Land a targeted ACPX adapter fix that drops unsupported timeout controls only for known agents that reject them, with regression coverage and coordination with the broader Claude timeout PR.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main forwards non-Codex timeout config options, acpx maps that to session/set_config_option, and Gemini ACP does not expose that handler. I did not run a credentialed Gemini ACP turn locally.

## Solution Assessment

Yes, the PR's Gemini-specific guard is a narrower maintainable fix than a global timeout drop because it preserves timeout forwarding for other ACP agents. The remaining judgment is how to reconcile it with https://github.com/openclaw/openclaw/pull/75306.
