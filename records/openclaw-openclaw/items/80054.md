## Summary

Keep open: current main still forwards non-Codex ACP timeout config options, and this PR is a focused fix for the same unsupported-control pattern already handled for Codex. The patch looks correct, but it should be coordinated with the broader open ACP timeout-control work before landing both.

## What This Changes

The PR adds Gemini ACP command detection so ACPX no-ops `timeout` and `timeout_seconds` config options for `gemini --acp` and `gemini --experimental-acp`, plus focused regression coverage and a generated Swift protocol refresh already present on current main.

## Best Possible Solution

Land the narrow Gemini ACP adapter no-op with regression coverage, or fold the Gemini case into the broader timeout-control fix if maintainers choose one unified ACP policy.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main maps `timeoutSeconds` into a timeout config option and ACPX only filters that key for Codex, while Gemini ACP upstream exposes mode/model controls but no generic config-option handler. I did not run a live credentialed Gemini ACP turn in this read-only pass.

## Solution Assessment

Yes, the proposed adapter-boundary skip is a narrow maintainable fix because it mirrors the existing Codex ACP timeout no-op and leaves non-timeout controls untouched. The only better alternative is to intentionally consolidate this with the broader timeout-control PR rather than merge two divergent policies.
