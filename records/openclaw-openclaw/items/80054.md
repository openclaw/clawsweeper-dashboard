## Summary

Keep open: source review supports the bug boundary, but this external PR still has only format/unit proof and one low-priority edge-case gap for older/custom Gemini ACP commands.

## What This Changes

The PR changes the ACPX runtime so Gemini ACP commands ignore `timeout` and `timeout_seconds` runtime config controls instead of forwarding them, with focused unit coverage.

## Best Possible Solution

Land a narrow ACPX adapter fix that suppresses unsupported Gemini timeout controls while preserving other config controls, after adding real Gemini ACP proof and covering the experimental-flag command form if supported.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: persist `timeoutSeconds` on a Gemini ACP session, then run a turn so the ACP control plane sends `timeout`/`timeout_seconds` through the current ACPX adapter. I did not run a live Gemini ACP session in this read-only review.

## Solution Assessment

Mostly yes: handling the no-op at the ACPX adapter boundary matches the existing Codex special case and avoids changing generic ACP control-plane behavior. The safer final version should also recognize Gemini's `--experimental-acp` form and include real after-fix behavior proof.
