## Summary

Keep open for normal PR review. Current main still forwards Gemini ACP timeout config controls, this branch is a focused adapter-boundary fix with no blocking correctness or security finding, and merge sequencing should account for the open broader timeout-control PRs.

## What This Changes

The PR detects Gemini ACP commands in the ACPX runtime and skips `timeout`/`timeout_seconds` config forwarding for them, with unit coverage for `--acp` and `--experimental-acp`.

## Best Possible Solution

Land one narrow timeout-control fix that prevents unsupported Gemini timeout config calls while preserving non-timeout forwarding, or fold this exact behavior into the chosen broader ACP timeout-control PR.

## Reproduction Assessment

Yes, by source inspection. Persist `timeoutSeconds` for a Gemini ACP session: current main maps it to `timeout`/`timeout_seconds`, applies it before the turn, and forwards non-Codex config options even though Gemini ACP docs do not advertise `session/set_config_option`.

## Solution Assessment

Mostly yes. The adapter-local no-op matches the existing Codex precedent and keeps unrelated Gemini ACP config forwarding intact; maintainers should still sequence it against the broader open timeout-control PRs.
