## Summary

Keep this PR open for normal review and landing: the change is a narrow ACPX adapter fix for a source-reproducible Gemini ACP timeout-control failure, and I found no blocking correctness or security finding.

## What This Changes

The PR changes the ACPX runtime so Gemini ACP timeout config controls are ignored instead of forwarded, with regression coverage for `gemini --acp` and `gemini --experimental-acp`.

## Best Possible Solution

Land a small ACPX adapter fix that preserves the generic manager contract, no-ops only Gemini ACP timeout controls, and keeps forwarding all other Gemini and non-Gemini config options.

## Reproduction Assessment

Yes from source inspection, though I did not run a live Gemini turn. Current main applies persisted `timeoutSeconds` through `setConfigOption`, and the ACPX adapter only suppresses that key for Codex, so a Gemini ACP command reaches the delegate with an unsupported timeout config option.

## Solution Assessment

Yes. Matching the existing Codex ACP timeout no-op at the ACPX adapter boundary is the narrowest maintainable fix because the manager remains generic while the vendor-specific unsupported control is handled where the command identity is known.
