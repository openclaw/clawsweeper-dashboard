## Summary

Keep open: current main still forwards these timeout controls for Gemini, upstream Gemini ACP does not advertise or handle config-option requests, and the proposed runtime/test change is focused with no blocking code findings. Landing still needs the red PR validation resolved.

## What This Changes

The PR adds ACPX runtime handling and tests to no-op `timeout`/`timeout_seconds` config controls for Gemini ACP commands while preserving other config forwarding, plus a generated Swift protocol field refresh.

## Best Possible Solution

Land a rebased version of the focused ACPX timeout no-op after required checks are green, or deliberately fold it into the broader ACP timeout-policy work in https://github.com/openclaw/openclaw/pull/75306.

## Reproduction Assessment

Yes, via source-level reproduction: current main forwards timeout controls for non-Codex ACP agents, OpenClaw defaults Gemini ACP to `gemini --acp`, and upstream Gemini ACP exposes no config-option handler. I did not run a live credentialed Gemini turn.

## Solution Assessment

Yes for the documented Gemini ACP command: mirroring the existing Codex timeout no-op at the OpenClaw ACPX wrapper boundary is narrow and preserves non-timeout forwarding. The main alternative is a broader timeout skip shared with https://github.com/openclaw/openclaw/pull/75306.
