## Summary

Keep open: current `main` still has the non-activating loader restore behavior, and this PR’s helper-only change does not address the restore boundary that can drop memory corpus supplements.

## What This Changes

The PR changes `registerMemoryCorpusSupplement` so later same-plugin registrations are ignored instead of replacing the existing supplement.

## Best Possible Solution

Fix the loader state restoration or documented registration boundary so active memory corpus supplements survive lazy non-activating loads without changing same-plugin replacement semantics by accident.

## Reproduction Assessment

No. Source inspection shows why this patch cannot fix the stated lazy restore path, but the exact service/lazy sequence still needs a focused current-main regression test.

## Solution Assessment

No. The narrow maintainable fix belongs at the loader restore boundary or SDK lifecycle contract, not in a helper change that makes all same-plugin corpus supplement registrations first-wins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Fix the restore path instead of making supplements first-wins:** `src/plugins/memory-state.ts:157-161`
  - body: The reported failure happens when a non-activating load restores `previousMemoryCorpusSupplements` after `register()` returns. This helper is bypassed if the service does not re-register on the second load, and any registration made during that second load is overwritten by `restoreMemoryPluginState`; it also ignores later legitimate same-plugin replacements. Please preserve the active supplement at the loader restore boundary and add a lazy-load regression test.
  - confidence: 0.84
