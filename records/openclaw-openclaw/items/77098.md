## Summary

Keep open: the reported memory supplement loss is plausible enough to review, but this PR is not safe to merge as-is because the helper change does not fix the loader restore path described by the PR and there is no regression coverage for the lazy re-register sequence.

## What This Changes

The PR changes `registerMemoryCorpusSupplement` in `src/plugins/memory-state.ts` from replacing a plugin's existing corpus supplement to ignoring later registrations for the same plugin.

## Best Possible Solution

Add a regression test for #77039's lazy non-activating plugin load and fix the loader state restoration or registration boundary that actually drops the active supplement, while preserving intentional replacement semantics unless the SDK contract is updated.

## Reproduction Assessment

No. Source inspection shows the proposed helper change cannot affect the stated restore path, but the exact current-main sequence that loses the supplement still needs a focused lazy-load regression test.

## Solution Assessment

No. The narrow maintainable fix should target `loadOpenClawPlugins` memory snapshot/restore behavior, or the service/runtime registration contract, rather than making all same-plugin corpus supplement registration first-wins.

## Review Findings

Full review comments:

- **[P2] Fix the restore path instead of making supplements first-wins:** `src/plugins/memory-state.ts:157-161`
  - body: The failure described here happens when a non-activating load restores `previousMemoryCorpusSupplements` after `register()` runs. Please add a regression test for the lazy re-register path and preserve or merge the active supplement where the loader restores memory state.
