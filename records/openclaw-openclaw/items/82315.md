## Summary

Current main already addresses the stale Control UI bundle path that matches the reported protocol-mismatch loop, and the apparent `Unauthorized` URL is the token-protected browser-control plugin endpoint rather than the Gateway dashboard. The fix is current-main-only and not in the latest `v2026.5.12` release.

## What This Changes

The issue reports that after upgrading to 2026.5.12 on Windows 10, the browser dashboard loops on Gateway protocol mismatch and an attempted browser-control URL returns `Unauthorized`.

## Best Possible Solution

Keep the build-scoped service-worker cache rotation and protocol-mismatch recovery docs; users on `v2026.5.12` should open the dashboard on port `18789` and hard-refresh or clear site data until a release containing the fix is available.

## Reproduction Assessment

Yes at source level: a stale Control UI client whose protocol range does not include protocol 4 reaches the current server's `protocol mismatch` close path, and the issue logs show that exact failure class. I did not run the Windows Edge upgrade path live.

## Solution Assessment

Yes. The current-main fix addresses the stale served-dashboard bundle cause without weakening the intentional protocol-v4 rejection path; the separate `18791` `Unauthorized` observation is expected plugin auth behavior, not the dashboard endpoint.
