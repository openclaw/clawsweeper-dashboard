## Summary

Current main addresses the reported protocol-mismatch-after-update path by rotating the Control UI service-worker cache per build and documenting the hard-refresh/site-data recovery path for already-stale tabs; the direct `127.0.0.1:18791` Unauthorized response is the separate browser-control token gate, not the Gateway Control UI.

## What This Changes

The issue reports a Windows 10 npm upgrade to 2026.5.12 leaving the browser Control UI in repeated `protocol mismatch` WebSocket failures, with a direct browser-control port visit showing `Unauthorized`.

## Best Possible Solution

Keep the current build-scoped service-worker cache behavior, and have affected 2026.5.12 users hard-refresh or clear Control UI site data until a release containing the fix is installed.

## Reproduction Assessment

Yes at source level: the shipped 2026.5.12 service worker used one fixed cache name while the Gateway protocol rejected stale Control UI bundles, matching the repeated `protocol mismatch` logs. I did not live-run Windows 10 Edge against current main.

## Solution Assessment

Yes. Rotating the service-worker cache per build is the narrow maintainable fix because it prevents old Control UI bundles from continuing to run against a newer Gateway without weakening auth or protocol checks.
