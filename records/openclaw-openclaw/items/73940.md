## Summary

Keep open. The Feishu WebSocket retry-exhaustion bug is still not implemented on current main, and this PR targets the right missing channel-owned recovery path, but the proposed patch has a concrete backoff/security concern and overlaps an open maintainer-labeled alternative PR, so it needs maintainer direction before merge.

## What This Changes

The PR adds Feishu WSClient lifecycle callbacks, a supervisor loop that recreates clients after SDK retry exhaustion or stalled reconnect state, bot identity preservation across replacement cycles, tests, and a changelog entry.

## Best Possible Solution

Use one maintainer-approved Feishu plugin transport implementation as the canonical #52618 fix: observe terminal SDK retry exhaustion through the Lark lifecycle contract, preserve bot identity only across recovery cycles, keep shutdown cleanup deterministic, and ensure supervisor backoff increases across rapid reconnect failures unless a connection has been stable long enough to reset safely.

## Reproduction Assessment

Yes. A focused reproduction is to simulate a WSClient whose start() returns, then invoke the SDK onError callback with the documented "WebSocket reconnect exhausted after N attempts" terminal error and verify current main never recreates the client while the PR does; the PR finding is reproducible by repeating onReady/onError cycles and observing the delay remains at 1000ms.

## Solution Assessment

No, not as currently proposed. The Feishu-owned supervisor direction is appropriate, but resetting backoff after any prior connection is not the narrowest safe behavior, and the implementation should be reconciled with the open maintainer alternative before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset backoff only after a stable connection:** `extensions/feishu/src/monitor.transport.ts:318-321`
  - body: This resets `attempt` whenever the cycle has ever reached `onReady` or `onReconnected`, so a flaky network that briefly connects and then exhausts SDK retries will recreate clients at the 1s minimum forever. That defeats the bounded backoff and can amplify WSClient churn and logs; keep the attempt increasing unless the connection stayed healthy for a meaningful interval, or add equivalent rate limiting/jitter.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity availability concern in the new supervisor backoff behavior.

Concerns:

- **[low] Backoff reset can amplify reconnect churn:** `extensions/feishu/src/monitor.transport.ts:318`
  - body: After any prior successful connection, the supervisor resets `attempt` before scheduling client recreation. Repeated brief connect/disconnect failures can therefore remain at the 1000ms minimum delay, increasing socket churn and log volume under unstable network conditions.
