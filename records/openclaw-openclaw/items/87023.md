## Summary

Keep open: current main and v2026.6.8 still lack first-class MCP binary-to-channel relay, and PR #94663 is an open partial implementation candidate rather than merged coverage for the broader channel/API/hook request.

## What This Changes

The issue requests first-class relay of MCP binary content blocks to channel attachments with safe transcript rewrite and a documented extension seam.

## Best Possible Solution

Define a narrow maintainer-approved MCP media relay contract that stages safe MCP binary content into OpenClaw-owned media, routes through existing channel outbound adapters, and rewrites transcript/model context explicitly.

## Reproduction Assessment

Yes, source inspection on current main shows MCP audio/resources are converted to text, embedded base64 images are not staged as sendable media, and MCP local media paths are filtered unless HTTP(S).

## Solution Assessment

No complete current solution exists; the right path needs a narrow SDK/media/channel contract and security review rather than an unrestricted public byte sink or hardcoded MCP forwarding.
