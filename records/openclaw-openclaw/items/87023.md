## Summary

Keep open: current main and v2026.6.9 still do not provide first-class MCP binary-to-channel relay, and the active implementation candidate at https://github.com/openclaw/openclaw/pull/94663 is open rather than merged coverage for the broader API, channel, hook, and transcript behavior.

## What This Changes

The issue requests a first-class OpenClaw path for MCP non-text `CallToolResult.content[]` blocks to be preserved, safely staged, delivered as channel attachments, and rewritten out of persisted/model-visible transcript context.

## Best Possible Solution

Define and land a narrow maintainer-approved MCP media relay contract that stages safe MCP binary content as OpenClaw-owned media, reuses existing outbound channel delivery, and rewrites model/transcript state without exposing unrestricted byte-sending APIs.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the missing capability: current main coerces MCP audio/resources to text, does not stage embedded base64 image data as media, and filters MCP-provenance local media paths. I did not run a live MCP-to-channel attachment flow because this is a broad feature request rather than one failing command path.

## Solution Assessment

Unclear as proposed: the direction is valid, but the best permanent shape needs maintainer-approved API and security boundaries. The safer current path is the narrower host-owned staging approach in the open candidate PR, with native channel and hook behavior reviewed as follow-up product/API decisions.
