## Summary

Keep open: current main has useful partial foundations for MCP content preservation, tool-result middleware, transcript rewrite hooks, and channel media payloads, but it still does not provide the requested first-class MCP binary-to-channel relay. Source inspection shows MCP image blocks are preserved as tool-result content, not converted to sendable channel media; local MCP-provenance media remains intentionally untrusted; and session attachment delivery is still constrained to trusted/bundled-style active-session paths rather than a general public channel attachment sink.

## What This Changes

The issue requests first-class relay of MCP non-text CallToolResult content to chat channels as attachments, with transcript rewrite support.

## Best Possible Solution

Design a narrow MCP-to-reply-media contract that converts safe content blocks into channel-deliverable media through the existing delivery pipeline, preserves transcript/model context with explicit rewrite semantics, and gates credentialed attachment sending behind a maintainer-approved plugin/session trust model.

## Reproduction Assessment

Yes by source inspection for the missing automatic MCP relay path. Current main preserves valid MCP image blocks, but the sendable-media path requires details.media/path/URL and explicitly blocks local MCP-provenance media.

## Solution Assessment

No, current main is not the full solution for this issue. The requested direction is aligned, but the maintainable fix should be a narrow SDK/channel/media contract rather than hard-coding MCP media forwarding in core or widening channel-send access without a trust model.
